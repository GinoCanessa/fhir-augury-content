## Resolution Description

> Update the PDDI specification to indicate that a CDS service may choose to use a DetectedIssue resource to convey additional information about a drug-drug interaction. If a DetectedIssue is provided, it should be incorporated within the CDS Hooks response as a suggested action with type create.

## Analysis

### Background

The PDDI (Potential Drug-Drug Interaction) CDS Implementation Guide currently defines how a CDS Hooks service returns response cards containing drug-drug interaction alerts. Cards include `summary`, `indicator`, `detail`, `source`, and `suggestions` with `actions` (of type `create`, `update`, or `delete`) carrying FHIR resources like `MedicationRequest`.

The ticket requests that the IG be updated to allow a CDS service to optionally include a `DetectedIssue` FHIR resource in the response, enabling richer, structured representation of DDI information beyond the free-text `detail` and `summary` fields of a Card.

### DetectedIssue Resource Relevance

The `DetectedIssue` resource (from FHIR core) is specifically designed for drug-drug interactions and similar clinical safety concerns. Key elements that add value beyond the current Card format include:

- **`status`** – lifecycle state (e.g., `preliminary`, `final`)
- **`category`** / **`code`** – coded classification (e.g., `DRG` for Drug Interaction Alert)
- **`severity`** – coded severity (high/moderate/low), complementing the Card `indicator`
- **`subject`** – reference to the patient
- **`implicated`** – references to the specific medications involved (MedicationRequest, MedicationStatement, etc.)
- **`evidence`** – supporting evidence for the detected issue
- **`detail`** – human-readable description of the risk
- **`qualityOfEvidence`** / **`expectedOnsetType`** / **`medicationClass`** / **`managementCode`** – structured metadata
- **`mitigation`** – actions already taken or recommended, with dates and authors
- **`reference`** – URI to relevant literature or guidelines

### Prior Art (FHIR-18804)

Ticket [FHIR-18804](https://jira.hl7.org/browse/FHIR-18804) ("DetectedIssue in CDS service response - PDDI #1") previously explored this concept but was deferred in 2020 with the comment: "We are no longer proposing to include a DetectedIssue resource in Card.suggestion.action." The current ticket (FHIR-40035) revisits this with a clear resolution directing implementation via `suggestion.action` with type `create`.

### Key Design Decision

Per the resolution and comment from bryn.rhodes, the chosen approach is **option B**: model the DetectedIssue directly as an action in the CDS Hooks response. Specifically, the DetectedIssue should be provided as a `resource` within a `suggestion.action` of type `create`. This leverages the existing CDS Hooks action mechanism without requiring any CDS Hooks specification changes.

## Feature Proposal

### What Changes

1. **Normative/guidance text** in the PDDI IG stating that a CDS service MAY include a `DetectedIssue` resource in a CDS Hooks response card suggestion action with type `create`.
2. **Card Actions documentation** updated for each knowledge artifact (Warfarin+NSAIDs and Digoxin+Cyclosporine) to list `DetectedIssue` as an optional `create` action resource.
3. **Example JSON responses** showing a CDS Hooks card that includes a DetectedIssue `create` action alongside existing MedicationRequest actions.
4. **Documentation** describing how a DetectedIssue maps to the PDDI minimal information model elements already described in the IG.

### What Does NOT Change

- The CDS Hooks specification itself is not modified.
- Existing card structure, indicator logic, and decision trees remain unchanged.
- The DetectedIssue is **optional** (MAY), not required — backward compatibility is preserved.

## Detailed Implementation Plan

### Step 1: Update CDS Service Specification – Normative Recommendations

**File:** `input/pagecontent/cds-service-spec.md`

In the "Summary of Normative Recommendations" section near the top of the file, add a new recommendation:

```markdown
* A CDS service MAY include a `DetectedIssue` resource within a CDS Hooks response
  card's `suggestions[].actions[]` with `type` set to `create` to convey structured,
  machine-readable information about a detected drug-drug interaction. If provided,
  the DetectedIssue resource SHOULD populate at minimum the `status`, `code`,
  `severity`, `implicated`, and `detail` elements.
```

### Step 5: Update Documentation Page

**File:** `input/pagecontent/documentation.md`

Add a new subsection (e.g., "6.x.x Using DetectedIssue in CDS Hooks Responses") covering:

1. **Purpose:** Explain that while the Card `detail` and `summary` provide human-readable DDI information, a `DetectedIssue` resource provides a structured, machine-readable representation that can be persisted by the EHR for downstream use cases such as:
   - Tracking unresolved detected issues across encounters
   - Re-evaluating DDIs when patient data changes
   - Clinical quality reporting
   - Medication reconciliation workflows

2. **Mechanism:** Describe how to include DetectedIssue within `suggestions[].actions[]` using type `create`, consistent with the CDS Hooks specification.

3. **Mapping to Minimal Information Model:** Show how DetectedIssue elements map to the PDDI minimal information model elements already described in the IG:
   | PDDI Element | DetectedIssue Element |
   |---|---|
   | Drugs Involved | `implicated` (references to medications) |
   | Clinical Consequences | `detail` |
   | Severity/Seriousness | `severity` |
   | Evidence | `evidence.code`, `evidence.detail`, `reference` |
   | Recommended Actions | `mitigation.action` |
   | Mechanism of Interaction | `detail` or extension |

4. **Conformance:** State that inclusion of DetectedIssue is OPTIONAL (MAY). Services not including it remain fully conformant.

## Implementation Considerations

1. **CDS Hooks Compatibility:** The DetectedIssue resource is used via the standard CDS Hooks suggestion action mechanism (type `create` with a FHIR resource). No changes to the CDS Hooks specification are required.

2. **EHR Behavior:** EHRs that do not understand the DetectedIssue action can safely ignore it per CDS Hooks convention. The existing card summary/detail/indicator still convey the alert information.

3. **Separate vs. Combined Suggestions:** The DetectedIssue should be in its own suggestion (not bundled with therapeutic alternative suggestions like MedicationRequest creates) so the EHR can independently accept/reject the DDI recording vs. the medication substitution.

4. **DetectedIssue Status Lifecycle:** The initial status should be `preliminary` when returned by the CDS service. The EHR would update to `final` upon clinician acknowledgment, or the mitigation elements could be populated when the clinician takes action.

5. **No Profile Required (Initially):** The initial implementation can reference the base DetectedIssue resource. A PDDI-specific DetectedIssue profile could be considered in a future ballot cycle if more constrained conformance is needed.

## Estimated Complexity

**Medium** — The changes are primarily narrative/documentation updates and example additions within the existing PDDI IG. No structural changes to the IG's resource definitions, profiles, or build infrastructure are required. The main effort is in crafting clear normative language and comprehensive examples.
