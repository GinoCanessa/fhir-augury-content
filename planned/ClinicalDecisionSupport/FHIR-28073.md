## Description

The ticket requests that nullability of arguments and return types be documented in the operator/function signatures throughout the CQL reference (Appendix B). Currently, signatures do not indicate whether arguments or return values can be `null`. For example, the Equal operator signature is:

```cql
=<T>(left T, right T) Boolean
```

But Equal can return `null` (when one or both arguments are null and the result is unknown). The proposed notation uses `?` to indicate nullability:

```cql
=<T>(left T, right T) Boolean?
```

## Resolution Description (verbatim)

> Agreed, document when arguments and results are nullable using the notation `?`
>
> Add a description of this notation to the discussion describing signature syntax at the beginning of the reference.

## Feature Proposal

### Overview

Add nullable type notation (`?` suffix) to all operator and function signatures in the CQL Reference (Appendix B) to clearly indicate which arguments accept `null` and which return types may be `null`. Also add an introductory paragraph at the beginning of the reference explaining the notation.

### Two-Part Change

1. **Signature Syntax Documentation**: Add a new paragraph to the existing signature-description prose at the beginning of `09-b-cqlreference.md` (before the "Types" section) that explains the `?` notation.

2. **Signature Updates**: Review and update all ~189 operator/function signatures to annotate nullable arguments and return types with `?`.

### Nullability Classification Rules

For implementing the signature updates, the following rules apply based on CQL semantics:

#### Return Type Nullability (`?` on return type)

| Rule | Operators Affected |
|------|--------------------|
| **Null propagation**: If any argument is null, result is null | Most arithmetic, string, date/time operators |
| **Three-valued logic**: Result can be null due to unknown | `and`, `or`, `xor`, `implies`, `not` |
| **Comparison with null**: `=`, `!=`, `<`, `>`, `<=`, `>=` return null when argument is null | Comparison operators (except `~` and `!~`) |
| **Conversion failure**: Invalid conversion returns null | `ToBoolean`, `ToInteger`, `ToDecimal`, `ToDate`, `ToDateTime`, `ToTime`, `ToQuantity`, `ToString` |
| **Empty/missing elements**: Can return null from collections | `First`, `Last`, `Indexer`, `Singleton From`, `Mode`, `Median` |
| **Always non-null return**: Never returns null | `Equivalent` (`~`), `Not Equivalent` (`!~`), `IsNull`, `IsTrue`, `IsFalse`, `Exists`, `AllTrue`, `AnyTrue`, `Count`, `Now`, `Today`, `TimeOfDay`, `Maximum`, `Minimum`, `Length` (List) |

#### Argument Nullability (`?` on argument types)

| Rule | Pattern |
|------|---------|
| **Arguments that accept null by design** | `Coalesce` arguments, `IsNull` argument, equivalence operators |
| **Arguments where null triggers null result** | Most operators — null propagation |
| **Arguments that must not be null** | Structural arguments like the list in aggregate functions |

### Key Design Decisions Required

1. **Argument vs. Return nullability scope**: The ticket description shows `?` on the return type (`Boolean?`). The resolution says "arguments and results." The implementer must decide for each of the ~189 signatures whether arguments also need `?` annotation.

2. **Generic type parameter nullability**: For signatures like `=<T>(left T, right T) Boolean?`, should the generic `T` itself carry `?` (e.g., `T?`) to indicate null-accepting arguments? Or is it implicit that generic `T` always includes null? This should be clarified with the work group — the CQL type system treats `null` as a value of any type, so annotating `T?` may be redundant for arguments, but annotating the return type as `T?` vs `T` is meaningful.

3. **Consistency with `?` in format strings**: The reference already uses `?` to mean "optional" in format string patterns (line 24). The new introductory text should differentiate the nullability `?` notation from the format string `?` notation.

## Implementation Plan

### Step 1: Add Nullability Notation Documentation

**File**: `input/pages/09-b-cqlreference.md`

**Location**: After the existing prose describing signature syntax (the paragraph beginning "For each operator or function, the signature, semantics, and usually an example are provided...") and before the formatting strings section (`{: #formatting-strings}`).

**Content to add** (draft):

```markdown
{: #nullability-notation}
In addition, the signatures use the `?` notation to indicate nullability. A type followed by `?`
indicates that the value may be `null`. When used on a return type, it indicates that the operator
may return `null` under certain conditions (typically due to null propagation or three-valued logic).
When used on an argument type, it indicates that the argument accepts `null` values. For example:

​```cql
=<T>(left T, right T) Boolean?
​```

indicates that the _equal_ operator returns a nullable `Boolean` (i.e., the result may be `true`,
`false`, or `null`), while:

​```cql
~<T>(left T, right T) Boolean
​```

indicates that the _equivalent_ operator always returns a non-null `Boolean` value.
```

### Step 2: Categorize All Signatures for Nullability

Before editing, systematically categorize every signature by reviewing the **Description** section of each operator to determine:
- Whether null arguments produce a null result (null propagation)
- Whether the operator has special null handling (e.g., equivalence, three-valued logic)
- Whether the return type is always non-null

This categorization should produce a lookup table mapping each operator to its nullable arguments and return type.

### Step 3: Update Signatures — By Section

Each section's signatures must be updated. Below is guidance per section:

#### 3a. Logical Operators
- `and(left Boolean?, right Boolean?) Boolean?` — three-valued logic
- `implies(left Boolean?, right Boolean?) Boolean?`
- `not(argument Boolean?) Boolean?`
- `or(left Boolean?, right Boolean?) Boolean?`
- `xor(left Boolean?, right Boolean?) Boolean?`

#### 3b. Type Operators
- `as<T>(argument Any) T?` — returns null if argument is not of type T
- `Convert`: returns nullable on conversion failure
- `To*` operators: return nullable on invalid input
- `Is`: returns `Boolean` (never null)
- `ConvertsTo*`: return `Boolean` (never null)
- `Children`/`Descendants`: return `List<Any>` (never null, but elements may be null)

#### 3c. Nullological Operators
- `Coalesce<T>(argument1 T?, argument2 T?, ...) T?` — arguments are explicitly nullable; result is nullable if all arguments are null
- `is null(argument Any) Boolean` — never null
- `is false(argument Boolean?) Boolean` — never null
- `is true(argument Boolean?) Boolean` — never null

#### 3d. Comparison Operators
- `=`, `!=`, `<`, `>`, `<=`, `>=`: return `Boolean?` (null if either argument is null)
- `~`, `!~`: return `Boolean` (never null, handles null inputs)

#### 3e. Arithmetic Operators
- Most return nullable (null propagation): `Abs`, `Add`, `Ceiling`, `Divide`, `Floor`, `Exp`, `Log`, `Ln`, `Modulo`, `Multiply`, `Negate`, `Power`, `Round`, `Subtract`, `Truncate`, `Truncated Divide`
- `Maximum`/`Minimum`: return `T` (never null)
- `Predecessor`/`Successor`: return `T?` (null if argument is null; error if at boundary)
- `Precision`: return `Integer?`

#### 3f. String Operators
- Most propagate null: `Combine`, `Concatenate`, `EndsWith`, `Indexer`, `LastPositionOf`, `Lower`, `Matches`, `PositionOf`, `ReplaceMatches`, `Split`, `StartsWith`, `Substring`, `Upper`
- `Length`: returns `Integer?` (null if argument is null)

#### 3g. Date and Time Operators
- `Now()`, `Today()`, `TimeOfDay()`: return non-nullable
- `Date()`, `DateTime()`, `Time()`: return non-nullable (constructed values)
- Comparison operators (After, Before, Same As, etc.): return `Boolean?`
- Arithmetic (Add, Subtract): return nullable
- `Duration`/`Difference`: return `Integer?`

#### 3h. Interval Operators
- `start of`, `end of`: return `T?`
- `contains`, `in`, `includes`, etc.: return `Boolean?`
- `intersect`, `except`: return `Interval<T>?`
- `union`: returns `Interval<T>?` (null if non-adjacent/non-overlapping)
- `Collapse`, `Expand`: return `List<Interval<T>>?`
- `Size`, `Width`: return `T?`
- `Point From`: return `T?`

#### 3i. List Operators
- `First`, `Last`: return `T?` (null if list is empty)
- `Indexer`: return `T?`
- `Singleton From`: return `T?`
- `IndexOf`: return `Integer?`
- `Exists`: return `Boolean` (never null)
- `Contains`, `In`: return `Boolean?`
- `Includes`, `Included In`: return `Boolean?`
- `Length`: return `Integer` (never null; 0 for null list)
- `Distinct`, `Except`, `Intersect`, `Union`, `Flatten`, `Skip`, `Tail`, `Take`: return list (generally non-null)

#### 3j. Aggregate Functions
- `Count`: return `Integer` (never null; 0 for null/empty)
- `AllTrue`, `AnyTrue`: return `Boolean` (never null)
- `Avg`, `GeometricMean`, `Max`, `Min`, `Median`, `Mode`, `Product`, `Sum`, `StdDev`, `PopulationStdDev`, `Variance`, `PopulationVariance`: return nullable (null for empty list)

#### 3k. Clinical Operators
- `AgeInYears()`, `AgeInYearsAt()`, `CalculateAgeInYears()`, `CalculateAgeInYearsAt()`: return `Integer?`

#### 3l. Errors and Messaging
- `Message`: returns `T` (passes through source)

### Step 4: Verify Consistency

After updating all signatures:
1. Ensure the `?` notation is used consistently across all overloads of the same operator
2. Cross-check with the operator descriptions to confirm null behavior matches the annotated signature
3. Verify no conflict with the `?` used in format string descriptions (different context, should be clear from surrounding text)

### Step 5: Review and PR

- Create a branch (e.g., `FHIR-28073-nullability-notation`)
- Submit PR to `HL7/cql` repository
- Reference FHIR-28073 in the PR description

## Risks and Considerations

| Risk | Mitigation |
|------|------------|
| **Massive scope**: ~189 signatures need individual review | Systematic categorization by section; use null propagation rules to batch-classify |
| **Generic type nullability ambiguity**: Whether `T?` is needed for generic parameters | Propose convention to work group: reserve `?` for return types; note in prose that generic arguments accept null unless otherwise stated |
| **Breaking existing tooling**: Tools parsing CQL signatures may not expect `?` | This is documentation-only (Appendix B pseudo-syntax), not grammar changes; no tooling impact |
| **Potential for errors**: Incorrectly marking an operator as nullable/non-nullable | Each operator's description already documents null behavior; use description as source of truth |
| **Format string `?` conflict**: Existing use of `?` for "optional" in format strings | Contextually distinct; add clarifying note in the new nullability section |

## Estimated Effort

**Medium-High** — While the conceptual change is simple (add `?` annotation), the implementation requires careful review of all ~189 operator descriptions to determine correct nullability for each argument and return type. Estimated at 4–8 hours for a domain expert familiar with CQL semantics.
