## Resolution Description

> Agreed, add more specific examples of categories in the example ValueSet for ActivityDefinition and PlanDefinition topic.

## Problem Statement

The `DefinitionTopic` CodeSystem (`http://terminology.hl7.org/CodeSystem/definition-topic`) contains only three high-level, abstract codes:

- `treatment` — Treatment
- `education` — Education
- `assessment` — Assessment

These describe the broad function of a PlanDefinition or ActivityDefinition but do not provide more specific categorization of the *content* of the activity. The ticket requests adding more granular, content-oriented example codes such as screening subtypes (e.g., diabetes-screening, mammography-screening) that would complement the existing high-level codes and support harmonization with CDS Hooks `card.source.topic`.

## Feature Proposal

Add additional, more specific example concept codes to the `DefinitionTopic` CodeSystem in the HL7 Unified Terminology (UTG) repository. These codes should represent content-level categories that drill down from the existing high-level function categories. Suggested codes based on the ticket description include screening-related categories:

- `screening` — general screening category
- `diabetes-screening`
- `mammography-screening`
- `colonoscopy-screening`
- `fecal-occult-blood-screening`
- `zika-screening`

The ValueSet (`http://terminology.hl7.org/ValueSet/definition-topic`) already includes all codes from the CodeSystem via `compose.include`, so adding codes to the CodeSystem will automatically expand the ValueSet.

## ⚠️ Critical Conflict: Topic Element Deprecation and Removal

**This ticket's implementation is significantly complicated by subsequent decisions that deprecate and remove the `topic` element entirely.**

### Deprecation History

1. **FHIR-38841** (Published, Persuasive with Modification) — Deprecated `topic` on metadata resources in favor of using `useContext` with code `http://terminology.hl7.org/CodeSystem/usage-context-type#topic`. The deprecation note is already present in the R6 source:

   > "DEPRECATION NOTE: For consistency, implementations are encouraged to migrate to using the new 'topic' code in the useContext element."

2. **FHIR-55564** (Resolved Persuasive) — Resolves to **remove** the `topic` element from `ActivityDefinition`, `EventDefinition`, `PlanDefinition`, and `Library` entirely in R6, since they were deprecated in R5.

3. **FHIR-55431** (Duplicate of FHIR-55564) — Specifically targeted removing `ActivityDefinition.topic`.

### Current State in Source

Both `ActivityDefinition.topic` and `PlanDefinition.topic` are already marked with `structuredefinition-standards-status` = `deprecated` in the FHIR core source:

- `source/activitydefinition/structuredefinition-ActivityDefinition.xml` (line 723)
- `source/plandefinition/structuredefinition-PlanDefinition.xml` (line 763)

### Impact on This Ticket

While the `topic` elements are being removed from the resources, the `DefinitionTopic` CodeSystem and ValueSet themselves **remain useful** because:

1. They are used as the example binding for the **`artifact-topic` extension** (`http://hl7.org/fhir/StructureDefinition/artifact-topic`) in HL7/fhir-extensions, which applies to many resource types
2. They can be used with `useContext.code = topic` as the migration path for the deprecated `topic` element
3. CDS Hooks `card.source.topic` may reference these codes for interoperability

**Recommendation:** Proceed with adding codes to the CodeSystem, but confirm with the CDS work group that this is still desired given the deprecation/removal context. The codes remain valuable for the `artifact-topic` extension and `useContext`-based approach.

## Detailed Implementation Plan

### Step 1: Add Content-Category Codes to CodeSystem (HL7/UTG)

**File:** `input/sourceOfTruth/fhir/codeSystems/CodeSystem-definition-topic.xml`

Add new `<concept>` elements after the existing three codes. The exact codes should be finalized with the CDS work group, but based on the ticket, a reasonable set would be:

```xml
<!-- Existing codes: treatment, education, assessment -->

<concept>
  <code value="screening"/>
  <display value="Screening"/>
  <definition value="The definition is related to screening of the patient."/>
</concept>
```

Additional, more specific screening subcategories could also be added, depending on the work group's preference for granularity:

```xml
<concept>
  <code value="diabetes-screening"/>
  <display value="Diabetes Screening"/>
  <definition value="The definition is related to screening the patient for diabetes."/>
</concept>
<concept>
  <code value="mammography-screening"/>
  <display value="Mammography Screening"/>
  <definition value="The definition is related to mammography screening of the patient."/>
</concept>
<concept>
  <code value="colonoscopy-screening"/>
  <display value="Colonoscopy Screening"/>
  <definition value="The definition is related to colonoscopy screening of the patient."/>
</concept>
<concept>
  <code value="fecal-occult-blood-screening"/>
  <display value="Fecal Occult Blood Screening"/>
  <definition value="The definition is related to fecal occult blood screening of the patient."/>
</concept>
<concept>
  <code value="zika-screening"/>
  <display value="Zika Screening"/>
  <definition value="The definition is related to screening the patient for Zika virus."/>
</concept>
```

**Version update:** Bump `<version>` from `1.0.1` to `1.1.0` and update `<date>` and `<meta><lastUpdated>`.

### Step 2: Verify ValueSet Inclusion (HL7/UTG)

**File:** `input/sourceOfTruth/fhir/valueSets/ValueSet-definition-topic.xml`

No structural changes required. The ValueSet already includes all codes from the CodeSystem:

```xml
<compose>
  <include>
    <system value="http://terminology.hl7.org/CodeSystem/definition-topic"/>
  </include>
</compose>
```

Bump `<version>` and `<date>` to match the CodeSystem update.

### Step 3: Review FHIR Core ValueSet (HL7/fhir)

**File:** `source/activitydefinition/valueset-definition-topic.xml`

This FHIR Core copy of the ValueSet already includes the CodeSystem. Update the `<date>` if a synchronized release is planned.

Note: This ValueSet also includes `definition-use` via `<valueSet value="http://hl7.org/fhir/ValueSet/definition-use"/>`, which is a separate concern (StructureDefinition use codes).

### Step 4: UTG Process Compliance

Per the comment from Bryn Rhodes on this ticket:

> "The definition topic code system resides in terminology.hl7.org now, so this change will need a UTG ticket"

A corresponding UTG ticket may need to be filed (or may already exist) to authorize changes to the CodeSystem through the UTG change management process.

## Open Questions

1. **Scope of new codes:** The ticket mentions specific screening examples. Should these be added literally, or should a broader but still illustrative set of content categories be defined? The resolution says "add more specific examples of categories" — suggesting illustrative examples rather than an exhaustive list.

2. **Hierarchy vs. flat list:** Should new codes be organized hierarchically (e.g., `screening` as parent with subtypes) or remain a flat list? The existing CodeSystem is flat.

3. **UTG ticket:** Has the required UTG change ticket been filed? The comment from Bryn Rhodes (2023-03-12) specifically calls this out as a prerequisite.

4. **Conflict with FHIR-55564:** Given that `topic` elements are being removed from resources in R6, should this ticket be de-prioritized or re-scoped to focus on the `artifact-topic` extension and `useContext` use cases?

5. **CDS Hooks alignment:** The original motivation was harmonization with CDS Hooks `card.source.topic`. Is this still a goal, and if so, what specific codes does the CDS Hooks community need?

## Recommendation

**Proceed with caution.** While the resolution is clear ("add more specific examples"), the subsequent deprecation and removal of the `topic` element (FHIR-55564) significantly changes the context. The implementation should:

1. Focus on the **UTG CodeSystem** as the only required change
2. Add a conservative set of screening-related example codes (at minimum `screening` as a general category)
3. Coordinate with the CDS work group to confirm the exact code list
4. File the required **UTG change ticket** if not already done
5. Ensure the codes are positioned as useful for **`useContext`-based topic categorization** and the **`artifact-topic` extension**, not just the deprecated `topic` element
