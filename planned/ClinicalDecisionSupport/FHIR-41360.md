## 1. Summary

The original request asks to add the ability to invoke `$package` at the *server* level (i.e. `[base]/$package`) and to add an `artifact` parameter. The resolution **modifies** this as follows:

1. Allow the `$package` operation to be invoked at the **server level** (`[base]/$package`).
2. Add a `resourceType` parameter (instead of the originally proposed `artifact` parameter).
3. Document that `resourceType` can **only** be used with the `id` parameter when invoking at the server level.
4. Document parameter exclusivity rules:
   - `id` is exclusive of `url`, `version`, and `identifier`.
   - `url` (and optionally `version`) can be used together, exclusive of `resourceType`, `id`, and `identifier`.
   - `identifier` (and optionally `version`) can be used together, exclusive of `resourceType`, `id`, and `identifier`.

## 3. Current State

The `$package` operation is defined in FSH at:
- **`input/fsh/operation-definitions/package-operation.fsh`**

Current invocation-level settings:
```fsh
* system = false    # ← NOT available at server level
* type = true       # available at type level, e.g. [base]/Library/$package
* instance = true   # available at instance level, e.g. [base]/Library/123/$package
```

Current `id` parameter:
```fsh
* parameter[+]
  * name = #id
  * min = 0
  * max = "1"
  * use = #in
  * type = #string
  * documentation = "The logical id of an existing Resource to package on the server."
```

There is **no** `resourceType` parameter today. The `url`, `version`, and `identifier` parameters exist but have no exclusivity documentation.

The operation is published at: `http://hl7.org/fhir/uv/crmi/OperationDefinition/crmi-package`

## 4. Feature Proposal

### 4.1 Enable Server-Level Invocation

Change `system` from `false` to `true` so the operation can be invoked as:

```
POST [base]/$package
```

This enables a client to package **any** resource type without needing to know the type endpoint up front, by providing `resourceType` + `id` (or `url`/`identifier`) as parameters.

### 4.2 Add `resourceType` Parameter

Add a new input parameter `resourceType` to identify the FHIR resource type when invoking at the server level:

```
resourceType  0..1  code
```

### 4.3 Document Parameter Exclusivity Rules

Add documentation clarifying which parameter combinations are valid:

| Invocation Level | Valid Combinations |
|------------------|--------------------|
| **Server** `[base]/$package` | `resourceType` + `id`; or `url` (+ optional `version`); or `identifier` (+ optional `version`) |
| **Type** `[base]/[Type]/$package` | `id`; or `url` (+ optional `version`); or `identifier` (+ optional `version`) |
| **Instance** `[base]/[Type]/[id]/$package` | No identification parameters needed (resource is identified by the URL) |

Exclusivity constraints:
- `id` is **exclusive of** `url`, `version`, and `identifier`
- `url` (+ optional `version`) is **exclusive of** `resourceType`, `id`, and `identifier`
- `identifier` (+ optional `version`) is **exclusive of** `resourceType`, `id`, and `url`
- `resourceType` can **only** be used with the `id` parameter when invoking at the server level

## 6. Detailed Implementation Plan

### Step 1: Update the Operation Definition FSH

**File:** `input/fsh/operation-definitions/package-operation.fsh`

**1a. Enable server-level invocation:**

Change:
```fsh
* system = false
```
To:
```fsh
* system = true
```

**1b. Add `resourceType` parameter** (insert before the existing `id` parameter):

```fsh
* parameter[+]
  * name = #resourceType
  * min = 0
  * max = "1"
  * use = #in
  * type = #code
  * documentation = """
The type of the resource to package. This parameter is used when the operation 
is invoked at the server level to indicate the type of the resource being packaged. 
The resourceType parameter can only be used with the id parameter when invoking at 
the server level. It is exclusive of the url and identifier parameters.
"""
```

**1c. Update `id` parameter documentation:**

Change:
```fsh
  * documentation = "The logical id of an existing Resource to package on the server."
```
To:
```fsh
  * documentation = """
The logical id of an existing Resource to package on the server. When used at 
the server level, the resourceType parameter must also be specified. The id 
parameter is exclusive of the url, version, and identifier parameters.
"""
```

**1d. Update `url` parameter documentation:**

Change:
```fsh
  * documentation = "A canonical or artifact reference to a Resource to package on the server."
```
To:
```fsh
  * documentation = """
A canonical or artifact reference to a Resource to package on the server. 
The url parameter (and optionally version) can be used together, and is 
exclusive of resourceType, id, and identifier parameters.
"""
```

**1e. Update `version` parameter documentation:**

Change:
```fsh
  * documentation = "The version of the Resource"
```
To:
```fsh
  * documentation = """
The version of the Resource. May be used with the url parameter or the 
identifier parameter, but not with the id or resourceType parameters alone.
"""
```

**1f. Update `identifier` parameter documentation:**

Change:
```fsh
  * documentation = "A business identifier of the Resource."
```
To:
```fsh
  * documentation = """
A business identifier of the Resource. The identifier parameter (and optionally 
version) can be used together, and is exclusive of resourceType, id, and url 
parameters.
"""
```

### Step 2: Update Documentation Pages

**2a. `input/pages/distribution.md`**

In the *"$package and $data-requirements"* section, after the existing description of `$package`, add text explaining server-level invocation:

> The `$package` operation can be invoked at three levels:
> - **Server level**: `POST [base]/$package` — requires a `resourceType` parameter with `id`, or `url`/`identifier` to identify the resource.
> - **Type level**: `POST [base]/[ResourceType]/$package` — uses `id`, `url`, or `identifier` to identify the resource within the type.
> - **Instance level**: `POST [base]/[ResourceType]/[id]/$package` — the resource is identified by the URL path.

**2b. `input/pages/operations.md`**

Update the description of the CRMI Package operation in the operations table to mention server-level invocation:

> Packages a specified canonical resource for use in a target environment, optionally including related content such as dependencies, components, and testing cases and data. **Can be invoked at the server, type, or instance level.**

**2c. `input/pages/changes.md`**

Add a changelog entry under the current ballot heading:

```markdown
* [FHIR-41360](https://jira.hl7.org/browse/FHIR-41360): Added server-level invocation support for the $package operation, including a new `resourceType` parameter and documentation of parameter exclusivity rules
```

### Step 3: Review Capability Statements

Review the three capability statement XML files in `input/resources/` to check whether they declare the `$package` operation. If so, update them to include server-level operation support. If the capability statements don't currently list `$package` at specific interaction levels, this step may be a no-op.

1. Run the IG Publisher build to ensure the FSH compiles correctly and the generated OperationDefinition JSON/XML reflects all changes.
2. Verify the generated narrative renders the new `resourceType` parameter and updated parameter documentation.
3. Verify the changes page renders correctly.

## 8. Testing Considerations

- **Server-level invocation with `resourceType` + `id`**: Verify `POST [base]/$package` with `resourceType=Library&id=example` returns the correct package.
- **Server-level invocation with `url`**: Verify `POST [base]/$package` with `url=http://example.org/Library/example` works without `resourceType`.
- **Server-level invocation with `identifier`**: Verify `POST [base]/$package` with `identifier=urn:example:lib1` works.
- **Exclusivity enforcement**: Verify that combining `id` with `url` or `identifier` returns an error.
- **Backward compatibility**: Ensure existing type-level and instance-level invocations continue to work unchanged.
