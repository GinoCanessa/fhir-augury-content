The CDS work group accepted the request to add (or, where one already exists,

the answers (a QuestionnaireResponse) to generate further results**. Bryn

1. publish a new self-contained PlanDefinition example in `source/plandefinition/`, or

This plan covers both legs so the ticket can be closed in either form.

## Feature Proposal

### Problem Statement

The PlanDefinition resource page currently ships a number of CDS / order-set /
protocol examples (Low Suicide Risk Order Set, Chlamydia Screening, Exclusive
Breastfeeding, Opioid CDS, Chest Pain Cardiology Order Set, Zika, KDN5, etc.)
but **none of them models the canonical "questionnaire-driven decision
support" pattern**, where:

- a PlanDefinition action presents a `Questionnaire` to a user,
- the resulting `QuestionnaireResponse` is consumed by subsequent actions, and
- those actions branch / generate further outputs (recommendations,
  RequestGroup entries, MeasureReports, etc.) based on the answers.

A grep of `source/plandefinition/*.xml` for `Questionnaire` returns matches
only inside `plandefinition-example-cardiology-os.xml`, where two
`<contained><Questionnaire>` blocks (`CollectReferralReason`,
`CardiologyConsultationGoal`) are present but are **not wired into the
`action.definitionCanonical` / `action.input` / `action.condition`
machinery** — i.e., the example contains questionnaires but does not
actually drive subsequent actions from their answers. Implementers
asking "how do I model a questionnaire-driven plan?" therefore have no
worked example to reference.

### Proposed Change

Add a new self-contained PlanDefinition example that exercises the
Questionnaire / QuestionnaireResponse pattern end-to-end, and register it
in the existing example list and notes so it is rendered into the
PlanDefinition specification page.

Concretely:

1. **Create** `source/plandefinition/plandefinition-questionnaire-example.xml`
   (suggested logical id: `questionnaire-example`, suggested display
   `QuestionnaireDrivenExample`) modelled on the existing
   `plandefinition-example-cardiology-os.xml` and
   `plandefinition-options-example.xml` patterns, with at minimum:
   - One `<contained><Questionnaire id="…"/>` defining the questions to
     ask (or, alternatively, a sibling `Questionnaire/…` example in
     `source/questionnaire/` that the PlanDefinition references by
     canonical URL).
   - A first `<action>` whose `definitionCanonical` (or
     `input.type=Questionnaire` + `input.profile`) points at that
     Questionnaire — this is the "ask the user" step.
   - One or more downstream `<action>` blocks whose `<condition>`
     `expression.language="text/fhirpath"` (or `text/cql`) evaluates the
     `%questionnaireResponse` (or named input) to gate execution and/or
     whose `<dynamicValue>` populates output `RequestGroup` /
     recommendation fields from the answers.
   - A final action that produces the "result" the resolution refers to
     (e.g., a recommended Task / ServiceRequest / CarePlan).
2. **Register** the new example in
   `source/plandefinition/list-PlanDefinition-examples.xml` by appending
   a new `<entry>` block following the existing pattern (description
   extension, title extension whose value is the file stem,
   `<item><reference value="PlanDefinition/<id>"/><display value="…"/></item>`).
3. **Reference** the new example from the narrative in
   `source/plandefinition/plandefinition-notes.html` (or
   `plandefinition-introduction.html` / `plandefinition-examples-header.xml`,
   whichever currently introduces the example list) with a one-paragraph
   explanation of the questionnaire-driven pattern and a link to the new
   instance.
4. *(Alternative, only if the CQF/Argonaut artifact is preferred over a
   bespoke example.)* Skip steps 1–2 and instead add a paragraph + link
   in `plandefinition-notes.html` pointing at the external
   PlanDefinition (Argonaut Questionnaire / CQF
   `questionnaire-enable-when` style example) and explaining the
   pattern.

The committed plan below assumes leg (1)–(3) — adding a self-contained
example — because it is the form most consistent with how every other
PlanDefinition example currently ships in core, and because the
external artifact referenced by Bryn Rhodes in 2018 has not been wired
into core in the years since.

### Design Rationale

- **Consistency with existing examples.** All current PlanDefinition
  examples live as `source/plandefinition/plandefinition-<slug>.xml`
  files registered in `list-PlanDefinition-examples.xml`. Adding the
  new instance the same way avoids inventing a new mechanism.
- **Self-contained (`<contained>` Questionnaire) keeps the example
  publishable** without coordinating with the Questionnaire resource
  page or introducing a cross-resource example dependency. The
  Questionnaire dir already contains many independent examples
  (`questionnaire-example-bluebook.xml`,
  `questionnaire-example-gcs.xml`, etc.); referencing one of those by
  canonical URL is also acceptable and is left as an editorial choice.
- **`%questionnaireResponse` / FHIRPath in `action.condition` and
  `action.dynamicValue`** is the documented mechanism for "generating
  results based on the results" in current core; using it makes the
  example normative-pattern-shaped rather than novel.
- **Argonaut/CQF reference** (the alternative leg) was the original
  resolution intent in 2018 but was never followed through; checking it
  in as a comment-only pointer would still satisfy the resolution text
  ("…or point to one if available") with minimal risk.

## Implementation Plan

### Prerequisites

- None. All referenced templates (`plandefinition-example-cardiology-os.xml`,
  `plandefinition-options-example.xml`, the Questionnaire examples in
  `source/questionnaire/`, the action-* CodeSystems, and
  `list-PlanDefinition-examples.xml`) are already present in the
  `db79dcd` clone of HL7/fhir master.
- Editorial decision: **self-contained new example** (this plan) vs.
  **pointer to external CQF/Argonaut artifact** (alternative). Confirm
  with the CDS WG owner (Bryn Rhodes) before authoring; the plan below
  assumes the self-contained variant.

### Step-by-Step Tasks

#### HL7/fhir

1. **Author the new PlanDefinition example instance**
   - File: `source/plandefinition/plandefinition-questionnaire-example.xml` *(new)*
   - Action: Create a new FHIR XML PlanDefinition using
     `plandefinition-example-cardiology-os.xml` as a structural template
     (root `<PlanDefinition xmlns="http://hl7.org/fhir">`,
     `<id value="questionnaire-example"/>`, `<url>`, `<name>`,
     `<title>`, `<status value="draft"/>`, `<experimental value="true"/>`,
     `<date>`, `<publisher>`, `<description>` etc.).
   - Details:
     - Add a `<contained><Questionnaire id="ScreeningQuestionnaire">…</Questionnaire></contained>`
       block (or use a sibling `Questionnaire` example in
       `source/questionnaire/` and reference it by canonical URL — pick
       one and stick with it). Include 2–4 simple `<item>` entries with
       `linkId`, `text`, `type`, `required`.
     - First `<action>`: `<title>Collect screening answers</title>`,
       `<definitionCanonical value="#ScreeningQuestionnaire"/>` (or the
       chosen canonical). This is the "ask the user" step that yields
       a `QuestionnaireResponse`.
     - Second `<action>`: `<title>Evaluate response</title>` with
       `<condition><kind value="applicability"/><expression><language value="text/fhirpath"/><expression value="%questionnaireResponse.item.where(linkId='1').answer.value = true"/></expression></condition>`
       (exact FHIRPath to match the contained Questionnaire's linkIds /
       answer types).
     - Third `<action>`: produces the "result" — e.g.,
       `<definitionCanonical value="ActivityDefinition/…"/>` or an
       inline `<dynamicValue>` populating a `RequestGroup` / Task. Use
       `dynamicValue.expression` to pull values out of the
       QuestionnaireResponse.
     - If `action.code` is used, draw codings from the already-published
       `action-code` CodeSystem at
       `source/plandefinition/codesystem-action-code.xml`.
     - Validate against `structuredefinition-PlanDefinition.xml` (in
       the same folder) — only the `<differential>` is hand-edited there;
       do not touch its `<snapshot>`.

2. **Register the new example**
   - File: `source/plandefinition/list-PlanDefinition-examples.xml`
   - Action: Append a new `<entry>` immediately before the closing
     `</List>`, mirroring the existing entry shape:
     ```xml
     <entry>
       <extension url="http://hl7.org/fhir/build/StructureDefinition/description">
         <valueString value="Questionnaire-driven Plan Definition example"/>
       </extension>
       <extension url="http://hl7.org/fhir/build/StructureDefinition/title">
         <valueString value="plandefinition-questionnaire-example"/>
       </extension>
       <item>
         <reference value="PlanDefinition/questionnaire-example"/>
         <display value="QuestionnaireDrivenExample"/>
       </item>
     </entry>
     ```
   - Details: The `title` extension's value must equal the new file's
     stem (`plandefinition-questionnaire-example`). The
     `reference` value must equal `PlanDefinition/<id>` from task 1.

3. **Add a narrative pointer in the resource notes**
   - File: `source/plandefinition/plandefinition-notes.html`
     *(verify; if the existing examples are introduced from
     `plandefinition-introduction.html` or
     `plandefinition-examples-header.xml`, edit that page instead so the
     pointer renders next to the example list).*
   - Action: Add a short paragraph (1–3 sentences) describing the
     questionnaire-driven decision-support pattern and linking to the
     new example:
     `<p>The <a href="plandefinition-questionnaire-example.html">QuestionnaireDrivenExample</a> illustrates how a PlanDefinition can present a Questionnaire to the user and use the resulting QuestionnaireResponse — via FHIRPath expressions in <code>action.condition</code> and <code>action.dynamicValue</code> — to drive subsequent actions and produce recommendations.</p>`
   - Details: Hand-edit only — this is bespoke-publisher HTML, not
     SUSHI/IG-publisher input. Do not introduce any new pages; the
     publisher renders one HTML page per example automatically from
     the registered list entry.

4. **(Optional, alternative leg of the resolution) Add the pointer to
   the external CQF/Argonaut artifact**
   - File: `source/plandefinition/plandefinition-notes.html`
   - Action: If the WG prefers "point to one if available" over a new
     in-core example, replace task 3's paragraph with one that links to
     the external (CQF / Argonaut) PlanDefinition and skip tasks 1–2.
   - Details: Confirm the canonical URL with the WG before checking in.
     This leg is only listed for completeness; the primary plan
     remains tasks 1–3.

### Validation Checklist

- [ ] New PlanDefinition instance validates against
      `source/plandefinition/structuredefinition-PlanDefinition.xml`
      (no validator errors / warnings about cardinality, required
      elements, or invariant failures).
- [ ] Element definitions in the contained Questionnaire have valid
      `linkId`, `type`, and `required` values.
- [ ] `action.condition.expression.language` is one of the allowed
      values (`text/fhirpath`, `text/cql`, …) and the expression
      parses.
- [ ] `action.definitionCanonical` resolves (either to the contained
      Questionnaire via `#fragment` or to a sibling canonical URL).
- [ ] `list-PlanDefinition-examples.xml` validates as a FHIR `List`
      and the new `<entry>` mirrors existing `<entry>` shape exactly.
- [ ] Bespoke HL7 publisher build (`./build.sh` / `publish.bat`) runs
      to completion and produces a rendered HTML page for the new
      example, linked from `plandefinition-examples.html`.
- [ ] Narrative pointer in the notes HTML renders without broken
      links (`plandefinition-questionnaire-example.html` exists in
      publisher output).
- [ ] No regressions in existing PlanDefinition examples or in the
      PlanDefinition resource page.

### Testing Considerations

- Re-publish the `source/plandefinition/` folder and confirm the new
  example appears in `plandefinition-examples.html` with description
  "Questionnaire-driven Plan Definition example".
- Validate the new instance with a FHIR validator (Java validator or
  FHIR Shorthand-independent validator) against R5 / current core.
- Spot-check the FHIRPath expressions used in `action.condition` /
  `action.dynamicValue` against a synthetic QuestionnaireResponse to
  ensure they evaluate as intended (a quick `fhirpath` CLI run is
  enough; no committed tests are needed for example content).
- No interoperability or backward-compatibility test is required —
  this is purely additive content.

### Open Questions

- **Editorial choice (in-core vs. external pointer).** The 2018-08-16
  comment from Bryn Rhodes promised to "reference [the Argonaut query
  project example] once it is available." Confirm with the CDS WG
  whether to (a) author a new in-core example as planned (tasks 1–3),
  or (b) commit only a pointer paragraph to the existing CQF / Argonaut
  artifact (task 4).
- **Contained Questionnaire vs. sibling canonical reference.** Either
  pattern is valid; pick one for the example. A contained Questionnaire
  keeps the example self-describing and avoids cross-resource
  publishing dependencies.
- **Output result resource.** Choose between RequestGroup, Task,
  ServiceRequest, or CarePlan as the "result" generated from the
  answers. RequestGroup is the most common pattern in existing
  PlanDefinition examples (see `plandefinition-opioidcds-*.xml`) and
  is the recommended default.
- **Narrative location.** Verify whether `plandefinition-notes.html`,
  `plandefinition-introduction.html`, or
  `plandefinition-examples-header.xml` is currently the page that
  introduces the example list, and place the new paragraph there
  before publishing.
- **Publisher gotcha (from briefing).** Edits go directly into
  `source/<resource>/` XML and HTML — there is no SUSHI / IG-publisher
  input pipeline to update; do not look for `sushi-config.yaml`,
  `ig.ini`, `input/`, `fsh-generated/`, or `output/`.
