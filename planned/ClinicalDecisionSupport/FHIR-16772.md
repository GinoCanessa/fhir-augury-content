### Description

The submitter requests that an example PlanDefinition be added that:
1. Refers to a Questionnaire (via `action.definitionCanonical`)
2. Generates clinical results (e.g., Observations, RiskAssessments) based on the QuestionnaireResponse

### Resolution Description

> Agreed, will add such an example, or point to one if available.

### Key Comment

Bryn Rhodes (2018-08-16): "Such an example is being developed as part of the Argonaut query project, we will reference that once it is available."

**Note:** The Argonaut query example was never added to the FHIR Core specification. No existing example in the repository demonstrates the full flow of questionnaire → result generation.

---

## Feature Proposal

### What This Ticket Requires

Add a **new PlanDefinition example** to the FHIR Core specification that demonstrates the complete workflow of:

1. **Defining a Questionnaire** — either contained or referenced — that collects structured clinical data from a user
2. **Referencing the Questionnaire** from a PlanDefinition action using `definitionCanonical`
3. **Processing the QuestionnaireResponse** — using CQL expressions or FHIRPath to extract answers
4. **Generating clinical resources** — creating Observations, RiskAssessments, or other result resources based on the questionnaire answers using `dynamicValue` elements

This fills a gap in the existing examples. While the cardiology order set example (`plandefinition-example-cardiology-os.xml`) contains Questionnaires referenced via `definitionCanonical`, it does not show how questionnaire results drive the generation of downstream clinical resources.

### Proposed Example Scenario

**PHQ-9 Depression Screening Assessment** — A clinically realistic scenario where:
- A PHQ-9 questionnaire is administered to screen for depression
- The total score is computed from the questionnaire responses
- Based on the score, an Observation (depression screening result) is generated
- A conditional action generates a ServiceRequest for a mental health referral if the score exceeds a threshold

This scenario is well-established in clinical practice, easy to understand, and exercises all the key features requested.

## Implementation Plan

### Step 1: Create the New Example File

**File:** `source/plandefinition/plandefinition-example-questionnaire-response.xml`

Create a new PlanDefinition example XML file in the `HL7/fhir` repository under `source/plandefinition/`. The example should follow the established pattern of existing examples (e.g., `plandefinition-example-cardiology-os.xml`, `plandefinition-options-example.xml`).

#### Structure of the Example

```xml
<PlanDefinition xmlns="http://hl7.org/fhir">
  <id value="example-questionnaire-response"/>
  <!-- Contained resources: Questionnaire, Library (CQL logic) -->
  <contained>
    <Questionnaire>
      <!-- PHQ-9 Depression Screening Questionnaire items -->
    </Questionnaire>
  </contained>
  <contained>
    <Library>
      <!-- CQL logic to score the questionnaire and determine actions -->
    </Library>
  </contained>
  
  <!-- Metadata -->
  <url value="http://hl7.org/fhir/PlanDefinition/example-questionnaire-response"/>
  <name value="QuestionnaireResponseExample"/>
  <title value="Questionnaire Response-Based PlanDefinition Example"/>
  <type>
    <coding>
      <system value="http://terminology.hl7.org/CodeSystem/plan-definition-type"/>
      <code value="eca-rule"/>
    </coding>
  </type>
  <status value="draft"/>
  <description value="Example PlanDefinition demonstrating how to use a Questionnaire
    to collect data and generate clinical resources based on the responses."/>
  <library value="#phq9-logic"/>
  
  <!-- Action 1: Administer the Questionnaire -->
  <action>
    <title value="Administer PHQ-9 Depression Screening"/>
    <definitionCanonical value="#phq9-questionnaire"/>
  </action>
  
  <!-- Action 2: Generate Observation from score (conditional on response) -->
  <action>
    <title value="Record Depression Screening Result"/>
    <condition>
      <kind value="applicability"/>
      <expression>
        <language value="text/cql-expression"/>
        <expression value="HasCompletedQuestionnaire"/>
      </expression>
    </condition>
    <definitionCanonical value="http://hl7.org/fhir/ActivityDefinition/observation"/>
    <dynamicValue>
      <path value="code"/>
      <expression>
        <language value="text/cql-expression"/>
        <expression value="PHQ9ScreeningCode"/>
      </expression>
    </dynamicValue>
    <dynamicValue>
      <path value="value"/>
      <expression>
        <language value="text/cql-expression"/>
        <expression value="PHQ9TotalScore"/>
      </expression>
    </dynamicValue>
    <dynamicValue>
      <path value="interpretation"/>
      <expression>
        <language value="text/cql-expression"/>
        <expression value="PHQ9Interpretation"/>
      </expression>
    </dynamicValue>
  </action>
  
  <!-- Action 3: Conditional referral based on score threshold -->
  <action>
    <title value="Refer to Mental Health if Indicated"/>
    <condition>
      <kind value="applicability"/>
      <expression>
        <language value="text/cql-expression"/>
        <expression value="ShouldReferToMentalHealth"/>
      </expression>
    </condition>
    <definitionCanonical value="http://hl7.org/fhir/ActivityDefinition/referral"/>
    <dynamicValue>
      <path value="reasonCode"/>
      <expression>
        <language value="text/cql-expression"/>
        <expression value="ReferralReason"/>
      </expression>
    </dynamicValue>
  </action>
</PlanDefinition>
```

#### Key Design Points

1. **Contained Questionnaire**: The PHQ-9 questionnaire is contained within the PlanDefinition, following the pattern established in `plandefinition-example-cardiology-os.xml` (lines 493–515).

2. **`definitionCanonical` reference**: The first action references the contained Questionnaire using `definitionCanonical value="#phq9-questionnaire"`, which is the established pattern (see cardiology example lines 836, 840).

3. **CQL-based dynamic values**: Subsequent actions use CQL expressions (via a contained Library) to:
   - Compute the PHQ-9 total score from the QuestionnaireResponse
   - Map the score to an Observation with appropriate LOINC codes
   - Determine whether a referral threshold is met

4. **Conditional actions**: The `condition` elements with `applicability` kind ensure actions only fire when appropriate (e.g., questionnaire is completed, score exceeds threshold).

5. **Result generation**: The `dynamicValue` elements on the second action demonstrate how to populate an Observation resource with values derived from questionnaire answers — this is the core capability the ticket requests.

### Step 2: Register the Example in the Examples List

**File:** `source/plandefinition/list-PlanDefinition-examples.xml`

Add a new `<entry>` element to register the example. Insert it after the existing entries (before the closing `</List>` tag):

```xml
<entry>
  <extension url="http://hl7.org/fhir/build/StructureDefinition/description">
    <valueString value="Questionnaire Response-Based PlanDefinition Example"/>
  </extension>
  <extension url="http://hl7.org/fhir/build/StructureDefinition/title">
    <valueString value="plandefinition-example-questionnaire-response"/>
  </extension>
  <item>
    <reference value="PlanDefinition/example-questionnaire-response"/>
    <display value="QuestionnaireResponseExample"/>
  </item>
</entry>
```

## Existing Patterns and Precedents

| Example | Pattern Used | Relevant Aspect |
|---------|-------------|-----------------|
| `plandefinition-example-cardiology-os.xml` | Contained Questionnaires + `definitionCanonical` | Shows Questionnaire references from actions (lines 493–515, 836–841), but does NOT show result generation |
| `plandefinition-example.xml` (Low Suicide Risk) | ActivityDefinition + dynamicValue | Shows generating request resources via CQL expressions |
| `plandefinition-example-episode-of-care.xml` | dynamicValue with CQL | Shows CQL-driven dynamic value population |
| `plandefinition-options-example.xml` | Contained ActivityDefinitions | Shows contained resources referenced by actions |
| `plandefinition-chlamydia-screening-intervention.xml` | ECA rule with conditions | Shows conditional action execution |

The new example should combine patterns from these existing examples: contained Questionnaire (cardiology), CQL-driven dynamic values (episode-of-care, cardiology), and conditional actions (chlamydia screening).

## Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Example is too complex or unrealistic | Low | PHQ-9 is a well-known, standardized screening tool with clear scoring rules |
| CQL expressions in contained Library may not validate in all toolchains | Medium | Keep CQL minimal and well-documented; consider including pseudocode comments |

| Example may become outdated as PlanDefinition evolves | Low | Use stable, established patterns already present in other examples |

## Summary

This is a **straightforward, additive change** requiring:
- **1 new file**: A PlanDefinition example XML demonstrating questionnaire-driven result generation
- **1 edit**: Add the example to the examples list
- **1 optional edit**: Update the examples header

No breaking changes, no structural modifications, no new extensions needed. All required resource capabilities (`definitionCanonical` → Questionnaire, `dynamicValue`, `condition`) are already part of the PlanDefinition StructureDefinition.
