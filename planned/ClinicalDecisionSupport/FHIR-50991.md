## Description

To support specifying parameter constraints, allow the targetConstraint extension to be used on the Library.parameter element.

## Resolution Description

> Agreed, extend the context of use for the cqf-targetConstraint extension to include ParameterDefinition, so that it can be used to provide a constraint on the allowable values for the given parameter.

## Feature Proposal

Add `ParameterDefinition` as a new context of use to the `targetConstraint` extension (`http://hl7.org/fhir/StructureDefinition/targetConstraint`). Because `Library.parameter` is of type `ParameterDefinition`, adding the datatype as a context enables the extension to be applied on any `ParameterDefinition` element — including `Library.parameter` — to express constraints on allowable parameter values.

This follows the same pattern used by [FHIR-50962](https://jira.hl7.org/browse/FHIR-50962), which added `ParameterDefinition` as a context to the `cqf-parameterDefinition` extension.

**ParameterDefinition** is a FHIR metadata type (complex datatype) used in:
- `Library.parameter` — parameter definitions for CQL libraries
- `OperationDefinition.parameter` (indirectly, via BackboneElement)
- `TriggerDefinition.data` (via DataRequirement)

The resolution specifically targets ParameterDefinition (the datatype) rather than `Library.parameter` (a specific element path). This is the broader, more reusable approach — it permits the extension wherever ParameterDefinition appears, not just on Library.

**Precedent — FHIR-50962**: The `cqf-parameterDefinition` extension (lines 41–48 of `StructureDefinition-cqf-parameterDefinition.xml`) already has `ParameterDefinition` as a context with a `contextInvariant` restricting usage to ParameterDefinition instances with type `BackboneElement` or `Extension`. The `targetConstraint` resolution does **not** call for a similar contextInvariant, since parameter constraints are meaningful on parameters of any type.

## Implementation Plan

### Step 1: Add `ParameterDefinition` Context to `targetConstraint`

**File**: `input/definitions/Element/StructureDefinition-targetConstraint.xml`

Add a new `<context>` block for `ParameterDefinition` after the existing Questionnaire contexts (after line 102, before line 103 `<type value="Extension"/>`):

```xml
  <context>
    <type value="element"/>
    <expression value="ParameterDefinition"/>
  </context>
```

This should be inserted after the last existing `<context>` block (the `Questionnaire.item` context ending at line 102) and before the `<type value="Extension"/>` element at line 103.

**Resulting context list** (16 contexts total — the 15 existing ones plus `ParameterDefinition`):
- ActivityDefinition, ActivityDefinition.participant, ActivityDefinition.dosage
- Measure, Measure.group, Measure.group.population, Measure.group.stratifier, Measure.supplementalData
- PlanDefinition, PlanDefinition.goal, PlanDefinition.goal.target, PlanDefinition.actor, PlanDefinition.action
- Questionnaire, Questionnaire.item
- **ParameterDefinition** ← new

### Step 2: Consider Updating the Description (Optional)

The current `<description>` (line 28) reads:

> Specifies a constraint that is enforced on instantiated (or target) resources. This extension can be applied to definitional resources (such as ActivityDefinition or Measure)...

Since `ParameterDefinition` is a datatype (not a definitional resource), consider appending a clause:

```
...or to parameter definitions (ParameterDefinition) to constrain allowable parameter values.
```

This is an editorial improvement, not strictly required by the resolution.

### Step 3: Validate

1. Run the FHIR IG Publisher on the `fhir-extensions` project to confirm:
   - The StructureDefinition is valid
   - The new context appears in the generated extension page
   - The extension appears under the `ParameterDefinition` extensions list in `extensions-metadatatypes.html`
2. Verify no `contextInvariant` is needed — unlike `cqf-parameterDefinition` (which restricts to `BackboneElement`/`Extension` types), `targetConstraint` should be usable on ParameterDefinition of any type

## Risks and Considerations

1. **Minimal risk change** — This is a straightforward context-of-use addition with no structural changes to the extension itself. The same pattern was already applied for `cqf-parameterDefinition` in FHIR-50962.
2. **No contextInvariant needed** — The resolution does not restrict which ParameterDefinition types can use the extension. Parameter constraints (e.g., "Measurement Period must be exactly one year") are meaningful for any parameter type, so no restricting invariant is required.
3. **Broad applicability** — Adding context on `ParameterDefinition` (the datatype) rather than `Library.parameter` (a specific element) means the extension is available wherever ParameterDefinition is used. This is intentional per the resolution and consistent with the original request's use case.
4. **Coordination with FHIR-48683** — That ticket (already Published) calls for examples of parameter constraints using targetConstraint. Once this context is added, those examples can be authored in the "Using CQL With FHIR" IG.
5. **IG Publisher auto-generation** — The `extensions-metadatatypes.xml` page uses `{% include extension-list-ParameterDefinition.xhtml %}` which is auto-generated by the IG Publisher based on extension contexts. No manual documentation update is needed.
