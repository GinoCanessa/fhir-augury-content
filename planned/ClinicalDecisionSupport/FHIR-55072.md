## Feature Proposal

### Problem Statement

`RequestOrchestration` (and its companion `PlanDefinition`) currently bind three CDS-owned coded elements to ValueSets/CodeSystems whose codes are illustrative rather than normative:

- `RequestOrchestration.code` → `http://hl7.org/fhir/ValueSet/action-code` (CodeSystem `http://hl7.org/fhir/action-code`, name `ActionCode`, title "Action Code").
- `RequestOrchestration.reason` → `http://hl7.org/fhir/ValueSet/action-reason-code` (CodeSystem `http://hl7.org/fhir/action-reason-code`, name `ActionReasonCode`, title "Action Reason Code").
- `RequestOrchestration.action.participant.function` → `http://hl7.org/fhir/ValueSet/action-participant-function` (CodeSystem `http://hl7.org/fhir/action-participant-function`, name `ActionParticipantFunction`, title "Action Participant Function").

All six artifacts currently declare `<status value="active"/>` and `<experimental value="false"/>`. Implementers reading the spec can mistakenly assume these codes are formally curated FHIR vocabulary, when they are in fact placeholder examples that the WG considered migrating to THO (Terminologies.HL7.org). The ticket discussion concluded that THO migration is not required, but the artifacts' nature as examples must be made explicit so that downstream consumers do not treat them as a stable terminology contract.

### Proposed Change

For each of the six artifacts (3 ValueSets + 3 CodeSystems) under `source/plandefinition/`, perform the following coordinated edits:

1. **Mark experimental.** Set `<experimental value="true"/>` (currently `false`).
2. **Re-id with `example` token.** Change `<id value="action-code"/>` to `<id value="action-code-example"/>` (and analogous for the other five). Keeping the `example` suffix matches existing precedent in the FHIR Core spec (e.g., `valueset-example.xml`, `codesystem-example.xml`, `example-characteristic-codes`).
3. **Re-URL.** Change `<url value="http://hl7.org/fhir/ValueSet/action-code"/>` to `<url value="http://hl7.org/fhir/ValueSet/action-code-example"/>` (and analogous for the CodeSystem URLs `http://hl7.org/fhir/action-code` → `http://hl7.org/fhir/action-code-example`). The `<compose><include><system>` reference in each ValueSet must be updated to the new CodeSystem URL.
4. **Re-name.** Change `<name value="ActionCode"/>` to `<name value="ActionCodeExample"/>` (PascalCase, append `Example`). Apply the same pattern to `ActionReasonCode` → `ActionReasonCodeExample` and `ActionParticipantFunction` → `ActionParticipantFunctionExample`.
5. **Re-title.** Change `<title value="Action Code"/>` to `<title value="Action Code Example"/>` (and analogous for the other two pairs).
6. **Update bindings.** In the StructureDefinitions for `RequestOrchestration` and `PlanDefinition`, repoint each `<binding><valueSet>` element to the new ValueSet URL (`…/action-code-example`, `…/action-reason-code-example`, `…/action-participant-function-example`). Binding strength (`example`) is unchanged.
7. **Rename the source files** from `valueset-action-code.xml` → `valueset-action-code-example.xml` (and the five siblings). The filename matches the artifact `id` by HL7 publisher convention; failing to rename will break publisher-generated page slugs (`valueset-action-code-example.html`).

Example, `source/plandefinition/valueset-action-code.xml` lines 4, 14, 19–20, 22 (current → proposed):

```xml
<id value="action-code"/>                 →  <id value="action-code-example"/>
<url value="http://hl7.org/fhir/ValueSet/action-code"/>
                                          →  <url value="http://hl7.org/fhir/ValueSet/action-code-example"/>
<name value="ActionCode"/>                →  <name value="ActionCodeExample"/>
<title value="Action Code"/>              →  <title value="Action Code Example"/>
<experimental value="false"/>             →  <experimental value="true"/>
<!-- inside <compose><include> -->
<system value="http://hl7.org/fhir/action-code"/>
                                          →  <system value="http://hl7.org/fhir/action-code-example"/>
```

The OID `<identifier>` blocks (e.g., `urn:oid:2.16.840.1.113883.4.642.3.3051`) are intrinsic identifiers and **must not** change; they continue to identify the same logical artifact across versions.

### Design Rationale

- The resolution explicitly approves the "Example" relabel rather than a THO migration, because (a) the WG is comfortable that these codes can churn without coordinating with V2/CDA, and (b) the artifacts are already informative (`<structuredefinition-standards-status>` = `informative`, `fmm` = 1 or 3). The relabel makes the example-only nature visible in every conformance tool that reads `experimental`, `name`, or `url`.
- Appending `Example` (rather than prefixing) preserves alphabetic grouping in the ValueSet index (`action-code`, `action-code-example` sort adjacent) and matches existing FHIR Core precedent (`example-characteristic-codes`, `valueset-example`, etc.).
- Binding strength is intentionally left unchanged. Resolution wording is silent on strength; the existing strength on these elements is `example`, which already aligns with the relabel and needs no further action.
- The `<identifier>` OID is not changed because OIDs are global identifiers for the logical concept; the ValueSet is being renamed but remains the same set of approved illustrative codes.

### Breaking Changes

This change **is breaking** for downstream consumers that reference the artifacts by their canonical URL or id:

- **Canonical URL change.** `http://hl7.org/fhir/ValueSet/action-code`, `…/action-reason-code`, and `…/action-participant-function` (plus the three CodeSystem URLs) are retired in favour of `…-example` URLs. Any IG, profile, terminology server, or implementation that resolves these URLs by string will need to be updated. Per FHIR's normal handling, the previous URLs should be considered withdrawn for FHIR R6+, not aliased.
- **Id / filename change.** The published page filenames (`valueset-action-code.html` etc.) move to `…-example.html`. External pages linking to the old URLs (including the resolution's own `https://build.fhir.org/valueset-action-code.html` links) will 404 unless the publisher team adds redirects.
- **Binding URL change in StructureDefinitions.** Profiles that derive from `RequestOrchestration` or `PlanDefinition` and inherit the binding will continue to validate (they pick up the new `valueSet`), but tools that snapshot the old binding URL will see a delta on regeneration.
- **Name change.** Code-generation tooling that emits class/enum names from `ValueSet.name` (e.g., `ActionCode` → `ActionCodeExample`) will produce different identifiers. This is breaking for clients depending on the previous generated names.

`experimental` flipping `false` → `true` is itself a metadata change but is not in the FHIR breaking-change classification.
