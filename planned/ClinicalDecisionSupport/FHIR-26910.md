## Feature Proposal

Create a new `cpg-unknownBehavior` extension that specifies expected behavior when there is insufficient information to evaluate a given expression, data requirement, or parameter definition. This addresses a gap in the Clinical Practice Guidelines IG where guideline authors currently have no standard way to indicate what systems should do when encountering unknown/missing data during clinical reasoning evaluation.

The implementation involves three new FHIR conformance artifacts:

1. **CodeSystem** ŌĆō `cpg-unknown-behavior-cs` with four codes: `ignore`, `warning`, `prompt`, `require`
2. **ValueSet** ŌĆō `cpg-unknown-behavior-vs` including all codes from the CodeSystem
3. **Extension** ŌĆō `cpg-unknownBehavior` of type `code` (0..1), applicable to `Expression`, `DataRequirement`, and `ParameterDefinition`, with a required binding to the ValueSet

Additionally, narrative guidance must be added to the IG describing each behavior option, with special emphasis on the `require` option's implications for clinical workflows.

## Detailed Implementation Plan

### Step 1: Create CodeSystem

**File**: `input/fsh/terminology/codesystem-cpg-unknown-behavior.fsh` (new)

```fsh
CodeSystem: CPGUnknownBehaviorCS
Id: cpg-unknown-behavior-cs
Title: "CPG Unknown Behavior Code System"
Description: "Specifies expected behavior when there is not enough information present to evaluate a given expression, data requirement, or parameter definition."
* insert CodeSystemMetadata(cpg-unknown-behavior)
* #ignore "Ignore" "The default evaluation path is followed (e.g. assume false in a population or eligibility criteria)"
* #warning "Warning" "The default evaluation path is followed but a warning is surfaced"
* #prompt "Prompt" "User interaction is required to provide the data"
* #require "Require" "An error is raised, the default evaluation path is interrupted"
```

**Notes**:
- The `CodeSystemMetadata` ruleset (from `rules.fsh`) sets the canonical URL to `http://hl7.org/fhir/uv/cpg/CodeSystem/cpg-unknown-behavior-cs`, adds knowledge artifact extensions, and links to the corresponding ValueSet.
- Case-sensitive is set to `true` by the ruleset.

### Step 2: Create ValueSet

**File**: `input/fsh/terminology/valueset-cpg-unknown-behavior.fsh` (new)

```fsh
ValueSet: CPGUnknownBehaviorVS
Id: cpg-unknown-behavior-vs
Title: "CPG Unknown Behavior Value Set"
Description: "Specifies expected behavior when there is not enough information present to evaluate a given expression, data requirement, or parameter definition."
* insert ValueSetMetadata(cpg-unknown-behavior-vs)
* include codes from system CPGUnknownBehaviorCS
```

**Notes**:
- The `ValueSetMetadata` ruleset sets the canonical URL to `http://hl7.org/fhir/uv/cpg/ValueSet/cpg-unknown-behavior-vs` and adds knowledge artifact extensions.

### Step 3: Create Extension

**File**: `input/fsh/extensions/extension-cpg-unknown-behavior.fsh` (new)

```fsh
Extension: CPGUnknownBehavior
Id: cpg-unknownBehavior
Title: "CPG Unknown Behavior Extension"
Description: "Specifies expected behavior when there is not enough information present to evaluate a given expression, data requirement, or parameter definition. This allows guideline authors to indicate whether missing data should be ignored, generate a warning, prompt for user input, or raise an error."
* insert StructureDefinitionMetadata(cpg-unknownBehavior)
* insert ExtensionContext(Expression)
* insert ExtensionContext(DataRequirement)
* insert ExtensionContext(ParameterDefinition)
* value[x] only code
* valueCode from $cpg-unknown-behavior-vs (required)
  * ^binding.description = "Specifies expected behavior when data is insufficient for evaluation"
```

**Notes**:
- The `StructureDefinitionMetadata` ruleset sets the canonical URL to `http://hl7.org/fhir/uv/cpg/StructureDefinition/cpg-unknownBehavior`.
- Three `ExtensionContext` insertions register the extension for use on `Expression`, `DataRequirement`, and `ParameterDefinition` types.
- The cardinality is 0..1 (FSH default for extensions).
- The value type is `code` with a `required` binding to the new ValueSet.

### Step 4: Add Aliases

**File**: `input/fsh/aliases.fsh` (modify)

Add the following alias entries (insert alphabetically among existing `$cpg-*` aliases):

```fsh
Alias: $cpg-unknown-behavior-vs = http://hl7.org/fhir/uv/cpg/ValueSet/cpg-unknown-behavior-vs
Alias: $cpg-unknownBehavior = http://hl7.org/fhir/uv/cpg/StructureDefinition/cpg-unknownBehavior
```

### Step 5: Add Narrative Guidance

**File**: Relevant page content file under `input/pagecontent/` (identify the appropriate clinical reasoning or extensions page)

Add a section describing the `cpg-unknownBehavior` extension with:

1. **Purpose**: When guideline-based clinical reasoning encounters expressions, data requirements, or parameters with insufficient data, the `cpg-unknownBehavior` extension allows guideline authors to specify expected system behavior.

2. **Behavior descriptions** for each code value:
   - `ignore` ŌĆō System follows the default evaluation path silently (e.g., treats a missing Boolean as `false`)
   - `warning` ŌĆō System follows the default path but surfaces a warning to the user or log
   - `prompt` ŌĆō System pauses and requires user interaction to supply the missing data
   - `require` ŌĆō System raises an error and interrupts the default evaluation path

3. **Guidance on `require`**: Workflows implementing the `require` behavior must take clinical context into account. Systems should ensure that users have the ability to proceed and take appropriate action, exercising clinical judgement. The `require` option should not create a hard block that prevents clinicians from delivering care when data is unavailable ŌĆö rather it should surface the need for the data while preserving the clinician's ability to override or proceed.
