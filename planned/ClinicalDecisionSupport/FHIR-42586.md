## Description

The ticket raises three concerns about the SHA hash guidance in the CRMI specification's "Preparing Artifacts" section:

1. **Cross-syntax hashing**: Will the SHA hash work appropriately when a resource is serialized in different formats (JSON vs XML)? Whitespace and formatting differences between syntaxes mean a naïve byte-level hash will not match across representations.
2. **Invalid system value**: The specification uses `"sha256"` as though it were a valid code system, but `meta.security` is of type `Coding` and requires a proper URI for `system` plus a `code` — not just a bare value.
3. **Code registration**: The desired code must actually be registered in a real code system.

## Resolution Description

> Agreed, we will reach out to the Security Workgroup to determine the appropriate way to identify the algorithm used for computing the hash.
>
> If needed, we will approach TSMG about establishing a Naming System for the algorithm (if one is not defined already)

## Feature Proposal

Update the CRMI IG's "Preparing Artifacts" SHA hash guidance to:

1. **Reference a proper code system** for identifying the hash algorithm — the existing UTG `IntegrityCheckAlgorithm` code system (`http://terminology.hl7.org/CodeSystem/v3-IntegrityCheckAlgorithm`) already defines `SHA-1` and `SHA-256` codes, or coordinate with the Security WG and TSMG to identify/create a more comprehensive code system.
2. **Specify the canonicalization method** to use before hashing, referencing the FHIR core Signatures page's defined canonicalization methods (Canonical JSON via RFC 8785 and/or Canonical XML via C14N 1.1).
3. **Clarify the `meta.security` Coding** to use the proper `system` URI and `code` — not an ad-hoc string like `"sha256"`.
4. **Address cross-syntax concerns** by mandating that the hash is computed over a canonical serialization (ensuring consistent hashing regardless of the format used at any point in the lifecycle).

### Key Technical Context

**Existing CRMI text** (quoted in FHIR-45297):
> OPTIONAL: A SHA hash for each resource maybe generated to confirm immutability of the resource. This SHA SHOULD be based on the contents of the resource, not including the `.meta`. The value can be stored as a `meta.security` coding. The metadata about the resource **MAY** change such that it does not substantially alter or modify the resource.

**FHIR Core Canonicalization Methods** (from `signatures.html`):
- `http://hl7.org/fhir/canonicalization/json` — JSON Canonicalization Scheme per RFC 8785
- `http://hl7.org/fhir/canonicalization/json-xml` — JSON canonicalization with XHTML canonicalized per XML rules (safe across syntaxes)
- `http://hl7.org/fhir/canonicalization/xml` — XML Canonical Method per Canonical XML 1.1
- Variant `#static` — omits `Resource.meta` and Narrative, suitable for artifact immutability checks

**UTG IntegrityCheckAlgorithm** (from `v3-IntegrityCheckAlgorithm.xml`):
- System: `http://terminology.hl7.org/CodeSystem/v3-IntegrityCheckAlgorithm`
- Codes: `SHA-1` (secure hash algorithm - 1), `SHA-256` (secure hash algorithm - 256)

## Implementation Plan

### Prerequisites

Before implementing, the CDS work group should:

1. **Coordinate with the Security WG** to confirm the appropriate code system for identifying hash algorithms. The existing UTG `IntegrityCheckAlgorithm` code system is a strong candidate but only has SHA-1 and SHA-256.
2. **Coordinate with TSMG** to determine whether `IntegrityCheckAlgorithm` should be expanded, or a new code system / NamingSystem should be established (per the resolution description).
3. **Review FHIR-45297 implementation** — Since FHIR-45297 (also Persuasive, now Applied) covers the same area with a more detailed resolution (provide detailed description of digital signatures including the [crmi-signing-poc](https://github.com/reason-healthcare/crmi-signing-poc)), the two tickets should be implemented together to avoid conflicting changes.

### Step 1: Determine the Hash Algorithm Code System

**Decision needed from Security WG / TSMG**:

- **Option A**: Use the existing UTG `IntegrityCheckAlgorithm` code system (`http://terminology.hl7.org/CodeSystem/v3-IntegrityCheckAlgorithm`). This requires adding codes for any additional algorithms needed (SHA-384, SHA-512, SHA3-256, etc.).
  - File: `HL7_UTG/input/sourceOfTruth/v3/codeSystems/v3-IntegrityCheckAlgorithm.xml`
- **Option B**: Create a new NamingSystem or code system specifically for CRMI artifact hashing use cases.
- **Option C**: Reference an external code system (e.g., IANA hash algorithms registry) via a FHIR NamingSystem.

### Step 2: Update CRMI "Preparing Artifacts" SHA Hash Guidance

**Repository**: CRMI IG (not in cached repos — likely `HL7/crmi-ig`)

Replace the existing under-specified SHA hash text with guidance that:

1. **Specifies the code system** for `meta.security` Coding:
   - `system`: The URI of the chosen code system (e.g., `http://terminology.hl7.org/CodeSystem/v3-IntegrityCheckAlgorithm`)
   - `code`: The algorithm code (e.g., `SHA-256`)

2. **Specifies the canonicalization method**: The hash MUST be computed on a canonical serialization of the resource content (excluding `.meta`). Recommended approach:
   - Use the `#static` canonicalization variant (`http://hl7.org/fhir/canonicalization/{fmt}#static`) which excludes `Resource.meta` and Narrative — this aligns with the intent of hashing "the contents of the resource, not including `.meta`"
   - If cross-syntax portability is required, use `json-xml` canonicalization (JSON with XHTML canonicalized per XML rules)
   - Document explicitly which canonicalization method is used (possibly as another `meta.security` coding or as part of the hash coding's display/extension)

3. **Addresses cross-syntax concerns**: Mandate that the hash is computed over a canonical form, so the same hash is reproducible regardless of whether the resource is stored as JSON or XML. Reference the FHIR core `signatures.html` page for the canonicalization algorithms.

4. **Provides example**: Show a concrete example of a `meta.security` element with the correct system, code, and hash value.

### Step 3: Coordinate with FHIR-45297 Implementation

Since FHIR-45297 is already Applied with a resolution to provide detailed digital signature descriptions (referencing the [crmi-signing-poc](https://github.com/reason-healthcare/crmi-signing-poc)), the FHIR-42586 changes should be integrated into that same effort. Specifically:

- The hash algorithm identification (FHIR-42586) feeds into the broader signing process description (FHIR-45297)
- The canonicalization method specification (FHIR-42586) is a prerequisite for the signing process (FHIR-45297)
- Both tickets target the same CRMI IG section

### Step 4: Update UTG (if Option A)

If the Security WG confirms use of `IntegrityCheckAlgorithm`:

**File**: `cache/github/repos/HL7_UTG/clone/input/sourceOfTruth/v3/codeSystems/v3-IntegrityCheckAlgorithm.xml`

Consider adding additional codes to support the algorithms identified by the Security WG (e.g., SHA-384, SHA-512, SHA3-256). This aligns with FHIR-47198's direction of allowing policy-driven algorithm choices.

## Risks and Considerations

1. **CRMI IG source not in cached repos** — The primary changes target the CRMI IG, whose source repository is not among the cached clones. The implementation will need to locate and modify the CRMI IG repository directly.
2. **Coordination dependency** — The resolution explicitly requires consultation with the Security WG and TSMG before changes can be finalized. This is a blocking prerequisite.
3. **Overlap with FHIR-45297** — Both tickets modify the same section of the CRMI spec. They should be implemented as a single coordinated change to avoid merge conflicts and ensure consistency.
4. **Backward compatibility** — Existing implementations that use bare `"sha256"` strings in `meta.security` will need migration guidance when the proper Coding structure is specified.
5. **Canonicalization complexity** — The `#static` variant excludes `.meta` entirely, which is semantically correct for this use case but means the hash value itself (stored in `.meta.security`) is inherently excluded from the hash computation. This is the intended behavior but should be documented clearly.
