# Implementation Plan: FHIR-40097

**Title:** Clarify CDS Hooks indicator values  
**Status:** Resolved - change required  
**Resolution:** Persuasive  
**Work Group:** Clinical Decision Support  
**Specification:** Potential Drug-Drug Interaction (PDDI) (FHIR)  
**Resolved:** 2025-04-02  

---

## Resolution Summary

The PDDI CDS Implementation Guide defines its own local `PDDI-CDS-indicator` CodeSystem with values `{info, warning, hard-stop}`. This conflicts with the CDS Hooks specification and the HL7 Terminology (THO) authoritative CodeSystem (`http://terminology.hl7.org/CodeSystem/cdshooks-indicator`), which defines `{info, warning, critical}`. The inconsistency is compounded by the fact that the PDDI IG examples already use `critical` rather than `hard-stop`.

**Resolution:** Remove the local `PDDI-CDS-indicator` CodeSystem and its accompanying ValueSet from the PDDI IG. Replace all references to the local code system and `hard-stop` code with references to the THO-published CodeSystem and ValueSet, using the canonical code `critical`.

> This is a **Persuasive** resolution — implementation is required.

---

## Feature Proposal

### Problem Statement

The PDDI CDS IG defines a local CodeSystem `PDDI-CDS-indicator` containing codes `{info, warning, hard-stop}`. This creates two problems:

1. **Terminology divergence:** The CDS Hooks specification and THO canonical CodeSystem use `critical` instead of `hard-stop`. Implementers cannot reliably map PDDI CDS cards to standard CDS Hooks indicator values.
2. **Internal inconsistency:** The PDDI IG's own narrative examples (e.g., in `cds-service-spec.html`) use `critical`, not `hard-stop`, contradicting the locally defined CodeSystem.

The THO CodeSystem for CDS Hooks indicators is now officially published at:
- **CodeSystem:** `http://terminology.hl7.org/CodeSystem/cdshooks-indicator`
- **THO page:** https://terminology.hl7.org/CodeSystem-cdshooks-indicator.html

### Proposed Change

1. **Remove** the local `PDDI-CDS-indicator` CodeSystem resource from the PDDI IG source.
2. **Remove** the local ValueSet that includes only `PDDI-CDS-indicator` codes.
3. **Update** all bindings, profiles, examples, and narrative that reference the local CodeSystem or the `hard-stop` code to instead reference:
   - CodeSystem: `http://terminology.hl7.org/CodeSystem/cdshooks-indicator`
   - ValueSet: `http://terminology.hl7.org/ValueSet/cdshooks-indicator` (or the FHIR core equivalent)
   - Code: `critical` (replacing `hard-stop`)
4. **Update** the CDS service specification narrative to consistently use `critical` throughout.

### Design Rationale

- The THO CodeSystem `cdshooks-indicator` (id: `cdshooks-indicator`, version `3.0.0`) is the authoritative source for CDS Hooks indicator terminology in HL7. It defines exactly `info`, `warning`, and `critical` — the same values required by the CDS Hooks specification.
- The old `http://cds-hooks.hl7.org/CodeSystem/indicator` URL is deprecated (status: `retired`) in the FHIR core repo (`source/terminologies/codesystem-cdshooks-indicator.xml`) with a `successor` relation pointing to the THO URL.
- Reusing THO terminology eliminates duplication, avoids maintenance of a redundant code system, and ensures interoperability with all CDS Hooks-compliant systems.

## Impact Analysis

### Affected Repositories

The PDDI CDS IG is maintained in a separate HL7 GitHub repository (not present in the local cache). Based on the PDDI specification URL `http://hl7.org/fhir/uv/pddi/`, the repository is likely:

- **`HL7/fhir-pddi-ig`** (GitHub: https://github.com/HL7/fhir-pddi-ig) — primary target for all changes

Files likely to be affected within that repository:
- `input/vocabulary/CodeSystem-PDDI-CDS-indicator.json` (or `.xml`) — **DELETE**
- `input/vocabulary/ValueSet-PDDI-CDS-indicator.json` (or `.xml`) — **DELETE** (or repurpose to reference THO)
- `input/pagecontent/cds-service-spec.md` (or `.xml`) — update narrative and examples
- `input/pagecontent/*.md` — any page referencing `hard-stop` or the local CodeSystem URL
- `input/examples/*.json` — CDS Hooks card examples using `hard-stop`
- `input/profiles/*.json` / `*.xml` — any profile binding to the local ValueSet
- `sushi-config.yaml` or `ig.ini` — may reference removed resources

**Reference repositories (read-only, no changes needed):**
- `HL7_UTG` (`cache/github/repos/HL7_UTG/clone/input/sourceOfTruth/fhir/codeSystems/CodeSystem-cdshooks-indicator.xml`) — the THO source of truth; confirms `critical` is correct
- `HL7_fhir` (`cache/github/repos/HL7_fhir/clone/source/terminologies/codesystem-cdshooks-indicator.xml`) — deprecated predecessor with `retired` status

### Breaking Changes

- **Terminology binding change:** Any PDDI IG profile that binds to the local `PDDI-CDS-indicator` ValueSet will need to be updated to bind to the THO ValueSet. This is a breaking change for implementers who coded to the old `hard-stop` value.
- **Code value change:** `hard-stop` → `critical` is a semantic code change. Existing PDDI CDS service implementations must update their response cards to use `critical`.
- **CodeSystem URL change:** From a local IG-defined URL to `http://terminology.hl7.org/CodeSystem/cdshooks-indicator`.

Mitigation: Since the IG is still under development (ballot cycle), this is an appropriate time to make breaking changes. A migration note in the change log and a version bump are recommended.

## Implementation Plan

### Prerequisites

1. Confirm the exact GitHub repository for the PDDI CDS IG (expected: `HL7/fhir-pddi-ig`).
2. Obtain a local clone of the repository.
3. Verify the THO ValueSet canonical URL for `cdshooks-indicator`. Check:
   - https://terminology.hl7.org/ValueSet-cdshooks-indicator.html
   - The UTG source: `cache/github/repos/HL7_UTG/clone/input/sourceOfTruth/fhir/codeSystems/CodeSystem-cdshooks-indicator.xml`
4. Confirm which version of the PDDI IG is being balloted (to determine the correct `ig.ini` / `sushi-config.yaml` version).

### Step-by-Step Tasks

#### Task 1 — Delete the local CodeSystem resource
- Locate and delete `input/vocabulary/CodeSystem-PDDI-CDS-indicator.{json,xml}` (or equivalent path).
- Remove it from the IG resource list in `sushi-config.yaml` or `ig.ini` if explicitly listed.

#### Task 2 — Replace or delete the local ValueSet
- Locate `input/vocabulary/ValueSet-PDDI-CDS-indicator.{json,xml}`.
- **Option A (preferred):** Delete the file entirely and update all bindings in profiles/examples to reference the THO ValueSet canonical URL.
- **Option B:** Repurpose the ValueSet to be a purely extensional binding to `http://terminology.hl7.org/CodeSystem/cdshooks-indicator`, including `info`, `warning`, and `critical`.

#### Task 3 — Update all profile bindings
- Search the IG source for any `ValueSet/PDDI-CDS-indicator` or `http://hl7.org/fhir/uv/pddi/ValueSet/PDDI-CDS-indicator` binding references.
- Replace with the THO ValueSet canonical URL or a FHIR core binding.

#### Task 4 — Replace `hard-stop` with `critical` in all examples
- Search for `"hard-stop"` or `hard-stop` across all `input/examples/` and `input/pagecontent/` files.
- Replace each occurrence with `critical` (code) and ensure the system is updated to `http://terminology.hl7.org/CodeSystem/cdshooks-indicator`.

#### Task 5 — Update narrative documentation
- In `input/pagecontent/cds-service-spec.md` (or equivalent), update:
  - Any table or list of indicator values to remove `hard-stop` and show `{info, warning, critical}`.
  - Any text comparing PDDI indicator values to CDS Hooks to reflect the unified set.
  - Ensure all code snippets in narrative use `critical`.
- Add a note or summary table pointing readers to the THO CodeSystem URL.

#### Task 6 — Update the change log / version history
- Add an entry noting the removal of the local `PDDI-CDS-indicator` code system.
- Flag `hard-stop` → `critical` as a breaking change.

#### Task 7 — Run IG Publisher and validate
- Run `_genonce.sh` (or `_genonce.bat`) to rebuild the IG.
- Confirm no broken references to deleted resources.
- Confirm all examples validate against the updated profiles.
- Confirm no QA errors related to unknown CodeSystem or ValueSet references.

### Validation Checklist

- [ ] No FHIR resource in the IG source references `PDDI-CDS-indicator` CodeSystem or ValueSet by its old local URL.
- [ ] No occurrence of code `hard-stop` in any JSON, XML, or Markdown file in the IG.
- [ ] All CDS Hooks card examples use `critical` from `http://terminology.hl7.org/CodeSystem/cdshooks-indicator`.
- [ ] Profile bindings point to THO or FHIR core `cdshooks-indicator` ValueSet.
- [ ] IG Publisher QA report shows zero errors related to the indicator code system/value set.
- [ ] Narrative in `cds-service-spec` consistently lists `{info, warning, critical}` (not `hard-stop`).
- [ ] Change log entry documents the breaking change.
- [ ] The deleted CodeSystem and ValueSet are removed from the IG's resource listing.

### Testing Considerations

- **Terminology validation:** Verify that FHIR validators can resolve `http://terminology.hl7.org/CodeSystem/cdshooks-indicator` during IG Publisher build (requires network access or a local tx.fhir.org cache entry).
- **Example conformance:** All example CDS Hooks response cards should validate with `indicator.system = http://terminology.hl7.org/CodeSystem/cdshooks-indicator` and `indicator.code` in `{info, warning, critical}`.
- **Backward compatibility note:** Document in release notes that existing PDDI CDS service implementations using `hard-stop` must migrate to `critical`.

### Open Questions

1. **ValueSet canonical URL for THO binding:** What is the exact canonical URL of the THO `cdshooks-indicator` ValueSet? (Verify at https://terminology.hl7.org/ValueSet-cdshooks-indicator.html before coding the binding.)
2. **Profile binding strength:** Should bindings be `required` or `extensible`? The CDS Hooks spec implies `required`, but confirm with the PDDI CDS WG.
3. **Existing implementers:** Has the PDDI WG notified existing PDDI CDS service developers about the `hard-stop` → `critical` change? A ballot comment or errata notice may be needed.
4. **Other PDDI-specific indicators:** Are there any PDDI-specific severity indicators beyond the CDS Hooks set (e.g., `hard-stop` was meant to convey a stronger urgency than `critical`)? If so, the WG should determine whether an extension or a PDDI-specific profile of the CDS Hooks card is more appropriate than adding a custom code.
