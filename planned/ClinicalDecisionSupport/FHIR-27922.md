## Description

Healthcare does not fit neatly into the levels that BPMN presumes for pools and lanes (Pool = organization, Lane = department/individual). Healthcare has many variable layers, and the rules around interactions between layers are inconsistent. The recommendation is to always map FHIR "actor" to BPMN "lane". If there is a desire to break lanes into pools, an extension should be used.

## Resolution Description

> Agreed, add documentation as suggested, and define two extensions:
>
> - **cpg-planDefinitionActor** – Same structure as R5 `PlanDefinition.actor`
> - **cpg-actorReference** – On `PlanDefinition.action` to specify actor
> - **cpg-locationdefinition** – Profile of Location with mode = kind
> - **cpg-locationReference** – On `PlanDefinition.action` to specify location
>
> Location could potentially map to pools, but it would place restrictions on the types of PlanDefinitions that could be mapped to BPMN.

### Comment (Bryn Rhodes, 2020-07-06)

> Also consider defining an extension to describe actors for an R4 PlanDefinition.

## Feature Proposal

### Goal

Add BPMN swimlane/pool mapping documentation to the CPG IG, and define extensions that enable R4 PlanDefinitions to express actor and location information equivalent to the R5 `PlanDefinition.actor` backbone element. This allows CPG PlanDefinitions to be mapped to BPMN diagrams where actors correspond to lanes and locations can optionally correspond to pools.

### Deliverables

1. **Documentation**: New section in the existing BPMN-to-PlanDefinition methodology page explaining how actors/participants map to BPMN lanes and how locations can optionally map to pools.
2. **Extension `cpg-planDefinitionActor`**: A complex extension on PlanDefinition that mirrors the R5 `PlanDefinition.actor` backbone element (title, description, option with type/typeCanonical/typeReference/role).
3. **Extension `cpg-actorReference`**: A simple extension on `PlanDefinition.action` that references one of the plan-level actors defined via `cpg-planDefinitionActor`.
4. **Extension `cpg-locationReference`**: A simple extension on `PlanDefinition.action` that references a CPGLocationDefinition to specify where the action takes place.
5. **Profile updates**: Update relevant CPG PlanDefinition profiles to incorporate the new extensions.

> **Note:** The `cpg-locationdefinition` profile already exists at `input/fsh/profiles/domain-profiles/cpg-locationdefinition.fsh` with `mode = #kind`, so no new Location profile is needed.

### R5 PlanDefinition.actor Structure (Reference for Extension Design)

The R5 `PlanDefinition.actor` backbone element (from `source/plandefinition/structuredefinition-PlanDefinition.xml` in `HL7/fhir`) has this structure:

```
PlanDefinition.actor (BackboneElement, 0..*)
  ├── title (string, 0..1) – User-visible title
  ├── description (markdown, 0..1) – Describes the actor
  └── option (BackboneElement, 1..*)
      ├── type (code, 0..1) – action-participant-type value set (required)
      ├── typeCanonical (canonical, 0..1) – Reference to ActorDefinition, CapabilityStatement, etc.
      ├── typeReference (Reference, 0..1) – Reference to CareTeam, Device, Location, Organization, etc.
      └── role (CodeableConcept, 0..1) – action-participant-role value set (example)
```

## Implementation Plan

### Task 1: Define `cpg-planDefinitionActor` Extension

**File:** `input/fsh/extensions/extension-cpg-plan-definition-actor.fsh` (new)

Create a complex (multi-value) extension that mirrors the R5 `PlanDefinition.actor` backbone element. This extension applies to `PlanDefinition` and allows R4-based CPG definitions to express plan-level actors.

```fsh
Extension: CPGPlanDefinitionActor
Id: cpg-planDefinitionActor
Title: "CPG Plan Definition Actor"
Description: "Defines an actor within the plan definition, mirroring the R5 PlanDefinition.actor backbone element."
* insert StructureDefinitionMetadata(cpg-planDefinitionActor)
* insert ExtensionContext(PlanDefinition)
* . 0..*
* extension contains
    title 0..1 and
    description 0..1 and
    option 1..*
* extension[title].value[x] only string
* extension[description].value[x] only markdown
* extension[option]
  * extension contains
      type 0..1 and
      typeCanonical 0..1 and
      typeReference 0..1 and
      role 0..1
  * extension[type].value[x] only code
  * extension[type].valueCode from http://hl7.org/fhir/ValueSet/action-participant-type (required)
  * extension[typeCanonical].value[x] only canonical
  * extension[typeReference].value[x] only Reference
  * extension[role].value[x] only CodeableConcept
```

**Rationale:** The resolution explicitly states "cpg-planDefinitionActor - Same structure as R5 PlanDefinition.actor". This complex extension provides all sub-elements of the R5 backbone.

### Task 2: Define `cpg-actorReference` Extension

**File:** `input/fsh/extensions/extension-cpg-actor-reference.fsh` (new)

Create a simple extension on `PlanDefinition.action` that references a plan-level actor defined via `cpg-planDefinitionActor`. The value is a string matching the title of the actor defined at the plan level (since extensions can't directly reference other extensions by index).

```fsh
Extension: CPGActorReference
Id: cpg-actorReference
Title: "CPG Actor Reference"
Description: "References an actor defined at the plan level to associate with this action, enabling BPMN lane assignment."
* insert StructureDefinitionMetadata(cpg-actorReference)
* insert ExtensionContext(PlanDefinition.action)
* . 0..*
* value[x] only string
```

**Rationale:** The resolution states "cpg-actorReference - On PlanDefinition.action to specify actor". This links actions to plan-level actors, enabling BPMN lane assignment.

### Task 3: Define `cpg-locationReference` Extension

**File:** `input/fsh/extensions/extension-cpg-location-reference.fsh` (new)

Create a simple extension on `PlanDefinition.action` that references a `CPGLocationDefinition` to specify where an action takes place.

```fsh

Id: cpg-locationReference
Title: "CPG Location Reference"
Description: "References a CPG Location Definition to specify where the action takes place, enabling optional BPMN pool assignment."

* . 0..*

```

**Rationale:** The resolution states "cpg-locationReference - On PlanDefinition.action to specify location". This allows optional mapping to BPMN pools.

### Task 6: Add BPMN Actor/Lane/Pool Documentation

**File:** `input/pagecontent/documentation-methodology.md` (modify)

Add a new section after the existing "Workflow Pattern Transformations" section. The section should cover:

#### Proposed Content Outline

```markdown
### Actors, Lanes, and Pools

BPMN uses lanes and pools to organize activities by the actors responsible for performing
them. This section describes how PlanDefinition actors and participants map to these BPMN
constructs.

#### Mapping Actors to Lanes

In BPMN, a lane represents a partition within a pool that organizes activities by role or
responsibility. In FHIR PlanDefinition, actors defined using the `cpg-planDefinitionActor`
extension (or `PlanDefinition.actor` in R5) correspond to BPMN lanes.

| BPMN | PlanDefinition |
|------|---------------|
| Lane | PlanDefinition.actor (R5) / cpg-planDefinitionActor extension (R4) |
| Lane name | actor.title |
| Lane participant | actor.option (type, typeCanonical, typeReference, role) |
| Activity in lane | PlanDefinition.action with cpg-actorReference matching actor title |

The recommendation is to **always map 'actor' to 'lane'**. Each actor defined at the plan
level produces a lane in the corresponding BPMN diagram. Actions are assigned to lanes
using the `cpg-actorReference` extension on `PlanDefinition.action`.

#### Mapping Locations to Pools (Optional)

In BPMN, a pool represents a participant in a collaboration, typically an organization or
system boundary. Healthcare does not fit neatly into the organizational levels that BPMN
presumes (Pool = organization, Lane = department/individual). In healthcare, there could
be many organizational layers, and the rules around interactions between layers are
variable.

If there is a desire to organize lanes into pools, the `cpg-locationReference` extension
on `PlanDefinition.action` can be used to reference a `CPGLocationDefinition` (a profile
of Location with `mode = kind`). Location types can then be used as pool boundaries.

| BPMN | PlanDefinition |
|------|---------------|
| Pool | CPGLocationDefinition (referenced via cpg-locationReference) |
| Pool name | Location.name |
| Pool type | Location.type |

**Note:** Using location as a pool mapping places restrictions on the types of
PlanDefinitions that can be mapped to BPMN, as not all clinical workflows have
meaningful location-based organizational boundaries.

#### Healthcare Considerations

Healthcare workflows often involve:
- Multiple organizational layers that don't map cleanly to the two-level pool/lane
  hierarchy that BPMN assumes
- Variable interaction rules between different organizational layers
- Actors that span organizational boundaries (e.g., a specialist who works across
  departments)

For these reasons, the CPG IG recommends starting with actor-to-lane mapping and only
introducing pool-based organization when the workflow has clear location-based boundaries.
```

## Risks and Considerations

1. **R4 vs R5 Compatibility**: The `cpg-planDefinitionActor` extension replicates R5 `PlanDefinition.actor` for R4 use. When the CPG IG migrates to R5, this extension would become redundant and should be deprecated in favor of the native element.
2. **Actor Reference Mechanism**: The design uses a string-based reference (actor title) to link actions to actors. An alternative would be a URI or index-based approach. The implementation should document the chosen mechanism clearly.
3. **BPMN Pool Mapping Limitations**: As noted in the resolution, mapping Location to pools restricts which PlanDefinitions can generate valid BPMN diagrams. Documentation should clearly state this is optional.
4. **Existing `participant` Element**: R4 `PlanDefinition.action.participant` already exists with type/role. The new actor extensions supplement (not replace) this element, providing plan-level actor definitions that participants can reference.
5. **Downstream Profile Impact**: Domain profiles (pathway, strategy, recommendation, workflow) that inherit from `CPGComputablePlanDefinition` will automatically inherit the new extensions. No explicit changes needed to those profiles unless specific constraints are desired.
