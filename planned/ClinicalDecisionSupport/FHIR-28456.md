## Original Request

The submitter requested that the CDS Hooks specification evaluate the optionality of key metadata elements — specifically `prefetch`, `url`, `topic`, `uuid` (for source), `suggestions`, and `actions` — with respect to requirements for retrospective analysis of CDS Hook effectiveness and efficiency. The concern is that when these elements are OPTIONAL, implementations will frequently omit them, making it impossible to aggregate data for quality improvement and learning health system goals.

## Resolution Description

> We will add a ballot/STU note that:
> We are seeking feedback on the use of topic in specific use cases.
>
> Review each optional element and provide guidance for how the element may be applicable within an example use case.

The resolution does **not** change any element from OPTIONAL to REQUIRED. Instead, it adds:
1. A **ballot/STU note** seeking community feedback on the use of `source.topic` in specific use cases
2. **Guidance text** for each relevant optional element explaining how it may be applicable, illustrated with example use cases

## Feature Proposal

### What Changes

Add non-normative guidance and a ballot note to the CDS Hooks specification to help implementers understand the value and applicability of optional metadata elements. This consists of:

1. **STU Note on `source.topic`**: Insert a ballot/STU note (admonition block) near the `source.topic` field definition in the specification, seeking implementer feedback on how `topic` is used in practice and in which use cases it provides value.

2. **Guidance paragraphs for optional elements**: For each of the following optional elements, add a brief guidance paragraph (or expand existing description) explaining when and why the element should be populated, with a concrete example use case:
   - `card.uuid` — needed for feedback/audit trails
   - `source.url` — provides traceability to the evidence source
   - `source.topic` — enables filtering, categorization, and analytics
   - `suggestions` / `suggestion.actions` — enable actionable decision support and outcome tracking
   - `suggestion.uuid` — required for feedback tracking of accepted suggestions
   - `prefetch` (on Discovery) — improves performance and enables complete data capture
   - `card.overrideReasons` — enables structured capture of override rationale for retrospective analysis

## Detailed Implementation Plan

### Step 1: Add STU/Ballot Note for `source.topic`

**File:** `docs/specification/current.md`
**Location:** Immediately after the `Source` attributes table and the existing `source` example JSON block

**Action:** Insert an admonition block (MkDocs `!!! note` syntax) with the STU note:

```markdown
!!! note "STU Note: Seeking Feedback on `topic`"
    We are seeking feedback on the use of `topic` in specific use cases. The `topic` field on the `source` 
    object provides a high-level categorization of the card content that can be useful for filtering, searching, 
    or ordering related cards. Implementers are encouraged to provide feedback on:
    
    - Which clinical domains or use cases benefit from populating `topic`
    - Whether a standardized set of topic codes would be valuable
    - How CDS Clients currently use or could use `topic` for card management and display
```

### Step 2: Add Guidance for `card.uuid`

**File:** `docs/specification/current.md`
**Location:** After the Card Attributes table, before the Source section

**Action:** Add a guidance paragraph:

```markdown
#### Guidance on Optional Card Elements

While `uuid` is optional, CDS Services that support the [Feedback](#feedback) endpoint SHOULD populate `uuid` 
on every card. Without a `uuid`, a CDS Client cannot provide feedback on a card, and retrospective analysis 
of CDS effectiveness (e.g., acceptance rates, override patterns) is not possible. For example, a medication 
interaction service that returns cards with `uuid` values enables the health system to track how often 
clinicians accept vs. override the interaction warnings, supporting continuous quality improvement.
```

### Step 3: Add Guidance for `source.url` and `source.topic`

**File:** `docs/specification/current.md`
**Location:** After the Source attributes table, before the existing `source` JSON example

**Action:** Add guidance paragraph:

```markdown
**Guidance on Source Metadata:**
Populating `url` on the source provides clinicians and administrators with traceability back to the 
evidence or organizational source behind the decision support. For example, a drug-drug interaction 
service might set `source.url` to link to the clinical evidence repository entry for the interaction.

The `topic` field enables CDS Clients to organize, filter, and prioritize cards from multiple services. 
For example, a service providing opioid prescribing guidance might use a `topic` coding such as 
`{"system": "http://example.org/topics", "code": "opioid-safety", "display": "Opioid Safety"}` to 
allow the CDS Client to group all opioid-related cards together or apply institution-specific display rules.
```

### Step 4: Add Guidance for `suggestions` and `suggestion.uuid`

**File:** `docs/specification/current.md`
**Location:** After the Suggestion attributes table

**Action:** Add guidance paragraph:

```markdown
**Guidance on Suggestion Metadata:**
CDS Services SHOULD populate `uuid` on each suggestion when the service supports the 
[Feedback](#feedback) endpoint. The `suggestion.uuid` is required for a CDS Client to report which 
specific suggestion was accepted. Without it, the service cannot determine which of potentially 
multiple suggestions on a card was selected by the clinician.

Populating `actions` within suggestions enables the CDS Client to take concrete, discrete steps 
on behalf of the user (e.g., creating a medication order, updating a condition). For example, 
a diabetes management service could return a suggestion with an action to create a hemoglobin A1c 
lab order, enabling one-click ordering directly from the card.
```

### Step 5: Add Guidance for `prefetch` in Discovery

**File:** `docs/specification/current.md`
**Location:** In the Discovery section, after the CDS Service attributes table

**Action:** Add guidance paragraph:

```markdown
**Guidance on Prefetch:**
While `prefetch` is optional in the discovery response, CDS Services SHOULD define prefetch templates 
for the data they require. Prefetch improves service performance (supporting the target of 500ms response 
times) by allowing CDS Clients to provide needed data proactively rather than requiring the service to 
make separate FHIR API calls. For example, a service checking medication interactions should define 
prefetch templates for the patient's current medication list to avoid additional round-trips.
```

### Step 6: Add Cross-Reference in Feedback Section

**File:** `docs/specification/current.md`
**Location:** At the beginning of the Feedback section, after the introductory paragraph

**Action:** Add a note connecting optional metadata elements to feedback capabilities:

```markdown
!!! info "Prerequisites for Effective Feedback"
    To enable meaningful feedback, CDS Services should populate `card.uuid` on every card and 
    `suggestion.uuid` on every suggestion. Without these identifiers, CDS Clients cannot report 
    which cards were acted upon or which suggestions were accepted. Organizations seeking to perform 
    retrospective analysis of CDS effectiveness should ensure their CDS Services populate these fields 
    and that their CDS Clients support the feedback endpoint.
```

### Step 7 (Optional): Update Best Practices

**File:** `docs/best-practices.md`
**Location:** Under the "CDS Service Developers" section

**Action:** Add a bullet point:

```markdown
* Populate optional metadata elements (`card.uuid`, `suggestion.uuid`, `source.topic`, `source.url`, 
  `overrideReasons`) to enable retrospective analysis and quality improvement. These fields are essential 
  for organizations leveraging the feedback endpoint to track CDS effectiveness.
```

- This ticket was originally filed in 2020, deferred post-STU2 publication, and re-opened in 2022. The final resolution was agreed upon in an October 8, 2025 call with Floyd Eisenberg, Bryn Rhodes, Isaac Vetter, and Vadim Peretokin.
