## Resolution Description

> Submitter will provide examples to review with the committee to establish guidance and potential specification for how to use GuidanceResponse and PlanDefinition together to implement behavior of a decision support service.

## Problem Statement

PlanDefinition does not currently provide a specification for how GuidanceResponse fits into the decision support workflow. When defining `dynamicValue.path` in a PlanDefinition, the correct path depends on the structure of the GuidanceResponse result — e.g., whether the response uses a RequestOrchestration (formerly RequestGroup) or a CarePlan. Without explicit guidance, implementers must guess how to bridge the gap between the definitional PlanDefinition/$apply output and the GuidanceResponse wrapper that communicates results to a consuming system.

### Current State Analysis

The FHIR specification today has these relevant pieces:

1. **PlanDefinition.$apply** — produces a Bundle of request resources (RequestOrchestration, individual requests like MedicationRequest, ServiceRequest, etc.). Documented in `plandefinition-notes.xml`.
2. **GuidanceResponse** — captures the formal response to a decision support request, including `module[x]` (what was evaluated), `result` (proposed actions, references to request resources), and `outputParameters`. Documented in `guidanceresponse-introduction.xml`.
3. **CDS on FHIR page** (`clinicalreasoning-cds-on-fhir.html`) — describes the CDS Hooks integration pattern where a CDS Service performs `$apply` against a PlanDefinition and transforms the resulting RequestOrchestration into a CDS Hooks response. However, this page does not describe the role of GuidanceResponse in this flow.
4. **GuidanceResponse.module[x]** — can accept a `canonical` type, which can reference a PlanDefinition, but this is not explicitly documented as a pattern.
5. **GuidanceResponse.result** — references request resources (including RequestOrchestration, CarePlan, etc.), which are exactly the output of PlanDefinition.$apply, but the connection is not explicitly specified.

### Key Gap

There is no documentation that explicitly describes:
- How GuidanceResponse wraps the results of a PlanDefinition $apply operation
- How `GuidanceResponse.module[x]` should reference the evaluated PlanDefinition
- How `GuidanceResponse.result` maps to the output of $apply
- How `dynamicValue.path` expressions in a PlanDefinition should be written when the intended output context includes a GuidanceResponse
- Complete examples showing the end-to-end pattern

### Important Comment (bryn.rhodes, 2023-03-12)

> The $apply operations are specifically about constructing the request resources involved. It would be up to a layer that was calling the $apply to provide information and processing semantics for the GuidanceResponse resource. I looked at adding something like a %response variable to the $apply processing context, but it really feels like that would be reaching outside the context of the $apply operation itself, and that there ought to be a higher level operation defined to support like. CDS Hooks service request is that operation, and the documentation provided in the decision support topic provides an illustration of how the guidance response content can be mapped to the CDS Hooks service request/response interaction. Suggest reopening this topic for further discussion.

This comment clarifies that the resolution should focus on **guidance and examples** rather than structural changes to $apply. The GuidanceResponse is an outer envelope applied by a layer above $apply, and the specification should document this pattern clearly.

## Feature Proposal

Add narrative guidance and examples to the FHIR Core specification that document how GuidanceResponse and PlanDefinition work together in a decision support service workflow. This involves:

1. **Narrative guidance** in PlanDefinition notes explaining how GuidanceResponse wraps $apply results
2. **Narrative guidance** in GuidanceResponse notes explaining the relationship to PlanDefinition
3. **Updated introduction** for GuidanceResponse documenting the PlanDefinition connection
4. **A new example** (or enhanced existing example) showing GuidanceResponse wrapping PlanDefinition $apply output

No structural changes to resource definitions are proposed; the `module[x]` canonical type already supports referencing a PlanDefinition, and `result` already supports referencing RequestOrchestration and other request resources.

## Detailed Implementation Plan

### Task 1: Update GuidanceResponse Introduction — Boundaries and Relationships

**File:** `source/guidanceresponse/guidanceresponse-introduction.xml`

**Current state:** The introduction has a commented-out Boundaries and Relationships section placeholder. The Scope and Usage section mentions CDS Hooks alignment and references `clinicalreasoning-cds-on-fhir.html#evaluation` but does not describe the relationship with PlanDefinition.

**Changes:**
- Uncomment and populate the Boundaries and Relationships section
- Add content explaining:
  - `GuidanceResponse.module[x]` as `moduleCanonical` can reference a PlanDefinition to indicate what was evaluated
  - `GuidanceResponse.result` carries the request resources produced by `PlanDefinition/$apply` (typically a RequestOrchestration)
  - GuidanceResponse acts as an envelope/wrapper around the $apply output, adding metadata (status, performer, occurrence time, evaluation messages, output parameters, data requirements)
  - The $apply operation itself produces the clinical content; GuidanceResponse records the interaction context

### Task 2: Expand GuidanceResponse Notes

**File:** `source/guidanceresponse/guidanceresponse-notes.xml`

**Current state:** Contains only a brief section on GuidanceResponse search behavior.

**Changes:**
- Add a new section "Using GuidanceResponse with PlanDefinition" before the existing search section
- Document the pattern:
  1. A decision support service receives a request (e.g., CDS Hooks hook, or direct invocation)
  2. The service identifies the appropriate PlanDefinition
  3. The service invokes `PlanDefinition/$apply` with the relevant context (subject, encounter, practitioner, etc.)
  4. The $apply operation returns a Bundle containing request resources (typically organized in a RequestOrchestration)
  5. The service constructs a GuidanceResponse:
     - Sets `moduleCanonical` to the canonical URL of the PlanDefinition
     - Sets `status` based on the outcome of the evaluation
     - Sets `result` to reference the RequestOrchestration (or other request resources) from the $apply output
     - Populates `outputParameters` with any additional evaluation outputs
     - Populates `dataRequirement` if additional data is needed
  6. The GuidanceResponse is returned to the caller
- Add a note about dynamicValue.path: when authoring PlanDefinition `dynamicValue.path` expressions, the path should target the request resource structure (e.g., RequestOrchestration action elements via `%action`), not the GuidanceResponse structure, because $apply produces request resources, not GuidanceResponse resources. The GuidanceResponse wrapping happens at a layer above $apply.

### Task 3: Add GuidanceResponse Context to PlanDefinition Notes

**File:** `source/plandefinition/plandefinition-notes.xml`

**Current state:** Has detailed documentation of the $apply process but no mention of GuidanceResponse. The note at the end says results are transient and it's up to the caller to determine how they are carried out.

**Changes:**
- Add a new subsection after the existing "Applying a PlanDefinition" section (after the paragraph about transient results, around line 70-71), titled "Decision Support Service Integration"
- Content should explain:
  - When PlanDefinition.$apply is used as part of a decision support service, the results are typically wrapped in a GuidanceResponse resource
  - The GuidanceResponse provides a container for the status, metadata, and output of the evaluation, while the $apply output provides the clinical content
  - Reference the GuidanceResponse resource documentation for details on the wrapper structure
  - Reference the CDS-on-FHIR page (or Using Clinical Reasoning IG per FHIR-53085) for integration patterns with CDS Hooks
  - Clarify that `dynamicValue.path` expressions target the request resources produced by $apply (not the GuidanceResponse), and that the GuidanceResponse wrapping is handled by the calling layer

### Task 4: Create or Enhance Example

**File:** New file `source/guidanceresponse/guidanceresponse-plandefinition-example.xml` (or enhance existing `guidanceresponse-example.xml`)

**Recommendation:** Create a new separate example to clearly illustrate the PlanDefinition-driven pattern, keeping the existing example (which shows a simple URI-based module) as-is.

**Example content should demonstrate:**
- `GuidanceResponse` with `moduleCanonical` referencing a PlanDefinition (e.g., a clinical decision support rule)
- `status` of `success`
- `result` referencing a contained or external RequestOrchestration that was produced by $apply
- `subject`, `encounter`, and `performer` populated
- `outputParameters` with evaluation-specific parameters
- Optionally a contained RequestOrchestration showing actions with proposal intent

**If creating a new example:**
- Also update `source/guidanceresponse/list-GuidanceResponse-examples.xml` to register the new example
- Also update `source/guidanceresponse/guidanceresponse-examples-header.xml` if needed to describe the new example

### Task 5: Review and Cross-Reference

- Ensure the new narrative content cross-references appropriately:
  - GuidanceResponse introduction → PlanDefinition notes
  - PlanDefinition notes → GuidanceResponse notes  
  - Both → CDS-on-FHIR page / Using Clinical Reasoning IG
- Verify no contradictions with existing content
- Ensure alignment with FHIR-53085 direction (content moving to Using Clinical Reasoning IG)
