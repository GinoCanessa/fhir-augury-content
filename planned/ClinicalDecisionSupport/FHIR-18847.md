## Resolution Summary

**Resolution:** Persuasive.

Quoted resolution description:

> Submitter will provide examples to review with the committee to establish
> guidance and potential specification for how to use GuidanceResponse and
> PlanDefinition together to implement behavior of a decision support service.

In short, the committee accepted the submitter's premise — that PlanDefinition
lacks normative guidance on how to use it together with GuidanceResponse —
and committed to adding that guidance based on submitter-supplied examples.
The deliverable is documentation/prose (plus illustrative examples), not a
schema change: explain how a decision support service layer can produce a
GuidanceResponse whose `result` points at the output of `PlanDefinition/$apply`
(a RequestOrchestration, a CarePlan, or an individual request resource), and
clarify the `dynamicValue.path` implications when that wrapping resource varies.

A later comment from bryn.rhodes (2023-03-12) qualifies the resolution: the
`$apply` operation itself is scoped to producing request resources, and the
orchestration of inputs/outputs around a GuidanceResponse "really feels like
[it would reach] outside the context of the $apply operation itself." He points
at CDS Hooks as the higher-level operation where that mapping is already
illustrated, and explicitly suggests **reopening this topic for further
discussion**. That caveat must be carried forward into Open Questions — the
plan below implements the *documentation* portion of the Persuasive resolution
but flags that the work group may elect to redirect this to CDS Hooks
cross-references instead of adding new semantics to `$apply`.

## Feature Proposal

### Problem Statement

As of the ticket's original submission, the PlanDefinition specification page
describes how `$apply` produces a RequestOrchestration (historically a
RequestGroup) and/or request resources, but it does not describe how
`PlanDefinition` composes with `GuidanceResponse` when a decision support
service wraps that output for return to a caller. Concretely, the submitter
observed that `PlanDefinition.action.dynamicValue.path` expressions are
written against the target *request* resource (e.g., a RequestOrchestration
action, a CarePlan activity) — so an author writing a PlanDefinition intended
to run inside a decision support service does not know whether the path should
be rooted at the RequestOrchestration, the CarePlan, or the GuidanceResponse
itself. The spec offers no anchor text answering that question.

A confirming read of the current clone (`HL7/fhir` @ `db79dcd`) shows this
gap is still present:

- `source/plandefinition/plandefinition-notes.xml` — the "Applying a
  PlanDefinition" narrative walks through RequestOrchestration / CarePlan /
  Task wrapping but never names `GuidanceResponse` (`Select-String
  GuidanceResponse` returns no matches in `source/plandefinition/`).
- `source/plandefinition/plandefinition-introduction.xml` — Scope/Boundaries
  prose likewise does not mention GuidanceResponse.
- `source/guidanceresponse/guidanceresponse-notes.xml` — scoped to
  "GuidanceResponse Search"; no cross-link back to PlanDefinition.
- `source/guidanceresponse/guidanceresponse-example.xml` — an example exists
  but is not tied into PlanDefinition narrative.

### Proposed Change

A documentation-only change to the PlanDefinition and GuidanceResponse
topic pages, plus a narrow clarification of `dynamicValue.path` semantics.
No StructureDefinition or schema edits.

1. **New subsection in `source/plandefinition/plandefinition-notes.xml`:
   "Using PlanDefinition with GuidanceResponse"** — to be inserted after the
   existing "Applying a PlanDefinition" / `$apply`-result paragraphs and
   before "Event Resources". Content (to be confirmed by WG):

   - A decision support service implementing `PlanDefinition/$apply` MAY
     return the result wrapped in a `GuidanceResponse`. In that case:
     - `GuidanceResponse.module[x]` references (or embeds) the applied
       PlanDefinition canonical.
     - `GuidanceResponse.result` references the primary output of `$apply`
       — typically a `RequestOrchestration`, but MAY be a `CarePlan`, a
       single request resource (e.g., `ServiceRequest`,
       `MedicationRequest`), or a `Bundle` of the above.
     - `GuidanceResponse.dataRequirement` reflects the data the
       PlanDefinition (and its libraries) declared it needs.
   - Clarify that `dynamicValue.path` is **always rooted at the resource the
     `$apply` operation produces for the enclosing action** — not at the
     wrapping `GuidanceResponse`. That is, the same PlanDefinition works
     whether or not a decision support service wraps its output in a
     GuidanceResponse; the wrapping is done by the service layer *after*
     `$apply` returns and is out of scope for `dynamicValue` expressions.
   - Point to CDS Hooks as the interoperable transport when the decision
     support service is invoked remotely, and reference the
     `clinicalreasoning-topics.html` discussion that illustrates how
     GuidanceResponse content can be mapped to the CDS Hooks
     service-request/service-response interaction (per bryn.rhodes 2023
     comment).
   - Show (inline snippet) a skeleton response in which a GuidanceResponse
     with `status=success`, `module[x]=PlanDefinition/…`, and
     `result=Reference(RequestOrchestration/…)` wraps an existing
     `plandefinition-example.xml`-style RequestOrchestration. The snippet
     MAY reuse `source/guidanceresponse/guidanceresponse-example.xml` as
     the target of the Reference so no new example file is strictly
     required.

2. **Cross-link update in
   `source/guidanceresponse/guidanceresponse-notes.xml`:** add a new
   leading section "Relationship to PlanDefinition" that mirrors the above,
   stating that a GuidanceResponse returned by a decision support service
   MAY carry the `$apply` output in `result` and pointing readers to the
   new section in `plandefinition.html#applying-a-plandefinition` (or its
   anchor equivalent). Preserve the existing "GuidanceResponse Search"
   section below it.

3. **No change** to:
   - `source/plandefinition/structuredefinition-PlanDefinition.xml` — no
     cardinality, type, or binding changes are implied by the resolution.
   - `source/guidanceresponse/structuredefinition-GuidanceResponse.xml` —
     same rationale.
   - `operationdefinition-PlanDefinition-apply.xml` — the resolution does
     not add a `%response` context variable or change `$apply` parameters;
     bryn.rhodes explicitly rejected reaching outside the `$apply` scope.
   - Search parameters, value sets, code systems, or examples (the existing
     `guidanceresponse-example.xml` already demonstrates a usable shape).

### Design Rationale

- **Documentation, not schema.** The resolution text ("establish guidance
  and potential specification") and the 2023 clarifying comment both point
  at narrative clarification rather than a new `$apply` parameter. The
  briefing's **Update narrative / prose** recipe (from the FhirCore
  playbook) is the exact fit: edit the `source/<resource>/*-notes.xml`
  authored HTML fragments directly.
- **Keep `dynamicValue.path` rooted at the $apply target.** Making
  `dynamicValue.path` resolve against a potential outer wrapper
  (GuidanceResponse) would create a PlanDefinition whose validity depends
  on whether a wrapping service is present — an implementation detail of
  the caller, not the PlanDefinition itself. The proposed wording preserves
  the current contract: paths are always written against the resource
  produced by the enclosing action.
- **Prefer cross-links to CDS Hooks.** bryn.rhodes's argument that the
  "higher level operation" for decision support is already described by
  CDS Hooks is persuasive; authoring a parallel decision-support operation
  inside Core would duplicate effort. Cross-linking discharges the Core
  spec's obligation without forking the narrative.
- **No new example file.** The existing `guidanceresponse-example.xml`
  plus `plandefinition-example.xml` are sufficient to illustrate the wrap
  pattern; adding another example inflates the publisher's example-list
  footprint (`list-GuidanceResponse-examples.xml`,
  `list-PlanDefinition-examples.xml`) for no additional explanatory power.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add "Using PlanDefinition with GuidanceResponse" subsection.**
   - File: `source/plandefinition/plandefinition-notes.xml`
   - Action: Insert a new `<h3>Using PlanDefinition with
     GuidanceResponse</h3>` block immediately after the paragraph ending
     "…the result effectively represents a proposed set of activities, and
     it is up to the caller to determine whether and how those activities
     are actually carried out." and before the existing
     `<h3>Event Resources</h3>`. Wrap the new section with an
     `<a name="guidanceresponse"></a>` anchor so other pages can link to
     `plandefinition.html#guidanceresponse`.
   - Details:
     - Describe that a decision support service MAY wrap the `$apply`
       output in a `GuidanceResponse` and set
       `GuidanceResponse.result = Reference(...)` to the
       RequestOrchestration / CarePlan / request resource produced by
       `$apply`, and `GuidanceResponse.module[x]` to the applied
       PlanDefinition.
     - State explicitly that `PlanDefinition.action.dynamicValue.path`
       is rooted at the `$apply` target resource for that action — **not**
       at any wrapping GuidanceResponse. Give a worked example: if the
       enclosing action produces a `ServiceRequest`, `path = "priority"`
       means `ServiceRequest.priority`; the presence or absence of a
       GuidanceResponse wrapper is irrelevant.
     - Cross-link: `<a href="guidanceresponse.html">GuidanceResponse</a>`,

     - Include a small inline XML skeleton showing a `GuidanceResponse`
       with `status=success`, `moduleCanonical`, and
       `result=Reference(RequestOrchestration/example)`. Do not author
       this as a separate example file; inline it inside the notes
