## Description

This ticket is a ballot comment (line item #53) from the Clinical Practice Guidelines (CPG) IG STU1 ballot (September 2019). The commenter asks:

> Why is this extension exclusive to CQF? It should be reviewed by the work group to determine whether this extension has a broader potential applicability and be registered for as broad a use as is reasonable.

## Resolution Description

> Agreed, we are considering this extension for inclusion in (and perhaps first class representation within) the base FHIR specification.

## Implementer Comment

Bryn Rhodes (2020-11-24):
> This change is applied but I get an error that I am performing an illegal operation when I try to mark it applied.

## Feature Proposal

### What Is Being Asked

The ballot commenter (Paul Knapp) filed a series of identical comments (FHIR-24138/CPG#53, FHIR-24140/CPG#54, FHIR-24142/CPG#55, FHIR-24146/CPG#57) questioning why certain CQF extensions in the CPG Implementation Guide were exclusively scoped to CQF use. The request is that these extensions be reviewed for broader applicability and registered at the widest reasonable context (e.g., in the FHIR Extensions Pack or the core FHIR specification) rather than remaining confined to the CPG IG.

### What This Ticket Specifically Covers (CPG #53)

The exact extension referenced by CPG ballot line item #53 cannot be definitively traced from the available data — the original ballot spreadsheet mapping line items to specific IG artifacts is not accessible. However, the pattern across the sibling tickets makes the intent clear: whichever CQF extension this line item refers to should be promoted from CPG-IG-only scope to the FHIR Extensions Pack or core specification.

### Current State of CQF Extensions

Since this ballot comment was filed (2019), a significant number of CQF extensions have been promoted to the FHIR Extensions Pack (`HL7/fhir-extensions`). As of the current repository state, 60+ `cqf-*` extensions are registered in the extensions pack with appropriate broad contexts:

| Extension | Context | Location in Extensions Pack |
|-----------|---------|---------------------------|
| `cqf-knowledgeCapability` | `DomainResource` | `input/definitions/Resource/` |
| `cqf-knowledgeRepresentationLevel` | `DomainResource` | `input/definitions/Resource/` |
| `cqf-expression` | `Element` | `input/definitions/Element/` |
| `cqf-calculatedValue` | `Element` | `input/definitions/Element/` |
| `cqf-qualityOfEvidence` | `PlanDefinition.action` | `input/definitions/Element/` |
| `cqf-strengthOfRecommendation` | `PlanDefinition.action` | `input/definitions/Element/` |
| `cqf-library` | (multiple) | `input/definitions/Resource/` |
| `cqf-directReferenceCode` | (multiple) | `input/definitions/Resource/` |
| `cqf-inputParameters` | (multiple) | `input/definitions/Resource/` |
| `cqf-logicDefinition` | (multiple) | `input/definitions/Resource/` |
| ...and 50+ more | various | various subdirectories |

Additionally, some concepts originally in CQF extensions were promoted to first-class elements in the FHIR R5 core specification (e.g., the `Expression` datatype was influenced by `cqf-expression`).
