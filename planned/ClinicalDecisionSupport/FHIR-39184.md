## Original Request

The ticket requested enhancing implicit conversion to quantity values to include units inferrable from context. The motivating example:

```cql
MedicationRequest.expectedSupplyDuration between 30 and 60 days
```

The natural reading implies the lower boundary `30` should also be interpreted as `30 days`, but the CQL translator does not infer units during implicit conversion of integer/decimal to quantity. The integer `30` is implicitly converted to the quantity `30 '1'` (with default unit `'1'`), causing a unit mismatch and a run-time error.

## Resolution Description

> Because it is not always the case that units can be correctly inferred from context (i.e. the user may have intended to have different units in the first argument), this should not be done.
>
> Note that because the example will result in a run-time, rather than compile-time error, add discussion about this case and that implementation environments should make use of unit conversion validation to help warn users at compile-time when this situation occurs.
>
> The minimum viable would be to simply provide warnings whenever the implicit conversion from integer/decimal to quantity results in a quantity with a default unit.

**Disposition: The original request to infer units from context is rejected. Instead, the spec should add documentation/discussion, and recommend compile-time warnings.**

## Impact Analysis

### Nature of Change

This is a **documentation-only change** to the CQL specification. No behavioral changes to the type system, no grammar changes, and no ELM schema changes are required.

The resolution calls for two things:

1. **Spec discussion**: Add narrative text discussing the pitfall of implicit integer/decimal → quantity conversion with default units in contexts like `between`, and the resulting run-time errors.
2. **Implementation guidance**: Recommend that CQL implementations (translators/compilers) produce compile-time warnings when an implicit conversion from `Integer` or `Decimal` to `Quantity` results in a quantity with the default unit (`'1'`).

### Risk Assessment

**Low risk.** This is additive documentation. No existing behavior changes. No breaking changes to syntax, ELM, or type system.

## Detailed Implementation Plan

### File 1: `input/pages/03-developersguide.md` (Developer's Guide)

**Section: Implicit Conversions** (near Table 3-J)

The implicit conversions table already shows that `Integer → Quantity` and `Decimal → Quantity` are implicit. The `ToQuantity(Integer)` and `ToQuantity(Decimal)` operators are documented as producing "a Quantity with a default unit ('1')".

**Action:** After the implicit conversion table (Table 3-J) and its surrounding discussion, add a new subsection or note block discussing the default-unit pitfall:

```markdown
##### Default Unit Warnings for Implicit Quantity Conversion

When an Integer or Decimal value is implicitly converted to a Quantity, the resulting
Quantity has a default unit of '1'. In contexts where the Quantity is subsequently
compared or used in operations with Quantities that have specific units, this default
unit will typically result in a unit mismatch and a run-time error rather than the
intuitively expected behavior. For example:

    MedicationRequest.expectedSupplyDuration between 30 and 60 days

In this expression, `60 days` is a Quantity literal with unit `'days'`, but `30` is an
Integer that is implicitly converted to the Quantity `30 '1'`. Because the units do not
match (`'1'` vs `'days'`), the `between` operation will fail at run-time.

Although it may seem that the unit `'days'` could be inferred from the context of the
`between` expression, it is not always the case that units can be correctly inferred.
The user may have intended different units for the first argument. Therefore, CQL does
not perform unit inference during implicit conversion.

> **Implementation Note:** Implementation environments SHOULD provide compile-time
> warnings whenever an implicit conversion from Integer or Decimal to Quantity results
> in a Quantity with the default unit ('1'). Such warnings help authors identify
> potential unit mismatch errors before run-time. At minimum, implementations SHOULD
> warn in the context of binary operators (such as `between`, comparisons, and
> arithmetic) where the other operand is a Quantity with an explicit unit.
```
