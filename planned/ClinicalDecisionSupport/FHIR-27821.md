## Resolution Description

> Add an example of a PlanDefinition that represents a manufacturing protocol as described in the examples in the comments.

## Problem Statement

The original request asked for `PlanDefinition.action.definition[x]` to support referencing `Procedure` resources directly (or for the binding to become extensible) so that non-clinical use cases (e.g., manufacturing protocols, physical security inspections, environment inspections) could be represented without requiring an intermediate `ActivityDefinition` layer.

After extensive discussion between the reporter (Peter Bomberg) and CDS work group (Bryn Rhodes), the resolution was modified to:
1. **Not** change the `definition[x]` type list or binding.
2. Clarify that the existing mechanism — using `ActivityDefinition` with `kind=ServiceRequest` to describe abstract procedures, and `RequestOrchestration` for instances — already supports these use cases.
3. **Add an example** PlanDefinition representing a manufacturing protocol to demonstrate this pattern.

## Feature Proposal

Add a new PlanDefinition example resource file to the FHIR Core specification that illustrates a **manufacturing protocol** use case. The example should demonstrate:

1. A `PlanDefinition` with `type` of `clinical-protocol` (or potentially no type, since this is non-clinical) that models a manufacturing/inspection protocol.
2. Hierarchical actions with sub-activities (e.g., environment inspection → air quality, lighting, temperature, humidity sub-checks).
3. Use of contained `ActivityDefinition` resources with `kind=ServiceRequest` to define each inspection/manufacturing step.
4. Use of `definitionCanonical` references from actions to the contained `ActivityDefinition` resources.
5. Sequencing relationships between actions using `relatedAction` where appropriate.

This demonstrates that non-clinical protocols can be represented using the existing PlanDefinition structure without modifying `definition[x]`.

## Impact Analysis

### Scope of Change

This is an **example-only** change. No structural modifications to the PlanDefinition resource, no new elements, no binding changes. The impact is limited to:
- One new XML example file
- One update to the examples list file
- Optionally, a brief mention in the examples header or introduction

### Risk Assessment

**Low risk.** Adding an example cannot break existing behavior, schemas, or validation. The example must be valid against the existing PlanDefinition StructureDefinition.

## Detailed Implementation Plan

### Step 1: Create New Example File

**File:** `source/plandefinition/plandefinition-example-manufacturing-protocol.xml`
**Repository:** `HL7/fhir`

Create a new PlanDefinition XML example resource modeled as a manufacturing protocol. Based on the comments (Peter Bomberg's examples), the example should represent an **environment inspection protocol** with sub-activities for various environmental checks.

#### Recommended Example Structure

```xml
<PlanDefinition xmlns="http://hl7.org/fhir">
  <id value="example-manufacturing-protocol"/>
  <text>
    <status value="generated"/>
    <div xmlns="http://www.w3.org/1999/xhtml">
      <!-- Generated narrative for the manufacturing protocol example -->
    </div>
  </text>

  <!-- Contained ActivityDefinitions for each inspection sub-activity -->
  <contained>
    <ActivityDefinition>
      <id value="ad-air-quality-check"/>
      <status value="draft"/>
      <description value="Perform air quality assessment in the manufacturing environment"/>
      <kind value="ServiceRequest"/>
      <code>
        <text value="Air Quality Check"/>
      </code>
    </ActivityDefinition>
  </contained>
  <contained>
    <ActivityDefinition>
      <id value="ad-temperature-check"/>
      <status value="draft"/>
      <description value="Measure and record ambient temperature"/>
      <kind value="ServiceRequest"/>
      <code>
        <text value="Temperature Check"/>
      </code>
    </ActivityDefinition>
  </contained>
  <contained>
    <ActivityDefinition>
      <id value="ad-humidity-check"/>
      <status value="draft"/>
      <description value="Measure and record relative humidity"/>
      <kind value="ServiceRequest"/>
      <code>
        <text value="Humidity Check"/>
      </code>
    </ActivityDefinition>
  </contained>
  <contained>
    <ActivityDefinition>
      <id value="ad-lighting-check"/>
      <status value="draft"/>
      <description value="Assess lighting conditions in the manufacturing area"/>
      <kind value="ServiceRequest"/>
      <code>
        <text value="Lighting Assessment"/>
      </code>
    </ActivityDefinition>
  </contained>
  <contained>
    <ActivityDefinition>
      <id value="ad-airflow-check"/>
      <status value="draft"/>
      <description value="Verify airflow patterns and ventilation system operation"/>
      <kind value="ServiceRequest"/>
      <code>
        <text value="Airflow Verification"/>
      </code>
    </ActivityDefinition>
  </contained>

  <identifier>
    <system value="http://example.org/fhir/manufacturing"/>
    <value value="env-inspection-protocol-1"/>
  </identifier>
  <name value="ManufacturingEnvironmentInspectionProtocol"/>
  <title value="Manufacturing Environment Inspection Protocol"/>
  <type>
    <coding>
      <code value="clinical-protocol"/>
      <!-- NOTE: The type code system currently only has clinical protocol;
           a non-clinical protocol type may be considered in the future.
           For now, this illustrates the general protocol pattern. -->
    </coding>
  </type>
  <status value="draft"/>
  <description value="An example protocol for performing a manufacturing environment
    inspection, including checks for air quality, temperature, humidity, lighting,
    and airflow. This example demonstrates the use of PlanDefinition for
    non-clinical protocols using the existing definition[x] mechanism with
    contained ActivityDefinition resources."/>
  <purpose value="To demonstrate how PlanDefinition can be used to represent
    non-clinical manufacturing and inspection protocols"/>

  <!-- Top-level action: Environment Inspection (group) -->
  <action>
    <title value="Environment Inspection"/>
    <description value="Perform all environmental checks for the manufacturing area"/>
    <groupingBehavior value="logical-group"/>
    <selectionBehavior value="all"/>

    <!-- Sub-action 1: Air Quality -->
    <action id="air-quality-action">
      <title value="Air Quality Check"/>
      <description value="Assess air quality including particulate matter and contaminant levels"/>
      <requiredBehavior value="must"/>
      <definitionCanonical value="#ad-air-quality-check"/>
    </action>

    <!-- Sub-action 2: Temperature -->
    <action id="temperature-action">
      <title value="Temperature Check"/>
      <description value="Measure ambient temperature and verify within acceptable range"/>
      <requiredBehavior value="must"/>
      <definitionCanonical value="#ad-temperature-check"/>
    </action>

    <!-- Sub-action 3: Humidity -->
    <action id="humidity-action">
      <title value="Humidity Check"/>
      <description value="Measure relative humidity and verify within acceptable range"/>
      <requiredBehavior value="must"/>
      <definitionCanonical value="#ad-humidity-check"/>
    </action>

    <!-- Sub-action 4: Lighting -->
    <action id="lighting-action">
      <title value="Lighting Assessment"/>
      <description value="Assess lighting conditions for compliance with manufacturing standards"/>
      <requiredBehavior value="must"/>
      <definitionCanonical value="#ad-lighting-check"/>
    </action>

    <!-- Sub-action 5: Airflow -->
    <action id="airflow-action">
      <title value="Airflow Verification"/>
      <description value="Verify airflow patterns and ventilation system operation"/>
      <requiredBehavior value="must"/>
      <definitionCanonical value="#ad-airflow-check"/>
    </action>
  </action>
</PlanDefinition>
```

#### Key Design Decisions

1. **Contained ActivityDefinitions with `kind=ServiceRequest`**: Follows the pattern established by existing examples (e.g., `plandefinition-protocol-example.xml`, `plandefinition-options-example.xml`) where contained ActivityDefinitions are referenced via `definitionCanonical` with fragment references (`#id`).

2. **Hierarchical action structure**: The top-level action is a logical group (`groupingBehavior=logical-group`, `selectionBehavior=all`) containing sub-actions for each inspection type. This demonstrates the reusable package concept described by the reporter.

3. **No changes to `definition[x]`**: The example uses only `definitionCanonical` referencing `ActivityDefinition`, proving the existing mechanism supports non-clinical use cases.

4. **Type code**: Uses `clinical-protocol` as the closest available type code. An implementation note could mention that future versions may add a more general `protocol` type code.

### Step 2: Register the Example in the Examples List

**File:** `source/plandefinition/list-PlanDefinition-examples.xml`
**Repository:** `HL7/fhir`

Add a new `<entry>` element to register the manufacturing protocol example:

```xml
<entry>
  <extension url="http://hl7.org/fhir/build/StructureDefinition/description">
    <valueString value="Manufacturing Environment Inspection Protocol"/>
  </extension>
  <extension url="http://hl7.org/fhir/build/StructureDefinition/title">
    <valueString value="plandefinition-example-manufacturing-protocol"/>
  </extension>
  <item>
    <reference value="PlanDefinition/example-manufacturing-protocol"/>
    <display value="ManufacturingProtocolExample"/>
  </item>
</entry>
```

This should be added alongside existing entries (e.g., after the Chest Pain Coronary Artery Disease entry at line 222).

### Step 3: Verify Example Validity (Optional)

After creating the example file:
1. Ensure the XML is well-formed.
2. Validate that `definitionCanonical` fragment references (`#ad-*`) correctly match the `id` values of the contained `ActivityDefinition` resources.
3. Verify the example resource conforms to the PlanDefinition StructureDefinition (`structuredefinition-PlanDefinition.xml`).
4. Run the FHIR build tooling (`./build.sh` or equivalent) to ensure no build errors.

## Testing Considerations

1. **Build validation**: The FHIR build process validates all example resources. The new example must pass validation.
2. **Narrative generation**: If the build tooling auto-generates narratives, the `<text>` element may be regenerated. The initial `<text>` can use a placeholder or generated content.
3. **Cross-reference integrity**: All contained resource `id` values must match their corresponding `definitionCanonical` fragment references.

## Open Questions

1. **PQ/CMC example**: Bryn Rhodes' latest comment (2025-03-18) asks whether the PQ/CMC (Pharmaceutical Quality/Chemistry, Manufacturing, and Controls) project has specific manufacturing protocol examples from their implementation guides. If so, using a real-world PQ/CMC example would be more authoritative than a generic manufacturing protocol. This should be coordinated with BR&R work group.

2. **Type code**: The `clinical-protocol` type code is somewhat misleading for a non-clinical use case. Should a new type code (e.g., `protocol`) be proposed separately, or should the example omit the type element? The existing code system (`http://terminology.hl7.org/CodeSystem/plan-definition-type`) would need a separate ticket for new codes.

3. **OID assignment**: Existing examples use OIDs for identifiers (e.g., `urn:oid:2.16.840.1.113883.4.642.11.*`). A new OID should be assigned for this example following the existing numbering scheme.

## Summary

This is a straightforward, low-risk change that adds a single new example file and registers it in the examples list. The example demonstrates that the existing `PlanDefinition` + `ActivityDefinition` mechanism (using `definitionCanonical` to contained ActivityDefinitions with `kind=ServiceRequest`) is sufficient for non-clinical manufacturing/inspection protocols — addressing the original concern without modifying `definition[x]` types or bindings.
