## Resolution Description

> Agreed, add this convention:
>
> Re-use of an alias name in a peer or child context within complex queries can make it difficult to follow what is happening in a query. Although it is allowed by the language due to scoping rules, this practice should be discouraged. Give examples of both peer and child contexts re-using an alias name to illustrate the anti-pattern.

## Implementation Required

**Yes** — Resolution is "Persuasive", which requires implementation.

## Feature Proposal

Add a new best-practice subsection to the CQL specification's Formatting Conventions appendix (Appendix G) that:

1. States the convention that alias names should not be re-used in peer or child scopes within complex queries, even though the language allows it via identifier hiding/scoping rules.
2. Provides a CQL code example illustrating alias reuse in **peer scope** (e.g., two `with` clauses in the same query reusing the same alias name).
3. Provides a CQL code example illustrating alias reuse in a **child scope** (e.g., a subquery nested within a parent query reusing the parent's alias name).
4. For each anti-pattern example, shows the corrected (recommended) version with unique alias names.

### Step 1: Add "Alias Reuse" best-practice subsection

**File:** `input/pages/14-g-formattingconventions.md`

**Location:** Under the existing `### Best Practices` section, after the existing `#### Direct Reference Codes` subsection (which is currently the only best-practice entry and ends at the bottom of the file).

**Action:** Append a new `#### Alias Names` subsection.

**Proposed content:**

````markdown
#### Alias Names

Re-use of an alias name in a peer or child context within complex queries can make it difficult to follow what is happening in a query. Although it is allowed by the language due to scoping rules, this practice should be discouraged.

The following example illustrates alias name reuse in a **peer scope** (two `with` clauses in the same query that both use the alias `X`):

```cql
// Anti-pattern: peer scope alias reuse
[Encounter: "Inpatient"] E
  with [Condition: "Diabetes"] X
    such that X.onset during E.period
  with [Condition: "Hypertension"] X
    such that X.onset during E.period
```

In this case, both `with` clauses define an alias named `X`, which makes it difficult to determine which condition is being referenced. The recommended approach is to use distinct alias names:

```cql
// Recommended: unique aliases for peer scopes
[Encounter: "Inpatient"] E
  with [Condition: "Diabetes"] D
    such that D.onset during E.period
  with [Condition: "Hypertension"] H
    such that H.onset during E.period
```

The following example illustrates alias name reuse in a **child scope** (a subquery that reuses the alias from the parent query):

```cql
// Anti-pattern: child scope alias reuse
[Encounter: "Inpatient"] E
  where exists (
    [Condition: "Acute Pharyngitis"] E
      where E.onset during MeasurementPeriod
  )
```

In this case, the subquery defines an alias `E` that hides the outer query alias, making it unclear which `E` is intended in each context. The recommended approach is to use a distinct alias name in the subquery:

```cql
// Recommended: unique alias in child scope
[Encounter: "Inpatient"] E
  where exists (
    [Condition: "Acute Pharyngitis"] P
      where P.onset during E.period
  )
```
````
