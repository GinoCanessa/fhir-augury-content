## Resolution Summary

The CDS work group accepted the comment as **Persuasive**. The approved
resolution is:

> Agreed, remove the `topic` element from `ActivityDefinition`,
> `EventDefinition`, `PlanDefinition`, and `Library` since they were
> deprecated in R5.
>
> Ensure that the R4 conversions to R5 and R6 handle representing `topic`
> using a `useContext` with a code of `topic`.
>
> Note that due to an issue with structure definition rendering in the
> ballot4 content, these elements did not display as "deprecated", but
> they were marked as deprecated in the StructureDefinitions.

Scope per the resolution is therefore the four definitional resources
**ActivityDefinition, EventDefinition, Library, PlanDefinition**. The
`Measure.topic` / `Measure.guidance` / `Measure.subject[x]` and
`Observation.bodySite` items mentioned in the original ticket description
are explicitly **not** part of this resolution (Bryn Rhodes' comment
confirms scope; the duplicate per-resource tickets enumerated by Isaac
Vetter — FHIR-55431 / 55434 / 55435 / 55436 / 55437 / 55438 / 55443 —
should each adopt this same resolution).

## Feature Proposal

### Problem Statement

R6 ballot 4 still carries a `topic` element on `ActivityDefinition`,
`EventDefinition`, `Library`, and `PlanDefinition` that has been marked
**deprecated** in the StructureDefinitions since R5 (see the
`structuredefinition-standards-status` extension on each element).
Per <http://hl7.org/fhir/versions.html#deprecation>, an element that was
Trial Use or Deprecated in the previous release may be **withdrawn** at
the next major release, and R6 is intended as a normative release.

A separate rendering bug in ballot 4 hid the deprecation flag in the
generated HTML (the comment fields still say "DEPRECATION NOTE" but the
element header no longer renders as deprecated), which made the issue
harder to spot but does not change the underlying state in source. The
correct behaviour is to remove these deprecated elements from R6 rather
than ship a normative release that re-asserts an element nobody is
supposed to use.

The `topic` concept itself is preserved through `useContext` with a
`useContext.code` of `topic` (CodeSystem
`http://terminology.hl7.org/CodeSystem/usage-context-type#topic`), which
is already the migration path called out in the existing DEPRECATION
NOTE comment on each element.

### Proposed Change

Make the following concrete edits in `HL7/fhir`:

1. **Remove the deprecated `topic` element** from each of these
   StructureDefinitions (the entire `<element id="...topic">` block,
   `<extension url=".../structuredefinition-standards-status">
   <valueCode value="deprecated"/></extension>` and all child
   `short` / `definition` / `comment` / `requirements` / `min` / `max`
   / `type` / `binding` / `mapping`):
   - `source/activitydefinition/structuredefinition-ActivityDefinition.xml` — `ActivityDefinition.topic` (`0..*` `CodeableConcept`, example binding to `http://hl7.org/fhir/ValueSet/definition-topic`).
   - `source/eventdefinition/structuredefinition-EventDefinition.xml` — `EventDefinition.topic` (same shape).
   - `source/library/structuredefinition-Library.xml` — `Library.topic` (same shape).
   - `source/plandefinition/structuredefinition-PlanDefinition.xml` — `PlanDefinition.topic` (same shape, plus a `<mapping identity="workflow" map="Definition.subject[x]"/>` that goes with it).
2. **Remove the corresponding `topic` SearchParameter** entries from
   each per-resource search-param bundle (these currently carry
   `<status value="normative"/>` on the search parameter; they must go
   with the element):
   - `source/activitydefinition/bundle-ActivityDefinition-search-params.xml` — `code=topic`, `expression=ActivityDefinition.topic`.
   - `source/eventdefinition/bundle-EventDefinition-search-params.xml` — `code=topic`, `expression=EventDefinition.topic`.
   - `source/library/bundle-Library-search-params.xml` — `code=topic`, `expression=Library.topic`.
   - `source/plandefinition/bundle-PlanDefinition-search-params.xml` — `code=topic`, `expression=PlanDefinition.topic`.
3. **Update the Knowledge Repository CapabilityStatement** to drop the
   four `searchParam` references that point at the now-removed search
   parameters:
   - `source/capabilitystatement/capabilitystatement-knowledge-repository.xml` — remove the `topic` `searchParam` entries that reference `SearchParameter/Library-topic` (line ~141), `SearchParameter/PlanDefinition-topic` (line ~208), and `SearchParameter/ActivityDefinition-topic` (line ~275). The `Measure-topic` reference (line ~342) is **out of scope** for this ticket and stays.
4. **Strip `topic` usages from the affected resource examples** so they
   continue to validate after the element is removed. Per a clone scan
   the following example files contain a `<topic>` element today and
   need the element deleted (and, where editorially helpful, replaced
   by an equivalent `<useContext>` with
   `useContext.code = http://terminology.hl7.org/CodeSystem/usage-context-type#topic`
   and the same CodeableConcept moved into `useContext.valueCodeableConcept`):
   - `source/activitydefinition/`: `activitydefinition-example.xml`, `activitydefinition-medicationorder-example.xml`, `activitydefinition-predecessor-example.xml`, `activitydefinition-servicerequest-example.xml`.
   - `source/library/`: `library-composition-example.xml`, `library-example.xml`, `library-exclusive-breastfeeding-cds-logic.xml`, `library-exclusive-breastfeeding-cqm-logic.xml`, `library-fhir-helpers.xml`, `library-fhir-model-definition.xml`, `library-mmi-suiciderisk-orderset-logic.xml`, `library-omtk-logic.xml`, `library-omtk-modelinfo.xml`, `library-opioidcds-common.xml`, `library-opioidcds-recommendation-{04,05,07,08,10,11}.xml`, `library-predecessor-example.xml`, `library-quick-model-definition.xml`, `library-zika-virus-intervention-logic.xml`.
   - `source/plandefinition/`: `plandefinition-chlamydia-screening-intervention.xml`, `plandefinition-example.xml`, `plandefinition-exclusive-breastfeeding-intervention-{01,02,03,04}.xml`, `plandefinition-opioidcds-{04,05,07,08,10,11}.xml`, `plandefinition-predecessor-example.xml`, `plandefinition-zika-virus-intervention.xml`.
5. **Do not remove `valueset-definition-topic.xml`** at
   `source/activitydefinition/valueset-definition-topic.xml`. The
   ValueSet remains referenced by the broader artifact-extensions
   spreadsheet (`source/profiles/artifact-extensions-spreadsheet.xml`)
   and by Measure / metadata-resource bindings outside the scope of
   this resolution. Leave it in place.
6. **Document the migration path on the affected resource pages.** The
   resolution requires a clear cross-version story; for each of the
   four resources, add a short paragraph in the resource notes/intro
   (or the per-resource changes page used by FhirCore) stating that
   `topic` has been removed in R6 and the equivalent value MUST be
   represented as a `useContext` element with
   `useContext.code = topic` from the
   `usage-context-type` CodeSystem, with the previous CodeableConcept
   moved into `useContext.valueCodeableConcept`.
7. **R4 → R5 and R4 → R6 cross-version maps.** The resolution
   explicitly requires that conversion logic represent the old `topic`
   field as a `useContext` with code `topic`. These maps live in
   `HL7/fhir-cross-version` (not `HL7/fhir`) — see Open Questions; this
   plan flags the change but the actual edits are out of scope for the
   `HL7/fhir` repo work and are not enumerated as tasks here.

### Design Rationale

- The DEPRECATION NOTE comment already on each `topic` element directs
  implementers to `useContext.code = topic`. Withdrawing the element in
  R6 simply finishes the migration that was started in R5, consistent
  with `versions.html#deprecation` policy for elements that were
  deprecated in the previous release.
- Replacing `topic` with `useContext` (rather than introducing a new
  element) keeps the categorisation capability intact for consumers
  that need it without re-asserting a parallel mechanism. The
  `usage-context-type` CodeSystem already defines `topic`, so no new
  terminology is required.
- Removing the corresponding `SearchParameter` (rather than
  re-pointing it at `useContext`) is consistent with how other
  deprecated/removed elements have been handled in R6: a generic
  `context-type-value` style search across `useContext` is already
  available on each of these resources.
- Scope is intentionally limited to the four resources named in the
  resolution. The comments thread shows the duplicates were filed
  per-resource (FHIR-55431/55434/55435/55436/55437/55438/55443) and
  the Measure-related items are governed by their own tickets — those
  must adopt the same resolution but should be implemented under
  their own keys to keep the audit trail clean.

### Breaking Changes

This change **is breaking** and must be called out in the R5 → R6
changes/notes:

- **Element removed:** `ActivityDefinition.topic`, `EventDefinition.topic`, `Library.topic`, `PlanDefinition.topic` (`0..*` `CodeableConcept`) — content sent in this position by R5 producers will no longer validate against R6.
- **SearchParameter removed:** `ActivityDefinition-topic`, `EventDefinition-topic`, `Library-topic`, `PlanDefinition-topic` (all currently `status=normative` in source). Servers that previously advertised these search parameters will lose them and clients querying `?topic=` against those endpoints will need to be migrated to `?context-type-value=` (or equivalent) querying the `useContext` element with code `topic`.
- **No type or binding changes** to other elements; the bindings on `useContext.code` and `useContext.valueCodeableConcept` are unchanged.
- **No new required elements** — `topic` was `0..*`, so no element minimum cardinality is being increased.

### Related Specifications

- **`MetadataResource`** (`source/metadataresource/structuredefinition-MetadataResource.xml`) and the broader Definition pattern carry the `useContext` element these instances will move to. No edit needed, but the migration narrative on the four resource pages should link out to `metadatatypes.html#UsageContext` for context-type semantics.
- **`SearchParameter` `*-context-type-value`** style search parameters on each affected resource are the intended replacement query path; verify their `expression` continues to traverse `useContext` after the change.
- **`Knowledge Repository` capability statement** (`capabilitystatement-knowledge-repository.xml`) advertises the now-removed search parameters and must be kept consistent.
- **`HL7/fhir-cross-version`** (out-of-scope repo) — R4 conversion maps for ActivityDefinition / EventDefinition / Library / PlanDefinition need the `topic → useContext` mapping rule per the resolution.
