## Description

PlanDefinition references several value sets that are potential THO migration candidates. The resolution does **not** migrate to THO; instead, the agreed approach is to clearly mark them as example/experimental content so they aren't mistaken for normative terminology.

### Approved Resolution

> "Agreed, mark the ValueSets and CodeSystems involved **experimental**, and add **Example** to the name, title, id, and url for each."

## Feature Proposal

### Renaming Scheme

For each artifact, apply the following transformation:

| Property | Current Pattern | New Pattern |
|----------|----------------|-------------|
| `id` | `action-code` | `action-code-example` |
| `name` | `ActionCode` | `ActionCodeExample` |
| `title` | `Action Code` | `Action Code Example` |
| `url` | `http://hl7.org/fhir/action-code` | `http://hl7.org/fhir/action-code-example` |
| `experimental` | `false` | `true` |

The same pattern applies to all three CodeSystem/ValueSet pairs (`action-code`, `action-reason-code`, `action-participant-function`).

## Detailed Implementation Plan

### Step 1: Rename and Update CodeSystem Files

For each of the three CodeSystems, create the new file (renamed with `-example` suffix) and delete the old file. Within each file apply all five changes (id, url, name, title, experimental).

**Example — `codesystem-action-code.xml` → `codesystem-action-code-example.xml`:**

```xml
<!-- BEFORE -->
<id value="action-code"/>
<url value="http://hl7.org/fhir/action-code"/>
<name value="ActionCode"/>
<title value="Action Code"/>
<experimental value="false"/>
<valueSet value="http://hl7.org/fhir/ValueSet/action-code"/>

<!-- AFTER -->
<id value="action-code-example"/>
<url value="http://hl7.org/fhir/action-code-example"/>
<name value="ActionCodeExample"/>
<title value="Action Code Example"/>
<experimental value="true"/>
<valueSet value="http://hl7.org/fhir/ValueSet/action-code-example"/>
```

Repeat analogously for:
- `codesystem-action-reason-code.xml` → `codesystem-action-reason-code-example.xml`
- `codesystem-action-participant-function.xml` → `codesystem-action-participant-function-example.xml`

### Step 2: Rename and Update ValueSet Files

For each of the three ValueSets, create the new file (renamed with `-example` suffix) and delete the old file. Update id, url, name, title, experimental, and the `compose/include/system` reference.

**Example — `valueset-action-code.xml` → `valueset-action-code-example.xml`:**

```xml
<!-- BEFORE -->
<id value="action-code"/>
<url value="http://hl7.org/fhir/ValueSet/action-code"/>
<name value="ActionCode"/>
<title value="Action Code"/>
<experimental value="false"/>
<compose>
  <include>
    <system value="http://hl7.org/fhir/action-code"/>
  </include>
</compose>

<!-- AFTER -->
<id value="action-code-example"/>
<url value="http://hl7.org/fhir/ValueSet/action-code-example"/>
<name value="ActionCodeExample"/>
<title value="Action Code Example"/>
<experimental value="true"/>
<compose>
  <include>
    <system value="http://hl7.org/fhir/action-code-example"/>
  </include>
</compose>
```

Repeat analogously for:
- `valueset-action-reason-code.xml` → `valueset-action-reason-code-example.xml`

## Risks and Considerations

1. **Breaking change for downstream consumers**: Changing the canonical URLs of CodeSystems and ValueSets is a breaking change. Any implementation guides, profiles, or implementations that reference `http://hl7.org/fhir/ValueSet/action-code` (etc.) will need to update. Since these are example-strength bindings, the practical impact is limited.

2. **Sibling ticket coordination**: The three CodeSystem/ValueSet artifacts are shared across PlanDefinition, RequestOrchestration, and ActivityDefinition. All binding reference updates should be applied atomically in a single commit/PR to avoid build failures.

3. **OID stability**: The OID values in `oids.ini` should remain unchanged — only the URL keys are updated.

4. **SVG diagrams**: The SVG files under `source/plandefinition/`, `source/requestorchestration/`, and `source/activitydefinition/` embed value set references. These are typically auto-generated during the build and should not require manual editing, but should be verified.
