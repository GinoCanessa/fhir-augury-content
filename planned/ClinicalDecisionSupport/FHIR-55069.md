## Feature Proposal

### Problem Statement

The PlanDefinition module ships three coded element bindings — `PlanDefinition.action.code`,
`PlanDefinition.action.reason`, and `PlanDefinition.action.participant.function` — that are
backed by ValueSets/CodeSystems authored in the core specification (`action-code`,
`action-reason-code`, `action-participant-function`). Each of these is presently marked
`experimental="false"` and `status="active"`, with descriptions that explicitly state they
"provide examples of …" the codes in question. The bindings are `example`-strength only.

In a Vocabulary review of PlanDefinition the comment thread (Comment 35, Lloyd McKenzie)
identified these as candidates for migration to the Terminology Harmonization Organization
(THO). Migration to THO is not required, but because the value sets are illustrative rather
than normative, marking them as not-experimental and giving them production-style identifiers
overstates their authority and complicates downstream reuse (e.g., CDA / V2 implementers
expecting THO-managed terminology). The CDS work group accepted a lighter-weight fix that
keeps the codes in core but explicitly brands them as examples.

### Proposed Change

For each of the three CodeSystem/ValueSet pairs, modify the resource as follows. The
proposed renamed identifiers below follow the existing FHIR core "example-…" convention used
elsewhere in the spec; if the work group prefers an alternative spelling (e.g.,
`action-example-code`, `Example` suffix rather than prefix) the same edits apply with that
spelling substituted.

### Design Rationale

The work group considered, but did not adopt, full migration of these value sets to THO. The
"Persuasive" resolution chooses the lighter intervention: keep the artifacts in core but
brand them honestly. Three signals already in the artifacts support this:

- The descriptions read "Provides examples of …" — `experimental=true` aligns the metadata
  with that prose.
- All three bindings on PlanDefinition / ActivityDefinition / RequestOrchestration are
  `example`-strength, so renaming the canonical URLs has no normative impact on conformance.
- The "Example…" prefix on `id`/`url`/`name`/`title` matches the convention used elsewhere
  in the spec for illustrative-only terminology and removes the implication that these are
  authoritative HL7 code systems suitable for direct V2/CDA reuse.

Migration to THO remains an option for a future ticket; this resolution intentionally keeps
the artifacts in `HL7/fhir` so no cross-repo coordination is required now.

### Breaking Changes

This change is **technically breaking for consumers who hard-code the old canonical URLs**,
even though the bindings themselves are `example`-strength:

- The CodeSystem URLs `http://hl7.org/fhir/action-code`, `http://hl7.org/fhir/action-reason-code`,
  and `http://hl7.org/fhir/action-participant-function` are removed from the core spec at
  these old URLs.
- The ValueSet canonicals `http://hl7.org/fhir/ValueSet/action-code` (and the two siblings)
  are likewise removed.
- The id-based file names are renamed (id is the publisher's filename driver); downloadable
  artifact URLs (e.g., `valueset-action-code.json`) change.

Because all three are FMM 1 / standards-status `informative`, the breakage envelope is
narrow, but it should be called out in the IG-cross-version notes and the PlanDefinition /
ActivityDefinition / RequestOrchestration change logs. The `experimental=true` flip is
metadata-only and not breaking.

No element cardinalities, types, search parameters, invariants, or required-strength
bindings change.
