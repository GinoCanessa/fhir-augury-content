## Description

The EvidenceTableReportPackage Profile is to be removed entirely. The ParticipantFlowReport, BaselineMeasureReport, and OutcomeMeasureReport Profiles are to be simplified and rebased on EvidenceReport (a profile of the Composition resource) instead of deriving from EvidenceTableReportPackage.

### Background & Motivation

The predecessor ticket [FHIR-50995](https://jira.hl7.org/browse/FHIR-50995) (Applied) explains the rationale:

- The three report profiles (ParticipantFlowReport, BaselineMeasureReport, OutcomeMeasureReport) were all derived from EvidenceTableReportPackage, which used a complex nested-section model: each results section (per VariableDefinition) contained subsection slices defined by "type of group the evidence is about," each with an entry referencing a single Evidence Resource.
- This model was **overly complex**, did not support data conversion for large datasets (e.g. ClinicalTrials.gov records with results), and redundantly structured group-type data already captured in `Evidence.variableDefinition`.
- The **simplified approach** removes section nesting; instead, each results section uses a flat `entry` list referencing Evidence Resources for the same VariableDefinition (role: "outcome"). The `variableDefinition` roles of "population" and "exposure" on the Evidence resources themselves determine the group type.
- This approach was pre-applied and tested with ClinicalTrials.gov data conversions using many patterns (single group, 2–3 groups, comparative/non-comparative evidence, missing data, etc.).

The related ticket [FHIR-46135](https://jira.hl7.org/browse/FHIR-46135) (Applied) created OutcomeMeasureReport and reorganized profile derivation hierarchies.

The companion ticket [FHIR-55701](https://jira.hl7.org/browse/FHIR-55701) (Applied) modifies ComparativeEvidenceReport to remove its Baseline Measures and Participant Flow sections (they now reference standalone report profiles) and restructures its sections.

## Feature Proposal

### Profile Hierarchy Change

**Before (current deprecated structure):**
```
Composition
  └── EvidenceReport
        └── EvidenceReportPackage
              └── EvidenceTableReportPackage     ← REMOVE
                    ├── ParticipantFlowReport     ← complex, nested sections
                    ├── BaselineMeasureReport     ← complex, nested sections
                    └── OutcomeMeasureReport      ← complex, nested sections
```

**After (proposed simplified structure):**
```
Composition
  └── EvidenceReport                             ← direct base
        ├── ParticipantFlowReport                ← simplified, flat entries
        ├── BaselineMeasureReport                ← simplified, flat entries
        └── OutcomeMeasureReport                 ← simplified, flat entries
```

### Structural Simplification

The key structural change in each of the three report profiles:

| Aspect | Before (EvidenceTableReportPackage-based) | After (EvidenceReport-based) |
|--------|------------------------------------------|------------------------------|
| **Base profile** | EvidenceTableReportPackage | EvidenceReport |
| **Results sections** | Nested: section → subsection slices by group type → entry (1 Evidence) | Flat: section → entry (0..\* Evidence Resources) |
| **Group identification** | Encoded in section slice structure | Encoded in `Evidence.variableDefinition` (roles: population, exposure) |
| **Complexity** | High (multiple nesting levels) | Low (flat entry list) |
| **Large dataset support** | Poor (doesn't scale for ClinicalTrials.gov) | Good (tested with many patterns) |

## Impact Analysis

### Scope

This change is **profile-level only** — it does not modify the FHIR core Composition resource or the base EvidenceReport profile definition. It affects:

1. **4 StructureDefinitions** — 1 removal (EvidenceTableReportPackage), 3 rewrites (ParticipantFlowReport, BaselineMeasureReport, OutcomeMeasureReport)
2. **IG manifest** — `FHIR-ebm-incubator.xml` artifact entries
3. **Documentation** — `link-list.md` references
4. **Examples** — any Composition instances conforming to these profiles (e.g., `Composition/343007`)

### Risk Assessment

| Risk | Level | Mitigation |

| Build/validation failures | **Medium** | New StructureDefinitions must correctly profile Composition and validate against FHIR R6 Composition constraints |

## Implementation Plan

> **Note:** Given the current repository state (all EBM profiles deprecated, IG restructured for Citation/Contributorship only), the implementation may need to occur in a separate branch or repository targeting the main EBMonFHIR IG rather than the incubator. The plan below covers the logical changes required regardless of where they land.

### Step 1: Remove EvidenceTableReportPackage Profile

**Action:** Ensure `EvidenceTableReportPackage` StructureDefinition is fully removed.

- **Verify** no StructureDefinition source file exists (confirmed: none found in filesystem)
- **Update** `FHIR-ebm-incubator.xml` line 282: keep `deprecated="true"` flag on the artifact entry (for publication history tracking) — no further action needed since it is already deprecated
- **Verify** no other profiles reference `evidence-table-report-package` as a base (confirmed: only the three report profiles did, and they are being rebased)
- **Remove** the `[EvidenceTableReportPackage]` reference from `link-list.md` if present (not currently present — no action needed)

**Files:** `FHIR-ebm-incubator.xml` (no change needed — already deprecated)

### Step 2: Create Simplified ParticipantFlowReport Profile

**Action:** Create a new StructureDefinition for ParticipantFlowReport based on EvidenceReport (Composition).

**New file:** `input/resources/ebm/StructureDefinition-participant-flow-report.xml` (or FSH equivalent)

**Profile definition:**
- `baseDefinition`: URL of EvidenceReport profile (e.g., `http://hl7.org/fhir/uv/ebm/StructureDefinition/evidence-report`)
- `type`: Composition
- **Sections:**
  - Results section with `entry` (0..\*) referencing Evidence resources (ParticipantFlowEvidence)
  - No nested subsection slices per group type
  - Section focus references a VariableDefinition (ParticipantFlowEvidenceVariable)
- Remove all EvidenceTableReportPackage-inherited section constraints

**IG manifest:** Add non-deprecated artifact entry in `FHIR-ebm-incubator.xml`

### Step 3: Create Simplified BaselineMeasureReport Profile

**Action:** Create a new StructureDefinition for BaselineMeasureReport based on EvidenceReport (Composition).

**New file:** `input/resources/ebm/StructureDefinition-baseline-measure-report.xml` (or FSH equivalent)

**Profile definition:**
- `baseDefinition`: URL of EvidenceReport profile
- `type`: Composition
- **Sections:**
  - Results section with `entry` (0..\*) referencing Evidence resources (BaselineMeasureEvidence)
  - No nested subsection slices per group type
  - Section focus references a VariableDefinition
- Remove all EvidenceTableReportPackage-inherited section constraints

**IG manifest:** Add non-deprecated artifact entry in `FHIR-ebm-incubator.xml`

### Step 4: Create Simplified OutcomeMeasureReport Profile

**Action:** Create a new StructureDefinition for OutcomeMeasureReport based on EvidenceReport (Composition).

**New file:** `input/resources/ebm/StructureDefinition-outcome-measure-report.xml` (or FSH equivalent)

## Open Questions

1. **Repository location:** The incubator IG has been restructured to focus on Citation and Contributorship only. Should these three report profiles be created in this incubator repo, or should they be created in the main EBMonFHIR IG repository (`HL7/ebm`)? The current index.md explicitly states this is for "Additional Resources beyond those in FHIR R6."

2. **EvidenceReport base profile:** The EvidenceReport profile itself is also deprecated in this repo. If the three reports derive from it, the EvidenceReport profile (or its canonical URL from the main EBMonFHIR IG) must be available as a dependency.

3. **Profile format:** The current repo uses raw XML StructureDefinitions (not FSH) for its active resources (Citation, Contributorship). Should new profiles follow the same format?

4. **Example instances:** Should new simplified example instances be created to demonstrate the three report profiles with ClinicalTrials.gov-style data?
