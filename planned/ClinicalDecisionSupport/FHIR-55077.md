## Resolution Summary

The committee accepted the ticket as **Persuasive** with the following directive (verbatim from the resolution description):

> "Agreed, mark the ValueSets and CodeSystems involved experimental, and add Example to the name, title, id, and url for each."

The two artifact pairs in scope are:

- `ValueSet/risk-assessment-method` + `CodeSystem/risk-assessment-method` (bound to `RiskAssessment.method`)
- `ValueSet/risk-assessment-outcome` + `CodeSystem/risk-assessment-outcome` (bound to `RiskAssessment.prediction.outcome`)

The migration to THO that the original ticket asked about was *not* adopted; the committee instead chose to clearly flag these as example-only content so that downstream specs (CDA, V2, IGs) do not treat them as normative HL7-managed terminology.

Note on current state in the clone (`HL7/fhir` @ `db79dcd`): all four artifacts already carry `<experimental value="true"/>`, so the `experimental` half of the resolution is already in place. The outstanding work is the **rename** (id, url, name, title) on all four artifacts plus updating the two `RiskAssessment` element bindings that point at the old `ValueSet/risk-assessment-method` and `ValueSet/risk-assessment-outcome` URLs.

## Feature Proposal

### Problem Statement

`RiskAssessment.method` (preferred binding) and `RiskAssessment.prediction.outcome` (example binding) are currently bound to two FHIR-published, CDS-owned value sets — `ValueSet/risk-assessment-method` and `ValueSet/risk-assessment-outcome` — each backed by a CodeSystem with the same id under `http://hl7.org/fhir/`. Lloyd McKenzie's ticket flagged these as "potentially re-useable THO-candidate value sets": their definitions could plausibly evolve over time and could be re-used by CDA / V2 / other HL7 product families, which would normally argue for migration to THO.

The CDS work group (re-routed by Michelle Miller in comment 65894 because RiskAssessment is owned by CDS, not PC) declined the THO migration but agreed the artifacts should be unambiguously labelled as **example-only**. The current id / url / name / title give no surface signal that they are illustrative rather than authoritative; the only signal is the value of `<experimental>` and the body of the `<description>` text. The resolution closes that gap.

### Proposed Change

Rename the four artifacts so that the word "Example" appears in each of `id`, `url`, `name`, and `title`, and update the two RiskAssessment element bindings to point at the renamed ValueSets. Recommended target identifiers (using the `-example` suffix convention seen elsewhere in the spec, e.g. `valueset-audit-event-type-example.xml` / id `audit-event-type-example` / name `AuditEventIDExample` / title "Example Audit Event ID"):

| Old | New |
|---|---|
| id `risk-assessment-method` | id `risk-assessment-method-example` |
| url `http://hl7.org/fhir/ValueSet/risk-assessment-method` | url `http://hl7.org/fhir/ValueSet/risk-assessment-method-example` |
| url `http://hl7.org/fhir/risk-assessment-method` (CodeSystem) | url `http://hl7.org/fhir/risk-assessment-method-example` |
| name `RiskAssessmentMethod` | name `RiskAssessmentMethodExample` |
| title `Risk Assessment Method` | title `Example Risk Assessment Method` |
| id `risk-assessment-outcome` | id `risk-assessment-outcome-example` |
| url `http://hl7.org/fhir/ValueSet/risk-assessment-outcome` | url `http://hl7.org/fhir/ValueSet/risk-assessment-outcome-example` |
| url `http://hl7.org/fhir/risk-assessment-outcome` (CodeSystem) | url `http://hl7.org/fhir/risk-assessment-outcome-example` |
| name `RiskAssessmentOutcome` | name `RiskAssessmentOutcomeExample` |
| title `Risk Assessment Outcome` | title `Example Risk Assessment Outcome` |

In each CodeSystem file the inner `<valueSet value="…"/>` back-pointer must be updated to the new ValueSet url. In each ValueSet file the `<compose><include><system value="…"/>` reference must be updated to the new CodeSystem url. In `structuredefinition-RiskAssessment.xml` three bindings must be repointed:

- Element `RiskAssessment.method` (line ~211) — `<valueSet value="http://hl7.org/fhir/ValueSet/risk-assessment-method"/>` → new url.
- Element `RiskAssessment.method` differential entry at line ~238 (a second binding block in the same SD) — same update.
- Element `RiskAssessment.prediction.outcome` (line ~465) — `<valueSet value="http://hl7.org/fhir/ValueSet/risk-assessment-outcome"/>` → new url.

Confirm `<experimental value="true"/>` remains set on all four artifacts (already true in `db79dcd`).

### Design Rationale

- The resolution explicitly chose the lighter-weight "rename + experimental" approach over a THO migration, on the basis that adding "Example" to the user-visible identifiers gives implementers an unmistakable signal without forcing churn on a published canonical url that is expected to be effectively a placeholder.
- The `-example` suffix on the id/url plus the `Example` token at the end of `name` and the start of `title` matches the convention already established in the spec (see `audit-event-type-example`, `audit-entity-detail-type-example`, `audit-event-outcome-detail-example`, `object-role-example`), which is the closest precedent. An alternative `example-…` prefix is also used in the spec (e.g. `example-characteristic-codes`, `example-cpt-all`), but that variant is mostly used for value sets that are themselves illustrative inputs to a feature rather than for "the published default value set is an example, not a curated terminology" — which is the case here.
- Keeping the binding strength (`preferred` for `.method`, `example` for `.prediction.outcome`) unchanged respects the resolution wording, which limits itself to renaming and the `experimental` flag.
- The CodeSystem urls (`http://hl7.org/fhir/risk-assessment-method` and `http://hl7.org/fhir/risk-assessment-outcome`) are HL7-published canonicals; renaming them is a breaking change for any external implementer who has hard-coded the old urls. The committee accepted that consequence implicitly by mandating that "Example" appear in the **url** of each artifact.

- `name` change (computable identifier) for all four artifacts.

The bindings on `RiskAssessment.method` (preferred) and `RiskAssessment.prediction.outcome` (example) are not themselves cardinality- or strength-changing, so the resource shape is unchanged. ### Open Questions

- **Suffix vs prefix convention.** The spec uses both `valueset-<x>-example.xml` (e.g. `audit-event-type-example`) and `valueset-example-<x>.xml` (e.g. `example-characteristic-codes`). This plan recommends the suffix form because the AuditEvent precedent is the closest semantic match, but CDS should confirm before merging.
- **File rename.** Per FhirCore convention the on-disk filename usually mirrors the resource id. The plan recommends renaming the four files in lockstep with the id changes (Tasks 1–4) and updating any publisher manifests accordingly (Task 7), but if the publisher harness has hard-coded filename expectations the safer fall-back is to leave the filenames and only change the inner `<id>` — at the cost of an id/filename mismatch.
- **Snapshot-area edits.** The briefing's gotcha about only editing `<differential>` content in StructureDefinitions is taken at face value here. If the local publisher pipeline is *not* run before the PR is opened, reviewers will see stale `<snapshot>` content; ensure the standard publisher run regenerates `<snapshot>` after Task 5.
- **Sibling tickets.** This ticket is one resource's worth of "Comment 43 — potentially re-useable THO-candidate value sets" follow-up. If CDS or PMs want sibling tickets handled together (e.g. so the `R6-Normative` lock window covers them all), surface those in the same PR; the cross-reference query for FHIR-55077 returned no related Jira tickets, so any grouping must be coordinated externally.
- **Redirect / page-slug strategy.** Renaming a published canonical changes the slug under `https://hl7.org/fhir/`. Confirm with the publisher team whether redirect entries are required for the four renamed pages (`valueset-risk-assessment-method.html`, `valueset-risk-assessment-outcome.html`, and the two CodeSystem pages).
