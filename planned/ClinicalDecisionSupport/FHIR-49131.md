## 1. Summary

The `artifact-approvalDate`, `artifact-lastReviewDate`, and `artifact-effectivePeriod` extensions (FMM1, CDS work group) are duplicates of `resource-approvalDate`, `resource-lastReviewDate`, and `resource-effectivePeriod` (FMM3, FHIR Infrastructure work group). The artifact-level extensions should be deprecated in favor of the resource-level extensions.

The resource-level extensions currently have a limited context-of-use list. Their contexts must be expanded to cover all the resource types that the artifact-level extensions support, so that consumers of the artifact-level extensions can migrate to the resource-level equivalents.

Additionally, Composition, ResearchStudy, and List must be added to all three resource-level extensions.

---

## 2. Feature Proposal

### 2a. Deprecate the artifact-level extensions

Mark the following three extension definitions as **retired** and add deprecation notes pointing to the resource-level replacements:

- `artifact-approvalDate` → use `resource-approvalDate`
- `artifact-lastReviewDate` → use `resource-lastReviewDate`
- `artifact-effectivePeriod` → use `resource-effectivePeriod`

### 2b. Expand resource-level extension contexts

Update the context-of-use for each resource-level extension to include all resource types listed in the resolution, ensuring full coverage of the deprecated artifact-level extensions' contexts.

## 5. Open Questions / Considerations

1. **Version-specific annotations for new contexts:** The artifact-level extensions use `startFhirVersion` annotations for resources like ClinicalUseDefinition (4.3), ConditionDefinition (5.0), DeviceDefinition (4.0), ExampleScenario (4.0), MedicationKnowledge (4.0), SubstanceDefinition (4.3), SubscriptionTopic (4.3), TerminologyCapabilities (4.0). The resource-level extensions should replicate these same version constraints. For resources where the element was introduced natively in R5 (CodeSystem, ConceptMap, NamingSystem, ObservationDefinition, SpecimenDefinition, SubscriptionTopic, ValueSet), use `endFhirVersion: 4.3` to indicate the extension is only applicable pre-R5.

2. **SubscriptionTopic dual annotation:** SubscriptionTopic was introduced in R4B (startFhirVersion 4.3) but also got native approvalDate/lastReviewDate/effectivePeriod in R5 (endFhirVersion 4.3). Since start and end are the same version, the extension would apply only for FHIR 4.3 (R4B) for this resource. Verify this is the intended behavior.

3. **Existing `endFhirVersion: 4.3` on CodeSystem, ConceptMap, NamingSystem, ValueSet:** The resource-level extensions already have these restrictions. They should be preserved as-is since the resolution confirms these resources got native elements in R5.

4. **EBMonFHIR IG impact:** Per the FHIR-53078 comment, EBMonFHIR IG uses the artifact-level extensions. Downstream IGs should be notified to migrate to the resource-level equivalents.

5. **CRMI IG companion change:** FHIR-49132 (already Applied, PR #76 merged) updates the CRMI publishable profiles to use resource-level extensions. This confirms the fhir-extensions change is a prerequisite/parallel effort.
