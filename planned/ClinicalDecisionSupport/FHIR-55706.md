## Description

The ticket requests that the `DichotomousIntendedOutcome` and `ParticipantFlowEvidenceVariable` profiles of `EvidenceVariable` set the `handling` element value to `#boolean` instead of `#dichotomous`.

**Referenced URL:** <https://build.fhir.org/ig/HL7/ebm/en/StructureDefinition-dichotomous-intended-outcome.html>

### Resolution Description

> Set the handling element value to 'boolean' in the DichotomousIntendedOutcome and ParticipantFlowEvidenceVariable Profiles of EvidenceVariable.

### Approval

Block vote approved by CDS at the [2026-02-11 meeting](https://confluence.hl7.org/spaces/CDS/pages/413252843/2026-02-11+-+Meeting+Agenda).

---

## Feature Proposal

### Goal

Change the fixed value of `EvidenceVariable.handling` from `#dichotomous` to `#boolean` in two EvidenceVariable profiles: **DichotomousIntendedOutcome** and **ParticipantFlowEvidenceVariable**. Both profiles represent dichotomous outcomes/variables and should use the `boolean` handling code (which means "The variable is dichotomous, expressed as a boolean") rather than the `dichotomous` code (which means "The variable is dichotomous, expressed as a pair of values").

### Rationale

- The `boolean` handling code is semantically more appropriate for these profiles, as their outcomes are expressed as boolean (true/false) values rather than as a pair of named dichotomous categories.
- This aligns with the broader evolution of `EvidenceVariable.handling` (see related ticket FHIR-51456 which changed handling from a `code` to a `CodeableConcept` datatype with an extensible value set).

### Profile Hierarchy

Both affected profiles share the same parent chain:
```
EvidenceVariable (FHIR R6 base)
  └── VariableDefinition (p-variable-definition.fsh)
        ├── DichotomousIntendedOutcome (p-dichotomous-intended-outcome.fsh)
        ├── ParticipantFlowEvidenceVariable (p-participant-flow-evidence-variable.fsh)
        ├── GroupAssignment (p-group-assignment.fsh)
        └── ParticipantFlowReasonEvidenceVariable (p-participant-flow-reason-evidence-variable.fsh)
```

The parent `VariableDefinition` profile constrains `handling 1..1` (required), so each child profile must specify a value.
