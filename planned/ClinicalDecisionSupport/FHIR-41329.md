## Description

The ticket requests better support for canonical resolution by defining operations or search patterns:

1. **System-level search by URL and version** — Given any canonical URL (with or without version), search at the server root and return any resources that match that canonical reference across all canonical resource types.
2. **Determine whether a resource is "canonical"** — Given a resource, provide a mechanism to know whether it is a canonical resource.
3. **Package identification** *(from comment by Bryn Rhodes)* — Support a mechanism to answer "What package (i.e., implementation guide) is this canonical defined in?"

The ticket notes that some servers (e.g., Firely) already support root-level search by `url`, while others (e.g., HAPI FHIR) do not.

## Feature Proposal

### Objective

Define and document search capabilities for canonical resolution in the CRMI IG, enabling clients to:

- Resolve canonical references via system-level search across all canonical resource types
- Determine which resource types are canonical
- Identify the source package/IG for a canonical resource

### Approach

Since the resolution is "define search capabilities as suggested," the implementation should formalize search patterns that some servers already support, making them normative expectations in the CRMI IG's Artifact Repository Service capability requirements.

**Three capabilities to define:**

1. **System-level canonical search**: Document and formalize the use of `GET [base]?url=<canonical-url>&version=<version>` as a cross-resource-type search for canonical resources. This leverages FHIR Core's existing system-level search (`GET [base]?params...`) with the `url` and `version` search parameters already defined on the `CanonicalResource` interface.

2. **Canonical resource identification**: Document the complete list of canonical resource types (those implementing the `CanonicalResource` interface) and how clients can discover which types a server supports via CapabilityStatement introspection.

3. **Package source identification**: Leverage the existing `package-source` extension (`http://hl7.org/fhir/StructureDefinition/package-source`) defined in fhir-extensions to indicate the package/IG in which a canonical is defined.

## Detailed Implementation Plan

### Change 1: Define System-Level Canonical Search in CRMI IG

**Repository**: `HL7/fhir-uv-crmi` (not in local cache)

**What**: Add a section to the Artifact Repository Service page defining normative expectations for system-level canonical resolution search.

**Details**:

The CRMI Artifact Repository Service (referenced from `clinicalreasoning-knowledge-artifact-distribution.html` at `https://hl7.org/fhir/uv/crmi/artifact-repository-service.html`) should define:

1. **System-level search by `url`**: Servers conforming to the Artifact Repository Service **SHALL** support searching at the system level using the `url` search parameter:
   ```
   GET [base]?url=http://example.org/fhir/ValueSet/example
   ```
   This returns a Bundle containing any canonical resources across all supported resource types that match the given URL.

2. **System-level search by `url` and `version`**: Servers **SHALL** support combining `url` with `version`:
   ```
   GET [base]?url=http://example.org/fhir/ValueSet/example&version=1.0.0
   ```

3. **Filtering by `_type`**: Servers **SHOULD** support the `_type` parameter to narrow results:
   ```
   GET [base]?url=http://example.org/fhir/ValueSet/example&_type=ValueSet
   ```

4. **Versioned canonical syntax**: Servers **SHOULD** support the `[url]|[version]` syntax on the `url` parameter per FHIR Core search semantics:
   ```
   GET [base]?url=http://example.org/fhir/ValueSet/example|1.0.0
   ```

**Rationale**: The `url` search parameter is defined on CanonicalResource and inherited by all canonical resource types. System-level search is supported by FHIR Core. This formalizes the expectation that Artifact Repository servers support this combination, which some servers (Firely) already do.

### Change 2: Document Canonical Resource Identification in CRMI IG

**Repository**: `HL7/fhir-uv-crmi` (not in local cache)

**What**: Add documentation explaining how to determine whether a resource type is "canonical" — answering the ticket's question "Given a resource, how do I know whether it is a canonical resource?"

**Details**:

Add a section (or extend existing guidance) covering:

1. **Interface-based identification**: A resource type is canonical if it implements the `CanonicalResource` interface. This can be determined by:
   - Checking the StructureDefinition for the resource type to see if it lists `http://hl7.org/fhir/StructureDefinition/CanonicalResource` as an interface
   - Consulting the canonical resource type list in the FHIR specification

2. **Runtime identification**: At runtime, a canonical resource can be identified by the presence of the `url` element (the canonical URL). The `_type` parameter combined with a list of known canonical resource types enables discovery.

3. **CapabilityStatement introspection**: A server's CapabilityStatement indicates which resource types it supports. Cross-referencing with the canonical resource type list tells clients which canonical types are available.

4. **Complete canonical resource type listing**: Include the full list of 28 canonical resource types from FHIR R6.

### Change 3: Reference Package Source Extension for Package Identification

**Repository**: `HL7/fhir-uv-crmi` (not in local cache)

**What**: Add guidance on using the `package-source` extension to answer "What package/IG is this canonical defined in?"

The CRMI IG should:

2. **Document** how clients can query for this information
3. **Reference** FHIR-38990's resolution for the design context

## Relationship to FHIR-49674 ($resolve Operation)

FHIR-49674 (Applied, Persuasive) defines a `$resolve` operation:
```
GET [base]/$resolve?type=w&canonical=x&source-url=y&source-version=z&duplicates=error
```

This is **complementary** to the search-based approach in FHIR-41329:
- **Search approach** (FHIR-41329): Uses standard FHIR search semantics, returns a Bundle, works with existing search infrastructure
- **Operation approach** (FHIR-49674): Purpose-built operation with additional context (source resource info, duplicate handling), returns a single resource

Both should be defined in the CRMI IG. The search approach is the "basic" mechanism; the `$resolve` operation provides richer semantics.

## Testing Considerations

1. **System-level search**: Verify `GET [base]?url=<canonical-url>` returns matching resources across all canonical types
2. **Version filtering**: Verify `GET [base]?url=<url>&version=<ver>` correctly filters by version
3. **Type filtering**: Verify `GET [base]?url=<url>&_type=ValueSet` narrows to specific resource types
4. **Versioned URL syntax**: Verify `GET [base]?url=<url>|<version>` works with version algorithm matching
5. **Package source**: Verify `package-source` extension is populated on returned resources when available
6. **Empty results**: Verify graceful handling when no canonical matches
