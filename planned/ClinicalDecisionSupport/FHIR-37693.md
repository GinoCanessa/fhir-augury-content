## 1. Summary

The ticket requests a clear, standalone paragraph defining what is meant by a "remote" CDS service in the PDDI CDS Implementation Guide. The three points to clarify are:

1. A remote CDS service is **not a component of the EHR**.
2. It **uses API and web standards for communication**.
3. It **could be a servlet container hosted within the organization's data architecture/firewall or hosted offsite**. In either case, **HIPAA rules apply**.

### Resolution Description

> "Agreed, add clarification of remote CDS service as suggested"

**Verdict: Requires implementation** (Persuasive).

---

## 2. Feature Proposal

### Current State

The `index.md` page of the PDDI CDS IG (under **Context → Assumptions → Technical**) already contains a sentence addressing these points, but it is embedded as a subordinate "Note that..." clause within a dense paragraph about technical assumptions:

> "Note that the remote service would not a component of the EHR, would use API and web standards for communication, and could be a servlet container hosted within the organization's data architecture / firewall or hosted offsite. HIPAA privacy and data protection rules would apply to all aspects of the interaction between the EHR and the CDS service."

**Issues with current text:**
- Buried inside a long paragraph rather than standing as a distinct, prominent definition
- Contains a grammatical error: "would not **a** component" (missing "be")
- Uses future conditional tense ("would") rather than direct declarative language

### Proposed Change

Extract and rewrite the "remote CDS service" definition as a **clear, standalone paragraph** (or small subsection) immediately after the sentence that introduces the concept of CDS as a "real-time remote service." The new text should:

1. Use direct, declarative language defining "remote" in this context
2. Present each characteristic as a distinct point (bulleted list or structured paragraph)
3. Fix the grammatical error
4. Explicitly reference HIPAA applicability regardless of hosting location

## 5. Detailed Implementation Plan

### Step 1: Modify `input/pagecontent/index.md`

**Current text** (Technical Assumptions paragraph):

```markdown
An important requirement of CDS is that it provide clinically relevant information to the clinician 
at the right time. This implementation guide assumes that CDS is provided as a real-time remote 
service which the EHR (client) subscribes to at various points in the workflow of clinicians. The 
service requests clinical decision support based on the current patient context. The EHR client 
sends relevant contextual information as part of the request and receives clinically relevant 
suggestions describing potential actions to be taken. The CDS service SHOULD run efficiently to 
meet clinician expectations and not interrupt from the clinical workflow. Delays in presenting CDS 
information may lead to unsuccessful CDS adoption and clinician frustration. Note that the remote 
service would not a component of the EHR, would use API and web standards for communication, and 
could be a servlet container hosted within the organization's data architecture / firewall or 
hosted offsite. HIPAA privacy and data protection rules would apply to all aspects of the 
interaction between the EHR and the CDS service.
```

**Proposed replacement** – split into two paragraphs, with the second being the new standalone clarification:

```markdown
An important requirement of CDS is that it provide clinically relevant information to the clinician 
at the right time. This implementation guide assumes that CDS is provided as a real-time remote 
service which the EHR (client) subscribes to at various points in the workflow of clinicians. The 
service requests clinical decision support based on the current patient context. The EHR client 
sends relevant contextual information as part of the request and receives clinically relevant 
suggestions describing potential actions to be taken. The CDS service SHOULD run efficiently to 
meet clinician expectations and not interrupt from the clinical workflow. Delays in presenting CDS 
information may lead to unsuccessful CDS adoption and clinician frustration.

In this implementation guide, a "remote" CDS service is one that:

- Is **not** a component of the EHR itself, but rather a separate service that the EHR communicates with
- Uses **API and web standards** for communication between the EHR client and the CDS service
- Could be a **servlet container or similar application hosted within the organization's data architecture and firewall**, or could be **hosted offsite** by a third party

Regardless of where the CDS service is hosted, **HIPAA privacy and data protection rules apply** to all aspects of the interaction between the EHR and the CDS service.
```
