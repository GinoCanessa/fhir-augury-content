## Resolution Summary

The Pharmacy WG agreed (resolution: **Persuasive**) that:

> "We have agreed to remove MedicationKnowledge resource and move the
> elements to FormularyItem or other resources as applicable."

Concretely this means: (1) **delete the `MedicationKnowledge` resource**
from the FHIR Core specification, and (2) **redistribute its elements**
to `FormularyItem` (the primary destination), `Medication`, or other
existing pharmacy resources where they fit better. Only the
"resource removal + element redistribution" decision is recorded; the
exact per-element mapping was *not* itemised in the resolution and must
be agreed on the implementation tracker / Pharmacy WG call before any
SD edits land. Cross-references and the cost / packaging / regulatory /
storage / indication / monograph / classification subhierarchies are the
material that needs a new home.

## Feature Proposal

### Problem Statement

`MedicationKnowledge` was added to FHIR R4 as a broad container for
"things you know about a medication" (cost, packaging, regulatory
status, monographs, storage guidelines, administration / indication
guidelines, relationships to other knowledge, etc.). When EU national
medicines specifications (and cross-border use cases) were profiled
against R5/R6, the WG observed that:

- The same conceptual data is being represented variably across
  countries — there is no consistent "MedicationKnowledge" shape in real
  EU implementations.
- Most jurisdictions already capture the formulary / business view
  (price, coverage, substitution rules, status in the formulary)
  through `FormularyItem` and the definitional / pharmaceutical view
  (form, ingredient, strength, batch) through `Medication`.
- `MedicationKnowledge` therefore largely duplicates `Medication` and
  `FormularyItem`, and its presence encourages divergent profiling
  rather than a consistent core model.

The ticket also surfaces that `MedicationKnowledge` has carried a
"not completely stable" warning since DSTU (see FHIR-23116), has had
multiple rounds of redefinition (FHIR-25937, FHIR-24098, FHIR-42901,
FHIR-48651, FHIR-45284), and continues to need structural changes to
satisfy any one community — reinforcing the WG view that the resource
itself is the wrong abstraction.

### Proposed Change

Apply the resolution as a coordinated removal-and-redistribution change
inside `HL7/fhir`:

1. **Remove `MedicationKnowledge` from the FHIR Core build.**
   - Delete the entire `source/medicationknowledge/` authoring folder
     (StructureDefinition, search-params bundle, lists, examples,
     introduction, notes, fivews mapping, SVG, code systems and value
     sets that are scoped to MedicationKnowledge only — see Affected
     Files below for the enumerated set).
   - Remove the three `medicationknowledge=` registrations from
     `source/fhir.ini` (publisher resource list, `[business-version]`,
     `[normative-version]`-style buckets at lines 580 / 726 / 870 in
     the current `db79dcd` clone).
   - Remove the OID assignment from `source/oids.ini` (line 608).
   - Remove the `MedicationKnowledge` row from
     `source/resource-name-tracker.xml` (line 419).
   - Remove the `MedicationKnowledge` enumerator from
     `source/library/library-fhir-model-definition-content.xml` so the
     generated `fhir-types` ValueSet/CodeSystem no longer advertises it.
   - Add a deprecation / removal entry to `source/history.html`.

2. **Redistribute element semantics.** For each top-level
   `MedicationKnowledge` element observed in the current
   StructureDefinition (`code`, `status`, `identifier`,
   `relatedMedicationKnowledge`, `associatedMedication`, `productType`,
   `monograph`, `preparationInstruction`, `cost`, `monitoringProgram`,
   `indicationGuideline`, `medicineClassification`, `packaging`,
   `clinicalUseIssue`, `storageGuideline`, `regulatory`, `definitional`,
   `name`, `author`, `jurisdiction`), decide its destination:

   | MedicationKnowledge element | Proposed destination |
   |-----------------------------|----------------------|
   | `code`, `status`, `identifier`, `name`, `jurisdiction` | Already covered on `Medication` / `FormularyItem`; map to existing elements, no new structure. |
   | `cost`, `packaging`, `monitoringProgram`, `relatedMedicationKnowledge` (renamed → `relatedItem`) | Add to `FormularyItem` (formulary / business view). |
   | `definitional` (ingredient, drug characteristic, intended route, doseForm), `productType`, `preparationInstruction` | Migrate the still-missing pieces into `Medication` (most are already there; only `productType` and `preparationInstruction` are net-new on `Medication`). |
   | `monograph`, `medicineClassification`, `indicationGuideline`, `clinicalUseIssue`, `storageGuideline`, `regulatory`, `associatedMedication`, `author` | Move to `FormularyItem` (knowledge / regulatory view) **or** to a Pharmacy WG follow-up resource if FormularyItem owners reject scope. Each of these MUST be ticketed individually — see Open Questions. |

   For every migrated element preserve: short, definition, cardinality,
   type, and binding (strength + ValueSet URL). Where an element only
   moves (no semantic change), do not renumber differential paths that
   are unaffected.

3. **Move retained MedicationKnowledge ValueSets / CodeSystems** that
   are reused on the destination resource:
   - `valueset-medication-codes.xml`,
     `valueset-medication-form-codes.xml`,
     `valueset-medication-cost-category.xml`,
     `valueset-medicationknowledge-characteristic.xml`,
     `valueset-medicationknowledge-package-type.xml`,
     `codesystem-medication-cost-category.xml`
     → relocate under `source/formularyitem/` (or `source/medication/`
     for the "definitional" ones) and rename URLs to drop
     `medicationknowledge` from the canonical where the binding is no
     longer MK-specific.
   - Delete `valueset-medicationknowledge-status.xml`,
     `valueset-medication-knowledge-status.xml`, and
     `codesystem-medicationknowledge-status.xml` — `FormularyItem`
     already has its own `formularyitem-status` CodeSystem/ValueSet.

4. **Update narrative and cross-references.**
   - Edit `source/medications-module.html` to remove
     MedicationKnowledge from the Medication module diagram / table and
     describe the new "Medication + FormularyItem" pairing.
   - Edit `source/formularyitem/formularyitem-introduction.xml`,
     `formularyitem-notes.xml`, and `formularyitem-html.xml` to absorb
     the migrated narrative (cost, packaging, regulatory, monograph
     story).
   - Edit `source/medication/` introduction / notes to absorb anything
     that landed on `Medication`.
   - Add example coverage on FormularyItem (or update
     `formularyitem/formularyitemexample01.xml`) demonstrating the
     migrated cost / packaging / regulatory data.

5. **Search parameters.** For every search parameter on
   `MedicationKnowledge` (in
   `source/medicationknowledge/bundle-MedicationKnowledge-search-params.xml`),
   re-anchor it on the destination resource (`FormularyItem` /
   `Medication`) and add to that resource's search-params bundle. Any
   parameter that has no destination home is dropped.

6. **History / changelog.** Add a `breaking change` entry in
   `source/history.html` and the per-resource `_changelog.txt` files
   for `medication`, `formularyitem`, and (the now-deleted)
   `medicationknowledge`.

### Design Rationale

- The resolution is explicit ("remove MedicationKnowledge"), so the
  high-level shape is fixed by the WG vote — design choice is limited
  to the per-element destination.
- `Medication` is the canonical "what is this drug" resource and
  already carries `code`, `status`, `marketingAuthorizationHolder`,
  `doseForm`, `ingredient`, `batch`, `definition`. It is the natural
  home for `definitional`, `productType`, and
  `preparationInstruction` — they describe the intrinsic product, not
  formulary policy.
- `FormularyItem` is already the "what does *this* organisation /
  jurisdiction do with this medication" resource. Cost, packaging,
  monitoring program, monographs, indication / classification, and
  regulatory status are formulary / policy concerns, so they belong
  there. This keeps a clean *intrinsic vs. policy* split that EU
  national specs were already converging on, which is the original
  motivation for the ticket.
- Alternatives considered: (a) merge MK directly into `Medication`
  (rejected — would bloat `Medication` with formulary policy and
  recreate the EU divergence problem); (b) leave MK in place and just
  deprecate (rejected — the WG explicitly voted to remove, not
  deprecate-in-place; previous "stabilise MK" tickets such as
  FHIR-25937, FHIR-42901, FHIR-48651 show that incremental fixes have
  not converged).

### Breaking Changes

Yes — this ticket is intrinsically breaking. Specifically:

- **Resource removal.** `MedicationKnowledge` ceases to exist as a
  FHIR Core resource. Any client reading or writing it against an
  R6/post-R6 server will fail. Endpoints `[base]/MedicationKnowledge`
  must be retired.
- **Profile / IG breakage.** Every IG that profiles
  `MedicationKnowledge` (PDex Formulary, IPS medication knowledge
  bindings, several EU national IGs) is broken until they retarget
  onto `FormularyItem` or `Medication`.
- **Search parameter URL changes.** Every MK search parameter URL
  (`http://hl7.org/fhir/SearchParameter/MedicationKnowledge-*`) is
  removed; the re-homed equivalents have new canonical URLs under
  `FormularyItem` / `Medication`.
- **Terminology URL changes.** `medicationknowledge-status`
  CodeSystem/ValueSet are removed. `medicationknowledge-characteristic`
  and `medicationknowledge-package-type` ValueSets are renamed.
- **`fhir-types` enumeration.** The generated CodeSystem/ValueSet
  drops `MedicationKnowledge`, which is itself a normative URL.
- **REST capability statements.** Server CapabilityStatements that
  declared support for `MedicationKnowledge` will need to be
  regenerated.
