## Resolution Summary

**Resolution:** Persuasive.
**Resolution Description:** *"We will create the examples using the information provided."*

The Pharmacy WG has agreed to add worked MedicationRequest examples that
demonstrate how to order **medical gases** (specifically HeliOx therapy
with a supplementary titratable O₂ order). The examples must embody the
modelling pattern suggested in the ticket description:

- Identify the gas using a SNOMED CT coded `MedicationRequest.medication`.
- Express flow rate via `MedicationRequest.dosageInstruction.doseAndRate.rateQuantity`
  (fixed flow) or `…rateRange` (titratable range).
- Capture the administration device (non-rebreather mask / CPAP /
  biPAP / nasal cannula / endotracheal IPPV) via
  `MedicationRequest.device`.
- Capture the clinical indication via `MedicationRequest.reason`
  (CodeableReference → `Condition.code`, using SNOMED CT — e.g., Croup
  71186008, COPD 13645005, acute exacerbation of COPD 195951007, acute
  exacerbation of asthma 708090002).
- Express the pre-condition for the supplementary O₂ ("if SpO₂ < 88%")
  via the existing `dosageInstruction.asNeededFor` / asNeeded pattern
  (see `medicationrequest0305.xml` `asNeededCodeableConcept` precedent
  and the `doseAndRate` `rateRange` precedent in
  `medicationrequest0315.xml`).
- Link the HeliOx order and the supplementary O₂ order via a shared
  `MedicationRequest.groupIdentifier`.

No changes are required to the MedicationRequest StructureDefinition —
all required elements already exist in the current `master` (verified
against `structuredefinition-MedicationRequest.xml`, which defines
`MedicationRequest.medication`, `.dosageInstruction`, `.device`,
`.reason`, and `.groupIdentifier`).

## Feature Proposal

### Problem Statement

FHIR Core ships a rich library of MedicationRequest examples covering
oral, topical, IV admixture, chemotherapy, and miscellaneous dosage
forms, but **none of them demonstrate how to order an inhaled medical
gas** (O₂, HeliOx, nitric oxide, etc.). Implementers on the
`#implementers` Zulip stream ([Example of oxygen
prescription](https://chat.fhir.org/#narrow/stream/179166-implementers/topic/Example.20of.20oxygen.20prescription))
surfaced that gas orders need a specific combination of elements —
coded gas, flow-rate as a rate (not a dose), administration device, and
often a pre-condition threshold — and that the existing example corpus
does not show how to fit this into MedicationRequest. Stephen Chu
proposed the concrete HeliOx and supplementary O₂ examples captured in
this ticket and asked that they be added.

The current example list in
`source/medicationrequest/list-MedicationRequest-examples.xml` and the
narrative overview in
`source/medicationrequest/medicationrequest-examples-header.xml` have
no entries under a "medical gas" or "respiratory therapy" grouping.

### Proposed Change

Add **two new MedicationRequest example instances** to the FHIR Core
spec, register them in the examples `List`, and surface them in the
examples-header narrative.

**Example 1 — `medicationrequest0351.xml` — HeliOx therapy order**

- `id`: `medicationrequest0351`
- `identifier`: example identifier (follow `medicationrequest0350.xml`
  style — `http://www.bmc.nl/portal/prescriptions` + a sample value).
- `groupIdentifier`: shared value (e.g.,
  `system="http://www.bmc.nl/portal/prescriptions/group"`,
  `value="heliox-o2-combo-001"`) — same value as Example 2.
- `status` = `active`, `intent` = `order`.
- `medication`: `CodeableReference` with contained `Medication`
  resource carrying `code.coding` = SNOMED CT `895694006` (helium 790
  mL/L and oxygen 210 mL/L gas for inhalation). The ticket quotes
  `895694000`; confirm the canonical concept ID against current
  SNOMED CT. Display: "HeliOx 79/21 inhalation gas" (note the only
  approved SNOMED concept matches 79/21, not 70/30 — record the 70/30
  request as an open question below).
- `subject`: `Patient/pat1` (reuse existing example patient as
  `medicationrequest0349`/`0350` do).
- `authoredOn`: a sample date.
- `requester`: `Practitioner/f007` (reuse existing example practitioner).
- `reason`: `CodeableReference` with an inline `concept` whose
  `coding` is SNOMED CT `195951007` "Acute exacerbation of chronic
  obstructive airway disease (disorder)".
- `device`: `CodeableReference` with an inline `concept` whose
  `coding` carries a SNOMED CT device code representing a
  non-rebreather mask (e.g., `336623009` non-rebreather oxygen mask).
  If the resolver cannot identify an exact SCTID during authoring,
  use a `text` description and leave a comment so the Pharmacy WG can
  confirm — do not invent a SCTID silently.
- `dosageInstruction[0]`:
  - `route`: SNOMED CT `447694001` "Respiratory tract route".
  - `doseAndRate.type` = `ordered`,
    `doseAndRate.rateQuantity` = `6 L/min` (UCUM `L/min`).

**Example 2 — `medicationrequest0352.xml` — titratable supplementary O₂**

- `id`: `medicationrequest0352`.
- `groupIdentifier`: same value as Example 1 (links the pair).
- `status` = `active`, `intent` = `order`.
- `medication`: `CodeableReference` with contained `Medication`
  whose `code.coding` is SNOMED CT `27879005` "Oxygen (substance)".
- `subject`: `Patient/pat1`.
- `requester`: `Practitioner/f007`.
- `reason`: `CodeableReference` with inline SNOMED CT concept for
  "Hypoxaemia" (SCTID `389087006`) — records the clinical indication
  for the supplementary titration.
- `device`: `CodeableReference` with inline SNOMED CT concept
  `371820004` "Nasal cannula (physical object)" (or equivalent; if the
  precise code needs WG confirmation, use `text` and flag).
- `dosageInstruction[0]`:
  - `route`: SNOMED CT `447694001` "Respiratory tract route".
  - `asNeededFor` (or `asNeededCodeableConcept` per the current
    StructureDefinition — see `medicationrequest0305.xml` for
    precedent): SNOMED CT concept for "SpO₂ less than 88%" (if a
    single SCTID is not readily available, supply a short `text`
    value "Titrate if SpO₂ < 88%").
  - `doseAndRate.type` = `ordered`,
    `doseAndRate.rateRange.low` = `1 L/min`,
    `doseAndRate.rateRange.high` = `3 L/min` (UCUM `L/min`). This
    mirrors the `rateRange` structure already used in
    `medicationrequest0315.xml`.

**Registration changes**

- Add two new `<entry>` blocks to
  `source/medicationrequest/list-MedicationRequest-examples.xml`
  (one per example), each with the `description` build-extension set
  to a short human description ("HeliOx therapy order — gas inhalation
  with flow rate and non-rebreather mask" / "Supplementary titratable
  O₂ order — nasal cannula, rate range, pre-condition SpO₂<88%,
  linked to HeliOx order via groupIdentifier") and the `title`
  build-extension equal to the XML file's base name
  (`medicationrequest0351` / `medicationrequest0352`). The `<item>`
  reference follows the same `MedicationRequest/medrx03XX` or
  `MedicationRequest/medicationrequest03XX` pattern as the last two
  entries — pick one style and stay consistent with the example
  `id` chosen above.
- Add a new subsection (or bullets under an existing closest-match
  section such as "MISCELLANEOUS DOSAGE FORMS") to
  `source/medicationrequest/medicationrequest-examples-header.xml`
  titled "MEDICAL GAS ORDERS" that introduces respiratory-therapy
  orders and links to both new examples with one-line descriptions.
- Add a line to `source/medicationrequest/_changelog.txt` in the
  existing format:

  ```
  * Add medical-gas MedicationRequest examples (FHIR-45330, https://jira.hl7.org/browse/FHIR-45330)
    Added HeliOx and supplementary-O2 MedicationRequest examples, linked via groupIdentifier.
    Files: source/medicationrequest/medicationrequest0351.xml, source/medicationrequest/medicationrequest0352.xml, source/medicationrequest/list-MedicationRequest-examples.xml, source/medicationrequest/medicationrequest-examples-header.xml
  ```

### Design Rationale

- MedicationRequest already supports every element the examples need
  (verified against the differential in
  `structuredefinition-MedicationRequest.xml`), so an **examples-only
  change** is the minimum-risk fix that honours the resolution ("We
  will create the examples using the information provided").
- Ticket discussion proposed `MedicationRequest.device` as a
  `CodeableReference` to a DeviceDefinition; the current FHIR Core
  definition exposes `MedicationRequest.device` directly and
  CodeableReference allows either a concept or a reference — the
  examples use inline concepts so they render without extra
  contained DeviceDefinition resources, consistent with the style of
  `medicationrequest0350.xml` (inline `reason.concept`).
- `rateQuantity` is used for the fixed HeliOx flow and `rateRange` for
  the titratable O₂, matching the distinction implementers asked
  about and reusing the existing `rateRange` precedent from
  `medicationrequest0315.xml`.
- `groupIdentifier` is the resolution's chosen linkage mechanism and
  the same field is already defined on MedicationRequest (path
  `MedicationRequest.groupIdentifier` in the differential).
- Reusing `Patient/pat1` and `Practitioner/f007` keeps the examples
  consistent with the corpus (see `medicationrequest0349.xml`,
  `medicationrequest0350.xml`) and avoids introducing stub resources.

### Open Questions

4. **Briefing staleness vs. ticket scope.** The saved briefing was
