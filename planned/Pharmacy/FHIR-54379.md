## Resolution Summary

**Resolution:** Persuasive — change required.

The applied resolution text on the ticket is:

> "We will update the definition to: 'Returns instance(s) of Medication resource that have this serial number'"

Concretely, the `description` of the `Medication-serial-number` SearchParameter must be replaced from the current (incorrect, copy/paste-derived) text:

> `Returns medications in a instance with this lot number`

to the resolved wording:

> `Returns instance(s) of Medication resource that have this serial number`

No change to `code`, `type`, `expression`, or `processingMode` is mandated by the resolution.

## Feature Proposal

### Problem Statement

The `serial-number` SearchParameter on the `Medication` resource currently has a description that is a copy/paste of the neighbouring `lot-number` SearchParameter:

- Current `description`: `Returns medications in a instance with this lot number`
- The parameter `code` is `serial-number` and the FHIRPath `expression` is `Medication.identifier`.
- Reporter (Eric Haas) flagged that the description ("lot number") contradicts the search target ("identifier"), which is misleading to implementers.

The wording mismatch is confusing because (a) the parameter is named `serial-number`, (b) it actually searches `Medication.identifier`, but (c) the human-readable definition tells the reader it is searching a "lot number in an instance" — which is what the sibling `lot-number` SearchParameter does (`expression = Medication.instance.lotNumber`).

### Proposed Change

Update one element of the `Medication-serial-number` SearchParameter authored in `source/medication/bundle-Medication-search-params.xml`:

- **File:** `source/medication/bundle-Medication-search-params.xml`
- **Element:** `Bundle.entry[].resource.SearchParameter[id='Medication-serial-number'].description`
- **From:** `Returns medications in a instance with this lot number`
- **To:** `Returns instance(s) of Medication resource that have this serial number`

All other SearchParameter fields remain unchanged in scope of the resolution:

- `id`: `Medication-serial-number`
- `code`: `serial-number`
- `type`: `token`
- `expression`: `Medication.identifier`
- `url`: `http://hl7.org/fhir/build/SearchParameter/Medication-serial-number`
- `processingMode`: `normal`
- standards-status extension: `normative`

XML diff (illustrative):

```xml
<SearchParameter>
  <id value="Medication-serial-number"/>
  ...
  <url value="http://hl7.org/fhir/build/SearchParameter/Medication-serial-number"/>
- <description value="Returns medications in a instance with this lot number"/>
+ <description value="Returns instance(s) of Medication resource that have this serial number"/>
  <code value="serial-number"/>
  <type value="token"/>
  <expression value="Medication.identifier"/>
  ...
</SearchParameter>
```

### Design Rationale

- The resolution is a `Technical Correction` (Persuasive) — narrowly scoped to fix a misleading description that almost certainly originated as a copy/paste from the sibling `lot-number` SearchParameter immediately above it in the same bundle (lines 129–135 of `bundle-Medication-search-params.xml`).
- The applied wording ("Returns instance(s) of Medication resource that have this serial number") is the exact text agreed in the ticket's resolution description; it accurately describes the actual search behaviour (matching on `Medication.identifier`, which is normatively how a serial number is carried on a Medication instance).
- Changing the SearchParameter `expression` was explicitly **not** requested. The reporter's own remark — "the search is on the identifier" — frames `Medication.identifier` as the correct target; the bug is purely in the human-readable description.
- The reporter's originally proposed wording ("Returns medications in a instance with this identifier/serial number") was refined by Pharmacy to drop the "in a instance" phrasing (which was itself a residue of the lot-number copy) and to use "instance(s) of Medication resource" — the canonical way to refer to a resource instance in FHIR narrative.
