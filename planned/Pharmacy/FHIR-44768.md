## Resolution Summary

Resolution: **Not Persuasive with Modification.**

Verbatim resolution description (from Jira):

> Will update descriptions and comments for information source and derived from. Note you can currently use the `PractitionerRole.code`, `Organization.type`, `RelatedPerson.relationship` to specify a code instead of indicating a specific individual.

The committee declined the original request to make `MedicationStatement.informationSource` (and the related `derivedFrom`) a `CodeableReference` (see committee comment from `jduteau`, 2024-11-11: "We don't think we need to change to a `CodeableReference` at this time"). Instead, the committee agreed to **clarify the existing element documentation** to point implementers at the already-existing pattern of using a referenced resource that itself carries a categorical code:

- `Practitioner` / `PractitionerRole` â€” use `PractitionerRole.code` to express the role (e.g., pharmacist, nurse).
- `Organization` â€” use `Organization.type` to express the kind of organization (e.g., a pharmacy, an LTC facility).
- `RelatedPerson` â€” use `RelatedPerson.relationship` to express the relationship type (e.g., spouse, parent, caregiver).

This is a **documentation-only / narrative** change. No structural changes to `MedicationStatement` (no new elements, no cardinality changes, no type changes, no binding changes, no new extensions, no terminology additions).

## Feature Proposal

### Problem Statement

`MedicationStatement.informationSource` is typed as `Reference(Patient | Practitioner | PractitionerRole | RelatedPerson | Organization | Device | Group)` (see `structuredefinition-MedicationStatement.xml` lines 360â€“375 in `HL7/fhir`). When recording medication adherence, implementers (Epic, Oracle/Cerner per the reporter's comment) frequently want to document only the **type/role** of the source â€” "spouse", "patient's pharmacy", "transfer-facility MAR", etc. â€” rather than incurring the workflow cost of identifying a specific individual or organization instance. The original ticket proposed allowing a code in addition to a reference (i.e., `CodeableReference`).

The committee determined that the structural change is unnecessary because the existing reference targets already carry coded role/type/relationship elements that satisfy the use case (`PractitionerRole.code`, `Organization.type`, `RelatedPerson.relationship`). The gap is therefore one of **discoverability** in the spec: implementers reading the `informationSource` (and `derivedFrom`) element documentation are not currently told that the categorical-only pattern is supported via the referenced resources.

### Proposed Change

Update narrative text on two elements of the `MedicationStatement` StructureDefinition (in `source/medicationstatement/structuredefinition-MedicationStatement.xml`), and one entry in the resource changelog. No `min`/`max`, `type`, `targetProfile`, `binding`, `mapping`, or invariant changes.

**1. `MedicationStatement.informationSource`** (current XML at lines 360â€“384; element has `<short>` and `<definition>` only â€” no existing `<comment>`):

- **Keep:** `short`, `min=0`, `max=*`, `type Reference` with all seven existing `targetProfile` entries (Device, Patient, Practitioner, PractitionerRole, RelatedPerson, Organization, Group), and both `mapping` entries (`w5` â†’ `FiveWs.source`; `rim` â†’ `.participation[typeCode=INF]â€¦`).
- **Update `<definition>`** â€” preserve the existing prose, then add a sentence telling implementers they can categorize without naming a specific party. Suggested text:

  > "The person or organization that provided the information about the taking of this medication. When only the *type* of source is known (e.g., a spouse, a pharmacy, a long-term care facility) rather than the specific individual or organization, implementers may reference an instance whose categorical element conveys that type â€” `PractitionerRole.code` for a practitioner role, `Organization.type` for an organization, or `RelatedPerson.relationship` for a person related to the patient â€” without needing to identify a specific individual. Note: Use `derivedFrom` when a `MedicationStatement` is derived from other resources, e.g. `Claim` or `MedicationRequest`."

- **Add a new `<comment>` element** (inserted between `<definition>` and `<min>`, matching sibling-element ordering in the same file) summarising the same guidance for implementers who scan the comment column:

  > "When the source is recorded as a category rather than a specific party (e.g., 'spouse', 'patient's pharmacy', 'transfer facility'), reference a `RelatedPerson`, `PractitionerRole`, `Organization`, etc. that carries only the appropriate type/role/relationship code (`RelatedPerson.relationship`, `PractitionerRole.code`, `Organization.type`) without populating the identifying details. This avoids requiring users to collect identifying information that is not clinically necessary for adherence reporting."

**2. `MedicationStatement.derivedFrom`** (current XML at lines 385â€“400; element already has both `<definition>` and `<comment>`):

- **Keep:** `short`, `min=0`, `max=*`, `type Reference(Resource)`, and the existing `rim` mapping.
- **Update `<comment>`** â€” preserve the existing comment prose, then append the same categorical guidance, scoped to derivation sources where applicable:

  > "Likely references would be to `MedicationRequest`, `MedicationDispense`, `Claim`, `Observation` or `QuestionnaireAnswers`. The most common use cases for deriving a `MedicationStatement` come from creating a `MedicationStatement` from a `MedicationRequest` or from a lab observation or a claim. It should be noted that the amount of information that is available varies from the type of resource that you derive the `MedicationStatement` from. When the derivation source is known only categorically (e.g., a paper MAR from a transfer facility, a verbal interview with a family member), the same pattern described on `informationSource` applies â€” reference a resource (`Organization`, `RelatedPerson`, `PractitionerRole`) whose type/role/relationship code carries the category, rather than identifying a specific individual."

**3. Resource changelog** (`source/medicationstatement/_changelog.txt`):

Append a new bullet documenting the change, following the existing two-entry style in the file (each entry: a heading line with ticket key + URL, an indented description, and a `Files:` line listing edited paths).

### Design Rationale

- **Aligns with committee disposition.** The resolution explicitly chose narrative clarification over a `CodeableReference` upgrade, citing the existing categorical elements on the target resources. The plan implements exactly that: documentation pointing at `PractitionerRole.code`, `Organization.type`, `RelatedPerson.relationship`.
- **Non-breaking.** Adding prose to `<definition>` and a new `<comment>` does not change any conformance constraint. Existing instances remain valid; existing implementations continue to work. This matches the playbook recipe for "Update narrative / prose".
- **Consistent with FHIR style.** Other elements in this same StructureDefinition (e.g., `derivedFrom`, `reason`) use `<comment>` to convey usage guidance distinct from the formal definition. Adding a `<comment>` to `informationSource` matches that pattern.
- **`derivedFrom` is included** because the resolution text explicitly names "information source and derived from", and the same categorical-source pattern is equally relevant when an adherence statement is derived from a paper MAR or a verbal interview.
- **Alternative considered (rejected by the committee).** Re-typing `informationSource` (and/or `derivedFrom`) as `CodeableReference` would cleanly express the "category only" use case at the element level, but is a breaking structural change and was explicitly declined; see `jduteau` 2024-11-11 comment.
