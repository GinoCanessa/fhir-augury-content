## Resolution Summary

Resolution: **Persuasive with Modification**.

Per the resolution description:

1. **No new MedicationDispense elements are required.** The "number of dispenses already supplied" need is already met by the existing `MedicationDispense.fillNumber` element (0..1 `positiveInt`, present in the current `structuredefinition-MedicationDispense.xml` at lines 571–581 of the clone). No change is required for that aspect of the request.
2. **Authorized refills should be conveyed by one of two existing mechanisms** (no new core element):
   - Reference (or contain) the associated `MedicationRequest`, whose `dispenseRequest.numberOfRepeatsAllowed` element already carries the authorized repeat count, via `MedicationDispense.authorizingPrescription`; or
   - Use a FHIR extension on `MedicationDispense` to carry the authorized repeat count, where an extension-based pattern is preferred by the implementer/IG.
3. **The spec will be updated with an example** demonstrating the first option — a `MedicationDispense` that **contains** a `MedicationRequest` (with `numberOfRepeatsAllowed`) and references it from `authorizingPrescription`.

The resolution explicitly rejects adding `MedicationDispense.numberOfRepeatsAuthorised` or `MedicationDispense.numberDispensed` to the base resource (and does not introduce them as core modifier extensions either).

## Feature Proposal

### Problem Statement

Transition-of-care use cases (the ticket cites Australia's Aged Care transfer summaries) need to convey, alongside each `MedicationDispense`, both (a) how many repeats the prescription authorized and (b) how many of those repeats have already been dispensed (e.g. "5 authorized, 2 already dispensed"). The reporter believed that `MedicationDispense` was missing the necessary modifier elements (`numberOfRepeatsAuthorised`, `numberDispensed`) and proposed adding them to the base resource (or as modifier extensions).

In fact, R6 `MedicationDispense` already carries the "fills already done" count via `MedicationDispense.fillNumber` (a `positiveInt` documented as "the known number of the fill over the entire lifetime of the prescription"), and the authorized repeat count lives on the prescription side as `MedicationRequest.dispenseRequest.numberOfRepeatsAllowed`. The gap is therefore documentary, not structural: implementers need a worked example that shows the two-resource pattern end-to-end.

### Proposed Change

**Documentation/example-only change in `HL7/fhir`. No StructureDefinition edits.**

1. **Add a new MedicationDispense example** — `medicationdispense0336` (next free number after the existing range; current list ends at `medicationdispense0335`, with `0323` and `0333` already skipped). The example demonstrates the resolution's recommended pattern:
   - A `MedicationDispense` resource that **contains** a `MedicationRequest` resource representing the authorizing prescription.
   - The contained `MedicationRequest` carries `dispenseRequest.numberOfRepeatsAllowed = 5` (i.e. 5 authorized repeats), aligning with the ticket's Pulmicort Turbuhaler 400 mcg / budesonide scenario.
   - The `MedicationDispense` references the contained request via `authorizingPrescription` (`reference = "#<contained-id>"`).
   - The `MedicationDispense` sets `fillNumber = 2` to convey "this is the 2nd of the authorized fills".
   - A coded medication, `subject`, `status = completed`, `whenHandedOver`, `quantity`, and `daysSupply` should be populated to make the example realistic and parseable by the publisher (use the same shape as the existing `medicationdispense0301.xml`, which already shows the `<contained>` + `authorizingPrescription` `reference="#…"` idiom but contains a `Medication` rather than a `MedicationRequest`).

2. **Register the new example** by adding a new `<entry>` to `source/medicationdispense/list-MedicationDispense-examples.xml`, following the exact structure used by the other entries (a `description` extension, a `title` extension whose `valueString` matches the file stem, and an `<item><reference value="MedicationDispense/meddisp0336"/><display value="..."/></item>` block).

3. **Optionally cross-link from the narrative.** Add a short paragraph to `source/medicationdispense/medicationdispense-introduction.xml` (under "Background and Context") that points implementers at the new example for transition-of-care scenarios, and explains the two-pathway choice (contained `MedicationRequest` vs. extension) that the resolution endorses. This paragraph should explicitly mention `MedicationDispense.fillNumber` as the canonical home for "how many fills so far" and `MedicationRequest.dispenseRequest.numberOfRepeatsAllowed` as the canonical home for "how many fills authorized". *This step is recommended but not strictly mandated by the resolution; flag for the work group if they want only the example.*

Sketch of the new example (illustrative — not a substitute for authoring):

```xml
<MedicationDispense xmlns="http://hl7.org/fhir">
  <id value="meddisp0336"/>
  <contained>
    <MedicationRequest>
      <id value="medrx0336"/>
      <status value="active"/>
      <intent value="order"/>
      <medication>
        <concept>
          <coding>
            <system value="http://snomed.info/sct"/>
            <code value="..."/>
            <display value="Pulmicort Turbuhaler 400 microgram/actuation powder for inhalation (budesonide)"/>
          </coding>
        </concept>
      </medication>
      <subject><reference value="Patient/pat1"/></subject>
      <dispenseRequest>
        <numberOfRepeatsAllowed value="5"/>
        <quantity>
          <value value="1"/>
          <unit value="inhaler"/>
        </quantity>
      </dispenseRequest>
    </MedicationRequest>
  </contained>
  <status value="completed"/>
  <medication>
    <reference><reference value="#medrx0336/medication"/></reference>
  </medication>
  <subject><reference value="Patient/pat1"/></subject>
  <authorizingPrescription>
    <reference value="#medrx0336"/>
  </authorizingPrescription>
  <quantity>
    <value value="1"/>
    <unit value="inhaler"/>
  </quantity>
  <whenHandedOver value="2025-02-20"/>
  <fillNumber value="2"/>
</MedicationDispense>
```

### Design Rationale

- **No new core elements** — the resolution explicitly favours reusing what is already there (`fillNumber` for fills-done; `MedicationRequest.dispenseRequest.numberOfRepeatsAllowed` accessed via `authorizingPrescription` for fills-authorized) over adding redundant modifier elements to `MedicationDispense`. This avoids two sources of truth for "authorized repeats" (one on the request, one duplicated on each dispense) and avoids breaking changes to a widely implemented resource.
- **Containment over external reference** in the example — when a `MedicationDispense` is exchanged out of the context of its prescription system (transition-of-care, document-style exchange like Aged Care transfer summaries), the receiver may not be able to dereference an external `MedicationRequest`. Containing the request preserves the authorized-repeat datum in the same payload while still using the standard `authorizingPrescription` link with a `#fragment` reference. This is the pattern the resolution recommends.
- **Extension as the alternative** — explicitly called out in the resolution for IGs that prefer not to carry a contained `MedicationRequest`. No extension is being defined or registered as part of this ticket; that is left to implementers/IGs.
- **Consistency with existing examples** — `medicationdispense0301.xml` already demonstrates the `<contained>…</contained>` + `<authorizingPrescription><reference value="#…"/></authorizingPrescription>` shape (today with a contained `Medication`); the new example simply applies the same idiom to a contained `MedicationRequest`.
