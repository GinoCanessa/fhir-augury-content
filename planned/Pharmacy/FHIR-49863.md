## Resolution Summary

**Resolution:** Persuasive with Modification.

The Pharmacy WG accepted the intent of the request (an explicit semantic
flag distinguishing a record-of-request from an authoritative request),
but rather than adding a new sibling element they will **rename and
re‑purpose the existing `MedicationRequest.reported` boolean** element.
Verbatim from the resolution description:

- Rename `reported` to **`isRecordOfRequest`**, and note that it maps to
  the `reported[x] boolean` element on the Request workflow pattern.
- **Make it a modifier element** (`isModifier=true`).
- New short:
  > Whether this is record of a Medication Request or the actual request
  > itself.
- New formal definition:
  > The information in this instance is a record of the Medication
  > Request and is not request or order itself and should not be
  > considered as such, i.e. the information should not be used for
  > medication dispense | supply and administration.

The 2025‑11‑03 follow‑up comment from Jose Costa‑Teixeira notes the WG
is "waiting for the issue creator to confirm if we need to add something
else (comment, isModifier..)" — the resolution text already commits to
making it a modifier, so this plan implements the modifier flag; any
additional comment text should be confirmed with the reporter (see Open
Questions).

## Feature Proposal

### Problem Statement

In R5, `MedicationRequest.reported[x]` (boolean | Reference) overlaps
semantically with `informationSource` and is easy to misread alongside
`status = active` and `intent = order`. The reporter notes that when
MedicationRequest is reused inside summary IGs (IPS, Australian Patient
Summary IG, Aged Care Clinical Document IG) to communicate that there is
**a record of** a known order — not the order itself — implementers need
an unambiguous, modifier‑strength flag stating "this instance must not
be used as authority for dispensing or administration." The current
`reported` boolean does not carry that strength: it is not a modifier,
its short text talks about "true if this is reported rather than
original record" without forbidding downstream action, and it is named
in a way that invites confusion with the data‑provenance concept of
"reported by".

### Proposed Change

Rename and strengthen the existing element rather than add a new one.

In `source/medicationrequest/structuredefinition-MedicationRequest.xml`
(differential):

- **Rename** the element formerly known as `MedicationRequest.reported`
  to **`MedicationRequest.isRecordOfRequest`** (id, path, and any
  `<element id="…">`/`<path value="…">` references).
- **Type:** `boolean`, **cardinality:** `0..1` (unchanged from the
  boolean choice of `reported[x]`; the choice is collapsed to a single
  boolean type per the resolution wording "maps to the `reported[x]
  boolean` element on the Request workflow pattern").
- **`isModifier` = `true`**, with `isModifierReason` = "This element is
  labeled as a modifier because this element explicitly makes the
  request not an authoritative record but for information purposes."
  (text already present, currently commented out at lines 638–639 of
  the StructureDefinition).
- **`isSummary` = `true`** (already set).
- **`short`:** `Whether this is record of a Medication Request or the
  actual request itself`.
- **`definition`:** `The information in this instance is a record of
  the Medication Request and is not request or order itself and should
  not be considered as such, i.e. the information should not be used
  for medication dispense | supply and administration.`
- **`comment`:** `If not populated, then assume that this is the
  original record and not reported` (already present at line 632; keep).
- **Mapping to the Request workflow pattern:** add a `<mapping>` (or
  remove the corresponding `<unmappedElement>` exception, see below)
  recording that `MedicationRequest.isRecordOfRequest` maps to
  `Request.reported` on the Request workflow pattern.

Inspection confirms that the rename, short text, definition, type, and
cardinality are **already applied** in
`source/medicationrequest/structuredefinition-MedicationRequest.xml`
(element at lines 628–640). The remaining work is:

1. Uncomment the `isModifier`/`isModifierReason` block (lines 638–639).
2. Update the request‑pattern mapping exceptions
   (`source/medicationrequest/medicationrequest-request-mapping-exceptions.xml`,
   line 288) so that `Request.reported` is no longer listed as
   `<unmappedElement>` — it now maps to
   `MedicationRequest.isRecordOfRequest`. If the resource and pattern
   diverge on naming/type, this becomes a `<divergentElement>` entry
   describing the rename.
3. Fix the malformed example XML in
   `source/medicationrequest/medicationrequest0340.xml` (line 37
   contains `<isRecordOfRequest value="true"></reported>` — mismatched
   closing tag, must become `<isRecordOfRequest value="true"/>`).
4. Sweep the rest of `source/medicationrequest/` for any remaining uses
   of `reported`/`reportedBoolean`/`reportedReference` in examples or
   prose and either remove them or convert to `isRecordOfRequest`.

### Design Rationale

- Reusing the existing slot keeps the `Request` workflow pattern
  alignment intact (the resolution explicitly preserves the mapping to
  `Request.reported[x] boolean`) and avoids inflating the resource with
  parallel flags.
- Promoting the element to **modifier** is the substantive semantic
  change: it forces consumers (CDS, dispensing systems, eMAR) to honour
  the "do not act on this" intent of the instance, which the original
  optional boolean never enforced. This is exactly the absolute‑clarity
  outcome requested in the ticket.
- Dropping the `Reference` choice (i.e. collapsing `reported[x]` to a
  pure boolean) is consistent with the resolution wording and with the
  fact that "who reported it" is already captured by
  `informationSource`. Provenance of the report belongs there;
  `isRecordOfRequest` carries only the modifier semantic.
- The new name reads as a predicate ("is this instance a record of a
  request?") rather than a verb form ("was this reported?"), removing
  the ambiguity with `informationSource` that the reporter highlighted.

### Breaking Changes

Yes — this change is **backward‑incompatible** with R5 even though it
re‑uses the same element slot:

- **Element rename:** `MedicationRequest.reported[x]` →
  `MedicationRequest.isRecordOfRequest`. Implementations that read
  `reported`, `reportedBoolean`, or `reportedReference` will need to
  switch to `isRecordOfRequest`.
- **Type narrowing:** the `Reference` choice of `reported[x]` is
  removed; only `boolean` survives. Existing instances using
  `reportedReference` (provenance of the report) must move that
  reference to `informationSource`.
- **Modifier promotion:** `isModifier` flips from absent to `true`.
  Consumers that previously ignored `reported` MUST now honour
  `isRecordOfRequest=true` and refrain from acting on the instance for
  dispensing/administration. This changes the conformance contract.

### Related Specifications

- **Request workflow pattern** (`source/request.html` and
  `source/patterns/`) — `Request.reported[x]` is the pattern element
  this resource maps to. The mapping exception file is the only
  in‑resource artifact that records the relationship; no edit to the
  pattern itself is implied.
- **MedicationDispense / MedicationAdministration / MedicationStatement**
  — these resources do not carry an analogous flag; the modifier on
  `MedicationRequest` is the authoritative signal that downstream
  resources should not be derived from this instance. No conformance
  edits required, but cross‑resource narrative could be revisited in a
  follow‑up.
- Downstream IGs (**IPS**, **AU Patient Summary**, **Aged Care Clinical
  Document FHIR IG**) reference `MedicationRequest` and will need to
  update profiles/examples to use `isRecordOfRequest` — out of scope
  for HL7/fhir but worth flagging to the WG.

### Step-by-Step Tasks

#### HL7/fhir

1. **Make `isRecordOfRequest` a modifier element**
   - File: `source/medicationrequest/structuredefinition-MedicationRequest.xml`
   - Action: In the `<element id="MedicationRequest.isRecordOfRequest">`
     block (currently lines 628–640), remove the surrounding
     `<!-- … -->` comment markers around the `<isModifier value="true"/>`
     and `<isModifierReason value="…"/>` lines so both elements become
     active. Keep `<isSummary value="true"/>` as is.
   - Details: The resulting block should contain, in order: `path`,
     `short`, `definition`, `comment`, `min=0`, `max=1`,
     `type=boolean`, `isModifier=true`,
     `isModifierReason="This element is labeled as a modifier because
     this element explicitly makes the request not an authoritative
     record but for information purposes."`, `isSummary=true`.

2. **Confirm rename is fully applied across the StructureDefinition**
   - File: `source/medicationrequest/structuredefinition-MedicationRequest.xml`
   - Action: Search the file for any residual `reported`, `reportedBoolean`,
     or `reportedReference` references (element ids, paths,
     `differentiates` slicing, mappings). The only allowed remaining
     occurrence is the workflow‑pattern mapping value (e.g.
     `Request.reported` inside a `<mapping>` entry, see next step).
     Update or delete any stragglers.
   - Details: Cross‑check against the resolution wording — only
     `MedicationRequest.isRecordOfRequest` should appear as the element
     id/path; the boolean choice is the only type.

3. **Add Request‑pattern mapping for `isRecordOfRequest`**
   - File: `source/medicationrequest/structuredefinition-MedicationRequest.xml`
   - Action: Inside the `<element id="MedicationRequest.isRecordOfRequest">`
     block, add (or verify the presence of) a `<mapping>` child whose
     `identity` matches the workflow‑pattern mapping identity used
     elsewhere in the file (commonly `workflow`) with `map` value
     `Request.reported`.
   - Details: This is the in‑file expression of "maps to the
     `reported[x] boolean` element on the Request workflow pattern" from
     the resolution.

4. **Update the request‑pattern mapping exceptions file**
   - File: `source/medicationrequest/medicationrequest-request-mapping-exceptions.xml`
   - Action: Remove the line
     `<unmappedElement patternPath="Request.reported" reason="Unknown"/>`
     (line 288) and replace it with a `<divergentElement>` entry that
     records the rename and type narrowing, e.g.
     `<divergentElement patternPath="Request.reported" resourcePath="MedicationRequest.isRecordOfRequest">`
     with a `<missingTypes _pattern="Reference(Any)" reason="Renamed to isRecordOfRequest; only the boolean choice is retained on MedicationRequest"/>`
     child and matching `<shortUnmatched>` / `<definitionUnmatched>` if
     the divergence checker requires them.
   - Details: If the WG prefers no divergence entry at all (because the
     in‑file `<mapping>` already declares the relationship), simply
     delete the `<unmappedElement>` line. Whichever route is taken, the
     pattern checker must no longer flag `Request.reported` as
     unmapped.

5. **Fix the malformed example XML in medicationrequest0340**
   - File: `source/medicationrequest/medicationrequest0340.xml`
   - Action: Replace line 37
     `<isRecordOfRequest value="true"></reported>` with the
     well‑formed self‑closing element `<isRecordOfRequest value="true"/>`.
   - Details: The current text is not parseable XML (open tag
     `isRecordOfRequest`, close tag `reported`) and will fail validation.
     This is a leftover from a partial rename.

6. **Sweep remaining examples and narrative for `reported` usage**
   - File: all `source/medicationrequest/medicationrequest*.xml`
     example files and `source/medicationrequest/medicationrequest-introduction.xml`,
     `medicationrequest-notes.xml`, `medicationrequest-html.xml`.
   - Action: Search for `<reported`, `reportedBoolean`,
     `reportedReference`, `MedicationRequest.reported` and update any
     surviving references. In `medicationrequest-introduction.xml`
     line 9 ("…may be used to report orders/request from external
     systems that have been reported for informational purposes…"),
     consider tightening the wording to reference
     `isRecordOfRequest` as the modifier that governs this usage —
     confirm scope with the WG before editing prose.
   - Details: Search‑params bundle
     (`bundle-MedicationRequest-search-params.xml`) was inspected and
     contains no `reported` search parameter; no change needed there.

7. **Do not edit the `<snapshot>` block**
   - File: `source/medicationrequest/structuredefinition-MedicationRequest.xml`
   - Action: Confirm all edits stay inside `<differential>`. Per the
     repo briefing, `<snapshot>` is regenerated by the publisher. Any
     manual snapshot edits will be lost and may break QA.
   - Details: Likewise leave `tools/`, `qa/`, `schema/`, and any
     `medicationrequest-spreadsheet.xml` derived companion file alone.
