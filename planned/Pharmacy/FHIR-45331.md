## Resolution Summary

The committee disposition is **Persuasive**. The recorded resolution
description is short and operational:

> Once the examples are created, we will inform O&O about them so they can
> add them to the appropriate Device resources.

In context, this means: the Pharmacy work group has accepted the request
to surface Pharmacy-authored, Device-related medication examples (the
canonical case being O₂ administered via a nasal cannula at a specified
flow rate) from the Device-side specification pages. The Pharmacy-side
work — actually creating those `MedicationRequest` /
`MedicationAdministration` examples — is tracked in the partner ticket
[FHIR-45329](https://jira.hl7.org/browse/FHIR-45329) ("Include example of
MedicationRequest, etc. involving Device"). FHIR-45331 is the
*Device-page-side* of that pair: once the examples exist, the Device,
DeviceDefinition (and possibly DeviceAssociation / DeviceMetric / Device
Module) pages should link to them so a reader landing on a Device page
can discover the cross-domain examples.

Because the Device family of resources is owned by the **Orders &
Observations (O&O / OO)** work group (see the "Responsible Owner: Work
Group OO" line in `source/device-module.html`), the actual page edits
must be coordinated with O&O even though the ticket sits with Pharmacy.

## Feature Proposal

### Problem Statement

Today the Device-side pages in FHIR Core do not surface any of the
Pharmacy-modelled examples in which a device is part of a medication
order or administration (e.g., oxygen via nasal cannula with a flow-rate
setting, IV pump-driven infusions, triple-lumen catheter routing).
Implementers reading `device.html`, `devicedefinition.html`,
`deviceassociation.html`, `devicemetric.html` or the Device Module
overview page get no pointer to those use cases, even when the supporting
example artifacts exist in the Pharmacy resource folders. The Zulip
discussion that triggered this ticket
([Example of oxygen prescription](https://chat.fhir.org/#narrow/stream/179166-implementers/topic/Example.20of.20oxygen.20prescription))
documented exactly this gap: implementers asking how to model a flow-rate
on an oxygen prescription had to be hand-walked to Pharmacy-side modelling
because the Device pages did not advertise the pattern. Pharmacy filed
FHIR-45329 to *create* the examples and FHIR-45331 to ensure they are
*linked from* the Device pages once they exist.

### Proposed Change

Add inbound references on the Device-family specification pages pointing
to the Pharmacy-authored Device-related medication examples produced by
FHIR-45329. Concretely:

1. **Device** (`source/device/device-notes.xml`, in the existing Notes
   section that already cross-links `device-examples.html`): add a short
   bullet/sub-paragraph in the Notes block (after the "Status Elements" /
   `conformsTo` notes added for R6) pointing to the Pharmacy
   medication examples and explaining when a `Device` instance is
   appropriate (i.e., the device used for an administration, such as the
   specific nasal cannula instance referenced from
   `MedicationAdministration`). Use anchor links of the form
   `medicationadministration-example-<name>.html` /
   `medicationrequest-example-<name>.html` once FHIR-45329 finalises the
   example file names.

2. **DeviceDefinition** (`source/devicedefinition/devicedefinition-notes.xml`):
   the current Notes block is essentially empty (`<ul></ul>`). Add a
   first Notes bullet that explains when `DeviceDefinition` (rather than
   `Device`) is the right reference target from a Pharmacy resource —
   e.g., when an order specifies the *kind* of device (an "IV pump" or
   "nasal cannula" type) without binding to a specific instance — and
   link out to the corresponding Pharmacy examples (`MedicationRequest`
   referencing a `DeviceDefinition`).

3. **DeviceAssociation** (`source/deviceassociation/deviceassociation-notes.xml`):
   the file is currently a placeholder
   (`<!-- insert xhtml content here -->`). If the Pharmacy examples
   include a `DeviceAssociation` instance (e.g., associating a pump to
   the patient for the duration of an administration), add a Notes block
   that introduces this usage pattern and links the example. If the
   examples do **not** exercise `DeviceAssociation`, leave this page
   alone and record the decision in the ticket — do not invent a
   reference.

4. **DeviceMetric** (`source/devicemetric/devicemetric-notes.xml`): the
   existing Notes block already discusses RTM-managed metric codes. If
   the Pharmacy oxygen example carries observed flow-rate via
   `DeviceMetric`, add a second bullet pointing to it as a worked
   example. If the examples model the dose/flow purely on the
   `Medication*` side (e.g., `dosageInstruction.rate[x]`), do **not**
   link from `devicemetric.html` — same rule as DeviceAssociation.

5. **Device Module overview** (`source/device-module.html`): in the
   "actual clinical use and/or behavior" paragraph (already mentions
   Observation, DeviceAlert, Procedure), add a short trailing sentence
   or sub-bullet in the surrounding narrative that points readers at the
   Pharmacy `Medication*` examples as another concrete use case for
   Device. This is the highest-traffic discovery point and should
   anchor-link the same example pages used above.

The new prose on each page is small (1–3 sentences plus 1–4 anchor
links). No StructureDefinition, no terminology binding, no search
parameter changes are required — this is a documentation-only,
discoverability change.

### Design Rationale

- **Discoverability over duplication.** The committee explicitly chose
  to *reference* the Pharmacy examples from the Device pages rather than
  copy them. This keeps a single source of truth on the Pharmacy side
  (FHIR-45329) and avoids the maintenance burden of forked copies.
- **Notes-section placement.** The FHIR Core convention for "see also"
  guidance lives in each resource's `…-notes.xml` block (visible in
  `source/device/device-notes.xml`, which already cross-links
  `device-examples.html`, `device-mappings.html`, etc.). Putting the new
  references there matches existing patterns rather than inventing a new
  page section.
- **Cross-WG coordination.** The Device family is owned by O&O
  (`Responsible Owner: Work Group OO` in `source/device-module.html`).
  The resolution explicitly says Pharmacy "will inform O&O" once the
  examples exist; this plan keeps the actual page edits as the
  O&O-coordinated step rather than having Pharmacy commit directly to
  Device pages without sign-off.
- **Conditional touches.** DeviceAssociation and DeviceMetric are listed
  in the original ticket as "possibly" affected. Whether they get a
  reference depends entirely on whether the FHIR-45329 examples
  instantiate those resources. The plan makes that conditional explicit
  so we do not add references to nonexistent example artifacts.

## Implementation Plan

### Prerequisites

1. **FHIR-45329 must be applied first.** The Pharmacy
   `MedicationRequest` / `MedicationAdministration` examples
   (referencing `Device` / `DeviceDefinition`) must exist in the build
   so that the new anchor links on the Device pages point at real
   artifacts. Specifically, the file names of the new example instances
   (e.g., `medicationrequest-example-oxygen-cannula.xml`) must be known
   before this ticket's edits can be finalised.
2. **O&O sign-off.** Per the resolution ("we will inform O&O about
   them"), the Pharmacy work group should confirm with O&O before
   committing edits to the Device-family pages, since O&O is the
   responsible owner of those pages.
3. **Decide DeviceAssociation / DeviceMetric inclusion.** Inspect the
   FHIR-45329 example set; mark tasks 3 and 4 below as "skip" if those
   resources are not exercised.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add Pharmacy-example reference to Device Notes**
   - File: `source/device/device-notes.xml`
   - Action: Append a new sub-section (or extend the closing paragraph
     of the existing Notes block) titled e.g. *"Use in Medication
     Administration"* that (a) explains when a specific `Device`
     instance should be referenced from a `MedicationAdministration`
     (versus referencing a `DeviceDefinition` from a
     `MedicationRequest`) and (b) links to the FHIR-45329 example(s) by
     anchor (e.g., `<a href="medicationadministration-example-oxygen-cannula.html">…</a>`).
   - Details: Keep the new content inside the existing root `<div>`;
     follow the surrounding `<a name="…"/><h3>…</h3>` pattern used by
     the `identifierandtype`, `udi`, `status`, and `conformsTo` blocks.
     Preserve the existing schema attributes on the root `<div>`.

2. **Populate the empty DeviceDefinition Notes block**
   - File: `source/devicedefinition/devicedefinition-notes.xml`
   - Action: Replace the empty `<ul></ul>` under the existing
     `<a name="notes"></a>` / `<h2>Notes:</h2>` with one or more bullets.
     The first bullet should describe when a Pharmacy resource (e.g.,
     `MedicationRequest`) should reference `DeviceDefinition` (i.e., to
     express the *kind* of device — "an IV pump", "a nasal cannula" —
     without committing to a specific instance), and link to the
     corresponding FHIR-45329 example(s).
   - Details: Match the markup style already used in
     `source/devicemetric/devicemetric-notes.xml` (bullet list, inline
     `<a href="…">` links). Do not change the surrounding
     `<a name="notes"></a>` anchor — other pages may already link to it.

3. **(Conditional) Add DeviceAssociation Notes content**
   - File: `source/deviceassociation/deviceassociation-notes.xml`
   - Action: Only if FHIR-45329 produces a `DeviceAssociation` instance
     for the medication example: replace the
     `<!-- insert xhtml content here -->` placeholder with a Notes
     block (`<a name="notes"/>` + `<h2>Notes:</h2>` + `<ul><li>…</li></ul>`
     mirroring `devicedefinition-notes.xml`) describing the Pharmacy
     use case and linking the example. Otherwise leave the file
     untouched and note the skip in the ticket.
   - Details: If the file is left as-is, double-check that the existing
     placeholder comment is still acceptable to the publisher (it
     currently is — the file is referenced from the build).

4. **(Conditional) Extend DeviceMetric Notes**
   - File: `source/devicemetric/devicemetric-notes.xml`
   - Action: Only if FHIR-45329 produces a `DeviceMetric` instance
     (e.g., observed oxygen flow-rate as a metric): append a second
     `<li>` after the existing RTM Management Service bullet that
     points at the Pharmacy example and explains the `DeviceMetric` →
     metric-code wiring used.
   - Details: Preserve the existing bullet and its links. Do not touch
     the `<a name="notes"/>` anchor.

5. **Surface examples on the Device Module overview page**
   - File: `source/device-module.html`
   - Action: In the introduction (immediately after the `Index` table
     and the `Device-related FHIR Resources: The Big picture` paragraph
     that already lists the kinds of device-related use cases), add one
     short paragraph or bullet under the existing narrative that
     advertises the Pharmacy-authored medication examples as a worked
     case study, with anchor links to the same example pages used in
     tasks 1–4. Do not introduce a new top-level `<h3>` unless the
     reviewer asks for one — a short paragraph keeps the page
     consistent with surrounding prose.
   - Details: Keep the existing `Responsible Owner: Work Group OO`
     header and STU note untouched. The new paragraph belongs **inside**
     the main `<body>` content flow, not in the navbar/header includes.

6. **Run a link-integrity sweep**
   - File: (no edit; sweep across `source/`)
   - Action: After tasks 1–5, grep for the new anchor strings (e.g.,
     `medicationadministration-example-…html`,
     `medicationrequest-example-…html`) repo-wide and confirm each
     resolves to a generated page (i.e., the example files exist in
     the same `source/medication*/` folders FHIR-45329 populated). Also
     re-grep for any pre-existing `device.html#…` /
     `devicedefinition.html#…` anchors that you may have inadvertently
     shadowed.
   - Details: This catches the gotcha called out in the FhirCore
     briefing about anchor collisions before the publisher build runs.
