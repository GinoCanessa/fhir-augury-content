## Resolution Summary

Resolution: **Persuasive**. The ticket reports three minor textual defects in the
published MedicationRequest pages and asks for them to be corrected verbatim:

1. §14.1.1 (Scope and Usage), sentence about ordering medications/substances
   not to be taken — currently reads *"It can can also be used to order
   medication or substances NOT be taken."* Required edits:
   - remove the duplicate `can`,
   - change `medication` to the plural `medications`,
   - insert the missing `to` (`NOT to be taken`).
2. §14.1.4.1, the value-set table heading currently rendered as
   *"MedicationReuqest Dispenser**Insruction**Codes"* — fix the typo
   `Insruction` → `Instruction` (and incidentally `Reuqest` → `Request`,
   which originates from the same source string).
3. §14.1.1, prose reference to *"using the Device Request"* — should be the
   single token / resource name **DeviceRequest** (matching the link target
   used elsewhere in the same paragraph).

No design changes; this is a documentation-only technical correction.

## Feature Proposal

### Problem Statement

The MedicationRequest specification page contains three small textual defects
that publish into the normative FHIR Core spec. They are visible in
`source/medicationrequest/medicationrequest-introduction.xml` and the
`valueset-medicationrequest-dispenser-instructions.xml` ValueSet description,
which together feed the rendered §14.1.1 and §14.1.4.1 sections of the
MedicationRequest page. The defects are grammatical / typographic only — the
semantics of MedicationRequest are unaffected — but the current text is
ungrammatical ("can can"), refers to *Device Request* as two words instead of
the FHIR resource name `DeviceRequest`, and surfaces the misspellings
"Reuqest" and "Insruction" in a value-set table title in the published spec.

### Proposed Change

Make three minimal edits to authored source files in `HL7/fhir`:

1. **`source/medicationrequest/medicationrequest-introduction.xml`, line 7**
   Replace the sentence

   > It can can also be used to order medication or substances NOT be taken.

   with

   > It can also be used to order medications or substances NOT to be taken.

2. **`source/medicationrequest/medicationrequest-introduction.xml`, line 8**
   In the sentence beginning *"This resource would not be used when ordering a
   device(s)…"*, change `Device Request` to `DeviceRequest`. To match the
   existing pattern in line 24 of the same file (`<a href="devicerequest.html">DeviceRequest</a>`),
   wrap the corrected token in the same anchor:

   > These types of devices would be ordered using the
   > `<a href="devicerequest.html">DeviceRequest</a>` or the SupplyRequest
   > resources.

   (`SupplyRequest` is left as-is for this ticket; it is already a single
   token here and a separate hyperlink decision.)

3. **`source/medicationrequest/valueset-medicationrequest-dispenser-instructions.xml`, line 23**
   Change the `description` value from

   > `MedicationRequest Dispenser Insruction Codes`

   to

   > `MedicationRequest Dispenser Instruction Codes`

   This is the string that the publisher renders as the §14.1.4.1 table
   heading the ticket calls out. (The sibling
   `codesystem-medicationrequest-dispenser-instructions.xml` already uses the
   correctly spelled `MedicationRequest dispenser instructions` and does not
   need to be changed.) The ticket also flags `Reuqest` in the rendered
   heading; that token does not appear in the source value sets in the clone,
   so no separate edit is required — the corrected description above resolves
   the reported text.

### Design Rationale

The resolution and the original ticket call for verbatim textual fixes; no
behavioural, structural, cardinality, or terminology-binding change is
proposed or required. Edits are made in the authored source (not the
generated HTML) because the `HL7/fhir` repo is published by the bespoke HL7
publisher, which renders `source/<resource>/<resource>-introduction.xml` and
the value-set XML into the resource page (per the FhirCore briefing's
"Authoring root(s)" / "Generated areas (do not edit)" rules). Hyperlinking
`DeviceRequest` mirrors the existing pattern already used three lines below
in the same file (`medicationrequest-introduction.xml` line 24), keeping the
prose internally consistent.
