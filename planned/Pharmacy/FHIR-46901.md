## Feature Proposal

### Problem Statement

`Medication` is reused to represent at least three distinct kinds of
records — generic formulations (e.g., Paracetamol 500mg tablets),
commercial / branded products (e.g., Tylenol 500mg tablets, possibly
with batch information), and compounded / extemporaneous preparations
(e.g., HC 1%, Salicylic Acid 5% in Glaxal Base). Today nothing on the
resource itself distinguishes these flavours. That matters for real
workflows: in CPOE prescribing, a clinician needs to search generic and
branded products but should not see patient-specific compounds, while a
dispense workflow needs the opposite slice. The resource has no element
or coded marker that lets servers, clients, or query clauses partition
`Medication` records by that distinction, so the gap blocks safe filtering
and search.

### Proposed Change

Implement the new standard extension exactly as the resolution requires,
in the `HL7/fhir-extensions` IG (the standard extensions pack, where all
new core-applicable extensions now live). The skeleton already exists at
`input/definitions/Medication/StructureDefinition-medication-type.xml` and
the value set at `input/definitions/Medication/ValueSet-medication-type.xml`.
Three things still need to land to fully satisfy the resolution:

1. **Confirm the extension definition matches the resolution and Grahame's
   "needs additional definitions" note.** Current state on `master` of
   `HL7/fhir-extensions` (verified directly in the cached clone):
   - `StructureDefinition/medication-type` (id: `medication-type`,
     name: `MedicationType`, title: `Medication Type`)
   - URL: `http://hl7.org/fhir/StructureDefinition/medication-type`
   - `kind = complex-type`, `type = Extension`, derivation `constraint`
   - Two `<context>` entries: `element` / `Medication` and `element` /
     `Medication.code.coding` ✓
   - `Extension.value[x]` is a single `CodeableConcept`, `min = 1`,
     `max = 1`, with example binding to
     `http://hl7.org/fhir/ValueSet/medication-type` ✓
   - `Extension.extension` `max = 0` (no nested sub-extensions) ✓
   - Description and short text were rewritten to explain how the
     extension is interpreted differently depending on context (whole
     `Medication` vs an individual `Medication.code.coding`).
   - Standards-status `trial-use`, FMM `1`, work-group `phx`.
   - The intended ValueSet binding URL is
     `http://hl7.org/fhir/ValueSet/medication-type` — this is correct
     and matches the published `ValueSet/medication-type`. (See Open
     Questions for the brief detour where the binding pointed at a
     `medication-type-vs` URL that doesn't exist as a sibling resource;
     commit `4b68eb9` reverted it back to the canonical URL above.)

2. **Add the supporting intro / notes prose for the extension page.**
   Other extensions in this IG ship a sibling
   `input/pagecontent/StructureDefinition-<id>-intro.md` that the
   publisher splices into the rendered extension page (cf.
   `StructureDefinition-patient-interpreterRequired-intro.md`). No
   `StructureDefinition-medication-type-intro.md` (or `.xml`) exists
   today. Add one that articulates, in user-facing prose, the two
   interpretation modes from the resolution:
   - On `Medication`: classifies the overall record (generic /
     formulation, branded / commercial product, extemporaneous
     preparation).
   - On `Medication.code.coding`: classifies the meaning of one
     individual coding within the resource's `code` CodeableConcept,
     so a `Medication` instance can carry several codings drawn from
     different SNOMED CT product axes and be unambiguous about which
     axis each coding represents.

3. **Add the two example instances explicitly demanded by the
   resolution.** The resolution language ("create examples of its use
   against Medication and of its use against Medication.code.coding") is
   the only piece not yet evidenced in the clone. Create:
   - A `Medication` example using the extension at the resource level
     (e.g., a Tylenol-style branded product where the extension's
     `valueCodeableConcept` codes `763158003 |Medicinal product
     (product)|`).
   - A `Medication` example using the extension at the
     `Medication.code.coding` level — the resource carries multiple
     codings in `Medication.code` (e.g., a branded product code and the
     underlying generic code), and each coding carries its own
     `medication-type` extension distinguishing branded vs generic.

The ValueSet itself does not need further work. `ValueSet/medication-type`
already enumerates the three SNOMED CT codes from the resolution (with
display strings) and is published from the same Medication folder, with
the SNOMED CT IPA copyright clause expected in this IG.

### Design Rationale

The resolution is unambiguous: a single optional **extension** carrying a
**CodeableConcept** with an **example-strength** binding to a small
SNOMED CT–rooted ValueSet. That deliberately avoids:

- Adding a new element to `Medication` itself, which would be a breaking
  change to a widely deployed resource and would require an N+1 release
  cycle for implementers.
- Using a required or extensible binding, which would force every
  Medication producer to map their existing terminology to this exact
  set of three SNOMED concepts — far stricter than the WG decided.
- Restricting the extension to only one context. By giving it both a
  `Medication`-level and a `Medication.code.coding`-level context, the
  same definition supports the two patterns implementers actually need
  (resource-level classification vs disambiguating individual codings),
  rather than spawning two near-duplicate extensions.

Ricardo's original ticket only asked the question; comments make it
clear the WG considered the alternative of a new top-level element on
`Medication` and rejected it on backward-compatibility grounds.

### Breaking Changes

No breaking changes identified. The artifact is a new optional standard
extension (`min = 0` at every use site, since it is invoked via
`Extension.value[x]` only when a producer chooses to populate it), with
an **example**-strength binding (no conformance burden on existing
producers). No existing element on `Medication` or any other resource
changes type, cardinality, or binding. No existing artifact is removed
or deprecated.
