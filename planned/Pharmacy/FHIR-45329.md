## Resolution Summary

**Resolution:** Persuasive.

From the resolution description:

> We will create an example MedicationRequest that echoes the work we are
> doing for MedicationAdministration in FHIR-33986 and MedicationRequest in
> FHIR-45330. This involves creating an example that references a
> DeviceDefinition in the MedicationRequest and a Device in the
> MedicationAdministration. The devices are listed in the comments.

Concretely, this ticket commits Pharmacy to authoring a set of example
instances that demonstrate ordering/administering a medication delivered via a
device (the motivating use case on Zulip was an oxygen prescription with
flow-rate settings). The devices called out in the ticket comments (by
Pharmacy co-chair john.hatem, 2024-05-20) are:

- IV pump (as both `Device` and `DeviceDefinition`)
- Triple Lumen Catheter (as `DeviceDefinition`)
- Nasal Canula (as `DeviceDefinition`)

The original ticket text also asks for the scenario to be carried through
"from request to dispense to administration to statement" — i.e., a linked
set of examples across MedicationRequest → MedicationDispense →
MedicationAdministration → MedicationStatement, using an oxygen / nasal
canula scenario as the canonical worked example.

Labels `R6-Examples, R6-Text` confirm this is an examples-and-narrative
change, not a schema change.

## Feature Proposal

### Problem Statement

The FHIR core Medication\* resources (MedicationRequest,
MedicationDispense, MedicationAdministration, MedicationStatement) already
support referencing a device — MedicationRequest has a `device` element and
the MedicationAdministration examples in FHIR-33986 exercise distal-port
selection on a triple lumen catheter — but the published R5/R6 example set
does not include a worked case of a medication that is *delivered through*
a device with device-controlled dose parameters. The community Zulip thread
[Example of oxygen prescription](https://chat.fhir.org/#narrow/stream/179166-implementers/topic/Example.20of.20oxygen.20prescription)
surfaced that implementers had to invent their own modelling, and Pharmacy
agreed on 2024-03-18 on a preferred shape. Without a published example,
downstream IGs and implementers have no canonical reference for:

- ordering oxygen (or another gas) with a flow-rate / delivery setting;
- naming the delivery device (nasal canula, IV pump, triple lumen catheter)
  in the request via `DeviceDefinition` and in the administration via
  `Device`;
- propagating that linkage through dispense, administration, and statement.

FHIR-45330 ("Add medical gases order examples to MedicationRequest") and
FHIR-33986 ("Review how to populate administered via distal port in triple
lumen catheter for MedicationAdministration") are the partner tickets; this
ticket is the umbrella that ties the end-to-end chain together. FHIR-45331
was also filed as a parallel "point at these examples from the Device
resource" follow-up.

### Proposed Change

Add a linked set of example instances in `HL7/fhir`, all authored under
`source/<resource>/`, and register each one in the matching
`list-<Resource>-examples.xml` manifest so the publisher picks them up. The
proposed shape:

1. **DeviceDefinition examples (new)** under `source/devicedefinition/`:
   - `devicedefinition-example-ivpump.xml` — IV pump (Device Definition).
   - `devicedefinition-example-triple-lumen-catheter.xml` — Triple Lumen
     Catheter.
   - `devicedefinition-example-nasal-canula.xml` — Nasal Canula.

2. **Device example (new)** under `source/device/`:
   - `device-example-ivpump.xml` — a concrete IV pump instance that
     references the IV-pump `DeviceDefinition` above. (The ticket comment
     lists "IV pump for Device" specifically.)

3. **MedicationRequest example (new)** under `source/medicationrequest/`:
   - `medicationrequest-oxygen-nasal-canula.xml` (or
     `medicationrequest03xx.xml` following the existing numeric naming
     convention already in use — see below) — orders oxygen delivered via
     the nasal canula, with `device` referencing the nasal-canula
     `DeviceDefinition` and `dosageInstruction.rate[x]` expressing the flow
     rate. This mirrors the scope of FHIR-45330 for medical gases.

4. **MedicationDispense example (new)** under `source/medicationdispense/`:
   - `medicationdispense03xx.xml` — dispense event against the
     oxygen MedicationRequest, carrying the link back to the request.

5. **MedicationAdministration example (new)** under
   `source/medicationadministration/`:
   - `medicationadministration03xx.xml` — administration record that
     references the MedicationRequest above and a concrete oxygen-delivery
     `Device` (e.g., the IV pump or a nasal-canula device instance aligned
     with FHIR-33986's distal-port work).

6. **MedicationStatement example (new)** under `source/medicationstatement/`:
   - `medicationstatementexampleNN.xml` (next free integer after
     `medicationstatementexample10.xml`) — patient-reported statement that
     closes the chain, linking back to the MedicationRequest.

7. **Manifest updates.** For each new file above, add an `<entry>` to the
   matching `list-<Resource>-examples.xml` with a `description`, `title`
   (matching the filename stem), and `item/reference` of
   `<Resource>/<id>`. Patterns to follow are already in-repo — e.g.,
   `source/medicationrequest/list-MedicationRequest-examples.xml` uses the
   `http://hl7.org/fhir/build/StructureDefinition/description` and `/title`
   extensions on each entry.

8. **Narrative tie-ins (label `R6-Text`).** Add a short paragraph in the
   MedicationRequest (`source/medicationrequest/medicationrequest-notes.xml`
   or the `-introduction.xml`, whichever the page currently uses for
   "Examples" prose) pointing readers at the oxygen chain as the worked
   example for device-mediated administration, and a matching paragraph in
   the MedicationAdministration notes cross-referencing the distal-port
   example from FHIR-33986. Exact file to be chosen by inspection at edit
   time — both the `-notes.xml` and `-introduction.xml` patterns exist in
   each resource folder.

### Design Rationale

- **Examples, not schema.** The labels (`R6-Examples`, `R6-Text`) and the
  resolution text are explicit: no structural/element changes are made.
  The Pharmacy resolution is to demonstrate a pattern that the existing
  R5/R6 structure already supports (MedicationRequest already has a
  `device` element; MedicationAdministration already references `Device`).
- **DeviceDefinition in the request, Device in the administration.** This
  matches the resolution description verbatim. Conceptually it aligns with
  the rest of FHIR: a request points at a *kind* of thing (definition),
  while an event points at the *actual* thing used (instance).
- **Oxygen as the canonical scenario.** It is the use case that drove the
  Zulip discussion and is the explicit motivator in the ticket; it also
  exercises a non-pill dose form (flow rate) which the existing
  MedicationRequest example set does not yet cover.
- **Use the existing numeric naming convention.**
  `source/medicationrequest/` already has `medicationrequest0301.xml` …
  `medicationrequest0350.xml` plus a handful of named examples
  (`medicationrequestexample1.xml` …). Either convention is acceptable per
  the existing mix, but numeric-series names keep these examples next to
  their Pharmacy peers in directory listings and in
  `list-MedicationRequest-examples.xml`.
- **No separate IG.** HL7/fhir is a bespoke-publisher repo
  (`source/fhir.ini` driven; not sushi/ig.ini), so the changes land as
  authored XML in the resource folders and as `<entry>` additions in the
  `list-*-examples.xml` manifests.

## Implementation Plan

### Step-by-Step Tasks

#### HL7/fhir

1. **Add DeviceDefinition — IV pump**
   - File: `source/devicedefinition/devicedefinition-example-ivpump.xml`
   - Action: Create a new `DeviceDefinition` XML following the pattern of
     `source/devicedefinition/devicedefinition-example.xml` and
     `devicedefinition-example-prodspec.xml`.
   - Details: `id="ivpump"`, `deviceName` / `classification` appropriate
     for an infusion pump. Include at least `deviceName`,
     `classification`, and `udiDeviceIdentifier` stubs to mirror the
     completeness of existing examples.

2. **Add DeviceDefinition — Triple lumen catheter**
   - File: `source/devicedefinition/devicedefinition-example-triple-lumen-catheter.xml`
   - Action: Create a new `DeviceDefinition`.
   - Details: `id="triple-lumen-catheter"`; model the three lumens via
     `property` or `component` entries so that FHIR-33986's distal-port
     reference has something concrete to point at. Confirm with FHIR-33986
     before committing the shape.

3. **Add DeviceDefinition — Nasal canula**
   - File: `source/devicedefinition/devicedefinition-example-nasal-canula.xml`
   - Action: Create a new `DeviceDefinition`.
   - Details: `id="nasal-canula"`; include classification codings and a
     human-readable `deviceName` of "Nasal Canula".

4. **Register the new DeviceDefinition examples**
   - File: `source/devicedefinition/list-DeviceDefinition-examples.xml`
   - Action: Add three `<entry>` blocks following the in-file pattern —
     each with the `build/StructureDefinition/description` and `/title`
     extensions and an `<item><reference value="DeviceDefinition/<id>"/>`
     plus a user-facing `<display>`.
   - Details: Titles should match the new filename stems
     (`devicedefinition-example-ivpump`, etc.).

5. **Add Device — IV pump instance**
   - File: `source/device/device-example-ivpump.xml`
   - Action: Create a concrete `Device` that references the IV-pump
     DeviceDefinition (`definition.reference =
     DeviceDefinition/ivpump`).
   - Details: Follow the style of `source/device/device-example-udi1.xml`
     / `device-example-pacemaker.xml`. Include `status`, `serialNumber`
     (placeholder), and a `type` coding consistent with the
     DeviceDefinition.

6. **Add MedicationRequest — oxygen via nasal canula**
   - File: `source/medicationrequest/medicationrequest0351.xml` (or
     lowest unused `03xx` number at the time of authoring)
   - Action: Create the oxygen prescription that demonstrates the
     Pharmacy-agreed pattern.
   - Details:
     - `medication.reference` or `medication.concept` for oxygen (RxNorm
       or SNOMED CT, per existing example conventions).
     - `device` referencing
       `DeviceDefinition/nasal-canula`.
     - `dosageInstruction[0].doseAndRate.rateQuantity` expressing the
       flow rate (e.g., `2 L/min`), or `rateRange` for a titration
       range, to match the Zulip-discussed modelling.
     - `subject`, `requester`, `authoredOn` set to plausible values
       consistent with the existing `medicationrequest03xx` examples.

7. **Register the new MedicationRequest example**
   - File: `source/medicationrequest/list-MedicationRequest-examples.xml`
   - Action: Add an `<entry>` for the new file with description, title,
     and `item/reference value="MedicationRequest/<id>"`.
   - Details: Use a title such as "Oxygen prescription via nasal canula"
     in the `description` extension.

8. **Add MedicationDispense — oxygen dispense**
   - File: `source/medicationdispense/medicationdispense0336.xml` (or
     next free number)
   - Action: Create a `MedicationDispense` that references the
     MedicationRequest from step 6 via
     `authorizingPrescription[0].reference`.
   - Details: Include `medication`, `quantity`, `whenHandedOver`, and any
     `dosageInstruction` override consistent with the ordered flow rate.

9. **Register the new MedicationDispense example**
   - File: `source/medicationdispense/list-MedicationDispense-examples.xml`
   - Action: Add an `<entry>` for the new file.

10. **Add MedicationAdministration — oxygen administration**
    - File: `source/medicationadministration/medicationadministration0314.xml`
      (or next free number)
    - Action: Create a `MedicationAdministration` that references the
      MedicationRequest from step 6 (`request.reference`) and a concrete
      `Device` (`device[0].reference`). For the administration device,
      either reuse `Device/ivpump` from step 5 or author an
      oxygen-delivery Device in this file's folder that references
      `DeviceDefinition/nasal-canula`; choose the option that is
      consistent with the distal-port pattern adopted in FHIR-33986.
    - Details: Include `effectiveDateTime`, `status = completed`, and
      `dosage.rateQuantity` matching the ordered flow rate.

11. **Register the new MedicationAdministration example**
    - File: `source/medicationadministration/list-MedicationAdministration-examples.xml`
    - Action: Add an `<entry>` for the new file.

12. **Add MedicationStatement — patient-reported use**
    - File: `source/medicationstatement/medicationstatementexample11.xml`
    - Action: Create a `MedicationStatement` with
      `derivedFrom` (or `basedOn`, per the R6 shape of the resource at
      the time of authoring) pointing back at the MedicationRequest
      from step 6, closing the request-to-statement chain called out in
      the ticket body.
    - Details: `status = recorded`, `subject` matching the other
      examples, `medication` pointing at oxygen.

13. **Register the new MedicationStatement example**
    - File: `source/medicationstatement/list-MedicationStatement-examples.xml`
      *(create if absent, following the
      `list-MedicationDispense-examples.xml` template)*
    - Action: Add an `<entry>` for the new file.

14. *(Optional, label `R6-Text`)* **Narrative tie-in on MedicationRequest**
    - File: `source/medicationrequest/medicationrequest-notes.xml`
    - Action: Add a short paragraph introducing the oxygen / nasal-canula
      worked example and pointing implementers at the linked chain
      (dispense, administration, statement).
    - Details: Match the existing heading structure in the file.

15. *(Optional, label `R6-Text`)* **Narrative tie-in on MedicationAdministration**
    - File: `source/medicationadministration/medicationadministration-notes.xml`
    - Action: Add a short paragraph cross-referencing the new
      administration example and the distal-port pattern from FHIR-33986.
