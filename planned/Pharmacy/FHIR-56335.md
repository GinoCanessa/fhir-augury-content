## Resolution Summary

**Resolution:** Persuasive with Modification.

> "Remove the medication specific codesystem and use the codesystem from the
> pattern, making sure that the ValueSet includes only the current existing
> codes — i.e. not adding any codes, just switching code systems."

Concretely: retire `http://hl7.org/fhir/CodeSystem/medicationrequest-intent`,
re-base the `medicationrequest-intent` ValueSet on the shared
`http://hl7.org/fhir/request-intent` CodeSystem (the canonical Request
pattern code system), and constrain the `compose` so the ValueSet
enumerates exactly the eight codes that exist in the current
medication-specific CodeSystem today: `proposal`, `plan`, `order`,
`original-order`, `reflex-order`, `filler-order`, `instance-order`, and
`option`. The `directive` code that exists in `request-intent` must
**not** be picked up — the resolution explicitly forbids adding codes.

The MedicationRequest binding URL
(`http://hl7.org/fhir/ValueSet/medicationrequest-intent`) and the
ValueSet's identity (id, OID, canonical URL) are preserved so existing
consumers and the StructureDefinition binding continue to resolve.

## Feature Proposal

### Problem Statement

`MedicationRequest.intent` is bound (required) to
`http://hl7.org/fhir/ValueSet/medicationrequest-intent`, which today
composes from a medication-specific CodeSystem
`http://hl7.org/fhir/CodeSystem/medicationrequest-intent`
(`source/medicationrequest/codesystem-medicationrequest-intent.xml`).
Every code in that medication-specific CodeSystem (`proposal`, `plan`,
`order`, `original-order`, `reflex-order`, `filler-order`,
`instance-order`, `option`) already exists in the cross-resource Request
pattern's `request-intent` CodeSystem
(`http://hl7.org/fhir/request-intent`, defined in
`source/request/codesystem-request-intent.xml`).

Maintaining a parallel medication-specific CodeSystem with codes that
duplicate the Request-pattern CodeSystem causes:

- Divergence risk: definitions can drift between the two CodeSystems
  (definitions are already worded slightly differently between the two
  files in this clone — e.g., the `proposal` definition wording differs).
- Cross-resource interoperability friction: implementers handling
  multiple Request-pattern resources have to map between two CodeSystems
  for the same conceptual codes.
- Inconsistency with the Request pattern, which the MedicationRequest
  StructureDefinition already maps to (`workflow` mapping
  `Request.intent`).

### Proposed Change

1. **Delete the medication-specific CodeSystem.** Remove
   `source/medicationrequest/codesystem-medicationrequest-intent.xml`
   (URL `http://hl7.org/fhir/CodeSystem/medicationrequest-intent`,
   OID `2.16.840.1.113883.4.642.4.1378`, id `medicationrequest-intent`).

2. **Re-base the existing ValueSet on `request-intent`.** Update
   `source/medicationrequest/valueset-medicationrequest-intent.xml` so
   that:

   - `id`, `url`
     (`http://hl7.org/fhir/ValueSet/medicationrequest-intent`),
     `identifier` (OID `2.16.840.1.113883.4.642.3.1321`), `name`
     (`MedicationRequestIntent`), `title`, `status`, `experimental`,
     publisher, contact, description, and the workgroup /
     standards-status extensions remain unchanged.
   - The `compose` block changes from a bare include of
     `http://hl7.org/fhir/CodeSystem/medicationrequest-intent` to an
     enumerated include of the eight current codes against
     `http://hl7.org/fhir/request-intent`. Example shape:

     ```xml
     <compose>
       <include>
         <system value="http://hl7.org/fhir/request-intent"/>
         <concept><code value="proposal"/></concept>
         <concept><code value="plan"/></concept>
         <concept><code value="order"/></concept>
         <concept><code value="original-order"/></concept>
         <concept><code value="reflex-order"/></concept>
         <concept><code value="filler-order"/></concept>
         <concept><code value="instance-order"/></concept>
         <concept><code value="option"/></concept>
       </include>
     </compose>
     ```

   The `directive` code (present in `request-intent` but **not** in the
   current `medicationrequest-intent` CodeSystem) must be omitted, per
   the resolution's "not adding any codes" constraint.

3. **Remove the deleted CodeSystem from the OID registry.** Drop the
   `http://hl7.org/fhir/CodeSystem/medicationrequest-intent =
   2.16.840.1.113883.4.642.4.1378` line from `source/oids.ini`
   (currently line 1082). The ValueSet OID line at line 614 stays.

4. **No change to the binding.** The `MedicationRequest.intent` element
   in `source/medicationrequest/structuredefinition-MedicationRequest.xml`
   already binds to
   `http://hl7.org/fhir/ValueSet/medicationrequest-intent` (line 310,
   strength `required`); since we are preserving the ValueSet's URL and
   id, the binding need not change. Likewise, the resource diagram
   (`medicationrequest.svg` line 215) references
   `valueset-medicationrequest-intent.html` and continues to resolve.

### Design Rationale

- **Aligns with the Request pattern.** MedicationRequest is mapped to
  `Request.intent` via the `workflow` mapping (StructureDefinition
  line 314). Reusing the Request-pattern CodeSystem makes that mapping
  semantically literal at the terminology layer rather than only at the
  conceptual layer.
- **Preserves the ValueSet identity.** Keeping the same ValueSet
  `id`/`url`/OID is the lowest-impact way to retire the duplicate
  CodeSystem without forcing implementers to chase a new binding URL.
  Existing instances and the MedicationRequest StructureDefinition
  binding need no edit.
- **Honors "not adding any codes".** Enumerating only the eight current
  codes (rather than including the whole `request-intent` CodeSystem)
  keeps the value set surface unchanged for existing implementers; in
  particular, `directive` — which exists in `request-intent` but never
  in `medicationrequest-intent` — stays out of MedicationRequest's
  bound value set. Adding it would be an in-scope topic for a separate
  ticket.
- **Alternative considered: deprecating but retaining the medication
  CodeSystem.** Rejected because the resolution explicitly says "remove
  the medication codesystem" and the duplication itself is the problem
  being fixed.

### Breaking Changes

This is a backward-incompatible change for any consumer that resolves
the medication-specific CodeSystem URL directly:

- **CodeSystem `http://hl7.org/fhir/CodeSystem/medicationrequest-intent`
  (OID `2.16.840.1.113883.4.642.4.1378`) is removed.** Resources or
  systems that reference this URL/OID (e.g., `Coding.system =
  "http://hl7.org/fhir/CodeSystem/medicationrequest-intent"` on
  pre-existing instances) will no longer resolve. Conformant
  MedicationRequest instances should now use
  `system = "http://hl7.org/fhir/request-intent"`.
- **No breaking changes at the binding layer.** The ValueSet URL, OID,
  and code set are preserved. `MedicationRequest.intent` continues to
  accept exactly the same eight codes; only the underlying
  `system` namespace shifts.
- **No new required elements** are introduced.

### Related Specifications

  `request-intent` CodeSystem and ValueSet that the medication-specific
  ValueSet now reuses. No edits needed here; this ticket simply consumes

  (`structuredefinition-MedicationRequest.xml:310`) keeps pointing at

- **Other Request-pattern resources** (e.g., `ServiceRequest`,
  `CommunicationRequest`, `MedicationDispense`-side requests, etc.)

## Implementation Plan

### Prerequisites

- None. The target CodeSystem (`http://hl7.org/fhir/request-intent`)
  already exists in this clone at
  `source/request/codesystem-request-intent.xml` and contains all eight
  codes the ValueSet needs to enumerate.

### Step-by-Step Tasks

#### HL7/fhir

1. **Re-base `medicationrequest-intent` ValueSet on `request-intent`.**
   - File: `source/medicationrequest/valueset-medicationrequest-intent.xml`
   - Action: Replace the existing `<compose>` block (which currently
     contains a single `<include>` with
     `<system value="http://hl7.org/fhir/CodeSystem/medicationrequest-intent"/>`)
     with an `<include>` whose `<system>` is
     `http://hl7.org/fhir/request-intent` and which lists exactly eight
     `<concept>` children for the codes `proposal`, `plan`, `order`,
     `original-order`, `reflex-order`, `filler-order`, `instance-order`,
     and `option`, in that order (matches today's hierarchy traversal in
     the medication-specific CodeSystem).
   - Details: Do **not** include `directive`. Do **not** alter the
     ValueSet's `id`, `url`, `identifier` (OID), `name`, `title`,
     `status`, `experimental`, publisher, contact, description, or the
     `structuredefinition-wg` (`phx`) and
     `structuredefinition-standards-status` (`normative`) extensions.
     Each `<concept>` child only needs `<code value="…"/>`; explicit
     `<display>` is optional and, if added, should match the Request
     pattern's display strings (e.g., `Proposal`, `Plan`, `Order`,
     `Original Order`, `Reflex Order`, `Filler Order`, `Instance Order`,
     `Option`). Verify well-formedness against the FHIR XML schema.

2. **Delete the medication-specific CodeSystem file.**
   - File: `source/medicationrequest/codesystem-medicationrequest-intent.xml`
   - Action: Remove the file from the working tree (`git rm`).
   - Details: This retires CodeSystem URL
     `http://hl7.org/fhir/CodeSystem/medicationrequest-intent` (OID
     `2.16.840.1.113883.4.642.4.1378`). The ValueSet that previously
     referenced it has already been updated in Task 1, so nothing else
     in `source/medicationrequest/` should resolve to this CodeSystem
     after the change.

3. **Remove the retired CodeSystem from the OID registry.**
   - File: `source/oids.ini`
   - Action: Delete the line
     `http://hl7.org/fhir/CodeSystem/medicationrequest-intent = 2.16.840.1.113883.4.642.4.1378`
     (currently line 1082).
   - Details: Leave the ValueSet OID line at line 614
     (`http://hl7.org/fhir/ValueSet/medicationrequest-intent =
     2.16.840.1.113883.4.642.3.1321`) untouched — that ValueSet still
     exists and keeps its OID.

4. **Verify no remaining references to the retired CodeSystem URL.**
   - File: repository-wide search under `source/`.
   - Action: Run a grep for both `CodeSystem/medicationrequest-intent`
     and the literal OID `2.16.840.1.113883.4.642.4.1378`. Expected
     post-change result: zero hits (the only authored references in
     this clone today are the file deleted in Task 2 and the
     `oids.ini` entry removed in Task 3).
   - Details: No edits expected to
     `structuredefinition-MedicationRequest.xml` (the binding points at
     the ValueSet URL, which is unchanged), to `medicationrequest.svg`
     (its `xlink:href` is `valueset-medicationrequest-intent.html`,
     also unchanged), or to other files in `source/medicationrequest/`.

### Open Questions

  `<display>` strings be included explicitly in the new `<include>`,
  or omitted (letting the publisher take them from the source
  CodeSystem)? The current ValueSet does not enumerate concepts, so
  there is no precedent in this artifact. Defer to Pharmacy WG /
  publisher convention; either is conformant.
- **`directive` deliberately excluded** — confirm with the Pharmacy WG
  that no follow-up ticket is intended to add `directive` to
  `MedicationRequest.intent`. The resolution is unambiguous on this
  point, but it is worth surfacing because `directive` is the only
  semantic gap between `request-intent` and the retired
  `medicationrequest-intent` CodeSystem.
