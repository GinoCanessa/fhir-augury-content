## Feature Proposal

### Problem Statement

The FHIR QA build has flagged the MedicationKnowledge resource with a large batch of "must have a binding", "name overlaps with parent", "should use internally defined codes", and "RIM Mapping is required" findings. These make the resource fail the publisher's QA rules and leave downstream implementers without normative terminology guidance for every coded category field. The ticket was taken by Pharmacy as a resource-wide clean-up and rolled into the R5 changes already in-flight for MedicationKnowledge.

### Proposed Change

All changes are to the MedicationKnowledge StructureDefinition in `HL7/fhir` `source/medicationknowledge/`. In the clone at `db79dcd`, the R5 reshape is partly applied (`administrationGuideline` has been renamed to `indicationGuideline.dosingGuideline`; `kineticCharacteristic` and `packaging.material` have been removed; `cost.effectiveDate`, `cost.source`, `cost.cost[x]` now exist; `definitional.*` tree has been introduced). The remaining gaps against this ticket's resolution are:

1. **Add example-strength bindings** on the following elements in `source/medicationknowledge/structuredefinition-MedicationKnowledge.xml` (differential). All currently have **no `<binding>` child** (verified by scanning the differential):

   | Element path (current) | ValueSet to bind | Source | Strength |
   |---|---|---|---|
   | `MedicationKnowledge.relatedMedicationKnowledge.type` | `http://hl7.org/fhir/ValueSet/medicationknowledge-product-type` (new; shared with `productType`) | new exemplar in `source/medicationknowledge/` | example |
   | `MedicationKnowledge.productType` | `http://hl7.org/fhir/ValueSet/medicationknowledge-product-type` (new) | new exemplar | example |
   | `MedicationKnowledge.monograph.type` | `http://hl7.org/fhir/ValueSet/medicationknowledge-monograph-type` (new, seeded from V3 `ActMedicationDocumentCode`: MONDDI, MONEDL, MONEDA, MONCPR, MONCPA, PROTIND, MANUF) | new exemplar | example |
   | `MedicationKnowledge.cost.type` | `http://hl7.org/fhir/ValueSet/medication-cost-category` (already present at `valueset-medication-cost-category.xml`) | existing | example |
   | `MedicationKnowledge.monitoringProgram.type` | `http://terminology.hl7.org/ValueSet/v3-ActMonitoringProtocolCode` | UTG (existing) | example |

   - `MedicationKnowledge.medicineClassification.classification` → `MedicationKnowledge.medicineClassification.value`.
   - `MedicationKnowledge.regulatory.regulatoryAuthority` → `MedicationKnowledge.regulatory.authority`.

3. **`status` element:** The ticket's resolution for the "internally defined codes" warning needs final sign-off (comment defers to Lloyd). In the clone, `MedicationKnowledge.status` is currently bound `required` to `http://hl7.org/fhir/ValueSet/publication-status`. If that stays, the ticket's `status` finding is already satisfied by a core-defined ValueSet and the local `valueset-medicationknowledge-status.xml` / `codesystem-medicationknowledge-status.xml` pair should either be retired or retained for jurisdictional reuse; this is flagged in **Open Questions** below and must not be silently changed.

4. **RIM Mapping:** Port the Medication resource's RIM mapping into MedicationKnowledge. The SD already declares the `rim` mapping identity (`http://hl7.org/v3`) in its header, so the work is adding `<mapping identity="rim" map="…"/>` rows on `MedicationKnowledge` (and sub-paths where Medication has them) in the differential.

5. **Package-type CodeSystem definition:** Add a `<definition>` to the `amp` concept of `http://terminology.hl7.org/CodeSystem/medicationknowledge-package-type` in the package-type CodeSystem file in this repo.

6. **Element definition polish** (non-binding):
   - `monitoringProgram.type` — append REMS and PDMP as example programs in the element short/definition text.
   - `packaging.material`: *(Not applicable — the element has been removed from the current differential; record the ticket resolution as satisfied via removal.)*

### Design Rationale

- **Exemplar (example-strength) bindings** are the established FHIR pattern for category-style `CodeableConcept.type` fields whose values are jurisdiction- or vendor-specific; this matches the Pharmacy WG direction captured in the disposition comment ("Create exemplar value set for this one").
- **Reuse existing UTG/Terminology ValueSets** (`v2-0477`, `v3-ActSubstanceAdminSubstitutionCode`, `v3-ActMonitoringProtocolCode`, `medicationknowledge-characteristic`, `medication-cost-category`) wherever the WG pointed to one in the disposition — avoids inventing parallel sets and keeps UTG as the terminology home.
- **Sharing** the same exemplar ValueSet between `productType` and `relatedMedicationKnowledge.type` follows the disposition comment ("Use same exemplar as for Product Type") and reduces maintenance cost.
- **Renaming `classification` → `value` and `regulatoryAuthority` → `authority`** removes the publisher warnings about child/parent name overlap; these are **breaking** naming changes consistent with the R5 reshape (which has already renamed other MedicationKnowledge elements in this clone).
- **RIM mapping inheritance** from Medication aligns with the disposition ("Use same mapping as for Medication") and satisfies the publisher's "RIM Mapping is required" gate.
