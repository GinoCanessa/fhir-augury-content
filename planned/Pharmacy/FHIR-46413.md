## Resolution Summary

Resolution: **Persuasive**.

The agreed change is to create a **standard extension (0..\*) on `Medication`**
that captures additional characteristics as a typed key/value pair. Verbatim
from the resolution description:

> We will create a standard extension (0..\*) for Medication to provide
> characteristcs:
>
> - characteristic.type CodeableConcept 1..1
> - characteristic.value[x] 1..1 boolean, dateTime, decimal, integer, string,
>   CodeableConcept, Quantity, Ratio

The shape mirrors the `characteristic` backbone elements already present on
`MedicationKnowledge` and `MedicinalProductDefinition`, but is delivered as a
standard extension on `Medication` (which has no native `characteristic`
element). The allowed value datatypes (`boolean`, `dateTime`, `decimal`,
`integer`, `string`, `CodeableConcept`, `Quantity`, `Ratio`) match the set
adopted by IHE MPD and called out by the reporter in the 2025-05-13 comment.

## Feature Proposal

### Problem Statement

Analysis of national `Medication` implementations shows that many
jurisdictions define their own bespoke extensions to carry medication
characteristics (e.g., colour, shape, imprint, storage temperature, package
size). The result is a proliferation of incompatible national extensions for
overlapping concepts. `Medication` itself has no general mechanism for
expressing a typed characteristic, even though sister resources
(`MedicationKnowledge.drugCharacteristic`, `MedicinalProductDefinition.characteristic`)
already define this exact pattern as a backbone element. A single, common,
HL7-published extension on `Medication` reduces national fragmentation while
keeping the core resource compact.

### Proposed Change

Add a complex standard extension to the FHIR Extensions Pack
(`HL7/fhir-extensions`):

- **Canonical URL:** `http://hl7.org/fhir/StructureDefinition/medication-characteristic`
- **Name / Title:** `MedicationCharacteristic` / "Medication Characteristic"
- **Context:** element `Medication` (so `Medication.extension` can carry
  zero-to-many of these)
- **Cardinality on host:** 0..\* (per resolution)
- **Type:** complex extension with two named sub-extension slices:

  | Slice | URL | Cardinality | Type |
  |-------|-----|-------------|------|
  | `type`  | `type`  | 1..1 | `CodeableConcept` (binding `example` → `http://hl7.org/fhir/ValueSet/medication-characteristic-type`) |
  | `value` | `value` | 1..1 | choice of `boolean`, `dateTime`, `decimal`, `integer`, `string`, `CodeableConcept`, `Quantity`, `Ratio` |

- **Top-level `Extension.value[x]`:** 0..0 (complex extension, value carried
  by sub-extensions).
- **Work group / FMM / status:** `phx` / FMM 1 / `trial-use` (consistent with
  other Pharmacy-owned extensions in the pack).

Supporting terminology, also published in `HL7/fhir-extensions`:

- **CodeSystem:** `http://hl7.org/fhir/medication-characteristic-type`
  (`MedicationCharacteristicType`, `experimental = true`, `complete`,
  example concepts adapted from
  `MedicationKnowledge.drugCharacteristic.type`, e.g., `imprint`, `size`,
  `color`, `shape`, `coating`, etc.).
- **ValueSet:** `http://hl7.org/fhir/ValueSet/medication-characteristic-type`
  (`MedicationCharacteristicType`, `experimental = true`, includes the above
  CodeSystem).

### Design Rationale

- Mirrors the established `characteristic.type` / `characteristic.value[x]`
  pattern from `MedicationKnowledge` and `MedicinalProductDefinition`, so
  implementers familiar with those resources can reuse the same modelling.
- Delivered as a **standard extension** rather than adding a backbone
  element to core `Medication`, because (a) the core team's general policy
  for `Medication` is to keep the resource lean and push optional structure
  to extensions, and (b) doing so avoids breaking changes for the R5 wire
  format.
- The value `[x]` set (`boolean`, `dateTime`, `decimal`, `integer`, `string`,
  `CodeableConcept`, `Quantity`, `Ratio`) matches the set the reporter and
  IHE MPD converged on in the 2025-05-13 comment, intentionally excluding
  attachment / reference types that would invite scope creep.
- Binding strength on `type` is `example` (not `preferred` or `extensible`)
  so that jurisdictions can keep using their own characteristic codes
  without conformance failures, while still having a published illustrative
  set to point at — addressing the comment that flagged the need to
  "review existing extensions" before locking down terminology.
- An "alternative" of adding a `classification` element to core
  `Medication` was raised in the 2024-09-27 comment, but that addresses a
  different concern (categorising the medication itself, not describing
  observable physical properties); it is out of scope for this resolution
  and is being handled separately (see also the sibling
  `medication-classification` extension already in the pack).

## Impact Analysis

### Step-by-Step Tasks

#### HL7/fhir-extensions

1. **Create the Medication characteristic extension StructureDefinition.**
   - File: `input/definitions/Medication/StructureDefinition-medication-characteristic.xml`
   - Action: Add a new complex Extension StructureDefinition.
   - Details:
     - `id` = `medication-characteristic`,
       `url` = `http://hl7.org/fhir/StructureDefinition/medication-characteristic`,
       `name` = `MedicationCharacteristic`,
       `title` = `Medication Characteristic`.
     - Set `kind = complex-type`, `type = Extension`,
       `baseDefinition = http://hl7.org/fhir/StructureDefinition/Extension`,
       `derivation = constraint`.
     - `context.type = element`, `context.expression = Medication`.
     - WG/FMM/standards-status extensions: `phx`, `1`, `trial-use`.
     - Differential:
       - `Extension.extension` min `2`.
       - Slice `type` (1..1, `fixedUri = "type"`, `value[x] = CodeableConcept`,
         binding `example` → `http://hl7.org/fhir/ValueSet/medication-characteristic-type`).
       - Slice `value` (1..1, `fixedUri = "value"`, `value[x]` choice of
         `boolean | dateTime | decimal | integer | string | CodeableConcept |
         Quantity | Ratio`).
       - `Extension.url` `fixedUri` = the canonical above.
       - `Extension.value[x]` `max = 0`.

2. **Create the example CodeSystem for characteristic types.**
   - File: `input/definitions/Medication/CodeSystem-medication-characteristic-type.xml`
   - Action: Add a new CodeSystem.
   - Details:
     - `id` = `medication-characteristic-type`,
       `url` = `http://hl7.org/fhir/medication-characteristic-type`,
       `name` = `MedicationCharacteristicType`,
       `title` = "Medication Characteristic Type".
     - `experimental = true`, `caseSensitive = true`, `content = complete`.
     - WG = `phx`, FMM = `1`, status = `trial-use`.
     - Seed concepts adapted from `MedicationKnowledge.drugCharacteristic.type`
       (e.g., `imprint`, `size`, `color`, `shape`, `coating`, `scoring`,
       `flavor`, `image`); each concept needs `display` and `definition`.

3. **Create the bound ValueSet for characteristic types.**
   - File: `input/definitions/Medication/ValueSet-medication-characteristic-type.xml`
   - Action: Add a new ValueSet that includes the CodeSystem from step 2.
   - Details:
     - `id` = `medication-characteristic-type`,
       `url` = `http://hl7.org/fhir/ValueSet/medication-characteristic-type`,
       `name` = `MedicationCharacteristicType`,
       `experimental = true`.
     - `compose.include[0].system = http://hl7.org/fhir/medication-characteristic-type`
       (no concept filter — include all).
     - Same WG/FMM/status metadata as the CodeSystem.

4. **Register the new artifacts with the IG.**
   - File: `extensions-ig.xml` (and/or the relevant page-content index in
     `input/pagecontent/`).
   - Action: Confirm the new files are picked up by the IG-publisher
     (file-name discovery should auto-list them; verify they appear in the
     generated artifact index after a local build).
   - Details: Run `_genonce.sh` (or `.bat`) and inspect
     `output/artifacts.html` for the three new entries; resolve any QA
     warnings raised against the new files.

5. **Add a usage / examples paragraph (optional but recommended).**
   - File: a Pharmacy-WG page under `input/pagecontent/` (e.g., a
     `medication-extensions.md` if one exists; otherwise add a short
     section to the existing extensions overview).
   - Action: Document when implementers should reach for
     `medication-characteristic` versus the existing
     `MedicationKnowledge.drugCharacteristic`, and show one inline JSON or
     XML example using a CodeableConcept value (e.g., colour) and one using
     a Quantity value (e.g., storage temperature).
