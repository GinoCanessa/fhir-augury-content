## Resolution Summary

**Resolution:** Persuasive

**Resolution description (verbatim):** *"since this is a new element we can change as requested"*

The committee accepted the request to relocate the new boolean
`MedicationRequest.isRecordOfRequest` element so that it sits **immediately
after `MedicationRequest.supportingInformation`** in the differential. The
rationale captured in the ticket is that placing the modifier-style flag
adjacent to `supportingInformation` makes the relationship between the two
clearer (the flag explains the context of the supporting information /
informationSource pair). Because `isRecordOfRequest` is a brand-new element
in this cycle, reordering it carries no downstream conformance baggage.

## Feature Proposal

### Problem Statement

`MedicationRequest.isRecordOfRequest` is a new boolean element added to
MedicationRequest to indicate that the instance is *a record of* a request
that was placed elsewhere, rather than the authoritative request itself. As
currently authored in `source/medicationrequest/structuredefinition-MedicationRequest.xml`
the element is positioned after `supportingInformation`, `authoredOn`, and
`requester` (line 628), which separates it from the related context-setting
elements `informationSource` (line 491) and `supportingInformation` (line
544). Reviewers reading the resource top-to-bottom (or the rendered logical
view) encounter the "is this a real request or a record of one?" flag long
after the elements whose meaning it conditions.

The submitter (Ron G. Parker, ballot comment 31) asked that the flag be
moved up next to `supportingInformation`, so the contextual elements
(`informationSource` → `supportingInformation` → `isRecordOfRequest`) read
together.

### Proposed Change

Reorder one element in the MedicationRequest StructureDefinition
differential so that the resulting authoring order is:

```
…
MedicationRequest.informationSource    (unchanged, stays at its current slot)
MedicationRequest.encounter            (unchanged)
MedicationRequest.supportingInformation
MedicationRequest.isRecordOfRequest    ← moved here
MedicationRequest.authoredOn
MedicationRequest.requester
MedicationRequest.performerType
…
```

Concretely, in
`source/medicationrequest/structuredefinition-MedicationRequest.xml` move
the entire `<element id="MedicationRequest.isRecordOfRequest">…</element>`
block (today at lines 628–645) so that it appears immediately after the
closing `</element>` of `MedicationRequest.supportingInformation` (today
ending at line 567) and immediately before
`<element id="MedicationRequest.authoredOn">` (today at line 568).

No definitional content changes:

- **Path:** `MedicationRequest.isRecordOfRequest` (unchanged)
- **Short:** *"Whether this is record of a Medication Request or the actual request itself"* (unchanged)
- **Definition:** *"The information in this instance is a record of the Medication Request and is not request or order itself and should not be considered as such, i.e. the information should not be used for medication dispense | supply and administration."* (unchanged)
- **Comment:** *"If not populated, then assume that this is the original record and not reported"* (unchanged)
- **Cardinality:** 0..1 (unchanged)
- **Type:** `boolean` (unchanged)
- **isModifier:** still commented out (unchanged) — see Open Questions
- **isSummary:** `true` (unchanged)
- **rim mapping:** `.participation[typeCode=INF].role` (unchanged)

Snippet showing the resulting order around the move target (illustrative):

```xml
    <element id="MedicationRequest.supportingInformation">
      <path value="MedicationRequest.supportingInformation"/>
      <!-- … unchanged … -->
    </element>
    <element id="MedicationRequest.isRecordOfRequest">
      <path value="MedicationRequest.isRecordOfRequest"/>
      <short value="Whether this is record of a Medication Request or the actual request itself"/>
      <!-- …block moved verbatim… -->
    </element>
    <element id="MedicationRequest.authoredOn">
      <path value="MedicationRequest.authoredOn"/>
      <!-- … unchanged … -->
    </element>
```

The published (snapshot) view will be regenerated by the HL7 publisher from
the differential; do not hand-edit any `<snapshot>` block.

In addition, FHIR XML instances must respect the StructureDefinition
element order. The example file
`source/medicationrequest/medicationrequest0340.xml` currently emits
`<isRecordOfRequest>` after `<requester>` (line 37) — this becomes invalid
once the differential is reordered, so the example must be re-ordered to
emit `<isRecordOfRequest>` immediately after any `<supportingInformation>`
slot and before `<authoredOn>` / `<requester>`.

### Design Rationale

- **Submitter intent.** The ballot comment (Ron G. Parker, "Comment 31")
  explicitly asks for adjacency to `supportingInformation` because the flag
  "give[s] the context of the supporting information"; reordering is the
  minimal change that satisfies the request.
- **No conformance cost.** `isRecordOfRequest` is being introduced in this
  cycle, so there is no installed base depending on its current ordinal
  position. The committee's resolution explicitly leans on this fact
  ("since this is a new element we can change as requested").
- **Adjacency to related context elements.** `informationSource`,
  `supportingInformation`, and `isRecordOfRequest` together describe how
  the request was sourced/contextualised; co-locating them improves
  readability of both the rendered specification and the differential
  itself. The `informationSource` definition already cross-references
  `.isRecordOfRequest` (line 494 of the StructureDefinition), making the
  proximity natural.
- **Alternatives considered.** Inserting `isRecordOfRequest` directly
  after `informationSource` (i.e., before `encounter` and
  `supportingInformation`) was implicitly rejected by the submitter's
  phrasing — they specifically asked for "right after
  supportingInformation". No alternative placement was floated in the
  comments.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Source of truth for MedicationRequest's StructureDefinition
  and examples. The reorder lives entirely here.
- **Affected Files:**
  - `source/medicationrequest/structuredefinition-MedicationRequest.xml` —
    move the `MedicationRequest.isRecordOfRequest` `<element>` block
    (currently lines 628–645) up to sit between
    `MedicationRequest.supportingInformation` (lines 544–567) and
    `MedicationRequest.authoredOn` (currently line 568).
  - `source/medicationrequest/medicationrequest0340.xml` — reorder the
    `<isRecordOfRequest value="true">…</isRecordOfRequest>` element
    (currently line 37, after `<requester>`) so that it precedes
    `<authoredOn>` (currently line 32) and `<requester>` (currently line
    33), matching the new resource element order.
- **Change Scope:** **Minor** (single-resource reorder; no schema changes,
  no new elements, no terminology, no behavioural change).

### Breaking Changes

**No breaking changes identified.**

- `isRecordOfRequest` is new in this cycle (the resolution explicitly
  invokes that fact), so no implementer is yet relying on its previous
  ordinal position.
- Cardinality (0..1), type (`boolean`), `isSummary` (`true`), the (still
  commented-out) `isModifier` flag, mappings, short, definition, and
  comment all remain unchanged.
- JSON instances are unaffected (JSON is order-insensitive). XML instances
  must follow the new order, which is why the one example that uses the
  element is being updated alongside the StructureDefinition.

### Related Specifications

- `MedicationRequest.informationSource` — its definition explicitly
  references `.isRecordOfRequest` ("Normally when this element is
  populated, the .isRecordOfRequest element would be set to 'true'."); the
  cross-reference text is unaffected by the reorder.
- `MedicationRequest.supportingInformation` — adjacent to the moved
  element after this change; no text change required.
- No downstream resources reference `MedicationRequest.isRecordOfRequest`
  by ordinal position, so no IG / profile / extension repo work is
  triggered.

### Related Tickets

- **FHIR-55006:** *"Is the element isRecordOfRequest appropriate? Wouldn't
  that be a MedicationStatement?"* — Triaged / Unresolved (also Pharmacy,
  also reported by Ron G. Parker, also reported as ballot Comment 31's
  follow-up). This ticket questions whether `isRecordOfRequest` should
  exist on MedicationRequest at all (vs. being represented as a
  MedicationStatement). It does **not** conflict with FHIR-55005's
  purely-positional change in the short term — even if FHIR-55006 later
  results in the element being removed, FHIR-55005's reorder simply
  becomes moot. However, implementers should be aware that a future
  resolution to FHIR-55006 may delete the element entirely.

  No other cross-references were returned by `cross-referenced` for
  FHIR-55005 (`hits: []`).

### Terminology Impact

**No terminology impact.** The element is `boolean`, with no binding;
neither HL7/UTG nor any ValueSet/CodeSystem is touched.

## Implementation Plan

### Prerequisites

None. The change is self-contained inside `HL7/fhir`. There are no
dependent ticket resolutions, no terminology additions, and no new
extension definitions required.

### Step-by-Step Tasks

#### HL7/fhir

1. **Move `MedicationRequest.isRecordOfRequest` in the StructureDefinition differential.**
   - File: `source/medicationrequest/structuredefinition-MedicationRequest.xml`
   - Action: Cut the entire `<element id="MedicationRequest.isRecordOfRequest">…</element>` block (currently lines 628–645) and paste it immediately after the closing `</element>` of `MedicationRequest.supportingInformation` (currently line 567) and immediately before `<element id="MedicationRequest.authoredOn">` (currently line 568).
   - Details:
     - Stay strictly inside the `<differential>` block (opens at line 86, closes near line 1238). Do **not** touch any `<snapshot>` content — it is regenerated by the publisher.
     - Preserve the block verbatim, including the commented-out `<!-- <isModifier value="true"/> <isModifierReason …/> -->` lines, the `<isSummary value="true"/>` flag, and the rim `<mapping>` entry.
     - Resulting differential order around the change:
       `…informationSource → encounter → supportingInformation → isRecordOfRequest → authoredOn → requester → performerType…`
     - Verify with `Select-String '<element id="MedicationRequest\.' source/medicationrequest/structuredefinition-MedicationRequest.xml` that `isRecordOfRequest` now appears between `supportingInformation` and `authoredOn`.

2. **Reorder `<isRecordOfRequest>` in the matching example instance.**
   - File: `source/medicationrequest/medicationrequest0340.xml`
   - Action: Move the `<isRecordOfRequest value="true">…</isRecordOfRequest>` element (currently line 37, after `<requester>`) so that it appears immediately before `<authoredOn>` (currently line 32). The example does not currently include `<supportingInformation>`, so positioning before `<authoredOn>` is sufficient to match the new StructureDefinition order.
   - Details:
     - This is required because FHIR XML serialisation must follow the resource's element order; without this fix the example will fail XML validation against the regenerated MedicationRequest schema.
     - Do not change the element's value or any attributes; only change its position within the parent `<MedicationRequest>` element.

3. **Run a repo-wide check for any other affected XML examples.**
   - File: _(search, not edit)_
   - Action: Run `Select-String -Path source -Filter '*.xml' -Pattern '<isRecordOfRequest'` (or equivalent grep) to confirm that the only MedicationRequest XML instance that emits `<isRecordOfRequest>` is `medicationrequest0340.xml`. The pre-existing match in `source/medicationadministration/immunization-example-refused.xml` (line 21) is on an Immunization resource, not a MedicationRequest, and is unaffected by this ticket.
   - Details: If new examples are discovered that include `<isRecordOfRequest>` after `<authoredOn>` / `<requester>`, apply the same reorder as Task 2 to each.

4. **No prose / introduction page change required.**
   - File: `source/medicationrequest/medicationrequest-introduction.xml` (and `medicationrequest-notes.xml`)
   - Action: None. Verified that neither file mentions `isRecordOfRequest` ordering. The change is purely structural.

### Testing Considerations

- **Re-publish locally** to confirm the snapshot is regenerated with
  `isRecordOfRequest` now sitting between `supportingInformation` and
  `authoredOn`, and to surface any XML-order validation errors in
  examples.
- **XML validation** of `medicationrequest0340.xml` (and any other
  example flagged in Task 3) against the new schema, since FHIR XML is
  order-sensitive.
- **Diff the rendered MedicationRequest page** before/after to confirm the
  only visible change is the row position of `isRecordOfRequest` in the
  element table; short/definition/cardinality/type cells should be
  identical.
- **Edge case:** confirm that QA does not flag the moved element as
  "deleted + added" in a way that breaks cross-version comparisons; if
  the build has a stable element-order check, this reorder is its
  expected output.
- **Interoperability:** JSON instances are unaffected (order-insensitive).
  XML producers/consumers that hard-coded element offsets will need to
  re-generate from the new schema, but since the element is new this
  cycle there should be no installed base to worry about.

### Open Questions

- **Interaction with FHIR-55006.** FHIR-55006 is unresolved and questions
  whether `isRecordOfRequest` belongs on MedicationRequest at all. The
  Pharmacy WG may want to confirm that they want FHIR-55005 applied even
  though FHIR-55006 could later remove the element. The skill's
  recommendation is to apply FHIR-55005 as approved (the resolution is
  unconditional and the change is trivial to revert) and let FHIR-55006
  proceed independently.
- **`isModifier` status.** The `<isModifier value="true"/>` /
  `<isModifierReason …/>` lines are present but commented out in the
  current file (lines 638–639). This ticket does not ask for that to
  change, but reviewers should confirm the commented state is intentional
  given the modifier-style semantics described in the element definition.
  No action taken in this plan beyond preserving the comment verbatim
  during the move.
- **Briefing scope.** The persisted `HL7/fhir` briefing is scoped to
  FHIR-42980 (mapping-language). Its general facts (authoring roots,
  generated areas, gotchas) apply directly to this ticket, but its
  ticket-specific Recommended Change Recipes are not tailored to a
  StructureDefinition reorder. The plan above relies on the briefing's
  general FhirCore guidance plus direct verification against the clone
  (`db79dcd`).
