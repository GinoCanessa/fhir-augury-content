## Resolution Summary

**Resolution: Persuasive with Modification.**

Per the resolution description, the work group agreed to:

1. **Restore `MedicationStatement.basedOn`** as it existed in R4:
   `0..* Reference(MedicationRequest | CarePlan | ServiceRequest)`.
2. **Add comments to both `.basedOn` and `.derivedFrom`** that disambiguate
   the two reference elements:
   - `.basedOn` — *what order is being fulfilled with this statement (or
     what order allowed the medication to be given/taken).*
   - `.derivedFrom` — *to be used to point at sources of information that
     allowed to infer the MedicationStatement, or are used as "evidence"
     of that statement.*
3. **The editor may adjust the wording if needed.**

No other elements are touched by this resolution.

## Feature Proposal

### Problem Statement

`MedicationStatement.basedOn` was present in FHIR R4 with the typing
`0..* Reference(MedicationRequest | CarePlan | ServiceRequest)`. It was
removed in 2021 by HL7/fhir commit
[`7453f87`](https://github.com/HL7/fhir/commit/7453f87bfcb5eeba6020113412508cb95e5d4f0a)
(Pharmacy disposition batch covering FHIR-31017, 31016, 31633, 31365,
28598, 31508, F31573). The reporter notes that of those tickets, only
FHIR-31508 (about `partOf`) is the closest in scope, and there is no
on-record decision that justifies removing `basedOn` itself.

The current build (clone HEAD `db79dcd`) reflects that removal: the
StructureDefinition at
`source/medicationstatement/structuredefinition-MedicationStatement.xml`
goes directly from `MedicationStatement.partOf` (line 121) to
`MedicationStatement.status` (line 140), with no `basedOn` element. The
existing `MedicationStatement.derivedFrom` (line 385) is the only
inbound-reference element on the resource and is described loosely
enough that implementers have been using it to express both
"derived/evidence" and "fulfills this order" semantics, which the work
group considers a modelling error.

The gap therefore has two prongs:

- A useful workflow link (Statement → Request) that exists on the
  sibling resource `MedicationRequest.basedOn` (see
  `source/medicationrequest/structuredefinition-MedicationRequest.xml`,
  line 143) is missing on `MedicationStatement`, so consumers cannot
  cleanly answer "which order did this statement fulfill?" without
  overloading `derivedFrom`.
- The split of intent between `basedOn` ("fulfills") and `derivedFrom`
  ("evidence / inferred from") is not documented on the resource.

### Proposed Change

Make a single, scoped edit to the `MedicationStatement` StructureDefinition
plus targeted prose updates:

1. **Add a new element `MedicationStatement.basedOn`** to the
   differential, inserted between `MedicationStatement.partOf`
   (current line 121) and `MedicationStatement.status` (current line
   140), with the following definition (pattern lifted from the R4
   element and aligned with the sibling `MedicationRequest.basedOn`):

   - `id` / `path`: `MedicationStatement.basedOn`
   - `short`: `Fulfils plan, proposal or order`
   - `definition`: `A plan, proposal or order that is fulfilled in whole
     or in part by this MedicationStatement.`
   - `comment` *(per resolution)*: `basedOn is what order is being
     fulfilled with this statement (or what order allowed the medication
     to be given/taken). Use derivedFrom (not basedOn) to point at
     sources of information that allowed the MedicationStatement to be
     inferred, or that are used as "evidence" of the statement.`
   - `min` / `max`: `0` / `*`
   - `type`: `Reference` with `targetProfile` entries (in this order):
     - `http://hl7.org/fhir/StructureDefinition/MedicationRequest`
     - `http://hl7.org/fhir/StructureDefinition/CarePlan`
     - `http://hl7.org/fhir/StructureDefinition/ServiceRequest`
   - `isSummary`: `true` (matches the R4 element and the sibling
     `MedicationRequest.basedOn`).
   - `mapping`:
     - `workflow` → `Event.basedOn`
     - `rim` → `.outboundRelationship[typeCode=FLFS].target[classCode=SBADM or PROC, moodCode=RQO or PRP or PLAN]`

2. **Update `MedicationStatement.derivedFrom`** (current line 385) to
   add a `comment` (or extend the existing `comment` immediately after
   the `definition` on line 388) reflecting the resolution's intent
   text:
   - Append: `derivedFrom is to be used to point at sources of
     information that allowed the MedicationStatement to be inferred,
     or that are used as "evidence" of that statement (e.g., a Claim,
     Observation or QuestionnaireResponse). Use basedOn (not
     derivedFrom) when recording which order the statement is
     fulfilling.`
   - The existing `comment` on `derivedFrom` lists likely reference
     targets (MedicationRequest, MedicationDispense, Claim,
     Observation, QuestionnaireAnswers); consider trimming
     `MedicationRequest` from that example list, because the
     MedicationRequest case now belongs in `basedOn`. Editor's
     discretion per the resolution.

3. **Refresh the introduction / scope prose** at
   `source/medicationstatement/medicationstatement-introduction.xml`
   line 72, which currently reads:

   > To indicate the link between a MedicationStatement instance and
   > the Medication Request, Dispense, or Administration that was used
   > to derive the MedicationStatement, the reference should be placed
   > in the MedicationStatement.derivedFrom element.

   The MedicationRequest half of that sentence now belongs in
   `basedOn`. Rewrite as something like:

   > To indicate which order this MedicationStatement is fulfilling
   > (e.g., a MedicationRequest, ServiceRequest, or CarePlan), use
   > MedicationStatement.basedOn. To indicate other sources of
   > information that allowed the MedicationStatement to be derived or
   > inferred (e.g., a MedicationDispense, MedicationAdministration,
   > Claim, Observation, or QuestionnaireResponse), use
   > MedicationStatement.derivedFrom.

4. **Add a `_changelog.txt` entry** at
   `source/medicationstatement/_changelog.txt` summarising the change
   and citing FHIR-55888, in the same format as the two existing
   entries in that file.

No new search parameters, ValueSets, CodeSystems, extensions, or
operations are introduced; the sibling `MedicationRequest` does not ship
a `based-on` SearchParameter either (verified — no `basedOn` /
`based-on` token in
`source/medicationrequest/bundle-MedicationRequest-search-params.xml`),
so adding a `MedicationStatement-based-on` search parameter is **out of
scope** unless Pharmacy explicitly asks for one.

### Design Rationale

- **Restoration vs. invention.** The resolution language explicitly
  says "restore … as it was in R4", so the cardinality (`0..*`) and
  target-profile set (`MedicationRequest | CarePlan | ServiceRequest`)
  are taken from R4, not re-derived. This minimises churn for R4
  implementers who have already coded against the element and have
  been waiting for it to come back.
- **Symmetry with `MedicationRequest.basedOn`.** The sibling resource
  carries `basedOn` as `0..*`, summary-true, with the workflow mapping
  `Request.basedOn`. Mirroring that on the `Event` side
  (`Event.basedOn`) keeps the workflow pattern consistent and matches
  how every other event-pattern resource (e.g., MedicationDispense,
  Procedure) wires the element. `RequestOrchestration` was added as a
  target on `MedicationRequest.basedOn` in R5; the resolution does
  **not** ask us to add it to MedicationStatement, so we stay strictly
  within the R4 target set the resolution names.
- **Disambiguation of `basedOn` vs. `derivedFrom` is a documentation
  fix, not a behavioural one.** Both elements remain `0..*`, both can
  technically reference a `MedicationRequest` instance — but the
  comments now state which one to choose, addressing the reporter's
  underlying concern that the two semantics had collapsed onto a
  single element.
- **Editor latitude.** The resolution explicitly grants editorial
  discretion on the wording. The proposed text above sticks as
  closely as possible to the resolution's own phrasing so reviewers
  can verify intent without translation.

### Breaking Changes

Adding a new optional element (`0..*`) is, in isolation, a
non-breaking schema change for consumers — they can ignore it. There
are nonetheless two implementer-visible impacts worth calling out
honestly:

- **Producer behaviour reinterpretation.** Implementers who previously
  packed "this statement fulfils order X" into `derivedFrom` are now
  invited (by the comment text) to move that data into `basedOn`. That
  is not enforced by schema, but reviewers and downstream profiles
  will start to flag the legacy usage as a modelling error. Impact is
  **soft / behavioural**, not wire-breaking.
- **Profiles that constrained `derivedFrom`** (if any exist in
  downstream IGs) may need to re-balance constraints between the two
  elements. No core-profile change is required by this ticket.

No other backward-incompatible changes (no required-element
additions, no type changes, no removed elements, no constraint
tightening, no binding-strength promotions, no search-parameter
changes).
