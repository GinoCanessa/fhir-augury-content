## Resolution Description

> This has already been done with the ClinicalDocument IG project.

## Original Request

The ticket requested:

1. Develop a proper profile on Composition in FHIR R4.0 and FHIR R5.0 for use by **all** CDA IGs — not just C-CDA R2.1.
2. Provide mapping at a broader level to align **all** CDA documents with the FHIR Composition resource.
3. Include CDA-to-FHIR mapping in the profile itself.
4. Create the profile **separate from** the C-CDA on FHIR IG.
5. Embed the profile with the Composition resource in the **base FHIR standard**.

## Resolution Interpretation

The resolution "Persuasive with Modification" acknowledges the validity of the request but modifies the approach: rather than embedding the profile directly in the base FHIR standard, the work was accomplished through a **separate Implementation Guide** — the [FHIR Clinical Documents IG](https://build.fhir.org/ig/HL7/fhir-clinical-document/index.html) (`hl7.fhir.uv.fhir-clinical-document`), published at STU 1 (v1.0.1) as a **universal realm** guide.

This IG satisfies the core requirements of FHIR-41149:
- ✅ A proper profile on Composition for CDA header alignment
- ✅ Universal scope — usable by all CDA IGs, not just C-CDA
- ✅ CDA-to-FHIR mapping included in the profiles
- ✅ Separate from the C-CDA on FHIR IG
- ⚠️ **Modified**: Not embedded in the base FHIR standard, but published as a standalone IG

## Feature Proposal

Since the resolution states this work has "already been done," the primary implementation tasks are:

1. **Verify completeness** of the FHIR Clinical Documents IG against the ticket requirements.
2. **Update base FHIR spec references** to point to the new IG where the old ClinicalDocument profile is referenced.
3. **Deprecate legacy artifacts** in the base FHIR spec that are superseded by the new IG.
4. **Ensure cross-IG alignment** so downstream IGs (C-CDA on FHIR, IPS, etc.) can derive from the new profiles.

## Impact Analysis

### What Already Exists (FHIR Clinical Documents IG)

**Repository**: [HL7/fhir-clinical-document](https://github.com/HL7/fhir-clinical-document)
**FHIR Version**: R4 (4.0.1)
**Status**: STU 1 — active (v1.0.1)

#### Profiles Defined

| Profile | Parent | ID | Purpose |
|---------|--------|----|---------|
| `ClinicalDocumentComposition` | `Composition` | `clinical-document-composition` | Universal Composition profile for clinical documents with CDA mappings |
| `ClinicalDocumentBundle` | `Bundle` | `clinical-document-bundle` | Document Bundle profile with document-level constraints |

#### Extensions Defined (in `input/fsh/extensions.fsh`)

| Extension | Purpose | CDA Mapping |
|-----------|---------|-------------|
| `DataEntererExtension` | Data entry person who transferred content into the document | `assignedEntity.dataEnterer` |
| `InformantExtension` | Information source for document content | `informantChoice.informant` |
| `InformationRecipientExtension` | Intended recipient (PRCP, TRC, IRCP) | `intendedRecipient.informationRecipient` |
| `ParticipantExtension` | General participants (not data-enterer/informant/recipient) | `associatedEntity.participant` |
| `ConsentExtension` | Patient consent reference | `authorization.Consent` |
| `ChangeMade` | Annotation of document changes (experimental) | N/A |

#### Key Composition Profile Constraints

- Uses R5 backport extensions for `Composition.version` and `Composition.status`
- `category` requires `107903-7 "Clinical note"` (1..*)
- `subject` restricted to `Patient | Group` (1..1)
- `attester` sliced into `legal_attester` and `professional_attester`
- `relatesTo` sliced for `replaced_document` and `appended_document`
- `section` requires `title` (1..1) and is must-support
- CDA mapping identity declared throughout
- Supports `basedOn` via R5 backport extension (for CDA `inFulfillmentOf`)

#### CDA Mapping Coverage

Per the IG's mapping page (`input/pagecontent/mapping.md`), scope includes:
1. CDA header attributes
2. CDA header participants (excluding entity profiles)
3. CDA header relationships (excluding related Act profiles)
4. CDA body choice
5. CDA section attributes
6. CDA section participants (excluding RIM entity profiles)
7. CDA section relationships (excluding RIM Act profiles)

## Detailed Implementation Plan

### Phase 1: Update Base FHIR Spec References (HL7/fhir)

Since the FHIR Clinical Documents IG now provides the canonical ClinicalDocument profile, the base FHIR spec should be updated to reference it appropriately.

#### Task 1.1: Update Composition Introduction Text

**File**: `source/composition/composition-introduction.xml`
**Line ~23**: Currently reads:
```xml
The <a href="composition-core.html">Clinical Document profile</a> constrains Composition to specify a clinical document
(matching <a href="http://www.hl7.org/implement/standards/product_brief.cfm?product_id=7">CDA</a>).
```

**Change to**: Update the reference to acknowledge both the legacy in-spec profile and the newer, more comprehensive FHIR Clinical Documents IG as the recommended starting point for new clinical document work.

#### Task 1.2: Update ClinicalDocument Profile Introduction

**File**: `source/composition/composition-clinicaldocument-introduction.xml`

**Action**: Add a prominent note at the top directing implementers to the [FHIR Clinical Documents IG](https://hl7.org/fhir/uv/fhir-clinical-document/) as the canonical and more comprehensive profile for clinical document representation in FHIR.

#### Task 1.3: Deprecate Legacy CDA Profile

**File**: `source/profiles/cda.profile.xml`

**Action**: This is a very old (2014) legacy profile. Consider:
- Setting status to `retired` or adding deprecation notice
- Adding a note directing users to the FHIR Clinical Documents IG
- This aligns with the IG's own reconciliation plan which proposes "deprecating the FHIR core ClinicalDocument profile in FHIR R6"

#### Task 2.2: R5 Gap Assessment

The ticket originally requested profiles for both R4 and R5. The current IG is R4-only but uses R5 backport extensions. An R5 version of the IG is in the reconciliation plan:

> "Plan for artifacts to derive from this IG by FHIR R6. Preparation includes: (1) backport key FHIR R6 capabilities now; (2) propose deprecating the FHIR core ClinicalDocument profile in FHIR R6; (3) work with artifact developers; (4) Propose revisions to FHIR R6 Composition that subsume Composition extensions."

**R5/R6 considerations** (informational — out of scope for this ticket):
- R5 adds `Composition.version` natively (replacing the backported extension)
- R5 adds `Composition.note` (may overlap with `ChangeMade` extension)
- R6 may add `Composition.participant` as a backbone element (replacing participant extensions)
- R6 may add `Composition.basedOn` (replacing the basedOn extension)
