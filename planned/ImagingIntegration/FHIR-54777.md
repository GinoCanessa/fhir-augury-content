## Resolution Summary

The committee accepted the request as **Persuasive** and approved an HTML
replacement for the *Comprehensive Imaging Scheduled Workflow* bullet that
describes the ImagingStudy created by the image archive. Each previously
italicised attribute name (`subject`, `basedOn`, `identifier`, `endpoint`,
`modality`) is to be **fully qualified** with the `ImagingStudy.` prefix,
wrapped in `<code>` and pointed at the correct resource link, e.g.:

```html
<li><code>ImagingStudy.subject</code> is current <a href="patient.html">Patient</a></li>
<li><code>ImagingStudy.basedOn</code> includes current <a href="servicerequest.html">ServiceRequest</a></li>
<li><code>ImagingStudy.identifier</code> is Study Instance UID</li>
<li><code>ImagingStudy.endpoint</code> points to image archive WADO-RS service</li>
<li><code>ImagingStudy.modality</code> is US</li>
```

The same `ImagingStudy.` qualification applies to later steps in the same
use case that mention `modality` and that introduce KOS series content
(e.g. `ImagingStudy.modality is now US,KOS`, KOS series details added to
`ImagingStudy.series`).

The use-case prose is also restructured so that workflow steps that do
not relate directly to FHIR (radiology reading workflow, image archive
notification mechanisms) are clearly demarcated. This portion of the
restructure is shared with FHIR-54794, which was applied in the same
commit.

## Feature Proposal

### Problem Statement

In the *Comprehensive Imaging Scheduled Workflow* narrative on the
ImagingStudy page (section 10.4.5.5.1), the steps that describe the
ImagingStudy created by the image archive list each attribute as a bare
italicised word — `subject`, `basedOn`, `identifier`, `endpoint`,
`modality`. Several of those names also exist on `ImagingStudy.series`
(e.g. `endpoint`, `modality`, `identifier`). Without a path qualifier,
readers cannot tell whether the use case is asserting that a value lives
on the top-level resource or on a series. The ticket reporter
(J. Medema) flagged `endpoint` in particular as ambiguous.

### Proposed Change

Edit `source/imagingstudy/imagingstudy-notes.xml` in `HL7/fhir` to
qualify every attribute reference inside the *Comprehensive Imaging
Scheduled Workflow* `<ul>` with the `ImagingStudy.` prefix and to wrap
the qualified path in `<code>` (matching the rest of the page). The
specific changes, applied to the bullet `<li>Image archive creates an
<a href="imagingstudy.html">ImagingStudy</a>` and the later KOS update
bullet, are:

| Before | After |
|---|---|
| `<code>subject</code> is current Patient` | `<code>ImagingStudy.subject</code> is current Patient` |
| `<code>basedOn</code> includes current ServiceRequest` | `<code>ImagingStudy.basedOn</code> includes current ServiceRequest` |
| `<code>identifier</code> is Study Instance UID` | `<code>ImagingStudy.identifier</code> is Study Instance UID` |
| `<code>endpoint</code> points to image archive WADO-RS service` | `<code>ImagingStudy.endpoint</code> points to image archive WADO-RS service` |
| `<code>modality</code> is US` | `<code>ImagingStudy.modality</code> is US` |
| `<code>modality</code> is now US,KOS` | `<code>ImagingStudy.modality</code> is now US,KOS` |
| `KOS series added` | `KOS series details added to <code>ImagingStudy.series</code>` |

In addition (rolled together with FHIR-54794 in the same edit), reshape
the surrounding bullets so that the EMR-notification step and the
radiology-reading-workstation step are nested under their own
`<li>` headings — explicitly tagging the radiology reading workflow as
**non-FHIR** — rather than appearing as a flat sequence of peer bullets.

### Design Rationale

- **Disambiguation.** `ImagingStudy.endpoint`, `ImagingStudy.modality`,
  and `ImagingStudy.identifier` all have direct counterparts at the
  series level (`ImagingStudy.series.endpoint`, `series.modality`,
  `series.identifier`). The same notes file already establishes the
  override semantics between the two levels (lines 61–66 of
  `imagingstudy-notes.xml`), which makes any unqualified reference in
  the use-case prose immediately ambiguous.
- **Consistency with the rest of the page.** Other prose on
  `imagingstudy-notes.xml` (e.g. the Endpoint discussion and the
  scheduled workflow inputs at line ~311) already uses the dotted
  `ImagingStudy.endpoint` form wrapped in `<code>`. Qualifying the use
  case brings it in line with the surrounding style.
- **Minimum-impact narrative fix.** No StructureDefinition, search
  parameter, binding, or invariant is touched — only the human-readable
  use-case description. The committee's *Persuasive* disposition with a
  verbatim XML replacement explicitly approves this narrow form.
- **Pairing with FHIR-54794.** The committee opted to perform the
  flat-list-to-nested restructuring at the same time, since both
  tickets edit the same `<ul>` and would otherwise step on each other.

## Implementation Plan

### Prerequisites

None. The edit is self-contained. (FHIR-54794 should be applied in the
same commit / PR to keep the bullet restructure consistent — see
*Open Questions*.)

### Step-by-Step Tasks

#### HL7/fhir

1. **Qualify ImagingStudy attributes in the scheduled-workflow use case**
   - File: `source/imagingstudy/imagingstudy-notes.xml`
   - Action: Inside the `<h3>Use Cases</h3>` →
     `<a name="scheduled"></a><h4>Comprehensive Imaging Scheduled
     Workflow</h4>` block (around line 247 onward), in the `<li>`
     "Image archive creates an `<a href="imagingstudy.html">ImagingStudy</a>`"
     sub-bulleted `<ul>`, replace each unqualified `<code>{name}</code>`
     with `<code>ImagingStudy.{name}</code>` for `subject`, `basedOn`,
     `identifier`, `endpoint`, and `modality`.
   - Details: Use the resolution's verbatim wording (see the table in
     *Proposed Change*). Keep all `<a href="...html">` links intact —
     `patient.html` for `subject`, `servicerequest.html` for `basedOn`.

2. **Qualify the KOS-update bullets in the same use case**
   - File: `source/imagingstudy/imagingstudy-notes.xml`
   - Action: In the later `<li>Image archive updates imaging study to
     include KOS` bullet, change `<code>modality</code> is now US,KOS`
     to `<code>ImagingStudy.modality</code> is now US,KOS`, and change
     `KOS series added` to
     `KOS series details added to <code>ImagingStudy.series</code>`.
   - Details: This carries the same disambiguation rule into the
     subsequent step that mutates the study.

3. **Re-nest the workflow bullets so non-FHIR steps are grouped (joint
   with FHIR-54794)**
   - File: `source/imagingstudy/imagingstudy-notes.xml`
   - Action: Convert the previously flat sequence
     ("Image archive notifies EMR…", "Radiology reading workstation
     retrieves study images…", "Radiologist reads study…", "Radiology
     reading workstation sends DICOM Key Object Selection (KOS)…")
     into:
     - `<li>Image archive notifies EMR of imaging study<ul><li>See
       <a href="imagingstudy.html#workflow">ImagingStudy Create and
       Update Workflow</a> for possible notification mechanisms</li></ul></li>`
     - `<li>Radiology reading workflow (non-FHIR)<ul>` containing the
       three radiology-reading sub-bullets `</ul></li>`.
   - Details: This is the structural half of the change shared with
     FHIR-54794 and is required so the qualified bullets above land in
     a clean parent structure.

4. **Verify no other use-case prose on the page reintroduces unqualified
   attribute names**
   - File: `source/imagingstudy/imagingstudy-notes.xml`
   - Action: Search the file for `<code>endpoint</code>`,
     `<code>modality</code>`, `<code>identifier</code>`,
     `<code>subject</code>`, and `<code>basedOn</code>` (without the
     `ImagingStudy.` prefix) and qualify any remaining occurrences in
     prose contexts where the attribute could plausibly mean either the
     study or series level.
   - Details: The endpoint discussion at lines 61–66 already uses the
     qualified form and should not be touched. Confine edits to
     ambiguous bare references inside use cases / examples.
