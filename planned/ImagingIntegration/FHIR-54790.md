## Feature Proposal

### Problem Statement

Both `ImagingStudy` and `ImagingSelection` document how a referenced
`Endpoint` of `connectionType = dicom-wado-rs` is used to retrieve DICOM
content. The two sections were originally intended to mirror each other (with
appropriate substitutions of the resource name), but they have drifted:

- `source/imagingstudy/imagingstudy-notes.xml` (lines 78–139) describes the
  WADO-RS service, gives a base GET example, then continues with three
  additional paragraphs covering:
  1. The "rendered" sub-resource query parameters (dimensions, quality,
     region of interest, brightness/contrast, demographic burn-in, frame
     selection).
  2. A worked example URL that adds `/rendered?viewport=...` to retrieve a
     400×400 region.
  3. A thumbnail retrieval example (`/thumbnail`) when the WADO-RS service
     supports DICOMweb thumbnails.
  4. A pointer to "DICOM PS 3.18 Section 10.4" for further details.

- `source/imagingselection/imagingselection-notes.xml` (lines 207–239) ends
  immediately after the equivalent base GET example and is missing the four
  items above.

This inconsistency is gratuitous — there is no resource-specific reason for
ImagingSelection to omit the rendered/thumbnail/PS3.18 material — and it
makes the spec harder to use because implementers have to flip between the
two pages to find the rendering guidance. The reporter explicitly asked for
either alignment or a documented reason for the omission, and the WG chose
alignment.

### Proposed Change

Edit `source/imagingselection/imagingselection-notes.xml` so that the
WADO-RS subsection (`<a name="wado-rs"></a>` … up to but not including
`<a name="wado-uri"></a>`) carries the same trailing prose as the
ImagingStudy WADO-RS subsection, with the resource name and example
ImagingSelection-specific element references kept consistent with the
existing ImagingSelection prose.

Specifically, **after** the existing closing `</pre>` on line 239 and
**before** the `<a name="wado-uri"></a>` anchor on line 240, insert the
following four blocks (adapted from `imagingstudy-notes.xml` lines 110–139):

1. `<p>` — `Query parameters on the "rendered" sub-resource can control
   other aspects of the rendering including: the rendered dimensions, the
   quality (compression ratio), the region of interest to render, the
   brightness/contrast (window center/width) adjustments, and whether to
   "burn" patient or study demographics into the rendered result. Specific
   frames of a multiframe instance can be retrieved using the frames
   sub-resource.`
2. `<p>` — `The example above can be extended with parameters that request
   an image of a region extending from the top-left corner of the original
   image to 1000 pixels across and 3000 pixels right, to be rendered to a
   size of 400 columns by 400 rows:` followed by a `<pre>` block containing
   the worked GET URL with `/rendered?viewport=400,400,0,0,1000,3000` and
   `Accept: image/jpeg`.
3. `<p>` — `If the specified WADO-RS service supports the DICOMweb thumbnail
   resource, a representative image of the study can be requested, for
   example, to display alongside the study. The URL would look as follows:`
   followed by a `<pre>` block containing the worked thumbnail GET URL.
4. `<p>` — `For further details on DICOM WADO-RS capabilities including
   additional rendering parameters, see <a
   href="https://dicom.nema.org/medical/dicom/current/output/chtml/part18/sect_10.4.html">DICOM
   PS 3.18 Section 10.4</a>.`

The example study/series/instance UIDs and the WADO-RS base URL must reuse
the values already established earlier in the ImagingSelection WADO-RS
subsection (lines 227–230) — i.e. the same `https://pacs.hospital.org/wado-rs`
base, study UID `1.2.250.1.59.40211.12345678.678910`, series UID
`1.2.250.1.59.40211.789001276.14556172.67789`, instance UID
`1.2.250.1.59.40211.2678810.87991027.899772.2`. This keeps the worked
examples consistent with the immediately preceding GET example and avoids
introducing a third parallel set of UIDs.

Verbatim drop-in (the prose / examples are identical to ImagingStudy because
the ImagingSelection example block already uses the same UIDs and base URL):

```html
<p>
Query parameters on the "rendered" sub-resource can control other aspects of the rendering including: the rendered dimensions, the
quality (compression ratio), the region of interest to render, the brightness/contrast (window center/width)
adjustments, and whether to "burn" patient or study demographics into the rendered result. Specific frames
of a multiframe instance can be retrieved using the frames sub-resource.
</p>
<p>
The example above can be
extended with parameters that request an image of a region extending from
the top-left corner of the original image to 1000 pixels across and 3000 pixels right, to be rendered to a size of 400 columns by 400 rows:
</p>
<pre>

GET https://pacs.hospital.org/wado-rs/studies/1.2.250.1.59.40211.12345678.678910/series/1.2.250.1.59.40211.789001276.14556172.67789/instances/1.2.250.1.59.40211.2678810.87991027.899772.2<b>/rendered?viewport=400,400,0,0,1000,3000</b> HTTP/1.1
Accept: image/jpeg

</pre>
<p>
If the specified WADO-RS service supports the DICOMweb thumbnail resource, a representative image of the study can be requested, for example, to display alongside the study. The URL would look as follows:
</p>
<pre>

GET https://pacs.hospital.org/wado-rs/studies/1.2.250.1.59.40211.12345678.678910/thumbnail HTTP/1.1
Accept: image/jpeg

</pre>
<p>
For further details on DICOM WADO-RS capabilities including additional rendering parameters,
see <a href="https://dicom.nema.org/medical/dicom/current/output/chtml/part18/sect_10.4.html">DICOM PS 3.18 Section 10.4</a>.
</p>
```

No StructureDefinition, ValueSet, CodeSystem, search parameter, example
resource, or invariant changes are required. The change is confined to a
single HTML notes page.

### Design Rationale

- **Consistency over divergence.** The two resources both describe the same
  external standard (DICOM WADO-RS). Documenting the rendered/thumbnail
  parameters on one resource page but not the other has no editorial
  justification — both resources can carry an `Endpoint.connectionType` of
  `dicom-wado-rs`, and the same query parameters apply equally. The WG
  reached this conclusion in the 6-0-0 vote on 3 Mar 2026.
- **Cite the same DICOM normative reference (PS 3.18 §10.4).** Pointing
  implementers at the DICOM source of truth is more durable than re-stating
  rendering parameter semantics inside FHIR, and matches what
  ImagingStudy already does.
- **Reuse the existing ImagingSelection example UIDs.** The earlier
  paragraphs in `imagingselection-notes.xml` already construct a worked
  example with a specific base URL and UIDs; the inserted `/rendered` and
  `/thumbnail` examples build on the same example so a reader sees one
  coherent walkthrough rather than two unrelated ones.
- **Alternative considered: trim ImagingStudy's WADO-RS section instead.**
  Rejected — the rendered-parameters discussion is genuinely useful guidance
  for both resources, and removing it from ImagingStudy would lose
  information the WG already considers in scope.
- **Alternative considered: factor the WADO-RS prose into a shared include.**
  Rejected as out of scope — the bespoke HL7 publisher uses per-resource
  `*-notes.xml` files and there is no current pattern for sharing prose
  between resource notes pages. Introducing one would be a larger editorial
  change than this ticket warrants. The PR linked from the ticket
  (HL7/fhir#4043) takes the same in-place duplication approach.

   `<code>Endpoint.connectionType</code>` (line 210). The WG resolution
   wording ("Update imagingselection.html#wado-rs to be consistent with
   ImagingStudy as requested") is ambiguous on whether this also gets
   harmonised. **Default recommendation:** make the link styling consistent
   too (small, low-risk editorial improvement), but flag it explicitly in
   the PR description so the WG can confirm or revert.

   intentionally calls out series-level retrieval ("The WADO-RS service
   also allows retrieval of study or series level information." — line
   218–219) which ImagingStudy does not state. This ImagingSelection-specific
   sentence is preserved by the proposed edit; confirm with the editor that
   the WG did not intend to drop it.
