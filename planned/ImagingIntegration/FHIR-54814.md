## Resolution Summary

Resolution: **Persuasive**. The ImagingStudy and ImagingSelection "Boundaries and Relationships" sections must explicitly state that `ImagingSelection.derivedFrom` references an `ImagingStudy` when the images were selected from it. The resolution description gives the approved prose nearly verbatim:

> The ImagingStudy resource is used to store details of an entire DICOM Study and its relationships to other resources, such as Patient, ServiceRequest, and Encounter.
>
> The ImagingSelection resource is used to reference a subset of a study. Instances of this resource are created for specific clinical purposes and are unlikely to change once created. **ImagingSelection.derivedFrom references an ImagingStudy when the images were selected from it.**

The comment trail confirms this was applied in HL7/fhir commit `c429239e` ("Clarified relationship between ImagingStudy and ImagingSelection"), which also adds an entry to the per-resource "what changed" bullet lists.

## Feature Proposal

### Problem Statement

Both the ImagingSelection and ImagingStudy specification pages describe a boundaries/relationships block, but neither page states explicitly that the link between the two resources is carried on `ImagingSelection.derivedFrom`. The reporter observed that readers have to infer the linkage from the element definition in the StructureDefinition. The page also mentions DocumentReference as another derivedFrom target in the same paragraph context, further obscuring which resource is the primary imaging linkage. The resolution chooses the narrower, clearer wording focused on the ImagingStudy linkage.

### Proposed Change

Apply documentation-only (narrative) edits to the two resource introduction pages in the HL7/fhir bespoke publisher source tree:

1. In `source/imagingstudy/imagingstudy-introduction.xml`, in the "Boundaries and Relationships" block:
   - Fix the sentence enumerating ImagingStudy relationships so the comma-splice before "ServiceRequest and Encounter" becomes a proper list: `…such as Patient, ServiceRequest, and Encounter.` (also pluralises "relationship" → "relationships").
   - Extend the ImagingSelection paragraph with the sentence: `ImagingSelection.derivedFrom references an ImagingStudy when the images were selected from it.` The link text uses `<a href="imagingstudy.html">ImagingSelection.derivedFrom</a>` and `<a href="imagingstudy.html">ImagingStudy</a>` (matching the existing cross-link style in the file).
2. In `source/imagingselection/imagingselection-introduction.xml`, apply the same two prose updates in the equivalent "Boundaries and Relationships" paragraph (the page carries a parallel copy).
3. In both introduction pages, add a bullet to the top-level "What's new" / changes list: `<li>Clarified relationship between ImagingStudy and ImagingSelection</li>`.

No StructureDefinition, search parameter, value set, example, or schema changes are required. The `<differential>` content of `source/imagingstudy/structuredefinition-ImagingStudy.xml` is unaffected by this ticket (the same commit touches it only for the unrelated FHIR-55483 `numberOfFrames` change).

### Design Rationale

The resolution explicitly prefers stating the linkage in each resource's own Boundaries and Relationships section rather than tacking it onto a combined paragraph that also covers DocumentReference — this keeps the statement close to where readers will be looking when trying to understand how a given resource connects to the rest of the imaging story. The wording mirrors the element definition on `ImagingSelection.derivedFrom` ("The imaging study from which the imaging selection is derived"), so authors do not need to coin new semantics. Adding a bullet to the per-release change list is the standard FhirCore pattern for surfacing narrative clarifications, and matches the bullets added for sibling tickets in the same commit.
