## Resolution Summary

**Resolution:** Persuasive.

The resolution requires adding a new note to the **ImagingSelection** Notes
page describing how to encode a DICOM UID when it is used as an
`ImagingSelection.identifier`. The exact wording approved in the resolution
is:

> ### 10.4.5.X DICOM UIDs as Identifiers
>
> In some cases, the `ImagingSelection.identifier` will be a DICOM UID, such
> as an Observation UID (see
> [DICOM PS 3.3 Table C.17-6](https://dicom.nema.org/medical/dicom/current/output/chtml/part03/sect_C.17.3.4.html#table_C.17-6)).
>
> When using a DICOM UID as the identifier, use the Identifier system of
> `"urn:dicom:uid"`, and prefix the UID value with `"urn:oid:"`.

The "10.4.5.X" subsection number is illustrative — the bespoke FHIR
publisher numbers `<h3>` blocks in the page automatically based on insertion
order, so the implementation only needs to add the heading "DICOM UIDs as
Identifiers" plus the prose at the appropriate location in
`source/imagingselection/imagingselection-notes.xml`.

The Jira comment by the reporter (`jwhitby`) points to commit

## Feature Proposal

### Problem Statement

`ImagingSelection.identifier` is an `Identifier` element that, in DICOM
workflows, is frequently populated with a DICOM UID (for example, an
Observation UID per DICOM PS 3.3 Table C.17-6). DICOM UIDs are bare OIDs
(e.g., `1.2.840.10008.…`) and the FHIR Identifier datatype does not, on its
own, tell an implementer how to express such a UID interoperably.

The companion ImagingStudy page already documents the standard FHIR
encoding for DICOM UIDs (`Identifier.system = "urn:dicom:uid"`,
`Identifier.value = "urn:oid:<UID>"`) at
<https://build.fhir.org/imagingstudy.html#dicom-uids>, but the
ImagingSelection Notes page has no equivalent guidance. Implementers
reading the ImagingSelection page in isolation are left to guess at the
encoding, leading to inconsistent identifier values across servers and
clients.

### Proposed Change

Add a new note to the ImagingSelection Notes page,
`source/imagingselection/imagingselection-notes.xml`, defining how to
encode a DICOM UID when it appears as `ImagingSelection.identifier`.

Concretely, append the following block immediately before the closing
`</div>` at the end of the file (i.e., as the final `<h3>` block in the
Notes section, after the existing laterality `<ul>` that ends the file):

```xml
  <a name="dicom-uids"></a>
  <h3>DICOM UIDs as Identifiers</h3>
  <p>
    In some cases, the <code>ImagingSelection.identifier</code> will be a DICOM UID, such as an Observation UID (see <a href="https://dicom.nema.org/medical/dicom/current/output/chtml/part03/sect_C.17.3.4.html#table_C.17-6">DICOM PS 3.3 Table C.17-6</a>).

    When using a DICOM UID as the identifier, use the Identifier system of <code>"urn:dicom:uid"</code>, and prefix the UID value with <code>"urn:oid:"</code>.
  </p>
```

Key points:

- **Anchor:** `dicom-uids` (matches the pattern used by ImagingStudy at
  `imagingstudy.html#dicom-uids` so the two pages share a consistent
  fragment name).
- **Heading:** `<h3>DICOM UIDs as Identifiers</h3>` — `<h3>` matches the
  surrounding subsection level inside the Notes div; the publisher will
  assign the actual numeric prefix (the resolution wrote it as
  "10.4.5.X").
- **External link:** to DICOM PS 3.3 Table C.17-6, kept as a plain
  `https://dicom.nema.org/...` URL (no `target="_blank"`/`rel` attributes,
  matching the surrounding style in this file).
- **Encoding rules:** Identifier `system = "urn:dicom:uid"` and `value`
  prefixed with `"urn:oid:"` — identical to ImagingStudy guidance, which is
  the entire reason the note exists.

No StructureDefinition, ValueSet, CodeSystem, search parameter, or example
instance changes are required.

### Design Rationale

- **Consistency with ImagingStudy.** ImagingStudy already documents this
  exact encoding pattern (`urn:dicom:uid` + `urn:oid:` prefix). Mirroring
  the wording — and the fragment anchor name `dicom-uids` — on
  ImagingSelection avoids divergent guidance for two closely related
  resources that are typically used together.
- **Minimal-surface documentation change.** The resolution does not
  request any structural change to the resource (no new element, no
  modified cardinality, no new binding). The guidance is purely advisory,
  so a Notes entry — not an `ElementDefinition.comment` on `identifier` —
  is the right home; readers consulting the resource page reach the Notes
  via the standard tabbed layout.
- **Place at the bottom of the Notes div.** The existing Notes file is a
  flat sequence of `<h3>` topics inside a single `<div>`; adding the new
  heading at the end (after the laterality `<ul>`) keeps the diff small
  and avoids renumbering disruption to existing inbound links such as
  `#endpoints`, `#wado-rs`, `#wado-uri`.
- **Alternative considered:** add the guidance to the ImagingSelection
  introduction (`imagingselection-introduction.xml`) instead. Rejected
  because identifier-encoding mechanics are reference material, not scope
  framing, and the parallel content on ImagingStudy lives in its Notes
  section.
