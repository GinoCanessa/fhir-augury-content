## Resolution Summary

The resolution is **Persuasive** with `change required`. A new error-severity invariant `ist-8` is to be added to the `ImagingStudy` resource:

- **Key:** `ist-8`
- **Severity:** `error`
- **Human:** "If started and series.started are both present, the started value SHALL be less than or equal to all of the series.started values"
- **Expression:** `started.empty() or series.where(started.empty() or started >= %resource.started).count() = series.count()`

The resolution explicitly notes that the invariant **expression must be updated to match the human value** (the human text was tightened from the original ticket wording that referenced a single `series.started`). The invariant is to be added at the root `ImagingStudy` element so it is in scope for both `ImagingStudy.started` and `ImagingStudy.series.started`. The resolution comment links commit [c429239](https://github.com/HL7/fhir/commit/c429239eac28753c38f5211945b94996e2c86b05) as the applied implementation reference.

## Feature Proposal

### Problem Statement

`ImagingStudy.started` records when the study was started; `ImagingStudy.series.started` records when each series within that study was started. Today nothing in the StructureDefinition prevents a server or sender from publishing an `ImagingStudy` whose `started` value is *after* one or more of its `series.started` values, even though that is logically impossible — a series cannot have begun before the study that contains it.

The existing `ist-3`/`ist-4`/`ist-5`/`ist-6` constraints already enforce numeric sanity between the rolled-up `numberOf*` counters and the series content. There is currently no parallel temporal sanity check between `started` and `series.started`. This ticket fills that gap with an additional invariant in the same family.

### Proposed Change

Add a single error-level constraint to the root `ImagingStudy` element in `source/imagingstudy/structuredefinition-ImagingStudy.xml` and wire it as a `condition` on the two elements it constrains.

1. **New constraint on `ImagingStudy` root element** (alongside `ist-1`…`ist-7`):

   ```xml
   <constraint>
     <key value="ist-8"/>
     <severity value="error"/>
     <human value="If started and series.started are both present, the started value SHALL be less than or equal to all of the series.started values"/>
     <expression value="started.empty() or series.where(started.empty() or started >= %resource.started).count() = series.count()"/>
   </constraint>
   ```

   Insert immediately after the `ist-7` constraint block (currently ending at line 132 of `structuredefinition-ImagingStudy.xml`) and before the `<mapping>` blocks for the root element (line 133+). XML-escape `>=` as `&gt;=` per the existing file convention (cf. `ist-2`'s `&lt;`).

2. **Add `<condition value="ist-8"/>` to `ImagingStudy.started`** (element starting at line 317). Place it after the existing element body in the same position the other elements use for `<condition>` (e.g., as on `ImagingStudy.identifier` at lines 162–163).

3. **Add `<condition value="ist-8"/>` to `ImagingStudy.series.started`** (element starting at line 830).

4. **Add invariant tests** under `source/imagingstudy/invariant-tests/` mirroring the existing pattern (`ist-N.fK.fail.xml` / `ist-N.pK.pass.xml`):
   - `ist-8.p1.pass.xml` — both `started` and at least one `series.started` present, with `started <= min(series.started)`.
   - `ist-8.p2.pass.xml` — `started` absent (constraint trivially satisfied).
   - `ist-8.p3.pass.xml` — `series.started` absent on every series (constraint trivially satisfied via the inner `started.empty()` clause).
   - `ist-8.f1.fail.xml` — `started` strictly later than the only `series.started`.
   - `ist-8.f2.fail.xml` — multiple series, one of whose `series.started` is earlier than `started`.

### Design Rationale

- **Consistency with the existing constraint family.** The resolution explicitly frames the new check as a sibling of the `numberOf*` sanity constraints (`ist-3`…`ist-6`). Those are all attached to the root `ImagingStudy` element so they can reach across `ImagingStudy.started` ↔ `series.started` simultaneously, and the new rule has the same shape.
- **Phrasing of the FHIRPath.** The chosen form
  `started.empty() or series.where(started.empty() or started >= %resource.started).count() = series.count()`
  closely mirrors `ist-6` (`series.where(numberOfInstances.empty() or instance.empty() or numberOfInstances = instance.count()).count() = series.count()`), which is the canonical pattern for "for every series, predicate holds" in this resource.
- **Why `>=` instead of `>` in the expression.** The human text says "less than or equal to all `series.started` values," i.e. study-start ≤ every series-start. Equivalently, every series-start ≥ study-start. The resolution comment notes the original draft had a stricter `<` form and was corrected so the expression matches the human "less than or equal" wording — an equal study-start and series-start is valid (a series may begin at the same instant the study is opened).
- **Why error-severity.** A `started` later than `series.started` is not a stylistic issue — it is an internal contradiction in the resource. The other `ist-*` sanity rules are all `error`; matching that severity keeps validators behaving consistently.
- **Why no change to `series.started` cardinality or type.** The constraint is a cross-element invariant only; there is no need to alter cardinality (`0..1` on both ends) or type (`dateTime` on both ends). Existing `min`/`max`/`type` are unchanged.

### Breaking Changes

This is a **soft breaking change for conformance**. No element type, cardinality, or binding changes. However, instances that were previously valid in which `ImagingStudy.started` is later than any `ImagingStudy.series.started` will now fail validation under `ist-8`. Implementations that intentionally model `started` as the time the study was *closed/finalized* rather than *opened*, or that derived `started` from a downstream timestamp, may suddenly see validation errors. This should be called out in the resource changes notes for the next release.

No element removal, no required-element addition, no type change, no required-binding promotion, no search-parameter change.

   - File: `source/imagingstudy/invariant-tests/ist-8.p1.pass.xml`
   - Action: Create a new minimal `ImagingStudy` instance with `id` `ist-8.p1.pass`, a single `urn:dicom:uid` identifier (so `ist-1` is satisfied), `started` set to e.g. `2024-01-01T08:00:00Z`, and one `series` whose `started` is `2024-01-01T08:30:00Z`. 5. **Add invariant-test pass instance: `started` absent**
   - File: `source/imagingstudy/invariant-tests/ist-8.p2.pass.xml`
   - Action: Create an `ImagingStudy` with no `started` element and one or more `series.started` values. Confirms the leading `started.empty()` short-circuit.
   - Details: Keep `series` minimal; satisfy `ist-1` with a single `urn:dicom:uid` identifier when including a series.

6. **Add invariant-test pass instance: no `series.started`**
   - File: `source/imagingstudy/invariant-tests/ist-8.p3.pass.xml`
   - Action: Create an `ImagingStudy` with `started` set and one `series` element that omits `started`. Confirms the inner `started.empty()` per-series short-circuit.
   - Details: Same identifier/series pattern as p1.

7. **Add invariant-test fail instance: single series with earlier `series.started`**
   - File: `source/imagingstudy/invariant-tests/ist-8.f1.fail.xml`
   - Action: Create an `ImagingStudy` where `started = 2024-01-01T09:00:00Z` and the single `series.started = 2024-01-01T08:00:00Z`.
   - Details: Validation should report a violation of `ist-8` only (do not violate any other `ist-*`).

8. **Add invariant-test fail instance: one of several series-starts is earlier**
   - File: `source/imagingstudy/invariant-tests/ist-8.f2.fail.xml`
   - Action: Create an `ImagingStudy` with two `series` elements; the first's `started` is later than `ImagingStudy.started`, the second's `started` is earlier. Confirms the `count() = series.count()` aggregation semantics.

### Testing Considerations

- **Revalidate ImagingStudy examples.** Run the publisher / validator over `source/imagingstudy/imagingstudy-example.xml` and `source/imagingstudy/imagingstudy-example-xr.xml` to confirm they still pass after `ist-8` is added. If either example has `started` later than any `series.started`, fix the example timestamps in a separate, clearly-scoped commit.
- **Edge cases to verify:**
  - `started` present, no `series` at all → should pass (the `series.where(...).count() = series.count()` with `series.count() = 0` is `0 = 0`).
  - `started` present, every `series.started` absent → should pass (inner `started.empty()` short-circuits each series).
  - Equal timestamps (`started == series.started`) → should pass (uses `>=`, not `>`).
  - Series-start microseconds before study-start → should fail (no rounding tolerance is intended).
- **Interoperability considerations.** DICOM-derived `StudyDate`/`StudyTime` and per-series `SeriesDate`/`SeriesTime` should already satisfy this rule for any well-formed DICOM origin; flag in the change notes that re-encoders should not down-fill `ImagingStudy.started` with the *latest* series start.
- **Cross-validator parity.** Confirm the new invariant is exercised by both Java and .NET reference validators (the existing `ist-*` invariants are; `ist-8` should be equivalent because it uses only standard FHIRPath operators and `%resource`).

### Open Questions

- **`%resource` scope assumption.** The proposed expression uses `%resource.started` from inside `series.where(...)`. This depends on `%resource` resolving to the root `ImagingStudy` even when the constraint is evaluated against a `series` sub-collection. Confirm this is the intended interpretation across the validators in scope; if any validator scopes `%resource` differently, the expression may need to be rewritten using `$this` capture (e.g., `started.empty() or series.where(started.empty() or started >= started).count() = series.count()` evaluated with appropriate context binding). The resolution wording assumes the `%resource` form works.
- **Should the new invariant also be added to FHIR R4/R4B errata or only R5+?** The ticket targets `FHIR Core (FHIR)` without a version qualifier; the implementing commit will determine the target branch. Confirm with the Imaging Integration WG whether a back-port is in scope.
- **Are existing in-the-wild ImagingStudy instances with `started > series.started` known to exist?** If so, the change notes should highlight this as a substantive validation tightening rather than a silent fix, even though formally only the constraint set is new.
