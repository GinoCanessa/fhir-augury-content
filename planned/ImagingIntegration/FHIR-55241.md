## Resolution Summary

The resolution (`Persuasive with Modification`) requires two coordinated changes
to the `ImagingSelection` resource so that the mandatory
`instance.imageRegion2D.regionType` element always has a usable code:

1. **Terminology — add an "unknown" escape valve.** Update the value set
   `http://hl7.org/fhir/ValueSet/imagingselection-2dgraphictype` (sourced from
   the matching code system
   `http://hl7.org/fhir/imagingselection-2dgraphictype`) to include a new
   concept:

   | system | code | display | definition |
   |---|---|---|---|
   | `http://hl7.org/fhir/imagingselection-2dgraphictype` | `unknown` | `UNKNOWN` | `Unknown is used only when none of the other values apply` |

2. **Definitions — fix the DICOM cross-reference comment** on
   `ImagingSelection.instance.imageRegion2D.regionType`:
   - Remove the link to **DICOM PS3.3 C.10.5.1.2**
     (`https://dicom.nema.org/medical/dicom/current/output/chtml/part03/sect_C.10.5.html#sect_C.10.5.1.2`).
   - Replace it with **DICOM PS 3.3 C.18.6.1.2**
     (`https://dicom.nema.org/medical/dicom/current/output/chtml/part03/sect_C.18.6.html#sect_C.18.6.1.2`).

Both items appear to be implemented in the in-flight HL7 FHIR pull request
[HL7/fhir#4043](https://github.com/HL7/fhir/pull/4043) (commit
`d473d96d2dfaa21c75670a1b674a92612b567bc2` cited by jwhitby in the ticket
comments).

## Feature Proposal

### Problem Statement

`ImagingSelection.instance.imageRegion2D.regionType` is a mandatory element
(`min = 1`) with a required binding to
`ValueSet/imagingselection-2dgraphictype`. The current value set is closed
(`immutable = true`, `compose.include` of the single code system
`imagingselection-2dgraphictype`, complete content) and contains only the five
DICOM-derived geometric primitives `point`, `polyline`, `multipoint`, `circle`,
`ellipse`. There is no `unknown` / `other` code, so a system that:

- needs to record a 2D region whose type cannot be expressed in the existing
  five concepts (e.g., a region described in a future R7/R8 vocabulary that is
  being down-converted to R6), or
- has lost or never received the original region-type information,

cannot produce a conformant `ImagingSelection.instance.imageRegion2D` instance
at all, even though the surrounding backbone is otherwise complete. This
violates the general FHIR pattern of providing an escape-valve concept on every
required-bound mandatory element so down-conversion and partial-information
scenarios remain expressible. (See discussion between Lloyd McKenzie and
jwhitby on the ticket: the workaround of treating "other" as a point cloud
would silently change semantics rather than acknowledge the unknown.)

Separately, the DICOM `<comment>` annotation attached to that same element
points at section **C.10.5.1.2** (Graphic Type within Graphic Annotation
Sequence), but the surrounding ImagingSelection comments and the sibling
`coordinate` element already point at the **C.18.6** family (Spatial 2D
Coordinates / Graphic Type), which is the correct DICOM home for these
primitives. The mismatch is a documentation defect that the same resolution
fixes.

### Proposed Change

**Change A — Add `unknown` to the 2D graphic-type code system and value set.**

In `source/imagingselection/codesystem-imagingselection-2dgraphictype.xml`,
append a new `<concept>` entry after the existing `ellipse` concept:

```xml
<concept>
  <code value="unknown"/>
  <display value="UNKNOWN"/>
  <definition value="Unknown is used only when none of the other values apply."/>
</concept>
```

The matching value set
`source/imagingselection/valueset-imagingselection-2dgraphictype.xml` requires
no compose change — it already pulls the entire
`http://hl7.org/fhir/imagingselection-2dgraphictype` code system via an
unrestricted `<include>`, so the new code is picked up automatically.

The new concept follows the convention used elsewhere in core (e.g., the
`unknown` codes in `data-absent-reason`, `null-flavor`-style escape valves):

- Code is lowercase (`unknown`) to match the other lower-case codes in the
  system (`point`, `polyline`, `multipoint`, `circle`, `ellipse`).
- Display is upper-case (`UNKNOWN`) to match the existing display convention
  (`POINT`, `POLYLINE`, …) which mirrors DICOM's CS VR style.
- Definition uses the exact wording supplied by the resolution: "Unknown is
  used only when none of the other values apply."

**Change B — Fix the DICOM cross-reference comment on `regionType`.**

In `source/imagingselection/structuredefinition-ImagingSelection.xml`, replace
the `<comment>` on the `ImagingSelection.instance.imageRegion2D.regionType`
element (currently at line 803):

- Before:
  `See [DICOM PS3.3 C.10.5.1.2](https://dicom.nema.org/medical/dicom/current/output/chtml/part03/sect_C.10.5.html#sect_C.10.5.1.2).`
- After:
  `See [DICOM PS 3.3 C.18.6.1.2](https://dicom.nema.org/medical/dicom/current/output/chtml/part03/sect_C.18.6.html#sect_C.18.6.1.2).`

This brings the regionType citation in line with the C.18.6 reference already
present on the sibling `coordinate` element (line 827) and matches the DICOM
section identified in the resolution.

### Design Rationale

- **Add a code rather than relax the binding.** The required binding to a
  closed terminology is by design (DICOM-aligned, normative since FHIR R6,
  `immutable = true`). Loosening to extensible would change the conformance
  contract and require re-balloting; adding a single new concept keeps the
  binding strength and immutability intact and is the lightest-touch fix that
  satisfies the use case.
- **`unknown` rather than `other`.** Lloyd McKenzie's comment on the ticket
  explicitly motivates `unknown`/`other` interchangeably as an escape valve;
  the resolution table chose `unknown`, matching the well-established core
  pattern (e.g., `data-absent-reason#unknown`,
  `administrative-gender#unknown`). It also aligns with DICOM's notion that an
  unrecognised or missing graphic-type cannot be mechanically rendered, but
  the surrounding selection should still be transmissible.
- **Re-using the existing code system.** The 2D graphic-type code system is
  ImagingSelection-specific and `complete`; adding a sixth concept does not
  conflict with any other content. There is no need for a parallel
  `data-absent-reason`-style extension because the resolution opts to keep the
  semantics inside the bound terminology.
- **Comment fix uses the existing C.18.6 anchor style** already present in the
  same StructureDefinition for the matching `coordinate` element (line 827),
  so the two related elements now cite the same DICOM section family and
  remain self-consistent.
- **Sibling ticket FHIR-55242** applies the identical pattern to
  `imageRegion3D.regionType` (3D graphic types). Both tickets are referenced
  from the same PR (HL7/fhir#4043), and both should be merged together so the
  2D and 3D escape-valve story is symmetric.

### Breaking Changes

No breaking changes identified.

- **Adding a new concept to an existing code system / value set is additive.**
  The value set has `immutable = true`, but immutability prevents *redefinition*
  of existing membership; appending a new concept to the underlying code
  system is the canonical mechanism for evolving such value sets between
  ballot cycles. Existing instances using `point`, `polyline`, `multipoint`,
  `circle`, or `ellipse` remain valid byte-for-byte.
- **Required-bound consumers** that exhaustively switch on the five existing
  codes will see a sixth value when interoperating with newer producers; this
  is the intended behaviour of the escape valve and is the same risk profile
  as every prior `unknown`-style concept added in core.
- **Comment update is purely documentary** — the URL swap does not affect
  conformance, validation, or wire format.
- The CodeSystem already has `version = 6.0.0`; if the publish process
  treats an in-cycle concept addition as needing a version bump, that is a
  publisher-side decision and is called out under Open Questions, but it is
  not a breaking change to consumers.
