## Resolution Summary

Resolution: **Persuasive**.

The committee agreed that the prose paragraphs that introduce the
`dicom-wado-rs` and `dicom-wado-uri` `Endpoint.connectionType` codes in the
ImagingStudy and ImagingSelection notes should be reinforced with concrete
JSON/XML example references, rather than describing them only in narrative
form. The applied solution is two-fold:

1. **Add a WADO-URI example to the Endpoint resource** (tracked as the
   prerequisite ticket [FHIR-56147](https://jira.hl7.org/browse/FHIR-56147)
   "Add Example WADO-URI Endpoint"). A WADO-RS example
   (`endpoint-example-wadors`) already exists in the clone; the WADO-URI
   counterpart does not.
2. **Update the ImagingStudy and ImagingSelection endpoint notes** so that
   the existing `dicom-wado-rs` and `dicom-wado-uri` paragraphs each link
   to the corresponding endpoint example page. Per the resolution
   description (XML snippet quoted on the ticket):

   > An `Endpoint.connectionType` of code `dicom-wado-rs`, system
   > `http://terminology.hl7.org/CodeSystem/endpoint-connection-type`,
   > identifies a DICOM WADO-RS service (see
   > [example of DICOM WADO-RS endpoint](/endpoint-example-wadors.html)).

   …and analogously for `dicom-wado-uri` pointing at
   `/endpoint-example-wado-uri.html`.

The author's referenced implementation is in
[HL7/fhir PR #4043](https://github.com/HL7/fhir/pull/4043) (commit
`c429239eac28753c38f5211945b94996e2c86b05`).

## Feature Proposal

### Problem Statement

The current ImagingStudy and ImagingSelection notes describe the two
DICOM endpoint connection types (`dicom-wado-rs`, `dicom-wado-uri`) only
in prose. A reader who wants to see a concrete instance of an Endpoint
resource configured for either service has no in-page link to follow,
even though a WADO-RS endpoint example
(`endpoint-example-wadors`) already ships with the spec. The reporter
asked for an explicit JSON/XML representation reference at both the
WADO-RS sentence (10.4.5.3.1) and the WADO-URI sentence (10.4.5.3.2);
the WG agreed to provide both representations via links to canonical
example pages, and to add a WADO-URI example since none exists today.

### Proposed Change

Two coordinated changes in `HL7/fhir`:

1. **Add the missing WADO-URI Endpoint example** (under FHIR-56147 — see
   Prerequisites). Create a sibling of
   `source/endpoint/endpoint-example-wadors.xml` named
   `source/endpoint/endpoint-example-wado-uri.xml` whose
   `Endpoint.connectionType` carries
   `system="http://terminology.hl7.org/CodeSystem/endpoint-connection-type"`
   and `code="dicom-wado-uri"`, with `Endpoint.address` set to the
   conventional fictional base URL `https://pacs.hospital.org/wado-uri`
   used elsewhere in the imaging notes. Wire the new example into:
   - `source/endpoint/list-Endpoint-examples.xml` (so the example is
     advertised in the resource's example list), and
   - `source/endpoint/endpoint-examples-header.xml` (so it gets a row in
     the rendered examples table; mirror the row produced for
     `endpoint-example-wadors`).

2. **Add example links** to the four `<h4>WADO-RS</h4>` /
   `<h4>WADO-URI</h4>` introductory paragraphs in the imaging notes. In
   each of these paragraphs append a parenthetical
   "(see [example of DICOM WADO-RS endpoint](endpoint-example-wadors.html))"
   or "(see [example of DICOM WADO-URI endpoint](endpoint-example-wado-uri.html))"
   immediately after the existing "identifies a DICOM WADO-… service."
   sentence. The four affected blocks are:
   - `source/imagingstudy/imagingstudy-notes.xml` line **82** (WADO-RS,
     under the `<a name="wado-rs"></a>` anchor at line 78).
   - `source/imagingstudy/imagingstudy-notes.xml` line **143** (WADO-URI,
     under the `<a name="wado-uri"></a>` anchor at line 140).
   - `source/imagingselection/imagingselection-notes.xml` line **211**
     (WADO-RS, under `<a name="wado-rs"></a>` at line 207).
   - `source/imagingselection/imagingselection-notes.xml` line **243**
     (WADO-URI, under `<a name="wado-uri"></a>` at line 240).

   Use page-relative hrefs (`endpoint-example-wadors.html`,
   `endpoint-example-wado-uri.html`) — these are the standard publisher-
   generated example pages alongside `endpoint.html`.

Example of the resulting sentence (WADO-RS variant, as quoted in the
resolution):

```xml
<p>
An <code><a href="endpoint.html">Endpoint</a>.connectionType</code> of code
<code>dicom-wado-rs</code>, system
<code>http://terminology.hl7.org/CodeSystem/endpoint-connection-type</code>,
identifies a DICOM WADO-RS service
(see <a href="endpoint-example-wadors.html">example of DICOM WADO-RS endpoint</a>).
</p>
```

### Design Rationale

- The publisher already renders an Endpoint example page per example in
  both JSON and XML (e.g. `endpoint-example-wadors.json.html` and
  `endpoint-example-wadors.xml.html` are linked from the existing
  imagingstudy notes), so a single `…-wadors.html` / `…-wado-uri.html`
  link gives the reader access to both serialisations on the rendered
  spec — directly answering the reporter's request for "JSON/XML
  representation".
- jeroen explicitly asked for "both, so not only JSON/XML"; using the
  example landing page (which links to both formats *and* keeps the
  prose unchanged) preserves the existing narrative while adding the
  concrete instance reference.
- Linking to `endpoint-example-wadors` rather than copying its contents
  inline keeps the imaging notes cohesive and avoids duplicating
  example content.
- Adding the WADO-URI example file (FHIR-56147) closes the asymmetry
  flagged by the committee — the WADO-RS section already had an example
  to link to; WADO-URI did not.

### Open Questions

- **Wording consistency for the parenthetical link.** The resolution
  XML quotes "(see example of DICOM WADO-RS endpoint)" / "(see example
  of DICOM WADO-URI endpoint)". This plan adopts that wording verbatim;
  if the WG prefers a different visible link text (e.g. "see WADO-RS
  Endpoint example") the four edits should be updated identically.
- **FHIR-56147 status.** At cache time FHIR-56147 is `Submitted` /
  `Unresolved`. If the WG has not approved the new example by the time
  this ticket is implemented, either roll the example creation into the
  same PR (jwhitby has offered to do so) or split the work so that
  task 5 / task 7 (the WADO-URI links) ship together with the example
  file to avoid dangling references.
- **Leading-slash hrefs.** The resolution snippet uses `/endpoint-…html`
  (root-relative). Existing intra-spec links in these notes files use
  page-relative hrefs without a leading slash (e.g. `endpoint.html`,
  `patient.html`). This plan follows the existing convention; confirm
  with the work group / publisher behaviour if root-relative is in fact
  required.
