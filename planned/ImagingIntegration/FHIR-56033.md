## Resolution Summary

**Resolution: Persuasive.** The reporter observed that the dedicated "Accession Number" prose in the Imaging Study Implementation Notes (sections 10.4.5.2, with later cross-references at 10.4.5.5 and 10.4.5.7) describes accession numbers being carried on `ImagingStudy.basedOn.identifier` (with `ACSN` type), while the `<comment>` on `ImagingStudy.identifier` simultaneously suggests that "an accession number MAY be used to relate a Task, an ImagingStudy and a DiagnosticReport." Those two descriptions are inconsistent.

The agreed resolution is to **remove the accession-number example sentence from the `ImagingStudy.identifier` element's `<comment>`**, leaving the Implementation Notes as the single authoritative location that describes accession-number handling. Jonathan Whitby confirmed via the `<applied>` comment on 2026-04-01 with commit [c429239](https://github.com/HL7/fhir/commit/c429239eac28753c38f5211945b94996e2c86b05) ("Remove reference to Accession Number under ImagingStudy.identifier").

## Feature Proposal

### Problem Statement

The ImagingStudy resource describes how accession numbers are conveyed in two different places, which conflict:

1. **Imaging Study Implementation Notes** (`source/imagingstudy/imagingstudy-notes.xml`, lines 33–308 and 401–438) — has a dedicated "Accession Number" subsection (`<a name="accession-number"></a>`, section 10.4.5.2) and follow-on text in 10.4.5.5/10.4.5.7 that explicitly route accession numbers through `ImagingStudy.basedOn.identifier` using the `ACSN` identifier type, with example JSON. The corresponding `ImagingStudy.basedOn` element in the StructureDefinition repeats this guidance ("If the ImagingStudy is associated with an Accession Number, this field SHOULD include a reference to that value … identifier.type = ACSN", `<alias value="AccessionNumber"/>`).
2. **`ImagingStudy.identifier` element comment** (`source/imagingstudy/structuredefinition-ImagingStudy.xml`, current line ~157 in the pre-commit baseline) — contains the contradictory illustration: "*…or an accession number MAY be used to relate a Task, an ImagingStudy and a DiagnosticReport.*" This implies accession numbers may live directly on `ImagingStudy.identifier`, which conflicts with the dedicated guidance and with the established Imaging Integration WG position that DICOM Study Instance UID is the canonical value for `ImagingStudy.identifier`.

The reporter requested that the mention of accession be removed from `ImagingStudy.identifier`. The committee agreed.

### Proposed Change

In `source/imagingstudy/structuredefinition-ImagingStudy.xml`, on the `ImagingStudy.identifier` `<element>` (path `ImagingStudy.identifier`), edit the `<comment value="…"/>` attribute to delete the example clause that mentions accession numbers. The pre/post wording is:

- **Before** (the relevant sentence inside the comment):
  > It is best practice for the identifier to only appear on a single resource instance, however business practices MAY occasionally dictate that multiple resource instances with the same identifier can exist - possibly even with different resource types.  **For example, multiple Patient and a Person resource instance might share the same social insurance number, or an accession number MAY be used to relate a Task, an ImagingStudy and a DiagnosticReport.**  See discussion under [Imaging Study Implementation Notes](imagingstudy.html#dicom-uids) for encoding of DICOM Study Instance UID.

- **After** (sentence removed in its entirety; surrounding text preserved):
  > It is best practice for the identifier to only appear on a single resource instance, however business practices MAY occasionally dictate that multiple resource instances with the same identifier can exist - possibly even with different resource types.  See discussion under [Imaging Study Implementation Notes](imagingstudy.html#dicom-uids) for encoding of DICOM Study Instance UID.

No other files are modified for this ticket. The `<requirements>`, `<alias>`, mapping entries, and the `Accession Number` prose in `imagingstudy-notes.xml` are correct as-is and are kept.

### Design Rationale

- Accession-number handling is already fully specified on `ImagingStudy.basedOn` (with `ACSN` identifier type) and described in the Implementation Notes. Repeating a contradictory example on `ImagingStudy.identifier` invited misuse.
- The Imaging Integration WG treats DICOM Study Instance UID as the typical/expected value for `ImagingStudy.identifier`; the surviving sentence already calls this out via the `dicom-uids` link.
- Deleting the example (rather than rewording it) is preferred because the generic "different resource types share an identifier" point is still illustrated by the social-insurance-number example, which is uncontroversial and not Imaging-specific.
- The reporter explicitly excluded the broader `basedOn`-vs-`identifier` design question from this ticket; the change is scoped to remove the inconsistency only.

- **DiagnosticReport / Task** — the deleted sentence implied a cross-resource identifier shared via `ImagingStudy.identifier`. No corresponding text needs to be added to those resources because accession-number linkage between Task → ImagingStudy → DiagnosticReport is achieved via `basedOn` references and the `ACSN` identifier type, which is already documented.
