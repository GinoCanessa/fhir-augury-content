## Resolution Summary

Resolution: **Persuasive with Modification**. Rewrite steps 6–12 of the
*Comprehensive Imaging Scheduled Workflow* use case in the ImagingStudy
page so that:

1. Step 6 gains a link to the `Endpoint` resource on the `endpoint`
   bullet (so every FHIR resource mentioned is linked, matching the
   style used in steps 1–5).
2. The radiology-reading sub-steps (current steps 8–10: workstation
   retrieves images, radiologist picks key images, workstation sends
   KOS to archive) are grouped under a single parent step titled
   **"Radiology reading workflow (non-FHIR)"** so it is explicit that
   those steps are outside FHIR scope.
3. Step 7 ("Image archive notifies EMR of imaging study") gains a
   nested bullet pointing at `imagingstudy.html#workflow` for possible
   notification mechanisms.
4. The "Image archive updates imaging study to include KOS" step uses
   the fully-qualified `ImagingStudy.modality` and `ImagingStudy.series`
   element paths (instead of the bare `modality` / "KOS series added").

The exact XML target provided by the resolution (`source/imagingstudy/imagingstudy-notes.xml`,
use-case section) is reproduced verbatim under *Proposed Change* below.

## Feature Proposal

### Problem Statement

In `source/imagingstudy/imagingstudy-notes.xml` under `<a name="use-case"></a>`
→ `<h4>Comprehensive Imaging Scheduled Workflow</h4>`, steps 1–6 link every
FHIR resource to its definition (e.g. `<a href="servicerequest.html">ServiceRequest</a>`,
`<a href="patient.html">Patient</a>`, `<a href="imagingstudy.html">ImagingStudy</a>`).
Starting at step 7 the prose drops that convention: "EMR", "imaging
study", "Radiology reading workstation", "KOS", and `modality`/`series`
are written as plain text. The reporter (Ana Kostadinovska) notes this
is confusing: readers cannot tell which nouns are FHIR resources/elements
and which are external systems. The resolution also flags that some
steps describe a DICOM-only radiology-reading workflow that has no FHIR
interaction at all; grouping them makes the FHIR-vs-non-FHIR boundary
explicit.

### Proposed Change

Edit the use-case list in
`source/imagingstudy/imagingstudy-notes.xml` (lines 253–286 in the
current clone at `db79dcd`). Replace the current `<ol>` with the structure
below.

- **Step 6 (existing):** add an `<a href="endpoint.html">Endpoint</a>`
  link to the `endpoint` bullet so it reads
  "`endpoint` points to image archive WADO-RS service
  (`<a href="endpoint.html">Endpoint</a>` resource)".
- **Step 7 (existing "Image archive notifies EMR…"):** keep the text
  but add a nested `<ul><li>` pointing at
  `<a href="imagingstudy.html#workflow">ImagingStudy Create and Update Workflow</a>`
  for "possible notification mechanisms" (the resolution's explicit
  wording).
- **Former steps 8–10 → one grouped step:** collapse the three
  workstation/radiologist/KOS-send steps under a new parent
  `<li>Radiology reading workflow (non-FHIR)</li>` whose nested `<ul>`
  holds the three original bullets unchanged in wording:
  - "Radiology reading workstation retrieves study images from image archive"
  - "Radiologist reads study and selects several images as 'key images'"
  - "Radiology reading workstation sends DICOM Key Object Selection (KOS) to image archive"
- **Former step 11 ("Image archive updates imaging study to include KOS"):**
  change the two nested bullets from bare identifiers to fully-qualified
  element paths — `<code>ImagingStudy.modality</code> is now US,KOS`
  and "KOS series details added to `<code>ImagingStudy.series</code>`".
- **Former step 12 ("Imaging study available to EMR clients"):**
  retain verbatim.

Concretely, the resulting outer list (steps 6–12 renumbered to 6–10
after grouping) is:

```xml
<li>Image archive creates an <a href="imagingstudy.html">ImagingStudy</a>
  <ul>
    <li><code>subject</code> is current <a href="patient.html">Patient</a></li>
    <li><code>basedOn</code> includes current <a href="servicerequest.html">ServiceRequest</a></li>
    <li><code>identifier</code> is Study Instance UID</li>
    <li><code>endpoint</code> points to image archive WADO-RS service (<a href="endpoint.html">Endpoint</a> resource)</li>
    <li><code>modality</code> is US</li>
  </ul>
</li>
<li>Image archive notifies EMR of imaging study
  <ul>
    <li>See <a href="imagingstudy.html#workflow">ImagingStudy Create and Update Workflow</a> for possible notification mechanisms</li>
  </ul>
</li>
<li>Radiology reading workflow (non-FHIR)
  <ul>
    <li>Radiology reading workstation retrieves study images from image archive</li>
    <li>Radiologist reads study and selects several images as "key images"</li>
    <li>Radiology reading workstation sends DICOM Key Object Selection (KOS) to image archive</li>
  </ul>
</li>
<li>Image archive updates imaging study to include KOS
  <ul>
    <li><code>ImagingStudy.modality</code> is now US,KOS</li>
    <li>KOS series details added to <code>ImagingStudy.series</code></li>
  </ul>
</li>
<li>Imaging study available to EMR clients</li>
```

No structural/StructureDefinition change is required; this is a pure
narrative edit. No terminology change. No schema change.

### Design Rationale

- Linking every FHIR resource mention is already the convention in steps
  1–5 of the same list and throughout the rest of `imagingstudy-notes.xml`
  (see `imagingselection.html`, `patient.html`, `servicerequest.html`,
  `observation.html`, `documentreference.html` links in the Boundaries
  and Relationships section of the sibling introduction file). Extending
  the convention to steps 6–12 is the least-surprise fix that the
  reporter asked for.
- Explicitly marking the radiology-reading sub-workflow **(non-FHIR)**
  is consistent with the sentence already in the section ("*Note:*
  reporting workflow is not described in this use case.") and with
  imaging WG practice of delimiting DICOM-native steps from FHIR-bearing
  steps. It avoids readers assuming KOS send/retrieve is a FHIR
  interaction.
- Using `ImagingStudy.modality` / `ImagingStudy.series` (FHIR Path form)
  instead of bare `modality` / prose "KOS series added" matches the
  cross-reference the ingestion index already extracts for this ticket
  (see Keywords: `imagingstudy` as `fhir_path`, `ImagingStudy.series`
  as `fhir_path`) and is consistent with the applied-vote commit
  [c429239](https://github.com/HL7/fhir/commit/c429239eac28753c38f5211945b94996e2c86b05),
  which touched the same file with +23/-16 lines in
  `source/imagingstudy/imagingstudy-notes.xml`.
- No change to the adjacent `ehr-usecase` block is warranted — it
  already uses fully-qualified element paths (`ImagingStudy.identifier`,
  `ImagingStudy.modality`, `ImagingStudy.endpoint`, etc.), so the
  clarity problem is genuinely confined to the `scheduled` use case.

### Open Questions

- **Endpoint link wording.** The resolution says "add link to Endpoint
  resource in step 6" but does not fix exact wording. The plan above
  uses the parenthesised form `(<a href="endpoint.html">Endpoint</a>
  resource)` to mirror how `<a href="patient.html">Patient</a>` etc. are
  linked in the same list. An alternative — wrapping the `<code>endpoint</code>`
  word directly in the anchor — would also be defensible. Confirm with
  the Imaging Integration WG / jwhitby before merging.
- **List renumbering.** Grouping the three radiology-reading steps
  collapses the 12-step list to 10 steps. If any other HL7 specification
  page or external reference cites "step 8 of the Comprehensive
  Imaging Scheduled Workflow" by number, those references will need
  updating. A repo-wide grep for "step 8"/"step 9"/"step 10"
  near `imagingstudy` is advised before merging.
- **Bundled ticket (FHIR-54777).** That ticket's resolution also edits
  the same block (attribute-level clarification,
  `ImagingStudy.endpoint` vs. `ImagingStudy.series.endpoint`). Coordinate
  so both sets of edits land together, matching the shape already
