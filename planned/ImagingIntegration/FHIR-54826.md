## Resolution Summary

**Resolution:** Persuasive.

Resolution description (verbatim): *"Remove note from end of https://build.fhir.org/imagingselection.html#10.5.5.1"*.

The closing note in the **Imaging Selection Subsets** section (§10.5.5.1) of the ImagingSelection page asserts that each ImagingSelection instance can only contain one type of subset, on the grounds that a DICOM Series contains only one type of instance. That premise is incorrect — a DICOM Series can contain instances with different SOP Classes (e.g., image instances mixed with Presentation State or SR instances), so the derived claim is unsound. The agreed fix is to delete the note entirely; no replacement text is required by the resolution.

The applied commit (already merged on master) is [`c429239`](https://github.com/HL7/fhir/commit/c429239eac28753c38f5211945b94996e2c86b05) — *"FHIR-54826 ** Remove unneccessary note from ImagingSelection 10.5.5.1"* — which modified `source/imagingselection/imagingselection-notes.xml`.

## Feature Proposal

### Problem Statement

The final paragraph of the *Imaging Selection Subsets* subsection of the ImagingSelection notes page currently reads:

> *Note: Because an imaging selection instance is limited to a single DICOM Series and each DICOM Series can contain only one type of instance - image, SR document, radiotherapy object, etc. - each imaging selection instance can only contain one type of subset.*

This is factually incorrect. A DICOM Series can legitimately contain SOP Instances with different SOP Classes (`instance.sopClass` values). The prose above the note already communicates, correctly, that *the value of the `instance.sopClass` element determines which type of subset can be present* — i.e. the constraint is at the **instance** level, not the **series** level. The mis-stated note therefore contradicts the surrounding text and must be removed.

### Proposed Change

Delete the `<p>Note: …</p>` element at the end of the `<h3>Imaging Selection Subsets</h3>` block in `source/imagingselection/imagingselection-notes.xml` (line 49 of the current file on `master` @ `db79dcd`). Specifically:

- Remove exactly this element:

  ```xml
  <p>Note: Because an imaging selection instance is limited to a single DICOM Series and each DICOM Series can contain only one type of instance - image, SR document, radiotherapy object, etc. - each imaging selection instance can only contain one type of subset.</p>
  ```

- Leave the preceding explanatory prose (paragraph "The value of the `instance.sopClass` element determines which type of subset can be present.", the `For example:` list, and the two `<li>` bullets for Enhanced CT / Enhanced SR SOP Classes) in place — they are correct and self-contained once the trailing note is removed.
- Leave the `<h3>Imaging Selection Subsets</h3>` heading and its anchor in place so that any inbound `#…` links from other pages remain valid.

No other file in the `source/imagingselection/` folder, no structure definition, no example, and no terminology artifact is affected by this change.

### Design Rationale

The resolution is documentation-only and narrowly scoped: the surrounding paragraphs (and the two illustrative SOP Class examples for Enhanced CT and Enhanced SR) already convey the correct model — subset applicability is driven per-instance by `instance.sopClass`, not per-series. Rewriting the note risks re-asserting a constraint that does not exist in DICOM. The workgroup-approved action is therefore simple deletion. The committer's own message on `c429239` confirms the intent: *"Remove unneccessary note from ImagingSelection 10.5.5.1"*.
