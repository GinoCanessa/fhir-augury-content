## Resolution Summary

**Resolution:** Persuasive.

The reporter pointed out that section 10.4.9.5 *ImagingStudy — DICOM Tag
Mapping* contains a stray `+` in the DICOM Tag Mapping column for the
**specimen** row. The committee agreed: the `+` is superfluous and must
be removed so the mapping reads naturally as a list of DICOM attributes
joined by `and`.

The applied fix (referenced by jwhitby in the only ticket comment) is in
`HL7/fhir` PR #4043, commit `79ad0b0`, which edits the dicom mapping on
`ImagingStudy.series.specimen` inside the resource's StructureDefinition
so that:

- **Before:** `Specimen Identifier (0040,0551) or Specimen UID
  (0040,0554) and + Issuer of the Specimen Identifier Sequence
  (0040,0562)`
- **After:** `Specimen Identifier (0040,0551) or Specimen UID
  (0040,0554) and Issuer of the Specimen Identifier Sequence
  (0040,0562)`

The commit also fixes the analogous stray `+` on `ImagingStudy.series.started`
(`Series Date (0008,0021) + and Series Time (0008,0031)` → `Series Date
(0008,0021) and Series Time (0008,0031)`) and a double-space typo in
invariant `ist-3`. Those are out of scope for FHIR-54785 itself — the
sibling ticket FHIR-54786 covers the `series.started` mapping — but they
share the same PR/commit and the same file, so they need to be
coordinated to avoid merge conflicts.

## Feature Proposal

### Problem Statement

The R6 DICOM Tag Mapping table for `ImagingStudy` (specification page
section 10.4.9.5, generated from the `<mapping identity="dicom">`
entries inside `structuredefinition-ImagingStudy.xml`) renders the
specimen mapping as:

> Specimen Identifier (0040,0551) or Specimen UID (0040,0554) **and +**
> Issuer of the Specimen Identifier Sequence (0040,0562)

The `+` has no operator semantics in this column — the column uses
plain English connectors (`or`, `and`) — so it reads as a typo and
makes the implementer guess whether `+` carries any additional meaning
beyond `and`. The reporter (Jeroen Medema) flagged it under the
`R6-DICOMTagMapping` label as part of the broader R6 review of the
DICOM mapping table.

### Proposed Change

Edit a single `<map value="…"/>` string inside the `<mapping
identity="dicom">` block of the `ImagingStudy.series.specimen`
element in `source/imagingstudy/structuredefinition-ImagingStudy.xml`:

- **Element:** `ImagingStudy.series.specimen`
- **Mapping identity:** `dicom`
- **Old `map` value:**
  `Specimen Identifier (0040,0551) or Specimen UID (0040,0554) and + Issuer of the Specimen Identifier Sequence (0040,0562)`
- **New `map` value:**
  `Specimen Identifier (0040,0551) or Specimen UID (0040,0554) and Issuer of the Specimen Identifier Sequence (0040,0562)`

No other elements, cardinalities, types, bindings, invariants, search
parameters, or examples change. The DICOM tag references (`(0040,0551)`,
`(0040,0554)`, `(0040,0562)`) are preserved verbatim. The publisher will
regenerate the rendered DICOM Tag Mapping table on `imagingstudy.html`
from this differential value.

### Design Rationale

- The DICOM Tag Mapping column is human-readable prose with `or` / `and`
  connectives; introducing arithmetic-style `+` is inconsistent with
  every other row in the table and was unintentional.
- Removing the `+` (rather than redefining what `+` means) preserves the
  established convention and matches the wording the reporter and the
  committee approved (`Persuasive`).
- The fix is the minimum viable edit — a single token deletion plus the
  surrounding whitespace — which is exactly what was applied in commit
  `79ad0b0`. No alternative wording (e.g., `, plus`) was considered in
  the discussion; the committee accepted the deletion outright.
