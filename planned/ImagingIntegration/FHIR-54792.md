## Resolution Summary

**Resolution:** Persuasive — *"Make sections consistent between imagingselection and imagingstudy."*

The WADO-URI sub-section in `ImagingSelection` ends after the basic WADO-URI
GET example. `ImagingStudy`'s WADO-URI sub-section continues with three
additional paragraphs that describe (a) the additional query parameters that
control rendered dimensions, quality, region of interest, brightness/contrast
adjustments, demographic burn-in and frame selection; (b) an extended GET
example that uses `rows`, `columns`, and `region`; and (c) a pointer to
DICOM PS 3.18 Chapter 9. The resolution requires those three paragraphs to be
copied into the corresponding WADO-URI block of `ImagingSelection` so that
both resources document WADO-URI identically.

The same PR (`HL7/fhir#4043`, commit
[`19581c2`](https://github.com/HL7/fhir/pull/4043/commits/19581c2d8b644a45ef2bbc0c8e0a17f6cdd21dd3))
also addresses the sibling `WADO-RS` alignment ticket FHIR-54790; this plan is
scoped only to the WADO-URI portion of FHIR-54792 (the ticket title and body
are explicit about the WADO-URI section), but a developer executing this plan
should expect to encounter both blocks in the same edit window.

## Feature Proposal

### Problem Statement

`ImagingStudy` and `ImagingSelection` both contain a `<h4>WADO-URI</h4>`
sub-section under their respective Notes pages
(`source/imagingstudy/imagingstudy-notes.xml` and
`source/imagingselection/imagingselection-notes.xml`). The introductory
prose and the canonical `GET …/wado-uri?requestType=WADO&studyUID=…` example
are present in both files. However the trailing content — the paragraph
beginning *"Additional query parameters can control other aspects of the
rendering including rendered dimensions, …"*, the extended example with
`rows=100&columns=100&region=0,0,0.5,1`, and the *"For further details on
DICOM WADO-URI capabilities …"* link to DICOM PS 3.18 Chapter 9 — is only
present in the `ImagingStudy` Notes page. Readers viewing
`imagingselection.html#wado-uri` therefore lose the rendering-parameter
guidance and the upstream DICOM specification reference. This is an
inconsistency between two closely-related resources that both reference the
same WADO-URI service.

### Proposed Change

In `source/imagingselection/imagingselection-notes.xml`, immediately after
the existing `<pre>…Accept: application/dicom</pre>` block that closes the
basic WADO-URI example (and immediately before the `<a name="laterality"></a>`
anchor that starts the next section), insert the same three paragraphs that
follow the equivalent example in
`source/imagingstudy/imagingstudy-notes.xml`:

1. A `<p>` paragraph describing the additional query parameters that can
   control rendering: rendered dimensions, quality (compression ratio),
   region of interest, brightness/contrast (window center/width), whether
   to "burn" patient or study demographics into the rendered result, and
   which frame of a multiframe instance to retrieve.
2. A `<p>` paragraph introducing the extended example, followed by a `<pre>`
   block containing
   `GET https://pacs.hospital.org/wado-uri?requestType=WADO&amp;studyUID=…&amp;seriesUID=…&amp;objectUID=…&amp;<b>rows=100&amp;columns=100&amp;region=0,0,0.5,1 </b> HTTP/1.1`
   with an `Accept: image/jpeg` header. Reuse the same Study/Series/SOP
   Instance UIDs already used in the preceding ImagingSelection example so
   the URLs read as a continuation rather than a fresh scenario.
3. A `<p>` paragraph stating *"For further details on DICOM WADO-URI
   capabilities including additional rendering parameters, see"* followed by
   an `<a href="https://dicom.nema.org/medical/dicom/current/output/chtml/part18/chapter_9.html">DICOM
   PS 3.18 Chapter 9</a>` link.

Add a corresponding bullet to the change-log block at the top of
`source/imagingselection/imagingselection-introduction.xml`, e.g.:

```xml
<li>Update ImagingSelection <code>#wado-uri</code> section to be consistent with ImagingStudy.</li>
```

(If FHIR-54790 is being applied at the same time the bullet may be combined
to read *"Update ImagingSelection `#wado-rs` and `#wado-uri` sections to be
consistent with ImagingStudy."*, mirroring the wording already used in commit
`19581c2`.)

No StructureDefinitions, search parameters, examples, ValueSets, or
CodeSystems are touched. The change is confined to two HTML/XML narrative
files under `source/imagingselection/`.

### Design Rationale

* The Imaging Integration WG resolved this as **Persuasive** with the
  one-line direction *"Make sections consistent between imagingselection and
  imagingstudy."* The cleanest, lowest-risk reading is to copy the missing
  prose verbatim from `ImagingStudy` rather than to remove the trailing
  prose from `ImagingStudy` or to write a fresh, paraphrased version — both
  ImagingStudy and ImagingSelection point at the same WADO-URI service so
  the wording is equally accurate for both.
* Using the same Study/Series/SOP Instance UIDs that already appear in the
  ImagingSelection WADO-URI example keeps the extended example readable as
  a continuation and matches the pattern used on the ImagingStudy page.
* The change-log entry under
  `imagingselection-introduction.xml` is the standard place where
  Imaging Integration records cross-version notes for `ImagingSelection`;
  the existing `<li>` entries for `imageRegion2D/3D` and DICOM UIDs follow
  the same pattern.
* Alternatives considered:
  * *Remove the trailing prose from ImagingStudy* — rejected: the WG vote
    was to add to ImagingSelection, not to subtract from ImagingStudy, and
    the ImagingStudy text predates ImagingSelection.
  * *Add a hyperlink from `imagingselection.html#wado-uri` to
    `imagingstudy.html#wado-uri`* — rejected: the resolution explicitly
    requires sectional consistency, not a redirect.

## Implementation Plan

### Prerequisites

- None. All required source content is already present in the working

### Step-by-Step Tasks

#### HL7/fhir

1. **Mirror the WADO-URI rendering-parameter prose into ImagingSelection.**
   - File: `source/imagingselection/imagingselection-notes.xml`
   - Action: Locate the WADO-URI block (anchor `<a name="wado-uri"></a>`
     followed by `<h4>WADO-URI</h4>`). Find the `<pre>` element that ends
     the basic WADO-URI GET example
     (`Accept: application/dicom` … `</pre>`), which is the last element
     in the WADO-URI block before the next anchor
     (`<a name="laterality"></a>`). Immediately after that closing
     `</pre>`, insert a `<p>` that begins *"Additional query parameters
     can control other aspects of the rendering including rendered
     dimensions, quality (compression ratio), the region of interest
     within the image to render, brightness/contrast (window center/width)
     adjustments, whether to "burn" patient or study demographics into the
     rendered result, and which frame of a multiframe instance to
     retrieve."* — using the same wording as the equivalent paragraph in
     `source/imagingstudy/imagingstudy-notes.xml` (look for the same
     opening phrase on that page). Use the typographic curly quotes
     (`“…”`) around `burn`, matching the source page.
   - Details: Insert as a sibling element of the surrounding `<p>` /
     `<pre>` blocks, preserving the existing indentation (2-space) used
     elsewhere in `imagingselection-notes.xml`. Do not wrap the inserted
     content in any new container element. Do not move or remove the
     `<a name="wado-uri"></a>` or `<a name="laterality"></a>` anchors.

2. **Add the extended WADO-URI example to ImagingSelection.**
   - File: `source/imagingselection/imagingselection-notes.xml`
   - Action: Immediately after the paragraph added in task 1, insert a
     `<p>` that introduces an extended example: *"The example above can be
     extended with parameters that request a 100x100 image thumbnail of
     the left half of the image to be retrieved:"* followed by a `<pre>`
     containing the extended GET line:
     `GET https://pacs.hospital.org/wado-uri?requestType=WADO&amp;studyUID=1.2.250.1.59.40211.12345678.678910&amp;seriesUID=1.2.250.1.59.40211.789001276.14556172.67789&amp;objectUID=1.2.250.1.59.40211.2678810.87991027.899772.2&amp;<b>rows=100&amp;columns=100&amp;region=0,0,0.5,1 </b> HTTP/1.1`
     and an `Accept: image/jpeg` line.
   - Details: Reuse the same Study Instance UID
     (`1.2.250.1.59.40211.12345678.678910`), Series Instance UID
     (`1.2.250.1.59.40211.789001276.14556172.67789`), and SOP Instance
     UID (`1.2.250.1.59.40211.2678810.87991027.899772.2`) that already
     appear in the basic WADO-URI example earlier in the same WADO-URI
     block, so the extended example reads as a continuation. The `<b>`
     emphasis on `rows=100&columns=100&region=0,0,0.5,1` matches the
     ImagingStudy page exactly. Use `&amp;` (not `&`) for query-string
     ampersands since the file is XML.

3. **Add the DICOM PS 3.18 Chapter 9 reference paragraph.**
   - File: `source/imagingselection/imagingselection-notes.xml`
   - Action: Immediately after the extended-example `<pre>` from task 2,
     insert a final `<p>` that reads: *"For further details on DICOM
     WADO-URI capabilities including additional rendering parameters,
     see"* followed by `<a href="https://dicom.nema.org/medical/dicom/current/output/chtml/part18/chapter_9.html">DICOM PS 3.18 Chapter 9</a>` and a closing period.
   - Details: This is the last new element inserted in the WADO-URI
     block. The next sibling element must remain the existing
     `<a name="laterality"></a>` / `<h3>Body Site and Laterality</h3>`
     pair — confirm that anchor is intact after the edit. Use the exact
     URL and link text from the ImagingStudy page so the rendered output
     matches.

4. **Add a cross-version change-log bullet to ImagingSelection.**
   - File: `source/imagingselection/imagingselection-introduction.xml`
   - Action: Inside the existing `<blockquote>` / `<ul>` change-log block
     near the top of the file (the same `<ul>` that contains the
     `imageRegion2D/3D` and DICOM-UIDs bullets), append a new `<li>`:
     `<li>Update ImagingSelection <code>#wado-uri</code> section to be consistent with ImagingStudy.</li>`
   - Details: If the WADO-RS sibling change (FHIR-54790) is being applied
     in the same commit, combine the bullet to match the upstream PR's
     wording: `<li>Update ImagingSelection <code>#wado-rs</code> and <code>#wado-uri</code> sections to be consistent with ImagingStudy.</li>`.
     Preserve the existing indentation pattern of the surrounding `<li>`
     elements.

5. **Verify rendered output and link integrity.**
   - File: (no edit) — inspection of `source/imagingselection/`,
     `source/imagingstudy/`, and any cross-references in `source/*.html`.
   - Action: Run a repo-wide search for `imagingselection.html#wado-uri`
     and for the bare `#wado-uri` anchor on adjacent pages to confirm no
     inbound link is broken by the edits, and re-confirm that
     `<a name="wado-uri"></a>` and `<a name="laterality"></a>` survive
     unmodified inside `imagingselection-notes.xml`.
   - Details: If the local environment supports a partial HL7 publisher
     run (`./build.sh -resource imagingselection`), regenerate the
     ImagingSelection page and visually compare its WADO-URI block
     against the ImagingStudy page side-by-side. The two `<h4>WADO-URI</h4>`
     sections should now contain the same paragraphs and the same
     extended example shape (only the leading anchor and headings differ
     in surrounding context).

### Open Questions

- **Scope of the bullet:** Should the new bullet in
  `imagingselection-introduction.xml` cover only WADO-URI (this ticket),
  or also WADO-RS (FHIR-54790)? Commit `19581c2` on PR
  [`HL7/fhir#4043`](https://github.com/HL7/fhir/pull/4043) chose a single
  combined bullet. If FHIR-54790 is being implemented separately, this
  plan should produce a WADO-URI-only bullet; otherwise prefer the
  combined wording that the WG-approved commit already uses.
- **Verbatim copy vs. light editorialisation:** The resolution
  (*"Make sections consistent"*) does not specify whether the wording
  must be identical or merely equivalent. This plan assumes verbatim
  copy from `imagingstudy-notes.xml` because that is the lowest-risk
  reading, matches commit `19581c2`, and avoids introducing new editorial
  content that would itself need WG review. If the WG prefers a single
  shared snippet (e.g., extracted into an include), that is a separate,
  larger change that should be raised as a follow-up ticket.
- **Out-of-scope concerns surfaced in the ticket:** None — unlike the
  related mapping-language ticket flagged in the repo briefing, the
  comments on FHIR-54792 contain only the PR-commit link and no
  out-of-scope items. Nothing additional needs to be split into a
  follow-up ticket.
