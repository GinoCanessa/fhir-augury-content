## Resolution Summary

Resolution: **Persuasive** — "Update text as proposed."

The reporter flagged a wording bug in the ImagingSelection notes. The existing bullet reads:

> `frameNumber` to identify the list of frames included in the selection

…and must be changed to:

> `frameNumber` to identify the list of **frame numbers** included in the selection

The phrase "list of frames" is misleading because `ImagingSelection.instance.frameNumber` is a repeating `positiveInt` — it carries frame numbers, not the frames themselves. The element's own StructureDefinition already says "A list of frame numbers selected from a multiframe SOP Instance" (see `source/imagingselection/structuredefinition-ImagingSelection.xml:717`), so this resolution just brings the Notes prose into line with the element definition.

## Feature Proposal

### Problem Statement

The ImagingSelection resource Notes page contains a bullet in the "Multiframe Image Selection" sub-section that describes `instance.frameNumber` as identifying a "list of frames included in the selection". This is imprecise: `instance.frameNumber` is a repeating `positiveInt` that carries the ordinal frame numbers within a multiframe SOP Instance, not the frame payloads. A reader comparing the Notes prose to the StructureDefinition for `ImagingSelection.instance.frameNumber` (whose `definition` already says "A list of frame numbers selected from a multiframe SOP Instance") sees an inconsistency between the two. The resolution is to correct the Notes so the terminology lines up with the element definition.

### Proposed Change

Make a single wording change in `source/imagingselection/imagingselection-notes.xml`, in the "Multiframe Image Selection" sub-section's bulleted list describing the required `instance` content.

At line 128:

```xml
<li><code>frameNumber</code> to identify the list of frames included in the selection</li>
```

becomes:

```xml
<li><code>frameNumber</code> to identify the list of frame numbers included in the selection</li>
```

No other element changes, cardinality changes, terminology bindings, invariants, search parameters, or examples are required. The jwhitby comment on the ticket points at the exact commit (`HL7/fhir@d473d96`) that applies this wording change as part of PR #4043 (a roll-up PR for several Imaging Integration tickets).

### Design Rationale

Chosen because:

- The resolution on the Jira ticket is a literal string replacement ("Update text as proposed.") — the reporter proposed the new wording verbatim and the work group accepted it as `Persuasive`.
- The replacement restores consistency with `ImagingSelection.instance.frameNumber`'s existing `definition` in `structuredefinition-ImagingSelection.xml`, which already uses "frame numbers".
- Alternative — rewording the whole paragraph — was not proposed and would exceed the resolution's intent.
- Alternative — changing the StructureDefinition `short`/`definition` — is unnecessary because those are already correct.
