## Resolution Summary

**Resolution:** Persuasive.

The resolution requires extending the `imagingselection-3dgraphictype` CodeSystem
(`http://hl7.org/fhir/imagingselection-3dgraphictype`) with an additional concept:

| System | Code | Display | Definition |
|--------|------|---------|------------|
| `http://hl7.org/fhir/imagingselection-3dgraphictype` | `unknown` | `UNKNOWN` | `Unknown is used only when none of the other values apply` |

The reason given by the submitter (Lloyd McKenzie): the element
`ImagingSelection.imageRegion3D.regionType` is mandatory (`min = 1`) with a
required binding, but the existing value set offers no `unknown`/`other`
escape valve, leaving systems with no way to populate the element when none
of the listed graphic types applies.

A linked PR comment from Jonathan Whitby
(<https://github.com/HL7/fhir/pull/4043>, commit
`d473d96d2dfaa21c75670a1b674a92612b567bc2`) confirms the work landed under
the joint message `FHIR-55241 / FHIR-55242 ** Added "unknown" options to
image3D.regionType and image2D.regionType`. FHIR-55242 owns the **3D** half
of that pair (FHIR-55241 owns 2D). The PR-HEAD copy of the codesystem
shows the new concept added with a slightly tightened definition
(`used only when none of the other values apply.`) versus the verbatim
resolution wording.

The companion ValueSet
(`source/imagingselection/valueset-imagingselection-3dgraphictype.xml`)
includes the entire CodeSystem (`<include><system value=".../imagingselection-3dgraphictype"/>`)
with no concept enumeration, so it does **not** need editing — the new
concept flows through automatically.

## Feature Proposal

### Problem Statement

`ImagingSelection.imageRegion3D.regionType` is defined as a 1..1 element
with a required binding to
`http://hl7.org/fhir/ValueSet/imagingselection-3dgraphictype`. The current
value set enumerates only specific DICOM 3D graphic types
(`point`, `multipoint`, `polyline`, `polygon`, `ellipse`, `ellipsoid`).

In real-world DICOM data, the source Graphic Type
(DICOM PS3.3 C.18.9.1.2) may carry a value that does not map to any of the
listed concepts, or be effectively missing/unknown when the imaging
selection still needs to be expressed in FHIR. Because the element is
mandatory and the binding is required, implementers have **no compliant
way** to populate the element in those cases — they must either fabricate
a value or skip the resource entirely. The submitter requests an explicit
escape-valve concept (`unknown`, `other`, or equivalent) so the resource
remains usable end-to-end.

### Proposed Change

Add a single concept, `unknown`, to the FHIR Core `imagingselection-3dgraphictype`
CodeSystem and update the ImagingSelection StructureDefinition's element
short description to surface the new value:

1. **CodeSystem `imagingselection-3dgraphictype`**
   - **System URL:** `http://hl7.org/fhir/imagingselection-3dgraphictype`
   - **New `<concept>`:**
     - `code` = `unknown`
     - `display` = `UNKNOWN`
     - `definition` = `used only when none of the other values apply.`
       (tightened phrasing from the resolution table; matches the wording
       already merged in PR HL7/fhir#4043 for the 2D mirror and adopted
       there as the canonical form)
   - Append after the existing `ellipsoid` concept; keep `caseSensitive`
     `true` and `content` `complete` unchanged.
2. **StructureDefinition `ImagingSelection`**, element
   `ImagingSelection.imageRegion3D.regionType` (in the `<differential>`):
   - Change `<short>` from
     `point | multipoint | polyline | polygon | ellipse | ellipsoid`
     to
     `point | multipoint | polyline | polygon | ellipse | ellipsoid | unknown`.
   - Leave `<min>`, `<max>`, `<type>`, `<binding>`, `<mapping>`,
     `<comment>` (DICOM PS3.3 C.18.9.1.2 reference), and `<alias>`
     (`GraphicType`) unchanged.
3. **ValueSet `imagingselection-3dgraphictype`** — no edit needed (it
   includes the whole CodeSystem by reference).
4. **Resource ballot note** — confirm
   `source/imagingselection/imagingselection-introduction.xml`'s
   "Note to Balloters" already lists "Added `unknown` value to
   `imageRegion2D.regionType` and `imageRegion3D.regionType` code systems."
   The PR-HEAD copy already does; if applying this ticket independently of
   PR HL7/fhir#4043, the bullet must be added.

The 2D mirror (FHIR-55241) makes the parallel edits in
`codesystem-imagingselection-2dgraphictype.xml` and the
`ImagingSelection.instance.imageRegion2D.regionType` element. The two
tickets are intended to land together, and PR HL7/fhir#4043 already does
exactly that.

### DICOM Cross-Reference Note

The user request asks to "fix any DICOM cross-reference comments on
`ImagingSelection.instance.imageRegion3D.regionType`". The current and
PR-HEAD value of the element's `<comment>` is
`See [DICOM PS3.3 C.18.9.1.2](http://dicom.nema.org/medical/dicom/current/output/chtml/part03/sect_C.18.9.html#sect_C.18.9.1.2).`
PR HL7/fhir#4043 left this comment untouched (only the `<short>` was
extended). The DICOM section is the correct one for 3D Graphic Type, so
**no DICOM-comment fix is required for 3D**. Tracked here as an explicit
"checked, no change needed" outcome, distinct from the 2D mirror where
the corresponding comment was reviewed under FHIR-55241.

### Design Rationale

- **Match the existing pattern.** Several other FHIR Core required
  bindings already include an `unknown` escape concept (e.g.,
  `administrative-gender`, observation/medication status code systems).
  Reusing the `unknown` code (rather than `other`) keeps terminology
  vocabulary consistent within FHIR Core.
- **Minimum-disruption change.** Adding a concept to an existing
  experimental-false but normative-flagged CodeSystem is preferred over
  changing the binding strength or making the element optional, both of
  which would be much larger compatibility events.
- **Symmetry with 2D.** Resolving 2D (FHIR-55241) and 3D (FHIR-55242)
  with identical mechanics avoids creating an asymmetric API where one
  region type has an escape valve and the other does not.
- **Definition wording.** The PR adopted
  `used only when none of the other values apply.` — a tightened version
  of the resolution table's `Unknown is used only when none of the other
  values apply`. This matches the phrasing chosen for the 2D mirror and
  reads cleanly when prefixed by the display label.
- **No alternative seriously considered.** `other` was mentioned as an
  acceptable equivalent in the ticket, but `unknown` is preferred because
  it more directly encodes the actual semantic ("the source data did not
  give us a usable graphic type") and aligns with neighbouring FHIR
  vocabularies.

### Breaking Changes

No breaking changes identified. The change is purely additive:

- A new code is being **added** to a required-bound CodeSystem. Existing

### Testing Considerations

- **Resource revalidation:** Re-run the FHIR publisher's resource
  validation for ImagingSelection. The CodeSystem expansion and
  ValueSet `$expand` should now include the seventh `unknown` code.
- **Example coverage:** Existing examples already use concrete graphic
  types (`point`, `ellipsoid`, etc.) and continue to be valid. Optional:
  add a small example or test fixture that uses `regionType = unknown`
  to confirm round-trip parsing and validation, but the resolution does
  not mandate a new example.
- **Edge cases:**
  - A consumer system that hard-codes the prior six-value enumeration
    (e.g., a switch statement) will not crash — `unknown` is a new
    valid code, not a breaking change. Implementations should be
    advised (via the ballot note) to handle the new code defensively.
  - Validators pinned to CodeSystem `version=6.0.0` will reject
    `unknown` until they pick up the new version. Bump `<version>` if
    the publisher's release process does not auto-rev it.
- **Interoperability:** Confirm that DICOM-to-FHIR mapping guidance
  (where present in the `dicom` mappings section of the SD) does not
  imply a 1:1 obligation to populate from a DICOM Graphic Type — the
  new `unknown` concept is intentionally outside the DICOM enumeration.
