## Resolution Summary

Resolution: **Persuasive**. Add a new element `ImagingStudy.series.instance.numberOfFrames` to ImagingStudy in FHIR Core. The exact differential proposed in the resolution is:

```xml
<element id="ImagingStudy.series.instance.numberOfFrames">
  <path value="ImagingStudy.series.instance.numberOfFrames"/>
  <short value="The number of this instance in the series"/>
  <definition value="The number frames in a multiframe image."/>
  <alias value="NumberOfFrames"/>
  <min value="0"/>
  <max value="1"/>
  <type>
    <code value="unsignedInt"/>
  </type>
  <example>
    <label value="General"/>
    <valueUnsignedInt value="1"/>
  </example>
  <mapping>
    <identity value="dicom"/>

  </mapping>
```

## Feature Proposal

### Problem Statement

DICOM instances may be multiframe — a single SOP Instance can contain many image frames (e.g., enhanced MR/CT, US multiframe, NM, XA). The DICOM attribute *Number of Frames* `(0028,0008)` carries that count. The current `ImagingStudy.series.instance` backbone exposes `uid`, `sopClass`, `number`, and `title`, but has no way to express how many frames a referenced instance contains. Consumers reading an ImagingStudy therefore cannot tell that an instance is multiframe, or how large it is, without retrieving the underlying DICOM object — which is the gap DICOM CP 2595 addresses.

### Proposed Change

Add one new element to ImagingStudy:

- **Path:** `ImagingStudy.series.instance.numberOfFrames`
- **Short:** `The number of this instance in the series`
- **Definition:** `The number frames in a multiframe image.`
- **Alias:** `NumberOfFrames`
- **Cardinality:** `0..1`
- **Type:** `unsignedInt`
- **Example:** label `General`, value `1`
- **DICOM mapping:** `Number Of Frames (0028,0008)`
- **isSummary:** not set in the resolution (omit, matching siblings such as `instance.title`)

This is purely additive; no existing element changes cardinality, type, or binding. No new invariant or search parameter is required by the resolution.

### Design Rationale

- Mirrors the DICOM data element directly (PS3.3 Image Pixel module), as proposed by DICOM CP 2595, so producers can populate it from the DICOM source without transformation.
- `unsignedInt 0..1` is consistent with the sibling `ImagingStudy.series.instance.number` (also `unsignedInt 0..1`) and with how other "count" elements in ImagingStudy are typed (`series.numberOfInstances`, `series.numberOfSeries`).
- Optional cardinality keeps the change non-breaking: legacy producers that omit frame counts remain conformant; consumers who care about multiframe behaviour can rely on the element when present.
- An explicit DICOM `mapping` keeps round-tripping unambiguous and matches the pattern used by every other `ImagingStudy.series.instance.*` element.
