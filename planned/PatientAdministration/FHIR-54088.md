## Resolution Summary

The ticket is a Technical Correction with resolution **Persuasive** and resolution
description: *"will be fixed as suggested"*. The original request is verbatim:

> Please correct encounter.partof to Encounter.partOf

In other words, a casing typo in the rendered text of a link on the Patient resource's
notes page must be corrected to the canonical FHIR path casing
(`Encounter.partOf`). The link target (`href`) already uses the correct casing; only
the visible link text is wrong.

## Feature Proposal

### Problem Statement

The Patient resource notes page (`source/patient/patient-notes.xml`) contains a
bullet under the "boundaries / relationships" prose that links to the Encounter
resource's `partOf` element. The visible anchor text is the lowercase
`encounter.partof`, which is not a valid FHIR path. FHIR paths are
PascalCase-by-resource and camelCase-by-element, so the correct rendering is
`Encounter.partOf`. The href fragment (`encounter-definitions.html#Encounter.partOf`)
already uses the canonical casing, so the discrepancy is purely in displayed text and
gives readers an incorrect path to copy. There are no other instances of the
lowercase form in `source/` (a case-insensitive grep across `source/` returns only
this one occurrence as malformed; all other matches are already `Encounter.partOf`).

### Proposed Change

In `source/patient/patient-notes.xml`, line 160, change the visible link text
`encounter.partof` to `Encounter.partOf`. The surrounding `<a href="...">…</a>`
markup, the href value, and all neighbouring prose remain untouched.

Before:

```xml
<li>To relate the encounters of a mother and her baby in a maternity encounter,
for administrative and billing purposes: use the <a href="encounter-definitions.html#Encounter.partOf">encounter.partof</a> property</li>
```

After:

```xml
<li>To relate the encounters of a mother and her baby in a maternity encounter,
for administrative and billing purposes: use the <a href="encounter-definitions.html#Encounter.partOf">Encounter.partOf</a> property</li>
```

### Design Rationale

- FHIR uses PascalCase resource names and camelCase element names; `Encounter.partOf`
  is the established canonical form and is what every other reference in the spec
  uses (verified by grep across `source/`).
- The fix is purely cosmetic in the rendered HTML — it does not alter the link
  target, IDs, anchors, or any conformance artefact — so it is the minimal change
  that satisfies the resolution "will be fixed as suggested".
- No alternative was discussed in the ticket; the reporter's wording is itself the
  agreed solution and the resolver accepted it verbatim.
