## Resolution Summary

**Resolution:** Persuasive — *"Update as described"*.

The committee accepted the reporter's recommendation verbatim: change the FHIRPath
expression of the `RelatedPerson` `role` SearchParameter from
`RelatedPerson.relationship` to `RelatedPerson.role` so the expression matches the
search parameter's stated semantics (functional role, not personal relationship).

## Feature Proposal

### Problem Statement

`RelatedPerson` defines two distinct elements:

- `RelatedPerson.relationship` — the *personal* relationship (e.g., mother,
  spouse, friend), backed by the `v3-PersonalRelationshipRoleType` value set.
- `RelatedPerson.role` — the *functional* role (e.g., emergency contact,
  guardian, power of attorney).

The element comments in the StructureDefinition explicitly cross-reference one
another to make this distinction clear (verified in
`source/relatedperson/structuredefinition-RelatedPerson.xml` lines 184–221).

The `role` SearchParameter advertises that it searches the functional role:

> `<description value="The functional relationship between the patient and the relatedperson"/>`
> `<code value="role"/>`

…and its `path` build-extension is correctly set to `RelatedPerson.role`. However,
the actual `<expression>` is `RelatedPerson.relationship` — almost certainly a
copy-paste from the immediately preceding `relationship` SearchParameter entry.
As a result, queries like `GET RelatedPerson?role=…` silently return matches
based on the *personal* relationship code instead of the functional role,
producing semantically wrong results without any error or warning.

### Proposed Change

Single source-file edit in `HL7/fhir`:

- **File:** `source/relatedperson/bundle-RelatedPerson-search-params.xml`
- **Entry:** the `SearchParameter` with `<id value="RelatedPerson-role"/>` (lines 349–363)
- **Change line 361** from:

  ```xml
  <expression value="RelatedPerson.relationship"/>
  ```

  to:

  ```xml
  <expression value="RelatedPerson.role"/>
  ```

No other fields in the entry change — `id`, `url`, `description`, `code`, `type`
(`token`), `processingMode`, the `standards-status` extension (`normative`), and
the `path` build-extension (already `RelatedPerson.role`) are all already
internally consistent with the corrected expression.

The sibling `RelatedPerson-relationship` SearchParameter entry (lines 332–345),
which legitimately uses `<expression value="RelatedPerson.relationship"/>`, is
left untouched.

### Design Rationale

- The fix matches the unambiguous resolution disposition (`Persuasive` —
  *"Update as described"*).
- The corrected expression aligns the SearchParameter with both its own
  `description` and its existing `path` build-extension.
- It preserves the established 1:1 mapping between the resource's two distinct
  elements and their respective token search parameters (`relationship` →
  `RelatedPerson.relationship`, `role` → `RelatedPerson.role`), which is the
  pattern already used everywhere else in the bundle.
- The element types are compatible: both `relationship` and `role` are
  `CodeableConcept` (token), so the SearchParameter `type` (`token`) does not
  need to change.
- No alternative was discussed in the ticket; the bug is a clear copy-paste
  error and the recommendation was accepted verbatim.

  - `source/relatedperson/bundle-RelatedPerson-search-params.xml` — change the

- **Change Scope:** Minor (single attribute value).

### Breaking Changes

This is a behavioural correction to a published SearchParameter. Whether it is
"breaking" depends on perspective:

- **Specification-correctness perspective:** Not breaking. The current
  expression contradicts the SearchParameter's `description` and `path`
  extension; any conformant server today is implementing a buggy parameter.
- **Wire-compatibility perspective:** *Potentially behaviour-changing* for
  servers that have implemented the published expression literally. After the
  fix, `GET RelatedPerson?role={code}` will resolve against
  `RelatedPerson.role` (functional role codes) instead of
  `RelatedPerson.relationship` (personal-relationship codes). Clients that
  were (incorrectly) relying on `role=` to filter by personal relationship
  should switch to `relationship=`, which has always been the semantically
  correct parameter for that use.

There are no cardinality, type, or terminology-binding changes, and no element
is added or removed. No StructureDefinition or invariant is affected.

### Related Specifications

- `RelatedPerson` StructureDefinition
  (`source/relatedperson/structuredefinition-RelatedPerson.xml`) — the source
  of truth for the `RelatedPerson.role` and `RelatedPerson.relationship`
  elements. **No edit required**, but the corrected SearchParameter must
  remain consistent with the element comments at lines 188 and 221, which
  already cross-reference each other (no change needed).
- The sibling `RelatedPerson-relationship` SearchParameter (same bundle,
  lines 332–345) — left unchanged; it correctly targets
  `RelatedPerson.relationship`.
- Downstream IGs (Incubator / IG repos) that profile `RelatedPerson` and
  reference the core `role` SearchParameter inherit the fix automatically and

re-points the existing token search to the correct element.

### Testing Considerations

- **Search-parameter resolution:** Re-run any conformance / search-parameter
  validation that the publisher performs against the updated bundle. Confirm
  the FHIRPath expression `RelatedPerson.role` resolves cleanly against the
  current `RelatedPerson` StructureDefinition (it should — the element exists
  with type `CodeableConcept`, matching the `token` search type).
- **Server behaviour:** Servers should re-index `RelatedPerson` instances
  against the corrected expression. A minimal test-matrix is:
  - Create a `RelatedPerson` with `relationship` = personal code (e.g.,
    `MTH`) and `role` = functional code (e.g., `emergency contact`).
  - `GET RelatedPerson?role={functional code}` must now return that
    instance; previously it would have returned only instances whose
    `relationship` matched the supplied code.
  - `GET RelatedPerson?relationship={personal code}` must continue to
    behave as before (unchanged SearchParameter).
- **Edge cases:** Instances with a `role` value but no `relationship` (and
  vice versa) — both should now be searchable by their respective parameters
  with no overlap.
- **Interoperability:** Implementers who relied on the buggy behaviour need
  to migrate their queries; flag this in the resource's change-history /
  publication notes if such notes are maintained for SearchParameter
  corrections.
