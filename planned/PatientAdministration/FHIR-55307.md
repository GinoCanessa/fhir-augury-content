## Feature Proposal

### Problem Statement

The Person resource specification page contains the contraction `doesn't` in prose. HL7 FHIR editorial convention (as reflected in this tracker's disposition) is that specification narrative should spell abbreviations/contractions out. A single occurrence was reported on the Person page (imported comment 271, reporter Lloyd McKenzie). Verified in the clone: `source/person/person-notes.xml` line 75 contains `it doesn’t disclose where the information is being used` (curly-apostrophe form), inside the `[%stu-note system%] … [%end-note%]` usage note for the "System / Master Index" style of Person usage. No other `doesn` / `don't` / `won't` / `can't`-style contractions were found on other Person authoring files in the clone.

### Proposed Change

In `source/person/person-notes.xml`, within the `[%stu-note system%]` note block (around line 75), replace the contraction `doesn’t` with the fully spelled-out form `does not`. After the edit, the sentence reads:

> This style of usage has fewer security implications, as it does not disclose where the information is being used.

No other element, structure, cardinality, binding, or example changes are required. This is a pure narrative/prose fix. No StructureDefinition, ValueSet, CodeSystem, SearchParameter, operation, or invariant is touched.

### Design Rationale

- The briefing for `HL7/fhir` identifies `source/<resource>/*.xml` and `source/*.html` as the authored content that the bespoke HL7 publisher consumes directly; `person-notes.xml` is exactly that — hand-authored narrative included into the Person page. It is the correct file to edit for a prose fix on the Person page.
- The change is the minimal surgical edit that matches the ticket's stated specifics ("Specifics: \"doesn't\""). Broader copy-editing (hunting for other contractions across Person or other resource pages) is deliberately not bundled in, because (a) the ticket only reports this one token, and (b) a repo-wide contractions sweep is gated on the open FMG policy discussion referenced in egelkraut's comment — doing that sweep here would pre-empt that decision.
- Using "does not" (not "does-not", not removing the clause) preserves meaning and tone while satisfying the "fully spelled out" requirement in the ticket.
