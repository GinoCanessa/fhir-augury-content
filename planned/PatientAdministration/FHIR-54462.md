## Resolution Summary

**Resolution:** Persuasive — *"Will update to be a modifer to be consistent with the other resources."*

The committee accepted the proposal: `PractitionerRole.active` will be flagged
as a modifier element (`isModifier = true`) so that it behaves consistently
with the equivalent `active` element on `Patient` and `Practitioner`. No
other behavioural or structural changes were called out in the resolution.

## Feature Proposal

### Problem Statement

`PractitionerRole.active` (boolean, 0..1) currently signals whether a
PractitionerRole record is in active use, but it is **not** declared as a
modifier element in the StructureDefinition. By contrast, the analogous
`active` elements on the closely related resources `Patient` and
`Practitioner` are both flagged `isModifier = true` (see
`source/patient/structuredefinition-Patient.xml` line 161 ff. and
`source/practitioner/structuredefinition-Practitioner.xml` line 145 ff.,
both of which carry `<isModifier value="true"/>` and an `isModifierReason`
of "This element is labelled as a modifier because it is a status element
that can indicate that a record should not be treated as valid").

The reporter's argument — that an inactive PractitionerRole fundamentally
changes how a consuming system should interpret references to it — is
exactly the modifier-element semantic, and the inconsistency with the
sibling resources is unwarranted. The Patient Administration work group
agreed and resolved the ticket as Persuasive.

### Proposed Change

Make `PractitionerRole.active` a modifier element by adding the standard
`isModifier` / `isModifierReason` pair to its element definition in the
PractitionerRole StructureDefinition differential, mirroring the wording
already used on `Patient.active` and `Practitioner.active`.

Concrete edit to `source/practitionerrole/structuredefinition-PractitionerRole.xml`,
inside the `<element id="PractitionerRole.active">` block (currently
starting at line 149 of the differential), insert two new lines between
the existing `<meaningWhenMissing>` line and the existing `<isSummary>`
line so the block reads:

```xml
<element id="PractitionerRole.active">
  <path value="PractitionerRole.active"/>
  <short value="Whether this practitioner role record is in active use"/>
  <definition value="Whether this practitioner role record is in active use. Some systems may use this property to mark non-active practitioners, such as those that are not currently employed."/>
  <comment value="If this value is false, you may refer to the period to see when the role was in active use. If there is no period specified, no inference can be made about when it was active."/>
  <requirements value="Need to be able to mark a practitioner role record as not to be used because it was created in error, or otherwise no longer in active use."/>
  <min value="0"/>
  <max value="1"/>
  <type>
    <code value="boolean"/>
  </type>
  <meaningWhenMissing value="This resource is generally assumed to be active if no value is provided for the active element"/>
  <isModifier value="true"/>
  <isModifierReason value="This element is labelled as a modifier because it is a status element that can indicate that a record should not be treated as valid"/>
  <isSummary value="true"/>
  …
</element>
```

No change to cardinality (`0..1`), type (`boolean`), `short`, `definition`,
`comment`, `requirements`, `meaningWhenMissing`, `isSummary`, or any of
the existing mappings (`w5`, `v2`, `rim`, `interface`). No change to
search parameters, examples, narrative, or terminology bindings.

### Design Rationale

- **Consistency with sibling resources.** `Patient.active` and
  `Practitioner.active` are already modifier elements with an identical
  `isModifierReason` ("This element is labelled as a modifier because it
  is a status element that can indicate that a record should not be
  treated as valid"). Reusing that exact wording keeps PractitionerRole
  aligned with the established pattern for status-style booleans on
  administrative resources.
- **Semantic correctness.** When `active = false`, downstream systems
  must treat references to the PractitionerRole differently (e.g., not
  schedule against it, not surface in pick lists, not honour its
  privileges). This is precisely the contract that `isModifier = true`
  exists to communicate to validators and implementers.
- **Minimal blast radius.** The edit is a two-line addition inside a
  single `<element>` block in one StructureDefinition; no other
  artefacts are mechanically dependent on the modifier flag (search
  parameters, mappings, and examples remain valid as-is).
- **Alternatives considered (implicit).** The alternative of leaving
  PractitionerRole inconsistent with Patient/Practitioner was rejected
  by the work group; no other alternative was discussed in the ticket.

## Impact Analysis

### Affected Repositories

  no cardinality, type, binding, or search-parameter change).

### Breaking Changes

**Yes — this is a backwards-incompatible change for conformance.**

Promoting an existing element from non-modifier to `isModifier = true`
changes the conformance contract: implementers consuming PractitionerRole
instances are now obliged to understand the `active` element. Specifically:

- The FHIR rule "applications must understand all modifier elements"
  (or reject the resource) means existing R4/R4B/R5 PractitionerRole
  consumers that silently ignored `active` would, under the new
  publication, be non-conformant if they continue to do so.
- No on-the-wire instance becomes invalid (the element's name, type,
  cardinality, and value space are unchanged), so persisted instances
  do not need migration.
- Profiles and IGs that derive from PractitionerRole inherit the new
  modifier flag automatically and may need to revisit their narrative
  if they previously documented `active` as non-modifying.

The ticket's resolution wording ("update to be a modifer to be consistent
with the other resources") explicitly accepts this consistency-driven
breaking change.
