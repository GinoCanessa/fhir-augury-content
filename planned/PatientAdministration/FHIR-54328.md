## Resolution Summary

**Resolution:** Not Persuasive with Modification.

The committee split the request into two parts:

1. **Adding invariants that enforce "when `mode = kind`, the instance-specific
   elements must be empty" тАФ declined.** The Location Mode narrative itself
   carves out an exception ("_however this information could still be included
   if was relevant, such as when it is a generic item, but not globally generic,
   e.g. a Burgers Medical Centre ambulance_"). Because the listed elements are
   still permitted when `mode = 'kind'` in edge cases, a FHIRPath invariant is
   not possible. No StructureDefinition change is made for the invariant part.

2. **Expanding the list of elements typically scoped to `mode = 'instance'` тАФ
   accepted.** The following three elements are added to the bulleted list in
   the Location Mode notes:
   - `Location.hoursOfOperation`
   - `Location.virtualService`
   - `Location.endpoint`

3. **Softening the lead-in sentence тАФ accepted.** The sentence introducing the
   list is re-worded so that it is guidance rather than a prohibition,
   reconciling it with the inline "_however this information could still be
   includedтАж_" clarification that follows:

   - **From:** _"Some of Location's data elements are only relevant when mode
     is 'instance' and should not be used when mode is 'kind':"_
   - **To:** _"Some of Location's data elements are **typically** only relevant
     when mode is 'instance' and **would** not be used when mode is 'kind':"_
     (the word `should` is struck; `typically` and `would` are added тАФ emphasis
     in the rendered HTML uses `<b>`/`<del>` per the resolution text).

## Feature Proposal

### Problem Statement

The Location resource's "Location Mode" narrative declares that a set of
elements (`identifier`, `telecom`, `address`, `position`, `status`,
`managingOrganization`) are "only relevant when mode is 'instance'", but

1. The list is incomplete тАФ `hoursOfOperation`, `virtualService`, and
   `endpoint` are also predominantly instance-scoped but are omitted from the
   list, so implementers working from the notes page miss them.
2. The phrasing is stronger than the spec actually intends. The paragraph
   immediately qualifies the rule with an "_however this information could
   still be includedтАж_" example, which makes the "only relevant" / "should not
   be used" wording self-contradictory. The reporter observed this tension and
   originally asked for a hard FHIRPath invariant.

### Proposed Change

All changes occur on a single authored HTML fragment in the Location resource
folder of `HL7/fhir`. No StructureDefinition, ValueSet, search parameter, or
example changes are required.

**File:** `source/location/location-notes.xml`, inside the existing
`<a name="loc-mode"></a> <h3>Location Mode</h3>` section.

1. **Soften the lead-in sentence** (currently line 34). Replace:

   ```html
   Some of Location's data elements are only relevant when mode is 'instance' and should not be used when mode is 'kind':<br/>
   ```

   with:

   ```html
   Some of Location's data elements are <b>typically</b> only relevant when mode is 'instance' and <del>should</del> <b>would</b> not be used when mode is 'kind':<br/>
   ```

   (The resolution specifies `<b>` for the added words and `<del>` for the
   struck `should`. Keep the trailing `<br/>` and the following `<i>` caveat
   paragraph intact.)

2. **Extend the bulleted list** (currently lines 38тАУ45) from six items to
   nine by appending, in the order given by the resolution:

   ```html
   <li>Location.hoursOfOperation</li>
   <li>Location.virtualService</li>
   <li>Location.endpoint</li>
   ```

   immediately after the existing `<li>Location.managingOrganization</li>`
   bullet and before the closing `</ul>`.

3. **Do not add any invariant** to `source/location/structuredefinition-Location.xml`.
   The resolution explicitly declined the invariant because the narrative
   permits these elements under `mode = 'kind'` in the "generic but not
   globally generic" case. Any `<constraint>` added to enforce emptiness
   would contradict that carve-out and must not be introduced.

### Design Rationale

- **Why reject the invariant?** The resolution quotes the existing in-page
  exception ("_a Burgers Medical Centre ambulance_") and notes that because
  the information can still be legitimately present on a `kind`-mode Location,
  an invariant would reject valid instances. The committee therefore preferred
  a documentation-only softening rather than a normative constraint.
- **Why expand the list?** `Location.hoursOfOperation` is a schedule of a
  specific physical or logical location; `Location.virtualService` carries
  concrete virtual-service endpoints (conference URLs, dial-ins); and
  `Location.endpoint` references Endpoint resources pointing at a specific
  running system. All three share the same "instance-scoped by default"
  property as `telecom`/`address` and were simply missed when the original list
  was authored.
- **Why the wording change?** "`typically` only relevant тАж `would` not be
  used" aligns with the qualifying "_however this information could still be
  includedтАж_" sentence that already follows the list, removing the internal
  contradiction without losing the guidance's practical force.
