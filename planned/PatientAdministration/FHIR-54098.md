## Resolution Summary

The ticket flags a single-word typo in the Encounter resource specification page,
`encounter.html` § 8.11.1.1 ("Admission data"). The sentence currently reads:

> "Local systems may have multiple different types of leave/hold and these can use
> appropriate combinations **fo** the status/subjectStatus fields to represent this."

The resolution (`Persuasive`) accepts the reporter's correction: change `fo` →
`for`. The resolution description states: "The error occurs in
https://build.fhir.org/encounter.html#8.11.1.1 — will be fixed accordingly." No
other content change is required.

## Feature Proposal

### Problem Statement

The narrative under § 8.11.1.1 of the Encounter specification page contains a
typographical error ("`fo`" where "`for`" was clearly intended). The sentence is
otherwise grammatical, and the surrounding paragraph discusses how local systems
combine `Encounter.status` and `Encounter.subjectStatus` to represent
on-leave / on-hold scenarios. The typo makes one sentence misparse and looks
unprofessional in the published specification, but it does not change normative
meaning. There are no associated cross-references (Jira, Zulip, GitHub PRs,
commits) for this ticket.

### Proposed Change

Edit the Encounter introduction page in the HL7/fhir repo to correct the typo:

- File: `source/encounter/encounter-introduction.xml`
- Located at line 87 (verified in the cached clone @ `db79dcd`).
- Current text:
  ```
  …types of leave/hold and these can use appropriate combinations fo the status/subjectStatus fields to represent this.
  ```
- Corrected text:
  ```
  …types of leave/hold and these can use appropriate combinations for the status/subjectStatus fields to represent this.
  ```

That is the entire change: a single character insertion (`fo` → `for`). No other
files, examples, StructureDefinitions, ValueSets, or pages need to be touched.

### Design Rationale

This is a pure typographical correction; there is no design choice involved. The
fix is made directly in the authored XML page rather than the rendered
`encounter.html` because the HL7 publisher generates the HTML from the
`source/encounter/encounter-introduction.xml` fragment (per the briefing's
"Authoring root(s)" / "Generated areas" rules — `source/` is the only authoring
root, and HTML pages under `source/` are read directly). The corrected sentence
remains semantically identical to the author's intent and matches the surrounding
prose style (lowercase, inline `for`).
