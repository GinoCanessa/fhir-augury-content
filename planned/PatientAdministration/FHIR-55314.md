## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"We will add 'unknown' to the
codesystem/valueset to support the escape valve."*

Concretely, the Patient Administration work group approved adding an
`unknown` concept to the `SlotStatus` CodeSystem so that `Slot.status`
(a 1..1 `code` with a required binding to
`http://hl7.org/fhir/ValueSet/slotstatus`) can be populated in cases
where none of the existing concepts (`busy`, `free`, `busy-unavailable`,
`busy-tentative`, `entered-in-error`) actually fit. Because the value
set is defined as `<compose><include><system
value="http://hl7.org/fhir/slotstatus"/></include></compose>` with
`immutable=true`, adding the code to the CodeSystem automatically
expands the ValueSet — no separate ValueSet edit is required.

## Feature Proposal

### Problem Statement

`Slot.status` is a mandatory (`1..1`) `code` element with a required
binding to the `SlotStatus` value set. Today the value set only offers
functional states (`busy`, `free`, two busy variants) plus
`entered-in-error`. When a system must expose a Slot but genuinely does
not know its free/busy state — e.g., slots imported from an upstream
schedule that withheld status, an integration in a degraded mode, or
data received via a bridge that does not carry the concept — there is
no legal value to assert. Given the required binding and the 1..1
cardinality, this leaves the Slot instance un-authorable: implementers
must either refuse the instance or mis-classify it (commonly as `busy`
or `free`), both of which are worse than being explicit about ignorance.

The ticket reporter (Lloyd McKenzie, imported comment 278) asks for an
"escape valve" concept — `unknown`, `other`, or equivalent — to be
added to the value set so that required-binding conformance can be met
without fabricating a functional meaning.

### Proposed Change

Add a single concept `unknown` to the `SlotStatus` CodeSystem. The
value set will pick it up automatically via its existing
`include system=http://hl7.org/fhir/slotstatus`. The `Slot`
StructureDefinition's short/definition enumeration for `Slot.status`
must be updated to include the new code. The CodeSystem is normative
(`content=complete`, `standards-status=normative`, `fmm=3`) so the
concept's `definition` must be crisp enough to support long-term
conformance.

Concrete changes:

1. **CodeSystem `slotstatus` — add concept**
   (`source/slot/codesystem-slotstatus.xml`)

   ```xml
   <concept>
     <code value="unknown"/>
     <display value="Unknown"/>
     <definition value="The status of the slot is not known. This code
       is intended to be used when no other code in this value set is
       appropriate (for example, a slot imported from a source system
       that does not carry status information)."/>
   </concept>
   ```

   Place the new `<concept>` after `entered-in-error`, preserving the
   existing order so the semantically "unusual" codes sit at the end.

2. **StructureDefinition `Slot` — refresh the status summary strings**
   (`source/slot/structuredefinition-Slot.xml`, `element id="Slot.status"`,
   lines 209–229)

   - `<short value="busy | free | busy-unavailable | busy-tentative | entered-in-error"/>`
     → `"busy | free | busy-unavailable | busy-tentative | entered-in-error | unknown"`
   - `<definition value="busy | free | busy-unavailable | busy-tentative | entered-in-error."/>`
     → `"busy | free | busy-unavailable | busy-tentative | entered-in-error | unknown."`

   The `<binding>` block is unchanged — still required, still points at
   `http://hl7.org/fhir/ValueSet/slotstatus`.

3. **ValueSet `slotstatus` — no content change required**
   (`source/slot/valueset-slotstatus.xml`)

   Confirmed by inspection: the `<compose>` clause is a bare
   `<include><system value="http://hl7.org/fhir/slotstatus"/></include>`
   with no `filter`/`concept` enumeration, so the addition propagates
   automatically. The version on the CodeSystem and ValueSet may be
   bumped by the publisher as part of the release; no manual change is
   needed here.

4. **Optional: add an example illustrating the new state**
   `source/slot/slot-example-unknown.xml` plus an entry in
   `list-Slot-examples.xml`. The briefing's Authoring Root rules
   support adding new per-resource authoring files in
   `source/<resource>/`. Treat this as nice-to-have; the resolution
   does not mandate it.

Out of scope (explicitly): changing the cardinality of `Slot.status`,
changing the binding strength, or changing the committee-notes
statement about why `Slot.status` is not a modifier element.

### Design Rationale

- **`unknown` over `other`.** The reporter offered three options
  (`unknown`, `other`, "or some equivalent"). `unknown` is the
  conventional FHIR status escape-valve code (appears in many other
  required status value sets) and communicates "I cannot assert any of
  the real values" rather than "I have a value, but it is not in your
  list" — the latter is what `other` typically implies and is rarely
  what scheduling integrations actually want to say.
- **CodeSystem add, not ValueSet add.** The `slotstatus` ValueSet is
  `immutable=true` and composes the entire CodeSystem unchanged.
  Adding to the CodeSystem is therefore both necessary (so the code
  has a formal definition) and sufficient (the ValueSet picks it up
  automatically). Adding a second `<include>` to the ValueSet would
  be redundant and would fight the `immutable` flag.
- **Keep the element required and 1..1.** The escape valve replaces
  the need to loosen cardinality or make the binding example/preferred
  — both were raised as alternatives in the general FHIR pattern but
  are more disruptive than an added code. The resolution explicitly
  picks the latter route.
- **Leave `entered-in-error` in place.** `entered-in-error` remains
  semantically distinct ("this record should never have existed");
  `unknown` is needed even when the record is valid.

### Breaking Changes

Adding a new code to a **required-binding value set** on a **normative**
resource is the sensitive case here. Honest assessment:

- **Existing instances:** no change — the resolution only adds a code;
  none are renamed, removed, or re-coded. Persisted Slot resources
  remain valid.
- **Producers:** backward-compatible — they may continue to produce
  any of the five original codes.
- **Consumers / validators:** must now accept `unknown` as a legal
  value. Validators that hard-coded the five-value enumeration (rather
  than consulting the value set) will reject instances using the new
  code. Per FHIR's own rules, adding a concept to a required-binding
  value set is a substantive change and requires work-group sign-off
  — which Patient Administration has given (resolution = Persuasive).

Net: **not a wire-format breaking change, but a substantive normative
terminology change.** Downstream IGs that re-slice `Slot.status` with
their own narrower bindings are unaffected because their slice takes
precedence; IGs that simply re-use the base binding inherit the new
code.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add the `unknown` concept to the `SlotStatus` CodeSystem.**
   - File: `source/slot/codesystem-slotstatus.xml`
   - Action: Insert a new `<concept>` element as the last concept,
     immediately after the existing `entered-in-error` concept (which
     currently ends on or around line 73 of the file).
   - Details: Use

     ```xml
     <concept>
       <code value="unknown"/>
       <display value="Unknown"/>
       <definition value="The status of the slot is not known. This
         code is intended to be used when no other code in this value
         set is appropriate (for example, a slot imported from a source
         system that does not carry status information)."/>
     </concept>
     ```

     Preserve two-space indentation and leave the existing
     `caseSensitive`, `valueSet`, and `content` elements untouched.
     Do **not** bump `<version>` by hand; the publisher controls
     CodeSystem versions.

2. **Update the `Slot.status` element's short and definition strings.**
   - File: `source/slot/structuredefinition-Slot.xml`
   - Action: In the `<element id="Slot.status">` block (lines
     209–229 in clone `db79dcd`), change:
     - `<short value="busy | free | busy-unavailable | busy-tentative | entered-in-error"/>`
       → `<short value="busy | free | busy-unavailable | busy-tentative | entered-in-error | unknown"/>`
     - `<definition value="busy | free | busy-unavailable | busy-tentative | entered-in-error."/>`
       → `<definition value="busy | free | busy-unavailable | busy-tentative | entered-in-error | unknown."/>`
   - Details: Only the `<short>` and `<definition>` strings change.
     Do not touch `<min>`, `<max>`, `<type>`, `<isSummary>`, the
     committee-notes extension, the `entered-in-error-status`
     extension, the `<binding>` block, or the `<mapping>` block. The
     StructureDefinition's `<snapshot>` (if authored) is
     publisher-regenerated per the briefing's gotcha — leave any
     snapshot block untouched; the publisher will sync it.

3. **Verify the `SlotStatus` ValueSet needs no change.**
   - File: `source/slot/valueset-slotstatus.xml`
   - Action: Read-only check. Confirm the `<compose>` is still
     `<include><system value="http://hl7.org/fhir/slotstatus"/></include>`
     (no `concept` or `filter` sub-elements). No edit expected.
   - Details: If a future PR ever adds an enumerated `<concept>`
     list to this ValueSet's `<include>`, the new code would have
     to be added there too. Leave a note in the commit message so
     reviewers know this was intentionally not edited.

4. *(Optional)* **Add an example slot in `unknown` state.**
   - File: `source/slot/slot-example-unknown.xml` (new) +
     `source/slot/list-Slot-examples.xml` (entry added)
   - Action: Author a minimal Slot example that copies
     `slot-example-busy.xml`'s shape, sets `status=unknown`, and
     gives it a stable `id` (e.g., `example-unknown`). Register it
     in `list-Slot-examples.xml` alongside the existing
     `example`, `example-busy`, `example-hcs`, `example-tentative`,
     `example-unavailable` entries.
   - Details: Not required by the resolution; include only if the
     PR author wants to exercise the new code through the build.
     Skip if this PR is intentionally scoped to the terminology
     change alone.
