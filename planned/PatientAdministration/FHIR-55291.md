resource page that should be expanded per HL7 publishing style ("Abbreviations
need to be fully spelled out"). The resolution accepts the request, so the
contraction must be replaced with the spelled-out form ("does not"). No

## Feature Proposal

### Problem Statement

The Encounter resource introduction on the published `encounter.html`
page contains the contraction "doesn't" in the paragraph describing
admission semantics for outpatient visits. HL7 specification authoring
style (and the request captured in this technical correction) calls for
contractions/abbreviations to be expanded so the prose is unambiguous and
friendly to non-native English readers and downstream translation.

### Proposed Change

In `source/encounter/encounter-introduction.xml` (the authored content
that the bespoke HL7 publisher renders into `encounter.html`), expand
the single user-visible contraction:

- **File:** `source/encounter/encounter-introduction.xml`
- **Line ~90 (current text):**
  > "…hence the encounter doesn't have a fixed definition of admitted."
- **Replacement:**
  > "…hence the encounter does not have a fixed definition of admitted."

No surrounding prose, structure, anchors, or examples need to change.

The other clone hit for the substring `doesn` —
`source/encounter/structuredefinition-Encounter.xml:1105`, inside an
`http://hl7.org/fhir/build/StructureDefinition/committee-notes`
extension (`"This concept doesn’t exist in v3…"`) — is **not** part of
the published encounter page (it is an internal committee note, and uses
a curly apostrophe `’` rather than the straight `'` quoted in the
ticket). It is out of scope for this ticket; do not edit it as part of
this change.

### Design Rationale

- The ticket text explicitly names the offending token (`"doesn't"`) and
  the page (`encounter`); a literal expansion to "does not" is the
  minimal, faithful application of the resolution.
- "does not" is the standard expansion already used elsewhere in the FHIR
  core narrative (consistency with existing prose).
- Editing the source authoring file (`encounter-introduction.xml`) — not
  any generated HTML — matches the FhirCore briefing's authoring rules
  (the bespoke HL7 publisher renders these XML narrative fragments into
  the final `encounter.html`).
- The committee-note extension hit is intentionally left untouched: it is
  authoring metadata, not user-facing page content, so it falls outside
  the "page: encounter" scope identified in the ticket.
