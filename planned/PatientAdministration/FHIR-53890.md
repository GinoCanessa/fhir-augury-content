## Resolution Summary

Resolution type: **Persuasive**.

Resolution description (verbatim, paraphrased from Jira):

> The text on the "Standard Extesion: Associated Encounter" will be removed as noted.
> We will also follow up with the spec tooling to see if the link to the deprecated extension can be suppressed from being displayed.

Concretely, this ticket requires a single narrative edit to the Encounter resource's *Boundaries and Relationships* (B&R) page: remove the paragraph that points readers at the now-deprecated `Associated Encounter` standard extension. A secondary, non-normative follow-up is to check with the publisher/tooling team whether links to deprecated extensions can be suppressed automatically; that follow-up is **not** in scope for the content edit in this plan.

## Feature Proposal

### Problem Statement

The B&R section of the Encounter resource currently instructs implementers to use the `Associated Encounter` standard extension "to reference an encounter where there is no property that already defines this association on the resource." That guidance is obsolete: the `Associated Encounter` extension has been deprecated in favour of the native `Encounter.partOf` element (and, more broadly, the encounter-context pattern used by individual resources). Leaving the paragraph in place actively steers new implementers toward a deprecated artifact, contradicting the rest of the Encounter specification (see `encounter-introduction.xml` lines 24, 37 and `encounter-notes.xml` lines 14–27, which already direct readers to `Encounter.partOf`).

The Zulip thread [#committers > deprecated extensions](https://chat.fhir.org/#narrow/channel/179165-committers/topic/deprecated.20extensions/with/570618698) that triggered the ticket also raises a broader tooling concern — deprecated extensions should not silently appear under a resource's "references to resource" section without any visual indication — but the resolution confines the content change to removing the stale paragraph.

### Proposed Change

**Single content change, in `source/encounter/encounter-introduction.xml`:**

Delete the entire `<p>` block at lines 120–123 in the current clone (HEAD `db79dcd`):

```xml
<p>
Standard Extension: <b>Associated Encounter</b><br/>
This extension should be used to reference an encounter where there is no property that already defines this association on the resource.
</p>
```

No replacement prose is required. The surrounding paragraphs (Appointment relationship, Communication relationship) already stand on their own. The `<a name="bnr"></a>` / `<h2>Boundaries and Relationships</h2>` heading and the closing `</div>` must remain untouched so the `#bnr` anchor used by other spec pages keeps resolving.

No other occurrences of `Associated Encounter` or `associatedEncounter` exist under `source/encounter/` (verified in the clone), so this is the only narrative change.

**Out-of-scope follow-up (tracked here, not implemented):** the resolution commits the WG to asking the publisher/tooling team whether links to deprecated extensions can be suppressed in the auto-generated "references to resource" block. That is a publisher/tooling change (outside `source/<resource>/`), and should be handled separately — likely via a new ticket or a Zulip thread with the tooling team. See *Open Questions* below.

### Design Rationale

- **Consistency:** The rest of the Encounter page (introduction and notes) already routes readers to `Encounter.partOf` for encounter-to-encounter relationships. Removing the stale paragraph aligns B&R with that guidance.
- **Minimal narrative surface:** Replacing the paragraph with "this extension is deprecated" was considered but rejected — the WG's stated resolution is to remove the statement, and the publisher already renders deprecation metadata at the extension level. Duplicating that in prose on Encounter would re-introduce the same maintenance drift that caused this ticket.
- **Anchor preservation:** Deleting only the paragraph (not the `<a name="bnr"></a>` / `<h2>` / surrounding `<div>`) avoids breaking inbound links of the form `encounter.html#bnr` used elsewhere in the spec.
- **Tooling follow-up kept separate:** Suppressing auto-generated links to deprecated extensions is a generic publisher concern that will recur for every resource with a deprecated extension; bundling it into this resource-level content edit would understate its scope.
