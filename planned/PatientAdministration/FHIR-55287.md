## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** "We will add the 'unknown' to the appointment status code system (and thus valueset)."

The change is to add a new `unknown` concept to the `AppointmentStatus` CodeSystem (`http://hl7.org/fhir/appointmentstatus`). Because the bound ValueSet (`http://hl7.org/fhir/ValueSet/appointmentstatus`) is `immutable=true` with a single `include` of the entire CodeSystem, the new code is automatically picked up by the ValueSet and by every place that bindings reference it (notably `Appointment.status`, which is a mandatory element with a required binding to this ValueSet).

## Feature Proposal

### Problem Statement

`Appointment.status` is `1..1` with a `required` binding to `ValueSet/appointmentstatus`. The current code system enumerates only operationally meaningful states (`proposed`, `pending`, `booked`, `arrived`, `fulfilled`, `cancelled`, `noshow`, `entered-in-error`, `checked-in`, `waitlist`) and provides no "unknown" or "other" escape valve. As reported by Lloyd McKenzie (imported as Comment 251), this can make `Appointment` impossible to populate in legitimate cases — e.g., when an Appointment record is being exchanged but the originating system did not record (or cannot reliably infer) the appointment's lifecycle state. Because the binding is `required` on a mandatory element, there is no other conformant way to satisfy the cardinality without misrepresenting the appointment state.

### Proposed Change

Add a single new concept to the `AppointmentStatus` CodeSystem.

- **CodeSystem:** `http://hl7.org/fhir/appointmentstatus` (id `appointmentstatus`).
- **New concept:**
  - `code`: `unknown`
  - `display`: `Unknown`
  - `definition`: A short statement that the appointment's status is not known. Suggested wording (consistent with similar "unknown" codes elsewhere in the spec): *"The current status of the appointment is not known. Note that 'unknown' is not the same as 'pending' or 'proposed'; 'unknown' is the status when the system providing the appointment information does not know what state the appointment is currently in."*
- **ValueSet:** No structural change required. `ValueSet/appointmentstatus` (`immutable=true`) composes by `include` of the whole `http://hl7.org/fhir/appointmentstatus` system, so the new code flows through automatically.
- **Appointment StructureDefinition:** Update the `Appointment.status` element's `<short>` enumeration (currently `proposed | pending | booked | arrived | fulfilled | cancelled | noshow | entered-in-error | checked-in | waitlist`) to append `| unknown` so that the human-readable summary continues to reflect the bound ValueSet. The element's `definition`, cardinality (`1..1`), type (`code`), `binding.strength` (`required`), and `binding.valueSet` URL all stay the same.
- **Examples (optional but recommended):** Update or add an example instance that demonstrates `Appointment.status = unknown` so QA validates the new code end-to-end.

### Design Rationale

- Adding `unknown` is the same "escape valve" pattern used by many other required-bound status code systems in FHIR (e.g., the various resource `*-status` value sets that include `unknown`). It is the minimal, least-disruptive way to remove the conformance trap reported by Lloyd McKenzie.
- The Patient Administration work group's resolution explicitly chose `unknown` (rather than `other`), aligning with prevailing FHIR practice. `other` would imply a known-but-unlisted state, which is not the gap; the gap is "the source system cannot tell us the state".
- The change is additive at the CodeSystem level. Because the ValueSet is `immutable` and includes the whole system, no ValueSet edit is needed and no `compose.exclude` filtering is required to suppress the new code anywhere it is currently bound.
- Keeping the `binding.strength` at `required` preserves interoperability; the new code is still part of the same closed list, simply with one more valid value.

### Breaking Changes

No breaking changes identified for the conformance contract:

- The change is purely additive at the CodeSystem level.
- `Appointment.status` cardinality (`1..1`), type (`code`), and `binding.strength` (`required`) are unchanged.
- No existing code is removed, deprecated, or repurposed.

Note for implementers: code that exhaustively switches over `AppointmentStatus` values (e.g., generated enums in client SDKs, server-side state machines) will see a new enum value and may need a default/fallback branch added. This is the standard expected impact of adding a code to a required-bound CodeSystem and is not a spec-level breaking change.

Out of scope for this ticket; flagged here only to prevent confusion with the similarly-named `participationstatus` and `participantrequired` code systems sitting alongside in `source/appointment/`.
- IG profiles that further constrain `Appointment.status` (e.g., US Core Appointment, if profiled) may need to consider whether to permit `unknown`. That assessment is the responsibility of the downstream IGs and is not part of this core change.

## Implementation Plan

### Step-by-Step Tasks

#### HL7/fhir

1. **Add the `unknown` concept to the AppointmentStatus CodeSystem.**
   - File: `source/appointment/codesystem-appointmentstatus.xml`
   - Action: Append a new `<concept>` block after the existing `waitlist` concept (the last one in the file).
   - Details: Use:

     ```xml
     <concept>
       <code value="unknown"/>
       <display value="Unknown"/>
       <definition value="The current status of the appointment is not known. Note that &quot;unknown&quot; is not the same as &quot;pending&quot; or &quot;proposed&quot;; &quot;unknown&quot; is the status when the system providing the appointment information does not know what state the appointment is currently in."/>
     </concept>
     ```

     Bump `<version>` and `<date>` per the publisher's conventions for normative artifact edits.

2. **(Verify only) Confirm the AppointmentStatus ValueSet still composes the full CodeSystem.**
   - File: `source/appointment/valueset-appointmentstatus.xml`
   - Action: No structural edit. Confirm that `<compose><include><system value="http://hl7.org/fhir/appointmentstatus"/></include></compose>` is unchanged and that `<immutable value="true"/>` remains.
   - Details: If the publisher requires CodeSystem and ValueSet to share a `<version>`/`<date>` for normative content, mirror the bumped values from Task 1 here.

3. **Update the `<short>` enumeration on `Appointment.status`.**
   - File: `source/appointment/structuredefinition-Appointment.xml`
   - Action: In the `<differential>`, locate the `Appointment.status` element (line 184 in the current clone). Change the `<short>` value (line 186) from
     `proposed | pending | booked | arrived | fulfilled | cancelled | noshow | entered-in-error | checked-in | waitlist`
     to
     `proposed | pending | booked | arrived | fulfilled | cancelled | noshow | entered-in-error | checked-in | waitlist | unknown`.
   - Details: Do **not** edit the `<snapshot>` block — it is regenerated by the publisher (per the FhirCore briefing's "Generated areas (do not edit)"). Leave `cardinality` (`1..1`), `type=code`, `binding.strength=required`, and `binding.valueSet=http://hl7.org/fhir/ValueSet/appointmentstatus` untouched.

4. **(Optional but recommended) Add an example demonstrating `status="unknown"`.**
   - File: `source/appointment/appointment-example-unknown-status.xml` (new) — or update an existing example such as `appointment-example2doctors.xml`.
   - Action: Author a minimal valid `Appointment` instance whose `status` element is `<status value="unknown"/>`, with an `extension` or narrative note explaining the scenario (e.g., a record imported from a system that did not preserve the lifecycle state).
   - Details: If a new file is added, register it in `source/appointment/list-Appointment-examples.xml` so the publisher picks it up. Keep the file alongside the existing examples for consistency with the per-resource folder convention.

5. **Run the publisher locally to regenerate snapshots and QA outputs.**

Confirm the `Appointment` resource page, the `appointmentstatus` CodeSystem page, and the `appointmentstatus` ValueSet expansion all render with the new `unknown` code, and that QA reports no new errors against the touched files.
