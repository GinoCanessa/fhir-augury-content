## Resolution Summary

**Resolution:** Persuasive with Modification.

The resolution requires two coordinated changes to the `Endpoint` resource:

1. **Expand the example ValueSet `endpoint-connection-type`** to additionally
   include the HL7 v3 NullFlavor concept **`UNK` (unknown)** from
   `http://terminology.hl7.org/CodeSystem/v3-NullFlavor`
   (per https://terminology.hl7.org/1.0.0/CodeSystem-v3-NullFlavor.html). This
   gives implementers a standardized "escape" code when no published
   connection-type code is appropriate, instead of inventing local "other"
   codes.
2. **Update the `Endpoint.connectionType` element comment** in the
   StructureDefinition to add guidance that when the connection type is
   unknown (or not yet known), a clarifying note can be supplied via
   `connectionType.text`.

The original 2026-03-03 disposition (4-0-0) included both `OTH/other` and
`UNK/unknown`. That disposition was reverted by an 8-0-0 motion on
2026-03-31 (Brian Postlethwaite / Reinhard Egelkraut), and the final
re-resolved disposition lists only `UNK/unknown` plus the comment-text
update. The plan below implements the **final** disposition.

The ticket also records a forward-looking note (not a code change): "Having
multiple 'other' codes for the connection types in different implementation
guides is fine, as that 'other' code would be semantically distinct in the
context of each IG."

## Feature Proposal

### Problem Statement

`Endpoint.connectionType` is mandatory (cardinality `1..*`) and bound (with
example strength) to the `endpoint-connection-type` ValueSet, which composes
codes only from `http://terminology.hl7.org/CodeSystem/endpoint-connection-type`.
Because the element is required, an implementer with an endpoint whose
underlying protocol/profile is not represented in that example value set has
no standardized way to populate the element. The reporter (John Moehrke)
notes that the value set should provide a code that can be used "when there
is no appropriate code available", and the WG agreed (Sonja Ziegler:
"â€¦inclusion of a standardized null flavor or 'escape mechanism' would
preclude everyone creating their own flavor of such").

In addition, the existing element comment provides only XDS-style protocol
guidance and gives implementers no instruction on how to add free-text
clarification when the chosen code is `UNK` (or otherwise insufficient).

### Proposed Change

**Change A â€” ValueSet `endpoint-connection-type`**
(`source/endpoint/valueset-endpoint-connection-type.xml`):

Add a second `<include>` block to the `<compose>` so the value set composes
codes from **both** of:

- `http://terminology.hl7.org/CodeSystem/endpoint-connection-type`
  (existing; whole code system)
- `http://terminology.hl7.org/CodeSystem/v3-NullFlavor` filtered to the
  single concept `UNK`

Concretely, append to the existing `<compose>`:

```xml
<include>
  <system value="http://terminology.hl7.org/CodeSystem/v3-NullFlavor"/>
  <concept>
    <code value="UNK"/>
    <display value="unknown"/>
  </concept>
</include>
```

Binding strength remains `example`. ValueSet `version`, `status`,
`experimental`, identifiers, `name`, `title`, and the workgroup/standards
extensions are unchanged. The narrative `<description>` may optionally be
extended to mention that the v3 NullFlavor `UNK` is provided as an escape
code, but that is editorial â€” the normative change is the `compose.include`
addition.

**Change B â€” StructureDefinition `Endpoint`**
(`source/endpoint/structuredefinition-Endpoint.xml`, element
`Endpoint.connectionType`, currently the `<comment>` at line 138):

Replace the existing `<comment value="â€¦">`:

> *For additional connectivity details for the protocol, extensions will be
> used at this point, as in the XDS example. If there are multiple payload
> types or mimetypes they are all applicable for all connection types, and
> all have the same status.*

with the updated text from the resolution:

> *For additional connectivity details for the protocol, extensions will be
> used at this point, as in the XDS example. If there are multiple payload
> types or mimetypes they are all applicable for all connection types, and
> all have the same status. If a connection type is not known, or not known
> **yet**, then a comment may be added to the connectionType.text.*

Implementation note: in the StructureDefinition XML the `<comment>` value is
plain text (no inline HTML), so the `<b>yet</b>` emphasis from the Jira
description should appear as plain `yet` in the XML attribute. The bolding
is preserved when other authoring formats (e.g., the spec's published
narrative) re-render the text.

No change is needed to:
- `min`/`max` (still `1..*`),
- `type` (still `CodeableConcept`),
- the `binding` block (still example, still bound to
  `http://hl7.org/fhir/ValueSet/endpoint-connection-type`),
- the `<short>` or `<definition>`.

### Design Rationale

- **Reuses an existing, governed code system.** The HL7 v3 NullFlavor code
  system already publishes `UNK` for "unknown" with well-understood
  semantics across HL7 product families. Reusing it is preferable to
  minting a new local "unknown" code in the
  `endpoint-connection-type` CodeSystem, which would diverge from
  established null-flavor handling.
- **Preserves the example binding.** The binding remains `example`, so
  this change does not increase conformance burden â€” implementers were
  already free to use codes outside the ValueSet â€” but it gives a
  recommended landing place for "no code applies", which the WG felt
  prevents implementer-by-implementer divergence (Sonja Ziegler comment).
- **Final disposition narrowed to `UNK` only.** The earlier
  4-0-0 disposition added both `OTH` and `UNK`; the 8-0-0 reopen and
  re-resolution kept only `UNK`. The narrower set avoids the apparent
  redundancy with the per-IG "other" pattern that the resolution explicitly
  endorses ("multiple 'other' codes â€¦ in different IGs â€¦ is fine").
- **`connectionType.text` as the explanatory channel.** Because
  `Endpoint.connectionType` is a `CodeableConcept`, `text` is already the
  conformant place to carry free-text clarification when no Coding is
  appropriate (or when the code is `UNK`). The comment update simply makes
  this guidance explicit on the element where it is most likely to be read.

### Open Questions

- **CodeSystem version pinning.** The resolution links to the v1.0.0
  THO publication of `v3-NullFlavor`. The `<include>` proposed above
  does not pin a CodeSystem version (matching the existing pattern in
  `endpoint-connection-type`'s ValueSet, which also does not pin
  `endpoint-connection-type` versions). Confirm this is the intended
  approach before merging; if pinning is desired, add a
  `<version value="â€¦"/>` inside the new `<include>`.
- **Optional editorial expansion.** The resolution does not mandate
  editing the ValueSet's `<description>` or the resource introduction.
  Decide whether to add a short note in `<description>` (and possibly in
  `source/endpoint/endpoint-notes.xml`) pointing implementers at
  `UNK` + `connectionType.text` as the recommended pattern; this is
  editorial, not normative, but improves discoverability.
- **Example demonstrating the new pattern.** Consider whether to add a
  small example endpoint instance that uses
  `connectionType.coding = v3-NullFlavor#UNK` plus a populated
  `connectionType.text`. Not required by the resolution, but it would
  document the intended usage on the rendered Examples tab.
