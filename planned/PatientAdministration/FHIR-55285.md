## Resolution Summary

Resolution: **Not Persuasive with Modification**.

Verbatim resolution description:

> The data types for both `Appointment.recurrenceTemplate.monthlyTemplate.nthweekofMonth` and `.dayOfWeek` should be changed to `code` from `Coding`. This is in alignment with other bindings across resources.
>
> These attributes are optional to allow for more flexibility.

Concretely, the resolution requires two narrow StructureDefinition edits in
the Appointment resource:

1. Change `Appointment.recurrenceTemplate.monthlyTemplate.nthWeekOfMonth.type` from
   `Coding` to `code` (binding to `http://hl7.org/fhir/ValueSet/week-of-month`,
   strength `required`, retained).
2. Change `Appointment.recurrenceTemplate.monthlyTemplate.dayOfWeek.type` from
   `Coding` to `code` (binding to `http://hl7.org/fhir/ValueSet/days-of-week`,
   strength `required`, retained).

Both elements remain optional (`min=0`, `max=1` — already the case in the
current source). The original ticket also raised a separate question about
migrating the `week-of-month` CodeSystem/ValueSet to THO; the approved
resolution explicitly does **not** include the THO migration — the agreed
change is the type narrowing only.

## Feature Proposal

### Problem Statement

In the current FHIR Core source, `Appointment.recurrenceTemplate.monthlyTemplate.nthWeekOfMonth`
and `Appointment.recurrenceTemplate.monthlyTemplate.dayOfWeek` are typed as
`Coding` with required bindings to `ValueSet/week-of-month` and
`ValueSet/days-of-week` respectively. Using `Coding` for an element with a
`required` binding to a small, closed code list is inconsistent with the
established FHIR convention (e.g., `Appointment.status`, `*.dayOfWeek` in
`Timing`, weekday usages elsewhere) where the chosen type is `code`. The
extra structural overhead of `Coding` (`system`, `version`, `display`,
`userSelected`) provides no value when the binding is required to a single
closed system, and can produce ambiguity in instances (e.g., omitted
`system` or display strings that disagree with the canonical display).

The ticket also flagged the related concern that `CodeSystem/ValueSet
week-of-month` is a candidate for migration to THO so that other product
families (CDA, V2) could share the same value set. The PA work group
declined to bundle that migration into this change ("Not Persuasive with
Modification") and scoped the resolution down to the type fix only.

### Proposed Change

Edit the differential of the Appointment StructureDefinition so that both
elements use a primitive `code` type bound to the same value sets they are
bound to today.

Target file: `source/appointment/structuredefinition-Appointment.xml`.

For element `Appointment.recurrenceTemplate.monthlyTemplate.nthWeekOfMonth`
(currently lines 1309–1326 in the cached clone @ `db79dcd`):

- Keep `id`, `path`, `short`, `definition`, `min` (`0`), `max` (`1`).
- Change the `<type><code value="Coding"/></type>` to `<type><code value="code"/></type>`.
- Keep the existing `<binding>` block unchanged:
  - `bindingName` extension: `WeekOfMonth`
  - `strength`: `required`
  - `description`: `The set of weeks in a month.`
  - `valueSet`: `http://hl7.org/fhir/ValueSet/week-of-month`

For element `Appointment.recurrenceTemplate.monthlyTemplate.dayOfWeek`
(currently lines 1327–1345):

- Keep `id`, `path`, `short`, `definition`, `comment`, `min` (`0`), `max` (`1`).
- Change the `<type><code value="Coding"/></type>` to `<type><code value="code"/></type>`.
- Keep the existing `<binding>` block unchanged:
  - `bindingName` extension: `DaysOfWeek`
  - `strength`: `required`
  - `description`: `The days of the week.`
  - `valueSet`: `http://hl7.org/fhir/ValueSet/days-of-week`

After the edits, the relevant differential element fragment for `nthWeekOfMonth`
should look like:

```xml
<element id="Appointment.recurrenceTemplate.monthlyTemplate.nthWeekOfMonth">
  <path value="Appointment.recurrenceTemplate.monthlyTemplate.nthWeekOfMonth"/>
  <short value="Indicates which week of the month the appointment should occur"/>
  <definition value="Indicates which week within a month the appointments in the series of recurring appointments should occur on."/>
  <min value="0"/>
  <max value="1"/>
  <type>
    <code value="code"/>
  </type>
  <binding>
    <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
      <valueString value="WeekOfMonth"/>
    </extension>
    <strength value="required"/>
    <description value="The set of weeks in a month."/>
    <valueSet value="http://hl7.org/fhir/ValueSet/week-of-month"/>
  </binding>
</element>
```

(`dayOfWeek` is the parallel change with its own binding metadata
preserved.)

No other source artifacts in `source/appointment/` reference these
elements:

- A search of `source/appointment/*.xml` for `nthWeekOfMonth` and
  `monthlyTemplate.dayOfWeek` returns matches only inside
  `structuredefinition-Appointment.xml`.
- `bundle-Appointment-search-params.xml` defines no search parameters on
  these elements.
- The `appointment-introduction.xml` and `appointment-notes.xml`
  narrative pages do not mention either element.
- The repo-wide search for `ValueSet/week-of-month` returns matches only
  in `codesystem-week-of-month.xml`, `valueset-week-of-month.xml`, and
  the Appointment SD itself (line 1324) — confirming the value set is
  used only by this one element and no cross-resource reference is
  affected.

The companion `valueset-week-of-month.xml` and
`codesystem-week-of-month.xml` need no changes: the value set already
contains the codes that will continue to be the bound `required` value
set for the new `code`-typed element.

### Design Rationale

The PA work group's resolution explicitly states the change is "in
alignment with other bindings across resources." For an element with a
`required` binding to a small closed code system, `code` is the
established FHIR pattern (e.g., `Timing.repeat.dayOfWeek` is `code` bound
to the same `days-of-week` ValueSet, and `Appointment.status` /
`Appointment.recurrenceTemplate.recurrenceType` use `code` with required
bindings). Keeping `Coding` here would introduce inconsistency, would
allow instances that omit `system`/`code` or supply mismatched
`display` text, and would force implementers to write extra glue around a
binding that is already pinned to a single `required` value set.

Two alternatives were considered in the ticket discussion and rejected:

- **Migrate `week-of-month` CodeSystem/ValueSet to THO** so the codes can
  be reused by V2/CDA. PA marked this part as Not Persuasive — there is no
  current cross-spec consumer demand strong enough to justify the
  migration overhead, and migration can be revisited later without
  affecting the type narrowing now.
- **Make the elements `1..1` required** while changing the type. The
  resolution explicitly preserves `0..1` ("These attributes are optional
  to allow for more flexibility"), since not every monthly-recurrence
  template needs both axes.

### Breaking Changes

This change **is breaking** for instance producers and consumers, even
though it does not change cardinality:

- The wire format for both elements changes from a `Coding` JSON/XML
  object (with `system`, `code`, `display`, etc.) to a primitive `code`
  string. Any existing instance that populated either element will fail
  validation against the new StructureDefinition.
- Consumers that traverse `Appointment.recurrenceTemplate.monthlyTemplate.nthWeekOfMonth.code`
  or `.system` will need to read the primitive value directly instead.
- FHIRPath / search-parameter expressions that descend into `.coding` or
  `.code` of these elements must be rewritten to address the primitive.
- The required binding to `ValueSet/week-of-month` and
  `ValueSet/days-of-week` is preserved, so the **set of allowed values**
  does not change; only the **type/shape** of the carrier changes.

The cardinality (`0..1`) is unchanged, so this is not a "newly required
element" break, but it is still a backwards-incompatible structural
change that should be called out in the resource's changelog/version
notes when the build is published.

### Testing Considerations

- **Revalidate** Appointment after the change: regenerate the
  `<snapshot>` via the HL7 publisher (`publish.sh` / `build.sh`) and
  confirm both element snapshots show `type.code = code` and the
  `required` bindings are preserved.
- **Examples:** none of the existing `appointment-example*.xml` files
  populate `recurrenceTemplate.monthlyTemplate.nthWeekOfMonth` or
  `monthlyTemplate.dayOfWeek` (verified by `Select-String` over
  `source/appointment/*.xml`). Consider adding or extending an example
  that exercises a monthly-by-week pattern (e.g., "second Tuesday of
  every month") so downstream toolers have a sample of the new
  primitive shape, e.g.:

  ```xml
  <recurrenceTemplate>
    <recurrenceType>
      <coding>
        <system value="http://terminology.hl7.org/CodeSystem/appointment-recurrence-type"/>
        <code value="monthly"/>
      </coding>
    </recurrenceType>
    <monthlyTemplate>
      <nthWeekOfMonth value="second"/>
      <dayOfWeek value="tue"/>
      <monthInterval value="1"/>
    </monthlyTemplate>
  </recurrenceTemplate>
  ```

- **Edge cases to verify:**
  - Instances that previously sent the elements as `Coding` objects must
    now fail validation (regression confirmation that the type really
    changed).
  - Instances using only `nthWeekOfMonth` without `dayOfWeek` (and vice
    versa) still validate, since both are `0..1`.
  - All codes from the bound value sets continue to be accepted as
    primitive `code` values.
- **Interoperability:** call this change out in the Appointment resource
  changelog / R6 ballot notes — it is a wire-format break (see Breaking
  Changes) even though the value space is unchanged.
