## Feature Proposal

### Problem Statement

`InsuranceProduct` was split out of `InsurancePlan` (see `fhir.ini` migration-note entries at lines 1835–1867 and 2336–2368 of `source/fhir.ini`, and the two early split commits `b9ef5a8515` / `3df002d6fc`). When Reinhard Egelkraut tried to run the HL7 publisher against the FHIR Core spec in April 2024 the resource produced "a couple of errors" during the build process (two attached screenshots, `24797_FHIR_InsuranceProduct_Build-Errors-1.png` and `24796_FHIR_InsuranceProduct_Build-Errors-2.png`). The errors are not reproduced in the ticket text, so the authoritative problem statement is simply that the then-current `source/insuranceproduct/` content did not publish cleanly, and Patient Administration accepted the technical correction to make it do so.

### Proposed Change

Perform a bounded technical-edit pass over the `source/insuranceproduct/` authoring tree in `HL7/fhir` (master branch) so that the publisher runs without emitting errors or new warnings for `InsuranceProduct`. Concretely:

1. **Re-run the publisher locally** against current `source/insuranceproduct/` and capture the current error/warning list. This replaces the two unreadable screenshots as the live work list.
2. **Fix any hand-authored XML issues the publisher reports** in the following in-scope files (all confirmed present in the clone at `db79dcd`):
   - `source/insuranceproduct/structuredefinition-InsuranceProduct.xml` — only the `<differential>` block is hand-edited per the FhirCore playbook; `<snapshot>` is generator output and must not be touched. Typical cleanup targets: element definition text that wraps across lines and accidentally introduces spurious whitespace / line breaks inside attribute values, missing or stale `mapping` identities, `condition`/`constraint` keys that do not resolve, and any `binding` that points at a ValueSet no longer present.
   - `source/insuranceproduct/bundle-InsuranceProduct-search-params.xml` — verify every `SearchParameter` has `url`, `code`, `type`, `expression`, `processingMode`, and that `expression` values parse against the current StructureDefinition paths.
   - `source/insuranceproduct/insuranceproduct-example.xml`, `insuranceproduct-example1.xml`, `insuranceproduct-example-onc.xml` — ensure each example validates against the StructureDefinition (correct element ordering, valid `id`/`identifier` datatypes, presence of a `<text>` narrative where required, references resolvable within the bundle of examples).
   - `source/insuranceproduct/insuranceproduct-introduction.xml`, `insuranceproduct-notes.xml`, `insuranceproduct-examples-header.xml` — valid XHTML, balanced tags, all `href` / `src` links reachable. The March 2025 commit `e27dd9ba45 "Fix: invalid URL link in InsuranceProduct"` already patched one such link in the introduction; a re-read is warranted to catch any remaining dangling links.
   - `source/insuranceproduct/insuranceproduct-fivews-mapping-exceptions.xml` — confirm the list of `unmappedElement` entries still matches the FiveWs pattern and `InsuranceProduct` differential. The upstream "get build completing" commit `21109129b7` rewrote the mapping-exception files across every resource; any regression needs to be reapplied here.
   - `source/insuranceproduct/list-InsuranceProduct-examples.xml`, `list-InsuranceProduct-operations.xml`, `list-InsuranceProduct-packs.xml` — ensure each listed entry exists and renders.
   - `source/insuranceproduct/valueset-insuranceproduct-applicability.xml`, `valueset-insuranceproduct-type.xml`, `valueset-publication-status.xml` — confirm `url`, `version`, `status`, `compose`, and any `expansion` content validates; terminology service resolves all referenced code systems.
   - `source/insuranceproduct/invariant-tests/` — keep existing invariant test fixtures consistent with the `ipn-1` constraint declared in the StructureDefinition (`(identifier.count() + name.count()) > 0`).
3. **Align registration metadata**: confirm the existing rows in `source/fhir.ini` are still valid — `insuranceproduct=InsuranceProduct` (line 287), work-group `pa` (613), security-category `business` (757), FMM `1` (897), standards-status `normative` (line 1042). No new entries are required by this ticket.
4. **Do not extend scope**: no new elements, no cardinality changes, no new bindings. Labels `AdditionalResources, TechnicalEdit` and resolution type `Persuasive` (not `Persuasive with Modifier`) confine this to a build-hygiene fix.

No example instances beyond the three already in the repo are proposed.

### Design Rationale

- A `Technical Correction` / `TechnicalEdit` ticket with a one-line resolution directs the implementer to make the publisher pass cleanly, not to redesign the resource. The `ipn-1` invariant, element set, normative status, and FMM level are unchanged from when the ticket was opened.
- Grounding the fix list in a fresh publisher run (rather than in the original 2024 screenshots) is the only reliable way to know which errors still exist at clone `db79dcd`; between April 2024 and the current HEAD several commits have already touched `source/insuranceproduct/*` (`21109129b7 get build completing`, `e27dd9ba45 Fix: invalid URL link`, `9987f9c0c3 Remove FormularyItem, and update status codes`, `e3c3d0fe20` / `5c20769cc7` "fix status codes", `c72229a425` / `02f79b0a3b` ballot-note rounds). Some of the originally-reported errors are likely already gone.
- Only `<differential>` content in `StructureDefinition` files is hand-edited per the FhirCore playbook (see Warnings in the Repo Context below). Confining edits to the differential preserves the publisher's ability to regenerate `<snapshot>` without drift.
- Alternatives considered:
  - *Close as Duplicate* of a more specific build-error ticket — rejected: the ticket was resolved `Persuasive`, not `Duplicate`.
  - *Expand scope to re-scrub InsuranceProduct content* — rejected: the labels are `TechnicalEdit`, not `SubstantiveChange`.
