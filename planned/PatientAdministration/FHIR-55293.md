## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** "We will add the code **`unknown`** as an
escape as requested."

The ticket reports that `Endpoint.status` is a 1..1 element with a **required**
binding to `ValueSet/endpoint-status`, yet the value set has no "unknown" /
"other" escape concept. In scenarios where the actual operational status of an
endpoint is not known to the sender, the resource becomes impossible to
populate. The approved fix is to add a single new concept, `unknown`, to the
backing CodeSystem so it flows into the required value set.

## Feature Proposal

### Problem Statement

`Endpoint.status` is mandatory (cardinality `1..1`) and bound with strength
**required** to `http://hl7.org/fhir/ValueSet/endpoint-status`
(StructureDefinition lines 104–124 of
`source/endpoint/structuredefinition-Endpoint.xml`). The CodeSystem
`http://hl7.org/fhir/endpoint-status`
(`source/endpoint/codesystem-endpoint-status.xml`) currently defines only
six operational codes: `active`, `limited`, `suspended`, `error`, `off`,
`entered-in-error`. None of these accommodates a sender that is required to
declare an Endpoint but does not actually know its current operational
status. Because the binding is `required`, an implementer has no conformant
escape valve and must invent one via extension or omit/fabricate a value.

### Proposed Change

Add one new concept to CodeSystem `http://hl7.org/fhir/endpoint-status`:

- **code:** `unknown`
- **display:** `Unknown`
- **definition:** `The state of the endpoint is not known.` (suggested; PA
  to finalise wording at publication. The definition must convey that this
  is an explicit escape value for use when the actual status cannot be
  determined, not a distinct operational state.)

No change is required to the ValueSet `endpoint-status` itself: it composes
the entire CodeSystem via a single `<include><system .../></include>`
(no `concept` filter), so the new code flows through automatically.

Downstream updates required by the new code:

1. The element `<short>` on `Endpoint.status` (line 106 of
   `structuredefinition-Endpoint.xml`) currently reads
   `active | limited | suspended | error | off | entered-in-error` and must
   be extended to include `unknown`.
2. The `<definition>` on `Endpoint.status` (line 107) should be reviewed so
   it remains accurate when `unknown` is a legal value (current text: "The
   endpoint status represents whether the endpoint can currently be used
   for connections or why it can't be used.").

No new elements, extensions, search parameters, invariants, or cardinality
changes are required.

### Design Rationale

- The reporter asked for `unknown`, `other`, or "some equivalent escape
  valve". Patient Administration chose **only `unknown`**. This is
  consistent with other FHIR core status value sets that add `unknown` as
  a deliberate escape concept rather than `other` (which is typically used
  for modeling gaps, not unknown operational state).
- Adding the code to the CodeSystem (rather than altering the binding
  strength to `extensible`/`preferred`, or removing the `required`
  constraint) preserves interoperability: every conformant consumer will
  still validate against the same closed list, just with one additional
  legal value.
- Because the ValueSet includes the CodeSystem wholesale, the change is
  localised to a single file plus a `<short>` text touch-up in the
  StructureDefinition, minimising blast radius.

## Implementation Plan

### Prerequisites

- None. The change is self-contained in `HL7/fhir` and does not depend on
  any other ticket, extension registration, or UTG submission.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add `unknown` concept to the Endpoint status CodeSystem**
   - File: `source/endpoint/codesystem-endpoint-status.xml`
   - Action: Append a new `<concept>` element after the existing
     `entered-in-error` concept (and before the closing `</CodeSystem>`
     tag).
   - Details: Use the element shape already in the file:
     ```xml
     <concept>
       <code value="unknown"/>
       <display value="Unknown"/>
       <definition value="The state of the endpoint is not known."/>
     </concept>
     ```
     Confirm final wording of `<definition>` with PA before publication;
     the definition must make clear this is an explicit escape value
     used when the operational status cannot be determined, not a
     distinct operational state.

2. **Update the `Endpoint.status` short text**
   - File: `source/endpoint/structuredefinition-Endpoint.xml`
   - Action: On the element `Endpoint.status` (id attribute on line 104),
     change the `<short>` value on line 106 from
     `active | limited | suspended | error | off | entered-in-error`
     to
     `active | limited | suspended | error | off | entered-in-error | unknown`.
   - Details: Do not touch `<snapshot>` content — it is regenerated by
     the publisher. Confine the edit to the `<element id="Endpoint.status">`
     block inside the authored differential content.

3. **(Optional) Refine the `Endpoint.status` element definition**
   - File: `source/endpoint/structuredefinition-Endpoint.xml`
   - Action: Consider updating the `<definition>` on line 107 to
     acknowledge the new escape value — e.g., append a sentence such as
     "Use `unknown` when the current operational status of the endpoint
     cannot be determined."
   - Details: Optional, subject to PA editorial preference. Not strictly
     required by the resolution.

4. **Verify no other artefacts hard-code the six-value enumeration**
   - File: repo-wide (search only; no edits expected)
   - Action: From the repo root, run a search for the literal short-text
     string and for references to the ValueSet URL to confirm no other
     authored page lists the codes explicitly:
     - `rg "active \| limited \| suspended \| error \| off \| entered-in-error" source/`
     - `rg "ValueSet/endpoint-status" source/`
   - Details: Update any additional occurrences found to include
     `unknown`. Do not edit files under `tools/`, `qa/`, `schema/`, or
     any `<snapshot>` block.
