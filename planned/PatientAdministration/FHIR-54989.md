## Resolution Summary

Resolution: **Persuasive**.

The resolution will update the **short description** of `Location.code` so it
no longer implies the element is used for identification (which is the role
of `Location.identifier`). The exact change called for by the resolution:

- **From:** `Codes that identify this location`
- **To:** `Codes that **represent** this location`

The resolution explicitly states that the underlying long-form `definition`
and `comment` already describe the proper usage well, so only the `<short>`
text needs to change. Broader guidance on how to use this datatype property
together with terminology services is being developed in the Administration
Incubator and is **out of scope** for this ticket.

## Feature Proposal

### Problem Statement

Reviewers (Comment 15, Ron G. Parker) noted that the purpose of and
difference between `Location.identifier` and `Location.code` is unclear in
the current specification:

- `Location.code` is currently bound (example strength) to the ISO 3166
  country-code value set (`http://hl7.org/fhir/ValueSet/iso3166-1-2`),
  which overlaps semantically with `Location.address.country`.
- The current short description "Codes that **identify** this location"
  reads as if `Location.code` is an identifier, blurring the line with
  `Location.identifier`.

The cross-references for this ticket confirm the affected paths:
`Location.identifier`, `Location.code`, and `Location.address.country` (no
external GitHub issues, PRs, or commits are linked).

### Proposed Change

Single, targeted edit to the `Location` StructureDefinition differential in
`HL7/fhir`:

- **Element:** `Location.code` (id `Location.code`, path `Location.code`).
- **Field changed:** `<short>` value only.
  - Old: `Codes that identify this location`
  - New: `Codes that represent this location`
- **Unchanged:** `<definition>`, `<comment>`, cardinality (`0..*`), type
  (`CodeableConcept`), `isSummary` (`true`), and the existing example
  binding to `http://hl7.org/fhir/ValueSet/iso3166-1-2` (binding name
  `LocationCode`, strength `example`).

XML diff (in `source/location/structuredefinition-Location.xml`, line 177):

```xml
<element id="Location.code">
  <path value="Location.code"/>
- <short value="Codes that identify this location"/>
+ <short value="Codes that represent this location"/>
  <definition value="A coded representation of this Location, where the codes may be found in a geospatial system."/>
  ...
</element>
```

No other element, no extension, no value set, and no search parameter is
modified by this ticket. The resolution defers any deeper guidance on
`Location.code` semantics and terminology to work being done in the
Administration Incubator.

### Design Rationale

- The resolution discussion concluded that the existing `<definition>` and
  `<comment>` text on `Location.code` already explains the intended usage
  ("a coded representation of this Location… geospatial or administrative
  regions… for identifiers prefer `Location.identifier`"). Only the
  one-line `<short>` is misleading because it uses the verb "identify".
- "Represent" is consistent with the existing `<definition>` ("A coded
  **representation** of this Location") and with the binding description
  ("Codes for **representing** locations, such as country codes."), so the
  short text now matches the rest of the element.
- Alternatives such as removing `Location.code` entirely, retyping it, or
  re-binding it to a different value set were not chosen — the resolution
  is explicitly minimal ("Persuasive" with a wording-only fix), with the
  larger semantic clarification handed to the Administration Incubator.
