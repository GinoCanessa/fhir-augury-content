## Resolution Summary

Resolution: **Persuasive with Modification**.

The ticket flags a naming mismatch between two value sets used on `RelatedPerson`:

- The value set bound to `RelatedPerson.role` (the functional role) is named `PatientRelationshipType`, even though it is *not* bound to the `relationship` element.
- The value set bound to `RelatedPerson.relationship` (the personal relationship) is the v3 THO value set `PersonalRelationshipRoleType`, whose name also contains "Role" despite being for personal relationships.

The WG accepted a modified fix scoped to FHIR Core:

1. Rename the functional-role value set (the one currently named `PatientRelationshipType`, at `source/relatedperson/valueset-relatedperson-relationshiptype.xml`):
   - `name`: `PatientRelationshipType` → `PatientContactFunctionalRole`
   - `title`: `Patient Relationship Type` → `Patient Contact Functional Role`
   - `description`: `"A set of codes that can be used to indicate the relationship between a Patient and a Related Person."` → `"A set of codes that can be used to indicate the functional role of an individual to a Patient (used on Patient contacts and RelatedPerson)."`
2. The `PersonalRelationshipRoleType` value set is a v3 THO value set; its rename (to `Personal Relationship Type`) is out of scope here and must be filed as a separate THO request.

The split that led to two value sets on `RelatedPerson` was introduced by [FHIR-15841](https://jira.hl7.org/browse/FHIR-15841); FHIR-54561 only clarifies the naming produced by that split.

## Feature Proposal

### Problem Statement

`RelatedPerson` has two concept-bound elements:

- `RelatedPerson.relationship` — the *personal* relationship (mother, spouse, etc.), bound to the v3 THO value set `http://terminology.hl7.org/ValueSet/v3-PersonalRelationshipRoleType` with bindingName `PatientRelationshipType`.
- `RelatedPerson.role` — the *functional* role (emergency contact, interpreter, guarantor, etc.), bound to `http://hl7.org/fhir/ValueSet/relatedperson-relationshiptype` with bindingName `ContactRelationship`.

The value set at `ValueSet/relatedperson-relationshiptype` carries `name="PatientRelationshipType"` and `title="Patient Relationship Type"`. The word "relationship" strongly implies it belongs to `RelatedPerson.relationship`, but the binding on `.relationship` actually points at the v3 THO value set, and `PatientRelationshipType` is bound to `.role`. This is inverted from what the names suggest and is a concrete source of confusion for implementers reading the spec.

### Proposed Change

In FHIR Core, rename and redescribe the functional-role value set so its identity matches its binding:

| Field | Before | After |
|-------|--------|-------|
| `ValueSet.name` | `PatientRelationshipType` | `PatientContactFunctionalRole` |
| `ValueSet.title` | `Patient Relationship Type` | `Patient Contact Functional Role` |
| `ValueSet.description` | `A set of codes that can be used to indicate the relationship between a Patient and a Related Person.` | `A set of codes that can be used to indicate the functional role of an individual to a Patient (used on Patient contacts and RelatedPerson).` |

The canonical URL (`http://hl7.org/fhir/ValueSet/relatedperson-relationshiptype`), the `id` (`relatedperson-relationshiptype`), the OIDs, the `version`, and all `<compose>` contents stay unchanged. This keeps the binding URL on `RelatedPerson.role` (and on any external consumer) stable; only the human-facing name/title/description change.

No changes are proposed in this ticket to:

- The v3 THO `PersonalRelationshipRoleType` value set (separate THO ticket).
- The binding URLs on `RelatedPerson.relationship` or `RelatedPerson.role`.
- The code systems referenced from the `<compose>` block.
- The bindingName extensions on the elements (see Open Questions).

### Design Rationale

- The chosen name aligns the value set identity with the element it is bound to (`RelatedPerson.role`, the *functional* role) and with the `.role` element's own comment ("This property is for functional relationships. Personal relationships are represented in `RelatedPerson.relationship`." — `source/relatedperson/structuredefinition-RelatedPerson.xml:221`).
- Because `PatientContactFunctionalRole` is used both by `Patient.contact` in spirit (the same list of functional contact roles) and `RelatedPerson.role`, the updated description explicitly names both usages.
- Keeping `id`, canonical `url`, and OIDs unchanged avoids a breaking rename for implementers and preserves the existing binding in `structuredefinition-RelatedPerson.xml` without edits to the element's `<valueSet>` target.
- The WG deliberately left `PersonalRelationshipRoleType` alone because it is owned by THO (v3), not FHIR Core.

### Breaking Changes

No breaking changes identified:

- The canonical `url` (`http://hl7.org/fhir/ValueSet/relatedperson-relationshiptype`) is unchanged, so existing bindings and implementer references continue to resolve.
- The `id` and all OIDs are unchanged.
- `ValueSet.name` is a computable identifier, so the rename is a compatibility-affecting change for consumers that look up the value set by `name` (as opposed to by `url`). In FHIR, the canonical lookup is by `url`, and name-based lookup is discouraged; the WG's acceptance implies this cost is acceptable. Flag in Open Questions for explicit confirmation.
- No changes to `<compose>` membership, so the code set stays identical.

### Related Specifications

- `RelatedPerson.role` element in `source/relatedperson/structuredefinition-RelatedPerson.xml` (line 217+) — its `<valueSet>` binding target (line 235) is unchanged, so no edit is required here, but reviewers should verify the referenced bindingName remains coherent with the new value-set name.
- `Patient.contact.relationship` (`source/patient/structuredefinition-Patient.xml:550`) binds to `v3-PersonalRelationshipRoleType`, not this value set — unaffected by this ticket (and a candidate to reconsider if a future ticket touches Patient.contact functional roles).

### Related Tickets
