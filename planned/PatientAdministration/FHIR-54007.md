## Resolution Summary

**Resolution:** Persuasive.

The Scope and Usage section of `PractitionerRole` currently contains the
sentence:

> "The role, specialty, Location telecom and HealthcareService properties can be repeated..."

That list mixes element names with search-parameter names (e.g. `role` is a
search parameter on `PractitionerRole.code`; `telecom` is the search
parameter on `PractitionerRole.contact.telecom`). Because the surrounding
prose is talking about *properties* (i.e., resource elements), the list
must be reworded to use the actual element names. The approved replacement
sentence is:

> "The code, specialty, location, contact and healthcareService properties can be repeated..."

This is a documentation-only technical correction; no schema, cardinality,
binding, or behavioral change is required.

## Feature Proposal

### Problem Statement

The Scope and Usage narrative of the PractitionerRole resource references
elements by names that do not exist on the resource. Specifically, "role"
and "Location telecom" are not element paths on `PractitionerRole` — they
are search-parameter names that resolve to `PractitionerRole.code` and
`PractitionerRole.contact.telecom` respectively. Implementers reading the
narrative cannot match the listed names back to the resource definition,
which is misleading and undermines the documentation. The reporter
(maheshravella) flagged this as a typo in the Scope and Usage section.

### Proposed Change

Edit a single sentence in the PractitionerRole introduction page so the
listed property names match actual `PractitionerRole` element names:

- Replace
  `The role, specialty, Location telecom and HealthcareService properties can be repeated...`
  with
  `The code, specialty, location, contact and healthcareService properties can be repeated...`

Mapping of the change (old → new), all on `PractitionerRole`:

| Old token (in narrative) | New token (element name) |
|---|---|
| `role` (search parameter)             | `code` (`PractitionerRole.code`) |
| `specialty`                           | `specialty` (unchanged) |
| `Location telecom` (search parameter) | `location` + `contact` (`PractitionerRole.location`, `PractitionerRole.contact`) |
| `HealthcareService`                   | `healthcareService` (`PractitionerRole.healthcareService`) |

Note that the original sentence collapsed two distinct elements
(`location` and `contact`) into the single search-parameter token
"Location telecom". The corrected sentence lists them separately, per the
resolution wording.

No other changes (cardinality, type, binding, search parameter
definitions, examples) are required by the resolution.

### Design Rationale

The Scope and Usage section is prose written in terms of resource
*properties*. Mixing in search-parameter names is internally inconsistent
and confusing — a reader trying to find `role` or `Location telecom` on
the resource will not find them. The resolution explicitly chose to align
the list with the actual element names rather than rewriting the sentence
to be search-parameter-oriented, because the surrounding paragraph
("...properties can be repeated if required in other instances of the
PractitionerRole...") is clearly element-level language. This is the
minimal, low-risk fix consistent with the existing narrative style and
with how other resource introductions describe their elements.
