## Resolution Summary

Resolution: **Persuasive**.

Per the ticket's resolution description: *"We will seek several examples of the InsuranceProduct/InsurancePlan resources and add to the specification (replacing/updating the existing stub examples)."*

Concretely, the existing near-empty `InsurancePlan/example` and `InsuranceProduct/example` instances in the FHIR Core build must be replaced (or supplemented) with realistic, element-rich example instances that exercise meaningful portions of each resource, and those new examples must be registered with the FHIR publisher so they appear on the resource's Examples page. The in-ticket comment from Brian Postlethwaite suggests sourcing (with appropriate adaptation) examples from the US National Directory of Healthcare (NDH) IG's `ndh-InsurancePlan` profile as a starting point.

## Feature Proposal

### Problem Statement

The published Examples pages for InsurancePlan and InsuranceProduct currently carry stub instances that convey almost nothing about how the resource is intended to be populated:

- `InsurancePlan/example` — only an `id`; no `name`, no `status`, no `type`, no `coverageArea`, no `plan`, no `contact`.
- `InsuranceProduct/example` — only `id` and `name="foo"`; does not exercise `ownedBy`, `administeredBy`, `coverageArea`, `coverage`, `plan`, `network`, or `endpoint`.

This is highly visible (these are the first worked examples implementers see when they land on the resource page from the build) and it undermines implementer confidence in the resources, particularly now that InsuranceProduct has been split from the legacy InsurancePlan structure. The ticket also flags a secondary, out-of-scope JIRA tooling issue (InsuranceProduct cannot yet be selected as a Related Artifact); that is not in the resolution and is not addressed by this plan.

### Proposed Change

Deliver richer canonical examples for both resources, plus any supporting plumbing needed so the HL7 publisher surfaces them. Work is confined to the two per-resource authoring folders in `HL7/fhir`:

1. **InsurancePlan (`source/insuranceplan/`)**

   a. Flesh out `insuranceplan-example.xml` (the canonical "general" example with `id="example"`) into a realistic narrative-bearing instance. At a minimum populate: `text` (generated narrative `div`), `identifier`, `status`, `type` (CodeableConcept), `name`, `alias`, `period`, `ownedBy` (reference to an Organization), `administeredBy`, `coverageArea` (reference to a Location), `contact.purpose` / `contact.name` / `contact.telecom` / `contact.address`, `endpoint`, plus at least one `plan` with `type`, `coverageArea`, `network`, `generalCost`, and `specificCost.category` + `specificCost.benefit.type` + `specificCost.benefit.cost`.

   b. Keep (or revise) `insuranceplan-example1.xml` as a second, minimally-different example (e.g., a Silver plan) so the publisher's multi-example layout is exercised. Remove the stray `[` typo currently present in the `schemaLocation` of `insuranceplan-example.xml`.

   c. Optionally introduce a third example (e.g., `insuranceplan-example-ndh.xml`) adapted from the US NDH IG's `ndh-InsurancePlan` illustration, scrubbed of IG-specific profile/extension URLs so it validates against core.

   d. Register every new example in `list-InsurancePlan-examples.xml` (the publisher's examples index for this resource) with `description` + `title` extensions and an `item.reference` pointing at `InsurancePlan/<id>`.

   e. Update `insuranceplan-examples-header.xml` with a short narrative paragraph describing what each example is meant to illustrate (the file is currently an empty `<div/>` shell).

2. **InsuranceProduct (`source/insuranceproduct/`)**

   a. Replace the stub `insuranceproduct-example.xml` (`id="example"`, `name="foo"`) with a realistic "general product" instance exercising `identifier`, `status`, `type`, `name`, `alias`, `period`, `ownedBy`, `administeredBy`, `coverageArea`, at least one `contact` with `purpose`+`telecom`+`address`, `endpoint`, and one or more `coverage` entries with `type`, `network`, and `benefit.type`+`benefit.limit`+`benefit.requirement`. The existing `insuranceproduct-example1.xml` (the HL7 staff plan) is already closer to the target shape and should be retained as the secondary example, with minor clean-ups (consistent indentation, add a `status`, verify all referenced `Location`, `Organization`, and `Endpoint` ids exist in the core examples corpus or use the standard stub ids used elsewhere in `source/`).

   b. Optionally add an NDH-derived `insuranceproduct-example-ndh.xml` as above.

   c. Register every example in `list-InsuranceProduct-examples.xml`.

   d. Update `insuranceproduct-examples-header.xml` with a paragraph describing each example.

3. **Cross-linking.** Each new example in `source/insuranceplan/` should reference an InsuranceProduct (via `InsurancePlan.product` if the resource carries such a reference in the current build) so the "tour" flows InsuranceProduct → InsurancePlan rather than two disconnected stubs. The exact element name must be verified against `source/insuranceplan/structuredefinition-InsurancePlan.xml` before authoring.

No StructureDefinition changes, no ValueSet/CodeSystem changes, no search-parameter changes, no narrative (`-introduction`, `-notes`) changes, no terminology binding changes are required by the resolution — the work is purely "add/replace example instances and register them".

### Design Rationale

- **Scope faithful to the resolution.** The resolution text explicitly says "replacing/updating the existing stub examples"; it does not authorise structural changes, new elements, or scope changes to either resource. The plan therefore deliberately keeps away from `structuredefinition-InsurancePlan.xml`, `structuredefinition-InsuranceProduct.xml`, `*-notes.xml`, `*-introduction.xml`, and all ValueSets in these folders.
- **Mirror the existing "one general + one concrete" pattern** already used by other Patient Administration resources in `source/` (e.g., `source/organization/organization-example*.xml`). That minimises publisher surprises and keeps the Examples tab on each resource page consistent across the spec.
- **Use the NDH IG only as a source of realistic data shapes**, not as a direct copy. NDH is R4 and uses profile-specific extensions and slicing that do not exist in core; naive copy/paste would either fail validation or require authoring those extensions, which is out of scope.
- **Keep ids stable.** The canonical example retains `id="example"` so no inbound links (e.g., `InsurancePlan/example`) break elsewhere in the spec or in tests.

### Testing Considerations

- Rebuild the spec locally using the standard publisher entry points (`publish.sh` / `publish.bat`) and visit `insuranceplan.html`, `insuranceplan-example.html`, `insuranceplan-examples.html`, `insuranceproduct.html`, `insuranceproduct-example.html`, `insuranceproduct-examples.html` to visually confirm each example renders with a narrative and realistic data.
- Run the instance validator over every new/rewritten example (`java -jar org.hl7.fhir.validator.jar ...` or the validator baked into the publisher output). Any `information`-level warnings about example-bound codes are acceptable; `error`-level cardinality/invariant findings are not.

- Interoperability: implementers landing on the Examples page should be able to copy the XML into a server POST and have it accepted; verify POSTing each new example to a reference server (HAPI) returns 201 with no validator complaints.
