## Resolution Summary

**Resolution:** Persuasive — "Will remove the additional text as noted."

The ticket targets the `RelatedPerson.communication` element. Its current
definition reads *"A language which may be used to communicate with the
related person about the patient's health."* The approved change removes the
qualifying clause *"about the patient's health"* (and its trailing period
adjustment), yielding *"A language which may be used to communicate with the
related person."* No cardinality, type, or binding changes are approved — the
change is purely to the human-readable definition/short text.

## Feature Proposal

### Problem Statement

`RelatedPerson` is used for more than health-related communication. It covers
personal relationships, emergency contacts, caregivers, and other non-clinical
roles (see `source/relatedperson/relatedperson-introduction.xml`, which
describes RelatedPersons as typically having "a personal relationship or
non-healthcare-specific professional relationship to the patient"). The
current wording of `RelatedPerson.communication` — "communicate with the
related person about the patient's health" — improperly narrows the purpose of
the element to healthcare communication, which conflicts with the broader
scope of the resource. The reporter (Eric Haas) flagged this as an editorial
inconsistency; the Patient Administration work group agreed to remove the
narrowing phrase.

### Proposed Change

In the RelatedPerson StructureDefinition differential, edit the
`RelatedPerson.communication` element so that:

- `short` changes from `"A language which may be used to communicate with the
  related person about the patient's health"` to `"A language which may be
  used to communicate with the related person"`.
- `definition` changes from `"A language which may be used to communicate
  with the related person about the patient's health."` to `"A language which
  may be used to communicate with the related person."` (trailing period
  preserved).

No other fields on the element change (min/max, type `BackboneElement`,
comment about default-local-language behaviour, RIM / interface mappings all
stay as-is).

**Ancillary consistency note (open question — see below):** The child element
`RelatedPerson.communication.language` carries the same phrasing in its
`short` value (line 435:
`"The language which can be used to communicate with the related person about
the patient's health"`). The resolution text speaks only of "the additional
text as noted," which in the ticket description refers to the parent
`communication` element; it does not explicitly call out the child's short
text. Recommend confirming with the work group whether the child's `short`
should be updated for consistency in the same change.

### Design Rationale

The change is a narrow editorial correction, not a semantic or structural
change:

- It aligns the element's definition with the broader scope of the
  `RelatedPerson` resource and with the parallel `Patient.communication`
  element (whose definition does not contain the "about the patient's health"
  qualifier).
- It preserves the element name, cardinality (`0..*`), type
  (`BackboneElement`), child structure (`.language`, `.preferred`), and both
  RIM (`LanguageCommunication`) and interface (`ParticipantLiving.communication`)
  mappings — so downstream conformance, interface compatibility, and logical
  model alignment are unaffected.
- Because the comment on the element already explains the element's purpose in
  terms of interpreter needs and multi-modal proficiency (rather than
  health-specific communication), striking the narrower phrase from the
  definition brings the prose into agreement with what the comment already
  says.

### Open Questions

- **Scope of "the additional text as noted."** The ticket description focuses
  on the parent `RelatedPerson.communication` element. The child element
  `RelatedPerson.communication.language` carries the identical phrasing in
  its `short` value (line 435). Task 2 above should only be applied if the
  Patient Administration work group confirms they intended the consistency
  edit to include the child element. If not, leave line 435 as-is and close
  the ticket after Task 1.
- **Comment field review.** The existing `comment` on
  `RelatedPerson.communication` (line 417) talks about interpreter needs and
  does not contain the offending phrase, so it does not need editing. Worth
  a final sanity check during PR review.
- **Downstream IG impact.** The briefing notes that general FhirCore
  touch-points to downstream IGs do not apply to this change. IGs that
  profile `RelatedPerson.communication` generally override `short`/`definition`
  themselves; no cascading updates are required in `HL7/fhir` for this
  ticket, but IG authors may want to review their own prose for the same
  wording.
