## Resolution Summary

The resolution is **Persuasive**. Encounter.serviceType currently has
inconsistent text between its `short` display and `definition`, and its
wording diverges from the deliberately-aligned Appointment.serviceType
element. The approved fix (per the resolution table) updates
`Encounter.serviceType` so its `short` and `definition` exactly mirror
the Appointment.serviceType wording (substituting "encounter" for
"appointment"):

| Property | New Value |
|----------|-----------|
| Element Id | `Encounter.serviceType` |
| Definition | `The specific service that is to be performed during this encounter.` |
| Short Display | `The specific service that is to be performed during this encounter` |

Per the reporter's comment, the change is intended to be applied
**after the current ballot closes (during R6 reconciliation)**. The
ticket is labelled `Normative-NonSubstantive`, signalling no semantic
change to the element's meaning, cardinality, type, or binding — only
text alignment.

## Feature Proposal

### Problem Statement

`Encounter.serviceType` today carries two different stories about what
it represents:

- `short` value: `Specific type of service`
- `definition` value: `Broad categorization of the service that is to be provided (e.g. cardiology).`

The short display says "specific" while the formal definition says
"broad categorization", which is internally contradictory and confuses
implementers about whether the element is for a granular service code
or a high-level grouping.

Patient Administration had previously aligned `Appointment.serviceType`
with `Encounter.serviceType` (same type `CodeableReference(HealthcareService)`,
same cardinality `0..*`, parallel binding), but the `short` /
`definition` text on the two elements was never reconciled.
Appointment.serviceType reads:

- `short`: `The specific service that is to be performed during this appointment`
- `definition`: `The specific service that is to be performed during this appointment.`

The two siblings should describe themselves the same way.

### Proposed Change

Update the `Encounter.serviceType` element definition in the FHIR Core
StructureDefinition for Encounter so its `short` and `definition`
mirror the Appointment.serviceType wording (substituting "encounter").

**File:** `source/encounter/structuredefinition-Encounter.xml`

**Element:** the `<element id="Encounter.serviceType">` block inside
`<differential>`.

**Before (current values in the clone @ db79dcd):**

```xml
<element id="Encounter.serviceType">
  <path value="Encounter.serviceType"/>
  <short value="Specific type of service"/>
  <definition value="Broad categorization of the service that is to be provided (e.g. cardiology)."/>
  ...
```

**After:**

```xml
<element id="Encounter.serviceType">
  <path value="Encounter.serviceType"/>
  <short value="The specific service that is to be performed during this encounter"/>
  <definition value="The specific service that is to be performed during this encounter."/>
  ...
```

Nothing else on the element changes:

- `min` / `max` remain `0..*`.
- `type` remains `CodeableReference` targeting
  `http://hl7.org/fhir/StructureDefinition/HealthcareService`.
- The `binding` (strength `example`, name `EncounterServiceType`) is
  untouched.
- `isSummary` remains `true`.
- No new constraints, comments, mappings, or aliases are added.

### Design Rationale

- **Cross-element consistency.** The Patient Administration WG already
  agreed Encounter.serviceType and Appointment.serviceType should be
  parallel; adopting Appointment's existing wording verbatim (with
  "encounter" substituted) is the cheapest way to remove the divergence
  without re-litigating the element's semantics.
- **Resolves an internal contradiction.** "Specific type of service"
  vs. "Broad categorization … (e.g. cardiology)" cannot both be right.
  The resolution chose the "specific service" framing — matching the
  existing `CodeableReference(HealthcareService)` type, which references
  a specific service instance rather than a coarse category.
- **Non-substantive by label.** The ticket carries
  `Normative-NonSubstantive`, signalling that the WG views this as a
  text/clarity fix only. No conformance change, no instance migration.
- **Alternatives not chosen.** Updating Appointment.serviceType to match
  the old Encounter wording, or inventing fresh text for both, would
  re-open the underlying semantics debate and likely require a
  substantive vote; reusing Appointment's already-balloted wording
  avoids that.

### Step-by-Step Tasks

#### HL7/fhir

1. **Update Encounter.serviceType `short` display**
   - File: `source/encounter/structuredefinition-Encounter.xml`
   - Action: In the `<differential>` element block whose
     `<element id="Encounter.serviceType">` carries
     `<path value="Encounter.serviceType"/>`, replace the
     `<short value="Specific type of service"/>` line with
     `<short value="The specific service that is to be performed during this encounter"/>`.
   - Details: Note there is **no trailing period** on the `short`
     attribute, matching Appointment.serviceType's existing convention.

2. **Update Encounter.serviceType `definition`**
   - File: `source/encounter/structuredefinition-Encounter.xml`
   - Action: In the same `<element>` block, replace the
     `<definition value="Broad categorization of the service that is to be provided (e.g. cardiology)."/>`
     line with
     `<definition value="The specific service that is to be performed during this encounter."/>`.
   - Details: Trailing period **is** present on `definition`, again
     matching Appointment.serviceType. Do not touch any sibling
     attributes (`min`, `max`, `type`, `binding`, `isSummary`) on this
     element.

3. **Confirm `<snapshot>` is not hand-edited**
   - File: `source/encounter/structuredefinition-Encounter.xml`
   - Action: Verify that any `<snapshot>` block within the file remains
     untouched in the diff. Per the briefing, snapshots are regenerated
     by the publisher; they will be refreshed on the next publish and
     must not be edited manually in this PR.
   - Details: If the local diff shows snapshot changes, revert them and
     let the publisher regenerate.
