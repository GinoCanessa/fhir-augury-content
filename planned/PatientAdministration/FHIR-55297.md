## Feature Proposal

### Problem Statement

`HealthcareService` binds several CodeableConcept elements to value sets and code systems that today live in the FHIR core specification (`source/healthcareservice/`):

- `HealthcareService.category` → `http://hl7.org/fhir/ValueSet/service-category`
- `HealthcareService.type` → `http://hl7.org/fhir/ValueSet/service-type`
- `HealthcareService.specialty` → `http://hl7.org/fhir/ValueSet/c80-practice-codes`
- `HealthcareService.serviceProvisionCode` → `http://hl7.org/fhir/ValueSet/service-provision-conditions`
- `HealthcareService.eligibility.code` → `http://hl7.org/fhir/ValueSet/eligibility`
- `HealthcareService.program` → `http://hl7.org/fhir/ValueSet/program`
- `HealthcareService.characteristic` → `http://hl7.org/fhir/ValueSet/service-mode`
- `HealthcareService.referralMethod` → `http://hl7.org/fhir/ValueSet/service-referral-method`

These value sets are general administrative terminologies useful well outside FHIR R-series — in CDA, V2, and other product families — and several of them have been re-published in THO (`http://terminology.hl7.org`) as version-independent assets. Continuing to bind core resources to the FHIR-namespaced copies pins them to a single FHIR release cycle and forks the source-of-truth between core and THO.

The Patient Administration WG, at the January 26 WGM, agreed to formally migrate the remaining unmigrated value sets used by `HealthcareService` (other than `communication`, which is its own tracker) and to re-point the bindings at the THO copies for the value sets that are *already* in THO.

### Proposed Change

Two coordinated workstreams: re-bind in `HL7/fhir`, and migrate authoritative artifacts in `HL7/UTG`.

**A. `HL7/fhir` — re-bind HealthcareService bindings to the THO ValueSet URLs.**

In `source/healthcareservice/structuredefinition-HealthcareService.xml`, change the `<binding><valueSet value="..."/></binding>` URL on each of the following elements from `http://hl7.org/fhir/ValueSet/<name>` to `http://terminology.hl7.org/ValueSet/<name>`. Strength stays as authored (all are `example` today, except where the existing differential says otherwise — do not change strength as part of this ticket).

**B. `HL7/UTG` — publish the newly migrated terminologies.**

Add the following authoritative artifacts under `input/sourceOfTruth/fhir/`:

- `valueSets/ValueSet-eligibility.xml` with canonical `http://terminology.hl7.org/ValueSet/eligibility`, `name = "Eligibility"`. Composition: `include` the new `http://terminology.hl7.org/CodeSystem/eligibility` CodeSystem.
- `codeSystems/CodeSystem-eligibility.xml` with canonical `http://terminology.hl7.org/CodeSystem/eligibility`, `name = "Eligibility"`. Migrate every concept and `display`/`definition` from `cache/github/repos/HL7_fhir/clone/source/healthcareservice/codesystem-eligibility.xml` (the FHIR core copy at canonical `http://hl7.org/fhir/eligibility`).
- `valueSets/ValueSet-service-mode.xml` with canonical `http://terminology.hl7.org/ValueSet/service-mode`, `name = "ServiceMode"`. Composition: `include` `http://terminology.hl7.org/CodeSystem/service-mode`.
- `codeSystems/CodeSystem-service-mode.xml` with canonical `http://terminology.hl7.org/CodeSystem/service-mode`, `name = "ServiceMode"`. Migrate every concept from `cache/github/repos/HL7_fhir/clone/source/healthcareservice/codesystem-service-mode.xml` (canonical `http://hl7.org/fhir/service-mode`).

Use the existing migrated peers (`ValueSet-service-type.xml`, `CodeSystem-service-type.xml`, `ValueSet-service-category.xml`, etc.) as templates for steward extensions, `<status>`, `<experimental>`, `<caseSensitive>`, `<content>`, and `<concept>` packaging.

The `specialty` (`c80-practice-codes`) migration is bundled with the parallel OrgAffiliation/PractitionerRole work; that ticket — not this one — owns adding the THO artifact and choosing its canonical URL. This plan assumes that once that artifact lands, all three of HealthcareService, OrgAffiliation, and PractitionerRole re-bind to the same THO canonical URL in lockstep.

### Design Rationale

- The resolution is explicit about **which** value sets go to THO and which stay; the plan mirrors that list 1:1 without expanding scope (e.g., `communication` is deliberately untouched here per the resolution).
- For value sets the resolution states are "already in THO," the only action is the canonical-URL change in the binding plus removing the now-duplicate core-side ValueSet files; we do *not* re-publish or re-version those terminologies from core.
- For value sets newly migrated, we keep the slug (`eligibility`, `service-mode`) and `name` (`Eligibility`, `ServiceMode`) identical to the existing core artifact so downstream implementers do not see two competing concepts. Only the canonical namespace changes (FHIR core → THO).
- Strength changes are out of scope: the resolution explicitly addresses location of the value set (FHIR core vs THO), not binding strength. Existing `example` strengths are preserved.
- Specialty is intentionally deferred to align with the OrgAffiliation/PractitionerRole work to avoid landing a one-off URL choice that the parallel ticket then has to re-revise.

### Breaking Changes

The canonical-URL change is a **breaking change for any consumer** that pinned to the old `http://hl7.org/fhir/ValueSet/<name>` canonicals for `service-category`, `service-type`, `service-provision-conditions`, `program`, `service-referral-method`, `eligibility`, or `service-mode`, including:

- Implementations that resolve those canonicals against a local terminology server (will 404 once the core-side copies are deleted in the next core ballot).
- IGs and profiles in downstream repos that constrain `HealthcareService` and re-state these bindings using the FHIR-core URL — they will need a follow-up edit.
- Validators that snapshot `HealthcareService` against pre-migration core packages will report a binding-URL mismatch when revalidated against post-migration core.

For the `eligibility` and `service-mode` CodeSystems, instances using the existing system URLs `http://hl7.org/fhir/eligibility` / `http://hl7.org/fhir/service-mode` will become invalid once those CodeSystems are removed from core, unless UTG re-publishes with the same URLs (this plan assumes UTG re-publishes under `http://terminology.hl7.org/CodeSystem/<name>` per the convention of the already-migrated peers, which means instances must be re-coded). PA WG should confirm the system-URL story explicitly before merging.

`HealthcareService.specialty` is *not* re-bound in this ticket — it is deferred to the parallel OrgAffiliation/PractitionerRole specialty-migration ticket. No breaking change there in this PR.
