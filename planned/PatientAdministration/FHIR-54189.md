## Resolution Summary

Resolution: **Persuasive**.

Resolution description (verbatim from the ticket): *"Will add the requested
search parameter for the slot end date."*

Concretely, this requires adding a new `date`-typed `SearchParameter` named
`end` to the `Slot` resource, with expression `Slot.end`, mirroring the
existing `start` search parameter (which targets `Slot.start`). No other
behavioural changes were requested.

## Feature Proposal

### Problem Statement

`Slot` already exposes a `start` search parameter (`Slot.start`, type
`date`) but does not expose a parameter for `Slot.end`. Clients that need
to constrain results to slots whose end falls within a particular window
cannot do so using a single `_search` interaction; they must either
post-filter on the client or chain server queries. The reporter's use case
("a patient is only available in the morning, slots must both start and
end within 08:00–12:00") cannot be expressed in a single FHIR search
without an `end` parameter on `Slot`.

`Slot.end` is a `1..1 instant` element on `Slot` (see
`structuredefinition-Slot.xml` lines 254–272) and is `isSummary=true`, so
it is a natural and well-defined target for a `date`-typed search
parameter.

### Proposed Change

Add a new entry to the Slot search-parameter bundle that defines an `end`
search parameter symmetric with the existing `start` parameter:

- **id:** `Slot-end`
- **url:** `http://hl7.org/fhir/build/SearchParameter/Slot-end`
- **code:** `end`
- **base / resource:** `Slot`
- **type:** `date`
- **expression / build path:** `Slot.end`
- **description:** `"Slot end date/time."` (symmetric with `Slot-start`'s
  `"Appointment date/time."`; consider re-titling Slot-start in a future
  ticket if the WG wants matched wording, but that is out of scope here).
- **standards-status extension:** `normative` (matches every other Slot
  search parameter in the bundle).
- **processingMode:** `normal` (matches the rest of the bundle).

Concrete XML to add as a new `<entry>` to
`source/slot/bundle-Slot-search-params.xml`, placed between the existing
`Slot-start` (id `Slot-start`, code `start`) and `Slot-status` entries to
keep the alphabetical ordering already used in the file:

```xml
<entry>
  <resource>
    <SearchParameter>
      <id value="Slot-end"/>
      <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
        <valueCode value="normative"/>
      </extension>
      <extension url="http://hl7.org/fhir/build/StructureDefinition/path">
        <valueString value="Slot.end"/>
      </extension>
      <url value="http://hl7.org/fhir/build/SearchParameter/Slot-end"/>
      <description value="Slot end date/time."/>
      <code value="end"/>
      <type value="date"/>
      <expression value="Slot.end"/>
      <processingMode value="normal"/>
    </SearchParameter>
  </resource>
</entry>
```

No other resource files require functional changes. The HL7 publisher
regenerates the per-resource search-parameter table on `slot.html` and the
global search-parameter registry from this bundle, so no hand-editing of
generated artifacts is needed.

### Design Rationale

- **Symmetry with `start`.** `Slot.start` already has a `date` search
  parameter (`Slot-start`, code `start`, expression `Slot.start`). The
  resolution explicitly asks for the equivalent on `Slot.end`, so the
  cleanest design is a sibling parameter with the same shape.
- **Type = `date`.** FHIR's `date` search parameter type natively supports
  the `instant` data type and the comparator prefixes (`eq`, `gt`, `lt`,
  `ge`, `le`, `sa`, `eb`, `ap`) needed to express "ends before X" or
  "ends within window". This is the same choice made for `Slot-start`.
- **Code = `end`.** Mirrors the simple, single-word code used for
  `start`. Combined-window queries can already be expressed by repeating
  the parameter with comparators (`?start=ge2026-01-15T08:00&end=le2026-01-15T12:00`),
  which directly satisfies the use case in the ticket description.
- **Standards status `normative`.** Slot is a normative resource and
  every existing Slot search parameter carries the `normative`
  standards-status extension; a new `end` parameter must do the same to
  remain internally consistent.
- **No new composite parameter.** A `start-end` composite was not
  requested in the resolution and is not needed: the reporter's use case
  is satisfied by combining `start` and `end` in the same query. Adding a
  composite would be a separate change and is left out of scope.

- **Change Scope:** Minor (single additive `<entry>` in one bundle file).

### Breaking Changes

No breaking changes identified.

- The change is purely additive: a brand-new search parameter `code=end`
  on `Slot`. No existing parameter is renamed, retyped, or removed.

  instances and existing servers/clients are unaffected.
- The new code `end` does not collide with any existing Slot search

  `service-category`, `service-type`, `service-type-reference`,

  the per-resource bundles; the new Slot `end` parameter will be picked
  up without additional editing.
- **`Schedule` and `Appointment`.** Both reference Slot in scheduling
  workflows. Neither needs a code change for this ticket: `Slot.end`
  is a property of Slot, not Schedule or Appointment. Documentation
  in those pages does not need to change to enable the new parameter.

## Implementation Plan

### Prerequisites

None. `Slot.end` already exists on the Slot StructureDefinition with
`min=1`, `max=1`, `type=instant`, and `isSummary=true`
(`source/slot/structuredefinition-Slot.xml` lines 254–272). No upstream
ticket, terminology addition, or extension definition is required before
this work can proceed.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add the `Slot-end` SearchParameter entry to the Slot search-parameter bundle.**
   - File: `source/slot/bundle-Slot-search-params.xml`
   - Action: Insert a new `<entry>` between the existing `Slot-start`
     entry (currently the second-to-last entry in the bundle) and the
     `Slot-status` entry (the last entry), preserving the alphabetical
     ordering by `id` used elsewhere in the file.
   - Details: The new `<entry>` must contain a `SearchParameter` with:
     - `<id value="Slot-end"/>`
     - `structuredefinition-standards-status` extension
       `<valueCode value="normative"/>`
     - `build/StructureDefinition/path` extension
       `<valueString value="Slot.end"/>`
     - `<url value="http://hl7.org/fhir/build/SearchParameter/Slot-end"/>`
     - `<description value="Slot end date/time."/>`
     - `<code value="end"/>`
     - `<type value="date"/>`
     - `<expression value="Slot.end"/>`
     - `<processingMode value="normal"/>`
     Use the verbatim XML in the *Proposed Change* section above.

2. **Run the local publisher (or rely on CI) to regenerate derived artifacts.**
   - File: (no manual edits) — publisher regenerates the per-resource
     search-parameter table on the Slot HTML, the global
     `searchparameter-registry`, and any JSON/Turtle search-parameter
     definitions.
   - Action: Confirm that the generated output now contains a `Slot-end`
     SearchParameter and that the per-resource Slot search-parameter
     table includes a row for `end` with type `date` and expression
     `Slot.end`.
   - Details: Do **not** hand-edit any generated file. Per the briefing,
     `tools/`, `qa/`, `schema/`, and publish-style outputs are
     out-of-scope for source edits.

3. **Optional housekeeping (non-blocking):** If the WG wants the
   description text on `Slot-start` and `Slot-end` to match more closely
   (e.g., both reading "Slot start date/time." / "Slot end date/time."),
   raise a separate ticket. The current ticket only authorises adding
   `end`; do not silently rewrite `Slot-start`'s description.

### Testing Considerations

- **Revalidate** the Slot resource definition and the
  `bundle-Slot-search-params.xml` bundle through the publisher's
  validation step; confirm no new errors or warnings are introduced.
- **Sample queries** to exercise the new parameter against any reference
  server / test harness that consumes the build output:
  - `GET [base]/Slot?end=lt2026-02-01T12:00:00Z`
  - `GET [base]/Slot?start=ge2026-02-01T08:00:00Z&end=le2026-02-01T12:00:00Z`
    (the morning-availability use case from the ticket)
  - `GET [base]/Slot?end=sa2026-02-01T17:00:00Z` (slot ends strictly
    after 5pm)
- **Edge cases:** verify that the comparator prefixes (`eq`, `ne`, `gt`,
  `lt`, `ge`, `le`, `sa`, `eb`, `ap`) all work as they do for
  `Slot.start`, since both elements are `instant`.
- **Interoperability:** existing servers that do not implement `end`
  should simply ignore the parameter (per FHIR search semantics for
  unknown parameters with `Prefer: handling=lenient`); document any
  server-side capability statement updates as a downstream concern, not
  part of this spec change.
