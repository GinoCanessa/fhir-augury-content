## Resolution Summary

The ticket asks whether `Encounter.admission.reAdmission` should be a `boolean`
rather than a `CodeableConcept`, and — if it stays a `CodeableConcept` — argues
that the example binding (HL7 v2 table `v2-0092`) is too sparse to be useful.

The approved resolution (Persuasive) is **to keep the existing `CodeableConcept`
type and binding, but document *why* a coded type was chosen**. Specifically,
the reAdmission element must receive a `<comment>` explaining that:

- Some systems carry their own value set here to differentiate the
  reasons/circumstances of a re-admission — e.g., "complications from health
  conditions", "treatment non-compliance", "medication errors", "discharged
  against advice".
- The data item was also modelled as a user-defined table in HL7 v2, so a
  `CodeableConcept` permits easier v2 → FHIR migration than a plain boolean.

No datatype change, no cardinality change, no binding-strength change. No
modification to the referenced ValueSet (`http://terminology.hl7.org/ValueSet/v2-0092`).

## Feature Proposal

### Problem Statement

`Encounter.admission.reAdmission` is currently declared as a `CodeableConcept`
with an `example` binding to `http://terminology.hl7.org/ValueSet/v2-0092`
("Re-Admission Indicator", derived from HL7 v2 table 0092, which contains a
single code `R` = "Re-admission"). A reader of the Encounter specification
reasonably asks: *"if the only coded concept is 'is a re-admission', why is this
not a boolean?"* The spec does not currently answer that question in-line, so
implementers are left to guess (and to question the design, as this ticket
demonstrates).

### Proposed Change

Add a `<comment>` child element to the existing
`<element id="Encounter.admission.reAdmission">` definition in the Encounter
StructureDefinition's `<differential>`. The comment text (as dictated by the
resolution) is:

> Some systems will use their own valueset here where they choose to
> differentiate reasons/circumstances of the re-admission. Such as
> "complications from health conditions", "treatment non-compliance",
> "medication errors", "discharged against advice". This was also modelled
> as a user defined table in v2, thus permitting easier migration.

No other attributes of the element change:

- `path` = `Encounter.admission.reAdmission` (unchanged)
- `short` = "Indicates that the patient is being re-admitted" (unchanged)
- `definition` = "Indicates that this encounter is directly related to a prior
  admission, often because the conditions addressed in the prior admission were
  not fully addressed." (unchanged)
- `min`/`max` = `0..1` (unchanged)
- `type.code` = `CodeableConcept` (unchanged)
- `binding.strength` = `example`, `binding.valueSet` =
  `http://terminology.hl7.org/ValueSet/v2-0092` (unchanged)
- `binding.extension elementdefinition-bindingName` = `ReAdmissionType`
  (unchanged)
- v2/rim mappings (unchanged)

Example inserted fragment (inside the existing element, placed between
`<definition>` and `<min>` per FHIR ElementDefinition ordering):

```xml
<comment value="Some systems will use their own valueset here where they choose to differentiate reasons/circumstances of the re-admission. Such as &quot;complications from health conditions&quot;, &quot;treatment non-compliance&quot;, &quot;medication errors&quot;, &quot;discharged against advice&quot;. This was also modelled as a user defined table in v2, thus permitting easier migration."/>
```

### Design Rationale

- **Keep `CodeableConcept`, not `boolean`.** The resolution explicitly preserves
  the coded type because real-world systems classify the *kind* of re-admission
  (not merely its occurrence), and because v2 modelled this as a user-defined
  table — switching to `boolean` would drop that structural carryover and
  degrade v2 → FHIR mapping fidelity. A `boolean` can be trivially recovered
  from "is the element populated?", so coded remains strictly more expressive.
- **Keep the `example` binding and the existing `v2-0092` ValueSet.** The
  resolution does not expand the terminology: the comment explicitly tells
  implementers they are expected to bring their own value set. An `example`
  strength correctly signals that.
- **Use `<comment>` (i.e., `ElementDefinition.comment`), not `<definition>`.**
  `comment` is the conventional home for "usage notes / rationale" text on an
  element; `definition` remains the normative single-sentence definition. This
  matches existing FHIR authoring practice throughout `source/encounter/`.
- **Do not touch `terminology.hl7.org` / `HL7/UTG`.** `v2-0092` is governed by
  the UTG (Unified Terminology Governance) repository, but the resolution does
  not change any codes, displays, or ValueSet membership, so no UTG work is
  required by this ticket.
