## Resolution Summary

**Resolution:** Persuasive with Modification.

Resolution description (verbatim from Jira): *"To cater for cases where the participant status is unknown, we will make the property appointment.participant.status optional."*

The originally-requested change (adding an `unknown` / `other` "escape valve" code to the `ParticipationStatus` value set bound to `Appointment.participant.status`) was modified during disposition. Instead of altering the bound terminology, the WG agreed to relax the cardinality of `Appointment.participant.status` from `1..1` to `0..1`, so that a participant entry can omit the status when it is not known. The required binding is retained for cases where a status is supplied.

## Feature Proposal

### Problem Statement

`Appointment.participant.status` is currently a mandatory element (`min=1`) with a required binding to `http://hl7.org/fhir/ValueSet/participationstatus`, which expands to `accepted | declined | tentative | needs-action`. The bound value set has no `unknown` / `other` concept. In real-world workflows (e.g., when an appointment is being assembled from partial data, when a third-party participant has not yet been contacted, or when bridging from systems that do not capture this distinction) there may be no truthful value to populate from the required value set. With the current cardinality and required binding, the participant entry — and therefore the entire `Appointment` — cannot be validly populated.

This was raised by Lloyd McKenzie (ticket reporter) and dispositioned by the Patient Administration work group.

### Proposed Change

Relax the cardinality of `Appointment.participant.status`:

- **Element:** `Appointment.participant.status`
- **Path:** `Appointment.participant.status`
- **Cardinality change:** `1..1` → `0..1` (i.e., `<min>` from `1` to `0`; `<max>` remains `1`)
- **Type:** `code` — unchanged
- **Binding:** required to `http://hl7.org/fhir/ValueSet/participationstatus` — unchanged
- **`isSummary`:** `true` — unchanged
- **Definition / short / mappings:** unchanged

No changes are required to:

- `valueset-participationstatus.xml` or `codesystem-participationstatus.xml` (the WG explicitly chose the cardinality route over adding an "escape valve" code).
- The Appointment search parameter `Appointment.participant.status` — the FHIRPath continues to resolve to the (now optional) element.
- `<snapshot>` content inside `structuredefinition-Appointment.xml` — that block is regenerated by the publisher from the `<differential>`.

### Design Rationale

The disposition note ("Persuasive with Modification") records the WG's preference for the cardinality fix over a terminology fix. Reasons that flow naturally from FHIR design conventions:

- `ParticipationStatus` describes a *response state* by an actor toward an appointment (`accepted`, `declined`, `tentative`, `needs-action`). Adding `unknown` / `other` would dilute the semantic meaning and would conflict with the established pattern of using cardinality, not sentinel codes, to express "not known".
- Making the element optional is a smaller, lower-risk change and is local to one StructureDefinition; adding codes to a required-bound value set would ripple through every consumer of `ParticipationStatus`.
- The required binding is preserved, so when a status *is* supplied it must still be one of the four canonical participation states — preserving downstream interoperability.

The ticket comments do not record alternatives beyond "add unknown/other to the value set" (the original request) and "make the element optional" (the chosen modification).

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole repository for the change. Hosts the `Appointment` StructureDefinition whose differential cardinality is being relaxed.
- **Affected Files:**
  - `source/appointment/structuredefinition-Appointment.xml` — change `<min value="1"/>` to `<min value="0"/>` on the `Appointment.participant.status` element (line 1032 in the verified clone). All other attributes of the element are preserved.
- **Change Scope:** Minor (single-file, single-attribute differential edit; no terminology changes; no new elements; no schema-shape change).

### Breaking Changes

**This is technically a wire-compatible / forward-compatible change for producers but a backward-incompatible change for consumers / validators in the strict sense:**

- Existing valid R*current* `Appointment` instances remain valid: they all carry `participant.status`, which is still permitted (`0..1` ⊇ `1..1`).
- New instances may now omit `participant.status`. Consumers, validators, and downstream profiles that were written against `min=1` will see instances they previously would have rejected. Implementers relying on `participant.status` always being present must update guards/null-handling.
- Required binding is unchanged — when present, value must still come from `ParticipationStatus`.
- No element removal, no type change, no binding strength change, no search parameter change.

This should be called out in the publication notes / changelog as a relaxed-cardinality change on `Appointment.participant.status` (`1..1` → `0..1`).
