## Resolution Summary

The applied resolution is **Persuasive with Modification**. Two QA-report
warnings were raised against the new `PersonalRelationship` resource:

1. *"A resource must have an 'entered in error' status."* — addressed by
   adding a status-style element. The exact resolution wording is:

   > we will add an active status element as it is already used for
   > Patient, RelatedPerson etc.
   >
   > - Element name: active
   > - datatype: boolean
   > - short definition: Whether this relationship information is in active use
   > - definition: Whether this relationship information is in active use.
   > - isModifier: true
   > - cardinality: 0..1

2. *"All resources should have an identifier."* — explicitly **not**
   addressed by adding an element:

   > any use cases for it
   > we will ask the responsible persons for the QA report to suppress
   > this warning

moved to **FMM Level 1**.

## Feature Proposal

### Problem Statement

`PersonalRelationship` is a new (FMM-0, draft) resource being incubated
under the Patient Administration work group. The publisher's QA pass
emits two structural warnings against it:

- It has no element capable of expressing an `entered-in-error` /
  inactive state, which is the universal mechanism FHIR uses to retract
  data that cannot simply be deleted.
- It has no `identifier` element, which the QA report flags because most
  resources are expected to be independently identifiable.

The original ticket description observes that the resource has *no*
status-style element today, and asks whether one should be added; and
that an `identifier` is not needed because `PersonalRelationship`
exists only to associate two already-identified individuals.

### Proposed Change

Make two coordinated edits to the `PersonalRelationship` StructureDefinition
in `HL7/admin-incubator`, plus a maturity bump:

1. **Add a new top-level element `PersonalRelationship.active`** with the
   following definition:

   - Path: `PersonalRelationship.active`
   - Short: `Whether this relationship information is in active use`
   - Definition: `Whether this relationship information is in active use.`
   - Type: `boolean`
   - Cardinality: `0..1`
   - `isModifier`: `true` (with an `isModifierReason` along the lines of
     "This element is labelled as a modifier because it is a status
     element that can indicate that a relationship record should not be
     treated as valid.")
   - `isSummary`: `true` (consistent with how `Patient.active` and
     `RelatedPerson.active` are surfaced)
   - Insertion point: place `active` as the first child element of
     `PersonalRelationship`, immediately after the root element block
     and before `PersonalRelationship.source`, matching the convention
     used by `Patient`/`RelatedPerson` (where `active` sits near the top
     of the differential).

2. **Bump the resource's FMM level from 0 → 1** by editing the
   `structuredefinition-fmm` extension on the StructureDefinition
   (`<valueInteger value="0"/>` → `<valueInteger value="1"/>`). The
   resolution explicitly ties FMM 1 to applying these changes.

3. **Suppress the "All resources should have an identifier" QA warning**
   for `PersonalRelationship` rather than adding an element. The
   resolution defers any `identifier` element until concrete use cases
   appear and asks for the warning to be quieted at the QA layer. In an
   IG-publisher-driven repo this is normally done by adding an entry to
   a suppressed-messages file (see Open Questions — no such file
   currently exists in `HL7/admin-incubator` and the exact mechanism for
   this incubator's QA pipeline needs confirmation with the QA owners).

No other elements, search parameters, or value-set bindings are altered
by this ticket.

Example differential snippet to add (XML, matching the file's existing
style):

```xml
<element id="PersonalRelationship.active">
  <path value="PersonalRelationship.active"/>
  <short value="Whether this relationship information is in active use"/>
  <definition value="Whether this relationship information is in active use."/>
  <min value="0"/>
  <max value="1"/>
  <type>
    <code value="boolean"/>
  </type>
  <isModifier value="true"/>
  <isModifierReason value="This element is labelled as a modifier because it is a status element that can indicate that a relationship record should not be treated as valid"/>
  <isSummary value="true"/>
</element>
```

### Design Rationale

- **Consistency with sibling Individuals resources.** `Patient.active`,
  `RelatedPerson.active`, and `Person.active` are all `boolean 0..1`
  modifier elements with identical short text and the same intent: a
  cheap, universal way to mark a record as no-longer-valid without
  deleting it. The resolution explicitly cites this pattern. Reusing
  `boolean active` (rather than introducing a coded `status` element
  with values like `active | entered-in-error`) keeps the resource
  aligned with its closest neighbours and avoids inventing a new value
  set.
- **`isModifier=true` is required.** Anything that can flip the
  validity of all the other data on the resource must be a modifier per
  FHIR's modifier-element rules. This matches `Patient.active`'s
  treatment.
- **Defer `identifier` until there is a use case.** The resolution is
  explicit that no `identifier` element is added. Adding a
  speculative element now would create churn for downstream IGs and
  contradict the resolution; a future ticket can revisit if a real
  business identifier requirement appears. Suppressing the QA warning
  is the agreed mitigation.
- **FMM 1 is appropriate post-fix.** With a modifier `active` element
  in place the resource clears the structural QA warning that was
  blocking maturity progression; FMM 1 is the next step and is what
  the resolution authorises.

No breaking changes. The new element is `0..1` and optional, so

brand-new element; it does not retroactively affect existing
instances. The FMM bump from 0 to 1 is a maturity advance, not a
behavioural change. The QA-warning suppression is a build-pipeline
configuration change with no on-the-wire impact.

### Related Specifications
