## Resolution Summary

**Resolution:** Persuasive — *"The formating and the link will be corrected."*

The Administration Module page (`administration-module.html`) renders the
table-row description for each resource by pulling `StructureDefinition.description`
from the resource's authored XML. For the **Group** row this currently shows up as
malformed text containing visible `\r\n\r\n` escape sequences, stray `*`
characters, and a markdown link `[below](group.html#group-usage)` that is
rendered as a `<span class="error">` (i.e., the publisher rejected it). The fix
is to clean up the `<description>` value in
`source/group/structuredefinition-Group.xml` so that:

1. The `\r\n\r\n` literal escape becomes a real paragraph break (`&#10;&#10;`,
   matching the convention used by every other description in the file and the
   sibling `<definition>` value on line 86).
2. The two stray `*` characters surrounding `collectively` and `NOTE:` are
   removed (they are not intended as emphasis or as footnote markers; their
   presence is what causes the markdown parser to misinterpret the trailing
   `[below](group.html#group-usage)` link as malformed).
3. With the asterisks gone, the existing markdown link
   `[below](group.html#group-usage)` will render as a proper `<a>` to the
   `group-usage` anchor that already exists in `source/group/group-notes.xml`
   line 22 (`<a name="group-usage"> </a>`).

No publisher logic, page template, or anchor target needs to change — only
the `<description value="...">` string on line 52 of the Group
StructureDefinition.

## Feature Proposal

### Problem Statement

On the rendered Administration Module page
([administration-module.html](https://hl7.org/fhir/administration-module.html)),
the table cell that summarises the **Group** resource is broken in three ways
visible to readers:

- The text contains literal `\r\n\r\n` characters (escape sequences that were
  never decoded), instead of the intended paragraph break before `NOTE:`.
- Stray `*` characters appear immediately after `collectively` and immediately
  before `NOTE:`, with no surrounding emphasis pair. They look like an
  abandoned attempt at a footnote marker.
- The cross-link to the "Groups as Families and Households" section, written
  as the markdown `[below](group.html#group-usage)`, is rendered as
  `<b><span class="error">[below]</span>(group.html#group-usage)</b>` — i.e.,
  the publisher's markdown processor flagged it as invalid and emitted error
  markup, leaving the user with no functioning link.

The regression was introduced by commit `7ef14b8c57` (FHIR-50525, *"Group
definition is ambiguous"*), which rewrote the `<description>` value but
replaced the previous proper newline entities (`&#10;&#10;`) with a literal
`\r\n\r\n`, added the stray asterisks, and appended the markdown link in a
context where the surrounding malformed emphasis prevents it from parsing.
Prior to that commit the description was clean.

The reporter (Reinhard Egelkraut) requests that both the formatting and the
link be corrected so the Admin Module page once again renders the Group
description as readable, well-formed HTML.

### Proposed Change

Edit a single line of one file:

- **File:** `source/group/structuredefinition-Group.xml` (line 52, the
  `<description value="...">` element of the Group StructureDefinition).

Replace the current value (logically):

> Represents a defined collection of entities that may be discussed or acted
> upon collectively but which are not typically expected to act
> collectively\*. These collections are also not typically formally or legally
> recognized.\r\n\r\n\*NOTE: Group may be used to define families or households,
> which in some circumstances may act collectively or have a degree of legal or
> formal recognition. This should be considered an exception. When Group is
> used for types of entities other than Patient or RelatedPerson, the
> expectation remains that the Group will not act collectively or have formal
> recognition - use Organization if these behaviors are needed. See more
> discussion \[below\](group.html#group-usage)

with:

> Represents a defined collection of entities that may be discussed or acted
> upon collectively but which are not typically expected to act collectively.
> These collections are also not typically formally or legally
> recognized.&#10;&#10;NOTE: Group may be used to define families or
> households, which in some circumstances may act collectively or have a
> degree of legal or formal recognition. This should be considered an
> exception. When Group is used for types of entities other than Patient or
> RelatedPerson, the expectation remains that the Group will not act
> collectively or have formal recognition - use Organization if these
> behaviors are needed. See more discussion
> \[below\](group.html#group-usage).

Concretely, three textual edits inside the single `value="..."` attribute:

1. `collectively*. These` → `collectively. These` (drop the `*`).
2. `\r\n\r\n*NOTE:` → `&#10;&#10;NOTE:` (replace literal escape with two LF
   character entities and drop the `*`).
3. Add a terminating period after `(group.html#group-usage)` to match the
   sentence punctuation used in every other description sentence in the file
   (optional but recommended for prose hygiene).

No other edits in the file. The XML structure, fhirVersion, contact blocks,
mappings, and `<differential>` content (including the line-86 `<definition>`,
which is already well-formed) all remain untouched.

### Design Rationale

- **Use `&#10;&#10;`, not literal `\r\n\r\n`.** Every other multi-paragraph
  description in `source/**/structuredefinition-*.xml` — including the
  sibling `<definition>` element on line 86 of the same file — uses the
  numeric character entity `&#10;` (LF) for paragraph breaks. The publisher
  decodes `&#10;` to a real newline before passing the markdown to the
  renderer. A literal backslash-r-backslash-n sequence is just text and is
  printed verbatim. A repo-wide grep for `\\r\\n` inside
  `<description value="...">` returns this single Group occurrence, confirming
  it is an outlier rather than a pattern.
- **Remove the asterisks rather than try to preserve them.** The bracketed
  segments `*…*` would be parsed as emphasis by the markdown processor, and
  the unmatched/stray placement is what corrupts the surrounding tokens
  (including the trailing link). They have no semantic role here and their
  removal restores the meaning the original sentence had before FHIR-50525.
- **Keep the markdown link, do not rewrite it as raw HTML.** Markdown links
  inside `<description value="...">` are used and render correctly elsewhere
  in the codebase — examples include
  `source/bundle/structuredefinition-Bundle.xml`,
  `source/servicerequest/structuredefinition-ServiceRequest.xml`, and
  `source/valueset/structuredefinition-ValueSet.xml`. Once the malformed
  emphasis is gone, `[below](group.html#group-usage)` will round-trip through
  the publisher into a normal `<a href="group.html#group-usage">below</a>`.
  The destination anchor (`<a name="group-usage"> </a>` on line 22 of
  `source/group/group-notes.xml`) already exists and is unchanged.
- **Out of scope.** The ticket asks only for the Admin Module table entry to
  render correctly. It does not ask to change the wording of the description,
  the sibling `<definition>` value, the Group resource's structure, or the
  Admin Module page template. Those are deliberately not touched.
