## Resolution Summary

Resolution: **Not Persuasive with Modification**. The reporter's proposal to rename or collapse `source`/`target` into a multi-cardinality `participants` element is rejected. Instead, the resolution requires *clarifying* that PersonalRelationship intentionally models a **single uni-directional** relationship between two individuals, by:

1. **Adding a new "Notes" / use-case section** to the resource page that describes several scenarios:
   - A simple extended family with parents / brothers / sisters / grandparents.
   - A foster home and the people and relationships within it.
   - A blended family showing step-children / adopted children.
2. **Adding actual example instances** for each of the above scenarios.
3. **Adding guidance** on the resource page stating (verbatim from the resolution):
   > The PersonalRelationship resource captures a single uni-directional relationship and not multiple relationships between a set of individuals. Properties such as `confidence`, `period`, `asserter` can all be different for each of the relationships, and could have different values for the different directions.
   > Some types of relationships might infer an inverse relationship, such as brother / sibling / father / child.
4. **Adding a new search parameter `party`** which references both `PersonalRelationship.source` and `PersonalRelationship.target` (analogous to the existing `patient` SP and modelled after Patient-style "either side" SPs), so consumers can discover relationships without having to know which side is source vs. target.

No element rename, no element removal, no cardinality change to `source`/`target`.

## Feature Proposal

### Problem Statement

The PersonalRelationship resource (currently a draft, FMM 0, Base.Individuals category — see `StructureDefinition-PersonalRelationship.xml`) defines two `1..1 Reference` elements named `source` and `target`. The reporter contends the names imply directionality that is unnatural for symmetric relationships (siblings, coworkers, spouses) and forces duplicate resource instances when modelling reciprocal relationships (Maria↔José; one parent ↔ four children). The reporter proposes renaming the elements to neutral terms (`participant1/participant2`, `partyA/partyB`, …) or collapsing them into a single `participants` element with cardinality `2..*`, optionally with a per-participant `type` BackboneElement to express directionality where it is needed.

The resolution rejects this redesign because the resource is *deliberately* uni-directional — each instance describes one direction of one relationship, with its own confidence / period / asserter. The current confusion is documentation, not modelling: the spec does not yet make this design intent visible, does not enumerate the canonical use cases, ships only a single thin example (`personalrelationship-example.xml`, a wife/husband stub), and forces consumers to use both `source` and `target` SearchParameters when they really just want "any relationship involving this individual".

### Proposed Change

Make four concrete edits in `HL7/admin-incubator` (where the PersonalRelationship resource currently lives — see "Repo Context" below):

1. **Resource intro page (`input/pagecontent/StructureDefinition-PersonalRelationship-intro.xml`)** — add a new section (after "Boundaries and Relationships") titled **"Notes"** (or "Modelling guidance") that contains the verbatim guidance text from the resolution about uni-directional semantics, per-direction property values, and inverse-relationship inference. Add a second new section **"Use Case Scenarios"** with three subsections — Extended family, Foster home, Blended family — each describing the scenario in prose and linking to the example bundles below.

2. **New example instances** under `input/resources/personalrelationship/`:
   - `personalrelationship-extended-family.xml` (Bundle, type `collection`) — Patients for the parents, children, and grandparents, plus PersonalRelationship instances expressing each pairwise relationship in both directions where appropriate.
   - `personalrelationship-foster-home.xml` (Bundle) — foster parent(s), biological parent(s), foster child(ren), with a mix of `relationshipType` codings (e.g., `FTH`, `FOSTERCH`) and use of `period` to show that the foster placement is time-bounded.
   - `personalrelationship-blended-family.xml` (Bundle) — biological parent, step-parent, biological children, step-children / adopted children, demonstrating use of step / adoptive `relationshipType` codes and per-direction `confidence` / `asserter` differences.
   Each Bundle should be self-contained (Patients + PersonalRelationships) so the Notes section can link to a single artifact per scenario.

3. **Examples list (`input/resources/personalrelationship/list-PersonalRelationship-examples.xml`)** — add three new `<entry>` items, one per Bundle above, with appropriate `description` / `title` extensions, so the publisher renders them on the resource page.

4. **Search parameters (`input/resources/personalrelationship/bundle-PersonalRelationship-search-params.xml`)** — add a new `SearchParameter` entry:
   - `id`: `PersonalRelationship-party`
   - `code`: `party`
   - `type`: `reference`
   - `description`: `The individual on either side of the relationship (source or target)`
   - `expression`: `PersonalRelationship.source | PersonalRelationship.target`
   - `<extension url=".../path"><valueString value="PersonalRelationship.source,PersonalRelationship.target"/>` (matching the build-tool extension already used for `PersonalRelationship-patient`)
   - `processingMode`: `normal`
   - `targetProfile`s: `Patient`, `RelatedPerson`, `Person` (mirroring the targets on `source` / `target`).

   No change is needed to the StructureDefinition's elements; `source` and `target` remain `1..1`.

### Design Rationale

- **Preserve uni-directional design.** The resource is intentionally minimal and one-directional so that each direction can carry its own `confidence`, `period`, and `asserter`. Collapsing into a multi-cardinality `participants` element would lose the ability to model "Maria says José is her husband with high confidence since 2010" independently from José's symmetric assertion (which might have a different asserter or period). The resolution makes this explicit in prose rather than via a model change.
- **Consistency with Patient-style SPs.** Patient already exposes a "two-sided" SP (`Patient-link` etc.) and the existing `PersonalRelationship-patient` SP already concatenates `source` + `target`. Adding `party` follows the same idiom and gives implementers a stable, side-agnostic discovery path.
- **Documentation-only fix is the lowest-impact path.** The resource is FMM 0 / draft and lives in an incubator IG — heavy modelling changes would invalidate any existing implementer experiments. Adding examples and a Notes section is purely additive.
- **Inverse-relationship guidance is informative, not normative.** The resolution explicitly says some relationships *might* infer an inverse — it does not require an automatic-inverse mechanism, so no invariant or conformance text is needed.

### Step-by-Step Tasks

#### HL7/admin-incubator

1. **Add Notes section to the PersonalRelationship intro page**
   - File: `input/pagecontent/StructureDefinition-PersonalRelationship-intro.xml`
   - Action: Inside the outer wrapping `<div xmlns="http://www.w3.org/1999/xhtml">`, after the existing "Boundaries and Relationships" inner `<div>`, append a new inner `<div>` containing `<a name="notes"></a><h3>Notes</h3>` and the verbatim guidance text from the resolution: that PersonalRelationship captures a single uni-directional relationship; that `confidence`, `period`, and `asserter` may differ between the two directions; and that some relationship types (brother, sibling, father, child) imply an inverse but the inverse must be stated as a separate instance if its properties are needed.
   - Details: Use valid XHTML (`<p>`, `<ul>`/`<li>`); preserve the `xsi:schemaLocation` on the outermost `<div>`.

2. **Add Use Case Scenarios section to the PersonalRelationship intro page**
   - File: `input/pagecontent/StructureDefinition-PersonalRelationship-intro.xml`
   - Action: After the new Notes `<div>`, add another `<div>` with `<a name="scenarios"></a><h3>Use Case Scenarios</h3>` and three `<h4>` subsections — "Extended family", "Foster home", "Blended family" — each with prose describing the scenario and an inline link to the corresponding example Bundle (e.g., `<a href="Bundle-personalrelationship-extended-family.html">Extended family example</a>`).
   - Details: Bundle file IDs must match those produced by the publisher for the new example Bundles in tasks 4–6.

3. **Add the `party` SearchParameter**
   - File: `input/resources/personalrelationship/bundle-PersonalRelationship-search-params.xml`
   - Action: Add a new `<entry><resource><SearchParameter>…</SearchParameter></resource></entry>` block with `id` `PersonalRelationship-party`, `code` `party`, `type` `reference`, `expression` `PersonalRelationship.source | PersonalRelationship.target`, the build-tool `path` extension `PersonalRelationship.source,PersonalRelationship.target`, `description` "The individual on either side of the relationship (source or target)", `processingMode` `normal`, and a canonical `<url>` of `http://hl7.org/fhir/build/SearchParameter/PersonalRelationship-party`.
   - Details: Mirror the existing `PersonalRelationship-patient` entry's structural style (which already uses the same comma-separated `path` extension and a `|`-joined `expression`). No `target` element is required by the build, but if other SPs in the bundle list it, include `Patient | RelatedPerson | Person`.

4. **Create the Extended Family example Bundle**
   - File: `input/resources/personalrelationship/personalrelationship-extended-family.xml` *(new)*
   - Action: Create a `Bundle` of `type` `collection` with `id` `personalrelationship-extended-family` containing Patient resources for two parents, two children (siblings), and two grandparents, plus PersonalRelationship instances for each parent↔child, sibling↔sibling, and grandparent↔grandchild pair. Use `relationshipType` codings from `v3-RoleCode` (e.g., `FTH`, `MTH`, `SON`, `DAU`, `SIB`, `GRFTH`, `GRMTH`, `GRNDSON`/`GRNDDAU`).
   - Details: Use unique, human-readable `id` values (e.g., `pr-extfam-father-of-alice`); set `period.start` for each relationship; include at least one inverse pair (e.g., father→child *and* child→father) to demonstrate that each direction is its own resource. Set `xsi:schemaLocation` to `http://hl7.org/fhir ../../publish/Bundle.xsd`.

5. **Create the Foster Home example Bundle**
   - File: `input/resources/personalrelationship/personalrelationship-foster-home.xml` *(new)*
   - Action: Create a Bundle with foster parent(s), biological parent(s), and foster child(ren). Express both the foster relationships (with bounded `period`) and the biological parent↔child relationships (open-ended `period`). Demonstrate per-direction differences in `asserter` (e.g., the social worker asserts the foster placement; the biological parent asserts the biological relationship).
   - Details: Use the `FOSTERCH` / `FOSTERDAU` / `FOSTERSON` and reciprocal foster-parent codes from v3-RoleCode. Demonstrate `confidence` differing between directions where appropriate.

6. **Create the Blended Family example Bundle**
   - File: `input/resources/personalrelationship/personalrelationship-blended-family.xml` *(new)*
   - Action: Create a Bundle with one biological parent, one step-parent, biological children of each parent, and at least one adopted child. Express step-parent↔step-child, biological parent↔biological child, and adoptive parent↔adopted child relationships.
   - Details: Use `STPFTH`/`STPMTH`/`STPSON`/`STPDAU` and `ADOPTF`/`ADOPTM`/`ADOPTP`-style codes from v3-RoleCode. Include `period.start` for each step / adoptive relationship reflecting the date the relationship was established.

7. **Register the three new example Bundles in the examples list**
   - File: `input/resources/personalrelationship/list-PersonalRelationship-examples.xml`
   - Action: Add three new `<entry>` blocks, one per new Bundle. For each entry set the `description` and `title` build extensions and the `<item><reference value="Bundle/personalrelationship-extended-family"/>` (and equivalents for foster-home and blended-family), with a human-readable `<display>` value.
   - Details: Keep the entries in the order Extended → Foster → Blended so they appear in the same order as the Use Case Scenarios subsections added in task 2.
