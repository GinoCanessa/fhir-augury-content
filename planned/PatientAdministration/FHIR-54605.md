## Resolution Summary

The resolution is **Persuasive**. The description text on the `date`
SearchParameter for `PractitionerRole` will be updated as follows
(verbatim from the resolution description):

- **From:** `The period during which the practitioner is authorized to perform in these role(s)`
- **To:** `Date(s) within the period during which the practitioner is authorized to perform in these role(s)`

The change is a single string edit on one SearchParameter `description`
element. No structural, cardinality, type, binding, expression, or
search-parameter-code changes are required.

## Feature Proposal

### Problem Statement

The `PractitionerRole` resource exposes a search parameter named `date`
of type `date` whose `expression` targets `PractitionerRole.period`
(a `Period`). The description currently reads "The period during which
the practitioner is authorized to perform in these role(s)" — wording
that mirrors the underlying element's `<short>` rather than describing
what the search actually does. As Russell Rogers reported, when a client
supplies a date value, the server returns PractitionerRoles whose
`period` includes that date; the description should make the
date-falls-within-period semantics explicit so implementers do not
mis-read the parameter as accepting/returning the period itself.

### Proposed Change

Update the `<description>` of the `PractitionerRole-date` SearchParameter
in `source/practitionerrole/bundle-PractitionerRole-search-params.xml`
(line 54) so the SearchParameter entry reads:

```xml
<SearchParameter>
  <id value="PractitionerRole-date"/>
  ...
  <url value="http://hl7.org/fhir/build/SearchParameter/PractitionerRole-date"/>
  <description value="Date(s) within the period during which the practitioner is authorized to perform in these role(s)"/>
  <code value="date"/>
  <type value="date"/>
  <expression value="PractitionerRole.period"/>
  <processingMode value="normal"/>
</SearchParameter>
```

No other fields on this SearchParameter (`code`, `type`, `expression`,
`processingMode`, `url`, standards-status extension, build-path
extension) change.

### Design Rationale

- The wording is a verbatim quote from the work-group-approved
  resolution description, so adopting it preserves the agreed prose.
- The change is intentionally narrow: only the SearchParameter's
  `description` is rewritten. The matching `<short>` on
  `PractitionerRole.period` in
  `source/practitionerrole/structuredefinition-PractitionerRole.xml`
  (line 184) is the *element*'s short label, not search semantics, and
  the resolution does not call for it to change. Touching it would
  conflate two distinct artifacts and could regress the resource's
  element documentation.
- The SearchParameter is marked `normative` (via
  `structuredefinition-standards-status` = `normative`), but per FHIR
  policy a non-substantive description-text clarification is allowed
  without a normative break — the search semantics, code, type, and
  expression are unchanged.
