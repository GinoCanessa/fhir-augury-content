## Resolution Summary

Resolution: **Persuasive**.

Resolution description (verbatim from Jira): *"The current build will be updated to change the binding strength on `Encounter.participant.type` to Preferred."*

Concretely, the only change required is to relax the binding strength on the `Encounter.participant.type` element from `extensible` to `preferred`. The bound ValueSet (`http://hl7.org/fhir/ValueSet/encounter-participant-type`) and its description ("Role of participant in encounter.") remain unchanged.

## Feature Proposal

### Problem Statement

The `Encounter.participant.type` element today carries an **extensible** binding to `http://hl7.org/fhir/ValueSet/encounter-participant-type`. Per the reporter (Eric Haas), the codes in that ValueSet (drawn from v3 ParticipationType plus a small FHIR-defined supplement) may not be in active use by some jurisdictions or implementation communities. An extensible binding obliges implementers to use a code from the ValueSet whenever the concept is covered by it, which is too strong a constraint when local terminologies are the practical norm for naming participant roles in an encounter.

### Proposed Change

Single, surgical change in HL7/fhir:

- **File:** `source/encounter/structuredefinition-Encounter.xml`
- **Element:** `Encounter.participant.type` (element id on line 566 of the current clone HEAD `db79dcd`)
- **Edit:** within the `<binding>` block on lines 579–586, change `<strength value="extensible"/>` (line 583) to `<strength value="preferred"/>`.
- **Leave unchanged:**
  - `elementdefinition-bindingName` extension (`ParticipantType`).
  - `description`: `"Role of participant in encounter."`.
  - `valueSet`: `http://hl7.org/fhir/ValueSet/encounter-participant-type`.
  - All other element fields (`short`, `definition`, `comment`, `min`/`max`, `type`, conditions `enc-1`/`enc-2`, `isSummary`, mappings).

After the edit the binding block will read:

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="ParticipantType"/>
  </extension>
  <strength value="preferred"/>
  <description value="Role of participant in encounter."/>
  <valueSet value="http://hl7.org/fhir/ValueSet/encounter-participant-type"/>
</binding>
```

No ValueSet or CodeSystem changes, no narrative changes, no example changes, no search-parameter changes are required.

### Design Rationale

- The resolution explicitly directs a binding strength change to `preferred`. `preferred` is the standard FHIR mechanism for "we recommend these codes, but local terminologies are acceptable" — exactly the scenario the reporter describes.
- Keeping the ValueSet and the binding name (`ParticipantType`) intact preserves backwards compatibility for implementers already using the codes, and keeps the recommendation visible. Replacing or retiring the ValueSet was not requested and would be a larger, breaking change.
- Alternative `example` binding was not chosen by the work group; `preferred` retains a stronger recommendation than `example` while still permitting jurisdictional code systems.
- The change is local to one element on one resource and does not affect any other Encounter element or any other resource that references Encounter.
