Resolution: **Persuasive**. The ticket aggregates ~21 RDF/Turtle serialization
changes for R6 (some breaking, some OWL/ShEx-only, some pure documentation).
JSON and XML serializations are unaffected. The agreed change set, taken from
the ticket description, comments (notably dbooth 2025‑10‑28 13:47 and
2025‑10‑30 20:42), and the linked W3C HCLS checklist
(`w3c/hcls-fhir-rdf#176`), is:

1. Rename `fhir:link` → `fhir:l` (avoid clash with existing unrelated use of
   `fhir:link`); apply to `fhir:canonical` resources too. (`hcls-fhir-rdf#193`)
2. Add an OPTIONAL `fhir:l` link on every `fhir:Reference`, as a sibling of the
   `a fhir:Reference` type assertion. (`hcls-fhir-rdf#190`)
3. Add an OPTIONAL `fhir:l` link wherever a value has FHIR type `uri` or one
   of its subtypes (`url`, `canonical`, `uuid`, `oid`). (`hcls-fhir-rdf#192`)
4. When a `fhir:l` link is generated from a URI containing a vertical-bar
   version (e.g. `…|v123`), convert the bar into a `?version=v123` query
   string. (`hcls-fhir-rdf#191`)
5. Change object type of `fhir:div` (Narrative) from `xsd:string` to

   → `a fhir:Quantity`). Update OWL & HAPI to match what R5 already did.
   (`hcls-fhir-rdf#138`)
7. Correct semantics of hash IDs when generating a `fhir:l` link from an `id`

8. Disambiguate punned primitive types — **per dbooth 2025‑10‑28 correction**,

## Feature Proposal

### Problem Statement

In R5, FHIR's RDF/Turtle serialization, OWL ontology, and ShEx schema have a
collection of small but real defects:

- The property name `fhir:link` collides with another, unrelated `fhir:link`
  use, so it cannot continue to mean "follow this URI".
- URI-typed primitives, `Reference`, and `canonical` have no easy way to be
  followed in an RDF graph; round-tripping is faithful but RDF processors
  cannot easily traverse references.
- Pipe-delimited canonical versions (`url|version`) are not valid URI fragments
  to a generic RDF tool.
- `fhir:div` is typed as `xsd:string`, which loses the fact that it is XHTML.
- Polymorphic `value[x]` properties lack consistent `rdf:type` assertions in
  generated artifacts.
- A `fhir:l` link generated from a hash id (`#foo`) is mishandled.
- The OWL ontology has class/property name collisions ("punning") because
  primitive types share names with properties (`fhir:code` is both).
- Several smaller OWL/ShEx/ontology defects accumulated (hierarchy duplicates,
  missing `isSummary`/`isDefinedBy`, unnamed structures, profile representation,
  `w5:` prefix inconsistency, missing version/datetime stamps, ShEx for
  canonicals, etc.).
- `rdf.html` and `datatypes.html#representations` no longer accurately describe
  what the generators produce.

JSON and XML serializations are unaffected; this is an RDF/Turtle/OWL/ShEx and
documentation cleanup that lands as the R6 baseline for FHIR RDF.

### Design Rationale

- `fhir:l` (single letter) was chosen specifically to avoid the existing
  `fhir:link` collision; keeping it short keeps generated Turtle compact and
  matches the existing `fhir:v` short-name pattern for primitive values.
- Adding `fhir:l` on URI-typed primitives, `Reference`, and `canonical` makes
  generated Turtle directly traversable by generic RDF tooling without
  changing the existing `fhir:reference` / `fhir:v` round-trip-faithful form.
- Rewriting `…|v123` to `…?version=v123` produces a syntactically valid URI
  while preserving the version semantics, and it round-trips back to the
  pipe form trivially.
- `rdf:XMLLiteral` for `fhir:div` correctly types Narrative XHTML; the bnode
  standoff (item 19) makes `fhir:div` shape-consistent with every other
  primitive type, simplifying ShEx and OWL.
- Capitalizing primitive class names (item 8 — **the agreed fix, not the
  original `fhirbt:`/`fhirdt:` prefix proposal**) is the minimum intervention
  that resolves OWL punning while keeping a single `fhir:` namespace, which
  the WG considered preferable to splitting the namespace.
- Generating `rdf:type` for polymorphic properties matches what R5 already
  does for instances and finally aligns the OWL ontology and HAPI-generated
  artifacts with the documented behavior.
- The `w5:` → `fhirw5:` rename, and adding version/datetime headers to ShEx
  and ontology files, are purely conventional alignment with the rest of the
  FHIR RDF prefix family and with FHIR build provenance.

Alternatives considered (visible in comments / W3C minutes): introducing
`fhirbt:`/`fhirdt:` namespaces for primitive vs. data types (rejected
2025‑10‑21 in favour of capitalization); leaving `fhir:link` in place
(rejected because of the documented clash); explicit `a fhir:Reference` on
every reference (rejected — it adds no information and is now commented out
of the canonical example).

### Breaking Changes

Yes — this is intentionally a breaking RDF change set from R5 to R6. The
breaking changes are:

- **Property rename `fhir:link` → `fhir:l`.** Any R5 RDF consumer that
  relied on `fhir:link` will not see the property in R6 output.
- **`fhir:div` object type changes from `xsd:string` to `rdf:XMLLiteral`
  with a bnode standoff** (items 5 + 19). Consumers that parsed
  `fhir:div` as a plain string literal must update.
- **OWL primitive class names change** to be Capitalized (`fhir:Code`,
  `fhir:Uri`, …) instead of sharing names with properties (item 8).
  OWL/ShEx consumers that bound on lower-case `fhir:code` etc. as
  classes must update.
- **Turtle prefix `w5:` → `fhirw5:`** in `w5.ttl` (item 18). Any consumer
  that hard-coded the old prefix string must update; expanded IRIs do not
  change.
- **Hash-id (`#foo`) handling for generated `fhir:l`** (item 7) is a
  semantics change for previously-malformed R5 output.

The OWL-only items (9, 10, 11, 12, 13, 17, 21) are non-breaking
ontology refinements. The new OPTIONAL `fhir:l` siblings (items 2, 3) are
additive — existing consumers that read `fhir:reference`/`fhir:v` continue
to work unchanged.
