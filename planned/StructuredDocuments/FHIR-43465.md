## Resolution Summary

The resolution (Persuasive) recorded on the ticket is:

> SDWG agrees to transfer ownership of this CodeSystem and ValueSet to the OO workgroup.

Concretely, the affected artifacts are:

- **CodeSystem** `http://hl7.org/fhir/catalog-type` (id `catalog-type`)
- **ValueSet** `http://hl7.org/fhir/ValueSet/catalog-type` (id `catalog-type`)

The applied-vote follow-up comments from gdolin (2025-10-09) reiterate that ownership of the CatalogType code system and value set must move to OO, and observe in passing that "CatalogType is a profile on Composition. There is a long history on Catalog and this tracker issue was requested to be owned by OO. It should be looked at by OO to determine if it should be carried forward into R6 or not." The resolution text itself names only the `CodeSystem` and `ValueSet`, so the implementation plan below limits the work-group reassignment to those two files. The related decision about whether the `Composition` profile that uses this binding should also move (or be retired in R6) is intentionally **not** scoped into this ticket.

## Feature Proposal

### Problem Statement

The `CatalogType` CodeSystem and ValueSet currently live in `HL7/fhir` under `source/composition/` and carry the SDWG (`sd`) work-group stamp on their `structuredefinition-wg` extension. The actual users of these terminologies are the OO-owned Catalog work (the description points at FHIR-37439, which is a value-set change OO needs to make and has carried into the Catalog IG). With ownership marked as SDWG, OO cannot make that change without a hand-off, so the binding is effectively orphaned: the consumer (OO Catalog) cannot evolve the terminology that constrains it, and the nominal owner (SDWG) does not maintain it.

### Proposed Change

Update the `structuredefinition-wg` extension on the two CatalogType terminology artifacts so the publisher reports OO as the responsible work group:

1. `source/composition/codesystem-catalog-type.xml` — change `<valueCode value="sd"/>` to `<valueCode value="oo"/>` on the `structuredefinition-wg` extension at the top of the resource (currently lines 3–5).
2. `source/composition/valueset-catalog-type.xml` — change `<valueCode value="sd"/>` to `<valueCode value="oo"/>` on the `structuredefinition-wg` extension at the top of the resource (currently lines 3–5).

No other metadata on these files needs to change for the ownership transfer itself: `url`, `name`, `id`, `version`, `status`, `experimental`, `caseSensitive`, `content`, the four `concept` entries on the CodeSystem, and the single `compose.include.system` reference on the ValueSet are all unchanged. The `structuredefinition-standards-status` extension (`informative`) is unchanged, the OID assignments in `source/oids.ini` (`http://hl7.org/fhir/catalogType` = `2.16.840.1.113883.4.642.4.2013` and `http://hl7.org/fhir/ValueSet/catalogType` = `2.16.840.1.113883.4.642.3.968`) are unchanged, and the consuming binding in `source/composition/structuredefinition-profile-catalog.xml` (which references `http://hl7.org/fhir/ValueSet/catalog-type` on the Catalog profile's `type` element) does not move because the resolution does not transfer the profile.

Example (CodeSystem, after edit — abbreviated):

```xml
<CodeSystem xmlns="http://hl7.org/fhir">
  <id value="catalog-type"/>
  <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-wg">
    <valueCode value="oo"/>
  </extension>
  <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
    <valueCode value="informative"/>
  </extension>
  <url value="http://hl7.org/fhir/catalog-type"/>
  ...
</CodeSystem>
```

### Design Rationale

In FhirCore, work-group ownership is declared per-artifact via the `structuredefinition-wg` extension; this is what `source/wglist.html`'s `<%wgreport%>` directive aggregates into the published Workgroup List Report (the page itself is templated and contains no per-artifact data — see `source/wglist.html`). Updating the extension on each affected file is therefore both necessary and sufficient: there is no separate workgroup-ownership manifest in the repo to keep in sync. Surveyed OO-owned artifacts under `source/biologicallyderivedproduct/` use exactly this pattern with `<valueCode value="oo"/>`, confirming the target string.

The decision to leave `structuredefinition-profile-catalog.xml` (still `wg = sd`) untouched follows the resolution wording verbatim ("this CodeSystem and ValueSet"). The R6-carry-forward question raised in the second gdolin comment is a follow-on disposition for OO and is not actionable from this ticket; it is captured in **Open Questions** below.

## Implementation Plan

### Prerequisites

None. The change is a self-contained, two-file metadata edit and does not depend on any other ticket or any UTG/IG companion change.

### Step-by-Step Tasks

#### HL7/fhir

1. **Reassign the CatalogType CodeSystem to OO.**
   - File: `source/composition/codesystem-catalog-type.xml`
   - Action: In the `structuredefinition-wg` extension at the top of the resource, change `<valueCode value="sd"/>` to `<valueCode value="oo"/>`.
   - Details: Only the inner `valueCode` value changes; keep the surrounding `<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-wg">` element and the sibling `structuredefinition-standards-status` extension (`informative`) intact. Do not touch `id`, `url`, `version`, `name`, `title`, `status`, `experimental`, `caseSensitive`, `content`, or any of the `concept` entries.

2. **Reassign the CatalogType ValueSet to OO.**
   - File: `source/composition/valueset-catalog-type.xml`
   - Action: In the `structuredefinition-wg` extension at the top of the resource, change `<valueCode value="sd"/>` to `<valueCode value="oo"/>`.
   - Details: Only the inner `valueCode` value changes; preserve everything else, including `compose.include.system` = `http://hl7.org/fhir/catalog-type` (which still points at the renamed-owner CodeSystem above).

3. **Confirm no companion ownership manifest needs editing.**
   - File: `source/wglist.html`
   - Action: Read-only verification — confirm the page is templated (`<%wgreport%>` directive) and carries no per-artifact ownership data; no edit required.
   - Details: The bespoke FhirCore publisher aggregates `structuredefinition-wg` from each artifact at build time. This step exists as an explicit checkpoint so a later reviewer does not assume something else needs updating.
