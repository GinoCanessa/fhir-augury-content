## Feature Proposal

### Problem Statement

The `Composition.relatesTo.type` element binds (extensibly) to the `ArtifactRelationshipType` value set, which contains a large vocabulary of relationship codes. Many of those codes overlap directly with concepts that are first-class in the `Provenance` resource:

| `relatesTo.type` code | Provenance equivalent |
|---|---|
| `derived-from` | `Provenance.entity.role = source` + a derivation activity |
| `transforms` / `transformed-into` | `Provenance.entity.role = source` + a transformation activity |
| `amends` / `corrects` / `replaces` | `Provenance.entity.role = revision` |
| `retracts` | `Provenance.entity.role = removal` |
| `signs` | `Provenance.signature` (and additionally `Composition.attester`) |
| `cites` / `quotation` | `Provenance.entity.role = quotation` |

The overlap creates three concrete interoperability risks called out in the ticket:

1. The same fact can be recorded in two (or, for `signs`, three) different places with no guidance on which is authoritative, leading to inconsistent implementation.
2. `Provenance` carries the *who / when / why / which activity* context that `relatesTo` cannot express because `relatesTo` conflates the related entity with the activity that produced the relationship.
3. For signatures, a single signature could be recorded at `Composition.relatesTo.type = "signs"`, at `Composition.attester`, and at `Provenance.signature`, with no documented precedence rule.

The discussion in the comments confirmed (per John Moehrke) that the duplication is intentional — resources are allowed to replicate "Five Ws" functionality of Provenance when that functionality is important for that resource — but (per Eric Haas) this expectation is not self-evident to the average reader and is not currently called out anywhere in the Composition specification. The result is that the spec implies users have a free choice between two distinct modeling approaches without telling them how to choose.

### Proposed Change

Add a new "Composition.relatesTo and Provenance" subsection to the Composition narrative pages, with three worked sub-points and one or more examples. Concretely:

**A. New narrative subsection in the Composition specification page** (`source/composition/composition-introduction.xml`), placed inside the existing **Background and Context** `<div>` (i.e., after the **Composition Status Codes** subsection and before / after the **Note for CDA aware readers** subsection, with a new `<a name="provenance"></a>` anchor and an `<h3>` heading such as **"Composition.relatesTo, Composition.attester and Provenance"**). The subsection covers:

  1. **General principle.** State the rule explicitly (the reporter's draft is a usable starting point, subject to Lisa & Gay's wording):
     > Use `Composition.relatesTo` to record *static, bibliographic or structural* relationships between this Composition and other artifacts (FHIR or non-FHIR) — i.e., facts that are intrinsic to *what this Composition is* and that travel with the document. Use a separate `Provenance` resource when the relationship represents an *activity* that occurred at a specific time, by a specific agent, for a stated reason — i.e., facts about *how this Composition came to exist*. The two are complementary, not alternatives: a single Composition may carry both a `relatesTo` entry (e.g., `replaces` an earlier composition) **and** a `Provenance` resource that records the agent, time, and policy that authorised the replacement.

  2. **Per-code mapping table** mirroring the table in the ticket description, listing each overlapping `relatesTo.type` code (`derived-from`, `transforms` / `transformed-into`, `amends` / `corrects` / `replaces`, `retracts`, `signs`, `cites` / `quotation` concepts) alongside the Provenance element that records the same activity-flavoured fact. The table should make clear that the `relatesTo` code states *that* a relationship exists, while the matching `Provenance` records *who, when, and how*.

  3. **Signatures specifically.** Document the precedence rule between the three signature-bearing locations:
     - `Composition.relatesTo.type = "signs"` — used when *this Composition signs another artifact* (it is a relationship from this document to the signed thing). Not the place to record the signature *of this Composition*.
     - `Composition.attester` — the canonical, in-document attestation record for *this Composition*. Implementations consuming a Composition should treat `attester` as the authoritative attestation/signature container for the document itself.
     - `Provenance.signature` — used when the signature is recorded as part of an external Provenance resource (e.g., a counter-signature added later by a different agent, or a non-FHIR signature wrapper). Also the canonical home for non-FHIR document signatures (per John Moehrke's comment that "FHIR Signatures are not the only signatures in the world").

  4. **`relatesTo` lifecycle codes vs. `Composition.status`.** Address question 4 from the ticket explicitly: codes such as `amends`, `replaces`, `retracts` describe a *relationship between two Compositions*, whereas `Composition.status` values such as `amended`, `entered-in-error` describe the *state of this Composition*. They are intentionally separate axes — for example, an `amended` composition will typically also carry a `relatesTo` entry of type `amends` pointing at the prior version, but the two are not redundant: the status records what this document *is*, the `relatesTo` records *what it is in relation to*.

  5. **At least one worked example.** For instance: a Composition that supersedes a prior discharge summary, showing both the `relatesTo` snippet (as already illustrated in the existing `Composition.relatesTo` element comment) and the accompanying `Provenance` resource that records the supersession activity, agent and timestamp. The existing inline JSON snippet in `Composition.relatesTo`'s comment can be referenced from the new section to avoid duplication.

**B. Element-level cross-pointers in the Composition StructureDefinition** (`source/composition/structuredefinition-Composition.xml`). Append (do **not** replace) wording to the `<comment>` of the following elements that points readers to the new narrative section:

  - `Composition.relatesTo` (line 771, `<comment>` at line 778) — append a sentence such as: *"See [Composition.relatesTo, Composition.attester and Provenance](composition.html#provenance) for guidance on when to record a relationship here versus on a separate Provenance resource."*
  - `Composition.relatesTo.type` (line 797) — currently has no `<comment>`; add one that points to the same section and specifically calls out the overlap-with-Provenance codes.
  - `Composition.attester` (find the element block; not loaded in this plan but present in the same file) — append a sentence pointing at the same section, noting that `attester` is the canonical in-document attestation record and that `Provenance.signature` and `relatesTo.type = "signs"` serve different purposes.

  These element-level pointers keep the resolution honest for readers who are looking at the formal definition rather than the narrative page. They are **prose-only `<comment>` text**; they do not change short, definition, cardinality, type, or binding.

**C. No terminology / value-set change.** The committee's resolution explicitly chose the documentation route over the reporter's "consider removing overlapping codes" recommendation. `codesystem-artifact-relationship-type.xml`, `valueset-artifact-relationship-type.xml`, and the binding on `Composition.relatesTo.type` (extensible to `http://terminology.hl7.org/ValueSet/artifact-relationship-type`, `bindingName=ArtifactRelationshipType`) all stay exactly as they are.

### Design Rationale

- **The committee deliberately preserved the codes.** John Moehrke's comments make the rationale explicit: per the Five Ws report, resources are expected to replicate Provenance-flavoured functionality when that functionality is important to the resource's primary use, and the Provenance vocabulary is aligned with W3C PROV for standards compatibility. Removing codes would break consumers that already use them and would not actually solve the underlying problem (which is that users do not know *when* to choose which mechanism). Adding guidance addresses the real defect — missing direction — without breaking conformance.
- **Narrative subsection over global "Boundaries and Relationships" addition.** The existing **Boundaries and Relationships** section in `composition-introduction.xml` already enumerates other *grouping* structures (List, Group, Bundle, QuestionnaireResponse). Provenance is not a grouping structure; it is an orthogonal "Five Ws" record. Putting the new guidance in **Background and Context** (which already houses similar cross-cutting topics like `Composition.status` semantics and the CDA mapping note) is more natural and avoids overloading the "Boundaries" section with a different kind of comparison.
- **Element-level pointers, not full duplication.** The full guidance lives on the narrative page (one source of truth). The StructureDefinition `<comment>` updates are deliberately short pointers so that readers of the formal definition (e.g., via downstream IG publishers) are not left without the cross-link, but the narrative is not duplicated in N places.
- **Reporter's draft text is a starting point, not the final wording.** The resolution names Lisa Nelson and Gay Dolin as the authors of the prose, with committee review. The implementation tasks below treat the reporter's example sentence as a placeholder and require the actual wording to be developed by the committee owners.
