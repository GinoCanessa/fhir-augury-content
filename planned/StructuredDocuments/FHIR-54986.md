## Resolution Summary

**Resolution:** Persuasive — *"This is a technical correction where excluded codes
are listed in short. Will edit to add the guidance stated in the related tickets
(e.g. 53695)."*

The reporter observed that the value set bound to `Composition.participant.type`
(participation function codes such as `AUT`, `AUTHEN`, `CST`, `LA`, `RCT`, `SBJ`)
appears to overlap with the dedicated, named Composition elements
`.author`, `.attester`, `.subject`, and `.custodian`, and asked whether either
the codes or the dedicated elements should be removed. The Structured Documents
work group decided **not** to remove either side. Instead, this is treated as a
technical correction: the specification must clearly explain to readers that
those codes are intentionally excluded from the bound value set
(`http://terminology.hl7.org/ValueSet/fhir-clinical-doc-participant`) precisely
because they have first-class, dedicated elements on `Composition`, and that
`Composition.participant` is reserved for *other* participation roles. The
guidance to be added mirrors what is already being applied for FHIR-53695,
FHIR-54103, FHIR-54182, and FHIR-54343 — the four tickets the gdolin / Corey
Spears comment thread asked us to coordinate this fix with.

## Feature Proposal

### Problem Statement

`Composition` exposes named elements for the four most common document
participation roles — `Composition.author`, `Composition.attester`,
`Composition.subject`, and `Composition.custodian`. In addition,
`Composition.participant` (1..\*, `CodeableConcept` typed at
`.participant.type`) is bound (preferred) to
`http://terminology.hl7.org/ValueSet/fhir-clinical-doc-participant`. That value
set deliberately *excludes* the v3 participation codes that correspond to the
named elements (`AUT`, `AUTHEN`, `CST`, `LA`, `RCT`, `SBJ`), but the FHIR Core
specification does not explain this loudly enough on the resource page or on
the element itself, so implementers regularly read the original v3
ParticipationType enumeration and assume those codes are still legal here.
Reporter Ron G. Parker (Comment 12) recommended either pruning the value set
further or removing the named elements; the work group decided the spec text
itself is the gap.

The relevant excerpt from the current `differential` in
`source/composition/structuredefinition-Composition.xml` shows that
`Composition.participant.type` already carries the corrected short
("*Meaning and purpose of participation, in creation of the clinical
document*") and the binding `<description>` already states *"Author,
Authenticator, Custodian, Legal Authenticator, Record Target and Subject are
found in distinct, named fields in FHIR. Thus, these types are excluded."* —
that is the FHIR-53695 fix. What is still missing is (a) a `<comment>` on
`Composition.participant.type` that points implementers at the named elements,
and (b) parallel narrative on the resource page so a reader of
`composition.html` reaches the same conclusion without first opening the
value set.

### Proposed Change

Make a documentation-only change in `HL7/fhir`. No cardinality, type, binding,
or invariant changes.

1. **`Composition.participant.type` element** — in
   `source/composition/structuredefinition-Composition.xml`, add a `<comment>`
   to the existing `<element id="Composition.participant.type">` block (after
   `<definition>` and before `<min>`):

   ```xml
   <comment value="Use Composition.participant only for participation roles that
     are NOT covered by the dedicated Composition.author, Composition.attester,
     Composition.subject, and Composition.custodian elements. The bound value
     set http://terminology.hl7.org/ValueSet/fhir-clinical-doc-participant
     intentionally excludes the v3 ParticipationType codes AUT (Author),
     AUTHEN (Authenticator), LA (Legal Authenticator), CST (Custodian), SBJ
     (Subject), and RCT (Record Target) because each of those roles has a
     first-class, named element on Composition. Do not duplicate those roles
     here."/>
   ```

   Leave the existing `<short>`, `<definition>`, `<min>`, `<max>`, `<type>`,
   and `<binding>` blocks untouched (they reflect the FHIR-53695 fix).

2. **Composition resource introduction** — in
   `source/composition/composition-introduction.xml`, in the `<h2>Boundaries
   and Relationships</h2>` (anchor `bnr`) section, add a short paragraph
   explaining the same split between named participant elements and
   `participant`. Suggested wording (HTML, to drop in just after the bullet
   list comparing Composition with `List` / `Group` / `Bundle` /
   `QuestionnaireResponse`):

   ```html
   <p>
   Composition exposes <a href="composition-definitions.html#Composition.author">author</a>,
   <a href="composition-definitions.html#Composition.attester">attester</a>,
   <a href="composition-definitions.html#Composition.subject">subject</a>,
   and <a href="composition-definitions.html#Composition.custodian">custodian</a>
   as dedicated, named elements. The generic
   <a href="composition-definitions.html#Composition.participant">participant</a>
   element is reserved for participation roles that are NOT one of those four;
   the value set bound to <code>Composition.participant.type</code>
   (<a href="https://terminology.hl7.org/ValueSet/fhir-clinical-doc-participant">FHIR Clinical Document Participant</a>)
   intentionally excludes the v3 codes <code>AUT</code>, <code>AUTHEN</code>,
   <code>LA</code>, <code>CST</code>, <code>SBJ</code>, and <code>RCT</code>
   for that reason.
   </p>
   ```

3. **Ballot note bookkeeping** — `composition-introduction.xml` already
   contains a `ballot-note` block listing the four sibling tickets (FHIR-53695,
   -54103, -54182, -54343) plus FHIR-54986 as part of the “key changes since
   6.0.0-ballot4”, and a release-notes `<ul>` listing the same five JIRA
   numbers. Update the ballot-note prose to explicitly mention the new
   guidance comment (e.g., append: *"and added an explanatory comment on
   Composition.participant.type clarifying that AUT/AUTHEN/LA/CST/SBJ/RCT are
   excluded because Composition has dedicated author/attester/custodian/subject
   elements (FHIR-54986)."*). Keep the existing `<li>` for FHIR-54986 in the
   release-notes list — it is already present.

### Design Rationale

- The work group explicitly rejected both alternatives proposed in the
  original comment (drop the codes, or drop the elements). The named elements
  pre-date the participant backbone and are widely used by CDA-on-FHIR
  implementations; removing them is a major breaking change. Pruning the
  value set further is unnecessary because the right v3 codes are *already*
  excluded.
- The corrective fix therefore has to live in **specification text** rather
  than in the StructureDefinition shape. Per the playbook, the natural homes
  are the element's own `<comment>` (so it shows up in
  `Composition-definitions.html` and in every downstream IG that profiles the
  element) and the resource introduction page (so the answer is visible
  before a reader drills into the element).
- Mirroring FHIR-53695's pattern (which added the binding `<description>`)
  keeps all five coordinated tickets internally consistent.
- The change touches only `<differential>` content (the new `<comment>`); the
  publisher will regenerate the snapshot. Per the briefing's gotcha *"Only
  `<differential>` content in StructureDefinitions is hand-edited"*, no
  `<snapshot>` edits are needed or allowed.
