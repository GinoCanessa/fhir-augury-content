## Resolution Summary

The applied resolution is **Persuasive**. Resolution description (verbatim):

> Replace short description for Composition.participant.type with "Meaning and purpose of participation, in creation of the clinical document"

The reporter flagged a contradiction between the Composition.participant.type binding short list (`AUT | AUTHEN | CST | LA | RCT | SBJ`) shown on the resource page and the bound ValueSet (`http://terminology.hl7.org/ValueSet/fhir-clinical-doc-participant`), which explicitly **excludes** exactly those v3-ParticipationType codes. The Structured Documents WG decided the right fix is at the **short description** level: rather than enumerating codes that the ValueSet excludes (which is what produced the published `AUT | AUTHEN | CST | LA | RCT | SBJ` short string on the resource page), the short text should describe the *meaning* of the element. FHIR-54182 is one of five duplicate trackers (FHIR-53695, FHIR-54103, FHIR-54182, FHIR-54343, FHIR-54986) that were coordinated and resolved together; the implementing commit is attributed to the lead ticket FHIR-53695 with FHIR-54182 listed alongside.

## Feature Proposal

### Problem Statement

On the published FHIR R6 ballot4 Composition page, the binding row for `Composition.participant.type` advertises the short binding string `AUT | AUTHEN | CST | LA | RCT | SBJ` (Preferred). Those codes come from `http://terminology.hl7.org/CodeSystem/v3-ParticipationType`, but the bound ValueSet `http://terminology.hl7.org/ValueSet/fhir-clinical-doc-participant` *excludes* exactly that set of codes (because Author, Authenticator, Custodian, Legal Authenticator, Record Target and Subject already have first-class fields elsewhere in FHIR — see the binding `<description>` on the SD). The result is that the resource page contradicts the value set page, which is what `FHIR-54182` reports as "value set confusion".

The underlying source of the confusing short string is the legacy practice of enumerating the example/expected codes inside the element's `<short>` text. For this binding the enumerated codes are precisely the *excluded* ones, so any normalization of that short text must drop the code list and instead describe what the element *means*. The duplicate tickets (FHIR-53695, FHIR-54103) make the same point from a "short description should be corrected" angle, and FHIR-54343 makes it from the "lists codes which are excluded" angle. All five were dispositioned together as Persuasive.

### Proposed Change

A single, surgical edit to `Composition.participant.type` in the FHIR Core Composition StructureDefinition:

- **Element:** `Composition.participant.type`
- **Path:** `Composition.participant.type`
- **Cardinality:** unchanged (`0..*`)
- **Type:** unchanged (`CodeableConcept`)
- **Binding:** unchanged — strength `preferred`, valueSet `http://terminology.hl7.org/ValueSet/fhir-clinical-doc-participant`, binding name `FHIRClinicalDocParticipantVs`, binding `<description>` retained ("FHIR Clinical Document Participant Types. Author, Authenticator, Custodian, Legal Authenticator, Record Target and Subject are found in distinct, named fields in FHIR. Thus, these types are excluded.")
- **Old `<short>`:** `AUT | AUTHEN | CST | LA | RCT | SBJ` (the v3 codes that the ValueSet excludes)
- **New `<short>`:** `Meaning and purpose of participation, in creation of the clinical document`
- **`<definition>`:** kept consistent with the new short — already present in the clone as `Meaning and purpose of participation, in creation of the clinical document. The list includes: data entry person, informant, primary information recipient, and tracker. Other types are allowed.`

The companion `composition-introduction.xml` ballot-comment list is updated to credit this ticket (and the other duplicates) under the Composition introduction so the change is traceable on the rendered page.

No ValueSet or CodeSystem changes are proposed — the disposition is explicitly that the *description* was misleading, not the *binding*. The exclusion list inside `ValueSet/fhir-clinical-doc-participant` is correct and stays as-is.

Example after-edit snippet (from `source/composition/structuredefinition-Composition.xml`):

```xml
<element id="Composition.participant.type">
  <path value="Composition.participant.type"/>
  <short value="Meaning and purpose of participation, in creation of the clinical document"/>
  <definition value="Meaning and purpose of participation, in creation of the clinical document. The list includes: data entry person, informant, primary information recipient, and tracker. Other types are allowed."/>
  ...
  <binding>
    <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
      <valueString value="FHIRClinicalDocParticipantVs"/>
    </extension>
    <strength value="preferred"/>
    <description value="FHIR Clinical Document Participant Types. Author, Authenticator, Custodian, Legal Authenticator, Record Target and Subject are found in distinct, named fields in FHIR. Thus, these types are excluded."/>
    <valueSet value="http://terminology.hl7.org/ValueSet/fhir-clinical-doc-participant"/>
  </binding>
</element>
```

### Design Rationale

- **Match the binding semantics, not the excluded code list.** The bound ValueSet explicitly excludes the v3-ParticipationType codes that previously formed the short string, so listing them in the short description is actively misleading to implementers. Removing the code list and describing the element's meaning aligns with the disposition adopted by SDWG.
- **Lowest-risk fix.** Cardinality, type, binding strength, and binding URL are unchanged; only descriptive metadata changes. This avoids any conformance impact for resources already validated against the binding.
- **Consistency with adjacent elements.** `Composition.participant.function` is already described as "Exact function of the participant in the creation of the clinical document"; the new `Composition.participant.type` short ("Meaning and purpose of participation, in creation of the clinical document") parallels that wording, making the type/function distinction clearer on the rendered page.
- **One disposition for five duplicate trackers.** SDWG coordinated FHIR-53695, FHIR-54103, FHIR-54182, FHIR-54343, FHIR-54986 (per JIRAUSER18251's 2026-01-22 comment on this ticket). Implementing this change once under the lead ticket FHIR-53695 closes all five.
