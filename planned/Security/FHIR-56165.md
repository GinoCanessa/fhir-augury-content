## Resolution Summary

**Resolution:** Persuasive — *"add the example as given"*.

The applied resolution is to import the R6 *Provenance of a Patient Merge*
example authored by John Moehrke in his Corrections IG
(`https://build.fhir.org/ig/JohnMoehrke/correction/branches/R6/index.html#merge-patient`)
into the FHIR Core Provenance examples set. Per the ticket, the **R6** form
of the example is the authoritative one to import because it follows the
R6-CI rule that `Provenance.target` may point at resources that have been
removed (as is the case for the source patient of a merge); the R4 form
exists but does not exercise that rule and is **not** what is being added.

## Feature Proposal

### Problem Statement

FHIR Core Provenance has examples for create / delete / signature / consent
/ correction-replacement workflows, but no example covering the
**Patient `$merge`** operation. A discussion in the Patient Administration
WG Zulip channel (Lloyd McKenzie / Elliot Silver / John Moehrke,
[`#179254 > Provenance of a Patient Merge`](https://chat.fhir.org/#narrow/channel/179254-patient-administration-WG/topic/Provenance.20of.20a.20Patient.20Merge))
established that Provenance is the right artifact for tracking a
`$merge`, and an example already exists in the Corrections IG. Without a
worked example in core, implementers using the new Patient `$merge`
operation have no reference for how to record the merge as Provenance,
and in particular how to handle the source Patient (which is removed by
the merge) on the `Provenance.target` reference.

### Proposed Change

Add **one new Provenance example instance** to FHIR Core that records the
provenance of a Patient `$merge`:

- **Resource type:** `Provenance`
- **Logical id:** `example-patient-merge` (matches the existing
  `provenance-example-<scenario>` pattern used for
  `example-correction-replacement`, `example-delete`, etc.).
- **File:** `source/provenance/provenance-example-patient-merge.xml`
  — XML instance copied as-given from the R6 build of
  `JohnMoehrke/correction` (`#merge-patient` section).
- **Examples list registration:** add a new `<entry>` in
  `source/provenance/list-Provenance-examples.xml` pointing at
  `Provenance/example-patient-merge`, with a short description and a
  display title (e.g., *"Provenance of a Patient Merge"*).

The example is expected to exhibit the following shape (consistent with
the resolution text and the R6 source):

- `Provenance.target[]` — references both the surviving Patient and the
  source Patient that was removed by the merge. This is the specific R6
  behaviour the ticket flags ("`.target` pointing at resources that are
  removed too").
- `Provenance.recorded` — timestamp of the merge.
- `Provenance.activity` — coded activity for the merge (per the source
  example in the Corrections IG).
- `Provenance.agent` — the actor that performed the merge.
- `Provenance.patient` — the surviving Patient.
- `Provenance.entity` — the source / merged-away Patient with the
  appropriate `role` (e.g., `removal`) consistent with how
  `provenance-example-correction-replacement.xml` already uses
  `<role value="removal"/>` for the historical resource.

The implementer must use the **R6** form from the corrections IG verbatim
(the resolution is "add the example **as given**"); this plan does not
re-author the instance content.

### Design Rationale

- The Security WG owns Provenance and has accepted the proposal as
  *Persuasive*, with the resolution explicitly directing reuse of the
  existing IG example rather than authoring new content.
- The R6 form is preferred because R6 is the active CI build for FHIR
  Core, and R6's relaxed rule on `Provenance.target` referencing
  no-longer-existing resources is exactly what makes the merge example
  valid in core. Importing the R4 form would require either dropping the
  removed-target reference or violating R4 rules — both worse than using
  the R6 form directly.
- Adding the example follows the established pattern in
  `source/provenance/` of one XML file per example plus one entry in
  `list-Provenance-examples.xml` (e.g.,
  `provenance-example-correction-replacement.xml`,
  `provenance-example-delete.xml`,
  `provenance-example-create-consent.xml`).

### Step-by-Step Tasks

#### HL7/fhir

1. **Create the new Provenance example instance.**
   - File: `source/provenance/provenance-example-patient-merge.xml`
     *(new)*
   - Action: Copy the R6 *Patient Merge* Provenance example from
     `https://build.fhir.org/ig/JohnMoehrke/correction/branches/R6/index.html#merge-patient`
     verbatim into a new XML file using the same authoring conventions
     as the existing sibling files (e.g.,
     `provenance-example-correction-replacement.xml`,
     `provenance-example-delete.xml`).
   - Details:
     - `Provenance.id` = `example-patient-merge`.
     - Preserve the R6 example's use of `Provenance.target` referencing
       both the surviving Patient and the removed source Patient — this
       is the specific R6 behaviour called out in the ticket and must
       not be edited away.
     - Use existing v3-ActReason / provenance-participant-type /
       provenance-entity-role coded values consistent with the R6
       example; do **not** introduce new codes.
     - Do **not** add any `<snapshot>`-style or generated content; this
       file is purely hand-authored example data.

2. **Register the new example in the Provenance examples manifest.**
   - File: `source/provenance/list-Provenance-examples.xml`
   - Action: Append a new `<entry>` after the existing
     `provenance-example-correction-replacement` entry, following the
     same shape used by every entry in the file.
   - Details: The new entry should contain:
     - `<extension url="http://hl7.org/fhir/build/StructureDefinition/description">`
       with a `<valueString>` such as
       *"Provenance of a Patient `$merge` (R6 form: target references the
       removed source Patient)."*
     - `<extension url="http://hl7.org/fhir/build/StructureDefinition/title">`
       with `<valueString value="provenance-example-patient-merge"/>`
       (this title must equal the file stem, matching the convention
       used by every other entry in the manifest).
     - `<item><reference value="Provenance/example-patient-merge"/>`
       `<display value="Provenance of a Patient Merge"/></item>`.
