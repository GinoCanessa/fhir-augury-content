## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"will change to 0..1"*.

The committee accepted the reporter's point: declaring the Confidentiality
tag set as `1..1` on the Security Classification table in
`security-labels.html` is misleading and, taken at face value, would imply
that any resource lacking a `meta.security` Confidentiality tag is invalid
against the base specification. The agreed change is to relax the stated
cardinality of the Confidentiality tag set from `1..1` to `0..1`.

The assignee's comment (John Moehrke, 2026-02-19) clarifies the underlying
intent inherited from the HL7 Healthcare Privacy and Security Classification
System (HCS): *if any codes appear in `meta.security`, then at least a
Confidentiality code should be present.* That nuance is **not** being
encoded as a constraint in the resolution — the approved fix is purely the
cardinality drop to `0..1`. Any further expression of the conditional
"if other security codes are present, a Confidentiality code is required"
behaviour is out of scope for this ticket.

## Feature Proposal

### Problem Statement

The page `source/security-labels.html` documents how FHIR uses
HCS-derived security labels on `Resource.meta.security`. The
*Security Classification* table (immediately below the prose introducing
the HCS-derived classification) lists a single tag set, **Confidentiality**,
with a stated cardinality of `1..1`:

```
| Tag Set        | Car.   | Description                          | Example Tags                       |
| Confidentiality | 1..1  | Security label metadata classifying… | Unrestricted, Normal, Very Restricted |
```

(See `source/security-labels.html` lines 349–366 in clone HEAD
`db79dcd`.)

Read literally, `1..1` says every FHIR resource must carry exactly one
Confidentiality security label. That is not the case in practice or
elsewhere in the specification: `Resource.meta.security` itself is `0..*`
and the FHIR base does not require a Confidentiality tag on every resource.
The original ticket calls this out as both confusing and, if interpreted
strictly, "crazy" — it would invalidate the vast majority of conforming
resources.

### Proposed Change

Single-cell prose edit in `source/security-labels.html`:

- In the *Security Classification* table (the table whose header row is
  `Tag Set | Car. | Description | Example Tags` and whose only data row
  is the `Confidentiality` row), change the value of the `Car.` cell from
  `1..1` to `0..1`.
- Concretely, replace `<td><code>1..1</code></td>` on line 361 with
  `<td><code>0..1</code></td>`. No other cell in the row, and no
  surrounding prose, changes.

After the edit, the row reads:

```
| Confidentiality | 0..1 | Security label metadata… | Unrestricted, Normal, Very Restricted |
```

This brings the documented cardinality of the Confidentiality tag set
into alignment with the other tag sets on the same page (Policy is
`0..1`; Sensitivity, Compartment, Integrity are `0..*` — see
`source/security-labels.html` lines 381, 387, 393, 399).

No StructureDefinition, ValueSet, CodeSystem, search parameter, or
example needs to change. `Meta.security` itself is unaffected — it
remains `0..*` and bound to the All Security Labels value set by its
existing definition; only the documented per-tag-set cardinality on the
HCS table is being relaxed.

### Design Rationale

- **Faithful to the resolution.** The resolution text is a literal
  instruction (`"will change to 0..1"`), so the implementation is the
  minimum edit that satisfies it.
- **Internally consistent.** The other rows in the *Security Classification*
  / *Security Category* tables already use `0..1` or `0..*`. Leaving
  Confidentiality at `1..1` was an outlier inherited from HCS that did
  not survive review.
- **Avoids false conformance constraint.** `1..1` on a documentation
  table reads as a base-spec constraint to many implementers. Dropping
  to `0..1` removes that misreading without losing the meaning that, when
  present, a resource carries at most one Confidentiality classification.
- **Defers the conditional rule.** Encoding the assignee's intent
  (*"if `meta.security` is populated at all, a Confidentiality code
  should be present"*) would require either an invariant on `Resource`
  or new prose with a SHALL/SHOULD statement. The committee chose not
  to adopt that here; reopening it would need a new ticket.
