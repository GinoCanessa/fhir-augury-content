## Resolution Summary

Resolution: **Persuasive**. Resolution description: *"Remove the hyperlinks"*.

The bullet list on the FHIR security page contains two list items whose link
targets do not resolve to anything on the page — the bullets for **"Input
Validation"** and **"When using OAuth"** are wrapped in `<a href="#input">` and
`<a href="#oauth">` respectively, but no `<a name="input">` or `<a
name="oauth">` anchor exists anywhere in `source/security.html`. The agreed
fix is to drop the broken hyperlinks while leaving the bullet text and its
surrounding prose unchanged.

## Feature Proposal

### Problem Statement

`source/security.html` (the published `security.html` page of the FHIR Core
specification) presents a bullet list of security considerations. Two of those
bullets currently link to in-page anchors that do not exist:

- Line 47: `<li><a href="#input">Input Validation</a> - Validate all input
  …</li>`
- Line 48: `<li><a href="#oauth">When using OAuth</a> - Consider the draft
  <a href="…">Best-Current-Practice for OAuth</a></li>`

A grep of `source/security.html` for `name="input"`, `name="oauth"`,
`id="input"`, and `id="oauth"` returns no matches — the only `<a name=…>`
anchors on the page are `general`, `access-control`, `SecPrivConsiderations`,
`Anonymous`, `Business`, `Individual`, `Patient`, `Unknowable`, `http`,
`authentication`, `binding`, `AuthZmethods`, `AccessDenied`, `audit`,
`attachments`, `labels`, `narrative`, and `stylesheets`. Clicking either of
the two bullet links therefore jumps to nothing and the URL fragment dangles,
which is a user-visible bug confirmed by the ticket reporter (and shown in
the screenshot attached to the ticket).

The other bullets in the same list (e.g., `#http`, `#authentication`,
`#binding`, `#audit`, `#attachments`, `#narrative`) all do resolve to real
anchors on the page. The ticket asks specifically that the two broken links
be removed rather than back-filled with new sections.

### Proposed Change

Edit `source/security.html` to **remove the two broken `<a href>` wrappers**
around "Input Validation" and "When using OAuth". The bullet text, the
surrounding `<li>…</li>` markup, the trailing prose after each label, and any
*other* hyperlinks on those bullets (e.g., the `<a
href="https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics">Best-Current-Practice
for OAuth</a>` link on bullet 48) MUST stay intact.

Concretely:

- **Line 47 — Input Validation bullet.** Change

  ```html
  <li><a href="#input">Input Validation</a>  - Validate all input received from other actors to assure the data is well formed and does not contain content that would cause unwanted system behavior. Testing ensures that the input is not susceptible to data input validation errors by using techniques such as fuzzing, invalid input attacks, and injection attacks.</li>
  ```

  to

  ```html
  <li>Input Validation  - Validate all input received from other actors to assure the data is well formed and does not contain content that would cause unwanted system behavior. Testing ensures that the input is not susceptible to data input validation errors by using techniques such as fuzzing, invalid input attacks, and injection attacks.</li>
  ```

- **Line 48 — When using OAuth bullet.** Change

  ```html
  <li><a href="#oauth">When using OAuth</a> - Consider the draft <a href="https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics">Best-Current-Practice for OAuth</a></li>
  ```

  to

  ```html
  <li>When using OAuth - Consider the draft <a href="https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics">Best-Current-Practice for OAuth</a></li>
  ```

No other lines in `source/security.html` need to change. No other files in
`HL7/fhir` need to change (the `#input` and `#oauth` fragments are not
referenced elsewhere in the clone — see the validation step below).

### Design Rationale

- The reporter / assignee (John Moehrke, Security WG co-chair) explicitly
  requested **removal** of the hrefs, not the addition of new anchored
  sub-sections. The Security page already covers Input Validation and OAuth
  topics in narrative form further down the page (see the "When using OAuth"
  guidance around lines 226–339 and the `narrative.html#security` /
  `auditevent.html` references in the Narrative section), so adding a new
  back-targeted anchor is unnecessary and would expand scope beyond the
  resolution.
- Unwrapping the `<a>` while keeping the bullet text avoids changing the
  visible label, the surrounding sentence, or the overall document
  structure — which is the minimum-impact way to honor the resolution.
- The other `<a href="#…">` bullets in the same list still point at real
  anchors and are therefore left untouched.
