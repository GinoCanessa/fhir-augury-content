## Resolution Summary

The resolution is **Persuasive with Modification**. Two distinct actions are required:

1. **Move the Break-the-Glass implementation guidance (current section 6.1.1.3 "Break the Glass" in `source/security-labels.html`) to the Security Principles page (`source/security.html`).** The reporter's concern is that the BTG examples in 6.1.1.3.1 ("using OAuth", showing a UDAP `hl7-b2b` extension) and 6.1.1.3.2 ("using http Category", showing an HTTP `Category:` header) describe transport/authorization-layer mechanisms rather than the FHIR-Core `meta.security` approach introduced in 6.1.1.1, so they belong with general security principles, not with the FHIR security-label representation page.
2. **On the UDAP citation specifically**, when the BTG content is moved/retained:
   - Use the IG's proper current name: **"FAST Security for Scalable Registration, Authentication, and Authorization"** (UDAP was renamed to FAST Security).
   - Replace the existing version-specific link `https://hl7.org/fhir/us/udap-security/STU1.1/b2b.html#b2b-authorization-extension-object` with the non-version-specific build link `https://build.fhir.org/ig/HL7/fhir-udap-security-ig/b2b.html`.

The reporter's original observation that 6.1.1.1 ("Representing Security Labels", which uses `meta.security`) is internally consistent should be preserved — that section is **not** moved.

## Feature Proposal

### Problem Statement

In the current FHIR Core specification, `source/security-labels.html` contains:

- §6.1.1.1 *Representing Security Labels* — explains that FHIR security labels are carried in `Resource.meta.security`.
- §6.1.1.3 *Break the Glass* — defines the BTG security label, then immediately follows with two implementation examples (§6.1.1.3.1 OAuth/UDAP, §6.1.1.3.2 HTTP `Category` header) that do **not** demonstrate `meta.security` and instead show transport-layer / authorization-layer signalling.

This is internally inconsistent within the *Representing Security Labels* page, which the reporter (EssienGe) flagged: a reader of `security-labels.html` reasonably expects the examples to mirror §6.1.1.1's `meta.security` pattern. The mismatch is doubly confusing because:

- The OAuth example references HL7 UDAP Security, which has since been renamed to "FAST Security for Scalable Registration, Authentication, and Authorization" — the cited IG name is no longer current.
- The cited UDAP link is version-pinned (`STU1.1/...`) rather than the canonical non-version-specific build URL, so it is brittle as the IG evolves.
- The reporter could not locate the specific UDAP example anchor in the referenced specification, suggesting the link target may already be stale.

### Proposed Change

Two coupled documentation edits in `HL7/fhir`, both in the bespoke-publisher authored HTML pages under `source/`:

**A. Relocate the BTG implementation-method discussion** out of `source/security-labels.html` and into `source/security.html` ("FHIR Security" / Security Principles).

In `source/security-labels.html` (current lines ~258–336), the block to move begins at:

```html
<a name="break-the-glass"></a>
<h2>Break the Glass</h2>
```

and runs through the closing `</div>` of the HTTP `Category:` example (line ~336), immediately before the next `<a name="hcs"></a>` block. Per the resolution this entire BTG section — including the `[%dragons-start%]…[%dragons-end%]` block, the `<h3>using OAuth</h3>` subsection, and the `<h3>using http Category</h3>` subsection — moves to `source/security.html`, where it is appended into the Security Principles page (logical home: between the existing `<a name="audit"></a>` *Audit Logging* section and `<a name="attachments"></a>` *Attachments*, or alternatively at the end of the page just before the `<a name="labels"></a>` *Security Labels* cross-reference; see Open Questions). The relocated heading should remain `<h2>Break the Glass</h2>` with the `<a name="break-the-glass"></a>` anchor preserved unchanged so existing inbound links continue to resolve once they are repointed to `security.html#break-the-glass`.

In `source/security-labels.html`, leave §6.1.1.1 (`<h2>Representing Security Labels</h2>`) untouched — this is the page's stable, FHIR-Core-aligned guidance and the reporter is satisfied with it. The BTG **security-label code itself** (the `v3-ActReason#BTG` table row at lines ~268–276) is part of the generic security-label catalog and may either travel with the BTG section to `security.html` or remain on `security-labels.html` as a labelled-purpose-of-use entry. The cleanest reading of the resolution ("move 6.1.1.3 to the Security Principles page") is to move the entire `<a name="break-the-glass"></a>…</h2>` block with its descriptive `<table class="grid">`, then in `security-labels.html` add a one-paragraph stub that points to the new location (e.g., `<p>See <a href="security.html#break-the-glass">Break the Glass</a> on the Security Principles page.</p>`) so the section number does not silently disappear.

**B. Update the UDAP citation** in the relocated `<h3>using OAuth</h3>` subsection (currently `source/security-labels.html` line 299):

- Change the link text from `HL7 UDAP Security` to `FAST Security for Scalable Registration, Authentication, and Authorization`.
- Change the `href` from `https://hl7.org/fhir/us/udap-security/STU1.1/b2b.html#b2b-authorization-extension-object` to `https://build.fhir.org/ig/HL7/fhir-udap-security-ig/b2b.html` (drop the version segment and the `#b2b-authorization-extension-object` fragment, per the resolution's non-version-specific guidance).
- Verify that the `hl7-b2b` extension example (lines 300–313) is still present in that IG; if it has been removed/renamed, John Moehrke's first comment authorises substituting the equivalent **IHE-IUA** PurposeOfUse extension example instead. (Tracked as an Open Question — do not silently switch without a check.)

### Design Rationale

- **Consistency within a page is itself a FHIR Core editorial principle.** `source/security-labels.html` is scoped to *how FHIR represents security labels* (i.e., `meta.security`). Implementation-layer mechanisms for *requesting* a labelled access (OAuth scopes, HTTP headers) belong with general security guidance, not with the label-representation reference. This is exactly the reporter's argument and the WG accepted it.
- **Security Principles (`source/security.html`) already hosts cross-cutting how-to material** (Authentication, Authorization, Audit Logging, Attachments). BTG's "how do I signal this on the wire" guidance fits this page's existing rhythm — and the Security Labels page can link back via the existing `<a name="labels"></a>` anchor.
- **Preserving the `<a name="break-the-glass"></a>` anchor** (just under a different page) minimises link rot. A short forwarding stub on `security-labels.html` further protects bookmarks while the spec re-renders.
- **Non-version-specific UDAP/FAST link** matches FHIR Core's standing practice for citing companion IGs that ballot independently — STU-pinned URLs become incorrect at the next IG release.
- **Considered alternative — delete 6.1.1.3.1/6.1.1.3.2 outright** (as raised in John Moehrke's 2026-01-27 comment): rejected by the resolution. The WG chose to *relocate and refresh* rather than *remove*, so the implementation guidance is preserved for readers but in the correct page.
