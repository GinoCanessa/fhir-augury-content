## Problem Statement

FHIR-27164 (Patient Administration) added guidance on representing user identities in FHIR resources to the **Administration Module** page (`administration-module.html`). As a companion action, PA recommended that the **Security** work group update its own **Security & Privacy Module** page (`secpriv-module.html#user`) with:

1. A new use-case paragraph on *how to represent user identity in FHIR Resources*.
2. Guidance that user identity SHALL be communicated in the `.identifier` element, not `.telecom`.
3. A note about the impact of duplicating the user identifier across Person/Patient/Practitioner/RelatedPerson.
4. A cross-reference back to the PA "Representing User Identities" section in `administration-module.html`.

The "Persuasive with Modification" resolution incorporates David Pyke's suggestion to add the word "also" (i.e., "…email address that may **also** be recorded in .telecom…").

## Resolution Description (Approved Text)

The approved edit adds a new use-case block **before** the existing "Access to protected Resources…" use case in the "User Identity and Access Context" section (§6.0.5.2):

> **_Use case: "How to represent User identity in FHIR Resources"_**
>
> _The user identity shall be communicated in the business .identifier element. Where the user identifier has other uses, such as an email address that may **also** be recorded in .telecom, these replications of the user identity are expected, but the formal user identity mapping shall be found in .identifier. For encoding guidance see [Administrative Page – Privacy and Security section]()_

The `[Administrative Page – Privacy and Security section]()` placeholder link must resolve to `administration-module.html#representing-user-identities` (or the appropriate anchor).

Add a new "How to represent User identity in FHIR Resources" use-case paragraph to the **User Identity and Access Context** section of the Security & Privacy module page. ## Detailed Implementation Plan

### Change 1: Insert user-identity use-case in `source/secpriv-module.html`

**File:** `source/secpriv-module.html` (HL7/fhir)  
**Location:** Lines 204–214, inside the `<a name="user"></a>` section, **after** the `<h4>` heading and **before** the existing "Access to protected Resources…" use case.

**Current content (lines 204–214):**

```html
<a name="user"></a>
<h4>
User Identity and Access Context
</h4>

<p>

<em>Use case:</em> "Access to protected Resources are enabled though user Role-Based, 
Context-Based, and/or Attribute-Based Access Control."

</p>
```

**Proposed content:**

```html
<a name="user"></a>
<h4>
User Identity and Access Context
</h4>

<p>
<em>Use case:</em> "How to represent User identity in FHIR Resources"
</p>

<p>
The user identity shall be communicated in the business <code>.identifier</code> element.
Where the user identifier has other uses, such as an email address that may also be
recorded in <code>.telecom</code>, these replications of the user identity are expected,
but the formal user identity mapping shall be found in <code>.identifier</code>.
For encoding guidance see the
<a href="administration-module.html#user-identities">Administration Module – Representing User Identities</a>
section.
</p>

<p>

<em>Use case:</em> "Access to protected Resources are enabled though user Role-Based, 
Context-Based, and/or Attribute-Based Access Control."

</p>
```

**Key decisions:**

1. **Anchor link target:** The PA page currently has `<h4>Representing User Identities</h4>` (line 277 of `administration-module.html`). There is no explicit `<a name="...">` anchor before that heading. The implementation should **either**:
   - Add an anchor `<a name="user-identities"></a>` before line 277 in `administration-module.html`, **or**
   - Use the auto-generated heading anchor if the FHIR build toolchain creates one.
   - The implementer should verify which anchor format the build system uses and adjust the link accordingly.

2. **"also" modification:** Per David Pyke's comment (the basis for "Persuasive with Modification"), the word "also" is included: "…email address that may **also** be recorded in .telecom…"

3. **HTML formatting:** The new text uses `<code>` tags for `.identifier` and `.telecom` to match the formatting conventions used elsewhere in the Security & Privacy module page.

### Optional: Verify/Add anchor in `source/administration-module.html`

**File:** `source/administration-module.html` (HL7/fhir)
