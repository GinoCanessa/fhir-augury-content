## Resolution Summary

The resolution is **Persuasive with Modification**. The Security work group is to amend the *User Identity and Access Context* section of the Security and Privacy module (rendered as §6.0.5.2 at https://build.fhir.org/secpriv-module.html#user) by inserting a new "Use case" block immediately above the existing *"Access to protected Resources are enabled though user Role-Based, Context-Based, and/or Attribute-Based Access Control."* use case.

The exact text approved by the resolution (with David Pyke's PA-suggested modification of "may **also** be recorded in" applied) is:

> **Use case: "How to represent User identity in FHIR Resources"**
>
> The user identity shall be communicated in the business `.identifier` element. Where the user identifier has other uses, such as an email address that may **also** be recorded in `.telecom`, these replications of the user identity are expected, but the formal user identity mapping shall be found in `.identifier`. For encoding guidance see [Administrative Page - Privacy and Security section](administration-module.html#secpriv).

The placeholder `[Administrative Page - Privacy and Security section]()` in the resolution must be filled in with a working link to the matching PA-side section that was authored under FHIR-27164 (the "Representing User Identities" subsection inside the Administration module's Security and Privacy section, anchor `#secpriv` in `administration-module.html`).

## Feature Proposal

### Problem Statement

The Security and Privacy module (`secpriv-module.html`) does not currently tell implementers *how* to represent a user identity in FHIR resources. Patient Administration revised its own guidance under FHIR-27164 (the "Representing User Identities" subsection in `administration-module.html#secpriv`), which states that systems MAY use `.identifier` on Patient/RelatedPerson/Practitioner/Person, MAY duplicate that identifier across related Person/etc. resources, and MUST NOT use `.telecom` to carry user identity. The Security module has no parallel statement and no pointer back to the PA guidance, so a reader of the Security module is left without the canonical answer to "where does the user identifier live?" and risks duplicating or relocating it inconsistently. PA's 28 Jan 2025 WGM (per the ticket description) explicitly recommends Security add this guidance and at minimum link back to the PA section that was revised by FHIR-27164.

### Proposed Change

Edit `source/secpriv-module.html`, in the `<a name="user"></a>` / *User Identity and Access Context* `<h4>` section (currently at lines 204–238 of the cached clone at `db79dcd`), to insert a new "Use case" paragraph block **before** the existing first use case (`<em>Use case:</em> "Access to protected Resources..."` at line 211).

Concretely, immediately after the closing `</h4>` on line 207 and before the existing `<p>` on line 209, insert:

```html
<p>
<em>Use case:</em> "How to represent User identity in FHIR Resources"
</p>
<p>
The user identity shall be communicated in the business <code>.identifier</code>
element. Where the user identifier has other uses, such as an email address
that may <em>also</em> be recorded in <code>.telecom</code>, these replications
of the user identity are expected, but the formal user identity mapping shall
be found in <code>.identifier</code>. For encoding guidance see
<a href="administration-module.html#secpriv">Administrative Page - Privacy and
Security section</a>.
</p>
```

Notes on the wording:

- The phrasing "shall" / "shall be" is taken verbatim from the resolution text. It is normative-sounding; if the work group ultimately wants to soften it to "SHOULD" to match the rest of the section, that should be raised as an open question rather than silently changed.
- "may **also** be recorded in" reflects David Pyke's accepted PA modification (comment of 2025-05-15).
- The bracketed placeholder in the resolution (`[Administrative Page - Privacy and Security section]()`) is wired up to `administration-module.html#secpriv`, which is the anchor for the *Security and Privacy* `<h3>` block that contains the PA-authored *Representing User Identities* `<h4>` (see `source/administration-module.html` lines 253–286 — the section authored under FHIR-27164).
- John Moehrke's 2026-02-19 comment ("are the identified Resources needing to be updated too?") is a self-asked question. The resolution text does **not** mandate any change to Patient / RelatedPerson / Practitioner / Person / Person resources themselves; their handling lives in the PA section that this update points to. That question is captured under *Open Questions* below.

No other parts of `secpriv-module.html` are touched, and no other files in the publisher are touched.

### Design Rationale

- The Security and Privacy module is the natural home for guidance about *how user identity is carried in security/access contexts*; the PA module is the natural home for *resource modelling guidance*. The chosen approach (a brief normative pointer in Security, with the resource-level detail living in PA) avoids duplicating the PA guidance and avoids the two pages drifting out of sync — exactly the risk the reporter raised about "duplicating the user identifier property between person/patient,etc".
- Cross-linking to `administration-module.html#secpriv` rather than re-stating the PA enumeration keeps a single source of truth (the work performed under FHIR-27164) and means future PA updates automatically apply.
- Inserting the new use case **before** the existing Role/Attribute/Context-Based Access Control use case follows the order in the resolution snippet.
- The wording is intentionally aligned with the PA section it points to: PA's *Representing User Identities* bullets state "DO NOT use .telecom properties to represent user identities" and "If an e-mail is used as both a user identifier and as a method of communication, then you'd put the same e-mail in both spots (identifier and telecom)" — the new Security paragraph restates the same rule (`.identifier` is canonical; replication into `.telecom` is allowed when the value also serves a non-identity purpose) without contradicting it.
