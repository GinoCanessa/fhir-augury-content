## Resolution Summary

**Resolution: Persuasive** (resolution description: "yes").

The Security WG accepted the reporter's recommendation as a drop-in change.
The FHIRPath expression of the `Provenance-when` SearchParameter must be
updated from:

```
(Provenance.occurred.ofType(dateTime))
```

to:

```
Provenance.occurred.ofType(dateTime) | Provenance.occurred.ofType(Period)
```

so that the `when` search parameter matches Provenance resources that use
either the `occurredDateTime` or `occurredPeriod` choice of the
`Provenance.occurred[x]` element. The reporter verified the new
expression on fhirpath-lab (HAPI-R6 engine) against both choice
variants; John Moehrke's only question in comments was whether the
proposed expression was drop-in, which was confirmed before the
Persuasive resolution.

## Feature Proposal

### Problem Statement

`Provenance.occurred[x]` is a choice between `dateTime` and `Period`
(see `source/provenance/structuredefinition-Provenance.xml`). The
normative `when` SearchParameter defined for Provenance, however, only
walks the `dateTime` branch of that choice:

- File: `cache/github/repos/HL7_fhir/clone/source/provenance/bundle-Provenance-search-params.xml`
- Lines 247–264 (`SearchParameter/Provenance-when`):
  - `<expression value="(Provenance.occurred.ofType(dateTime))"/>` (line 261)
  - `path` build extension: `<valueString value="Provenance.occurredDateTime"/>` (line 254)

Consequence: any Provenance instance that records *when* using
`occurredPeriod` is invisible to the standard `Provenance?when=...` date
search — a silent correctness gap for servers and clients that follow
the Provenance guidance allowing either choice.

### Proposed Change

Update `SearchParameter/Provenance-when` so the search covers both
branches of `Provenance.occurred[x]`:

1. **Expression** (line 261): replace

   ```xml
   <expression value="(Provenance.occurred.ofType(dateTime))"/>
   ```

   with

   ```xml
   <expression value="Provenance.occurred.ofType(dateTime) | Provenance.occurred.ofType(Period)"/>
   ```

   (No surrounding parentheses — the resolution quotes the expression
   without them; FHIRPath does not require them.)

2. **`path` build extension** (lines 253–255): extend the single-valued
   `path` string so the HL7 publisher emits both element paths as the
   documented path for this search parameter. The existing pattern for
   multi-branch paths elsewhere in the clone is a newline/space-separated
   list of element paths inside the `valueString`. Replace

   ```xml
   <extension url="http://hl7.org/fhir/build/StructureDefinition/path">
     <valueString value="Provenance.occurredDateTime"/>
   </extension>
   ```

   with

   ```xml
   <extension url="http://hl7.org/fhir/build/StructureDefinition/path">
     <valueString value="Provenance.occurredDateTime, Provenance.occurredPeriod"/>
   </extension>
   ```

   (Verify the exact separator against another SearchParameter in the
   clone that already lists multiple paths — the publisher is strict
   about how this string is tokenised. Fall back to keeping only
   `Provenance.occurredDateTime` if the publisher rejects the combined
   form; the expression itself is the normative artifact.)

3. Leave `code`, `type` (`date`), `status` (`active`),
   `standards-status` (`normative`), `description` ("When the activity
   occurred"), and `url` unchanged. No base-resource StructureDefinition
   change is required — `Provenance.occurred[x]` already has both
   `dateTime` and `Period` as allowed types.

### Design Rationale

- `Type.ofType(X)` on a choice element is the idiomatic FHIRPath way to
  reach a single branch of an `[x]` polymorphic element; unioning the
  two branches with `|` is the canonical pattern already used by other
  FHIR R6 search parameters with `date|Period` choice targets.
- Search parameter `type` is `date`, which FHIR's date search engine
  already handles uniformly for both `dateTime` and `Period`; no server
  parsing change is implied.
- The resolution explicitly accepted the reporter's exact expression
  ("Persuasive" + "yes"), and the reporter confirmed on fhirpath-lab
  that the expression evaluates correctly against both
  `occurredDateTime` and `occurredPeriod` fixtures. No alternative was
  proposed in comments.
- Keeping the outer parentheses of the original expression is not
  meaningful; dropping them matches the resolution wording and avoids
  an unnecessary churn signal for diff readers.

### Breaking Changes

No breaking changes. The change **widens** the set of resources matched
by `GET [base]/Provenance?when=…` to also include instances whose
`occurred[x]` is a `Period`; it does not remove or repurpose the
parameter or alter its `type`, `code`, `url`, or `description`.
Conforming servers that previously ignored `occurredPeriod` must now
start considering it, which is the intended correctness fix rather than
an incompatible contract change. Clients that used to assume only
`occurredDateTime` was searchable may now receive additional matches —
that is a search-result expansion, not a breaking API change.

### Related Specifications

- `Provenance.occurred[x]` in
  `source/provenance/structuredefinition-Provenance.xml` defines the
  choice of `dateTime | Period` — no change needed, but the updated
  SearchParameter must continue to match its type signature.
- Any profile that constrains `Provenance.occurred[x]` to a single
  choice (e.g., downstream IGs) is unaffected because the base
  SearchParameter still works for whichever branch the profile keeps.

### Testing Considerations

- Build the publisher once (outside this plan's authoring scope) to
  verify the SearchParameter bundle regenerates cleanly and that
  `provenance.html`'s search-parameter table reflects the new
  expression.
- Add / update a Provenance example that uses `occurredPeriod` (one
  exists already — confirm with a search under
  `source/provenance/provenance-example*.xml`). The FHIRPath test suite
  for search parameters should exercise the new expression against
  both `occurredDateTime` and `occurredPeriod` instances and assert a
  match in each case.
- Edge cases:
  - `occurredPeriod` with only a `start` (no `end`) — the date search
    should treat the period as an open-ended interval per the FHIR date
    search rules.
  - `occurredPeriod` with only an `end` — symmetrical.
  - Resources with neither `occurredDateTime` nor `occurredPeriod`
    should still be excluded.
- Interop: FHIR validators and reference servers (HAPI, Firely,
  Grahame's) should pick up the new expression automatically once the
  published SearchParameter bundle is regenerated.
