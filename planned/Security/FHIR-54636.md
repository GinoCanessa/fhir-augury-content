## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):**
> Remove expression. If it is later needed it can be better explained in a specific extension.

The Security WG accepted that the `expression` choice on `Permission.rule.data` is unjustified: its intended use is unclear, the rationale for invoking it is unclear, and supporting `Expression` here is implementation-expensive. The agreed action is to **delete the `Permission.rule.data.expression` element** from the `Permission` StructureDefinition. If the capability is ever re-required, it should be re-introduced via a dedicated extension rather than as a core element on `Permission`.

## Feature Proposal

### Problem Statement

The `Permission` resource (security/privacy WG) has, on its `rule.data` backbone, an element `expression` of type `Expression` (cardinality `0..1`). The original ticket states three concerns:

1. **Unclear semantics** ‚Äî the spec does not adequately describe how `expression` is meant to be used to identify the data covered by a rule.
2. **Unclear motivation** ‚Äî there is no clear use case justifying putting this in core; the other `rule.data` selectors (`resource`, `security`, `period`, `purpose`) already provide most needed selection power.
3. **High implementation cost** ‚Äî implementing/evaluating `Expression` (FHIRPath / CQL / etc.) inside an authorization decision path is expensive and operationally risky.

The reporter (also the WG lead, John Moehrke) suggests that, if a rule-data selection beyond what the existing structured fields provide is ever needed, it would be cleaner to address that requirement with a focused extension, with documentation tailored to that specific use case.

### Proposed Change

Delete the element `Permission.rule.data.expression` from the `Permission` StructureDefinition wherever it currently lives (today: the `HL7/data-access-policies` IG ‚Äî see Repo Context below), plus all artifacts that exist solely to support it.

Concretely, the prior `<differential>` definition (captured from the historical `source/permission/structuredefinition-Permission.xml` in `HL7/fhir` at commit `0e2a8a7f83^`, immediately before Permission was relocated out of core) was:

```xml
<element id="Permission.rule.data.expression">
  <path value="Permission.rule.data.expression"/>
  <short value="Expression identifying the data"/>
  <definition value="Used when other data selection elements are insufficient."/>
  <min value="0"/>
  <max value="1"/>
  <type>
    <code value="Expression"/>
  </type>
  <isSummary value="true"/>
  <mapping>
    <identity value="fhirconsent"/>
    <map value="Consent.provision.class, Consent.provision.code, Consent.provision.expression"/>
  </mapping>
</element>
```

The deletion must:

- Remove the entire `<element>` block for `Permission.rule.data.expression` from the differential of the Permission StructureDefinition.
- Remove any narrative bullets / descriptions / scope notes that mention `expression` as a way of identifying covered data on `Permission.rule.data`.
- Remove or update any examples that populate `rule.data.expression`.
- Remove any invariant or constraint expressions that reference `expression` on `Permission.rule.data` (e.g., a `prm-N` invariant that requires "at least one of resource | security | period | purpose | expression must be present" must be reworded to drop `expression`).
- Update the `fhirconsent` mapping entry on `Permission.rule.data` so that the row mapping `Consent.provision.expression` is removed (the mapping for `Consent.provision.class`/`Consent.provision.code` may still belong, on a different element).

No replacement element is added. No replacement extension is defined as part of this ticket ‚Äî the resolution explicitly leaves any future extension to a separate piece of work.

### Design Rationale

- The Security WG's rationale (per the resolution and the description) is that `Permission.rule.data.expression` lacks a clear, documented use case and pushes a heavy evaluation burden into authorization decision points. Removing it eliminates the dead/risky surface and brings `Permission.rule.data` in line with the rest of `Permission`, which selects data declaratively (by resource type, security label, period, purpose).
- Choosing **deletion** over **deprecation** is consistent with `Permission`'s current pre-normative status (it is being authored out of `HL7/data-access-policies`, not yet locked into a normative core release at this clone HEAD), so a hard removal is permissible without a deprecation cycle.
- Choosing **"future extension if needed"** over **inline retention with better docs** is consistent with FHIR's general guidance that experimental or rarely-used capabilities live as extensions until adoption justifies promotion.
- The reporter and assignee are the same person (the WG lead), so there is no parallel design to reconcile from comments ‚Äî the ticket has no comments and no Zulip cross-references in Augury at the time of planning.

### Breaking Changes

- **Yes ‚Äî element removal.** Removing `Permission.rule.data.expression` is a backward-incompatible change for any consumer that has authored `Permission` instances populating that element. Because `Permission` is currently pre-normative (per its current location in an IG rather than the core release at this HEAD), the WG is permitted to take the removal without a deprecation cycle, but the impact must be acknowledged in the IG's change history / release notes. Any consuming system that has been validating against an earlier version of the `Permission` profile **will reject removal** if their validators are configured against the older snapshot ‚Äî however, instances that simply omit the element will continue to validate.
- No type changes, no required-element introductions, no terminology binding strength changes, no search parameter changes.

### Related Specifications

- `Consent` ‚Äî the prior `fhirconsent` mapping on `Permission.rule.data.expression` mapped to `Consent.provision.expression`.
