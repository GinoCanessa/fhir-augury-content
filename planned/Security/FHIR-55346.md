## Feature Proposal

### Problem Statement

`Provenance.entity.role` is a mandatory element (1..1) with a required binding to `http://hl7.org/fhir/ValueSet/provenance-entity-role`. The bound CodeSystem today lists only five activity-centric codes — `revision`, `quotation`, `source`, `instantiates`, `removal` (`source/provenance/codesystem-provenance-entity-role.xml` lines 51–75) — and offers no "unknown" or "other" escape valve. As raised by Lloyd McKenzie (Comment 310), this makes the containing `entity` BackboneElement impossible to populate when:

- legacy / imported data does not map cleanly to any of the five state-table codes, and
- a future FHIR version introduces a new role that must be down-converted to R6.

Without an escape valve the only workarounds are dropping `Provenance.entity` altogether or mis-classifying the role, both of which degrade provenance fidelity. John Moehrke's comments flag the design concern that a generic `unknown`/`other` may be misused, which the resolution addresses by narrowing scope to `other` with precise "none of the codes apply" semantics plus an extension hook for elaboration.

### Proposed Change

Add exactly one new `<concept>` to the `ProvenanceEntityRole` CodeSystem:

- **CodeSystem:** `http://hl7.org/fhir/provenance-entity-role`
- **Code:** `other`
- **Display:** `Other`
- **Definition:** `The authoring/source system knows that none of the codes apply. An extension may elaboration on the way the data was used.` *(verbatim from the applied resolution; note the ungrammatical "may elaboration" is preserved to match the resolution text — flagged as an Open Question below in case editorial cleanup to "may elaborate" is desired)*

No other changes to the CodeSystem metadata (`version`, `status`, `caseSensitive`, `hierarchyMeaning`, `content=complete`) are required. The new concept is a sibling of the existing five codes — no hierarchy (`<property>` / `concept.concept`) is introduced.

Derived artifacts that flow automatically from the CodeSystem edit:

- `ValueSet/provenance-entity-role` — composes the entire CodeSystem (`<include><system value="http://hl7.org/fhir/provenance-entity-role"/></include>`, no `<concept>` enumeration), so `other` is included without editing `valueset-provenance-entity-role.xml`.
- `Provenance.entity.role` binding in `StructureDefinition/Provenance` — remains `required` + `immutable`; no differential change.

Example snippet to append (just before `</CodeSystem>`):

```xml
<concept>
  <code value="other"/>
  <display value="Other"/>
  <definition value="The authoring/source system knows that none of the codes apply. An extension may elaboration on the way the data was used."/>
</concept>
```

### Design Rationale

- **Minimal, targeted escape valve.** Adding a single `other` code (rather than both `unknown` and `other`) matches the resolution wording and answers Moehrke's concern about users not understanding the intent — the definition explicitly says "none of the codes apply," which is a stronger semantic than a bare `unknown`.
- **Extension-based elaboration.** The definition points implementers at an extension to explain *how* the entity was used when `other` is chosen, preserving the state-table character of the element while permitting legacy/down-conversion cases.
- **ValueSet-via-CodeSystem composition.** Because the bound ValueSet includes the entire CodeSystem, adding the code in one place propagates consistently; there is no risk of the ValueSet and CodeSystem drifting.
- **Normative artifact.** Both the CodeSystem and ValueSet carry `standards-status = normative` with FMM 6 (lines 8–13 of each file). Adding a new code to a normative required-bound CodeSystem is a substantive change that must clear the normative-change bar; the resolution of "Persuasive with Modification" on a Highest-priority Security-owned ticket records that approval.

## Implementation Plan

### Prerequisites

- None. The change is a single-file terminology addition with no upstream dependencies (no new extensions to define, no upstream UTG change, no prerequisite ticket resolution).

### Step-by-Step Tasks

#### HL7/fhir

1. **Add the `other` concept to the Provenance entity-role CodeSystem**
   - File: `source/provenance/codesystem-provenance-entity-role.xml`
   - Action: Insert a new `<concept>` block immediately before the closing `</CodeSystem>` element (after the existing `removal` concept at line 75).
   - Details:
     - `code`: `other`
     - `display`: `Other`
     - `definition`: `The authoring/source system knows that none of the codes apply. An extension may elaboration on the way the data was used.` (verbatim from the ticket resolution)
     - Do **not** modify the CodeSystem `version`, `date`, `status`, `caseSensitive`, `hierarchyMeaning`, `content`, or `valueSet` pointers — only the concept list grows by one.
     - Do **not** add a `<property>` or nest the concept under another concept; it is a flat sibling of the other five codes.

2. **Leave the ValueSet unchanged (verification task)**
   - File: `source/provenance/valueset-provenance-entity-role.xml`
   - Action: Confirm the `<compose>` block still contains only `<include><system value="http://hl7.org/fhir/provenance-entity-role"/></include>` (no `<concept>` enumeration). No edit.
   - Details: Because the ValueSet is `immutable=true` and composes the whole CodeSystem, the new `other` code is included automatically. If a future editor ever replaces the system-wide include with an explicit concept list, `other` must be added there too — call this out in the commit message so reviewers don't re-introduce an enumeration.

3. **Leave the Provenance StructureDefinition binding unchanged (verification task)**
   - File: `source/provenance/structuredefinition-Provenance.xml`
   - Action: Confirm `Provenance.entity.role` remains bound `required` to `http://hl7.org/fhir/ValueSet/provenance-entity-role` with `ElementDefinition.binding` unchanged. No differential edit.
   - Details: Only the `<differential>` is hand-edited in this repo; the `<snapshot>` will regenerate during publish. Do not touch the snapshot.

4. **(Optional) Add a demonstration example exercising the `other` code**
   - File: new `source/provenance/provenance-example-entity-other.xml` (if the Security WG wants a worked example), or extend an existing example.
   - Action: If added, register the new example in `source/provenance/list-Provenance-examples.xml`.
   - Details: Not required by the resolution; include only if the WG wants documentation coverage. If added, also ensure the example carries a simple extension on `Provenance.entity` illustrating the "extension may elaboration" clause.
