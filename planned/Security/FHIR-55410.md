## Resolution Summary

**Resolution:** Persuasive with Modification — "edit as directed".

The resolution approves the reporter's direction: the text that currently reads *"It MAY include consuming, processing, transforming, …"* in the Provenance resource needs clarification and tightening of its conformance language. Specifically:

1. Clarify that Provenance **does not** track actions that only consume data (e.g. a query) or merely process it without producing a new entity (e.g. validating a resource). Consumed entities are only recorded when they are *involved in the creation of a new version* of some other entity.
2. This is descriptive, not normative, text — stop using the uppercase RFC 2119 keyword `MAY`. Use lowercase `may` (or a plain declarative rewording) because this sentence is not stating a conformance requirement.

"Edit as directed" means the resolution explicitly delegates the exact prose to the assignee (John Moehrke), but the shape of the edit is fixed by the two points above.

## Feature Proposal

### Problem Statement

The Provenance resource today defines the `Provenance.activity` concept with the sentence:

> "An activity is something that occurs over a period of time and acts upon or with entities; it MAY include consuming, processing, transforming, modifying, relocating, using, or generating entities."

Two problems:

- **Overreach.** Readers infer that *any* consume-or-process action should be recorded as a Provenance. That is not what Provenance is for: a Provenance instance is only produced when a new entity (or a new version of an entity) is created. Pure consumers (read/query) and pure processors that don't emit a new entity (e.g. validation) never generate a Provenance. The current wording invites incorrect implementations that create Provenance instances for queries or validation calls.
- **Mis-use of conformance language.** The uppercase `MAY` is a RFC 2119 conformance keyword. The sentence is prose describing the W3C-PROV-inspired notion of an "activity"; it is not imposing a conformance option. Using `MAY` here muddies the spec's normative surface and is inconsistent with how FHIR uses 2119 keywords elsewhere (reserved for stated conformance options on implementers, not for prose definitions of conceptual terms).

The same problematic sentence appears verbatim in the human-readable Provenance introduction page, so both the machine-readable StructureDefinition and the narrative page need the edit.

### Proposed Change

Apply the following coordinated edits in `HL7/fhir` (single repo, documentation/StructureDefinition-only change — no schema change, no cardinality or type change):

1. **`source/provenance/structuredefinition-Provenance.xml` — `Provenance.activity` element**, currently at line ~313, `<definition value="…">`:

   Rewrite the definition from:

   > "An activity is something that occurs over a period of time and acts upon or with entities; it MAY include consuming, processing, transforming, modifying, relocating, using, or generating entities."

   to wording along the lines of (final prose at editor's discretion per "edit as directed"):

   > "An activity is something that occurs over a period of time and acts upon or with entities. In Provenance, an activity is recorded only when it produces a new entity or a new version of an existing entity. Examples include transforming, modifying, relocating, or generating entities. Actions that only consume data (for example, a query) or that only process data without producing a new entity (for example, validating a resource) do not by themselves result in a Provenance record; consumed entities are referenced from Provenance only when they contributed to the creation or new version of some other entity."

   The word `MAY` must be removed from this sentence. If a non-normative "may" is retained it must be lowercase.

2. **`source/provenance/provenance-introduction.xml` — Activity bullet**, currently at line ~73 of the intro HTML:

   Rewrite:

   > "Activity - An activity is something that occurs over a time period and acts upon or with entities. It MAY include consuming, processing, transforming, modifying, relocating, using, or generating entities."

   to the same clarified text as above (or the intro-style summary form of it), again replacing uppercase `MAY` with plain prose. Keep the list item inside the existing `<ul>` structure; do not change the Entity or Agent bullets.

3. **Optional sweep** within the same page for the same pattern: the intro also uses uppercase `MAY` in the Entity and Agent bullets ("entities MAY be real or imaginary", "An Agent MAY be a person, device, system, …"). The resolution does not explicitly direct edits to those two bullets, so they are flagged as open questions (see below) rather than changed silently.

No examples need to be altered — the change is prose-only and the existing `provenance-example*.xml` instances remain structurally valid (cardinality, type, and binding of `Provenance.activity` are unchanged).

### Design Rationale

- The resolution says "edit as directed", and the reporter's direction in the description is specific and tightly scoped: clarify what consuming/processing means for Provenance, and stop using `MAY` as a conformance verb here. That is the minimum change.
- Keeping the element's cardinality (`0..1`) and type (`CodeableConcept` bound to the `provenance-activity-type` binding) unchanged preserves backward compatibility. No schema or wire format change is introduced.
- Mirroring the exact wording between the StructureDefinition `<definition>` and the human-readable introduction page is a long-standing FhirCore convention and matches the convention recorded in the repo briefing (introduction pages and StructureDefinition definitions are both hand-authored in `source/<resource>/`). Editing both in one commit prevents drift between the narrative and the machine-readable definition that will be regenerated into the rendered spec.
- Downgrading `MAY` to "may" (or removing it in favour of declarative prose) is consistent with the rest of the Provenance resource narrative, which mixes normative statements (correctly capitalised) with definitional prose (lowercase).
