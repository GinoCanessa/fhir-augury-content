## Resolution Summary

**Resolution:** Persuasive with Modification.

The committee accepted that the relationship between `AuditEvent.agent.type` and `AuditEvent.agent.role` is unclear with respect to ISO 21298 *functional* vs *structural* roles, but rejected the reporter's proposal to wholly equate `.type` with "functional role" / `.role` with "structural role". Instead, `.type` is to be aligned with the underlying ASTM/DICOM/IHE-ATNA element from which it derives — namely "how the agent participated in this specific activity" — while `.role` is recast as the broader set of authorising roles the agent holds, which *may* be drawn from ISO 21298 functional and/or structural role concepts but is not required to be.

The exact element-level edits agreed in the resolution (verbatim from the resolution description):

> **AuditEvent.agent.type**
> - Definition: How the agent participated.
> - Requirements: What was the function of the agent in the activity that is being recorded. Where an agent played multiple types of participation, they will be listed as multiple `.agent` elements representing each type of participation.
> - Comment: This may be an indication of "Functional Role" but does not need to correlate to ISO 21298:2018 - Functional and Structural Roles.
>
> **AuditEvent.agent.role**
> - Definition: Roles the agent holds that authorize the participation in the activity.
> - Requirements: The roles that reflect the agent authorizations related to the activity.
> - Comment: These roles may be an indication of "Functional Role" and/or "Structural Role" concept as documented in ISO 21298:2018 - Functional and Structural Roles.

In addition, the reporter's final comment (2026-03-24) requested — and was tacitly accepted by the assignee — a short clarifying note in the AuditEvent narrative that **participation in the activity is NOT the same thing as a functional role**, even though the two concepts may overlap. That clarification is to be added to the AuditEvent notes/content page (Section 6.4) so it is discoverable without reading the element binding text.

The resolution is scoped to **AuditEvent**. The reporter's original proposal also touched Provenance, but the assignee's final disposition does not modify Provenance text; Provenance changes are out of scope for this ticket.

## Feature Proposal

### Problem Statement

Section 6.4 of the AuditEvent specification (the AuditEvent resource page, Notes, and the binding description for `participation-role-type`) currently describes `AuditEvent.agent.type` and `AuditEvent.agent.role` in language that is anchored to ISO 21298:2018 *functional role* / *structural role* concepts (mirrored verbatim in the element `requirements` text — e.g. "Functional roles reflect functional aspects of relationships between entities. Functional roles are bound to the realization/performance of acts…"). This wording strongly implies — particularly to readers unfamiliar with ISO 21298 — that the FHIR elements are direct projections of those ISO concepts, when they are not:

- `AuditEvent.agent.type` is `0..1` and is intended to convey *how the agent participated in this specific activity* (the ATNA / DICOM / RFC 3881 lineage), not the abstract ISO functional-role concept (which can be many).
- `AuditEvent.agent.role` is `0..*` and is intended to be a free, locally-meaningful set of roles authorising participation, optionally aligned with ISO structural and/or functional roles, but not required to be.

The element `definition`, `comment`, and `requirements` text on the StructureDefinition, and the section 6.4 narrative, do not currently make these distinctions visible. Implementers reading the public spec at `hl7.org/fhir/6.0.0-ballot4/auditevent-definitions.html#AuditEvent.agent.type` are likely to mis-equate `.type` with ISO 21298 *functional role* (as the reporter explicitly did), which is precisely what the Security WG does not intend.

**3. AuditEvent narrative — add a short "Structural vs Functional vs Participation" clarification on the AuditEvent content page** (the assignee accepted in the 2026-03-24 exchange that "if possible, it would be good if you could add some note indicating that participation is not the same as functional roles, even if they may be related sometimes"). This clarification belongs on the AuditEvent notes page (`source/auditevent/auditevent-notes.xml`) — the same page that already hosts §6.4 sub-sections such as "Using Coded Values", "Event codes for Common Scenarios", and "authorization and agent.authorization". A new `<h3>` section titled e.g. "Agent role vs agent type" should be added near the top (before "Event codes for Common Scenarios") with text along these lines (final wording at author discretion, preserving the substantive points):

> The `agent.type` element records *how the agent participated in the specific activity that is being audited* — i.e., the role the agent played in this single AuditEvent. It is bound (preferred) to the `participation-role-type` value set and is `0..1`; an agent that played more than one kind of participation in the same activity is recorded as multiple `agent` elements, one per type of participation.
>
> The `agent.role` element records *the roles the agent holds that authorize their participation*. It is `0..*` and bound (example) to the `security-role-type-example` value set. These roles MAY be drawn from the *Functional* and/or *Structural* role concepts of ISO 21298:2018 (Health Informatics — Functional and structural roles), but implementers are not required to align with ISO 21298; locally-defined or domain-specific role vocabularies are also valid.
>
> Note: "participation in the activity" (`agent.type`) and "functional role" (as defined by ISO 21298) are related but not equivalent concepts. ISO 21298 functional roles are abstract role concepts that may be assigned across many acts; `agent.type` is the concrete participation in the single activity recorded by this AuditEvent.

Example snippet (illustrative — not part of the resolution, included to make the contrast concrete):

```xml
<agent>
  <type>
    <coding>
      <system value="http://terminology.hl7.org/CodeSystem/v3-ParticipationType"/>
      <code value="AUT"/>     <!-- author: how the agent participated in this event -->
    </coding>
  </type>
  <role>
    <coding>
      <system value="http://terminology.hl7.org/CodeSystem/v3-RoleClass"/>
      <code value="PROV"/>    <!-- structural authorization role: 'healthcare provider' -->
    </coding>
  </role>
  <who><reference value="Practitioner/example"/></who>
</agent>
```

### Design Rationale

- **Anchored in the inherited information model.** The AuditEvent resource derives from RFC 3881 → DICOM Audit Log Message → IHE-ATNA. In all of those, the equivalent of `.type` is "how the agent participated in *this* event", not an abstract role concept. The new `.type` text restores that lineage as the primary definition (the assignee made this argument explicitly on 2026-03-19 by quoting the AuditEvent Scope and Usage list of derivative standards).
- **ISO 21298 retained as guidance, not as a binding constraint.** The reporter's strict mapping ( `.role` ↔ structural, `.type` ↔ functional) was rejected because (a) `.type` is `0..1` while an agent can have many functional roles per ISO, and (b) the Security WG explicitly does not want to mandate ISO 21298 alignment. The new `.role` comment keeps ISO 21298 as a *possible* source (covering both functional and structural concepts) so implementers who use ISO 21298 still get a clear hook, while implementers who don't are not constrained.
- **"Security role" wording removed from `.role` definition** as specifically requested by the reporter (2026-02-02) and not contested in the discussion. The element text now centres on *authorization for participation* rather than the narrower "security role" framing.
- **Narrative clarifier on the resource page** addresses the reporter's primary complaint that the distinction is buried in the binding text in §6.4.4.1 and is "easily overlooked". Putting the clarification in `auditevent-notes.xml` (which renders as part of the AuditEvent resource page §6.4) makes it discoverable to anyone reading the resource overview.
- **Provenance left unchanged.** Although the reporter proposed parallel edits to `Provenance.agent.role` and `Provenance.agent.type`, the resolution text is silent on Provenance and the assignee did not commit to Provenance edits in the discussion. Touching Provenance here would exceed the resolution; if alignment is desired it should be raised as a separate Security WG ticket.
