## Problem Statement

The `Provenance.activity` element is bound (example strength) to the `provenance-activity-type` ValueSet. The Provenance page explicitly discusses recording the *deletion/removal* of a resource (the "Provenance of Removal and Replace" section) and even provides a delete example (`provenance-example-delete`). However, the `provenance-activity-type` ValueSet does **not** include a `destroy` concept from the `iso-21089-lifecycle` code system, even though that code system already defines `destroy` ("Destroy/Delete Record Lifecycle Event"). The ticket asks to:

1. Add `destroy` to the `provenance-activity-type` ValueSet.
2. Ensure the Provenance notes section references the `destroy` vocabulary for the `activity` element when discussing deletion/removal.

## Feature Proposal

Add the `destroy` code from `http://terminology.hl7.org/CodeSystem/iso-21089-lifecycle` to the `provenance-activity-type` ValueSet, and update the "Provenance of Removal and Replace" narrative section to reference `destroy` as the recommended `Provenance.activity` vocabulary for deletion events.

## Detailed Implementation Plan

### Change 1: Add `destroy` to `provenance-activity-type` ValueSet

**File:** `source/terminologies/valueset-provenance-activity-type.xml`  
**Repository:** HL7/fhir

The ValueSet currently includes 7 concepts from `iso-21089-lifecycle`: `amend`, `originate`, `merge`, `deidentify`, `receive`, `transform`, `verify`. Add `destroy` to this list.

**Current content (lines 45–69):**
```xml
<compose>
  <include>
    <system value="http://terminology.hl7.org/CodeSystem/iso-21089-lifecycle"/>
    <concept>
      <code value="amend"/>
    </concept>
    <concept>
      <code value="originate"/>
    </concept>
    <concept>
      <code value="merge"/>
    </concept>
    <concept>
      <code value="deidentify"/>
    </concept>
    <concept>
      <code value="receive"/>
    </concept>
    <concept>
      <code value="transform"/>
    </concept>
    <concept>
      <code value="verify"/>
    </concept>
  </include>
</compose>
```

**Proposed change:** Add a new `<concept>` element for `destroy`. A logical placement is after `deidentify` (alphabetical within the lifecycle codes) or at the end of the list before `</include>`:

```xml
<compose>
  <include>
    <system value="http://terminology.hl7.org/CodeSystem/iso-21089-lifecycle"/>
    <concept>
      <code value="amend"/>
    </concept>
    <concept>
      <code value="originate"/>
    </concept>
    <concept>
      <code value="merge"/>
    </concept>
    <concept>
      <code value="deidentify"/>
    </concept>
    <concept>
      <code value="destroy"/>
    </concept>
    <concept>
      <code value="receive"/>
    </concept>
    <concept>
      <code value="transform"/>
    </concept>
    <concept>
      <code value="verify"/>
    </concept>
  </include>
</compose>
```

**Rationale:** The `destroy` code is defined in `iso-21089-lifecycle` (display: "Destroy/Delete Record Lifecycle Event", definition: "Occurs when an agent causes the system to permanently erase record entry content from the system."). This is the appropriate vocabulary for deletion/removal activity in Provenance, and the Provenance page already discusses this use case.

### Change 2: Update "Provenance of Removal and Replace" narrative

**File:** `source/provenance/provenance-notes.xml`  
**Repository:** HL7/fhir

The "Provenance of Removal and Replace" section (lines 74–87) currently describes how to use Provenance to record deletion but does **not** mention what `Provenance.activity` code to use. The section should explicitly reference the `destroy` vocabulary from `iso-21089-lifecycle`.

**Current text (lines 74–87):**
```html
<a name="removal"> </a>
<h3>Provenance of Removal and Replace</h3>
<p>
When a Provenance resource is used to record the removal of a resource, the Provenance.target 
references the removed resource, and the Provenance.entity indicates the removed resource as well. 
For example, <a href="provenance-example-delete.html">example delete</a> shows the use of 
Provenance to record the deletion of a resource. 
If versioning is supported, 
the version that was removed is referenced in Provenance.target; if versioning is not supported then 
Provenance.target contains the non-version reference. 
</p>
```

**Proposed change:** Add a sentence indicating that `Provenance.activity` should use the `destroy` code from `iso-21089-lifecycle`:

```html
<a name="removal"> </a>
<h3>Provenance of Removal and Replace</h3>
<p>
When a Provenance resource is used to record the removal of a resource, the Provenance.target 
references the removed resource, and the Provenance.entity indicates the removed resource as well. 
The <code>Provenance.activity</code> SHOULD use the <code>destroy</code> code from the 
<a href="http://terminology.hl7.org/CodeSystem/iso-21089-lifecycle">ISO 21089 Record Lifecycle</a> 
code system to indicate the deletion activity.
For example, <a href="provenance-example-delete.html">example delete</a> shows the use of 
Provenance to record the deletion of a resource. 
If versioning is supported, 
the version that was removed is referenced in Provenance.target; if versioning is not supported then 
Provenance.target contains the non-version reference. 
</p>
```

**Rationale:** The ticket explicitly asks to "make sure the section speaks to the vocabulary `destroy` for activity." This guides implementers to use the correct activity code when recording deletion provenance.

## Testing Considerations

- Verify the FHIR build succeeds with the updated ValueSet (the `destroy` code must resolve against the `iso-21089-lifecycle` code system).
- Verify the expanded ValueSet includes all 8 concepts (the 7 existing plus `destroy`).
- Confirm the Provenance narrative renders correctly with the new text and link.
- The existing `provenance-example-delete.xml` uses `v3-DataOperation#DELETE` rather than `iso-21089-lifecycle#destroy`. Since the binding is example strength, both are valid. No example change is strictly required by this ticket, though updating the example to use `destroy` could be considered as a follow-up for consistency.

- The `destroy` code already exists in the upstream `iso-21089-lifecycle` code system in UTG — no new code system changes are needed.
- The narrative change is purely informational guidance, not a conformance constraint.
- No structural or constraint changes to the Provenance StructureDefinition are involved.
