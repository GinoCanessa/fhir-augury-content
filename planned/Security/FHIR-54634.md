## Resolution Summary

The ticket is resolved **Persuasive**. The Permission resource must allow
authorization rules to be scoped by `Location`, both at the resource root and
in each `rule.activity`, so a Permission can express "this rule applies (or
explicitly does NOT apply) to requests from a given Location." The reporter's
own clarification on the ticket — "Location may be needed at the root, but
also in each rule." — is the operative direction. Since the resolution is
Persuasive (not Persuasive with Modification), the change implements the
ticket as written: add a `Location` reference at the Permission root and at
`Permission.rule.activity`, supporting both inclusion and negation
(via `Permission.rule.type = deny` together with the activity match).

## Feature Proposal

### Problem Statement

Permission today expresses what activities are permitted/denied via
`Permission.rule.activity` (`actor`, `action`, `purpose`, `dataResource` /
`reference`, `request`). There is no way to constrain a rule (or a
Permission as a whole) to requests originating from, or targeting, a
particular `Location`. Real-world authorization policies regularly need
location-scoped rules — e.g., "this clinician may read these records only
when accessing from the ED workstation," or "deny external network
locations." Without an `activity.location`, policy authors must either
encode location out-of-band or rely on `purpose` / `actor` proxies, both of
which are lossy and opaque to a Permission engine.

### Proposed Change

Add a `Location` reference to Permission, both at the resource root and in
each rule's activity:

1. **`Permission.rule.activity.location`** *(new element)*
   - Cardinality: `0..*`
   - Type: `Reference(Location)`
   - Short: "The Location(s) this activity rule applies to"
   - Definition: "When present, this rule only matches requests whose
     originating or targeted Location is among (or, when used inside a
     `deny` rule, is one of) the referenced Locations. Absent means the
     rule is not scoped by Location."
   - Summary: true (consistent with sibling `actor` / `action` / `purpose`).

2. **`Permission.location`** *(new root-level element, optional)*
   - Cardinality: `0..*`
   - Type: `Reference(Location)`
   - Short: "Location(s) this Permission as a whole is scoped to"
   - Definition: "When present, the entire Permission only takes effect for
     requests bound to one of these Locations. Acts as an outer filter that
     each rule's `activity.location`, if any, further narrows."
   - Summary: true.

3. **Negation semantics.** Negation ("NOT a Location") is expressed with the
   existing `Permission.rule.type = deny` pattern combined with
   `activity.location`, rather than introducing a new boolean. This matches
   how the resource already handles "NOT this actor / NOT this action."
   Document this explicitly in the resource scope/usage notes so authors
   don't reach for a custom extension.

4. **Search parameter.** Add a `Permission.location` search parameter:
   - Name: `location`
   - Type: `reference`, target `Location`
   - Expression: `Permission.location | Permission.rule.activity.location`
   - Aligns with the `Permission.actor` / `Permission.activity` search
     parameters added under FHIR-50219 / FHIR-50221.

5. **Examples / narrative.** Update at least one existing Permission example
   to demonstrate `rule.activity.location`, and add a short paragraph to the
   resource narrative describing location-scoped rules and the deny pattern.

### Design Rationale

- **Mirrors existing `activity` shape.** `Permission.rule.activity` already
  layers per-rule constraints on top of the rule envelope; adding
  `location` there keeps the granularity story consistent with `actor`,
  `action`, `purpose`, and `dataResource`.
- **Root + per-rule, per the reporter.** John Moehrke (reporter, assignee,
  and Security WG co-chair) explicitly called out both placements in the
  one comment on the ticket. Implementing only one would not satisfy the
  Persuasive resolution.
- **Negation via `deny`, not a new flag.** Permission already supports
  inclusion/exclusion through `rule.type` of `permit` vs `deny`. Reusing
  that pattern avoids adding orthogonal "not" semantics that other
  `activity.*` siblings would also need.
- **Search parameter scope.** A combined expression
  (`Permission.location | Permission.rule.activity.location`) lets clients
  find any Permission relevant to a Location without having to know whether
  the policy author scoped at the root or at the rule.
