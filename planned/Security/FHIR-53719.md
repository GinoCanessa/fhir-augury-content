## Feature Proposal

### Problem Statement

The base FHIR specification's Security narrative
(`source/security.html`, "Authentication" section) currently states that
either UDAP Consumer-Facing or UDAP Business-to-Business **SHOULD** be
implemented "as appropriate" for scalable security, that the
Registration and Discovery sections of UDAP Security **SHOULD** also be
implemented, and that **Tiered OAuth for User Authentication SHOULD be
implemented to securely scale trust with new OIDC credential issuers**
(lines 228–232).

The reporter argues this is awfully premature for a base-spec normative
recommendation:

- UDAP Consumer-Facing / B2B presuppose substantial network
  infrastructure (TLS certs, directories, trust framework) that exists
  in only a few mature national interop networks. Recommending it at
  SHOULD strength in the base spec implies broad applicability that
  doesn't exist in practice.
- UDAP Tiered OAuth lacks the deployed adoption to justify being
  recommended in the base spec at all.
- The current SHOULDs sit at the same conformance strength the same page
  uses for foundational technologies like TLS and NTP, which
  overstates UDAP's maturity in the global FHIR audience.

### Proposed Change

**Repository:** `HL7/fhir` (FhirCore)
**File:** `source/security.html`
**Section:** `<a name="authentication"></a>` / `<h2>Authentication</h2>`,
the `<p>` block currently spanning lines 228–232.

Two coordinated edits inside that `<p>`:

1. **Downgrade the UDAP SHOULDs to MAY.** Rewrite the first two
   sentences of the paragraph so that:
   - "either UDAP Consumer-Facing or UDAP Business-to-Business **SHOULD**
     be implemented, as appropriate" → "either UDAP Consumer-Facing or
     UDAP Business-to-Business **MAY** be implemented, as appropriate".
   - "The Registration and Discovery sections of UDAP Security **SHOULD**
     also be implemented to securely scale the addition of new client
     and server endpoints" → "The Registration and Discovery sections of
     UDAP Security **MAY** also be implemented to securely scale the
     addition of new client and server endpoints" (this is the UDAP
     dynamic-registration sentence the reporter calls "UDAP dynreg").

2. **Remove the Tiered OAuth recommendation.** Delete the third
   sentence in its entirety (current line 231):
   "Tiered <a href=\"http://hl7.org/fhir/us/udap-security/user.html\">OAuth
   for User Authentication</a> SHOULD be implemented to securely scale
   trust with new OIDC credential issuers." Also remove the now-orphaned
   external link to `http://hl7.org/fhir/us/udap-security/user.html`
   (the link only exists inside that sentence).

The neighbouring paragraphs are not changed:

- The OpenID Connect recommendation in the preceding `<p>` (lines
  223–227) is left as-is.
- The "All systems SHALL protect authenticator mechanisms…" paragraph
  (lines 233–235) is left as-is — it is the unaffected SHALL.
- The `<a href="http://hl7.org/fhir/us/udap-security/b2b.html#b2b-authorization-extension-object">UDAP B2B Authorization Extension Object</a>`
  reference further down the page (line 339, in the consent /
  business-to-business workflow paragraph) is **not** in scope; the
  reporter's complaint is specifically about the SHOULD-strength
  recommendation in the Authentication section, not the descriptive
  mention of the B2B extension object in a different context.

`source/security-labels.html` (line 299) also references UDAP Security
but only as a documentary example of `purpose-of-use` (`hl7-b2b` OAuth
extension); it carries no SHOULD/SHALL conformance language about
implementing UDAP and is **not** edited by this ticket.

#### Suggested replacement HTML

```html
<p>
For scalable security, either <a href="http://hl7.org/fhir/us/udap-security/consumer.html">UDAP Consumer-Facing</a> or <a href="http://hl7.org/fhir/us/udap-security/b2b.html">UDAP Business-to-Business</a> MAY be implemented, as appropriate.
The Registration and Discovery sections of <a href="http://hl7.org/fhir/us/udap-security/index.html">UDAP Security</a> MAY also be implemented to securely scale the addition of new client and server endpoints.
</p>
```

(The Tiered OAuth sentence is dropped; the paragraph is otherwise
character-identical except for `SHOULD` → `MAY` in two places.)

### Design Rationale

- **Persuasive disposition with no modifying comment.** The Security
  WG's recorded decision is "Persuasive"; with no resolution
  description and no Jira comments narrowing the scope, the most
  faithful implementation is the reporter's literal request: dynreg →
  MAY, Tiered OAuth removed.
- **Conformance-strength calibration.** The base FHIR spec already uses
  SHALL for "protect authenticator mechanisms" and SHOULD for things
  like CORS and OpenID Connect that are broadly deployable. Reserving
  SHOULD for technologies with broad-deployment evidence (TLS, OIDC,
  CORS) and using MAY for technologies that depend on mature
  national-network infrastructure (UDAP dynreg, consumer/B2B trust
  frameworks) is internally consistent with the rest of
  `source/security.html`.
- **Removal vs. demotion of Tiered OAuth.** The reporter explicitly
  asked to *remove mention* of Tiered OAuth (not to demote it).
  Demoting would still imply base-spec endorsement of an option whose
  deployed footprint the reporter says is too small to justify the
  mention; removal is the option that matches the ticket text.
- **Scope discipline.** The ticket targets one paragraph in one page.
  The unrelated B2B-authorization-extension-object reference further
  down the same page is descriptive of an extension mechanism, not a
  conformance recommendation, so it is intentionally out of scope.

### Breaking Changes

No technical breaking changes — this is a normative-strength downgrade
in narrative text, not a schema or terminology change. However, it
**does** weaken normative guidance:

- Implementers who read the base spec to choose a security framework
  will no longer see UDAP Consumer-Facing / B2B as a SHOULD.
- The base spec will no longer mention Tiered OAuth at all; readers who
  previously found it via this page will need to look at the UDAP IG
  directly.

These should be called out in the FHIR change-tracker / version-history
entry for the publish, but they do not break wire-level conformance for
any artifact.
