## Resolution Summary

The resolution is **Persuasive with Modification**. From the ticket's resolution description:

> Add a Verify example, following the Remove/Delete example.

The reporter observed that the example ValueSet for `Provenance.activity` includes a `verify` code, but — as with `delete` — it is not obvious how `Provenance.target` relates to `Provenance.entity.what` when the verification evidence is held entirely inside the Provenance resource and no update is performed on the verified resource. The agreed fix is to add a worked example that shows this pattern, modelled on the existing `provenance-example-delete` example rather than changing the resource model, activity code, or prose semantics. No StructureDefinition or ValueSet changes are requested by the resolution.

## Feature Proposal

### Problem Statement

The Provenance resource's activity ValueSet (`http://terminology.hl7.org/ValueSet/v3-DataOperation` via the example binding) contains the code `VERIFY`. Implementers have asked how to represent a verification activity in Provenance when verification does not produce a new version of the verified resource — the target of the verification *is* the entity whose content was checked, so `Provenance.target` and `Provenance.entity.what` end up referencing the same resource. The spec already addresses the analogous case for `DELETE` in the "Provenance of Removal and Replace" section of `source/provenance/provenance-notes.xml` (clone HEAD `db79dcd`, lines 74–87) and ships a companion example at `source/provenance/provenance-example-delete.xml`. No equivalent example exists for `verify`, so readers have no canonical pattern to copy.

### Proposed Change

Add one new example instance and register it with the Provenance examples list, mirroring how `provenance-example-delete` is wired in today.

1. **New example file** — `source/provenance/provenance-example-verify.xml`
   - Modelled line-for-line on `source/provenance/provenance-example-delete.xml` (clone HEAD `db79dcd`).
   - `Provenance.id` = `example-verify`.
   - `Provenance.target` references the resource whose content is being verified (e.g., a version-specific reference to an existing example such as `Observation/example/_history/1`, consistent with how the delete example uses a version-specific target reference).
   - `Provenance.recorded` = a realistic fixed timestamp (match the style of `provenance-example-delete.xml`).
   - `Provenance.activity.coding` = `system=http://terminology.hl7.org/CodeSystem/v3-DataOperation`, `code=VERIFY`.
   - `Provenance.agent.type.coding` = `system=http://terminology.hl7.org/CodeSystem/provenance-participant-type`, `code=verifier` if present in that CodeSystem at `db79dcd`; otherwise fall back to `author` as the delete example does, and surface this in Open Questions.
   - `Provenance.agent.who.display` = a representative verifier name.
   - `Provenance.entity.role` = `source` (from `source/provenance/codesystem-provenance-entity-role.xml` at `db79dcd`, line 62) — the verified resource is the source of the verification evidence. The `removal` code used by the delete example is semantically specific to removal and is not the right fit for verify.
   - `Provenance.entity.what.reference` = the same reference used in `Provenance.target`, demonstrating the pattern the ticket is documenting (target == entity.what when no update is performed).

2. **Register the example in the examples list** — `source/provenance/list-Provenance-examples.xml`
   - Add a new `<entry>` after the existing `provenance-example-delete` entry (currently lines ~154–164 in the clone), using the build extensions already used by sibling entries:
     - `description`: `Provenance of a verify activity where the evidence is held entirely in the Provenance.`
     - `title`: `provenance-example-verify`
     - `item.reference`: `Provenance/example-verify`
     - `item.display`: `Provenance of Verify`

3. **Prose cross-link (minor, secondary)** — `source/provenance/provenance-notes.xml`
   - Optional but aligned with the ticket's stated gap: extend the existing "Provenance of Removal and Replace" treatment (lines 74–87) with a short paragraph (or a new `<a name="verify"></a>` / `<h3>Provenance of Verify</h3>` block in the same style) that explains the verify case and hyperlinks to `provenance-example-verify.html`. This is not strictly required by the resolution text ("Add a Verify example"), so it is called out as a discretionary sub-task and flagged in Open Questions.

No changes are proposed to:
- `source/provenance/structuredefinition-Provenance.xml` (no element additions/changes).
- `source/provenance/codesystem-provenance-entity-role.xml` or `valueset-provenance-entity-role.xml` (existing `source` role is reused).
- Any other repository — the change is confined to `HL7/fhir`.

### Design Rationale

The resolution explicitly says to *follow* the Remove/Delete example, so copying the shape of `provenance-example-delete.xml` (including the target == entity.what duplication) is the most faithful expression of the approved solution. Using `entity.role = source` rather than inventing a new role keeps the example within the current terminology; `source` is already in `codesystem-provenance-entity-role.xml` and semantically covers "the resource whose content is the source of the verification judgment". Reusing the example-delete ValueSet bindings (`v3-DataOperation` with code `VERIFY`) is consistent with the existing Provenance activity example binding and avoids scope creep into terminology (which would have required UTG work and a separate ticket). Picking an existing `/_history/N` target reference — as the delete example does — keeps the example self-contained without needing new supporting resources.
