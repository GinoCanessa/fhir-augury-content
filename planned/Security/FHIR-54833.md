## Resolution Summary

Resolution: **Not Persuasive with Modification**.

The work group affirms that the term **`authorization`** is the correct name for
the element (i.e., the rename from R4 `Provenance.reason` →
`Provenance.authorization` stands). However, with modification:

> Drop the datatype back to `CodeableConcept`, as we cannot come up with any
> useful use of `Reference` that is not already covered by other elements in
> the resource (e.g. `.basedOn`).

Reviewer note in the resolution also reminds that R4 uses of
`Provenance.reason.text` as a freeform text blob explaining *why* the activity
happened are now carried by **`Provenance.why`** (markdown). No action is
required there for this ticket — `Provenance.why` already exists in the R6
draft as `markdown` (0..1).

Net change required by this ticket: change the type of `Provenance.authorization`
from `CodeableReference` to `CodeableConcept` (everything else about the element
— name, short, definition, requirements, alias, cardinality 0..*, binding to
v3-PurposeOfUse at strength `example`, mappings — stays).

## Feature Proposal

### Problem Statement

Between R4 and the current R6 ballot draft, the Security WG renamed
`Provenance.reason` (CodeableConcept, extensible binding to PurposeOfUse) to
`Provenance.authorization` (CodeableReference, example binding to PurposeOfUse)
and added `Provenance.why` (markdown, 0..1) to carry freeform reason text.

The reporter (Brett Marquard) questioned both the rename ("authorization" vs
something like "approved purposes") and, implicitly, whether the broadened
`CodeableReference` type was justified. After WG discussion (see resolution and
John Moehrke's comment that the change was made to "support broader
consideration of the authorization, such as a medical order, rather than just
the security purpose of use concept"), the WG agreed that:

1. `authorization` is the right name and definition.
2. There is no actual `Reference()` target that is not already better expressed
   elsewhere in `Provenance` (notably `Provenance.basedOn`, which carries the
   resource(s) that authorized/triggered the event).

So `CodeableReference` is over-engineered for the intent; `CodeableConcept`
suffices and avoids redundancy with `.basedOn`.

### Proposed Change

In `HL7/fhir`, change one element type in the core Provenance
StructureDefinition and update the four examples that use the
`CodeableReference`-shaped authorization payload:

- **`Provenance.authorization`** — change `<type><code value="CodeableReference"/></type>`
  to `<type><code value="CodeableConcept"/></type>`. Leave everything else on
  the element unchanged:
  - path: `Provenance.authorization`
  - short: `Authorization (purposeOfUse) related to the event`
  - definition: `The authorization (e.g., PurposeOfUse) that was used during the event being recorded.`
  - requirements: `Record of any relevant security context, not restricted to purposeOfUse valueSet. May include security compartments, refrain, obligation, or other security tags.`
  - alias: `PurposeOfEvent`
  - cardinality: `0..*`
  - binding: `example` to `http://terminology.hl7.org/ValueSet/v3-PurposeOfUse`
    (binding name `ProvenanceReason`)
  - mappings: workflow `Event.reason`, w5 `FiveWs.why[x]`, rim `unique(./reasonCode)`,
    fhirauditevent `AuditEvent.purposeOfEvent`

- **Examples** — the four example instances currently serialize the
  `CodeableReference` `concept` choice as `<authorization><concept><coding>…</coding></concept></authorization>`.
  When the type becomes `CodeableConcept`, the wrapping `<concept>` element must
  be removed so the payload is `<authorization><coding>…</coding></authorization>`.
  Files: `provenance-example-advanced.xml`,
  `provenance-example-correction-replacement.xml`,
  `provenance-example-delete.xml`, `provenance-example-sig.xml`.

  Example before/after for `provenance-example-correction-replacement.xml`:

  ```xml
  <!-- before -->
  <authorization>
    <concept>
      <coding>
        <system value="http://terminology.hl7.org/CodeSystem/v3-ActReason"/>
        <code value="PATSFTY"/>
      </coding>
    </concept>
  </authorization>

  <!-- after -->
  <authorization>
    <coding>
      <system value="http://terminology.hl7.org/CodeSystem/v3-ActReason"/>
      <code value="PATSFTY"/>
    </coding>
  </authorization>
  ```

- **`profile-provenance-relevant-history` (Relevant History profile)** — the
  profile only adds a `comment` and `mustSupport` to `Provenance.authorization`
  and does **not** override `<type>`. No change required there; it inherits the
  new `CodeableConcept` type from the base resource.

- **Workflow / Event-pattern divergence document
  (`provenance-event-mapping-exceptions.xml`)** — the existing `divergentElement`
  for `patternPath="Event.reason" resourcePath="Provenance.authorization"`
  documents short/definition divergence, not type divergence, so no edit is
  strictly required. The `CodeableReference(...)` mention later in the file is
  on the `Provenance.why` divergence block, not on authorization, so it does
  not need touching for this ticket.

- **`Provenance.why`** — no change. Already present as `markdown` 0..1, with
  alias `reason`, exactly as the resolution note describes.

### Design Rationale

- The WG explicitly considered and rejected the broader name change ("approved
  purposes"). The element conveys *both* the formal security PurposeOfUse code
  *and* other authorization-style codes (e.g. ActReason codes such as `TREAT`,
  `PATSFTY`, `METAMGT` that already appear in the examples). `authorization`
  generalises correctly across those uses.
- Dropping from `CodeableReference` back to `CodeableConcept` eliminates
  redundancy with `Provenance.basedOn` (which is already
  `Reference(Any)`-shaped via the existing R6 draft) for any case where the
  authorizing instrument is itself a FHIR resource (medical order,
  ServiceRequest, Consent, etc.). This keeps a single canonical place for the
  resource pointer and reserves `authorization` for the coded purpose itself.
- All four pre-existing R6-ballot examples already populate only the `concept`
  choice — there is no example using the `reference` choice — which
  corroborates the WG's "no useful Reference use we can come up with" finding
  and means the change is non-disruptive to the existing example set beyond a
  one-element re-wrap.
- Binding strength stays at `example` rather than reverting to the R4
  `extensible` strength: the WG's desire to allow non-PurposeOfUse codes (e.g.
  ActReason TREAT/PATSFTY) is exactly what `example` strength enables.

### Breaking Changes

This is a breaking change **relative to prior R6 ballot drafts** that already
shipped `Provenance.authorization` as `CodeableReference`:

- **Type narrowing (CodeableReference → CodeableConcept).** Any implementer who
  populated `Provenance.authorization.reference` (the `Reference` choice of
  `CodeableReference`) loses the ability to do so. They must either:
  - move that reference into `Provenance.basedOn` (the WG's stated rationale
    for the change), or
  - drop the reference and retain only the coded value in
    `Provenance.authorization`.
- **Wire-format change for the `concept` choice.** Instances populating the
  `concept` choice change shape from `<authorization><concept><coding>…</authorization>`
  to `<authorization><coding>…</authorization>` (and the equivalent JSON path
  shortening, `authorization[i].concept.coding[…]` → `authorization[i].coding[…]`).

Relative to **R4** (`Provenance.reason : CodeableConcept`), this is *not* a
new break: the element is still single-valued at the coding-level shape and
binds the same `v3-PurposeOfUse` ValueSet (now at `example` rather than
`extensible` strength, which is a relaxation, not a tightening).

### Related Specifications

- **`Provenance.basedOn`** — the WG's named "where to put a reference instead"
  element. No edit required, but the resolution explicitly leans on its
  existence; the plan's task list includes verifying it remains available for
  the use cases this change defers to it.
- **`Provenance.why`** — already added (markdown 0..1) and explicitly called out
  in the resolution as the new home for R4's `Provenance.reason.text` freeform
  usage. No edit required.
- **AuditEvent.purposeOfEvent** — preserved by the existing
  `fhirauditevent` mapping on `Provenance.authorization`; that mapping's
  semantics are unaffected by the type narrowing.
- **Workflow `Event.reason` pattern** — the divergence file
  (`provenance-event-mapping-exceptions.xml`) already documents that
  Provenance diverges from the pattern on this element; no new divergence
  introduced by narrowing the type.

### Testing Considerations

- **Revalidate `Provenance` and the relevant-history profile** after the
  publisher build to confirm the inherited type flows through.
- **Re-run example validation** for the four edited example instances; they
  should pass against the new `CodeableConcept` type.
- **Edge cases:**
  - A `Provenance.authorization` element with multiple codings in one
    `CodeableConcept` (current examples each have one coding only — adding a
    multi-coding example is optional but would exercise the new shape).
  - A `Provenance.authorization` paired with a populated `Provenance.basedOn`
    (the WG's intended substitute for the lost `Reference` choice). Consider
    adding or updating an example that demonstrates this pairing to make the
    deferral explicit.
- **Interoperability:**
  - Publish a brief migration note for implementers who tracked earlier R6
    ballots: `Provenance.authorization.concept.coding[…]` collapses to
    `Provenance.authorization.coding[…]`, and any
    `Provenance.authorization.reference` usage moves to `Provenance.basedOn`.
  - Confirm the FHIRPath / search-parameter expressions across the repo still
    parse with the narrower type (no SP currently targets this element, so
    risk is low).
