## Summary

The ticket identifies an inconsistency between the *definition* of `AuditEvent.agent.type` (which says "Functional Role") and the *contents* of its bound ValueSet `participation-role-type`, which includes codes from `extra-security-role-type` (e.g., `humanuser`, `authserver`, `datacollector`). These codes describe *what kind of agent* something is (agent classification), not *what functional role* it played in the event.

In practice — especially through IHE BALP — `agent.type` has been used with classification codes like `humanuser` and `client`, while `agent.role` has become a catch-all for any additional role information beyond its intended "structural role" scope.

## Resolution Description

> Remove the `extra-security-role-type` from the `valueset-participation-role-type`.

This aligns the ValueSet contents with the element definition: `agent.type` should carry *functional* participation roles, not agent classification codes.

## Feature Proposal

Remove the `<include>` block for `http://terminology.hl7.org/CodeSystem/extra-security-role-type` from the `participation-role-type` ValueSet. This is a targeted ValueSet composition change — the CodeSystem itself (`extra-security-role-type`) in HL7/UTG is **not** being deleted or deprecated, it is simply no longer included in this particular ValueSet. The ValueSet will still include codes from `provenance-agent-type` and DICOM CID 402, which contain actual functional participation roles (author, verifier, assembler, etc.).

Additionally, the 8 AuditEvent examples in the FHIR core spec that currently use `extra-security-role-type#humanuser` in `agent.type` should be evaluated. Since the binding is `preferred` (not `required`), using codes outside the ValueSet is technically valid, but the examples should ideally demonstrate best practice by using codes from the preferred ValueSet.

### Cross-Resource Impact

The `participation-role-type` ValueSet is bound by **multiple resources** beyond AuditEvent:

| Resource | Element | Binding Strength |
|----------|---------|------------------|
| `AuditEvent.agent.type` | `agent.type` | preferred |
| `Provenance.agent.type` | `agent.type` | example |
| `Condition.participant.function` | `participant.function` | extensible |
| `AllergyIntolerance.participant.function` | `participant.function` | extensible |
| `Consent.provision.actor.role` | `provision.actor.role` | extensible |
| `Contract` | (performer function) | extensible |
| `FamilyMemberHistory` | (participant function) | extensible |

Removing `extra-security-role-type` from the ValueSet composition reduces the available codes for **all** of these bindings. However:
- The `extra-security-role-type` codes (`humanuser`, `authserver`, `datacollector`, `dataprocessor`, `datasubject`) are **agent classification concepts** that are semantically inappropriate as *functional participation roles* for any of these elements.
- The ValueSet is marked **extensible** (`valueset-extensible` extension = `true`), so implementations can still use these codes if needed via CodeableConcept.
- The Provenance binding is `example` strength, so no conformance impact there.

### Downstream / IHE BALP Impact

IHE's Basic Audit Log Patterns (BALP) extensively uses `extra-security-role-type#humanuser` in `agent.type`. After this change, BALP profiles will be using codes outside the preferred ValueSet. This is valid (preferred binding allows other codes), but BALP may want to evaluate whether to:
1. Continue using `humanuser` (still conformant with preferred binding)
2. Migrate to DICOM or other functional role codes
3. Move `humanuser` usage from `agent.type` to `agent.role`

## Detailed Implementation Plan

### Step 1: Modify the ValueSet (HL7/fhir)

**File:** `source/terminologies/valueset-participation-role-type.xml`

Remove the third `<include>` block (lines 43–45) that pulls in `extra-security-role-type`:

```xml
<!-- REMOVE THIS BLOCK -->
<include>
  <system value="http://terminology.hl7.org/CodeSystem/extra-security-role-type"/>
</include>
```

The resulting `<compose>` section should contain only two `<include>` entries:
1. `http://terminology.hl7.org/ValueSet/provenance-agent-type` (functional participation codes)
2. DICOM `http://dicom.nema.org/resources/ontology/DCM` CID 402 (DICOM audit trail codes)

### Step 2: Review AuditEvent Examples (HL7/fhir)

Eight AuditEvent examples currently use `extra-security-role-type#humanuser` in `agent.type`. The workgroup should decide whether to:

**Option A — Leave examples as-is:** The `preferred` binding strength allows codes outside the ValueSet. The examples still validate. Add a comment noting the code is from a system not in the preferred ValueSet.

**Option B — Replace with a DICOM participation code:** Replace `humanuser` with an appropriate DICOM CID 402 code that is a functional participation role (e.g., `110150` for "Application" or a more specific DICOM role code if the agent's function can be determined from context).

**Option C — Move to `agent.role`:** Move the `humanuser` coding from `agent.type` to `agent.role` (which is a better semantic fit for agent classification), and either leave `agent.type` empty or populate it with a true functional role code from `provenance-agent-type`.

**Recommendation:** Option C most faithfully implements the resolution's intent (separating functional roles from agent classification). However, this is a workgroup decision.

**Affected example files (all in `source/auditevent/`):**
- `auditevent-example-error.xml` (line 50)
- `audit-event-example-login.xml` (line 30)
- `audit-event-example-logout.xml` (line 30)
- `audit-event-example-vread.xml` (line 30)
- `audit-event-example-search.xml` (line 31)
- `audit-event-example-media.xml` (line 49)
- `audit-event-example-pixQuery.xml` (line 65)
- `audit-event-example-create-traceID.xml` (line 31)

### Step 3: No Changes to HL7/UTG

The `CodeSystem-extra-security-role-type.xml` in `input/sourceOfTruth/fhir/codeSystems/` is **not** modified or deprecated. The codes remain available for use; they are simply no longer included in the `participation-role-type` ValueSet composition.

### Step 4: Coordinate with Related Ticket FHIR-53882

[FHIR-53882](https://jira.hl7.org/browse/FHIR-53882) ("Bring more clarity on agent.role and agent.type") is also resolved as "Persuasive with Modification" for the Security work group. That ticket focuses on improving documentation/definitions for `agent.type` vs `agent.role`. The two tickets are complementary:
- **FHIR-55889** (this ticket): Removes inappropriate codes from the ValueSet
- **FHIR-53882**: Clarifies the narrative/documentation

These should be coordinated to ensure the updated definitions align with the cleaned-up ValueSet.
