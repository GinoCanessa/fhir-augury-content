## Resolution Summary

Resolution: **Persuasive with Modification**.

The Security WG agreed the section is confusing and approved targeted prose
edits to security-labels.html § "Core Security Labels":

1. **Rename the section** from "Core Security Labels" to **"Guidance on
   Security Vocabulary Use"** and **mark it as Informative**.
2. **Rewrite the opening paragraph** so that it stops calling the labels
   "core" and stops asserting a SHOULD-use rule for all FHIR applications.
   The exact diff applied is:

   > "This specification defines a set of ~~core~~ security
   > ~~labels~~ **vocabulary** for ~~all~~ FHIR systems.
   > ~~All conformant FHIR Applications SHOULD use these labels where
   > appropriate.~~ For all these labels, how they are operationalized -
   > their use and interpretation - is subject to the applicable Mutual
   > Trust Framework agreement as described above."

3. **Re-order and rename the three patterns of use** from
   `(1) Bundle, (2) Context, (3) Meta Data` to
   **`(1) Meta Data, (2) Meta Bundle, (3) Context`** — including dropping
   the word "Label" from "Security Label vocabulary" → "Security
   vocabulary", and renaming the second pattern "Bundle" → "Meta Bundle".
4. **Re-order the descriptive paragraphs** that follow so they appear in
   the new order (Meta Data first, then Meta Bundle, then Context).
5. **Have the "Context" paragraph point at the example use of
   PurposeOfUse**, so readers can see how a request-level security label
   is actually carried in current FHIR.

No code, schema, or terminology changes are required. The remainder of the
section (the table of "Name/Tag" entries from "Purpose of Use" downward,
the Break-the-Glass section, and the HCS section) is untouched.

## Feature Proposal

### Problem Statement

The reporter (EssienGe) found § 6.1.1.2 "Core Security Labels" in
security-labels.html confusing on three counts:

- The phrase "set of core security labels" is undefined — it is unclear
  whether the table that follows lists *the* core labels, or just
  categories whose underlying codes are all "core", and there is no stated
  contrast between "core" and "non-core".
- The "three patterns of use" sentence (Bundle / Context / Meta Data)
  reads like a definition of *use cases for security labels in general*
  but is buried inside a section nominally about "core" labels, so the
  paragraph feels misplaced (the reporter suggests it belongs near
  § 6.1.1.1, which today only describes the Meta Data case).
- The distinction between pattern (1) "Bundle" and pattern (3) "Meta
  Data" is not clear (both ultimately use `meta.security`), and the
  spec does not say whether pattern (2) "Context" is supported in the
  current FHIR spec at all, or how.

John Moehrke (assignee) and the WG accepted that the existing section
overstates a "core" concept that the spec does not actually define and
underexplains the three usage patterns. Comments also note that the spec
is historically inconsistent in using "label" vs "tag", but the
resolution does not attempt to fix that broader inconsistency here.

   > Security/Privacy considerations of a data set, and (3) Context:

### Design Rationale

- **Why drop "core" from the section title and prose.** The spec does
  not actually define a closed set of "core" security labels — the table
  in this section enumerates *category* terminologies (PurposeOfUse,
  Confidentiality, sensitivity codes, integrity codes, Refrain,
  Obligation, etc.) sourced from external HL7 terminology and from the
  HCS. There is no companion "non-core" section, so calling these "core"
  invites exactly the question the reporter raised. The resolution
  reframes the section as **guidance** on how the existing security
  vocabulary is *used*, which matches what the prose actually does.
- **Why mark Informative.** The content is descriptive guidance about
  how to apply already-defined vocabularies; it is not new normative
  conformance language. Removing the "All conformant FHIR Applications
  SHOULD use these labels where appropriate" sentence is consistent with
  this — there was no underlying conformance test to enforce it.
- **Why reorder Meta Data → Meta Bundle → Context.** The reporter
  observed (and the WG agreed) that Meta Data is the most common
  case, Bundle is a specialisation of Meta Data on `Bundle.meta.security`
  (hence rename to "Meta Bundle" to make the relationship explicit), and
  Context is the least common / most operationally constrained case.
  Listing them in that order matches a reader's likely reading path.
- **Why point Context at PurposeOfUse.** PurposeOfUse is the one
  request-level security tag that *does* have established usage in
  current FHIR (in `_security` search params, in SMART/UDAP scopes, and
  in OAuth assertions). Pointing the reader at that example answers
  EssienGe's question "Is the (2) context case supported in the current
  FHIR spec, and if so, how?" without expanding the resolution into a
  larger feature.
- **Alternatives considered and rejected.** The original ticket
  suggested moving the "three patterns of use" paragraph into § 6.1.1.1.
  The resolution instead keeps it in place and rewrites the surrounding
  framing — a smaller, lower-risk edit that preserves all inbound
  `#core` anchors and keeps the table of vocabularies adjacent to its
  introductory paragraph.

### Open Questions

1. **Subsection-level "Informative" marker mechanism.** The resolution
   says "Mark as Informative", but `source/security-labels.html` is
   declared Normative at the page level (line 15). The FhirCore
   playbook does not document an unambiguous, agreed pattern for
   downgrading a single subsection's standards-status inside a
   Normative page (no surrounding example was found in the cached
   clone via grep). Confirm with the Security WG / FHIR Methodology /
   FMG which marker to apply (inline `<a
   href="versions.html#std-process">Informative</a>` badge, a
   `[%stdstatus … %]` macro, or a class-based marker such as
   `class="mixed-fmm-info"`).
2. **Target of the PurposeOfUse pointer in the Context paragraph.**
   The resolution says "have the context paragraph point at the
   example use of PurposeOfUse" but does not specify whether it should
   link (a) the in-page Purpose-of-Use table row, (b) the external
   `terminology.hl7.org` ValueSet page already used by that table row,
   or (c) a worked example resource elsewhere in the spec. Task 7
   defaults to (a) with fallback to (b); confirm intent before merge.
3. **"Label" vs "tag" inconsistency.** John Moehrke notes in the
   ticket comments that the spec is historically inconsistent in using
   "label" vs "tag". The approved resolution does not attempt to
   reconcile this beyond dropping "Label" from "Security Label
   vocabulary" → "Security vocabulary". Do not silently broaden the
   edit to other occurrences of "label/tag" elsewhere in
   `security-labels.html` or `security.html`; if reconciliation is
   wanted, it should be a separate ticket.
