## Resolution Summary

**Resolution:** Persuasive — "Agree to update binding to THO value set."

The FHIR Core `Goal Priority` ValueSet (`http://hl7.org/fhir/ValueSet/goal-priority`) is essentially a duplicate of the THO-maintained ValueSet (`http://terminology.hl7.org/ValueSet/goal-priority`, Goal priority — HL7 Terminology v7.0.1). Patient Care (2026-02-12, jlyle) agreed with the reporter's proposal to:

1. Re-bind `Goal.priority` (and, by extension per Bryn's 2025-10-10 comment, the equivalent binding on `PlanDefinition.goal.priority`) to the THO ValueSet.
2. Remove the now-redundant FHIR-core `goal-priority` ValueSet.

CDS (PlanDefinition owner) explicitly agreed to follow Patient Care's decision on the shared binding, so the change includes PlanDefinition.

## Feature Proposal

### Problem Statement

FHIR Core currently ships `ValueSet/goal-priority`, whose sole `compose.include` is the THO CodeSystem `http://terminology.hl7.org/CodeSystem/goal-priority` (verified at `source/goal/valueset-goal-priority.xml:39-42`). The same three-code set is already published by THO as `http://terminology.hl7.org/ValueSet/goal-priority`. Maintaining two ValueSet definitions for an identical concept set is unnecessary duplication, creates a divergence risk between FHIR release cycles and THO releases, and contradicts the broader Patient Care / THO policy of hosting terminology in THO.

### Proposed Change

Re-point every `Goal Priority` binding in FHIR Core at the THO ValueSet and retire the core copy. Specifically:

1. **Goal.priority** (`source/goal/structuredefinition-Goal.xml`, element id `Goal.priority`, line 212): change the binding's `<valueSet>` from `http://hl7.org/fhir/ValueSet/goal-priority` to `http://terminology.hl7.org/ValueSet/goal-priority`. Preserve strength (`preferred`), `bindingName` (`GoalPriority`), and description.
2. **Goal.acceptance.priority** (`source/goal/structuredefinition-Goal.xml`, element id `Goal.acceptance.priority`, line 386): same URL swap. Preserve strength (`example`), `bindingName`, and description.
3. **PlanDefinition.goal.priority** (`source/plandefinition/structuredefinition-PlanDefinition.xml`, element id `PlanDefinition.goal.priority`, line 974): same URL swap. Preserve strength (`preferred`), `bindingName` (`GoalPriority`), and description.
4. **Retire the core ValueSet**: delete `source/goal/valueset-goal-priority.xml` and remove its OID row from `source/oids.ini` (line 463, `http://hl7.org/fhir/ValueSet/goal-priority = 2.16.840.1.113883.4.642.3.273`). No other `source/*.ini`, `source/*.html`, or resource-list registration refers to `valueset-goal-priority`.
5. Leave the Goal example (`source/goal/goal-example.xml:23`) unchanged — it references the THO **CodeSystem** (`http://terminology.hl7.org/CodeSystem/goal-priority`), not the ValueSet URL, and remains valid against the new binding.

The post-change expected binding element on all three elements:

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="GoalPriority"/>
  </extension>
  <strength value="..."/>  <!-- unchanged per element -->
  <description value="..."/> <!-- unchanged per element -->
  <valueSet value="http://terminology.hl7.org/ValueSet/goal-priority"/>
</binding>
```

### Design Rationale

- The core ValueSet is a pure pass-through (`compose.include` with no filter) of the THO CodeSystem, so every bound instance is semantically identical under both URLs — no instance-data impact.
- Hosting the ValueSet in THO aligns with the established pattern for shared, cross-spec vocabulary and the explicit direction from the ticket's `THO` label. Binding `Goal.priority` to the THO ValueSet means future code additions/retirements flow through THO's release process rather than requiring a FHIR-core republish.
- Applying the same swap to `PlanDefinition.goal.priority` keeps the two resources consistent — Bryn's comment and CDS's deferral make this explicit. `Goal.acceptance.priority` (same `GoalPriority` bindingName) is included for the same reason.
- `PlanDefinition.action.priority` (line 1415) is *not* in scope — it binds to `request-priority` (routine/urgent/asap/stat), a different ValueSet.

- **Role:** Sole repository holding the bindings and the retiring ValueSet. All specification-level changes live here.

  - `source/oids.ini` — remove line 463 (`http://hl7.org/fhir/ValueSet/goal-priority = 2.16.840.1.113883.4.642.3.273`).
- **Change Scope:** Minor (three one-line URL edits, one file deletion, one ini-line removal; no instance-format change, no new elements, no cardinality change).

### Breaking Changes

- **Binding URL change is a conformance-surface change.** Implementers that key off the *ValueSet canonical URL* (e.g., validators with pinned value-set caches, downstream IGs that extend these bindings) will see a new canonical after this change. Because the two ValueSets enumerate the same codes (both pull from the identical THO CodeSystem), instance conformance of existing resources is **not** affected — any `Coding.system = http://terminology.hl7.org/CodeSystem/goal-priority` with the three defined codes continues to validate.
- **ValueSet retirement (`http://hl7.org/fhir/ValueSet/goal-priority`) is a breaking change** for any external IG or implementation that binds to that URL directly or expands it through the terminology server. OID `2.16.840.1.113883.4.642.3.273` is similarly retired from the FHIR namespace. Mitigation: THO publishes the equivalent ValueSet; consumers should migrate to the THO canonical.
- Search parameters: no `Goal` or `PlanDefinition` search parameter targets `priority` in a way that depends on the ValueSet URL (verified in the Goal/PlanDefinition search-params bundles under their respective folders). No search-parameter change needed.
