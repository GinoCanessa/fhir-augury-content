## Resolution Summary

**Resolution:** Persuasive with Modification (Patient Care vote, 2026-02-12 — Rob Hausam / Emma Jones: 0-0-4).

The work group adopted Kevin O'Donnell's clarification proposal with two
small editorial modifications. The agreed resolution text is to **replace
the existing `Condition.verificationStatus` comment** with:

> *Although the Terminology Binding is Required, the data type is
> CodeableConcept which allows multiple codings. **If populated**, one
> code from the binding must be present, but additional codes with
> greater specificity, drawn from **other terminologies**, may also be
> present if needed.*

The two modifications relative to the reporter's draft are:

1. Insert "**If populated**," after "multiple codings." to make explicit
   that the requirement only applies when the element actually carries a
   value (the element itself is `0..1`).
2. Replace the SNOMED-specific phrase ("perhaps drawn from SNOMED") with
   the terminology-neutral wording "drawn from other terminologies", so
   the comment does not privilege one code system.

The reporter's three other questions — adding similar text on
`Condition.clinicalStatus`, the rationale for the abdominal-pain ED
example, and search semantics for absent `verificationStatus` — were
**not** taken up by the resolution. Only the `verificationStatus`
comment is to change.

## Feature Proposal

### Problem Statement

`Condition.verificationStatus` is bound to the value set
`http://hl7.org/fhir/ValueSet/condition-ver-status` with binding
strength **required**, yet the existing element comment talks about
using SNOMED codes for additional specificity. This appears to
contradict the required binding and confuses implementers about whether
non-binding codes are permitted at all on this element.

The pre-existing comment also mixed two separate ideas — when the
element may be omitted ("not required … abdominal pain in the ED") and
when additional codings may be supplied — which made the guidance
harder to follow than necessary.

The reporter (Kevin O'Donnell) proposed a clarification that explains
the well-known FHIR pattern: a `CodeableConcept` with a required binding
allows additional `coding` entries beyond the bound one, provided one of
the bound codes is present.

### Proposed Change

Make a **single, non-substantive** prose change inside the FHIR Core
spec source:

1. In `source/condition/structuredefinition-Condition.xml`, on the
   `<element id="Condition.verificationStatus">` block, replace the
   value of the `<comment>` child with the resolution text exactly:

   ```
   Although the terminology binding is required, the data type is
   CodeableConcept which allows multiple codings. If populated, one
   code from the binding must be present, but additional codes with
   greater specificity, drawn from other terminologies, may also be
   present if needed.
   ```

   No other facets of the element change — `path`, `short`,
   `definition`, `min` (`0`), `max` (`1`), `type` (`CodeableConcept`),
   `binding` (required → `condition-ver-status`), `mustSupport`,
   `isModifier`, `isSummary`, mappings, etc. all stay as-is.

2. On the same element, append the ticket key to the
   `committee-notes` build extension to record the change for ballot
   provenance, e.g.:

   ```xml
   <extension url="http://hl7.org/fhir/build/StructureDefinition/committee-notes">
     <valueString value="GF#11281, GF#16193, J#28458, J#54914"/>
   </extension>
   ```

3. In `source/condition/condition-introduction.xml`, list this ticket
   under the **Non-Substantive Changes** bullet of the "Note to
   Balloters" block (and in the "All Changes" block) for the next
   ballot, with the wording:

   > [FHIR-54914](https://jira.hl7.org/browse/FHIR-54914): Clarified
   > Condition.verificationStatus comment

This change is **documentation only**. There is no change to
cardinality, type, binding strength or value-set, search parameters,
invariants, examples, or any other normative or behavioural artefact.

`Condition.clinicalStatus` is **not** in scope for this ticket; do not
edit it as part of this fix even though the reporter raised it.

### Design Rationale

- **Aligns with the FHIR rule for `CodeableConcept` + required
  binding.** The general FHIR pattern (documented for terminology
  bindings) is that a required binding constrains at least one
  `coding`, while the `CodeableConcept` container may still carry
  additional codings from other terminologies. The new comment simply
  states this rule on the element where implementers were getting
  confused.
- **Removes the SNOMED-specific framing.** FHIR Core comments avoid
  privileging a single code system unless the binding itself does so.
  The "other terminologies" wording keeps the door open to whichever
  external terminology an implementer needs (SNOMED CT, ICD-10, local
  codes, etc.).
- **Adds "If populated" to disambiguate the `0..1` cardinality.** The
  element is optional, so the requirement that "one code from the
  binding must be present" only bites when the element is actually
  used. Without the qualifier, readers could misread the comment as
  making the element effectively mandatory.
- **Conservative scope.** The work group declined to extend the same
  text to `clinicalStatus` or to redesign the ED-abdominal-pain
  example, treating those as separate questions. This plan respects
  that decision and limits the patch surface accordingly.
