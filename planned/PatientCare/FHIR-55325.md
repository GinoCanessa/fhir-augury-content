## Resolution Summary

The Patient Care WG (3/5/2026 call) found the request **Not Persuasive with
Modification**. The escape valve concept the reporter asked for already
exists in the bound value set: `FamilyMemberHistory.status` includes the
code `health-unknown` ("Health information for this family member is
unavailable/unknown."). No new code is needed in the
`history-status` CodeSystem / ValueSet.

The modification required by the resolution is a **documentation update**
to the Notes section of the FamilyMemberHistory resource page. The
existing "Reviewed, No Information Available for a Family Member" note
currently directs implementers to `FamilyMemberHistory.dataAbsentReason`;
it must be re-pointed to `FamilyMemberHistory.status = health-unknown`.

Exact replacement text from the resolution:

> **Reviewed, No Information Available for a Family Member**
>
> When an individual family member's history is not available,
> FamilyMemberHistory.status can be used to indicate health-unknown.

## Feature Proposal

### Problem Statement

The reporter (Lloyd McKenzie, ticket comment 289) observed that
`FamilyMemberHistory.status` is a 1..1 element with a `required` binding
to `ValueSet/history-status` and asked for an "unknown" / "other" escape
valve so the element could always be populated when no information is
available about a family member.

On review, the value set already provides that escape valve via the
`health-unknown` code (defined in `CodeSystem/history-status` as "Health
information for this family member is unavailable/unknown."). However,
the FamilyMemberHistory Notes page does not surface this option:
the "Reviewed, No Information Available for a Family Member" note instead
points implementers to the optional `FamilyMemberHistory.dataAbsentReason`
element, which obscures the simpler in-binding solution and reinforces
the perception that there is no escape valve on `status`.

### Proposed Change

Single, prose-only change in the FamilyMemberHistory Notes file.

- **File:** `source/familymemberhistory/familymemberhistory-notes.xml`
- **Section:** the `<p><b>Reviewed, No Information Available for a Family Member</b></p>` block followed by its descriptive `<p>` paragraph (currently referencing `FamilyMemberHistory.dataAbsentReason`).
- **Replace** the current paragraph:
  > When an individual family member's history is not available,
  > `FamilyMemberHistory.dataAbsentReason` can be used to indicate why
  > that family member's history is not available (e.g. subject unknown).
- **With** the resolution-approved text:
  > When an individual family member's history is not available,
  > `FamilyMemberHistory.status` can be used to indicate `health-unknown`.

No structural changes:

- `CodeSystem/history-status` — no edit. The `health-unknown` concept is
  already present at version 6.0.0.
- `ValueSet/history-status` — no edit. It is `immutable=true` and
  composed solely from `http://hl7.org/fhir/history-status`, so the
  `health-unknown` code is already in the bound value set.
- `StructureDefinition/FamilyMemberHistory` — no edit. The cardinality
  (1..1), type (`code`) and `required` binding of `status` are unchanged;
  `dataAbsentReason` (0..1) is left in place — the resolution does not
  remove it, only stops promoting it as the recommended escape valve in
  this scenario.

### Design Rationale

- **Use the binding's own escape valve before introducing parallel ones.**
  Where a required binding already includes an "unknown" concept,
  Patient Care prefers the in-binding solution to keep instances simple
  and to keep the bound value set the authoritative place to look for
  status semantics.
- **Consistency with the Notes page's other bullets.** The neighbouring
  bullets ("Not Reviewed, Not Asked", "Reviewed, None Identified for
  Family", "Reviewed, No Known Problems or Negated Condition…") each
  point implementers at a single canonical mechanism. Re-pointing the
  "No Information Available" bullet at `status = health-unknown` matches
  that pattern and removes the implicit recommendation to use
  `dataAbsentReason` as the primary escape valve.
- **Avoids a breaking change.** Adding a new required code, changing
  cardinality, or weakening the binding strength would all be
  potentially breaking. A documentation-only change is the least
  invasive option that addresses the reporter's concern.
- **Alternatives considered (and rejected):**
  - Adding `unknown` / `other` codes to `CodeSystem/history-status` —
    rejected because `health-unknown` already covers the use case and a
    second synonym would create redundant codes in a normative system.
  - Loosening the binding strength on `FamilyMemberHistory.status` —
    rejected; the WG view per the resolution is that no escape valve is
    missing.

### Open Questions

- **Should the `dataAbsentReason` element itself be deprecated or
  flagged?** The resolution only changes the Notes guidance. Leaving
  `FamilyMemberHistory.dataAbsentReason` in the StructureDefinition is
  the safe interpretation, but if Patient Care actually wants the
  element removed or marked as the secondary mechanism, a follow-up
  ticket is needed. Confirm with the WG before broadening the change.
- **Should an inline anchor / link to the `health-unknown` concept be
  added in the new paragraph?** The resolution gives plain prose; some
  neighbouring notes use anchor links to other resources. Either is
  acceptable; default to matching the existing style of the
  surrounding paragraphs (plain element/code names, no anchor) unless
  the WG asks otherwise.
