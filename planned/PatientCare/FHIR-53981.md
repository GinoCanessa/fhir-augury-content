## Resolution Summary

Resolution: **Persuasive with Modification**. Per the Patient Care call on
2026-02-19 (michelle.m.miller's applied-vote comment), the agreed change
is narrower than the submitter's "split into four elements" proposal:

- Rename `Goal.statusReason` → `Goal.lifecycleStatusReason` (it already
  conveys the reason for the lifecycle status; the rename removes the
  ambiguity introduced by the bare `status*` prefix).
- Rename `Goal.statusDate` → `Goal.achievementStatusDate` (it already
  conveys when the achievement status took effect; **also fix the typo**
  in the existing short — "achievment" → "achievement").
- Reorder the elements so `Goal.lifecycleStatusReason` immediately
  follows `Goal.lifecycleStatus`, and `Goal.achievementStatusDate`
  immediately follows `Goal.achievementStatus` (matching the Event
  pattern's "status, then statusReason" ordering convention).

Explicitly **out of scope** of this resolution (Patient Care declined):
renaming `Goal.acceptance.status`, adding a reason for achievement,
adding a date for lifecycle, or splitting into four parallel
date/reason elements.

## Feature Proposal

### Problem Statement

`Goal` carries two parallel status concepts —
`Goal.lifecycleStatus` (proposed | planned | accepted | active | …) and
`Goal.achievementStatus` (in-progress | improving | achieved | …). It
also defines two "status*"-prefixed siblings:

- `Goal.statusDate` — short "When goal achievment status took effect"
  (sic), defined as "Identifies when the current achievement status took
  effect."
- `Goal.statusReason` — short "Reason for current lifecycle status",
  defined as "Captures the reason for the current lifecycle status."

Despite sharing the `status*` prefix, the date applies to
`achievementStatus` and the reason applies to `lifecycleStatus`. The
ticket reporter (AndyHampton) argues this is too prescriptive and easy
to misuse, and proposed splitting into four elements (one date and one
reason for each of the two statuses) or introducing an inline data
type. There is also a long-standing typo ("achievment") in the
`statusDate` short.

The resolution adopts a lighter-weight fix: rename the two existing
elements so their names announce which status they apply to, and reorder
them next to the status they describe. The two "absent" combinations
(date for lifecycle, reason for achievement) are explicitly deferred —
no current demand.

### Proposed Change

     `definition`, and `comment` all remain unchanged. The

   - **Move** the element so that it immediately follows
     `<element id="Goal.lifecycleStatus">` (currently at line 115).

   - `<element id="Goal.statusDate">` → `<element id="Goal.achievementStatusDate">`
   - `<path value="Goal.statusDate"/>` → `<path value="Goal.achievementStatusDate"/>`

   `source/goal/invariant-tests/gol-1.f1.fail.xml` — rename

```xml
    <display value="In Progress"/>

…
<achievementStatusDate value="2016-02-14"/>
<lifecycleStatusReason>
  <text value="Patient wants to defer weight loss until after honeymoon."/>
</lifecycleStatusReason>
```

### Design Rationale

- The existing definitions already describe the right concepts — the
  problem is purely naming. Renaming (rather than adding new elements)
  preserves intent and avoids introducing unused parallel paths the
  WG explicitly declined to add.
- The Event pattern in FHIR places `statusReason` immediately after
  `status`. Reordering `lifecycleStatusReason` after `lifecycleStatus`
  and `achievementStatusDate` after `achievementStatus` aligns Goal
  with that pattern at the per-status-pair level.
- Patient Care declined renaming `Goal.acceptance.status` because the
  enclosing `acceptance` BackboneElement already disambiguates it.
- Patient Care declined adding `lifecycleStatusDate` and
  `achievementStatusReason` because no implementer has requested them;
  the submitter's concern was disambiguation, not coverage.
- The four-way split / inline data-type alternative (the submitter's
  preferred option) was considered but rejected on the call as larger
  than needed for the actual use case.
- These are breaking, non-passive changes — see the Breaking Changes
  section below.

### Breaking Changes

This resolution is **breaking / non-passive**, and the WG comment
explicitly acknowledged this trade-off:

- `Goal.statusDate` → `Goal.achievementStatusDate`. Existing
  instances using the old element name will not validate against the
  new StructureDefinition; existing FHIRPath / search expressions
  referencing `Goal.statusDate` must be updated.
- `Goal.statusReason` → `Goal.lifecycleStatusReason`. Same caveat.
- The two elements move position in the `<differential>`. While
  position is not normative, downstream profiles that slice the
  resource by ordinal index (rare) would be affected.
- No type, cardinality, modifier, summary-flag, or binding-strength
  changes — so the only break is the path/element name (and ordering).

### Related Specifications
