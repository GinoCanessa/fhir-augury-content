## Resolution Summary

The Patient Care work group's 11/20/2025 conference call applied the following
resolution (Persuasive with Modification) to FHIR-50413:

- **Name:** Remove `previous`; add `comparesTo`.
- **Cardinality:** `0..*`.
- **Definition (description):** *"Reference(s) to other assessments that were
  conducted on this patient. Assessments are often/usually ongoing in nature; a
  care provider (practitioner or team) will make new assessments on an ongoing
  basis as new data arises or the patient's conditions changes."*
- **References (target types):** `Reference(RelatedArtifact | ClinicalAssessment)`
  — quoted verbatim from the applied vote. `ClinicalAssessment` is not a real
  FHIR resource, and the prior 11/13 vote phrased the option as
  *"Replace ClinicalAssessment with RelatedArtifact – OR – Add RelatedArtifact
  to Reference (RelatedArtifact | ClinicalAssessment)"*. The current
  `ClinicalImpression.previous` element targets `ClinicalImpression`. This
  almost certainly means the new `comparesTo` element should target
  `Reference(RelatedArtifact | ClinicalImpression)` (i.e., add `RelatedArtifact`
  to the existing `ClinicalImpression`-only reference). See **Open Questions**.

The 11/13 vote also tracked a "Description (to do next time)" and "Comments
(to do next time)" continuation that the 11/20 motion ultimately replaced
with the simpler "remove previous / add comparesTo" wording above.

## Feature Proposal

### Problem Statement

`ClinicalImpression.previous` (FHIR Core) is currently a single-cardinality
reference to one earlier `ClinicalImpression`:

```
<element id="ClinicalImpression.previous">
  <path value="ClinicalImpression.previous"/>
  <short value="Reference to last assessment"/>
  <definition value="A reference to the last assessment that was conducted on
                     this patient. ..."/>
  <comment value="It is always likely that multiple previous assessments exist
                  for a patient. ..."/>
  <min value="0"/>
  <max value="1"/>
  <type>
    <code value="Reference"/>
    <targetProfile
      value="http://hl7.org/fhir/StructureDefinition/ClinicalImpression"/>
  </type>
</element>
```
(`source/clinicalimpression/structuredefinition-ClinicalImpression.xml`,
lines 321–332.)

The submitter (Diana Ovelgoenne) and discussion participants (Stephen,
Michelle Miller) noted that real-world clinical work — chronic disease
follow-up (oncology, cardiology, orthopedics), MRI/imaging review, etc. —
routinely compares the current impression against **multiple** prior
assessments, not just the last one. The element name `previous` and the
1..1-style "last assessment" framing both fight that workflow. The closely
related ticket [FHIR-46614](https://jira.hl7.org/browse/FHIR-46614)
("cardinality of ClinicalImpression.previous should be 0..*") already
captures the cardinality side of the same gap.

The originally-suggested fix in this ticket was to copy the `relatesTo`
backbone pattern from `DiagnosticReport`. The work group rejected the full
backbone and instead settled on a flat 0..* reference renamed to
`comparesTo`, motivated by the "compare to multiple historical assessments"
use case and a preference for `comparesTo`/`comparison` over the more
abstract `relatesTo`.

### Proposed Change

In `HL7/fhir`:

1. **Delete the `ClinicalImpression.previous` element** from
   `source/clinicalimpression/structuredefinition-ClinicalImpression.xml`
   (the `<element id="ClinicalImpression.previous">` block at lines 321–332
   shown above, including its surrounding whitespace).

2. **Add a new `ClinicalImpression.comparesTo` element** in the same
   StructureDefinition, in the same position the deleted `previous` element
   occupied (between `ClinicalImpression.note` and
   `ClinicalImpression.problem`, preserving the existing element ordering of
   the resource). Concrete values:

   ```xml
   <element id="ClinicalImpression.comparesTo">
     <path value="ClinicalImpression.comparesTo"/>
     <short value="Other assessments compared with this one"/>
     <definition value="Reference(s) to other assessments that were conducted
                        on this patient. Assessments are often/usually ongoing
                        in nature; a care provider (practitioner or team) will
                        make new assessments on an ongoing basis as new data
                        arises or the patient's conditions changes."/>
     <min value="0"/>
     <max value="*"/>
     <type>
       <code value="Reference"/>
       <targetProfile
         value="http://hl7.org/fhir/StructureDefinition/RelatedArtifact"/>
       <targetProfile
         value="http://hl7.org/fhir/StructureDefinition/ClinicalImpression"/>
     </type>
   </element>
   ```

   Notes:
   - `RelatedArtifact` is a **datatype**, not a resource. It cannot be a
     `Reference` `targetProfile`. The applied wording
     `Reference(RelatedArtifact | ClinicalAssessment)` is therefore not
     literally implementable as a Reference. Two possible reads, both flagged
     in **Open Questions**:
       (a) The element should be a `RelatedArtifact` complex type
           (cardinality `0..*`) instead of a `Reference`, so that each entry
           carries a `RelatedArtifact.type` code (e.g., `predecessor`,
           `derived-from`) plus the `resource` reference to the prior
           `ClinicalImpression`. This matches the 11/13 discussion of "which
           code means previous?".
       (b) The element stays a `Reference(ClinicalImpression)` (drop the
           `RelatedArtifact` target) and a separate `relatedArtifact` style
           element is added, or `RelatedArtifact` is exposed via an
           extension. Less likely given the explicit motion text.
   - The 11/20 motion does **not** mention `relatesTo` / removal of the
     `comment` field — keep the existing `<comment>` text (or update it to
     match the new plural framing) per work-group preference; flagged
     below as an open question.

3. **Rename the search parameter** in
   `source/clinicalimpression/bundle-ClinicalImpression-search-params.xml`
   from `previous` to `comparesTo`:
   - `<id value="ClinicalImpression-previous"/>` →
     `<id value="ClinicalImpression-comparesTo"/>`
   - `<valueString value="ClinicalImpression.previous"/>` →
     `<valueString value="ClinicalImpression.comparesTo"/>`
   - `<url value="http://hl7.org/fhir/build/SearchParameter/ClinicalImpression-previous"/>`
     →
     `<url value="http://hl7.org/fhir/build/SearchParameter/ClinicalImpression-comparesTo"/>`
   - `<code value="previous"/>` →
     `<code value="comparesTo"/>`
   - `<expression value="ClinicalImpression.previous"/>` →
     `<expression value="ClinicalImpression.comparesTo"/>`
   - `<description>` value updated to match the new short
     ("Other assessments compared with this one") rather than
     "Reference to last assessment".
   - If the search parameter is intended to also be queryable when the
     element is a `RelatedArtifact` (option (a) above), the FHIRPath
     expression becomes `ClinicalImpression.comparesTo.resource` and the
     `type` may become `reference` over a chained path — confirm with the
     final element shape.

4. **Refresh narrative cross-references to "previous"** in the
   ClinicalImpression authoring folder so the introduction and notes match
   the new element name and intent. In particular,
   `source/clinicalimpression/clinicalimpression-introduction.xml` line 18
   currently reads:

   > "...the clinical impression can explicitly reference both care plans
   > (preceding and resulting) and reference a previous impression that
   > this impression follows."

   Update the trailing clause to reflect that ClinicalImpression can now
   reference *one or more* prior impressions for comparison (e.g., "...and
   reference one or more prior impressions that this impression compares
   against."). The two `<li>` bullets in the same file at lines 34–35 use
   "previous" only as the English word ("the previous question") and do
   not need to change.

### Design Rationale

- **Plural semantics over single "last" semantics.** Both the submitter
  (oncology/cardiac/orthopedic follow-up) and the work-group discussion
  agreed that comparing to several prior assessments is the dominant real
  workflow; `min=0`/`max=1` was the wrong cardinality.
- **`comparesTo` over `previous` / `relatesTo`.** The work group explicitly
  rejected the originally-proposed `relatesTo` (too abstract, overlaps with
  `supportingInfo`) and the existing `previous` (implies a single immediate
  predecessor). Stephen's preference for "comparesTo" was adopted because it
  matches the actual clinical action (comparison), while still naturally
  taking 0..* cardinality.
- **Flat list rather than `DiagnosticReport.relatesTo` backbone.** The
  submitter's original suggestion was to copy the
  `DiagnosticReport.relatesTo` backbone (a code + target). The work group
  did not adopt that; the motion describes a single flat element, not a
  backbone with a `code` child. If the intent is to retain a "kind of
  relationship" code (e.g., `predecessor`), the natural FHIR-idiomatic way
  to do that *without* adding a custom backbone is to type the element as
  `RelatedArtifact` (option (a) under Proposed Change) — which is also the
  most charitable read of the literal motion text.
- **Backward-incompatible rename is acceptable here** because
  `ClinicalImpression` is not Normative (it carries the resource
  Maturity Level / standards-status of the rest of the resource, with
  `previous` not flagged `isSummary` or `isModifier`), so removing
  `previous` and introducing `comparesTo` is within scope for a
  trial-use change.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Source of truth for the `ClinicalImpression` resource
  definition, its search parameters, and its specification narrative. All
  changes for this ticket land here.
- **Affected Files:**
  - `source/clinicalimpression/structuredefinition-ClinicalImpression.xml`
    — remove the `ClinicalImpression.previous` element (lines 321–332) and
    insert a new `ClinicalImpression.comparesTo` element with
    cardinality `0..*` and the resolution's definition text, in the same
    differential position.
  - `source/clinicalimpression/bundle-ClinicalImpression-search-params.xml`
    — rename the `ClinicalImpression-previous` SearchParameter
    (id, url, code, expression, FHIRPath valueString, description) to
    `ClinicalImpression-comparesTo`.
  - `source/clinicalimpression/clinicalimpression-introduction.xml`
    — update the line-18 prose so the "previous impression that this
    impression follows" clause matches the new plural-comparison framing.
- **Change Scope:** Moderate — element rename + cardinality change + type
  re-target on a published (Trial Use) resource, plus a search-parameter
  rename and a narrative tweak. Not a single-line edit; not a sprawling
  cross-resource change.

### Breaking Changes

- **Yes — element rename and removal.** Removing
  `ClinicalImpression.previous` and introducing
  `ClinicalImpression.comparesTo` is a breaking change for any
  implementer that produces or consumes the `previous` element or the
  `previous` search parameter:
  - JSON/XML payloads that carry the `previous` field will no longer
    validate against the next published version of `ClinicalImpression`.
  - The search parameter URL
    `http://hl7.org/fhir/SearchParameter/ClinicalImpression-previous` and
    its `previous` token are renamed; existing client search calls
    (`GET [base]/ClinicalImpression?previous=...`) will fail or return
    no results until updated.
  - The cardinality change (`0..1` → `0..*`) means single-valued client
    parsers that bound the element to a single object must accept arrays.
  - The type change (adding `RelatedArtifact` as a possible target form)
    further widens what an instance may contain.
- **No `isModifier`/`isSummary` impact** — the existing `previous`
  element is neither, so summary-mode and modifier-extension behavior do
  not change.

### Related Specifications

- **`RelatedArtifact` datatype** (`source/datatypes/`) — referenced by the
  new element. No edit required to the datatype itself, but the resource
  becomes a new consumer of `RelatedArtifact` and the consumer list /
  cross-reference page (`source/relatedartifact.html`-style targets, if
  present in this build) may regenerate.
- **`ClinicalImpression`-using IGs / profiles** — any downstream
  Implementation Guide that profiles `ClinicalImpression.previous` (e.g.,
  cardinality tightenings or `mustSupport` flags) will need to be
  republished against the renamed element. Out of scope for this ticket
  but should be flagged in release notes.
- **No FiveWs / event mapping change** —
  `clinicalimpression-event-mapping-exceptions.xml` and
  `clinicalimpression-fivews-mapping-exceptions.xml` do not currently
  call out `previous`, so no mapping rebalance is required.

### Related Tickets

- **[FHIR-46614](https://jira.hl7.org/browse/FHIR-46614):** *"cardinality
  of ClinicalImpression.previous should be 0..*"* — directly overlapping
  request from Russell Leftwich (Triaged, Patient Care, Highest priority).
  This ticket subsumes it: removing `previous` and introducing
  `comparesTo` at `0..*` resolves the cardinality concern and should be

### Terminology Impact

No terminology impact. `ClinicalImpression.comparesTo` is typed as a
`Reference` (and possibly a `RelatedArtifact` complex type — see Open
Questions); neither is a coded element. The four ClinicalImpression
ValueSets in the source folder
(`valueset-clinicalimpression-change-pattern.xml`,
`valueset-clinicalimpression-prognosis.xml`,
`valueset-clinicalimpression-status-reason.xml`,
`valueset-clinicalimpression-status.xml`,
plus `valueset-investigation-sets.xml`) are unaffected, so no UTG change
is needed.

### Open Questions

- **Type shape of `comparesTo`.** The applied resolution literally says
  `Reference(RelatedArtifact | ClinicalAssessment)`. `RelatedArtifact`
  is a datatype (not referenceable) and `ClinicalAssessment` is not a
  FHIR resource — almost certainly a transcription of the discussion
  shorthand for `ClinicalImpression`. The two implementable readings
  (RelatedArtifact-typed list vs. Reference(ClinicalImpression) plus a
  separate RelatedArtifact mechanism) produce different on-the-wire
  representations and different search-parameter expressions. **Patient
  Care must clarify before implementation begins.**
- **Element comment.** Should the existing `<comment value="It is always
  likely that multiple previous assessments exist..."/>` be retained
  verbatim, deleted, or rewritten now that cardinality is `0..*`? The
  motion did not address the comment line.
- **Search-parameter standards-status.** The current
  `ClinicalImpression-previous` parameter carries
  `valueCode value="normative"`. Renaming to `comparesTo` produces a new
  parameter URL; the work group should confirm whether the new
  parameter inherits `normative` or is reset (e.g., to `trial-use`).
- **Relationship to FHIR-46614.** Once this ticket is implemented,
  FHIR-46614 should be closed as resolved by FHIR-50413 (or marked as
  superseded). Not a blocker but should not be forgotten.
- **Briefing parameterisation.** The cached
  `cache/github/repos/HL7_fhir/repo-analysis/briefing.md` was generated
  for a different ticket (FHIR-42980, mapping-language prose). Its
  *general* FhirCore facts (authoring root, generated areas, gotchas
  about `<snapshot>`/spreadsheets) were applied above; the
  ticket-specific "Ticket-Relevant Paths" / "Recommended Change
  Recipes" / "Cross-Repo Touch Points" sections were re-derived for
  FHIR-50413 from the FhirCore playbook conventions and direct clone
  inspection. A future `repo-analysis HL7/fhir if-stale` run scoped to
  FHIR-50413 would replace this re-derivation with persisted
