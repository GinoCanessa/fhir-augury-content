## Resolution Summary

The Patient Care WG voted in the Sept 2025 Thursday Q1 session to advance the
**Procedure** resource to **Normative** status at **FMM 6** in **R6** (the
upcoming FHIR Release). The resolution wording, captured on the ticket, is:

> "In alignment with the Sept 2025 Thurs Q1 vote, Procedure is planned to go
> normative (FMM 6) in R6"
> ([2025 Sept Patient Care WGM Agenda and Minutes](https://confluence.hl7.org/spaces/PC/pages/358875423/2025+Sept+Patient+Care+WGM+Agenda+and+Minutes))

Concretely this requires updating Procedure's standards-status / FMM /
normative-version metadata in both the resource StructureDefinition and the
build's `source/fhir.ini` registries to reflect a Normative resource with FMM 6
whose first-normative version is 6.0.0.

## Feature Proposal

### Problem Statement

Procedure has been at Trial Use for the entire DSTU/STU lifecycle of FHIR. The
current cached HEAD of `HL7/fhir` (`db79dcd`, R6 development branch) reflects an
incomplete advancement: the resource StructureDefinition already declares
`structuredefinition-standards-status = normative`, but the FMM extension is
still `3` (and `source/fhir.ini [fmm]` carries `procedure=4`), and there is no
`structuredefinition-normative-version` extension, no `[normative-package]`
entry, and no `[first-normative-version]` entry for Procedure. The SD's own
`<status>` is also still `draft` (peer normative resources such as Patient sit
at `active`).

The Patient Care WG vote at the Sept 2025 WGM authorises completing the
advancement so that R6 ships Procedure as Normative at FMM 6, consistent with
how Patient/Observation/CapabilityStatement are declared.

### Proposed Change

Apply the standard "advance a resource to normative" edits, mirroring the
treatment of Patient (FMM 5, normative-version 4.0.0) but with Procedure's
target metadata (FMM 6, normative-version 6.0.0):

1. **`source/procedure/structuredefinition-Procedure.xml`** — update the
   maturity / standards-status block:
   - Keep `structuredefinition-standards-status` `valueCode="normative"` (already
     set).
   - Add a new sibling extension
     `structuredefinition-normative-version` with
     `valueCode="6.0.0"` (currently absent; Patient carries a parallel
     extension valued `4.0.0`).
   - Change `structuredefinition-fmm` `valueInteger` from `3` to `6`.
   - Change the SD's `<status value="draft"/>` to `<status value="active"/>`
     (line 34) so the SD itself is active, matching the other normative
     resources.

2. **`source/fhir.ini` `[fmm]` section** — change `procedure=4` (line 895) to
   `procedure=6`.

3. **`source/fhir.ini` `[normative-package]` section** (lines 931–947) — add a
   new entry `Procedure=patient` (placing Procedure in the same Patient-aligned
   ballot package used by `Patient` and `Observation`; confirm with PC chairs
   before merge if a different package code is preferred).

4. **`source/fhir.ini` `[first-normative-version]` section** (lines 949–964) —
   add `Procedure=6.0.0`, recording R6 as Procedure's first normative version.

No element-level shape changes (cardinality, type, binding, invariants,
search parameters, examples) are mandated by this resolution — it is a pure
maturity/status promotion.

### Design Rationale

This change is dictated by the Sept 2025 Patient Care WGM vote and by the
existing FHIR convention for advancing a resource to normative:

- The combination `standards-status=normative` + `fmm=6` +
  `normative-version=6.0.0` + SD `status=active` is the canonical metadata
  pattern (see `source/patient/structuredefinition-Patient.xml` lines 11–37 for
  the FMM 5 / 4.0.0 analogue).
- The build's `source/fhir.ini` carries parallel registries that the publisher
  consumes for cross-resource pages (resourcelist, maturity tables, normative
  package documentation). Updating only the SD without the ini registries
  produces inconsistent generated content.
- Promotion to FMM 6 (rather than FMM 5) was explicitly approved at the WGM —
  FMM 6 is the "Normative" ceiling, applied because the resource has had broad
  implementation experience over multiple STU cycles.

No alternative was discussed in the ticket beyond the WGM-approved promotion.

- **Change Scope:** Minor (metadata only; no schema/element changes).

### Breaking Changes

Promoting a resource to Normative is a **governance breaking change** even when
the wire format is unchanged: implementers gain stronger backward-compatibility
guarantees, and any retroactive shape changes become substantially harder. From
a runtime/conformance perspective:

- No element added, removed, or retyped.
- No cardinality changes.
- No binding strength changes.
- No new invariants or search parameters.

So at the data-shape level there are **no breaking changes** for instances or
existing implementations validating against R5 Procedure.

- **Resource list / maturity dashboard pages** (auto-generated from

### Step-by-Step Tasks

#### HL7/fhir

1. **Bump Procedure's FMM extension to 6.**
   - File: `source/procedure/structuredefinition-Procedure.xml`
   - Action: Change the `structuredefinition-fmm` extension value from
     `<valueInteger value="3"/>` to `<valueInteger value="6"/>` (currently
     line 15, inside the extension block at lines 14–16).
   - Details: Do not touch the `<snapshot>` further down in the file — it is
     publisher-generated.

2. **Add `structuredefinition-normative-version` extension.**
   - File: `source/procedure/structuredefinition-Procedure.xml`
   - Action: Insert a new extension immediately after the existing
     `structuredefinition-standards-status` extension (after line 13), mirroring
     Patient's analogue at lines 14–16:
     ```xml
       <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-normative-version">
         <valueCode value="6.0.0"/>
       </extension>
     ```
   - Details: Place it before the `structuredefinition-fmm` extension so the
     ordering matches the Patient SD.

3. **Set the SD's own `<status>` to `active`.**
   - File: `source/procedure/structuredefinition-Procedure.xml`
   - Action: On line 34, change `<status value="draft"/>` to
     `<status value="active"/>`.
   - Details: Matches the convention used by every other normative core
     resource (e.g., Patient line 37).

4. **Update Procedure's FMM in `[fmm]`.**
   - File: `source/fhir.ini`
   - Action: On line 895, change `procedure=4` to `procedure=6`.
   - Details: Keep alphabetic position unchanged.

5. **Add Procedure to `[normative-package]`.**
   - File: `source/fhir.ini`
   - Action: Insert `Procedure=patient` into the `[normative-package]` block
     (lines 931–947) after the existing `Observation=observation` entry on
     line 947. Substitute the WG-confirmed package name from the Prerequisites
     step if it differs from `patient`.
   - Details: This is the registry the publisher consults to know which ballot
     package a normative resource belongs to.

6. **Add Procedure to `[first-normative-version]`.**
   - File: `source/fhir.ini`
   - Action: Insert `Procedure=6.0.0` into the `[first-normative-version]`
     block (lines 949–964) after `Observation=4.0.0` on line 964.
   - Details: R6 ships as `6.0.0`; this records that R6 is the first normative
     version of Procedure.
