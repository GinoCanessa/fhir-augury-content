## Resolution Summary

Resolution: **Persuasive**.

Resolution description (verbatim): *"Make a FHIR ConceptMap across allergy clinical
status value sets (shown above) and make the ConceptMap available via CCDA value
set registry page and via FHIR Terminology > ConceptMaps"*.

Concretely, the work group approved (vote Rob McClure / Jay Lyle: 11‑0‑0) the
authoring of a new FHIR `ConceptMap` resource that maps the **FHIR
AllergyIntolerance clinical-status value set**
(`http://hl7.org/fhir/ValueSet/allergyintolerance-clinical`, backed by
`CodeSystem` `http://terminology.hl7.org/CodeSystem/allergyintolerance-clinical`
— codes `active | inactive | resolved`) to the corresponding C-CDA allergy
clinical-status value set (HL7 v3 `ActStatus`-derived value set used by the
C-CDA Allergy Concern Act / `statusCode`). The ConceptMap must be registered so
that it (a) renders on the FHIR **Terminology → ConceptMaps** index
(`terminologies-conceptmaps.html`, which is assembled from
`list-ConceptMap-examples.xml` via the `<%conceptmapslist%>` macro), and (b) is
reachable from the CCDA value‑set registry page so CCDA implementers can find
the cross‑walk.

The associated comment captures an explicit non‑blocking follow‑up: once the
publisher supports it, the new ConceptMap should also be surfaced on the
AllergyIntolerance "Terminology Bindings" table — this is deferred by the
resolution and is *not* in scope here.

## Feature Proposal

### Problem Statement

C-CDA and FHIR each define their own allergy clinical-status value sets, and
implementations that bridge C-CDA and FHIR (e.g., CDA→FHIR transformations,
clinical‑status translation in allergy reconciliation) must hand‑roll the
mapping. The resolution notes that the FHIR core specification already
publishes similar cross-walks for other concepts (see
`source/conceptmap/conceptmap-example.xml`, which maps FHIR `address-use` to
`v3-AddressUse`), but no such published ConceptMap exists for allergy clinical
status. FHIR-28088 closes that gap by promoting the mapping from tribal
knowledge to a first-class, registered ConceptMap artefact that both the FHIR
Terminology index and the CCDA value‑set registry point at.

### Proposed Change

Add a new ConceptMap to FHIR Core, register it in the ConceptMap example list,
and wire it in so the FHIR `terminologies-conceptmaps.html` page and the CCDA
value‑set registry page both surface it. Nothing else in the AllergyIntolerance
resource definition changes.

**New ConceptMap resource** (new file
`source/conceptmap/conceptmap-allergyintolerance-clinical.xml`):

- `id`: `allergyintolerance-clinical`
- `url`: `http://hl7.org/fhir/ConceptMap/allergyintolerance-clinical`
- `name` / `title`: `FHIRv3AllergyIntoleranceClinicalStatus` /
  `FHIR/v3 AllergyIntolerance Clinical Status Mapping`
- `status`: `draft` (initial publication; WG may bump to `active`)
- `experimental`: `true` for first publication, mirroring `conceptmap-101`
- `publisher`: `HL7 International / Patient Care`
- Work‑group extension:
  `http://hl7.org/fhir/StructureDefinition/structuredefinition-wg` →
  `pc` (consistent with `valueset-allergyintolerance-clinical.xml`).
- `useContext`:
  `usage-context-type = venue` / `valueCodeableConcept.text = "for CCDA Usage"`
  (same pattern as `conceptmap-101`, so this ConceptMap shows up in the
  CCDA-scoped filter on the Terminology page).
- `jurisdiction`: `urn:iso:std:iso:3166` → `US` (CCDA is US‑realm).
- `purpose`: "To help implementers map AllergyIntolerance.clinicalStatus
  between FHIR and C-CDA / HL7 v3 allergy clinical-status coding."
- `copyright`: `Creative Commons 0`
- `sourceScopeUri`:
  `http://hl7.org/fhir/ValueSet/allergyintolerance-clinical`
- `targetScopeUri`: the C-CDA-side allergy clinical-status ValueSet canonical
  under `http://terminology.hl7.org`. Exact URL to be confirmed at authoring
  time from UTG / the CCDA registry page (see Open Questions); the mapping
  resource itself targets the `ActStatus`-derived codes
  (`active | suspended | aborted | completed`).
- `group.source`:
  `http://terminology.hl7.org/CodeSystem/allergyintolerance-clinical`
- `group.target`: `http://terminology.hl7.org/CodeSystem/v3-ActStatus`
- `group.element` entries (draft, pending WG confirmation at
  publication time — the resolution references a mapping table "shown above"
  in a source document we do not have copy of in cache, so these are the
  clinically conventional correspondences):
  - `active` → `active` (`relationship = equivalent`)
  - `inactive` → `suspended` (`relationship = related-to`; v3 has no exact
    "inactive" concept — `suspended` is the closest semantic)
  - `resolved` → `completed` (`relationship = equivalent`)

**Registration** (edit `source/conceptmap/list-ConceptMap-examples.xml`): add a
new `entry` with `valueString = "conceptmap-allergyintolerance-clinical"` and
`reference = "ConceptMap/allergyintolerance-clinical"`. The existing
`<%conceptmapslist%>` macro on `source/terminologies-conceptmaps.html` then
emits the row automatically — no edit to that page is required.

**CCDA value‑set registry wiring**: the CCDA side consumes FHIR artefacts by
canonical URL. Once the ConceptMap is published at
`http://hl7.org/fhir/ConceptMap/allergyintolerance-clinical`, add a reference
to it from whatever page renders the CCDA allergy clinical‑status value set
entry (the ticket calls this "the CCDA value set registry page"). That page
lives outside `HL7/fhir` — see Open Questions; no HL7/fhir change is required
beyond publishing the ConceptMap at a stable URL and exposing it via the
Terminology index.

**Out of scope (explicitly deferred by the resolution/comments):**

- Linking the new ConceptMap from AllergyIntolerance's Terminology Bindings
  table. The comment from Michelle Miller says "In the future, consider
  including it on FHIR AllergyIntolerance Terminology Bindings table when
  tooling supports it." Do *not* edit
  `source/allergyintolerance/structuredefinition-AllergyIntolerance.xml` or
  `allergyintolerance-*-notes.xml`/`-introduction.xml` under this ticket.

### Design Rationale

- Mirrors the established cross-walk pattern already in `HL7/fhir`. The
  nearest analogue in the clone is `source/conceptmap/conceptmap-example.xml`
  (FHIR `address-use` ↔ `v3-AddressUse`), which uses `sourceScopeUri` /
  `targetScopeUri`, `useContext` `"for CCDA Usage"`, and a single `group`
  keyed by the FHIR and v3 code systems. Re-using that shape keeps tooling,
  validation, and the `conceptmapslist` macro behaviour consistent.
- Registering via `list-ConceptMap-examples.xml` is the path the publisher
  already walks to build `terminologies-conceptmaps.html` (confirmed by
  reading `source/terminologies-conceptmaps.html`: the body is essentially
  `<%conceptmapslist%>` plus a header). No new publisher plumbing is needed.
- Keeping the AllergyIntolerance resource untouched respects the explicit
  deferral in the committee vote.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Authors and publishes the new ConceptMap artefact and exposes it
  via the FHIR Terminology → ConceptMaps index.
- **Affected Files:**
  - `source/conceptmap/conceptmap-allergyintolerance-clinical.xml` — new

No breaking changes. This ticket only adds a new informative/experimental
ConceptMap. It does not alter the AllergyIntolerance resource,
`ValueSet/allergyintolerance-clinical`, any CodeSystem, any binding strength,
any cardinality, or any search parameter.

  outside this repo but is the reason the new ConceptMap exists.

## Implementation Plan

### Prerequisites

- Confirm the exact canonical URL of the C-CDA allergy clinical-status
  ValueSet to use in `targetScopeUri` (see Open Questions). This does not
  block drafting the ConceptMap body — the `group.target` code‑system URI
  (`http://terminology.hl7.org/CodeSystem/v3-ActStatus`) is unambiguous — but

- Confirm the three code-to-code mappings with Patient Care (clinically: the

### Step-by-Step Tasks

#### HL7/fhir

1. **Create the new ConceptMap example file.**
   - File: `source/conceptmap/conceptmap-allergyintolerance-clinical.xml`
   - Action: Add a new XML file following the shape of
     `source/conceptmap/conceptmap-example.xml`.
   - Details:
     - `id = allergyintolerance-clinical`,
       `url = http://hl7.org/fhir/ConceptMap/allergyintolerance-clinical`.
     - Include the `structuredefinition-wg` extension with
       `valueCode = "pc"` (matches the Patient Care ownership used on
       `valueset-allergyintolerance-clinical.xml`).
     - `name = FHIRv3AllergyIntoleranceClinicalStatus`,
       `title = FHIR/v3 AllergyIntolerance Clinical Status Mapping`.
     - `status = draft`, `experimental = true`,
       `publisher = "HL7 International / Patient Care"`, one `contact` with
       `system = url, value = http://hl7.org/fhir`.
     - `useContext`:
       `code = usage-context-type|venue`,
       `valueCodeableConcept.text = "for CCDA Usage"` (mirrors
       `conceptmap-example.xml`).
     - `jurisdiction`: `urn:iso:std:iso:3166|US`.
     - `purpose = "To help implementers map AllergyIntolerance.clinicalStatus between FHIR and C-CDA / HL7 v3 allergy clinical-status coding."`
     - `copyright = "Creative Commons 0"`.
     - `sourceScopeUri = http://hl7.org/fhir/ValueSet/allergyintolerance-clinical`.
     - `targetScopeUri`: the CCDA allergy clinical-status ValueSet canonical
       (confirm with UTG — see Open Questions).
     - Exactly one `group` with
       `source = http://terminology.hl7.org/CodeSystem/allergyintolerance-clinical`
       and
       `target = http://terminology.hl7.org/CodeSystem/v3-ActStatus`.
     - Three `element` children matching the draft mapping:
       `active → active (equivalent)`,
       `inactive → suspended (related-to)`,
       `resolved → completed (equivalent)`.
     - Include `<display>` values on both source and target codes for
       renderer hinting.

2. **Register the new ConceptMap in the example list.**
   - File: `source/conceptmap/list-ConceptMap-examples.xml`
   - Action: Add a new `entry` block after the existing entries.
   - Details: The new entry must contain (a) an extension with
     `valueString = "conceptmap-allergyintolerance-clinical"` (the file id,
     matching the pattern used by other entries such as
     `conceptmap-example-specimen-type`), and (b) an `item.reference`
     pointing at `ConceptMap/allergyintolerance-clinical`. This is the only
     change needed to make the resource show up on
     `source/terminologies-conceptmaps.html` — that page renders through
     the `<%conceptmapslist%>` macro and does not need a direct edit.

3. **Self-check the publish locally** (if the author has the HL7 publisher
   toolchain set up).
   - Action: Run `./publish.sh` (or the Windows equivalent) and open the
     generated `terminologies-conceptmaps.html` to confirm the new row is
     rendered and the ConceptMap canonical resolves. Note that
     `build.sh`/`publish.sh`/`gradlew`/`vsts-ci.yml` are publisher plumbing
     (per briefing) — do not edit them.

### Open Questions

1. **Exact `targetScopeUri`.** The resolution references "allergy clinical
   status value sets (shown above)" but the ticket body does not include the
   CCDA-side ValueSet canonical. Confirm with Rob McClure / Robert Hausam /
   Marc Duteau (CC'd on the resolution comment) which UTG / CCDA ValueSet
   canonical is the intended target — candidates are a CCDA-published
   allergy-status ValueSet on terminology.hl7.org, or a direct reference to
   the v3 `ActStatus` codes enumerated in the `group`.
2. **`inactive → suspended` relationship.** v3 `ActStatus` has no direct
   "inactive" concept. Patient Care should confirm whether `suspended`
   (modeled as `related-to`) or `aborted` (`related-to`) is the correct
   clinical target, and whether the mapping should be asymmetric
   (FHIR `inactive` is broader than either v3 concept alone).
3. **CCDA registry page edit.** The resolution also asks that the ConceptMap
   be reachable "via CCDA value set registry page". That page is not inside
   `HL7/fhir`; the briefing confirms no cross-repo touch points for this
   ticket within the FhirCore clone. Confirm with the CCDA publisher
   (likely a separate PR on the CCDA IG / `ccda-examples` tooling) that the
   canonical URL
   `http://hl7.org/fhir/ConceptMap/allergyintolerance-clinical` is all they
   need, and track that follow-up outside this ticket.
4. **Standards status.** Initial `status = draft` / `experimental = true`
   mirrors `conceptmap-example.xml`. Patient Care may prefer `active` given
   the 11-0-0 vote; confirm before merge.
5. **Deferred follow-up ticket.** The comment "In the future, consider
   including it on FHIR AllergyIntolerance Terminology Bindings table when
   tooling supports it" should be captured as a separate tracker so it is
   not lost — out of scope here.
