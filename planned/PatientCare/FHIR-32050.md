## Resolution Summary

The Patient Care WG approved (Sept 2025 WGM, Thurs Q1 vote) raising the
`AllergyIntolerance` resource to **Normative** status as part of the R6
publication. Resolution wording (verbatim):

> In alignment with the Sept 2025 Thurs Q1 vote, AllergyIntolerance is
> planned to go normative in R6.

Reference: [2025 Sept Patient Care WGM Agenda and Minutes](https://confluence.hl7.org/spaces/PC/pages/358875423/2025+Sept+Patient+Care+WGM+Agenda+and+Minutes).

RLeftwich participates in the change-application discussion and notes
that the transition lands in R6 тАФ consistent with the resolution.

The previously-published ticket **FHIR-36062** ("Designate
AllergyIntolerance type and reaction as Trial Use (non-normative)")
must be honored as part of this transition: when the resource itself
is promoted to Normative, the `type` element and the `reaction`
backbone element are to be flagged element-level Trial Use so they
remain non-normative.

## Feature Proposal

### Problem Statement

`AllergyIntolerance` has been at FMM 3 (Trial Use) since R4. The
Patient Care WG has determined the resource is mature, widely
implemented, and ready to be locked in for R6 with normative semantics.
Today the publisher sources still classify the resource as FMM 3 even
though the StructureDefinition's resource-level `standards-status`
extension already carries `normative` (apparently set in anticipation
of this change) тАФ but the supporting publisher metadata in
`source/fhir.ini` does not yet declare it normative, the StructureDefinition
is missing the `structuredefinition-normative-version` extension, the
publication `<status>` is still `draft`, and FMM is still 3. As a
result the publisher will not actually treat the resource as Normative
in R6 output.

### Proposed Change

Promote `AllergyIntolerance` to Normative for R6 with the following
concrete edits, mirroring the established pattern used by `Patient`
and `Observation`:

1. **StructureDefinition (`structuredefinition-AllergyIntolerance.xml`)**
   - Add a new `structuredefinition-normative-version` extension
     immediately after the existing `structuredefinition-standards-status`
     extension (insertion point: line 13, between the closing
     `</extension>` of `standards-status` and the opening of `fmm`):

     ```xml
     <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-normative-version">
       <valueCode value="6.0.0"/>
     </extension>
     ```

   - Bump the `structuredefinition-fmm` extension `valueInteger` from
     `3` to `5` (line 15). FMM 5 is the value used by Patient and is
     consistent with first-time-normative resources that retain a
     numeric FMM.
   - Change `<status value="draft"/>` to `<status value="active"/>`
     (line 34) to match Patient/Observation's convention for active
     normative resources.
   - For each of the two elements held back from normative scope per
     FHIR-36062, add an element-level standards-status extension as
     the first child of the `<element>`:

     ```xml
     <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
       <valueCode value="trial-use"/>
     </extension>
     ```

     - `AllergyIntolerance.type` (element starting at line 186)
     - `AllergyIntolerance.reaction` (backbone element starting at
       line 604)

   *(The resource-level `structuredefinition-standards-status` of
   `normative` already present at line 11тАУ13 is correct and stays.)*

2. **Publisher metadata (`source/fhir.ini`)**
   - `[fmm]`: change `allergyintolerance=3` (line 794) тЖТ `allergyintolerance=5`.
   - `[standards-status]`: insert a new line `AllergyIntolerance=normative`
     in the alphabetised block (currently lines 967тАУ1047). Place it
     after `AdverseEvent=normative` (line 1025) to maintain alphabetical
     ordering within the block.
   - `[normative-package]`: insert `AllergyIntolerance=patient` after
     line 946 (`Patient=patient`). PC is the owning WG; the existing
     normative `patient` package is the natural home unless PC elects
     to introduce a dedicated `medications` or `clinical` package
     (see Open Questions).
   - `[first-normative-version]`: insert `AllergyIntolerance=6.0.0`
     after line 963 (`Patient=4.0.0`).

3. **No changes** are required to the AllergyIntolerance value sets,
   code systems, examples, search parameters, narrative pages, or
   mapping spreadsheets to effect the maturity change itself. They
   are only edited if the publisher build flags conformance issues
   that the new normative status surfaces.

### Design Rationale

- The four-section pattern in `source/fhir.ini`
  (`[fmm]` / `[standards-status]` / `[normative-package]` /
  `[first-normative-version]`) plus the StructureDefinition extensions
  is exactly how Patient and Observation transitioned to Normative in
  R4. Replicating that pattern minimizes publisher surprises.
- The resource SD already carries `standards-status=normative`, so the
  intent has been pre-staged; the missing pieces are the
  `normative-version` extension, the FMM bump, the `<status>` flip,
  and the `fhir.ini` declarations the publisher actually consumes.
- Element-level Trial Use marking on `type` and `reaction` was
  approved in **FHIR-36062** (status: Published) тАФ that disposition
  is binding and must be preserved as part of this change.

## Impact Analysis

### Affected Repositories

- **Role:** The single source of truth for the `AllergyIntolerance`
  StructureDefinition and for the publisher metadata that determines
  resource standards status, FMM, and normative packaging.
- **Affected Files:**
  - `source/allergyintolerance/structuredefinition-AllergyIntolerance.xml` тАФ add `structuredefinition-normative-version` extension (`6.0.0`), bump `structuredefinition-fmm` from `3` to `5`, change `<status>` from `draft` to `active`, add element-level `standards-status=trial-use` extensions on `AllergyIntolerance.type` (line 186) and `AllergyIntolerance.reaction` (line 604).
  - `source/fhir.ini` тАФ update `[fmm]` (line 794), and add entries in `[standards-status]`, `[normative-package]`, `[first-normative-version]` for `AllergyIntolerance`.
- **Change Scope:** **Moderate.** Few lines change, but the semantic
  effect (locking the wire format and many element definitions under
  the FHIR Normative ballot) is significant.

### Breaking Changes

Treated honestly: **the maturity promotion itself is not a wire-format
breaking change**, because no element cardinalities, types, bindings,
or invariants change. However:

- Going Normative is itself a **conformance-impacting milestone**: it
  freezes the in-scope elements of `AllergyIntolerance` against
  future incompatible change without going through the FHIR
  inverse-version-compatibility process.
- The two carved-out elements (`type` and `reaction`) remain Trial
  Use; implementers must understand they are explicitly *not* part of
  the normative content even though the parent resource is.
- Implementers tracking the resource's `<status>` value (`draft` тЖТ
  `active`) or the `structuredefinition-fmm` extension (`3` тЖТ `5`)
  may have CI/conformance tooling that branches on these values.
  This is generally desirable behaviour but should be called out in
  R6 release notes.

### Related Specifications

- No other FHIR Core resource directly references AllergyIntolerance's
  maturity state; references like `Reference(AllergyIntolerance)` in
  other resources are unaffected by maturity changes.
- Downstream IGs in the `IgRepositories` set (e.g., US Core, IPS, AU
  Core profiles on AllergyIntolerance) inherit normative status
  semantically but require no editorial change in this ticket.

### Related Tickets
