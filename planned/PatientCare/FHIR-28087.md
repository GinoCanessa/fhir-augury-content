## Resolution Summary

Resolution: **Persuasive**.

From the ticket's resolution description:

> Make a FHIR ConceptMap across clinical status / problem status value sets and make the ConceptMap available via CCDA value set registry page and via FHIR Terminology &gt; ConceptMaps.

Concretely, this ticket requires Patient Care / Vocab to author a new `ConceptMap` resource whose **source** is the FHIR `Condition.clinicalStatus` value set (`http://hl7.org/fhir/ValueSet/condition-clinical`, aka *ConditionClinicalStatusCodes*) and whose **target** is the C-CDA "Problem Status" value set used on Problem Concern Act / Problem Observation (the CDA Problem Status registry entry). The resulting ConceptMap must then be:

1. Published in the FHIR Core specification so that it appears on the auto-generated *Mappings between Value Sets Defined in FHIR* page (`terminologies-conceptmaps.html`), i.e. *FHIR Terminology &gt; ConceptMaps*.
2. Registered on the CCDA value-set-registry page on Confluence / the HL7 Terminology (UTG/THO) side so consumers coming from the CDA world can discover it.

The Michelle Miller comment adds a future-looking note: once tooling supports it, the same ConceptMap should also be surfaced in Condition's *Terminology Bindings* table. That is not in the scope of this ticket — it is tracked as a follow-up.

## Feature Proposal

### Problem Statement

FHIR Condition uses `ConditionClinicalStatusCodes` (`active | recurrence | relapse | inactive | remission | resolved`) on `Condition.clinicalStatus`. C-CDA's Problem Concern Act / Problem Observation uses a separate "Problem Status" value set (HL7 Act Status–derived, with members such as *Active*, *Inactive*, *Resolved*). Implementers mapping between C-CDA problem lists and FHIR Condition repeatedly need to translate between these two value sets, but there is no normative, discoverable mapping published by HL7 in either the FHIR spec or the CCDA value-set registry. Each implementer reinvents the mapping, with no authoritative home for corrections.

The ticket description (2020-07-16, Michelle Miller, Patient Care FHIR call) states: "Make a FHIR ConceptMap across clinical status / problem status value sets … and make the ConceptMap available via CCDA value set registry page and via FHIR Terminology &gt; ConceptMaps." The resolution vote at that meeting was *Persuasive*; no alternate wording was recorded in the ticket.

### Proposed Change

Create a new `ConceptMap` resource, published as part of FHIR Core, that declares the source/target mapping described in the resolution:

- **Resource id:** `condition-clinical-to-ccda-problem-status` (suggested; the final id may be picked by Vocab — there is no fixed naming rule in the ticket).
- **Canonical URL:** `http://hl7.org/fhir/ConceptMap/condition-clinical-to-ccda-problem-status`
- **Name / Title:** `ConditionClinicalStatusToCCDAProblemStatus` / *Condition Clinical Status → C-CDA Problem Status*
- **Status / experimental:** `draft` / `false` (consistent with other new cross-standard ConceptMaps such as `conceptmap-example-specimen-type`).
- **Publisher / contact:** `HL7 International / Patient Care` with a Vocab secondary contact, mirroring the pattern used by existing ConceptMap XMLs under `source/conceptmap/`.
- **sourceScope[x]:** `sourceScopeUri = http://hl7.org/fhir/ValueSet/condition-clinical` (the *ConditionClinicalStatusCodes* value set defined in `source/condition/valueset-condition-clinical.xml`).
- **targetScope[x]:** `targetScopeUri` pointing at the canonical URL of the C-CDA Problem Status value set in HL7 Terminology / UTG. The authoritative canonical will be taken from UTG; the placeholder expected form is `http://terminology.hl7.org/ValueSet/v3-ProblemStatus` or the current UTG canonical for the CCDA Problem Status registry entry.
- **group.source / group.target:** the code system underlying each value set — `http://terminology.hl7.org/CodeSystem/condition-clinical` for the FHIR side, and the HL7 v3 `ActStatus` (or whichever code system UTG confirms for CCDA Problem Status) for the target side.
- **element / target mappings:** one `element` per code in *ConditionClinicalStatusCodes*, each with a `target.relationship` of `equivalent`, `source-is-narrower-than-target`, or `source-is-broader-than-target` as vocab-ratified. Initial suggested mapping (to be confirmed by the Vocab / PC facilitators listed on the ticket — Rob McClure, Robert Hausam, Marc Duteau):
  - `active` ↔ Active (`equivalent`)
  - `recurrence` → Active (`source-is-narrower-than-target`)
  - `relapse` → Active (`source-is-narrower-than-target`)
  - `inactive` ↔ Inactive (`equivalent`)
  - `remission` → Inactive (`source-is-narrower-than-target`)
  - `resolved` ↔ Resolved / Completed (`equivalent`, using the CCDA-preferred code)
- **Optional notes:** include a short `ConceptMap.description` explaining that the mapping is intended for CDA ↔ FHIR Condition clinical-status translation and that reverse mapping (CDA → FHIR) is not automatically valid for every source code.

In addition to the resource itself:

- Register the new ConceptMap in `source/fhir.ini` under the `[ConceptMap]` publisher section that currently enumerates the example maps (`conceptmap=ConceptMap`, `[codesystems]`, `[valuesets]` — the exact key used by `source/conceptmap/*.xml` entries — see §"Step-by-Step Tasks" below), so the publisher picks it up and it appears on the auto-generated `terminologies-conceptmaps.html` registry (which is rendered from `<%conceptmapslist%>` at build time — no manual edit of that HTML page is required).
- Add it to `source/conceptmap/list-ConceptMap-examples.xml` so it shows up in the ConceptMap examples index alongside `conceptmap-example-specimen-type` and the others.
- Update the CCDA Value Set Registry page (maintained on Confluence / UTG, not in this repo) to link back to the new ConceptMap's canonical URL.

### Design Rationale

- **Authoring as a concrete ConceptMap resource** (rather than narrative prose) follows the pattern already established by all other cross-code-system maps in the spec (`source/conceptmap/conceptmap-103.xml`, `conceptmap-example-specimen-type.xml`, `conceptmap-message-adt-a04-to-bundle.xml`). It makes the mapping machine-consumable via `$translate` and via the same *Mappings between Value Sets Defined in FHIR* page the resolution explicitly targets.
- **Publishing in FHIR Core and linking from the CDA registry** (rather than authoring the map in UTG and linking the other way) matches Michelle Miller's exact phrasing — "available via CCDA value set registry page **and** via FHIR Terminology &gt; ConceptMaps" — and the existing HL7 practice of hosting cross-standard maps alongside the source value set's spec.
- **`sourceScopeUri` / `targetScopeUri`** is the R5+ ConceptMap shape (`ConceptMap.source[x]=@ConceptMap.sourceScope[x]` renaming is already recorded in `source/fhir.ini` at line 1641-1642), which is the form the clone's current publisher expects.
- The Condition facilitators' follow-up (surfacing the map on Condition's Terminology Bindings table) is deferred because the ticket comment explicitly marks it as "in the future, consider including it … when tooling supports it" — i.e., out of scope now.
