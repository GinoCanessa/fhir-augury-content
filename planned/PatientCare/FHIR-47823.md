## Resolution Summary

**Resolution: Not Persuasive with Modification.** The committee declined to redesign `Procedure.focalDevice.action` as a backbone element mirroring `ServiceRequest.orderDetails.parameters`. Instead, two modifications were agreed at the WGM (Wed 2025-05-13 Q1):

1. **Update `Procedure.outcome` definitional comment** to make explicit that detailed device-parameter settings recorded as part of a procedure should be carried as `Procedure.outcome` `Reference(Observation)` (cardinality already `0..*`), with one Observation per parameter. The worked example written into the comment is the ICU ventilator case — `Procedure.code = "ventilator parameter adjustment"` with `Procedure.outcome` references to Observations for respiratory rate, tidal volume, FiO2, and PEEP.
2. **Add a Procedure XML example** demonstrating the pattern (Emma Jones to supply the ventilator-parameter-adjustment instance and the supporting Observation instances).

Distinguishing the two concerns flagged in committee discussion:
- *What the device settings should be* → handled by `DeviceMetric` / order details, not Procedure.
- *What the device was actually set to during the procedure* → captured via `Procedure.outcome` `Reference(Observation)`, which is what this ticket implements.

No structural change to `Procedure.focalDevice.action` (still `CodeableConcept`) and no change to `Device` / `DeviceDefinition` are required by the resolution. Coordination with OO/Devices was discussed (March 2025: "CONSIDERED (need coordination with Device/OO)") but the resolution path is documentation-only inside Procedure.

## Feature Proposal

### Problem Statement

`Procedure.focalDevice.action` (`CodeableConcept`) is too thin to record what a clinician actually changed on a complex implantable or wearable device (ICDs, insulin pumps, ventilators) during a procedure. `ServiceRequest` already supports rich parameter capture via `ServiceRequest.orderDetails.parameterFocus` (`CodeableReference(Device | DeviceDefinition | …)`) plus a `parameters` backbone (`code` + `value[x]`, `0..*`), but `Procedure` exposes no analogous structure for *recording* what was actually set. Authors were therefore reaching for `focalDevice.action` and finding it insufficient.

### Proposed Change

This is a **documentation + example** change to the `Procedure` resource in `HL7/fhir`. There is no element addition, type change, cardinality change, or new extension.

1. **`Procedure.outcome` element comment** — extend the existing `<comment>` text to teach implementers that procedure-time device parameter settings should be recorded as Observation references in `Procedure.outcome` (the element is already `0..*` `CodeableReference(Observation)`), and include the ICU ventilator example. The exact wording adopted by the WG and merged via PR #3618 / commit `b78abd9` is:

   > *"… In the ICU where patient is intubated and attached to a ventilator, the ventilator parameter settings can include respiratory rate, tidal volume, FiO2, and PEEP. The Procedure.code = ventilator parameter adjustment. The Procedure.outcome is 0..\* Reference(Observation) where each Observation instance supports documenting respiratory rate, tidal volume, FiO2, and PEEP."*

2. **`committee-notes` build extension on `Procedure.outcome`** — append `J#47823` to the existing `<valueString>` so the change is traceable in the publisher (`"J#40516, J#47823"`).

3. **New ventilator-parameter-adjustment example** — add a Procedure example instance whose `code` denotes "ventilator parameter adjustment" and whose `outcome` contains 4 `CodeableReference(Observation)` entries for respiratory rate, tidal volume, FiO2, and PEEP, plus the four supporting `Observation` example instances (`contained` or sibling resource files, mirroring the convention used by other `procedure-example-*` files). Register the new Procedure example in `source/procedure/list-Procedure-examples.xml`.

### Design Rationale

- Reuses existing structure (`Procedure.outcome` `0..*` `CodeableReference(Observation)`) — no breaking schema changes, no parallel pattern for the same concept.
- Keeps the *intent* (what should the device do — orderable settings on a request) on `ServiceRequest.orderDetails`, and the *fact* (what was actually set during the encounter) on `Procedure.outcome`. This matches the WGM discussion separating DeviceMetric/order intent from procedure-time recording.
- Avoids redesigning `Procedure.focalDevice.action` into a backbone, which would have been a breaking change to a long-standing element and would have duplicated the parameter-capture pattern already established on `ServiceRequest`.
- Leaves the door open for a future Devices/OO ticket to define a richer DeviceMetric/parameter-snapshot pattern without blocking this fix.
