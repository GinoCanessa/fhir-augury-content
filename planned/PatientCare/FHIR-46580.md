## Resolution Summary

**Resolution:** Persuasive with Modification.

The reporter requested that the CarePlan resource's Maturity Model level (FMM)
be raised to at least FMM 5. The applied resolution modifies that request:

> "In alignment with the Sept 2025 Thurs Q1 vote, CarePlan is planned to go
> normative (FMM = 6) in R6."

So the change to apply for R6 is to set CarePlan's standards-status to
`normative` (already so on the StructureDefinition in the current clone) and
its FMM value to `6` (currently `2` in both `source/fhir.ini` and the
StructureDefinition extension). No other behavioural / element changes to the
resource are requested by this ticket.

## Feature Proposal

### Problem Statement

The CarePlan resource is widely profiled and deployed (US Core, UK Core, MCC
profile across multiple production release cycles) but its publisher-tracked
maturity in the FHIR Core repo is still recorded as FMM 2. The FMM value
shown in the published spec is driven by `source/fhir.ini` `[fmm]` and by the
`structuredefinition-fmm` extension on the resource's StructureDefinition;
both currently say `2`, which contradicts the resource's actual deployment
maturity and the Sept 2025 Patient Care WGM decision to publish CarePlan as
normative in R6.

### Proposed Change

Set CarePlan's published maturity to FMM 6 and confirm its standards-status
is `normative`, by editing the two authoring locations the publisher reads:

1. **`source/fhir.ini`** — in the `[fmm]` section, change the line
   `careplan=2` to `careplan=6`.
2. **`source/careplan/structuredefinition-CarePlan.xml`** — in the
   `<differential>`-side header extensions, change the
   `structuredefinition-fmm` extension's `<valueInteger value="2"/>` to
   `<valueInteger value="6"/>`. The sibling
   `structuredefinition-standards-status` extension already carries
   `<valueCode value="normative"/>` and does not need to be changed.

No CarePlan element definitions, cardinalities, bindings, search parameters,
operations, examples, or invariants change as part of this ticket. This is a
pure metadata/maturity bump.

### Design Rationale

- The Patient Care WG voted in Sept 2025 (Thurs Q1) to take CarePlan
  normative in R6 with FMM = 6. The applied resolution explicitly invokes
  that vote, so FMM = 6 (not the originally requested FMM = 5) is the
  authoritative target.
- Both the `[fmm]` table in `source/fhir.ini` and the
  `structuredefinition-fmm` extension on the StructureDefinition feed the
  published page; they must agree to avoid mismatched values in the rendered
  spec.
- The StructureDefinition's `structuredefinition-standards-status` extension
  is already `normative` in the current clone, so the FMM bump completes the
  intent of the resolution without re-asserting standards-status.

## Implementation Plan

### Prerequisites

- Confirm that the R6 publication cycle is the target branch for the change
  (the resolution explicitly says "in R6"). All edits land on the `master`
  branch of `HL7/fhir`.
- No dependent tickets need to land first; no extension or terminology
  prerequisites.

### Step-by-Step Tasks

#### HL7/fhir

1. **Bump CarePlan FMM in the publisher's FMM table.**
   - File: `source/fhir.ini`
   - Action: In the `[fmm]` section (header at line 788), change the line
     `careplan=2` (currently at line 805) to `careplan=6`. Leave every other
     line in the `[fmm]` section untouched.
   - Details: This is the table the bespoke publisher reads to render the
     "Maturity Level" badge on the resource page. No surrounding sections
     (`[standards-status]`, `[fmg-approval]`, etc.) need to change — CarePlan
     is not currently listed in `[standards-status]` (the StructureDefinition
     carries that flag directly).

2. **Bump CarePlan FMM in the StructureDefinition.**
   - File: `source/careplan/structuredefinition-CarePlan.xml`
   - Action: In the header extensions block (between `<meta>` and
     `<url>`), change:
     ```xml
     <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-fmm">
       <valueInteger value="2"/>
     </extension>
     ```
     to use `<valueInteger value="6"/>`. Do not modify the sibling
     `structuredefinition-standards-status` extension (already
     `normative`), the `structuredefinition-category`,
     `structuredefinition-security-category`, `structuredefinition-wg`, or
     any element under `<differential>` / `<snapshot>`.
   - Details: Edit only the `valueInteger` attribute on the FMM extension.
     The `<snapshot>` block lower in the file is regenerated by the
     publisher and must not be hand-edited.

3. **Spot-check that no other authored CarePlan files redundantly assert an
   FMM value.**
   - File: `source/careplan/` (directory scan)
   - Action: `grep -ri "structuredefinition-fmm" source/careplan/` and
     confirm the only hit is the one edited in task 2. Also `grep -i
     "fmm" source/careplan/careplan-introduction.xml source/careplan/careplan-notes.xml`
     to confirm no narrative page hard-codes "FMM 2" / "Maturity Level 2".
   - Details: If a hard-coded mention is found in introduction/notes, update
     it to match FMM 6 in the same commit. (At HEAD `db79dcd` the briefing
     does not record any such mention; this is a defensive check.)

### Open Questions

- The applied resolution states FMM = 6 ("normative"). Per the FHIR maturity
  model summary in the ticket body, FMM 5 is the documented prerequisite
  step; FMM 6 corresponds to fully normative content. The plan above
  follows the resolution literally (FMM 6) — confirm with Patient Care WG
  whether any intermediate FMM 5 publication is desired before flipping to
  6, or whether the R6 release cycle goes straight to 6.
- The current StructureDefinition already carries
  `standards-status=normative` while `[standards-status]` in
  `source/fhir.ini` does not list CarePlan. If repository convention is to
  also list normative resources in the `[standards-status]` section of
  `source/fhir.ini`, an entry `CarePlan=normative` may need to be added in
  that section (just before the `[wg-info]` header at line 1048). Confirm
  with a Patient Care WG / FMG editor before adding to keep the change
  minimal.
