## Feature Proposal

### Problem Statement

In FHIR R5, `Goal.outcome` provided a clear, core mechanism to link a Goal to
the Observations that record evaluation/progress toward it. In the current R6
draft `Goal.outcome` has been removed, and the intended replacement —
referencing the Goal from the Observation via the
`resource-pertainsToGoal` extension — is **not** mentioned anywhere in the
Goal narrative or in §9.6.2 *Boundaries and Relationships*. The only
discoverable Goal→Observation link in that section is the single sentence
"Goals are often evaluated using Observations" (no mechanism), and the most
discoverable element from Goal toward Observation is `Goal.addresses`, which
is for a different purpose (clinical context/rationale, not progress
evaluation).

This leaves implementers migrating R5→R6 without a discoverable, normative
pattern, and risks improper use of `Goal.addresses` to carry the
evaluation-of-progress relationship.

### Proposed Change

Edit the Goal resource's *Boundaries and Relationships* narrative
(`source/goal/goal-introduction.xml`, around line 52) so that the existing
sentence that simply states "Goals are often evaluated using Observations"
also identifies, by URL, the standard extension that carries the
Observation→Goal link, and clarifies the directionality (the link is held on
the Observation, not on the Goal).

Suggested replacement prose for the existing one-sentence paragraph (the new
content is inline; cross-links use the publisher's existing relative form):

```html
<p>
Goals are often evaluated using <a href="observation.html">Observations</a>.
The link from such an evaluation/progress <a href="observation.html">Observation</a>
back to the <a href="goal.html">Goal</a> it pertains to is carried by the
standard <a href="extension-resource-pertainstogoal.html">resource-pertainsToGoal</a>
extension (canonical URL
<code>http://hl7.org/fhir/StructureDefinition/resource-pertainsToGoal</code>),
which is placed on the Observation. <code>Goal.addresses</code> is <em>not</em>
the correct element for this purpose: it is intended to reference the
<a href="condition.html">Condition</a>, problem, or other context that the Goal
is responding to, not the Observations used to measure progress toward the Goal.
</p>
```

Notes on the wording:

- The canonical URL of the extension is confirmed in
  `source/oids.ini` (`http://hl7.org/fhir/StructureDefinition/resource-pertainsToGoal`)
  and is already referenced from `source/careplan/structuredefinition-CarePlan.xml`
  on `CarePlan.activity.plannedActivityReference` — so the same URL/style is
  already in use elsewhere in the spec.
- The extension's published page in the FHIR build is conventionally named
  `extension-resource-pertainstogoal.html` (lowercase) — verify the exact
  filename produced by the publisher for this build before merging the link;
  if the publisher emits a different slug, update the `<a href>` accordingly.
- No element-level changes (no new element on Goal, no new extension, no
  binding change) are made in this ticket. `Goal.outcome` is **not**
  reintroduced.

No new examples, search parameters, ValueSets, or CodeSystems are required by
this change.

### Design Rationale

- The change is the minimum needed to close the discoverability gap created
  by the R5→R6 removal of `Goal.outcome`, and is exactly what the WG voted
  on (Option 1).
- Re-adding a `Goal.outcome` element (Option 2) was explicitly rejected here
  and previously on [FHIR-48360](https://jira.hl7.org/browse/FHIR-48360); the
  WG's preference is to express this relationship as a generic "this resource
  pertains to that Goal" extension on the *referencing* resource (the
  Observation), keeping Goal narrow.
- Placing the link on the Observation (via `resource-pertainsToGoal`) rather
  than on the Goal also avoids forcing every Goal to carry a back-pointer to
  every evaluation Observation, which scales poorly and would require a
  Goal update each time a new Observation is recorded.
- Calling out that `Goal.addresses` is **not** the right element here is
  important because it is currently the most discoverable Goal-side link to
  Observation in the R6 draft and is, per the ticket, the most likely
  element for implementers to misuse.
