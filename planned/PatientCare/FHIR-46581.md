## Resolution Summary

**Resolution:** Persuasive with Modification.

The September 2025 Patient Care WGM (Thursday Q1 vote, captured in the
[2025 Sept Patient Care WGM Agenda and Minutes](https://confluence.hl7.org/spaces/PC/pages/358875423/2025+Sept+Patient+Care+WGM+Agenda+and+Minutes))
approved that **CareTeam is planned to go normative (FMM = 6) in R6**.

Concretely, this means:

- The CareTeam resource's Maturity Level (FMM) is to be raised from its
  current value of **2** (per `source/fhir.ini` `[fmm]`
  `careteam=2`) to **6**, signalling intent to publish CareTeam as
  Normative content in the R6 release cycle.
- The "Modification" relative to the originally requested FMM 5 is the
  step directly to FMM 6 / Normative for R6, in line with the WG vote.

This ticket is the FMM bump itself; it does not by itself complete the
full Normative ballot machinery (see Open Questions for the related,
intentionally-deferred bookkeeping).

## Feature Proposal

### Problem Statement

CareTeam has been profiled and used widely in production (US Core,
UK Core, MCC profile across multiple formal release cycles). Despite
this, the published FMM for the CareTeam resource is still **2**
(`source/fhir.ini` line 806: `careteam=2`, mirrored in
`source/careteam/structuredefinition-CareTeam.xml` as
`<extension url="…structuredefinition-fmm"><valueInteger value="2"/>`).
That understates real-world maturity and is inconsistent with the
already-present `standards-status` extension value of `normative` in the
hand-edited resource artifacts (StructureDefinition, search-parameter
bundle, status CodeSystem and ValueSet).

The Patient Care WG voted in September 2025 to advance CareTeam to
Normative for R6. The published maturity indicator must be updated to
match.

### Proposed Change

Raise CareTeam's Maturity Level from 2 to 6 across the
`HL7/fhir` (FhirCore) authoring inputs, matching the pattern used by
other R6-track-to-normative resources (e.g., `imagingselection=6`,
`imagingstudy=6` already present in `source/fhir.ini` `[fmm]`).

Specifically:

1. **`source/fhir.ini`, `[fmm]` section:** change `careteam=2` to
   `careteam=6`.
2. **`source/careteam/structuredefinition-CareTeam.xml`:** change the
   top-level
   `<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-fmm">`
   value from `<valueInteger value="2"/>` to
   `<valueInteger value="6"/>`. The `standards-status` extension already
   carries `<valueCode value="normative"/>` and does not need to be
   re-asserted by this ticket.
3. **`source/careteam/codesystem-care-team-status.xml`:** raise the
   companion `structuredefinition-fmm` extension from
   `<valueInteger value="2"/>` to `<valueInteger value="6"/>` so the
   CodeSystem maturity matches its parent resource (it is already
   marked `standards-status=normative`).
4. **`source/careteam/valueset-care-team-status.xml`:** raise the
   companion `structuredefinition-fmm` extension from
   `<valueInteger value="2"/>` to `<valueInteger value="6"/>` for the
   same reason. (`valueset-care-team-category.xml` and
   `valueset-participant-role.xml` do not currently carry an FMM
   extension and are out of scope for this bump.)

No element-level differential changes, cardinality changes, type
changes, or terminology binding changes are required by this
resolution. This is a metadata bump only.

### Design Rationale

- Aligning the FMM extension on the StructureDefinition with the
  `[fmm]` entry in `fhir.ini` is required because the publisher uses
  both as authoritative inputs and any divergence is flagged by QA.
- Setting `careteam=6` (instead of `=5`) follows the project convention
  shown by `imagingselection=6` and `imagingstudy=6` in the same
  `[fmm]` section, which are likewise on the R6 normative track.
- The associated companion CodeSystem (`care-team-status`) and
  status ValueSet are already `standards-status=normative` in their
  hand-edited artifacts; raising their FMM to match avoids QA noise
  about a normative artifact at FMM 2.
- The ticket does **not** add CareTeam to `[normative-package]`,
  `[first-normative-version]`, or `[standards-status]` overrides,
  because the resolution language is "**planned to go normative…in
  R6**" and the WG vote authorises the FMM advancement that signals
  that intent. Promoting CareTeam fully into the normative package is a
  follow-on concern that, by Patient Care convention, requires its own
  ballot tracking and QA cycle (see Open Questions).

### Step-by-Step Tasks

#### HL7/fhir (FhirCore)

1. **Raise CareTeam FMM in the central `[fmm]` registry**
   - File: `source/fhir.ini`
   - Action: In the `[fmm]` section (header at line 788), change the
     `careteam=2` line (currently line 806) to `careteam=6`.
   - Details: Leave all surrounding entries untouched. The new value
     mirrors the convention used by `imagingselection=6` and
     `imagingstudy=6` in the same section.

2. **Raise CareTeam FMM on the StructureDefinition**
   - File: `source/careteam/structuredefinition-CareTeam.xml`
   - Action: In the top-level extension block, change
     `<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-fmm"><valueInteger value="2"/></extension>`
     to use `<valueInteger value="6"/>` (currently around line 14–16).
   - Details: Do not touch the `structuredefinition-standards-status`
     extension (already `normative`), the `<differential>` content, or
     the `<snapshot>` block. Authored top-level extensions only.

3. **Raise CareTeam status CodeSystem FMM to match**
   - File: `source/careteam/codesystem-care-team-status.xml`
   - Action: Change the top-level
     `structuredefinition-fmm` extension from
     `<valueInteger value="2"/>` to `<valueInteger value="6"/>`.
   - Details: The CodeSystem is already
     `standards-status=normative`; raising FMM aligns it with the
     parent resource and prevents the publisher's QA from flagging an
     FMM/standards-status mismatch.

4. **Raise CareTeam status ValueSet FMM to match**

   - Action: Change the top-level

   - Details: Same reasoning as the CodeSystem. Do **not** add an FMM
     extension to `valueset-care-team-category.xml` or
     `valueset-participant-role.xml` — those do not currently carry

### Open Questions

- **Full normative promotion vs. FMM advance only.** The resolution
  text is "planned to go normative (FMM = 6) in R6." This plan
  implements the FMM advance only and intentionally does **not** add
  CareTeam to `source/fhir.ini` `[normative-package]` (line 931),
  `[first-normative-version]` (line 949), or the per-resource
  `[standards-status]` overrides (line 966). If Patient Care intends
  this single ticket to also complete those bookkeeping entries
  (selecting a normative-package bucket and the first normative
  version string for R6), confirm before merging and add the
  corresponding lines.
- **Standards-status authoring inconsistency.** The hand-edited
  `structuredefinition-CareTeam.xml` already declares
  `<standards-status value="normative"/>` even though `[fmm]` has
  long carried `careteam=2`. Confirm with the WG that this pre-existing
  authored value is intentional and should remain `normative` after
  the FMM bump (the briefing's "only `<differential>` is hand-edited"
  rule does not apply to the top-level extension block, so this value
  *is* authored input).
- **`careteam-notes.xml` callouts.** A reviewer should glance at
  `source/careteam/careteam-notes.xml` and `careteam-introduction.xml`
  to confirm there is no narrative text that quotes the old FMM value
  ("FMM 2") that would also need updating. None was found in this
  scan, but it is worth a final pre-merge check.
