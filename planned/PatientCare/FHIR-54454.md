## Resolution Summary

**Resolution:** Not Persuasive with Modification.

The committee did not adopt the reporter's literal request (extending the
list of category values shown in the `Condition.category` description text
to include `diagnostic-report-impression`). Instead, the Patient Care
2026-02-12 motion (jlyle) decided:

> "This is an example binding; listing the values in 'short' is useful for
> required bindings but may be confusing in this case. We will replace this
> list with sufficiently vague text describing the broad category:
> *Classification of type of condition*."

So the approved change is to **replace** the enumerated `short` value on
`Condition.category` (currently `problem-list-item | encounter-diagnosis`)
with the generic phrase **`Classification of type of condition`**, rather
than expanding the list. As Michelle Miller noted in the comments, this is
a non-substantive (editorial) change — it does not alter the binding, the
ValueSet, the cardinality, or the type of `Condition.category`.

## Feature Proposal

### Problem Statement

The `Condition.category` element binds (example strength) to
`valueset-condition-category.xml`, whose contents include
`problem-list-item`, `encounter-diagnosis`, and — added later —
`diagnostic-report-impression`. The element's `short` text on the Condition
page, however, still reads `problem-list-item | encounter-diagnosis`,
which:

1. Is out-of-date relative to the bound ValueSet (missing
   `diagnostic-report-impression`).
2. Misleads implementers into treating the inline list as authoritative,
   even though the binding is `example` (not `required`).

The reporter proposed adding `diagnostic-report-impression` to the inline
list. Patient Care decided that any enumerated list of values in the
`short` of an example-bound element is itself the wrong pattern, and that
the list should be replaced with a generic descriptor.

### Proposed Change

A single, narrowly scoped edit to the Condition StructureDefinition's
hand-edited `<differential>` block, plus a substantive-changes log entry
on the Condition introduction page.

**Element to modify:** `Condition.category`

- **Path:** `Condition.category`
- **Cardinality:** unchanged (`0..*`)
- **Type:** unchanged (`CodeableConcept`)
- **Binding:** unchanged (`example`,
  `http://hl7.org/fhir/ValueSet/condition-category`)
- **`short`:**
  - Before: `problem-list-item | encounter-diagnosis`
  - After:  `Classification of type of condition`
- **`definition` / `comment`:** unchanged.
- **Committee-notes extension** (`http://hl7.org/fhir/build/StructureDefinition/committee-notes`):
  append this ticket to the existing `valueString`:
  - Before: `J#48358`
  - After:  `J#48358, J#54454`

No change to `valueset-condition-category.xml`, no change to the binding
strength, no change to the search parameters, examples, or other
Condition.* elements.

The Condition introduction page should also list this ticket in its
"Substantive Changes" / change-log section so consumers of the published
spec can find it.

### Design Rationale

- **Mirrors the committee motion verbatim.** The wording
  *"Classification of type of condition"* is exactly the phrase agreed by
  Patient Care on 2026-02-12.
- **Aligns with FHIR `short` conventions.** Inline `|`-separated value
  enumerations in `short` are an established convention for **required**
  bindings (e.g., `Observation.status`), where the listed codes are the
  only legal values. They are misleading when the binding is `example`,
  because implementers may legitimately use other codes. A descriptive
  phrase is the standard pattern for example/preferred/extensible
  bindings.
- **Avoids drift.** Embedding a value list in `short` couples the prose
  to the ValueSet; future ValueSet expansions (such as the addition of
  `diagnostic-report-impression` that triggered this ticket) silently
  invalidate the prose. A category-level descriptor decouples them.
- **Alternative considered (and rejected):** simply appending
  `diagnostic-report-impression` to the existing list. Rejected because
  it perpetuates the misleading enumeration pattern and would re-break
  the moment another category value is added.

## Implementation Plan

### Prerequisites

None. All artifacts referenced (the Condition StructureDefinition, the
Condition introduction page, the `condition-category` ValueSet) already
exist in `HL7/fhir` at clone HEAD `db79dcd`.

### Step-by-Step Tasks

#### HL7/fhir

1. **Update `Condition.category` `<short>` and committee-notes**
   - File: `source/condition/structuredefinition-Condition.xml`
   - Action: In the `<differential>` `<element id="Condition.category">`
     block, (a) change `<short value="problem-list-item | encounter-diagnosis"/>`
     to `<short value="Classification of type of condition"/>`, and
     (b) inside the sibling
     `<extension url="http://hl7.org/fhir/build/StructureDefinition/committee-notes">`
     change `<valueString value="J#48358"/>` to
     `<valueString value="J#48358, J#54454"/>`.
   - Details: Do not modify the `<snapshot>` block — it is regenerated
     by the HL7 publisher (briefing "Generated areas (do not edit)").
     Keep the `<path>`, `<min>`, `<max>`, `<type>`, `<definition>`,
     `<comment>`, and `<binding>` of `Condition.category` exactly as
     they are. Binding strength stays `example` and the bound ValueSet
     stays `http://hl7.org/fhir/ValueSet/condition-category`.

2. **Log the change in the Condition introduction page**
   - File: `source/condition/condition-introduction.xml`
   - Action: Add a new `<li>` under the resource's Substantive Changes /
     change-log section linking to the ticket, e.g.:
     `<li><a href="https://jira.hl7.org/browse/FHIR-54454">FHIR-54454</a>: Clarified Condition.category short display</li>`.
   - Details: Place the entry alongside the FHIR-54914 entry (the
     companion ticket landing in the same publisher cycle) so both
     show up in the per-resource change list rendered on the Condition
     page.
