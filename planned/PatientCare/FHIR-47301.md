## Resolution Summary

The resolution is **Not Persuasive with Modification**. The committee declined
the original request to add a new `HealthcareService` reference element to
CareTeam, but agreed on two documentation-only modifications. From the
resolution description:

> 1. Add an example with no patient
> 2. Add a "notes" paragraph describing how CareTeam is used when there is no
>    patient specified

Rationale captured in the 2026-02-26 Patient Care FHIR conference-call comment:

- `CareTeam.participant.member` and `CareTeam.participant.onBehalfOf` already
  support `Reference(HealthcareService)` in R6, so the directory-oriented
  "what service does the team provide" question is already answerable via
  existing elements.
- `CareTeam.category` (example-bound) conveys the type of care team (e.g.,
  a "diabetic CareTeam"), and `CareTeam.reason` (CodeableConcept) conveys the
  condition/reason — also already present.
- A CareTeam can be virtual; physical location is modeled through
  `HealthcareService.location` rather than on CareTeam directly.
- `CareTeam.subject` is already `0..1`, so modeling a non-patient-specific
  ("directory-style") CareTeam requires no schema change — only guidance and
  an example to make the usage discoverable.

No element, cardinality, binding, or Reference-target changes are required.

## Feature Proposal

### Problem Statement

The CareTeam resource is used in two materially different ways:

1. **Patient-/group-specific** care teams (the "hospital" framing): a team
    caring for a specific `Patient` or `Group`. This is the pattern modeled by
    the current canonical example (`careteam-example`), which sets
    `CareTeam.subject = Patient/example`.
2. **Non-patient-specific / directory** care teams (the "ACO" framing): a
    standing team that delivers a kind of service (e.g., a "diabetic CareTeam"
    at a payer/ACO or in a provider directory) and is *not* attached to any
    individual patient or group.

`CareTeam.subject` has cardinality `0..1` (verified in
`source/careteam/structuredefinition-CareTeam.xml:200–208`), so the
directory-style usage is already valid FHIR. However:

- There is **no worked example** in the specification of a CareTeam without a
  `subject`. Implementers reading the resource page only see the subject-bound
  pattern and infer (incorrectly) that `subject` is effectively required or
  that a new element (e.g., a top-level `HealthcareService` reference) is
  needed to capture directory use.
- The current `careteam-introduction.xml` "Scope and Usage" mentions "an event,
  prior to a subject being identified (such as a code blue team)" but does
  *not* cover the ongoing, directory-style CareTeam use case, nor does it
  explain which existing elements carry directory information
  (`category` for team type, `reason` for condition,
  `participant.member`/`onBehalfOf` for `HealthcareService`,
  `HealthcareService.location` for where the team operates).
- `careteam-notes.xml` presently contains only a Provenance note — there is
  no "Notes" entry that addresses the no-patient pattern.

The original ticket (and the FAST National Directory IG's use of a
HealthcareService extension on CareTeam) reflect exactly this gap.

### Proposed Change

Documentation-only changes in `HL7/fhir`, confined to the `source/careteam/`
authoring folder:

1. **Add a new worked example: `careteam-example-directory`** (id
   `example-directory`), a CareTeam with **no `subject`** that demonstrates
   the directory/ACO pattern. Concrete shape:

   - `id`: `example-directory`
   - `identifier`: organization-scoped identifier (e.g., value
     `"diabetes-care-team-001"`)
   - `status`: `active`
   - `category`: CodeableConcept with `text` (e.g., "Diabetes care team") —
     and where feasible a LOINC coding consistent with existing
     `valueset-care-team-category` entries (the example-bound value set).
   - `name`: `"Diabetes Care Team"`
   - **No `subject` element** (this is the point of the example).
   - `participant`: at least two participants showing `member` typed as
     `Reference(Practitioner)` (e.g., an endocrinologist) and
     `Reference(HealthcareService)` (e.g., "Diabetes Education Service"),
     with `onBehalfOf` referencing an `Organization`.
   - `reason`: `concept.text = "Diabetes mellitus management"`.
   - `managingOrganization`: reference to an `Organization`.
   - Narrative `<text>` that calls out this is a non-patient-specific
     directory-style team.

   The example MUST schema-validate against
   `schema/CareTeam.xsd` (same schemaLocation used by `careteam-example.xml`).

2. **Register the new example** by adding a matching `<entry>` to
   `source/careteam/list-CareTeam-examples.xml` (the HL7-publisher registry of
   examples for this resource).

3. **Add a "no-patient" paragraph to `source/careteam/careteam-notes.xml`**
   (which currently holds only a Provenance note), explaining that:
   - `CareTeam.subject` is optional (`0..1`); a CareTeam without a `subject`
     represents a non-patient-specific team such as a directory-level or
     standing team (e.g., a "diabetic CareTeam" offered by an ACO / provider
     organization).
   - The *type* of team is conveyed via `CareTeam.category`; the
     condition/purpose via `CareTeam.reason`.
   - The *services* the team provides are modeled by referencing a
     `HealthcareService` from `CareTeam.participant.member` and/or
     `CareTeam.participant.onBehalfOf` — no top-level HealthcareService
     element is added.
   - Physical location (when relevant) belongs on the referenced
     `HealthcareService.location`, since a CareTeam may be virtual.
   - Cross-link the new `careteam-example-directory` example.

No changes to `structuredefinition-CareTeam.xml`, ValueSets, CodeSystems,
search parameters, or invariants.

### Design Rationale

- The committee explicitly rejected adding a new element (`HealthcareService`
  reference at the CareTeam or CareTeam.participant level). The R6 model
  already carries that information through `participant.member` /
  `participant.onBehalfOf`, which were extended to allow
  `Reference(HealthcareService)`.
- The committee confirmed CareTeam must encompass *both* patient-specific
  and directory-style teams, so the fix belongs in narrative + an example,
  not in the structure.
- Adding a second example (rather than mutating `careteam-example`) preserves
  the existing patient-bound example that downstream IGs, tests, and
  tutorials already reference.
- Placing the prose in `careteam-notes.xml` rather than
  `careteam-introduction.xml` matches the existing pattern: "Scope and Usage"
  enumerates the kinds of subjects at a high level; "Notes" is where
  usage-pattern guidance lives.
- Consistency with the FAST National Directory IG: that IG used an
  extension because R4 lacked the Reference(HealthcareService) target on
  `participant.member`; in R6 core that extension is no longer necessary and
  the notes should make that explicit.

### Step-by-Step Tasks

#### HL7/fhir

1. **Create the new no-patient CareTeam example instance**
   - File: `source/careteam/careteam-example-directory.xml`  *(new)*
   - Action: Author a FHIR R6 CareTeam XML instance, following the same
     header and schema location used by `source/careteam/careteam-example.xml`
     (root element `<CareTeam xmlns="http://hl7.org/fhir" …
     xsi:schemaLocation="http://hl7.org/fhir ../../schema/CareTeam.xsd">`).
   - Details:
     - `<id value="example-directory"/>`.
     - `<text>` with `<status value="generated"/>` and an `<xhtml:div>`
       narrative that explicitly says this is a non-patient-specific
       directory/standing team (e.g., "Diabetes Care Team — directory
       example, not tied to a specific patient").
     - `<identifier><value value="diabetes-care-team-001"/></identifier>`.
     - `<status value="active"/>`.
     - `<category>` — use a LOINC coding from
       `valueset-care-team-category` if a suitable one exists for the
       chosen scenario; otherwise `<text value="Diabetes care team"/>`.
     - `<name value="Diabetes Care Team"/>`.
     - **Do NOT include a `<subject>` element.** This absence is the
       point of the example.
     - `<participant>` entries:
       - One `member` referencing a contained or external `Practitioner`
         (e.g., endocrinologist), with `onBehalfOf` → `Organization`.
       - One `member` referencing a `HealthcareService` (e.g.,
         `HealthcareService/example` or a locally contained one labeled
         "Diabetes Education Service"), demonstrating the R6
         `Reference(HealthcareService)` target.
       - Each `participant.role` with a `text` value (e.g.,
         "Endocrinology", "Diabetes education").
     - `<reason><concept><text value="Diabetes mellitus management"/></concept></reason>`.
     - `<managingOrganization>` referencing an `Organization` (an
       existing example org such as `Organization/f001` is acceptable).
     - Ensure the file schema-validates locally before PR (the publisher
       will also validate it during build).

2. **Register the new example in the publisher's examples list**
   - File: `source/careteam/list-CareTeam-examples.xml`
   - Action: Append a new `<entry>` sibling to the existing entry for
     `CareTeam/example`, following the same shape used there.
   - Details: The entry must include the two publisher-specific extensions
     (`http://hl7.org/fhir/build/StructureDefinition/description` and
     `.../title`), e.g.:
     ```xml
     <entry>
       <extension url="http://hl7.org/fhir/build/StructureDefinition/description">
         <valueString value="Directory-style CareTeam with no patient subject (e.g., diabetes care team)"/>
       </extension>
       <extension url="http://hl7.org/fhir/build/StructureDefinition/title">
         <valueString value="careteam-example-directory"/>
       </extension>
       <item>
         <reference value="CareTeam/example-directory"/>
         <display value="Directory Care Team (no patient)"/>
       </item>
     </entry>
     ```

3. **Add a "Notes" paragraph documenting the no-patient usage pattern**
   - File: `source/careteam/careteam-notes.xml`
   - Action: Inside the existing `<div>` (after the current
     Provenance paragraph and still under the `<h3>Notes</h3>` heading),
     add a new `<p>` block that covers non-patient-specific CareTeams.
   - Details: The paragraph should:
     - State that `CareTeam.subject` is optional (`0..1`) and that a
       CareTeam with no `subject` represents a non-patient-specific team
       such as a standing/directory team (e.g., a diabetes care team
       offered by an ACO or provider organization).
     - Explain which existing elements carry directory-oriented
       information:
       - `CareTeam.category` (example-bound) for the *type* of team.
       - `CareTeam.reason` for the *clinical condition / purpose*.
       - `CareTeam.participant.member` and
         `CareTeam.participant.onBehalfOf` may reference
         `HealthcareService` (as well as Practitioner, PractitionerRole,
         RelatedPerson, Patient, Organization, CareTeam,
         Group) to identify the services the team delivers. An example
         link: `<a href="healthcareservice.html">HealthcareService</a>`.
       - Physical location, when relevant, belongs on the referenced
         `HealthcareService.location` — because a CareTeam may be
         virtual, location is not modeled directly on CareTeam.
     - Cross-link the new example: `<a
       href="careteam-example-directory.xml.html">directory CareTeam
       example</a>` (confirm the exact generated file suffix the
       publisher uses for CareTeam examples — `careteam-example.xml.html`
       is the pattern used by sibling examples).

4. **(Optional, if the WG wants visibility in Scope & Usage) — small
   addition to the introduction**
   - File: `source/careteam/careteam-introduction.xml`
   - Action: *Only if requested by the WG during PR review.* Extend the
     "assigned to" bullet list in the Scope and Usage section with an
     item such as: "a service or standing program (e.g., a directory /
     non-patient-specific team such as a diabetes care team), in which
     case `subject` is omitted and the team is described by `category`
     and `reason`."
   - Details: Keep this edit minimal; the authoritative prose for the
     new pattern lives in `careteam-notes.xml` per task 3.
   - Note: The resolution strictly asks only for (1) example and
     (2) notes paragraph, so this task is **not required** and is listed
     only as an opt-in follow-up. Default is to **skip** it.

### Testing Considerations

- Run the HL7 FHIR publisher locally (or in CI via
  `master-branch-pipeline.yml` on a PR branch) to build the CareTeam
  resource page and confirm the example renders and registers.
- Schema-validate the new example XML in isolation (`xmllint --schema
  schema/CareTeam.xsd source/careteam/careteam-example-directory.xml` or
  equivalent) before the publisher run — fails fast on authoring
  mistakes.
- Edge cases to verify in the generated output:
  - The example table lists both `careteam-example` (with subject) and
    `careteam-example-directory` (without subject).
  - The JSON/Turtle/XML serializations generated from the XML source
    round-trip cleanly and show the absent `subject` element.
  - The rendered narrative of the new example says it is not
    patient-specific.
- Interoperability consideration: consumers that assumed
  `CareTeam.subject` was always present will now see a canonical
  counter-example in the core spec. This is correct (subject has always
  been `0..1`), but release notes for the containing publication should
  call out the clarified guidance.

### Open Questions

- **Category code selection.** Should the directory example use a coded
  `category` (preferred if `valueset-care-team-category` contains a
  suitable LOINC code for "diabetes / disease-management team") or a
  `category.text` only? Decision needed before PR so the example
  represents realistic usage. If no suitable code exists, proceed with
  `text` only (the binding is example-strength).
- **HealthcareService participant — contained vs. external.** The
  existing `careteam-example.xml` uses a `contained` Practitioner; for
  the directory example, using a `contained` HealthcareService keeps the
  example self-contained and deterministic. Confirm with WG reviewer
  which style they prefer.
- **Scope and Usage amendment (Task 4).** The resolution lists only
  (1) example and (2) notes paragraph. Confirm with the WG whether they
  also want the Scope-and-Usage bullet list in
  `careteam-introduction.xml` amended to explicitly name the
  directory/standing-team pattern. Default per resolution: **skip**.

  ticket-scoped to `FHIR-42980` and its playbook SHA no longer matches
