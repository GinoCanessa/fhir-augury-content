## Resolution Summary

**Resolution:** Persuasive with Modification.

Resolution description (verbatim): *"In alignment with the Sept 2025 Thurs Q1 vote, Goal is planned to go normative (FMM 6) in R6"* (references the [2025 Sept Patient Care WGM Agenda and Minutes](https://confluence.hl7.org/spaces/PC/pages/358875423/2025+Sept+Patient+Care+WGM+Agenda+and+Minutes)).

The ticket requested raising the Goal resource's maturity to at least FMM 5. Patient Care's adopted resolution is stronger: promote Goal to **Normative** in R6 with **FMM 6**, rather than just FMM 5. Implementation therefore means setting Goal's `structuredefinition-fmm` extension to `6` and marking the resource as normative from R6 (adding a `structuredefinition-normative-version` of `6.0.0`) — matching how other already-normative resources (e.g., Patient) are modelled.

## Feature Proposal

### Problem Statement

Goal has been profiled and used in production for several release cycles — notably by the MCC profile, US Core, and the Danish national implementation — yet its FHIR Maturity Model level in R5 remained FMM 2. This under-represents the real-world deployment of the resource and blocks downstream IGs from relying on version stability guarantees. Patient Care reviewed the evidence at the September 2025 WGM and voted that Goal is ready to advance to Normative status in R6.

### Proposed Change

Update the Goal StructureDefinition in the FHIR core build so that R6 publishes Goal as Normative (FMM 6):

1. **`source/goal/structuredefinition-Goal.xml`** — change the `structuredefinition-fmm` extension value from `2` to `6` (line 15). This is the single authoritative place the publisher reads the FMM from for the Goal resource page and the generated resource list.
2. **`source/goal/structuredefinition-Goal.xml`** — confirm/retain the existing `structuredefinition-standards-status` extension value of `normative` (line 12).
3. **`source/goal/structuredefinition-Goal.xml`** — add a `structuredefinition-normative-version` extension with `valueCode` `4.3.0` or `6.0.0` as appropriate for the first release publishing Goal as normative. The Patient resource uses this pattern (line 14: `<valueCode value="4.0.0"/>`), locking in the release in which it first became normative. Use `6.0.0` so the publisher records R6 as the normative-from version.

Concrete XML (extension block in `structuredefinition-Goal.xml` after change):

```xml
<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
  <valueCode value="normative"/>
</extension>
<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-normative-version">
  <valueCode value="6.0.0"/>
</extension>
<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-fmm">
  <valueInteger value="6"/>
</extension>
```

No element cardinalities, types, bindings, invariants, or search parameters are changed by this ticket. This is a metadata-only promotion of the resource's maturity/standards status.

### Design Rationale

- FMM 6 is the numeric FMM value reserved for normative content (see `source/versions.html` line 148: `<tr><td><b>FMM 6</b> … title="Normative Content" …`). It is the only FMM value consistent with `standards-status = normative`.
- The modelling precedent in the same repository is Patient (`source/patient/structuredefinition-Patient.xml`), which carries `standards-status=normative`, `normative-version=4.0.0`, and `fmm=5` (locked at the level it reached when going normative). Goal's resolution calls out FMM 6 explicitly, so the analogous extension block for Goal uses FMM 6 and `normative-version=6.0.0`.
- The existing Goal StructureDefinition already has `standards-status=normative` (line 12) but still shows `fmm=2` (line 15) and lacks a `normative-version`. That state is self-inconsistent and confirms the ticket's R6-normative direction is already partially staged in the build — this ticket finishes the job.
- The alternative of setting FMM to 5 (as originally requested) was considered and rejected by the work group at the Sept 2025 WGM in favour of going straight to normative — see the Resolution Description.
