## Resolution Summary

Resolution: **Persuasive with Modification**.

Rather than the reporter's original suggested wording ("It establishes the
meaning of the value and/or components."), the work group approved a more
nuanced rewrite of the **Comments** for both `Observation.code` and
`Observation.value[x]`, plus a cross-link to the existing
"Interoperability Issues using code value pairs in FHIR" section on the
Observation resource page (`observation.html#code-interop`).

Exact wording approved in the resolution description:

- **`Observation.code` — Comments (replaces existing Comments):**
  > *All* code-value and, if present, component.code-component.value pairs
  > need to be taken into account to correctly understand the meaning of
  > the observation. All codings in `Observation.code` are expected to be
  > sufficiently precise to allow a user with the code and value to be
  > able to understand the value — though not all codes will necessarily
  > convey the same level of detail. Codings in `Observation.category`
  > SHOULD be concepts that are too broad to be able to understand the
  > meaning of the `Observation.value` without also having an
  > `Observation.code`.
  >
  > See [Interoperability Issues using code value pairs in FHIR](observation.html#code-interop)
  > for additional information.

- **`Observation.value[x]` — Comments (extends existing list with one new
  bullet):**
  > - See [Interoperability Issues using code value pairs in FHIR](observation.html#code-interop)
  >   for additional information.

`Definition` text and `Short` displays for both elements remain unchanged
(verified against the resolution table — the values shown there match the
current StructureDefinition).

The change has already been merged to `master` in PR
[HL7/fhir#3940](https://github.com/HL7/fhir/pull/3940) — commit
[`7f746aa`](https://github.com/HL7/fhir/commit/7f746aa3032f72241bbd812ad00ced70e651629a)
on 2025-11-06 (per Rob Hausam's applied-vote comment). This plan
documents the implementation as applied for traceability and to support

## Feature Proposal

### Problem Statement

The original `Observation.code` Comments described how multiple
code/value pairs interact and how `Observation.category` should be
broader, but it never tied the reader to the resource's existing
**"Interoperability Issues using code value pairs in FHIR"** narrative
(the `code-interop` section on the Observation page). The reporter's
core concern — that nothing in the element-level definition makes clear
that the code is what gives the value its meaning — was therefore not
addressed in the structured definition itself, only in free-text guidance
elsewhere on the page. The same gap existed at `Observation.value[x]`,
where the Comments list previously linked only to the general "Notes"
section.

Discussion in the ticket comments (Lloyd McKenzie, Andrea Pitkus, 2025-10)
made clear that a blanket statement like "the code defines the meaning"
is not safe — local lab compendium codes, unmapped local codes, and
codes that do not encode the analyte fully all break that simplification.
The agreed-upon resolution therefore avoids a rewrite of the element
**Definition** and instead strengthens the **Comments** with a pointer
to the page-level discussion that already covers these caveats.

### Proposed Change

Two surgical edits in `source/observation/structuredefinition-Observation.xml`,
both inside the `<differential>` block:

1. **`Observation.code` element** (`<element>` at differential index for
   `Observation.code`, currently around line 420). Replace the existing
   `<comment value="…"/>` with the same prose plus, separated by a blank
   line (encoded as `&#xA;&#xA;`), the new sentence:

   > See [Interoperability Issues using code value pairs in FHIR](observation.html#code-interop) for additional information.

2. **`Observation.value[x]` element** (`<element>` at differential index
   for `Observation.value[x]`, currently around line 689). Append a new
   final bullet to the existing Markdown list inside the `<comment
   value="…"/>` (separated by `&#xA;- `):

   > - See [Interoperability Issues using code value pairs in FHIR](observation.html#code-interop) for additional information.

No `<definition>`, `<short>`, `<min>`, `<max>`, `<type>`, `<binding>`, or
constraint changes are required. No new elements, no slicing, no new
extensions, no terminology changes.

### Design Rationale

- The work group preferred linking out to the existing detailed
  discussion (`#code-interop`) rather than baking a one-liner like
  "code establishes the meaning of the value" into the element
  Definition. The discussion in comments showed several real-world
  scenarios (local lab compendium codes, missing LOINC mappings) where
  the simple statement is incorrect, so a pointer to the nuanced
  page-level guidance is safer than an absolute claim in the formal
  Definition.
- The `Comments` field (FHIR ElementDefinition `comment`) is the right
  surface for this: it is rendered into the resource page's element
  table without altering the formal Definition that downstream IGs may
  quote verbatim.
- Keeping the change inside `<differential>` is consistent with the
  HL7/fhir authoring rules — the publisher regenerates `<snapshot>`
  blocks (briefing, "Generated areas (do not edit)").
- Cross-linking with `observation.html#code-interop` reuses an
  already-published anchor (verified at
  `source/observation/observation-notes.xml:167`:
  `<a name="code-interop"></a>`), so no new anchor is introduced and no
  inbound link can be broken.
