## Resolution Summary

**Resolution: Persuasive.** Resolution description (from ticket snapshot):
"We will make the requested changes."

Concretely, the ticket asks for the FormatCode value set to include the
XDS-I (DICOM Key Object Selection) code:

- **Code:** `1.2.840.10008.5.1.4.1.1.88.59`
- **System (as cited in ticket):** `1.2.840.10008.2.6.1` — i.e., DICOM,
  which in FHIR is usually represented via the URI
  `http://dicom.nema.org/resources/ontology/DCM`.
- **Definition:** "DICOM KOS (Image Manifest) (XDS-I)"

The long comment thread (2018 → 2025) clarifies **where** the change must
land: the FormatCode value set is maintained in HL7 Terminology (THO), and
the issue is being processed as a THO change under tracker
[UP-726](https://jira.hl7.org/browse/UP-726). The equivalent change was
already done for another FormatCode addition via UP-523. The FHIR Core
specification's binding on `DocumentReference.content.format` does not
change — only the bound value set (which lives in HL7/UTG) is updated.

The `ExternallyBlocked` and `Pending_UTG` labels reflect exactly this:
the code has been in IHE's FormatCode value set for years, but IHE's
code-system expansion failed; the FHIR core binding re-points to the
HL7-managed composition in THO so that HL7 can enumerate the missing
DICOM code explicitly.

## Feature Proposal

### Problem Statement

`DocumentReference.content.format` is bound (extensible) to
`http://terminology.hl7.org/ValueSet/v3-HL7FormatCodes`, which is
defined in HL7 Terminology (THO) under HL7/UTG at
`input/sourceOfTruth/v3/valueSets/v3-HL7FormatCodes.xml`. That value set
currently composes:

1. All `status = active` codes from IHE's
   `http://ihe.net/fhir/ihe.formatcode.fhir/CodeSystem/formatcode`.
2. All codes from the HL7-owned
   `http://terminology.hl7.org/CodeSystem/v3-HL7DocumentFormatCodes`.

The DICOM-defined Key Object Selection (KOS) / XDS-I code
`1.2.840.10008.5.1.4.1.1.88.59` is documented on the IHE FormatCodes
wiki as an XDS-I format code, but it is **not** present (or not
expandable as `active`) in either included code system. As a result, a
DocumentReference describing an XDS-I DICOM Key Object Selection
document cannot use a FormatCode value that the HL7 value set will
validate, even though the resolution of the ticket (and UP-726) has
agreed this is a legitimate DocumentReference FormatCode.

Separately, IHE maintains the formatcode CodeSystem but has not added
this code there, and — per John Moehrke's 2025-07 comment — the IHE IG's
current terminology expansion fails QA for this code. The ticket has
therefore been re-homed to THO/UTG.

### Proposed Change

Add the DICOM Key Object Selection code to the
`v3-HL7FormatCodes` ValueSet in HL7/UTG so that it expands cleanly and
is available for `DocumentReference.content.format` bindings. The file
already contains a commented-out `<include>` stub with the exact intent;
the fix is to activate (and correct) that include.

**Target file:** `input/sourceOfTruth/v3/valueSets/v3-HL7FormatCodes.xml`
(in `HL7/UTG`).

**Add (uncomment + adjust) the following `<include>` in
`ValueSet.compose`:**

```xml
<include>
  <system value="http://dicom.nema.org/resources/ontology/DCM"/>
  <concept>
    <code value="1.2.840.10008.5.1.4.1.1.88.59"/>
    <display value="Key Object Selection Document"/>
  </concept>
</include>
```

**Version bump:** increment `ValueSet.version` (currently `2.1.0`) to
`2.2.0` and update `ValueSet.date` to the publication date of the change.

**No change** to:
- `DocumentReference.content.format` element definition or binding in
  HL7/fhir (it already references
  `http://terminology.hl7.org/ValueSet/v3-HL7FormatCodes`, which is the
  artifact being updated).
- The `v3-HL7DocumentFormatCodes` CodeSystem (the code belongs in the
  DICOM namespace, not HL7's namespace).
- The IHE `ihe.formatcode.fhir` CodeSystem (out of HL7's editorial
  control; separate IHE tracker if a parallel IHE fix is ever desired).

If the terminology server cannot resolve the DICOM code system (or the
QA of an unreachable reference is unacceptable), a fallback **Option B**
is to add the code as a concept inside the HL7-owned
`v3-HL7DocumentFormatCodes` CodeSystem with a HL7-namespaced display,
keeping the OID-shaped code as-is. This should only be taken if
`http://dicom.nema.org/resources/ontology/DCM` cannot be relied upon at
expansion time; Option A is preferred because the semantics (DICOM SOP
class UID) genuinely belong to DICOM, not HL7.

### Design Rationale

- **Preserve semantic ownership.** The code is a DICOM SOP Class UID.
  Binding it to the DICOM CodeSystem URI
  (`http://dicom.nema.org/resources/ontology/DCM`) is the pattern used
  across FHIR core (e.g., several DICOM-aligned ValueSets use that same
  URI). Burying the code inside `v3-HL7DocumentFormatCodes` would
  misattribute ownership.
- **Match the existing intent.** The `v3-HL7FormatCodes.xml` file
  already contains a commented-out `<include>` with exactly this system
  and code — the authoring intent was recorded but deactivated while
  the authoring question ("should IHE fix this upstream?") was being
  worked. The comment thread on FHIR-17298 now resolves that question:
  IHE won't fix it upstream, so HL7 will enumerate it directly.
- **Minimal blast radius.** Extending a value-set composition is a pure
  additive change. No FHIR Core binding, element, or search parameter
  changes.
- **Follows the UP-523 precedent.** The same work-group path (OO
  ticket → THO/UTG tracker) was used successfully for an earlier
  FormatCode addition to this same ValueSet; reusing the pattern keeps
  governance clean.
