## Resolution Summary

The Orders & Observations work group accepted the request as **Persuasive**
(resolved 2025-10-02, labeled `Ready_to_Apply`). The resolution directs the
publisher to add an explicit `classification` BackboneElement to the
`InventoryItem` resource, modeled directly on
`DeviceDefinition.classification`, instead of relying on a slice of the
existing `characteristic` element. The exact shape called out in the
resolution table is:

- **`InventoryItem.classification`** — `0..*` `BackboneElement`, summary (Σ).
  - Short: *What kind of inventory item this is*
  - Definition: (none supplied in the resolution; mirror the parent shape)
  - Comment: *In this element various classifications can be used*
- **`InventoryItem.classification.type`** — `1..1` `CodeableConcept`, summary (Σ).
  - Short: *A classification of the inventory item*
  - Definition: *A classification or nomenclature for the inventory item*
  - Comment: *In this element various classification systems may be used*
- **`InventoryItem.classification.justification`** — `0..*` `RelatedArtifact`.
  - Short: *Further information qualifying this classification of the inventoryItem*
  - Definition: *Additional information supporting the classification of the inventoryItem*

In addition, the resolution requires a new search parameter:

- **`InventoryItem-type`** — token, expression `InventoryItem.classification.type`,
  description *"The type of inventory item"*.

## Feature Proposal

### Problem Statement

`InventoryItem` (FHIR R6 / build) currently lacks a first-class classification
element. Implementations modelling pharmacy, supply, or device inventory
almost always need to attach one or more code-system-based classifications
(e.g., GMDN, EMDN, SNOMED CT, national product codes) to an inventory entry.
Today, callers have to overload `InventoryItem.code` (single concept for the
specific item) or shoehorn the classification into
`InventoryItem.characteristic` via slicing on `characteristicType`. The
ticket reporter (Jose Costa-Teixeira) flagged that this implicit pattern is
both fragile and inconsistent with peer resources — `DeviceDefinition`
already exposes an explicit `classification` BackboneElement for exactly
this purpose. The work group agreed and resolved the ticket Persuasive.

### Proposed Change

Add an explicit `classification` BackboneElement to
`source/inventoryitem/structuredefinition-InventoryItem.xml`, structured as
follows (paths, cardinality, types, isSummary flags, and short/definition
text are taken verbatim from the resolution; the structure mirrors the
sibling `DeviceDefinition.classification` block at lines 781–824 of
`source/devicedefinition/structuredefinition-DeviceDefinition.xml`):

```xml
<element id="InventoryItem.classification">
  <path value="InventoryItem.classification"/>
  <short value="What kind of inventory item this is"/>
  <definition value="What kind of inventory item this is."/>
  <comment value="In this element various classifications can be used, such as GMDN, EMDN, SNOMED CT, risk classes, national product codes."/>
  <min value="0"/>
  <max value="*"/>
  <type>
    <code value="BackboneElement"/>
  </type>
  <isSummary value="true"/>
</element>
<element id="InventoryItem.classification.type">
  <path value="InventoryItem.classification.type"/>
  <short value="A classification of the inventory item"/>
  <definition value="A classification or nomenclature for the inventory item."/>
  <comment value="In this element various classification systems may be used."/>
  <min value="1"/>
  <max value="1"/>
  <type>
    <code value="CodeableConcept"/>
  </type>
  <isSummary value="true"/>
</element>
<element id="InventoryItem.classification.justification">
  <path value="InventoryItem.classification.justification"/>
  <short value="Further information qualifying this classification of the inventoryItem"/>
  <definition value="Additional information supporting the classification of the inventoryItem."/>
  <min value="0"/>
  <max value="*"/>
  <type>
    <code value="RelatedArtifact"/>
  </type>
</element>
```

Insertion point: between the existing `InventoryItem.code` block (ends at
line 132 in the current SD) and `InventoryItem.name` (starts at line 133).
This places `classification` immediately after `code`, matching the order
in which it appears in the resolution narrative and giving it a position
that mirrors the early-in-the-resource placement of
`DeviceDefinition.classification`.

Add the new search parameter as a fifth `<entry>` in
`source/inventoryitem/bundle-InventoryItem-search-params.xml` (existing
entries: `code`, `subject`, `status`, `identifier`):

```xml
<entry>
  <resource>
    <SearchParameter>
      <id value="InventoryItem-type"/>
      <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
        <valueCode value="normative"/>
      </extension>
      <extension url="http://hl7.org/fhir/build/StructureDefinition/path">
        <valueString value="InventoryItem.classification.type"/>
      </extension>
      <url value="http://hl7.org/fhir/build/SearchParameter/InventoryItem-type"/>
      <description value="The type of inventory item"/>
      <code value="type"/>
      <type value="token"/>
      <expression value="InventoryItem.classification.type"/>
      <processingMode value="normal"/>
    </SearchParameter>
  </resource>
</entry>
```

No new ValueSet binding is introduced — `classification.type` is unbound
(consistent with `DeviceDefinition.classification.type`, which only carries
an example binding to `device-type`; the resolution does not specify a
binding for InventoryItem and explicitly lists multiple competing code
systems as valid).

### Design Rationale

- **Mirrors a peer resource.** `DeviceDefinition.classification` (XML lines
  781–824) uses the same triple of `[BackboneElement] / type:CodeableConcept
  1..1 / justification:RelatedArtifact 0..*`. Reusing that exact shape gives
  implementers a familiar pattern and keeps cross-resource consistency.
- **Explicit element vs. characteristic slice.** The reporter's core
  argument — echoed by the resolution — is that classification is a
  near-universal need for inventory items and should not depend on
  per-implementer slicing of `characteristic`. An explicit element is
  discoverable, summary-flagged, and searchable.
- **Summary flags.** The resolution table marks both `classification` and
  `classification.type` with Σ; this matches `DeviceDefinition` and ensures
  the classification is visible in summary views and bundle searches.
- **No binding.** Because real-world inventory classifications span
  proprietary, regulatory, and clinical code systems (GMDN, EMDN, SNOMED
  CT, NDC, etc.), constraining `type` to a single ValueSet would harm
  interoperability. Leaving it unbound lets implementers choose; this
  matches the resolution wording ("various classifications can be used").
- **Search parameter naming.** `type` is already used by other resources
  (e.g., `DeviceDefinition-type`) for "the kind/classification" search;
  reusing it here keeps the search-name vocabulary consistent.

No breaking changes identified. All additions are new optional elements
(`classification` itself is `0..*`) and a new search parameter. Existing
instances remain valid; existing search behaviour is unaffected. Note that
`classification.type` is `1..1` only when the parent `classification`
backbone is present, which is the conventional "required-within-optional"
shape used by sibling resources and does not constitute a breaking change.

## Implementation Plan

### Prerequisites

None. The change is self-contained within `HL7/fhir`. No dependent
ticket, no terminology addition, and no extension definition must land
first.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add `InventoryItem.classification` differential elements**
   - File: `source/inventoryitem/structuredefinition-InventoryItem.xml`
   - Action: In the `<differential>` block, insert three new `<element>`
     definitions (`InventoryItem.classification`,
     `InventoryItem.classification.type`,
     `InventoryItem.classification.justification`) immediately after the
     existing `InventoryItem.code` element (currently lines 122–132) and
     before `InventoryItem.name` (currently line 133).
   - Details: Use the exact short, definition, comment, cardinality, type,
     and `isSummary` values shown in the **Proposed Change** section
     above. Do not modify the `<snapshot>` block (regenerated by the
     publisher). Preserve XML indentation conventions used in surrounding
     elements (two-space indent, one element per file line where
     practical).

2. **Add the `InventoryItem-type` search parameter**
   - File: `source/inventoryitem/bundle-InventoryItem-search-params.xml`
   - Action: Append a new `<entry>` after the existing `identifier` entry
     and before the closing `</Bundle>` tag.
   - Details: `id="InventoryItem-type"`, `code="type"`, `type="token"`,
     `expression="InventoryItem.classification.type"`,
     `description="The type of inventory item"`,
     `url="http://hl7.org/fhir/build/SearchParameter/InventoryItem-type"`,
     and the `path` build extension set to
     `InventoryItem.classification.type`. Mirror the
     `structuredefinition-standards-status = normative` extension and
     `processingMode = normal` from the sibling entries already present
     in the bundle.

3. **Demonstrate the new element in the example instance**
   - File: `source/inventoryitem/inventoryitem-example.xml`
   - Action: Replace the placeholder `<!-- insert contents here -->`
     section with at least one `<classification>` block containing a
     `<type>` CodeableConcept (e.g., a SNOMED CT or GMDN code) and,
     optionally, a `<justification>` `RelatedArtifact`.
   - Details: Keep the existing `id="example"` and `status="active"`
     elements. Use a code system the publisher already references in
     other examples to avoid introducing unresolvable URIs. Validate
     that the example still parses against the regenerated schema.

4. **Refresh narrative pages for the new element**
   - File: `source/inventoryitem/inventoryitem-introduction.xml`
   - Action: Review the resource scope/boundary prose; if it currently
     instructs implementers to use `characteristic` for classification,
     update the wording to point at `classification` instead and note
     that `characteristic` remains for other descriptive properties.
   - Details: Keep edits minimal and HTML-safe (the publisher injects
     this snippet into the resource page).

5. **Audit the notes page for stale guidance**
   - File: `source/inventoryitem/inventoryitem-notes.xml`
   - Action: Read the notes page and remove or amend any sentence that
     conflicts with having an explicit `classification` element.
   - Details: The current notes file contains the standard draft-resource
     STU note; verify nothing else needs adjustment. If no conflicting
     prose exists, no edit is required — record that explicitly in the
     PR description.

6. **(Verification only) Confirm no other source files reference InventoryItem classification**
   - Files: Repo-wide search across `source/`.
   - Action: Run `git grep -n "InventoryItem.classification"` and
     `git grep -n "InventoryItem-type"` to ensure the new paths/search
     codes are not already used elsewhere with a different meaning.
   - Details: This is a safety check before the publisher run; no edit
     expected.

### Testing Considerations

- **Resource validation:** Run the publisher (or the targeted SD
  validation step it exposes) on `InventoryItem` and ensure no errors or
  new warnings are introduced. Confirm the regenerated snapshot includes
  the three new element definitions.
- **Search parameter validation:** Confirm the publisher's search-param
  consistency check passes — specifically that the `expression`
  `InventoryItem.classification.type` resolves and that the new param's
  `code` (`type`) does not collide with any other InventoryItem search
  param (today the bundle defines `code`, `subject`, `status`,
  `identifier`; `type` is free).
- **Example revalidation:** Validate the updated `inventoryitem-example`
  XML against the regenerated `InventoryItem.xsd` (or whatever schema
  artefact the publisher emits) to ensure the new `classification` block
  is well-formed.
- **Edge cases to verify:** an InventoryItem with multiple
  `classification` entries (different code systems); a classification
  with a `justification` referencing an external URL via
  `RelatedArtifact`; absence of `classification` (still valid because
  cardinality is 0..*).
- **Interoperability sanity check:** Confirm peer-resource consistency
  by diffing the new XML against
  `source/devicedefinition/structuredefinition-DeviceDefinition.xml`
  lines 781–824 — the only intentional differences should be the
  resource name, the short/definition text, and the absence of the
  example binding to `device-type`.

### Open Questions

- **`classification.type` binding strength.** The resolution table omits a
  binding. `DeviceDefinition.classification.type` carries an example
  binding to `device-type`. Decision needed before merge: leave
  `InventoryItem.classification.type` entirely unbound (current proposal,
  matches the resolution), or attach an `example` binding to a
  general-purpose ValueSet for editor tooling support. The plan
  currently implements the former.
- **Definition text on the parent `classification` element.** The
  resolution table leaves the `Definition:` cell blank for the parent
  element. The plan fills it with `"What kind of inventory item this is."`
  (echoing the short description), matching the
  `DeviceDefinition.classification` convention. Confirm with the work
  group editor before merge.
- **Cross-link in narrative.** Whether to add an explicit pointer from the
  InventoryItem narrative to `DeviceDefinition.classification` (helpful
  for implementers but not required by the resolution). The plan flags
  this as an optional narrative refresh in Task 4.
- **Snapshot regeneration cadence.** The FhirCore playbook says
  `<snapshot>` is publisher-generated; confirm the local publisher run
  before PR will pick up the new differential elements without manual
  snapshot edits.
