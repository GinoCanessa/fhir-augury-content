## Resolution Summary

**Resolution:** Persuasive with Modification.

The committee will add the `unknown` code from the canonical FHIR status
CodeSystem (`http://hl7.org/fhir/resource-status`, display *"unknown"*,
definition *"Authoring system does not know the status"*) to the
`DocumentReference.status` binding so that authoring systems have an
escape valve when the status is not known. The code is taken from the
shared `resource-status` CodeSystem rather than being added to
`document-reference-status`, so the change is a ValueSet‑composition
change, not a CodeSystem‑content change.

Resolution text, verbatim:

> Will add: `http://hl7.org/fhir/resource-status` → `unknown` / unknown /
> "Authoring system does not know the status" to DocumentReference.status.

Ticket comments note explicitly that `DocumentReference.status` remains
a modifier after this change (the `isModifier` / `isModifierReason`
framing for `entered-in-error` is not removed). A follow-up Zulip/Jira
comment from the assignee flagged that adding `unknown`/`other` *could*
justify removing the modifier, but Lloyd McKenzie replied that it would
not — and no removal of modifier flags is part of the approved
resolution.

## Feature Proposal

### Problem Statement

`DocumentReference.status` is mandatory (`1..1`) with a *required*
binding to `http://hl7.org/fhir/ValueSet/document-reference-status`.
That ValueSet is implicitly the full content of the
`document-reference-status` CodeSystem (see
`source/documentreference/codesystem-document-reference-status.xml`
lines 37–53), which defines only three concepts: `current`,
`superseded`, and `entered-in-error`. None of those represent
"status is not known".

When a sender cannot assert any of those three states (for example, when
receiving a document from an external feed that does not carry a
lifecycle status), they are forced either to mis-classify the reference
or to omit a mandatory element — neither of which is a legal or
clinically safe choice. Reporter (Lloyd McKenzie) asked for an
`unknown` / `other` escape valve so the element can always be populated
truthfully.

### Proposed Change

Bind `DocumentReference.status` to an explicit ValueSet that composes
the existing `document-reference-status` CodeSystem with a single code
(`unknown`) drawn from the shared `resource-status` CodeSystem.

Concretely:

1. **New ValueSet resource** (authored file:
   `source/documentreference/valueset-document-reference-status.xml`):

   ```xml
   <ValueSet xmlns="http://hl7.org/fhir">
     <id value="document-reference-status"/>
     <url value="http://hl7.org/fhir/ValueSet/document-reference-status"/>
     <identifier>
       <system value="urn:ietf:rfc:3986"/>
       <value value="urn:oid:2.16.840.1.113883.4.642.3.7"/>
     </identifier>
     <name value="DocumentReferenceStatus"/>
     <title value="DocumentReference Status"/>
     <status value="active"/>
     <experimental value="false"/>
     <description value="The status of the document reference."/>
     <compose>
       <include>
         <system value="http://hl7.org/fhir/document-reference-status"/>
       </include>
       <include>
         <system value="http://hl7.org/fhir/resource-status"/>
         <concept>
           <code value="unknown"/>
           <display value="unknown"/>
         </concept>
       </include>
     </compose>
   </ValueSet>
   ```

   Preserve the existing OID for this ValueSet
   (`2.16.840.1.113883.4.642.3.7`, already registered in
   `source/oids.ini:376`) plus the normative standards-status extensions
   and work-group extension that apply to the `document-reference-status`
   CodeSystem (`oo`, `normative`, `fmm=3`) so the ValueSet's metadata is
   consistent with its content.

   This pattern mirrors `source/fivews/valueset-resource-status.xml`
   (ValueSet composing from `http://hl7.org/fhir/resource-status`), which
   is the nearest in-repo example of an explicit compose against the
   canonical status CodeSystem.

2. **Detach the ValueSet from the CodeSystem's implicit generation.**
   Remove (or keep but rely on the explicit file winning) the
   `<valueSet value="http://hl7.org/fhir/ValueSet/document-reference-status"/>`
   line at `source/documentreference/codesystem-document-reference-status.xml:37`
   so that the publisher does not try to auto-generate a competing
   implicit ValueSet with the same canonical URL. (Verify behaviour with
   the assignee — see Open Questions.)

3. **Update the StructureDefinition short form and binding description.**
   In `source/documentreference/structuredefinition-DocumentReference.xml`
   at the `DocumentReference.status` element (lines 236–262):
   - Line 238: change
     `<short value="current | superseded | entered-in-error"/>` to
     `<short value="current | superseded | entered-in-error | unknown"/>`.
   - Line 260: leave the `<description>` of the binding as
     "The status of the document reference." (still accurate).
   - Do **not** change `isModifier` (line 246) or `isModifierReason`
     (line 247). Per Lloyd McKenzie's comment dated 2026-01-29, the
     presence of `unknown` does not change the element's modifier
     status; `entered-in-error` is still the modifier-worthy code.

4. **Add an example or update an existing example** under
   `source/documentreference/` that uses `status="unknown"`, so
   implementations can see the new code exercised (see
   `documentreference-example-comprehensive.xml` or add a small
   `documentreference-example-unknown-status.xml`; register it in the
   list of examples: `list-DocumentReference-examples.xml`). The
   resolution does not mandate this, but it is normal practice for any
   binding-expansion change.

5. **Add release-note text** (label `R6-Release-note` is already on
   the ticket). The DocumentReference introduction/notes page
   (`source/documentreference/documentreference-introduction.xml` or
   `documentreference-notes.xml`) should note that `status` now permits
   `unknown` and point at the canonical `resource-status` CodeSystem.

### Design Rationale

- **Use the shared CodeSystem, not a local one.** The resolution
  explicitly draws `unknown` from `http://hl7.org/fhir/resource-status`
  rather than minting a local `document-reference-status#unknown`. This
  keeps DocumentReference aligned with the FHIR-wide convention that
  `unknown` is the canonical escape valve across status-bearing
  resources, and it avoids fragmenting the concept across many local
  CodeSystems.
- **Compose rather than extend the local CodeSystem.** The local
  CodeSystem (`document-reference-status`) is marked `normative` with
  `fmm=3`; adding a new concept to a normative CodeSystem is a much
  heavier change than binding to a ValueSet that composes the existing
  CodeSystem plus a single external code.
- **Modifier flag unchanged.** The assignee (John D.L. Nolen) raised
  the concern that adding `unknown` might argue for dropping
  `isModifier`, but the reporter (Lloyd McKenzie) replied that the
  modifier semantics are driven by `entered-in-error`, which remains.
  The resolution records no modifier change, so neither `isModifier`
  nor `isModifierReason` should be touched.
- **Required binding retained.** Because the change is about widening
  the required ValueSet rather than changing binding strength, the
  `<strength value="required"/>` at line 259 stays. Only the ValueSet's
  content is different.

- **Affected Files:**

    — **new**: explicit compose of
    `http://hl7.org/fhir/document-reference-status` (all codes) and the
    single `unknown` code from
    `http://hl7.org/fhir/resource-status`.
  - `source/documentreference/codesystem-document-reference-status.xml`
    — remove or review the implicit
    `<valueSet value="http://hl7.org/fhir/ValueSet/document-reference-status"/>`
    (line 37) so the new explicit ValueSet does not collide.
  - `source/documentreference/structuredefinition-DocumentReference.xml`
    — update `short` on `DocumentReference.status` (line 238) to include
    `| unknown`.
  - *(optional)* `source/documentreference/documentreference-notes.xml`
    or `documentreference-introduction.xml` — brief R6 release note
    prose.
  - *(optional)* `source/documentreference/list-DocumentReference-examples.xml`
    + new example XML — add an example instance with `status="unknown"`.
- **Change Scope:** Minor. One new XML file, small edits to one
  CodeSystem and one StructureDefinition differential, plus optional
  documentation and example touches. No schema, no code, no generated
  output is hand-edited.

### Breaking Changes

- **Required binding widened.** From a consumer's perspective, this is
  **non-breaking** for senders (any document that validated before
  still validates — the new `unknown` code is purely additive) but
  **potentially breaking for receivers** written against the old
  ValueSet enumeration: they must now accept and reason about
  `status="unknown"` in addition to `current | superseded |
  entered-in-error`. This is the standard additive-code impact and
  should be called out in the R6 release note.
- No cardinality, type, or modifier flag changes.
- No change to `strength = required`, so implementations may not send
  any code outside the new ValueSet.

### Related Specifications

- `http://hl7.org/fhir/ValueSet/resource-status` (defined at
  `source/fivews/valueset-resource-status.xml`) — the existing ValueSet
  that already composes all of `resource-status`. The new DocumentReference
  ValueSet deliberately does *not* use this ValueSet wholesale; it
  selects only `unknown` to avoid polluting DocumentReference.status
  with the rest of the canonical status enumeration.
- Other status-bearing resources with required bindings that lack an
  `unknown` escape valve may ultimately need the same treatment — not
  in scope here.
- Concept map generation: if a ConceptMap from
  `document-reference-status` exists, it is unaffected because the
  existing local codes are unchanged. `unknown` is mapped upstream via
  `resource-status` mappings.
