## Resolution Summary

Resolution: **Persuasive with Modification**.

The approved resolution adds a new repeating `subspecification` element to the
`conformsTo` backbone of both `Device` and `DeviceDefinition` so devices can
declare conformance to sub-standard items such as IHE Named Options, DICOM SOP
classes, and Bluetooth profile features. Per the 2025-06-30 disposition update
(noted by Marti Velezis in the 2025-08-12 comment), an `actor` element was
also added to the resolution to capture the IHE-style actor that the device
implements.

Concretely, on `Device.conformsTo` and `DeviceDefinition.conformsTo`:

- **`actor` 0..1 CodeableConcept** — "The actor that conforms to the stated
  specification and subspecification(s)."
- **`subspecification` 0..* CodeableConcept** — "Identifies the clause or
  option within the actor, or if no actor is specified, within the
  specification, to which the device conforms." Definition: "An option or
  clause within the specification the device conforms to."

Companion changes called for in the disposition:

- New search parameters: `subspecification` (token) and a composite
  `specification`+`subspecification` (paired with the existing
  `specification` token search parameter).
- Update `device-example-Bodimetrics.xml` to add a `conformsTo` instance with
  `specification = Bluetooth` and `subspecification = Basic ECG (Heart Rate)`.
- Add a `subspecification` example value (DICOM SOP class
  `urn:oid:1.2.840.10008.5.1.4.1.1.2`, system `urn:dicom:uid`) to the
  example ValueSet for `subspecification`.

> **Caveat from August 2025 committee follow-up.** The 2025-08-27 comments
> from costateixeira and Marti Velezis state that the requested ValueSet
> work cannot be done as drafted and that the committee will "revise the
> ticket and pull off the code systems/value sets work." This plan therefore
> implements the StructureDefinition + SearchParameter + example-instance
> portions of the disposition (which Marti confirmed on 2025-08-12 should be
> applied) and treats the new ValueSet content for `subspecification` as a
> deferred, committee-blocked item. See **Open Questions**.

## Feature Proposal

### Problem Statement

FHIR-39242 renamed `Device.specialization` / `DeviceDefinition.specialization`
to `conformsTo`, and at the time the work group acknowledged a remaining gap:
the `conformsTo` backbone (currently `category`, `specification`, `version`,
plus `source` on `DeviceDefinition`) cannot express conformance to a
*sub-standard* unit — for example:

- An IHE Actor + Named Option ("Document Repository, asynchronous web
  services exchange option" inside IHE XDS).
- A DICOM SOP class ("CT Image Storage" inside DICOM).
- A Bluetooth Health Device Profile service ("HDP / Basic ECG (Heart
  Rate)" inside Bluetooth).

Today implementers either overload `specification.coding` with codes that
mix the standard with one of its options, or fall back to a free-text
narrative, neither of which is computable. The reporter (Elliot Silver) and
the OO/DEV joint working session on 2024-05-20 concluded that a dedicated,
repeating `subspecification` slot is required, and follow-up discussion
(esilver / john_moehrke 2024-05-27/28) made it clear that distinguishing
the *role* the device plays in the standard (an IHE-style actor, e.g.,
provider vs. consumer of a DICOM SOP class) is also necessary so the claim
is sufficiently expressive — driving the additional `actor` element added
to the disposition in June 2025.

### Proposed Change

In **`source/device/structuredefinition-Device.xml`** and
**`source/devicedefinition/structuredefinition-DeviceDefinition.xml`**, add
two new child elements to the `conformsTo` BackboneElement, placed between
the existing `specification` and `version` children (matching the order the
disposition table renders them):

1. **`{Resource}.conformsTo.actor`**
   - Cardinality: `0..1`
   - Type: `CodeableConcept`
   - Short: "The actor that conforms to the stated specification and
     subspecification(s)."
   - Definition: "The actor (e.g., an IHE Actor) within the named
     specification that the device implements. When present, any
     subspecifications are scoped to this actor."
   - Binding: none defined yet (no value set in the disposition); leave
     unbound or attach an example binding once the OO committee provides a
     value set.

2. **`{Resource}.conformsTo.subspecification`**
   - Cardinality: `0..*`
   - Type: `CodeableConcept`
   - Short: "Identifies the clause or option within the actor, or if no
     actor is specified, within the specification, to which the device
     conforms."
   - Definition: "An option or clause within the specification the device
     conforms to."
   - Comment: examples include the asynchronous web-services option on
     the IHE XDS Document Registry actor, the DICOM CT Image Storage SOP
     class, and the Bluetooth Health Device Profile (HDP) / Basic ECG
     (Heart Rate) feature.
   - Binding: example binding to the existing `subspecification` example
     ValueSet (see deferred terminology task below; if the value set is
     not yet published, omit the `<binding>` and add a TODO note).

For `DeviceDefinition`, mirror the elements with the same definitions and
mark them `isSummary="true"` to match the surrounding
`DeviceDefinition.conformsTo.*` elements (which all carry
`isSummary="true"` today).

In **`source/device/bundle-Device-search-params.xml`**, add:

3. **SearchParameter `Device-subspecification`** — `token`, expression
   `Device.conformsTo.subspecification`, code `subspecification`,
   description "An option or clause within the specification the device
   conforms to."
4. **SearchParameter `Device-specification-subspecification`** —
   `composite`, expression `Device.conformsTo`, two components:
   `specification` (`specification.ofType(CodeableConcept)`) and
   `subspecification` (`subspecification.ofType(CodeableConcept)`),
   description "A composite of both specification and subspecification."

In **`source/devicedefinition/bundle-DeviceDefinition-search-params.xml`**,
add the equivalent two SearchParameters for `DeviceDefinition.conformsTo`.

In **`source/device/device-example-Bodimetrics.xml`**, add an additional
`<conformsTo>` block that uses `specification` = Bluetooth and
`subspecification` = Basic ECG (Heart Rate), demonstrating the new element.

In **`source/device/valueset-device-specification-type.xml`** (or a new
`valueset-device-subspecification-example.xml`, see Open Questions) add the
DICOM SOP class concept `urn:oid:1.2.840.10008.5.1.4.1.1.2` from system
`urn:dicom:uid` as an example concept.

### Design Rationale

- A repeating `0..*` `CodeableConcept` mirrors the existing
  `specification` slot's data shape and keeps the backbone uniformly
  coded. Implementers already know how to populate `category` /
  `specification` with a coded `CodeableConcept` and the new field
  follows the same pattern.
- Putting `actor` and `subspecification` *inside* the `conformsTo`
  backbone (rather than creating peer elements on `Device` /
  `DeviceDefinition`, or carrying the data on a core extension) keeps
  one assertion of conformance — actor + specification + subspecification
  + version — atomic and unambiguous. This was the alternative weighed
  against extensions in the original ticket description and rejected by
  the 2024-05-20 Devices working session.
- The composite `specification`+`subspecification` search parameter
  matches the pattern already used for the existing
  `specification`+`version` composite (`Device-specification-version`,
  see lines 310–333 of `bundle-Device-search-params.xml`), so clients can
  query for a specific (specification, subspecification) pair without
  cross-resource joins.
- Adding `actor` as `0..1` (rather than `0..*`) was the disposition's
  choice; a device claim made by a single resource instance is for one
  role at a time, and a device that implements multiple actors models
  each as a separate `conformsTo` repetition — preserving 1:1 scoping
  between actor and the subspecifications under it.

### Breaking Changes

**No breaking changes** to core conformance of *existing* `Device` /
`DeviceDefinition` instances:

- `actor` and `subspecification` are both **optional** (`0..1` and
  `0..*`); existing instances without them remain valid.
- No element renames, type changes, or cardinality reductions on
  existing `conformsTo.*` children.
- The new search parameters are additive; no existing parameter is
  modified.

Caveats worth flagging in the spec change-log:

- Implementers that previously crammed sub-standard semantics into
  `specification.coding` will be encouraged (but not required) to
  migrate to `subspecification`. This is a *guidance* shift, not a
  conformance break.
- StructureDefinition snapshots regenerate, so any client that diffs the
  raw snapshot will see additions even where it didn't change behaviour.
