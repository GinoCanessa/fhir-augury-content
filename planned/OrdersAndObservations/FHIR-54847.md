## Resolution Summary

Resolution: **Persuasive with Modification**. The applied resolution has three
parts:

1. Create a new ValueSet `parameter-focus-code` that is the **union of the
   `device-type` and `medication-codes` Value Sets**, and bind it
   **extensible** to `ServiceRequest.orderDetail.parameterFocus[x]` (the
   `CodeableConcept` choice).
2. Change the long definition of `ServiceRequest.orderDetail.parameterFocus[x]`
   to: *"Indicates the context of the order details by reference or code when
   the resource instance is not available."*
3. Add a section to the ServiceRequest **Notes** explaining how
   `orderDetail` and `parameterFocus[x]` are used (drawn from the chat /
   committers Zulip discussion captured in comments).

Note from JIRAUSER19407 (2026-02-27): the resolution language about
"changing the element type code, 0..1 to extensible binding" needed
revisiting because *you cannot have an extensible binding on a `code`* — the
binding must be applied to the `CodeableConcept` choice of the choice
element, not to the element as a whole. The resolution is intended to add a
binding only to the `CodeableConcept` type.

A draft implementation exists at
[HL7/fhir#4016](https://github.com/HL7/fhir/pull/4016) ("FHIR-54847: Add
binding and improve definitions for parameterFocus[x]") by jdlnolen — open
and draft as of 2026-04-09. The PR diverges slightly from the resolution
text (it currently uses softer "short" wording and adds a `comment` rather
than fully replacing the long definition) and should be reconciled with the
resolution before merge.

## Feature Proposal

### Problem Statement

`ServiceRequest.orderDetail.parameterFocus[x]` is a choice element that
includes `Reference`, `canonical`, and `CodeableConcept`. The
`CodeableConcept` choice was added during R6 work (per the FHIR-39389 thread
referenced in comments) so that implementers without access to definitional
resources such as `DeviceDefinition` or `Medication` could still indicate
the focus of an order detail in principle.

However, in 6.0.0-ballot4 the `CodeableConcept` choice has **no binding at
all**, and the element's existing `short` and `definition` are essentially
identical and tied to the "by reference" phrasing — leaving implementers
unable to tell what kinds of codes are appropriate or what the element is
intended to convey. The reporter explicitly flagged that the short and long
definitions "are not very different and don't provide enough detail for me
to understand what focus is used for."

### Proposed Change

Apply the three resolution items in `HL7/fhir`, all under
`source/servicerequest/`:

1. **New ValueSet `parameter-focus-code`** (`ValueSet/parameter-focus-code`,
   canonical `http://hl7.org/fhir/ValueSet/parameter-focus-code`):
   - `name` = `ParameterFocusCode`, `title` = "Parameter Focus Code".
   - WG extension `valueCode` = `oo`; standards-status `trial-use`.
   - `status` = `draft`, `experimental` = `false`.
   - Description: codes identifying the focus item (e.g., medication,
     device, substance) that order detail parameters apply to; explicitly
     called out as the union of Device Type and Medication Codes value
     sets.
   - SNOMED CT copyright clause (because `medication-codes` includes SNOMED
     CT content).
   - `compose.include` two `valueSet` entries:
     - `http://hl7.org/fhir/ValueSet/device-type`
     - `http://hl7.org/fhir/ValueSet/medication-codes`
   - File: `source/servicerequest/valueset-parameter-focus-code.xml`.

2. **Bind `parameterFocus[x]` (CodeableConcept) extensible** to the new
   ValueSet in `source/servicerequest/structuredefinition-ServiceRequest.xml`
   on the `ServiceRequest.orderDetail.parameterFocus[x]` element:
   - `binding.strength` = `extensible`
   - `binding.valueSet` =
     `http://hl7.org/fhir/ValueSet/parameter-focus-code`
   - `binding.description` = "Codes identifying the focus item (e.g.,
     medication, device, substance) that the order detail parameters apply
     to."
   - `elementdefinition-bindingName` extension `valueString` =
     `OrderDetailParameterFocus`.
   - The binding is on the choice element, but per FHIR rules the binding
     applies to the `CodeableConcept` choice of the type; this satisfies
     the 2026-02-27 caveat about not being able to bind a `code`.

3. **Definition / short / comment text** on the same element (per the
   resolution and the Zulip-derived note proposed by Eric Silver on
   2026-03-03):
   - `short` = "The product or item that the order detail parameters apply
     to" (PR #4016 wording) **OR** keep aligned to the resolution wording
     "The context of the order details by reference or code". The PR
     currently uses the former — confirm with the WG which to ship.
   - `definition` = "Indicates the context of the order details by
     reference or code when the resource instance is not available." (per
     resolution, item 2).
   - Add an element-level `comment` capturing usage guidance — when the
     reference choice is preferred, when canonical is appropriate, and
     when CodeableConcept is acceptable (text is in the 2026-03-03 esilver
     comment on the ticket, lightly cleaned per the same author's
     follow-up: drop the extra space in "patient,  body positioning" and
     remove the commas after "orderDetail" and "values").

4. **ServiceRequest Notes** update in
   `source/servicerequest/servicerequest-notes.xml` — add a new `<li>` (or
   short subsection with anchor) explaining `orderDetail` /
   `parameterFocus[x]` usage. Use the cleaned esilver text:
   *"Use orderDetail to provide details about the request, such as
   information to collect from the patient, body positioning, appropriate
   device settings, preferred bandage size, etc. Each repetition of
   orderDetail includes a list of parameter codes and values providing the
   details. If the details are about a specific device, substance,
   medication, etc. involved in the service, that item is indicated in the
   parameterFocus[x] element; if the details are more generally about the
   service itself, parameterFocus is omitted.*

   *The parameterFocus[x] choice element allows several ways to identify
   the target of the parameters:*

   - *parameterFocusReference — used when there is a specific resource
     that is the target of the focus (most common; e.g., a specific
     ventilator Device at the patient's bedside).*
   - *parameterFocusCanonical — used when a canonical resource provides
     the definition of a type of resource (e.g., "DeviceCo's AwesomeVent
     model 3").*
   - *parameterFocusCodeableConcept — used to specify, for example,
     device or medication types as a coded value; specificity is
     implementation-dependent.*

   *Generally parameterFocusReference or parameterFocusCanonical provide
   more information; however, implementations that do not use definitional
   resources (such as DeviceDefinition or Medication) might prefer
   parameterFocusCodeableConcept.*

   *The orderDetail.parameter.code identifies the detail being specified.
   The codes used are typically dependent on the focus item (if any) and
   may be implementation specific. Similarly, the type of the
   orderDetail.parameter.value[x] choice element and the appropriate value
   will depend on the focus item (if any) and the parameter code."*

### Design Rationale

- **Union ValueSet over inventing a new code system.** The committers
  thread (Zulip topic 582871954) and the resolution converge on reusing
  existing terminology (`device-type` and `medication-codes`) rather than
  creating bespoke parameter-focus codes. This keeps the binding consistent
  with the kinds of *resources* that can be referenced via
  `parameterFocusReference` (Device, DeviceDefinition, Medication, etc.),
  which is the design intent recorded in the 2026-02-20 author comment
  (binding "should generally reflect the classes of resource that are
  referenceable").
- **Extensible (not required) binding.** Implementers may need codes
  outside Device Type / Medication Codes (e.g., substances or biologically
  derived products); extensible permits this without forcing terminology
  expansion.
- **Binding applied to the CodeableConcept choice only.** The 2026-02-27
  comment correctly points out that an extensible binding cannot apply to a
  `code` element. By placing the binding on `parameterFocus[x]` (and
  letting the publisher resolve it for the `CodeableConcept` choice), the
  resolution is realized without violating the `code` rule. (The `Reference`
  and `canonical` choices already constrain via `targetProfile`.)
- **Notes section over per-element prose only.** The orderDetail flow
  (parameter + parameterFocus + parameter.code/value) is not obvious from
  any single element definition; the resolution and the chat both call for
  consolidated narrative in `servicerequest-notes.xml` so callers can find
  it on the resource page.

### Breaking Changes

**No breaking changes for the resource shape.** Adding an `extensible`
binding to a previously unbound `CodeableConcept` choice is **conformance
tightening**: instances that previously used arbitrary codes remain
schematically valid, but extensible binding now expects a code from
`parameter-focus-code` *unless none in the ValueSet adequately represents
the concept*. Implementations that currently use unrelated codes should
review whether their codes belong in Device Type / Medication Codes (or
should be added through extension).

There is **no element renaming, no cardinality change, and no type change**
to `parameterFocus[x]`. The `Reference` and `canonical` choices are
unaffected.

### Related Specifications

- `Device`, `DeviceDefinition`, `Medication`, `Substance`,
  `BiologicallyDerivedProduct` — referenced indirectly via the union
  ValueSet's underlying `device-type` and `medication-codes` (and via the
  `Reference`/`canonical` choices on `parameterFocus[x]`). No conformance
  edits required in their definitions.
- Other resources that align with workflow `Request.product` mapping
  (workflow pattern docs) — already mapped; no spec change needed.
- Downstream IGs profiling `ServiceRequest.orderDetail` in
  `IgRepositories` may inherit or tighten the new binding; this is
  informational only.

### Related Tickets

- **FHIR-39389 — *DeviceDefinition should follow definitional resource
  modelling*:** Cited in the 2026-02-20 resolution comment as the upstream
  cause. Changing `parameterFocus` from `CodeableReference` to `Reference`
  removed the ability to express the focus as a code; the new
  `parameterFocusCodeableConcept` choice (and now its binding) restores it.
  FHIR-54847 complements FHIR-39389; no conflict.
- The `cross-referenced` query for FHIR-54847 returned only PR #4016 and

### Terminology Impact

- **New ValueSet:** `http://hl7.org/fhir/ValueSet/parameter-focus-code`
  ("Parameter Focus Code"), composed as the union of:
  - `http://hl7.org/fhir/ValueSet/device-type`
  - `http://hl7.org/fhir/ValueSet/medication-codes`
- **No new CodeSystem.** No content additions are required in `HL7/UTG`;

- **WG extension** `structuredefinition-wg` = `oo` and standards-status
