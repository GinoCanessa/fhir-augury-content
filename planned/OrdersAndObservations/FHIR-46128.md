## Resolution Summary

**Resolution:** Persuasive with Modification.

The resolution updates the language bindings on two backbone-element children of
`InventoryItem` so that they no longer force callers into the curated
`CommonLanguages` (`http://hl7.org/fhir/ValueSet/languages`) value set:

1. **`InventoryItem.name.language`** — change the binding to
   `http://hl7.org/fhir/ValueSet/all-languages` and set the strength to
   **`preferred`** (was `required` against `languages`). Keep cardinality `1..1`
   and type `code`. Keep the `bindingName` extension `NameLanguage` and the
   `binding-definition` "The languages for an item name".
2. **`InventoryItem.description.language`** — change the binding to
   `http://hl7.org/fhir/ValueSet/all-languages` and set the strength to
   **`preferred`** (was `required` against `languages`). Keep cardinality `0..1`
   and type `code`. Keep the `bindingName` extension `ItemDescriptionLanguage`
   and the `binding-definition` "The languages for a description of an item".

The ticket title and description only call out `description.language`, but the
resolution-description tables (rendered from build.fhir.org) explicitly also
update `name.language` with the same change. Both elements must be updated.

The resolution cross-references FHIR-46074 (Terminology Infrastructure), which
made the analogous change for `TerminologyCapabilities.codeSystem.version.language`.

## Feature Proposal

### Problem Statement

`InventoryItem.name.language` and `InventoryItem.description.language` are bound
to `http://hl7.org/fhir/ValueSet/languages` (CommonLanguages — a small curated
list) with strength `required`. That binding prevents implementers from
recording item names and descriptions in any human language outside the
curated subset, even though IETF BCP 47 already provides codes for every
natural language. The reporter notes that FHIR Core elsewhere prefers
`http://hl7.org/fhir/ValueSet/all-languages` (the full IETF tag set) for
`*.language` elements, and FHIR-46074 already established this pattern for
`TerminologyCapabilities.codeSystem.version.language` for the same reason.

### Proposed Change

Edit `source/inventoryitem/structuredefinition-InventoryItem.xml` in
`HL7/fhir` and update the two `<binding>` blocks identified below. No other
files in the InventoryItem directory reference these bindings (search params,
examples, code system / value set helpers, and narrative pages do not mention
these elements' bindings).

**1. `InventoryItem.name.language` (element id at line 166):**

Current `<binding>` (lines 177–186):

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="NameLanguage"/>
  </extension>
  <extension url="http://hl7.org/fhir/tools/StructureDefinition/binding-definition">
    <valueString value="The languages for an item name"/>
  </extension>
  <strength value="required"/>
  <description value="Name languages."/>
  <valueSet value="http://hl7.org/fhir/ValueSet/languages"/>
</binding>
```

Becomes:

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="NameLanguage"/>
  </extension>
  <extension url="http://hl7.org/fhir/tools/StructureDefinition/binding-definition">
    <valueString value="The languages for an item name"/>
  </extension>
  <strength value="preferred"/>
  <description value="Name languages."/>
  <valueSet value="http://hl7.org/fhir/ValueSet/all-languages"/>
</binding>
```

**2. `InventoryItem.description.language` (element id at line 240):**

Current `<binding>` (lines 250–259):

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="ItemDescriptionLanguage"/>
  </extension>
  <extension url="http://hl7.org/fhir/tools/StructureDefinition/binding-definition">
    <valueString value="The languages for a description of an item"/>
  </extension>
  <strength value="required"/>
  <description value="Description languages."/>
  <valueSet value="http://hl7.org/fhir/ValueSet/languages"/>
</binding>
```

Becomes:

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="ItemDescriptionLanguage"/>
  </extension>
  <extension url="http://hl7.org/fhir/tools/StructureDefinition/binding-definition">
    <valueString value="The languages for a description of an item"/>
  </extension>
  <strength value="preferred"/>
  <description value="Description languages."/>
  <valueSet value="http://hl7.org/fhir/ValueSet/all-languages"/>
</binding>
```

Both edits sit inside `<differential>` (the only hand-edited region of an
HL7/fhir StructureDefinition); the publisher will regenerate `<snapshot>`.

### Design Rationale

- **Consistency with FHIR-46074.** The Terminology Infrastructure WG already
  applied the same fix to `TerminologyCapabilities.codeSystem.version.language`
  (now bound to `all-languages`). Aligning InventoryItem's two language
  elements removes a real-world implementer pain point identified during
  ballot for v5.0/R6.
- **`all-languages` is the right VS for free-form `code` language tags.** Item
  names and descriptions are authored content; restricting them to a curated
  short list of common languages forces non-conformant data for any
  legitimate IETF BCP 47 tag outside that list.
- **Preferred (not extensible) matches the resolution wording.** The
  resolution-description tables explicitly show "make binding strength
  (Preferred)" for both elements, even though the ticket title offers
  "preferred or extensible" as alternatives. The applied wording governs.
- **Keep the binding name and binding-definition extensions.** They are
  authored, semantic markers used by the publisher and downstream tooling and
  are not part of the change.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole repository: defines the InventoryItem StructureDefinition
  whose two language bindings are being relaxed.
- **Affected Files:**
  - `source/inventoryitem/structuredefinition-InventoryItem.xml` — edit
    `<binding>` for `InventoryItem.name.language` (lines 177–186) and
    `InventoryItem.description.language` (lines 250–259); change `<strength>`
    `required` → `preferred` and `<valueSet>` `…/ValueSet/languages` →
    `…/ValueSet/all-languages` in both blocks.
- **Change Scope:** Minor.

### Breaking Changes

- **Yes — binding-strength relaxation from `required` to `preferred`** on a
  normative-track resource element is technically a backward-compatible change
  for *producers* (any previously valid instance using a CommonLanguages tag
  is still valid against the new `all-languages` value set, which is a
  superset) but is a **conformance-relevant change** for *consumers/validators*
  that previously rejected any code outside CommonLanguages. Validators must
  be updated/re-run; downstream profiles that re-tighten the binding to
  `required` against CommonLanguages will continue to work but now diverge
  from base.
- **Value-set change** from `…/ValueSet/languages` to `…/ValueSet/all-languages`
  also changes the canonical URL referenced by tooling; profile authors that
  hard-code the old canonical for slicing or invariants need to be aware.
- No element is added, removed, retyped, or recardinalized.

### Related Specifications

- **`TerminologyCapabilities.codeSystem.version.language`** — already updated
  by FHIR-46074; uses the same `all-languages` value set (with an additional
  `starter` binding to `languages`). InventoryItem will become consistent.
- Other resources that bind `*.language` elements to `all-languages` (e.g.,
  `Resource.language`, `CodeSystem.concept.designation.language`) are
  unaffected; this ticket does not change the underlying value sets.
- Downstream IGs that profile InventoryItem (e.g., supply-chain or pharmacy
  IGs) will inherit the relaxed binding and may need to revisit any local
  tightening.

### Related Tickets

- **FHIR-46074:** *Required binding strength to CommonLanguages should be
  changed to Preferred* (Terminology Infrastructure, status `Applied`,
  resolution `Persuasive with Modification`). Same reporter, same pattern,
  applied to `TerminologyCapabilities.codeSystem.version.language`. Sets the
  precedent for the change here. **Note divergence**: in 46074 the publisher
  kept `<strength>required</strength>` and added an `<additional purpose="starter">`
  pointing at `…/ValueSet/languages`. The 46128 resolution explicitly says
  `preferred` and does not mandate a starter binding — see Open Questions.

### Terminology Impact

- No new CodeSystem or ValueSet is created. The replacement value set
  `http://hl7.org/fhir/ValueSet/all-languages` already exists in
  `source/terminologies/valueset-all-languages.xml` (`<name value="AllLanguages"/>`).
- No `HL7/UTG` change required.

### Step-by-Step Tasks

#### HL7/fhir

1. **Relax `InventoryItem.name.language` binding to AllLanguages / preferred.**
   - File: `source/inventoryitem/structuredefinition-InventoryItem.xml`
   - Action: Inside the `<element id="InventoryItem.name.language">` block
     (starts line 166), in its `<binding>` (lines 177–186), change
     `<strength value="required"/>` → `<strength value="preferred"/>` and
     `<valueSet value="http://hl7.org/fhir/ValueSet/languages"/>` →
     `<valueSet value="http://hl7.org/fhir/ValueSet/all-languages"/>`.
   - Details: Leave the `bindingName` extension (`NameLanguage`), the
     `binding-definition` extension ("The languages for an item name"), and
     the `<description value="Name languages."/>` line unchanged. Edit only
     within `<differential>`; do not modify any `<snapshot>` content.

2. **Relax `InventoryItem.description.language` binding to AllLanguages / preferred.**
   - File: `source/inventoryitem/structuredefinition-InventoryItem.xml`
   - Action: Inside the `<element id="InventoryItem.description.language">`
     block (starts line 240), in its `<binding>` (lines 250–259), apply the
     same two edits: `<strength>required</strength>` → `preferred`, and
     `<valueSet>` `…/ValueSet/languages` → `…/ValueSet/all-languages`.
   - Details: Leave the `bindingName` extension (`ItemDescriptionLanguage`),
     the `binding-definition` extension, and the
     `<description value="Description languages."/>` line unchanged.

3. **(Optional) Mirror FHIR-46074's "starter" binding pattern.**
   - File: `source/inventoryitem/structuredefinition-InventoryItem.xml`
   - Action: If WG decides to follow the FHIR-46074 implementation exactly,
     after each updated `<valueSet value="…all-languages"/>` line add:
     `<additional><purpose value="starter"/><valueSet value="http://hl7.org/fhir/ValueSet/languages"/></additional>`
     (with a unique `<key>` per binding, e.g., `b-ii-name-1`, `b-ii-desc-1`).
   - Details: **Do not apply by default** — the FHIR-46128 resolution wording
     does not call for it. Only apply if the assignee confirms the WG wants
     the same dual binding shape as TerminologyCapabilities. Track via Open
     Questions below.
