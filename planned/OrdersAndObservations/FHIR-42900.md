## Resolution Summary

The committee found the request **Persuasive** (re-opened 2024-10-10 by motion 13-0-0
to ensure the full canonical-resource scope is implemented, not just the type label).
The agreed solution, captured in the original 2023-12-07 vote and reaffirmed by the
re-opening motion, is:

> "We agree that it should follow the definition pattern, and agree that this should
> also include the Canonical Resource (i.e., address all of the canonical resource
> elements required)."

Concretely this means **`DeviceDefinition` must be specialized from
`CanonicalResource`** (`baseDefinition =
http://hl7.org/fhir/StructureDefinition/CanonicalResource`) and **all canonical
resource elements** must be present on the resource (url, identifier-as-business,
version, versionAlgorithm[x], name, title, status, experimental, date, publisher,
contact, description, useContext, jurisdiction, purpose, copyright, copyrightLabel,
approvalDate, lastReviewDate, effectivePeriod). Vote impact was recorded as
**Non-compatible / Enhancement** for **R6**.

Companion ticket **FHIR-39389** ("DeviceDefinition should follow definitional
resource modelling", resolution *Not Persuasive with Modification*) tracks the
"definition pattern" portion; the assignee's 2025-07-10 / 2025-08-12 comments
defer joint completion-tracking to that ticket.

## Feature Proposal

### Problem Statement

`DeviceDefinition` is a definitional artifact (it describes the kind/model of a
device, distinct from the instance resource `Device`), and is used in many
specialized definitional flows (e.g., medicinal-product co-packaging, device
inventory catalogues). Prior to this work it was modelled as a plain
`DomainResource`: it had no canonical URL, no `version`, no `status`, no
publication metadata (`publisher`, `contact`, `useContext`, `jurisdiction`,
`approvalDate`, `lastReviewDate`, `effectivePeriod`, etc.), and could not be
referenced canonically the way `ObservationDefinition`, `ActivityDefinition`,
`PlanDefinition`, and `EventDefinition` already are. That gap meant catalogues of
device definitions could not be versioned, governed, or referenced by canonical
URL — inconsistent with the rest of the definitional-artifact family and
explicitly called out by the QA reviewer (Marti Velezis, 2024-05-04) as
"the Type was updated to indicate it is a Canonical Resource ... but I don't see
any of the other Canonical requirements applied."

### Proposed Change

Make `DeviceDefinition` a specialization of `CanonicalResource` and surface every
element that pattern requires, with the cardinalities, types, and bindings
established by `CanonicalResource` itself. Concrete edits:

1. **Re-base the StructureDefinition.** In
   `source/devicedefinition/structuredefinition-DeviceDefinition.xml`:
   - `<baseDefinition value="http://hl7.org/fhir/StructureDefinition/CanonicalResource"/>`
     (was `DomainResource`).
   - Build template extension `http://hl7.org/fhir/build/StructureDefinition/template`
     → `valueString = "CanonicalResource"`.
   - Add `<mapping><identity value="workflow"/><map value="Definition"/></mapping>`
     on the resource root and on `DeviceDefinition.url` (the workflow / definition
     pattern mapping).

2. **Add the canonical-resource element set** to the differential, in the
   conventional CanonicalResource order, all `0..1` unless noted, with definitions
   copied from the canonical-resource template:
   - `url : uri (0..1, isSummary)` — "Canonical identifier for this DeviceDefinition,
     represented as an absolute URI (globally unique)".
   - `identifier : Identifier (0..*, isSummary)` — business identifiers (already
     existed as a non-canonical element; keep, but align mapping `FiveWs.identifier`).
   - `version : string (0..1, isSummary)` — business version of the DeviceDefinition.
   - `versionAlgorithm[x] : string | Coding (0..1, isSummary)` — how versions compare.
   - `name : string (0..1, isSummary)` — computer-friendly name.
   - `title : string (0..1, isSummary)` — human-friendly title.
   - `status : code (1..1, isSummary, isModifier)` bound **required** to the
     publication-status value set (`http://hl7.org/fhir/ValueSet/publication-status`).
   - `experimental : boolean (0..1, isSummary)`.
   - `date : dateTime (0..1, isSummary)` — date last changed.
   - `publisher : string (0..1, isSummary)`.
   - `contact : ContactDetail (0..*, isSummary)`.
   - `description : markdown (0..1)`.
   - `useContext : UsageContext (0..*, isSummary)`.
   - `jurisdiction : CodeableConcept (0..*, isSummary)`.
   - `purpose : markdown (0..1)`.
   - `copyright : markdown (0..1)`.
   - `copyrightLabel : string (0..1)`.
   - `approvalDate : date (0..1)`.
   - `lastReviewDate : date (0..1)`.
   - `effectivePeriod : Period (0..1, isSummary)`.

3. **Preserve existing device-specific elements.** `partNumber`, `manufacturer[x]`
   (Reference(Organization) | string), `modelNumber`, etc., remain — they are
   device-domain elements layered on top of the canonical-resource base. The
   QA-cleanup must avoid the historical mistake of commenting them out while
   inserting canonical elements.

4. **Add canonical-resource search parameters** in
   `source/devicedefinition/bundle-DeviceDefinition-search-params.xml`:
   `DeviceDefinition-url`, `-version`, `-name`, `-title`, `-status`, `-date`,
   `-context`, `-context-quantity`, `-context-type`, `-context-type-quantity`,
   `-context-type-value`, `-jurisdiction`, `-publisher`, etc., mirroring the
   pattern used by `ObservationDefinition`'s search-param bundle. Mark
   normative-status to match the resource.

5. **Realign `device-definition-status` CodeSystem** in
   `source/devicedefinition/codesystem-device-definition-status.xml` so it is no
   longer used as the binding for the resource `status` element (that element now
   binds to `publication-status`); retain the historical operational-status codes
   under their existing identifier where still referenced, or re-purpose to the
   correct concept domain.

6. **Update the FiveWs mapping exceptions** in
   `source/devicedefinition/devicedefinition-fivews-mapping-exceptions.xml`:
   add `FiveWs.source` (publisher/contact now exist), drop `FiveWs.witness` and
   `FiveWs.done` since the new `effectivePeriod` / `publisher` map them.

7. **Register the resource category** in `publish.ini` so the publisher places
   `DeviceDefinition` under `Specialized.Definitional Artifacts` alongside
   `ActivityDefinition`, `EventDefinition`, `PlanDefinition`, `ObservationDefinition`:
   `DeviceDefinition = Specialized.Definitional Artifacts`.

8. **Refresh examples** so they round-trip against the new differential: at minimum
   `source/devicedefinition/devicedefinition-example.xml` must declare a `status`
   (now mandatory) and may declare a `url`. Cross-resource examples that embed
   contained `DeviceDefinition` instances
   (`source/medicinalproductdefinition/medicinalproductdefinition-example-co-packaged-liquid-and-syringe.xml`,
   its `-complete` sibling, and
   `source/packagedproductdefinition/packagedproductdefinition-example-co-packaged-liquid-and-syringe.xml`)
   need the same `status` addition.

9. **Add `DeviceDefinition` rows to the global `status-codes` table** in
   `source/status-codes.xml` so the publisher renders the correct status binding
   per element on cross-resource pages.

### Design Rationale

The committee's discussion (and the QA review by the assignee) makes clear that
"adding `CanonicalResource` to the type field" without surfacing the elements is
not enough — the value of the change is canonical referenceability, versioning,
and governance metadata, all of which live on the inherited elements. Following
the `ObservationDefinition` precedent keeps the OO definitional family internally
consistent and is the same pattern endorsed by the cross-WG canonical-resource
working group (see `canonicalresource.html`). The change is explicitly labelled
**R6-Structural** and was scored **Non-compatible / Enhancement** at vote, so it
is correctly landed in the R6 cycle rather than back-patched to R5 / R4B. Naming
the existing business `version` separately from the canonical `version` was
considered (Marti Velezis 2024-05-04) but rejected in favour of the canonical
semantic; existing instances using `version` as a "device version" must migrate
to a different element / extension at the next major release.

### Breaking Changes

- **`DeviceDefinition.status` becomes `1..1`** (mandatory) bound `required` to
  `publication-status`. Existing instances without a status are now invalid.
- **Resource base type changes** from `DomainResource` to `CanonicalResource`.
  Tools that compile against the resource hierarchy must recompile.
- **`DeviceDefinition.version` semantics change** from "device-business version"
  to canonical "business version of the artifact." Any system using `version`
  to model device-revision will need to migrate to a different element /
  extension.
- **New search parameters** are additive (not breaking on their own), but
  `status` search now exists and may surprise servers that previously had no
  status concept on this resource.
- All breaking changes are scoped to **R6** (the ticket is labelled
  `R6-Structural` and the vote recorded "Impact: Non-compatible").

### Related Specifications

- **CanonicalResource** (`http://hl7.org/fhir/StructureDefinition/CanonicalResource`)
  — new base type; any change to the canonical-resource template propagates here.
- **ObservationDefinition** — the explicit precedent in the original ticket;
  pattern conformance should be checked against it.
- **ActivityDefinition / EventDefinition / PlanDefinition** — the
  `Specialized.Definitional Artifacts` family `DeviceDefinition` now joins in

- **Device** (instance resource) — its `definition` element references
  `DeviceDefinition`; no element change is required, but documentation should
  acknowledge that `DeviceDefinition` references can now be canonical URLs.
- **MedicinalProductDefinition / PackagedProductDefinition** — embed contained
  `DeviceDefinition` instances; their examples must be revalidated.

  no UTG change required).
- **`device-definition-status`** CodeSystem

  it for backward documentation. This decision must be recorded in

- No `HL7/UTG` repository change is required.
