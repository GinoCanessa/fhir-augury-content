## Resolution Summary

**Resolution: Persuasive.** Add a new `SearchParameter` for the
`BiologicallyDerivedProduct` resource targeting
`BiologicallyDerivedProduct.collection.sourceOrganization`, with the
following resolution-mandated characteristics:

- **Name (code):** `source-organization`
- **Type:** `reference`
- **Description:** "The organization that facilitated the collection of the product"
- **Expression:** `BiologicallyDerivedProduct.collection.sourceOrganization`

The reporter's later comment (2026-04-10) flags ambiguity: the analogous
parameter for `BiologicallyDerivedProduct.collection.sourcePatient` is
called simply `patient`, and the as-implemented PR uses `organization`
rather than `source-organization`. The resolution text remains the
authoritative spec for this plan; any deviation must be confirmed by the
work group (see Open Questions).

## Feature Proposal

### Problem Statement

`BiologicallyDerivedProduct` (BDP) records the organization that
facilitated product collection in
`BiologicallyDerivedProduct.collection.sourceOrganization` (a
`Reference(Organization)`), but until R6 there was no `SearchParameter`
exposing that element. Servers had no standard way to answer queries
like "find all BDP records collected by organization X" — a common
operational need for biorepositories, blood banks, and tissue
establishments coordinating across collection sites. Sibling collection
elements such as `collection.sourcePatient` (search parameter `patient`)
and `collection.collector` (search parameter `collector`) are already
searchable, leaving `sourceOrganization` as a noticeable gap.

### Proposed Change

Add a single new `SearchParameter` entry to the BDP search-params bundle:

- **id:** `BiologicallyDerivedProduct-collection-sourceOrganization`
- **url:** `http://hl7.org/fhir/build/SearchParameter/BiologicallyDerivedProduct-collection-sourceOrganization`
- **code:** `source-organization` *(per resolution text — currently
  implemented as `organization`; see Open Questions)*
- **type:** `reference`
- **base:** `BiologicallyDerivedProduct`
- **expression:** `BiologicallyDerivedProduct.collection.sourceOrganization`
- **path build extension:** `BiologicallyDerivedProduct.collection.sourceOrganization`
- **standards-status extension:** `normative`
- **status:** `active`
- **processingMode:** `normal`
- **description:** "The organization that facilitated the collection of the product"

Add a release-note bullet pointing at FHIR-56179 in the BDP introduction
page so the change is surfaced under the R6 release notes block.

No StructureDefinition change is required — the underlying element
`BiologicallyDerivedProduct.collection.sourceOrganization` already
exists as `Reference(Organization)`.

### Design Rationale

- BDP already exposes parallel collection elements via reference search
  parameters (`patient` → `collection.sourcePatient`, `collector` →
  `collection.collector`). Adding `source-organization` /
  `organization` for `collection.sourceOrganization` completes the
  pattern and uses the same `processingMode=normal`,
  `standards-status=normative`, `status=active` conventions already
  used by every other entry in the BDP search-param bundle.
- Reference type is the only sensible choice — the target element is a
  `Reference(Organization)`.
- The PR author chose `code=organization` to match the precedent set by
  `code=patient` (for `collection.sourcePatient`); the resolution text
  chose `source-organization` to make the source-vs-other-organization
  distinction explicit. Both have FHIR precedent. Bringing this back to
  the work group resolves a one-bit naming question rather than a
  design question.

   ticket so the historical record matches the implemented spec.
