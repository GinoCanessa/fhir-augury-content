## Resolution Summary

The ticket flagged the `request-orderdetail-parameter-code` CodeSystem (and the
ValueSet of the same name, bound to `DeviceRequest.parameter.code`) as a
candidate for migration to THO. The Orders & Observations work group decided
**not** to migrate to THO at this time, but to mark the binding as
experimental so consumers understand its maturity.

Resolution wording (from the ticket): "We will make a change to:
**request-orderdetail-parameter-code**: change the **ValueSet** to
experimental. The **code system is already experimental and does not need
revised.**"

Concretely the only change required is to flip
`ValueSet/request-orderdetail-parameter-code.experimental` from `false` to
`true`. The CodeSystem is already `experimental="true"` and is left alone.

## Feature Proposal

### Problem Statement

`ValueSet/request-orderdetail-parameter-code` is bound to
`DeviceRequest.parameter.code` (and is referenced from ServiceRequest
artifacts) but its experimental status disagrees with the underlying
CodeSystem of the same name:

- `CodeSystem/request-orderdetail-parameter-code` declares
  `experimental="true"` (informative, FMM 0).
- `ValueSet/request-orderdetail-parameter-code` declares
  `experimental="false"` despite drawing exclusively from that experimental
  CodeSystem and being marked informative / FMM 0 itself.

This mismatch is misleading: implementers reading the ValueSet metadata see a
non-experimental binding even though the only codes available come from an
experimental, low-maturity CodeSystem. The Orders & Observations work group
also reviewed whether to migrate this content to the Terminology
Harmonization Organization (THO), but concluded the existing definition is
unlikely to need cross-product re-use in the near term and chose not to
migrate. Marking the ValueSet experimental is the minimal change that
correctly communicates current maturity without committing to a THO move.

### Proposed Change

Edit `source/servicerequest/valueset-request-orderdetail-parameter-code.xml`:

- Change the element on line 19 from
  `<experimental value="false"/>` to
  `<experimental value="true"/>`.

No other changes:

- The CodeSystem (`source/servicerequest/codesystem-request-orderdetail-parameter-code.xml`)
  already has `<experimental value="true"/>` on line 19 — leave it as-is.
- No element/cardinality/binding changes on DeviceRequest or ServiceRequest.
  Both resources continue to bind their respective `parameter.code` element to
  `http://hl7.org/fhir/ValueSet/request-orderdetail-parameter-code`; only the
  ValueSet's `experimental` flag changes.
- No THO migration is performed under this ticket.

Resulting ValueSet metadata (after change):

```xml
<ValueSet xmlns="http://hl7.org/fhir">
  <id value="request-orderdetail-parameter-code"/>
  ...
  <name value="RequestOrderDetailParameterCode"/>
  <title value="Request OrderDetail Parameter Code"/>
  <status value="active"/>
  <experimental value="true"/>   <!-- was: false -->
  ...
</ValueSet>
```

### Design Rationale

- **Internal consistency.** A ValueSet that composes only an experimental
  CodeSystem should itself be flagged experimental; otherwise downstream
  tooling and implementers receive contradictory maturity signals.
- **Minimal disruption.** The work group considered THO migration (the
  ticket carries the `THO-candidate` label) but the resolution explicitly
  declines migration. Flipping a single boolean preserves URLs, IDs, and
  bindings, requiring no co-ordinated change in DeviceRequest, ServiceRequest,
  or any downstream IG.
- **Signals review intent.** Marking the ValueSet experimental keeps the
  door open for further work (including a future THO move or content rework)
  without locking implementers into the current code list.
