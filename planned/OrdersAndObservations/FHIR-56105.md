## Resolution Summary

**Resolution:** Persuasive. **Resolution description:** *"This has been addressed in R6."*

DocumentReference must no longer be presented as a Diagnostics resource. It is to appear only as a Foundation/Documents resource throughout the FHIR specification. The single applied-vote comment (JIRAUSER19407, 2026-03-31) adds a follow-up requirement: *"Introduction text needs to be updated to reference this ticket"* — i.e., the Diagnostics Module introduction (and the R6 release note) must explicitly call out that DocumentReference has been removed from this module, citing FHIR-56105.

Concretely the resolution requires:

1. Removing DocumentReference from every place it is enumerated as a Diagnostics resource (module index, module resource table, top-level "modules" landing fragment, and the cross-module hierarchy file that drives the publisher).
2. Confirming DocumentReference remains correctly listed under the Foundation/Documents grouping it already belongs to (no add — it is already there).
3. Adding a short narrative note in the Diagnostics Module introduction explaining the change and referencing FHIR-56105, plus a corresponding R6 release-note entry (label `R6-Release-note`).

## Feature Proposal

### Problem Statement

DocumentReference is currently classified as one of the Diagnostics-module resources in several places in the FHIR R6 build:

- `source/diagnostics-module.html` — listed in the module Index (line 46) and in the "resources covered by this module" table (`[%res-item DocumentReference%]`, line 99), and the maturity section (line 389) treats DocumentReference's FMM as a Diagnostics-module fact.
- `source/modules-fragment.html` — appears inside the `<div id="diagnostics">` content paragraph (line 151) on the modules landing grid.
- `source/hierarchy.xml` — duplicated under both `Foundation/Documents` (line 134) **and** `Diagnostics` (line 307), so the publisher's TOC / breadcrumbs treat DocumentReference as a Diagnostics resource.

DocumentReference, however, is a generic reference to any kind of externally-authored document — clinical summaries, care-provisioning documents, financial documents, CDAs, PDFs, etc. — and is not specific to diagnostic content. It is already correctly grouped with `Composition` under Foundation/Documents (`source/resourcelist.html` line 73‑74, `source/foundation-module.html` lines 50‑51 and 103‑104, and `source/documents.html` which describes both "FHIR documents" and the DocumentReference resource as the Documents framework). The Diagnostics-module classification therefore mis-represents the resource's scope and creates two incompatible "homes" in the spec navigation.

### Proposed Change

Remove DocumentReference from the Diagnostics module wherever it is currently enumerated, leave its existing Foundation/Documents listings untouched, and document the move.

Specifically:

1. **`source/hierarchy.xml`** — delete the line `<page type="resource" resource="DocumentReference"/>` from the `<page title="Diagnostics" filename="diagnostics-module.html" …>` block (currently line 307, between `Observation` and `DiagnosticReport`). Leave the corresponding entry inside the `Foundation/Documents` block (line 134) intact.

2. **`source/diagnostics-module.html`** —
   - Remove `<li><a href="documentreference.html">DocumentReference</a></li>` from the module Index list (line 46), so the first column lists only `Observation` and `DiagnosticReport`.
   - Remove the `[%res-item DocumentReference%]` row from the "Resources involved in Diagnostics" table (line 99), preserving the surrounding rows (`Observation`, `DiagnosticReport`, `ServiceRequest`, `ImagingSelection`, `ImagingStudy`, `Specimen`, `BodyStructure`).
   - Remove the bullet that asserts DocumentReference's maturity inside this module (line 389: *"The DocumentReference resource has been tested and used in production tooling and has reached maturity level FMM=4."*). The bullet belongs in the Foundation module (where DocumentReference's maturity is already representable); the Foundation page does not need a new bullet because that page does not currently call out per-resource FMM in the same way — but the Diagnostics-module sentence must not remain.
   - Add a short note in the Diagnostics Module introduction (immediately under `<h3>Introduction</h3>`, around line 22) that explicitly references FHIR-56105 and explains that DocumentReference is no longer part of this module. Suggested wording (HTML, in line with existing tone):
     > `<p>Note: as of this release the <a href="documentreference.html">DocumentReference</a> resource is no longer part of the Diagnostics Module. It is now grouped solely under the <a href="foundation-module.html">Foundation Module</a> (Documents) because it is used to reference any kind of document (clinical, administrative, financial, etc.), not just diagnostic content. See <a href="https://jira.hl7.org/browse/FHIR-56105">FHIR-56105</a>.</p>`

3. **`source/modules-fragment.html`** — remove the line `<a href="documentreference.html">DocumentReference</a>,` from the `<div id="diagnostics">` `<p class="content">` block (line 151), so the listed Diagnostics resources read `Observation, DiagnosticReport, Specimen, ImagingStudy, etc.` Verify the trailing comma/whitespace stays consistent with the surrounding markup.

4. **R6 release note** (`R6-Release-note` label) — add a one-line entry under the Foundation/Documents (or Diagnostics) section of `source/history.html` (and any per-module release-note fragment used by the OO work group's R6 notes) reading approximately:
   > `<li>FHIR-56105: DocumentReference has been removed from the Diagnostics Module and is now listed only under Foundation/Documents.</li>`
   The exact list it goes into should match the convention used by other R6 release notes already present in `source/history.html` (the file currently mentions DocumentReference at line 128 in a different context, which is unrelated and must not be edited).

### Design Rationale

- DocumentReference is content-agnostic: it can wrap any kind of document, including non-clinical ones. The Foundation/Documents grouping (alongside `Composition`) already exists to describe FHIR's general "document" framework, and `source/documents.html` is the conceptual page for that framework. Listing DocumentReference under both Diagnostics and Foundation/Documents creates duplicate categorisation that is inconsistent with how every other resource is presented.
- The cleanest solution is removal from Diagnostics rather than relocation, because the Foundation/Documents listings are already correct and complete (`source/resourcelist.html` `Foundation.Documents` set, `source/foundation-module.html` Documents bullet list, `source/hierarchy.xml` Foundation/Documents block). No new home needs to be invented.
- The publisher's hierarchy is driven by `source/hierarchy.xml`; `diagnostics-module.html` and `modules-fragment.html` are the human-authored narrative views. All three must be updated together for the navigation, the module page, and the homepage modules grid to agree.
- The introductory note + R6 release-note combination is the standard FHIR pattern for breaking-presentation-changes and is what the applied-vote comment is asking for. It also avoids dangling expectations from R5 implementers who navigate to the Diagnostics Module looking for DocumentReference.

### Breaking Changes

No schema-level breaking changes. The DocumentReference StructureDefinition, its elements, cardinalities, types, bindings, search parameters, examples, and operations are entirely unchanged.

There is, however, a **navigational/categorisation change** that may affect documentation tooling, downstream IGs, or readers who deep-link into the Diagnostics module expecting DocumentReference to be enumerated there:

- Anyone whose tooling parses `source/hierarchy.xml` to derive per-module resource sets will see DocumentReference disappear from the Diagnostics module set. If their tooling assumes a resource is in *exactly one* module, the new state aligns with that assumption (DocumentReference is now only in Foundation/Documents); if their tooling assumes the R5 layout, it will need to be re-baselined.
- Inbound deep-links of the form `diagnostics-module.html#…` aimed at DocumentReference will no longer find the resource on the page. (No `<a name="…">` anchors specific to DocumentReference exist in `diagnostics-module.html`, so there is no specific anchor to redirect — but a brief note in the introduction (Step 4 of Tasks) softens the change.)
