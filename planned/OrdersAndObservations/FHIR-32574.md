## Feature Proposal

### Problem Statement

`Device` is a candidate to go normative (label `R6-FMM`). To be eligible, HL7 requires a Level 3 QA assessment against the standard FMM criteria (see https://confluence.hl7.org/display/FHIR/Conformance+QA+Criteria). At the time the ticket was opened (2021), no Level 3 QA had been performed on `Device`; the most recent FMM on the resource is `2` (`source/device/structuredefinition-Device.xml:15` — `<valueInteger value="2"/>`, and `source/fhir.ini:824` — `device=2`). The 2025 QA assessment (linked from Jira and attached as `OO_FMM_QA_DEV.docx`) identified a concrete, bounded list of informational findings that have to be resolved or suppressed before the maturity level can be raised.

### Proposed Change

The change is FMM-promotion plumbing plus targeted QA clean-up in `HL7/fhir`, confined to the `source/device/` authoring folder and the top-level `source/fhir.ini`:

1. **Clear or formally suppress each QA informational message** listed in the resolution summary. For each item:
   - **Search-parameter expressions with no example coverage** (`Device.biological-source-event`, `Device.definition`, `Device.location`, `Device.parent`): add at least one Device example under `source/device/` that populates the corresponding element so the QA expression resolves, or add the specific QA message to `suppressed-messages.txt` with a justification. Preferred path: update examples, because the search parameters are real and should be testable.
   - **`udi-carrier-hrf` style-guide violation**: rename the search-parameter code/id to a style-guide-compliant form (e.g., `udi-carrier-hrf` → `udi-carrier-hrf` is the offender; replace with `udiCarrier-hrf`-style token per the FHIR search-parameter naming guidance, or — if the name is already published and renaming would be breaking — suppress the message with a note that the name is grandfathered).
   - **Child/parent naming conflicts** (`Device.deviceVersion`, `Device.udiCarrier.carrierAIDC`, `Device.udiCarrier.carrierHRF`): these are long-standing names. The resolution does not authorise a rename (that would be a breaking change to `Device` content and belongs to separate tickets). The practical action is therefore to **suppress** each message in `suppressed-messages.txt` with a short justification (historic naming, rename out of scope for this ticket).
   - **Mapping-pattern exceptions**: ensure `source/device/device-fivews-mapping-exceptions.xml` and `source/device/device-participant-mapping-exceptions.xml` list a non-`Unknown` `reason` for every `Device` deviation from the `fivews` / `participant` pattern. Where a deviation is genuinely unexplained, open a follow-up ticket (see "Related Tickets" below) rather than inventing a reason.
2. **File follow-up tickets** for any finding that cannot be cleared in this ticket without making a substantive design change (e.g., renames of `deviceVersion`, `carrierAIDC`, `carrierHRF`; rename of `udi-carrier-hrf` if it is deemed breaking). This is explicitly called out in the QA comment: *"more tickets need to be created for each type."*
3. **Bump the FMM level** once the QA list is green:
   - `source/device/structuredefinition-Device.xml` — change the `structuredefinition-fmm` extension (currently at line 14–16) from `<valueInteger value="2"/>` to `<valueInteger value="3"/>`.
   - `source/fhir.ini` — change `device=2` on line 824 (the FMM section) to `device=3`. The other `device=…` entries in `fhir.ini` (work-group `oo`, security-category `business`, abbreviation `dev`, approval `approved`, normative status) are unrelated and must not be changed.
4. **Attach the completed QA template** (`OO_FMM_QA_DEV.docx`) to the ticket/Confluence record, if not already present. No change inside `HL7/fhir` is required for this step.

### Design Rationale

- The motion is already agreed ("Persuasive"), so the design question is purely *how* to perform the FMM-3 bump, not *whether*.
- Rewriting long-standing `Device` element names (`deviceVersion`, `carrierAIDC`, `carrierHRF`) is a breaking change that would ripple through every implementer and downstream IG. The QA assessor explicitly acknowledges follow-up tickets are expected; suppression with justification is the conservative, scope-matching choice for this ticket.
- Adding example values is preferred over suppression for "no found values in examples" because the underlying search parameters are real and useful; suppressing them would be hiding a gap in test coverage, not a substantive modelling problem.
- `fhir.ini` is the sole authoritative source of per-resource FMM levels in this repo (see `source/fhir.ini` lines 823–829); the `structuredefinition-fmm` extension on the individual StructureDefinition is the mirrored value the publisher renders. Both must be updated together or the publisher QA surfaces a new mismatch.

   - File: Choose an existing `source/device/device-example*.xml` that already references a `DeviceDefinition` (or add one), populating `Device.definition` as a `Reference(DeviceDefinition)`.
   - Action: Ensure the Device instance sets `Device.definition` to a resolvable `Reference`.
