## Resolution Summary

**Resolution:** Persuasive.

Resolution description (verbatim from the ticket):

> Device was removed in error. We will add back a reference to Device to
> container.device, processing.device, and collection.device to the choice
> element. This will be an additional choice of deviceReference with a
> datatype of Reference(Device).

In concrete terms, the `device[x]` choice on each of `Specimen.collection`,
`Specimen.processing`, and `Specimen.container` must include `Reference(Device)`
as one of its allowed types (alongside the existing `CodeableConcept` and
`canonical(DeviceDefinition)` choices). The implicit choice element name for
the new type is `deviceReference`.

## Feature Proposal

### Problem Statement

FHIR-39389 (DeviceDefinition should follow definitional resource modelling)
restructured the `device[x]` choice elements on `Specimen` so that they offer
`CodeableConcept` and `canonical(DeviceDefinition)` only. In doing so it
removed the existing `Reference(Device)` choice that previously let an
implementer point at a specific instance of a `Device` (the actual catheter,
processor, or container instance, as opposed to its definition). The reporter
(Gregory Boege) flagged this as an unintended regression: there are real
workflows — e.g., associating a collection with the specific catheter used,
or a container with the specific tube/holder instance — where a definitional
reference is not sufficient and an instance reference is required.

The change is a technical correction restoring the previously-supported
capability without otherwise altering the choice's semantics.

### Proposed Change

In `source/specimen/structuredefinition-Specimen.xml`, add an additional
`<type>` entry for `Reference` with `targetProfile` =
`http://hl7.org/fhir/StructureDefinition/Device` to each of the following
three choice elements:

1. `Specimen.collection.device[x]` (currently at `min=0 max=1`, types
   `CodeableConcept` + `canonical(DeviceDefinition)`).
2. `Specimen.processing.device[x]` (currently at `min=0 max=1`, types
   `CodeableConcept` + `canonical(DeviceDefinition)`).
3. `Specimen.container.device[x]` (currently at `min=1 max=1`, types
   `CodeableConcept` + `canonical(DeviceDefinition)`).

After the change, every one of those three elements MUST offer all three of:

- `CodeableConcept` (existing — for inline coded values bound to the relevant
  example ValueSet such as `containerdevice-code`, `processingdevice-code`,
  or `specimen-container-type`);
- `Reference` with `targetProfile` `Device` (restored — for an instance
  reference, exposed as `deviceReference`);
- `canonical` with `targetProfile` `DeviceDefinition` (existing — for a
  definitional reference, exposed as `deviceCanonical`).

Cardinalities, bindings, mappings, and short/definition text on the three
elements remain unchanged.

Concrete XML fragment to insert into each element (placement: between the
existing `<type><code value="Reference"/>...</type>` (which currently doesn't
exist) — i.e. add a fresh block adjacent to the existing `CodeableConcept`
and `canonical` blocks). For example, in `Specimen.container.device[x]` the
final element should look like:

```xml
<element id="Specimen.container.device[x]">
  <path value="Specimen.container.device[x]"/>
  <short value="Device resource for the container"/>
  <definition value="The device resource for the the container holding the specimen. If the container is in a holder then the referenced device will point to a parent device."/>
  <min value="1"/>
  <max value="1"/>
  <type>
    <code value="CodeableConcept"/>
  </type>
  <type>
    <code value="Reference"/>
    <targetProfile value="http://hl7.org/fhir/StructureDefinition/Device"/>
  </type>
  <type>
    <code value="canonical"/>
    <targetProfile value="http://hl7.org/fhir/StructureDefinition/DeviceDefinition"/>
  </type>
  <binding> ...unchanged... </binding>
  <mapping> ...unchanged... </mapping>
</element>
```

### Design Rationale

- **Restores prior behavior.** Implementers were already using
  `Reference(Device)` for these three elements before FHIR-39389 was applied.
  Removing it broke a real, in-the-wild use case (an instance pointer to the
  specific physical device).
- **Pure additive type expansion.** Adding a third type to an existing
  choice does not alter the meaning of the element, does not change
  cardinality, does not change the binding, and does not invalidate any
  existing instance — `CodeableConcept`-valued and
  `canonical(DeviceDefinition)`-valued instances remain valid.
- **Symmetric across the three sites.** `collection.device[x]`,
  `processing.device[x]`, and `container.device[x]` all model the same
  conceptual question ("which device played this role?") and have always
  shared the same type list. The fix preserves that symmetry rather than
  treating one of the three differently.
- **Definitional vs. instance distinction is preserved.** The resolution
  intentionally keeps `canonical(DeviceDefinition)` (the FHIR-39389 outcome)
  and adds `Reference(Device)` alongside it — `deviceCanonical` for "kind of
  device" and `deviceReference` for "this specific device".

### Breaking Changes

No breaking changes. This is a pure additive type expansion on existing
choice elements:

- Cardinality is unchanged on all three elements.
- The previously-allowed types (`CodeableConcept`,
  `canonical(DeviceDefinition)`) remain allowed, so all instances valid
  before the change remain valid after.
- Bindings and binding strengths are unchanged.

The only consumer-facing implication is that validators and code
generators will start accepting/producing a `deviceReference` choice
where they previously rejected it. That is the intended restoration of
behavior, not a regression.

## Implementation Plan

### Prerequisites

None. The `Device` resource and its canonical URL
`http://hl7.org/fhir/StructureDefinition/Device` already exist in
`HL7/fhir` and are the same resource the elements referenced before
FHIR-39389. No upstream artifact has to land first.

### Step-by-Step Tasks

#### HL7/fhir

1. **Restore `Reference(Device)` on `Specimen.collection.device[x]`.**
   - File: `source/specimen/structuredefinition-Specimen.xml`
   - Action: Inside the `<element id="Specimen.collection.device[x]">`
     block (begins around line 530 in the current clone), add a third
     `<type>` child after the existing `CodeableConcept` `<type>` and
     before (or after) the existing `canonical(DeviceDefinition)`
     `<type>`:
     ```xml
     <type>
       <code value="Reference"/>
       <targetProfile value="http://hl7.org/fhir/StructureDefinition/Device"/>
     </type>
     ```
   - Details: Do not modify `<min>`, `<max>`, `<short>`, `<definition>`,
     `<binding>`, or `<mapping>`. Only touch `<differential>` content.
     The publisher regenerates `<snapshot>`.

2. **Restore `Reference(Device)` on `Specimen.processing.device[x]`.**
   - File: `source/specimen/structuredefinition-Specimen.xml`
   - Action: Inside the `<element id="Specimen.processing.device[x]">`
     block (begins around line 721 in the current clone), add the same
     `<type>` block as in task 1.
   - Details: Same constraints as task 1 (differential-only, types
     symmetric across the three sites).

3. **Restore `Reference(Device)` on `Specimen.container.device[x]`.**
   - File: `source/specimen/structuredefinition-Specimen.xml`
   - Action: Inside the `<element id="Specimen.container.device[x]">`
     block (begins around line 808 in the current clone), add the same
     `<type>` block as in task 1.
   - Details: Same constraints as task 1. Note this element is
     `min=1 max=1`; the additive type does not change that.

4. **(Optional) Add or update an example to exercise `deviceReference`.**
   - File: e.g., `source/specimen/specimen-example.xml` or one of the

   - Action: Add a `containerDeviceReference` (or equivalent on
     `collection`/`processing`) pointing at a `Device` instance to give
     the rendered Specimen page a worked example of the restored choice.
   - Details: Optional — the resolution does not require it. Skip if
     the work group has not asked for it. Do not edit

5. **Run the local publisher and review the regenerated outputs.**
