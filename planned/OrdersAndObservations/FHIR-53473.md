## Resolution Summary

Resolution: **Persuasive**.

The ticket directs five updates to the BodyStructure SearchParameter bundle so
that the published codes follow FHIR's `kebab-case` convention and the
descriptions accurately convey the parameter's purpose:

1. **`excluded_structure` → `excluded-structure`** (token, expression
   `BodyStructure.excludedStructure.structure`). New description: *"The
   excluded body structure(s)"* (replacing *"Body site excludedStructure
   structure"*).
2. **`identifier`** (token, expression `BodyStructure.identifier`). New
   description: *"The unique id for the BodyStructure"* (replacing
   *"Bodystructure identifier"*). In Common — aligns with the
   `clinical-identifier` common search parameter (58 resources).
3. **`included_structure` → `included-structure`** (token, expression
   `BodyStructure.includedStructure.structure`). New description: *"The
   included body structure(s)"* (replacing *"Body site includedStructure
   structure"*).
4. **`morphology`** (token, expression
   `BodyStructure.includedStructure.morphology`). New description: *"The kind
   of structure"* (replacing *"Kind of Structure"*).
5. **`patient`** (reference → Patient, expression `BodyStructure.patient`).
   New description: *"The patient referenced for the identified body
   structure"* (replacing *"Who this is about"*). In Common — aligns with the
   `clinical-patient` common search parameter (60 resources).

## Feature Proposal

### Problem Statement

Two of the BodyStructure-specific search parameters (`included_structure` and
`excluded_structure`) use underscores in their codes, which violates the FHIR
naming convention that search-parameter codes use lowercase hyphenated
identifiers (kebab-case). In addition, several existing descriptions are
either ungrammatical (`"Body site includedStructure structure"`,
`"Bodystructure identifier"`), too generic (`"Who this is about"`,
`"Kind of Structure"`), or non-descriptive of the parameter's specific role
on BodyStructure. The Orders & Observations work group reviewed these in QA
(see the linked O&O Committers Zulip thread) and agreed the codes and
descriptions need to be normalised.

### Proposed Change

Edit `source/bodystructure/bundle-BodyStructure-search-params.xml` so the
five `SearchParameter` entries reflect the renamed codes and updated
descriptions. The expression, type, base resource, and processingMode are
unchanged.

| Current `code` | New `code` | Current `description` | New `description` | `expression` (unchanged) | `type` |
|---|---|---|---|---|---|
| `excluded_structure` | `excluded-structure` | Body site excludedStructure structure | The excluded body structure(s) | `BodyStructure.excludedStructure.structure` | token |
| `identifier` | `identifier` | Bodystructure identifier | The unique id for the BodyStructure | `BodyStructure.identifier` | token |
| `included_structure` | `included-structure` | Body site includedStructure structure | The included body structure(s) | `BodyStructure.includedStructure.structure` | token |
| `morphology` | `morphology` | Kind of Structure | The kind of structure | `BodyStructure.includedStructure.morphology` | token |
| `patient` | `patient` | Who this is about | The patient referenced for the identified body structure | `BodyStructure.patient` | reference |

For consistency with the `code` rename, also update the matching SearchParameter
`id` and `url` so that the resource identity tracks the published code (the
existing pattern in this bundle is `BodyStructure-<code>` for IDs and
`http://hl7.org/fhir/build/SearchParameter/BodyStructure-<code>` for URLs):

- `id` `BodyStructure-includedStructure-structure` → `BodyStructure-included-structure`; `url` likewise.
- `id` `BodyStructure-excludedStructure-structure` → `BodyStructure-excluded-structure`; `url` likewise.
- `id` `BodyStructure-identifier`, `BodyStructure-morphology`, `BodyStructure-patient`: unchanged.

(The id rename is flagged again in **Open Questions** in case the work group
prefers to keep historical IDs for stability — but kebab-cased IDs are the
norm in `bundle-*-search-params.xml` files for other resources.)

### Design Rationale

- FHIR search parameter codes are conventionally lowercase kebab-case
  (e.g., `birth-date`, `family-name`, `body-site`); underscores are
  non-conformant. The rename brings BodyStructure in line with every other
  resource in the spec.
- The new descriptions mirror the wording style used by other clinical
  resources (e.g., the common `patient` parameter description is contextualised
  for the resource — `"The patient referenced for the identified body
  structure"` parallels how `Condition.patient` and others phrase it).
- `identifier` and `patient` remain "in common" with the
  `clinical-identifier` (58 resources) and `clinical-patient` (60 resources)
  registry entries respectively — only the per-resource description changes,
  not the common parameter.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Owns the BodyStructure resource and its SearchParameter bundle.
- **Affected Files:**
  - `source/bodystructure/bundle-BodyStructure-search-params.xml` — rename
    `included_structure` → `included-structure` and `excluded_structure` →
    `excluded-structure` (codes, IDs, URLs); update the `description` value
    on all five entries (`identifier`, `included-structure`,
    `excluded-structure`, `morphology`, `patient`).
- **Change Scope:** Minor (single bundle file, mechanical rename + five
  description text edits). However the search-parameter `code` rename has
  downstream visibility (see Breaking Changes).

### Breaking Changes

Renaming a published SearchParameter `code` is a **breaking change** for any
client or server that relies on the previous URL form
`GET [base]/BodyStructure?included_structure=…` or
`?excluded_structure=…`. Servers will need to advertise the new codes in
their `CapabilityStatement.rest.resource.searchParam` and clients must update
their query construction.

The other three changes (`identifier`, `morphology`, `patient`) only touch
human-readable `description` text — no behaviour, cardinality, or expression
change — and are **not** breaking.

### Related Specifications

- `searchparameter-registry.html` (`clinical-identifier`, `clinical-patient`)
  — BodyStructure's `identifier` and `patient` parameters appear in the
  registry's "in common" rollups. These are aggregated by the publisher from
  per-resource bundles, so editing the bundle is sufficient; no manual
  registry edit should be needed, but the published page should be reviewed
  after a publisher run.
- `bodystructure-search.html` (anchors `#BodyStructure-includedstructure`,
  `#BodyStructure-excludedstructure`, `#clinical-identifier`,
  `#BodyStructure-morphology`, `#clinical-patient`) — generated from the
  bundle; anchors derived from the renamed codes will change.

### Step-by-Step Tasks

#### HL7/fhir

1. **Rename `included_structure` SearchParameter to `included-structure`**
   - File: `source/bodystructure/bundle-BodyStructure-search-params.xml`
   - Action: In the `SearchParameter` whose `expression` is
     `BodyStructure.includedStructure.structure`:
     - `<id value="BodyStructure-includedStructure-structure"/>` →
       `<id value="BodyStructure-included-structure"/>`
     - `<url value="http://hl7.org/fhir/build/SearchParameter/BodyStructure-includedStructure-structure"/>` →
       `<url value="http://hl7.org/fhir/build/SearchParameter/BodyStructure-included-structure"/>`
     - `<code value="included_structure"/>` →
       `<code value="included-structure"/>`
     - `<description value="Body site includedStructure structure"/>` →
       `<description value="The included body structure(s)"/>`
   - Details: Leave `type=token`, `expression`, `processingMode`, and the
     `standards-status` / `path` extensions unchanged.

2. **Rename `excluded_structure` SearchParameter to `excluded-structure`**
   - File: `source/bodystructure/bundle-BodyStructure-search-params.xml`
   - Action: In the `SearchParameter` whose `expression` is
     `BodyStructure.excludedStructure.structure`:
     - `<id value="BodyStructure-excludedStructure-structure"/>` →
       `<id value="BodyStructure-excluded-structure"/>`
     - `<url value="http://hl7.org/fhir/build/SearchParameter/BodyStructure-excludedStructure-structure"/>` →

       `<code value="excluded-structure"/>`
     - `<description value="Body site excludedStructure structure"/>` →
       `<description value="The excluded body structure(s)"/>`
   - Details: Same constraints as task 1.

3. **Update `identifier` description**
   - File: `source/bodystructure/bundle-BodyStructure-search-params.xml`

   - Details: `id`, `url`, `code`, `expression`, and `type` are unchanged.

### Open Questions

- **Should the SearchParameter `id` and `url` be renamed alongside the
  `code`?** This plan assumes yes, to keep the bundle's
  `BodyStructure-<code>` naming pattern consistent. If O&O prefers to retain
  the historical `id`/`url` for resource-identity stability, only the
  `<code>` and `<description>` lines should be touched in tasks 1 and 2.
  Confirm with the assignee (Jose Costa-Teixeira) before merging.
- **Where exactly is the "In Common" wiring for `identifier` and `patient`
  produced?** The publisher aggregates per-resource bundles into
  `searchparameter-registry.html`; if there is a hand-authored mapping that
  pins the older description, it would also need to be touched. The
  briefing's repo-sweep guidance covers this in task 6.
- **Are there downstream IGs (e.g., US Core, IPS, IPA) that pin the old
  `included_structure` / `excluded_structure` codes?** The briefing notes the
  general FhirCore touch point to "downstream IGs under `IgRepositories`"
  but flags none active for this ticket. A quick check of the major IG

  rename.
