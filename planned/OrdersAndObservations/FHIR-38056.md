## Resolution Summary

Resolution: **Persuasive with Modification**.

Applied motion (from the ticket's resolution description):

- **Include (new) value sets for risk class** applicable outside the US (OUS) — e.g., EU MDR risk classes (I, IIa, IIb, III) and comparable regional risk classifications; keep FDA/US classes in scope as example content where available.
- **Keep the binding to `example`** on `DeviceDefinition.classification.type` (do *not* strengthen to `extensible`, as the ticket originally mused about).
- **Add description text on the `classification` definition** explaining why both "device type" and "risk class" legitimately fit under a single `classification` backbone element — so implementers understand the element is intentionally broad.

The 1/19/2023 WGM discussion (captured in the single applied comment) also explored adding an example ValueSet covering selected device types (from ISO 11073) plus risk classes, and a possible future `category` element to separate "device type" from "risk class" with overlap to `conformsTo`. The motion that *passed* only commits to the three bullets above; the `category` element split is explicitly out of scope for this ticket.

## Feature Proposal

### Problem Statement

`DeviceDefinition.classification.type` (1..1 `CodeableConcept`, example binding) is currently bound to `http://hl7.org/fhir/ValueSet/device-type` (name `DeviceType`). That ValueSet is composed from SNOMED CT "is-a 49062001 (Device)" and selected ISO 11073 device codes — i.e., **device kind/type** concepts only. It does **not** contain risk/safety class concepts such as FDA medical-device class I/II/III or EU MDR class I / IIa / IIb / III.

At the same time, the element's `short`/`definition` already read "A classification or risk class of the device model," and the parent `classification` element's `comment` already enumerates "GMDN, EMDN, SNOMED CT, risk classes, national product codes." So the wire contract invites risk-class codes while the bound example ValueSet offers only device-type codes. Implementers are therefore misled about what `classification.type` is for, and have no example pool of risk-class codes to draw from.

### Proposed Change

Three coordinated edits in `HL7/fhir`:

1. **Add one or more new example ValueSets covering risk class (OUS)** under `source/device/` (co-located with the existing `valueset-device-type.xml`). Concretely, at minimum a new `ValueSet/device-risk-class` that enumerates the EU MDR device risk classes, and (where licensing permits) comparable OUS risk-class codes. Suggested shape:
   - `url` = `http://hl7.org/fhir/ValueSet/device-risk-class`
   - `name` = `DeviceRiskClass`, `title` = `Device Risk Class`
   - `status` = `active`, `experimental` = `true`
   - WG extension `structuredefinition-wg` = `oo`, standards-status `informative` (to match the sibling `device-type` ValueSet).
   - Compose an include for a new example CodeSystem (e.g. `http://hl7.org/fhir/device-risk-class-eu-mdr` or similar) enumerating at least codes `class-I`, `class-IIa`, `class-IIb`, `class-III` with EU MDR 2017/745 displays. If additional OUS regimes (e.g., Health Canada, TGA, PMDA) are in scope, add as separate includes; otherwise leave as a single example system and note extensibility.
   - The new CodeSystem(s) live alongside the ValueSet (e.g. `source/device/codesystem-device-risk-class-eu-mdr.xml`). Follow the existing `codesystem-devicedefinition-*-example.xml` pattern already present in `source/devicedefinition/`.
2. **Keep `DeviceDefinition.classification.type` bound to `device-type` with `strength=example`**, but update the binding description / add an alternate-binding or companion narrative so that users are pointed at both the existing DeviceType ValueSet (device kind) *and* the new DeviceRiskClass ValueSet (risk class). The passed motion explicitly keeps the binding strength at `example`, so **do not** promote to `extensible`. Practical options, any of which is acceptable if agreed with the WG:
   - Enrich the existing `<binding><description>` on `DeviceDefinition.classification.type` (line 811 of `structuredefinition-DeviceDefinition.xml`) to read e.g. `"Type of device e.g. according to official classification. For risk/safety class (OUS), see http://hl7.org/fhir/ValueSet/device-risk-class."`, keeping the primary `valueSet` pointer unchanged; *or*
   - Leave the bound ValueSet as `device-type` and add the risk-class ValueSet as an additional example cited in the resource's narrative (`devicedefinition-introduction.xml`) and in the new description added per bullet 3 below.
3. **Add definition/comment text on `DeviceDefinition.classification`** (the backbone, not `.type`) explaining that "classification" in this resource is intentionally broad and covers both (a) device kind/type (e.g., GMDN, EMDN, SNOMED CT device-is-a) and (b) risk/safety class (e.g., FDA class I/II/III, EU MDR class I / IIa / IIb / III), and that implementers `SHOULD` send a separate `classification` repetition per dimension rather than conflating them in one `type`. This replaces the current terse `definition` of "What kind of device or device system this is." The existing `comment` on the backbone already lists "risk classes" — extend it to state the rationale.

No change to cardinality, data type, or element structure of `DeviceDefinition.classification.*`. No new element (`category`) is introduced — that was debated but not part of the approved motion.

### Design Rationale

- **Binding strength stays `example`** because `classification.type` is intentionally a catch-all for multiple OUS regulatory regimes that HL7 cannot enumerate exhaustively, and strengthening to `extensible` would force implementers into a single suggested concept list per instance. The motion reaffirms this.
- **Add a sibling example ValueSet rather than stuffing risk-class codes into `device-type`.** Mixing kind and risk-class concepts into one ValueSet would make the existing `device-type` ValueSet self-contradictory (it is scoped by SNOMED CT `is-a 49062001 Device`) and would still not tell implementers that the two dimensions are different. Separate example ValueSets keep the concept boundary clean and allow a future non-breaking promotion to `extensible` per dimension.
- **Expand the backbone definition, not `.type`'s definition**, because the conflation confusion lives at the backbone level (one `classification` entry per dimension). Documenting it on `.type` alone would not explain why implementers should use multiple repetitions.
- **Defer the `category` element split** (debated at 2023-01 WGM) to a separate change request; the motion as captured in `resolution_description` does not commit to that structural change.

### Open Questions

1. **OUS scope breadth.** The motion says "value sets for risk class (OUS)" (plural). Does OO commit to EU MDR only for R6, or also Health Canada (class I–IV), TGA (class I / IIa / IIb / III / AIMD), PMDA (class I–IV), etc.? Impacts whether we ship one `device-risk-class` ValueSet with multiple includes or several per-regime ValueSets.
2. **URL/name for the risk-class CodeSystem(s).** EU MDR codes are not HL7-owned; the CodeSystem must be published as an example under an `hl7.org/fhir/...` namespace. Confirm the exact URL pattern (`device-risk-class-eu-mdr` vs. `eu-mdr-device-class`) with the WG / vocab lead.
3. **Inclusion of FDA (US) classes.** The motion says OUS explicitly, but the binding description arguably should point at *some* US example too. Confirm whether FDA classes I/II/III belong in a new US-scoped example CodeSystem or are left to implementer discretion.
4. **Alternate binding vs. description-only pointer.** Should the risk-class ValueSet be attached to `classification.type` as an *additional* binding (e.g., via `elementdefinition-additionalBinding`) or only cited in the description text? The motion does not specify; a description-only pointer is the minimum compliant option.
5. **Overlap with `DeviceDefinition.conformsTo`.** The WGM comment flags that risk class may also be carried via `conformsTo`. Should the new backbone definition text explicitly disambiguate (e.g., "`classification` carries the risk class assertion; `conformsTo` carries the declaration of conformance to a specific regulation")? Recommended yes, but confirm wording with OO.
6. **R6 / R6-Vocab label** — the `R6-Vocab` label on the ticket implies targeting the R6 cycle; confirm the motion is being landed on the current `master` (R6) branch, not an Rx backport.
