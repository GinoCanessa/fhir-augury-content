## Resolution Summary

**Resolution:** Persuasive — *"We will make the requested changes."*

The Vital Signs profiles' "Mandatory Requirements" narrative (rendered in
the profile page's *Text Summary* tab) currently states that the
mandatory LOINC code is carried in `Observation.category`. That is
factually wrong: `Observation.category` carries the fixed
`vital-signs` category coding (system
`http://terminology.hl7.org/CodeSystem/observation-category`), while the
LOINC code that identifies the specific vital sign measurement belongs
in `Observation.code`. Apply a uniform fix across all affected Vital
Signs profile narratives so the bullet reads *"One code in
`Observation.code` which must have a fixed
`Observation.code.coding.system`=`http://loinc.org` and a fixed
`Observation.code.coding.code`=`<loinc-code>`"*. The recent revisions
that introduced this wording were applied **inconsistently and
incompletely**, so every Vital Signs profile descended from
`vitalsigns` must be audited and corrected.

## Feature Proposal

### Problem Statement

Each derived Vital Signs profile (e.g., bodyheight, bodyweight, BMI,
heart rate, respiratory rate, body temperature, head circumference,
oxygen saturation, blood pressure, vitals panel) carries a
human-authored "Complete Summary of the Mandatory Requirements" block in
two extensions on the StructureDefinition root:

- `http://hl7.org/fhir/StructureDefinition/structuredefinition-summary`
  (`valueMarkdown`)
- `http://hl7.org/fhir/tools/StructureDefinition/profile-summary`
  (`valueString`)

These extensions feed the *Text Summary* tab of the published profile
page. In every affected profile the first numbered bullet currently
reads (verbatim, e.g. for bodyheight):

> 1.  One code in `Observation.category` which must have
>     -   a fixed `` `Observation.category.coding.system` ``=**'http ://loinc.org'**
>     -   a fixed  `` `Observation.category.coding.code` ``= **'8302-2'**

This is wrong on two counts: (a) `Observation.category` is fixed to the
`observation-category` CodeSystem with code `vital-signs`, not to a
LOINC code, and (b) the LOINC measurement code (e.g., `8302-2` for body
height) is constrained on `Observation.code`, not `Observation.category`.
Implementers reading the Text Summary tab are being told to put the
LOINC measurement code in the wrong element.

The `oxygensat` profile shows that an earlier pass started fixing this
but stopped halfway: its bullet now reads *"One code in
`Observation.code` which must have a fixed
`Observation.code.coding.system`=…"* but the next sub-bullet still
says *"a fixed `Observation.category.coding.code`=`'2708-6'`"*. The
`bp` profile is similarly half-fixed (system bullet still uses
`category`, code bullet uses `code`). The remaining profiles
(bodyheight, bodytemp, bodyweight, bmi, headcircum, heartrate,
resprate, vitalspanel) are uniformly wrong. The parent `vitalsigns`
profile is correct (it distinguishes "A category in
`Observation.category`" from "A code in `Observation.code`") and is
**not** in scope.

The reporter's only comment thread asks for the location; the
resolution ("We will make the requested changes") is unconditional.

### Proposed Change

For each affected profile, edit both the `valueMarkdown` and
`valueString` "Mandatory Requirements" narratives so that bullet 1 of
the Mandatory Requirements list refers consistently to
`Observation.code` (not `Observation.category`):

**Before** (e.g., bodyheight):

```
1.  One code in `Observation.category` which must have
    -   a fixed `Observation.category.coding.system`=**'http ://loinc.org'**
    -   a fixed  `Observation.category.coding.code`= **'8302-2'**
    -   8306-3 -Body height - lying (i.e., body length - typically used
        for infants) MAY be included as an additional observation code
        - The code (or codes) for the measurement obtained  in
        `Observation.code`. All codes SHALL have a system value
```

**After** (target wording for every affected profile, with each
profile's existing LOINC code preserved):

```
1.  One code in `Observation.code` which must have
    -   a fixed `Observation.code.coding.system`=**'http://loinc.org'**
    -   a fixed `Observation.code.coding.code`= **'<loinc-code>'**
    -   <profile-specific trailing notes preserved verbatim>
```

The change is purely a substitution of the path token `category` →
`code` inside three occurrences in bullet 1 of each narrative
(`Observation.category` → `Observation.code`,
`Observation.category.coding.system` → `Observation.code.coding.system`,
`Observation.category.coding.code` → `Observation.code.coding.code`).
All other prose, surrounding bullets, LOINC code values, line breaks
(`&#xD;` for most files, `&#xA;` for `oxygensat`), and trailing notes
(e.g., bodyheight's "8306-3 - Body height - lying" addition) must be
preserved exactly.

For the two partially-fixed profiles, only the still-wrong sub-bullets
need to change:

- `oxygensat`: in both extensions, change `` `Observation.category.coding.code` ``=`'2708-6'` to `` `Observation.code.coding.code` ``=`'2708-6'`.
- `bp`: in both extensions, change `` `Observation.category` `` (in the leading sentence) and `` `Observation.category.coding.system` ``=`'http ://loinc.org'` to use `` `Observation.code` `` / `` `Observation.code.coding.system` ``. The `'85354-9'` sub-bullet is already correct.

This change is **narrative-only**. No element definitions, cardinalities,
bindings, fixed values, mustSupport flags, or invariants are modified.
The actual constraints on `Observation.category:VSCat` (fixed pattern
`vital-signs`) and on `Observation.code` (LOINC binding plus the
profile-specific fixed code where applicable) are already correct in
the differential and are unchanged.

### Design Rationale

- The wording change is the literal correction requested by the
  reporter and approved by the work group ("Persuasive — We will make
  the requested changes"), so no design alternatives are in scope.
- Editing the two summary extensions (`structuredefinition-summary` and
  `tools/StructureDefinition/profile-summary`) is the right place
  because the bespoke FHIR publisher reads those values to render the
  *Text Summary* tab; there is no separately-authored HTML page for
  this content.
- Per the FhirCore briefing, only `<differential>` content of
  StructureDefinitions is hand-edited; `<snapshot>` blocks are
  regenerated. The summary extensions live at the SD root (above the
  differential and snapshot), so they are authored content and safe to
  edit.
- The fix is applied uniformly to every Vital Signs profile so the
  publisher's *Text Summary* output is consistent, addressing the
  ticket's complaint that prior revisions were applied
  "inconsistently and incompletely."
