## Resolution Summary

The Orders & Observations work group voted **Persuasive with Modification**. The narrative on the DiagnosticReport page that currently asserts "reports **SHALL** always include an effective[x] element" conflicts with the element's `0..1` cardinality and overstates the requirement. The resolution text (verbatim from Jira) replaces the existing paragraph in the *Clinically Relevant Time* section with a softened "best practice" wording, and additionally calls for the `DiagnosticReport.effective[x]` element `comment` to be replaced with a hyperlink pointing at this same section so that the two locations no longer diverge.

Existing paragraph (authored in `source/diagnosticreport/diagnosticreport-notes.xml`, lines 41–43):

> If specimens were taken, the clinically relevant time of the report can be derived from the specimen collection times, but since detailed specimen information is not always available, and nor is the clinically relevant time always exactly the specimen collection time (e.g. complex timed tests), the reports SHALL always include an effective[x] element. Note that [HL7 V2](http://www.hl7.org/implement/standards/product_brief.cfm?product_id=185) messages often carry a diagnostically relevant time without carrying any specimen information.

Approved replacement (verbatim from the resolution description):

> The best practice is to always include an effective[x] element. If specimens were taken, often the clinically relevant time of the report can be derived from the specimen collection times. However, detailed specimen information is not always available nor is the clinically relevant time always exactly the specimen collection time (e.g. complex timed tests). Note that [HL7 V2](http://www.hl7.org/implement/standards/product_brief.cfm?product_id=185) messages often carry a diagnostically relevant time without carrying any specimen information.

Additionally: the `<comment value="..."/>` on `DiagnosticReport.effective[x]` in `structuredefinition-DiagnosticReport.xml` (line 395) is to be replaced with a hyperlink to the updated *Clinically Relevant Time* section on the DiagnosticReport page. The linked PR is [HL7/fhir#4023](https://github.com/HL7/fhir/pull/4023) (not present in the local Augury cache at time of planning).

## Feature Proposal

### Problem Statement

The submitter (Bas van den Heuvel) observed that the narrative in the DiagnosticReport *Clinically Relevant Time* section states the report "SHALL always include an effective[x] element", yet `DiagnosticReport.effective[x]` is defined with cardinality `0..1` in the StructureDefinition. The narrative therefore contradicts the normative cardinality. The original rationale ("because there can be more than one time") was also judged confusing. The WG needs either to tighten cardinality to `1..1`, or to relax the prose — the WG chose to relax the prose to a "best practice" recommendation, keeping `0..1` cardinality unchanged.

### Proposed Change

Three concrete edits in a single repository (`HL7/fhir`):

1. **Rewrite the *Clinically Relevant Time* paragraph** in `source/diagnosticreport/diagnosticreport-notes.xml` so that it no longer says "SHALL always include", replacing it with the exact resolution text quoted above ("The best practice is to always include an effective[x] element. …"). The existing `<a href="diagnosticreport-definitions.html#DiagnosticReport.effective_x_">effective[x]</a>` link in the preceding sentence (line 40) is unchanged.
2. **Add an anchor** immediately before `<h3>Clinically Relevant Time</h3>` on line 38 so that other pages (including the new element comment) can link stably to it. Use the same style already used elsewhere in the same file (`<a name="notes"></a>`, `<a name="genetics"></a>`), e.g. `<a name="Clinically-Relevant-Time"></a>`.
3. **Replace the `comment` value on `DiagnosticReport.effective[x]`** in `source/diagnosticreport/structuredefinition-DiagnosticReport.xml` (line 395) so that instead of restating the narrative, it redirects readers to the section above. FHIR definitions render markdown, so a value of the form `See [Clinically Relevant Time](diagnosticreport.html#Clinically-Relevant-Time) for guidance on populating this element.` is appropriate. The exact anchor must match whatever is added in task 2.

No cardinality, type, binding, `min`/`max`, or mapping change is made on `DiagnosticReport.effective[x]`; it remains `0..1` of `dateTime | Period`.

### Design Rationale

The WG preferred a narrative relaxation over a cardinality change because raising `effective[x]` to `1..1` would be a significant breaking change to existing R4/R5 implementers and example instances, with no clinical justification beyond the confusing sentence itself. Rewording to "best practice is to always include" keeps guidance strong while making it non-normative, consistent with how other recommended-but-optional elements are discussed in FHIR narrative. Linking the element `comment` to the notes section — rather than duplicating the guidance — avoids a recurrence of the same drift in the future and matches the resolution description exactly ("we will also replace the text in the comment for .effective[x] to be a hyperlink to this section").
