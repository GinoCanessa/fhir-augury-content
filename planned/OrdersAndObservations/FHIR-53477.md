## Resolution Summary

The resolution is **Persuasive**. The HCP vote on 23/03/2026 (per Robin White,
ICCBBA) refined the original "redirect to external URLs" wording to a more
specific instruction:

> The existing codes in the FHIR code systems should be referenced as a
> **fragment** of the external code systems in THO.

Background (also recorded in the comments): the externally-defined ISBT 128
content is now published in HL7 Terminology (THO) as NamingSystems and
externally-defined CodeSystem maintenance resources (`ECS_MR`):

- ISBT 128 Product Description Codes — `NamingSystem-ISBT128ProductDescriptionCodes`,
  `ECS_MR-ISBT128ProductDescriptionCodes`.
- ISBT 128 Property Types — `NamingSystem-ISBT128PropertyTypes`,
  `ECS_MR-ISBT128PropertyTypes`.

In concrete terms the resolution requires three things in the FHIR Core
spec:

1. Convert the two FHIR-side CodeSystems (`biologicallyderived-productcodes`
   and `biologicallyderived-product-property-type-codes`) into **fragment**
   CodeSystems that point at the corresponding ISBT 128 external code
   systems now defined in THO, rather than asserting `content = complete`.
2. Capitalise the leading-alpha character of every code listed in
   `codesystem-biologicallyderived-productcodes.xml` (e.g., `e0398` → `E0398`,
   `s1128` → `S1128`, `t1396` → `T1396`) and the corresponding `display`
   anchors so the codes match the external system's actual values. The
   resolution lists nine codes explicitly: `E0398`, `S1128`, `S1194`,
   `S1195`, `S1310`, `S1398`, `S2598`, `E4377`, `T1396`.
3. Verify that the corresponding ValueSet expansions still resolve once
   the CodeSystem `content` and code casing change.

The original ticket text additionally proposed replacing the FHIR-side
`<url>` with the ISBT 128 canonical URLs
(`https://www.isbt128.org/uri/productdescriptioncode` and
`https://www.isbt128.org/uri`). The HCP refinement supersedes that
literal URL swap with the "fragment of THO ECS_MR" pattern; see **Open
Questions** for the exact `<url>` value question that needs to be
confirmed before authoring.

## Feature Proposal

### Problem Statement

`BiologicallyDerivedProduct.productCode` and the property-type binding on
`BiologicallyDerivedProduct.property.type` (see
`structuredefinition-BiologicallyDerivedProduct.xml` lines 131 and 399)
both reference FHIR-Core CodeSystems whose content is a hand-curated
subset of ISBT 128 codes published by ICCBBA. The FHIR Core CodeSystems
declare `content = complete` even though they enumerate only a handful
of codes from a much larger external standard, and the codes carry a
lower-case leading alpha (e.g., `e0398`, `s1128`, `t1396`) that does not
match the actual ISBT 128 value, which capitalises the leading alpha
(e.g., `E0398`, `S1128`, `T1396`). This makes the FHIR Core artifacts
both technically incorrect (`content` should be `fragment`) and
semantically wrong (the codes do not match the issuing standard).

The Orders & Observations work group raised FHIR-53477 to align the
FHIR Core CodeSystems with the authoritative ISBT 128 source. The
external content has since been registered in HL7 Terminology
(NamingSystem + ECS\_MR for both Product Description Codes and Property
Types — see comment of 2026-03-09), enabling the fragment pattern.

### Proposed Change

Treat both FHIR-Core CodeSystems as **fragments** of the corresponding
ISBT 128 external code systems now described in THO, and correct the
code casing in the Product Description CodeSystem.

**A. `codesystem-biologicallyderived-productcodes.xml`**

- Set `<content value="fragment"/>` (currently `complete`).
- Add a NamingSystem/ECS\_MR pointer in `<description>` that refers to
  the THO record
  (`https://terminology.hl7.org/NamingSystem/ISBT128ProductDescriptionCodes`)
  so consumers know where the authoritative content lives.
- Capitalise the leading alpha of every `<concept>.code` and update the
  matching `display` text where the lower-case form is repeated:
  - `e0398` → `E0398`
  - `s1128` → `S1128`
  - `s1194` → `S1194`
  - `s1195` → `S1195`
  - `s1310` → `S1310`
  - `s1398` → `S1398`
  - `s2598` → `S2598`
  - `e4377` → `E4377`
  - `t1396` → `T1396`
- Leave `<url>`, `<identifier>` (OID `2.16.840.1.113883.4.642.4.1998`),
  `<jurisdiction>`, and `<copyright>` unchanged unless the Open
  Questions resolve in favour of the literal ISBT 128 URL swap.

**B. `codesystem-biologicallyderived-product-property-type-codes.xml`**

- Set `<content value="fragment"/>`.
- Update `<description>` so it explicitly says this is a fragment of the
  ISBT 128 Property Types code system as registered in THO
  (`https://terminology.hl7.org/NamingSystem/ISBT128PropertyTypes`); the
  current description string is truncated mid-sentence
  (`...subset of the codes defined in http://terminology.hl7.org/CodeSystem/ib`)
  and must be rewritten as part of this fix.
- Existing concepts (`collectiontype`, `aborhd`, `singleeuropeancode`,
  `redcellantigen`) are not in the resolution's capitalisation table and
  are kept as-is. Confirm with the WG (Open Questions) whether ISBT 128
  expects mixed-case display values such as `CollectionType`.

**C. `valueset-biologicallyderived-productcodes.xml`**

- The `<compose><include><system>` value remains
  `http://hl7.org/fhir/biologicallyderived-productcodes` (no codes
  enumerated inline), so no edits are required unless Open Question 1
  resolves in favour of an external URL swap.

**D. `valueset-biologicallyderived-product-property-type-codes.xml`**

- Same as ValueSet (C): `<compose><include><system>` is
  `http://hl7.org/fhir/biologicallyderived-product-property-type-codes`;
  no in-line `<concept>` entries to edit.

**E. Release notes**

- Append a bullet to
  `BiologicallyDerivedProduct-release-notes.xml` recording the
  fragment-conversion and code casing fix, with `[%jira-task FHIR-53477%]`.

**F. Validation: ValueSet expansion**

- After the casing change, re-expand both ValueSets in the publisher and
  confirm every entry in the expansion uses the upper-cased codes. This
  is explicitly called out in the ticket
  ("Check to make sure the expansions still work once the URI is updated
  for the codeSystem").

### Design Rationale

The FHIR `CodeSystem.content = fragment` value exists exactly for this
case: a CodeSystem resource whose intent is to surface a small,
WG-curated subset of an externally-defined code system. THO's
NamingSystem + ECS\_MR pair is the established place to record the
external system; pointing FHIR Core at it via a fragment CodeSystem
keeps the binding surface stable for `BiologicallyDerivedProduct` while
delegating authoritative content management to ICCBBA/THO. The HCP
vote explicitly chose this pattern over a hard URL swap because (a) it
preserves backward compatibility for instances already using the
existing FHIR Core URLs, and (b) it avoids dragging the canonical
`https://www.isbt128.org/uri/...` URI through every ValueSet binding in
FHIR Core and downstream IGs.

The leading-alpha capitalisation is purely a correctness fix: the
codes shipped today are not ISBT 128 codes — ISBT 128 always uses an
upper-case alpha prefix. Changing the casing brings the FHIR-Core
fragment into agreement with the external system that it now
explicitly declares itself a fragment of.

### Breaking Changes

- **Code value casing change (E0398, S1128, S1194, S1195, S1310, S1398,
  S2598, E4377, T1396).** This is technically a breaking change for any
  consumer or instance carrying the lower-case form of these codes
  against `http://hl7.org/fhir/biologicallyderived-productcodes`.
  CodeSystems are `caseSensitive = true`, so an instance with
  `code = e0398` will no longer validate after the fix. Given the
  resource is FMM 1 / informative and the bindings are
  `strength = example`, real-world impact is likely low, but the change
  must be called out in release notes.
- **`content = fragment` change.** Tools that previously assumed they
  could rely on the FHIR-Core CodeSystem as a complete enumeration of
  ISBT 128 codes will now have to recognise it as a fragment and treat
  unknown ISBT 128 codes as not necessarily invalid. This is a
  conformance / tooling concern more than a wire-format break.
