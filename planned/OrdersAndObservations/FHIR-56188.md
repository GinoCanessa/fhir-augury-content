## Resolution Summary

**Resolution:** Persuasive with Modification.

Quoted resolution description: *"We will remove the reference to TermInfo pattern and remove the last item referencing TermInfo referencing guidance note section."*

Concretely, this requires removing the single list item in the Observation resource's guidance narrative that references the retired HL7 V3 "Terminfo" implementation guide (the "Assertion pattern" bullet, which is the last `<li>` of the Guidance `<ol>` in `source/observation/observation-notes.xml`). Rather than correcting the "TermInfo" vs. "Terminfo" spelling discussed in the ticket description (see FHIR-55270 context), the committee chose to drop the reference entirely because the document being cited is a retired standard.

## Feature Proposal

### Problem Statement

The Observation resource's guidance narrative (rendered into `observation.html` via `source/observation/observation-notes.xml`) currently tells implementers to `SHOULD NOT use the Assertion pattern as described in HL7 Version 3 Implementation Guide: Terminfo - Using SNOMED CT in CDA R2 Models, Release 1` and links to the HL7 product-brief page for that document.

Two problems were raised on the ticket:

1. The casing of the referenced standard is wrong. FHIR-55270 previously changed the spelling to "Terminfo" (lower-case "i"), but the HL7 standards grid uses **TermInfo** (upper-case "I").
2. More importantly, **TermInfo is a retired HL7 standard** — it only surfaces on the standards-grid search when retired standards are included. Telling FHIR R6 implementers to align their "Assertion pattern" guidance against a withdrawn document is not appropriate.

The Orders & Observations work group resolved the ticket by removing the reference rather than fixing the spelling.

### Proposed Change

Edit `source/observation/observation-notes.xml` and remove the `<li>` that references TermInfo from the "Guidance" ordered list (the one under `<a name="guidance"></a>` / `<h4>Guidance:</h4>`). In the clone at HEAD `db79dcd` this is lines 258–259:

```xml
<li>SHOULD NOT use the <em>Assertion</em> pattern as described in <a href="http://www.hl7.org/implement/standards/product_brief.cfm?product_id=418">HL7 Version 3 Implementation Guide: Terminfo - Using SNOMED CT in CDA R2 Models, Release 1</a>. ( The code is 'ASSERTION' and the value is a SNOMED CT concept or expression )
  </li>
```

After removal, the enclosing `<ol>` that opens at line 210 must still close cleanly with `</ol>` (currently at line 260), and the immediately following `<a name="refine">` anchor / "Refining the interpretation of an Observation…" section must remain in place. The preceding Guidance item ("Recommended rules for case 4 pattern", lines 251–257) becomes the new last item of that list.

No other prose changes are required — the item referenced in the ticket ("spelling") is being deleted, so the "TermInfo" vs. "Terminfo" spelling question is moot. A repository-wide grep for `Terminfo` / `TermInfo` in the clone (`source/`) shows `observation-notes.xml:258` as the **only** remaining occurrence, so this single deletion fully discharges both clauses of the resolution ("remove the reference to TermInfo pattern" and "remove the last item referencing TermInfo referencing guidance note section").

### Design Rationale

- **Consistency with HL7's retirement policy.** Normative FHIR guidance should not cite a standard HL7 itself has retired — implementers following such a pointer would land on a withdrawn document.
- **Targeted deletion, not rewording.** The committee explicitly chose removal over rewording, which avoids the difficult question of what the non-retired replacement guidance for an "Assertion pattern" would be. That debate can be picked up in a future ticket if needed.
- **Minimal blast radius.** The other four guidance items (cases 1–4) remain untouched and continue to describe the recommended Observation.code / Observation.value patterns. Removing only the negative "SHOULD NOT" bullet does not change any of the positive recommendations.
- **No StructureDefinition impact.** The change is narrative-only in a page snippet file; there is no cardinality, binding, invariant, or search-parameter change. Per the FhirCore playbook this is a "Update narrative / prose" recipe.
