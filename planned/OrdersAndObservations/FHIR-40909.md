## Resolution Summary

The approved motion (resolution: **Persuasive**) is:

> Motion to add:
> - **CW025 — MR SAFE**
> - **CW026 — MR CONDITIONAL**
> - **CW027 — MR UNSAFE**
> - **Containing_latex — Containing latex (MDR, MDD, AIMDD)**
>
> as examples to `http://build.fhir.org/valueset-device-safety.html` to support EUDAMED, sourced from the EU MDCG document `md_udi-device-enumerations_en_0.pdf` (2022-08).

The four codes come from the EU EUDAMED enumerations published by the European Commission. They are conceptually identical to four NCI Thesaurus codes already enumerated in `ValueSet/device-safety` (C106045 MR Safe, C106046 MR Conditional, C106047 MR Unsafe, C101673 Labeled as Containing Natural Rubber Latex), but they originate in a different controlled vocabulary (EUDAMED MDR/MDD/AIMDD).

The thread of comments (Costa-Teixeira → Velezis, 2025-01-05 through 2025-08-22) makes it explicit that **the resolution cannot be applied verbatim until a `system` URL exists for the EUDAMED enumerations**. As of 2025-08-22 the codes are PDF-only, the EU has not responded with a canonical URL, and Marti escalated the question to TI/TSMG (meeting 2025-08-28) to decide whether HL7 may mint a `CodeSystem` (likely under THO/UTG) or use an OID. The `Pending-SME-Input` and `R6-Vocab` labels reflect that this is open R6-Vocab work.

This plan therefore covers **(a)** the deterministic part — once a CodeSystem identifier exists, exactly what changes are made in `HL7/fhir` — and **(b)** the prerequisite work in HL7 terminology infrastructure that must precede part (a).

## Feature Proposal

### Problem Statement

`ValueSet/device-safety` (`http://hl7.org/fhir/ValueSet/device-safety`) currently enumerates only NCI Thesaurus codes for MR-environment safety and natural-rubber-latex labeling. EU implementers using **EUDAMED** (the European database for medical devices, established under the MDR, MDD and AIMDD regulations) classify the same concepts with a different set of identifiers (CW025 / CW026 / CW027 for MR Safe/Conditional/Unsafe, and `Containing_latex` for natural-rubber-latex labeling). Because these codes do not appear anywhere in `ValueSet/device-safety`, EUDAMED-aligned implementations cannot use the example value set verbatim and have no anchor in the FHIR spec for cross-walking EUDAMED values to the existing NCI concepts.

The work group has agreed (Persuasive) that the example `device-safety` value set should illustrate both vocabularies, signalling to implementers that EUDAMED codes are a legitimate alternative way to express device safety characteristics.

### Proposed Change

Add four EUDAMED concepts to `source/device/valueset-device-safety.xml` as a **second `<include>` block** (so each `<system>` URL groups its own concepts cleanly), with definitions transcribed from the MDCG enumerations PDF:

| EUDAMED code | Display | Conceptually equivalent NCI code (already in VS) |
|---|---|---|
| `CW025` | MR Safe | C106045 |
| `CW026` | MR Conditional | C106046 |
| `CW027` | MR Unsafe | C106047 |
| `Containing_latex` | Containing latex (MDR, MDD, AIMDD) | C101673 |

Concrete edit (illustrative — depends on the system URL chosen in the prerequisite step):

```xml
<include>
  <extension url="http://hl7.org/fhir/StructureDefinition/valueset-systemName">
    <valueString value="EUDAMED Device Enumerations"/>
  </extension>
  <system value="{EUDAMED_CODESYSTEM_URL}"/>
  <concept>
    <extension url="http://hl7.org/fhir/StructureDefinition/valueset-concept-definition">
      <valueString value="EUDAMED designation that the device has demonstrated safety in the MR environment within defined conditions (equivalent to NCI C106045 MR Safe). Source: EU MDCG md_udi-device-enumerations 2022-08."/>
    </extension>
    <code value="CW025"/>
    <display value="MR Safe"/>
  </concept>
  <concept>
    <extension url="http://hl7.org/fhir/StructureDefinition/valueset-concept-definition">
      <valueString value="EUDAMED designation that the device is conditionally safe in the MR environment under specified conditions (equivalent to NCI C106046 MR Conditional)."/>
    </extension>
    <code value="CW026"/>
    <display value="MR Conditional"/>
  </concept>
  <concept>
    <extension url="http://hl7.org/fhir/StructureDefinition/valueset-concept-definition">
      <valueString value="EUDAMED designation that the device poses unacceptable risks in the MR environment (equivalent to NCI C106047 MR Unsafe)."/>
    </extension>
    <code value="CW027"/>
    <display value="MR Unsafe"/>
  </concept>
  <concept>
    <extension url="http://hl7.org/fhir/StructureDefinition/valueset-concept-definition">
      <valueString value="EUDAMED designation that the device contains natural rubber latex, per MDR / MDD / AIMDD labeling (equivalent to NCI C101673 Labeled as Containing Natural Rubber Latex)."/>
    </extension>
    <code value="Containing_latex"/>
    <display value="Containing latex (MDR, MDD, AIMDD)"/>
  </concept>
</include>
```

Update `<description>` and the human-readable narrative to mention EUDAMED alongside NCI Thesaurus, and bump `<version>` per normal authoring conventions.

### Design Rationale

- **Two `<include>` blocks, one per code system.** This is consistent with existing `ValueSet/device-safety` style (single `<include>` per system, with `valueset-systemName` extension naming the source) and with how the FHIR spec composes example value sets that pull from multiple terminologies. It keeps each concept's `system` unambiguous.
- **Use `valueset-concept-definition` extensions** for the EUDAMED concepts (the existing NCI rows already use this pattern). This is required because the codes are defined locally in the ValueSet rather than resolved from a published CodeSystem snapshot.
- **Cross-reference to NCI in the definition text.** The four EUDAMED codes are semantically redundant with existing NCI codes; the `valueset-concept-definition` text explicitly notes the equivalence so implementers know to treat them as alternates rather than independent concepts. (The Costa-Teixeira / Velezis exchange in 2025-01 confirms this duplication is intentional and approved.)
- **`immutable=true` is preserved.** The ValueSet is currently `<immutable value="true"/>`. Adding new `<include>` content is technically a change to the value set's logical content, but because the change happens in the development branch (R6) before publication, it does not violate post-publication immutability of an already-released artifact. The `<version>` bump signals the new revision.
- **The `system` URL question is the gating decision.** Comments on the ticket (2025-07-09, 2025-07-25, 2025-08-22) make clear three options were on the table: (1) a published EU/EUDAMED canonical URL — desired but not available; (2) an OID under `urn:oid:` — viable fallback; (3) an HL7 THO/UTG-minted CodeSystem proxying the EUDAMED enumeration — the route Velezis raised with TI/TSMG. The choice is a TSMG matter; this plan defers naming the URL until that decision lands and treats `{EUDAMED_CODESYSTEM_URL}` as the single substitution required to apply the change.

#### HL7/UTG (Utg) — *conditional*

### Breaking Changes

No breaking changes. The change is purely additive: new `<concept>` entries inside `ValueSet/device-safety`, which is `experimental="false"` but bound as an **example** binding wherever it is used. Existing implementations that consume only the NCI codes are unaffected; existing instances remain valid; no element cardinality, type, or invariant is touched.

### Related Specifications

- **`Device` resource (`source/device/structuredefinition-Device.xml`)** — references `ValueSet/device-safety` via the `Device.safety` element binding. The binding itself is unchanged (still `example` strength, same canonical URL, same value set), so no edit to `structuredefinition-Device.xml` is required. Republishing the spec will pick up the new concepts in the rendered `valueset-device-safety.html` page automatically.
- **`DeviceDefinition`** — also references device-safety for its `safety` element in some FHIR versions; same reasoning — no edit required, just regenerated narrative.

None identified.

### Terminology Impact

- The four codes already exist *as concepts* in the value set under NCI codes; what's added is alternate **codings** for the same concepts from the EUDAMED enumeration.
- A new `<system>` URL is introduced into `ValueSet/device-safety`. The exact value depends on the prerequisite TSMG decision:
  - **Option A — official EUDAMED URL:** preferred but not currently available (per Velezis's request to the EU, no response received as of 2025-07-09).
  - **Option B — OID:** e.g., a `urn:oid:` form for the EUDAMED enumeration. Quickest to apply but undesirable as a long-term identifier.
  - **Option C — HL7 THO/UTG-minted CodeSystem:** would create a new `CodeSystem` in `HL7/UTG` enumerating CW025/CW026/CW027/Containing_latex with provenance to the MDCG PDF; the ValueSet then references the THO canonical URL.
- No change to existing NCI Thesaurus references.
- No CodeSystem resource is added or modified inside `HL7/fhir` itself for any of these options.
