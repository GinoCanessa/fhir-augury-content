## Resolution Summary

**Resolution:** Persuasive with Modification.

The Orders & Observations work group agreed to **replace** the current text of
the BodyStructure resource's *Boundaries and Relationships* section with the
following two paragraphs (verbatim from the ticket's resolution description):

> BodyStructure is used to explicitly identify and describe an anatomical
> structure when more detail or persistence is required than can be conveyed
> by a code alone.
>
> Where a resource needs to represent an anatomical structure and supports
> both coded and referenced representation, the preferred approach is to
> allow reference to BodyStructure (e.g., via CodeableReference). This enables
> reuse of the same anatomical structure across resources and supports
> additional detail such as laterality, qualifiers, morphology, or spatial
> description.

A third sentence ("A simple coded representation is sufficient when no
additional detail or reuse is required.") was discussed by John D.L. Nolen in
the 2026-03-17 comment as a candidate addition but was **not** carried into the
final resolution description. Implementation should follow the resolution
description and omit that sentence unless the work group later directs
otherwise (see Open Questions).

The ticket's narrative also asks that resources making use of this pattern
(ImagingStudy, ImagingSelection, Observation, ServiceRequest, …) be listed
"in this section together with short description" and that the section serve
as guidance for other resources. The approved resolution did **not** include
such a list — it only restates the principle. The plan therefore implements
the resolution as written and flags the broader request as out of scope for
this ticket.

## Feature Proposal

### Problem Statement

The BodyStructure resource is the canonical place to describe an anatomical
structure with more granularity than a single code can carry (laterality,
qualifiers, morphology, spatial description, included/excluded structures,
etc.). Across recent design discussions, the work groups have converged on
a pattern: where a resource needs to represent anatomical location and
supports both a coded value and a referenced structure, the field should be
typed as `CodeableReference(BodyStructure)` (e.g., `Observation.bodySite`,
`ServiceRequest.bodySite`, `ImagingStudy`/`ImagingSelection` body-site fields).

The current *Boundaries and Relationships* prose on the BodyStructure page
does not state this principle. It only says, in one sentence, that
`BodyStructure` should be used "consistent with the code" when a resource's
code implies a body structure, and that `bodySite` can be used otherwise.
That wording predates the CodeableReference convergence and gives no
guidance to resource authors on when to allow a reference to
`BodyStructure`, or why. This leaves the cross-resource pattern undocumented
and forces each resource design to re-litigate the question.

### Proposed Change

**Repository:** `HL7/fhir` (FhirCore).

**File:** `source/bodystructure/bodystructure-introduction.xml`.

**Change:** Replace the entire body of the `<h2>Boundaries and Relationships</h2>`
`<div>` (currently one `<p>` element following the `<a name="bnr"></a>` anchor)
with two `<p>` elements containing the resolution text. Concretely, the
existing block:

```xml
<div>
<a name="bnr"></a>
<h2>Boundaries and Relationships</h2>
<p>
If a resource's code value implies a body structure, then the bodyStructure resource should be used consistent with the code and to provide any additional detail. The bodySite element in any resource can be used when the resource's code is not implicit about the body structure and does not require extensive detail.
</p>
</div>
```

…is replaced with:

```xml
<div>
<a name="bnr"></a>
<h2>Boundaries and Relationships</h2>
<p>
BodyStructure is used to explicitly identify and describe an anatomical structure when more detail or persistence is required than can be conveyed by a code alone.
</p>
<p>
Where a resource needs to represent an anatomical structure and supports both coded and referenced representation, the preferred approach is to allow reference to BodyStructure (e.g., via <a href="references.html#CodeableReference">CodeableReference</a>). This enables reuse of the same anatomical structure across resources and supports additional detail such as laterality, qualifiers, morphology, or spatial description.
</p>
</div>
```

The `<a name="bnr"></a>` anchor MUST be preserved so that any inbound links of
the form `bodystructure.html#bnr` continue to resolve. No other elements in
`bodystructure-introduction.xml` are modified.

No StructureDefinition, ValueSet, search-parameter, schema, or example file is
changed; the resolution is purely narrative.

### Design Rationale

- **Faithful to resolution.** The two paragraphs are the verbatim resolution
  text from `metadata.resolution_description`. "Persuasive with Modification"
  is signalled in the ticket comment thread (the third sentence proposed
  during discussion was dropped from the resolution description), so the
  modification is carried by adopting only the two-paragraph version.
- **Replaces, not appends.** The ticket explicitly says the principle is
  "nooplace mentioned" today, and the resolution wording sets out *the*
  recommended pattern. Leaving the legacy sentence in alongside the new
  prose would create two competing statements about when to use
  `bodySite` vs. `Reference(BodyStructure)`. Replacement avoids that.
- **Hyperlinking `CodeableReference`.** The resolution text mentions
  CodeableReference as an example; FhirCore convention is to link
  cross-references to types into the spec (`references.html#CodeableReference`).
  This is a small editorial improvement consistent with the rest of
  `source/bodystructure/bodystructure-introduction.xml` style and with
  surrounding pages such as `procedure.html` / `observation.html` already
  linked from the Scope and Usage paragraph. If FMG editorial preference is
  to keep the prose link-free, the `<a>` can be dropped without affecting
  meaning.
- **Scope discipline.** The ticket narrative requested an enumerated list of
  resources that follow the pattern (ImagingStudy, ImagingSelection,
  Observation, ServiceRequest, …). The approved resolution does not contain
  such a list; adding one here would (a) exceed the resolution and (b)
  duplicate information that already lives on each referencing resource's
  own page. That request is recorded under Open Questions.
