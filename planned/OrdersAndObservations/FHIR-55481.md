## Resolution Summary

**Resolution: Persuasive with Modification.**

FHIR-55276 retargets `Specimen.role`'s binding from the FHIR-local
`http://hl7.org/fhir/ValueSet/specimen-role` (backed by the FHIR-local
`http://hl7.org/fhir/specimen-role` CodeSystem) to a THO-based value set that
reuses `http://terminology.hl7.org/CodeSystem/v2-0369`. Once that retarget
lands, this ticket **removes the now-unused FHIR specimen-role ValueSet and
CodeSystem** from the core FHIR specification.

The original ticket also proposed adding a Note-to-Balloters on the Specimen
page flagging the non-compatible change to `Specimen.role`'s binding. The
approved resolution explicitly modifies that: *"We will not include a note
to Balloters at this time."* — so no balloter note is in scope.

## Feature Proposal

### Problem Statement

The core FHIR specification currently defines both
`ValueSet/specimen-role` and `CodeSystem/specimen-role` (see
`source/specimen/valueset-specimen-role.xml` and
`source/specimen/codesystem-specimen-role.xml` in the `HL7/fhir` clone).
After FHIR-55276 is applied, `Specimen.role` no longer binds to that value
set — it binds to a THO-hosted value set based on
`http://terminology.hl7.org/CodeSystem/v2-0369`. Leaving the FHIR-local
specimen-role artifacts published alongside the new THO-based binding
creates a confusing parallel definition that nothing in the core spec
references and whose codes (lowercase FHIR variants) differ from the
authoritative THO codes.

### Proposed Change

Remove the two FHIR-local specimen-role terminology artifacts from
`HL7/fhir`, and scrub the two stray references to their canonical URLs in
the publisher's OID registry:

1. **Delete** `source/specimen/codesystem-specimen-role.xml` — the
   `CodeSystem` with `url = http://hl7.org/fhir/specimen-role` (id
   `specimen-role`, name/title `SpecimenRole` / `Specimen Role`, OID
   `2.16.840.1.113883.4.642.4.2060`).
2. **Delete** `source/specimen/valueset-specimen-role.xml` — the
   `ValueSet` with `url = http://hl7.org/fhir/ValueSet/specimen-role`
   (id `specimen-role`, OID `2.16.840.1.113883.4.642.3.3216`),
   currently composed as `include system
   http://hl7.org/fhir/specimen-role`.
3. **Edit** `source/oids.ini` — remove the two lines that register the
   deleted canonicals:
   - Line 858: `http://hl7.org/fhir/ValueSet/specimen-role = 2.16.840.1.113883.4.642.3.3216`
   - Line 1474: `http://hl7.org/fhir/specimen-role = 2.16.840.1.113883.4.642.4.2060`
4. **Edit** `source/specimen/specimen.svg` — the diagram currently links
   `Specimen.role` to `valueset-specimen-role.html` (line 230,
   `xlink:href="valueset-specimen-role.html"`). After FHIR-55276 retargets
   the binding, this link points at a page that will no longer be
   generated. Repoint the link to the new THO value set page surfaced
   by FHIR-55276, or remove the link, matching whatever pattern
   FHIR-55276 uses for the other specimen diagrams that now reference
   THO.

Do **not** add a Note-to-Balloters to
`source/specimen/specimen-notes.xml` — the resolution explicitly removed
that from scope.

No edit is required to `source/specimen/structuredefinition-Specimen.xml`
in this ticket: the `Specimen.role` binding line at
`structuredefinition-Specimen.xml:330`
(`<valueSet value="http://hl7.org/fhir/ValueSet/specimen-role"/>`) is
owned by **FHIR-55276**, which retargets it to the THO value set. This
ticket must run **after** FHIR-55276 has changed that URL; otherwise the
live `Specimen.role` binding would dangle.

### Design Rationale

- The approved motion on FHIR-55276 chose to **reuse** the THO
  `v2-0369` code system rather than keep a parallel FHIR-local copy,
  making the FHIR-local specimen-role VS/CS redundant.
- Leaving unused canonicals published would perpetuate two URLs
  (`http://hl7.org/fhir/...` and `http://terminology.hl7.org/...`) for
  the same concept, harming interop and complicating V2↔FHIR mapping
  (the very use-case FHIR-55276 called out).
- Dropping the Note-to-Balloters follows the Applied vote on this
  ticket and defers any balloter communication to the work group's
  normal release-note process (ticket is labelled
  `R6-Release-note`).

### Breaking Changes

Yes — this is a non-compatible change, consistent with the ticket's
`R6-Release-note` label.

- The canonical URLs `http://hl7.org/fhir/ValueSet/specimen-role` and
  `http://hl7.org/fhir/specimen-role` (CodeSystem) will no longer
  resolve in the core FHIR package after this change. Any implementer

  codes lowercase → uppercase per the THO `v2-0369` source) is driven
  by FHIR-55276, but it lands as a pair with this removal, so the
  R6 release note must cover both.

## Implementation Plan
