## Resolution Summary

**Resolution:** Persuasive with Modification

**Resolution description (verbatim):** "We will remove the requirement from .subject entirely."

The committee accepted that the current `<requirements>` text on
`DiagnosticReport.subject` ("SHALL know the subject context.") conflicts with
the element's `0..1` cardinality. The agreed modification is to remove the
`<requirements>` element from `DiagnosticReport.subject` altogether rather
than rephrase it to a lower-case "should".

> ⚠️ The PR linked from the ticket (HL7/fhir#4021) does **not** match this
> resolution description. It rewrites the text to `"Should know the subject
> context."` instead of removing the `<requirements>` element. See the Open
> Questions section — this plan follows the written resolution, which says
> the requirement is removed entirely.

## Feature Proposal

### Problem Statement

`DiagnosticReport.subject` has cardinality `0..1` (optional, single) but its
requirements prose states "SHALL know the subject context.". The two
statements contradict each other: a SHALL cannot apply to an optional
element. The ticket's reporter offered three possible fixes — remove the
requirement, lower-case it, or change SHALL to SHOULD — and the O&O work
group picked removal.

### Proposed Change

In `source/diagnosticreport/structuredefinition-DiagnosticReport.xml`, within
the element definition whose `id` is `DiagnosticReport.subject` (path
`DiagnosticReport.subject`, currently at line 309–330 of the clone at
`db79dcd`), **delete the line**:

```xml
<requirements value="SHALL know the subject context."/>
```

No other fields on the element change: `short`, `definition`, `alias`, `min
= 0`, `max = 1`, types, mappings, and the element ordering all remain as-is.
No changes are made to the resource narrative
(`source/diagnosticreport/diagnosticreport-introduction.xml`) beyond the
standard R6 release-note roll-up that lists the applied ticket.

### Design Rationale

The work group judged that, because a DiagnosticReport can legitimately be
issued without a known subject (per the element's `0..1` cardinality and the
existing definition's note that diagnostic services "also perform analyses
on specimens collected from a variety of other sources"), there is no
universal requirement to state. Rewording SHALL → SHOULD would still imply
a normative expectation that does not hold for all DiagnosticReport
instances (e.g., anonymous public-health surveillance specimens, quality-
control samples). Dropping the `<requirements>` sentence entirely removes
the contradiction without introducing a new, still-inaccurate, soft
normative claim. The `requirements` element is optional in
StructureDefinition, so its absence is valid.
