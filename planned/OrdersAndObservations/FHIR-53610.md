## Resolution Summary

**Resolution:** Persuasive.

The committee agreed to remove `Observation.valueReference` and
`Observation.component.valueReference` from the Observation resource, along
with the matching `value-reference` and `component-value-reference` search
parameters. These elements/search parameters were originally added as the
Observation-side anchor for `MolecularSequence`/`MolecularDefinition`
references; once `MolecularSequence` was removed from FHIR Core they became
orphaned (no remaining target profile, no use case in core).

A merged implementation already exists on `master` as PR
[HL7/fhir#3999](https://github.com/HL7/fhir/pull/3999) (merge commit
`f30cf02b`, work commit `c5527236`), so this plan documents the resolution
as applied — not a new design exercise.

## Feature Proposal

### Problem Statement

`Observation.value[x]` and `Observation.component.value[x]` historically
included `Reference` as a permitted choice type. The only intended use of
that choice was to point at definitional resources from the genomics work
(originally `MolecularSequence`, then `MolecularDefinition`). The element
comments in `structuredefinition-Observation.xml` even spelled this out:

> "The usage of valueReference is restricted to the MolecularDefinition
> resources as a definitional resource, not as a patient-specific finding."

When `MolecularSequence` was removed from FHIR Core, the `Reference` choice
on `value[x]` lost its only target profile. The corresponding search
parameters `Observation-value-reference` (code `value-reference`) and
`Observation-component-value-reference` (code `component-value-reference`)
likewise became orphaned: they index a choice that no longer has a meaningful
target. Leaving these in place is misleading to implementers (the spec
appears to support a Reference value with no documented target) and
generates dead conformance surface (search parameters with no realistic
hits).

### Proposed Change

Make the following changes in `HL7/fhir`:

1. **Remove the `Reference` choice from `Observation.value[x]`.**
   - Element path: `Observation.value[x]`.
   - Action: delete the `<type><code value="Reference"/></type>` entry
     (with its commented `targetProfile` for `MolecularDefinition`).
   - Remaining permitted types are unchanged: `Quantity`, `CodeableConcept`,
     `string`, `boolean`, `integer`, `Range`, `Ratio`, `SampledData`,
     `time`, `dateTime`, `Period`, `Attachment`.
   - Cardinality (`0..1`) and conditions (`obs-7`, `obs-6`, `obs-11`) stay
     as-is.

2. **Remove the `Reference` choice from `Observation.component.value[x]`.**
   - Element path: `Observation.component.value[x]`.
   - Action: delete the corresponding `<type><code value="Reference"/></type>`
     entry (with commented `targetProfile`).
   - Cardinality (`0..1`) and condition `obs-10` stay as-is.

3. **Strip the MolecularDefinition restriction sentences from the element
   comments.** In both the `Observation.value[x]` and
   `Observation.component.value[x]` `<comment>` values, remove the bullet:

   > "The usage of valueReference is restricted to the MolecularDefinition
   > resources as a definitional resource, not as a patient-specific
   > finding."

   leaving the surrounding guidance about choosing a datatype based on
   `code`, the `CodeableConcept`/`Attachment` advice, and the pointer to the
   Notes section intact.

4. **Delete the orphaned search parameters from
   `bundle-Observation-search-params.xml`.** Remove all `<entry>` elements
   carrying:
   - `SearchParameter.id = Observation-value-reference` (code
     `value-reference`, expression `Observation.value.ofType(Reference)`).
   - `SearchParameter.id = Observation-component-value-reference` (code
     `component-value-reference`, expression
     `Observation.component.value.ofType(Reference)`).

   Note: the bundle currently contains duplicate copies of each of these
   two entries (four entries total in the area starting around line 765);
   all four must go.

5. **No new elements, extensions, value sets, or invariants are introduced.**

### Design Rationale

The change is a clean orphan removal driven by an upstream deletion
(`MolecularSequence`/`MolecularDefinition` no longer in core). Keeping a
choice type whose only target profile no longer exists in the core
specification:

- breaks the Observation guidance ("the datatype for `value[x]` should be
  determined by `code`") because there is no defined coded use for
  `Reference`;
- leaves the explanatory comment ("restricted to MolecularDefinition")
  pointing at a non-existent definitional resource;
- ships search parameters that no real instance can populate.

Alternatives considered (implicit in the resolution):

- **Repoint `valueReference` at a different definitional resource.** Not
  pursued — there is no replacement target in FHIR Core, and any future
  genomics IG can re-introduce a profile-level extension or its own
  `value[x]` slice if needed.
- **Keep the elements / search parameters but flag as deprecated.** Not
  pursued — this is a Technical Correction, the resolution wording is
  "Remove", and PR #3999 does an outright removal.

This change is consistent with the repo's general pattern of pruning core
content when the upstream resource it depended on is removed.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole repository for the change. Hosts the Observation
  StructureDefinition and the Observation search-parameter bundle, both of
  which are hand-authored under `source/observation/`.
- **Affected Files:**
  - `source/observation/structuredefinition-Observation.xml` — delete
    `<type><code value="Reference"/>…</type>` from `Observation.value[x]`
    and `Observation.component.value[x]`; remove the
    "restricted to the MolecularDefinition resources" sentence from both
    `<comment>` values.
  - `source/observation/bundle-Observation-search-params.xml` — delete
    every `<entry>` whose `SearchParameter.id` is
    `Observation-value-reference` or
    `Observation-component-value-reference` (PR #3999 removed four such
    entries; verify against current `master` before deleting).
- **Change Scope:** Minor (two files, removals only, no new artefacts).

### Breaking Changes

This **is** a breaking change for any client that relies on:

- A `valueReference` element on `Observation` or `Observation.component` —
  instances populating `value` with a `Reference` will no longer validate
  against the core profile.
- The `Observation` search parameters `value-reference` and
  `component-value-reference` — servers advertising or clients querying
  these will need to drop them.

The ticket type is `Technical Correction` with resolution `Persuasive`,
and the resolution explicitly directs removal, so the breakage is
intentional and aligned with the upstream `MolecularSequence` removal.

### Related Specifications

- Any IG that profiles `Observation` and slices `value[x]` or constrains
  `value-reference` / `component-value-reference` search parameters will
  need to be revisited. Genomics-adjacent IGs are the most likely
  candidates.
- The Notes section on `observation.html` already de-emphasises the
  Reference choice; it should be re-read after the edit to confirm no
  remaining references to "valueReference" or to MolecularDefinition slip
  through.

### Related Tickets

- No additional Jira tickets surfaced via `cross-referenced` for
  `FHIR-53610`. The four cross-references are: two FHIR-artifact mentions
  (`Observation.valueReference`, `Observation.component.valueReference`)
  and two GitHub commits in HL7/fhir (`c5527236` work commit and
  `f30cf02b` merge commit for PR #3999). No companion or conflicting
  tickets identified.

### Terminology Impact

No terminology impact. The removed elements are typed `Reference`, not
coded; no ValueSet or CodeSystem in `HL7/UTG` is affected.

#### HL7/fhir

   `Observation.value[x]`.**
   - File: `source/observation/structuredefinition-Observation.xml`
   - Action: in the `<element>` whose `<path>` is `Observation.value[x]`,
     delete the `<type>` block whose `<code value="Reference"/>` (and the
     adjacent commented-out `targetProfile` line for
     `MolecularDefinition`). In the same element's `<comment>` value,
