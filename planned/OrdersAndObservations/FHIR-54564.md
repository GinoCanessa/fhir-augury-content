## Resolution Summary

**Resolution:** Persuasive with Modification.

> *Motion to update the reference to have a target type of "Any" to the definition of `.statusReason`.*

`Task.statusReason` is defined as a `CodeableReference` but its `<type>` element in `structuredefinition-Task.xml` carries no `targetProfile`, so the specification does not state which resource types are valid reference targets. The approved resolution is to add an explicit target of "Any" (i.e., the abstract `Resource` type) to the element's `CodeableReference` type declaration. After the change, `Task.statusReason.reference` may point at any resource (the same pattern already used by sibling element `Task.reason`).

## Feature Proposal

### Problem Statement

`Task.statusReason` was changed from `CodeableConcept` to `CodeableReference` in R5 (FHIR-29118). When an element of type `Reference` or `CodeableReference` carries no `targetProfile`, the FHIR publisher and downstream tooling treat the target set as unspecified rather than "Any" — implementers and validators are left to guess which resource types are legal reference targets. Because the ticket comment explicitly notes the R5 history and there is no narrative override on the Task page, the most charitable reading today is "unspecified", which is exactly the gap the ticket flags. The O&O Workgroup's resolution is to make the intent explicit: any resource may be referenced from `Task.statusReason.reference`.

### Proposed Change

In `source/task/structuredefinition-Task.xml`, locate the `<element id="Task.statusReason">` block (currently lines 232–262 of the differential) and add a single `<targetProfile>` to the existing `CodeableReference` `<type>`:

Before:

```xml
<type>
  <code value="CodeableReference"/>
</type>
```

After:

```xml
<type>
  <code value="CodeableReference"/>
  <targetProfile value="http://hl7.org/fhir/StructureDefinition/Resource"/>
</type>
```

No other element fields change: `min` (`0`), `max` (`*`), `short`, `definition`, `comment`, `isSummary` (`true`), the `example` binding to `http://hl7.org/fhir/ValueSet/task-status-reason`, and the existing RIM and workflow mappings all remain as-is.

### Design Rationale

This mirrors the established intra-resource pattern: `Task.reason` (also a `CodeableReference`, lines 793–812 of the same file) already declares `<targetProfile value="http://hl7.org/fhir/StructureDefinition/Resource"/>`. Using the abstract `Resource` profile is the canonical way in FHIR Core to encode "Any" for `Reference`/`CodeableReference` targets — it is consistent with both the workflow pattern used elsewhere on `Task` and the FHIR R6 reference-targeting conventions documented on `references.html`. It avoids enumerating an exhaustive (and inevitably incomplete) list of resource types, which is appropriate because `statusReason` is intentionally open-ended (a Task can be held, failed, or refused for reasons that may be described by virtually any resource — Provenance, DocumentReference, Communication, Observation, Encounter, etc.). No alternative was suggested by the ticket discussion; the single comment from the assignee only documents the R5 type-change history and does not propose a constrained target list.
