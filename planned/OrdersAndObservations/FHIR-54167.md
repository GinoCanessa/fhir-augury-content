## Feature Proposal

### Problem Statement

The original `DocumentReference.basedOn` element description in R6 draft read "A procedure that is fulfilled in whole or in part by the creation of this media." That wording was wrong on two counts: (1) every target profile on `basedOn` is a request/order resource (`Appointment | AppointmentResponse | CarePlan | Claim | CommunicationRequest | Contract | CoverageEligibilityRequest | DeviceRequest | EnrollmentRequest | MedicationRequest | NutritionOrder | RequestOrchestration | ServiceRequest | VisionPrescription`) — none is `Procedure`; (2) `DocumentReference` indexes *documents*, so "this media" is inconsistent with the rest of the resource's prose, which uses "document".

FHIR-54536 ("DocumentReference.basedOn points to requests not events") then generalised the description to use the workflow term **"activity"** (so it correctly covers the request/order profiles listed on the element) and replaced "this media" with "this document". That change was made to the element's `<short>` and `<definition>` in the StructureDefinition but **not** to the matching SearchParameter description in `bundle-DocumentReference-search-params.xml`, which still reads `"Procedure that caused this media to be created"`. FHIR-54167's remaining deliverable is to bring the SearchParameter description into line with the element's `<short>`, so the published SearchParameter no longer contradicts the resource definition.

### Proposed Change

Edit a single `<description value="…"/>` on the `DocumentReference-based-on` SearchParameter in the R6 authoring source:

- **File:** `source/documentreference/bundle-DocumentReference-search-params.xml`
- **Line 54:** change
  - Before: `<description value="Procedure that caused this media to be created"/>`
  - After:  `<description value="Activity that caused this document to be created"/>`

Rationale for the exact replacement text: the SearchParameter's description should mirror the element `<short>` it indexes. The current `<short>` on `DocumentReference.basedOn` (line 200 of `structuredefinition-DocumentReference.xml`) is already `"Activity that caused this document to be created"` — reusing that string keeps search-parameter metadata and element metadata identical, as is standard practice for FHIR Core search parameters derived from a single element.

No changes are needed to:

- `structuredefinition-DocumentReference.xml` — `<short>` and `<definition>` already updated by FHIR-54536.
- `documentreference-introduction.xml` — the STU-note bullet for FHIR-54167 is already present (line 26).
- `documentreference-notes.xml` / examples — no stale "procedure … media" wording remains.

### Design Rationale

- **Use "activity", not "request or response".** The ticket's original resolution text reads "A request or response that is fulfilled…". FHIR-54536 chose the broader term **"activity"** instead, which is the canonical workflow term on `Event.basedOn` mappings (see the `<mapping>` entry `workflow` → `Event.basedOn` at line 225 of the SD). The QA note on this ticket confirms the merged wording is the FHIR-54536 "activity" form; aligning the SearchParameter to that wording (rather than reintroducing "request or response") preserves consistency with the element definition that is already in the build.
- **Minimal, non-breaking edit.** The description of a SearchParameter is informational. Changing it does not alter the search expression, type, target resources, or code, so no consumers can break. This is a documentation-only fix appropriate for the R6 release-note label already on the ticket.
- **Scope discipline.** The QA comment explicitly scopes PR #4040 to the short-description fix. Rolling the SearchParameter change into the same narrow PR keeps the ticket closable without entangling it with FHIR-54536 territory.

### Open Questions

- **Use "activity" vs. the resolution's literal "request or response"?** The resolution text in Jira says "A request or response that is fulfilled…". FHIR-54536's applied edit uses "An activity that is fulfilled … this document." and the current `<short>` is "Activity that caused this document to be created." This plan aligns the SearchParameter description to the **applied** wording ("Activity … this document") rather than the literal Jira resolution wording, on the grounds that (a) the QA comment on FHIR-54167 explicitly defers to the FHIR-54536 wording as already‑applied, and (b) search-parameter descriptions conventionally mirror the element `<short>`. If the work group prefers the literal resolution wording, the target string in Task 1 should instead be `"A request or response fulfilled in whole or in part by the creation of this document"`; this is mutually exclusive with the element `<short>` and would re-open a parallel edit on `structuredefinition-DocumentReference.xml` lines 200–201, enlarging the scope of FHIR-54167 beyond what PR #4040's draft scoped (short-definition media→document only).
- **Anchor / inbound-link audit (briefing gotcha).** The briefing flags that HTML anchor removals should be audited before merging; this plan removes no anchors, so the gotcha is inapplicable — noted here only to confirm it was considered.
