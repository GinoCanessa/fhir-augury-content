## Resolution Summary

The ticket is resolved as **Persuasive**. The approved motion is:

> "Motion to find persuasive to retain flexibility for future use cases to assign multiple identifiers to a packaging instance. And update the definition accordingly."

The comment by jdlnolen on 2026-02-20 ("change cardinality and plural change") confirms the concrete action: change the cardinality of `DeviceDefinition.packaging.identifier` from `0..1` to `0..*` and update the `short`/`definition` text to reflect that multiple identifiers may be assigned to a packaging instance. Per FHIR naming convention, the element name `identifier` stays singular even when it becomes repeating (consistent with `Patient.identifier`, `Organization.identifier`, etc.).

## Feature Proposal

### Problem Statement

In the current `DeviceDefinition` StructureDefinition, `DeviceDefinition.packaging.identifier` is constrained to `0..1`. As noted in the ticket description, identifiers for real-world packaging are almost always assignable by multiple entities over time (e.g., manufacturer-assigned GTIN/UDI-style identifiers plus distributor- or regulator-assigned identifiers). The current cardinality forces implementers to pick one identifier per packaging instance, losing fidelity and diverging from the pattern used almost everywhere else in FHIR where `.identifier` is `0..*`.

### Proposed Change

Change the `DeviceDefinition.packaging.identifier` element to be repeating and refresh its documentation:

- **Element id / path:** `DeviceDefinition.packaging.identifier` (unchanged)
- **Element name:** `identifier` — **unchanged** (singular, per FHIR convention for repeating identifier elements)
- **Type:** `Identifier` (unchanged)
- **Cardinality:** `0..1` → **`0..*`** (change `<max value="1"/>` to `<max value="*"/>`)
- **`short`:** `"Business identifier of the packaged medication"` → **`"Business identifier(s) of the packaged medication"`**
- **`definition`:** `"The business identifier of the packaged medication."` → **`"Business identifier(s) assigned to the packaged medication. A packaging instance may be assigned identifiers by multiple entities (e.g., manufacturer, distributor, regulator) and/or over time."`**

No other element changes. The referenced `structuredefinition-DeviceDefinition.xml` is the differential-only authoring copy — the publisher regenerates snapshots, so only the differential `<element>` block needs editing.

### Design Rationale

- **Consistency with FHIR patterns.** Repeating `.identifier` is the standard across almost all FHIR resources and backbone elements (e.g., `Patient.identifier 0..*`, `Substance.identifier 0..*`, `Medication.identifier 0..*`). Making packaging match removes a surprising exception.
- **Real-world fidelity.** As stated in the resolution motion and ticket description, packaging identifiers are routinely assigned by multiple entities/systems; constraining to one forces implementers to drop information or invent workarounds.
- **Backwards-compatible-in-practice loosening.** Widening `0..1` to `0..*` does not invalidate existing instances (a single identifier still satisfies the new cardinality). It does, however, require consumers to accept a repeating structure — see Breaking Changes below.
- **Name stays singular.** FHIR naming convention keeps `.identifier` singular regardless of cardinality; no element rename is implied by "plural change" in the applied-vote comment — that comment refers to pluralising the narrative prose in `short`/`definition`, not the element name.

### Affected Repositories

- **Role:** Sole repository affected. Hosts the authoritative `DeviceDefinition` StructureDefinition differential.

- **Change Scope:** Minor (single element, three attribute edits in one differential block).

### Breaking Changes

Widening `0..1` → `0..*` is technically a **non-breaking schema loosening** for *producers* of `DeviceDefinition` (any instance valid against the old cardinality is also valid against the new one). However it is a **breaking change for consumers** that assume a single-identifier structure:

- **XML/JSON shape change for consumers.** Code that reads `packaging.identifier` as a singleton object must now handle an array. In FHIR JSON, a repeating element is represented as a JSON array, which is a structural/parser-level change for clients with strict schema bindings (e.g., generated class libraries where the property type changes from `Identifier` to `List<Identifier>`).
- **Search parameters.** No existing search parameter (verified in `source/devicedefinition/bundle-DeviceDefinition-search-params.xml`) targets `DeviceDefinition.packaging.identifier`, so no search-parameter breakage.
- **No terminology, type, or removal changes.**

Per the `R6-Release-note` label on the ticket, this must be called out as a release note for R6.

### Related Specifications

- **`DeviceDefinition.packaging.udiDeviceIdentifier`** — sibling element, already `0..*`. After this change, `packaging.identifier` aligns with it.
- **`DeviceDefinition.packaging.packaging`** — recursive backbone via `<contentReference value="#DeviceDefinition.packaging"/>` (line 1037). The contentReference automatically inherits the new `identifier` cardinality for nested packaging levels; no edit needed.
- **`DeviceDefinition.identifier`** (top-level) — already `0..*`, so the new `packaging.identifier` is consistent.
- **Related resources with packaging identifiers** (e.g., Medication packaging structures, Substance, Ingredient) already use `0..*` identifier; this change brings DeviceDefinition in line.
