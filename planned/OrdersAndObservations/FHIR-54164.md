## Resolution Summary

Resolution: **Persuasive with Modification**.

The committee approved updating `DiagnosticReport.study` so that its prose
matches the (already-applied) removal of `GenomicStudy` from its allowed
target types. Concretely the resolution table on the ticket calls for:

- **Definition** — keep the existing text up through "…provide views of
  the source images." and **strike (delete)** the trailing sentence:
  *"A GenomicStudy might comprise one or more analyses, each serving a
  specific purpose. These analyses may vary in method (e.g.,
  karyotyping, CNV, or SNV detection), performer, software, devices
  used, or regions targeted."*
- **Comments** — strike (delete) the leading sentence
  *"For laboratory-type studies like GenomeStudy, type resources will be
  used for tracking additional metadata and workflow aspects of complex
  studies."* and retain the remaining ImagingStudy/media-overlap
  sentences.
- **Short Display, Cardinality (0..\*), Type (Reference(ImagingStudy)),
  Summary (false)** — unchanged.

The original ticket text proposed simplifying the Definition to a
shorter ImagingStudy-only sentence and "remove the Comment" entirely;
the resolution table is the modified version that was approved (longer
Definition retained, only GenomicStudy mentions struck from each).

The ticket is also tagged **R6-Release-note** and **Ready_to_Apply**, so
the DiagnosticReport release-note list must include `FHIR-54164`.

## Feature Proposal

### Problem Statement

Earlier R6 work removed `GenomicStudy` as an allowed target of
`DiagnosticReport.study` (the element type now resolves to
`Reference(ImagingStudy)` only — see line 638 of
`structuredefinition-DiagnosticReport.xml`, which lists a single
`targetProfile` of `…/StructureDefinition/ImagingStudy`). However, the
element's authored `<definition>` and `<comment>` text still describe
`GenomicStudy`/`GenomeStudy` at length, telling implementers about
karyotyping/CNV/SNV analyses and "laboratory-type studies like
GenomeStudy". This contradicts the structural definition: a reader
following the prose would believe they can point `study` at a
`GenomicStudy`, only to discover the type binding rejects it.

The ticket fixes that prose-vs-binding mismatch ahead of the next R6
ballot publication.

### Proposed Change

Edit the `<element id="DiagnosticReport.study">` block in
`source/diagnosticreport/structuredefinition-DiagnosticReport.xml`
(`<differential>`, currently lines 629–640) so that:

1. The `<definition>` value drops the trailing GenomicStudy sentence and
   becomes:

   > One or more links to full details of any study performed during
   > the diagnostic investigation. An ImagingStudy might comprise a set
   > of radiologic images obtained via a procedure that are analyzed as
   > a group. Typically, this is imaging performed by DICOM enabled
   > modalities, but this is not required. A fully enabled PACS viewer
   > can use this information to provide views of the source images.

2. The `<comment>` value drops the leading "laboratory-type studies
   like GenomeStudy" sentence and becomes:

   > ImagingStudy and the media element are somewhat overlapping -
   > typically, the list of image references in the media element will
   > also be found in one of the imaging study resources. However, each
   > caters to different types of displays for different types of
   > purposes. Neither, either, or both may be provided.

   (Per the resolution table. See "Open Questions" below — the existing
   PR #4025 deletes the entire `<comment>` element instead, which
   matches the *original* ticket text but not the resolution table.)

3. `<short>`, `<min>`, `<max>`, and `<type>` (with the single
   `targetProfile` of `…/ImagingStudy`) are unchanged.

Add a release-note entry to
`source/diagnosticreport/diagnosticreport-introduction.xml`, inside the
existing `<ul>` of resolved tickets (currently ends at line 26 with
`FHIR-54163`):

```xml
<li><a href="https://jira.hl7.org/browse/FHIR-54164">FHIR-54164</a></li>
```

No example, search-parameter, terminology, mapping, or invariant
changes are required. The element's structural shape (cardinality,
type, target profile, isSummary) stays the same.

### Design Rationale

The change is a pure prose-to-structure reconciliation. The structural
binding (single `targetProfile = ImagingStudy`) is the source of truth;
the `<definition>` and `<comment>` must not advertise capabilities the
type system does not allow. Re-introducing GenomicStudy as a target was
not on the table — that decision was made (and applied) by an earlier
ticket, which is exactly why this cleanup ticket exists.

The committee chose the *modified* form (keep the longer Definition,
strike only the GenomicStudy sentences) over the reporter's original
proposal (replace Definition wholesale; remove Comment entirely)
because the surviving ImagingStudy/PACS/media-overlap guidance is still
useful to implementers and worth preserving.
