## Resolution Summary

Resolution: **Persuasive with Modification**.

The ticket flagged four CodeSystem/ValueSet pairs in BiologicallyDerivedProduct
as THO-migration candidates. The committee concluded:

1. **Two of the bindings are already being moved to external code systems** by
   other tickets and require no action here:
   - `CodeSystem/biologicallyderived-productcodes` /
     `ValueSet/biologicallyderived-productcodes` bound to
     `BiologicallyDerivedProduct.productCode` — handled by
     **FHIR-55000**, **FHIR-53477**, **FHIR-49979**.
   - `CodeSystem/biologicallyderived-product-property-type-codes` /
     `ValueSet/biologicallyderived-product-property-type-codes` bound to
     `BiologicallyDerivedProduct.property.type` — handled by
     **FHIR-55000**, **FHIR-53477**, **FHIR-49979**.

2. **The other two are example values without probable re-use elsewhere** in
   FHIR Core or related specifications. Per the modified resolution (committee
   vote David Barwin / Riki Merrick **11 – 0 – 1**, 2026-01-30), **both the
   CodeSystem and the ValueSet** in each pair must be marked
   `experimental="true"` to align with FHIR example-terminology rules
   (Lloyd McKenzie comment 2026-01-30: "If the code system is experimental, the
   value set should also be."):
   - `CodeSystem/product-category` (`BiologicallyDerivedProductCategory`) and
     `ValueSet/product-category`, bound to
     `BiologicallyDerivedProduct.productCategory`.
   - `CodeSystem/biologicallyderived-product-status`
     (`BiologicallyDerivedProductStatus`) and
     `ValueSet/biologicallyderived-product-status`, bound to
     `BiologicallyDerivedProduct.productStatus`.

No element-level changes to the `BiologicallyDerivedProduct` StructureDefinition
are required; the bindings, paths, and ValueSet URLs are unchanged.

## Feature Proposal

### Problem Statement

`BiologicallyDerivedProduct` carries four FHIR-Core CodeSystem/ValueSet pairs
that the THO-candidate review flagged as potentially needing migration to THO
or, alternatively, explicit annotation as example-only terminology. Today all
four resources have `experimental="false"`, which incorrectly signals to
implementers that the example-only `productCategory` and `productStatus`
codes are intended for production use. The other two bindings (`productCode`,
`property.type`) are already being routed to external code systems via
parallel tickets, but the example-only pairs were not yet correctly annotated.

### Proposed Change

Two distinct fixes — one already covered elsewhere, one to perform here.

**Out of scope for this ticket** (tracked separately and to be applied before
R6 is published):

- Re-binding `BiologicallyDerivedProduct.productCode` and
  `BiologicallyDerivedProduct.property.type` to external code systems
  (FHIR-55000, FHIR-53477, FHIR-49979).

**In scope for this ticket — flip `experimental` to `true` on four artifacts:**

| File | Element | Current | New |
|---|---|---|---|
| `source/biologicallyderivedproduct/codesystem-product-category.xml` | `<experimental>` (line 23) | `false` | `true` |
| `source/biologicallyderivedproduct/valueset-product-category.xml` | `<experimental>` (line 23) | `false` | `true` |
| `source/biologicallyderivedproduct/codesystem-biologicallyderived-product-status.xml` | `<experimental>` (line 26) | `false` | `true` |
| `source/biologicallyderivedproduct/valueset-biologicallyderived-product-status.xml` | `<experimental>` (line 23) | `false` | `true` |

No other element of these resources changes (URL, status, content, concepts,
bindings all remain as-is). The `BiologicallyDerivedProduct` StructureDefinition
itself is not edited — its example-strength bindings to
`http://hl7.org/fhir/ValueSet/product-category` (line 113) and
`http://hl7.org/fhir/ValueSet/biologicallyderived-product-status` (line 271)
are preserved verbatim.

### Design Rationale

- `experimental=true` is the standard FHIR signal for terminology that is
  **not intended for production use**. The committee's modified resolution
  treats these two CodeSystems as "example examples": codes provided to make
  the spec illustrative, not to be normatively adopted by implementers.
- Marking **both** the CodeSystem and the ValueSet experimental keeps the two
  artifacts internally consistent. Per Lloyd McKenzie's 2026-01-30 comment, it
  would be incoherent to publish a non-experimental ValueSet whose only
  content comes from an experimental CodeSystem.
- Migration to THO was considered and rejected for this pair on the grounds
  that the codes have no probable re-use in CDA/V2 or elsewhere in FHIR Core.
  Annotating them as experimental is the lighter-touch alternative that
  satisfies the THO-candidate review without scheduling a terminology move.
- The earlier "Not Persuasive" resolution (2026-01-29) was reverted on
  2026-01-30 specifically because it left the example-only pairs without any
  experimental marking; this plan implements the corrected disposition.
