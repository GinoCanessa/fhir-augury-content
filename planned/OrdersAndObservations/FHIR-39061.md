## Resolution Summary

Resolution: **Not Persuasive with Modification**.

The OO work group declined to add search parameters for every `value[x]`
datatype on `Observation` in this ticket. The approved modification is a
policy/process statement rather than a one-shot artifact change:

- "Prioritize the addition with search parameters that are needed based on
  use cases given that all value search parameters for each data type
  cannot be completed before R5."
- "Implementers can create their own search parameters if they need."
- "OO will prioritize the addition of search parameters as use cases are
  submitted and work to create them over time."

In other words, no bulk addition of `value-*` SearchParameters is approved
under this ticket. Future additions are gated on a submitted use case and
will be tracked via separate tickets. The only in-spec follow-up action
implied by the modification is to make the existing narrative reflect this
stance so readers do not expect a complete value-typed search-parameter
matrix.

## Feature Proposal

### Problem Statement

`Observation.value[x]` is a choice element with a wide type set
(`Quantity`, `CodeableConcept`, `string`, `boolean`, `integer`, `Range`,
`Ratio`, `SampledData`, `time`, `dateTime`, `Period`, `Attachment`,
`Reference`). The Observation search-parameter table only ships
SearchParameters for a subset of those types, namely `value-quantity`,
`value-concept`, `value-date`, `value-string` (plus the parallel
`code-value-*` and `combo-*` and `combo-code-value-*` variants). This is
verifiable in the clone:

- `cache/github/repos/HL7_fhir/clone/source/observation/bundle-Observation-search-params.xml`
  defines `Observation-code-value-concept`, `Observation-code-value-date`,
  `Observation-code-value-quantity`, `Observation-code-value-string`,
  `Observation-combo-code-value-concept`,
  `Observation-combo-code-value-quantity`,
  `Observation-combo-value-concept`, `Observation-combo-value-quantity`
  (lines 65–276). There is no `value-boolean`, `value-integer`,
  `value-range`, `value-ratio`, `value-sampleddata`, `value-time`,
  `value-period`, `value-attachment`, or `value-reference`.
- `cache/github/repos/HL7_fhir/clone/source/observation/observation-notes.xml`
  line 380 already acknowledges the gap in narrative form:
  "Because there are multiple types allowed for the *value* element,
  multiple value search parameters are defined. There is no standard
  parameter for searching values of type Ratio."

The reporter (Bas van den Heuvel, via Ana Kostadinovska) read the
incomplete coverage as an oversight in "the table" and asked for the
missing rows to be filled in.

### Proposed Change

Per the resolution, **no new SearchParameter resources are to be added in
this ticket**. The change is documentary/process:

1. Update the existing narrative in
   `source/observation/observation-notes.xml` (the bullet at line 380) so
   that it reflects the agreed OO stance:
   - Acknowledge that not every `value[x]` datatype has a registered
     standard search parameter (today the list is `value-quantity`,
     `value-concept`, `value-date`, `value-string`, plus their `combo-`
     and `code-value-` companions).
   - State that additional `value-*` search parameters will be added by
     OO over time, prioritised by submitted use cases, rather than
     completed exhaustively up front.
   - Note that implementers MAY define their own SearchParameters for
     `value[x]` datatypes that are not covered by the core specification
     (e.g., `value-boolean`, `value-integer`, `value-range`,
     `value-ratio`, `value-sampleddata`, `value-time`, `value-period`,
     `value-attachment`, `value-reference`), following the standard
     SearchParameter conformance rules.

   The current sentence ("There is no standard parameter for searching
   values of type Ratio") singles out one missing type, which is
   misleading — the proposed prose generalises it.

2. Optionally add a short row or sentence to the search-parameter
   narrative on the Observation page (if a "table" exists in
   `source/observation/observation.html` or its narrative siblings) that
   cross-links to the bullet above, so the reporter's "table" complaint
   is addressed in place. Verify against the clone before adding any
   table row — if no such table is rendered as authored content (the
   search-parameter rows are auto-generated by the publisher from
   `bundle-Observation-search-params.xml`), this sub-task is dropped and
   the entirety of the change lives in `observation-notes.xml`.

3. Do **not** add new `<entry>` blocks to
   `source/observation/bundle-Observation-search-params.xml`.
   New `value-*` SearchParameters require their own use-case-driven
   ticket per the resolution.

### Design Rationale

- The resolution is a deliberate scoping decision: OO acknowledged the
  gap but rejected the bulk-fill approach. Encoding that decision in
  prose prevents the same complaint from being re-filed and gives
  implementers a clear, sanctioned escape hatch (custom
  SearchParameters).
- Touching only the narrative respects the Pending-SME-Input label —
  individual `value-*` parameter definitions need clinical/SME input on
  modifier suffixes, target types (especially `Reference`), and Range vs
  Quantity range semantics that are out of scope here.
- Generalising the existing one-off Ratio sentence avoids implying that
  Ratio is uniquely missing while several others are not.
