## Resolution Summary

**Resolution:** Persuasive with Modification.

The committee approved the addition of an **example binding** to a **new
FHIR ValueSet** that will cover both `DeviceDefinition.guideline.contraindication`
(this ticket) **and** `DeviceDefinition.guideline.indication` (sister ticket
FHIR-54863). The two elements must share the same ValueSet.

The new ValueSet is to be backed by SNOMED CT, composed of concepts
selected via filters anchored at:

- `64572001 | Disease (disorder) |`
- `404684003 | Clinical finding (finding) |`

Per the applied-vote comment by JIRAUSER19407 on 2026‑03‑30:
"FHIR-54864 and FHIR-54863 should share the same valueset." An earlier
comment (2026‑02‑27) from the same author proposed reusing
`http://hl7.org/fhir/ValueSet/condition-code`, but the final motion
diverges from that and creates a new, narrower ValueSet limited to the
two SNOMED hierarchies above. The implementation must follow the motion,
not the earlier comment.

## Feature Proposal

### Problem Statement

`DeviceDefinition.guideline.contraindication` is a `0..*` `CodeableConcept`
element used to describe situations in which a device should not be used
because it may cause harm. The current `structuredefinition-DeviceDefinition.xml`
defines the element with no `<binding>` at all (lines 1403–1413 of
`source/devicedefinition/structuredefinition-DeviceDefinition.xml`), so
implementers have no curated set of codes — even at example strength —
to draw on. The same gap exists on the sibling element
`DeviceDefinition.guideline.indication` (FHIR-54863).

By contrast, the neighbouring `DeviceDefinition.guideline.warning`
element already has an example binding to
`http://hl7.org/fhir/ValueSet/devicedefinition-warnings-example`,
demonstrating the established pattern within the same resource for
giving CodeableConcept elements at least an example binding.

### Proposed Change

Add an **example-strength** binding on both
`DeviceDefinition.guideline.contraindication` and (per the shared-valueset
motion) `DeviceDefinition.guideline.indication` to a new SNOMED-CT-backed
ValueSet that covers diseases and clinical findings.

Concrete artifacts:

1. **New ValueSet** (file: `source/devicedefinition/valueset-devicedefinition-guideline-condition.xml`)
   - `id`: `devicedefinition-guideline-condition`
   - `url`: `http://hl7.org/fhir/ValueSet/devicedefinition-guideline-condition`
   - `name`: `DeviceDefinitionGuidelineCondition`
   - `title`: `DeviceDefinition Guideline Condition Codes`
   - `status`: `active`; `experimental`: `true`
   - `publisher`: `HL7 International`
   - WG extension `structuredefinition-wg` = `oo`; standards-status =
     `informative` (consistent with `valueset-devicedefinition-warnings-example.xml`).
   - `description`: "Codes describing clinical conditions or findings
     that are indications for, or contraindications to, the use of a
     device. Drawn from SNOMED CT diseases (disorder) and clinical
     findings (finding)."
   - `copyright`: standard SNOMED CT affiliate-license boilerplate
     (copy verbatim from `source/condition/valueset-condition-code.xml`).
   - `compose.include` (two separate `include` blocks, one per
     hierarchy named in the resolution):
     ```xml
     <compose>
       <include>
         <system value="http://snomed.info/sct"/>
         <filter>
           <property value="concept"/>
           <op value="is-a"/>
           <value value="64572001"/>
         </filter>
       </include>
       <include>
         <system value="http://snomed.info/sct"/>
         <filter>
           <property value="concept"/>
           <op value="is-a"/>
           <value value="404684003"/>
         </filter>
       </include>
     </compose>
     ```
   - Note: `64572001 | Disease (disorder) |` is a descendant of
     `404684003 | Clinical finding (finding) |` in SNOMED CT, so the
     second filter strictly subsumes the first. Both are included as
     separate filters because the resolution explicitly enumerates both
     anchor concepts. See "Open Questions" below.

2. **Edit `source/devicedefinition/structuredefinition-DeviceDefinition.xml`**
   - On the `DeviceDefinition.guideline.contraindication` element
     (lines 1403–1413), add a `<binding>` block immediately after the
     `<type>` block (mirroring the existing pattern used by
     `DeviceDefinition.guideline.warning` at lines 1424–1431):
     ```xml
     <binding>
       <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
         <valueString value="DeviceDefinitionGuidelineCondition"/>
       </extension>
       <strength value="example"/>
       <description value="Clinical conditions or findings that may indicate or contraindicate use of a device."/>
       <valueSet value="http://hl7.org/fhir/ValueSet/devicedefinition-guideline-condition"/>
     </binding>
     ```
   - Apply the **same** `<binding>` block to
     `DeviceDefinition.guideline.indication` (lines 1392–1402). This is
     the FHIR-54863 change; it is included here because the resolution
     mandates a shared valueset and the two tickets are tied together
     by the applied-vote comment.

No changes are required to `source/bindings.ini` by hand — the file
header states: "This file is maintained by the tooling program. Never
edit it by hand." The new binding name will be assigned by the
publisher tooling on the next build.

### Design Rationale

- **Example strength** matches the resolution wording ("add an example
  binding") and is consistent with the only other coded element in
  `DeviceDefinition.guideline` (`.warning`), which is also bound at
  example strength.
- **Single shared ValueSet** for `.indication` and `.contraindication`
  follows the explicit applied-vote comment ("should share the same
  valueset") and avoids two near-identical SNOMED filter ValueSets.
  Indication and contraindication describe the same conceptual space
  (clinical conditions/findings); the polarity is conveyed by the
  element name, not the code.
- **SNOMED filter rather than reusing `condition-code`.** The earlier
  comment proposing `http://hl7.org/fhir/ValueSet/condition-code` was
  superseded by the final motion. `condition-code` includes
  `404684003` plus an additional standalone code
  (`160245001 | No current problems or disability`) — that extra code
  is appropriate for `Condition.code` but not for indications or
  contraindications of a device, so the new ValueSet is a cleaner fit.
- **Two filters (disorder + finding) rather than just finding.**
  Although `Disease (disorder)` is a subtype of `Clinical finding
  (finding)`, the motion enumerates both. Encoding both filters keeps
  the ValueSet definition faithful to the resolution-of-record and
  makes the curators' intent explicit. (See Open Questions.)
- **File location.** Resource-local ValueSets live alongside the
  `StructureDefinition` under `source/<resource>/` and are picked up by
  the publisher without an entry in `source/fhir.ini`'s `[valuesets]`
  section (which is reserved for cross-resource shared ValueSets).
  `source/devicedefinition/valueset-devicedefinition-warnings-example.xml`
  is the local precedent.

No breaking changes. The change introduces an **example-strength**
binding on elements that were previously unbound. Example bindings are

ValueSet — so existing instances and conforming implementations remain
valid. No element types, cardinalities, or required bindings are
altered.

### Terminology Impact

- **No changes to HL7/UTG.** SNOMED CT is an external code system;
  HL7 does not maintain its concepts. The two anchor codes
  (`64572001`, `404684003`) are pre-existing SNOMED concepts.
- **New ValueSet** `http://hl7.org/fhir/ValueSet/devicedefinition-guideline-condition`

  SNOMED-CT copyright boilerplate (as used in
  `source/condition/valueset-condition-code.xml`) must be included
  on the new ValueSet.
- Because the binding strength is *example*, the SNOMED affiliate
  licensing requirement is non-blocking for implementers, but the
  copyright statement remains mandatory on the resource itself.

### Open Questions

- **Both filters vs. one?** SNOMED CT's `64572001 | Disease (disorder) |`
  is a subtype of `404684003 | Clinical finding (finding) |`, so the
  `is-a 404684003` filter alone would already include all of disease.
  The motion enumerates both anchors; this plan encodes both as
  separate `include` filters to honour the wording verbatim. If the
  reviewer prefers the simpler single-filter form, drop the
  `64572001` include.
- **Terminology-binding caveat (briefing gotcha — irrelevant here).**
  The briefing flags `<snapshot>` regeneration and the
  hand-edit-only-`<differential>` rule; both are addressed by Tasks
  2–4. No other gotchas from the briefing apply.
- **FHIR-54863 ownership.** Tasks 1 and 3 implicitly close the sister
  ticket. Confirm with the work group whether that ticket should be
  resolved by the same commit or kept open with a "implemented under
  FHIR-54864" note.
- **Optional second `include`** of the standalone code
  `160245001 | No current problems or disability |` (as
  `condition-code` does) is **not** included here — it does not fit
  the indication/contraindication semantics. Flagged in case the
  reviewer expects parity with `condition-code`.
