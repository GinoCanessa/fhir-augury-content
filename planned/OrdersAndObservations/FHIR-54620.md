## Resolution Summary

The resolution is **Persuasive**. Resolution description (verbatim):

> ServiceRequest.focus will be marked as a modifier

Concretely, the `ServiceRequest.focus` element must become a modifier element with the same rationale already used on `Observation.focus` and `Procedure.focus`. The reporter explicitly proposed the reason text:

> "true (Reason: This element is labeled as a modifier because it changes who is the target of the request. See the use cases in the definition.)"

## Feature Proposal

### Problem Statement

`ServiceRequest.focus` carries the same semantics as `Observation.focus` and `Procedure.focus`: it identifies the thing the request is *about* when that is not the subject of record (e.g., a fetus, a spouse, a body structure, another request). On Observation and Procedure, `focus` is already flagged as `isModifier=true` because it changes who/what the record is really about — a consumer that ignores it will misinterpret the data.

On ServiceRequest, `focus` is currently **not** marked as a modifier (see `cache/github/repos/HL7_fhir/clone/source/servicerequest/structuredefinition-ServiceRequest.xml` lines 734–753: the element has `<isSummary value="true"/>` but no `<isModifier>` / `<isModifierReason>`). This is an inconsistency in the R6 draft that the ticket asks to correct.

### Proposed Change

Make a single, targeted change to the ServiceRequest StructureDefinition:

- **Element:** `ServiceRequest.focus` (id `ServiceRequest.focus`, path `ServiceRequest.focus`).
- **Add child element:** `<isModifier value="true"/>`.
- **Add child element:** `<isModifierReason value="This element is labeled as a modifier because it changes who is the target of the request. See the use cases in the definition."/>`.
- **Placement:** immediately after the element's `<type>` block and before the existing `<isSummary value="true"/>`, matching the ordering used on `Observation.focus` (see `source/observation/structuredefinition-Observation.xml` around the `Observation.focus` element, where `isModifier` / `isModifierReason` precede `isSummary`).
- **Cardinality / type / binding / summary flag:** unchanged (`0..*`, `Reference(Resource)`, `isSummary=true`).

Resulting differential fragment (illustrative, matching sibling style):

```xml
<element id="ServiceRequest.focus">
  <path value="ServiceRequest.focus"/>
  <short value="What the service request is about, when it is not about the subject of record"/>
  <definition value="The actual focus of a service request when it is not the subject of record ..."/>
  <min value="0"/>
  <max value="*"/>
  <type>
    <code value="Reference"/>
    <targetProfile value="http://hl7.org/fhir/StructureDefinition/Resource"/>
  </type>
  <isModifier value="true"/>
  <isModifierReason value="This element is labeled as a modifier because it changes who is the target of the request. See the use cases in the definition."/>
  <isSummary value="true"/>
  <mapping>
    <identity value="w5"/>
    <map value="FiveWs.subject"/>
  </mapping>
  ...
</element>
```

No other files in `source/servicerequest/` need content changes: short/definition, mappings, examples, and search parameters remain correct. The `<snapshot>` block is absent from this file (differential-only; `<differential>` opens at line 80 and the file has no `<snapshot>` element), so there is no hand-edited snapshot to keep in sync — the publisher regenerates it.

### Design Rationale

- **Consistency with siblings.** `Observation.focus` and `Procedure.focus` are already modifier elements with essentially the same reason text; aligning ServiceRequest removes a cross-resource inconsistency that the reporter explicitly flagged.
- **Correctness.** `focus` redirects the clinical interpretation from the subject of record to a different entity (fetus, caregiver, body structure, another request). A consumer that silently drops or ignores `focus` would attribute the request to the wrong party — the textbook case for `isModifier=true`.
- **Reason text choice.** The wording proposed in the ticket ("changes who is the target of the request. See the use cases in the definition.") is a near-verbatim parallel of the Observation reason ("changes who is the target of the observation. See the use cases in the definition.") already present in `source/observation/structuredefinition-Observation.xml` — keeping them parallel was the resolution committee's clear intent ("same logic as the other resources").
- **Alternatives considered.** Adding a constraint/invariant instead of `isModifier` was not proposed and would not convey the same consumer-processing obligation. Editing only the narrative (`servicerequest-notes.xml` / `servicerequest-introduction.xml`) would not affect conformance; the resolution specifically asks for the modifier flag.

### Breaking Changes

Yes — this is a **conformance-relevant breaking change**, consistent with the `R6-Release-note` label on the ticket:

- Flipping an element from non-modifier to `isModifier=true` changes consumer obligations. A system that previously ignored unknown modifiers was allowed to drop `ServiceRequest.focus`; once the element is `isModifier=true`, a processor that does not understand it must refuse to process the resource (per the FHIR modifier-element rules). Existing instances are unchanged syntactically, but consumers' *handling* obligations change.
- Existing ServiceRequest examples and instances in the repo that populate `focus` (e.g., `source/servicerequest/servicerequest-example-*.xml`) remain valid; no example updates are required by the resolution itself.
- No cardinality, type, or binding change — so instance XML/JSON schemas remain compatible.

This warrants an entry in the R6 release notes (already anticipated by the `R6-Release-note` label).
