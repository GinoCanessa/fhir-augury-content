## Resolution Summary

Resolution: **Persuasive with Modification**.

Rather than adding a new `Observation.context` element or widening the
cardinality of `Observation.method` to `0..*`, the working group resolved to
clarify — via editorial changes only — that the existing
`Observation.interpretationContext` element (already `0..*`,
`CodeableReference(Any)`) is the correct home for the contextual details
described in the ticket (patient standing, fasting, cardiac phase, imaging
view, sensor type, averaging/adjustment of the value, etc.).

The exact agreed changes (per the resolution description) apply to the
`Observation.interpretationContext` element definition:

- **Element Id:** `Observation.interpretationContext` (unchanged — the
  reporter had asked to rename to `Observation.context`; the committee
  declined renaming a mature element).
- **Short Display:** `Context for understanding the observation` (unchanged
  from current source; reaffirmed).
- **Definition (new):** *"Contextual information that is relevant to
  understanding the observation value or interpretation. The referenced
  information may be concurrent or precede the observation."*
  (replacing the current *"Other preceding or concurrent information that
  is critical to understand the context and significance of the
  observation."*)
- **Comments (appended sentence):** append
  *"Similarly, information about the timing, view, or imaging mode may be
  relevant to understanding an imaging measurement."* to the existing
  comments paragraph (existing text retained).

No cardinality, type, binding, or invariant changes are approved by this
resolution.

## Feature Proposal

### Problem Statement

The ticket (filed by Kevin O'Donnell) observes that details about the
context in which an Observation is produced — e.g., *patient standing*,
*patient fasting*, *breath-held*, *apical 4-chamber view*, *Doppler mode*,
*value averaged over three readings*, *value normalised for body weight* —
have been treated as belonging in `Observation.method`, which is `0..1`.
Because many observations require several such contextual qualifiers
simultaneously, users have been forced either to pre-coordinate all
combinations into custom codes (combinatorial explosion, poor
searchability) or to misuse `method` by overloading it.

During OO discussion on 2026-03-20 and in the subsequent comment thread,
Lorraine pointed the reporter at the existing
`Observation.interpretationContext` element
(`0..*`, `CodeableReference(Any)`), which is already structurally
sufficient — the problem is that its name and its definition are framed
narrowly around *interpretation*, not around the observation value itself,
causing implementers (and the original reporter) to overlook it for the
imaging/measurement use cases listed in the ticket.

### Proposed Change

A single-repository, editorial change to the element definition of
`Observation.interpretationContext` in the Observation resource
StructureDefinition, in `HL7/fhir`.

In `source/observation/structuredefinition-Observation.xml`, on the
`<element id="Observation.interpretationContext">` element:

1. Replace the `<definition value="…"/>` attribute value from

   > Other preceding or concurrent information that is critical to
   > understand the context and significance of the observation.

   to

   > Contextual information that is relevant to understanding the
   > observation value or interpretation. The referenced information may
   > be concurrent or precede the observation.

2. Extend the `<comment value="…"/>` attribute value by appending — at
   the end of the existing sentence — the new sentence:

   > Similarly, information about the timing, view, or imaging mode may
   > be relevant to understanding an imaging measurement.

   so the full comment reads:

   > This element provides essential contextual information that affects
   > how the observation should be interpreted. For example, knowing that
   > a patient is diabetic, fasting, or paraplegic may be critical to
   > properly understanding lab results or other observations. Similarly,
   > information about the timing, view, or imaging mode may be relevant
   > to understanding an imaging measurement.

3. **Explicitly NOT changed** by this ticket:
   - `<short value="Context for understanding the observation"/>` —
     already matches the resolution text; leave as-is.
   - `<min value="0"/>` / `<max value="*"/>` — already `0..*`.
   - `<type><code value="CodeableReference"/> …` — unchanged.
   - `<binding>` / ValueSet `observation-interpretation-context` —
     unchanged (strength remains `example`).
   - Element id/path — **not** renamed (committee declined; see comment
     from JIRAUSER19407, 2026-04-08).
   - `Observation.method` — cardinality **not** widened; no change.
   - No new `Observation.context` element is added.

### Design Rationale

- **Reuse over new element.** `Observation.interpretationContext` already
  provides the `0..*` `CodeableReference(Any)` shape the reporter asked
  for. Adding a parallel `Observation.context` would create overlap and
  force implementers to choose between near-identical slots; the OO
  committee explicitly preferred reuse.
- **No rename of a mature element.** Renaming
  `interpretationContext` → `context` would be a breaking wire-level
  change for every implementer already using R5/R6 drafts. The committee
  chose a narrower, editorial fix (JIRAUSER19407 comment, 2026-04-08:
  *"The element name is unlikely to be renamed as Observation is quite
  mature but definitional/editorial content can be adjusted as long as
  original uses remain valid."*). The reporter accepted this
  (JIRAUSER28753 comment, 2026-04-08: *"the definition & comment changes
  would meet our needs and retains all prior usage."*).
- **Definition broadening.** The new definition swaps "critical to
  understand the context and significance of the observation" for
  "relevant to understanding the observation value or interpretation",
  so it clearly covers both the `value[x]` branch (imaging/physiological
  context) and the `interpretation` branch (patient clinical context).
  This matches the ticket's examples, which mix both categories
  (standing/fasting vs. cardiac phase/imaging view/mode).
- **Comment extension, not replacement.** Existing deployments may cite
  the lab-oriented example sentence; keeping it and appending an imaging
  sentence preserves backward usability of the normative commentary.

## Implementation Plan

### Prerequisites

None. This change is self-contained within `HL7/fhir` and does not
depend on any other ticket, ValueSet, or extension.

### Step-by-Step Tasks

#### HL7/fhir

1. **Update the `<definition>` of `Observation.interpretationContext`.**
   - File: `source/observation/structuredefinition-Observation.xml`
   - Action: On the `<element id="Observation.interpretationContext">`
     block (currently at lines ~820–843 of the differential), replace
     the value of the `<definition value="…"/>` attribute from
     `Other preceding or concurrent information that is critical to
     understand the context and significance of the observation.`
     to
     `Contextual information that is relevant to understanding the
     observation value or interpretation.  The referenced information
     may be concurrent or precede the observation.`
   - Details: Editorial. Only this one `<definition>` attribute value
     changes on this element. Do not touch `<short>`, `<min>`, `<max>`,
     `<type>`, `<binding>`, or `<mapping>` on this element.

2. **Extend the `<comment>` of `Observation.interpretationContext`.**
   - File: `source/observation/structuredefinition-Observation.xml`
   - Action: On the same `<element id="Observation.interpretationContext">`
     block, append the sentence
     ` Similarly, information about the timing, view, or imaging mode
     may be relevant to understanding an imaging measurement.`
     to the existing `<comment value="…"/>` attribute. Resulting full
     value:
     `This element provides essential contextual information that
     affects how the observation should be interpreted. For example,
     knowing that a patient is diabetic, fasting, or paraplegic may be
     critical to properly understanding lab results or other
     observations. Similarly, information about the timing, view, or
     imaging mode may be relevant to understanding an imaging
     measurement.`
   - Details: Editorial. Preserve the existing sentences verbatim and
     only append the new sentence.

3. **Scan narrative pages for text that narrows
   `interpretationContext` to "interpretation only" and reconcile.**
   - File: `source/observation/observation-introduction.xml`
   - File: `source/observation/observation-notes.xml`
   - Action: Read each file and search for any reference to
     `interpretationContext` (or prose framing context as relevant only
     to `Observation.interpretation`). If any such sentence constrains
     the element to lab/clinical interpretation only, adjust the prose
     to match the broadened definition (covers both `value[x]` and
     `interpretation`). If no such sentence exists, leave the file
     unchanged — this task may be a no-op.
   - Details: The bespoke HL7 publisher authors `source/*.html` and
     `source/<resource>/*.xml` narrative blocks directly; edits go
     inline. Do not look for `sushi-config.yaml`, `input/`, or
     `fsh-generated/` — none exist (per briefing).

4. **Do NOT edit the `<snapshot>` block of the Observation
   StructureDefinition.**
   - File: `source/observation/structuredefinition-Observation.xml`
   - Action: Leave any `<snapshot>` content untouched; the HL7
     publisher regenerates it from the differential on the next
     publish run.
   - Details: FhirCore briefing warning — only `<differential>` content
     in StructureDefinitions is hand-edited.

5. **Do NOT add or widen any other element.**
   - File: `source/observation/structuredefinition-Observation.xml`
   - Action: Explicitly do **not** add an `Observation.context`
     element, do **not** change `Observation.method` cardinality from
     `0..1` to `0..*`, and do **not** rename
     `Observation.interpretationContext`. These were all considered and
     rejected by the committee; the resolution is confined to the two
     editorial changes above.
   - Details: Guard rail — ensures the implementer does not over-apply
     the ticket's *description* (which asked for bigger changes) at the
     expense of the ticket's *resolution*.
