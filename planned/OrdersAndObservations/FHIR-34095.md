## Resolution Summary

Resolution: **Persuasive with Modification**.

Map InventoryReport's existing elements onto the Workflow Event pattern,
adding the following six mappings (per the resolution table):

| InventoryReport element | Event Pattern element |
|-|-|
| `InventoryReport.status` | `Event.status` |
| `InventoryReport.item.item` (i.e. `InventoryReport.inventoryListing.item.item`) | `Event.product` |
| `InventoryReport.reportedDateTime` | `Event.occurrenceDateTime` |
| `InventoryReport.reportingPeriod` | `Event.occurrencePeriod` |
| `InventoryReport.reporter` | `Event.reportedReference` |
| `InventoryReport.inventoryListing.location` | `Event.location` |

The follow-up comment from marti_velezis records that the remaining Event
pattern elements (`basedOn`, `partOf`, `researchStudy`, `statusReason`,
`category`, `code`, `subject`, `encounter`, `performer`, `performer.function`,
`performer.actor`, `reason`, `relevantHistory`) were reviewed and have **no
use case** in InventoryReport at the time of review, so they remain
unmapped (and continue to be declared as such in
`inventoryreport-event-mapping-exceptions.xml`).

## Feature Proposal

### Problem Statement

`InventoryReport` represents an action that has been (or is being) performed
— recording the state of inventory at a point in time. Per the FHIR Workflow
Pattern (the Event variant), every "event-like" resource is expected to
participate in the cross-resource Event mapping so that consumers can reason
about it uniformly with other event resources (Procedure, Observation,
MedicationAdministration, etc.).

At clone HEAD `db79dcdfe1`, `InventoryReport`'s
`StructureDefinition` only declares Workflow/Event mappings on three nodes:

- `InventoryReport` → `Event` (resource-level)
- `InventoryReport.identifier` → `Event.identifier`
- `InventoryReport.status` → `Event.status`

All other elements that obviously correspond to Event-pattern slots
(`reportedDateTime`, `reportingPeriod`, `reporter`,
`inventoryListing.location`, `inventoryListing.item.item`) carry **no**
`workflow` mapping at all, and `inventoryreport-event-mapping-exceptions.xml`
still flags `Event.location`, `Event.product`, `Event.reported`, and
`Event.occurrence` as `<unmappedElement>`s. That is the gap the resolution
closes.

### Proposed Change

Two coordinated edits inside `source/inventoryreport/`:

1. **Add element-level Event-pattern mappings** to
   `structuredefinition-InventoryReport.xml`. For each of the five elements
   listed below, add a `<mapping>` child (immediately after the existing
   element body, in the same position other resources use for workflow
   mappings — i.e. after `<isSummary>` / after the `<type>` block, alongside
   any existing `w5` / `rim` / `dicom` mappings). The Event pattern is
   identified in this SD by `<identity value="workflow"/>`
   (uri `http://hl7.org/fhir/workflow`, name "Workflow Pattern"); no new
   `<mapping>` declaration at the top of the SD is required.

   | Element id | New mapping to add |
   |-|-|
   | `InventoryReport.reportedDateTime` | `<mapping><identity value="workflow"/><map value="Event.occurrenceDateTime"/></mapping>` |
   | `InventoryReport.reportingPeriod` | `<mapping><identity value="workflow"/><map value="Event.occurrencePeriod"/></mapping>` |
   | `InventoryReport.reporter` | `<mapping><identity value="workflow"/><map value="Event.reportedReference"/></mapping>` |
   | `InventoryReport.inventoryListing.location` | `<mapping><identity value="workflow"/><map value="Event.location"/></mapping>` |
   | `InventoryReport.inventoryListing.item.item` | `<mapping><identity value="workflow"/><map value="Event.product"/></mapping>` |

   No mapping is added for `InventoryReport.status` — it already has
   `<mapping><identity value="workflow"/><map value="Event.status"/></mapping>`
   at clone HEAD.

2. **Update `inventoryreport-event-mapping-exceptions.xml`** so it no
   longer claims those slots are unmapped. Remove the following four
   `<unmappedElement>` lines:

   - `<unmappedElement patternPath="Event.location" reason="Unknown"/>`
   - `<unmappedElement patternPath="Event.product" reason="Unknown"/>`
   - `<unmappedElement patternPath="Event.reported" reason="Unknown"/>`
   - `<unmappedElement patternPath="Event.occurrence" reason="Unknown"/>`

   Leave the remaining `<unmappedElement>` rows in place (`Event.partOf`,
   `Event.reason`, `Event.relevantHistory`, `Event.code`, `Event.subject`,
   `Event.statusReason`, `Event.performer.actor`,
   `Event.performer.function`, `Event.note`, `Event.category`,
   `Event.encounter`, `Event.recorded`, `Event.performer`,
   `Event.researchStudy`) — these match the comment-table elements
   confirmed as having no use case. The two existing `<divergentElement>`
   blocks (`Event.identifier`, `Event.status`) remain unchanged: the
   mappings exist, but the short/definition text legitimately diverges
   from the pattern.

### Design Rationale

- **The resolution table is authoritative.** It explicitly lists
  source→target paths; the implementation simply transcribes them onto
  the existing Workflow-pattern mapping (`identity="workflow"`) already
  declared at the top of `structuredefinition-InventoryReport.xml`.
- **Choice elements map to typed slices.** The Event pattern uses
  `Event.occurrence[x]` (choice of dateTime/Period) and
  `Event.reported[x]` (choice of dateTime/Reference). Because
  `InventoryReport` already discriminates these as separate typed
  elements (`reportedDateTime`, `reportingPeriod`, `reporter` typed as
  `Reference`), the mappings target the typed forms
  (`Event.occurrenceDateTime`, `Event.occurrencePeriod`,
  `Event.reportedReference`) exactly as the resolution table dictates.
  This is consistent with how other resources map to typed Event choices
  (e.g., DiagnosticReport, Observation).
- **Unmapped-by-design elements are preserved as exceptions.** The
  comment from marti_velezis enumerates which Event slots have "no use
  case" in InventoryReport. Keeping their `<unmappedElement>` rows in
  the exceptions file is what the patterns tooling expects — silent
  omission would surface as a build warning from `mappingExceptions.xsd`.
- **No spec-page or example changes are required.** The mappings tab is
  generated by the publisher from the SD, so no edits to
  `inventoryreport-introduction.xml`, `inventoryreport-notes.xml`, or
  example instances are needed for the mappings to render.

### Open Questions

- The Event pattern uses choice elements `Event.occurrence[x]` and
  `Event.reported[x]`. The resolution table specifies the typed choice
  forms (`Event.occurrenceDateTime`, `Event.occurrencePeriod`,
  `Event.reportedReference`), and the existing exceptions file used the
  un-typed names (`Event.occurrence`, `Event.reported`) for the
  `<unmappedElement>` rows. The plan removes those un-typed exception
  rows entirely on the assumption that satisfying any typed slice of a
  choice satisfies the pattern slot. If publisher tooling instead
  expects every untyped choice to remain explicitly listed when not
  *all* typed forms are mapped, the row(s) may need to stay; verify on
  first publisher run and re-add if QA complains.
- The briefing's "Recommended Change Recipes" section is scoped to
  FHIR-42980 (a documentation edit) and does not enumerate a recipe for
  SD-mapping additions. The plan follows the generic FhirCore rule
  ("only `<differential>` content is hand-edited") and the precedent set
  by other Event-conformant resources. If a more specific recipe is
  preferred, a refreshed `repo-analysis` run scoped to FHIR-34095 would
  surface it.
