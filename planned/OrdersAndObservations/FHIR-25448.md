## Resolution Summary

The resolution is **Persuasive**. From the resolution description:

> Motion to change the definition of `online` to "The device is online." as an errata.

The single, narrowly-scoped change is to fix the `definition` text of the
`online` concept in the `device-status-reason` CodeSystem. Today the
definition is the verbatim string used for the `off` concept ("The device
is off."), which is clearly a copy/paste error. The resolution replaces
that string with "The device is online." and tags the change as an R4
errata (per the `R4-Errata` label and the 2021-08-23 comment by
hbuitendijk: "We discussed that this should be queued into an R4 errata
queue"). No structural or cardinality changes are required, no other
concepts are affected, and no display/code changes are made.

## Feature Proposal

### Problem Statement

The R4 publication of the `device-status-reason` CodeSystem
(`http://terminology.hl7.org/CodeSystem/device-status-reason`) lists the
concept `online` with display "Online" but with a definition that
incorrectly reads "The device is off." — the same definition used for
the `off` concept. This is misleading to implementers reading the
generated `codesystem-device-status-reason.html`, and it causes any tool
that surfaces the `definition` element (terminology servers, generated
documentation, downstream IGs binding to this ValueSet) to show wrong
text. The current build of the file (now maintained in HL7/UTG) still
carries the bad definition (line 45 of
`input/sourceOfTruth/fhir/codeSystems/CodeSystem-device-status-reason.xml`),
so this is also live in the next THO release, not just R4.

### Proposed Change

Edit the `online` concept in the `device-status-reason` CodeSystem so
that its `definition` matches the concept it is documenting.

**Single targeted edit** in
`HL7/UTG`:`input/sourceOfTruth/fhir/codeSystems/CodeSystem-device-status-reason.xml`,
inside the `<concept>` whose `<code value="online"/>`:

- Before:
  ```xml
  <concept>
    <code value="online"/>
    <display value="Online"/>
    <definition value="The device is off."/>
  </concept>
  ```
- After:
  ```xml
  <concept>
    <code value="online"/>
    <display value="Online"/>
    <definition value="The device is online."/>
  </concept>
  ```

Nothing else in the file changes. `code`, `display`, ordering of
concepts, `caseSensitive`, `content`, and the bound ValueSet
(`http://terminology.hl7.org/ValueSet/device-status-reason`) are all
left untouched.

Since the ticket carries the `R4-Errata` label, the same correction also
needs to be applied to whatever R4 publication artifact HL7 uses for
errata (the published R4 site at
`http://hl7.org/fhir/R4/codesystem-device-status-reason.html` is the
target). The `Unable-to-Apply` label on this ticket reflects that the
HL7/fhir current-build clone does **not** carry the R4-era source of
this CodeSystem (it has been migrated out of `HL7/fhir` to
`HL7/UTG` — `Get-ChildItem` over `cache/github/repos/HL7_fhir/clone/source`
returns no `device-status-reason*` files; only `codesystem-device-status.xml`
and `valueset-device-status.xml` for the parent Device.status concept
exist there). The errata application therefore happens in the R4 errata
process governed by FMG, not by editing a file in `HL7/fhir`.

### Design Rationale

- **Minimal, targeted fix.** The reporter and the resolution agree
  this is a copy/paste defect, not a semantic redesign. The smallest
  possible textual change ("The device is off." → "The device is
  online.") restores the intent without disturbing any consumers.
- **Mirrors the pattern of sibling concepts.** Other concepts in the
  same CodeSystem follow the same template "The device is `<state>`."
  (`paused` → "The device is paused.", `offline` → "The device is
  offline.", `not-ready` → "The device is not ready.", `off` → "The
  device is off."). Using "The device is online." keeps `online`
  consistent with that pattern.
- **Errata vs. new normative cycle.** FMG/OO already discussed
  (2021-08-23 comment) routing this to the R4 errata queue rather than
  waiting for a new normative ballot, because the change is a defect
  correction with no breaking impact.
- **Scope deliberately narrow.** Comment 55351 raises a broader
  redesign (statusReason becoming a documentation-only status with new
  reason codes, alongside `operationalStatus` /
  `associationStatus` work tracked under FHIR-24696). That is a
  separate, larger workstream and is explicitly **not** in scope for
  this ticket — this ticket only fixes the misleading definition.
