## What a future plan will need to cover (for context only — not a plan)

Recorded here so orchestrators can see why those repos are needed; **do not
treat as authoritative file paths** — they must be verified against the cloned
repos once briefings exist.

- **`HL7/fhir-extensions`**: add a new complex `Extension` StructureDefinition
  `observation-supportingDevice` with:
  - `context`: element `Observation`
  - `extension:role` — 0..1, `value[x]` = CodeableConcept (1..),
    short: "Code indicating the role of the supporting device".
    Bound to a new ValueSet to be sourced from
    - https://hl7.org/fhir/uv/livd/2024Jan/ValueSet-livd-device-type.html
      (at minimum `instrument-platform` and `test-kit`), and/or
    - https://usnistgov.github.io/v2plusDemo/vocab/0384-Substance-Type.html
    (binding strength: **example** or **extensible** per ralf.herzog's
    resolution comment — resolve during planning).
  - `extension:device` — 1..1, `value[x]` = Reference(Device) (1..),
    short: "A reference to the supporting device".
  - `Extension.url` = `http://hl7.org/fhir/StructureDefinition/observation-supportingDevice`
    (the example-org URL in the ticket snapshot is a placeholder).
  - `Extension.value[x]` max = 0 (complex extension).
  - Description text per the resolution (including the "not a substitute for
    Observation.subject / Observation.focus / Observation.device" guidance and
    the `[[[Procedure]]]` cross-link — note Marti's 2025-10-14 comment asking
    to "ensure this [[[Procedure]]] resolves correctly in the description").
- **ValueSet / CodeSystem work**: the new `SupportingDeviceRole` (working
  name) ValueSet must be created; whether it lives in `HL7/fhir-extensions`
  or `HL7/UTG` depends on the extension pack's conventions and needs a
  briefing to confirm.
- **`HL7/fhir`** (only if the briefing surfaces required prose updates):
  possibly a back-link from `source/observation/` narrative noting the new
  extension; the resolution does not mandate this.
