## Feature Proposal

### Problem Statement

A pool of long-lived, OO-owned change requests against `Observation`,
`Device`, `DeviceAssociation`, `DeviceMetric`, `DeviceDefinition`,
`DiagnosticReport`, `InventoryItem`, `InventoryReport`, and related profiles
had been deferred over multiple ballot cycles. With the FHIR Core normative
publication on the horizon, OO needs every deferred OO-scoped ticket to
either be applied or formally re-dispositioned (Persuasive / Persuasive with
Mod / Not Persuasive / Withdrawn / Duplicate) so nothing slips into Normative
in an unresolved state. CGP/Steering deferred them; OO has now accepted the
hand-off.

### Proposed Change

The "change" required by FHIR-54951 itself is a **Jira workflow action**, not
a code/spec edit. Concretely:

1. For each deferred ticket still listed without `<del>` in FHIR-54951's
   description (the seven tickets enumerated above), transition it from its
   current closed/deferred state back to `Open` (or `Triaged`) in Jira and
   reassign the work group to **Orders & Observations** if it is not already
   OO.
2. Schedule each reopened ticket for disposition at an upcoming OO call
   (per the WGM minutes referenced in the comment).
3. Once each individual ticket is re-dispositioned, the substantive
   `HL7/fhir` change (if any) is implemented under that ticket's own
   identifier — not under FHIR-54951.
4. Optionally, update FHIR-54951's description to strike through additional
   tickets as they are individually applied or closed, so the master list
   stays accurate.

There are **no** XML, HTML, JSON, FSH, or terminology files in
`cache/github/repos/HL7_fhir/clone/` that change as a direct result of
FHIR-54951.

### Design Rationale

The Persuasive vote was a procedural one: it sanctions reopening the
deferred backlog rather than letting Normative ship with stale "deferred"
tickets that had never been formally dispositioned. Splitting the actual
content edits into the individual child tickets (rather than bundling them
under FHIR-54951) preserves traceability — each spec change keeps its own
disposition record, applied-vote, and implementation notes.
