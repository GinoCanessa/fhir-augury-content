## Feature Proposal

### Problem Statement

`DeviceDefinition` is a **definitional** resource (it describes a kind/model of device, not a specific instance), but historically it was modeled as a plain `DomainResource` rather than as a `CanonicalResource`. As a result:

- It lacked publication metadata (`url`, `version`, `status`, `publisher`, `description`, `useContext`, `jurisdiction`, `purpose`, `copyright`, …) that all other definitional artifacts (`ActivityDefinition`, `PlanDefinition`, `ObservationDefinition`, `SpecimenDefinition`, etc.) carry.
- It could not be referenced by canonical URL, so consumers could not pin to a specific published version of a device model. Every reference to it was a server-relative `Reference(DeviceDefinition)` with no version pinning semantics.
- It did not appear in the cross-spec "Canonical URLs" listing on the `Resource` page, breaking discoverability.

The reporter (Elliot Silver, 2022-11-07) and supporting Zulip discussion in `#devices > DeviceDefinition as canonical` (Aug 2024) called for the resource to adopt the definitional-resource pattern and for inbound references to switch to `canonical(DeviceDefinition)`.

### Design Rationale

- **Pattern consistency**: every other "Definition" resource (`ActivityDefinition`, `PlanDefinition`, `ObservationDefinition`, `SpecimenDefinition`, …) is a `CanonicalResource`. Treating `DeviceDefinition` the same way removes a long-standing inconsistency flagged in the original ticket and confirmed in the 2024-10-10 OO vote.
- **Choice over replacement**: where a previous `Reference(Device | DeviceDefinition)` existed, the resolution keeps `Reference(Device)` and adds a sibling `canonical(DeviceDefinition)` rather than removing the reference. This preserves existing instance semantics for runtime device handles while letting consumers cite a published device model by canonical URL. Marti Velezis's applied-vote tables explicitly show this `[x]` choice pattern across `DeviceRequest.product`, `Specimen.*Device`, `ObservationDefinition.device`, `ServiceRequest.parameterFocus`, etc.
- **CanonicalResource via interface**: `DeviceDefinition` declares `baseDefinition = CanonicalResource` and uses the build template `CanonicalResource`, which is how all other definitional resources are spelled in this repo. This automatically wires it into the `Canonical URLs` listing rendered from `resource-notes.xml` and the implementing-resources block in `canonicalresource-introduction.xml`.
- **Why defer non-OO updates**: the OO-only scope keeps this ticket tractable and matches the applied-vote agreement. Cross-WG canonical-leg additions involve normative-status, FMM, and binding considerations specific to each owning WG and are best handled as targeted tickets where each WG can review impact.

### Breaking Changes

This ticket *does* introduce backward-incompatible changes relative to R5 (the underlying conversion was applied between R5 and R6 master):

- **Type change:** `Device.definition` switched from `Reference(DeviceDefinition)` to `canonical(DeviceDefinition)`. Existing R5 instances using `reference: "DeviceDefinition/123"` are not valid R6 — implementers must produce canonical URLs.
- **Type change:** `DeviceDefinition.hasPart.definition[x]` (formerly `hasPart.reference: Reference(DeviceDefinition)`) is now `canonical(DeviceDefinition) | CodeableConcept`. Element renamed and retyped.
- **Type change:** `DeviceDefinition.link.relatedDevice[x]` is now `canonical(DeviceDefinition) | CodeableConcept` rather than `CodeableReference(DeviceDefinition)`.
- **Choice expansion (additive but materially different):** `DeviceRequest.product`, `NutritionOrder.enteralFormula.deliveryDevice`, `ObservationDefinition.device`, `ServiceRequest.parameterFocus`, and three sites on `Specimen` now expose a `canonical(DeviceDefinition)` leg in addition to (or instead of) the prior `Reference(... | DeviceDefinition)`. Senders aiming at a device *model* must now use the canonical leg.
- **New constraint:** `cnl-1` (URL-vs-OID) on `DeviceDefinition.url`.
- **Resource categorisation:** `DeviceDefinition` is now a `CanonicalResource`, which carries downstream impacts (it shows up in canonical-aware operations such as `$current-canonical`, package conformance, etc.).

The R6 release notes / migration page should call this out as a non-compatible change for the `DeviceDefinition` resource and any resource that referenced it.
