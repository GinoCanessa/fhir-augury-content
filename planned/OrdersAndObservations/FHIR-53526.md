## Resolution Summary

Resolution: **Persuasive**.

Verbatim resolution description: *"Will update to fix order of code and status to have status first and then code and make the parallel changes in the examples."*

Concretely, the differential of the `NutritionProduct` StructureDefinition currently lists `NutritionProduct.code` before `NutritionProduct.status`. The fix reorders these two top-level elements so `status` precedes `code` (the conventional ordering used by other resources such as `Observation`), and applies the same reordering of the `<status>` and `<code>` elements in every NutritionProduct example instance.

## Feature Proposal

### Problem Statement

NutritionProduct uses an unconventional element order: `code` is declared before `status` at the top level of the resource. The reporter notes this differs from the standard HL7 pattern (e.g., `Observation`, where `status` precedes `code`), and the same backwards order is reflected in all of the published example instances. This is purely a presentation / consistency issue — there is no semantic change — but it is an inconsistency the work group has agreed to fix.

The original ticket description (Rik Smithies):

> I cant find the reference but I think there is a convention that status comes before code, and NutritionProduct has this backwards. It is definitely different to e.g. Observation

The 2025-12-01 HCP meeting agreed this is persuasive (per the comment from JIRAUSER19407 on 2026-02-20 referencing that meeting's "will fix" tag).

### Proposed Change

In `HL7/fhir`, in the NutritionProduct resource folder (`source/nutritionproduct/`):

1. **StructureDefinition** (`source/nutritionproduct/structuredefinition-NutritionProduct.xml`): inside the single `<differential>` block, swap the position of the two `<element>` blocks so that `<element id="NutritionProduct.status"> ... </element>` (currently lines 119–152 in the cached clone) appears immediately after `<element id="NutritionProduct">` (the resource-level element ending at line 86) and **before** `<element id="NutritionProduct.code"> ... </element>` (currently lines 87–118). No content of either element changes — only their relative position. Cardinality, type, binding, isModifier, isSummary, mappings, and definitions are preserved verbatim.

   After the change, the top-level `<path>` order in the differential is:
   ```
   NutritionProduct
   NutritionProduct.status   (was lines 119–152)
   NutritionProduct.code     (was lines 87–118)
   NutritionProduct.category
   NutritionProduct.manufacturer
   ... (unchanged from here)
   ```

2. **Example instances** (six files): in each example, move the existing `<status value="..."/>` element so it appears **before** the `<code> ... </code>` block. Today every example places the `<code>` block (with nested `<coding>` children) starting at the second-line position and the `<status>` element after it. The change is a pure XML element reorder — values are unchanged.

   Files and their current top-of-file positions (resource-level only):
   - `nutritionproduct-example.xml` — `<code>` block starts line 4; `<status value="active"/>` line 11.
   - `nutritionproduct-example-adultformula.xml` — `<code>` line 4; `<status value="active"/>` line 11.
   - `nutritionproduct-example-apples.xml` — `<code>` line 3; `<status value='active'/>` line 11.
   - `nutritionproduct-example-highprotein-enteral.xml` — `<code>` line 4; `<status value="active"/>` line 11.
   - `nutritionproduct-example-peanutbutter.xml` — `<code>` line 3; `<status value='active'/>` line 11.
   - `nutritionproduct-example-shake.xml` — `<code>` line 3; `<status value='active'/>` line 11.

   In every example, the inner `<code value="..."/>` elements that appear later in the file belong to *nested* sibling complex elements (e.g., `nutrient.amount.code`, `instance.identifier`, `ingredient.item.concept.coding.code`) and **must not** be touched — only the resource-level `<code>` (the `NutritionProduct.code` field) is moved.

No element definitions, value sets, terminology, search parameters, narratives, or operations are added, removed, or modified. The `<snapshot>` section is generated by the publisher and is not present in the authoring file (only `<differential>`), per the HL7/fhir authoring conventions.

### Design Rationale

- **Convention alignment.** Across HL7 FHIR resources that have both a `status` and a `code` element (Observation, MedicationRequest, ServiceRequest, NutritionOrder, …), `status` is consistently placed first. Aligning NutritionProduct removes a small but real surprise for implementers reading the resource definition or inspecting examples.
- **Parallelism between definition and examples.** The work group agreed the examples must be updated in lockstep so the rendered XML/JSON examples in the published spec match the new element order.
- **Lowest-impact change shape.** No cardinality, binding, type, or definition is altered, so this is a pure ordering / cosmetic correction. There is no impact on instance validity (XML element order between siblings is significant in FHIR XML serialization, but instances using either order remain schema-valid because the schema accepts the canonical order; existing instances using the old order would need to be re-serialized for strict canonical form, which is what the example update accomplishes).
- **No alternatives considered in the resolution.** The resolution text is unambiguous ("status first and then code" + "parallel changes in the examples"), so no alternative ordering or partial fix is on the table.
