## Resolution Summary

**Resolution: Persuasive with Modification.** Per the resolution description on
the ticket:

1. **Remove** the two FHIR-defined CodeSystems
   `http://hl7.org/fhir/biologicallyderived-productcodes` and
   `http://hl7.org/fhir/biologicallyderived-product-property-type-codes` from
   the FHIR Core specification (they are not complete code systems — at best
   they are fragments/examples of ISBT 128 content owned by ICCBBA).
2. **Redefine** the matching ValueSets so they include codes from the
   HTA-endorsed ISBT 128 code system rather than from a hl7.org-hosted
   CodeSystem resource. The HTA-endorsed identifiers (ISBT 128 page on the HTA
   Confluence space) are:
   - URI: `https://www.isbt128.org/uri`
   - OID: `2.16.840.1.113883.6.18`
3. **Choose a content strategy** for what the ValueSets actually expand to —
   the resolution suggests, as a near-term option, exposing a partial-list
   (`fragment`) ValueSet via the HL7 terminology server, with a longer-term
   option of a full ISBT-128-backed FHIR server. The two existing concept
   subsets (HCT/blood/marrow product codes; collectiontype, aborhd,
   singleeuropeancode, redcellantigen property types) are the minimum content
   carried over.

The driving rationale (Rob Hausam, captured in the ticket description) is
that these codes are ISBT 128 IP and the canonical URL must not be
`http://hl7.org/fhir/biologicallyderived-productcodes`; HTA must own the
canonical code system identity.

## Feature Proposal

### Problem Statement

Two CodeSystem resources currently shipped with FHIR Core declare
`content = complete` and a `http://hl7.org/fhir/...` canonical URL even though
the content is a small subset of ISBT 128, an externally-owned proprietary
standard maintained by ICCBBA:

- `source/biologicallyderivedproduct/codesystem-biologicallyderived-productcodes.xml`
  — 9 ISBT 128 Product Description Codes (e.g. `e0398`, `s1128`, `s1194`,
  `t1396`).
- `source/biologicallyderivedproduct/codesystem-biologicallyderived-product-property-type-codes.xml`
  — 4 ISBT 128 data-element pointers (`collectiontype`, `aborhd`,
  `singleeuropeancode`, `redcellantigen`), each whose definition simply
  references `https://www.isbt128.org/uri/...`.

Both files declare `<content value="complete"/>` (verified in the clone) which
is incorrect: the resources contain a tiny fraction of ISBT 128 and were
never intended to be authoritative. FHIR-49688 already corrected the maturity
flags / immutable flag on the surrounding ValueSets but explicitly deferred
the `content` and canonical-identity question to a follow-up — this ticket.
Marking them `example` was attempted in the FHIR-49688 PR but reverted after
it was discovered the publisher cannot expand a ValueSet whose backing
CodeSystem is `example`. That left the misleading `complete` flag in place
and motivated the present ticket.

The FHIR-Core ownership of those URLs also conflicts with HTA policy: ISBT
128 is HTA-endorsed under the canonical `https://www.isbt128.org/uri` (OID
`2.16.840.1.113883.6.18`), and HL7 must not republish ICCBBA-owned codes
under a `hl7.org/fhir/...` system URL.

### Proposed Change

In `HL7/fhir`, in the `source/biologicallyderivedproduct/` authoring folder:

1. **Delete the two FHIR-Core CodeSystem resources**
   - `source/biologicallyderivedproduct/codesystem-biologicallyderived-productcodes.xml`
   - `source/biologicallyderivedproduct/codesystem-biologicallyderived-product-property-type-codes.xml`

2. **Redefine the two ValueSets** so they no longer compose from
   `http://hl7.org/fhir/biologicallyderived-productcodes` /
   `http://hl7.org/fhir/biologicallyderived-product-property-type-codes`,
   but instead reference the HTA-endorsed ISBT 128 system
   (`https://www.isbt128.org/uri`).

   In `source/biologicallyderivedproduct/valueset-biologicallyderived-productcodes.xml`,
   change the `<compose>` from:

   ```xml
   <compose>
     <include>
       <system value="http://hl7.org/fhir/biologicallyderived-productcodes"/>
     </include>
   </compose>
   ```

   to an enumerated subset against ISBT 128, retaining the 9 Product
   Description Codes that the previous CodeSystem listed:

   ```xml
   <compose>
     <include>
       <system value="https://www.isbt128.org/uri"/>
       <concept>
         <code value="e0398"/>
         <display value="RED BLOOD CELLS|CPD&gt;AS5/450mL/refg|Irr|ResLeu"/>
       </concept>
       <concept><code value="s1128"/><display value="HPC, APHERESIS/Citrate/XX/refg/Mobilized"/></concept>
       <concept><code value="s1194"/><display value="HPC, APHERESIS|NS/XX/&lt;=-120C|10% DMSO|Cryopreserved|Mobilized"/></concept>
       <concept><code value="s1195"/><display value="HPC, APHERESIS|NS/XX/&lt;=-120C|5% DMSO|Cryopreserved|Mobilized"/></concept>
       <concept><code value="s1310"/><display value="HPC, APHERESIS|None/XX/refg|3rd Party Comp:Yes|Other Additives:Yes|Mobilized|CD34 enriched"/></concept>
       <concept><code value="s1398"/><display value="HPC, MARROW|NS/XX/rt|Plasma reduced"/></concept>
       <concept><code value="s2598"/><display value="HPC, MARROW|NS/XX/&lt;=-150C|10% DMSO|3rd Party Comp:Yes|Cryopreserved|RBC reduced"/></concept>
       <concept><code value="e4377"/><display value="Apheresis RED BLOOD CELLS|ACD-A/XX/refg|Irradiated|1st container"/></concept>
       <concept><code value="t1396"/><display value="BONE, FEMUR|Frozen|Right|Radiation sterilization"/></concept>
     </include>
   </compose>
   ```

   Update the ValueSet `<description>` to note that this is an example /
   fragment subset of ISBT 128 Product Description Codes for illustration
   only, with the canonical content owned by ICCBBA at
   `https://www.isbt128.org/uri`. Keep the `<copyright>` ICCBBA notice. Also
   add an `<identifier>` entry for the HTA-endorsed OID
   `2.16.840.1.113883.6.18` so terminology services can resolve the
   underlying system. The ValueSet's own `<url>` and `<id>` (canonical
   `http://hl7.org/fhir/ValueSet/biologicallyderived-productcodes`) MUST stay
   unchanged so that the `BiologicallyDerivedProduct.productCode` binding in
   `structuredefinition-BiologicallyDerivedProduct.xml` (line 131) keeps
   resolving.

   In `source/biologicallyderivedproduct/valueset-biologicallyderived-product-property-type-codes.xml`,
   apply the same pattern, switching the `<include>` to
   `https://www.isbt128.org/uri` and enumerating the four ISBT 128 data-
   element references that currently exist:

   ```xml
   <compose>
     <include>
       <system value="https://www.isbt128.org/uri"/>
       <concept><code value="CollectionType"/><display value="Collection Type"/></concept>
       <concept><code value="ABORhD"/><display value="ABORhD"/></concept>
       <concept><code value="SingleEuropeanCode"/><display value="Single European Code"/></concept>
       <concept><code value="RedCellAntigen"/><display value="Red Cell Antigen"/></concept>
     </include>
   </compose>
   ```

   (Code casing follows the URIs that the now-deleted CodeSystem pointed at,
   e.g. `https://www.isbt128.org/uri/CollectionType`. Confirm the precise
   ISBT 128 URI fragment with HTA / ICCBBA before merge — see Open Questions.)

3. **Keep the StructureDefinition bindings unchanged** in
   `source/biologicallyderivedproduct/structuredefinition-BiologicallyDerivedProduct.xml`:
   - Line 131 — `BiologicallyDerivedProduct.productCode` binding to
     `http://hl7.org/fhir/ValueSet/biologicallyderived-productcodes` (strength
     `example`).
   - Line 399 — `BiologicallyDerivedProduct.property.type` binding to
     `http://hl7.org/fhir/ValueSet/biologicallyderived-product-property-type-codes`
     (strength `example`).

   These are example-strength bindings, so reshuffling the underlying
   ValueSet content does not affect conformance. (No analogous bindings exist
   in `BiologicallyDerivedProductDispense` — confirmed by grep across
   `source/biologicallyderivedproductdispense/` — so the dispense resource is
   not touched by this ticket despite appearing in the ticket title.)

### Design Rationale

- **HTA / ICCBBA ownership.** Per Rob Hausam's comment on the ticket and the
  HTA Confluence ISBT 128 page, the canonical URL for ISBT 128 must be
  `https://www.isbt128.org/uri` (OID `2.16.840.1.113883.6.18`). HL7 must not
  republish those codes under a `http://hl7.org/fhir/...` system URL.
- **`example` content type was rejected** because the publisher and
  terminology server cannot expand ValueSets backed by an `example`
  CodeSystem; this was tried in the FHIR-49688 PR and reverted. Removing the
  CodeSystem resources entirely sidesteps that tooling limitation while
  honouring the IP boundary.
- **Enumerating concepts in the ValueSet `<compose>`** preserves the existing
  example codes (so existing examples and documentation snippets remain
  resolvable and readable) without claiming HL7 ownership of the underlying
  system. This matches the resolution's "partial list (fragment) in the
  terminology server" near-term option; the longer-term "full via FHIR
  Server" option is left as a follow-up because it depends on ICCBBA
  licensing and an externally-hosted endpoint that does not yet exist.
- **Keep the FHIR ValueSet canonicals stable** so that the example-strength
  bindings on `BiologicallyDerivedProduct.productCode` and
  `BiologicallyDerivedProduct.property.type` continue to resolve and so that
  no breaking change is introduced for implementers binding to those URLs.

### Breaking Changes

- **CodeSystem URLs disappear.** Both
  `http://hl7.org/fhir/biologicallyderived-productcodes` and
  `http://hl7.org/fhir/biologicallyderived-product-property-type-codes` are
  withdrawn. Any external implementer who imported codes via the FHIR
  CodeSystem URL (rather than via the ValueSet) will get an unresolvable
  reference. This is intentional per the resolution but is a **breaking
  change for direct CodeSystem consumers** and must be called out in the
  release notes for this resource.
- **ValueSet semantics change** from "all codes from a small HL7-hosted
  CodeSystem" to "an enumerated set of codes drawn from the externally-owned
  ISBT 128 system". The canonical URLs and IDs of the ValueSets stay the
  same, and the StructureDefinition bindings remain example strength, so no
  conformance check that exists today will start failing — but expansions
  will now contain ISBT-128-system-coded codings instead of
  `hl7.org/fhir/...`-system-coded codings, which is a wire-level change for
  any consumer that pinned the system in their parser.
- No changes to required/preferred/extensible bindings (both bindings remain
  `example`).

### Terminology Impact

- **Two CodeSystems withdrawn from FHIR Core:**
  `http://hl7.org/fhir/biologicallyderived-productcodes` and
  `http://hl7.org/fhir/biologicallyderived-product-property-type-codes`.
- **Two ValueSets rebound** to `https://www.isbt128.org/uri` (ISBT 128
  HTA-endorsed canonical URI; OID `2.16.840.1.113883.6.18`).
- **No UTG (HL7/UTG) edits required.** The HTA-endorsed system is externally
  owned by ICCBBA and is not (and should not be) republished in UTG by HL7.
  If the HL7 terminology server needs to host a partial expansion as a
  short-term fragment (the resolution's "partial list (fragment) in the
  terminology server" option), that is operational work on the
  tx.fhir.org-equivalent server and not a content change in HL7/UTG or
  HL7/fhir.
- **Tooling impact:** The new ValueSets are enumerated `<concept>` lists, so
  expansion does not depend on any registered CodeSystem resource. The
  publisher should expand them cleanly without the failure mode that
  occurred when the prior PR tried `content="example"`.
