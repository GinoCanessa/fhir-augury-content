## Summary

The OO Incubator IG (`HL7/oo-incubator`) contains StructureDefinitions, CodeSystems, and ValueSets for 8 resource types that were moved from R5 core FHIR. The ticket reports that CodeSystems and ValueSets are "not all listed in the artifacts" and that "some of them are pointing to R5 content." The resolution is **Persuasive**, meaning the request is accepted and implementation is required.

## Feature Proposal

### Current State

The incubator IG includes CodeSystems and ValueSets for 8 resources: BiologicallyDerivedProductDispense, DeviceDispense, DeviceUsage, InventoryItem, InventoryReport, SupplyDelivery, SupplyRequest, and Transport. However, the terminology artifacts have an inconsistent URL scheme:

- **5 CodeSystems** have been properly migrated to incubator-local URLs (`http://hl7.org/fhir/uv/oo-incubator/CodeSystem/<name>`):
  - `biologicallyderivedproductdispense-match-status`
  - `biologicallyderivedproductdispense-origin-relationship`
  - `biologicallyderivedproductdispense-performer-function`
  - `devicedispense-status-reason`
  - `devicedispense-status`

- **18 CodeSystems** still use R5 core FHIR URLs (`http://hl7.org/fhir/<name>`), pointing to content owned by R5 core:
  - `biologicallyderivedproductdispense-status`
  - `deviceusage-adherence-code`, `deviceusage-adherence-reason`, `deviceusage-status`
  - `inventoryitem-nametype`, `inventoryitem-status`
  - `inventoryreport-counttype`, `inventoryreport-status`
  - `supplydelivery-stage`, `supplydelivery-status`, `supplydelivery-supplyitemcondition`, `supplydelivery-supplyitemtype`
  - `supplyrequest-status`
  - `transport-code`, `transport-intent`, `transport-status-reason`, `transport-status`, `transport`

- **All ValueSets** use incubator-local URLs for their own canonical URLs (correct), but **18 ValueSets reference R5-URL CodeSystems** in their `compose` sections, creating an unresolved dependency on R5 core content.

The `special-url-base: http://hl7.org/fhir` parameter in `sushi-config.yaml` allows R5 URLs to pass validation, masking the problem.

### Proposed Change

Migrate all 18 remaining R5-URL CodeSystems to use incubator-local canonical URLs (`http://hl7.org/fhir/uv/oo-incubator/CodeSystem/<name>`), following the same pattern already established for the 5 migrated CodeSystems. Update all ValueSet `compose` sections to reference the new incubator-local CodeSystem URLs. This makes the incubator IG self-contained and removes the dependency on R5 core terminology content.

### Justification

- Consistency: 5 CodeSystems already use incubator URLs; the remaining 18 should follow the same pattern.
- Self-containment: The incubator IG should own and publish all its terminology artifacts rather than depending on R5 core.
- Artifact visibility: With R5 URLs, these CodeSystems may not appear correctly in the IG artifact listing because they look like external references.
- Resolution is Persuasive — the request is valid and accepted by the work group.

### Risk Assessment

- **Low risk:** Changes are mechanical (find-and-replace on URL values) and follow an established pattern (5 CodeSystems already migrated).
- **Consideration:** The `special-url-base` setting permits the IG to publish artifacts with `http://hl7.org/fhir/` URLs. The StructureDefinitions themselves still use `http://hl7.org/fhir/StructureDefinition/<ResourceName>` URLs, so `special-url-base` likely needs to remain even after CodeSystem migration.
- **Consideration:** The CodeSystems' `<valueSet>` elements (which cross-reference back to the ValueSet they populate) should also be verified to use incubator-local URLs — some R5-URL CodeSystems already correctly reference incubator-local ValueSets in this element (e.g., `biologicallyderivedproductdispense-status` has `<valueSet value="http://hl7.org/fhir/uv/oo-incubator/ValueSet/...">`).
