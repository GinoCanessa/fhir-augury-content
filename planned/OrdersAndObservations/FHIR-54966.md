## Resolution Summary

The approved resolution (Persuasive with Modification) is:

> *"Motion to update the authorship where OO and Dev are co-authors and OO is the gatekeeper of and managing the JIRAs addressing the module page. Also include an acknowledgement section at the end similar to the one present for the Clinical Reasoning Module"*

Concretely, the Device module page must be changed so that:

1. Authorship is shown as **co-authored** by Orders & Observations (OO) and Health Care Devices (DEV) work groups (today the page lists OO as the sole "Responsible Owner").
2. It is recorded that OO remains the **gatekeeper** for the module page — i.e., OO continues to triage/manage Jira tickets that target the module page itself (with applicable tickets being routed to the responsible WG, as confirmed by Devices WG in the ticket comment from 2026-01-27).
3. An **Acknowledgements** section is added at the end of the page, modelled on the one at the end of `clinicalreasoning-module.html` (a single short paragraph naming the two co-authoring WGs).

This is a documentation-only change to a single specification page; no resources, profiles, value sets, or extensions are affected.

## Feature Proposal

### Problem Statement

The Device module page (`source/device-module.html`) currently advertises a single Responsible Owner — Work Group OO — via the standard module header:

```html
<table class="colsi"><tr><td id="wg">Responsible Owner: Work Group
  <a _target="blank" href="[%wg oo%]">[%wgt oo%]</a> </td>
  <td id="ballot">…</td></tr></table>
```

In practice the Device module is the joint concern of Orders & Observations and the Health Care Devices (DEV) work group: DEV owns the device-related resources (Device, DeviceAlert, DeviceAssociation, DeviceDefinition, DeviceMetric, DeviceRequest, DeviceUsage, DeviceDispense), while OO has historically managed the cross-cutting module page that introduces and indexes them. The current single-owner presentation:

- understates DEV's role as a substantive author of the module's content; and
- gives readers no indication of how Jira tickets for the module page are routed (which is the operational question that prompted this ticket — see the comment from `mcros` confirming DEV's agreement on co-authorship and OO's gatekeeping role for triage).

The page also lacks an "Acknowledgements" closing section, which is the convention used elsewhere in core (notably `source/clinicalreasoning-module.html`) for joint-authored modules.

### Proposed Change

Edit `source/device-module.html` to make three concrete adjustments:

1. **Replace the single-WG header cell with a co-author cell**, matching the two-WG pattern used in `clinicalreasoning-module.html` (`<a …>[%wgt cds%]</a> &amp; <a …>[%wgt cqi%]</a>`).

   New cell:

   ```html
   <td id="wg">Responsible Owner: Work Groups
     <a _target="blank" href="[%wg oo%]">[%wgt oo%]</a> &amp;
     <a _target="blank" href="[%wg dev%]">[%wgt dev%]</a>
   </td>
   ```

   The `[%wg dev%]` / `[%wgt dev%]` shortcodes correspond to the Health Care Devices work group ("DEV WG" as named in the ticket and comment). Confirm the exact shortcode against the publisher's WG table before merge — see Open Questions.

2. **Add a short "module-management" note** so the gatekeeper role is visible to readers. The existing intro is the natural place. Append a sentence to (or add a sibling paragraph after) the current note-to-implementers stu-note block, e.g.:

   ```html
   <p>This module page is co-authored by Orders &amp; Observations (OO) and
   Health Care Devices (DEV). OO acts as the gatekeeper for the module page
   itself and triages incoming Jira tickets, routing each to the responsible
   work group.</p>
   ```

3. **Add an Acknowledgements section at the bottom of the page**, immediately before `[%file newfooter%]` (line 315 in the current clone HEAD). Use the same shape as `clinicalreasoning-module.html` lines 341–345:

   ```html
   <a name="acknowledgements"></a>
   <h3>Acknowledgements</h3>

   <p>The Device module page and the guidance it provides are the joint work
   of the HL7 Orders &amp; Observations and Health Care Devices Work Groups.
   Orders &amp; Observations stewards the module page and triages incoming
   Jira tickets to the responsible work group.</p>
   ```

No other elements (the Note to Implementers stu-note, Introduction, Index, "Big picture" diagram, Common Use Cases, the resources table at the bottom, or the `O&O welcomes feedback` paragraph) need to be touched as part of this ticket. The closing OO-feedback paragraph at line 311–313 may stay as-is or be lightly broadened to "OO and DEV welcome feedback…" as a stylistic alignment, but that wording change is optional and not required by the resolution.

### Design Rationale

- The two-WG header pattern is already used in core (`clinicalreasoning-module.html` shows CDS & CQI), so reusing it preserves consistency and keeps the publisher-rendered "Responsible Owner" link rendering working out of the box.
- An end-of-page Acknowledgements section is the resolution's explicit ask ("similar to the one present for the Clinical Reasoning Module"). That page's Acknowledgements is a single short paragraph naming both WGs, which is the model imitated above.
- Recording OO as gatekeeper directly in the page (rather than only in the Jira workflow) lets readers and future ticket triagers see the routing rule without leaving the spec. The Devices WG comment on 2026-01-27 ("Ticket triaging by OO group assigning the ticket to a responsible WG") explicitly endorses this arrangement, so capturing it on the page is aligned with WG consensus.
- Alternatives considered:
  - *Make DEV the sole owner and remove OO.* Rejected — both the original ticket and the resolution are explicit that this is **co-authorship**, with OO retaining the gatekeeper/triage role.
  - *Capture co-authorship only in the Acknowledgements section, leaving the header as OO-only.* Rejected — the header "Responsible Owner" cell is the canonical machine-readable signal of authorship in core, and leaving it OO-only would contradict the resolution and confuse downstream tooling that reads the WG table.
