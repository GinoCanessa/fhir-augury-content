## Resolution Summary

The committee resolution (Persuasive with Modification) is:

> *"Remove MPHO Profile as it is not ready for Normative status as we move to R6. This should be moved to the OO Additional Resources (which may be renamed) IG instead."*

In other words, none of the three options described in the original ticket
(fix the slicing discriminator, remove the slicing only, or remove the profile
entirely) are taken in core. Instead the profile itself is removed from FHIR
Core for R6 because (a) the slicing discriminator on
`category.coding.code` / `category.coding.system` is invalid against an example
binding and (b) the profile, in its current shape, is not ready to publish at
Normative maturity. The artifact is to be relocated to an Orders &
Observations "Additional Resources" IG (in practice — and per Marti
Velezis's 2025-11-22 comment — staged in `HL7/oo-incubator` until that IG
exists, and tracked for ballot under follow-up ticket
[FHIR-54998](https://jira.hl7.org/browse/FHIR-54998)).

The companion ballot note already in `biologicallyderivedproduct-introduction.xml`
explicitly lists "The Medical Product of Human Origin (MPHO) Profile is still
under development" as moved to the OO Incubator IG, so this plan only needs
to make the source tree match that statement.

## Feature Proposal

### Problem Statement

The MedicalProductOfHumanOrigin profile (`StructureDefinition/medicalproductofhumanorigin`)
currently shipped from FHIR Core (`source/biologicallyderivedproduct/`) declares
slicing on `BiologicallyDerivedProduct.productCategory` using two discriminators:

```
<discriminator>
  <type value="value"/>
  <path value="coding.code"/>
</discriminator>
<discriminator>
  <type value="value"/>
  <path value="coding.system"/>
</discriminator>
```

These discriminators are not valid: `productCategory` is an `example`-bound
CodeableConcept, so a `value`-type discriminator on `coding.code`/`coding.system`
cannot reliably select slices, and the publisher emits an error. Beyond the
technical defect, the profile body is sparse — a handful of cardinality
constraints — and was reviewed by the work group as not strong enough to
justify Normative inclusion in FHIR Core for R6. The discussion thread on
Zulip ([#O&O-Committers > FHIR-49665](https://chat.fhir.org/#narrow/channel/520162-O.26O-Committers/topic/FHIR-49665))
plus subsequent applied-vote comments converged on removing the profile from
core rather than patching the slicing.

### Proposed Change

Remove the MedicalProductOfHumanOrigin (MPHO) profile and its supporting
artifacts from the FHIR Core source tree (`HL7/fhir`), and adjust narrative
and example metadata so the BiologicallyDerivedProduct module no longer
publishes or references that profile. Concretely:

1. **Delete the StructureDefinition** for the profile:
   `source/biologicallyderivedproduct/structuredefinition-profile-medicalproductofhumanorigin.xml`
   (URL `http://hl7.org/fhir/StructureDefinition/medicalproductofhumanorigin`,
   name `MedicalProductOfHumanOrigin`, derivation `constraint` on
   `BiologicallyDerivedProduct`, currently 153 lines including the invalid
   `productCategory` slicing block at lines 78–87 plus the
   `productCategory:MPHOCode` slice at lines 88–98 and the cardinality
   tightenings on `productCode`, `identifier`, `biologicalSourceEvent`).

2. **Delete the per-profile authored prose**:
   - `source/biologicallyderivedproduct/profile-mphobiologicallyderivedproduct-introduction.xml`
     (effectively empty stub).
   - `source/biologicallyderivedproduct/biologicallyderivedproduct-mphobiologicallyderivedproduct-profile-notes.xml`
     (effectively empty stub).

3. **Delete the in-core IG wrapper** that referenced the profile:
   `source/biologicallyderivedproduct/implementationguide-BiologicallyDerivedProduct-core.xml`
   (its only `definition.resource.reference` was
   `StructureDefinition/medicalproductofhumanorigin`; with that profile gone,
   the IG wrapper has no remaining content and the publisher should not be
   asked to ship an empty IG entry).

4. **Drop the `meta.profile` claim from the two MPHO-flavoured examples** so
   they no longer assert conformance to a profile that no longer exists in core:
   - `source/biologicallyderivedproduct/biologicallyderivedproduct-example-apheresis-platelet.xml`
     — remove the `<meta><profile value="…/medicalproductofhumanorigin"/></meta>`
     block (lines 4–6).
   - `source/biologicallyderivedproduct/biologicallyderivedproduct-example-red-cell-antigen.xml`
     — remove the equivalent `<meta>` block (lines 4–6).

   Keep the example payloads themselves: they are still valid base
   `BiologicallyDerivedProduct` instances that exercise the ISBT 128
   identifiers, `biologicalSourceEvent`, and `property` slots, and they
   are listed in `list-BiologicallyDerivedProduct-examples.xml`.

5. **Update the BDP module narrative** so the prose stops promising an
   in-core profile:
   - `source/biologically-derived-product-module.html` — line 45
     ("A profile for MPHO has been developed that specifies that essential
     traceability identifiers are required.") should be replaced with a
     pointer to the OO Incubator IG, e.g. *"A profile for MPHO is being
     developed in the [OO Incubator IG](https://build.fhir.org/ig/HL7/oo-incubator)."*
     The MPHO definition sentence at line 33 stays.

6. **Confirm the existing balloter note**:
   `source/biologicallyderivedproduct/biologicallyderivedproduct-introduction.xml`
   already states that the MPHO Profile has been "moved to the OO Incubator IG".
   Verify that wording is still accurate for the R6 ballot publication and
   leave it in place. No edit unless the balloter note needs trimming after
   the deletions land.

7. **Out of scope but worth noting**: the `mpho` code in the
   `product-category` CodeSystem (`source/biologicallyderivedproduct/codesystem-product-category.xml`,
   lines 73–74) remains — it is the example-binding code, used by base
   `BiologicallyDerivedProduct` examples, and is independent of the profile.
   The `MPHO Unique Identifier` / ISBT-128 mapping rows in
   `structuredefinition-BiologicallyDerivedProduct.xml` (lines 101, 189)
   describe the base resource, not the profile, and stay as-is.
   `source/supplydelivery/supplydelivery-example-mphodelivery.xml` does not
   declare conformance to the deleted profile (it just uses MPHO-shaped
   data), so it stays.

The actual relocation of the profile content into `HL7/oo-incubator` (or the
eventual OO Additional Resources IG) is tracked separately by ballot ticket
**FHIR-54998** and is not in the scope of this implementation plan; this plan
only completes the **removal** half from FHIR Core.

### Design Rationale

- **Why delete rather than fix the slicing.** Patching the discriminator
  (`$this`-style discriminator on a CodeableConcept slice with `pattern`
  matching) would technically work, but the work-group consensus —
  Lloyd McKenzie (2025-10-03), Jose Costa-Teixeira (2025-08-27), David Hay /
  Yanick Gaudet (applied-vote thread) — was that a profile defined by a
  handful of cardinality constraints over an example-bound element does not
  meet the bar for Normative inclusion in Core. Fixing the slicing in place
  would lock that bar lower for R6.
- **Why move to an Additional Resources / Incubator IG rather than drop the
  work entirely.** David Hay (JIRAUSER19407, 2025-10-03) explicitly
  conditioned his agreement on the destination being an
  "additional resources" IG that allows the profile to keep maturing. The
  resolution adopts that condition. Marti Velezis (2025-11-22) confirmed
  that, in the interim, `HL7/oo-incubator` is the staging location until the
  named IG is set up.
- **Why delete the wrapper IG file.** `implementationguide-BiologicallyDerivedProduct-core.xml`
  was a thin in-core IG with a single resource reference to the profile.
  Once the profile is removed, the wrapper has no remaining definition and
  would either fail validation or publish an empty IG; deletion is the
  cleanest outcome. The base resource itself ships from
  `structuredefinition-BiologicallyDerivedProduct.xml` independently of any
  IG wrapper.
- **Why strip `meta.profile` from the two examples instead of deleting
  them.** Both examples are still useful illustrations of populating a
  base BiologicallyDerivedProduct with ISBT 128–style identifiers and
  properties, and they are already enrolled in the BDP examples list.
  Asserting conformance to a deleted profile would, by contrast, break
  validation and confuse implementers.
- **Why the `mpho` category code stays in the product-category CodeSystem.**
  The example binding on `BiologicallyDerivedProduct.productCategory`
  remains, the `mpho` code is genuinely useful as one example category, and
  base examples (`biologicallyderivedproduct-example.xml`) use it. The
  original ticket's discussion of the example binding being "not strong
  enough" is precisely the reason the profile is being removed; the code
  itself is fine.

### Breaking Changes

Yes, this is a backward-incompatible change for any implementer who is
asserting `meta.profile = http://hl7.org/fhir/StructureDefinition/medicalproductofhumanorigin`:

- The profile URL `http://hl7.org/fhir/StructureDefinition/medicalproductofhumanorigin`
  will resolve to *Not Found* in R6 core. Existing R5 instances claiming
  conformance to that profile will fail validation against R6 core.
- The in-core IG `BiologicallyDerivedProduct-core` (id `BiologicallyDerivedProductCore`)
  is removed.
- Two formerly profile-conformant examples lose their `meta.profile` claim
  (the underlying instance data is unchanged).

These are intentional and align with the resolution: the profile is moving
to a separately-versioned IG rather than persisting under a Core URL at a
maturity level the work group does not endorse for Normative.

There is **no** breaking change to the base `BiologicallyDerivedProduct`
resource itself, to its search parameters, or to the `product-category`
CodeSystem.
