## Feature Proposal

Remove the `mustSupport` flag from `DeviceDefinition.type` in the LIVD DeviceDefinition profile (`devicedefinition-uv-livd`). The element retains its `0..1` cardinality and its extensible binding to `livd-device-type`. Only the Must Support designation is removed.

## Affected Repository

> **REPO_NOT_CACHED: HL7/livd**

The LIVD IG lives in the [HL7/livd](https://github.com/HL7/livd) repository, which is **not** in the cached repository list. All file references below are based on GitHub API inspection of the repository at commit `041d810`.

## Impact Analysis

### Scope: Minimal (single-element flag change)

This is a profile metadata change — removing `mustSupport: true` from one element. It does not change cardinality, type, bindings, or any structural aspect of the profile.

### Affected Files

| File | Change Type | Description |
|------|-------------|-------------|
| `input/fsh/devicedefinition-uv-livd.fsh` | Modify | Remove MS / add explicit `^mustSupport = false` on `type` |

### Files NOT Affected

| File | Reason |
|------|--------|
| `input/fsh/extensions.fsh` | The `DeviceDefinitionClassificationType` extension is separate from the base `type` element |
| `input/fsh/bundle-uv-livd.fsh` | Bundle profile does not constrain DeviceDefinition.type |
| `input/fsh/observationdefinition-uv-livd.fsh` | Different resource profile |
| `input/examples/devicedefinition-livd-example.json` | Example does not include MS metadata |
| `sushi-config.yaml` | No config changes needed |

### Breaking Change Assessment

**Non-breaking.** Removing Must Support is a relaxation of conformance expectations. Implementations that previously populated `DeviceDefinition.type` will continue to work. Implementations that relied on `type` being Must Support will no longer be required to support it, but can still optionally use it.

## Implementation Plan

### Current State Analysis

The current FSH file (`input/fsh/devicedefinition-uv-livd.fsh`) at HEAD (`041d810`) shows:

```fsh
* type 0..1
* type from http://hl7.org/fhir/uv/livd/ValueSet/livd-device-type (extensible)
* type ^binding.description = "IVD device types for LIVD device definitions (extensible)."
```

The `MS` keyword is **not present** on the `type` line. However, the QA note from December 2025 indicates that Must Support is still appearing in the generated profile output. This suggests one of:
1. The IG output has not been regenerated since the FSH was last edited
2. The `type` element needs an **explicit** `^mustSupport = false` to suppress MS that may be carried over in the differential

### Step 1: Explicitly Remove mustSupport from `type`

**File:** `input/fsh/devicedefinition-uv-livd.fsh`

**Change:** Add `^mustSupport = false` to the `type` element, following the pattern used by other non-MS elements in this profile (e.g., `* identifier ^mustSupport = false`).

**Before:**
```fsh
* type 0..1
* type from http://hl7.org/fhir/uv/livd/ValueSet/livd-device-type (extensible)
* type ^binding.description = "IVD device types for LIVD device definitions (extensible)."
```

**After:**
```fsh
* type 0..1
* type ^mustSupport = false
* type from http://hl7.org/fhir/uv/livd/ValueSet/livd-device-type (extensible)
* type ^binding.description = "IVD device types for LIVD device definitions (extensible)."
```

### Step 2: Verify Generated Output

After the change, rebuild the IG with SUSHI and verify:
1. The generated `StructureDefinition-devicedefinition-uv-livd.json` does **not** include `"mustSupport": true` on the `type` element in the differential
2. The `type` element retains cardinality `0..1`
3. The extensible binding to `livd-device-type` ValueSet is preserved
4. No new QA errors are introduced

```bash
# From the HL7/livd repo root
sushi .
# Then run the IG publisher
./_genonce.sh  # or _genonce.bat on Windows
```

### Step 3: Verify Examples Still Validate

The example file `input/examples/devicedefinition-livd-example.json` should continue to validate without errors against the updated profile. Since the change only removes a Must Support flag (a relaxation), no example changes are needed.
