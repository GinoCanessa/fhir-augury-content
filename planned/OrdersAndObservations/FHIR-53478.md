## Resolution Summary

**Resolution:** Persuasive (no resolution-description text or applied-vote
comments are present in the Jira snapshot — `comments` count = 0,
`metadata.resolution_description` not populated).

The ticket asks O&O to **finalize** the THO move for three CodeSystems
that the Core spec build currently still hosts but that are now
maintained at terminology.hl7.org. The build/published pages at
`build.fhir.org/codesystem-<id>.html` should be replaced by redirects to
the corresponding `terminology.hl7.org/CodeSystem-<id>.html` pages, i.e.,
the local Core CodeSystem definitions stop being authored in
`HL7/fhir` and the canonical home becomes THO. Verbatim from the
description:

> Finalize THO moves in the Core spec for the following codeSystems:
> - DeviceAssociation Status Reason — `…/codesystem-deviceassociation-status-reason.html` should be redirected to `…/CodeSystem-deviceassociation-status-reason.html`
> - DeviceAssociation Relationship — `…/codesystem-deviceassociation-relationship.html` should be redirected to `…/CodeSystem-deviceassociation-relationship.html`
> - NutritionIntake Status Reason — `…/codesystem-nutritionintake-status-reason.html` should be redirected to `…/CodeSystem-nutritionintake-status-reason.html`

Per the FMG note already in `source/ballot-intro.html`:

> "Moving CodeSystems from core to THO is only a substantive change if
> (a) the CodeSystem is bound to a Coding, CodeableConcept, or
> CodeableReference, and (b) the URL changes."

The three CodeSystems here are referenced through ValueSets bound on
CodeableConcept elements (`DeviceAssociation.statusReason`,
`DeviceAssociation.relationship`, `NutritionIntake.statusReason`), so the
CS URL changes (`http://hl7.org/fhir/...` →
`http://terminology.hl7.org/CodeSystem/...`) are by definition
substantive — but they are also the changes the WG explicitly
agreed to in pre-ballot notes.

## Feature Proposal

### Problem Statement

Three O&O CodeSystems were authored inside the FHIR Core spec while THO
copies were being prepared. Pre-ballot notes in `source/ballot-intro.html`
(line 130) and `source/nutritionintake/nutritionintake-introduction.xml`
already announce the intent to move
`nutritionintake-status-reason` to THO. The DeviceAssociation pair
(`deviceassociation-status-reason`, `deviceassociation-relationship`)
followed the same trajectory. The published pages on `build.fhir.org`
still resolve to Core-hosted definitions; the Core build needs to stop
authoring these CSs locally and redirect to the THO-hosted pages so
there is exactly one canonical home for each system.

Current Core-hosted state (verified in clone `db79dcd`):

- `source/deviceassociation/codesystem-deviceassociation-status-reason.xml` — `<url value="http://hl7.org/fhir/deviceassociation-status-reason"/>` (note: no `/CodeSystem/` segment), `wg=oo`, `standards-status=normative`, 8 concepts.
- `source/deviceassociation/codesystem-deviceassociation-relationship.xml` — `<url value="http://hl7.org/fhir/deviceassociation-relationship"/>`, `wg=oo`, `standards-status=normative`.
- `source/nutritionintake/codesystem-nutritionintake-status-reason.xml` — `<url value="http://hl7.org/fhir/CodeSystem/nutritionintake-status-reason"/>`, OID `2.16.840.1.113883.4.642.4.2129`, `wg=oo`, `standards-status=informative`, 7 concepts (renamed from numeric to mnemonic codes in FHIR-53303 / commit `5ece12a376`).

### Proposed Change

Carry out the standard "finalize THO move" pattern for each of the three
CodeSystems. The ValueSets stay in Core (they are the binding targets
referenced from the StructureDefinitions) but their `compose.include.system`
is rewritten to the new THO canonical URL.

For **each** of the three CodeSystems:

1. **Delete** the local Core CodeSystem XML so the publisher stops
   emitting `codesystem-<id>.html` and `CodeSystem-<id>.json/xml` in
   Core. The publisher's existing redirect machinery (`source/redirect.*`)
   takes care of producing a cross-version redirect at the old Core URL
   to the THO page.
2. **Rewrite** the binding ValueSet's `<compose><include><system>…` from
   the old Core CS URL to the new THO URL:
   - `http://hl7.org/fhir/deviceassociation-status-reason` → `http://terminology.hl7.org/CodeSystem/deviceassociation-status-reason`
   - `http://hl7.org/fhir/deviceassociation-relationship` → `http://terminology.hl7.org/CodeSystem/deviceassociation-relationship`
   - `http://hl7.org/fhir/CodeSystem/nutritionintake-status-reason` → `http://terminology.hl7.org/CodeSystem/nutritionintake-status-reason`
3. Leave the **ValueSet URLs** unchanged
   (`http://hl7.org/fhir/ValueSet/<id>`) so that the StructureDefinition
   bindings continue to resolve and no resource-side change is needed.
4. **Remove** the CS row from `source/oids.ini` (and only the CS row —
   the ValueSet OID for `nutritionintake-status-reason` stays because the
   VS itself remains Core-hosted). Specifically remove the line:
   `http://hl7.org/fhir/CodeSystem/nutritionintake-status-reason = 2.16.840.1.113883.4.642.4.2129`.
   The two DeviceAssociation CSs have no OID rows in `oids.ini`, so no
   action is needed for those.
5. **Update the balloter notes** that still describe the move as
   "expected" so they no longer claim a future-tense move:
   - `source/nutritionintake/nutritionintake-introduction.xml` — both the
     change-summary `<li>` and the `Note to Balloters` paragraph that say
     the CS "is currently defined in the FHIR specification, but is
     expected to be moved to HL7 Terminology (THO)" should be reworded
     to "has been moved to HL7 Terminology (THO)" (or removed for the
     change-summary bullet, since "moved to THO" is the actual change for
     this ticket).
   - `source/ballot-intro.html` line 130 — same rewording for the
     `nutritionintake-status-reason` bullet. No equivalent line currently
     exists for the two DeviceAssociation CSs (the ballot-intro file does
     not mention them), so no addition is required there; if the WG
     wants the move called out explicitly, add bullets adjacent to
     line 130.

Example (concrete) ValueSet diff — `valueset-deviceassociation-status-reason.xml`:

```xml
   <compose>
     <include>
-      <system value="http://hl7.org/fhir/deviceassociation-status-reason"/>
+      <system value="http://terminology.hl7.org/CodeSystem/deviceassociation-status-reason"/>
     </include>
   </compose>
```

### Design Rationale

- This is the WG-blessed "finalize THO move" pattern. The ballot-intro
  language already in the spec ("Moving CodeSystems from core to THO is
  only a substantive change if (a) the CodeSystem is bound to a Coding,
  CodeableConcept, or CodeableReference, and (b) the URL changes") tells
  balloters exactly what to expect; carrying through the move keeps the
  spec consistent with that note.
- Deleting the local CS rather than `deprecated`-stamping it is the
  appropriate path here because THO already hosts the authoritative
  copy (see ticket links to `terminology.hl7.org/CodeSystem-…`). The
  alternative — keeping a deprecated-stamped local CS, as was done for
  CDSHooks in commit `51d8fb83b6` (FHIR-19786) — applies when the THO
  copy uses a different URL and the Core URL must remain resolvable
  with a deprecation marker. Here the move is a straight rename of the
  canonical home; redirects handle the URL change.
- Keeping the ValueSets Core-hosted (with rewritten `compose.system`) is
  the cheapest correct change: it preserves every existing
  StructureDefinition binding URL, so no resource-side or
  search-parameter changes are needed.
- For `nutritionintake-status-reason` the VS additionally composes a
  SNOMED concept (`419099009 Dead`) and an unbounded `include` of the
  full local CS. The full-CS `include` is what is being repointed; the
  SNOMED `include` is left untouched.

### Breaking Changes

The CodeSystem URL changes are **substantive (i.e., breaking) per the
spec's own definition** (see `source/ballot-intro.html` line 34: "Moving
CodeSystems from core to THO is only a substantive change if (a) the
CodeSystem is bound to a Coding, CodeableConcept, or CodeableReference,
and (b) the URL changes"):

- `DeviceAssociation.relationship` (CodeableConcept, extensible binding) —
  bound CS URL changes from `http://hl7.org/fhir/deviceassociation-relationship`
  to `http://terminology.hl7.org/CodeSystem/deviceassociation-relationship`.
- `DeviceAssociation.statusReason` (CodeableConcept) — bound CS URL
  changes for `deviceassociation-status-reason` (no `/CodeSystem/` segment
  in the old URL!) to the new THO `…/CodeSystem/deviceassociation-status-reason`.
- `NutritionIntake.statusReason` (CodeableConcept) — bound CS URL
  changes from `http://hl7.org/fhir/CodeSystem/nutritionintake-status-reason`
  to `http://terminology.hl7.org/CodeSystem/nutritionintake-status-reason`.

The **ValueSet** canonical URLs are unchanged, so resource instances that
reference codings by `system` will still validate against the bound
ValueSet provided the consuming server resolves the THO CS.
Implementations that hardcoded the old `http://hl7.org/fhir/...` system
URL on `Coding.system` will need to update.

No element cardinalities, types, search parameters, or invariants are
changed by this ticket.

### Terminology Impact

This ticket *is* the terminology change. Concretely:

- Three CodeSystem canonical URLs change to live under
  `http://terminology.hl7.org/CodeSystem/` (with the leading
  `/CodeSystem/` segment, which two of the three did not previously
  have).
- Three ValueSet definitions are rewritten (only the
  `compose/include/system` value) to reference the new CS URLs.

  expected to be carried by THO.

  StructureDefinition `<binding>` elements.
