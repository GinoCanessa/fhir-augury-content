## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"Change the binding strength as suggested to preferred."*

The two `routeOfAdministration` bindings on `NutritionOrder` are currently
declared as `extensible` against `http://hl7.org/fhir/ValueSet/enteral-route`.
The committee accepted the recommendation to relax the binding strength to
`preferred` on both elements, on the basis that the referenced code system may
not be used by every jurisdiction or implementation community. The ValueSet
itself is unchanged; only the `<strength>` value on each `<binding>` flips
from `extensible` → `preferred`.

Affected elements:

- `NutritionOrder.additive.routeOfAdministration` (binding name
  `AdditiveRouteOfAdministration`)
- `NutritionOrder.enteralFormula.routeOfAdministration` (binding name
  `EnteralRouteOfAdministration`)

## Feature Proposal

### Problem Statement

Both `NutritionOrder.additive.routeOfAdministration` and
`NutritionOrder.enteralFormula.routeOfAdministration` are bound `extensible` to
the ValueSet `http://hl7.org/fhir/ValueSet/enteral-route`. An `extensible`
binding requires implementers to use a code from the ValueSet whenever a
suitable code exists in it, and only permits a non-listed code when the
ValueSet does not cover the required concept. Eric Haas (reporter) flagged
that the underlying code system(s) feeding `enteral-route` may not be
available, licensed, or recognised in some jurisdictions or implementation
communities, making the `extensible` requirement impractical in those
settings.

### Proposed Change

In `source/nutritionorder/structuredefinition-NutritionOrder.xml`, change the
`<strength>` element of each `<binding>` block on the two
`routeOfAdministration` elements from `extensible` to `preferred`.

Concretely, two single-line edits in the differential:

1. **`NutritionOrder.enteralFormula.routeOfAdministration`** (binding currently at line 1183–1190 in the cached clone):
   - Before: `<strength value="extensible"/>` (line 1187)
   - After:  `<strength value="preferred"/>`
   - The `<extension url=".../elementdefinition-bindingName"><valueString
     value="EnteralRouteOfAdministration"/></extension>`, the `<description>`
     ("Codes specifying the route of administration of enteral formula."), and
     the `<valueSet value="http://hl7.org/fhir/ValueSet/enteral-route"/>`
     reference all stay exactly as they are. Only the strength changes.

2. **`NutritionOrder.additive.routeOfAdministration`** (binding currently at line 1470–1477):
   - Before: `<strength value="extensible"/>` (line 1474)
   - After:  `<strength value="preferred"/>`
   - The `AdditiveRouteOfAdministration` binding name, the description ("Codes
     specifying the route of administration of additive."), and the same
     `enteral-route` ValueSet reference are all preserved.

No other changes are required: cardinality (0..*), type (CodeableConcept),
mappings (v2 ODS-4, RIM `.routeCode`), short/definition text, and the bound
ValueSet are all unchanged.

Resulting binding shape (both elements):

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="EnteralRouteOfAdministration"/>  <!-- or AdditiveRouteOfAdministration -->
  </extension>
  <strength value="preferred"/>
  <description value="Codes specifying the route of administration of enteral formula."/>  <!-- or additive -->
  <valueSet value="http://hl7.org/fhir/ValueSet/enteral-route"/>
</binding>
```

### Design Rationale

- **Direct application of the resolution.** The committee explicitly approved
  *"Change the binding strength as suggested to preferred."* The reporter's
  recommendation block of the ticket reads *"Change the binding to preferred"*,
  so the resolution is unambiguous and surgical.
- **Preferred vs. extensible.** `preferred` advises implementers to use the
  bound ValueSet when possible but does not require local code-system use when
  another code system is more appropriate to the jurisdiction. This directly
  addresses the reporter's concern that some implementation communities may
  not use the underlying code systems behind `enteral-route`.
- **ValueSet retained.** The resolution does not call for removing or
  redefining the ValueSet, so `http://hl7.org/fhir/ValueSet/enteral-route`
  (defined in `valueset-enteral-route.xml` in the same folder) is preserved
  untouched. The change is strictly a binding-strength relaxation.
- **Symmetry.** Both `routeOfAdministration` elements are conceptually
  parallel (one for enteral formula, one for additive) and the ticket lists
  both. They must be changed together to keep the resource internally
  consistent.
- **Alternatives considered and rejected.** Switching the binding to `example`
  would have been more permissive than the resolution requested. Splitting the
  ValueSet, expanding it, or pointing it at a jurisdiction-neutral code system
  would have been a larger change not endorsed by the resolution.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Houses the canonical `NutritionOrder` StructureDefinition whose
  binding strengths are being relaxed.
- **Affected Files:**
  - `source/nutritionorder/structuredefinition-NutritionOrder.xml` — change
    `<strength value="extensible"/>` → `<strength value="preferred"/>` inside
    the `<binding>` blocks of `NutritionOrder.enteralFormula.routeOfAdministration`
    (line 1187) and `NutritionOrder.additive.routeOfAdministration` (line 1474).
- **Change Scope:** Minor (two single-line edits in one file; no schema, no
  examples, no terminology changes).

### Breaking Changes

**No breaking changes identified.** Relaxing a binding from `extensible` to
`preferred` is, by FHIR convention, a non-breaking change: every instance that
previously satisfied the `extensible` rule (i.e., used a code from
`enteral-route` or supplied a code outside the ValueSet only when no listed
concept fit) trivially satisfies the more permissive `preferred` rule.
Conformance resources, profiles, and instances that already constrained
themselves to `enteral-route` continue to validate. Implementers gain
flexibility; none lose capability.

The minor downstream consequences worth noting (but not breaking):

- Profiles in IGs that explicitly re-bind these elements to `extensible`
  remain valid and continue to enforce their stricter rule — they are simply
  no longer redundant.
- Validators that previously emitted information-level "extensible binding"
  messages will instead emit "preferred binding" messages.

### Related Specifications

- `valueset-enteral-route.xml` (in the same folder) is referenced unchanged.
- No other resource in `source/` references either of the two affected
  elements directly; the change is contained within `NutritionOrder`.
- Downstream IGs that profile `NutritionOrder.additive` or
  `NutritionOrder.enteralFormula` (e.g., nutrition-related US realm IGs) may
  wish to re-tighten the binding in their own profiles, but that work is out
