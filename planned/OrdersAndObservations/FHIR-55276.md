## Resolution Summary

The applied resolution is **Persuasive with Modification**:

> Motion to update the `specimen-role` valueset to be based on the THO code
> system (`https://terminology.hl7.org/CodeSystem-v2-0369.html`) and
> register the `specimen-role` valueset in THO.

Concretely this requires two coordinated changes:

1. Re-base `ValueSet/specimen-role` (currently bound at preferred strength
   to `Specimen.role`) so its `compose.include.system` is the THO
   `v2-0369` CodeSystem (`http://terminology.hl7.org/CodeSystem/v2-0369`)
   instead of the FHIR-internal `http://hl7.org/fhir/specimen-role`.
2. Register the `specimen-role` ValueSet definition in the THO
   (`HL7/UTG`) repository so the canonical
   (`http://hl7.org/fhir/ValueSet/specimen-role`) is owned/published from
   THO going forward.

A reviewer (rikim, 2026-01-27) flagged in-flight that the existing FHIR
`specimen-role` CodeSystem uses the **lowercase** v2-0369 abbreviations
(`b`, `c`, `e`, `f`, `o`, `p`, `q`, `r`, `v`) while the THO `v2-0369`
CodeSystem publishes them in the canonical V2 case (typically uppercase
in v2 tables). Whether this rebase is a pure URL swap or a **code-value
change** as well is the one outstanding modification flagged on the
ticket and is captured under Open Questions below.

## Feature Proposal

### Problem Statement

Per the ticket: the Specimen resource references several value sets
that are THO-migration candidates. The specific case called out is
`CodeSystem: specimen-role (SpecimenRole) referenced by ValueSet
specimen-role bound to element Specimen.role`. Today this code system
is defined inside FHIR core (`http://hl7.org/fhir/specimen-role`,
`source/specimen/codesystem-specimen-role.xml`) even though the same
concept set is already standardised in HL7 V2 table 0369 and published
through THO (`v2-0369`,
`http://terminology.hl7.org/CodeSystem/v2-0369`). Keeping a FHIR-only
duplicate prevents reuse from V2, CDA, and other product families and
forces any future expansion of the concept set to ride the FHIR
publication cycle.

### Proposed Change

**HL7/fhir (FhirCore) — re-base the ValueSet:**

- Edit `source/specimen/valueset-specimen-role.xml`:
  - Keep the canonical URL `http://hl7.org/fhir/ValueSet/specimen-role`
    (the resolution explicitly *registers* this canonical in THO; it is
    not being renamed).
  - In the `<compose>` block, replace the current
    `<include><system value="http://hl7.org/fhir/specimen-role"/></include>`
    with an include that references the THO code system —
    `<include><system value="http://terminology.hl7.org/CodeSystem/v2-0369"/></include>`.
  - Drop `<immutable value="true"/>` if THO will own and may extend the
    set (otherwise leave; see Open Questions).
  - Update `<date>` and bump publisher/contact text only if the project's
    R6 publication conventions require it.

- Retire `source/specimen/codesystem-specimen-role.xml` from the FHIR
  core publication. The FhirCore briefing's general gotcha applies:
  removing a top-level artifact requires updating its enclosing list and
  any cross-references rather than silently deleting the file.
  Specifically:
  - Remove the `codesystem-specimen-role.xml` file (or mark it
    deprecated per the project's standard retirement pattern).
  - Update `source/specimen/list-Specimen-packs.xml` /
    `list-Specimen-examples.xml` if either enumerates the retired
    CodeSystem (verify before edit — they do not currently enumerate it
    in this clone).
  - Remove the `codesystem-specimen-role` entry from
    `source/specimen/implementationguide-Specimen-core.xml` if listed
    there (verify against the IG file).
  - Search for and update any inbound links of the form
    `codesystem-specimen-role.html` or
    `http://hl7.org/fhir/specimen-role` elsewhere in `source/` so they
    redirect to the v2-0369 page on terminology.hl7.org.

- Edit `source/specimen/structuredefinition-Specimen.xml` (line 312–331)
  if the binding strength or `bindingName` needs adjustment — but the
  resolution does **not** ask for a strength change, so leave
  `strength=preferred` and `valueSet=http://hl7.org/fhir/ValueSet/specimen-role`
  as-is. The ValueSet canonical is unchanged; only its composition
  changes.

- Add an R6 release note (per the `R6-Release-note` label) describing
  the rebase and pointing implementers at v2-0369 for the authoritative
  code list.

**HL7/UTG — register the ValueSet in THO:**

- Add a `ValueSet/specimen-role` definition under the appropriate THO
  authoring path (typically `input/sourceOfTruth/ValueSets/` for HL7
  external value sets) whose `compose.include.system` is
  `http://terminology.hl7.org/CodeSystem/v2-0369`, and whose `url`
  remains `http://hl7.org/fhir/ValueSet/specimen-role` so existing
  bindings continue to resolve.
- No new CodeSystem needs to be added to UTG — `v2-0369` already exists
  in THO.

### Design Rationale

- **Reuse over duplication.** The original ticket explicitly frames the
  test as "if it's realistic for the value set definition to need to
  change in the coming years, or the value sets might be useful in CDA,
  V2 or other specs". Specimen role is a textbook fit: V2 has owned the
  same concept set for years (table 0369), and rebasing onto v2-0369
  lets V2/CDA implementers reuse the same vocabulary natively rather
  than maintaining a FHIR-only parallel.
- **Keep the canonical.** Registering the ValueSet (not just a new one)
  in THO preserves the existing `http://hl7.org/fhir/ValueSet/specimen-role`
  canonical so all current `Specimen.role` bindings, examples, and
  downstream IG dependencies continue to resolve unchanged. Only the
  underlying CodeSystem moves.
- **Preferred binding stays preferred.** The resolution makes no
  binding-strength change. `Specimen.role` is preferred today and
  remains preferred — implementers were never required to use these
  codes, so swapping the underlying system does not raise the bar.
- **Lowercase vs. v2 case.** The rikim comment surfaces a real
  modification: today's FHIR CodeSystem uses single-letter lowercase
  codes (`b`, `c`, `e`, `f`, `o`, `p`, `q`, `r`, `v`) while v2-0369
  publishes the same concepts. This must be reconciled before
  publication (see Open Questions); the resolution motion itself is
  silent on case and that is the "with Modification" hook.

### Breaking Changes

- **Yes — code value change is possible (and likely).** The current
  FHIR `specimen-role` CodeSystem uses single-letter lowercase codes
  (`b`, `c`, `e`, `f`, `o`, `p`, `q`, `r`, `v`). The THO `v2-0369`
  CodeSystem publishes the same concepts in their canonical V2 form. If
  the rebase keeps lowercase via per-concept inclusion/aliasing the
  change is non-breaking on the wire; if the rebase adopts v2-0369's
  native codes (which the resolution comment notes "would align better
  with implementations in V2") then existing instances coding
  `Specimen.role` as `{system: http://hl7.org/fhir/specimen-role,
  code: p}` will no longer validate against the rebased ValueSet — that
  is a breaking change for R6 implementers, even at a `preferred`
  binding, because the system URL changes regardless.
- **Yes — system URL change.** Even with a pure URL swap and no
  code-value change, the `system` element in coded `Specimen.role`
  values changes from `http://hl7.org/fhir/specimen-role` to
  `http://terminology.hl7.org/CodeSystem/v2-0369`. Validators that
  pin instances to the FHIR-side system will fail; this is breaking at
  the data level even though the binding strength remains `preferred`.
- **Retirement of a normative-status CodeSystem.** The current
  CodeSystem carries `standards-status=normative` and `fmm=2`. Retiring
  a normative artifact between R5 and R6 must be called out in the
  release notes and in the spec's normative-content tracking.
