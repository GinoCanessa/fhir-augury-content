## Resolution Summary

**Resolution:** Persuasive with Modification.

Quoting the resolution description verbatim:

> Motion to add an example binding to a new FHIR value set to cover both of the fields mentioned in these tickets. The code system would be existing SNOMED CT with a filter bound to 64572001 | Disease (disorder) and 404684003 | Clinical finding (finding) for both of these.

Concretely, the resolution requires:

1. Create a **new ValueSet** defined in FHIR Core that composes SNOMED CT concepts using an `is-a` filter against both `64572001` (Disease (disorder)) and `404684003` (Clinical finding (finding)).
2. Attach that ValueSet as an **example-strength binding** to `DeviceDefinition.guideline.indication` (this ticket) and — per the applied-vote comment and companion ticket FHIR-54864 — also to `DeviceDefinition.guideline.contraindication`. The two fields **share the same ValueSet**.

Note on the Zulip/comment reference to `http://hl7.org/fhir/ValueSet/condition-code`: that was an early suggestion ("Could do an example binding to …"). The resolution supersedes it by specifying a *new* ValueSet spanning both SNOMED CT filters (condition-code is limited to `404684003`). `condition-code` is a useful pattern reference only.

## Feature Proposal

### Problem Statement

`DeviceDefinition.guideline.indication` is a `0..* CodeableConcept` describing "A clinical condition for which the device was designed to be used" (source: `source/devicedefinition/structuredefinition-DeviceDefinition.xml` line 1392–1402). The element has **no `<binding>`** — FHIR's authoring convention is that every `CodeableConcept` element carries at least an example binding so implementers have guidance on which terminology to use. The sibling `DeviceDefinition.guideline.contraindication` (lines 1403–1413) has the same gap and is tracked by FHIR-54864; the sibling `DeviceDefinition.guideline.warning` (lines 1414–1431) already demonstrates the expected pattern (example binding to `devicedefinition-warnings-example`).

### Proposed Change

**1. New ValueSet (`HL7/fhir`)**

Create a new ValueSet file under `source/devicedefinition/` composing SNOMED CT concepts via two `is-a` filters. Proposed metadata (modelled on the existing sibling `valueset-devicedefinition-warnings-example.xml` and on `source/condition/valueset-condition-code.xml`):

- **id:** `devicedefinition-indication` *(see Open Questions — the resolution did not specify an id; this one mirrors the field name, is shared by indication + contraindication, and fits the sibling pattern `devicedefinition-<aspect>`.)*
- **url:** `http://hl7.org/fhir/ValueSet/devicedefinition-indication`
- **name:** `DeviceDefinitionIndication` **title:** `Device Indication / Contraindication Codes`
- **status:** `active`, **experimental:** `true`
- **structuredefinition-wg** extension: `oo` (matches `devicedefinition-warnings-example`).
- **structuredefinition-standards-status** extension: `informative` (matches sibling).
- **copyright:** reuse the standard SNOMED CT affiliate-licence boilerplate copied from `source/condition/valueset-condition-code.xml` lines 38–39.
- **compose.include:** two `include` blocks, both `<system value="http://snomed.info/sct"/>`, each with a `<filter>` of `<property value="concept"/> <op value="is-a"/>`:
  - `<value value="64572001"/>` — Disease (disorder)
  - `<value value="404684003"/>` — Clinical finding (finding)

Illustrative skeleton (XML authored form expected by the bespoke publisher):

```xml
<ValueSet xmlns="http://hl7.org/fhir">
  <id value="devicedefinition-indication"/>
  <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-wg">
    <valueCode value="oo"/>
  </extension>
  <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
    <valueCode value="informative"/>
  </extension>
  <url value="http://hl7.org/fhir/ValueSet/devicedefinition-indication"/>
  <version value="6.0.0"/>
  <name value="DeviceDefinitionIndication"/>
  <title value="Device Indication / Contraindication Codes"/>
  <status value="active"/>
  <experimental value="true"/>
  <publisher value="HL7 International / Orders and Observations"/>
  <description value="Example codes describing clinical conditions for which a device is indicated or contraindicated."/>
  <copyright value="This resource includes content from SNOMED Clinical Terms® (SNOMED CT®) which is copyright of the International Health Terminology Standards Development Organisation (IHTSDO). Implementers of these specifications must have the appropriate SNOMED CT Affiliate license - for more information contact http://www.snomed.org/snomed-ct/get-snomed-ct or info@snomed.org"/>
  <compose>
    <include>
      <system value="http://snomed.info/sct"/>
      <filter>
        <property value="concept"/>
        <op value="is-a"/>
        <value value="64572001"/>
      </filter>
    </include>
    <include>
      <system value="http://snomed.info/sct"/>
      <filter>
        <property value="concept"/>
        <op value="is-a"/>
        <value value="404684003"/>
      </filter>
    </include>
  </compose>
</ValueSet>
```

**2. Binding on `DeviceDefinition.guideline.indication` (this ticket)**

Add a `<binding>` child to the element at `source/devicedefinition/structuredefinition-DeviceDefinition.xml` line 1392 (before the closing `</element>` on line 1402), modelled on the `warning` element's binding block (lines 1424–1431):

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="DeviceDefinitionIndication"/>
  </extension>
  <strength value="example"/>
  <description value="Clinical conditions for which a device is indicated (or contraindicated)."/>
  <valueSet value="http://hl7.org/fhir/ValueSet/devicedefinition-indication"/>
</binding>
```

**3. Companion binding on `DeviceDefinition.guideline.contraindication` (FHIR-54864, out of scope here but must be coordinated)**

The same `<binding>` block is to be attached to the element at lines 1403–1413 as part of FHIR-54864. Because the applied-vote comment explicitly ties the two tickets ("should share the same valueset"), the ValueSet id/url chosen here must accommodate both fields (reflected in the proposed title/description above).

### Design Rationale

- **Example strength** is the only defensible choice given the resolution's wording ("*example* binding") and the reality that device indications/contraindications vary across jurisdictions and device types. This matches the sibling `guideline.warning` element which also binds example.
- **Shared ValueSet** follows the applied-vote comment from JIRAUSER19407 linking FHIR-54863 and FHIR-54864; duplicating the same composition into two separate ValueSets would be churn with no implementer benefit.
- **SNOMED CT filter choice** (`64572001` + `404684003`) comes directly from the resolution. Using `is-a` filters on those parents produces a wide but semantically coherent set spanning both diseases/disorders and clinical findings, which is what both indication and contraindication realistically need.
- **Publisher extensions (`wg=oo`, `standards-status=informative`, experimental=true)** match the sibling `devicedefinition-warnings-example` ValueSet — staying consistent with Orders & Observations authoring for this resource.
- **Not reusing `condition-code`:** `condition-code` includes only the `404684003` filter (plus one absent concept). Broadening it to add `64572001` would change existing `Condition.code` semantics — a much larger, unrelated impact — so creating a dedicated DeviceDefinition-scoped VS is the lower-risk choice and aligns with the resolution wording ("a new FHIR value set").
