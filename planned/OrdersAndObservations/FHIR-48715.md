## Feature Proposal

### Problem Statement

`Specimen.condition` currently has an **extensible** binding to the
HL7 v2 table 0493 value set
(`http://terminology.hl7.org/ValueSet/v2-0493`). v2 itself binds SPM-24
as **example** with only a few suggested codes, so the FHIR extensible
strength is stricter than the upstream v2 contract and prevents other
HL7 product families (notably US Core) from migrating to a more
complete SNOMED CT-based vocabulary while remaining conformant. OO has
prepared a SNOMED CT international subset that should be offered
alongside the v2 table for cross-product reuse.

### Proposed Change

1. **Relax the primary binding** on `Specimen.condition`:
   - Path: `Specimen.condition`
   - Binding strength: `extensible` → `example`
   - Value set (unchanged): `http://terminology.hl7.org/ValueSet/v2-0493`
   - Binding name (unchanged, e.g., `SpecimenCondition` per current spec)

2. **Add an additional binding** on `Specimen.condition.binding`
   pointing at a new SCT-based value set:
   - `additional.purpose`: `starter`
   - `additional.valueSet`: `http://hl7.org/fhir/ValueSet/specimen-condition-sct`
   - `additional.documentation`: "SNOMED CT International descendants of
     881221000124104 Specimen condition finding (finding)."

3. **Create a new ValueSet** `specimen-condition-sct`
   (URL `http://hl7.org/fhir/ValueSet/specimen-condition-sct`) with:
   - Profile: `http://hl7.org/fhir/StructureDefinition/shareablevalueset`
   - WG extension: `oo`
   - Standards-status: `trial-use`; FMM: `2`
   - Status: `draft`; experimental: `false`
   - Compose: include `http://snomed.info/sct` filter
     `concept descendent-of 881221000124104`
   - Standard SNOMED CT copyright block

   (Full XML draft is in the 2025-09-14 ticket comment.)

### Design Rationale

- Aligns FHIR's binding strength with the upstream v2 binding
  (`example`), removing an unintended over-constraint that blocks IGs
  from declaring richer terminology choices while staying conformant.
- Keeps the v2 value set as the primary binding so existing
  cross-product mappings continue to work.
- Uses an **additional binding with `purpose=starter`** rather than
  replacing the primary binding, so implementers see SNOMED CT as a
  recommended starting point without forcing it on all jurisdictions.
- Anchors the SCT value set on the parent concept "Specimen condition
  finding (finding)" so the value set automatically grows with future
  SCT International releases (intensional definition, no enumerated
  members to maintain).

update their tooling. No type, cardinality, or required-element changes
