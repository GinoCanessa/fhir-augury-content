## Resolution Summary

The resolution is **Not Persuasive with Modification**. From the ticket's resolution description:

> An extension is not needed, the approach in the comments should be added as an example in the specification on how to add an annotation as an observation for a SampledData observation.

In other words: **no schema change** to `Observation`, `Observation.component`, or the `SampledData` datatype; and **no new extension**. Instead, the Observation specification must be updated to show — by example — how to annotate samples within a `SampledData` waveform using paired Observations that reference the waveform via `derivedFrom`.

The O&O / Dev WG discussion on the ticket converged on two equally-acceptable patterns (both to be included per the 2025-07-02 comment from E. Silver: *"Dev WG finds both options as acceptable (and thinks there may be additional approaches), and recommends including both"*):

- **Option A — "Period annotation" Observation.** A second Observation with `code` set to the annotation concept (e.g., atrial flutter, pacer beat, inspiration start), `derivedFrom` pointing at the waveform Observation, and `valuePeriod` identifying the annotated time window on the source waveform.
- **Option B — "Coded SampledData" annotation Observation.** A second Observation with the same `origin` / `interval` / `intervalUnit` / `dimensions` as the source waveform and a `valueSampledData` whose `data` is a per-sample code stream (e.g., `N N N A A A A N N …`), with `codeMap` pointing at a (typically contained) `ConceptMap` that defines the annotation codes. Again, `derivedFrom` references the source waveform. Option B is already enabled by the existing `SampledData.codeMap` facility described in `source/datatypes.html` (SampledData section); no datatype change is required.

The resolution explicitly rejects the reporter's original proposal to extend `Observation.component` with a sample-index pointer. It also rejects the intermediate proposal (Corey Spears, 2024-12-02) to define a dedicated extension on `Observation.focus` / `Observation.note` carrying sample-index ranges — that extension is *not* to be built.

## Feature Proposal

### Problem Statement

For `SampledData`-valued Observations representing waveforms (e.g., an ECG snippet from a monitor, a flow/pressure trace from a ventilator), implementers need a documented, interoperable way to mark that *this particular sample* or *this sample range* represents a clinically-relevant event — a pacemaker beat, the start of inspiration, an episode of atrial flutter, etc.

The original ticket (2019) proposed extending `Observation.component` with a sample-pointer. The current `Observation` specification:

- Defines `Observation.value[x]` including `valueSampledData` (`source/observation/observation-notes.xml:340`).
- Defines `Observation.hasMember` / `Observation.derivedFrom` for linking related Observations (`source/observation/observation-notes.xml:78–96`).
- Ships an EKG `SampledData` example (`source/observation/observation-example-sample-data.xml` → `Observation/ekg`, registered as `observation-example-sample-data` in `list-Observation-examples.xml`).

…but does **not** show an implementer how to annotate a location or range within that waveform. There is no guidance paragraph and no worked example covering the annotation use case. Implementers therefore invent one-off extensions or misuse `component`, harming interoperability for ECG/ventilator waveform reporting.

### Proposed Change

All edits land in the FHIR Core repository (`HL7/fhir`) under `source/observation/`. No StructureDefinition differential changes. No extension definitions. No terminology/UTG changes.

1. **Add Option A example — period-annotation Observation.** New file `source/observation/observation-example-sample-data-annotation-period.xml` (Observation id `ekg-annotation-period`), derived from the existing `Observation/ekg`:
   - `status` = `final`
   - `code` = LOINC `LA17085-4` "Atrial flutter" (the condition identified on the waveform), aligned with the ticket comment's worked example
   - `subject` = `Patient/f001` (same as the existing EKG example)
   - `effectiveDateTime` = a point inside the source waveform's timeline
   - `performer` = `Practitioner/f005` (same as the existing EKG example)
   - `valuePeriod` with `start`/`end` set to the annotated interval within the waveform
   - `derivedFrom` = `Reference(Observation/ekg)`
2. **Add Option B example — coded-SampledData annotation Observation.** New file `source/observation/observation-example-sample-data-annotation-coded.xml` (Observation id `ekg-annotation-coded`), derived from `Observation/ekg`:
   - `contained` holds a `ConceptMap` (id `ekg-Mapping`) mapping local codes (`N`, `A`, …) to LOINC concepts such as `LA25095-3` "Normal Sinus Rhythm" / `LA17085-4` "Atrial flutter", `status` = `active`, `experimental` = `true`, single `group` with `target` = LOINC.
   - `code` identical or parallel to the source waveform's observation code
   - `valueSampledData` with the **same** `origin.value`, `interval`, `intervalUnit`, and `dimensions` as `Observation/ekg`, `codeMap` = canonical reference (`#ekg-Mapping`) to the contained ConceptMap, and `data` a per-sample code string (e.g., `"N N N N … A A A A … N N N"`) whose length and positions align to the source waveform.
   - `derivedFrom` = `Reference(Observation/ekg)`
3. **Register both new examples** in `source/observation/list-Observation-examples.xml` with descriptive `valueString` titles ("EKG annotation via derivedFrom + valuePeriod" and "EKG annotation via derivedFrom + coded valueSampledData"), following the existing entry pattern used for `observation-example-sample-data`.
4. **Add a narrative paragraph** to `source/observation/observation-notes.xml`, co-located with the existing SampledData bullet (currently at line 340 of `observation-notes.xml`) and/or the `hasMember` / `derivedFrom` section (lines 78–96), explaining:
   - Annotations on a `SampledData` waveform are represented as a **separate Observation** that references the waveform via `derivedFrom`.
   - **Option A** — point/range annotations: use `code` for the annotation concept and `valuePeriod` (or `effectivePeriod`) to mark the annotated interval; link with `derivedFrom` to the waveform Observation. Point to `observation-example-sample-data-annotation-period`.
   - **Option B** — per-sample annotations across the whole (or a clipped portion of the) series: use `valueSampledData` with the same `origin` / `interval` / `dimensions` as the source, `codeMap` pointing at a `ConceptMap` of annotation codes, and a parallel `data` string. Cross-link to the existing `SampledData.codeMap` description in `source/datatypes.html` (SampledData section) and point to `observation-example-sample-data-annotation-coded`.
   - Explicitly state that an extension on `Observation.component`, `Observation.focus`, or `Observation.note` is **not** needed for this use case.
5. **(Optional, editorial)** Add a forward link from the SampledData section of `source/datatypes.html` (around line 1322, after the `codeMap` / `ConceptMap` paragraph) to the new annotation narrative in `observation.html#notes` so datatype readers discover the Option B worked example.

No changes to:
- `source/observation/structuredefinition-Observation.xml` (no element changes).
- `source/datatypes/sampleddata.xml` (per briefing, this is a legacy spreadsheet, and no datatype change is required anyway).
- Search parameters (`bundle-Observation-search-params.xml`) — the new examples rely only on already-searchable elements (`code`, `subject`, `derived-from`).

### Design Rationale

- **Pure example + narrative change matches the resolution.** The resolution description is explicit: "An extension is not needed, the approach in the comments should be added as an example." Adding examples and narrative — with no schema or extension change — is the minimum, faithful implementation of that resolution.
- **Two examples, not one.** The 2025-07-02 Dev WG comment recommended including both patterns because the right choice depends on the clinical question (is the annotation a discrete interval, or a per-sample classification?). Shipping both worked examples prevents implementers from assuming only one is conformant.
- **Reuse existing machinery.** Option A uses the already-documented `derivedFrom` pattern (`observation-notes.xml:78–96`), which is also used by `observation-example-bmi-using-related`. Option B uses `SampledData.codeMap` + contained `ConceptMap`, which is already specified in `datatypes.html` (lines 1322–1331) but currently has no Observation-level worked example. Both reuses avoid new conformance surface.
- **Examples build on `Observation/ekg`.** Anchoring both new examples to `Observation/ekg` (the existing `observation-example-sample-data`) means readers see exactly how annotations compose with an already-shipped waveform example, and it mirrors the worked examples in Eric Silver's 2025-06-25 comment.
- **Rejected alternatives** (recorded on the ticket):
  - Extending `Observation.component` with a sample-pointer element — rejected as unnecessary; `derivedFrom` + `valuePeriod` (Option A) already expresses "this observation annotates samples in that observation."
  - A bespoke extension on `Observation.focus` / `Observation.note` carrying a sample-index or sample-range (Corey Spears proposal, 2024-12-02) — no longer pursued; the resolution states no extension is needed.
  - Relying solely on `observation-example-sample-data` as-is (R6 already allows `valueSampledData` — see `rikim` 2024-02-01 comment) — insufficient, because the existing example does not illustrate annotation.

### Step-by-Step Tasks

#### HL7/fhir

1. **Author the Option A example (period annotation).**
   - File: `source/observation/observation-example-sample-data-annotation-period.xml`
   - Action: Create a new Observation XML instance with `id = "ekg-annotation-period"` following the worked FSH/JSON in the 2025-06-25 ticket comment.
   - Details: Required elements:
     - `<status value="final"/>`
     - `<code>` with LOINC coding (`system = http://loinc.org`, `code = LA17085-4`, `display = "Atrial flutter"`).
     - `<subject><reference value="Patient/f001"/><display value="P. van de Heuvel"/></subject>`.
     - `<effectiveDateTime value="2015-02-19T09:30:35+01:00"/>` (a time within the source `Observation/ekg`'s window).
     - `<performer><reference value="Practitioner/f005"/><display value="A. Langeveld"/></performer>`.
     - `<valuePeriod><start value="2015-02-19T09:30:35.240+01:00"/><end value="2015-02-19T09:30:35.320+01:00"/></valuePeriod>` — the annotated interval on the waveform.
     - `<derivedFrom><reference value="Observation/ekg"/></derivedFrom>`.
     - Root element `<Observation xmlns="http://hl7.org/fhir">`, matching the header style of sibling examples (see `observation-example-sample-data.xml`).

2. **Author the Option B example (coded SampledData annotation).**
   - File: `source/observation/observation-example-sample-data-annotation-coded.xml`
   - Action: Create a new Observation XML instance with `id = "ekg-annotation-coded"` following the worked FSH/JSON in the 2025-06-25 ticket comment.
   - Details: Required structure:
     - `<contained>` holds a `<ConceptMap>` with `id = "ekg-Mapping"`, `status = "active"`, `experimental = "true"`, one `<group>` with `target = "http://loinc.org"`, and two `<element>` entries (`N` → `LA25095-3` "Normal Sinus Rhythm" `equivalent`; `A` → `LA17085-4` "Atrial flutter" `equivalent`).
     - `<status value="final"/>`.
     - `<code>` parallel to the source waveform's code (e.g., LOINC `76281-5` "Type of arrhythmia on EKG") — the `code` identifies the annotation's subject, not the annotation value.
     - `<subject>` / `<effectiveDateTime>` / `<performer>` matching the source `Observation/ekg` (reuse `Patient/f001`, `Practitioner/f005`, `2015-02-19T09:30:35+01:00`).
     - `<valueSampledData>` with `<origin><value value="2048"/></origin>`, `<interval value="10"/>`, `<intervalUnit value="ms"/>`, `<dimensions value="1"/>`, `<codeMap value="#ekg-Mapping"/>` (fragment reference to the contained ConceptMap), and `<data value="N N N N N N N N … A A A A A A A A A … N N N …"/>` — the code string must have the same sample count as (or be a defensible clip of) the source waveform to make the mapping meaningful.
     - `<derivedFrom><reference value="Observation/ekg"/></derivedFrom>`.

3. **Register both new examples in the Observation example list.**
   - File: `source/observation/list-Observation-examples.xml`
   - Action: Insert two new `<entry>` elements immediately after the existing `observation-example-sample-data` entry (which references `Observation/ekg`).
   - Details: Each `<entry>` follows the established pattern:
     ```xml
     <entry>
       <extension url="http://hl7.org/fhir/build/StructureDefinition/description">
         <valueString value="EKG annotation via derivedFrom + valuePeriod (Option A)"/>
       </extension>
       <extension url="http://hl7.org/fhir/build/StructureDefinition/title">
         <valueString value="observation-example-sample-data-annotation-period"/>
       </extension>
       <item>
         <reference value="Observation/ekg-annotation-period"/>
         <display value="EKG Annotation (Period)"/>
       </item>
     </entry>
     ```
     and analogously for `observation-example-sample-data-annotation-coded` / `Observation/ekg-annotation-coded` / "EKG Annotation (Coded SampledData)".

4. **Add narrative explaining both annotation patterns.**
   - File: `source/observation/observation-notes.xml`
   - Action: Insert a new subsection (e.g., `<a name="sampleddata-annotation"></a><h3>Annotating SampledData Waveforms</h3>`) either (a) immediately after the `valueSampledData` bullet at line 340 inside the Value[x] and Datatypes `<ul>`, or (b) as a sibling `<h3>` inside the section that contains `Observation.hasMember and Observation.derivedFrom` (around line 78), whichever reviewers prefer for flow.
   - Details: The paragraph must cover:
     - Statement that annotating one or more samples in a `SampledData` waveform is done with a **separate Observation** that uses `derivedFrom` to reference the waveform Observation — no extension and no change to `Observation.component` is required.
     - Option A: `code` = annotation concept, `valuePeriod` (or `effectivePeriod`) = annotated interval on the source waveform; point to `observation-example-sample-data-annotation-period.html`.
     - Option B: `valueSampledData` with the same `origin` / `interval` / `intervalUnit` / `dimensions` as the source waveform; `codeMap` references a (typically contained) `ConceptMap` enumerating the annotation codes; `data` is the parallel per-sample code stream. Cross-link to the SampledData section of `datatypes.html` for the `codeMap` semantics. Point to `observation-example-sample-data-annotation-coded.html`.
     - Note that `hasMember` can group the source waveform and its annotation Observations when they are reported together.

5. **(Optional) Cross-link from the SampledData datatype page.**
   - File: `source/datatypes.html`
   - Action: After the `codeMap` / `ConceptMap` paragraph (around lines 1322–1331), add a short sentence: "For a worked example of `codeMap` used to annotate samples in an Observation waveform, see <a href="observation.html#sampleddata-annotation">Annotating SampledData Waveforms</a> in the Observation resource notes." These are stable LOINC codes and a reasonable choice, but the O&O co-chairs may prefer a less arrhythmia-specific pair (e.g., pacer beat / inspiration start, per the original ticket use cases) to match the motivating examples. Confirm choice before merging.
- **Placement of the new narrative.** Should "Annotating SampledData Waveforms" sit (a) next to the `valueSampledData` bullet in the Value[x] and Datatypes list (`observation-notes.xml:340`), (b) as a new `<h3>` sibling to the existing `Observation.hasMember and Observation.derivedFrom` section (~line 78–96), or (c) both (brief mention at 340, full section near 78)? - **Gotcha: legacy datatype spreadsheet.** `source/datatypes/sampleddata.xml` is an Excel spreadsheet (per the briefing's "Warnings / Gotchas"); do not attempt to add the annotation narrative there. Keep all narrative changes in `source/datatypes.html` and `source/observation/observation-notes.xml`.
- **Scope of `hasMember`.** The narrative should mention that when a waveform and its annotation Observations are reported as a set, `hasMember` on a grouping Observation is a reasonable pattern (already documented at `observation-notes.xml:78–96`). Confirm whether reviewers want a third example illustrating the `hasMember` grouping, or whether the two `derivedFrom` examples suffice.
