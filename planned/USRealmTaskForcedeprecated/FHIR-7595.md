## Feature Proposal

### Problem Statement

In 2015 (the comment refers to 2015-May ballot core #886) the reporter
observed that US-realm implementation guides were each independently
constraining common FHIR datatypes (the example given is
`Patient.address`) in separate, overlapping ways. The reporter proposed
that US Core itself define constrained datatypes (or "constrainable"
profiles) so downstream IGs could reuse a single canonical constraint.

### Proposed Change

The committee did **not** persuade the technical proposal. Instead the
disposition asks USRSC (the US Realm Steering Committee) to:

1. Establish a process to receive and approve US Realm profiles.
2. Nominate certain approved profiles as **US Realm Base Profiles**.
3. Direct future US Realm FHIR artifacts to start from those base
   profiles for consistency where possible and reasonable.

These three actions are **organizational / process changes for USRSC**.
There is no derived FHIR Core change request, no US Core artifact
change request, and no terminology change request embedded in the
disposition.

### Design Rationale

- The technical mechanism the reporter asked for (constrained datatype
  profiles) already exists in FHIR (`StructureDefinition` with
  `kind=complex-type`, `derivation=constraint`), so no spec change is
  needed to enable it (Grahame's comment).
- The substantive disagreement is about **which** US-realm constraints
  should be canonical, which is a stewardship question, not an editorial
  one — hence the recommendation to delegate to USRSC.
- The work group of record (US Realm Task Force) has since been
  deprecated, and the ticket was last touched only to clean up an
  unrelated `Related Artifact(s): DeviceMetric` reference (see comments
  from Elliot Silver and Lloyd McKenzie, 2025-07-03 and Javier Espina,
  2026-01-12). No technical follow-up has been requested in the
  intervening ten years.

## Implementation Plan

### Prerequisites

This ticket does not produce a technical implementation task for any
HL7-maintained repository. The action items live with the US Realm
Steering Committee (USRSC):

- USRSC must agree to own and operate a process for receiving and
  approving US Realm profiles.
- USRSC must agree on the criteria and mechanism for nominating "US
  Realm Base Profiles".

These are not actionable inside the FHIR Core or US Core repositories.

### Step-by-Step Tasks

**No repository changes are required by this resolution.**

The resolution text is an instruction to USRSC, not a normative change
to any balloted artifact. There is no `StructureDefinition`, page,
extension, ValueSet, CodeSystem, search parameter, or operation to
edit. No PRs or commits referencing this ticket exist in cached
repositories.

If, in the future, USRSC produces:

- A published process document → it would land in a USRSC governance
  space (Confluence / HL7 website), not in `HL7/fhir`.
- A nominated US Realm Base Profile → it would be authored in the
  relevant US Realm IG repository (e.g. `HL7/US-Core` or a new IG
  repository) under that IG's normal authoring workflow, **as a
  separate Jira ticket**, and would not be tracked under this ticket.
