## Resolution Summary

InM (Infrastructure & Messaging) agreed that `MessageHeader.response.code` needs an
"escape valve" so the element can be populated in cases that are not strictly
`ok`, `transient-error`, or `fatal-error`. The approved resolution
(**Persuasive with Modification**) makes three concrete changes to FHIR Core
in HL7/fhir:

1. **Add a new `non-fatal-error` concept** to the `http://hl7.org/fhir/response-code`
   CodeSystem with display **"Non-Fatal Error"** and definition
   *"The message was accepted and processed but with one or more errors. The
   `MessageHeader.response.detail` SHALL describe the issue."* This is the
   replacement for the requested `unknown` / `other` codes — InM explicitly
   rejected those generic escape-valves in favour of a single, meaningful new
   concept.
2. **Change the datatype of `MessageHeader.response.code`** from `code` to
   `CodeableConcept`. This supports future codes from other code systems
   without further schema changes.
3. **Loosen the binding** so the new `CodeableConcept` element is **extensible**
   (rather than required) to the `response-code` ValueSet. The resolution
   states the element should "be extensible to allow for other errors codes
   to be considered in the future."

The resolution table also corrects the existing `fatal-error` definition to
reference `MessageHeader.response.detail` (singular, the actual SD element
id is `MessageHeader.response.details` but the prose in the table uses the
narrative phrase "the response narrative" — see Open Questions below).

A draft implementation already exists as PR
[HL7/fhir#4007](https://github.com/HL7/fhir/pull/4007) by isaacvetter
("MessageHeader.response.code and response.details"). This plan is written
against the current `master` clone and should be cross-checked with that PR
before opening a new one.

## Feature Proposal

### Problem Statement

`MessageHeader.response.code` is a **mandatory** (1..1) element with a
**required** binding to the `http://hl7.org/fhir/ValueSet/response-code`
ValueSet, which contains only three concepts: `ok`, `transient-error`,
`fatal-error`. There is no concept for situations such as:

- A relay agent that has accepted custody and forwarded the message but
  cannot yet say whether downstream processing was `ok` (Lloyd McKenzie's
  example in the comments).
- A receiver that processed the bulk of a message successfully but
  encountered non-fatal issues with some content (warnings, partial
  failures, advisory messages) that the sender should know about.

Because the binding is `required` and the element is `1..1`, implementers
have no compliant way to express these states — the element is impossible
to populate honestly. The original ticket asked for an `unknown` / `other`
escape valve; InM preferred a more semantically specific concept.

### Proposed Change

Three coordinated changes in `HL7/fhir`:

**Change 1 — CodeSystem: add `non-fatal-error`.**

In `source/messageheader/codesystem-response-code.xml`, append a new
`<concept>` after the existing `fatal-error` concept:

```xml
<concept>
  <code value="non-fatal-error"/>
  <display value="Non-Fatal Error"/>
  <definition value="The message was accepted and processed but with one or more errors. The MessageHeader.response.detail SHALL describe the issue."/>
</concept>
```

The CodeSystem is `complete` and its bound ValueSet (`response-code`) uses
`<immutable value="true"/>` with a single `<include>` of the whole code
system, so adding the concept to the CodeSystem automatically expands the
ValueSet.

**Change 2 — StructureDefinition: change datatype to `CodeableConcept`,
binding to `extensible`, and update prose.**

In `source/messageheader/structuredefinition-MessageHeader.xml` on the
`MessageHeader.response.code` element (currently lines 477–505):

- `<short>` — change from `"ok | transient-error | fatal-error"` to
  `"ok | transient-error | fatal-error | non-fatal-error +"` (the trailing
  `+` is the FHIR convention for an extensible binding).
- `<type><code value="code"/></type>` — change to
  `<type><code value="CodeableConcept"/></type>`.
- `<binding><strength value="required"/></binding>` — change to
  `<strength value="extensible"/>`.
- Leave `<min value="1"/>`, `<max value="1"/>`, `<isSummary value="true"/>`,
  the `bindingName` extension (`ResponseType`), the `<valueSet>` reference
  to `http://hl7.org/fhir/ValueSet/response-code`, and the v2 / RIM mappings
  unchanged.
- Consider extending the `<comment>` to mention that when
  `non-fatal-error` is sent, `MessageHeader.response.details` MUST be
  populated (parallels the existing `fatal-error` comment on
  `response.details`, which already says *"If any of the issues are errors,
  the response code SHALL be an error."*).

**Change 3 — Search parameter prose update.**

In `source/messageheader/bundle-MessageHeader-search-params.xml`, update the
`MessageHeader-code` SearchParameter's `<description>` from
`"ok | transient-error | fatal-error"` to
`"ok | transient-error | fatal-error | non-fatal-error +"`. The
SearchParameter `type` stays `token` (token works for both `code` and
`CodeableConcept` element types) and the `expression`
`MessageHeader.response.code` is unchanged.

**Change 4 — Examples must use `CodeableConcept` form.**

The two example instances that populate `MessageHeader.response.code` need
to wrap the value in a `coding`:

- `source/messageheader/messageheader-example.xml` line 52 — replace
  `<code value="ok"/>` with the `CodeableConcept` form.
- `source/messageheader/message-response-link.xml` line 35 — replace
  `<code value="ok"/>` with the `CodeableConcept` form.

The new payload in each case:

```xml
<code>
  <coding>
    <system value="http://hl7.org/fhir/response-code"/>
    <code value="ok"/>
  </coding>
</code>
```

### Design Rationale

- **Why `non-fatal-error` instead of `unknown` / `other`.** From the
  resolution: *"InM agrees… the use of unknown or other is not desirable
  but will add a value for `non-fatal-error` to accommodate circumstances
  where a message was successful but some content may have errored."* A
  semantically specific code lets receivers and senders agree on the
  intent without needing out-of-band understanding of what `other` means.
- **Why `CodeableConcept` instead of `code`.** The resolution explicitly
  calls for the datatype change so future codes from other systems can be
  added without another spec-level breaking change. This is consistent with
  how FHIR generally widens code-typed elements when extensibility is
  needed (the same pattern was used on multiple resources during R4 → R5).
- **Why `extensible` instead of `required`.** Pairs with the
  `CodeableConcept` change. Required + CodeableConcept would defeat the
  purpose; `extensible` lets implementers add their own coded concepts when
  none of the canonical concepts apply, while still nudging implementers
  toward the canonical set for interoperability.
- **Alternatives considered.** Adding `unknown` and/or `other` to the
  existing required-bound `code` element. Rejected because it does not
  give implementers any future runway and the comments in the ticket
  argued (Anthony Julian asking Lloyd for `unknown` examples) that
  `unknown` was hard to justify on its own.

### Breaking Changes

This is a **breaking change** and should be called out as such in the
release notes:

- **Datatype change `code` → `CodeableConcept`** on
  `MessageHeader.response.code`. Any consumer or producer using a R4/R5
  `MessageHeader` instance wire format will not validate against the
  new schema (the inner JSON / XML shape changes from a primitive to a
  structured object). Existing instances need to be transformed.
- **Binding strength `required` → `extensible`.** This is a *widening*
  change so existing required-set-only senders remain conformant, but
  validators that previously rejected non-canonical codes will now accept
  them.
- **CodeSystem and ValueSet are marked `normative` (FMM 4)** — see the
  `<extension url=".../structuredefinition-standards-status">` block in
  both `codesystem-response-code.xml` and `valueset-response-code.xml`.
  Adding a code to a normative code system is allowed (substantive
  expansions are OK in subsequent normative versions) but the addition
  must be documented in the substantive change notes for this cycle.
- **No removed elements / constraints**, no search parameter type change
  (token still applies), no removed concepts.

### Related Specifications

- **FHIR Messaging framework** (`source/messaging.html` and related

- **`$process-message` operation** — the operation definition lives at

- **Bundle/Message profile downstream IGs** that constrain

### Testing Considerations

- **Revalidate** all `MessageHeader` examples in
  `source/messageheader/` (both files updated above plus any other
  bundles that contain a `MessageHeader` with a populated `response`).
- **Search-parameter tests:** issue a token search such as
  `MessageHeader?code=non-fatal-error` against a test server and confirm
  the new code is matched.
- **Negative test:** a `MessageHeader` instance using the **old**
  primitive form (`<code value="ok"/>` directly under `<response>`)
  should now fail schema validation — useful as a migration smoke test.
- **Edge case:** a `MessageHeader` populated with an
  out-of-ValueSet coded value (e.g., a custom local code) should
  validate (extensible binding) and produce a warning, not an error.
- **Interoperability:** confirm that any existing R4/R5 → R6 mapping
  notes / migration guidance documents the schema break on
  `MessageHeader.response.code`.

### Open Questions

- **Resolution table wording vs SD element id.** The resolution table
  describes the new code's definition as referring to
  `MessageHeader.response.detail` (singular), but the actual SD element
  id is `MessageHeader.response.details` (plural). PR #4007 may already
  have made a choice here. Recommend keeping the SD element id
  (`MessageHeader.response.details`) in the new concept's `definition`
  text as written in this plan; flag for InM if they intended the
  singular form for narrative reasons.
- **Comment on `MessageHeader.response.code` SHALL-populate-details.**
  The plan's task 2 suggests adding a SHALL-style comment that pairs
  `non-fatal-error` with mandatory `response.details`. Confirm with
  InM whether they want that strengthening (it is consistent with the
  existing `fatal-error` comment on `response.details`) or would
  prefer to leave it as a SHOULD.
- **`<comment>` on `MessageHeader.response.details`** currently says
  *"If any of the issues are errors, the response code SHALL be an
  error."* With four concepts in play, "an error" still reads
  unambiguously (covers `fatal-error` and `non-fatal-error`), but InM
  may want to clarify by listing both.
- **Standards-status / FMM impact.** The CodeSystem and ValueSet are
  `normative` at FMM 4 today. Adding a concept to a normative artifact
  in a subsequent normative version is allowed, but confirm that the
  current cycle's normative-balloting rules consider this a substantive
  change that requires re-balloting of the affected resources — the
  datatype change on `MessageHeader.response.code` almost certainly
  does.
- **Coordination with PR
  [HL7/fhir#4007](https://github.com/HL7/fhir/pull/4007).** The PR
  exists but its current contents were not reviewed as part of this
  plan; confirm its diff aligns with the six file changes above
  before opening any new work.
