## Feature Proposal

### Problem Statement

The ballot comment that produced FHIR-55243 (Comment 209, imported by Lloyd McKenzie) flagged `message-events` as a potential THO migration candidate because:

- `MessageDefinition.event[x]` is bound (example) to `ValueSet/message-events`, which in turn drew from the FHIR-only `CodeSystem/message-events`.
- Message event codes are useful beyond FHIR (CDA, V2) and are expected to evolve across releases, so the authoritative list should live in THO rather than the FHIR core spec.

FHIR-43807 addressed the same concern for `MessageHeader.event[x]` by (a) rewiring the shared `message-events` ValueSet to include the V2-0003 ValueSet from THO (`https://terminology.hl7.org/5.4.0/ValueSet-v2-0003.html`) and (b) rewriting the `MessageHeader.event[x]` ElementDefinition text to state that the binding is exemplar, the element is intentionally unbound, and implementers should treat V2-0003 as a good example source.

Because `MessageDefinition.event[x]` shares the same ValueSet, it inherits the terminology fix automatically, but its ElementDefinition has **not** yet been updated: the `short`/`definition` on `MessageDefinition.event[x]` still read `"Event code or link to the EventDefinition"` / `"Event code or link to the EventDefinition."` with no mention of intentional unbinding or the V2 example source. Readers of the MessageDefinition page therefore get weaker guidance than readers of the MessageHeader page, even though the two elements share the same binding.

### Proposed Change

Apply FHIR-43807's editorial treatment to the `MessageDefinition` side of the pair.

1. **`source/messagedefinition/structuredefinition-MessageDefinition.xml`** — on the `MessageDefinition.event[x]` element (id `MessageDefinition.event[x]`, currently at lines 530–554):
   - Keep `min=1`, `max=1`, `type=Coding | uri`, `isSummary=true` unchanged.
   - Keep the binding element unchanged: `strength=example`, `bindingName=MessageEvent`, `isCommonBinding=true`, `valueSet=http://hl7.org/fhir/ValueSet/message-events`.
   - Replace `<short value="Event code  or link to the EventDefinition"/>` with wording consistent with the MessageHeader short (adjusted for the definition/profile context, e.g. *"Event code or link to the EventDefinition for this message"*).
   - Replace the one-line `<definition>` with an expanded definition mirroring MessageHeader's (lines 107 in `structuredefinition-MessageHeader.xml`), adapted for MessageDefinition. Target text (final wording to be confirmed by I&M):

     > Code that identifies the event this MessageDefinition describes and connects it with its EventDefinition when applicable. Events are defined by codes, or by URIs that identify events, or by reference to an EventDefinition that defines the event directly. This element is intentionally unbound, and the selection of appropriate event codes is delegated to implementations and implementation guides. For codes, a good set of example codes is found in [the V2 event codes](https://terminology.hl7.org/ValueSet-v2-0003.html).

   - Leave the `<binding>` `description` (`"One of the message events defined as part of this version of FHIR."`) in place unless I&M decides to align it with the MessageHeader wording; this is a low-risk clarifying change but is not called out by the resolution.

2. **`source/messageheader/valueset-message-events.xml`** — **no change required**. Already composes `https://terminology.hl7.org/5.4.0/ValueSet-v2-0003.html` (lines 87–91). Verify during QA that FHIR-43807 is actually in the applied state at implementation time; if not, this ticket must wait (resolution text says "once it has been applied").

3. **`source/messagedefinition/messagedefinition-introduction.xml`** / **`messagedefinition-notes.xml`** — optional prose update. A short paragraph could be added pointing at V2-0003 as the canonical example source, mirroring the messaging module text. No narrative mentions of `event`/`message-events`/`v2-0003` exist in these files today (grep confirmed), so adding one is an I&M editorial call, not a resolution requirement.

### Design Rationale

- **Consistency with FHIR-43807.** MessageDefinition.event[x] and MessageHeader.event[x] are paired elements sharing a common binding (`elementdefinition-isCommonBinding=true`, `bindingName=MessageEvent`). Divergent definition text on a common binding is a maintenance hazard and a real source of ballot confusion.
- **No new binding required.** Because FHIR-43807 already rewired the shared `message-events` ValueSet to include V2-0003, MessageDefinition automatically consumes the broadened value set. Rebinding `MessageDefinition.event[x]` directly to `ValueSet-v2-0003` would (a) break the `isCommonBinding` pairing, (b) duplicate work already absorbed at the ValueSet level, and (c) require a THO coordination round the resolution does not authorise.
- **Binding strength stays `example`.** The resolution text and the FHIR-43807 precedent both keep `strength=example` / "intentionally unbound". Tightening to `extensible`/`preferred` would be a breaking change and is not what I&M approved.
- **No deprecation of FHIR `CodeSystem/message-events`.** The resolution does not call for retiring the FHIR-owned CodeSystem; the ValueSet continues to include V2-0003 alongside (or in place of) the legacy include, matching what FHIR-43807 shipped.

### Breaking Changes

No breaking changes. The binding canonical (`http://hl7.org/fhir/ValueSet/message-events`), strength (`example`), cardinality (`1..1`), and type choice (`Coding | uri`) are all unchanged. Implementations that bind to the shared ValueSet automatically pick up V2-0003 codes, but that shift was already delivered by FHIR-43807 at the ValueSet level and is back-compatible (example-strength bindings are non-normative).

### Related Specifications

- **MessageHeader.event[x]** (`source/messageheader/structuredefinition-MessageHeader.xml`) — shares the `MessageEvent` common binding (`elementdefinition-isCommonBinding=true`). The MessageDefinition wording change introduced here should stay editorially aligned with it.
- **`ValueSet/message-events`** (`http://hl7.org/fhir/ValueSet/message-events`) — shared definition ValueSet. Already rewired to V2-0003 by FHIR-43807.
- **`CodeSystem/message-events`** (`http://hl7.org/fhir/message-events`) — legacy FHIR-owned code system. Not deleted; not modified by this ticket.
- **EventDefinition** and **SubscriptionTopic** — referenced in the MessageHeader definition text via URI; the proposed MessageDefinition wording should acknowledge EventDefinition (it is the one MessageDefinition.event[x]'s `uri` variant explicitly targets) and may optionally mention SubscriptionTopic for symmetry.
- **V2 table 0003** (`https://terminology.hl7.org/5.4.0/ValueSet-v2-0003.html`) — external THO artifact, referenced by URL only.

### Related Tickets

- **FHIR-43807 — Resource Message Header** (Persuasive with Modification, resolved 2024-02-02). Companion ticket. Its resolution — bind to v2-0003 and rewrite the element documentation as exemplar / intentionally unbound — is the template FHIR-55243 must mirror on the MessageDefinition side.
