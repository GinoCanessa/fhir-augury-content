## Resolution Summary

The InM work group accepted the ticket as **Persuasive with Modification**. The
approved resolution has two concrete actions:

1. **Bind `MessageHeader.event[x]`** to the V2 event type value set
   `https://terminology.hl7.org/5.4.0/ValueSet-v2-0003.html`
   (canonical `http://terminology.hl7.org/ValueSet/v2-0003`), retaining an
   **example** binding strength.
2. **Update the documentation** for the binding so that it reads, verbatim:
   > The value set is exemplar: element is intentionally unbound. It's up to
   > implementation guides and local implementations to define the codes they
   > need for their specific use-cases.

Per `ajulian` (2025-09-18) and `JIRAUSER24630` (2026-01-28), the ticket was
previously marked applied but the binding swap was never performed in
`structuredefinition-MessageHeader.xml`; the InM Zulip thread
`#fhir-messages > MessageHeader.event` (Zulip id `540258156`) confirms that
"InM will apply it." The element's `<definition>` text was already updated in a
prior pass — it now mentions the V2 event codes and notes that the element is
"intentionally unbound" — but the `<binding>` block still points at the old
exemplar `http://hl7.org/fhir/ValueSet/message-events`. This plan finishes the
unfinished work: it changes the bound `valueSet` and rewrites the binding
`<description>` to match the resolution.

## Feature Proposal

### Problem Statement

The MessageHeader.event[x] element is the trigger code for a FHIR message. The
ticket reports that the published binding cell on the MessageHeader resource
page renders as `??` because the element's binding is in an awkward state: the
element carries an `example` binding to `ValueSet/message-events` (a ValueSet
that simply re-exposes the legacy FHIR `message-events` code system), and the
text accompanying the binding never explained what implementers were supposed
to do, leaving a "controlled terminology TBD" impression.

The InM resolution is that the FHIR core specification does not want to
maintain a curated list of message event codes — those are a local /
implementation-guide concern. However, leaving the cell with a broken binding
is not acceptable. The agreed remedy is to use HL7 V2's well-known
`v2-0003` (Event Type) value set as the **exemplar** value set so the cell
renders something meaningful, while the prose makes clear the binding is
example-only and IGs/implementers are expected to substitute their own codes.

### Proposed Change

In **`HL7/fhir`** repository, in
`source/messageheader/structuredefinition-MessageHeader.xml`, modify the
`<binding>` block on the `MessageHeader.event[x]` element as follows:

- **`<binding>.<valueSet>`** — change from
  `http://hl7.org/fhir/ValueSet/message-events`
  to
  `http://terminology.hl7.org/ValueSet/v2-0003`.
- **`<binding>.<strength>`** — leave at `example` (per the resolution: the
  binding is exemplar; strength `example` is the FHIR mechanism that says so).
- **`<binding>.<description>`** — replace the current text
  ("One of the message events defined as part of this version of FHIR.") with:
  > The value set is exemplar: element is intentionally unbound. It's up to
  > implementation guides and local implementations to define the codes they
  > need for their specific use-cases.
- **`<binding>` extensions** — keep the `elementdefinition-bindingName` (value
  `MessageEvent`) and `elementdefinition-isCommonBinding` extensions in place;
  no change is required there. The binding name is preserved so existing
  cross-spec references (e.g., bindings tables) remain stable.

The element-level `<definition>` already reflects the agreed wording (it
explicitly notes the element is "intentionally unbound" and points at the V2
event codes as a "good set of example codes"); no change is required there
beyond a final consistency pass to ensure the V2 link in the prose matches the
binding canonical.

The local `valueset-message-events.xml` and its companion
`codesystem-message-events.xml` (referenced as
`http://hl7.org/fhir/message-events`) are no longer the bound ValueSet for
`MessageHeader.event[x]`. They are still legitimate FHIR specification
artefacts and are referenced from other narrative pages, so this plan does not
delete them — see Open Questions on whether they should be marked deprecated
in a follow-up ticket.

### Design Rationale

- **Why `v2-0003`?** The InM Zulip post by `ajulian` (2024-01-31) explicitly
  surfaced V2-0003 as the de-facto example code set ("the missing value set in
  question looks a lot like…"); the resolution confirms it. Using a
  terminology.hl7.org ValueSet rather than a FHIR-core ValueSet also signals
  that the FHIR spec is not trying to own / curate message event codes.
- **Why keep strength `example`?** The resolution language ("is exemplar:
  element is intentionally unbound") maps directly to FHIR's `example` binding
  strength definition. Using `preferred` or stronger would contradict
  "intentionally unbound" and would impose conformance expectations on
  implementers that the work group does not want.
- **Why not mint a new FHIR ValueSet?** That option was considered in the
  ticket comments and rejected — `ajulian` (2024-02-02): "There's no
  appropriate terminology we wish to bind to, and we don't want to define one
  at this time." Reusing the existing V2 set avoids new spec maintenance.
- **Why also rewrite the binding description?** The current description
  ("One of the message events defined as part of this version of FHIR") is now
  factually wrong — the bound ValueSet is no longer authored by the FHIR core
  spec. The resolution provides the replacement wording verbatim.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole repository with content changes — the binding lives in the
  core FHIR MessageHeader StructureDefinition.
- **Affected Files:**
  - `source/messageheader/structuredefinition-MessageHeader.xml` —
    swap the `<valueSet>` URL on the `MessageHeader.event[x]` `<binding>`

    binding-name / common-binding extensions as-is.
  - `source/messageheader/messageheader-notes.xml` — revise the bullet
    "The value set MessageEvent is populated by the authors of the resources

- **Change Scope:** Minor — three localised edits, no schema or invariant
  changes.

### Breaking Changes

- The bound ValueSet canonical changes from
  `http://hl7.org/fhir/ValueSet/message-events` to
  `http://terminology.hl7.org/ValueSet/v2-0003`. Because the binding strength
  is `example` (and remains `example`), this is **not** a conformance breaking
  change — example bindings impose no conformance expectations. However, any
  implementer or downstream IG that treated the canonical of the example
  ValueSet as a stable identifier (e.g., as a starting point for a derived
  required binding) will need to repoint. Document this in the resource's
  change log.
- No element cardinality, type, search-parameter, or invariant changes are
  introduced.

### Related Specifications

- **HL7 Terminology (THO) `v2-0003`** — becomes a referenced (exemplar) value
  set from FHIR core. No change to THO is required; the binding takes a
  dependency on an already-published THO artefact.
- `valueset-message-events.xml` and `codesystem-message-events.xml` (canonical
  `http://hl7.org/fhir/ValueSet/message-events` and
  `http://hl7.org/fhir/message-events`) remain in the FHIR specification but
  lose their last in-spec binder. They are still listed in
  `bundle-MessageHeader-search-params.xml` neighbours and in the artefact
  registry; nothing requires their removal as part of this ticket.
- `source/messaging.html` and `source/messagedefinition.html` should be
  re-read for consistency; their current statements ("This specification does
  not define any events") remain accurate after the change.

  in `source/messageheader/` remain in the publication build and continue to

### Open Questions

- **Should `valueset-message-events.xml` and `codesystem-message-events.xml`
  be deprecated?** This ticket does not require their removal; FHIR-30201
  already noted they are underused. Recommend a follow-up ticket if InM wants
  to mark them deprecated. The briefing's gotcha — "out-of-scope concerns
  flagged in comments need a separate ticket; do not silently roll them in" —
  applies here.
- **Versioned vs unversioned canonical.** The resolution cites the THO 5.4.0
  versioned page (`https://terminology.hl7.org/5.4.0/ValueSet-v2-0003.html`).
  The plan binds to the unversioned canonical
  (`http://terminology.hl7.org/ValueSet/v2-0003`) so the FHIR build picks the
  THO version pinned in its own dependency configuration. Confirm with InM
  that an unversioned binding is acceptable; if a pinned version is required,
  add `|<version>` to the `<valueSet>` URL (e.g.,
  `http://terminology.hl7.org/ValueSet/v2-0003|5.4.0`).
- **Snapshot regeneration.** Confirm the publisher pipeline regenerates the
  `<snapshot>` block; if a manual snapshot rebuild step is needed for this
  resource, capture it in the PR description so reviewers know to run it.
