## Resolution Summary

Resolution: **Persuasive**.

Resolution description (verbatim from FHIR-53423):

> Change DeviceMetric.calibration.state to CodeableConcept with binding of extensible.
>
> Change unspecified value to unknown with appropriate update to definition/display.

Two concrete edits are approved:

1. Retype `DeviceMetric.calibration.state` from `code` to `CodeableConcept` and relax the binding strength on the existing `metric-calibration-state` ValueSet from `required` to `extensible`.
2. In the `metric-calibration-state` CodeSystem, replace the `unspecified` concept with `unknown` and update its `display` and `definition` accordingly.

The change was agreed at the Devices WG call on 2025-11-10 and applied as Persuasive on 2026-01-28.

## Feature Proposal

### Problem Statement

`DeviceMetric.calibration.state` is currently modelled as a primitive `code` bound `required` to the `DeviceMetricCalibrationState` CodeSystem (`source/devicemetric/codesystem-metric-calibration-state.xml`, concepts: `not-calibrated`, `calibration-required`, `calibrated`, `unspecified`). Two problems:

- The `required` binding plus primitive `code` type forces every implementer worldwide to describe calibration state with exactly one of four fixed codes. Real device-calibration lifecycles are richer (e.g., vendor-specific intermediate states, "in progress", "overdue", "expired", manufacturer codes); there is no extension point without a model change.
- The concept `unspecified` is ambiguous: it is used to mean "we do not know the state", but `unspecified` reads as "the sender declined to state". FHIR's canonical wording for "the state is not known" is `unknown`, and `DeviceMetric.calibration.state` already has `meaningWhenMissing = "If absent, the calibration state is unknown."` (`structuredefinition-DeviceMetric.xml:409`), making the inconsistency visible.

### Proposed Change

Scope: single repository (`HL7/fhir`), single resource folder (`source/devicemetric/`). Three files change.

**1. `source/devicemetric/structuredefinition-DeviceMetric.xml`** — element `DeviceMetric.calibration.state` (lines 399–422 in the clone at `db79dcd`):

- `<type><code value="code"/></type>` → `<type><code value="CodeableConcept"/></type>`
- `<binding><strength value="required"/> … </binding>` → `<strength value="extensible"/>` (keep `bindingName = DeviceMetricCalibrationState`, description, and `valueSet = http://hl7.org/fhir/ValueSet/metric-calibration-state`).
- `<short value="not-calibrated | calibration-required | calibrated | unspecified"/>` → `<short value="not-calibrated | calibration-required | calibrated | unknown"/>` to reflect the renamed code (the `short` is the standard "pipe list" shown in the column view for bound primitive codes; retaining it with the new term keeps the row legible even though the type is now `CodeableConcept`).
- Leave min/max (`0..1`), `isSummary=false`, `meaningWhenMissing`, DIM mapping, and the `path`/`id` unchanged.

**2. `source/devicemetric/codesystem-metric-calibration-state.xml`** — rename the fourth concept (lines 65–69):

- `<code value="unspecified"/>` → `<code value="unknown"/>`
- `<display value="Unspecified"/>` → `<display value="Unknown"/>`
- `<definition value="The state of calibration of this metric is unspecified."/>` → `<definition value="The state of calibration of this metric is unknown."/>`
- Keep the CodeSystem `url`, OID identifiers, `version`, `name`, `status`, `content=complete`, and the `valueSet` back-link intact. Leave the other three concepts (`not-calibrated`, `calibration-required`, `calibrated`) untouched.

**3. `source/devicemetric/valueset-metric-calibration-state.xml`** — review only:

- The ValueSet includes the whole `http://hl7.org/fhir/metric-calibration-state` CodeSystem, so the rename flows through automatically with no edits to `<compose>`.
- `<immutable value="true"/>` on the ValueSet means "the set of codes this ValueSet designates cannot change meaning" — it does not prevent the underlying CodeSystem from being edited, but it does create a publication-time contract that the old `unspecified` code-semantic disappears when the CS is republished. This is consistent with the "Persuasive" resolution and does not require additional edits here, but it should be flagged for the Devices WG reviewer.
- The Devices WG may wish to remove `<immutable value="true"/>` given that the binding is now `extensible` and downstream tooling commonly treats immutable ValueSets as closed. This is an **open question** (see below), not a required edit.

**4. `source/devicemetric/devicemetric-example.xml`** — uses `<state value="calibrated"/>`. The primitive-shorthand `value=` syntax is valid for `CodeableConcept.coding.code` only when expressed as `<state><coding><system value="…"/><code value="calibrated"/></coding></state>`. The example must be updated to use the CodeableConcept shape, otherwise publisher validation will fail after the type change.

### Design Rationale

- **Extensible binding is the standard FHIR pattern** for device/clinical status codes where the HL7-defined codes cover the common cases but cannot reasonably enumerate every vendor-specific state. The sibling element `DeviceMetric.calibration.type` is already modelled exactly this way (CodeableConcept + extensible binding to `metric-calibration-type`, `structuredefinition-DeviceMetric.xml:376–398`), so this change brings `.state` into alignment with `.type` and with the broader DeviceMetric pattern.
- **`unknown` is the canonical FHIR spelling** for "the value is not known" (e.g., `Patient.gender` uses `unknown`, `Observation.dataAbsentReason` has `unknown`). Renaming `unspecified` → `unknown` removes the "sender declined" connotation and matches the element's existing `meaningWhenMissing`.
- **Rename rather than deprecate**: the resolution says "Change unspecified value to unknown", and the code has no documented deployed usage in the spec (no example references it), so a clean rename is appropriate. A deprecation-style retirement (keep `unspecified`, add `unknown`) was not selected by the WG and would leave two codes meaning the same thing in a CodeSystem whose binding is becoming extensible anyway.

## Impact Analysis

### Affected Repositories

- **Role:** Sole authoring repository. Holds the DeviceMetric StructureDefinition, the `metric-calibration-state` CodeSystem and ValueSet, and the DeviceMetric example that exercises `calibration.state`.
- **Affected Files:**
  - `source/devicemetric/structuredefinition-DeviceMetric.xml` — retype `DeviceMetric.calibration.state` from `code` to `CodeableConcept`; change binding strength from `required` to `extensible`; update `short` to use `unknown` instead of `unspecified`.
  - `source/devicemetric/codesystem-metric-calibration-state.xml` — rename concept `unspecified` to `unknown`; update its `display` and `definition`.
  - `source/devicemetric/valueset-metric-calibration-state.xml` — no content change; include as an open question whether `<immutable value="true"/>` should be dropped now that the binding is extensible.
  - `source/devicemetric/devicemetric-example.xml` — convert `<state value="calibrated"/>` to the CodeableConcept shape so the example validates after the type change.
- **Change Scope:** Moderate. Two concepts change meaning on the wire (type shape of `calibration.state`; rename of one code).

### Breaking Changes

Yes — this is a breaking change on two axes. Per HL7's normal R6 publication rules this is acceptable only because DeviceMetric is not a normative resource element here; nonetheless, implementers must be warned.

- **Type change `code` → `CodeableConcept`.** Instances currently serialized as `<state value="calibrated"/>` (XML) or `"state": "calibrated"` (JSON) will no longer validate against the new StructureDefinition. They must become `<state><coding><system value="http://hl7.org/fhir/metric-calibration-state"/><code value="calibrated"/></coding></state>`. Every existing DeviceMetric instance and every server/client that produces or consumes `.state` is affected.
- **Code rename `unspecified` → `unknown`.** Instances coded as `unspecified` from the `http://hl7.org/fhir/metric-calibration-state` system are no longer valid against the republished CodeSystem (the old code ceases to exist; the CS is `content=complete`).
- **Binding strength `required` → `extensible`** is strictly a *loosening*, not a breaking change on its own — every previously-valid value remains valid. Combined with the type change it does newly permit additional (extension) codes.

### Related Specifications

- `DeviceMetric.calibration.type` (sibling element, already CodeableConcept + extensible binding to `metric-calibration-type`) is the template this change aligns to; it is not itself modified.
- No other core resource references `DeviceMetric.calibration.state` directly. No cross-resource invariant or search parameter touches `.state` (confirmed by grep of `source/devicemetric/bundle-DeviceMetric-search-params.xml`, which defines searches over `type`, `category`, `source`, `parent` — not `calibration.state`).
- IGs that profile DeviceMetric (not inventoried here — none surfaced via Step 1b cross-references) may need to re-profile `.state` after the type change.

## Implementation Plan

### Prerequisites

None. All referenced artefacts exist in the clone at `db79dcd` and the change is self-contained within `source/devicemetric/`.

### Step-by-Step Tasks

#### HL7/fhir

1. **Retype `DeviceMetric.calibration.state` to `CodeableConcept`.**
   - File: `source/devicemetric/structuredefinition-DeviceMetric.xml`
   - Action: In the `<element id="DeviceMetric.calibration.state">` block (lines 399–422 at `db79dcd`), change `<type><code value="code"/></type>` to `<type><code value="CodeableConcept"/></type>`.
   - Details: Edit only the `<differential>`. Do not modify the `<snapshot>` — it is regenerated by the publisher (briefing §Facts).

2. **Relax the binding strength to `extensible`.**
   - File: `source/devicemetric/structuredefinition-DeviceMetric.xml`
   - Action: In the same element's `<binding>` block, change `<strength value="required"/>` to `<strength value="extensible"/>`.
   - Details: Keep the `bindingName` extension (`DeviceMetricCalibrationState`), the `<description>`, and the `<valueSet>` (`http://hl7.org/fhir/ValueSet/metric-calibration-state`) unchanged.

3. **Update the element `short` to reflect the renamed code.**
   - File: `source/devicemetric/structuredefinition-DeviceMetric.xml`
   - Action: Change `<short value="not-calibrated | calibration-required | calibrated | unspecified"/>` to `<short value="not-calibrated | calibration-required | calibrated | unknown"/>`.
   - Details: Leave `<definition>`, `<min>`, `<max>`, `<isSummary>`, `<meaningWhenMissing>`, and the DIM mapping untouched.

4. **Rename the `unspecified` concept to `unknown` in the CodeSystem.**
   - File: `source/devicemetric/codesystem-metric-calibration-state.xml`
   - Action: In the fourth `<concept>` block (lines 65–69), change `<code value="unspecified"/>` to `<code value="unknown"/>`, `<display value="Unspecified"/>` to `<display value="Unknown"/>`, and `<definition value="The state of calibration of this metric is unspecified."/>` to `<definition value="The state of calibration of this metric is unknown."/>`.
   - Details: Do not add or remove concepts. Do not change the CodeSystem `url`, OID identifiers, `version`, `caseSensitive`, or the `valueSet` back-link. The other three concepts (`not-calibrated`, `calibration-required`, `calibrated`) are untouched.

5. **Update the DeviceMetric example to use CodeableConcept shape for `state`.**
   - File: `source/devicemetric/devicemetric-example.xml`
   - Action: Replace `<state value="calibrated"/>` (line 43) with:
     ```xml
     <state>
       <coding>
         <system value="http://hl7.org/fhir/metric-calibration-state"/>
         <code value="calibrated"/>
       </coding>
     </state>
     ```
   - Details: Required so the example continues to validate against the new `CodeableConcept` type; otherwise the publisher will raise a type-mismatch error.

6. **(Optional, WG decision) Drop `immutable` from the calibration-state ValueSet.**
   - File: `source/devicemetric/valueset-metric-calibration-state.xml`
   - Action: Pending Devices WG confirmation, remove `<immutable value="true"/>`.
   - Details: Not strictly required by the resolution. Flagged because an extensible binding against an `immutable` ValueSet is a mixed signal for terminology tooling. Keep as-is unless the WG approves the change in review.
