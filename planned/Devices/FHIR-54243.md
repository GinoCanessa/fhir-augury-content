## Resolution Summary

Resolution: **Persuasive**.

Per the resolution description (and confirmed by Mike Cros's 2026-02-09
applied-vote comment):

> *"this is a duplicate for state as related to FHIR-53423; this is a
> duplicate for color as related to FHIR-54464; the other code elements are
> staying."*

In other words, the substantive changes proposed by this umbrella ticket are
absorbed into two narrower siblings:

- **`DeviceMetric.calibration.state`** — handled by **FHIR-53423** ("make
  `DeviceMetric.calibration.state` extensible and change `unspecified` to
  `unknown`"). That ticket changes the binding strength (and the underlying
  CodeSystem/ValueSet) but does **not** retype the element to
  Coding/CodeableConcept.
- **`DeviceMetric.color`** — handled by **FHIR-54464**
  ("`DeviceMetric.color` should be CodeableConcept"). That ticket retypes
  the element from `code` to `CodeableConcept`.
- **All other `code`-typed elements on `DeviceMetric`** —
  `DeviceMetric.status`, `DeviceMetric.operationalStatus`,
  `DeviceMetric.category`, `DeviceMetric.availability`, and
  `DeviceMetric.calibration.type` — explicitly **stay as `code`**. The
  resolution rejects the umbrella proposal to widen them to Coding /
  CodeableConcept.

Therefore, **no direct change to `source/devicemetric/` is required by
FHIR-54243 itself.** The ticket's role in the implementation backlog is to
(a) record the work group's decision that the broader umbrella change is
not adopted, and (b) point downstream tooling/reviewers at the two sibling
tickets that carry the actual edits.

## Feature Proposal

### Problem Statement

The original ticket (filed by Erik Moll, 2026-01-14) generalises the
concern raised in **FHIR-51486**: `code`-typed elements cannot carry a
code-system or version, cannot offer alternate codings, and therefore limit
interoperability. The reporter asks the Devices WG to consider widening
`code` elements on `DeviceMetric` to `Coding` or `CodeableConcept`,
mirroring the broader R6-era discussion.

The current state of `source/devicemetric/structuredefinition-DeviceMetric.xml`
(clone HEAD `db79dcd`) shows the following `code`-typed elements:

| Path | Line | Binding |
| ---- | ---- | ------- |
| `DeviceMetric.status` | 118 | required → `metric-status` |
| `DeviceMetric.operationalStatus` | 144 | required → `metric-operational-status` |
| `DeviceMetric.category` | 173 | required → `metric-category` |
| `DeviceMetric.color` | 299 | required → `color-codes` |
| `DeviceMetric.availability` | 339 | required → `metric-availability` |
| `DeviceMetric.calibration.type` | 377 | required → `metric-calibration-type` |
| `DeviceMetric.calibration.state` | 400 | required → `metric-calibration-state` |

(`DeviceMetric.type` is already `CodeableConcept`, line 208, so it is not
in scope.)

### Proposed Change

The Devices WG's adopted resolution is **narrow**, not the umbrella
widening that was proposed:

1. **Do nothing in this ticket's scope** beyond documenting that
   FHIR-54243 is closed as Persuasive and the work is split into
   FHIR-53423 and FHIR-54464.
2. **Track FHIR-53423** for `DeviceMetric.calibration.state`. That ticket
   keeps the type as `code` but loosens the binding to extensible and
   replaces the `unspecified` concept with `unknown` in the bound
   ValueSet/CodeSystem.
3. **Track FHIR-54464** for `DeviceMetric.color`. That ticket retypes the
   element from `code` to `CodeableConcept` (and carries any associated
   binding/strength tweaks).
4. **Leave `status`, `operationalStatus`, `category`, `availability`, and
   `calibration.type` unchanged** — the resolution explicitly states "the
   other code elements are staying."

No new elements, extensions, invariants, search parameters, examples, or
narrative text are introduced by FHIR-54243 itself.

### Design Rationale

The Devices WG declined the umbrella widening for several implicit
reasons that follow from the resolution:

- **Stability of required bindings.** `status`, `operationalStatus`,
  `category`, `availability`, and `calibration.type` already use required
  bindings to small, well-curated CodeSystems owned by the Devices WG.
  Widening them to Coding/CodeableConcept would loosen interoperability
  guarantees with no demonstrated implementer need (the original ticket
  cites only general R6 considerations, not concrete use-cases on these
  elements).
- **Targeted fixes preferred over umbrella retypes.** Where there *is* a
  concrete implementer need (`color` benefits from alternate codings;
  `calibration.state` needs an extensibility/`unknown` story), the WG
  filed dedicated tickets (FHIR-54464, FHIR-53423) so each change can be
  scoped, balloted, and tracked individually.
- **R6 finality.** With R6 being the final FHIR release, the bar for
  type-widening of `code` elements is high; only changes that are
  affirmatively required by implementers were accepted.
