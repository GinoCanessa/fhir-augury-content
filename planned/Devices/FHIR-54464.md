## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"change color to codable concept with required binding"*.

The disposition keeps the existing required binding to
`http://hl7.org/fhir/ValueSet/color-codes` but widens the element type
from `code` to `CodeableConcept` so that callers can carry alternate
codings (e.g., a vendor or display-system equivalent) alongside the
canonical color, while still being conformant against the required
value set via at least one coding in `DeviceMetric.color.coding`.

## Feature Proposal

### Problem Statement

`DeviceMetric.color` is currently typed as a primitive `code` with a
**required** binding to the `ColorCodesOrRGB` value set
(`http://hl7.org/fhir/ValueSet/color-codes`). That value set composes
two code systems — `http://hl7.org/fhir/color-names` (CSS4 color names)
and `http://hl7.org/fhir/color-rgb` (`#RRGGBB` hex codes) — but a
primitive `code` cannot carry the `system` of the value it expresses.
The reporter notes:

- Implementations cannot record alternate codings of the same color
  (vendor color palettes, display-system specific identifiers, etc.).
- Although in practice the two source code systems are visually
  distinguishable (hex vs. CSS color name), the architectural risk of a
  `code` element bound to a multi-system value set remains: there is no
  in-instance way to declare which code system a value came from.
- Future expansion (additional codings, translations, display text) is
  blocked by the primitive type.

### Proposed Change

**Single substantive change in `HL7/fhir`:** widen the type of
`DeviceMetric.color` from `code` to `CodeableConcept`, preserving the
existing required binding.

In `source/devicemetric/structuredefinition-DeviceMetric.xml`
(differential element `DeviceMetric.color`, lines 298–318 in the
current clone HEAD `db79dcd`):

- `<type><code value="code"/></type>` → `<type><code value="CodeableConcept"/></type>`
- Leave `min`=0, `max`=1 unchanged.
- Leave `isSummary value="false"` unchanged.
- Leave `meaningWhenMissing` unchanged
  (`"If absent, there is no preferred color for rendering."`).
- Keep the binding block as-is:
  - `strength` = `required`
  - `valueSet` = `http://hl7.org/fhir/ValueSet/color-codes`
  - `description` = `"Describes the typical color of representation."`
- Update `<short>` so it no longer implies a single primitive value —
  e.g., from `"Color name (from CSS4) or #RRGGBB code"` to something
  like `"Color of the metric (CSS4 name or #RRGGBB code)"` (still
  pointing at the same value set, but no longer suggesting only a
  primitive code is allowed).
- Definition text (`<definition>`) does not require functional change;
  the existing prose about Patient Monitor / ECG / Pleth color
  conventions still reads correctly for a `CodeableConcept`.
- The DIM mapping (`Metric::Color`) is unaffected.

The required binding semantics carry over from `code`:
[`Terminologies — Required`](https://hl7.org/fhir/terminologies.html#required)
mandates that for `CodeableConcept` at least one `coding` entry
references a code drawn from the value set; additional `coding` entries
in other systems and `text` are then permitted.

**Example update.** `source/devicemetric/devicemetric-example.xml`
currently encodes the value as a primitive
(`<color value="blue"/>`, line 27). Replace with the equivalent
`CodeableConcept` form, e.g.:

```xml
<color>
  <coding>
    <system value="http://hl7.org/fhir/color-names"/>
    <code value="blue"/>
  </coding>
</color>
```

### Design Rationale

- **Aligns with the rest of the resource.** Sibling elements such as
  `category`, `operationalStatus`, and `calibration.type/state` are
  already `CodeableConcept`; only `color` and a handful of status-style
  codes were primitives. Widening to `CodeableConcept` is the
  consistent pattern for an element that may legitimately need
  alternate codings or a display string.
- **Resolves the multi-system value-set risk** identified by the
  reporter: each `coding` carries its own `system`, so consumers can
  unambiguously distinguish a CSS color name from a hex value (or any
  future supplementary system) without relying on lexical heuristics.
- **Required binding is retained**, satisfying the workgroup's
  interoperability goal that there be a single canonical
  representation: one `coding` from `ValueSet/color-codes` is required;
  any further codings are additive.
- **No need for `Coding`** (instead of `CodeableConcept`): the workgroup
  comment in the original ticket explicitly cites support for
  *alternate codings* and *future expansion*, which `CodeableConcept`
  enables and `Coding` does not.

- **Change Scope:** Moderate. One element type change in one

  invariant, no value-set composition change.

### Breaking Changes

**Yes — this is a breaking change.** Changing
`DeviceMetric.color` from `code` to `CodeableConcept`:

- Alters the wire/JSON/XML shape of the element. Existing instances
  authored against the previous version (e.g.,
  `"color": "blue"` in JSON, `<color value="blue"/>` in XML) are
  invalid against the new StructureDefinition.
- Affects every consumer that reads or writes
  `DeviceMetric.color` — parsers, generated language bindings,
  validators, and FHIRPath expressions over `DeviceMetric.color` (now
  `DeviceMetric.color.coding.code` or `DeviceMetric.color.text`).
- Requires consumers to migrate; there is no in-instance backwards
  compatible representation for a primitive `code` and a
  `CodeableConcept` simultaneously.

The required binding strength itself is unchanged, so there is no
*additional* binding-strength escalation, but the type widening is
itself a breaking change that must be documented in the version
release notes.
