## Resolution Summary

The Devices WG resolution (Persuasive with Modification) accepts moving the
DeviceAlert and DeviceMetric terminology artifacts listed in the ticket
description out of FHIR Core into THO. In addition, the resolution
expands the scope to include the `metric-calibration-state` CodeSystem
and ValueSet (per FHIR-53423). The bindings on the affected
`DeviceAlert` and `DeviceMetric` data elements must be updated to point
at the new (THO-hosted) ValueSets.

The actual *creation* of the corresponding artifacts inside THO is
called out as a separate UP ticket and is therefore **out of scope for
this plan**; this plan covers only the FHIR Core (HL7/fhir) side of the
move: deletion of the in-core CodeSystem/ValueSet files, removal of
their registry entries, and re-pointing the bindings.

Verbatim resolution: *"Accept the movement of the terminology artifacts
listed in the description. Also move CS and VS for
metric-calibration-state per FHIR-53423. We will also make the
necessary changes to the binding targets of affected data elements in
DeviceMetric and DeviceAlert."*

## Feature Proposal

### Problem Statement

Per TSMG guidance summarised by Lloyd McKenzie on the ticket
(*#TSMG > Terminology moved from the base spec to THO*), once a code
system reaches FMM3+ and is not used in a `code`-typed (schema-bound)
binding and is not flagged experimental, it must live external to FHIR
Core (typically THO) rather than in the base spec. The
DeviceAlert and DeviceMetric terminology that the Devices WG plans to
mature past R6 ballot meets that condition, so the relevant CodeSystems
and the ValueSets that wrap them currently live in the wrong
repository. Leaving them in core would block the Devices WG from
versioning or expanding them on a faster cycle than the FHIR core
release.

### Design Rationale

This is a relocation, not a redesign. Lloyd's TSMG guidance cited on
the ticket establishes the rule: FMM3+ CodeSystems that are not
schema-bound (`code`-typed) and not experimental must live in THO. The
Devices WG (10/27/25 minutes) and Elliot Silver's per-element analysis
(10/30/25) jointly classified these specific artifacts as candidates;
the resolution accepts that classification with the addition of
`metric-calibration-state` from FHIR-53423.

The ticket explicitly defers the THO additions to a separate UP
ticket. That keeps the FHIR Core change atomic — it can land as soon
as THO can host the new canonical URLs — and avoids tangling the
publish runs of the two repositories.

### Breaking Changes

- **Canonical URL changes for 13 CodeSystems and 12 ValueSets** are
  inherently breaking for any consumer that resolves the old
  `http://hl7.org/fhir/...` canonical URLs. Any IG that imports
  DeviceAlert or DeviceMetric and refers to these terminologies by
  canonical URL will need to update its references once the THO
  artifacts are published.
- Binding *strengths* on `DeviceAlert` and `DeviceMetric` elements are
  not being changed by this plan, but the ValueSet they point at is —
  consumers re-validating with the new terminologies will get the same
  semantic check against (presumably) the same code lists, so behaviour
  should be preserved as long as THO publishes the same content.
- Removal of OID registry rows from `oids.ini` reassigns the
  authoritative location of those OIDs; downstream OID consumers should
  not be affected (the OID values themselves do not change), but this
  is worth calling out in the change log.
