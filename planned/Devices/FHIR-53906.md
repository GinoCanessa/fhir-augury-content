## Resolution Summary

**Resolution:** Persuasive — *"Make the requested changes as proposed."*

The reporter observes that the **DeviceAlert** specification page is rendered
with a **Draft** standards-status banner at the top of the page, and that the
StructureDefinition view shows a `D` flag for the entire resource. Both
should instead reflect **Normative** status (`N` flag). The Devices Work Group
agreed; the request is to flip DeviceAlert's standards-status from `draft` to
`normative` so the banner and per-row flag are consistent with the Work
Group's intended status for the resource.

## Feature Proposal

### Problem Statement

The DeviceAlert resource is, per the Work Group's intent, **Normative** in
this release, and `source/fhir.ini` already records this in the
`[standards-status]` section (line 1047: `DeviceAlert=normative`). The
`[fmg-approval]` section (line ~1355) similarly records `DeviceAlert=approved`.
However, the per-resource artifacts that drive the rendered page banner and
the row-level flag in the StructureDefinition view still carry the `draft`
value. Specifically, the `structuredefinition-DeviceAlert.xml`'s
`structuredefinition-standards-status` extension is set to `draft`, which
causes the publisher to:

- render the page-top banner as "Draft" rather than "Normative", and
- render a `D` (draft) flag against the resource row rather than `N`
  (normative).

This leaves the rendered specification inconsistent with the Work Group's
position and with the `[standards-status]` section of `fhir.ini`.

### Proposed Change

Update the `structuredefinition-standards-status` extension on the DeviceAlert
StructureDefinition from `draft` to `normative`. Concretely, in
`source/devicealert/structuredefinition-DeviceAlert.xml` (lines 12–14):

```xml
<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
  <valueCode value="normative" />
</extension>
```

(currently `<valueCode value="draft" />`).

This single-token change drives both the page-top banner ("Normative") and
the per-row standards-status flag (`N`) on the DeviceAlert specification
page, which were the two symptoms called out in the ticket.

No other changes to DeviceAlert metadata are mandated by the resolution. In
particular:

- `structuredefinition-fmm` remains `2` (the FMM is independent of standards
  status; it is not part of this ticket).
- `structuredefinition-wg` remains `devices`.
- The DeviceAlert entry in `source/fhir.ini`'s `[standards-status]` section
  already says `normative`, so no edit there is required — the file is
  already consistent with the proposed SD change.
- The `[fmg-approval]` section already records `DeviceAlert=approved`.

Optional, depending on Work Group convention (see Open Questions): a
`structuredefinition-normative-version` extension may be added immediately
after the standards-status extension, e.g.:

```xml
<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-normative-version">
  <valueCode value="6.0.0" />
</extension>
```

This is the pattern used by Patient (`4.0.0`), CapabilityStatement, Bundle,
Observation, etc. Many other resources marked `normative` in the
standards-status extension (e.g., MedicinalProductDefinition,
ActorDefinition, Requirements) do **not** carry this extension, so the
pattern is not universally enforced. Only add it if the Devices WG / FMG
intend to record 6.0.0 as the normative-from version for DeviceAlert.

### Design Rationale

The reporter's request is mechanical and the resolution is "Make the
requested changes as proposed." The fix follows the established mechanism
the publisher uses to drive both the page banner and the per-row flag —
the `structuredefinition-standards-status` extension — and aligns the SD
with the existing `fhir.ini` entries that already declare DeviceAlert as
`normative` and `approved`. No alternative was proposed in the ticket
discussion (there are no comments) and no alternative is needed: the same
mechanism is used by every existing normative resource (Patient, Bundle,
Observation, etc.).

`StructureDefinition.status` (PublicationStatus) is intentionally left
out of the required change. Survey of existing normative resources in the
repo shows the convention is mixed: many resources tagged `normative` in
the standards-status extension still carry `<status value="draft"/>`
(e.g., AdministrableProductDefinition, MedicinalProductDefinition,
PackagedProductDefinition, ActorDefinition, Requirements,
ManufacturedItemDefinition, ConceptMap, NamingSystem,
TerminologyCapabilities). Only the symptoms the ticket calls out (banner
+ row flag) are driven by the standards-status extension. Treating
`<status>` is captured below as an explicit Open Question rather than a
silent edit.

change for downstream implementers — the resource's content cannot break
backwards compatibility once normative. In strict element-cardinality /
binding terms, **no element-level breaking changes** are introduced by
this ticket; the change is purely a status flip on the SD wrapper. There
are no element additions, type changes, cardinality changes, binding
strength changes, or removed constraints.

### Step-by-Step Tasks

#### HL7/fhir

1. **Flip DeviceAlert's standards-status extension to `normative`**
   - File: `source/devicealert/structuredefinition-DeviceAlert.xml`
   - Action: In the `structuredefinition-standards-status` extension at
     the top of the file (currently lines 12–14), change
     `<valueCode value="draft" />` to `<valueCode value="normative" />`.
   - Details: The surrounding extension URL

     `name`, `title`, `description`, etc.) stay unchanged. Do not touch
     `<differential>` or `<snapshot>` content.

2. **(Optional) Add a `structuredefinition-normative-version` extension**

