## Feature Proposal

### Problem Statement

CGIT observed in 2015 that the FHIR specification did not document what
sections or materials an Implementation Guide should contain, nor how
those sections should be organised. Implementers authoring IGs had no
normative or informative reference for scope, actors, interactions,
conformance resources, or test scripts/requirements. The ticket
description captures this gap verbatim:

> CGIT thinks it is useful to determine what sections and materials
> can/should/must be in a FHIR implementation guide (e.g. scope,
> actors, interactions, conformance resources, test scripts/requirements)
> and how these are put into sections.

Discussion in comments (Andy McKenzie, 2015-05-25) agreed the topic was
important but not ready for DSTU 2 codification. Conformance WG
(Nathan Bunker, 2021-03-02) took an action item to provide direction.
The 2019-01-23 resolution committed to a **draft** good-practice section
in core, linked to both the Profiling page and the `ImplementationGuide`
resource, and deferred any DSTU/Normative promotion to a later review.

### Proposed Change

Add a single, clearly-marked **draft** "Good Practice Guidance" block
that enumerates the expected IG sections, anchored in the two locations
required by the resolution. The change is documentation-only; no
StructureDefinition, element, cardinality, binding, or search-parameter
change is required.

**1. `source/profiling.html` — add a new sub-section.**

Insert a new `<h3 id="ig-good-practice">Good Practice for Implementation
Guides (Draft)</h3>` sub-section between the existing
`[%ig guidance%]` paragraph (ends at line 73) and the `Glossary`
sub-section (begins at line 74). The sub-section MUST:

- Explicitly label itself "Draft" in the heading and in an opening
  `<p class="stu-note">` (or equivalent notice block) stating that this
  guidance is not normative and may be promoted in a future release.
- Enumerate, with a bullet list, the expected IG sections identified by
  CGIT:
  - **Scope** — what the IG covers and does not cover; target
    jurisdictions, domains, use cases.
  - **Actors** — the systems / roles the IG expects to interact.
  - **Interactions** — the transactions / RESTful API usage the IG
    constrains or requires.
  - **Conformance resources** — `StructureDefinition` profiles &
    extensions, `ValueSet`, `CodeSystem`, `ConceptMap`,
    `CapabilityStatement`, `OperationDefinition`, `SearchParameter`,
    `ImplementationGuide` itself.
  - **Examples** — at least one instance per profile.
  - **Test scripts / requirements** — `TestScript`, `TestPlan`, or
    prose-form testing requirements.
  - **Distribution** — NPM package metadata and dependencies.
- Include a short paragraph on how these are organised into pages /
  sections inside an IG (Introduction, Artifacts, Support, Changes).
- Cross-link to the `ImplementationGuide` resource page
  (`implementationguide.html`) and to the Guidance Implementation Guide
  (the `[%ig guidance%]` template expands to the canonical URL at
  publish time).

**2. `source/implementationguide/implementationguide-notes.xml` — add a
matching sub-section.**

Insert a new `<a name="good-practice"></a><h3>Good Practice for IG
Authors (Draft)</h3>` block into `implementationguide-notes.xml`,
placed after the existing `<a name="authoring"></a>` block (so it
immediately follows "Authoring Implementation Guides"). The block
should be a short summary — one paragraph plus the same bullet list as
above — and link back to
`profiling.html#ig-good-practice` as the authoritative location, and
onward to `[%ig guidance%]` for the full Guidance Implementation Guide.

**3. Cross-link only — do not duplicate content.**

The full bullet list is authored once (on `profiling.html`). The
`implementationguide-notes.xml` block SHOULD summarise in 1–3 sentences
and link, to avoid divergence.

### Design Rationale

- **Resolution wording is prescriptive about *where* the change lives.**
  "Linked to the Profiling page and the `ImplementationGuide` resource"
  dictates both anchor locations; anything else would not satisfy the
  persuasive disposition.
- **"Draft" is mandatory.** The resolution commits to *evaluating*
  promotion "as part of Release 3". The heading and an STU-note
  callout preserve that deferral.
- **Single-author, cross-linked** is the standard FhirCore pattern for
  cross-cutting prose (see existing
  `implementationguide-notes.xml` → `packages.html#composition`
  cross-link for an analogous case). Duplicating the list in two places
  would create a maintenance burden and is rejected.
- **An alternative — removing the ticket as already satisfied by the
  Guidance IG link** — is rejected because the Guidance IG link is a
  bare one-liner (line 70 of `profiling.html`); it does not satisfy
  the resolution's commitment to an explicit "draft section" that names
  the expected sections/materials. The implementation plan here is the
  minimum change that can close the ticket consistently with its
  resolution.

### Step-by-Step Tasks

#### HL7/fhir

1. **Insert "Good Practice for Implementation Guides (Draft)"
   sub-section on the Profiling page.**
   - File: `source/profiling.html`
   - Action: Between the end of the existing `[%ig guidance%]`
     paragraph (currently ending at line 73, `</p>`) and the
     `<a name="glossary"></a>` anchor (currently line 74), insert a
     new block:
     ```html
     <a name="ig-good-practice"></a>
     <h3>Good Practice for Implementation Guides (Draft)</h3>
     <p class="stu-note">
       This guidance is <b>draft</b>. The Conformance Work Group will
       evaluate promoting it to normative status in a future release of
       FHIR. See also the <a href="[%ig guidance%]">Guidance
       Implementation Guide</a> for the authoritative, evolving version.
     </p>
     <p>A FHIR Implementation Guide is typically expected to document
       the following sections / materials:</p>
     <ul>
       <li><b>Scope</b> — what the IG covers and does not cover,
         target jurisdictions, domains, and use cases.</li>
       <li><b>Actors</b> — the systems or roles expected to
         interact under the IG.</li>
       <li><b>Interactions</b> — the RESTful / messaging / document /
         service transactions that the IG constrains or requires.</li>
       <li><b>Conformance resources</b> —
         <a href="structuredefinition.html">profiles</a> and
         extensions, <a href="valueset.html">ValueSets</a>,
         <a href="codesystem.html">CodeSystems</a>,
         <a href="conceptmap.html">ConceptMaps</a>,
         <a href="capabilitystatement.html">CapabilityStatements</a>,
         <a href="operationdefinition.html">OperationDefinitions</a>,
         <a href="searchparameter.html">SearchParameters</a>, and the
         <a href="implementationguide.html">ImplementationGuide</a>
         resource itself.</li>
       <li><b>Examples</b> — at least one instance per profile to
         illustrate intent.</li>
       <li><b>Test scripts / requirements</b> —
         <a href="testscript.html">TestScript</a>,
         <a href="testplan.html">TestPlan</a>, or prose-form testing
         requirements.</li>
       <li><b>Distribution</b> — NPM package metadata and dependency
         declarations (see <a href="packages.html">FHIR NPM Packages</a>).</li>
     </ul>
     <p>These are typically organised into pages or sections:
       Introduction (scope, actors, use cases), Artifacts (conformance
       resources and examples), Support (downloads, testing, change
       log), and Changes / History. See the
       <a href="implementationguide.html#good-practice">notes on the
       <code>ImplementationGuide</code> resource</a> for the author's
       perspective on the same list.</p>
     ```
   - Details: Use the `[%ig guidance%]` template macro — do not hard-code
     the URL. Use HL7 page-relative links (`valueset.html`,
     `testscript.html`, etc.) consistent with the rest of
     `profiling.html`. Ensure the `stu-note` CSS class exists in the
     publisher's stylesheet (it does — used elsewhere on
     `profiling.html`); if the project uses a different draft-marker
     class in the current HEAD, match that.

2. **Insert matching "Good Practice for IG Authors (Draft)" block on
   the `ImplementationGuide` resource notes.**
   - File: `source/implementationguide/implementationguide-notes.xml`
   - Action: Immediately after the existing `<a name="authoring"></a>`
     block (which ends with the paragraph referencing the Guidance IG),
     and before the `<a name="roles"></a>` block, insert:
     ```xml
     <a name="good-practice"></a>
     <h3>Good Practice for IG Authors (Draft)</h3>
     <p>
       An Implementation Guide is typically expected to document its
       <b>Scope</b>, <b>Actors</b>, <b>Interactions</b>,
       <b>Conformance resources</b> (profiles, extensions, value sets,
       code systems, concept maps, capability statements, operation
       definitions, search parameters, and the ImplementationGuide
       resource itself), <b>Examples</b>, <b>Test scripts /
       requirements</b>, and <b>Distribution</b> (NPM package)
       information, usually organised into Introduction, Artifacts,
       Support, and Changes sections.
     </p>
     <p>
       This list is <b>draft</b>; see
       <a href="profiling.html#ig-good-practice">Good Practice for
       Implementation Guides</a> on the Profiling page for the
       authoritative core enumeration, and the
       <a href="[%ig guidance%]">Guidance Implementation Guide</a> for
       the evolving detailed guidance.
     </p>
     ```
   - Details: Preserve the surrounding `<div xmlns…>` wrapper and XHTML
     schema location already declared at the top of the file. Match the
