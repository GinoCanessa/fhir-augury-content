## Problem Statement

The EU Laboratory Report IG currently allows `Patient` as the `Specimen.subject` to indicate the source of a specimen. This creates problems when the specimen source is a non-human animal (e.g., a dog that bit someone, a tick carrying Lyme disease):

1. **Semantic mismatch** — A dog that bites a patient is not a "subject of care" in the human healthcare system. Using `Patient` for such an entity is incorrect.
2. **Confusion risk** — Allowing multiple `Patient` resources in a report Bundle (one human, one animal) is confusing. Systems may follow the `Specimen.subject` Patient reference and incorrectly treat the animal as "the" patient.
3. **System contamination** — Non-human Patient records appearing in human-only systems could lead to pick-lists containing animals, which is impractical and embarrassing.

## Resolution Description

The resolution (Persuasive with Modification) aligns with the approach voted on in [FHIR-53384](https://jira.hl7.org/browse/FHIR-53384) (Applied in FHIR Core R6), which adds `RelatedPerson` to `Specimen.subject` reference targets. The four-part resolution for the EU Lab Report IG is:

1. **Create an Extension on `.subject`** with `valueReference` targeting only `RelatedPerson` (may be changed to an R6 backport extension when available)
2. **Create a Profile on `RelatedPerson`** with the [`practitioner-animalSpecies`](https://hl7.org/fhir/extensions/StructureDefinition-practitioner-animalSpecies.html) extension (value set binding TBD)
3. **Add guidance**: for a non-identifiable specimen source (e.g., tick — SNOMED `710069003 | Tick specimen (specimen) |`), `Specimen.type` with the appropriate code shall be used
4. **Set `Specimen.type` to `1..1`** (use data absent reason if unknown)

## Impact Analysis

### Scope

This is a **profile-level change** within the EU Laboratory Report IG. It does not require changes to FHIR Core or the FHIR Extensions pack — FHIR-53384 has already handled the core spec changes (adding `RelatedPerson` to `Specimen.subject` in R6).

### Breaking Changes

- **`Specimen.type` cardinality change from `0..1` to `1..1`**: This is a **tightening constraint** that could affect existing EU Lab Report instances that omit `Specimen.type`. Mitigation: the resolution specifies using `DataAbsentReason` if unknown, so compliant instances can always provide a value.
- **Behavioral impact on Specimen.subject**: Adding the extension to allow `RelatedPerson` is additive. The real behavioral change is in the **guidance** — implementers should stop using `Patient` for animal specimen sources in human-context lab reports and use `RelatedPerson` instead.

### Dependencies

- **FHIR-53384** (Applied) — Prerequisite for the R6 backport extension approach. If the EU Lab IG targets R5, a custom extension is needed until the R6 backport extension becomes available.
- **`practitioner-animalSpecies` extension** — Already exists and supports `RelatedPerson` context. The EU Lab IG profile on `RelatedPerson` would use this extension directly.
- **Value set binding for animal species** — The resolution notes "vs binding?" as TBD. The existing `http://hl7.org/fhir/ValueSet/animal-species` from FHIR Extensions is the natural candidate (example binding strength).

## Implementation Plan

> **Note**: All file paths below are in the `hl7-eu/laboratory` repository, which is NOT in the cached repos. Exact paths must be verified against that repo's structure.

### Task 1: Create Extension on Specimen.subject for RelatedPerson reference

**Goal**: Allow `Specimen.subject` in the EU Lab Specimen profile to reference `RelatedPerson`, using an extension (since R5 `Specimen.subject` does not include `RelatedPerson` natively).

**Details**:
- Create a new `StructureDefinition` of type `Extension` that adds a `valueReference` targeting only `RelatedPerson`
- Set context to `Specimen.subject`
- This is a bridge extension until R6 (where `RelatedPerson` is natively on `Specimen.subject`) — document that this may be replaced with an R6 backport extension when available
- File: likely `input/fsh/extensions/SpecimenSubjectRelatedPerson.fsh` or equivalent FSH/XML definition

**FSH sketch**:
```fsh
Extension: SpecimenSubjectRelatedPersonEuLab
Id: specimen-subject-relatedperson-eu-lab
Title: "Specimen Subject - RelatedPerson (EU Lab)"
Description: "Extension to allow Specimen.subject to reference a RelatedPerson, for cases where the specimen source is not the subject of care (e.g., animal bite). This extension may be replaced by an R6 backport extension when available."
Context: Specimen.subject
* value[x] only Reference(RelatedPerson)
```

### Task 2: Create RelatedPerson Profile with animalSpecies extension

**Goal**: Define a profile on `RelatedPerson` that includes the `practitioner-animalSpecies` extension for identifying the species of the animal.

**Details**:
- Create a `StructureDefinition` profiling `RelatedPerson`
- Include the `practitioner-animalSpecies` extension (URL: `http://hl7.org/fhir/StructureDefinition/practitioner-animalSpecies`)
- Determine value set binding — the natural candidate is `http://hl7.org/fhir/ValueSet/animal-species` (currently `example` binding strength in the extension; the EU Lab profile may want to use `preferred` or `extensible`)
- File: likely `input/fsh/profiles/RelatedPersonAnimalEuLab.fsh` or equivalent

**FSH sketch**:
```fsh
Profile: RelatedPersonAnimalEuLab
Parent: RelatedPerson
Id: relatedperson-animal-eu-lab
Title: "RelatedPerson - Animal (EU Lab)"
Description: "Profile on RelatedPerson for representing a non-human specimen source (e.g., dog, tick) in the EU Laboratory Report context."
* extension contains
    http://hl7.org/fhir/StructureDefinition/practitioner-animalSpecies named animalSpecies 0..1 MS
* extension[animalSpecies].valueCodeableConcept from http://hl7.org/fhir/ValueSet/animal-species (preferred)
```

### Task 3: Update EU Lab Specimen Profile — set Specimen.type to 1..1

**Goal**: Tighten the cardinality of `Specimen.type` to require it.

**Details**:
- In the existing EU Lab Specimen profile, change `Specimen.type` from `0..1` to `1..1`
- Add guidance that `DataAbsentReason` extension should be used if the type is unknown
- This ensures that for non-identifiable specimen sources (e.g., tick), the specimen type conveys the nature of the specimen directly
- File: the existing EU Lab Specimen profile FSH/XML file (must be located in the `hl7-eu/laboratory` repo)

### Task 4: Add Guidance Documentation

**Goal**: Document when to use `Specimen.type` vs `Specimen.subject` (via RelatedPerson) for non-human specimen sources.

**Details**:
- Add narrative guidance to the IG stating:
  - For a **non-identifiable specimen source** (e.g., tick, mosquito, environmental sample): use `Specimen.type` with the appropriate SNOMED code (e.g., `710069003 | Tick specimen (specimen) |`). No `RelatedPerson` is needed.
  - For an **identifiable animal specimen source** (e.g., a known dog): use the `SpecimenSubjectRelatedPerson` extension to reference a `RelatedPersonAnimalEuLab` instance with the `animalSpecies` extension.
  - Do **not** use `Patient` for animal specimen sources in a human healthcare context.
- File: likely `input/pagecontent/specimen.md` or similar IG page

### Task 5: Update Examples

**Goal**: Update or create specimen examples that demonstrate the correct patterns.

**Details**:
- Ensure existing examples that used `Patient` for animal specimen sources are updated to use `RelatedPerson` via the new extension
- Add an example of a tick specimen using only `Specimen.type` (no separate subject resource)
- Add an example of a dog-bite scenario using the `RelatedPersonAnimalEuLab` profile

## Zulip Discussion Summary

The [PA WG Zulip thread](https://chat.fhir.org/#narrow/stream/patient%20administration%20WG/topic/Specimen%20source%20RelatedPerson%20%7C%20Patient) (Nov 2025 – Mar 2026, 50 messages, 10 participants) established consensus that:

- `Patient` is semantically wrong for animals that are not subjects of care in a human system
- `RelatedPerson` is the best fit, despite the name not being ideal for animals
- The `practitioner-animalSpecies` extension already supports `RelatedPerson` context
- FHIR-53384 was created and voted on in O&O to add `RelatedPerson` to `Specimen.subject` in R6 Core
- For R5-based IGs, a bridge extension is needed until R6 backport extensions are available

## Open Questions

1. **Value set binding strength** for `animalSpecies` on the EU Lab `RelatedPerson` profile — the resolution notes "vs binding?" as TBD
2. **R6 backport extension availability** — when available, the custom extension (Task 1) should be replaced
3. **Bundle cardinality** — the original ticket proposed limiting to one Patient per bundle; the resolution does not explicitly address this. Should the EU Lab Bundle profile constrain `Patient` entries to exactly one?
