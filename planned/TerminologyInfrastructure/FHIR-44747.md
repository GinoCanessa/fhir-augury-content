## Resolution Summary

The committee accepted the request as **Persuasive with Modification**. Rather than linking the ValueSet operation pages directly to the tooling-IG `tx-issue-type` CodeSystem (which is not part of FHIR Core), the agreed change is to **add an implementer note that points to the Terminology Ecosystem IG** in the following four places:

1. `Codesystem-operation-validate-code` — the `$validate-code` operation page on `CodeSystem`.
2. `Valueset-operation-validate-code` — the `$validate-code` operation page on `ValueSet`.
3. `Valueset-operation-expand` — the `$expand` operation page on `ValueSet`.
4. The **Terminology Service** page.

Exact resolution text to be added to each location (verbatim from the resolution description):

> Implementers should note that additional requirements are made on terminology servers that wish to be part of the public HL7 terminology ecosystem in the [terminology ecosystem IG](https://hl7.org/fhir/uv/tx-ecosystem/).

The original ask (linking `tx-issue-type` from the operation pages, or constraining the `issues` out-parameter to a tx-specific OperationOutcome profile) is **explicitly not adopted** in this resolution: TI's prior tracker discussion (2024-04-22) noted that core cannot reference an unreleased tooling-IG artifact, and the resolution narrows the change to a single ecosystem-IG pointer.

## Feature Proposal

### Problem Statement

The `ValueSet/$validate-code`, `ValueSet/$expand`, and `CodeSystem/$validate-code` operations all surface terminology-specific error and warning information in their `OperationOutcome` results, but the FHIR Core operation pages and the Terminology Service page give implementers no signal that *additional* conformance requirements (issue-type coding, server behaviors, etc.) exist for servers that participate in the public HL7 terminology ecosystem. Those additional requirements live in the [Terminology Ecosystem IG](https://hl7.org/fhir/uv/tx-ecosystem/) but there is no inbound pointer from FHIR Core. As Gino's 2025-02-27 comment summarises: "this ticket is just asking to add a link […] somewhere in the definition and page content."

### Proposed Change

Add a short implementer note (one sentence with an outbound link to `https://hl7.org/fhir/uv/tx-ecosystem/`) to each of the three terminology operation definitions and to the Terminology Service page. Concretely:

- **`source/valueset/operationdefinition-ValueSet-expand.xml`** — append the implementer-note sentence to the existing `<extension url="http://hl7.org/fhir/build/StructureDefinition/footer">/<valueMarkdown>` block (line ~39). The footer markdown is what the publisher renders below the operation parameter table on `valueset-operation-expand.html`.
- **`source/valueset/operationdefinition-ValueSet-validate-code.xml`** — append the same implementer-note sentence to the existing footer `<valueMarkdown>` (line ~39). Renders below the operation table on `valueset-operation-validate-code.html`.
- **`source/codesystem/operationdefinition-CodeSystem-validate-code.xml`** — this OperationDefinition does **not** currently carry a footer extension. Add a new `<extension url="http://hl7.org/fhir/build/StructureDefinition/footer">` with `<valueMarkdown value="…"/>` containing the implementer note, placed alongside the other build-only extensions near the top of the resource (i.e., after the existing `http://hl7.org/fhir/build/StructureDefinition/example` extensions and before the `<url>` element, mirroring the placement used in the two ValueSet operation files).
- **`source/terminology-service.html`** — add an HTML paragraph carrying the same note. Drop it into the page introduction (immediately before the `<a name="conformance"></a>` anchor / `<h3>Conformance</h3>` heading near the top of the file) so it is visible to anyone landing on the page.

Suggested rendered markdown for the operation footers:

```markdown
Implementers should note that additional requirements are made on terminology servers that wish to be part of the public HL7 terminology ecosystem in the [terminology ecosystem IG](https://hl7.org/fhir/uv/tx-ecosystem/).
```

Suggested HTML snippet for `terminology-service.html`:

```html
<p>
Implementers should note that additional requirements are made on terminology servers that wish
to be part of the public HL7 terminology ecosystem in the
<a href="https://hl7.org/fhir/uv/tx-ecosystem/">terminology ecosystem IG</a>.
</p>
```

No StructureDefinition / ValueSet / CodeSystem changes are made — `OperationOutcome.issue.code` binding, the `tx-issue-type` CodeSystem, and the operation parameter signatures all remain untouched.

### Design Rationale

- **Persuasive *with Modification*.** The original ticket asked for a direct link to `tx-issue-type`, optionally with a profile on `OperationOutcome` and a `targetProfile` on the `issues` out-parameter. The TI 2024-04-22 tracker call ruled that out: core cannot link to an unreleased tooling-IG artifact, and `OperationOutcome` is a generic resource not specific to terminology. The resolution therefore replaces a code-system-specific link with a single, durable pointer to the **Terminology Ecosystem IG**, which is the canonical home for additional terminology-server conformance requirements. This is an inexpensive, fully backward-compatible change that addresses the implementer-discoverability problem without coupling FHIR Core to non-core artifacts.
- **Footer extension over `<description>` edit.** The build-only `footer` extension is the established mechanism in FHIR Core for adding implementer prose below an OperationDefinition's parameter table (already used by both `ValueSet-expand` and `ValueSet-validate-code`). Using it for `CodeSystem-validate-code` keeps authoring style consistent across the three terminology-operation pages.
- **Same wording in all four locations.** The resolution description gives the exact text; using identical wording everywhere keeps the implementer signal consistent and trivially greppable.
