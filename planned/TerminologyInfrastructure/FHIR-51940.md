## Resolution Summary

The resolution (Persuasive) directs that a new input parameter be added to
`CodeSystem/$validate-code` that aligns with the existing `useSupplement`
parameter on `ValueSet/$validate-code`. The exact specification from the
resolution description is:

- **Name:** `useSupplement`
- **Cardinality:** `0..*`
- **Type:** `canonical`
- **Documentation:** "When using this input parameter, all provided
  supplements must be used when validating the display or properties of the
  concept code. Use of this parameter should result in $validate-code
  behaving the same way as if the supplements were included in an 'all
  codes' value set definition using the
  http://hl7.org/fhir/StructureDefinition/valueset-supplement"

Note the resolution wording is slightly stronger than the original ticket
description: it explicitly covers both **display** and **property**
validation, and frames the equivalent semantics in terms of an "all codes"
value set carrying the `valueset-supplement` extension (rather than the
"value set definition" wording in the original).

## Feature Proposal

### Problem Statement

Today `CodeSystem/$validate-code` has no way for a client to direct the
server to consult one or more `CodeSystem` supplements when validating a
code's display (or properties). The companion operation
`ValueSet/$validate-code` already exposes a `useSupplement` input parameter
that lets the caller pin specific supplements; that capability does not
exist on the `CodeSystem` variant.

This gap blocks the local-display-validation use case raised in
FHIR-50851 ("Define an extension for CodeSystem to allow defining private
code system spaces"). When a client holds a code/display pair drawn from a
local supplement (e.g., a translated or organisation-specific display), it
cannot ask the terminology server to validate that pair against the base
CodeSystem **plus** the supplement(s) — there is no parameter to nominate
the supplement(s).

### Proposed Change

Add one new input parameter to the `CodeSystem/$validate-code` operation
definition:

- **Name:** `useSupplement`
- **Use:** `in`
- **Min / Max:** `0` / `*`
- **Type:** `canonical`
- **Documentation:** "When using this input parameter, all provided
  supplements must be used when validating the display or properties of the
  concept code. Use of this parameter should result in $validate-code
  behaving the same way as if the supplements were included in an 'all
  codes' value set definition using the
  [[[http://hl7.org/fhir/StructureDefinition/valueset-supplement]]]"

The `[[[…]]]` form is the publisher's link-resolution syntax used by the
sibling parameter on `ValueSet/$validate-code` (verified in
`source/valueset/operationdefinition-ValueSet-validate-code.xml`, line 180);
keeping the same form preserves consistent rendering.

The new parameter sits naturally alongside the existing `displayLanguage`
input parameter (the last `in` parameter today) and before the `out`
parameters block. Snippet (XML, to be inserted in the operation
definition):

```xml
<parameter>
  <name value="useSupplement"/>
  <use value="in"/>
  <min value="0"/>
  <max value="*"/>
  <documentation value="When using this input parameter, all provided supplements must be used when validating the display or properties of the concept code. Use of this parameter should result in $validate-code behaving the same way as if the supplements were included in an &quot;all codes&quot; value set definition using the [[[http://hl7.org/fhir/StructureDefinition/valueset-supplement]]]"/>
  <type value="canonical"/>
</parameter>
```

No `scope` element is required (the equivalent parameter on
`ValueSet/$validate-code` likewise has no `scope`); the parameter is valid
at both type-level and instance-level invocations.

### Design Rationale

- **Parity with `ValueSet/$validate-code`.** The existing `useSupplement`
  parameter on that operation has identical name, cardinality (`0..*`),
  type (`canonical`), and intended semantics. Mirroring it on the
  `CodeSystem` variant gives implementers a single, predictable pattern
  for "validate against base + supplements", regardless of whether the
  client is hitting `ValueSet/$validate-code` or
  `CodeSystem/$validate-code`.
- **Canonical typing.** Supplements are themselves `CodeSystem` resources
  with `content = supplement` and a `supplements` element pointing at the
  base CodeSystem. They are addressed by canonical URL, so `canonical` is
  the right input type (matching the ValueSet variant).
- **Cardinality `0..*`.** Multiple supplements may legitimately apply to a
  single base CodeSystem (e.g., translation supplement + property
  supplement + jurisdictional display supplement). The ValueSet variant
  already allows this.
- **Behavioural equivalence anchor.** The resolution explicitly frames the
  semantics by reference to the `valueset-supplement` extension applied to
  an "all codes" value set definition. This both reuses an existing,
  understood normative mechanism and avoids re-specifying merge semantics
  from scratch.
- **Alternative considered (and rejected by the resolution wording):**
  passing supplements inline as `CodeSystem` resources (analogous to the
  existing `codeSystem` input). The resolution chose the canonical-URL
  form for consistency with the ValueSet variant; clients that need to
  hand the server an unknown supplement can still do so by other means
  (e.g., transaction-scoped POST), but the operation's parameter surface
  stays narrow.

## Implementation Plan

### Prerequisites

None. The `valueset-supplement` extension referenced from the new
parameter documentation already exists in `HL7/fhir-extensions`, and the
`CodeSystem` supplement model (`content = supplement`,
`CodeSystem.supplements`) is already normative. FHIR-50851's outcome
does not gate this change — `useSupplement` is useful in its own right
for any deployment that uses CodeSystem supplements.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add `useSupplement` input parameter to `CodeSystem/$validate-code`**
   - File: `source/codesystem/operationdefinition-CodeSystem-validate-code.xml`
   - Action: Insert a new `<parameter>` block immediately after the
     existing `displayLanguage` parameter (which currently ends at line
     146 with `</parameter>`) and immediately before the first output
     parameter, `result` (currently starts at line 147 with
     `<parameter>`).
   - Details: The inserted block must be:

     ```xml
     <parameter>
       <name value="useSupplement"/>
       <use value="in"/>
       <min value="0"/>
       <max value="*"/>
       <documentation value="When using this input parameter, all provided supplements must be used when validating the display or properties of the concept code. Use of this parameter should result in $validate-code behaving the same way as if the supplements were included in an &quot;all codes&quot; value set definition using the [[[http://hl7.org/fhir/StructureDefinition/valueset-supplement]]]"/>
       <type value="canonical"/>
     </parameter>
     ```

     Notes: (a) Do **not** include a `<scope>` element — the matching
     parameter on `ValueSet/$validate-code` (lines 175–182 of
     `source/valueset/operationdefinition-ValueSet-validate-code.xml`)
     omits `scope`, and the new parameter applies to both type-level and
     instance-level invocations of the operation. (b) Use the
     publisher's `[[[…]]]` link syntax around the canonical URL, exactly
     as the ValueSet variant does, so the URL renders as a hyperlink in
     the published page. (c) Quote the inner double-quotes around `all
     codes` as `&quot;` per FHIR XML attribute escaping (mirrors the
     escape style used elsewhere in the same file, e.g., the description
     string at line 57).

2. *(Optional, recommended)* **Add a worked example showing `useSupplement`**
   - File: `source/codesystem/$validate-code-request.txt`

   - Details: Optional and at the workgroup's discretion — the

     keep the example syntactically consistent with the existing entries

3. **Local sanity check before raising the PR**
   - File: `source/codesystem/operationdefinition-CodeSystem-validate-code.xml`
   - Action: Re-validate that the file is well-formed XML, that the new
     parameter block sits between `displayLanguage` and `result`, and

   - Details: Inspect the file in a viewer and confirm parameter

### Testing Considerations

- **Specification build:** the bespoke HL7 publisher must successfully
  re-render the `CodeSystem/$validate-code` operation page with the new
  parameter shown in the parameter table and the supplement extension
  rendered as a hyperlink.
- **Cross-page consistency:** confirm that the rendered descriptions of
  `useSupplement` on `CodeSystem/$validate-code` and
  `ValueSet/$validate-code` are textually consistent (both reference the
  same `valueset-supplement` extension URL).
- **Server implementations** (out of scope for this repo, but worth
  flagging to implementers): terminology servers should treat the
  parameter as optional input, accept `0..*` canonical URLs, resolve
  each to a `CodeSystem` resource with `content = supplement` whose
  `supplements` element matches the base CodeSystem under validation,
  and then perform display/property validation as if the supplement's
  designations and properties had been merged into the base CodeSystem
  for the purposes of the operation.
- **Edge cases to verify against future server-side test suites:**
  - Caller passes a `useSupplement` whose `supplements` element does not
    match the base CodeSystem URL → should be a validation error /
    OperationOutcome issue.
  - Multiple `useSupplement` values that disagree on display for the
    same code → server-defined precedence; document if not already
    covered.
  - `useSupplement` combined with `displayLanguage` → supplement
    designations in the requested language should take precedence.

### Open Questions

- **Worked example:** the plan leaves the sample request update as
  optional. The Terminology Infrastructure WG may wish to mandate one
  for parity with how `ValueSet/$validate-code` is illustrated.
- **Scope element:** the matching parameter on `ValueSet/$validate-code`
  has no `scope`. This plan likewise omits `scope`. If the WG later
  decides this parameter should only be valid at the type level (or
  only at the instance level), a `<scope value="…"/>` element will need
  to be added. The resolution text does not constrain this, so the
  default-mirror approach is taken here.
- **Cross-references:** the cross-reference index returned no GitHub PRs
  / commits, no related Jira tickets beyond the FHIR-50851 link
  embedded in the description, and no Zulip threads for FHIR-51940.
  Implementers should still grep the latest publisher build for any
