## Resolution Summary

The committee accepted the report as **Persuasive**: the `Variable Role` CodeSystem
(`http://hl7.org/fhir/variable-role`) currently has its `name` and `title`
elements reversed relative to the FHIR conventions. `name` must be a
machine-readable computational name (PascalCase, no spaces) and `title` must
be the human-friendly label. Today the CodeSystem has
`name="Variable Role"` and `title="VariableRole"`, which is backwards.
The fix is to swap the two values so `name="VariableRole"` and
`title="Variable Role"`.

## Feature Proposal

### Problem Statement

In `source/evidence/codesystem-variable-role.xml` the FHIR Core
`variable-role` CodeSystem violates the conformance convention used
throughout the specification (and used by its companion ValueSet
`source/evidence/valueset-variable-role.xml`, which already has
`name="EvidenceVariableRole"` and `title="Evidence Variable Role"`):

```xml
<name value="Variable Role"/>
<title value="VariableRole"/>
```

`CodeSystem.name` is required to be a string usable as a code-generation
identifier (machine-readable, typically PascalCase, no whitespace).
`CodeSystem.title` is the optional human-friendly display string. The
current values are reversed: the human-friendly form sits in `name` and
the computational form sits in `title`. This breaks tooling that uses
`CodeSystem.name` to generate class/enum identifiers, and produces a
poor display in any UI that surfaces `title`.

The defect was reported as a **Technical Correction**, indicating the
change is editorial in nature rather than a substantive design change.

### Proposed Change

Swap the values of `<name>` and `<title>` on the
`http://hl7.org/fhir/variable-role` CodeSystem so they become:

```xml
<name value="VariableRole"/>
<title value="Variable Role"/>
```

Concretely:

- File: `source/evidence/codesystem-variable-role.xml`
- Element: `CodeSystem.name` — change value from `Variable Role` to `VariableRole`.
- Element: `CodeSystem.title` — change value from `VariableRole` to `Variable Role`.

No other elements (`id`, `url`, `version`, `identifier`, `status`,
`description`, `caseSensitive`, `content`, `valueSet`, the four
concepts, or the WG/standards-status/FMM extensions) change. The
companion ValueSet (`source/evidence/valueset-variable-role.xml`)
already follows the convention and is **not** modified.

After the swap the CodeSystem aligns with the bound ValueSet:

| Artifact | name | title |
|----------|------|-------|
| CodeSystem `variable-role` (after) | `VariableRole` | `Variable Role` |
| ValueSet `variable-role` (unchanged) | `EvidenceVariableRole` | `Evidence Variable Role` |

(The differing tokens between CodeSystem.name and ValueSet.name
— `VariableRole` vs `EvidenceVariableRole` — pre-date this ticket and
are out of scope; the ticket is strictly the swap.)

### Design Rationale

FHIR specifies that `CanonicalResource.name` "SHOULD be usable as an
identifier for the module by machine processing applications such as
code generation" and that `title` is the "short, descriptive,
user-friendly title". Every other CodeSystem in `source/evidence/`
follows that convention; the `variable-role` CodeSystem is an outlier.
Swapping the values is the minimal, lowest-risk fix that brings the
artifact into line with both the FHIR base specification and its own
sibling ValueSet — no alternative was considered because the resolution
explicitly asks only for the swap.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Hosts the authoritative `http://hl7.org/fhir/variable-role` CodeSystem under `source/evidence/`.
- **Affected Files:**
  - `source/evidence/codesystem-variable-role.xml` — swap the values of `<name>` and `<title>`.
- **Change Scope:** Minor (two attribute-value edits on a single line each in one XML file).

### Breaking Changes

- Strictly, `CodeSystem.name` is a computational identifier, so changing it from `"Variable Role"` to `"VariableRole"` is an **observable change** to anyone who keyed off the previous (incorrect) value. In practice:
  - The current `name` value (`"Variable Role"`) contains a space and is not a valid programming identifier, so well-behaved code generators would have rejected or sanitised it; few real consumers can have used it verbatim.
  - The CodeSystem's `id`, `url`, `version`, OID identifier, `valueSet`, and concept codes/displays are unchanged, so all instance data and bindings remain valid.
- The CodeSystem is marked `normative` (FMM 1) via `structuredefinition-standards-status`. The committee categorised this as a **Technical Correction** under that status, signalling the change is editorial and acceptable for normative content.
- **Verdict:** No instance-data or wire-format break. There is a theoretical surface-name change for code-generation tooling, which is the stated intent of the ticket.

### Related Specifications

- `source/evidence/valueset-variable-role.xml` — bound ValueSet that includes this CodeSystem. Already follows convention; no edit required, but it is the consistency reference for the swap.
- The CodeSystem is bound from the `EvidenceVariable` resource (and related Evidence-family resources) under `source/evidencevariable/` and `source/evidence/`. Those resources reference the CodeSystem by `url`, not by `name`/`title`, so they are not affected.

### Related Tickets

- **None identified.** Cross-reference lookup for `FHIR-54573` returned zero hits across Jira, GitHub, Zulip, and Confluence; keyword-related search surfaces no companion tickets.

### Terminology Impact

- This ticket *is* a terminology change, but it is metadata-only — no concept codes, displays, definitions, hierarchy, or `valueSet` membership are altered. No downstream ValueSet expansion, binding, or subsumption result is affected.
- No UTG (`HL7/UTG`) edit is required. The deprecated UTG copy (`http://terminology.hl7.org/CodeSystem/variable-role`) is a different CodeSystem and already uses the corrected naming convention.

## Implementation Plan

### Prerequisites

- None. The change is self-contained within a single authored XML file in `HL7/fhir`. No dependent ticket, terminology addition, or extension definition is needed.

### Step-by-Step Tasks

#### HL7/fhir

1. **Swap the `name` element value to the machine-readable form.**
   - File: `source/evidence/codesystem-variable-role.xml`
   - Action: Change `<name value="Variable Role"/>` (currently line 20) to `<name value="VariableRole"/>`.
   - Details: `CodeSystem.name` must be a computational identifier suitable for code generation; PascalCase with no whitespace satisfies the constraint and matches the previous (mis-placed) `title` text.

2. **Swap the `title` element value to the human-friendly form.**
   - File: `source/evidence/codesystem-variable-role.xml`
   - Action: Change `<title value="VariableRole"/>` (currently line 21) to `<title value="Variable Role"/>`.
   - Details: `CodeSystem.title` is the user-facing display label. Using the spaced form mirrors the previously misplaced `name` text and aligns visually with the `EvidenceVariable` resource that consumes the binding.

3. **Verify no companion authored artifact still references the old name string.**
   - File: repo-wide grep across `source/`.
   - Action: Search for the literal string `"Variable Role"` and confirm any hits that remain are either prose mentions in HTML pages (acceptable — those are display copy) or the corrected `<title>` line above. No other authored CodeSystem/ValueSet/StructureDefinition should carry `name="Variable Role"`.
   - Details: Hand-editing one XML attribute should be sufficient; this step is just a guardrail to rule out a stale duplicate authoring file.

### Validation Checklist

- [ ] `source/evidence/codesystem-variable-role.xml` parses as valid XML and validates as a FHIR `CodeSystem` (publisher run completes without errors for the `evidence` resource group).

- [ ] Publisher-rendered pages for `EvidenceVariable` and the `variable-role` CodeSystem display the new `title` ("Variable Role") in headings and the new `name` ("VariableRole") in machine-readable contexts.

- [ ] Cross-references (`EvidenceVariable.role`, `EvidenceVariable.characteristic.role`, etc.) that bind to `http://hl7.org/fhir/ValueSet/variable-role` continue to resolve unchanged.
