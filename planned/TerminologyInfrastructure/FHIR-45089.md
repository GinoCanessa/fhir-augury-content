## Resolution Summary

**Resolution:** Persuasive — *"Agree with reporter to rename the profile."*

The Terminology Infrastructure work group accepted the proposal to rename the
base FHIR Core `ExecutableValueSet` profile to `ExpandedValueSet`, aligning the
core specification with the renaming already adopted in the CRMI IG (per the
parent ticket FHIR-44508 and the 2024-01-29 TI/CQI joint discussion). The
ticket carries the `R6` label (target release) and `TEXTONLY` label (the change
is concentrated in profile name / display text rather than structural ValueSet
content). No specific element-level edits were requested in the resolution; the
intent is a rename of the profile itself (URL/id, name, title, and any prose
that references the old name).

## Feature Proposal

### Problem Statement

The FHIR Core specification defines a profile on `ValueSet` whose URL/id is
`executablevalueset` and whose Name/Title is `ExecutableValueSet` (OID
assignment recorded in `source/oids.ini` line 1929; intro placeholder at
`source/valueset/profile-executablevalueset-introduction.xml`). The CRMI IG
ballot 1 reconciliation concluded that "ExpandedValueSet" more accurately
describes what the profile communicates (a ValueSet whose expansion has been
materialised), and the CRMI IG renamed its profile accordingly. With CRMI
having moved, the core profile name is now out of step with the only IG that
re-uses this concept, creating cross-spec confusion and forcing CRMI to choose
between matching CRMI history or matching FHIR Core. FHIR-44508 explicitly
spawned this R6 ticket to bring the core into line.

### Proposed Change

Rename the FHIR Core `ExecutableValueSet` profile to `ExpandedValueSet`. As a
`TEXTONLY` change against `R6`, the rename is to be carried out without altering
the ValueSet structural constraints expressed by the profile. Concretely:

- **Profile id / URL:** change `executablevalueset` →
  `expandedvalueset` (URL becomes
  `http://hl7.org/fhir/StructureDefinition/expandedvalueset`).
- **Profile Name:** `ExecutableValueSet` → `ExpandedValueSet`.
- **Profile Title / display:** `Executable ValueSet` → `Expanded ValueSet`
  (mirroring the title style used by the sibling `ShareableValueSet`,
  `ComputableValueSet`, `PublishableValueSet`).
- **Intro / narrative file:** rename
  `source/valueset/profile-executablevalueset-introduction.xml` →
  `source/valueset/profile-expandedvalueset-introduction.xml` (the file is
  currently an empty `<div>` placeholder — verified in the clone).
- **OID registry:** in `source/oids.ini`, replace the line
  `http://hl7.org/fhir/StructureDefinition/executablevalueset = 2.16.840.1.113883.4.642.5.1203`
  with the equivalent `…/expandedvalueset = 2.16.840.1.113883.4.642.5.1203`,
  preserving the OID so the registry identity is stable across the rename
  (consistent with how OIDs are kept across non-substantive renames).
- **Generated history snapshot:** `tools/history/release5/profiles-others.xml`
  contains historical R5 publisher output for `ExecutableValueSet` (id, URL,
  name) — leave as-is (R5 history) and let the publisher emit a new R6 entry
  under the new name on the next build.

The ticket is `TEXTONLY`, so no changes are proposed to the ValueSet base
resource (`source/valueset/structuredefinition-ValueSet.xml`) nor to the
individual profile constraints (cardinalities, types, bindings, invariants).
This is a name/identity rename only.

### Design Rationale

- The CRMI IG (the principal downstream consumer) has already adopted
  `ExpandedValueSet`; aligning the core avoids a permanent naming mismatch
  between the base spec and its primary IG (FHIR-44508 commentary).
- "Expanded" matches the semantics of the constraints (the profile requires the
  expansion to be present), which is what the CRMI reconciliation
  specifically called out.
- Sibling profiles (`shareablevalueset`, `computablevalueset`,
  `publishablevalueset`) already use simple lowercase ids without dashes and
  CamelCase Names; using `expandedvalueset` / `ExpandedValueSet` keeps the
  family pattern consistent.
- Preserving the OID across the rename matches how stable identifiers are
  treated for cosmetic/textual renames in `source/oids.ini` and avoids
  invalidating any external systems that bound to the OID rather than the URL.

### Breaking Changes

A profile **URL** rename is, by FHIR conventions, a breaking change for any
consumer that resolves
`http://hl7.org/fhir/StructureDefinition/executablevalueset` or asserts
`meta.profile = …/executablevalueset` on instances. In R6 this is acceptable
(R6 is a major version) and was the explicit intent of the resolution and the
upstream CRMI alignment. Specifically:

- Any conformant resource that declares
  `meta.profile = http://hl7.org/fhir/StructureDefinition/executablevalueset`
  will need to migrate to the new `…/expandedvalueset` URL.
- Any IG that imports / extends the old URL (notably any pre-rename CRMI
  artefact still pointing at FHIR Core) must update the reference.
- The OID `2.16.840.1.113883.4.642.5.1203` is preserved across the rename, so
  OID-keyed registry lookups continue to resolve.
- No change to the underlying constraints, so instances that already conform
  to `ExecutableValueSet` will continue to conform structurally to
  `ExpandedValueSet` after the rename.

### Step-by-Step Tasks

#### HL7/fhir

1. **Rename the profile intro file.**
   - File: `source/valueset/profile-executablevalueset-introduction.xml`
   - Action: Move/rename to
     `source/valueset/profile-expandedvalueset-introduction.xml`.
   - Details: File contents in the clone are an empty `<div>` placeholder
     (verified). Preserve the empty placeholder skeleton so the new file is
     wired in as the intro hook for the renamed profile (matching the pattern
     used by the sibling `profile-shareablevalueset-introduction.xml`,
     `profile-computablevalueset-introduction.xml`, and
     `profile-publishablevalueset-introduction.xml`).

2. **Update the OID assignment to the new profile URL.**
   - File: `source/oids.ini` (line 1929 in the cached clone)
   - Action: Replace
     `http://hl7.org/fhir/StructureDefinition/executablevalueset = 2.16.840.1.113883.4.642.5.1203`
     with
     `http://hl7.org/fhir/StructureDefinition/expandedvalueset = 2.16.840.1.113883.4.642.5.1203`.
   - Details: Keep the OID (`2.16.840.1.113883.4.642.5.1203`) unchanged so the
     registry identity is stable across the textual rename. Keep alphabetical
     ordering in the surrounding block consistent (the `expanded…` entry will
     sort earlier than `executable…`, so move it to the correct alphabetical
     slot next to `family-member-history-genetics-observation` neighbours
     rather than leaving it in the old position).

3. **Update the profile's authoritative declaration site to the new id / URL /
   name / title.** *(target file to be confirmed per Prerequisites)*
   - File: the publisher author file that currently emits
     `id=executablevalueset`, `url=http://hl7.org/fhir/StructureDefinition/executablevalueset`,
     `name=ExecutableValueSet`. (Not present in `source/` per `git grep`; see
     Open Questions.)
   - Action: Change
     - `id`: `executablevalueset` → `expandedvalueset`
     - `url`: `…/executablevalueset` → `…/expandedvalueset`
     - `name`: `ExecutableValueSet` → `ExpandedValueSet`
     - `title` (if present): `Executable ValueSet` → `Expanded ValueSet`
   - Details: Do **not** modify any structural constraint elements
     (cardinalities, types, bindings, invariants). The ticket is `TEXTONLY` /
     identifier-only. Do not edit
     `tools/history/release5/profiles-others.xml` — that is generated R5
     history and is in the briefing's "do not edit" set.

4. **Sweep for residual `executablevalueset` / `ExecutableValueSet` references
   inside `source/`.**
   - File: scope = `source/**`
   - Action: After steps 1–3, run
     `git grep -i -E "executablevalueset|Executable\s*ValueSet|StructureDefinition/executablevalueset"`
     against `source/` and address any remaining hits (anchors, intro prose,
     cross-page links, profile lists in `source/*.html`, `source/fhir.ini`
     manifest entries). The current clone has zero residual hits in `source/`
     beyond the two files above, but that should be re-verified after the
     declaration site (task 3) is updated.
   - Details: As the briefing's anchor-deletion gotcha highlights, also grep
     for the bare URL fragment / page name (`executablevalueset.html` if such
     a page is generated) so no inbound link is left dangling.

5. **Verify intro file wiring picks up the new placeholder.**
   - File: `source/valueset/profile-expandedvalueset-introduction.xml`
   - Action: Build the spec locally (per the briefing's "bespoke HL7
     publisher" notes, via `publish.sh` / `build.sh`) and confirm the rendered
     `expandedvalueset.html` page exists, references the renamed intro, and
     that no `executablevalueset.html` page is emitted.
   - Details: This validates that step 1's filename matches the publisher's
     intro-lookup convention (`profile-<id>-introduction.xml`).
