## Resolution Summary

The ticket originally proposed adding a new `handle-unclosed-expansion` boolean
input parameter (default `false`) to `ValueSet/$expand` so clients could opt in
to receiving expansions flagged with the `valueset-unclosed` extension. A
"Persuasive" vote on 2025-05-14 approved adding the parameter, and that
parameter was implemented in `source/valueset/operationdefinition-ValueSet-expand.xml`
(currently lines 231–238 of the cloned file).

On 2026-03-09 the Terminology Infrastructure WG **reverted** that resolution
to **"Not Persuasive with Modification"** because (a) defaulting to `false`
would silently break almost every SNOMED-backed value set client (Michael
Lawley, 2026-02-11) and (b) the previously approved parameter was never
applied throughout the rest of the spec, leaving the publication out of sync
with itself.

The replacement, agreed-upon resolution is **documentation-only**:

1. **Do not add the `handle-unclosed-expansion` parameter.** (And, since it
   was added speculatively after the now-superseded May 2025 vote, it must be
   **removed** so the spec is internally consistent.)
2. On the `$expand` operation page (`valueset-operation-expand.html`,
   §4.9.18.1), append the following sentence to the end of the last narrative
   paragraph before the examples:

   > Clients SHOULD be able to handle incomplete or unclosed value set
   > expansions. Servers return such ValueSet resources for a variety of
   > reasons (e.g., fragment, post-coordinated concepts, code systems with a
   > grammar, size). Servers SHOULD indicate incomplete or unclosed value set
   > expansions using the
   > [http://hl7.org/fhir/StructureDefinition/valueset-unclosed](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-valueset-unclosed.html).
3. In the same narrative, change the existing sentence:

   > **From:** "If the value set itself is unbounded due to the inclusion of
   > **post-coordinated value sets (e.g. SNOMED CT, UCUM)**, then the extension
   > [valueset-unclosed] can be used to indicate that the expansion is
   > incomplete"
   >
   > **To:** "If the value set itself is unbounded due to the inclusion of
   > **post-coordinated concept codes (e.g. SNOMED CT)**, then the extension
   > [valueset-unclosed] can be used to indicate that the expansion is
   > incomplete**.**"

   (i.e., narrow "value sets (e.g. SNOMED CT, UCUM)" → "concept codes (e.g.
   SNOMED CT)" and add the missing terminating period.)

The resolution explicitly notes that the broader question of paging
implications and whether clients should be able to *opt out of*
post-coordinated codes deserves a separate ticket.

## Feature Proposal

### Problem Statement

The `valueset-unclosed` extension
(`http://hl7.org/fhir/StructureDefinition/valueset-unclosed`) is defined to
let a server indicate that a returned `$expand` result is incomplete (e.g.,
because it contains post-coordinated SNOMED CT concepts whose codes cannot
all be enumerated). The original concern was that the extension's own
description says it should only be returned to clients that explicitly
indicate they handle it, yet the FHIR Core specification provides no formal
hand-shake for that capability.

The originally proposed solution — adding a `handle-unclosed-expansion`
boolean input to `$expand` defaulting to `false` — was rejected on appeal
because almost every SNOMED-backed value set is "unclosed" by nature, so a
default of `false` would either (a) cause servers to start returning errors
to existing clients, or (b) require every existing client to be updated
before the next FHIR release, both of which constitute a substantively
breaking change to a normative operation. The WG concluded that the correct
fix is to make the existing behaviour explicit in the specification text:
clients are expected to cope with unclosed expansions, and servers continue
to flag them via the existing extension.

### Proposed Change

All edits are confined to **`HL7/fhir`**, repository category **FhirCore**.

**Change 1 — Remove the speculative `$expand` parameter.**

Delete the `<parameter>` block for `handle-unclosed-expansion` from
`source/valueset/operationdefinition-ValueSet-expand.xml` (currently
lines 231–238 in clone `db79dcdfe1`):

```xml
<parameter>
  <name value="handle-unclosed-expansion"/>
  <use value="in"/>
  <min value="0"/>
  <max value="1"/>
  <documentation value="If true this asserts that you will correctly handle an unclosed expansion and the returned expansion SHALL include the valueset-unclosed extension if the value set is unclosed.  If handle-unclosed-expansion is set to false the server SHALL return an error if the value set is unclosed."/>
  <type value="boolean"/>
</parameter>
```

The remaining input parameters (`property` immediately before, and
`exclude-system` immediately after) keep their current positions.

**Change 2 — Update the operation's narrative text.**

The "From" paragraph the resolution quotes appears verbatim **twice** in
`source/valueset/operationdefinition-ValueSet-expand.xml`:

- Line 40 — inside the
  `http://hl7.org/fhir/build/StructureDefinition/footer` extension's
  `<valueMarkdown value="…"/>` (rendered as the operation page's footer
  block).
- Line 63 — inside the OperationDefinition's own `<comment value="…"/>`
  (rendered as §4.9.18.1 *Comment* on `valueset-operation-expand.html`).

Both occurrences must be edited identically. Within each value:

- Replace the substring
  `post-coordinated value sets (e.g. SNOMED CT, UCUM)` with
  `post-coordinated concept codes (e.g. SNOMED CT)`.
- Append a `.` immediately after `expansion is incomplete` so the sentence
  ends in a period.
- Append a new paragraph (preceded by `&#xA;&#xA;` to produce a blank line in
  the rendered Markdown):

  ```
  Clients SHOULD be able to handle incomplete or unclosed value set expansions. Servers return such ValueSet resources for a variety of reasons (e.g., fragment, post-coordinated concepts, code systems with a grammar, size). Servers SHOULD indicate incomplete or unclosed value set expansions using the [[[http://hl7.org/fhir/StructureDefinition/valueset-unclosed]]]
  ```

  (Use the `[[[...]]]` triple-bracket form because that is the publisher's
  link convention everywhere else in this file — the Markdown post-processor
  rewrites it into the live link to the extension page.)

After the edits, the closing paragraph of both blocks should read:

> When a server cannot correctly expand a value set because it does not
> fully understand the code systems (e.g. it has the wrong version, or
> incomplete definitions) then it SHALL return an error. If the value set
> itself is unbounded due to the inclusion of post-coordinated concept
> codes (e.g. SNOMED CT), then the extension
> `[[[http://hl7.org/fhir/StructureDefinition/valueset-unclosed]]]` can be
> used to indicate that the expansion is incomplete.
>
> Clients SHOULD be able to handle incomplete or unclosed value set
> expansions. Servers return such ValueSet resources for a variety of
> reasons (e.g., fragment, post-coordinated concepts, code systems with a
> grammar, size). Servers SHOULD indicate incomplete or unclosed value set
> expansions using the
> `[[[http://hl7.org/fhir/StructureDefinition/valueset-unclosed]]]`.

No new resources, profiles, extensions, search parameters, value sets, code
systems, or invariants are introduced. No element cardinalities, types, or
bindings change. The `valueset-unclosed` extension itself
(`http://hl7.org/fhir/StructureDefinition/valueset-unclosed`, defined in the
`HL7/fhir-extensions` IG) is **not** modified — it is only referenced.

### Design Rationale

- **Why prose, not a parameter?** A boolean parameter that defaults to
  `false` would change every existing client's effective behaviour overnight
  for any value set that includes SNOMED CT (the dominant real-world case).
  M. Lawley flagged this as a breaking change on 2026-02-11; the WG agreed
  on 2026-03-09. A boolean defaulting to `true` would have no semantic
  effect (it would just confirm the current behaviour), making the parameter
  redundant.
- **Why `SHOULD`, not `SHALL`?** Both clauses use `SHOULD` because some
  edge-case clients (e.g., bulk export pipelines that materialise codes
  ahead of time) genuinely cannot tolerate unbounded expansions; they remain
  free to special-case unclosed responses without the spec calling them
  non-conformant.
- **Why narrow "value sets (e.g. SNOMED CT, UCUM)" → "concept codes (e.g.
  SNOMED CT)"?** UCUM expansions are bounded in the way `$expand` cares
  about; the looseness only comes from post-coordination of *concept codes*,
  most prominently in SNOMED CT. The change tightens the example to the
  case the extension was actually designed for.
- **Out-of-scope follow-on.** Comment 4 records that the WG considered
  whether clients should be able to *exclude* post-coordinated concepts
  from `$expand` results (paging-cost / "too-costly" interaction). That
  belongs in a separate ticket and is explicitly not addressed here.

### Breaking Changes

- **Removal of the `handle-unclosed-expansion` input parameter** from the
  normative `ValueSet/$expand` OperationDefinition. The parameter was added
  to the `master` branch after the (since-reverted) 2025-05-14 vote and is
  present at lines 231–238 of the cached clone. Any client or server that
  was developed against an interim build of R6 since May 2025 and that
  speaks this parameter will need to drop it. This is a backward-
  incompatible change *relative to the interim spec*, although it is
  *forward-compatible* with the previously published normative R5 surface
  (which never had this parameter).

  Net assessment: **breaking against unreleased interim R6 only** — the
  resolution is, by intent, a course-correction back to the R5-shaped
  surface for this operation. It should be called out in R6 ballot notes
  and substantive-change tracking.

- **Wording change** ("post-coordinated value sets (e.g. SNOMED CT, UCUM)" →
  "post-coordinated concept codes (e.g. SNOMED CT)") narrows the documented
  applicability of the `valueset-unclosed` extension. Servers that currently
  return the extension for UCUM expansions would, under the new wording, no
  longer be doing so against an explicitly-documented use case — but the
  extension's own conformance does not forbid this, so it is not a hard
  break.

- **No element-level breaking changes** to any resource: no element added,
  removed, retyped, re-cardinalised, or re-bound.
