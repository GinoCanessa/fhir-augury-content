## Feature Proposal

### Problem Statement

The published table for `http://hl7.org/fhir/ValueSet/iso3166-1-2` shows ISO 3166-1 alpha-3 (3-letter) codes, but the value set's intensional definition restricts membership to alpha-2 codes via the filter `code matches regex [A-Z]{2}`. The codes shown in the rendered table therefore cannot satisfy the filter, so the value set is internally inconsistent / unusable as published.

Comments on the ticket establish:

- The expansion bug was a tx.fhir.org rendering issue; the CI-build definition is correct on its face, but the value set is a duplicate of `http://terminology.hl7.org/ValueSet/v3-Country2` (see `terminology.hl7.org/ValueSet-v3-Country2.html`) which already expands correctly.
- Maturity level of the FHIR-core copy is 0; in R5 the value set was owned by Terminology Infrastructure, and in R6 / CI build ownership is recorded as FHIR Infrastructure (see the `<extension url="…structuredefinition-wg"><valueCode value="fhir"/></extension>` on the ValueSet). Neither work group sees a reason to keep a duplicate of a THO value set in core.
- The only example binding in core that points at this value set is `Location.code` (R6), confirmed in the ticket comment and in `source/location/structuredefinition-Location.xml` line 192.

### Proposed Change

In `HL7/fhir` (`master`), retire the duplicate value set. The supported FHIR mechanism for retiring a value set without breaking the canonical / OID identity is to flip the `<status>` element from `active` to `retired`, leave the canonical URL, OID identifier, and version in place, and add a note in `<description>` pointing readers at the surviving THO value set. Concretely:

- File: `source/terminologies/valueset-iso3166-1-2.xml`
  - Change `<status value="active"/>` → `<status value="retired"/>`.
  - Update `<description>` from
    `"This value set defines the ISO 3166 Part 1 2-letter codes"`
    to something like
    `"Retired. Use http://terminology.hl7.org/ValueSet/v3-Country2 (HL7 Terminology) instead. This value set previously defined the ISO 3166 Part 1 alpha-2 codes; it duplicated the THO value set and the published table did not match the intensional regex filter (FHIR-51120)."`
  - Leave `<id>`, `<identifier>` (OID `2.16.840.1.113883.4.642.3.3351`), `<version>`, `<name>`, `<title>`, the `structuredefinition-wg` extension, and the existing `<compose>` block in place. Retiring rather than deleting preserves the canonical URL / OID for any historical references.

- File: `source/fhir.ini`
  - Leave the existing registration line `iso3166-1-2=terminologies\valueset-iso3166-1-2.xml` (line 1292) unchanged so the publisher continues to render and link the (now retired) page. Removing the registration would break the publish if anything still links the canonical URL; retired status is sufficient signaling.

- File: `source/oids.ini`
  - Leave `http://hl7.org/fhir/ValueSet/iso3166-1-2 = 2.16.840.1.113883.4.642.3.3351` (line 1038) in place. OID retention is required even after retirement so the identifier registry remains stable.

- File: `source/location/structuredefinition-Location.xml`
  - **No edit in this ticket.** The `Location.code` example binding (line 192) currently references `http://hl7.org/fhir/ValueSet/iso3166-1-2`. The resolution explicitly assigns the rebinding to a follow-up Patient Administration ticket. Flag this as a coordination concern (see Open Questions): once that PA ticket lands, the binding should be redirected to `http://terminology.hl7.org/ValueSet/v3-Country2` (or another THO value set PA chooses).

The matching 3-letter sibling `valueset-iso3166-1-3.xml` (`http://hl7.org/fhir/ValueSet/iso3166-1-3`) is **not** in scope. The ticket and resolution are exclusively about the 2-letter value set; the 3-letter sibling has the correct `[A-Z]{3}` regex and is left untouched.

### Design Rationale

- **Retire, don't delete.** FHIR resources are versioned by canonical URL; deleting a published value set would orphan inbound canonical references. Setting `<status value="retired"/>` is the conformance-correct way to signal "do not use" while preserving the canonical and OID for traceability. This matches comment guidance from Lloyd McKenzie ("we'd be doing schema enumeration… [no good reason]") and Carmela Couderc ("LM already said it's ok to remove… ok with resolving the ticket as proposed").
- **Point at the THO replacement in the description.** The TI tracker call (2025-12-01) and 2026-04-06 follow-up established `http://terminology.hl7.org/ValueSet/v3-Country2` as the surviving authority. Putting that URL in the retired value set's `<description>` gives implementers a programmatic breadcrumb without relying solely on release notes.
- **Defer the Location.code rebinding.** The resolution explicitly carves out the Patient Administration follow-up. Rebinding `Location.code` here would (a) cross work-group boundaries (TI is retiring; PA owns Location), and (b) pre-empt PA's choice of replacement value set. Honour the carve-out and document the dependency.
- **Don't touch `fhir.ini` or `oids.ini`.** The publisher driver and OID registry should continue to recognise the canonical so the retired-status page is still generated and the OID stays reserved. This mirrors how other retired value sets remain registered.
