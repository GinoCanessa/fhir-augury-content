## Resolution Summary

The ticket is resolved **Persuasive with Modification**. The approved
resolution (carmela_couderc, 2025-12-15) is a documentation-only
clarification — there is **no new structural element** added to
`CodeSystem.property` (the earlier 2024-09-26 idea of adding a sibling
"value set reference" element was dropped in favour of pointing at the
existing `codesystem-property-valueset` extension). Specifically:

1. In the `concept-property-type` CodeSystem, extend the definition of
   the `code` concept from
   > "The property value is a code that identifies a concept defined in the code system."

   to
   > "The property value is a code that identifies a concept defined in the code system **or in an another code system**."

2. In the `CodeSystem` StructureDefinition, change the definition of
   `CodeSystem.property.type` from
   > "The type of the property value. Properties of type \"code\" contain a code defined by the code system (e.g. a reference to another defined concept)."

   to
   > "The type of the property value. Properties of type \"code\" contain a code defined by the code system **or another code system**."

3. Add a comment on `CodeSystem.property.type`:
   > "If type is code, and a defined set of codes is to be used, these codes can be specified by using the extension [codesystem-property-valueset](https://hl7.org/fhir/extensions/StructureDefinition-codesystem-property-valueset.html)."

(Verbatim wording is taken from the resolution_description in the
ticket snapshot. The phrase "or in an another code system" is reproduced
as written; the implementer should leave the typo verbatim or seek WG
sign-off before normalising it — see Open Questions.)

## Feature Proposal

### Problem Statement

The FHIR specification is internally inconsistent about what a
`CodeSystem.property` of type `code` means. The `concept-property-type`
CodeSystem defines `code` as "a code that identifies a concept defined
in the code system", and the StructureDefinition of
`CodeSystem.property.type` reinforces that with "Properties of type
'code' contain a code defined by the code system (e.g. a reference to
another defined concept)". However, several standard properties shipped
in the spec — `status`, `synonym`, `comment` — clearly use codes that
are **not** defined in the owning code system, and THO has 1200+ uses of
`code`-typed properties for the same dual purpose (internal references
**and** values from a controlled set in another code system). The
result is that the prose definition contradicts current and intended
practice, and implementers cannot tell from the type alone whether a
`code` value is an internal reference, an external code, or merely a
free-form `code`-typed token.

### Proposed Change

Apply the three documentation edits listed in the Resolution Summary,
all inside `HL7/fhir`:

1. **`source/codesystem/codesystem-concept-property-type.xml`** —
   broaden the `code` concept's `<definition>` so the spec acknowledges
   that a `code`-typed property value may identify a concept in the
   owning code system **or in another code system**.

2. **`source/codesystem/structuredefinition-CodeSystem.xml`** — update
   the `<definition>` of element `CodeSystem.property.type` (line 848 in
   the current clone) to match the broadened semantics; the parenthetical
   "(e.g. a reference to another defined concept)" is replaced wholesale.

3. **`source/codesystem/structuredefinition-CodeSystem.xml`** — add a
   new `<comment>` child on the `CodeSystem.property.type` element
   pointing implementers at the
   `codesystem-property-valueset` extension when a constrained set of
   codes is intended. The element does not currently carry a `<comment>`
   (verified in the clone at lines 845–863), so this is a clean
   insertion immediately after `<definition>` and before `<min>`, in
   line with FHIR ElementDefinition child ordering.

No structural change is made: no new sibling element is added to
`CodeSystem.property`, no cardinality or type changes, no new search
parameters, no terminology-binding strength change. The existing
`required` binding of `CodeSystem.property.type` to
`http://hl7.org/fhir/ValueSet/concept-property-type` is unaffected.

### Design Rationale

Two alternatives were on the table during the multi-WGM discussion:

* **(A) Add a sibling element** to `CodeSystem.property` that references
  a value set for the allowable property values, falling back to "any
  internal reference" when absent. This was the 2024-09-26 joint FHIR-I
  / TI direction (carmela_couderc comment, 2024-09-26) and was the
  reporter's original R6 preference.
* **(B) Reuse the existing `codesystem-property-valueset` extension**
  and only correct the prose so the existing dual usage is no longer
  contradicted by the spec text.

Option (B) was selected at the 2025-11-20 TI tracker call (marc.duteau,
"codesystem-property-valueset seems to meet the need ... is that good
enough?") and reaffirmed by the reporter on 2025-12-08 ("I could live
with it remaining an extension, but if so, we still need to correct the
definitions, and it might be useful to mention the extension"). The
final resolution drafted by Carmela Couderc on 2025-12-15 implements (B)
verbatim. Choosing (B) avoids a breaking change to a normative resource
(`CodeSystem.property` is part of the normative `CodeSystem` boundary
and adding a new element with implementer expectations would be a
breaking design change), and it leverages a deliberately-published
extension that already exists in `HL7/fhir-extensions`. The September
2024 minutes (FHIR-I / TI joint, action item: "TI will write up a
specific resolution to the ticket. FHIR-I will be happy with the
resolution and TI can resolve and apply without their review.") give TI
the authority to apply this resolution unilaterally.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole repository for this change. Holds both the
  `concept-property-type` CodeSystem instance and the StructureDefinition
  for the `CodeSystem` resource that contains `CodeSystem.property.type`.
- **Affected Files:**
  - `source/codesystem/codesystem-concept-property-type.xml` — broaden
    the `<definition>` of the `code` concept (line 50) to acknowledge
    cross-code-system references.
  - `source/codesystem/structuredefinition-CodeSystem.xml` — replace the
    `<definition>` of element `CodeSystem.property.type` (line 848) and
    insert a new `<comment>` child between `<definition>` and `<min>`
    (lines 848 → 849) referencing the `codesystem-property-valueset`
    extension.
- **Change Scope:** **Minor** (prose-only; no cardinality, type,
  binding, invariant, or search-parameter changes; no new elements).

### Breaking Changes

No breaking changes identified.

The semantics of `CodeSystem.property` of type `code` are being
*relaxed*, not tightened: implementations that already used `code`-typed
properties for cross-code-system references (e.g., the THO usage cited
by the reporter, and the spec-internal `status` property) become
spec-conformant rather than spec-violating. No existing instance
becomes invalid, no invariant is added, and the `required` binding of
`CodeSystem.property.type` to the `concept-property-type` ValueSet is
unchanged. The only behavioural shift is that a validator can no longer
treat "the value of a `code`-typed property is not a code in this code
system" as an automatic error — but no published validator currently
enforces that constraint anyway.

### Related Specifications

- `CodeSystem` resource (normative `CodeSystem.property.type` element).
- The `codesystem-property-valueset` extension in `HL7/fhir-extensions`
  is referenced from the new `<comment>` and should remain reachable at
  `https://hl7.org/fhir/extensions/StructureDefinition-codesystem-property-valueset.html`.
- The `concept-property-type` ValueSet
  (`http://hl7.org/fhir/ValueSet/concept-property-type`) is unchanged
  in membership.
- THO/UTG content using `code`-typed properties cross-code-system
  becomes nominally conformant; no change required in `HL7/UTG`.

### Related Tickets

The reporter's description and the cross-reference set surface two
sibling tickets in the same area:

- **FHIR-44959** — "the standard code system property 'comment' has
  type 'code' which is wrong." Closely related: this ticket broadens
  what `code`-typed properties may legally do, but FHIR-44959 argues
  `comment` is simply mis-typed (should be `string`). The two changes
  are compatible — applying FHIR-44961 does not retroactively make
  `comment` correct (free text is not a "code defined in any code
  system"), so FHIR-44959 should still be implemented separately.
- **FHIR-44960** — "the definition of the standard code system
  property 'synonym' is ambiguous." Also compatible: FHIR-44961 makes
  the use of `code` for cross-code-system references legal, which is
  one of the interpretations FHIR-44960 is trying to disambiguate. The
  two should be cross-referenced in implementation notes.

No other tickets surfaced via cross-references for FHIR-44961 itself
(the `cross-referenced` call returned only the Zulip terminology thread
at `179202-terminology/finding.20an.20active.20concepts`, the September
2024 WGM minutes Confluence page, and FHIR mentions of
`CodeSystem.property` and `CodeSystem.property.type`).

### Step-by-Step Tasks

#### HL7/fhir

1. **Broaden the `code` concept definition in the
   `concept-property-type` CodeSystem.**
   - File: `source/codesystem/codesystem-concept-property-type.xml`
   - Action: At line 50, replace the `<definition>` value on the
     `code` concept (the one whose `<code value="code"/>` sits at line
     48 and `<display value="code (internal reference)"/>` at line 49).
   - Details:
     - Old:
       `<definition value="The property value is a code that identifies a concept defined in the code system."/>`
     - New:
       `<definition value="The property value is a code that identifies a concept defined in the code system or in an another code system."/>`
     - The verbatim resolution wording ("or in an another code system")
       is retained; see Open Questions for the typo discussion.
     - Do not modify the `<display>` ("code (internal reference)") in
       this commit; renaming the display is out of scope for the
       resolution and could itself be a breaking presentational change.
       Flag it as a candidate follow-up if reviewers raise it.

2. **Update the definition of `CodeSystem.property.type` in the
   `CodeSystem` StructureDefinition.**
   - File: `source/codesystem/structuredefinition-CodeSystem.xml`
   - Action: At line 848, inside the `<element id="CodeSystem.property.type">` block (lines 845–863), replace the
     existing `<definition value="…"/>` line.
   - Details:
     - Old:
       `<definition value="The type of the property value. Properties of type &quot;code&quot; contain a code defined by the code system (e.g. a reference to another defined concept)."/>`
     - New:
       `<definition value="The type of the property value. Properties of type &quot;code&quot; contain a code defined by the code system or another code system."/>`
     - Keep the existing escaping convention (`&quot;` for inline
       double-quotes around `code`); other element definitions in this
       file use the same convention.

3. **Add a `<comment>` to `CodeSystem.property.type` referencing the
   `codesystem-property-valueset` extension.**
   - File: `source/codesystem/structuredefinition-CodeSystem.xml`
   - Action: Insert a new `<comment>` child immediately after the
     updated `<definition>` (i.e., between current lines 848 and 849)
     and before `<min value="1"/>`. The element does not currently
     have a `<comment>` (confirmed in the clone), so this is a fresh
     insertion at the canonical ElementDefinition child position
     (after `definition`, before `min`).
   - Details:
     - New line:
       `<comment value="If type is code, and a defined set of codes is to be used, these codes can be specified by using the extension http://hl7.org/fhir/StructureDefinition/codesystem-property-valueset (see [codesystem-property-valueset](https://hl7.org/fhir/extensions/StructureDefinition-codesystem-property-valueset.html))."/>`
     - The canonical extension URL must appear in the comment so
       implementations can resolve it programmatically; the rendered
       hyperlink is for the human-facing publisher output. Match the
       inline-link style used elsewhere in this StructureDefinition
       (e.g., line 826 uses `[Concept Properties](codesystem-concept-properties.html)`).
     - Verify XML well-formedness: the comment value must escape any
       `<`/`&` characters; the proposed text contains none.

4. **Run a repo-wide check for stale prose duplicates.**
   - Files: anywhere under `source/` that quotes the old definition
     verbatim (e.g., narrative pages discussing `CodeSystem.property.type`).
   - Action: `git --no-pager grep -n "code defined by the code system (e.g. a reference to another defined concept)"`
     in the clone; update any narrative copies to match the new
     definition or rephrase to avoid duplicating it.
   - Details: Optional but recommended. The bespoke HL7 publisher does
     not auto-propagate definition text into prose pages; out-of-date
     duplicates would silently survive otherwise.

### Open Questions

1. **Verbatim grammar in the resolution.** The approved resolution
   text reads "a concept defined in the code system **or in an another
   code system**." The doubled article ("an another") is almost
   certainly a typo for "or in another code system". The plan above
   reproduces the resolution verbatim per the "Read the resolution
   description carefully" rule. **Decision needed before merge:**
   apply verbatim or normalise to "or in another code system"? A
   one-line confirmation from TI on the tracker should suffice; the
   2024-09-2024 minutes already authorise TI to apply without FHIR-I
   review.

2. **Display-value alignment.** The `code` concept in
   `concept-property-type` has display `"code (internal reference)"`,
   which after this change understates the concept's broadened
   meaning. Renaming the display (e.g., to `"code"` or
   `"code (internal or external reference)"`) is **not** in the
   resolution and is therefore out of scope here, but reviewers may
   raise it. Recommend opening a follow-up ticket rather than rolling
   it in silently — consistent with the briefing's gotcha that
   out-of-scope ticket comments should not be silently absorbed.

3. **Comment phrasing — extension URL form.** Should the new
   `<comment>` quote the canonical extension URL
   (`http://hl7.org/fhir/StructureDefinition/codesystem-property-valueset`)
   or only the human-facing link? The plan above includes both for
   robustness; if TI prefers the URL-free form used elsewhere in the
   StructureDefinition, drop the canonical URL and keep only the
   inline `[codesystem-property-valueset](…)` markdown link.
