## Resolution Summary

**Resolution:** Persuasive with Modification.

The committee accepted the proposed new operator but renamed it from the
proposed `of` to **`property-value-of`** to be self-describing in the
filter-operator code system. The applied wording from the resolution is:

- **Code:** `property-value-of`
- **Display:** `Property Value Of`
- **Definition:** *"Includes the concepts that are used as property values from
  the set of concepts in the provided filter.value element value
  (comma-separated list or value set). If the provided filter.value element
  value is a value set, any concepts not in the filter defining code system
  are ignored."*

The resolution explicitly notes the change must be made in **two places**:
(1) the formal CodeSystem (machine-readable `<concept>` entries) and (2) the
narrative table rendered at
`https://build.fhir.org/codesystem-filter-operator.html`. Within the
authoring repo, both surfaces live inside the single resource file
`source/valueset/codesystem-filter-operator.xml` — the `<text><div>` table
is hand-authored alongside the formal `<concept>` block.

## Feature Proposal

### Problem Statement

`ValueSet.compose.include.filter` lets an author select members of a code
system by *property value*. The existing operators (`=`, `in`, `is-a`,
`descendent-of`, `regex`, `exists`, …) all answer the question *"which
concepts have property X with value Y?"*. There is no inverse operator that
answers *"which concepts are used as the property value Y on some other
concept?"*.

The reporter's example: in LOINC the `component` property of any LOINC
observation code points at a LOINC Part code (`LP…`). Today an author can
write `property = component, operator = =, value = LP12345` to get every
observation whose component is `LP12345`. There is no way to ask the
inverse — *"give me every code that appears as the value of the `component`
property on any LOINC observation"* — i.e., enumerate the LOINC Parts that
are actually used as components. This is a real authoring gap for any
terminology that exposes inter-concept relationships through string-valued
properties (LOINC Parts, SNOMED relationship targets surfaced as strings,
etc.).

### Proposed Change

Add one new concept to the `filter-operator` CodeSystem
(`http://hl7.org/fhir/filter-operator`) and the parallel narrative table:

| Field | Value |
|-------|-------|
| Code | `property-value-of` |
| Display | `Property Value Of` |
| Definition | Includes the concepts that are used as property values from the set of concepts in the provided filter.value element value (comma-separated list or value set). If the provided filter.value element value is a value set, any concepts not in the filter defining code system are ignored. |
| Comments | (none) |

Concrete authoring deltas in `source/valueset/codesystem-filter-operator.xml`:

1. **Append a `<concept>` block** at the end of the existing concept list
   (after the `exists` concept, mirroring the order of the other entries):

   ```xml
   <concept>
     <code value="property-value-of"/>
     <display value="Property Value Of"/>
     <definition value="Includes the concepts that are used as property values from the set of concepts in the provided filter.value element value (comma-separated list or value set). If the provided filter.value element value is a value set, any concepts not in the filter defining code system are ignored."/>
   </concept>
   ```

2. **Append a matching `<tr>` row** to the `<table class="codes">` block
   inside `<text><div>`, following the same `<a name="filter-operator-…">`
   anchor pattern used by every existing row:

   ```xml
   <tr>
     <td style="white-space:nowrap">property-value-of
       <a name="filter-operator-property-value-of"> </a>
     </td>
     <td>Property Value Of</td>
     <td>Includes the concepts that are used as property values from the set of concepts in the provided filter.value element value (comma-separated list or value set). If the provided filter.value element value is a value set, any concepts not in the filter defining code system are ignored.</td>
     <td/>
   </tr>
   ```

Worked example demonstrating the new operator (matches the reporter's
LOINC scenario, updated to use the resolved code name):

```json
"include": [{
  "system": "http://loinc.org",
  "filter": [{
    "property": "component",
    "op": "property-value-of",
    "value": "http://loinc.org?vs"
  }]
}]
```

### Design Rationale

- **Why a new operator (vs. a reverse-property convention):** the existing
  operator vocabulary is enumerated by exactly this CodeSystem — the
  binding is `required` and immutable (the bound ValueSet sets
  `<immutable value="true"/>` and includes the entire CodeSystem). Adding
  a code is the only conformant way to express new behaviour without
  forking the binding.
- **Why the renamed code `property-value-of` (not the proposed `of`):**
  `of` is too short to be self-describing in lists/UI and collides with
  natural English usage in narrative text. The resolution's modification
  pins the name to its semantics: *the concept is the value of a property
  of one of the concepts in the filter value*. This matches the naming
  cadence of the existing entries (`is-a`, `descendent-of`, `child-of`,
  `descendent-leaf`, `is-not-a`, `not-in`).
- **Why allow both comma-separated lists and a value set in `filter.value`:**
  consistent with the existing `in` / `not-in` operators, whose definitions
  already accept "the set of codes or concepts specified in the provided
  value (comma-separated list)". The new operator extends the contract to
  also accept a value set canonical, which is more practical for the
  motivating LOINC use case (`http://loinc.org?vs`) and is reinforced by
  FHIR-51115's URI usage in `filter.value`.
- **Why the cross-system clipping clause:** when `filter.value` resolves to
  a value set whose members come from multiple code systems, only members
  drawn from the *defining* code system can be evaluated — concepts from
  other systems are silently ignored rather than raising an error. This
  keeps the operator usable against value sets without requiring authors
  to pre-filter them.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Hosts the `filter-operator` CodeSystem authored inside the
  ValueSet resource folder. The change is fully contained in this repo.
- **Affected Files:**
  - `source/valueset/codesystem-filter-operator.xml` — append one
    `<concept>` entry (`property-value-of`) and the parallel `<tr>` row
    inside the existing `<text><div><table class="codes">…</table>` block.
- **Change Scope:** Minor (single new code in an open-content,
  case-sensitive code system; one file edited).

### Breaking Changes

No breaking changes identified.

- The bound ValueSet (`http://hl7.org/fhir/ValueSet/filter-operator`)
  includes the entire CodeSystem (`<compose><include><system .../></include></compose>`
  with no `<concept>` enumeration), so the new code becomes a member
  automatically without touching the ValueSet file.
- The ValueSet is `immutable=true` only with respect to *which CodeSystem
  it draws from*, not the membership of that CodeSystem; adding a code
  does not violate immutability.
- The CodeSystem is `content=complete` with `caseSensitive=true`; adding
  a new code does not change the meaning of any existing code, and the
  new code does not collide (case-insensitively) with any existing code.
- Existing `required`-strength bindings continue to validate previously
  valid instances; this is purely additive.
- The CodeSystem and ValueSet are both `normative` / FMM 5; per FHIR
  versioning policy, additive changes to a normative code system are
  permitted as substantive content additions but are not breaking.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add the formal `property-value-of` concept**
   - File: `source/valueset/codesystem-filter-operator.xml`
   - Action: Append a new `<concept>` block at the end of the existing
     `<concept>` list (after the `exists` entry that closes the file
     before `</CodeSystem>`).
   - Details:
     - `<code value="property-value-of"/>`

If the provided filter.value element value is a value set, any concepts not in the filter defining code system are ignored."/>`

2. **Mirror the new code in the narrative table**
   - File: `source/valueset/codesystem-filter-operator.xml`
   - Action: Inside the `<text><div xmlns="http://www.w3.org/1999/xhtml">`
     element's `<table class="codes">`, append a new `<tr>` after the
     existing `exists` row (which is currently the last data row).
   - Details: Use the same four-column shape and anchor convention as
     every other row:

     ```xml
     <tr>
       <td style="white-space:nowrap">property-value-of
         <a name="filter-operator-property-value-of"> </a>
       </td>
       <td>Property Value Of</td>

       <td/>
     </tr>
     ```

     Keeping the `<a name="filter-operator-property-value-of">` anchor
     in line with the existing `filter-operator-=`, `filter-operator-is-a`,
     `filter-operator-descendent-of`, … pattern preserves stable inbound
     fragment links to the rendered `codesystem-filter-operator.html`.

3. **Bump the CodeSystem `<meta><lastUpdated>` and date stamps if

### Testing Considerations

- **Re-publish locally:** run the standard FHIR-core publish (`publish.sh`
  / `build.sh` per the briefing's facts) and confirm the
  `codesystem-filter-operator.html` page renders the new row and that
  the value set expansion at `valueset-filter-operator.html` includes
  `property-value-of`.
- **Schema/spec validation:** the QA pass should report zero new
  validation errors. Pay attention to any whitespace / XML schema warnings
  introduced by the appended block.
- **Server semantics (out-of-scope for this ticket but worth flagging
  to implementers):** terminology servers implementing `$expand` will
  need to interpret the new operator. The definition specifies the
  cross-system clipping rule when `filter.value` is a value set — this
  is the principal edge case to test in implementations:
  - `filter.value` is a comma-separated list of codes from the bound
    system.
  - `filter.value` is a value set canonical whose members are entirely
    within the bound system.
  - `filter.value` is a value set canonical whose members span multiple
    systems (must silently ignore concepts outside the bound system).
  - `filter.value` references a missing or empty value set (server
    behaviour follows existing `$expand` error contract; not changed by
    this ticket).
- **Examples:** consider adding (in a follow-up, since this ticket is
  TEXTONLY) a small ValueSet example exercising `property-value-of`
  against a fixed code system, alongside the existing
  `valueset-example-filter.xml` in `source/valueset/`.
