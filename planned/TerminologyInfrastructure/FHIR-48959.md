## Resolution Summary

The resolution (Persuasive with Modification) directs three concrete narrative
edits to the ValueSet specification page:

1. **Section 4.9.6.5 — Referencing CodeSystem Fragments.** Remove the
   existing single sentence "The CodeSystem referenced in a `ValueSet.compose`
   may reference a CodeSystem fragment." and replace it with two new
   paragraphs:
   - "A CodeSystem fragment MAY be used to generate an expansion in the
     absence of the complete code system."
   - "A CodeSystem fragment cannot be referenced directly. ValueSet
     definitions reference fragments in the same way the CodeSystem is
     referenced. For fragments represented as FHIR CodeSystem resources,
     `CodeSystem.content = fragment` indicates the CodeSystem resource
     represents a fragment."
2. **Section 4.9.10 — Expansion process for `compose.include`.** Add a
   fourth bullet to the `Otherwise: For each compose.include` ordered
   list: "If the server cannot determine that the expansion based on the
   fragments is complete, the server SHALL include the output parameter
   in `ValueSet.expansion.parameter`. The parameter value SHOULD be the
   CodeSystem fragment uri. The parameter name SHOULD be
   `used-fragment`." The reporter's 2026-04-08 comment notes that all
   fragments of a CodeSystem share the same canonical URL, so the
   parameter value really has to identify the resource (e.g.,
   `CodeSystem.url` of the specific fragment instance), not just the
   shared canonical — this nuance must be folded into the wording (see
   Open Questions).
3. **Section 4.9.10 — Expansion Parameters table.** Add a new row to the
   third table (the response-only "used-*" parameters table):
   - Name: `used-fragment`
   - Type: `uri`
   - Description: "The canonical URL (system | {version} if available)
     of a code system that was used based on a fragment during the
     expansion of the value set. This parameter SHALL be provided if the
     server cannot guarantee that the use of this and other fragments
     resulted in a complete expansion. If the server can guarantee that
     the expansion is complete, this parameter SHALL NOT be provided
     even if fragments were used."

The original ticket also asked the WG to fix the "Request for Feedback"
callout that ends with "Ongoing feedback is welcome at [link to be
provided]"; the resolution text does not address that side request, so
it is captured as an Open Question rather than a planned change.

## Feature Proposal

### Problem Statement

The FHIR Core ValueSet narrative (currently `source/valueset/valueset-notes.xml`,
rendered as `valueset.html`) underspecifies how `ValueSet.compose` interacts
with CodeSystem fragments:

- Section 4.9.6.5 ("Referencing CodeSystem Fragments") only states that
  the CodeSystem referenced in `ValueSet.compose` "may reference a
  CodeSystem fragment", but does not explain the consequences for
  expansion or the fact that fragments are referenced via the same
  canonical as the parent CodeSystem (and therefore cannot be referenced
  directly by URL).
- Section 4.9.10 ("Value Set Expansion"), which describes the conceptual
  expansion algorithm for each `compose.include`, makes no mention of
  fragments at all. As Elliot Silver notes in the ticket description,
  this means an expansion produced from one set of fragments can differ
  from an expansion produced from a different set of fragments without
  the difference being recorded in `ValueSet.expansion.parameter`. There
  is no defined parameter to surface "this expansion was produced from a
  fragment and might therefore be incomplete".
- The existing third table in 4.9.10 lists `used-system`,
  `used-supplement`, and `used-valueset`, but lacks a `used-fragment`
  peer.

Zulip discussion ([terminology] "Fragments and closure" #486556226,
"Adding codes to fragment CS" #532938974, and "Tracker item FHIR-48959"
#582882594) and the 2025-09-17 WGM minutes captured in the ticket
comments confirm that fragments are widely understood to be
underspecified, and that the WG wants the spec to (a) acknowledge that
fragment-based expansions may be incomplete and (b) give servers a
standard parameter to flag that condition.

### Proposed Change

All edits are to a single file — `source/valueset/valueset-notes.xml`
(rendered as the `4.9 ValueSet` page on the published spec). The
resolution dictates the prose to use almost verbatim; the implementation
should preserve the existing publisher conventions (`<a name="…">`
anchors, `<code>` for FHIRPath/element fragments, `[%dragons-start%] /
[%dragons-end%]` markers for "warning" passages, `<table class="grid">`
for parameter tables).

**Change A — Section 4.9.6.5 "Referencing CodeSystem Fragments" (lines 212–222).**
Keep the `<a name="fragments"></a>` anchor, the `<h3>` heading, the
existing first paragraph about the VSD Partition, and the existing
SNOMED CT explanatory paragraph. Replace the final paragraph
(currently line 220–222: "The CodeSystem referenced in a
`<code>ValueSet.compose</code>` may reference a CodeSystem fragment.")
with two new paragraphs:

```xml
<p>
A CodeSystem fragment <strong>MAY</strong> be used to generate an
expansion in the absence of the complete code system.
</p>
<p>
A CodeSystem fragment cannot be referenced directly. ValueSet
definitions reference fragments in the same way the
<a href="codesystem.html">CodeSystem</a> is referenced. For fragments
represented as FHIR CodeSystem resources,
<code>CodeSystem.content = fragment</code> indicates the CodeSystem
resource represents a fragment.
</p>
```

(The case of "MAY" in the resolution should be carried through as an
HTML conformance keyword consistent with surrounding usage on the page.
The sample above uses `<strong>` to match how SHALL/SHOULD/MAY are
emphasised elsewhere in `source/valueset/valueset-notes.xml`; an editor
should confirm by inspection of the existing page rendering and pick the
matching style.)

**Change B — Section 4.9.10 "Value Set Expansion", `compose.include` algorithm (lines 500–510).**
Add a fourth `<li>` to the `<ol>` inside the
`<li>Otherwise: For each <i>compose.include</i>:</li>` outer item,
following the existing items 1 ("If there is a system…"), 2 ("For each
`valueSet`…"), and 3 ("Add the intersection…"). Proposed wording (an
elaboration of the resolution that incorporates the reporter's
2026-04-08 comment about identifying the *fragment instance* rather than
the shared canonical URL):

```xml
<li>If the expansion was produced from one or more <a href="codesystem.html#fragments">CodeSystem fragments</a>
and the server cannot determine that the union of those fragments is
complete with respect to the referenced code system, the server
<strong>SHALL</strong> record this in
<code>ValueSet.expansion.parameter</code> using a parameter with
<code>name = "used-fragment"</code> and <code>value</code> set to a uri
that identifies the specific fragment that contributed to the
expansion (typically the <code>CodeSystem.url</code> of the fragment
instance, since all fragments of a code system share the same
canonical url). One <code>used-fragment</code> parameter
<strong>SHOULD</strong> be included for each fragment used.
If the server can guarantee that the expansion is complete, the
<code>used-fragment</code> parameter <strong>SHALL NOT</strong> be
included, even if fragments were used to produce it.</li>
```

**Change C — Section 4.9.10 Expansion Parameters table 3 (lines 618–622).**
Add a new `<tr>` row to the third `<table class="grid">` (the
response-only "used-*" table currently containing `used-system`,
`used-supplement`, `used-valueset`):

```xml
<tr><td><code>used-fragment</code></td><td><a href="datatypes.html#uri">uri</a></td><td>A uri that identifies a CodeSystem fragment that was used during the expansion of the value set. This parameter <strong>SHALL</strong> be provided if the server cannot guarantee that the use of this and any other fragments resulted in a complete expansion of the referenced code system. If the server can guarantee that the expansion is complete, this parameter <strong>SHALL NOT</strong> be provided even if fragments were used. Because all fragments of a code system share the same canonical url, the value of <code>used-fragment</code> typically identifies the specific fragment instance (e.g. via the fragment's <code>CodeSystem.url</code>) rather than the shared canonical.</td></tr>
```

**Change D — Dragon-text update (lines 522–528).**
The resolution explicitly asks to "[add fragments] to the dragon text".
The existing `[%dragons-start%]` block at lines 522–528 already warns
about CodeSystem supplements being subject to ongoing experimentation;
extend it with a parallel sentence about CodeSystem fragments, e.g.:

```xml
<p class="warning">
The impact of <a href="codesystem.html#fragments">CodeSystem fragments</a>
on <a href="valueset-operation-expand.html">value set expansion</a> -
including how partial-fragment expansions are signalled via the
<code>used-fragment</code> output parameter - is also subject to
ongoing experimentation and implementation testing, and further
clarification and additional rules might be proposed in future
versions of this specification.
</p>
```

This sentence sits inside the existing dragons block (no need to open a
new one).

### Design Rationale

- **Mirrors the existing `used-system` / `used-supplement` / `used-valueset`
  pattern.** The third 4.9.10 table is already the "things the expansion
  consumed and is reporting back to you" table; `used-fragment` belongs
  there as a peer with the same `uri` type and the same
  response-parameter semantics.
- **`SHALL` (not SHOULD) when completeness can't be guaranteed.** This is
  what the resolution text specifies and matches the analogous
  `Applications generating expansions SHALL use this parameter if the
  CodeSystem.content value is "fragment"` sentence already in table 2
  (line 608). Consistent strength of conformance language across the two
  fragment-related parameters reduces confusion.
- **Identify the fragment instance, not the canonical.** The reporter's
  2026-04-08 comment correctly observes that the canonical URL is shared
  across fragments and the complete code system, so it cannot
  distinguish which fragment was used. The resolution text says "SHOULD
  be the CodeSystem fragment uri", which aligns with this if interpreted
  as the fragment's `CodeSystem.url`. The proposed wording above makes
  that explicit so implementers don't accidentally emit the shared
  canonical and defeat the purpose of the parameter.
- **Removing the under-specified sentence and replacing it with two MAY
  / cannot-be-referenced-directly statements** brings 4.9.6.5 into line
  with how `CodeSystem.content = fragment` actually works (a fragment is
  modelled as a CodeSystem resource with `content = fragment`, not as a
  separately-referenceable URL). This was explicitly agreed at the
  2025-09-17 WGM (per ticket comments).
- **Dragon text** signals to implementers that fragment-aware expansion
  is still settling, paralleling the existing supplements caveat.
- **Single-file scope** keeps the change recipe simple: this is a
  prose-only change to one HTML authoring file under `source/`, with no
  StructureDefinition, schema, or example-instance changes. It cleanly
  matches the FhirCore "Update narrative / prose" recipe in the saved
  briefing.

### Breaking Changes

No breaking changes identified. All edits are clarifications of existing
behaviour:
- 4.9.6.5 paragraph swap removes one under-specified sentence and adds
  guidance about how `ValueSet.compose` references fragments. Existing
  ValueSet instances remain valid.
- The new `used-fragment` expansion output parameter is **additive** —
  servers that do not produce fragment-based expansions are unaffected,
  and servers that already produce fragment-based expansions only need
  to start emitting the parameter when they cannot guarantee
  completeness. Clients are not required to consume it.
- No element cardinality, type, binding, or invariant changes.
- No search-parameter changes.

### Related Specifications

- **`codesystem.html#fragments`** — The narrative anchor for CodeSystem
  fragments and `CodeSystem.content = fragment`. Already linked from
  4.9.6.5 ("FHIR CodeSystem Fragments"); the new prose should keep that
  link bidirectional. Consider whether `codesystem.html#fragments`
  should reciprocally point to the new `used-fragment` parameter (a
  small follow-up edit, optional, not required by the resolution).
- **`valueset-operation-expand.html` ($expand operation definition)** —
  The third table of 4.9.10 documents *response* parameters that
  `$expand` may emit. Implementers of `$expand` will look here for the
  authoritative list. The `used-fragment` row is the authoritative
  definition; the OperationDefinition itself does not enumerate
  response parameters of this kind, so no edit is needed to
  `source/valueset/operationdefinition-ValueSet-expand.xml`.
- **`valueset.html#parameters` / `#expansion` / `#fragments` anchors** —
  Used by other spec pages; preserved by this change.

### Terminology Impact

No terminology impact. The change does not add or modify any
CodeSystem, ValueSet, or ConceptMap; it only documents how an existing
mechanism (`CodeSystem.content = fragment`) interacts with ValueSet
expansion and adds one new expansion *output parameter name* (a string
literal `used-fragment`, not a coded value).

The `valueset-expansion-processing-rule` CodeSystem at
`source/valueset/codesystem-expansion-processing-rule.xml` was inspected
for relevance and is *not* affected — it enumerates processing-rule
codes, not response parameter names.

### Open Questions

- **Identifier of the fragment in `used-fragment`'s value.** The
  resolution text says "SHOULD be the CodeSystem fragment uri", but
  Elliot Silver's 2026-04-08 comment correctly observes that the
  *canonical URL* is shared across all fragments of a code system, so
  it cannot distinguish which fragment was used. This plan assumes the
  intended value is the fragment instance's `CodeSystem.url` (i.e., the
  resource URL of the specific CodeSystem-with-`content = fragment`
  resource that was consumed) and the plan's wording for both Change B
  and Change C reflects that assumption. **Confirm with the WG before
  merging.** If the WG instead wants the value to be a versioned
  canonical (e.g., `system|version`), the wording in steps 2 and 4 must
  be adjusted accordingly.
- **"Request for Feedback" callout link.** The original ticket
  description includes a second, separate request: replace the broken
  "Ongoing feedback is welcome at [link to be provided]" placeholder in
  the Request-for-Feedback callout. The resolution text does **not**
  address this. Either (a) treat it as out of scope for FHIR-48959 and
  open a follow-up ticket, or (b) confirm with the WG that they want it
  rolled into this change and which URL to use. This plan does **not**
  include an edit for that callout.
- **Reciprocal note on `codesystem.html#fragments`.** Should the
  CodeSystem narrative also gain a "see also `valueset.html#parameters`
  for `used-fragment`" pointer? Optional, not required by the
  resolution; flag for the reviewer.
- **Backwards-compat with existing fragment expansions.** The Zulip
  thread "[terminology] Fragments and closure" (#486556226) hints at
  closure-mode interactions. This plan does not touch closure
  semantics; if the WG wants closure mentioned alongside `used-fragment`
  in the dragon text, expand Change D accordingly.
