## Resolution Summary

**Resolution:** Persuasive.

**Resolution Description (verbatim):** "Change the datatype for `match.originMap` output parameter to canonical."

The output parameter `match.originMap` of the `ConceptMap/$translate`
operation currently has `type = uri`. The agreed change is to retype it
to `canonical`. The element's documentation already describes it as a
"canonical reference to the concept map from which this mapping comes
from", so no narrative change to the parameter's `documentation` is
required. Comment from JIRAUSER19808 (2025-12-05) clarifies that
although the resolution text says "input parameter", `match.originMap`
is in fact an *output* parameter — this plan treats it as the output
part it is (`use = out`, inside the outer `match` parameter). Comment
from carmela_couderc (2026-01-26) notes that other `uri`-typed
parameters that should likely become `canonical` are deliberately
deferred to separate tickets and are out of scope here.

## Feature Proposal

### Problem Statement

`ConceptMap/$translate` returns one or more `match` parameters; each
match optionally carries an `originMap` part identifying which
ConceptMap produced the mapping. Today that part is typed `uri`, which
is too weak for the intent: when a server holds multiple resource
instances of the same canonical ConceptMap at different versions, a
plain URI in `originMap` cannot disambiguate which version produced a
given match (clients would have to reissue `$translate` per
`conceptMapVersion` to find out). FHIR's `canonical` datatype was
designed precisely for "URL plus optional `|version`" references to
canonical resources, and the parameter's own documentation already
calls it a "canonical reference", so the type/documentation are
inconsistent.

### Proposed Change

In the `ConceptMap/$translate` OperationDefinition, retype the
`match.originMap` output parameter from `uri` to `canonical`. Concretely,
in `source/conceptmap/operationdefinition-ConceptMap-translate.xml`:

- Change the inner `<part>` whose `<name value="originMap"/>` /
  `<use value="out"/>` / `<min value="0"/>` / `<max value="1"/>` from
  `<type value="uri"/>` to `<type value="canonical"/>`.
- Leave `name`, `use`, cardinality (`0..1`), and the existing
  `<documentation value="The canonical reference to the concept map
  from which this mapping comes from"/>` unchanged.
- No `targetProfile` is required (canonical may but need not constrain
  to `ConceptMap`); the resolution does not call for one and adding it
  would be out-of-scope tightening. (See Open Questions.)

In the worked example
`source/conceptmap/$translate-response.txt`, update the two illustrative
responses so the `originMap` part demonstrates the new canonical type:

- Rename the (pre-existing, mis-spelled) `"originalMap"` parts to
  `"originMap"` so the example actually matches the OperationDefinition.
- Change the value from `"valueString": "http://hl7.org/fhir/ConceptMap/…"`
  to `"valueCanonical": "http://hl7.org/fhir/ConceptMap/…"`.
- For the multi-match example (which exists specifically to illustrate
  matches originating from different ConceptMaps), make at least one
  `valueCanonical` carry a `|<version>` suffix
  (e.g., `"http://hl7.org/fhir/ConceptMap/cm-composition-status-v3|1.0.0"`)
  so the example demonstrates the disambiguation use case the ticket
  cites.
- Rename the second match's `"source"` part (which appears to be a
  typo for `originMap` in the existing file) to `"originMap"` with
  `valueCanonical` for consistency.

### Design Rationale

- **Aligns type with intent.** The parameter documentation already
  reads "canonical reference to the concept map"; FHIR's `canonical`
  datatype is the correct primitive for "URL + optional `|version`"
  references to canonical resources such as ConceptMap.
- **Solves the stated use case.** A `canonical` value can carry the
  exact ConceptMap version (`|<businessVersion>`), letting clients
  distinguish matches returned from multiple instances of the same
  canonical ConceptMap without re-issuing `$translate` once per
  `conceptMapVersion`.
- **Minimal, surgical edit.** Only the `<type>` element changes; no
  cardinality, name, or documentation changes are needed. This matches
  the "Persuasive" disposition (apply as proposed) rather than
  "Persuasive with Modification".
- **Scope discipline.** carmela_couderc's comment explicitly defers
  other `uri`→`canonical` retypings to separate tickets; this plan does
  not opportunistically touch other parameters.

### Breaking Changes

This is a backward-incompatible parameter type change for clients of
the `$translate` operation:

- **Output type narrowed.** `uri` → `canonical`. `canonical` is a
  syntactic subset of `uri` (it must be a canonical URL, optionally
  with `|version` and `#fragment`), so a server that previously
  returned an arbitrary URI in `originMap` would no longer be
  conformant. In practice, any server that already followed the
  documentation ("canonical reference to the concept map") was
  effectively returning a canonical, so wire-compatibility is high,
  but the conformance contract has tightened — this should be called
  out as a technical correction in change notes for the next
  publication.
- **Client parsing.** Clients receiving the parameter as JSON will see
  the value carried under `valueCanonical` instead of `valueUri`;
  clients that hard-coded `valueUri` lookup will need to update.
