## Resolution Summary

The original request proposed creating a **new** extension (`CodeSystem-property-allowed-value`) to constrain allowed values for `string`-typed CodeSystem properties. After extensive discussion (May 2024 WGM, March 2026 WGM, and Zulip threads), the work group resolved **Not Persuasive with Modification**: rather than creating a new extension, the **existing** `codesystem-property-valueset` extension (created via [FHIR-35978](https://jira.hl7.org/browse/FHIR-35978)) should be revised to also cover `string`-typed properties, in addition to the `code` and `Coding` types it already supports.

The resolution specifies exact text changes to the extension's `definition` and `comment` fields.

---

## Feature Proposal

### Problem Statement

CodeSystem properties typed as `code` constrain their values to codes defined within the code system itself (or a referenced code system). However, when a property is typed as `string`, there is no mechanism to constrain which string values are permitted. This forces code system authors into an uncomfortable choice:

1. Use `code` type, which requires defining every allowed value as an actual concept in the code system (polluting the concept space with non-instance codes).
2. Use `string` type, but accept that any arbitrary string is valid with no validation possible.

The `codesystem-property-valueset` extension already allows binding a ValueSet to `code` and `Coding`-typed properties, but it explicitly excludes `string`-typed properties in its current definition.

### Proposed Change

Modify the existing `codesystem-property-valueset` extension definition to extend its applicability to `string`-typed properties. This is a **text-only change** to the extension's `definition` and `comment` elements — the extension's structure (canonical(ValueSet)), context (CodeSystem.property), cardinality (0..1), and URL remain unchanged.

Specifically, the changes are:

1. **Definition** — Change from:
   > "Where CodeSystem properties are of type code or Coding, this ValueSet defines the permitted set of concepts to be used in CodeSystem.concept.property."

   To:
   > "Where CodeSystem properties are of type code, Coding, **or string**, this ValueSet defines the permitted set of **codes or string values** to be used in CodeSystem.concept.property."

2. **Comment** — Change from:
   > "This could be used during validation to ensure that not only is a specific property of the correct type (based on CodeSystem.property.type) but also where coded from the correct valueset. It could also be used in a user interface to guide code system designers to only pick from the appropriate source."

   To:
   > "This could be used during validation to ensure that not only is a property instance of the correct type (based on CodeSystem.property.type) but the value is from the valueset. It could also be used in a user interface to guide users to only select from the appropriate source."

### Design Rationale

- **Reuse over proliferation**: Rather than creating a new extension with overlapping purpose, extending an existing extension is simpler for implementers and avoids confusion about which extension to use.
- **ValueSet-based constraint**: Leveraging ValueSets (with implicit code systems or VCL) is the standard FHIR mechanism for constraining coded/string values, maintaining consistency with the broader terminology framework.
- **Non-breaking**: The extension URL, structure, and cardinality are unchanged. The only change broadens the *documented applicability* — existing uses for `code` and `Coding` properties remain fully valid.

### Breaking Changes

**None.** This change is purely additive — it broadens the documented scope of an existing extension without changing its URL, structure, data type, cardinality, or context. All existing usages of the extension on `code` and `Coding` properties remain valid.

Validators that currently check the extension against the definition text may need to update their logic to also validate `string`-typed property values against the bound ValueSet, but this is a **new capability**, not a breaking change.

### Terminology Impact

- **Validation tools** (e.g., the FHIR Validator) should be updated to validate `string`-typed property values against the bound ValueSet when the `codesystem-property-valueset` extension is present. Currently validation is only performed for `code` and `Coding` types.
- **Implicit CodeSystem for strings**: A related Zulip thread (Michael Lawley, 2026-03-17) proposes defining an implicit CodeSystem URI for all strings (e.g., `http://hl7.org/fhir/CodeSystem/string`) to simplify creating ValueSets that constrain string properties without requiring an explicit CodeSystem. This is a **separate, future enhancement** not part of this ticket's scope.
- **UTG / THO**: Code systems in THO that use `string`-typed properties (currently unconstrained) could adopt this extension to add validation. No immediate changes required.
