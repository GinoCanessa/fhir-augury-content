## Feature Proposal

### Problem Statement

The R6 ballot text in `terminologies.html` says only that additional
bindings "do not replace the main binding (the main binding always
applies)" but is silent on the **set-relationship** between an
additional binding's value set and the main binding's value set. As a
result IG authors are receiving conflicting guidance — some advisors
insist additional bindings must always be subsets/refinements of the
main binding, others say they need not be — which leads to incompatible
modeling assumptions across IGs, inconsistent validator behaviour, and
implementers inferring intent from informal advice. The reporter (Gay
Dolin) cites the HL7 FHIR IG Creation Zulip channel as a venue where
this ambiguity has surfaced, and links companion tickets
[FHIR-54128](https://jira.hl7.org/browse/FHIR-54128) and
[FHIR-54590](https://jira.hl7.org/browse/FHIR-54590) which raise the
same underlying concern from different angles.

### Proposed Change

This is a **documentation-only** change confined to the FHIR Core
specification authoring tree (`HL7/fhir`, authoring root `source/`).
There are no resource StructureDefinition, ValueSet, CodeSystem, or
search-parameter changes.

**Change B — Insert two new paragraphs after the existing
"non-overlapping value sets" paragraph.**

- File: `source/terminologies.html`
- Location: immediately **after** the closing `</p>` on **line 249**
  (which ends the paragraph beginning "It's possible for there to be
  multiple applicable required bindings to non-overlapping value
  sets…") and **before** the `<a name="references"></a>` anchor on line
  252.
- New content (HTML):

```html
<p>
The conceptual spaces of all bindings on an element must be overlapping.
For example, it would not be appropriate to have a binding for concepts
representing color and a binding for concepts representing shape on the
same element unless handled through slices.
</p>

<p>
Note that if the element data type is a <code>code</code> or
<code>Coding</code>, the value must satisfy all bindings. If the
conformance expectations of the bindings are disjoint, it will not be
possible to satisfy the bindings and no data will be valid.
</p>
```

The first of these two paragraphs is, per the resolution, the same
content that resolves
[FHIR-54590](https://jira.hl7.org/browse/FHIR-54590) (slicing /
validation expectations for additional bindings). It must be inserted
once and **only** once via this ticket; FHIR-54590's plan must not
re-author it.

**Change C — Add a new "Examples for Additional bindings" section to
the binding examples page.**

- File: `source/terminologies-binding-examples.html`
- Location: append a new `<h3>` section after the existing
  `<h3>Examples for Example bindings</h3>` block (which currently ends
  the document at the `<ul>` on lines ~338–343, just before
  `[%file newfooter%]`).
- Content: at least one worked example for each of the cases the
  resolution clarifies. Recommended minimum coverage:
  1. An additional binding whose value set **is a subset** of the main
     binding (e.g., a `purpose = preferred` extra binding narrowing a
     `required` main binding).
  2. An additional binding whose value set **overlaps but is not a
     subset** of the main binding (e.g., a jurisdictional / realm
     overlay using `usage`).
  3. An additional binding whose value set **extends beyond** the main
     binding (e.g., a `purpose = candidate` or `purpose = ui` value set
     that adds context-specific concepts for non-conformance use).
  4. A short narrative example illustrating that on a `code` or
     `Coding` element, a value must satisfy *all* bindings, contrasted
     with the `CodeableConcept` / `CodeableReference` case where
     multiple Codings may be used to satisfy disjoint required
     bindings.
  Each example should follow the existing page's pattern: a short prose
  setup paragraph followed by a `<pre class="json">` block showing the
  ElementDefinition (or fragment) with `binding` and
  `binding.additional` populated. The narrative for each example must
  state explicitly which of the three relationships
  (subset/overlap/extend) it illustrates and reference back to
  `terminologies.html#binding`.

### Design Rationale

- The wording "are not required to be subsets … may overlap with,
  extend beyond, or differ from it" was the working-group's explicit
  choice (Carmela Couderc, 2026-01-28) and resolves the ambiguity in
  favour of the **non-subset-permitted** interpretation. This is
  consistent with the existing `binding.additional.purpose` codes
  (`preferred`, `extensible`, `required`, `candidate`, `current`,
  `preferred`, `ui`, `starter`, `component`, …) which already imply
  contexts (e.g., `ui`, `candidate`) where the additional value set is
  legitimately *broader* than the main binding rather than a refinement
  of it.
- The companion "conceptual spaces must be overlapping" sentence is
  required to prevent the obvious abuse (color × shape on the same
  element) and is the same sentence agreed for FHIR-54590, so adding it
  here closes both tickets in one edit.
- The "code or Coding" caveat preserves the existing behaviour
  documented in the prior paragraph (multiple required bindings to
  non-overlapping value sets are only satisfiable on `CodeableConcept`
  / `CodeableReference`) and makes the disjoint-conformance failure
  mode explicit so validators have a normative hook.
- Worked examples are placed on the binding-examples page rather than
  inline in `terminologies.html` to keep the normative page concise;
  this matches the pattern already used for the four binding strengths
  (Required / Extensible / Preferred / Example), each of which is
  *defined* on `terminologies.html` and *exemplified* on
  `terminologies-binding-examples.html`.

### Breaking Changes

No breaking changes identified. The change is a **clarification** of
existing under-specified behaviour:

- The rewritten paragraph (Change A) preserves the existing rule that
  the main binding always applies and merely makes the *non-subset*
  interpretation explicit. Implementations that already assumed
  additional bindings could be non-subsets are unaffected;
  implementations that assumed strict subset-only need to relax that
  check, but no instance data becomes invalid as a result.
- The "conceptual spaces must be overlapping" sentence (Change B,
  paragraph 1) is a soft constraint expressed as guidance ("would not
  be appropriate"), not a new invariant — no validator-breaking
  change.
- The "code or Coding" sentence (Change B, paragraph 2) restates an
  existing logical consequence of the prior paragraph; it does not
  introduce a new conformance rule, only documents one.
- No element cardinality, type, binding strength, or search-parameter
  changes occur.
