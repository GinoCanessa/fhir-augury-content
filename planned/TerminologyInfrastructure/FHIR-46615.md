## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"Will remove the reference to
compose-createdBy extension in the Publishable profile."*

The change is narrow and mechanical: delete the slice on
`ValueSet.compose.extension` that names the
`http://hl7.org/fhir/StructureDefinition/valueset-compose-createdBy`
extension from the `PublishableValueSet` profile's differential. This
follows from the resolution of [FHIR-40703](https://jira.hl7.org/browse/FHIR-40703)
("Clean up definition of extension compose-createdBy (or deprecate it)"),
which deprecated the extension itself; once deprecated, the Publishable
profile should no longer carry an explicit slice that calls for it.

## Feature Proposal

### Problem Statement

The `PublishableValueSet` profile previously declared an explicit
extension slice on `ValueSet.compose` named `composeCreatedBy` that
required (or at least named) the now-deprecated
`valueset-compose-createdBy` extension. Carrying a slice that points at
a deprecated extension creates two problems:

1. It implies that publishers of a "Publishable" ValueSet are still
   expected to consider populating `compose-createdBy`, contradicting
   FHIR-40703's deprecation.
2. Tooling that drives off the profile's differential (validators,
   IG-publisher snapshot generation, conformance reporters) will keep
   surfacing the extension as an in-scope element of the profile.

### Proposed Change

Remove the entire `<element id="ValueSet.compose.extension:composeCreatedBy">`
block from the `PublishableValueSet` profile's differential. The block
to be removed is exactly:

```xml
<element id="ValueSet.compose.extension:composeCreatedBy">
  <path value="ValueSet.compose.extension"/>
  <sliceName value="composeCreatedBy"/>
  <min value="0"/>
  <max value="1"/>
  <type>
    <code value="Extension"/>
    <profile value="http://hl7.org/fhir/StructureDefinition/valueset-compose-createdBy"/>
  </type>
</element>
```

(Verbatim from the pre-migration version of
`source/valueset/structuredefinition-profile-publishablevalueset.xml`
at commit `3958dd1e7c~1`.)

No other element in the profile is touched. Specifically:

- The sibling slice `composeCreationDate` (referencing
  `valueset-compose-creationDate`) is **not** in scope for this ticket
  and must be left intact.
- The differential's outer `ValueSet.compose` element entry is **not**
  removed — only the named slice on its extension.
- No changes are made to the `valueset-compose-createdBy` extension
  itself; that artefact is governed by FHIR-40703.

### Design Rationale

- The resolution is direct and unambiguous, with no alternative options
  raised in the ticket comments (none exist on the issue).
- Removing the slice is the minimum change that breaks the profile's
  formal dependency on the deprecated extension while preserving every
  other constraint the profile imposes on publishable value sets.
- Leaving the extension definition in place (in `HL7/fhir-extensions`)
  is correct: deprecation is signalled at the extension level by
  FHIR-40703, and the profile simply stops naming it.

### Breaking Changes

No breaking changes identified for consumers of FHIR Core.

- The `valueset-compose-createdBy` extension itself is not removed by
  this ticket — instances that already populate it remain valid against
  the base `ValueSet` resource (extensions on `compose` are unrestricted
  there).
- Profiles that previously claimed conformance to `PublishableValueSet`
  with `composeCreatedBy` populated remain conformant after the slice
  is removed (the slice was `0..1`, so removing it relaxes, not
  tightens, the constraint).
- Downstream IGs that derive from `PublishableValueSet` should be
  unaffected; if any added their own slice on `composeCreatedBy`, it
  remains valid in the derived profile.
