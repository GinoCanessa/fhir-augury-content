## Resolution Summary

Resolution: **Persuasive with Modification**.

The applied resolution requires:

1. Add a new element **`TerminologyCapabilities.codeSystem.version.content`**:
   - Cardinality: `1..1`
   - Type: `code`
   - Binding strength: `required`, bound to a new ValueSet (described below)
   - `isSummary = true`
   - Definition: *"The extent the content of the code system version (the concepts and codes it defines) as supported by the server."*
2. Add a **new ValueSet** to the core spec drawn from the existing `CodeSystem Content Mode` code system, restricted to the codes `fragment`, `complete`, `not-present`:
   - `name` = `TerminologyCapabilitiesContentMode`
   - `title` = `Terminology Capabilities Content Mode`
   - `url` = `http://hl7.org/fhir/ValueSet/terminologycapabilities-content-mode`

The resolution explicitly notes that `content = supplement` will be constant across versions, justifying keeping the existing `TerminologyCapabilities.codeSystem.content` (which retains the broader `codesystem-content-mode` binding including `supplement` and `example`) while adding the per-version element bound to the more restricted set.

## Feature Proposal

### Problem Statement

The `TerminologyCapabilities` resource is used by terminology servers to advertise which CodeSystems they host and to what extent. Today the server can declare the *content mode* (`fragment`, `complete`, `not-present`, `example`, `supplement`) only at `TerminologyCapabilities.codeSystem.content` — i.e. once per CodeSystem URL. In practice, however, the content mode can vary per version: a server may host the *complete* content of CodeSystem X version 2024 but only a *fragment* of version 2020, or hold a snapshot of one published version while merely declaring the existence of another. The current model cannot express this, leading to inaccurate `TerminologyCapabilities` statements.

### Proposed Change

**Element addition** — add `TerminologyCapabilities.codeSystem.version.content` as a new sibling of the other `TerminologyCapabilities.codeSystem.version.*` elements (placed alphabetically between `compositional` and `filter`, i.e. consistent with the ordering used for similar `code`-typed metadata elements):

- `id` / `path`: `TerminologyCapabilities.codeSystem.version.content`
- `short`: `not-present | fragment | complete`
- `definition`: `The extent the content of the code system version (the concepts and codes it defines) as supported by the server.`
- `min`: `1`
- `max`: `1`
- `type`: `code`
- `isSummary`: `true`
- `binding`:
  - `bindingName` extension: `TerminologyCapabilitiesContentMode`
  - `strength`: `required`
  - `description`: `The extent of the content of a single version of a code system supported by the server.`
  - `valueSet`: `http://hl7.org/fhir/ValueSet/terminologycapabilities-content-mode|6.0.0`

**ValueSet addition** — create `source/codesystem/valueset-terminologycapabilities-content-mode.xml`:

- `id`: `terminologycapabilities-content-mode`
- `url`: `http://hl7.org/fhir/ValueSet/terminologycapabilities-content-mode`
- `name`: `TerminologyCapabilitiesContentMode`
- `title`: `Terminology Capabilities Content Mode`
- `status`: `active`
- `experimental`: `false`
- `immutable`: `true`
- WG extension: `vocab`
- `compose.include`:
  - `system` = `http://hl7.org/fhir/codesystem-content-mode`
  - `concept`: `not-present`, `fragment`, `complete` (omitting `example` and `supplement`)

**Existing `TerminologyCapabilities.codeSystem.content`** — leave the binding (`codesystem-content-mode`) and definition unchanged. Update the short label/definition narrative if helpful to clarify it conveys the *aggregated* / cross-version content state (especially `supplement`), and that per-version detail belongs at `.version.content`.

**Example instances** — update `terminologycapabilities-example.xml` to include a `version.content` value on at least one `codeSystem.version` entry to demonstrate the new element.

### Design Rationale

- A new required element at `version` is the most explicit way to capture per-version content state without overloading the existing summary at the `codeSystem` level.
- A *new* ValueSet (rather than reusing `codesystem-content-mode`) is necessary because the resolution restricts per-version content to `fragment | complete | not-present`. `supplement` is, by definition, per-CodeSystem (a supplement applies to the whole CodeSystem) and `example` is not a meaningful per-version statement of completeness; both are intentionally excluded. Keeping the broader binding on the parent `.content` retains backward expressivity.
- `isSummary = true` mirrors `TerminologyCapabilities.codeSystem.content` so that summarised `TerminologyCapabilities` statements continue to convey the most operationally meaningful field.
- `1..1` is consistent with the resolution and with the precedent set by the existing `.content` element. A required statement avoids ambiguity about whether a missing value implies "complete" or "unknown".
- The valueset URL pattern follows the convention used for other resource-specific bindings drawn from a shared code system (e.g. `http://hl7.org/fhir/ValueSet/<resource>-<aspect>`).

- **Change Scope:** **Moderate**. Two structural artefacts (one new file), one example update, optional narrative.

### Breaking Changes

**Yes — this is a breaking change.** Adding a `1..1` (required) element to `TerminologyCapabilities.codeSystem.version` means existing `TerminologyCapabilities` instances that include any `codeSystem.version` entries become invalid until they are amended to populate `version.content`. Servers and clients that produce `TerminologyCapabilities` resources will need to update their generators.

Mitigation options the WG should weigh (none currently selected by the resolution; flagged in *Open Questions*):

- Down-grade cardinality to `0..1` to preserve backward compatibility (deviates from the literal resolution text).
- Document a recommended default (`fragment` is the safest assumption when a server publishes versions but has not declared content state).

No other elements are removed, retyped, or have their bindings altered. The existing `TerminologyCapabilities.codeSystem.content` binding (`codesystem-content-mode|6.0.0`, required) is preserved unchanged.

## Implementation Plan

### Prerequisites

- None. The underlying `codesystem-content-mode` CodeSystem already defines all three required codes (`not-present`, `fragment`, `complete`).
- Confirm with WG editors the desired placement of the new element within the `version` sub-tree (alphabetical vs. semantic grouping). The plan below assumes placement immediately before `version.filter`.

### Step-by-Step Tasks

#### HL7/fhir

1. **Create the new ValueSet file**
   - File: `source/codesystem/valueset-terminologycapabilities-content-mode.xml` (new)
   - Action: Create the file using `valueset-codesystem-content-mode.xml` as the template.
   - Details: Set `id` = `terminologycapabilities-content-mode`; `url` = `http://hl7.org/fhir/ValueSet/terminologycapabilities-content-mode`; `name` = `TerminologyCapabilitiesContentMode`; `title` = `Terminology Capabilities Content Mode`; `version` = `6.0.0`; `status` = `active`; `experimental` = `false`; `immutable` = `true`; `description` = "The extent of the content of a single version of a code system as supported by a terminology server."; WG extension `vocab`; `standards-status` `normative` (mirroring the parent CodeSystem); `compose.include` with `system` = `http://hl7.org/fhir/codesystem-content-mode` and three `concept` entries (`not-present`, `fragment`, `complete` — codes only; displays come from the CodeSystem). Include the standard `identifier` block with a freshly allocated OID per HL7 convention (request from the OID registrar if not pre-allocated).

2. **Add the new element to the TerminologyCapabilities StructureDefinition (differential only)**
   - File: `source/terminologycapabilities/structuredefinition-TerminologyCapabilities.xml`
   - Action: Inside `<differential>`, insert a new `<element>` for `TerminologyCapabilities.codeSystem.version.content`. Place it immediately before the existing `<element id="TerminologyCapabilities.codeSystem.version.filter">` block (current line 719) so it sits with the other version-level metadata elements.
   - Details:
     ```xml
     <element id="TerminologyCapabilities.codeSystem.version.content">
       <path value="TerminologyCapabilities.codeSystem.version.content"/>
       <short value="not-present | fragment | complete"/>
       <definition value="The extent the content of the code system version (the concepts and codes it defines) as supported by the server."/>
       <min value="1"/>
       <max value="1"/>
       <type>
         <code value="code"/>
       </type>
       <isSummary value="true"/>
       <binding>
         <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
           <valueString value="TerminologyCapabilitiesContentMode"/>
         </extension>
         <strength value="required"/>
         <description value="The extent of the content of a single version of a code system supported by the server."/>
         <valueSet value="http://hl7.org/fhir/ValueSet/terminologycapabilities-content-mode|6.0.0"/>
       </binding>
     </element>
     ```
     Do **not** edit `<snapshot>` — the publisher regenerates it from `<differential>`.

3. **Update the example instance to demonstrate the new element**
   - File: `source/terminologycapabilities/terminologycapabilities-example.xml`
   - Action: For each `codeSystem.version` entry already present in the example, add a `<content value="…"/>` child (use `complete` or `fragment` as appropriate to illustrate the per-version variability called out in the ticket). Ensure ordering matches the differential element order.
   - Details: This both validates the new required cardinality and serves as the published example for downstream implementers.

4. **(Optional) Clarifying narrative**
   - File: `source/terminologycapabilities/terminologycapabilities-introduction.xml` *(or `terminologycapabilities-notes.xml`)*
   - Action: Add one or two sentences explaining that `codeSystem.content` describes the aggregated content state across versions (notably whether the resource acts as a supplement) and that `codeSystem.version.content` states the per-version content mode for each enumerated version.
   - Details: Keep wording terse; the resolution did not mandate prose changes.

5. **Bindings/registry hygiene**
   - File: `source/terminologies.html` (and any per-version binding registry the publisher consumes — check via grep for `terminologycapabilities-content-mode` after creating the ValueSet).
   - Action: If the new ValueSet must be listed in a registry table, add the row alongside `codesystem-content-mode` (vocabulary WG, FMM/normative status, `required` binding usage).
   - Details: Only required if the publisher fails to auto-discover the new ValueSet; verify with a local publish.

### Testing Considerations

- **Revalidation:** Re-run validation against `terminologycapabilities-example.xml` and any other `TerminologyCapabilities` examples elsewhere in the spec or shipped IGs reachable by the FHIR validator.
- **Example instances:** Add (or extend the existing) example to show two versions of the same CodeSystem with different `content` values (e.g. v1 `complete`, v2 `fragment`) — this is the canonical motivating case from the ticket.
- **Edge cases:**
  - A `codeSystem.version` entry must now always populate `content`. Confirm publisher-shipped examples and any test fixtures used by reference implementations.
  - Ensure that a `TerminologyCapabilities` resource with `codeSystem.content = supplement` and no `version` entries remains valid (no required `version.content` triggered when `version` is absent).
  - Round-trip JSON ↔ XML serialisation of the new element.
- **Interoperability:** Notify implementers via the publication change log that `version.content` is a new required element and provide migration guidance (defaulting to `fragment` for unknown legacy states is a reasonable suggestion).

### Open Questions

- **Cardinality vs. backwards compatibility.** The resolution states `1..1`, but adding a required element to an element that already has shipped instances is a breaking change. Confirm with the Vocabulary WG whether the intent is strictly `1..1` or whether `0..1` (with documented default) is acceptable to preserve compatibility for existing servers.
- **Element placement.** Should `version.content` sit immediately after `version.compositional` (alphabetical-ish: code/isDefault/supplement/compositional/content/language/filter/property), or grouped semantically with `filter`/`property` near the end of the version block? Editor preference required.
- **Search parameter.** The resolution does not request a search parameter on the new element. Confirm the WG does not want one (matches the absence of an SP on the existing `.content`).
- **Narrative scope.** Does the WG want a clarifying note on `TerminologyCapabilities.codeSystem.content` itself (e.g. "per-version detail belongs at `version.content`; this element conveys aggregated state, including `supplement`")? The plan flags this as optional.
- **OID allocation** for the new ValueSet — must be coordinated with the HL7 OID registrar before the ValueSet file is finalised.
- **Coordination with FHIR-48960** ("Clarify fragment and supplement relationships"). If both tickets land in the same release, the narrative around `supplement` should be reviewed jointly so the per-version exclusion of `supplement` from the new ValueSet aligns with FHIR-48960’s framing.
