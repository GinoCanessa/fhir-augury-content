## Resolution Summary

**Resolution:** Persuasive with Modification.

The applied resolution states:

> All representations of this code system example reference value set and not
> code system. Change the text in all examples from "value set" to "code
> system".

The resolution illustrates the change with TTL, JSON, and XML excerpts of the
`reason-medication-not-given-codes` CodeSystem, all of which currently carry a
`description` element worded as if the resource were a ValueSet
(*"This value set is provided as an example. The value set to instantiate this
attribute …"*). The fix is to reword the offending `description` so it
describes a CodeSystem, in every serialization form.

In the HL7/fhir source repository the JSON and TTL representations are
generated by the bespoke FHIR publisher from the authored XML. The change
therefore only needs to be applied to the single XML source file; the published
JSON/TTL/HTML artefacts will pick up the new wording on the next build.

## Feature Proposal

### Problem Statement

The example `CodeSystem` `http://hl7.org/fhir/reason-medication-not-given`
(file `source/medicationadministration/codesystem-reason-medication-not-given-codes.xml`)
contains a `description` that talks about a *value set*:

> *"This value set is provided as an example. The value set to instantiate this
> attribute should be drawn from a robust terminology code system that consists
> of or contains concepts to support the medication process."*

This description is a verbatim copy of language appropriate to a `ValueSet` and
is misleading on a `CodeSystem` resource. The reporter notes "Clearly not a
description of a CodeSystem". The same wrong wording is propagated by the
publisher into the JSON, TTL, and HTML renderings of the example
(e.g. `build.fhir.org/codesystem-reason-medication-not-given-codes.json.html`).
There is no separate authored JSON or TTL file — they are publisher output —
so a single source-of-truth fix in the XML resolves all representations.

### Proposed Change

Edit the `description` element of the example `CodeSystem`
`reason-medication-not-given-codes` so it accurately describes a
`CodeSystem` rather than a `ValueSet`.

- **File:** `source/medicationadministration/codesystem-reason-medication-not-given-codes.xml`
- **Element:** `CodeSystem.description` (currently a single `<description value="…"/>` line).
- **Current value:**

  > This value set is provided as an example. The value set to instantiate
  > this attribute should be drawn from a robust terminology code system that
  > consists of or contains concepts to support the medication process.

- **New value (replace the two occurrences of "value set" with "code
  system"):**

  > This code system is provided as an example. The code system to instantiate
  > this attribute should be drawn from a robust terminology code system that
  > consists of or contains concepts to support the medication process.

  This is the literal `value-set → code-system` substitution that the
  resolution prescribes ("Change the text in all examples from 'value set' to
  'code system'") and matches the TTL/JSON/XML excerpts shown in the
  resolution description.

No other authored files require editing. The companion ValueSet
`source/medicationadministration/valueset-reason-medication-not-given-codes.xml`
already has a different, ValueSet-appropriate description ("This value set
includes all medication refused, medication not administered, …") and is out
of scope.

### Design Rationale

- The reporter and resolution both target the wording on the
  `CodeSystem` resource — the resource type whose semantics dictate that the
  text refer to a "code system". The resolution explicitly prescribes the
  substitution rather than a deeper rewrite, so a literal swap preserves the
  surrounding sentence structure that the resolution example shows.
- HL7/fhir is built by the bespoke HL7 publisher (per the cached
  briefing); JSON/TTL/HTML examples are regenerated from authored XML at
  publish time. Editing only the XML keeps the change consistent with how the
  repository is maintained and avoids touching generated output.
- The CodeSystem is `experimental="false"` but `status="draft"` and serves as
  an *example* CodeSystem only. Description-only wording fixes carry no
  conformance impact, no terminology binding implications, and no breaking
  changes for implementers.
