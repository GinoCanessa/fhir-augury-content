## Feature Proposal

### Problem Statement

In the current CodeSystem narrative (§4.8.12.1 Concept Status, in `source/codesystem/codesystem-notes.xml`), the relationship between three overlapping concept properties — the `status` code property, the `inactive` boolean property, and the date properties (`effectiveDate`, `deprecationDate`, `retirementDate`) — is described as a set of "rules" but the spec acknowledges (today, lines 342–347 of `codesystem-notes.xml`) that "There is wide variation in the life cycles supported by the different code systems… These ideas overlap with setting the concept to 'inactive'. In addition, concepts have dates associated with their retirement or deprecation and some code systems imply the status by providing dates."

The reporter (John Moehrke) flagged that this narrative is unclear about whether implementers should populate **both** `status` and `inactive`, or pick one — and, when they conflict, which property wins. Comments on the 2024-12-02 and 2025-01-13 tracker calls confirmed the underlying issue:

- A code system might publish `status = active` together with `inactive = true`, or `status = retired` together with `inactive = false` — directly contradictory states (carmela_couderc, 2025-01-13).
- Computing `inactive` from a "current date" relative to `deprecationDate`/`retirementDate` is ambiguous (which clock? Which `lastUpdated`? The CodeSystem's `period`?) (mjlawley, 2025-01-16).
- Retroactively interpreting historical clinical records as referencing "inactive" codes breaks longitudinal/forwards/backwards compatibility (peter_jordan, 2025-01-13).

The spec needs to state, in plain language, what a consuming system should do when these properties disagree.

### Proposed Change

Append a single new paragraph to the bottom of the **Concept Status** subsection of the CodeSystem narrative, immediately after the current closing paragraph at lines 365–368 of `source/codesystem/codesystem-notes.xml` (the paragraph beginning "The status also relates to properties such as deprecation and retirement date…") and before the `<a name="synonyms"></a>` anchor at line 370.

Paragraph text (verbatim from the resolution):

> Due to variations in editorial process and the uncertainty introduced by distributed lifecycle management, these rules are not always followed, though they should be considered best practice. When they are not followed we recommend that the status overrides any date related properties when present, and codes should only be considered inactive if the inactive concept property has a value of true or in the presence of editorial intent.

Concretely, this establishes two recommendations (non-normative — "we recommend"):

1. **Status > dates.** When the `status` concept property is present and conflicts with any date-based property (`effectiveDate`, `deprecationDate`, `retirementDate`), the value of `status` overrides the date-derived interpretation.
2. **Inactivity is explicit.** A code is only to be considered inactive when the `inactive` concept property is explicitly `true`, or where there is editorial intent (e.g., the source code system's own publication has marked it so) — i.e., do **not** infer `inactive = true` from a past `deprecationDate` or `retirementDate` alone.

No other normative element, cardinality, type, binding, invariant, or search parameter changes. No StructureDefinition edit. No CodeSystem/ValueSet content change.

### Design Rationale

- The resolution is intentionally framed as guidance ("we recommend"), not a SHALL-level invariant, because the 2025-01-13 tracker call concluded that "each code system might operate differently — and there isn't necessarily any consistency to be expected across code systems with regards to valuing these properties." Codifying the precedence as a constraint would invalidate existing real-world content from publishers that follow other conventions.
- Putting the rule "status overrides any date related properties" into the spec directly addresses mjlawley's concern about computing inactivity relative to a "current date" — implementers are guided **not** to do that arithmetic when `status` is present.
- Requiring an explicit `inactive = true` (or "editorial intent") to mark a code inactive directly addresses peter_jordan's longitudinal-compatibility concern: a historical record referencing a code that was later given a `retirementDate` does not retroactively become a reference to an "inactive" code, because the consumer is told not to infer inactivity from dates alone.
- The change is placed at the **bottom** of §4.8.12.1 (per the resolution) rather than rewriting the existing rules list (lines 358–364), preserving in-flight editorial advice while adding the fallback recommendation when the rules are not followed.
- The alternative options floated on 2024-12-02 (deprecating `deprecationDate`/`retirementDate`, adding hard SHALL invariants tying status values to dates, rewording "Deprecated/Retired" bullets) were discussed but did not make it into the approved resolution text and are therefore **out of scope** for this ticket.

2. **Should `deprecationDate` / `retirementDate` actually be deprecated, per option (1) of the 2024-12-02 discussion?**
   The 2024-12-02 comment also floated deprecating those two date properties entirely in favor of `status` + `inactive` + `effectiveDate`. This is **not** in the approved resolution and would itself be a separate (and larger) Persuasive-with-Mod proposal. Out of scope here, but worth flagging because the new paragraph implicitly deprioritizes those date properties ("status overrides any date related properties").

3. **What is "editorial intent"?** The new paragraph leans on the phrase "in the presence of editorial intent" to broaden when a code may be considered inactive. The spec gives no formal definition. If the WG wants to clarify (e.g., a sentence saying "by 'editorial intent' we mean that the publishing source code system has explicitly marked the code as inactive in its own native lifecycle, even if it has not populated the FHIR `inactive` property"), that clarification could be added in the same edit. **Recommendation:** ask the WG whether to inline a one-sentence gloss or leave the term intentionally permissive.

4. **No automated test or invariant is being added.** The 2025-01-13 tracker call asked "Is it worthwhile to try and define invariants/constraints for the [conflicting status/inactive] scenarios?" The approved resolution did not add any. If the WG later decides to add a SHOULD-style invariant on `CodeSystem.concept.property`, that would be a follow-up ticket.
