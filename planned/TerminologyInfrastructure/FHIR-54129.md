## Feature Proposal

### Problem Statement

The current narrative on the `codesystem.html` page (sections 4.8, 4.8.1, 4.8.2 and
4.8.3.1) carries several pre-existing problems:

- The lead description uses the obsolete phrase *"key properties"* and does not
  acknowledge **code system fragments** as first-class content alongside
  supplements.
- The Scope and Usage section anchors itself to the old V3 "Core Principles"
  document, despite the FHIR Terminology Fundamentals page now being the
  authoritative source.
- The list-bullet about case sensitivity confuses *codes* with *concept codes*,
  which matters for code systems that distinguish identifiers from labels.
- The Boundaries and Relationships paragraph mixes a normative description of
  `NamingSystem` together with an editorial argument about *who* maintains each
  resource, which neither belongs in the resource definition nor reflects current
  practice.
- The CodeSystem Identification bullet ordering puts the implementation-local
  `CodeSystem.id` *before* the canonical `CodeSystem.url`, the opposite of the
  emphasis given everywhere else in the spec.
- The `CodeSystem.id` bullet duplicates generic guidance that already appears on
  `resource.html#id`.

These are documentation defects only — there is no schema impact — but they
materially affect how implementers read the resource scope and identification
guidance.

### Proposed Change

Apply the literal substitutions captured in *Resolution Summary* above. All
edits land in the bespoke HL7 publisher's authoring sources for the CodeSystem
resource, located under `source/codesystem/` in `HL7/fhir`:

- **Resource lead (rendered as section 4.8).** In
  `source/codesystem/structuredefinition-CodeSystem.xml`, update the two values
  that drive the resource lead description:
  - `StructureDefinition.description` (line 54)
  - the root-element `definition` for `CodeSystem` (line 88)

  to the new text *"The CodeSystem resource is used to declare the existence of
  and describe a code system, code system fragment or code system supplement and
  its attributes, and optionally define its content."*

- **Sections 4.8.1, 4.8.2 and 4.8.3.1.** All three are authored in
  `source/codesystem/codesystem-introduction.xml`:
  - **4.8.1** is rendered from the `<a name="scope">` block. Replace the
    paragraphs and bullets as itemised in *Resolution Summary §2*.
  - **4.8.2** is rendered from the `<a name="relationships">` block. Replace the
    `CodeSystem` vs `NamingSystem` paragraph as itemised in *Resolution Summary §3*.
  - **4.8.3.1** is rendered from the `<a name="ident">` block, nested under the
    `<a name="context">` Background and Context section. Re-order the three
    bullets, trim the `CodeSystem.id` bullet, drop *"location"* from the
    surrounding paragraph, and prepend the new struck-through introductory phrase
    to the `CodeSystem.identifier` bullet (using `<del>` markup).

No `<snapshot>` blocks, search parameters, operation definitions, or example
instances change. No new elements, types, cardinalities, or terminology bindings
are introduced.

### Design Rationale

- The ticket is filed as a **Technical Correction** with resolution
  **Persuasive**; the description doubles as the resolution text, so the agreed
  approach is to apply the wording the reporter proposed.
- Putting `CodeSystem.url` first matches the rest of the FHIR core narrative
  (Coding.system, ValueSet binding, the canonical-URL contract for resources)
  and is consistent with the `CodeSystem.url` being the only identifier that is
  *stable across servers*.
- Trimming the `CodeSystem.id` bullet to the CodeSystem-specific point removes
  duplicate prose; the generic "logical id on a server" guidance already lives
  on `resource.html#id`, which is linked from the surrounding context.
- Re-anchoring 4.8.1's first paragraph from the V3 Core Principles document to
  the FHIR Terminology Fundamentals page tracks the reorganisation of the
  terminology pages already underway in the FHIR R6 cycle.
- Editing the SD-level `description` keeps the same narrative in the resource
  reference table at the top of the page (which is generated from the SD), so
  the page lead and the SD never diverge.
