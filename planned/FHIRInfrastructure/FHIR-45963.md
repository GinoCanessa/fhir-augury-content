## Resolution Summary

The ticket reports that the sentence *"This pattern fixes the value of the
string, so that this value is valid:"* — followed by an XML/JSON tabset
containing `<element value="This is a string value"/>` — appears twice in
immediate succession on a FHIR specification page. The resolution was set
to **Persuasive** with **Resolved – change required**, meaning the
duplication is confirmed and the redundant block must be removed. No
free-text resolution description, applied-vote comment, or linked PR was
recorded; the corrective action is simply to delete the duplicate
paragraph + tabset so the page reads cleanly.

The duplicated content lives on the *Element Definition Examples* page
(`source/elementdefinition-examples.html`) in the **Pattern Examples →
Primitive Types** subsection. Lines 50–62 and 63–75 in the current clone
are byte-for-byte identical (a `<p>` followed by an
`[%example-start%]` / `[%example-json%]` / `[%example-end%]` tabset).

## Feature Proposal

### Problem Statement

The Pattern Examples page in FHIR Core duplicates a paragraph and its
associated XML/JSON tabset. After the introductory `patternString`
example, the prose *"This pattern fixes the value of the string, so that
this value is valid:"* and its accompanying tabset (`<element
value="This is a string value"/>` plus the JSON form) are repeated
verbatim before the page continues with *"as is this:"*. Readers
following the example flow encounter the same example twice with no
intervening explanation, which obscures the intent of the section
(showing one valid form, then a second valid form with an extension,
then an invalid form).

### Proposed Change

In `source/elementdefinition-examples.html`, delete one of the two
identical paragraph + tabset pairs on lines 50–62 / 63–75 so that a
single instance remains. Concretely, remove the second occurrence (lines
63–75 in the current `db79dcd` HEAD):

```html
<p>
This pattern fixes the value of the string, so
that this value is valid:
</p>
[%example-start%]
<pre class="xml">
  &lt;element value=&quot;This is a string value&quot;/&gt;
</pre>
[%example-json%]
<pre class="json">
  "element" : "This is a string value"
</pre>
[%example-end%]
```

After the edit, the surrounding flow should read:

1. Intro `patternString` example (lines 41–49, unchanged).
2. *"This pattern fixes the value of the string, so that this value is
   valid:"* + the `<element value="This is a string value"/>` tabset
   (kept once).
3. *"as is this:"* + the extended-element tabset (unchanged, currently
   lines 76–97).
4. *"but this is not:"* + the invalid-content tabset (unchanged,
   currently lines 98–109).

No other prose, anchors (`pattern-examples`, `pattern-primitive`,
`pattern-complex`), tables, or cross-links are affected. The page's
`[%settitle …%]`, `<%dtxheader …%>`, work-group / standards-status row,
and downstream complex-type examples remain intact.

### Design Rationale

The change is a pure technical correction: the duplicated block is
clearly an authoring slip (identical XML, identical JSON, identical
prose, with the second copy breaking the "valid → also valid → not
valid" three-way contrast that the section is designed to teach). The
Persuasive resolution endorses the simplest fix — delete the duplicate
— rather than rewriting the section. There is no alternative structure
to weigh: removing the redundant block restores the intended pedagogical
sequence without changing semantics, normative behavior, or any anchor
target. Deleting the *second* occurrence (rather than the first) keeps
the block that immediately follows the `patternString` introduction,
preserving the most natural read order.
