## Resolution Summary

The committee resolved this as **Not Persuasive with Modification**. The
proposal to add an `abstract` boolean flag to `ImplementationGuide` was not
accepted. Instead, the resolution splits the work two ways:

1. **Codified categorisation** of an IG's intended purpose (the part that
   would have replaced the requested `abstract` flag) is deferred to and will
   be handled under **FHIR-48768** ("Define terminology so that authors can
   use usageContext to categorise profiles and IGs"). FHIR-48768 is currently
   `Unresolved` / `Waiting for Input`, so no terminology work is performed in
   this ticket.
2. **A new narrative subsection** must be added to the ImplementationGuide
   notes page, titled **"Declaring conformance to guides"**, with the exact
   verbatim wording given in the resolution description (reproduced in
   "Proposed Change" below). The new prose explicitly leans on
   `ImplementationGuide.useContext` as the (current) computable surface for
   surfacing intended purpose pending FHIR-48768.

Net effect for this ticket: a single, narrative-only addition to
`source/implementationguide/implementationguide-notes.xml`. No
StructureDefinition, schema, search-parameter, terminology, or example
changes are required.

## Feature Proposal

### Problem Statement

Some implementation guides are not intended to be implemented directly; they
exist as a reusable substrate for derivative IGs (examples cited in the
ticket: Clinical Order Workflow / COW, several national "base" IGs). Other
specifications behave similarly without being expressed as
`ImplementationGuide` resources at all (e.g., SMARTv2 scopes). Today the
specification gives readers and contracting/regulatory authors no guidance
distinguishing IGs that are appropriate to point at in conformance language
("you SHALL implement IG X") from IGs that are libraries / building blocks
that need additional, downstream constraint before they can meaningfully be
implemented. The reporter framed this by analogy to C#'s `abstract` keyword.

### Proposed Change

Add a new authored subsection to the ImplementationGuide notes page. The
notes are assembled from the per-resource notes XML; for `ImplementationGuide`
that is `source/implementationguide/implementationguide-notes.xml`. Insert a
new `<h3>` section with anchor `conformance` (or another anchor that does not
collide with the existing `packaging`, `authoring`, `roles`, `versions`,
`default` anchors in that file). The natural insertion point is immediately
**after** the existing `<h3>Roles of Implementation Guides</h3>` block and
**before** `<a name="versions"></a><h3>Multi-Version Implementation
Guides</h3>`, because the new content is about *how IGs are referenced as
conformance targets* — a logical follow-on to "Roles of Implementation
Guides".

The section content must be the exact prose from the resolution:

```html
<a name="conformance"></a>
<h3>Declaring conformance to guides</h3>
<p>
It's not uncommon for contractual or regulatory language to require
implementation of certain implementation guides. However, not all guides
are appropriate to be referenced in this manner. For example, some
implementation guides may define libraries of artifacts or make general
context-agnostic recommendations that require additional guidance/constraint
to be useable. Other guides will define a number of CapabilityStatements
and/or ActorDefinitions and when setting implementation expectations, it's
necessary to point to those child resources in order to meaningfully assert
implementation expectations.
</p>
<p>
Implementation guides <b>SHOULD</b> make clear what their intended purposes
and uses are. The
<a href="implementationguide-definitions.html#ImplementationGuide.useContext">ImplementationGuide.useContext</a>
element may be helpful in surfacing these expectations in a more computable
fashion.
</p>
```

Notes on the markup (chosen to match conventions already present in
`implementationguide-notes.xml`):

- Use `<h3>` (the file's existing section level) and a leading
  `<a name="..."></a>` anchor as the file's existing sections do
  (`packaging`, `authoring`, `roles`, `versions`, `default`).
- Hyperlink `ImplementationGuide.useContext` to
  `implementationguide-definitions.html#ImplementationGuide.useContext`,
  mirroring the already-present link pattern for
  `ImplementationGuide.fhirVersion` further down the same file.
- Use `<b>SHOULD</b>` (the file is already authored as XHTML and uses inline
  markup for emphasis); keep "SHOULD" upper-case to preserve normative
  intent.
- The two paragraphs in the resolution are quoted verbatim with only
  whitespace/markup normalisation (the quoted text in the resolution is
  copy-pasted from a JIRA editor and uses curly quotes / interleaved
  whitespace; emit straight ASCII quotes and a single regular space after
  punctuation, matching the rest of the file).

No other artifacts are added, removed, or modified. In particular:

- `structuredefinition-ImplementationGuide.xml` is **not** changed (no new
  `abstract` element — this was the rejected portion of the request).
- No new ValueSet or CodeSystem is introduced; categorisation codes are the
  scope of FHIR-48768.
- `bundle-ImplementationGuide-search-params.xml` is unchanged (no new
  searchable element).
- No example IGs (`implementationguide-example.xml`,
  `list-ImplementationGuide-examples.xml`) need updating because no
  resource-instance change is being made.

### Design Rationale

- The work group explicitly rejected adding an `abstract` boolean. A boolean
  flag is too coarse for the use cases enumerated in the ticket and in
  comments (abstract IGs vs. IGs that *contain* abstract profiles vs.
  informative-only/narrative-only IGs — esilver's comment). A coded element,
  driven by a ValueSet, is the agreed direction, and that work is owned by
  FHIR-48768 so as not to fork two parallel solutions.
- In the meantime, the spec needs to give implementers and contract/policy
  authors guidance, hence the narrative-only addition here. Pointing them at
  `ImplementationGuide.useContext` re-uses an element that already exists
  on `ImplementationGuide` and is the most plausible long-term landing zone
  for the FHIR-48768 ValueSet.
- Editing the per-resource notes file (rather than `implementationguide.html`
  or `implementationguide-definitions.html`) is consistent with the FhirCore
  authoring convention: per-resource long-form prose lives in
  `source/<resource>/<resource>-notes.xml` and is composed by the publisher
  into the final resource page.

### Open Questions

- **Anchor name:** the resolution does not prescribe the HTML anchor for
  the new subsection. `#conformance` is proposed (short, memorable, matches
  the section title). If FHIR Infrastructure prefers a less generic name
  (e.g., `#declaring-conformance`) to leave `#conformance` free for future
  use elsewhere on the IG page, the implementer should swap it in step 1
  and step 2.
- **Section ordering:** placement between "Roles of Implementation Guides"
  and "Multi-Version Implementation Guides" is a judgement call grounded
  in topical adjacency. The resolution does not specify a position; if the
  WG would rather place it at the very end of the notes file (after
  "Default Profiles") that is equally defensible. Confirm with the WG
  before merging if there is any doubt.
- **Forward compatibility with FHIR-48768:** when FHIR-48768 lands a
  concrete ValueSet for IG purpose categorisation, the second paragraph of
  the new section may want to be tightened from "may be helpful" to a
  reference to the new ValueSet. That is a follow-up edit, not a
  prerequisite, and is not in scope here.
- **Per the briefing's gotcha about the publisher:** confirm that
  `implementationguide-notes.xml` is the file actually composed into the
  rendered IG resource page (the file naming convention strongly suggests
  it is, and it is the only `*-notes.xml` under
  `source/implementationguide/`). If the publisher pipeline pulls notes
  from a different file, redirect the edit accordingly.
