## Resolution Summary

**Resolution:** Persuasive. The ticket reports a specification inconsistency:
the `Narrative` datatype is Normative, but `Narrative.status` has a `required`
binding to the ValueSet `http://hl7.org/fhir/ValueSet/narrative-status`, which
is currently published as "informative". Per the originating Zulip thread
([#implementers > Required binding to informative valueset](https://chat.fhir.org/#narrow/stream/implementers/topic/Required%20binding%20to%20informative%20valueset)),
Lloyd McKenzie's direction is unambiguous:

> "the narrative-status value set should absolutely be marked as normative,
> given that the narrative type itself is normative"
>
> "The assertion that the value set is 'informative' is an error. There shouldn't
> be **any** informative value sets."

The fix, therefore, is to change the standards-status of both the
`ValueSet/narrative-status` **and** its backing `CodeSystem/narrative-status`
from informative to normative, aligning them with the standards-status of the
`Narrative` datatype (already `normative`, per `source/fhir.ini` line 976).
No code values, displays, definitions, URIs, OIDs, or binding on
`Narrative.status` are changed.

## Feature Proposal

### Problem Statement

`Narrative` is a normative FHIR datatype (ANSI-approved, with the stability
guarantees that implies). `Narrative.status` carries a `required` binding
(`1..1`, type `code`) to `http://hl7.org/fhir/ValueSet/narrative-status`, whose
codes are defined by `http://hl7.org/fhir/CodeSystem/narrative-status`
(`generated | extensions | additional | empty` — see
`source/datatypes/narrative.xml` rows around line 865 and the
`narrative-status` worksheet at line 6090).

The published `narrative-status` ValueSet is labelled `Standards Status:
Informative`. Because a `required` binding from a normative element mandates
that instances use codes from that ValueSet, the ValueSet and its CodeSystem
are *de facto* normative: any breaking change to them would break the
normative `Narrative` contract. Publishing them as informative therefore
misrepresents their real stability guarantees and contradicts the statement
that "Informative" means "does not make rules that implementers are required
to follow".

### Proposed Change

Promote the standards-status of the narrative-status terminology to
`normative` so that the published metadata matches reality.

Target artifacts (all in `HL7/fhir`):

- **ValueSet** `http://hl7.org/fhir/ValueSet/narrative-status`
  - Change `Standards Status` from `Informative` → `Normative`.
  - Mark the ValueSet as being normative from the same FHIR version in which
    `Narrative` itself became normative (R4 — `Narrative=4.0.0` per the
    `[normative-version]` / equivalent block on `source/fhir.ini` lines 945 and
    962, where peer normative VS/resources (e.g., `ValueSet=4.0.0`) are
    likewise anchored to R4). `Normative Version: 4.0.0`.
- **CodeSystem** `http://hl7.org/fhir/CodeSystem/narrative-status`
  - Same promotion: `Standards Status: Normative`, `Normative Version: 4.0.0`.
  - This is required for consistency: a normative ValueSet cannot have a
    `required` binding implicitly derived from an informative backing
    CodeSystem without creating the same contradiction one step down.

No other fields change:

- URLs, identifiers/OIDs, `name`, `title`, descriptions, `caseSensitive`, and
  the four concept codes (`generated`, `extensions`, `additional`, `empty`)
  with their existing displays/definitions are **unchanged**.
- `Narrative.status`' binding strength stays `required`; its reference
  continues to be `http://hl7.org/fhir/ValueSet/narrative-status`.
- The `Narrative` StructureDefinition requires no changes — only the
  referenced terminology's standards-status metadata is updated.

### Design Rationale

- Consistency with the normative `Narrative` datatype: a `required` binding
  from a normative element whose codes *must* come from a specific ValueSet
  effectively promises that ValueSet (and its CodeSystem) will not change
  incompatibly. Labelling it `Informative` contradicts that promise. This is
  exactly the alignment Lloyd called for in the Zulip thread.
- Consistency with peers: sibling enumerations for other normative datatypes
  (`Identifier.use → ValueSet/identifier-use`, `HumanName.use →
  ValueSet/name-use`) are already published as normative — `narrative-status`
  is an anomaly among these `use`-style required bindings. (Confirmed in the
  R5 history snapshot at
  `tools/history/release5/valuesets.xml` lines 17985/18283/147030/116973.)
- Aligning with Lloyd's stronger assertion that there should be no
  informative ValueSets at all. (Cleaning up every other informative VS is
  out of scope for this ticket; addressing `narrative-status` specifically
  is in scope.)
- Alternative considered — weakening `Narrative.status`' binding strength
  (e.g., to `extensible`) — was implicitly rejected by the resolution and by
  Lloyd's recommendation to promote the ValueSet; loosening a binding on a
  normative element would itself be a breaking change.

## Impact Analysis

### Affected Repositories

- **Role:** Single point of change. Publishes both the `Narrative` datatype and its bound `narrative-status` ValueSet + CodeSystem. Standards-status metadata is promoted from Informative to Normative in the authoring source so that the regenerated VS/CS carry the new label.
- **Affected Files:**
  - `source/fhir.ini` — add (or amend) the entry that drives the publisher's standards-status labelling for `narrative-status` VS/CS. Most likely a new line in the existing `[standards-status]` section (model after `Narrative=normative` at line 976) and a matching `narrative-status=4.0.0` entry in the `[normative-version]` block (model after `ValueSet=4.0.0` at line 962). Exact key name needs confirmation — see Open Questions.
  - `source/datatypes/narrative.xml` — only if the publisher resolves standards-status for generated VS/CS from the spreadsheet rather than from `fhir.ini`. If so, the targeted edit is on the `NarrativeStatus` row in the `Bindings` worksheet (lines ~5628–5640) and/or on the `narrative-status` worksheet header (line ~6090). The spreadsheet may need a `Standards Status` / `Normative Version` cell added — this is a metadata-only change, not a code/value edit.
  - `source/narrative.html` — scan for any hand-authored prose that characterises `narrative-status` as informative, example "NarrativeStatus is informative" or similar phrasing, and bring it in line with the new normative status.
- **Change Scope:** Minor (metadata-only; no code values, no binding strength, no path/cardinality changes).

### Breaking Changes

**No breaking changes.** Standards-status promotion from Informative →
Normative tightens stability guarantees going forward but does not change any
instance's conformance: every instance that validated before (with one of the
four existing codes `generated | extensions | additional | empty`) still
validates after. The `required` binding on `Narrative.status` is unchanged.

A normative label does lock the terminology against future incompatible
changes — that is the point of the ticket — but does not invalidate existing
artefacts. FHIR balloting/QA rules will require the standards-status change
itself to go through the appropriate process, but technical-correction
tickets are the standard vehicle for that, matching this ticket's Type.

### Related Specifications

- `Narrative` datatype — its `status` element consumes `ValueSet/narrative-status` via `required` binding (per spreadsheet rows in `source/datatypes/narrative.xml` line 853 onward). Definition text in the VS/CS is already normative-appropriate.
- Every normative resource that allows a `Narrative` (essentially every `DomainResource` — hence the keyword `domainresource` appearing in the ticket's extracted keywords): their stability promise implicitly depends on the stability of `narrative-status`. Promotion strengthens, rather than perturbs, those promises.
- Peer `*-use` normative ValueSets (`ValueSet/identifier-use`, `ValueSet/name-use`) — not edited, but serve as the conformance pattern this change aligns `narrative-status` with.

### Related Tickets

- No related Jira tickets were returned by `cross-referenced FHIR-42868` (one result only: the filing Zulip message #394712135). `get_jira_item_links` returned an empty array. If a future ticket is raised to sweep up other informative-but-required-binding ValueSets (per Lloyd's broader remark that no ValueSet should be informative), it should reference this ticket as prior art.

- `CodeSystem/narrative-status` — same promotion.
- Concept list (`generated`, `extensions`, `additional`, `empty`) and their
  definitions are unchanged.

HL7/UTG does **not** carry narrative-status, so no UTG changes are required
