## Feature Proposal

### Problem Statement

A ValueSet binding on a FHIR Infrastructure-owned resource currently exposes the binding name as `characterist ic` (two words separated by a space) rather than `characteristic`. Binding names appear in the published specification as the human-readable label for the binding and are reused by downstream tooling (IG Publisher cross-links, definitions JSON, R6 ballot HTML), so a whitespace-broken name looks like a defect and breaks exact-match tooling that keys off the binding name.

### Proposed Change

In the single `elementdefinition-bindingName` extension whose `valueString` is currently `characterist ic`, change that `valueString` to `characteristic`. Concretely:

```xml
<!-- Before -->
<extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
  <valueString value="characterist ic"/>
</extension>

<!-- After -->
<extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
  <valueString value="characteristic"/>
</extension>
```

If the authoritative source for the element is an authoring spreadsheet (`source/<resource>/<resource>-spreadsheet.xml`) rather than the hand-maintained StructureDefinition XML, the equivalent edit is to the "Binding Name" cell for that element — replacing `characterist ic` with `characteristic`. The binding strength, the bound ValueSet URL, and every sibling extension must remain untouched.

No example instances, invariants, search parameters, or ValueSet/CodeSystem artifacts change. This is a one-character deletion.

### Design Rationale

Binding names in FHIR Core use short CamelCase/lowercase identifiers without whitespace (e.g. `BindingStrength`, `PublicationStatus`, `AddressUse`). A space-separated binding name is therefore both a typo and a style violation. The resolution ("Fix the typo", `metadata.resolution = Persuasive`) endorses the reporter's proposed correction verbatim, so no alternative designs are on the table. No Zulip discussion, related tickets, or comments were surfaced by the `cross-referenced` and `keywords` calls — there is no design debate to reconcile.
