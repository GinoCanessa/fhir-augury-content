## Resolution Summary

Resolution type: **Persuasive with Modification**. Two concrete changes are required to the GraphDefinition narrative examples:

1. Replace the placeholder example that links to the (now defunct) RCPA page
   `http://fhir.hl7.org.au/fhir/rcpa/pprofiles.html`. The resolution text
   says: "update the example (in both places) to pick one of the two
   approaches and will comment to explain which approach we picked and note
   that that choice is for example purposes only, not a definitive best
   practice for the representation of this CDA construct." Because the
   original RCPA page is gone, the link is to be replaced with the Wayback
   Machine snapshot
   `https://web.archive.org/web/20211128093752/http://fhir.hl7.org.au/fhir/rcpa/pprofiles.html`,
   and the TODO ("Todo: make this an actual graph definition, and clone
   into the main spec") is to be **tracked as "need a volunteer to author
   the actual GraphDefinition"** rather than left as an inline source
   comment.
2. Remove the standalone TODO comment
   `"Todo: how would this be validated? - where is the graph referred to?"`
   because it is already resolved by [FHIR-44865](https://jira.hl7.org/browse/FHIR-44865).

A third TODO referenced in the original ticket description —
`<!-- any section entry. Todo: this recurses; are we profiling this at all levels? -->`
— appears in the same file in the Composition-example XML snippet. The
resolution wording ("pick one of the two approaches … comment to explain
which approach we picked") is most naturally read as applying to this
same pair of TODOs inside the illustrative XML block, which needs to be
resolved into a concrete choice (not left as an open question in the
narrative).

## Feature Proposal

### Problem Statement

The narrative/notes page for the `GraphDefinition` resource contained
three inline `Todo:` markers left over from the original authoring
effort:

- Inside the XML example under *Simple Examples*: a comment on
  `Composition.section.entry` that questions whether the graph is being
  profiled at all levels (recursion concern).
- Under *Summarize a set of profiles on resources*: a parenthetical
  asking for someone to convert a link-to-an-external-Australian-profile
  into a real FHIR GraphDefinition and clone it into the main spec.
- Under *Document Relationship rules*: an unresolved question on how
  compartment-consistency rules would be validated.

These TODOs shipped into the published specification. The ticket asks
for them to be resolved so that GraphDefinition narrative does not
expose editorial TODOs to readers.

### Proposed Change

Apply the three edits below to the GraphDefinition notes/narrative page
in the authored HL7/fhir source.

1. **Replace the dead RCPA link and the "make this an actual graph definition" TODO.**
   In the paragraph that currently reads (approximately):

   > Here's `<a href="http://fhir.hl7.org.au/fhir/rcpa/pprofiles.html">an example</a>` of
   > the kind of summary this represents. (Todo: make this an actual graph definition,
   > and clone into the main spec)

   - Change the `href` to
     `https://web.archive.org/web/20211128093752/http://fhir.hl7.org.au/fhir/rcpa/pprofiles.html`
     (and add `rel="noopener"` / a note that the live page is gone, if
     the house style permits).
   - Remove the parenthetical `(Todo: …)` from the narrative.
   - Do **not** silently swallow the remaining work item: open (or link
     to) a follow-up HL7/fhir GitHub issue titled "Author a concrete
     GraphDefinition example mirroring the RCPA profile summary" so the
     need for a volunteer-authored example is tracked rather than left
     inline.

2. **Remove the compartment-validation TODO.**
   In the paragraph immediately after the compartment-rule code table in
   the *Document Relationship rules* section, delete the sentence
   `Todo: how would this be validated? - where is the graph referred to?`
   entirely. If desired, replace it with a single sentence such as:
   "See [FHIR-44865](https://jira.hl7.org/browse/FHIR-44865) for the
   `$graph` operation's validation behaviour." The resolution
   explicitly states FHIR-44865 covers this concern.

3. **Resolve the "any section entry" XML-comment TODO.**
   In both copies of the `Composition.section.entry`-to-`list` link
   inside the *Simple Examples* XML (the first minimal example and the
   second one that adds the `<compartment>` block), replace the inline
   comment
   `<!-- any section entry. Todo: this recurses; are we profiling this at all levels? -->`
   with a non-TODO explanatory comment that commits to one of the two
   approaches — per the resolution, pick the approach that **does not**
   recurse at all Composition.section levels (i.e. treat the link as
   applying at the immediate section only) and state that this choice is
   "for example purposes only, not a definitive best practice for
   representing the CDA `section.entry` construct."

Example replacement comment:

```xml
<!-- Composition.section.entry at the outer section only (nested
     sections are out of scope for this example). Shown for example
     purposes only; not a definitive best practice for representing
     the CDA section.entry construct. -->
```

### Design Rationale

- The resolution wording is deliberate: it explicitly refuses to make
  this a normative best practice and instead scopes the change to
  "example purposes only." Keeping an explanatory comment in the
  example — rather than deleting the example or silently rewriting it —
  preserves reader understanding of why the example was structured that
  way.
- Redirecting the external link to a Wayback Machine snapshot is the
  lowest-risk way to honour the original citation now that the live
  RCPA page has moved.
- Removing the compartment-validation TODO is safe because
  [FHIR-44865](https://jira.hl7.org/browse/FHIR-44865) owns that
  question; leaving a cross-reference to it keeps the narrative
  navigable without re-opening debate inside this page.
- Tracking the "author a real GraphDefinition" request as a follow-up
  GitHub issue (rather than leaving a TODO in the spec) separates
  spec-cleanup (this ticket) from new authoring work (a volunteer
  exercise).
