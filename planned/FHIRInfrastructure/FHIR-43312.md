## Resolution Summary

The committee resolved the ticket as **Persuasive with Modification**. Rather than introducing a new top-level `offer` intent code on `ServiceRequest.intent`, the resolution requires:

1. Add a level-2 code under `proposal` called **`solicit-offer`** to capture an "ask" ServiceRequest.
2. Add a level-2 code under `proposal` called **`offer-response`** to capture a response to an "ask" ServiceRequest.
3. Add the following to the notes section of ServiceRequest:
   > "To capture the workflow around the solicitation of potential ServiceRequests based on time, price, etc, use the level 2 code `solicit-offer` on the `ask` ServiceRequest. For the ServiceRequest response to an `ask` ServiceRequest, use the level 2 code `offer-response` along with a `basedOn` reference to the `ask` ServiceRequest."

Per applied-vote comment from John D.L. Nolen on 2025-09-19, the change was applied to the build in commit `954ae5d` on PR [HL7/fhir#3677](https://github.com/HL7/fhir/pull/3677). Inspection of the clone shows the commit (and `servicerequest-notes.xml`) only addresses **part** of the resolution — see Impact Analysis for the gap that still needs to be closed before QA can clear `PendingQA`.

## Feature Proposal

### Problem Statement

`ServiceRequest` today has no way to express the "bidding" / "offer-to-fulfil" workflow described in the ticket: a prospective filler proposing fulfilment based on a paper or digital request token, or multiple potential fillers offering service against a single request with their own constraints (location, time, cost). The original ticket and the linked Zulip thread *[workflow] Bidding on Tasks* (`405178604`) and *[implementers] Parallel processing communication workflow* (`436328783`) note that R4/R5 support for this pattern is limited. The reporter initially asked for a new top-level `intent` code (`offer`); the resolution refines this to two child codes under the existing `proposal` intent so that "ask" and "response" ServiceRequests are clearly distinguishable while remaining within the proposal semantic (no authorization to act).

### Proposed Change

Two coordinated changes against the FHIR Core build:

1. **`request-intent` CodeSystem** — add two child concepts beneath the existing `proposal` concept:
   - `code = solicit-offer`, `display = Solicit Offer`, `definition` = "A proposal that solicits offers for fulfilment of a service request, e.g., where multiple potential fillers may respond with constraints such as location, time, or cost."
   - `code = offer-response`, `display = Offer Response`, `definition` = "A proposal made in response to a `solicit-offer` ServiceRequest, in which a prospective filler offers to fulfil the solicited request. The offering ServiceRequest should reference the soliciting ServiceRequest via `basedOn`."

   Both concepts are level-2 codes nested inside the `<concept>` block whose `<code value="proposal"/>` (mirroring the existing `original-order`/`reflex-order`/`filler-order`/`instance-order` nesting under `order`).

2. **`ServiceRequest.intent` element short label** — already updated by commit `954ae5d` to `proposal | solicit-offer | offer-response | plan | directive | order | original-order | reflex-order | filler-order | instance-order | option`. No further edit required to the StructureDefinition itself; the same `short` change was applied (correctly) to all 6 sibling request-pattern resources that bind to `ValueSet/request-intent` (ActivityDefinition, CommunicationRequest, DeviceRequest, NutritionOrder, RequestOrchestration, SupplyRequest). The `<binding>` continues to reference `http://hl7.org/fhir/ValueSet/request-intent`, which (via `<include><system value=".../request-intent"/></include>`) will automatically expand to include the two new codes once they are added to the CodeSystem.

3. **`servicerequest-notes.xml`** — already contains the required workflow paragraph (verified in clone). No further edit required.

Example "ask" ServiceRequest fragment:

```xml
<intent value="solicit-offer"/>
```

Example response:

```xml
<intent value="offer-response"/>
<basedOn>
  <reference value="ServiceRequest/ask-123"/>
</basedOn>
```

### Design Rationale

The committee preferred deepening the existing `proposal` intent over inventing a new sibling top-level code. Rationale evident from the comment thread and the resolution:

- Both an "ask" and a response to an "ask" are still **proposals** — neither carries authorization to act. Modeling them as level-2 children of `proposal` preserves the existing `proposal | plan | directive | order | option` semantic spine documented in `request-intent` CodeSystem definitions.
- Pattern matches the established level-2 pattern under `order` (`original-order`, `reflex-order`, `filler-order`, `instance-order`), so consumers that already know to coalesce level-2 codes to their parent for coarse processing get the right behaviour for free.
- Keeps the change additive at the CodeSystem level — no existing consumer breaks; `proposal` remains valid, and existing instances do not need to be re-coded.
- Esilver's OO comment noted concern that the broader bidding workflow needed FHIR-I review before adopting a wholly new top-level code; the level-2 codes plus a pointer in `servicerequest-notes.xml` is the lighter-weight expression of that workflow.

The alternative (a top-level `offer` intent as originally proposed) was not chosen because it would have implied semantics outside the `proposal` family and required redefining or expanding the meaning of `intent` itself.

The `request-intent` ValueSet (`http://hl7.org/fhir/ValueSet/request-intent`) is required-bound from `intent` on ActivityDefinition, CommunicationRequest, DeviceRequest, NutritionOrder, RequestOrchestration, ServiceRequest, and SupplyRequest. Once the CodeSystem gains the two new codes, all seven resources can carry `intent = solicit-offer` or `intent = offer-response`. ### Related Tickets

No directly related Jira tickets surfaced via `cross-referenced FHIR-43312` (only the two Zulip threads above). A targeted review of other `request-intent` or `ServiceRequest.intent` tickets is not required for this change because the modification is additive and confined to one CodeSystem.

### Terminology Impact

- `CodeSystem/request-intent` (`http://hl7.org/fhir/request-intent`) — adds two concepts (`solicit-offer`, `offer-response`) nested under `proposal`. CodeSystem is normative; concept additions are non-breaking.

- No HL7/UTG impact: `request-intent` is a FHIR-Core terminology, not UTG-managed.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add `solicit-offer` and `offer-response` child concepts under `proposal` in the request-intent CodeSystem**
   - File: `source/request/codesystem-request-intent.xml`
   - Action: Inside the existing `<concept>` block whose `<code value="proposal"/>` is on line ~25, insert two nested `<concept>` children (mirroring the structure already used for `original-order`/`reflex-order`/`filler-order`/`instance-order` under `order`).
   - Details:
     ```xml
     <concept>
       <code value="proposal"/>
       <display value="Proposal"/>
       <definition value="The request is a suggestion made by someone/something that does not have an intention to ensure it occurs and without providing an authorization to act."/>
       <concept>
         <code value="solicit-offer"/>
         <display value="Solicit Offer"/>
         <definition value="A proposal that solicits offers for fulfilment of a service request from one or more prospective fillers, who may respond with constraints such as location, time, or cost."/>
       </concept>
       <concept>
         <code value="offer-response"/>
         <display value="Offer Response"/>
         <definition value="A proposal made in response to a 'solicit-offer' request, in which a prospective filler offers to fulfil the solicited request. The offering request should reference the soliciting request via basedOn."/>
       </concept>
     </concept>
     ```
