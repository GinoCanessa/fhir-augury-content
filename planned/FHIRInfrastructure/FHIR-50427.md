## Resolution Summary

The resolution is **Persuasive** (Block-Vote-6). The narrative for `Attachment.url`
must be reworded so that the "absolute URL / de-referenceable" requirement is
scoped to the case where `data` is absent. Specifically, the current wording:

> "When populated, the `url` element SHALL be an absolute url that can be
> de-referenced to the attachment's content."

is to be replaced with:

> "If the `data` element is not populated and the `url` element is populated,
> the `url` element SHALL be an absolute url that can be de-referenced to the
> attachment's content."

The change loosens the absolute-URL/de-referenceability requirement when both
`data` and `url` are present (the attachment's content is already inline, so
the URL only needs to point at the same content and is permitted to be a
relative or transient reference).

## Feature Proposal

### Problem Statement

The current Attachment narrative makes an unconditional SHALL statement that
any populated `Attachment.url` must be an absolute, de-referenceable URL.
That is too strong: when `Attachment.data` is also populated, the URL is a
co-pointer to the same content (and the current Comments on `Attachment.url`
already explicitly allow relative URLs, `cid:` references, transient
wrapping-envelope locations, and Bundle-internal URIs). The unconditional
SHALL therefore contradicts the rest of the Attachment Comments and over-
constrains conformant senders that include both inline data and a non-
absolute URL.

### Proposed Change

Reword the `Attachment.url` narrative so the absolute-URL / de-reference
requirement is conditional on `data` being absent. Concrete edit:

- **Where:** the Comments cell for the `Attachment.url` row in the
  Attachment spreadsheet (`source/datatypes/attachment.xml`, the row
  beginning at line 925 with cells running through line 949, comment cell
  at line 942).
- **What to change:** the sentence in that Comments cell that today reads
  "If a URL is provided, it SHALL resolve to actual data." (line 942) is to
  be replaced / expanded to:

  > "If the `data` element is not populated and the `url` element is
  > populated, the `url` element SHALL be an absolute URL that can be
  > de-referenced to the attachment's content."

- Retain the surrounding existing sentences in the same cell that already
  permit relative URLs, `cid:` references, transient wrapping envelopes
  and Bundle-internal URIs when `data` is also populated.
- No change to cardinality (`Attachment.url` remains `0..1`, type `url`),
  no change to invariants, no change to bindings.

Resulting Comments-cell text (proposed):

> "If both `data` and `url` are provided, the `url` SHALL point to the
> same content as the `data` contains. URLs may be relative references or
> may reference transient locations such as a wrapping envelope using
> `cid:` though this has ramifications for using signatures. Relative
> URLs are interpreted relative to the service URL, like a resource
> reference, rather than relative to the resource itself. URLs may also
> be URIs only resolvable within a Bundle containing the resource with
> the attachment type. **If the `data` element is not populated and the
> `url` element is populated, the `url` element SHALL be an absolute URL
> that can be de-referenced to the attachment's content.**"

### Design Rationale

The two-sentence pairing in the resolution makes the conformance rule
internally consistent with the rest of the Attachment Comments, which
already enumerate the legitimate non-absolute URL forms. By gating the
SHALL on "data not populated", the spec correctly distinguishes:

- the **co-pointer** case (`data` + `url` both present): URL only needs to
  point at the same content and may be relative / `cid:` / Bundle-scoped, and
- the **by-reference** case (`url` only): the URL is the sole carrier of the
  attachment payload and therefore must be an absolute, de-referenceable URL.

No alternative wording is discussed in the ticket; the resolution was
adopted as a Block-Vote item, indicating the working group treated it as
non-controversial editorial cleanup.

### Breaking Changes

No breaking changes. The change **relaxes** an existing SHALL by scoping
it to a narrower precondition (`data` absent + `url` present); existing
instances that satisfy the prior unconditional SHALL still satisfy the
narrower one. No element is added, removed, retyped, or rebound. No
invariants change.

1. **Update the `Attachment.url` Comments cell wording**
   - File: `source/datatypes/attachment.xml`
   - Action: In the `<Row>` that begins at line 925 (the row whose first
     cell contains `Attachment.url`), locate the Comments `<Cell
     ss:StyleID="s88">` at line 942. Replace the trailing sentence
     `"If a URL is provided, it SHALL resolve to actual data."` with:
     `"If the data element is not populated and the url element is
     populated, the url element SHALL be an absolute URL that can be
     de-referenced to the attachment's content."`

     - Preserve all preceding sentences in the same cell (the relative-URL,
       `cid:`, and Bundle-scoped-URI guidance must remain).

       ss:StyleID="s88"><Data ss:Type="String">…</Data><NamedCell

       `<Data>` changes.
     - Do not modify any other row, do not re-order rows, and do not
       touch the workbook-level styles or named ranges.
     - Do not edit any generated artefact (e.g., a publisher-derived
       StructureDefinition for Attachment, the rendered `datatypes.html`
