## Resolution Summary

**Resolution:** *Persuasive with Modification* — "This will be addressed by new designed CapabilityStatement using a code."

The ticket asked for a declarative way (dropped in R4 when `CapabilityStatement.acceptUnknown` was removed per [FHIR-14647](https://jira.hl7.org/browse/FHIR-14647)) for a server to state whether it accepts unknown extensions. The committee accepted the requirement, but modified the proposal away from the reporter's suggested dedicated `acceptUnknownExtensions` *boolean* and toward a single **code-typed** element on the (then-redesigned) CapabilityStatement that enumerates the possible behaviours. The enumeration is already materialised in-tree as the `unknown-content-code` CodeSystem/ValueSet with four codes — `no`, `extensions`, `elements`, `both` — which directly supersedes the earlier STU3 `acceptUnknown` coded element.

Because the label `CapabilityStatement2` was a working-title for the redesign that was subsequently folded back into the single `CapabilityStatement` resource (no `CapabilityStatement2` artefact exists in `HL7/fhir@db79dcd`), the practical implementation of the resolution is to re-introduce a single coded element on `CapabilityStatement`, bound (required) to the existing `unknown-content-code` ValueSet.

## Feature Proposal

### Problem Statement

In STU3, `CapabilityStatement.acceptUnknown` (bound to `unknown-content-code`) allowed a server to declare its behaviour when it encountered unknown elements or extensions. FHIR-14647 judged that element "awkward" and removed it in R4. Since then there is **no declarative way** in a CapabilityStatement instance for a server to communicate whether unknown extensions are silently dropped, returned in an error, or accepted — a gap that makes capability negotiation and client-side defensive coding harder, and that several interoperability profiles (notably those built on top of CapabilityStatement for conformance testing) have had to work around with custom extensions.

The STU3 binding artefacts (`unknown-content-code` CodeSystem and ValueSet) were kept alive in the core spec (both at `<version value="6.0.0"/>` in `source/capabilitystatement/`), anticipating re-introduction of the element — they are currently orphaned (not referenced from the `CapabilityStatement` StructureDefinition) and only appear in the XML narrative and OID registry.

### Proposed Change

Re-introduce a coded element on `CapabilityStatement` that declares the server's behaviour on receipt of unknown content, bound to the existing `unknown-content-code` ValueSet.

- **Element path:** `CapabilityStatement.acceptUnknown`
  - (The committee's resolution text doesn't fix the element name; `acceptUnknown` matches the STU3 name and the existing ValueSet title "Unknown Content Code". An alternative `acceptUnknownContent` is also defensible — to be confirmed in Open Questions.)
- **Cardinality:** `0..1`
- **Type:** `code`
- **Binding:** required → `http://hl7.org/fhir/ValueSet/unknown-content-code`
  - Values (already defined in `codesystem-unknown-content-code.xml`):
    - `no` — Neither Elements or Extensions
    - `extensions` — Unknown Extensions (accepts unknown extensions, rejects unknown elements)
    - `elements` — Unknown Elements (accepts unknown elements, rejects unknown extensions)
    - `both` — Unknown Elements and Extensions
- **Short:** `Behaviour on receipt of unknown content`
- **Definition:** "A code that indicates whether the application accepts unknown elements or extensions when reading resources. Applicable to CapabilityStatements of `kind = instance` or `kind = capability`; SHOULD NOT appear on `kind = requirements` unless the requirement is for a specific behaviour. Does **not** apply to modifier extensions, which are governed by the element modifier rules (§ Extensibility)."
- **Comments:** "Behaviour on receipt of *unknown modifier extensions* is defined by the modifier-extension rules and MUST NOT be overridden by this element."
- **Placement in the StructureDefinition:** between `CapabilityStatement.implementationGuide` (ends ~line 762) and `CapabilityStatement.rest` (starts line 763) in `source/capabilitystatement/structuredefinition-CapabilityStatement.xml`, alongside the other resource-level conformance facts (`format`, `patchFormat`, `acceptLanguage`, `implementationGuide`). This matches the "application-wide capabilities" grouping, not the per-`rest` grouping, because the behaviour is a property of the application, not of a specific interaction mode.
- **Invariants:** add `cpb-acc-1` — *"acceptUnknown should not be present when kind = requirements unless the CapabilityStatement is specifying required behaviour"* — expression `kind != 'requirements' or acceptUnknown.exists().not() or (acceptUnknown.exists() and text.exists())` (wording to be finalised by WG).
- **ValueSet housekeeping:** `valueset-unknown-content-code.xml` is currently marked `experimental=true`. Flip to `experimental=false` to match the (non-experimental) CodeSystem now that the value set is referenced again.
- **Narrative / prose:**
  - `source/capabilitystatement/capabilitystatement-introduction.xml` — add a paragraph explaining the new element and its relationship to modifier-extensions.
  - `source/capabilitystatement/capabilitystatement-notes.xml` — add guidance on how servers should populate it and how clients should interpret it during capability negotiation.
- **Examples:** add `acceptUnknown` to at least the canonical `capabilitystatement-example.xml` and `capabilitystatement-phr-example.xml` (value `extensions` is a reasonable default for demo servers).

### Design Rationale

1. **Single coded element over a boolean.** The committee explicitly chose a code over the reporter's proposed boolean. A code captures the four meaningful combinations (elements only, extensions only, both, neither) — a single boolean cannot.
2. **Reuse of `unknown-content-code`.** The CodeSystem and ValueSet are already in-tree at version 6.0.0, with OIDs carrying an `old`/`historical` identifier chain back to STU3, which means implementers who honoured the STU3 element have continuity.
3. **Placement at the resource root, not under `rest`.** Acceptance of unknown content is an application property, not a per-REST-mode property; placing it at the root mirrors its STU3 position (right before `rest`).
4. **Exclusion of modifier extensions.** Consistent with the reporter's original request and with FHIR's existing rule that unknown modifier extensions MUST cause rejection; a narrative "does not apply to modifier extensions" is sufficient.
5. **Alternatives rejected.** A separate `acceptUnknownExtensions` *boolean* (reporter's original proposal) was rejected because it would require a second boolean for unknown elements and would not cleanly express the "neither" state; it was also rejected implicitly by the resolution's "using a code" wording.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Only repository that changes. Hosts the `CapabilityStatement` StructureDefinition, the `unknown-content-code` terminology, and the capability-statement narrative pages.
- **Affected Files:**
  - `source/capabilitystatement/structuredefinition-CapabilityStatement.xml` — add `acceptUnknown` element (diff ~25 lines in `<differential>`), plus optional invariant entry.
  - `source/capabilitystatement/valueset-unknown-content-code.xml` — change `<experimental value="true"/>` → `<experimental value="false"/>`.
  - `source/capabilitystatement/capabilitystatement-introduction.xml` — add paragraph documenting the element.
  - `source/capabilitystatement/capabilitystatement-notes.xml` — add usage-and-interpretation notes.
  - `source/capabilitystatement/capabilitystatement-example.xml` — add `<acceptUnknown value="extensions"/>`.
  - `source/capabilitystatement/capabilitystatement-phr-example.xml` — add `<acceptUnknown value="..."/>`.
- **Change Scope:** Moderate — a single new coded element on a widely-implemented conformance resource, reusing existing terminology; narrative updates and example updates are straightforward.

### Breaking Changes

- **New optional element (`0..1`) at resource root — not a breaking wire change.** Existing instances remain valid and existing CapabilityStatements that omit `acceptUnknown` are unambiguous (interpreted as "unspecified").
- **Tooling / code-generation impact:** implementers that auto-generate classes from the StructureDefinition will need to regenerate; any downstream profile that forbade extra elements via a `closed` slicing will need to be re-checked.
- **Reinstated terminology is not breaking** — the ValueSet and CodeSystem already exist.
- **Not a breaking change, but behavioural:** clients that previously had to assume unknown-extension behaviour can now short-circuit to the declared value; existing servers that don't set the element keep the status-quo ambiguity.

### Related Specifications

- **Extensibility page (`source/extensibility.html` and siblings)** — the rule that unknown modifier extensions MUST cause rejection remains authoritative; the new element's definition must link to it and explicitly defer to it for modifier extensions.
- **`conformance-rules.html` / `capabilitystatement.html`** — top-level narrative pages may need a one-line mention that the new element supersedes the removed STU3 `acceptUnknown`.
- **`TerminologyCapabilities`** — has its own capability-surface model; no change here, but the WG should verify no parallel "accept unknown" concept needs to be mirrored.
- **OperationOutcome** — servers that choose `no` or `extensions` will likely raise `OperationOutcome` with `code = business-rule` or `structure`; no SD change, but documenting the expected error pattern in `capabilitystatement-notes.xml` is helpful.

### Related Tickets

- **FHIR-14647:** *CapabilityStatement.acceptUnknown is awkward* — the ticket that removed the element in R4. This ticket (FHIR-27932) explicitly reverses that decision with a redesigned, code-typed re-introduction. Mention FHIR-14647 in the new element's comments for historical traceability.
- **FHIR-29212:** *CapabilityStatement state support of async* — surfaces next to FHIR-27932 in the WG status report (per `fhir-augury` cross-references). Not conflicting; both are additive coded elements on `CapabilityStatement`. Implementation should coordinate on element ordering in the differential to avoid merge conflicts if both land in the same publishing cycle.

### Terminology Impact

- **No new terminology created.** Re-uses the already-in-tree `unknown-content-code` CodeSystem and ValueSet at `source/capabilitystatement/codesystem-unknown-content-code.xml` and `source/capabilitystatement/valueset-unknown-content-code.xml` (both at version 6.0.0).
- **Only change:** flip the ValueSet's `<experimental>` flag from `true` to `false` so that it matches the non-experimental CodeSystem and the fact that it is now normatively referenced from a resource StructureDefinition.
- **No UTG change required** — this terminology lives in core `HL7/fhir`, not in `HL7/UTG`.
