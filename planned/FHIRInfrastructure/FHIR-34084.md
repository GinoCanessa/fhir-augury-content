## Resolution Summary

**Resolution:** Persuasive.

The approved resolution is, verbatim:

> MnM will create a Confluence page
> (<https://confluence.hl7.org/display/FHIR/Mitigating+Breaking+Changes>)
> and will work on adding guidance to that page.

In other words, MnM (Modeling and Methodology) committed to authoring a
*living* Confluence document that enumerates the kinds of breaking
changes that may need to be applied to normative FHIR resources, and the
mitigation patterns the community can use to avoid actually breaking
conformant implementations (e.g., introduce a new parallel element,
deprecate the existing element and add a new one, use an extension,
introduce a "grouping" resource and reference it instead of changing
cardinality in place, etc.). The deliverable is the Confluence page
itself — there is **no normative spec change** required by this
resolution.

## Feature Proposal

### Problem Statement

FMG asked MnM to develop options — and criteria for choosing between
them — for situations where a *normative* FHIR resource encounters a
requirement that would, if applied directly, constitute a breaking
change. Examples called out in the original ticket include:

- An element that is `0..1` and now needs to repeat (cardinality
  expansion).
- An element whose type or binding needs to change in a non-additive
  way.
- A correction that would require renaming or removing an existing
  element.

Because resources marked Normative carry a backward-compatibility
commitment, the community needs a documented, agreed-upon menu of
mitigation patterns plus guidance on when each is appropriate. Without
such guidance, work groups facing these situations either stall or
make ad-hoc decisions that may diverge across resources.

### Proposed Change

This is a **methodology / documentation** ticket. The proposed
deliverable is the Confluence page
[*Mitigating Breaking Changes*](https://confluence.hl7.org/display/FHIR/Mitigating+Breaking+Changes),
maintained in the HL7 Confluence FHIR space.

The page should, at minimum, enumerate the following mitigation patterns
(drawn directly from the ticket text and the comments) and provide
selection criteria for each:

1. **Add a new parallel element** alongside the existing one (e.g.,
   `foo[x]` plus a new `foos[x]` for the repeating case), with the
   original retained for backward compatibility.
2. **Deprecate the existing element and add a new one** with the
   correct definition, marking the old as deprecated via the standard
   status / `obligation` / `deprecation` mechanisms.
3. **Use an extension** to convey the additional content rather than
   modifying the normative element.
4. **Create a "grouping" resource** that bundles the repeating content
   and reference it from the original element, so cardinality of the
   normative element does not have to change.
5. **Use modifier extensions / invariants** when behavioural changes
   are needed without touching the structural model.

For each pattern the Confluence page should document:

- The kind of breaking change it mitigates (cardinality expansion,
  type narrowing, semantic refinement, removal, etc.).
- Pros / cons (instance-size impact, search-parameter impact,
  validator behaviour, downstream profiling impact).
- Selection criteria — when this option is preferred and when it is
  not.
- A short worked example.

The page is explicitly intended to be a **living document**, updated
as MnM and other work groups encounter new normative-mitigation
scenarios.

### Design Rationale

The resolution disposition is **Persuasive**, but the disposition
deliberately does *not* mandate a single canonical mitigation pattern.
This is appropriate because:

- Different resource shapes (composition, references, value-typed
  elements) admit different mitigation patterns; codifying a single
  rule would be premature.
- The Normative commitment is enforced at the level of
  *implementation conformance*, not at the level of
  *modelling style*. A menu-of-options approach lets each work group
  choose the lowest-impact mitigation while still honouring the
  Normative contract.
- The Confluence page format allows the guidance to evolve with
  experience (the comments explicitly call this out: "*will become a
  living document over time*"), whereas baking the guidance into the
  spec narrative would freeze it on the ballot cycle.

The alternative considered in the comments (and partially adopted) was
to begin enumerating concrete cases on the existing
[*Mitigating Breaking Changes*](https://confluence.hl7.org/display/FHIR/Mitigating+Breaking+Changes)
Confluence page rather than create new tooling or new spec sections.
That alternative was selected.

## Implementation Plan

### Prerequisites

- Edit access to the HL7 Confluence FHIR space, in particular the
  page <https://confluence.hl7.org/display/FHIR/Mitigating+Breaking+Changes>.
- MnM agreement on the initial enumeration of mitigation patterns.

### Step-by-Step Tasks

This ticket has no GitHub-tracked tasks; the entire plan executes
against the Confluence page. Tasks are ordered by dependency.

#### HL7 Confluence (FHIR space)

1. **Confirm the Confluence page exists and is owned by MnM.**
   - Page: <https://confluence.hl7.org/display/FHIR/Mitigating+Breaking+Changes>
   - Action: Verify that the page exists in the FHIR Confluence
     space and that page restrictions allow MnM members to edit it.
     If it does not exist, create it.
   - Details: Add MnM to the page watchers so subsequent edits are
     announced.

2. **Add an introduction explaining the page's purpose.**
   - Page: *Mitigating Breaking Changes*
   - Action: Add an opening section that states (a) the page is a
     living MnM document, (b) it enumerates mitigation patterns for
     situations where a Normative resource would otherwise require a
     breaking change, and (c) it provides criteria for selecting
     between patterns.
   - Details: Link back to the Normative section of the FHIR spec
     and to FMG's Normative governance documentation.

3. **Enumerate the initial mitigation patterns.**
   - Page: *Mitigating Breaking Changes*
   - Action: Add one subsection per pattern listed in the Proposed
     Change above (parallel element, deprecate-and-add, extension,
     grouping resource, modifier extension). Each subsection should
     contain: trigger scenario, mechanism, pros, cons, worked
     example.
   - Details: Cardinality-expansion (`0..1` → `0..*`) is the
     scenario explicitly named in the ticket and should be the
     first worked example.

4. **Document the selection criteria.**
   - Page: *Mitigating Breaking Changes*
   - Action: Add a section that, for each kind of would-be breaking
     change (cardinality expansion, type narrowing, semantic
     refinement, removal, rename), lists which mitigation patterns
     are applicable and which are preferred.
   - Details: Frame the criteria around *implementer impact* —
     instance-size impact, search-parameter impact, validator
     behaviour, downstream profiling impact.

5. **Mark the page as a living document and define a maintenance
   cadence.**
   - Page: *Mitigating Breaking Changes*
   - Action: Add a footer noting the page is maintained by MnM and
     requesting that work groups encountering new
     normative-mitigation scenarios open a Jira ticket against MnM
     so the page can be extended.
   - Details: Reference FHIR-34084 as the originating ticket.

6. **Announce the page on the appropriate Zulip stream and to
   FMG.**
   - Action: Post a short announcement to the MnM Zulip stream and
     notify FMG that the requested guidance is now available, so
     FMG (the original requester) can close out the request on
     their side.
   - Details: Link the Confluence page and FHIR-34084.

### Testing Considerations

There is no software to test. The page should be reviewed at the
next MnM call; any work group that has previously hit a normative
mitigation situation should be invited to sanity-check the
enumerated patterns against their real-world experience.

When a future ticket proposes a structural change to a Normative
resource, treat the application of this guidance as the regression
test: if the work group cannot pick a mitigation from the
documented menu, the page is missing a pattern and should be
extended.

### Open Questions

- **Where should the page be referenced from the spec?** The
  resolution does not say whether the *Mitigating Breaking Changes*
  Confluence page should be linked from the FHIR spec narrative
  (e.g., from the Normative section) or remain a Confluence-only
  reference. MnM should decide.
- **Versioning of the guidance.** As a "living document", changes
  to the mitigation menu may retroactively affect how previously
  closed tickets would have been resolved. Whether to track a
  changelog on the page (or via Confluence revision history alone)
  is open.
- **Interaction with the `obligation` framework.** The
  *deprecate-and-add* pattern overlaps with the conformance
  obligation work; the page should clarify whether deprecation is
  signalled via element status, obligation extensions, or both.
