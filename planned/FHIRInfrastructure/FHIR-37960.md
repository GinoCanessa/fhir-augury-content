The committee reached the resolution **"Persuasive with Modification"** with the
following actionable text (verbatim from the resolution description):

> *"We will submit a THO task to add IRI as an additional standard identifier
> NamingSystem"*

The current resolution is the **re-opened** disposition recorded on

2022-12-12 disposition (Grahame Grieve / Josh Mandel: 16-0-0), which was

1. The change is delivered against **HL7 Terminology (THO)** — i.e., the
   `HL7/UTG` repository — by registering a new identifier `NamingSystem`,
   not by editing FHIR Core resources or extensions.
2. The metadata for the new entry must **explicitly state that IRI values
   cannot be used in `Identifier.system` or `Coding.system`**. They are
   only enabled for `Identifier.value` (with `Identifier.system =
   urn:ietf:rfc:3987`).

The reporter's preferred metadata for the new identifier system was:

- **Identifier:** Internationalized Resource Identifiers (IRIs) as defined
  by RFC 3987
- **URI / Identifier.system value:** `urn:ietf:rfc:3987`
- **OID:** to be registered if necessary
- **Type:** none
- **Comment:** As defined by RFC 3987, which also provides an algorithm
  for converting IRIs to URIs. Example:
  `https://hi.wikipedia.org/wiki/हृदय`

## Feature Proposal

### Problem Statement

The "Known Identifier Systems" registry (`identifier-registry.html` in
FHIR Core, now hosted on terminology.hl7.org) lists a Coding/Identifier
system value for **URIs** (`urn:ietf:rfc:3986`) but does not list one for
**IRIs** (RFC 3987). IRIs are the internationalised form of URIs that
permit Unicode characters; they are widely used by RDF tooling, modern
HTTP libraries, and browsers, particularly for non-Latin-script
identifiers.

Without a registered identifier system value for IRIs, FHIR implementers
who carry IRI-shaped identifiers must either (a) invent their own
`Coding.system` / `Identifier.system` URI for purely local use, or (b)
percent-encode the IRI into a URI and then reuse the existing
`urn:ietf:rfc:3986` registration. Both approaches are lossy: option (b)
discards the human-readable form (which is the entire point of an IRI),
and option (a) prevents interoperability.

The original ticket asked for IRI to be added next to URI on the FHIR
Core registry page. The resolution moved that work to THO so that the
entry can be maintained alongside the rest of the actively-curated
identifier `NamingSystem`s (the registry on the FHIR Core page itself
has been migrated and now just links to THO).

### Proposed Change

Add a new identifier-system `NamingSystem` to **HL7 Terminology (THO)**
(repository **`HL7/UTG`**) registering IRIs. The implementation has two
parts: the THO content change, plus a coordination touch-point in FHIR
Core.

**1. New THO `NamingSystem` (primary deliverable)**

A new `NamingSystem` resource describing the IRI identifier system, with
the following content:

- `name` / `id`: a stable identifier following THO's naming convention
  for identifier systems (e.g., `iri`).
- `kind`: `identifier`.
- `status`: `active`.
- `responsible`: HL7 International / FHIR Infrastructure (the work group
  that voted the resolution).
- `description`: "Internationalized Resource Identifiers (IRIs) as
  defined by RFC 3987."
- `usage` (free-text guidance): MUST state that IRI values are only
  permitted in `Identifier.value` when `Identifier.system =
  urn:ietf:rfc:3987`, and that they MUST NOT be used as the value of
  `Identifier.system` or `Coding.system`. Implementers needing to carry
  an IRI in those fields must percent-encode it into a URI per RFC 3987
  §3.1 and use `urn:ietf:rfc:3986`.
- `uniqueId`:
  - `{ type: uri, value: "urn:ietf:rfc:3987", preferred: true }`
  - (Optional) an OID `uniqueId` if HTA registers one; otherwise omit.
- Comment / publisher metadata: cite RFC 3987 and include the worked
  example `https://hi.wikipedia.org/wiki/हृदय` (and its percent-encoded
  URI form) so consumers see why the IRI form matters.

**2. FHIR Core touch-points (coordination only)**

The FHIR Core "Known Identifier Systems" page
(`source/identifier-registry.html` in `HL7/fhir`) has already been
migrated: it no longer contains an in-page list and instead points
readers to `http://terminology.hl7.org/identifiers.html`. Once the THO
`NamingSystem` is published, no edit to `identifier-registry.html`
itself is required. Verify after publication that the THO identifiers
page renders the new IRI entry; if it does, FHIR Core needs no further
change for this ticket.

### Design Rationale

- **Re-using `urn:ietf:rfc:3987` as the system value** is consistent
  with the existing pattern for the URI entry (`urn:ietf:rfc:3986`). It
  makes the IRI/URI pair symmetric and discoverable, and it relies on a
  stable IETF reference that is unlikely to be rescinded.
- **Adding the entry in THO rather than in FHIR Core** matches the
  current state of the world: the registry has already been migrated to
  terminology.hl7.org and the FHIR Core page explicitly says new
  identifier systems will be curated there. The resolution text ("submit
  a THO task") makes this explicit.
- **Hard restriction on `Identifier.system` / `Coding.system`** comes
  directly from the re-opened resolution and from the reporter's own
  comment (2022-12-01, JIRAUSER23302) which confirms the use case is
  identifier values only, with no need for IRI-shaped CodeSystem URIs.
  Encoding this restriction in the `NamingSystem.usage` text keeps it
  visible to implementers reading the registry rather than buried in
  the Jira history.
- **Leaving the OID unspecified** mirrors the reporter's "to be
  registered if necessary" note. Asking HTA to mint one is cheap if
  needed but should not block the initial publication.
- An alternative considered (and rejected by the resolution) was
  duplicating the IRI entry directly into the FHIR Core
  `identifier-registry.html` file. That was rejected because the page
  is being phased out as a content-bearing registry; future maintenance
  would diverge between the two locations.

  an IRI and confirm the new registry text gives implementers a clear
  reason why this is not permitted (this is documentation testing,
  not validator testing — there is no machine-checkable rule).
