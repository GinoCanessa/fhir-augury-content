## Resolution Summary

**Resolution:** Persuasive.

The resolution text ("Will remove this whole section:") directs the removal of
the entire `Messaging CapabilityStatement rules:` top-level bullet from the
CapabilityStatement notes — including all four nested sub-bullets:

1. The "interpretation of request and response depends on the mode…" bullet.
2. "If a request or response is not specified for an event, then no rules are
   made for it."
3. "Events that are not listed are not supported."
4. The MessageDefinition / `supportedMessage` / `event` paragraph that was the
   original trigger for the ticket.

The original ticket only asked to revisit the MessageDefinition bullet (since
MessageDefinition is moving to normative in R6), but the work group's approved
disposition is broader: retire the entire messaging-rules section rather than
rewrite only the MessageDefinition sentence.

## Feature Proposal

### Problem Statement

The CapabilityStatement page has a `Notes` section that lists per-mode rules.
The "Messaging CapabilityStatement rules:" sub-list contains a bullet stating
that MessageDefinition is "newly proposed and is still considered 'draft'" and
that `supportedMessage` "has not yet been reviewed by ballot." This wording is
obsolete: MessageDefinition is moving to normative in R6, `supportedMessage`
has been balloted, and the "may meet implementer needs better — feedback
welcome" framing is no longer appropriate for a normative specification.

Once that stale paragraph is removed, the three remaining bullets in the same
sub-list (request/response interpretation by `mode`, missing-request/response
semantics, and "events not listed are not supported") are either implicit in
the CapabilityStatement.messaging StructureDefinition / invariants or no longer
add value as standalone prose in the notes page — and per the applied
resolution the whole section is to be retired.

### Proposed Change

Pure narrative / prose edit in the FhirCore repository, no StructureDefinition
or terminology changes.

**File:** `source/capabilitystatement/capabilitystatement-notes.xml`

**Change:** Remove the entire `<li>Messaging CapabilityStatement rules: … </li>`
block that currently sits between the `RESTful CapabilityStatement rules`
block and the `Document CapabilityStatement rules` block — i.e., the outer
`<li>` spanning from the line beginning `<li>Messaging CapabilityStatement
rules:` (currently line 22) through its closing `</li>` (currently line 34).

Concretely, delete:

```xml
 <li>Messaging CapabilityStatement rules:
  <ul class="dense">
   <li>The interpretation of request and response depends on the mode. If the mode is sender,
   then request specifies what the application sends, and response specifies what it accepts.
   If the mode is "receiver", then this is reversed</li>
   <li>If a request or response is not specified for an event, then no rules are made for it</li>
   <li>Events that are not listed are not supported</li>
   <li>The <a href="messagedefinition.html">MessageDefinition</a> resource is newly proposed and is still considered 'draft'.
     The <code>supportedMessage</code> element can be used in place of the <code>event</code> and the work group believes it may meet
     implementer needs better, however because the new mechanism has not yet been reviewed by ballot, the older 'event' mechanism has
     been retained.  Implementers may use one or the other to define their capabilities.  Feedback is welcome.</li>
  </ul>
 </li>
```

After the edit, the surrounding bullet list should flow directly from
`RESTful CapabilityStatement rules` to `Document CapabilityStatement rules`
with no intervening Messaging item.

No other authored pages in `source/` reference the removed bullets by anchor
(the `<li>` is not tagged with an `<a name="…"/>`), so no inbound-fragment
cleanup is required. The MessageDefinition link itself remains valid in other
places in the spec; only this notes-page bullet is removed.

### Design Rationale

- The disposition is explicit ("Will remove this whole section"), so the
  implementation mirrors the resolution verbatim rather than attempting a
  rewrite.
- The removed prose was advisory framing from an earlier maturity window
  ("newly proposed," "not yet been reviewed by ballot," "feedback is welcome").
  With MessageDefinition moving toward Normative in R6 and `supportedMessage`
  already balloted, that framing is factually wrong for the upcoming version.
- The non-MessageDefinition sub-bullets (mode interpretation, missing
  request/response, "events not listed are not supported") are restatements
  of what is already captured by the CapabilityStatement.messaging
  StructureDefinition and its element definitions; removing them from the
  notes page reduces duplication rather than losing normative content.
- Alternative considered: edit only the final (MessageDefinition) bullet.
  Rejected by the work group — the resolution text explicitly extends the
  scope to the whole messaging-rules list.
