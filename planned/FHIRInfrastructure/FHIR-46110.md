## Resolution Summary

Resolution: **Not Persuasive with Modification**.

Verbatim resolution description:

> We will add guidance to `CapabilityStatement.rest.searchParam` and
> `CapabilityStatement.rest.resource.searchParam` that says:
>
> If documenting a parameter that is not a 'true' SearchParameter (such
> as `_format`, `_count`, etc.), the `.definition` element SHALL be
> omitted and the type SHOULD be `'special'`.

The reporter's stronger ask — to *remove* the existing comment that
tells implementers to include control search parameters such as
`_sort` / `_count` in `searchParam` — was **not** accepted. The
committee instead retained the existing prose and added a normative
sentence (SHALL omit `.definition`, SHOULD set `type = 'special'`) that
reconciles the apparent contradiction with the search-results-parameters
guidance elsewhere in the spec.

The change is text-only on two `<comment value="…"/>` attributes on the
core `CapabilityStatement` `StructureDefinition`. There are no
cardinality, type, binding, invariant, or search-parameter changes.

## Feature Proposal

### Problem Statement

In FHIR R5, the `comment` on
`CapabilityStatement.rest.resource.searchParam` (and the parallel
comment on `CapabilityStatement.rest.searchParam`) tells implementers
that the listed search parameters "should include the control search
parameters such as `_sort`, `_count`, etc.". Elsewhere in the
specification, search **result** parameters (`_sort`, `_format`,
`_pretty`, `_total`, …) are explicitly described as *not* being real
search parameters and as not having `SearchParameter` definitions, so
they cannot be slotted into `searchParam` the way a "true" search
parameter (with a canonical `SearchParameter.definition`) can.

Implementers (see the Zulip thread *"R5 CapabilityStatement Canonical
URL does not resolve"* on stream `#IG-creation`, and the linked
attachments in the comments) hit this contradiction in practice: they
either omit `_include` / `_revinclude` / `_count` from the
`CapabilityStatement` and lose the ability to advertise support, or they
include them and get publisher / QA errors because there is no
resolvable `SearchParameter` canonical to point `.definition` at.

### Proposed Change

Edit the core `CapabilityStatement` `StructureDefinition` so that the
two `searchParam` element comments document the escape valve that lets
control / result parameters be advertised without a `.definition`
canonical:

- **Element `CapabilityStatement.rest.resource.searchParam`** — append
  to the existing `comment` value (after the existing
  `…linked SearchParameter definitions.` sentence, separated by a
  newline `&#xA;`):

  > If documenting a parameter that is not a 'true' SearchParameter
  > (such as `_format`, `_count`, etc.), the `.definition` element
  > SHALL be omitted and the type SHOULD be `'special'`.

- **Element `CapabilityStatement.rest.searchParam`** — append the
  identical sentence to that element's `comment` value, again separated
  by `&#xA;`.

No other element is modified. `min` / `max` / `type` /
`contentReference` / `binding` for both elements stay exactly as
authored. The only edit is to the two `comment value="…"` strings on
the `<element>` tags whose `id` attributes are
`CapabilityStatement.rest.resource.searchParam` and
`CapabilityStatement.rest.searchParam`.

The new prose is normative-style ("SHALL" / "SHOULD") so the bound
value set `http://hl7.org/fhir/ValueSet/search-param-type` (required
binding) must continue to contain the `special` code — it already does
(see `valueset-system-restful-interaction.xml` neighbours and
`codesystem-search-param-type.xml`), so no terminology work is needed.

### Design Rationale

- **Minimal, additive prose change.** The committee chose to clarify
  rather than restructure: the existing guidance to advertise control
  parameters in `searchParam` is preserved (because real-world
  CapabilityStatements do legitimately need to communicate that, e.g.,
  `_count` is supported on a given resource endpoint), but the new
  sentence makes the semantics consistent with the rest of the search
  module — control / result parameters do not have a `SearchParameter`
  resource, so `.definition` (canonical → `SearchParameter`) is not
  applicable, and `type` is forced to the `special` code in the
  required-bound `search-param-type` value set.
- **Reuses existing `special` slot in `SearchParamType`.** The
  `special` code already exists in `codesystem-search-param-type.xml`
  precisely for parameters that do not fit the `number | date | string
  | token | reference | composite | quantity | uri` axes; this
  resolution simply documents that it is the right code for control /
  result parameters.
- **No invariant added.** A formal invariant that asserts
  "`type = 'special' implies definition is absent`" was *not* added.
  The wording is `SHALL be omitted` / `SHOULD be 'special'`, which is
  prose-level conformance only. This matches the FHIR-Infrastructure
  pattern of preferring narrative guidance for cross-element shape
  rules that are easy to misread as machine-checkable but in practice
  have legitimate exceptions (e.g., implementation-defined "special"
  parameters that *do* have a custom `SearchParameter`).
- **Alternative considered (and rejected):** Removing the original
  "should include control search parameters" sentence outright (the
  reporter's preferred fix). This would have left implementers without
  any guidance on how / where to advertise support for control
  parameters in `CapabilityStatement`, and was rejected as
  *Not Persuasive*.

### Breaking Changes

No breaking changes identified. The change is narrative-only:

- No element is added, removed, renamed, or retyped.
- No cardinality is changed (`min` / `max` are unchanged on both
  affected elements).
- The required-bound `SearchParamType` value set
  (`http://hl7.org/fhir/ValueSet/search-param-type`) already includes
  the `special` code; no terminology change is needed.
- No invariant is added or modified. `cpb-12`
  (`searchParam.select(name).isDistinct()`) is untouched and still
  applies to entries authored under either element.

The new sentence does introduce a normative `SHALL` ("the `.definition`
element SHALL be omitted") for control / result parameters. Any
existing `CapabilityStatement` instance that lists, e.g., `_count` as a
`searchParam` *with* a `definition` canonical is now non-conformant in
prose terms; however, since control parameters never had a resolvable
`SearchParameter` canonical to begin with, no real-world conformant
instance is expected to be affected.

### Related Specifications

  — the resource pointed at by `.definition` (target profile

  result parameters do not have `SearchParameter` resources.
- The bound code system / value set
  `codesystem-search-param-type.xml` /
