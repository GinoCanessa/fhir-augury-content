## Resolution Summary

The committee resolution (`Persuasive with Modification`) approves three concrete
edits to the `ExampleScenario` resource definition:

1. **Add a new element** `ExampleScenario.process.step.operation.definition`,
   cardinality `0..1`, type `canonical(OperationDefinition)`, with:
   - **short:** `Operation invoked`
   - **definition:** *"If this operation represents the invocation of a FHIR
     operation, points to the canonical definition of that operation"*
2. **Add a constraint** to `ExampleScenario.process.step.operation` stating
   that `definition` is only permitted when `type.code = 'operation'`.
3. **Add a `rationale`** (i.e., a `requirements` description) on
   `ExampleScenario.process.step.operation.type` reading: *"This binding is
   extensible to provide support for non-FHIR interactions as well."*

The pattern intentionally mirrors `ExampleScenario.actor.definition →
canonical(ActorDefinition)`, which already exists in the resource.

## Feature Proposal

### Problem Statement

`ExampleScenario` already lets an actor be tied to a formal `ActorDefinition`
via `ExampleScenario.actor.definition` (`0..1 canonical(ActorDefinition)`).
There is, however, no equivalent way to link a step's *operation* to its
formal definition. When the step represents an invocation of a real FHIR
operation, authors and tooling cannot point at the canonical
`OperationDefinition` that describes the operation's parameters and behaviour.
The reporter (Anja Schwab) raised the inconsistency and proposed adding a
`canonical(OperationDefinition)` element that parallels the actor pattern.

### Proposed Change

Make a single, structural edit to the `ExampleScenario` StructureDefinition
plus two metadata additions on neighbouring elements:

**A. New element**

- **Path / id:** `ExampleScenario.process.step.operation.definition`
- **Cardinality:** `0..1`
- **Type:** `canonical` with `targetProfile =
  http://hl7.org/fhir/StructureDefinition/OperationDefinition`
- **Short:** `Operation invoked`
- **Definition:** *"If this operation represents the invocation of a FHIR
  operation, points to the canonical definition of that operation."*
- **Condition:** reference the new constraint key (see B).

Element XML (mirrors the existing `ExampleScenario.actor.definition` block at
lines 551–563 of the SD):

```xml
<element id="ExampleScenario.process.step.operation.definition">
  <path value="ExampleScenario.process.step.operation.definition"/>
  <short value="Operation invoked"/>
  <definition value="If this operation represents the invocation of a FHIR operation, points to the canonical definition of that operation."/>
  <min value="0"/>
  <max value="1"/>
  <type>
    <code value="canonical"/>
    <targetProfile value="http://hl7.org/fhir/StructureDefinition/OperationDefinition"/>
  </type>
  <condition value="exs-25"/>
</element>
```

Insert this element immediately after
`ExampleScenario.process.step.operation.type` (current line 1015) so that the
`definition` element sits adjacent to the `type` it constrains, and re-order
the existing `title`, `initiator`, `receiver`, `description`,
`initiatorActive`, `receiverActive`, `request`, `response` children below it.

**B. New constraint on `ExampleScenario.process.step.operation`**

The next available key in this SD is `exs-25` (existing keys run through
`exs-24`). Add it as a child `<constraint>` of the
`ExampleScenario.process.step.operation` BackboneElement (current lines
978–1001):

```xml
<constraint>
  <key value="exs-25"/>
  <severity value="error"/>
  <human value="definition is only permitted if type.code = 'operation'"/>
  <expression value="definition.exists() implies type.code = 'operation'"/>
</constraint>
```

Also add `<condition value="exs-25"/>` on the new `definition` element (already
shown above) so the constraint is surfaced in the per-element view.

> Note: the literal token `'operation'` is the value the resolution text
> calls out. The element under constraint is `type` whose datatype is
> `Coding`, so the FHIRPath expression must reference `type.code` (the
> `Coding.code` element). If reviewers prefer to also pin the system, the
> expression can be tightened to `type.system =
> 'http://hl7.org/fhir/restful-interaction' and type.code = 'operation'` —
> see Open Questions.

**C. Rationale (`requirements`) on
`ExampleScenario.process.step.operation.type`**

Add a `<requirements>` child to the existing
`ExampleScenario.process.step.operation.type` element (current lines
1002–1015), capturing the resolution's "rationale" wording. In the FHIR
StructureDefinition vocabulary, `ElementDefinition.requirements` is the
appropriate slot for a "why this is the way it is" rationale on an element.

```xml
<requirements value="This binding is extensible to provide support for non-FHIR interactions as well."/>
```

Place it after `<definition>` and before `<min>`, consistent with element
ordering elsewhere in the SD.

### Design Rationale

- **Parallel to `actor.definition`.** The reporter's argument and the
  resolution both explicitly mirror the existing
  `ExampleScenario.actor.definition` → `canonical(ActorDefinition)` pattern.
  Reusing the same shape keeps `ExampleScenario` internally consistent and
  matches the precedent set when `ActorDefinition` was introduced.
- **`canonical` rather than `Reference`.** `OperationDefinition` is a
  canonical resource and the existing `actor.definition` likewise uses
  `canonical(ActorDefinition)`. A `Reference` would inappropriately tie the
  step to an instance.
- **Cardinality `0..1`.** `definition` is metadata that further specifies an
  already-modelled step; making it required would break every existing
  ExampleScenario instance and force authors to find or invent an
  OperationDefinition for non-FHIR or generic flows.
- **Constraint scoped to the operation backbone.** The constraint must be on
  the parent `ExampleScenario.process.step.operation` element rather than on
  `definition` itself, because it expresses a relationship between two
  sibling sub-elements (`definition` and `type`). All other multi-element
  invariants in this SD (`exs-17`, `exs-18`, `exs-19`, `exs-22`, …) follow
  the same placement convention.
- **`requirements` is the canonical "rationale" slot.** The resolution uses
  the word "rationale", but `ElementDefinition` already has a dedicated
  `requirements` element for design rationale text; reusing it avoids
  introducing an extension just to carry a sentence of prose.

### Breaking Changes

The change is **additive and non-breaking**:

- The new `definition` element is `0..1`, so existing instances remain valid.
- The new constraint `exs-25` is `definition.exists() implies type.code =
  'operation'`. Because no existing instance can already populate the new
  `definition` element, `definition.exists()` is necessarily `false` for
  every legacy instance, and the implication is vacuously satisfied. New
  authors who *do* set `definition` while leaving `type.code` non-`operation`
  will see a validation error — that is the intent of the constraint.
- Adding `requirements` text to `…operation.type` is documentation only; it
  does not affect conformance.

No breaking changes identified.

### Testing Considerations

- **Resource validation:** Re-validate every existing `ExampleScenario`
  example (`examplescenario-example.xml`,
  `examplescenario-example-laborder.xml`) and any examples that live under
  `source/examplescenario/invariant-tests/` to confirm the additive change
  does not break them and that `exs-25` fires only when expected.
- **Invariant edge cases:**
  1. `operation` with `definition` set and `type.code = 'operation'` →
     valid.
  2. `operation` with `definition` set and `type.code = 'create'` (or any
     other code) → must error on `exs-25`.
  3. `operation` with `definition` set and `type` absent entirely →
     `type.code` is empty, `type.code = 'operation'` is `false`, so
     `exs-25` errors. This is the intended (strict) behaviour because the
     resolution wording is unambiguous; revisit only if Open Questions
     point otherwise.
  4. `operation` with no `definition` → vacuously valid.
- **Publisher / IG build:** Run the standard `publish.sh` flow for the
  ExampleScenario resource page to confirm the new element appears in the
  formal definitions table and that the `requirements` text is rendered on
  the `…operation.type` row.
- **Interoperability:** Existing producers that have never populated
  `definition` continue to round-trip cleanly; consumers that do not
  recognise the new element MUST ignore it per FHIR's standard
  forward-compatibility rules.

### Open Questions

1. **Constraint precision on `type.code`.** The resolution wording is
   `type.code = 'operation'`. The bound ValueSet
   (`examplescenario-operation-codes`) is *extensible*, so a user could in
   principle pick a non-canonical `code` value `'operation'` from a
   different system. If WG wants to be strict, tighten `exs-25`'s
   expression to also pin `type.system` (e.g.,
   `http://hl7.org/fhir/restful-interaction`). Confirm with FHIR
   Infrastructure before merge.
2. **Constraint behaviour when `type` is absent.** As written, `exs-25`
   errors if `definition` is set but `type` is empty. This is the literal
   reading of the resolution text. If WG intends `definition` to be
   permissible whenever the step is *intended* to represent an operation
   (regardless of whether `type` was filled in), the expression should
   become `definition.exists() implies (type.empty() or type.code =
   'operation')`. Flag for WG sign-off.
3. **Search parameter.** Should
   `ExampleScenario.process.step.operation.definition` be exposed as a
   search parameter (`reference` / `composite`) so registries can find
   ExampleScenarios that exercise a given OperationDefinition? Out of
   scope of the resolution as written, but a natural follow-up.
4. **Briefing scope.** The cached briefing's "Ticket-Relevant Paths" block
   is scoped to FHIR-42980 (Mapping Language) rather than this ticket. The
   general FhirCore facts (authoring root, generated areas, build system,
   gotchas) still apply, and the ExampleScenario paths cited above were
   verified directly against clone HEAD `db79dcd`. No re-run of
   `repo-analysis` is required for this plan.
