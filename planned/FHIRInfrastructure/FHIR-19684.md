## Resolution Summary

**Resolution:** Persuasive with Modification.

The committee agreed to add a single GraphDefinition example showing the
implementation of `Patient/$everything` on a hypothetical server. The
second half of the original request — defining a new operation that lets
servers expose a server-specific `$everything` GraphDefinition — was
explicitly **out of scope** for this resolution; the committee saw no
demand from implementers and chose to wait until at least one server
project asked for it.

Resolution description (verbatim):

> We will create a GraphDefinition example showing the implementation of
> Patient/$everything on a hypothetical server. We will not define an
> operation for retrieving a server-specific $everything GraphDefinition
> until/unless servers that implement $everything indicate a desire to
> expose such an operation.

The most recent comment (mckenzie, 2022-09-23) further sequences the
work: this example is **gated on FHIR-25474** ("Update GraphDefinition
resource to allow it to describe a graph"). Until Lloyd's
GraphDefinition rework lands, authoring an `Patient/$everything` example
in the new GraphDefinition shape is premature.

## Feature Proposal

### Problem Statement

`Patient/$everything` is one of the most widely-implemented FHIR
operations, but the specification gives no machine-readable description
of what a server is expected (or allowed) to return. Implementers
currently rely on prose plus the request/response example pair under
`source/patient/` (`$everything-request.txt`, `$everything-response.txt`)
to understand the operation; there is no GraphDefinition instance
demonstrating how the traversal could be expressed declaratively. As a
result:

- Newcomers to GraphDefinition have no end-to-end, recognisable example.
- Server authors cannot point clients at a normative-style document that
  states "this is the graph my `$everything` walks".
- The GraphDefinition resource itself lacks a high-profile worked
  example anchored to a familiar operation.

### Proposed Change

Add **one GraphDefinition example** instance — call it
`GraphDefinition/patient-everything-example` (final id to be confirmed
during authoring) — that models a hypothetical server's implementation
of `Patient/$everything`. The example must:

1. Use `Patient` as the start node.
2. Walk the typical `$everything` reverse-references: `Encounter` (via
   `Encounter.subject`), `Observation` (`Observation.subject`),
   `Condition` (`Condition.subject`), `Procedure` (`Procedure.subject`),
   `MedicationRequest` (`MedicationRequest.subject`),
   `AllergyIntolerance` (`AllergyIntolerance.patient`),
   `DiagnosticReport` (`DiagnosticReport.subject`), and any other
   resources actually referenced by the existing
   `source/patient/$everything-response.txt` Bundle (the example must
   stay consistent with the response example already shipped).
3. Include forward links from `Encounter` to `Practitioner` /
   `Location` (or whichever participants the response example shows) so
   the graph demonstrates both `reverse` and forward traversal.
4. Carry narrative copy that **explicitly states the example is one
   hypothetical implementation**, mirroring the language of the
   resolution; the spec must not imply normativity.
5. Be referenced from the `Patient/$everything` operation page so
   readers actually find it.

The example must be authored against whatever shape GraphDefinition has
**after** FHIR-25474 lands. Authoring it before FHIR-25474 will produce
an example using a structure that is about to be replaced.

The second half of the original request (a server-specific
`$everything-graph` operation) is **not implemented** per the
resolution.

### Design Rationale

- Adding an example is the smallest, lowest-risk change that satisfies
  the request: no resource shape changes, no new operation, no
  normative consequences.
- Anchoring the GraphDefinition to `Patient/$everything` gives
  implementers an example built on the most familiar operation in core
  FHIR, maximising educational value.
- Deferring the "server returns its own GraphDefinition" capability
  follows YAGNI plus the committee's explicit reasoning: there is no
  implementer demand. If demand emerges, a separate ticket should add a
  dedicated operation rather than retro-fitting it onto this example.
- Sequencing behind FHIR-25474 (called out by mckenzie 2022-09-23)
  prevents wasted re-authoring work — the new example should be born in
  the post-rework GraphDefinition idiom.

## Implementation Plan

### Prerequisites

1. **FHIR-25474 must be applied first.** Do not author this example
   against the pre-rework GraphDefinition shape — it will be discarded.
2. **A current `repo-analysis` briefing for `HL7/api-incubator-ig`
   must exist** before implementation begins, so that authoring root,
   recipe names, and exact file paths are grounded rather than guessed.
3. Confirm in the api-incubator-ig clone that the migrated
   `Patient/$everything` OperationDefinition is present (it is no
   longer in `HL7/fhir`); the example will reference it by canonical URL.

### Step-by-Step Tasks

#### HL7/api-incubator-ig (primary)

1. **Author the GraphDefinition example for `Patient/$everything`**
   - File: `<authoring-root>/GraphDefinition-patient-everything-example.<json|xml|fsh>`
     (path/extension to be taken from the IG's briefing once produced).
   - Action: Create a new GraphDefinition resource in the post-FHIR-25474
     idiom whose start is `Patient` and whose links walk the same set
     of resources currently appearing in
     `cache/github/repos/HL7_fhir/clone/source/patient/$everything-response.txt`
     (so the example and the existing response Bundle stay
     consistent). Include reverse links from `Encounter`,
     `Observation`, `Condition`, `Procedure`, `MedicationRequest`,
     `AllergyIntolerance`, `DiagnosticReport` (and any other
     resources in the response), plus forward links for participants
     (`Practitioner`, `Location`).
   - Details:
     - Pick a stable canonical URL under the api-incubator-ig
       canonical base.
     - Set `status = draft` (or whatever the IG's house style for
       examples is — confirm from its briefing).
     - Narrative must explicitly state this is **one hypothetical
       server's implementation** of `Patient/$everything`, mirroring
       the resolution language; do not imply normativity.
     - Where the OperationDefinition for `Patient/$everything` (now
       hosted in this IG) carries an `inputParameters` /
       `outputParameters` reference style, link the example via
       whichever element FHIR-25474 provides for binding a graph to
       an operation.

2. **Cross-link the example from the `Patient/$everything` operation page**
   - File: `<authoring-root>/<page-that-introduces-Patient-$everything>`
     (e.g., the operation's `.md`/`.xml` description page).
   - Action: Add a short paragraph: "An example GraphDefinition
     showing one hypothetical server's implementation of this
     operation is available at [GraphDefinition/patient-everything-example](…)."
   - Details: Make it explicit that the example is illustrative, not
     normative.

3. **Cross-link the example from the GraphDefinition resource page(s)**
   - File: `<authoring-root>/<graphdefinition-page>` (the IG's main
     page or example index for GraphDefinition).
   - Action: Add the new example to the resource's example list/table
     so it is discoverable from the GraphDefinition page itself.

4. **Validate via the IG publisher**
   - Action: Run the IG publisher (per the api-incubator-ig briefing's
     build instructions) and confirm the new GraphDefinition
     validates, the narrative renders, and the cross-links resolve.

#### HL7/fhir (secondary, optional)

5. **Verify whether `Patient/$everything` is still described in core**
   - File(s) to inspect:
     `cache/github/repos/HL7_fhir/clone/source/patient/patient.html`,
     `cache/github/repos/HL7_fhir/clone/source/patient/patient-introduction.xml`,

     `cache/github/repos/HL7_fhir/clone/source/patient/list-Patient-operations.xml`

     narrative / prose" recipe; no publisher config or schema changes.

### Testing Considerations

- Re-run the IG publisher for `HL7/api-incubator-ig` and visually
  inspect the rendered GraphDefinition example page, the operation
  page, and the GraphDefinition resource page.
- If the FhirCore cross-link is added, run the FhirCore publisher

  canonical URL.

  *illustrative*; reviewers should not be able to mistake it for a
