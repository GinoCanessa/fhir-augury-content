## Resolution Summary

The resolution (Persuasive) agrees that the type list on
`DataRequirement.valueFilter.value[x]` (currently `dateTime | Period |
Duration`) was a copy/paste error from `dateFilter.value[x]` and contradicts
the element's own definition ("Value filters specify additional constraints
on the data for elements *other than* code-valued or date-valued"). The
approved fix mirrors the cqf-valueFilter extension that this element is
based on, with `RatioRange` added by Grahame Grieve in the FHIR-I Zulip
discussion:

> We will update the value[x] types to: **boolean, integer, string,
> Quantity, Range, Ratio, and RatioRange**.
>
> In addition, we will define a constraint to ensure that either a path or
> searchParam must be provided, but not both.

The new constraint mirrors the existing `drq-1` (codeFilter) and `drq-2`
(dateFilter) invariants — `path.exists() xor searchParam.exists()` — and
attaches to `DataRequirement.valueFilter`.

## Feature Proposal

### Problem Statement

`DataRequirement` provides three sibling filter elements — `codeFilter`,
`dateFilter`, and `valueFilter` — which collectively let a data
requirement narrow the set of returned instances along three orthogonal
axes (terminology, time, and arbitrary value). The first two are
internally consistent: `codeFilter.value*` slots are coded, `dateFilter.
value[x]` is `dateTime | Period | Duration`, and each has a `path xor
searchParam` invariant (`drq-1`, `drq-2`).

`valueFilter` was added to cover *everything else* — non-code, non-date
constraints (e.g., observed value > 7) — but its `value[x]` was authored
with the same `dateTime | Period | Duration` set as `dateFilter`, which
makes it effectively useless for its stated purpose. It also lacks the
`path xor searchParam` invariant that the other two filters carry.

The Zulip thread on `fhir/infrastructure-wg` (topic
"DataRequirement.valueFilter (FHIR-53984)", 2026-01-27) confirmed this
was a copy/paste mistake: the cqf-valueFilter extension that
`DataRequirement.valueFilter` is modelled on declares
`Boolean | Integer | String | Quantity | Ratio | Range`. Grahame Grieve
added `RatioRange` to keep the modern set complete; Bryn Rhodes accepted
that and noted he would file a follow-up to align the extension.

DanielSchuster's original ticket comment also flagged the missing
constraint analogous to `drq-1` / `drq-2`.

### Proposed Change

Two surgical edits to the `DataRequirement` complex datatype, both made
in the bespoke spreadsheet authoring file
`source/datatypes/datarequirement.xml`.

**Change A — Replace `valueFilter.value[x]` types.**

- Element path: `DataRequirement.valueFilter.value[x]`
- Cardinality: `0..1` (unchanged)
- Type before: `dateTime | Period | Duration`
- Type after: `boolean | integer | string | Quantity | Range | Ratio |
  RatioRange`
- Short label after: "The value of the filter, as a boolean, integer,
  string, Quantity, Range, Ratio, or RatioRange value"
- Definition after: "The value of the filter." (current text is
  acceptable; the type-specific narrative in the existing description
  ("If period is specified… If dateTime is specified…") was carried over
  from `dateFilter` and should be dropped — it does not match the new
  type set.)

**Change B — Add a new `path xor searchParam` invariant on
`DataRequirement.valueFilter`.**

Add a new row to the `Invariants` worksheet of
`source/datatypes/datarequirement.xml`, following the template of
existing rows id=1 (codeFilter) and id=2 (dateFilter):

| Field | Value |
|-|-|
| Id | `3` |
| Name | `value - filter or searchParam` |
| Severity | `error` |
| Context | `DataRequirement.valueFilter` |
| English | `Either a path or a searchParam must be provided, but not both` |
| Expression | `path.exists() xor searchParam.exists()` |
| XPath | `(exists(f:path) and not(exists(f:searchParam))) or (not(exists(f:path)) and exists(f:searchParam))` |

Then wire the invariant ID `3` into the constraint-reference column of
the `DataRequirement.valueFilter.path` and
`DataRequirement.valueFilter.searchParam` rows on the `Data Elements`
worksheet (mirroring how `dateFilter.path` and `dateFilter.searchParam`
each carry the value `2` in that column today).

### Design Rationale

- **Type set parity with cqf-valueFilter.** The
  `cqf-valueFilter` extension
  (`https://hl7.org/fhir/extensions/StructureDefinition-cqf-valueFilter.html`)
  is the documented R4 backport of this element and the canonical
  reference for which types make sense for non-code, non-date value
  filtering. Aligning the core element to that set restores the original
  intent. Adding `RatioRange` (Grahame's amendment) keeps DataRequirement
  consistent with FHIR's modern Quantity-family pattern (`Range`,
  `Ratio`, `RatioRange` typically appear together where numeric ranges
  are admitted).
- **Invariant parity with sibling filters.** `codeFilter` and
  `dateFilter` already require exactly one of `path` or `searchParam`.
  `valueFilter` was always intended to follow the same shape; the
  missing constraint is an authoring oversight, not a deliberate
  loosening (confirmed by Bryn Rhodes on Zulip).
- **Out-of-scope amendments.** The cqf-valueFilter extension itself does
  not yet list `RatioRange`; Bryn committed on Zulip to file a separate
  ticket against `HL7/fhir-extensions` to add it. That work is not
  included here.

  - `source/datatypes/datarequirement.xml` —

      `DataRequirement.valueFilter.value[x]` from `dateTime|Period|
      Duration` to `boolean|integer|string|Quantity|Range|Ratio|
      RatioRange`, refresh its short label and definition to drop the
      stale "Period/DateTime/Duration" wording, and set the

- **Change Scope:** Minor (single authoring file, two coupled edits).

### Breaking Changes

This is a **breaking change** in the strict FHIR sense, but in a
direction that improves rather than restricts validity for almost all
real instances:

- **Type set change on `DataRequirement.valueFilter.value[x]`**:
  removes `dateTime`, `Period`, and `Duration` and adds `boolean`,
  `integer`, `string`, `Quantity`, `Range`, `Ratio`, `RatioRange`. Any
  pre-existing instance that populated `valueDateTime`, `valuePeriod`,
  or `valueDuration` will no longer validate; the Zulip discussion
  notes there is no evidence such instances exist in the wild ("the
  fact that the type list is so useless but no one has commented until
  this task suggests that it's not really getting used"). Date-valued
  filtering remains expressible via the sibling `dateFilter`.
- **New invariant on `DataRequirement.valueFilter`**: any existing
  instance that supplies neither, or both, of `valueFilter.path` and
  `valueFilter.searchParam` will newly fail validation. Same caveat
  about real-world usage applies.

No other elements, cardinalities, or bindings change.

## Implementation Plan

### Prerequisites

- An installation of Excel (or another SpreadsheetML 2003 XML editor)
  capable of round-tripping
  `source/datatypes/datarequirement.xml` without re-flowing the file's
  XML structure. Editing this file by hand in a text editor is
  error-prone and discouraged — see the briefing gotcha.
- No upstream ticket dependencies. The `HL7/fhir-extensions` follow-up
  for `cqf-valueFilter` is independent and can land later.

### Step-by-Step Tasks

#### HL7/fhir

1. **Replace the type list on `DataRequirement.valueFilter.value[x]`.**
   - File: `source/datatypes/datarequirement.xml`
   - Action: On the `Data Elements` worksheet, locate the row whose
     element path is `DataRequirement.valueFilter.value[x]` and replace
     the type cell value `dateTime|Period|Duration` with
     `boolean|integer|string|Quantity|Range|Ratio|RatioRange`.
   - Details: Cardinality stays `0..1`. The type cell uses the
     `pipe-delimited` format already in use for choice elements
     elsewhere in the workbook (e.g., `dateFilter.value[x]`).

2. **Refresh the short label and definition for
   `DataRequirement.valueFilter.value[x]`.**
   - File: `source/datatypes/datarequirement.xml`
   - Action: In the same row, replace the short label
     `"The value of the filter, as a Period, DateTime, or Duration
     value"` with `"The value of the filter"` (or, more descriptively,
     `"The value of the filter, as a boolean, integer, string,
     Quantity, Range, Ratio, or RatioRange value"`). Replace the
     definition text — currently begins
     `"The value of the filter. If period is specified… If dateTime is
     specified…"` — with `"The value of the filter."` Drop the
     type-specific clauses that referenced Period/dateTime; they no
     longer apply.
   - Details: Keep wording terse and consistent with the parallel
     short label on `codeFilter.code` etc. Do not add comparator-
     related guidance here — `value[x]` does not use the value-filter
     comparator binding.

3. **Add the new "value - filter or searchParam" invariant.**
   - File: `source/datatypes/datarequirement.xml`
   - Action: On the `Invariants` worksheet, append a new row directly
     below the existing `dateFilter` invariant (`id=2`) with the
     following column values:
     - **Id:** `3`
     - **Name:** `value - filter or searchParam`
     - **Severity:** `error`
     - **Context:** `DataRequirement.valueFilter`
     - **English:**
       `Either a path or a searchParam must be provided, but not both`
     - **Expression:** `path.exists() xor searchParam.exists()`
     - **XPath:** `(exists(f:path) and not(exists(f:searchParam))) or
       (not(exists(f:path)) and exists(f:searchParam))`
   - Details: The id `3` produces invariant key `drq-3` once the
     publisher prefixes it (consistent with `drq-1`, `drq-2`). Use the
     identical English/Expression/XPath wording from row `id=2` to
     guarantee semantic parity.

4. **Wire the new invariant onto `valueFilter.path` and
   `valueFilter.searchParam`.**
   - File: `source/datatypes/datarequirement.xml`
   - Action: On the `Data Elements` worksheet, in the constraint-
     reference column (the same column that today carries `1` on the
     `codeFilter.path` / `codeFilter.searchParam` rows and `2` on the
     `dateFilter.path` / `dateFilter.searchParam` rows), set the value
     to `3` for both the `DataRequirement.valueFilter.path` and
     `DataRequirement.valueFilter.searchParam` rows.
   - Details: This is what causes the publisher to emit the
     constraint reference on the generated StructureDefinition for
     those elements.

5. **Local publisher build & QA review.**
   - Files: generated outputs under the publisher's working tree
     (not committed).
   - Action: Run the FHIR publisher locally (per the bespoke build
     described in the briefing — `source/fhir.ini` driven, **not**
     `sushi`/`ig.ini`) and verify:
     - The generated StructureDefinition for DataRequirement carries
       the new `value[x]` type list and the new `drq-3` constraint on
       `DataRequirement.valueFilter`.
     - The generated short / definition strings on
       `valueFilter.value[x]` reflect the rewritten cells.
     - QA does not introduce new errors on `qa.html` /
       `suppressed-messages.txt` lines that mention `DataRequirement`
       or `drq-`.
   - Details: Do not commit any generated artefacts; only the
     spreadsheet edits should appear in the diff.

### Testing Considerations

- **Revalidate `DataRequirement` instances and any resource that
  embeds `DataRequirement`** (notably `Library.dataRequirement`,
  `Measure.basisRequirement` / `dataRequirement`,
  `PlanDefinition.action.input` / `output`).
- **Author or update at least one example** under `source/` that
  populates `valueFilter` using one of the new types (e.g.,
  `valueQuantity` or `valueRange`) so generated documentation has a
  worked example.
- **Negative-test the new constraint:** create an example fragment
  that supplies neither `valueFilter.path` nor `valueFilter.searchParam`
  (or both) and confirm the publisher's QA flags it via `drq-3`.
- **Edge cases to verify:**
  - `valueFilter` with `valueQuantity` + `comparator = '>'` round-trips
    correctly.
  - `valueFilter` with `valueBoolean` (the simplest new type) is
    accepted.
  - `valueFilter` with `valueDateTime` is now rejected (no longer in
    the type set).
- **Interoperability consideration:** none of FHIR's automated
  upgrade tooling will silently rewrite a removed type — implementers
  with R4-style `valueDateTime` filters will need to rephrase via
  `dateFilter`. Call this out in the build's release note for
  `DataRequirement` if such notes are produced for the next ballot.

- **`cqf-valueFilter` extension follow-up.** Bryn committed on Zulip
