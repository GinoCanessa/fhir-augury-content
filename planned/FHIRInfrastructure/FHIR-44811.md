## Resolution Summary

**Resolution: Persuasive.** The applied disposition states:

> Remove <https://build.fhir.org/ig/FHIR/fhir-tools-ig/CodeSystem-tx-issue-type.html>
> from fhir-tools-ig and move into THO. (We anticipate that the
> `http://hl7.org/fhir/tools/CodeSystem` prefix will change as this migrates to
> THO and the impacts of this change can be managed by existing participants
> in the FHIR terminology ecosystem.)

In concrete terms, the work is a two-repo migration:

1. **Add** the `tx-issue-type` CodeSystem (and its companion ValueSet) to the
   HL7 Terminology (THO) publication, sourced from the **HL7/UTG** repository,
   under the new canonical prefix `http://terminology.hl7.org/CodeSystem/`
   (i.e., new canonical `http://terminology.hl7.org/CodeSystem/tx-issue-type`,
   and `…/ValueSet/tx-issue-type`).
2. **Remove** the CodeSystem (and any companion ValueSet) from
   **HL7/fhir-tools-ig**, replacing in-IG references with the new THO
   canonical URL.

The TI tracker call on 2024-04-22 explicitly agreed this CodeSystem and its
ValueSet should live in THO, and FHIR-I was identified as the sponsoring WG
for the UP/UTG ticket needed to introduce it. The change is acknowledged as
breaking for any consumer that hard-coded the `…/fhir/tools/CodeSystem/`
prefix; the resolution accepts that risk on the basis that the affected
audience is a small, identifiable set of terminology-ecosystem participants.

## Feature Proposal

### Problem Statement

The `tx-issue-type` CodeSystem classifies the issues that terminology servers
(`tx.fhir.org`, Ontoserver, etc.) return inside `OperationOutcome.issue`
entries from `$validate-code` and related operations. Each issue must carry a
machine-readable `type` so that clients (most notably the FHIR validator) can
decide how to react — for example, downgrading certain errors to warnings
based on binding strength, or surfacing partial-validation notices.

Today this CodeSystem is published only as part of the **fhir-tools-ig**
(`http://hl7.org/fhir/tools/CodeSystem/tx-issue-type`). That placement has
several problems:

- **Discoverability and authority.** Terminology server compliance test cases
  depend on this CodeSystem, but its location in a tools IG makes it look
  optional or implementation-specific rather than a normative terminology
  artifact.
- **Cross-spec reuse.** Other FHIR work — explicitly FHIR-44810 (documenting
  how a server signals incomplete validation using `processing-note`) —
  needs to reference the CodeSystem from outside the tools IG. Cross-IG
  dependencies on a tools IG are awkward.
- **Lifecycle.** Terminology lifecycle (deprecation, addition of new codes,
  versioning) belongs in the THO governance flow managed via UTG, not in a
  tooling IG release cycle.

### Proposed Change

Migrate the artifact to THO via the UTG repository, then deprecate the
fhir-tools-ig copy.

**1. New CodeSystem in HL7/UTG**

- **File:** `input/sourceOfTruth/fhir/codeSystems/CodeSystem-tx-issue-type.xml`
  (new), following the naming convention used by every other
  `terminology.hl7.org/CodeSystem/<id>` entry already in that directory
  (e.g., `CodeSystem-accepting-patients.xml`).
- **CodeSystem.url:** `http://terminology.hl7.org/CodeSystem/tx-issue-type`
- **CodeSystem.id:** `tx-issue-type`
- **CodeSystem.name:** `TxIssueType`
- **CodeSystem.title:** `Terminology Service Issue Type`
- **CodeSystem.status:** `active`
- **CodeSystem.experimental:** `false`
- **CodeSystem.publisher:** `Health Level Seven International`
- **CodeSystem.caseSensitive:** `true`
- **CodeSystem.content:** `complete`
- **CodeSystem.valueSet:** `http://terminology.hl7.org/ValueSet/tx-issue-type`
- **CodeSystem.copyright:** standard THO CC0 boilerplate (copy from any
  `unified/codeSystems/*.xml`).
- **WG extension:** `structuredefinition-wg` valueCode = `fhir` (FHIR
  Infrastructure is the sponsoring WG per the 2024-04-22 TI tracker call).
- **Concepts:** transcribe every concept currently published at
  <https://build.fhir.org/ig/FHIR/fhir-tools-ig/CodeSystem-tx-issue-type.html>
  exactly, preserving code, display, definition, hierarchy (parent/child via
  the standard `concept` nesting), and any concept properties already
  present (notably the `processing-note` code referenced by FHIR-44810).
- **Version:** `1.0.0` (initial THO publication).

**2. New ValueSet in HL7/UTG**

- **File:** `input/sourceOfTruth/fhir/valueSets/ValueSet-tx-issue-type.xml`
  (new).
- **ValueSet.url:** `http://terminology.hl7.org/ValueSet/tx-issue-type`
- **ValueSet.id:** `tx-issue-type`
- **ValueSet.name:** `TxIssueType`
- **ValueSet.title:** `Terminology Service Issue Type`
- **ValueSet.compose.include[0].system:**
  `http://terminology.hl7.org/CodeSystem/tx-issue-type` (no filter — include
  all).
- Same WG extension, copyright, status, and publisher as the CodeSystem.

**3. Deprecate / remove from HL7/fhir-tools-ig**

- Delete the source files that define the CodeSystem and ValueSet under the
  `http://hl7.org/fhir/tools/CodeSystem/tx-issue-type` and
  `http://hl7.org/fhir/tools/ValueSet/tx-issue-type` URLs (typically under
  `input/resources/` or `input/vocabulary/` in the IG — the exact path
  must be confirmed against the current `fhir-tools-ig` clone, which is
  not present in this workspace cache; see *Open Questions*).
- Remove the corresponding entries from any `ImplementationGuide`/`ig.ini`
  resource list and from any per-resource pages under `input/pages/`.
- Replace every in-IG reference to the old URLs with the THO equivalents
  (`http://terminology.hl7.org/CodeSystem/tx-issue-type`,
  `http://terminology.hl7.org/ValueSet/tx-issue-type`).
- Add a short migration note in the IG narrative pointing readers to the THO
  location, plus a `dependsOn` entry on the published THO IG version that
  introduces the CodeSystem.

### Design Rationale

- **THO over Core.** The ticket framed the choice as "core or THO"; the
  workgroup discussion (TI 2024-04-22, plus comments from McClure and
  Snell) converged on THO because (a) THO governs all `terminology.hl7.org`
  artifacts under UTG, (b) lifecycle changes can ship without a Core ballot,
  and (c) every other terminology artifact of similar character already
  lives there. Putting it in Core would tie the cadence to FHIR Core
  releases unnecessarily.
- **URL change is unavoidable.** The resolution explicitly acknowledges and
  accepts the canonical-URL change from the `…/fhir/tools/` prefix to the
  `…/terminology.hl7.org/` prefix. That is the definitive UTG/THO
  namespace and matches every existing `fhir/codeSystems/*.xml` entry in
  the UTG `sourceOfTruth` (e.g., `accepting-patients` is published as
  `http://terminology.hl7.org/CodeSystem/accepting-patients`).
- **Same WG ownership.** FHIR Infrastructure (the sponsoring WG for this
  ticket) becomes the maintaining WG of the new THO artifact. This matches
  the ticket's `work_group` field and the 2024-04-22 disposition note.
- **Companion ValueSet.** The fhir-tools-ig publishes a ValueSet alongside
  the CodeSystem; THO consistently pairs every authored CodeSystem with a
  bound "all codes" ValueSet under `valueSets/`. Carrying the ValueSet
  across keeps existing bindings (in client code, validators, and
  FHIR-44810's planned guidance) straightforward to retarget.

### Breaking Changes

**Yes — this change is breaking for terminology-ecosystem consumers.**

- The canonical URL of the CodeSystem changes from
  `http://hl7.org/fhir/tools/CodeSystem/tx-issue-type` to
  `http://terminology.hl7.org/CodeSystem/tx-issue-type`. Any code,
  validator config, server profile, or test harness that compares against
  the old URL will need to be updated.
- The same applies to the bound ValueSet
  (`http://hl7.org/fhir/tools/ValueSet/tx-issue-type` →
  `http://terminology.hl7.org/ValueSet/tx-issue-type`).
- Concept codes themselves are preserved, so `OperationOutcome.issue.code`
  values do not change in their string form — only the system that gives
  them meaning changes URL.

The resolution explicitly accepts this break ("the impacts of this change
can be managed by existing participants in the FHIR terminology
ecosystem").

### Related Specifications

- **FHIR Core (HL7/fhir).** The validator and the documented behavior of
  `$validate-code` are written against the existence of this CodeSystem.
  Once relocated, any documentation or examples that cite the old URL
  must be updated. *(Best handled as a follow-up edit when the next
  related ticket — e.g., FHIR-44810 — is implemented.)*
- **HL7 Terminology (THO) publication.** The new artifacts will appear in
  the next THO release built from UTG.
- **FHIR Tools IG.** Continues to exist; loses these two artifacts and
  gains a `dependsOn` on THO.
