## Resolution Summary

Resolution type: **Persuasive**.

The resolution adds normative narrative to the FHIR Core specification
explaining how extension `context` behaves when an extension is carried on
an element that is itself being communicated via a cross- / inter-version
extension. The exact wording to be added (from the resolution description) is:

> "In some situations, there will be a need to send an extension on an
> element that is being communicated using an inter-version extension.
> The extension to be communicated will have a 'context' of the element
> defined in that foreign version and will often NOT have an explicit
> context of the inter-version extension - whether a 'standard' extension
> that was explicitly defined for the purpose or one using the special
> syntax defined here that is implicitly defined. In either case,
> validators SHALL behave as though the context includes the relevant
> cross-version extension."

No StructureDefinition, ValueSet, CodeSystem, or schema changes are
required. The change is a documentation addition to the section that
defines the cross-version extension mechanism.

## Feature Proposal

### Problem Statement

When an element is backported (or forward-ported) into a different FHIR
version via a cross-version / inter-version extension (either an
explicitly-authored "standard" cross-version extension or one defined
implicitly via the URL pattern
`http://hl7.org/fhir/[defining_version]/StructureDefinition/extension-[Path]`
documented in `source/versions.html` at the `#extensions` anchor), the
backported "element" is physically expressed as an extension on the
host resource. Users then frequently need to attach further extensions
to that backported element — e.g. the standard extension for
`CodeableReference` when a newer-version element of type
`CodeableReference` is backported into R4 (see the Zulip discussion
"Backporting CodeableReference", stream `fhir/infrastructure-wg`, which
directly references this ticket).

The FHIR Core specification currently does not address this situation.
The sub-extension's `StructureDefinition.context` was defined against
the foreign-version element (e.g., `CodeableReference`), not against
the inter-version extension container that carries it, so a naïve
validator would reject the sub-extension as appearing in an invalid
context. The ticket asks the specification to state that this usage is
valid and to instruct validators accordingly.

### Proposed Change

Add a new paragraph (the verbatim wording quoted above in
"Resolution Summary") to `source/versions.html` inside the existing
"Extensions for converting between versions" section (the block that
begins at the `<a name="extensions"></a>` anchor on line 622 and ends
before the `<a name="extensions-examples-primitive"></a>` / `<h4>Formal
Definitions for extensions</h4>` block at line 699–703).

The natural insertion point is immediately after the existing closing
paragraph "Further details can be found in the individual cross-version
definition packages." at line 694–696 and before the blank-line
separator at line 697–698, so the new text sits as the final
substantive paragraph of the `#extensions` section, ahead of the
"Formal Definitions for extensions" sub-section.

Concretely, wrap the resolution text in a `<p>` element, matching the
surrounding HTML style:

```html
<p>
In some situations, there will be a need to send an extension on an
element that is being communicated using an inter-version extension.
The extension to be communicated will have a <code>context</code> of
the element defined in that foreign version and will often NOT have an
explicit context of the inter-version extension - whether a "standard"
extension that was explicitly defined for the purpose or one using the
special syntax defined here that is implicitly defined. In either
case, validators <b>SHALL</b> behave as though the context includes
the relevant cross-version extension.
</p>
```

No other files require changes:

- No `StructureDefinition.context` authoring changes are needed on
  individual extension definitions — the resolution explicitly takes
  the form "validators SHALL behave as though the context includes the
  relevant cross-version extension", i.e. the rule lives in the prose,
  not in each extension's context list.
- No ValueSet / CodeSystem changes.
- No schema, search parameter, or operation changes.

### Design Rationale

- The specification already defines the inter-version extension
  mechanism in exactly one place (`source/versions.html` at the
  `#extensions` anchor). Co-locating the new validator guidance with
  the mechanism keeps all inter-version semantics in one narrative.
- Expressing the rule as a validator SHALL, rather than amending every
  cross-version extension definition's `context`, avoids an explosion
  of per-extension context entries (the implicit URL-pattern form is
  effectively unbounded — one per element per foreign version) and
  keeps the implicit cross-version extensions usable without each one
  needing to re-declare contexts for every possible nested extension.
- The wording ("standard extension that was explicitly defined for the
  purpose or one using the special syntax defined here that is
  implicitly defined") maps cleanly onto the two forms the section
  already describes: explicit standard extensions (the Formal
  Definitions table that follows) and the implicit
  `http://hl7.org/fhir/[defining_version]/StructureDefinition/extension-[Path]`
  pattern documented in the same section at lines 640–696.
- Mckenzie's comment on the ticket ("Grahame will see if he can figure
  out how to make this work") signalled that mechanism-level change
  was considered; the accepted resolution is the lighter-weight
  narrative-only approach.

### Breaking Changes

No breaking changes at the data-model level. The prose clarifies a
previously-undefined behaviour and adds a `SHALL` for validators:
conformant validators that were already permitting this usage are
unaffected; validators that were rejecting sub-extensions inside
inter-version extensions on the grounds of missing `context` must now
accept them. This is a validator-behaviour tightening, but it relaxes
(rather than tightens) accepted-instance content, so deployed
instances remain valid.

### Open Questions

- The resolution does not prescribe an exact insertion point within
  the `#extensions` section. This plan places the new paragraph at
  the end of that section (immediately before the "Formal
  Definitions for extensions" sub-section), since it is a
  closing-rule that applies to the whole mechanism described above
  it. An editor may reasonably choose to place it earlier (e.g.,
  immediately after the opening two paragraphs that introduce the
  round-tripping problem). Either placement satisfies the
  resolution; the trailing placement has been chosen to avoid
  interrupting the "URL pattern → examples → formal definitions"
  narrative arc that currently flows lines 640–703.
- The resolution refers to "the relevant cross-version extension" in
  the singular. Nested inter-version extensions (e.g., an R4 backport
  of an element whose type is itself only defined via another
  cross-version extension) are not explicitly addressed; the natural
  reading is transitive ("validators SHALL behave as though the
  context includes **each** relevant cross-version extension along
  the chain"), but this is not stated. If this is intended, a future
  editorial pass may want to clarify — out of scope for this ticket.
