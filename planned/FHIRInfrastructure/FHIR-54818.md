## Resolution Summary

Resolution: **Persuasive with Modification**.

Per the resolution description:

> "Will remove the sentence, though for a different reason (the referenced content is old & out-of-date). It's fine for this specification to point to external documents for 'additional considerations'. Such content has no impact on conformance. However, there are places we have external references that are intended to have conformance impact and we're not always super clear in differentiating.
>
> We will define a style guide that ensures that external references clearly disambiguate whether the reference is informative or intended to have conformance weight and will review the external references in the core spec to ensure they comply with this guidance."

Concretely, two threads of work:

1. **Immediate, in-scope of this ticket:** remove the "Additional discussion on inter-versioning issues can be found here: …Interversion+Compatibility" sentence from the inter-versioning section of `source/versions.html`. The Confluence page is from 2019, out-of-date, and cited from a normative part of the spec.
2. **Follow-up (referenced by this ticket, broader scope):** define an editorial style guide for external references in the core spec that disambiguates "informative" vs "conformance-bearing" links, and review existing external references for compliance. Tracked as a side-effect on Zulip ([fhir/infrastructure-wg › Conformance expectations for external references](https://chat.fhir.org/#narrow/stream/fhir%2Finfrastructure-wg/topic/Conformance%20expectations%20for%20external%20references/near/570436801)).

## Feature Proposal

### Problem Statement

The reporter identified that `source/versions.html` (which renders to the normative "Managing Versions of FHIR" page) contains a link to a non-normative HL7 Confluence page on inter-version compatibility:

> "Additional discussion on inter-versioning issues can be found here: https://confluence.hl7.org/display/FHIR/Interversion+Compatibility."

The Confluence page is uncontrolled content from 2019: it can be edited or deleted at any time by anyone with Confluence access, and is not subject to the normative ballot/change-control process that governs the FHIR specification. Embedding it in a normative section creates an implicit conformance dependency on uncontrolled, stale material that the WG has no commitment to maintain.

The WG agreed in resolution to remove the sentence (with a slightly different rationale — the referenced content is old and out-of-date — and noting that it is acceptable to point at external documents for purely informative "additional considerations", but that the spec is currently inconsistent about signalling whether an external reference is informative or carries conformance weight).

### Proposed Change

**Repository:** `HL7/fhir` (FhirCore).

**File:** `source/versions.html`.

**Change:** Delete the trailing block in the "Managing Inter-version compatibility" area that currently reads:

```html
<hr/>
<p>
Additional discussion on inter-versioning issues can be found here:
<a href="https://confluence.hl7.org/display/FHIR/Interversion+Compatibility">https://confluence.hl7.org/display/FHIR/Interversion+Compatibility</a>.
</p>
```

In the current clone (`db79dcdf`) this is lines 571–575 of `source/versions.html`, sitting between the deprecated-artifacts list (ending line 570) and the `<a name="studev"></a> <h3>Managing Change during the development process</h3>` heading (line 578).

After the change:

- The deprecated-artifacts `<ul>` block ends at line 570.
- The next sibling block is the `<a name="studev"></a> <h3>Managing Change during the development process</h3>` section (currently line 578).
- The `<hr/>` separator that visually demarcated the Confluence link is also removed (it has no remaining content to separate; the natural section break under `<h3>Managing Change during the development process</h3>` provides the structural divider).

No other resource, profile, ValueSet, search parameter, or example is affected by this removal. A repository-wide grep (`git grep -i "interversion"` against `source/`) finds only two matches:

- `source/r3maps.html:58` — references the GitHub `interversion` packages folder. This is **not** the Confluence page targeted by this ticket and **must not** be removed.
- `source/versions.html:574` — the line being removed.

**Out of scope but tracked (style-guide follow-up):** the resolution also commits the WG to (a) defining a style guide that distinguishes informative from conformance-bearing external references, and (b) reviewing existing external references in the core spec against that guide. Both are larger pieces of editorial work that should be handled under separate tickets/Zulip-coordinated follow-ups (the Zulip topic *"Conformance expectations for external references"* in `fhir/infrastructure-wg` is already the home for that discussion). They are listed under Open Questions below so they are not lost, but they are not part of this ticket's code change.

### Design Rationale

- The Confluence page is uncontrolled content: it has no version pinning, no ballot process, and no SLA on continued existence. A normative spec page should not appear to delegate "additional considerations" to such a target without a clear "informative only" framing — and even with one, the WG agreed the 2019 content is too stale to be useful.
- Pure deletion (rather than rewriting the sentence to add an "informative only" qualifier) is what the resolution explicitly approves. The WG considered the alternative of keeping the sentence with disambiguation, but landed on removal because the referenced content is out-of-date and the broader disambiguation question is being handled by the forthcoming style guide rather than ad-hoc per-link rewording.
- Removing the trailing `<hr/>` along with the paragraph keeps the rendered page clean: the `<hr/>` exists only to separate the Confluence-link paragraph from the deprecated-artifacts list above it; with the paragraph gone, the rule serves no purpose.
- The change is purely narrative on a top-level `source/*.html` authoring page, which the briefing confirms is the supported edit point for prose pages in this repo (the bespoke HL7 publisher reads `source/*.html` directly; there is no sushi/IG-publisher pipeline to regenerate it).
