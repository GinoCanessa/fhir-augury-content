## Resolution Summary

Resolution: **Persuasive** (no `resolution_description` is recorded on the
ticket; no comments present). The reporter pointed out that the table of
links to the cross-version extension packages on the FHIR specification
contains URLs of the form `http://hl7.org/fhir/<defining_version>/extensions/<target_version>/package.tgz`
(e.g., `http://hl7.org/fhir/3.0/extensions/4.0/package.tgz`), all of which
302-redirect to a path under `/STU3/extensions/...` that returns HTTP 404.
Because the resolution is "Persuasive" with no narrative, the implied
fix is the only one consistent with the report: **change the broken
`/extensions/<X>/package.tgz` links so that they resolve**, either by
re-publishing the packages at the URLs the table advertises or by
pointing the table at URLs that are actually published.

In the current (post-R5) world the cross-version extensions are no longer
shipped as `hl7.fhir.extensions.r*` packages co-located under
`hl7.org/fhir/<v>/extensions/<v>/`; they have been moved to per-pair
`hl7.fhir.uv.xver-<defining>.<target>` IGs published at
`https://hl7.org/fhir/uv/xver-<defining>.<target>/`. The table on
`source/versions.html` should therefore be repointed at those IG URLs.

## Feature Proposal

### Problem Statement

`source/versions.html` (anchor `#extensions`, sub-heading "Formal
Definitions for extensions") publishes a matrix of "R*N* Extensions" rows
× "R*N* Package" columns. Historically each populated cell contained
both the package name (e.g., `hl7.fhir.extensions.r3:4.0.1`) and an
`<img src="npm16.png"/>` linked to
`http://hl7.org/fhir/<defining_version>/extensions/<target_version>/package.tgz`.
Those `package.tgz` URLs 302-redirect (per the ticket, e.g.
`http://hl7.org/fhir/3.0/extensions/4.0/package.tgz`
→ `http://hl7.org/fhir/STU3/extensions/4.0/package.tgz`) and the
redirect target is a 404. The table therefore promises downloads of the
cross-version extension packages that nobody can actually fetch, which
is misleading for implementers trying to wire up cross-version tooling.

### Proposed Change

Edit `source/versions.html` and replace every cell in the
"Formal Definitions for extensions" table that currently uses the
`hl7.fhir.extensions.r*:*` / `http://hl7.org/fhir/<X>/extensions/<Y>/package.tgz`
form with a working link to the corresponding cross-version extension IG.
The table already exists at `source/versions.html` lines 705–769
(at clone HEAD `db79dcd`); each non-`n/a` cell should look like:

```html
<td><a href="https://hl7.org/fhir/uv/xver-<defining>.<target>/">hl7.fhir.uv.xver-<defining>.<target></a></td>
```

Concretely, the matrix becomes:

| Row \ Column | R2 Package | R3 Package | R4 Package | R4B Package | R5 Package | R6 Package |
|---|---|---|---|---|---|---|
| R2 Extensions | n/a | `hl7.fhir.uv.xver-r3.r2` | `hl7.fhir.uv.xver-r4.r2` | `hl7.fhir.uv.xver-r4b.r2` | `hl7.fhir.uv.xver-r5.r2` | `hl7.fhir.uv.xver-r6.r2` |
| R3 Extensions | `hl7.fhir.uv.xver-r2.r3` | n/a | `hl7.fhir.uv.xver-r4.r3` | `hl7.fhir.uv.xver-r4b.r3` | `hl7.fhir.uv.xver-r5.r3` | `hl7.fhir.uv.xver-r6.r3` |
| R4 Extensions | `hl7.fhir.uv.xver-r2.r4` | `hl7.fhir.uv.xver-r3.r4` | n/a | `hl7.fhir.uv.xver-r4b.r4` | `hl7.fhir.uv.xver-r5.r4` | `hl7.fhir.uv.xver-r6.r4` |
| R4B Extensions | `hl7.fhir.uv.xver-r2.r4b` | `hl7.fhir.uv.xver-r3.r4b` | `hl7.fhir.uv.xver-r4.r4b` | n/a | `hl7.fhir.uv.xver-r5.r4b` | `hl7.fhir.uv.xver-r6.r4b` |
| R5 Extensions | `hl7.fhir.uv.xver-r2.r5` | `hl7.fhir.uv.xver-r3.r5` | `hl7.fhir.uv.xver-r4.r5` | `hl7.fhir.uv.xver-r4b.r5` | n/a | `hl7.fhir.uv.xver-r6.r5` |

Also remove the now-meaningless `<img src="npm16.png"/>` icons and the
embedded `hl7.fhir.extensions.r*:*` package coordinates, and update the
caveat paragraph immediately under the table from

> Note for balloters: the R5 packages will be created when R5 is finalized. Until then, these are broken links.

to

> Note that some guides linked above will not exist at the time of publication (e.g., extensions based-on or targeting this release). The links will be correct and will resolve once the packages are available.

This is the wording already chosen for the same fix in FHIR-42924; reusing
it keeps the page internally consistent.

### Design Rationale

- The cross-version extensions are no longer co-published with the core
  spec; per `source/history.html` (line 158): "All the extensions have
  been moved to a new package `hl7.fhir.uv.extensions` published at
  `http://hl7.org/fhir/extensions`." Pointing the table at the per-pair
  `hl7.fhir.uv.xver-<defining>.<target>` IGs is the only choice that
  remains consistent with where the artifacts actually live.
- Linking to the IG home page (`https://hl7.org/fhir/uv/xver-...`)
  rather than the package tarball means the link survives package
  re-versioning — implementers can still find the published `*.tgz`
  from the IG landing page, and stale version coordinates are no longer
  baked into the spec.
- Replacing the original "Note for balloters …" sentence with the
  forward-looking "some guides linked above will not exist at the time
  of publication …" wording keeps the page honest in cases where a
  given xver-IG is published on a different schedule than the core
  spec, while no longer implying that the broken links are normal.
- The ticket's resolution carries no narrative, so the only design
  alternative on record is "republish files at the old URLs". That was
  rejected by the broader R5/R6 publication architecture, which has
  already retired the `hl7.fhir.extensions.r*` package family in favor
  of `hl7.fhir.uv.xver-*`.

  sushi / IG-publisher pipeline.

  (the `<a name="extensions">` "Extensions for converting between

- **Applicable change recipes:** "Update narrative / prose" — direct
  edit of the `source/*.html` page, replacing inline link markup. No

    (`#extensions`, `#extensions-definitions`) must be preserved

  IGs that the new links point at live in their own per-pair
  HL7 repositories, but FHIR-36021 does not require any change there
  — it only repoints existing links at IGs that are already published
  under `hl7.org/fhir/uv/xver-*`.

### Breaking Changes

No breaking changes to FHIR conformance. The fix is documentation-only:

- No StructureDefinition, ValueSet, CodeSystem, Operation, or search
  parameter is added, removed, or modified.
- The user-visible package coordinates do change: the `<td>` text
  switches from `hl7.fhir.extensions.r*:*` to
  `hl7.fhir.uv.xver-<defining>.<target>`. Implementers who scraped the
  table for package IDs will need to update their tooling, but the old
  coordinates were already broken (the linked tarballs 404), so no
  working implementation depended on them.

### Related Specifications

- The `hl7.fhir.uv.xver-<defining>.<target>` cross-version extension
  IGs (e.g., `hl7.fhir.uv.xver-r4.r3`, …, `hl7.fhir.uv.xver-r6.r5`)
  become the canonical landing pages for the artifacts the table
  references. Their publication state on `hl7.org/fhir/uv/xver-*` is
  what determines whether each new link resolves at any given moment.
- `source/basic/basic-notes.xml`,

- `source/history.html` (line 158) already records the move of
  cross-version extensions to the `hl7.fhir.uv.extensions` family —
