## Feature Proposal

### Problem Statement

The "Formal Definitions for extensions" table in the FHIR Core
specification enumerates, for each source-version of cross-version
extensions, the published NPM package name per target-version column.
The final row — **R5 Extensions** — was authored as an exact duplicate
of the **R4 Extensions** row (`hl7.fhir.extensions.r4:*`), producing a
table that (a) self-contradicts the heading on its own row, and (b)
breaks the consistent `r{source}` naming pattern established by the R2,
R3, and R4 rows. The erroneous row also marks the R5 Package column
as a valid package rather than `n/a` (a row cannot publish extensions
into its own FHIR release). Implementations or readers trying to pick a
package by row/column end up pointed at a non-existent / wrong package.

### Proposed Change

Apply the corrected row as written in the ticket's "should it not be"
block:

| Row label        | R2 Package                   | R3 Package                   | R4 Package                   | R5 Package |
|------------------|------------------------------|------------------------------|------------------------------|------------|
| R5 Extensions:   | `hl7.fhir.extensions.r5:1.0.2` | `hl7.fhir.extensions.r5:3.0.1` | `hl7.fhir.extensions.r5:4.0.1` | `n/a`      |

Concretely:

- Replace every `hl7.fhir.extensions.r4:*` token in the R5 row with the
  equivalent `hl7.fhir.extensions.r5:*` token (matching the version
  suffixes `1.0.2`, `3.0.1`, `4.0.1`).
- Move the `n/a` cell from the R4 position to the R5 position so the
  diagonal self-reference `n/a` pattern continues down the table.
- Leave the four preceding rows (R2 / R3 / R4 Extensions) unchanged.

No element, cardinality, or terminology change is involved — this is a
pure documentation / authored-HTML fix.

### Design Rationale

The corrected row is the only one that preserves the established
invariant of the table: the row labelled *R`n` Extensions* publishes
the `hl7.fhir.extensions.r{n}` package family across the other
versions' columns, and `n/a` lives on the `R{n}` column for that row
(you cannot "publish R5 extensions *into* R5"). Fixing the row in place
is strictly narrower than re-authoring the table, and matches the
reporter's explicit corrected example in the ticket body. No
alternative is discussed on the ticket or in the linked Zulip thread;
the change is a non-substantive technical correction as reflected by
the ticket type (*Technical Correction*) and resolution (*Persuasive*).
