## Resolution Summary

The resolution is **Persuasive** with the resolution description: *"Will add 'unknown' to the value set."*

Concretely, this means adding a new `unknown` concept to the
`http://hl7.org/fhir/questionnaire-answers-status` CodeSystem (which
backs the value set bound to `QuestionnaireResponse.status`). The added
concept provides the missing "escape valve" so that a
`QuestionnaireResponse` instance can satisfy the mandatory, required-bound
`status` element when the authoring system genuinely does not know the
lifecycle state of the response.

The existing element-level comment on `QuestionnaireResponse.status`
already documents this exact intent — *"Unknown does not represent
'other' — one of the defined statuses must apply. Unknown is used when
the authoring system is not sure what the current status is."* — but no
matching code currently exists. This ticket aligns the code system with
the comment.

## Feature Proposal

### Problem Statement

`QuestionnaireResponse.status` is `min=1`, `isModifier=true`, with a
`required` binding to the value set
`http://hl7.org/fhir/ValueSet/questionnaire-answers-status`. The value
set today composes the `http://hl7.org/fhir/questionnaire-answers-status`
CodeSystem, which contains only:

- `in-progress`
- `completed`
- `amended`
- `entered-in-error`
- `stopped`

None of those codes covers the case in which the lifecycle state of the
response is genuinely unknown (e.g., when records are imported from an
external system that did not carry status, or when an intermediary that
must populate the resource cannot determine its state). Because the
binding is `required` and the element is mandatory, there is no
conformant value an implementer can choose in that situation, leaving the
resource impossible to populate. The element's own comment already
anticipates an `unknown` code, but it was never defined.

### Proposed Change

Add a single new concept to the `questionnaire-answers-status` CodeSystem:

- **System:** `http://hl7.org/fhir/questionnaire-answers-status`
- **Code:** `unknown`
- **Display:** `Unknown`
- **Definition:** *"The current state of this QuestionnaireResponse is
  not known. Note: This concept is not to be used for "other" — one of
  the listed statuses is presumed to apply, but the authoring/source
  system does not know which one."* (matches the wording used by other
  FHIR `*-status` "unknown" codes and aligns with the existing
  `QuestionnaireResponse.status` element comment.)

No other concept changes are required. The bound ValueSet is
`<immutable>true</immutable>` and is composed solely of `include
<system>http://hl7.org/fhir/questionnaire-answers-status</system>` with
no concept enumeration or filter, so the new code is picked up
automatically — no value-set edit is needed beyond a normal version bump
that the publisher applies.

The `QuestionnaireResponse` StructureDefinition's element-level `short`
text for `QuestionnaireResponse.status` must be updated from:

> `in-progress | completed | amended | entered-in-error | stopped`

to:

> `in-progress | completed | amended | entered-in-error | stopped | unknown`

The existing element `<comment>` should be retained essentially as-is —
its wording already explains the intended semantics of `unknown` — though
it can be lightly tightened to reflect that the code now exists rather
than being purely conceptual.

### Design Rationale

- **Consistent with FHIR-wide pattern.** Many other required-bound,
  mandatory `*.status` elements in core (e.g., `Observation.status`,
  `DiagnosticReport.status`, `MedicationRequest.status`) include an
  `unknown` (and in some cases `other`) escape-valve code for exactly
  this reason. Adding `unknown` here brings `QuestionnaireResponse` in
  line with that established pattern.
- **Resolves a hard conformance bug.** The combination of `min=1` +
  `required` binding + missing escape valve is unsatisfiable for the
  scenarios the reporter describes; no clever workaround at the binding
  level (e.g., a profile loosening the binding) is available because the
  binding is `required` in core.
- **Author intent already documented.** The element's `<comment>`
  already differentiates `unknown` from `other` and constrains its use
  to "the authoring system is not sure what the current status is."
  Adding the code makes the prose match reality rather than introducing
  new semantics.
- **Alternatives considered but rejected by the resolution.** The
  resolution explicitly chose `unknown` only; an `other` code was not
  added because the existing comment specifically rules that semantic
  out ("Unknown does not represent 'other'"). Loosening the binding
  strength was not chosen because that would be a much larger,
  cross-cutting change to a normative element.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Owns both the CodeSystem definition that gains the new
  `unknown` concept and the `QuestionnaireResponse` StructureDefinition
  whose `status` element narrative must be refreshed.
- **Affected Files:**
  - `source/questionnaireresponse/codesystem-questionnaire-answers-status.xml`
    — add a sixth `<concept>` block with code `unknown`, display
    `Unknown`, and the agreed definition.
  - `source/questionnaireresponse/structuredefinition-QuestionnaireResponse.xml`
    — update `<short>` on the `QuestionnaireResponse.status` element to
    include `| unknown`; optionally tighten the `<comment>` so it speaks
    of `unknown` as a defined code.
  - `source/questionnaireresponse/valueset-questionnaire-answers-status.xml`
    — no concept-level edit (composes the full system), but the
    publisher will pick up the new code automatically; review on
    regeneration.
- **Change Scope:** Minor — three small edits in one resource folder.

### Breaking Changes

This is a **substantive (and arguably breaking) change to a normative,
FMM 5/Normative-status CodeSystem and ValueSet** even though it adds
rather than removes a code:

- Receivers that pattern-match on the closed set
  `{in-progress, completed, amended, entered-in-error, stopped}` may
  fail validation or default-handling logic when they encounter
  `unknown`.
- The `QuestionnaireResponse.status` binding is `required`, so any
  conformance test suite that previously asserted "values outside the
  five known codes are invalid" will need updating.
- `unknown` does not change `isModifier` semantics, but downstream
  systems treating non-`entered-in-error` codes as "valid clinical
  state" must decide how to handle the explicit *unknown* state.

These should be called out explicitly in the publication notes for the
release that ships this change.

### Related Specifications

- The bound ValueSet `http://hl7.org/fhir/ValueSet/questionnaire-answers-status`
  is referenced from `QuestionnaireResponse.status` only in core, but
  may be reused by downstream IGs (e.g., HL7 SDC — see the `SDC` label
  on this ticket) and by SearchParameters that constrain `status`. Any
  IG that profiles `QuestionnaireResponse.status` with a tighter binding
  (e.g., excluding the new `unknown` code) is unaffected, but IGs
  relying on the core binding should verify that `unknown` is acceptable
  in their workflows.
- No core SearchParameter changes are implied; the
  `QuestionnaireResponse-status` SearchParameter continues to work
  unchanged (it just gains another valid value).

  not UTG.

## Implementation Plan

### Prerequisites

- None. The change is self-contained in HL7/fhir core source files. No
  dependent ticket needs to land first; no extension or external
  terminology has to be defined.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add `unknown` concept to the QuestionnaireResponseStatus CodeSystem.**
   - File: `source/questionnaireresponse/codesystem-questionnaire-answers-status.xml`
   - Action: After the existing `<concept>` block for `stopped`, append a
     new `<concept>` element.
   - Details:
     ```xml
     <concept>
       <code value="unknown"/>
       <display value="Unknown"/>
       <definition value="The current state of this QuestionnaireResponse is not known. Note: This concept is not to be used for &quot;other&quot; - one of the listed statuses is presumed to apply, but the authoring/source system does not know which one."/>
     </concept>
     ```
     Do not modify the CodeSystem `<version>`, `<date>`, `<status>`,
     `<caseSensitive>`, or `<content>` values — those are managed by the
     publisher / release process. Keep concept ordering consistent with
     the file's existing style (one `<concept>` per block, no extra
     attributes).

2. **Update the `QuestionnaireResponse.status` element narrative.**
   - File: `source/questionnaireresponse/structuredefinition-QuestionnaireResponse.xml`
   - Action: On the `<element id="QuestionnaireResponse.status">` block
     (currently around lines 187–211), update the `<short>` value, and
     optionally refine the `<comment>` value.
   - Details:
     - Change `<short value="in-progress | completed | amended | entered-in-error | stopped"/>`
       to `<short value="in-progress | completed | amended | entered-in-error | stopped | unknown"/>`.
     - Optionally update `<comment>` from
       *"Unknown does not represent &quot;other&quot; - one of the defined
       statuses must apply.  Unknown is used when the authoring system is
       not sure what the current status is."* to wording that no longer
       reads as forward-looking, e.g., *"The `unknown` code does not
       represent &quot;other&quot; - one of the other defined statuses is
       presumed to apply.  `unknown` is used only when the authoring
       system cannot determine which of the other statuses currently
       applies."*
     - Leave the `<binding>` element unchanged — same value set URL,
       same `required` strength.
     - Do **not** edit any `<snapshot>` content; the publisher will
       regenerate it.

3. **Re-run the publisher locally to confirm the value set picks up the new code.**
   - File: n/a (build verification)
   - Action: After the two edits above, run the publisher's
     terminology generation step (`build.sh` / `publish.sh` per the
     repo's standard local build) and confirm that the regenerated
     `valueset-questionnaire-answers-status.xml` snapshot/expansion now
     lists the `unknown` code in addition to the original five.
   - Details: No manual edit to
     `source/questionnaireresponse/valueset-questionnaire-answers-status.xml`
     is required because its `<compose>` is `<include><system>http://hl7.org/fhir/questionnaire-answers-status</system></include>` with no
     concept filter. Verify in CI logs that no QA error is raised for
     the value set.

4. **Update the `QuestionnaireResponse` resource introduction / notes if they enumerate statuses.**
   - File: `source/questionnaireresponse/questionnaireresponse-introduction.xml`
     and `source/questionnaireresponse/questionnaireresponse-notes.xml`
   - Action: Grep both files for any prose that enumerates the five
     existing status codes; if found, add `unknown` to the list and a
     one-sentence description matching the new concept definition. If
     neither file mentions the codes by name (the current copies do
     not), no edit is needed — record a "no change" note in the PR.
   - Details: A repo-wide grep for the literal string
     `entered-in-error` within `source/questionnaireresponse/` is the
     fastest way to find every prose enumeration that may need
     refreshing.

5. **Update `QuestionnaireResponse` examples if a clearly-`unknown` example is desired (optional).**
   - File: `source/questionnaireresponse/questionnaireresponse-example*.xml`
   - Action: Optionally add or adapt one example instance that uses
     `status="unknown"` to demonstrate the new code; otherwise leave
     the existing examples as-is. This is a nice-to-have, not a
     resolution requirement.
   - Details: If added, ensure `examples-header.xml` /
     `list-QuestionnaireResponse-examples.xml` are updated to include
     the new file per the repo's existing example-registration pattern.

### Open Questions

- **Final wording of the `unknown` definition.** The proposal above
  reuses the established FHIR pattern, but the FHIR Infrastructure WG
  may want to harmonise with the wording used in
  `Observation.status`'s `unknown` definition. Confirm the exact
  definition text with the WG before merging.
- **Element-level `<comment>` rewording.** Step 2 proposes a small
  wording refresh because the current comment reads as if `unknown`
  did not yet exist. The committee may prefer to leave the existing
  text untouched; resolve this in PR review.
- **Optional `unknown` example.** Whether to ship a new example
  showing `status="unknown"` (Step 5) is a documentation choice for
  the WG, not a hard requirement of the resolution.
- **SDC IG follow-up.** The ticket carries the `SDC` label. If the
  HL7 SDC IG profiles `QuestionnaireResponse.status` with its own
  binding, SDC editors should review whether to allow or exclude the
  new `unknown` code. This is out of scope for the core change but
  worth flagging in the PR description.
