## Resolution Summary

The committee marked this ticket **Persuasive**. No free-text resolution
description or applied-vote comments are present in the Jira record (the
ticket has zero comments and no `resolution_description` field), so the
resolution must be inferred from the ticket title/description and the
prior Zulip thread that prompted it ([#questionnaire > maxLength for
boolean type?](https://chat.fhir.org/#narrow/stream/questionnaire/topic/maxLength%20for%20boolean%20type%3F),
where Brian Postlethwaite confirmed: *"Yeah, that doesn't seem right.
Can you log it?"*).

The accepted change is therefore: **make the FHIR Core specification
clearly state that `Questionnaire.item.maxLength` does not apply to
`boolean` type items** (or, more generally, only applies to the simple
question types already enumerated by invariant `que-10`). The rule is
already enforced on instance data — `que-10` restricts `maxLength` to
`type in ('string' | 'decimal' | 'integer' | 'text' | 'date' |
'dateTime' | 'time' | 'url')` (see
`source/questionnaire/structuredefinition-Questionnaire.xml:789-795`) —
but the human-readable element documentation on the published page does
not communicate that restriction; it only mentions `base64binary`, which
is itself not in the allowed list. The fix is documentation alignment in
the StructureDefinition's element-level comment, which the publisher
then renders into `questionnaire-definitions.html`.

## Feature Proposal

### Problem Statement

`Questionnaire.item.maxLength` is an `integer` element that limits the
character length of an answer. The instance-level constraint `que-10`
already restricts the element to a closed list of "simple" question
types (`string`, `decimal`, `integer`, `text`, `date`, `dateTime`,
`time`, `url`, plus `optionOrString`-constrained items). However, the
element's published definition page does not call out this restriction;
the only type-specific note attached to `Questionnaire.item.maxLength`
is the `comment`:

> "For base64binary, reflects the number of characters representing the
> encoded data, not the number of bytes of the binary data. The value
> may come from the ElementDefinition referred to by .definition."

This is misleading on two fronts:

1. It implies `maxLength` applies to `base64binary` items, but
   `base64binary` is not in `que-10`'s allow-list, so it is *not*
   permitted on `boolean` (the original complaint), `base64binary`,
   `attachment`, `coding`, `quantity`, `reference`, `group`, `display`,
   etc.
2. It says nothing about the types that *are* allowed, so an implementer
   reading
   <https://hl7.org/fhir/questionnaire-definitions.html#Questionnaire.item.maxLength>
   reasonably concludes `maxLength` is universally applicable — exactly
   the confusion the reporter raised.

### Proposed Change

Update the `Questionnaire.item.maxLength` element documentation so the
generated definitions page explicitly enumerates which `item.type`
values support `maxLength` and confirms it does **not** apply to
`boolean` (and the other excluded types). Concretely:

- **File:** `source/questionnaire/structuredefinition-Questionnaire.xml`
- **Element:** `Questionnaire.item.maxLength` (lines 1245-1264 in the
  current clone).
- **Action:** Replace the existing `<comment value="…"/>` (line 1249)
  with a comment that:
  1. Lists the question types `maxLength` is applicable to, mirroring
     `que-10`: `string`, `text`, `url`, `integer`, `decimal`, `date`,
     `dateTime`, `time`, plus items whose `answerConstraint` is
     `optionOrString`.
  2. States explicitly that `maxLength` SHALL NOT be specified for other
     types — calling out `boolean` (the ticket's exemplar) along with
     the rest of the excluded types.
  3. Preserves the existing useful guidance that, where supported, the
     value may come from the `ElementDefinition` referenced by
     `.definition`. (The current sentence about `base64binary`
     character-vs-byte counting is dropped: `base64binary` is not in
     `que-10`'s allow-list, so the guidance is no longer applicable on
     this element.)

  Suggested replacement text (concrete wording subject to WG editorial
  polish, but the substance is fixed by the resolution):

  > "Only applies to simple question types: `string`, `text`, `url`,
  > `integer`, `decimal`, `date`, `dateTime`, and `time`, plus items
  > whose `answerConstraint` is `optionOrString`. SHALL NOT be specified
  > for other item types (including `boolean`, `group`, `display`,
  > `attachment`, `coding`, `quantity`, `reference`, etc.) — see
  > invariant `que-10`. Where supported, the value may come from the
  > `ElementDefinition` referred to by `.definition`."

- Optionally tighten the `<short value="No more than these many
  characters"/>` (line 1247) to "Maximum allowed answer length (simple
  text/numeric/date items only)" so the element's one-line summary on
  the resource page also signals the type restriction. This is a small
  editorial nicety; if the WG prefers minimal diff, leave `short`
  untouched.

No constraint change is required: `que-10` already enforces the rule.
No type change, cardinality change, or binding change is required.

### Design Rationale

- The behaviour is already correct on the wire — `que-10` rejects a
  `boolean` item that carries `maxLength`. The defect is purely
  *documentation*, which matches the ticket's `Type: Technical
  Correction` and the reporter's request to "correct the documentation."
- Aligning the element `comment` with the invariant's allow-list keeps
  the spec internally consistent: the published definitions page is
  generated from the StructureDefinition, so editing the comment is the
  single source of truth.
- The current `base64binary` sentence pre-dates the introduction of
  `que-10` (which closed the allow-list) and is now actively misleading
  because `base64binary` is not on that list. Removing it eliminates a
  contradiction between the element comment and the invariant.
- Naming `boolean` explicitly addresses the exact case the reporter
  raised and the Zulip thread agreed on, without inventing a new
  exclusion list (the exclusion is just "everything not in `que-10`").
- Alternative considered: expanding `que-10` to permit additional types
  (the approach proposed in [FHIR-53143](https://jira.hl7.org/browse/FHIR-53143)
  for `dateTime`). That ticket was resolved **No Change** — `dateTime`
  is already allowed — and there is no consensus to broaden `que-10`
  here. So this fix is documentation-only.
