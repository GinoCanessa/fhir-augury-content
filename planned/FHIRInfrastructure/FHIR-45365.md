## Problem Statement

The Subscriptions Backport IG defines several extensions (e.g., `backport-heartbeat-period`, `backport-payload-content`, `backport-channel-type`, etc.). The **profiles** that use these extensions correctly constrain their cardinality (e.g., `0..1`, `1..1`), but the **extension StructureDefinitions themselves** do not specify `min` and `max` cardinality values. This means the extensions are implicitly `0..*` at the definition level, which is semantically incorrect for extensions that should only appear once (e.g., heartbeat period, timeout, max count).

## Resolution Description

> Propose updating all extension definitions to set the min and max cardinality.

## REPO_NOT_CACHED: HL7/fhir-subscription-backport-ig

The primary affected repository (`HL7/fhir-subscription-backport-ig`) is **not** in the cached repo list. The analysis below is based on live GitHub API reads of the current `master` branch.

## Feature Proposal

Add explicit `min` (`^min`) and `max` (`^max`) cardinality constraints to every extension definition in the Subscriptions Backport IG. The cardinality values should match what the `BackportSubscription` profile already specifies when it uses each extension.

### Extensions Requiring Cardinality

The following extensions are defined without explicit cardinality and need `^min` / `^max` added:

| Extension | FSH Name | Current Cardinality | Proposed Cardinality | Source File |
|-----------|----------|--------------------|--------------------|-------------|
| `backport-channel-type` | `BackportChannelType` | (none → implicit 0..*) | 0..1 | `BackportSubscription.fsh` |
| `backport-filter-criteria` | `BackportFilterCriteria` | (none → implicit 0..*) | 0..* | `BackportSubscription.fsh` |
| `backport-payload-content` | `BackportPayloadContent` | (none → implicit 0..*) | 0..1 | `BackportSubscription.fsh` |
| `backport-heartbeat-period` | `BackportHeartbeatPeriod` | (none → implicit 0..*) | 0..1 | `BackportSubscription.fsh` |
| `backport-timeout` | `BackportTimeout` | (none → implicit 0..*) | 0..1 | `BackportSubscription.fsh` |
| `backport-max-count` | `BackportMaxCount` | (none → implicit 0..*) | 0..1 | `BackportSubscription.fsh` |
| `backport-related-query` | `BackportRelatedQuery` | (none → implicit 0..*) | 0..* | `Extensions.fsh` |

**Rationale for proposed values:**
- Extensions used via `0..1` in the `BackportSubscription` profile (heartbeat period, timeout, max count, channel type, payload content) should be `0..1` at the definition level.
- `BackportFilterCriteria` is used as `0..*` in the profile, so the extension definition should allow `0..*`.
- `BackportRelatedQuery` is used with `notificationEvent` which can have multiple related queries, so `0..*` is appropriate.

### Risk Assessment

- **Low risk.** Adding explicit cardinality to extension definitions is a non-breaking, clarifying change. Profiles already enforce these cardinalities. This change makes the extension definitions self-documenting and consistent with how they are actually used.
- **No behavioral change** for systems that already follow the profile constraints.
- **Potential warning resolution**: Some validators may flag extensions without explicit cardinality; this fix would eliminate those warnings.

## Detailed Implementation Plan

### Step 1: Add cardinality to extensions in `BackportSubscription.fsh`

For each of the 6 extension definitions in this file, add the appropriate FSH cardinality rules. In FHIR Shorthand, this is done by adding rules to the extension root element.

**BackportChannelType** — add after the `insert StructureJurisdiction` line:
```fsh
Extension:   BackportChannelType
Id:          backport-channel-type
Title:       "Backported R5 Additional Channel Types"
Description: "Additional channel types not defined before FHIR R5."
* insert StructureJurisdiction
* ^context[0].type = #element
* ^context[0].expression = "Subscription.channel.type"
+ * . 0..1                          // ← ADD: cardinality
* value[x] only Coding
```

**BackportFilterCriteria** — cardinality `0..*` (multiple filters allowed):
```fsh
Extension:   BackportFilterCriteria
...
* insert StructureJurisdiction
* ^context[0].type = #element
* ^context[0].expression = "Subscription.criteria"
+ * . 0..*                          // ← ADD: cardinality
* value[x] only string
```

**BackportPayloadContent** — cardinality `0..1`:
```fsh
Extension:   BackportPayloadContent
...
* insert StructureJurisdiction
* ^context[0].type = #element
* ^context[0].expression = "Subscription.channel.payload"
+ * . 0..1                          // ← ADD: cardinality
* value[x] only code
```

**BackportHeartbeatPeriod** — cardinality `0..1`:
```fsh
Extension:   BackportHeartbeatPeriod
...
* insert StructureJurisdiction
* ^context[0].type = #element
* ^context[0].expression = "Subscription.channel"
+ * . 0..1                          // ← ADD: cardinality
* value[x] only unsignedInt
```

**BackportTimeout** — cardinality `0..1`:
```fsh
Extension:   BackportTimeout
...
* insert StructureJurisdiction
* ^context[0].type = #element
* ^context[0].expression = "Subscription.channel"
+ * . 0..1                          // ← ADD: cardinality
* value[x] only unsignedInt
```

**BackportMaxCount** — cardinality `0..1`:
```fsh
Extension:   BackportMaxCount
...
* insert StructureJurisdiction
* ^context[0].type = #element
* ^context[0].expression = "Subscription.channel"
+ * . 0..1                          // ← ADD: cardinality
* value[x] only positiveInt
```

### Step 2: Add cardinality to extension in `Extensions.fsh`

**BackportRelatedQuery** — cardinality `0..*`:
```fsh
Extension:   BackportRelatedQuery
...
* insert StructureJurisdiction
* insert ExtensionContext(SubscriptionStatus.notificationEvent)
* insert ExtensionContext(SubscriptionTopic.notificationShape)
* insert ExtensionContext(Basic.extension)
+ * . 0..*                          // ← ADD: cardinality
* extension contains
    queryType 0..1 MS and
    query 1..1 MS
```

### Step 3: Validate

1. Run the IG Publisher (SUSHI + IG Publisher) to ensure no errors are introduced:
   ```bash
   ./_genonce.sh   # or _genonce.bat on Windows
   ```
2. Verify the generated StructureDefinition JSON/XML files contain the correct `min` and `max` values on the extension root element.
3. Review the generated extension pages on the IG to confirm the cardinality is displayed correctly.
