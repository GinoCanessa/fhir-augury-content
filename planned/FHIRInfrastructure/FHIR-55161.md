## Resolution Summary

Resolution: **Not Persuasive with Modification**.

Verbatim resolution description: *"Will correct the binding to be to
[https://build.fhir.org/valueset-extended-resource-types.html](https://build.fhir.org/valueset-extended-resource-types.html)
which is version-independent."*

Concretely, the committee declined the reporter's request to add a literal
`unknown`/`other` escape‑valve concept to the bound value set. Instead, the
required binding on `CompartmentDefinition.resource.code` will be re‑pointed
from the version‑locked `http://hl7.org/fhir/ValueSet/resource-types` to the
version‑independent `http://hl7.org/fhir/ValueSet/extended-resource-types`.
That value set already includes (a) the current core resource types, (b) the
historical / renamed FHIR resource types via
`http://hl7.org/fhir/fhir-old-types`, and (c) additional resource types via
`http://hl7.org/fhir/tools/CodeSystem/additional-resources` — collectively
serving as the escape valve.

## Feature Proposal

### Problem Statement

`CompartmentDefinition.resource.code` is mandatory (`min=1`) with a `required`
strength binding to `http://hl7.org/fhir/ValueSet/resource-types` (see
`source/compartmentdefinition/structuredefinition-CompartmentDefinition.xml`,
element `CompartmentDefinition.resource.code`, binding block). That value
set is locked to the resource types defined in *this* version of FHIR and
contains no `unknown` / `other` escape concept. As a consequence:

- An author who needs to describe compartment membership for a resource type
  defined in a prior FHIR version, in a downstream IG, or in an HL7 tooling
  "additional resources" code system has no valid code to populate the
  element with, even though the element is mandatory.
- Servers/profiles that span FHIR versions cannot author a single
  CompartmentDefinition that survives renames or removals of resource types.

The reporter (Lloyd McKenzie) asked for an `unknown`/`other` concept in the
bound value set as the escape valve.

### Proposed Change

A single, surgical change to the `CompartmentDefinition` StructureDefinition:

- **Element:** `CompartmentDefinition.resource.code`
- **Change:** retarget the `binding.valueSet` from
  `http://hl7.org/fhir/ValueSet/resource-types` to
  `http://hl7.org/fhir/ValueSet/extended-resource-types`.
- **Binding strength:** unchanged (`required`).
- **Binding name extension** (`elementdefinition-bindingName`): change the
  `valueString` from `ResourceType` to `ExtendedResourceType` so the binding
  name continues to match the value set it references (this matches the
  naming pattern used by the existing target value set, whose `name` is
  `ExtendedResourceTypes`).
- **Binding `description`:** change the description from
  *"One of the resource types defined as part of this version of FHIR."* to
  something that reflects the broader scope, e.g.
  *"A FHIR resource type — including resource types from previous versions of
  FHIR and additional resource types defined by HL7 tooling — that is
  supported by the server."* (Exact wording is editorial; it must no longer
  imply "this version of FHIR" since the new binding is intentionally
  version-independent.)
- **`isCommonBinding`** extension on the binding: leave as `true` (the
  binding is still expected to be reused).

The cardinality, type (`code`), `isSummary`, `short`, and `definition` of the
element itself do **not** change.

No new value set, code system, extension, or example needs to be created —
`http://hl7.org/fhir/ValueSet/extended-resource-types` already exists in this
repo at
`source/capabilitystatement/valueset-extended-resource-types.xml` and already
unions `ValueSet/resource-types`, `CodeSystem/fhir-old-types`, and the
tooling `additional-resources` code system. The existing
`compartmentdefinition-example.xml` (which uses core resource codes such as
`Device`, `Communication`, `CommunicationRequest`) remains valid under the
new binding because the new value set is a strict superset of the old one.

### Design Rationale

The committee chose a value-set retarget over the reporter's
`unknown`/`other` proposal for several reasons that are visible in the
resolution wording and in the structure of the existing
`extended-resource-types` value set:

- Adding a literal `unknown` / `other` code to a *required*, version-locked
  binding bound to a code-typed element would invite authors to use a
  meaningless sentinel for cases where they actually do know the resource
  type but it just isn't in core (e.g. a renamed type or an IG-defined
  type). That degrades search/compartment semantics rather than fixing them.
- `extended-resource-types` is purpose-built for exactly this scenario:
  current core types ∪ historical FHIR types ∪ HL7 tooling additional
  resources. Its description (in `valueset-extended-resource-types.xml`)
  states that "the binding to additional resources is not version fixed,
  and will change over time", which makes the binding genuinely
  version-independent without weakening it from `required`.
- Other CanonicalResource elements that name resource types (e.g. on
  `Consent`, `ImplementationGuide`, `MessageDefinition`, `Questionnaire`,
  `ExampleScenario`, `CodeableReference`, `Reference`, see `Select-String`
  results in Step 4 below) also bind to `ValueSet/resource-types`. The
  resolution intentionally does **not** ripple this change to those
  elements; it is scoped to the one element the ticket called out.

### Breaking Changes

Honestly assessed: **technically backward-compatible at the instance
level, but a binding-target change with semantic implications.**

- All instances valid against the previous binding remain valid against the
  new one, because `extended-resource-types` is a superset of
  `resource-types` (it `<include>`s `ValueSet/resource-types` directly, plus
  `fhir-old-types` and the tooling `additional-resources` code system —
  see `source/capabilitystatement/valueset-extended-resource-types.xml`).
  No existing CompartmentDefinition example or test instance becomes
  invalid.
- However, validators and code generators that pinned the bound value set
  by URL (`http://hl7.org/fhir/ValueSet/resource-types`) — e.g. to generate
  a `ResourceType` enum used to type `resource.code` in client SDKs — will
  see the binding URL change. Downstream tooling that auto-generates
  enums will produce a different (broader) enum. Profilers that have
  re-bound or constrained `resource.code` against the old value set will
  need to retarget.
- The `bindingName` (used by some code generators to name the enum) is
  proposed to change from `ResourceType` to `ExtendedResourceType`, which
  *is* a source-incompatible rename for those generators. Tooling owners
  should be alerted.

  definitions and consumers that resolve `binding.valueSet` dynamically
  should continue to work; only the URL of the bound value set changes.
