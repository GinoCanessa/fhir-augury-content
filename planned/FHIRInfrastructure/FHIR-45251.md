## Resolution Summary

Resolution: **Persuasive**. The paragraph in `operations.html` that currently
uses the non-conformance word *"may"* to describe GET invocation of stateless,
primitive-parameter operations is confusing because it is immediately followed
by a `SHALL support` server requirement. The approved fix is a pure-wording
change: replace the informative *"may be invoked"* with non-conformance prose
that states the operation **is** invokable via HTTP GET, preserving the
existing server `SHALL` requirement. The ticket proposes:

> If all the parameters for the operation are primitive types and the operation
> has affectsState = false, the operation **is also invokable** by performing
> an HTTP GET operation where all of the values of the parameters are appended
> to the URL in the search portion of the URL (e.g. after the '?' character).
> Servers SHALL support this method of invocation.

**Applied state:** This resolution has already been implemented on `master` of
`HL7/fhir` in commit [`18b4d75a69`](https://github.com/HL7/fhir/commit/18b4d75a69)
("FHIR-45251 - Remove confusing non-conformance 'may' from operations
documentation", 2025-11-03). The committed wording is semantically equivalent
but slightly varied: *"the operation **is also able to be invoked** by
performing an HTTP GET operation…"* (see `source/operations.html` line 240 in
clone HEAD `db79dcd`). The `AwaitingMerge` label on the Jira ticket is stale
relative to clone HEAD. This plan documents the change as designed; no further
normative content work is required unless the committee wants the exact
"invokable" wording from the ticket.

## Feature Proposal

### Problem Statement

The "Executing an Operation" section of `operations.html` contains a paragraph
that mixes an informative *"may"* with a normative *"SHALL"* in consecutive
sentences:

> …the operation **may be invoked** by performing an HTTP GET operation where
> all of the values of the parameters are appended to the URL in the search
> portion of the URL (e.g. after the '?' character). **Servers SHALL support**
> this method of invocation.

Per FHIR conformance conventions, "may" is a non-conformance (informative)
word, but the sentence it appears in is describing a mechanism that the very
next sentence requires servers to support. Readers (including the implementer
in Zulip thread [*"Understanding GET invoked extended operations"*](https://chat.fhir.org/#narrow/stream/179166-implementers))
have flagged this as contradictory — "may" implies optional, while "SHALL"
mandates support. The fix is a small prose correction on a single
specification page.

### Proposed Change

Edit the single paragraph in `source/operations.html` (currently lines
237–243 at clone HEAD `db79dcd`) so the sentence no longer uses a
non-conformance modal verb on the client side of the description. The ticket
proposal:

- **From:** "the operation **may be invoked** by performing an HTTP GET
  operation …"
- **To:** "the operation **is also invokable** by performing an HTTP GET
  operation …"

The trailing "Servers SHALL support this method of invocation." is retained
verbatim. No other paragraphs, examples, or anchors on the page change.

Equivalent already-merged variant (commit `18b4d75a69`): "the operation **is
also able to be invoked** by performing an HTTP GET operation …". Both drop
the non-conformance "may" while preserving meaning; either satisfies the
resolution.

### Design Rationale

- **Minimal, prose-only change.** The resolution type is *Technical
  Correction*; scope is deliberately confined to removing a non-conformance
  word. No normative behaviour, cardinality, binding, or schema changes are
  implied.
- **Keep the server `SHALL`.** Only the client-side phrasing is ambiguous. The
  server requirement to support GET invocation is already correctly normative
  and must remain.
- **Consistency with the rest of `operations.html`.** Other paragraphs on the
  same page (e.g., lines 63–64 and 98–103) already use a correct mix of
  informative *"MAY be invoked using GET"* on the client side with the
  conformance conditions spelled out; the edited paragraph simply brings the
  detailed restatement in line by avoiding lower-case "may" right before a
  `SHALL`.
- **Out-of-scope suggestions deferred.** The reporter's own follow-up comment
  about "add content to the HTTP page detailing HTTP requirements — e.g., if
  GET and POST are both required, can a server return HTTP 405 for one or the
  other … also document expectations for executing during batch/transaction
  bundles" is explicitly not part of this technical correction and should be
  tracked as a separate ticket if desired.
