## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"Will update to reference a location in THO (though we have to get it there first and prototype to make sure that will work)."*

The ticket flags an inline `todo:` marker on the Resource page (the page is rendered from `source/resource/resource-introduction.xml`) inside the *"Additional Resources defined Outside This Specification"* section. The marker currently reads:

> "HL7 publishes the list of approved resource at **(somewhere?)** (todo: this needs further work, but at this time the functional list is published here: [IG Registry](https://raw.githubusercontent.com/FHIR/ig-registry/refs/heads/master/additional-resources.json))."

The agreed-upon resolution is to remove the `todo:` admission and the `(somewhere?)` placeholder, replacing them with a stable reference to a location in **terminology.hl7.org (THO)** that lists / registers the HL7-approved Additional Resources. ## Feature Proposal

### Problem Statement

The Resource specification page (`resource.html`, sourced from
`source/resource/resource-introduction.xml`) is the entry point for the
abstract `Resource` type and explains the rules for *Additional Resources*
defined outside the FHIR Core specification. The current copy admits its
own incompleteness with a `todo:` parenthetical and a `(somewhere?)`
placeholder for the canonical list of HL7-approved additional resources,
falling back to a raw GitHub URL into the `FHIR/ig-registry` repo.

This is poor balloted-spec hygiene:

- A `todo:` marker in normative-adjacent prose signals that the spec is
  unfinished to readers and ballot reviewers.
- The fallback link is a raw `githubusercontent.com` URL into a non-HL7
  branch of an external GitHub project â€” fragile, unbranded, and not a
  long-term canonical citation.
- Implementers who follow the rules in the same section ("Additional
  Resources to be used like this SHALL be registered with HL7" /
  "Unapproved resources SHALL NOT be used") have no clean way to look up
  the authoritative HL7 list.

The reporter (Lloyd McKenzie, FHIR Infrastructure) flagged this via the
"Incomplete content indicated on" automated sweep that searches authored
spec pages for words like `todo`, `tbd`, `somewhere`, etc. (Jira-imported
comment 154.)

### Proposed Change

Make a single, targeted edit to **`source/resource/resource-introduction.xml`**
(no other files):

1. Remove the `(somewhere?)` placeholder and the parenthetical
   `(todo: this needs further work, â€¦)` admission from the paragraph that
   currently spans lines 63â€“67.
2. Replace them with a normative-tone reference to the THO-hosted list
   of HL7-approved Additional Resources. The exact target URL is to be
   supplied by Vocabulary WG once the resource is published to THO; the
   placeholder used in the patch should be the canonical
   `http://terminology.hl7.org/...` URL agreed during the prototype
   (see Open Questions for the candidate forms).
3. Retain the IG Registry JSON link only as an *implementation note*
   pointing at the machine-readable mirror, not as the primary citation.

Concretely, the existing paragraph:

```html
<p>
HL7 publishes the list of approved resource at <b>(somewhere?)</b> (todo: this needs further work, but at
this time the functional list is published here: <a href="https://raw.githubusercontent.com/FHIR/ig-registry/refs/heads/master/additional-resources.json">IG Registry</a>).
If you wish to register such a resource, the process starts by making a proposal <a href="https://chat.fhir.org/#narrow/channel/491058-Additional-Resource-Proposals">here</a>.
</p>
```

â€¦should become (illustrative â€” exact THO URL TBC by Vocab WG):

```html
<p>
HL7 publishes the list of approved Additional Resources at
<a href="http://terminology.hl7.org/AdditionalResources">terminology.hl7.org</a>.
If you wish to register such a resource, the process starts by making a proposal
<a href="https://chat.fhir.org/#narrow/channel/491058-Additional-Resource-Proposals">on the
Additional Resource Proposals stream</a>.
</p>
<p>
A machine-readable mirror of the same list is maintained in the
<a href="https://github.com/FHIR/ig-registry">FHIR IG Registry</a>
(<a href="https://raw.githubusercontent.com/FHIR/ig-registry/refs/heads/master/additional-resources.json"><code>additional-resources.json</code></a>).
</p>
```

This deletes both the `(somewhere?)` placeholder and the `todo:` text,
satisfies the "Incomplete content" sweep, and gives readers a stable HL7
URL while preserving the machine-readable link implementers already use.

### Design Rationale

- **Why THO, not the FHIR Core spec.** The list of *Additional Resources*
  is by definition out-of-core and grows asynchronously from FHIR Core
  ballots. Hosting it on `terminology.hl7.org` (HL7's terminology /
  registry-content publication site) avoids re-balloting the core spec
  every time a new Additional Resource is approved. This is consistent
  with how other HL7-managed registries are surfaced from the spec.
- **Why keep the `ig-registry` link.** It is the existing
  machine-readable source of truth used by tooling; demoting it to an
  "also see" note preserves implementer continuity without making it the
  primary citation.
- **Why a documentation-only change.** Nothing in the rules listed
  immediately above the affected paragraph (`SHALL be registered`,
  `SHALL NOT be used`, naming, StructureDefinition publication) changes
  â€” only the pointer to *where* the list lives. No StructureDefinition,
  ValueSet, search parameter, or invariant is touched.
- **Alternatives considered.** Embedding the list directly in
  `resource.html` was rejected by the resolution (it would freeze the
  list per ballot). A raw `chat.fhir.org` link as the primary citation
  was rejected because the chat stream is for *proposals*, not the
  authoritative published list.
