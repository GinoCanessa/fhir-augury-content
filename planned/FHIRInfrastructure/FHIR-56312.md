## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"Will do this. Will also define unit tests in the shared test suite."*

Concretely, the resolution approves two things:

1. Extend the FHIR-flavour FHIRPath function `htmlChecks()` so that, in addition to accepting a single `xhtml` element, it also accepts items of type `string`. When invoked on a string, the string is parsed as the content of an XHTML `<div>` and the function returns `false` if the string does not parse as valid HTML or if it fails the HTML-usage rules referenced from `narrative.html#rules`. It continues to return empty on unsupported element kinds or on a collection of elements.
2. Add unit tests that exercise the new string behaviour to the shared FHIRPath test suite (used cross-implementation).

The reporter also flagged a compatibility note in a comment: existing servers that historically return `false` (instead of empty) when `htmlChecks()` is invoked on a string will now diverge from the spec. This is informational, not a blocker, but must be surfaced in the change description.

## Feature Proposal

### Problem Statement

The FHIR profile of FHIRPath defines `htmlChecks() : Boolean` (see `source/fhirpath.html`, section `fn-htmlChecks`). As specified today, the function is only meaningful on a single `xhtml` element — it returns true/false against the Narrative HTML rules for that element and returns *empty* for anything else. In practice, HTML content is not confined to `Narrative.div`: several extensions carry HTML fragments, and profiles may want to place HTML into a normal element (e.g., `Parameters.parameter.value[x] : string`, `Observation.value[x] : string`). Today there is no in-spec way to assert the same safety checks against those string-valued HTML fragments — invariant authors must either hand-roll regexes or give up on the check.

The Zulip thread [#fhirpath > Expanding htmlChecks()](https://chat.fhir.org/#narrow/channel/179266-fhirpath/topic/Expanding.20htmlChecks.28.29) reached consensus that the cleanest fix is to extend `htmlChecks()` itself so the same function can be applied to `string`-typed values, with the string being parsed as the contents of a `<div>` before the Narrative rules are evaluated.

### Proposed Change

**FHIR Core (`HL7/fhir`, `source/fhirpath.html`, section `fn-htmlChecks`)** — update the prose definition of `htmlChecks()` to add string support. The normative wording is:

> When invoked on a single [xhtml](narrative.html#xhtml) element, the function returns true if the [rules around HTML usage](narrative.html#rules) are met, and false if they are not. If invoked on items of type 'string', the contents of the string are parsed as the content of a `<div>` and the function returns false if the string does not parse as valid HTML or fails the rules around HTML usage. The return value is empty on any other kind of element or if invoked on a collection of elements.

This is exactly the text applied in commit `149475c` (see Impact Analysis → Related PRs/Commits below). That commit is already on `master` (the specification's default branch) at the current clone HEAD `db79dcd`, so the main prose task is already landed; the remaining work for this ticket is the catch-up items listed in the Implementation Plan.

**FHIR Core — ballot-notes block (`source/fhirpath.html`, top of the page, block `#bn1`)** — the "Compatible, Substantive Changes" bullet currently points at the wrong Jira ticket:

```html
<li><a href="https://jira.hl7.org/browse/FHIR-56303">FHIR-56303</a>: Allow htmlChecks()
to be invoked on strings in addition to xhtml</li>
```

The correct change request is **FHIR-56312**, not FHIR-56303. This bullet was added by follow-up commit `8d42f18` ("Added change description") and must be corrected before ballot.

**Shared FHIRPath test suite** — add unit tests covering the new string behaviour. The "shared test suite" referenced by the resolution is the cross-implementation FHIRPath test harness consumed by engines such as HAPI, Firely, pathling, and fhirpath.js (externally hosted; not one of the cached repos). Each test case must specify: input expression, input resource/primitive context, expected result. Cover at least:

- `'<div xmlns="http://www.w3.org/1999/xhtml">hello</div>'.htmlChecks()` → `true`
- `'<div xmlns="http://www.w3.org/1999/xhtml"><script>bad</script></div>'.htmlChecks()` → `false` (rule violation)
- `'not <valid'.htmlChecks()` → `false` (un-parseable)
- `'<p>plain</p>'.htmlChecks()` → `true` (parsed as div contents; element `<p>` is permitted by Narrative rules)
- `{}.htmlChecks()` → empty (absent input)
- `('a' | 'b').htmlChecks()` → empty (collection input)
- Existing `Narrative.div.htmlChecks()` happy / unhappy paths (to assert no regression for the xhtml path).

**Compatibility note (ballot callout in `source/fhirpath.html` or companion R6 migration docs)** — note that previously-shipped servers may return `false` (rather than empty) when `htmlChecks()` is invoked on a string, so deployed clients that relied on that divergent behaviour will see a spec-vs-implementation mismatch until they upgrade. Lloyd's comment frames this as "may be frustrating from a compatibility perspective" but ultimately a safety improvement; reflect that tone.

### Design Rationale

- **Minimal surface-area change.** Keeping the operation on a single function (`htmlChecks()`) rather than introducing a new `htmlChecksString()` preserves the FHIRPath ergonomics and means that invariants can be written uniformly regardless of whether the subject element is an `xhtml` node or a string value.
- **"Parsed as the content of a div" semantics** are the natural extension: the Narrative rules are defined over XHTML with `<div>` as the implicit root, so adopting the same wrapper makes the string and xhtml code paths equivalent post-parse.
- **Return-empty-on-collection** continues to match existing FHIRPath conventions and avoids ambiguity about how to aggregate per-item truthiness.
- **Unit tests in the shared suite** are required because multiple implementations host `htmlChecks()`; without a shared test vector, behaviour will drift across engines and the "old servers return `false` instead of empty" compatibility gap surfaced in Lloyd's comment will recur.

### Breaking Changes

- **Compatible, substantive** per the ballot-note classification. Under the R6 normative process this is not a breaking *specification* change — strings previously yielded empty (no behaviour promised) and will now yield a defined boolean. However:
  - Deployed servers that previously returned `false` for string input (as documented in Lloyd's comment) will become spec-non-conformant until upgraded. Downstream invariant authors who encoded `... .htmlChecks() = true` relying on the old "always false" behaviour will see invariants that previously trivially failed now trivially pass for valid HTML strings.
  - No element, cardinality, type, or terminology-binding changes. Search parameters unaffected.

### Related Specifications

- **`Narrative` datatype** (`source/datatypes/narrative.*`) — `htmlChecks()` continues to reference `narrative.html#rules` for the permitted-HTML ruleset; no textual change is required in the Narrative page, but reviewers should re-read it to confirm the "content of a div" wrapping semantics are consistent with `Narrative.div` expectations.
- **Any invariant using `htmlChecks()`** — today limited to invariants on `xhtml` elements (`txt-1` / `txt-2` on `Narrative.div`). No FHIR Core invariant currently invokes `htmlChecks()` on a string; downstream profiles and extensions can now begin doing so.
- **Extensions and profiles** carrying HTML in string elements (mentioned in the ticket: Parameters, Observation.value[x]) gain the ability to author safety invariants; no automatic core update is required.
