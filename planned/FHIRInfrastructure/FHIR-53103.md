## Resolution Summary

**Resolution: Persuasive.** Per the resolution description: *"Move the
VersionAlgorithm code system and value set to THO. Will keep the code
system URL the same."*

Concretely:

- The `VersionAlgorithm` `CodeSystem` and its companion `ValueSet`
  currently authored in HL7/fhir at
  `source/canonicalresource/codesystem-version-algorithm.xml` and
  `source/canonicalresource/valueset-version-algorithm.xml` must be
  removed from FHIR Core and re-homed in THO (HL7/UTG) so they can
  evolve independently of the base specification.
- The CodeSystem canonical URL is to remain
  `http://hl7.org/fhir/version-algorithm` (per the resolution text).
  By implication the ValueSet URL
  `http://hl7.org/fhir/ValueSet/version-algorithm` is also retained,
  so all extensible bindings already present on `CanonicalResource`
  and the ~28 inheriting resources continue to resolve unchanged after
  the move.
- The OID registrations in `source/oids.ini`
  (`http://hl7.org/fhir/ValueSet/version-algorithm = 2.16.840.1.113883.4.642.3.3103`
  and `http://hl7.org/fhir/version-algorithm = 2.16.840.1.113883.4.642.4.1989`)
  must travel with the artifacts to UTG so the OID/URL mapping is
  preserved and the entries no longer assert ownership inside FHIR
  Core's `oids.ini`.

## Feature Proposal

### Problem Statement

`VersionAlgorithm` is bound `extensible` everywhere it is used (the
binding originates on `CanonicalResource.versionAlgorithm[x]` and is
inherited by every canonical-resource StructureDefinition in the spec —
ActivityDefinition through ValueSet, see Affected Files below). Per
TSMG / UTG policy as articulated by Bryn Rhodes, Grahame Grieve, and
Lloyd McKenzie in the Zulip thread
[#TSMG > Terminology moved from the base spec to THO](https://chat.fhir.org/#narrow/stream/TSMG/topic/Terminology%20moved%20from%20the%20base%20spec%20to%20THO/near/547354094):

- Terminology that is **not version-specific** and is bound
  **extensible** (rather than `code`/`required` or `experimental`)
  should live in THO so it can be maintained on a faster cadence than
  the base FHIR specification, and so additional codes can be added
  without a normative ballot.
- Keeping `VersionAlgorithm` in core means future additions are
  effectively gated on FHIR ballots even though the binding is
  extensible — the wrong outcome for a vocabulary that describes a
  general-purpose mechanism (semver, integer, alphabetical, natural,
  date, etc.).

### Proposed Change

1. **Add the artifacts to HL7/UTG (THO source-of-truth).** Create
   `input/sourceOfTruth/fhir/codeSystems/CodeSystem-version-algorithm.xml`
   and `input/sourceOfTruth/fhir/valueSets/ValueSet-version-algorithm.xml`
   in HL7/UTG, derived from the existing FHIR Core files. Per the
   resolution, **retain** the canonical URLs
   `http://hl7.org/fhir/version-algorithm` and
   `http://hl7.org/fhir/ValueSet/version-algorithm` (this is an
   exception to the standard UTG convention of using
   `http://terminology.hl7.org/...` URLs — see Open Questions).
   Preserve identifiers, OIDs (`2.16.840.1.113883.4.642.4.1989` for the
   CS, `2.16.840.1.113883.4.642.3.3103` for the VS), `caseSensitive`,
   `content="complete"`, `status`, `experimental="false"`,
   `description`, and the full `<concept>` list (semver, integer,
   alpha, natural, date, …). Keep `VersionAlgorithm` as `name` /
   `title` and tag the WG extension as `fhir`.
2. **Register them in the UTG control manifests.** Add `<entry>`
   blocks for `CodeSystem/version-algorithm` and
   `ValueSet/version-algorithm` to
   `input/sourceOfTruth/control-manifests/fhir-Rendering.xml`
   (alphabetical order, alongside neighbouring entries such as
   `version-independent-set` if present), and to
   `input/sourceOfTruth/control-manifests/fhir-Normative.xml` to
   match the `normative` standards-status currently asserted in the
   FHIR Core copies.
3. **Remove the artifacts from HL7/fhir.** Delete
   `source/canonicalresource/codesystem-version-algorithm.xml` and
   `source/canonicalresource/valueset-version-algorithm.xml`. Drop
   the two `version-algorithm` entries from `source/oids.ini`. Do
   **not** edit any of the StructureDefinitions that bind the value
   set: because the resolution preserves the URL, every
   `<binding><valueSet value="http://hl7.org/fhir/ValueSet/version-algorithm"/>`
   in `CanonicalResource.xml` and the ~28 inheriting resources
   continues to resolve correctly via the THO copy.
4. **Update FHIR Core dependency wiring** so the publisher pulls
   `VersionAlgorithm` from the THO package. In a FhirCore build, this
   typically means adding / bumping a dependency declaration for the
   THO package in the publisher configuration (`source/publish.ini`
   / equivalent) so that the `extensible` bindings continue to
   validate against the THO-resolved value set; verify the exact
   declaration with the publisher maintainers before merging (see
   Open Questions).

### Design Rationale

- The Zulip discussion converged that artifacts that are extensible
  bindings (not `code`/`required`, not `experimental`) belong in THO,
  precisely so they can grow without a base-spec ballot. Bryn Rhodes
  (the ticket reporter) cited this as the rationale, and Lloyd
  McKenzie reaffirmed: *"once their maturity level is FMM3+ need to be
  external or receive special TSMG dispensation to not live in THO."*
  `VersionAlgorithm` already carries `<fmm value="1"/>` and
  `<standards-status value="normative"/>` in core, so this is also a
  forward-compatibility move ahead of R6 normative-lock.
- Keeping the CodeSystem URL unchanged (per the resolution text)
  avoids breaking instances and downstream profiles that already
  validate against `http://hl7.org/fhir/version-algorithm`. This is
  the explicit author-of-resolution choice and overrides the general
  TSMG preference (also expressed in the same Zulip thread by Grahame
  and Lloyd) that THO content should use `terminology.hl7.org`-rooted
  URLs. Implementers should escalate to TSMG / FMG before merging if
  this exception will not be granted (see Open Questions).
- The alternative — leaving `VersionAlgorithm` in core — was rejected
  because the binding is `extensible` and the vocabulary is plainly
  not version-specific (it describes how *any* version string is
  ordered), so the policy cleanly applies.

### Breaking Changes

- **No instance-level breaking changes** if the resolution is followed as written: the CodeSystem URL `http://hl7.org/fhir/version-algorithm` and the ValueSet URL `http://hl7.org/fhir/ValueSet/version-algorithm` are preserved, so existing instances and profiles that bind to or use these URLs continue to validate.
- **Packaging-level change:** consumers that previously loaded `VersionAlgorithm` from the FHIR Core package will, after this change, need to also resolve it from the THO package. Tooling that pins to a single package and does not load THO transitively may report an unresolved binding until its dependency list is updated.
- **Policy-level risk:** retaining the `http://hl7.org/fhir/...`-rooted URL on a THO-resident artifact deviates from the established UTG/THO convention surfaced in the Zulip thread. If TSMG/FMG do not grant the exception, the URL will need to change to `http://terminology.hl7.org/CodeSystem/version-algorithm` (CS) and `http://terminology.hl7.org/ValueSet/version-algorithm` (VS), which **would** be a breaking change for every existing instance that uses a literal `http://hl7.org/fhir/version-algorithm` `Coding.system`.

## Implementation Plan

- TSMG/FMG sign-off on retaining the `http://hl7.org/fhir/...` URLs on a THO-resident CodeSystem/ValueSet. The Zulip thread reflects open dispute on this exact point; implementers should not start UTG authoring until this is confirmed in writing.
- A `repo-analysis HL7/UTG` run to refresh the HL7/UTG briefing (currently missing). The paths in this plan are derived directly from the clone but a full briefing will surface UTG-specific gotchas (e.g., release-tracking entries, history files) that this skill could not authoritatively enumerate.
- Confirmation from the FHIR publisher maintainers of the exact dependency-declaration mechanism for THO content in the FHIR Core build (see Open Questions).

   - Action: Execute `_genonce.bat` (Windows) or `_genonce.sh` (Linux/macOS) and confirm the new artifacts render and pass UTG QA.

- **Re-validate every canonical resource instance/example in HL7/fhir** (every `*-example*.xml` under `source/<resource>/`) that populates `versionAlgorithm[x]` — they should still validate against the now-THO-resident `VersionAlgorithm` ValueSet via the preserved URL.
