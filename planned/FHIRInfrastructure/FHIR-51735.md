## Resolution Summary

The decimal-primitive regex contains an extra `}` after the exponent quantifier:

```
-?(0|[1-9][0-9]{0,17})(\.[0-9]{1,17})?([eE][+-]?[0-9]{1,9}})?
                                                       ^^ extra }
```

The regex must be corrected to:

```
-?(0|[1-9][0-9]{0,17})(\.[0-9]{1,17})?([eE][+-]?[0-9]{1,9})?
```

Resolution recorded as **Persuasive** with no inline resolution-description text. The
ticket's Description body is essentially "As reported by Stefan Adolf by email" plus
an inlined screenshot, so all detail comes from the title (`Extra '}' in decimal regex`)
and the regex actually present in the published spec. The companion ticket
[FHIR-53492 — Incorrect Regex for Decimal Datatype](https://jira.hl7.org/browse/FHIR-53492)
(reporter SeanTan, resolution **Persuasive with Modification**, status **Applied**) has
already landed the fix on `master` in commit `2a1360e0` (2025-12-15), which edits exactly
the two authoring locations described below. FHIR-51735 is functionally the same defect
as FHIR-53492 and is satisfied by that same fix.

## Feature Proposal

### Problem Statement

The published R5 StructureDefinition for the `decimal` primitive type carries an
invalid regex constraint with an unbalanced `}` at the end of the exponent group
(`{1,9}}` instead of `{1,9}`). Strict regex engines will treat the trailing `}` as a
literal or reject the pattern as malformed; lenient engines silently accept it. Either
way, validators that round-trip the regex from the StructureDefinition's
`StructureDefinition/regex` extension can warn or fail on the supposedly-canonical
decimal pattern, undermining trust in the published primitive types and producing
spurious validator output for otherwise-valid `decimal` values such as `1.5e2`.

The defect originates in the FHIR Core authoring workbook (`source/datatypes/primitives.xml`,
the spreadsheet that drives the `decimal` row's regex column) and is reflected in the
narrative XML schema page (`source/xml.html`). From there it propagates into every
publisher-generated artifact: `profiles-types.xml`, `fhir-all-xsd`, `fhir.schema.json`,
the data-types reference page, and the cross-version snapshots. The version-history
copy in `tools/history/release5/profiles-types.xml` shows the bug's footprint in two
places (the snapshot and differential `regex` extensions on `decimal.value`, lines 1906
and 1937 of the cached clone).

### Proposed Change

Correct the decimal regex by removing the spurious `}`:

- **Element:** `decimal.value` (the `value` attribute of the `decimal` primitive type)
- **Constraint type:** `StructureDefinition/regex` extension (and `xs:pattern` in the schema)
- **Old value:** `-?(0|[1-9][0-9]{0,17})(\.[0-9]{1,17})?([eE][+-]?[0-9]{1,9}})?`
- **New value:** `-?(0|[1-9][0-9]{0,17})(\.[0-9]{1,17})?([eE][+-]?[0-9]{1,9})?`

The fix is a single-character delete in two authoring files:

1. `source/datatypes/primitives.xml` line 134 — the `decimal` row's regex cell in the
   spreadsheet that the publisher reads.
2. `source/xml.html` line 314 — the narrative `xs:pattern` example shown on the XML
   formats page so the documentation matches the schema.

All publisher-generated artifacts (StructureDefinitions, XSDs, JSON schemas, history
snapshots) regenerate from those two sources on the next publish.

### Design Rationale

- **Minimal-impact correction.** The semantic intent of the regex is unchanged; the
  trailing `}` is an obvious typo, not a deliberate constraint, so a literal one-character
  fix preserves every conformant decimal value and only narrows the accepted set on
  strict engines that previously refused the malformed pattern.
- **Authoring-root only.** Per the `HL7/fhir` repo briefing, `source/datatypes/primitives.xml`
  is the authoritative spreadsheet for primitive metadata and `source/xml.html` is an
  authored narrative page. Both sit under the `source/` authoring root. Generated areas
  (`tools/history/...`, `<snapshot>` blocks, `schema/`, `qa/`) are not edited directly —
  the publisher will regenerate them from the corrected source.
- **Consistency with prior precedent.** FHIR-53492 fixed the identical defect using
  exactly this approach (commit `2a1360e0`, two-line patch). Replaying / verifying that
  fix is the only sensible path; introducing any larger regex rewrite would invent scope
  the resolution does not authorise.

- **Change Scope:** Minor (two-line typo fix, no semantic change).

### Breaking Changes

No breaking changes. The corrected regex is strictly more well-formed than the
existing one; any decimal value previously accepted by lenient engines (which treated
the extra `}` as a literal that never matched in the captured group) remains accepted,
and strict engines that rejected the malformed pattern will now compile and accept
the same conformant decimals. There is no impact on instances that conform to the
intended decimal grammar. Validators that hardcoded the old pattern string to compare
against the StructureDefinition will see a one-character drift and may need a regen.

### Related Specifications

- The XML schema (`fhir-all-xsd`, `fhir-single.xsd`) carries an `xs:pattern` for
  `decimal-primitive` that mirrors the same regex; it regenerates from the spreadsheet
  fix.
- The JSON schema (`fhir.schema.json`) emits a `pattern` keyword with the same regex
  for `decimal`; same regeneration path.
- The data-types reference (`source/datatypes.html`) includes the regex in the
  primitive-type tables; regenerated by the publisher.
- `decimal` is referenced from many resources (`Quantity.value`, `Money.value`,
  `MonetaryComponent.amount.value`, `SampledData.data` decimal token grammar,
  `Observation.valueQuantity`, …). None of those need changes — they reference the
  type, not the literal regex.

### Testing Considerations

- **What to revalidate.** Re-run the publisher's primitive-type validation pass and
  any QA step that compiles the regex extracted from `decimal.value`'s
  `StructureDefinition/regex` extension. Re-run examples that exercise decimal-typed
  elements (Quantity, Money, MonetaryComponent, Observation.valueQuantity,
  SampledData.data tokens) to confirm no false rejections.
- **Example instances.** No new examples are needed. Existing examples already cover
  scientific-notation decimals (e.g., `Observation` examples with very small/large
  quantities); confirm they still validate.
- **Edge cases.** Verify the boundary: 17 digits before the decimal, 17 fractional
  digits, 9-digit exponent magnitudes, sign on exponent. Confirm `1e1234567890` (10
  exponent digits) is rejected, and `1e123456789` (9 digits) is accepted.
- **Interoperability.** Validators that hardcode the regex string for comparison
  against the StructureDefinition extension should be regenerated against the new
  StructureDefinition. Note in release notes (R6 / next R5 corrigendum) that the
  regex string changed by exactly one character.

### Open Questions

- **R5 corrigendum vs. R6-only.** The corrected regex is on `master` (R6). Whether
  the published R5 errata page or a republished R5 snapshot needs to be issued is a
  publishing-process decision for FHIR Infrastructure / FMG. The briefing flags
  `tools/history/release5/` as generated, so any R5 republish goes through that
  pipeline rather than a hand-edit.
- **Resolution-description text missing.** FHIR-51735's snapshot exposes no
  `resolution_description`; the only narrative is the title and "As reported by
  Stefan Adolf by email" plus an inlined screenshot. This plan therefore reads the
  intent from the title and the actual diff of the companion ticket FHIR-53492. If a
  reviewer surfaces a different intent from the original email thread, the plan
  should be re-checked against that text.
- **Anchoring `^`/`$`.** The corrected pattern is unanchored. FHIR primitive regexes
  are intentionally unanchored because the consuming engine anchors them
  contextually. No change requested here, but worth flagging if any consumer is
  observed to misuse the pattern (out of scope for FHIR-51735).
