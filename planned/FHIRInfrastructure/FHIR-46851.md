## Resolution Summary

**Resolution:** Persuasive.

The resolution replaces the existing `_sort` bullet for `date`-typed search
parameters on `search.html` with a generalised, range-aware version that:

1. Broadens the scope from "date Period" to **any range-valued search
   parameter** (date period, range of numeric values, etc.).
2. Keeps the rule that selecting a sort value from a range is
   **server-defined**, but explicitly notes that custom search parameters
   (e.g. one targeting `Period.end`) can be defined when prescriptive
   behaviour is needed.
3. Changes "most reference implementations search by the `start` of the
   period" to "most reference implementations search by the **lower bound**
   of the period/range", and adds a recommendation: use the
   **lowest/earliest** value an element touches when sorting **ascending**
   and the **highest/latest** value when sorting **descending**.

Exact replacement text from the Jira resolution description:

> When sorting on search parameters that contain ranges of values (e.g. a
> date period, range of numeric values, etc.) the sort behavior is
> server-defined with respect to selecting a value. If more prescribed
> search behavior is needed, custom search parameters that assert specific
> behavior can be defined (e.g., a parameter targeting the `Period.end`).
> While most reference implementations search by the lower bound of the
> period/range, it is recommended to use the lowest/earliest value an
> element touches when sorting in ascending order and the highest/latest
> value an element touches when sorting in descending order.

## Feature Proposal

### Problem Statement

The `_sort` documentation on `search.html` only addresses the case where a
`date` search parameter is sourced from something other than an
instant/dateTime/date — and even then only mentions `Period`, saying the
result is server-defined and that "most reference implementations search
by the `start` of the period."

This leaves two gaps:

- It is silent on the **mixed-precision case** raised in the original
  ticket: when an `Observation.date` (or similar) is searched across
  values that are a mix of `date`, `dateTime`, and `instant`, every value
  is conceptually a period (`2010-08` ≡ `2010-08-01T00:00:00` →
  `2010-08-31T23:59:59`), but the spec gives no guidance on how to order
  them.
- It is silent on **other range-valued search parameters** (e.g. `Range`
  on a numeric quantity), which have exactly the same selection problem
  as `Period`.

The Zulip thread
[implementers > FHIR-46851 : Sort date recommendation](https://chat.fhir.org/#narrow/channel/179166-implementers/topic/FHIR-46851.20.3A.20Sort.20date.20recommendation)
confirmed there is no consensus to make a single rule mandatory (search
is normative), but there is consensus to (a) generalise the wording
beyond `date`/`Period` and (b) add a non-mandatory recommendation that
ascending sorts use the lowest/earliest value an element touches and
descending sorts use the highest/latest.

### Proposed Change

Edit the existing `<li>` bullet inside the `_sort` notes list on
`source/search.html` (currently lines 5067–5070, the bullet that begins
"When sorting on search parameters of `date` type, ...") and replace its
prose with the resolution text:

- Change the bullet's lead-in from "When sorting on search parameters of
  `date` type, if the source data is not a single instance (e.g., a
  `Period`)" to "When sorting on search parameters that contain ranges of
  values (e.g. a date period, range of numeric values, etc.)".
- Append "with respect to selecting a value" after "the sort behavior is
  server-defined".
- Update the "custom search parameters" sentence to give the example
  "(e.g., a parameter targeting the `Period.end`)".
- Replace the trailing sentence ("Note that most reference
  implementations search by the `start` of the period.") with a
  combined sentence that (i) keeps the descriptive "lower bound of the
  period/range" observation and (ii) adds the **recommendation** to use
  the lowest/earliest value an element touches when sorting ascending
  and the highest/latest when sorting descending.

Concrete proposed HTML for the bullet (preserving the existing `<li>`
structure and the existing `<a href="datatypes.html#Period">Period</a>`
link where helpful):

```html
<li>When sorting on search parameters that contain ranges of values
    (e.g., a <a href="datatypes.html#Period">date period</a>, range of
    numeric values, etc.) the sort behavior is server-defined with
    respect to selecting a value. If more prescribed search behavior is
    needed, custom search parameters that assert specific behavior can
    be defined (e.g., a parameter targeting the <code>Period.end</code>).
    While most reference implementations search by the lower bound of
    the period/range, it is recommended to use the lowest/earliest value
    an element touches when sorting in ascending order and the
    highest/latest value an element touches when sorting in descending
    order.</li>
```

(Wording is fixed by the resolution; only HTML formatting — `<code>` for
`Period.end`, the `Period` link — is at editorial discretion to match
neighbouring bullets in the same list.)

No other elements, search parameters, search-parameter definitions,
StructureDefinitions, ValueSets, or examples need to change. This is a
**prose-only** change to a single specification page.

### Design Rationale

- The resolution generalises the rule from `date`/`Period` to **any
  range-valued search parameter**, which is correct: `Range` (numeric)
  and `Period` (temporal) both expose the same "which point do you sort
  on?" problem, and `Quantity`-typed search parameters can also surface
  ranges. Generalising avoids having to amend the same sentence again
  when the next range-valued search type comes up.
- "Server-defined with respect to selecting a value" is more precise than
  "server-defined" alone — it scopes the latitude to the *choice of
  point within the range*, not to the entire sort algorithm.
- The recommendation (lowest-touched for ascending, highest-touched for
  descending) implements the intuition raised by Bas van den Heuvel in
  the Jira comments and the Zulip thread: any single date/datetime is
  itself a virtual period, so sorting "by the lowest value an element
  touches" places `2010-08` ahead of `2010-08-15T12:00:00` in an
  ascending sort, and behind it in a descending sort, in a way that is
  consistent across mixed precisions.
- Keeping it as a recommendation (not a `SHALL`/`fhir-conformance`
  span) preserves backward compatibility with implementations that
  already sort by `Period.start` for both directions, which the WG
  considered important because search is normative.
- The "custom search parameter targeting `Period.end`" example is kept
  verbatim from the resolution because it is the canonical workaround
  when an implementer truly needs a hard-coded sort point.
