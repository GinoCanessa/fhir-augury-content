## Resolution Summary

The ticket is **Persuasive** â€” the request as filed is accepted. The
FhirAugury-cached snapshot does not carry an explicit
`resolution_description`; the operative wording therefore comes from the
ticket description itself, which the workgroup accepted:

> "Receiving updates about Tasks (either there's a new one or someone's
> made an update to an existing one) is a foundational part of all of
> the Task-based workflows. We should include some examples of what
> that process looks like when polling and when using subscription. On
> the subscription side, we should show two variations â€” one with the
> full workflow including discovering subscription topics and creating
> the subscription [...], as well as a 'minimal' workflow where the
> initial steps of discovery and subscription creation happen
> out-of-band. These examples should live in the Examples tab alongside
> the other examples we plan to create."

Concretely, this means three new worked examples must be added to the
**Task** resource's *Examples* tab:

1. A **polling** example showing how a client discovers Task changes by
   periodic search.
2. A **full subscription** example showing the entire flow: discovering
   the Subscription topic, creating the Subscription, and receiving
   notifications about Tasks.
3. A **minimal subscription** example where topic discovery and
   Subscription creation are assumed to have happened out-of-band, and
   only the notification-handling part is illustrated.

## Feature Proposal

### Problem Statement

Task-based workflows in FHIR Core depend on consumers being able to
*react* to Task creation and updates, but the current Task page does
not contain any worked examples that demonstrate **how** a client
actually discovers those changes. Implementers see the resource
definition and the state machine, but no end-to-end walk-through of:

- a polling client that periodically searches for Tasks it cares about,
- a subscription-based client that performs the full discovery /
  creation / receive cycle, or
- a subscription-based client that only handles notifications because
  topic discovery and Subscription creation are arranged out-of-band
  (the common production pattern).

The result is that every IG and implementer has to reinvent these
patterns. This ticket fills that gap by anchoring three canonical
examples on the Task resource page itself, alongside the existing
Task examples (`task-example1`..`task-example6`,
`task-example-fm-*`, `task-cpg-example-1`, etc.) registered in
`source/task/list-Task-examples.xml`.

### Proposed Change

Add three new worked examples to the Task resource's **Examples** tab.
Each example is a normal FHIR Bundle (preferred â€” it lets the example
include the requests/responses that make the workflow concrete) plus a
short narrative. Concretely:

1. **`task-example-poll`** â€” *Polling for Task changes*
   - **File:** `source/task/task-example-poll.xml`
   - **Resource type:** `Bundle` of type `transaction` (or `batch`)
     containing the sequence of `GET /Task?...&_lastUpdated=gt{...}`
     requests a polling client would issue, with example responses as
     `entry.resource` Bundles.
   - **Narrative content:** describe how the client establishes a
     baseline `_lastUpdated`, how it computes the next-poll cursor,
     and how it deals with new vs. updated Tasks (using
     `Task.lastModified` / `Bundle.entry.search.mode`).
   - **Cross-link:** in the Task `Notes` page (`task-notes.xml`) or the
     introduction (`task-introduction.xml`), add a sentence under the
     state-machine discussion that points readers at the new example.

2. **`task-example-subscription-full`** â€” *Full subscription workflow*
   - **File:** `source/task/task-example-subscription-full.xml`
   - **Resource type:** `Bundle` containing, in order:
     a. a `GET` for the SubscriptionTopic catalog (e.g.,
        `GET /SubscriptionTopic?resource=Task`) and its example
        response,
     b. a `POST /Subscription` instance whose
        `Subscription.topic` references the discovered
        `SubscriptionTopic`, `channelType = rest-hook`, and a
        suitable `filterBy` on Task (e.g., owner / status), and
     c. one or more `Bundle` notifications of type
        `subscription-notification` carrying Task `event-notification`
        events.
   - **Narrative:** call out the discovery â†’ create â†’ notify flow and
     reference the Subscription page's existing channel diagrams in
     `source/subscription/workflows.md`
     (rest-hook / websocket / messaging / email).

3. **`task-example-subscription-minimal`** â€” *Minimal subscription
   workflow (discovery + creation out-of-band)*
   - **File:** `source/task/task-example-subscription-minimal.xml`
   - **Resource type:** `Bundle` containing only the
     `subscription-notification` Bundles a client would receive,
     plus a one-paragraph narrative noting that the
     `SubscriptionTopic` discovery and the `POST /Subscription` were
     handled out-of-band.
   - This is the **production** pattern most implementers actually
     ship.

For each new example, the **registration** changes are identical:

- Add an `<entry>` for the new example to
  `source/task/list-Task-examples.xml`, with the same `description`,
  `title`, and `<item><reference value="Bundle/{id}"/></item>`
  pattern used by the existing entries.
- The HL7 publisher will then surface them in the Task page's
  *Examples* tab automatically (no `source/*.html` edits are
  required for that tab â€” the tab is materialised from the list).

No changes to `structuredefinition-Task.xml`, the Task ValueSets /
CodeSystems, or `fhir.ini` are required: this is a documentation-only
addition.

### Design Rationale

- The Examples tab on every resource page is exactly the surface the
  ticket asks for ("These examples should live in the Examples tab").
- Putting all three examples on Task itself (rather than on
  Subscription) keeps them discoverable from the workflow context an
  implementer is already in. Subscription's own workflow diagrams in
  `source/subscription/workflows.md` cover *channel mechanics*; these
  new examples cover *Task-specific* polling/subscription patterns,
  which is complementary, not duplicative.
- Splitting subscription into "full" and "minimal" mirrors the exact
  request in the ticket ("two variations") and reflects the dominant
  real-world deployment style: most production clients are handed a
  pre-provisioned Subscription and only have to deal with notifications.
- Using `Bundle` as the example container lets each example carry the
  *interaction sequence* (requests, responses, notification bundles)
  rather than a single isolated resource snapshot â€” which is what
  makes a "polling vs. subscription" demonstration meaningful.
- Registering examples via `list-Task-examples.xml` (rather than ad-hoc
  page HTML) is the established FhirCore recipe and what every other
  Task example in the clone uses.

### Step-by-Step Tasks

#### HL7/fhir

1. **Author the polling example Bundle.**
   - File: `source/task/task-example-poll.xml` *(new)*
   - Action: Create a new XML file containing a `<Bundle>` whose
     entries narrate a polling client's interaction with the Task
     endpoint: an initial `GET /Task?owner=...&_lastUpdated=ge{T0}`
     followed by example response Bundles (`type = searchset`)
     containing one or two `Task` resources reused from the existing
     Task example set, then a follow-up
     `GET /Task?owner=...&_lastUpdated=gt{T1}` showing a delta poll.
   - Details: Use `<id value="example-poll"/>` (or similar),
     `<status value="generated"/>`, and a narrative `<text>` block

     internally consistent so readers can follow the cursor.

   - File: `source/task/task-example-subscription-full.xml` *(new)*

     `endpoint`, and a suitable `filterBy`,
