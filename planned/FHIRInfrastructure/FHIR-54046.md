## Resolution Summary

The resolution (Persuasive) directs the following changes:

1. Update the cardinality of `SubscriptionTopic.trigger.event` from `0..1` to `0..*`.
2. Add documentation explaining that **multiple `CodeableConcept` values** in `trigger.event` are intended for **consolidating distinct events that share the same trigger semantics** (e.g., several `patient-status-changed` events that all share the same `canFilterBy` / `notificationShape`).
3. Clarify that when multiple codes **represent the same event** (e.g., an HL7 v2 ADT event code and a local hospital code that mean the same thing), those codes belong in a **single `CodeableConcept` value as multiple `coding` entries**, not as separate `event` repetitions.
4. Explicitly note that `SubscriptionStatus` already supports the multi-event pattern, so **no changes are required there**.

Exact wording from the resolution:

> Will update `SubscriptionTopic.trigger.event` to be `0..*`. Will add documentation explaining that multiple `CodeableConcept` values should only be used when representing different events (e.g., consolidating the trigger for multiple patient-status-changed events) and clarify that using different codes representing the same event should still be placed in the same `CodeableConcept` value as multiple `coding` values (e.g., using an ADT event code and a local hospital code that represents the same event). Note that this is already available in the `SubscriptionStatus` resource, so no changes are required there.

## Feature Proposal

### Problem Statement

`SubscriptionTopic.trigger.event` is currently constrained to `0..1` (single `CodeableConcept`). When a topic author needs to express that a trigger fires on **several distinct external events that share identical filter, shape, and resource semantics** — the canonical motivating example is "a long group of ADT events, but not all of them, with no convenient parent code to use" — the only option today is to repeat the entire `SubscriptionTopic.trigger` backbone for each event. That forces duplication of `resource`, `supportedInteraction`, `queryCriteria`, `fhirPathCriteria`, `canFilterBy`, and `notificationShape` for every event in the set, even though those values are identical across the repetitions.

The `SubscriptionStatus` resource already represents notification events as a repeating element, so this constraint is also inconsistent within the Subscriptions framework.

### Proposed Change

**Element cardinality change** (StructureDefinition):

- Element: `SubscriptionTopic.trigger.event`
- Type: `CodeableConcept` (unchanged)
- Cardinality: `0..1` → **`0..*`**
- Binding: `http://terminology.hl7.org/ValueSet/v2-0003`, strength `example` (unchanged)
- `isSummary`: `true` (unchanged)
- `short` and `definition`: refresh to convey that the element now accepts multiple distinct events sharing a single trigger envelope. Suggested wording:
  - `short`: `Events which can trigger a notification from the SubscriptionTopic`
  - `definition`: `One or more well-defined events which can be used to trigger notifications from the SubscriptionTopic. Multiple repetitions SHALL only be used to consolidate distinct events that share identical trigger semantics (e.g., the same resource, supportedInteraction, queryCriteria, fhirPathCriteria, canFilterBy, and notificationShape). When multiple codes represent the same logical event (for example, an HL7 v2 ADT code and a local site code that name the same occurrence), those codes SHALL be placed as multiple coding values within a single CodeableConcept, not as separate repetitions of trigger.event.`

**Narrative documentation update** (`source/subscriptiontopic/subscriptiontopic-notes.xml`, "Defining Subscription Topics" → event-trigger paragraphs around line 595–625):

Add a paragraph (or extend the existing event-trigger paragraph) with the substance of the resolution:

- When `trigger.event` is repeated, every repetition is treated as an alternate event that fires the same trigger (identical `resource`, `supportedInteraction`, `queryCriteria`/`fhirPathCriteria`, `canFilterBy`, and `notificationShape`). Authors **SHOULD** use repetitions only to consolidate triggers that would otherwise be duplicate `trigger` backbones — e.g., several `patient-status-changed` events under HL7 v2 ADT where there is no single parent event code that captures the desired subset.
- When two or more codes describe the **same** event (e.g., a v2 `ADT^A01` code and a hospital-internal code for the same admission event), encode them as multiple `coding` entries in a **single** `CodeableConcept`, not as separate repetitions of `trigger.event`. This preserves the standard "translation" semantics of `CodeableConcept` and keeps the trigger count meaningful.
- Cross-reference: `SubscriptionStatus.notificationEvent` already supports multiple events per notification, so this change brings `SubscriptionTopic` into alignment with what implementations are already producing on the status side.

**No change** to `SubscriptionStatus` — call this out explicitly in the change set / commit message so reviewers do not look for a paired edit.

**Example instance illustrating multi-event consolidation** (recommended but optional). The existing admission examples each carry a single `<event>` with one coding. Adding one new example (or an extra `<event>` to one of the existing admission examples) would demonstrate the new pattern. A minimal snippet:

```xml
<trigger>
  <description value="Encounter resource entering 'in-progress' for any HL7 v2 admit-class event"/>
  <resource value="http://hl7.org/fhir/StructureDefinition/Encounter"/>
  <supportedInteraction value="create"/>
  <supportedInteraction value="update"/>
  <event>
    <coding>
      <system value="http://terminology.hl7.org/CodeSystem/v2-0003"/>
      <code value="A01"/>
      <display value="ADT/ACK - Admit/visit notification"/>
    </coding>
    <!-- Same event represented in a local code system: keep in one CodeableConcept. -->
    <coding>
      <system value="http://example.org/local-hospital-events"/>
      <code value="ADMIT"/>
    </coding>
  </event>
  <event>
    <coding>
      <system value="http://terminology.hl7.org/CodeSystem/v2-0003"/>
      <code value="A04"/>
      <display value="ADT/ACK - Register a patient"/>
    </coding>
  </event>
  <event>
    <coding>
      <system value="http://terminology.hl7.org/CodeSystem/v2-0003"/>
      <code value="A05"/>
      <display value="ADT/ACK - Pre-admit a patient"/>
    </coding>
  </event>
  ...
</trigger>
```

### Design Rationale

- **Mirrors `SubscriptionStatus`.** The status side already represents `notificationEvent` as repeating; making the topic side repeating closes a gratuitous asymmetry in the framework.
- **Avoids duplicating backbone state.** Repeating the entire `SubscriptionTopic.trigger` for events that share `resource`, `supportedInteraction`, `queryCriteria`, `fhirPathCriteria`, `canFilterBy`, and `notificationShape` produces large, fragile definitions that are easy to update inconsistently. A repeating `event` is the smallest precise representation.
- **Preserves CodeableConcept semantics.** The resolution is careful to draw the line between "different events" (separate repetitions) and "different codings of the same event" (multiple `coding` inside one `CodeableConcept`). That keeps `CodeableConcept`'s "translation" meaning intact and prevents implementations from double-counting one event as several.
- **Backward compatibility.** Loosening `0..1` → `0..*` is non-breaking for instances that already use a single `event`; existing instances and consumers continue to work without modification.
- **Alternative considered.** The reporter explicitly noted that the multi-event situation can be expressed today via repeated `trigger` backbones, but rejected that as too duplicative; the work group agreed (Persuasive resolution).

### Open Questions

- **Use of `SHALL` vs `SHOULD` in the new definition text.** The resolution uses the word "should". The draft definition above uses `SHALL` for the two normative restrictions (consolidation requires identical trigger semantics; same-event-different-codings goes in one `CodeableConcept`). Confirm with the work group whether they want `SHALL` (normative) or `SHOULD` (recommended) — this affects conformance.
- **New example vs in-place extension.** Decide whether to add a dedicated `subscriptiontopic-example-admission-multi-event.xml` or extend an existing example. A new example is clearer for ballot readers; extending an existing one keeps the example count steady.
- **Sweep scope.** The plan keeps the singular/plural language sweep inside the SubscriptionTopic surface only. If the work group wants it widened to `source/subscriptions.html` or any topic-based-subscription IG cross-link in `source/`, raise that explicitly so the change does not silently grow.
