## Resolution Summary

Resolution is **Persuasive** — four small textual corrections on the ActorDefinition resource page (as seen in the 6.0.0-ballot1 snapshot, sections 5.11.1 and 5.11.2):

1. Section 5.11.1 — "The ActorDefinition Resource an actor that can plays the role..." → "The ActorDefinition Resource **is** an actor that can **play** the role..."
2. Section 5.11.1 — "...business process languages will refer the actors..." → "...business process languages will refer **to** the actors..."
3. Section 5.11.2 — "...Actors may be asociated with..." → "...Actors may be **associated** with..."
4. Section 5.11.2 — link text "ExampleXcenario" → "ExampleScenario" (the `href` to `examplescenario.html` is already correct; only the anchor text is wrong).

These are pure prose typo fixes; no schema, binding, cardinality, terminology, or behavioural impact.

## Feature Proposal

### Problem Statement

The ActorDefinition specification page in the FHIR 6.0.0-ballot1 publication contains four copy-editing defects (a missing verb, a missing preposition, a misspelling, and a mangled link label). They do not change the meaning of the specification, but they are visible to every reader of the resource page and degrade the apparent quality of the ballot content. The FHIR Infrastructure work group has accepted the correction as Persuasive.

### Proposed Change

Apply the four edits listed in the Resolution Summary to the authored source of the ActorDefinition page. The FHIR Core publisher builds `actordefinition.html` from per-resource authoring fragments under `source/actordefinition/` (see Repo Context below). The text in sections 5.11.1 ("Scope and Usage") and 5.11.2 ("Boundaries and Relationships") is authored in `source/actordefinition/actordefinition-introduction.xml`. The surrounding narrative that could also host these sentences (notes / examples) lives in `actordefinition-notes.xml`.

Concretely the agreed fixes, stated as exact before→after strings, are:

| # | Before | After |
|---|--------|-------|
| 1 | `The ActorDefinition Resource an actor that can plays the role` | `The ActorDefinition Resource is an actor that can play the role` |
| 2 | `business process languages will refer the actors` | `business process languages will refer to the actors` |
| 3 | `Actors may be asociated with` | `Actors may be associated with` |
| 4 | `<a href="examplescenario.html" …>ExampleXcenario</a>` | `<a href="examplescenario.html" …>ExampleScenario</a>` |

No element, cardinality, binding, invariant, search parameter, or example instance changes are required. No terminology (UTG) change is required. No extension change is required.

**Current-clone observation:** In the current `HL7/fhir` clone at `db79dcd`, `source/actordefinition/actordefinition-introduction.xml` already reads "represents an actor that can play the role of a participant" and uses "ExampleScenarios" as the link label. The other two typos ("refer the actors", "asociated") do not appear in the current authoring fragments either. The most likely situations are:

- The typos existed in an earlier draft that was later reorganised (the 6.0.0-ballot1 HTML text exposed by the ticket does not match current `source/actordefinition/*.xml` phrasing), or
- The fixes have already been applied to master since ballot1 was cut.

Step 4a below directs the implementer to re-verify each exact string at implementation time; any string that is already corrected can be ticked off without a code change, and the ticket can be closed with "verified already corrected in master" if all four are already clean.

### Design Rationale

Typo corrections in FHIR Core specification prose are done by editing the authored HTML/XML fragments under `source/<resource>/` directly — this is the "Update narrative / prose" recipe from the FhirCore playbook. Per the briefing, `source/*.html` and per-resource `source/<resource>/*.xml` fragments are authored content, not generated, so the edits go in-place. No alternative mechanism (extension, profile, code system) is appropriate because the change is purely editorial.
