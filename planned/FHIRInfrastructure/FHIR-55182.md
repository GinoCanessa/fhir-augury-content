## Resolution Summary

Resolution: **Not Persuasive with Modification**.

The committee declined to add an `unknown` / `other` escape-valve code to the
`questionnaire-enable-operator` value set. Instead, they agreed that the
existing `enableWhenExpression` extension is the standard mechanism for
expressing conditional rules that cannot be modelled with the structured
`enableWhen` element, and that this should be made explicit in the
`Questionnaire.item.enableWhen` comment.

Exact resolution text (to be added to the `enableWhen` comments):

> "If there is a need to express conditional rules more complex than can be
> expressed with the enableWhen structure, systems should use the
> `enableWhenExpression` extension.  For example, logic that involves a
> mixture of `and` and `or` logic, checking for membership in a value set
> or within a range, etc."

The only required specification change is appending this sentence to the
existing `<comment>` value on the `Questionnaire.item.enableWhen` element in
the Questionnaire StructureDefinition. No value-set, code-system, or
cardinality changes are adopted.

## Feature Proposal

### Problem Statement

`Questionnaire.item.enableWhen.operator` is 1..1 with a required binding to
`http://hl7.org/fhir/ValueSet/questionnaire-enable-operator`, whose codes
are limited to concrete comparison operators (`exists`, `=`, `!=`, `>`, `<`,
`>=`, `<=`). There is no `unknown` or `other` code, so an author who needs
conditional logic that does not fit one of those operators — e.g., a mix of
AND and OR logic, value-set membership, or range checks — cannot legally
populate `operator` and therefore cannot use the structured `enableWhen`
element at all. The reporter asked for an escape-valve code so such items
can at least be constructed.

The committee decided the escape valve is not needed, because FHIR already
provides the `enableWhenExpression` extension (SDC-derived, now distributed
via `HL7/fhir-extensions`) for expressing arbitrarily complex conditional
logic via a FHIRPath expression. The gap is purely that the Questionnaire
resource's own narrative does not tell authors to reach for that extension
when the structured `enableWhen` is insufficient.

### Proposed Change

Append a new sentence to the existing `<comment>` value of the
`Questionnaire.item.enableWhen` element (element id
`Questionnaire.item.enableWhen`) in
`source/questionnaire/structuredefinition-Questionnaire.xml`.

- **Element:** `Questionnaire.item.enableWhen` (not `.operator` — the
  guidance is about the whole conditional, so it belongs on the backbone
  element whose comment already discusses enableWhen semantics).
- **Change type:** modification of existing `<comment value="..."/>`
  attribute; no cardinality, type, binding, constraint, or modifier-flag
  changes.
- **Text to append** (verbatim from the resolution, with the broken
  `[enableWhenExpression]()` markdown link in the resolution replaced by a
  prose reference to the extension — the publisher renders StructureDefinition
  comments as HTML, not markdown, and the link needs to be authored as an
  `<a>` to the extension's canonical URL):

  > If there is a need to express conditional rules more complex than can be
  > expressed with the enableWhen structure, systems should use the
  > [enableWhenExpression](http://hl7.org/fhir/StructureDefinition/questionnaire-enableWhenExpression)
  > extension.  For example, logic that involves a mixture of `and` and `or`
  > logic, checking for membership in a value set or within a range, etc.

  In the XML `<comment value="..."/>` attribute, quote characters must be
  escaped (`&quot;`) and angle brackets (`&lt;`, `&gt;`), and the sentence
  is concatenated onto the end of the existing comment (after the existing
  "... setting constraints around expected precision, etc." sentence), using
  two spaces as separator to match the spacing already used between
  sentences in that comment.

No changes are made to:

- `source/questionnaire/valueset-questionnaire-enable-operator.xml`
- `source/questionnaire/codesystem-questionnaire-enable-operator.xml`
- Any element other than `Questionnaire.item.enableWhen`.
- The `<snapshot>` block — the file contains only `<differential>` (lines
  84–1503 of `structuredefinition-Questionnaire.xml`); the publisher will
  regenerate the snapshot automatically.
- Any invariant or constraint; the existing `que-7` and `que-18`
  constraints on `enableWhen` are untouched.

### Design Rationale

- The resolution ("Not Persuasive with Modification") explicitly rejects
  adding a new code to the required value set. Adding an escape-valve code
  would either (a) be a breaking change to every existing implementation
  that switches on the enum, or (b) make the structured semantics
  ambiguous.
- `enableWhenExpression` already exists and is the standard HL7-blessed
  escape valve for expressions that cannot be modelled structurally. The
  extension's canonical URL is
  `http://hl7.org/fhir/StructureDefinition/questionnaire-enableWhenExpression`
  and it lives in the `HL7/fhir-extensions` repo (per this repo's
  "extensions removed from core" policy noted in the briefing).
- The change is documentation-only and purely additive — it does not alter
  conformance behaviour, so no implementer who is conformant today is made
  non-conformant by the edit.
- Placing the sentence on `Questionnaire.item.enableWhen` (the backbone),
  rather than on `.operator`, matches the scope of the guidance: the whole
  enableWhen structure (question + operator + answer) is what is
  insufficient for complex logic, not the operator alone. The existing
  comment on that element already talks about multi-repetition and
  enableBehavior semantics, so this is the natural home.

  resolution text contains a broken markdown link (`[enableWhenExpression]()`).
  The author must choose between (a) an embedded HTML anchor
  (`<a href="...">enableWhenExpression</a>`) or (b) a bare canonical URL
  in parentheses, whichever matches current FhirCore authoring practice
  in neighbouring comments. Verify by grepping
  `source/questionnaire/structuredefinition-Questionnaire.xml` and
  adjacent SDs for existing extension links in `<comment>` values before
  picking a form.
- **Canonical URL stability.** Confirm that
  `http://hl7.org/fhir/StructureDefinition/questionnaire-enableWhenExpression`
  is still the current canonical URL for this extension in
  `HL7/fhir-extensions`; if the extension has been renamed or republished
  under a different canonical, substitute that URL.
