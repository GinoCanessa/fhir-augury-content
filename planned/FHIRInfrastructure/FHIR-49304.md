## Resolution Summary

The resolution is **Persuasive with Modification**. Two concrete changes were
approved:

1. **Add `HealthcareService` as a permitted performer in the Event pattern.**
   The Request pattern was already updated to allow `HealthcareService` as a
   performer, but the Event pattern lags behind, which prevents
   `HealthcareService` from appearing as the actor of an Event-pattern
   resource.
2. **Add a new "Actors in Workflow" section to the Workflow page** with the
   exact bullet-list text supplied in the resolution, codifying which
   resources may participate in workflow activities and in what capacity
   (requester, performer, consumable/focus, with/without responsibility).

PA reviewed the proposed disposition at the January 2025 WGM (Monday Q2) and
agreed with FHIR-I's assertions about which actor types are allowed to do
what (comments by `mckenzie` 2025-01-20, `egelkraut` 2025-01-27).

The verbatim text approved for the new "Actors in Workflow" section is:

> FHIR defines a range of resources that can participate in activity
> resources — as requesters, performers, etc. There are guidelines about
> what types of resources can participate in which ways:
>
> - Person cannot ever be a participant (always use Patient, Practitioner,
>   PractitionerRole, or RelatedPerson)
> - Group can be a participant but typically cannot take responsibility. One
>   exception is if the Group represents a family or household — families and
>   households are permitted to be performers and requesters. Groups of
>   practitioners **cannot** take responsibility — use CareTeam,
>   HealthcareService or Organization instead.
> - Location cannot be a requester or performer (use CareTeam,
>   HealthcareService, or Organization instead)
> - Endpoint cannot be a requester or performer (use Device, HealthcareService,
>   or Organization)
> - CareTeam, Device, DeviceDefinition, HealthcareService, Patient,
>   Practitioner, PractitionerRole, RelatedPerson, and Organization can all be
>   actors and take responsibility for actions (though there may be business
>   rules about what types of actors are allowed to take on certain roles in
>   certain contexts).
> - BiologicallyDerivedProduct, InventoryItem, Medication, NutritionProduct,
>   Specimen, and Substance can participate as consumables or focuses but
>   cannot have responsibility.

## Feature Proposal

### Problem Statement

The Workflow module's two activity-pattern logical models — Request and Event
— diverge in what they allow as a performer/actor. The Request pattern
already includes `HealthcareService` in the type set of `Request.performer`
(see `source/request/request-spreadsheet.xml`, the Request.performer row,
reference type set
`Reference(Practitioner|PractitionerRole|Organization|CareTeam|HealthcareService|Patient|Device|RelatedPerson)`),
but the Event pattern's `Event.performer.actor` row in
`source/event/event-spreadsheet.xml` is only
`Reference(Practitioner|PractitionerRole|Organization|CareTeam|Patient|Device|RelatedPerson)`
— missing `HealthcareService`. The Zulip thread on
`#179256-Orders-and-Observation-WG > HealthcareService as a participant`
flagged that this asymmetry blocks Event-pattern resources from naming a
`HealthcareService` as the performer, which is a real-world need (e.g., a
laboratory service performing an `Observation`/`DiagnosticReport`).

The reporter additionally observed that the Workflow specification is silent
on which FHIR resources may legitimately act as agents (requesters /
performers / responsible parties) in workflow activities. The current
`source/workflow.html` page documents the Request, Event, and Definition
patterns and how they relate, but does not enumerate which resource types
may participate, in what capacity, and with what responsibility constraints.

### Proposed Change

**Change 1 — Event pattern: extend `Event.performer.actor` reference set.**

- File: `source/event/event-spreadsheet.xml`.
- Row to edit: the `Event.performer.actor` row (currently around line 1357 –
  1366 in the spreadsheet XML).
- Cell to edit: the type cell currently containing
  `Reference(Practitioner|PractitionerRole|Organization|CareTeam|Patient|Device|RelatedPerson)`.
- New value: insert `HealthcareService` into the reference type list to
  match the Request pattern's ordering. The agreed type set is
  `Reference(Practitioner|PractitionerRole|Organization|CareTeam|HealthcareService|Patient|Device|RelatedPerson)`.
- Cardinality (`1..1`), `who.actor` mapping, and the surrounding
  `Event.performer` / `Event.performer.function` rows are not changed.
- Consequence: `Event.performer.actor` becomes symmetric with
  `Request.performer` in terms of permitted actor resource types, and any
  Event-pattern-conformant resource (`Observation`, `Procedure`,
  `Encounter`, `MedicationAdministration`, `DiagnosticReport`, etc.) that
  re-states `performer.actor` will now legitimately allow
  `Reference(HealthcareService)` as a value when the resource maintainers
  choose to expose it.

### Design Rationale

- **Symmetry with Request pattern.** The Request pattern already permits
  `HealthcareService` as a performer (verified in
  `source/request/request-spreadsheet.xml`). Aligning the Event pattern
  removes an unintended asymmetry rather than introducing a new design
  decision.
- **Single source of truth for actor rules.** Documenting the permitted
  actor resources on `workflow.html` (the canonical Workflow module page)
  rather than scattering the guidance across individual resource pages keeps
  the policy maintainable and discoverable. The bullet text was reviewed
  by Patient Administration WG, the workgroup most affected by the
  Patient/Practitioner/RelatedPerson/Group/Location/Organization/HealthcareService
  set, and accepted as written.
- **Anchor stability.** Using `<a name="actors"></a>` mirrors the existing
  anchor convention on `workflow.html` (`request`, `event`, `definition`,
  `extensions`) so other pages can deep-link to
  `workflow.html#actors`.
- **No change to Request pattern.** The resolution explicitly scopes the
  pattern change to Event; Request is already correct and is referenced
  here only as the authoritative model.
