## Feature Proposal

### Problem Statement

In the FHIR Core specification page that documents the primitive types
(`source/datatypes.html`), the row describing the `instant` primitive
type contains a sentence that is meant to point readers at the
`timezone` extension so they can record the actual zone (rather than
just an offset). The reference, however, is malformed:

```
Actual timezone codes can be sent using the
[[[http://hl7.org/fhir/StructureDefinition/timezone extension]]],
if desired.
```

The HL7 FHIR publisher recognises `[[[ <canonical-url> ]]]` as a macro
that expands to a hyperlink to the named StructureDefinition (this
pattern is used dozens of times across `source/`, e.g.
`source/datatypes.html` line 1795:
`extension: [[[http://hl7.org/fhir/StructureDefinition/questionnaireresponse-signature]]].`).
Because the bracketed token here contains an embedded space and the
trailing word `extension`, the publisher cannot resolve it as a
canonical URL, so the literal `[[[…]]]` markup is rendered verbatim in
the published `datatypes.html` page. The reader sees raw publisher
syntax instead of a working link to the timezone extension.

The dateTime row immediately above (line 177) handles the same concept
correctly using the `<%extension … %>` form, demonstrating that this is
purely a typo confined to the `instant` row.

### Proposed Change

Edit `source/datatypes.html` so that the `instant` row's "Value Domain"
cell uses a well-formed publisher macro for the timezone extension.

- **File:** `source/datatypes.html`
- **Location:** line 218, inside `<tr>` … `<td>instant…</td>` block
  (anchor `<a name="instant"></a>` at line 216).
- **Before (current text on line 218):**

  ```
      Actual timezone codes can be sent using the [[[http://hl7.org/fhir/StructureDefinition/timezone extension]]], if desired.  Sub-millisecond precision is optionally allowed.
  ```

- **After:**

  ```
      Actual timezone codes can be sent using the [[[http://hl7.org/fhir/StructureDefinition/timezone]]] extension, if desired.  Sub-millisecond precision is optionally allowed.
  ```

The substantive change is moving the word `extension` outside the
triple-bracket macro and removing the stray space inside the URL token,
so the publisher emits an `<a>` tag pointing to the
`extension-timezone.html` page generated for
`http://hl7.org/fhir/StructureDefinition/timezone`. The wording
("Actual timezone codes can be sent using the … extension, if desired")
is preserved verbatim so that translators and downstream readers see no
semantic change.

No other file changes are required for the resolution as written. The
identical sentence on the dateTime row (line 177) already uses the
`<%extension http://hl7.org/fhir/StructureDefinition/timezone%>` form
and renders correctly, so it should be left alone unless the work group
chooses to harmonise both rows on a single macro style — which would be
out of scope for this Technical Correction ticket.

### Design Rationale

- The triple-bracket `[[[ canonical-url ]]]` form is the established
  convention in this clone for inline links to extensions (see the long
  list of occurrences across `source/**/structuredefinition-*.xml` and
  the second occurrence in `source/datatypes.html` line 1795). Adopting
  the same form here keeps the `instant` row consistent with the rest of
  the corpus, requires the smallest possible diff, and avoids changing
  the publisher rendering pipeline.
- Moving `extension` outside the brackets (rather than removing it)
  preserves the original prose. A reader of the rendered page sees
  "using the *timezone* extension, if desired" — exactly the intent of
  the original sentence.
- An alternative — switching this row to the `<%extension URL%>` macro
  to match the dateTime row above — was considered. It produces the
  same rendered output but introduces a second style into a row whose
  bug has nothing to do with macro choice; it would also drift from the
  surrounding `[[[…]]]` convention in the rest of `datatypes.html`. The
  resolution explicitly characterises the issue as "a new typo", so the
  minimal, in-style repair is preferred.
- The resolution description notes that the same defect exists in R4
  and R4B but not in R5/CI build. Because this skill operates against
  the master clone (`HL7/fhir` @ `db79dcd`), the fix is described
  against `master`. The work-group lead should additionally backport
  the same one-line edit to the R4 and R4B maintenance branches called
  out in the resolution.
