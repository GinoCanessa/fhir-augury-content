## Feature Proposal

### Problem Statement

Modern medication and lab workflows commonly need to convey a **multi-line
prescription / requisition** — a single business "order" composed of multiple
medication or service line items. National implementations (ISO 17523, IHE
CMPD, IHE MPD survey: 7 of 11 countries) confirmed the requirement, but FHIR
does not consistently signal *how* to bind line items together.

Implementers have diverged across at least four approaches:

- `Request.groupIdentifier` (with the binding kept outside FHIR);
- `RequestOrchestration` (single-status group);
- `Bundle` (no semantic glue);
- `CarePlan` (heavyweight wrapper).

In addition, the existing Request-pattern guidance contradicted itself by
forbidding `groupIdentifier` on the "option" Requests referenced by a
`RequestOrchestration` — which prevents a real-world case where a
`RequestOrchestration` is itself part of a larger requisition that *does*
have a shared `groupIdentifier`. Searching for a "prescription" was also
under-specified: clients had to guess between `Request.identifier`,
`Request.groupIdentifier`, `RequestOrchestration.identifier`, and so on.

### Proposed Change

Three concrete changes to `HL7/fhir`:

**(A) New shared search parameter `group-or-identifier`.**

Add a `token` search parameter named `group-or-identifier` to every
Request-pattern resource that already carries a `groupIdentifier` element.
The parameter's expression unions `<Resource>.identifier` and
`<Resource>.groupIdentifier`, allowing a single client query to retrieve a
"prescription" / "requisition" regardless of whether it was modelled as a
single Request, a set of Requests sharing a `groupIdentifier`, or an
explicit `RequestOrchestration`.

Reference template — already present in MedicationRequest and ServiceRequest
(`source/medicationrequest/bundle-MedicationRequest-search-params.xml` and
`source/servicerequest/bundle-ServiceRequest-search-params.xml`):

```xml
<entry>
  <resource>
    <SearchParameter>
      <id value="<Resource>-group-or-identifier"/>
      <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
        <valueCode value="normative"/>     <!-- match the resource's own normative/trial-use status -->
      </extension>
      <extension url="http://hl7.org/fhir/build/StructureDefinition/path">
        <valueString value="<Resource>.groupIdentifier | <Resource>.identifier"/>
      </extension>
      <url value="http://hl7.org/fhir/build/SearchParameter/<Resource>-group-or-identifier"/>
      <description value="Group ID or other identifier"/>
      <code value="group-or-identifier"/>
      <type value="token"/>
      <expression value="<Resource>.groupIdentifier | <Resource>.identifier"/>
      <processingMode value="normal"/>
    </SearchParameter>
  </resource>
</entry>
```

In-scope target resources (verified in the clone — they currently expose
`group-identifier` and a `groupIdentifier` element but **lack**
`group-or-identifier`):

- `CommunicationRequest`
- `DeviceRequest`
- `NutritionOrder`
- `RequestOrchestration`
- `Task`

`MedicationRequest` and `ServiceRequest` are already done; do not duplicate.

**(B) Remove the prohibition on option-level `groupIdentifier`.**

In `source/request/request-spreadsheet-notes.xml`, inside the
`RequestOrchestration` subsection of *Compound Requests* (around lines
146–150), remove the second sentence of the trailing paragraph so that the
remaining text reads:

> "A RequestOrchestration might have a .groupIdentifier if it was created as
> part of a larger collection of independent orders."

The sentence to delete is exactly:

> "The different 'option' requests referenced by the RequestOrchestration
> SHOULD NOT have a .groupIdentifier specified as they are all considered
> part of the RequestOrchestration order."

**(C) Add the new Compound Requests guidance paragraph.**

In the same file (`source/request/request-spreadsheet-notes.xml`), add a new
paragraph at the bottom of the *Compound Requests* section — before the
following `<a name="requisitionid"></a>` / `<h3>Shared requisition id</h3>`
subsection (i.e., insert immediately after the existing intro `<ul>` of
the three compound-request approaches, around line 84) — containing the
verbatim guidance from resolution item 3 (see *Resolution Summary* above).

### Design Rationale

- The composite-style search parameter follows existing FHIR practice (cf.
  the `MedicationRequest`/`ServiceRequest` precedents already merged) and
  preserves single-resource queryability rather than introducing a new
  "prescription" resource. This directly answers the ticket's "Searching
  for a 'prescription' is ambiguous" objection without forcing clients to
  guess which element to query.
- Removing the `SHOULD NOT` sentence aligns the spec with the immediately
  preceding sentence (which acknowledges that an Orchestration can itself
  carry a `groupIdentifier` for a larger requisition). Without the change,
  the Compound Requests narrative is internally inconsistent.
- The new guidance paragraph captures the consensus reached in the
  resolution discussion: there is intentionally no first-class
  "prescription" wrapper resource; the Request pattern + groupIdentifier +
  optional RequestOrchestration is the canonical mechanism. Brian Postlethwaite's
  comment on the ticket ("'grouping' and RequestOrchestration have two
  different purposes. Both are needed.") is the rationale for keeping both
  mechanisms documented and queryable through a single search.
- `Task.focus 0..1 → 0..*` and per-Request resource cross-links (resolution
  items 4 and 5) were intentionally split out as separate change requests
  so that each Request-resource workgroup can sign off on its own page,
  and so that Task's cardinality change goes through the normal CR review
  for that resource. This plan respects that split.
