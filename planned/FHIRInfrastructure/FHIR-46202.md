## Resolution Summary

**Resolution:** Persuasive (no free-text `resolution_description` and no comments
were returned by the CLI snapshot for this ticket; the `cross-referenced` query

The reporter pointed out that the regex shown on
[`datatypes.html#code`](https://www.hl7.org/fhir/datatypes.html#code) for the
`code` primitive type does not match the canonical form found in
`fhir.schema.json`. The correct regex (as quoted verbatim in the ticket
description) is:

```
^[^\s]+(\s[^\s]+)*$
```

i.e. anchored start/end and using `\s` (whitespace metacharacter) rather than
a literal space character between the captured tokens.

A `Persuasive` resolution on a `Technical Correction` ticket means the work
group accepted the change as described in the ticket; the implementation is to
make the documented regex agree with the canonical schema regex.

## Feature Proposal

### Problem Statement

`fhir.schema.json` (the JSON Schema published as part of the spec) defines the
pattern for the `code` primitive type as:

```
^[^\s]+(\s[^\s]+)*$
```

The narrative documentation on `datatypes.html` for the `code` primitive
currently shows a different (and weaker) regex:

```
[^\s]+( [^\s]+)*
```

The two differ in two material ways:

1. **Anchoring.** The documented form lacks `^` and `$`, so it accepts any
   string that *contains* a matching substring (e.g. trailing whitespace, a
   tab, or a newline before/after the value). The schema form is anchored and
   therefore rejects such values, matching the prose definition that `code`
   has "no leading or trailing whitespace".
2. **Internal whitespace.** The documented form uses a literal space `' '`
   between non-whitespace runs. The schema form uses `\s`, which would also
   admit tabs / newlines / other whitespace characters between tokens. The
   prose definition of `code`, however, says "*no whitespace other than
   single spaces in the contents*" — which is what the literal-space form
   captures, and what the `\s` form does **not**.

So the two forms in the published spec disagree with each other, and each one
disagrees with the prose in a different way. The ticket calls this out as a
documentation defect and proposes adopting the schema form on the documentation
page.

### Proposed Change

Update the canonical source-of-truth for the `code` primitive's regex so that
the documentation page renders the same regex that ships in `fhir.schema.json`.

In the HL7/fhir clone the regex template `[%regex code%]` on
`source/datatypes.html` (line 155) is resolved by the publisher from the
`Patterns` worksheet of `source/datatypes/primitives.xml`. The `code` row in
that worksheet (line ~265, the cell named `code`) currently contains:

```
[^\s]+( [^\s]+)*
```

Replace that cell's value with the form quoted in the ticket:

```
^[^\s]+(\s[^\s]+)*$
```

No other primitive types referenced from the `code` row need to change: the
edit is one cell in the `Patterns` worksheet of one spreadsheet file. The
publisher will pick this up everywhere `[%regex code%]` is expanded
(`source/datatypes.html`, plus any other narrative page that templates the
same token).

### Design Rationale

- The reporter explicitly cites `fhir.schema.json` as the authoritative source
  and asks the documentation to be brought into line with it. Because
  `fhir.schema.json` is generated from the same spec build, the simplest way to
  honour "Persuasive" is to use its exact pattern verbatim on the doc page.
- The `^` / `$` anchors are necessary for the regex to actually express the
  prose constraint ("no leading or trailing whitespace"). Without them the
  pattern is satisfied by any string that merely contains a single
  non-whitespace token, which is clearly not the intent.
- The `\s` vs literal-space discrepancy described above is a known wrinkle
  worth raising in **Open Questions** below: the prose says "single spaces",
  but the canonical schema regex uses `\s`. The Persuasive resolution names
  the schema form, so this plan applies it; if the work group ultimately wants
  the regex to match the prose exactly, the right value would be
  `^[^\s]+( [^\s]+)*$` (anchored, literal space). That second alternative is
  not what the ticket asked for and is not what is being applied here.

   - File: `source/datatypes/primitives.xml`

     `code` (the row carrying `<NamedCell ss:Name="code"/>`, currently around
     line 259-266). Replace the value of the `RegEx` cell from

     format (`<Cell><Data ss:Type="String">...</Data></Cell>`). Edit the
     literal text inside the `<Data>` element only; do not alter the

     row.

2. **Verify the rendered documentation picks up the new value**
   - File: `source/datatypes.html` (line 155)
   - Action: No source edit. After step 1, run the publisher (or inspect the

   - File: (generated; not in `source/`)
   - Action: After publishing, diff the regenerated `fhir.schema.json`

   - Details: This is a sanity check, not an edit. It ensures the publisher
     hasn't been emitting two different patterns from two different sources;

### Open Questions

- **`\s` vs literal space in the canonical regex.** The prose for `code`
  says "*no whitespace other than **single spaces** in the contents*". The
  schema-form regex (`^[^\s]+(\s[^\s]+)*$`) actually allows tabs, newlines,
  etc. between tokens because `\s` is broader than `' '`. The Persuasive
  resolution explicitly names the schema form, so this plan applies it
  verbatim. If the work group later prefers the regex to match the prose
  exactly, the value should instead be `^[^\s]+( [^\s]+)*$` (anchored,
  literal space) — and `fhir.schema.json` would also need updating to
  agree. Worth flagging on the PR for FMG / FHIR-I awareness, but **not** a
  blocker for this ticket.
- **Confirm the `[%regex code%]` macro targets `Patterns!code` specifically.**
  The spreadsheet defines a `code` `NamedCell` only in the `Patterns`
  worksheet (the `Imports` worksheet has no row for `code`). The plan
  assumes the publisher resolves `[%regex code%]` against that named cell.
  If there is a publisher convention to prefer `Imports` over `Patterns`,
  the edit may need to be made (or duplicated) in a different worksheet —
  this should be verified during PR review by anyone familiar with the
  publisher's spreadsheet-loading code.
- **Briefing scope.** The cached HL7/fhir briefing (`meta.json`
  `ticket_key: FHIR-42980`) was generated for an unrelated FML-page ticket.
  Its general FhirCore facts (build system, authoring root, generated areas,

  pre-computed for the `code` primitive. The path used in this plan

  verified directly against the clone (`db79dcd`).
