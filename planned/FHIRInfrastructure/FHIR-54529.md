## Resolution Summary

Resolution: **Persuasive with Modification**.

The committee will replace the sentence in the documents specification that
describes how a Bundle stylesheet `<link>`'s `url` may be expressed. The
exact rewording dictated by the resolution is:

Change

> The url can be an absolute reference to a CSS stylesheet or a relative
> reference to a Binary resource that carries a CSS stylesheet. Stylesheet
> references can only refer to a CSS stylesheet — other forms of stylesheet
> are not acceptable.

to

> The url can be an absolute reference to a CSS stylesheet or a reference
> following the [Bundle resolution rules](bundle.html#references) to a
> Binary resource that carries a CSS stylesheet. Stylesheet references can
> only refer to a CSS stylesheet — other forms of stylesheet are not
> acceptable.

The change is purely prose: instead of describing the Bundle-internal
reference as "a relative reference," the spec now points readers at the
formal Bundle reference-resolution rules at `bundle.html#references` for
how the `url` can resolve to a Binary entry inside the same Bundle.

## Feature Proposal

### Problem Statement

The Documents specification (`source/documents.html`, "Stylesheets" /
`#css` section, lines ~195–200) currently states that a Bundle stylesheet
link's `url` "can be an absolute reference to a CSS stylesheet or a
relative reference to a Binary resource that carries a CSS stylesheet."
The reporter (Bas van den Heuvel) noted that "relative reference" is
unclear in this context — implementers cannot tell from this sentence
alone how the url is meant to resolve to a `Binary` entry that lives
elsewhere in the same Bundle (i.e., that it is matched against the
entry's `fullUrl` per the Bundle reference-resolution algorithm). The
ticket asks for an explicit clarification rather than leaving readers to
guess.

### Proposed Change

Edit the prose paragraph in `source/documents.html` (the paragraph
introduced at `<a name="css"></a>` describing the Bundle `<link
relation="stylesheet">` `url`) so that the second clause links to the
formal Bundle reference rules instead of saying "a relative reference":

- Replace the existing sentence (HTML lines 195–200):

  ```html
  <p>
  The <code>url</code> can be an absolute reference to a CSS stylesheet or
  a relative reference to a Binary resource that carries a
  CSS stylesheet. Stylesheet references can only refer to
  a CSS stylesheet - other forms of stylesheet are not acceptable.
  </p>
  ```

- With:

  ```html
  <p>
  The <code>url</code> can be an absolute reference to a CSS stylesheet or
  a reference following the <a href="bundle.html#references">Bundle resolution rules</a>
  to a Binary resource that carries a CSS stylesheet. Stylesheet references
  can only refer to a CSS stylesheet - other forms of stylesheet are not acceptable.
  </p>
  ```

No other paragraphs, examples, or surrounding markup change. The
`<a name="css"></a>` anchor and the surrounding stylesheet discussion
(security note link, "relative (internal) references SHOULD be used,"
etc.) are preserved verbatim.

### Design Rationale

- **Faithful to the resolution.** The resolution explicitly dictates the
  before/after wording; the only judgement call is rendering the Markdown
  link `[Bundle resolution rules](bundle.html#references)` in the
  surrounding HTML page as `<a href="bundle.html#references">Bundle
  resolution rules</a>`. `documents.html` is authored as HTML (see Repo
  Context below), so an inline `<a>` is the correct rendering.
- **Anchor is real.** `bundle.html#references` resolves to the
  `Resolving references in Bundles` section in
  `source/bundle/bundle-notes.xml` (`<a name="references"></a>`,
  authored at line 209), so the new link will not dangle.
- **Reuses an existing concept.** Bundles already define a single
  authoritative reference-resolution algorithm. Pointing at it removes
  the need for the Documents page to re-explain Bundle-internal
  resolution semantics, and avoids inconsistent restatements drifting
  out of sync.
- **No structural change considered.** The reporter suggested adding a
  sentence about the `fullUrl` mechanic; the committee chose the lighter
  link-to-the-rules approach instead, which the resolution makes
  explicit. No alternative wording was selected.
