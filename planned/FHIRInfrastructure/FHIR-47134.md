## Resolution Summary

Resolution: **Not Persuasive with Modification**.

The committee declined to enumerate, per functional list, an exhaustive list
of acceptable status values that a server should include. Instead, the
agreed modification is to add narrative guidance making clear that
inclusion in a `$current-*` functional list is **not** a pure function of
a resource's `status` (or `clinicalStatus`) value, and that business rules
legitimately influence whether an item is still "current".

Verbatim text to add (per the ticket's Resolution Description), positioned
**immediately before** the sentence `"FHIR defines the following names for
functional lists"` in `source/lifecycle.html`:

> The choice of whether a resource is included in a 'current' list is not
> solely driven by status. The period when a record is considered still
> 'current' may vary based on business rules. For example, a
> MedicationRequest or MedicationStatement that was only recently changed
> to 'completed' might still remain as a 'current' medication. Medications
> with a longer half-life or where they are community-based rather than
> in-patient medications might remain on the 'current' list for longer.
>
> Different systems may have differing rules for when a particular
> instance should appear in the $current list. (This is one of the
> reasons for having an operation rather than merely querying based on
> status.)

No element/cardinality/binding changes are required; this is a pure
narrative edit to a single specification page.

## Feature Proposal

### Problem Statement

FHIR Core defines four functional list names — `$current-problem`,
`$current-medications`, `$current-allergies`, `$current-drug-allergies`
(see `lifecycle.html` "Current Resource Lists") — but does not say what
`status` (or `clinicalStatus`) values a server is expected to use when
deciding which resource instances belong on each list. The reporter
(Brett Marquard) raised concern that absent guidance, implementers will
construct materially different "current" lists from the same underlying
data (e.g., is a `Condition` with every `clinicalStatus` value
appropriate for `$current-problem`? does
`MedicationStatement.status = recorded` belong in `$current-medications`?).
At minimum, the requestor asked for cross-links from the functional-list
definitions to the relevant status elements so implementers have a
starting point.

The committee considered enumerating acceptable status values per
functional list and found that approach unworkable: the membership of a
"current" list legitimately depends on local business rules that go
beyond the resource's status (e.g., post-discharge medication windows,
half-life considerations, community vs. in-patient context). The
committee therefore opted for an explanatory paragraph rather than a
normative status enumeration.

### Proposed Change

A single, content-only edit to `source/lifecycle.html` in `HL7/fhir`:
insert two new `<p>` elements inside the `Current Resource Lists`
section (anchor `current`), positioned **immediately before** the
paragraph beginning `"FHIR defines the following names for functional
lists:"`.

The two paragraphs to insert (with the exact wording from the
resolution):

```html
<p>The choice of whether a resource is included in a 'current' list is
   not solely driven by status. The period when a record is considered
   still 'current' may vary based on business rules. For example, a
   MedicationRequest or MedicationStatement that was only recently
   changed to 'completed' might still remain as a 'current' medication.
   Medications with a longer half-life or where they are community-based
   rather than in-patient medications might remain on the 'current' list
   for longer.</p>
<p>Different systems may have differing rules for when a particular
   instance should appear in the $current list. (This is one of the
   reasons for having an operation rather than merely querying based on
   status.)</p>
```

No StructureDefinition, ValueSet, CodeSystem, search parameter, example,
or operation definition is modified.

### Design Rationale

- **Narrative over enumeration.** The committee's modification (vs. the
  reporter's literal ask) is principled: status alone is insufficient to
  determine list membership. Encoding a fixed status-to-list mapping
  would over-constrain implementations and embed clinical/business
  policy into the core specification.
- **Co-located with the functional-list definitions.** Placing the
  guidance immediately before the "FHIR defines the following names for
  functional lists:" paragraph ensures readers encounter the caveat
  before the list of names and the implicit invitation to map them to
  status values.
- **Operation, not query.** The second paragraph reinforces an existing
  design choice already made elsewhere (`_list` search parameter,
  `lifecycle.html#current` referenced from `search.html` line 6081):
  functional lists are exposed as named lists (effectively an operation
  surface) precisely so server-side rules can shape the result, rather
  than clients reconstructing them from status filters.
- **No element-level change is needed** because the originally requested
  cross-links (e.g., to `Condition.clinicalStatus`,
  `MedicationStatement.status`) would risk implying the opposite of the
  agreed message — that status is the determinant.
