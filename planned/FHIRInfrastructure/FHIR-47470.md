## Resolution Summary

Resolution type: **Persuasive with Modification**.

Resolution text (verbatim):

> Will create a 'starter' additional binding as proposed with one additional code: **stylesheet** which is used in documents.

Concretely, this means adding a second (`additional`) binding on `Bundle.link.relation` with `purpose = starter` whose ValueSet enumerates the small set of IANA link-relation codes that FHIR Bundles use in practice: **first, last, self, prev, next**, plus the modification code **stylesheet** (used in document Bundles). The existing `required` binding to `http://hl7.org/fhir/ValueSet/iana-link-relations` is unchanged.

## Feature Proposal

### Problem Statement

`Bundle.link.relation` is bound (strength: `required`) to `http://hl7.org/fhir/ValueSet/iana-link-relations`, which enumerates roughly 120 IANA-registered link-relation codes (see `source/bundle/codesystem-iana-link-relations.xml`; the registered codes include `first`, `last`, `next`, `prev`, `self`, `stylesheet`, and many others). In practice, Bundle producers and consumers use only a handful of these — the navigation links `first`, `last`, `self`, `prev`, `next` — plus `stylesheet` for document Bundles. Implementers browsing the full 120-entry ValueSet have no signal that only a small starter set is in common use, which is a poor authoring experience and encourages divergence.

The ticket reporter (Bas van den Heuvel) proposed adding a preferred/starter ValueSet via `binding.additional` to highlight the common cases without removing the ability to use any IANA code.

### Proposed Change

1. **New ValueSet** in the `bundle/` module listing the starter set of link-relation codes:

   - Id: `bundle-link-relations-common` (sibling to `valueset-iana-link-relations.xml`).
   - URL: `http://hl7.org/fhir/ValueSet/bundle-link-relations-common`.
   - Name: `BundleLinkRelationsCommon` / Title: `Common Bundle Link Relation Types`.
   - Status: `active`, experimental `false`, version `6.0.0`.
   - Description: "The subset of IANA link-relation types that are commonly used in FHIR `Bundle.link` / `Bundle.entry.link`: the navigation links `first`, `last`, `self`, `prev`, and `next`, plus `stylesheet` (used in document Bundles)."
   - Compose: a single `include` from `http://hl7.org/fhir/CodeSystem/iana-link-relations` enumerating the six codes.
   - Work-group extension (`structuredefinition-wg` = `fhir`) and standards-status extension consistent with `valueset-iana-link-relations.xml`.

   Example skeleton:

   ```xml
   <ValueSet xmlns="http://hl7.org/fhir">
     <id value="bundle-link-relations-common"/>
     <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-wg">
       <valueCode value="fhir"/>
     </extension>
     <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
       <valueCode value="normative"/>
     </extension>
     <url value="http://hl7.org/fhir/ValueSet/bundle-link-relations-common"/>
     <version value="6.0.0"/>
     <name value="BundleLinkRelationsCommon"/>
     <title value="Common Bundle Link Relation Types"/>
     <status value="active"/>
     <experimental value="false"/>
     <description value="The IANA link-relation types commonly used in FHIR Bundle.link / Bundle.entry.link: first, last, self, prev, next, and stylesheet (used in document Bundles)."/>
     <compose>
       <include>
         <system value="http://hl7.org/fhir/CodeSystem/iana-link-relations"/>
         <concept><code value="first"/></concept>
         <concept><code value="last"/></concept>
         <concept><code value="self"/></concept>
         <concept><code value="prev"/></concept>
         <concept><code value="next"/></concept>
         <concept><code value="stylesheet"/></concept>
       </include>
     </compose>
   </ValueSet>
   ```

2. **Add an `<additional>` binding** to `Bundle.link.relation` in `source/bundle/structuredefinition-Bundle.xml` (element id `Bundle.link.relation`, currently at lines 330–345). Preserve the existing `required` binding; append a sibling `additional` block inside `<binding>`:

   ```xml
   <binding>
     <strength value="required"/>
     <valueSet value="http://hl7.org/fhir/ValueSet/iana-link-relations"/>
     <additional>
       <key value="b-bdl-1"/>
       <purpose value="starter"/>
       <valueSet value="http://hl7.org/fhir/ValueSet/bundle-link-relations-common"/>
       <documentation value="Starter set of IANA link-relation codes used by nearly all FHIR Bundles: the navigation links first/last/self/prev/next and stylesheet (for document Bundles)."/>
     </additional>
   </binding>
   ```

   The shape of the `<additional>` block mirrors the existing Patient.communication.language additional binding in `source/patient/structuredefinition-Patient.xml` lines 851–855, which is the published precedent cited in the ticket comment ("We're doing this for Patient.language, so assume it's ok").

3. **No narrative / prose page changes are required.** The `source/bundle/bundle-introduction.xml` and `source/bundle/bundle-notes.xml` pages do not currently describe link-relation usage at that level of detail; no update is needed unless a WG reviewer requests one.

### Design Rationale

- **Consistency with Patient.language** — the ticket resolution explicitly points at `Patient.communication.language` as precedent. That element keeps a `required` binding to `all-languages` and layers a `starter`-purpose additional binding to `languages`. Applying the same pattern on `Bundle.link.relation` keeps FHIR's conformance semantics unchanged (any IANA code remains valid) while giving implementers a curated starter list.
- **`purpose = starter`** is the correct additional-binding purpose per the `http://hl7.org/fhir/ValueSet/additional-binding-purpose` ValueSet: "useful as a starter set" rather than `preferred` (which has different conformance implications) or `required`.
- **Why these six codes** — `first`, `last`, `self`, `prev`, `next` are the navigation links called out by the reporter; `stylesheet` was added by the WG on resolution because it is the one additional code used in document Bundles. All six codes already exist in `source/bundle/codesystem-iana-link-relations.xml` (verified at lines 178, 293, 358, 443, 508, 548).
- **New ValueSet rather than inline `concept` list in the binding** — FHIR additional-bindings reference a `valueSet` canonical. A dedicated ValueSet is discoverable from the terminology pages and usable by tooling (IG generators, validators, UI pickers) independently of `Bundle`.

### Open Questions

- **Naming of the starter ValueSet** (`bundle-link-relations-common` vs. `bundle-link-relations-starter` vs. `common-bundle-link-relations`). The resolution text does not fix the id; the name chosen here matches the "common values" phrasing of the ticket title and mirrors the `bundle-type`/`search-entry-mode`/`http-verb` sibling naming. A WG reviewer should confirm before merge.
- **Additional-binding `key` value** (`b-bdl-1` proposed, by analogy to Patient's `b-pat-1`). If FHIR Infrastructure WG already has a different key-naming scheme for Bundle, use that instead.
- **Scope of `stylesheet` usage** — the resolution mentions it for document Bundles specifically; the starter ValueSet does not carry context-of-use metadata. If the WG wants to distinguish "navigation starter" from "document starter", it would need two ValueSets and two `<additional>` entries (one with `purpose=starter` for navigation and a second — possibly with a different purpose code — for `stylesheet`). Default assumption: a single combined starter ValueSet, per the resolution wording ("one additional code: stylesheet").
