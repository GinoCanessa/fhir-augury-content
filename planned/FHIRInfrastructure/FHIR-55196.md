## Resolution Summary

Resolution: **Not Persuasive with Modification**.

The committee declined to add an `unknown` / `other` escape-valve concept to
the `Subscription.status` value set. Instead, the definition of the existing
`off` code will be broadened so that the set of five concepts
(`requested`, `active`, `error`, `off`, `entered-in-error`) covers every
state a Subscription can legitimately be in. Per the resolution text, the
definition of `off` changes from:

> "Too many errors have occurred or the subscription has expired."

to:

> "The subscription is not presently active.  E.g. Too many errors have occurred or the subscription has expired."

With that broadening, any future state would be a specialisation of one of
the existing codes, so no escape-valve concept is required.

## Feature Proposal

### Problem Statement

`Subscription.status` (R5 Subscription resource) is a 1..1 element with a
required binding to the `SubscriptionStatus` CodeSystem/ValueSet. The
reporter pointed out that the value set has no `unknown` / `other` /
catch-all concept. If a server’s Subscription is in a state that does not
cleanly fit `requested`, `active`, `error`, `off`, or `entered-in-error`,
there is no legal code and the mandatory element cannot be populated –
making the resource impossible to instantiate in those edge cases.

### Proposed Change

A narrow, documentation-only change to the `SubscriptionStatus` CodeSystem:

- **Artifact:** CodeSystem `http://hl7.org/fhir/subscription-status`
  (the in-line CodeSystem whose codes are surfaced through the
  `http://hl7.org/fhir/ValueSet/subscription-status` value set bound to
  `Subscription.status`).
- **Concept changed:** code `off` (display `Off`).
- **Field changed:** `concept.definition`.
  - From: `Too many errors have occurred or the subscription has expired.`
  - To: `The subscription is not presently active.  E.g. Too many errors have occurred or the subscription has expired.`
- **No other concepts are added, removed, or renamed.** `requested`,
  `active`, `error`, `off`, and `entered-in-error` remain the full set.
- **No change to code, display, or hierarchy.** Bindings, cardinality,
  strength, and URL/OID identifiers are unchanged.

Example (unchanged) value-set usage:

```xml
<status value="off"/>
```

Only the human-readable definition carried in the CodeSystem and rendered
into the specification shifts meaning.

### Design Rationale

The committee considered (and declined) the reporter’s proposal to add a
new `unknown`/`other` concept. Adding such a concept to an already-published
required-bound status value set would:

- create a new legal instance state that every existing Subscription
  server/consumer would have to handle; and
- weaken the semantics of `Subscription.status` by allowing a "don't
  know" response for a required element that every server is expected to
  authoritatively own.

Instead, the resolution observes that the five existing codes are
semantically exhaustive once `off` is read as "not presently active (for
any reason)" rather than only "errored out / expired". The textual
broadening preserves backward compatibility (no code churn, same
cardinality, same required binding) while closing the gap the reporter
identified. This mirrors the FHIR pattern of preferring definition-level
clarifications over value-set growth for normative required-bound status
elements.

### Step-by-Step Tasks

#### HL7/fhir

1. **Broaden the `off` concept definition in the SubscriptionStatus CodeSystem.**
   - File: `source/subscription/codesystem-subscription-status.xml`
   - Action: In the `<concept>` block whose `<code value="off"/>`
     appears (currently line 39–42), change the `<definition>` value
     from
     `"Too many errors have occurred or the subscription has expired."`
     to
     `"The subscription is not presently active.  E.g. Too many errors have occurred or the subscription has expired."`
     (note the two-space gap between `active.` and `E.g.`, as written in
     the resolution).
   - Details: Leave `<code value="off"/>`, `<display value="Off"/>`, and
     every other concept (`requested`, `active`, `error`,
     `entered-in-error`) untouched. Do not edit
     `valueset-subscription-status.xml`; it derives from this CodeSystem.
     Do not edit `<snapshot>` blocks anywhere — they are regenerated by
     the publisher.

2. **Sanity-check for stray copies of the old definition text.**
   - File: repository-wide (`source/**`).
   - Action: Run a case-sensitive search for the literal string
     `Too many errors have occurred or the subscription has expired.`
     Expect exactly one match before the edit (the line above) and zero
     after.
   - Details: A pre-flight grep during this plan's preparation found a
     single occurrence in the clone at HEAD
     `db79dcdfe196d35bd0e74c58c655a4c1c8f534f7`, confirming the change
     is localised. If future edits introduce additional copies (e.g.
     hand-authored narrative in `subscription.html` or
     `subscription-notes.xml`), update them in the same commit.
