## Resolution Summary

The resolution (**Persuasive**) states:

> Will add to the relevant elements in `CapabilityStatement` that `*` is an
> allowed value and used to signal support for requests using an `*` in place
> of a specific value. In a `CapabilityStatement`, listing a `*` makes no
> assertions beyond that support — the actual search parameters available
> still must be listed individually.

The "relevant elements" in scope are `CapabilityStatement.rest.resource.searchInclude`
and `CapabilityStatement.rest.resource.searchRevInclude` (the latter explicitly
mentioned in the ticket body as "also applies to searchRevInclude"). The change
is prose-only on the element `comment` text in the CapabilityStatement
StructureDefinition; no cardinality, type, or binding changes are required.

## Feature Proposal

### Problem Statement

The current (pre-resolution) narrative for
`CapabilityStatement.rest.resource.searchInclude` explains how to document
supported `_include` values (e.g., `CarePlan:activity-reference:DeviceRequest`)
and how iterative includes are expressed, but it does not say whether the
wildcard `*` is a valid value. Implementers observed production R4 servers
publishing `"searchInclude": ["*"]` (and mixed lists like `["*", "ExplanationOfBenefit:..."]`),
and validators such as Touchstone were behaving inconsistently because the
specification was silent on the question (see Zulip thread
[implementers > CapabilityStatement.rest.resource.searchInclude](https://chat.fhir.org/#narrow/stream/implementers/topic/CapabilityStatement.rest.resource.searchInclude),
which motivated this ticket and the sibling FHIR-46380). The same silence
applied to `searchRevInclude`, which delegates its narrative to the
`searchInclude` comment.

### Proposed Change

Update the `<comment>` text on the `searchInclude` and `searchRevInclude`
element definitions inside
`source/capabilitystatement/structuredefinition-CapabilityStatement.xml` to
explicitly document the `*` wildcard:

- **Element `CapabilityStatement.rest.resource.searchInclude`** — insert the
  following two sentences into the existing `<comment>` (after "If this list
  is empty, the server does not support includes." and before the paragraph
  about iterative `_include`):

  > The wildcard, `*`, is an allowed value and used to signal support for
  > requests using an `\*` in place of a specific value. In a
  > `CapabilityStatement`, listing an `*` makes no assertions beyond that
  > support — the actual search parameters available still must be listed
  > individually.

- **Element `CapabilityStatement.rest.resource.searchRevInclude`** — the
  current `<comment value="See \`CapabilityStatement.rest.resource.searchInclude\` comments."/>`
  delegates entirely to the `searchInclude` narrative. Because the resolution
  wording references "the relevant elements" (plural) and implementers hit the
  ambiguity on both sides, keep the delegating "See …" pointer but verify the
  rendered `capabilitystatement-definitions.html` page (produced by the
  publisher) cross-links correctly. No separate prose needs to be authored for
  `searchRevInclude` — the cross-reference picks up the new wildcard sentence
  automatically.

No other elements need the wildcard call-out: the ticket scope is explicitly
`searchInclude` / `searchRevInclude`.

> **Status check against the clone at HEAD `db79dcd`:** the
> `searchInclude` `<comment>` in
> `source/capabilitystatement/structuredefinition-CapabilityStatement.xml`
> (lines 1150–1160) already contains the two wildcard sentences above, and
> `searchRevInclude` (lines 1161–1171) already uses the
> "See `CapabilityStatement.rest.resource.searchInclude` comments." pointer.
> The resolution therefore appears to already be applied in source. The tasks
> below are written as a verification + finishing checklist rather than a
> fresh authoring pass; if any item is already complete, mark it done rather
> than re-editing.

### Design Rationale

- The change is surgical: it closes the ambiguity called out in the ticket
  without introducing a new structural construct. Keeping the element type
  (`string`, 0..*) untouched avoids breaking existing instances.
- Authoring the wildcard narrative **only** on `searchInclude` and having
  `searchRevInclude` delegate via "See …" is the same pattern already in use
  for these two elements and keeps the prose single-sourced. The resolution
  text is preserved almost verbatim in the comment, so the intent recorded on
  the ticket is directly traceable in the spec.
- The resolution explicitly declines to make `*` carry any additional
  semantics ("makes no assertions beyond that support"). That protects server
  implementations already using the value (e.g., the CarinBB EOB server
  described in the Zulip thread) while also giving validator authors a clear
  rule that a `*` entry does not imply `Resource:*`-style combinations.
