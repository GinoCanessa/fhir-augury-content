## 1 · Summary

The ticket asks for clarification on how wildcard `_include` support should be expressed in a `CapabilityStatement`. Specifically:

1. How should support for wildcard include (`*`) be specified in `CapabilityStatement.rest.resource.searchInclude`?
2. If wildcard include support is specified, does that mean all individual includes are implicitly supported, or must they be listed separately?

The same questions apply to `searchRevInclude`.

### Resolution Description

> Will add to the relevant elements in `CapabilityStatement` that `*` is an allowed value and used to signal support for requests using an `*` in place of a specific value.
>
> In a `CapabilityStatement`, listing a `*` makes no assertions beyond that support — the actual search parameters available still must be listed individually.

---

## 2 · Context & Discussion

A [Zulip thread](https://chat.fhir.org/#narrow/stream/implementers/topic/CapabilityStatement.rest.resource.searchInclude) on the **implementers** stream captured the original discussion:

- **Michele Mottini** observed that production servers use `*` in `searchInclude` but the spec didn't explicitly address whether it was valid or what it meant.
- **Gino Canessa** confirmed the colon-separated syntax is correct (e.g., `CarePlan:activity-reference:DeviceRequest`) and noted the wildcard `*` needed explicit clarification.
- **Craig McClendon** reported a real-world issue where a HAPI-based server listed both `*` and individual includes, but Touchstone expected `ExplanationOfBenefit:*` rather than a bare `*`, highlighting the ambiguity.

---

## 3 · Feature Proposal

Add two sentences to the `comment` element of `CapabilityStatement.rest.resource.searchInclude` in the StructureDefinition, inserted after the sentence about empty lists meaning no include support:

> The wildcard, `*`, is an allowed value and used to signal support for requests using an `*` in place of a specific value. In a `CapabilityStatement`, listing an `*` makes no assertions beyond that support — the actual search parameters available still must be listed individually.

The `searchRevInclude` element already defers to `searchInclude` via: _"See `CapabilityStatement.rest.resource.searchInclude` comments."_ — so no separate change is needed there.
