## Resolution Summary

**Resolution:** Persuasive.

The committee adopted the conventions described in the work-in-progress
branch:

> Will adopt the conventions as discussed at this WGM and as documented
> here:
> https://build.fhir.org/branches/2024-01-gg-xver-extensions2/versions.html#extensions-examples-complex

In short, the cross-version extension specification needs concrete,
worked examples covering the harder cases — in particular complex
datatypes like `CodeableReference`, primitive→typed transitions, and
mandatory-element handling — together with a precise URL/pathing
convention for choice elements (the `[x]` suffix has to be URL-encoded
as `%5Bx%5D` in the extension URL `[Path]` segment, as confirmed in the
follow-up `[IG creation] missing extension` thread on Zulip).

The most recent ticket comment (ginocanessa, 2025-11-07) signals an
implementation refinement: the bulk of the worked-example content is
being **moved into the cross-version (`xver-*`) packages** rather than
hosted exclusively inline in the FHIR core spec, with the core spec
retaining the convention statement and links into those packages. The
ticket will be confirmed once that content is live across the
cross-version package set.

## Feature Proposal

### Problem Statement

`source/versions.html` in HL7/fhir defines a set of cross-version
extensions whose URLs follow the pattern
`http://hl7.org/fhir/[defining_version]/StructureDefinition/extension-[Path]`,
where `[Path]` is an `ElementDefinition.id`. Today the page:

- Names three example sub-sections (the anchors
  `extensions-examples-primitive`, `extensions-examples-complex`, and
  `extensions-examples-mandatory` exist on lines 699–701 of
  `source/versions.html`) but **carries no content under any of them** —
  they sit immediately before `<h4>Formal Definitions for extensions</h4>`
  with no prose, no examples, and no headings of their own.
- Does not explain how `[x]` choice-element suffixes are represented in
  the extension URL. The Zulip discussion `[IG creation] missing
  extension` (2024-05-09) shows real implementers landing on the
  `%5Bx%5D` URL-encoded form only by archeology through expired CI
  branches — the published spec does not state the rule.
- Does not describe how transitions of complex datatypes such as
  `CodeableReference` (R5+) are represented as cross-version extensions
  back to versions where the type does not exist (DSTU2/R3/R4/R4B), nor
  how primitives that change type, nor mandatory elements that newly
  appear are represented.

The gap is exactly what the original ticket text states: "Need to
describe better how to handle stuff like CodeableReference."

### Proposed Change

The change has two coordinated parts. The first is a documentation
update inside HL7/fhir; the second is a content-population effort in
the `hl7.fhir.uv.xver-*` cross-version packages (out of cache scope —
see Cross-Repo Touch Points).

**Part A — `source/versions.html` in HL7/fhir.** Replace the empty
anchor block at lines 699–701 with three populated sub-sections that
embody the conventions adopted at the WGM:

1. **Primitive examples** (`<a name="extensions-examples-primitive">` →
   `<h4>` heading + worked example). Show, for an element whose type
   changed between versions (e.g., a string→code or code→Coding move),
   how the cross-version extension carries the value in the source
   version's JSON form, with one full StructureDefinition snippet and
   one instance snippet.
2. **Complex datatype examples** (`<a name="extensions-examples-complex">`
   → `<h4>` heading + worked example). Use `CodeableReference` as the
   canonical case: show the cross-version extension definition that
   carries an R5 `CodeableReference` value into an R4/R4B context as
   the choice `Reference | CodeableConcept`, including the value
   structure for each branch and the URL form
   `http://hl7.org/fhir/5.0/StructureDefinition/extension-<Resource>.<element>`
   (with the `%5Bx%5D` rule applied where the element is itself a
   choice — see step 2 below).
3. **Mandatory element examples** (`<a name="extensions-examples-mandatory">`
   → `<h4>` heading + worked example). Show how an element that is
   mandatory in the newer version but absent from the older version is
   represented by the cross-version extension, and clarify the
   conformance impact on the older-version instance (the extension is
   not "mandatory" in the older version's StructureDefinition; it is a
   recommended extension that allows round-tripping).

In addition, in the prose around the URL pattern (currently lines
640–693), add a normative paragraph specifying that:

- `[Path]` is the `ElementDefinition.id` from the relevant
  `StructureDefinition` in the `[defining_version]`.
- For choice elements, the `[x]` token MUST appear in the URL
  percent-encoded as `%5Bx%5D` (e.g.,
  `http://hl7.org/fhir/4.0/StructureDefinition/extension-Observation.value%5Bx%5D`
  for R4 `Observation.value[x]`). Add at least one row to the example
  URL table at lines 689–693 demonstrating this form.
- Provide a forward link from each new sub-section into the relevant
  `hl7.fhir.uv.xver-*` package examples once they exist.

**Part B — Cross-version packages
(`hl7.fhir.uv.xver-r{N}.r{M}`, e.g., `xver-r5.r4`).** These IGs are
**not** in the local FhirAugury cache (no
`cache/github/repos/HL7_xver-*` directory) and so cannot have their
file paths verified inline. The implementation tasks against them
are:

- Add a per-package `versions.html`-equivalent (or `extensions.html`)
  page that mirrors the same three example categories
  (primitive / complex / mandatory) with package-specific `extension-*`
  StructureDefinitions called out by URL.
- Ensure every shipped `extension-*.json|xml` for an `[x]` choice
  element uses the `%5Bx%5D` URL-encoded form so that the URL on the
  wire matches the one documented in `source/versions.html`.
- Confirm a `CodeableReference` example StructureDefinition exists in
  the R5→R4/R4B and R5→R3 packages and is referenced from the new
  example sub-section in `source/versions.html`.

(Per the 2025-11-07 comment, Part B is the "content is moving" half
that will be confirmed before the ticket is closed.)

### Design Rationale

The Persuasive resolution explicitly cites the
`2024-01-gg-xver-extensions2` build branch as the design of record;
that branch already contains the populated example sub-sections and
URL-encoding rule. The implementation is therefore mechanical adoption
of an already-agreed text — there are no alternative designs to weigh.

The Zulip thread `[IG creation] missing extension` (Corey Spears,
2024-05-09) demonstrates the cost of *not* shipping these conventions:
implementers reverse-engineer the `%5Bx%5D` rule from a CI branch that
"no longer exists", which is exactly the scenario the resolution is
intended to retire.

The Zulip thread `Backporting CodeableReference` (Lloyd McKenzie,
2024-02-02) confirms that ticket [FHIR-44514](https://jira.hl7.org/browse/FHIR-44514)
chose a workaround (changing the RSG extension to a
`CodeableConcept | Reference` choice) precisely *because* this ticket
had not yet shipped. That workaround can be revisited only after
Part A is published.

The `MovingToGuide` label confirms that — over the long run — the bulk
of the worked content belongs in the dedicated cross-version
package(s) rather than as an ever-growing block in the core spec, hence
the Part B split.

#### HL7/xver-* (cross-version IGs — JiraSpecArtifacts/Ig family, not in local cache)

  scheduled. Expected touch points: each package's `versions.html` /

- **Change Scope:** Major (across ~30 packages on the R2/R3/R4/R4B/R5/R6

### Breaking Changes

- **No breaking changes for the published `source/versions.html`
  page itself** — it adds prose and example sub-sections under existing
  anchors without removing or renaming anything.
- **Potentially breaking for already-published `xver-*` extensions
  whose URLs do NOT use `%5Bx%5D`.** If any shipped `extension-*` for a
  choice element uses a different URL form (e.g., literal `[x]`,
  stripped `x`, or any other variation) then bringing those URLs into
  line with the newly-normative `%5Bx%5D` convention would change the
  canonical URL — which is by definition a breaking change for any
  consumer that referenced the old URL. The Part B implementation must
  audit existing URLs and decide per-case whether to rename, alias, or
  freeze (out of scope to decide here; flag in Open Questions).
- **No type/cardinality changes to any FHIR resource element.**
