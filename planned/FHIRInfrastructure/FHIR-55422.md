## Resolution Summary

The committee accepted the concern as **Persuasive with Modification**. Rather
than altering the normative `Attachment` datatype, the resolution is to define
a **new extension** with the following exact shape:

- **URL / id:** `alternateHash` (HL7 extension; published in the HL7
  fhir-extensions IG).
- **Cardinality on the host element:** `0..*`.
- **Context:** `Attachment` (element context).
- **Components (sub-extensions, both `1..1`):**
  - `algorithm` — `Coding`, **extensible** binding to a Terminology.HL7.Org
    (THO) value set seeded from
    `https://build.fhir.org/ig/FHIR/openehr-base-ig/ValueSet-integrity-check-algorithms.html`
    (unless a more authoritative source is identified during implementation).
  - `hash` — `string` (the digest expressed in the algorithm’s native textual
    form, mirroring how openEHR encodes its `integrity_check`).

The existing normative `Attachment.hash` (SHA-1, base64Binary) element is
**unchanged**; the resolution explicitly notes that SHA-1 hashes should
continue to use that element, and `alternateHash` is the path for any other
algorithm.

## Feature Proposal

### Problem Statement

`Attachment.hash` in core FHIR is normative and pinned to SHA-1 (encoded as
`base64Binary`). Implementations whose source systems compute a different
digest (SHA-256, SHA-512, BLAKE3, MD5, etc. — exactly the situation that
surfaced while mapping FHIR Attachment ↔ openEHR `integrity_check` /
`integrity_check_algorithm`) must either drop the digest or recompute SHA-1
purely to populate the FHIR field. Both options are lossy: the former
discards integrity information the source already had, the latter forces a
weak/deprecated algorithm onto receivers. Because `Attachment` is normative,
the element itself cannot change, but the extension mechanism can carry the
missing information without breaking conformance.

### Proposed Change

Define a new HL7 standard extension and, in support of its binding, a THO
value set (and matching code system if no authoritative external source is
adopted).

**1. Extension `alternateHash` (HL7/fhir-extensions)**

- **Canonical URL:** `http://hl7.org/fhir/StructureDefinition/alternateHash`
- **id:** `alternateHash`
- **kind:** `complex-type`, `type = Extension`,
  `baseDefinition = http://hl7.org/fhir/StructureDefinition/Extension`,
  `derivation = constraint`.
- **Status / FMM:** `draft`, `experimental = true`, `fmm = 0` (initial
  publication; uplift later if adopted).
- **Publisher / WG:** `HL7 International / FHIR Infrastructure`
  (`structuredefinition-wg = fhir`).
- **Context:**
  ```xml
  <context>
    <type value="element"/>
    <expression value="Attachment"/>
  </context>
  ```
- **Cardinality at the host:** `0..*` on `Attachment.extension:alternateHash`.
- **Differential (sliced sub-extensions, value[x] disallowed at the root):**
  - `Extension` — short: *“Hash of the data using an algorithm other than
    SHA-1”*; min `0`, max `*`.
  - `Extension.extension:algorithm` — `1..1`,
    `value[x] -> valueCoding`, binding strength `extensible`,
    `valueSet = http://hl7.org/fhir/ValueSet/integrity-check-algorithms`,
    short: *“Algorithm used to produce the hash”*. Must **not** be SHA-1
    (note in `comment`: *“If the algorithm is SHA-1, populate
    `Attachment.hash` directly instead of using this extension.”*).
  - `Extension.extension:hash` — `1..1`, `value[x] -> valueString`, short:
    *“Hash value in the algorithm’s native textual encoding”*.
  - `Extension.url` — fixed to `alternateHash`.
  - `Extension.value[x]` — `0..0` (so `value[x]` is prohibited at the root,
    forcing use of the two sub-extensions).

  Example use:
  ```xml
  <Attachment>
    <contentType value="application/pdf"/>
    <url value="https://example.org/report.pdf"/>
    <extension url="http://hl7.org/fhir/StructureDefinition/alternateHash">
      <extension url="algorithm">
        <valueCoding>
          <system value="http://terminology.hl7.org/CodeSystem/integrity-check-algorithms"/>
          <code value="SHA-256"/>
          <display value="SHA-256"/>
        </valueCoding>
      </extension>
      <extension url="hash">
        <valueString value="e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"/>
      </extension>
    </extension>
  </Attachment>
  ```

**2. ValueSet / CodeSystem `integrity-check-algorithms` (HL7/UTG → THO)**

Seed the value set from the openEHR base IG’s
`integrity-check-algorithms` value set. Concrete codes drawn from that source
(verify exact strings against the openEHR IG at implementation time):
`SHA-1`, `SHA-256`, `SHA-384`, `SHA-512`, `MD5`, plus any others enumerated
there. SHA-1 is included for completeness of the algorithm list but the
extension comment steers authors toward the core `Attachment.hash` element
when SHA-1 is used.

- **ValueSet canonical:**
  `http://hl7.org/fhir/ValueSet/integrity-check-algorithms`.
- **CodeSystem canonical (if hosted at THO):**
  `http://terminology.hl7.org/CodeSystem/integrity-check-algorithms`.
- **Binding strength used by the extension:** `extensible` (per the
  resolution), so implementers may carry algorithms outside the base list
  while still encouraging uniform vocabulary.
- **Authoritative-source check:** before publishing the new CodeSystem, the
  Vocabulary review must confirm whether IANA Hash Function Textual Names
  (https://www.iana.org/assignments/hash-function-text-names/) or NIST CSRC
  Hash Algorithms is a more authoritative external source — the resolution
  text leaves that door open.

**3. Documentation (HL7/fhir, optional but recommended)**

A short cross-reference paragraph in the Attachment narrative
(`source/datatypes.html` — Attachment section) noting that non-SHA-1 digests
should be expressed via the `alternateHash` extension. This keeps core users
from re-discovering the gap. Resolution doesn’t mandate the doc tweak; flag
as optional follow-up rather than a blocker.

### Design Rationale

- **Extension over datatype change.** `Attachment` is a normative datatype;
  altering the element list (e.g., adding an `algorithm` sibling to `hash`)
  would be a breaking change to a normative artifact. An extension is the
  designed escape valve for exactly this case.
- **Multi-component extension over a sibling extension pair.** Algorithm
  and hash are meaningless without each other, so they are bundled as
  sub-extensions inside a single `alternateHash` extension. This mirrors
  established patterns in `fhir-extensions` (e.g., `alternate-codes`,
  `iso21090-uncertainty`), which use `Extension.extension` slices and
  forbid the root `value[x]`.
- **`Coding` for the algorithm.** Allows future systems beyond the seed
  value set (extensible binding) and admits richer metadata (system,
  version, display) than a bare `code`, matching how openEHR represents
  `integrity_check_algorithm`.
- **`string` (not `base64Binary`) for the hash.** Different algorithms have
  different canonical textual encodings (hex, base64, multihash). A free
  `string` lets each algorithm carry its native form without ambiguity, and
  matches the openEHR `integrity_check` representation.
- **Cardinality `0..*`.** A producer may want to publish multiple digests
  (e.g., SHA-256 *and* BLAKE3) for the same byte stream; `0..*` allows that
  without further plumbing.
- **Keep `Attachment.hash` for SHA-1.** Avoids forking the SHA-1 path
  through two mechanisms and preserves backward compatibility with all
  existing instances.
