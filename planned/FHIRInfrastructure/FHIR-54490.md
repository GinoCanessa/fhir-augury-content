## Resolution Summary

Resolution: **Persuasive**.

The resolution description states verbatim:

> Will add "E.g., 'Patient/123' rather than 'http://someserver.org/Patient/123'." after "If the URL is a relative reference, it is interpreted in the same way as a resource reference."

Concretely, the Attachment narrative on the FHIR datatypes page must be amended to clarify, with an example, that a relative `Attachment.url` is a FHIR resource reference of the form `Patient/123` (relative to the FHIR server base URL), not an absolute URL such as `http://someserver.org/Patient/123`.

## Feature Proposal

### Problem Statement

The Attachment datatype currently says that when `Attachment.url` is a relative reference it is "interpreted in the same way as a resource reference" and links to `references.html#references`. The reporter (Bas van den Heuvel) noted this leaves the exact form of a relative reference ambiguous: is it `Patient/123`, `./Patient/123`, or `http://Patient/123`? Implementers shouldn't have to chase the cross-link to discover that the intended form is `Patient/123` evaluated against the FHIR server base URL. The ticket asks for an inline clarification.

### Proposed Change

A single, narrowly scoped narrative addition to the Attachment section of the datatypes specification page.

- **Repo:** `HL7/fhir`
- **File:** `source/datatypes.html`
- **Location:** the paragraph beginning "The actual content of an Attachment can be conveyed directly using the `data` element…" — specifically the sentence on line 593 that ends "…it is interpreted in the same way as a [resource reference](references.html#references)."
- **Edit:** append the following sentence immediately after that sentence (still inside the same `<p>` block):

  ```html
  E.g., <code>Patient/123</code> rather than <code>http://someserver.org/Patient/123</code>.
  ```

  Rationale for using `<code>` wrapping: the surrounding paragraph already wraps element names and tokens (e.g., `<code>data</code>`, `<code>URL</code>`, `cid:`) in `<code>`, so the example values should match the existing typographic convention.

- **No other artifacts change.** The Attachment StructureDefinition (`source/datatypes/attachment.xml` is the legacy SpreadsheetML — see Gotchas; the actual XML StructureDefinition for datatypes lives in the publisher pipeline) does not contain the same prose; the `Attachment.url` element definition is unchanged. No invariant, cardinality, type, or binding change is required.

After applying the change, the resulting paragraph reads:

> The actual content of an Attachment can be conveyed directly using the `data` element or a `URL` reference can be provided. If both are provided, the reference SHALL point to the same content as found in the data. The reference can never be reused to point to some different data (i.e., the reference is version specific). The `URL` reference SHALL point to a location that resolves to actual data; some URIs such as cid: meet this requirement. If the URL is a relative reference, it is interpreted in the same way as a [resource reference](references.html#references). E.g., `Patient/123` rather than `http://someserver.org/Patient/123`.

### Design Rationale

The resolution explicitly approves an inline example over a cross-reference rewrite or a behavioral change. Using the same `Patient/123` form documented under `references.html#references` keeps Attachment consistent with the broader FHIR Reference rules (relative references are interpreted against the FHIR server base, exactly as they are for `Reference(...)` elements). Showing the absolute-URL counter-example removes the specific ambiguity the reporter raised (`Patient/dlkdjak` vs `./Patient/kadsd` vs `http://Patient/akdasjlk`). No alternative is evident from the ticket — there are no comments and no Zulip/GitHub cross-references to suggest a different design was considered.
