## Resolution Summary

The committee declined to add this capability to the **core** `CapabilityStatement` resource. Instead, the work moves to the new **CapabilityStatement2 IG** (`HL7/capstmt`, the Application Feature Framework IG), which takes a terminology / FeatureDefinition-driven approach to expressing server features.

The originally approved disposition was to add a code `unqualified-search` (boolean true/false) to the IG's feature value set, with context `CapabilityStatement.rest` and `CapabilityStatement.rest.resource`.

A subsequent **2025-05-13 comment from Rick Geimer (assignee) refines the disposition**:

> "Instead of adding a code to the feature value set, we should create an example instance of FeatureDefinition for this use case."

So the implementation work is to author a new **`FeatureDefinition` example instance** in the CapabilityStatement2 IG (`HL7/capstmt`) that declares server support for unqualified searches at the `CapabilityStatement.rest` / `CapabilityStatement.rest.resource` level. No change is required in `HL7/fhir`.

The ticket itself should also be reassigned in Jira from "FHIR Core (FHIR)" to the CapabilityStatement2 IG specification.

## Feature Proposal

### Problem Statement

Today, a FHIR server's CapabilityStatement cannot explicitly declare whether the server supports unqualified searches (i.e. `GET [base]/Resource` with no search parameters). This matters for Patient Access APIs, where servers commonly rely on the implicit identity carried by the access token to scope an unqualified `GET /Patient` to the authorized patient — while other servers (e.g., Epic's Observation endpoint) require at least one search parameter. Today the only way to discover this is to brute-force the request. The reporter (Brendan Keeler) raised this on the [#implementers Zulip thread](https://chat.fhir.org/#narrow/stream/179166-implementers/topic/CapabilityStatement.20and.20unqualified.20searches) and filed FHIR-41457.

### Proposed Change

Add a new example `FeatureDefinition` instance to the CapabilityStatement2 IG (`HL7/capstmt`) declaring an "unqualified search" feature with these characteristics (subject to confirmation against the IG's current authoring patterns once the briefing is available — see Open Questions):

- **Resource:** `FeatureDefinition`
- **Suggested instance id / name:** `UnqualifiedSearch`
- **Canonical URL:** `http://hl7.org/fhir/uv/application-feature/FeatureDefinition/UnqualifiedSearch` (form mirrors the existing `FeatureSupport` / `FavoriteColor` examples in `input/fsh/instances/FeatureDefinition/`).
- **status:** `active`
- **valueType:** `boolean` (true = server supports `GET [base]/{Resource}` with no search parameters; false = explicitly does not).
- **description / short:** "Indicates whether the server supports unqualified search (a `GET` on the resource type with no search parameters)."
- **contextDefinition:** Anchored at `CapabilityStatement.rest` and `CapabilityStatement.rest.resource` (per the resolution text). When supplied at `CapabilityStatement.rest`, the value is the server-wide default; when supplied at `CapabilityStatement.rest.resource`, it overrides the default for that specific resource type — matching the access pattern (e.g., a server may support unqualified `GET /Patient` but not unqualified `GET /Observation`).
- **contextExpressionLanguage:** `text/url` (matches `FeatureSupport.fsh`).
- **context:** entries pointing at `CapabilityStatement#CapabilityStatement.rest` and `CapabilityStatement#CapabilityStatement.rest.resource`.

A companion update to `FeatureCapabilityStatementExample.fsh` (under `input/fsh/instances/CapabilityStatement/`) should add a `feature` extension demonstrating the new `FeatureDefinition` in use — at least once at `rest` level and once at a `rest.resource` entry — to give implementers a concrete pattern.

### Design Rationale

- The committee decided the core `CapabilityStatement` should not grow an element for this concern; the Application Feature Framework (CapabilityStatement2 IG) is the designated extension surface for arbitrary server-feature declarations.
- The 2025-05-13 update prefers an **example FeatureDefinition** over a code in a fixed value set so the framework remains terminology-agnostic and the example serves as the canonical pattern for implementers.
- Anchoring context at both `CapabilityStatement.rest` and `CapabilityStatement.rest.resource` matches the resolution text and reflects real-world deployments where unqualified-search behaviour can vary per resource type.
