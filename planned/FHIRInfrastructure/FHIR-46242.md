## Resolution Summary

**Resolution:** Persuasive with Modification.

Verbatim resolution description:

> We will define two properties as part of CapabilityStatement2 that can be
> used on `interaction` (whether `rest.interaction` or
> `rest.resource.interaction`) that can indicate the profile that must hold
> for data passed "into" the interaction and for data "returned from" the
> interaction. (For some interactions, only one of those properties will be
> relevant.)

In concrete terms for the CapabilityStatement2 / Application Feature
Framework work currently hosted in **HL7/capstmt**, this means defining
two new `FeatureDefinition` resources (the framework's mechanism for
declaring conformance properties) — one for the **input** (request)
profile of a system-level or resource-level interaction, and one for the
**output** (response) profile — both scoped via `context` to
`CapabilityStatement.rest.interaction` and
`CapabilityStatement.rest.resource.interaction`. Each property's value is
a `canonical(StructureDefinition)` reference identifying the profile the
data must conform to. This allows `batch`/`transaction` (and any other
system-level interaction such as `search-system`/`history-system`, plus
resource-level `create`/`update`/etc.) to declare the accepted/returned
Bundle (or other resource) profile, satisfying the original request
without changing the base CapabilityStatement resource itself.

## Feature Proposal

### Problem Statement

The base `CapabilityStatement` resource has no machine-readable way to
declare the profile that a server expects on the request body of a
batch/transaction interaction (or returns in the response). The
batch/transaction handler is not modelled as an `OperationDefinition`,
so the standard `OperationDefinition.parameter.targetProfile` mechanism
that exists for $-operations cannot be reused. Implementations such as
IHE MHD currently abuse `interaction.documentation` (free text) for the
same purpose, which is opaque to validators and tooling. The reporter
asks for a first-class extension/property so that a server can publish
"on POST to `/` (batch/transaction), I accept Bundles conforming to
profile X". The same need applies to other interactions (e.g., the
profile of resources accepted on `create`/`update`, or the profile of
the `Bundle` returned by `search-system`/`history-system`).

### Proposed Change

Add two new `FeatureDefinition` instances to the **HL7/capstmt** IG
(CapabilityStatement2 / Application Feature Framework) so that they can
be declared on `CapabilityStatement.rest.interaction` and
`CapabilityStatement.rest.resource.interaction` via the existing
`Feature` extension (`http://hl7.org/fhir/uv/application-feature/StructureDefinition/feature`):

1. **`InteractionInputProfile`** — the profile that data **passed into**
   the interaction must conform to.
   - `url`: `http://hl7.org/fhir/uv/application-feature/FeatureDefinition/InteractionInputProfile`
   - `name`: `InteractionInputProfile`
   - `status`: `active`
   - `description`: "The profile (canonical URL of a StructureDefinition)
     that the request payload of this interaction must conform to. For
     `batch`/`transaction`, this is the Bundle profile; for `create` /
     `update` / `patch`, this is the resource profile expected on the
     request body. Repeats when more than one alternative profile is
     accepted."
   - `valueType`: `canonical` (target type `StructureDefinition`)
   - `contextDefinition`: "Applied to a `CapabilityStatement.rest.interaction`
     or `CapabilityStatement.rest.resource.interaction` element where the
     interaction has a request body."
   - `context` (using `contextExpressionLanguage = #text/url`):
     - `http://hl7.org/fhir/StructureDefinition/CapabilityStatement#CapabilityStatement.rest.interaction`
     - `http://hl7.org/fhir/StructureDefinition/CapabilityStatement#CapabilityStatement.rest.resource.interaction`

2. **`InteractionOutputProfile`** — the profile that data **returned
   from** the interaction conforms to.
   - `url`: `http://hl7.org/fhir/uv/application-feature/FeatureDefinition/InteractionOutputProfile`
   - `name`: `InteractionOutputProfile`
   - `status`: `active`
   - `description`: "The profile (canonical URL of a StructureDefinition)
     that the response payload of this interaction conforms to. For
     `batch`/`transaction` and `search-system`/`history-system`, this is
     the response Bundle profile. Repeats when alternative response
     profiles are possible."
   - `valueType`: `canonical` (target type `StructureDefinition`)
   - `contextDefinition`: same scoping as input.
   - `context`: same two URLs as input.

The resolution explicitly notes: "for some interactions, only one of
those properties will be relevant." That is naturally handled because
each `FeatureDefinition` is independent and can be declared zero or more
times per interaction.

A worked example (FSH) showing how a server would declare these on a
batch/transaction interaction, to be added to the existing
`FeatureCapabilityStatementExample.fsh`:

```fsh
* rest[+].mode = #server
* rest[=].interaction[+].code = #transaction
* rest[=].interaction[=].extension[+].url = "http://hl7.org/fhir/uv/application-feature/StructureDefinition/feature"
* rest[=].interaction[=].extension[=].extension[+].url = "definition"
* rest[=].interaction[=].extension[=].extension[=].valueCanonical = "http://hl7.org/fhir/uv/application-feature/FeatureDefinition/InteractionInputProfile"
* rest[=].interaction[=].extension[=].extension[+].url = "value"
* rest[=].interaction[=].extension[=].extension[=].valueCanonical = "http://example.org/fhir/StructureDefinition/MyTransactionBundle"
```

### Design Rationale

- The CapabilityStatement2 incubator already defines a generic feature
  declaration mechanism (`FeatureDefinition` + `Feature` extension). The
  resolution explicitly scopes the work to CapabilityStatement2 and
  speaks of "two properties", which maps directly onto two
  `FeatureDefinition` instances rather than a bespoke extension on the
  base `CapabilityStatement` resource. This avoids touching the
  normative base resource for an experimental capability and keeps the
  CS2 incubator self-contained.
- Using `valueType = canonical` (target `StructureDefinition`) — rather
  than free text or a code — gives validators a tool to enforce the
  declared profile on submitted/returned bundles.
- Scoping to both `rest.interaction` and `rest.resource.interaction`
  covers the originally requested batch/transaction case (system-level
  interaction) and incidentally enables the same declaration for
  per-resource `create`/`update`/etc., which the discussion thread
  identified as a parallel need.
- Modelling input and output as **separate** definitions (rather than a
  single property with sub-extensions) matches the resolution wording
  ("two properties") and lets each side be omitted independently for
  interactions where only one direction has a meaningful payload.
- The IHE MHD pattern of using `interaction.documentation` is rejected
  because it is unstructured prose and cannot be enforced or discovered
  by tooling.

### Testing Considerations

- Validate the example CapabilityStatement (`FeatureCapabilityStatementExample`) end-to-end: parse → validate → confirm the two `Feature` extension instances on the `transaction` interaction round-trip.
- Construct a sample request Bundle that **matches** and another that **violates** the declared `InteractionInputProfile` and confirm a profile-aware validator can use the canonical to perform validation.
- Repeat for response Bundle vs `InteractionOutputProfile`.
- Spot-check that the same features attach cleanly to a `rest.resource.interaction` (e.g., `create` on `Patient`) — this exercises the second context expression.
- Cross-check tooling that already knows about the `Feature` extension URL (`http://hl7.org/fhir/uv/application-feature/StructureDefinition/feature`) treats `valueCanonical` correctly inside the `value` sub-extension.

### Open Questions

- **Exact instance names / canonical URLs.** The resolution says "two
  properties" but does not name them. `InteractionInputProfile` /
  `InteractionOutputProfile` are proposed; the work group should
  confirm before publication so the canonical URLs are stable.
- **`valueType` choice.** `#canonical` is the most useful for tooling,
  but the existing `FeatureDefinition` examples in the IG use `#code`.
  Confirm that the framework's `valueType` accepts `canonical` end-to-end
  (publisher rendering, `$feature-query` operation behaviour, the
  `Feature.value` sub-extension's `value[x]` choice). If not, fall back
  to `#string` carrying a canonical URL and document the convention.
- **Cardinality on a single interaction.** Should multiple alternative
  input/output profiles be allowed (i.e., declare the feature multiple
  times)? Recommended **yes** to mirror `OperationDefinition` semantics,
  but worth a confirming note from the WG.
- **Scope beyond batch/transaction.** The resolution mentions
  `rest.interaction` *and* `rest.resource.interaction`; confirm that
  declaring these features on resource-level interactions
  (`create`/`update`/etc.) is in-scope for this ticket or should be
  documented as a follow-on.
