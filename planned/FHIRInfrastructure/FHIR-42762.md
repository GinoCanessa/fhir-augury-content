## Resolution Summary

**Resolution:** Persuasive with Modification.

The committee agreed to **define a `SearchParameter` for the
`workflow-episodeOfCare` extension**. The search parameter will live on /
be visible from the extension itself, **not** on the resources that use
the extension — that is the standard behaviour for search parameters
defined against extensions (the resolution cites the
`patient-mothersMaidenName` extension as a reference example:
`https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-patient-mothersMaidenName.html`).

The "modification" relative to the original ask is that the search
parameter will **not** be wired into the per-resource search-parameter
pages of every Event resource. Now that extensions are maintained in a
separate IG (`HL7/fhir-extensions`), surfacing extension-defined search
parameters on core-resource pages would be a tooling change, and the
resolver explicitly says that promoting it onto resource pages is out of
scope here (it's a tooling question for Zulip, not a content change for
this ticket).

## Feature Proposal

### Problem Statement

The core spec offers `Encounter`-based search on most Event resources
(via the `workflow-encounter` search parameter family / direct `encounter`
elements), but there is no equivalent way for clients to query Event
resources by associated `EpisodeOfCare`. The `workflow-episodeOfCare`
extension already exists and is widely applicable (its SD lists ~40 base
resource contexts in the Extensions IG — see the file referenced below),
yet there is no `SearchParameter` defined against it, so the data carried
in the extension is effectively non-searchable in a standard, portable
way.

The companion ticket FHIR-42761 (which proposed making `encounter`
elements directly reference `EpisodeOfCare` as well) is referenced as the
fall-back assumption: if that broader change is not adopted, this ticket
is the minimum needed to give implementers a search path to
EpisodeOfCare-tagged events.

### Proposed Change

Add a single new `SearchParameter` resource to the `HL7/fhir-extensions`
IG, modelled on the existing `SearchParameter-workflow-researchStudy.xml`
(same shape, same workflow- prefix, same "extension search parameter"
pattern).

Concrete shape:

- **Resource type:** `SearchParameter`
- **id / file name:** `workflow-episodeOfCare`
  (file: `SearchParameter-workflow-episodeOfCare.xml`)
- **url:** `http://hl7.org/fhir/SearchParameter/workflow-episodeOfCare`
- **identifier (OID):** allocate the next OID in the
  `urn:oid:2.16.840.1.113883.4.642.31.x` series used by the existing
  workflow- search parameters in this IG (e.g., `workflow-researchStudy`
  uses `…31.1`; pick the next free integer in that branch — verify
  against the other `SearchParameter-workflow-*.xml` files at
  authoring time).
- **version:** `5.0.0` (matches the version stamped on the sibling
  `workflow-*` extension and search parameter files in the same folder).
- **name:** `EpisodeOfCareSearchParameter`
- **title:** `Episode Of Care SearchParameter`
- **status:** `draft`
- **experimental:** `true`
- **publisher / contact:** `HL7 International / FHIR Infrastructure`,
  with `http://www.hl7.org/Special/committees/fiwg` (this is the WG that
  owns the `workflow-episodeOfCare` extension — see its
  `structuredefinition-wg` extension value `fhir` and its existing
  publisher string).
- **standards-status extension:** `draft` (as on the other `workflow-*`
  search parameters in this folder).
- **wg extension:** `fhir`.
- **description:** `SearchParameter for the workflow-episodeOfCare
  extension`.
- **code:** `workflow-episodeOfCare` (use the prefixed form, matching
  `workflow-barrierRef` etc., so it doesn't collide with any future
  resource-native `episodeOfCare` search parameter).
- **type:** `reference`.
- **base:** the full list of resources that appear as `<context>`
  entries in `StructureDefinition-workflow-episodeOfCare.xml`. From the
  current SD that is:
  AdverseEvent, Appointment, Basic, CarePlan, ChargeItem,
  ClinicalImpression, Communication, CommunicationRequest, Composition,
  Condition, Consent, DetectedIssue, DeviceRequest, DeviceUsage,
  DiagnosticReport, DocumentReference, EnrollmentRequest,
  EnrollmentResponse, FamilyMemberHistory, Flag, Goal, ImagingStudy,
  Immunization, ImmunizationEvaluation, ImmunizationRecommendation,
  Invoice, List, MeasureReport, MedicationAdministration,
  MedicationDispense, MedicationRequest, MedicationStatement,
  MolecularSequence, NutritionIntake, NutritionOrder, Observation,
  Procedure, QuestionnaireResponse, RequestGroup, RequestOrchestration,
  ResearchSubject, RiskAssessment, ServiceRequest, SupplyDelivery,
  SupplyRequest, VisionPrescription.
  (Re-derive this list from the SD at authoring time — version-specific
  contexts must still be enumerated as `base` entries since
  `SearchParameter.base` does not itself carry version-specific use.)
- **target:** `EpisodeOfCare` (the extension's `value[x]` is constrained
  to `Reference(EpisodeOfCare)` in the SD).
- **expression:** Use the same shape as
  `SearchParameter-workflow-researchStudy.xml`, i.e.
  `DomainResource.extension('http://hl7.org/fhir/StructureDefinition/workflow-episodeOfCare').value`
  This single-line `DomainResource.extension(...)` form is the simpler
  pattern in the same folder; the per-base disjunction form used by
  `workflow-barrierRef` is also acceptable but unnecessarily verbose
  here.
- **processingMode:** `normal`.

No changes to `StructureDefinition-workflow-episodeOfCare.xml` itself
are required.

Per the resolver's "Persuasive with Modification" wording, **do not**
attempt to inject the new search parameter into the per-resource
search-parameter pages of every Event resource in `HL7/fhir`. That part
of the original request is explicitly deferred as a tooling
conversation.

### Design Rationale

- The existing extension already carries enough type information
  (`Reference(EpisodeOfCare)`) for a reference-typed search parameter to
  be well-defined.
- The chosen shape mirrors `SearchParameter-workflow-researchStudy.xml`,
  which is the closest precedent in the same folder
  (`input/definitions/multiple/`): same workflow- prefix, same "draft /
  experimental, FHIR Infrastructure or BR&R-published, wide `base` list,
  reference `target`, `DomainResource.extension(...).value`
  expression". Following the precedent keeps tooling, validation, and
  publication consistent.
- Defining the SearchParameter against the extension itself (as opposed
  to publishing per-resource SearchParameters) is the canonical FHIR
  pattern and is exactly what the resolver pointed at with the
  `patient-mothersMaidenName` example.
- The `code` is intentionally `workflow-episodeOfCare` (with prefix) to
  match other workflow- extension search parameters and avoid colliding
  with any future resource-native `episodeOfCare` parameter (e.g., on
  Encounter), which would be a separate piece of work.

## Impact Analysis

### Affected Repositories

#### HL7/fhir-extensions (FhirIg / Extensions Pack)

- **Role:** Owns `workflow-episodeOfCare` and is where the new
  `SearchParameter` must be authored.

  artifacts).

### Breaking Changes

None. This is a purely additive change: a new `SearchParameter` artifact
in the Extensions IG. No existing element, binding, or expression is
altered. Existing servers that do not implement the new parameter remain
conformant; servers that do implement it gain a new query capability.

### Related Specifications

- `StructureDefinition/workflow-episodeOfCare` (Extensions IG) — the
  extension whose `value[x]` is the target of this search parameter.
  Its `<context>` list is the authoritative source for the new
  `SearchParameter.base` list.
- `EpisodeOfCare` (FHIR Core) — the `target` resource for the new
  reference search parameter. No core changes are required.
- Sibling precedents in the Extensions IG:
  `SearchParameter/workflow-researchStudy`,
  `SearchParameter/workflow-barrierRef`,
  `SearchParameter/workflow-barrierCode`,
  `SearchParameter/workflow-protectiveFactorRef`,
  `SearchParameter/workflow-protectiveFactorCode`.
