## Resolution Summary

The resolution (`Persuasive with Modification`) approves rewording the
`CapabilityStatement.imports` element comment so that it no longer
contradicts the element's definition. The current comment forbids any
overlap "directly or indirectly" between imported CapabilityStatements,
which is incompatible with the definition's allowance for the importing
CapabilityStatement to "repeat the same resources, interactions and
operations to add additional details to them".

The approved replacement text (verbatim from the resolution) is:

> The contents of any sibling imported CapabilityStatements SHALL NOT
> overlap., i.e. they cannot refer either directly or in their hierarchy
> to the same rest/resource, operations/name, searchparam/name,
> interaction/code, messaging/endpoint, document/mode pair as is found
> in a sibling import. However the importing CapabilityStatement is
> permitted to overlap - i.e. it may further refine rules around
> interactions, operations, etc. imported from another
> CapabilityStatement. If a CapabilityStatement has a hierarchy of
> imports, the whole hierarchy must be navigated to determine the full
> set of resources, interactions, operations, etc. that are being
> imported (including any refinements at lower levels of the hierarchy)
> when checking for overlap with sibling imports.

In short: distinguish "sibling imports" (which still must not overlap)
from "hierarchical / chained imports" (where the importer may legally
refine the imported capabilities). The pre-existing second paragraph of
the comment ("A capability statement that imports another
CapabilityStatement automatically instantiates it too…") is left in
place.

> **Note on current repo state:** PR #3784 (commit `2f9c759f29`,
> 2025-10-20) has already landed this exact wording change on `master`,
> with two minor copy-edit deviations from the resolution string: a
> trailing punctuation cleanup (`overlap., i.e.` → `overlap, i.e.`) and
> using a comma instead of an em-dash before "i.e. it may further
> refine". The plan below describes the work as the verbatim spec
> change; if ratifying the merged form is acceptable, only the
> validation / cross-link checks below remain outstanding.

## Feature Proposal

### Problem Statement

The element comment on `CapabilityStatement.imports` directly
contradicts the element's `definition`:

- **Definition** says the importing server "may repeat the same
  resources, interactions and operations to add additional details to
  them."
- **Comment** says "the contents of any directly or indirectly imported
  CapabilityStatements SHALL NOT overlap, i.e. they cannot refer to the
  same rest/resource, operations/name, searchparam/name,
  interaction/code, messaging/endpoint, document/mode pair."

Implementers cannot reconcile these statements: the definition permits
an importer to refine an imported resource entry, but the comment
appears to forbid the importer (or any indirect import) from referring
to the same resource at all. This was raised on the
`#implementers > CapabilityStatement imports` Zulip thread (stream
179166) which led to the ticket being filed.

### Proposed Change

Edit one element comment in the FHIR Core CapabilityStatement
StructureDefinition.

- **File:** `source/capabilitystatement/structuredefinition-CapabilityStatement.xml`
- **Element:** `CapabilityStatement.imports` (`<element id="CapabilityStatement.imports">`, currently around line 543)
- **Change:** Replace the first paragraph of the `<comment value="…"/>`
  attribute. Specifically, the leading sentence

  > the contents of any directly or indirectly imported
  > CapabilityStatements SHALL NOT overlap, i.e. they cannot refer to
  > the same rest/resource, operations/name, searchparam/name,
  > interaction/code, messaging/endpoint, document/mode pair.

  is replaced with the resolution text quoted in the **Resolution
  Summary** above.

- **Preserve:** the second paragraph of the comment (separated by
  `&#xA;&#xA;`):

  > A capability statement that imports another CapabilityStatement
  > automatically instantiates it too (though this is often not a very
  > useful statement for the kinds of CapabilityStatements that are
  > suitable for importing).

- **Do not change:** `<short>`, `<definition>`, `<min>`, `<max>`,
  `<type>`, or `<isSummary>` on the same element. Cardinality remains
  `0..*`, type remains `canonical(CapabilityStatement)`, and the
  element stays in the summary set.

No other source files require edits — `Select-String` over
`source/capabilitystatement/*.{xml,html}` and the broader `source/`
tree shows the phrase "SHALL NOT overlap" only in
`structuredefinition-CapabilityStatement.xml`. The
`capabilitystatement.html` narrative does not duplicate this comment.

### Design Rationale

The new wording introduces an explicit distinction that the original
comment lacked:

1. **Sibling imports** — two CapabilityStatements imported by the same
   parent — must not collide on the listed identity tuples
   (`rest/resource`, `operations/name`, `searchparam/name`,
   `interaction/code`, `messaging/endpoint`, `document/mode`). This
   keeps the union of sibling capabilities unambiguous.
2. **Hierarchical / chained imports** — an importer importing a
   CapabilityStatement that itself imports more — may legally repeat
   identifiers from any of its (transitive) imports because the
   importer's purpose is precisely to *refine* them. This matches the
   element's existing definition and the natural use case for the
   element.
3. **Hierarchy traversal** — overlap checking against siblings is
   defined over the *flattened* import hierarchy, so refinements at
   deeper levels still count for the sibling-overlap test.

This is a documentation / definitional clarification only: no
StructureDefinition cardinality, type, binding, or invariant is
introduced or modified. Choosing to clarify the existing comment (vs.
adding a new constraint or invariant) is consistent with FHIR
Infrastructure's preference for narrative clarification when the
underlying conformance rule is unchanged. The Zulip thread and the
ticket comments both pointed at "what does 'overlap' mean" as the core
ambiguity, and the resolution addresses exactly that.

### Step-by-Step Tasks

#### HL7/fhir

1. **Replace the `CapabilityStatement.imports` element comment.**
   - File: `source/capabilitystatement/structuredefinition-CapabilityStatement.xml`
   - Action: Inside `<element id="CapabilityStatement.imports">`,
     replace the entire `<comment value="…"/>` attribute value with the
     two-paragraph string below, joined by `&#xA;&#xA;` exactly as the
     surrounding XML already does:

     - Paragraph 2 (unchanged): "A capability statement that imports
       another CapabilityStatement automatically instantiates it too

     does, and make sure `&#xA;` is preserved (not converted to a raw

2. **Sanity-check for stray copies of the legacy wording.**
   - File: repo-wide search
   - Action: Run `git grep -n "directly or indirectly imported"` and
