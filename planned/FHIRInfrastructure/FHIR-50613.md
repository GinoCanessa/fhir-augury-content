## Resolution Summary

**Resolution:** Persuasive.

Resolution description (from the ticket): *"Will add a clarification indicating
that the relationship is an inclusive or, however they must all express the same
behavior."*

Concretely, `SubscriptionTopic.trigger` exposes three parallel ways to express a
trigger's criteria — `queryCriteria`, `fhirPathCriteria`, and `event` — and the
current narrative does not state what populating more than one at the same time
means. The approved change is a narrative/definition clarification only: when
more than one of those child elements is populated on a single trigger, the
relationship is an inclusive OR, **and** all populated forms must express the
same behavior (i.e., they must be semantically equivalent translations of the
same trigger condition). No new invariant / constraint is being added (the
ticket explicitly considered `consider adding a constraint to only allow one`,
and the resolution chose clarification over a constraint).

## Feature Proposal

### Problem Statement

The `SubscriptionTopic.trigger` backbone element defines a trigger via one of
three criterion carriers: `queryCriteria` (FHIR query), `fhirPathCriteria`
(FHIRPath expression), and `event` (CodeableConcept). The narrative for
SubscriptionTopic and the Topic-Based Subscription pages clearly describe each
carrier individually, but neither the resource-level prose nor the element
definitions in
`source/subscriptiontopic/structuredefinition-SubscriptionTopic.xml` state what
it means to populate more than one on the same trigger — AND vs OR, or whether
they are required to be semantically equivalent translations of one another.
This ambiguity was reported in FHIR-50613 by Weiyu Zhang.

### Proposed Change

Add a clarifying **comment** to the `SubscriptionTopic.trigger` element in the
StructureDefinition differential, stating that `queryCriteria`,
`fhirPathCriteria`, and `event` are inclusive-OR when more than one is present
on the same trigger, **and** that when multiple are populated they must all
express the same trigger behavior (i.e., be semantic translations of one
another, not additional independent conditions).

- **File:**
  `cache/github/repos/HL7_fhir/clone/source/subscriptiontopic/structuredefinition-SubscriptionTopic.xml`
  (authoring path in `HL7/fhir`:
  `source/subscriptiontopic/structuredefinition-SubscriptionTopic.xml`).
- **Element:** `SubscriptionTopic.trigger` (currently at lines 600–614 of the
  differential; has `short`, `definition`, cardinality `0..*`, `type

- **New element-level `<comment>`** (to be inserted after line 603's

  XML snippet to add (immediately after the existing `<definition … />` on

  ```xml

However, when multiple are populated they are expected to express the same trigger behavior (i.e., they are semantic translations of one another) — not independent additional trigger conditions. FHIRPath invariant is added (the resolution explicitly rejected the
"constraint to only allow one" alternative proposed in the ticket description).

### Design Rationale

- The resolution explicitly chose *clarification* over *constraint*. Adding a
  constraint (e.g., FHIRPath invariant `queryCriteria.empty() or
  fhirPathCriteria.empty()` etc.) would be a backward-incompatible behavioral
  change to R5 / R6 SubscriptionTopic and is out of scope.
- Placing the clarification on `SubscriptionTopic.trigger` (the parent
  BackboneElement) rather than duplicating it on each of the three child
  elements keeps the information authoritative in one location, next to the
  existing "Multiple triggers are considered OR joined" statement in the
  element's `definition`. The two OR-semantics statements (trigger-level vs.
  criteria-carrier-level) then live adjacent to one another and are easier to
  keep consistent.
- Using `<comment>` rather than extending `<definition>` follows the FHIR Core
  convention already used on neighbouring elements in this SD — e.g.,
  `SubscriptionTopic.trigger.description` (line 619),
  `SubscriptionTopic.trigger.resource` (line 631), and
  `SubscriptionTopic.trigger.queryCriteria` (line 676) each carry supplemental
  usage guidance in a `<comment>`.

### Open Questions

- The resolution phrase "inclusive or, however they must all express the
  same behavior" is slightly tension-y: inclusive OR on semantically
  equivalent forms is effectively redundant. The proposed comment resolves
  this by framing multiple populated carriers as *semantic translations* of
  a single trigger — reviewers should confirm that wording matches the WG's
  intent before merge.
- Whether to additionally add a short paragraph in
  `source/subscriptiontopic/subscriptiontopic-notes.xml` (Implementation
  Notes) restating the clarification for implementers is a style choice;
  the resolution only commits to adding the clarification (singular). The
  plan above keeps the change surgical and at the element level, matching
  the Repo Context recipe "Update narrative / prose" with the smallest
  possible footprint. If the WG prefers a narrative-page echo as well, add
  a second task to this plan — the briefing's gotcha about anchor hygiene
  does **not** apply here because no `<a name="…"/>` is being added or
  removed.
- No Zulip discussions, related Jira tickets, or GitHub PRs/issues were
  surfaced by `cross-referenced FHIR-50613` (single self-referential hit
  only). If the assignee is aware of an off-record discussion that narrowed
  the wording further, it should be folded into Task 1's `<comment>` text
