## Resolution Summary

Resolution: **Persuasive with Modification**.

The FHIRPath page’s `resolve()` section is to be expanded in two ways:

1. **Add clarifying prose** that engine behavior on resolution is implementation-defined:
   > "When implementing resolve(), it is at the discretion of the engine what protocols, FHIR formats and versions are supported and whether and which logical models are handled."
2. **Add a concrete example** demonstrating that `resolve()` operates on plain URI/URL/canonical strings (not only on `Reference`):
   ```
   DocumentReference.content.attachment.url.resolve() // When using DocumentReference as an index entry to resources stored elsewhere
   ```

No behavioral or schema change — this is a documentation-only change to a single specification page.

## Feature Proposal

### Problem Statement

The `resolve()` function description on the FHIRPath page (`source/fhirpath.html`, section `fn-resolve`, lines 344–356 at clone HEAD `db79dcd`) states that a string that is a `uri`, `canonical`, or `url` can be resolved, but the two worked examples only exercise `Reference` inputs:

```fhirpath
Patient.managingOrganization.resolve().name
Observation.subject.where(resolve() is Patient)
```

Implementers and spec readers have reported (per ticket description) that it is not obvious from the current wording that `resolve()` can be invoked directly on a URI-typed element such as `Attachment.url`. The section also does not set expectations about engine latitude (supported protocols, formats, versions, logical models), which has caused interoperability confusion.

### Proposed Change

Edit `source/fhirpath.html` in `HL7/fhir`, in the `resolve()` function section anchored at `<a name="fn-resolve">` (currently lines 344–356):

1. **Add a new paragraph** after the existing two paragraphs describing `resolve()` behavior and before the `<pre class="fhirpath">` example block:
   ```html
   <p>When implementing <code>resolve()</code>, it is at the discretion of the engine what protocols,
      FHIR formats and versions are supported and whether and which logical models are handled.</p>
   ```
2. **Append a third example line** inside the existing `<pre class="fhirpath">` block, after the two current examples:
   ```fhirpath
   DocumentReference.content.attachment.url.resolve() // When using DocumentReference as an index entry to resources stored elsewhere
   ```

Resulting block (illustrative):

```html
<a name="fn-resolve"> </a>
<h5><b>resolve() : collection</b></h5>
<p>For each item in the collection, if it is a string that is a <a href="datatypes.html#uri">uri</a>
   (or <a href="datatypes.html#canonical">canonical</a> or <a href="datatypes.html#url">url</a>), locate the
   target of the reference, and add it to the resulting collection. If the item does not resolve to a resource,
   the item is ignored and nothing is added to the output collection.</p>
<p>The items in the collection may also represent a Reference, in which case the <code>Reference.reference</code> is resolved.
  If the input is empty, the return value will be an empty collection.</p>
<p>When implementing <code>resolve()</code>, it is at the discretion of the engine what protocols,
   FHIR formats and versions are supported and whether and which logical models are handled.</p>

<pre class="fhirpath">
Patient.managingOrganization.resolve().name
Observation.subject.where(resolve() is Patient)
DocumentReference.content.attachment.url.resolve() // When using DocumentReference as an index entry to resources stored elsewhere
</pre>
```

### Design Rationale

- The resolution (`metadata.resolution_description`) quotes the exact text and example to add verbatim; the implementation is to transcribe those into the FHIRPath page with no interpretive changes.
- Placing the engine-latitude paragraph *before* the example block keeps it adjacent to the normative behavior description (it qualifies "locate the target of the reference"), and ensures the example block still reads as a single contiguous code sample.
- Appending the new example to the existing `<pre class="fhirpath">` rather than creating a second `<pre>` block preserves the page's rendered structure and matches the style already used for other multi-line function examples elsewhere on the page.
- The example intentionally uses `Attachment.url` to address the reporter's concrete scenario (resolving a URI on `DocumentReference.content.attachment`). The inline comment `// When using DocumentReference as an index entry to resources stored elsewhere` is carried through from the resolution text to clarify *why* this pattern is useful.
- Alternatives considered and rejected:
  - Editing the datatypes page (`uri`, `canonical`, `url`) instead of the FHIRPath page — rejected because the ticket was explicitly raised on, and triaged to, the FHIRPath Core page (see the single comment from `corey_spears`: *"Issue raised on the FHIR Core FHIRPath page, assigning to FHIR-I"*).
  - Introducing normative "SHALL/SHOULD" language around engine latitude — rejected; the resolution wording is deliberately permissive ("at the discretion of the engine").
