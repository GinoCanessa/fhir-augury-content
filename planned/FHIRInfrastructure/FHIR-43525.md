## Resolution Summary

The ticket title — "Work on TODO (second one)" — and description quote a single
TODO embedded in the GraphDefinition resource notes:

> *Todo: how would this be validated? - where is the graph referred to?*

The resolution is **Persuasive**. The resolution description points at the
companion ticket [FHIR-43656](https://jira.hl7.org/browse/FHIR-43656)
("GraphDefinition examples contain todo items"), whose resolution states
explicitly:

> "Will remove **'Todo: how would this be validated? - where is the graph
> referred to?'** as it's resolved by
> [FHIR-44865](https://jira.hl7.org/browse/FHIR-44865)."

So the concrete required change is: **delete the "Todo: how would this be
validated? — where is the graph referred to?" paragraph** from the
GraphDefinition notes page. The underlying question (how a GraphDefinition is
actually invoked / validated) is being addressed by FHIR-44865 ("Validate
should also be able to take a GraphDefinition"), which adds a `graph`
parameter to `$validate` — and so this stand-alone TODO no longer needs to
remain in the prose.

## Feature Proposal

### Problem Statement

The published GraphDefinition resource notes contain an unresolved authoring
TODO that leaks editorial uncertainty into the normative-track narrative.
Specifically, immediately after the table that defines the four compartment
consistency codes (`identical`, `matching`, `different`, `custom`), the page
asks the reader the rhetorical/editorial question:

> *Todo: how would this be validated? — where is the graph referred to?*

This is unhelpful for implementers, signals an unfinished spec, and — per the
FHIR-43656 resolution and FHIR-44865 — is now substantively answered by the
addition of a `graph` parameter to the `$validate` operation. The TODO should
therefore be removed (not answered inline), since the answer belongs in the
`$validate` operation documentation rather than in the GraphDefinition notes.

### Proposed Change

Edit `input/pagecontent/StructureDefinition-GraphDefinition-notes.xml` in the
**HL7/api-incubator-ig** repository (GraphDefinition has migrated out of
HL7/fhir into this incubator IG):

1. Remove the paragraph at approximately **line 164–166**:

   ```xml
   <p>
   Todo: how would this be validated? - where is the graph referred to?
   </p>
   ```

2. Do **not** replace it with answer prose in this file. Per the resolution,
   the question is resolved by FHIR-44865, which lives on the `$validate`
   operation. Optionally, a single-sentence cross-reference may be added in
   place of the TODO, e.g.:

   ```xml
   <p>
   A <code>GraphDefinition</code> can be supplied to the
   <a href="OperationDefinition-Resource-validate.html">Resource $validate</a>
   operation via its <code>graph</code> parameter, which a server may use to
   check that an instance and its references conform to the graph (see
   <a href="https://jira.hl7.org/browse/FHIR-44865">FHIR-44865</a>).
   </p>
   ```

   This is offered as an enhancement; the strict requirement of this ticket
   (per FHIR-43656) is removal of the TODO paragraph only.

3. Leave the surrounding compartment-consistency table and the `<a name="examples"></a>`
   anchor / `<h4>Simple Examples</h4>` heading that follow the TODO untouched.

No StructureDefinition, ValueSet, CodeSystem, search-parameter, or example
instance changes are required by this ticket. The other TODOs visible in the
same file (lines 33, 202, 232) are out of scope here — they belong to
FHIR-43656 (the example TODOs) and to other companion tickets.

### Design Rationale

- The TODO is editorial, not normative. Removing it cleans up the publication
  without changing any conformance behaviour.
- The substantive question ("how would this be validated?") is being addressed
  in a different artifact (`$validate` operation, via FHIR-44865), which is
  the correct location for invocation/validation semantics — the
  GraphDefinition resource page should describe the resource, not the
  operations that consume it.
- Adding a soft cross-reference (option 2 in *Proposed Change*) preserves
  discoverability for readers who would have reached for the TODO answer,
  without duplicating content that belongs on the operation page. The applied
  vote on FHIR-43656 only mandates removal, so the cross-reference is left as
  an editorial option.
