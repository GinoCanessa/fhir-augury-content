## Resolution Summary

Resolution: **Persuasive**. Resolution description: *"Do this, changing 'downright miserable' to 'exceptionally complex'"*.

That is, apply the two changes proposed in the ticket description verbatim, with the single agreed wording substitution:

1. Reorganise the *Slicing* section of `profiling.html` so that *Discriminator*, *Slice Cardinality*, and *Default Slice* become children (sub-sections) of *Slicing*, since they all describe aspects of slicing.
2. Add a new sub-topic **"Slicing Snapshot Generation"** under *Slicing* with the prose supplied in the ticket, **with the word "downright miserable" replaced by "exceptionally complex"** (per the resolution).

The new sub-topic text (after substitution) reads:

> "Snapshot generation with slicing is complicated, because there are two sets of rules that theoretically come into play — the rules inherited from a slice declared in a parent profile, and rules defined on the 'base' slicing element in the derived profile. This essentially provides two 'base' elements, and there are edge cases that make trying to combine multiple bases plus the differential **exceptionally complex**.
>
> For this reason, when generating snapshots, the constraints of the base (slicing) elements are **not** included in the snapshots produced for the slices. The base for a slice in a derived profile is the same slice in the snapshot of the parent profile. If there is no corresponding slice in the parent profile, then the base for the slice is the 'base' element in the parent profile.
>
> Even though the constraints of the slicing element are not reflected in the snapshots of the slices, all of the constraints in the slicing element still apply. For example, if a binding is present in the slicing element, that binding holds for all slices. If a slicing element declares mustSupport, then the element as a whole is mustSupport, which intrinsically means that all slices are automatically mustSupport (whether they declare it or not)."

The resolution is documentation/clarification only — no normative SHALL is added (consistent with the ticket comments from Lloyd McKenzie and Grahame Grieve). This is not a behavioural change to snapshot generation algorithms; it captures long-standing behaviour in prose.

## Feature Proposal

### Problem Statement

The current `profiling.html` page presents *Slicing*, *Discriminator*, *Slice Cardinality*, and *Default Slice* as four sibling `<h3>` sections (lines 519–520, 576–577, 772–773, 785–786 of `source/profiling.html`). The latter three are conceptually all aspects of slicing, but the page hierarchy does not express that relationship, and there is no documentation of how snapshots are generated for slices.

That gap surfaces as real-world confusion. The cross-referenced Zulip thread *"[IG creation] Slices not inheriting preferred bindings from root"* (Zulip ids 512422089 and 513935731) shows IG authors and tooling implementers disagreeing about whether constraints declared on the slicer element (e.g., a `binding` or `mustSupport`) propagate into the snapshots of individual slices. The behaviour has been consistent in tooling for years (the slicer's constraints are *not* copied into per-slice snapshots, but they *do* still apply), but it is undocumented in the spec, leading to inconsistent author expectations and tooling validation messages.

The ticket notes (Lloyd McKenzie comments) confirm the intent is **clarification**, not a change to the algorithm: "This isn't a change, it's documenting what the snapshot generation behavior has long been."

### Proposed Change

**Change 1 — Reparent three `<h3>` sub-topics under "Slicing".**

Within `source/profiling.html`:

- The sections currently rendered as `<h3>Discriminator</h3>` (line 577, anchor `discriminator`), `<h3>Slice Cardinality</h3>` (line 773, anchor `slice-cardinality`), and `<h3>Default Slice</h3>` (line 786, anchor `default-slice`) become children of `<h3>Slicing</h3>` (line 520, anchor `slicing`).
- In the bespoke HL7 publisher's HTML authoring style, the standard way to express that subordination is to demote the three child headings from `<h3>` to `<h4>`, while leaving `<h3>Slicing</h3>` as the parent. The existing `<a name="discriminator"></a>`, `<a name="slice-cardinality"></a>`, and `<a name="default-slice"></a>` anchors must be retained verbatim (other pages and external IGs link to them).
- Do not change the surrounding text content of those three sub-sections.

**Change 2 — Add a new "Slicing Snapshot Generation" sub-topic.**

Insert a new `<h4>` sub-section as the **last child** of *Slicing* — i.e., after the new `<h4>Default Slice</h4>` block (currently ending at line 805) and before the next sibling `<h3>Profiling portions of a resource</h3>` block at the existing `<a name="partial"></a>` anchor (line 807).

Suggested authored block (HTML, matching the surrounding style):

```html
<a name="slicing-snapshot-generation"></a>
<h4>Slicing Snapshot Generation</h4>
<p>
Snapshot generation with slicing is complicated, because there are two
sets of rules that theoretically come into play - the rules inherited
from a slice declared in a parent profile, and rules defined on the
'base' slicing element in the derived profile. This essentially provides
two 'base' elements, and there are edge cases that make trying to
combine multiple bases plus the differential exceptionally complex.
</p>
<p>
For this reason, when generating snapshots, the constraints of the base
(slicing) elements are <b>not</b> included in the snapshots produced for
the slices. The base for a slice in a derived profile is the same slice
in the snapshot of the parent profile. If there is no corresponding
slice in the parent profile, then the base for the slice is the 'base'
element in the parent profile.
</p>
<p>
Even though the constraints of the slicing element are not reflected in
the snapshots of the slices, all of the constraints in the slicing
element still apply. For example, if a binding is present in the
slicing element, that binding holds for all slices. If a slicing
element declares mustSupport, then the element as a whole is
mustSupport, which intrinsically means that all slices are
automatically mustSupport (whether they declare it or not).
</p>
```

Anchor name `slicing-snapshot-generation` is recommended for consistency with the existing kebab-case anchors on this page (`slice-cardinality`, `default-slice`, `slicing`).

### Design Rationale

- The hierarchy change reflects how the topics are actually related; *Discriminator*, *Slice Cardinality*, and *Default Slice* describe aspects of *Slicing*, not parallel concepts. Reparenting them via `<h3>` → `<h4>` (rather than physical re-ordering) is the minimum-disruption way to express that in the existing HTML authoring style and preserves all inbound `#discriminator`, `#slice-cardinality`, and `#default-slice` anchor links.
- The new sub-topic codifies behaviour that has long been implemented identically across HL7 tooling (publisher / validator / sushi). Comments on the ticket (Lloyd McKenzie x2) confirm that the intent is to document, not change, this behaviour. Grahame Grieve explicitly notes a SHALL would be inappropriate ("it's too late to make this a SHALL, and we should settle for a validation warning"); accordingly the prose deliberately avoids normative SHALL/SHOULD and uses descriptive language.
- The wording substitution "downright miserable" → "exceptionally complex" was the explicit term of the resolution (Lloyd offered "exceptionally difficult" in comments; the resolution chose "exceptionally complex"). Use the resolution's exact wording.
- Ehaas's alternative re-wording (proposed in a comment) was *not* the wording adopted by the work group as the resolution, so it is not implemented here. It is captured under Open Questions in case the editor prefers to weave any of it in non-normatively, but the default position is to apply the exact text from the ticket description (with the one substitution).
