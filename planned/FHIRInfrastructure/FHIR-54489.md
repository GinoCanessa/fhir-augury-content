## Resolution Summary

**Resolution: Persuasive.** The committee accepts the report that the
per-resource "Version Diff" tab is showing incorrect "code removed"
deltas for value-set bound elements whose codes live below the top
level of the bound ValueSet (i.e. nested / hierarchical codes with
level > 1). The resolution explicitly classifies this as a **tooling
issue** rather than a specification change and directs the fix to the
publisher tooling: **HL7/kindling**, tracked as
[`HL7/kindling#212` — "Diff logic isn't handling nested codes"](https://github.com/HL7/kindling/issues/212).

The resolution text also notes that, regardless of when the kindling
fix lands, the ballot publication will continue to carry the standing
caveat that **the version-diff view is not normative and is not
subject to ballot**. No change is required in the FHIR Core
specification source itself, in any of the affected resources
(DocumentReference / Condition / Observation), or in the bound
ValueSets — the underlying value sets and bindings are correct; only
the diff renderer mis-reports them.

A duplicate ballot comment ([FHIR-54571](https://jira.hl7.org/browse/FHIR-54571))
was filed by Patient Care and transferred to FHIR-I; both resolve via
the same kindling tooling fix.

## Feature Proposal

### Problem Statement

When the FHIR R6 ballot was rendered, the per-resource **Version Diff**
tab reported that several codes had been *removed* from the ValueSets
bound to required-strength `code`-typed elements, even though:

1. The element's normal documentation continued to show those codes as
   valid required values, and
2. The bound ValueSets themselves still contained the codes (no
   removal, no deprecation).

Concrete instances reported by the submitter and confirmed against the
ballot rendering:

| Resource | Element | Binding |
|---|---|---|
| DocumentReference | `docStatus` | composition-status (required) |
| Condition | `clinicalStatus` | condition-clinical (required) |
| Condition | `verificationStatus` | condition-ver-status (required) |
| Observation | `status` | observation-status (required) |

The common factor identified by the reporter is that every code that
the diff incorrectly flagged as "removed" is a **child code at level
> 1** in its ValueSet hierarchy (i.e. it is nested under a parent
concept rather than appearing as a top-level concept in the
`ValueSet.compose` / expansion). Top-level codes diff correctly. This
strongly indicates that the diff generator is only walking the first
level of the ValueSet expansion / `concept` tree when computing the
"prior version" set used for the comparison, and is therefore treating
every nested code as "missing from the previous version" → reporting
it as removed in the diff direction it is rendered.

The defect is purely cosmetic at the spec-content level but is
high-impact for ballot reviewers, who are using the diff view as their
primary signal of what changed between R5 and R6 and are being
mis-led into filing further unnecessary tracker items.

### Proposed Change

**Scope:** all changes are to publisher tooling. **No FHIR Core
specification artifacts (resources, profiles, ValueSets, CodeSystems,
extensions, search parameters, examples, narrative) need to be
modified.**

In **HL7/kindling**, fix the version-diff generator so that the
"existing codes" / "previous codes" comparison logic for a value-set
bound `code`/`Coding`/`CodeableConcept` element walks the **full
hierarchy** of the bound ValueSet (or its expansion), not just the
top-level concepts. Concretely, the fix in `HL7/kindling#212` should:

1. When building the "previous-version code set" and the
   "current-version code set" for an element's required/extensible
   binding, recursively descend `ValueSet.compose.include[*].concept`
   and any `concept.concept` children (and likewise any expansion
   `ValueSet.expansion.contains[*].contains` children), so every code
   at every depth contributes to the set.
2. Equivalently, prefer comparing **fully expanded** ValueSets over
   raw `compose` walks; if expansion is unavailable, ensure the
   recursive walk covers nested `concept` children.
3. Re-run the diff in unit / smoke tests against at least the four
   resource/element pairs cited above and confirm that the diff page
   reports **no spurious "code removed"** entries for them when the
   underlying bindings have not changed between R5 and the current
   build.

**Documentation caveat (already in place):** the published spec
already states that the version-diff view is informational and not
subject to ballot. No edit to that caveat is required by this ticket;
it is recorded here only because the resolution explicitly relies on
it.

### Design Rationale

The committee chose a tooling-only fix (rather than a normative spec
change) because:

- The underlying ValueSets, bindings, and element definitions are all
  correct. The defect is in how the publisher *compares* them between
  versions, not in what the spec says.
- A normative change would inappropriately treat a renderer bug as a
  spec issue, polluting the change log and forcing re-review of
  unchanged artifacts.
- The version-diff view is explicitly out-of-scope for ballot, so the
  bug does not block ballot acceptance — but it does cause noise (the
  duplicate FHIR-54571 was filed by another work group for the same
  symptom on Patient Care resources). Fixing it in kindling resolves
  every affected resource at once and prevents future duplicate
  trackers.
- Limiting the fix to kindling avoids any risk of perturbing the
  per-resource StructureDefinitions, which would invalidate
  downstream conformance artifacts unnecessarily.

## Implementation Plan

### Prerequisites

- Access to the **HL7/kindling** repository (this fhir-augury
  workspace does not have a clone of it; the implementer must work
  against an external checkout).
- A local FHIR Core build environment capable of running kindling
  end-to-end so the regenerated version-diff pages can be inspected.
- The four cited resource pages (DocumentReference, Condition,
  Observation; the first two via two elements each) available as a
  smoke-test fixture.

### Step-by-Step Tasks

#### HL7/kindling

1. **Reproduce the defect locally**
   - File: n/a (build/run only)
   - Action: Build kindling from current `master`, run the FHIR Core
     publisher, and open the version-diff tab for `DocumentReference`,
     `Condition`, and `Observation`. Confirm that nested codes from
     `composition-status`, `condition-clinical`,
     `condition-ver-status`, and `observation-status` are reported as
     "removed" in the diff.
   - Details: Capture the offending output as the regression baseline
     for task 5.

2. **Locate the version-diff code-set comparator**
   - File: kindling source for the per-resource version-diff renderer
     (the "existing codes comparison logic" Brian Postlethwaite
     referenced on Zulip).
   - Action: Identify the helper that materialises the set of bound
     codes for a `code` / `Coding` / `CodeableConcept` element from
     its bound ValueSet, used by the comparator that produces the
     "added" / "removed" code lists in the diff.
   - Details: Both the previous-version side and the current-version
     side must call the same helper; record both call sites.

3. **Fix the ValueSet walk to descend nested concepts**
   - File: same helper as task 2.
   - Action: Replace any single-level iteration over
     `ValueSet.compose.include[*].concept` with a recursive walk that
     also descends `concept.concept` children. If the helper consumes
     an expansion, replace the single-level iteration over
     `ValueSet.expansion.contains[*]` with a recursive walk that also
     descends `contains.contains` children. Prefer using a fully
     expanded ValueSet when one is available, falling back to the
     recursive compose walk otherwise.
   - Details: The set produced for each side of the diff must be the
     **transitive closure** of all codes at every depth, deduplicated
     by `system|code`.

4. **Apply the same fix to both sides of the comparator**
   - File: the comparator call sites identified in task 2.
   - Action: Verify (and adjust if necessary) that *both* the
     previous-version code set and the current-version code set are
     built via the recursive helper. Asymmetric behaviour will simply
     invert the bug into spurious "code added" entries.
   - Details: Add an assertion or unit precondition that both sides
     went through the same code path.

5. **Add regression coverage**
   - File: kindling test suite (test fixtures + assertions).
   - Action: Add a unit test that constructs a small ValueSet with
     codes at depths 1 and 2, runs the diff helper against an
     identical "previous" version, and asserts that the
     "added" and "removed" sets are both empty.
   - Details: Also add a smoke-level assertion (manual or scripted)
     that re-running the FHIR Core publisher produces a clean diff
     for `DocumentReference.docStatus`, `Condition.clinicalStatus`,
     `Condition.verificationStatus`, and `Observation.status`
     between R5 and the current build.

6. **Close the tracking issues**
   - File: n/a (issue tracker)
   - Action: On merge, close [`HL7/kindling#212`](https://github.com/HL7/kindling/issues/212)
     and reference both `FHIR-54489` and the duplicate `FHIR-54571`
     in the close comment so Patient Care reviewers see the
     resolution.
   - Details: Note in the close comment that the spec source did not
     change and that affected resource pages will diff correctly at
     the next publish.

### Testing Considerations

- **Resources to revalidate (publisher diff only, no spec
  validation):** DocumentReference, Condition, Observation, and any
  resource with a required or extensible binding to a hierarchical
  ValueSet.
- **Fixtures:** a small synthetic ValueSet containing concepts at
  depth 1 and depth 2, used in a kindling unit test as the
  "previous" and "current" inputs to assert empty add/remove sets.
- **Edge cases:**
  - ValueSets defined purely via `compose.include.concept` with
    `concept.concept` nesting.
  - ValueSets defined purely via an `expansion.contains.contains`
    tree.
  - ValueSets that mix `include.system` (no inline concepts) with
    `include.concept` siblings.
  - Bindings whose ValueSet legitimately *did* change between R5 and
    the current build — these must still produce correct, non-empty
    add/remove sets (do not over-correct into silence).
- **Interoperability considerations:** none — wire format and
  conformance behaviour are unaffected; the only artifact that
  changes is the human-readable diff page in the published spec.
