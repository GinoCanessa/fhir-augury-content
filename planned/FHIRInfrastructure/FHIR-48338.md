## Resolution Summary

**Resolution:** Persuasive with Modification.

**Resolution description (verbatim):** "We will consolidate the content in *Internal Id References* into the section *Linking between Data and Narrative*."

The approved change is editorial / narrative-prose only. The two adjacent subsections of `source/narrative.html` that both discuss `id`/`idref`-style linking between a resource's structured data and its XHTML narrative are to be merged into a single section, keyed off the existing *Linking between Data and Narrative* heading. The original reporter's request to bless `idref` references from `Composition.text` / `Composition.section.text` into a document `Bundle` is **not** accepted as written — comment from Lloyd McKenzie (2025-01-20) is definitive: the `id` uniqueness scope in FHIR is the resource, `/` is illegal in an `IDREF`, and cross-resource narrative links must use the existing `narrativeLink` / `originalText` extensions described in *Linking between Data and Narrative*. Grahame McKenzie's follow-up (2025-01-27) committed to rewriting the relevant section of the Composition spec, which the WG then scoped down to consolidating narrative.html sections at resolution time.

## Feature Proposal

### Problem Statement

`source/narrative.html` currently contains two separate `<h3>` sections that discuss the same underlying concern — how XHTML narrative links to structured data via `id`/`idref`:

- `<h3>Internal Id References</h3>` at line 219 (anchor `#internal`) describes the XML `id`/JSON `id` property, the resource-scoped uniqueness of `id` values, and the fact that when resources are combined in a `Bundle`, `id` duplicates across resources are allowed and must be managed by readers.
- `<h3>Linking between Data and Narrative</h3>` at line 339 (anchor `#linking`) describes the `narrativeLink` and `originalText` extensions — including the canonical `Bundle`/`Composition` example where narrative in one entry is referenced from structured data in another entry via a `fullUrl#fragment` URL.

Reporters at Connectathon 37 and in the IPS Zulip thread (stream 207835-IPS, topic *Referencing from IPS narrative to data*) read the two sections as being in tension: the "Internal Id References" section reads as though `idref` could resolve into another `Bundle` entry, while "Linking between Data and Narrative" makes it clear that cross-entry narrative references must go through `originalText` / `narrativeLink`. The WG's disposition is that the two sections tell one story and should be presented as one section, with the `id`/`idref` mechanics framed as an input to the cross-resource linking story rather than as a parallel, competing mechanism.

### Proposed Change

Perform a **single-file, documentation-only** edit to `source/narrative.html`:

1. **Fold the body of *Internal Id References* into *Linking between Data and Narrative*.** Move the prose paragraphs at lines 220–243 (inside `<h3>Internal Id References</h3>`) so that they appear at the top of the *Linking between Data and Narrative* section (starting at line 339), before the current introductory paragraph about `narrativeLink` / `originalText`. Order after the merge:
   1. Lead paragraph: `id`/`idref` attribute mechanics, incl. "in JSON, the property `id` is used which is equivalent to the XML attribute `id`".
   2. Resource-scoped uniqueness paragraph (`id` SHALL be unique within the resource; contained resources share the parent's scope).
   3. Bundle-scoped caveat paragraph (duplicate `id` values across entries are permitted and must be managed by readers) — explicitly framed as the reason cross-resource narrative links cannot rely on `id`/`idref` alone.
   4. Bridge sentence (new, one sentence) stating: because `idref` cannot cross resource boundaries and cannot carry the `Resource/id` prefix needed to do so, cross-resource narrative linking uses the extensions described below.
   5. Existing *Linking between Data and Narrative* intro paragraph (`narrativeLink` / `originalText`) and the two existing code examples (JSON `Condition` with `originalText`, XML `Bundle` with `Composition` + `Condition`).
   6. Preserve the "images not contained in the resource" paragraph currently at lines 238–243 — this is about embedding images, not about id/idref linking, and belongs earlier in the page alongside *Image References* (line 190). Move it up to the end of the *Image References* section rather than into the merged section.
2. **Delete the old `<h3>Internal Id References</h3>` heading** at line 219 along with the `<a name="internal"></a>` anchor at line 218, but **preserve backward compatibility for inbound links** by keeping `<a name="internal"></a>` as a synonym anchor on the merged *Linking between Data and Narrative* section (placed immediately next to the existing `<a name="linking"></a>` anchor at line 338). Two pages in this repo link to `narrative.html#internal` (see Impact Analysis) and the anchor must continue to resolve to the consolidated section.
3. **No schema, StructureDefinition, ValueSet, CodeSystem, or search-parameter changes.** No change to `Composition`, `Composition.text`, `Composition.section.text`, `Bundle`, `Reference`, `narrativeLink`, or `originalText`. The reporter's original ask — blessing `idref` targets that reach into other `Bundle` entries — is explicitly **not** being adopted; the merged section will continue to state the same invariants that *Internal Id References* and the Lloyd McKenzie comment already assert.

### Design Rationale

- The WG accepted the reporter's point that the spec is confusing, but rejected the proposed mechanics. The `XML Schema` rules for `ID`/`IDREF` do not allow `/` and scope resolution to the enclosing "document" (for FHIR, a single resource). Any prose that suggested `idref` could cross resource boundaries would directly contradict the XML/JSON serialization rules that FHIR has shipped for years and would be a breaking behavioural change for validators. Consolidating the two narrative sections lets the spec state the constraint once and immediately pivot to the supported cross-resource pattern (`narrativeLink` / `originalText`).
- Merging into *Linking between Data and Narrative* (rather than the reverse) is preferred because that section already contains the canonical `Bundle`/`Composition`/`Condition` example that the IPS thread was really asking about, and because it already carries the generic title.
- Keeping `<a name="internal"></a>` on the merged section avoids breaking the two existing inbound links (`source/references.html` line 738, `source/xml.html` line 50) and any external deep-links to `narrative.html#internal` — a non-trivial concern for a page that has been stable for several FHIR versions.
