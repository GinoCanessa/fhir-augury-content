## Resolution Summary

The ticket is resolved as **Persuasive with Modification**. The approved
resolution (verbatim from the Jira `resolution_description`) is:

> Will update `SubscriptionStatus.type` to be `coding` with an extensible
> binding to the same value set.

The original request was to add an "unknown"/"other" escape-valve concept
to the required-bound value set. Instead of extending the value set, the
work group chose the structurally equivalent fix: change the type of
`SubscriptionStatus.type` from `code` to `Coding` and relax the binding
strength from `required` to `extensible`, keeping the same value set
(`http://hl7.org/fhir/ValueSet/subscription-notification-type`). That
allows senders to convey an out-of-value-set concept when the predefined
codes do not fit, without polluting the canonical value set with generic
"unknown"/"other" codes.

## Feature Proposal

### Problem Statement

`SubscriptionStatus.type` is currently defined as:

- Type: `code`
- Cardinality: `1..1`
- `isModifier` = true
- Binding: `required` to
  `http://hl7.org/fhir/ValueSet/subscription-notification-type`

The referenced value set contains only five concrete notification types
(`handshake`, `heartbeat`, `event-notification`, `query-status`,
`query-event`) and no catch-all concept. Because the element is mandatory
and required-bound, a sender that encounters a scenario the five codes
don't describe has no conformant way to populate the element — the
resource becomes impossible to construct. This was flagged by Lloyd
McKenzie (Comment 163) as a missing "escape valve" on a 1..1 required
element.

### Proposed Change

Apply the resolution literally in
`HL7/fhir`, `source/subscriptionstatus/`:

1. **Element type change on `SubscriptionStatus.type`** in
   `structuredefinition-SubscriptionStatus.xml`:
   - `<type><code value="code"/></type>` → `<type><code value="Coding"/></type>`
   - Keep `min=1`, `max=1`, `isModifier=true`, `isSummary=true`.
   - Keep `isModifierReason` (or revisit it — see Open Questions — but
     it is not part of the resolution).

2. **Binding change on `SubscriptionStatus.type`** in the same file:
   - `<strength value="required"/>` → `<strength value="extensible"/>`
   - Keep `valueSet` =
     `http://hl7.org/fhir/ValueSet/subscription-notification-type`.
   - Keep the `SubscriptionNotificationType` binding name extension.
   - Update the `<binding><description>` to reflect that non-listed
     concepts are allowed when none of the listed concepts fit.

3. **Update the short description** on
   `SubscriptionStatus.type`. The current `short` is
   `"handshake | heartbeat | event-notification | query-status | query-event"`,
   which is the canonical `code`-typed shorthand. With a `Coding` +
   extensible binding, a more appropriate short is something like
   `"Type of notification, typically: handshake | heartbeat | event-notification | query-status | query-event"`,
   mirroring the convention used elsewhere for extensible-bound `Coding`
   elements.

4. **Update invariants `sst-1` and `sst-2`** in
   `structuredefinition-SubscriptionStatus.xml`. Their FHIRPath
   expressions compare `type` directly against string literals, which
   only works when `type` is a primitive `code`. Once `type` is a
   `Coding`, the comparisons must target the child `code` element:
   - `sst-1` current:
     `(type = 'event-notification' or type = 'query-event') implies notificationEvent.exists()`
   - `sst-1` new:
     `(type.code = 'event-notification' or type.code = 'query-event') implies notificationEvent.exists()`
   - `sst-2` current:
     `type = 'query-status' implies status.exists()`
   - `sst-2` new:
     `type.code = 'query-status' implies status.exists()`

5. **Update every example and invariant-test instance** under
   `source/subscriptionstatus/` that populates `SubscriptionStatus.type`.
   All existing instances use the primitive form
   `<type value="event-notification"/>`; with `Coding` this must become:

   ```xml
   <type>
     <system value="http://hl7.org/fhir/subscription-notification-type"/>
     <code value="event-notification"/>
   </type>
   ```

   (System URL pulled from `codesystem-subscription-notification-type.xml`:
   `<url value="http://hl7.org/fhir/subscription-notification-type"/>`.)

6. **No change to the CodeSystem or ValueSet.** Per the resolution, we
   are not adding `unknown`/`other` codes; the existing content of
   `codesystem-subscription-notification-type.xml` and
   `valueset-subscription-notification-type.xml` stays as-is.

### Design Rationale

- Extensible binding on a `Coding` is the FHIR-standard escape valve:
  senders MUST use a listed concept when one fits, but MAY supply an
  alternative `Coding` (their own system + code, or a `text`-only
  `Coding`) when none fits. This directly addresses the reporter's
  "impossible to populate" concern without forcing HL7 to mint generic
  `unknown`/`other` concepts into an otherwise tightly-scoped code
  system.
- Keeping the existing value set (rather than adding catch-all codes)
  preserves the semantic cleanliness of
  `subscription-notification-type`: those five codes describe the five
  real notification types, and alternative codings are clearly marked as
  out-of-spec by virtue of not matching the value set.
- `Coding` (rather than `CodeableConcept`) matches the original
  single-concept cardinality; `type` is conceptually one notification
  type per status, not a list of translations.
- `isModifier=true` is preserved because the meaning of the resource
  still changes with `type` (e.g., whether `notificationEvent` is
  required via `sst-1`/`sst-2`).

### Breaking Changes

This is a **breaking change** on the wire format of `SubscriptionStatus`:

- `SubscriptionStatus.type` is a type change from `code` (an XML/JSON
  primitive) to `Coding` (a structured datatype). Every existing
  conformant instance (`<type value="event-notification"/>` in XML,
  `"type": "event-notification"` in JSON) becomes structurally invalid
  and must be rewritten to the `Coding` form. There is no
  forward-compatible way to read or produce the new element from code
  generated against the R5 definition.
- Relaxing the binding from `required` → `extensible` is, in isolation,
  non-breaking for senders that were already using the listed codes and
  is generally seen as a liberalisation for receivers. The element-type
  change is the dominant breaking aspect.
- Invariant expressions `sst-1` and `sst-2` change, which is not
  itself a wire-format break but will cause existing validators that
  cache the prior FHIRPath to report spurious failures until refreshed.

Because `SubscriptionStatus` is already marked `normative` in its
CodeSystem extension block, this change needs to be called out clearly
in release notes / the resource's change log.
