## Resolution Summary

Resolution: **Not Persuasive with Modification**.

Verbatim resolution description:

> Will will adjust this binding to be version-independent, which will make the problem go away. We will advise tool-smiths to behave as though prior releases of FHIR also bound in a version-independent manner.

The committee did not accept the literal request to add an `unknown` / `other` "escape valve" code to the `FHIR-version` value set. Instead, the binding of `CapabilityStatement.fhirVersion` (and, by parity, the other elements bound to the same `FHIR-version` value set) is to be made *version-independent* — i.e., a "floating" binding whose contents are not frozen at FHIR Core's publication time and that can therefore legitimately contain FHIR versions published *after* the FHIR Core release that carries the `CapabilityStatement` definition. Tool-smiths are to be told (in spec prose) to retroactively treat the same binding in prior FHIR releases as having the same version-independent semantics.

Per ticket comments:

- The mechanism is intended to be analogous to the existing version-independent bindings used for language (BCP 47), MIME types (BCP 13), and UCUM — i.e., the value set definition is composed of an external/extensible code system rather than a locked, version-specific enumeration.
- Grahame Grieve was tagged to identify the concrete binding mechanism.
- A subsequent comment notes that THO (terminology.hl7.org) being version-bound at FHIR Core publication time means a package-based binding still locks to a specific snapshot — this caveat must be acknowledged in the plan and in the spec prose.

## Feature Proposal

### Problem Statement

`CapabilityStatement.fhirVersion` is `1..1` with a `required` binding to `http://hl7.org/fhir/ValueSet/FHIR-version`. That value set is composed from `http://hl7.org/fhir/FHIR-version`, a CodeSystem whose `concept` list is hand-maintained inside FHIR Core (`source/capabilitystatement/codesystem-FHIR-version.xml`, mirrored in the `FHIR-version` worksheet of `source/terminologies/bindings.xml`). The list therefore enumerates only the FHIR versions known *at FHIR Core publication time*.

Consequences:

- A server that is conformant to a *future* FHIR version (e.g., 7.0.0 published after FHIR R6) cannot legally express that fact in a CapabilityStatement built against an R6-bound resource definition, because `7.0.0` is not in the R6 snapshot of the value set.
- The element is mandatory (`min=1`) with `required` strength, so there is no legal "I don't know" or "newer than I know about" code.
- The same problem applies to every other element bound to the same value set: `ImplementationGuide.fhirVersion`, `ImplementationGuide.definition.resource.fhirVersion`, and `StructureDefinition.fhirVersion`.

The reporter originally proposed adding an `unknown` / `other` escape valve. The committee rejected that and instead resolved the deeper problem by changing the binding semantics from "locked to this FHIR version's snapshot of FHIR versions" to "version-independent (floating)".

### Proposed Change

Change the binding of every element currently bound to `http://hl7.org/fhir/ValueSet/FHIR-version` so that, instead of being a frozen enumeration of FHIR releases known at this FHIR Core publication time, it is treated as version-independent in the same way that BCP-13 (mime types), BCP-47 (languages), and UCUM are treated. Concretely:

1. **Re-author the `FHIR-version` value set as a floating value set.**
   - Keep `<url value="http://hl7.org/fhir/ValueSet/FHIR-version"/>` (do not break canonical references).
   - Replace its single `<compose><include><system value="http://hl7.org/fhir/FHIR-version"/></include></compose>` (currently implicit through the `valueSet` element on the CodeSystem) with an explicit, immutable=false ValueSet whose `<compose>` references the FHIR-version code system *without* version pinning, and whose narrative + description state that members are not constrained to versions published as of this FHIR Core release.
   - Set `<immutable value="false"/>` (the existing `version-independent-resource-types` value set is immutable=true, but unlike that set the FHIR-version set must be allowed to grow).
   - Add the same metadata extensions used on other floating value sets (wg=`fhir`, standards-status=`normative`, fmm=`5`).

2. **Mark the binding as floating on every ElementDefinition that consumes it.**
   - Add the binding-name extension (already present) plus the agreed-upon "floating binding" marker. Two viable mechanisms appear in FHIR today; pick one and apply consistently:
     - (a) A new ElementDefinition.binding extension `http://hl7.org/fhir/StructureDefinition/elementdefinition-binding-version-independent` (boolean, default false). This mirrors the explicit-flag pattern used for `elementdefinition-isCommonBinding`.
     - (b) Reuse / repurpose the conventions used for `mimetypes`, `all-languages`, and UCUM bindings (which today rely on the value set's compose pointing at an external URI namespace). For FHIR-version this is the chosen mechanism per the committee discussion; the binding itself does not need a new extension if the value set's `compose` is structured as floating.
   - Recommended: implement (b) (no new extension) because it matches the precedents named in the resolution comment ("language/mime/UCUM"). Keep (a) as a fallback to discuss with vocab if (b) proves insufficient.

3. **Update spec prose** to explain that:
   - The `FHIR-version` value set is a floating / version-independent binding. Implementers MAY use a code that names a FHIR version published after this release of FHIR Core.
   - Tool-smiths SHOULD treat the same binding in *prior* FHIR releases as having the same version-independent semantics (per the resolution).
   - The package-based binding caveat raised in the ticket comments (THO snapshots being locked at publication time) is acknowledged: the floating semantics are normative regardless of the THO snapshot a particular package happens to ship.

4. **No code list changes.** Do not add `unknown` or `other` to `codesystem-FHIR-version.xml` / the `FHIR-version` worksheet in `bindings.xml` — the resolution explicitly went a different direction.

5. **No cardinality / type changes** to the consuming elements. `min`, `max`, `type` (`code`), `isSummary`, and binding `strength` all remain as today.

### Design Rationale

- Adding an `unknown`/`other` code (the original proposal) would only paper over the gap and would propagate ambiguous values into ecosystems that need to reason about which FHIR version a server actually supports. The committee explicitly rejected that approach.
- Making the binding version-independent matches the mechanism FHIR already uses for code systems whose membership cannot reasonably be frozen at one release (BCP-13 mime types, BCP-47 languages, UCUM). Aligning on that established pattern keeps tooling consistent.
- Keeping the binding strength `required` preserves the constraint that `fhirVersion` MUST be a member of the (now-floating) FHIR-version value set; it does not weaken the conformance posture, it just acknowledges that the legal set evolves over time independent of FHIR Core's release cadence.
- Advising tool-smiths to behave retroactively this way (per the resolution wording) avoids forcing existing R4/R5 deployments to upgrade their definitions just to be allowed to declare a newer FHIR version.

### Breaking Changes

- **Wire-format breaking changes:** **None.** No element is added, removed, retyped, or re-cardinalized. Binding `strength` stays `required`. The set of *currently-known* FHIR version codes is not reduced.
- **Conformance-posture changes worth flagging:** A FHIR version code published after this FHIR Core release becomes valid where it was previously *technically* non-conformant. Validators that hard-coded the published-at-release-time membership (rather than the live value set) will need an update. This is an intentional, forward-compatible loosening — call it out in the changelog but do **not** classify as breaking.
- **Retroactive guidance:** The resolution explicitly directs tool-smiths to treat *prior* FHIR releases' bindings the same way. That is normative guidance, not a wire-format change, and so is not breaking — but downstream conformance tooling owners must be notified.
