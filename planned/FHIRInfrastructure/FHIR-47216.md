## Resolution Summary

The reporter pointed out that the FHIR Infrastructure "Version Management
Policy" page contains a "Note for balloters: the R5 packages will be
created when R5 is finalized. Until then, these are broken links." caption
underneath the cross-version-extensions package table — a comment that no
longer makes sense now that R5 has been finalized and R6 is in flight, and
that the R5 column header is missing from the same table.

The Work Group resolved the ticket as **Persuasive** with the following
exact resolution description:

> "We will update all tables on that page to have R6 as well as R5. (Will
> also fix the missing header for R5 in the mentioned table.)"

So the agreed change is two-fold:

1. Extend every package table on the Version Management Policy page so
   that it includes both an **R5** column/row and a new **R6** column/row
   (in addition to the existing R2/R3/R4 columns and R2/R3/R4/R5 rows).
2. Fix the missing **R5** header on the row that, in the description's
   table, currently has cells for `hl7.fhir.extensions.r4:1.0.2` /
   `:3.0.1` / `:4.0.1` / `n/a` but no "R5 Extensions:" label.

The trailing R5-specific balloter note becomes obsolete once the R5
column is real and populated, and should be removed (or rewritten to
reference R6 in the same way) per the resolution wording.

## Feature Proposal

### Problem Statement

The FHIR Infrastructure Work Group maintains a "Version Management
Policy" page (the source page is on confluence.hl7.org; the description
is a paste of the rendered Confluence content, which is why every table
cell carries the `confluenceTable` / `confluenceTd` CSS classes). On
that page there is a matrix of `hl7.fhir.extensions.r{N}` packages
showing, for each authored-against major FHIR version (R2/R3/R4/R5),
which package version is published against which target FHIR version
(R2/R3/R4 columns).

Two concrete defects exist on that table today:

1. The "R5 Extensions:" row in the original Confluence markup is missing
   its leading header cell — it has four data cells
   (`hl7.fhir.extensions.r4:1.0.2`, `:3.0.1`, `:4.0.1`, `n/a`) but no
   label cell, so the row renders without an "R5 Extensions:" caption
   even though the data clearly belongs to it.
2. The page predates R5 being final and predates R6 being on the radar:
   there is no R5 column, no R6 column, no R6 row, and the page closes
   with the now-stale balloter note "*Note for balloters: the R5
   packages will be created when R5 is finalized. Until then, these are
   broken links.*"

The reporter's title — "R5 comment in version management policy page" —
flags the stale balloter note specifically; the resolution broadens the
fix to also align the table itself with the R5/R6 reality.

### Proposed Change

Edit the FHIR Infrastructure "Version Management Policy" Confluence page
(confluence.hl7.org) as follows. All edits are to one HTML/Confluence
page; no FHIR resource, profile, or extension definition changes are
involved.

1. **Add the missing "R5 Extensions:" row header.** In the row whose
   first data cell currently is `hl7.fhir.extensions.r4:1.0.2 …
   1.0/package.tgz`, prepend a `<td class="confluenceTd">R5
   Extensions:</td>` (i.e., the row should read: *R5 Extensions:* |
   `hl7.fhir.extensions.r4:1.0.2` | `hl7.fhir.extensions.r4:3.0.1` |
   `hl7.fhir.extensions.r4:4.0.1` | `n/a`).
   *Note:* the existing data is for `hl7.fhir.extensions.r4:*` packages,
   not `r5`; that is consistent with the row labelling because
   "R5 Extensions" here means *extensions usable from an R5
   StructureDefinition*, sourced from the published `r4` extension
   packages. If the Work Group intends to repoint these cells at the
   `hl7.fhir.extensions.r5:*` package family, that should be raised as a
   follow-up clarification ticket — the resolution as written only adds
   the missing header, it does not relabel the cells.

2. **Add an R5 column.** In every row (the header row and each
   `R{2,3,4,5}+R6 Extensions` row), add an "R5 Package" column to the
   right of the existing "R4 Package" column, populated with the
   appropriate `hl7.fhir.extensions.r{N}:5.0.0 …
   /5.0/package.tgz` cell (or `n/a` where N == 5).

3. **Add an R6 column.** Add an additional "R6 Package" column to the
   right of the new R5 column, populated with
   `hl7.fhir.extensions.r{N}:6.0.0 … /6.0/package.tgz` cells (or `n/a`
   where N == 6) in each existing row, mirroring the published cross-
   version extensions package family.

4. **Add an "R6 Extensions:" row.** Add a new row below the R5
   Extensions row, headed `R6 Extensions:`, populated with cells for
   `hl7.fhir.extensions.r6:2.0.0`, `:3.0.0`, `:4.0.0`, `:5.0.0`, and
   `n/a` for the R6 column, matching the package URL pattern
   `http://hl7.org/fhir/6.0/extensions/{N}.0/package.tgz`.

5. **Remove (or update) the stale R5 balloter note.** Remove the
   trailing paragraph

   > *"Note for balloters: the R5 packages will be created when R5 is
   > finalized. Until then, these are broken links."*

   It is no longer accurate now that the R5 column is real. If the
   Work Group wants to keep an equivalent caveat for R6, replace the
   removed paragraph with an analogous note scoped to R6 (e.g., "The
   R6 package links will resolve once R6 is published; until then they
   may 404."). Otherwise simply delete the paragraph.

6. **Apply the same column/row additions to every other package table on
   the page.** The resolution explicitly says "all tables on that page"
   — i.e., not just the `hl7.fhir.extensions.r{N}` cross-version
   extensions table but also any sibling tables on the Version
   Management Policy page that follow the same R2/R3/R4 column shape
   (typically the core-package, examples-package, search-params, and
   conformance-package tables).

Example of the corrected R5 row after edit (HTML, Confluence-flavoured):

```html
<tr>
  <td class="confluenceTd">R5 Extensions:</td>
  <td class="confluenceTd">hl7.fhir.extensions.r4:1.0.2 …</td>
  <td class="confluenceTd">hl7.fhir.extensions.r4:3.0.1 …</td>
  <td class="confluenceTd">hl7.fhir.extensions.r4:4.0.1 …</td>
  <td class="confluenceTd">hl7.fhir.extensions.r4:5.0.0 …</td>  <!-- NEW R5 col -->
  <td class="confluenceTd">hl7.fhir.extensions.r4:6.0.0 …</td>  <!-- NEW R6 col -->
</tr>
```

### Design Rationale

- The change is purely editorial / documentation: the underlying package
  ecosystem already supports R5 (final) and is being extended for R6, so
  the policy page is simply catching up to reality.
- Adding both R5 and R6 in one edit (rather than only R5) matches the
  resolution wording verbatim ("all tables on that page to have R6 as
  well as R5") and avoids a second editorial pass when R6 ballots.
- Fixing the missing header in-place (instead of, say, rebuilding the
  table) is the minimum-surface fix consistent with a "Technical
  Correction" issue type and the Persuasive resolution. The resolution
  did not authorize relabelling the existing data cells, so this plan
  preserves the existing `r4:*` package values in the R5 Extensions row
  and flags the relabelling question as an open item.
- Removing the stale balloter note is implied by the resolution: once
  the R5 column is materialized with real package versions, the
  caveat "the R5 packages will be created when R5 is finalized" is both
  false and confusing; leaving it in would contradict the new column.

### Open Questions

1. **Should the R5 Extensions row's *data* cells (currently
   `hl7.fhir.extensions.r4:*`) be relabelled to point at the
   `hl7.fhir.extensions.r5:*` package family?** The resolution does not
   say; it only authorizes adding the missing R5 row header. If the WG
   intended the data to also be repointed, that needs a follow-up
   ticket or an explicit scope clarification before this edit ships.

   This plan assumes the existing pattern
   (`hl7.fhir.extensions.r{N}:6.0.0` against
   `…/6.0/package.tgz`) holds, but the actual published numbers must
   be sourced from `HL7/fhir-extensions` before the cells are filled.
3. **Should a replacement R6 balloter caveat be written?** The
   resolution says "removed or adapted to R6"; the implementer should
   confirm with the WG whether removal alone is acceptable, or whether
   an R6-scoped equivalent must take its place.
4. **Which other tables on the page are in scope?** The resolution
   says "all tables", but the description only quotes the cross-
   version extensions table. The implementer must skim the live
   Confluence page and apply the same column/row expansion to every
   table that has the R2/R3/R4 column shape, then list each one in the
   applied-vote comment for traceability.
5. **Is there a content-exclusion / branding constraint** (e.g., a
   requirement to keep the balloter caveat as long as R6 is in
   ballot)? If so, prefer the "replace with R6 caveat" path over
   outright removal.
