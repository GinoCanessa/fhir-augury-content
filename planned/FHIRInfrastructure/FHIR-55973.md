## Resolution Summary

Resolution: **Persuasive**. The resolution text spells out three concrete changes:

1. Add an invariant to `QuestionnaireResponse.item` with expression
   `answer.value.isDistinct()` and human text
   *"The same answer cannot be specified more than once for the same question"*.
2. In `Questionnaire.item.repeats`, update the comment wording from
   *"If a question is marked as repeats=true, then multiple answers can be
   provided"* to *"If a question is marked as repeats=true, then multiple
   distinct answers can be provided"*, and update the definition from
   *"may include multiple answers"* to *"may include multiple distinct
   answers"*.
3. In the `Questionnaire.item.repeats` comment, promote the sentence
   *"When repeats=true for a group, it'll be represented with multiple items
   with the same linkId in the QuestionnaireResponse. For a question, it'll
   be represented by a single item with that linkId with multiple answers."*
   to be the first paragraph of the comment.

## Feature Proposal

### Problem Statement

When a Questionnaire question has `repeats=true` and an answer list
(`answerOption`/`answerValueSet`), the specification did not explicitly
state whether a respondent can select the same choice more than once. The
Zulip thread `[questionnaire] selecting the same answer more than once`
(stream 179255-questionnaire) surfaced real forms where ambiguity produced
differing UI behaviour. The current `Questionnaire.item.repeats` definition
(`may include multiple answers`) and comment (`multiple answers can be
provided`) are silent on distinctness, and `QuestionnaireResponse.item`
has no invariant preventing duplicate values in its `answer` array —
`qrs-1` forbids co-occurrence of `answer` and `item`, `qrs-2` enforces
uniqueness of child `linkId`, and `qrs-3` constrains `linkId` whitespace,
but none cover duplicate answer values.

### Proposed Change

**Normative rule (QuestionnaireResponse):** Add a new constraint on the
`QuestionnaireResponse.item` element. The next available key in the
existing sequence (`qrs-1`, `qrs-2`, `qrs-3`) is `qrs-4`.

- key: `qrs-4`
- severity: `error`
- human: `The same answer cannot be specified more than once for the same question`
- expression: `answer.value.isDistinct()`
- source: `http://hl7.org/fhir/StructureDefinition/QuestionnaireResponse`

**Prose (Questionnaire.item.repeats):**

- Replace the existing `definition` text
  *"An indication, if true, that a QuestionnaireResponse for this item
  may include multiple answers associated with a single instance of this
  item (for question-type items) or multiple repetitions of the item
  (for group-type items)."*
  with the same sentence but with "multiple answers" replaced by
  "multiple distinct answers":
  *"An indication, if true, that a QuestionnaireResponse for this item
  may include multiple distinct answers associated with a single instance
  of this item (for question-type items) or multiple repetitions of the
  item (for group-type items)."*
- Rewrite the `comment` so it begins with the previously-trailing
  sentence about how group- vs question-repeats are represented in
  QuestionnaireResponse, then carries the remaining existing prose with
  the wording update `multiple answers can be provided` →
  `multiple distinct answers can be provided`. Suggested new comment:

  > When repeats=true for a group, it'll be represented with multiple
  > items with the same linkId in the QuestionnaireResponse. For a
  > question, it'll be represented by a single item with that linkId
  > with multiple answers.
  >
  > If a question is marked as repeats=true, then multiple distinct
  > answers can be provided for the question in the corresponding
  > QuestionnaireResponse. When rendering the questionnaire, it is up
  > to the rendering software whether to render the question text for
  > each answer repetition (i.e. "repeat the question") or to simply
  > allow entry/selection of multiple answers for the question (repeat
  > the answers). Which is most appropriate visually may depend on the
  > type of answer as well as whether there are nested items.
  >
  > The resulting QuestionnaireResponse will be populated the same way
  > regardless of rendering - one 'question' item with multiple answer
  > values.
  >
  > The value may come from the ElementDefinition referred to by
  > .definition.

### Design Rationale

The resolution was marked *Persuasive* and mirrors FHIR's usual pattern
for content-shape rules: a FHIRPath invariant on the resource that
directly states the constraint and lives next to the data it guards.
`answer.value.isDistinct()` evaluates over the repeating `answer` child
of `QuestionnaireResponse.item`, so placing the constraint on
`QuestionnaireResponse.item` (rather than on the collection element
`answer` itself) matches the precedent set by `qrs-1` and `qrs-2`, which
both hang off `QuestionnaireResponse.item` and reason over its children.

`isDistinct()` compares using the FHIRPath `=` operator, which for
`answer.value[x]` choice types will compare by the underlying datatype —
so two `valueCoding` answers with the same `system`+`code` collide,
matching author intent from the Zulip thread.

Updating `Questionnaire.item.repeats` prose in the same ticket keeps
authoring and response rules aligned: the Questionnaire side tells
designers "repeats means *distinct* multiple answers"; the
QuestionnaireResponse side enforces it. Re-ordering the comment so the
group-vs-question representation paragraph leads is purely editorial —
the resolution calls it out explicitly because it is the sentence
readers most often miss.

### Breaking Changes

Yes — this is a backward-incompatible validation change. Existing
QuestionnaireResponse instances that repeat the same answer value for a
single question (e.g., selecting the same `Coding` twice in a
`repeats=true` choice question) validate today and will fail after
`qrs-4` is added. This is an intentional tightening, but it should be
called out explicitly in migration / change-log notes because any
implementer currently relying on duplicate-answer semantics is affected.
No cardinality, type, or binding changes are introduced.

### Related Specifications

- `Questionnaire.item.answerOption`, `Questionnaire.item.answerValueSet`,
  `Questionnaire.item.answerConstraint` — govern the domain of values
  whose distinctness is now enforced; no edits required, but the new
  rule should be consistent with how `answerConstraint` describes
  allowed answer shapes.
- `Questionnaire.item.repeats` and `QuestionnaireResponse.item.answer`
  are the two sides of the same concept; doc and invariant edits land
  in both resources.
- Downstream: Structured Data Capture (SDC) IG — the ticket carries the

### Testing Considerations

- Revalidate `QuestionnaireResponse` against the updated SD with both a
  duplicate-answer instance (must fail `qrs-4`) and a distinct-answer
  instance (must pass).
- Edge cases to verify:
  - Duplicate `valueCoding` with identical `system`+`code` but
    differing `display` — FHIRPath `=` on `Coding` should treat these
    as equal; confirm `qrs-4` fails.
  - Same primitive `valueString` repeated — must fail.
  - Two distinct `valueCoding` entries from the same ValueSet —
    must pass.
  - Item with zero or one answer — trivially passes (isDistinct on an
    empty or singleton collection is true).
  - Nested repeating group items with duplicated answers in sibling
    items — `qrs-4` is per-item, so duplicates in *different* item
    instances must still pass.
- Interop: downstream SDC consumers (label `SDC-CORE`) and form
  rendering engines should re-run their conformance suites; any form
  currently exploiting duplicate-answer semantics needs an errata.
