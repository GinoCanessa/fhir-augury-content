## Resolution Summary

**Resolution:** Persuasive with Modification

**Resolution description (verbatim):** *"We will propagate these search parameters across all resources where the underlying elements exist. (There are a few limited places where canonical elements are missing from certain resources and this was a deliberate choice.)"*

In other words, every resource derived from `CanonicalResource` (and the related metadata-bearing resources called out by the ticket) must define the full CanonicalResource search-parameter family — `context`, `context-quantity`, `context-reference`, `context-type`, `context-type-quantity`, `context-type-value`, `date`, `description`, `jurisdiction`, `name`, `publisher`, `status`, `title`, `url`, `version` — **wherever the underlying canonical element actually exists on the resource**. Where the underlying element is intentionally absent (e.g., `CompartmentDefinition.identifier`, `CompartmentDefinition.jurisdiction`), the search parameter is still **not** added; that omission is deliberate per the resolution.

The audit table the reporter assembled (see ticket description, sourced from the Search Parameter Registry CanonicalResource block at https://build.fhir.org/searchparameter-registry.html#canonicalresource) is the working list of gaps to close. No new template parameters are being defined on `CanonicalResource` itself; the ticket is purely about propagating existing CanonicalResource template parameters into the per-resource `bundle-<Resource>-search-params.xml` files.

## Feature Proposal

### Problem Statement

`CanonicalResource` (`source/canonicalresource/bundle-CanonicalResource-search-params.xml` in `HL7/fhir`) declares a templated set of search parameters — `context`, `context-quantity`, `context-reference`, `context-type`, `context-type-quantity`, `context-type-value`, `date`, `description`, `jurisdiction`, `name`, `publisher`, `status`, `title`, `url`, `version` — using `{{name}}` / `{{title}}` placeholders, and the FHIR documentation describes them as "search parameters defined by this interface for all implementations".

In practice, however, the published Search Parameter Registry (https://build.fhir.org/searchparameter-registry.html#canonicalresource) shows that several resources derived from CanonicalResource do **not** instantiate the full template set in their per-resource search-params bundles. The reporter compiled the following gap matrix (X = missing in that resource):

Inspection of the per-resource bundles in the cache (`source/<resource>/bundle-<Resource>-search-params.xml`) confirms each gap above; e.g., `bundle-SubscriptionTopic-search-params.xml` carries `date`, `identifier`, `status`, `title`, `url`, `version`, `experimental` but none of the `context*`, `description`, `jurisdiction`, or `publisher` parameters. `bundle-DeviceDefinition-search-params.xml` has no `context*`, `date`, `description`, `jurisdiction`, `status`, or `title` parameters. Similar omissions are visible for the other listed resources.

Inspection of the corresponding StructureDefinitions (e.g., `structuredefinition-SubscriptionTopic.xml`, `structuredefinition-DeviceDefinition.xml`, …) confirms that every one of the affected resources **does** carry the underlying canonical elements (`useContext`, `date`, `description`, `jurisdiction`, `publisher`, `status`, `title`, `version`, `name`, `identifier`) in its differential — except for `CompartmentDefinition`, whose differential intentionally omits `identifier` and `jurisdiction`. That single resource is therefore the "deliberate choice" the resolution carves out; everything else is just an authoring oversight.

### Proposed Change

For each (resource, parameter) cell flagged "X" in the table above, **add an entry to that resource's `bundle-<Resource>-search-params.xml`** matching the corresponding template entry in `bundle-CanonicalResource-search-params.xml`, with `{{name}}` substituted for the resource name and `{{title}}` substituted for the resource's display title.

Concretely, each new `<entry>` MUST be modeled on the matching CanonicalResource template entry and have:

- `SearchParameter.id` = `<Resource>-<code>` (e.g., `SubscriptionTopic-context`)
- `SearchParameter.url` = `http://hl7.org/fhir/build/SearchParameter/<Resource>-<code>`
- `SearchParameter.code` = the canonical code (`context`, `context-quantity`, `context-reference`, `context-type`, `context-type-quantity`, `context-type-value`, `date`, `description`, `jurisdiction`, `name`, `publisher`, `status`, `title`, `version`)
- `SearchParameter.type` = same as the CanonicalResource template (`token` for context*/jurisdiction/status, `quantity` for context-quantity, `reference` for context-reference, `composite` for the `context-type-*` composites, `date` for `date`, `string` for `description`/`name`/`publisher`/`title`/`version`, `uri` for `url`)
- `SearchParameter.expression` = the CanonicalResource template expression with `{{name}}` replaced by the resource name (e.g., `(SubscriptionTopic.useContext.value.ofType(CodeableConcept))` for `context`)
- `SearchParameter.description` = the CanonicalResource template description with `{{title}}` replaced by the resource's display title (e.g., "A use context assigned to the subscription topic")
- The same `processingMode` (`normal`) and the same `structuredefinition-standards-status` extension (`normative`) used by the CanonicalResource template
- The build extension `http://hl7.org/fhir/build/StructureDefinition/path` populated with the `{{name}}`-substituted path string (e.g., `SubscriptionTopic.useContext.valueCodeableConcept`), matching the template
- Composite parameters (`context-type-quantity`, `context-type-value`) MUST keep their `<component>` blocks (`context-type` + `context-quantity` / `context-type` + `context`), copied verbatim from the template

**Explicitly excluded from this ticket** (the "deliberate" carve-outs flagged by the resolution):

- `CompartmentDefinition` — do **not** add `identifier` (no `CompartmentDefinition.identifier` element) and do **not** add `jurisdiction` (no `CompartmentDefinition.jurisdiction` element). Both omissions in the registry table reflect missing underlying elements.
- `url` is not missing anywhere per the reporter's own audit, so no work is needed for `url`.

**Example added entry** — `SubscriptionTopic-context` in `bundle-SubscriptionTopic-search-params.xml`, derived from the CanonicalResource template (`bundle-CanonicalResource-search-params.xml`):

```xml
<entry>
  <resource>
    <SearchParameter>
      <id value="SubscriptionTopic-context"/>
      <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
        <valueCode value="normative"/>
      </extension>
      <extension url="http://hl7.org/fhir/build/StructureDefinition/path">
        <valueString value="SubscriptionTopic.useContext.valueCodeableConcept"/>
      </extension>
      <url value="http://hl7.org/fhir/build/SearchParameter/SubscriptionTopic-context"/>
      <description value="A use context assigned to the subscription topic"/>
      <code value="context"/>
      <type value="token"/>
      <expression value="(SubscriptionTopic.useContext.value.ofType(CodeableConcept))"/>
      <processingMode value="normal"/>
    </SearchParameter>
  </resource>
</entry>
```

The same shape applies for every other `(resource, code)` pair in the gap list.

### Design Rationale

- **Consistency with the CanonicalResource interface contract.** The text "search parameters defined by this interface for all implementations" is the design intent; the registry today simply diverges from it. Propagating the template entries restores the documented contract without needing any spec-text change.
- **Minimum-change, mechanical approach.** Each addition is a literal placeholder substitution against the existing CanonicalResource template entries. There are no new parameter codes, no new types, no new bindings, and no new expressions. This minimises both review burden and the risk of introducing a subtly different parameter shape per resource.
- **Respect the deliberate carve-outs.** The resolution is explicit that some omissions are intentional — namely where the underlying element is not present on the resource. The plan above therefore keeps `CompartmentDefinition.identifier` and `CompartmentDefinition.jurisdiction` un-added, since those elements are absent from the differential. No other deliberate carve-out has been identified.
- **Keep `url` untouched.** The reporter's audit confirms `url` is not missing anywhere; adding redundant entries would just create churn.
- **Per-resource bundles, not a single shared SearchParameter resource.** The repo's existing pattern is one SearchParameter resource per (resource, code), authored inside the per-resource bundle. The plan follows that pattern rather than introducing a new sharing mechanism.
