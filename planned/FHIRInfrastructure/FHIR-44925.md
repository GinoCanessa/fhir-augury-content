## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"We will add a section with some examples with an explanation of usage."*

The approved change is to extend the FHIR Trademark section of the license
page with a new sub-section that defines what constitutes a "trademark use"
versus a "non-trademark use" and illustrates the distinction with concrete
examples drawn from the Zulip discussion linked on the ticket. The ticket
specifically asks for:

1. A clear definition of "trademark use" (Grahame's framing) so the existing
   bullet that refers to "non-trademark uses" has an anchor in the text.
2. The point that trademark restrictions apply even to titles of slide
   decks, articles, and similar artifacts.
3. Lloyd's example *"FHIRing on All Cylinders"* as a slide-deck title that
   needs permission.
4. Additional worked examples such as:
   - *"CompanyX's FHIR Products"* as a website title;
   - the hostname `companyx-fhir-products`;
   - *"CompanyX's Products for FHIR"* as a website title,
   identified as trademark uses (or clearly flagged as requiring HL7
   permission).

## Feature Proposal

### Problem Statement

`source/license.html` contains the FHIR Trademark section (anchor
`#trademark`, lines 70–97). The current text:

- Refers to "non-trademark uses of the foregoing marks" (line 84) without
  explaining what makes a use a *trademark* use in the first place, so
  implementers cannot self-classify their use.
- Prohibits use of the marks "in URL domains or to brand your product or
  service" (lines 86–87) without illustrating the edge cases that arise in
  practice (slide-deck titles, blog post titles, hostnames containing the
  word "fhir", phrases like "Products for FHIR" versus "FHIR Products").
- Points readers at the fhir.org community-license and product-license
  forms (line 91) but gives no gut-check examples to help an implementer
  decide whether they need to submit a form before publishing.

The resulting ambiguity has been discussed on Zulip
(`implementers > Trademark question about text to include`, stream 179166)
and has recurred often enough that a documentation fix was approved.

### Proposed Change

Add a new sub-section inside the FHIR Trademark section of
`source/license.html`, immediately after the existing `<ul>` that ends at
line 93 and before the "Trademark FAQs are posted…" paragraph (line 95).
The new content has three parts:

1. **Definition paragraph.** A short paragraph that defines a "trademark
   use" as a use of the word "FHIR" (or the FHIR flame icon) as, or as
   part of, the name or branding of a product, service, event, website,
   domain, or publication. Contrast with a "non-trademark use" (referential
   or nominative — e.g., saying that a product "supports FHIR" or
   "implements the HL7® FHIR® standard"). This paragraph should also call
   out explicitly that titles of slide decks, articles, videos, and
   similar artifacts count as branding for this purpose and therefore fall
   under the trademark-use rules, even when the artifact itself is free to
   download.

2. **Examples table.** A two-column HTML table (`Example` / `Classification`)
   listing concrete examples and whether each is a trademark use that
   requires HL7 written permission via the product-license / event-license
   forms referenced higher up in the section. At minimum include:

   | Example | Classification |
   |---|---|
   | `"FHIRing on All Cylinders"` (slide-deck or presentation title) | Trademark use — requires permission |
   | `"CompanyX's FHIR Products"` as a website title | Trademark use — requires permission |
   | `companyx-fhir-products` as a hostname / domain label | Trademark use — requires permission (URL domain) |
   | `"CompanyX's Products for FHIR"` as a website title | Trademark use — requires permission |
   | "CompanyX's product supports the HL7® FHIR® standard" (marketing copy) | Non-trademark (nominative) use — no permission required, but include the ® and attribution per the bullet above |
   | A blog post titled "Our experience implementing FHIR" | Non-trademark (nominative) use — no permission required |

   The exact classification of the *"Products for FHIR"* form is flagged
   in the ticket as uncertain; the author of the change should confirm
   with the work group / trademark owner before the examples table is
   published (see Open Questions).

3. **Cross-link.** Either reuse the existing product-license link
   (`http://www.fhir.org/product-license`) or add a short reminder bullet
   pointing readers to the forms for cases classified as trademark uses.

No existing text inside the `<ul>` (lines 77–93) should be deleted; the
bullet list's rules remain in force. The change is purely additive prose
that sits between the `<ul>` and the FAQs paragraph.

### Design Rationale

- **Scope is documentation-only, matching the resolution.** "Persuasive"
  with the wording *"We will add a section with some examples…"* rules
  out structural changes (no new pages, no new resources, no StructureDef
  edits). Following the FhirCore "Update narrative / prose" recipe from
  the repo briefing keeps the change where the publisher expects hand-
  edited content.
- **Additive rather than rewriting.** The existing bullet list is
  legally-reviewed language. Adding a definition + examples block below
  the list avoids touching that language while still closing the
  "non-trademark use" gap the ticket calls out.
- **Examples come directly from the ticket.** Using Paul Lynch's
  "CompanyX" examples and Lloyd's "FHIRing on All Cylinders" example
  keeps the documentation aligned with the community consensus that
  produced the Persuasive vote.
- **A table is used instead of prose bullets** so the Classification
  column makes the trademark-use / non-trademark-use split visually
  unambiguous, which is the information gap Grahame's framing was meant
  to fill.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add "Trademark use vs. non-trademark use" definition paragraph.**
   - File: `source/license.html`
   - Action: Immediately after the closing `</ul>` at line 93 and before
     the `<p>Trademark FAQs are posted…</p>` paragraph at line 95, insert
     a new `<a name="trademark-use-vs-non-trademark"></a>` anchor followed
     by an `<h4>` (keep `<h3>` for "FHIR Trademark"; use `<h4>` for the
     sub-section so the section hierarchy stays well-formed) such as
     `<h4>Trademark use vs. non-trademark use</h4>`, then a `<p>` that
     defines the two categories.
   - Details: The paragraph should (a) define *trademark use* as use of
     "FHIR" or the FHIR flame icon as — or as part of — the name or
     branding of a product, service, event, website, domain, publication,
     or other artifact (including titles of slide decks, talks, articles,
     and videos), and (b) define *non-trademark use* as a purely
     referential / nominative use that does not identify the user's own
     offering (e.g., "our product supports the HL7® FHIR® standard").
     Reuse the existing ® / attribution wording style already used on
     lines 81–85 for consistency.

2. **Add examples table.**
   - File: `source/license.html`
   - Action: Directly after the paragraph added in task 1, insert an HTML
     `<table>` with header row `<th>Example</th><th>Classification</th>`
     and one `<tr>` per example from the *Proposed Change → Examples
     table* section above. Use the same plain `<table>` markup style
     already used elsewhere in `license.html` (see the terminologies
     table starting around line 100+ of the same file) to match the
     rendered look.
   - Details: Each *trademark use* row should explicitly say "requires
     written permission via the product-license / community-license
     forms" and link to the two fhir.org permission-request URLs already
     referenced on line 91
     (`http://www.fhir.org/product-license`,
     `http://www.fhir.org/community-license`). *Non-trademark use* rows
     should link back to the bullet on line 84 that specifies the ® +
     attribution text.

3. **Close-out paragraph (optional but recommended).**
   - File: `source/license.html`
   - Action: After the examples table, add a short `<p>` that reminds the
     reader: "If in doubt, contact HL7trademarks@HL7.org before
     publishing" (reusing the mailto link already present on line 96), to
     make the escalation path visible without repeating the FAQs
     paragraph.
   - Details: Keep the existing FAQs paragraph (line 95) unchanged;
     the new paragraph sits above it.

4. **Anchor / inbound-link check.**
   - File: `source/license.html` and the rest of `source/`.
   - Action: Before submitting, run a repo-wide search for any existing
     inbound references to anchors on `license.html` (e.g.,
