## Resolution Summary

**Resolution:** Persuasive — *"Will fix. Typo also present in R4 and should fix there if we ever do another technical correction to that spec (yeah right)."*

The seventh bullet in the "Notes" list on the primitive data types page mis-spells the `dateTime` primitive type as `datetime` (lowercase `t`). The agreed fix is a one-character casing correction on that bullet so the type name matches the canonical FHIR primitive name `dateTime`. The R4 occurrence is intentionally **out of scope** for this ticket (would only be addressed as part of a future R4 technical correction, which is not planned).

## Feature Proposal

### Problem Statement

The FHIR primitive data types page (`source/datatypes.html`) contains a notes section that, in its seventh bullet, refers to the `dateTime` primitive as `datetime`:

> Leap second are allowed in the **datetime**, instant and time types. …

Throughout the rest of the specification the primitive is consistently spelled `dateTime` (capital `T`) — including in the same file at line 175 (`<td>dateTime<a name="dateTime"></a><a name="datetime"></a></td>`). The lowercase form is a plain typo: it does not match the FHIR primitive type name and is mildly confusing for readers cross-referencing the type definition. The ticket reporter (Nick Radov) flagged this and the work group voted **Persuasive** to correct it.

### Proposed Change

Edit the single bullet in `source/datatypes.html` (line 426 in the current clone, HEAD `db79dcd`) so the type name matches the FHIR primitive:

- **Before:**
  > Leap second are allowed in the `datetime`, instant and time types. …
- **After:**
  > Leap second are allowed in the `dateTime`, instant and time types. …

That is the only character to change — replace the lowercase `t` in `datetime` with an uppercase `T` so the bullet reads `dateTime`. No other text in the bullet, no other lines in the file, and no other files in the repository need to be touched for this ticket.

The existing dual anchors on line 175 (`<a name="dateTime"></a><a name="datetime"></a>`) remain untouched: they intentionally support both casings for inbound links, and the resolution does not ask for them to change.

### Design Rationale

- **Consistency with the canonical primitive name.** The FHIR primitive is defined as `dateTime` everywhere else (StructureDefinition file name `datetime.profile.*`, but display name `dateTime` — see line 175 of `source/datatypes.html`). The notes bullet is the outlier.
- **Smallest possible diff.** A "Persuasive" / "Technical Correction" resolution should be applied with the minimum edit that satisfies the ticket. There is no design alternative to weigh — the resolution comment is unambiguous ("Will fix.").
- **Anchors preserved.** Keeping both `<a name="dateTime">` and `<a name="datetime">` anchors avoids breaking any historical inbound links that may use the lowercase fragment, even though the visible prose is fixed to the canonical casing.
- **R4 explicitly deferred.** The resolution comment explicitly notes the same typo exists in R4 but defers fixing it to "another technical correction to that spec (yeah right)" — i.e., not now. The plan therefore does not touch any R4 branch / artifact.
