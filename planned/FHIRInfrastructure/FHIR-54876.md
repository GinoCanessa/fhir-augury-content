## Resolution Summary

Resolution: **Persuasive with Modification**. The committee agreed to clarify the
semantics of the `replaces` element on the **Request** workflow pattern (which
ServiceRequest, MedicationRequest, and the other Request resources track) so
that it explicitly supports prolongation / continuation / supersession use
cases, and so that it is clear that "replaces" does **not** automatically
invalidate the prior order. Four concrete edits were approved:

1. Change the element **short** (currently `Request(s) replaced by this <title>`)
   to: **`Request(s) replaced, superseded, or continued by this <title>`**.
2. Change the element **definition** (currently
   `Completed or terminated request(s) whose function is taken by this new <title>.`)
   to:
   **`Indicates that this <title> should not be fulfilled until the referenced request(s) are completed or discontinued and also that the function of the referenced requests is taken by this new <title>.`**
3. Append to the existing **requirements** text (currently
   `Allows tracing the continuation of a therapy or administrative process instantiated through multiple requests.`) the sentence:
   **`Also ensures that subsequent orders to continue a therapy aren't accidentally fulfilled in parallel - i.e. it creates a pre-condition that the previous order(s) are done before the new one starts.`**
4. Add a new sentence to the element **comments**:
   **`Replacement does not invalidate the prior order. If the intention is to discontinue the prior order and switch to this new order, the status of the prior order must be manually changed - the presence of a replacement order cannot be presumed to automatically change the status of the prior order, though some business flows MAY opt to do this automatically.`**

The `<title>` token is the placeholder used by the Request workflow pattern
(spreadsheet uses `{{title}}`); each derived resource substitutes its own
resource name (e.g., `request`, `service request`, `care plan`).

## Feature Proposal

### Problem Statement

Throughout the workflow specification, `Request.replaces` has an ambiguous
short/definition wording that early implementers have read as "this request
invalidates the previous request." The reporter and the
[`#workflow > Prolongation / refills`](https://chat.fhir.org/#narrow/channel/179196-workflow/topic/Prolongation.20.2F.20refills/with/568472449)
Zulip thread show a real use case — a prescriber writes an initial prescription
plus a continuation/refill at the same time — that the current text does not
describe well, and where the implication of automatic invalidation is wrong.
Nothing about the model needs to change; the prose around it needs to be made
unambiguous and to explicitly cover prolongation / renewal / continuation.

### Proposed Change

Apply the four text edits above to the **Request** workflow pattern element
`Request.replaces`, which is the canonical source for all derived resources.
In the HL7/fhir clone the pattern lives in the bespoke spreadsheet
`source/request/request-spreadsheet.xml` (see lines 1056–1059 for the
short / definition / requirements / comment cells with the `{{title}}`
substitution token). Specifically:

- **Short** cell at `request-spreadsheet.xml:1056` →
  `Request(s) replaced, superseded, or continued by this {{title}}`.
- **Definition** cell at `request-spreadsheet.xml:1057` →
  `Indicates that this {{title}} should not be fulfilled until the referenced request(s) are completed or discontinued and also that the function of the referenced requests is taken by this new {{title}}.`
- **Requirements** cell at `request-spreadsheet.xml:1058` → append
  ` Also ensures that subsequent orders to continue a therapy aren't accidentally fulfilled in parallel - i.e. it creates a pre-condition that the previous order(s) are done before the new one starts.` to the existing text.
- **Comment** cell at `request-spreadsheet.xml:1059` → append a new sentence
  beginning ` Replacement does not invalidate the prior order. …` to the
  existing "The replacement could be because…" text.

For each Request-pattern-following resource whose `*.replaces` element
hand-copies (rather than inherits) the pattern wording, propagate the
equivalent text edits so the published narrative matches the resolved pattern.
From the clone these are:

- `CommunicationRequest.replaces`
  (`source/communicationrequest/structuredefinition-CommunicationRequest.xml` ll. 133–141)
  — currently uses pattern wording verbatim with `request` substituted for
  `<title>`. Update `short`, `definition`, `comment`, `requirements`.
- `RequestOrchestration.replaces`
  (`source/requestorchestration/structuredefinition-RequestOrchestration.xml` ll. 157–164)
  — same as CommunicationRequest. Update `short`, `definition`, `comment`,
  `requirements`.
- `CarePlan.replaces`
  (`source/careplan/structuredefinition-CarePlan.xml` ll. 159–167) — uses the
  pattern shape but with "care plan" wording. Update `short`, `definition`,
  `comment`, `requirements` to mirror the new pattern with `care plan`
  substituted for `<title>` (e.g., short →
  `Care plan(s) replaced, superseded, or continued by this care plan`).
- `ServiceRequest.replaces`
  (`source/servicerequest/structuredefinition-ServiceRequest.xml` ll. 195–230)
  — currently has its own short
  (`What request replaces`) and definition
  (`The request takes the place of the referenced completed or terminated request(s).`),
  and no comment/requirements on this element. Replace short/definition with
  the new pattern wording (substituting `service request` for `<title>`) and
  add the new comment + requirements. The ticket title and use case are
  ServiceRequest-centric, so this resource is the highest-visibility target.
- `DeviceRequest.replaces`
  (`source/devicerequest/structuredefinition-DeviceRequest.xml` ll. 173–191)
  — same shape as ServiceRequest. Update short/definition; add comment +
  requirements.
- `SupplyRequest.replaces`
  (`source/supplyrequest/structuredefinition-SupplyRequest.xml` ll. 220–248)
  — same shape as ServiceRequest. Update short/definition; add comment +
  requirements.
- `Appointment.replaces`
  (`source/appointment/structuredefinition-Appointment.xml` ll. 491–510) —
  has its own definition specific to appointment cancellations
  (`…in cases where there is a cancellation, the details of the cancellation can be found in the cancellationReason property…`).
  Apply the new pattern short
  (`Appointment(s) replaced, superseded, or continued by this appointment`),
  retain the appointment-specific cancellation sentence in the definition
  but prefix it with the new pattern definition wording, and add the new
  comment text.

`MedicationRequest` does not expose a `MedicationRequest.replaces` element;
its `MedicationRequest.priorPrescription` element maps to `Request.replaces`
(see `source/medicationrequest/structuredefinition-MedicationRequest.xml` ll.
166–179). Out of scope for this ticket — the resolution targets `replaces`
specifically; renaming/retitling `priorPrescription` is not part of the agreed
change and should be opened separately if desired. (Noted in Open Questions.)

`MessageDefinition.replaces` (`source/messagedefinition/...`) is a different
concept (replacement of the message definition itself) and does not derive
from the Request pattern — out of scope.

### Design Rationale

- The Request workflow pattern is the single source of truth for `replaces`
  semantics. Editing the pattern keeps all participating resources aligned
  and keeps the publisher's pattern-conformance checks happy.
- Adding "superseded, or continued" to the short clarifies the prolongation/
  renewal use cases the reporter raised without changing cardinality, type,
  reference targets, or invariants — i.e., this is purely a documentation
  change with no breaking impact.
- The new comment sentence explicitly addresses the misreading observed in
  early implementations ("`replaces` invalidates the prior order"). It states
  the rule (status of the prior order is *not* automatically changed) while
  permitting business flows that opt into automatic status changes — so it
  documents existing behaviour rather than introducing new normative
  requirements.
- Updating the per-resource hand-copies in addition to the pattern is
  required because those resources do not regenerate their definitions from
  the pattern at publish time; without the propagation, the published
  ServiceRequest / CommunicationRequest / etc. pages would still show the
  old wording.
