## Resolution Summary

**Resolution:** Persuasive with Modification.

The submitter asked for `Bundle.signature` to be changed from `0..1` to `0..*`. The Work Group accepted the underlying use case (multiple parties signing the same Bundle for different purposes — e.g., Proof of Origin, Proof of Receipt) but declined to alter `Bundle.signature` itself for backward-compatibility reasons. Instead, the approved fix is to define a **new core extension** as follows (verbatim from the resolution description):

- **Name:** `AdditionalSignature`
- **Type:** `Signature`
- **Cardinality:** `0..*`
- **Context:** element `Bundle`
- **Short:** "More signatures"
- **Definition:** "Additional signatures that apply to the Bundle content (i.e. where a single Bundle.signature is not sufficient)"
- **Comment:** "There is no distinction in meaning between the signature repetition that appears in Bundle.signature vs. those that are sent in this extension. Systems and profiles SHALL NOT ascribe different meaning to signatures sent in Bundle.signature vs. here."
- **Requirements:** "This is handled as a separate element rather than increasing the cardinality for Bundle.signature for backward compatibility reasons. If this element becomes widely used, we'll evaluate migrating the extension to Bundle."

No change is made to `Bundle.signature`'s cardinality. No breaking changes to existing Bundle instances.

## Feature Proposal

### Problem Statement

Today `Bundle.signature` has cardinality `0..1`, so a Bundle can carry at most one signature. Real-world document-exchange scenarios (discussed on the [#implementers > Signatures in Bundle for multiple purposes](https://chat.fhir.org/#narrow/stream/179166-implementers/topic/Signatures.20in.20Bundle.20for.20multiple.20purposes) Zulip thread and in the ticket comments by John Moehrke and Corey Spears) require multiple distinct signatures on the same Bundle — e.g., Proof of Origin from the source system and Proof of Receipt from the receiver — each produced by a different party and/or covering a different purpose. The current cardinality forces implementers either to choose one signature, nest Bundles, or reach for non-standard extensions.

### Proposed Change

Define a new standard extension in HL7 FHIR core that allows additional signatures to be attached to a Bundle:

- **Canonical URL:** `http://hl7.org/fhir/StructureDefinition/bundle-AdditionalSignature` (canonical to be confirmed during authoring; follows the existing core-extension naming pattern used by e.g. `operationoutcome-issue-source`).
- **id / name:** `id` = `bundle-AdditionalSignature`; `name` = `AdditionalSignature`; `title` = `Additional Signature`.
- **kind:** `complex-type`; **type:** `Extension`; **baseDefinition:** `http://hl7.org/fhir/StructureDefinition/Extension`; **derivation:** `constraint`; **abstract:** `false`.
- **Context:** `type=element`, `expression=Bundle`.
- **Differential elements:**
  - `Extension` — `short` = "More signatures"; `definition` = (text above); `comment` = (text above); `requirements` = (text above); `min=0`, `max=*`.
  - `Extension.extension` — `max=0` (simple extension, no sub-extensions).
  - `Extension.url` — `fixedUri` = the canonical URL above.
  - `Extension.value[x]` — `min=1`; single `type.code` = `Signature`.
- **Standards-status / FMM:** inherit Bundle's `normative` track; FMM as agreed by the WG (recommend `0` / `trial-use` initially because new, noting the resolution's "If this element becomes widely used, we'll evaluate migrating the extension to Bundle" language).
- **Narrative update:** extend the Signatures section of the Bundle page to describe the new extension, reference the identical-meaning clause, and link to the extension page.
- **Example:** add or update a Bundle example that carries both `Bundle.signature` and one `AdditionalSignature` extension to demonstrate multi-party signing.

### Design Rationale

- Adding a sibling extension instead of widening `Bundle.signature` to `0..*` is explicitly called out in the resolution as a **backward-compatibility** measure: existing consumers that assume at most one signature on Bundle continue to work.
- Making the extension's `Extension.value[x]` a single `Signature` and placing it at `0..*` on Bundle reproduces the desired `0..*` cardinality for additional signatures without changing the core Bundle element.
- The Comment clause (no distinction in meaning vs. `Bundle.signature`) prevents downstream profiles from inventing semantic splits that would fragment interoperability.
- The Requirements clause leaves the door open for a future core-element migration if adoption warrants, matching the resolution text.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Home of the Bundle resource definition and all hand-authored core extensions. The new `AdditionalSignature` extension, per the resolution's precedent and existing authoring patterns, is a core extension authored alongside Bundle.
- **Affected Files:**
  - `source/bundle/structuredefinition-extension-bundle-AdditionalSignature.xml` — **new**: the StructureDefinition for the extension (differential only; publisher generates the snapshot).
  - `source/bundle/bundle-notes.xml` — extend the "Signatures" section (currently lines 381–392) with a paragraph introducing the new extension, pointing at its page, and restating the no-semantic-difference clause from the Comment.
  - `source/bundle/bundle-example.xml` (or a new sibling example file) — optional but recommended: demonstrate a Bundle carrying both `Bundle.signature` and one `AdditionalSignature` extension.
- **Change Scope:** Minor (one new extension file, one narrative paragraph, optional example).

### Breaking Changes

No breaking changes identified. `Bundle.signature` is left at `0..1`; the new `AdditionalSignature` extension is `0..*` and optional, so all existing Bundle instances remain valid. The resolution's choice of an extension over a cardinality change is explicitly motivated by backward compatibility.

### Related Specifications

- **Datatypes:** `Signature` (`datatypes.html#Signature`) — the value type carried by the extension. No change required, but the Bundle notes narrative already links there and should continue to.
- **Profile pages for Bundle sub-types** (`structuredefinition-profile-{document,message,transaction,history,search-set,batch,batch-response,subscription-notification,transaction-response}-bundle.xml`): inherit context `Bundle`, so the extension applies automatically; no profile edits required unless a WG chooses to constrain it.
- **Provenance**: `Bundle.signature`'s definition (line 666) already points implementers to Provenance-based signatures; the new extension should similarly not compete with that guidance — the narrative edit should keep that pointer.

### Related Tickets

- No direct Jira cross-references surfaced by `cross-referenced FHIR-42961`. Only a Zulip thread and a resource-path reference (`Bundle.signature`) came back. If a conflicting or dependent ticket exists, it was not captured in the cross-reference index at the time of planning.

### Terminology Impact

No terminology impact. The extension carries a `Signature` value; no new CodeSystem or ValueSet is introduced or bound. The existing `Signature.type` binding is reused unchanged.

  - A Bundle with only `AdditionalSignature` (no `Bundle.signature`) — must validate.

  - An `AdditionalSignature` whose `value[x]` is absent — must fail validation (because `Extension.value[x]` `min=1`).

### Open Questions

- **Home repository for the extension.** The resolution text does not explicitly pin the extension to FHIR core vs. `HL7/fhir-extensions`. Authoring in core (under `source/bundle/`) matches the pattern of existing core extensions reviewed in the clone and the briefing's characterization of the core repo. If FMG/WG prefers `HL7/fhir-extensions`, that repo must be added to scope and `repo-analysis` run for it before implementation — the briefing explicitly flags `HL7/fhir-extensions` as the general FhirCore extension touch-point and does not supply a current briefing for it.
- **FMM and standards-status.** Not specified by the resolution. Recommend `fmm=0` / `standards-status=trial-use`; confirm with FHIR-I before submission.
- **Canonical URL form.** `bundle-AdditionalSignature` vs. `bundle-additional-signature`. Existing core-extension examples use lower-case-hyphen naming (e.g., `operationoutcome-issue-source`); however the resolution uses the pascal-case name `AdditionalSignature` for the extension's `name`. A final decision on URL casing should be taken during authoring to match current FHIR Infrastructure convention.
- **OID allocation.** A new OID from the FHIR core series is required for the `<identifier>` element.
- **SearchParameter / summary flag.** The resolution does not state whether the extension should participate in the Bundle summary or be searchable. Current core extensions (see the operationoutcome examples) do not declare SearchParameters by default; confirm with the WG.
