## Resolution Summary

**Resolution:** Persuasive with Modification

**Resolution description (verbatim):** *"We will migrate as requested, but will also migrate (not remove) `all-distance-units` and `all-time-units`, as they are used as max bindings."*

The ticket originally asked to:

1. Migrate `ucum-common` and `ucum-units` from FHIR Core to THO (UTG) — already tracked by UP-481.
2. **Remove** `all-distance-units` and `all-time-units` from FHIR Core without a THO replacement.

The committee, on the FHIR-I call of 2025-03-10 (motion Grahame Grieve / Corey Spears 12-0-0; vote also recorded), reverted the prior 2024-03-25 "Persuasive" disposition and adopted the modification: **all four ValueSets are migrated to THO**; none of them are merely deleted. The two `all-*-units` ValueSets must be retained because the datatype spreadsheets bind them as max value sets on `Distance.code`, `Duration.code`, and `Age.code`.

For HL7/fhir master this means **deleting the four ValueSet source files from the core spec** and **letting the canonical URLs be served by THO** (publication of the matching THO content is the responsibility of UP-481 / the UTG release that lands `ucum-common`, `ucum-units`, `all-distance-units`, `all-time-units`). All in-spec bindings whose `valueSet` reference is the unchanged canonical URL (`http://hl7.org/fhir/ValueSet/ucum-*`, `http://hl7.org/fhir/ValueSet/all-*-units`) continue to resolve once THO publishes them; the publisher registration and OID listings local to core are removed.

## Feature Proposal

### Problem Statement

FHIR Core currently authors and publishes four UCUM-related ValueSets:

- `source/valueset/valueset-ucum-common.xml` — `http://hl7.org/fhir/ValueSet/ucum-common` (Common UCUM Units, hand-curated concept list).
- `source/datatypes/valueset-ucum-units.xml` — `http://hl7.org/fhir/ValueSet/ucum-units` (entire UCUM code system, intended to become immutable).
- `source/datatypes/valueset-all-distance-units.xml` — filter `canonical = m` against UCUM (used as a max binding by `Distance.code`).
- `source/datatypes/valueset-all-time-units.xml` — filter `canonical = a` against UCUM (used as a max binding by `Duration.code` and `Age.code`).

These value sets are general terminology assets, not FHIR-specific bindings. The January 2023 WGM agreed they belong in THO so they can be released independently of the FHIR Core release cycle and shared with non-FHIR HL7 products. UP-481 already tracks the THO side. FHIR-44735 is the matching FHIR Core ticket and, after the 2025-03-10 reconsideration, requires that **all four** ValueSets be removed from the core spec source (with THO assuming maintenance and publication under the same canonical URLs).

### Proposed Change

In **HL7/fhir master**:

1. **Delete the four authoring ValueSet files** so the FHIR Core publisher no longer emits them:
   - `source/valueset/valueset-ucum-common.xml`
   - `source/datatypes/valueset-ucum-units.xml`
   - `source/datatypes/valueset-all-distance-units.xml`
   - `source/datatypes/valueset-all-time-units.xml`

2. **Remove the publisher registrations** for the two `all-*-units` ValueSets in `source/fhir.ini` (lines 1289–1290 in the current clone, inside `[valuesets]`):
   - `all-time-units=datatypes\valueset-all-time-units.xml`
   - `all-distance-units=datatypes\valueset-all-distance-units.xml`

   `ucum-common` and `ucum-units` are not currently registered in `[valuesets]`; no `fhir.ini` change is required for them.

3. **Remove the OID assignments** in `source/oids.ini` so core no longer claims the OIDs (THO will continue to assert them):
   - `http://hl7.org/fhir/ValueSet/all-distance-units = 2.16.840.1.113883.4.642.3.3303` (line 113)
   - `http://hl7.org/fhir/ValueSet/all-time-units = 2.16.840.1.113883.4.642.3.3304` (line 116)
   - `http://hl7.org/fhir/ValueSet/ucum-common = 2.16.840.1.113883.4.642.3.3144` (line 940)
   - `http://hl7.org/fhir/ValueSet/ucum-units = 2.16.840.1.113883.4.642.3.912` (line 941)

4. **Remove the example list entry** in `source/valueset/list-ValueSet-examples.xml` (lines 71–77 in the current clone) that lists `valueset-ucum-common` / `ValueSet/ucum-common`, since the local page disappears.

5. **Leave canonical-URL bindings unchanged** in StructureDefinitions:
   - `source/devicemetric/structuredefinition-DeviceMetric.xml` line 283 — `<valueSet value="http://hl7.org/fhir/ValueSet/ucum-units" />`.
   - `source/observationdefinition/structuredefinition-ObservationDefinition.xml` lines 788 and 1092 — `<valueSet value="http://hl7.org/fhir/ValueSet/ucum-units"/>`.

   The canonical URL is preserved across the migration (THO publishes the same `http://hl7.org/fhir/ValueSet/ucum-*` and `http://hl7.org/fhir/ValueSet/all-*-units` URLs); the binding text does not change. Confirm the publisher resolves the cross-version reference once THO is the source of truth — see Open Questions.

6. **Datatype spreadsheet bindings (Quantity / Distance / Duration / Age / SampledData and the *-extensions spreadsheets):** the spreadsheets reference the ValueSets by *page name* (`valueset-ucum-units`, `valueset-all-distance-units`, `valueset-all-time-units`) in the `value set` / max-binding cells. Because these become non-local pages, the publisher must either (a) be taught to resolve them to THO-hosted pages or (b) the spreadsheet cells must be updated to use full canonical URLs / external page references. The decision is editorial and should be confirmed with the publisher maintainers — see Open Questions and the matching task in the Implementation Plan.

7. **Cross-spec note in `source/datatypes.html` / Quantity narrative:** none of those pages currently link to the four ValueSets by anchor, but a search for `valueset-ucum-units.html`, `valueset-ucum-common.html`, `valueset-all-distance-units.html`, `valueset-all-time-units.html` should be re-run before the PR lands so any prose link is repointed at the THO-hosted page.

No schema, JSON definitions, search parameters, profiles, or invariants are added or changed. No element cardinality or type changes. No new extensions.

### Design Rationale

- **Honour the FHIR-I + UTG agreement.** January 2023 WGM agreed that UCUM-derived ValueSets are terminology assets, not FHIR-specific bindings. THO already exists to own this kind of content with an independent release cadence (UP-481 carries the THO side).
- **Preserve the canonical URLs.** When a ValueSet migrates between HL7 publication channels, retaining the canonical URL avoids breaking every existing binding — both inside FHIR Core (DeviceMetric, ObservationDefinition, Distance/Duration/Age/SampledData) and in downstream IGs that reference these ValueSets. This is the standard HL7 migration pattern.
- **Migrate, do not delete, the `all-*-units` ValueSets.** The 2025-03-10 reversal recognised that `Distance.code` / `Duration.code` / `Age.code` use these ValueSets as max bindings; deleting them would silently demote the maximum-allowed-units constraint on those datatypes. Migration preserves the constraint without keeping authoring duplication in core.
- **Make `ucum-units` immutable in THO.** The original ticket text records that `ucum-units` is "all UCUM codes" and should be marked immutable. Implementing immutability is part of the THO authoring task (UP-481), not this ticket; FHIR Core just stops shipping its own copy.
- **Alternative considered: keep the ValueSets in core but mark them informative.** Rejected at the WGM and again on 2025-03-10 — the WG explicitly committed to migration so external products can consume them without depending on a FHIR Core release.
- **Alternative considered (the original 2024-03-25 disposition): delete `all-distance-units` and `all-time-units` outright without THO replacements.** Reverted because of the max-binding usage in datatypes; would have been a silent loosening of the spec.

### Breaking Changes

- **Conformance impact:** None for consumers — the canonical URLs are retained, all four ValueSets continue to exist (in THO), and the binding strengths / contents are unchanged. Existing instances and downstream IG bindings remain valid.
- **Tooling impact:** Any tool that previously fetched these ValueSets from the FHIR Core publication (e.g., expanded into `valueset-ucum-units.json` under the FHIR `R6`/`build` package) must now resolve them from THO. This is a behaviour change for offline tooling that pre-loads only the FHIR Core package. Call this out in the FHIR Core release notes.
- **Internal publisher impact:** Until THO is publishing all four ValueSets, the FHIR Core build will show unresolved-binding warnings on `Distance.code`, `Duration.code`, `Age.code`, `DeviceMetric.unit`, `ObservationDefinition.permittedUnit`, and `ObservationDefinition.qualifiedValue.permittedUnit`. This is a hard sequencing requirement (see Prerequisites).

No element-level breaking changes (no required→optional, no removed elements, no type changes, no binding-strength changes, no search-parameter changes, no invariant changes).
