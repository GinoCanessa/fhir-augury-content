## Resolution Summary

Resolution: **Persuasive with Modification** (FHIR Infrastructure, applied 2025-09-22).

The current short definition of the `Request.insurance` element (and, by extension, the `request-insurance` extension that surfaces it on resources that follow the Request pattern but do not have a native `insurance` element — e.g., `NutritionOrder`, `CarePlan`) reads:

> Insurance plans, coverage extensions, pre-authorizations and/or pre-determinations that may be relevant in delivering the requested service.

It is to be replaced with the two-paragraph wording agreed by Financial Management and re-affirmed by FHIR Infrastructure (which owns the extension via the Request pattern):

> The relevant insurance Coverage for this request to perform Eligibility checking, authorization, pre-determination or claiming for goods and/or services renderable or rendered, etc..
>
> Responses from insurers (ClaimResponse) containing responses to prior authorization or pre-determination requests to guide in the choosing of goods and services informed by reimbursement guidance, and to provide authorization numbers which may need to be quoted on future claims for reimbursement.

No structural change (cardinality, type, binding, or context) is required — only the human-readable `definition` text on the pattern element, the per-resource `definition` overrides where the pattern text was copied, and the corresponding entries in the per-resource `*-request-mapping-exceptions.xml` files that record the pattern-vs-resource definitions.

## Feature Proposal

### Problem Statement

Cooper Thompson (Epic) raised in 2021 that the existing one-line definition of the `request-insurance` extension is ambiguous: it does not say whether the referenced `Coverage` is expected to be partially populated and validated at check-in, fully populated and validated, informational only for triggering prior-auth workflows, or scoped exclusively to prior-auth use. The same ambiguity applies wherever the Request pattern's `insurance` element appears as a native element (DeviceRequest, MedicationRequest, ServiceRequest, Task, Transport, EnrollmentRequest) and wherever it appears as the `request-insurance` extension on resources that follow the Request pattern but omit the element natively (NutritionOrder, CarePlan, …).

The ticket bounced through Orders & Observations (NutritionOrder), then Financial Management (FM), and finally back to FHIR Infrastructure (FHIR-I) because the extension is owned by the Request pattern, not by any one resource. FM (pknapp, 2025-01-30) reviewed the extension as "reasonable for the use case and complete and appropriate in its structure", and proposed clarified prose covering both Coverage uses (eligibility/authorization/claim inputs) and `ClaimResponse` uses (authorization numbers, prior-auth/predetermination responses). lconstab (2025-09-10/11) moved the ticket to FHIR-I for application against the Request pattern.

### Proposed Change

1. **Update the Request pattern element `Request.insurance` `definition` value** in the spreadsheet that drives the pattern, replacing the single-sentence text with the two-paragraph wording from the resolution.
2. **Update each Request-pattern-derived resource StructureDefinition** that copied the pattern-style definition verbatim, replacing the per-resource `definition` text with the new wording (preserving any resource-specific tail clause where it is meaningful, e.g. "to the Task", "to the Transport", "for the requested service" — see Implementation Plan for the per-resource recommendation).
3. **Update the `*-request-mapping-exceptions.xml` files** for each resource where the divergence row records the old pattern text, so the `<_pattern>` value reflects the new pattern wording. Where the resource's own `<resource>` value continues to differ, leave that override in place (and refresh it only where the resolution wording supersedes it).
4. **No change is required to the `request-insurance` extension's URL, context, type, or cardinality** — the extension already targets `Reference(Coverage|ClaimResponse)`, `0..*`, and is contextualized to the Request pattern. The extension's own `definition` is generated from the pattern row, so updating the pattern propagates to the published extension automatically.
5. **No examples, search parameters, ValueSets, CodeSystems, or invariants need to change.**

### Design Rationale

- The Request pattern is the single point of truth for `Request.insurance`; FM and FHIR-I have agreed the wording, and applying it at the pattern level (rather than re-stating it per resource) keeps the eight downstream resource definitions consistent.
- Two paragraphs (Coverage usage; ClaimResponse usage) match the structure of the resolution and mirror how the existing `Reference(Coverage|ClaimResponse)` type is used in practice.
- The mapping-exceptions infrastructure already exists precisely to record where individual Request-pattern resources legitimately diverge from the pattern wording (e.g., Task uses "relevant to the Task"); we keep that mechanism and only realign the `_pattern` side and any per-resource overrides whose only point of departure was the original ambiguous wording.
- Alternatives considered (per the comments): (a) move ownership to FM — rejected because the extension belongs to the Request pattern owned by FHIR-I; (b) restrict the extension's reference target to `Coverage` only — not adopted because FM explicitly intends the `ClaimResponse` target to convey prior-auth responses; (c) introduce an `Account` reference container for ordered coverages (cooper.thompson, 2022-09-20) — out of scope for this ticket.

### Open Questions

1. **Resource-specific tail clauses.** For DeviceRequest, MedicationRequest, ServiceRequest, Task, and Transport, the current per-resource definitions append a resource-specific tail ("for delivering the requested device", "for delivering the requested service", "to the Task", "to the Transport"). The resolution does not state whether these should be (a) replaced wholesale by the pattern wording (cleanest, removes most divergent rows from the mapping-exceptions files) or (b) preserved as a parenthetical addition to the first paragraph (keeps resource-specific framing). The plan recommends (a) for DeviceRequest, MedicationRequest, ServiceRequest (the "for delivering …" tails are mechanical wording artefacts of the original ambiguous text) and (b) for Task and Transport (where "relevant to the Task/Transport" is a substantive narrowing since those resources span more than a single fulfilment). FHIR-I should confirm before implementation.
2. **Spreadsheet vs. derived `request-insurance` StructureDefinition.** Confirm with the publisher maintainers that editing the `Request.insurance` row in `source/request/request-spreadsheet.xml` is sufficient to regenerate the published `http://hl7.org/fhir/StructureDefinition/request-insurance` extension definition (i.e., that no committed snapshot of the extension StructureDefinition lives elsewhere and needs a parallel edit). The grep showed no hand-authored `structuredefinition-request-insurance.*` file under `source/`, which is consistent with full regeneration, but a maintainer sanity-check is prudent.
