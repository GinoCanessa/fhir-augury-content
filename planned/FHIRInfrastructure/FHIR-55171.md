## Resolution Summary

Resolution type: **Persuasive**.

Resolution description (verbatim from the ticket):

> Will move the code system and value set to THO. As part of the move, we'll change the code system URL to be something sensible.

Concretely this means:

1. Migrate `CodeSystem` `list-item-flag` (logical name `PatientMedicineChangeTypes`,
   currently published from `HL7/fhir` at the OID URL `urn:oid:1.2.36.1.2001.1001.101.104.16592`)
   into the Terminology Harmonization (THO / UTG) source of truth.
2. As part of that move, replace the OID-style `CodeSystem.url` with a sensible
   THO-style canonical URL (i.e., a `http://terminology.hl7.org/CodeSystem/...`
   URL) so the code system is consumable like every other THO code system.
3. Migrate the matching `ValueSet` `list-item-flag` to THO as well, so its
   `compose.include.system` reference can move from the OID to the new
   THO URL alongside the CodeSystem.
4. Re-point the existing `List.entry.flag` binding in the FHIR Core `List`
   StructureDefinition at the new THO ValueSet canonical, and remove the
   in-core copies of the code system and value set.

Binding strength on `List.entry.flag` is **example** (verified in the clone),
so the binding update is non-breaking for conformant systems but is still
required for the references to resolve cleanly post-move.

## Feature Proposal

### Problem Statement

`List` references a small family of value sets that are good migration
candidates for the Terminology Harmonization (THO / UTG) repository. The
specific candidate flagged by the ticket is the `list-item-flag`
(PatientMedicineChangeTypes) code system, used by the `list-item-flag`
ValueSet bound (example strength) to `List.entry.flag`.

Two concrete problems with the present in-core definitions:

- The CodeSystem is published from `HL7/fhir` (in `source/list/`), but it is
  a piece of clinical-list terminology that is realistically shareable with
  CDA, V2, and other specifications. Keeping it inside the FHIR Core repo
  makes cross-spec reuse and lifecycle management awkward.
- The CodeSystem's canonical URL is an OID (`urn:oid:1.2.36.1.2001.1001.101.104.16592`),
  which is unlike every other modern HL7 code system and difficult to
  recognise / reference. The resolution explicitly calls out that the URL
  should change "to be something sensible" as part of the move.

### Proposed Change

Two-repo change:

**A. HL7/UTG (Terminology Harmonization)** — add the migrated artifacts:

- New file `input/sourceOfTruth/unified/codeSystems/CodeSystem-list-item-flag.xml`
  containing a CodeSystem with:
  - `id` = `list-item-flag`
  - `url` = **`http://terminology.hl7.org/CodeSystem/list-item-flag`**
    (sensible THO-style URL replacing the legacy OID; the legacy OID
    `urn:oid:1.2.36.1.2001.1001.101.104.16592` should be retained as an
    `identifier` so existing references continue to resolve)
  - `name` = `PatientMedicineChangeTypes`, `title` = `Patient Medicine Change Types`
  - `status` = `active`, `experimental` = `false`, `caseSensitive` = `true`
  - `content` = `complete`
  - `valueSet` pointer = `http://terminology.hl7.org/ValueSet/list-item-flag`
  - The same six concepts currently in the HL7/fhir copy: `01 Unchanged`,
    `02 Changed`, `03 Cancelled`, `04 Prescribed`, `05 Ceased`,
    `06 Suspended`, with the existing `definition` text preserved verbatim.
  - Existing NEHTA `copyright` block preserved.
  - Standards-status / wg metadata aligned with UTG conventions
    (responsible WG: `fhir`).

- New file `input/sourceOfTruth/unified/valueSets/ValueSet-list-item-flag.xml`
  containing a ValueSet with:
  - `id` = `list-item-flag`
  - `url` = **`http://terminology.hl7.org/ValueSet/list-item-flag`**
    (the legacy `http://hl7.org/fhir/ValueSet/list-item-flag` URL plus
    the OID `urn:oid:2.16.840.1.113883.4.642.3.320` retained as
    `identifier` entries so existing references are not orphaned)
  - `name` / `title` / `status` / `publisher` / `contact` / `description`
    carried across from the HL7/fhir copy
  - `compose.include.system` updated to **`http://terminology.hl7.org/CodeSystem/list-item-flag`**
    (replacing the OID system reference)

**B. HL7/fhir (FHIR Core)** — remove the in-core copies and re-point the
binding:

- Delete `source/list/codesystem-list-item-flag.xml`.
- Delete `source/list/valueset-list-item-flag.xml`.
- Edit `source/list/structuredefinition-List.xml`, the `List.entry.flag`
  element binding (currently around lines 407–425): change
  `<valueSet value="http://hl7.org/fhir/ValueSet/list-item-flag"/>` to
  `<valueSet value="http://terminology.hl7.org/ValueSet/list-item-flag"/>`.
  Leave `binding.strength` = `example` and the `bindingName` extension
  (`ListItemFlag`) unchanged.
- No other `source/list/*.xml` files reference `list-item-flag`
  (verified by repo-wide search; only the three files above match).

### Design Rationale

- **Persuasive resolution.** The disposition is `Persuasive`, so the
  approved direction is to perform the migration rather than re-debate
  whether to migrate.
- **THO is the right home.** The CodeSystem is a clinical-list concept
  vocabulary, plausibly reusable in CDA/V2 — exactly the criterion the
  reporter calls out for THO migration.
- **Sensible URL.** Replacing the OID URL with
  `http://terminology.hl7.org/CodeSystem/list-item-flag` matches the
  established THO naming pattern used by every other CodeSystem in
  `input/sourceOfTruth/unified/codeSystems/`. The OID is preserved as
  an `identifier` so prior references remain resolvable.
- **Example binding kept.** The current binding strength on
  `List.entry.flag` is `example`. Changing strength is out of scope for
  this ticket (and the resolution does not ask for it), so only the
  `valueSet` canonical changes.
- **Single source of truth.** Removing the HL7/fhir copies is essential
  to avoid two divergent definitions of the same code system.

### Breaking Changes

- **Canonical URL change for the CodeSystem.** The CodeSystem moves from
  `urn:oid:1.2.36.1.2001.1001.101.104.16592` to
  `http://terminology.hl7.org/CodeSystem/list-item-flag`. Any client
  that resolves the CodeSystem by its old OID URL will need to follow
  the new canonical. This is mitigated by carrying the OID forward as
  an `identifier`, but consumers that key on `url` rather than
  `identifier` will see a breaking change and must be updated.
- **Canonical URL change for the ValueSet.** Likewise, the ValueSet
  canonical moves from `http://hl7.org/fhir/ValueSet/list-item-flag` to
  `http://terminology.hl7.org/ValueSet/list-item-flag`. The
  `List.entry.flag` binding is updated in lockstep, but external
  conformance resources / IG profiles that bind to the old ValueSet
  URL will need to be re-pointed (or rely on the preserved
  `identifier` for resolution).
- The `List.entry.flag` binding strength remains `example`, so this
  is **not** a breaking change to conformance for the resource itself.

### Related Specifications

- `List` (FHIR Core) — only resource binding affected.
- Any downstream IG, extensions pack, or profile that binds to
  `http://hl7.org/fhir/ValueSet/list-item-flag` would benefit from

### Terminology Impact

This change is **entirely a terminology migration** — moving one
CodeSystem and one ValueSet from `HL7/fhir` to `HL7/UTG` and renaming
the CodeSystem URL. Specifically:

- CodeSystem `http://terminology.hl7.org/CodeSystem/list-item-flag`
  (new canonical) replacing `urn:oid:1.2.36.1.2001.1001.101.104.16592`.
- ValueSet `http://terminology.hl7.org/ValueSet/list-item-flag`
  (new canonical) replacing
  `http://hl7.org/fhir/ValueSet/list-item-flag`.
- Both old URLs / OIDs preserved as `identifier` entries on the
  migrated artifacts so existing references can still be resolved.

3. The migration must land in UTG **before** the HL7/fhir deletions

### Open Questions

1. **Exact THO canonical URL.** The resolution says "something
   sensible". This plan assumes
   `http://terminology.hl7.org/CodeSystem/list-item-flag` and
   `http://terminology.hl7.org/ValueSet/list-item-flag`, matching
   neighbouring entries in `HL7/UTG/input/sourceOfTruth/unified/`,
   but the FHIR-I work group should confirm before merge. Renaming
   `list-item-flag` itself (e.g., to `patient-medicine-change-types`
   to match the `name`/`title`) is a possibility worth raising with
   the WG since the URL is changing anyway.
2. **Missing `HL7/UTG` repo briefing.** Per the `ticket-plan` skill
   rules, this plan should normally halt and ask for
   `repo-analysis HL7/UTG` before producing UTG-side tasks. The plan
   above proceeds using clone-observed conventions for UTG (file
   layout under `input/sourceOfTruth/unified/codeSystems/` and
   `.../valueSets/`) but every UTG-side path and the registration
   step (Task 3) **must be re-validated against an up-to-date
   `HL7/UTG` briefing** before implementation begins.
3. **Standards status / experimental flag.** The current ValueSet has
   `experimental="true"` while the CodeSystem has `experimental="false"`.
   The migration is a good opportunity to align these; the plan
   carries the existing values over verbatim by default but the WG
   should decide whether to change them as part of the move.
4. **Whether to also migrate the other in-core list-related
   ValueSets.** The ticket explicitly scopes only `list-item-flag`,
   but `source/list/` contains several other value sets
   (`list-empty-reason`, `list-example-codes`, `list-listpurposecodes`,
   `list-mode`, `list-order`, `list-status`). These are out of scope
   for FHIR-55171 and should not be folded in silently.
