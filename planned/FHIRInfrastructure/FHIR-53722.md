## Resolution Summary

The ticket requests that elements already flagged as deprecated in the FHIR R6 ballot be withdrawn (removed) where appropriate, since keeping deprecated elements in a new major version intended for normativity is unnecessary. The resolution is **Persuasive with Modification** — the committee agreed that many deprecated elements should be removed, but individual decisions are needed per element.

**FHIR Infrastructure owns the resolution for four elements:**

1. **`Bundle.signature`** — **Remove the element.** Was Trial Use since R4; should not remain in R6.
2. **`StructureDefinition.keyword`** — **Keep but ensure deprecated flag renders properly.** This element is already normative and cannot be removed; it will remain flagged as deprecated.
3. **`MetadataResource.topic`** — **Remove the element from the abstract resource.** Keep the `topic` search parameter but update its expression to search on `useContext` with code `topic`.
4. **`OperationOutcome.issue.location`** — **Remove the element.** Deprecated since R4.

**Guidance to other work groups** (via 17 sub-tickets):
- If the element has the deprecated flag since R4, remove it.
- If the element has a deprecated comment and is trial-use, or the resource was TU in R4, or the element was added in R5+, remove it.
- If the element has a deprecated comment but no flag and doesn't meet removal criteria, add the deprecated flag.

---

## Feature Proposal

### What Changes

Remove three deprecated elements entirely from the FHIR R6 specification and fix the deprecation rendering for one normative element. Update the `topic` search parameter on `MetadataResource` to point at `useContext` instead of the removed `.topic` element.

### Why

R6 is intended as a normative release. Deprecated elements that have been deprecated since R4 (two major versions ago) should not persist — they add unnecessary complexity to FHIR models and implementer tooling. Removing them now, during the major version boundary, is the correct time per [FHIR deprecation policy](http://hl7.org/fhir/versions.html#deprecation).

### Deprecation Rendering Issue

The ticket notes that the deprecation flag "doesn't seem to be rendered into the R6-ballot4 StructureDefinitions." This is likely a build tooling issue. The resolution calls for investigating why. The XML structures for deprecated status use two nesting patterns:

**Pattern A** (Bundle.signature, MetadataResource.topic — `valueCode` as direct child):
```xml
<extension url="...structuredefinition-standards-status">
  <extension url="...structuredefinition-standards-status-reason">
    <valueString value="reason"/>
  </extension>
  <valueCode value="deprecated"/>
</extension>
```

**Pattern B** (OperationOutcome.issue.location, StructureDefinition.keyword — reason nested inside `valueCode`):
```xml
<extension url="...structuredefinition-standards-status">
  <valueCode value="deprecated">
    <extension url="...structuredefinition-standards-status-reason">
      <valueMarkdown value="reason"/>
    </extension>
  </valueCode>
</extension>
```

Both patterns should be checked in the FHIR build tooling to ensure consistent rendering.

## Implementation Plan

### Phase 1: Remove `Bundle.signature` (FHIR-55430)

**Priority:** High — straightforward removal with no normative constraints

1. **Remove element definition** from `source/bundle/structuredefinition-Bundle.xml`
   - Delete lines 657–673 (the entire `<element id="Bundle.signature">` block)

2. **Update narrative** in `source/bundle/bundle-notes.xml`
   - Remove/rewrite the "Signatures" section (lines 381–391) to indicate that `Bundle.signature` has been withdrawn in R6
   - Direct readers to the Provenance-based signatures page

3. **Update signatures page** at `source/signatures.html`
   - Line 464–466: Remove the paragraph about `Bundle.signature` being deprecated; instead state it was removed in R6
   - Lines 472–473: Update the worked example note to use Provenance-based signatures
   - Line 643: Remove reference to "Bundle.signature removed" in payload verification text

4. **Update versions page** at `source/versions.html`
   - Line 690: The `Bundle.signature` URL is used as an example of cross-version extensions. Either update the example to use a different removed element, or annotate that this is an example of an element removed in R6.

5. **Cross-version extension**: Coordinate with `HL7/fhir-extensions` maintainers to create `http://hl7.org/fhir/5.0/StructureDefinition/extension-Bundle.signature` for R5→R6 migration support.

### Phase 2: Fix `StructureDefinition.keyword` rendering (FHIR-55445)

**Priority:** Medium — no structural change; rendering investigation

1. **Investigate deprecation flag rendering** in the FHIR build tooling
   - The element already has the deprecated flag (lines 599–604 in `structuredefinition-StructureDefinition.xml`)
   - The search parameter also has the deprecated flag (line 345 in `bundle-StructureDefinition-search-params.xml`)
   - Determine why the flag doesn't render in the published StructureDefinition

2. **If rendering fix is a build tool issue**, file a separate ticket or fix the build tooling to handle both extension nesting patterns (Pattern A and B above).

3. **No element removal** — `StructureDefinition.keyword` is normative and must remain.

### Phase 3: Remove `MetadataResource.topic` (FHIR-55439)

**Priority:** High — has cascading impact on 16+ derived resources

1. **Remove element definition** from `source/metadataresource/structuredefinition-MetadataResource.xml`
   - Delete lines 155–176 (the entire `<element id="MetadataResource.topic">` block)

2. **Update search parameter** in `source/metadataresource/bundle-MetadataResource-search-params.xml`
   - Keep the `topic` search parameter entry (lines 89–101)
   - Change the `expression` (line 98) from `{{name}}.topic` to search via `useContext`:
     ```xml
     <expression value="{{name}}.useContext.where(code.where(system='http://terminology.hl7.org/CodeSystem/usage-context-type' and code='topic')).value.ofType(CodeableConcept)"/>
     ```
   - Update the `path` extension (line 93) similarly
   - Update the `description` (line 95) to reflect the new search target

3. **Coordinate with other work groups** to ensure derived resources also remove their `topic` elements simultaneously. This is critical — removing the abstract definition without updating derived resources could break the build.

4. **Verify ValueSet `definition-topic`** — this ValueSet (`http://hl7.org/fhir/ValueSet/definition-topic`) may still be referenced by the `useContext`-based approach; confirm it remains available.

### Phase 4: Remove `OperationOutcome.issue.location` (FHIR-55442)

**Priority:** High — straightforward removal; deprecated since R4

1. **Remove element definition** from `source/operationoutcome/structuredefinition-OperationOutcome.xml`
   - Delete lines 218–245 (the entire `<element id="OperationOutcome.issue.location">` block)

2. **Update examples:**
   - `source/operationoutcome/operationoutcome-example.xml` — Remove line 17: `<location value="/f:Patient/f:gender"/>`
   - `source/operationoutcome/operationoutcome-example-validationfail.xml` — Remove line 16: `<location value="/f:Patient/f:identifier"/>`
   - `source/operationoutcome/operationoutcome-example-searchfail.xml` — Remove line 16: `<location value="http.name:exact"/>`

3. **Update narrative** in `source/operationoutcome/operationoutcome-notes.xml`
   - Line 27 references "location of an issue" — this is generic enough to remain
   - No major changes needed since the notes already focus on `expression`

4. **Cross-version extension**: Coordinate with `HL7/fhir-extensions` maintainers to create `http://hl7.org/fhir/5.0/StructureDefinition/extension-OperationOutcome.issue.location` for R5→R6 migration.

1. **Phase 2** (StructureDefinition.keyword rendering fix) — can be done independently
2. **Phase 1** (Bundle.signature removal) — can be done independently
3. **Phase 4** (OperationOutcome.issue.location removal) — can be done independently
