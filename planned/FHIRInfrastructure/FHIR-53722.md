## Resolution Summary

The ticket asks that elements still flagged as `deprecated` in R6 be removed where the rules in `versions.html#deprecation` allow it, rather than being shipped deprecated in a new major (intended-normative) release.

The FHIR Infrastructure WG resolved the ticket as **Persuasive with Modification** with the following FHIR-I-specific dispositions:

- **`Bundle.signature`** — **remove** the element. It has been Trial Use since R4, so it should not remain.
- **`StructureDefinition.keyword`** — **keep** in the spec, flagged as deprecated. The element cannot be removed because it is already normative. Investigate why the deprecation flag is not rendering and fix the rendering.
- **`MetadataResource.topic`** — **remove** the element from the abstract resource, but **keep the search parameter** and update its expression to search on `useContext` with code `topic`. Each derived metadata resource must remove the element at the same time (or update its definition to no longer derive from `MetadataResource` if it has a different reason). Also investigate why the deprecation flag is not rendering.
- **`OperationOutcome.issue.location`** — **remove** the element. It has been deprecated since R4.

Guidance issued to other work groups (handled via the sub-tickets in *Related Tickets* below):

- If an element has had the deprecated flag since R4 → remove the element.
- If an element has a "deprecated" comment **and** a Trial Use flag, **or** the resource was Trial Use in R4, **or** the element was added in R5+ → remove the element.
- If an element has a "deprecated" comment **and no flag** → add the deprecated flag.

## Feature Proposal

### Problem Statement

R6 ballot 4 still ships a number of elements flagged as deprecated. Per `http://hl7.org/fhir/versions.html#deprecation`, deprecated content should not survive a major-version transition unless backward-compatibility (e.g., normative status) prevents removal. Carrying these elements forward forces every R6 model and consumer to reason about elements that the spec itself says should not be used. The reporter also noted that the deprecation flag is not currently being rendered into the published R6-ballot4 StructureDefinitions even though the source XML still carries the standards-status `deprecated` value, which makes it harder for implementers to identify the deprecated set.

### Proposed Change

Implement the four FHIR-I-owned dispositions in `HL7/fhir`, and (separately) coordinate the per-resource sub-tickets owned by other WGs.

2. **Keep `StructureDefinition.keyword`, fix deprecation-flag rendering**
   - File: `source/structuredefinition/structuredefinition-StructureDefinition.xml`
   - Action: Leave the element in place but **fix the malformed `structuredefinition-standards-status` extension** at lines ~599–605. At HEAD the XML is:

     ```xml
     <extension url=".../structuredefinition-standards-status">
       <valueCode value="deprecated">
         <extension url=".../structuredefinition-standards-status-reason">
           <valueMarkdown value="Use the artifact-topic extension instead"/>
         </extension>
       </valueCode>
     </extension>
     ```

     The `standards-status-reason` extension is incorrectly nested **inside** `<valueCode>`, which is a primitive type and cannot contain a child extension. This is the root cause of the deprecation flag not being rendered.

     Restructure to mirror the working `Bundle.signature` shape — i.e., put the `standards-status-reason` extension as a **sibling** of the `valueCode`, both children of the parent `standards-status` extension, and self-close `<valueCode value="deprecated"/>`:

     ```xml
     <extension url=".../structuredefinition-standards-status">
       <extension url=".../structuredefinition-standards-status-reason">
         <valueMarkdown value="Use the artifact-topic extension instead"/>
       </extension>
       <valueCode value="deprecated"/>
     </extension>
     ```
   - Verify the existing `short`, `definition`, and `comment` text on the element continues to call out the deprecation and to point at the `artifact-topic` extension.

3. **Remove `MetadataResource.topic`; rebind the `topic` search parameter to `useContext`**
   - File: `source/metadataresource/structuredefinition-MetadataResource.xml`
   - Action: Delete the entire `<element id="MetadataResource.topic">…</element>` block (currently lines ~155 through the closing tag), including its (similarly malformed — see below) `standards-status` extension.     
     ```xml
     <expression value="{{name}}.useContext.where(code.system='http://terminology.hl7.org/CodeSystem/usage-context-type' and code.code='topic').value.ofType(CodeableConcept)"/>
     ```

     Also update the `{{name}}.topic` `valueString` example expansion in the parameter's `extension` block to the same `useContext`-based form, and update the `description` to make clear the parameter now searches `useContext` for the well-known `topic` code.
   - Cross-resource follow-up (covered by the per-resource sub-tickets — see *Related Tickets*): every derived metadata resource that currently inherits or surfaces a `topic` element must remove it in lock-step. See sub-tickets FHIR-55431 (ActivityDefinition), FHIR-55432 (CodeSystem), FHIR-55433 (ConceptMap), FHIR-55434 (EventDefinition), FHIR-55435 (Library), FHIR-55436 (Measure), FHIR-55439 (MetadataResource — tracker), FHIR-55440 (NamingSystem), FHIR-55443 (PlanDefinition), FHIR-55446 (ValueSet). ### Design Rationale

- **Consistency with the deprecation policy.** `versions.html#deprecation` already says deprecated Trial Use content should not survive a major-version boundary unless normative status forces it. The four FHIR-I dispositions apply that rule literally: remove what we can (Bundle.signature, MetadataResource.topic, OperationOutcome.issue.location), and only retain what normative status pins in place (StructureDefinition.keyword).
- **Replacement paths already exist.** None of the removals leave implementers without a path: signatures move to `Provenance` (already documented in `signatures.html#Bundle`); `MetadataResource.topic` is replaced by the `useContext` pattern using `usage-context-type#topic`, which the resolution explicitly calls out; `OperationOutcome.issue.location` is replaced by the format-independent `issue.expression`, which has been the recommended path since R4.
- **Search-parameter continuity.** Keeping the `topic` SearchParameter on `MetadataResource` (and re-pointing its expression at `useContext`) preserves the public search surface for clients while removing the deprecated element. This is the deliberate "modification" portion of *Persuasive with Modification* — it answers James Jahns's concern without breaking the discovery path that consumers already use.
- **Render-bug fix is mandatory, not cosmetic.** The reporter explicitly noted that the deprecation flag is not rendering for several of these elements. Investigation against the clone shows the cause: in `StructureDefinition.keyword`, `OperationOutcome.issue.location`, and `MetadataResource.topic` the `standards-status-reason` extension is nested **inside** `<valueCode>` rather than being a sibling. This makes the wrapper `standards-status` extension malformed, which suppresses rendering. Fixing the shape on the surviving element (SD.keyword) makes the deprecation visible to humans and tooling. The other two malformed instances disappear with the element-removals in tasks 1 and 4 above.
- **Sub-tickets for non-FHIR-I elements.** The other deprecated elements listed in the original ticket are owned by other WGs and were split out into sub-tickets (FHIR-55430 … FHIR-55446). FHIR-I should not implement those in this PR — its plan is bounded by the four bullets above.

### Breaking Changes

Yes — three of the four dispositions are wire-level breaking changes for any client that has not yet migrated:

- **Removing `Bundle.signature`** — Bundle instances that previously embedded a digital signature on the bundle itself will no longer validate. Implementers must move signing to the `Provenance` resource (already the documented path; `signatures.html#Bundle`).
- **Removing `MetadataResource.topic`** — All derived metadata resources that surface `.topic` as an instance element will lose that element. Existing instances must move topic data into `useContext` with `code = http://terminology.hl7.org/CodeSystem/usage-context-type#topic`. The `topic` SearchParameter continues to work but its underlying FHIRPath changes, so any server that pre-computes search indexes must reindex.
- **Removing `OperationOutcome.issue.location`** — Any client populating `issue.location` (XPath strings) will lose that element. Replacement is `issue.expression` (FHIRPath), which has been the recommended path since R4.

The `StructureDefinition.keyword` rendering fix is **not** a breaking change — element shape, cardinality, and binding remain unchanged; only the rendering of the existing `deprecated` flag is corrected.
