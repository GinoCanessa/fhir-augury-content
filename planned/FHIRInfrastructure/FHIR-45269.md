## Resolution Summary

**Resolution:** Persuasive with Modification.

**Resolution description (verbatim):** *"We will define a feature for this in
the new CapabilityStatement IG."*

Rather than adding a new element to the core `CapabilityStatement` resource or
a new extension in the FHIR Extensions pack, the work group agreed to express
the server's maximum-supported `_count` as a **Feature** in the new
Application Feature Framework IG (`HL7/capstmt`, canonical
`http://hl7.org/fhir/uv/application-feature`). Servers that support the
framework will be able to advertise their max `_count` by declaring this
feature on their `CapabilityStatement`, letting clients discover the value
from conformance metadata instead of probing by trial and error.

## Feature Proposal

### Problem Statement

FHIR's search pagination is deliberately opaque: clients are expected to
follow `next` links rather than jump to arbitrary pages. However, the
`_count` search result parameter is standard, and servers in practice cap it
at widely varying implementation-specific limits (e.g., 100, 1000, 10000,
etc.). There is today no conformance-level way for a server to advertise
its maximum supported `_count`:

- `_count` is a **search result parameter**, not a `SearchParameter`
  resource, so it has no natural slot inside `CapabilityStatement.rest.resource`.
- Clients currently discover the maximum through trial and error — either by
  requesting an absurdly large `_count` and seeing what comes back, or by
  empirically cataloguing per-server behaviour. For high-volume clients
  (especially bulk payer/EHR integrations) this makes it impossible to size
  fetch plans up front, and problematic server caps can stay hidden until a
  patient with a large result set surfaces in production.

Zulip discussion (stream `implementers`, topic *_count and
CapabilityStatements*, 2024-04-08 → 2024-04-10) confirmed the work group's
strong preference to keep pagination opaque, but Gino Canessa explicitly
proposed advertising the max page size as "a great request for the Feature
CapabilityStatement IG." The reporter filed FHIR-45269 against that
suggestion, and the resolution adopted it.

### Proposed Change

Add a new `FeatureDefinition` instance to the Application Feature Framework
IG (`HL7/capstmt`) that lets a server declare its maximum supported `_count`
per resource type (and optionally at the server level).

Concretely:

- **Feature URL:** `http://hl7.org/fhir/uv/application-feature/FeatureDefinition/MaxPageSize`
  (name kept generic because it covers the max number of match resources per
  page, which is what `_count` controls).
- **name:** `MaxPageSize`
- **description:** "The maximum number of match resources the server will
  return per page for a search interaction. This is the largest value the
  server will honour for the `_count` search result parameter; the server
  MAY return fewer resources per page than this value."
- **status:** `active`
- **valueType:** `integer` (positive integer; the largest honoured `_count`).
- **contextDefinition / context:** usable both at the root of
  `CapabilityStatement` (server-wide max) and on
  `CapabilityStatement.rest.resource` (per-resource-type max). Use the same
  URL-form context expression language as the existing `FeatureSupport`
  example:
  - `http://hl7.org/fhir/StructureDefinition/CapabilityStatement#CapabilityStatement`
  - `http://hl7.org/fhir/StructureDefinition/CapabilityStatement#CapabilityStatement.rest.resource`
- **contextExpressionLanguage:** `#text/url`
- **Qualifiers:** none required for STU1 (a per-interaction qualifier — e.g.,
  `search-type` vs `search-system` — can be considered later if the WG wants
  separate limits per interaction).

Add an FSH `Instance` under
`input/fsh/instances/FeatureDefinition/MaxPageSize.fsh`, and a
`FeatureCapabilityStatement` example that declares the feature both at the
server level and on at least one resource (e.g., `Observation`), so clients
and implementers have a concrete template.

Update `input/pagecontent/specification.md` to list `MaxPageSize` under
*Features Defined in this Specification* alongside the existing
`FeatureSupport` feature, with a short narrative that cross-references the
core spec's `_count` semantics (the server MAY return fewer resources than
the requested `_count`, even when below the advertised max).

### Design Rationale

- **Resolution alignment.** The work group explicitly chose the Feature
  Framework over a core `CapabilityStatement` element or a generic
  `CapabilityStatement` extension, which keeps the change isolated to a
  single trial-use IG and avoids touching a normative page.
- **Consistent with existing IG patterns.** The IG already ships exactly one
  "real" feature (`FeatureSupport`) plus one example (`FavoriteColor`). The
  proposed `MaxPageSize` feature reuses the same
  `Instance … InstanceOf: FeatureDefinition Usage: #definition` shape, the
  same URL-form `contextExpressionLanguage`, and the same context-on-the-CS
  pattern demonstrated by `FeatureSupport`, so no new machinery is needed.
- **Integer value type.** `_count` is an unsigned integer in FHIR core;
  matching the type in the FeatureDefinition's `valueType` keeps the
  framework-level declaration faithful and allows the example and any
  tooling to round-trip without casting.
- **Server-wide and per-resource contexts.** Real servers sometimes apply a
  global cap (e.g., 1000) and tighter caps for particularly heavy resources
  (e.g., `DocumentReference`). Allowing the feature on
  `CapabilityStatement.rest.resource` lets those servers advertise both,
  matching the `_count`-related behaviour discussed on Zulip (Paul Church,
  Michele Mottini).
- **Non-binding semantics preserved.** The description explicitly states
  that the server MAY still short a page. This preserves the normative
  behaviour of search (Lloyd McKenzie, Elliot Silver, Gino Canessa on
  Zulip) while giving clients an upper bound.

Alternatives considered (and rejected):

- *New element on core `CapabilityStatement.rest`.* Rejected — normative
  page, larger blast radius, and the Feature Framework is the intended
  vehicle per the resolution.
- *New extension in `hl7.fhir.uv.extensions`.* Mentioned by Gino on Zulip as
  a possibility, but the resolution text specifically picks the
  CapabilityStatement IG, so the extension route is deferred.
- *New search result parameter (e.g., `_useMaxCount`).* Out of scope for
  this ticket — it does not solve the advertisement problem, only the
  request-time hinting problem.

### Step-by-Step Tasks

#### HL7/capstmt

1. **Create the `MaxPageSize` FeatureDefinition instance.**
   - File: `input/fsh/instances/FeatureDefinition/MaxPageSize.fsh` *(new)*
   - Action: Add a new FSH `Instance` using `FeatureSupport.fsh` as the
     template.
   - Details:
     ```fsh
     Instance:   MaxPageSize
     InstanceOf: FeatureDefinition
     Description: """Advertises the maximum number of match resources the server will return per page for a search interaction (i.e., the largest honoured value of the `_count` search result parameter)."""
     Usage:  #definition
     * text.status = #additional
     * text.div = "<div xmlns=\"http://www.w3.org/1999/xhtml\">The maximum number of match resources the server will return per page for a search interaction.</div>"
     * url = "http://hl7.org/fhir/uv/application-feature/FeatureDefinition/MaxPageSize"
     * name = "MaxPageSize"
     * description = "The maximum number of match resources the server will return per page for a search interaction. This is the largest value the server will honour for the `_count` search result parameter; the server MAY still return fewer resources per page than this value."
     * status = #active
     * valueType = #integer
     * contextDefinition = "Context is either a CapabilityStatement (server-wide maximum) or a CapabilityStatement.rest.resource entry (per-resource-type maximum). Expressed using a URL of the form Canonical(StructureDefinition)#{element-id}."
     * contextExpressionLanguage = #text/url
     * context[+] = "http://hl7.org/fhir/StructureDefinition/CapabilityStatement#CapabilityStatement"
     * context[+] = "http://hl7.org/fhir/StructureDefinition/CapabilityStatement#CapabilityStatement.rest.resource"
     ```

2. **Register the FeatureDefinition as a logical-model `Binary` resource.**
   - File: `sushi-config.yaml`
   - Action: Under `resources:`, add a `Binary/MaxPageSize` entry mirroring
     the existing `Binary/FeatureSupport` entry.
   - Details:
     ```yaml
       Binary/MaxPageSize:
         extension:
           - url: http://hl7.org/fhir/tools/StructureDefinition/implementationguide-resource-logical
             valueCanonical: http://hl7.org/fhir/uv/application-feature/StructureDefinition/FeatureDefinition
     ```
     Verify during a local build that the IG publisher picks up the new
     instance; adjust if the IG uses a different packaging convention once
     the `repo-analysis` briefing is generated.

3. **Extend the example `FeatureCapabilityStatement`.**
   - File: `input/fsh/instances/CapabilityStatement/FeatureCapabilityStatementExample.fsh`
   - Action: Append a new `feature` slice that declares `MaxPageSize` with
     an illustrative value, plus (optionally) a per-resource declaration on
     the existing `rest` block.
   - Details (server-wide + per-resource):
     ```fsh
     * extension[feature][+].extension[definition].valueCanonical = "http://hl7.org/fhir/uv/application-feature/FeatureDefinition/MaxPageSize"
     * extension[feature][=].extension[value].valueInteger = 1000
     ```
     If a per-resource example is desired, add a
     `rest.resource` entry (e.g., `Observation`) with the same extension
     pattern using `extension[feature][+]` to preserve slice order (see
     Gotcha about `manualSliceOrdering`).

4. **Document `MaxPageSize` in the specification narrative.**
   - File: `input/pagecontent/specification.md`
   - Action: Under *Features Defined in this Specification*, add a new
     subsection for `MaxPageSize` between or after the existing
     `FeatureSupport` section.
   - Details: Briefly describe the feature's purpose, link to the new
     `Binary-MaxPageSize.html` artifact page, and include a short paragraph
     clarifying that the value is an upper bound — servers MAY still
     return fewer results per page, and clients MUST continue to follow
     `next` links rather than construct pagination URLs themselves.
     Cross-link to FHIR core `_count` (search result parameter) and to
     `CapabilityStatement.rest.resource` for the per-resource context.

5. **Local build and publisher QA.**
   - Files: n/a (runs `_genonce.sh` / `_genonce.bat`).
   - Action: Run the IG publisher locally; verify that
     `Binary-MaxPageSize.html` renders, that the example CapabilityStatement
     shows both the original `FeatureSupport` / `FavoriteColor` features
     and the new `MaxPageSize` declaration, and that no new QA warnings
     are introduced beyond those listed in `input/ignoreWarnings.txt`.

### Testing Considerations

- **Publisher build.** The primary test is a clean local run of the IG
  publisher, verifying new artifact pages and the updated example.
- **Example round-trip.** Confirm the augmented
  `FeatureCapabilityStatementExample` instance serialises to valid JSON
  and XML, and that `extension[feature].extension[value].valueInteger`
  appears as expected.
- **Conformance walk-through.** Manually trace the client workflow: given
  a CapabilityStatement containing a `MaxPageSize` feature with
  `valueInteger = 1000`, a client should be able to read it without any
  extra parsing rules beyond those already needed for `FeatureSupport` and
  `FavoriteColor`.
- **Interoperability note.** Because the feature is optional and new,
  back-compat is automatic; a server that never declares it behaves
  exactly as it does today.

### Open Questions

- **Briefing absent for `HL7/capstmt`.** Run the `repo-analysis` skill

  narrative before publishing, because the canonical URL is immutable once

  requests it.

  CapabilityStatement IG, but confirm no parallel ticket/PR is in flight
