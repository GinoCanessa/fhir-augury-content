## Resolution Summary

Resolution: **Persuasive**. The applied solution has three concrete parts:

1. Add a "best practice" invariant to `Questionnaire.item.answerOption.value[x]`, `Questionnaire.item.enableWhen.answer[x]`, and `Questionnaire.item.initial.value[x]` that fires when the active type is `Coding` and either `code` or `system` is missing. Human text:
   > "When using codings, both code and system should be present. See FHIR specification Questionnaire 'Using Coded Answers' usage notes for guidance."
2. Add a new section titled **Using Coded Answers** to the Questionnaire usage notes (Resource Notes page) explaining why `Coding`s should carry both `code` and `system`, recommending `string` when only a display value is available, and showing the "synthesise a CodeSystem URL from the Questionnaire URL plus itemId" workaround for ad-hoc codes.
3. Add a `<comment>` (or comment-extension) pointer on `answerOption.value[x]`, `enableWhen.answer[x]`, and `initial.value[x]` that says
   > "See [here](questionnaire.html#coded-answers) for additional guidance on using Coding in Questionnaires."

The resolution body is the authoritative source; no follow-on votes or applied-vote comments were recorded on the ticket (the `comments` array is empty).

## Feature Proposal

### Problem Statement

Questionnaire `item.answerOption.value[x]`, `item.enableWhen.answer[x]`, and `item.initial.value[x]` all permit `Coding` as one of their allowed types (see lines 1085–1144, 1310–1386, and 1403–1470 in `source/questionnaire/structuredefinition-Questionnaire.xml`). FHIR currently does not nudge Questionnaire authors away from publishing `Coding`s that lack a `system`, lack a `code`, or carry only a display value — yet, as the resolution observes, "a Coding with no code cannot be meaningfully computed on, and a Coding with no system has no semantic meaning." Authors that genuinely have only a display string should be using the `string` type instead. The Questionnaire spec offers no guidance on this today; the "Permitted values for questions" section (`source/questionnaire/questionnaire-notes.xml`, lines 251–278) talks about `answerValueSet` vs. `answerOption` but says nothing about how `Coding`-typed answers should be populated.

### Proposed Change

Three discrete deliverables, all in `HL7/fhir`:

**A. New best-practice invariant `que-19` (single FHIRPath, three element attachments).**

Add the constraint definition to each of:

- `Questionnaire.item.answerOption.value[x]` (line 1332)
- `Questionnaire.item.enableWhen.answer[x]` (line 1085)
- `Questionnaire.item.initial.value[x]` (line 1425)

Constraint XML (mirroring the existing best-practice pattern at `que-15`, line 880–890):

```xml
<constraint>
  <key value="que-19"/>
  <severity value="warning">
    <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bestpractice">
      <valueBoolean value="true"/>
    </extension>
  </severity>
  <human value="When using codings, both code and system should be present.  See FHIR specification Questionnaire 'Using Coded Answers' usage notes for guidance."/>
  <expression value="$this is Coding implies ($this.code.exists() and $this.system.exists())"/>
  <source value="http://hl7.org/fhir/StructureDefinition/Questionnaire"/>
</constraint>
```

The expression is anchored at the `value[x]` / `answer[x]` element itself; `$this is Coding` short-circuits when the active choice is any other type. Use the next sequential key (`que-19`); existing keys top out at `que-18b` (see the constraint inventory in `structuredefinition-Questionnaire.xml`).

**B. New "Using Coded Answers" section in `source/questionnaire/questionnaire-notes.xml`.**

Insert immediately after the existing `<h3>Permitted values for questions</h3>` block (closes line 277) and before the `<a name="valuesets"> </a>` anchor at line 279:

```html
<a name="coded-answers"> </a>
<h3>Using Coded Answers</h3>
<p>
It's not uncommon for legacy questionnaires to have "coded" questions that have a code but no system or sometimes even just a display value.  While this is not prohibited, it's not good practice.
</p>
<p>
A Coding with no code cannot be meaningfully computed on, and a Coding with no system has no semantic meaning.  If only a display is available, consider using the <code>string</code> type.  If the codes are "made up", simply define a CodeSystem URL based on the Questionnaire URL and itemId or some other unique string.
</p>
```

The page anchor name `coded-answers` is the link target referenced from the StructureDefinition comments and from the new invariant's human text (the resolution uses the page's own anchor scheme; existing anchors in this file include `valuesets`, `booleans`, `polymorphic`).

**C. Pointer comments on the three `value[x]` / `answer[x]` elements.**

Replace (or extend) the `<comment value="..."/>` element on each of the three elements so the existing prose is preserved and a sentence pointing at the new section is appended:

- `Questionnaire.item.answerOption.value[x]` (existing comment line 1339): existing text `"The data type of the value must agree with the item.type."` → append `" See [here](questionnaire.html#coded-answers) for additional guidance on using Coding in Questionnaires."`
- `Questionnaire.item.enableWhen.answer[x]` (line 1085 has no `<comment>` today; insert one immediately after `<short>` at line 1090): set comment to `"See [here](questionnaire.html#coded-answers) for additional guidance on using Coding in Questionnaires."`
- `Questionnaire.item.initial.value[x]` (existing comment line 1435): existing text `"The type of the initial value must be consistent with the type of the item."` → append `" See [here](questionnaire.html#coded-answers) for additional guidance on using Coding in Questionnaires."`

The placeholder `[here]` in the resolution maps to the markdown link `[here](questionnaire.html#coded-answers)`; the ElementDefinition `comment` field is rendered as markdown by the publisher.

### Design Rationale

- **Best-practice invariant, not error-level.** The resolution explicitly calls these out as "best practice" — matching the precedent set by `que-15` (LinkId length warning). That keeps existing instances valid while still flagging the smell during validation. The `elementdefinition-bestpractice` extension on `<severity>` is the canonical signal.
- **One key, three attachments vs. three separate keys.** The applied resolution describes a single human-facing rule that just happens to attach in three places; reusing one key (`que-19`) keeps the rule book short. This mirrors how `que-11` already appears as a `<condition>` on three separate elements (`item`, `answerOption`, `initial`) in the same SD.
- **Anchor in the notes page rather than a new HTML page.** Existing Questionnaire usage notes already host similar guidance sections (`Useful Value Sets`, `Handling Boolean Answers`, `Questions with multiple answer types`) — adding `Using Coded Answers` keeps related guidance co-located and consistent with the publisher's link-resolution rules for `questionnaire.html#fragment`.
- **No `string`-type promotion or schema break.** The change deliberately avoids removing `Coding` from any `value[x]` choice list — it would break interoperability with deployed Questionnaires. A warning + documentation is the lightest-touch nudge that satisfies the ticket.
