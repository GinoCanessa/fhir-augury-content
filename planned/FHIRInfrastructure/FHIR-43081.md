## Resolution Summary

The submitter proposed adding a new complex extension `http://hl7.org/fhir/StructureDefinition/authorization-hint` with two sub-extensions — a required `Coding` named `type` and an optional `string` named `value` — scoped to `SubscriptionStatus.notificationEvent`. The intent is to allow servers to include authorization information (e.g., OAuth challenge tokens) alongside subscription notifications so recipients can tie subsequent resource queries back to the original notification event.

**Resolution (Persuasive with Modification):** Accepted, but the modification requires that standard coding values also be defined in THO (HL7 Terminology) as part of core, and the coded element be bound to that value set.

---

## Feature Proposal

### Problem

When a server sends a subscription notification (e.g., `id-only` or `empty` notifications), the recipient often needs to query back for referenced resources. There is currently no standardized mechanism to include authorization context — such as token exchange hints or OAuth challenge tokens — within the notification itself. This forces implementations to use ad-hoc approaches that lack interoperability.

### Solution

Add a repeating `authorizationHint` backbone element (or extension for backport) on `SubscriptionStatus.notificationEvent` that carries:

1. **`authorizationType`** (1..1, `Coding`): Classifies the authorization hint (e.g., `oAuthChallengeToken`).
2. **`value`** (0..1, `string`): The authorization value itself (e.g., a token or code).

Per the resolution modification, a standard CodeSystem and ValueSet must be defined in THO to provide a controlled vocabulary for `authorizationType`, with an extensible binding on the element.

### Prior Art / Related Work

- **FHIR-46494**: The R6 core implementation commit (`0857a991`) that added `SubscriptionStatus.notificationEvent.authorizationHint` as a BackboneElement to the core spec.
- **FHIR-43275**: "Add 'notified pull' option to notifications" — companion feature adding `relatedQuery` to `notificationEvent`, already Applied.
- **FHIR-55630**: "Create Standard Task.input code for authorization hints" — downstream ticket requesting a Task.input code for the same authorization-hint pattern for non-Subscription workflows.
- **FHIR-55629**: "Combine + Expand on Guidance for Sharing Information" — references `authorization_hint` in the context of workflow information sharing.
- **Zulip thread** ([cds hooks > Security guidance for multi-system hook responses](https://chat.fhir.org/#narrow/stream/cds%20hooks/topic/Security%20guidance%20for%20multi-system%20hook%20responses)): Discusses the same authorization pattern for CDS Hooks where delegated systems need to tie subsequent queries back to the initial notification.

### Gaps Identified in R6 Core

- **No binding on `authorizationType`**: The `authorizationType` element currently has type `Coding` but no ValueSet binding. The resolution explicitly requires standard coding values in THO.
- **Example uses placeholder system**: The example in `notification-id-only-query.xml` uses `http://example.org/fhir/CodeSystem-authorization-type` rather than an official HL7 system.
- **RIM mapping is placeholder**: All three elements have `<map value="N/A (to add?)"/>`.

### Downstream Impact

- **FHIR-55630** (Orders & Observations) depends on the authorization-hint pattern being standardized to create a matching `Task.input` code.
- **CDS Hooks** has analogous needs (per the Zulip thread); a shared solution for tying subsequent queries to initial notifications would benefit both Subscriptions and CDS Hooks.

## Implementation Plan

### Phase 1: Define Terminology in THO (HL7/UTG)

**Goal:** Create a CodeSystem and ValueSet for authorization hint types.

#### Step 1.1: Create CodeSystem

Create a new file in `sourceOfTruth/fhir/codeSystems/`:

- **File:** `CodeSystem-subscription-notification-authorization-type.xml`
- **URL:** `http://hl7.org/fhir/subscription-notification-authorization-type` (or similar)
- **Content:** Define initial codes such as:
  - `oauth2-challenge-token` — "OAuth2 Challenge Token": A token that can be exchanged during an OAuth2 workflow to obtain an access token scoped to the notification context.
  - Additional codes as agreed by the work group (e.g., `smart-launch-context`, `bearer-token`, etc.)
- **Properties:** `status: draft`, `content: complete`, `caseSensitive: true`

#### Step 1.2: Create ValueSet

Create a new file in `sourceOfTruth/fhir/valueSets/`:

- **File:** `ValueSet-subscription-notification-authorization-type.xml`
- **URL:** `http://hl7.org/fhir/ValueSet/subscription-notification-authorization-type`
- **Compose:** Include all codes from the CodeSystem above.

### Phase 2: Bind Terminology to Core Element (HL7/fhir)

**Goal:** Add a binding to `SubscriptionStatus.notificationEvent.authorizationHint.authorizationType`.

#### Step 2.1: Add Binding to StructureDefinition

- **File:** `source/subscriptionstatus/structuredefinition-SubscriptionStatus.xml`
- **Element:** `SubscriptionStatus.notificationEvent.authorizationHint.authorizationType`
- **Change:** Add a `<binding>` element after the `<type>` element:

```xml
<binding>
  <strength value="extensible"/>
  <description value="Classification of the authorization hint."/>
  <valueSet value="http://hl7.org/fhir/ValueSet/subscription-notification-authorization-type"/>
</binding>
```

#### Step 2.2: Update Example

- **File:** `source/subscriptionstatus/notification-id-only-query.xml`
- **Change:** Replace the placeholder system `http://example.org/fhir/CodeSystem-authorization-type` with the official system URL `http://hl7.org/fhir/subscription-notification-authorization-type`.

#### Step 2.3: Update Resource Packs

- **File:** `source/subscriptionstatus/list-SubscriptionStatus-packs.xml`
- **Change:** If ValueSet/CodeSystem packing is required, add the new resources.

### Phase 3: Backport IG Extension (HL7/fhir-subscription-backport-ig)

> **REPO_NOT_CACHED: HL7/fhir-subscription-backport-ig**
>
> This is the primary target repository for this ticket. The implementation details below are based on the ticket description, the R6 core implementation, and the pattern established by FHIR-43275 (`backport-related-query`).

**Goal:** Define a complex extension for use in R4/R4B notifications.

#### Step 3.1: Create Extension StructureDefinition

Create a new extension definition (likely FSH or XML):

- **Extension URL:** `http://hl7.org/fhir/uv/subscriptions-backport/StructureDefinition/notification-authorization-hint` (per FHIR-55630 reference)
- **Context:** `SubscriptionStatus.notificationEvent` (R5 backport Parameters part)
- **Structure:**
  - Sub-extension `type` (1..1, `valueCoding`): Classification of the authorization hint, bound to the ValueSet from Phase 1.
  - Sub-extension `value` (0..1, `valueString`): The authorization value (e.g., a token).
- **Cardinality on parent:** 0..*

Follow the pattern of the existing `backport-related-query` extension from FHIR-43275.

#### Step 3.2: Add to Notification Parameters Profile

Update the backport notification Parameters profile to include the new extension on the `notification-event` part, mirroring how `related-query` was added.

#### Step 3.3: Add Documentation

Add an "Authorization Hinting" section to the backport IG narrative, paralleling the content in `subscription-notes.xml` from the core spec.

#### Step 3.4: Add Examples

Create or update notification examples to demonstrate usage of the `notification-authorization-hint` extension with the backport Parameters structure.

### Phase 4: Validation and Testing

#### Step 4.1: Verify Core Build

- Build the FHIR core spec after Phase 2 changes to ensure the StructureDefinition validates correctly with the new binding.
- Confirm the example passes validation against the updated StructureDefinition.

#### Step 4.2: Verify Backport IG Build

- Build the Subscriptions Backport IG after Phase 3 changes.
- Validate examples against the new extension profile.

#### Step 4.3: Cross-Check Downstream

- Verify that the CodeSystem/ValueSet URLs are consistent across core, UTG, and the backport IG.
- Ensure the terminology binding strength (extensible) allows implementations to define custom authorization types while encouraging use of standardized codes.

## Open Questions

1. **Code system content:** What specific authorization type codes should be defined initially? The example uses `oauth2-challenge-token`, but the work group should agree on a starter set (e.g., should `smart-launch-context` or `bearer-token` be included?).
2. **RIM mappings:** The current RIM mapping is `N/A (to add?)` for all three elements. Should meaningful RIM mappings be provided?
3. **Broader scoping:** The original ticket suggested additional context scoping to `Endpoint`, `reference(Endpoint)`, and `Organization` if added as a core extension. Since the R6 implementation uses a BackboneElement on `SubscriptionStatus.notificationEvent`, should additional contexts be addressed separately?
4. **CDS Hooks alignment:** Per the Zulip discussion, CDS Hooks has the same authorization delegation problem. Should the terminology and pattern be designed to be reusable by CDS Hooks, or is that a separate effort?
