## Resolution Summary

The reporter observed that on the main datatypes page
(`source/datatypes.html`, rendered as `datatypes.html`) the *Constraints*
box is missing for both `SampledData` and `RelativeTime`, even though
their StructureDefinitions declare invariants (`sdd-1` for SampledData
and `rlt-1` / `rlt-2` for RelativeTime, which do show up in the detailed
view at `datatypes-definitions.html#...`).

The applied resolution (`Persuasive`) is recorded on the ticket as:

> This is fixed for SampledData but is still broken for RelativeTime

This matches what is in the clone (see Impact Analysis below): a
`<div class="use"><p><b>Constraints</b></p>[%dt.constraints SampledData%]</div>`
block exists in the SampledData section of `source/datatypes.html`
(added by commit `8577415` as part of FHIR-49899 work), but the
equivalent block for `RelativeTime` is still missing. The remaining
implementation work is therefore to add the analogous constraints block
to the `RelativeTime` section of the same page so that `rlt-1` and
`rlt-2` render in the main datatypes page, not only in the detailed
view.

## Feature Proposal

### Problem Statement

FHIR specification pages for complex datatypes include a *Constraints*
box that renders the invariants declared on that type's
StructureDefinition. The box is produced by the HL7 publisher when it
encounters the `[%dt.constraints <TypeName>%]` tag on a page. Both
`SampledData` and `RelativeTime` declare invariants in their
spreadsheet-authored definitions
(`source/datatypes/sampleddata.xml` "Invariants" worksheet — `sdd-1`
equivalent invariant on the `SampledData` context; and
`source/datatypes/relativetime.xml` "Invariants" worksheet — two
invariants on the `RelativeTime` context: `contextReference.empty() or
contextDefinition.empty()` and the `contextPath` / `contextReference` /
`contextDefinition` triad), yet the reporter observed that on
`http://hl7.org/fhir/6.0.0-ballot3/datatypes.html` those invariants
never appear in the per-type Constraints box on the main page. The
invariants *do* appear in the detailed view
(`datatypes-definitions.html#...`), which is driven by a different
publisher path (snapshot-based), so the issue is not with the
StructureDefinition itself; it is with the `source/datatypes.html`
authoring file not requesting the constraints block for these types.

Since the last FHIR 6.0.0 ballot, the SampledData case was fixed (see
`git log` entry `8577415 FHIR-49899` on `source/datatypes.html`, which
added the `<div class="use">…[%dt.constraints SampledData%]…</div>` block
to the SampledData section). The RelativeTime case was overlooked and
is still not rendering a constraints box.

### Proposed Change

Make a single, surgical edit to `source/datatypes.html` in `HL7/fhir`:

- Insert a `Constraints` block inside the `RelativeTime` `<h2>` section,
  mirroring the block already present for SampledData. Specifically,
  between the last prose paragraph of the section
  (`<p>The method to express a natural language description of the
  RelativeTime value is to use the text element.</p>` at line 1761)
  and the "used in the following places" paragraph
  (`<p>RelativeTime is used in the following places: [%dtusage
  RelativeTime%]</p>` at line 1763), insert:

  ```html
  <div class="use">
  <p><b>Constraints</b></p>
  [%dt.constraints RelativeTime%]
  </div>
  ```

No other file needs to change. The RelativeTime StructureDefinition
already carries the two invariants (see
`source/datatypes/relativetime.xml` → Invariants worksheet) with
correct `Severity`, `Context=RelativeTime`, English description and
FHIRPath `Expression`, so the publisher has everything it needs to
populate the box once the `[%dt.constraints RelativeTime%]` tag is
present. No spreadsheet edit is required and the publisher
(`tools/`) is out of scope per the HL7/fhir briefing.

SampledData already has the constraints block (lines 1333–1336 of
`source/datatypes.html`); it remains untouched.

### Design Rationale

- **Consistency with SampledData.** The already-applied fix for
  SampledData (FHIR-49899, commit `8577415`) uses exactly this
  `<div class="use">…[%dt.constraints <Type>%]…</div>` pattern
  immediately before the `[%dtusage <Type>%]` paragraph. Applying the
  same pattern to RelativeTime keeps both complex datatypes consistent
  with each other and with the other types on the page that already
  show a Constraints box (e.g., `Attachment` line 620, `Quantity` line
  951, `Period` line 1284, `HumanName` line 1524, `Address` line 1570,
  `Timing` line 1718, `Signature` line 1782).
- **Minimal risk.** The invariants themselves are unchanged (wire
  format, FHIRPath, and detailed view are all already correct). Only
  the main datatype page's rendering of the invariants is affected.
- **Matches the approved resolution text.** The `Persuasive`
  resolution recorded on the ticket says SampledData is already fixed
  and RelativeTime is still broken, so the only remaining work is the
  RelativeTime half of the original two-part fix.
- **Alternatives considered.** Moving this rendering into the
  publisher (so that every datatype automatically gets a Constraints
  box when invariants exist) would avoid future recurrences, but that
  lives in `tools/` which the HL7/fhir briefing explicitly flags as
  out-of-scope / publisher harness. It is also outside the scope of a
  Technical Correction ticket.

## Implementation Plan

### Prerequisites

None. The RelativeTime invariants (`rlt-1`, `rlt-2`) already exist in
`source/datatypes/relativetime.xml` (Invariants worksheet rows 2–3,
context `RelativeTime`) and the publisher's `[%dt.constraints %]` tag
is already proven to work for other types on the same page.

#### HL7/fhir
