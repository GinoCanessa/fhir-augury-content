## 1. Problem Statement

The `NamingSystem` resource allows specifying a `publisher` (the entity responsible for the resource) and a `responsible` party (the organization responsible for issuing identifiers/codes in the namespace), but both are encoded as plain strings. This prevents linking these textual names to the actual `Organization`, `Practitioner`, or other resources that represent the entity. The submitter requests the ability to reference an `Organization` resource for the publisher and responsible party.

## 2. Resolution Summary

The resolution (Persuasive with Modification) calls for defining **three new extensions** rather than modifying the core resource types. All three provide a `Reference` companion to an existing string-typed element:

1. **`Canonical-publisherReference`** ŌĆö scope: `CanonicalResource`
   - *"A reference to the entity that serves as the publisher for this resource"*
   - Type: `Reference(Organization | Practitioner | PractitionerRole | CareTeam)`

2. **`ContactDetail-reference`** ŌĆö scope: `ContactDetail`
   - *"A reference to the entity whose context-specific contact information is provided"*
   - Type: `Reference(Organization | Practitioner | PractitionerRole | CareTeam | Patient | RelatedPerson)`

3. **`NamingSystem-responsibleReference`** ŌĆö scope: `NamingSystem`
   - *"A reference to the entity who maintains the NamingSystem"*
   - Type: `Reference(Organization | Practitioner | PractitionerRole | CareTeam)`

---

## 3. Feature Proposal

### Approach
Create three new FHIR extension StructureDefinitions in **HL7/fhir-extensions**. Each follows the standard simple-value extension pattern used throughout the repo (single `value[x]` of type `Reference`). The extensions allow systems to provide machine-readable references alongside existing human-readable string elements, without breaking changes to core resource definitions.

### Relationship to Existing Extensions
| Existing Extension | Relationship |
|---|---|
| `artifact-contactDetailReference` (`input/definitions/datatypes/`) | **Overlaps** with proposed `ContactDetail-reference`. The existing extension already has context `ContactDetail` and targets `Practitioner`, `PractitionerRole`, `Patient`, `RelatedPerson`, `Organization`. The resolution adds `CareTeam` to the target list. The work group must decide whether to: (a) update this existing extension to add `CareTeam` and rename/alias it, or (b) create a second parallel extension. **Recommendation: update the existing `artifact-contactDetailReference` to add `CareTeam` as a target profile**, since it already covers the same context and purpose. |
| `artifact-publisher` (`input/definitions/Resource/`) | A string-typed publisher extension for non-canonical resources. The new `Canonical-publisherReference` is the Reference companion, **not** a replacement. Both may coexist. |

## 5. Detailed Implementation Plan

### 5.1 Create `canonicalresource-publisherReference` Extension

**File:** `input/definitions/CanonicalResource/StructureDefinition-canonicalresource-publisherReference.xml`

Create a new XML StructureDefinition following the pattern of existing CanonicalResource extensions (e.g., `canonicalresource-short-description.xml`):

- **id:** `canonicalresource-publisherReference`
- **url:** `http://hl7.org/fhir/StructureDefinition/canonicalresource-publisherReference`
- **name:** `CRPublisherReference`
- **title:** `CanonicalResource Publisher Reference`
- **wg extension:** `fhir` (FHIR Infrastructure)
- **fmm:** `1` (new extension)
- **standards-status:** `trial-use`
- **status:** `active`
- **context:** `{ type: "element", expression: "CanonicalResource" }`
- **description:** *"A reference to the entity that serves as the publisher for this resource."*
- **value[x]:** `Reference` with targetProfiles:
  - `http://hl7.org/fhir/StructureDefinition/Organization`
  - `http://hl7.org/fhir/StructureDefinition/Practitioner`
  - `http://hl7.org/fhir/StructureDefinition/PractitionerRole`
  - `http://hl7.org/fhir/StructureDefinition/CareTeam`
- **cardinality:** `0..1`

**Template reference:** Use `StructureDefinition-canonicalresource-short-description.xml` as structural template; use `StructureDefinition-artifact-contactDetailReference.xml` as the model for the Reference value type pattern.

### 5.2 Create `namingsystem-responsibleReference` Extension

**File:** `input/definitions/NamingSystem/StructureDefinition-namingsystem-responsibleReference.xml`

Create a new XML StructureDefinition following the pattern of existing NamingSystem extensions (e.g., `namingsystem-checkDigit.xml`):

- **id:** `namingsystem-responsibleReference`
- **url:** `http://hl7.org/fhir/StructureDefinition/namingsystem-responsibleReference`
- **name:** `NSResponsibleReference`
- **title:** `NamingSystem Responsible Reference`
- **wg extension:** `fhir` (FHIR Infrastructure)
- **fmm:** `1` (new extension)
- **standards-status:** `trial-use`
- **status:** `active`
- **context:** `{ type: "element", expression: "NamingSystem" }`
- **description:** *"A reference to the entity who maintains the NamingSystem, providing a machine-readable companion to NamingSystem.responsible."*
- **value[x]:** `Reference` with targetProfiles:
  - `http://hl7.org/fhir/StructureDefinition/Organization`
  - `http://hl7.org/fhir/StructureDefinition/Practitioner`
  - `http://hl7.org/fhir/StructureDefinition/PractitionerRole`
  - `http://hl7.org/fhir/StructureDefinition/CareTeam`
- **cardinality:** `0..1`

**Template reference:** Use `StructureDefinition-namingsystem-checkDigit.xml` as structural template.

### 5.3 Update `artifact-contactDetailReference` Extension

**File:** `input/definitions/datatypes/StructureDefinition-artifact-contactDetailReference.xml`

This existing extension already covers the `ContactDetail-reference` use case from the resolution (context: `ContactDetail`, value: `Reference`). It currently targets `Practitioner`, `PractitionerRole`, `Patient`, `RelatedPerson`, `Organization`.

**Change:** Add `CareTeam` to the target profile list in the `value[x]` element:

```xml
<!-- Add after existing targetProfiles in Extension.value[x] -->
<targetProfile value="http://hl7.org/fhir/StructureDefinition/CareTeam"/>
```

**Rationale:** The resolution specifies the ContactDetail extension should allow `Reference(Organization|Practitioner|PractitionerRole|CareTeam|Patient|RelatedPerson)`. The existing extension already has all except `CareTeam`.

> **ŌÜĀ’ĖÅ Open Question:** The existing extension is owned by work group `cds` (Clinical Decision Support). FHIR Infrastructure should coordinate with CDS on adding the `CareTeam` target. If CDS objects, a separate extension could be created, but this would be duplicative.

## 6. Testing Considerations

- **Validation:** Each new StructureDefinition should pass the FHIR IG publisher validation when the `fhir-extensions` IG is built.
- **Context correctness:** Verify `CanonicalResource` context works across all canonical resource types (NamingSystem, ValueSet, CodeSystem, etc.).
- **Example resources:** Consider adding example resources in `input/examples/` demonstrating usage of each extension on a NamingSystem instance.
- **Cross-version:** New extensions should declare `fhirVersion` consistent with the current target (6.0.0 in FHIR core; 5.0.0 in fhir-extensions).
