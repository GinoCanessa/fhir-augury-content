## Resolution Summary

**Resolution:** Persuasive.

The September 2025 WGM (FHIR Infrastructure, Wed Q3, Pittsburgh) voted to change how
references to *additional resources* (resources defined outside the core
specification â€” see `resource.html#additional`) are expressed on the wire. The
current text at `references.html#additional` requires the
`alternate-reference` extension for every reference whose target is an
additional resource. The approved resolution is:

> References to additional resources can be made using normal (plain)
> `Reference.reference` references **if this is declared in the
> StructureDefinition of the additional resource.**

In other words: the StructureDefinition that defines an additional resource
must be able to opt-in to being the target of a normal `Reference`. When it
does, `Reference.reference` (and `Reference.type`, via the `type`-uri pattern
already documented for additional resources at `references.html` line 124)
is the correct encoding â€” `alternate-reference` is no longer required for
that case, only for the "future/past FHIR version" scenario and for
additional-resource targets whose SDs have not made the declaration.

The resolution text is short; it does not yet specify the exact mechanism
(extension URL, element name) for the declaration. The implementer must
pick a concrete mechanism (see Open Questions) and roll it into the edit
to `structuredefinition.html#additional` / `structuredefinition-notes.xml`
and the prose on `references.html#additional`. A transcript of the WGM
discussion was to be attached to the ticket by the Infrastructure co-chair
and should be consulted before the wording is finalised.

## Feature Proposal

### Problem Statement

The `source/references.html` "Additional References" section (lines 304â€“330)
currently states, without qualification, that the only way to reference an
additional resource from an element whose core targets do not include it is
to use the `alternate-reference` extension (defined in
`HL7/fhir-extensions`):

> In both of these cases, the solution is to use the alternate-reference
> extension. It allows referencing any type of resource - including
> additional resources. â€¦

This forces a heavy-weight encoding (an extension wrapping the `Reference`)
on every exchange that points at an additional resource, even when the
additional resource's maintainers have published a stable StructureDefinition
and intend implementations to treat it as a first-class reference target.
The WGM discussion concluded that normal `Reference.reference` should be
permitted for additional resources whose SDs explicitly opt in, consistent
with how `references.html` line 124 already describes the `type` URI for
additional resources.

### Proposed Change

### Design Rationale

- **Consistency with the existing "additional resource as a `type`
  uri" pattern.** `references.html` line 124 already says that for
  additional resources, `Reference.type` is the canonical URI of the SD
  and `Reference.reference` is a normal relative/absolute URL. The
  resolution simply removes the "but you still have to wrap it in an
  extension" gate from the natural wire form.
- **Opt-in, not automatic.** The WGM text explicitly requires the
  declaration in the SD. This preserves the current default (additional
  resources are *not* core targets) and avoids silently enabling every
  registered additional resource as a reference target â€” which was a
  concern raised when the `alternate-reference` extension was originally
  introduced.
- **Extension-based declaration is the established HL7 mechanism** for
  SD-level metadata (see the `implements` extension referenced at
  structuredefinition-notes line 241â€“243). A new element on
  StructureDefinition would be a much larger change and is not required
  to meet the resolution.
- **Alternate-reference retained for genuine fallbacks.** The inter-version
  scenario (first bullet of Â§Additional References) cannot use normal
  references and still needs `alternate-reference`, so the extension stays
  in the spec; only the guidance for the second bullet changes.

### Breaking Changes

- **Semantic broadening.** Instances that previously *had* to be
  transmitted as `alternate-reference` may now be transmitted as normal
  `Reference.reference`. Existing implementations that consume
  `alternate-reference` still work; existing producers are not forced to
  change. **No wire-level breaking change.**
- **Validation tightening at the SD boundary.** A profile whose
  `Reference.type.targetProfile` points at an additional resource's SD is
  now valid **only** if that SD carries the opt-in extension. This is a
  new validation rule that implementations may not yet enforce; it is
  technically new behaviour but does not break any previously-conformant
  instance (since previously such profiling was disallowed entirely â€”
  Â§Additional References line 316: "Core elements aren't permitted to
  have resource targets that are additional resources").
- **Migration section edit.** Lines 325â€“330 of `references.html`
  currently document the additionalâ†’core migration pain. The new two-path
  story makes that description insufficient; the updated text needs to
  address the possibility that an additional resource is reachable via
  normal reference *before* it moves to core, which is itself a migration
  edge.

Overall: **No backward-incompatible wire changes.** The change is
permissive, opt-in, and additive.
