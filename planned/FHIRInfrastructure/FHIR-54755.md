## Resolution Summary

The resolution is **Persuasive with Modification**. It requires two concrete edits to the
narrative-binding documentation page (`source/narrative.html`):

1. **Add a new section** titled **"References to other resources"**, placed **above** the
   existing **"Internal Id References"** section, with this content (verbatim from the
   resolution):

   > A narrative may include a link to another resource by using an `<a/>` tag with the
   > `href` being a relative or absolute reference to the target resource. When navigating
   > such a link, the system displaying the narrative SHOULD make available the target
   > resource via its normal resource rendering mechanism.
   >
   > E.g.
   >
   > Patient: `<a href="Patient/example">(link)</a>`

2. **Remove the "Internal Id References" section** and **move the "Linking between Data
   and Narrative" section up** (into the slot the removed section vacated, immediately
   following the new "References to other resources" section). **Keep the anchor tags
   for both** removed/moved sections so that existing inbound deep-links
   (`narrative.html#internal`, `narrative.html#linking`) continue to resolve.

The resolution does not propose any new extension, code system, value set, or schema
change — this is entirely a specification-prose update to one HTML page.

## Feature Proposal

### Problem Statement

The "Linking between Data and Narrative" section of `narrative.html` only covers the
existing `narrativeLink` and `originalText` extensions, which target structured data
elements. The reporter (Bas van den Heuvel) observed that a server producing narrative
XHTML often wants to reference **another resource** as the source of the rendered
information, and the spec does not currently say how to express that — leaving readers
to guess between an inline `<a>` text link, an IHE-IMR-style reference, or some custom
extension. The page also still carries an "Internal Id References" section that
predates the consolidated narrative-linking guidance and is now largely redundant /
out of order in the page flow.

### Proposed Change

Edit `source/narrative.html` (FHIR Core authoring root `source/`) as follows:

1. **Insert a new `<h3>` section** with anchor `references` titled
   **"References to other resources"** containing the resolution's prose. Suggested
   markup, modelled on the surrounding sections in the page:

   ```html
   <a name="references"></a>
   <h3>References to other resources</h3>
   <p>
   A narrative may include a link to another resource by using an
   <code>&lt;a/&gt;</code> tag with the <code>href</code> being a relative or absolute
   reference to the target resource. When navigating such a link, the system
   displaying the narrative <b>SHOULD</b> make available the target resource via its
   normal resource rendering mechanism.
   </p>
   <p>E.g.</p>
   <pre class="xml">
   Patient: &lt;a href="Patient/example"&gt;(link)&lt;/a&gt;
   </pre>
   ```

   The new section is inserted **above** the current "Internal Id References" section
   (currently at `narrative.html:218`).

2. **Remove the body** of the "Internal Id References" section
   (`narrative.html:218–243`), i.e. the `<h3>` heading and its four `<p>` paragraphs,
   while **retaining the `<a name="internal"></a>` anchor tag** so existing inbound
   links from `source/xml.html` and `source/references.html` (see Related Specifications)
   do not break.

3. **Move the "Linking between Data and Narrative" section** (currently
   `narrative.html:338–406`, anchor `linking`) **up** the page so it immediately
   follows the new "References to other resources" section, and **retain its
   `<a name="linking"></a>` anchor tag** at the new location.

After the change the section order in `source/narrative.html` becomes:

1. `Narrative` (intro)
2. `XHTML Content`
3. `Multi-language support for Narratives`
4. `Image References`
5. **`References to other resources`** *(new)*
6. **`Linking between Data and Narrative`** *(moved up; was below Styling the XHTML)*
7. `Styling the XHTML`
8. `Clinical Safety Concerns`

Anchors `internal` and `linking` are preserved at their new in-page positions; the new
section adds anchor `references`.

### Design Rationale

- **Reuse of plain XHTML `<a href>`.** Narratives are XHTML, and the FHIR
  specification already allows arbitrary XHTML in `Narrative.div` subject to the
  Narrative rules. Mandating a plain `<a href="<resource>">` (relative or absolute
  resource reference) is the lowest-friction option and is consistent with how
  `Image References` and `Internal Id References` currently leverage core XHTML
  attributes rather than inventing a FHIR-specific markup. The committee chose this
  over IHE-IMR-style markup or a new extension precisely because no new artifact is
  needed.
- **`SHOULD` (not `SHALL`) on the rendering side.** Narrative renderers may operate
  in contexts where the target resource is not resolvable (printed copy, sandboxed
  viewer); `SHOULD` matches the existing tone of the page (cf. the data-URL guidance
  for embedded images at line 239).
- **Section reordering.** Putting "References to other resources" + "Linking between
  Data and Narrative" together, **before** the styling/safety material, groups all
  cross-reference patterns (out-of-resource links, internal id references, narrative
  ↔ data links) in one place. The "Internal Id References" content is being dropped
  because its substance is now subsumed by the new section plus the existing
  `narrativeLink` / `originalText` guidance in "Linking between Data and Narrative".
- **Anchor preservation** is non-negotiable: a repo-wide search confirms inbound
  links to `narrative.html#internal` from `source/xml.html` and
  `source/references.html`. Keeping the bare `<a name="internal"></a>` and
  `<a name="linking"></a>` markers avoids dangling fragment URLs even though their
  surrounding content is removed or relocated.

  This is flagged under *Open Questions* below; no plan-time test is needed.
- **Interoperability.** No on-the-wire changes; no FHIR resource instance is altered.
  Ballot reviewers should focus on prose clarity rather than conformance impact.

### Open Questions

- **Anchor placement when a section body is removed.** The resolution literally says
  "keeping the anchor tags for both". The plan assumes both anchors should be
  preserved at the position of the relocated "Linking between Data and Narrative"
  section (so `narrative.html#internal` lands the reader at the consolidated
  cross-reference material). If the work-group instead intended the `internal`
  anchor to stay where it is and become a marker pointing forward, the implementer
  should confirm before merging.
- **Heading level for the new section.** The plan uses `<h3>`, matching all sibling
  sub-sections of the page-level `<h2>Narrative</h2>`. Confirm with the work-group
  that the new content is intended at the same level rather than as a `<h4>` under
  "Linking between Data and Narrative".
- **Cross-server / external resource references.** The new prose permits "absolute
  reference to the target resource" without further qualification. Some reviewers
  may want a brief safety / privacy note about following links to external servers
  from rendered narrative; this is not in the resolution, so the plan does not add
  it, but it is worth raising during ballot reconciliation.
- **Audit of inbound `#linking` references in downstream repositories.** Plan-time
  grep was scoped to `source/` in `HL7/fhir`. Before the change ships, it would be
  prudent to also search `HL7/fhir-extensions` and key IG repos for
  `narrative.html#linking` to confirm no downstream link breaks even though the
  anchor is preserved. (No cross-repo touch points were flagged by the briefing, so
  this is a low-risk verification rather than a required task.)
