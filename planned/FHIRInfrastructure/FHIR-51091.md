## Resolution Summary

Resolution: **Persuasive with Modification**. The contexts table on the
ElementDefinition page (the "Interpretation of ElementDefinition in
different contexts" grid) currently covers only *Type definition* and
*Constraint definition* where "Type definition" is implicitly read as any
`StructureDefinition` whose `derivation = specialization`. That reading
incorrectly sweeps logical models into the same rules as resources /
complex types / primitive types. The approved fix is:

1. **Replace the "expected" / "Not expected" vocabulary with "required"
   / "prohibited"** throughout the table (and drop the explanatory
   bullet that defines Expected/Not Expected).
   - This is flagged in the resolution as *theoretically breaking* ã it
     changes the formal force of the entries, so existing models in the
     ecosystem must be reviewed (Grahame Grieve) and toolsmiths asked
     how their code handles these cells before merge.
2. In the **Type definition, following elements** column mark `slicing`
   as **prohibited** (today it is `optional`).
3. **Rename the "Type definition, ãÎ" columns to "FHIR Type definition,
   ãÎ"** and add a new paired column **"Logical Type definition,
   following elements"** (and, per the 2025-07-21 @ginocanessa comment,
   draft matching content for a **"Logical Type definition, root
   element"** column too ã to be confirmed with the WG).
4. **Logical Type definition column contents** are the same as the FHIR
   Type definition column *except* for the following overrides:
   - `short` / `definition`: **required ã recommended**
   - `fixed[x]`, `pattern[x]`, `minValue[x]`, `maxValue[x]`, `maxLength`:
     **prohibited ã optional**
   - `isSummary`: leave as `optional`, but add a footnote:
     *"The definition of isSummary is tied to its use in RESTful search
     and therefore is not technically relevant for logical models,
     however its use is not prohibited because of the possibility that
     the concept might still have meaning in some logical model
     contexts."*
5. Per the @ginocanessa comment, the nearby **Path** table / prose (on
   the same page, anchor `#path`) must also be reviewed to see whether
   logical-model-specific wording is needed there.

## Feature Proposal

### Problem Statement

The `<h2>Interpretation of ElementDefinition in different contexts</h2>`
table (`cache/github/repos/HL7_fhir/clone/source/elementdefinition.html`,
lines 131ã328) has only four columns:

- Type definition, root element
- Type definition, following elements
- Constraint Definition, root element
- Constraint Definition, following elements

The `Type definition` columns are applied by tooling to *any*
`StructureDefinition` with `derivation = specialization`, including
logical models (`kind = logical`). That means the normative rules today
say, among other things, that a logical-model element **must not** use
`minValue[x]`, `maxValue[x]`, `maxLength`, `fixed[x]`, or `pattern[x]`,
and its `short`/`definition` is **required**. Those constraints make
sense for primitive, complex, and resource specialisations (which are
generic building blocks), but they are wrong for logical models, which
routinely encode context-specific numeric bounds, string lengths, or
fixed example values. The resolution corrects this by splitting the
"Type definition" columns into two parallel tracks (FHIR types vs.
logical models) and tightens the terminology the table uses.

### Design Rationale

- The resolution's word swap (`expected`/`Not expected` ã
  `required`/`prohibited`) aligns the contexts table with the
  conformance vocabulary already used in the rest of the page
  (`prohibited`, `required`, `recommended`, `optional`) and removes an
  ambiguous "SHOULD/SHOULD-NOT-but-not-invalid" middle ground that
  toolsmiths have to special-case.
- Splitting the Type-definition track into **FHIR Type definition** and
  **Logical Type definition** is a purely documentation-shaped change
  that keeps the existing rules intact for primitive-type /
  complex-type / resource specialisations while exposing the
  logical-model-specific relaxations (`fixed[x]`, `pattern[x]`,
  `minValue[x]`, `maxValue[x]`, `maxLength` all allowed, short /
  definition only *recommended*). This matches real practice (e.g.,
  `id`, `string` primitives use `maxLength` via invariants today).
- The resolution preserves `isSummary` as *optional* for logical models
  rather than *prohibited* so models that legitimately use it for
  application-layer summaries aren't rejected; the footnote records
  that this is intentional but discouraged.
- Alternatives considered implicitly in the ticket thread:
  - *Do nothing because the page is normative* ã rejected; the current
    wording is an inadvertent over-reach and the WG judges the fix
    theoretical-breaking-at-most.
  - *Keep one Type-definition track and gate by a footnote* ã rejected
    as harder to read and tool-friendly than parallel columns.

### Breaking Changes

- **Theoretically breaking.** The resolution itself flags this: swapping
  `expected` / `Not expected` for `required` / `prohibited` strengthens
  the force of several cells (notably `type`, `isSummary`, `condition`
  in the root-element columns, and `base` in all columns). Any
  StructureDefinition or tool that relied on the permissive
  "Expected/Not Expected" reading ã where a non-conforming element was
  still considered valid ã will be tightened. The resolution explicitly
  requires Grahame Grieve to check existing models in the ecosystem and
  for toolsmiths to be consulted before merge.
- The new Logical-Type-definition columns *loosen* rules for
  `kind = logical` StructureDefinitions (allowing `fixed[x]`,
  `pattern[x]`, `minValue[x]`, `maxValue[x]`, `maxLength`, and
  downgrading `short`/`definition` from required to recommended). This
  is backward-compatible for consumers of existing logical models ã
  authors gain permission, they do not lose any.
- Marking `slicing` as `prohibited` in FHIR-Type-definition, following
  elements is a tightening: any specialisation-style
  StructureDefinition that currently places `slicing` on a non-root
  element becomes non-conformant. Validate via Grahame's ecosystem
  sweep.
