## Resolution Summary

**Resolution: Persuasive with Modification.** The resolution text directs that changes be made **"in the appropriate publications outside of the FHIR core specification"** — i.e., the IG Publisher tooling and the Best Practices (`FHIR/ig-guidance`) IG, not the FHIR R6 specification itself. Specifically:

1. Ensure that the `Parameters` file used in creating the IG is included in the IG output folder and the published package file.
2. Generate a fragment that renders the parameters-file content for human consumption, and (in HL7) set a requirement that the fragment must be rendered within the IG.
3. Make explicit that the parameters used for expansions in the IG only apply to the IG itself. If editors have requirements or recommendations for the broader implementation community, they must state them explicitly and account for the impact of dependencies.
4. Recommend (HL7-wide) that IGs use expansion parameters sparingly and delegate version selection to as late in the process as possible.
5. Add a link from the Best Practices IG (`FHIR/ig-guidance`) to the core spec `index.html` page (`https://build.fhir.org/ig/FHIR/ig-guidance/index.html` is the Best Practices IG's home).

The "policy" half of the original ticket (whether distinct parameters are needed for publication vs downstream validation, and rules for inheriting parameters from dependencies) was resolved by the points above: parameters apply only to the IG itself, and any conformance-relevant constraint (e.g., code-system version, `activeOnly`) must be expressed inside the value-set definition itself, not as an out-of-band expansion parameter.

## Feature Proposal

### Problem Statement

Today an IG can be built using value-set expansion parameters (code-system version pins, language, `activeOnly`, `displayLanguage`, etc.) that materially shape the rendered expansions, but those parameters are not surfaced in the published IG. Readers and downstream implementers cannot tell:

- Which parameters were used to build the IG's expansions.
- Whether those parameters were chosen for publishing convenience or as a conformance constraint on implementers.
- How to reproduce the published expansions (e.g., for testing).
- What happens when a dependent IG was built with a different set of parameters than the IG that depends on it.

This creates ambiguity about which terminology constraints implementers must honour, and silent drift when an IG inherits dependencies built with incompatible expansion parameters.

### Proposed Change

Three coordinated changes, all **outside the FHIR R6 core specification**:

1. **IG Publisher (`HL7/fhir-ig-publisher`, not in cache):**
   - When packaging an IG, copy the `Parameters` resource that drove the build (the one passed via `parameters` in `sushi-config.yaml` / `ig.ini` / `ImplementationGuide.definition.parameter` and any expansion-parameter resource referenced from it) into the IG's output folder **and** include it inside the NPM package tarball alongside the other conformance resources.
   - Generate an HTML fragment (e.g., `expansion-parameters.xhtml` or similar) that renders the contents of that `Parameters` resource in human-readable form (parameter name, value, source — defaulted vs explicitly set), and make it available to IG authors as an includable fragment (akin to existing publisher-generated fragments for resource lists, dependency tables, etc.).
   - Cause the IG Publisher to fail / warn when an HL7-realm IG is published without that fragment being rendered somewhere in the IG (see policy point 4 below).

2. **Best Practices IG (`FHIR/ig-guidance`, briefing not loaded — see Open Questions):**
   - Add a new policy section (or extend an existing terminology / publication-policy page) stating, in HL7-normative language for HL7 IGs:
     - The expansion parameters used during IG publication apply **only** to the IG's own published expansions. They are not, by themselves, a conformance statement to downstream implementers.
     - If an IG editor wants to constrain implementers (e.g., to a specific code-system version or to `activeOnly = true`), that constraint **must** be expressed in the value-set definition itself (e.g., `ValueSet.compose.include.version`, an extension on the `ValueSet`, or accompanying narrative), not left implicit in expansion parameters.
     - HL7 IGs **must** render the auto-generated expansion-parameters fragment somewhere in the IG.
     - HL7 recommends IGs use expansion parameters sparingly and delegate code-system-version selection to as late in the process as possible (i.e., let implementers pick the version unless there is a real interoperability reason to pin).
     - When an IG depends on other IGs that were built with different expansion parameters, the depending IG's parameters apply to its own value-set expansions; conflicts must be resolved by the editor at design time, not silently inherited.
   - Add a link from `index.html` (or the natural landing page) of `FHIR/ig-guidance` to the FHIR core spec's `index.html`, per the resolution text.

3. **(No change to HL7/fhir core spec.)** The resolution explicitly scopes the change "outside of the FHIR core specification". The only core-spec-adjacent action implied is the inbound link **from** the Best Practices IG **to** core spec `index.html`, which is an edit to `FHIR/ig-guidance`, not to `HL7/fhir`.

### Design Rationale

- **Why the value-set, not the expansion parameters, carries conformance constraints:** during the 2025-08-04 TI tracker discussion (comment by carmela_couderc, summarising consensus), the group concluded that expansion parameters are a build-time tooling concern; if a parameter value (e.g., a SNOMED edition or `activeOnly`) is conformance-relevant, it belongs in the `ValueSet` definition where validators and downstream tools can see it without having to consult the IG's build configuration. This avoids creating a parallel conformance channel ("hidden in build params") that validators would not naturally honour.
- **Why surface the parameters anyway:** even when not conformance-bearing, the parameters are necessary for reproducibility and testing. Shipping the `Parameters` file in the package and rendering a human-readable fragment lets implementers reproduce the exact expansions the IG was built against.
- **Why not inherit dependency parameters automatically:** the second part of the original ticket ("not currently examining the differences between expansion parameters between an IG and the IGs it depends on") was resolved per the September-2025 motion (marc.duteau): "FHIR-I will handle both the task of surfacing expansion parameters and the task of not inheriting expansion parameters." Auto-inheritance was explicitly rejected because dependent IGs may legitimately need different expansion contexts.
- **Why the Best Practices IG is the right home for the policy:** TI's July-2025 disposition (mckenzie comment) said TI lands the *policy*; FHIR-I handles the *rendering*. The HL7-wide normative wording for IG authors lives most naturally in `FHIR/ig-guidance` (the Best Practices IG), with the IG Publisher enforcing it mechanically.

### Breaking Changes

- **For IG authors (HL7 realm):** Once the IG Publisher enforces "must render the parameters fragment", existing HL7 IGs that do not render it will fail / warn at publish time. This is a behavioural breaking change for authors, but a non-breaking change for published artefacts and conformance.
- **For downstream consumers / validators:** No breaking changes. Adding the `Parameters` resource to the package is additive; existing tooling that ignores unknown package contents is unaffected.
- **For value-set authors:** The policy clarifies (does not change) where conformance-relevant constraints belong; authors who were *implicitly* relying on expansion parameters to constrain implementers should now move those constraints into the value-set definition. This is a soft breaking change in expectations, not in machine-readable artefacts.

### Related Specifications

- **`Parameters` resource** (FHIR core) — the artefact being shipped and rendered. No change to its definition; this work just gives it a new lifecycle role inside an IG package.
- **`ValueSet` resource** (FHIR core) — relevant because the policy says conformance-bearing constraints (code-system version, `activeOnly`, etc.) belong in `ValueSet.compose` or in `ValueSet` extensions rather than in the IG's expansion parameters. No change to the `ValueSet` resource itself is required.
- **`ImplementationGuide` resource** (FHIR core) — the `definition.parameter` element is where the expansion parameters reference is currently captured; no change to the resource is implied, only to how the publisher uses it.
- **`$expand` operation** (terminology module) — the parameters being shipped are precisely those passed to `$expand`. The policy text in `FHIR/ig-guidance` should cross-link the relevant `$expand` parameters (e.g., `system-version`, `force-system-version`, `activeOnly`, `displayLanguage`).
