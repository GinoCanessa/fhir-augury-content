## Resolution Summary

**Resolution:** Persuasive.

Resolution description (verbatim): *"Will add an invariant saying that alternatives can't have alternatives in their descendant steps"*.

Concretely, a new FHIRPath constraint must be added to the `ExampleScenario.process.step.alternative` element such that none of the steps reachable through `alternative.step` (and any nested `.process.step` sub-processes within them) may themselves contain an `alternative`. The only outstanding implementation action called out in comments (mckenzie, 2025-11-06) is "make the invariant and tests".

## Feature Proposal

### Problem Statement

`ExampleScenario.process.step.alternative.step` is a content-reference back to `ExampleScenario.process.step`, which itself has an `alternative` child. The structure therefore permits arbitrarily deep nesting of alternatives-within-alternatives. The ticket notes two concerns with allowing this:

1. **Rendering:** renderers / diagrams for ExampleScenario become visually unmanageable once alternatives branch recursively.
2. **No use case:** authors have not produced a concrete need to model alternatives inside alternatives.

The specification is missing a constraint that prevents this recursion, so the schema currently allows content that the WG has agreed is not intended to be authored.

### Proposed Change

Add one new invariant to the ExampleScenario StructureDefinition, on the element `ExampleScenario.process.step.alternative`:

- **Key:** `exs-25` (next available after `exs-24`; in-clone invariants `exs-1`..`exs-24` are all used).
- **Severity:** `error`.
- **Human:** `Alternatives cannot contain further alternatives in their descendant steps`.
- **Expression:** `descendants().alternative.empty()`
  - Rationale: at the `alternative` context, `descendants()` traverses every element reachable through `alternative.step` (and its nested `process.step`). If any of those descendants has an `alternative` child, the instance violates the constraint. This catches both direct nesting (`step.alternative.step.alternative`) and indirect nesting via `step.alternative.step.process.step.alternative`.
- **XPath:** not authored (FHIR Core has moved away from authoring XPath for new invariants; none of the recent `exs-*` constraints include xpath, and the publisher derives what it needs from the FHIRPath).

Add a `<condition value="exs-25"/>` back-reference on `ExampleScenario.process.step.alternative.step` (the element whose contents the invariant constrains) for consistency with how existing invariants such as `exs-13` are back-referenced on `alternative`/`alternative.title` (lines 1116, 1130).

Add matching invariant-test files under `source/examplescenario/invariant-tests/`, following the established naming pattern (`exs-N.fN.fail.xml` for instances that must fail, `exs-N.pN.pass.xml` for instances that must pass):

- `exs-25.f1.fail.xml` — minimal instance with `step.alternative.step.alternative` (one-level nested alternative-in-alternative).
- `exs-25.f2.fail.xml` — alternative nested one level deeper, via `step.alternative.step.process.step.alternative`, to confirm the `descendants()` traversal catches the sub-process path, not just the direct contentReference path.
- `exs-25.p1.pass.xml` — instance with a populated `step.alternative.step` that contains only `operation`/`process`/`workflow` content and no `alternative` (must pass).

### Design Rationale

- **Invariant over schema restructure.** The resolution is explicit: an invariant, not a structural change. Restructuring `alternative.step` to reference a trimmed version of `process.step` without `alternative` would be a breaking change to the content model and is out of scope.
- **`descendants().alternative.empty()`.** This is the simplest FHIRPath that enforces the intent. It matches the style of other ExampleScenario invariants (e.g., `exs-13`'s `alternative.title.count() = alternative.title.distinct().count()` — short, FHIRPath-only, no xpath).
- **Test coverage.** The existing `invariant-tests/` directory is the mechanism the bespoke publisher uses to validate that an invariant both fires when it should and does not fire when it shouldn't. The WG's "tests" ask from the comment is addressed by adding the f/p files listed above.
- **Severity error vs warning.** All sibling exs-* invariants on `process.step` and `alternative` (exs-13, exs-22) use `error`. A warning would not prevent rendering breakage, so `error` is consistent with WG precedent and with the ticket's framing ("can't have").

## Implementation Plan

### Prerequisites

- None. The invariant uses only core FHIRPath (`descendants()`, `empty()`), which is supported by the FHIR validator and publisher already in use.
- Before landing, run a grep across `source/examplescenario/` example files to confirm no shipped example instance nests alternatives; fix any that do first (under the same PR, if found).

### Step-by-Step Tasks

#### HL7/fhir

1. **Sanity-check shipped ExampleScenario examples.**
   - File(s): `source/examplescenario/examplescenario-example.xml`, `source/examplescenario/examplescenario-example-laborder.xml`, and any other `examplescenario-example*.xml` under `source/examplescenario/`.
   - Action: Grep/inspect to confirm that no `<alternative>` element contains a descendant `<alternative>` (directly under its `<step>` or via nested `<step><process><step><alternative>`).
   - Details: If a shipped example does nest alternatives, restructure it first (typically by flattening the inner alternative into a sibling step of the outer alternative) and note the restructure in the PR description. This avoids `exs-25` regressing the publish build.

2. **Add the exs-25 invariant to the ExampleScenario StructureDefinition.**
   - File: `source/examplescenario/structuredefinition-ExampleScenario.xml`
   - Action: Insert a new `<constraint>` block inside the `ExampleScenario.process.step.alternative` element (currently defined at lines 1106–1117), immediately after the existing `<condition value="exs-13"/>` and before `</element>`.
   - Details (authoring XML):
     ```xml
     <constraint>
       <key value="exs-25"/>
       <severity value="error"/>
       <human value="Alternatives cannot contain further alternatives in their descendant steps"/>
       <expression value="descendants().alternative.empty()"/>
     </constraint>
     ```
   - Notes: Do not touch any `<snapshot>` block; only the `<differential>` section (which ends at line ~1164) is hand-edited. Preserve existing whitespace/indentation conventions used for the sibling `exs-13`/`exs-22` constraint blocks.

3. **Back-reference the invariant on `alternative.step`.**
   - File: `source/examplescenario/structuredefinition-ExampleScenario.xml`
   - Action: On `ExampleScenario.process.step.alternative.step` (currently at lines 1145–1153), add `<condition value="exs-25"/>` before the closing `</element>` tag.
   - Details: This mirrors the back-reference pattern already used (e.g., `<condition value="exs-13"/>` on `alternative` at line 1116 and on `alternative.title` at line 1130). The back-reference is informational but is how the ExampleScenario resource consistently tags elements whose content is constrained by a sibling invariant.

4. **Add a failing fixture: direct nesting via `alternative.step.alternative`.**
   - File: `source/examplescenario/invariant-tests/exs-25.f1.fail.xml` (new)
   - Action: Create a minimal ExampleScenario instance (model the existing `exs-13.f1.fail.xml` for boilerplate) containing a single `<process>` with one `<step>` that has an `<alternative>` whose `<step>` in turn contains another `<alternative>`.
   - Details: Name it `<ExampleScenario>` with `id`, `url`, `name`, `status`, `purpose`, one `<actor>`, one `<instance>` as required by other invariants (to avoid unrelated exs-* failing). The nested structure under `<process>` should be:
     ```xml
     <step>
       <number value="1"/>
       <operation><title value="Outer"/><initiator value="A"/><receiver value="B"/></operation>
       <alternative>
         <title value="Alt 1"/>
         <step>
           <number value="1a"/>
           <operation><title value="Inner op"/><initiator value="A"/><receiver value="B"/></operation>
           <alternative>           <!-- triggers exs-25 -->
             <title value="Nested alt"/>
           </alternative>
         </step>
       </alternative>
     </step>
     ```
   - Notes: Ensure the file satisfies exs-13/exs-22 and other existing invariants so the only failure the publisher reports is exs-25.

5. **Add a failing fixture: indirect nesting via `alternative.step.process.step.alternative`.**
   - File: `source/examplescenario/invariant-tests/exs-25.f2.fail.xml` (new)
   - Action: Same boilerplate as `f1`, but the inner alternative is reached through a sub-process (`step.alternative[0].step[0].process.step[0].alternative[0]`), exercising the `descendants()` traversal through `process.step`.
   - Details: This proves the invariant catches the recursive path and not only the direct `alternative.step.alternative` path.

6. **Add a passing fixture: alternative with steps but no nested alternative.**
   - File: `source/examplescenario/invariant-tests/exs-25.p1.pass.xml` (new)
   - Action: Create an ExampleScenario instance with one `<step>` whose `<alternative>` contains one or two `<step>` entries that each have `<operation>` only (no nested `<alternative>`). The publisher must find no exs-25 violation.
   - Details: Confirms exs-25 does not over-fire on legitimate single-level alternatives — which is the normal case authored today.

### Testing Considerations

- **Revalidation scope:** only the ExampleScenario resource and its example instances need revalidating; no cascading snapshot regen is required beyond the normal publisher pass (which always rebuilds ExampleScenario's snapshot from the edited differential).
- **Edge cases to verify:**
  - Alternative whose `step` uses `<process>` with deeper nesting (f2 fixture) — confirms `descendants()` crosses the contentReference hop through `process.step`.
  - Alternative with many sibling steps, none of which have alternative — confirms no over-firing.
  - Empty alternative (no step) — should still pass exs-25 (trivially `descendants().alternative` is empty).
- **Interoperability:** servers/validators that re-run FHIRPath constraints against existing ExampleScenario instances will begin flagging previously-accepted nested-alternative content. Release notes for the build that lands this invariant should mention exs-25 explicitly so implementers know to flatten any pre-existing nested alternatives.
