## Resolution Summary

The committee disposition is **Persuasive with Modification**. The agreed
changes to the `UsageContext` metadata datatype are:

1. Add `canonical(ValueSet)` as an allowed type for `UsageContext.value[x]`,
   so a context can be expressed as "any code drawn from this value set"
   rather than enumerating one `UsageContext` per code.
2. Add documentation clarifying the interpretation of the `value[x]` choice:
   *"If a 'range' is specified, then the context means that the artifact
   applies for any value of the context that falls within range. If a
   'valueset' is specified, then the context means that the artifact
   applies if a code matches the value set."*
3. Add documentation to `UsageContext` stating: *"If multiple usageContexts
   are provided, there is no specific interpretation of 'and' or 'or'.
   I.e. if an age context and a gender context are provided, there is no
   implication that both age and gender must hold."*
4. Change the short/definition of `UsageContext.value[x]` from
   *"A value that defines the context specified in this context of use…"*
   to *"A value or set of possible values that defines the context …"*.

The committee accepted the use case but rejected the reporter's suggestion
to also add `Reference(PlanDefinition)` / `canonical(PlanDefinition)`
(per Grahame Grieve's comment "I don't follow where PlanDefinition fits in").
Only the value-set form is added.

## Feature Proposal

### Problem Statement

`UsageContext` lets an artifact (profile, knowledge module, etc.) declare
the context(s) in which it applies. Today `UsageContext.value[x]` is
typed as `CodeableConcept | Quantity | Range |
Reference(PlanDefinition|ResearchStudy|InsurancePlan|HealthcareService|Group|Location|Organization)`
(see the spreadsheet row in
`source/datatypes/usagecontext.xml`, line ~865).

The reporter's use case — additional bindings of the form *"if
`Observation.code` is from value set Q1, then
`Observation.valueCodeableConcept` shall be bound to value set A1"* —
requires putting "the set of codes Q1" into `UsageContext.value[x]`.
The current types only allow:

- A single `CodeableConcept` (one code), forcing implementers to author one
  `UsageContext` per code in Q1 (tedious and fragile, and the "or"
  semantics across rows are not actually defined — see point 3 above).
- A `Quantity` / `Range` (numeric only).
- A reference to a small fixed list of resources, none of which represent
  a generic value set.

There is no way to say "any code from this value set" in a single
`UsageContext`. The same gap shows up for jurisdiction sets and language
sets, both of which the reporter calls out as common.

### Proposed Change

Make four coordinated edits, all in `HL7/fhir`:

1. **Datatype: extend `UsageContext.value[x]` types.**
   - File: `source/datatypes/usagecontext.xml` (the legacy authoring
     spreadsheet for the UsageContext datatype).
   - Worksheet: `Data Elements`, row whose first cell is
     `UsageContext.value[x]` (around line 865 of the XML).
   - Change the `Type` cell (currently `CodeableConcept | Quantity | Range
     | Reference(PlanDefinition|ResearchStudy|InsurancePlan|HealthcareService|Group|Location|Organization)`)
     to add `canonical(ValueSet)`:
     `CodeableConcept | Quantity | Range | Reference(PlanDefinition|ResearchStudy|InsurancePlan|HealthcareService|Group|Location|Organization) | canonical(ValueSet)`.
   - Cardinality stays `1..1`. The publisher will pick up the new choice
     and regenerate `valueCanonical` in the StructureDefinition snapshot.

2. **Datatype: change the `value[x]` short and definition text.**
   - Same row in `source/datatypes/usagecontext.xml`.
   - Replace the short *"Value that defines the context"* with
     *"Value or set of possible values that defines the context"*.
   - Replace the definition *"A value that defines the context specified
     in this context of use. The interpretation of the value is defined by
     the code"* with
     *"A value or set of possible values that defines the context
     specified in this context of use. The interpretation of the value is
     defined by the code."*

3. **Page narrative: explain `range` vs `valueset` semantics on the
   UsageContext element.**
   - File: `source/metadatatypes.html`, UsageContext section starting at
     `<a name="UsageContext"></a>` (line 339).
   - Insert a paragraph immediately after the existing `<p>The
     UsageContext structure defines the context of use for a module.</p>`
     and before the `[%dt UsageContext 1%]` macro:

     ```html
     <p>
     The interpretation of <code>UsageContext.value[x]</code> depends on
     the type chosen:
     </p>
     <ul>
       <li>If a <code>Range</code> is specified, the context means that
       the artifact applies for any value of the context that falls
       within the range.</li>
       <li>If a <code>canonical(ValueSet)</code> is specified, the
       context means that the artifact applies if a code matches the
       value set.</li>
     </ul>
     ```

4. **Page narrative: clarify that multiple `UsageContext` entries are
   not implicitly conjoined.**
   - Same file/section as task 3, immediately after the bulleted
     interpretation list:

     ```html
     <p>
     If multiple <code>UsageContext</code>s are provided, there is no
     specific interpretation of "and" or "or". For example, if an age
     context and a gender context are both supplied, there is no
     implication that both the age and the gender must hold for the
     artifact to apply.
     </p>
     ```

The detailed-descriptions page (`source/metadatatypes-definitions.html`)
is rendered from the `<%dictionary UsageContext%>` macro and does not
need a hand edit — the publisher will regenerate the per-element table
from the updated spreadsheet.

### Design Rationale

- **`canonical(ValueSet)` over `Reference(ValueSet)`.** Value sets are
  canonical artifacts identified by URL+version; FHIR convention is to
  reference them with `canonical` (cf. additional bindings, profiles,
  questionnaire bindings). A literal `Reference` would be inconsistent
  and would not support versioned URLs.
- **Mirrors `Quantity`/`Range`.** Grahame's comment notes "We already
  have valueQuantity/valueRange, so it's the same pattern" — adding
  `valueCanonical` for value sets follows the existing precedent of
  expressing a *range* of acceptable values rather than a single point.
- **Documentation of `and`/`or` semantics.** The reporter implicitly
  assumed multiple `UsageContext` entries combine with "or"; existing
  guidance is silent. The committee chose to make the silence explicit
  rather than retrofit a binding decision that would invalidate
  in-the-wild artifacts.
- **PlanDefinition reference rejected.** Mike McKenzie raised
  `Reference(PlanDefinition)` / `canonical(PlanDefinition)` as a
  possibility; Grahame pushed back ("I don't follow where PlanDefinition
  fits in?"). The accepted resolution does not include it, so it is out
  of scope for this ticket.

### Breaking Changes

Adding `canonical(ValueSet)` to a `value[x]` choice element is **not
backward incompatible** for existing instances: every previously-valid
`UsageContext` (using `valueCodeableConcept`, `valueQuantity`,
`valueRange`, or one of the listed `valueReference` targets) remains
valid. Validators and code generators will need to recognise the new
`valueCanonical` slot, but no existing data becomes invalid.

The definition-text change is editorial; the short label change is
editorial.

**Net assessment:** No breaking changes for instance authors. Tooling
that pattern-matches the exact set of `value[x]` types (e.g.,
hand-rolled validators, code generators with hard-coded type lists,
profile authors who slice on `value[x]`) will need to be updated to
handle `valueCanonical`.

### Related Specifications

`UsageContext` is referenced (via `[%dtusage UsageContext%]`) by every
canonical-resource that carries `useContext`, including but not limited
to: `StructureDefinition`, `ValueSet`, `CodeSystem`,
`ConceptMap`, `CapabilityStatement`, `OperationDefinition`,
`SearchParameter`, `MessageDefinition`, `GraphDefinition`,
`ImplementationGuide`, `NamingSystem`, `TerminologyCapabilities`,

The `metadatatypes-definitions.html` rendering is regenerated by the
