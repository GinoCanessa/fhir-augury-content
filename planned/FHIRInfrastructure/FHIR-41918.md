## Resolution Summary

**Resolution:** Persuasive with Modification.

The resolution clarifies that the presence of one or more `targetProfile` values
on a `Reference` or `canonical` element does **not** imply that the reference
will contain enough information to resolve. The approved change is to add the
following Comments text to `ElementDefinition.targetProfile` (i.e., the
`ElementDefinition.type.targetProfile` element in the ElementDefinition
data-type definition), verbatim from the resolution:

> "The presence of one or more targetProfiles for a Reference or canonical
> **does not** mean that the Reference or canonical will contain enough
> information to allow resolution.  E.g. A Reference might just have
> `.display`, or a canonical might have an extension in place of a value.
> The expectation with the targetProfile is that if the element **does**
> resolve to a resource, that resource SHALL comply with at least one of
> the listed targetProfile options."

No structural change (cardinality, type, binding, or invariant) is required.

## Feature Proposal

### Problem Statement

The current definition of `ElementDefinition.type.targetProfile` leaves
implementers unsure whether declaring a `targetProfile` creates a resolution
obligation — i.e., whether a Reference/canonical carrying a `targetProfile`
must actually resolve to an instance of that profile. This ambiguity was
raised on Zulip (`179190-united-states` → topic
"reference.identifier conformance",
https://chat.fhir.org/#narrow/stream/179190-united-states/topic/reference.2Eidentifier.20conformance),
where participants debated whether a profile-targeted reference that carries
only `.display` or `.identifier` (no `.reference`) violates conformance.

The existing `ElementDefinition.type.targetProfile` row in the
ElementDefinition spreadsheet has a populated "Definition" ("Used when the
type is 'Reference' or 'canonical', ... the content must conform to at least
one of them...") but an empty "Comments" cell, so there is nowhere the
resolution guidance currently lives.

### Proposed Change

Add the resolution-approved Comments text to `ElementDefinition.type.targetProfile`
in the ElementDefinition data-type spreadsheet. Concretely:

- **Element:** `ElementDefinition.type.targetProfile`
- **Change:** populate the (currently empty) Comments column for this element
  with the exact wording quoted in the Resolution Summary above.
- **Type / cardinality / binding / invariants:** unchanged. The existing
  invariants (`targetProfile is only allowed if the type is Reference or
  canonical`; `targetProfiles SHOULD be unique`) remain as-is.
- **Scope:** single cell edit in one file; no schema, no snapshot, no
  terminology changes.

Rendered-spec effect: the generated `elementdefinition.html` /
`elementdefinition-definitions.html` pages will display the new Comments
paragraph under the `ElementDefinition.type.targetProfile` entry,
clarifying the non-obligation of resolution.

### Design Rationale

- The resolution explicitly chose to add a **comment** rather than tighten
  conformance, because the existing semantics (a Reference may carry
  `.display` / `.identifier` without a resolvable `.reference`) are
  intentional and well-established; the ticket is a clarification, not a
  behavioural change.
- Placing the guidance on `ElementDefinition.type.targetProfile` (the
  normative definitional home of `targetProfile`) means every consumer of
  the spec — HTML pages, StructureDefinition snapshots, implementation
  guides — picks up the clarification automatically via the bespoke
  publisher, avoiding a manual fan-out to per-resource or per-profile pages.
- The phrasing matches the Zulip consensus captured in the ticket: profiles
  are a *conformance gate when resolved*, not a *resolution obligation*.
- Alternatives considered (inferred from the resolution wording): adding a
  new invariant, or tightening to require `.reference` when `targetProfile`
  is present. Both were rejected — they would be backward-incompatible and
  contradict existing practice (e.g., many profiles intentionally allow
  `.display`-only references).

  inside the string, or (c) left as plain prose and re-emphasised later in
  the narrative. The briefing does not prescribe which; resolve before
  merge to avoid a follow-up edit.
