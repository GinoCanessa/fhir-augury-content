## Resolution Summary

The resolution is **Persuasive**. The committee's resolution description reads:

> Will correct this example to use the revised text syntax and will include a reference to the text syntax.

Concretely, the unclear free‑form, dot‑indented pseudo example currently used to illustrate `$graph?graph=med-package` on the GraphDefinition page must be re‑written using the GraphDefinition **Text Representation** (the `node …; link … -> …;` short‑hand grammar that is documented further down on the same page), and a hyperlink (or anchor reference) to that Text Representation section must be added immediately around the example so a reader is no longer surprised by an undocumented notation.

## Feature Proposal

### Problem Statement

The GraphDefinition narrative (in `source/graphdefinition/graphdefinition-notes.xml`, "Fetching a graph of resources" sub‑section under "Using GraphDefinitions") shows what a hypothetical `med-package` GraphDefinition "would look approximately like" using an ad‑hoc indented dotted‑path notation:

```
MedicationDispense
  .subject
  .context
  .performer.actor
  .authorizingPrescription
     .requester.agent
  .substitution.responsibleParty
```

This notation is not introduced anywhere in the FHIR specification and is not the actual XML/JSON serialisation of a GraphDefinition. Readers have no reliable way to map the indentation back to GraphDefinition `node`/`link` constructs, and the example contradicts the rest of the page, which uses either real XML excerpts or the formally defined Text Representation grammar (`<h4>Text Representation</h4>`, anchor `#text`). The reporter (Bas van den Heuvel) explicitly asked that the notation either be introduced or replaced with a picture.

### Proposed Change

In the GraphDefinition narrative page (historically `source/graphdefinition/graphdefinition-notes.xml`, immediately around the `$graph?graph=med-package` example):

1. **Replace the ad‑hoc dotted example.** Remove the existing `<pre>` block:

   ```
   MedicationDispense
     .subject
     .context
     .performer.actor
     .authorizingPrescription
        .requester.agent
     .substitution.responsibleParty
   ```

   …and substitute an equivalent example written in the GraphDefinition Text Representation that the page already documents in the "Text Representation" section (anchor `#text`). The replacement should faithfully reproduce the same graph (`MedicationDispense` as the start node, with outgoing links via `MedicationDispense.subject` → Patient, `MedicationDispense.context` → Encounter, `MedicationDispense.performer.actor` → Practitioner/Organization, `MedicationDispense.authorizingPrescription` → MedicationRequest, then `MedicationRequest.requester.agent` → Practitioner, and `MedicationDispense.substitution.responsibleParty` → Practitioner). Use the canonical short‑hand the page itself defines, e.g.

   ```
   node start dispense 'The dispense being fetched' = MedicationDispense;
   node subj 'Subject of the dispense'              = Patient;
   node ctx  'Encounter context'                    = Encounter;
   node perf 'Performer (actor)'                    = Practitioner;
   node rx   'The authorizing prescription'         = MedicationRequest;
   node req  'The requester (agent)'                = Practitioner;
   node sub  'Substitution responsible party'       = Practitioner;

   link 'subject'                  = dispense[MedicationDispense.subject]                        -> subj;
   link 'context'                  = dispense[MedicationDispense.context]                        -> ctx;
   link 'performer'                = dispense[MedicationDispense.performer.actor]                -> perf;
   link 'authorizingPrescription'  = dispense[MedicationDispense.authorizingPrescription]        -> rx;
   link 'requester'                = rx[MedicationRequest.requester.agent]                       -> req;
   link 'substitution party'       = dispense[MedicationDispense.substitution.responsibleParty]  -> sub;
   ```

   The exact node descriptions and the resource type for `actor`/`agent`/`responsibleParty` (which are `Reference(...)` choices in the underlying StructureDefinition) should be set to whichever single choice is consistent with the rest of the example narrative; the goal is illustrative, not normative.

2. **Add an introducing reference to the Text Representation section.** Immediately before the new `<pre>` block, change the existing introductory sentence so that it both names the syntax and links to its definition. Suggested wording:

   > In this case, written using the GraphDefinition <a href="#text">Text Representation</a>, the graph definition would look approximately like this:

   The anchor `#text` already exists on the same page (the `<a name="text"></a>` immediately above `<h4>Text Representation</h4>`), so no new anchor is required.

3. **No schema/StructureDefinition changes.** The change is documentation‑only — `structuredefinition-GraphDefinition.xml`, the search‑params bundle, the operation definitions, and the example GraphDefinition resources are unaffected.

### Design Rationale

- The fix matches the resolution verbatim: use the *revised text syntax* (the formally defined `node`/`link` short‑hand) and *include a reference to the text syntax* (a hyperlink to the `#text` anchor).
- Re‑using a notation that is already defined on the same page is preferable to introducing a brand‑new informal notation or to embedding a generated image, because (a) the Text Representation is normative within GraphDefinition's own scope, (b) it stays usable in the published HTML, search indexes and accessibility tooling, and (c) it doesn't require new image generation in the publisher pipeline.
- An alternative — replacing the example with a rendered SVG diagram — was implicitly suggested by the reporter ("introduce the notation or use a picture"), but is heavier (requires authoring + maintaining an SVG, regenerating on each change) and still wouldn't introduce the notation that is used elsewhere in the page. The Text Representation already serves the introductory purpose and aligns with the page's other examples.

### Step-by-Step Tasks

#### HL7/fhir (or its successor host for GraphDefinition)

1. **Replace the unclear `med-package` `<pre>` example with the Text Representation form.**
   - File: `source/graphdefinition/graphdefinition-notes.xml` (historical path; retarget to the resource's new home if it has moved out of `HL7/fhir`).
   - Action: Inside the "Fetching a graph of resources" sub‑section (anchor `#fetch`), locate the `<pre>` block that begins with `MedicationDispense\n  .subject\n  .context\n  .performer.actor\n  .authorizingPrescription\n     .requester.agent\n  .substitution.responsibleParty` and replace it with an equivalent example written using the GraphDefinition Text Representation already defined later on the same page (`<h4>Text Representation</h4>`, anchor `#text`).
   - Details: Use `node start dispense … = MedicationDispense;` plus one `node` per linked resource (subject/context/performer/authorizingPrescription/requester/substitution responsibleParty), followed by one `link '<description>' = source[<MedicationDispense|MedicationRequest>.<path>] -> target;` per outgoing edge. Preserve the full set of paths from the original example so the example remains semantically identical. Do **not** introduce search parameters or compartment rules — those are out of scope for the example's intent.

2. **Update the sentence that introduces the example to name and link to the syntax.**
   - File: `source/graphdefinition/graphdefinition-notes.xml` (same caveat as task 1).
   - Action: Edit the introducing sentence (currently: *"In this case, the graph definition would look approximately like this:"*) so that it explicitly names the Text Representation and contains a hyperlink to the existing `#text` anchor.
   - Details: Suggested replacement: *"In this case, written using the GraphDefinition <a href="#text">Text Representation</a>, the graph definition would look approximately like this:"*. Keep wording short; the goal is to (a) name the notation and (b) provide a click‑through to its formal definition, both of which the resolution explicitly requires.

3. **Sweep the page for residual references to the old notation.**
   - File: `source/graphdefinition/graphdefinition-notes.xml`.
   - Action: Search the file for any remaining dot‑indented pseudo notation and either rewrite it in the Text Representation or remove it. Confirm the page contains no other `<pre>` blocks that still rely on the unintroduced syntax.
   - Details: Only the `med-package` example was flagged in the ticket; this sweep is defensive to ensure no second instance remains uncited.
