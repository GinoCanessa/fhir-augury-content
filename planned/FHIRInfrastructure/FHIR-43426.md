## Problem Statement

When viewing the Detailed Description tab on any profile (e.g., in US Core or C-CDA IGs), the **differential elements table** renders the upper cardinality as `?` for elements whose cardinality has not been changed from the parent specification. The conformance rules do not explain what this symbol means, and users may confuse it with the `?!` modifier element flag.

### Examples Cited

- [US Core AllergyIntolerance definitions](http://build.fhir.org/ig/HL7/US-Core/StructureDefinition-us-core-allergyintolerance-definitions.html) — Differential Elements Table tab
- [C-CDA AllergiesAndIntolerancesSection definitions](http://hl7.org/cda/us/ccda/2024Jan/StructureDefinition-AllergiesAndIntolerancesSection-definitions.html) — Differential Elements Table tab

## Resolution Description

> Will provide an explanation of what the `?` means here: <https://build.fhir.org/ig/FHIR/ig-guidance/readingIgs.html> (which is what IGs should be pointing to for help in understanding profile rendering).
>
> Specifically, `?` in a differential means that **the cardinality has not been changed from whatever it was in the parent specification**.

## Feature Proposal

Add documentation to the **readingIgs** page in the `FHIR/ig-guidance` IG to explain the meaning of `?` in cardinality columns of differential element views. The explanation should:

1. Clarify that `?` appears **only** in differential views (not snapshot/key elements/must-support views).
2. Define that `?` means "the cardinality (minimum, maximum, or both) has not been constrained and remains whatever it was in the parent specification."
3. Distinguish `?` (unchanged cardinality marker) from `?!` (modifier element flag) to prevent confusion — which was the original reporter's concern.

## Detailed Implementation Plan

### Step 1: Add `?` cardinality explanation to the Cardinality column description

**File:** `FHIR/ig-guidance` → `input/pagecontent/readingIgs.xml`

**Location:** Inside the `<table class="grid">` that describes the table columns, within the `Card.` row (approximately the section starting with `<td>Card.</td>`).

**Current text:**
```xml
<tr>
  <td>Card.</td>
  <td>Cardinality: the lower and upper bounds on how many times this element is allowed to appear
      within the containing model or element.  In some <a href="#sdviews">views</a>, cardinality
      may be greyed out, indicating that they are unchanged from the reference model for that view.</td>
</tr>
```

**Proposed text:**
```xml
<tr>
  <td>Card.</td>
  <td>Cardinality: the lower and upper bounds on how many times this element is allowed to appear
      within the containing model or element.  In some <a href="#sdviews">views</a>, cardinality
      may be greyed out, indicating that they are unchanged from the reference model for that view.
      In the <b>differential</b> view, a <code>?</code> in the minimum or maximum cardinality
      position indicates that the cardinality has not been changed from whatever it was in the
      parent specification.  For example, <code>1..?</code> means the minimum cardinality has
      been set to 1 but the maximum is unchanged, while <code>?..?</code> means neither bound
      has been constrained.  Note: <code>?</code> in a cardinality value is distinct from the
      <code>?!</code> flag, which indicates a
      <a href="{{site.data.fhir.path}}conformance-rules.html#isModifier">modifier element</a>.</td>
</tr>
```

### Step 2 (Optional): Add `?` to the Differential view description

**Location:** In the `<ul>` describing the model views, within the `<li><b>Differential</b>` item.

**Current text excerpt:**
> Element properties that are unchanged from the parent model are partially opaque, looking greyish.

**Proposed addition** (append after "looking greyish"):
> Similarly, cardinality values that are unchanged from the parent model are shown as `?`.
