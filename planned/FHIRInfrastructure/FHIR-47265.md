## Feature Proposal

### Problem Statement

Implementers regularly need to combine multiple `Request` resources into a
single logical order (a "prescription", "requisition", or other compound
request). Authoritative guidance for how to do this — using shared
`identifier`, `groupIdentifier`, or `RequestOrchestration` — already lives on
the Request pattern page in section 12.4.10 (`request.html#compound`,
"Compound Requests"). Today most individual Request-resource pages do not
point readers at that section, so each resource is read in isolation and
implementers re-discover (and inconsistently re-invent) compound-request
patterns. This ticket closes that gap by adding a forward link to
`request.html#compound` from every Request-pattern resource page.

### Proposed Change

For every FHIR resource that implements the **Request pattern** (i.e. the
resources that carry a `<resource>-request-mapping-exceptions.xml` mapping
file under `source/<resource>/`), insert a short paragraph in that resource's
introduction (or notes) page pointing to the Compound Requests section of the
Request pattern.

Use the wording already in place on `MedicationRequest` and `SupplyRequest`
(adapted per resource so the leading sentence reads naturally for that
resource's domain — "single medication", "single item", "single test", etc.):

> The {Resource} resource allows requesting only a single
> {medication / test / supply / item / appointment / …}. If a workflow
> requires requesting multiple items simultaneously, this is done using
> multiple instances of this resource. These instances can be linked in
> different ways, depending on the needs of the workflow. For guidance, refer
> to [the Request pattern](request.html#compound).

Where a resource page already has a paragraph that discusses single-vs-multi
ordering (e.g. SupplyRequest), only the trailing sentence with the
`request.html#compound` link needs to be added; do not duplicate the
preceding sentences.

### Design Rationale

- Mirrors the wording already in MedicationRequest / SupplyRequest, keeping
  authoring consistent across the canonical Request set.
- Uses a relative HTML link (`request.html#compound`) so the publisher
  resolves it correctly across all build outputs and ballots, matching the
  existing usage on the two resources that already have the link.
- Edits target the per-resource `*-introduction.xml` (or `*-notes.xml`)
  files under `source/<resource>/`, which the bespoke HL7 publisher reads as
  authored prose — consistent with how MedicationRequest and SupplyRequest
  carry their version of this paragraph today.
- A pure prose addition keeps this ticket squarely "Persuasive": no
  StructureDefinition or schema changes, no breaking impact.
