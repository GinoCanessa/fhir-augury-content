## Feature Proposal

### Problem Statement

At the time the ticket was filed (2024-09-06), `source/testing.html` was the
canonical FHIR "Testing FHIR" page in the FHIR Core specification. It
described the TestScript resource and its execution lifecycle in some
depth, but:

- It contained **no visual artifact** (no embedded SVG / image, no
  flowchart, no UML diagram). The reporter argued that readers struggle to
  understand how TestPlan, TestScript, and the surrounding testing
  artifacts relate to each other from prose alone.
- It under-represented the other testing-adjacent canonical resources
  introduced in R5 / R6 development — **ActorDefinition** (defines actors
  participating in testing scenarios), **Requirements** (defines what a
  test is verifying), and **TestPlan** (planning layer above TestScript).
  The page read primarily as a TestScript guide.

The combined gap — no diagram, plus narrow scope — is what the resolution
agreed to fix.

### Proposed Change

Two coordinated edits to the FHIR Testing page (historically
`source/testing.html`):

1. **Add a relationship diagram.** Insert a new SVG (preferred — matches
   the per-resource `*.svg` convention used throughout `source/`, e.g.
   `source/requirements/requirements.svg`,
   `source/observation/observation.svg`) that shows the relationships
   among:
   - **Requirements** — what is being verified
   - **ActorDefinition** — who/what participates
   - **TestPlan** — the planning layer (groups TestScripts, references
     Requirements and ActorDefinitions)
   - **TestScript** — the executable layer (concrete operations and
     assertions)
   - **TestReport** — the result of executing a TestScript
   - Supporting cross-links to **CapabilityStatement**, **ExampleScenario**,
     and **PlanDefinition** where they intersect with testing
   The SVG should sit alongside the page (e.g. `source/testing.svg`) and
   be referenced from a new "Resource Relationships" subsection placed
   near the top of the page, before the existing TestScript-specific
   workflow content.

2. **Expand the prose.** Rewrite / augment the introductory section so
   that it walks the reader through the four canonical resources
   (Requirements → ActorDefinition → TestPlan → TestScript) and then
   funnels into the existing TestScript execution workflow content. New
   sections (or subsections) needed:
   - "Testing Artifacts in FHIR" — short overview paragraph per resource
     (Requirements, ActorDefinition, TestPlan, TestScript, TestReport)
     with `<a href="...">` links to each resource page.
   - The diagram referenced in (1).
   - A short "How they fit together" paragraph that mirrors the diagram
     in narrative form.

The existing TestScript-specific subsections (Execution / Workflow /
Fixtures / Profiles / Variables / Operations / Assertions / Examples) are
retained unchanged.

### Design Rationale

- **SVG over raster image.** Every per-resource diagram in
  `source/<resource>/<resource>.svg` is SVG. The publisher already serves
  SVG natively from authored `source/*.svg` files (e.g.,
  `source/resources.svg`, `source/uml-example.svg`,
  `source/subscription-registration-rest.svg`), so adding
  `source/testing.svg` reuses an established pattern.
- **Diagram near the top.** The reporter's stated need is that a reader
  who lands on the page should *first* see how the testing resources
  relate. Placing the diagram and the new artifact overview before the
  existing TestScript-only workflow keeps the original deep-dive intact
  while satisfying the resolution.
- **Don't restructure existing TestScript content.** The resolution
  asks for *expansion*, not rewriting. Keeping the existing Execution /
  Fixtures / Profiles / Variables / Operations / Assertions sections
  unchanged minimises risk of breaking inbound anchor links from other
  pages and external IGs.
- **Alternative considered: per-resource diagram embedded in each
  resource's intro.** Rejected by the resolution wording — the
  diagram is requested on the *Testing* page so it shows the
  relationships, not on the individual resource pages.

         <li><a href="actordefinition.html">ActorDefinition</a> — the

         <li><a href="testplan.html">TestPlan</a> — the planning

           Requirements and ActorDefinitions.</li>
         <li><a href="testscript.html">TestScript</a> — the
