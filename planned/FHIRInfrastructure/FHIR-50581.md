## Resolution Summary

Resolution type: **Persuasive**.

The ticket reports that the page discussing implementation obligations speaks about an `obligation` "feature" / "element" of `ElementDefinition`, but no such element exists on `ElementDefinition`. Obligations on an `ElementDefinition` are expressed via the **Obligation Extension** (`http://hl7.org/fhir/StructureDefinition/obligation`), not a built-in ElementDefinition property. The two offending phrases — "`obligation` features of `ElementDefinition`" and "using the `obligation` element" — must be rewritten to refer to the Obligation Extension instead.

## Feature Proposal

### Problem Statement

In the FHIR Core specification (the `obligations.html` "Implementation Obligations" page, rendered on the ballot as `elementdefinition.html`-adjacent content), two sentences incorrectly describe obligations as if they were a built-in feature/element of `ElementDefinition`:

1. Introductory paragraph — "Implementation guides can make rules about the behavior of applications by using the `mustSupport` and `obligation` features of `ElementDefinition` in profiles." (cache clone: `source/obligations.html:27-28`).
2. MustSupport bullet list — "Providing detailed expectations about the element using the `obligation` element" (cache clone: `source/obligations.html:61`).

Neither `obligation` nor "obligation element" exists on `ElementDefinition`. The correct mechanism is the standalone **Obligation Extension** (`StructureDefinition/obligation`), which is already referenced correctly further down the same page (line 90: `<%extension http://hl7.org/fhir/StructureDefinition/obligation%>`) and on `elementdefinition.html` (line 561). These two lingering phrases are therefore both factually wrong and inconsistent with the rest of the page.

This is a documentation-only technical correction; no normative behaviour, StructureDefinition, ValueSet, CodeSystem, or search parameter is affected.

### Proposed Change

Edit `source/obligations.html` in `HL7/fhir` and replace each incorrect phrase with a reference to the Obligation Extension, reusing the publisher's existing extension-link template so the rendered text matches the style already used at line 90 of the same file and line 561 of `elementdefinition.html`.

Concrete edits:

1. **Line 27-29** — change the opening sentence so `mustSupport` remains an ElementDefinition feature but `obligation` is separated out as an extension:

   Before:
   ```html
   <p>
   Implementation guides can make rules about the behavior of applications by using the <code>mustSupport</code>
   and <code>obligation</code> features of <a href="elementdefinition.html">ElementDefinition</a> in profiles.
   </p>
   ```

   After:
   ```html
   <p>
   Implementation guides can make rules about the behavior of applications by using the <code>mustSupport</code>
   feature of <a href="elementdefinition.html">ElementDefinition</a> and the
   <%extension http://hl7.org/fhir/StructureDefinition/obligation%> extension in profiles.
   </p>
   ```

2. **Line 61** — change the bullet list item to reference the Obligation Extension:

   Before:
   ```html
   <li>Providing detailed expectations about the element using the <code>obligation</code> element</li>
   ```

   After:
   ```html
   <li>Providing detailed expectations about the element using the
   <%extension http://hl7.org/fhir/StructureDefinition/obligation%> extension</li>
   ```

No other content on `obligations.html` needs to change; the `<h3>Obligations</h3>` section starting at line 85 already uses the extension template correctly.

### Design Rationale

- The `<%extension ...%>` publisher macro is already the established idiom in this page (line 90) and on `elementdefinition.html` (line 561) to render a hyperlink to the extension's published page; using it again keeps prose consistent and ensures the generated anchor tracks the extension's canonical URL.
- `mustSupport` is genuinely a property on `ElementDefinition`, so it is preserved as a feature of `ElementDefinition`; only the false pairing with `obligation` is removed.
- Inlining a plain text "Obligation Extension" string would bypass the publisher's extension-link rewriting and risk a dead link when the extensions pack's URL layout changes. Using the macro is the safer, more maintainable fix.
- No alternative was seriously considered because the ticket itself states the required direction ("This should instead refer to the Obligation Extension") and there is no Zulip/cross-reference discussion proposing anything else (cross-referenced search returned 0 hits).
