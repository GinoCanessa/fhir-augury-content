## Resolution Summary

Resolution: **Persuasive with Modification**.

Resolution description (verbatim):

> We will change the cardinality to 0..1. Will add a meaningIfMissing of
> "The mechanism by which group membership is determined is unknown or
> cannot be expressed using one of the standard codes."

Concretely, `Group.membership` is relaxed from `1..1` to `0..1`, and a
`meaningWhenMissing` (ElementDefinition's element for the FHIR
"meaningIfMissing" concept) is added so that an absent value carries a
well-defined interpretation. The required binding to
`group-membership-basis` is retained; the escape-valve behaviour comes
from omission (combined with `meaningWhenMissing`) rather than a new
code in the ValueSet.

## Feature Proposal

### Problem Statement

`Group.membership` is currently a mandatory element (`min=1`) with a
required binding to `ValueSet/group-membership-basis` (codes:
`definitional`, `conceptual`, `enumerated`). The bound ValueSet has no
`unknown` / `other` code, so in cases where none of the three codes
applies — or where the authoring system does not know which applies —
the containing `Group` cannot be validly populated. The reporter (Lloyd
McKenzie, comment 131) asked for an explicit escape valve, either via a
new concept in the ValueSet or another mechanism.

### Proposed Change

Apply the approved resolution in `HL7/fhir`, authoring root
`source/group/`:

1. **Relax cardinality of `Group.membership` to `0..1`.**
   - File: `source/group/structuredefinition-Group.xml`
   - Element `Group.membership` (id `Group.membership`, currently at
     lines 427–447 of the differential).
   - Change `<min value="1"/>` → `<min value="0"/>` (line 432).
     `<max value="1"/>` is unchanged.
   - Keep the existing required binding to
     `http://hl7.org/fhir/ValueSet/group-membership-basis` and the
     `<condition value="grp-2"/>` reference.

2. **Add `meaningWhenMissing` to the same element.**
   - After the existing `<definition>`/`<requirements>` block (and
     before `<min>`, per ElementDefinition ordering where
     `meaningWhenMissing` appears later than `requirements`; the
     publisher re-orders as needed — place it immediately after
     `<requirements>`):

     ```xml
     <meaningWhenMissing value="The mechanism by which group membership is determined is unknown or cannot be expressed using one of the standard codes."/>
     ```

   - Do **not** add new codes to `ValueSet/group-membership-basis` or
     `CodeSystem/group-membership-basis` — the resolution uses
     `meaningWhenMissing` instead of an `unknown`/`other` code.

3. **Leave the ValueSet and CodeSystem unchanged.**
   - `source/group/valueset-group-membership-basis.xml` and
     `source/group/codesystem-group-membership-basis.xml` stay as-is.
     The escape valve is the absence of a value, not a new code.

No changes to the `Group.member` element (or any other element) are
required by the resolution.

### Design Rationale

- The resolution chose `meaningWhenMissing` over extending the
  required-bound ValueSet. This keeps the value-set closed to the three
  well-defined bases (`definitional`, `conceptual`, `enumerated`) and
  avoids introducing a weaselly code like `other`/`unknown` inside a
  coded set that is otherwise exhaustive by construction.
- Making the element optional with a `meaningWhenMissing` is a standard
  FHIR pattern for providing an "escape valve" without weakening a
  required binding.
- The invariant `grp-2` ("Group.type SHALL be defined if
  Group.membership is either 'definitional' or 'enumerated'") already
  uses the expression `type.exists() or (membership = 'conceptual')`.
  That expression remains well-defined when `membership` is absent (the
  equality evaluates empty/false, so the invariant requires
  `type.exists()` when `membership` is not populated), which is a
  reasonable behaviour — see Open Questions below for FMG confirmation.

### Testing Considerations

- **Resource revalidation:** publisher must revalidate `Group` and all

- **New negative example (optional but recommended):** a minimal

- **Edge cases:**
  - Instance with `membership` absent and `type` absent → should fail
    `grp-2` (because the expression requires `type.exists()` when
    `membership` is not `'conceptual'`). Confirm this is the intended
    behaviour with FMG.
  - Instance with `membership` absent and `type` present → should pass
    `grp-2`; treat as "unknown basis" per `meaningWhenMissing`.
- **Interoperability:** existing implementations that populate
  `membership` see no wire-format change. Implementations that consume
  `Group` must now handle the absent case; consider a brief mention in
  the resource's `group-notes.xml` if the WG wants implementer guidance
  (not required by the resolution).

### Open Questions

- **Invariant `grp-2` semantics when `membership` is absent.** The
  current expression `type.exists() or (membership = 'conceptual')`
  will require `type.exists()` whenever `membership` is missing, which
  aligns with the spirit of the invariant (only "conceptual" groups
  may omit `type`). Confirm with FHIR Infrastructure that this
  behaviour is intended, or whether the invariant should be updated to
  explicitly allow `membership.empty()` (e.g.,
  `type.exists() or membership.empty() or (membership = 'conceptual')`).
- **Narrative updates.** The resolution text does not mandate prose
  changes in `group-introduction.xml` or `group-notes.xml`. If any
  existing narrative asserts that "membership is always one of the
  three basis codes", it should be softened; a quick editorial pass is
  recommended during implementation.
- **Exemplar of the escape-valve case.** Whether to ship a new
  `group-example-*.xml` that demonstrates absent `membership` is a
  work-group editorial choice, not a resolution requirement.
