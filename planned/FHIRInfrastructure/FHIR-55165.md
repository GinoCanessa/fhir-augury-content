# Implementation Plan: FHIR-55165

**Title:** Group.membership has no escape valve
**Status:** Resolved - change required
**Resolution:** Persuasive with Modification
**Work Group:** FHIR Infrastructure
**Specification:** FHIR Core (FHIR)
**Resolved:** 2026-03-16

---

## Resolution Summary

The resolution changes the cardinality of `Group.membership` from `1..1` to `0..1` and adds a `meaningWhenMissing` value of: *"The mechanism by which group membership is determined is unknown or cannot be expressed using one of the standard codes."*

This approach avoids adding new codes (e.g., "unknown" or "other") to the `GroupMembershipBasis` CodeSystem/ValueSet and instead provides an escape valve by making the element optional. When the membership basis is omitted, the `meaningWhenMissing` text conveys that the basis is unknown or inexpressible.

## Feature Proposal

### Problem Statement

`Group.membership` is currently a mandatory element (`1..1`) with a required binding to the `GroupMembershipBasis` ValueSet, which contains only three codes: `definitional`, `conceptual`, and `enumerated`. There is no "unknown" or "other" concept in the ValueSet.

This means there is no valid way to populate a Group resource when the basis for membership is not known or does not fit one of the three defined concepts. Since the binding is `required`, no alternative codes can be used, and since the cardinality is `1..1`, the element cannot be omitted. This creates an impossible-to-populate situation for some real-world use cases.

### Proposed Change

1. **Change cardinality** of `Group.membership` from `1..1` to `0..1` in the StructureDefinition.
2. **Add `meaningWhenMissing`** text to the `Group.membership` element definition: *"The mechanism by which group membership is determined is unknown or cannot be expressed using one of the standard codes."*
3. **Update the `grp-2` invariant** to handle the case where `membership` is absent — the existing expression `type.exists() or (membership = 'conceptual')` would incorrectly require `type` when `membership` is omitted. The expression must be updated to also accept absent `membership`.
4. **Update narrative documentation** in `group-notes.xml` to reflect the optionality and explain when omission is appropriate.

### Design Rationale

Rather than adding "unknown" or "other" codes to the `GroupMembershipBasis` CodeSystem (which could weaken the semantics of the ValueSet and encourage imprecise coding), making the element optional with a `meaningWhenMissing` is a cleaner FHIR pattern. This approach:

- Preserves the precise semantics of the three existing codes
- Provides a clear "escape valve" for cases where membership basis is unknown
- Follows established FHIR patterns (e.g., `Group.experimental`, `Group.combinationMethod`, `Group.member.inactive` already use `meaningWhenMissing` in the same resource)
- Does not require terminology changes (no CodeSystem/ValueSet modification)

### Affected Repositories

#### HL7/fhir (FHIR Core Specification)
- **Role:** Primary specification repository containing the Group resource definition
- **Affected Files:**
  - `source/group/structuredefinition-Group.xml` — Change `min value="1"` to `min value="0"` on the `Group.membership` element (line 432); add `meaningWhenMissing` element; update `grp-2` invariant expression (line 94)
  - `source/group/group-notes.xml` — Update narrative around `Group.membership` optionality (lines 73–78) to note the element is now optional and explain the meaning when absent
- **Change Scope:** Minor

### Breaking Changes

- **Cardinality relaxation (1..1 → 0..1):** This is a **non-breaking** change for consumers — systems that always populate `membership` will continue to work. However, consumers that assume `membership` is always present will need to handle the absent case.
- **Invariant `grp-2` update:** The invariant expression change is technically a loosening (it accepts more instances), so this is **non-breaking** for data producers. Validators that hard-code the old expression will need to update.
- **No terminology changes:** The `GroupMembershipBasis` CodeSystem and ValueSet remain unchanged. No ValueSet expansion impact.

### Related Specifications

- **Group resource invariant `grp-2`:** Currently defined as `type.exists() or (membership = 'conceptual')`. Must be updated to also accept absent `membership`.
- **Search parameter `Group-membership`:** The search parameter on `Group.membership` (token type) will naturally handle 0..1 cardinality — searches simply won't match Groups where `membership` is absent. No changes needed.

## Implementation Plan

### Prerequisites

None. This is a standalone change to the Group resource definition.

### Step-by-Step Tasks

#### HL7/fhir

**Task 1: Change `Group.membership` cardinality from 1..1 to 0..1**
- **File:** `source/group/structuredefinition-Group.xml`
- **Location:** Line 432
- **Change:** Replace `<min value="1"/>` with `<min value="0"/>`

**Task 2: Add `meaningWhenMissing` to `Group.membership`**
- **File:** `source/group/structuredefinition-Group.xml`
- **Location:** After line 437 (after `<isSummary value="true"/>`)
- **Change:** Add:
  ```xml
  <meaningWhenMissing value="The mechanism by which group membership is determined is unknown or cannot be expressed using one of the standard codes."/>
  ```

**Task 3: Update invariant `grp-2` expression to handle absent membership**
- **File:** `source/group/structuredefinition-Group.xml`
- **Location:** Lines 91–94 (the `grp-2` constraint)
- **Change:** Update the FHIRPath expression from:
  ```
  type.exists() or (membership = 'conceptual')
  ```
  to:
  ```
  type.exists() or membership.exists().not() or (membership = 'conceptual')
  ```
- **Rationale:** The invariant currently requires `type` when `membership` is not `'conceptual'`. With `membership` now optional, an absent value should not force `type` to be required. The human-readable description should be updated to:
  *"Group.type SHALL be defined if Group.membership is either 'definitional' or 'enumerated'"* (no change needed — the existing human text already accurately describes the intent; only the expression needs updating).

**Task 4: Add invariant test for absent membership (grp-2 passing case)**
- **File:** `source/group/invariant-tests/grp-2.p5.pass.json` (new file)
- **Change:** Add a new passing test where `membership` is omitted entirely and `type` is also absent, verifying the updated invariant accepts this case:
  ```json
  {
    "resourceType": "Group",
    "id": "grp-2.p5.pass.json",
    "name": "Unknown membership group"
  }
  ```

**Task 5: Update narrative documentation for Group.membership optionality**
- **File:** `source/group/group-notes.xml`
- **Location:** Lines 73–78 (the `Group.membership` section)
- **Change:** Update the text at line 74 from *"Group.membership requires one of three codes:"* to something like *"Group.membership, when populated, uses one of three codes:"*. Add a note explaining that when `Group.membership` is absent, it means the membership basis is unknown or cannot be expressed with the standard codes.

### Testing Considerations

1. **Validate the updated StructureDefinition** passes the FHIR build tooling without errors.
2. **Run existing invariant tests** (`grp-2.f1.fail.json`, `grp-2.p1.pass.json`, `grp-2.p3.pass.json`, `grp-2.p4.pass.json`) to confirm no regressions with the updated expression.
3. **Verify the new invariant test** (`grp-2.p5.pass.json`) passes — a Group without `membership` (and without `type`) should be valid.
4. **Validate all existing Group examples** continue to pass — since they all populate `membership`, the cardinality change from 1..1 to 0..1 should not affect them.
5. **Test search parameter behavior** — confirm that searching `membership=enumerated` does not match a Group where `membership` is absent.
6. **Build the IG** and verify the Group resource page renders correctly with the updated cardinality and `meaningWhenMissing` text.

### Open Questions

1. **Invariant `grp-2` behavior when membership absent:** The current invariant requires `type` unless `membership = 'conceptual'`. With the cardinality change, should an absent `membership` also exempt the Group from needing `type`? This plan assumes **yes** (an unknown membership basis should not force the Group type to be specified), but this should be confirmed with the work group.

2. **IsModifier consideration:** Should `Group.membership` become an `isModifier` element now that its absence carries specific meaning? Currently it is not marked as a modifier. The `meaningWhenMissing` pattern typically does not require `isModifier`, but this is worth confirming.

3. **Impact on profiles:** Profiles that set `Group.membership` to `1..1` (overriding the base) will not be affected. However, profiles that inherit the base cardinality will now allow absent `membership`. Profile owners should be notified of this change.
