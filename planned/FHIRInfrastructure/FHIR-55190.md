## Resolution Summary

Resolution: **Not Persuasive with Modification**.

Per the resolution description: *"Will add a comment saying 'If pre-adopting a new type from a future FHIR version, use the `special` code and communicate the alternate type as an extension.'"*

The committee declined to add a new `unknown`/`other` concept to the `search-param-type` value set (the requested change) — the existing `special` code is treated as the escape valve. Instead, an `ElementDefinition.comment` is to be added to `SearchParameter.type` documenting the agreed convention: when an implementer needs to use a search parameter `type` value defined in a later version of FHIR than the one they are publishing against, they should set `SearchParameter.type` to `special` and convey the actual intended type via an extension.

## Feature Proposal

### Problem Statement

`SearchParameter.type` is a 1..1 element with a `required` binding to `http://hl7.org/fhir/ValueSet/search-param-type`. The reporter (Lloyd McKenzie) observed that the value set has no generic `unknown` or `other` concept, which means the element cannot be populated when the conceptual search type is not represented in the bound value set — for example, when pre-adopting a new search type from a future FHIR version. The original request was to add an `unknown`/`other`/equivalent escape valve concept to the value set.

### Proposed Change

Do **not** modify the `search-param-type` CodeSystem or ValueSet. Instead, add a single `<comment>` to the `SearchParameter.type` `ElementDefinition` in the SearchParameter StructureDefinition, immediately after the existing `<definition>` element (and before `<min>`), following the placement convention used by sibling elements such as `SearchParameter.expression` (line 559) and `SearchParameter.processingMode` (line 595 area).

Concrete edit:

- File: `source/searchparameter/structuredefinition-SearchParameter.xml`
- Element: `<element id="SearchParameter.type">` (currently lines 531–554).
- Insert between line 534 (`<definition .../>`) and line 535 (`<min value="1"/>`):

  ```xml
  <comment value="If pre-adopting a new type from a future FHIR version, use the 'special' code and communicate the alternate type as an extension."/>
  ```

No changes are required to:

- `source/codesystems/` or `source/valuesets/` entries for `search-param-type` — the `special` code already exists in the bound value set (it is listed in the element's `<short>` value `number | date | string | token | reference | composite | quantity | uri | special | resource`).
- The element's cardinality, type, condition, isSummary, or binding — all remain unchanged.
- Any other resource or page.

### Design Rationale

- The committee's modification chose to **document a convention over the existing `special` code** rather than expand the value set. This avoids: (a) introducing a new code with semantics that overlap `special`; (b) churn in downstream tooling and validators that already enumerate the bound codes; and (c) the risk that an `unknown`/`other` code becomes a dumping ground that obscures the intended search semantics.
- Using `special` + an extension preserves the `required` binding's validity while giving implementers a documented forward-compatibility mechanism for pre-adopting future search types.
- Placing the guidance in `ElementDefinition.comment` (rather than in narrative on the SearchParameter page) ensures the rule travels with the element definition into derived snapshots, IG publisher output, and tooling that surfaces element comments to authors.
