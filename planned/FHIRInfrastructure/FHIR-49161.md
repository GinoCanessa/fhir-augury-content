## Resolution Summary

The ticket reports a typo in the FHIRPath expression of Questionnaire invariant
`que-10` ("Maximum length can only be declared for simple question types"). The
expression compares `answerConstraint` to the literal string `'optionOrString'`,
but that is **not** a valid code in the
`http://hl7.org/fhir/questionnaire-answer-constraint` CodeSystem. The valid
codes are `optionsOnly`, `optionsOrType`, and `optionsOrString` — note the
plural `options`. The resolution (Persuasive) requires changing
`'optionOrString'` to `'optionsOrString'` so the invariant actually fires/passes
as intended.

## Feature Proposal

### Problem Statement

Invariant `que-10` on `Questionnaire.item` permits a `maxLength` to be set
either when the item type is one of the simple primitive types
(`string`, `decimal`, `integer`, `text`, `date`, `dateTime`, `time`, `url`) or
when `answerConstraint` allows free-text answers. The intent of the second
clause is to also allow `maxLength` on coded items whose `answerConstraint`
permits open-text answers — i.e., the code `optionsOrString` from the
`questionnaire-answer-constraint` CodeSystem.

The expression as published compares against `'optionOrString'` (singular
"option"), which is not a member of that CodeSystem. As a result, the second
disjunct of `que-10` can never evaluate to `true`, so any `Questionnaire.item`
that uses `answerConstraint = 'optionsOrString'` together with a `maxLength`
will incorrectly fail validation against `que-10`.

Brian Postlethwaite's comment also notes — out of scope for this fix but worth
recording — that the project should add negative test resources covering all
published invariants.

### Proposed Change

In `source/questionnaire/structuredefinition-Questionnaire.xml`, in the
`<constraint>` element with `<key value="que-10"/>`, change the FHIRPath
expression from:

```fhirpath
(type in ('string' | 'decimal' | 'integer' | 'text' | 'date' | 'dateTime' | 'time' | 'url')) or answerConstraint='optionOrString' or maxLength.empty()
```

to:

```fhirpath
(type in ('string' | 'decimal' | 'integer' | 'text' | 'date' | 'dateTime' | 'time' | 'url')) or answerConstraint='optionsOrString' or maxLength.empty()
```

Concretely: in the `<expression value="..."/>` attribute on line 793, change
the substring `answerConstraint='optionOrString'` to
`answerConstraint='optionsOrString'` (insert a single `s` after `option`). No
other attributes of the constraint (`key`, `severity`, `human`, `source`)
change. The three sibling `<condition value="que-10"/>` markers on
`Questionnaire.item.answerOption`, `Questionnaire.item.answerValueSet`, and the
related element (lines 994, 1255, 1276) do not need any change — they only
reference the key.

### Design Rationale

The `questionnaire-answer-constraint` CodeSystem in the same folder
(`source/questionnaire/codesystem-questionnaire-answer-constraint.xml`,
lines 25/30/35) defines exactly three codes:

- `optionsOnly`
- `optionsOrType`
- `optionsOrString`

`'optionOrString'` matches none of these, so the existing expression is dead
code in the second disjunct. The fix simply restores the originally intended
semantics: when the author has declared that an item permits options *or* a
free-text string, the `maxLength` constraint may be applied to the free-text
portion. No alternative wording was considered because the resolution is a
literal typo correction — the surrounding logic, severity, and human-readable
message are all already correct. Comparing against the spelling used in the
authoritative CodeSystem also ensures the expression remains stable should a
future change tighten validation of FHIRPath string literals against bound
ValueSets.

No breaking changes identified in the strict sense — but note the change is
*behaviour-altering* in a benign direction:

- Before: `que-10`'s second disjunct never matches, so a Questionnaire item

  would *fail* the invariant.

resource that previously passed. No instances are made non-conformant by the
change, so no migration is required.
