## Resolution Summary

Resolution: **Persuasive**. The ticket reports a typo in the FHIR Core
documentation — the text **"the busines version"** must read **"the business
version"**. The fix is a single-character correction (insert the missing `s`)
in the narrative text describing the format of business versions on canonical
resources. No comments, applied-vote text, or resolution_description beyond
the "Persuasive" disposition were returned by the Jira source; the scope is
therefore taken directly from the ticket description.

## Feature Proposal

### Problem Statement

The FHIR Core specification page describing the format of the
`businessVersion` on canonical resources contains a typographical error:
the word "business" is misspelled as "busines" (missing a trailing `s`) in
the sentence that prescribes that the business version SHOULD NOT contain a
`#` character. The typo appears once in the authored HTML for
`resource-notes.html` and surfaces in the published spec. It is a cosmetic /
readability defect only — no normative meaning is changed — but the page is
part of the FHIR Core narrative and should be correct.

### Proposed Change

Correct the spelling of "business" on line 758 of
`source/resource/resource-notes.xml` in `HL7/fhir`.

- Current text (line 758):
  > `For business versions in canonical resources, the busines version SHOULD NOT contain a <code>#</code>`
- Corrected text:
  > `For business versions in canonical resources, the business version SHOULD NOT contain a <code>#</code>`

Change detail:
- Exactly one character is inserted (`busines` → `business`).
- The surrounding markup (`<a name="bv-format"></a>` anchor, `<p>` wrapping,
  `<code>#</code>` inline element, SHOULD NOT normative language) is
  preserved verbatim.
- No other occurrence of `busines` (as a non-plural stem) exists under
  `source/` in the clone — a repo-wide grep for `\bbusines\b` returns this
  single hit (`source/resource/resource-notes.xml:758`).

### Design Rationale

This is a pure documentation typo fix. The repo briefing for `HL7/fhir`
identifies `source/` as the sole authoring root and flags that "bespoke
publisher reads `source/*.html` pages as authored content; HTML edits go in
directly" — `source/resource/resource-notes.xml` is exactly such an authored
narrative file (XHTML content consumed by the HL7 publisher). Editing the
file directly is the recommended recipe; no tooling regeneration, no
StructureDefinition differential, no terminology artefact, and no generated
area is involved. The correction preserves all normative content and simply
restores the intended English spelling.
