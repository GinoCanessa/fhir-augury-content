## Resolution Summary

Resolution: **Persuasive**. The invariant `sdd-1` on the `SampledData` datatype contains a typographical error: the word `SHALL` is misspelled as `SAHLL` in the human-readable text of the constraint. The current text reads:

> *A SampledData SAHLL have either an interval and offsets but not both*

The fix is a literal spelling correction — change `SAHLL` to `SHALL` everywhere it appears in the `sdd-1` invariant text within the `SampledData` datatype definition. The FHIRPath `expression` (`interval.exists().not() xor offsets.exists().not()`), severity, and key are unchanged. Khalid Shahin (the reporter) noted in Zulip (`committers` > `Closing this Jira ticket for FHIR Infrastructure`, 2025-05-12) that the fix was carried in `HL7/fhir` PR [#3378](https://github.com/HL7/fhir/pull/3378) ("ResearchStudy-March-14-2025-B"), commit `ffbab9b791` ("J#49760 ResearchStudy changes and J#42848 sampledata.xml"). The `master` branch of `HL7/fhir` (clone HEAD `db79dcd`) already shows `SHALL` at `source/datatypes/sampleddata.xml` line 3428, confirming the change has been applied. The ticket itself is still in `Resolved - change required` and needs to be flagged Published; no further authoring change is required in `source/`.

## Feature Proposal

### Problem Statement

The `SampledData` datatype defines invariant `sdd-1` to enforce that a `SampledData` instance carries either an `interval`/`intervalUnit` *or* `offsets`, but not both. The human-readable form of the constraint embedded in the StructureDefinition (and rendered into the published spec, snapshots, and downstream artifacts) contains a typo: `SAHLL` instead of `SHALL`. This is purely a documentation/display defect — the FHIRPath expression that actually enforces the rule is correct — but the misspelling is visible to every implementer reading the SampledData page or the rendered constraints in any IG that re-publishes the type. There is no other gap; the constraint's semantics, severity (`error`), and machine-checkable expression are all correct.

### Proposed Change

Edit `HL7/fhir` to replace the misspelled `SAHLL` with `SHALL` in the human text of invariant `sdd-1` on `SampledData`.

- **Authoring file:** `source/datatypes/sampleddata.xml` (per the briefing, files under `source/datatypes/<type>.xml` are Excel SpreadsheetML, not FHIR XML — edits go in the spreadsheet cell content, not in a `<human>` element). The cell currently reads:

  ```text
  A SampledData SAHLL have either an interval and offsets but not both
  ```

  Change it to:

  ```text
  A SampledData SHALL have either an interval and offsets but not both
  ```

  No other column on that row (FHIRPath expression, key, severity) changes.

- **Generated artifacts:** the publisher regenerates `<snapshot>` blocks and the per-release `tools/history/` archives from the spreadsheet on the next build; do not hand-edit those. The historical archive `tools/history/release5/profiles-types.xml` still contains the misspelt phrase but is a frozen release snapshot and is explicitly out of scope (generated area per the briefing).

- **Published / merged:** the change is already on `master` (commit `ffbab9b791`, merged via PR [#3378](https://github.com/HL7/fhir/pull/3378)). The remaining administrative action on the ticket is to mark it `Published`.

Example of the fixed invariant as it should appear in the rendered StructureDefinition constraint:

```xml
<constraint>
  <key value="sdd-1"/>
  <severity value="error"/>
  <human value="A SampledData SHALL have either an interval and offsets but not both"/>
  <expression value="interval.exists().not() xor offsets.exists().not()"/>
</constraint>
```

### Design Rationale

The change is a pure typographical correction with no behavioural impact: the FHIRPath expression that machines evaluate is unchanged. There is no design decision beyond "spell `SHALL` correctly", which aligns with the FHIR convention that normative keywords (`SHALL`, `SHOULD`, `MAY`) are written in upper-case per RFC 2119 / IETF BCP 14. No alternative was considered or warranted. The Zulip note from the reporter confirms this was the intended fix, and the resolution `Persuasive` indicates FHIR Infrastructure agreed without modification.
