## Resolution Summary

Resolution: **Persuasive**. The committee agreed to broaden
`ElementDefinition.valueAlternatives` so it is no longer restricted to
primitive elements. The applied resolution text is:

> Do this. Also add a note that says: "valueAlternatives applies only to
> the elements it is declared on. It does not propagate to child elements."

Concretely, the resolution adopts the changes proposed in the ticket
description verbatim:

1. **Short** of `ElementDefinition.valueAlternatives` changes from
   `Extensions that are allowed to replace a primitive value` to
   `Extensions that are allowed to replace a value/content`.
2. **Definition** changes from
   `Specifies a list of extensions that can appear in place of a primitive value.`
   to
   `Specifies a list of extensions that can appear in place of a value (for primitive types) or non-extension child elements (for complex types and backbone elements).`
3. **Usage note** sentence
   `...it SHALL have either a value or at least one of the listed extensions`
   becomes
   `...it SHALL have at least one of the listed extensions, a value (if primitive), or a non-extension child element (if a complex type or backbone element)`.
4. The standing usage-note assertion that `valueAlternatives` "is only
   permitted on elements with primitive types" must be removed (it is the
   constraint being lifted), and the new note from the resolution must
   be added: `valueAlternatives applies only to the elements it is
   declared on. It does not propagate to child elements.` Grahame's
   review comment ("we should comment that the list of extensions does
   not apply to sub-properties") aligns with this same note.

Narrative pages that frame `valueAlternatives` as primitive-only must
also be revised so they describe the broadened semantics.

## Feature Proposal

### Problem Statement

Today, `ElementDefinition.valueAlternatives` may only be set on elements
of primitive type. When a profile author wants to require "at least one
component, or one of these extensions" on a complex/backbone element
(e.g., on `Quantity`, `CodeableConcept`, or a BackboneElement), the only
modelling option is to slice the extension and enumerate every allowed
slice. This bulks up the snapshot, clutters renderings, and is harder to
read than a simple list of allowed alternative extensions. The Zulip
discussion in `fhir/infrastructure-wg` ("Why is valueAlternatives
limited to primitives only?") confirms FMG agreement (Lloyd McKenzie /
Grahame Grieve) that the primitive-only restriction is unnecessary.

### Proposed Change

Generalise `ElementDefinition.valueAlternatives` so it can be declared
on **any** element (primitive, complex, or backbone), with the rule that
"the value/content is satisfied" varies by category:

- **Primitive element:** the element either has a `value`, or has at
  least one of the listed alternative extensions.
- **Complex type or BackboneElement:** the element either has at least
  one non-extension child element present, or has at least one of the
  listed alternative extensions.

The element's cardinality, type (`canonical(StructureDefinition)`), and
the existing co-occurrence constraint
`mustHaveValue.value implies valueAlternatives.empty()` (rule "Can't
have valueAlternatives if mustHaveValue is true") all remain unchanged
— only the prose / definition text and the generalised semantics
change.

Specifically, in `source/datatypes/elementdefinition.xml` (the
ElementDefinition spreadsheet), update the row whose Path cell is
`ElementDefinition.valueAlternatives`:

- **Short** cell: `Extensions that are allowed to replace a value/content`
- **Definition** cell: `Specifies a list of extensions that can appear in place of a value (for primitive types) or non-extension child elements (for complex types and backbone elements)`
- **Comments / usage-note** cell: replace the existing text
  > If mustHaveValue is true, then this element should not be present, since no extensions are allowed in place of the value. Note that this element has no impact if the value is present. When valueAlternatives is present, it means that if the element is present, it SHALL have either a value or at least one of the listed extensions. This element is only permitted on elements with primitive types.

  with
  > If mustHaveValue is true, then this element should not be present, since no extensions are allowed in place of the value. Note that this element has no impact if the value (or, for complex types and backbone elements, a non-extension child element) is present. When valueAlternatives is present, it means that if the element is present, it SHALL have at least one of the listed extensions, a value (if primitive), or a non-extension child element (if a complex type or backbone element). valueAlternatives applies only to the elements it is declared on; it does not propagate to child elements.

  (i.e. the "only permitted on elements with primitive types" sentence
  is dropped, the SHALL clause is rewritten to the resolution wording,
  and the new "applies only to the elements it is declared on…" note is
  appended.)

In `source/elementdefinition.html` (the narrative for ElementDefinition),
update the paragraph that currently reads:

> Alternatively, profiles may wish to allow the primitive value to be
> replaced by some extensions but not others. In this case, the profiles
> can list the allowable extensions using the `valueAlternatives`
> element, which contains a list of the extensions that can appear if
> the primitive value is not present. Note that this list is a short cut
> for actually profiling the extensions on the primitive data type and
> making a co-occurence constraint, but **has no effect when the value
> is present**.

so that it covers complex types and backbone elements as well — e.g.:

> Alternatively, profiles may wish to allow the value (for primitive
> types) or the non-extension content (for complex types and backbone
> elements) to be replaced by some extensions but not others. In this
> case, the profiles can list the allowable extensions using the
> `valueAlternatives` element, which contains a list of the extensions
> that can appear if no value or non-extension child element is present.
> Note that this list is a short cut for profiling the extensions on
> the element and making a co-occurrence constraint, but **has no
> effect when the value (or any non-extension child element) is
> present**. `valueAlternatives` only applies to the element on which
> it is declared; it does not propagate to that element's children.

No change is required to `source/exchanging.html` — the two
references there ("properties `mustHaveValue` or `valueAlternatives`
to disallow the use of extensions like this, or control which
extensions are allowed in place of a value") are already neutral with
respect to primitive vs. complex.

### Design Rationale

- **Consistency across the type system.** `mustHaveValue` is
  legitimately primitive-only (it would otherwise need to enumerate
  which sub-elements are required). `valueAlternatives`, by contrast,
  only ever lists extensions, so the same machinery can govern any
  element kind without per-type complications. This was the explicit
  argument in the Zulip thread.
- **Authoring ergonomics.** The slicing workaround required today
  ("slice the extension element and enumerate the allowed slice
  profiles") materially inflates snapshot output. Allowing
  `valueAlternatives` on complex/backbone elements is a strict
  simplification with no expressive loss.
- **No type or cardinality change.** The element remains
  `0..* canonical(StructureDefinition)`; only the human-facing
  semantics broaden. Existing instances of `valueAlternatives` on
  primitive elements continue to mean exactly what they did before.
- **Scoping note.** The added "does not propagate to child elements"
  sentence (resolution + Grahame's review comment) prevents
  implementers from interpreting a list on, say, `Patient.name` as
  also applying to `Patient.name.given`.
- **Alternatives considered.** Keeping the slicing-only workflow was
  rejected as too verbose. Introducing a parallel new element for
  complex types was rejected as redundant — the resolution simply
  drops the artificial primitive restriction.

### Breaking Changes

No breaking changes identified.

- The element's cardinality, type and existing constraint
  (`mustHaveValue.value implies valueAlternatives.empty()`) are
  unchanged.
- Existing instances of `valueAlternatives` on primitive elements
  retain their meaning exactly.
- The change is *broadening*: previously-disallowed usage (declaring
  `valueAlternatives` on a complex or backbone element) becomes
  allowed. Implementations that validated and rejected such usage
  will need to be updated, but no previously valid content becomes
  invalid.
- Validators that hard-coded "valueAlternatives only on primitive
  elements" will need to relax that check; this is a
  validator-implementation consideration, not a normative breaking
  change in resources/instances.

### Related Specifications

- `ElementDefinition.mustHaveValue` (sibling element) — semantics are
  unchanged but the comments cell's reference to mustHaveValue should
  remain consistent with the new wording.
- ElementDefinition narrative on the "Missing Data" / extensions-in-
  place-of-value pattern (`source/elementdefinition.html`) and the
  Exchanging Data page (`source/exchanging.html`) — both reference
  `valueAlternatives` and must read consistently with the broadened
  scope.
- The `mustHaveValue.value implies valueAlternatives.empty()`
  invariant continues to apply uniformly across primitive, complex and
  backbone elements (since `mustHaveValue` itself stays primitive-only,
  the implication is vacuously satisfied for non-primitive elements).
- No StructureDefinition snapshots in core resources need
  hand-editing; the publisher regenerates them from the updated
  `ElementDefinition` definition.
