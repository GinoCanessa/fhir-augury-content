## Feature Proposal

### Problem Statement

The `versions.html` page in FHIR Core carries an overall standards-status of
**Normative** (see the page header: `<table class="colsn">` and the in-page
"Standards Status: Normative" link). However, the page also contains the
section "Plans for future releases" (anchor `#plans`, currently at
`source/versions.html` lines 776–817), which describes HL7's intentions for
future FHIR releases (e.g., moving Foundation/Base/Clinical resources to
Normative, planned content updates, pointers to the build snapshot and the
product director's blog).

Forward-looking plans are inherently subject to change and cannot be
balloted as Normative. Leaving the section under the page's overall
Normative banner risks readers (and downstream regulators / SDOs that take
"Normative" at face value) treating those plans as binding commitments.
The reporter explicitly objects to "normative plans" and asks that the
section be labeled **Informative** instead.

### Proposed Change

Apply the project's standard **Informative** marker to the heading of the
"Plans for future releases" subsection on `source/versions.html` so that
the section is visibly demoted from the page's overall Normative status.

The repo already has an established pattern for inline standards-status
flagging (used throughout `source/documentation.html` lines 38–122):

```html
<a href="versions.html#std-process" title="Informative Content" class="informative-flag">I</a>
```

Concretely, modify the `<h3>` for the section so it carries the Informative
flag immediately after the title text. Today the section opens as:

```html
<a name="plans"></a>
<h3>Plans for future releases</h3>
```

and should become:

```html
<a name="plans"></a>
<h3>Plans for future releases <a href="versions.html#std-process" title="Informative Content" class="informative-flag">I</a></h3>
```

Mirror the same flag on the table-of-contents entry near the top of the
file so the change is discoverable from the page TOC. Currently
(`source/versions.html` line 29):

```html
<li><a href="#plans">Plans for future releases</a></li>
```

becomes:

```html
<li><a href="#plans">Plans for future releases</a> <a href="versions.html#std-process" title="Informative Content" class="informative-flag">I</a></li>
```

No prose changes are required: the body of the section already reads as
forward-looking guidance and is consistent with Informative content.

### Design Rationale

- The `informative-flag` / `class="informative-flag"` CSS hook (rendered
  as a green "I" badge by the publisher) is the canonical, repo-wide way
  to mark a single artifact or section as Informative when the surrounding
  page or list carries a different status. It is the same construct used
  for every Informative entry listed on `source/documentation.html` (lines
  38–122) and on numerous TOC entries elsewhere.
- Marking only the subsection — rather than re-classifying the whole
  `versions.html` page — preserves the Normative status of the genuinely
  normative content on that page (the standards-development process,
  inter-version change rules, FHIR Maturity Model, etc.) while correctly
  demoting only the forward-looking plans.
- Adding the flag both at the heading and in the TOC matches what
  `documentation.html` does for its Informative entries, which is the
  closest authored precedent in the same authoring root.
- The resolution is simply "Persuasive" with no further normative text
  prescribing markup; following the established `informative-flag`
  pattern is the minimum, lowest-risk change that satisfies the reporter's
  request.
