## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"Grahame will talk to the webmaster and see if we can get redirects for all of the URLs defined in `mappingSpaces.xml` on the HL7 website."*

The approved disposition is an **operational fix on the HL7 website**, not a
content change in the FHIR specification: the link-text values already
shown next to each mapping heading (e.g., `http://hl7.org/v2`,
`http://hl7.org/v3`) will remain as-is in `source/mappingSpaces.xml`, and
the HL7 webmaster will be asked to add HTTP redirects so that each of
those short URLs resolves to the same destination currently used in the
`<link>` attribute (the HL7 `product_brief.cfm` pages, `dicomstandard.org`
current, etc.). The in-repo option of changing the link text to match
`<link>` destinations was considered and not chosen.

## Feature Proposal

### Problem Statement

On every resource's **Mappings** tab, the FHIR publisher renders section
headings in the form:

> HL7 V2 Mapping (http://hl7.org/v2)
> RIM Mapping (http://hl7.org/v3)

The parenthesised text is both the displayed label **and** the `href`
target of the anchor (it comes from the `<url>` element in
`source/mappingSpaces.xml`). Clicking the link works because of
browser/publisher-level navigation, but the raw URL — i.e., what a user
gets when they copy the visible text and paste it into a new browser
tab — currently returns HTTP 404 on the HL7 site. This affects the
**majority** of FHIR Core resources, because V2 and RIM mappings are
ubiquitous, and extends to any other mapping whose `<url>` does not
resolve to a live page on the HL7 website (notably `http://hl7.org/v3`
and `http://hl7.org/v3/cda`).

Supporting context:
- Nick Radov (reporter) noted the short links used to work and appear to
  have been broken by recent HL7 website changes.
- Brian Postlethwaite's comment identified this as a repeated pattern
  across many resources rather than a per-resource authoring bug.
- Lloyd McKenzie confirmed Grahame would engage the HL7 webmaster.

### Proposed Change

**Primary (approved) change — HL7 website:** add HTTP redirects so every
`<url>` listed in `source/mappingSpaces.xml` resolves to a useful page.
For the URLs specifically highlighted in the ticket this means:

- `http://hl7.org/v2` → `http://www.hl7.org/implement/standards/product_brief.cfm?product_id=185` (the value of the sibling `<link>` in the `v2` space).
- `http://hl7.org/v3` → `http://www.hl7.org/implement/standards/product_brief.cfm?product_id=186` (the value of the sibling `<link>` in the `rim` space).
- `http://hl7.org/v3/cda` → `http://www.hl7.org/implement/standards/product_brief.cfm?product_id=7` (the value of the sibling `<link>` in the `cda` space).

…and the same principle (`<url>` → `<link>`) for every other `<space>` in
`source/mappingSpaces.xml` whose `<url>` is hosted on `hl7.org` and does
not already resolve. Non-HL7 URLs (e.g., `http://loinc.org`,
`http://snomed.org/attributebinding`, `https://dicomstandard.org/current`,
`http://ietf.org/rfc/2445`, `http://metadata-standards.org/11179/`,
`http://openehr.org`, `http://siframework.org/...`,
`http://www.cdisc.org/define-xml`, etc.) are out of scope — those are the
third party's responsibility.

**Secondary (repo-side) change — HL7/fhir, contingent on the webmaster
result:** no content change is required if the redirects are put in
place. *Only if* the webmaster cannot create the redirects does the
fallback become "change the link text in FHIR to match the actual
destinations", i.e. swap the rendering so that the visible text inside
parentheses on the Mappings tab is the `<link>` value (or the `<url>` is
replaced by the `<link>` value in `mappingSpaces.xml`). That fallback is
documented below in the Implementation Plan as an optional branch.

### Design Rationale

- The ticket frames two options explicitly — "create working short
  links" **or** "change the link text in FHIR to match the actual
  destinations". The resolution prefers the first because:
  - The short URLs (`http://hl7.org/v2`, `http://hl7.org/v3`,
    `http://hl7.org/v3/cda`) are already published in thousands of
    rendered pages across every FHIR version; keeping them stable
    and making them resolvable is less disruptive than changing the
    visible label everywhere.
  - `mappingSpaces.xml` deliberately distinguishes `<url>` (formal,
    stable identifier) from `<link>` (the pragmatic "click here"
    target). The comment block at the top of the file
    (lines 3–11) describes `<url>` as *"the formal URL — should also
    resolve to a useful document if possible"*, which squarely frames
    the fix as "make the formal URL resolve", not "redefine the formal
    URL".
- Alternatives considered:
  - **Rewriting `<url>` values to point at `product_brief.cfm`:**
    rejected because it changes the displayed identifier across the
    whole spec and conflates the formal URL with the product-brief
    redirect target.
  - **Per-resource edits to each Mappings tab:** rejected as
    infeasible and because Brian Postlethwaite and the publisher
    authoring model both confirm this is centralized in a single
    tooling input file (`source/mappingSpaces.xml`).

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Source of the displayed URLs (`source/mappingSpaces.xml`).
  Under the **approved** resolution, no edit is required here; the file
  is what documents which URLs the webmaster must make resolve. Under
  the **fallback** branch, `mappingSpaces.xml` is the only file changed.
- **Affected Files:**
  - `source/mappingSpaces.xml` — *only if the fallback branch is taken.*
    Change would adjust the `<url>` (or the publisher's label
    rendering) for each `<space>` whose short URL cannot be made to
    resolve, so that the text shown in parentheses on every Mappings
    tab matches a resolvable destination.
- **Change Scope:** Minor (single file, ≤ ~20 lines) if the fallback is
  taken; **No change** if the primary resolution succeeds.

#### HL7 website (not a GitHub repository)

- **Role:** Add redirects from each `<url>` value in
  `source/mappingSpaces.xml` to the corresponding `<link>` destination.
  This is handled by the HL7 webmaster via web-server or CMS
  configuration and is not tracked in any of the configured cached
  repos.
- **Affected Files:** n/a (web-server redirect configuration).
- **Change Scope:** Minor — a handful of redirect rules covering
  `http://hl7.org/v2`, `http://hl7.org/v3`, `http://hl7.org/v3/cda`, and
  the FHIR-hosted `<url>` values already pointing at `hl7.org/fhir/...`
  (which generally do resolve and are out of scope).

### Breaking Changes

No breaking changes identified. The displayed link text and `href`
targets on every Mappings tab remain byte-for-byte identical under the
approved resolution — only the server-side behaviour of the target URLs
changes (from 404 to 3xx redirect). Implementers who have copy/pasted
the short URLs into their own material will see those URLs start
working, which is strictly an improvement.

## Implementation Plan

### Prerequisites

- Confirm with the HL7 webmaster (Eric, per the 2024-12-16 comment from
  Lloyd McKenzie) that HTTP redirects can be added on the `hl7.org`
  host for the URLs listed in `source/mappingSpaces.xml`.
- Enumerate the exact set of short URLs that currently 404 versus those
  that already resolve — the redirect list is driven by that
  enumeration, not by blanket coverage of every `<space>`.

### Step-by-Step Tasks

#### HL7 website (webmaster — primary / approved resolution)

1. **Inventory the short URLs that need redirects**
   - File: `cache/github/repos/HL7_fhir/clone/source/mappingSpaces.xml`
     (read-only input; no edit)
   - Action: extract every `<url>` value from `<space>` entries whose
     host is under `hl7.org`. For each, record the sibling `<link>`
     value (if present) as the redirect target.
   - Details: at minimum this includes:
     - `http://hl7.org/v2` → `http://www.hl7.org/implement/standards/product_brief.cfm?product_id=185` (`v2` space, lines 45–54).
     - `http://hl7.org/v3` → `http://www.hl7.org/implement/standards/product_brief.cfm?product_id=186` (`rim` space, lines 110–119).
     - `http://hl7.org/v3/cda` → `http://www.hl7.org/implement/standards/product_brief.cfm?product_id=7` (`cda` space, lines 101–109).
     `<space>` entries whose `<url>` already points at
     `http://hl7.org/fhir/...` (e.g., `fivews`, `workflow`, `interface`)
     are already served by the FHIR publication site and should be
     verified; if they resolve, no redirect is needed.

2. **Request redirects from the HL7 webmaster**
   - File: n/a (out-of-band email / ticket to Eric)
   - Action: send the inventoried list to the webmaster and ask for
     HTTP 301 (or 302) redirects from each short URL to the recorded
     target.
   - Details: preserve the case/path exactly as it appears in
     `mappingSpaces.xml`; use 301 where the mapping is intended to be
     permanent. Confirm that the redirects work over both `http` and
     `https` and from both `hl7.org` and `www.hl7.org` hostnames.

3. **Verify the redirects**
   - File: n/a (browser / `curl -I` check)
   - Action: for each redirected URL, confirm a 3xx response that lands
     at the intended `product_brief.cfm` (or equivalent) destination.
   - Details: spot-check from at least one rendered resource's Mappings
     tab on the current FHIR publication site (e.g., Patient,
     Observation) by copying the short URL and pasting it into a fresh
     browser session.

#### HL7/fhir (GitHub repository — fallback branch, only if redirects cannot be added)

4. **Fallback: align the displayed link text with the live destination**
   - File: `source/mappingSpaces.xml`
   - Action: for each `<space>` whose short `<url>` the webmaster
     cannot make resolvable, change the `<url>` value to the same
     value currently held in the sibling `<link>` element (or introduce
     a publisher-side tweak that renders the `<link>` value in
     parentheses instead of the `<url>` value).
   - Details: worked example for the `v2` space (lines 45–54):

     ```xml
     <space>
       <url>http://www.hl7.org/implement/standards/product_brief.cfm?product_id=185</url>
       <link>http://www.hl7.org/implement/standards/product_brief.cfm?product_id=185</link>
       <columnName>v2 Mapping</columnName>
       <title>HL7 V2 Mapping</title>
       <id>v2</id>
       <sort>1</sort>
       <publish>true</publish>
       <preamble>…</preamble>
     </space>
     ```

     Apply the same `<url>` ⟵ `<link>` swap for the `rim` (lines
     110–119), `cda` (lines 101–109), and any other affected
     `hl7.org`-hosted `<space>` entries. Do **not** edit per-resource
     files or the generated `<snapshot>` blocks in any
     StructureDefinition — the publisher re-renders every resource's
     Mappings tab from `mappingSpaces.xml` on the next build.

5. **Fallback: repo-wide sanity check for stale references**
   - File: repository-wide grep (read-only)
   - Action: after editing `mappingSpaces.xml`, search for any page or
     tooling fixture that hard-codes the old short URL string (e.g.,
     `http://hl7.org/v2`, `http://hl7.org/v3`) as the *link target*
     rather than as descriptive prose.
   - Details: descriptive prose references in `source/*.html` that
     *mention* `http://hl7.org/v2` in text may legitimately remain;
     the concern is HTML anchors (`href="http://hl7.org/v2"`) that
     would still 404 after the fallback. Fix or remove any such
     hard-coded anchors surfaced by the grep.
