## Resolution Summary

**Resolution:** Not Persuasive with Modification.

The resolver determined that no schema change is required on `ExampleScenario.instance`:

> "The specification is already able to handle non-FHIR data. CDA, v2, CSV, or
> other data can be pointed to as an instance. It would simply be a Binary
> reference. The IG tooling can determine (or be informed of) the appropriate
> mime type the Binary has and, if appropriate, can even indicate the logical
> model the instance is to be validated against.
>
> We will update the spec to include an ExampleScenario that shows exchanges
> and examples of non-FHIR data."

The concrete modification, therefore, is **documentation only**: add a new
worked `ExampleScenario` example that demonstrates how to point
`ExampleScenario.instance` at non-FHIR content (CDA, v2, CSV, or similar) via
a `Binary` reference, and surface that in the ExampleScenario page narrative
so the capability is discoverable.

## Feature Proposal

### Problem Statement

The original ticket (Comment 18, imported by Ron G. Parker against the
`ExampleScenario.instance` element) requested that the specification "support
non-FHIR data" and asked for a review of `structureX` / `content` elements to
ensure sufficient flexibility.

Inspection of the current StructureDefinition at
`source/examplescenario/structuredefinition-ExampleScenario.xml` confirms
that the capability already exists:

- `ExampleScenario.instance.structureType` (lines 635–654) is a `Coding` with
  an **extensible** binding to
  `http://hl7.org/fhir/ValueSet/examplescenario-instance-type`. That ValueSet
  (`source/examplescenario/valueset-examplescenario-instance-type.xml`)
  already composes `http://hl7.org/fhir/ValueSet/resource-types` **and**
  `http://terminology.hl7.org/CodeSystem/v2-0354`, so v2 message-structure
  codes are explicitly in scope, and `extensible` permits CDA / CSV / Word /
  PDF / OpenEHR / X12 / any other structure.
- `ExampleScenario.instance.structureVersion` (lines 655–666) carries release
  info when the structure is not FHIR ("I.e. what release of FHIR, X12,
  OpenEHR, etc. is instance compliant with"), and invariant `exs-1` at line
  581–582 makes it required when `structureType` is not a FHIR resource.
- `ExampleScenario.instance.structureProfile[x]` (lines 667–679) takes a
  `canonical` or a plain `uri` — so a non-FHIR profile / template / schema
  URL is permitted.
- `ExampleScenario.instance.content` (lines 707–718) is a `Reference`, and
  its formal `comment` states: "If not conveying FHIR data or not using the
  same version of FHIR as this ExampleScenario instance, the reference must
  be to a Binary." `Binary.contentType` then supplies the mime type (`text/cda+xml`,
  `application/hl7-v2`, `text/csv`, `application/pdf`, etc.).

The gap is purely illustrative: none of the existing examples
(`examplescenario-example.xml`, `examplescenario-example-laborder.xml`) uses
a non-FHIR `structureType`, and the narrative at
`examplescenario-notes.xml` does not call out the pattern. Implementers
reading the spec therefore miss that the capability is already there.

### Proposed Change

Add a new, self-contained ExampleScenario example — proposed key
`example-nonfhir` — that exchanges at least one non-FHIR instance via a
`Binary` reference, and wire it into the spec's publication plumbing plus
the ExampleScenario narrative.

Concrete content for the new example:

1. A small, realistic scenario showing two actors (e.g., a sending system
   and a receiving system) exchanging:
   - one **FHIR** instance (to anchor the pattern against an already-known
     structure, e.g., a `DocumentReference` or `MessageHeader`), and
   - one **non-FHIR** instance pointed at a `contained` (or external)
     `Binary`, using one or more of the codes allowed by
     `examplescenario-instance-type`, for example:
     - CDA — `structureType = http://terminology.hl7.org/CodeSystem/v2-0354#CDA` (v2-0354 includes `CDA`) with `structureVersion = "R2"` and `structureProfile = "http://hl7.org/cda/us/ccda/StructureDefinition/ContinuityofCareDocumentCCD"`, `content → Binary` with `contentType = application/cda+xml`.
     - v2.x — `structureType = http://terminology.hl7.org/CodeSystem/v2-0354#ADT_A01` (or similar v2 message code from that CodeSystem) with `structureVersion = "2.5.1"`, `content → Binary` with `contentType = application/hl7-v2`.
     - CSV / Word / PDF — per the ticket discussion (esilver's concern about
       needing a LogicalModel for every non-FHIR document): demonstrate the
       no-`structureProfile` case so readers see that binding is optional
       when no LogicalModel exists. `structureType` can use an extension
       to point at a mime-typed code, or a CodeSystem entry the author
       defines locally; minimally, `content → Binary` with
       `contentType = text/csv` / `application/msword` / `application/pdf`.
2. A short `process` / `step` block showing the operation so the scenario
   renders meaningfully in the generated diagram (the publisher uses the
   sibling `.plantuml` file to render flow pictures — see
   `examplescenario-example.plantuml`).

Wiring / publication changes:

- New authoring file
  `source/examplescenario/examplescenario-example-nonfhir.xml` (an
  `ExampleScenario` resource whose `id` is `example-nonfhir`).
- New companion diagram source
  `source/examplescenario/examplescenario-example-nonfhir.plantuml`
  mirroring the style of `examplescenario-example.plantuml`.
- Add an `<entry>` to
  `source/examplescenario/list-ExampleScenario-examples.xml` referencing
  `ExampleScenario/example-nonfhir` with a `title` /
  `description` extension consistent with the two existing entries.
- Extend the narrative at
  `source/examplescenario/examplescenario-notes.xml` with a new
  `<a name="nonfhir"> </a>` / `<h3>Non-FHIR data</h3>` section that:
  - states the rule (`content` must point at a `Binary`; use
    `Binary.contentType` for the mime type; `structureType` identifies the
    structure via v2-0354 or any extensible-binding code; `structureVersion`
    is required when not FHIR; `structureProfile[x]` optionally names a
    LogicalModel / profile / external schema);
  - answers the concern raised by esilver on 2024-02-12 in the ticket
    comments — a LogicalModel is **not** required to include a non-FHIR
    instance; it is an option when validation against a model is desired;
  - links to the new `example-nonfhir` worked example.

No change to `structuredefinition-ExampleScenario.xml`, to
`valueset-examplescenario-instance-type.xml`, or to extension definitions
is required; the resolution explicitly rejected widening the schema and
rejected repurposing existing extensions ("Don't think we need to use any
of these extensions." — mckenzie, 2024-02-12).

### Design Rationale

The resolution is deliberately minimal. Three alternatives were discussed
and rejected in the ticket comments:

- **Adding a dedicated non-FHIR structure element** (the original ask).
  Rejected because the mime type on `Binary` already carries the structural
  information needed to dispatch the instance.
- **Reusing existing IG extensions that flag non-FHIR instance examples
  elsewhere in the spec** (mckenzie, 2024-02-12: "Should we modify the
  context of those extensions to also make them useable for
  ExampleScenario.instance, or something different here"). Rejected —
  "Don't think we need to use any of these extensions." The
  `structureType` + `content → Binary` pattern is considered sufficient.
- **Requiring a LogicalModel for every non-FHIR instance** (esilver
  objection). Rejected by the resolver; LogicalModels are optional.

The chosen path — a published worked example plus a narrative callout —
is consistent with how FHIR routinely closes "the capability is there,
but undiscoverable" gaps elsewhere (see the `examplescenario-example-laborder`
precedent for a second worked scenario added purely for pedagogy, listed
alongside the general example in `list-ExampleScenario-examples.xml`).

### Step-by-Step Tasks

#### HL7/fhir

1. **Author the new ExampleScenario example resource.**
   - File: `source/examplescenario/examplescenario-example-nonfhir.xml`
   - Action: Create a new `<ExampleScenario>` authored resource with
     `id = example-nonfhir` modelled after
     `source/examplescenario/examplescenario-example.xml`.
   - Details:
     - Include at least two `<actor>` entries (e.g., a sending EHR and a
       receiving document repository).
     - Define at least three `<instance>` blocks:
       - one FHIR instance (e.g., `structureType` using
         `http://hl7.org/fhir/fhir-types#DocumentReference`) to anchor the
         scenario in a known structure;
       - one CDA instance with
         `structureType.system = http://terminology.hl7.org/CodeSystem/v2-0354`
         / `code = CDA`, `structureVersion = "R2"`, a
         `structureProfile` pointing to a public CDA profile URL, and
         `content` referencing a contained `Binary` whose
         `contentType = application/cda+xml`;
       - one "unstructured" instance (e.g., CSV or PDF) with an
         appropriate v2-0354 or locally-defined `structureType` code,
         **no** `structureProfile[x]`, and `content` referencing a
         contained `Binary` with `contentType = text/csv` (or
         `application/pdf`), to make explicit that a LogicalModel is
         optional.
     - Supply a `<process>` with enough `<step>` entries (each with
       `<operation>` → `<request>` / `<response>` as appropriate) to
       produce a meaningful rendered diagram.
     - Honour invariant `exs-1` by always populating `structureVersion`
       on non-FHIR instances.
     - Populate `url`, `version`, `name`, `title`, `status`,
       `experimental`, `date`, and `publisher` consistently with the
       existing `examplescenario-example.xml`.

2. **Author the companion PlantUML diagram source.**
   - File: `source/examplescenario/examplescenario-example-nonfhir.plantuml`
   - Action: Create a sequence diagram mirroring the style of
     `examplescenario-example.plantuml`, depicting the actors and the
     exchange of the FHIR + non-FHIR instances defined above.
   - Details: Keep node / participant names aligned with the `key`
     values used in `<actor>` and `<instance>` in the XML so the
     generated page cross-links correctly.

3. **Register the new example in the examples list.**
   - File: `source/examplescenario/list-ExampleScenario-examples.xml`
   - Action: Add a third `<entry>` after the existing
     `examplescenario-example-laborder` entry.
   - Details: Use the same shape as the first two entries:
     ```xml
     <entry>
       <extension url="http://hl7.org/fhir/build/StructureDefinition/description">
         <valueString value="Example of examplescenario - Exchange of non-FHIR data (CDA / CSV) via Binary"/>
       </extension>
       <extension url="http://hl7.org/fhir/build/StructureDefinition/title">
         <valueString value="examplescenario-example-nonfhir"/>
       </extension>
       <item>
         <reference value="ExampleScenario/example-nonfhir"/>
         <display value="Non-FHIR data exchange"/>
       </item>
     </entry>
     ```

4. **Document the non-FHIR pattern on the ExampleScenario page.**
   - File: `source/examplescenario/examplescenario-notes.xml`
   - Action: Insert a new `<a name="nonfhir"> </a>` anchor followed by
     an `<h3>Non-FHIR data</h3>` section at the end of the file
     (after the existing `Contained` section).
   - Details: Narrative should cover:
     - The three levers that make non-FHIR data work without schema
       change: `ExampleScenario.instance.structureType` (Coding,
       extensible binding, already includes v2-0354 for CDA / v2), the
       required `structureVersion` when `structureType` is not FHIR
       (invariant `exs-1`), and `content` pointing to a `Binary` whose
       `contentType` is the mime type (cite the `comment` at line 711
       of `structuredefinition-ExampleScenario.xml`).
     - The optionality of `structureProfile[x]` — a LogicalModel /
       profile / external schema URL is allowed but not required.
     - An explicit link to the new `ExampleScenario/example-nonfhir`
       worked example.

### Open Questions

1. **Choice of v2-0354 code(s) for the CDA and "unstructured" instances.**
   The ticket comments do not pin specific codes. A reasonable default
   is `CDA` for the CDA example; for CSV / PDF an extensible-binding
   approach using a locally-defined code or an ad-hoc system is
   preferable to shoehorning into v2-0354. This should be confirmed with
   FHIR-I before merging.
2. **Whether to exercise `structureProfile[x]` as `canonical` vs `uri`.**
   The ticket does not specify. Recommendation: show the `uri` form for
   the CDA example (external CDA template URL) and omit
   `structureProfile` entirely for the CSV / PDF example to illustrate
   the LogicalModel-optional case esilver raised.
3. **Briefing scope.** The cached per-repo briefing was generated with
   `ticket_key = FHIR-42980`, so its *Ticket-Relevant Paths* section is
   scoped to the mapping-language ticket. The briefing's Facts,
   Authoring roots, Generated areas, Recommended Change Recipes, and
   Warnings / Gotchas are repo-wide and remain authoritative for this
   plan (clone HEAD and playbook SHA both match `meta.json`, so the
