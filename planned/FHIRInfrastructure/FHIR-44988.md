## Feature Proposal

### Problem Statement

When data flows through intermediaries (IPS bundles, multi-source aggregators, deduplication pipelines), implementers want to preserve the originating-server identity of each resource alongside the data itself. `meta.source` records the *server*, but there is no consensus mechanism on the resource for recording the originating *resource URL* in `Identifier.type`.

The reporter checked v2-0203 (`identifierType`) and found `RI` ("Resource identifier") to be a generalized v2 concept that does not cleanly describe a FHIR resource URL, and no other existing code is a good fit. The Multi-domain-environments Zulip thread (`Resource id as identifier`, message 426475585) confirmed there is no off-the-shelf code that implementers should adopt.

### Proposed Change

**1. New concept in `http://terminology.hl7.org/CodeSystem/v2-0203` (HL7/UTG):**

| Field | Value |
|-|-|
| code | `FURL` |
| display | `FHIR resource URL` |
| definition | `The full URL of the FHIR resource on an originating server. Preserving the URL as an identifier is useful especially for deduplication purposes when the resource is transmitted away from the server.` |
| status property | `active` |

The concept must follow the existing v2 concept-property pattern used throughout `cs-v2-0203.xml` (e.g., the existing `RI` entry at lines 1733ŌĆō1749 of `input/sourceOfTruth/v2/codeSystems/cs-v2-0203.xml`):

- Wrapper `<concept id="..."/>` with the next available numeric `id` in the file (the file currently ends at `id="2242"` per the file's 2,242-line length / contiguous numbering convention; UTG editors should pick the next free id following the file's existing numbering rules).
- `<code>`, `<display>`, `<definition>` matching the table above.
- A `<property><code value="v2-concCommentAsPub"/><valueString value="..."/></property>` carrying the same definition text (this matches the `RI`, `RN`, and `RPH` siblings in the same file).
- A `<property><code value="status"/><valueCode value="active"/></property>`.
- No deprecation properties.
- Bump `<version>` of the code system from `5.0.0` to the next patch/minor as set by UTG release procedure for additive content.

**2. FHIR Core ValueSet `identifier-type` (HL7/fhir):**

`source/terminologies/valueset-identifier-type.xml` enumerates a curated subset of `v2-0203` codes (`DL`, `PPN`, `BRN`, `MR`, `MCN`, `EN`, `TAX`, `NIIP`, `PRN`, `MD`, `DR`, `ACSN`, `UDI`, `SNO`, `SB`, `PLAC`, `FILL`, `JHN` ŌĆö see lines 30ŌĆō84 of that file). Add a `<concept><code value="FURL"/></concept>` entry inside the existing `<include>` block so this code is part of the extensible binding suggestion users see for `Identifier.type`.

**3. FHIR-side usage rule:**

The resolution adds an interpretation rule that does not exist in the v2 code system itself: when this `Identifier.type` is used inside a FHIR `Identifier`, `value` MUST be the full URL of the resource on the originating server, and `system` MUST be `urn:ietf:rfc:3986`. This belongs in narrative on the FHIR side ŌĆö most naturally as a row/note in the existing identifier-registry / datatypes documentation in `HL7/fhir`. (See Open Questions for path determination.)

**Example FHIR `Identifier`:**

```json
{
  "use": "official",
  "type": {
    "coding": [{
      "system": "http://terminology.hl7.org/CodeSystem/v2-0203",
      "code": "FURL",
      "display": "FHIR resource URL"
    }]
  },
  "system": "urn:ietf:rfc:3986",
  "value": "https://server.example.org/fhir/Patient/123"
}
```

### Design Rationale

- **Extends the existing v2-0203 code system rather than minting a new one.** This keeps `Identifier.type` consumers from having to learn another binding, and matches the pattern used for every other identifier-type code in FHIR.
- **Reuses `urn:ietf:rfc:3986` as the identifier `system`.** This is the standard FHIR pattern for URL-as-identifier (`urn:ietf:rfc:3986` is already the system used for `ValueSet.identifier`/`CodeSystem.identifier` URN-OID identifiers in this repo), so the rule "when used in FHIR, value is the URL and system is rfc:3986" is consistent with prevailing FHIR practice.
- **Re-using `RI` was rejected.** The existing `RI` concept is defined as "A generalized resource identifier" with v2-style use-case text about respiratory therapists, exam rooms, etc. (see the `v2-concComment` property at line 1739 of `cs-v2-0203.xml`). It is not specific to FHIR resource URLs and would create ambiguity in cross-paradigm mappings.
- **Adding only to the v2 code system would technically suffice for the v2 value set** (`vs-v2-0203.xml` includes the whole code system and so picks up the new concept automatically), but the FHIR Core `identifier-type` ValueSet is an enumerated subset and would silently *not* expose `FURL` unless edited explicitly. The resolution's "and will add it to the value set" is therefore read here as covering the FHIR Core ValueSet as well; this should be confirmed in the Open Questions.
