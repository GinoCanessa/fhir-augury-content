## 1. Resolution Summary

The resolution directs creation of a new extension with the following characteristics:

- **URL:** `http://hl7.org/fhir/StructureDefinition/binding-conceptDomain`
- **Cardinality:** 0..1
- **Value type:** `Coding`
- **Context:** `ElementDefinition.binding`

The extension allows FHIR StructureDefinitions to associate an element's binding with a v2/v3 *concept domain*, expressed as a Coding. This is needed so that FHIR StructureDefinitions can fully represent v2 and v3 definitions. In R5 the binding strength would be `example`; in R6+ it would be `unbound`.

The broader concept-domain mechanism also involves:
- A CodeSystem defining all concept domains (code + display + markdown definition)
- ValueSets connected to concept domains via either a CodeSystem supplement property or via `ValueSet.topic`

However, **this ticket's scope is limited to creating the extension itself**.

---

## 2. Feature Proposal

### What Changes

Create a new FHIR extension resource (`StructureDefinition`) named `binding-conceptDomain` in the FHIR Extensions Pack. The extension:

1. Has context `ElementDefinition.binding` — it is placed on the binding element of any element definition.
2. Carries a `Coding` value representing the concept domain.
3. Has cardinality 0..1 (at most one concept domain per binding).
4. Follows the same structural pattern as existing `ElementDefinition.binding` extensions (e.g., `elementdefinition-bindingName`, `elementdefinition-isCommonBinding`).

### Why

FHIR StructureDefinitions are used to represent v2 and v3 data definitions, but currently lack a mechanism to express concept domain associations — a core vocabulary construct in HL7 v2/v3. This extension closes that gap, enabling complete representation of legacy HL7 vocabulary bindings within FHIR conformance resources.

    <valueInteger value="1"/>

  <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
    <valueCode value="trial-use"/>
  </extension>

  <identifier>

### Step 2: Assign an OID

Replace the placeholder `urn:oid:2.16.840.1.113883.4.642.5.XXXX` with the next available OID from the HL7 OID registry (branch `2.16.840.1.113883.4.642.5`). Inspect existing extensions in the repo to find the next sequential number.

### Step 3: Validate the IG Build

Run the FHIR IG Publisher build to verify:
- The extension is discovered and rendered correctly
- No QA errors are introduced
- The extension page appears in the published output

### Implementation Notes

1. **Naming convention:** The ticket explicitly specifies the URL `http://hl7.org/fhir/StructureDefinition/binding-conceptDomain`. This uses a `binding-` prefix rather than the `elementdefinition-` prefix used by older sibling extensions. The file and `id` should use `binding-conceptDomain` to match the URL tail.

2. **FMM level:** Set to 1 (draft/trial-use) since this is a new extension. The `standards-status` is set to `trial-use` to match similar new extensions.

3. **No binding on the Coding value:** The Coding is not bound to a specific CodeSystem/ValueSet in the extension definition itself, because the concept domain CodeSystem is defined externally (potentially in UTG or as a separate resource). Consumers will reference concept domains by their code+system in the Coding.

4. **No sub-extensions:** This is a simple-value extension (no nested extensions), consistent with the resolution directive of "0..1 of type Coding."

5. **Work group code:** `fhir` (matching FHIR Infrastructure WG convention used by all sibling extensions).

## 7. Open Questions

1. **OID assignment:** A new OID under `2.16.840.1.113883.4.642.5` must be allocated. The implementer should check the registry or use the next available sequential number.

2. **Concept domain CodeSystem:** The ticket mentions a CodeSystem defining all concept domains (code + display + markdown definition). This may be a separate piece of work (possibly in HL7/UTG). The extension itself does not depend on this CodeSystem existing yet.

3. **Documentation / rendered HTML:** A comment from mfaughn asks about rendering concept domain associations in HTML output. This is a tooling concern (FHIR IG Publisher rendering) and may require a separate ticket.
