## Resolution Summary

The ticket was filed to add a new `generated` code to the `request-status`
code system so that `ImmunizationRecommendation.status` (requested by the
sibling ticket FHIR-53251) could express a request that has been generated
by a decision-support engine without being claimable as `active` or
`completed`.

The approved resolution (type **Persuasive with Modification**) replaces
that proposal with a broader, pattern-level change:

> "We will define a new status called `activated`. Will update `active`
> and `completed` to be specializations of `activated`."
>
> Definition: *"Indicates that the Request has transitioned to active but
> that there is no feedback loop that will allow evaluation of whether
> the authorized action has successfully been completed — i.e. it is
> unknown whether the Request is still active, on hold, completed, or
> revoked, but it is known that the resource isn't draft or marked as
> 'in error'."*

So instead of adding `generated` as a sibling leaf code, the resolution
adds `activated` as a new parent concept in the Request Status hierarchy
and re-parents `active` and `completed` under it.

## Feature Proposal

### Problem Statement

The Request Pattern's status code system (`http://hl7.org/fhir/request-status`)
currently defines: `draft`, `active`, `on-hold`, `entered-in-error`,
`unknown`, and (nested under `ended`) `completed` and `revoked`. Systems
that generate a Request but have no feedback loop to determine whether it
is still in force, fulfilled, on hold, or revoked have no appropriate code
to use. Consumers looking at a Request driven by such a system cannot
reliably assert `active` (it may have already ended) nor `completed` (it
may still be open), and `unknown` is too weak because it does not
distinguish "no feedback loop" from "not sure if this was ever valid".

ImmunizationRecommendation (FHIR-53251) is the concrete driver: it is
aligned to the Request pattern but is typically machine-generated with no
downstream lifecycle tracking.

### Proposed Change

Edit the core Request Status code system and its bound value set:

1. **`CodeSystem/request-status`** — add a new concept `activated` and
   move the existing `active` and `completed` concepts so that they are
   nested under `activated`. Concept definition (verbatim from the
   resolution):

   > "Indicates that the Request has transitioned to active but that
   > there is no feedback loop that will allow evaluation of whether the
   > authorized action has successfully been completed - i.e. it is
   > unknown whether the Request is still active, on hold, completed, or
   > revoked, but is is known that the resource isn't draft or marked as
   > 'in error'."

   Proposed concept XML (using the nested-concept style already used by
   `ended`/`completed`/`revoked` in the file):

   ```xml
   <concept>
     <code value="activated"/>
     <display value="Activated"/>
     <definition value="Indicates that the Request has transitioned to active but that there is no feedback loop that will allow evaluation of whether the authorized action has successfully been completed - i.e. it is unknown whether the Request is still active, on hold, completed, or revoked, but is is known that the resource isn't draft or marked as 'in error'."/>
     <concept>
       <code value="active"/>
       <display value="Active"/>
       <definition value="The request is in force and ready to be acted upon."/>
     </concept>
     <concept>
       <code value="completed"/>
       <display value="Completed"/>
       <definition value="The activity described by the request has been fully performed. No further activity will occur."/>
     </concept>
   </concept>
   ```

   Note: `completed` is currently nested under `ended`. Because FHIR
   `CodeSystem` nested `<concept>` syntax only expresses a single parent,
   moving `completed` under `activated` removes the lexical nesting under
   `ended`. To preserve the subsumption relationship to `ended`, add a
   `<concept.property>` entry (or equivalent `parent`/`child`
   relationship — the existing file does not declare a
   `hierarchyMeaning`, so that will likely need to be added as well).
   This is flagged in Open Questions.

2. **`ValueSet/request-status`** — no change required; it already
   `<compose><include>`s the whole `http://hl7.org/fhir/request-status`
   system, so the new `activated` code automatically flows into every
   resource bound to this value set.

3. **Request pattern narrative** — the Request pattern spreadsheet /
   page must be refreshed to describe the new `activated` state, its
   intended usage, and where it sits in the state machine.
   (`source/request/request-spreadsheet.xml` mentions `request-status`;
   the publisher generates the user-facing page from the spreadsheet.)

4. **Sibling ticket alignment (FHIR-53251)** — the approved wording of
   this ticket supersedes the sibling ticket's original plan to use a
   new `generated` code. Any work on FHIR-53251 should bind
   `ImmunizationRecommendation.status` to the (unchanged) `request-status`
   value set, expecting implementers to use `activated` for generated-
   only recommendations. Confirm before implementation.

### Design Rationale

- The code system is **normative** (standards-status extension at top
  of the file), so adding a new non-breaking concept is preferred over
  introducing a parallel code system for ImmunizationRecommendation.
- A new parent `activated` keeps the leaf codes `active` and
  `completed` unchanged (same codes, same definitions, same bindings
  everywhere downstream), so existing instances and implementations
  remain valid — only the *hierarchy* changes.
- Rejecting the originally-proposed `generated` leaf avoids polluting
  every request-bound resource (CarePlan, CommunicationRequest,
  DeviceRequest, NutritionOrder, RequestOrchestration, ServiceRequest,
  etc.) with an ImmunizationRecommendation-specific concept that would
  make little sense in most of their state machines (concern raised by
  @jmandel on the 2025-11-24 FHIR-I core call).
- Using a parent concept lets ImmunizationRecommendation bind to
  `activated` semantically ("activated, no feedback loop") without
  forcing consumers of every other request-bound resource to reason
  about a new code they will never emit.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Houses the authoritative `CodeSystem/request-status`,
  `ValueSet/request-status`, and the Request pattern specification
  that defines the state machine.
- **Affected Files:**
  - `source/request/codesystem-request-status.xml` — add `activated`
    parent concept; nest existing `active` and `completed` concepts
    underneath; adjust hierarchy declarations (see Open Questions on
    `completed` being under both `ended` and `activated`).
  - `source/request/request-spreadsheet-introduction.xml` and/or
    `source/request/request-spreadsheet-notes.xml` — extend the
    Request pattern prose to describe the new `activated` state, its
    semantics ("no feedback loop") and its relationship to `active`
    and `completed`.
  - `source/request/request-spreadsheet.xml` — verify (and update if
    needed) any state-table cells referencing `request-status` codes.
  - `source/request/valueset-request-status.xml` — **no edits**;
    verified that it includes the whole code system, so the new code
    propagates automatically.
  - `source/careplan/structuredefinition-CarePlan.xml`,
    `source/communicationrequest/structuredefinition-CommunicationRequest.xml`,
    `source/devicerequest/structuredefinition-DeviceRequest.xml`,
    `source/nutritionorder/structuredefinition-NutritionOrder.xml`,
    `source/requestorchestration/structuredefinition-RequestOrchestration.xml`,
    `source/servicerequest/structuredefinition-ServiceRequest.xml` —
    **no `<differential>` edits**. These resources bind to
    `ValueSet/request-status`; new codes flow through automatically.
    Their `<snapshot>` blocks will be regenerated by the publisher
    (do not hand-edit — briefing gotcha).
- **Change Scope:** **Moderate.** The CodeSystem itself is a small,
  surgical XML edit; the knock-on effect is that every resource bound
  to `request-status` gains a new permissible code. Narrative updates
  to the Request pattern page are also required. Normative status of
  the code system means the change needs an explicit discussion of
  whether hierarchy reorganization is backwards compatible (codes
  unchanged → yes at the instance level; parent/child relationships
  changed → could surprise subsumption-based tooling).

### Breaking Changes

- **No instance-level breaking change.** No codes are removed or
  renamed. Instances using `active`, `completed`, or any other
  existing code remain valid.
- **Potential tooling impact.** Re-parenting `active` and `completed`
  changes the subsumption graph. A terminology server that previously
  answered "is `completed` a child of `ended`?" with *yes* may now
  answer differently depending on how the multi-parent relationship
  is encoded (see Open Questions). Document the intended subsumption
  semantics explicitly.
- The code system is `<valueCode value="normative"/>`, so this change
  must be reviewed against normative-content change rules before
  publication.

### Open Questions

- **Hierarchy encoding for `completed`.** The resolution says
  `completed` is a specialization of `activated`, but the current
  file also nests `completed` under `ended`. Because `CodeSystem`
  nested `<concept>` syntax supports only one lexical parent, one
  of the following must be chosen:
  1. Keep `completed` nested under `ended` (preserving its
     "ended lifecycle" semantics) and declare the
     `completed is-a activated` relationship via a
     `<concept.property>` entry of type `parent`/`Coding`.
  2. Move `completed` to nest under `activated` and declare the
     `completed is-a ended` relationship via the same `property`
     mechanism.
  3. Use `<property>` entries for *both* parent relationships and
     avoid nesting for `completed` entirely.
  FHIR-I should confirm which representation is preferred before
  the XML edit is made.
- **`hierarchyMeaning` value.** The file currently has no
  `hierarchyMeaning`. Adding one is recommended but technically a
  change to a normative resource; confirm FHIR-I is comfortable
  declaring `is-a`.
- **Narrative wording.** The exact prose for the Request pattern
  page describing `activated` was not spelled out in the resolution
  beyond the concept definition; FHIR-I should review the
  narrative wording before commit.
- **Mirror concepts in `MedicationRequest` / `SupplyRequest`
  status code systems?** These have their own status code systems
  aligned with Request. The resolution is silent on whether
  Pharmacy / OO should mirror `activated` there; notify those WGs
  so they can file follow-on tickets if desired.
- **FHIR-53251 alignment.** Confirm with FHIR-I that the sibling
  ImmunizationRecommendation ticket will bind to the unchanged
  `ValueSet/request-status` and consume `activated` rather than
  reviving the original `generated` proposal.
