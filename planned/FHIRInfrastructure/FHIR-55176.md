## Resolution Summary

**Resolution:** *Not Persuasive with Modification* — the request to add an
`unknown`/`other` escape-valve concept to the `OperationKind` value set is
declined, but the committee agreed to address the underlying concern via
documentation. The exact resolution wording is:

> Will add a comment saying "If there is a need to pre-adopt new codes from
> a future version, set this element to `'operation'` and convey the
> additional concept in an extension."

In other words, the value set bound to `OperationDefinition.kind` is left
unchanged (it remains the two-concept `OperationKind` system: `operation`
and `query`), and a guidance comment is added to the
`OperationDefinition.kind` element directing implementers to fall back to
`operation` plus an extension when they need to convey a code that is not
yet in the value set.

## Feature Proposal

### Problem Statement

`OperationDefinition.kind` is a 1..1 element with a `required` binding to
the `OperationKind` value set
(`http://hl7.org/fhir/ValueSet/operation-kind`). That value set currently
contains only two concepts — `operation` and `query` — and includes no
`unknown` / `other` escape-valve concept. Lloyd McKenzie raised the concern
(comment 142, imported by Lloyd) that a mandatory element with a required
binding and no escape valve can become impossible to populate when an
implementer needs to express a kind that is not (yet) one of the two
defined values, e.g. when pre-adopting a future code.

### Proposed Change

Add a guidance comment on the `OperationDefinition.kind` element so that
the existing `<comment>` is extended (not replaced) with the resolution
text. Concretely, in
`source/operationdefinition/structuredefinition-OperationDefinition.xml`
inside the `<differential>` block, the element at `OperationDefinition.kind`:

- **Path:** `OperationDefinition.kind`
- **Type:** `code` (unchanged)
- **Cardinality:** 1..1 (unchanged)
- **Binding:** `required` to
  `http://hl7.org/fhir/ValueSet/operation-kind` (unchanged)
- **Comment (current):** `Named queries are invoked differently, and have
  different capabilities.`
- **Comment (proposed):** `Named queries are invoked differently, and have
  different capabilities. If there is a need to pre-adopt new codes from a
  future version, set this element to 'operation' and convey the
  additional concept in an extension.`

No change to the `OperationKind` `CodeSystem`
(`source/operationdefinition/codesystem-operation-kind.xml`) or the
`OperationKind` `ValueSet`
(`source/operationdefinition/valueset-operation-kind.xml`). They retain
exactly the two concepts `operation` and `query`.

No new extension is being defined here. The resolution refers to the
generic mechanism of attaching a modifierExtension/extension carrying the
pre-adopted concept; defining a specific extension URL is out of scope of
this ticket.

### Design Rationale

The committee considered (and declined) the original ask of adding
`unknown`/`other` to the value set. Adding such codes to a normative,
required-bound value set has well-known downsides: it weakens validation
(`unknown` is allowed everywhere), it does not actually convey the
intended semantic, and once normative, the value set is hard to evolve. The
chosen approach preserves the closed, two-concept `OperationKind` system
and instead documents the escape hatch already inherent in FHIR: when a
new concept needs to be conveyed before the value set catches up, pick the
nearest existing concept (`operation`) and carry the additional semantic
in an extension. This is the same pattern used elsewhere in the spec for
required-bound, normative value sets and is consistent with FHIR's general
extension-based forwards-compatibility approach.

## Implementation Plan

### Prerequisites

None. The change is self-contained inside one file in `HL7/fhir`.

### Step-by-Step Tasks

#### HL7/fhir

1. **Extend the `<comment>` on `OperationDefinition.kind` with the
   pre-adoption guidance.**
   - File: `source/operationdefinition/structuredefinition-OperationDefinition.xml`
   - Action: In the `<differential>` block, on the element with
     `id="OperationDefinition.kind"` /
     `<path value="OperationDefinition.kind"/>` (line 287 in the current
     clone), update the `<comment>` attribute (line 291) from:

     ```xml
     <comment value="Named queries are invoked differently, and have different capabilities."/>
     ```

     to:

     ```xml
     <comment value="Named queries are invoked differently, and have different capabilities. If there is a need to pre-adopt new codes from a future version, set this element to 'operation' and convey the additional concept in an extension."/>
     ```
   - Details: Append (do not replace) the new sentence so the existing
     guidance about named queries is preserved. Use straight ASCII
     single-quotes around `operation` to match the resolution wording
     exactly. Do not modify `<short>`, `<definition>`, `<min>`, `<max>`,
     `<type>`, any `<condition>`, `<isSummary>`, `<binding>`, or
     `<mapping>` on this element. Do not edit any `<snapshot>` content
     (none present in this file; `<snapshot>` is regenerated by the
     publisher in any case).
