## Feature Proposal

### Problem Statement

In R4 the spec published a ValueSet at the canonical URL
`http://hl7.org/fhir/ValueSet/operation-outcome`, intended as the
implementer-facing list of `OperationOutcome.issue.details` codes. Starting
in R4B that ValueSet was no longer published, but the URL was still
advertised as a known FHIR ValueSet (and is still referenced by external
guides and the CI build). The CodeSystem itself was relocated to THO at
`http://terminology.hl7.org/CodeSystem/operation-outcome`. The result is a
dangling canonical: implementers and validators that resolve
`http://hl7.org/fhir/ValueSet/operation-outcome` get a 404 against
hl7.org/fhir even though the URL is still treated as the registered
identifier in `source/oids.ini` (line 673 — see Repo Context below).

### Proposed Change

Two related sub-changes:

**A. R5/current main (`HL7/fhir`, `source/operationoutcome/`):**

1. Author a real ValueSet artifact for the canonical
   `http://hl7.org/fhir/ValueSet/operation-outcome`, modeled on the
   sibling `valueset-issue-severity.xml` and `valueset-issue-type.xml` in
   the same directory:
   - `id`: `operation-outcome`
   - `url`: `http://hl7.org/fhir/ValueSet/operation-outcome`
   - `identifier`: `urn:oid:2.16.840.1.113883.4.642.3.412` (the OID
     already reserved at `source/oids.ini:673`)
   - `name` / `title`: `OperationOutcome` / `Operation Outcome Codes`
   - `status`: `active`, `experimental`: `false`
   - `description`: text matching the historical R4 description ("Operation
     Outcome codes used by FHIR test servers (this is not standardized
     and includes codes for test purposes only)" — exact wording to be
     pulled from the R4 publication during implementation).
   - `compose.include[0].system`:
     `http://terminology.hl7.org/CodeSystem/operation-outcome` (THO; the
     codes already live there). No `version` pin — defer to the
     implementer's THO snapshot policy.
   - WG / FMM / standards-status extensions matching neighboring
     ValueSets in `source/operationoutcome/` (`fhir` WG, FMM as agreed
     by FHIR-I, status `trial-use` unless FHIR-I requests otherwise).
2. Wire the new file into the OperationOutcome publication: add it to
   `source/operationoutcome/list-OperationOutcome-operations.xml` (or
   the appropriate sibling list — currently the directory has lists for
   examples, operations, and packs but no list explicitly enumerating
   ValueSets; if the FhirCore convention is that ValueSets in a resource
   folder are auto-picked up, no list edit is required and the file is
   the only change).
3. Confirm `source/oids.ini` line 673 still points at the new ValueSet
   (no edit required — the OID is already reserved).

**B. R4B technical correction (only if/when opened):**

1. Backport the same ValueSet artifact onto the R4B branch of `HL7/fhir`
   (folder layout in R4B mirrors current main for `operationoutcome`).
2. No binding changes to the OperationOutcome StructureDefinition are
   needed — `OperationOutcome.issue.code` is bound to
   `ValueSet/issue-type`, `OperationOutcome.issue.severity` to
   `ValueSet/issue-severity`, and `OperationOutcome.issue.details`
   carries an example binding which can now resolve to the restored
   ValueSet.

### Design Rationale

- The OID for the URL is already reserved in `source/oids.ini` (line
  673), and the CodeSystem already lives on THO. Re-authoring a thin
  ValueSet that composes the THO CodeSystem is the lowest-impact way to
  fulfill the resolution while preserving the historical canonical URL.
- This matches the reporter's second option ("the valueSet appear in
  FHIR core like it did in R4") and is consistent with the WG's "fixed
  in R5" assertion.
- Moving the ValueSet to THO (the reporter's first option) would change
  the canonical URL and break implementers who still resolve
  `http://hl7.org/fhir/ValueSet/operation-outcome` — explicitly avoided.
- The pattern (ValueSet authored in the resource folder, composing a THO
  CodeSystem) is already used by the sibling
  `source/operationoutcome/valueset-issue-severity.xml` and
  `valueset-issue-type.xml`, so no new authoring convention is being
  introduced.
