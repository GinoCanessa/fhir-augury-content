## Resolution Summary

The resolution is **Persuasive**. The free-text `resolution_description` field is
not populated on this ticket; the agreed change follows directly from the
reporter's description and the single comment by `esilver`.

The `definition` of `ElementDefinition.comment` currently reads:

> "Explanatory notes and implementation guidance about the data element,
> including notes about how to use the data properly, exceptions to proper use,
> etc. (Note: The text you are reading is specified in
> ElementDefinition.comment)."

The parenthetical "(Note: The text you are reading is specified in
ElementDefinition.comment)" is factually wrong: the surrounding sentence is
itself the **definition** of `ElementDefinition.comment`, not its `comment`.
The fix is to remove that incorrect parenthetical from the `definition` cell of
the `ElementDefinition.comment` row in the authoring spreadsheet so the
self-referential meta-note is no longer false. The element's `comment` cell
("If it is possible to capture usage rules using constraints, that mechanism
should be used in preference to this element") is correct and is **not**
changed.

This corrects a regression introduced by the prior FHIR-28307 fix
(see comment by `esilver` on the ticket).

## Feature Proposal

### Problem Statement

`ElementDefinition.comment.definition` claims that "the text you are reading is
specified in `ElementDefinition.comment`", but the text is actually carried by
`ElementDefinition.definition`. The note was apparently introduced to
disambiguate the two fields after FHIR-28307, but it ended up attached to the
wrong cell, so the publisher now ships a self-referential note that is
self-contradicting in every generated artifact (HTML page, StructureDefinition
JSON/XML, FHIR snapshot, downstream IG renderings).

The cross-referenced Zulip thread `[conformance] Usage of
ElementDefinition.definition vs .comment` is the discussion that produced
FHIR-46200; it confirms the reporter's reading and there is no counter-proposal
in the chat or in Jira comments.

### Proposed Change

Edit the row for `ElementDefinition.comment` in the authoring spreadsheet
`source/datatypes/elementdefinition.xml` so the **Definition** cell drops the
incorrect parenthetical, leaving:

> "Explanatory notes and implementation guidance about the data element,
> including notes about how to use the data properly, exceptions to proper use,
> etc."

Concretely, in the row whose first cell is
`<Data ss:Type="String">ElementDefinition.comment</Data>` (clone HEAD `db79dcd`,
line 1306), the cell currently reading:

```xml
<Cell ss:StyleID="s89"><Data ss:Type="String">Explanatory notes and implementation guidance about the data element, including notes about how to use the data properly, exceptions to proper use, etc. (Note: The text you are reading is specified in ElementDefinition.comment)</Data><NamedCell ss:Name="_FilterDatabase"/></Cell>
```

becomes:

```xml
<Cell ss:StyleID="s89"><Data ss:Type="String">Explanatory notes and implementation guidance about the data element, including notes about how to use the data properly, exceptions to proper use, etc.</Data><NamedCell ss:Name="_FilterDatabase"/></Cell>
```

No other cell in this row changes:

- `Short` cell stays "Comments about the use of this element".
- `Comment` cell (style `s89`, after the `s90` padding cell) stays "If it is
  possible to capture usage rules using constraints, that mechanism should be
  used in preference to this element".
- Cardinality (`0..1`), type (`markdown`), summary flag (`y`), RIM mapping,
  v2 mapping, and 11179 mapping cells are untouched.

No row is added or removed. No other `ElementDefinition.*` row is touched.

### Design Rationale

- The reporter's reading is unambiguous and is not contested in the only ticket
  comment or in the cross-referenced Zulip thread; the resolution is
  "Persuasive" with no counter-text, which means the simplest correction —
  removing the wrong parenthetical — is the agreed fix.
- A more elaborate alternative ("(Note: this text is specified in
  `ElementDefinition.definition`)") was considered but is not required: the
  enclosing tooling already records which `ElementDefinition` field a string
  came from, so a meta-note on the value adds no information once the false
  one is removed. Keeping the definition free of self-referential meta-notes
  matches every other row in the spreadsheet.
- The change is confined to the canonical authoring source. The publisher
  regenerates the HTML pages, StructureDefinition JSON/XML, snapshots, and
  downstream views from this spreadsheet, so no parallel edit in
  `source/elementdefinition.html`, `source/elementdefinition-definitions.html`,
  or any generated StructureDefinition file is needed.

    edited directly.

    snapshot artifact.
- **Cross-repo touch points:** None. Documentation-only correction in a single
  authoring spreadsheet. The briefing's general FhirCore touch points
  (`HL7/UTG`, `HL7/fhir-extensions`, downstream IGs) do not apply.

### Open Questions

- **Whether to substitute a corrected meta-note rather than delete.** The
  resolution text is silent on this, the only ticket comment simply blames
  FHIR-28307, and the linked Zulip thread does not propose specific
  replacement wording. The plan above deletes the parenthetical outright,
  which is the minimal change that makes the definition true. If the work
  group wants to keep a self-locating hint, an acceptable alternative is
  "(Note: this text is the *definition* of ElementDefinition.comment; the
  separate `comment` field is shown below.)" — but adopting that requires an
  explicit note from the resolver and is **not** implemented here.
- **Spreadsheet-edit gotcha** (from briefing): editors that "fix"
  SpreadsheetML can silently rewrite the file (renumber styles, drop
  `_FilterDatabase` named cells, change line endings). The implementer must
  edit the cell with a plain text editor and diff against `db79dcd` before
  committing.
