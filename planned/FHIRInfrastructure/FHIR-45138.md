## Resolution Summary

**Resolution:** Not Persuasive with Modification.

The reporter's request to change `ImplementationGuide.global.profile` cardinality from 1..1 to 1..* was declined as written. Instead, the resolution applied is:

> ImplementationGuide.global is already 0..* with 1..1 type and 1..1 profile.
> We will add a rule that says that multiple ImplementationGuide.global of the same type means one of any of the same type applies.
> We will update all of the element descriptions under ImplementationGuide.global to be consistent with this clarification.
> We will also add an example using multiple globals.

Concretely, the deliverable is:

1. Cardinalities on `ImplementationGuide.global` (0..*), `ImplementationGuide.global.type` (1..1) and `ImplementationGuide.global.profile` (1..1) are **unchanged**.
2. A new normative rule is added: when multiple `global` entries share the same `type`, conformance to **any one** of the listed profiles is sufficient (i.e., they are OR'd, not AND'd).
3. The `short`/`definition`/`comment` text on each of the three element definitions under `ImplementationGuide.global` is updated to be consistent with that "OR-of-profiles per type" semantics.
4. A new example (or an extension of the existing example) is published demonstrating multiple `global` entries with the same `type` but different profiles.

## Feature Proposal

### Problem Statement

The current spec models default profiles for an Implementation Guide as a list of `(type, profile)` pairs where each entry's `profile` is 1..1. The reporter encountered a real-world case (US Core 6.1.0) where a single former profile (`Condition`) was split into two profiles (`Condition Encounter Diagnosis` and `Condition Problems and Health Concerns`). With the elements as currently specified, an IG cannot say "every `Condition` resource MUST conform to one of these two profiles" — there is no way to declare alternation per resource type.

The reporter proposed lifting `ImplementationGuide.global.profile` to 1..*. The resolution rejects that data-model change but acknowledges the underlying need by reinterpreting the existing repetition of `ImplementationGuide.global` itself: when the same `type` appears in multiple `global` entries, conforming to **any one** of those entries' profiles satisfies the global default for that type. This preserves the data model while delivering the semantics the reporter was asking for, and it keeps backward-compatible serialization for existing IGs that only declare one `global` per type.

The Zulip thread cross-referenced from the ticket ([implementers › Changing ImplementationGuide.global.profile to 1..*](https://chat.fhir.org/#narrow/stream/179166-implementers/topic/Changing.20ImplementationGuide.2Eglobal.2Eprofile.20to.201.2E.2E*)) is the public surface where this design was discussed; it is the source the resolution is drawn from.

### Proposed Change

The change set is documentation/profile-only — no element add/remove, no cardinality change, no type change.

1. **Element definition updates** in `source/implementationguide/structuredefinition-ImplementationGuide.xml` (differential only; the publisher regenerates `<snapshot>`):

   - `ImplementationGuide.global` (path; cardinality 0..*; BackboneElement; isSummary=true; **unchanged**)
     - `short`: change from "Profiles that apply globally" to something like **"Default profile(s) per resource type"**.
     - `definition`: rephrase to make explicit that `global` is a list, that each entry binds a single profile to a single resource type, and that **multiple entries with the same `type` mean instances of that type need only conform to *one* of the listed profiles** (logical OR per type).
     - `comment`: keep the link to `[Default Profiles](implementationguide.html#default)` and add a sentence explicitly stating the OR-per-type rule and that cross-type combinations remain AND (every type's default must be satisfied).

   - `ImplementationGuide.global.type` (cardinality 1..1; code; binding `http://hl7.org/fhir/ValueSet/resource-types`; **unchanged**)
     - `short`: clarify to e.g. **"Resource type that this default applies to"**.
     - `definition`: rephrase so it no longer reads as "all instances must conform to (this profile)" in isolation, but as "the resource type this default-profile entry constrains; the same `type` may appear in more than one `global` entry".
     - `comment`: leave the existing denormalization rationale, append a sentence that repeated `type` values are expected and meaningful (alternation).

   - `ImplementationGuide.global.profile` (cardinality 1..1; canonical(StructureDefinition); **unchanged**)
     - `short`: change from "Profile that all resources must conform to" to **"One acceptable profile for resources of this type"** (or similar — must drop the "all resources must conform" phrasing, which is the source of the misreading).
     - `definition`: rephrase to "A reference to **one** profile to which an instance of this `type` may conform; if the IG declares multiple `global` entries with the same `type`, instances satisfy the default by conforming to **any one** of those profiles."
     - `comment`: (currently absent) add a short comment cross-linking to the [Default Profiles](implementationguide.html#default) narrative.

2. **Narrative update** in `source/implementationguide/implementationguide-notes.xml`, in the existing `<a name="default"></a>` "Default Profiles" section. Add a new paragraph (and a small concrete example block) that:
   - States the rule: multiple `ImplementationGuide.global` with the same `type` are interpreted as "one of these profiles applies" (logical OR).
   - States the corollary: `global` entries with different `type` values are independent — each type's default(s) must be satisfied (logical AND across types).
   - Notes that an instance satisfies the default by conforming to any profile derived from one of the listed profiles (this is already stated for the single-global case and needs to be reaffirmed for the multi-global case).

3. **New example** demonstrating the multi-global pattern. Two viable shapes:

   - **Preferred:** add a new example file `source/implementationguide/implementationguide-example-globals.xml` with two `<global>` entries of `<type value="Condition"/>` pointing at the two US Core 6.1.0 Condition profiles (Encounter Diagnosis and Problems & Health Concerns), and register it in `source/implementationguide/list-ImplementationGuide-examples.xml`. This keeps the existing minimal `implementationguide-example.xml` as-is.
   - Alternative: extend `implementationguide-example.xml` itself with a second `<global>` for the same `type`. Lower change footprint but mixes two pedagogical concerns in one example.

   Example snippet for the preferred shape:

   ```xml
   <global>
     <type value="Condition"/>
     <profile value="http://hl7.org/fhir/us/core/StructureDefinition/us-core-condition-encounter-diagnosis"/>
   </global>
   <global>
     <type value="Condition"/>
     <profile value="http://hl7.org/fhir/us/core/StructureDefinition/us-core-condition-problems-health-concerns"/>
   </global>
   ```

### Design Rationale

- **Preserves wire-format compatibility.** No element cardinality changes, so existing IG instances remain valid and existing tooling does not need to handle a new repeating element. This is the dominant reason the original 1..* proposal was declined.
- **Uses an already-repeating structure.** `ImplementationGuide.global` is already 0..*; the resolution simply assigns previously-undefined semantics to the case where the same `type` appears more than once. Prior to this clarification, repeating `type` was effectively ambiguous (would it have meant AND? OR? duplicate-and-error?), so this is also a latent-defect fix.
- **Mirrors the existing "derived profile" allowance.** The current narrative already says "a resource can conform to the default profile by conforming to any profile derived from it." Extending that to "or any one of the listed profiles for the same type" is a natural generalization.
- **Keeps `profile` 1..1 inside a `global` entry.** Each entry remains a `(type, profile)` tuple, which simplifies tooling that walks the list — no list-of-lists.
- The Zulip thread referenced in the ticket (the only out-of-Jira discussion linked) is where this design was negotiated; no alternative was retained that would require a data-model change.

### Breaking Changes

No breaking changes identified at the data-model level: cardinalities, types, bindings, and constraints on `ImplementationGuide.global*` are unchanged, so all existing `ImplementationGuide` instances remain valid against the new StructureDefinition.

There is, however, a **semantic clarification with potential conformance impact** for IGs / tooling that previously relied on "the same `type` appears at most once in `global`":

- IG validators that treated repeated-`type` `global` entries as an error (or that AND'd the profiles together) will now be wrong. They must update to interpret repeated `type` as a per-type profile alternation (OR).
- IG authors who unintentionally listed the same `type` more than once previously had undefined behavior; under the new rule their IG suddenly has a precise (and probably looser) conformance meaning. This is worth a note in the IG release notes.

### Related Specifications

- Conformance / validation guidance pages (`profiling.html`, `validation.html` if they discuss IG-level defaults). No file edit identified as required here from the resolution; flagged as an open question.
