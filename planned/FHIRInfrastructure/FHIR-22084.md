## 1. Resolution Summary

**Ballot Comment (PRO #44):**
> "Also include requirements for SoF Launch support as client or app."

**Resolution Description (Persuasive):**
> "Will clarify SoF Launch specific requirements for EHRs."

The ballot commenter (Keith W. Boone) identified that the IG specifies SMART on FHIR (SoF) Launch requirements only at a high level, without clearly distinguishing the specific launch-related obligations for EHRs (acting as launch platforms / authorization servers) versus external PRO apps (acting as SMART clients). The resolution commits to clarifying EHR-specific SoF Launch requirements.

---

## 2. Feature Proposal

### Current State

The PRO IG currently addresses SMART on FHIR in three places:

1. **`source/pages/capstatements.md` — "SMART on FHIR App Interactions" subsections:**
   - **EHR actor:** Requires support for "EHR Launch Sequence" from the SMART on FHIR App Authorization guide, and information retrieval using US Core profiles during launch.
   - **External PRO Administration System actor:** Requires support for "EHR Launch Sequence" as a client, and requesting information from EHR during launch.

2. **`source/pages/guidance.md` — Implementation Step 4:**
   - **EHR actor (Step 4):** "Implement SMART on FHIR Authorization Server" with a link to SMART on FHIR Sandbox.
   - **External PRO Administration System (Step 4):** "Implement SMART on FHIR Authorization for clients" — register with EHR's Authorization Server to enable launching from within the EHR.

3. **`source/pages/pro-overview.md` — Data Flow DF3:**
   - Describes SMART on FHIR App launch from EHR with patient context (Step 4), but only covers the **EHR Launch** flow.

### Gaps Identified

- **Missing granularity on EHR-side launch obligations:** The IG says EHRs SHALL support "EHR Launch Sequence" but does not enumerate specific SoF Launch capabilities that EHRs must declare (e.g., `launch-ehr`, `context-ehr-patient`, `permission-patient`, `sso-openid-connect`).
- **No Standalone Launch requirements:** Related ticket FHIR-22079 covers this separately, but the current SoF section doesn't acknowledge that Standalone Launch could also be relevant (e.g., patient self-service on a tablet).
- **Client vs. Server roles are implicit:** The "SMART on FHIR App Interactions" subsections mix client and server responsibilities. EHR requirements should explicitly state the EHR is acting as a **SMART authorization server and resource server**, while External PRO Apps are acting as **SMART clients**.
- **Link to outdated SMART specification:** Current links point to `http://docs.smarthealthit.org/authorization` rather than the HL7-published `http://hl7.org/fhir/smart-app-launch`.

### Proposed Changes

1. **Clarify EHR-specific SoF Launch requirements** in `capstatements.md` under the EHR "SMART on FHIR App Interactions" subsection:
   - Explicitly state the EHR acts as a **SMART on FHIR Authorization Server** and **FHIR Resource Server**.
   - Enumerate the specific SMART capabilities the EHR SHALL declare (e.g., `launch-ehr`, `context-ehr-patient`, `permission-patient`).
   - Require the EHR to support token introspection or token validation for the External PRO App.

2. **Clarify External PRO App SoF requirements** as a SMART **client/app**:
   - Explicitly state the External PRO App acts as a **SMART on FHIR client application**.
   - Require the External PRO App to support the SMART App Launch protocol as a **confidential client** or **public client** (as appropriate).
   - Require the app to request appropriate scopes (e.g., `patient/QuestionnaireResponse.write`, `patient/Questionnaire.read`).

3. **Update SMART on FHIR links** from `http://docs.smarthealthit.org/authorization` to `http://hl7.org/fhir/smart-app-launch`.

4. **Update guidance.md** — Align Step 4 implementation guidance for both EHR and External PRO actors with the clarified requirements.

## 4. Implementation Plan

### Step 1: Update `source/pages/capstatements.md` — EHR "SMART on FHIR App Interactions"

**Location:** The "SMART on FHIR App Interactions" subsection under "EHR or Care Delivery Health IT System Conformance Requirements" (~line 120 in capstatements.md).

**Current text (approximate):**
```markdown
##### SMART on FHIR App Interactions 

For EHR or Care Delivery Health IT Systems supporting the collection of PRO data 
using External PRO Administration System by passing patient context information 
the following requirements apply. 

The EHR or Care Delivery Health IT System **SHALL** support the **EHR Launch Sequence** 
from the [SMART on FHIR App Authorization guide](http://docs.smarthealthit.org/authorization) 
applicable to the EHR.

The EHR or Care Delivery Health IT System **SHALL** support information retrieval during 
**EHR Launch Sequence** using tokens and parameters for 
[US Core profiles](http://hl7.org/fhir/us/core/index.html)

In order to receive collected PRO data, The EHR or Care Delivery Health IT System **SHALL** 
support the SDC QuestionnnaireResponse profile and interactions outlined earlier in the section.
```

**Proposed replacement:**
```markdown
##### SMART on FHIR App Interactions 

For EHR or Care Delivery Health IT Systems supporting the collection of PRO data 
using an External PRO Administration System by passing patient context information, 
the following requirements apply. In these interactions, the EHR acts as both a 
**SMART on FHIR Authorization Server** and a **FHIR Resource Server**.

The EHR or Care Delivery Health IT System **SHALL**:
* Support the **EHR Launch Sequence** as defined in the 
  [SMART App Launch IG](http://hl7.org/fhir/smart-app-launch) applicable to the EHR, 
  including the `launch-ehr` capability.
* Provide patient context (`context-ehr-patient`) to the launched application as part 
  of the EHR Launch flow.
* Support authorization and token exchange for the External PRO Administration System 
  acting as a SMART client, including issuing access tokens with appropriate scopes.
* Support the `permission-patient` capability to allow the External PRO Administration 
  System to request patient-level scopes (e.g., `patient/QuestionnaireResponse.write`, 
  `patient/Questionnaire.read`).
* Support information retrieval during the **EHR Launch Sequence** using tokens and 
  parameters for [US Core profiles](http://hl7.org/fhir/us/core/index.html).

The EHR or Care Delivery Health IT System **SHOULD**:
* Declare its supported SMART capabilities in the CapabilityStatement `security` element 
  using the SMART capabilities code system.

In order to receive collected PRO data, the EHR or Care Delivery Health IT System **SHALL** 
support the SDC QuestionnaireResponse profile and interactions outlined earlier in this section.
```

### Step 2: Update `source/pages/capstatements.md` — External PRO App "SMART on FHIR App Interactions"

**Location:** The "SMART on FHIR App Interactions" subsection under "External PRO Administration System Conformance Requirements."

**Current text (approximate):**
```markdown
##### SMART on FHIR App Interactions 

For External PRO Administration Systems interacting with EHRs to receive patient 
and other clinical context data from the EHR the following requirements apply:

The External PRO Administration System **SHALL** support the **EHR Launch Sequence** 
from the [SMART on FHIR App Authorization guide](http://docs.smarthealthit.org/authorization) 
as a client.

The External PRO Administration System **SHALL** request information from EHR during 
**EHR Launch Sequence** using tokens and parameters for 
[US Core profiles](http://hl7.org/fhir/us/core/index.html) as a client.
```

**Proposed replacement:**
```markdown
##### SMART on FHIR App Interactions 

For External PRO Administration Systems interacting with EHRs to receive patient 
and other clinical context data from the EHR, the following requirements apply. In 
these interactions, the External PRO Administration System acts as a 
**SMART on FHIR client application**.

The External PRO Administration System **SHALL**:
* Support the **EHR Launch Sequence** as defined in the 
  [SMART App Launch IG](http://hl7.org/fhir/smart-app-launch) as a SMART client 
  (confidential or public app, as appropriate for the deployment model).
* Be registered with the EHR's authorization server and support the OAuth 2.0 
  authorization code flow.
* Request appropriate patient-level scopes during authorization (e.g., 
  `patient/QuestionnaireResponse.write`, `patient/Questionnaire.read`, 
  `patient/Patient.read`).
* Use the patient context received during the EHR Launch flow to correctly 
  associate collected PRO data with the patient.
* Support information retrieval from the EHR during the **EHR Launch Sequence** 
  using access tokens for [US Core profiles](http://hl7.org/fhir/us/core/index.html).
```

- Link to the HL7-published SMART App Launch IG instead of `sandbox.smarthealthit.org`.
