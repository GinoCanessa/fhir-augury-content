## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** "Will add an example of each. Likely Patient.name.given for the first and something resolving to observations with different bindings on category or code for the second."

Concretely, two new worked examples must be added to the FHIR Core slicing-and-discriminator documentation:

1. A **positional slicing** example — preferably using `Patient.name.given` — that shows how to use `discriminator.type = position` together with `slicing.ordered = true` so that slices are distinguished by their index in the list rather than by the value of any element.
2. A **"by value within a referenced resource" slicing** example — preferably using a `List` or `Composition`/`Bundle`-style container of `Reference(Observation)` entries that is sliced by a value found inside the target Observation (e.g., `Observation.category` or `Observation.code`), demonstrating how the discriminator path uses `resolve()` and how the disambiguating constraint lives on a *second* profile (the Observation-side profile) rather than on the entry itself.

## Feature Proposal

### Problem Statement

The existing FHIR slicing documentation (`source/profiling.html` and the
worked-examples page `source/profiling-examples.html`) explains the five
discriminator types and gives examples for the most common patterns (slicing
`Patient.telecom` by `system`+`use`, "Fixed Order Slicing" of telecom with
`ordered=true` and no discriminator, slicing `Observation.component` by `code`
for blood-pressure components, slicing extensions, and the LipidReport example
that slices `DiagnosticReport.result` by the `targetProfile` of the
referenced resource).

Two important slicing patterns are described in prose but never demonstrated
end-to-end:

- **Positional slicing** (`discriminator.type = position`). The discriminator
  type is defined in `profiling.html` (around line 668) as: "Used to match
  slices based on their index. This is only possible if all but the last
  slice have a fixed cardinality where `min` > 0 and `min` = `max`. The last
  slice MAY have `min` != `max`. Typical example: slice on Practitioner.name
  to require that the first name be the usual name for the practitioner, and
  it must be present." Despite being one of the five discriminator types,
  there is **no worked StructureDefinition example** of positional slicing on
  the examples page. The closest existing example, "Fixed Order Slicing"
  (`profiling-examples.html` line 186), uses `ordered=true` with **no**
  discriminator and still constrains each slice with fixed values, which is
  not the same as discriminating purely by index.
- **Slicing by a value contained inside a referenced resource**
  (`discriminator.type = value` with a path that crosses `resolve()`).
  `profiling.html` describes this case (lines 617, 738, 743) and the
  LipidReport example uses `discriminator.type = profile` over
  `resolve()`, but there is no example that uses `discriminator.type = value`
  to slice on a coded element (e.g., `category` or `code`) of the referenced
  resource — the case the reporter explicitly asks for.

The reporter (Elliot Silver) notes that both patterns are non-intuitive
without a concrete example, and the resolver (Ward Weistra) agreed in the
"Persuasive" disposition that an example of each should be added.

### Proposed Change

Add two new worked examples to `source/profiling-examples.html`, immediately
after the existing "Fixed Order Slicing" section (which is conceptually
adjacent to positional slicing) and before the existing
"Slicing based on a referenced Resource Type" / LipidReport example.

#### Example A — Positional slicing on `Patient.name.given`

A short narrative section followed by a JSON instance and a
StructureDefinition snippet, modelled on the existing "Fixed Order Slicing"
example. The profile says:

- `Patient.name` cardinality is constrained so a single official name
  applies (e.g., `name` 1..1, with `name.use = official`).
- `Patient.name.given` cardinality is `2..2` (or `2..*` with the last slice
  open).
- `Patient.name.given` is sliced with `discriminator.type = position`,
  `discriminator.path = "$this"`, `slicing.ordered = true`,
  `slicing.rules = closed` (or `open` if the trailing slice is unbounded).
- Slice 1 (`firstName`) has `min=1`, `max=1` — the first `given` is the
  given name.
- Slice 2 (`middleName`) has `min=1`, `max=1` — the second `given` is the
  middle name.

Sample JSON instance:

```json
{
  "resourceType": "Patient",
  "name": [{
    "use": "official",
    "family": "Smith",
    "given": ["Jane", "Marie"]
  }]
}
```

Sample StructureDefinition snippet (XML, matching the style of the existing
examples):

```xml
<element id="Patient.name.given:">
  <path value="Patient.name.given"/>
  <slicing>
    <discriminator>
      <type value="position"/>
      <path value="$this"/>
    </discriminator>
    <description value="First given is the first name; second given is the middle name. Distinguished by position in the list."/>
    <rules value="closed"/>
    <ordered value="true"/>
  </slicing>
  <min value="2"/>
  <max value="2"/>
</element>

<element id="Patient.name.given:firstName">
  <path value="Patient.name.given"/>
  <sliceName value="firstName"/>
  <min value="1"/>
  <max value="1"/>
  <short value="First given name (must be present and appear first)"/>
</element>

<element id="Patient.name.given:middleName">
  <path value="Patient.name.given"/>
  <sliceName value="middleName"/>
  <min value="1"/>
  <max value="1"/>
  <short value="Middle name (must be present and appear second)"/>
</element>
```

The narrative must explicitly call out the rule from `profiling.html`: every
slice except the last must have a fixed cardinality with `min = max`, and
`slicing.ordered` must be `true`.

#### Example B — Slicing entries by a value in the referenced resource

A second new section on `profiling-examples.html` showing how to slice a list
of references where the disambiguator lives **inside the referenced
resource**. The most natural candidate (per the resolution) is a profile on a
`List` (or generic container) of `Reference(Observation)` where slices are
distinguished by `Observation.category`:

- Container resource: `List` (used in the discussion of `List.item` in
  `profiling.html` lines 738–743) with `List.entry.item` typed as
  `Reference(Observation)`.
- `List.entry` is sliced with:
  - `discriminator.type = value`
  - `discriminator.path = "item.resolve().category"`
  - `slicing.rules = open`, `ordered = false`.
- Slice "vitalSigns" — `min=1`, the entry must reference an Observation that
  conforms to a sibling profile `ObservationVitalSigns` whose `category` is
  fixed/patterned to the LOINC `vital-signs` code.
- Slice "laboratory" — `min=0`, the entry references an Observation
  conforming to a sibling profile `ObservationLaboratory` with `category`
  fixed/patterned to `laboratory`.

The example must include **both** the slicing profile (the List profile) and
the two referenced-Observation profiles (`ObservationVitalSigns` and
`ObservationLaboratory`), each fixing/patterning `Observation.category` to
the appropriate code from
`http://terminology.hl7.org/CodeSystem/observation-category`. This is
exactly the "the disambiguating constraint lives on a second profile that
fixes a value within the referenced resource" pattern that the ticket asks
to demonstrate.

Sample JSON instance:

```json
{
  "resourceType": "List",
  "status": "current",
  "mode": "snapshot",
  "entry": [
    { "item": { "reference": "Observation/bp-1" } },
    { "item": { "reference": "Observation/glucose-1" } }
  ]
}
```

Sample StructureDefinition snippet for the List-side slicing:

```xml
<element id="List.entry:">
  <path value="List.entry"/>
  <slicing>
    <discriminator>
      <type value="value"/>
      <path value="item.resolve().category"/>
    </discriminator>
    <description value="Entries are differentiated by the category of the referenced Observation"/>
    <rules value="open"/>
  </slicing>
</element>

<element id="List.entry:vitalSigns">
  <path value="List.entry"/>
  <sliceName value="vitalSigns"/>
  <min value="1"/>
</element>
<element id="List.entry:vitalSigns.item">
  <path value="List.entry.item"/>
  <type>
    <code value="Reference"/>
    <targetProfile value="http://example.org/fhir/StructureDefinition/ObservationVitalSigns"/>
  </type>
</element>

<element id="List.entry:laboratory">
  <path value="List.entry"/>
  <sliceName value="laboratory"/>
  <min value="0"/>
</element>
<element id="List.entry:laboratory.item">
  <path value="List.entry.item"/>
  <type>
    <code value="Reference"/>
    <targetProfile value="http://example.org/fhir/StructureDefinition/ObservationLaboratory"/>
  </type>
</element>
```

…and a snippet of the referenced-side profile (one shown for brevity):

```xml
<!-- ObservationVitalSigns -->
<element id="Observation.category">
  <path value="Observation.category"/>
  <min value="1"/>
  <patternCodeableConcept>
    <coding>
      <system value="http://terminology.hl7.org/CodeSystem/observation-category"/>
      <code value="vital-signs"/>
    </coding>
  </patternCodeableConcept>
</element>
```

### Design Rationale

- **Same page, same style.** The reporter explicitly contrasts these two
  patterns with the existing slicing examples; co-locating them on
  `profiling-examples.html` (already linked from `profiling.html` line 756)
  makes the pattern set complete without a new page.
- **Use the resolver's suggested artifacts.** The resolution explicitly
  names `Patient.name.given` for the positional case and "observations with
  different bindings on category or code" for the referenced-resource case,
  so the examples should match those choices to remain faithful to the
  approved resolution.
- **Pair the slicing profile with the referenced-side profile.** The whole
  point of the second example (per the ticket description: "depending on a
  second profile which fixes a value within the referenced resource") is
  that the discriminating constraint cannot live on the slicing profile
  alone — it must live on the *target* profile. The example must show both
  sides or it will not actually demonstrate the non-intuitive behaviour the
  reporter is asking to clarify.
- **Consistency with the `position` rule already in `profiling.html`.** The
  text on `profiling.html` (lines 668–672) already states the cardinality
  rule for positional slicing; the new example must obey it (slices except
  the last have `min = max`) so the documentation remains internally
  consistent.
- **No spec/StructureDefinition change required.** This is a documentation
  change only — the `position` discriminator type already exists in
  `discriminator-type` and `resolve()` is already a permitted FHIRPath
  function in the discriminator path, so no terminology, schema, or
  StructureDefinition file needs editing.
