## Resolution Summary

The ticket is marked **Persuasive**: the reported typo is accepted and a
correction is required. Under the *Key to Type Icons* heading on the Resource
Definitions specification page, the fifth bullet (the *Datatype* icon entry)
contains the misspelling **"uppwer case letter"**, which must be corrected to
**"upper case letter"**. No other behaviour, structure, or wording change is
called for.

## Feature Proposal

### Problem Statement

The Resource Definitions specification page renders a "Key to Type Icons"
bulleted list explaining what each icon in the resource/element hierarchy
diagrams represents. The fifth bullet — describing the *Datatype* icon for
complex types — currently reads:

> "All complex type names defined in this specification start with an
> **uppwer** case letter"

`uppwer` is a typo for `upper`. The defect is purely editorial: the page is
authored HTML that ships verbatim into the published spec, so the misspelling
is visible to every reader of the FHIR Core specification's type-icon legend.

The misspelling was confirmed in the live clone at
`source/formats.html`, line 128, in the `<li>` for the *Datatype* icon
inside the `<ul>` that follows the *Key to Type Icons* heading.

### Proposed Change

Replace the single token `uppwer` with `upper` on line 128 of
`source/formats.html`. The edit is a one-character correction inside this
sentence (within the *Datatype* `<li>`):

> "All complex type names defined in this specification start with an
> uppwer case letter"

becomes:

> "All complex type names defined in this specification start with an
> upper case letter"

No other characters in the bullet, no other bullets in the list, no anchors,
and no surrounding markup change. The list ordering, the icon `src` references,
and all other items in the `<ul>` are unaffected.

### Design Rationale

This is a "Technical Correction" resolved as Persuasive — the only design
question is *what is the correct spelling*, which is unambiguously `upper`.
The fix is intentionally minimal:

- It preserves the existing wording style of the sibling bullets (e.g., the
  *Primitive Datatype* bullet says "start with a lower case letter"), keeping
  the legend internally consistent.
- It avoids any rephrasing or restructuring of the legend, which would
  expand scope beyond what the resolution authorises.
- The HL7 publisher reads `source/*.html` pages as authored content, so the
  fix is applied directly in the HTML — no regeneration step or template
  change is required.
