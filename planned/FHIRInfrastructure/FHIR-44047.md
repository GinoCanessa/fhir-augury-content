## Resolution Summary

**Resolution:** Persuasive.
**Resolution description (verbatim):** *"Should make a note to review this as we review content for R6."*

The committee accepted the reporter's observation that the abbreviations
"e.g." (exempli gratia) and "i.e." (id est) are inconsistently punctuated
throughout the FHIR Core specification — most occurrences omit the trailing
comma that Chicago/AMA-style English prescribes ("e.g.,", "i.e.,"). The
committee resolved to fold the cleanup into the broader R6 content-review
sweep rather than landing a one-off mass edit at resolution time. The
implementation work is therefore a deliberate, reviewable pass over the
authored content of HL7/fhir during R6 ballot preparation, normalising
"e.g." → "e.g.," and "i.e." → "i.e.," (with judgement applied — see
Open Questions). The ticket title itself contains a typo ("gtatia" →
"gratia") which is informational only and does not require a code change.

## Feature Proposal

### Problem Statement

The FHIR Core specification mixes "e.g." / "i.e." (no trailing comma) with
"e.g.," / "i.e.," (with trailing comma) throughout its narrative — both
in the top-level HTML pages under `source/*.html` and in the prose-bearing
elements of resource definitions (`short`, `definition`, `comment`,
`requirements`, `purpose`, ValueSet/CodeSystem `description`, etc.) under
`source/<resource>/*.xml`. The dominant English style guides (Chicago,
AMA, APA) treat both abbreviations as parenthetical sentence connectors
that take a comma after the period; omitting the comma is widely regarded
as incorrect punctuation. The inconsistency makes the spec look unpolished
and complicates downstream tooling that scrapes definitions for display
(IGs, registries, validators) — those tools surface the inconsistency to
implementers who then file follow-up tickets. A baseline grep of the
current `master` clone (`db79dcd`) gives a sense of the asymmetry:

| Location | `e.g.` (no comma) | `e.g.,` (with comma) | `i.e.` (no comma) | `i.e.,` (with comma) |
|---|---|---|---|---|
| `source/*.html` | 26 | 618 | 12 | 135 |
| `source/**/*.xml` (recursive) | ~2,296 | (not counted) | ~362 | (not counted) |

The HTML pages are *already* mostly comma-correct; the long tail of
remaining `e.g.`/`i.e.` instances lives in the resource-definition XML
files, which are exactly what the R6 content review covers.

### Proposed Change

A targeted, file-by-file punctuation sweep of the FHIR Core authored
content, scoped strictly to authored prose:

1. In `source/*.html` (top-level specification pages), normalise every
   parenthetical "e.g." → "e.g.," and "i.e." → "i.e.,". This is a small
   set (~38 occurrences across 26 + 12) and can be done manually with
   per-file review.

2. In `source/<resource>/*.xml` resource and conformance files,
   normalise the same pattern — but only inside the *hand-edited* element
   text fields named below, and only inside `<differential>` (never
   `<snapshot>`, which is regenerated). The element fields in scope are:

   - `StructureDefinition.differential.element[*].short.@value`
   - `StructureDefinition.differential.element[*].definition.@value`
   - `StructureDefinition.differential.element[*].comment.@value`
   - `StructureDefinition.differential.element[*].requirements.@value`
   - `StructureDefinition.differential.element[*].meaningWhenMissing.@value`
   - `StructureDefinition.purpose.@value`
   - `ValueSet.description.@value`, `CodeSystem.description.@value`
   - `OperationDefinition.description.@value`,
     `OperationDefinition.parameter[*].documentation.@value`
   - `SearchParameter.description.@value`
   - `CapabilityStatement` narrative fields (`description`, `purpose`,
     `documentation`)
   - Any other `xs:string`-typed prose field authored by humans in the
     differential.

3. Skip files and regions that are derived/generated:

   - `source/datatypes/<type>.xml` (Excel spreadsheets per briefing —
     not real FHIR XML).
   - `source/<resource>/<resource>-spreadsheet.xml` where a sibling
     StructureDefinition exists (legacy/derived).
   - `<snapshot>` blocks anywhere in StructureDefinitions (regenerated
     by the publisher).
   - Code values (`<code value="..."/>`), URLs, file paths, regex
     patterns, FHIRPath/CQL expressions, and example JSON/XML inside
     narrative `<div>` blocks where the author intentionally quoted
     non-prose content.

4. The transformation is purely textual:
   - `e.g.` followed by whitespace → `e.g.,` followed by whitespace.
   - `i.e.` followed by whitespace → `i.e.,` followed by whitespace.
   - Already-correct `e.g.,` / `i.e.,` are left alone.
   - Trailing-punctuation cases (`e.g.)`, `e.g.:`) are left alone — the
     comma is not used when the abbreviation is followed immediately by
     other punctuation.
   - Sentence-final `e.g.` / `i.e.` (followed by `</p>` or `.`) are
     vanishingly rare and should be reviewed by a human; they are not
     mass-rewritten.

5. The ticket title's misspelling "gtatia" → "gratia" is metadata in
   Jira and **not** a content change in the repository; no code action
   is required.

### Design Rationale

- The committee explicitly chose to fold this into the R6 content
  review (resolution: *"Should make a note to review this as we review
  content for R6."*) rather than land a standalone mass-edit PR at
  resolution time. That implies the implementation should be staged as
  part of the R6 ballot prep cycle, where every author is already
  re-reading their resource's prose.
- Comment thread shows mild disagreement (Costa Teixeira) on whether
  comma-less usage is *strictly* incorrect; the resolution sided with
  the reporter (Jeff Brown) on the basis that consistency within the
  spec is itself a value, even if either form is defensible in
  isolation. The plan therefore prefers consistency over per-author
  preference: every occurrence becomes the comma form.
- The change is scoped to *authored* prose because the FHIR publisher
  regenerates `<snapshot>` blocks and other derived artifacts; editing
  those would be silently overwritten and create false diffs.
- A pure regex sweep is rejected (see Open Questions) because of the
  long tail of legitimate non-comma cases (`e.g.)`, `e.g.:`,
  end-of-sentence `e.g.`, code/URL fragments inside CDATA-style
  narrative). The plan instead treats this as a guided, per-file
  review with a regex *prompt* rather than a regex *replace*.

    etc.) is *not* sushi / IG-publisher; do not look for `input/`,
    `fsh-generated/`, `sushi-config.yaml` — none exist.

  `HL7/UTG`, `HL7/fhir-extensions`, and IG repos are not touched by a
  documentation-only change confined to HL7/fhir authored content.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole repository for the change. All authored FHIR Core
  prose (HTML pages and resource/conformance XML differentials) lives
  here.
- **Affected Files:** (categories rather than an enumerated list,
  because the sweep touches dozens of files)
  - `source/*.html` — ~38 occurrences across the top-level
    specification pages (26 `e.g.` + 12 `i.e.` without trailing comma
    in the current `db79dcd` clone). Each page edited in place.
  - `source/<resource>/*.xml` (StructureDefinition, ValueSet,
    CodeSystem, OperationDefinition, SearchParameter,
    CapabilityStatement) — ~2,658 raw `e.g.`/`i.e.` matches in
    `source/**/*.xml`, of which the in-scope subset is the prose
    fields enumerated under *Proposed Change* item 2, restricted to
    `<differential>` blocks. Realistic in-scope count is materially
    smaller than the raw grep number once `<snapshot>`, code values,
    and `source/datatypes/*.xml` spreadsheets are excluded.
  - `source/structuremap/*.xml`, `source/observation/*.xml`,
    `source/patient/*.xml`, `source/medication/*.xml`,
    `source/diagnosticreport/*.xml`, etc. — every per-resource folder
    that ships authored FHIR Core content is in scope; none is
    structurally privileged over another.
- **Change Scope:** Moderate — many small, mechanical edits across
  many files; no schema, build, or behavioural change.

### Breaking Changes

No breaking changes. The change is pure prose punctuation in narrative
fields. No element cardinality, type, binding, invariant, or search
parameter is altered. Downstream consumers that scrape `definition` or
`description` text will see textual diffs but no semantic change.

## Implementation Plan

### Prerequisites

- The R6 content-review pass must be the framing for this work
  (per resolution wording). Schedule the sweep against an R6 ballot
  preparation milestone rather than an out-of-cycle PR, so that
  per-resource owners review the diffs in the same pass as their
  R6 substantive edits.
- Confirm with FHIR Infrastructure (and the publisher maintainers)
  that no automated lint/CI rule will be added that *forces* the
  comma form — this sweep is a one-time normalisation, not a new
  publisher rule, unless the WG wants to also add a
  `qa/`/`suppressed-messages.txt`-style check.
- Verify briefing freshness immediately before starting: re-run
  `repo-analysis HL7/fhir if-stale` if the clone HEAD has advanced
  past `db79dcd` by then.

### Step-by-Step Tasks

#### HL7/fhir

1. **Sweep top-level specification HTML pages.**
   - File: `source/*.html` (each page in turn — there are ~26 pages
     containing comma-less `e.g.` and ~12 containing comma-less
     `i.e.` in the `db79dcd` snapshot).
   - Action: For each page, run a regex *find* (not blind replace)
     for `\be\.g\.[^,)]` and `\bi\.e\.[^,)]`. For each hit, manually
     decide between (a) inserting a comma to produce `e.g.,` /
     `i.e.,` followed by the same trailing whitespace/character, or
     (b) leaving the occurrence alone if it falls under one of the
     skip rules in *Proposed Change* item 4 (sentence-final, code
     fragment, intentionally non-parenthetical).
   - Details: Preserve all surrounding HTML markup, anchors
     (`<a name="...">`), and `<span>` wrappers. Diffs should be
     comma-only insertions.

2. **Sweep authored XML prose in StructureDefinition differentials.**
   - File: `source/<resource>/structuredefinition-*.xml` and
     `source/<resource>/profile-*.xml` (every per-resource folder
     under `source/`).
   - Action: For each file, locate `<differential>` and within it
     each `<element>`'s `<short>`, `<definition>`, `<comment>`,
     `<requirements>`, `<meaningWhenMissing>`. Apply the same
     find-and-judge pass as Task 1. Also check the resource-level
     `<purpose>` if present.
   - Details: **Do not edit anything inside `<snapshot>`** — those
     fields are regenerated by the publisher (briefing §Warnings).
     Skip `source/datatypes/<type>.xml` (Excel spreadsheets, per
     briefing) and any `<resource>-spreadsheet.xml` where a sibling
     StructureDefinition exists.

3. **Sweep authored XML prose in ValueSet, CodeSystem,
   OperationDefinition, SearchParameter, CapabilityStatement.**
   - File: `source/<resource>/valueset-*.xml`,
     `source/<resource>/codesystem-*.xml`,
     `source/<resource>/operation-*.xml`,
     `source/<resource>/searchparameter-*.xml`,
     `source/<resource>/capabilitystatement-*.xml`, plus the
     centralised `source/valuesets.xml`, `source/codesystems.xml`,
     `source/searchparameter-registry.xml` if those are
     hand-authored (verify in clone before editing each).
   - Action: For each file, normalise comma-less `e.g.`/`i.e.` in
     `<description>`, `<documentation>`, parameter
     `<documentation>` fields only. Same find-and-judge approach.
   - Details: Do not alter `<code value="..."/>` strings, URIs,
     identifiers, or `<expression>` / FHIRPath fragments — only
     human-language prose fields. Aggregate registry XML files
     (`source/valuesets.xml`, etc.) may be derived; check for a
     header comment indicating generation before editing and skip
     if generated.

4. **Cross-check anchor and link integrity.**
   - File: repo-wide.
   - Action: After Tasks 1–3, run a search for any links of the
     form `e.g.\` or `i.e.\` inside HTML `href`/`src` attributes
     (e.g., file names that happen to contain the literal `e.g.`)
     to confirm none was accidentally rewritten. Also re-verify
     that no `<a name="...">` anchor was disturbed by an HTML edit
     in Task 1.
   - Details: This is a guard against the briefing's general
     anchor-preservation gotcha; the punctuation sweep is unlikely
     to touch anchors but the check is cheap.

5. **Run the publisher locally on the changed files.**
   - File: n/a — uses `publish.sh` / `build.sh` from the repo root
     (publisher tooling is itself out of scope per briefing).
   - Action: Run a local publisher build limited to (or at least
     including) the changed resources to confirm no narrative-XML
     edit broke XML well-formedness. Inspect the produced
     `qa.html` / suppressed-messages output for new errors or
     warnings.
   - Details: This is the cheapest backstop against a stray
     character (e.g., a comma inserted *inside* a tag rather than
     after `e.g.`).

6. **Open the PR(s) against the R6-ballot working branch.**
   - File: n/a.
   - Action: Stage the changes as one PR per resource folder (or a
     small number of grouped PRs) so that per-resource owners can
     review diffs in their own area. Reference FHIR-44047 in each
     PR title/body.
   - Details: A single mega-PR is discouraged — too many touched
     files for resource owners to meaningfully review.

### Testing Considerations

- **Re-validate** every touched StructureDefinition / ValueSet /
  CodeSystem with the FHIR validator as part of the publisher run;
  the change should be a no-op semantically.
- **Re-publish** the spec locally and diff the rendered HTML for a
  sample of edited pages (`patient.html`, `observation.html`,
  `medication.html`) against the previous build to confirm the only
  visible change is the inserted commas inside narrative.
- **Edge case to verify:** narrative inside `<div xmlns="...">`
  blocks (the human-readable resource summary) — these are
  authored prose and are in scope; confirm the comma additions
  flow through the rendered narrative.
- **Interoperability:** none — pure narrative.
- **Downstream IGs:** no direct test is needed; the next time an IG
  pulls a new core spec snapshot it will see the corrected text.

### Open Questions

1. **Scope of the sweep across XML files.** The plan above limits
   editing to authored prose fields inside `<differential>` and to
   ValueSet/CodeSystem `<description>` etc. Is the FHIR
   Infrastructure WG comfortable with that scoping, or do they want
   a broader sweep that also touches narrative-`<div>` blocks
   (which are *technically* generated for some resources but
   hand-authored for others)? The briefing's authoring boundary
   helps but does not explicitly call out the narrative-`<div>`
   case.
2. **Sentence-final occurrences.** A small minority of hits are
   sentence-final ("...several factors, e.g."), where the comma
   form is awkward. The plan flags these for manual judgement. Does
   the WG want a default ruling — leave as-is, or rewrite the
   sentence to avoid trailing abbreviation?
3. **Should a permanent lint rule be added?** If yes, a publisher
   `qa/` rule (or `suppressed-messages.txt`-adjacent check) would
   prevent regression in R6 and beyond. The resolution does not
   ask for this, so the plan does not include it, but it is a
   natural follow-up.
4. **Comment-thread style disagreement.** Costa Teixeira pushed
   back on the *correctness* framing in the comments. The
   resolution sided with the reporter, but a reviewer noting this
   pushback may want a one-line note in the PR description
   acknowledging that consistency (not strict correctness) is the
   driver. Confirm the WG is comfortable with that framing.
5. **Briefing scope.** The persisted briefing's *Ticket-Relevant
   Paths* section is scoped to FHIR-42980 (a different ticket) —
   the path information used above is therefore drawn from the
   briefing's general *Facts* / *Warnings* / *Recommended Change
   Recipes* sections, which are repo-wide and apply to any
   FhirCore narrative-edit ticket. If a ticket-specific briefing
   is desired for FHIR-44047, re-run `repo-analysis HL7/fhir`
   scoped to this ticket before implementation.
