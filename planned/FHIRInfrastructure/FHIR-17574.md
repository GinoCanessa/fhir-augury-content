## Resolution Summary

**Resolution:** Persuasive with Modification.

> "We will define a standard extension on address called classification as 0..1
> with a preferred binding to a terminology based on the proposed set of codes.
> (We'll try to use existing codes rather than make up new ones.)"

The PA conference call on 2025-07-22 (cooper.thompson) refined the resolution:

- The extension's value set should be `v3-ServiceDeliveryLocationRoleType`
  (`https://terminology.hl7.org/5.1.0/ValueSet-v3-ServiceDeliveryLocationRoleType.html`),
  which is the same value set already used for `Location.type` and which
  contains the codes called out in the original ticket (assisted living
  facility, school, dormitory, long-term care / nursing home, correctional
  facility, day care, etc.).
- The extension is **not** to be permitted on `Location` directly — because
  `Location.type` already exists and `Location.address` is `0..1`, there is
  already a 1:1 mapping between the value set and an address inside `Location`.
  Context is therefore restricted to `Address`.

## Feature Proposal

### Problem Statement

`Address.use` (home / work / temp / old / billing) classifies the role of an
address relative to the contact, but does not capture the *kind of place* the
address represents (assisted living facility, dormitory, correctional
facility, shelter, etc.). For epidemiology and public-health workflows
(highlighted again during COVID by craig.newman, 2023-05-08), and for
consumers that need to know the "living arrangement" associated with an
address, `Address.use` is insufficient. The Location resource carries this
information via `Location.type`, but a free-standing `Address` (e.g., on
`Patient.address`, `Person.address`, `Practitioner.address`) is not bound to
a Location and so has no way to express it.

### Proposed Change

Add a new standard extension to the FHIR Extensions Pack:

- **Canonical URL:** `http://hl7.org/fhir/StructureDefinition/address-classification`
- **Id / file slug:** `address-classification`
- **Name:** `AddressClassification`
- **Title:** `Address Classification`
- **Status:** `active`, experimental `false`
- **Publisher / WG:** HL7 International / FHIR Infrastructure (FHIR-I owns
  the Address data type; the resolution explicitly returned the disposition
  to FHIR-I — comment by craig.newman, 2023-05-09).
- **fhirVersion:** `5.0.0` (consistent with the other address-* extensions
  shipped from this repo).
- **Context:** single `element` context with `expression` = `Address`. **Do
  not** add a `Location` context (per the 2025-07-22 PA decision).
- **Cardinality on the context:** `0..1` (a given Address can be classified
  at most once; the resolution explicitly states `0..1`).
- **Type / value:** `Extension.value[x]` constrained to a single
  `valueCodeableConcept` slice (`min=0`, `max=1`, `type=CodeableConcept`).
- **Binding:**
  - `strength` = `preferred` (per resolution wording: *"preferred binding to
    a terminology based on the proposed set of codes"*).
  - `valueSet` = `http://terminology.hl7.org/ValueSet/v3-ServiceDeliveryLocationRoleType`.
- **Short:** `Classification of the place this address identifies`.
- **Definition:** *"A classification of the kind of place this address
  identifies (e.g., assisted living facility, school, dormitory, long-term
  care / nursing home, correctional facility, shelter, day-care / group
  baby-sit, other congregate living facility). This is in addition to the
  role distinction already captured by `Address.use` (home / work / etc.) and
  is intended for use cases — including public health and epidemiology —
  that need to characterise the physical setting an address represents
  without modelling it as a Location resource."*

The extension follows the same shape and metadata block as the existing
`address-official` extension already shipped from
`HL7/fhir-extensions/input/definitions/datatypes/StructureDefinition-address-official.xml`,
so authoring should be a copy-and-modify of that template.

### Design Rationale

- **Extension, not core element.** Both Grahame Grieve (2018-08-07,
  2019-11-23) and the PHER / PA work groups landed on an extension. The
  original concept (a coded refinement of `Address.use`) is too narrow and
  too IG-specific for a normative core data-type element, but is well-suited
  to a standard extension that consumers can opt into.
- **Reuse `v3-ServiceDeliveryLocationRoleType` instead of inventing a new
  value set.** This is the value set already bound to `Location.type` and it
  covers every example the reporter listed. The resolution explicitly
  prefers existing codes; the PA call confirmed this set is the right one.
  This keeps the meaning of the extension aligned with `Location.type` so
  that an Address that *would* have been backed by a Location resource
  still carries the same code system.
- **Preferred (not required / extensible) binding.** The resolution
  specifies *preferred*. This lets implementers use codes outside the set
  when none of the existing codes fit the local use case, while still
  steering interoperability toward the v3 set.
- **No Location context.** `Location` already has `Location.type` and
  `Location.address` is `0..1`, so adding the extension to `Location` would
  create two ways to express the same fact. Context is restricted to
  `Address` (PA call, 2025-07-22).
- **Place in `HL7/fhir-extensions`, not core.** The standard-extensions IG
  is the supported home for new cross-resource extensions; this ticket is a
  pure extension addition with no change to the Address data type itself or
  to any resource StructureDefinition in `HL7/fhir`.

### Related Specifications

   - File: `input/definitions/datatypes/StructureDefinition-address-classification.xml`

     `input/definitions/datatypes/StructureDefinition-address-official.xml`.
   - Details:
     - `<id value="address-classification"/>`
     - `structuredefinition-wg` extension `valueCode = "fhir"`,
       `structuredefinition-fmm` extension `valueInteger = 1` (new — same

       `structuredefinition-standards-status` extension `valueCode =
       "trial-use"`.
     - `<url value="http://hl7.org/fhir/StructureDefinition/address-classification"/>`
     - `<identifier>` with `system = "urn:ietf:rfc:3986"` and a freshly
       minted OID `urn:oid:2.16.840.1.113883.4.642.5.<n>`.
     - `<version value="5.0.0"/>` (matches sibling extensions in this
       directory).

       `<title value="Address Classification"/>`,
       `<status value="active"/>`, `<experimental value="false"/>`,
       `<date value="<today>"/>`.

       (HL7 International / FHIR Infrastructure;
       `http://www.hl7.org/Special/committees/fiwg`).
     - `<description>` carrying the resolution prose (see Proposed Change).
     - `<fhirVersion value="5.0.0"/>`, `<kind value="complex-type"/>`,
       `<abstract value="false"/>`, `<type value="Extension"/>`,
       `<baseDefinition value="http://hl7.org/fhir/StructureDefinition/Extension"/>`,
       `<derivation value="constraint"/>`.
     - `<context><type value="element"/><expression value="Address"/></context>`
       — single context, **no Location context**.

### Testing Considerations

- Build the extensions IG end-to-end and inspect the rendered page for
  `address-classification` (title, definition, value-set link, contexts).
- Construct an example `Patient` with two `address` entries: one with the
  extension carrying a `v3-ServiceDeliveryLocationRoleType` code (e.g.,
  `NURS` for nursing home), one without — confirm both validate.
- Construct a negative example with a code outside the value set and
  confirm validators issue a *warning* (preferred binding) rather than an
  error.
- Confirm the extension can be sliced into `Patient.address.extension`,
  `Person.address.extension`, `Practitioner.address.extension`, and
  `Organization.address.extension` (i.e., the `Address` element context
  matches every resource that uses Address).
- Confirm validators do **not** allow the extension on `Location.address`
  (the spec restricts context to `Address` so it *will* be permitted there
  too, since `Location.address` is an `Address` — the PA decision is
  about not adding a `Location` resource-level context, not about
  forbidding the extension on `Location.address`. Document this
  distinction in the description if reviewers raise the question).
- Cross-version impact: the extension is `fhirVersion 5.0.0`; verify
  publication metadata and downstream IGs that pin to fhir-extensions
