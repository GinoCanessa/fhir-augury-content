## Resolution Summary

Resolution type: **Persuasive with Modification**.

Per the ticket resolution description, the change is to add a new paragraph to the
"Related Pattern: Asynchronous Interaction Request" section (balloted as
§3.2.6.1.1.1):

> There are other [architectural approaches](exchanging.html) (Task, messaging, etc.)
> that can also provide asynchronous responses.

No other text in the section is altered; the existing blockquote pointing readers at
the async-bundle pattern is retained and the new paragraph is appended beneath it
(linking to `exchanging.html`, which is the core spec's "Exchanging Data" page).

## Feature Proposal

### Problem Statement

The FHIR Asynchronous Request Patterns documentation currently describes the two
canonical async patterns (Asynchronous Bulk Data Request and Asynchronous Interaction
Request) and, in the "Related Pattern: Asynchronous Interaction Request" section,
cross-links the two patterns to one another. It does not mention the other
architectural approaches in FHIR that can also deliver asynchronous responses —
notably Task-based workflows (`exchanging-request.html#task`) and messaging
(`exchanging-messaging.html#async`). A reader arriving at the async pattern section
may reasonably conclude those are the only options, when in fact Task and messaging
are first-class alternatives described in the "Exchanging Data" section
(`exchanging.html`).

### Proposed Change

Add a single new paragraph to the "Related Pattern: Asynchronous Interaction
Request" section, immediately after the existing `<p>For use cases that return small
amounts of data but may take a lot of time to process, see <a
href="async-bundle.html">Asynchronous Interaction Request</a>.</p>` paragraph:

```html
<p>There are other <a href="exchanging.html">architectural approaches</a>
(Task, messaging, etc.) that can also provide asynchronous responses.</p>
```

Notes:

- The resolution description shows the link target rendered as `<span
  class="error">[architectural approaches]</span>(exchanging.html)`. That markup is
  the Jira "broken wiki-link" rendering of the intended hyperlink; the HTML edit
  should produce an ordinary `<a href="exchanging.html">architectural
  approaches</a>` anchor.
- No heading, id, or anchor is added — this is a single additional `<p>` inside the
  existing section body.
- The edit is narrative-only; no StructureDefinition, ValueSet, or data-model
  artifact changes.

### Design Rationale

The resolution was adjudicated by FHIR Infrastructure as "Persuasive with
Modification". The intent is to nudge readers of the async-patterns doc toward the
adjacent exchange guidance rather than to move or duplicate content. A single
pointer paragraph is the minimum viable change and is consistent with how the rest
of the async page cross-links: existing links go out to `async-bundle.html`,
`async-bulk.html`, `operations.html`, and `http.html` using plain anchor tags inside
`<p>` elements. Keeping the edit to one extra paragraph preserves the existing
information architecture and avoids expanding the "Asynchronous Request Patterns"
page into a full comparison of exchange styles (which is already covered in
`exchanging.html`).
