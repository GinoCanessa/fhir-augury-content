## Resolution Summary

The resolution (Persuasive) is to reword the **Restful interface** row in
the "Forward and backward compatibility rules" table on the FHIR
versioning page so that the rule clearly applies to the names/portions
the **specification mandates** (e.g., the `Patient` and `_history`
portions of `[base]/Patient/123/_history/4`), rather than reading as a
constraint on deployed servers' endpoints generally.

Verbatim, the cell text is to change from:

> **Restful interface** Existing endpoints will not be renamed or
> removed, nor have their expected behavior changed in a manner that
> would cause reasonable systems designed against prior versions to be
> non-interoperable. Additional endpoints and interactions may be
> introduced.

to:

> **Restful interface** Where the specification mandates specific names
> for endpoints or portions thereof (e.g. the 'Patient' or '\_history'
> portions of `[base]/Patient/123/_history/4`), future versions will
> not rename, remove, or change the behavior of invoking those
> endpoints in a manner that would cause reasonable systems designed
> against prior versions to be non-interoperable. Additional endpoints
> and interactions may be introduced.

No other changes to the table or surrounding sections are required by
the resolution.

## Feature Proposal

### Problem Statement

The "Forward and backward compatibility rules" section on
`versions.html` currently reads as if the FHIR specification places
forward/backward-compatibility obligations on the URLs of any FHIR
server endpoint. The reporter (Bas van den Heuvel) points out that:

1. The spec is normative for **FHIR content**, not for deployments —
   deployments are not normative.
2. The word "endpoint" in the existing wording is ambiguous; it reads
   like the operator's base URL is in scope, when in fact the
   compatibility commitment only covers the names/portions that the
   FHIR specification itself mandates (resource type segments,
   `_history`, etc.).

The result is a rule that, taken literally, over-reaches its own
scope and causes confusion about what the specification is actually
promising across versions.

### Proposed Change

Edit a single cell in the compatibility-rules table on
`source/versions.html` (the **Restful interface** row, currently
lines 493–499 of the clone at HEAD `db79dcd`):

- **File:** `source/versions.html`
- **Section:** `Forward and backward compatibility rules` table,
  `<tr>` whose first `<td>` is `Restful interface`.
- **Action:** Replace the contents of the second `<td>` (the rule
  text) with the resolution wording shown in the Resolution Summary
  above.
- **Notes on encoding:** The new text contains a literal `[base]`
  placeholder. The existing FHIR convention on this page is to render
  example URL templates with `[base]` shown literally (matching how
  `http.html` renders them); keep the same convention rather than
  introducing new markup. Render the underscore in `_history` as a
  plain underscore (no escaping needed inside `<td>` HTML).
- **Anchors / IDs:** The row has no `id`/`name` anchor, so no inbound
  links break.
- **Surrounding rows / table structure:** Leave untouched. Do not
  reorder the table or move the rule to a new section — the
  resolution explicitly rewords this single cell rather than
  relocating the content (the reporter's suggestion to move the
  remark elsewhere was not adopted; the WG kept it in place but
  rewrote it for clarity).

Illustrative diff (semantic, not literal whitespace):

```html
<tr>
  <td>Restful interface</td>
  <td>
-     Existing endpoints will not be renamed or removed, nor have their expected behavior changed in a manner that would cause reasonable systems designed
-     against prior versions to be non-interoperable.  Additional endpoints and interactions may be introduced.
+     Where the specification mandates specific names for endpoints or portions thereof
+     (e.g. the 'Patient' or '_history' portions of <code>[base]/Patient/123/_history/4</code>),
+     future versions will not rename, remove, or change the behavior of invoking those endpoints
+     in a manner that would cause reasonable systems designed against prior versions to be non-interoperable.
+     Additional endpoints and interactions may be introduced.
  </td>
</tr>
```

### Design Rationale

- **Scope alignment.** The rewording binds the compatibility promise
  to "names the specification mandates" (resource-type segments,
  `_history`, etc.), which is what the FHIR core actually controls.
  The original wording was easy to misread as constraining
  implementer-chosen endpoint URLs.
- **Minimal surface change.** The WG chose to rewrite the cell rather
  than restructure `versions.html` or push content to `http.html` (the
  reporter's alternative). That keeps the compatibility-rules table
  intact as the single canonical home for these promises and avoids
  splitting closely related rules across two pages.
- **Consistency with existing examples.** Reusing the
  `[base]/Patient/123/_history/4` URL template aligns with the URL
  shape used throughout `http.html` and the RESTful API page, so
  readers immediately recognize what "portions thereof" refers to.
