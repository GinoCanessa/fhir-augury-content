## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"Will change the anchor for 5.4.6.5 Logical Models to `logical-models` as the other anchor is generated by a code snippet."*

Concretely: on `structuredefinition.html` there are currently two anchors with the same fragment id `#logical`:

1. One inside the **Common Mapping Targets** table (section 5.4.6.4), generated by the `[%mappings-table%]` publisher macro from the `<id>logical</id>` row in `source/mappingSpaces.xml` (the "Logical Mapping" row). This anchor is generated by code and cannot be moved without changing the mappings-table generation.
2. One immediately preceding the `<h3>Logical Models</h3>` heading (section 5.4.6.5), authored by hand in `source/structuredefinition/structuredefinition-notes.xml`.

The fix is to rename the hand-authored anchor (#2) from `logical` to `logical-models`, leaving the generated `#logical` to point unambiguously at the Logical Mapping row in the mappings table. All inbound links inside the spec that today say `structuredefinition.html#logical` but mean *"Logical Models"* must be updated to `structuredefinition.html#logical-models` so they keep landing on the same content.

## Feature Proposal

### Problem Statement

The page `structuredefinition.html` (built primarily from `source/structuredefinition/structuredefinition-notes.xml`) contains two anchors named `logical` in close proximity:

- `5.4.6.4 Common Mapping Targets` — the "Logical mapping" row of the generated mappings table emits `<a name="logical">` (or `id="logical"`) from `mappingSpaces.xml`'s `<id>logical</id>`.
- `5.4.6.5 Logical Models` — a hand-authored `<a name="logical"></a>` immediately before `<h3>Logical Models</h3>`.

Because HTML fragment resolution picks the first matching id on the page, every link pointing at `structuredefinition.html#logical` lands on the Common Mapping Targets row rather than the Logical Models section, which is what the surrounding prose in pages such as `elementdefinition.html`, `profiling.html`, `references.html`, and `questionnaire-notes.xml` actually means. The reporter (Max Masnick, 2023-01-31) provided a screenshot showing the navigation arrow next to the section heading also pointing at the ambiguous fragment. This is a long-standing R4 *and* R5 documentation defect.

### Proposed Change

**Single-file authoring change**, plus inbound link updates so existing references keep working:

1. In `source/structuredefinition/structuredefinition-notes.xml`, change the hand-authored anchor at line 209 from:

   ```xml
   <a name="logical"></a>
   <h3>Logical Models</h3>
   ```

   to:

   ```xml
   <a name="logical-models"></a>
   <h3>Logical Models</h3>
   ```

   (Equivalent: add `id="logical-models"` to the `<h3>` if the surrounding sections use that pattern; the existing pattern in this file is `<a name="…"></a>` immediately before `<h3>`, so keep it.)

2. Update every inbound link in `source/` that currently uses `structuredefinition.html#logical` and is semantically about *Logical Models* (not the Logical Mapping row) to `structuredefinition.html#logical-models`. The five hits in this clone are:

   - `source/elementdefinition.html` line 540 — *"default values can be defined in [Logical Models]"* — clearly about Logical Models.
   - `source/profiling.html` line 299 — *"default values may be defined for [logical models]"* — clearly about Logical Models.
   - `source/references.html` line 121 — *"In order to support [Logical Models]"* — clearly about Logical Models.
   - `source/questionnaire/questionnaire-notes.xml` line 198 — *"reference to the [logical model] 'xxx'"* — clearly about Logical Models.
   - `source/fhir.ini` line 110 — `logical=structuredefinition.html#logical` in the `[redirects]`/link-shortcut block. This shortcut is named `logical`; whether to repoint it to `#logical-models` or leave it alone depends on what the shortcut is intended to mean. See **Open Questions**.

3. Do **not** modify `source/mappingSpaces.xml`. The `<id>logical</id>` row is the well-defined identifier for the "Logical Mapping" target across all of FHIR (e.g., `http://hl7.org/fhir/logical`) and is the anchor the resolution intentionally leaves in place to own `#logical`.

### Design Rationale

The resolution comment explicitly chose this direction because the other anchor is *generated* — moving it would mean either renaming the well-known `logical` mapping target id (used in `<url>http://hl7.org/fhir/logical</url>` and throughout `mappingSpaces.xml`) or special-casing the publisher's mappings-table emitter. Both are far more invasive than renaming a single hand-authored anchor. The `logical-models` name is descriptive, matches the section heading verbatim, and avoids any further collision with mapping-target ids.

Updating inbound `#logical` links to `#logical-models` preserves user-visible behavior: today those links *intend* to land on the Logical Models section but accidentally land on the mapping-table row; after the fix, leaving them as `#logical` would make that mis-landing official rather than accidental. Updating them is therefore part of the same defect fix, not a separate concern.
