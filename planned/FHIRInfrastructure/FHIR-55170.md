## Resolution Summary

Resolution: **Not Persuasive with Modification**.

The ticket asked for an `unknown`/`other` escape-valve concept to be added to the
`guide-page-generation` value set. The work group declined to extend the value
set. Instead, the resolution agreed two changes (verbatim):

1. We will remove the `ig-3` constraint.
2. We will add documentation to the element indicating that if there are new
   capabilities or capabilities adopted from future versions, they should use a
   code of `generated` and use an extension to specify the new mechanism to
   create the page. `source` may or may not be relevant.

In other words: keep the four-code value set (`html | markdown | xml |
generated`) as-is, drop the constraint that forbids `source` when
`generation = 'generated'`, and document `generated` (paired with an extension)
as the forward-compatible escape valve.

## Feature Proposal

### Problem Statement

`ImplementationGuide.definition.page.generation` is `1..1` with a required
binding to `http://hl7.org/fhir/ValueSet/guide-page-generation`, whose
underlying `CodeSystem` (`source/implementationguide/codesystem-guide-page-generation.xml`,
`http://hl7.org/fhir/guide-page-generation`) defines exactly four codes:
`html`, `markdown`, `xml`, `generated`. There is no `unknown` / `other` code.
When an IG author needs to express a page-generation mechanism that does not
match any of those four (e.g., a new templating system, or a generation
strategy adopted from a future spec version), they have no legal value to
populate the element with, which makes the whole `definition.page` block
impossible to author conformantly.

The situation is made worse by the `ig-3` invariant on
`ImplementationGuide.definition.page`:

```
generation='generated' implies source.empty()
```

That invariant prevents using `generated` as a fallback alongside an extension
that points at the actual source of the page, because as soon as `generation`
is `generated`, `source[x]` must be empty.

### Proposed Change

Two coordinated changes to the `ImplementationGuide` resource definition in
`HL7/fhir`. No CodeSystem or ValueSet change is in scope (the work group
explicitly declined adding `unknown`/`other`).

1. **Remove the `ig-3` constraint** from `ImplementationGuide.definition.page`.

   In `source/implementationguide/structuredefinition-ImplementationGuide.xml`,
   delete the `<constraint>` block at lines 775–780:

   ```xml
   <constraint>
     <key value="ig-3"/>
     <severity value="error"/>
     <human value="Source must be absent if 'generated' is generated"/>
     <expression value="generation='generated' implies source.empty()"/>
   </constraint>
   ```

   Also remove the two `<condition value="ig-3"/>` back-references that point
   at this key:
   - line 801, on element `ImplementationGuide.definition.page.source[x]`;
   - line 836, on element `ImplementationGuide.definition.page.generation`.

   With the constraint and conditions gone, an IG author may legitimately
   populate `source[x]` even when `generation = 'generated'`.

2. **Update the documentation on `ImplementationGuide.definition.page.generation`**
   to describe `generated` as the forward-compatible escape valve.

   In the same StructureDefinition, expand the element at lines 827–845. The
   existing `<definition>` ("A code that indicates how the page is generated.")
   stays. Add a `<comment>` immediately after it, on or just after line 830,
   with the spirit of the resolution. Suggested wording (work group may polish):

   > If a new page-generation capability is needed — either a brand-new
   > mechanism or one adopted from a later FHIR version — use the code
   > `generated` and convey the actual generation mechanism via an extension
   > on this element. `source[x]` may or may not be relevant in that case
   > (the `ig-3` constraint that previously forbade it has been removed in
   > this version of FHIR).

   No change to the binding, cardinality, type, value set, or short label
   (`html | markdown | xml | generated`).

The `guide-page-generation` `CodeSystem` and `ValueSet` are intentionally
**not** modified.

### Design Rationale

The work group chose this path over adding `unknown`/`other` because:

- The four existing codes describe how the publisher should *process* the
  page. An `unknown`/`other` code carries no processing semantics — a
  publisher seeing it would not know what to do, so it does not actually
  unlock new authoring.
- `generated` already means "the publisher (not the author) is responsible
  for the page content", which is the correct semantic for any future
  mechanism: the publisher is told to generate, and an accompanying
  extension tells it *how*. This keeps the four-code semantics tight while
  giving authors a forward-compatible escape valve.
- The `ig-3` constraint blocked the very pattern the resolution wants to
  endorse (`generation='generated'` plus an extension that may need
  `source[x]` to point at templates, partial source, or generator input).
  Removing the constraint is the minimum change needed to make the
  documented pattern legal.

The change is documentation + constraint removal only; no breaking impact on
existing instances that already validate against `ig-3`, since a removed
constraint can never start failing things that previously passed.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole owner of the change. Hosts the `ImplementationGuide`
  StructureDefinition and the `guide-page-generation` value set / code system.
- **Affected Files:**
  - `source/implementationguide/structuredefinition-ImplementationGuide.xml`
    — remove `ig-3` `<constraint>` (lines 775–780), remove the two
    `<condition value="ig-3"/>` entries (lines 801, 836), add a `<comment>`

  - `source/implementationguide/invariant-tests/ig-3.f1.fail.xml` — delete
    (the fixture exercises a constraint that no longer exists; leaving it
    in place would either silently start passing or become a misleading
    "fail" fixture for a removed key).
- **Change Scope:** Minor.

### Breaking Changes

No breaking changes for instance authors:

- Removing an invariant cannot turn a previously-valid instance invalid.
  Instances that satisfied `ig-3` (i.e., did not set `source[x]` when
  `generation = 'generated'`) remain valid. Instances that previously failed
  `ig-3` (set both at once) become valid.
- Cardinality, type, binding strength, and value set membership of
  `ImplementationGuide.definition.page.generation` are unchanged, so the
  required-binding behaviour is preserved.

Potentially observable impact for tooling:

- Validators and IG-publisher implementations that hard-code or test for
  `ig-3` will need to drop that key. This is a tooling-side adjustment, not
  a backwards-incompatible change to the spec contract.

### Related Specifications

ecosystem and by every published HL7 FHIR Implementation Guide, but no other

and no other StructureDefinition in `source/implementationguide/` reuses the
`ig-3` key. No conformance updates required outside this folder.

## Implementation Plan

### Prerequisites

None. All changes live in a single per-resource folder in `HL7/fhir`; no
upstream ticket, no terminology change, no extension definition is required
first. (If an IG author wants to *use* the new pattern, they will need an
appropriate `Extension` to attach to `definition.page.generation` describing
the alternate generation mechanism, but defining that extension is out of
scope for this ticket — the resolution only obliges the spec to *allow* the
pattern.)

### Step-by-Step Tasks

#### HL7/fhir

1. **Remove the `ig-3` constraint from `ImplementationGuide.definition.page`.**
   - File: `source/implementationguide/structuredefinition-ImplementationGuide.xml`
   - Action: Delete the `<constraint>` element at lines 775–780 (the
     `<key value="ig-3"/>` block ending in
     `<expression value="generation='generated' implies source.empty()"/>`).
   - Details: This is inside the `<differential>` of the
     `ImplementationGuide.definition.page` element. Do not touch the
     `<snapshot>` — the publisher regenerates it.

2. **Drop the `ig-3` `<condition>` back-reference on `definition.page.source[x]`.**
   - File: `source/implementationguide/structuredefinition-ImplementationGuide.xml`
   - Action: Delete the line `<condition value="ig-3"/>` at line 801 inside
     element `ImplementationGuide.definition.page.source[x]`.
   - Details: Leave the surrounding element (cardinality 0..1, types
     `url | string | markdown`, translatable extension) untouched.

3. **Drop the `ig-3` `<condition>` back-reference on `definition.page.generation`.**
   - File: `source/implementationguide/structuredefinition-ImplementationGuide.xml`
   - Action: Delete the line `<condition value="ig-3"/>` at line 836 inside
     element `ImplementationGuide.definition.page.generation`.
   - Details: Keep the `1..1` cardinality, the `code` type, and the required
     binding to `http://hl7.org/fhir/ValueSet/guide-page-generation` exactly
     as they are.

4. **Add the escape-valve documentation to `definition.page.generation`.**
   - File: `source/implementationguide/structuredefinition-ImplementationGuide.xml`
   - Action: Inside the `ImplementationGuide.definition.page.generation`
     element (the block at lines 827–845, after task 3), add a `<comment>`
     child immediately after `<definition value="A code that indicates how
     the page is generated."/>` (around line 830), reading approximately:

     > "If a new page-generation capability is needed — either a brand-new
     > mechanism or one adopted from a later FHIR version — use the code
     > `generated` and convey the actual generation mechanism via an
     > extension on this element. `source[x]` may or may not be relevant
     > in that case."

   - Details: This is the documentation prescribed by item 2 of the
     resolution. Final wording is the work group's call; the structural
     placement (a single `<comment>` on the `generation` element) is what
     matters. Do not change `<short>`, `<min>`, `<max>`, `<type>`, or
     `<binding>`.

5. **Delete the now-orphaned `ig-3` "must-fail" invariant fixture.**
   - File: `source/implementationguide/invariant-tests/ig-3.f1.fail.xml`
   - Action: Delete the file.
   - Details: The fixture intentionally violates `ig-3` (sets both
     `sourceString = "demosource"` and `generation = "generated"` on a
     `definition.page`). Once `ig-3` is gone, the fixture no longer
     exercises anything. Sibling fixtures (`cnl-0.f1.fail.xml`,
     `cnl-1.f1.fail.xml`, `ig-1.f1.fail.xml`, `ig-2.f1.fail.xml`,
     `ig-2.p1.pass.xml`) are unrelated and stay. No manifest or index file
     in this folder lists the fixtures, so no further bookkeeping is needed
     beyond the delete.

### Open Questions

1. **Exact wording of the new `<comment>`.** The resolution sketches the
   semantics ("use `generated` and an extension; `source` may or may not be
   relevant") but does not provide a final sentence. The work group should
   confirm the wording before publication.
2. **Companion extension.** The resolution endorses "use an extension to
   specify the new mechanism" but does not name or define that extension.
   No extension is added by this ticket. The work group may want a
   follow-up ticket against `HL7/fhir-extensions` to define a canonical
   "page-generation-mechanism" extension; that is explicitly out of scope
   here per the briefing's "do not silently roll in adjacent concerns"
   guidance.
3. **Test-harness manifest.** No manifest file in
   `source/implementationguide/invariant-tests/` enumerates the fixtures
   in this clone. If a higher-level harness (under `tools/` or in the
   publisher) maintains its own list, that list will need the `ig-3.f1.fail.xml`
   entry removed; flag for the maintainer at PR review time.
