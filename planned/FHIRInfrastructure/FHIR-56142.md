## Resolution Summary

Resolution = **Persuasive**. Per the applied resolution (mckenzie, 2026-04-13):

1. Add a new `Other` concept to the `request-intent` CodeSystem with the following definition:

   > "Indicates that the intent value being communicated does not fall within any of the allowed intent codes for the version of FHIR being communicated. The specific intent concept SHOULD be communicated in an extension. Systems will not be able to safely process (and might not accept) instances with this code unless they are familiar with the extension communicating the actual intent."

2. Add a new section titled **"Choosing an intent code"** to the Request pattern's "notes" page, containing the following four paragraphs verbatim:

   > "In most cases, a sending system will know the specific intent code to transmit. Systems SHOULD transmit the most specific code applicable. I.e. It is better to transmit "filler-order" or "instance-order" than just "order". However, if a system is not sure of which more specific code applies (because their system doesn't differentiate), it is acceptable to transmit a higher-level code. Consuming systems SHOULD be able to handle both generic and specific codes."
   >
   > "In rare circumstances, a system might have legacy or external data that does not differentiate whether the record represents a plan, proposal, or order. In this situation, the system SHOULD use an 'intent' of "proposal" as it is the safest and most encompassing concept. Systems that infer intent in this way should never be used to initiate workflow."
   >
   > "The 'Other' code is reserved for use where none of the existing codes are applicable. In most cases, this will be when expressing a concept introduced in a future version of FHIR that needs to be conveyed in an older version of FHIR."

The applied vote (Jose Costa Teixeira / Reinhard Egelkraut, 4-0-0) classifies this as **Compatible, substantive; Category: Enhancement**. Vote dispositions explicitly note that `data-absent-reason` does not have an `Other` concept, so the new `Other` code is added to `request-intent` itself rather than being borrowed from `http://terminology.hl7.org/CodeSystem/data-absent-reason`.

Because this same problem was logged against multiple resources whose `intent` element binds to `http://hl7.org/fhir/ValueSet/request-intent`, the persuasive resolution here is intended to also resolve:

- FHIR-55322 (CommunicationRequest)
- FHIR-55275 (ServiceRequest)
- FHIR-55262 (DeviceRequest)
- FHIR-55267 (NutritionOrder)

## Feature Proposal

### Problem Statement

`Request.intent` is mandatory (`1..1`) on every resource that follows the Request pattern, with a **required** binding to `http://hl7.org/fhir/ValueSet/request-intent`. The current value set (`proposal`, `plan`, `directive`, `order` + `original-order`/`reflex-order`/`filler-order`/`instance-order`, `option`) provides no escape valve such as `unknown` or `other`. As a result, when a system holds a request whose intent legitimately cannot be classified into any of the existing codes (e.g., legacy data, transcoded content from a future FHIR version, partner-specific intents), the resource cannot be populated at all — the required binding forces a code that may misrepresent the data.

The same gap was reported independently for CommunicationRequest (FHIR-55322), ServiceRequest (FHIR-55275), DeviceRequest (FHIR-55262) and NutritionOrder (FHIR-55267). All of those resources bind `intent` to the same `request-intent` ValueSet (confirmed in the clone — see Repo Context below), so the fix is single-sourced in `source/request/`.

### Proposed Change

Two coordinated edits, both confined to `HL7/fhir`:

1. **Add the `Other` concept to the `request-intent` CodeSystem.**
   - File: `source/request/codesystem-request-intent.xml`
   - Insert a new top-level (sibling of `proposal`/`plan`/`directive`/`order`/`option`) `<concept>` element after the existing `option` concept (before `</CodeSystem>`):
     ```xml
     <concept>
       <code value="other"/>
       <display value="Other"/>
       <definition value="Indicates that the intent value being communicated does not fall within any of the allowed intent codes for the version of FHIR being communicated.  The specific intent concept SHOULD be communicated in an extension.  Systems will not be able to safely process (and might not accept) instances with this code unless they are familiar with the extension communicating the actual intent."/>
     </concept>
     ```
   - The CodeSystem is `<status value="active"/>` and has `standards-status = normative`. Adding a new code is a substantive but compatible (additive) change to a normative artifact and must follow the FMG additive-content rules; no other concept is altered.
   - The `request-intent` ValueSet (`source/request/valueset-request-intent.xml`) composes `<include><system value="http://hl7.org/fhir/request-intent"/></include>` with no filter and no enumerated concepts, so the new code automatically flows into the bound ValueSet. **No edit is required to the ValueSet file.**

2. **Add a "Choosing an intent code" notes section to the Request pattern page.**
   - File: `source/request/request-spreadsheet-notes.xml`
   - Insert a new `<h2>` block (with `<a name="intent"></a>` anchor) using the verbatim wording from the resolution. Place it as a peer of the other top-level sections (e.g., immediately before `<a name="notes"> </a>` at line 174 or immediately after the existing `Updating requests` section at line 62, whichever the FHIR-I editor prefers — this plan recommends placing it directly **after** `Updating requests` so general "how to choose codes" guidance precedes the more specific `notes`/`supportingInfo` discussion). The skeleton:
     ```xml
     <a name="intent"></a>
     <h2>Choosing an intent code</h2>
     <p>In most cases, a sending system will know the specific intent code to transmit. ...</p>
     <p>In rare circumstances, a system might have legacy or external data ...</p>
     <p>The 'Other' code is reserved for use where none of the existing codes are applicable. ...</p>
     ```
   - All three paragraph bodies are taken verbatim from the resolution text (see Resolution Summary above).

### Design Rationale

- The committee considered re-using `Other` from `http://terminology.hl7.org/CodeSystem/data-absent-reason` (per the original 2026-04-13 vote text), but mckenzie's revert noted that `data-absent-reason` does not actually contain an `Other` concept. The final resolution therefore mints `other` directly inside `request-intent`, which is also more consistent with the pattern-owned semantics (this is an *intent* escape valve, not a data-absent reason).
- Adding `other` rather than `unknown` matches the agreed-upon definition: the *intent* exists and is known to the source, but cannot be classified — it is not "unknown". The notes guidance also explicitly directs implementers to fall back to `proposal` (the safest existing code) rather than `other` whenever the actual situation is "we don't differentiate plan/proposal/order".
- The accompanying notes section is required because, without it, implementers would have no guidance on when to prefer specific vs. generic codes, when to use `proposal` as a safe fallback, and when `other` is appropriate. This is a precondition for `other` not being abused as a generic dumping ground.
- Because `request-intent` is composed by reference (no concept enumeration in the ValueSet), no consuming resource StructureDefinition needs to change. This minimises ripple across CommunicationRequest, ServiceRequest, DeviceRequest, NutritionOrder, SupplyRequest, ActivityDefinition, RequestOrchestration.

### Breaking Changes

No breaking changes. The change is purely additive:

- Adding a new code to a required-bound ValueSet means producers may now emit `other`; pre-change consumers will reject `other` as an unknown code. Per the FMG's standard treatment of additive ValueSet changes against required bindings, this is classified by the committee vote as **Compatible, substantive (Enhancement)** rather than breaking — consumers are expected to handle additive updates within a major-version line.
- No element cardinality, type, or binding changes.
- No removal of existing codes.
- No search-parameter changes.
- No invariant changes.

         transmit <code>filler-order</code> or <code>instance-order</code> than just <code>order</code>.

         doesn't differentiate), it is acceptable to transmit a higher-level code. Consuming systems
         <strong>SHOULD</strong> be able to handle both generic and specific codes.
       </p>
       <p>
         In rare circumstances, a system might have legacy or external data that does not
         differentiate whether the record represents a plan, proposal, or order. In this situation,
         the system <strong>SHOULD</strong> use an <code>intent</code> of <code>proposal</code> as it
         is the safest and most encompassing concept. Systems that infer intent in this way should
         never be used to initiate workflow.
       </p>
       <p>
         The <code>other</code> code is reserved for use where none of the existing codes are

     - Wording is taken verbatim from the resolution; the only typographic adjustments are wrapping `SHOULD` in `<strong>`, the code names in `<code>`, and converting straight to typographic spacing — these match the conventions already used elsewhere in this file (see `Fulfillment/Execution information`, lines 30–61, which uses `<code>`, `<em>` and `<strong>` similarly). ### Testing Considerations

- Re-run the standard publish for `source/request/` and the consumer resources listed under "Related Specifications". The publisher's QA pass should flag any binding-validation regressions.
- Build a small example instance for at least one consumer resource (e.g., `ServiceRequest.intent = "other"`) plus an `extension` slot carrying a fictional intent concept, and confirm that:
  - It validates against the resource's StructureDefinition with a `required` binding.
  - The default expansion of `ValueSet/request-intent` includes `other`.
- Edge cases to verify:
  - Producer that emits `intent="other"` without an accompanying intent extension — instance is technically valid but the new notes guidance discourages it; this is an editorial check, not a validator check.
  - Producer of legacy data that previously tried to omit `intent` (was impossible due to required binding) can now choose `proposal` (per the new guidance) or `other` (when truly unclassifiable).
- Interoperability: confirm that downstream IGs that already constrain `intent` to a subset of `request-intent` codes (e.g., US Core profiles) are not silently broken by the additive change. Where a downstream profile uses `<binding strength="required">` to a *narrower* ValueSet, the new `other` code does not appear there automatically — that is the desired behaviour.
