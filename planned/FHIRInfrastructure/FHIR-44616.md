## Resolution Summary

The resolution is **Not Persuasive with Modification**. No new mechanism is being added to the FHIR core specification, *but* the underlying problem is acknowledged and the modification is to be addressed as **guidance in the cross-version extensions guide** rather than as a normative change to the core extension model.

The single resolution comment on the ticket (Gino Canessa, 2025-11-04) is explicit:

> "Will update this ticket once the relevant content is confirmed in the guides."

The `MovingToGuide` label confirms the disposition: the resolution moves the explanatory work into the cross-version extensions implementation guide(s) (the `hl7.fhir.uv.xver-*` packages, currently published from `HL7/cross-version`). The reporter's two-paragraph problem statement frames what that guidance must cover:

1. How to satisfy a `1..1` mandatory polymorphic element in version *N* when the chosen value is one of the additional types introduced only in version *N+1* (the worked example is `Task.input.value[x]`, which permits more types in R5 than in R4 — using an inter-version extension to carry the R5 value still leaves the R4 `value[x]` slot formally empty and therefore violates `1..1`).
2. Whether the same mechanism can be reused to satisfy mandatory elements that are **removed** or made **optional** in a later version.

The Zulip thread `[fhir/infrastructure-wg] Backporting CodeableReference` (msg `418829848`, Aug 2023) is the substantive design discussion. Two candidate encodings were debated and the WG did not reach consensus on a single shape — which is consistent with "not persuasive" for a normative core change but persuasive enough to warrant guidance.

## Feature Proposal

### Problem Statement

The FHIR core specification today (see `source/versions.html`, "Extensions for converting between versions", lines ~622–697) tells implementers how to use cross-version extensions for elements that "do not exist", "have types that do not exist", or have "changes that prevent direct representation (e.g. maximum cardinality changes from 1 to *)". It does **not** address the asymmetric case the ticket calls out:

- A mandatory polymorphic element (`value[x]` typed `1..1`) exists in both versions.
- Version *N+1* widens the allowed type list (R5 `Task.input.value[x]` permits more datatypes than R4).
- An R4 instance wants to carry an R5-only datatype via the cross-version extension `http://hl7.org/fhir/5.0/StructureDefinition/extension-Task.input.value`.
- The cross-version extension does not occupy the `value[x]` slot itself, so the R4 schema/cardinality `value[x] 1..1` is unsatisfied even though the data is logically present.

The same gap exists in the inverse direction: an element that is mandatory in version *N* and removed (or relaxed to `0..1`) in version *N+1*, where converted instances need to honour the older mandatory while signalling that the modern semantics no longer require it.

The problem is real (the Zulip thread and the cross-version extension authors hit it for `CodeableReference` backporting), but the WG concluded that a normative additive change to the core extension model is not warranted; clarifying the existing pattern in the guide is sufficient.

### Proposed Change

This is a **documentation / guidance** change, not a schema or extension-definition change. Concretely:

1. **In the cross-version extensions IG (`HL7/cross-version` → `hl7.fhir.uv.xver-*` packages):** add a dedicated narrative section — call it *"Satisfying mandatory elements when types differ across versions"* — that:
   - Describes the asymmetry (mandatory in version *N*; representable in *N+1* only).
   - Specifies how the existing **data-absent-reason** extension (`http://hl7.org/fhir/StructureDefinition/data-absent-reason`) is to be applied to the otherwise-empty mandatory `value[x]` (or other mandatory) slot, with code `unsupported` (per ValueSet `data-absent-reason` — code already exists), to signal "the value is carried by an inter-version extension on the parent, not by this slot."
   - Pairs that with the corresponding inter-version extension on the parent element (the existing `extension-[Path]` URL pattern documented in `source/versions.html`).
   - Provides at least one fully-worked example using `Task.input.value[x]` (R4 instance carrying an R5-only datatype) and at least one inverse example (R5 instance representing an element that was mandatory in R4 and removed/relaxed in R5).
   - States that no new extension URL is being minted by this guidance; it is a documented composition of two existing mechanisms (DAR + cross-version `extension-[Path]`).
2. **In `HL7/fhir` core, `source/versions.html`:** add a one-paragraph cross-reference at the end of the "Extensions for converting between versions" section pointing to the new guide section, so that core readers find the pattern. The empty anchor `<a name="extensions-examples-mandatory"></a>` already present at line 701 of `source/versions.html` is the natural insertion point — it suggests this scenario was scoped at one time but never written.
3. **Defer the second sub-question** (mandatory elements *removed* in a future version) into the same guide section by treating it as the symmetric case of the first.

The keyword analysis on the ticket reinforces the DAR-`unsupported` framing: `unsupported` (BM25 8.83, count 3) and `dar` (BM25 6.12, count 2) are both top-ranked terms, indicating that the WG conversation that produced "Not Persuasive with Modification" had already converged on data-absent-reason as the slot-filling mechanism rather than minting a new "satisfied-by-interversion-extension" extension.

### Design Rationale

- **Not Persuasive (no core change):** the FHIR core extension model already supplies all of the building blocks — cross-version extensions on the parent (defined in `source/versions.html`), and `data-absent-reason` for an otherwise-empty mandatory primitive/complex slot. Inventing a new core extension for "this slot is satisfied by an inter-version extension" would duplicate functionality and would only ever be meaningful inside the cross-version conversion process — exactly the scope of the cross-version IG.
- **With Modification (guide change):** the Zulip discussion shows competent implementers reaching different shapes (the `_datatype` flag, the per-type `extension-Extension.value*` flag, the URL-bearing `_datatypeUrl`, the per-type-anchor proposal). That divergence is itself the evidence that the guide is currently silent and needs to give one canonical answer.
- **DAR `unsupported` is the right slot-filler** because (a) it is normative and already understood by validators, (b) the code `unsupported` ("The source system wasn't capable of supporting this element") accurately describes the R4 schema's view of an R5-only datatype, and (c) it leaves the actual value carried unambiguously by the inter-version extension on the parent.
- **Defer the symmetric case** rather than splitting it out, because the same DAR + xver-extension composition handles both directions, and a single guide section with a forward and an inverse example is clearer than two parallel sections.
