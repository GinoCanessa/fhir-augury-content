## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"This should be either deprecated or removed, as appropriate."*

The FHIR-I work group agreed that the THO CodeSystem
`http://terminology.hl7.org/CodeSystem/basic-resource-type` (published at
`https://build.fhir.org/ig/HL7/UTG/CodeSystem-basic-resource-type.html`) should
not continue to exist as an active, example-binding terminology. The approved
direction is to **deprecate** (preferred where external implementers may still
be using the codes) or **remove** (if safe) the CodeSystem ŌĆö and the associated
ValueSet that wraps it ŌĆö from THO/UTG, and to ensure the FHIR Core `Basic`
resource no longer points users at it as a source of resource-type codes.

## Feature Proposal

### Problem Statement

The THO CodeSystem `basic-resource-type` enumerates a frozen list of
placeholder "resource types" (e.g., `consent`, `referral`, `advevent`,
`aptmtreq`, `transfer`, `diet`, `adminact`, `exposure`, ŌĆ”) that were intended
to be carried in `Basic.code` for concepts that did not yet have a dedicated
FHIR resource. Nearly every one of these placeholder codes has long since been
replaced by a real FHIR resource (`Consent`, `ServiceRequest`, `AdverseEvent`,
`Appointment`, `Task`/encounter movement, `NutritionOrder`, etc.), and `Basic`
itself is a last-resort resource. The CodeSystem therefore:

- Advertises a set of codes that should effectively never be used in new data,
  yet is published with `status=active` and `experimental=false` in THO
  (`input/sourceOfTruth/fhir/codeSystems/CodeSystem-basic-resource-type.xml`,
  version 1.0.1), which implies endorsement.
- Is wrapped by two ValueSets that still bind into FHIR Core:
  - `http://terminology.hl7.org/ValueSet/basic-resource-type` (THO/UTG).
  - `http://hl7.org/fhir/ValueSet/basic-resource-type` (FHIR Core, in
    `source/basic/valueset-basic-resource-type.xml`).
- Is referenced as the example binding for `Basic.code` in the core
  specification and from `source/basic/basic-example.xml`, which perpetuates
  the expectation that implementers reach for these codes.

The reporter (Lloyd McKenzie) flagged that this CodeSystem "shouldn't exist"
and requested a THO proposal to remove it. The resolution accepts the
request as Persuasive and leaves the final choice between *deprecate* and
*remove* to the maintainer, based on whether any known implementers still rely
on it.

### Proposed Change

**Goal:** stop publishing `basic-resource-type` as an active terminology and
stop steering FHIR `Basic` users toward it.

Concretely:

1. **THO/UTG ŌĆö CodeSystem.**
   In `input/sourceOfTruth/fhir/codeSystems/CodeSystem-basic-resource-type.xml`
   (version `1.0.1`), apply the deprecation pattern that THO already uses for
   retiring content. At minimum:
   - Flip `<status value="active"/>` ŌåÆ `<status value="retired"/>`.
   - Flip `<experimental value="false"/>` ŌåÆ `<experimental value="true"/>`
     (optional, per THO editor preference) or leave `false` but mark the
     CodeSystem retired.
   - Add a `<extension url="http://hl7.org/fhir/StructureDefinition/codesystem-deprecated"><valueCode value="..."/></extension>` at the CodeSystem level (or a retirement date via the standard status-date
     pattern used elsewhere in THO) so the generated IG page renders the
     deprecation prominently.
   - Prepend a deprecation sentence to `<description>` explaining that the
     CodeSystem is retired because the placeholder codes have been superseded
     by first-class FHIR resources, and that implementers should use the
     appropriate dedicated resource (e.g., `Consent`, `AdverseEvent`,
     `Appointment`, `NutritionOrder`, etc.) instead of `Basic.code`.
   - Bump `<version>` (e.g., `1.0.1` ŌåÆ `1.1.0`) and update `<meta><lastUpdated>`.
   - **If the WG instead chooses full removal** (only if a review confirms no
     known external dependencies): delete the file and the sibling ValueSet,
     and record the retirement in the THO publication-request / release notes
     instead of editing in place.

2. **THO/UTG ŌĆö wrapping ValueSet.**
   `input/sourceOfTruth/fhir/valueSets/ValueSet-basic-resource-type.xml` must
   be kept consistent with the CodeSystem: same `status` (`retired`),
   matching `<description>` deprecation note, version bump. If the CodeSystem
   is fully removed, this ValueSet must also be removed.

3. **FHIR Core ŌĆö ValueSet copy.**
   `source/basic/valueset-basic-resource-type.xml` (already `status=draft`,
   `experimental=true`, version `6.0.0`) must be updated to match the THO
   outcome ŌĆö move to `status=retired`, update its `<description>` to point
   readers to the real resources, and (if THO removes the CodeSystem) remove
   the file. Any FHIR-Core page or resource that binds `Basic.code` to this
   ValueSet should be reviewed so the binding is either removed or its
   description clearly labels the ValueSet as retired guidance.

4. **FHIR Core ŌĆö `Basic` resource & example.**
   - Re-examine the example binding on `Basic.code` in
     `source/basic/` (StructureDefinition / profile page). It should continue
     to be an example binding at most; update the binding description so it
     stops recommending `basic-resource-type` and instead steers implementers
     to the appropriate dedicated FHIR resource.
   - Update `source/basic/basic-example.xml` (line 59 references this
     CodeSystem) so the example does not reinforce the retired codes ŌĆö either
     substitute a more plausible example code or add a comment noting the
     CodeSystem is retired.
   - Update `source/oids.ini` (line 157) only if the OID entry is removed
     alongside full deletion; for a deprecation, leave the OID mapping in
     place.

5. **Release-note / history entries.**
   Record the retirement in THO's publication request / history page and
   (where applicable) in FHIR Core's resource-level change log for `Basic`.

### Design Rationale

- **Deprecate-over-delete by default.** THO's convention for phasing out
  published terminology is to set `status=retired`, keep the artifact
  addressable, and document the deprecation in the description. This
  preserves URL/OID stability for existing instances in the wild ŌĆö important
  for a CodeSystem that has been published and referenced for years. Full
  removal is only appropriate when the maintainer has confidence no consumer
  depends on the artifact, which matches the resolution wording ("either
  deprecated or removed, **as appropriate**").
- **Scope is intentionally narrow.** The resolution does not call for
  changing the cardinality or type of `Basic.code`, nor for retiring the
  `Basic` resource itself. The proposal limits changes to terminology status
  and descriptive prose that points readers at the dedicated resources.
- **Matched pair (CodeSystem + ValueSet).** The two artifacts are published
  as a pair in both THO/UTG and FHIR Core; updating only one would leave a
  ValueSet referencing a retired CodeSystem (or vice-versa), which is a
  conformance smell. They must move together.
- **Labels context.** The `defer` label on the Jira ticket indicates the
  change was accepted but scheduled for a future THO release rather than an
  urgent R5 patch; the plan is therefore sized for a normal THO publication
  cycle rather than a hotfix.

### Breaking Changes

- **Retiring a published CodeSystem is a soft breaking change for
  consumers.** Instances in the wild that validate against the retired
  ValueSet will still resolve (canonical URL/OID unchanged) ŌĆö but any
  downstream binding that promotes the codes to `required` strength would
  start emitting warnings. In FHIR Core the binding on `Basic.code` is
  example-strength, so no conformance break is expected.
- **Full removal would be a hard breaking change** ŌĆö canonical URL stops
  resolving and any instance carrying a code from the system becomes
  unresolvable. This is why the proposal prefers deprecation unless the
  WG confirms zero known consumers.
- **Version bumps** on the CodeSystem and ValueSet mean any profile
  pinning to `version=1.0.1` will need to re-pin (normal THO release
  behaviour).
