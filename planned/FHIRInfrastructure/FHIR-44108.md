## Problem Statement

The [notifications page](https://build.fhir.org/ig/HL7/fhir-subscription-backport-ig/notifications.html#fhir-r4b-and-later) references three examples for the `backport-related-query` extension:

1. [Backported SubscriptionTopic: R4B Encounter Complete](https://build.fhir.org/ig/HL7/fhir-subscription-backport-ig/SubscriptionTopic-r4b-encounter-complete.html)
2. [R4B Notification: Id Only with Query](https://build.fhir.org/ig/HL7/fhir-subscription-backport-ig/Bundle-r4b-notification-id-only-with-query.html)
3. [R4B Notification: Full Resource with Query](https://build.fhir.org/ig/HL7/fhir-subscription-backport-ig/Bundle-r4b-notification-full-resource-with-query.html)

The reporter noted that the first link (the R4B SubscriptionTopic example) did **not** actually include a `backport-related-query` extension, making the reference misleading. The proposed fix was to update the example to add one.

## Resolution Description

> Propose adding a related query to the R4B topic definition.

### Current State Analysis

**Likely already implemented.** Inspection of the current `BackportTopics.fsh` on the default branch shows that the `BackportSubscriptionTopicExampleEncounterCompleteR4B` instance **already contains** the `backport-related-query` extension on its `notificationShape`:

```fsh
* notificationShape[=].extension[+].url = $relatedQueryExt
* notificationShape[=].extension[=].extension[+].url = "queryType"
* notificationShape[=].extension[=].extension[=].valueCoding = http://example.org/query-types#prescribed "Prescribed medications"
* notificationShape[=].extension[=].extension[+].url = "query"
* notificationShape[=].extension[=].extension[=].valueString = "http://example.org/fhir/Encounter/[id]/$prescribed-medications"
```

This extension was likely added as part of **[FHIR-43275](https://jira.hl7.org/browse/FHIR-43275)** ("Add 'notified pull' option to notifications"), which is documented in the STU 1.2.0 changelog as "Updated example [Backported SubscriptionTopic: R4B Encounter Complete]".

The `$relatedQueryExt` alias resolves to `http://hl7.org/fhir/uv/subscriptions-backport/StructureDefinition/backport-related-query` (defined in `input/fsh/Common.fsh`).

**However**, FHIR-44108 is **not** tracked in `input/pagecontent/ig_changelog.md`. This suggests:
- The substantive code change was already applied via FHIR-43275
- FHIR-44108 may have been filed against a published version that hadn't incorporated the fix yet
- At minimum, the changelog needs a FHIR-44108 entry

### Step 2: Apply Code Change (if not already present)

**If the extension is NOT present** in the current R4B SubscriptionTopic example, add it to `input/fsh/BackportTopics.fsh` in the `BackportSubscriptionTopicExampleEncounterCompleteR4B` instance.

Add the following after the `notificationShape[=].include` entries (around the end of the `notificationShape` block):

```fsh
* notificationShape[=].extension[+].url = $relatedQueryExt
* notificationShape[=].extension[=].extension[+].url = "queryType"
* notificationShape[=].extension[=].extension[=].valueCoding = http://example.org/query-types#prescribed "Prescribed medications"
* notificationShape[=].extension[=].extension[+].url = "query"
* notificationShape[=].extension[=].extension[=].valueString = "http://example.org/fhir/Encounter/[id]/$prescribed-medications"
```

This mirrors the pattern already used in:
- The R4 version (`BackportSubscriptionTopicExampleEncounterCompleteR4`) — same extension in the `notificationShape`
- The R4B notification examples (`BackportNotificationExampleIdOnlyWithQuery`, `BackportNotificationExampleFullResourceWithQuery`) — same extension on `SubscriptionStatus.notificationEvent`

**If the extension IS already present** (as current code inspection suggests), no FSH changes are needed.
