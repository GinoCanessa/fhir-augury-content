## Resolution Summary

**Resolution:** Persuasive with Modification (vote 11-0-0; Impact: Non-substantive; Category: Clarification).

Add a usage-notes section called **"Use of 'identifier' and similar elements"** to **both the Request and the Event workflow patterns**, and update the **Identifier data type** to point at those new sections. The section must cover:

1. The `identifier` element is intended to uniquely point to a single instance, with the following exceptions:
   - Both `Identifier.system` and `Identifier.value` must be present for an Identifier to be unique.
   - When `Identifier.period` is populated, identity only applies within that period.
   - Identifiers may not always actually be unique (e.g. SSNs across patients due to fraud, entry error, or duplicate records).
2. Identifiers are **not** intended to convey identifiers for *other* business objects. Example: a "request" identifier on a result must not appear in `[result].identifier`; it belongs on `[result].basedOn.identifier`. *(Per the 2025-02-10 disposition note from McKenzie, bullet 2 was retained as originally written rather than the shorter PA-Q4-WGM 2025 wording proposed by egelkraut, because bullet 2 sets the rule and bullet 5 covers the workaround.)*
3. Some legacy systems put non-identifier content into identifier elements (e.g. an insurance subscriber id in `Patient.identifier` instead of `Coverage.subscriber`); systems should be tolerant of identifiers that prove non-unique even though that practice is incorrect.
4. Systems may enforce uniqueness for specific identifier systems within their own scope; only under that enforcement may an identifier be safely assumed to resolve to a single record.
5. To convey *categories* for resources, use elements other than `Resource.identifier` ŌĆö e.g. `.code`, `.category`, dedicated elements such as `.groupIdentifier`, extensions, or `meta.tag`.

The guidance must also cover other elements of type **Reference** or **Identifier** that appear on the respective patterns (so the same wording lives on both Request and Event because both patterns expose `identifier`, `basedOn`, `groupIdentifier`, etc.).

## Feature Proposal

### Problem Statement

FHIR's definition of `Identifier` is generic and historically silent on three questions that have caused real implementer confusion (per the linked zulip thread `fhir/infrastructure-wg > Identifier definition` and the PA Confluence page "Using FHIR Identifiers"):

- **Scope** ŌĆö should an `Identifier` element only ever point at the *underlying instance*, or can it also identify a *broader concept* (e.g. an accession number, a requisition, a group)?
- **Categorization vs. identification** ŌĆö implementers conflate the two and stuff coded category values into `.identifier` (e.g. an order/request id on a result, a subscriber id on a patient).
- **Uniqueness assumptions** ŌĆö implementers assume identifiers are always unique and fail in edge cases (period scoping, system+value pairing, legacy/duplicate data, intentional non-uniqueness).

The Request and Event workflow patterns are the natural place to put this guidance because every resource that follows these patterns inherits `identifier`, `basedOn`, `groupIdentifier`, `requester`/`performer`, etc. and re-asks the same questions. The Identifier data type page is the secondary surface that should *link* to the pattern guidance rather than duplicate it.

### Proposed Change

This is a **prose-only / non-substantive (Category: Clarification)** documentation change confined to `HL7/fhir`. Work breaks into two parts:

**Part A ŌĆö already landed in `b6dac519ee` (verified against `cache/github/repos/HL7_fhir/clone`):**

- `source/request/request-spreadsheet-notes.xml` ŌĆö add a new top-level `<a name="identifier"> </a>` anchor and `<h2>Use of 'identifier' and similar elements</h2>` section containing the five-bullet block above (with bullet 1 nested as a sub-list of three exceptions). This sits among the other named sub-sections of the Request pattern's notes (e.g. `extensions`, `statemachine`, `fulfillment`, `updating`, `compound`, ŌĆ”). Confirmed at line 152 of the current file.
- `source/event/event-spreadsheet-notes.xml` ŌĆö add the same anchor and `<h2>Use of 'identifier' and similar elements</h2>` section (the Event variant covers `event.identifier`, `event.basedOn`, etc. and shares the wording). Confirmed at line 54 of the current file.

Together these add **22 lines to request notes and 21 lines to event notes** (commit stats: `+43/-0`, two files modified).

**Part B ŌĆö still TODO (the focus of this implementation plan):**

- `source/datatypes/identifier.xml` ŌĆö add a usage-note row pointing to the two new pattern sections. This is the Excel-spreadsheet authoring file (per the briefing's Gotchas: `source/datatypes/<type>.xml` files are *Excel*, not FHIR XML); the change must be made by opening the spreadsheet, editing the relevant `Definition`/`Comments`/`Notes` cell on the root `Identifier` row, and re-saving so the publisher will pick it up. Suggested wording for the cross-link cell:

  > For guidance on appropriate use of `identifier` and related elements (e.g. `basedOn`, `groupIdentifier`, `partOf`) see the corresponding sections of the [Request pattern](request.html#identifier) and the [Event pattern](event.html#identifier).

  The publisher renders that into the Identifier datatype detail page (`datatypes.html#Identifier` and the standalone `identifier.html` rendering) ŌĆö no separate HTML edit is required.

- (Optional, recommended) Mirror the same cross-link on the **Reference** data type's notes if any of the five bullets are read as applying to `Reference` (the resolution explicitly says "The guidance should also cover other elements of type Reference or Identifier that appear on the respective patterns" ŌĆö that is satisfied by including `basedOn`/`partOf` discussion in the pattern sections themselves, but a one-line cross-link from the Reference datatype to the same anchors keeps both sides of the resolution observable from the data-type pages too).

### Design Rationale

- **Pattern-page placement** keeps the rule next to the elements it governs (`basedOn`, `groupIdentifier`, `partOf`, `requester` etc. all live on Request/Event), and avoids duplicating the same five bullets across every Request/Event resource. This matches existing precedent in `request-spreadsheet-notes.xml` for cross-cutting topics like Fulfillment/Execution and Compound Requests.
- **Bullet 2 retained verbatim, not the WGM-2025 PA-Q4 shortening**, because ŌĆö per McKenzie's 2025-02-10 comment ŌĆö bullet 2 sets the prohibition while bullet 5 (which the WGM-2025 proposal would have made redundant) describes the *workaround* (use `.basedOn.identifier`, `.groupIdentifier`, `.code`, `.category`, etc.). Removing the "classifiers / categories" clause from bullet 2 would have weakened the rule.
- **Identifier-datatype links instead of inlining** keeps the canonical text in one place (the patterns) and lets the data-type page be a thin pointer ŌĆö consistent with how the datatype pages already link out to workflow guidance.
- **Non-substantive / Clarification impact** is appropriate because no element definitions, cardinalities, types, bindings, search parameters or invariants are changed; only narrative is added.
