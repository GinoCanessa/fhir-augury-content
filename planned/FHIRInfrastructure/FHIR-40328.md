## Resolution Summary

Resolution type is **Persuasive with Modification**. From the ticket's Resolution Description:

> Will provide a `meaningIfMissing` for each that says "If not declared, then it is up to server discretion whether multiple repetitions are allowed." Post-R5, we will also add a warning that complains if these elements are not declared.

The resolution has **two parts**:

1. **Already applied (R5, commit `3625f3cd19`, 2023-03-20)** — `meaningWhenMissing` text was added to `SearchParameter.multipleOr` (line 630 of `structuredefinition-SearchParameter.xml`) and `SearchParameter.multipleAnd` (line 641). Maintainer mckenzie's 2023 comments note "Only applied part" and "Still need to add the warning post-R5".
2. **Outstanding (post-R5 — i.e., this work)** — add warning-severity invariants on the `SearchParameter` root that fire when `multipleOr` or `multipleAnd` are not declared. Gino Canessa's 2025-11-04 comment explicitly reassigned the ticket because "Adding the invariants now will cause issues with build — assigning so that it can be done in lockstep with the related changes"; i.e., the invariants must be introduced together with populating `multipleOr`/`multipleAnd` on the many existing SearchParameter instances currently missing them.

This plan therefore covers the **outstanding post-R5 work** only.

## Feature Proposal

### Problem Statement

`SearchParameter.multipleOr` (0..1 boolean) and `SearchParameter.multipleAnd` (0..1 boolean) declare whether a server supports OR-/AND-combinations of a given parameter. Originally, the base spec did not state how a missing element should be interpreted. Elliot Silver's ticket asked whether a SearchParameter with no `multipleAnd` value is permitted to repeat or not, and asked the WG to either (a) raise the cardinality to 1..1 or (b) declare a `meaningIfMissing`.

The WG chose option (b) for R5, giving the prose "If not declared, then it is up to server discretion whether multiple repetitions are allowed." The WG also committed to strengthening the guidance post-R5 with a warning-level invariant that flags SearchParameters which omit either element, nudging authors to declare intent explicitly without turning existing content into a hard validation failure.

### Proposed Change

Edit the base `SearchParameter` StructureDefinition to add two warning-level invariants on the root element, then ensure existing SearchParameter instances in the core spec declare `multipleOr`/`multipleAnd` so the new warnings don't regress the build.

1. **Add invariant `spd-6`** to the root `SearchParameter` element in `source/searchparameter/structuredefinition-SearchParameter.xml`:
   - `key`: `spd-6`
   - `severity`: `warning`
   - `human`: `SearchParameter SHOULD declare multipleOr so that servers know whether multiple 'or'-combined values are supported for this parameter`
   - `expression`: `multipleOr.exists()`
   - `source`: `http://hl7.org/fhir/StructureDefinition/SearchParameter`
2. **Add invariant `spd-7`** to the same element:
   - `key`: `spd-7`
   - `severity`: `warning`
   - `human`: `SearchParameter SHOULD declare multipleAnd so that servers know whether the parameter may be repeated with AND semantics`
   - `expression`: `multipleAnd.exists()`
   - `source`: `http://hl7.org/fhir/StructureDefinition/SearchParameter`
3. **Populate `multipleOr` and `multipleAnd`** on the 2,026 SearchParameter instances across `source/**/*search-params*.xml` that currently omit them, so the new warnings do not flood the publisher output. (A survey of the clone found 147 files containing 2,052 `SearchParameter` instances; 2,026 are missing one or both elements — see Impact Analysis.) For each instance, add explicit `true`/`false` values that reflect the parameter's intent as already described in its prose.
4. **Add invariant-test fixtures** `source/searchparameter/invariant-tests/spd-6.f1.fail.xml` and `spd-7.f1.fail.xml`, each a minimal SearchParameter that deliberately omits the respective element and is expected to trigger only the corresponding warning.
5. Leave the existing `meaningWhenMissing` text on both elements unchanged — it continues to describe behaviour for any instance that (despite the warning) still omits the element.

### Design Rationale

- The WG explicitly chose `meaningIfMissing` + a warning rather than raising cardinality to 1..1. This preserves backwards compatibility with every SearchParameter already published against R4/R5 (cardinality stays 0..1) while giving authors a visible nudge. That trade-off is the core of "Persuasive with Modification".
- Severity `warning` matches the style of existing advisory invariants on `SearchParameter` (e.g. `cnl-0`) and of advisory invariants elsewhere in the spec. An `error` severity would turn every legacy SearchParameter into a validation failure — the WG did not agree to that.
- Keys `spd-6` and `spd-7` continue the existing `spd-N` sequence used on the root `SearchParameter` element (currently `spd-1` through `spd-5`). The `cnl-*` keys are reserved for canonical-resource-wide invariants inherited into this resource and are deliberately not reused.
- Expressions are trivial `exists()` checks; no FHIRPath subtleties apply because both elements are 0..1 booleans on the root.
- Populating `multipleOr`/`multipleAnd` on existing instances in lockstep is the explicit gating concern raised by Gino Canessa (2025-11-04); without it, the publisher build surfaces ~2k new warnings and the invariants cannot be safely landed.

### Breaking Changes

- **Validator warnings on every legacy SearchParameter** that is re-validated against the R6 profile without declaring `multipleOr` / `multipleAnd`. This is intentional (the point of the invariants) and was explicitly chosen over an `error`-severity constraint or a cardinality increase. Not a structural breaking change, but external SearchParameter authors will see new advisory output.
- No cardinality change. No type change. No terminology binding change. Element definitions and `meaningWhenMissing` text are unchanged.
- No search parameter schema change (the resource's own SearchParameters are not the subject — the resource definition is).

### Related Specifications

- Any IG that defines SearchParameters against FHIR R6 (`HL7/fhir-extensions`, downstream `IgRepositories`, HL7 accelerators' IGs) will surface the new warnings when re-validated. None require structural changes; authors choose to populate or suppress.
- `CapabilityStatement.rest.resource.searchParam` references SearchParameters by `definition` but does not repeat `multipleOr`/`multipleAnd`; no change needed.
- `OperationDefinition.parameter.searchType` is unaffected.

## Implementation Plan

### Prerequisites

- Confirm with the WG / Grahame that the warning text and expressions proposed for `spd-6` / `spd-7` are acceptable before the bulk content sweep begins. Re-authoring 2,026 SearchParameter instances and then changing the invariant is expensive.
- Confirm the target branch/release. Because Gino's 2025-11-04 comment defers the change "in lockstep with the related changes", this work should land on the active R6 build branch (not a patch branch of R5).
- Ensure the local publisher build is green on master before starting, so new warnings are clearly attributable to this change.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add invariant `spd-6` (multipleOr SHOULD be declared).**
   - File: `source/searchparameter/structuredefinition-SearchParameter.xml`
   - Action: In the root `SearchParameter` element's `<differential>` constraint list, insert a new `<constraint>` block immediately after the existing `spd-5` block (which ends at line 135, before the first `<mapping>` at line 136).
   - Details:
     ```xml
     <constraint>
       <key value="spd-6"/>
       <severity value="warning"/>
       <human value="SearchParameter SHOULD declare multipleOr so that servers know whether multiple 'or'-combined values are supported for this parameter"/>
       <expression value="multipleOr.exists()"/>
       <source value="http://hl7.org/fhir/StructureDefinition/SearchParameter"/>
     </constraint>
     ```
     Do not touch the `<snapshot>` block — the publisher regenerates it.

2. **Add invariant `spd-7` (multipleAnd SHOULD be declared).**
   - File: `source/searchparameter/structuredefinition-SearchParameter.xml`
   - Action: Insert immediately after the new `spd-6` block.
   - Details:
     ```xml
     <constraint>
       <key value="spd-7"/>
       <severity value="warning"/>
       <human value="SearchParameter SHOULD declare multipleAnd so that servers know whether the parameter may be repeated with AND semantics"/>
       <expression value="multipleAnd.exists()"/>
       <source value="http://hl7.org/fhir/StructureDefinition/SearchParameter"/>
     </constraint>
     ```

3. **Create invariant-test fixture for `spd-6`.**
   - File: `source/searchparameter/invariant-tests/spd-6.f1.fail.xml`
   - Action: New file. Clone the structure of `spd-1.f1.fail.xml` and craft a minimal SearchParameter that omits `<multipleOr>` (optionally including `<multipleAnd>` so the fixture isolates the `spd-6` failure).
   - Details: Must be schema-valid apart from missing `multipleOr`; the publisher's invariant-test harness expects a single expected failure per fixture file (naming convention `<key>.f1.fail.xml`).

4. **Create invariant-test fixture for `spd-7`.**
   - File: `source/searchparameter/invariant-tests/spd-7.f1.fail.xml`
   - Action: New file; same pattern as task 3 but omitting `<multipleAnd>`.
   - Details: Include `<multipleOr value="true"/>` (or `false`) so the fixture isolates the `spd-7` failure and does not also trip `spd-6`.

5. **Populate `<multipleOr>` and `<multipleAnd>` in the core SearchParameter bundle.**
   - File: `source/searchparameter/bundle-SearchParameter-search-params.xml`
   - Action: For each of the 22 `<SearchParameter>` entries, add `<multipleOr value="…"/>` and `<multipleAnd value="…"/>` just after `<expression>` / `<xpath>` / `<processingMode>` (matching the order in the StructureDefinition).
   - Details: Values must reflect the parameter's intent (e.g. identifier token parameters typically allow both; unique-key parameters may set one or both to `false`). Do **not** default everything to `true` without inspection — that is a spec statement, not a cosmetic fill.

6. **Populate `<multipleOr>` and `<multipleAnd>` on all remaining SearchParameter instances.**
   - File(s): `source/**/*search-params*.xml` (146 files outside the `searchparameter/` folder; ~2,004 `<SearchParameter>` entries).
   - Action: For each SearchParameter entry lacking either element, add explicit boolean values reflecting the parameter's semantics.
   - Details: This is the "lockstep" work called out by Gino Canessa's 2025-11-04 comment. Suggested workflow:
     1. Script a first pass that lists (file, SearchParameter code, existing prose) for every missing element to triage which values are obvious.
     2. For the obvious cases (e.g. most token/string/date parameters with straightforward semantics) set `multipleOr=true`, `multipleAnd=true` where the existing spec prose already implies it is combinable.
     3. Flag the non-obvious cases for WG review before committing values.
     Skip examples that are intentionally minimal test fixtures in `source/searchparameter/invariant-tests/` beyond the two added by this ticket; they are SHOULD-only targets, and those fixtures may deliberately need to remain missing.

7. **Run the publisher locally and verify invariant counts.**
   - Action: Run `publish.sh` / `build.sh` (or the gradle wrapper entry used by CI) on the changed branch.
   - Details: Expect zero new `spd-6` / `spd-7` warnings after task 6 completes. If warnings remain, they identify SearchParameter instances still needing explicit values — treat each as a follow-up edit, not a suppression.

- **Scope of the content sweep.** Task 6 updates ~2,000 instances in the core spec. The WG should confirm whether (a) the content sweep is in scope for this ticket or (b) it should be split into a follow-up ticket (one per work group or per resource folder). (This addresses the Gotchas surfaced in Repo Context.)
