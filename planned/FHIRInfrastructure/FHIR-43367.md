## Resolution Summary

**Resolution:** Persuasive — *"Fixed as part of Grahame's overall update"* (resolution description from the Jira snapshot).

The reporter showed that the existing inter-version-extensions example in the FHIR Core specification used an **R4 `ValueSet`** instance to demonstrate "this is how the new R5 element looks in R4 via a cross-version extension". The instance referenced `ValueSet.expansion.contains.property` and a `subProperty` cross-version extension. The problem: `ValueSet.expansion.contains.property` did not exist in R4 — that path was added in R5. So the supposedly-R4 example was not a valid R4 ValueSet. The reporter requested that the example either (a) be reworked so the `property` element really is treated as an inter-version extension, or (b) be replaced with a different example that *is* a valid demonstration of cross-version extensions.

The Persuasive resolution accepts the report. The agreed implementation (reflected in the in-repo fix) is option **(b)**: drop the broken `ValueSet.expansion.contains.property.subProperty` JSON pair entirely from `source/versions.html` and replace it with a new, broader **"Examples"** section that demonstrates cross-version extensions using R5/R6 → R4 backports of `Immunization.administeredProduct` (`Reference`), `Parameters.parameter.value[x]` for a `CodeableReference` choice, the `immunization-procedure` extension (`CodeableReference` value), and `individual-genderIdentity.supportingInfo` (`CodeableReference` in a sub-extension). All replacement examples use elements/types that genuinely exist in the source version being illustrated, so the JSON is valid in both directions.

## Feature Proposal

### Problem Statement

The FHIR Core specification's "Inter-Version Extensions" guidance on `source/versions.html` previously contained a worked example that purported to show "this is the R5 form, and this is what it would look like in R4 using a cross-version extension". The two JSON instances both used the path `ValueSet.expansion.contains.property` (and its `subProperty` child). That path was introduced in R5; it does not exist in R4. As a result:

1. The "R4 form" instance was not a valid R4 `ValueSet` — it claimed to demonstrate how to backport R5 content into a valid R4 instance, but the surrounding `property` element it relied on is not part of R4.
2. Implementers reading the page were given a misleading template for cross-version extensions, undermining the very pattern the section is teaching.
3. The sample only covered one shape (a primitive nested inside a complex extension at a property/sub-property location). Cross-version extensions actually have several distinct shapes — a complex datatype on a resource, a complex datatype in a `value[x]` choice, a complex datatype already inside an extension, and a complex datatype inside a sub-extension — none of which were illustrated.

The reporter (Corey Spears) flagged this on 2023-12-11. FMG/FHIR-I accepted the report as Persuasive and the example was reworked.

### Proposed Change

Edit **`source/versions.html`** (sole authored file affected) inside the cross-version-extensions section (around the `extensions-examples-*` anchors, just below the `Mappings between primitive types across versions` table at ~line 802 of the pre-fix file):

1. **Delete** the existing `<p><b>Example of a complex extension</b></p>` block plus the two `<pre class="json" fragment="ValueSet">…</pre>` instances that show:
   - the R5 `ValueSet` with `expansion.contains.property.subProperty`, and
   - the R4 `ValueSet` that re-implements `subProperty` via a nested cross-version extension at `http://hl7.org/fhir/5.0/StructureDefinition/extension-ValueSet.expansion.contains.property.subProperty`.
2. **Insert a new `<a name="extensions-examples"></a><h4>Examples</h4>` subsection** (anchor reused so any inbound link to `versions.html#extensions-examples` continues to resolve) that contains four labelled pairs of JSON instances. Each pair shows the natural form in the newer version followed by the cross-version-extension form in R4:

   | Sub-heading | Source-version form | R4 cross-version-extension form |
   |---|---|---|
   | "A complex datatype in a resource" | `Immunization.administeredProduct` (a `Reference`/`CodeableReference`) in R6 | R4 `Immunization` carrying it via `extension[url=http://hl7.org/fhir/4.0/StructureDefinition/extension-Immunization.administeredProduct]` with a nested `reference` sub-extension |
   | "A complex datatype in a resource for a choice element" | R6 `Parameters.parameter.valueCodeableReference` | R4 `Parameters.parameter` carrying a `value[x]` cross-version extension at `…extension-Parameters.parameter.value%5Bx%5D` whose first sub-extension is `_datatype` = `"CodeableReference"` and whose remaining sub-extensions carry the value's parts |
   | "A complex data type in an extension" | R5 `Immunization` with `immunization-procedure` extension (`valueCodeableReference`) | R4 `Immunization` with the same `immunization-procedure` extension URL but child sub-extensions `_datatype` = `"CodeableReference"`, `concept`, `reference` |
   | "A complex data type in an sub-extension" | R5 `Patient` with `individual-genderIdentity.supportingInfo` carrying a `valueCodeableReference` | R4 `Patient` with the same extension but `supportingInfo` re-expressed as a sub-extension whose first child is `_datatype` = `"valueCodeableReference"`, followed by `concept` and `reference` sub-extensions |

3. Each example is wrapped in `<pre class="json">…</pre>` (the `fragment="ValueSet"` attribute that the old `pre` blocks used is removed, since the new examples are illustrative-only and not intended to be parsed as fragments of a real test resource).
4. Reuse the existing anchor pattern: keep the pre-existing `<a name="extensions-examples-primitive"></a>`, `<a name="extensions-examples-complex"></a>`, `<a name="extensions-examples-mandatory"></a>` stack a few lines below (they already exist in `versions.html`) so other pages linking into the section keep resolving. Add the new umbrella `<a name="extensions-examples"></a>` anchor at the head of the new "Examples" `<h4>` so it can be linked as a unit.

No other authored file is modified. There is no terminology change, no StructureDefinition change, no schema change. The cross-version extension URLs cited in the new examples (`http://hl7.org/fhir/4.0/StructureDefinition/extension-Immunization.administeredProduct`, `…/extension-Parameters.parameter.value%5Bx%5D`, the `immunization-procedure` and `individual-genderIdentity` core extensions) already exist; the page just references them.

### Design Rationale

- **Picking option (b) over option (a).** Option (a) — "make the `property` element itself an inter-version extension" — would have meant inventing an `…/extension-ValueSet.expansion.contains.property` cross-version extension on top of, and inside of, the existing `subProperty` cross-version extension. That nests one cross-version extension inside another, which (i) is not how the cross-version packages are actually structured (each path defined in an earlier version gets one extension, not a parent/child stack the reader has to assemble) and (ii) compounds the very confusion the reporter flagged. Option (b) — replace with a new, valid example — keeps the teaching function of the section without inventing a fictional extension stack.
- **Why several examples instead of one.** The Zulip thread referenced from the implementing commit (`#fhir/infrastructure-wg > Backporting CodeableReference`) settled on illustrating each of the distinct shapes a complex datatype can take. A single example necessarily hides the differences between "complex datatype as a resource element", "complex datatype as a `value[x]` choice", "complex datatype as an extension's value", and "complex datatype inside a sub-extension"; each requires a different cross-version-extension layout (notably, the `_datatype` discriminator is needed for the `value[x]`/extension cases but not for the resource-element case).
- **Why `Immunization.administeredProduct`, `Parameters.parameter.value[x]`, `immunization-procedure`, `individual-genderIdentity.supportingInfo`.** These are real elements/extensions that exist in R5/R6 and not in R4 (or whose datatype changed between versions), so they are genuine cross-version-extension candidates. The original `ValueSet.expansion.contains.property` example was rejected because the *parent path* didn't exist in R4 — choosing replacement examples whose parent path **does** exist in R4 is essential to fixing the underlying defect.
- **Anchor stability.** Existing `<a name="extensions-examples-primitive">` / `…-complex` / `…-mandatory` anchors are preserved (the rewrite kept them in place a few dozen lines further down). The new umbrella `<a name="extensions-examples"></a>` is additive.
