## Resolution Summary

**Resolution:** Persuasive.

Quoting the resolution description verbatim:

> We will add an example showing the case of an async batch-request.
>
> In this scenario, the response is a `batch-response` bundle for the async request, where the first (only) entry is another `batch-response` bundle with the results of the batch request.

In concrete terms, the Asynchronous Interaction Request Pattern page must be updated to show what the completed status response looks like when the *kicked-off* interaction is itself a `batch` request. The async wrapper is a `batch-response` Bundle whose first (and only) entry contains another `batch-response` Bundle (the result of the nested batch), not a flat bundle mixing the async wrapper entry with the inner batch entries.

## Feature Proposal

### Problem Statement

The Asynchronous Interaction Request Pattern page (`async-bundle.html`, currently authored in `HL7/api-incubator-ig`) specifies that when the status request returns `200 OK`, the body is a `batch-response` Bundle whose first entry is the outcome of the kicked-off interaction. Today the page illustrates this for:

- a single-resource operation (first entry's `response` + optional `resource`),
- an error case (first entry's `response.status` = 4xx with an OperationOutcome in `response.outcome`), and
- a search operation that returns multiple resources (first entry's `resource` is a nested `searchset` Bundle).

The specification never explicitly covers the case where the kicked-off interaction is a FHIR `batch` request. As the reporter (Josh Mandel) and the linked Zulip thread (`#implementers > ✔ Async Batch Responses`) pointed out, without an explicit example, implementers may collapse the two layers into a single flat `batch-response` Bundle — losing the contract that the first entry of the outer Bundle always carries the outcome of the original kick-off, independent of how many individual responses the nested batch contained.

### Proposed Change

Add a new worked example to `async-bundle.html` under the **Status Request → Response - Complete** section (immediately after the existing "multiple resources / search" example), illustrating the `Bundle` of `Bundle` shape for an async batch request:

- Outer Bundle: `resourceType: "Bundle"`, `type: "batch-response"`.
- Outer Bundle has exactly one `entry`.
- That single entry's `response.status` is `200 OK` (the outcome of the kick-off request — i.e., the batch was accepted and processed).
- That single entry's `resource` is itself a `Bundle` with `type: "batch-response"`, whose `entry[]` array holds one `response` (and optional `resource`) per operation in the originally submitted batch.

Concrete JSON skeleton to author (finalize element values during editing):

```json
{
  "resourceType": "Bundle",
  "type": "batch-response",
  "entry": [{
    "response": { "status": "200 OK" },
    "resource": {
      "resourceType": "Bundle",
      "type": "batch-response",
      "entry": [
        {
          "response": { "status": "201 Created", "location": "Observation/123" },
          "resource": { "resourceType": "Observation", "id": "123" }
        },
        {
          "response": { "status": "200 OK", "location": "Patient/456" },
          "resource": { "resourceType": "Patient", "id": "456" }
        }
      ]
    }
  }]
}
```

Accompany the snippet with one or two sentences of narrative that:

1. State the shape explicitly: "When the kicked-off interaction is a FHIR `batch`, the status response is a `batch-response` Bundle whose single entry contains another `batch-response` Bundle holding the per-operation results."
2. Emphasize (consistent with the existing paragraph beginning "The outcome of the kick-off request SHALL be present as the first entry in the bundle") that the nested structure is required — a flat Bundle that merges both layers is non-conformant.

No normative wording on `Bundle.type`, cardinality, or status codes needs to change; only an additional illustrative example plus a small amount of clarifying prose. The parallel structure with the existing `searchset` example (where the inner resource is a different `Bundle.type`) should be preserved.

### Design Rationale

- **Consistency with existing examples.** The page already uses the pattern "outer `batch-response` Bundle wraps the kick-off outcome; the inner `resource` carries the real payload, even if that payload is itself a Bundle" for the multi-resource/search case. Extending that to `batch` simply instantiates the same rule on a new inner `Bundle.type`, rather than inventing a new shape.
- **Preserves the kick-off-outcome invariant.** The spec's SHALL — "The outcome of the kick-off request SHALL be present as the first entry in the bundle" — only makes sense if the outer Bundle remains the async wrapper. A flat Bundle would conflate kick-off outcome with individual batch-operation outcomes and break that invariant; the explicit example closes the ambiguity that prompted the Zulip discussion.
- **Minimal, non-breaking.** Adding an example (plus one short paragraph) is a documentation-only change. It matches the "Update narrative / prose" recipe rather than any StructureDefinition or conformance change, and does not require re-balloting normative text.
