## Feature Proposal

### Problem Statement

When discussing derivative profiles built on top of a profile of the
`Parameters` resource, reviewers were unsure which kinds of changes are
permitted. The original ticket enumerates the ambiguous cases:

- Restricting `parameter:myParam.value` to `string` only (where the
  parent profile allowed `string | uri | url`).
- Adding a new part `c` to a complex-parameter slice whose parent
  defined only parts `a` and `b`.
- Adding a brand-new named-parameter slice (e.g. `extended`) to
  `Parameters.parameter`.
- Restricting `Parameters.parameter:content.resource` to a specific
  profile.

The specification already has general rules for these cases (in
`profiling.html`'s Re-profiling and Re-slicing section), but there is
no worked example that ties them together on the `Parameters`
resource, which is the place where this pattern surfaces most
frequently (operation inputs/outputs, messaging, etc.). The resolution
comment confirms that "Parameters is no different than anything else" —
so the fix is to make that clear in the prose rather than to introduce
new profiling rules.

A secondary concern is discoverability of the companion FHIR-52921
language: `OperationDefinition`-derivation rules directly govern what a
derived `inputProfile` / `outputProfile` (which are profiles of
`Parameters`) may look like. Without a cross-link, authors of
`Parameters` profiles can miss that constraint.

### Proposed Change

**File touched:** `source/profiling.html`
(authoring root = `source/`, per `HL7/fhir` briefing).

Target location: the existing section

```
<a name="reslicing"></a>
<h3>Re-profiling and Re-slicing</h3>
```

at line 918 of `source/profiling.html`, immediately before the
`<a name="extensions"></a>` block at line 977. Insert a new subsection
(suggested `<h4>` and anchor `<a name="reslicing-parameters-example"></a>`)
at the end of the Re-profiling and Re-slicing content (after the
re-slicing rules at line 974) containing:

1. A short framing sentence: profiling `Parameters` follows the general
   re-profiling / re-slicing rules; the example below walks through the
   cases that most often cause confusion.
2. A worked example mirroring the ticket's four bullets, using a
   fictitious base profile `MyParametersBase` with exactly the slices
   from the ticket description:
   - `parameter:myParam` — value types `string | uri | url`
   - `parameter:complex` — parts `a`, `b`
   - `parameter:content` — expects a `Resource`
3. For each of two variants of the base profile (slicing `rule = open`
   vs. `rule = closed`), show a derivative profile `MyParametersChild`
   that applies the four changes from the ticket and state explicitly
   which are valid, which are not, and why. Concretely:

   | Change in derivative | `rule = open` on `parameter` | `rule = closed` on `parameter` |
   |---|---|---|
   | Tighten `parameter:myParam.value[x]` to `string` only | Allowed (tightening allowed types on an existing slice — re-profiling rule) | Allowed (same — closed slicing does not prevent tightening existing slices) |
   | Add part `c` to `parameter:complex` | Depends on whether the parent's `parameter:complex.part` is itself sliced open or closed; add commentary showing both cases | Same commentary — the outer `parameter` slicing rule does not govern the inner `part` slicing |
   | Add a brand-new slice `parameter:extended` | Allowed (new slice permitted when outer slicing is open) | **Not allowed** (closed slicing forbids new slices) |
   | Constrain `parameter:content.resource` to a specific profile | Allowed (tightening the allowed profile set is a normal constraint; if profiles were already listed, only tightening — not loosening — is permitted) | Allowed (same rationale) |

4. A closing paragraph linking to the driving rule in
   `operationdefinition.html#base` (the FHIR-52921 language) with
   phrasing such as: "When a profile of `Parameters` is referenced from
   an `OperationDefinition.inputProfile` or `outputProfile`, derivative
   profiles are additionally constrained by
   [Derived OperationDefinitions](operationdefinition.html#base) — in
   particular a derived operation's profile must remain compatible with
   the original (all profiled parameters with `min > 0` are still
   profiled; all profiled parameters are profiled with equivalent or
   tighter constraints)."

XML-ish sketch of the inserted StructureDefinition fragments (HTML
pre-formatted blocks, matching the existing style used around
line 900–916 of the same file):

```xml
<!-- base profile: MyParametersBase, outer slicing rule = open -->
<element id="Parameters.parameter">
  <path value="Parameters.parameter"/>
  <slicing>
    <discriminator><type value="value"/><path value="name"/></discriminator>
    <rules value="open"/>
  </slicing>
</element>
<element id="Parameters.parameter:myParam">
  <path value="Parameters.parameter"/>
  <sliceName value="myParam"/>
</element>
<element id="Parameters.parameter:myParam.value[x]">
  <path value="Parameters.parameter.value[x]"/>
  <type><code value="string"/></type>
  <type><code value="uri"/></type>
  <type><code value="url"/></type>
</element>
<!-- parameter:complex with parts a, b; parameter:content with resource -->
```

No example files need to be added under `source/examples/` — the
example is authored inline in `profiling.html` using `<pre>` blocks,
consistent with how the surrounding `reslicing` and earlier
`<pre class="xml">` blocks (lines ~893–916) are done.

### Design Rationale

- **Minimal-impact fix.** The reviewer comment on the ticket (Lloyd
  McKenzie, 2025-09-08) confirms there is **no special rule** for
  profiling `Parameters`; therefore the fix must live in documentation
  only. Adding an example in the existing Re-profiling and Re-slicing
  section reuses the page readers already land on when they ask the
  question.
- **Placement choice.** The Re-profiling and Re-slicing section
  (`profiling.html#reslicing`, line 918) already contains the applicable
  rules ("ElementDefinition.slicing.rule can be constrained from open to
  closed", the re-slicing rules, and the open/closed semantics). Adding
  the example here keeps rule + illustration together rather than
  splitting them across pages.
- **Open vs. closed framing.** The resolution explicitly calls out
  "open vs. closed slicing"; using that as the table axis keeps the
  example aligned with the resolution wording and makes the "add new
  slice" vs. "tighten existing slice" distinction visually obvious.
- **Cross-link to FHIR-52921.** The ticket is the direct follow-up of
  FHIR-52921 (referenced in the resolution and in FHIR-52921's
  description / its applying commit `f415ed7` on 2025-12-14, which
  edited `source/operationdefinition/operationdefinition-notes.xml`).
  Linking to `operationdefinition.html#base` (the anchor that renders
  `<h3>Derived OperationDefinitions</h3>` at line 4 of that XML file)
  ensures a reader of the `Parameters` example can find the companion
  `OperationDefinition` constraints without knowing to look for them.
- **Alternatives considered.**
  - *New standalone page.* Rejected — the content is a short example,
    not a new concept; dropping it into the existing section costs one
    paragraph and a table.
  - *Example in `parameters.html` resource page.* Rejected — the rule
    being illustrated is a general profiling rule, not a
    `Parameters`-specific one; authors searching for profiling guidance
    land on `profiling.html` first.
