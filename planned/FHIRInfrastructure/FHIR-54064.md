## Resolution Summary

The resolution is **Persuasive**. Per the resolution description:

> Will append to the sentence "The Signature.data is base64 encoded JWS-Signature as specified in RFC 7515: JSON Web Signature (JWS)" the following: **"NOTE: Because JWS is already base-64 encoded, this means that the signature will be double-encoded."**

This is a documentation-only clarification to the JWS rules in the `Signature` data type page. No StructureDefinition, ValueSet, CodeSystem, schema, or behavior is changed; the only edit is appending a `NOTE:` clause to the existing bullet that points readers at RFC 7515.

## Feature Proposal

### Problem Statement

The `Signature` data type page currently asserts that `Signature.data` "is base64 encoded JWS-Signature as specified in RFC 7515: JSON Web Signature (JWS)". The reporter points out that this is internally ambiguous:

- `Signature.data` is typed as `base64Binary` in FHIR — i.e., its wire form is always base64.
- RFC 7515 itself specifies that a JWS is **base64url** encoded.

The combination implies that to populate `Signature.data` from a compact-serialization JWS, the (already base64url-encoded) JWS must be base64-encoded a second time. The current text never says that out loud, leaving implementers to either guess or to interpret "base64 encoded JWS-Signature" inconsistently. The ticket asks the spec to be explicit about the double-encoding so that producers and consumers of `Signature.data` agree on the wire bytes.

### Proposed Change

Edit `source/signatures.html` to append a clarifying NOTE to the first bullet of the **JWS/JSON Signatures** section (the `<a name="JWS">` block), preserving the existing link to RFC 7515.

- **File:** `source/signatures.html`
- **Section:** `<h3>JWS/JSON Signatures</h3>` (anchor `#JWS`), opening `<ul>` at line 367.
- **Bullet to modify (line 368):**

  Current:

  ```html
   <li>The Signature.data is base64 encoded JWS-Signature as specified in <a href="https://tools.ietf.org/html/rfc7515">RFC 7515: JSON Web Signature (JWS)</a></li>
  ```

  After:

  ```html
   <li>The Signature.data is base64 encoded JWS-Signature as specified in <a href="https://tools.ietf.org/html/rfc7515">RFC 7515: JSON Web Signature (JWS)</a>. NOTE: Because JWS is already base-64 encoded, this means that the signature will be double-encoded.</li>
  ```

The XML Digital Signatures bullet at line 395 is intentionally **not** modified — XMLDSig is not pre-base64-encoded, so the double-encoding remark does not apply there.

No StructureDefinition, ValueSet, CodeSystem, search parameter, or example needs to change; the wire format of `Signature.data` is already `base64Binary` and the resolution explicitly endorses (rather than alters) the existing double-encoded representation.

### Design Rationale

- The resolution is "Persuasive" with a verbatim wording committed in `metadata.resolution_description`. The plan adopts that wording exactly so the published spec matches what the work group voted on.
- Editing prose only, in the same bullet that already cites RFC 7515, keeps the clarification adjacent to the rule it qualifies and avoids reordering the JWS rule list.
- An alternative — switching `Signature.data` to a pre-decoded representation (e.g., the raw JWS string in `string`) — would be a breaking type change and was not what the work group resolved. The Persuasive resolution explicitly chooses to *document* the double-encoding rather than eliminate it.
- The XMLDSig sibling bullet is left alone because its source bytes are XML, not a base64url payload, so the "already base-64 encoded" caveat would be incorrect there.
