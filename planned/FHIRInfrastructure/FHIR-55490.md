## Resolution Summary

Resolution: **Persuasive**. The committee agreed to add `orderMeaning` to three
Backbone elements where the order of repetitions is semantically significant.
The agreed prose (verbatim from the resolution description) is:

- **Questionnaire.item** â€” "The order items are listed indicates the order in
  which they should be presented to the user (if they are enabled)"
- **Questionnaire.item.answerOption** â€” "The order of the options indicates the
  order in which they should be presented to the user by default. Extensions
  may drive changes to the presentation order."
- **QuestionnaireResponse.item** â€” "The order of the items is the order in
  which the questions were presented to the user when completing the
  Questionnaire."

No code changes, no cardinality changes, no terminology changes â€” purely a
metadata addition to the StructureDefinitions of `Questionnaire` and
`QuestionnaireResponse`.

## Feature Proposal

### Problem Statement

`ElementDefinition.orderMeaning` is the FHIR-defined slot for telling
implementers whether the order of repeating sibling elements carries meaning.
For repeating elements where order does not matter, `orderMeaning` is left
absent (the default assumption per the spec is that order is not significant).
Today, neither `Questionnaire.item`, `Questionnaire.item.answerOption`, nor
`QuestionnaireResponse.item` set `orderMeaning`, even though their order is
semantically significant:

- `Questionnaire.item` â€” items are presented to the responder in document
  order (the parent `Questionnaire` description already states "The content of
  the questionnaire is constructed from an ordered, hierarchical collection of
  items", confirmed at line 731 of
  `source/questionnaire/structuredefinition-Questionnaire.xml`).
- `Questionnaire.item.answerOption` â€” options are presented in document order
  by default (extensions may override).
- `QuestionnaireResponse.item` â€” preserves the presentation order used at
  capture time, which is needed to reconstruct the responder's experience.

The omission was raised on Zulip
(`#questionnaire > orderMeaning`, 2026-01-28). Lloyd McKenzie confirmed it
was an oversight, Brian Postlethwaite asked for the same on
`QuestionnaireResponse`, and Paul Lynch opened FHIR-55490. The committee
resolved it as Persuasive with the wording quoted above.

### Proposed Change

Add `<orderMeaning>` to the `<differential>` of each of the three element
definitions, placed immediately after `<type>` and before any `<constraint>`
or `<mapping>` (matching the existing pattern in
`source/bundle/structuredefinition-Bundle.xml` line 370 and
`source/examplescenario/structuredefinition-ExampleScenario.xml`).

In `source/questionnaire/structuredefinition-Questionnaire.xml`:

- On the `<element id="Questionnaire.item">` definition (currently at line
  727; the `<type><code value="BackboneElement"/></type>` is on line 731 in
  the differential), insert:

  ```xml
  <orderMeaning value="The order items are listed indicates the order in which they should be presented to the user (if they are enabled)"/>
  ```

- On the `<element id="Questionnaire.item.answerOption">` definition
  (currently at line 1310; the `<type><code value="BackboneElement"/></type>`
  is on line 1314 in the differential), insert:

  ```xml
  <orderMeaning value="The order of the options indicates the order in which they should be presented to the user by default.  Extensions may drive changes to the presentation order."/>
  ```

In `source/questionnaireresponse/structuredefinition-QuestionnaireResponse.xml`:

- On the `<element id="QuestionnaireResponse.item">` definition (currently at
  line 373; the `<type><code value="BackboneElement"/></type>` block ends near
  line 388 in the differential), insert:

  ```xml
  <orderMeaning value="The order of the items is the order in which the questions were presented to the user when completing the Questionnaire."/>
  ```

No other elements, examples, or pages are touched. The publisher will
regenerate `<snapshot>` content; do not hand-edit snapshots (per the briefing).

### Design Rationale

`orderMeaning` is the spec-sanctioned mechanism for declaring that sibling
order is meaningful. Using it (rather than only mentioning order in `comment`
or `definition`) lets downstream tooling â€” diff/merge utilities, validators,
publication renderers â€” surface the constraint consistently, the same way it
already does for `Bundle.entry` and the various `ExampleScenario` repeating
backbones.

The exact wording is taken directly from the resolution description, so there
is no editorial freedom to exercise. Putting `orderMeaning` only on
`Questionnaire.item` and `Questionnaire.item.answerOption` (and not on
`Questionnaire` itself) matches the Zulip discussion: `Questionnaire` is the
container, and order significance lives at the repeating-child level. The
extension-level concern Paul Lynch raised on Zulip ("orderMatter perhaps
should be set on Questionnaire and Questionnaire.item.extension") is **not**
in the committee resolution and must not be silently rolled in.
