## Resolution Summary

The ticket was resolved **Persuasive** by FHIR Infrastructure with no
free-text resolution description and no comments captured in Augury. The
substance of the fix is therefore taken directly from the ticket
description, which raises three concrete defects on the Metadatatypes
and Datatypes pages:

1. The `Contributor` data type is still drawn in the Metadatatypes
   diagram even though the type itself has been removed from the
   specification.
2. `Contributor` is still listed as a usage site for `ContactDetail` on
   the Metadatatypes page (`ContactDetail is used in the following
   places: ŌĆ”`).
3. `CodeableReference` is rendered under "Special Purpose Datatypes"
   on `datatypes.html`, but it is defined as a general-purpose Complex
   data type (declared in `source/fhir.ini` under `[types]` alongside
   `CodeableConcept`, `Coding`, etc.).

A `Persuasive` resolution requires the spec to be updated so that all
three observations stop being true. (1) and (2) are downstream
consequences of fully removing the `Contributor` data type; (3) is a
diagram-categorisation bug independent of `Contributor`.

## Feature Proposal

### Problem Statement

The Metadatatypes and Datatypes pages of the FHIR Core specification
currently misrepresent the type framework:

- The Metadatatypes diagram (`source/datatypes/metadatatypes.diagram`,
  rendered into `source/metadatatypes.html` via the
  `[%diagram datatypes/metadatatypes.diagram 3dt%]` macro) used to
  include `Contributor`, even though `Contributor` was removed from
  R5+. While the diagram listing in this repo no longer mentions
  `Contributor` at clone HEAD `db79dcd` (see "Impact Analysis" below),
  related residue from the deletion still exists in `source/fhir.ini`
  (a stale `contributor=metadatatypes` `[type-pages]` mapping at line
  99) and may still be flowing through the publisher's `[%dtusage ŌĆ” %]`
  macro into the "ContactDetail is used in the following places" line
  on `metadatatypes.html` (line 64).
- `CodeableReference` is listed in
  `source/datatypes/specialtypes.diagram`, which is rendered into the
  "Special Purpose Datatypes" panel of `source/datatypes.html` (line 31:
  `[%diagram datatypes/specialtypes.diagram 4dt%]`). This contradicts
  its declaration in `source/fhir.ini` `[types]` (line 18) where
  `CodeableReference` sits with the other general-purpose complex
  types. Readers cross-referencing the datatypes page and the type
  framework get conflicting categorisations.

### Proposed Change

Three concrete edits are required.

**A. Re-categorise `CodeableReference` as a general-purpose Complex
data type in the diagrams.**

- Remove the `CodeableReference` token from the
  `classes=DataType,BackboneType,Reference,Narrative,Extension,Meta,ElementDefinition,Dosage,xhtml,CodeableReference,DosageDetails,DosageCondition,DosageSafety`
  line of `source/datatypes/specialtypes.diagram`.
- Remove the corresponding `CodeableReference=-80;-50` directive from
  the `[directions]` block of `specialtypes.diagram` and re-balance
  the remaining `[size]` width if visual layout warrants it (the
  current width is `440`; after removing CodeableReference the diagram
  may comfortably narrow ŌĆö check render before committing).
- Add `CodeableReference` to the `classes=` line of
  `source/datatypes/alltypes.diagram` (which currently lists
  `Coding,CodeableConcept,ŌĆ”` as the General-Purpose Datatypes
  rendered in the right-hand panel of `datatypes.html`).
- Add a `CodeableReference=<x>;<y>` direction line to the
  `[directions]` block of `alltypes.diagram`, placed near `Coding` /
  `CodeableConcept` (e.g. `CodeableReference=-100;0` ŌĆö exact offset
  to be verified visually after publisher render).

This makes the Datatypes page diagrams consistent with the
authoritative `[types]` listing in `source/fhir.ini`.

**B. Confirm and finish the `Contributor` removal so the Metadatatypes
diagram and "used-in" listing are clean.**

The bulk of the `Contributor` deletion landed in commit
`03893addeb` ("fully remove Contributor data type
(FHIR-21220 Add 'publisher' to contributor type valueset)"), which is
ancestral to the analysed HEAD `db79dcd`. As a result:

- `source/datatypes/metadatatypes.diagram` no longer lists
  `Contributor` (verified at HEAD).
- `source/datatypes/contributor.xml` is deleted.

However, `source/fhir.ini` still contains a stale
`[type-pages]` entry at line 99: `contributor=metadatatypes`. Because
the publisher consults `[type-pages]` when rendering datatype usage
listings, this entry can keep `Contributor` appearing in
`[%dtusage ContactDetail%]` and similar lookups. The remediation:

- Delete line 99 (`contributor=metadatatypes`) from
  `source/fhir.ini`.
- After running the publisher, re-inspect
  `source/metadatatypes.html` rendered output for the line
  `ContactDetail is used in the following places: ŌĆ”` (current
  source line 64, macro `[%dtusage ContactDetail%]`) and confirm
  `Contributor` no longer appears. If it does, trace the publisher's
  `dtusage` resolver in `tools/` to find the residual reference.

**C. Final visual / link sweep.**

- Re-render the spec locally and visually compare the
  Metadatatypes diagram, the Datatypes "Special Purpose" diagram,
  and the Datatypes "General-Purpose" diagram against the proposed
  state above.
- Run a repo-wide search for `Contributor` under `source/` to make
  sure no other narrative page (e.g. ActivityDefinition or
  PlanDefinition prose) still cross-links to `metadatatypes.html#Contributor`.
  Note: 03893addeb already cleaned the structuredefinition XML for
  ActivityDefinition / PlanDefinition / Parameters / Task, so the
  expected residue is documentation prose only.

### Design Rationale

- The `[types]` block in `source/fhir.ini` is the FHIR-I-approved
  inventory of general-purpose reusable data types ("FHIR-I must
  approve changes to this list" ŌĆö line 12 of `fhir.ini`). Anything
  that lives in that block should appear in the General-Purpose
  Datatypes diagram, not the Special Purpose one. Moving
  `CodeableReference` to `alltypes.diagram` aligns the rendered
  documentation with the type-framework registry instead of
  introducing a new categorisation rule.
- The `Contributor` clean-up is purely deletion-residue work: there
  is no design choice ŌĆö the type does not exist anymore, so any
  dangling reference to it is a defect. The `fhir.ini`
  `[type-pages]` entry is the most likely remaining channel by which
  the publisher could still emit `Contributor` text into rendered
  HTML.
- All edits are confined to authored files (`source/*.html`,
  `source/datatypes/*.diagram`, `source/fhir.ini`); no
  `<snapshot>` blocks, no Excel-form `source/datatypes/<type>.xml`
  spreadsheets, and no generated areas are touched, in line with
  the FhirCore briefing's authoring rules.
