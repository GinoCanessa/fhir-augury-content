## Resolution Summary

The resolution is **Persuasive**, with the description: *"We will add more explanation about how the text representation can be used."*

Concretely, the GraphDefinition specification's *Text Representation* section must be expanded so that a reader can understand:

- What the section is (a convenience short-hand syntax for authoring GraphDefinition instances), as opposed to FHIR Shorthand (FSH) or another DSL;
- When and how it is intended to be used (e.g., as the value of the `definition` parameter on the `$graph` operation, where supported, and/or as a tooling / authoring shortcut that is later serialised to a normal JSON/XML GraphDefinition resource);
- How it relates to the formal GraphDefinition resource (one-to-one mapping of `node`/`link` statements to `GraphDefinition.node` / `GraphDefinition.link` elements);
- Any limitations / caveats (informative status, no formal grammar yet, whitespace handling already noted in the page).

The resolution does **not** call for a change to the GraphDefinition StructureDefinition itself, examples, terminology, or other resources — it is a documentation-only clarification scoped to the existing "Text Representation" sub-section.

## Feature Proposal

### Problem Statement

The reporter (Ricardo Quintano) flagged that the GraphDefinition page presents a "Text Representation" section without explaining what it is or how it relates to the rest of FHIR. In particular it is not clear whether this short-hand is:

- A separate authoring DSL (similar to FHIR Shorthand) that gets compiled into GraphDefinition instances, or
- A literal value that can be passed to a server (e.g., via a query parameter), or
- An informal teaching aid that has no normative use.

The current notes file (`input/pagecontent/StructureDefinition-GraphDefinition-notes.xml`) introduces a short-hand example earlier in the page (line 96, where a server "may also allow clients to pass in their own graph definition using a text representation" as the `definition` parameter to `$graph`), but the dedicated *Text Representation* section starting at line 312 jumps straight into syntax (`node`, `link`) without restating that purpose, the audience, or the mapping back to the GraphDefinition resource. A reader landing on `#text` from the table of contents has no context.

### Proposed Change

Edit `input/pagecontent/StructureDefinition-GraphDefinition-notes.xml`, in the section anchored `<a name="text"></a>` / `<h4>Text Representation</h4>` (currently lines 311–312), inserting one or two new explanatory paragraphs *before* the existing "For convenience…" sentence (current line 313–316) and tightening the closing prose. The new prose should cover:

1. **Purpose.** State explicitly that the Text Representation is a convenience short-hand for authoring or transmitting a GraphDefinition. Clarify that it is **not** related to FHIR Shorthand (FSH), and that it is informative — the JSON/XML serialization of GraphDefinition is the normative form.
2. **Where it is used.** Cross-reference the earlier discussion at the `#fetch` section (line 96 of the same file) where the short-hand can appear as the `definition` parameter on the `$graph` operation; note that tooling MAY also accept it as input and convert to a `GraphDefinition` resource.
3. **Mapping to the resource.** Add one short paragraph (or a small bullet list) stating that each `node` statement corresponds to a `GraphDefinition.node` element (with `nodeId`, `description`, `type`, optional `profile`) and each `link` statement corresponds to a `GraphDefinition.link` element (with the bracketed expression mapping to `link.path`, the arrow target to `link.targetId`, the `?params` to `link.params`, and the trailing compartment clause to `link.compartment`).
4. **Status / caveats.** Note that the syntax has no formal grammar published in this IG and is provided to make the examples easier to read; servers are not required to accept it.

Suggested wording (illustrative — work-group is free to refine):

```html
<a name="text"></a>
<h4>Text Representation</h4>
<p>
For convenience, a graph definition may be written using a compact text short-hand
in addition to its normative JSON or XML serialization. This short-hand is provided
purely as an authoring and documentation aid: it is informative, has no formal
grammar in this IG, and is not related to
<a href="https://hl7.org/fhir/uv/shorthand/">FHIR Shorthand (FSH)</a>.
</p>
<p>
The short-hand may appear in two places:
</p>
<ul>
  <li>As the value of the <code>definition</code> parameter on the
    <a href="OperationDefinition-GraphDefinition-graph.html">$graph operation</a>,
    where a server documents support for it (see
    <a href="#fetch">Fetching a graph of resources</a> above).</li>
  <li>As input to tooling that converts the text into a
    <code>GraphDefinition</code> resource for storage or exchange.</li>
</ul>
<p>
Each statement in the short-hand maps directly to a GraphDefinition element:
</p>
<ul>
  <li><code>node</code> &rarr; one entry in <code>GraphDefinition.node</code>
    (<code>nodeId</code>, <code>description</code>, <code>type</code>, optional <code>profile</code>).</li>
  <li><code>link</code> &rarr; one entry in <code>GraphDefinition.link</code>
    (bracketed expression &rarr; <code>link.path</code>, arrow target &rarr; <code>link.targetId</code>,
    <code>?params</code> &rarr; <code>link.params</code>, trailing compartment clause &rarr; <code>link.compartment</code>).</li>
</ul>
<p>
A graph definition is introduced by two or more node statements:
</p>
<!-- existing example pre/code blocks continue unchanged from current line 317 onward -->
```

No other files require changes.

### Design Rationale

- **Documentation-only fix.** The reporter explicitly asked for clarification, not a syntactic change, and the resolution ("more explaination about how the text representation can be used") matches that.
- **Minimal disruption.** Inserting clarifying prose before the existing example preserves all current content (node/link syntax, the worked examples on lines 317–375, and the `#text` anchor) so inbound links and search results stay valid.
- **Cross-link instead of duplication.** The `$graph` operation use-case is already described at the top of the same page (line 96 area) and in `OperationDefinition-GraphDefinition-graph.html`; cross-linking avoids two sources of truth.
- **Explicit "not FSH" disclaimer.** The reporter's confusion ("Is this the ShortHand syntax to create GraphDefinition instances?") is best addressed by naming FHIR Shorthand and stating the relationship explicitly, rather than relying on the reader to infer it.
- **Mapping table.** Spelling out node/link → GraphDefinition element correspondences gives implementers a deterministic reading of the short-hand without inventing a formal grammar (which would be a much larger normative change and is out of scope for a Persuasive resolution).
