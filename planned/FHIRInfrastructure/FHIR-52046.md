## Resolution Summary

**Resolution: Persuasive.** The committee accepted the reporter's argument that
the existing `tst-7`/`tst-8`/`tst-9` invariants on `TestScript.*.action.operation`
incorrectly enumerate the operation type codes that require a `sourceId` (the
request payload). The resolution text — verbatim from the ticket — is:

> For tst-7, will change description to: "Operation must have a source id unless
> the method is one that doesn't always require a payload or the operation type
> is one that doesn't always require a payload" with FHIRPath of:
>
> `sourceId.exists() or method in ('get' | 'delete' | 'head') or not(type.code in ('create' | 'create-conditional' | 'update' | 'update-conditional' | 'patch' | 'patch-conditional' | 'transaction' | 'batch'))`
>
> We will remove the other two because they're already inherited from where
> Operation is defined.

Concretely, the change has two parts:

1. **Replace** the existing `tst-7` invariant on the canonical Operation
   backbone definition (`TestScript.setup.action.operation`) with a new human
   description and a new FHIRPath expression that switches from a positive
   allow-list of "no-payload" type codes to a negative deny-list of
   payload-bearing type codes (`create*`, `update*`, `patch*`, `transaction`,
   `batch`) plus a method-based escape (`get`/`delete`/`head`).
2. **Remove** the duplicated copies of the same operation-shape invariant from
   the other action paths (`test.action.operation`, `teardown.action.operation`,
   `common.action.operation`), since those paths use `contentReference` to
   `#TestScript.setup.action.operation` and therefore inherit its constraints
   automatically.

## Feature Proposal

### Problem Statement

`TestScript` defines an "Operation must have a `sourceId` unless …" invariant
(currently `tst-4`/`tst-10`/`tst-15`/`tst-17` in the live R6 source — historically
`tst-7`/`tst-8`/`tst-9` in earlier drafts) on every `*.action.operation` path. The
exception list enumerates `'capabilities' | 'search' | 'transaction' | 'history'`
as the operation type codes for which a `sourceId` is _not_ required.

The reporter identified two correctness problems and one structural one:

- **`transaction` is wrong on the deny side.** A FHIR `transaction` is a Bundle
  whose contents must be supplied by the test author; it is meaningless to
  declare a transaction operation with no `sourceId` pointing at a fixture.
  Listing `transaction` as an exception lets invalid TestScripts pass
  validation.
- **`batch` is missing.** `batch` has the same payload requirement as
  `transaction`. Either both belong on the exception list or neither does. The
  current rule treats `batch` as requiring a `sourceId` while exempting
  `transaction`, which is internally inconsistent.
- **Coverage gap on level-2 codes.** `valueset-testscript-operation-codes`
  enumerates many more level-2 codes than the four currently named in the
  invariant. The invariant should reason about the _category_ of operation
  (read-only / payload-bearing) rather than try to enumerate every spec-level
  code.

The resolution addresses all three by (a) inverting the rule to a deny-list of
the operations that _do_ require a payload (`create*`, `update*`, `patch*`,
`transaction`, `batch`) and (b) adding the HTTP-method escape so verb-based
overrides (`get`, `delete`, `head` — none of which need a body) satisfy the
constraint regardless of `type.code`.

### Proposed Change

**Affected resource:** `TestScript`, in
`HL7/fhir-testing-ig/input/resources/testscript/StructureDefinition-TestScript.xml`
(the canonical `TestScript` StructureDefinition; no separate JSON authoring
file is present).

**Change 1 — rewrite the canonical operation invariant (the resolution refers
to it as `tst-7`; in the live source it is `tst-4`, on element
`TestScript.setup.action.operation`):**

| Field | Before (live source) | After (resolution) |
|---|---|---|
| `key` | `tst-4` | `tst-4` _(or `tst-7`; see Open Questions)_ |
| `severity` | `error` | `error` |
| `human` | "Setup operation will contain either sourceId or targetId or params or url." | "Operation must have a source id unless the method is one that doesn't always require a payload or the operation type is one that doesn't always require a payload" |
| `expression` | `sourceId.exists() or (targetId.count() + url.count() + params.count() = 1) or (type.code in ('capabilities' \| 'search' \| 'transaction' \| 'history'))` | `sourceId.exists() or method in ('get' \| 'delete' \| 'head') or not(type.code in ('create' \| 'create-conditional' \| 'update' \| 'update-conditional' \| 'patch' \| 'patch-conditional' \| 'transaction' \| 'batch'))` |
| `source` | `http://hl7.org/fhir/StructureDefinition/TestScript\|0.1.0-SNAPSHOT` | unchanged |

**Change 2 — remove the duplicate operation-shape constraint blocks** from the
sibling action paths that already `contentReference` the canonical definition:

- `TestScript.test.action.operation` → currently re-declares `tst-10` with the
  same expression as old `tst-4`. **Remove the entire `<constraint>` block.**
- `TestScript.teardown.action.operation` → currently re-declares `tst-15` with
  the same expression. **Remove the entire `<constraint>` block.**
- `TestScript.common.action.operation` → currently re-declares `tst-17` with
  the same expression. **Remove the entire `<constraint>` block.**

After removal those elements still carry their `<contentReference
value="#TestScript.setup.action.operation"/>`, so the new (rewritten)
constraint propagates automatically.

**No change** is required to:

- The `valueset-testscript-operation-codes` ValueSet — the new expression
  enumerates type-code literals directly and does not depend on a binding
  change.
- `TestScript.variable.sourceId`, `TestScript.setup.action.assert.sourceId`,
  `TestScript.setup.action.assert.compareToSourceId`, or any of the other
  existing `tst-*` invariants on assert/variable paths (`tst-2`, `tst-5`,
  `tst-7`, `tst-8`, `tst-9`, `tst-11`, `tst-18` in the live source). The
  reporter's `tst-7`/`tst-8`/`tst-9` references in the Jira description are
  using older invariant numbering for the same operation-shape rule, not the
  current assert-shape invariants that happen to share those keys today.

### Design Rationale

- **Inverting the polarity of the type-code list** future-proofs the invariant
  against new level-2 codes added to `testscript-operation-codes`. New
  read-style codes (e.g., future `vread`-like codes) automatically satisfy the
  rule without a spec edit, whereas only payload-bearing codes — a closed,
  stable set tied to the FHIR REST verbs — need to stay enumerated.
- **Including `method`** acknowledges that a TestScript author can override the
  HTTP verb explicitly. If the verb is `get`/`delete`/`head`, no body is sent
  on the wire regardless of what the operation `type.code` says, so requiring
  a `sourceId` would be wrong.
- **Deduplicating via `contentReference`** removes a long-standing maintenance
  hazard: the four parallel copies of this invariant have drifted before (the
  ticket itself is partly a symptom of that), and FHIR's `contentReference`
  semantics already say constraints on the referenced backbone propagate to
  the reference site. Carrying four copies forward would re-create the same
  drift the next time the rule is tweaked.
- **Alternative considered:** keeping the four copies and only fixing the
  expression in each. Rejected by the resolution because the duplication itself
  is the root cause of past bugs and there is no semantic reason for the four
  paths to diverge.

### Breaking Changes

This change is **a tightening of a validation rule** and is therefore
backward-incompatible for any TestScript instance that exploits the old text:

- **TestScripts with a `type.code = 'transaction'` operation and no
  `sourceId`** will start failing validation. They were arguably invalid
  before (the resolution's whole point is that the old rule was wrong to
  permit this), but tooling that relied on the published invariant will flag
  them as new errors.
- **TestScripts with `type.code = 'capabilities'` or `'history'` and no
  `sourceId`/`targetId`/`params`/`url`** were previously valid via the
  type-code escape; under the new rule they remain valid because
  `capabilities`/`history` are not in the deny-list of payload-bearing codes.
  No regression.
- The `targetId.count() + url.count() + params.count() = 1` clause in the
  current expression is **dropped** in the resolution's new expression. That
  means a `'create'`-typed operation that today satisfies the rule by setting
  `url` (and no `sourceId`) would, under the new rule, fail validation
  because `'create'` is on the deny-list and there is no `method`-based
  escape unless the author also sets `method` to `get`/`delete`/`head`. This
  is the most material breaking change and should be called out in the IG's
  release notes / change log.

### Testing Considerations

- **Positive cases (must continue to pass):**
  - `read`/`vread`/`history`/`search`/`capabilities`-typed operations with
    no `sourceId` — satisfied via the `not(type.code in (…))` clause.
  - Any operation whose `method` is overridden to `get`, `delete`, or `head`
    — satisfied via the `method in (…)` clause regardless of `type.code`.
  - Any operation that explicitly sets `sourceId` to a defined fixture.
- **Negative cases (must fail under the new rule):**
  - `type.code = 'transaction'` with no `sourceId` (the headline bug).
  - `type.code = 'batch'` with no `sourceId` (newly covered).
  - `type.code = 'create'`/`'update'`/`'patch'` (and their `*-conditional`
    siblings) with no `sourceId` and no `method` override to a body-less verb.
- **Edge cases:**
  - `type.code` absent (e.g., the operation defines only `method` and `url`):
    `not(type.code in (…))` is `true` (the deny-list cannot match an absent
    code), so the invariant is satisfied. Confirm this matches WG intent —
    the resolution wording suggests yes.
  - `method` absent: the `method in (…)` clause is `false`, so the
    invariant falls through to the type-code check. No regression.
- **Interoperability:** existing test engines that hard-coded the old
  expression (with the `'capabilities' | 'search' | 'transaction' | 'history'`
  exemption) will need to update. Coordinate the change with at least one
  reference engine (e.g., AEGIS Touchstone, the assignee's own engine) to
  confirm the new expression is parseable and evaluates as expected on a
  representative TestScript corpus.
