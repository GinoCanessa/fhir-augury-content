## Feature Proposal

### Problem Statement

When an R4 Implementation Guide attempts to pre-adopt an R5 element using cross-version extensions (e.g., `http://hl7.org/fhir/5.0/StructureDefinition/extension-Appointment.virtualService`), tooling like the IG Publisher generates validation errors because the element's data type (`VirtualServiceDetail`) only exists in R5 and is not recognized in R4 contexts.

**`VirtualServiceDetail`** is an R5 metadata type with the following structure:
- `VirtualServiceDetail.channelType` (0..1, Coding)
- `VirtualServiceDetail.address[x]` (0..1, url | string | ContactPoint | ExtendedContactDetail)
- `VirtualServiceDetail.additionalInfo` (0..*, url)
- `VirtualServiceDetail.maxParticipants` (0..1, positiveInt)
- `VirtualServiceDetail.sessionKey` (0..1, string)

This type is used by three R5 resource elements:
- `Appointment.virtualService` (0..*)
- `Encounter.virtualService` (0..*)
- `Location.virtualService` (0..*)

Because `VirtualServiceDetail` does not exist in R4 (or R4B), any cross-version extension pointing to these elements fails validation. The reporter specifically needs this for the Canadian eReferral/eConsult specification, which needs `Appointment.virtualService` in R4.

### Proposed Change

The resolution delegates this to the cross-version extension packages (xver). The implementation requires:

1. **Cross-version extension package definitions** — The xver packages (specifically `hl7.fhir.uv.xver-r4.r5` for R5-in-R4) must include complete, valid StructureDefinition resources for extensions like `extension-Appointment.virtualService` that properly decompose `VirtualServiceDetail` into nested sub-extensions using only R4-compatible types.

2. **`_datatype` annotation** — The existing `_datatype` extension (`http://hl7.org/fhir/StructureDefinition/_datatype`) from the fhir-extensions pack provides the mechanism to annotate the original R5 type name (`VirtualServiceDetail`) on the extension, enabling tooling to understand the source type during cross-version round-tripping.

3. **Cross-version companion IG** — Per the related ticket FHIR-47645 (Persuasive with Modification, Highest priority), the cross-version extensions section of `versions.html` is being split into a separate companion IG. The detailed definitions and guidance for handling R5 data types will reside in that guide.

4. **Tooling updates** — The IG Publisher and FHIR Validator need updates to recognize and correctly process cross-version extension definitions that reference data types from other FHIR versions, using the `_datatype` annotation and `version-specific-use`/`version-specific-value` extensions.

### Design Rationale

The cross-version extension mechanism already has a standard URL pattern (`http://hl7.org/fhir/[version]/StructureDefinition/extension-[Path]`) and the `_datatype` extension to handle type mismatches. The gap is that the actual xver package definitions for elements typed with R5-only data types (like `VirtualServiceDetail`) have not been published yet. Rather than inventing a custom workaround, the resolution correctly identifies that the existing cross-version framework, once fully populated with definitions for R5 types, will address this need systematically for all R5-only types, not just `VirtualServiceDetail`.

### Step-by-Step Tasks

#### Task 1: Define the VirtualServiceDetail cross-version extension decomposition

The R5→R4 xver package must define how `VirtualServiceDetail` decomposes when used as a cross-version extension. Since `VirtualServiceDetail` doesn't exist in R4, the extension must represent it as a complex extension with nested sub-extensions:

**Extension URL:** `http://hl7.org/fhir/5.0/StructureDefinition/extension-Appointment.virtualService`

**Structure (nested sub-extensions):**
- `channelType` → `valueCoding` (Coding exists in R4)
- `address[x]` → type-specific sub-extensions:
  - `addressUrl` → `valueUrl`
  - `addressString` → `valueString`
  - `addressContactPoint` → `valueContactPoint` (ContactPoint exists in R4)
  - `addressExtendedContactDetail` → complex extension (ExtendedContactDetail is also R5-only; needs its own decomposition)
- `additionalInfo` → `valueUrl` (repeating)
- `maxParticipants` → `valuePositiveInt`
- `sessionKey` → `valueString`

The extension should carry the `_datatype` annotation:
```json
{
  "url": "http://hl7.org/fhir/StructureDefinition/_datatype",
  "valueString": "VirtualServiceDetail"
}
```

#### Task 2: Define equivalent extensions for Encounter and Location

Similar extension definitions are needed for:
- `http://hl7.org/fhir/5.0/StructureDefinition/extension-Encounter.virtualService`
- `http://hl7.org/fhir/5.0/StructureDefinition/extension-Location.virtualService`

These share the same `VirtualServiceDetail` decomposition structure.

#### Task 3: Handle the ExtendedContactDetail sub-type

`VirtualServiceDetail.address[x]` includes `ExtendedContactDetail` as one of its choice types. `ExtendedContactDetail` is also an R5-only data type that must be decomposed. The xver package must also include proper definitions for `ExtendedContactDetail` conversion (this may already exist in the `conversions-metadatatypes.xml` for fhir-extensions).

**File:** `HL7/fhir-extensions` — `input/pagecontent/conversions-metadatatypes.xml` (lines 78–81 contain the `ExtendedContactDetail` conversion map reference).

#### Task 4: Update the cross-version extensions companion IG

Per FHIR-47645, the cross-version extensions section is moving to a companion IG. The documentation must:
1. Describe the pattern for handling R5-only data types in cross-version extensions
2. Include VirtualServiceDetail as a worked example
3. Reference the `_datatype` extension mechanism
4. Document that elements with R5-only types are explicitly in scope for cross-version extensions (per FHIR-44614: "elements with types that do not exist")

**File:** `HL7/fhir` — `source/versions.html` (lines 679–683) already states extensions are defined for "elements with types that do not exist." The companion IG should expand this with specific guidance.

#### Task 5: Update tooling (IG Publisher / FHIR Validator)

The IG Publisher must be updated to:
1. Resolve cross-version extension URLs against the xver packages
2. Recognize the `_datatype` extension to understand the source type
3. Accept complex extension decompositions of R5-only types without generating validation errors
4. Use `version-specific-use` and `version-specific-value` extensions where definitions differ between versions

> **Note:** Tooling updates are outside the spec repos but are essential for this ticket to be fully resolved. The assignee's comment confirms: "Note that tooling will need updates at that time."

#### Task 6: Publish the xver packages

The cross-version extension packages referenced in `versions.html` (line 739: `hl7.fhir.uv.xver-r4.r5`, line 755: `hl7.fhir.uv.xver-r5.r4`, etc.) must be generated and published with the new VirtualServiceDetail extension definitions included.
