## Feature Proposal

### Problem Statement

When an implementer authors an extension in a pre-R5 IG that pre-adopts an R5 element — for example referencing `http://hl7.org/fhir/5.0/StructureDefinition/extension-Appointment.virtualService` — and the pre-adopted element's `type` is a datatype only defined in R5 (here `VirtualServiceDetail`), tooling such as the IG Publisher (IGP) emits an "invalid constrained type" error during QA. The error is shown in the screenshot attached to the Jira ticket and in `qa.json` (attachment 25087).

This blocks the Canadian eReferral / eConsult specification (and any IG using the R5 cross-version convention) from cleanly conveying R5-shaped information in an R4/R4B context, even though the data carried is a legitimate R5 pre-adoption. The reporter asked for either a tooling fix or an alternative modelling pattern that would survive the R6 migration.

### Proposed Change

Per the resolution, the change is **not** in `HL7/fhir` core. The accepted fix has three external pieces:

1. **Publish the cross-version extensions package(s)** that provide canonical pre-adoption extensions for R5 elements. These packages own the URL pattern, the StructureDefinition shape, and the rules for how an R5-only datatype is represented when the extension is consumed in an R4/R4B IG.

   - Concretely, an extension equivalent to `extension-Appointment.virtualService` will be defined in the cross-version-extensions package, with a `value[x]` (or sub-extension structure) typed in a way the publisher and validator can resolve in R4/R4B context.
   - The exact URL and structure are deferred to the cross-version-extensions package authors.

2. **Update IG Publisher / Validator tooling** to recognise cross-version extensions and to suppress the "invalid constrained type" error when the constraint comes from a higher FHIR version's datatype that the cross-version package has explicitly mapped/declared.

3. **Update FHIR Core documentation** (only once the guide is published) to point implementers at the cross-version-extensions package as the supported pattern for pre-adopting R5 elements, so future readers do not invent the `http://hl7.org/fhir/5.0/StructureDefinition/extension-...` pattern by hand.

No structural change to any HL7/fhir resource, profile, or extension is required by this ticket. No new element, datatype, search parameter, terminology binding, or invariant is being added to FHIR Core.

### Design Rationale

- **Keeps Core stable.** The reporter's request was for a tooling/modelling improvement, not a change to `Appointment` or any other R4 resource. Pushing the solution into the cross-version-extensions package preserves the FHIR Core surface and avoids version-specific carve-outs in resource StructureDefinitions.
- **Single source of truth for cross-version pre-adoption.** Multiple IGs (CA:eReC, others) need the same pattern. A shared package gives them one canonical URL and structure rather than each IG hand-rolling `http://hl7.org/fhir/5.0/...` extension references that today's tooling rejects.
- **Smooth R6 migration path.** The reporter explicitly asked for a model that "will translate well when we move to R6". A package-managed pre-adoption extension can be republished or aliased for R6 without touching IG content.
- **Tooling, not modelling.** The error is a tooling/QA artefact (IGP rejecting a datatype it does not know about), so the fix is correctly scoped to the tools and to a package that the tools can be taught about, rather than to FHIR Core's StructureDefinitions.

Alternatives implicitly rejected (per `mckenzie` / `ginocanessa` comments):

- **Adding R5 datatypes into R4/R4B core** — out of scope; would defeat the version model.
- **Per-IG hand-rolled extensions** — what implementers are doing today, and the source of the QA failure.
