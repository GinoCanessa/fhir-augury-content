## Feature Proposal

### Problem Statement

`source/mapping-language.html` (the FML specification page) contains three literal `TODO` placeholders that publish into the normative spec text. The reporter (Lloyd McKenzie, ballot comment 79) flagged them as "phrases that imply there is missing/incomplete content." Specification pages must not ship with TODOs — they either need real prose or the section must be removed (or moved out to an Incubator project). The parent ticket (FHIR-42980) sat unresolved from 2023 because no one supplied the missing prose; the FHIR-I work group has now agreed the resolution above.

### Proposed Change

Edit a single file, `source/mapping-language.html`, in three places:

**Change 1 — `list_modes` bullet (around line 483, inside the target-statement `<ul>`).**

Current text:

> `<li><b>list_modes</b>: control how elements that repeat are managed when the transform rule is evaluated. Values can be 'first', 'share', 'last' and 'single'. TODO: What do these do?</li>`

Replace with prose that defines each list mode value. The four codes correspond to the FHIR-defined `MapTargetListMode` value set already used by the `StructureMap.group.rule.target.listMode` element:

- **`first`** — only emit the rule's output for the first item produced by the source iteration; subsequent matches are discarded.
- **`share`** — share / collapse identical produced values across iterations into a single target item (de-duplicate by value).
- **`last`** — only emit the rule's output for the last item produced by the source iteration; earlier matches are discarded.
- **`single`** — assert that exactly one item is produced; raise an error if more than one source iteration would emit.

(The exact normative wording must be cross-checked against `source/structuremap/structuredefinition-StructureMap.xml` element `StructureMap.group.rule.target.listMode` and the bound value set so the FML page and the resource definition agree.)

**Change 2 — paragraph after the transform-functions table (around line 656).**

Current text:

> `<p>TODO: explain how optional parameters work with transforms (append only?), document list mode</p>`

Replace with two short paragraphs:

1. **Optional parameters on transform functions** — explain that, for transform functions whose signature includes an optional trailing parameter (e.g., `append(source, source...)`, `cast(source, type?)`), omitting the optional parameter causes the engine to apply the function in its append-only / inferred form: i.e., for `append`, append all supplied sources in order producing a single concatenated value; for `cast`, infer the target type from the assignment context when `type` is absent. The wording should be modelled on the existing prose for required parameters in the same table so the page reads consistently.
2. **List-mode interaction with transforms** — explain how the `list_modes` qualifier on the target (Change 1 above) interacts with a transform that itself produces multiple values: the transform is evaluated per source iteration, then the resulting values are passed through the `first` / `share` / `last` / `single` filter before being assigned to the target element. A forward link (`<a href="#target">target statement</a>`) to the `list_modes` bullet should be added.

**Change 3 — remove the `Mapping Support API` section (around line 918).**

Delete the following block in its entirety:

```html
<a name="api"></a>
<h3>Mapping Support API</h3><p>todo</p>
```

After deletion the page must read directly from the closing `</p>` of the preceding `Formal Grammar` section into `<a name="reserved"></a><h3>Reserved keywords</h3>`. The section heading number (`7.4.0.12`) auto-renumbers via the publisher; no other adjustment is needed in this file.

### Design Rationale

- The three changes mirror the per-TODO actions that FHIR-I voted on in the parent ticket (FHIR-42980 applied-vote comment by ginocanessa, 2026-01-12). This ticket's only obligation is "must resolve with the same resolution," so the implementation is the same edit set.
- For Change 1, the `list_modes` value definitions are sourced from the existing FHIR `StructureMap` resource and its bound `MapTargetListMode` value set rather than being newly invented — this keeps the prose aligned with the resource definition.
- For Change 2, the new prose deliberately adds a cross-reference back to the `list_modes` bullet so the two new pieces of documentation reinforce each other (the parent TODO explicitly asks for "document list mode" in addition to optional-parameter behaviour).
- For Change 3, removal (rather than authoring new prose) is what the work group voted for — the Mapping Support API material is being relocated to an Incubator project and is therefore out of scope for FHIR Core.
- Out-of-scope items (the `cast` `source, type?` question mark; missing `dateOp` documentation) are intentionally **not** addressed here, per Corey Spears's comment that they require their own ticket.
