## Feature Proposal

### Problem Statement

`source/exchanging.html` and its sibling `exchanging-*.html` sub-pages
describe ~10 different FHIR data-exchange patterns (REST CRUD, search,
batch/transaction, polling, subscription push, subscription pull,
operations, messaging query, messaging notification, human-mediated
request via Task, asynchronous patterns). The pages describe each
pattern in prose and link out to specs (e.g., `subscription.html`,
`messaging.html`, `task.html`) but they do **not** include any
machine-readable, walk-through example of the actual interactions
between `data source`, `data consumer`, and any intermediaries.

The clone confirms the gap: a literal HTML comment at
`source/exchanging.html` line 421 reads:

```html
<!-- TODO - create ExampleScenarios showing each of the workflows -->
```

The ExampleScenario resource is purpose-built for exactly this — a
sequenced description of actors, instances, and steps in an
interoperability flow. Da Vinci originally requested one per exchange
type so implementers had a concrete, runnable example of each pattern.
Today the only ExampleScenario instances in core are
`source/examplescenario/examplescenario-example.xml` (a generic toy
example) and `source/examplescenario/examplescenario-example-laborder.xml`
(a lab order tracked via Task — already partially satisfies the "Task"
half of the resolution).

### Proposed Change

Per the modified resolution, deliver **a small, prioritised set** of new
ExampleScenario instances — not one per exchange type. Concretely:

1. **Add new ExampleScenario instances** under
   `source/examplescenario/`, mirroring the existing two-file pattern
   (`<id>.xml` plus `<id>.plantuml`), for the patterns the Workflow
   call agrees are "complex enough":

   | New instance id | Exchange pattern (page) | Why complex enough |
   | --- | --- | --- |
   | `examplescenario-example-subscription-push` | `exchanging-subscription.html#push` | Subscription push: `Subscription` create + criteria evaluation + notification with full payload + security implications. |
   | `examplescenario-example-subscription-pull` | `exchanging-subscription.html#pull` | Subscription notification-only + subsequent pull query; two distinct flows. |
   | `examplescenario-example-messaging-notify` | `exchanging-messaging.html#notify` | MessageHeader + event payload + acknowledgement (and optional negative ack). |
   | `examplescenario-example-operation-async` | `exchanging-operation.html` (async pattern referenced from `async.html`) | Operation kick-off + 202 + status polling + result retrieval (the async pattern explicitly called out around line 938 of `exchanging.html`). |

   The existing `examplescenario-example-laborder.xml` already covers the
   Task / human-intervention pattern called out in the resolution and
   does **not** need to be duplicated; instead, link it from
   `exchanging-request.html`.

   The exact instance set is subject to Workflow-call confirmation
   (see Open Questions). The pattern is the same regardless of which
   subset is picked: each instance is one XML file + one PlantUML file
   plus one entry in the registration list.

2. **Register every new instance** in
   `source/examplescenario/list-ExampleScenario-examples.xml` with one
   `<entry>` element per instance, following the existing two entries:

   ```xml
   <entry>
     <extension url="http://hl7.org/fhir/build/StructureDefinition/description">
       <valueString value="Example of examplescenario - Subscription push notification"/>
     </extension>
     <extension url="http://hl7.org/fhir/build/StructureDefinition/title">
       <valueString value="examplescenario-example-subscription-push"/>
     </extension>
     <item>
       <reference value="ExampleScenario/example-subscription-push"/>
       <display value="Subscription push notification"/>
     </item>
   </entry>
   ```

3. **Wire each new instance into the relevant `exchanging-*.html`
   sub-page** by adding a "See ExampleScenario" link near the top of
   the page (before any anchor blocks the publisher emits), e.g. in
   `source/exchanging-subscription.html`:

   ```html
   <p>
     See <a href="examplescenario-example-subscription-push.html">ExampleScenario:
     Subscription push notification</a> for a worked example of this flow.
   </p>
   ```

4. **Remove (or reword) the line-421 TODO comment** in
   `source/exchanging.html`. Replace it with a short sentence pointing
   readers at the available ExampleScenarios, e.g.:

   ```html
   <p>
     For worked examples of the more complex flows above, see the
     <a href="examplescenario.html">ExampleScenario</a> instances:
     <a href="examplescenario-example-subscription-push.html">subscription push</a>,
     <a href="examplescenario-example-subscription-pull.html">subscription pull</a>,
     <a href="examplescenario-example-messaging-notify.html">messaging notification</a>,
     <a href="examplescenario-example-operation-async.html">async operation</a>,
     and <a href="examplescenario-example-laborder.html">Task-based human-mediated request</a>.
   </p>
   ```

5. **Optionally also link the existing `examplescenario-example-laborder`
   instance** from `source/exchanging-request.html` since that is the
   exchange page for the Task-mediated flow.

No StructureDefinition, ValueSet, CodeSystem, schema, or grammar
changes are required. The ExampleScenario resource definition itself
(`source/examplescenario/structuredefinition-ExampleScenario.xml`) is
**not** edited — only new instance examples and one HTML comment are
touched.

### Design Rationale

- The resolution is explicit that "all" is not the goal; only "complex"
  flows get a scenario. The proposed subset is taken directly from the
  resolution text (Subscription, Task) plus the two patterns
  (messaging notifications, async operations) that share the same
  multi-step / multi-actor / out-of-band-callback complexity profile
  and are otherwise hard to grok from prose alone. Pure REST CRUD,
  search, polling, and exchanging-rest batch/transaction are
  intentionally **not** in scope — they are well covered by `http.html`
  and `search.html`, and a one-step ExampleScenario adds little value.
- Following the existing `examplescenario-example-laborder.xml` +
  `examplescenario-example-laborder.plantuml` pattern keeps the
  publisher happy (the bespoke HL7 publisher already knows how to
  render those pairs) and avoids inventing new tooling. The PlantUML
  file is what produces the rendered sequence diagram visible in the
  built page.
- Registering each instance in
  `list-ExampleScenario-examples.xml` is required so the publisher
  picks up the new files and includes them in the resource's example
  list (consistent with how `example` and `example-laborder` are
  registered today).
- Linking from each `exchanging-*.html` sub-page (rather than only
  from the parent index) means a reader who lands on, say,
  `exchanging-subscription.html` from a search result still gets the
  scenario.
- Removing the TODO comment closes the literal in-source marker that
  this ticket is tracking; replacing it with prose pointing at the
  scenarios provides the navigation aid the original reporter wanted.
