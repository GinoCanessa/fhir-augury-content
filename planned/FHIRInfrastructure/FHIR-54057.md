## Resolution Summary

The committee accepted the cleanup as **Persuasive with Modification**. The
approved resolution has two parts:

1. **Add two new codes** to the FHIR restful-interaction code system so that
   conditional read variants can be expressed via `interaction.code` rather
   than via the type-level `conditionalRead` element:
   - `read-conditional-modified-since` (covers the existing
     `conditional-read-status#modified-since` semantics — `If-Modified-Since`).
   - `read-conditional-not-match` (covers the existing
     `conditional-read-status#not-match` semantics — `If-None-Match`).

2. **Deprecate the type-level conditional* elements** on
   `CapabilityStatement.rest.resource`:
   - `conditionalCreate`, `conditionalRead`, `conditionalUpdate`,
     `conditionalDelete`, `conditionalPatch` — all marked deprecated, with
     instructions to use the corresponding `interaction.code` values
     (`create-conditional`, `read-conditional-modified-since` /
     `read-conditional-not-match`, `update-conditional`,
     `delete-conditional-single` / `delete-conditional-multiple`,
     `patch-conditional`).
   - The associated **value sets and code systems** are also deprecated where
     they exist — concretely `CodeSystem/conditional-read-status`,
     `ValueSet/conditional-read-status`, `CodeSystem/conditional-delete-status`,
     and `ValueSet/conditional-delete-status` (verified in the clone). There is
     no separate VS/CS for the boolean `conditionalCreate`, `conditionalUpdate`,
     or `conditionalPatch` flags.

The elements are not removed in this cycle — only deprecated — so existing
CapabilityStatement instances remain valid.

## Feature Proposal

### Problem Statement

`CapabilityStatement.rest.resource` carries five element-level signals for
conditional interaction support that pre-date the modern `interaction` backbone:

- `conditionalCreate` (boolean, 0..1)
- `conditionalRead` (code, 0..1, bound to ConditionalReadStatus)
- `conditionalUpdate` (boolean, 0..1)
- `conditionalPatch` (boolean, 0..1)
- `conditionalDelete` (code, 0..1, bound to ConditionalDeleteStatus)

The `restful-interaction` code system already enumerates the corresponding
behaviors directly on `interaction.code`:

- `create-conditional`
- `update-conditional`
- `patch-conditional`
- `delete-conditional-single`, `delete-conditional-multiple`

This leaves two ways to declare the same capability, which is redundant,
inconsistent (booleans vs. codes vs. structured `interaction` repeats), and a
source of authoring confusion. The reporter notes that conditional read is the
only case not yet representable purely via `interaction.code`, because the
HTTP-header-driven variants (`If-Modified-Since`, `If-None-Match`,
full-support) require finer granularity than a single `read-conditional` code.

### Proposed Change

**Repository:** `HL7/fhir` (FhirCore)

#### 1. Extend `CodeSystem/restful-interaction` with two new conditional-read codes

File: `source/codesystem/codesystem-restful-interaction.xml`

Add two new `<concept>` entries (anywhere after the existing `read` /
`vread` concepts, alongside the other `*-conditional` codes):

```xml
<concept>
  <code value="read-conditional-modified-since"/>
  <display value="read-conditional-modified-since"/>
  <definition value="Read the current state of the resource, conditional on the resource having been modified since a client-supplied timestamp (HTTP If-Modified-Since)."/>
</concept>
<concept>
  <code value="read-conditional-not-match"/>
  <display value="read-conditional-not-match"/>
  <definition value="Read the current state of the resource, conditional on the resource version not matching a client-supplied ETag (HTTP If-None-Match)."/>
</concept>
```

These are pure additions to a `content="complete"` code system; bump no
existing concept semantics.

#### 2. Add the new codes to `ValueSet/type-restful-interaction`

File: `source/capabilitystatement/valueset-type-restful-interaction.xml`

Add the two new codes to the `compose.include` block (and to the rendered
`text.div` table for narrative consistency):

```xml
<concept>
  <code value="read-conditional-modified-since"/>
</concept>
<concept>
  <code value="read-conditional-not-match"/>
</concept>
```

(`ValueSet/system-restful-interaction` does **not** need updating — these are
type/instance-level interactions, not system-level.)

#### 3. Deprecate the five conditional* elements in CapabilityStatement

File: `source/capabilitystatement/structuredefinition-CapabilityStatement.xml`

For each of the elements listed below, add a
`structuredefinition-standards-status` extension carrying
`valueCode="deprecated"` plus a nested
`structuredefinition-standards-status-reason` markdown extension that points
authors at the `interaction.code` replacement. The existing `min`/`max`/`type`
must remain unchanged so that existing instances remain valid.

Pattern (the same shape used by the `outcome` concept in
`source/bundle/codesystem-search-entry-mode.xml` lines 60–71):

```xml
<extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
  <valueCode value="deprecated">
    <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status-reason">
      <valueMarkdown value="Deprecated in favor of declaring the corresponding code on `CapabilityStatement.rest.resource.interaction.code` (see FHIR-54057)."/>
    </extension>
  </valueCode>
</extension>
```

Targets (current line numbers in the clone @ `db79dcd`):

| Element id | Lines | Replacement on `interaction.code` |
|---|---|---|
| `CapabilityStatement.rest.resource.conditionalCreate` | 1061–1071 | `create-conditional` |
| `CapabilityStatement.rest.resource.conditionalRead` | 1072–1090 | `read-conditional-modified-since` and/or `read-conditional-not-match` (a server with both is full-support) |
| `CapabilityStatement.rest.resource.conditionalUpdate` | 1091–1101 | `update-conditional` |
| `CapabilityStatement.rest.resource.conditionalPatch` | 1102–1112 | `patch-conditional` |
| `CapabilityStatement.rest.resource.conditionalDelete` | 1113–1131 | `delete-conditional-single` and/or `delete-conditional-multiple` |

Also update each element's `<comment>` to add a one-sentence pointer to the
new approach (e.g., "Deprecated; use `interaction.code = create-conditional`
instead.") so the rendered documentation surfaces the deprecation prominently.

#### 4. Update the `interaction.code` short / display

File: `source/capabilitystatement/structuredefinition-CapabilityStatement.xml`,
line 990.

Append the two new codes to the `<short>` so the rendered table lists them:

```
read | read-conditional-modified-since | read-conditional-not-match | vread | update | update-conditional | patch | patch-conditional | delete | delete-conditional-single | delete-conditional-multiple | delete-history | delete-history-version | history-instance | history-type | create | create-conditional | search-type
```

#### 5. Deprecate the conditional-read and conditional-delete VS/CS

Files (mark the **whole resource** deprecated by changing the value of the
top-level `structuredefinition-standards-status` extension from `normative`

- `source/capabilitystatement/codesystem-conditional-read-status.xml`
- `source/capabilitystatement/valueset-conditional-read-status.xml`
- `source/capabilitystatement/codesystem-conditional-delete-status.xml`
- `source/capabilitystatement/valueset-conditional-delete-status.xml`

The deprecation reason should read:
"Deprecated in FHIR R6 by FHIR-54057. Use the corresponding `*-conditional`

There are no `conditional-create-status` / `conditional-update-status` /
`conditional-patch-status` value sets in the clone (the original elements are
plain booleans), so this step is limited to the read and delete pairs.

#### 6. Update CapabilityStatement narrative

File: `source/capabilitystatement/capabilitystatement-notes.xml` (and, if it
exists in the rendered page, `capabilitystatement-introduction.xml`).

Add a short paragraph explaining the migration: that the five

#### 7. Update the example to use `interaction.code`

File: `source/capabilitystatement/capabilitystatement-example.xml` (and the

### Design Rationale

- The committee chose **deprecation, not removal**, to preserve backward
  compatibility for instances and downstream IGs that already populate the
  five `conditional*` elements. The data model stays valid; tooling can warn.
- The two new `read-conditional-*` codes preserve the granularity that
  `ConditionalReadStatus` provided (`modified-since`, `not-match`,
  `full-support`) without inventing a new element. Full support is expressed
  by repeating `interaction` with both new codes, which is consistent with how
  `delete-conditional-single` + `delete-conditional-multiple` already work.
- The standards-status / standards-status-reason extension pair is the
  established mechanism for code-level and resource-level deprecation in this
  repo (see `source/bundle/codesystem-search-entry-mode.xml`); reusing it
  keeps the deprecation machine-readable for IG publishers and tooling.
- The reporter explicitly excluded `conditionalRead` from the original cleanup
  because a single boolean / single code could not capture the HTTP-header
  variants. The committee's modification — adding the two new codes — closes
  that gap so the cleanup can be uniform across all five flags.

### Breaking Changes

- **No instance-level breaking changes.** All five `conditional*` elements
  remain optional (0..1) with their existing types and bindings, so existing
  CapabilityStatement instances continue to validate.
- **Soft conformance impact** for tooling that treats `standards-status =
  deprecated` as an error: anything still authoring the deprecated elements
  or referencing the deprecated VS/CS will surface warnings. This is the
  intended signal.
- **No code system-level breaking change.** The two new codes in
  `restful-interaction` are additions to a `content="complete"` code system;
  no existing code is renamed, removed, or re-defined.

### Related Specifications

- `ValueSet/type-restful-interaction` and `ValueSet/system-restful-interaction`
  bind `CapabilityStatement.rest.resource.interaction.code` and
  `CapabilityStatement.rest.interaction.code` respectively. Only the
  type-level value set needs the new codes; system-level read variants are
  not in scope.
- `ValueSet/conditional-read-status` (bound by
  `CapabilityStatement.rest.resource.conditionalRead`) and
  `ValueSet/conditional-delete-status` (bound by
  `CapabilityStatement.rest.resource.conditionalDelete`) become deprecated;
  the bindings remain in place because the elements themselves are only
  deprecated.
- Downstream IGs and capability profiles (e.g., US Core, IPS, SMART) that
  declare conditional support via the deprecated elements will eventually
  want to migrate; document the migration in the release notes but do not
  attempt to refactor IG repos as part of this ticket.
