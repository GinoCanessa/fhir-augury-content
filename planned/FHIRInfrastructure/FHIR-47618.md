## Resolution Summary

Resolution: **Persuasive**.

Resolution description (verbatim): *"Will check all links and will fix those that are broken. Will not change http to https unless http does not work. (Distinction between http and https will soon be disappearing anyhow.)"*

Concretely, this means:

1. Walk every external link on the "Mapping to Other Standards" page and verify it resolves.
2. Replace any broken link with a working equivalent (target the same authoritative resource where possible).
3. **Do not** perform a blanket `http://` → `https://` rewrite. Only switch a scheme when the `http://` form is itself broken.
4. The DICOM normative reference, which the reporter called out as broken, must be updated to point at `https://www.dicomstandard.org/`.

## Feature Proposal

### Problem Statement

The "Mapping to Other Standards" page (rendered as `mappings.html`) lists each external mapping space FHIR provides mappings into (HL7 v2, CDA, RIM, DICOM, LOINC, SNOMED CT, vCard, IHE XDS / SDC / DEX, openEHR, ISO 11179, BRIDG, ISO/IEEE 11073-10201/-10206/-10207, IDMP, ISBT 128, MDMI, NCPDP SCRIPT, ClinicalTrials.gov, etc.). The data driving the table lives in `source/mappingSpaces.xml`, where each `<space>` carries both a formal `<url>` (namespace identifier) and a navigable `<link>`.

Several of those `<link>` URLs are stale or dead — the reporter explicitly flagged the DICOM normative reference. A user hitting one of those links from the spec lands on a 404 / parked / redirected page, which undermines the value of the mappings page. The ticket also notes that many `<link>` values are still `http://`, but the resolution makes clear that only broken `http://` links must be migrated, not the working ones.

### Proposed Change

Edit `source/mappingSpaces.xml` to repair broken `<link>` targets, leaving the formal `<url>` namespace identifiers untouched (those identifiers are referenced from StructureDefinition `mapping.uri` values across the spec and must not be renamed). The page wrapper `source/mappings.html` itself has no embedded URLs (it just emits `[%mappings-table%]`) and does not need editing.

Specific edits known from the ticket:

- **DICOM mapping space** (`<id>dicom</id>`, `<sort>7</sort>`):
  - Update `<link>https://www.dicomstandard.org/current</link>` to `<link>https://www.dicomstandard.org/</link>` (the reporter's proposed replacement). The `current` sub-path is the broken target.
  - Apply the same fix to the historic duplicate DICOM entry (`<sort>49</sort>`, `<url>http://nema.org/dicom</url>`) which currently also points at `https://www.dicomstandard.org/current`.
  - Leave the `<url>` values (`https://dicomstandard.org/current`, `http://nema.org/dicom`) untouched — they are stable namespace identifiers used as `mapping.uri` keys in resources.

For the rest of the file, the implementer must perform a link-check pass (see Implementation Plan §"Link audit"). For each `<link>` that fails, choose the minimum change that restores a useful target:

1. If the same site exposes the resource at a new path → update the path.
2. If the resource has moved to a different host → point at the new authoritative host (prefer the standards-body home page or the canonical landing page for the standard, never a Web Archive snapshot unless no live equivalent exists).
3. If only the scheme is broken (i.e., `http://example.org/foo` 404s but `https://example.org/foo` works) → switch scheme. This is the only sanctioned http→https rewrite, per the resolution.
4. If no live equivalent exists → leave the `<link>` element empty (matching the existing pattern used by `<id>logical</id>` and `<id>w5</id>`) so the column heading still renders without a clickable URL, and add an XML comment recording when/why it was cleared.

Do not change `<columnName>`, `<title>`, `<id>`, `<sort>`, `<publish>`, `<sparse>`, `<pattern>`, or `<preamble>` content. Do not introduce new `<space>` entries; this is a maintenance pass, not a content addition.

### Design Rationale

- The resolution is explicit on scope: fix what is broken, do not rewrite working URLs. The bullet "Will not change http to https unless http does not work" guards against churn that would generate large diffs and risk breaking external citations to the published spec.
- The `<url>` element of each `<space>` is a stable identifier, used as the `uri` attribute on `<mapping>` elements inside StructureDefinition definitions across the entire spec. Changing those would silently break every per-element mapping reference. Only the `<link>` (the human-facing navigation target) is in scope.
- The DICOM fix matches what the reporter requested verbatim (`https://www.dicomstandard.org/`), so there is no design ambiguity for that entry.
- Clearing rather than fabricating a substitute link, when the original has no live equivalent, mirrors the existing convention already present in the file and avoids inventing an authoritative target.

### Step-by-Step Tasks

#### HL7/fhir

1. **Apply the DICOM normative-reference fix.**
   - File: `source/mappingSpaces.xml`
   - Action: In the `<space>` whose `<id>dicom</id>` and `<sort>7</sort>`, change `<link>https://www.dicomstandard.org/current</link>` to `<link>https://www.dicomstandard.org/</link>`. Apply the same change to the historic duplicate DICOM entry (`<id>dicom</id>` with `<sort>49</sort>`, currently commented as `<!-- historic -->` and carrying `<url>http://nema.org/dicom</url>`).
   - Details: Leave `<url>https://dicomstandard.org/current</url>` and `<url>http://nema.org/dicom</url>` untouched — these are namespace identifiers used as `mapping.uri` keys across StructureDefinitions and must not be renamed.

2. **Run a link audit over every `<link>` in `source/mappingSpaces.xml`.**
   - File: `source/mappingSpaces.xml`
   - Action: Extract every `<link>...</link>` value (including the live and historic DICOM entries, the v2 product-brief link, the CDA / RIM `product_brief.cfm` links, IHE wiki links such as `http://wiki.ihe.net/index.php?title=Data_Element_Exchange` and `http://wiki.siframework.org/...`, NCPDP, FDA UDI, OMG ServD, BRIDG, ISO/IEEE 11073, ISBT 128, NCI BRIDG, etc.) and probe each one (HEAD or GET) for a 2xx / 3xx-to-2xx response.
   - Details: For each non-2xx outcome, follow the Proposed Change decision tree:
     1. Same site, new path → update the path.
     2. Different host now authoritative → repoint to the canonical landing page for that standard.
     3. Only the scheme is broken → switch `http://` to `https://` (this is the only http→https rewrite sanctioned by the resolution).
     4. No live equivalent → empty the `<link>` element (`<link></link>`) and leave a short XML comment recording the date and reason, mirroring the existing pattern used by the `logical` and `w5` spaces.
   - Do not modify any of: `<url>`, `<columnName>`, `<title>`, `<id>`, `<sort>`, `<publish>`, `<sparse>`, `<pattern>`, `<preamble>`.

3. **Spot-check `source/mappings.html` for inline links.**
   - File: `source/mappings.html`
   - Action: Re-read the page and confirm the body contains no external `<a href>` URLs (current clone HEAD does not). If any are found, run them through the same audit and apply the same decision tree.
   - Details: The page is a thin wrapper around `[%mappings-table%]`; the table content all originates in `mappingSpaces.xml`, so edits here should be minimal-to-none.

4. **Sanity-check that no formal `<url>` was renamed.**
   - File: `source/mappingSpaces.xml`
   - Action: Diff the edited file against `master` and confirm only `<link>` elements were changed.
   - Details: A `mapping.uri` rename in any `<space>` would silently break every per-element mapping reference (`<mapping identity="dicom" />`, etc.) across the spec. Guard against it explicitly.

5. **Repo-wide search for stale `mappings.html` fragment links.**
   - File: `source/**`
   - Action: Search for incoming references like `mappings.html#`, `mappingSpaces`, and any `mapping.uri="..."` to make sure nothing depends on a `<link>` value that has just been blanked. (Per-resource `<mapping identity="..."/>` entries depend on `<id>`, not `<link>`, so they should be unaffected.)
   - Details: Inspired by the briefing's gotcha about removing anchors elsewhere. No edits expected, but a quick grep prevents surprises during publication.

### Open Questions

- **Audit cadence and authority for "broken" determination.** The resolution authorizes the implementer to fix broken links but does not enumerate which links beyond DICOM are broken at resolution time. The implementer should run the audit themselves and record results (e.g., in the PR description) so reviewers can confirm each repointed `<link>`.
- **Empty-link policy.** When no live equivalent exists, this plan recommends emptying the `<link>` (matching the existing `logical` / `w5` precedent) rather than removing the `<space>` or inventing a substitute. Reviewers should confirm this is acceptable; the alternative would be to keep a known-dead URL, which the resolution explicitly rejects.
- **Scope of `<url>` preservation.** This plan treats every `<url>` as a stable namespace identifier. If the work group ever needs to retire a mapping space outright, that would be a separate ticket — not in scope here.
