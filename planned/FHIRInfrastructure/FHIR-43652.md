## Resolution Summary

Resolution: **Persuasive with Modification**.

Per the resolution description: drop `GraphDefinition.link.sliceName` and replace it with a new element `GraphDefinition.link.elementId` with the same type (`string`) and cardinality (`0..1`) as the removed element:

- **Short:** "Id of link source element in node profile"
- **Definition:** "The ElementDefinition.id of the element that corresponds to the node in the node profile for which this link rule applies."
- **Comments:** "This is only necessary in addition to the path in situations where the profile has slicing such that there are multiple ElementDefinitions which share the same path. If included in such cases, the linkage rule only applies to the slice specified. If not included, the link rule applies independent of slicing. ElementId SHALL only be present if the node corresponding to the sourceId has a declared profile."

The intent is to remove the ambiguity of `sliceName` (it was never clear whether the slice referred to the source or target resource) by anchoring the link to a specific `ElementDefinition.id` on the source node's declared profile.

## Feature Proposal

### Problem Statement

The original ticket question — *"Is this a slice on the source or target resource?"* — captures a real ambiguity in `GraphDefinition.link.sliceName`. The element's name and one-line definition ("Which slice (if profiled)") do not say *which* of the two endpoints of a `link` (`sourceId` or `targetId`) the slice belongs to, nor how it should be resolved when multiple `ElementDefinition`s share the same `path` in the source profile. As `GraphDefinition` underpins the `$graph` operation and document/profile-driven traversal, this ambiguity makes link rules unimplementable in the presence of slicing.

### Proposed Change

Apply the following edits to `GraphDefinition.link`:

1. **Remove element:** `GraphDefinition.link.sliceName`
   - Current: `string`, `0..1`, short = "Which slice (if profiled)", definition = "Which slice (if profiled)."

2. **Add element:** `GraphDefinition.link.elementId`
   - Path: `GraphDefinition.link.elementId`
   - Type: `string`
   - Cardinality: `0..1`
   - Short: `Id of link source element in node profile`
   - Definition: `The ElementDefinition.id of the element that corresponds to the node in the node profile for which this link rule applies.`
   - Comments: `This is only necessary in addition to the path in situations where the profile has slicing such that there are multiple ElementDefinitions which share the same path. If included in such cases, the linkage rule only applies to the slice specified. If not included, the link rule applies independent of slicing. ElementId SHALL only be present if the node corresponding to the sourceId has a declared profile.`
   - Position: in place of the removed `sliceName` element (between `link.path` and `link.targetId`) so the ElementDefinition ordering of `link` children remains contiguous.

3. **Element-id renumbering:** Because `elementId` replaces `sliceName` 1-for-1, downstream child element ids/positions inside `link` are unchanged.

4. **Update narrative / text-representation docs** that reference `sliceName` (currently `input/pagecontent/StructureDefinition-GraphDefinition-notes.xml` documents the shorthand `sourceId[{path}:{sliceName}]`). Replace every occurrence of `sliceName` with `elementId` in narrative, including the link shorthand grammar, any worked examples, and the surrounding prose.

5. **Update or add an example** demonstrating the use of `elementId` against a sliced profile. Existing `graphdefinition-example.xml` does not use the old `sliceName`, so no example data must be migrated, but a new "sliced profile" example would help readers.

### Design Rationale

- **Anchors the rule unambiguously** to the source node — the new comment text explicitly ties `elementId` to `sourceId`'s declared profile, eliminating the source-vs-target ambiguity.
- **Uses `ElementDefinition.id`**, the canonical, stable handle for a specific element within a profile (slice or otherwise), rather than the slice name alone. This is consistent with how other FHIR artifacts (e.g., differential elements, ElementDefinition.contentReference) reference profile-internal elements.
- **Preserves cardinality and type** (`string`, `0..1`), minimizing wire-format and parser impact: a `<sliceName>` valued instance becomes a `<elementId>` valued instance with no schema-shape change beyond the element name.
- **Renaming rather than redefining** clearly signals to existing implementers that the semantics have changed, avoiding silent reinterpretation of pre-existing `sliceName` data.

### Breaking Changes

**Yes — this is a breaking change** for any system that has implemented `GraphDefinition.link.sliceName`:

- The element name on the wire changes from `sliceName` to `elementId`. Any persisted `GraphDefinition` instances or generators using `sliceName` will fail validation against the new StructureDefinition.
- The semantics also change: the new element holds an `ElementDefinition.id` (not a slice name). Existing data that put a slice name into `sliceName` is **not** semantically equivalent to the new `elementId` value space.
- Migration guidance should accompany the IG release notes: implementers must rewrite `sliceName="X"` to `elementId="<element id of slice>"` against the relevant node profile.

### Related Specifications

- **GraphDefinition `$graph` operation** — link traversal semantics depend on this element. Operation narrative may need a refresh if it currently illustrates `sliceName`.
- **ElementDefinition.id / slicing** — the new element's definition explicitly references `ElementDefinition.id`; ensure cross-links in narrative point at the ElementDefinition slicing page.
- **Composition `$document` operation** and other resources/operations that reference GraphDefinition by canonical URL are not directly affected (the canonical URL and resource id are unchanged).
