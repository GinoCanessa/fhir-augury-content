## Resolution Summary

The ticket reports that the cross-version-extension table on the FHIR Version Management Policy page lists DSTU2, R3, R4, R4B, and R5 as possible values for the `[defining_version]` URL token, but omits R6. The resolution refers to companion ticket [FHIR-47216](https://jira.hl7.org/browse/FHIR-47216), whose resolution description states verbatim:

> *We will update all tables on that page to have R6 as well as R5. (Will also fix the missing header for R5 in the mentioned table.)*

For FHIR-47208 specifically, the actionable change is to add an `R6` row to the `[defining_version]` table. A scan of the `source/versions.html` page in the current `HL7/fhir` clone (HEAD `db79dcd`) confirms that this is the only `class="grid"` table on the page that lists FHIR version codes and is missing an `R6` entry — every other table on the page is either unrelated to the version axis (FMM levels, standards-status definitions, version-string component definitions) or already includes R6 columns/rows (the cross-version package matrix at lines 706–769 already has full R6 coverage).

## Feature Proposal

### Problem Statement

The Version Management Policy page (`source/versions.html`) defines, in the "Extensions for Converting between Versions" section, the URL pattern `http://hl7.org/fhir/[defining_version]/StructureDefinition/extension-[Path]` for cross-version extensions. The page lists the legal values of `[defining_version]` in a small table immediately after the URL pattern. As of clone HEAD `db79dcd`, that table contains rows for DSTU2 (1.0), R3 (3.0), R4 (4.0), R4B (4.3), and R5 (5.0) — but no row for R6. Implementers reading the R6 build of this page therefore have no documented major-version token to use when constructing or interpreting cross-version extension URLs targeting R6, even though the package matrix further down the page (the format/extensions cross-tab at lines 706–769) already references `xver-r6.*` packages. The mismatch is a documentation correctness defect: one table on the page asserts R6 cross-version extensions exist, while another table in the same section implies the legal version-token list stops at R5.

### Proposed Change

Edit `source/versions.html` and add a new `<tr>` row for R6 at the end of the cross-version-extensions `[defining_version]` table (currently lines 652–673), immediately before the closing `</table>` on line 673. The new row must follow the same shape as the existing R5 row, must use the canonical lower-case `r6` URL path (matching the `r4`, `r4b`, `r5` rows already in the table — *not* the upper-case style used by the older `DSTU2`/`STU3` historical rows), and must mark R6 as the current version since R6 will be the published version of the spec where this change lands.

Concretely, insert after line 672 (after the existing R5 row's closing `</tr>`):

```html
<tr>
  <td><a href="http://hl7.org/fhir/r6">FHIR R6</a> (this version)</td>
  <td>6.0</td>
</tr>
```

…and update the existing R5 row (line 670) to remove the `(this version)` annotation, replacing it with a parenthetical that distinguishes R5 from R6 in line with the historical pattern (e.g., `<a href="http://hl7.org/fhir/r5">FHIR R5</a> (mixed STU/Normative)`, mirroring the way R4's row is annotated), so that exactly one row carries the `(this version)` marker. The exact annotation wording for R5 is left to the editor's discretion as long as it (a) drops `(this version)` and (b) is factually accurate for R5; if no other annotation is desired, leaving the row as plain `<a href="http://hl7.org/fhir/r5">FHIR R5</a>` is also acceptable.

No other tables on the page require changes for this ticket:

- The table at lines 706–769 (`Format:` / `R2 Package` / … / `R6 Package`) already lists R6 in both the column header and the row header, with `xver-r6.*` package links populated.
- The `<table class="lines">` at lines 195–251 (version-string component definitions) is not a per-version table and already mentions Release 6 in prose ("FHIR Release 6 (Normative) is version 6", line 206).
- The `<table>` blocks at lines 49–117, 142–149, 349–544, 553–556 are about FMM levels, lifecycle states and standards-status categories — they do not enumerate FHIR versions and are out of scope for this ticket.

### Design Rationale

- **Consistency with the package matrix.** The R6 row makes the `[defining_version]` table internally consistent with the package matrix on the same page that already publishes `xver-r6.r2`, `xver-r6.r3`, `xver-r6.r4`, `xver-r6.r4b`, `xver-r6.r5` and the inverse `xver-{r2..r5}.r6` packages. Implementers must be able to map those package names back to the documented `[defining_version]` token set.
- **URL path style.** The R4, R4B, R5 rows all use the lower-case path style (`/r4`, `/r4b`, `/r5`) introduced after the DSTU2/STU3 era. The new R6 row should match the modern style (`/r6`) to align with the canonical URL pattern used elsewhere in the spec and in the existing `xver-r6.*` package URLs already on this page.
- **`(this version)` marker.** The R5 row currently carries `(this version)` because the file was last edited for R5. R6 is the spec being published where this change lands, so the marker moves to the R6 row. Leaving both rows marked as `(this version)` would be incorrect; leaving only R5 marked would mis-state the publishing version.
- **Alternative considered: bulk rewrite.** FHIR-47216's resolution wording ("update all tables on that page to have R6 as well as R5") could be read as a sweeping change. A line-by-line audit of `source/versions.html` (see Proposed Change) shows only this one table is actually missing R6; doing more than that on this ticket would silently expand scope. Any other R6 gap on the page should be handled by FHIR-47216 itself or a follow-up ticket, not folded in here.
