## Feature Proposal

### Problem Statement

The R6 ballot 1 rendering of the top-level Datatypes page exhibits two
defects:

1. **Format is truncated and askew** — visual layout breaks on the
   published page (likely a malformed table row, an unclosed tag, a stray
   liquid-style `[%…%]` macro that did not expand, or a layout/macro that
   widens the page beyond its column).
2. **Links are problematic** — one or more `<a href="…">` targets on the
   page are broken (stale anchors, missing target pages, or wrong
   fragment IDs).

Because Datatypes is a top-of-spec landing page that almost every reader
of the FHIR specification visits, even minor rendering/link defects are
high-visibility and erode confidence in the ballot artifact. The ticket
was filed against the ballot snapshot itself, so the fix has to land in
the authoring source (`source/datatypes.html`) on `master` so the next
publish run produces a clean page.

### Proposed Change

Repair `source/datatypes.html` on `HL7/fhir@master` so the page renders
without truncation and with all links resolving. The change is a
content/markup fix only — no FHIR resource, element, profile, extension,
or terminology is added, removed, or modified. Concretely:

- **Validate page-level HTML well-formedness.** Walk the file end-to-end
  and confirm every block-level element (`<table>`, `<tr>`, `<td>`,
  `<div>`, `<p>`, `<ol>`, `<ul>`, `<li>`) is balanced. Particular
  attention to the per-primitive table that begins around line 82 and to
  the `Primitive Types` / `General-Purpose Datatypes` row of `<div
  class="row">` cards near the top of the page (lines 38–46), since
  truncation symptoms typically originate in unbalanced table rows or
  div containers.
- **Validate every publisher macro expands.** All `[%…%]` tokens (e.g.,
  `[%settitle Datatypes%]`, `[%file newheader%]`, `[%file newnavbar%]`,
  `[%dtheader base%]`, `[%dtref Reference%]`, `[%diagram
  datatypes/allprimitivetypes.diagram 1dt%]`,
  `[%diagram datatypes/alltypes.diagram 2dt%]`,
  `[%diagram datatypes/metadatatypes.diagram 3dt%]`,
  `[%diagram datatypes/specialtypes.diagram 4dt%]`,
  `[%regex base64Binary%]`, `[%wg fhir%]`, `[%wgt fhir%]`,
  `<%extension http://hl7.org/fhir/StructureDefinition/timezone%>`)
  must be syntactically correct so the publisher actually substitutes
  them. Any unbalanced `[%` / `%]` or stray `%` sneaking into the page
  body causes the publisher to emit raw, "askew" output.
- **Validate every link target.** For each `<a href="…">` on the page —
  including same-page anchors (`#primitive`, `#complex`, `#timing`,
  `#base64Binary`, `#boolean`, `#canonical`, `#code`, `#date`,
  `#dateTime`, `#decimal`, …) and cross-page links (`types.html`,
  `resourcelist.html`, `patterns.html`, `metadatatypes.html`,
  `dosage.html`, `elementdefinition.html`, `extensibility.html`,
  `terminologies.html`, `valueset.html`, `references.html`,
  `datatypes-examples.html`, `versions.html`, the per-primitive
  `*.profile.xml.html` / `*.profile.json.html` definition pages) —
  confirm the target file or anchor exists in the publisher source.
  Repair stale targets in place; remove or repoint links whose target
  no longer exists.
- **Re-publish locally and visually inspect** the rendered
  `datatypes.html` to confirm the page is no longer truncated and the
  card row, primitive-types table, and complex-type / metadata sections
  flow correctly.

No element / cardinality / binding / search-parameter / invariant
changes are required.

### Design Rationale

The ticket is a Technical Correction with a Persuasive disposition and a
free-text symptom report only ("truncated and askew", "links are
problematic"). The corresponding authoring source — `source/datatypes.html`
in `HL7/fhir` — is the single file that the bespoke HL7 publisher reads
to produce `datatypes.html` in any published ballot or release. The
playbook's "Update narrative / prose" recipe (per the saved briefing for
`HL7/fhir`) covers exactly this kind of authored-HTML edit: edit the
`.html` file directly under `source/` and re-run the publisher. There is
no alternative authoring path (no FSH, no IG-publisher, no FSH-generated
output) for this page.

We restrict the change to markup repair because:

- The specification's *content* (set of primitive types, value domains,
  XML / JSON representations, regex bindings) is governed by other
  tickets and resources and is out of scope here.
- The reporter's complaint was strictly about rendering and links, not
  about the meaning of any datatype.
- Persuasive without a substantive `resolution_description` indicates
  the WG accepted the report as a publication-quality defect rather than
  a normative change.

### Step-by-Step Tasks

#### HL7/fhir

1. **Reproduce the rendering defect.**
   - File: *(none — investigation step)*
   - Action: Open the published page referenced in the ticket
     (<https://hl7.org/fhir/6.0.0-ballot1/datatypes.html>) and identify
     exactly *where* the page is "truncated and askew" and *which*
     links are broken. Diff the rendered DOM against
     `source/datatypes.html` to localise each defect to a specific
     authoring construct (table row, div card, `[%…%]` macro, or
     `<a href>`).
   - Details: Capture the concrete defect list (e.g., "primitive-types
     table truncates after row N", "card row collapses to one column",
     "`#xyz` anchor has no target") so each defect can be addressed by
     a discrete edit below. Without this list the remaining tasks are
     guesses.

2. **Repair HTML well-formedness in `source/datatypes.html`.**
   - File: `source/datatypes.html`
   - Action: For every defect identified in Task 1, balance the
     offending markup. Typical fixes are: close a `<td>` / `<tr>` /
     `<div>` / `<p>` that was left open; correct an attribute (e.g., a
     stray quote) that broke parsing; remove a stray character that
     prevented a `[%…%]` macro from being recognised by the publisher.
   - Details: The vulnerable areas to inspect first are the introductory
     card row at lines 38–46 (the `<div class="row">` containing the
     `[%diagram …%]` calls for `allprimitivetypes.diagram`,
     `alltypes.diagram`, `metadatatypes.diagram`, and
     `specialtypes.diagram`) and the primitive-types `<table
     class="list">` starting at line 82. Each primitive row is a
     three-`<tr>` group (definition + regex / definition links + size
     note where present); confirm each group's `<tr>`/`<td>` count and
     `style="border-top: 0px silver solid"` continuation cells are
     consistent.

3. **Repair broken links in `source/datatypes.html`.**
   - File: `source/datatypes.html`
   - Action: For each `<a href="…">` flagged in Task 1, either correct
     the target to an existing page/anchor or remove the link if the
     target has been retired. Cross-page targets must exist as
     `source/<target>.html`; same-page anchors must match an `<a
     name="…">` or element `id` in the same file.
   - Details: Common targets to spot-check — `types.html`,
     `resourcelist.html`, `patterns.html`, `metadatatypes.html`,
     `dosage.html`, `elementdefinition.html`, `extensibility.html`,
     `extensibility.html#primitives`, `extensibility.html#modifier`,
     `extensibility.html#list`, `terminologies.html`,
     `terminologies.html#string`, `valueset.html`, `references.html`,
     `references.html#canonical`, `references.html#canonical-list`,
     `datatypes-examples.html#primitives`, `versions.html#std-process`,
     and the per-primitive `*.profile.xml.html` /
     `*.profile.json.html` files. Same-page anchors include
     `#primitive`, `#complex`, `#modifiers`, and the per-type names
     (`#base64Binary`, `#boolean`, `#canonical`, `#code`, `#date`,
     `#dateTime`, `#decimal`, …).

4. **Verify publisher macros still expand.**
   - File: `source/datatypes.html`
   - Action: Search the file for `[%` and `%]` (and `<%` / `%>` for
     extension macros) and confirm each pair is balanced and uses a
     macro name the publisher knows. Pay special attention to
     `[%settitle Datatypes%]`, `[%file newheader%]`, `[%file
     newnavbar%]`, `[%dtheader base%]`, the `[%dtref …%]` set, the
     `[%diagram …%]` set, `[%wg fhir%]`, `[%wgt fhir%]`, `[%regex
     <type>%]`, and `<%extension
     http://hl7.org/fhir/StructureDefinition/timezone%>`.
   - Details: Unbalanced or misspelt macros pass straight through the
     publisher and appear as raw `[%…%]` text in the rendered page, a
     classic cause of "askew" output.

5. **Sweep for inbound references to repaired anchors.**
   - File: *(repo-wide grep, no edit unless a stale link is found)*
   - Action: For any anchor renamed or removed in Task 3, grep
     `source/` for inbound references (e.g., `datatypes.html#<old>`)
     and either repoint or remove them so no neighbour page is left
     with a dangling fragment.
   - Details: This mirrors the briefing's general guidance to sweep
     for inbound `#anchor` references whenever an anchor is touched —
     the same hygiene rule the FML page repair followed.

6. **Re-publish locally and visually verify.**
   - File: *(build artifact, not a source edit)*
   - Action: Run the bespoke HL7 publisher (`publish.sh` /
     `publish.bat`) to regenerate `datatypes.html`, then open the
     rendered page in a browser and confirm:
     (a) the introductory card row renders as a 2×2 grid of named
         diagrams,
     (b) the primitive-types table flows top-to-bottom without
         truncation,
     (c) every `<a href>` in the page resolves (no 404, no dangling
         fragment),
     (d) no raw `[%…%]` or `<%…%>` text leaks into the rendered DOM.
   - Details: Do not commit until the rendered output reproduces the
     intended layout. Do not edit anything under `tools/`, `qa/`,
     `schema/`, or other generated areas during this step.
