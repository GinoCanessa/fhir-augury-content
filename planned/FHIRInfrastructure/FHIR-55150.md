## Resolution Summary

**Resolution:** Not Persuasive with Modification.

The committee declined to add an `unknown`/`other` escape-valve concept to the
`capability-statement-kind` ValueSet, on the rationale that **`requirements`
is a safe fallback** for cases where the authoring system does not have
specific knowledge of an instance, product, or class of implementation.

Instead, two narrative changes are approved on `CapabilityStatement.kind`:

1. **Replace the element `definition`** from:

   > "The way that this statement is intended to be used, to describe an
   > actual running instance of software, a particular product (kind, not
   > instance of software) or a class of implementation (e.g. a desired
   > purchase)."

   to:

   > "The way that this statement is intended to be used, to describe an
   > actual running instance of software, a particular product (kind, not
   > instance of software) or a general statement of implementation
   > expectations (e.g. a desired purchase, a formal specification, etc.)."

2. **Add a new `comment`** on the same element:

   > "Unless knowledge of a specific system or implementation are known, use
   > 'requirements'"

The single comment on the ticket (Lloyd McKenzie, 2026-02-09) records that
the committee's first inclination — relaxing the cardinality from `1..1` to
`0..1` — was rejected because it would violate the cross-version stability
rule "Minimum element cardinalities will not be changed". Grahame Grieve was
asked to start a Zulip discussion about that meta-rule, but no relaxation has
been adopted, so the cardinality of `CapabilityStatement.kind` remains
`1..1`.

## Feature Proposal

### Problem Statement

`CapabilityStatement.kind` is mandatory (`1..1`) with a required binding to
`http://hl7.org/fhir/ValueSet/capability-statement-kind`, whose only codes are
`instance`, `capability`, and `requirements`. The reporter argued that none of
these reads as an "unknown / other" escape valve, leaving authors with no way
to populate the element when the CapabilityStatement does not cleanly fit any
of the three categories. The original ticket asked to add an `unknown`/`other`
concept to the value set.

### Proposed Change

Per the resolution, the value set is **not** changed and the cardinality is
**not** relaxed. The change is purely to the `ElementDefinition` for
`CapabilityStatement.kind` inside the core CapabilityStatement
StructureDefinition:

- **File:** `source/capabilitystatement/structuredefinition-CapabilityStatement.xml`
- **Element:** `CapabilityStatement.kind` (currently at lines 488–511 of the
  `<differential>` block; differential spans lines 81–1572).
- **Edit 1 — replace `<definition>` value (line 491):**

  ```xml
  <definition value="The way that this statement is intended to be used, to describe an actual running instance of software, a particular product (kind, not instance of software) or a general statement of implementation expectations (e.g. a desired purchase, a formal specification, etc.)."/>
  ```

  (Diff vs. current: the trailing clause "or a class of implementation (e.g. a
  desired purchase)." becomes "or a general statement of implementation
  expectations (e.g. a desired purchase, a formal specification, etc.).")

- **Edit 2 — insert a new `<comment>` element immediately after the
  `<definition>` and before `<requirements>` (between current lines 491 and
  492), respecting `ElementDefinition`'s child-order
  (`short` → `definition` → `comment` → `requirements`):**

  ```xml
  <comment value="Unless knowledge of a specific system or implementation are known, use 'requirements'"/>
  ```

No other elements, bindings, invariants (`cpb-3`, `cpb-14`, `cpb-15`,
`cpb-16`), value sets, code systems, or examples are altered. The
`capability-statement-kind` CodeSystem and ValueSet under
`source/capabilitystatement/codesystem-capability-statement-kind.xml` and
`source/capabilitystatement/valueset-capability-statement-kind.xml` are
deliberately left untouched.

### Design Rationale

- The committee judged that `requirements` already provides a semantically
  correct fallback when the author is describing an aspirational or
  not-yet-instantiated capability — including formal specifications, RFPs,
  and the originally-cited "desired purchase" use case. Adding `unknown`/
  `other` would introduce ambiguity (two ways to represent the same intent)
  without enabling any new use case.
- A `<comment>` on the element is the standard FHIR mechanism for steering
  implementers toward the right code when a required binding has subtle
  selection rules; this is consistent with the pattern used elsewhere in the
  spec where one code in a small enumeration is the conservative default.
- Broadening the `<definition>` from "a class of implementation (e.g. a
  desired purchase)" to "a general statement of implementation expectations
  (e.g. a desired purchase, a formal specification, etc.)" makes it textually
  obvious that the existing `requirements` code already covers cases such as
  a formal specification or RFP — the use cases the reporter was trying to
  solve with an escape valve.
- The alternative of changing `min` from `1` to `0` was explicitly considered
  and rejected in the only ticket comment because it would violate the
  cross-version stability rule "Minimum element cardinalities will not be
  changed". A separate Zulip discussion on relaxing that meta-rule was
  flagged but is out of scope here.
