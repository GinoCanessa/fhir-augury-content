## Resolution Summary

**Resolution:** Persuasive with Modification
(vote Jose Costa Teixeira / Richard Ettema: 13-0-0; Impact: Non-substantive;
Category: Enhancement).

Exact resolution text: *"Will define a code system as per the attached PDF
(and google sheet in the comments) in THO and a preferred binding."*

Concretely, this requires two coordinated pieces of work:

1. **In THO / UTG** — create a new CodeSystem (and accompanying ValueSet) of
   requirement-category codes drawn from the attached PDF
   (`Candidate Requirements Categories - Sheet1.pdf`) and the linked Google
   sheet. That work is tracked separately under **UP-783**.
2. **In FHIR Core** — change the binding on
   `Requirements.statement.category` from `example` (with no
   `<valueSet>`) to **`preferred`**, pointing at the ValueSet created in THO.

The originally proposed code set (ticket description) was:
`functional`, `performance`, `security`, `usability`, `scalability`,
`accessibility`. The applied resolution modified that set to adopt the
expanded list in the attached PDF / Google sheet (the "with Modification"
part of the disposition). The authoritative codes are defined by the
downstream THO CodeSystem produced under UP-783; this core-spec change
only references the resulting ValueSet.

## Feature Proposal

### Problem Statement

`Requirements.statement.category` is a `0..*` `Coding` element in the
Requirements resource (R5). Today (clone `db79dcd`,
`source/requirements/structuredefinition-Requirements.xml` line 643) its
`<binding>` is:

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="Category"/>
  </extension>
  <strength value="example"/>
  <description value="e.g., functional, safety, performance, accessibility"/>
</binding>
```

The binding declares strength `example` but supplies **no `<valueSet>`
reference**. Implementers therefore have only the prose hint in
`<description>` to work from, with no machine-processable set of
candidate codes. That makes it difficult to author consistent
Requirements instances, reason about category axes across Requirements
resources, or build UIs that help users pick a category. The ticket and
the Zulip thread
(`#fhir/infrastructure-wg > Category codes for requirements`) establish
workgroup consensus that an explicit, published list is needed.

### Proposed Change

**A. Terminology (THO / UTG — tracked under UP-783, *prerequisite* for
this ticket):**

- New `CodeSystem` published on THO at
  `http://terminology.hl7.org/CodeSystem/requirement-category`, populated
  from the "Candidate Requirements Categories" PDF / Google sheet. The
  initial starter set named in the ticket description was `functional`,
  `performance`, `security`, `usability`, `scalability`,
  `accessibility`; the final code list is whatever UP-783 lands (the
  resolution explicitly defers to the attached PDF + sheet).
- Matching `ValueSet` on THO at
  `http://terminology.hl7.org/ValueSet/requirement-category`
  (`compose.include.system =
  http://terminology.hl7.org/CodeSystem/requirement-category`, all codes).

Note the system URI differs from the ticket reporter's original
suggestion of `http://hl7.org/fhir/requirement-category`: because the
resolution places the artifact in **THO** rather than in the FHIR core
spec, the canonical lives under `terminology.hl7.org`, consistent with
other THO-hosted category code systems.

**B. FHIR Core (this ticket, HL7/fhir):**

Update the binding on `Requirements.statement.category` in
`source/requirements/structuredefinition-Requirements.xml` to:

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="RequirementCategory"/>
  </extension>
  <strength value="preferred"/>
  <valueSet value="http://terminology.hl7.org/ValueSet/requirement-category"/>
  <description value="High-level categorization of a Requirements statement (e.g., functional, performance, security, usability, scalability, accessibility)."/>
</binding>
```

No changes to cardinality (`0..*`), type (`Coding`), `isSummary`, short,
definition, or mapping — only the binding strength, the new
`<valueSet>`, the `bindingName` (upgraded from generic "Category" to
"RequirementCategory" since the binding is now a named,
externally-defined VS), and a tightened `<description>` that mirrors the
ValueSet scope.

### Design Rationale

- **Preferred, not required or extensible.** The resolution explicitly
  says *"a preferred binding"*. `preferred` communicates "this is the
  recommended value set, but implementers may choose other codes with
  justification" — appropriate for a `0..*` categorization element
  whose axes (functional vs. performance vs. security, etc.) are
  acknowledged in the element definition to be "not mutually exclusive"
  and not exhaustive.
- **Terminology lives in THO, not core.** Consistent with HL7's
  post-R4 pattern of hosting code systems that aren't FHIR-specific (or
  that may evolve independently of the core release cycle) in THO.
  Multiple existing Requirements bindings already reference THO URIs
  (e.g., `http://terminology.hl7.org/ValueSet/jurisdiction` on
  `Requirements.jurisdiction`, line 400), so this is idiomatic for the
  resource.
- **Keeps core change non-substantive.** The vote record tags the
  change *Impact: Non-substantive; Category: Enhancement*. The core
  edit is therefore a binding tweak only; no new element, no cardinality
  or type change, no new invariants.
- **Alternatives considered** (from comments / Zulip):
  - *Using the reporter's proposed `http://hl7.org/fhir/requirement-category`
    in core.* Rejected by the WG in favor of THO (resolution wording).
  - *Adopting the Da Vinci CRD `CodeSystem-temp` set directly
    (referenced by mckenzie 2026-01-27).* Used as input/inspiration
    rather than adopted wholesale — the final set is the PDF / Google
    sheet (expanded beyond CRD's set).
  - *Staying at `example` strength with just a `<valueSet>` filled in.*
    Rejected — the ticket's whole point is that implementers need a
    recommended set, which is what `preferred` signals.

### Step-by-Step Tasks

#### HL7/UTG (via UP-783 — listed here only for sequencing)

1. **Author `CodeSystem-requirement-category`.**
   - Action: Create the CodeSystem resource on THO with canonical
     `http://terminology.hl7.org/CodeSystem/requirement-category`,
     status `active`, experimental `false`, content `complete`,
     populated from the Candidate Requirements Categories PDF / Google
     sheet.
   - Details: Specific UTG file paths and layout are out of scope for

2. **Author `ValueSet-requirement-category`.**
   - Action: Create the ValueSet with canonical
     `http://terminology.hl7.org/ValueSet/requirement-category`,
     `compose.include` of the CodeSystem above (all codes).
   - Details: Same as above.

#### HL7/fhir (FhirCore)

3. **Update the `Requirements.statement.category` binding to
   `preferred` and point at the new THO ValueSet.**
   - File: `source/requirements/structuredefinition-Requirements.xml`
   - Action: In the `<element id="Requirements.statement.category">`
     block (currently starting at line 643 in clone `db79dcd`), replace
     the existing `<binding>` (lines ~659–669) with:
     ```xml
     <binding>
       <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
         <valueString value="RequirementCategory"/>
       </extension>
       <strength value="preferred"/>
       <valueSet value="http://terminology.hl7.org/ValueSet/requirement-category"/>
       <description value="High-level categorization of a Requirements statement (e.g., functional, performance, security, usability, scalability, accessibility)."/>
     </binding>
     ```
   - Details: Edit only the `<differential>` view — do not touch any
     `<snapshot>` block (the publisher regenerates it). Keep the
     element's `<short>`, `<definition>`, `<min>`, `<max>`, `<type>`,
     `<isSummary>`, and `<mapping>` unchanged.

4. **Verify no downstream content references
   `Requirements.statement.category` with an incompatible assumption.**
   - File: `source/requirements/` (all files), plus repo-wide grep for
     `Requirements.statement.category` and for the literal
     `requirement-category`.
   - Action: Repo-wide search to confirm no existing prose, examples,
     profiles, or Quick-Reference tables hard-code an alternative
     CodeSystem/ValueSet for this element. In the two example
     instances (`Requirements-example1.xml`, `Requirements-example2.xml`),
     confirm none currently populates `.statement.category` with a
     code that would now fall outside the preferred set in a way that
     would be misleading in the published spec.
   - Details: No edits expected; this is a sanity sweep.

5. **(Optional but recommended) Add a short example category coding in
   one Requirements example.**
   - File: `source/requirements/Requirements-example1.xml` **or**
     `source/requirements/Requirements-example2.xml`
   - Action: Add a `<statement><category>` entry using one of the new
     codes from the THO CodeSystem (e.g., `functional`) so the published
     resource page demonstrates the new binding.
   - Details: Only do this if the WG confirms; the resolution does not
     mandate example updates. Leave out if there is any doubt.

### Testing Considerations

- **Revalidate** the Requirements StructureDefinition with the FHIR

  `preferred` ValueSet resolves.
- **Revalidate** `Requirements-example1.xml` and `Requirements-example2.xml`.
- **Publish preview** — spot-check the rendered Requirements resource
  page (`requirements.html`) to confirm the element's binding row shows

- **Downstream IG smoke check** — any IG that profiles Requirements
  (e.g., CRD, CQF) should be rebuilt on a snapshot that contains this
  binding to confirm they do not break (they should not, since
  `preferred` is permissive).
- **Edge case** — an instance that uses a code from outside the new
  ValueSet should still validate (with an informational/warning
  message, not an error). Confirm the tooling behaves accordingly.
