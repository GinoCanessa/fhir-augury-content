## Feature Proposal

### Problem Statement

The published FHIR Mapping Language (FML) specification page (`source/mapping-language.html` in `HL7/fhir`) contains three editorial `TODO` markers that have shipped to readers of the spec for multiple ballot cycles. They appear in normative-adjacent prose:

- The `list_modes` bullet defines the syntactic position of `list_modes` but explicitly defers the semantics ("TODO: What do these do?").
- The trailing paragraph after the transform-functions table defers description of how optional parameters and list mode behave with transforms.
- An entire `Mapping Support API` section consists solely of the word "todo", which is misleading and implies forthcoming normative content where none is being authored in core FHIR.

These omissions are a long-standing readability and conformance-clarity gap. Implementers have no authoritative description of `list_mode` semantics in core, despite `list_mode` being part of the rule grammar (see `source/mapping.g4`).

### Proposed Change

A pure documentation edit to a single specification page, `source/mapping-language.html`. No StructureDefinition, ValueSet, CodeSystem, or example file is touched.

**Edit 1 — Replace `list_modes` TODO with normative prose (around line 483).**

Replace the existing list item:

```html
<li><b>list_modes</b>: control how elements that repeat are managed when the transform rule is evaluated. Values can be 'first', 'share', 'last' and 'single'. TODO: What do these do?</li>
```

with prose along the lines of:

```html
<li><b>list_modes</b>: control how elements that repeat are managed when the transform rule is evaluated. Each value applies independently to the source side of the rule:
  <ul>
    <li><code>first</code> — only the first item of the repeating source element is processed by this rule; subsequent items are ignored for this rule.</li>
    <li><code>last</code> — only the last item of the repeating source element is processed by this rule; preceding items are ignored for this rule.</li>
    <li><code>share</code> — the rule is applied to every item, but the variable produced is shared across iterations so a single target value is built up rather than one per source item.</li>
    <li><code>single</code> — the rule asserts that the source element has exactly one item; the engine raises an error if zero or more than one items are present.</li>
  </ul>
  Multiple values may be combined where they do not conflict; conflicting combinations (for example <code>first</code> with <code>last</code>) are an error.
</li>
```

The exact wording must be reviewed by Grahame Grieve / FHIR-I before merge (see Open Questions); the structure above is the editorial shape proposed by this plan.

**Edit 2 — Replace the post-transform-table TODO paragraph (around line 656).**

Replace:

```html
<p>
TODO: explain how optional parameters work with transforms (append only?), document list mode
</p>
```

with two short paragraphs covering:

1. **Optional parameters in transforms.** When a transform function declares an optional parameter (typeset as `(...)` in the parameters column of the table above), omitting that parameter at the call site is permitted. For `append`-style transforms, omission means "append the supplied values only", with no separator/prefix. For other transforms (e.g., `translate`, `cp`), the per-transform behaviour is given in the Documentation column; default behaviour applies when the parameter is not supplied.
2. **List modes in transform rules.** The `list_mode` qualifier on a rule (see the bullet in the target-statement description above) interacts with transforms as follows: `first`/`last`/`single` constrain which source element is offered to the transform; `share` causes the transform's output variable to be reused across iterations rather than re-created per source item.

Again, exact wording is for FHIR-I editorial sign-off.

**Edit 3 — Remove the empty `Mapping Support API` section (around line 918).**

Delete the four contiguous lines:

```html
<a name="api"></a>
<h3>Mapping Support API</h3>

<p>todo</p>
```

so that the `Formal Grammar` section is followed directly by `Reserved keywords`. Also remove or fix any inbound references to the `#api` fragment of `mapping-language.html` from elsewhere in `HL7/fhir` and (if any are found) note them on the PR for downstream IGs.

### Design Rationale

- **Single-page, single-PR change.** The resolution scopes the work to one page; nothing in the StructureDefinitions, terminology, or examples is implicated. Keeping the change to `source/mapping-language.html` matches the Update narrative / prose recipe in the FhirCore playbook and minimises blast radius.
- **Prose, not grammar.** The grammar (`mapping.g4`) already enumerates `list_mode` tokens; the gap is purely descriptive. No grammar or invariant change is required, so the change is non-breaking.
- **Removing the API section rather than authoring it in core.** Per the resolution and the ticket's reporter/assignee discussion, the Mapping Support API content is being relocated to an Incubator project. Leaving an empty `<p>todo</p>` ships an empty conformance section to balloters; deleting it is the published preference.
- **Out-of-scope items are deliberately deferred.** The `cast` `source, type?` question mark and the `dateOp` transform are tracked outside this ticket per Corey Spears's 2026-01-29 comment. Folding them in here would broaden the change beyond the resolution.
