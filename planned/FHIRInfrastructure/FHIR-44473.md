## Feature Proposal

### Problem Statement

In running prose on the ExampleScenario resource pages, the resource name is
written inconsistently. Within `source/examplescenario/examplescenario-notes.xml`
the very first sentence opens with *"Example Scenarios help to make..."* —
treating the name as an English noun phrase with a space. The next paragraph in
the same file (and the rest of the ExampleScenario documentation set) uses the
canonical, single-token form *"ExampleScenarios"*. The inconsistency is
exactly the kind of editorial defect Ron G. Parker filed in the FHIR-444xx
spelling/typo sweep (see Related Tickets — FHIR-44486, FHIR-44480, FHIR-44488,
FHIR-48196, FHIR-48207, FHIR-42297, etc.). FHIR resource names are proper
nouns in FHIR documentation conventions and must be written as one CamelCase
token whenever they refer to the resource.

### Proposed Change

Edit `source/examplescenario/examplescenario-notes.xml` to replace the
space-separated form *"Example Scenarios"* with the correct CamelCase
*"ExampleScenarios"* in line 5:

- **Before** (line 5):
  > Example Scenarios help to make the expected behavior of a collection of
  > systems adhering to a guide more concrete.
- **After** (line 5):
  > ExampleScenarios help to make the expected behavior of a collection of
  > systems adhering to a guide more concrete.

This is a one-token, prose-only correction. No other edits are required in the
file because every other occurrence of the resource name in
`examplescenario-notes.xml` already uses the correct CamelCase form
(e.g. line 8 *"creating ExampleScenarios is often an order of magnitude…"*,
lines 11, 14, 17, 20).

The following spaced occurrences elsewhere in the cache **must not be
touched** by this ticket because they are intentional FHIR conventions — they
are the human-readable `title` of canonical resources / valuesets and the
`name`/`title` values in invariant test fixtures:

- `source/examplescenario/structuredefinition-ExampleScenario.xml` —
  `<title value="Example Scenario"/>` (canonical-resource human title, by design).
- `source/examplescenario/valueset-examplescenario-instance-type.xml`,
  `valueset-examplescenario-operation-codes.xml`,
  `valueset-examplescenario-resource-type.xml` —
  `<title value="Example Scenario …"/>` (intentional human titles).
- `source/examplescenario/invariant-tests/*.xml` —
  `<name value="Example Scenario"/>` in invariant test fixtures (test-only data).

### Design Rationale

- **Consistency within the same page.** The second sentence of the same
  paragraph already uses *"ExampleScenarios"*; making line 5 match removes the
  internal inconsistency.
- **FHIR house style.** Resource names in narrative prose are written as a
  single CamelCase token (`ExampleScenario`, `ActorDefinition`,
  `PlanDefinition`). The same notes file consistently treats other resource
  names this way (e.g. *PlanDefinitions* / *ActorDefinition* in the
  introduction file).
- **Minimum-impact edit.** A single-word substitution in a hand-authored notes
  file is the smallest change that satisfies the resolution and triggers no
  derived-artifact regeneration concerns, matching the FhirCore *"Update
  narrative / prose"* recipe.
- **Scope discipline.** The `<title value="Example Scenario"/>` on the
  StructureDefinition and ValueSets is the FHIR-defined human-readable
  display, not a misspelling, so it is explicitly out of scope here. Sweeping
  it in would be a behavioural change that the reporter did not request and
  would conflict with established FHIR canonical-resource conventions.
