## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** "Also look at creating regression tests on the regexes."

In context, the resolution accepts the report that the R5 `dateTime` regex incorrectly accepts a value with the `T<time>` portion present but no timezone offset (e.g., `2024-07-23T12:27:16.321661`). The fix originally applied between DSTU1 and DSTU2 under FHIR-8889 must be re-applied to the R5 regex so that, when hours/minutes/seconds are present, a timezone (`Z` or `┬▒hh:mm` / `14:00`) is required. The resolver additionally calls for regression tests on the primitive regexes so this class of regression can be detected automatically.

## Feature Proposal

### Problem Statement

The narrative for the `dateTime` primitive type in `source/datatypes.html` (lines 175ÔÇô186) is unambiguous:

> "If hours and minutes are specified, a timezone offset SHALL be populated."

and `source/datatypes/primitives.xml` line 194 (the authoring spreadsheet that drives the published StructureDefinition for `dateTime`) repeats:

> "If hours and minutes are specified, a UTC offset SHALL be populated."

However, the regex authored on line 198 of `source/datatypes/primitives.xml` does **not** enforce that constraint. The current value is:

```regex
([0-9]([0-9]([0-9][1-9]|[1-9]0)|[1-9]00)|[1-9]000)(-(0[1-9]|1[0-2])(-(0[1-9]|[1-2][0-9]|3[0-1])(T([01][0-9]|2[0-3]):[0-5][0-9]:([0-5][0-9]|60)(\.[0-9]{1,9})?(Z|(\+|-)((0[0-9]|1[0-3]):[0-5][0-9]|14:00)?)?)?)?)?
```

There are two stray `?` quantifiers inside the timezone alternation. Reading from the inside outward:

1. `((0[0-9]|1[0-3]):[0-5][0-9]|14:00)?` ÔÇö the offset value (`hh:mm` or `14:00`) is marked optional, so a literal `+` or `-` with no digits would also match.
2. `(Z|(\+|-)ÔÇŽ)?` ÔÇö the entire timezone group is optional even when the `T<time>` block is present.

The combined effect is that `2024-07-23T12:27:16.321661` (T-block present, no timezone) matches, in violation of both the narrative SHALL and the historical DSTU2 fix from FHIR-8889. Because this regex is propagated to the published `dateTime` StructureDefinition, the FHIR JSON Schema, and downstream language-binding regexes, every R5 validator that relies on the spec-published regex inherits the bug.

For comparison, the `instant` regex on line 155 of the same file is correctly written as a single non-optional timezone group:

```regex
ÔÇŽ(\.[0-9]{1,9})?(Z|(\+|-)((0[0-9]|1[0-3]):[0-5][0-9]|14:00))
```

### Proposed Change

**File:** `source/datatypes/primitives.xml`, line 198 (the `dateTime` row's `regex` cell).

**Action:** Remove the two stray `?` quantifiers inside the timezone subexpression so that the timezone is mandatory whenever the `T<time>` portion is present. The fully bracketed structure is preserved (date, month, day, time-block remain optional), only the *timezone* loses its optionality.

**Replace** the current cell value with:

```regex
([0-9]([0-9]([0-9][1-9]|[1-9]0)|[1-9]00)|[1-9]000)(-(0[1-9]|1[0-2])(-(0[1-9]|[1-2][0-9]|3[0-1])(T([01][0-9]|2[0-3]):[0-5][0-9]:([0-5][0-9]|60)(\.[0-9]{1,9})?(Z|(\+|-)((0[0-9]|1[0-3]):[0-5][0-9]|14:00)))?)?)?
```

Diff against the current text ÔÇö the only change is the removal of the two `?` after `14:00)` and after `((0[0-9]|1[0-3]):[0-5][0-9]|14:00)`:

```
ÔÇŽ(\.[0-9]{1,9})?(Z|(\+|-)((0[0-9]|1[0-3]):[0-5][0-9]|14:00)?)?)?)?)?
ÔÇŽ(\.[0-9]{1,9})?(Z|(\+|-)((0[0-9]|1[0-3]):[0-5][0-9]|14:00)))?)?)?
                                                          ^ removed ?  ^^ removed ?
```

**Behavioural effect** of the corrected regex:

- `2024` Ôťô (year only)
- `2024-07` Ôťô (gYearMonth)
- `2024-07-23` Ôťô (date)
- `2024-07-23T12:27:16Z` Ôťô (datetime with Z)
- `2024-07-23T12:27:16.321661+02:00` Ôťô (datetime with offset)
- `2024-07-23T12:27:16.321661-14:00` Ôťô (the `14:00` boundary case)
- `2024-07-23T12:27:16.321661` ÔťŚ (T-block present, no tz ÔÇö **now correctly rejected**)
- `2024-07-23T12:27:16.321661+` ÔťŚ (sign with no digits ÔÇö **now correctly rejected**)

**Regression tests** (per the resolution's explicit ask): introduce a regex-regression test fixture under `tests/` covering each FHIR primitive whose `value` is regex-constrained (`date`, `dateTime`, `time`, `instant`, `oid`, `uuid`, `id`, `code`, `decimal`, `integer`, `unsignedInt`, `positiveInt`, `base64Binary`, `uri`, `url`, `canonical`, `markdown`, `xhtml` where applicable) with positive and negative cases. The clone currently has no such fixture (`tests/resources/` only contains `liquid-tests.json` and `unicode.xml`); the format and harness for adding one is a coordination point with FHIR Infrastructure / publisher maintainers ÔÇö see Open Questions.

### Design Rationale

- **Restores the DSTU2 fix.** FHIR-8889 (referenced by the reporter) established the rule that the timezone must be present when a time portion is given. The resolution accepts the report as a regression and the proposed regex change re-applies that rule structurally rather than via narrative-only enforcement.
- **Aligns with the existing narrative.** Both `source/datatypes.html` (line 177) and `source/datatypes/primitives.xml` (line 194) already state the SHALL; the regex is the single source of machine-checkable truth and must agree with it.
- **Mirrors the `instant` regex.** `instant` (line 155) already has a non-optional terminal timezone group; using the same shape for the `dateTime` time-block keeps the two regexes consistent and self-documenting.
- **No alternative considered viable.** Solving the regression with an extra invariant on `dateTime` would require every implementer to evaluate FHIRPath in addition to the regex; the regex itself is the standard validation surface for primitives, and patching it is the smallest, most universally-effective change.
- **Regression test coverage** is mandated by the resolver's "Also look at creating regression tests on the regexes" ÔÇö without an automated test harness, the same class of off-by-one regex regression can recur silently in any future ballot cycle.

### Breaking Changes

**Yes ÔÇö this is a breaking change for any data that has been authored against the (incorrect) R5 regex and contains dateTimes with a `T<time>` portion but no timezone offset.**

- Instances such as `2024-07-23T12:27:16.321661` were silently valid against the R5 regex; they will now be rejected.
- Producers that emitted timezone-less dateTimes (because validators based on the published R5 regex did not flag them) must add a timezone ÔÇö `Z` if UTC was intended, or the appropriate `┬▒hh:mm` offset.
- The breakage is **intentional and corrective**: the narrative SHALL has always required the timezone, so any instance lacking it was non-conformant by spec, even though tooling failed to detect it. The change brings tooling back in line with the spec.
- No StructureDefinition differential changes; no element added/removed; no cardinality, type, or binding changes.

- **Edge cases:**
  - The `14:00` literal offset (the IDL boundary) must continue to match.
  - `+24:00` and `-24:00` must remain rejected.
  - The leap-second case `ÔÇŽ:60` (already supported) must remain valid with a timezone.

  - Pure-date and gYear/gYearMonth forms (no `T<time>` block) must continue to be valid without any timezone ÔÇö the timezone requirement applies only when the time-block is present, which is exactly what removing the trailing `?`s on the timezone group accomplishes.
