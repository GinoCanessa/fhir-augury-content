## Resolution Summary

Resolution: **Persuasive with Modification**.

> "Will remove the entire 'Open Issues' section, which includes the note to implementers."

Concretely, this requires deleting the `<h3>Open Issues</h3>` heading together with the `[%stu-note issues%] … [%end-note%]` implementer-note block that follows it on the Workflow specification page. No replacement prose is required.

## Feature Proposal

### Problem Statement

The Workflow page (`source/workflow.html`) still carries an "Open Issues" chapter containing questions that were intended to be resolved prior to the workflow content becoming normative. Although the page itself is informative, the questions it poses concern content that will be normative, so leaving these unresolved stakeholder questions in the published specification is inappropriate for the normative ballot track. The reporter notes that the questions either have been clarified or will be clarified by other means, and therefore should no longer appear on the page.

### Proposed Change

Remove the entire "Open Issues" block from `source/workflow.html`. In the current clone (HEAD `db79dcd`) this is lines **311–329**:

```html
        <h3>Open Issues</h3>
        [%stu-note issues%]
                <b>Workflow Issues</b>
            </p>
            <ul>
                <li>It is possible to replace some portions of the <a href="messageheader.html">MessageHeader</a> ...</li>
                <li>The <a href="operationdefinition.html">OperationDefinition</a> resource could be used ...</li>
                <li>The <a href="https://build.fhir.org/ig/HL7/oo-incubator/StructureDefinition-SupplyRequest.html">SupplyRequest</a>, <a href="devicerequest.html">DeviceRequest</a> and <a href="visionprescription.html">VisionPrescription</a> ...</li>
                <li>
                    SpecimenDefinition is defining a 'thing', not an action, so it is not appropriate for the Definitions group.
                </li>
            </ul>
            <p>
                Feedback is welcome <a href="http://hl7.org/fhir-issues">here</a>.
        [%end-note%]
```

The deletion includes:

- the `<h3>Open Issues</h3>` heading,
- the `[%stu-note issues%]` publisher macro and its closing `[%end-note%]`,
- the four `<li>` questions (MessageHeader/Task overlap, OperationDefinition for task types, SupplyRequest/DeviceRequest/VisionPrescription overlap, SpecimenDefinition placement), and
- the "Feedback is welcome" sentence inside the note.

The `[%file newfooter%]` include on the following line (currently line 331) must remain in place. After the deletion, the "Workflow pattern extensions" `<ul>` (ending around current line 308) should be the last content block before the footer include.

### Design Rationale

This mirrors the resolution text exactly: the committee voted to remove the `Open Issues` content rather than rewrite or relocate it. The questions were either addressed elsewhere or deemed out of scope for normative publication. Leaving them in place would (a) publish unresolved committee discussion inside what is supposed to be stable normative-supporting narrative, and (b) imply a call for public feedback (`hl7.org/fhir-issues`) on matters the work group considers closed. No alternative (e.g., moving the content to a separate informative page) was discussed in the resolution; the ticket reporter and resolver both endorse outright removal.
