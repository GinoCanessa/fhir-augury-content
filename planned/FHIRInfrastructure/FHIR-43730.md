## 1. Problem Statement

The reporter observed that using the `history` bundle type for subscription notification bundles seems impractical because it makes `Bundle.entry.request` mandatory for every entry (due to Bundle invariants `bdl-3` and `bdl-4`). They suggested using the `collection` bundle type as a "lighter" alternative.

## 2. Resolution Summary

**Not Persuasive with Modification** — The request to change from `history` to `collection` was rejected, but with a modification to improve documentation.

Key findings from the resolution:

1. **Historical reason:** The `history` bundle type was required by earlier iterations of the specification (before `SubscriptionStatus` was defined) in order to support `id-only` notifications.
2. **`collection` type limitation:** In FHIR R4, `collection` type bundles require each `Bundle.entry` to contain a resource (`entry.resource` SHALL be present), which would **preclude `id-only` notifications** where only resource URLs are provided without the resource body.
3. **Breaking change concern:** Although the strict technical requirement for `history` no longer exists (since `SubscriptionStatus` now carries the status information), switching to `collection` would be a breaking change.
4. **Community consensus:** The existing implementer community was polled and was not convinced the change was worth the disruption.

**Agreed action:** Add documentation to the IG explaining the rationale for choosing `history` over `collection`.

## 6. Implementation Plan

### Step 1: Expand "Bundle Type Considerations" in `notifications.md`

**File:** `input/pagecontent/notifications.md` (in `HL7/fhir-subscription-backport-ig`)

**Location:** After the existing "### Bundle Type Considerations" heading and its current content about `bdl-3`/`bdl-4`.

**Action:** Add a new paragraph block explaining the rationale. Suggested content:

```markdown
#### Why History Instead of Collection?

The use of the `history` bundle type for notifications is a deliberate design choice.  In FHIR R4, `collection` type bundles have a conformance requirement that each `Bundle.entry` element SHALL contain a resource (see [Bundle: Collection](http://hl7.org/fhir/R4/bundle.html#collection)).  This requirement would preclude the use of [`id-only`](payloads.html#id-only) notifications, where entries contain only resource URLs without the resource body.

Earlier iterations of this specification (prior to the definition of the `SubscriptionStatus` resource) required the `history` type specifically in order to provide `id-only` notifications.  While the introduction of `SubscriptionStatus` has removed the strict technical need for `history`, changing the bundle type would be a breaking change for existing implementations.  When this question was raised with the implementer community ([FHIR-43730](https://jira.hl7.org/browse/FHIR-43730)), the consensus was that the disruption was not warranted.
```
