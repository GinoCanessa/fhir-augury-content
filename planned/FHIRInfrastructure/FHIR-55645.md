## Resolution Summary

**Resolution:** Persuasive (no `resolution_description` populated on the
ticket; no comments or applied-vote text were returned by FhirAugury).

The accepted change is the action stated in the ticket description itself:
**update the FHIR Core "list of link types" so it reflects the current
upstream IANA Link Relation Types registry**, and ideally introduce a
mechanism to keep it in sync automatically.

In FHIR Core, the "list of link types" is the normative
`CodeSystem/iana-link-relations` (URL
`http://hl7.org/fhir/CodeSystem/iana-link-relations`, name
`LinkRelationTypes`), which is republished from the IANA registry at
<https://www.iana.org/assignments/link-relations/link-relations.xhtml#link-relations-1>.
That CodeSystem (and the wrapping `ValueSet/iana-link-relations` it feeds)
is the binding target for `Bundle.link.relation`, so it is what consumers
see when they look up "the list of link types" defined by FHIR.

## Feature Proposal

### Problem Statement

FHIR Core re-publishes the IANA Link Relation Types registry as
`CodeSystem/iana-link-relations` for use by `Bundle.link.relation` (and
indirectly by anyone binding to `ValueSet/iana-link-relations`). The
checked-in copy in `source/bundle/codesystem-iana-link-relations.xml`
currently contains **120 `<concept>` entries** at version `6.0.0` and is
declared `content="complete"`, `caseSensitive="true"`, normative.

The IANA registry has grown since this snapshot was taken (entries such
as `linkset`, `geofeed`, `manifest`, `compression-dictionary`,
`status` and a number of others have been added or had their
descriptions updated upstream), so the FHIR copy materially
under-represents what is now a valid `Bundle.link.relation` value. Lloyd
McKenzie filed FHIR-55645 тАФ *"List of link types isn't up-to-date"* тАФ
asking us to "Update from source (or figure out a way to auto-update)".
Because the CodeSystem is declared `complete`, the gap is observable as
a validation/expansion correctness problem, not just a documentation
gap: validators that consult this CodeSystem will reject codes that are
in fact registered with IANA.

### Proposed Change

Two coordinated pieces of work, both in **`HL7/fhir`** under
`source/bundle/`:

1. **Refresh the codes** in
   `source/bundle/codesystem-iana-link-relations.xml`:
   - For every entry currently listed at
     <https://www.iana.org/assignments/link-relations/link-relations.xhtml#link-relations-1>,
     ensure a `<concept>` with:
     - `<code value="<lower-case relation name>"/>` matching IANA's
       "Relation Name" column,
     - `<display value="<short prose>"/>` derived from IANA's
       "Description",
     - `<definition value="<same prose>"/>` (the existing file
       intentionally repeats display in definition; preserve that
       convention),
     - no `<deprecated>`/`<status>` extensions added unless IANA marks
       the relation as obsolete (none of the existing 120 entries carry
       such markers, so default behaviour is "no extension").
   - Add the missing relations (e.g., `linkset`, `geofeed`,
     `compression-dictionary`, `manifest`, `status`, etc., per the
     current IANA snapshot at the time of update тАФ the authoritative
     list is the registry, not this plan).
   - Update `<display>`/`<definition>` text for any existing relation
     whose IANA description has been revised. Preserve XML escaping
     (`&#xA;` line breaks) used in the current file so diffs stay clean.
   - Do **not** remove any relation that is still present in IANA, and
     do **not** silently delete a relation that has been retired by
     IANA тАФ flag retirements for WG review rather than dropping codes
     from a normative `complete` CodeSystem (see Open Questions).
   - Bump `<version value="6.0.0"/>` to the in-flight publication
     version per the publisher's normal version-bump rules; do not
     hand-edit the version unless that is what the publisher's release
     workflow expects for normative content.

2. **Auto-update mechanism (the "or figure out a way to auto-update"
   half of the resolution).** Add a small generator under `tools/` (or
   wherever the publisher already keeps content scrapers тАФ see Open
   Questions) that:
   - Fetches the IANA registry XML at
     <https://www.iana.org/assignments/link-relations/link-relations.xml>
     (the canonical machine-readable form),
   - Renders it into the FHIR XML shape used by
     `codesystem-iana-link-relations.xml`,
   - Writes the result back over the source file (idempotent тАФ running
     twice produces no diff if IANA has not changed),
   - Is invocable from a CI step or a `publish.sh`/`build.sh` hook so
     the file can be refreshed in a controlled, reviewable way (PR
     diff, not silent regeneration during publish).

The wrapping `ValueSet/iana-link-relations` (`source/bundle/valueset-iana-link-relations.xml`)
is a pure `<include><system .../></include>` over the CodeSystem and
needs **no edits** beyond a possible version bump consistent with the
CodeSystem.

`Bundle.link.relation` itself in `source/bundle/structuredefinition-Bundle.xml`
already binds to `http://hl7.org/fhir/ValueSet/iana-link-relations`
(line 343) тАФ that binding does not change.

### Design Rationale

- IANA owns the registry; FHIR Core re-publishes it for binding
  purposes. Keeping the FHIR copy in step with IANA is consistent with
  the existing publisher header on the file
  (`<publisher value="Internet Assigned Numbers Authority (IANA)"/>`)
  and the explicit `<description>` pointing at the IANA URL.
- A generator in `tools/` (PR-diff style) is preferred over a fully
  silent build-time fetch because (a) the CodeSystem is normative тАФ
  `<standards-status value="normative"/>` тАФ so changes need to be
  reviewable, and (b) the publisher must remain reproducible offline,
  per the FhirCore briefing's note that the build is bespoke and
  publisher-driven from `source/fhir.ini`.
- The alternative "just bump it once and forget" was implicitly
  rejected by the reporter ("or figure out a way to auto-update"); a
  one-shot refresh would put us back in this same position the next
  time IANA adds a relation.

### Breaking Changes

- **No breaking changes** in the strict FHIR sense: existing codes
  remain valid, the binding strength on `Bundle.link.relation` does
  not change, and the CodeSystem URL/name/version semantics stay the
  same.
- Behaviourally, validators using the previous snapshot will start
  accepting additional codes (those newly added by IANA). This is a
  *relaxation* тАФ the previous behaviour rejected codes that are in
  fact registered relations тАФ and is exactly the bug being fixed. It
  is not a backward-incompatible change.
- A `<version>` bump on a normative resource should still be reviewed
  by FHIR Infrastructure to confirm the publication-version policy is
  satisfied.

### Step-by-Step Tasks

#### HL7/fhir

1. **Snapshot the current IANA registry.**
   - File: *(working copy, not committed)*
   - Action: Fetch
     <https://www.iana.org/assignments/link-relations/link-relations.xml>
     and save locally.
   - Details: This is the authoritative machine-readable form of the
     registry; the HTML page is for humans. Use it as the source of
     truth for both this one-shot refresh and the generator written
     in task 3.

2. **Refresh `source/bundle/codesystem-iana-link-relations.xml`.**
   - File: `source/bundle/codesystem-iana-link-relations.xml`
   - Action: Replace the `<concept>` block list so the union of codes
     matches the IANA snapshot from task 1. For every IANA relation,
     emit a `<concept>` with `<code>`, `<display>`, and `<definition>`
     populated as described in *Proposed Change*. Update the
     `<version>` element per publisher policy.
   - Details: Preserve the existing surrounding metadata exactly
     (`<id>`, `<url>`, `<name value="LinkRelationTypes"/>`,
     `<title>`, `<status value="active"/>`,
     `<experimental value="false"/>`, the IANA `<publisher>`/`<contact>`
     block, `<caseSensitive value="true"/>`,
     `<content value="complete"/>`, the `structuredefinition-wg` and
     `structuredefinition-standards-status` extensions). Keep
     alphabetical order of `<concept>` entries and preserve the
     `&#xA;`-style line breaks already used in long definitions to
     keep diffs minimal.

3. **(Optional) bump `<version>` in
   `source/bundle/valueset-iana-link-relations.xml`** to match the
   CodeSystem.
   - File: `source/bundle/valueset-iana-link-relations.xml`
   - Action: Update the `<version>` element if (and only if) the
     CodeSystem version was bumped.
   - Details: No other change. The `<compose>` block stays as a
     single `<include><system value="http://hl7.org/fhir/CodeSystem/iana-link-relations"/></include>`.

4. **Add an IANA тЖТ FHIR CodeSystem generator script.**
   - File: `tools/<new>/update-iana-link-relations.<ext>` (final path
     to be confirmed; see Open Questions)
   - Action: Implement a script that (a) downloads
     `https://www.iana.org/assignments/link-relations/link-relations.xml`,
     (b) parses each `<record>` element (relation name + description),
     (c) renders a FHIR `CodeSystem` XML matching the layout of
     `source/bundle/codesystem-iana-link-relations.xml` (same
     extensions, same metadata, alphabetised `<concept>` list,
     `&#xA;`-escaped line breaks where IANA descriptions span lines),
     and (d) overwrites `source/bundle/codesystem-iana-link-relations.xml`.
   - Details: The script must be idempotent тАФ re-running with no
     upstream change must produce a zero-byte diff. Do **not** wire
     it into `publish.sh`/`build.sh` so that publishing stays
     deterministic and offline; instead run it on demand and commit
     the resulting diff via PR. Add a brief usage note (README or
     header comment) recording the IANA URL, the policy for handling
     IANA retirements (escalate to WG, do not silently delete), and
     the requirement that bumps to the CodeSystem `<version>` go
     through the publisher's normative-version process.

5. **Run the local FHIR publisher / QA over `source/bundle/`.**
   - File: *(no edit)*
   - Action: Run the standard FHIR Core build (`publish.sh` /
     `build.sh` per the briefing) against the changed files and
     confirm no QA regressions on the CodeSystem, the ValueSet
     expansion, or `Bundle`.
   - Details: Pay particular attention to validation messages that
     reference `iana-link-relations` and to any new "unknown code"
     or "duplicate code" errors that would indicate a generator bug.
