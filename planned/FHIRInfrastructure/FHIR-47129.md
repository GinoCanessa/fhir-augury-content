## Resolution Summary

The ticket is resolved **Persuasive** with the resolution description "Do this." (verbatim — no additional prose was supplied beyond the ticket body). The change required is the one set out in the ticket description and confirmed by the precedent fix tracked in [FHIR-28415](https://jira.hl7.org/browse/FHIR-28415):

The English description of the `drt-1` invariant on the **Duration** datatype and its FHIRPath expression do not agree.

- **Description (correct):** *"There SHALL be a code if there is a value and it SHALL be an expression of time. If system is present, it SHALL be UCUM."*
- **Current FHIRPath expression (incorrect):** `code.exists() implies ((system = %ucum) and value.exists())`

The current expression *requires* `value` whenever `code` is present, and *requires* `system = %ucum` whenever `code` is present. Neither matches the description: the description says (a) `code` is required when `value` is present (the inverse direction), and (b) `system` is constrained only when it is **present**. The same defect was repaired in R4B by FHIR-28415; this ticket asks that the equivalent fix be applied to R4, R5, and R6 (the current build).

The fix is to replace the expression with one structured the same way as the `Age` datatype's invariant (which is already correct), minus the non-negative-value clause that does not apply to Duration.

## Feature Proposal

### Problem Statement

The FHIRPath expression for `drt-1` is logically inconsistent with its English description and with the corresponding XPath. As written today in the Duration spreadsheet (`source/datatypes/duration.xml`, row containing the `drt-1` invariant):

| Cell | Current value |
|------|----------------|
| English | "There SHALL be a code if there is a value and it SHALL be an expression of time. If system is present, it SHALL be UCUM." |
| Expression | `code.exists() implies ((system = %ucum) and value.exists())` |
| XPath | `(f:code or not(f:value)) and (not(exists(f:system)) or f:system/@value='http://unitsofmeasure.org')` |

Two concrete bugs in the FHIRPath:

1. **Wrong implication direction.** The description constrains "code required when value present" (`value.exists() implies code.exists()`, equivalent to `code.exists() or value.empty()`). The current expression instead asserts "value required when code present", which neither the description nor the XPath asks for.
2. **Unconditional UCUM requirement when code is present.** The description only constrains `system` "if system is present". The current expression makes `system = %ucum` mandatory whenever `code` is present, even when no `system` element is supplied. The XPath gets this right with `not(exists(f:system)) or f:system/@value='…ucum'`.

The R4B branch fixed this under [FHIR-28415](https://jira.hl7.org/browse/FHIR-28415) (same title, same defect). R4, R5, and the current R6 build still carry the broken expression. The Zulip thread surfaced via cross-reference (`#implementers > Duration Rule Expression`, message `468228695`) is where the reporter recorded this gap before opening the Jira ticket.

### Proposed Change

Update the FHIRPath expression cell of the `drt-1` invariant in `source/datatypes/duration.xml` so that it faithfully encodes the English description and matches the structure of the (already-correct) `Age` invariant, minus the non-negative clause.

- **Resource / datatype:** `Duration`
- **Invariant key:** `drt-1`
- **Severity:** unchanged (`error`)
- **English description:** unchanged — *"There SHALL be a code if there is a value and it SHALL be an expression of time. If system is present, it SHALL be UCUM."*
- **New FHIRPath expression:**

  ```fhirpath
  (code.exists() or value.empty()) and (system.empty() or system = %ucum)
  ```

- **XPath:** unchanged — already consistent with the description:

  ```xpath
  (f:code or not(f:value)) and (not(exists(f:system)) or f:system/@value='http://unitsofmeasure.org')
  ```

For comparison, `Age`'s invariant (in `source/datatypes/age.xml`) reads:

```fhirpath
(code.exists() or value.empty()) and (system.empty() or system = %ucum) and (value.empty() or value.hasValue().not() or value > 0)
```

Duration deliberately omits the trailing `value > 0` clause because Duration values may be negative (e.g., representing offsets relative to an event).

No change to the description text, severity, XPath, cardinality, binding, or any other Duration element is required.

### Design Rationale

- **Mirrors the proven R4B fix.** FHIR-28415 already established the corrected form of the expression for R4B; FHIR-47129 is explicitly asking for parity in R4/R5/R6. Re-using the same shape minimises risk of introducing a different bug.
- **Aligns Duration with Age.** Age and Duration are both "specializations of Quantity" (per `source/datatypes.html#QuantityVariations`) with structurally identical invariants. Diverging FHIRPath shapes between them is a maintenance hazard.
- **Preserves the existing XPath.** The XPath on this row is already correct, so leaving it untouched both reduces churn and gives implementers using XPath validation a stable reference.
- **No semantic change to the description.** Tools that consume the description text (publishers, registry summaries) see no diff; only validators that evaluate the FHIRPath cell change behaviour, and only in the direction of becoming *more* permissive in cases the description never intended to forbid (e.g., a Duration with `value` and `code` but no `system`).

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Source of truth for the Duration datatype, including its invariants. The bespoke HL7 publisher reads the per-datatype spreadsheet to emit StructureDefinition, narrative, validator artifacts, and downstream packages.
- **Affected Files:**
  - `source/datatypes/duration.xml` — change exactly one cell: the FHIRPath expression for the `drt-1` invariant. No other rows or cells touched.
- **Change Scope:** **Minor** — single-cell edit in one authoring spreadsheet.

### Breaking Changes

The change is **backward compatible** in practice and is best characterised as a defect fix:

- The new expression is **strictly more permissive** than the old one. Resources that previously validated against `drt-1` continue to validate.
- Resources that the old expression incorrectly rejected — namely, Durations with `value` and `code` but no `system`, and Durations with `code` and `system = %ucum` but no `value` — will now correctly validate. These cases were already permitted by the description, the XPath, and R4B (post-FHIR-28415).
- No element cardinality, type, binding, or search parameter changes.

The change does, however, alter the **published normative FHIRPath expression** for an invariant that is part of the Duration StructureDefinition, which should be called out in the IG / specification changelog.

### Related Specifications

- **Quantity** and the other Quantity specialisations (`Age`, `Distance`, `Count`, `Money`, `MoneyQuantity`, `SimpleQuantity`) — listed in `source/datatypes.html` under `#QuantityVariations`. None require edits, but `Age` is the structural template the new expression mirrors.

### Terminology Impact

### Testing Considerations

- **Revalidate Duration examples and consumers.** Re-run the standard publisher validation pass over all bundled example resources that contain a `Duration` (search the clone for `<valueQuantity>` / `<duration>` elements typed as Duration, plus uses inside `Timing`, `MedicationRequest.dosageInstruction`, `Task.executionPeriod`-style fields). All previously passing examples must still pass.
- **Edge cases to verify:**
  - Duration with `value` only (no `code`, no `system`) — must fail (description requires `code` when `value` present).
  - Duration with `code` only — must pass.
  - Duration with `code` and `system = %ucum` and `value` — must pass.
  - Duration with `code` and a non-UCUM `system` — must fail.
  - Duration with `code` and `value` but no `system` — must pass under the new expression (was incorrectly failing under the old one).
  - Negative `value` with valid UCUM `code` and `system` — must pass (Duration intentionally permits negative values; this is why the `value > 0` clause from `Age` is omitted).
- **Interoperability considerations:** Validators (HAPI, .NET FHIR validator, fhir-validator-cli) consume the FHIRPath expression directly from the published StructureDefinition. Implementers running pre-fix and post-fix validators against the same Duration instance may see different results in the cases listed above; the post-fix behaviour is the correct one and matches R4B.
- **Regression sweep:** confirm no other invariant in `source/datatypes/duration.xml` was inadvertently modified (a diff of the file should show changes only inside the single `<Data>` element on line 3305).
