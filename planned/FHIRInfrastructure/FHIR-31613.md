## Resolution Summary

The applied resolution (Persuasive with Modification) commits to three behavioural clarifications, originally framed against the never-shipped `CapabilityStatement2`:

1. Provide an explicit way to declare that a server supports **all** (or **none**) of the per-resource-type filters on an `_include` / `_revinclude` join (i.e., the optional `:[targetType]` third segment).
2. Make clear that declaring support for the **non-resource-qualified** form (`Coverage:payor`) does **not** imply support for the **resource-qualified** form (`Coverage:payor:Patient`).
3. Specify that a server which supports the non-filtered form but not the requested filtered form **MAY** either (a) honour the request as the non-filtered include or (b) ignore the `_include` entirely; whichever it does, the server **SHALL** reflect the actual behaviour in the response `Bundle.link` `self` URL.

Status caveat: a 2023-02-15 FHIR-Infrastructure status report on Zulip (`#fhir/infrastructure-wg > Status Report`, message id `328143250`) groups FHIR-31613 with other "CapabilityStatement2 stuff â€” deferred for an implementation guide" items and explicitly notes "Nothing to do for R5". The ticket is still in **Resolved â€“ change required**, so the technical change is pending; this plan therefore proposes the smallest landing site for points 2 and 3 inside the existing R6 `CapabilityStatement` and `search.html`, and flags point 1 (the "all/none" declaration mechanism) as still requiring committee design before any normative-text edit can be merged.

## Feature Proposal

### Problem Statement

FHIR's `_include` / `_revinclude` syntax allows three forms (see `source/search.html` lines 5400â€“5417):

- `_include=*` (wildcard)
- `_include=[resource]:*` (per-resource wildcard)
- `_include=[resource]:[parameter]` (the standard reference-typed include)
- `_include=[resource]:[parameter]:[targetType]` (the optional third segment, restricting to a specific target type when the search parameter targets multiple resources, e.g. `Coverage:payor:Patient`)

`CapabilityStatement.rest.resource.searchInclude` (and the parallel `searchRevInclude`) is a `0..* string` whose comment (`source/capabilitystatement/structuredefinition-CapabilityStatement.xml`, line 1154) currently:

- explains the `*` wildcard and iterative includes, but
- never says whether enumerating `Coverage:payor` does or does not also imply support for the qualified form `Coverage:payor:Patient`,
- never says how a client should interpret a request that uses a qualified third-segment form against a server that only advertises the unqualified form, and
- offers no way to assert "I support all (or no) `[targetType]` filters" without enumerating each combination by hand.

The original Zulip thread (`#implementers > CapabilityStatement searchInclude documentation`, stream 179166) and the FHIR-I review converged on the resolution above. The fix never landed because the CapabilityStatement2 effort â€” the originally intended landing site â€” was deferred.

### Proposed Change

**Repo:** `HL7/fhir` (FhirCore). All edits below are `<differential>`-only / hand-authored HTML; no `<snapshot>` blocks are touched (per the briefing's authoring rules).

**A. Replace the comment on `CapabilityStatement.rest.resource.searchInclude`** to add the two clarifications and the self-link rule.

- File: `source/capabilitystatement/structuredefinition-CapabilityStatement.xml`
- Element: `CapabilityStatement.rest.resource.searchInclude` (line 1150â€“1160).
- Action: Append the following clauses to the existing `<comment value="â€¦"/>` (preserving the existing wildcard/iterative-include text):

  > Each entry is the literal `_include` parameter value the server supports, and SHALL be matched as a string. Listing the three-segment, target-type-qualified form (e.g. `Coverage:payor:Patient`) advertises support **only** for that filtered form; listing the unqualified form (e.g. `Coverage:payor`) advertises support **only** for the unfiltered form. Neither form implies support for the other; a server that supports both SHALL list both.
  >
  > If a client requests a target-type-qualified `_include` (e.g. `_include=Coverage:payor:Patient`) against a server that only advertises the unqualified form, the server MAY either (a) execute the include as if the third segment were absent, or (b) ignore the `_include` entirely. Whichever the server does, the executed semantics SHALL be reflected in the `self` link of the response `Bundle` (i.e. the `self` link SHALL include the `_include` actually evaluated, omitting any segment the server did not honour).

- Element: `CapabilityStatement.rest.resource.searchRevInclude` (line 1161â€“1171).
- Action: The existing comment is `See \`CapabilityStatement.rest.resource.searchInclude\` comments.` â€” leave it as a back-reference; the new clauses then apply by reference. No edit needed beyond verifying the back-reference still resolves after the searchInclude comment grows.

**B. Mirror the clarifications in the search-page narrative** so implementers reading either page get the same answer.

- File: `source/search.html`
- Action: After the existing fhir-conformance note at line 5418â€“5421 (which already says "implementers intending to support wildcard (`*`) segments SHOULD advertise them â€¦ in `CapabilityStatement.rest.resource.searchInclude`"), add a new short paragraph (still inside the `_include` section) along the lines of:

  > Listing `[resource]:[parameter]` and listing `[resource]:[parameter]:[targetType]` are independent assertions â€” neither implies the other. Servers SHALL list each form they support separately. If a client requests a `[resource]:[parameter]:[targetType]` form that the server does not advertise but the server does advertise the unqualified `[resource]:[parameter]` form, the server MAY either execute the include without the target-type filter or ignore the include altogether; the response `Bundle.link` with `relation=self` SHALL reflect the include parameter actually applied.

- Action (point 3 cross-check): the canonical-includes paragraph at lines 5580â€“5584 already asserts the "advertise via `searchInclude` / `revInclude`" pattern; no change needed there, but verify the new paragraph immediately follows the syntactic enumeration so a reader sees the conformance rule in context.

**C. (Deferred sub-feature) Mechanism to declare "all" / "none" of the resource-type filters.**

- The resolution's first bullet ("ensure there's an ability to say that you support all (or none) of the Resource filters") cannot be satisfied with the current `searchInclude` element shape (a free-text `0..* string`). Two implementation options exist:
  1. **Convention only** â€” extend the comment in (A) to define a convention: enumerating `Coverage:payor` *plus* `Coverage:payor:*` SHALL be interpreted as "all `[targetType]` values supported"; enumerating only `Coverage:payor` SHALL be interpreted as "no `[targetType]` filter supported". This requires no schema change.
  2. **Modelled** â€” add an `extension` (or, post-CapabilityStatement2-revival, a new sub-element) carrying a coded `all | none | enumerated` flag per advertised include. This is normative-modelling work and warrants its own ticket.
- This plan recommends **Option 1 (convention only) for the in-place fix**, defines the `:*` token explicitly in (A), and lists the modelled variant as an **Open Question** so it isn't silently lost again.

### Design Rationale

- The resolution was framed against `CapabilityStatement2` because at the time the WG expected a parallel resource. With CapabilityStatement2 deferred (Zulip status reports `295121258`, `295768221`, `328143250`), the only way to deliver the agreed semantics in R6 is to land them in the existing `CapabilityStatement` comment plus `search.html`. This is consistent with how the searchInclude comment already extends `_include` semantics (e.g. the iterative-include example added later).
- Tying the "what did the server actually do?" rule to `Bundle.link` `relation=self` reuses an existing, normative reflection mechanism that clients already inspect (`search.html` already discusses self-link contents elsewhere); no new structural element is needed for point 3.
- Stating "neither implies the other" rather than "qualified implies unqualified" matches the consensus from the `#implementers > CapabilityStatement searchInclude documentation` thread (Zulip `231669362`, the originating discussion linked from the ticket): implementers asked for an explicit declaration model, not an inferred one, because real servers commonly support one form but not the other.
- Putting the "all `[targetType]` values" convention behind the existing `:*` per-resource wildcard token (already defined in `search.html` line 5404) avoids inventing new syntax and keeps the convention-only path schema-compatible with R5.
