## Feature Proposal

### Problem Statement

The page in question asserts that "This specification is fully normative"
without qualification, yet the specification still contains pages and
artifacts that are marked **Informative**, **Draft**, or **STU**. The
reporter observes that the unqualified claim is not supported by the actual
content of the specification (for example, the roadmap page itself is
Informative), and that a clear rationale is needed for whether *all*
content is normative.

The pre-existing wording is in the bulleted list under the "notes about
breaking changes" heading in `source/ballot-intro.html` (line 43):

```html
<li>This specification is fully normative, so once this ballot is
    complete, no breaking changes are allowed in future publications</li>
```

This is the statement the FMG intends to scope down.

### Proposed Change

Replace the phrase "This specification is fully normative" with
"The implementable content in this specification is fully normative (i.e.
no '[draft]()' or '[STU]()' content remains)" in the single bullet on
`source/ballot-intro.html` (line 43). Keep the rest of the bullet (the
clause about breaking changes after the ballot) intact.

After the edit the bullet should read:

```html
<li>The implementable content in this specification is fully normative
    (i.e. no '[draft]()' or '[STU]()' content remains), so once this
    ballot is complete, no breaking changes are allowed in future
    publications</li>
```

Notes on the `'[draft]()'` / `'[STU]()'` tokens copied verbatim from the
resolution:

- In the Jira rendering these tokens appeared inside
  `<span class="error">…</span>`, which is how the HL7 publisher highlights
  unresolved Markdown-style link references (`[label](target)` with an
  empty target). The resolution text preserves those tokens as literal
  strings the reader will recognise as the standards-status labels used
  elsewhere in the spec.
- Preserve them as literal text (with the single quotes around each
  token) in the authored HTML. Do **not** try to render them as
  cross-links; the authors deliberately showed them as the visible
  markers a reader encounters on non-normative pages.

No other changes are required on this page, and no other page in
`source/` contains the exact phrase "This specification is fully
normative" (verified by grep against the clone).

### Design Rationale

- The reporter's concern is that the unqualified claim overstates the
  normativity of the publication; the fix narrows the claim to the
  *implementable* content and explicitly acknowledges that `[draft]`
  and `[STU]` markers would invalidate the claim, which is consistent
  with how those standards-status markers are used elsewhere in the
  spec (e.g., on resource and element banners).
- The change is a targeted wording edit on a single authored HTML
  page. No structural, schema, terminology, or conformance changes are
  required, so no alternatives were seriously considered — the
  resolution dictates the exact wording.
- Keeping the trailing clause ("so once this ballot is complete, no
  breaking changes are allowed in future publications") preserves the
  existing forward-compatibility guarantee that subsequent bullets rely
  on (e.g., the bullets that discuss forwards compatibility and R4
  normativity immediately below).
