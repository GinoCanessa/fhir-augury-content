## Resolution Summary

**Resolution:** Persuasive

**Resolution description (verbatim):** *"Will update the metadata on these types so they appear as abstract."*

The committee accepted the proposal that the two type-framework anchors `Base`
and `DataType` should be presented in the same "abstract" grouping as the
other foundational abstract types (`Element`, `BackboneElement`,
`PrimitiveType`, `BackboneType`). The agreed action is a metadata-only update:
adjust the catalog metadata that drives how these types are classified and
rendered so that Base and DataType are unambiguously surfaced as abstract
types in the spec.

## Feature Proposal

### Problem Statement

The reporter (Thomas Zhou) noted that both `Base` and `DataType` are abstract
types — `Base` is the root of the entire FHIR type system and `DataType` is
the abstract root of every concrete data type — yet, in the spec metadata
that drives type classification, they are not consistently grouped with the
other abstract types (`Element`, `BackboneElement`, `PrimitiveType`,
`BackboneType`). Moving them into the same abstract group would align the
published metadata with their abstract nature and remove the inconsistency
that triggered the ticket.

### Proposed Change

Apply the metadata classification change in the FHIR Core spec
(`HL7/fhir`), authoring root `source/`, so that Base and DataType are
catalogued and published as abstract types:

1. **`source/fhir.ini` — `[infrastructure]` section.**
   Confirm and (if needed) reassert the abstract flag for both types in the
   ini section the publisher uses to drive the abstract type list. In the
   current clone (`db79dcd`) this section already reads:

   ```ini
   [infrastructure]
   ; order matters here - must be in order of dependency
   ; FHIR-I must approve changes to this list
   Base=abstract
   Element=abstract
   BackboneElement=abstract
   DataType=abstract
   PrimitiveType=abstract
   BackboneType=abstract
   Extension
   Narrative
   ```

   Both `Base=abstract` and `DataType=abstract` are present. The plan must
   verify (a) that this is the section the publisher actually uses to
   produce the abstract-type grouping, and (b) that the order keeps Base
   first and DataType ahead of its concrete specializations
   (PrimitiveType, BackboneType). If a follow-up section (e.g.
   `[type-pages]`, `[type-characteristics]`, `[fmm]`, `[standards-status]`)
   contradicts this classification, update it to match.

2. **Published `StructureDefinition` metadata for `Base` and `DataType`.**
   Verify that the publisher emits `<abstract value="true"/>` in the
   generated `StructureDefinition-Base.xml` and
   `StructureDefinition-DataType.xml`. Snapshots are generated and out of
   scope per the briefing, but the source files must drive that flag.
   The two source spreadsheets are:

   - `source/datatypes/base.xml`
   - `source/datatypes/datatype.xml`

   Both are MS Office XML spreadsheets (the briefing flags this — they are
   not FHIR XML). Inspection shows the `Data Elements` worksheet does not
   currently expose an `Abstract` column for these two types. The
   `[infrastructure]` entry in `fhir.ini` is therefore the canonical
   abstract switch. If, after the publisher run, either generated
   `StructureDefinition` still ships with `<abstract value="false"/>`, the
   fix is to add the missing classifier — either by extending the
   spreadsheet header (matching what `element.xml` /
   `backboneelement.xml` / `primitivetype.xml` / `backbonetype.xml` use)
   or by promoting the `=abstract` flag in `[infrastructure]` to a fully
   honoured publisher input. Decide between these two options with FHIR-I
   before editing the spreadsheets.

3. **Type Framework prose pages.**
   `source/types.html` already documents Base and DataType under the
   "FHIR Type Framework" page that visualises the abstract types
   (`[%diagram datatypes/abstracts.diagram 3dt%]`), and
   `source/datatypes/abstracts.diagram` already lists both classes:
   `Base,Element,BackboneElement,DataType,Resource,DomainResource,PrimitiveType,BackboneType,…`.
   No prose change is required if the metadata fix above is sufficient,
   but the page must be re-rendered after the metadata change so the
   "abstract types" diagram and tab label remain accurate.

### Design Rationale

- Base and DataType *are* abstract; classifying them anywhere else is a
  factual inconsistency in the spec metadata.
- The fix is intentionally narrow and metadata-only ("Will update the
  metadata on these types so they appear as abstract") — no element,
  cardinality, binding, or invariant changes.
- Reusing the existing `[infrastructure]` ini section keeps the change
  consistent with how `Element`, `BackboneElement`, `PrimitiveType`, and
  `BackboneType` are already classified, instead of inventing a new
  grouping.
- No alternative approach was discussed in the ticket (no comments were
  recorded). The resolution wording confirms the metadata path was the
  agreed design.

### Breaking Changes

No breaking changes identified.

- The `<abstract>` flag on a `StructureDefinition` already reflects
  reality (Base and DataType are not instantiable). Flipping the
  metadata to expose that reality cannot break any conformant
  consumer — there are no instances of `Base` or `DataType` to
  invalidate, and no profiles target them as instantiable types.
- No element is added, removed, retyped, or rebound.
- No search parameter, invariant, or binding strength changes.

### Related Specifications

  — abstract resources catalogued in `[resource-infrastructure]`

### Open Questions

- **Where does the publisher actually read the abstract flag from?**
  `fhir.ini` `[infrastructure]` already lists `Base=abstract` and
  `DataType=abstract`, yet the ticket implies the published metadata
  does not currently treat them as abstract. The plan needs either
  (a) confirmation that this section is what flips
  `<abstract value="true"/>`, or (b) discovery of the additional input
  (a spreadsheet column, a hard-coded list in
  `tools/`/`HL7/kindling`, a `[type-characteristics]` entry, etc.)
  that has to be updated. Resolve before deciding whether Task 3 is a
  no-op or a real edit.
- **Spreadsheet "Abstract" column convention.** The `Data Elements`
  worksheet for `base.xml` and `datatype.xml` does not expose an
  obvious `Abstract` column. Confirm with the publisher maintainers
  whether sibling abstract types (`element.xml`, `primitivetype.xml`,
  `backboneelement.xml`, `backbonetype.xml`) carry an explicit
  abstract flag in the spreadsheet, or whether they too rely solely
  on `fhir.ini` `[infrastructure]`.
- **Publisher repo coverage.** If the abstract flag is decided in
  publisher code rather than data, the actual implementation would
  spill into `HL7/kindling` (or another publisher repo). That is out
  of scope for HL7/fhir and would need a sibling ticket — flag this
