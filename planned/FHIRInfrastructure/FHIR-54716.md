## Resolution Summary

Resolution: **Persuasive**.

The Questionnaire constraint `que-10` (`Maximum length can only be declared for
simple question types`) contains a typo in its FHIRPath expression: the literal
`'optionOrString'` is misspelled and must be `'optionsOrString'` to match the
actual code defined by `CodeSystem/questionnaire-answer-constraint`. The
ticket text explicitly states: *"Typo in constraint rule 'que-10': 'optionOrString'
in fhirpath constraint should be 'optionsOrString'."* No further changes are
requested.

## Feature Proposal

### Problem Statement

The Questionnaire resource defines invariant `que-10`, which permits
`maxLength` only when the question is a simple type **or** when
`answerConstraint` permits free-text-style answers. The FHIRPath expression
references the literal string `'optionOrString'`, but the value bound to
`Questionnaire.item.answerConstraint` is the code `optionsOrString`
(plural — see `source/questionnaire/codesystem-questionnaire-answer-constraint.xml`
line 35: `<code value="optionsOrString"/>`). Because the constraint compares
against a non-existent code, the `answerConstraint='optionOrString'` clause can
never be true, so any Questionnaire item that legitimately uses
`answerConstraint = optionsOrString` together with `maxLength` will be
incorrectly flagged as failing `que-10`. The element's own `short` description
on line 1267 of the StructureDefinition correctly lists
`"optionsOnly | optionsOrType | optionsOrString"`, confirming the canonical
spelling and reinforcing that line 793 is a pure transcription error.

### Proposed Change

In `source/questionnaire/structuredefinition-Questionnaire.xml`, in the
`<constraint>` element whose `<key value="que-10"/>` (the only constraint with
that key — verified by grep), change the FHIRPath `<expression>` from:

```
(type in ('string' | 'decimal' | 'integer' | 'text' | 'date' | 'dateTime' | 'time' | 'url')) or answerConstraint='optionOrString' or maxLength.empty()
```

to:

```
(type in ('string' | 'decimal' | 'integer' | 'text' | 'date' | 'dateTime' | 'time' | 'url')) or answerConstraint='optionsOrString' or maxLength.empty()
```

Single-character insertion: `optionOrString` → `option`**`s`**`OrString`. No
other element of the constraint changes (key, severity, human, source all
remain). No other file references the misspelled token (verified by recursive
grep across `source/`).

### Design Rationale

- **Minimal-surface technical correction.** The ticket type is *Technical
  Correction* and the resolution simply applies the reporter's exact request.
  No design decision is involved — the canonical code is already defined in
  `CodeSystem/questionnaire-answer-constraint`, and the StructureDefinition's
  own `short` text on the `answerConstraint` element already uses the correct
  spelling. This is purely aligning the invariant expression with the rest of
  the artifact.
- **Why not also update the `human` text?** The human-readable sentence
  ("Maximum length can only be declared for simple question types") is
  unchanged and still accurate; only the FHIRPath literal was wrong.
- **Why not deprecate / re-key the constraint?** `que-10` has been published
  with this key; renaming would be a breaking change far beyond the scope of a
  technical-correction ticket. Fixing the literal in place is the smallest
  change that restores intended behavior.

### Breaking Changes

No breaking changes for spec consumers in the conformance sense — no element,
type, cardinality, binding, search parameter, or operation changes.

Behavioral note for **validator implementers**: any implementation that
mechanically executes the published `que-10` FHIRPath will, after this fix,
correctly reject `Questionnaire` items where the question is not a simple type
**and** `answerConstraint != 'optionsOrString'` **and** `maxLength` is set.
Resources that were silently passing because the previous expression's
middle clause was permanently false (since no item could ever have
`answerConstraint = 'optionOrString'`) will now be evaluated against the
intended rule. This is the intended correction, not a regression. Validator
test fixtures that rely on the broken behavior must be updated.
