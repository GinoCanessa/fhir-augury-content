## Resolution Summary

The Work Group accepted this Technical Correction with resolution **Persuasive**.
The ticket reports that the word *"managable"* is used in the FHIR specification
where the standard English spelling *"manageable"* is intended (illustrated by a
quoted sentence from the extensibility narrative: *"To make the use of extensions
safe and managable, there is strict governance appliedŌĆ”"*). The agreed change is
a non-substantive spelling correction: replace `managable` with `manageable`
(and the same correction for the related forms `unmanagable` ŌåÆ `unmanageable`
and `managableness` ŌåÆ `manageableness`) wherever the misspelling occurs in
authored content. No semantic, structural, cardinality, binding, or invariant
change is required.

The ticket description is the only authoritative resolution text ŌĆö there is no
separate `resolution_description` and there are no comments, applied-vote
notes, related Jira tickets, related Zulip threads, related GitHub issues/PRs,
or other cross-references on this ticket (`cross-referenced` returned `total: 0`).

## Feature Proposal

### Problem Statement

The non-standard spelling *"managable"* appears throughout the authored FHIR
Core specification, most visibly in the canonical `Element.extension` /
`DomainResource.extension` / `DomainResource.modifierExtension` definition
text that is replicated across many StructureDefinitions and rendered into
the published narrative pages. The reporter cites the sentence on the
**Extensibility** page (`source/extensibility.html`, line 48):

> *"To make the use of extensions safe and **managable**, there is strict
> governance applied to the definition and use of extensions. Although any
> implementer can define and use extensions, there is a set of requirements
> that must be met as part of their use and definition."*

The misspelling is also present on other narrative pages (`exchanging.html`,
`overview-arch.html`, `workflow-management.html`, `workflow-ad-hoc.html`),
in the canonical extension-definition prose that is hand-authored into
several `source/datatypes/*.xml` and StructureDefinition files, in the
`StructureMap.group` definition, and in the `StructureMap` SVG diagrams.
The misspelling is also entered as an allowed word in the project spell-check
dictionary (`source/spelling/english.dic`), which is what has prevented it
from being flagged automatically until now.

### Proposed Change

Apply a documentation-only spelling correction across the `source/` tree of
`HL7/fhir`:

1. Replace every occurrence of the literal token `managable` with `manageable`
   in **authored** content under `source/`.
2. Replace `unmanagable` with `unmanageable` for consistency (occurs in
   `source/exchanging.html` and `source/workflow-ad-hoc.html`).
3. Remove the entries `managable`, `unmanagable`, and `managableness` from the
   project spell-check dictionary `source/spelling/english.dic` so that any
   future reintroduction of the misspelling is caught by tooling.

The clone HEAD (`db79dcd`) currently contains the misspelling in the following
33 files (verified via `git grep -l -i 'managable' -- 'source/*'`):

- Narrative / HTML pages
  - `source/extensibility.html` (the example cited by the reporter)
  - `source/exchanging.html`
  - `source/overview-arch.html`
  - `source/workflow-management.html`
  - `source/workflow-ad-hoc.html`
- Canonical extension / modifierExtension definition text (hand-authored)
  - `source/datatypes/element.xml`
  - `source/datatypes/backboneelement.xml`
  - `source/datatypes/backbonetype.xml`
  - `source/domainresource/structuredefinition-DomainResource.xml`
  - `source/profiles/cda.profile.xml`
  - `source/structuredefinition/structuredefinition-example.xml`
  - `source/basic/basic-introduction.xml`
  - `source/resource/resource-notes.xml` (text appears inside hand-authored
    HTML `<table>` snippets / element tooltips, not in a generated snapshot)
- StructureMap definition + diagrams
  - `source/structuremap/structuredefinition-StructureMap.xml`
    (the `StructureMap.group` element `<definition>` "Organizes the mapping
    into managable chunksŌĆ”")
  - `source/structuremap/structuremap.svg`
  - `source/structuremap/structuremap-old.svg`

    `managableness`)

  `source/structuredefinition/invariant-tests/`):
  `cnl-0.f1.fail.xml`, `cnl-1.f1.fail.xml`, `sdf-1.f1.fail.xml`,
  `sdf-1.f2.fail.xml`, `sdf-2.f1.fail.xml`, `sdf-5.f1.fail.xml`,
  `sdf-8.f1.fail.xml`, `sdf-8b.f1.fail.xml`, `sdf-11.f1.fail.xml`,
  `sdf-14.f1.fail.xml`, `sdf-15.f1.fail.xml`, `sdf-16.f1.fail.xml`,
  `sdf-18.f1.fail.xml`, `sdf-19.f1.fail.xml`, `sdf-27.f1.fail.xml`,
  `sdf-28.f1.fail.xml`. These each carry a copy of the canonical
  `extension`/`modifierExtension` definition text to make the fixture
  parseable; updating them keeps the fixture text in sync with the canonical
  definitions and avoids a future diff churning these files.

The replacement is a literal, case-preserving search-and-replace; there are no
sentence-meaning changes, and no other words (e.g., `manage`, `manager`,
`management`, `managed`) match the search.

### Design Rationale

- *"Managable"* is not standard English; *"manageable"* (preserving the silent
  `-e` of *manage*) is the conventional spelling. The Work Group resolved this
  as **Persuasive** ŌĆö i.e., the reporter is correct and the change should be
  made.
- The extension/modifierExtension definition text is the canonical
  human-authored prose that is copied (often by hand) into many places
  in the spec. Because the briefing notes that for FHIR Core only
  `<differential>` content of StructureDefinitions is hand-edited, the
  authored copies of this prose in `source/datatypes/*.xml`,
  `source/domainresource/structuredefinition-DomainResource.xml`,
  `source/profiles/cda.profile.xml`, and `source/structuredefinition/
  structuredefinition-example.xml` are exactly the right place to apply
  the fix at the source. `<snapshot>` blocks are regenerated by the
  publisher and must not be edited directly ŌĆö see the briefing's
  "Generated areas (do not edit)".
- Updating the dictionary closes the loop so that lint/spell-check passes
  flag any reintroduction of the misspelling, preventing regression.
- An alternative narrower fix (only the single sentence cited in the ticket
  on `extensibility.html`) would leave the misspelling in the canonical
  `Element.extension` definition source ŌĆö meaning the typo would
  immediately reappear on the published page next time the publisher
  regenerated it from the StructureDefinitions. The repo-wide fix is the
  correct interpretation of the resolution.
