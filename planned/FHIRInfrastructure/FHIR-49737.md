## Feature Proposal

### Problem Statement

`source/mappingSpaces.xml` is the master catalogue of external mapping
spaces that the FHIR publisher consumes when rendering each
StructureDefinition's "Mappings" tab and the global mappings index. Two
of the entries point at URLs that no longer resolve:

- The CDISC entry only carries a `<url>` (`http://www.cdisc.org/define-xml`)
  with no `<link>`, and the legacy CDISC URL no longer reaches the
  Define-XML page. Readers clicking through from a published FHIR mapping
  see a broken-link / redirect-loop experience.
- The BRIDG entry has a `<link>` pointing at a deep BRIDG 5.1 download
  package (`…/bridg-releases/release-51/release-package`) that has been
  retired by NCI. The deep path 404s; only the BRIDG site root
  (`https://bridgmodel.nci.nih.gov/`) is reliably available.

Because these are catalogue-level URLs, every consumer of the FHIR
publication (R5, R6, every snapshot/CI build) renders the broken targets
on every page that declares a `BRIDG5.1` or `define.xml2.0` mapping.

### Proposed Change

Edit one file: `source/mappingSpaces.xml`.

**1. CDISC Define-XML entry (lines 238–245 in the current clone).**

Update the `<url>` to the working URL **and** add a `<link>` so the
publisher renders an explicit, working hyperlink. The `<url>` is the
formal identifier; while no current StructureDefinition declares
`<identity value="define.xml2.0"/>` in this clone (confirmed by
repository-wide grep), keeping `<url>` and `<link>` aligned is the
convention used by the surrounding entries (e.g., `fivews`, `workflow`)
and avoids divergence if mappings are added later.

Before:

```xml
<space>
    <url>http://www.cdisc.org/define-xml</url>
    <columnName>CDISC Define.xml 2.0 Mapping</columnName>
    <title>CDISC Define XML 2.0 metadata standard</title>
    <id>define.xml2.0</id>
    <sort>18</sort>
    <publish>true</publish>
</space>
```

After:

```xml
<space>
    <url>https://www.cdisc.org/standards/data-exchange/define-xml</url>
    <link>https://www.cdisc.org/standards/data-exchange/define-xml</link>
    <columnName>CDISC Define.xml 2.0 Mapping</columnName>
    <title>CDISC Define XML 2.0 metadata standard</title>
    <id>define.xml2.0</id>
    <sort>18</sort>
    <publish>true</publish>
</space>
```

(The commented-out duplicate at lines 246–253 should also have its
`<url>` updated for consistency, since it is clearly a parked copy of
the live entry.)

**2. BRIDG 5.1 entry (lines 414–421 in the current clone).**

The `<url>` is already `https://bridgmodel.nci.nih.gov` (the BRIDG site
root, used as the mapping identity URI in
`source/researchstudy/structuredefinition-ResearchStudy.xml` and
`source/researchsubject/structuredefinition-ResearchSubject.xml` —
**do not change `<url>`**). Replace the broken `<link>` with the BRIDG
home page, matching the URL field reporter requested
(`https://bridgmodel.nci.nih.gov/`).

Before:

```xml
<space>
    <url>https://bridgmodel.nci.nih.gov</url>
    <link>https://bridgmodel.nci.nih.gov/download-model/bridg-releases/release-51/release-package</link>
    <columnName>BRIDG 5.1 Mapping</columnName>
    <title>BRIDG 5.1 Mapping</title>
    <id>BRIDG5.1</id>
    <sort>38</sort>
    <publish>true</publish>
</space>
```

After:

```xml
<space>
    <url>https://bridgmodel.nci.nih.gov</url>
    <link>https://bridgmodel.nci.nih.gov/</link>
    <columnName>BRIDG 5.1 Mapping</columnName>
    <title>BRIDG 5.1 Mapping</title>
    <id>BRIDG5.1</id>
    <sort>38</sort>
    <publish>true</publish>
</space>
```

### Design Rationale

- `mappingSpaces.xml` separates the `<url>` (formal identifier, also used
  as the `<uri>` value of `<mapping>` blocks inside StructureDefinitions
  via the matching `<identity value="…"/>`) from the optional `<link>`
  (the displayed hyperlink). For BRIDG, `<url>` is referenced as a
  mapping URI in StructureDefinitions, so it must remain stable; only
  `<link>` is updated.
- For CDISC the formal `<url>` is not currently bound to any
  StructureDefinition mapping in this clone, so updating it (in addition
  to introducing `<link>`) is safe and keeps the catalogue truthful.
- This is the "Update narrative / prose" recipe from the per-repo
  briefing's "Recommended Change Recipes", scoped to a non-HTML
  catalogue file: edit authored XML directly under `source/`; the
  publisher consumes it on the next build.
- No alternative to a direct edit was considered — the URLs are wrong
  on the face of it, the resolver requested specific replacements, and
  no resource mappings need to change.

### Step-by-Step Tasks

#### HL7/fhir

1. **Fix the CDISC Define-XML mapping space entry**
   - File: `source/mappingSpaces.xml`
   - Action: In the `<space>` block whose `<id>` is `define.xml2.0` (currently lines 238–245), change `<url>http://www.cdisc.org/define-xml</url>` to `<url>https://www.cdisc.org/standards/data-exchange/define-xml</url>` and insert `<link>https://www.cdisc.org/standards/data-exchange/define-xml</link>` immediately after the `<url>` element.
   - Details: Preserve sibling elements (`<columnName>`, `<title>`, `<id>`, `<sort>`, `<publish>`) and the existing element ordering used by other entries (e.g., `fivews`, `workflow`).

2. **Update the commented-out CDISC duplicate (consistency)**
   - File: `source/mappingSpaces.xml`
   - Action: In the commented block immediately following (currently lines 246–253), update the same `<url>` to the new value. Leave the block commented.
   - Details: Strictly optional, but keeps the parked copy from drifting from the live entry; cheap to do at the same time.

3. **Fix the BRIDG 5.1 mapping space entry**
   - File: `source/mappingSpaces.xml`
   - Action: In the `<space>` block whose `<id>` is `BRIDG5.1` (currently lines 414–421), replace `<link>https://bridgmodel.nci.nih.gov/download-model/bridg-releases/release-51/release-package</link>` with `<link>https://bridgmodel.nci.nih.gov/</link>`.
   - Details: Do **not** touch `<url>https://bridgmodel.nci.nih.gov</url>` — it is the mapping identity URI referenced from `source/researchstudy/structuredefinition-ResearchStudy.xml` and `source/researchsubject/structuredefinition-ResearchSubject.xml`.

4. **Manual smoke-check the two replacement URLs resolve**
   - File: n/a (out-of-tree verification)
   - Action: `curl -I https://www.cdisc.org/standards/data-exchange/define-xml` and `curl -I https://bridgmodel.nci.nih.gov/`; both should return a 2xx (or a tolerated redirect to a 2xx) — not a 404.
   - Details: Cheap insurance that we are not swapping one broken link for another.

5. **Run the publisher locally and inspect a consumer page**
   - File: n/a
   - Action: Run the standard local publisher build (per `publish.sh` / `build.sh`), then open the published `researchstudy.html` (or any page derived from a StructureDefinition with a `BRIDG5.1` mapping) and confirm the BRIDG link in the Mappings tab now points at `https://bridgmodel.nci.nih.gov/`. Spot-check the global mappings index page for the CDISC entry.
   - Details: Do not edit any file under `qa/`, `tools/`, `schema/`, `publish/`, or any generated `<snapshot>` block.

### Open Questions

- **Should the CDISC `<url>` be updated or only `<link>` added?** This plan updates both because no StructureDefinition currently references the `define.xml2.0` identity (verified by grep), and the surrounding entries align `<url>` with `<link>`. If the FHIR-Infrastructure WG would prefer to keep the legacy `<url>` for historical continuity and only add `<link>`, the change in Task 1 collapses to a single insertion — no other tasks change.
- **Should the commented-out CDISC duplicate be removed entirely rather than just refreshed?** The briefing's "do not silently roll in out-of-scope changes" guidance argues for leaving it commented. This plan only updates the URL inside the comment; deleting the block is intentionally out of scope.
