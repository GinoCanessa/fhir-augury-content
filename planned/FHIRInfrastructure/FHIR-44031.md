## Resolution Summary

**Resolution:** Persuasive — *"Will clean up the todos one way or the other."*

The Ad-Hoc Workflow Communication Patterns page (`workflow-ad-hoc.html`) currently
ships with a number of unresolved placeholders (`TODO`, `???`, `TBD`, and an empty
`<li>`). The resolution authorises the owners to remove these placeholders by
replacing them with substantive content **or** by deleting the relevant
incomplete passages — whichever is appropriate per item. The change is purely a
documentation cleanup confined to one specification page in `HL7/fhir`.

## Feature Proposal

### Problem Statement

The published page <https://hl7.org/fhir/6.0.0-ballot1/workflow-ad-hoc.html>
contains placeholders that leak into a normative-track ballot publication. The
ticket reporter highlighted two anchors specifically:

- `#12.11.1.5` — corresponds to the "Option D: Messaging request from placer to
  filler & acknowledgment" sub-section, which currently opens with an italicised
  `TODO: needs review and update`.
- `#optione` — "Option E: Services request from placer to filler &
  acknowledgment", which contains the most placeholders: a `TODO` banner, an
  empty step (`<li></li>`), `???` items in Benefits and Limitations, and a `TBD`
  Usage Recommendations.

In addition to those two, the same page carries equivalent placeholders in the
"Additional Scenarios" section. None of these are acceptable in a published
ballot artifact, and the goal of this ticket is to remove them all in one
sweep.

### Proposed Change

Edit `source/workflow-ad-hoc.html` in `HL7/fhir` to remove every
placeholder (`TODO`, `???`, `TBD`, empty `<li>`) currently present in the
authoring file. Concretely, address each of the following occurrences (line
numbers reference the file at clone HEAD `db79dcd`):

| Line | Section | Current text | Disposition |
|-----:|---------|--------------|-------------|
| 145  | Option D | `<i>TODO: needs review and update</i>` | Either replace with the agreed introductory paragraph for Option D (mirroring the style of Options A–C earlier in the page), or delete the italicised paragraph entirely if no extra prose is required. |
| 182  | Option E | `<i>TODO: Needs review, update and more work - there's not a lot of experience using FHIR services to manage the fulfillment process</i>` | Replace with a brief, accurate framing paragraph, **or** keep a short editorial note worded as plain prose (no `TODO:` prefix) acknowledging limited deployment experience. |
| 189  | Option E → Steps | `<li></li>` (empty list item) | Delete the empty `<li>` element so the ordered list renders cleanly with the three real steps. |
| 193  | Option E → Benefits | `<li>???</li>` | Replace with the actual benefits of an HTTP-services-based pattern (e.g., synchronous confirmation, reduced infrastructure compared with messaging, alignment with existing REST tooling), or — if the work group cannot agree on content — remove the empty `<ul>`/`<h4>Benefits</h4>` block. |
| 197  | Option E → Limitations | `<li>???</li>` | Replace with concrete limitations (e.g., synchronous coupling, harder to negotiate `no`, no built-in delivery guarantees), or remove the empty `<ul>`/`<h4>Limitations</h4>` block. |
| 201  | Option E → Usage Recommendations | `TBD` | Replace with a one-paragraph recommendation (when to choose services over messaging/REST polling), or remove the `<h4>Usage Recommendations</h4>` block if no recommendation is forthcoming. |
| 209  | Additional Scenarios | `<i>TODO: needs review and update. Possibly add options about using messaging and/or services instead of polling/subscription in above scenarios</i>` | Either replace with intro prose for the Additional Scenarios section, or simply delete the italicised paragraph (the existing `Querying the status…` subsection that follows is self-contained). |

For sub-sections where new prose is supplied, follow the existing pattern in the
same file (Options A, B, C all open with a single `<p>`-wrapped sentence and
then list `Steps`, `Benefits`, `Limitations`, `Usage Recommendations`, and
`Usage Examples`).

The page's anchors (`#optiond`, `#optione`, `#additional`) and the
already-published numbering must be preserved so external links such as the two
in the original ticket continue to resolve.

### Design Rationale

- The resolution explicitly allows either prose replacement *or* deletion
  ("one way or the other"). The disposition table above offers both
  options per item so the work group can pick the path of least cost
  while still ending up with no `TODO`/`???`/`TBD` strings on the page.
- Keeping anchors intact preserves the value of inbound links, including
  the two URLs cited on the ticket.
- All other Workflow pages (`workflow.html`, `workflow-management.html`,
  `workflow-communications.html`, `workflow-examples.html`,
  `workflow-module.html`) have already been checked in this clone — no
  comparable placeholders need to be cleaned up alongside this ticket
  (verified by ripgrep over `source/workflow*.html`).
- Because this is hand-authored HTML inside `source/`, no publisher
  regeneration is involved beyond the normal build.
