## Resolution Summary

**Resolution:** Persuasive.

> Will create an example and link to it from the Request pattern. Will also link
> to a multi-order example where we **don't** use RequestGroup.

Concretely, the resolution requires two deliverables:

1. A worked example of grouping multiple orders using what the ticket calls
   **RequestGroup** (in R6 this resource is **RequestOrchestration**; the ticket
   carries the `R6-Text` label). The Request pattern page must link to it.
2. A second worked example showing how to associate multiple orders **without**
   using RequestOrchestration (i.e., using the `Request.groupIdentifier` /
   `ServiceRequest.requisition` mechanism). The Request pattern page must link
   to it as well.

No structural/schema change to any resource is required — this is a
documentation + examples change. The open question raised in the original
description ("how would we link from the ServiceRequest examples to
requestgroup-examples") is answered by the resolution: add explicit links from
the Request pattern narrative.

## Feature Proposal

### Problem Statement

The Request pattern page (`request.html`, authored from
`source/request/request-spreadsheet-notes.xml`) already describes — in prose —
two mechanisms for linking multiple related orders:

- `Request.groupIdentifier` to tag independent but related requests as part of
  one logical "order set" (notes lines ~87–99).
- `RequestOrchestration` for coordinated, multi-action orders sharing a single
  status (notes lines ~130–149, section anchored at `<a name="requestorchestration"></a>`).

However, there are no end-to-end **examples** demonstrating either approach,
and ServiceRequest readers arriving from `servicerequest.html` have no example
pointer to RequestOrchestration, despite the ServiceRequest introduction
(`source/servicerequest/servicerequest-introduction.xml`, lines 41 and 44)
explicitly directing them to the Request pattern and to RequestOrchestration
for multi-item scenarios. The `source/requestorchestration/` folder ships only
two examples (`requestorchestration-example` — offset actions — and
`requestorchestration-kdn5-example`), neither of which is framed as a plain
"multi-item order" to match the narrative in the Request pattern.

The ticket description and resolution together ask for concrete, linked
examples so that implementers can see each mechanism in action rather than
reasoning from prose alone.

### Proposed Change

Add two multi-item-order examples and wire them into the relevant narrative
pages. All paths resolve against the `HL7/fhir` clone
(HEAD `db79dcd`) authoring root `source/`.

1. **New multi-item-order example that DOES use RequestOrchestration.**
   - Location: `source/requestorchestration/requestorchestration-example-multi-order.xml`
     (a `RequestOrchestration` resource instance).
   - Content: a single RequestOrchestration whose `action[]` entries reference
     2–3 distinct `ServiceRequest` instances (e.g., a lab panel + an imaging
     study + a referral), each carried as an `option`-intent child
     ServiceRequest under `action.resource`. Per
     `source/request/request-spreadsheet-notes.xml` lines 131–149, all
     referenced child requests must have `intent = option`, and their status
     is governed by the parent RequestOrchestration.
   - Also add the child ServiceRequest example instances used by the
     orchestration under `source/servicerequest/`, e.g.
     `servicerequest-example-multi-order-lab.xml`,
     `servicerequest-example-multi-order-imaging.xml`,
     `servicerequest-example-multi-order-referral.xml` (names chosen to match
     the existing `servicerequest-example-<slug>.xml` convention seen in the
     clone: `servicerequest-example-ambulation.xml`,
     `servicerequest-example-colonoscopy.xml`, etc.).
   - Register the new examples in:
     - `source/requestorchestration/list-RequestOrchestration-examples.xml`
       (add a new `<entry>` mirroring the existing entries, with
       `<reference value="RequestOrchestration/example-multi-order"/>` and an
       appropriate title/description).
     - `source/servicerequest/list-ServiceRequest-examples.xml` (add one
       `<entry>` per new child ServiceRequest).

2. **New multi-order example that does NOT use RequestOrchestration.**
   - Location: `source/servicerequest/servicerequest-example-groupidentifier-*.xml`
     (2–3 sibling ServiceRequest instances).
   - Content: 2–3 independent `ServiceRequest` instances sharing the same
     `ServiceRequest.requisition` (and/or `Request.groupIdentifier`), e.g. a
     CBC + a CMP + a lipid panel ordered together under one requisition
     number. Each has a normal (non-`option`) `intent` such as `order`, and
     each has its own independent `status`. This is the pattern described in
     `source/request/request-spreadsheet-notes.xml` lines 87–99 and in
     `source/servicerequest/servicerequest-introduction.xml` line 44
     ("ServiceRequest.requisition … for billing").
   - Register in `source/servicerequest/list-ServiceRequest-examples.xml`.

3. **Request pattern narrative — add links to both examples.**
   - File: `source/request/request-spreadsheet-notes.xml`.
   - In the existing `groupIdentifier` paragraph (around lines 87–99) append a
     sentence: "See the [multi-order example using `requisition`](servicerequest-example-groupidentifier-cbc.html) for a worked illustration of this pattern."
   - In the existing `RequestOrchestration` section (lines 130–149, anchor
     `requestorchestration`) append a sentence: "See
     [requestorchestration-example-multi-order](requestorchestration-example-multi-order.html)
     for a worked multi-item order using RequestOrchestration."

4. **ServiceRequest narrative — cross-link the same two examples.**
   - File: `source/servicerequest/servicerequest-introduction.xml`.
   - Lines 41 and 44 already point readers at the Request pattern and
     RequestOrchestration. Append inline "see example" links to the two new
     examples so that a reader on `servicerequest.html` can reach them in one
     click (answers the ticket's "how would we link from the ServiceRequest
     examples …" open question).

5. **R6 terminology cleanup.**
   - The ticket predates the `RequestGroup → RequestOrchestration` rename.
     Any new prose added above must use `RequestOrchestration` (the name
     present in the clone under `source/requestorchestration/` and in
     `source/request/request-spreadsheet-notes.xml`). No references to
     `RequestGroup` should be introduced.

### Design Rationale

- The resolution is explicit that both mechanisms are valid and should each
  get an example. Providing only the RequestOrchestration example would
  mislead readers into thinking orchestration is always required for
  multi-item orders, when `ServiceRequest.requisition` / `groupIdentifier` is
  the simpler and more common pattern (per the ServiceRequest introduction
  and the Request-pattern notes).
- Using `ServiceRequest` for the child resources in both examples keeps the
  artifacts concrete (the ticket was filed against ServiceRequest / Request
  pattern) and re-uses the existing example machinery under
  `source/servicerequest/`.
- Linking from both the Request pattern page and the ServiceRequest page
  directly answers the ticket's open question about discoverability, without
  introducing a new specification section.
- RequestOrchestration's `option`-intent rule (notes lines 135–142) is
  non-obvious; a worked example is the cheapest way to teach it and prevents
  implementers from incorrectly combining `option` and `order` intents in one
  orchestration.

3. **Create the child ServiceRequests for the orchestration.**
