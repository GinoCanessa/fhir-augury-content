## Resolution Summary

Resolution: **Persuasive**. The reporter notes that invariant `eld-11` on
`ElementDefinition` ("Binding can only be present for coded elements, string,
and uri if using FHIR-defined types") currently rejects a `binding` whenever
the element's type is `CodeableReference`, even though the `.concept` portion
of `CodeableReference` is a `CodeableConcept` for which a binding clearly
makes sense. The current FHIRPath expression is:

```
binding.empty() or type.code.empty() or type.code.contains(':') or
type.select((code = 'code') or (code = 'Coding') or (code = 'CodeableConcept')
or (code = 'Quantity') or (code = 'string') or (code = 'uri') or
(code = 'Duration')).exists()
```

The persuasive disposition requires that `CodeableReference` be added to the
list of types for which a binding is permitted, so that the invariant no
longer falsely flags `ElementDefinition` instances that bind a value set to
an element typed as `CodeableReference`.

## Feature Proposal

### Problem Statement

`ElementDefinition` carries the FHIR-published invariant `eld-11`, which is
intended to ensure that authors only attach a `binding` to elements whose
type can sensibly carry a coded value (`code`, `Coding`, `CodeableConcept`,
`Quantity`, `string`, `uri`, `Duration`, plus profile-typed elements
recognised via the `:` test). The `CodeableReference` datatype, added in
R5, was omitted from that allow-list. `CodeableReference` is a choice
between a `Reference` and a `CodeableConcept` (its `.concept` element), and
it is routinely used in places where authors do want to bind a value set —
for example, `MedicationRequest.medication`, `Condition.code` profile
points, and similar. Today the invariant prevents such bindings from
validating, forcing authors to either drop the binding or use workarounds
that defeat the purpose of `CodeableReference`.

### Proposed Change

Add `'CodeableReference'` to the list of accepted type codes in invariant
`eld-11` on `ElementDefinition`, in **both** the FHIRPath `expression` and
the legacy XPath `xpath` cells of the invariants table.

**New FHIRPath expression:**

```
binding.empty() or type.code.empty() or type.code.contains(':') or
type.select((code = 'code') or (code = 'Coding') or (code = 'CodeableConcept')
or (code = 'CodeableReference') or (code = 'Quantity') or (code = 'string')
or (code = 'uri') or (code = 'Duration')).exists()
```

**New XPath:**

```
not(exists(f:binding)) or (count(f:type/f:code) = 0)
or contains(f:type/f:code/@value, ':')
or f:type/f:code/@value=('code','Coding','CodeableConcept',
'CodeableReference','Quantity','string','uri','Duration')
```

Human-readable label ("Binding can only be present for coded elements,
string, and uri if using FHIR-defined types") does not need to change; the
adjective "coded" already covers `CodeableReference` once it is allow-listed.

No new elements, types, extensions, search parameters, terminology
artifacts, or operations are introduced.

### Design Rationale

- `CodeableReference` was introduced in R5 specifically so that an element
  could carry either a coded concept *or* a typed reference. Its `.concept`
  child is a `CodeableConcept`, so any binding placed on the parent element
  is interpreted as binding the `.concept` value — exactly the same rule
  that already applies to `CodeableConcept`.
- Sibling invariant on row 4914 of the same file
  (`type.where(code in 'code' | 'Coding' | 'CodeableConcept' |
  'CodeableReference' | 'uri' | 'string' | 'Quantity').count() > 1
  implies binding.empty()`) **already** treats `CodeableReference` as a
  bindable coded type when checking for ambiguous multi-type bindings. The
  fix to `eld-11` brings the two invariants into alignment — without it,
  the spec contradicts itself by allowing a multi-type element to mix
  `CodeableReference` with other coded types but rejecting a single-type
  `CodeableReference` element with a binding.
- The XPath expression is kept in sync with the FHIRPath expression because
  the bespoke HL7 publisher emits both into the generated StructureDefinition
  snapshot; downstream validators that fall back to XPath would otherwise
  reject the binding even after the FHIRPath update.
- Alternative considered: defining a separate `CodeableReference`-aware
  invariant. Rejected because it would duplicate the type allow-list and
  obscure the intent; the original invariant's allow-list is the natural
  place to capture the additional type.

- **Direct regression:** craft an `ElementDefinition` snippet with
  `type.code='CodeableReference'` and a `binding.valueSet` populated; verify
