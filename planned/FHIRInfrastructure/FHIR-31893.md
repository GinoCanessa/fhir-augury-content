In practice the resolution is a **process/governance commitment** rather than a single normative change to the specification. The MnM (Modeling and Methodology) work group — and per a 2025-10-11 comment by Bryn Rhodes, possibly Bryn personally with FHIR Infrastructure — agrees to:

1. Review every `Required` and `Extensible` ValueSet binding declared on elements of resources in the core FHIR specification.
2. Score each binding against the criteria in the *Guide to Designing Resources → Vocabulary* section (binding strength must match how stable and complete the value set is, what implementer freedom is intended, etc.).

## Feature Proposal

### Problem Statement

`Required` and `Extensible` are the two strongest FHIR binding strengths. `Required` means "implementations must use a code from this value set"; `Extensible` means "must use a code from this value set if a suitable concept exists". The *Guide to Designing Resources → Vocabulary* page sets out when each strength is appropriate (stability of the value set, completeness, expected interoperability impact, implementer burden, etc.).

Over many ballot cycles individual resources have been authored by different work groups at different times. Bindings have been chosen ticket-by-ticket without a single, comprehensive cross-resource review against the current Vocabulary guidance. The reporter's concern is that some `Required`/`Extensible` bindings in the core resources no longer satisfy the design rules — for example, value sets that are too small to be `Required`, value sets where extension is expected but the binding is `Required`, or stable value sets that could legitimately be tightened from `Extensible` to `Required`.

A pair of comments confirms scope and approach:

- **bryn.rhodes (2025-10-11):** *"I have scripting that has done this analysis for the current specification and am in the process of identifying recommendations here. I'm happy for this to be assigned to me if MnM is okay with that?"* — automation already exists.
- **ron_shapiro (2021-04-28):** records the formal approval vote.

The cross-reference index returns **no** linked items (`cross-referenced` returned `total: 0`), so no follow-up tickets have yet been logged that we can see from this side; the survey is still in flight.

### Proposed Change

Carry out a one-shot review and produce per-binding recommendations, then file targeted child tickets. Concretely:

1. **Enumerate every `Required` and `Extensible` binding in the core specification.**
   - Source: every `<binding strength="required"/>` and `<binding strength="extensible"/>` declaration in `<differential>` blocks of `source/<resource>/structuredefinition-<Resource>.xml` (and equivalent profile/data-type StructureDefinitions under `source/`).
   - Capture: resource name, element path, binding strength, ValueSet canonical URL, current owning work group of the resource.

2. **Score each binding against the Vocabulary guidance.** For each row produce one of:
   - `OK` — binding strength is appropriate; no action.
   - `Loosen` — should be `Extensible` (was `Required`) or `Preferred` (was `Extensible`) because the value set is incomplete / will need extension.
   - `Tighten` — could safely become `Required` (was `Extensible`) because the value set is stable and complete and interop demands it.
   - `Wrong-ValueSet` — strength is fine but the bound ValueSet is the wrong content.
   - `Needs-discussion` — automated scoring is ambiguous.

3. **For every non-`OK` row, create one child Jira ticket** in the resource's owning work group, of type *Change Request*, linked back to FHIR-31893 ("relates to"). The child ticket carries the concrete element + binding fix. This ticket is the parent/tracker.

4. **Publish the survey artefacts** (CSV / table of every binding scored, plus the recipes used) in a location MnM/FHIR-I can reference — at minimum attach the CSV to FHIR-31893; preferably also link from a Confluence page under the Vocabulary guide so future editors can see the rationale.

5. **Close FHIR-31893** once the survey CSV is published and the child tickets are filed. Track resolution progress via the linked children, not via this ticket.

No direct edits to `source/<resource>/structuredefinition-*.xml` are made under this ticket; those happen under the per-resource child tickets that the survey produces.

### Design Rationale

- The resolution explicitly delegates the *fixes* to individual work groups via new tickets ("create new Jira issues for workgroups to fix any bindings that are deemed incorrect"). Treating FHIR-31893 as a tracker rather than a code change keeps each binding change reviewable by its owning work group, which is the body authoritatively responsible for that resource's vocabulary.
- Bryn Rhodes' 2025-10-11 offer of existing scripted analysis is the natural execution path: scripted enumeration is the only realistic way to cover every `Required`/`Extensible` binding across all core resources, and a script is reproducible across publication cycles.
- A persistent CSV (or equivalent table) makes the survey auditable; future tickets can be checked against the same baseline.
- The label `Deferred` and the four-year gap between resolution (2021) and the recent Bryn comment (2025-10-11) suggest this has been waiting for someone to actually run the scan — assigning to whoever owns the script (Bryn) and treating the deliverable as "filed child tickets" turns it from an indefinite intent into a closeable task.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Source of every `Required`/`Extensible` binding declaration to be surveyed. Also the host of any optional published artefact (e.g., a survey CSV attached to the ticket or committed under `source/` as a working document — most likely **not** committed; attached to Jira instead).
- **Affected Files:**

- **Change Scope:** Minor *for this tracker ticket* (no code change). Cumulative scope across child tickets is potentially **Major** for the specification, since binding-strength changes are conformance-affecting (see Breaking Changes).

### Breaking Changes

This tracker ticket itself introduces no breaking change. However, the **child tickets** it spawns can — by definition — introduce breaking changes for implementers, depending on direction:

- **Tightening `Extensible` → `Required`** is breaking for any implementation that today sends a code outside the bound ValueSet; previously legal instances become non-conformant.
- **Loosening `Required` → `Extensible`** is breaking for consumers that today rely on receiving only codes from the bound ValueSet (they must now tolerate additional codes).
- **Changing the bound ValueSet** at `Required` strength is breaking even if strength is unchanged.

Each child ticket must call out its own breaking-change profile; the survey CSV should include a "breaking?" column to make this visible up front.

### Related Specifications

- **HL7/UTG** — defines the ValueSet/CodeSystem content backing most core bindings; binding-strength judgements depend on whether the value set is stable and complete.
- **HL7/fhir-extensions** — alternative path when an enumerated binding is judged inappropriate; an extension may be the right answer instead.
- **All FHIR Core resources and complex data types** — every resource StructureDefinition is in scope of the survey.
- **FHIR specification page:** *Guide to Designing Resources → Vocabulary* (Confluence) — the rubric the survey applies.

### Related Tickets

`cross-referenced` returned `total: 0` for FHIR-31893: there are no linked Jira tickets, GitHub items, Confluence pages, or Zulip threads in the cache. None identified.

### Terminology Impact

Indirect but pervasive. The survey will identify cases where:

- A bound ValueSet is incomplete / unstable enough that the binding strength must be loosened.
- A bound ValueSet must be **replaced** (point at a different UTG ValueSet).
- A new ValueSet needs to exist in UTG to support a tightening recommendation.

All concrete UTG changes happen under the child tickets, not under FHIR-31893. The survey should still tag every row with the bound ValueSet's canonical URL so the UTG impact is computable from the CSV.

### Step-by-Step Tasks

#### HL7/fhir

1. **Enumerate the survey universe.**
   - File(s): all `source/<resource>/structuredefinition-*.xml` reachable from the publisher's resource list (driven from `source/fhir.ini`).
   - Action: produce a list of every StructureDefinition that ships in core; exclude `implementations/`, `tests/`, `tools/`, `qa/`, `schema/`, and any non-core IG content.
   - Details: use the publisher's resource list (preferred) or, as fallback, every `source/<resource>/structuredefinition-<Resource>.xml`. Treat the `<differential>` block as the source of truth for bindings; do **not** read from `<snapshot>`. Skip `source/datatypes/<type>.xml` files that are actually Excel spreadsheets — handle real-XML data-type StructureDefinitions only.

2. **Extract every `Required` / `Extensible` binding.**
   - File(s): the StructureDefinitions enumerated in task 1 (read-only).
   - Action: for each `element` whose `binding/strength` is `required` or `extensible`, emit a row containing: `resource`, `elementPath`, `strength`, `valueSetCanonical` (with version if pinned), `bindingDescription` (if present), and the `workGroup` extension on the StructureDefinition (`http://hl7.org/fhir/StructureDefinition/structuredefinition-wg`).
   - Details: this is the input table for scoring. Bryn's existing script per his 2025-10-11 comment already does the analysis; align its output to this column set.

3. **Score each row against the Vocabulary rubric.**
   - File: survey CSV (working artefact under MnM / FHIR-I control; not committed under `source/`).
   - Action: assign each row one of `OK`, `Loosen`, `Tighten`, `Wrong-ValueSet`, `Needs-discussion`, with a one-sentence rationale citing the relevant rule from the *Guide to Designing Resources → Vocabulary* page.
   - Details: also flag whether the recommended change is breaking (see Breaking Changes) and whether a UTG change is a prerequisite.

4. **File a child Jira ticket for every non-`OK` row.**
   - Action: create one *Change Request* per row in the **owning work group** of the resource (taken from the `structuredefinition-wg` extension), titled e.g. `<Resource>.<elementPath>: <Strength> binding to <ValueSet> — <Loosen|Tighten|Wrong-ValueSet|Discuss>`. Body must state the current binding, the recommended binding, the rationale, and the breaking-change assessment. Link `relates to` → FHIR-31893.
   - Details: do **not** bundle multiple resources into one ticket; per-element granularity matches how the work groups will action them. Prefix the description with a pointer to the published survey CSV so the rationale is auditable.

5. **Publish the survey CSV and close FHIR-31893.**
   - Action: attach the final scored CSV to FHIR-31893 (and, ideally, link from the Vocabulary section of the *Guide to Designing Resources* Confluence page so future editors find it). Add a closing comment listing every child ticket key. Resolve / close FHIR-31893; child tickets carry the actual fixes from here.
   - Details: at this point this ticket's deliverable is complete — closure is **not** contingent on the children being resolved.

### Testing Considerations

- "Testing" here is reproducibility of the survey, not run-time behaviour:
  - Re-run Bryn's script against the same publisher tag and confirm the row count and contents match the published CSV exactly.
  - Spot-check 5–10 randomly chosen rows by hand against the StructureDefinition `<differential>` to verify the script reads the right element/binding.
  - Verify ownership: for every row, confirm the work-group attribution matches the `structuredefinition-wg` extension on the source StructureDefinition.
  - Confirm no in-scope StructureDefinition was missed by diffing the script's resource list against `source/fhir.ini`'s resource list.
- Implementation impact (for the eventual child tickets, not for FHIR-31893 itself): each binding-strength change should be re-validated by the publisher and any example instances under the affected resource re-checked for conformance.
