## Resolution Summary

The ticket was disposed **Persuasive** with no expanded resolution text and
no applied-vote comments captured in the source. The reported defect, in
full, is:

> If you click on HumanName, it doesn't jump

In context, this concerns the FHIR Core specification page that documents
the general-purpose datatypes. The expected behavior is that a link to
`datatypes.html#HumanName` (used widely from resource diagrams,
`datatypes-definitions.html`, the `search.html` page, the iso-21090
profile introduction, the per-resource SVG diagrams, etc.) jumps the
browser to the HumanName section of `source/datatypes.html`. The Persuasive
disposition accepts that this jump is broken in the published spec and
agrees a fix is required.

Because no implementation detail was recorded with the disposition, the
plan below executes the documentation-fix recipe and explicitly calls out
the diagnostic step (reproduce on the published page, identify whether the
failure is anchor-masking, the jQuery-UI tabs widget instantiated for
`#tabs-HumanName`, or a publisher template difference) before applying a
correction.

## Feature Proposal

### Problem Statement

In the published FHIR Core specification, links of the form
`datatypes.html#HumanName` (and `datatypes.html#humanname`) do not scroll
the browser to the HumanName section of the Datatypes page. Many places
in the spec link to that anchor:

- `source/search.html` — multiple references (lines ~898, 2746, 3416,
  3471, 6787) use both the `#HumanName` and `#humanname` casings.
- `source/history.html` — change-history bullets at ~619, 639.
- `source/profiles/iso-21090-introduction.xml` — extension introduction
  text at lines 47, 53.
- Per-resource SVG diagrams that link types to their definitions, e.g.
  `source/patient/patient.svg` (lines 133, 323),
  `source/relatedperson/relatedperson.svg` (line 187),
  `source/task/task.svg` (lines 850, 1044).
- Sibling datatype pages
  (`source/datatypes-definitions.html:172`,
  `source/datatypes-examples.html:1188`,
  `source/datatypes-mappings.html:187`,
  `source/datatypes-profiles.html:220`) that cross-link back to the base
  definition.

Failing this jump degrades usability for every reader who follows a
HumanName cross-reference. Peer datatype anchors on the same page
(`#Identifier`, `#Address`, `#ContactPoint`, etc.) work, so HumanName is
the outlier.

### Proposed Change

Restore correct anchor behavior for `datatypes.html#HumanName` (and the
lowercase `#humanname` alias). The exact source-side change depends on
which root cause the diagnostic step (Step 1 below) confirms; the three
candidates that source inspection cannot rule out are:

1. **Anchor-on-`<h2>` modernization.** Move the anchor from the legacy
   pre-element `<a name="…"></a>` form into an `id` attribute on the
   heading, mirroring whatever the publisher emits for Identifier /
   Address / ContactPoint. The current source at
   `source/datatypes.html:1441-1443` is:

   ```html
   <a name="HumanName"></a>
   <a name="humanname"></a>
   <h2 style="margin-bottom: 1px">HumanName</h2>
   ```

   Replace with (preserving both casings as ids for backward
   compatibility with existing inbound links — see the search/history/SVG
   list above):

   ```html
   <h2 id="HumanName" style="margin-bottom: 1px">HumanName<a id="humanname"></a></h2>
   ```

   or equivalent (e.g., heading `id="HumanName"` plus a sibling `<span
   id="humanname"></span>` immediately before the heading). The exact
   shape should match whichever pattern Identifier / Address /
   ContactPoint use after the publisher runs, so the rendered DOM is
   uniform.

2. **Tab-widget interference.** The page contains
   `$( '#tabs-HumanName' ).tabs(...)` (lines 1904 and 1929 of
   `source/datatypes.html`). If the jQuery-UI tabs init consumes the
   URL fragment for HumanName specifically (e.g., because the publisher
   emits a `#tabs-HumanName` element whose id collides with or is
   recomputed from the `HumanName` fragment, or because the tabs init
   resets `window.location.hash`), the fix is to call the tabs init
   without disturbing the existing fragment (e.g., guard the
   `store(ui.newTab.index())` call so it does not run on initial load).

3. **Publisher-template asymmetry.** The `[%dt HumanName 10%]` macro on
   line 1460 produces the tabbed details panel. Other types use the same
   macro pattern (`[%dt Identifier 9%]`, `[%dt Address 11%]`, etc.), so
   if HumanName-specific output differs, the fix is in the publisher
   template, not the source page. In that case the change has to be made
   in the FHIR publisher (out-of-tree repository), and this ticket's
   `HL7/fhir` clone change is limited to whatever follow-up is required
   to consume the publisher fix.

The implementation plan in §Implementation Plan walks through these
options in order; whichever the diagnostic confirms is the one that gets
applied.

### Design Rationale

- HumanName is a public, deeply-linked anchor target. Many inbound links
  use the mixed-case form `#HumanName` and a few use the lowercase
  `#humanname` (search.html lines 2746, 3416; iso-21090 intro lines 47,
  53). Both must continue to resolve, so the fix must keep both spellings
  as valid in-page targets.
- Aligning the HumanName anchor markup with the Identifier / Address /
  ContactPoint pattern follows the existing convention on the page rather
  than introducing a new one — the Repo Briefing's recipe is "edit the
  authored HTML to match the prevailing pattern", not "rework the
  publisher".
- Modifying client-side JavaScript (option 2) is preferred only if
  diagnosis shows the tabs widget is consuming the fragment, because
  publisher behavior changes affect every datatype page.
