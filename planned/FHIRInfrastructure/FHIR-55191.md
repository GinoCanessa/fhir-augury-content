## Resolution Summary

Resolution type: **Not Persuasive with Modification**.

The committee declined the reporter's request to add an `unknown` / `other` /
escape-valve concept to the `structure-definition-kind` value set. Instead,
per the resolution description, the agreed modification is to add a **comment**
to the `StructureDefinition.kind` element with the exact guidance:

> "If pre-adopting a new 'kind' from a future version, set the type to
> 'logical' and specify the details of the new kind as an extension."

This documents the existing escape-valve mechanism (use `logical` plus an
extension to describe the new kind) rather than expanding the required-bound
value set.

## Feature Proposal

### Problem Statement

`StructureDefinition.kind` is a mandatory (1..1) `code` element with a
**required** binding to
`http://hl7.org/fhir/ValueSet/structure-definition-kind`, whose concepts are
only `primitive-type | complex-type | resource | logical` (see
`source/structuredefinition/codesystem-structure-definition-kind.xml` and
`source/structuredefinition/valueset-structure-definition-kind.xml`). Because
the binding is required with no `unknown` / `other` concept, an implementer
who needs to represent a StructureDefinition for a newly-defined kind (for
example, one being introduced in a later version of FHIR) has no conformant
way to populate `kind` while staying within the current value set.

### Proposed Change

Add a `<comment>` child to the `StructureDefinition.kind` ElementDefinition in
the core differential. Per standard FHIR ElementDefinition ordering, the
`<comment>` appears immediately after `<definition>` and before `<min>`.

Exact comment text (from the resolution description):

```
If pre-adopting a new 'kind' from a future version, set the type to 'logical'
and specify the details of the new kind as an extension.
```

Resulting fragment (only the new line is added; surrounding lines shown for
context — do **not** touch `<snapshot>` content, which is regenerated):

```xml
<element id="StructureDefinition.kind">
    <path value="StructureDefinition.kind"/>
    <short value="primitive-type | complex-type | resource | logical"/>
    <definition value="Defines the kind of structure that this definition is describing."/>
    <comment value="If pre-adopting a new 'kind' from a future version, set the type to 'logical' and specify the details of the new kind as an extension."/>
    <min value="1"/>
    <max value="1"/>
    ...
```

No change to cardinality, type, binding, conditions (`sdf-3`, `sdf-8`,
`sdf-8a`, `sdf-11`, `sdf-15`, `sdf-15a`, `sdf-29`), `isSummary`, or the bound
value set. No change to the `structure-definition-kind` CodeSystem/ValueSet.

### Design Rationale

- The resolution explicitly rejects the reporter's proposal to widen the
  value set ("Not Persuasive with Modification"), so adding `unknown` /
  `other` to `codesystem-structure-definition-kind` is out of scope.
- The recommended workaround — `kind = logical` with an extension describing
  the real kind — already fits within existing FHIR conformance rules, so a
  narrative `<comment>` on the element is sufficient; no structural or
  terminology changes are needed.
- Placement as `<comment>` (rather than extending `<definition>`) follows
  existing usage in the same file — e.g., `StructureDefinition.abstract` uses
  `<comment>` for similar "how to use this in practice" guidance.
