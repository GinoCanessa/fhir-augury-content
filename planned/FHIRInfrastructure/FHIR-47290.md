## Resolution Summary

**Resolution:** Persuasive — "Update description as requested."

Specifically, in the `$get-ws-binding-token` operation on `Subscription`, the
`id` IN parameter's `documentation` value must be reworded from:

> "At the Instance level, this parameter is ignored. At the Resource level, one or more parameters containing **a FHIR id** for a Subscription to get a token for. In the absence of any specified ids, the server may either return a token for all Subscriptions available to the caller with a channel-type of websocket or fail the request."

to:

> "At the Instance level, this parameter is ignored. At the Resource level, one or more parameters containing **the logical id** for **the** Subscription to get a token for. In the absence of any specified ids, the server may either return a token for all Subscriptions available to the caller with a channel-type of websocket or fail the request."

The substantive change is replacing the ambiguous phrase "a FHIR id" with the
precise FHIR-spec term "the logical id" (the `Resource.id` value), and
tightening "a Subscription" to "the Subscription". The resolution was approved
as part of the 2025-03-31 FHIR Infrastructure CC block vote (Confluence page
`325460673`).

The later ticket comment (ginocanessa, 2025-11-04) adds the `MovingToGuide`
label context: for FHIR R6, the `$get-ws-binding-token` OperationDefinition
and its surrounding WebSocket details are moving out of FHIR Core into an
external guide; the text fix must land in whatever guide ultimately hosts the
operation for R6.

## Feature Proposal

### Problem Statement

The `id` IN parameter of `Subscription/$get-ws-binding-token` used the term
"a FHIR id", which is not a defined FHIR term. FHIR defines `Resource.id`
as the **logical id** of a resource. Callers reading the OperationDefinition
cannot tell whether "a FHIR id" means the logical id (`Resource.id`), a full
literal reference, a business identifier (`Subscription.identifier`), or
something else — which is precisely the ambiguity this operation's callers
hit in practice for websocket subscription binding.

The reporter (Bas van den Heuvel) proposed the precise wording; FHIR
Infrastructure accepted the change on 2025-03-31 as Persuasive.

### Proposed Change

Edit the `documentation` attribute of the `id` IN parameter of the
`Subscription-get-ws-binding-token` OperationDefinition so that it reads:

> "At the Instance level, this parameter is ignored. At the Resource level, one or more parameters containing the logical id for the Subscription to get a token for. In the absence of any specified ids, the server may either return a token for all Subscriptions available to the caller with a channel-type of websocket or fail the request."

Concretely, two substitutions on that single parameter's `documentation`
string:

1. "a FHIR id for a Subscription" → "the logical id for the Subscription"

No other attributes of the OperationDefinition change: the parameter's
`name` (`id`), `use` (`in`), `type` (`id`), `min`/`max` cardinality, and
canonical URL are unchanged. No structural or cardinality change; this is a
pure documentation edit.

### Design Rationale

- Uses FHIR-defined vocabulary. `Resource.id` is formally defined as the
  *logical id* of the resource; it is the correct and unambiguous referent
  here, since the parameter's `type` is already `id` (not `Reference` or
  `Identifier`).
- The wording pattern matches how other FHIR OperationDefinitions describe
  `id`-typed parameters.
- The change is Persuasive (documentation clarification), so it should not
  alter behaviour for any conforming server or client — no breaking change.
- Alternatives considered (implicit in the Jira thread): leaving the text
  unchanged (rejected — ambiguous), or replacing with a `Reference(Subscription)`
  parameter (not considered — would be a breaking type change and is out of
  scope of a Persuasive documentation edit).
