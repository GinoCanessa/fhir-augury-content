## Resolution Summary

The ticket is resolved **Persuasive with Modification**. The committee agreed
that the two ExampleScenario example narratives
(`examplescenario-example` and `examplescenario-example-laborder`) should not
rely on a PNG raster for their primary diagram — an SVG would be clearer and
accessible to visually-impaired readers. However, the immediate fix could not
be applied because the bespoke FHIR Core publisher (**Kindling**) did not
accept an SVG embedded in the ExampleScenario narrative `<text>/<div>` block.

The resolution description says verbatim:

> "We went with a PNG because the tooling didn't like having an SVG. We will
> surface a tooling issue with Kindling and see if we can get the issue
> resolved."

The ticket carries the `defer` label, confirming that the in-spec PNG → SVG
swap in `HL7/fhir` is gated on a tooling fix in `HL7/kindling`. The
implementation therefore has two parts:

1. File / surface a Kindling tooling issue so that SVG in ExampleScenario
   narratives round-trips through the publisher.
2. Once Kindling accepts SVG in narrative, replace the two PNGs with
   SVG equivalents in `HL7/fhir` and delete the PNG assets.

## Feature Proposal

### Problem Statement

`ExampleScenario` is an ideal resource for diagram-driven documentation: each
example scenario is, by its nature, a sequence/interaction diagram. The two
canonical examples published on hl7.org are
`examplescenario-example` (IHE Mobile Medication Administration) and
`examplescenario-example-laborder` (Lab order tracking with Task).

Inspection of the clone confirms the narrative in both files is almost
entirely a single raster image:

- `source/examplescenario/examplescenario-example.xml` line 22:
  `<img src="./examplescenario-example.png" alt="ExampleScenario diagram - IHE MMA example"/>`
- `source/examplescenario/examplescenario-example-laborder.xml` line 22:
  `<img src="./examplescenario-example-laborder.png" alt="ExampleScenario diagram - IHE MMA example"/>`
- PNG assets present alongside the narrative:
  - `source/examplescenario/examplescenario-example.png` (43,231 bytes)
  - `source/examplescenario/examplescenario-example-laborder.png` (86,531 bytes)

Rasterised diagrams have three concrete problems here:

1. **Accessibility.** The diagram is the entire content; screen readers get
   only the `alt` text, and visually-impaired users cannot interrogate the
   sequence (actors, messages, timing).
2. **Legibility.** PNGs do not scale; zoom degrades, and the spec is read at
   many device sizes.
3. **Authoring.** The PlantUML sources already exist in the repo
   (`examplescenario-example.plantuml`,
   `examplescenario-example-laborder.plantuml`). Today the PNG is the
   committed rendering of those `.plantuml` files, i.e. the committed
   artefact is downstream of an authoring artefact that can also emit SVG.

### Proposed Change

Deliver the change in two sequenced deliverables mirroring the resolution:

**Deliverable A — Kindling tooling fix (blocking prerequisite).**

File an issue against the Kindling publisher (`HL7/kindling`, not cached
locally) describing the observed behaviour:

- Title: "ExampleScenario narrative rejects inline/referenced SVG diagrams"
- Body:
  - Symptom: when the narrative `<text>/<div>` of an ExampleScenario (or the
    `.svg` referenced via `<img src="…">`) is SVG rather than PNG, Kindling
    errors / mangles / strips the content (exact symptom to be captured
    when the author reproduces the failure).
  - Reference context: FHIR-43904 (this ticket), plus the two pages above.
  - Expected: narrative authors can use SVG (either inline `<svg>` or
    `<img src="*.svg">`) in ExampleScenario `<text>` exactly as they can
    elsewhere in the spec — `source/<resource>/<resource>.svg` files are
    already consumed successfully for most resources (see e.g.
    `source/patient/patient.svg`, `source/observation/observation.svg`),
    so the defect is specific to either
    (a) the `.svg` extension in the ExampleScenario narrative folder, or
    (b) inline `<svg>` in an xhtml narrative.
- Acceptance: once Kindling passes an SVG narrative build, close the tooling
  issue and unblock Deliverable B.

**Deliverable B — PNG → SVG swap in `HL7/fhir` (unblocked by A).**

For each of the two example scenarios:

1. Render an SVG from the existing `.plantuml` source, committing it next to
   the XML:
   - `source/examplescenario/examplescenario-example.svg` from
     `source/examplescenario/examplescenario-example.plantuml`
   - `source/examplescenario/examplescenario-example-laborder.svg` from
     `source/examplescenario/examplescenario-example-laborder.plantuml`
   - Use PlantUML's SVG output (`plantuml -tsvg <file>.plantuml`) so the SVG
     is a faithful re-render of the existing PNG.
2. Update the narrative `<img>` tag in each XML to point at the new SVG and
   keep descriptive `alt` text (and fix the laborder `alt` which is currently
   "IHE MMA example" — see "Open Questions"):
   - In `examplescenario-example.xml` line 22: change
     `src="./examplescenario-example.png"` to
     `src="./examplescenario-example.svg"`.
   - In `examplescenario-example-laborder.xml` line 22: change
     `src="./examplescenario-example-laborder.png"` to
     `src="./examplescenario-example-laborder.svg"` and update the `alt`
     attribute to describe the lab-order scenario.
3. Delete the superseded raster assets so they do not drift out of sync:
   - `source/examplescenario/examplescenario-example.png`
   - `source/examplescenario/examplescenario-example-laborder.png`
4. Leave the `.plantuml` sources in place — they remain the diagram
   authoring artefact.

Optional (recommended) accessibility polish inside each narrative `<div>`:
add a short textual description of the interaction before/after the `<img>`
so that screen-reader users still get the narrative even if the SVG's inner
`<title>`/`<desc>` elements are not yet populated. This is low-risk HTML-only
content consistent with the publisher's handling of narrative `<div>`.

### Design Rationale

- **Consistency with existing spec practice.** The repo already carries
  hundreds of `source/<resource>/<resource>.svg` files (confirmed for
  `account`, `activitydefinition`, `adverseevent`, `allergyintolerance`,
  `appointment`, `artifactassessment`, `auditevent`, and many more). SVG is
  the default diagram format for FHIR Core; PNGs in ExampleScenario are the
  anomaly, not a norm.
- **Authoring source already exists.** Both PlantUML sources are committed
  alongside the PNG. PlantUML natively emits SVG, so the change re-renders
  the same authoring artefact in an additional format rather than introducing
  a new editorial source.
- **Accessibility is a first-class FHIR concern.** SVG exposes text nodes to
  assistive technology; PNG does not. The ticket explicitly calls this out.
- **Two-step sequencing is forced by the resolution.** The committee did not
  dispute the technical merit — they deferred the in-spec change to avoid
  breaking the publisher. That phrasing (reflected in the `defer` label) is
  the direct reason the plan has a blocking Kindling prerequisite rather
  than a one-PR swap.

## Implementation Plan

### Prerequisites

1. **Kindling accepts SVG in ExampleScenario narrative.** This is the
   blocking prerequisite captured by the resolution and the `defer` label.
   Deliverable B cannot merge until a Kindling build succeeds with an SVG
   narrative. Filing the Kindling issue (Task 1 below) is the first action.
2. **PlantUML available to re-render diagrams.** The SVG rendering uses
   the existing `.plantuml` authoring sources, so the author's
   environment needs a working `plantuml` (Graphviz-backed) CLI. No repo
   change is needed for this — PlantUML is a local dev tool.

### Testing Considerations

- **Publisher regression.** After the Kindling fix, run a full spec
  build locally and confirm that the two ExampleScenario pages publish
  with the SVG embedded, and that no other ExampleScenario-adjacent
  pages regress.
- **Screen-reader smoke test.** Open each generated

  announced, not just the `alt` attribute.
- **High-DPI / zoom legibility.** Confirm both SVGs remain legible at

  calls out.
- **Link integrity.** Re-run the spec's broken-link / asset-reference
  QA to confirm no page (inside or outside `source/examplescenario/`)

- **Interoperability.** `ExampleScenario.text.div` is narrative; no
  validator or profile changes. Existing validators consuming these
  instances must continue to accept them — spot-check by running the
  XML through the FHIR Validator after the narrative change.

### Open Questions

- **Kindling defect specifics.** The resolution tells us "the tooling
  didn't like having an SVG" but not the exact failure mode
  (parse-time rejection? narrative sanitiser stripping `<svg>` or
  `<img src="*.svg">`? publisher asset-copier not including `.svg`
  files from the ExampleScenario folder?). The Kindling issue (Task 1)
  must capture the exact symptom when reproduced; the HL7/fhir PR
  (Tasks 2–7) cannot merge until we know the fix covers the real
  failure mode.
- **Inline `<svg>` vs `<img src="*.svg">`.** The plan uses
  `<img src="*.svg">` because it is the minimal delta from today's
  PNG-via-`<img>` narrative and keeps the `.svg` as a separate
  re-renderable file. If the Kindling maintainers prefer inlining
  `<svg>` directly into the narrative (for single-file portability),
  Tasks 4–5 change shape — the `.svg` file would not be committed and
  its contents would be embedded into the XML `<div>`. Resolve with
  the Kindling issue.
- **`HL7/kindling` briefing not cached.** No `repo-analysis` briefing
  exists for `HL7/kindling` in this cache (the repo is not cloned
  under `cache/github/repos/`). Per the `ticket-plan` skill's rules,
  the plan does not invent Kindling file paths — the Kindling task is
  scoped as "file an issue" with reproduction detail, and any
  code-level Kindling work is left to that repo's maintainers once the
  issue is triaged.
- **Accessibility augmentation.** Should the narrative gain a short
  textual summary of the interaction (actors + flow) in addition to
  the SVG, so that accessibility does not depend entirely on
  PlantUML's `<desc>` / `<title>` emission? Optional but recommended;
  not blocking.
