## Resolution Summary

**Resolution:** Persuasive.

The approved change is to add a new concept to the
`http://terminology.hl7.org/CodeSystem/usage-context-type` CodeSystem — which
lives in HL7 Terminology (THO), maintained in `HL7/UTG` — through a UTG
request:

- **code:** `topic`
- **display:** `Topic`
- **definition (as worded in the resolution):** "A keyword describing a topic
  addressed by or that forms part of the scope of the resource."

Because the bound ValueSet `http://terminology.hl7.org/ValueSet/usage-context-type`
includes the entire CodeSystem (`<include><system>…/usage-context-type</system></include>`
with no explicit code list), the new concept will automatically appear in the
ValueSet expansion. No FHIR Core (HL7/fhir) edits are required by the
resolution itself.

A 2025-10-11 follow-up comment from `bryn.rhodes` notes that the code is
"now in the UsageContextType code system in THO", and inspection of
`HL7/UTG` clone HEAD confirms the `<concept code="topic"/>` entry is already
present in `input/sourceOfTruth/fhir/codeSystems/CodeSystem-usage-context-type.xml`.
This plan therefore covers (a) the original resolution actions and

is marked Applied.

## Feature Proposal

### Problem Statement

FHIR-38841 ("Remove Metadata Topic and recommend using Canonical UsageContext
property", resolved Persuasive with Modification in 2023) committed to
defining a dedicated `topic` concept in the UsageContextType CodeSystem so
that authors describing canonical resource metadata could express "topic"
through `useContext` rather than via a separate `topic` element. However,
the agreed `topic` code was never actually added to
`http://terminology.hl7.org/CodeSystem/usage-context-type`.

This leaves the CodeSystem missing one of the most commonly needed usage
contexts: a generic "this artifact is about <topic>" classifier. Resource
authors currently have to overload `focus`, `program`, or `workflow`, none
of which carry the intended general "topic / keyword" semantics, which in
turn defeats the FHIR-38841 plan to use UsageContext in place of a separate
`Metadata.topic` element.

### Proposed Change

Add a single new `<concept>` to the FHIR-flavored `CodeSystem-usage-context-type`
in HL7/UTG.

- **System:** `http://terminology.hl7.org/CodeSystem/usage-context-type`
- **Code:** `topic`
- **Display:** `Topic`
- **Definition:** A keyword describing a topic addressed by or that forms
  part of the scope of the resource.

Concrete edit to make in `HL7/UTG`:

`input/sourceOfTruth/fhir/codeSystems/CodeSystem-usage-context-type.xml` —
append, after the `jurisdiction` concept (the existing last entry in the
file), the following block:

```xml
  <concept>
    <code value="topic"/>
    <display value="Topic"/>
    <definition value="A keyword describing a topic addressed by or that forms part of the scope of the resource."/>
  </concept>
```

Bump CodeSystem `version` from the current `2.0.1` to the next THO
publication's version, and update `meta.lastUpdated` and `date` to the
release date for that THO publication, per the UTG project's normal
versioning rules.

No change is required in
`input/sourceOfTruth/fhir/valueSets/ValueSet-usage-context-type.xml`, because
its `compose.include` references the CodeSystem as a whole (no explicit
`concept` enumeration), so the new code will appear automatically in the
ValueSet expansion. The ValueSet `version` may still be incremented as part
of the THO release if UTG project conventions require it.

Example usage of the new concept once published:

```xml
<useContext>
  <code>
    <system value="http://terminology.hl7.org/CodeSystem/usage-context-type"/>
    <code value="topic"/>
  </code>
  <valueCodeableConcept>
    <text value="cardiology"/>
  </valueCodeableConcept>
</useContext>
```

### Design Rationale

- **Follows through on FHIR-38841.** That ticket's resolution promised a
  `topic` concept in UsageContextType so that the standalone `Metadata.topic`
  element could be deprecated in favor of `useContext`. FHIR-49931 is the
  missing follow-up.
- **CodeSystem-only change.** The bound ValueSet already includes the whole
  CodeSystem, so no ValueSet maintenance is necessary; this is the smallest
  and least error-prone delta.
- **Routing through UTG / THO.** Per the resolution text "through a UTG
  request", the change must be made in `HL7/UTG`, which is the upstream
  source-of-truth for `terminology.hl7.org` artifacts. Editing FHIR Core
  copies of usage-context-type would be wrong because UTG → THO is the
  authoritative publication chain.
- **Definition wording divergence.** The committed concept in the clone
  uses the FHIR-38841 wording ("Descriptive topics related to the content
  of the artifact. Topics provide a high-level categorization as well as
  keywords for the artifact that can be useful for filtering and
  searching.") rather than the FHIR-49931 resolution wording. See **Open
  Questions** for the recommended reconciliation; the *new-edit* form above
  uses the FHIR-49931 resolution wording verbatim, but reviewers may decide
  to keep the existing FHIR-38841 wording.

### Breaking Changes

No breaking changes identified.

- The change is purely additive: a new code is appended to an existing
  CodeSystem, and the bound ValueSet (which expands the whole CodeSystem)
  gains one extra concept.
- No existing concept's code, display, or definition is altered.
- No binding strength is changed.
- Implementations that today encounter a `topic` code where they previously
  did not should already be tolerant — the binding to UsageContextType in
  Core is `extensible`, so unknown-but-system-valid codes were already
  permitted.

### Related Specifications

- **UsageContext datatype** (FHIR Core) — its `code` element is bound to
  `http://terminology.hl7.org/ValueSet/usage-context-type`; the new `topic`
  code becomes available everywhere UsageContext is used (which is across
  all canonical / metadata resources: `CapabilityStatement`,
  `StructureDefinition`, `ValueSet`, `CodeSystem`, `OperationDefinition`,
  `Library`, `PlanDefinition`, `Measure`, `ActivityDefinition`,
  `Questionnaire`, `ConceptMap`, `NamingSystem`, etc.).
- **Search parameter `context-type`** (and the composite `context-type-value`
  family) on canonical resources — these reference the same ValueSet and
  will accept `topic` once republished, with no parameter-definition
  change required.
- **`Library.topic` / `PlanDefinition.topic` / `Measure.topic`** etc. —
  the existing dedicated `topic` element on knowledge resources is bound to
  a separate ValueSet (`http://hl7.org/fhir/ValueSet/definition-topic`) and
  is **not** affected by this change. FHIR-38841's broader intent (deprecate
  `Metadata.topic` in favor of UsageContext) is *not* in scope here and is
  not required by FHIR-49931's resolution.
