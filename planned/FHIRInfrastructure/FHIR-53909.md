## Resolution Summary

Resolution: **Persuasive**.

Per the ticket's resolution description, the section of `search.html` that deals with the `reference` search type will receive a new note clarifying how the `alternate-reference` extension interacts with reference-type search parameters. The exact approved wording (from the resolution description) is:

> "Instead of Reference.reference, some elements might instead contain the \[additionalReference\]\(\) extension. Systems SHOULD support indexing and searching against content found in this extension as well unless there are constraints in place that ensure that additionalReference is never present."

The `[additionalReference]()` marker in the resolution text is a placeholder for a link to the `alternate-reference` structure definition (the extension is referred to informally as "additionalReference"/"alternate-reference" throughout the spec). No other normative changes are requested.

## Feature Proposal

### Problem Statement

`references.html#additional` (clone: `source/references.html` lines 304–330) defines how the `alternate-reference` extension is used when a `Reference.reference` cannot be populated because the target is not a declared target type — for example, pointing `Observation.basedOn` at an [additional resource](https://build.fhir.org/resource.html#additional). However, the FHIR search specification (`source/search.html`, `#reference` section starting at line 3231) does not currently say anything about how reference-type search parameters behave when the target value lives inside the `alternate-reference` extension instead of in the standard `Reference.reference` element.

The reporter (James Jahns) asks: if `Observation.basedOn` uses the `alternate-reference` extension to point at an additional resource with id `123`, does `GET /Observation?based-on=123` match? Without guidance, server implementations — especially "pure" FHIR servers that drive indexing off `SearchParameter.expression` — will diverge, and IG authors will be forced to define new search parameters for every additional resource. The ticket requests that FHIR Core either (a) spell out that the intuitive matching behavior is expected, or (b) give alternate guidance for additional-resource IGs.

### Proposed Change

Add a single explanatory note inside the `reference` search-type subsection of `source/search.html` (`<a name="reference"></a><h4>reference</h4>` at line 3231). The note states a SHOULD-level expectation that reference search parameters also index/search values carried in the `alternate-reference` extension, unless a profile or constraint guarantees the extension cannot be present.

Concrete change:

- **File:** `source/search.html`
- **Location:** inside the `#reference` subsection, immediately after the introductory paragraph and the list of parameter forms (good candidate: after the "Notes" list that currently ends at line 3257, or as an additional `<li>` within that Notes list). Placing it inside the Notes list keeps the SHOULD with the other SHOULD-level reference-matching rule (line 3254–3255).
- **New prose (exact text from the resolution, with the marker expanded to a real hyperlink to the `alternate-reference` extension):**

  > Instead of `Reference.reference`, some elements might instead contain the [alternate-reference](https://hl7.org/fhir/extensions/5.3.0-ballot-tc1/StructureDefinition-alternate-reference.html) extension (sometimes referred to as "additionalReference"). Systems **SHOULD** support indexing and searching against content found in this extension as well, unless there are constraints in place that ensure the extension is never present.

  Use the same `<span class="fhir-conformance">…</span>` conformance wrapper already used for the existing SHOULD bullet at line 3254–3255 so the note is flagged by the publisher's conformance-highlighting machinery.

- The internal link target should match the pattern already used at `source/references.html` line 320 (the extension link to `StructureDefinition-alternate-reference.html` in the ballot-tc1 extensions IG). If the publisher macros support a short form (e.g., the extension publisher's `[[[ext-name]]]` style), that is preferred over hard-coding the ballot URL; see Open Questions.

No changes to `SearchParameter` resources, FHIRPath expressions, example instances, or terminology are required by the resolution.

### Design Rationale

- **Scope is intentionally narrow.** The resolution approved a *note*, not a normative algorithm or a new search-parameter semantics. It leaves implementation freedom while setting a SHOULD-level expectation, which matches FHIR's usual approach to behaviour that cannot be mechanically enforced from `SearchParameter.expression` alone.
- **Co-locating the note with the existing reference-matching Notes** (line 3251–3257 of `source/search.html`) keeps all reference-search quirks in one place and re-uses the existing `fhir-conformance` span convention.
- **Opt-out via profile constraints** is explicitly preserved — matches the pattern used elsewhere in FHIR ("unless …") and prevents the SHOULD from forcing cost on servers whose profiles already rule the extension out.
- **Alternatives considered (implicit in the ticket):** (a) *requiring* the behavior — rejected as too aggressive for servers driven purely off FHIRPath expressions; (b) *defining per-additional-resource search parameters* — rejected by the reporter as "quickly gets messy"; (c) *doing nothing* — rejected by the Persuasive vote because leaving it ambiguous produces divergent implementations.

### Open Questions

- **Link form for the extension.** The resolution uses the placeholder text `[additionalReference]()`. Does the HL7 publisher support a macro such as `[[[alternate-reference]]]` or similar for extensions-IG references, or should the absolute `https://hl7.org/fhir/extensions/5.3.0-ballot-tc1/StructureDefinition-alternate-reference.html` URL (already used at `source/references.html:320`) be repeated here? Default to the absolute URL unless a maintainer confirms a macro form.
- **Naming.** The extension is formally `alternate-reference`, but the resolution text calls it `additionalReference`. The proposed prose keeps `alternate-reference` as the link label (matching `source/references.html`) and parenthetically notes the `additionalReference` synonym. Confirm this with the Work Group if either name is preferred in isolation.
- **Placement (list vs standalone paragraph).** The plan attaches the note as a new `<li>` in the existing Notes `<ul>` (ending at `source/search.html:3256`) to keep all reference-search SHOULDs together. A standalone `<p>` after the Notes list is an acceptable alternative; choose during review.
