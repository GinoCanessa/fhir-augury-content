## Resolution Summary

The ticket asked for an "escape valve" code (e.g., `unknown`, `other`) to be
added to the value set bound to `CapabilityStatement.rest.resource.searchParam.type`,
because the element is mandatory with a required binding and there are
realistic situations in which none of the existing codes fit (notably,
down-conversion from a future FHIR version that introduces a new search-parameter
type).

The committee disposition is **Not Persuasive with Modification**: the value set
will **not** be expanded with a new code. Instead, a clarifying comment is to be
added to the `searchParam.type` element so that down-conversion implementers
have explicit guidance:

> "If down-converting from a version of FHIR that has additional types, they
> would map to 'special'."

This means the existing `special` code is the agreed-upon escape valve, and the
resolution is realised purely by adding a sentence to the element's `<comment>`
in the CapabilityStatement StructureDefinition. The bound CodeSystem
(`search-param-type`) is not changed.

## Feature Proposal

### Problem Statement

`CapabilityStatement.rest.resource.searchParam.type` has cardinality `1..1`
with a required binding to `http://hl7.org/fhir/ValueSet/search-param-type`
(codes: `number | date | string | token | reference | composite | quantity |
uri | special | resource`). When a server's CapabilityStatement is constructed
by down-converting from a later FHIR version that introduces additional search
parameter types, none of those new codes will be in the current value set, yet
the element must still be populated. Without explicit guidance, implementers
have no canonical way to fill the slot, leading to inconsistent or invalid
CapabilityStatements.

### Proposed Change

Add a sentence to the `<comment>` of the
`CapabilityStatement.rest.resource.searchParam.type` element in
`source/capabilitystatement/structuredefinition-CapabilityStatement.xml`
that points down-converters at the existing `special` code.

Concretely, change the element's `<comment>` from:

> "While this can be looked up from the definition, it is included here as a
> convenience for systems that autogenerate a query interface based on the
> server capability statement.  It SHALL be the same as the type in the search
> parameter definition."

to:

> "While this can be looked up from the definition, it is included here as a
> convenience for systems that autogenerate a query interface based on the
> server capability statement.  It SHALL be the same as the type in the search
> parameter definition. If down-converting from a version of FHIR that has
> additional types, they would map to 'special'."

No other element, binding, value set, or code system is modified. The bound
ValueSet (`http://hl7.org/fhir/ValueSet/search-param-type`) and its backing
CodeSystem (`source/capabilitystatement/codesystem-search-param-type.xml`)
remain untouched.

### Design Rationale

- The committee explicitly rejected expanding the value set with `unknown` /
  `other`. Adding a new required-binding code would itself be a breaking
  change for every server publishing a CapabilityStatement and every consumer
  parsing one, and would invalidate the current short-form summary
  (`number | date | string | token | reference | composite | quantity | uri |
  special | resource`).
- The existing `special` code is already the catch-all for search parameter
  types whose semantics fall outside the standard kinds, so it is the natural
  landing spot for unknown/future types in a down-conversion scenario.
- A `<comment>` change is non-normative as far as conformance machinery is
  concerned (it does not alter cardinality, type, or binding) and therefore
  carries the lowest possible risk profile while still giving down-converters
  written guidance.
- The change is localised to a single element in a single StructureDefinition,
  matching the bespoke HL7 publisher's "edit the differential, let the
  publisher regenerate the snapshot" model.

### Open Questions

- The same common binding (`SearchParamType`) is also bound to
  `SearchParameter.type` (`source/searchparameter/structuredefinition-SearchParameter.xml`,
  line 531). The resolution wording explicitly targets only
  `CapabilityStatement.rest.resource.searchParam.type` and does not direct a
  parallel edit on `SearchParameter.type`. **Open question:** should the
  same down-conversion sentence be mirrored on `SearchParameter.type` for
  consistency? Default per the resolution text: no — leave `SearchParameter.type`
  unchanged unless the WG explicitly asks for the parallel edit.
- The bespoke HL7 publisher is the only build path; there is no sushi /
  IG-publisher fallback, so any reviewer applying this change needs a working
  publisher environment to verify the snapshot regeneration. No action needed
  in the plan, but worth flagging for whoever picks up the ticket.
