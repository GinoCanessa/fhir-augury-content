## Resolution Summary

The committee accepted the report as **Persuasive**. The Versions page (`source/versions.html`) currently labels FHIR R5 as "(this version)" in its release history table. Because the work in scope for this ballot/build is FHIR R6, that label is wrong: R5 is now a prior release and R6 is the current one. The resolution requires:

1. Correct the offending row on the Versions page so that R5 is no longer labelled as the current/this release, and so that R6 is shown as the current release.
2. Sweep the rest of the specification for the same mistake — every other place where R5 is described as "this version", "this release", or otherwise the current release — and apply the same correction.

There is no resolution prose beyond `Persuasive`, no applied-vote comments, and no cross-references on the ticket; the implementation is therefore driven by the ticket description plus a clone-wide audit of "R5 as current release" wording.

## Feature Proposal

### Problem Statement

`source/versions.html` documents the formal milestone release history (DSTU1 → R5) and is the reader's canonical map of "where are we now in the release timeline?". The R5 row in that table is annotated `FHIR R5 (this version)`, which was correct while R5 was the current release but is now incorrect: this build publishes R6, not R5. Several other top-level specification pages copy the same row (`http.html`, `versioning.html`) or paraphrase the same statement in prose (`diff.html`, `diff-r4.html`, `diff-r4b.html`, `history.html`, `mapping-tutorial.html`), so a single-line fix on `versions.html` would leave the rest of the spec inconsistent and continue to mislead readers about which release they are looking at.

The reporter explicitly calls this out: "the implementation note refers to R5 as the current release. This should be R6 … this is not the only place in the spec where R5 is referred to as this release. Please fix the reference on this page and scan the rest of the specification for similar references."

### Proposed Change

Make the Versions page reflect that R6 is the current release, and apply the matching correction to every other top-level page in `source/` that still treats R5 as the current release. Concretely:

1. **`source/versions.html`** — in the milestone release table (lines ~640–673):
   - Re-label the existing R5 row from `<a href="http://hl7.org/fhir/r5">FHIR R5</a> (this version)` to plain `<a href="http://hl7.org/fhir/r5">FHIR R5</a>` (no annotation), keeping the existing `5.0` version cell.
   - Add a new trailing row for R6 with the same shape:
     ```html
     <tr>
       <td><a href="http://hl7.org/fhir/r6">FHIR R6</a> (this version)</td>
       <td>6.0</td>
     </tr>
     ```
   - Verify the surrounding prose ("the formal milestone releases for DSTU2 and on") still parses correctly.

2. **`source/http.html`** (line 514) — same edit pattern: drop the `(this version)` from the R5 row and add an R6 row in the same table where the page enumerates milestone release URLs.

3. **`source/versioning.html`** (line 184) — same edit pattern as `http.html` / `versions.html`.

4. **`source/diff.html`**, **`source/diff-r4.html`**, **`source/diff-r4b.html`** (each at line 26) — the prose "FHIR R4, R4B, R5 and this version" implicitly treats R5 as a prior release (good) but treats "this version" as R6. Confirm each file's "this version" phrasing is unambiguous now that R6 is current; if any sibling text in the same paragraph still calls R5 the current release, correct it. The list itself ("R4, R4B, R5") is correct as-is — these pages compare against prior releases, so R5 belongs in the "prior" list when R6 is current.

5. **`source/history.html`** (line 149) — the bullet `<p><b>R5 Milestone</b>. This version has cumulatively made…` describes the R5 milestone *at the time R5 was the current release*. Update the wording so "this version" no longer dangles as a present-tense claim about R5 being current; rewrite as historical narrative (e.g., `<b>R5 Milestone</b>. R5 cumulatively made 1000s of changes from R4+R4B…`). A new "R6 Milestone" entry is in scope only if other R6 milestone copy already exists on this page; do not invent new release-note prose here as part of this fix.

6. **`source/mapping-tutorial.html`** (line 592) — `This specification includes transforms that map between version R4B and this version (R5).` Replace the parenthetical with `(R6)` so the tutorial reflects the current release. Verify that any concrete examples on the page that still demonstrate R4B↔R5 mappings are intentional (they may be unchanged tutorial content) and only correct the "this version" claim.

The audit grep used to enumerate the sweep list (run from the clone root):

```powershell
Select-String -Path source\*.html -Pattern 'R5\s*\(this (version|release)\)|this (version|release).*R5|R5.*this (version|release)' -List
```

That grep, run against clone HEAD `db79dcd`, returned exactly the eight files listed above. Any new hits introduced before this ticket lands must also be corrected.

A separate, near-by typo in `source/versions.html` line 766 (the cell `<a href="https://hl7.org/fhir/uv/xver-r5.r6/">hl7.fhir.uv.xver-r6.r6</a>` — display text and href disagree, and it sits inside the "R6 Extensions" row that already lists `xver-r4b.r6` on the previous line) is **out of scope** for this ticket. It is flagged here only so a reader does not silently roll a second bug fix into a Technical Correction; file separately if desired.

### Design Rationale

The change is editorial — there is no FHIR design tradeoff to make. The choice to mirror the existing R5 row pattern (rather than restructure the table) preserves the page's long-standing visual layout and minimises diff surface for reviewers, which is appropriate for a Highest-priority Technical Correction. Doing the spec-wide sweep in a single ticket honours the reporter's explicit request and avoids leaving R5-as-current statements scattered across the spec, where they would re-confuse readers as soon as R6 ballots. No alternatives were discussed in the ticket (no comments, no cross-references); the resolution simply records that the request is Persuasive.
