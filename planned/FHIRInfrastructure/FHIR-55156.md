## Resolution Summary

Resolution: **Persuasive**.

Resolution description (Jira): *"Do the same as per FHIR-55154 (should be able to use the same code, just add it to this value set)."*

FHIR-55154's resolution adds an `other` code to the `restful-interaction` CodeSystem (and the sibling `type-restful-interaction` ValueSet) with the definition *"An interaction type that is not one of the standard interaction types defined in this version of FHIR."* and explicitly notes that an `unknown` code is not appropriate here.

For this ticket the concrete change is therefore: **add the `other` concept to the `system-restful-interaction` ValueSet** so that `CapabilityStatement.rest.interaction.code` (which is bound `required` to that value set) has an escape valve for system-level interactions that don't match one of the predefined codes. The CodeSystem concept itself is added by FHIR-55154 — this ticket reuses it.

## Feature Proposal

### Problem Statement

`CapabilityStatement.rest.interaction.code` is `1..1` with a `required` binding to the `http://hl7.org/fhir/ValueSet/system-restful-interaction` ValueSet. That value set today contains only four codes: `transaction`, `batch`, `search-system`, `history-system` (see `source/capabilitystatement/valueset-system-restful-interaction.xml`, lines 250–266 in the cached clone). When a server exposes a system-level RESTful interaction that is not one of those four (for example, a profile/extension-defined system interaction, or a server reporting capability for an experimental or implementation-specific system operation), there is no permitted code to populate `rest.interaction.code` with. The required binding therefore makes the parent `rest.interaction` element impossible to populate in those cases — which contradicts the intent of describing real-world server capabilities.

The same gap exists at the resource level (`CapabilityStatement.rest.resource.interaction.code` → `type-restful-interaction`) and is being closed by FHIR-55154 by adding an `other` code. This ticket extends that fix to the system-level value set so the two siblings stay symmetric.

### Proposed Change

In `source/capabilitystatement/valueset-system-restful-interaction.xml`:

1. Add a fifth `<concept>` to the existing single `<include>` (system `http://hl7.org/fhir/restful-interaction`):

   ```xml
   <concept>
     <code value="other"/>
   </concept>
   ```

2. Add a corresponding `<tr>` row to the table inside `text.div` for `other`, linking to `codesystem-restful-interaction.html#restful-interaction-other`, display `other`, definition *"An interaction type that is not one of the standard interaction types defined in this version of FHIR."* (matching the wording adopted by FHIR-55154).

3. Bump `<meta><lastUpdated>` and `<date>` to the publication-cycle date used for the rest of this build (consistent with how other terminology edits in the build are stamped).

The `other` concept itself is **defined in the `restful-interaction` CodeSystem by FHIR-55154**. This ticket only references it from the `system-restful-interaction` ValueSet — no CodeSystem edit is required here. If FHIR-55154 has not yet landed when this is implemented, add the concept to `source/codesystem/codesystem-restful-interaction.xml` as part of that ticket first; do not duplicate it here.

No `StructureDefinition` change is required: `CapabilityStatement.rest.interaction.code`'s binding URL (`http://hl7.org/fhir/ValueSet/system-restful-interaction`) and binding strength (`required`) remain unchanged — only the membership of the value set widens.

### Design Rationale

- **Consistency with FHIR-55154**: The reporter (Lloyd McKenzie) explicitly tied the two tickets together. Using the same code (`other`) and definition wording in both value sets keeps `rest.interaction.code` and `rest.resource.interaction.code` symmetric and avoids implementers having to learn a different escape valve at each level.
- **Why `other` and not `unknown`**: FHIR-55154's resolution discussion explicitly rejects `unknown` for this binding — an interaction is something the server *does*, so the server cannot meaningfully say it is `unknown`. `other` accurately conveys "I support an interaction outside the standard list".
- **Required binding preserved**: Implementer guides and tooling rely on the `required` binding strength to enumerate supported system-level interactions. Adding an escape-valve concept to the value set is preferable to relaxing the binding to `extensible`, because the latter would invite fully ad-hoc strings and break analytics across CapabilityStatements.
- **CodeSystem reuse**: The four current system-level codes (`transaction`, `batch`, `search-system`, `history-system`) are all sourced from the single `restful-interaction` CodeSystem (`source/codesystem/codesystem-restful-interaction.xml`). Adding `other` there once (via FHIR-55154) and including it from both sibling value sets is the minimal-impact path.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Owns the `restful-interaction` CodeSystem and the `system-restful-interaction` ValueSet that backs `CapabilityStatement.rest.interaction.code`'s required binding.
- **Affected Files:**
  - `source/capabilitystatement/valueset-system-restful-interaction.xml` — add the `other` concept to the `<compose>` and a corresponding row to the narrative table; update `meta.lastUpdated` and `date`.
  - `source/codesystem/codesystem-restful-interaction.xml` — **only if not already added by FHIR-55154**; add the `other` concept (display `other`, definition *"An interaction type that is not one of the standard interaction types defined in this version of FHIR."*).
  - *(No edit needed)* `source/capabilitystatement/structuredefinition-CapabilityStatement.xml` — the binding URL and strength are unchanged.
- **Change Scope:** Minor (additive terminology change to one value set, optionally one CodeSystem concept).

### Breaking Changes

No breaking changes. Adding a new code to a `required`-bound ValueSet:

- Does not invalidate any existing CapabilityStatement instances (existing codes remain valid).
- Does not change cardinality or types.
- Does not change the binding strength.
- The new `other` concept is opt-in for implementers who currently can't represent a non-standard system interaction.

The only behavioural ripple is that downstream validators and tooling that hard-code the four-member enum for system-level interactions will need to refresh their value-set caches. This is expected for any terminology addition and does not constitute a breaking change.

### Related Specifications

- `CapabilityStatement.rest.interaction` (and its sibling `CapabilityStatement.rest.resource.interaction`) are the only spec elements that bind to these two ValueSets. No other resource references `system-restful-interaction` directly.
- The narrative pages `capabilitystatement.html`, `http.html`, and `valueset-system-restful-interaction.html` (publisher-generated) will pick up the new code automatically; no prose changes are required unless a workgroup decides to explicitly call out the escape valve.

### Step-by-Step Tasks

#### HL7/fhir

1. **(Conditional) Add `other` concept to the restful-interaction CodeSystem**
   - File: `source/codesystem/codesystem-restful-interaction.xml`
   - Action: **Skip if FHIR-55154 has already added this.** Otherwise add a new top-level `<concept>` (sibling to `transaction`, `batch`, `operation`) at the end of the concept list (after the `operation` block ending at line 160):
     ```xml
     <concept>
       <code value="other"/>
       <display value="other"/>
       <definition value="An interaction type that is not one of the standard interaction types defined in this version of FHIR."/>
     </concept>
     ```
   - Details: Place it as a top-level concept (not nested under `history` or `search`), since it is not specialised from any of the abstract parents. Do not mark it `abstract`.

2. **Add `other` to the system-restful-interaction ValueSet `<compose>`**
   - File: `source/capabilitystatement/valueset-system-restful-interaction.xml`
   - Action: Inside the single `<compose><include>` (lines 250–265), append a new `<concept>` after the existing `history-system` entry:
     ```xml
     <concept>
       <code value="other"/>
     </concept>
     ```
   - Details: Keep the existing four codes in their current order. The `<include>` already targets `system="http://hl7.org/fhir/restful-interaction"`, so no system attribute changes are needed.

3. **Add `other` row to the ValueSet narrative table**
   - File: `source/capabilitystatement/valueset-system-restful-interaction.xml`
   - Action: Inside `<text><div>` (the hand-authored narrative, lines 8–207), append a new `<tr>` row to the existing `<table class="none">` after the `history-system` row (which ends around line 190):
     ```xml
     <tr>
       <td>
         <a href="codesystem-restful-interaction.html#restful-interaction-other">other</a>
       </td>
       <td>other</td>
       <td>An interaction type that is not one of the standard interaction types defined in this version of FHIR.</td>
     </tr>
     ```
   - Details: Match the indentation and surrounding blank-line whitespace style of the existing rows so the diff stays minimal. Keep `<text status="extensions"/>` unchanged.

4. **Bump publication metadata on the ValueSet**
   - File: `source/capabilitystatement/valueset-system-restful-interaction.xml`
   - Action: Update `<meta><lastUpdated value="…"/>` (line 6) and `<date value="…"/>` (line 237) to the current publication-cycle date used by the build (match what other terminology edits in the same build are stamping).
   - Details: Leave `<version value="6.0.0"/>` untouched unless the build cycle has bumped it elsewhere; do not edit `identifier`, `name`, `title`, or any extensions.

5. **Sanity-check no other location enumerates the four legacy system codes**
   - File: repo-wide grep
   - Action: Run a grep for `system-restful-interaction` and for the four-code tuple (`transaction`, `batch`, `search-system`, `history-system`) across `source/` to confirm no narrative page (e.g., `source/capabilitystatement.html`, `source/http.html`) hard-codes the enumeration in a way that would now be stale. If any does, update it to mention `other` or — preferably — replace the enumeration with a link to `valueset-system-restful-interaction.html`.
   - Details: This is a verification task, not necessarily an edit; record findings in the PR description.
