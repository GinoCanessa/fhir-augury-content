## Resolution Summary

The committee resolved this ticket as **Persuasive with Modification**: instead
of a new extension, a new code is added to the `QuantityComparator` value
set / code system. The exact resolution text is:

- **Code:** `~`
- **Display:** `Approximately`
- **Definition:** The actual value is an approximation.
- **Comment:** Note that a recommended value for the approximation is 10% of
  the stated value (or for a date, 10% of the gap between now and the date),
  but systems may choose other values where appropriate. Systems SHOULD use
  the `Quantity Confidence Interval` extension
  (`http://hl7.org/fhir/StructureDefinition/quantity-confidenceInterval`)
  to provide more detail when possible.

No extension is being added by this ticket; the referenced
`quantity-confidenceInterval` extension already exists (in the
`HL7/fhir-extensions` pack) and only needs to be cited in the comment.

## Feature Proposal

### Problem Statement

The `Quantity.comparator` element today only carries operators that express
strict ordering relations against the stated value (`<`, `<=`, `>=`, `>`)
plus the `ad` ("sufficient to achieve total") code. There is no way to
indicate that the stated value is itself an *approximation* — e.g.,
"approximately 10 mg". This gap was identified during openEHR ↔ FHIR data
mapping work, where openEHR's magnitude/approximate semantics could not be
round-tripped into a `Quantity` without resorting to ad-hoc extensions or a
loss of meaning.

### Proposed Change

Add a new concept to the `QuantityComparator` code system and (transitively)
the `QuantityComparator` value set bound to `Quantity.comparator` (and to
the comparator element on the `Quantity` specializations: `Age`, `Count`,
`Distance`, `Duration`, `Money`).

Concrete authoring change in the `quantity-comparator` worksheet of
`source/datatypes/quantity.xml` (which is the canonical FHIR spreadsheet
that drives both the `QuantityComparator` `CodeSystem` and `ValueSet`):

| Field | Value |
|-------|-------|
| Code | `~` |
| Display | `Approximately` |
| Definition | `The actual value is an approximation.` |
| Committee Notes / Comment | `Note that a recommended value for the approximation is 10% of the stated value (or for a date, 10% of the gap between now and the date), but systems may choose other values where appropriate. Systems SHOULD use the Quantity Confidence Interval extension (http://hl7.org/fhir/StructureDefinition/quantity-confidenceInterval) to provide more detail when possible.` |
| System / Parent / v2 / v3 | (leave blank, matching sibling rows) |

Resulting code system membership becomes (in order):
`<`, `<=`, `>=`, `>`, `ad`, `~`.

Example instance fragment after the change:

```xml
<valueQuantity>
  <value value="10"/>
  <comparator value="~"/>
  <unit value="mg"/>
  <system value="http://unitsofmeasure.org"/>
  <code value="mg"/>
</valueQuantity>
```

No structural change to `Quantity` itself is required; no new search
parameters, no new invariants, no profile changes.

### Design Rationale

- A new code on the existing required-bound value set is the smallest
  surface-area change that makes "approximate" first-class on every existing
  `Quantity`-typed element across the spec, including the specializations
  (`Age`, `Count`, `Distance`, `Duration`, `Money`) that reuse the same
  binding. An extension would have required every consumer to opt-in and
  would not have round-tripped with openEHR's native semantics.
- `~` matches common scientific/medical notation for "approximately", is
  unambiguous in XML/JSON serialisation, and parallels the single-character
  style of the existing comparator codes (`<`, `>`, `ad`).
- The 10% guidance is non-normative ("SHOULD"/"may choose other values"),
  so systems retain flexibility while gaining a portable default.
- Pointing implementers at `quantity-confidenceInterval` keeps the simple
  comparator code lightweight while preserving a path to richer statistical
  metadata when needed — alternatives considered in committee included
  defining a dedicated extension on `Quantity` for "approximate" semantics,
  which was rejected as redundant with the new code.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole repository for the change. Defines the
  `QuantityComparator` `CodeSystem` and the `ValueSet` that is `required`-
  bound to `Quantity.comparator` (and inherited by `Age.comparator`,
  `Count.comparator`, `Distance.comparator`, `Duration.comparator`,
  `Money.comparator` via the Quantity specializations).
- **Affected Files:**
  - `source/datatypes/quantity.xml` — add a new row in the
    `quantity-comparator` worksheet for code `~` / display `Approximately`
    with the resolution's definition and comment text. This is the only
    hand-authored change.
- **Change Scope:** Minor.

### Breaking Changes

Adding a new code to a `required`-bound value set is technically a
forward-compatible expansion for *producers* (existing instances remain
valid) but is a **breaking change for validating consumers**: a strict
validator built against the prior value set will reject instances using
`comparator = ~` until it picks up the new expansion. The spec community
has historically treated value-set expansions on required bindings this
way; this should be called out in the IG release notes for the version
that ships the change.

No element cardinality, type, invariant, or search parameter changes.

### Related Specifications

- `Age`, `Count`, `Distance`, `Duration`, `Money` — Quantity
  specializations that reuse the `QuantityComparator` binding; they
  automatically inherit the new code with no additional file edits.
- `Observation.value[x]` (Quantity), `Observation.referenceRange.low/high`
  (SimpleQuantity), `Condition.onsetAge` (Age), and many other elements
  typed `Quantity` or one of its specializations across the spec — all
  gain the ability to express `~` without any per-resource change.
- `quantity-confidenceInterval` extension in `HL7/fhir-extensions` —
  unchanged, but newly *referenced* from the comparator code's comment.

### Related Tickets

- A `cross-referenced FHIR-56000` lookup via fhir-augury returned 0 hits
  (`temp/plan/xref.json`), and a keyword-similarity search surfaced no
  competing or dependent tickets in the FHIR Infrastructure backlog at the
  time of planning. Re-check before merging in case later ballot tickets
  on `Quantity` have been opened.

### Terminology Impact

- `CodeSystem` `http://hl7.org/fhir/quantity-comparator` — gains one
  concept (`~` / `Approximately`).
- `ValueSet` `http://hl7.org/fhir/ValueSet/quantity-comparator` — its
  expansion automatically grows by one concept (it is an "include all
  codes from the comparator code system" value set).
- No UTG (`HL7/UTG`) change is required: `quantity-comparator` is a
  FHIR-namespace code system maintained inside the FhirCore repo, not in
  UTG. Likewise no `HL7/fhir-extensions` change is required.

### Testing Considerations

- **Revalidation targets:** any resource type containing a `Quantity`,
  `Age`, `Count`, `Distance`, `Duration`, or `Money`-typed element whose
  test fixtures include a `comparator` value. Adding a code to a
  required-bound value set should not invalidate any existing example
  (existing codes remain valid), but a regression run is the cheapest
  way to confirm.
- **Edge cases:**
  - JSON serialisation of `comparator: "~"` — confirm tooling handles
    the tilde without escaping issues.
  - Validators that cache value-set expansions need a refresh; document
    this in release notes.
  - FHIRPath / CQL filters that compare `comparator` against a literal
    set should be reviewed for any hard-coded enumerations.
- **Interoperability:** confirm round-tripping with openEHR's
  approximate-magnitude representation (the original mapping use case
  cited in the ticket).

- Should the FhirCore release notes call out the value-set expansion as
