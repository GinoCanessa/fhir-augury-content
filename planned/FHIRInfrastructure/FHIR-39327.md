## Resolution Summary

The committee resolved this **Persuasive with Modification**. Verbatim from the
resolution description:

> We will split the event into eventCode and eventReference and add an
> invariant that at least one must be present. Will add documentation that
> either or both can be present. If both are present, then the eventCode
> indicates an event that occurs within the context of the reference.
>
> Will provide examples for:
> - offset from date of birth (code only) (after end of)
> - offset from Procedure (event only) (before start of)
> - offset from start of treatment within a clinical trial (Reference to
>   ResearchStudy, eventCode of "start of treatment") (after end of)

Implementation status (from the comment trail and the current clone of
`HL7/fhir-extensions`):

- **Structural / definitional changes are already applied.** Reuben McKenzie's
  2023-03-19 comment ("Changes to extension - master b9e39bb") and Bryn
  Rhodes's 2025-07-23 commit `af8044e` ("[FHIR-34232]: Clarified behavior and
  usage of relative-date extension") sit on top of the
  `StructureDefinition-relative-date.xml` file. The extension now exposes
  `targetReference`, `targetCode`, `targetPath`, `relationship` (deprecated)
  and `offset` slices, and the `targetCode` `comment` element documents the
  "code-within-reference" semantics required by the resolution.
- **Examples are still outstanding.** Gino Canessa's 2025-07-21 follow-up
  comment is explicit: *"Please ensure that there are examples of this
  extension somewhere."* No example instances of the `relative-date`
  extension exist in the `HL7/fhir-extensions` clone today (no
  `input/examples/` folder present, and no `git grep` hits for the URL
  outside the StructureDefinition itself).
- **Naming divergence vs. resolution wording.** The resolution names the
  slices `eventCode` / `eventReference`; the applied implementation kept the
  pre-existing slice names `targetCode` / `targetReference`. The semantics
  match the resolution (event broken into code + reference; either may be
  used; if both, the code is interpreted within the reference) but the
  *names* differ. See Open Questions.
- **Cardinality vs. invariant.** The resolution calls for an invariant
  "at least one must be present". The current SD instead sets `min=1` on
  both `targetReference` and `targetCode`, which makes both *individually*
  required — i.e., it forces both, not "at least one". This is a divergence
  from the approved resolution and is the principal definitional gap that
  this plan must close. See Open Questions.

The remaining implementation work, therefore, is:

1. Reconcile the cardinality / invariant for `targetReference` /
   `targetCode` so the constraint matches the approved resolution
   ("at least one must be present").
2. Author the three example instances called out in the resolution.
3. Optionally, reconcile slice naming (or document the renaming decision in
   the extension's purpose / comment text).

## Feature Proposal

### Problem Statement

The original ticket (David Pyke, 2022-10-25) reported that the `relative-date`
extension shipped with no examples and no explanation of the `code` element,
making it effectively undocumented for implementers. Implementers could not
tell whether the "event" anchor was meant to be expressed as a code, a
reference, or both — and there were no worked examples to disambiguate.

The committee's resolution clarifies the model (split the anchor into a code
and a reference, both optional individually but at least one required, with
defined semantics when both are present) and commits to publishing three
worked examples spanning the expected usage patterns.

### Proposed Change

Apply the following to the `HL7/fhir-extensions` repository (extension lives
at `input/definitions/datatypes/StructureDefinition-relative-date.xml`,
canonical `http://hl7.org/fhir/StructureDefinition/relative-date`).

**A. Constraint change — make targetReference / targetCode jointly optional
but at-least-one required.**

In `StructureDefinition-relative-date.xml`:

- Change `Extension.extension:targetReference` `min` from `1` to `0`
  (line 67 in the current file).
- Change `Extension.extension:targetCode` `min` from `1` to `0`
  (line 97 in the current file).
- Add an invariant on `Extension` (the root differential element) to
  enforce "at least one of targetReference or targetCode SHALL be
  present", e.g.:

  ```xml
  <constraint>
    <key value="rde-1"/>
    <severity value="error"/>
    <human value="At least one of targetReference or targetCode SHALL be present"/>
    <expression value="extension('targetReference').exists() or extension('targetCode').exists()"/>
  </constraint>
  ```

  (Key string subject to publisher conventions in
  `HL7/fhir-extensions`; verify against existing extensions before merge.)

**B. Documentation — clarify "either or both" semantics on the root.**

Update the root `Extension` element's `comment` (currently:
*"This extension is used when a precise date is not known, but rather, the
date will be relative to some future event…"*) to add the resolution's
clarification text:

> Either `targetReference`, `targetCode`, or both SHALL be supplied. If
> both are present, the `targetCode` identifies an event that occurs
> *within the context of* the referenced resource (e.g. "start of
> treatment" within a referenced `ResearchStudy`).

The `targetCode` `comment` already contains a closely related sentence
("If present alongside targetReference, the kind of event is within the
specified event…"); update for wording consistency rather than duplicating.

**C. Examples — add the three instances called out in the resolution.**

Add three example Extension/Observation pairs (or Patient + Observation
pairs as appropriate; the publisher's convention in `HL7/fhir-extensions`
should drive the host-resource choice — verify with the project lead /
existing examples before authoring). Each should appear under
`input/examples/` (folder to be created if not present) and be registered
in `ig.ini` / the publisher's example list per the repo's conventions.

The three required examples:

1. **Offset from date of birth (code only, after end of).**
   - `targetCode`: a CodeableConcept identifying "date of birth" (e.g.
     LOINC 21112-8 *Birth date*, or a SNOMED CT equivalent — pick the
     code system already used elsewhere in `HL7/fhir-extensions`).
   - No `targetReference`.
   - `targetPath`: `Patient.birthDate` (illustrative, may be omitted).
   - `offset`: positive `Duration` (e.g. 18 years), with `relationship`
     left absent because the relationship element is deprecated; the
     sign of the offset encodes "after" per the offset element's
     comment.

2. **Offset from Procedure (event only, before start of).**
   - `targetReference`: `Reference(Procedure/example)`.
   - No `targetCode`.
   - `offset`: negative `Duration` (e.g. -2 weeks) to encode "before".

3. **Offset from start of treatment within a clinical trial.**
   - `targetReference`: `Reference(ResearchStudy/example)`.
   - `targetCode`: a CodeableConcept with display "start of treatment"
     (use the existing study-milestone code system if one is already in
     use in the repo; otherwise a `text`-only CodeableConcept is
     acceptable for an example).
   - `offset`: positive `Duration` (e.g. 30 days), encoding "after".

Each example must include a short narrative `text.div` that mirrors the
resolution wording so readers immediately see the pattern.

### Design Rationale

- **Splitting event into code + reference (already implemented as
  targetCode / targetReference)** mirrors the FHIR pattern used in
  `Timing.event`, `Schedule.actor`, and the CDS-Hooks RelativeTime
  proposal that this extension is slated to be replaced by once
  `RelativeTime` is generally available (per the extension's `purpose`
  field). Keeping both allows machine-computable anchors (reference) and
  human-meaningful anchors (code) to coexist.
- **"At-least-one" rather than "both required"** preserves the
  resolution's explicit statement that *either* a code *or* a reference is
  sufficient, while still preventing meaningless instances that anchor to
  nothing.
- **Examples deliberately span the three slot combinations** (code only,
  reference only, code + reference) so implementers can pattern-match
  their own use cases without re-reading the prose.
- The 2023 deferral note ("Process for doing [examples] for extensions
  isn't yet clear, so deferring that part to R6") is no longer a blocker:
  `HL7/fhir-extensions` is structured as a normal IG with an `input/`
  folder and supports `input/examples/` (the convention used by every
  other example-bearing extension in the repo). The 2025 follow-up
  comment from Gino Canessa effectively reactivates the examples task.

### Breaking Changes

- **Cardinality relaxation on `targetReference` and `targetCode`
  (`min: 1 → 0`) is technically a backward-compatible loosening for
  *consumers*** — instances previously valid remain valid. However, it
  is a **breaking change for tooling that depended on `min=1`** (e.g.,
  validators or generators that assumed both slices would always be
  populated, or producers that omitted one of the two slices and were
  silently non-conformant).
- **Adding the `rde-1` invariant** is a **strictly new constraint**:
  any existing instance that has *neither* `targetReference` nor
  `targetCode` will become invalid. Per the StructureDefinition's
  current shape both slices were `min=1`, so no conformant instance
  could have lacked them; this should be a no-op in practice but must
  be called out for ballot-comment purposes.
- No type changes, no removed elements, no terminology binding
  strength changes.
