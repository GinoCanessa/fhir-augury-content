## Resolution Summary

The applied resolution is **Persuasive**:

> We will remove these tables and will instead point to the examples.html page of the COW specification once there's a release out (along with a note to check for more recent versions of the IG).

In concrete terms: delete the two "Workflow states for …" example tables that currently live in the FHIR Core workflow communications page, and replace them with a short paragraph that links readers to the `examples.html` page of the Clinical Workflow ("COW") implementation guide, instructing them to also check for newer versions of that IG. The ticket carries the `defer` label because the swap is gated on the COW IG having a published release to point at.

## Feature Proposal

### Problem Statement

The current FHIR Core workflow documentation embeds two static example tables that walk a reader through the values of `Task.status`, `Task.statusReason`, and `Task.businessStatus` for (1) a lab order for a blood test, and (2) a referral from a PCP to a specialist. These tables live in `source/workflow-communications.html` (lines 108–228 and 229–500 of the current clone HEAD `db79dcd`). The reporter notes that these worked examples are out of date relative to current implementer practice, and that the Clinical Workflow ("COW") implementation guide now hosts richer, better-maintained equivalents on its own `examples.html` page. Maintaining duplicated, drifting examples in FHIR Core risks confusing implementers, and the FHIR Infrastructure work group has decided that the canonical home for these worked examples should be the COW IG rather than the base specification.

### Proposed Change

In `source/workflow-communications.html`:

1. **Remove the two example sections** in their entirety:
   - The `<h4>Example: Workflow states for a lab order for a blood test</h4>` heading and its preamble paragraph (around lines 108–110), together with the immediately following `<table class="grid">` (lines 111–228).
   - The `<h4>Example: Workflow states for a referral from a PCP to a specialist</h4>` heading and its preamble paragraph (around lines 229–231), together with the immediately following `<table class="grid">` (lines 232–500).
2. **Remove the lead-in sentence** in the preceding paragraph (lines 103–107) that says "The following two examples illustrate how that could be accomplished with regards of specific workflows. Note that these examples avoid any specificity on where the described resources exist, or any reference to a specific communication pattern. The goal of the examples is to demonstrate the use of the Task attributes to represent the workflow state.", because once the tables are gone the sentence has no referent. Keep the surrounding sentence about `Task.status`/`Task.statusReason`/`Task.businessStatus` providing the mechanism for tracking workflow state.
3. **Insert, in place of the removed tables**, a short replacement paragraph pointing to the COW IG examples page. Suggested wording (HTML):

   ```html
   <p>
       For worked examples that illustrate how the combination of <a href="task-definitions.html#Task.status">Task.status</a>,
       <a href="task-definitions.html#Task.statusReason">Task.statusReason</a> and
       <a href="task-definitions.html#Task.businessStatus">Task.businessStatus</a> evolve over the lifetime of a real workflow,
       see the <a href="{COW-EXAMPLES-URL}">examples page of the Clinical Workflow (COW) implementation guide</a>.
       Implementers should check the <a href="{COW-IG-URL}">COW IG home page</a> for the most recent published version of that guide.
   </p>
   ```

   The two `{COW-…}` placeholders MUST be filled with the canonical URLs of the first published COW release before this change is merged (the resolution explicitly defers the change until that release exists).
4. **No other files require edits.** Inbound links from `workflow-management.html` (`workflow-examples.html#optionF-ex1`, `#optionG-ex1`, `#optionI-ex1`) point at `workflow-examples.html`, not at the removed tables, and remain valid. A grep across `source/` for `#workflow-communications` fragment links should still be performed before merge to confirm no inbound link targets one of the removed sections by anchor.

### Design Rationale

The resolution chosen by FHIR Infrastructure delegates the maintenance of worked Task-state examples to the COW IG, which is iterating on this content faster than FHIR Core can. Inlining the examples in FHIR Core was the original approach, but it has produced stale content, so the work group has accepted Lloyd McKenzie's proposal to replace inline examples with a link plus a "check for newer versions" caveat. This mirrors how FHIR Core handles other domain-specific content that has migrated to a maintained IG (for example, pointing readers at downstream IGs for clinical document templates rather than embedding them). The `defer` label on the ticket is the explicit signal that the change must wait for a COW release to exist — landing the change before then would replace one stale artifact with a dead link.
