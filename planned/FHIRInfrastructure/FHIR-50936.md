## Resolution Summary

The resolution is **Persuasive**. The `OperationDefinition` for `resource-validate`
defines the `profile` input parameter twice with identical semantics
(`min=0`, `max=1`, `type=canonical`, `targetProfile=StructureDefinition`,
documentation about `mode=profile`). The agreed correction is to **remove the
second `<parameter>` block** named `profile` from the OperationDefinition,
leaving exactly one `profile` input parameter on the `$validate` operation.

There is no functional change to the operation contract — the second declaration
was an accidental duplicate; servers and clients have always seen a single
logical `profile` parameter.

## Feature Proposal

### Problem Statement

The published `OperationDefinition` for the resource-level `$validate`
operation contains two identical `<parameter>` blocks for the input parameter
named `profile`. This is invalid OperationDefinition content (parameter names
on a single operation are unique by convention), causes confusion for
implementers reading the published spec, and produces noise/errors in any
tooling that processes the OperationDefinition (e.g., generates client SDKs,
renders parameter tables, or validates the resource itself against the
metaspec).

### Proposed Change

In `source/resource/operationdefinition-Resource-validate.xml`, delete the
**second** of the two `<parameter>` elements whose `<name value="profile"/>`.
Both copies declare:

- `name = profile`
- `use = in`
- `min = 0`, `max = 1`
- `type = canonical`
- `targetProfile = http://hl7.org/fhir/StructureDefinition/StructureDefinition`
- `documentation = "If this is nominated, then the resource is validated
  against this specific profile. If a profile is nominated, and the server
  cannot validate against the nominated profile, it SHALL return an error. The
  profile parameter is required for mode=profile, and may be present in other
  modes"`

so the two blocks are byte-equal — either copy can be removed; the convention
is to keep the first occurrence and remove the duplicate that appears later in
the file (immediately after the `graph` parameter block).

After the edit there must be exactly **one** `<parameter>` element with
`<name value="profile"/>` in the file.

No other artifacts (example instances, search parameters, ValueSets,
CodeSystems, narrative, registries) need to change — the duplicate parameter
declaration was metadata-only.

### Design Rationale

- OperationDefinition input parameters are identified by `name`; two entries
  with the same name on the same operation is a structural defect, not a
  modelling choice.
- Both copies are identical, so deletion is purely a deduplication —
  there is no question of which variant to keep, and no impact on the
  operation's semantics, cardinality, type, or documentation.
- The reporter (Gino Canessa) explicitly recommended "removing the second
  one", and the ticket was resolved **Persuasive** at WG disposition, so the
  workgroup has approved exactly this minimal correction. No alternative
  designs (e.g., merging documentation, renaming one of the parameters) were
  put forward.

## Implementation Plan

### Prerequisites

None. This is a self-contained single-file deletion.

### Step-by-Step Tasks

#### HL7/fhir

1. **Remove the duplicate `profile` input parameter from the resource-level
   `$validate` OperationDefinition.**
   - File: `source/resource/operationdefinition-Resource-validate.xml`
   - Action: Delete the **second** `<parameter>` element whose child is
     `<name value="profile"/>`. In a pre-fix file this block sits immediately
     after the `graph` parameter block (between the `graph` parameter's
     closing `</parameter>` and the next `<parameter>` block, which begins
     with the `structuredefinition-standards-status` extension and the
     `resource` parameter). Keep the first `profile` parameter block intact.
   - Details: The two blocks are byte-equal:
     ```xml
     <parameter>
       <name value="profile"/>
       <use value="in"/>
       <min value="0"/>
       <max value="1"/>
       <documentation value="If this is nominated, then the resource is validated against this specific profile. If a profile is nominated, and the server cannot validate against the nominated profile, it SHALL return an error. The profile parameter is required for mode=profile, and may be present in other modes"/>
       <type value="canonical"/>
       <targetProfile value="http://hl7.org/fhir/StructureDefinition/StructureDefinition"/>
     </parameter>
     ```
     so removing the second occurrence is a 9-line deletion with no
     reformatting elsewhere. Reference implementation: commits `142aabe6` and
     `7c995e64` on master, which performed exactly this deletion.

2. **Verify the file parses and contains exactly one `profile` parameter.**
   - File: `source/resource/operationdefinition-Resource-validate.xml`
   - Action: Grep / XPath for `<name value="profile"/>` and confirm a single
     hit. Confirm the surrounding `<parameter>` blocks for `mode`, `resource`,
     `graph`, `jurisdiction`, `non-must-support-flagged`, etc. remain in their
     original order and content.
   - Details: e.g. `Select-String -Path source/resource/operationdefinition-Resource-validate.xml -Pattern '<name value="profile"/>'` should report one line.
