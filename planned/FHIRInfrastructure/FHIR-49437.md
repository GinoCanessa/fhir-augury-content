## Resolution Summary

The resolution is **Persuasive with Modification**:

> We will ensure that there are common search parameters for all pattern elements that are commonly searched. Specifically:
>
> - **Request** — `identifier`, `basedOn`, `replaces`, `groupIdentifier`, `status`, `intent`, `priority`, `doNotPerform`, `category`, `code`, `product`, `subject`, `encounter`, `occurrence`, `authoredOn`, `requester`, `performer`, `performerType`
> - **Event** — `identifier`, `basedOn`, `partOf`, `researchStudy`, `status`, `category`, `code`, `product`, `subject`, `encounter`, `occurrence`, `performer`, `performerType`, `location`
> - **Definition** — (use the common canonical / `MetadataResource` search parameters) plus `performerType`, `code`, `product`
>
> The resource-pattern mappings will be used to determine which element(s) back the search parameter for each resource. Where a resource lacks a core element but a core extension exists, the existing core extension is used.

A subsequent comment (Jose Costa-Teixeira, 2025-01-30) asks that the **`group-or-identifier`** search parameter (resolution of [FHIR-47251](https://jira.hl7.org/browse/FHIR-47251) — "Multiline orders / prescriptions") also be included in the Request set. This is in scope and treated as part of the implementation here (see Open Questions for explicit confirmation).

The work is therefore: define the named search parameters once per pattern (Request / Event / Definition) and ensure every resource that follows that pattern has the corresponding search parameter in its `bundle-<Resource>-search-params.xml`, with `expression` derived from the resource→pattern element mapping (or the agreed core extension when the element is absent).

## Feature Proposal

### Problem Statement

Resources that follow the same workflow pattern use slightly different element names for the same conceptual element (e.g., `MedicationRequest.authoredOn`, `Task.authoredOn`, `NutritionOrder.dateTime`; or `MedicationRequest.requester` vs `NutritionOrder.requester` vs `Task.requester`). Because the resources do not all expose the same `code` for those concepts as a search parameter, a single cross-resource query of the form

```
GET [base]?_type=NutritionOrder,MedicationRequest,CarePlan&recorder=DrWho
```

is impossible — each resource defines its own search parameter codes (e.g., `requester` is missing from `NutritionOrder` today; `authoredon` is missing from many requests; `performertype` only exists on a few). The Request, Event, and Definition patterns already exist as logical models in `source/request/`, `source/event/`, `source/definition/` and resources map their elements to those patterns via the `*-request-mapping-exceptions.xml` / `*-event-mapping-exceptions.xml` / `*-definition-mapping-exceptions.xml` files. The patterns, however, do not currently project a *common* set of search parameter codes onto their implementers.

The ticket proposes to close this gap by standardizing search parameter codes across all resources that implement a given pattern, using the pattern→resource element mapping as the source of truth.

### Proposed Change

Define the following common search parameter codes per pattern, and ensure each implementing resource carries a `SearchParameter` whose `code` matches and whose `expression` resolves to the resource-specific element identified by the pattern mapping (or to the agreed core extension when no native element exists).

**Authoring mechanics:**
- Search parameters are authored as `<SearchParameter>` entries inside each resource's `source/<resource>/bundle-<Resource>-search-params.xml`. There are 147 such bundle files in the clone today.
- Each new `SearchParameter` follows the existing entry shape (see `source/medicationrequest/bundle-MedicationRequest-search-params.xml` for the canonical pattern), supplying `id`, `url` (under `http://hl7.org/fhir/build/SearchParameter/<Resource>-<code>`), `description`, `code`, `type`, `expression`, and `processingMode`.
- Where the resource lacks the pattern element but a core extension exists, the `expression` uses `Resource.extension('http://hl7.org/fhir/StructureDefinition/<extension-name>').value...` per existing FHIR Core practice. The pattern→element mapping for each resource is determined by reading the resource's `<resource>-<pattern>-mapping-exceptions.xml` together with the pattern logical model (`source/request/request-spreadsheet.xml`, `source/event/event-spreadsheet.xml`, `source/definition/definition-spreadsheet.xml`).
- Where the resource already carries a search parameter for the same concept under a different `code` (e.g., `NutritionOrder.datetime` rather than `occurrence`; `MedicationRequest.authoredon` already present), the existing parameter is **preserved** for backward compatibility and a new parameter with the standardized `code` is added (or the existing parameter is renamed and an alias retained — to be confirmed in Open Questions). The starting position is "add the standardized code; never silently break an existing code."

### Design Rationale

- The pattern logical models in `source/request/`, `source/event/`, `source/definition/` already act as the consistency mechanism for *element* design across resources of a kind. Extending that consistency to *search parameters* is the natural next step and is exactly what the resolution describes.
- Using the pattern→resource mapping as the source of truth keeps the authoring surface aligned: when a resource changes its mapping into the pattern, the search parameter expressions can be regenerated/reviewed mechanically.
- Hyphenated codes (`based-on`, `group-identifier`, `performer-type`, `do-not-perform`, `authored-on`) follow established FHIR `SearchParameter.code` conventions (see existing `MedicationRequest-group-identifier`, `MedicationRequest-group-or-identifier`).
- Including `group-or-identifier` is consistent with the FHIR-47251 resolution, which already shipped that combined parameter for `MedicationRequest` (visible in `bundle-MedicationRequest-search-params.xml`); broadening it to all Request-pattern resources is the obvious generalization.
- Preserving existing parameters as aliases avoids breaking deployed clients while still giving cross-resource queries a single common code to use.

### Breaking Changes

- **Net new search parameter codes per resource — non-breaking in the additive sense.** Adding new `SearchParameter` resources with previously unused `code` values is backward-compatible for existing clients.
- **Risk: code collisions / behavior changes.** Where a resource currently exposes the same conceptual search under a different `code` (e.g., `NutritionOrder.datetime` vs the new common `occurrence`; `MedicationRequest.authoredon` vs `authored-on`; `intended-performertype` vs `performer-type`), introducing the standardized code does *not* by itself break anything, but **renaming or removing the existing parameter would**. This plan keeps the existing parameters as aliases (decision to confirm — see Open Questions). If the workgroup later decides to retire the legacy codes, that is a breaking change and must be staged across versions.
- **Risk: `subject` vs `patient`.** Several resources expose `patient` as a typed-narrowed alternative to `subject`. Adding a common `subject` parameter does not break `patient`; both should coexist, with `patient` defined as the existing "Patient-only" composite where it exists today.
- **Resources where the pattern element has no native element and no agreed extension exist.** For these, no SearchParameter can be added; that gap must be flagged per-resource (see Open Questions and per-task notes).

No structural breaking changes to existing resource definitions are introduced.

## Implementation Plan

### Prerequisites

1. Resolve the **alias vs rename** policy (Open Question 1) before touching files. The decision affects whether each per-resource edit is purely additive or also renames existing entries.
2. Resolve the **gap policy** (Open Question 2) — what to do for resources where the pattern element has no native equivalent and no agreed core extension. Without this, several Request and Event resources cannot be completed.
3. Build the per-resource mapping table (one row per (pattern, resource, common-code) → expression) from the pattern spreadsheets and `*-mapping-exceptions.xml` files. This table is the input to every per-bundle edit and should live as a working spreadsheet attached to the ticket; it is not a committed artifact in the repo.
