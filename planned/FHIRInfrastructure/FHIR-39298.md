In concrete terms, the resolution authorises adding contextual columns to the
"Overview of approaches" table on `exchanging.html` so that, for each
exchange approach, an implementer can see at a glance:

1. Whether the *client* and/or *server* end of the interaction is expected
   to be a FHIR-conformant implementation (e.g., a RESTful FHIR server vs.
   a CDS Hooks Service vs. a generic SPARQL endpoint).
2. Which transport protocol(s) the approach is defined over (e.g., HTTP /
   HTTPS, or "any transport" for things like Documents and Messages that
   can be exchanged out-of-band).
3. Which wire/MIME types are used (FHIR JSON / FHIR XML for REST,
   `application/json` for CDS Hooks, GraphQL document strings, RDF for
   SPARQL, etc.).

The resolution explicitly leaves room to **consolidate** existing columns
where the new context makes some current columns redundant or where a
single column can encode several facts. Because the ticket carries the
**Deferred** label and the resolution called out that it might slip past
R5, this work is targeted at R6.

The ticket has no comments and no GitHub PR or commit cross-references —
nothing has been implemented yet.

## Feature Proposal

### Problem Statement

The "Overview of approaches" table on `source/exchanging.html`
(`#approach-summary`) lists ~28 exchange approaches with four columns:
**Re-use**, **Adoption**, **Description**, and **Example**. Implementers
choosing between approaches today must follow each row's link to discover
basic technical facts that strongly influence the choice:

- **Is FHIR conformance required on each end?** A "REST create" row
  obviously implies a FHIR RESTful server on the receiving end, but
  rows like *CDS Hooks*, *SPARQL search*, and *FHIR Messaging
  Notifications* are not as transparent — CDS Hooks expects a
  CDS-Hooks-conformant service (which need not be a full FHIR server),
  SPARQL targets a generic RDF endpoint, and Messaging may be carried
  over non-HTTP transports.
- **What protocol does it run over?** Most approaches assume HTTP/HTTPS,
  but Messaging and Documents are deliberately transport-agnostic, and
  this is not visible from the table.
- **What MIME types are exchanged?** FHIR REST uses
  `application/fhir+json` / `application/fhir+xml`; CDS Hooks uses plain
  `application/json` over a CDS-Hooks-defined payload; GraphQL search
  uses GraphQL request bodies; SPARQL uses SPARQL query/results
  formats; Documents and Bundles can be FHIR JSON/XML/RDF.

These facts are decision-driving but currently buried. Reporter notes
that surfacing them in the comparison table itself would materially help
decision making (per the ticket description and the linked
`fhir/infrastructure-wg > Status Report` Zulip topic where the ticket
was triaged).

### Proposed Change

Edit a single FHIR Core specification page,
`source/exchanging.html`, and rework the `#approach-summary` table
under the `#approaches` anchor as follows.

**1. Add three new contextual columns** (subject to consolidation in
step 3):

- **FHIR endpoint** — encodes whether each end of the interaction is
  expected to be a FHIR-conformant implementation. Suggested coded
  values applied per row:
  - `Source` — only the *data source* must be a FHIR implementation
    (e.g., RESTful read/search/vread/history; SPARQL targets a
    FHIR RDF endpoint).
  - `Consumer` — only the *data consumer* must be a FHIR
    implementation (e.g., REST create / update / patch; REST batch /
    transaction Bundle to consumer; FHIR 'Process' Operation;
    Messaging notifications consumed by a FHIR system).
  - `Both` — both ends are expected to be FHIR implementations
    (e.g., Subscription, Task, CommunicationRequest, FHIR Messaging
    Queries, Polling).
  - `Neither / out-of-band` — the wire format is FHIR but neither
    endpoint need expose a FHIR REST API (e.g., FHIR Document
    exchanged via email / SFTP, Collection Bundle stored on a Batch
    endpoint that may not be a general FHIR server).
  - `CDS Hooks Service` — special case for the CDS Hooks row; the
    *consumer* is a CDS-Hooks-conformant service rather than a
    generic FHIR server.
- **Protocol** — the transport(s) defined by the approach. Suggested
  coded values:
  - `HTTPS` — required by FHIR REST in production (and used as the
    default for CDS Hooks, GraphQL search, SPARQL search, _filter
    search, CQL search, batch search, REST patch, etc.).
  - `HTTP/HTTPS` — for completeness when both are explicitly
    permitted (e.g., dev/test).
  - `Any` — for transport-agnostic approaches (FHIR Document, Batch
    Bundle, Transaction Bundle when used out-of-band, FHIR
    Messaging notifications which the spec explicitly permits over
    non-HTTP transports such as MLLP or queue-based messaging).
- **Wire / MIME types** — what is actually on the wire. Suggested
  values per row family:
  - `application/fhir+json`, `application/fhir+xml` — RESTful
    approaches (read, search, vread, history, create, update, patch,
    Bundle batch / transaction / collection, Subscription, Task,
    CommunicationRequest).
  - `application/json` (CDS Hooks payload) — CDS Hooks row.
  - GraphQL request body + `application/fhir+json` response —
    GraphQL search row.
  - SPARQL query + RDF result formats (e.g., `text/turtle`,
    `application/sparql-results+json`) — SPARQL search row.
  - Document/Bundle MIME (FHIR JSON / XML / RDF) — Document and
    Collection-Bundle rows.
  - FHIR message Bundle (`application/fhir+json`,
    `application/fhir+xml`) — FHIR Messaging Queries and Messaging
    Notifications.

**2. Cell-level value table** (to be authored in the markup; concrete
proposed values per existing row):

**3. Consolidation per the resolution.** Once the three new columns
are populated, audit the table for redundancy. The most defensible
consolidations are:

- Merging **Re-use** and **Adoption** into a single
  **Maturity / adoption** column with two-character codings (e.g.,
  `H/H`, `H/M`, `L/M`) and a legend. The current table already
  pairs the two values per row; combining them halves the column
  count and gives back horizontal space for the new columns.
- Optionally folding **Example** into the **Description** cell
  (rendered as a smaller-italic "Example: …" trailing line) so the
  table is no wider than today after the three new columns are
  added.

Both are explicitly editorial and need WG sign-off — flagged as an
Open Question below.

**4. Update narrative around the table** (the paragraphs at lines
~103–149 of `source/exchanging.html`) to describe the new columns
and their meanings, mirroring the existing paragraphs that explain
"Re-use" and "Adoption". Add a short sub-list explaining each new
column's coded values (FHIR endpoint, Protocol, Wire / MIME).

**5. Existing TODO comment** on line 421 (`<!-- TODO - create
ExampleScenarios showing each of the workflows -->`) is unrelated to
this ticket and is left untouched.

### Design Rationale

- The change is **purely editorial / documentation** on a single
  authored page. It introduces no new resource elements, no new
  ValueSets, no new search parameters, and no behavioural change to
  any FHIR API.
- Adding the columns directly to the comparison table — rather than
  burying the information in each linked sub-page — preserves the
  decision-support purpose the table already serves (reporter's stated
  motivation: "additional context to help with decision making").
- Coded short values (e.g., `Source` / `Consumer` / `Both`) keep the
  table readable; long-form prose stays in the linked sub-pages.
- The consolidation suggestion comes from the resolution itself
  ("May be able to consolidate the number of columns a bit"). Picking
  Re-use + Adoption to merge is the lowest-risk consolidation because
  the two values are already presented side-by-side per row.
- Marking Subscription / Messaging / Document as "Any" / "channel-
  dependent" for protocol matches the existing FHIR specification
  text in `subscription.html`, `messaging.html`, and `documents.html`,
  which deliberately do not mandate HTTP for those exchange patterns.
- This work is **R6 scope** based on the resolution language and the
  `Deferred` label.
