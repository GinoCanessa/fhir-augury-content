## Resolution Summary

Resolution: **Not Persuasive with Modification**.

The work group declined the reporter's request to allow `ExampleScenario` to point at arbitrary URLs / non-FHIR endpoints (everything referenced from an `ExampleScenario` must remain a FHIR resource so it can be packaged in an IG; raw payloads are conveyed via `Binary`). However, the work group agreed that `ExampleScenario` currently makes no statement about *how* an instance is serialised on the wire (XML, JSON, TTL, ND-JSON, or — for `Binary` — raw bytes), and that this is sometimes material to a scenario.

The approved modification is to define a **new extension** named **`ExampleScenario-exchange-syntax`** with the following exact properties (per the resolution description):

- **Cardinality:** `0..*`
- **Type:** `code`
- **Binding:** mime type (the standard FHIR `MimeType` value set, `http://hl7.org/fhir/ValueSet/mimetypes`)
- **Context:** `ExampleScenario.step.operation.request` and `ExampleScenario.step.operation.response`
  (in R5 these are realised as `ExampleScenario.process.step.operation.request` and `…response` — see Open Questions)
- **Short:** `Mime format of exchange`
- **Definition:** `A code describing the syntax in which the data is serialized for this exchange.`
- **Comment:** `If multiple codes are specified, then any of the listed syntaxes are permitted. If not specified, any syntax is allowed or the syntax is not considered relevant to the scenario.`

The resolution text closes with "Will let Elliot look at this before we vote", but the ticket has been moved to *Resolved – change required*, so the action item is to ship the extension as specified.

## Feature Proposal

### Problem Statement

`ExampleScenario` describes the operations exchanged in a scenario but does not let an author state which serialisation(s) of a request or response payload are in use. This matters for scenarios that are inherently format-specific — e.g., a `Binary` exchanged as raw bytes (a JPEG, PDF, or HL7 v2 message) versus the same payload base64-wrapped in a JSON `Binary`, or scenarios that constrain themselves to ND-JSON for bulk-data style flows. Without an explicit syntax marker, implementers reading the scenario have to guess which encoding the example is meant to demonstrate.

The original ticket also raised two adjacent concerns: (1) how to render an example whose payload is itself a `Binary` ("Binary in a Binary"), and (2) why `ExampleScenario` requires a `Reference` rather than allowing an arbitrary URL. Both were answered in comments — Binaries are not double-wrapped, and `Reference` is required so the target can live in an IG package — and the work group declined to change those rules. The remaining, agreed gap is the missing serialisation indicator, which this extension closes.

### Proposed Change

Define a new extension StructureDefinition with these characteristics:

- **id / file name:** `examplescenario-exchange-syntax` (file: `StructureDefinition-examplescenario-exchange-syntax.xml`)
- **url:** `http://hl7.org/fhir/StructureDefinition/examplescenario-exchange-syntax`
- **name:** `ExampleScenarioExchangeSyntax`
- **title:** `ExampleScenario Exchange Syntax`
- **status:** `active`
- **kind:** `complex-type`
- **type:** `Extension`
- **baseDefinition:** `http://hl7.org/fhir/StructureDefinition/Extension`
- **derivation:** `constraint`
- **publisher / wg:** HL7 International — FHIR Infrastructure (`structuredefinition-wg` = `fhir`)
- **context** (two entries, both `type=element`):
  - `ExampleScenario.process.step.operation.request`
  - `ExampleScenario.process.step.operation.response`
- **differential — root `Extension`:**
  - `min` = 0, `max` = `*`
  - `short` = `Mime format of exchange`
  - `definition` = `A code describing the syntax in which the data is serialized for this exchange.`
  - `comment` = `If multiple codes are specified, then any of the listed syntaxes are permitted. If not specified, any syntax is allowed or the syntax is not considered relevant to the scenario.`
- **differential — `Extension.url`:** fixed to the canonical url above.
- **differential — `Extension.value[x]`:**
  - sliced to `valueCode` only (`type.code = code`)
  - `binding`:
    - `strength` = `required` (matches the existing `MimeType` binding on every other mime-typed `code` element in core FHIR — see Open Questions if the WG actually wants `preferred`)
    - `valueSet` = `http://hl7.org/fhir/ValueSet/mimetypes`
    - `description` = `IETF mime types (BCP-13)`

Example fragment of usage (illustrative):

```xml
<request>
  <extension url="http://hl7.org/fhir/StructureDefinition/examplescenario-exchange-syntax">
    <valueCode value="application/fhir+json"/>
  </extension>
  <extension url="http://hl7.org/fhir/StructureDefinition/examplescenario-exchange-syntax">
    <valueCode value="application/fhir+xml"/>
  </extension>
  <reference value="Binary/example-jpeg"/>
</request>
```

### Design Rationale

- **Extension rather than core element.** The `Extension-pack` label on the ticket and the FHIR Infrastructure pattern of pushing post-publication additions into the HL7 extensions IG (rather than re-cutting `ExampleScenario` mid-cycle) mean this is shipped as a standalone extension StructureDefinition in `HL7/fhir-extensions`. The HL7/fhir core repo already says "extensions removed from / added outside core live in `HL7/fhir-extensions`" (per its repo briefing), so this is the consistent home.
- **`code` with mime-type binding.** Every other place in core FHIR that records a wire format uses `code` bound to `MimeType` (`http://hl7.org/fhir/ValueSet/mimetypes`, BCP-13). Using the same datatype and value set keeps the new extension grep-able and tooling-compatible (e.g., publisher mime-type validation, SUSHI completion).
- **`0..*` cardinality.** The resolution explicitly chose `0..*` so an author can list "either JSON or XML is acceptable here". The companion `comment` clarifies that absence ≠ "no syntax allowed" — it means the scenario is syntax-agnostic, which is the historical default.
- **Two contexts, not one.** Request and response can legitimately use different serialisations (e.g., upload raw bytes, receive JSON OperationOutcome); separate context entries let the extension appear independently on each side without forcing a paired declaration.
- **`Reference`-only payload retained.** The work group explicitly rejected loosening the rule that `request`/`response` must point at FHIR resources. The new extension addresses the *one* concrete authoring gap that the discussion converged on, leaving the IG-packaging requirement intact.

### Testing Considerations

- Re-validate any existing `ExampleScenario` instance in the IG to confirm the new context entries do not produce false-positive constraint failures on already-published examples.
- Build and load an example `ExampleScenario` that uses the extension twice on a `request` (e.g., `application/fhir+json` and `application/fhir+xml`) and once on a `response` (e.g., `application/octet-stream` for a raw `Binary`); confirm the extension renders correctly in HTML output and round-trips through JSON↔XML.
- Edge cases: empty extension (must validate as omission, not as an error), unknown mime type (should be rejected by the `required` binding), and the same code appearing twice on one element (the resolution does not forbid duplicates, but a publisher warning is acceptable).
- Interoperability: validators that consume the extensions IG should pick up the new SD on the next package release; smoke-test with the HL7 FHIR Validator against a sample scenario.
