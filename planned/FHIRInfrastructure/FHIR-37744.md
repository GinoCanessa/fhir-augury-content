## Resolution Summary

Resolution: **Persuasive**.

> Add FHIRPath Patch examples to the example transaction bundle.
> - One showing conditional PATCH
> - One showing "straight" PATCH

The resolution is to extend the existing `Bundle-transaction` example
(`source/bundle/bundle-transaction.xml`) with two new `entry` elements
that demonstrate the FHIRPath Patch operation (a `Parameters` resource
carrying a FHIRPath Patch operation, sent with HTTP method `PATCH`):

1. A "straight" PATCH against a known logical id (e.g.
   `Patient/123`).
2. A conditional PATCH whose request URL carries search parameters
   (e.g. `Patient?identifier=https://mymrn|789`).

The ticket text supplies a representative payload (a FHIRPath Patch
that adds `Patient.birthDate = 1930-01-01`), but explicitly notes that
"we won't use this specific example; want to pick something that makes
more sense for the example transaction scenario." Implementers should
choose a payload whose target resource and field are consistent with
the other entries already in the example bundle, while preserving the
shape (Parameters → parameter `name=operation` with parts `type`,
`path`, `name`, `value`).

## Feature Proposal

### Problem Statement

The `Bundle-transaction` example
(`source/bundle/bundle-transaction.xml`) is the canonical, walk-through
example referenced from the Bundle resource page and from the
RESTful-API "transaction" documentation. Today it demonstrates POST
(create), conditional create, PUT (update), conditional update, DELETE,
conditional DELETE, an operation invocation (`$lookup`), a search GET
and a conditional read — but it has **no** PATCH entry at all (verified
by grepping the file: no `PATCH` token appears in
`bundle-transaction.xml`). PATCH is one of the verbs explicitly
permitted by the Bundle invariants (`bdl-3`, `bdl-4`, `bdl-5` in
`structuredefinition-Bundle.xml`) and by the
`http-verb` CodeSystem/ValueSet, and the FHIRPath Patch is the
HL7-preferred patch flavour for FHIR. Implementers writing FHIRPath
Patch transactions therefore have no in-spec example to copy from,
which makes the conditional-PATCH case in particular under-specified
in practice.

### Proposed Change

Edit `source/bundle/bundle-transaction.xml` to add two new `<entry>`
blocks to the existing transaction `Bundle`. The existing entries and
their order are preserved; the new entries are appended in the natural
"create / conditional-create / update / conditional-update / delete /
conditional-delete / **patch / conditional-patch** / operation / read /
conditional-read" progression, i.e. immediately after the conditional
DELETE entry (currently lines 233–244 of the file) and before the
`$lookup` operation entry.

Both new entries use:

- `entry.fullUrl` = a fresh `urn:uuid:` value (do not reuse any
  existing `fullUrl` in the bundle).
- `entry.resource` = a `Parameters` resource (FHIR XML, default
  namespace `http://hl7.org/fhir`) carrying exactly **one**
  `parameter` named `operation` with four `part`s, in this order:
  - `type` — `valueCode` ∈ {`add`, `insert`, `delete`, `replace`,
    `move`}. Use `add` (matches the ticket's exemplar payload and is
    the simplest demonstrable case).
  - `path` — `valueString` (FHIRPath expression rooted at the target
    resource type, e.g. `Patient`).
  - `name` — `valueString` (the element name to add, when
    `type=add`).
  - `value` — a typed `value[x]` matching the target element's type.
- `entry.request.method` = `PATCH`.
- `entry.request.url` = either a logical id (straight PATCH) or a
  search expression (conditional PATCH).

**Entry A — straight PATCH** (suggested concrete content):

```xml
<entry>
  <fullUrl value="urn:uuid:1a2b3c4d-0001-4000-a000-000000000001"/>
  <resource>
    <!-- a FHIRPath Patch operation: add Patient.birthDate -->
    <Parameters>
      <parameter>
        <name value="operation"/>
        <part>
          <name value="type"/>
          <valueCode value="add"/>
        </part>
        <part>
          <name value="path"/>
          <valueString value="Patient"/>
        </part>
        <part>
          <name value="name"/>
          <valueString value="birthDate"/>
        </part>
        <part>
          <name value="value"/>
          <valueDate value="1930-01-01"/>
        </part>
      </parameter>
    </Parameters>
  </resource>
  <request>
    <!-- PATCH to [base]/Patient/[id] - that's a FHIRPath Patch -->
    <method value="PATCH"/>
    <url value="Patient/123"/>
  </request>
</entry>
```

**Entry B — conditional PATCH** (suggested concrete content):

```xml
<entry>
  <fullUrl value="urn:uuid:1a2b3c4d-0002-4000-a000-000000000002"/>
  <resource>
    <!-- a FHIRPath Patch operation: add Patient.birthDate -->
    <Parameters>
      <parameter>
        <name value="operation"/>
        <part>
          <name value="type"/>
          <valueCode value="add"/>
        </part>
        <part>
          <name value="path"/>
          <valueString value="Patient"/>
        </part>
        <part>
          <name value="name"/>
          <valueString value="birthDate"/>
        </part>
        <part>
          <name value="value"/>
          <valueDate value="1930-01-01"/>
        </part>
      </parameter>
    </Parameters>
  </resource>
  <request>
    <!-- PATCH to [base]/Patient?params - that's a conditional FHIRPath Patch -->
    <method value="PATCH"/>
    <url value="Patient?identifier=https://mymrn|789"/>
  </request>
</entry>
```

The XML comments mirror the style already used in the file (e.g. the
"`POST to [base]/Patient - that's a create`" and "`DELETE to
[base]/Patient?params - that's a conditional delete operation`"
comments).

If the editor prefers a less abstract payload, replace the `add`
operation against `birthDate` with one that mutates an element on a
resource that already exists in the bundle (e.g., adding a `gender`
or a `name.given` to one of the Patients already created above) — this
satisfies the ticket's "pick something that makes more sense for the
example transaction scenario" caveat without changing the shape of the
new entries.

The companion response example `source/bundle/bundle-response.xml`
should grow two matching response entries (HTTP `200 OK`, no body —
patch is treated like update for response status) so that the Bundle
example pair remains a complete request/response walkthrough. The
list page `source/bundle/list-Bundle-examples.xml` does not need to
change because both new entries are added inside the existing
`bundle-transaction` example (and the existing `bundle-response`
example).

### Design Rationale

- **FHIRPath Patch (Parameters resource), not JSON-Patch / XML-Patch.**
  The resolution explicitly says "FHIRPath Patch", and the ticket's
  exemplar payload uses the `Parameters`-with-`operation`-parameter
  shape that FHIRPath Patch defines. Demonstrating FHIRPath Patch in
  the canonical Bundle example also matches the HL7 preference (it is
  the only FHIR-defined patch format that is conveyable inside a
  `Bundle.entry.resource`, since JSON-Patch / XML-Patch require a
  `Binary` or non-FHIR media type and are not generally recommended in
  a transaction bundle).
- **Both straight and conditional forms.** The resolution explicitly
  asks for both. Conditional PATCH is the form most often missing
  from implementer tooling and the case the reporter raised, so
  showing both side-by-side is intentional.
- **Append, don't reorder.** The existing example is already heavily
  cross-referenced (URN-style `fullUrl` values are used as references
  elsewhere in the bundle and would be referenced from the matching
  `bundle-response.xml`). Appending after the conditional-DELETE
  entry preserves the create→update→delete→**patch**→operation→read
  progression and avoids invalidating any internal references.
- **Use new `urn:uuid:` `fullUrl`s.** The bundle's other entries each
  have a unique `fullUrl`; PATCH entries against existing ids
  technically don't need a `fullUrl`, but using one keeps the example
  consistent with the surrounding entries and makes the response
  bundle easier to align.

## Implementation Plan

### Prerequisites

None. All conformance artifacts (Bundle invariants, `http-verb`

example-instance edit.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add a "straight" PATCH entry to the transaction example.**
   - File: `source/bundle/bundle-transaction.xml`
   - Action: Insert a new `<entry>` immediately after the existing
     conditional-DELETE `</entry>` (currently line 244) and before the
     `$lookup` operation `<entry>` (currently starting at line 245).
   - Details: Follow the "Entry A" template under Proposed Change.
     Use a fresh `urn:uuid:` for `fullUrl`. The `<resource>` is a
     `Parameters` resource (default namespace `http://hl7.org/fhir`)
     with one `parameter name="operation"` containing four `part`s
     in order (`type=add`, `path=Patient`, `name=birthDate`,
     `value` as `valueDate`). Set
     `request.method = "PATCH"` and
     `request.url = "Patient/123"`. Add an XML comment matching the
     surrounding style (e.g.
     `<!-- a FHIRPath Patch operation: add Patient.birthDate -->`
     and `<!-- PATCH to [base]/Patient/[id] - that's a FHIRPath Patch -->`).

2. **Add a conditional PATCH entry to the transaction example.**
   - File: `source/bundle/bundle-transaction.xml`
   - Action: Insert a second new `<entry>` immediately after the
     entry added in Task 1, still before the `$lookup` operation
     `<entry>`.
   - Details: Follow the "Entry B" template under Proposed Change.
     Identical `Parameters` payload; differ only in
     `request.url`, which becomes a search expression
     (`Patient?identifier=https://mymrn|789`). Add an XML comment
     matching the surrounding style (e.g.
     `<!-- PATCH to [base]/Patient?params - that's a conditional FHIRPath Patch -->`).

3. **Append matching response entries to the response example.**
   - File: `source/bundle/bundle-response.xml`
   - Action: Append two `<entry>` blocks at the end of the response
     bundle (in the same order as the request entries added in Tasks
     1 and 2).
   - Details: Each response entry should mirror the style of the
     existing response entries in this file (`response.status = "200
     OK"`, optional `response.location` and `response.etag`, no
     `response.outcome`). For PATCH the result is the updated
     resource state; the example may either omit a body (status-only
     response) or include the resulting `Patient` — pick the form
     already used for PUT/conditional-PUT in this same file to keep
     style consistent.

4. **Local validation pass on the touched examples.**
   - File: `source/bundle/bundle-transaction.xml` and
     `source/bundle/bundle-response.xml`
   - Action: Run the spec build / publisher against the Bundle
     resource (e.g., `./build.sh` or the publisher entry point used
     for partial builds, per the publisher's existing instructions —
     do not introduce a new build mechanism). Confirm the new
     entries validate against `Bundle` and against the
     `transaction-bundle` / `transaction-response-bundle` profiles.
