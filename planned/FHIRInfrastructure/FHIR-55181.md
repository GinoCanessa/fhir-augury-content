## Resolution Summary

**Resolution:** Not Persuasive with Modification

The reporter asked for an `unknown` / `other` "escape valve" concept to be added
to the value set bound to `Questionnaire.item.type`. The committee declined to
add such a concept, on the grounds that `item.type` is too tightly coupled to
the allowed data types of the rest of the item metadata and to the
`QuestionnaireResponse` answer data type to safely support an "other" value.

In place of the requested value-set change, the resolution requires a
**documentation-only** change in two pieces:

1. Add a new section to the Questionnaire **Notes** page titled
   **"Handling alternative question types"**. The section text (verbatim from
   the resolution) is:

   > In most cases, if there's a need to capture an answer type not in the
   > list of allowed types, the preferred solution is to use a group with
   > child questions that define the needed data. However, if there's a
   > genuine need to convey an unsupported type (e.g. SampledData), the
   > recommended solution is as follows:
   >
   > - Set the item.type to 'string'
   > - Place a modifier extension on the item that indicates that the type
   >   isn't actually a string, but something else
   > - Communicate the answer (and if necessary initial values, min values,
   >   max values, etc. with an extension with a value of the desired data
   >   type)

2. Update **`Questionnaire.item.type`** so that its narrative (element
   comment) points readers to this new note section.

No value-set, code-system, or cardinality change is being made.

## Feature Proposal

### Problem Statement

`Questionnaire.item.type` is a mandatory (1..1) element with a `required`
binding to `http://hl7.org/fhir/ValueSet/item-type-useable`. The bound value
set enumerates a fixed list of types (group, display, boolean, decimal,
integer, date, dateTime, string, …) and contains no `unknown` / `other`
sentinel. As a result, when an author needs to ask a question whose answer
type is genuinely outside that list (e.g., `SampledData`, or a future
data type), there is no compliant value they can place in `item.type`, which
in turn makes the entire item impossible to populate and the resource
impossible to validate.

The reporter (Lloyd McKenzie, comment 147 on the imported ballot) asked for an
escape valve in the value set itself. The committee determined that such a
concept cannot safely be added because every other piece of `item` metadata
(`item.answerOption`, `item.initial[x]`, `item.answerValueSet`, etc.) and the
matching `QuestionnaireResponse.item.answer[x]` are typed off `item.type`;
adding `unknown` / `other` would leave those companion elements with no
defined behavior.

The resolution therefore addresses the *guidance gap*: implementers in this
situation today have no documented, interoperable workaround. The spec must
tell them what to do.

### Proposed Change

Two file edits in `HL7/fhir`, both purely narrative (no schema, no
StructureDefinition `<differential>` change to cardinality / type / binding,
no terminology change):

**A. New notes section: "Handling alternative question types"**

- File: `source/questionnaire/questionnaire-notes.xml`
- Add a new `<a name="alt-types"> </a>` anchor and `<h3>Handling alternative
  question types</h3>` section. A natural insertion point is **after** the
  existing `<h3>Question types</h3>` section (anchor `question-types`,
  currently around line 223) and **before** `<h3>Permitted values for
  questions</h3>` (anchor `choices`, currently around line 250), since the
  new content is a direct extension of "what types are allowed and what to
  do when none of them fit".
- Section content (HTML form, faithful to the resolution wording):

  ```html
  <a name="alt-types"> </a>
  <h3>Handling alternative question types</h3>
  <p>
  In most cases, if there's a need to capture an answer type not in the
  <a href="#question-types">list of allowed types</a>, the preferred
  solution is to use a <code>group</code> with child questions that define
  the needed data.  However, if there's a genuine need to convey an
  unsupported type (e.g. <a href="datatypes.html#SampledData">SampledData</a>),
  the recommended solution is as follows:
  </p>
  <ul>
    <li>Set the <code>item.type</code> to <code>'string'</code>.</li>
    <li>Place a modifier extension on the item that indicates that the type
        isn't actually a string, but something else.</li>
    <li>Communicate the answer (and, if necessary, initial values, min
        values, max values, etc.) with an extension whose value is of the
        desired data type.</li>
  </ul>
  ```

  The `<a href="datatypes.html#SampledData">` link is the standard
  cross-reference style used elsewhere in this notes file (compare the
  existing `datatypes.html#string`, `datatypes.html#uri`,
  `datatypes.html#Coding` links in the "Question types" table). The
  `#question-types` link points back to the immediately-preceding section so
  the reader knows what list is being escaped.

**B. Cross-reference from `Questionnaire.item.type`**

- File: `source/questionnaire/structuredefinition-Questionnaire.xml`
- Element: `<element id="Questionnaire.item.type">` (currently lines
  978–1010 in the clone at HEAD `db79dcd`).
- Append a sentence to the existing `<comment>` (currently:
  `Additional constraints on the type of answer can be conveyed by
  extensions. The value may come from the ElementDefinition referred to by
  .definition.`) so it becomes, e.g.:

  > Additional constraints on the type of answer can be conveyed by
  > extensions. The value may come from the ElementDefinition referred to
  > by `.definition`. For guidance on what to do when none of the listed
  > types is suitable, see [Handling alternative question
  > types](questionnaire.html#alt-types).

  No other field on the element changes — `min`/`max` (1..1), the `code`
  type, the required binding to `item-type-useable`, and all `condition`
  references stay exactly as today. The only edit is to the `<comment>`
  string, which the publisher renders as the element's narrative on
  `questionnaire.html` and `questionnaire-definitions.html`.

  *Note on form:* the resolution text says "Will update the
  Questionnaire.type to reference that note section." From the cross-reference
  context (the targets indexed are `Questionnaire.item.type` and
  `Questionnaire.type`) and the substance of the discussion, the intent is
  the element under discussion — `Questionnaire.item.type`. There is no
  element literally named `Questionnaire.type`. This should be confirmed
  with the resolver during implementation (see Open Questions).

### Design Rationale

- **Why not change the value set.** `item-type-useable` drives a network of
  type-dependent constraints (`item.answerOption.value[x]`,
  `item.initial.value[x]`, `item.answerValueSet`, the matching
  `QuestionnaireResponse.item.answer.value[x]`, and several `que-*`
  invariants, e.g. `que-3` / `que-5` / `que-9`). An `unknown` / `other` code
  would have no defined behavior in any of those contexts, so adding it
  would make the resource *less* interoperable, not more. The committee
  resolution explicitly states this: "there's no practical way to allow
  'other' types here."
- **Why "string + modifier extension + value extension".** It reuses
  primitives that already exist (`string` is in the value set; modifier
  extensions are the standard FHIR mechanism for "this thing doesn't mean
  what it normally means"; `value[x]`-typed extensions can carry arbitrary
  data types). Nothing in the base resource has to grow to support it.
- **Why a notes-page section rather than an extension on
  `item.type`.** The agreed mechanism is a *pattern* (combine three
  existing extension slots), not a single new artifact. The notes page is
  where Questionnaire already documents implementation patterns
  (`Handling Boolean Answers`, `Questions with multiple answer types`,
  `Questionnaires with Math`); the new section fits that template exactly.
- **Why also touch the StructureDefinition comment.** Without the
  cross-reference from the element narrative, an implementer reading the
  rendered element table on `questionnaire.html` would never see the new
  notes section. The resolution explicitly calls for this link.

### Related Specifications

- **`QuestionnaireResponse.item.answer.value[x]`** — the matching answer
  element. The new pattern relies on the existing rule that the
  `value[x]` data type on a response answer must agree with the
  questionnaire's `item.type`. Under the new pattern, `item.type` is
  `string`, so the response would carry `valueString` plus a
  matching `value[x]` extension carrying the "real" typed answer. No
  edit is required to `QuestionnaireResponse`, but cross-reading the
  resource's documentation to ensure consistency with the new notes
  section is recommended.
- **SDC IG (`hl7.fhir.uv.sdc`)** — out of scope for this ticket but
  worth flagging to the SDC editors (the ticket carries the `SDC`
  label). SDC may want to publish a concrete extension URL (or reuse an
  existing one) for the "modifier extension that says the type isn't
  really string" role described in the new pattern.
- **Extensions pack (`HL7/fhir-extensions`)** — no change required by
  this ticket; the pattern uses generic mechanisms.
