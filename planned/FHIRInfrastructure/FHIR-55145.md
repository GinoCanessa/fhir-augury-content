## Resolution Summary

Resolution type is **Not Persuasive with Modification** — the committee did not add an `unknown`/`other` code to the `Bundle.type` value set as originally requested, but agreed to a two-part modification:

1. Add `collection` as an exception to invariant `bdl-7` (so that a `collection` Bundle, like a `history` Bundle, is not required to have unique `fullUrl`s / distinct `meta.versionId`s).
2. Add a comment on `Bundle.type` stating: *"If dealing with some other type of Bundle that doesn't map cleanly to a more specialized type, use 'collection' as it imposes the least constraints and has the most flexible semantic."*

Together these changes give implementers an escape valve (`collection`) without expanding the required binding value set.

## Feature Proposal

### Problem Statement

`Bundle.type` is `1..1` with a required binding to `http://hl7.org/fhir/ValueSet/bundle-type`. The ValueSet has no `unknown`/`other` code, so when a sender has a Bundle that does not cleanly match any of the specialized types (`document`, `message`, `transaction`, `transaction-response`, `batch`, `batch-response`, `history`, `searchset`, `collection`, `subscription-notification`), they are forced to pick one of the specialized types and then potentially violate the invariants that type enforces. The reporter (Lloyd McKenzie) asked for an `unknown`/`other` code.

### Proposed Change

Two surgical edits to the `Bundle` StructureDefinition — no ValueSet or CodeSystem changes.

**1. Relax invariant `bdl-7` to also except `collection` Bundles.**

Current (`source/bundle/structuredefinition-Bundle.xml` line ~128-134):

```xml
<constraint>
    <key value="bdl-7"/>
    <severity value="error"/>
    <human value="FullUrl must be unique in a bundle, or else entries with the same fullUrl must have different meta.versionId (except in history bundles)"/>
    <expression value="(type = 'history') or entry.where(fullUrl.exists()).select(fullUrl&amp;iif(resource.meta.versionId.exists(), resource.meta.versionId, '')).isDistinct()"/>
    <source value="http://hl7.org/fhir/StructureDefinition/Bundle"/>
</constraint>
```

After:

```xml
<constraint>
    <key value="bdl-7"/>
    <severity value="error"/>
    <human value="FullUrl must be unique in a bundle, or else entries with the same fullUrl must have different meta.versionId (except in history or collection bundles)"/>
    <expression value="(type = 'history' or type = 'collection') or entry.where(fullUrl.exists()).select(fullUrl&amp;iif(resource.meta.versionId.exists(), resource.meta.versionId, '')).isDistinct()"/>
    <source value="http://hl7.org/fhir/StructureDefinition/Bundle"/>
</constraint>
```

**2. Extend the `Bundle.type` `<comment>` element.**

Current (`source/bundle/structuredefinition-Bundle.xml` line ~244):

```xml
<comment value="It's possible to use a bundle for other purposes (e.g. a document can be accepted as a transaction). This is primarily defined so that there can be specific rules for some of the bundle types."/>
```

After (append the resolution's exact wording):

```xml
<comment value="It's possible to use a bundle for other purposes (e.g. a document can be accepted as a transaction). This is primarily defined so that there can be specific rules for some of the bundle types. If dealing with some other type of Bundle that doesn't map cleanly to a more specialized type, use 'collection' as it imposes the least constraints and has the most flexible semantic."/>
```

### Design Rationale

The committee rejected adding a new `unknown`/`other` code because introducing a catch-all value into a required-bound code element is a breaking change for validators, would weaken conformance across every existing Bundle consumer, and would add a code for which no invariants or processing semantics could be defined. Instead, they leveraged the existing `collection` code, which is already the least-constrained Bundle type. The only remaining friction — that `collection` Bundles were still subject to `bdl-7`'s fullUrl-uniqueness rule — is removed by adding `collection` alongside `history` in the `bdl-7` exception clause. The new `<comment>` on `Bundle.type` makes the intent discoverable to authors reading the resource page.

- **Role:** Hosts the authoritative `Bundle` StructureDefinition whose `<differential>` contains both the `bdl-7` invariant and the `Bundle.type` element definition being modified.
- **Affected Files:**
  - `source/bundle/structuredefinition-Bundle.xml` — update the `<human>` text and `<expression>` of the `bdl-7` `<constraint>` (lines ~128–134); extend the `<comment>` on the `Bundle.type` `<element>` (line ~244).
- **Change Scope:** Minor (two localized edits, both inside `<differential>`, in a single file).

### Breaking Changes

The resolution relaxes, rather than tightens, the constraint: instances that were previously `collection` Bundles with duplicate `fullUrl`s + identical `meta.versionId` were invalid and now become valid. That is a **non-breaking, backward-compatible relaxation** for senders; validators that previously rejected such Bundles will need to pick up the new FHIR definition to avoid false-positive errors, but no previously-valid instance becomes invalid. The `<comment>` change is purely documentation.

**No breaking changes identified.**

### Related Specifications

- **Bundle profiles** in the same directory (`source/bundle/structuredefinition-profile-*-bundle.xml` — batch, batch-response, document, history, message, search-set, subscription-notification, transaction, transaction-response) inherit from `Bundle` and therefore inherit the updated `bdl-7`. No per-profile edits are required because each profile already narrows `Bundle.type` to a single value, and `collection` is not one of those values (so the new `type = 'collection'` branch of `bdl-7` is unreachable inside these profiles and cannot alter their semantics).

### Testing Considerations

- **Revalidation targets:** `Bundle` StructureDefinition and all nine Bundle profiles.
- **Example instances:** Verify each `source/bundle/bundle-*.xml` example still validates. Optionally add a new `collection`-type example that has two entries sharing a `fullUrl` to demonstrate the relaxed invariant.
- **Edge cases:**
  - `collection` Bundle with duplicate `fullUrl` and identical `meta.versionId` — must now validate (previously failed `bdl-7`).
  - `collection` Bundle with no `fullUrl`s — must still validate (was already valid; still covered by the `entry.where(fullUrl.exists())` filter).
  - `history` Bundle — unchanged, still excepted from `bdl-7`.
  - `transaction`/`document`/`batch`/etc. with duplicate fullUrls — must still fail `bdl-7` (the new `or type = 'collection'` branch does not short-circuit them).
- **Interoperability:** Validators that cache the prior FHIRPath expression will continue to reject `collection` Bundles with duplicate fullUrls until they pick up the new definition; this is a definition-version coordination concern, not a conformance break.
- **Invariant-test fixtures:** Inspect `source/bundle/invariant-tests/` for any fixture naming `bdl-7` and update / add as needed so the relaxation is covered.

### Open Questions

- **Ordering of invariant exceptions.** The resolution text doesn't mandate `(type = 'history' or type = 'collection')` vs. `(type in ('history' | 'collection'))`. The plan above preserves the existing expression shape (Boolean `or`) to minimise diff noise; the editor may prefer the `in (…)` form for readability — either is semantically equivalent.
- **Invariant-test coverage.** The briefing was ticket-scoped to FHIR-42980 and does not enumerate Bundle invariant-test fixtures. A quick `source/bundle/invariant-tests/` review is flagged in the checklist; if no `bdl-7`-with-`collection` fixture exists, adding one is a judgment call for the editor.
- **Bundle introduction / notes narrative.** The resolution does not require an update to `bundle-introduction.xml` / `bundle-notes.xml`, but the implementer may wish to add a one-line cross-reference pointing to the new `<comment>` guidance. Left as optional.
