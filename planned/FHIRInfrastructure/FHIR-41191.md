## Resolution Summary

Resolution: **Persuasive with Modification**.

Verbatim resolution description: *"This has already been fixed in R5 and later. Resolving this ticket as applicable to R4 in case a patch release is made."*

Concretely, the regex used to validate the FHIR `base64Binary` primitive datatype must allow the empty string, since per RFC 4648 §10 the empty string is a valid base64 encoding of a zero-length octet sequence. R5 and later already carry this fix; the R4 line of the specification still ships the older regex that rejects the empty string. The change to apply in R4 is to adopt the same regex pattern that is now in master/R5+ — namely one whose top-level quantifier is `*`/`?` (so zero characters match) rather than `+` (which forces at least one quartet). No semantic change to the datatype itself is required; this is a regex-only technical correction.

## Feature Proposal

### Problem Statement

The R4 normative definition of `base64Binary` declares the lexical regex (in the `source/datatypes/primitives.xml` spreadsheet that drives the publisher) as a pattern that requires at least one group of four base64 characters — effectively `(\s*([0-9a-zA-Z\+/=]){4}\s*)+` (R4-era form). That `+` quantifier makes the empty string fail to match, even though RFC 4648 §10 explicitly lists the empty string as the canonical base64 encoding of an empty input. Implementations that strictly enforce the published regex therefore reject conformant base64 values, which is contrary to the underlying standard.

The ticket reporter (Jens Villadsen, FHIR Infrastructure) asked that the regex be brought into line with RFC 4648. The fix has already landed on R5+ (master) — the spreadsheet cell for `base64Binary` in the cached HL7/fhir master clone now reads:

```
(?:[A-Za-z0-9+/]{4})*(?:[A-Za-z0-9+/]{2}==|[A-Za-z0-9+/]{3}=)?
```

…which trivially admits the empty string (`*` and `?` quantifiers around the only character-consuming groups). The remaining work is therefore a straight backport of that same pattern to R4 if/when an R4 technical-correction release is cut. The `R4-Technical-Correction` label and the `Persuasive with Modification` resolution confirm that no design discussion is open — only the mechanical patch on the R4 maintenance branch.

### Proposed Change

Replace the R4 `base64Binary` regex with the regex already in use on R5+/master. Concretely, on the R4 maintenance branch of `HL7/fhir`:

- File: `source/datatypes/primitives.xml` (Excel-XML spreadsheet that drives the publisher; it is the single authoritative authoring location for primitive metadata, per the briefing for HL7/fhir).
- Row: the `base64Binary` row (the row whose first cell is `<Data ss:Type="String">base64Binary</Data>`; in master this row also carries the second `base64Binary` cell as the XML schema name and `string` as the JSON form — the structure on R4 will be the same modulo any column reordering, which must be visually verified on the R4 branch before editing).
- Cell: the regex cell (sixth populated cell in the row on master; identifiable as the only cell whose content is a regex literal containing `[A-Za-z0-9+/]`).
- Old value (R4, expected): a pattern equivalent to `(\s*([0-9a-zA-Z\+/=]){4}\s*)+` — i.e., one or more groups of four base64 chars surrounded by optional whitespace.
- New value (target — copied verbatim from R5+/master): `(?:[A-Za-z0-9+/]{4})*(?:[A-Za-z0-9+/]{2}==|[A-Za-z0-9+/]{3}=)?`

This regex:

- Accepts the empty string (because `(?:[A-Za-z0-9+/]{4})*` matches zero quartets and the trailing `(?:...)?` is optional).
- Accepts any number of full quartets followed by an optional final padded group (`xx==` or `xxx=`).
- Mirrors RFC 4648 §10 examples exactly.

The regex is referenced from `source/datatypes.html` via the publisher template `[%regex base64Binary%]` (line 103 in master); that file is *not* edited because the template substitution will pick up the new value automatically when the publisher next runs.

No additional element, type, cardinality, binding, extension, search parameter, example, or invariant is added or modified by this ticket.

### Design Rationale

The decision is forced by RFC 4648 §10 and by the existing R5+/master fix. There is no design alternative worth exploring — keeping the empty-string-rejecting pattern would leave R4 inconsistent with the very RFC it cites in its own datatype description (`source/datatypes.html` line 95 already links `RFC 4648` directly). Adopting the *exact* R5 regex (rather than inventing a new one) keeps R4 and R5 lexically consistent for downstream tooling and avoids any new risk of accepting payloads R5 rejects (or vice versa).

The earlier R4 wrapper of `\s*…\s*` (whitespace tolerance) is intentionally dropped, because `source/datatypes.html` already states "base64Binary content does not include any whitespace or line feeds, but reading applications should ignore whitespace characters" — i.e., whitespace tolerance is a reader-side requirement, not a wire-format/lexical-space requirement, and so does not belong in the regex pattern. R5+/master made the same call.

- **Change Scope:** Minor (one cell in one spreadsheet; everything else regenerated).

### Breaking Changes

No breaking changes from a strict conformance standpoint: the new regex is a strict superset of the old one (every value that matched the old pattern still matches the new pattern; the new pattern additionally matches the empty string and unpadded multi-quartet sequences exactly as RFC 4648 defines them). Implementations that relied on the regex *rejecting* the empty string would observe a behavioural change, but that prior behaviour was non-conformant to RFC 4648 and is the bug being fixed. No element cardinality, no type, and no binding changes; no search parameters change.

Whitespace tolerance (`\s*…\s*`) present in the old R4 regex is intentionally removed. Implementations that previously parsed `base64Binary` literals containing embedded whitespace and relied on the regex to admit them will need to strip whitespace themselves before matching — but `source/datatypes.html` already documents that whitespace is a reader-side concern, so this aligns the regex with the long-standing prose.

### Related Specifications

No hand edits are needed in those resources' StructureDefinitions.
- The XML Schema (`fhir-base.xsd` / `fhir-single.xsd`) and JSON Schema for R4 are publisher-generated from the same primitives.xml row; they will pick up the new pattern after the publisher runs.

   - File: `source/datatypes/primitives.xml`

### Testing Considerations

- **Empty-string acceptance:** the canonical regression test for this ticket. Validate a `Binary` resource (`Binary.data = ""`) and an `Attachment` with `Attachment.data = ""` against the regenerated R4 schemas and StructureDefinitions; both must validate cleanly.
- **Padded base64 acceptance:** validate values such as `"AA=="`, `"AAA="`, and `"AAAA"` — all must continue to validate.
- **Multi-quartet acceptance:** validate longer values such as `"SGVsbG8gd29ybGQ="` — must continue to validate.
- **Whitespace rejection (intentional behaviour change):** validate `" AA== "` — under the new regex this should fail purely-lexical regex validation; downstream readers are still expected to strip whitespace per `datatypes.html`. Document the behaviour change in any R4 release notes if not already noted from R5.
- **Resources whose elements are typed `base64Binary`:** smoke-test at least `Binary`, `Attachment`, `AuditEvent.entity.detail.valueBase64Binary`, `Parameters.parameter.valueBase64Binary`, and `Device.udiCarrier.carrierAIDC` example instances to confirm the snapshot regeneration was clean.
- **Interoperability:** spot-check at least one reference R4 implementation (HAPI / firely-net-sdk / similar) to verify the new regex compiles and matches identically across regex engines (Java, .NET, JavaScript, Python `re`); the pattern is engine-agnostic and uses no extended features, so no incompatibility is expected.
