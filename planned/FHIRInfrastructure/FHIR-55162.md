## Resolution Summary

Resolution: **Persuasive**. The ticket reports that the `examplescenario`
page "contains one or more misspelled or abbreviated words. Abbreviations
need to be fully spelled out. Specifics: `doesn't`." The approved change
is therefore to expand the contraction `doesn't` to `does not` everywhere
it appears in the content that renders on the ExampleScenario
specification page (narrative fragments and the rendered element
definitions / comments from the core StructureDefinition).

## Feature Proposal

### Problem Statement

HL7 FHIR publication style requires prose on specification pages to use
fully spelled-out forms rather than English contractions, so that
translated and read-aloud renderings are unambiguous. The current
`examplescenario.html` page (assembled from files under
`source/examplescenario/` by the bespoke HL7 publisher) still contains
the contraction `doesn't` in both its narrative introduction and in two
`<comment>` values on the ExampleScenario StructureDefinition, both of
which surface in the rendered element table on the page. The ticket
reporter (Lloyd McKenzie) flagged these as a technical correction with
priority **Highest**.

### Proposed Change

Replace every occurrence of the contraction `doesn't` that contributes
to the rendered `examplescenario` page with the expanded form
`does not`, preserving surrounding sentence structure, punctuation, and
XML escaping. No structural, cardinality, binding, or behavioural
changes are introduced — this is a pure editorial / copy correction.

Concrete replacements confirmed against the clone
(`cache/github/repos/HL7_fhir/clone/` @ `db79dcd`):

1. `source/examplescenario/examplescenario-introduction.xml`, line 26 —
   narrative fragment:
   - Before: `…can define alternatives, it doesn't necessarily call out all of them.…`
   - After: `…can define alternatives, it does not necessarily call out all of them.…`
2. `source/examplescenario/structuredefinition-ExampleScenario.xml`,
   line 639 — `<comment value="…">` on the element whose definition
   describes the data-structure indicator (inside `<differential>`):
   - Before: `"This element may be omitted if describing a step that doesn't involve the sharing of a data structure - e.g. a human action"`
   - After: `"This element may be omitted if describing a step that does not involve the sharing of a data structure - e.g. a human action"`
3. `source/examplescenario/structuredefinition-ExampleScenario.xml`,
   line 723 — `<comment value="…">` on the "instance version" element
   (inside `<differential>`):
   - Before: `"Not used if an instance doesn't change"`
   - After: `"Not used if an instance does not change"`

### Design Rationale

- The ticket text explicitly calls out `doesn't` as the abbreviation to
  spell out, and the resolution is **Persuasive** (accepted as-is with
  no scope expansion). The three sites above are the only
  `doesn't` occurrences in `source/examplescenario/` and are therefore
  precisely the set that needs to change.
- `does not` is the standard expansion used elsewhere in the core FHIR
  specification prose; no alternative rewording (e.g., reshaping the
  sentence) is warranted for a technical correction of this scope.
- Edits are applied in the hand-authored `<differential>` section of
  the StructureDefinition and in the narrative `-introduction.xml`
  fragment, in line with the repo's authoring rules — not in generated
  `<snapshot>` blocks.
- Other contractions on the same page (e.g., `what's`, `it's`) are out
  of scope because the ticket's `Specifics` field names only `doesn't`;
  a broader sweep would require a separate ticket per the reporter's
  stated scoping.
