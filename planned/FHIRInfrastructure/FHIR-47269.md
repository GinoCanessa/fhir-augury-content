## Resolution Summary

Resolution type: **Persuasive with Modification**.

On the datatypes page for the Signature data type, add a new section titled
"Known Signature Issues" immediately **before** the XML and JSON signatures
section. The section must contain prose to the effect of:

> "There are situations where FHIR requires data to be transformed as part
> of the exchange process and these transformations will potentially cause
> signature issues unless a custom canonicalization mechanism is chosen
> that takes the changes into account. These include:
> - Expectations for translating relative URLs into absolute URLs or moving
>   referenced data inline in the [Bulk Data Exchange](async-bulk.html#…)
>   portion of the spec."

The exact anchor on the Bulk Data Exchange link was flagged as broken in
the original resolution text (`<span class="error">[Bulk Data
Exchange]</span>(async-bulk.html#3.2.6.1.5.0.4)`) and must be repaired
when the prose is landed — see the Open Questions section.

## Feature Proposal

### Problem Statement

The FHIR asynchronous Bulk Data Exchange pattern can require servers to
rewrite resources as part of the exchange process — for example,
translating relative URLs to absolute URLs, or moving referenced data
inline. When a client or intermediary includes Provenance resources in
the requested data, those rewrites silently invalidate any digital
signatures carried in `Provenance.signature` (or embedded in
`Bundle.signature`), because the signed canonical form of the payload
changes between the signer and the verifier. The current Signature
documentation (`source/signatures.html`) does not warn implementers that
this scenario exists, nor does it point them at the mitigation
(choosing a custom canonicalization that accommodates the documented
transforms). The originating ticket and the bulk-data / Security
discussions on Zulip confirm that implementers have already been bitten
by this in the Bulk Data context.

### Proposed Change

Documentation-only change to one page in `HL7/fhir`.

**File:** `source/signatures.html`

**Insertion point:** Between the existing `Digital Signature Methods`
section (ends at line 360 in the current clone) and the `JWS/JSON
Signatures` section (anchor `<a name="JWS"></a>` on line 361). The
resolution says "before the XML and JSON signatures section"; in the
current page, both the JSON (`JWS/JSON Signatures`, line 362) and XML
(`XML Digital Signatures`, line 389) signature sections are preceded
immediately by the JWS anchor on line 361, so inserting there places
the new section before both as required.

**New markup (approximately):**

```html
<a name="known-issues"></a>
<h3>Known Signature Issues</h3>

<p>
There are situations where FHIR requires data to be transformed as part
of the exchange process and these transformations will potentially cause
signature issues unless a custom canonicalization mechanism is chosen
that takes the changes into account. These include:
</p>
<ul>
  <li>Expectations for translating relative URLs into absolute URLs or
      moving referenced data inline in the
      <a href="async-bulk.html">Bulk Data Exchange</a> portion of the
      spec.</li>
</ul>
```

No changes to the Signature StructureDefinition, to `datatypes.html`,
or to any other resource are required by the resolution. The Signature
section in `source/datatypes.html` already links to `signatures.html`
(line 1775 of `datatypes.html`), so the new section is reachable from
the datatype reference without further cross-linking.

### Design Rationale

- The resolution is explicitly **"Persuasive with Modification"**: the
  Infrastructure WG declined to change the Signature structure, extension,
  or Provenance flow, and instead documented the known limitation so that
  implementers can select an appropriate canonicalization strategy.
  That matches the ticket's original question ("What is the recommended
  process…?") with guidance rather than a normative change.
- Placing the warning as an `<h3>` inside `signatures.html`, adjacent to
  the existing canonicalization and method sections, keeps all signature
  caveats co-located. The pre-existing `<h2>Cautions around the use of
  digital signatures</h2>` section (line 78) covers *governance* concerns
  (what is signed, who signed, ongoing change); the new section covers a
  different, *protocol-induced* concern and is appropriately a sibling
  `<h3>` under the `Digital Signature Methods` treatment.
- A bulleted list is used rather than prose because the resolution text
  opens the door for additional transforms to be listed in the future
  ("These include:"). The list structure makes future additions
  low-risk.
- John Moehrke's comment ("Signature datatype is owned by Security") is
  a governance note — the Security WG owns the datatype — but the
  ticket itself remains with FHIR Infrastructure and the authoring
  location is unchanged.
