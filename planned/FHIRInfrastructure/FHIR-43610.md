## Problem Description

The `backport-related-query` examples in the Subscriptions Backport IG are inconsistent:

1. **Topic definition** (`r4-encounter-complete` / `r4b-encounter-complete`): Defines a related query for **"Prescribed medications"** (`http://example.org/query-types#prescribed`) with query URL `http://example.org/fhir/Encounter/[id]/$prescribed-medications`.

2. **Notification examples** (`r4-notification-id-only-with-query`, `r4-notification-full-resource-with-query`, and R4B equivalents): Use a generic **"Example query"** (`http://example.org/query-types#example`) with a generic URL `http://example.org/fhir/$example?patient=...`. These notification examples also reference the **admission** topic (`$admissionTopic`) instead of the encounter-complete topic.

3. The `r4-notification-full-resource-with-query` contains two related queries—one generic "Example query" and one "Prescribed medications"—mixing both styles inconsistently.

The result is that someone reading the examples cannot trace a single consistent scenario from topic definition → subscription → notification.

## Resolution Description

> Update examples to ensure there is a consistent end-to-end example.
>
> Mod: add an additional page walking through the steps in order with the matching examples for each FHIR version.

## Related Tickets

- [FHIR-43608](https://jira.hl7.org/browse/FHIR-43608): "The query for notify pull wouldn't be in the Topic, only the query 'name' would" (Persuasive) — Clarifies that the topic definition may contain only the query _name_ (coded description), not the full query string, since the query URL may only be determined at runtime.

## Detailed Implementation Plan

### Task 1: Add encounter-complete aliases to `Common.fsh`

Add new aliases alongside the existing admission aliases so that the notification examples can reference the encounter-complete topic and subscription:

```fsh
Alias: $encounterCompleteTopic = http://hl7.org/fhir/uv/subscriptions-backport/SubscriptionTopic/r4-encounter-complete
Alias: $encounterCompleteSub = https://example.org/fhir/Subscription/encounter-complete
Alias: $encounterCompleteSubStatus = https://example.org/fhir/Subscription/encounter-complete/$status
```

Also add an inline `Encounter` instance with `status = #finished` for use in full-resource notification examples, to match the "encounter complete" trigger:

```fsh
Instance:    BackportNotificationEncounterComplete
InstanceOf:  Encounter
Usage:       #inline
* id            = "86009987-eabe-42bf-8c02-b112b18cb616"
* status        = #finished
* class         = http://terminology.hl7.org/CodeSystem/v3-ActCode#VR
* subject.reference = $notificationPatient
* subject.display   = "Example Patient"
```

This replaces the existing `BackportNotificationEncounter` (which has `status = #in-progress`) in the query examples so the encounter status is consistent with the "encounter complete" trigger.

### Task 2: Update R4 notification examples in `BackportNotificationR4.fsh`

#### 2a. Create a new RuleSet for encounter-complete notification status

The existing `AddParameterStatus` RuleSet hardcodes `$admissionSub`, `$admissionTopic`, and `$admissionSubStatus`. Add a parallel RuleSet (or parameterize the existing one) for encounter-complete:

```fsh
RuleSet: AddParameterStatusEncounterComplete(id, status, type, sinceStart)
  * entry[0].fullUrl  = "urn:uuid:{id}"
  * entry[0].resource = BackportNotificationStatusShellR4
  * entry[0].resource.id = "{id}"
  * entry[0].resource.parameter[subscription].name = "subscription"
  * entry[0].resource.parameter[subscription].valueReference.reference = $encounterCompleteSub
  * entry[0].resource.parameter[topic].name = "topic"
  * entry[0].resource.parameter[topic].valueCanonical = $encounterCompleteTopic
  * entry[0].resource.parameter[status].name = "status"
  * entry[0].resource.parameter[status].valueCode = {status}
  * entry[0].resource.parameter[type].name = "type"
  * entry[0].resource.parameter[type].valueCode = {type}
  * entry[0].resource.parameter[eventsSinceSubscriptionStart].name = "events-since-subscription-start"
  * entry[0].resource.parameter[eventsSinceSubscriptionStart].valueString = "{sinceStart}"
  * entry[0].request.method = #GET
  * entry[0].request.url = $encounterCompleteSubStatus
  * entry[0].response.status = "200"
```

Alternatively, modify the existing `AddParameterStatus` to be topic-agnostic by parameterizing the subscription/topic references. However, this would require updating all other examples that use `AddParameterStatus`, so a separate RuleSet is safer.

#### 2b. Update `BackportNotificationExampleIdOnlyWithQueryR4`

Change from admission topic to encounter-complete topic, and from generic "Example query" to "Prescribed medications":

**Current:**

### Task 4: Optionally adjust topic definition per FHIR-43608

Per the related ticket FHIR-43608 (resolution: Persuasive), the topic definition SHOULD contain only the query _name_ (coded description), not the full query string, since the query URL may depend on runtime parameters.

In `BackportTopics.fsh`, the R4B encounter-complete topic currently includes both `queryType` and `query` in the `notificationShape` extension. Per FHIR-43608, consider keeping the `queryType` but making the `query` element optional or removing it from the topic definition (the query string would only appear in the notification).

**This should be coordinated with the FHIR-43608 implementation.** If FHIR-43608 is implemented first, the topic examples may already omit the query string.

### Task 5: Create walkthrough page `input/pagecontent/walkthrough.md`

Create a new documentation page that walks through the complete end-to-end example scenario for both R4 and R4B. The page should cover:

1. **Scenario description**: A patient has an encounter that completes, and the subscriber wants to be notified with a link to retrieve prescribed medications.

2. **Step 1: Topic Definition** — Show and explain the `r4-encounter-complete` topic (R4) and `r4b-encounter-complete` topic (R4B), highlighting the `notificationShape` and `backport-related-query` extension with the "Prescribed medications" query type.

3. **Step 2: Subscription** — Show an example subscription subscribing to the encounter-complete topic (could reference existing `BackportSubscriptionExampleAdmission` adapted, or create a new example subscription for encounter-complete).

4. **Step 3: Notification (id-only)** — Show and explain the `r4-notification-id-only-with-query` (R4) and `r4b-notification-id-only-with-query` (R4B) bundles, highlighting how the related query in the notification matches the topic definition.

5. **Step 4: Notification (full-resource)** — Show and explain the `r4-notification-full-resource-with-query` (R4) and `r4b-notification-full-resource-with-query` (R4B) bundles.

6. **Step 5: Client resolves query** — Brief description of how a client would use the related query URL from the notification to retrieve the prescribed medications.
