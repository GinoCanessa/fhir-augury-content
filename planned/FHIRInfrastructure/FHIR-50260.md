## Resolution Summary

The committee resolved this as **Not Persuasive with Modification**. Rather than declaring `lowerLimit` / `upperLimit` to be unscaled (as the submitter proposed), the resolution chose the opposite interpretation — they are **scaled in the same way as the data values** — and asked that this be made explicit in the spec:

1. Add a comment to **both** `SampledData.lowerLimit` and `SampledData.upperLimit`:

   > "These values are scaled in the same manner as the data values. I.e. to find the true limit, multiply this value by the factor and add the origin."

   Rationale captured in the resolution: "We're choosing to go with this interpretation because the one example we have in the spec that has limits (in Observation) uses this approach."

2. Add the following to the description of the **SampledData** data type itself:

   > "If there is a need to capture dimensions that differ in scale, origin, limits, etc. this is typically done by having multiple SampledData instances - one in each Observation.component where the components represent different dimensional subsets."

   This addresses the side-note in the original ticket about `SampledData` only supporting per-instance (not per-dimension) `origin` / `factor` / limits.

No structural changes (no new elements, no cardinality / type / binding changes). This is a pure **prose / element-comment clarification** in the SampledData datatype.

## Feature Proposal

### Problem Statement

The current FHIR R4–R6 specification text for `SampledData.lowerLimit` and `SampledData.upperLimit` does not state whether the values are expressed in the **scaled** units used by the data points (i.e., subject to the `data[i] = SampledData.data[i] * factor + origin.value` transform) or in the **raw, post-conversion** measurement units. As reported by Erik Moll on Zulip ([conformance > interpretation of SampledData.lowerLimit and .upperLimit](https://chat.fhir.org/#narrow/channel/179177-conformance/topic/interpretation.20of.20SampledData.2ElowerLimit.20and.20.2EupperLimit)) and in this ticket, multiple implementers had assumed the unscaled interpretation, which conflicts with how the only canonical example in the spec (under `Observation`) actually uses these fields. Without disambiguation, two conforming implementations can serialise / interpret the same waveform incompatibly.

A secondary concern — the inability to vary `origin` / `factor` / limits per dimension within a single `SampledData` — surfaced in the ticket as a side note. The resolution does not change the data type to support per-dimension parameters; instead it documents the recommended workaround (split into multiple `SampledData` instances carried by separate `Observation.component` entries).

### Proposed Change

**Repository:** `HL7/fhir` (FhirCore)

Two prose-only edits, both within the SampledData datatype's authoring sources. No StructureDefinition cardinality / type / binding / invariant changes; no new elements; no terminology changes.

1. **Element-level Comments — `SampledData.lowerLimit`** (`source/datatypes/sampleddata.xml`, row at line 951–975, "Comments" column = position 16, currently empty cell at line 967)
   - Add cell text:
     > "These values are scaled in the same manner as the data values. I.e. to find the true limit, multiply this value by the factor and add the origin."

2. **Element-level Comments — `SampledData.upperLimit`** (`source/datatypes/sampleddata.xml`, row at line 976–1000, "Comments" column = position 16, currently empty cell at line 991)
   - Add the **same** cell text as above (resolution explicitly says "to both").

3. **Data-type level prose — SampledData narrative** (`source/datatypes.html`, SampledData section starting at line 1291)
   - Add a new paragraph (best placed after the existing multi-dimensional paragraph at lines 1309–1312, immediately before the codes-handling paragraph that begins around line 1313) reading:
     > "If there is a need to capture dimensions that differ in scale, origin, limits, etc., this is typically done by having multiple SampledData instances — one in each `Observation.component`, where the components represent different dimensional subsets."
   - Wrap as a `<p>…</p>` block consistent with the surrounding markup; link `Observation.component` to `observation-definitions.html#Observation.component` for navigation.

Optional secondary placement (to be confirmed during implementation, see Open Questions): the same dimensional-subset paragraph could additionally / alternatively be authored into the SampledData type-level "Comments" cell of the spreadsheet (`source/datatypes/sampleddata.xml`, root SampledData row at line 826–850, "Comments" column at line 842) so that it surfaces in the generated detailed-descriptions table as well as in the narrative page. The resolution says "the description of the data type", which most naturally maps to the narrative page; the spreadsheet duplication is a publish-quality-of-life choice.

### Design Rationale

- **Why scaled and not unscaled?** The resolution explicitly states the choice was driven by the existing canonical example in the spec (the `Observation` SampledData example) which uses scaled limits. Choosing the unscaled interpretation as originally proposed by the submitter would have invalidated that example and any implementations modelled on it. Picking the documented-by-example interpretation minimises real-world breakage.
- **Why a comment, not a definition rewrite?** The element `Definition` ("The lower limit of detection of the measured points. This is needed if any of the data points have the value 'L' (lower than detection limit)" — sampleddata.xml line 965) remains accurate; what's missing is the *unit semantics*. FHIR convention is to put implementer-facing semantics that complement (not replace) the definition into the element `Comments` field, which is exactly the slot used here.
- **Why workaround prose instead of new per-dimension elements?** The submitter's side-note about per-dimension `origin` / `factor` / `lowerLimit` / `upperLimit` is real but represents a structural change that was not approved. The committee has chosen to **document an existing pattern** (one SampledData per `Observation.component`) rather than introduce new optional repeating structures inside `SampledData`. This keeps R6 backward-compatible with R4 / R5 and defers any datatype expansion to a future change request.
