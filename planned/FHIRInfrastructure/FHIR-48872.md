## Resolution Summary

**Resolution:** Persuasive

Resolution description (verbatim from the snapshot): *"Make change to the title as suggested and change ownership to FHIR Infrastructure."*

The ticket calls out two concrete defects on the **Metadata Types** page of the FHIR Core specification:

1. The page **title** displays the typo `MetaDatatypes` and must read `Metadata Types`.
2. The **curating Work Group** is currently shown as *Modelling and Methodology (MnM)* and must be re-attributed to **FHIR Infrastructure**.

## Feature Proposal

### Problem Statement

The FHIR Core spec page that introduces the metadata-oriented general-purpose datatypes (ContactDetail, DataRequirement, ParameterDefinition, RelatedArtifact, TriggerDefinition, Expression, UsageContext, ExtendedContactDetail, VirtualServiceDetail, Availability, MonetaryComponent) is rendered with two presentation defects:

- The visible H1 reads `MetaDatatypes` (run-together, mid-cap "D"), which is inconsistent with the rest of the publication where the page is referred to as "Metadata Types" and is grammatically incorrect.
- The page header attributes ownership to the Modelling and Methodology (MnM) Work Group. Per the resolution, ownership has been transferred to FHIR Infrastructure (FI), and the page must reflect the new owner so reviewers, balloters, and downstream readers contact the correct WG and so WG-level reporting/QA picks the page up under FI.

The defects are limited to authored HTML pages in the bespoke FHIR publisher build (no spreadsheets, no StructureDefinitions, no terminology); they have no normative impact on the datatypes themselves.

### Proposed Change

Edit the four authored pages that compose the Metadata Types section (and the inbound link from the Datatypes index) so that:

1. **Page title (H1)** on `source/metadatatypes.html` line 15 changes from `<h1>MetaDatatypes</h1>` to `<h1>Metadata Types</h1>`.
2. **Inbound link text** on `source/datatypes.html` line 36 changes from `<a href="metadatatypes.html">MetaDatatypes</a>` to `<a href="metadatatypes.html">Metadata Types</a>`.
3. **Sub-page H2** on `source/metadatatypes-examples.html` line 14 changes from `<h2>MetaDatatypes - Examples</h2>` to `<h2>Metadata Types - Examples</h2>` (matches the wording already used on the sibling Definitions/Mappings/Profiles sub-pages).
4. **Sub-page settitle** on `source/metadatatypes-profiles.html` line 4 changes from `[%settitle MetaDatatype Profiles%]` to `[%settitle Metadata Type Profiles%]` (the corresponding H2 already reads "Metadata Types - Profiles", so only the browser-tab title is out of step).

The Work Group reassignment in the page header is already in effect: `source/metadatatypes.html` line 16, plus the four sibling pages, already use `[%wg fhir%]`/`[%wgt fhir%]`, and `source/fhir.ini` line 1057 maps `fhir=…fiwg… FHIR Infrastructure`. No further change is needed for the WG reattribution; verify in the validation step and call it out in the commit message so reviewers can confirm.

No file rename, no anchor change, and no link target change is required. All anchors (`#ContactDetail`, `#DataRequirement`, …) and the file name `metadatatypes.html` remain stable so external deep-links continue to resolve.

### Design Rationale

- The resolution explicitly says "Make change to the title as suggested and change ownership to FHIR Infrastructure." — i.e., update the displayed strings, not the file path. Renaming `metadatatypes.html` would break a large number of inbound links in the spec (see `source/datatypes.html` and the `metadatatypes-{definitions,mappings,profiles,examples}.html` cross-links) for no benefit, so the file name is intentionally preserved.
- The four sibling sub-pages already render "Metadata Types - Detailed Descriptions / Mappings / Profiles" via H2 and use `[%wg fhir%]` for ownership; the proposed change brings the remaining outliers (the H1 on the main page, the H2 on the Examples sub-page, the link label on `datatypes.html`, and the settitle on the Profiles sub-page) into alignment with that established pattern. This is the lowest-risk way to satisfy the resolution and yields a consistent reader experience.
- An earlier commit by the assignee (`865a65b10e`, "Changed title and workgroup", 2025-02-25) already fixed the `[%settitle …%]` and `[%wg …%]` macros on `metadatatypes.html` but missed the literal H1 string on that same page and the ancillary occurrences on the sibling pages and the Datatypes index. This plan completes that work.
