## Resolution Summary

The originally proposed `:term-filter` token search modifier is **not** being added.
Instead, the resolution (Persuasive with Modification) takes a different shape, recorded
in Michael Lawley's 2025-12-05 comment after discussion with Grahame Grieve:

1. **Define a well-known `designation` concept-property** in the
   `http://hl7.org/fhir/concept-properties` CodeSystem.
2. When used as `ValueSet.compose.include.filter.property` with
   `ValueSet.compose.include.filter.operator = "="`, the
   `ValueSet.compose.include.filter.value` is interpreted as a
   **case-insensitive wildcard match** in which `*` MUST be specified explicitly
   (so a value with no `*` is an anchored full-string match).
3. Two optional modifiers on the property name are defined for constraining which
   designations are matched:
   - `designation.use` — written `:[system]#code` (system optional, `#` required),
     e.g. `designation:|preferredForLanguage` or
     `designation:http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra|preferredForLanguage`.
   - `designation.language` — an RFC 4647 language-range preceded by `@`,
     e.g. `designation:@en,en-*` (combinable with the use modifier:
     `designation:…|preferredForLanguage@en`).
4. Examples (from the resolution comment), where the value is interpreted with
   wildcards as described:
   - `designation = "*tylenol*"`
   - `designation:|preferredForLanguage = "*tylenol*"`
   - `designation:http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra|preferredForLanguage = "*tylenol*"`
   - `designation:@en,en-* = "*tylenol*"`
   - `designation:http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra|preferredForLanguage@en = "*tylenol*"`
   - `designation = "tylenol"` (anchored, exact, case-insensitive)
5. Search-by-designation is then expressed via existing `:in` / `:not-in` token
   modifiers against an implicit ValueSet (typically authored using the VCL
   syntax codified at the IG-guidance VCL page); no new search modifier is
   added to FHIR Core's search machinery.

The substantive normative change to **FHIR Core (HL7/fhir)** is therefore the
addition of a single well-known concept-property (`designation`) with the
operator/wildcard/modifier semantics above. The broader VCL syntax that makes
the implicit ValueSet ergonomic is being maintained outside HL7/fhir master
(see `https://build.fhir.org/ig/FHIR/ig-guidance/vcl.html`), and is out of
scope for this ticket's HL7/fhir change.

## Feature Proposal

### Problem Statement

Today there is no way to express, in a single FHIR REST search, queries such as:

- "All medication prescriptions where `.code` has an official designation
  matching `tylenol`"
- "All medication prescriptions where `.code` is a concept with an
  `RXTERM_FORM` property matching `Tab`"

`ValueSet.compose.include.filter` already supports filtering by code-system
properties, but there is no standardized way to filter by **designation text**
(display/synonym text) at all, and no standardized property identifier for
designation matching. As a result, IGs (e.g., the Da Vinci PDex Drug Formulary
guide) have invented per-IG indirection resources (e.g., MedicationKnowledge as
a search-only side-cache) just to expose terminology designations to search.
This does not scale: every IG ends up defining a different cache shape and
overlapping definitions.

### Proposed Change

Add the `designation` well-known concept-property to the
`http://hl7.org/fhir/concept-properties` CodeSystem and document its
behaviour as a ValueSet filter property.

**Concrete authoring change in `HL7/fhir`:**

1. Add a new `<concept>` to `source/codesystem/codesystem-concept-properties.xml`
   with:
   - `code = "designation"`
   - `display = "Designation"`
   - `definition` describing:
     - That this property identifies the textual designations (display and
       synonyms) of the concept.
     - That when used in `ValueSet.compose.include.filter.property` with
       `operator = "="`, the `value` is matched **case-insensitively** as a
       wildcard pattern in which `*` MUST be explicit (so a value with no
       `*` is an anchored full-string match against the designation text).
     - That the property name MAY be qualified by:
       - a `:system#code` segment to constrain `designation.use`
         (system optional, `#` required), e.g.
         `designation:|preferredForLanguage`,
         `designation:http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra|preferredForLanguage`.
       - a `@language-range` segment (RFC 4647) to constrain
         `designation.language`, e.g. `designation:@en,en-*`.
       - both forms combined: `designation:…|preferredForLanguage@en`.
   - `<property><code value="dataType"/><valueCode value="string"/></property>`.

2. Cross-reference the new property from
   `source/valueset/valueset-notes.xml` in the section that explains
   `compose.include.filter` semantics, so readers know that `designation`
   is a well-known filter property with the wildcard / modifier semantics
   above (rather than a CodeSystem-specific, free-form property).

3. Add or extend an example in `source/valueset/valueset-example-filter.xml`
   (or a new `valueset-example-designation-filter.xml` registered alongside
   it via `source/valueset/list-ValueSet-examples.xml` and the spec's
   examples plumbing) showing a `compose.include.filter` using
   `property = "designation"`, `op = "="`, `value = "*tylenol*"`.

The original ticket title (`:term-filter` token search modifier) does **not**
get implemented: no entry is added to
`source/searchparameter/codesystem-search-modifier-code.xml`, and no new
section is added to `source/search.html`. Search-side use of designation
filtering is achieved by an `:in` / `:not-in` token modifier against an
implicit (VCL-authored) ValueSet that uses the new `designation` filter
property — and that VCL surface lives in HL7/ig-guidance, not in HL7/fhir.

### Design Rationale

- **Reuses existing machinery.** `ValueSet.compose.include.filter` already
  supports property-based filtering of CodeSystem content; reusing the
  property mechanism (rather than introducing a new search modifier
  `:term-filter`) keeps the search syntax simple and lets servers continue
  to express terminology constraints via implicit ValueSets and the
  existing `:in` / `:not-in` modifiers.
- **Aligns with Lloyd McKenzie's 2023-07-10 guidance** that the search
  syntax must not be able to express anything that
  `ValueSet.compose` cannot — by surfacing the capability as a
  CodeSystem property rather than a search modifier, parity with
  `ValueSet.compose` is automatic.
- **Standardises a well-known property** so that IGs no longer have to
  define their own per-system property names or cache resources to make
  designation text searchable. This was the explicit driver in the
  Da Vinci PDex Formulary discussion that originated the ticket.
- **Wildcard with explicit `*`** (rather than implicit `contains`) matches
  the precedent set by SNOMED ECL Description Filters
  (https://confluence.ihtsdotools.org/display/DOCECL/6.8+Description+Filters)
  cited by Peter Jordan, and gives IG authors a single rule that
  collapses to anchored exact-match when `*` is omitted.
- **Optional `use` and `language` qualifiers** are needed because real
  designations are typed by `Designation.use` and tagged by
  `Designation.language`; without these qualifiers the property could
  only match "any designation of any kind" which is too loose for
  formulary/drug-name use cases.
- **Alternative considered and rejected:** a new `:term-filter` token
  search modifier (the original proposal). Rejected because it would
  duplicate `ValueSet.compose` semantics in the search layer, contradict
  the WG's "search ⊆ ValueSet.compose" principle, and require every
  server to implement a new modifier instead of reusing the existing
  ValueSet-expansion pipeline.
- **Alternative considered and out of scope here:** a `vcl` well-known
  filter property whose value is a VCL expression. Michael Lawley noted
  this is not strictly necessary because `compose.include.valueSet`
  pointing at a VCL implicit ValueSet already covers it; the VCL piece
  is being progressed in HL7/ig-guidance separately.

- `ValueSet.compose` and the `$expand` operation — implementers expanding ValueSets that use the new property must implement the wildcard / use / language matching against `CodeSystem.concept.designation` (and equivalent designation sources).
- `:in` / `:not-in` token search modifiers — unchanged in definition, but are the recommended way to perform designation-based search by combining with an implicit ValueSet that uses the new property.

- **Spec build:** run the publisher locally to confirm the new `concept-properties` entry shows up in `concept-properties.html` (or the equivalent generated page) and that the new ValueSet example renders.
- **CodeSystem expansion:** expand `http://hl7.org/fhir/ValueSet/concept-properties` (or the equivalent generated full ValueSet) and verify `designation` is present.

- **Edge cases worth exercising:**
  - `value = "tylenol"` — anchored, full-string, case-insensitive match only; should NOT match a designation `Tylenol PM`.
  - `value = "*tylenol*"` — substring, case-insensitive; SHOULD match `Tylenol PM` and `Children's Tylenol`.
  - `designation:@en` — only matches designations whose `language` lies inside the RFC 4647 range `en`.
  - `designation:|preferredForLanguage` — system-less use code; matches any designation whose `use.code = preferredForLanguage` regardless of system.
  - Combined: `designation:http://…/hl7TermMaintInfra|preferredForLanguage@en = "*tylenol*"`.
- **Interoperability:** confirm via Zulip `[terminology]` that at least one widely deployed terminology server (Ontoserver, Snowstorm, tx.fhir.org) plans to implement the new well-known property semantics; this is the same audience that drove the ticket.

3. **Wildcard escape.** The resolution comment specifies `*` as the wildcard but does not explicitly define an escape mechanism if a designation literally contains `*`. The Vocab WG should decide whether to (a) say nothing, (b) define `\*` as a literal-star escape, or (c) declare literal `*` un-matchable. This should be resolved before publication.
4. **`use` qualifier when no system is given.** The resolution example `designation:|preferredForLanguage` uses an empty system. Confirm that the spec wording accepts an empty system as "match `use.code` regardless of system" (per the example), versus "match only when `use.system` is absent". The committee should choose explicitly.
5. **Search-side guidance.** The original ticket asked for a search modifier; the resolution defers to `:in` / `:not-in` against VCL implicit ValueSets. Should `source/search.html` carry a short non-normative pointer ("for designation-based search, use `:in` against an implicit ValueSet that filters by the `designation` concept-property") so implementers find the pattern? This is a possible scope addition worth confirming with the WG.
