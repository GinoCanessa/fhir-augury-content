## Resolution Summary

The resolution is **Persuasive with Modification**. Rather than changing
`List.source` cardinality from `0..1` to `0..*`, the work group decided to add a
sibling element `List.contributors` (0..*, `Reference`) with the same target
profiles as `List.source`. Verbatim from the resolution:

- **Element:** `contributors`, `0..*`, `Reference` with the same targets as
  `List.source`.
- **Short:** "who helps maintain the list".
- **Definition:** "Entities who have the ability to adjust the contents of the
  list".
- **Comments:** "If both 'source' and 'contributors' are present, then 'source'
  has primary responsibility. If contributors are present without source, then
  all contributors have shared responsibility. The source SHOULD NOT be also
  listed as a contributor. To indicate a practitioner working for an
  organization, use PractitionerRole, even if as a contained resource rather
  than having separate repetitions of this element for Practitioner and
  Organization."
- **Search:** Define a new `SearchParameter` for `List.contributors`, plus a
  combined search parameter that searches across both `List.source` and
  `List.contributors`.

`List.source` itself is **not** changed — its cardinality stays `0..1` and its
type/targets are unchanged.

## Feature Proposal

### Problem Statement

`List.source` records the entity responsible for deciding what is in the list.
After the multi-target expansion of agent references, callers commonly know
multiple participating entities (e.g., a `Practitioner` and an `Organization`)
that both helped maintain the list. The current `0..1` cardinality forces a
choice: either drop one party, or fall back to `PractitionerRole`, which only
works when such a resource exists. The reporter requests a way to record
multiple maintaining parties without losing the "primary responsibility"
semantics that `source` carries today.

### Proposed Change

Add a new sibling element on the `List` resource and the matching search
parameters. Specifically:

**StructureDefinition (`source/list/structuredefinition-List.xml`)**

Insert a new element immediately after the existing `List.source` element
(currently at lines 311–344):

- `id` / `path`: `List.contributors`
- `short`: `who helps maintain the list`
- `definition`: `Entities who have the ability to adjust the contents of the list`
- `comment`: `If both 'source' and 'contributors' are present, then 'source' has primary responsibility. If contributors are present without source, then all contributors have shared responsibility. The source SHOULD NOT be also listed as a contributor. To indicate a practitioner working for an organization, use PractitionerRole, even if as a contained resource rather than having separate repetitions of this element for Practitioner and Organization.`
- `min`: `0`, `max`: `*`
- `type`: `Reference` with `targetProfile` entries identical to `List.source`:
  - `http://hl7.org/fhir/StructureDefinition/Practitioner`
  - `http://hl7.org/fhir/StructureDefinition/PractitionerRole`
  - `http://hl7.org/fhir/StructureDefinition/Patient`
  - `http://hl7.org/fhir/StructureDefinition/Device`
  - `http://hl7.org/fhir/StructureDefinition/Organization`
  - `http://hl7.org/fhir/StructureDefinition/RelatedPerson`
  - `http://hl7.org/fhir/StructureDefinition/CareTeam`
  - `http://hl7.org/fhir/StructureDefinition/Group`
- `isSummary`: follow `List.source` precedent — set `true` (matches the
  existing maintenance/agent semantics already in the summary view).
- Mappings: copy the `w5` mapping (`FiveWs.author`) to keep the FiveWs view
  consistent; the `rim`/`xds` mappings are tied to the singular `source`
  semantics and may be omitted (or left for a workgroup mapper to fill in
  during review).
- `<snapshot>` block in the same StructureDefinition is generated by the HL7
  publisher and must NOT be edited by hand.

**SearchParameter bundle (`source/list/bundle-List-search-params.xml`)**

Add two new entries, modelled on the existing `List-source` entry (lines
158–175):

1. `List-contributors`
   - `id`: `List-contributors`
   - `url`: `http://hl7.org/fhir/build/SearchParameter/List-contributors`
   - `code`: `contributors`
   - `type`: `reference`
   - `expression`: `List.contributors`
   - `description`: `Who helps maintain the list`
   - `processingMode`: `normal`
   - `standards-status` extension: `normative` (same as siblings)
   - `build/StructureDefinition/path` extension: `valueString` =
     `List.contributors`

2. `List-source-or-contributors` (combined parameter requested by the
   resolution; final `code` to be confirmed by FHIR-I)
   - `id`: `List-source-or-contributors`
   - `url`:
     `http://hl7.org/fhir/build/SearchParameter/List-source-or-contributors`
   - `code`: `source-or-contributors` (proposed)
   - `type`: `reference`
   - `expression`: `List.source | List.contributors`
   - `description`: `Source or contributors of the list`
   - `processingMode`: `normal`
   - `standards-status` extension: `normative`
   - `build/StructureDefinition/path` extension: `valueString` =
     `List.source | List.contributors`

The combined search parameter targets the same set of resource types as
`List.source` and `List.contributors`, so no additional `target` declarations
are required beyond what the publisher derives from the expression.

**Narrative**

Update the prose in `source/list/list-notes.xml` (or
`source/list/list-introduction.xml` if the narrative for "source" lives there)
to introduce `contributors`, explain the "primary vs. shared responsibility"
rule, and reference the new search parameters. The exact insertion point is
discovered during implementation; both files exist in the clone.

**Examples (optional but recommended)**

Add or update at least one example in `source/list/` (e.g.,
`list-example.xml` or `list-example-medlist.xml`) to include `contributors`
alongside `source`, demonstrating the documented "source = primary,
contributors = shared" pattern.

### Design Rationale

The original ticket requested promoting `source` to `0..*`. The work group
chose the modified path because:

1. Bumping cardinality on an existing summary element is a backward-
   incompatible change for every consumer that already reads `source` as a
   single value (parsers, FHIRPath, terminology pipelines).
2. The single-`source` semantics ("primary responsibility / author") are
   meaningful and worth preserving. Splitting the concept into `source`
   (primary) plus `contributors` (shared) lets implementers express the
   richer relationship the ticket describes without erasing the existing
   semantic anchor.
3. The combined SearchParameter (`List.source | List.contributors`) gives
   implementers a single search code that returns lists where a given party
   is involved in any maintenance role — the practical capability the
   reporter wanted.
4. PractitionerRole remains the recommended way to bind a practitioner to an
   organization for a *single* contributor; the new element is for the
   distinct case of multiple independent contributing parties.

### Breaking Changes

No breaking changes for parsers or stored data. `List.source` keeps its
existing cardinality (`0..1`), type, target profiles, and summary flag.
`List.contributors` is a brand-new optional element (`0..*`); existing
instances remain valid.

A consumer that was *implicitly* assuming "the only maintainer reference on a
List is `source`" may now miss data if it does not also read
`contributors`, but that is a forward-compatibility concern for downstream
tooling, not a breaking change to the schema.

## Implementation Plan

### Step-by-Step Tasks

#### HL7/fhir

1. **Add `List.contributors` element to the StructureDefinition**
   - File: `source/list/structuredefinition-List.xml`
   - Action: Insert a new `<element>` block immediately after the existing
     `List.source` element (after current line 344, before
     `<element id="List.orderedBy">` at line 345).
   - Details:
     - `id` and `path`: `List.contributors`
     - `short`: `who helps maintain the list`
     - `definition`: `Entities who have the ability to adjust the contents of the list`
     - `comment`: per the resolution wording quoted in the Resolution
       Summary above.
     - `min`: `0`; `max`: `*`
     - `type`: `Reference` with the eight `targetProfile` entries identical
       to `List.source` (Practitioner, PractitionerRole, Patient, Device,
       Organization, RelatedPerson, CareTeam, Group).
     - `isSummary`: `true` (mirrors `List.source`).
     - Mapping: `<mapping><identity value="w5"/><map value="FiveWs.author"/></mapping>`.
     - Do **not** edit the `<snapshot>` block — the publisher regenerates
       it from the `<differential>`.

2. **Add `List-contributors` SearchParameter**
   - File: `source/list/bundle-List-search-params.xml`
   - Action: Add a new `<entry>` containing a `<SearchParameter>` modelled
     on the existing `List-source` entry (lines 158–175).
   - Details:
     - `id`: `List-contributors`
     - `url`: `http://hl7.org/fhir/build/SearchParameter/List-contributors`
     - `description`: `Who helps maintain the list`
     - `code`: `contributors`
     - `type`: `reference`
     - `expression`: `List.contributors`
     - `processingMode`: `normal`
     - Include the `structuredefinition-standards-status` extension with
       `valueCode` = `normative` and the `build/StructureDefinition/path`
       extension with `valueString` = `List.contributors`, matching the
       sibling search-parameter style.

3. **Add combined `List-source-or-contributors` SearchParameter**
   - File: `source/list/bundle-List-search-params.xml`
   - Action: Add a second new `<entry>` for the combined search parameter
     requested by the resolution.
   - Details:
     - `id`: `List-source-or-contributors`
     - `url`:
       `http://hl7.org/fhir/build/SearchParameter/List-source-or-contributors`
     - `description`: `Source or contributors of the list`
     - `code`: `source-or-contributors` (proposed; confirm with FHIR-I that
       the dash form is preferred over e.g. `maintainer`).
     - `type`: `reference`
     - `expression`: `List.source | List.contributors`
     - `processingMode`: `normal`
     - Include the `structuredefinition-standards-status` extension with
       `valueCode` = `normative` and the `build/StructureDefinition/path`
       extension with `valueString` =
       `List.source | List.contributors`.

4. **Update the List narrative to describe `contributors` and the new search parameters**
   - File: `source/list/list-notes.xml` (and/or
     `source/list/list-introduction.xml` — pick the file where the existing
     `source` discussion lives; both are present in the clone).
   - Action: Add a paragraph (or short subsection) introducing the
     `contributors` element, stating the "source = primary responsibility,
     contributors = shared responsibility, source SHOULD NOT also appear in
     contributors" rule, and noting that `PractitionerRole` (possibly
     contained) is the preferred way to express a single
     practitioner-in-organization.
   - Details: Mention both the `contributors` and the combined
     `source-or-contributors` search parameters so implementers can
     discover them from the resource page.

5. **Demonstrate the new element in an example**
   - File: at least one of `source/list/list-example.xml`,
     `source/list/list-example-medlist.xml`, or
     `source/list/list-example-long.xml`.
   - Action: Add a `<contributors>` reference (e.g., to an `Organization`)
     alongside the existing `<source>` reference in one of the
     human-curated examples to illustrate combined usage.
   - Details: Keep the example small; the goal is to show that `source`
     and `contributors` are populated independently and not to introduce
     new conformance fixtures.

### Open Questions

- **Final `code` for the combined search parameter.** The resolution says
  "a combo search parameter that looks at both source and contributor" but
  does not name it. This plan proposes `source-or-contributors`; FHIR-I
  should confirm before merge (alternatives: `maintainer`,
  `source-contributor`, `responsible`).
- **Element name spelling.** The resolution text uses both "contributor"
  (in prose) and "contributors" (in the proposed element name). This plan
  uses `contributors` as the element name verbatim from the resolution but
  uses `contributor` in the SearchParameter `code` of an alternative
  naming if FHIR-I prefers a singular search code.
- **`isSummary` on `List.contributors`.** Set to `true` here to mirror
  `List.source`. Confirm the work group is comfortable widening the summary
  view; if not, change to `false` in Task 1.
- **RIM / XDS mappings.** `List.source` carries `rim` and `xds` mappings
  tightly bound to a single-author concept. This plan omits them on
  `contributors`; a HL7 mapper should add equivalents if the work group
  wants the FiveWs / RIM / XDS views to remain complete.
- **Narrative file location.** Both `source/list/list-notes.xml` and
  `source/list/list-introduction.xml` exist in the clone. Implementer
  should add the narrative to whichever file currently discusses `source`
  (verify during implementation).
