### Resolution Description

> Now that we have moved extension definitions out of the core spec, we will do the following:
> 1. Update the publisher to be able to check the core spec and any dependency IGs for use of extensions defined in that IG
> 2. Update the tooling to render such cross-referenced examples in the examples tab for the extension
> 3. Spit out a warning if there are no examples that reference an extension either in the extension IG itself or an ancestor
> 4. Set an expectation for work groups to clean up all said warnings.

### Key Context from Comments

- **2023-02-12 (mckenzie):** Extensions now live in a separate IG (`hl7.fhir.uv.extensions`), making this "more complicated" — need to determine where examples live and how to detect missing ones.
- **2022-09-23 (mckenzie):** Shifted from FMG to FHIR-I. Can now add to official methodology. Requested post-R5 tooling tracker for warnings.

## Feature Proposal

### Goal

Ensure every approved FHIR extension has at least one discoverable usage example — either within its own IG, in the core spec, or in a dependency/ancestor IG — and that the IG Publisher provides clear feedback when examples are missing.

### Current State

1. **`HL7/fhir-extensions` repo** contains **~689 extension StructureDefinitions** with **zero example instances** in the IG itself.
2. The IG configuration (`extensions-ig.xml`) sets `no-usage-check = true`, suppressing all missing-example warnings.
3. The IG Publisher's validation (`PublisherProcessor.validateSD()`, line 718–724) already warns when an extension has no examples — but only checks **local IG files** via `countUsages()`.
4. The IG Publisher's renderer (`StructureDefinitionRenderer.references()`, lines 2307–2367) **already finds cross-IG examples** from three sources:
   - Core spec R5 examples (via `sdmap.details` from `hl7.fhir.r5.core`)
   - Dependency IGs (via `relatedIgs` loop)
   - Special `r5-examples.json` + `hl7.fhir.r5.examples` package (for `hl7.fhir.uv.extensions` only)
5. The `exampleList()` / `exampleTable()` methods (lines 1571–1615) only render **local IG examples** — cross-IG examples appear only in the `references()` "Usage" section.

### Proposed Changes

Four work streams aligned with the resolution:

| # | Resolution Item | Change Summary |
|---|-----------------|----------------|
| 1 | Check core spec + dependency IGs | Extend `countUsages()` to also check cross-IG sources (core spec, related IGs, R5 examples) |
| 2 | Render cross-referenced examples in examples tab | Extend `exampleList()`/`exampleTable()` to include cross-IG examples from `references()` sources |
| 3 | Warn if no examples anywhere | After cross-IG check, emit warning; remove `no-usage-check = true` from extensions IG |
| 4 | Set work group expectations | Add guidance to IG authoring documentation; work groups clean up remaining warnings |

## Detailed Implementation Plan

### Phase 1: Extend Validation to Check Cross-IG Examples

**Goal:** Make `PublisherProcessor.validateSD()` aware of examples from dependency IGs, core spec, and related packages.

#### Step 1.1: Extract cross-IG example discovery into a shared method

**File:** `PublisherProcessor.java`

Currently, `countUsages()` (line 1795) only iterates `pf.fileList` (local IG resources). The `StructureDefinitionRenderer.references()` method (line 2244) already has the full cross-IG logic but computes it per-extension during rendering.

**Changes:**
- Create a new method `countCrossIGUsages(String fixedUrl, StructureDefinition sd)` that checks:
  1. **Local IG files** (existing `countExtensionUsage()` logic)
  2. **Core spec R5 examples** via `sdmap.details` from `hl7.fhir.r5.core` package (mirror lines 2307–2320 of `StructureDefinitionRenderer`)
  3. **Related IGs** via the `relatedIgs` list (mirror lines 2345–2353)
  4. **R5 examples package** for extensions IG specifically (mirror lines 2355–2367)
- Cache the loaded `sdmap.details` and `relatedIgs` data at the `PublisherProcessor` level to avoid reloading per-extension

#### Step 1.2: Update `validateSD()` to use cross-IG counts

**File:** `PublisherProcessor.java` (lines 717–724)

**Current code (lines 718–724):**
```java
if (!this.pf.noUsageCheck) {
  if (sd.getType().equals("Extension")) {
    int c = countUsages(getFixedUrl(sd));
    if (c == 0) {
      f.getErrors().add(new ValidationMessage(..., "The Implementation Guide contains no examples for this extension", WARNING));
      r.getErrors().add(new ValidationMessage(..., "The Implementation Guide contains no examples for this extension", WARNING));
    }
  }
```

**New logic:**
```java
if (!this.pf.noUsageCheck) {
  if (sd.getType().equals("Extension")) {
    int localCount = countUsages(getFixedUrl(sd));
    if (localCount == 0) {
      int crossIGCount = countCrossIGUsages(getFixedUrl(sd), sd);
      if (crossIGCount == 0) {
        // No examples anywhere — WARNING
        f.getErrors().add(new ValidationMessage(...,
          "No examples found for this extension in this IG, the core spec, or any dependency IGs",
          WARNING));
      } else {
        // Examples exist in dependency/core — INFORMATION only
        f.getErrors().add(new ValidationMessage(...,
          "This IG contains no local examples for this extension, but " + crossIGCount +
          " usage(s) found in dependency IGs or the core specification",
          INFORMATION));
      }
    }
  }
}
```

### Phase 2: Render Cross-IG Examples in the Examples Tab

**Goal:** Show cross-referenced examples from dependency IGs and core spec in the extension's examples tab, not just in the references/usage section.

#### Step 2.1: Extend `exampleList()` and `exampleTable()`

**File:** `StructureDefinitionRenderer.java` (lines 1571–1615)

**Changes:**
- Add a new parameter or method variant that also accepts cross-IG examples
- After rendering local examples, add a section for cross-IG examples with clear source attribution:
  ```html
  <li><a href="...">Patient-example</a> <span class="source">(from hl7.fhir.r5.core)</span></li>
  ```
- Reuse the example discovery logic from `references()` (which already loads core spec via `sdmap.details`, related IGs, and R5 examples)

#### Step 2.2: Update `PublisherGenerator` to pass cross-IG examples

**File:** `PublisherGenerator.java` (lines 1978–1992)

The generator creates `example-list` and `example-table` fragments. Update it to:
1. Collect cross-IG examples using the same logic as `references()`
2. Pass them to the extended `exampleList()` / `exampleTable()` methods
3. Generate combined fragments that include both local and cross-IG examples

### Phase 3: Enable Warnings for the Extensions IG

**Goal:** Remove the `no-usage-check` suppression so the new cross-IG-aware checks can validate extension coverage.

#### Step 3.1: Update `extensions-ig.xml`

**File:** `HL7/fhir-extensions` → `input/extensions-ig.xml` (line 281–287)

**Current:**
```xml
<parameter>
  <code>
    <system value="http://hl7.org/fhir/guide-parameter-code"/>
    <code value="no-usage-check"/>
  </code>
  <value value="true"/>
</parameter>
```

**Change:** Remove this parameter block entirely, or change `<value value="true"/>` to `<value value="false"/>`.

> **Important:** This should only be done **after** Phase 1 is deployed, so that the cross-IG checking reduces false-positive warnings. Many extensions are already used in core spec R5 examples and will pass the check.

#### Step 3.2: Assess remaining warnings

After enabling checks, run the publisher on `fhir-extensions` to enumerate which extensions truly have no examples anywhere. This produces the work list for Phase 4.

### Phase 4: Work Group Cleanup and Documentation

**Goal:** Set expectations for work groups to provide examples for their extensions.

#### Step 4.1: Document the expectation

Add guidance to the IG authoring documentation (in `HL7/fhir` or the IG publisher documentation) that:
- Every extension StructureDefinition should have at least one discoverable example
- Examples can live in the extension's own IG, in the core spec, or in a dependency IG
- The publisher will warn when no examples are found

#### Step 4.2: Provide tooling support

Consider adding a publisher feature to auto-generate skeleton example resources for extensions, pre-populated with:
- The extension URL
- A placeholder value matching the extension's value type
- Appropriate context resource type

#### Step 4.3: Work group triage

Each work group responsible for extensions in `fhir-extensions` reviews the warnings for their extensions and either:
- Adds examples to the `fhir-extensions` IG (in `input/definitions/<resource>/`)
- Confirms that adequate examples exist in the core spec or other dependency IGs
- Marks specific extensions as intentionally example-free (e.g., deprecated extensions)
