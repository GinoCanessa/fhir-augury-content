## Resolution Summary

Resolution: **Persuasive with Modification**.

The committee re-opened the original (2022) "non-substantive" disposition on
2023-03-20 (Grahame Grieve / Josh Mandel, 14-0-0) and replaced it with the
following four-part resolution, reflecting that extension definitions had by
then been moved out of FHIR Core into a separate IG (HL7/fhir-extensions):

> Now that we have moved extension definitions out of the core spec, we will do
> the following:
> 1. Update the publisher to be able to check the core spec and any dependency
>    IGs for use of extensions defined in that IG.
> 2. Update the tooling to render such cross-referenced examples in the
>    examples tab for the extension.
> 3. Spit out a warning if there are no examples that reference an extension
>    either in the extension IG itself or an ancestor.
> 4. Set an expectation for work groups to clean up all said warnings.

In short: the change is a *tooling + methodology* change, not a resource
content change. The IG Publisher must learn to find example instances that use
each extension (anywhere in the extension's own IG, in its ancestor IGs, or in
FHIR Core when Core depends on the extension IG), surface them on the
extension's "Examples" tab, and warn when none exist; the FHIR-I methodology
documentation must then state the expectation that work groups eliminate
those warnings.

## Feature Proposal

### Problem Statement

Approved FHIR extensions historically shipped without instance-level examples,
so consumers had to construct usage from the StructureDefinition alone. The
original 2022 ticket asked for an example with every approved extension; a
non-substantive ("refer to FMG as a quality consideration") disposition was
recorded, but the issue was re-opened in 2023 because:

- Extension definitions had moved out of FHIR Core into the separate
  HL7/fhir-extensions IG. Examples that previously could have lived alongside
  the extension definition in Core no longer have an obvious home.
- There was no automated way to know which extensions lacked examples, so the
  "quality consideration" never produced enforceable feedback to work groups.
- The IG Publisher's per-extension "Examples" tab only surfaces examples that
  live in the same IG, not examples that live in dependent IGs (notably FHIR
  Core, which depends on fhir-extensions).

The ticket comment thread (mckenzie, 2023-02-12 and 2023-02-... onward) makes
the new framing explicit: "Now that we have extensions in a separate IG, this
is more complicated. Will need to discuss within FHIR-I where to make examples
live - and how to check if they're missing."

### Proposed Change

The implementation has three independent pieces of work, in three different
HL7 repositories:

1. **IG Publisher — cross-IG example discovery (HL7/fhir-ig-publisher).**
   For every StructureDefinition with `kind = complex-type` /
   `type = Extension` defined in the IG being built, the publisher must scan:
   (a) the example resources of the IG itself, and
   (b) the example resources of every IG that depends on this IG (transitively),
   for any element with `url = <extension's canonical URL>` (covering both
   modifier and non-modifier extensions, simple and complex extensions, and
   extensions used at any path including nested extensions). Each example
   resource that contains at least one usage of the extension is recorded as a
   "usage example" for that extension.

2. **IG Publisher — Examples tab rendering (HL7/fhir-ig-publisher).**
   The per-extension page generated by the publisher already has an
   "Examples" tab populated from intra-IG examples. Extend it to additionally
   render the cross-referenced example instances discovered in (1), grouping
   them by source IG so the reader can see "Examples in this IG" vs.
   "Examples in <ancestor IG name>".

3. **IG Publisher — missing-examples QA warning
   (HL7/fhir-ig-publisher).** After (1) runs, emit a QA warning of the form
   `Extension <canonical URL> has no example instance in this IG or in any IG
   that depends on it` for every extension whose usage-example list is empty.
   The warning must be suppressible via the standard
   `suppressed-messages.txt` mechanism so that intentionally
   example-less extensions can be exempted with rationale.

4. **Methodology — work-group expectation (HL7/fhir).**
   The FHIR-I methodology pages in FHIR Core that govern extension authoring
   (`source/defining-extensions.html`, `source/extensibility.html`, and the
   surrounding `source/extensibility-definitions.html` /
   `source/extensibility-examples.html` set) state the expectation that
   every approved extension SHOULD have at least one example instance in
   the extension's own IG or in an IG that depends on it, and that work
   groups SHOULD treat the publisher's missing-examples warning as a
   ballot-blocker / quality gate. This is the "set an expectation for work
   groups to clean up all said warnings" piece of the resolution.

5. **Examples backfill — HL7/fhir-extensions and ancestor IGs.** Once (1)–(3)
   land, the tooling will surface a list of extensions without examples in
   HL7/fhir-extensions. The work-group cleanup is a follow-on, ticket-driven
   activity per extension (or per work group), not a single change in this
   ticket. This plan therefore identifies it but does not enumerate the
   per-extension authoring tasks.

### Design Rationale

- **Cross-IG discovery (rather than co-locating examples) preserves the
  separation that motivated extracting extensions to fhir-extensions in the
  first place.** Examples often only make sense in the context of a host
  resource (`Patient`, `Observation`, …), so forcing them into the extension
  IG would produce contrived snippets. Letting the example live wherever it
  naturally belongs and discovering it via the extension's canonical URL
  matches the "extensions live in their IG, examples live where they're
  used" pattern already used elsewhere in FHIR tooling.
- **The publisher is the natural enforcement point.** It already builds the
  per-extension page and already emits QA warnings. Adding one more
  cross-IG sweep keeps the rule machine-checkable and removes the manual
  "FMG quality consideration" framing that the 2022 disposition relied on
  (and which the comment trail confirms never worked).
- **Methodology text in FHIR Core (not in fhir-extensions) is correct
  because FHIR Core owns the extensibility methodology.**
  `source/defining-extensions.html` and `source/extensibility.html` are the
  normative pages that other IGs cite; the new expectation belongs there.

## Implementation Plan

### Prerequisites

- Confirm with FHIR-I that the SHOULD framing (rather than SHALL) is the
  intended strength for the methodology text. The resolution wording
  ("set an expectation for work groups to clean up all said warnings")
  reads as SHOULD; a SHALL would block ballot.
- Confirm the suppression mechanism for the new warning. Standard IG
  Publisher behavior is `input/suppressed-messages.txt` (or
  `suppressed-messages.txt` at repo root in FHIR Core); the methodology
  text should reference whichever mechanism is canonical.
- (Out of cache) An up-to-date briefing for HL7/fhir-ig-publisher is
  required before the per-file publisher tasks below can be made
  concrete. The publisher repo is not in the Augury cache, so its tasks
  remain at the requirements level here.
- (Out of cache) A briefing for HL7/fhir-extensions is required before
  the per-extension backfill can be enumerated.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add "Examples for approved extensions" subsection.**
   - File: `source/defining-extensions.html`
   - Action: Insert a new subsection (heading text along the lines of
     "Examples for Approved Extensions") near the end of the page,
     before any "See also" / cross-link block. Anchor it (e.g.,
     `<a name="examples"></a>`) so other pages can link to it.
   - Details: The subsection MUST state, as authored prose:
     (a) every approved extension SHOULD have at least one example
     instance demonstrating its use; (b) the example MAY live in the
     extension's own IG OR in any IG that depends on the extension's IG
     (the IG Publisher will discover and surface such cross-IG examples
     on the extension's "Examples" tab); (c) the IG Publisher emits a
     QA warning when no such example exists, and work groups are
     expected to either supply an example or suppress the warning with
     a documented rationale via the standard suppression mechanism;
     (d) reference the suppression mechanism by name (do not just say
     "suppress" — name the file).

2. **Cross-link from the umbrella "Extensibility" page.**
   - File: `source/extensibility.html`
   - Action: Add a single-paragraph cross-link to the new
     `defining-extensions.html#examples` anchor in the part of the
     page that introduces "extension authoring".
   - Details: Use the existing `[%extensions-location%]…` style for the
     link if cross-IG, or a relative `defining-extensions.html#examples`
     link if intra-Core. Keep the wording short — one sentence
     stating that approved extensions SHOULD ship with at least one
     example, with the link.

3. **Reconcile sibling extensibility pages.**
   - File: `source/extensibility-definitions.html`
   - Action: Search for any text asserting examples are entirely
     optional or that they are out of scope for extensions. Update such
     text to reference the new SHOULD expectation, or remove it if it
     contradicts the new methodology.
   - Details: This is a consistency pass, not a content rewrite. If no
     contradictory text is found, leave the file untouched and record
     that fact in the PR description.

4. **Annotate the extensibility-examples page.**
   - File: `source/extensibility-examples.html`
   - Action: Add a short note (1–2 sentences) at the top pointing
     readers at the per-extension "Examples" tab in the extension IG
     (which, after the publisher change lands, will include cross-IG
     usage examples).
   - Details: Use the same `[%extensions-location%]…` link style used
     elsewhere on the page (line ~157 of `source/extensibility.html`
     uses `[%extensions-location%]StructureDefinition-…html`; mirror
     that pattern).

#### HL7/fhir-ig-publisher (out of Augury cache — requirements-level)

5. **Index extension usages across the IG and its dependents.**
   - File: *Not enumerated — requires HL7/fhir-ig-publisher briefing.*
   - Action: After the IG's example resources are loaded, walk every
     example for elements whose `url` matches the canonical URL of any
     extension defined in the IG (or in an IG it depends on, going
     up the dependency chain). Record the (`extensionCanonicalURL`,
     `exampleResourceId`, `sourceIgPackageId`) triples.
   - Details: Must handle modifier extensions, nested extensions,
     extensions on primitives (`_field`), and extensions inside
     `contained` resources. The match key is the canonical URL,
     including any version suffix when present.

6. **Render cross-IG usage examples on the extension page.**
   - File: *Not enumerated — requires HL7/fhir-ig-publisher briefing.*
   - Action: On each extension's generated page, in the existing
     "Examples" tab, add a section listing the cross-IG usage examples
     discovered in task 5, grouped by source IG.
   - Details: Each entry should be a hyperlink to the example
     resource's page in its source IG, plus a short snippet showing
     the extension element in context.

7. **Emit "missing examples" QA warning.**
   - File: *Not enumerated — requires HL7/fhir-ig-publisher briefing.*
   - Action: After tasks 5 and 6, for every extension defined in the
     IG that has zero usage examples (intra-IG and cross-IG), emit a
     QA warning of the form `Extension <canonicalURL> has no example
     instance in this IG or in any IG that depends on it`.
   - Details: Warning, not error. Suppressible via the standard
     `suppressed-messages.txt` mechanism. The warning text must
     include the canonical URL so suppression entries can target the
     specific extension.

#### HL7/fhir-extensions (out of Augury cache — follow-on, per-extension)

   - File: Per-extension; not enumerable here.
   - Action: After tasks 5–7 are deployed, run an IG build, harvest

  conformance behavior; downstream implementer impact is limited to
  receiving better documentation and (eventually) more example
