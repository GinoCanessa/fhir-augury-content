## Resolution Summary

**Resolution type:** Persuasive.

**Resolution description (verbatim):** *"Grahame will enhance the tooling and also propose the initial recommended set."*

During the 2020-02 WGM (Thu Q3), Methodology & Modeling (MnM) agreed that FHIR
should publish a **recommended set of datatypes for elements that express a
choice of timing** (for example, `effective[x]` / `occurrence[x]` style
elements). Committees would be expected to author timing choices using this
set; deviations would require MnM approval. Two concrete deliverables were
approved:

1. **Define** the recommended set of timing-choice datatypes (to be proposed
   by Grahame Grieve).
2. **Enhance the publisher tooling** so it flags resource/profile definitions
   whose timing choice deviates from the recommended set (a QA/compliance
   check), allowing committees to seek MnM approval before shipping a
   non-compliant choice.

The ticket is labelled `Deferred`; no cross-references or applied commits have
been recorded against FHIR-26023, so this plan treats the resolution as still
pending first implementation.

## Feature Proposal

### Problem Statement

Across FHIR Core and downstream IGs, different committees pick different
combinations of datatypes when modelling an element whose meaning is *"when
this happened / will happen"*. Observed variants in the current spec include
`dateTime | Period`, `dateTime | Period | Timing`, `dateTime | Period |
Timing | instant`, `date | Period`, and other ad‑hoc mixes. Without a
normative or recommended set:

- Implementers have to handle a different choice-of-types everywhere a
  timing concept appears, complicating generators, validators, and client
  SDKs.
- Profile authors copy whichever pattern is nearest, which propagates the
  inconsistency into IGs.
- MnM has no mechanical hook to review deviations, so variation is invisible
  until ballot.

MnM's resolution addresses both sides: a specification-level recommendation
plus a tooling check.

### Proposed Change

This is a **two‑part** change. Only the first part is in scope for the
`HL7/fhir` cached repo; the second part lives in the separate publisher
tooling repositories, which are not cached in this environment.

**Part A — Specification (HL7/fhir).** Add normative-track prose that
defines the recommended set of timing-choice datatypes and explains the
MnM deviation process. The authoritative home for "how FHIR describes a
choice of datatypes" today is `source/formats.html` under the
`Choice of Datatypes` section (anchor `#choice`, around line 192). The
recommended-set guidance belongs either there or on a new subsection linked
from both `formats.html#choice` and `datatypes.html#Timing`.

Proposed initial recommended set (to be finalised by Grahame as stated in
the resolution — this plan names the set so downstream tasks are concrete;
MnM/Grahame remains the approver):

| Level of precision needed | Recommended choice |
|-|-|
| Point-in-time event, instant-level precision | `dateTime \| Period` (default) |
| Point-in-time event, day-level precision only | `date \| Period` |
| Event that may recur / schedule | `dateTime \| Period \| Timing` |
| Event that may recur and needs sub-second precision | `instant \| Period \| Timing` |

The prose must:

- State that elements expressing *timing* (i.e. when an event occurred,
  will occur, or recurs) SHOULD use one of the recommended choices above.
- Require committees wishing to deviate to obtain MnM approval and record
  the rationale in the resource's scope/boundaries section.
- Cross-link to `datatypes.html#Timing` and to `profiling.html` guidance.

**Part B — Tooling (out-of-repo, tracked here for completeness).** Enhance
the HL7 publisher / IG-publisher QA to:

- Detect elements whose `type[*].code` set looks like a timing choice (the
  presence of `Timing`, or any combination of `dateTime`/`date`/`instant`
  with `Period`).
- Compare against the recommended set above and emit a QA warning (not an
  error) when the combination is not recommended, with a message pointing
  to the new `formats.html` section and to this ticket's MnM-approval path.
- Provide an allow‑list mechanism (e.g. a `suppressed-messages.txt` entry
  or a dedicated `mnm-approved-timing-choices.json` data file) so that
  MnM-approved deviations do not re-raise the warning on every build.

Because the publisher sources are **not** in the `HL7/fhir` clone (per the
briefing, `tools/` is a generated/harness area and the bespoke publisher is
out of scope for content edits), Part B requires work in
`HL7/org.hl7.fhir.core` (Java validator / publisher) and/or
`HL7/fhir-ig-publisher` — neither of which is in the local cache, so this
plan can only state the expected behaviour, not concrete file paths there.

### Design Rationale

- **Consistency > novelty.** The MnM decision explicitly chose a
  *recommended* rather than *required* set, preserving committee
  flexibility while giving the specification a documented default. A
  normative/SHOULD statement on `formats.html` is the least intrusive way
  to express that.
- **Co-locate with the existing `Choice of Datatypes` guidance.** The
  `formats.html#choice` section is where readers already look to
  understand `[x]` elements; scattering the timing-choice rule into a
  separate page would hide it.
- **Tooling as warning, not error.** Deviations are legitimate with MnM
  approval, so the check must not break builds — it must surface in QA
  reports and be suppressible via the standard HL7 suppression mechanism
  (`suppressed-messages.txt` is already present at the repo root per the
  briefing).
- **Alternatives considered.** (a) Making the set normative-required was
  rejected implicitly by the resolution wording ("recommended"). (b)
  Publishing the set only as a ValueSet/CodeSystem of type combinations
  was not chosen because the value is in human narrative + a tooling hook,
  not in a bindable code. (c) Enforcing via FHIRPath invariants on every
  affected element was rejected as impractical across hundreds of
  resources and IGs.

### Step-by-Step Tasks

#### HL7/fhir

   - File: `source/formats.html`
   - Action: Immediately after the existing `Choice of Datatypes`
     subsection (after the `<a href="choice-elements.json">` paragraph,

   - Details: Body should (a) define what qualifies as a "timing"

   - File: `source/datatypes.html`
   - Action: Inside the `<a name="Timing">` block (starting ~line 1610),

     as part of a choice-of-types element, see the recommended set at
     [Recommended Timing Choice](formats.html#timing-choice)."*
   - Details: Keep wording short; do not duplicate the table.

3. **Add MnM-approval note to `profiling.html`.**
   - File: `source/profiling.html`
   - Action: Add a short paragraph in the guidance area for new resources

   - Details: Before editing, verify anchor names on `profiling.html` to

4. **(Optional) Extend the Timing datatype `<differential>` narrative.**
   - File: `source/datatypes/timing.xml`
   - Action: In the top-level `<differential>` element narrative

     cardinality/type constraints.
   - Details: Confirm the file is a FHIR-XML StructureDefinition (not the
     `-spreadsheet.xml` Excel sibling) before editing.

5. **Check for anchor collisions and dangling inbound links.**
   - File: repo-wide grep over `source/**/*.html` and `source/**/*.xml`.
   - Action: Search for pre-existing uses of `timing-choice` and
     `#timing-choice`; if any exist in an unrelated context, pick a
     different anchor (e.g. `#recommended-timing-choice`).
   - Details: Also confirm the new section does not shadow
     `formats.html#choice`.

#### HL7/org.hl7.fhir.core and/or HL7/fhir-ig-publisher (out-of-repo — not cached)

6. **Implement the timing-choice QA check.**
   - File: Not determinable from this environment; repo is not cached.
   - Action: In the validator/publisher QA pass, for every
     `ElementDefinition` whose `id` ends with `[x]`, compute the set of

7. **Ship the recommended-set data file.**
   - File: Not determinable here; likely a resource in the publisher

   - Details: The `HL7/fhir` clone's `suppressed-messages.txt` at the
     repo root remains the standard per-IG opt-out path for
