## Resolution Summary

**Resolution:** Persuasive.

**Resolution text (verbatim):** "Where there are recommended value sets defined in the usageContext type codes, we will add additional bindings with purpose 'preferred' pointing to those value sets."

Concretely, the resolution asks that for each concept defined in the `http://terminology.hl7.org/CodeSystem/usage-context-type` CodeSystem that lists a recommended/associated value set in its concept definition, the `UsageContext` metadata datatype gains a companion additional binding keyed off `UsageContext.code` → that recommended `ValueSet`, with **`purpose = preferred`**. The binding strength does not replace the existing binding on `UsageContext.valueCodeableConcept`; it augments it so that tooling can suggest the appropriate domain value set when a particular `UsageContext.code` is used (for example, `gender` → `administrative-gender`). The ticket description notes that the binding strength ("required, extensible") may vary by concept, but the resolution standardises the purpose on `preferred`.

## Feature Proposal

### Problem Statement

`UsageContext` is the metadata datatype used on `CanonicalResource`-family resources (CapabilityStatement, ActivityDefinition, ActorDefinition, ValueSet, etc.) to describe the context in which an artifact applies. Today, `UsageContext.code` is bound to the `usage-context-type` CodeSystem (concepts such as `gender`, `focus`, `user`, `workflow`, `task`, `venue`, and the jurisdictional context), and `UsageContext.valueCodeableConcept` is bound to a very broad "everything" value set because it is a choice-typed value carrying whatever domain concept the author wants to pin.

The concept definitions in `http://terminology.hl7.org/CodeSystem/usage-context-type` already name a recommended value set for each concept (e.g., `gender` → `administrative-gender`, `workflow` → v3 `ActEncounterCode`), but those recommendations are not expressed on the datatype itself. As a result, validators and authoring tools cannot automatically flag a `UsageContext` in which `code=gender` but `valueCodeableConcept` is taken from a completely unrelated domain. The reporter (Elliot Silver) asks that the datatype slice `valueCodeableConcept` by `code` so each pairing has its own binding, and explicitly notes that the `jurisdiction` concept has a defined value set that is not currently called out.

### Proposed Change

Update the **additional bindings** file for `UsageContext` so that each usage-context-type concept with a recommended value set has a corresponding `<additional>` entry whose `<purpose value="preferred"/>` slice of the binding is keyed via a `usage` element of the form `UsageContext.code = <concept>`.

- **File:** `source/datatypes/UsageContext-additional-bindings.xml` (already introduced under FHIR-45507 on 2025-11-04; commit `e3fe9fdef2`).
- **Shape of each additional binding (after change):**

  ```xml
  <additional>
    <key value="uc-bind-<concept>"/>
    <purpose value="preferred"/>
    <valueSet value="<recommended ValueSet canonical>"/>
    <usage>
      <code value="UsageContext.code"/>
      <valueCodeableConcept>
        <coding>
          <system value="http://hl7.org/fhir/usage-context-type"/>
          <code value="<concept>"/>
        </coding>
      </valueCodeableConcept>
    </usage>
    <any value="false"/>
  </additional>
  ```

- **Concrete edits to the existing bindings** (current state inspected at clone HEAD `db79dcd`):
  - `uc-bind-gender` — change `<purpose value="required"/>` → `<purpose value="preferred"/>`. ValueSet remains `http://hl7.org/fhir/ValueSet/administrative-gender`.
  - `uc-bind-focus` — change `<purpose value="starter"/>` → `<purpose value="preferred"/>`. ValueSet remains `http://hl7.org/fhir/ValueSet/condition-code`.
  - `uc-bind-user` — change `<purpose value="starter"/>` → `<purpose value="preferred"/>`. ValueSet remains `http://hl7.org/fhir/ValueSet/provider-taxonomy`.
  - `uc-bind-workflow` — change `<purpose value="starter"/>` → `<purpose value="preferred"/>`. ValueSet remains `http://terminology.hl7.org/ValueSet/v3-ActEncounterCode`.
  - `uc-bind-task` — change `<purpose value="starter"/>` → `<purpose value="preferred"/>`. ValueSet remains `http://terminology.hl7.org/ValueSet/v3-ActTaskCode`.
  - `uc-bind-venue` — change `<purpose value="starter"/>` → `<purpose value="preferred"/>`. ValueSet remains `http://terminology.hl7.org/ValueSet/v3-ServiceDeliveryLocationRoleType`.
  - `uc-bind-jurisdiction` — change `<purpose value="exetensible"/>` (existing typo) → `<purpose value="preferred"/>`. ValueSet remains `http://terminology.hl7.org/ValueSet/jurisdiction`.
- **Coverage check:** review the concept definitions in `http://terminology.hl7.org/CodeSystem/usage-context-type` (external — published from HL7/UTG / THO); for every concept whose definition names a recommended value set, verify a corresponding `uc-bind-<concept>` slice exists. Add any that are missing using the template above. The ticket specifically calls out `jurisdiction` as being absent from the usage-context-type listing despite having a defined value set; the existing file already includes `uc-bind-jurisdiction`, so no new slice is needed — only the purpose/typo correction.

### Design Rationale

- **Purpose = `preferred` (not `required` or `extensible`).** The resolution is explicit: purpose is `preferred`. `preferred` is defined in `codesystem-additional-binding-purpose` as a recommendation that a particular value set should be used when applicable; this matches the CodeSystem's own phrasing that these value sets are "recommended", and avoids tightening the existing binding on `valueCodeableConcept` in ways that would break deployed `UsageContext` instances that choose a different value set (for example, using a local gender code system). The current `required` strength on `gender` and the mix of `starter`/`exetensible` on the rest are inconsistent with the resolution and must all collapse to `preferred`.
- **Slicing via the `<usage>` element, not profile slicing.** The `additional binding` mechanism already supports carrying a `usage` coding, which is exactly the "slice by code" pattern the ticket describes — pairing the value set to a particular `UsageContext.code`. Using profile-level discriminated slicing of `valueCodeableConcept` would require a profile on `UsageContext`, which is neither desirable for a core datatype nor what FHIR-45507 chose when it introduced `UsageContext-additional-bindings.xml`. Re-using that mechanism keeps the change localised and leaves room for IGs to add stricter `required`/`extensible` bindings of their own.
- **Typo fix is tightly coupled.** The existing `exetensible` on `uc-bind-jurisdiction` is clearly a typo for `extensible`, but the resolution replaces it with `preferred` anyway. Fixing it in the same change avoids leaving an invalid `AdditionalBindingPurposeVS` code behind.

### Affected Repositories

- **Role:** Houses the `UsageContext` metadata datatype authoring files and the additional-bindings sidecar.

  - `source/datatypes/UsageContext-additional-bindings.xml` — update the seven existing `<additional>` entries so that each `<purpose>` is `preferred`; fix the `exetensible` typo on `uc-bind-jurisdiction`; add any missing `uc-bind-<concept>` slice for concepts in `usage-context-type` that have a recommended value set but no existing entry (coverage check).
- **Change Scope:** Minor (single authoring file, no StructureDefinition differential or narrative changes).

### Breaking Changes

- **Gender: binding strength effectively loosens from `required` to `preferred`.** The existing `uc-bind-gender` carries `purpose=required`, which in the additional-bindings model is a strong statement that the value set *must* be used when `UsageContext.code=gender`. Moving to `preferred` is a *relaxation* — existing instances that used the `administrative-gender` value set remain valid, and instances that used a different code system now become explicitly tolerated. No validator that previously passed will start failing.
- **Jurisdiction: `exetensible` → `preferred`.** The old value was not a valid `AdditionalBindingPurposeVS` code (typo), so in practice any validator enforcing that binding was already emitting an error on this authoring file. After the fix the file becomes conformant; this is a bug fix rather than a breaking change.
- **Overall:** No changes to cardinality, type, element path, or the primary binding on `UsageContext.valueCodeableConcept`. Therefore no wire-format or schema break for downstream implementations. Resources validated against the regenerated UsageContext snapshot should see either no change or a relaxation in diagnostic severity. **No breaking changes to instance conformance are introduced.**

#### HL7/fhir

1. **Update all `<purpose>` values to `preferred` and fix the jurisdiction typo**
   - File: `source/datatypes/UsageContext-additional-bindings.xml`
