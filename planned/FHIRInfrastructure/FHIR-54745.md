## Resolution Summary

The committee voted **Persuasive** on the resolution:

> "This functionality is not ready for normative and needs to be moved to another specification."

In concrete terms, the `%factory` Type Factory API currently documented inside the Normative FHIRPath page must be **removed from FHIR Core** and **spun off into a separate Implementation Guide** (an "incubator" IG, per the ticket label `Incubator`). The reporter notes the functionality is useful but not widely deployed and therefore should mature outside the Normative track.

The destination IG is **not specified by the resolution**. The follow-up Zulip thread *"fhirpath extensions in FHIR Core"* in `fhir/infrastructure-wg` (Brian Postlethwaite, 2026-03-30) explicitly asks the WG which incubator should host it (candidate raised: `HL7/api-incubator-ig`, with the caveat that it appears to be failing) and whether a new incubator should be created. That decision is a **prerequisite open question** for this work.

## Feature Proposal

### Problem Statement

The Normative FHIRPath specification page (`source/fhirpath.html`) currently documents a `%factory` variable that exposes a "class factory" of construction methods for FHIR data types — primitive constructors, `Extension`, `Identifier`, `HumanName`, `ContactPoint`, `Address`, `Quantity`, `Coding`, `CodeableConcept`, plus the general `create(type)`, `withExtension(...)`, and `withProperty(...)` helpers (lines 578–861 of the page). The page itself acknowledges immaturity:

> "Note that a future version of FHIRPath may provide a factory framework directly, in which case this factory API may be withdrawn or deprecated." (`fhirpath.html` line 583–584)

Since the FHIRPath page is on the Normative track, leaving an experimental, lightly-deployed API inside it would lock immature content into Normative status. The reporter and the WG agreed (Persuasive resolution) that the Type Factory should mature in a separate, non-normative IG before any future return to the core.

### Proposed Change

1. **In `HL7/fhir`:** delete the entire **Type Factory** section from `source/fhirpath.html`.
   - Remove the anchor `<a name="factory"> </a>` and `<h3>Type Factory</h3>` block beginning at line 579.
   - Remove all per-function subsections through the end of `%factory.withProperty(...)` — i.e., lines 578 through 861 inclusive (the block ends immediately before `<a name="txapi"></a>` / `<h3>Terminology Service API</h3>` at line 862).
   - Removed anchors that must no longer exist (and therefore must not be linked to from any HL7/fhir page after this change): `#factory`, `#fn-factory-primitives`, `#fn-factory-Extension`, `#fn-factory-Identifier`, `#fn-factory-HumanName`, `#fn-factory-ContactPoint`, `#fn-factory-Address`, `#fn-factory-Quantity`, `#fn-factory-Coding`, `#fn-factory-CodeableConcept`, `#fn-factory-create`, `#fn-factory-withExtension`, `#fn-factory-withProperty`.

2. **In `HL7/fhir`:** repair the five inbound links from the FHIR Mapping Language transform table (`source/mapping-language.html` lines 627, 633, 639, 645, 651). After deletion of the Type Factory section these `fhirpath.html#fn-factory-*` hrefs become dangling fragments. Two acceptable repairs (pick per WG decision on the destination IG):
   - **Preferred (once destination IG is known):** rewrite each href to point at the corresponding section in the new incubator IG (e.g., `<incubator-ig-canonical>/StructureDefinition-fhirpath-typefactory.html#fn-factory-CodeableConcept`).
   - **Interim (destination IG not yet stood up):** drop the `<a href=...>` wrapping and replace the cell content with the literal function name in `<code>` (e.g., `<code>%factory.CodeableConcept()</code>`), plus a one-sentence note above the table that "the `%factory.*` functions referenced as the FHIRPath equivalents are defined in a separate incubator IG (TBD)."
   - In either case also fix the pre-existing markup defect on these lines: each cell currently ends with `</a>"></td>` — a stray `">` after the closing anchor tag. The fix is to drop the stray `">` so the cell ends `</a></td>` (or, in the interim variant, `</code></td>`).

3. **In the destination incubator IG (separate repository, scope of a follow-on ticket — see Open Questions):** publish a profile/specification page that re-hosts the Type Factory content removed in (1), preserving anchor names where practical so external bookmarks can be redirected with one rewrite rule.

### Design Rationale

- The resolution wording is unambiguous: the functionality is "not ready for normative" and must move to "another specification". Wholesale removal from `fhirpath.html` (rather than, e.g., demoting a single subsection to non-normative) is the only edit that aligns with both the Normative status of the page and the Persuasive vote.
- The Type Factory section is **self-contained** within `fhirpath.html` (single `<a name="factory">` anchor block, terminating cleanly at `<a name="txapi">`), so its removal does not entangle adjacent normative content (Terminology Service API is the next sibling section and is unaffected).
- The only intra-repo coupling is the FML transform-table cross-reference; addressing it in the same change keeps the spec internally consistent. Leaving the dangling fragments would silently break the spec at publish time (the bespoke HL7 publisher does not validate fragment targets).
- Choosing the destination IG is **out of scope of the resolution as worded** — the Zulip thread shows the WG itself has not yet decided. Any plan that hard-codes a destination would be guessing; this plan therefore treats destination selection as a prerequisite/open question rather than a task.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Removes the Type Factory section from the Normative FHIRPath page and repairs the FML page's now-dangling cross-references.
- **Affected Files:**
  - `source/fhirpath.html` — delete lines 578–861 (the entire `<a name="factory">` … pre-`<a name="txapi">` block, covering all `%factory.*` documentation).
  - `source/mapping-language.html` — rewrite (or interim-strip) the five `<a href="fhirpath.html#fn-factory-*">…</a>` cells in the transform table at lines 627, 633, 639, 645, 651, and fix the stray `">` markup defect on those lines.
- **Change Scope:** Moderate. The deletion in `fhirpath.html` is mechanical, but the page is Normative; review must confirm no other normative section depends on the removed anchors.

### Breaking Changes

- **API removed from Normative FHIR Core:** the `%factory` variable and all of its methods (`{primitive}`, `Extension`, `Identifier`, `HumanName`, `ContactPoint`, `Address`, `Quantity`, `Coding`, `CodeableConcept`, `create`, `withExtension`, `withProperty`) will no longer be defined by the FHIRPath page. Implementations that rely on the documented `%factory.*` syntax against the FHIR Core spec will lose their normative reference.
- **External fragment links break:** any external page bookmarking `…/fhirpath.html#factory` or `…/fhirpath.html#fn-factory-*` will dangle. A redirect rule (or equivalent publisher-level rewrite) from the old anchors to the new IG location should be considered as part of the destination IG's standup.
- This is a **functional removal**, not a backwards-compatible deprecation. State it clearly in the FHIR Core change log / ballot note.

### Related Specifications

- **FHIR Mapping Language (`mapping-language.html`)** — the FML transform table currently documents `cc`, `c`, `qty`, `id`, `cp` as shorthand for the corresponding `%factory.*` constructors. After the move, the FML page must either link to the new IG or stop pointing at FHIR Core for these definitions. (See FHIR-42980, which independently rewrites prose in the same file.)
- The new incubator IG (when chosen) will become the canonical home for `%factory.*` and may need to declare a dependency / cross-reference back to the FHIRPath Normative page.

### Related Tickets

- **FHIR-42980:** *(FML page TODO cleanup)* — also edits `source/mapping-language.html`. Does not conflict semantically (different lines), but anyone executing both should rebase the second one onto the first to avoid accidental merge conflicts in the same file.
- The cross-reference query returned only the Zulip mention above; no other Jira tickets referencing FHIR-54745 were found via `cross-referenced` (limit 50).

### Terminology Impact

No terminology impact. The change is an HTML/prose deletion and link repair only; no CodeSystem, ValueSet, or coded element is added, removed, or rebound.

### Open Questions

1. **Which incubator IG receives the Type Factory content?** The resolution does not name one; the Zulip thread (`fhir/infrastructure-wg` > "fhirpath extensions in FHIR Core") raises `HL7/api-incubator-ig` as a candidate but flags it as "appears to be failing", and asks whether a new incubator should be created. The WG must answer this before Task 2 can use the *preferred* link variant and before redirect rules can be authored.
2. **Redirect strategy for the removed anchors.** The publisher does not natively rewrite removed fragments. Decide whether the WG wants (a) hosting-level redirects from `…/fhirpath.html#factory*` to the new IG, (b) a stub paragraph on `fhirpath.html` pointing readers to the new IG, or (c) silent removal. (b) is the lowest-risk option but contradicts the "spin off cleanly" intent of the resolution; (a) is the cleanest user experience but requires coordination with the publishing pipeline.
3. **Coordination with FHIR-42980.** Both tickets edit `source/mapping-language.html`. Decide merge order (recommend FHIR-42980 first, then this ticket rebased on top, since FHIR-42980's edits are at lines 483, 656, and 918 and the link cells here sit at 627–651 within that same file).
4. **Companion ticket against the destination IG.** Standing up the new IG content is out of scope of FHIR-54745 as filed against FHIR Core. A companion ticket (or repo-level issue) should be opened against the chosen incubator repo to import the removed prose; this plan does not author that ticket.
