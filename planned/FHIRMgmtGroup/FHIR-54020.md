## Resolution Summary

Resolution: **Persuasive** — *"Agreed, will correct spelling of payer throughout."*

The Financial Management WG previously chose `payer` over the `payor` variant
and asked that this preference be applied consistently across all FHIR R6
content. The approved action is therefore a repo-wide spelling normalization
in narrative / definition / example text, replacing `payor` (and its
capitalised / plural forms) with `payer` everywhere it appears as the English
common noun. This is a technical correction; no element renames, cardinality
changes, terminology changes, or behavioural changes are mandated by the
resolution.

## Feature Proposal

### Problem Statement

R6 content is inconsistent in how it spells the common noun for an insurer
that pays a healthcare claim. Both `payer` and `payor` appear, often within
the same module and even within the same file (for example
`source/financial-module.html` lists `Payer` and `Payor` as separate
glossary rows). FM standardised on `payer` years ago; the Patient
Experience pass for R6 (the "PE" tracker batch in the title) is the
opportunity to make the rest of the spec match.

A repo-wide grep against the analyzed clone (`HL7/fhir`,
`db79dcd`, `git grep -i payor -- source/`) finds **26 files** under
`source/` containing `payor` / `Payor` / `payors` / `Payors` (full list in
the Affected Files section below). They fall into four buckets:

1. **Narrative HTML pages and resource introductions** (e.g.,
   `source/financial-module.html`, `source/help.html`,
   `source/overview-patient.html`,
   `source/coverageeligibilityresponse/coverageeligibilityresponse-introduction.xml`,
   `source/paymentreconciliation/paymentreconciliation-introduction.xml`).
2. **StructureDefinition `<purpose>` / `<definition>` / `<comment>` prose**
   (e.g., `source/coverage/structuredefinition-Coverage.xml`,
   `source/enrollmentrequest/structuredefinition-EnrollmentRequest.xml`,
   `source/coverageeligibilityrequest/structuredefinition-CoverageEligibilityRequest.xml`).
3. **Examples** that use `payor` as a contained-resource id / local
   reference (e.g., `source/communication/communication-example-fm-attachment.xml`
   and its siblings, `source/eligibilityresponse/eligibilityresponse-example-benefits-2.xml`,
   `source/processrequest/processrequest-example-poll-specific.xml`).
4. **Special files**: `source/spelling/english.dic` (the publisher's
   spell-check dictionary, where `payor` is currently whitelisted),
   `source/fhir.ini` (a `Coverage.payor=` historical-rename mapping
   entry), legacy `*-spreadsheet.xml` Excel artifacts under
   `source/eligibilityrequest/` and `source/eligibilityresponse/`,
   `source/library/library-fhir-model-definition-content.xml` (a model
   definition snapshot containing `<ns4:element name="payor">`), and two
   diagram SVGs under `source/coverage/` and
   `source/coverageeligibilityrequest/`.

### Proposed Change

Apply a case-preserving global substitution of the common-noun spelling
across `source/` only:

| Found | Replace with |
|-|-|
| `payor`   | `payer`   |
| `Payor`   | `Payer`   |
| `payors`  | `payers`  |
| `Payors`  | `Payers`  |

Scope rules — **do not** rewrite a match when it is any of the following
(these are not common-noun prose):

- A FHIR element / class / property identifier in machine-readable XML —
  e.g., `<path value="...">`, `<name value="...">`, `<id value="...">`,
  `<code value="...">`, `<reference value="#...">`, `<ns4:element name="...">`.
  (See "Affected Files" / Open Questions for the few places where this
  matters in practice.)
- The `Coverage.payor=` key in `source/fhir.ini` (this is a historical
  rename-mapping entry keyed by the *old* element name; rewriting the
  key would corrupt the cross-version mapping).
- Any binary payload (e.g., the base64 `<data>` block in
  `source/patient/patient-example-a.xml`, where `payor` only appears
  inside an embedded PNG byte stream).

The XML-prose buckets (`<purpose>`, `<definition>`, `<comment>`,
`<short>`, plain text inside intro / notes / examples-header XML, plain
text inside `*.html` pages) are all in scope. Narrative inside
StructureDefinitions is hand-authored (`<differential>`-style content per
the per-repo briefing); the `<snapshot>` blocks are publisher-generated
and need no manual edit — they will be regenerated on the next publish.

In addition:

- Remove the `payor` line from `source/spelling/english.dic` so the
  publisher's spell-checker no longer accepts the deprecated spelling
  going forward and would flag any future re-introduction.
- For the two `.svg` diagrams (`source/coverage/coverage.svg`,
  `source/coverageeligibilityrequest/coverageeligibilityrequest.svg`),
  edit the embedded `<text>` nodes that read `payor` so the rendered
  diagram label says `payer`.
- For example files where `payor` is used as a contained-resource id /
  local reference (e.g., `<id value="payor"/>` and the matching
  `<reference value="#payor"/>`), rename consistently to `payer` within
  the same file so the intra-bundle reference still resolves. These are
  example-only identifiers with no external consumers.

No StructureDefinition cardinalities, types, bindings, search
parameters, invariants, or canonical URLs are touched by this change.

### Design Rationale

- The resolution explicitly states "correct spelling of payer
  throughout," so the change is mechanical and global rather than
  selective. The job is to apply a single English-spelling preference,
  not to redesign any artifact.
- Limiting the rewrite to the publisher's authoring root (`source/`)
  matches the per-repo briefing's guidance that this is the only place
  hand-authored content lives in `HL7/fhir`. Generated areas
  (`<snapshot>`, `tools/`, `qa/`, `schema/`) are explicitly excluded.
- Excluding XML *identifier* contexts (element names, codes, ids,
  references) and the `fhir.ini` rename-mapping key is the
  least-invasive way to honour FM's intent without breaking
  machine-readable conformance or version-history tooling. A per-file
  review (vs. blind sed) is required for that reason.
- Removing `payor` from `source/spelling/english.dic` makes the
  preference enforceable on subsequent edits, so the inconsistency does
  not silently re-emerge.

### Breaking Changes

No breaking changes to the FHIR specification itself: no element renames,
type changes, cardinality changes, binding changes, or search-parameter
changes. The only renames are to **contained-resource local ids** inside
example files (`<id value="payor"/>` → `<id value="payer"/>`), which are
example-internal anchors and have no consumers outside the example
bundle.

The publisher dictionary edit (`source/spelling/english.dic`) is
deliberately a behavioural tightening: future authored content
containing `payor` will be flagged as a spelling error rather than
silently accepted. That is the *point* of the ticket and is not a
backwards-compatibility concern for published content.

### Related Specifications

None directly. This is a prose-only consistency pass within
`HL7/fhir`. Downstream IGs that quote FHIR Core narrative are
unaffected (they will pick up the new wording the next time they
re-pull).
