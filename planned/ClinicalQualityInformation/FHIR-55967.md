## Resolution Summary

Resolution: **Persuasive with Modification**. The approved change is:

- Copy the IN parameters `reporter` and `reporterResource` (as defined on `Measure-$evaluate`) onto `Measure-$care-gaps`.
- Remove the IN parameter `organization` from `Measure-$care-gaps` and remove `organizationResource` from both `Measure-$collect-data` and `Measure-$care-gaps`.
- On `Measure-$evaluate`, `Measure-$care-gaps`, and `Measure-$collect-data`, update the `reporterResource` description so its first sentence reads: *"The provider for which the report will be run, provided as a Practitioner, PractitionerRole, or Organization resource."*
- Constrain the `reporterResource` parameter type so it is one of `Organization`, `Practitioner`, or `PractitionerRole` (rather than the prior single-type `Organization`).

## Feature Proposal

### Problem Statement

The three Measure operations that accept a "reporter" of the measurement (`$evaluate`, `$care-gaps`, `$collect-data`) had inconsistent input parameters:

- `$evaluate` carried `reporter` (reference) plus `reporterResource` (typed as `Organization`).
- `$care-gaps` carried `organization` and `organizationResource` (typed as `Organization`) instead of the aligned `reporter` / `reporterResource` pair.
- `$collect-data` carried `reporterResource` (typed as `Organization`) and an additional `organizationResource` parameter, even though the description already framed it as "the provider for which the data-of-interest will be collected".

The naming and typing mismatch made it impossible to write portable client code that targets all three operations with the same provider context, and the `Organization`-only typing prevented passing a `Practitioner` or `PractitionerRole` ad‑hoc reporter (a use case explicitly supported by the existing prose pointing at `clinicalreasoning-quality-reporting.html#ad-hoc-organizations`).

### Proposed Change

Three OperationDefinitions are touched. All edits go in the `<differential>` content of the OperationDefinitions; no `<snapshot>` editing is required (the publisher regenerates that).

1. `Measure-$care-gaps` (`source/measure/operationdefinition-Measure-care-gaps.xml`):
   - **Remove** the existing `organization` IN parameter (string/reference, 0..1).
   - **Remove** the existing `organizationResource` IN parameter (Organization, 0..1).
   - **Add** an IN parameter `reporter`, mirroring `$evaluate`:
     - `use=in`, `min=0`, `max=1`, `type=string`, `searchType=reference`.
     - Documentation: *"The provider for which the report will be run. This may be a reference to a Practitioner, PractitionerRole, or Organization. If specified, systems MAY use this information to determine subjects for which the measure will be calculated, but how subjects are determined is implementation-specific. This parameter is reflected in the reporter element of the resulting MeasureReport(s). This parameter cannot be used with the reporterResource parameter."*
   - **Add** an IN parameter `reporterResource`, mirroring the new `$evaluate` shape:
     - `use=in`, `min=0`, `max=1`, `type=DomainResource`, `searchType=reference`.
     - Three `operationdefinition-allowed-type` extensions constraining the allowed resource types to `Practitioner`, `PractitionerRole`, and `Organization`.
     - Documentation first sentence: *"The provider for which the report will be run, provided as a Practitioner, PractitionerRole, or Organization resource."* followed by the standard ad‑hoc-organization explanatory sentences and the cross-reference to `reporter` ("This parameter cannot be used with the reporter parameter.").

2. `Measure-$collect-data` (`source/measure/operationdefinition-Measure-collect-data.xml`):
   - **Remove** the `organizationResource` IN parameter (Organization, 0..1).
   - **Modify** the existing `reporterResource` parameter:
     - Change `type` from `string` to `DomainResource`.
     - Add the three `operationdefinition-allowed-type` extensions (`Practitioner`, `PractitionerRole`, `Organization`).
     - Replace the documentation so the first sentence reads *"The provider for which the report will be run, provided as a Practitioner, PractitionerRole, or Organization resource."* (unify the wording with the other two operations; keep the ad‑hoc-organization sentence and the "cannot be used with the reporter parameter" sentence).

3. `Measure-$evaluate` (`source/measure/operationdefinition-Measure-evaluate.xml`):
   - **Modify** the existing `reporterResource` parameter:
     - Change `type` from `string` to `DomainResource`.
     - Add the three `operationdefinition-allowed-type` extensions (`Practitioner`, `PractitionerRole`, `Organization`).
     - Update the first sentence of the documentation to *"The provider for which the report will be run, provided as a Practitioner, PractitionerRole, or Organization resource."* (was: "...provided as an Organization resource.").
   - The `reporter` parameter on `$evaluate` is unchanged structurally (it is the source of the copy into `$care-gaps`); only verify wording remains aligned.

4. Ballot note (`source/measure/measure-introduction.xml`):
   - Add a "Non-compatible Changes" bullet for FHIR-55967 in the "Note to Balloters" section: *"Align reporter parameters in $evaluate, $care-gaps, and $collect-data"* with a link to the Jira ticket.

### Design Rationale

Resolution wording is explicit ("Persuasive with Modification") and the three-operation alignment removes a long-standing inconsistency. Using `type=DomainResource` plus `operationdefinition-allowed-type` extensions is the canonical FHIR pattern for declaring a polymorphic resource-typed operation parameter (rather than declaring three sibling parameters). Choosing `Practitioner | PractitionerRole | Organization` matches the existing ad‑hoc-organizations narrative on `clinicalreasoning-quality-reporting.html` and the typing already used elsewhere when a "reporter" (provider performing the report) is referenced. Removing `organization`/`organizationResource` from `$care-gaps` and `organizationResource` from `$collect-data` is the resolution’s explicit decision and avoids two parallel naming conventions for the same concept.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Hosts the three Measure OperationDefinitions and the Measure introduction page.
- **Affected Files:**
  - `source/measure/operationdefinition-Measure-care-gaps.xml` — remove `organization` and `organizationResource`; add `reporter` and `reporterResource` (typed `DomainResource` with `Practitioner`/`PractitionerRole`/`Organization` allowed-type extensions).
  - `source/measure/operationdefinition-Measure-collect-data.xml` — remove `organizationResource`; retype `reporterResource` to `DomainResource` with the three allowed-type extensions; rewrite first sentence.
  - `source/measure/operationdefinition-Measure-evaluate.xml` — retype `reporterResource` to `DomainResource` with the three allowed-type extensions; rewrite first sentence.
  - `source/measure/measure-introduction.xml` — add Non-compatible Changes ballot bullet for FHIR-55967.
- **Change Scope:** Moderate. Three operation contracts change shape; one is a non-backwards-compatible parameter rename/removal.

### Breaking Changes

Yes — explicitly tagged as **Non-compatible** in the Measure introduction:

- `$care-gaps`: removal of the `organization` and `organizationResource` IN parameters is a breaking change. Existing clients invoking `$care-gaps` with either parameter will need to switch to `reporter` / `reporterResource`.
- `$collect-data`: removal of the `organizationResource` IN parameter is a breaking change.
- `$evaluate` and `$collect-data`: `reporterResource` type widens from `Organization` to `DomainResource` constrained to `Practitioner | PractitionerRole | Organization`. Servers that previously rejected non-`Organization` instances can now legitimately receive them; clients that hard-coded `Organization` will continue to work.

### Related Specifications

- `MeasureReport.reporter` (the result element these parameters populate) — no schema change needed; its existing reference target set already includes `Practitioner`, `PractitionerRole`, `Organization`, so the widened input type is consistent.
- `clinicalreasoning-quality-reporting.html#ad-hoc-organizations` — the rationale section the parameter documentation links to; no edit needed but its narrative now applies symmetrically to all three operations.
- Other Measure operations (`$data-requirements`, `$submit-data`, `$evaluate-measure`) are not in scope; the resolution did not call them out.

### Related Tickets

- **FHIR-51028:** Clarified examples are experimental content — adjacent Non-compatible bullet in the same `measure-introduction.xml` "Note to Balloters" block. Complementary, not conflicting; both bullets co-exist.
- **FHIR-50488:** Ensuring `supplementalData` can be associated with specific measures in the invocation — recent change to the same OperationDefinition family. No element overlap with the reporter/organization parameters.
- No other open tickets surfaced via cross-references that touch the `reporter` / `organization` parameters on these three operations.

### Terminology Impact

No terminology impact. No CodeSystem or ValueSet changes are required; the change is structural (parameter shape and typing).

## Implementation Plan

### Prerequisites

None. The `operationdefinition-allowed-type` extension and the `DomainResource` typing pattern already exist elsewhere in core (e.g., the `subject` parameter on related operations), so no extension or terminology work needs to land first.

1. **Retype and re-document `reporterResource` on `$evaluate`**

   - Action: In the `reporterResource` `<parameter>` block, change `<type value="string"/>` to `<type value="DomainResource"/>` and prepend three `<extension url="http://hl7.org/fhir/StructureDefinition/operationdefinition-allowed-type">` elements with `valueUri` of `Practitioner`, `PractitionerRole`, and `Organization` respectively. 3. **Replace `organization`/`organizationResource` with `reporter`/`reporterResource` on `$care-gaps`**
