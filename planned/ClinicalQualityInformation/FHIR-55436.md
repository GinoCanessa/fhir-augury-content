## Resolution Summary

**Resolution:** Persuasive — *"Remove Measure.topic as proposed because this element was deprecated in R5."*

The committee has approved removal of the `Measure.topic` element from the Measure resource in R6. This is the per-element execution ticket spawned from parent FHIR-53722 (broader R6 deprecated-element clean-up). Per the parent ticket's removal guidelines, `Measure.topic` qualifies for outright removal because it carried an explicit `DEPRECATION NOTE` comment and the migration target (use `useContext` with `usage-context-type#topic`) is already documented in the deprecated element's `comment`.

## Feature Proposal

### Problem Statement

`Measure.topic` (0..* CodeableConcept, bound to `definition-topic` example value set) was marked deprecated in R5 via both a `structuredefinition-standards-status = deprecated` extension and an in-line `comment` directing implementers to migrate to `useContext.code = topic` + `useContext.valueCodeableConcept`. Carrying the element forward into R6 (a) clutters the Measure resource with an element no one is supposed to author against, (b) duplicates the `useContext`-based capability the WG endorses, and (c) preserves a `topic` SearchParameter and `topic`-typed slot in the Measure UML diagram that promote the deprecated path. The parent ticket FHIR-53722 establishes that R6 is the appropriate window to physically remove R5-deprecated elements.

### Proposed Change

Physically remove every artifact-level reference to `Measure.topic` from the Measure resource definition in `HL7/fhir`:

1. **StructureDefinition** — `source/measure/structuredefinition-Measure.xml`: delete the `<element id="Measure.topic">` differential block in its entirety (path, short, definition, deprecation extension/comment, requirements, 0..* cardinality, `CodeableConcept` type, example binding to `http://hl7.org/fhir/ValueSet/definition-topic` with binding name `DefinitionTopic`, and the rim mapping). The `<snapshot>` block is regenerated by the publisher and must not be hand-edited.
2. **SearchParameter bundle** — `source/measure/bundle-Measure-search-params.xml`: delete the `<entry>` carrying `SearchParameter` `Measure-topic` (code `topic`, type `token`, expression `Measure.topic`, URL `http://hl7.org/fhir/build/SearchParameter/Measure-topic`).
3. **UML diagram** — `source/measure/measure.svg`: remove the SVG group(s) rendering the `topic : 0..* CodeableConcept` row and any deprecation badge/struck-through styling for that row, then re-flow row spacing so the box closes cleanly.
4. **Resource introduction** — `source/measure/measure-introduction.xml`: add an R6 release-note bullet/paragraph announcing the removal of `Measure.topic`, with a pointer to the `usage-context-type#topic` migration path. (No prose currently markets `Measure.topic`; nothing else in the intro has to be deleted.)

After the change, no `Measure.topic` element, search parameter, or UML row should exist; `useContext` remains the sole way to express a measure's topic.

Reference implementation already exists in `HL7/fhir@06ab8128` ("[FHIR-55436]: Remove deprecated Measure.topic. Also removed element references from Measure bundle search params and measure.svg."), which is **already merged into the analyzed clone HEAD** (`db79dcd`). Stats: 4 files changed, 3 insertions, 60 deletions — exactly the scope above.

### Design Rationale

- The parent ticket FHIR-53722 explicitly authorizes removal (rather than merely re-flagging) for any deprecated element with either the `deprecated` standards-status flag since R4 or a deprecation `comment` plus a qualifying condition. `Measure.topic` carries the `valueCode="deprecated"` standards-status extension *and* a `DEPRECATION NOTE` comment in its description, satisfying the strongest removal criterion.
- The migration target (`useContext` with `usage-context-type#topic`) is a base-resource pattern shared by every canonical resource, so removal does not strand existing users — the replacement is documented inside the element being deleted.
- Keeping a `Measure-topic` SearchParameter alive after the element is gone would dangle a `Measure.topic` FHIRPath expression with no possible match; the parameter must be removed at the same time.
- The SVG must be hand-edited because `source/measure/measure.svg` is authored content (per the briefing, `source/` is the only authoring root and there is no SVG generator in the FhirCore build pipeline).

### Breaking Changes

**Yes — this is a breaking change for R6 consumers of R5 Measure.**

- `Measure.topic` (0..* CodeableConcept) is removed. Any R5 instance that populated `Measure.topic` will not round-trip into an R6-conformant Measure; senders must migrate the value(s) to `Measure.useContext` entries with `useContext.code = http://terminology.hl7.org/CodeSystem/usage-context-type#topic` and `useContext.valueCodeableConcept = <topic concept>`.
- The `topic` SearchParameter on Measure is removed. Servers honoring `GET [base]/Measure?topic=…` in R5 will return 4xx (or silently empty) in R6; clients should switch to `GET [base]/Measure?context-type=topic&context=…`.
- The change is consistent with the deprecated element's own `DEPRECATION NOTE`, so it is not a *surprise* breaking change, but the element-level/SearchParameter-level removal must be called out in the R6 release notes (the `measure-introduction.xml` bullet is the in-spec home for that note).

### Related Specifications

- **`http://terminology.hl7.org/CodeSystem/usage-context-type`** — the `topic` code remains the documented R6 migration target; no change required.
