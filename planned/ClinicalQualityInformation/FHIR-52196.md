## Resolution Summary

The submitter observed that US Core CareTeam has MustSupport flags on the target reference types of `CareTeam.participant.member` (specifically Practitioner, PractitionerRole, and RelatedPerson), but the derived QI Core CareTeam profile does not carry those MustSupport flags forward. The work group agreed this is an oversight and resolved as **Persuasive**: add the MS flag to the `participant.member` references in QI Core CareTeam to be consistent with the US Core implementation — specifically for Practitioner, PractitionerRole, and RelatedPerson target types.

---

## Feature Proposal

### Problem Statement

QI Core CareTeam inherits from (is based on) US Core CareTeam. In US Core STU8, `CareTeam.participant.member` has MustSupport applied to its target reference types: US Core Practitioner, US Core PractitionerRole, and US Core RelatedPerson. However, the derived QI Core CareTeam profile does not reproduce these MustSupport markers on the corresponding QICore-profiled references. This creates an inconsistency: implementers looking only at the QI Core profile may not realize they must support these reference targets, since the MS flags from the parent profile are not visually or structurally echoed in the child profile.

### Proposed Change

In the QI Core CareTeam StructureDefinition, add `mustSupport = true` to each of the three target reference type entries under `CareTeam.participant.member`:

1. **Reference(QICore Practitioner)** — `mustSupport: true`
2. **Reference(QICore PractitionerRole)** — `mustSupport: true`
3. **Reference(QICore RelatedPerson)** — `mustSupport: true`

This mirrors the US Core CareTeam profile's approach of flagging specific reference targets as MustSupport using the `type.targetProfile` extension mechanism.

### Design Rationale

- **Consistency with US Core**: QI Core is built on US Core; MS flags on reference targets should be inherited or explicitly restated to avoid ambiguity.
- **Implementer clarity**: Without explicit MS on the QI Core profile, measure developers and EHR implementers may miss that support for these reference types is required.
- **Pattern alignment**: This is the same pattern applied in other QI Core profiles (and is the subject of a sibling ticket FHIR-52547 for Condition.subject).
