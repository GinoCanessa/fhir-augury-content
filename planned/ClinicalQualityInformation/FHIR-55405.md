## Resolution Summary

**Resolution:** Persuasive — "Add search parameter as requested."

The committee accepted the change request to add a search parameter for the
existing `MeasureReport.category` element. No additional scope or alternative
design was specified in the resolution; the literal request from the ticket
description is the agreed solution.

## Feature Proposal

### Problem Statement

`MeasureReport` already defines a `category` element (an extensible
`CodeableConcept` bound to `http://hl7.org/fhir/ValueSet/measurereport-category`,
cardinality 0..1) that classifies the report (e.g., DEQM, Risk Adjustment,
Value-Based Performance). However, the resource ships no search parameter for
that element, so servers cannot filter `MeasureReport` instances by category
through the standard FHIR REST search API. Clients consuming MeasureReports
across multiple programs (DEQM vs. RA vs. VBP, etc.) must therefore retrieve
all reports and filter client-side, which is inefficient and inconsistent with
the way other coded classifier elements (e.g., `Condition.category`,
`Observation.category`, `DocumentReference.category`) are exposed.

### Proposed Change

Add a single `token` SearchParameter to `MeasureReport`:

- **Id:** `MeasureReport-category`
- **URL:** `http://hl7.org/fhir/build/SearchParameter/MeasureReport-category`
- **Code:** `category`
- **Type:** `token`
- **Base resource:** `MeasureReport`
- **Expression:** `MeasureReport.category`
- **Path extension (build form):** `MeasureReport.category`
- **Description:** "The category of the measure report (e.g., DEQM, RA, VBP)"
- **processingMode:** `normal`
- **Standards status:** `normative` (matches every sibling SearchParameter
  already in `bundle-MeasureReport-search-params.xml`).

The new entry is added to the existing
`source/measurereport/bundle-MeasureReport-search-params.xml` bundle alongside
the existing `MeasureReport-date`, `-evaluated-resource`, `-identifier`,
`-measure`, `-patient`, `-period`, `-reporter`, `-location`, `-status`, and
`-subject` entries. Insertion in alphabetical-by-code order places it between
`MeasureReport-evaluated-resource` and `MeasureReport-identifier`.

Example XML entry to add:

```xml
<entry>
  <resource>
    <SearchParameter>
      <id value="MeasureReport-category"/>
      <extension url="http://hl7.org/fhir/StructureDefinition/structuredefinition-standards-status">
        <valueCode value="normative"/>
      </extension>
      <extension url="http://hl7.org/fhir/build/StructureDefinition/path">
        <valueString value="MeasureReport.category"/>
      </extension>
      <url value="http://hl7.org/fhir/build/SearchParameter/MeasureReport-category"/>
      <description value="The category of the measure report (e.g., DEQM, RA, VBP)"/>
      <code value="category"/>
      <type value="token"/>
      <expression value="MeasureReport.category"/>
      <processingMode value="normal"/>
    </SearchParameter>
  </resource>
</entry>
```

### Design Rationale

- **`token` type:** `MeasureReport.category` is a `CodeableConcept`, which the
  FHIR search-parameter taxonomy maps to `token` (consistent with
  `Condition-category`, `Observation-category`, `DocumentReference-category`,
  etc.).
- **Cardinality 0..1 on the element:** A simple `MeasureReport.category`
  expression is sufficient; no `where()` filter or repetition handling is
  needed.
- **Extensible binding:** The search parameter does not enforce the value set
  binding — `token` search already supports searching by `system|code` or by
  `code` alone, which lets clients query both the suggested DEQM/RA/VBP codes
  and any extension codes a deployment uses.
- **Standards status `normative`:** Every other SearchParameter in the
  MeasureReport search-params bundle is `normative`; adding a new one at a
  lower maturity would be inconsistent with the resource-level FMM/maturity
  posture.
- **Code `category`:** Matches the element name and the cross-resource
  convention for categorising classifier search parameters.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Sole repository — defines the `MeasureReport` resource and its
  search parameters.
- **Affected Files:**
  - `source/measurereport/bundle-MeasureReport-search-params.xml` — add a new
    `<entry>` for `MeasureReport-category` (token, expression
    `MeasureReport.category`, normative).
- **Change Scope:** Minor.

### Breaking Changes

No breaking changes. The change adds a new optional SearchParameter; existing
`MeasureReport` instances and queries are unaffected. Servers that have not
yet implemented the parameter remain conformant — search parameter support is
optional unless flagged as `mandatory` in a CapabilityStatement.

### Related Specifications

- `MeasureReport.category` element definition in
  `source/measurereport/structuredefinition-MeasureReport.xml` (lines 122–141)
  remains unchanged; the new search parameter simply exposes it for query.
- The bound ValueSet `http://hl7.org/fhir/ValueSet/measurereport-category`
  (extensible) is unchanged.
- Implementation guides that profile `MeasureReport` (e.g., DEQM, Da Vinci RA,
  Da Vinci VBP) gain the ability to require this search parameter in their
  CapabilityStatements without re-defining it.

### Related Tickets

No related Jira tickets were surfaced via cross-references or keyword search;
the only cross-reference recorded for FHIR-55405 is the FHIR-path target
`MeasureReport.category` itself. If future tickets propose changes to the
cardinality, type, or binding of `MeasureReport.category`, this search
parameter's expression and type may need to be re-evaluated.

### Terminology Impact

No terminology impact. The change reuses the existing extensible binding to
`http://hl7.org/fhir/ValueSet/measurereport-category`; no CodeSystem or
ValueSet additions/edits are required in HL7/UTG or elsewhere.
