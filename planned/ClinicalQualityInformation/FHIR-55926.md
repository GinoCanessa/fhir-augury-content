## Resolution Summary

Resolution: **Persuasive**. The resolution description is "Update as in
description" — i.e., apply the exact prose changes supplied in the ticket
description.

Concretely, rewrite the `<documentation>` text of the `subject` and
`subjectGroup` input parameters of the `Measure/$evaluate` OperationDefinition
so that:

1. The two parameters are explicitly documented as mutually exclusive.
2. The "server picks subjects" fallback now triggers when **neither**
   `subject` **nor** `subjectGroup` is supplied (previously worded as "if no
   subject is provided").
3. The `subject` documentation no longer enumerates the list of allowed
   reference target resource types inline (that list is removed from the
   prose; it is already carried structurally via the parameter's allowed
   target types).

Target new text for `subject`:

> Subject(s) for which the measure will be calculated. Subjects provided in
> this parameter SHALL match the subjectType of the measure being evaluated
> (e.g. if the Measure.group.subjectType is Patient, the subject SHALL be a
> reference to a Patient or a Group of Patients). This parameter cannot be
> used with the subjectGroup parameter. If no subject or subjectGroup is
> provided, what subjects are evaluated is up to the server.

Target new text for `subjectGroup`:

> Subjects for which the measure will be evaluated, provided as a Group
> resource. Subjects provided in this parameter SHALL match the subjectType
> of the measure being evaluated. (e.g. if the Measure.group.subjectType is
> Patient, the subject SHALL be a reference to a Patient or a Group of
> Patients). This parameter cannot be used with the subject parameter. If no
> subject or subjectGroup is provided, what subjects are evaluated is up to
> the server.

## Feature Proposal

### Problem Statement

The current `Measure/$evaluate` OperationDefinition has two input parameters
that can identify the subject(s) of a measure calculation: `subject` (a
reference-style string parameter, allowed targets include Patient, Group,
Practitioner, Organization, Location, Device, etc.) and `subjectGroup` (a
`Group` resource parameter).

As authored in `source/measure/operationdefinition-Measure-evaluate.xml`, the
`subject` doc says _"If no subject is provided, what subjects are evaluated
is up to the server. This parameter cannot be used with the subjectGroup
parameter."_ and the `subjectGroup` doc says _"If no subject is provided,
what subjects are evaluated is up to the server. This parameter cannot be
used with the subject parameter."_ Two problems:

- **Ambiguous fallback trigger.** The phrase "if no subject is provided"
  is ambiguous — a reader cannot tell whether supplying `subjectGroup`
  alone is enough to avoid server-chosen subjects. The intended semantics,
  per the ticket, is that the server picks subjects only when **both**
  parameters are absent.
- **Redundant / out-of-place type list in `subject`.** The inline sentence
  "The subject may be a Patient, Practitioner, PractitionerRole,
  Organization, Location, Device, or Group." duplicates information that
  belongs structurally to the parameter (allowed target types) and muddies
  the prose about mutual exclusivity.

The ticket is reported as a Change Request with resolution **Persuasive**
and the resolution description is "Update as in description", meaning the
replacement text in the ticket description is the agreed solution.

### Proposed Change

Edit the two `<documentation value="…"/>` strings inside
`source/measure/operationdefinition-Measure-evaluate.xml`:

- **`subject` parameter** (currently at line ~101–102):
  - Remove the sentence: _"The subject may be a Patient, Practitioner,
    PractitionerRole, Organization, Location, Device, or Group."_
  - Reorder the two closing sentences so the exclusivity statement comes
    before the "server picks subjects" fallback, and broaden the fallback
    trigger to "If no subject or subjectGroup is provided".
  - Do **not** change `<type>`, `<searchType>`, `<min>`, `<max>`, or
    `<use>`. Only the `<documentation value="…"/>` content changes.
- **`subjectGroup` parameter** (currently at line ~111–112):
  - Collapse the double-space before the parenthetical "(e.g. if the
    Measure.group.subjectType is Patient, …)" to a single space.
  - Reorder the two closing sentences so the exclusivity statement comes
    before the "server picks subjects" fallback, and broaden the fallback
    trigger to "If no subject or subjectGroup is provided".
  - Do **not** change `<type>` (`Group`), `<min>`, `<max>`, `<use>`.

No structural changes (no new parameters, no changed cardinalities, no new
bindings, no new invariants). No changes to
`operationdefinition-Measure-evaluate-measure.xml` — that older definition
does not declare a `subjectGroup` parameter and is out of scope for this
clarification.

### Design Rationale

- The ticket explicitly supplies the replacement prose verbatim, and the
  CQI work group accepted it as Persuasive without further amendment (no
  applied-vote comments on the ticket). The design decision is therefore
  fixed: implement the text exactly as written.
- Removing the inline type list from `subject` is consistent with the rest
  of the OperationDefinition: allowed target resource types are carried
  structurally (e.g. via `operationdefinition-allowed-type` extensions
  elsewhere in the file), not as ad-hoc prose. Keeping the list only in
  one place avoids drift when target types are later adjusted.
- The "no subject **or subjectGroup**" wording matches the actual semantic
  intent: either parameter is sufficient to name the subjects. Only when
  both are omitted does the server default to choosing subjects itself.
- Alternative approaches considered in the ticket discussion: none —
  there are no ticket comments, so no alternatives were raised. Leaving
  the file untouched was implicitly rejected by the Persuasive vote.
