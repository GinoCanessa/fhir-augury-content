## Resolution Summary

**Resolution:** Persuasive with Modification.

The applied resolution (per FMWG, 2021-09-22 minutes) requires four work items:

1. Create a new code system that is **not** an example.
2. Add the proposed adjudication codes (from CARIN-BB and RTPBC alignment) into that code system; the system itself must not be marked experimental/example.
3. Review the codes for consistent style.
4. In the standard resources that bind `ValueSet/adjudication`, change the bindings to use this code system at strength **preferred** or **extensible** (no longer "example").

The codes to be added (from the ticket description) are: `coinsurance`, `noncovered`, `priorpayerpaid`, `paidbypatient`, `paidtoprovider`, `paidtopatient`, `memberliability`, `coupon-discount`, `member-discount`, `other-discount`, `discount`, `drugcost`, `other-cost`. These are sourced from `C4BBAdjudication` and `rtpbc-patient-pay-type`.

Inspection of the current cache shows partial completion already exists in HL7/UTG: the `CodeSystem` at `http://terminology.hl7.org/CodeSystem/adjudication` has already been promoted (`experimental="false"`, `content="complete"`, version `1.0.1`, lastUpdated `2024-04-24`). The new concepts have **not** yet been added, the HL7/fhir wrapper ValueSet still carries `experimental="true"` / `copyright="This is an example set."`, and two of the four core bindings to `ValueSet/adjudication` are still `example` rather than `preferred`/`extensible`.

## Feature Proposal

### Problem Statement

The FHIR R5 `Adjudication Value Codes` code system (`http://terminology.hl7.org/CodeSystem/adjudication`) is the canonical FM code system for the `adjudication.category` slot on `ClaimResponse`, `ExplanationOfBenefit`, and related resources. Historically it was published as a small example set covering only `submitted`, `copay`, `eligible`, `deductible`, `unallocdeduct`, `eligpercent`, `tax`, `benefit`. Implementer guidance from CARIN-BB (`C4BBAdjudication`) and RTPBC (`rtpbc-patient-pay-type`) demonstrated significant gaps for real-world payer/pharmacy adjudication: there is no canonical code for co-insurance, non-covered amounts, prior-payer paid, paid-to-provider/patient/member-liability splits, drug cost, or the various discount/coupon categories. Each downstream IG was inventing its own code system, defeating interoperability at the very element designed to make adjudication line items comparable across payers.

The ticket addresses this by promoting the core code system from "example" to a real, maintained code system and back-filling the missing concepts so that downstream IGs can rebind to (or extend) the core system instead of their own.

### Proposed Change

**A. HL7/UTG — `CodeSystem-adjudication.xml` (`input/sourceOfTruth/fhir/codeSystems/CodeSystem-adjudication.xml`)**

The system header already satisfies items (1) and (2)'s "not an example" requirement (`experimental="false"`, `content="complete"`, shareable profile, FM ownership, FMM=1). Bump `version` (currently `1.0.1`) and `meta.lastUpdated` and append the following 13 `<concept>` entries to the existing 8. Codes/displays/definitions are taken verbatim from the ticket description (whitespace trimmed):

| Code | Display | Definition |
|------|---------|------------|
| `coinsurance` | Co-insurance | The amount the insured individual pays, as a set percentage of the cost of covered medical services, as an out-of-pocket payment to the provider. Example: Insured pays 20% and the insurer pays 80%. |
| `noncovered` | Noncovered | The portion of the cost of this service that was deemed not eligible by the insurer because the service or member was not covered by the subscriber contract. |
| `priorpayerpaid` | Prior payer paid | The reduction in the payment amount to reflect the carrier as a secondary payor. |
| `paidbypatient` | Paid by patient | The amount paid by the patient at the point of service. |
| `paidtoprovider` | Paid to provider | The amount paid to the provider. |
| `paidtopatient` | Paid to patient | The amount paid to the patient. |
| `memberliability` | Member liability | The amount of the member's liability. |
| `coupon-discount` | Coupon discount amount | Coupon discount reducing the amount to be collected from the patient. |
| `member-discount` | Member discount amount | Member discount reducing the amount to be collected from the patient. |
| `other-discount` | Other discount | Other discount reducing the amount to be collected from the patient. |
| `discount` | Discount | The amount of the discount. |
| `drugcost` | Drug cost | Price paid for the drug excluding manufacturer or other discounts. It typically is the sum of the following components: ingredient cost, dispensing fee, sales tax, and vaccine administration. |
| `other-cost` | Other cost | Other amount to be collected from the patient. |

Resolution item (3) "review for consistent style" must be addressed before merging — the proposed codes mix two casing conventions (`paidbypatient` / `paidtoprovider` etc. vs. `coupon-discount` / `member-discount` / `other-discount` / `other-cost`). Recommendation, to be confirmed by FMWG, is to normalise the new identifiers to lower-kebab-case (`paid-by-patient`, `paid-to-provider`, `paid-to-patient`, `member-liability`, `prior-payer-paid`, `non-covered`, `drug-cost`) so the entire system reads consistently — leaving the eight pre-existing codes untouched to avoid breaking deployed instances. If FMWG instead prefers preserving submitter-supplied spelling, all 13 codes ship as-is. Either decision must be reflected in (a) the CodeSystem concepts and (b) the ValueSet wrapper, before bindings are promoted.

**B. HL7/UTG — `ValueSet-adjudication.xml` (`input/sourceOfTruth/fhir/valueSets/ValueSet-adjudication.xml`)**

Confirm the UTG ValueSet wrapper is also non-experimental and includes the system without enumerating concepts (the `compose.include` form), so the new codes are picked up automatically.

**C. HL7/fhir — `source/claimresponse/valueset-adjudication.xml`**

Update the in-core mirror so it is no longer an example:

- `<experimental value="true"/>` → `<experimental value="false"/>`
- Replace `<copyright value="This is an example set."/>` with the standard HL7 Terminology copyright (matching the UTG CodeSystem copyright).
- Bump `<version value="6.0.0"/>` to the current core build version on the next publication run.

**D. HL7/fhir — bindings on `ValueSet/adjudication`**

Promote the two remaining `example` bindings to **preferred** (matches the strength already used on the canonical `item.adjudication.category` slots), so the resource is internally consistent:

- `source/claimresponse/structuredefinition-ClaimResponse.xml` line 1905 — change `<strength value="example"/>` → `<strength value="preferred"/>` for the binding at line 1907 (`ValueSet/adjudication`, the `addItem.adjudication.category` slot).
- `source/explanationofbenefit/structuredefinition-ExplanationOfBenefit.xml` line 3424 — change `<strength value="example"/>` → `<strength value="preferred"/>` for the binding at line 3426 (`ValueSet/adjudication`, the EOB `addItem.adjudication.category` slot).

Leave the two existing `preferred` bindings (ClaimResponse line 801, EOB line 1900) as-is — they already satisfy the resolution.

### Design Rationale

- **Promote, don't replace.** The resolution explicitly directs creating "a new code system that is not an example" rather than retiring the existing one. In practice this has been implemented in UTG by promoting the existing canonical (`http://terminology.hl7.org/CodeSystem/adjudication`) from example to authoritative — preserving all existing bindings, instance data, and OIDs (`2.16.840.1.113883.4.642.1.1171`). Adding concepts to the existing canonical avoids forcing every downstream binder (CARIN-BB, RTPBC, EOB profiles, US-Core derivatives) to rewire their references.
- **Preferred over extensible** is consistent with what FM has already chosen on the two main `item.adjudication.category` slots. `extensible` would force every implementer to come from this list when a concept fits, which is too aggressive for a category whose long tail is genuinely IG-specific (e.g., dental, vision, pharmacy local categories). `preferred` signals "use these codes when they apply, otherwise extend" — exactly the pattern FM has been migrating toward.
- **Code-style normalisation** (kebab-case for all new codes) reflects what the resolution explicitly asked for in item (3), and matches how more recent FM/terminology codes (`coupon-discount`, `member-discount`, `other-discount`, `other-cost`) were already proposed in the ticket. Renaming the eight legacy codes is out of scope and would be a breaking change.
- **Alternatives considered.** (a) Creating a parallel `adjudication-extended` code system and union-ValueSet — rejected because it splits the canonical and forces every downstream IG binding to migrate. (b) Importing CARIN-BB / RTPBC code systems directly into the ValueSet — rejected because those code systems live in IGs and would create a cyclic core→IG dependency. (c) Leaving the existing example status — rejected; that is the very thing the resolution overturns.

## Impact Analysis

### Affected Repositories

#### HL7/UTG (Terminology)

- **Role:** Owns the canonical `CodeSystem` (and ValueSet wrapper) at `http://terminology.hl7.org/CodeSystem/adjudication`. All concept additions and the "no longer experimental" promotion live here.
- **Affected Files:**
  - `input/sourceOfTruth/fhir/codeSystems/CodeSystem-adjudication.xml` — append 13 concepts; bump `version`/`meta.lastUpdated`; (style normalisation per FMWG decision).
  - `input/sourceOfTruth/fhir/valueSets/ValueSet-adjudication.xml` — verify `experimental="false"`; ensure `compose.include` form (no enumerated concepts) so new concepts flow through automatically.
- **Change Scope:** Moderate (terminology-only; no schema changes, but additive concepts impact every consumer of the ValueSet).

#### HL7/fhir (FhirCore)

- **Role:** Contains the in-core mirror of the ValueSet and the bindings from FM resources that depend on it.
- **Affected Files:**
  - `source/claimresponse/valueset-adjudication.xml` — flip `experimental` from `true` to `false`; replace example-set copyright with HL7 Terminology copyright.
  - `source/claimresponse/structuredefinition-ClaimResponse.xml` — promote `example` → `preferred` for the `ValueSet/adjudication` binding at line 1907 (the binding at line 801 is already `preferred`; leave untouched).
  - `source/explanationofbenefit/structuredefinition-ExplanationOfBenefit.xml` — promote `example` → `preferred` for the `ValueSet/adjudication` binding at line 3426 (the binding at line 1902 is already `preferred`; leave untouched).
- **Change Scope:** Minor (four small, surgical edits across three files — all inside `<differential>`/authored content).

### Breaking Changes

- **Yes — binding strength promotion is a breaking change for conformance.** Moving `ClaimResponse.addItem.adjudication.category` and the corresponding EOB element from `example` to `preferred` does not invalidate existing instances, but it does change downstream conformance expectations and validator messages (a non-listed code now produces a "should" warning rather than being silently accepted). Downstream IGs that further constrain these elements (CARIN-BB, US Core EOB, dental/vision profiles) should be informed.
- **Code additions are additive** — not breaking on the wire. The CodeSystem `version` bump is consumer-visible but non-breaking.
- **Style normalisation** (if FMWG approves renaming the new codes to kebab-case) is **not** breaking against existing instances (no instances exist for codes that have not yet shipped) but would be breaking against any draft/pre-publication tooling that has already wired the as-submitted spellings (e.g., the CARIN-BB / RTPBC harmonisation work).
