## Resolution Summary

**Resolution:** Persuasive.

The Financial Management work group, with FMG agreement, had previously decided
that `EnrollmentRequest` (and its paired `EnrollmentResponse`) is out of scope
for FHIR Core and should be migrated to an implementation guide. That migration
had not yet happened in the FHIR Core source. The resolution requires the
relocation to be completed before the next ballot — i.e., remove
`EnrollmentRequest` (and `EnrollmentResponse`) from the FHIR Core specification
in `HL7/fhir` so that they no longer ship as part of the core ballot.

The ticket text only names `EnrollmentRequest`, but the two resources are
designed and modelled as a request/response pair (see the `[r3-renames]`
comment in `source/fhir.ini`: *"Enrollment became EnrollmentRequest/Response"*,
and the paired `[%res-item EnrollmentRequest%]` / `[%res-item EnrollmentResponse%]`
listings in `source/financial-module.html`). Removing the request without the
response would leave a dangling resource. Treat the pair as a single unit of
work and confirm with FM whether the response is in scope before merging
(captured under Open Questions).

## Feature Proposal

### Problem Statement

The `EnrollmentRequest` and `EnrollmentResponse` resources currently live in
FHIR Core under `source/enrollmentrequest/` and `source/enrollmentresponse/`
and are wired into many cross-cutting registries (`source/fhir.ini`,
`source/hierarchy.xml`, `source/compartments.xml`, `source/resourcelist.html`,
`source/financial-module.html`, `source/workflow.html`,
`source/searchparameter/common-search-parameters.xml`, `source/oids.ini`,
`source/status-codes.xml`, `source/resource-name-tracker.xml`). They are also
referenced from a number of other resources' `targetProfile`s
(`AuditEvent.entity.what`, `Composition.event.detail`,
`DocumentReference.context.related`, `GuidanceResponse.evaluationMessage` /
similar slot, `Provenance.entity.what`) and listed in workflow / financial
overview pages.

The FM work group and FMG agreed the Enrollment* resources are out of scope
for FHIR Core and belong in an implementation guide. Until they are removed,
the next ballot of FHIR Core would continue to publish out-of-scope content,
contradicting the work group decision and inflating the ballot surface that
balloters must review.

### Proposed Change

Remove the `EnrollmentRequest` (and paired `EnrollmentResponse`) resources
from the `HL7/fhir` core source so they do not ship in the next ballot:

1. **Delete the per-resource source folders** `source/enrollmentrequest/` and
   `source/enrollmentresponse/` in their entirety, including the
   StructureDefinitions
   (`structuredefinition-EnrollmentRequest.xml`,
   `structuredefinition-EnrollmentResponse.xml`), introductions, notes,
   examples, search-param bundles, mapping-exception files, list-* files, and
   the `enrollment-outcome` `CodeSystem` / `ValueSet` defined in
   `source/enrollmentresponse/`.
2. **Deregister both resources from `source/fhir.ini`** by removing every
   keyed entry across all sections that enumerate resources (lines 141–142,
   251–252, 545–546, 693–694, 838–839, 1150–1151, 1365–1366, and 2644 — all
   listed under "Affected Files" below) and **adding `EnrollmentRequest` and
   `EnrollmentResponse` to the `[removed-resources]` section** (currently
   ends near line ~447, with the legacy `Enrollment` entry already on line
   351). Add a comment near the existing `; Enrollment became
   EnrollmentRequest/Response` line (1461) noting that Enrollment* has now
   moved to an FM implementation guide and citing FHIR-55225.
3. **Drop them from the page registries**: remove the `<page type="resource"
   resource="EnrollmentRequest"/>` / `EnrollmentResponse` entries from
   `source/hierarchy.xml` (lines 383–384) and the corresponding rows from
   `source/compartments.xml` (lines 413, 418).
4. **Remove the resource list / module entries** in
   `source/resourcelist.html` (lines 204–205, 334–335),
   `source/financial-module.html` (lines 48–49, 79–80, 194 — including the

   239), `source/workflow-module.html` (line 112), `source/resourceguide.html`

5. **Remove `targetProfile` references** to

   the existing entries enumerate `Reference(EnrollmentRequest)` /

   tokens from each enumerated list rather than deleting the whole `<issue>`.
9. **Remove the `EnrollmentRequest.subject=EnrollmentRequest.candidate`

implementation guide is **not** in scope for this ticket; it is presumed to
happen in a separate FM IG repository (no such IG repo is referenced from
the cross-references for this ticket).

### Design Rationale

This change implements an already-agreed work-group decision; the design
question is *how* to extract the resources cleanly, not *whether*. The
chosen approach mirrors the standard FHIR Core pattern for resource
retirement, evidenced by the `[removed-resources]` section in
`source/fhir.ini` (containing `Eligibility`, `EligibilityRequest`,
`EligibilityResponse`, `Enrollment`, etc.) and the `[r3-renames]` comment
block (`; EligibilityRequest/Response became
CoverageEligibilityRequest/Response`, `; Enrollment became
EnrollmentRequest/Response`). Adding `EnrollmentRequest` /
`EnrollmentResponse` to `[removed-resources]` and deleting the source
folders is the same recipe that removed earlier resources from core — it
keeps the publisher's bookkeeping consistent and prevents any other page
registry from re-introducing them implicitly.

Removing `targetProfile` references (rather than replacing them with a
remote IG URL) keeps FHIR Core self-contained: cross-IG profile references
should not appear in core StructureDefinitions because that creates a
publish-time dependency on an IG that does not yet exist. Implementers who
need to point at the IG version of `EnrollmentRequest` can do so via
profile slicing inside the IG itself.

The ticket explicitly mentions only `EnrollmentRequest`, but
`EnrollmentResponse` is treated as in scope because (a) it is paired in
every workflow, module, and registry listing in the source, and (b) it has
no independent purpose without the request. The Open Questions section
captures this for FM confirmation.

### Breaking Changes

This is a breaking change for any implementer relying on
`EnrollmentRequest` or `EnrollmentResponse` as part of FHIR Core, and
specifically:

- Any resource currently allowing
  `Reference(... | EnrollmentRequest | ...)` will no longer accept
  `EnrollmentRequest` as a target profile in core. Affected:
  `AuditEvent.entity.what`, `Composition.event.detail`,
  `DocumentReference.context.related`,
  `GuidanceResponse.evaluationMessage` (or whichever element holds the
  reference at line 345 — verify when editing), and `Provenance.entity.what`.
- The two common search parameters previously declared with
  `<base value="EnrollmentRequest"/>` (`identifier` and `patient`) will no
  longer apply to `EnrollmentRequest`. Implementers who relied on
  searching `EnrollmentRequest?identifier=...` or
  `EnrollmentRequest?patient=...` against a core endpoint will need to
  switch to the IG version.
- The `enrollment-outcome` `CodeSystem` and `ValueSet`
  (`http://hl7.org/fhir/enrollment-outcome` family) are removed from core.
  Any implementer or downstream IG referencing those terminology URLs from
  core will break and must move to the IG-published versions.
- Removing both resources from FHIR Core is itself a breaking change
  relative to the prior ballot.
