## Resolution Summary

Resolution: **Persuasive**. Per the resolution description — *"Apply change of binding to Extensible as proposed."* — the binding strength on `Claim.careTeam.role` (which targets `http://hl7.org/fhir/ValueSet/claim-careteamrole`) must be changed from the current value to **`extensible`**. The same value set is reused by `ExplanationOfBenefit.careTeam.role` with an identical binding block, so the change must be applied in both StructureDefinitions to keep the two resources consistent (both currently declare `<strength value="preferred"/>` — see `source/claim/structuredefinition-Claim.xml:735` and `source/explanationofbenefit/structuredefinition-ExplanationOfBenefit.xml:866`).

Although the ticket title mentions "CodeSystem binding", FHIR bindings are always to a ValueSet. The binding to update is the `<binding>` block that references `ValueSet/claim-careteamrole`; the underlying CodeSystem (`http://terminology.hl7.org/CodeSystem/claimcareteamrole`, maintained in HL7/UTG) is not itself being re-bound.

## Feature Proposal

### Problem Statement

`Claim.careTeam.role` and `ExplanationOfBenefit.careTeam.role` are bound to `http://hl7.org/fhir/ValueSet/claim-careteamrole` with strength `preferred`. The ValueSet is flagged as `experimental = true` with description *"This value set includes sample Claim Care Team Role codes."* (see `source/claim/valueset-claim-careteamrole.xml:25,33`). The submitter (Financial Management WG) asserts that a `preferred` binding is weaker than what the WG intends for this element: implementers should be constrained to use codes from this ValueSet *or* codes whose meaning is covered by it, rather than being free to substitute any coding they wish. `extensible` is the FHIR binding strength that captures exactly this intent.

### Proposed Change

In both StructureDefinitions below, replace the `<strength>` element inside the `<binding>` block for the `careTeam.role` element so it reads `extensible` instead of `preferred`. No other fields in the binding change: the `elementdefinition-bindingName` extension (`CareTeamRole`), the `description` (`"The role codes for the care team members."`), and the `valueSet` URL (`http://hl7.org/fhir/ValueSet/claim-careteamrole`) all remain as-is.

Exact after-state:

```xml
<binding>
  <extension url="http://hl7.org/fhir/StructureDefinition/elementdefinition-bindingName">
    <valueString value="CareTeamRole"/>
  </extension>
  <strength value="extensible"/>
  <description value="The role codes for the care team members."/>
  <valueSet value="http://hl7.org/fhir/ValueSet/claim-careteamrole"/>
</binding>
```

No changes are required to the ValueSet definition itself (`source/claim/valueset-claim-careteamrole.xml`), to the referenced CodeSystem in HL7/UTG, or to any instance examples — the `extensible` strength still permits the codes already used in examples.

### Design Rationale

- `extensible` is the correct strength when a ValueSet is the authoritative source for the concept but implementers may add additional codes to cover use cases not anticipated by the base spec. That matches the stated intent of `claim-careteamrole` and is consistent with how similar role-style bindings are handled elsewhere in the Financial Management module.
- Keeping Claim and ExplanationOfBenefit aligned is an existing invariant of this module — both resources currently share identical `<binding>` blocks for `careTeam.role`, so both must be updated together to avoid drift.
- Changing strength from `preferred` → `extensible` does not invalidate any existing conformant instance (codes from the bound ValueSet remain valid; codes outside it that are currently tolerated under `preferred` remain tolerated under `extensible` provided they cover a concept not already in the ValueSet). See Breaking Changes for a full assessment.

### Breaking Changes

Strengthening a binding from `preferred` to `extensible` is a **normatively meaningful** change and should be recorded as such in the R5+ change log:

- Under `preferred`, implementers are *encouraged* but not required to use codes from the ValueSet; under `extensible`, implementers **must** use a code from the ValueSet when one covers the concept, and may only add codes for concepts not present in the ValueSet.
- Existing instances that use codes from `ValueSet/claim-careteamrole` remain valid.
- Existing instances that use codes **outside** the ValueSet whose concept **is** represented inside the ValueSet become non-conformant under the new strength. Given the ValueSet itself is marked `experimental = true` and described as a sample set, this is a real (if small) conformance impact for early adopters.

Recommend flagging this as a non-additive change in the R5+ change log and calling it out in the Claim and ExplanationOfBenefit resource change notes.
