## Feature Proposal

### Problem Statement

In R5, the EnrollmentResponse resource binds its `outcome` element to
`http://hl7.org/fhir/ValueSet/enrollment-outcome`, which itself includes
the CodeSystem `http://hl7.org/fhir/enrollment-outcome`. While the
ValueSet computational/human names already read as `EnrollmentOutcome` /
`Enrollment Outcome`, the underlying CodeSystem still carries
`ClaimProcessingCodes` / `Claim Processing Codes` / a "Claim Processing
Outcome codes" description â€” clearly copy-pasted from the
`http://hl7.org/fhir/ValueSet/claim-outcome` family used by Claim,
ClaimResponse, and friends. This produces:

- A `CodeSystem.name` collision: multiple distinct CodeSystems in core
  declare `name = "ClaimProcessingCodes"`, which violates the convention
  that `name` be a unique computational identifier, breaks generated code
  (class/enum names) in client SDKs, and is misleading to implementers
  reading the spec or terminology server output.
- A misleading title and description that talk about claim processing in
  a CodeSystem that exclusively codes EnrollmentResponse outcomes.

### Proposed Change

Edit the EnrollmentResponse outcome CodeSystem in the HL7/fhir core spec
to align its identifying metadata with the EnrollmentResponse resource and
its sibling ValueSet:

- File: `source/enrollmentresponse/codesystem-enrollment-outcome.xml`
  - `CodeSystem.name`: change from `ClaimProcessingCodes` to
    **`EnrollmentOutcome`** (mirrors the sibling ValueSet's `name`).
  - `CodeSystem.title`: change from `Claim Processing Codes` to
    **`Enrollment Outcome`** (mirrors the sibling ValueSet's `title`).
  - `CodeSystem.description`: change from
    "This value set includes Claim Processing Outcome codes." to
    **"This code system defines the set of codes used to indicate the
    outcome of processing of an EnrollmentRequest."** (aligns wording with
    the resource's `outcome` element and removes the misleading "Claim
    Processing" framing; also corrects the "value set" wording, since
    this artifact is a CodeSystem).

No changes to the canonical `url`, `id`, `OID`, `version`, `status`,
`caseSensitive`, `content`, bound `valueSet`, or the four enumerated
concepts (`queued`, `complete`, `error`, `partial`). The codes themselves
are correct and in active use â€” only the metadata identifying the
CodeSystem changes.

Example of the after-state (relevant lines only):

```xml
<name value="EnrollmentOutcome"/>
<title value="Enrollment Outcome"/>
...
<description value="This code system defines the set of codes used to indicate the outcome of processing of an EnrollmentRequest."/>
```

### Design Rationale

- **Consistency with the bound ValueSet.** The matching ValueSet at
  `source/enrollmentresponse/valueset-enrollment-outcome.xml` already
  uses `name = "EnrollmentOutcome"` and `title = "Enrollment Outcome"`,
  so renaming the CodeSystem to match restores the standard FHIR pattern
  in which a 1-system / 1-ValueSet pair shares its computational name.
- **`name` must be unique-ish.** The whole point of `Resource.name` on
  conformance resources is to give code generators a stable, distinctive
  identifier. Several outcome CodeSystems in the FM space currently share
  `name = "ClaimProcessingCodes"` (see *Related Tickets* below); fixing
  the EnrollmentResponse one is FHIR-40747's specific scope.
- **Resolution is "Persuasive with Modification".** The reporter
  proposed renaming both `name` and `title`; the workgroup's
  modification (per the resolution slip-of-paper "see 40817" and the
  `FM-R5+_ToBeApplied` label) is taken to mean: do the rename, and also
  scrub the copy-pasted description so it actually describes the
  enrollment outcome CodeSystem rather than a claim-processing one.
- **Alternative considered: rename to `EnrollmentOutcomeCodes`.** Other
  CodeSystems in core sometimes append `Codes` to their `name`. Rejected
  because the bound ValueSet uses `EnrollmentOutcome`; matching it
  exactly is the lower-friction option and avoids needing a second pass
  on the ValueSet.

### Breaking Changes

The CodeSystem's canonical `url`, `id`, `OID`, `version`, `caseSensitive`,

(`http://hl7.org/fhir/ValueSet/enrollment-outcome`) is unchanged. Code

`error`, `partial`) continue to validate.

The only consumer-visible deltas are:

- **Generated client code that hangs off `CodeSystem.name`.** SDKs that
  produce class/enum names from `CodeSystem.name` will now generate
  `EnrollmentOutcome` instead of `ClaimProcessingCodes` for this
  CodeSystem. Downstream consumers that imported the old name by
  identifier will need to re-bind. This is an intentional fix â€” the
  pre-fix name was a duplicate of the Claim/ClaimResponse family â€” but
  it is a source-level breaking change for code generators and should be
  called out in the R6 release notes.
- **Display-only metadata** (`title`, `description`) â€” non-breaking.

No breaking changes to instance validation or to the wire format.
