# Implementation Plan: FHIR-40748

**Title:** Payment outcome code system has ClaimProcessingCodes as a name
**Status:** Resolved - change required
**Resolution:** Persuasive with Modification
**Work Group:** Financial Mgmt
**Specification:** FHIR Core (FHIR)
**Resolved:** 2025-01-29

---

## Resolution Summary

The resolution for FHIR-40748 defers to FHIR-41817: *"see 41817 for resolution."*

FHIR-41817's resolution states: **"Propose that the three code systems are replaced with a single codesystem 'Processing Outcome', given that the codes are all the same, with a more limited valueset for PaymentReconciliation if the outcome element survives resource analysis in R6."**

Three code systems (`payment-outcome`, `eligibility-outcome`, `enrollment-outcome`) are identical copies of the `claim-outcome` code system — all containing the same four codes (`queued`, `complete`, `error`, `partial`) — but all three incorrectly retain `name="ClaimProcessingCodes"` and `title="Claim Processing Codes"` from copy-paste. The resolution consolidates these into a single CodeSystem.

## Feature Proposal

### Problem Statement

The `payment-outcome` CodeSystem (`http://hl7.org/fhir/payment-outcome`) has `name="ClaimProcessingCodes"` and `title="Claim Processing Codes"`, which are incorrect — these are copy-pasted values from the `claim-outcome` CodeSystem. The same problem exists for `eligibility-outcome` (FHIR-40746) and `enrollment-outcome` (FHIR-40747). All four code systems define the exact same four codes (`queued`, `complete`, `error`, `partial`) with identical definitions, making three of them redundant.

### Proposed Change

Per the FHIR-41817 resolution, consolidate the three duplicated code systems into a single new CodeSystem:

1. **Create a new unified CodeSystem** — `processing-outcome` with:
   - `name="ProcessingOutcomeCodes"`
   - `title="Processing Outcome Codes"`
   - URL: `http://hl7.org/fhir/processing-outcome` (or reuse `http://hl7.org/fhir/claim-outcome`)
   - Same four codes: `queued`, `complete`, `error`, `partial`

2. **Deprecate/remove three redundant CodeSystems:**
   - `http://hl7.org/fhir/payment-outcome` (this ticket)
   - `http://hl7.org/fhir/eligibility-outcome` (FHIR-40746)
   - `http://hl7.org/fhir/enrollment-outcome` (FHIR-40747)

3. **Update ValueSets** to reference the unified CodeSystem:
   - `payment-outcome` ValueSet → include from unified CodeSystem
   - `eligibility-outcome` ValueSet → include from unified CodeSystem
   - `enrollment-outcome` ValueSet → include from unified CodeSystem
   - `claim-outcome` ValueSet → include from unified CodeSystem (if claim-outcome CS is not the base)

4. **Consider a more limited ValueSet for PaymentReconciliation** if the `outcome` element survives R6 resource analysis.

**Alternative (minimal fix for this ticket alone):** If the consolidation is deferred, at minimum rename the `payment-outcome` CodeSystem:
- `name` → `"PaymentOutcomeCodes"`
- `title` → `"Payment Outcome Codes"`
- `description` → `"This value set includes Payment Processing Outcome codes."`

### Design Rationale

- All four CodeSystems define identical codes — maintaining separate copies creates confusion and naming collisions (three share the same `name`/`title`).
- The Zulip discussion (implementers > ClaimProcessingCode, 2023-08-18) confirms this was unintentional copy-paste, with community consensus that consolidation to a single CodeSystem with multiple ValueSets is the cleanest approach.
- Lloyd McKenzie noted that duplicate titles would fail validation in HL7 IGs but don't currently fail in the core spec, making this a latent issue.

### Breaking Changes

- **CodeSystem URL changes:** If the three CodeSystems are removed/consolidated, any system sending codes with the old CodeSystem URLs (`http://hl7.org/fhir/payment-outcome`, `http://hl7.org/fhir/eligibility-outcome`, `http://hl7.org/fhir/enrollment-outcome`) will need to update to use the new unified URL.
- **CodeSystem `name` changes:** The `name` field is a computable identifier. Changing from `ClaimProcessingCodes` to a new name could break CQL expressions or tooling that references the type by name (see `library-fhir-model-definition-content.xml` references to `FHIR.ClaimProcessingCodes`).
- **ValueSet bindings remain stable:** Since the ValueSet URLs don't change and the codes remain the same, the ValueSet-level bindings in StructureDefinitions are unaffected.
- **Severity:** Moderate for R6 (as a major version boundary this is acceptable); would be a breaking change if applied to R5.

## Implementation Plan

### Prerequisites

1. Coordinate with FHIR-41817, FHIR-40746, and FHIR-40747 — all four tickets share the same resolution and should be implemented together as a single change set.
2. Confirm with the Financial Mgmt work group whether to:
   - (a) Reuse the existing `claim-outcome` CodeSystem URL as the unified system, or
   - (b) Create a new `processing-outcome` CodeSystem URL.
3. Determine if the `PaymentReconciliation.outcome` element will survive R6 resource analysis (per the resolution note about a "more limited valueset").

### Step-by-Step Tasks

#### HL7/fhir — Option A: Consolidate onto existing `claim-outcome` CodeSystem

1. **Rename `claim-outcome` CodeSystem name/title** — File: `source/claimresponse/codesystem-claim-outcome.xml`
   - Action: Consider renaming `name` from `ClaimProcessingOutcomeCodes` to `ProcessingOutcomeCodes` and `title` to `Processing Outcome Codes` if this becomes the universal outcome CodeSystem. *(Optional — current name is already descriptive enough for ClaimResponse context.)*

2. **Update `payment-outcome` CodeSystem** — File: `source/paymentreconciliation/codesystem-payment-outcome.xml`
   - Action: Either (a) delete this file entirely if consolidating, or (b) rename `name` from `"ClaimProcessingCodes"` to `"PaymentOutcomeCodes"` and `title` to `"Payment Outcome Codes"` as a minimal fix.

3. **Update `payment-outcome` ValueSet** — File: `source/paymentreconciliation/valueset-payment-outcome.xml`
   - Action: Change `<system value="http://hl7.org/fhir/payment-outcome"/>` to `<system value="http://hl7.org/fhir/claim-outcome"/>` (or the new unified URL).

4. **Update `eligibility-outcome` CodeSystem** — File: `source/coverageeligibilityresponse/codesystem-eligibility-outcome.xml`
   - Action: Either delete or rename `name`/`title` (per FHIR-40746).

5. **Update `eligibility-outcome` ValueSet** — File: `source/coverageeligibilityresponse/valueset-eligibility-outcome.xml`
   - Action: Change include system to the unified CodeSystem URL.

6. **Update `enrollment-outcome` CodeSystem** — File: `source/enrollmentresponse/codesystem-enrollment-outcome.xml`
   - Action: Either delete or rename `name`/`title` (per FHIR-40747).

7. **Update `enrollment-outcome` ValueSet** — File: `source/enrollmentresponse/valueset-enrollment-outcome.xml`
   - Action: Change include system to the unified CodeSystem URL.

8. **Fix `claim-outcome` ValueSet name** — File: `source/claimresponse/valueset-claim-outcome.xml`
   - Action: Rename `name` from `"ClaimProcessingCodes"` to `"ClaimOutcome"` (matching the pattern of the other ValueSets which have correct names like `PaymentOutcome`, `EligibilityOutcome`).

9. **Update OIDs** — File: `source/oids.ini`
   - Action: If CodeSystem URLs change, update or remove the corresponding OID entries (lines 1235, 1242, 1387 for CodeSystems). If a new unified CodeSystem is created, add its OID entry.

10. **Update library model definition** — File: `source/library/library-fhir-model-definition-content.xml`
    - Action: If the type name changes from `ClaimProcessingCodes`, update references at lines 203, 1121, 1492, and 11260. *(Note: This file may be auto-generated by the build tooling.)*

### Open Questions

1. **Consolidation vs. rename?** The FHIR-41817 resolution proposes full consolidation into a single CodeSystem. Should this be implemented now, or should we start with simple renames (Option B) and consolidate later?
2. **Which URL for the unified CodeSystem?** Should we reuse `http://hl7.org/fhir/claim-outcome` or create a new `http://hl7.org/fhir/processing-outcome`?
3. **PaymentReconciliation outcome survival in R6?** The resolution mentions a "more limited valueset for PaymentReconciliation if the outcome element survives resource analysis in R6." Has this analysis been completed?
4. **Should the `claim-outcome` ValueSet name also be fixed?** It currently has `name="ClaimProcessingCodes"` which doesn't match the CodeSystem name `ClaimProcessingOutcomeCodes` — this appears to be a separate minor inconsistency.
