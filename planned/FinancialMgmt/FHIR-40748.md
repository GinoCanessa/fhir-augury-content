## Resolution Summary

The ticket's resolution description states "see 41817 for resolution." FHIR-41817 ("3 valuesets with identical Title and Name") is the umbrella ticket that consolidates the three sibling tickets FHIR-40746 (eligibility), FHIR-40747 (enrollment), and FHIR-40748 (payment), all of which report the same defect: the `claim-outcome`, `eligibility-outcome`, `enrollment-outcome`, and `payment-outcome` CodeSystems all share the `name = ClaimProcessingCodes` and `title = Claim Processing Codes`, so they are not uniquely identifiable by computable name.

The umbrella resolution on FHIR-41817 (resolution: **Persuasive with Modification**) is verbatim:

> "Propose that the three code systems are replaced with a single codesystem 'Processing Outcome', given that the codes are all the same, with a more limited valueset for PaymentReconciliation if the outcome element survives resource analysis in R6."

For FHIR-40748 specifically, the actionable change is therefore: retire the standalone `payment-outcome` CodeSystem in favour of a shared `processing-outcome` CodeSystem (canonical `http://hl7.org/fhir/processing-outcome`, computable name `ProcessingOutcome`, title `Processing Outcome`), and re-anchor the existing `payment-outcome` ValueSet so that it narrows the consolidated CodeSystem to the four codes currently bound by `PaymentReconciliation.outcome` (`queued`, `complete`, `error`, `partial`). The `PaymentReconciliation.outcome` element keeps its existing required binding to `http://hl7.org/fhir/ValueSet/payment-outcome`.

The label `FM-R5+_ToBeApplied` confirms this is an R5+ (i.e., R6) change that has been resolved but not yet applied to the spec source.

## Feature Proposal

### Problem Statement

Verified from the clone (`source/paymentreconciliation/codesystem-payment-outcome.xml` line 23–24):

```xml
<name value="ClaimProcessingCodes"/>
<title value="Claim Processing Codes"/>
```

Three other FM CodeSystems (`claim-outcome`, `eligibility-outcome`, `enrollment-outcome`) carry the same `name`/`title`. The CodeSystem resource definition requires `name` to be a computable (machine-friendly) identifier, and consumers (code generators, IG tooling, terminology browsers) rely on `name` being unique within a publication. Four CodeSystems sharing the same `name` collide in any tool that keys off `name`, and a human reading "Claim Processing Codes" cannot tell which CodeSystem they are looking at.

The four CodeSystems also have **identical concept lists** (`queued`, `complete`, `error`, `partial`, with the same display text and definitions). Renaming each in isolation would still leave four redundant CodeSystems duplicating the same vocabulary, which is the deeper concern raised on FHIR-41817.

### Proposed Change

Apply the FHIR-41817 umbrella resolution to the **payment** half of the trio. The single ticket FHIR-40748 is implemented as the following coordinated set of edits in `HL7/fhir`:

1. **Introduce a consolidated CodeSystem `processing-outcome`** under the Financial Mgmt area (proposed location: `source/paymentreconciliation/codesystem-processing-outcome.xml`, but the WG may prefer a neutral home such as `source/financial/`). Field values:
   - `id` = `processing-outcome`
   - `url` = `http://hl7.org/fhir/processing-outcome`
   - `name` = `ProcessingOutcome`
   - `title` = `Processing Outcome`
   - `status` = `active`
   - `experimental` = `false`
   - `caseSensitive` = `true`
   - `content` = `complete`
   - `description` = "Outcome codes shared by the financial-management ClaimResponse, CoverageEligibilityResponse, EnrollmentResponse and PaymentReconciliation `.outcome` elements."
   - Carry across the standards-status/fmm/wg extensions currently on `codesystem-payment-outcome.xml` (`fm`, `normative`, `fmm 2`).
   - Concepts: `queued`, `complete`, `error`, `partial` with the existing display/definition text (the four current entries are already identical across all four sibling CodeSystems).

2. **Replace the standalone `payment-outcome` CodeSystem** at `source/paymentreconciliation/codesystem-payment-outcome.xml`. Per the umbrella resolution the standalone CodeSystem is removed; the file is deleted (and any registry/index entries cleaned up — see Step 4 of the implementation plan below).

3. **Re-anchor the `payment-outcome` ValueSet** (`source/paymentreconciliation/valueset-payment-outcome.xml`) so that its `compose.include.system` points at `http://hl7.org/fhir/processing-outcome` instead of `http://hl7.org/fhir/payment-outcome`. The ValueSet keeps its existing canonical (`http://hl7.org/fhir/ValueSet/payment-outcome`), `name` (`PaymentOutcome`), `title` (`Payment Outcome`), and `id` (`payment-outcome`). Per the umbrella resolution, this is the "more limited valueset for PaymentReconciliation". Because the consolidated CodeSystem only contains the four codes currently in use, no explicit `concept` filter is required at the ValueSet level — but adding an explicit `concept` filter (`queued`, `complete`, `error`, `partial`) is acceptable and protects the binding from accidental widening if other codes are later added to `processing-outcome`.

   Example:

   ```xml
   <compose>
     <include>
       <system value="http://hl7.org/fhir/processing-outcome"/>
       <concept><code value="queued"/></concept>
       <concept><code value="complete"/></concept>
       <concept><code value="error"/></concept>
       <concept><code value="partial"/></concept>
     </include>
   </compose>
   ```

4. **Leave the `PaymentReconciliation.outcome` binding intact.** The element at `structuredefinition-PaymentReconciliation.xml` lines 319–338 already binds (required) to `http://hl7.org/fhir/ValueSet/payment-outcome` with bindingName `PaymentOutcome`. No change to the StructureDefinition is required for this ticket, because the ValueSet canonical does not move.

5. **Coordinate with the sibling tickets** (FHIR-40746 → `eligibility-outcome`, FHIR-40747 → `enrollment-outcome`, plus the analogous Persuasive-with-Modification work for `claim-outcome` referenced by FHIR-41817). Each sibling ticket is responsible for re-anchoring its own ValueSet at `http://hl7.org/fhir/processing-outcome` and deleting its own redundant CodeSystem. The new `processing-outcome` CodeSystem itself should only be added once — whichever ticket lands first creates it, and the others' plans must check for its presence rather than re-introducing it. Coordinating ownership of the new file is the only implementation question this ticket needs the WG to confirm (see *Open Questions*).

### Design Rationale

- **Honours the WG decision.** The resolution is `Persuasive with Modification` — the modification is precisely "consolidate into one CodeSystem, narrow the ValueSet". A simple "rename `name`/`title` to PaymentOutcome" fix would resolve FHIR-40748's literal text but ignore the agreed-upon resolution and would leave four near-identical CodeSystems shipping in R6.
- **Preserves backward compatibility for binding consumers.** Because the ValueSet canonical (`http://hl7.org/fhir/ValueSet/payment-outcome`) is unchanged, existing PaymentReconciliation conformance and instances continue to validate. The CodeSystem URL change is observable to direct CodeSystem consumers (rare for this CS) and to terminology servers; this is unavoidable under the chosen resolution and is the price of de-duplication.
- **Keeps the per-resource ValueSet narrow.** The umbrella resolution explicitly calls for a "more limited valueset for PaymentReconciliation". Restricting the `payment-outcome` ValueSet to the four codes currently in use prevents the binding from accidentally admitting future, payment-irrelevant outcome codes that the consolidated `processing-outcome` CodeSystem may grow.
- **Alternatives considered (and rejected per the resolution comments on FHIR-41817):** (a) rename only `name`/`title` on each of the four CodeSystems to `PaymentOutcome` etc. — rejected because it leaves four duplicate vocabularies; (b) collapse all four ValueSets into one — rejected because each resource binding should remain independently constrainable, especially if `PaymentReconciliation.outcome` survives R6 resource analysis with a narrower scope.
