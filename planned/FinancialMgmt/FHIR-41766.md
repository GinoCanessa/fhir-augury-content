## Resolution Summary

The committee disposition is **Persuasive with Modification**. The resolution
description (verbatim from Jira) states:

> For USCLS codes the codesystem will need to be removed or correct the
> copyright and codesystem name.
>
> For ex-visionprescriptionproduct rename the Codesystem to visionproduct
> and change the Copyright to the standard HL7 International copyright.

Concretely this means two scoped fixes to the FM-owned example CodeSystems
hosted in HL7 Terminology (THO):

1. **`http://terminology.hl7.org/CodeSystem/ex-USCLS`** — either retire it,
   or replace the placeholder `copyright` value (`"This is an example
   set."`) with USCLS-appropriate licensing text and ensure the
   `name`/`title` are correct. The reporter additionally requests that a
   non-`ex-` URI be defined by TSMG for "USCLS codes in general"; that
   piece is a TSMG follow-up and is out of scope for the FM
   implementation.
2. **`http://terminology.hl7.org/CodeSystem/ex-visionprescriptionproduct`**
   — rename the CodeSystem (resource `id`/`name`/canonical `url`) to
   **`visionproduct`** and replace the `copyright` value with the
   standard HL7 International copyright statement.

All downstream FHIR-Core ValueSets that compose these systems must follow
suit (corrected copyright text, and updated `system` URI for the renamed
vision product CodeSystem).

## Feature Proposal

### Problem Statement

Two FM-owned example CodeSystems published through HL7 Terminology
(THO) carry a placeholder `copyright` value of `"This is an example
set."`. That string describes the *content* (it is an example value
set) but is **not** a legal/license statement, which is what the
`CodeSystem.copyright` element is intended for. As a result:

- Implementers and downstream IGs cannot determine the actual licensing
  terms for `ex-USCLS` or `ex-visionprescriptionproduct`.
- The vision codes are HL7-defined and should carry HL7's standard
  copyright/license text, but currently do not.
- The USCLS codes are an external code set; the placeholder hides the
  fact that no real USCLS license has been declared on the FHIR-side
  representation.
- The CodeSystem id/url for the vision codes is namespaced as `ex-…`
  ("example"), implying it is throw-away example content. The committee
  has decided the vision codes are first-class HL7 content and should be
  renamed to drop the `ex-` prefix and the `prescription` qualifier.

The `valueset-service-uscls.xml` and `valueset-vision-product.xml`
ValueSets shipped from HL7/fhir core also still carry the
`"This is an example set."` placeholder copyright and reference the old
`ex-visionprescriptionproduct` CodeSystem URL, so the cleanup must
extend into core as well.

`input/sourceOfTruth/fhir/codeSystems/CodeSystem-ex-USCLS.xml`):

(`"This material derives from the HL7 Terminology (THO). THO is
copyright ©1989+ Health Level Seven International and is made available

https://terminology.hl7.org/license.html"`), which addresses the

  rest of this plan assumes the *retain-with-corrections* path because
  the system is still referenced by core artifacts.
- Verify the CodeSystem `name` (`USCLSCodes`) and `title`
  (`USCLS Codes`) accurately label this as "USCLS codes used by the FHIR
  examples" (consider updating the title/description to make the
  example-only / non-authoritative scope explicit, e.g.
  `"Example USCLS Codes"`, mirroring the vision-product wording).
- The TSMG-defined non-`ex-` URI for full USCLS is *not* being created
  by this ticket; flag it under Open Questions.

**B. HL7/UTG — rename `CodeSystem/ex-visionprescriptionproduct` →

`Claim.item.detail.productOrService` /

`http://hl7.org/fhir/ValueSet/vision-product` (no canonical change

### Design Rationale

- The FHIR `CodeSystem.copyright` element is normatively defined as
  carrying licensing/attribution text. Using it for a content-scope
  comment ("This is an example set.") is a category error and confuses
  implementers about license terms.
- Renaming the vision-product CodeSystem out of the `ex-` namespace
  reflects FM's position that these are durable HL7-curated codes (only
  two concepts, but used in conformance examples and downstream IGs)
  rather than throw-away example material. Aligning the ValueSet
  `name`/`title` keeps the FHIR-core artifact consistent with the
  renamed CodeSystem.
- The USCLS half is treated more conservatively — the resolution
  itself is "remove **or** correct" — because USCLS is ultimately an
  external code set and only TSMG can define a proper non-example URI
  for it. This plan corrects the copyright (the part FM owns today) and
  defers the URI question to TSMG.
- Keeping the FHIR-Core ValueSet canonical URLs (`…/ValueSet/service-uscls`
  and `…/ValueSet/vision-product`) intact avoids a breaking change to
  every binding in `Claim`, `ClaimResponse`, and `VisionPrescription`.

### Breaking Changes

- **Renaming the canonical URL of `CodeSystem/ex-visionprescriptionproduct`
  to `CodeSystem/visionproduct` is a breaking change for any consumer
  that has hard-coded the `ex-` URL.** Mitigation: UTG should publish
  the renamed CodeSystem with its OID
  (`urn:oid:2.16.840.1.113883.4.642.1.1188`) preserved and, per UTG
  practice, retire the old URL with a `replacedBy` pointer.
- All `Coding.system` instances in HL7/fhir examples that reference
  the old `ex-visionprescriptionproduct` URL must change in the same
  release; otherwise example validation will fail once UTG publishes
  the renamed CodeSystem.
- Correcting `CodeSystem.copyright` is **not** a breaking change in the
  conformance sense (no `concept` is removed or relabeled) but is a
  metadata/license-text change that downstream IGs may want to track.
