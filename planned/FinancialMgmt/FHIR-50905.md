## Resolution Summary

The ticket is **Persuasive** with resolution description: *"Apply changes to R5 claimsubtype as described."*

The "as described" wording refers to the original ticket body, which directs:

> Replace `ex-claimsubtype` CodeSystem with a standard CodeSystem in THO. Add: `inpatient`, `outpatient`.

In concrete terms the resolution requires:

1. Promote the existing THO CodeSystem currently published at
   `http://terminology.hl7.org/CodeSystem/ex-claimsubtype`
   ("Example Claim SubType Codes") from an **example** CodeSystem to a
   **regular / standard** THO CodeSystem (drop the `ex-` prefix and the
   "Example" framing in id, url, name, title, and description; flip
   `experimental` to `false`/keep `false` and remove the "sample" wording).
2. **Add** two new concepts to that CodeSystem: `inpatient` and `outpatient`
   (existing concepts `ortho` and `emergency` are retained).
3. Update the FHIR Core (`HL7/fhir`) `ValueSet/claim-subtype` and the THO
   `ValueSet/claim-subtype` to point at the renamed CodeSystem URL, drop
   the "Example" naming and the `experimental=true`/"This is an example
   set" copyright, so the binding used by `Claim.subType`,
   `ClaimResponse.subType` and `ExplanationOfBenefit.subType` resolves to
   a standard (non-example) value set.

The label `FM-R5+_ToBeApplied` indicates the change targets the **R5+
master branch** (i.e. the next FHIR release after R5), not a published
R5 ballot.

## Feature Proposal

### Problem Statement

`Claim.subType`, `ClaimResponse.subType` and
`ExplanationOfBenefit.subType` carry an *example* binding to
`http://hl7.org/fhir/ValueSet/claim-subtype`, which composes a single
THO CodeSystem `http://terminology.hl7.org/CodeSystem/ex-claimsubtype`.

That CodeSystem is currently shipped as an *example*:

- `id = ex-claimsubtype`, `name = ExampleClaimSubTypeCodes`, `title =
  "Example Claim SubType Codes"`, description begins "This value set
  includes sample Claim SubType codes…"
- It defines only two codes: `ortho` (Orthodontic Claim) and
  `emergency` (Emergency Claim).

The two most commonly used institutional sub-type distinctions —
**inpatient** and **outpatient** — are not in the code system, even
though the description text already says "within type institutional
there may be subtypes for emergency services, bed stay and
transportation". Implementers asking how to express
inpatient-vs-outpatient on a Claim/ClaimResponse/EOB therefore have no
HL7-published code to pick. The Financial Management work group has
agreed (per the resolution) that this CodeSystem should be promoted to
a "regular" standard THO CodeSystem and that `inpatient` and
`outpatient` should be added as first-class concepts.

### Proposed Change

**A. THO (`HL7/UTG`) — promote and extend the CodeSystem**

In `input/sourceOfTruth/fhir/codeSystems/CodeSystem-ex-claimsubtype.xml`,
replace the file with a "regular" (non-example) CodeSystem.
Recommended target shape:

- `id` = `claim-subtype` (drop `ex-` prefix)
- `url` = `http://terminology.hl7.org/CodeSystem/claim-subtype`
- `name` = `ClaimSubTypeCodes`
- `title` = `Claim SubType Codes`
- `status` = `active`, `experimental` = `false`
- `version` bumped (next THO publication version)
- `valueSet` companion reference updated to
  `http://terminology.hl7.org/ValueSet/claim-subtype`
- `description` reworded to drop "sample / example" language, e.g.
  *"This code system defines codes used to distinguish claim sub-types,
  e.g. for institutional claims distinguishing inpatient and outpatient
  encounters and other sub-categorisations such as emergency or
  orthodontic."*
- Concepts (preserve order, **add** the two new codes):
  - `ortho` — "Orthodontic Claim" — *A claim for orthodontic services.*
  - `emergency` — "Emergency Claim" — *A claim for emergency services.*
  - **`inpatient`** — "Inpatient Claim" — *A claim for services
    rendered to a patient who has been formally admitted as an
    inpatient.*
  - **`outpatient`** — "Outpatient Claim" — *A claim for services
    rendered to a patient who has not been formally admitted (e.g.
    ambulatory, day-case, or clinic outpatient encounter).*

Because the file id changes (`ex-claimsubtype` → `claim-subtype`), the
existing file should be **renamed** to
`CodeSystem-claim-subtype.xml`. Also update the THO control manifests
that enumerate it:

- `input/sourceOfTruth/control-manifests/fhir-Rendering.xml` line 712:
  `<reference value="CodeSystem/ex-claimsubtype"/>` →
  `<reference value="CodeSystem/claim-subtype"/>`
- `input/sourceOfTruth/control-manifests/allHL7-Rendering.xml` line
  886: same replacement.

The corresponding companion ValueSet in THO,
`input/sourceOfTruth/fhir/valueSets/ValueSet-claim-subtype.xml`, must
be updated so that:

- `name` / `title` drop "Example" and become `ClaimSubTypeCodes` /
  `Claim SubType Codes`
- `description` is reworded to drop "sample"
- the `<compose><include><system>` URL points at the renamed CodeSystem
  `http://terminology.hl7.org/CodeSystem/claim-subtype`
- `version` bumped to match the THO release.

**B. FHIR Core (`HL7/fhir`) — update the example ValueSet that core
publishes alongside Claim/ClaimResponse/EOB**

`source/claim/valueset-claim-subtype.xml` and
`source/explanationofbenefit/valueset-claim-subtype.xml` both define a
core ValueSet at `http://hl7.org/fhir/ValueSet/claim-subtype` whose
sole `compose.include.system` is the THO CodeSystem URL. Both copies
need to be updated to:

- change `compose.include.system` from
  `http://terminology.hl7.org/CodeSystem/ex-claimsubtype` to
  `http://terminology.hl7.org/CodeSystem/claim-subtype`
- drop the "Example" framing in `name`, `title`, and `description`
- in `source/claim/valueset-claim-subtype.xml`, set
  `experimental=false` and replace the `<copyright value="This is an
  example set."/>` with a copyright that aligns with the THO source
  (e.g. *"This material derives from the HL7 Terminology (THO). THO is
  copyright © Health Level Seven International and is made available
  under the CC0 designation."*).
- the second file (`explanationofbenefit/valueset-claim-subtype.xml`)
  is a near-empty stub that already lacks `id`/`url`; bring it into
  sync with the canonical `claim/` copy (or, preferably, delete the
  duplicate — see Open Questions).

The example instances referencing the CodeSystem
(`source/claim/claim-example-institutional.xml`,
`source/claimresponse/claimresponse-example.xml`,
`source/explanationofbenefit/explanationofbenefit-example-2.xml`)
already use codes from this system (`emergency`); their
`<system value="…/ex-claimsubtype"/>` references must be updated to the
new system URL `…/claim-subtype`. At least one example should be
updated to demonstrate the new `inpatient` (or `outpatient`) code.

### Design Rationale

- The Financial Management work group has resolved the ticket as
  **Persuasive** with no modification, accepting the proposal verbatim.
- "Regular" CodeSystem (vs. the `ex-` prefixed example) is the
  standard THO promotion pattern: drop the `ex-` prefix, drop the
  "Example" naming, set `experimental=false`, and let the system stand
  as an HL7-published, non-experimental code list. Other FM domains
  have followed this pattern when example code systems mature into
  agreed value sets.
- Adding `inpatient` and `outpatient` aligns with the existing
  description text that already calls out the institutional sub-type
  distinction, and gives implementers an HL7-blessed way to express
  the most common Claim/EOB sub-typing without inventing local codes.
- Keeping `ortho` and `emergency` preserves backward compatibility for
  any implementer that already populated `subType` from this system.

### Breaking Changes

- **Yes — canonical URL change for the CodeSystem.**
  `http://terminology.hl7.org/CodeSystem/ex-claimsubtype` is the
  current canonical URL. Renaming to
  `http://terminology.hl7.org/CodeSystem/claim-subtype` will break
  consumers that reference the old URL by string match. Mitigations:
  - Coordinate THO release notes / deprecation entry pointing the old
    URL at the new one.
  - The companion `ValueSet/claim-subtype` URL is unchanged
    (id/url stay the same), so resources binding to the ValueSet are
    unaffected — only direct references to the CodeSystem URL break.
- **Behavioural addition** — `inpatient` and `outpatient` are new
  codes; this is **non-breaking** (the binding on `*.subType` is
  *example*, not *required*, so receivers are not obliged to honour
  any specific code set).
- The sub-type binding strength on `Claim.subType` /
  `ClaimResponse.subType` / `ExplanationOfBenefit.subType` is **not**
  being tightened; if a future ticket proposes lifting it from
  `example` to `extensible`/`preferred` that would be the breaking
  change, but it is out of scope here.
