## Resolution Summary

The Jira resolution is **Persuasive with Modification** with the resolution
description `See 41817`. FHIR-41817 ("3 valuesets with identical Title and
Name") covers the same defect across the four "outcome" CodeSystems
(`claim-outcome`, `payment-outcome`, `eligibility-outcome`,
`enrollment-outcome`) and resolves with:

> Propose that the three code systems are replaced with a single
> codesystem 'Processing Outcome', given that the codes are all the same,
> with a more limited valueset for PaymentReconciliation if the outcome
> element survives resource analysis in R6.

What has actually been applied in the clone (HEAD `db79dcd`) follows a
*narrower* form of that resolution: rather than collapsing the four
CodeSystems into one, `claim-outcome` was renamed in place so its
`name`/`title` no longer collide with the other three. The remaining
three CodeSystems — including the one this ticket is filed against —
still carry `name="ClaimProcessingCodes"` / `title="Claim Processing
Codes"` and therefore still violate the uniqueness rule that FHIR-40746
and FHIR-41817 raised.

For FHIR-40746 specifically, the in-scope artifact is **only** the
Eligibility Outcome CodeSystem
(`source/coverageeligibilityresponse/codesystem-eligibility-outcome.xml`).
The corresponding ValueSet (`valueset-eligibility-outcome.xml`) was
already given a unique `name`/`title` (`EligibilityOutcome` / `Eligibility
Outcome`), so it does not need to change. The two sibling CodeSystems
that share the same defect (`payment-outcome`, `enrollment-outcome`) are
called out under *Related Tickets* and *Open Questions* but their fix
belongs under FHIR-41817, not here.

## Feature Proposal

### Problem Statement

Three distinct R5/R6 CodeSystems —
`http://hl7.org/fhir/eligibility-outcome`,
`http://hl7.org/fhir/payment-outcome`, and
`http://hl7.org/fhir/enrollment-outcome` — all share the same
`name="ClaimProcessingCodes"` and `title="Claim Processing Codes"`. This
violates the FHIR conformance rule that CodeSystem `name` should be a
unique computable identifier and `title` a unique human-friendly label.
It also misleads readers because none of those three CodeSystems is
actually the canonical "Claim Processing Codes" set — that label belongs
to the Claim Outcome / ClaimResponse axis.

FHIR-40746 was filed against the Eligibility Outcome CodeSystem in
particular: implementers loading
`codesystem-eligibility-outcome.xml` see a CodeSystem whose `name` and
`title` describe a different domain (claim processing) than the
`id`/`url` and the resource it is wired into
(CoverageEligibilityResponse). Tooling that resolves CodeSystems by
`name` (e.g., FHIRPath/IG-publisher lookups, Sushi imports, downstream
validators) cannot distinguish the three systems.

The companion ticket FHIR-41817 confirms the same defect on
`payment-outcome` and `enrollment-outcome` (and originally
`claim-outcome`, which has since been renamed to
`ClaimProcessingOutcomeCodes` / `Claim Processing Outcome Codes`).

### Proposed Change

Edit the Eligibility Outcome CodeSystem so its `name` and `title` are
unique to that resource, mirroring the rename pattern already applied to
`claim-outcome`:

- **File:** `source/coverageeligibilityresponse/codesystem-eligibility-outcome.xml`
- **Element changes:**
  - `<name value="ClaimProcessingCodes"/>` → `<name value="EligibilityOutcomeCodes"/>`
  - `<title value="Claim Processing Codes"/>` → `<title value="Eligibility Outcome Codes"/>`
- **No other changes** to the CodeSystem are required. In particular,
  do **not** modify:
  - `id` (`eligibility-outcome`) — stable URL slug consumers depend on.
  - `url` (`http://hl7.org/fhir/eligibility-outcome`) — canonical, must not change.
  - `identifier` (OID `urn:oid:2.16.840.1.113883.4.642.4.2072`).
  - `version` (`6.0.0`) — bumped by the publisher process, not by this edit.
  - The four `concept` entries (`queued`, `complete`, `error`, `partial`)
    or their `display` / `definition` text.
  - The bound `valueSet` reference
    (`http://hl7.org/fhir/ValueSet/eligibility-outcome`), whose
    ValueSet (`valueset-eligibility-outcome.xml`) already has the
    correct unique `name="EligibilityOutcome"` / `title="Eligibility
    Outcome"`.

Resulting `<name>`/`<title>` block (snippet):

```xml
<name value="EligibilityOutcomeCodes"/>
<title value="Eligibility Outcome Codes"/>
```

### Design Rationale

- **Consistency with the precedent already in the codebase.** The
  `claim-outcome` CodeSystem (the only one of the four that has already
  been re-applied in this clone) was renamed to
  `name="ClaimProcessingOutcomeCodes"` / `title="Claim Processing
  Outcome Codes"`. The same shape — `<DomainSpecific>OutcomeCodes` for
  the computable name, with a matching human title — applied to
  Eligibility yields `EligibilityOutcomeCodes` / `Eligibility Outcome
  Codes`. This keeps the four CodeSystems visually parallel and
  alphabetically clustered in catalog views.
- **Aligns the CodeSystem with the already-fixed ValueSet.** The
  matching ValueSet in
  `source/coverageeligibilityresponse/valueset-eligibility-outcome.xml`
  is already named `EligibilityOutcome` / `Eligibility Outcome`. Naming
  the CodeSystem `EligibilityOutcomeCodes` / `Eligibility Outcome
  Codes` makes the CS↔VS pair self-describing.
- **Why the `Codes` suffix on the CodeSystem but not the ValueSet?**
  Standard FHIR convention: ValueSet names describe the *concept*
  (`EligibilityOutcome`), CodeSystem names describe the *code list*
  (`EligibilityOutcomeCodes`). This is exactly the convention applied to
  the renamed `claim-outcome`.
- **Why not adopt the literal text of FHIR-41817's resolution
  (collapse all three CodeSystems into a single `Processing Outcome`
  CodeSystem)?** Two reasons:
  1. The FM WG has, in practice, taken the *unique-rename* path — the
     `claim-outcome` rename in this clone is the evidence — rather than
     the collapse path. Following that precedent avoids a much larger,
     URL-breaking change that has not been authored anywhere in the
     clone.
  2. Collapsing canonical URLs is a breaking change for every
     conformance resource currently bound to
     `http://hl7.org/fhir/eligibility-outcome`,
     `…/payment-outcome`, `…/enrollment-outcome`, and would need to be
     scoped by FHIR-41817 across all three siblings — outside the
     stated scope of FHIR-40746 (which names only the Eligibility
     Outcome CodeSystem).
- **Alternative considered:** `name="ClaimProcessingOutcomeCodes"` for
  every sibling. Rejected because it preserves the original collision
  the ticket was filed to fix and is contradicted by the
  already-applied `claim-outcome` rename, which is itself
  `ClaimProcessingOutcomeCodes`.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Hosts the authoritative XML source for the Eligibility
  Outcome CodeSystem and is the single repo published to
  `hl7.org/fhir/codesystem-eligibility-outcome.html`.
- **Affected Files:**
  - `source/coverageeligibilityresponse/codesystem-eligibility-outcome.xml`
    — change `<name value="ClaimProcessingCodes"/>` to
    `<name value="EligibilityOutcomeCodes"/>` and
    `<title value="Claim Processing Codes"/>` to
    `<title value="Eligibility Outcome Codes"/>`. No other lines change.
- **Change Scope:** Minor (two-attribute metadata edit to a single
  conformance resource).

### Breaking Changes

The change is technically a non-breaking metadata correction:

- The CodeSystem `url`, `id`, `version`, `identifier`, and code list are
  unchanged, so any resource bound to
  `http://hl7.org/fhir/eligibility-outcome` continues to resolve and
  validate.
- However, consumers who locate this CodeSystem by its `name` (e.g.,
  CQL/FHIRPath lookups, Sushi `Instance: ClaimProcessingCodes`,
  generated code that uses `name` as an identifier in a target
  language) will need to update their reference from
  `ClaimProcessingCodes` to `EligibilityOutcomeCodes`. In FHIR R5,
  CodeSystem `name` is conformance-relevant (machine-friendly
  identifier), so this should be called out in the changelog under the
  Eligibility / FM section even though no canonical URL moves.
- No code values or display strings change.

### Related Specifications

- **CoverageEligibilityResponse** — the resource that consumes this
  CodeSystem (via `ValueSet/eligibility-outcome` on
  `CoverageEligibilityResponse.outcome`). No StructureDefinition change
  is needed; the binding URL is stable.
- **ValueSet `eligibility-outcome`** — already named `EligibilityOutcome`
  / `Eligibility Outcome`; no change needed but worth re-validating
  after the CodeSystem rename to confirm the CS/VS pair still resolves.
- **CodeSystems `payment-outcome` and `enrollment-outcome`** — carry the
  same defect today but are out of scope for FHIR-40746; tracked under
  FHIR-41817.
- **CodeSystem `claim-outcome`** — already renamed to
  `ClaimProcessingOutcomeCodes`; serves as the precedent for this fix.

### Related Tickets

- **FHIR-41817:** "3 valuesets with identical Title and Name" —
  Persuasive with Modification. Same root defect across all four
  "outcome" CodeSystems. **Complements, does not conflict.** FHIR-40746
  applies the rename to the eligibility CodeSystem; FHIR-41817 is the
  umbrella ticket whose remaining work (rename `payment-outcome` and
  `enrollment-outcome`, evaluate the original "single Processing
  Outcome CodeSystem" proposal) should ride alongside this one. Both
  tickets carry the `FM-R5+_ToBeApplied` label, suggesting a coordinated
  apply window.
- **(Comment in FHIR-41817):** Nick George also noted
  `deviceassociation-status-reason` and `deviceassociation-reason`
  both named `DeviceAssociationCode`. That is an Orders & Observations
  / Devices concern, not Financial Mgmt, and is not part of FHIR-40746.

### Terminology Impact

This **is** a terminology change (a CodeSystem header edit), but it
does **not** require any work in `HL7/UTG`:

- The CodeSystem lives in `HL7/fhir` (the cross-repo touch points in
  the briefing show UTG is the canonical home only for terminology
  *moved out* of core).
- No code, definition, designation, or property is added, removed, or
  modified.
- No ValueSet expansion changes (the bound ValueSet's `compose.include`
  references the canonical URL, which is unchanged).
- Downstream validators using `$validate-code` against
  `http://hl7.org/fhir/eligibility-outcome` continue to behave
  identically.

### Open Questions

- **Naming form:** Does FM prefer the proposed
  `EligibilityOutcomeCodes` (parallel to the already-applied
  `ClaimProcessingOutcomeCodes` for `claim-outcome`), or the literal
  `EligibilityProcessingOutcomeCodes` form? The proposal here follows
  the file-id-driven convention also used by the matching ValueSet
  (`EligibilityOutcome`); confirm with the WG before merging.
- **Scope vs FHIR-41817:** Should the `payment-outcome` and
  `enrollment-outcome` renames be folded into the same PR as a single
  FM cleanup, or kept on FHIR-41817? The label `FM-R5+_ToBeApplied` is
  shared between both tickets, suggesting a single coordinated apply
  is acceptable, but FHIR-40746's literal scope is Eligibility only.
- **Original FHIR-41817 proposal:** That ticket's resolution text
  proposes collapsing the three CodeSystems into a single "Processing
  Outcome" CodeSystem. The pattern actually applied in the clone (rename
  `claim-outcome` rather than collapse) suggests the WG silently
  changed direction. Confirm this with FM before applying — if the WG
  still wants the collapse, this ticket's plan changes substantially
  (URL change, retirement of `eligibility-outcome`, ValueSet rebind).
- **Briefing scope:** The cached `repo-analysis/briefing.md` was

  `mapping-language.html`). Its repo-wide *Facts*, *Authoring root(s)*,
  and *Warnings* sections still apply, but the *Ticket-Relevant Paths*
  and *Recommended Change Recipes* sections are FHIR-42980-specific. A
  briefing refresh scoped to FHIR-40746 is **not** required for this
  plan (no playbook or clone-HEAD drift), but a future
  ticket-specific re-run would surface the eligibility-outcome path
  directly under *Ticket-Relevant Paths*.
