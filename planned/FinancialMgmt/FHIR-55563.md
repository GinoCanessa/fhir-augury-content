## Resolution Summary

**Resolution:** Persuasive with Modification

> "The Contract resource will be pulled from the R6 normative specification and included in an 'in development' specification."

In other words: Contract is **not** ready for normative publication in R6. The
ticket calls for the resource to be removed from the R6 normative ballot and
re-homed in a separate "in development" companion specification, where the
deeper refactor (splitting the contract document, contract definition, and
contract execution into distinct artifacts and reconciling its state machines
with the FHIR workflow pattern) can continue without blocking R6 publication
or making misleading normative commitments.

This is a **release-management / publication-scope** change rather than a
content-level change to the Contract StructureDefinition itself. The
resource's element definitions remain as they are today; what changes is
**where** it is published and **with what standards-status**.

## Feature Proposal

### Problem Statement

As of the original report, the `Contract` resource conflates three distinct
concepts:

1. The contract **document** (a signed/legally-significant artifact),
2. The contract **definition** (a template / contract type),
3. The contract **execution** (the run-time instantiation, terms in force,
   actions taken, etc.).

Reporter notes that this leads to:

- multiple, overlapping state machines on a single resource,
- a model that does not align with the FHIR workflow pattern (Definition →
  Request → Event), and
- an "incoherent" overall lifecycle.

Significant FM workflow-call effort produced an agreed refactor plan, but
that plan has not yet been applied. Publishing the resource as Normative in
R6 would lock in the current incoherent shape and make later refactoring
impossible without breaking normative content. The ticket therefore blocks
Contract from going Normative in R6.

### Proposed Change

Per the resolution, perform the following at the publication / spec-scope
level (no element-level changes to `StructureDefinition-Contract` are required
by *this* ticket):

1. **Remove Contract from the R6 Normative scope.** Set its
   standards-status so that R6 ships Contract as **trial-use / draft** (i.e.
   *not* normative). Concretely, ensure Contract does **not** appear in
   `[normative-package]` and does **not** receive a `[first-normative-version]`
   entry in the R6 publication, and add an explicit `Contract=trial-use`
   entry under `[standards-status]` in `source/fhir.ini` so the build
   stamps the resource accordingly.
2. **Move Contract into an "in development" specification.** Publish the
   Contract resource (its `StructureDefinition`, examples, value sets, code
   systems, narrative pages, and search-parameter bundle) as part of a
   separate "in development" / preview companion publication where the
   refactor work can proceed. Mechanically this is one of:
   - a new branch / IG-style companion publication owned by FM, or
   - inclusion in the existing FHIR "extras" / "draft companion" publication
     used for resources not yet ready for the main spec.
   The exact mechanism is a release-process decision (see Open Questions);
   either way Contract's source files are removed from the R6 main publish
   set and added to the in-development publish set.
3. **Document the move.** Update the Contract resource intro / notes pages
   to explain (a) that it is no longer on the normative track for R6, (b)
   why (the refactor identified by FM is still outstanding), and (c) where
   the in-development version lives. Add an entry to the R5→R6 changes
   notes recording the scope change.
4. **No structural changes to the Contract resource in this ticket.** The
   element model, terminology bindings, and search parameters are
   intentionally left as-is here. The refactor itself (splitting document /
   definition / execution and unifying the state machines) is separate
   workflow-call work and should be tracked under follow-up tickets in the
   in-development spec — *not* applied as part of FHIR-55563.

### Design Rationale

- **The resolution is a scope decision, not a model edit.** The applied
  resolution explicitly says "pulled from the R6 normative specification
  and included in an 'in development' specification." It does not direct a
  specific structural change. This plan therefore restricts itself to the
  publication-scope mechanics and explicitly defers the conceptual refactor.
- **Avoiding a premature Normative lock-in is a well-established FHIR
  practice.** Once an artifact is normative, breaking changes are
  prohibited. Allowing Contract to ride the R6 normative train in its
  current state would either (a) freeze a known-incoherent design or (b)
  force a withdrawal of normative status later — both bad outcomes. Trial
  use + in-development publication keeps the resource available to current
  implementers while reserving room for the refactor.
- **Keeping Contract published somewhere preserves continuity.** Pulling
  the resource from R6 normative does not mean removing it from the
  ecosystem. Publishing it in an in-development companion keeps existing
  R5/R6 implementers able to find an authoritative current version while
  the refactor is in flight, and keeps tooling (validators, IG authors) able
  to resolve `http://hl7.org/fhir/StructureDefinition/Contract`.
- **Alternatives considered:**
  - *Apply the refactor inline in R6.* Rejected by the resolution: the
    refactor work has been scoped on FM workflow calls but not yet
    implemented, and R6 cannot wait.
  - *Publish Contract as Normative anyway with a "this will change" note.*
    Rejected because a Normative artifact contractually cannot change in
    breaking ways; a disclaimer does not override that constraint.
  - *Withdraw Contract entirely.* Rejected because it remains in active use
    and the FM workflow plan envisions a refactor, not a removal.

### Breaking Changes

- **Standards-status downgrade.** Anything that consumed Contract on the
  expectation that it would be Normative in R6 will see it as Trial Use
  instead. This is a *contractual* change to expectations more than a
  wire-format change — instances are unaffected — but it is significant for
  conformance work and downstream IGs that planned to depend on a
  Normative Contract in R6.
- **Publication relocation.** Tooling that resolves
  `http://hl7.org/fhir/StructureDefinition/Contract` against the R6
  normative package may need to be redirected at the new in-development
  publication once Contract is no longer published from the main R6 build.
  Whether the canonical URL changes is a release-decision that should not
  be assumed; preserving the canonical URL and just changing the publishing
  artifact is strongly preferred.
- No breaking changes are introduced at the element / cardinality / type
  level by this ticket.

1. **Mark Contract as Trial Use in R6.**
   - File: `source/fhir.ini`
   - Action: Add `Contract=trial-use` under `[standards-status]`.
   - Details: Place alphabetically with the other resource-status entries.
     Verify Contract is **not** present under `[normative-package]` and

   - File: `source/contract/structuredefinition-Contract.xml`
