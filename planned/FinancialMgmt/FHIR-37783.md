## Resolution Summary

The committee resolution (Persuasive with Modification) is:

> Add text and a reference to the Financial section to point to the CDEX, or
> other FHIR section as appropriate, to demonstrate the approach of using
> Task for the request for and an Operation endpoint for submission of
> attachments.

Concretely, this requires editing the Financial Management module page to
(a) add narrative that recommends a Task-based pattern for *requesting*
attachments and a FHIR Operation endpoint for *submitting* attachments,
(b) link out to the Da Vinci CDex / HRex implementation guides as the
worked examples of that pattern, and (c) discourage the existing
CommunicationRequest/Communication-only solicited-attachment pattern that
the page currently presents as the primary solicited approach.

The single comment on the ticket (pknapp, 2025-10-21) flags that the
approach should be re-examined "in light of the changes/usage which has
occurred since the ticket was dispositioned" — implementers should sanity
check current CDex/HRex URLs and section anchors before merging.

## Feature Proposal

### Problem Statement

The current Financial Management module page
(`source/financial-module.html`) presents CommunicationRequest /
Communication as the primary solicited and unsolicited attachment
mechanism:

- The **Glossary** (lines 146, 168, 170) defines `Attachment`,
  `Solicited Attachment`, and `Unsolicited Attachment` as
  `Communication`-typed exchanges.
- The **Common Use Cases** table (lines 207–208) does list a
  Task-wrapped pattern (`Task code=deliver, input=Communication`;
  `Task code=poll, input=CommunicationRequest`), but presents it
  side-by-side with the CommunicationRequest pattern without preference.
- The **Relative Order of Use** table (lines 321–322) shows the
  insurer-requests-info / provider-supplies-info exchange exclusively
  with CommunicationRequest and Communication.
- The **Attachments – Supporting Information** section (lines 368–383)
  enumerates four approaches; the *Solicited* approach is described
  only as "a Communication in response to a CommunicationRequest from
  the insurer". The Task / FHIR Operation approach is mentioned in a
  single bullet ("Input") with no narrative, no example, and no
  cross-reference to a worked IG.

The Da Vinci HREX IG explicitly documents the downsides of using
CommunicationRequest for the request-for-information exchange
(http://hl7.org/fhir/us/davinci-hrex/exchanging-request.html#request),
and Da Vinci CDex uses a Task-based request + FHIR Operation
submission pattern instead. The original ticket
(also see related FHIR-37228, "Provide clarity on why solicited
workflow varies from existing guidance in financial module") notes
that the base-spec page is now out of step with the consensus
implementation pattern emerging in Da Vinci.

### Proposed Change

Edit `source/financial-module.html` to introduce a clear preference
ordering for attachment exchanges and to point readers to the Da Vinci
CDex (and HREX) IGs as the canonical worked examples. The following
specific edits implement the resolution:

1. **Rewrite the bullet list inside the Attachments section** (lines
   378–383) so that the Task + Operation pattern is presented first
   and recommended, the CommunicationRequest/Communication pattern is
   retained but explicitly discouraged, and a sentence references the
   Da Vinci CDex IG as the worked example. Suggested replacement
   (subject to FM editorial polish):

   ```html
   <ul>
     <li><b>Included</b>: in the <i>{resource}</i>.supportingInfo
         section.</li>
     <li><b>Task-based (recommended)</b>: a
         <a href="task.html">Task</a> requesting the attachment(s),
         with the actual content delivered either via a FHIR Operation
         endpoint (preferred for batch submission of attachments) or
         as <a href="task.html">Task</a>.input on a follow-up Task.
         This pattern is described in detail in the Da Vinci Clinical
         Data Exchange (CDex) implementation guide and in the Da Vinci
         Health Record Exchange (HREX) IG.</li>
     <li><b>Unsolicited (legacy)</b>: a
         <a href="communication.html">Communication</a> which refers
         to the Claim or Explanation of Benefit. This pattern is
         retained for backward compatibility but the Task-based
         approach above is preferred.</li>
     <li><b>Solicited (legacy / discouraged)</b>: a
         <a href="communication.html">Communication</a> in response to
         a <a href="communicationrequest.html">CommunicationRequest</a>
         from the insurer. <b>Use of CommunicationRequest for the
         request leg is discouraged</b>; see the Da Vinci HREX IG
         (<a href="http://hl7.org/fhir/us/davinci-hrex/exchanging-request.html#request">Exchanging
         a Request</a>) for the rationale and the Task-based
         alternative.</li>
   </ul>
   ```

2. **Add an introductory paragraph** immediately after the existing
   "<i>Supporting information may be provided…</i>" paragraph (i.e.,
   between line 377 and the `<ul>` on line 378) that summarises the
   preference ordering in prose and names CDex/HREX as the
   authoritative worked examples. This paragraph also satisfies the
   resolution's "Add text and a reference to the Financial section to
   point to the CDEX" requirement.

3. **Update the Glossary rows** for `Attachment`, `Solicited
   Attachment`, and `Unsolicited Attachment` (lines 146, 168, 170)
   so the "Resource Type" cell reads `Communication or Task` (and the
   description references the preferred Task/Operation pattern).
   Cardinality of the change: cosmetic / narrative only — no
   StructureDefinition changes.

4. **Update the Relative Order of Use table** (lines 321–322) so the
   "Insurer requests additional information" row shows
   `Task` (with a parenthetical "(or, legacy: CommunicationRequest)")
   and the "Provider submits supporting information" row shows
   `Operation` (with a parenthetical "(or, legacy: Communication)").
   This makes the in-page worked example consistent with the new
   guidance in the Attachments section.

5. **Optionally re-order or annotate the Common Use Cases rows**
   (lines 207–208) to flag the Task-based send/request as the
   recommended approach. This is a small editorial polish step; the
   rows already exist.

No StructureDefinition, search-parameter, ValueSet, CodeSystem,
extension, or operation definitions are added or removed by this
ticket. The change is **documentation-only**, confined to one HTML
authoring page.

### Design Rationale

- **Aligns base-spec guidance with adopted implementation patterns.**
  Da Vinci HREX and CDex have already converged on Task + Operation
  for attachment workflows; the base spec should not advertise an
  approach that the implementation community has explicitly moved
  away from.
- **Minimum-disruption editorial change.** Existing rows mentioning
  CommunicationRequest are retained (with a "discouraged" / "legacy"
  qualifier) so existing implementations that follow the page today
  are not broken by the prose change.
- **Single-page edit, no normative artifact churn.** The
  Persuasive-with-Modification resolution explicitly scopes the
  change to "text and a reference"; promoting it to a constraint /
  invariant change would exceed the resolution and require a fresh
  ballot disposition.
- **Cross-references to Da Vinci IGs (CDex, HREX) provide the worked
  examples** rather than duplicating their full guidance in the base
  spec, consistent with how the Financial Management module already
  delegates US-Realm specifics to Da Vinci.
