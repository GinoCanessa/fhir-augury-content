## Resolution Summary

**Resolution:** Persuasive — *"Add Consent.action to one or more examples"*.

The Consent resource defines a SearchParameter named `action` whose
expression is `Consent.repeat(provision).code` (the `Consent.action`
SearchParameter pointing at `Consent.provision.code`, defined in
`source/consent/bundle-Consent-search-params.xml`). The publisher's QA
emits an INFORMATION-level message because no example currently
populates that path at the level the QA scanner inspects:

> INFORMATION: Consent.action Search Parameter 'Consent.action' had no
> found values in any example. Consider reviewing the expression
> (Consent.provision.code)

The agreed fix is to populate `Consent.provision.code` in at least one
canonical example so the SearchParameter has demonstrable matches and
the QA message clears. A `12 Dec 2025` comment from the reporter notes
"Preemptive closure"; however, a search of the `HL7/fhir` clone log
since the ticket's resolution date (`2025-06-12`) shows no commit that
modifies a `source/consent/consent-example*.xml` file in a way that
adds `Consent.provision.code` content (the only consent-example commit
in that window — `d99d31d25e` — only renamed `verificationDate` →
`date`). The change called for by the resolution therefore still needs
to be applied.

## Feature Proposal

### Problem Statement

The `Consent-action` SearchParameter (code `action`, expression
`Consent.repeat(provision).code`) is shipped as `normative` /
`active` in `source/consent/bundle-Consent-search-params.xml`. The
HL7 FHIR publisher's QA pass walks each example listed in
`source/consent/list-Consent-examples.xml` and warns when an
SP expression yields no hits across the entire example corpus —
which is the case here. The two examples that *do* contain a
`<provision>/<code>` element (`consent-example-CDA.xml`,
`consent-example-signature.xml`) place `<code>` inside a **nested**
`<provision>` block, and the QA pass still reports zero hits, so the
INFORMATION message persists in the build. Beyond clearing the QA
noise, the absence of populated examples reduces the discoverability
and testability of the `action` search for implementers.

### Proposed Change

Add a populated `Consent.provision.code` element to one or more
existing Consent examples so that the `Consent-action` SearchParameter
has at least one demonstrable match in the published example set. The
minimum-viable change is to update **`source/consent/consent-example.xml`**
(the canonical "basic" example referenced first in
`list-Consent-examples.xml`), which today has only a single
`<provision>` carrying just a `<period>`. Concretely:

- Insert a `<code>` child into the existing top-level `<provision>` of
  `consent-example.xml`. Use a value drawn from the SP's binding
  `ValueSet/consent-content-code` (per the SD: binding `example`,
  `http://hl7.org/fhir/ValueSet/consent-content-code`). A LOINC code
  consistent with the patterns already used in `consent-example-CDA.xml`
  (e.g., LOINC `34133-9` "Summary of episode note") is appropriate and
  keeps the example self-consistent. Example XML fragment:

  ```xml
  <provision>
    <period>
      <start value="1964-01-01"/>
      <end value="2019-01-01"/>
    </period>
    <code>
      <coding>
        <system value="http://loinc.org"/>
        <code value="34133-9"/>
        <display value="Summary of episode note"/>
      </coding>
    </code>
  </provision>
  ```

- Optionally, mirror the same `<code>` addition into one or two
  scenario-rich examples (e.g., `consent-example-notLabs.xml` or
  `consent-example-notThis.xml`) so that the search returns more than
  one hit and demonstrates filtering by code value.

No StructureDefinition, SearchParameter, ValueSet, or narrative pages
are altered — this is an examples-only change.

### Design Rationale

- The resolution text (*"Add Consent.action to one or more
  examples"*) explicitly scopes the work to example data, not to the
  resource definition or the SearchParameter expression. The QA
  message itself recommends "reviewing the expression
  (Consent.provision.code)" as a hint, but the disposition the work
  group accepted is to satisfy the SP rather than change it.
- Updating the canonical `consent-example.xml` (id
  `consent-example-basic`) gives the SP at least one hit at the
  top-level provision (where the QA scanner is most likely to look),
  without disturbing the nested-provision examples that already model
  more complex scenarios (CDA / signature).
- LOINC `34133-9` is already the value used in
  `consent-example-CDA.xml`, keeping example data internally
  consistent and avoiding the introduction of a new code that would
  drag in unrelated terminology review.
- A pure data-only example change is the minimum-impact way to clear
  the build INFORMATION message; it carries no normative
  implications and no compatibility risk.

## Implementation Plan

### Prerequisites

None. All artifacts the change depends on (Consent SD, Consent-action

### Step-by-Step Tasks

#### HL7/fhir

1. **Populate `Consent.provision.code` in the canonical basic example.**
   - File: `source/consent/consent-example.xml`
   - Action: Inside the existing top-level `<provision>` element
     (currently containing only `<period>...</period>`), add a sibling
     `<code>` element with a single `<coding>` to a value drawn from
     the example-bound `consent-content-code` ValueSet.
   - Details: Use LOINC `34133-9` "Summary of episode note" to stay
     consistent with `consent-example-CDA.xml`. Resulting fragment:

     ```xml
     <provision>
       <period>
         <start value="1964-01-01"/>
         <end value="2019-01-01"/>
       </period>
       <code>
         <coding>
           <system value="http://loinc.org"/>
           <code value="34133-9"/>
           <display value="Summary of episode note"/>
         </coding>
       </code>
     </provision>
     ```
     Preserve element ordering (`period` before `code`) per the SD's
     element order. No narrative `<text>` regeneration is needed for
     this example (text is hand-authored prose unrelated to provision
     contents); leave the existing `<text>` block untouched.

2. **(Optional) Provide a second populated example to give the SP
   more than one hit.**
   - File: `source/consent/consent-example-notLabs.xml`
   - Action: In one of its existing `<provision>` blocks (where the
     scenario already includes laboratory-related content), add a
     `<code>` element with a LOINC code that matches the laboratory
     scope of the example (e.g., LOINC `26436-6` "Laboratory studies
     (set)" if appropriate, otherwise reuse `34133-9` from task 1).
   - Details: Confirm the chosen code is present in (or extensible
     under) `consent-content-code` before merging; if uncertain,
     reuse `34133-9` to avoid widening the example terminology
     surface. Preserve sibling-element ordering per the SD.

3. **(Optional) Provide a third populated example for variety.**
   - File: `source/consent/consent-example-notThis.xml`
   - Action: Same pattern as task 2, scoped to the
     "withhold/withdraw consent for a specific domain" scenario the
     example narrative already describes.
   - Details: Reuse `34133-9` if no domain-appropriate code is
     readily justifiable from `consent-content-code`. Tasks 2 and 3
     are optional; satisfying the ticket only requires task 1.

4. **Re-run the publisher and confirm the QA INFORMATION message
   clears.**
   - File: *(no edit)* — invoke the bespoke publisher per the
     briefing's build-system note (`publish.sh` / `build.sh` /
     `gradlew`, whichever is in use locally for QA-only runs).
   - Action: Verify that the build no longer reports
     `INFORMATION:Consent.action Search Parameter 'Consent.action'
     had no found values in any example` and that
     `qa.html` lists at least one hit for
     `SearchParameter/Consent-action` against the example bundle.
   - Details: If the message persists despite task 1, inspect the
     QA scanner's interpretation of `Consent.repeat(provision).code`
     (the existing nested-`<code>` placements in
     `consent-example-CDA.xml` and `consent-example-signature.xml`
     should *also* be hits under that expression but currently are
     not surfacing — note the discrepancy in the open questions
     section rather than silently rewriting the SP).

### Open Questions

- **Why are the existing nested `<provision>/<code>` elements in
  `consent-example-CDA.xml` and `consent-example-signature.xml` not
  already counted by the QA scanner?** The SP expression
  `Consent.repeat(provision).code` should descend into nested
  `<provision>` elements, so those examples *should* already produce
  hits. Before declaring this fix complete, confirm whether the
  publisher's QA is evaluating `repeat()` correctly or only checking
  the top-level provision. If the QA is the limiting factor, task 1
  (top-level provision) is sufficient regardless; if there is a
  genuine FHIRPath evaluation gap, raise a separate publisher
  ticket.
- **Code-value selection for the optional examples (tasks 2 and 3).**
  The example binding to `consent-content-code` is `example`
  strength, so any reasonable code is allowed. Confirm with the work
  group whether a domain-appropriate code is preferred, or whether
  reusing `34133-9` everywhere is acceptable.

  the unchanged example files.** The reporter's late comment implies
