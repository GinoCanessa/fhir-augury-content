## Resolution Summary

The committee disposition is **Persuasive**, with resolution description: *"Will provide new examples."* The follow-up comment from David Pyke (2025-06-12) requested the reporter indicate which examples should be added, and the ticket was preemptively closed on 2025-12-11 without the reporter supplying additional specifics. The actionable requirement is therefore: **add additional Consent example instances to the FHIR Core Consent resource page** that demonstrate the R5 Consent model (which materially changed from R4) across a broader set of realistic privacy-consent scenarios.

## Feature Proposal

### Problem Statement

The Consent resource model changed substantially in R5. The reporter notes that public guidance and example instances for the new model are limited and simple, particularly compared to the rich body of R4-era examples profiled by IHE PCF. As a result, implementers adopting R5 Consent have insufficient worked examples to follow when modelling common privacy-consent use cases. The Consent resource page in FHIR Core ships an examples list (`source/consent/list-Consent-examples.xml`) plus example instance files such as `consent-example.xml`, `consent-example-Emergency.xml`, `consent-example-Out.xml`, `consent-example-notThis/notTime/notOrg/notLabs/notAuthor/notSecLabel/notThem.xml`, `consent-example-CDA.xml`, `consent-example-grantor.xml`, `consent-example-OrgToOrg.xml`, `consent-example-pkb.xml`, `consent-example-provider.xml`, `consent-example-signature.xml`, and `consent-example-smartonfhir.xml`. While the set is broad in topic, individual examples are short and do not always exercise the new R5 element shapes (e.g., `decision`, restructured `provision` with nested provisions, `controller`, `manager`, `regulatoryBasis`, `policyBasis`, `policyText`, `verification`, etc.).

### Proposed Change

Add a curated set of new Consent example instances to `source/consent/` that exercise the R5 Consent model end-to-end, and register them in `source/consent/list-Consent-examples.xml` so they are published on the Consent resource page's Examples tab. Each new example must be a complete `Consent` instance file (`consent-example-<scenario>.xml`) following the conventions of the existing examples. The recommended new scenarios — chosen to fill gaps in the current set against R5's restructured model — are:

1. **Research consent with categories and policy reference** — `Consent.decision = permit`, `category` set with a research code, `regulatoryBasis` referencing an applicable code, `policyBasis.reference` pointing to an Organization-published policy, `provision.period` covering the study window, and `provision.action` constrained to research-related codes.
2. **Treatment consent with delegated decision-maker** — populated `controller` (the patient) and `manager` (a delegated Practitioner/RelatedPerson), `decision = permit`, `provision.actor` listing the delegated decision-maker role, and `verification` with `verificationType = family` and a verifying party reference.
3. **Break-glass / emergency override** — `decision = permit` with a nested `provision` whose `purpose` includes `ETREAT` (emergency treatment) and whose `securityLabel` widens disclosure for that scenario, demonstrating the nested-provision pattern that replaces R4's `except` blocks.
4. **Privacy consent with data-class / purpose / actor exclusions combined** — single example showing how multiple `provision` entries combine (a permit base plus restrictive nested provisions for `dataPeriod`, `code`, `actor`, and `purpose`) to mirror common IHE PCF patterns under the R5 shape.
5. **Withdrawn / revoked consent** — `status = inactive`, `decision = deny`, with `verification.verificationDate` recording the revocation date; no `provision` granting access.
6. **Notice-only / no-consent-required jurisdiction** — `decision = notApplicable` (or equivalent per R5 binding), with `regulatoryBasis` indicating the jurisdiction and `policyText` supplied as an `Attachment` reference, demonstrating use of `policyText` vs. `policyBasis`.
7. **Cross-organization data-sharing agreement** — Organization-to-Organization Consent that complements `consent-example-OrgToOrg.xml` but exercises R5 `controller`/`manager` and `policyBasis.reference` to a Contract.

For each new example file, also add an `<entry>` to `list-Consent-examples.xml` with the `description` and `title` build extensions plus an `<item><reference value="Consent/<id>"/><display value="..."/></item>` block, matching the style of existing entries.

Additionally, expand the narrative on the Consent resource page (`source/consent/consent-introduction.xml` and/or `source/consent/consent-notes.xml`) with a short "Worked Examples" subsection that links each new scenario to the policy/clinical pattern it illustrates, giving readers a guided path through the new R5 model.

### Design Rationale

- **Resolution-driven.** The committee disposition is "Persuasive" with the explicit commitment to *provide new examples*. The change is intentionally additive and documentation-only — no element, cardinality, or binding changes are introduced.
- **Targets R5 shape gaps.** The current example set predates much of the R5 restructuring (e.g., `decision`, nested `provision`, `controller`/`manager`, `regulatoryBasis` vs. `policyBasis` vs. `policyText`, `verification`). The proposed scenarios were selected to exercise these new elements explicitly so implementers can pattern-match.
- **Aligns with existing conventions.** New files follow the established `consent-example-<scenario>.xml` pattern in `source/consent/`, are wired into the Consent page through `list-Consent-examples.xml` (the file the publisher consumes for the Examples tab), and avoid bypassing the bespoke HL7 publisher pipeline (no `sushi-config.yaml`, `ig.ini`, or `input/` involved here).
- **Non-normative.** Examples are illustrative; they do not change conformance. This minimises ballot risk and keeps the change inside the work-group's stated scope ("provide new examples").
- **Defers the unanswered scoping question.** The reporter never enumerated the desired examples, so the proposed set is the planner's recommendation; the work group should ratify the list before the file-by-file work begins. See Open Questions.
