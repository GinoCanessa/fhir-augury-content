## Resolution Summary

**Resolution:** Persuasive — *"Add field to one or more examples"*.

The build emitted the publisher information message:

> `INFORMATION:Consent.manager Search Parameter 'Consent.manager' had no found values in any example. Consider reviewing the expression (Consent.manager)`

The agreed fix is to populate the `Consent.manager` element in at least one of the published `Consent` examples so the search parameter `manager` (FHIRPath `Consent.manager`) evaluates to at least one hit during publication, eliminating the warning.

The reporter added a "Preemptive closure" comment on 2025-12-11, suggesting they expect the message to clear on the next publisher run; this plan therefore aims to (a) verify the message is actually gone given examples already in the clone, and, only if it persists, (b) add `manager` to additional examples.

## Feature Proposal

### Problem Statement

The HL7 publisher runs a coverage check that walks every `SearchParameter` in a resource's bundle and evaluates the `expression` against the resource's bundled examples. When zero examples produce a value for the FHIRPath, the publisher emits an `INFORMATION` message recommending the expression be reviewed. For `Consent`, the `manager` SearchParameter (`expression = Consent.manager`, defined in `source/consent/bundle-Consent-search-params.xml` lines ~187–204) was raising this message at the time the ticket was filed, even though `Consent.manager` is a valid normative element on the resource (`source/consent/structuredefinition-Consent.xml`, element `Consent.manager`, type `Reference(HealthcareService | Organization | Patient | Practitioner | RelatedPerson | PractitionerRole)`, cardinality `0..*`).

The remedy is purely an example-data fix: add a populated `<manager>` element to one or more published examples so the FHIRPath has something to bind to.

### Proposed Change

1. **Verify** that the publisher message is no longer emitted in the current clone (HEAD `db79dcd`). Two existing examples in `source/consent/` already contain a `<manager>` element with a `Practitioner/f007` reference:
   - `source/consent/consent-example-notLabs.xml` (line 27)
   - `source/consent/consent-example-notSecLabel.xml` (line 26)

   These two examples were introduced in 2021 (commits `e88d9d3` / `5e99732` "Add search examples FHIR-31568"). They satisfy the resolution literally: at least one example has the field. If the publisher still emits the warning, it is most likely because those two examples are not registered in the example list driving the SearchParameter coverage check, or because of a separate processing-mode issue.

2. **If the message persists**, add a `<manager>` element to the canonical base example `source/consent/consent-example.xml` (the `Consent/consent-example-basic` instance) so the most-prominent example demonstrates the workflow-management actor. A minimal addition that fits the existing scenario (treatment authorization at "Burgers University Medical Center") is:

   ```xml
   <manager>
     <reference value="Organization/f001"/>
     <display value="Burgers University Medical Center"/>
   </manager>
   ```

   Insert it between `<controller>` and `<sourceAttachment>` (after the existing `<controller>` block, around line 28). `Organization/f001` is already referenced as the `controller` in this example, so no new referenced resource is required.

3. **Optionally**, also add a `<manager>` element to `source/consent/consent-example-Emergency.xml` and `source/consent/consent-example-No-Emergency.xml` so that the "policy-style" examples likewise demonstrate the element. Use the same `Organization/f001` (or `Practitioner/f007`) reference already used in those files' `<controller>` blocks.

No changes are required to:
- `source/consent/structuredefinition-Consent.xml` (the element exists and is normative).
- `source/consent/bundle-Consent-search-params.xml` (the SearchParameter expression is correct).

### Design Rationale

- The resolution explicitly says "Add field to one or more examples"; it does **not** ask for a SearchParameter, type, or cardinality change. The least-invasive fix is therefore the right one — keep all StructureDefinitions / SearchParameters untouched and only edit example XML.
- Choosing the canonical `consent-example.xml` as the primary edit gives the warning-suppression maximum signal (the publisher's coverage check almost certainly walks the registered example list, and this is the headline example).
- Reusing `Organization/f001` (already cross-referenced from the same example) avoids introducing new referential dependencies and keeps the example self-consistent.
- The reporter's "preemptive closure" comment indicates they are willing to close on the assumption the publisher will no longer warn; the plan above gives a deterministic fall-back if that assumption proves wrong.

   - Action: Execute the standard local publish (`./build.sh` or the publisher harness as configured) at clone HEAD `db79dcd` and grep the QA output for the literal string `Consent.manager Search Parameter 'Consent.manager' had no found values`.

2. **Add `<manager>` to the canonical base example.**
   - File: `source/consent/consent-example.xml`
   - Action: Insert a `<manager>` block immediately after the existing `<controller>` element (around line 28), before `<sourceAttachment>`.

3. **(Optional) Add `<manager>` to the Emergency / No-Emergency examples for broader coverage.**
   - Files: `source/consent/consent-example-Emergency.xml`, `source/consent/consent-example-No-Emergency.xml`

   - File: n/a (build verification).
   - Action: Re-run `./build.sh` (or the equivalent publisher invocation) and grep the QA log a second time. Confirm no `Consent.manager Search Parameter` INFORMATION line appears.
   - Details: If the warning persists despite multiple examples now populating `<manager>`, escalate — at that point the cause is not example coverage and a separate ticket should investigate the publisher's SearchParameter coverage check (out of scope for FHIR-50869).
