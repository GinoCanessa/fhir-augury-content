## Resolution Summary

The build emits an INFORMATION-level QA message:

> `INFORMATION:Consent.purpose Search Parameter 'Consent.purpose' had no found values in any example. Consider reviewing the expression (Consent.provision.purpose)`

The committee accepted the ticket as **Persuasive** with the resolution
description: *"Add Consent.purpose to one or more examples."* The fix is
example-only — no resource, search-parameter, or terminology changes
are required. The goal is for the publisher's example-coverage check on
the `Consent-purpose` SearchParameter (expression
`Consent.repeat(provision).purpose`) to find at least one populated
`Consent.provision.purpose` value in at least one shipped example so
the informational message no longer fires.

## Feature Proposal

### Problem Statement

`Consent` defines a SearchParameter `Consent-purpose`
(`http://hl7.org/fhir/build/SearchParameter/Consent-purpose`) with
expression `Consent.repeat(provision).purpose`
(`source/consent/bundle-Consent-search-params.xml`, lines 247–263).
The publisher's QA pass evaluates the parameter's effective FHIRPath
(`Consent.provision.purpose` per the build message) against each
shipped example and finds no matching values, raising an INFORMATION
message in every build.

Inspection of `source/consent/consent-example*.xml` shows three
examples that *do* contain a `<purpose>` element
(`consent-example-Emergency.xml` line 69,
`consent-example-No-Emergency.xml` lines 43 & 47,
`consent-example-provider.xml` line 36), but in every case the
`<purpose>` lives inside a *nested* `<provision>` (a
`provision.provision.purpose`), not a top-level
`Consent.provision.purpose`. The QA check the publisher runs evaluates
`Consent.provision.purpose` (single hop) rather than the full
`repeat(provision)` path, so none of the existing examples satisfy it.
The remaining 16 example Consent instances have no `<purpose>` at all.

The lack of any top-level `Consent.provision.purpose` value in the
shipped examples is a documentation/QA gap: implementers searching for
worked examples of the `purpose` search parameter find none.

### Proposed Change

Add at least one `<purpose>` Coding to a *top-level* `<provision>` in
one or more existing Consent examples so that the FHIRPath
`Consent.provision.purpose` returns a value for at least one shipped
example. The Coding(s) MUST be drawn from the extensible binding on
`Consent.provision.purpose`
(ValueSet `http://terminology.hl7.org/ValueSet/v3-PurposeOfUse`,
binding name `PurposeOfUse`,
strength `extensible`, per
`source/consent/structuredefinition-Consent.xml` lines 681–699).

Recommended minimal change (one example is sufficient to silence the
INFO; doing two improves coverage):

1. **`source/consent/consent-example.xml`** — the canonical "main"
   example. Currently has a single top-level `<provision>` with only a
   `<period>` child (lines 44–50). Add a `<purpose>` Coding inside that
   top-level `<provision>`, e.g.:

   ```xml
   <purpose>
     <system value="http://terminology.hl7.org/CodeSystem/v3-ActReason"/>
     <code value="TREAT"/>
     <display value="treatment"/>
   </purpose>
   ```

2. **`source/consent/consent-example-Emergency.xml`** *(optional, for
   coverage)* — promote (or duplicate) the existing nested
   `<purpose>` (lines 69–72, currently inside the inner permit
   `<provision>`) so that an equivalent `<purpose>` also appears in
   the outer top-level `<provision>` (which begins at line 43). The
   nested purpose can remain — adding a top-level one does not change
   the example's intent (an overall-deny with a permit exception for
   `ETREAT`); if anything, restating the purpose at the top level
   makes the directive easier to read.

In both files the narrative `<text>/<div>` block does not need to be
re-rendered for this change to be valid; the publisher regenerates
narrative as needed. If the human-curated narrative is preferred, add
a brief sentence describing the added purpose-of-use Coding.

No changes to `structuredefinition-Consent.xml`,
`bundle-Consent-search-params.xml`, the Consent introduction page, or
any ValueSet/CodeSystem are required.

### Design Rationale

- The resolution explicitly chose the minimal, example-only fix. The
  informational build message is purely a coverage warning — it is
  not asserting that the SearchParameter expression is wrong, only
  that no example exercises it.
- `Consent.provision.purpose` has cardinality `0..*` and an
  *extensible* binding to `v3-PurposeOfUse` (a v3 vocabulary the
  Consent examples already reference for nested purposes). Re-using
  the same `http://terminology.hl7.org/CodeSystem/v3-ActReason`
  CodeSystem keeps the new example data consistent with the existing
  Emergency/No-Emergency examples (`TREAT`, `ETREAT`) and avoids
  pulling in any new terminology.
- Editing `consent-example.xml` (the canonical example) is preferred
  because it is the most-cited example and is currently empty of
  purpose data. A single top-level `<purpose>` there is sufficient to
  silence the QA INFO; doing a second example is belt-and-braces and
  costs almost nothing.
- An alternative would be to change the SearchParameter expression
  from `Consent.repeat(provision).purpose` to a form the simple
  example checker definitely evaluates (e.g., `Consent.provision.purpose`
  alone, dropping nested-provision support). This was **not** chosen:
  it would silently narrow the search semantics (nested provisions
  would no longer be searchable by purpose) and is out of scope for a
  Persuasive technical-correction ticket whose resolution is
  example-only.

## Implementation Plan

### Prerequisites

None. The required ValueSet/CodeSystem (`v3-PurposeOfUse` /
`v3-ActReason`) are already referenced from existing Consent
examples; the `Consent.provision.purpose` element is already defined
(`0..*`, extensible binding). No dependent ticket must resolve first.

### Step-by-Step Tasks

#### HL7/fhir

1. **Add a top-level `Consent.provision.purpose` to the canonical Consent example.**
   - File: `source/consent/consent-example.xml`
   - Action: Inside the single existing top-level `<provision>` block
     (currently lines 44–50, containing only `<period>`), add a
     `<purpose>` child Coding bound to `v3-PurposeOfUse`.
   - Details:
     ```xml
     <purpose>
       <system value="http://terminology.hl7.org/CodeSystem/v3-ActReason"/>
       <code value="TREAT"/>
       <display value="treatment"/>
     </purpose>
     ```
     Place it immediately before the existing `<period>` element so
     element order matches the Consent StructureDefinition
     (`purpose` is defined before `dataPeriod`/`period` in the
     differential). Leave the rest of the example unchanged. The
     publisher will regenerate the `<text>` narrative; no manual
     narrative edits are required. If a hand-curated narrative is
     preferred, add a sentence such as "The provision applies for
     treatment purposes." to the existing `<div>`.

2. **(Optional, recommended) Promote a top-level purpose into the Emergency example.**
   - File: `source/consent/consent-example-Emergency.xml`
   - Action: Inside the outer top-level `<provision>` that begins at
     line 43 (the overall-deny block), add a `<purpose>` Coding
     mirroring the existing nested permit-purpose at lines 69–72.
   - Details:
     ```xml
     <purpose>
       <system value="http://terminology.hl7.org/CodeSystem/v3-ActReason"/>
       <code value="ETREAT"/>
       <display value="Emergency Treatment"/>
     </purpose>
     ```
     Insert it after the outer `<provision>`'s leading children and
     before the nested `<provision>` element so element order matches
     the SD. Leave the existing nested permit `<provision>` and its
     own `<purpose>` (lines 56–73) untouched — the example's
     "overall deny, permit-for-ETREAT" intent is preserved.

3. **Verify the QA INFO message is gone.**
   - File: (build output)
   - Action: Run the publisher (or rely on the next CI publish) and
     confirm the line
     `INFORMATION:Consent.purpose Search Parameter 'Consent.purpose' had no found values in any example. Consider reviewing the expression (Consent.provision.purpose)`
     no longer appears. If it persists after Task 1, complete Task 2
     (or pick another example) — the publisher's example-coverage
     check requires `Consent.provision.purpose` (top-level) to
     evaluate to a non-empty collection in at least one example.
   - Details: Do not silence the message via
     `suppressed-messages.txt`; the resolution explicitly calls for
     adding example data, not suppressing the QA output.

### Testing Considerations

- **Publisher run on `source/consent/`:** Confirm both edited
  examples validate, narrative regenerates cleanly, and the
  `Consent-purpose` informational message disappears. Confirm no new
  QA messages appear (e.g., binding warnings on the new Coding).
- **FHIRPath evaluation:** Manually evaluate
  `Consent.provision.purpose` and `Consent.repeat(provision).purpose`
  against the edited examples to confirm both expressions return the
  expected Coding(s). Useful in any FHIRPath evaluator
  (`tools/`-based or external).
- **Edge cases:**
  - Example XML element ordering must match the Consent SD
    differential (`purpose` before `dataPeriod`/`period`/`data`/
    nested `provision`). If the publisher rejects on ordering,
    relocate the new `<purpose>` accordingly.
  - The extensible binding allows codes outside `v3-PurposeOfUse`
    with justification, but reusing in-binding codes (`TREAT`,
    `ETREAT`) avoids any binding-strength warning.
- **Interoperability:** Implementers harvesting Consent examples for
  test fixtures or search-parameter exemplars will now find
  populated `purpose` data. No outbound contract changes — examples
  are non-normative.
