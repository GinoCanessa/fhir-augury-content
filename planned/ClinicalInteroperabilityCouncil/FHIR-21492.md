## Resolution Summary

The resolution is **Persuasive** with the resolution description "Auto-approved" — no
explicit change instructions were captured on the ticket. The ticket body is the
authoritative description of the work: the IG contains broken table markup
(stray `</tr></tr> </tr>` sequences and table rows that overflow on the right
because they have extra `<td>` cells, e.g. the `DEMOGRAPHICS` table). The
required action is therefore mechanical: fix the malformed HTML so every table
row has the correct number of cells and no orphaned closing tags.

A near-identical sibling ticket, **FHIR-21562** ("Fix table formatting issues -
CDMH #18"), was filed the same day against the *US Common Data Model
Harmonization for Research (CDMH) (FHIR)* specification under the
*Biomedical Research & Regulation* work group, was resolved Persuasive on
2019-07-02, and reached status **Applied**. FHIR-21492 is the duplicate that
landed on Clinical Interoperability Council / FHIR Core because of mis-tagging;
the substantive fix has already been implemented in the CDMH IG via
FHIR-21562. This plan describes what the change must look like in the CDMH IG
sources so the fix can be verified (and reapplied if any of the malformed
tables crept back in during a later regeneration).

## Feature Proposal

### Problem Statement

The CDMH IG (`hl7.org/fhir/us/cdmh`) — the Common Data Model Harmonization for
Research IG produced by the BR&R work group — contains hand-authored HTML
tables that map CDMH model elements (Demographics, Diagnoses, Medications,
Procedures, etc.) to FHIR resources. Several of those tables are
syntactically broken:

- Sequences such as `</tr></tr> </tr>` appear inside the table body, which
  most browsers tolerate but which trip up validators, accessibility tooling,
  and any downstream consumer that parses the IG output as XHTML.
- The `DEMOGRAPHICS` mapping table has rows with two extra `<td>` cells on
  the right, so those rows render as wider than the table header and "stick
  out" past the right edge of the surrounding container.

The original ticket explicitly cites these two failure modes ("`</tr></tr>
</tr>`" and "Table: DEMOGRAPHICS row has two extra cells to the right") and
notes that the issue is widespread ("Many of the tables").

### Proposed Change

Apply HTML hygiene fixes to the affected CDMH IG narrative pages:

1. **Remove orphaned `</tr>` tags.** For every authored CDMH page, find any
   occurrence of `</tr>` that is not paired with a preceding `<tr>` opening
   the same row, and delete it. Concretely, search for the substring
   `</tr></tr>` and reduce each run of consecutive `</tr>` closers to a
   single `</tr>` matching the row that was actually opened.
2. **Normalise the `DEMOGRAPHICS` table column count.** Identify the rows in
   the Demographics mapping table that have two more `<td>` cells than the
   table's `<thead>` declares, and drop the trailing extras (or, if those
   cells contain meaningful content, add the matching extra `<th>` columns
   to the header so every row has a uniform width).
3. **Sweep the remaining tables.** Repeat the same row-width audit for the
   other CDMH mapping tables (Diagnoses, Encounters, Labs/Observations,
   Medications, Procedures, Vital Signs, etc.). For each table, every
   `<tr>` in the body must have the same number of `<td>` cells as the
   header has `<th>` cells.

The change is purely cosmetic / structural HTML — no model semantics, no
mapping content, and no FHIR conformance artifacts (StructureDefinitions,
ValueSets, profiles) are affected. No element definitions, cardinalities,
bindings, search parameters, or invariants change.

### Design Rationale

- The resolution was auto-approved (Persuasive) with no alternative design
  proposed, so the obvious mechanical fix to the HTML is the correct
  interpretation.
- The fix is intentionally surgical: edit the authored HTML in place rather
  than restructuring the tables, because the *content* of the tables is
  already correct — only the markup is malformed.
- Adjusting the `DEMOGRAPHICS` table by trimming extra `<td>` cells (rather
  than widening the header) is preferred when those cells are blank; if any
  of them contain real content the header should be widened instead, so no
  information is lost. The implementer must inspect each affected row to
  decide.
- Keeping FHIR-21492 and FHIR-21562 in sync (they describe the same defect)
  avoids a future reviewer being confused by a "Resolved – change required"
  CIC ticket whose change actually shipped under a BR&R ticket.

### Step-by-Step Tasks

#### HL7/cdmh (or current CDMH IG repo)

1. **Locate the Demographics mapping table.**
   - File: TBD (likely under `input/pagecontent/`, `input/pages/`, or
     `pages/` — depends on the IG template in use).
   - Action: Identify the page that renders the `DEMOGRAPHICS` mapping
     table referenced in the ticket. Confirm the table's `<thead>` column
     count and inspect each `<tr>` in the body for extra trailing `<td>`
     cells.
   - Details: The ticket states the rows have "two extra cells to the
     right". For each affected row, decide whether the trailing cells are
     blank padding (delete them) or contain real content (then widen the
     `<thead>` to match by adding two new `<th>` columns with appropriate
     headings).

2. **Repair orphaned `</tr>` sequences across the IG.**
   - File: every authored HTML / Markdown-with-embedded-HTML page in the
     CDMH IG authoring root.
   - Action: Search for the literal substring `</tr></tr>` (and any
     run of two-or-more consecutive `</tr>` closers, including those
     separated by whitespace such as `</tr></tr> </tr>`). Reduce each run
     to a single `</tr>` aligned with the matching opening `<tr>`.
   - Details: The fix is mechanical but must be reviewed page-by-page:
     deleting a `</tr>` that *should* have been a `</td>` would silently
     break a row. Use an HTML/XHTML validator on each edited page to
     confirm the table structure is well-formed after the change.

3. **Sweep the remaining mapping tables for column-count mismatches.**
   - File: every CDMH mapping page (Diagnoses, Encounters, Labs/
     Observations, Medications, Procedures, Vital Signs, plus any other
     domain tables present).
   - Action: For each table, count `<th>` cells in the header and compare
     to the `<td>` count of every body `<tr>`. Where they differ, apply
     the same trim-or-widen decision used for `DEMOGRAPHICS`.
   - Details: The ticket says "Many of the tables contain formatting
     issues", implying `DEMOGRAPHICS` is illustrative rather than unique.

4. **Rebuild the IG and verify rendering.**
   - File: N/A (build step).
   - Action: Run the CDMH IG publisher (whatever the repo's
     `_genonce.sh` / `_updatePublisher.sh` / `sushi` workflow is) and
     visually inspect each previously affected page in the generated
     `output/`.
   - Details: Confirm tables no longer overflow their container on the
     right and that no row appears truncated, mis-aligned, or missing.

5. **Reconcile FHIR-21492 against FHIR-21562.**
   - File: N/A (Jira hygiene).
   - Action: Once the CDMH IG fix is confirmed shipped, transition
