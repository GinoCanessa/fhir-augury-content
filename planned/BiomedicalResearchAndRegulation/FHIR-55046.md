## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** *"make the identifier element ..*"*

The ticket requires changing the cardinality of `Ingredient.identifier` from
`0..1` to `0..*`, so that an `Ingredient` instance can carry multiple
business identifiers (one per assigning entity / system).

Cross-references show the change has already landed on `HL7/fhir` master:

- `HL7/fhir@284925a8` — *"FHIR-55046 Ingredient.identifier made 0..* (from 0..1)"* —
  flipped `<max>` from `1` to `*` in `source/ingredient/structuredefinition-Ingredient.xml`.
- `HL7/fhir@f0c7b541` — *"resource introduction updates …"* — added the
  ballot-note block to `source/ingredient/ingredient-introduction.xml` advertising
  the change for the next ballot.

This plan documents the canonical implementation steps so the change can be
re-applied, verified, or extended; both commits are already on the clone's HEAD
(`db79dcdfe196d35bd0e74c58c655a4c1c8f534f7`).

## Feature Proposal

### Problem Statement

From the ticket description:

> Identifiers can almost always be assigned by multiple entities over time,
> but this element is not marked as repeating. (Comment 12 — imported by:
> Lloyd McKenzie)

In R5 and through the R6 ballot, `Ingredient.identifier` was declared
`0..1`, which is inconsistent with the near-universal FHIR pattern where
`Resource.identifier` is `0..*` (e.g., `Medication.identifier`,
`Substance.identifier`, `SubstanceDefinition.identifier`,
`MedicinalProductDefinition.identifier`). Because identifiers are normally
assigned independently by multiple business systems (manufacturer SKU,
regulatory identifier, internal catalogue id, IDMP identifier, …), the
`0..1` cardinality forced implementers to drop or externalise legitimate
parallel identifiers. This is a pure cardinality fix with no type,
binding, semantic, or search-parameter changes required.

### Proposed Change

Modify the `Ingredient` resource's top-level `identifier` element:

- **Element:** `Ingredient.identifier`
- **Type:** `Identifier` *(unchanged)*
- **Cardinality:** `0..1` → **`0..*`**
- **Short:** *"An identifier or code by which the ingredient can be referenced"* *(unchanged)*
- **Definition:** *"The identifier(s) of this Ingredient that are assigned by business processes and/or used to refer to it when a direct URL reference to the resource itself is not appropriate."* *(already plural; unchanged)*
- **isSummary:** `true` *(unchanged)*
- **Mappings:** `idmp=n/a`, `w5=FiveWs.class` *(unchanged)*
- **Search parameter `Ingredient-identifier`** (token, expression
  `Ingredient.identifier`) — no change needed; token search parameters
  over repeating `Identifier` elements are already the norm (cf.
  `Patient-identifier`, `Medication-identifier`).

Add a ballot note to `ingredient-introduction.xml` announcing the
cardinality change for the next R6 ballot cycle, matching the sibling
notes being added for FHIR-55053 and FHIR-55057 in the same ballot batch.

### Design Rationale

- **Consistency with the broader spec.** Almost every resource-level
  `identifier` in FHIR is `0..*`; `Ingredient` was an outlier. The fix
  aligns it with its peers in the Medication Definition module
  (`Medication`, `MedicinalProductDefinition`, `SubstanceDefinition`,
  `AdministrableProductDefinition`, `ManufacturedItemDefinition`).
- **Real-world need.** Ingredients routinely carry multiple coexisting
  identifiers: a manufacturer's internal code, a regulator-assigned id,
  an IDMP identifier, and local formulary codes. `0..1` forced a single
  choice and lost information.
- **Backward-compatible widening.** Loosening `max` from `1` to `*` is
  non-breaking for existing instances — every instance valid under
  `0..1` remains valid under `0..*`.
- **Batched ballot messaging.** The same resolution pass relaxed
  `SubstanceDefinition.moiety.identifier` (FHIR-55057) and
  `RegulatedAuthorization.case.identifier` (FHIR-55053); the ballot note
  is delivered in the same commit for all three for ballot-cycle clarity.
- **Alternative considered — leave as 0..1 and add a repeating
  extension.** Rejected as inconsistent with the rest of FHIR and
  unnecessarily indirect for a core business concept. Not supported by
  any surfaced Zulip or comment discussion.

- **Role:** Defines the `Ingredient` resource, its StructureDefinition,

- **Affected Files:**
  - `source/ingredient/structuredefinition-Ingredient.xml` — in the
    `<differential>` element for `Ingredient.identifier` (id

  - `source/ingredient/ingredient-introduction.xml` — insert the

    Usage" section noting that `Ingredient.identifier` was made `0..*`

- **Change Scope:** Minor. Two small, surgical edits in a single resource
  folder; publisher regenerates all downstream artifacts (snapshot, JSON,
  Turtle, schema, spreadsheet view, resource page) from these inputs.

### Breaking Changes

**No breaking changes.** Widening `max` from `1` to `*` is backward
compatible: every `Ingredient` instance previously valid under `0..1`
(zero or one `identifier`) remains valid under `0..*`. Consumers that
only read the first `identifier` will continue to work; they simply may
miss additional identifiers. The token search parameter
`Ingredient-identifier` already operates over `Ingredient.identifier`
and requires no change — it will naturally index all repetitions.

### Related Specifications

No other resources reference `Ingredient.identifier` directly. Sibling
medication-definition resources (`Medication`, `MedicinalProductDefinition`,
`SubstanceDefinition`, `AdministrableProductDefinition`,
`ManufacturedItemDefinition`) already declare their top-level
`identifier` as `0..*` and therefore need no adjustment. The common
`Identifier` datatype definition is unchanged.

### Related Tickets

The same resolution sweep relaxed two sibling cardinalities in the
Medication Definition module; they were shipped alongside FHIR-55046 in
commit `f0c7b541` for joint ballot messaging:

- **FHIR-55057:** *SubstanceDefinition.moiety.identifier* — also made
  `0..*` (previously `0..1`). Complements this change; no conflict.
- **FHIR-55053:** *RegulatedAuthorization.case.identifier* — also made
  `0..*` (previously `0..1`). Complements this change; no conflict.

No conflicting open tickets were surfaced by cross-references or by the
ticket's keyword set (`ingredient`, `ingredient.identifier`, `repeat`,
`identifier`).

### Terminology Impact

**No terminology impact.** The change is to cardinality only; no
CodeSystem, ValueSet, or binding is added, removed, or altered. UTG is
not affected.

## Implementation Plan

### Prerequisites

None. This is a self-contained cardinality widening within a single
resource folder. No dependent tickets, extension authoring, or
terminology authoring are required.

### Step-by-Step Tasks

#### HL7/fhir

1. **Widen `Ingredient.identifier` cardinality in the StructureDefinition differential.**
   - File: `source/ingredient/structuredefinition-Ingredient.xml`
   - Action: In the `<element id="Ingredient.identifier">` block inside
     the `<differential>` (currently around line 97), change
     `<max value="1"/>` to `<max value="*"/>`.
   - Details: Do not edit the `<snapshot>` block — it is regenerated by
     the HL7 publisher. Leave every other child of the element
     (`path`, `short`, `definition`, `min`, `type`, `isSummary`,
     `mapping`) unchanged. This exactly matches the diff in
     `HL7/fhir@284925a8`.

2. **Add the balloter-facing note to the resource introduction page.**
   - File: `source/ingredient/ingredient-introduction.xml`
   - Action: Directly under the `<h2>Scope and Usage</h2>` "module page"
     callout and before the main `<p>A material used in the preparation
     of a medicinal/pharmaceutical product…</p>` paragraph, insert a
     `<blockquote class="ballot-note" id="bn1">` containing:
     - A **Note to Balloters** preamble explaining that the resource is
       being prepared for Normative status and that comment is sought
       on substantive changes since the previous R6 ballot.
     - A nested bullet list with one item: *"Ingredient.identifier,
       which was made 0..* (previously 0..1)"*.
   - Details: Mirrors the block added in `HL7/fhir@f0c7b541`. Use the
     same `class="ballot-note"` and `id="bn1"` conventions used by the
     sibling introductions (`regulatedauthorization-introduction.xml`,
     `substancedefinition-introduction.xml`) that were updated in the
     same commit.

3. *(Optional but recommended)* **Extend an example to showcase the new cardinality.**
   - File: `source/ingredient/ingredient-example.xml` *(or a new sibling example)*
   - Action: Add a second `<identifier>` element with a different
     `system` / `value` so the published resource page and JSON/XML
     examples visibly demonstrate that multiple identifiers are
     supported.
   - Details: This is not required by the resolution and was not
     performed in the merged commits, but it is the cheapest way to
     make the change visible to readers of the spec. Skip if the
     workgroup prefers to leave the example set stable.

### Testing Considerations

- **Revalidate** `Ingredient` examples
  (`source/ingredient/ingredient-example.xml`,
  `source/ingredient/ingredient-example-strength-repeat.xml`) against
  the regenerated StructureDefinition — all existing instances must
  still be valid.
- **Search-parameter smoke test:** confirm the published
  `Ingredient-identifier` search parameter still resolves and that a
  server indexing `Ingredient.identifier` as token will index every
  repetition (no FHIRPath change required, but worth exercising).
- **Edge case — multiple identifiers in one instance:** build a test
  `Ingredient` with ≥2 `identifier` entries from different `system`s to
  confirm JSON/XML/Turtle serialisation and validation all accept the
  repeated element.
- **Edge case — zero identifiers:** keep an existing identifier-less
  example (or ensure one exists) to confirm `min=0` behaviour is
  preserved.
- **Interoperability:** consumers that previously assumed at most one
  identifier (e.g., code that reads `Ingredient.identifier[0]` or a
  scalar identifier) will need to iterate; call this out in release
  notes for the R6 ballot. The ballot note added in task 2 is the
  canonical signalling mechanism.

### Open Questions

- **Snapshot in sync?** The `<snapshot>` block in
  `structuredefinition-Ingredient.xml` is publisher-generated. Confirm
  a clean publish run has been done so the snapshot's
  `Ingredient.identifier.max` also reads `*`. Addressed by the
  "Warnings / Gotchas" from the repo briefing: only the differential is
  hand-edited.
- **Example coverage (task 3).** The workgroup should decide whether to
  ship an example with repeating identifiers for this ballot cycle. Not
  done in the merged commits.
- **Downstream IGs.** Any IG that profiled `Ingredient.identifier` to
  `0..1` (matching the previous core cardinality) would continue to
  work — constraining `0..*` back to `0..1` is legal — but such IGs may
  want to revisit whether the core widening changes their intent.
  No such IG was surfaced by cross-references in the augury cache.
