## Feature Proposal

### Problem Statement

The current `state-change-reason` CodeSystem (canonical `http://terminology.hl7.org/CodeSystem/state-change-reason`, OID `2.16.840.1.113883.4.642.4.2009`) was authored in 2020 (last edit `2020-12-28T16:55:11+11:00`, version 6.0.0 in the in-tree shadow at `cache/github/repos/HL7_UTG/clone/input/sourceOfTruth/fhir/codeSystems/CodeSystem-state-change-reason.xml`). It contains 18 concepts including `adverseEvent`, `death`, `notEligible`, `eligibleButNo`, `enrolled`, `exclusion`, `followUpComplete`, `inclusion`, `informedConsentSigned`, `intOrObsComplete`, `nonCompliance`, `preReg`, `refuseConsent`, `screenFailure`, `studyClosed`, `studyTerminated`, `subjectRefusal`, `unwillingOrUnable`, `withdrawnConsent`.

BRR reviewed these terms in the context of `ResearchSubject.subjectState.reason` and `ResearchSubject.subjectMilestone.reason` (both bound — example strength — to ValueSet `http://terminology.hl7.org/ValueSet/state-change-reason` per `source/researchsubject/structuredefinition-ResearchSubject.xml` lines 276–294 and 332–347) and reached consensus that the term displays / coverage need to be updated to better describe research-subject lifecycle reasons. The agreed display set is documented on the linked Confluence page; the resolution explicitly notes that **codes (machine-readable identifiers) were not assigned by the workgroup and must be set when the UTG ticket is processed**.

### Proposed Change

1. **Terminology change (UTG / THO)** — `http://terminology.hl7.org/CodeSystem/state-change-reason`
   - Replace / amend the concept list to reflect the BRR-agreed display set on Confluence pageId=161075413.
   - Assign new `code` values (camelCase, consistent with the existing concept ids) for each new or renamed display, retaining existing codes wherever the display is unchanged or merely re-worded but semantically identical.
   - Where existing codes are removed or merged, follow UTG retirement conventions (do not delete; mark `<property>` `status = retired` / `deprecated`, and add `replacedBy` if applicable) so prior instance data remains resolvable.
   - Bump `<version>` (currently `6.0.0`) per UTG release rules and refresh `<date>`/`<meta><lastUpdated>`.
   - Update the parallel `ValueSet/state-change-reason` (`cache/github/repos/HL7_UTG/clone/input/sourceOfTruth/fhir/valueSets/ValueSet-state-change-reason.xml`) only if its composition rule changes (today it includes the whole CodeSystem, in which case no edit is needed beyond the version bump).
2. **No structural change to ResearchSubject.** Cardinality, type (`CodeableConcept`), binding strength (`example`), and ValueSet URL remain as-is. The narrative `<short>` ("State change reason") and `<definition>` ("The reason for the state change. If coded it should follow the formal subject state model.") on `ResearchSubject.subjectState.reason` and `ResearchSubject.subjectMilestone.reason` continue to hold.
3. **HL7/fhir core spec sync** — the in-tree shadow files
   - `source/researchsubject/__codesystem-state-change-reason.xml`
   - `source/researchsubject/__valueset-state-change-reason.xml`
   are mirror copies of the THO artifacts (the `__` prefix in the FhirCore `source/<resource>/` layout marks them as terminology-imported, not hand-authored). They need to be re-synced from THO once UP-584 lands so that the FHIR Core publish picks up the updated displays/codes; the sync is mechanical, not editorial.

### Design Rationale

- The work was assigned to UTG because `state-change-reason` lives in the THO governance scope (`http://terminology.hl7.org/CodeSystem/...`), not in the FHIR Core canonical space, and BRR is not the authoring WG of THO. Routing through UTG (UP-584) keeps the THO change controlled and traceable to BRR consensus.
- Keeping binding strength at `example` (rather than tightening) honours the original ResearchSubject design: implementations frequently need site-/protocol-specific reasons, and the THO list is illustrative of common categories. The resolution does not call for tightening.
- Codes were intentionally **deferred to UTG**. UTG editors typically prefer to coin codes alongside the canonical XML (rather than have a WG ad-libbing them on Confluence), to keep `code` ⇄ `display` ⇄ `definition` triples consistent with neighbouring CodeSystems.
