## Resolution Summary

The BRR work group resolved that the `StateChangeReason` CodeSystem (`http://terminology.hl7.org/CodeSystem/state-change-reason`) should be updated in HL7 Terminology (THO) with changes listed on a Confluence page documenting BRR consensus terminology for `ResearchSubject.subjectState.reason`. A UTG ticket (UP-584) was created to carry out the update.

The resolution specifically notes: *"We will make a UTG ticket to update the code system in Terminology.hl7.org with the changes listed on the linked confluence page. NOTE: the confluence page does not list the code values but has code displays. The code values will need to be determined when the UTG ticket is created."*

A key predecessor ticket, [FHIR-26868](https://jira.hl7.org/browse/FHIR-26868), previously resolved (Persuasive with Modification) and added 12 new reason codes to the FHIR core R6 copy of the CodeSystem. The FHIR core repo (`HL7/fhir`) already contains these codes in `source/researchsubject/__codesystem-state-change-reason.xml`. However, the canonical source of truth in the UTG repository has **not** been updated — it still contains only the original 19 codes. This ticket's implementation requires propagating the BRR-consensus codes to the UTG source of truth so they are published in THO.

## Feature Proposal

### Problem Statement

The `StateChangeReason` CodeSystem in THO (v2.0.0, published in THO v7.1.0) contains only 19 original codes that are insufficient to capture the full range of reasons a research subject's state may change. The BRR work group reached consensus on an expanded set of reason terms covering clinical trial scenarios such as disease relapse, failed randomization, loss to follow-up, pregnancy, protocol deviation, and study completion. These terms are documented on a Confluence page ([pageId=161075413, section "Terminology for subjectState.reason"](https://confluence.hl7.org/pages/viewpage.action?pageId=161075413)) and have already been incorporated into the FHIR core R6 CodeSystem definition, but the canonical UTG source of truth has not been updated.

### Proposed Change

Add the following 12 new concept codes to the `StateChangeReason` CodeSystem in the UTG repository. These codes match the BRR consensus from FHIR-26868 and are already present in the FHIR core R6 copy:

| Code | Display | Definition |
|------|---------|------------|
| `diseaseRelapse` | Disease Relapse | Relapse of disease. |
| `failedRandomization` | Failed Randomization | Subject failed randomization. |
| `lackOfEfficacy` | Lack Of Efficacy | The study activities were not efficient in dealing with the underlying issue. |
| `lostToFollowUp` | Lost To Follow Up | The subject can no longer be followed up. |
| `physicianDecision` | Physician Decision | A physician has decided the subject should no longer continue on the study. |
| `pregnancy` | Pregnancy | Subject is now pregnant and no longer meets the inclusion criteria. |
| `progressiveDisease` | Progressive Disease | The subject has a disease that has progressed to far to allow them to continue in the study. |
| `protocolDeviation` | Protocol Deviation | There has been a protocol deviation that cannot be corrected or compensated for. |
| `recovery` | Recovery | The subject has recovered and no longer meets the inclusion criteria. |
| `siteTerminated` | Site Terminated | The site where the subject was being treated is no longer in the study. |
| `technicalProblems` | Technical Problems | The subject cannot continue on the study due to technical issues. |
| `complete` | Complete | The study is now complete. |

Additionally, the Confluence page may contain further refinements to existing code displays or definitions beyond what was captured in FHIR-26868. Since the Confluence page is not publicly accessible for verification, implementers should consult the page directly (pageId=161075413, section "Terminology for subjectState.reason (FINAL - Jan 09, 2024)") to confirm the exact set of changes.

### Design Rationale

- The expanded code set supports real-world clinical trial workflows where subjects leave studies for diverse clinical, administrative, and regulatory reasons.
- Aligning the UTG source of truth with the FHIR core R6 definition eliminates the current divergence between the two repositories.
- Using UTG/THO as the single source of truth for this CodeSystem (rather than the FHIR core repo) follows HL7 terminology governance — the FHIR core copy is a downstream consumer.
- The UTG ticket UP-584 is the formal vehicle for this change per HL7 terminology change management processes.

## Impact Analysis

### Affected Repositories

#### HL7/UTG (Utg)
- **Role:** Canonical source of truth for the `StateChangeReason` CodeSystem published in THO
- **Affected Files:**
  - `input/sourceOfTruth/fhir/codeSystems/CodeSystem-state-change-reason.xml` — Add 12 new `<concept>` elements
  - `input/sourceOfTruth/fhir/valueSets/ValueSet-state-change-reason.xml` — No structural changes needed (the ValueSet includes all codes from the CodeSystem via `compose.include.system`), but version bump may be appropriate
  - `input/sourceOfTruth/control-manifests/fhir-Rendering.xml` — May need entry updates if new codes require rendering metadata
  - `input/sourceOfTruth/control-manifests/allHL7-Rendering.xml` — Same as above
- **Change Scope:** Moderate

#### HL7/fhir (FhirCore)
- **Role:** Contains the R6 working copy of the CodeSystem and ValueSet, and the ResearchSubject StructureDefinition that binds to the StateChangeReason ValueSet
- **Affected Files:**
  - `source/researchsubject/__codesystem-state-change-reason.xml` — **Already contains the 12 new codes** (v6.0.0). No changes needed unless UTG review results in display/definition adjustments.
  - `source/researchsubject/__valueset-state-change-reason.xml` — No changes needed (includes all codes from CodeSystem)
  - `source/researchsubject/structuredefinition-ResearchSubject.xml` — No changes needed; `ResearchSubject.subjectState.reason` already has an `example` binding to `http://terminology.hl7.org/ValueSet/state-change-reason` (line 286-293); `ResearchSubject.subjectMilestone.reason` also binds to the same ValueSet (line 340-347)
- **Change Scope:** Minor (likely no changes needed; already aligned)

### Breaking Changes

- **No breaking changes.** This is purely additive — 12 new codes are being added to a CodeSystem with `example` binding strength. No existing codes are being removed, renamed, or redefined.
- Systems consuming the existing 19 codes will continue to function. The new codes are opt-in.
- The ValueSet binding strength is `example`, meaning implementations are not required to use these specific codes.

### Related Specifications

- **ResearchSubject (R6):** The `subjectState.reason` element (0..1, CodeableConcept) has an `example` binding to the `StateChangeReason` ValueSet
- **ResearchSubject (R6):** The `subjectMilestone.reason` element (0..*, CodeableConcept, per FHIR-50388) also uses an `example` binding to the same ValueSet
