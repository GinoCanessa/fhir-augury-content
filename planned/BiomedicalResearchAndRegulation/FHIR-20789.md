# Implementation Plan: FHIR-20789

**Title:** Use of AdverseEvent for tracking adverse events in WHT CRN
**Status:** Resolved - change required
**Resolution:** Persuasive
**Work Group:** Biomedical Research & Regulation
**Specification:** US Coordinated Registry Network (CRN) (FHIR)
**Resolved:** 2019-10-23

---

## Resolution Summary

The resolution accepts the commenter's concern that the CRN IG's CDE mapping table may be incorrectly mapping adverse events and post-operative complications to the AdverseEvent resource. The AdverseEvent resource is primarily intended for causality determination and risk management, not simply recording clinical events that occurred.

The resolution requires this ticket to be addressed alongside FHIR-21016 ("Remove AdverseEvent?"). The approach is to:

1. Determine whether each specific data element in the CDE mapping table is for **risk management** (causality assessment) or for **identification of "what occurred"** (clinical observation).
2. Map data elements accordingly — risk management elements remain with AdverseEvent; clinical observation elements should use Condition or Observation instead.
3. Contact the FDA POP Clinical Working Group to clarify data requirements and whether AdverseEvent must be retained, and if so, which data elements.
4. Examine the FDA MDR adverse event codes (<https://www.fda.gov/medical-devices/mdr-adverse-event-codes/coding-resources>) to determine coding requirements.

Per the FHIR-21016 resolution update (2019-08-16), the FDA clinical WG plans to collect complications or adverse events that occur during surgery, in the short term (0–30 days), from 30–90 days, and longer-term (>90 days) after surgery. A list of specific complications was developed during a Delphi process with the work group.

## Feature Proposal

### Problem Statement

The CRN IG's CDE (Common Data Element) mapping table maps adverse events and post-operative complications directly to the FHIR AdverseEvent resource. However, the AdverseEvent resource is designed for causality determination and risk management — not for simple tracking of clinical complications or conditions. The resource's scope includes cause-and-effect analysis, suspect entities, causality assessment methods, and entity relatedness evaluations. If the CRN's intent is simply to record "what occurred" (e.g., a surgical complication happened), then Condition or Observation may be more appropriate resources. Using the wrong resource leads to semantic inaccuracy and interoperability issues with other systems that correctly interpret AdverseEvent as a causality-assessment artifact.

### Proposed Change

Review and revise the CRN IG's CDE mapping table to correctly categorize adverse event data elements:

1. **Retain AdverseEvent** for data elements that genuinely involve causality determination, risk management, or regulatory adverse event reporting (e.g., FDA MDR reporting of device-related adverse events where suspect entity and causality assessment are required).
2. **Re-map to Condition** data elements that represent post-operative complications or clinical conditions that occurred as a result of a procedure (e.g., surgical site infection, wound dehiscence) — these are clinical findings, not causality assessments.
3. **Re-map to Observation** data elements that represent clinical observations or findings related to adverse outcomes (e.g., lab abnormalities, clinical measurements) that do not require causality analysis.
4. **Update profiles and terminology bindings** in the CRN IG to reflect the revised mapping, incorporating FDA MDR adverse event codes where AdverseEvent is retained.
5. **Coordinate with Patient Care workgroup** to ensure alignment with the AdverseEvent resource's intended scope and the Clinical Care / Clinical Research AdverseEvent Implementation Guides.

### Design Rationale

The FHIR AdverseEvent resource's scope explicitly states: "An adverse event is the result of an intervention that caused unintentional harm... characterized by the need to capture cause-and-effect, severity, and outcome." Its structure includes `suspectEntity` with `causality` sub-elements (`assessmentMethod`, `entityRelatedness`, `author`), `contributingFactor`, `preventiveAction`, and `mitigatingAction` — all geared toward risk management analysis. Using AdverseEvent for simple complication tracking would leave most of these elements unused and misrepresent the data's semantics. Condition is the appropriate resource for clinical complications (post-operative findings), while Observation covers clinical measurements and findings. This approach aligns with FHIR core guidance and the Vulcan accelerator project's recommended AdverseEvent profiles.

## Impact Analysis

### Affected Repositories

#### US Coordinated Registry Network (CRN) IG (Not in cached repositories)
- **Role:** The primary IG being modified — contains the CDE mapping table, AdverseEvent profile(s), and related conformance artifacts.
- **Affected Files:** The CRN IG repository is not available in the local cache. The IG is published under the specification "US Coordinated Registry Network (CRN) (FHIR)" but no corresponding GitHub repository was found in the cached clones or on GitHub (HL7 organization).
- **Change Scope:** Major — requires re-evaluation of the entire CDE mapping table for adverse event and complication data elements, potential removal or revision of AdverseEvent profile, and introduction of Condition/Observation profiles.

#### HL7/fhir (FhirCore) — Reference Only
- **Role:** Contains the base AdverseEvent, Condition, and Observation resource definitions. No changes required to the core spec; these serve as reference for correct resource selection.
- **Reference Files:**
  - `source/adverseevent/structuredefinition-AdverseEvent.xml` — Base AdverseEvent StructureDefinition defining scope, causality elements, and boundaries
  - `source/adverseevent/adverseevent-introduction.xml` — Scope and Usage / Boundaries and Relationships documentation
  - `source/condition/structuredefinition-Condition.xml` — Base Condition resource for clinical findings
- **Change Scope:** None (reference only)

### Breaking Changes

This is a **breaking change** to the CRN IG. Data elements currently mapped to AdverseEvent would be re-mapped to Condition or Observation, changing:
- Resource types used in CDE mappings
- Profile structures and conformance expectations
- Terminology bindings on affected profiles
- Any existing implementations that followed the previous CDE mapping table

Systems currently producing or consuming AdverseEvent instances for post-operative complications per the CRN IG would need to migrate to Condition or Observation instances.

### Related Specifications

- **AdverseEvent Resource (FHIR Core R5):** Defines the base resource scope — causality determination and risk management. See `source/adverseevent/adverseevent-introduction.xml` for Boundaries and Relationships.
- **Condition Resource (FHIR Core):** Appropriate for representing clinical findings such as post-operative complications.
- **Observation Resource (FHIR Core):** Appropriate for clinical measurements and discrete findings.
- **Clinical Care AdverseEvent Implementation Guide** (Vulcan): Provides extensions, value sets, and examples for clinical care adverse event use cases.
- **Clinical Research AdverseEvent Implementation Guide** (Vulcan): Provides guidance for clinical research adverse event tracking.
- **FDA MDR Adverse Event Codes:** External coding system relevant for device-related adverse event reporting (<https://www.fda.gov/medical-devices/mdr-adverse-event-codes/coding-resources>).

### Terminology Impact

**Potential terminology impact** depending on the outcome of the FDA POP Clinical WG consultation:

- If AdverseEvent is retained for some data elements, terminology bindings must incorporate or reference the **FDA MDR adverse event codes** for device-related adverse events.
- If data elements are re-mapped to Condition, appropriate clinical terminology bindings (e.g., SNOMED CT, ICD-10) for post-operative complications would be needed.
- If data elements are re-mapped to Observation, appropriate observation code bindings would be needed.
- The list of specific complications developed during the Delphi process (referenced in FHIR-21016 resolution) may need to be formalized as a ValueSet.

## Implementation Plan

### Prerequisites

1. **Obtain FDA POP Clinical WG input:** The resolution explicitly requires contacting the FDA POP Clinical Working Group to determine:
   - Whether AdverseEvent must be retained in the CRN IG
   - Which specific data elements need to be retained if so
   - Whether the data collection is for extraction/reporting of existing records or prospective collection
2. **Obtain the Delphi process complications list:** The file referenced in FHIR-21016 (uploaded 2019-08-16 by zachmay on behalf of Gayathri Jayawardena) containing the specific complications of interest needs to be retrieved and reviewed.
3. **Review FDA MDR adverse event codes:** Examine the coding resources at <https://www.fda.gov/medical-devices/mdr-adverse-event-codes/coding-resources> to determine applicable code systems.
4. **Locate or establish the CRN IG repository:** The CRN IG source repository needs to be identified or created for implementation work.

### Step-by-Step Tasks

#### US Coordinated Registry Network (CRN) IG

1. **Categorize CDE mapping table entries**
   - File: CDE mapping table (location TBD — CRN IG repository not in cache)
   - Action: Review every adverse event and post-operative complication entry in the CDE mapping table
   - Details: For each data element, classify as (a) risk management / causality assessment → retain AdverseEvent, (b) clinical complication / condition → re-map to Condition, or (c) clinical observation / finding → re-map to Observation. Use FDA POP Clinical WG input and the Delphi complications list as the basis for classification.

2. **Revise or remove the AdverseEvent profile**
   - File: AdverseEvent StructureDefinition in CRN IG (location TBD)
   - Action: If AdverseEvent is retained for some elements, narrow the profile to only the causality/risk-management data elements. If no elements require causality assessment, remove the AdverseEvent profile entirely from the IG.
   - Details: If retained, ensure the profile includes appropriate use of `suspectEntity`, `causality`, and FDA MDR code bindings. Align with the Clinical Care or Clinical Research AdverseEvent Implementation Guide profiles as appropriate.

3. **Create or update Condition profile for post-operative complications**
   - File: New or existing Condition StructureDefinition in CRN IG (location TBD)
   - Action: Create a CRN-specific Condition profile for post-operative complications
   - Details: Include elements for `code` (bound to the Delphi complications list and/or SNOMED CT), `onset[x]` (to capture timing: during surgery, 0–30 days, 30–90 days, >90 days), `subject`, `encounter`, and `evidence`. Cardinality and constraints should reflect the CDE requirements.

4. **Create or update Observation profile for clinical findings**
   - File: New or existing Observation StructureDefinition in CRN IG (location TBD)
   - Action: Create a CRN-specific Observation profile if discrete clinical findings (e.g., lab values) related to adverse outcomes need to be captured separately from Condition
   - Details: Only needed if specific CDE elements are better modeled as observations rather than conditions.

5. **Update CDE mapping table**
   - File: CDE mapping documentation in CRN IG (location TBD)
   - Action: Update all resource mappings to reflect the revised categorization
   - Details: Document the rationale for each mapping decision, referencing the FDA POP Clinical WG guidance and this resolution.

6. **Create or update terminology artifacts**
   - File: ValueSet/CodeSystem definitions in CRN IG (location TBD)
   - Action: Formalize the Delphi complications list as a ValueSet; add FDA MDR code references where AdverseEvent is retained
   - Details: Ensure all coded elements have appropriate terminology bindings with correct binding strength.

7. **Update examples**
   - File: Example resources in CRN IG (location TBD)
   - Action: Create or update example instances for each affected resource type (AdverseEvent if retained, Condition, Observation)
   - Details: Examples should demonstrate the correct use of each resource for the CRN use case, including timing categories (surgical, 0–30 days, 30–90 days, >90 days).

8. **Update IG narrative and documentation**
   - File: IG page content in CRN IG (location TBD)
   - Action: Update the IG's narrative documentation to explain the resource selection rationale
   - Details: Include guidance on when to use AdverseEvent vs. Condition vs. Observation for CRN data collection, referencing the FHIR core resource scope and boundaries.

### Testing Considerations

- **Profile validation:** All revised profiles (AdverseEvent if retained, Condition, Observation) must validate against the FHIR R4/R5 base StructureDefinitions using the FHIR validator.
- **Example validation:** All example instances must validate against their respective CRN profiles.
- **CDE coverage:** Verify that every adverse event and post-operative complication CDE has a corresponding mapping to at least one FHIR resource and element.
- **Terminology validation:** All ValueSets must expand correctly; all coded elements in examples must be valid against their bound ValueSets.
- **Timing categories:** Verify that the Condition profile correctly supports the FDA clinical WG's timing categories (during surgery, 0–30 days, 30–90 days, >90 days post-surgery).
- **IG publisher build:** The full IG must build successfully with the FHIR IG Publisher with no errors.
- **Regression testing:** Ensure changes do not break existing CRN profiles or examples unrelated to adverse events.

### Open Questions

1. **CRN IG repository location:** The CRN IG source repository was not found in the cached GitHub clones or on the HL7 GitHub organization. The repository needs to be identified before implementation can begin.
2. **FDA POP Clinical WG response:** The resolution explicitly requires input from the FDA POP Clinical Working Group, which may not yet have been obtained. The implementation plan depends on their determination of whether AdverseEvent needs to be retained and which data elements are required.
3. **Delphi complications list:** The file referenced in FHIR-21016 (uploaded 2019-08-16) containing the specific complications of interest needs to be located and reviewed. It was attached to the Jira ticket but is not available through the CLI.
4. **FHIR version target:** The CRN IG's target FHIR version (R4 vs. R5) affects the available AdverseEvent elements. The R5 AdverseEvent has significant structural changes from R4 (e.g., `suspectEntity.causality` sub-elements, `contributingFactor`, `preventiveAction`, `mitigatingAction`).
5. **Coordination with FHIR-21016:** Since the resolution states this ticket should be addressed alongside FHIR-21016, both tickets should be implemented together. The implementation plan above covers the combined scope.
