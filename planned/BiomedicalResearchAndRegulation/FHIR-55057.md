## Resolution Summary

**Resolution:** Persuasive.

**Resolution description (verbatim):** "Make the identifier element maximal cardinality *".

Concretely, change the maximum cardinality of `SubstanceDefinition.moiety.identifier` from `1` to `*`, i.e. make the element `0..*` where it is currently `0..1`. This is a pure StructureDefinition differential edit in the SubstanceDefinition resource, plus the customary resource-introduction changelog entry.

## Feature Proposal

### Problem Statement

The original ticket (imported comment #23 from Lloyd McKenzie) notes that identifiers can almost always be assigned by multiple entities over time, yet `SubstanceDefinition.moiety.identifier` is currently constrained to a single repetition. This is inconsistent with the general FHIR treatment of `Identifier`-typed elements, which are repeating in the vast majority of resource definitions, and it forces implementers who need to carry multiple identifiers for a moiety (e.g., from different registries/authorities) to either lose information or round-trip through extensions.

### Proposed Change

Apply the approved resolution by relaxing cardinality on the existing moiety identifier element:

- **Element path:** `SubstanceDefinition.moiety.identifier`
- **Type:** `Identifier` (unchanged)
- **Cardinality:** `0..1` → `0..*`
- **Short / definition / binding / mapping:** unchanged ("Identifier by which this moiety substance is known").

In the authoring differential (`source/substancedefinition/structuredefinition-SubstanceDefinition.xml`), the only edit is to the `<max>` value on the `SubstanceDefinition.moiety.identifier` element differential (currently line 324):

```xml
<element id="SubstanceDefinition.moiety.identifier">
  ...
  <min value="0"/>
  <max value="*"/>   <!-- was 1 -->
  <type><code value="Identifier"/></type>
  ...
</element>
```

Additionally, add a bullet to the resource-introduction changelog (`source/substancedefinition/substancedefinition-introduction.xml`) under the existing "Updates to the following elements" list:

```xml
<li>SubstanceDefinition.moiety.identifier, which was made 0..* (previously 0..1)</li>
```

Both edits are already present in the `HL7/fhir` clone as commits `0bd34d0` (structure definition) and `f0c7b54` (introduction changelog), so this plan primarily documents the change that has been applied and captures the associated validation / cross-reference work.

### Design Rationale

- FHIR consistently treats `Identifier`-typed elements as repeating so that business identifiers from multiple assigning authorities can coexist; applying the same treatment to `moiety.identifier` aligns it with the sibling `SubstanceDefinition.identifier` (already `0..*`) and with analogous changes made under FHIR-55053 (`RegulatedAuthorization.case.identifier`) and FHIR-55046 (`Ingredient.identifier`) in the same commit window.
- Widening the upper bound of cardinality is a backward-compatible change for existing instances: any `0..1` instance is trivially valid against `0..*`. No downgrade in constraints, no type change, no binding change.
- No alternatives (e.g., modelling multiple identifiers via an extension) were discussed in the ticket; the resolution is a one-line verdict and the BR&R work group accepted it as Persuasive with highest priority.

- **`SubstanceDefinition.moiety` backbone element** — no structural change needed; only the child `identifier` cardinality moves.
- **Sibling `SubstanceDefinition.identifier`** — already `0..*`; the change brings the moiety-level identifier into alignment with the resource-level identifier pattern.
- **Search parameters** — `source/substancedefinition/bundle-SubstanceDefinition-search-params.xml` defines `SubstanceDefinition-identifier` against `SubstanceDefinition.identifier` (top-level). No moiety-identifier search parameter exists, so search definitions are unaffected.
- **Examples** — `source/substancedefinition/substancedefinition-example.xml` should be spot-checked but is not required to change; any existing 0..1 usage remains valid.
- **Downstream IGs** that profile SubstanceDefinition may want to reassert `0..1` if they depend on single-cardinality behaviour; this is noted informatively, not as a required edit.
