# Implementation Plan: FHIR-21013

**Title:** Patient.communication.language or Patient.communication.preferred?
**Status:** Resolved - change required
**Resolution:** Persuasive
**Work Group:** Biomedical Research & Regulation
**Specification:** US Coordinated Registry Network (CRN) (FHIR)
**Resolved:** 2019-08-28

---

## Resolution Summary

The resolution requires that the CRN IG's harmonized data element `pat_pref_language_spoken` be mapped to **both** `Patient.communication.language` and `Patient.communication.preferred`. Currently, the CDE-to-FHIR mapping table only maps `pat_pref_language_spoken` to `Patient.communication.language`, which captures the language itself but does not indicate that this is the patient's *preferred* language. By adding a mapping to the boolean element `Patient.communication.preferred`, the IG will correctly express that the communicated language is the patient's preferred one.

## Feature Proposal

### Problem Statement

The CRN IG's harmonized Common Data Element (CDE) template has an inconsistency between naming and FHIR mapping for the preferred language concept:

1. The CDE uses the harmonized element name `pat_pref_language_spoken`, where "pref" clearly denotes **preferred** language.
2. The CDE template's "long name" references "Preferred Language (ISO 639.2)."
3. However, the FHIR mapping only points to `Patient.communication.language` — which captures a communication language but does **not** indicate whether it is the patient's preferred language.

In the FHIR Patient resource, preferred language status is expressed through the boolean element `Patient.communication.preferred` (0..1), which exists alongside `Patient.communication.language` (1..1) within the `Patient.communication` backbone element. Without mapping to `Patient.communication.preferred`, implementers using the CRN IG cannot determine whether a given communication language entry represents the patient's preferred language or merely one of several languages the patient can communicate in.

### Proposed Change

Update the CDE-to-FHIR mapping table in the CRN IG to map `pat_pref_language_spoken` to both FHIR elements:

| Harmonized CDE | Current FHIR Mapping | Updated FHIR Mapping |
|----------------|---------------------|----------------------|
| `pat_pref_language_spoken` | `Patient.communication.language` | `Patient.communication.language`, `Patient.communication.preferred` |

This means that when populating `pat_pref_language_spoken`, implementers should:
1. Set `Patient.communication.language` to the appropriate language code (using ISO 639 / BCP 47).
2. Set `Patient.communication.preferred` to `true` to indicate that this is the patient's preferred communication language.

### Design Rationale

This approach was chosen because:

1. **Semantic completeness:** The harmonized element name `pat_pref_language_spoken` explicitly references a *preferred* language. Mapping only to `Patient.communication.language` loses the "preferred" semantics. The FHIR Patient resource already provides the `Patient.communication.preferred` boolean specifically for this purpose.
2. **No structural changes required to FHIR Core:** Both `Patient.communication.language` and `Patient.communication.preferred` already exist in the base Patient resource. This change only affects the CRN IG's mapping documentation.
3. **Alignment with FHIR design intent:** The FHIR specification explicitly designed the `communication` backbone element with both a language code and a preference indicator. Using both elements together correctly represents the concept of a preferred spoken language.
4. **Consistency with US Core and USCDI:** USCDI defines "Preferred Language" as a data class element, and related discussions (see FHIR-40773) have identified similar gaps in US Core. Aligning the CRN IG mapping ensures consistency.

### Breaking Changes

None. This is a documentation/mapping update within the CRN IG. It does not alter any StructureDefinition, profile, or constraint. Existing implementations that already populate `Patient.communication.language` will continue to work. The addition of `Patient.communication.preferred` to the mapping provides supplementary guidance — it does not invalidate any prior conformant data.

### Related Specifications

- **FHIR Core Patient resource** — Defines the `Patient.communication` backbone element with sub-elements `.language` (CodeableConcept, required, bound to `all-languages` ValueSet) and `.preferred` (boolean, optional).
- **US Core Patient profile** (`us-core-patient`) — The CRN IG references US Core's Patient profile. US Core constrains `Patient.communication.language` but does not currently constrain `Patient.communication.preferred` (as noted in FHIR-40773).
- **USCDI "Preferred Language"** — The United States Core Data for Interoperability defines "Preferred Language" as a data element. The CRN IG's harmonized element `pat_pref_language_spoken` corresponds to this USCDI concept.
- **HL7/cdmh (Common Data Model Harmonization)** — The `input/includes/PCORnet_CDM_v4_FHIR.html` file in HL7/cdmh also references `pat_pref_language_spoken` in a PCORnet-to-FHIR mapping context. Changes to CRN should be reviewed for consistency with CDMH mappings.

Resolution: Persuasive with Modification (applied to US Core). Directly related — same underlying issue in a different IG.

- **FHIR-14112:** "Patient.communication should have a way to designate if the preferred language is for written vs verbal" — Proposes extending the preferred language concept to distinguish written vs. verbal preference. Future consideration for CRN if more granular language preference is needed.
- **FHIR-8501:** "Document the 'system' for Patient.communication.language" — Discusses the coding system for language codes (ISO 639). Related to the language code binding used in the mapping.
- **FHIR-26654:** "Practitioner is missing a language preference indicator" — Extends the preferred language concept to Practitioner. Not directly relevant to Patient but shows the pattern is being applied more broadly.

### Terminology Impact

No terminology changes are required. The mapping update does not introduce or modify any ValueSets or CodeSystems. The existing terminology bindings apply:

- `Patient.communication.language` is bound to the `all-languages` ValueSet (required strength), which uses BCP 47 / ISO 639 language tags. This binding is unchanged.
- `Patient.communication.preferred` is a boolean and has no terminology binding.
