## Resolution Summary

**Resolution:** Persuasive — *"Proceed as suggested, with a temporary 'Example' binding strength pointing to an example value set in the FHIR build until the target value set is available in THO."*

Concretely, BRR approved the following changes to the `ResearchStudy` resource on 2025-11-18:

1. Rename element `ResearchStudy.primaryPurposeType` → `ResearchStudy.purposeType`.
2. Change cardinality from `0..1` to `0..*` (a study may now have multiple purposes; "primary" intent is dropped).
3. Bind to a new `ResearchStudyPurposeType` ValueSet that BRR will eventually publish in THO. Until that ValueSet is in THO, bind to an **example** value set hosted in the FHIR build (binding strength `example`, *not* the originally suggested `preferred`).

A subsequent comment from JIRAUSER18251 (2026-04-06) supplied the **definitions for the new CodeSystem** (21 codes) that should populate the example value set in the FHIR build:

`treatment`, `prevention`, `diagnostic`, `screening`, `supportive-care`, `health-services-research`, `basic-science`, `device-feasibility`, `safety`, `risk-association`, `efficacy`, `pharmacokinetic`, `pharmacodynamic`, `bioequivalence`, `dose-response`, `pharmacogenetic`, `pharmacogenomic`, `pharmacoeconomic`, `incidence`, `prevalence`, `prediction`.

The existing `research-study-prim-purp-type` CodeSystem only carries the first 8 of these.

## Feature Proposal

### Problem Statement

`ResearchStudy.primaryPurposeType` is a single-valued (`0..1`) `CodeableConcept` capturing one "primary" purpose for a study. Real-world authors — including the Evidence-Based Medicine on FHIR IG, the European Medicines Agency Clinical Trial Information System (EU CTIS), and other BRR projects — routinely need to record multiple, equally-weighted study purposes (e.g., a trial that is simultaneously a *treatment* and *safety* study). The "primary" framing forces these authors to either (a) discard genuine purpose information or (b) abuse another element. In addition, the current bound code list (`research-study-prim-purp-type`, 8 codes) is missing many study-purpose categories that BRR has identified (see the 21-code list above).

### Proposed Change

In `HL7/fhir`, in `source/researchstudy/structuredefinition-ResearchStudy.xml`:

* Rename the element:
  * `id` and `path`: `ResearchStudy.primaryPurposeType` → `ResearchStudy.purposeType`.
* Change cardinality:
  * `<min value="0"/>` (unchanged) and `<max value="1"/>` → `<max value="*"/>`.
* Keep `<type><code value="CodeableConcept"/></type>` and `<isSummary value="true"/>` unchanged.
* Update the binding block:
  * `<strength>` `preferred` → `example`.
  * `bindingName` extension `valueString`: `ResearchStudyPrimaryPurposeType` → `ResearchStudyPurposeType`.
  * `<valueSet>` should point at the (renamed/expanded) example value set in the FHIR build (see below). The committed change still references `http://terminology.hl7.org/ValueSet/research-study-prim-purp-type`; per the resolution it should be a build-local example value set (e.g., `http://hl7.org/fhir/ValueSet/research-study-purpose-type`) until BRR ships the canonical value set in THO.
  * Refresh the `<description>` (e.g., "Codes for an intent of the study.") to drop the "main" wording.
* Update the BRIDG mapping `<map>` text from `StudyProtocolVersion.primaryPurposeTypeCode` if BRR has provided a more accurate mapping; otherwise leave as a documented TODO (BRIDG name unchanged is acceptable in the short term).
* Refresh the `<short>` value to enumerate the wider set of codes (or use a generic short like "Intent of the study (one or more)") — the current short still lists the original 8 codes.

In `source/researchstudy/codesystem-research-study-prim-purp-type.xml` and `source/researchstudy/valueset-research-study-prim-purp-type.xml` (or sibling new files):

* Either (a) add the missing 13 codes (`safety`, `risk-association`, `efficacy`, `pharmacokinetic`, `pharmacodynamic`, `bioequivalence`, `dose-response`, `pharmacogenetic`, `pharmacogenomic`, `pharmacoeconomic`, `incidence`, `prevalence`, `prediction`) to the existing CodeSystem (renaming it / its value set to `research-study-purpose-type`), **or** (b) introduce a new pair `codesystem-research-study-purpose-type.xml` + `valueset-research-study-purpose-type.xml` carrying all 21 codes from the 2026-04-06 comment, with `experimental="true"` and a description noting that this is a temporary example value set that will be replaced by a THO-hosted `ResearchStudyPurposeType` value set.

### Design Rationale

* **Multiple purposes, not one "primary":** BRR's 2025-11-18 minutes captured consensus that several real workflows (EBMonFHIR, EU CTIS) cannot model their studies with a single primary purpose; making the element repeatable is the minimal, additive way to support those use cases without inventing a new sub-resource.
* **`example` strength, not `preferred`:** The original ticket text proposed `preferred`. The applied resolution explicitly downgrades this to `example`, because the canonical `ResearchStudyPurposeType` value set is not yet authored in THO. `example` strength signals that the build-local value set is illustrative only and will be replaced. Re-tightening the binding (to `preferred` or `extensible`) will be a follow-up ticket once the THO value set lands.
* **Renaming the element (not just relaxing cardinality on `primaryPurposeType`):** Dropping the "primary" prefix avoids semantic drift — the data is no longer about a singular "primary" purpose, so leaving the element name unchanged would mislead implementers and break the model's self-documenting nature.
* **Expanded code set:** The 21-code list in the 2026-04-06 comment is BRR's working definition of `ResearchStudyPurposeType`. Including it in the build-local example value set lets implementers and example authors exercise the new content even before the THO value set exists, and gives BRR a concrete artifact to migrate to THO.

### Breaking Changes

Yes — this is a **breaking change** for any consumer that:

- Reads or writes `ResearchStudy.primaryPurposeType` by name (XML element `<primaryPurposeType>`, JSON key `primaryPurposeType`, or FHIRPath `ResearchStudy.primaryPurposeType`). The element is renamed to `purposeType`.
- Assumes the element is single-valued. After the change `purposeType` is `0..*`, so client code that does `study.purposeType.coding[0]` (or similar singular access) without iterating will silently miss additional purposes.
- Relies on the previously-bound value set URL (`research-study-prim-purp-type`) at preferred strength. The strength is dropped to `example` and (per the resolution) the bound URL should change to a build-local example value set; the canonical value set will eventually live in THO under a new URL.

The strength downgrade (`preferred` → `example`) is itself a backward-incompatible binding change in the conformance sense, even though it is "looser" — code generators and validators that key on the `bindingName` `ResearchStudyPrimaryPurposeType` will need to re-generate.
