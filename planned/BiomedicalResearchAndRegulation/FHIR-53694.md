## Description

Add `DocumentReference` as a type option for `ResearchStudy.result 0..* Reference`. This is needed because the prior type option of `Citation` is no longer available as the Citation Resource became an Additional Resource. The `DocumentReference` Resource will allow linking externally to a Citation.

## Resolution Description

> Add DocumentReference as a type option for ResearchStudy.result 0..* Reference.

## Feature Proposal

### What changes

Add `DocumentReference` as an additional allowed reference target type on the `ResearchStudy.result` element. Currently `result` is typed as `Reference(Composition | DiagnosticReport | Evidence)`. After this change it will become `Reference(Composition | DiagnosticReport | DocumentReference | Evidence)`.

### Why

The `Citation` resource was previously an allowed target type for `ResearchStudy.result` but was removed when `Citation` became an Additional Resource (not part of the FHIR core normative/trial-use set). `DocumentReference` fills the gap by allowing implementers to link externally to citation information.

### Precedent

- **FHIR-46095** (Applied, Persuasive) ‚Äî added `Composition` as a reference type on `ResearchStudy.result`, following the same pattern of expanding allowed reference targets.
- **FHIR-49760** (Applied, Persuasive with Modification) ‚Äî broader restructuring of ResearchStudy to replace RelatedArtifact, which introduced `DocumentReference` as a reference target in other new elements (e.g. `documentation`).

## Impact Analysis

### Breaking Changes

**None.** This is a purely additive change ‚Äî adding a new `targetProfile` to an existing Reference type. No existing valid data becomes invalid. Systems that do not produce or consume `DocumentReference` references on `ResearchStudy.result` are unaffected.

### Backward Compatibility

Fully backward compatible. The element cardinality (`0..*`), data type (`Reference`), and all existing target profiles remain unchanged. Only a new target profile is appended.

## Implementation Plan

### Repository: HL7/fhir

All changes are within `source/researchstudy/` in the HL7/fhir repository.

---

#### Change 1: Add DocumentReference targetProfile to StructureDefinition

**File:** `source/researchstudy/structuredefinition-ResearchStudy.xml`
**Lines:** 1277‚Äì1282

**Current content:**
```xml
      <type>
        <code value="Reference"/>
        <targetProfile value="http://hl7.org/fhir/StructureDefinition/Composition"/>
        <targetProfile value="http://hl7.org/fhir/StructureDefinition/DiagnosticReport"/>
        <targetProfile value="http://hl7.org/fhir/StructureDefinition/Evidence"/>
      </type>
```

**Updated content:**
```xml
      <type>
        <code value="Reference"/>
        <targetProfile value="http://hl7.org/fhir/StructureDefinition/Composition"/>
        <targetProfile value="http://hl7.org/fhir/StructureDefinition/DiagnosticReport"/>
        <targetProfile value="http://hl7.org/fhir/StructureDefinition/DocumentReference"/>
        <targetProfile value="http://hl7.org/fhir/StructureDefinition/Evidence"/>
      </type>
```

**Rationale:** Insert the new `DocumentReference` targetProfile in alphabetical order among the existing target profiles, between `DiagnosticReport` and `Evidence`. This follows the existing ordering convention in the file.

#### Change 2: Update the SVG resource diagram

**File:** `source/researchstudy/researchstudy.svg`
**Lines:** ~441‚Äì446

**Current content (relevant fragment):**
```xml
   xlink:href="researchstudy-definitions.html#ResearchStudy.result">...result</a> : <a

   xlink:href="references.html#Reference">Reference</a> [0..*] ¬´ <a

```

**Action:** Add a `DocumentReference` link between `DiagnosticReport` and `Evidence` in the SVG text element, following the established pattern:
```xml
   xlink:href="diagnosticreport.html#DiagnosticReport">DiagnosticReport</a>|<a
   id="n13XX"
   xlink:href="documentreference.html#DocumentReference">DocumentReference</a>|<a
```

**Note:** The SVG may need visual width adjustments to accommodate the additional type text. The next available `id` attribute value should be used (incrementing from existing `n1345`). The SVG is auto-generated by the FHIR build tooling in many cases; if so, rebuilding the spec will regenerate it. Manual editing is a fallback.
