# Implementation Plan: FHIR-21016

**Title:** Remove AdverseEvent?
**Status:** Resolved - change required
**Resolution:** Persuasive
**Work Group:** Biomedical Research & Regulation
**Specification:** US Coordinated Registry Network (CRN) (FHIR)
**Resolved:** 2019-10-23

---

## Resolution Summary

The ballot comment questioned whether the AdverseEvent resource should be removed from the CRN IG, given that AdverseEvent was "not specific enough currently." The resolution (Persuasive) accepted the concern but did **not** call for outright removal. Instead, the work group agreed to:

1. Follow up with the FDA POP (Post-market Orthopedic Procedure) Clinical Working Group to determine specific data requirements for adverse event capture.
2. Examine the FDA MDR (Medical Device Reporting) adverse event codes at [FDA Coding Resources](https://www.fda.gov/medical-devices/mdr-adverse-event-codes/coding-resources) to determine coding requirements.
3. Clarify whether AdverseEvent should be retained and, if so, which data elements are necessary.

A subsequent update (August 2019) confirmed that the FDA clinical WG plans to collect complications/adverse events that occur:
- During surgery
- Short-term post-surgery (0–30 days)
- Medium-term post-surgery (30–90 days)
- Long-term post-surgery (>90 days)

A specific list of complications was developed via a Delphi process with the FDA clinical work group (referenced as an attached file in the Jira ticket).

The companion ticket [FHIR-20789](https://jira.hl7.org/browse/FHIR-20789) further clarifies that the resolution must determine whether the CRN's adverse event data elements relate to **risk management/causality** (→ AdverseEvent) or **identification of what occurred** (→ Condition/Observation), and map accordingly.

## Feature Proposal

### Problem Statement

The CRN IG's CDE (Common Data Element) mapping table maps adverse events and post-operative complications to the FHIR AdverseEvent resource. However, the AdverseEvent resource is fundamentally about **causality determination and risk management** (per the FHIR resource scope), not simply about recording that a complication occurred. The CRN IG did not specify which data elements were needed for adverse event capture, making the profile insufficiently constrained for implementers.

The FDA clinical WG has specific, surgery-focused complication tracking needs with defined time windows, which may or may not align with the AdverseEvent resource's scope. Some complications may be better represented as Condition or Observation resources.

### Proposed Change

1. **Retain AdverseEvent** in the CRN IG but constrain it to FDA-identified causality/risk-management scenarios (device-related adverse events requiring MDR reporting).
2. **Refine the AdverseEvent profile** with specific data elements, constrained value sets based on FDA MDR adverse event codes, and time-windowed occurrence tracking (surgery, 0–30 days, 30–90 days, >90 days).
3. **Add Condition/Observation profiles** (or update CDE mappings) for complications that are purely "what occurred" findings, not causality determinations.
4. **Update the CDE mapping table** to clarify which adverse event CDEs map to AdverseEvent vs. Condition/Observation.
5. **Incorporate the FDA Delphi-derived complications list** as a ValueSet or coded reference within the IG.

### Design Rationale

- The FHIR AdverseEvent resource scope is explicitly about causality determination (per the resource introduction: "AdverseEvent is about the causality determination. It's a risk management exercise."). Device-related adverse events that require FDA MDR reporting fit this scope.
- Post-surgical complications that are simply tracked as clinical findings (fracture, infection, etc.) are better modeled as Condition resources with onset timing relative to the procedure.
- This dual-resource approach aligns with the guidance from FHIR-20789 and the broader FHIR community understanding of resource boundaries.

## Impact Analysis

### Affected Repositories

#### CRN IG Repository (Not Cached)

The US Coordinated Registry Network (CRN) IG does not have a public GitHub repository in the known cached clone set, and no public HL7 GitHub repository was found for it. The CRN IG may be maintained in a private or pre-publication repository, or may be published only via the HL7 balloting infrastructure.

- **Role:** Primary target — contains the AdverseEvent profile, CDE mapping tables, and conformance artifacts for the CRN use case.
- **Affected Files:** CRN AdverseEvent StructureDefinition profile, CDE-to-FHIR mapping pages, example resources, ValueSet definitions.
- **Change Scope:** Major — requires profile refinement, new value sets, mapping table updates, and potentially new Condition/Observation profiles.

#### HL7/cdmh (Common Data Model Harmonization) (Not Cached — Related)

The CDMH IG at [HL7/cdmh](https://github.com/HL7/cdmh) contains a related CDMH AdverseEvent profile (`cdmh-adverseevent`) with extensions for PCORnet death-related data elements and an AdverseEvent example.

- **Role:** Related downstream consumer — uses an AdverseEvent profile aligned with the same FDA data model. Changes to the CRN IG's approach to AdverseEvent may need to be coordinated with CDMH.
- **Affected Files:**
  - `input/examples/AdverseEvent-example.json` — CDMH AdverseEvent example using R4 `event`, `date`, `severity` elements
  - `input/pagecontent/guidance.md` — mapping guidance referencing AdverseEvent
- **Change Scope:** Minor — review for consistency with CRN decisions; may need updates if the CRN removes or redefines its AdverseEvent profile.

#### HL7/fhir (FHIR Core) (FhirCore — Cached)

### Breaking Changes

- **For CRN IG implementers:** If the AdverseEvent profile is significantly refined (constrained elements, required value sets, new extensions for time-window categorization), existing CRN implementations that previously loosely used AdverseEvent will need to update their mappings and data capture workflows.
- **CDE mapping table changes:** Shifting some CDEs from AdverseEvent to Condition/Observation would be a breaking change in the mapping semantics, requiring implementers to re-map data flows.
- **CDMH alignment:** The CDMH IG's AdverseEvent example uses R4 elements (`event`, `date`, `severity`, `suspectEntity.instance`, `suspectEntity.causality`) — several of which were renamed/restructured in R5/R6. If CRN updates to align with the current FHIR version, CDMH would need to coordinate.

### Terminology Impact

- **New ValueSet needed:** A CRN-specific ValueSet incorporating the FDA MDR adverse event codes from [FDA Coding Resources](https://www.fda.gov/medical-devices/mdr-adverse-event-codes/coding-resources) to bind to `AdverseEvent.code`.
- **Complications ValueSet:** The Delphi-derived surgical complications list (attached to the Jira ticket) should be encoded as a ValueSet for use in either AdverseEvent or Condition profiles.
- **Existing UTG code systems:** The existing `adverse-event-category`, `adverse-event-severity`, and `adverse-event-seriousness` code systems in UTG are sufficient for CRN use and do not need modification.
- **Time-window categorization:** If the CRN IG adds a time-window classification (during surgery, 0–30 days, 30–90 days, >90 days), this may require a new CodeSystem/ValueSet for complication timing categories.

## Implementation Plan

### Prerequisites

1. **Obtain the FDA Delphi-derived complications list** — The attached file referenced in the Jira ticket update (August 2019) contains the specific complications of interest. This list is needed to create value sets.
2. **Confirm the CRN IG source repository location** — The CRN IG repository was not found in cached clones or on public HL7 GitHub. The editor or work group must identify where the IG source is maintained.
3. **Coordinate with Patient Care WG** — Per FHIR-20789, the Patient Care work group should be consulted on AdverseEvent vs. Condition/Observation boundaries.
4. **Determine target FHIR version** — The CRN IG was originally based on FHIR R4. Confirm whether updates target R4 or R5/R6 (significant structural changes to AdverseEvent between versions).

### Step-by-Step Tasks

#### Task 1: Classify CRN CDEs as Risk Management vs. Occurrence

- Review each CDE currently mapped to AdverseEvent in the CRN CDE mapping table.
- For each CDE, determine whether it captures:
  - **Causality/risk management data** → keep mapped to AdverseEvent
  - **Clinical finding/occurrence data** → re-map to Condition or Observation
- Document the classification rationale for each CDE.

#### Task 2: Create FDA MDR Adverse Event Codes ValueSet

- Obtain the FDA MDR adverse event coding resources.
- Create a CRN-specific ValueSet (e.g., `crn-adverse-event-code`) that includes relevant FDA MDR codes.
- Bind this to `AdverseEvent.code` (R5+) or `AdverseEvent.event` (R4) in the CRN profile with an extensible or preferred binding strength.

#### Task 3: Create Surgical Complications ValueSet

- Encode the Delphi-derived complications list as a FHIR ValueSet.
- Include codes from SNOMED CT, ICD-10, or FDA MDR as appropriate.
- Categorize complications by time window: during surgery, 0–30 days, 30–90 days, >90 days.

#### Task 4: Refine the CRN AdverseEvent Profile

- Update the CRN AdverseEvent StructureDefinition to:
  - Constrain `AdverseEvent.code` (or `event` in R4) with the FDA MDR ValueSet binding.
  - Require `AdverseEvent.actuality` (set to `actual`).
  - Constrain `AdverseEvent.category` to relevant categories (e.g., `device`, `procedure-mishap`).
  - Require `AdverseEvent.seriousness` with standard binding.
  - Require `AdverseEvent.suspectEntity` to link the adverse event to the suspected device or procedure.
  - Add an extension or use `AdverseEvent.detected` / `AdverseEvent.effect[x]` to capture the time window relative to surgery.
  - Require `AdverseEvent.study` if used within a research context.
- Add must-support flags on the constrained elements.

#### Task 5: Create CRN Condition Profile for Complications (if needed)

- If Task 1 identifies CDEs that are purely occurrence-based:
  - Create a CRN Condition profile for surgical complications.
  - Bind `Condition.code` to the surgical complications ValueSet.
  - Use `Condition.onset[x]` to capture timing relative to surgery.
  - Reference the Procedure resource via `Condition.evidence` or extensions.

#### Task 6: Update CDE Mapping Table

- Update the CRN IG's CDE-to-FHIR mapping page to reflect:
  - Which CDEs map to AdverseEvent (causality/risk management).
  - Which CDEs map to Condition or Observation (occurrence/finding).
  - Rationale for each mapping decision.

#### Task 7: Update Examples

- Update the CRN AdverseEvent example to demonstrate:
  - FDA MDR coding for the event type.
  - Seriousness classification.
  - Suspect entity linkage to a device or procedure.
  - Time window categorization.
- Add Condition examples for non-causality complications if new profiles are created.

#### Task 8: Update IG Narrative

- Update the CRN IG's narrative pages to explain:
  - Why AdverseEvent is retained.
  - The distinction between AdverseEvent (causality) and Condition (occurrence).
  - FDA MDR coding requirements.
  - The complications time-window classification approach.

### Testing Considerations

- **Profile validation:** Run the HL7 FHIR Validator against updated AdverseEvent and (if created) Condition profiles to ensure no structural errors.
- **Example validation:** Ensure all updated examples conform to their respective profiles.
- **Mapping completeness:** Verify that every CDE previously mapped to AdverseEvent is accounted for — either retained in AdverseEvent or re-mapped to Condition/Observation.
- **Terminology validation:** Confirm that FDA MDR codes resolve correctly and the ValueSet expansion is valid.
- **CDMH alignment:** Review the CDMH IG's AdverseEvent profile and example against the CRN changes to identify any inconsistencies.

### Open Questions

1. **Where is the CRN IG source repository?** The IG was not found on public HL7 GitHub. The work group must identify the repository for implementation.
2. **Target FHIR version?** The CRN IG was originally R4-based. FHIR R5/R6 significantly restructured AdverseEvent (e.g., `event` → `code`, `date` → `detected`, removal of `severity` in favor of `seriousness`, addition of `effect[x]`). The target version determines the profile structure.
3. **FDA Delphi complications list availability?** The attached file referenced in the August 2019 update is needed but may not be publicly accessible via Jira.
4. **Is the CRN IG still actively maintained?** The last Jira activity was January 2020. If the IG is inactive, this ticket may be deferred or closed without implementation.
5. **Coordination with FHIR-20789:** The companion ticket's resolution explicitly states it will be addressed "along with Tracker Item 21016." Both tickets should be implemented together to ensure consistency in the AdverseEvent vs. Condition/Observation determination.
6. **FDA MDR code system OID/URL:** What canonical URL should be used for the FDA MDR adverse event code system in the ValueSet? Is there an existing NamingSystem in UTG for FDA MDR codes?
