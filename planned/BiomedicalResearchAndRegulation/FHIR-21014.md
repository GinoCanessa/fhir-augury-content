**Title:** Include Common Data Elements in the IG
**Status:** Resolved - change required
**Resolution:** Persuasive
**Work Group:** Biomedical Research & Regulation
**Specification:** US Coordinated Registry Network (CRN) (FHIR)
**Resolved:** 2019-08-28

## Resolution Summary

The ballot commenter (Robin Taylor) requested that the full set of Common Data Elements (CDEs) be included directly in the IG content, rather than only being available via an external Google Drive spreadsheet link referenced from the Overview page. The resolution was marked **Persuasive**, meaning the request was accepted as-is.

The resolution description states:
> Add the complete data element spreadsheet to the Overview and the Mappings and Profiles pages. The link is present below for reference.

This means the external CDE spreadsheet content must be embedded as inline content within the IG itself, on both the Overview page and the Mappings and Profiles page.

## Feature Proposal

### Problem Statement

The WHT CRN IG defines a set of consensus-approved Common Data Elements (CDEs) that registries must collect. However, the full CDE list is only available as an external Google Drive spreadsheet link (`https://drive.google.com/file/d/1GGRZOJcLBAW_p_czcfaSe7xAirs8KPw6/view`), which is referenced from the Overview page. This is problematic because:

1. **Accessibility:** External links may become stale or inaccessible (the Google Drive link requires permissions).
2. **Self-containment:** An IG should be self-contained — readers should not need to leave the IG to find the core data elements it defines.
3. **Reviewability:** Ballot reviewers cannot effectively review or comment on content that lives outside the IG.

The Mappings and Profiles page (`profiles.md`) already contains a partial CDE-to-FHIR mapping table via the `CDEs_FHIR.html` include, but this is a mapping table (CDE → FHIR element → profile) and not the full CDE definition spreadsheet itself (which includes element definitions, descriptions, permitted values, etc.).

### Proposed Change

Embed the complete CDE spreadsheet as an HTML table directly within the IG, making it available on:

1. **The Overview page** (`source/pages/crn-overview.md`) — in the "WHT CRN Project Data Elements" section, replacing or supplementing the external Google Drive link with the full inline table.
2. **The Mappings and Profiles page** (`source/pages/profiles.md`) — alongside or above the existing CDE-to-FHIR mapping table, so readers can see both the full CDE definitions and how they map to FHIR.

The implementation should:
- Create a new HTML include file (e.g., `CDEs_Full.html`) containing the complete CDE spreadsheet content rendered as an HTML table.
- Include this file in both the Overview and Profiles pages using Jekyll `{% include %}` directives.
- Retain the external Google Drive link as a supplementary reference, but the IG content itself should be the authoritative representation.

### Design Rationale

- **HTML include file pattern:** The IG already follows this pattern — `CDEs_FHIR.html` is an HTML table include used in `profiles.md`. Using the same approach for the full CDE table maintains consistency.
- **Dual-page inclusion:** The resolution explicitly names both the Overview and the Mappings and Profiles pages. Including the same file in both pages avoids content duplication and ensures consistency.
- **Preserving the external link:** The Google Drive link should be kept as a reference for users who want the original spreadsheet format, but the IG itself should contain the complete data element definitions.

## Implementation Plan

### Prerequisites

1. **Obtain the CDE spreadsheet content.** The authoritative CDE list is in the Google Drive spreadsheet linked from the Overview. The implementer must access and extract the full content from this spreadsheet (or the NLM's CDE Repository at `https://cde.nlm.nih.gov/cde/search`) and convert it to HTML table format.
2. **Coordinate with FHIR-20967 and FHIR-20966.** These tickets modify the same "NLM's CDE Repository" section of the Overview page. Changes should be applied together to avoid merge conflicts.
3. **Coordinate with FHIR-21210.** That ticket asks for CDE definitions/descriptions in the mapping table and links to FHIR elements. If both are implemented together, the `CDEs_FHIR.html` table can be enhanced at the same time.

### Step-by-Step Tasks

#### HL7/coordinated-registry-network

1. **Create the full CDE HTML include file**
   - File: `source/pages/_includes/CDEs_Full.html`
   - Action: Create a new HTML file containing the complete CDE spreadsheet rendered as a styled HTML table. Follow the same CSS class pattern used in `source/pages/_includes/CDEs_FHIR.html` (`.ritz .waffle` CSS classes).
   - Details: The table should include all columns from the CDE spreadsheet: Data Element Name, Definition/Description, Data Type, Permitted Values/Value Set, Source (e.g., which FDA Clinical Working Group), and any other columns present in the original spreadsheet. Each row represents one Common Data Element.

2. **Embed the full CDE table in the Overview page**
   - File: `source/pages/crn-overview.md`
   - Action: In the "WHT CRN Project Data Elements" section (after the paragraph that currently says *"The data element list is called as the Common Data Elements (CDEs) for the CRN project..."*), add a Jekyll include directive: `{% include CDEs_Full.html %}`
   - Details: Keep the existing external Google Drive link as a supplementary reference (e.g., "The original spreadsheet is also available at [Common Data Elements](https://drive.google.com/...)"). The inline table becomes the primary representation. Add a brief introductory sentence such as: "The following table contains the complete set of WHT CRN Common Data Elements:"

3. **Embed the full CDE table in the Mappings and Profiles page**
   - File: `source/pages/profiles.md`
   - Action: Add the same Jekyll include directive `{% include CDEs_Full.html %}` above the existing "Mappings of CDEs to FHIR" section. Add an introductory heading such as `#### Common Data Elements` and a brief explanation.
   - Details: This allows readers to see the full CDE definitions on the same page as the FHIR mapping table, giving them context for understanding how each data element maps to FHIR resources and profiles.

4. **Verify the existing CDEs_FHIR.html mapping table**
   - File: `source/pages/_includes/CDEs_FHIR.html`
   - Action: Verify that the list of CDEs in the new full table (`CDEs_Full.html`) is consistent with the CDEs listed in the existing mapping table. Any CDEs present in the full spreadsheet but missing from the mapping table should be flagged for follow-up (potentially under FHIR-21210).
   - Details: The current `CDEs_FHIR.html` contains 31 data elements mapped to FHIR. The full spreadsheet may contain additional elements or more detailed definitions.

5. **Update the table of contents / navigation if needed**
   - File: `source/pages/toc.md`
   - Action: Verify that the table of contents properly reflects any new sections added to the Overview or Profiles pages. No changes expected unless new page-level sections are added.

### Testing Considerations

- **IG Publisher Build:** Run the IG Publisher to ensure the new HTML include file renders correctly and does not break the build. The IG uses the `jekyll` tool (as specified in `ig.json`).
- **Visual Inspection:** The full CDE table should be readable, properly styled, and consistent with the rest of the IG's visual design. Use the same `.ritz .waffle` CSS classes as `CDEs_FHIR.html`.
- **Content Accuracy:** The data in the embedded table must exactly match the consensus-approved CDE spreadsheet. Any discrepancies must be resolved with the work group.
- **Page Load Performance:** The full CDE spreadsheet may be large. Verify that embedding it as HTML doesn't cause excessive page load times. If the table is very large, consider whether a collapsible section or separate page is warranted.

### Open Questions

1. **CDE Spreadsheet Access:** The Google Drive link (`https://drive.google.com/file/d/1GGRZOJcLBAW_p_czcfaSe7xAirs8KPw6/view`) requires access permissions. The implementer will need access to this spreadsheet or must obtain the CDE content from the NLM CDE Repository at `https://cde.nlm.nih.gov/cde/search` to convert it to HTML.

2. **Spreadsheet Columns:** The exact columns in the CDE spreadsheet are not known from the IG content alone. The implementer should confirm which columns to include (e.g., Element Name, Definition, Data Type, Permitted Values, Notes, Source Working Group).

3. **Coordination with FHIR-21210:** That ticket asks for definitions/descriptions to be added to the CDE-to-FHIR mapping table and for links to FHIR element definitions. Should the `CDEs_Full.html` table and the enhanced `CDEs_FHIR.html` mapping table be combined into a single comprehensive table, or kept as two separate tables?

4. **Coordination with FHIR-20967 and FHIR-20966:** These tickets modify the "NLM's CDE Repository" section wording on the same Overview page. All three changes should be applied in the same commit or PR to avoid conflicts.

5. **IG Publisher Version:** The IG currently uses an older Jekyll-based IG Publisher toolchain. If the IG is being migrated to the newer FHIR IG Publisher (sushi/FSH), the implementation approach may differ. Confirm which toolchain version is current.
