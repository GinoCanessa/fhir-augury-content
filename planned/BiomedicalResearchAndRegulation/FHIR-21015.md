# Implementation Plan: FHIR-21015

**Title:** Include LOINC and other codes when specified
**Status:** Resolved - change required
**Resolution:** Persuasive
**Work Group:** Biomedical Research & Regulation
**Specification:** US Coordinated Registry Network (CRN) (FHIR)
**Resolved:** 2019-09-17

---

## Resolution Summary

The resolution requires the following changes to the WHT CRN Implementation Guide:

1. **Update the Terminology section** to link to the VSAC FHIR API at `https://cts.nlm.nih.gov/fhir/` for value sets.
2. **Link to value sets using FHIR API URLs** instead of or in addition to the existing browser-based VSAC/LOINC links.
3. **Add UMLS licensing guidance** — mention the specific licensing requirements to access value sets via a UMLS license.
4. **Verify value set accessibility** — confirm that all value sets referenced in the CDE-to-FHIR mapping spreadsheet are accessible via the VSAC FHIR API.
5. **Include non-accessible value sets directly in the IG** if they cannot be retrieved through the API.

The original request (from the ballot commenter) was broader — asking that LOINC and other question codes from the template spreadsheet be included wherever specified. The resolution narrowed this to a terminology section update focusing on VSAC FHIR API integration and value set accessibility.

## Feature Proposal

### Problem Statement

The WHT CRN IG's terminology section currently lists value sets with links to the VSAC browser UI (`https://vsac.nlm.nih.gov/valueset/...`) and LOINC details pages (`https://r.details.loinc.org/LOINC/...`). These links are useful for human browsing but do not provide machine-readable FHIR API access to the value sets. Additionally:

- There is no mention of the VSAC FHIR API endpoint for programmatic access.
- There is no guidance on the UMLS licensing requirements needed to use the VSAC API.
- It is unclear whether all value sets referenced by the CDE mapping are discoverable via standard FHIR terminology operations.
- LOINC codes that have been specified as Question Codes in the CRN template are not consistently surfaced in the IG.

### Proposed Change

1. **Rewrite the Terminology section** (`source/pages/terminology.md`) to:
   - Add a subsection introducing the VSAC FHIR API endpoint (`https://cts.nlm.nih.gov/fhir/`).
   - Convert existing value set references to use FHIR API canonical URLs where available (e.g., `https://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1180.14`).
   - Add a licensing notice explaining that access to VSAC value sets and LOINC codes requires a UMLS license, with a link to the UMLS license application page.
   - List all value sets from the CDE mapping spreadsheet, organized by domain area (demographics, procedures, conditions, observations, devices).

2. **Verify value set availability** via the VSAC FHIR API:
   - For each value set OID currently referenced, verify that `GET https://cts.nlm.nih.gov/fhir/ValueSet/{OID}` returns a valid FHIR ValueSet resource.
   - For value sets that are not accessible via the API, define them as inline ValueSet resources within the IG under `source/resources/`.

3. **Add LOINC question codes** to the CDE-to-FHIR mapping table (`source/pages/_includes/CDEs_FHIR.html`):
   - Where LOINC codes have been identified as Question Codes in the CRN template spreadsheet, add a column or annotation showing the LOINC code alongside each CDE mapping entry.

### Design Rationale

- Using the VSAC FHIR API endpoint ensures value sets can be retrieved programmatically by implementations, which is essential for terminology validation at runtime.
- UMLS licensing requirements are a common stumbling block for implementers; explicit documentation prevents confusion.
- Including non-accessible value sets directly in the IG ensures all terminology is self-contained when external APIs are unavailable.
- Adding LOINC question codes to the mapping table makes the IG more complete and aligns it with the original CDE template that drives the project.

## Implementation Plan

### Prerequisites

1. Obtain or verify access to a UMLS account to test VSAC FHIR API access.
2. Obtain the original CDE template spreadsheet referenced in the ticket (Google Drive link: `https://drive.google.com/file/d/1GGRZOJcLBAW_p_czcfaSe7xAirs8KPw6/view`) to identify all LOINC question codes.
3. Clone the `HL7/coordinated-registry-network` repository (not currently in the local cache).

### Step-by-Step Tasks

#### Task 1: Verify VSAC FHIR API Accessibility for All Referenced Value Sets

1. Using a UMLS API key, test each value set OID against the VSAC FHIR API:
   ```
   GET https://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1180.14
   GET https://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1180.19
   ```
2. For each LOINC code referenced as a question code, verify the answer value set is available:
   ```
   GET https://cts.nlm.nih.gov/fhir/ValueSet?code=66174-4
   ```
3. Document which value sets are accessible and which are not.
4. For any value sets not available via the VSAC FHIR API, prepare to include them directly in the IG.

#### Task 2: Identify LOINC Question Codes from the CDE Template

1. Review the CDE template spreadsheet to extract all LOINC question codes.
2. Map each LOINC code to the corresponding CDE in `CDEs_FHIR.html`.
3. Known LOINC codes from existing IG references:
   - `66174-4` — "Are you currently pregnant?" question
   - `63893-2` — "Outcome of Pregnancy" question
   - `66678-4` — "Diabetes - PhenX" question
4. Review the template for additional codes for: Height, Weight, Pregnancy Status, Pregnancy History, Pain During Sexual Activity, Pelvic Pain, Gestational Age, Depression, Irregular Menstruation, Endometriosis, etc.

#### Task 3: Rewrite `source/pages/terminology.md`

Replace the current content with an expanded section that includes:

1. **Overview paragraph** explaining that the CRN IG uses value sets from VSAC, LOINC, FHIR core, US Core, SDC, and PRO.

2. **VSAC FHIR API subsection:**
   ```markdown
   #### Accessing Value Sets via the VSAC FHIR API

   Value sets used by this IG are published in the Value Set Authority Center (VSAC)
   and can be accessed programmatically via the VSAC FHIR API:

   **FHIR API Base URL:** `https://cts.nlm.nih.gov/fhir/`

   Example: To retrieve the "CRN Procedures Performed" value set:
   `GET https://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1180.19`
   ```

3. **Licensing Requirements subsection:**
   ```markdown
   #### Licensing Requirements

   Access to value sets through the VSAC FHIR API requires a UMLS license.
   A UMLS license is free and can be obtained by registering at
   [https://uts.nlm.nih.gov/uts/](https://uts.nlm.nih.gov/uts/).

   LOINC codes used in this IG are subject to the LOINC license terms
   available at [https://loinc.org/license/](https://loinc.org/license/).
   ```

4. **Value Sets table** listing all value sets with both VSAC browser links and FHIR API URLs:

   | Value Set Name | OID | VSAC FHIR API URL |
   |----------------|-----|-------------------|
   | WHT CRN Concomitant Procedures Performed | 2.16.840.1.113762.1.4.1180.14 | `https://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1180.14` |
   | WHT CRN Procedures Performed | 2.16.840.1.113762.1.4.1180.19 | `https://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1180.19` |

5. **LOINC Question Codes table** listing codes used in the CDE template.

6. Retain the existing `{% include list-simple-valuesets.xhtml %}` and `{% include list-simple-codesystems.xhtml %}` includes.

#### Task 4: Update the CDE Mapping Table (if LOINC codes available)

1. Add a fourth column "Question Code" to the CDE-to-FHIR mapping table in `source/pages/_includes/CDEs_FHIR.html`.
2. Populate it with the LOINC question codes from the template for each applicable CDE row.
3. Example update for the "Pregnancy Status" row:

   | Harmonized CDE | FHIR Data Element | FHIR Resource/Profile | Question Code |
   |----------------|-------------------|-----------------------|---------------|
   | Pregnancy Status | Observation.code, Observation.value[x] | us-core-observationresults | LOINC 66174-4 |

#### Task 5: Include Non-Accessible Value Sets in the IG

1. For any value sets confirmed as not accessible via the VSAC FHIR API:
   - Create `ValueSet-*.xml` or `ValueSet-*.json` resource files in `source/resources/`.
   - Register them in `ig.json` under the `"resources"` section.
2. Update `terminology.md` to note which value sets are IG-local vs. VSAC-hosted.

#### Task 6: Update Implementation Guidance

1. In `source/pages/guidance.md`, update **Step 6: Implement Terminology Validations** to reference the VSAC FHIR API:
   ```markdown
   The External CRN Data Collection System should invoke the VSAC FHIR API
   at `https://cts.nlm.nih.gov/fhir/` to validate coded answers using the
   `$validate-code` operation. A UMLS API key is required.
   ```

### Open Questions

1. **CDE Template Access:** The original template is hosted on Google Drive (`https://drive.google.com/file/d/1GGRZOJcLBAW_p_czcfaSe7xAirs8KPw6/view`). Is this file still accessible? The complete list of LOINC question codes depends on reviewing this spreadsheet.
2. **Value Set Currency:** The VSAC value sets were created for the original WHT CRN project. Are there newer versions or updated OIDs that should be referenced?
3. **Scope of Question Codes:** The resolution says "include LOINC and other codes wherever they have been specified as Question Code in the template." Does "other codes" refer to SNOMED CT, ICD-10, CPT, or other code systems? The CDE template would clarify this.
4. **Repository Staleness:** The last commit to `HL7/coordinated-registry-network` was in March 2019 (pre-ballot resolution). All changes from this ticket and the other 9+ resolved CRN tickets from the same ballot cycle appear to remain unimplemented.
5. **IG Tooling:** The repository uses the older Jekyll-based IG Publisher tooling (`"tool": "jekyll"` in `ig.json`). Should the IG be migrated to the newer FHIR IG Publisher template (sushi/FSH) as part of this work, or should changes use the existing tooling?
6. **Coordination with FHIR-21210:** Ticket FHIR-21210 ("Mappings of CDEs to FHIR section could be more useful") likely overlaps with the CDE table changes proposed here. These tickets should be implemented together to avoid conflicting edits.
