# Implementation Plan: FHIR-39004

**Title:** Change Study Design Value Set content logical definition to use SEVCO once published in THO
**Status:** Resolved - change required
**Resolution:** Persuasive
**Work Group:** Biomedical Research & Regulation
**Specification:** FHIR Core (FHIR)
**Resolved:** 2025-03-13

---

## Resolution Summary

The ticket requests changing the Study Design ValueSet (`http://hl7.org/fhir/ValueSet/study-design`) from its current content logical definition (CLD) — which individually enumerates 73 SEVCO concepts — to a filter-based CLD that includes three root codes and all their descendants from the SEVCO external code system (`https://fevir.net/sevco`). The resolution is "Persuasive" and states: "Use https://fevir.net/sevco when it is listed with THO."

**Prerequisite status:** SEVCO is now registered in THO (HL7/UTG) as an external code system with:
- TSMG-endorsed URI: `https://fevir.net/sevco`
- NamingSystem: `input/sourceOfTruth/external/hta/namingSystems/SEVCO.xml`
- Metadata record: `input/pagecontent/ECS_MR-SEVCO.xml`

This prerequisite is satisfied — SEVCO is published in THO.

## Feature Proposal

### Problem Statement

The Study Design ValueSet currently enumerates all 73 SEVCO concepts individually within its `compose.include` block. Each concept is listed with its code, display, and multiple SNOMED CT designations (definitions and synonyms). This approach has several problems:

1. **Maintenance burden**: Every time SEVCO adds, removes, or renames a study design concept, the ValueSet must be manually updated with individual concept entries.
2. **Sustainability**: The original enumeration was a temporary fix for the R5 ballot because SEVCO was not yet registered as an external code system in THO.
3. **Inconsistency**: The CLD does not express the *intent* (all study design-related SEVCO codes and descendants), but rather a frozen snapshot of 73 specific codes.

### Proposed Change

Replace the current `compose.include` block (73 individually enumerated concepts) with a filter-based include using three `concept` entries with `is-a` filters that include each code and all its descendants from the SEVCO code system:

| Code | Display | Definition |
|------|---------|------------|
| `SEVCO:01000` | Study Design | A plan specification for how and what kinds of data will be gathered as part of an investigation which may produce testable explanations, conclusions and predictions or test a hypothesis. |
| `SEVCO:00998` | Study Design Process | A specification of a sequence of actions for a component or part of a study design. |
| `SEVCO:00999` | Study Design Feature | An aspect or characteristic of a study design. |

The new CLD should use the `filter` element with `property=concept`, `op=is-a`, and `value=<code>` for each of the three root codes, rather than enumerating individual `concept` elements.

### Design Rationale

- **Sustainability**: A filter-based CLD automatically includes new SEVCO descendant codes as they are published, reducing maintenance overhead.
- **Correctness**: The three root codes (`SEVCO:01000`, `SEVCO:00998`, `SEVCO:00999`) capture the complete domain of study design characteristics, study design processes, and study design features from SEVCO.
- **THO alignment**: Now that SEVCO has a registered URI in THO (`https://fevir.net/sevco`), the ValueSet can properly reference the external code system and rely on its hierarchy for expansion.
- **Consistency with companion ticket**: FHIR-39003 (also Persuasive) covers the complementary work of removing SEVCO codes from `research-study-classifiers` CodeSystem and creating a convenience copy in THO.

## Impact Analysis

### Affected Repositories

#### HL7/fhir (FhirCore)

- **Role:** Primary source of the Study Design ValueSet definition used by Evidence and ResearchStudy StructureDefinitions.
- **Affected Files:**
  - `source/evidence/valueset-study-design.xml` — **Primary change target.** Replace the 73 enumerated SEVCO concepts in `compose.include` with three `is-a` filter entries. This will reduce the file from ~1891 lines to approximately 40–50 lines.
  - `source/evidence/structuredefinition-Evidence.xml` — No structural change needed. The `Evidence.studyDesign` element (line 706) already binds to `http://hl7.org/fhir/ValueSet/study-design` with extensible strength. The binding URL remains unchanged.
  - `source/researchstudy/structuredefinition-ResearchStudy.xml` — No structural change needed. The `ResearchStudy.studyDesign` element (line 456) already binds to `http://hl7.org/fhir/ValueSet/study-design` with preferred strength. The binding URL remains unchanged.
  - `source/researchstudy/codesystem-research-study-classifiers.xml` — No change needed for this ticket. The CodeSystem no longer contains SEVCO codes (they were already removed or were never merged into the R6 branch). Companion ticket FHIR-39003 addresses this separately.
- **Change Scope:** Moderate — The change is conceptually simple (replacing enumerated concepts with filters) but affects a large file and the resulting expansion will differ if SEVCO's hierarchy has changed since the concepts were enumerated.

#### HL7/UTG (Utg)

- **Role:** THO source of truth for the ValueSet and the SEVCO external code system registration.
- **Affected Files:**
  - `input/sourceOfTruth/fhir/valueSets/ValueSet-study-design.xml` — **Must be updated in parallel.** This file mirrors the FHIR Core ValueSet at URL `http://terminology.hl7.org/ValueSet/study-design` and currently also enumerates 73 SEVCO concepts individually. Apply the same filter-based CLD change.
  - `input/sourceOfTruth/external/hta/namingSystems/SEVCO.xml` — No change needed. Already defines SEVCO with preferred URI `https://fevir.net/sevco`.
  - `input/pagecontent/ECS_MR-SEVCO.xml` — No change needed. Already contains the external code system metadata record for SEVCO.
- **Change Scope:** Moderate — Same conceptual change as FHIR Core, applied to the THO copy.

### Breaking Changes

- **ValueSet expansion may differ:** The filter-based CLD includes all descendants of the three root codes from the current SEVCO hierarchy. If SEVCO has added or reorganized concepts since the 73-code snapshot was taken, the expanded ValueSet will contain different codes. This is intentionally the desired behavior but may affect systems that validated against the exact set of 73 codes.
- **Binding impact:** Both `Evidence.studyDesign` (extensible binding) and `ResearchStudy.studyDesign` (preferred binding) use this ValueSet. Since neither binding is required strength, implementations using codes outside the new expansion will not break validation, but they may see different code suggestions.
- **Immutable flag consideration:** The FHIR Core ValueSet has `<immutable value="true"/>`. Changing from enumerated to filter-based inclusion conceptually changes what "immutable" means for this ValueSet. This should be reviewed — if the intent is that the expansion dynamically includes new SEVCO codes as they appear, the `immutable` flag should likely be set to `false` or removed, since the expansion will change over time as SEVCO evolves.

### Terminology Impact

- **SEVCO external code system** (`https://fevir.net/sevco`): Already registered in THO. No changes needed to the code system registration itself.
- **ValueSet expansion**: Will change from a fixed set of 73 concepts to a dynamically computed set based on SEVCO hierarchy. The expansion depends on the SEVCO terminology server being able to resolve `is-a` filters for the three root codes.
- **Terminology server requirements**: Terminology servers expanding this ValueSet must have access to the SEVCO code system and support hierarchical (`is-a`) filtering. This is a new runtime dependency.

## Implementation Plan

### Prerequisites

1. **SEVCO published in THO** — ✅ Confirmed. The NamingSystem (`SEVCO.xml`) and metadata record (`ECS_MR-SEVCO.xml`) exist in HL7/UTG.
2. **FHIR-39003 coordination** — Should be implemented in coordination with FHIR-39003 (removing SEVCO codes from `research-study-classifiers` CodeSystem). However, since the CodeSystem already appears to have no SEVCO codes in the current R6 branch, this ticket can proceed independently.
3. **SEVCO hierarchy verification** — Verify that the three root codes (`SEVCO:01000`, `SEVCO:00998`, `SEVCO:00999`) and their descendants in the current SEVCO release cover the intent of the 73 previously enumerated codes.

### Step-by-Step Tasks

#### Task 1: Update the FHIR Core ValueSet (`HL7/fhir`)

**File:** `source/evidence/valueset-study-design.xml`

Replace the entire `<compose>` block (currently lines 41–1890, containing 73 individually enumerated `<concept>` entries under a single `<include>`) with a new `<compose>` block using three `<include>` elements, each with an `is-a` filter:

```xml
<compose>
  <include>
    <system value="https://fevir.net/sevco"/>
    <filter>
      <property value="concept"/>
      <op value="is-a"/>
      <value value="SEVCO:01000"/>
    </filter>
  </include>
  <include>
    <system value="https://fevir.net/sevco"/>
    <filter>
      <property value="concept"/>
      <op value="is-a"/>
      <value value="SEVCO:00998"/>
    </filter>
  </include>
  <include>
    <system value="https://fevir.net/sevco"/>
    <filter>
      <property value="concept"/>
      <op value="is-a"/>
      <value value="SEVCO:00999"/>
    </filter>
  </include>
</compose>
```

> **Note:** The `is-a` filter includes the root code itself plus all descendants. If the SEVCO code system uses `concept` as the filter property for hierarchy traversal, use `concept`. If it uses a different property name (e.g., `parent`), adjust accordingly. Verify against THO/SEVCO documentation.

Additionally:
- Review and potentially remove or set to `false` the `<immutable value="true"/>` element (line 40), since the expansion will now change as SEVCO evolves.
- Remove the `<version value="2.0"/>` from the system include (line 44 in current file), unless the intent is to pin to a specific SEVCO version.
- Update the `<date>` element to reflect the change date.

#### Task 2: Update the UTG ValueSet (`HL7/UTG`)

**File:** `input/sourceOfTruth/fhir/valueSets/ValueSet-study-design.xml`

Apply the identical `<compose>` block change as Task 1. This file currently mirrors the FHIR Core ValueSet with the same 73 enumerated SEVCO concepts from system `https://fevir.net/sevco`.

#### Task 3: Review `immutable` flag

Both the FHIR Core and UTG copies of the ValueSet set `<immutable value="true"/>`. With the change to filter-based inclusion, the expansion will vary as SEVCO adds new descendant codes. Evaluate whether:
- `immutable` should be changed to `false` (recommended, since expansion will change)
- Or kept as `true` with a pinned SEVCO version (if stability is preferred over currency)

#### Task 4: Verify SEVCO hierarchy coverage

Before finalizing, compare the set of codes that would be included by the three `is-a` filters against the current 73 enumerated codes to ensure no unintended omissions or additions. This requires querying the SEVCO code system at `https://fevir.net/resources/CodeSystem/27270` for descendants of:
- `SEVCO:01000` (Study Design)
- `SEVCO:00998` (Study Design Process)
- `SEVCO:00999` (Study Design Feature)

#### Task 5: Update version metadata

In both files:
- Update `<version>` to the next appropriate version
- Update `<date>` to the change date
- Update `<meta><lastUpdated>` accordingly

### Testing Considerations

1. **Expansion validation**: Expand the ValueSet using a terminology server that has SEVCO loaded. Verify the expansion includes at least the 73 previously enumerated codes plus any new descendants.
2. **Binding validation**: Validate sample Evidence and ResearchStudy resources against the updated ValueSet to ensure existing codes still pass validation.
3. **Cross-version comparison**: Compare the expanded code list from the filter-based CLD against the previous enumerated list to identify any additions or removals, and confirm they are intentional.
4. **Terminology server compatibility**: Test that major terminology servers (e.g., tx.fhir.org) can resolve the `is-a` filter for SEVCO codes. If SEVCO is not yet loaded in the HL7 terminology server, this may need to be coordinated.

### Open Questions

1. **Immutable flag**: Should the ValueSet remain `immutable=true` (with a pinned SEVCO version) or change to `immutable=false` (allowing dynamic expansion as SEVCO evolves)? The work group should decide the intended behavior.
2. **SEVCO version pinning**: Should the ValueSet `compose.include` specify a `version` for the SEVCO system (e.g., `2.0`) to ensure stability, or omit it to always use the latest SEVCO release?
3. **Terminology server readiness**: Is SEVCO loaded and available on tx.fhir.org for expansion? If not, what is the timeline for this?
4. **Coordination with FHIR-39003**: The companion ticket (removing SEVCO from `research-study-classifiers` CodeSystem and creating a THO convenience copy) appears partially addressed already (the CodeSystem has no SEVCO codes in the current branch). Is FHIR-39003 still needed, and should these be implemented as a single PR?
5. **Filter property name**: The standard FHIR filter property for hierarchical inclusion is `concept` with op `is-a`. Verify this is correct for SEVCO's hierarchy representation in THO.
