## Description

Add `ResearchStudy.comparisonGroup.name` 0..1 string to allow simple labels for comparison groups. This is needed so that implementers do not need to invent and selectively use complex pathways to create labels for comparison groups. Use case established for converting European Union Clinical Trial Information System (EU CTIS) records to FHIR.

## Resolution Description

> Add ResearchStudy.comparisonGroup.name 0..1 string

## Feature Proposal

### What is being requested?

Re-introduce a `name` element on the `ResearchStudy.comparisonGroup` backbone element with cardinality **0..1** and datatype **string**. This provides a simple, human-readable label for each comparison group (arm) in a research study.

### Historical context

The `name` element previously existed in **R5** as `ResearchStudy.comparisonGroup.name` with cardinality **1..1 string** (required). It was removed during the R6 restructuring. This ticket re-introduces it with **relaxed cardinality (0..1)**, making it optional rather than required.

### Why is this needed?

Without a `name` element, implementers must resort to complex workarounds to label comparison groups — e.g., encoding names into `eligibility.display` or `description`. The EU CTIS use case demonstrates a concrete need for simple, structured labeling of comparison groups when converting external clinical trial registry data to FHIR.

### Breaking Changes

**None.** This is a purely additive change:
- New optional element (0..1) — existing instances remain valid
- No changes to existing elements or their cardinalities
- No terminology changes required
- Re-introduction of a well-understood R5 element at relaxed cardinality

### Compatibility Notes

- **R5 → R6 migration**: Systems that used the R5 `comparisonGroup.name` (which was 1..1) can now map directly to this element again. The relaxed cardinality (0..1) means R5 data will be valid without modification.
- **Serialization**: All serialization formats (JSON, XML, RDF) will pick up the new element automatically from the StructureDefinition.

## Detailed Implementation Plan

### File 1: StructureDefinition — Add `name` element

**File:** `source/researchstudy/structuredefinition-ResearchStudy.xml`
**Repository:** HL7/fhir

**Action:** Insert a new `<element>` block for `ResearchStudy.comparisonGroup.name` immediately after the `ResearchStudy.comparisonGroup` backbone element definition (after line 986, before `ResearchStudy.comparisonGroup.targetNumber`).

Placing `name` as the **first child** of `comparisonGroup` is consistent with the pattern used in R5 and with other backbone elements that start with a name/label (e.g., `ResearchStudy.objective` has a `name` first child).

**Element definition to add:**

```xml
    <element id="ResearchStudy.comparisonGroup.name">
      <path value="ResearchStudy.comparisonGroup.name"/>
      <short value="Label for study comparisonGroup"/>
      <definition value="Human-readable label for this comparisonGroup of the study."/>
      <min value="0"/>
      <max value="1"/>
      <type>
        <code value="string"/>
      </type>
    </element>
```

**Notes:**
- The `short` and `definition` texts are adapted from the R5 definition ("Unique, human-readable label for this comparisonGroup of the study.") but with "Unique" removed since uniqueness is not enforceable at the StructureDefinition level and the cardinality is now optional.
- No mappings to BRIDG or clinicaltrials-gov are included initially; these can be added separately if the work group identifies appropriate mappings.
