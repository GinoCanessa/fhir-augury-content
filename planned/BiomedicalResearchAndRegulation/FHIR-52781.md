### Original Comment

> No valid use case for the structure maps that are in the IG. Mapping Errors are misleading.

### Resolution Description

> Although the structure maps are not referenced directly in the use cases, we use them along with the narrative about the mappings to indicate in a structured way how to map from FHIR resources to OMOP tables.
>
> We do need to update them to be more complete and include the terminology mapping.

## Resolution Analysis

**Resolution "Not Persuasive with Modification"**: The work group rejected the premise that structure maps have no valid use case, but acknowledged that the maps contain errors and need to be updated. The status "Resolved — change required" confirms implementation is needed.

**What to implement**: The resolution mandates two specific changes:
1. **Make structure maps more complete** — add missing required fields, fix errors, ensure they align with the narrative guidance
2. **Include terminology mapping** — add `translate()` calls and ConceptMap references where maps currently copy raw codes

## Feature Proposal

### Overview

Update all 10 FHIR Mapping Language (`.fml`) structure maps in the FHIR to OMOP IG to fix errors, add missing required fields, and include terminology mapping via `translate()` calls. This brings the formal structure maps into alignment with the richer narrative mapping guidance already present in the IG.

### Goals

1. Fix known mapping errors (variable name bugs, incorrect field assignments)
2. Add missing required OMOP fields (`person_id`, domain PKs, `*_type_concept_id`)
3. Replace raw code copies with `translate()` calls referencing existing ConceptMaps
4. Ensure each map is internally consistent with its corresponding narrative intro page
5. Add comments explaining fields that cannot be structurally mapped (e.g., PK/FK identity management)

## Detailed Implementation Plan

### Phase 1: Fix Existing Bugs in Structure Maps

These are concrete errors that must be fixed regardless of completeness improvements.

#### Task 1.1: Fix `ConditionMap.fml` variable name bug

**File**: `input/maps/ConditionMap.fml`

**Bug**: Line referencing abatement uses `adt` instead of `abdt`:
```fml
src.abatement : dateTime as abdt -> tgt.condition_end_datetime = adt, ...
```

**Fix**: Change `adt` to `abdt`:
```fml
src.abatement : dateTime as abdt -> tgt.condition_end_datetime = abdt, tgt.condition_end_date = cast(abdt, "date");
```

#### Task 1.2: Fix `medication.fml` scoping bugs

**File**: `input/maps/medication.fml`

**Bug**: The `end` period mappings reference `fps` (the start date variable) instead of `fpe`:
```fml
s.end as fpe -> tgt.drug_exposure_end_datetime = fpe, tgt.drug_exposure_end_date = cast(fps, "date");
```
```fml
s.end as fpe -> tgt.verbatim_end_date = cast(fps, "date");
```

**Fix**: Replace `fps` with `fpe` in both lines:
```fml
s.end as fpe -> tgt.drug_exposure_end_datetime = fpe, tgt.drug_exposure_end_date = cast(fpe, "date");
```
```fml
s.end as fpe -> tgt.verbatim_end_date = cast(fpe, "date");
```

#### Task 1.3: Fix `Procedure.fml` incorrect triple assignment

**File**: `input/maps/Procedure.fml`

**Bug**: Raw code is assigned to `procedure_concept_id`, `procedure_source_value`, AND `procedure_source_concept_id` simultaneously:
```fml
sc.code as a -> tgt.procedure_concept_id, tgt.procedure_source_value, tgt.procedure_source_concept_id = a;
```

**Fix**: Per FHIR-52014, remove `procedure_source_concept_id`. Use `translate()` for `procedure_concept_id`, keep raw code for `procedure_source_value`:
```fml
sc.code as a -> tgt.procedure_concept_id = translate(a, '<ConceptMapURL>', 'code'), tgt.procedure_source_value = a;
```

### Phase 2: Add Missing Required Fields

Per FHIR-52010, add the following fields to each structure map where they are currently missing or commented out.

#### Task 2.1: Add `person_id` mapping pattern

All maps need a `person_id` mapping. Since FHIR IDs cannot be directly converted to OMOP integer IDs, use a comment-documented placeholder pattern:

```fml
// person_id: Requires external identity resolution from FHIR Patient reference to OMOP person_id
// This cannot be expressed purely in the mapping language and requires an external lookup service
src.subject -> tgt.person_id;  // placeholder — resolved externally
```

**Affected files**: `medication.fml`, `Observation.fml`, `Procedure.fml` (currently commented out); verify all others.

#### Task 2.2: Add domain primary key mapping pattern

Each map needs its domain PK (e.g., `observation_id`, `procedure_occurrence_id`). Same external resolution pattern:

```fml
// <domain>_id: Requires external identity generation/resolution
src.id -> tgt.<domain>_id;  // placeholder — assigned by target system
```

**Affected files**: `Observation.fml`, `Procedure.fml`, `medication.fml` (currently commented out).

#### Task 2.3: Add `*_type_concept_id` fields

Add type concept ID fields where missing. Use hardcoded OMOP concept IDs per the OMOP CDM conventions:

| Map | Field | Suggested Value | Meaning |
|-----|-------|----------------|---------|
| `Allergy.fml` | `observation_type_concept_id` | 32817 | EHR |
| `Observation.fml` | `observation_type_concept_id` | 32817 | EHR |
| `Measurement.fml` | `measurement_type_concept_id` | 32817 | EHR |
| `Procedure.fml` | `procedure_type_concept_id` | 32817 | EHR |
| `EncounterVisit.fml` | `visit_type_concept_id` | 32817 | EHR |
| `medication.fml` | — | — | Already maps `drug_type_concept_id` from category |

### Phase 3: Add Terminology Mapping (`translate()` Calls)

Per the resolution and FHIR-52549, structure maps must use `translate()` with ConceptMap references instead of copying raw FHIR codes into OMOP concept_id fields.

#### Task 3.1: Update `Observation.fml` — add `translate()` for concept fields

**Current** (copies raw code):
```fml
sc.code as a -> tgt.observation_concept_id = a;
```

**Updated**:
```fml
sc.code as a -> tgt.observation_concept_id = translate(a, '<ObservationConceptMapURL>', 'code'), tgt.observation_source_value = a;
```

Also fix `unit_concept_id` and `value_as_concept_id`:
```fml
s.unit as b -> tgt.unit_concept_id = translate(b, '<UnitConceptMapURL>', 'code'), tgt.unit_source_value = b;
```
```fml
sc.code as a -> tgt.value_as_concept_id = translate(a, '<ValueConceptMapURL>', 'code');
```

#### Task 3.2: Update `medication.fml` — add `translate()` for drug_concept_id

**Current** (copies raw code):
```fml
sc.code as a -> tgt.drug_concept_id = a;
```

**Updated**:
```fml
sc.code as a -> tgt.drug_concept_id = translate(a, '<DrugConceptMapURL>', 'code'), tgt.drug_source_value = a;
```

Also update `drug_type_concept_id`:
```fml
sc.code as a -> tgt.drug_type_concept_id = translate(a, '<DrugTypeConceptMapURL>', 'code');
```

#### Task 3.3: Update `Procedure.fml` — add `translate()` for procedure_concept_id

Already addressed in Task 1.3. Ensure a valid ConceptMap URL is used.

#### Task 3.4: Update `Measurement.fml` and `SimpleVitalSignsMap.fml`

Review and add `translate()` where raw codes are copied to `measurement_concept_id` and related fields.

#### Task 3.5: Create missing ConceptMap FSH resources

New ConceptMap FSH files may be needed:

| File to Create | Purpose |
|----------------|---------|
| `input/fsh/ProcedureConceptMaps.fsh` | Maps FHIR procedure codes → OMOP procedure_concept_id |
| `input/fsh/MedicationConceptMaps.fsh` | Maps FHIR medication codes → OMOP drug_concept_id |
| `input/fsh/ObservationConceptMaps.fsh` | Maps FHIR observation codes → OMOP observation_concept_id |

**Note**: The ConceptMaps need only be representative/illustrative since the actual FHIR→OMOP vocabulary mapping is managed in the OMOP vocabulary infrastructure, not in the IG itself.

### Phase 4: Remove Incorrect `*_source_concept_id` Mappings

Per FHIR-52014, remove mappings to `*_source_concept_id` fields that incorrectly contain raw codes instead of OMOP concept IDs.

#### Task 4.1: Audit and remove `*_source_concept_id` assignments

**Affected files** (known):
- `Procedure.fml`: Remove `procedure_source_concept_id = a` (Task 1.3)
- `ImmunizationMap.fml`: Check for `drug_source_concept_id`
- `EncounterVisit.fml`: Check for `visit_source_concept_id`
- `Allergy.fml`: Check for `observation_source_concept_id`

Replace with `*_source_value` where the raw code should go.

### Phase 5: Add Missing Dosage Mapping to Medication Map

Per FHIR-52495, the medication structure map must include dosage mappings discussed in the narrative.

#### Task 5.1: Add dosage group to `medication.fml`

Add a mapping from `MedicationStatement.dosage` to DrugExposure fields:
```fml
src.dosage first as d -> tgt then {
    d.doseAndRate first as dr -> tgt then {
        dr.dose : Quantity as dq -> tgt then {
            dq.value as v -> tgt.quantity = v;
            dq.unit as u -> tgt.dose_unit_source_value = u;
        };
    };
    d.route as r -> tgt then {
        r.coding as rc -> tgt then {
            rc.code as c -> tgt.route_concept_id = translate(c, '<RouteConceptMapURL>', 'code'), tgt.route_source_value = c;
        };
    };
};
```

### Phase 6: Align Narrative Intro Pages

After updating each `.fml` file, review its corresponding `StructureMap-*-intro.md` page to ensure the narrative description matches the updated structure map.

#### Task 6.1: Review and update each intro page

For each of the 9 intro markdown files in `input/pagecontent/StructureMap-*-intro.md`:

## Testing & Validation

1. **IG Build**: Run the FHIR IG Publisher (`_genonce.sh` / `_genonce.bat`) to verify all `.fml` files parse correctly as valid FHIR Mapping Language
2. **ConceptMap References**: Verify all `translate()` ConceptMap URLs reference ConceptMaps that exist in the IG
3. **Narrative Consistency**: Manual review of each intro page against its updated structure map
4. **Peer Review**: StructureMap changes should be reviewed by the BRR work group given the high number of related tickets

## Risks and Considerations

1. **Coordination with related tickets**: FHIR-52774, FHIR-52549, FHIR-52010, FHIR-52012, FHIR-52014, and FHIR-52495 all target the same files. These should ideally be implemented together in one PR to avoid merge conflicts.
2. **ConceptMap coverage**: Creating representative ConceptMaps for Procedure, Medication, and Observation domains requires domain expertise. The IG may choose to reference external terminology services rather than defining complete ConceptMaps inline.
3. **PK/FK identity management**: The resolution for FHIR-52010 acknowledged that `person_id` and domain PKs cannot be fully expressed in the mapping language. Clear comments explaining this limitation should be added.
4. **Draft status**: All maps are marked `status = 'draft'`. Consider whether any should advance to `active` after this update.
