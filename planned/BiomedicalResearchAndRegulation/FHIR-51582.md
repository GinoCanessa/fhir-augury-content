## Resolution Summary

The workgroup accepted this as **Persuasive**. The resolution states:

> *"We will fix the SQL to have the proper OMOP element names."*

The ticket identifies that example SQL in the CodeableConcept mapping pattern page includes `qualifier_source_value` as a column in an `INSERT INTO condition_occurrence` statement. However, `qualifier_source_value` is **not** a valid column in the OMOP CDM `condition_occurrence` table — it only exists on the `observation` table. The SQL must be corrected to use only valid OMOP column names for the target table.

## Feature Proposal

### Problem

The published ballot version (2025Sep) of the CodeableConcept Mapping Patterns page (`CodeableConceptPattern.html`) contained SQL examples that reference `qualifier_source_value` as a column in `condition_occurrence` INSERT statements. According to the OMOP CDM v5.4 specification, `qualifier_source_value` is a column only on the `observation` table, not on `condition_occurrence`.

The valid columns for `condition_occurrence` are:
- `condition_occurrence_id`, `person_id`, `condition_concept_id`, `condition_start_date`, `condition_start_datetime`, `condition_end_date`, `condition_end_datetime`, `condition_type_concept_id`, `condition_status_concept_id`, `stop_reason`, `provider_id`, `visit_occurrence_id`, `visit_detail_id`, `condition_source_value`, `condition_source_concept_id`, `condition_status_source_value`

### Proposed Solution

Remove `qualifier_source_value` from all `INSERT INTO condition_occurrence` SQL examples and their corresponding VALUES clauses. Update the mapping strategy narrative text to no longer reference using `qualifier_source_value` as a quality flag in condition_occurrence.

### Specific Changes Required

Three SQL blocks in the ballot version contain the invalid column:

#### 1. Example 2 — Ambiguous Clinical Language ("Patient has diabetes")

**Ballot SQL** (incorrect — `qualifier_source_value` is not a `condition_occurrence` column):
```sql
INSERT INTO condition_occurrence (
    ...
    condition_source_concept_id,
    qualifier_source_value          -- ❌ INVALID column
) VALUES (
    ...
    0,
    'LOW_SPECIFICITY'               -- ❌ INVALID value
);
```

**Fix**: Remove `qualifier_source_value` from column list and `'LOW_SPECIFICITY'` from VALUES. Update mapping strategy text from *"Map to general diabetes concept with quality flag for specificity limitation"* to guidance that does not prescribe use of an invalid column.

#### 2. Example 3a — Medical Abbreviations, History of MI

**Ballot SQL** (incorrect):
```sql
INSERT INTO condition_occurrence (
    ...
    condition_source_concept_id,
    qualifier_source_value          -- ❌ INVALID column
) VALUES (
    ...
    0,
    'HISTORY_OF'                    -- ❌ INVALID value
);
```

**Fix**: Remove `qualifier_source_value` from column list and `'HISTORY_OF'` from VALUES.

#### 3. Example 3b — SOB Complaint (observation table)

The observation INSERT also used `qualifier_source_value` with value `'PATIENT_COMPLAINT'`. While `qualifier_source_value` IS a valid column on `observation`, the related ticket [FHIR-51583](https://jira.hl7.org/browse/FHIR-51583) addresses the misuse of this column as a generic metadata tag, so this should also be corrected.

#### 4. Field Mapping Summary Table

Remove `qualifier_source_value` rows from the HTML summary table at the bottom of the page that listed `"LOW_SPECIFICITY"` and `"HISTORY_OF"` / `"PATIENT_COMPLAINT"` as mapped values.

### Pre-Implementation Status

6. **Updating** the field mapping summary table to remove `qualifier_source_value` rows
