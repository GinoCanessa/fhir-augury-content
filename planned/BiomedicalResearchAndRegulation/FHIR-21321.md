1. **Add to parenthetical statement:** Expand the parenthetical after `distinctIdentifier` from `(returned as donation_id)` to `(returned as donation_id from the Parse UDI API)` to clarify the source of the `donation_id` mapping.
2. **Correct element name:** Fix the typo `manufacturineDate` → `manufactureDate` to match the actual FHIR `Device.manufactureDate` element name.

## Feature Proposal

### Problem Statement

The device-crn profile's implementation guidance in the CRN IG contains two errors in the sentence describing the AccessGUDID Parse UDI API output:

**Current (incorrect) text:**
> The Parse UDI API allows users to pass a UDI and return each part of the UDI in a structured format (specifically the serialNumber, lotNumber, expirationDate, distinctIdentifier (returned as donation_id) or manufacturineDate).

The problems are:
1. The parenthetical `(returned as donation_id)` does not clarify that `donation_id` is the field name returned by the Parse UDI API, which could confuse implementers about where this mapping comes from.
2. The element name `manufacturineDate` is a typo — the correct FHIR Device resource element is `Device.manufactureDate` (no extra "ine").

### Proposed Change

Replace the sentence with the corrected version:

> The Parse UDI API allows users to pass a UDI and return each part of the UDI in a structured format (specifically the serialNumber, lotNumber, expirationDate, distinctIdentifier (returned as donation_id from the Parse UDI API) or manufactureDate).

This corrects both issues in a single text edit.

### Design Rationale

- The correction aligns the guidance text with the actual FHIR `Device.manufactureDate` element path, eliminating implementer confusion from the typo.
- Adding "from the Parse UDI API" to the parenthetical makes it explicit that `donation_id` is the API field name, matching the style used in the ticket description and the related ticket FHIR-20980.
- This is a documentation-only change with no structural or behavioral impact.
