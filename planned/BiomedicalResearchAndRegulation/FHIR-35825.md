# Implementation Plan: FHIR-35825

**Title:** The FHIR FHIRPath mapping for the AE has errors
**Status:** Resolved - change required
**Resolution:** Persuasive
**Work Group:** Biomedical Research & Regulation
**Specification:** CDISC Mappings (FHIR)
**Resolved:** 2022-01-18

---

## Resolution Summary

The resolution is **"Correct as suggested"** — the FHIRPath column and Element column in the AE (Adverse Event) domain mapping table are incorrect for 11 SDTM variables related to MedDRA coding of the adverse event term. These variables are currently incorrectly mapped to `Condition.code` via `AdverseEvent.resultingCondition.resolve().code`, but they should map directly to `AdverseEvent.event` (R4) since the CDISC Mappings IG targets FHIR R4 (`fhirVersion: 4.0.1`).

The semantic distinction is important: `AdverseEvent.event` represents the actual adverse event that occurred (coded in SNOMED CT or MedDRA), while `AdverseEvent.resultingCondition` represents a downstream consequence of the event (e.g., a broken leg resulting from a fall). The MedDRA coding hierarchy (AETERM, AEDECOD, AELLT, etc.) describes the event itself, not the resulting condition.

## Feature Proposal

### Problem Statement

The CDISC-to-FHIR mapping IG's Adverse Event (AE) domain table contains systematic errors in both the **Element** column and the **FHIRPath** column for 11 SDTM variables that represent the adverse event term and its MedDRA coding hierarchy. Currently:

- **Element column** incorrectly shows `Condition.code` (linking to `condition-definitions.html#Condition.code`)
- **FHIRPath column** incorrectly shows `AdverseEvent.resultingCondition.resolve().code`

These should instead reference `AdverseEvent.event`, which is the R4 element representing the specific event or incident that occurred (type: CodeableConcept, cardinality: 0..1).

The affected SDTM variables are:

| # | Variable | Description |
|---|----------|-------------|
| 1 | AETERM | Reported Term for the Adverse Event |
| 2 | AEDECOD | Dictionary-Derived Term (MedDRA Preferred Term) |
| 3 | AELLT | Lowest Level Term |
| 4 | AELLTCD | Lowest Level Term Code |
| 5 | AEPTCD | Preferred Term Code |
| 6 | AEHLT | High Level Term |
| 7 | AEHLTCD | High Level Term Code |
| 8 | AEHLGT | High Level Group Term |
| 9 | AEHLGTCD | High Level Group Term Code |
| 10 | AESOC | Primary System Organ Class |
| 11 | AESOCCD | Primary System Organ Class Code |

### Proposed Change

For each of the 11 affected SDTM variables, update the AE domain mapping:

1. **Element column**: Change from `Condition.code` → `AdverseEvent.event`
   - Update the hyperlink from `{{site.data.fhir.path}}condition-definitions.html#Condition.code` to `{{site.data.fhir.path}}adverseevent-definitions.html#AdverseEvent.event`
   - Update the element description from "Identification of the condition, problem or diagnosis" to "Specific event that occurred or that was averted"
   - Update cardinality display from `0..1 CodeableConcept` (same type, so just update the link context)
   - Update binding from `ConditionKind` (example) to `AdverseEventType` (example) referencing ValueSet `adverse-event-type`

2. **FHIRPath column**: Change from `AdverseEvent.resultingCondition.resolve().code` → `AdverseEvent.event`

> **Note:** In R5, `AdverseEvent.event` was renamed to `AdverseEvent.code`. The CDISC mapping IG targets R4 (`fhirVersion: 4.0.1`), so `AdverseEvent.event` is the correct element name. A future R5 migration would change this to `AdverseEvent.code`.

### Design Rationale

As explained by commenter Hugh Glover:

- `AdverseEvent.event` is the SNOMED/MedDRA-coded description of what happened (e.g., "fell out of bed", "bleeding in left eye"). This is what the SDTM AE term and MedDRA coding hierarchy describe.
- `AdverseEvent.resultingCondition` is the downstream medical consequence of the event (e.g., "broken leg", "blindness in left eye"). This is a reference to a separate Condition resource.

The MedDRA coding variables (AETERM, AEDECOD, AELLT, etc.) all describe the adverse event itself at various levels of the MedDRA hierarchy, not the resulting medical condition. Therefore, they belong mapped to `AdverseEvent.event`, not `Condition.code` via `resultingCondition`.

### Breaking Changes

- **No breaking changes to FHIR resources** — this is a mapping documentation correction only.
- **Semantic impact on implementers:** Systems that implemented the CDISC-to-FHIR mapping based on the incorrect table would have mapped AE term/coding data into `resultingCondition` (a Reference to Condition) instead of `event` (an inline CodeableConcept). Those implementations would need to be updated to place MedDRA codes directly in `AdverseEvent.event` rather than creating a separate Condition resource referenced via `resultingCondition`.

### Terminology Impact

- **No terminology changes required.** The `AdverseEvent.event` element has an `example` binding to `http://hl7.org/fhir/ValueSet/adverse-event-type` (SNOMED CT). The MedDRA coding used in CDISC is compatible with this binding strength (example bindings are non-restrictive).
- The mapping table's binding reference should change from `ConditionKind` (example) to `AdverseEventType` (example) for the affected rows.
