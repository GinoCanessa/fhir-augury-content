# Implementation Plan: FHIR-21017

**Title:** use procedure.location instead of location.identifier?
**Status:** Resolved - change required
**Resolution:** Persuasive
**Work Group:** Biomedical Research & Regulation
**Specification:** US Coordinated Registry Network (CRN) (FHIR)
**Resolved:** 2019-08-28

---

## Resolution Summary

The resolution directs implementers to modify the CDE (Common Data Element) mapping for "Facility ID" from `Location.identifier` to `Procedure.location.identifier`, making the data path explicit through the Procedure resource. The profile associated with this CDE should change from `us-core-location` to `procedure-crn`. The `procedure-crn` profile must be updated to constrain `Procedure.location` to reference the US Core Location profile (`us-core-location`), ensuring the facility identifier is discoverable through the procedure's location reference chain.

## Feature Proposal

### Problem Statement

The CRN IG's CDE-to-FHIR mapping table currently maps the "Facility ID" data element directly to `Location.identifier` using the `us-core-location` profile. This is semantically ambiguous because:

1. It does not specify _which_ Location resource is relevant — the mapping implies a standalone Location resource rather than one connected to the procedure context.
2. The CRN's purpose is to collect the identifier of "the facility where the procedure was performed." The relationship between the procedure and the facility is only implicit.
3. The FHIR Procedure resource already has a `Procedure.location` element (0..1, Reference(Location)) that explicitly captures "the location where the procedure actually happened."

By not using `Procedure.location`, the mapping misses the semantic link between the procedure and the facility, making it harder for implementers to understand how to populate a conformant CRN bundle.

### Proposed Change

Per the resolution:

1. **Update CDE mapping:** Change the "Facility ID" CDE mapping from `Location.identifier` (profile: `us-core-location`) to `Procedure.location.identifier` (profile: `procedure-crn`).
2. **Update procedure-crn profile:** Add a type constraint on `Procedure.location` that requires it to reference a resource conforming to the US Core Location profile (`http://hl7.org/fhir/us/core/StructureDefinition/us-core-location`).
3. **Maintain us-core-location reuse:** Implementers continue to use the `us-core-location` profile for the actual Location resource instance within the bundle; the change only affects how that resource is _reached_ (via `Procedure.location`).

### Design Rationale

- **Explicit data path:** `Procedure.location.identifier` provides a clear, unambiguous traversal path from the procedure to the facility ID, aligning with the CRN's intent to collect facility information in the context of a specific procedure.
- **Profile-level enforcement:** Constraining `Procedure.location` to reference `us-core-location` ensures type safety and conformance validation without introducing new extensions.
- **Consistency with FHIR core:** `Procedure.location` is the standard FHIR element for this purpose (R4 definition: "The location where the procedure actually happened"), so this change aligns the CRN IG with FHIR design intent.
- **No new elements or extensions required:** The change reuses existing FHIR elements and profiles.

- **Change Scope:** None (reference only)

### Breaking Changes

- **Minor breaking change for CRN implementers:** Implementers who previously created standalone Location resources and mapped Facility ID directly to `Location.identifier` without connecting them via `Procedure.location` will need to update their implementations to populate `Procedure.location` as a reference to the Location resource. This is a structural change in how the CRN bundle is composed.
- **Profile conformance impact:** If the `procedure-crn` profile adds a Must Support flag on `Procedure.location`, existing bundles that omit this element would no longer be fully conformant with the updated profile. However, since `Procedure.location` remains 0..1, it does not break syntactic validity.
- **No impact on core FHIR resources:** The change is entirely within the CRN IG scope.

### Related Specifications

- **US Core (R4):** The `us-core-location` profile is referenced as the target profile for `Procedure.location`. The CRN IG already depends on US Core R4 (declared in `ig.json` → `dependencyList`).
- **SDC (Structured Data Capture):** The CRN IG uses SDC profiles for questionnaires; this change does not affect SDC dependencies.

## Implementation Plan

1. Clone or fork `HL7/coordinated-registry-network` from GitHub (https://github.com/HL7/coordinated-registry-network).

3. Verify the `us-core-location` profile canonical URL: `http://hl7.org/fhir/us/core/StructureDefinition/us-core-location`.
4. Review any pending changes from related tickets (FHIR-19370, FHIR-21210, FHIR-21211) to avoid merge conflicts.

### Step-by-Step Tasks

#### Task 1: Update CDE Mapping Table

**File:** `source/pages/_includes/CDEs_FHIR.html`

Update the "Faciltiy ID" row in the HTML mapping table:

| Column | Current Value | New Value |
|--------|--------------|-----------|
| Harmonized Common Data Element | Faciltiy ID | Facility ID _(also fix typo)_ |
| FHIR Data Element | `Location.identifier` | `Procedure.location.identifier` |
| FHIR Resource/Profile/Extension | `us-core-location` | `procedure-crn` |

Specifically, change:
```html
<tr style='height:20px;'>
  <td class="maptable16">Faciltiy ID</td>
  <td class="maptable17">Location.identifier</td>
  <td class="maptable17">us-core-location</td>

```
To:
```html
<tr style='height:20px;'>
  <td class="maptable16">Facility ID</td>

  <td class="maptable17">procedure-crn</td>
```

#### Task 2: Update procedure-crn Profile Definition

**File:** `source/resources/procedure-crn-profile-spreadsheet.xml`

In the profile spreadsheet, update the `Procedure.location` element definition:

1. **Add or update the element row for `Procedure.location`** to include:
   - **Must Support:** `Y` (to indicate implementers must support this element)
   - **Type constraint:** Set the target profile on the Reference type to `http://hl7.org/fhir/us/core/StructureDefinition/us-core-location`
   - **Short description:** "Facility where the procedure was performed"
   - **Definition:** "Reference to the Location resource representing the facility where the procedure was performed. Use the us-core-location profile for the referenced Location resource."

2. Since this IG uses the legacy spreadsheet format, the exact XML structure for the type constraint row should follow the pattern already used for other elements in the spreadsheet. Look for the `<Row>` element corresponding to `Procedure.location` and update its type cell to include the profile reference.

#### Task 3: Update Narrative Documentation (if needed)

**File:** `source/pages/profiles.md`

Consider adding a note under the "CRN Specific Profiles" section clarifying that:
- The `procedure-crn` profile constrains `Procedure.location` to reference a `us-core-location`-conformant Location resource.
- Implementers should include the Location resource in the CRN bundle and reference it from `Procedure.location`.

#### Task 4: Update or Create Examples (if applicable)

**File:** `source/examples/` directory

If Procedure examples exist in the IG, update them to include a `Procedure.location` reference pointing to a Location resource with an `identifier` element. If no examples exist, consider creating one to demonstrate the correct pattern:

### Open Questions

1. **Legacy tooling:** The IG uses the legacy spreadsheet-based profile definition format and an older IG publisher. Should this ticket's implementation also migrate to FSH (FHIR Shorthand) or the newer IG publisher template? This could be a significant effort but would modernize the IG.
2. **Must Support on Procedure.location:** The resolution says to use `Procedure.location.identifier` for the Facility ID mapping. Should `Procedure.location` be marked as Must Support in the profile? The resolution implies yes, but this should be confirmed with the work group.
3. **Cardinality of Procedure.location:** The base cardinality is 0..1. Should the CRN profile tighten this to 1..1 (require a facility for every procedure)? The resolution does not explicitly mandate this.
4. **Repo activity:** The `HL7/coordinated-registry-network` repository has not been updated since March 2019 (pre-resolution). Multiple resolved tickets (FHIR-21210, FHIR-21211, FHIR-20788, etc.) also appear unapplied. A batch update addressing all pending resolved tickets may be more efficient.
5. **Coordination with FHIR-21210:** That ticket requests improvements to the CDE mapping table format. If implementing both together, the mapping table structure may need more extensive changes beyond just the Facility ID row.
