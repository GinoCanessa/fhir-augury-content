## Resolution Summary

**Resolution:** Persuasive

The Biomedical Research & Regulation work group voted the change as
Persuasive with the resolution description: **"change maximal cardinality
of identifier to \*"**.

Concretely, the upper cardinality of `RegulatedAuthorization.case.identifier`
must change from `0..1` to `0..*` so that a single case may carry more
than one identifier. No other elements, types, bindings, or invariants
are affected by the resolution as voted.

## Feature Proposal

### Problem Statement

`RegulatedAuthorization.case` represents the regulatory case or
procedure under which a regulated authorization is granted or amended.
The current schema only allows one `identifier` on the case
(`0..1`).

In practice, a regulatory case is routinely identified by more than one
external system over its lifetime — for example, a case can carry both
an originating regulator's procedure number and a downstream
agency's/sponsor's tracking identifier, and these may be present
simultaneously rather than supersede one another. The ticket
description (Lloyd McKenzie) states this as a general principle:
"Identifiers can almost always be assigned by multiple entities over
time, but this element is not marked as repeating."

This is consistent with two sibling fixes in the same R6 ballot cycle
that were applied as part of the same commit grouping:

- **FHIR-55046** — `Ingredient.identifier` made `0..*` (from `0..1`).
- **FHIR-55057** — `SubstanceDefinition.moiety.identifier` made `0..*` (from `0..1`).

### Proposed Change

Change exactly one cardinality and surface the change in the resource's
ballot-note:

1. In the `RegulatedAuthorization` StructureDefinition differential,
   change the upper cardinality of the element
   `RegulatedAuthorization.case.identifier` from `1` to `*`:

   ```xml
   <element id="RegulatedAuthorization.case.identifier">
     <path value="RegulatedAuthorization.case.identifier"/>
     <short value="Identifier by which this case can be referenced"/>
     <definition value="Identifier by which this case can be referenced."/>
     <min value="0"/>
     <max value="*"/>            <!-- was: <max value="1"/> -->
     <type>
       <code value="Identifier"/>
     </type>
     <isSummary value="true"/>
     ...
   </element>
   ```

   No change to `min`, `type`, `isSummary`, `mapping`, `short`, or
   `definition`.

2. Add a balloter-facing note to
   `regulatedauthorization-introduction.xml` recording the substantive
   change so it shows in the resource intro on the rendered page.
   Following the pattern already applied for the parallel R6 changes,
   the note is a `<blockquote class="ballot-note">` block listing the
   updated element and its prior cardinality.

No new elements, no new extensions, no terminology changes, and no
invariant/search-parameter changes are required.

### Design Rationale

- **Direct application of the resolution.** The resolution wording is
  explicit and unambiguous: "change maximal cardinality of identifier
  to *". There is no design degree of freedom to consider.
- **Consistent with FHIR's general treatment of `identifier`.** Across
  the spec the `identifier` element is almost always `0..*` to allow
  business identifiers from multiple assigners; `case.identifier` was
  the outlier on this resource.
- **Consistent with sibling resolutions in the same ballot cycle.**
  FHIR-55046 (`Ingredient.identifier`) and FHIR-55057
  (`SubstanceDefinition.moiety.identifier`) were resolved the same way
  and committed in the same group. Treating
  `RegulatedAuthorization.case.identifier` identically maintains
  cross-resource consistency in the Medication Definition module.
- **No alternative considered.** Because the change is widening
  (`0..1` → `0..*`), every existing instance remains valid, so no
  alternative such as introducing a new repeating element was needed.

### Breaking Changes

No breaking changes. The change widens cardinality from `0..1` to
`0..*`. Every existing valid instance (which had at most one
`case.identifier`) remains valid under the new cardinality. Consumers
written against the old schema must be prepared to encounter more than
one `identifier` per `case`, but this is an additive interoperability
expansion, not a backward-incompatible change.

  cardinality change.
- No StructureDefinition profile, search parameter, or operation in the
  HL7/fhir clone narrows or relies on `case.identifier` being a single
  value, so no downstream conformance updates are required.

## Implementation Plan

### Prerequisites

None. The resolution is self-contained, requires no prior terminology or
extension definitions, and has no dependent unresolved tickets.

### Step-by-Step Tasks

#### HL7/fhir

1. **Widen `RegulatedAuthorization.case.identifier` cardinality.**
   - File: `source/regulatedauthorization/structuredefinition-RegulatedAuthorization.xml`
   - Action: In the `<differential>`, locate the element with
     `id="RegulatedAuthorization.case.identifier"` and
     `path="RegulatedAuthorization.case.identifier"` (currently around
     lines 377–391). Change its `<max value="1"/>` to
     `<max value="*"/>`. Leave `<min value="0"/>`, the `Identifier`
     `<type>`, `<short>`, `<definition>`, `<isSummary>`, and the IDMP
     `<mapping>` unchanged.
   - Details: Do **not** touch the corresponding entry in `<snapshot>`
     (if present in your working copy) — the publisher regenerates
     snapshots from the differential. Do not modify any other element
     under `RegulatedAuthorization.case` (e.g., `case.type`,
     `case.status`, `case.date[x]`, `case.application`).

2. **Add ballot-note for the cardinality change to the resource
   introduction.**
   - File: `source/regulatedauthorization/regulatedauthorization-introduction.xml`
   - Action: Immediately after the existing Scope and Usage banner
     paragraph (the bordered "module page" callout) and before the
     "RegulatedAuthorization is a resource covering …" paragraph, insert
     a `<blockquote class="ballot-note" id="bn1">` containing a "Note
     to Balloters" header and a bulleted list whose single sub-bullet
     reads:
     `RegulatedAuthorization.case.identifier, which was made 0..* (previously 0..1)`.
   - Details: Match the structure used for the parallel R6 changes
     (FHIR-55046, FHIR-55057) so the rendered page lists the change
     consistently with sibling resources in the Medication Definition
     module. The block is rendered as-is by the publisher; no
     additional registration in any module-level page is required.

### Testing Considerations

- **Resource validation:** Re-run the publisher's validation over the
  RegulatedAuthorization resource to confirm the differential parses
  and the regenerated snapshot shows `0..*`.
- **Example coverage:** Optionally extend
  `regulatedauthorization-example.xml` (or add a small example) with
  two `case.identifier` entries to demonstrate the new repeatability;
  not required by the resolution, but useful for ballot reviewers.
- **Edge cases:** Confirm that an instance with zero `case.identifier`
  values still validates (`min=0` unchanged) and that two or more
  `Identifier` entries validate.
- **Interoperability:** Implementers parsing
  `RegulatedAuthorization.case.identifier` as a single object will need
  to switch to a list. Note this in ballot communication; the intro
  ballot-note already covers it.
