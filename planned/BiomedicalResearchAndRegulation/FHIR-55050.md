## Resolution Summary

**Resolution:** Not Persuasive with Modification.

The committee did **not** accept the requested change of adding an "unknown"/"other"
escape-valve concept to the All Languages value set. Instead, they accepted that
the underlying use case is real and decided that **making the element optional
better addresses the use requirement**.

Verbatim resolution description:

> Instead of adding an 'escape valve' to the All Languages value set, making the
> element optional better addresses the use requirement, so we will change
> name.usage.language cardinality to 0..1

Concretely: change `MedicinalProductDefinition.name.usage.language` cardinality
from `1..1` to `0..1`. The required binding to `http://hl7.org/fhir/ValueSet/all-languages`
is **not** changed.

## Feature Proposal

### Problem Statement

`MedicinalProductDefinition.name.usage.language` is currently a mandatory element
(`min=1`, `max=1`) with a `required` binding to the All Languages value set
(`http://hl7.org/fhir/ValueSet/all-languages`). Because the value set has no
`unknown` / `other` / equivalent escape-valve concept, there are real-world cases
where a `MedicinalProductDefinition.name.usage` instance cannot be populated at
all — the implementer has nothing valid to put into `language`. This effectively
makes the entire containing `usage` BackboneElement unusable in those cases,
which is a defect against the WG's own usage requirements.

The reporter (Lloyd McKenzie) proposed adding an `unknown`/`other` concept to the
value set as the fix.

### Proposed Change

**Single-element cardinality change in the FHIR Core MedicinalProductDefinition
StructureDefinition:**

- **Element:** `MedicinalProductDefinition.name.usage.language`
- **Path:** `MedicinalProductDefinition.name.usage.language`
- **Cardinality:** change `min` from `1` to `0` (i.e., `1..1` → `0..1`).
- **Type:** unchanged (`CodeableConcept`).
- **Binding:** unchanged — `strength=required`, `valueSet=http://hl7.org/fhir/ValueSet/all-languages`.
  The starter additional binding to `http://hl7.org/fhir/ValueSet/languages` is also
  unchanged.
- **Short / definition / mapping / isSummary:** unchanged.

The All Languages value set itself is **not** modified.

The parent `MedicinalProductDefinition.name.usage` BackboneElement is already
`0..*` (`min=0` at line 888 of the StructureDefinition); no change is needed
there. The sibling `name.usage.country` remains `1..1` and `name.usage.jurisdiction`
remains `0..1`, both unchanged.

After the edit, the differential element will read (only the `<min>` line changes):

```xml
<element id="MedicinalProductDefinition.name.usage.language">
  <path value="MedicinalProductDefinition.name.usage.language"/>
  <short value="Language code for this name"/>
  <definition value="Language code for this name."/>
  <min value="0"/>          <!-- was 1 -->
  <max value="1"/>
  <type>
    <code value="CodeableConcept"/>
  </type>
  <isSummary value="true"/>
  <binding>
    <!-- unchanged -->
  </binding>
  ...
</element>
```

### Design Rationale

The WG explicitly rejected the value-set route. Their reasoning, as captured in
the resolution description, is that an "unknown"/"other" escape valve in
`all-languages` would pollute a value set used widely across the spec (it is the
required binding for `Resource.language`, `Element.language`, `Attachment.language`,
`CodeSystem.language`, etc.) just to support one BackboneElement. Making the
element optional is a strictly local change that:

- Resolves the real use case (a `usage` entry whose language is genuinely unknown
  can simply omit `language`).
- Preserves the strong tag-based identity of the value set for every other
  consumer.
- Aligns with the existing pattern on the sibling `name.usage.jurisdiction`,
  which is already `0..1`.
- Is the minimum change necessary — no code-system or value-set churn, no
  ripple into UTG.

Implementers who currently always populate `language` continue to work
unchanged. Implementers who couldn't previously instantiate `name.usage` at all
gain a path forward by omitting `language`.

### Breaking Changes

This change relaxes a constraint (mandatory → optional). It is therefore:

- **Backward-compatible for producers**: any existing instance that already
  populated `name.usage.language` continues to validate.
- **Forward-incompatible for naive consumers**: code that assumes
  `name.usage.language` is always present (e.g., dereferences it without a
  presence check) may NPE on instances produced post-change. Consumers should
  null-check.
- **FMM / normative considerations**: MedicinalProductDefinition is currently
  trial-use; relaxing a cardinality on a trial-use resource is a permitted
  substantive change. There is no normative-grade compatibility concern blocking
  the edit, but the change should be called out in the resource's change-log /
  release notes per the WG's normal practice.

This is **not** an additive breaking change (no new required element, no type
change, no removed element, no binding-strength tightening, no search-parameter
removal).

### Related Specifications

- **Resource.language**, **Element.language**, **Attachment.language**,
  **CodeSystem.language**, etc., all also bind `required` to
  `http://hl7.org/fhir/ValueSet/all-languages`. They are **not** changed by this
  ticket — the WG explicitly chose the cardinality route precisely to avoid
  touching the shared value set.
- `MedicinalProductDefinition.name.usage` BackboneElement (`0..*`) and its
  siblings `country` (`1..1`) and `jurisdiction` (`0..1`) — unchanged.
- Profiles in IGs that profile `MedicinalProductDefinition.name.usage.language`
