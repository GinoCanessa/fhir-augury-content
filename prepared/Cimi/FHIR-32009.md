## Details

The submitter requests the ability to optionally capture blood pressure cuff size as a **numeric measurement** (e.g., 22–42 cm or inches) in addition to the existing coded terminology concept (e.g., small, medium, large). The motivation comes from real-world experience with BP device manufacturers (Omron, A&D, Withings), who do not consistently identify cuff sizes using standardized discrete size categories. The submitter's current workflow requires users to specify cuff size in centimetres, which is then mapped to terminology concepts via a published reference.

An additional consideration raised: if a numeric cuff size range with units is specified, should the corresponding terminology concept also be mandated?

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Add Optional Quantity Support for Cuff Size Component

**Rationale:** Accept the request to allow numeric cuff size capture, but implement it as an optional Quantity datatype alongside the existing CodeableConcept on the cuff size component of the Blood Pressure Observation. This aligns with the established pattern of cuff size as a component (per FHIR-24519, Lloyd McKenzie's guidance, and the Zulip consensus). The cuff size component would support either a CodeableConcept (for discrete sizes like small/medium/large) or a Quantity with a Range (for numeric values in cm or inches). Do **not** mandate that a coded concept must accompany a numeric value, since the purpose of this change is to accommodate devices that do not use standardized discrete sizes.

**Impact:** Requires updating the Blood Pressure profile in US Vital Signs with Qualifying Extensions to allow Quantity or Range as an alternative value type for the cuff size component. Value set bindings for coded cuff sizes remain unchanged.

### Disposition B: Not Persuasive with Modification — Recommend Mapping to Coded Values Only

**Rationale:** Acknowledge the real-world problem but decline to add a numeric representation to the profile. Instead, provide implementation guidance recommending that systems map numeric cuff sizes to the existing coded terminology (e.g., by using published manufacturer reference tables). This preserves semantic interoperability and avoids introducing a dual representation that receivers would need to interpret. Add a note in the profile explaining how numeric values from devices should be mapped to the existing value set concepts.

**Impact:** No structural change to the profile. New implementation guidance text added to the Blood Pressure profile narrative.

### Disposition C: Not Persuasive — Decline the Request

**Rationale:** The existing coded value set for cuff size (small, medium, large, etc.) is sufficient for clinical documentation. Physician survey data (FHIR-29282) rated cuff size as "less important" overall. Adding numeric cuff size dimensions introduces complexity without clear clinical benefit, as the coded size categories are what clinicians act upon. Numeric device-specific dimensions are a device-level concern better handled in the Device resource or outside the clinical observation.

**Impact:** No change to the specification.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The submitter raises a genuine interoperability problem: BP device manufacturers do not consistently map to standardized cuff size categories, and real-world implementations already require numeric-to-coded mapping workflows. Adding optional Quantity support to the cuff size component provides a standards-based path for systems that capture raw device measurements, while the existing CodeableConcept path remains available for systems that use discrete size categories. This approach is consistent with the established consensus (FHIR-24519, Zulip discussion) that cuff size belongs as a component, and it follows the FHIR principle of supporting both coded and measured data where clinically appropriate. Mandating a coded equivalent alongside a numeric value should **not** be required, preserving flexibility for implementers.
