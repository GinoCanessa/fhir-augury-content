## Summary

The submitter requests that when a ValueSet expansion does not contain a complete expansion, the `SUBSETTED` meta tag should be set on the resource instance. This is especially relevant for bundled ValueSet resources (e.g., in IG packages) where an application lacks the context of a specific terminology server call and cannot otherwise determine whether the expansion is partial.

After extended discussion, the work group resolved this as **Persuasive with Modification**. Rather than requiring the `SUBSETTED` tag, the resolution leverages the existing `ValueSet.expansion.total` element and paging parameters (`offset`, `count`) as the mechanism for clients to determine whether an expansion is complete. The proposed changes update the `$expand` operation documentation and the `ValueSet.expansion.total` element definition.

However, in the most recent discussion (2026-04-06), the work group noted important distinctions between **incomplete** and **unclosed** expansions, concluded that the `valueset-unclosed` extension is not the right mechanism for paging scenarios, and flagged that the `total` element may not be populated for unclosed value sets — requiring further consideration.

---

## Details

### Original Request

When a ValueSet expansion is not complete, the `SUBSETTED` tag should be applied so consumers know the expansion is partial. The concern is most acute for static/bundled expansions in IG packages where there is no live terminology server interaction.

### Resolution Description (as currently recorded)

> There is already an established mechanism for determining whether a given `ValueSet.expansion` element is a page: the `total` element is empty, or is greater than `contains.count()`.
>
> **Change 1** — Update the `$expand` operation documentation:
>
> _Current:_ "Servers are not obliged to support paging, but if they do, SHALL support both the offset and count parameters."
>
> _Proposed:_ "Servers are not obliged to support paging, but if they do, SHALL support both the offset and count parameters, and SHOULD provide the total element to indicate the total size of the expansion. If no `total` element is present, clients SHALL consider the result a page."
>
> **Change 2** — Add to the `ValueSet.expansion.total` element definition:
>
> "The `total` element SHOULD be present in an expansion. If it is not present, the expansion SHALL be treated as one page in a paged expansion."

### Key Discussion Points

- **Michael Lawley** asked for clarification on what "incomplete" means — paged results include `offset`/`count` parameters and are not truly `SUBSETTED` (2025-01-16).
- **January 2025 WGM** identified the `valueset-unclosed` extension as the existing mechanism (2025-01-29).
- **Submitter** (Gino Canessa) agreed any mechanism is acceptable as long as it is documented (2025-09-29).
- **September 2025** work group discussion discovered `valueset-unclosed` was incorrectly deprecated. A new ticket ([FHIR-53021](https://jira.hl7.org/browse/FHIR-53021)) was filed to un-deprecate it.
- **Bryn Rhodes** confirmed with implementers on Zulip that clients determine paged results by comparing `total` to `contains.count()` (2025-10-07).
- **Michael Lawley** cautioned that calculating `total` can be impossible (infinite), very hard, or very expensive (2025-10-07).
- **April 2026** discussion noted the difference between incomplete and unclosed expansions, that `valueset-unclosed` is not the right indicator for paging, and that `total` may not be populated for unclosed value sets — further options to be considered at a future meeting.

## Proposed Dispositions

### Disposition A: Accept (Persuasive with Modification — as resolved)

**Action:** Apply the resolution as written — update the `$expand` documentation and `ValueSet.expansion.total` element definition to clarify that servers SHOULD populate `total`, and if absent, clients SHALL treat the expansion as a page.

**Rationale:** The resolution addresses the submitter's core concern (no mechanism to detect incomplete expansions) by leveraging existing elements rather than introducing new tags. The submitter agreed this approach is acceptable. It formalizes existing implementation behavior confirmed by Grahame Grieve.

**Risk:** The April 2026 discussion identified that "clients SHALL consider the result a page" when `total` is absent may be too strong — it conflates *incomplete* expansions with *unclosed* expansions. Michael Lawley's Zulip input confirms that computing `total` can be impossible, so this rule could make many legitimate complete expansions appear paged.

---

### Disposition B: Refine the resolution to distinguish incomplete, paged, and unclosed expansions

**Action:** Accept the spirit of the resolution but revise the language to:
1. Keep the SHOULD for servers to populate `total` when paging.
2. Soften "clients SHALL consider the result a page" to "clients SHOULD NOT assume the expansion is complete" when `total` is absent.
3. Add guidance distinguishing three states: (a) **paged** (offset/count present, total available), (b) **unclosed** (expansion inherently incomplete, use `valueset-unclosed` extension), and (c) **complete** (total present and equals `contains.count()`, offset 0 or absent).
4. Coordinate with FHIR-53021 to ensure `valueset-unclosed` is un-deprecated.

**Rationale:** This addresses the concerns raised in the April 2026 meeting about the difference between incomplete and unclosed expansions. It avoids over-constraining servers that cannot compute `total` while still giving clients clear guidance. It also aligns with the R6 `ValueSet.expansion.next` element direction.

**Risk:** More complex specification language; requires coordination across multiple tickets and spec sections.

---

### Disposition C: Decline (Not Persuasive)

**Action:** Close the ticket. The existing mechanisms (`offset`/`count` parameters, `total` element, `valueset-unclosed` extension, and R6 `next` element) already provide sufficient information for clients to determine expansion completeness.

**Rationale:** The specification already offers multiple mechanisms. Adding more conformance language risks contradicting the practical reality that `total` cannot always be computed. The submitter's specific use case (IG packages with static expansions) could be addressed by IG authoring guidance rather than core spec changes.

**Risk:** Leaves the documentation gap that the submitter identified — implementers may not discover the existing mechanisms without explicit guidance.

---

## Recommendation

**Disposition B: Refine the resolution** is recommended.

The April 2026 work group discussion already identified that the current resolution language needs adjustment, distinguishing incomplete from unclosed expansions. The Zulip discussions confirm real implementer concerns about `total` being impossible to compute in some cases, making the SHALL language in the current resolution potentially problematic. Disposition B preserves the submitter's goal (clear documentation of mechanisms) while incorporating the nuances raised by the work group and community feedback. It also aligns with the parallel effort to un-deprecate `valueset-unclosed` (FHIR-53021).
