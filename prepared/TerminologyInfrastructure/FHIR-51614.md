## Problem Statement

The reporter identifies a usability gap in the FHIR spec regarding designations used for language translations. Both the CodeSystem and ValueSet resources contain nearly identical text describing how designations work, but there is a critical practical difference that is not clearly stated:

- **CodeSystem supplements** are the correct mechanism for adding language translations to 3rd-party code systems.
- **ValueSet designations** serve a different, more limited purpose — they assign alternative display text for use only within the context of a value set expansion, not for general-purpose translation.

The current spec text does not make this distinction clear, creating a "footgun" where implementers mistakenly add language designations to ValueSets instead of using CodeSystem supplements. The reporter had to consult Zulip to learn the correct approach.

Additionally, the reporter suggests the IG Publisher should emit a warning when language designations are added to ValueSets, since that pattern is almost always incorrect.

This earlier thread provided the foundational discussion. Key points:

- Vadim asked whether translations should go in a CodeSystem supplement or in ValueSet designations.
- **Elliot Silver** initially suggested the spec says only code system publishers should define supplements.
- **Grahame Grieve** clarified: supplements should not be defined *against* the code system publisher's policies, but there is no prohibition on 3rd parties creating supplements. He confirmed supplements are the correct approach.
- Grahame explained that **ValueSet designations** can assign a different display *for use in the context of the expansion*, but not otherwise (e.g., not in Codings).
- Topic was marked resolved; Vadim confirmed he would go with CodeSystem supplements.

## Proposed Dispositions

### Disposition A: Persuasive — Clarify Spec Text on Both Pages

**Action:** Accept the request. Update the designation guidance text on both the [CodeSystem](http://hl7.org/fhir/codesystem.html#designations) and [ValueSet](http://hl7.org/fhir/valueset.html#designations) pages to clearly distinguish their roles:

1. On the **CodeSystem designations** section, add explicit text stating that language translations for code systems (including 3rd-party systems) should be provided via **CodeSystem supplements** using designations with appropriate language tags and the `preferredForLanguage` use code.
2. On the **ValueSet designations** section, add a note clarifying that ValueSet designations are for assigning alternative display text *within the context of the value set expansion only*, and that language translations for a code system's concepts should be done via CodeSystem supplements, not ValueSet designations.
3. Reference the designation use code `http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra#preferredForLanguage` as suggested in the Jira comment.
4. Coordinate with FHIR-54129 (CodeSystem text technical corrections) to avoid conflicting edits.

**Rationale:** The community consensus (Grahame Grieve, Vadim, mjlawley) agrees the spec is unclear. This is a text-only change (TEXTONLY label) with no structural impact. The distinction is well-established in practice but poorly documented.

### Disposition B: Persuasive with Modification — Clarify Spec Text Only (No IG Publisher Warning)

**Action:** Accept the spec text clarification portion only (same as Disposition A items 1–4), but decline the request for an IG Publisher warning about language designations on ValueSets.

**Rationale:** As Grahame noted, it is *"not wrong to put them in the value set, just a much restricted purpose."* Adding a validator warning for something that is technically valid could produce false positives and confuse implementers who have legitimate reasons to use ValueSet designations with language tags (e.g., context-specific display overrides). The IG Publisher change would also fall outside the scope of FHIR Core spec text and would need a separate request to the tooling team.

### Disposition C: Not Persuasive — Existing Text Is Sufficient

**Action:** Decline the request. The existing text, combined with the broader CodeSystem supplements documentation, already provides the necessary guidance for implementers.

**Rationale:** The distinction between CodeSystem supplements and ValueSet designations is documented elsewhere in the spec (CodeSystem supplements section). Implementers are expected to read the specification holistically. Adding redundant guidance risks creating maintenance burden and potential inconsistencies, especially given the active changes in FHIR-54129.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended.

The core request — clearer spec text distinguishing CodeSystem supplement designations from ValueSet designations — has strong community support and addresses a real usability gap. The TEXTONLY label makes this low-risk. However, the IG Publisher warning should be out of scope for this ticket: Grahame Grieve explicitly noted that ValueSet language designations are not wrong, just limited in purpose. The text clarification alone will substantially improve the implementer experience. The `preferredForLanguage` code reference (per mjlawley's comment) adds concrete, actionable guidance. Coordination with FHIR-54129 is advisable to avoid conflicting edits to the CodeSystem page.
