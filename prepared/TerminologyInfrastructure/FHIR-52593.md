## Summary

The submitter requests that **"Deprecated"** be added as a distinct artifact status in the CRMI artifact lifecycle, separate from **"Retired"**. The rationale is that VSAC supports both Retired and Deprecated statuses, "Deprecated" has been used historically in C-CDA, and CRMI itself already uses "Deprecated" as a standards-status on some of its own IG artifacts.

**Proposed definition of Deprecated:**
- **Definition:** Still available/valid, but discouraged for use.
- **Purpose:** Signals to implementers that a better, newer, or safer alternative exists, and they should migrate away.
- **Status:** The feature/standard continues to exist and usually still works for backward compatibility.

**Resolution description (from ticket):** The base FHIR lifecycle statuses for artifacts cannot be modified, but the need can be met using the `valueset-workflowStatusDescription` extension. This extension will be added to the ShareableValueSet pattern and its use for describing "deprecated" and "retired" status will be documented.

## Ticket Comments Summary

**Bryn Rhodes** (5 comments, Sep–Oct 2025) explored several angles:
1. **Base FHIR constraint**: Adding "Deprecated" to PublicationStatus would require changes to all canonical resources in FHIR (an R6 base specification change). The current "retired" status ("should not be used") arguably subsumes "deprecated."
2. **Extension approach**: CRMI already uses the `standards-status` extension to mark features as deprecated. A separate extension or flag for deprecation may be the right approach.
3. **Alternative proposals**: Suggested a "releaseStatus" extension (values: pre-release, release, deprecated), or a "lifecycleStatus" extension. Referenced `template-status-code` and `standards-status` value sets from the FHIR extensions pack.
4. **Clarification requested**: Asked the submitter to clarify the precise distinction between "deprecated" and "retired."

## Proposed Dispositions

### Disposition A: Persuasive with Modification (Accept Current Resolution)
**Rationale:** Accept the resolution as stated. The `valueset-workflowStatusDescription` extension provides an existing, standards-compliant mechanism to convey "deprecated" semantics for value sets without requiring changes to base FHIR PublicationStatus. Adding this extension to the ShareableValueSet profile and documenting its use for "deprecated" vs. "retired" addresses the submitter's need pragmatically. This approach avoids the breaking change that was rejected in FHIR-31028 and works within the existing FHIR extension framework.

**Impact:** CRMI IG changes only (add extension to ShareableValueSet pattern, add documentation). No base FHIR changes required.

### Disposition B: Persuasive with Modification (Broader Lifecycle Extension)
**Rationale:** Go further than the current resolution by defining a formal CRMI lifecycle status extension (not limited to value sets) that supports "deprecated" alongside other statuses like "pre-release" and "release." This addresses Bryn Rhodes' suggestion of a "releaseStatus" or "lifecycleStatus" extension and would apply to all CRMI-conformant canonical resources, not just value sets. This would better align with the fact that THO already operationally uses "deprecated" for code systems and value sets alike.

**Impact:** More extensive CRMI IG changes. Requires defining a new extension, code system, and value set. Potentially affects all CRMI shareable resource profiles.

### Disposition C: Not Persuasive
**Rationale:** The existing "retired" status in PublicationStatus already covers the concept of "should not be used." Adding a distinct "deprecated" status—even via extension—risks fragmenting implementer understanding of artifact lifecycle. The validator already treats deprecated artifacts as retired (per Grahame Grieve's guidance). The prior attempt to add "Deprecated" to PublicationStatus was rejected (FHIR-31028), and the semantic boundary between "deprecated" and "retired" remains unclear, as Bryn Rhodes noted in the comments.

**Impact:** No changes required. Implementers continue using "retired" for both deprecated and retired artifacts.

---

## Recommendation

**→ Disposition A: Persuasive with Modification (Accept Current Resolution)**

This is the recommended disposition. The resolution pragmatically addresses a real operational need—VSAC, C-CDA, and THO all recognize "deprecated" as a distinct status—without requiring the base FHIR PublicationStatus changes that were previously rejected in FHIR-31028. Using `valueset-workflowStatusDescription` leverages an existing extension mechanism and scopes the change to CRMI documentation and the ShareableValueSet profile. The Zulip evidence (particularly the validator warning reported by Eric Haas) confirms that implementers are encountering this gap in practice. The workgroup should ensure the documentation clearly defines the distinction between "deprecated" (still valid, use discouraged) and "retired" (should not be used), which was the clarification Bryn Rhodes requested.
