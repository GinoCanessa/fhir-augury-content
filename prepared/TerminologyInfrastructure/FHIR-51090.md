## Description

The `ValueSet/$expand` and `ValueSet/$validate-code` operations have a `system-version` parameter for specifying the version of a code system used in the operation. They also both have a `default-valueset-version` parameter, which serves the same purpose but for value sets. The `default-valueset-version` naming is clearer, so the proposal is to:

1. Add `default-system-version`, identical to `system-version`
2. Deprecate `system-version`

in both operations.

## Discussion History

### Tracker Call — 2025-12-15 (carmela_couderc)

Discussed on the 2025-12-15 tracker call. Key points:
- **Concern** about having two ways to declare the code system version (deprecated `system-version` plus new `default-system-version`).
- Attendees were **leaning towards rejecting** the deprecate-and-add proposal.
- Noted the confusing similarity between `systemVersion` (CamelCase, a different parameter) and `system-version` (kebab-case).
- Proposed **alternative**: update the *description* of `system-version` to include the word "default" rather than creating a deprecated/active scenario.
- Proposed updated description: *"Specifies a default code system version to use for the operation, IF the value set definition does not specify a code system version."*
- Michael Lawley was asked for input on implementation impact.
- Ticket was slated for the joint TI/FHIR-I session in January 2026.

### Michael Lawley — 2026-02-25

- Ontoserver (R4) has pre-adopted support for all parameter name variations.
- Requires that if multiple "equivalent" (renamed) parameters are supplied, they must have the same value — no overriding.
- Prefers a **hybrid rewording**: *"Specifies a default code system version to use for the operation, IF the value set definition itself does not specify which code system version to use. [...] Note that this is a different parameter to systemVersion."*
- Clarified that `systemVersion` is the version of the CodeSystem referenced in the `system` (IN) parameter, whereas `system-version` applies to CodeSystems referenced within the ValueSet itself.
- Ontoserver uses `system-version` as a default system version fallback if an explicit `default-system-version` has not been specified.

### Bryn Rhodes — 2026-02-25

Agreed that the hybrid wording proposed by Michael Lawley is better.

## Analysis

### Core Issue
The `system-version` parameter name does not clearly communicate its "default" semantics. Implementers may confuse it with `systemVersion` (a different, CamelCase parameter that specifies the version of the CodeSystem in the `system` IN parameter). The `default-valueset-version` naming already established a clearer pattern for the value set equivalent.

### Discussion Evolution
The proposal has evolved significantly through discussion:
1. **Original proposal** (Bryn Rhodes, June 2025): Add `default-system-version` and deprecate `system-version`.
2. **Tracker call** (December 2025): Attendees leaned toward rejecting the deprecation approach; proposed instead to update the existing description.
3. **Implementer feedback** (Michael Lawley, February 2026): Supports hybrid approach — improve the description wording while noting the distinction from `systemVersion`. Ontoserver already supports all variants.
4. **Reporter agreement** (Bryn Rhodes, February 2026): Accepts the hybrid wording.

### Key Consideration
CRMI 2.0.0 already uses `default-system-version` in its manifest parameter naming (per FHIR-51734 and FHIR-51088). If FHIR Core does not adopt `default-system-version`, there will be a naming mismatch between CRMI and FHIR Core for the same semantic concept.

## Proposed Dispositions

### Disposition A: Persuasive — Update Description Only (No Rename)

**Action:** Update the description of `system-version` in `ValueSet/$expand` and `ValueSet/$validate-code` to use Michael Lawley's hybrid wording that includes the word "default" and clarifies the distinction from `systemVersion`. Do not add `default-system-version`. Do not deprecate `system-version`.

**Updated description:** *"Specifies a default code system version to use for the operation, IF the value set definition itself does not specify which code system version to use. The format is the same as a canonical URL: [system]|[version] - e.g. http://loinc.org|2.56. Note that this is a different parameter to systemVersion."*

**Rationale:** This was the direction favored by tracker call attendees and subsequently endorsed by both Michael Lawley and Bryn Rhodes. It clarifies semantics without introducing a deprecation cycle or parameter duplication. It avoids backward compatibility concerns while addressing the naming confusion.

**Risk:** Creates a divergence from CRMI 2.0.0's `default-system-version` naming convention.

### Disposition B: Persuasive — Add default-system-version and Deprecate system-version (Original Proposal)

**Action:** Add `default-system-version` as a new parameter identical to `system-version` in both `ValueSet/$expand` and `ValueSet/$validate-code`. Mark `system-version` as deprecated. Apply the improved description to the new parameter.

**Rationale:** This is the original proposal and aligns with CRMI 2.0.0 naming conventions (`default-system-version`, `default-valueset-version`, `default-canonical-version`). Provides long-term naming consistency across the ecosystem.

**Risk:** Introduces a deprecation cycle and parameter duplication, which was the primary concern raised on the December 2025 tracker call. Implementers must support both parameter names during the transition period.

### Disposition C: Not Persuasive — Reject

**Action:** No change. Leave the `system-version` parameter name and description as-is.

**Rationale:** The current parameter name and description are functional and widely implemented. The confusion between `system-version` and `systemVersion` can be addressed through documentation or education rather than specification changes.

**Risk:** Ongoing implementer confusion between `system-version` and `systemVersion`. Missed opportunity to improve clarity. Growing divergence from CRMI 2.0.0 naming conventions.

---

## Recommendation

**Disposition A: Persuasive — Update Description Only**

This disposition best reflects the consensus that emerged through the discussion. The tracker call attendees, the primary implementer (Michael Lawley/Ontoserver), and the original reporter (Bryn Rhodes) all converged on the hybrid wording approach as preferable to a full deprecation cycle. It delivers the core benefit — clarifying the "default" semantics and the distinction from `systemVersion` — without the backward compatibility concerns of introducing a new parameter name. The CRMI naming divergence is a consideration but is manageable given that CRMI already documents the mapping between CRMI 1.0.0 and 2.0.0 parameter names (per FHIR-51624).
