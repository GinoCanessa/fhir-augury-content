## Summary

The VA VHIE Clinical Data Quality Team (CDQT) requests that terminology servers be able to declare their support for implicit value set URL patterns. Currently, implicit value sets (e.g., those defined by code systems like SNOMED CT) exist as URI patterns but there is no standardized mechanism in the FHIR specification for a terminology server to advertise which implicit value set patterns it supports. The submitter argues that clients — both human users and automated systems — need a way to discover whether a given terminology server can expand a particular implicit value set.

## Details

**Background:** Implicit value sets are value sets defined by URL patterns associated with code systems (e.g., SNOMED CT implicit value sets for reference sets, ECL expressions, etc.). The FHIR spec documents these patterns but does not provide a standard way for servers to declare support for them.

**Discussion History (10 comments, 2023–2024):**

- **2023-08-21:** TI tracker call acknowledged the request as reasonable. Noted that tx.fhir.org already supplies this information, but not in a capabilities statement. Asked the submitter for use cases and clarified three key questions: (1) Does the server support implicit value sets from any code system? (2) Does it support them for specific code systems? (3) What are the URL patterns?
- **2023-10-02 / 2024-01-22 / 2024-02-12:** Multiple follow-ups requesting the submitter to provide additional detail.
- **2024-07-22:** TI tracker call identified that implementing this would require something in **TerminologyCapabilities**. The challenge is that implicit value sets have only a URI pattern, not a formal identifier — so a scheme to create identifiers for each code system's implicit value set patterns would need to be documented first.
- **2024-07-23 (Michael Lawley):** Asked whether this is for human consumption or automated use, and how automated clients would use the information. Also asked about relevance to the Tx Registry.
- **2024-07-24 (Peter Jordan):** Questioned what this adds beyond `CodeSystem.valueSet`. Noted that `CodeSystem.valueSet` isn't guaranteed to be an implicit value set and cannot hold multiple patterns.
- **2024-07-25:** External recommendation to use the TerminologyCapabilities resource, defined tightly in an IG, as the vehicle for this.
- **2024-07-29:** Carmela summarized that the change serves both human and automated use. A human could determine server capability for expanding a value set they are defining; an automated TS client could discover servers that support a particular implicit value set pattern. Also useful for the Tx Registry.
- **2024-09-26 (Joint FHIR-I/TI):** Agreed there is a place for this feature but it doesn't exist yet. Need formal identifiers for implicit value set patterns (currently only URL patterns exist). Noted this should not be server-specific.

**Key Technical Challenge:** Implicit value sets are identified by URI patterns, not stable identifiers. To declare support in TerminologyCapabilities, a formal identification scheme for implicit value set patterns would need to be created first.

**Cross-Reference:** FHIR-39080 mentions `CodeSystem.valueSet` — Peter Jordan noted this element is insufficient because it is not guaranteed to reference an implicit value set and can only hold a single URL.

## Proposed Dispositions

### Disposition A: Accept — Add implicit value set support declaration to TerminologyCapabilities (Persuasive with Modification)

**Resolution:** Persuasive with Modification

**Action:** Add a new element to `TerminologyCapabilities.codeSystem` (or a new backbone element) that allows a server to declare which implicit value set URL patterns it supports for each code system it hosts. This would involve:

1. Define a new repeating element (e.g., `TerminologyCapabilities.codeSystem.implicitValueSet`) that holds the URI pattern(s) for implicit value sets the server supports for a given code system.
2. Document how implicit value set patterns are identified (since they are URI-pattern-based, not single-URI identifiers).
3. This information would be useful both for human discovery and automated client routing (e.g., Tx Registry integration).

**Rationale:** The workgroup has acknowledged this is a reasonable request with a clear gap in the spec. TerminologyCapabilities is the appropriate resource. The modification addresses the need for a formal identification approach rather than the originally vague request.

### Disposition B: Defer to R7 — Needs prerequisite identifier scheme (Deferred)

**Resolution:** Deferred

**Action:** Defer this change to a future release. The September 2024 joint FHIR-I/TI session identified that implicit value sets currently only have URI patterns, not formal identifiers. Before this feature can be implemented in TerminologyCapabilities, a formal identification scheme for implicit value set patterns needs to be developed. Track this as a future R7 candidate once the identifier scheme work is completed.

**Rationale:** While the request is valid, the technical prerequisite (a formal identification scheme for implicit value set patterns) does not yet exist. Attempting to add this to TerminologyCapabilities without solving the identifier problem first could lead to an inconsistent or unimplementable design. Deferring allows time to develop the identifier scheme properly.

### Disposition C: Decline — Out of scope for the specification (Not Persuasive)

**Resolution:** Not Persuasive

**Action:** No change to the specification. Note that tx.fhir.org already provides this information informally, and individual servers/registries can advertise capabilities through implementation-specific mechanisms. The submitter did not provide detailed use cases despite multiple requests over 18+ months.

**Rationale:** The original request lacked sufficient specificity despite repeated requests for use cases. `CodeSystem.valueSet` already provides some of this information. Server-specific capability advertising can be handled through extensions or the Tx Registry without changes to the core spec. The implicit value set URI pattern identification challenge suggests this may be premature.

---

## Recommendation

**Disposition B: Defer to R7** is recommended.

The workgroup has established through multiple discussions (2023–2024) that this is a valid gap in the spec. The natural home for this feature is `TerminologyCapabilities`. However, the September 2024 joint session clearly identified a blocking prerequisite: implicit value sets are identified by URI patterns, not formal identifiers, and a proper identification scheme must be developed before TerminologyCapabilities can meaningfully declare support for them. Accepting now (Disposition A) risks premature design; declining (Disposition C) would dismiss a legitimate need acknowledged by TI members. Deferring allows the prerequisite identifier work to be completed while preserving the intent of the request.
