## Description

The ticket requests adding a reference to a new `expansion-parameters` extension (to be defined in the extensions pack) in the ManifestParameters discussion and the ManifestLibrary profile in the CRMI IG. This extension would indicate **"the parameters that were used to expand value sets in the IG"** — recording the actual parameters applied during publication.

The submitter distinguishes three related but distinct extension concepts:

1. **`cqf-expansionParameters`** — Specifies the expansion parameters that **should** be used to expand value sets in an IG (prescriptive/default).
2. **`expansion-parameters`** (new, not yet defined) — Records the parameters that **were** used to expand value sets during publication (descriptive/as-published).
3. **`cqf-inputParameters`** — Identifies the input expansion parameters used as input to the release/publication process.

## Resolution Details

The resolution (Persuasive) agrees to update the language in the [Canonical Resolution with a Manifest](https://build.fhir.org/ig/HL7/crmi-ig/version-manifest.html#canonical-resolution-with-a-manifest) section of the CRMI IG.

**Current text:**
> In addition, if the operation specifically involves value set expansion, the expansionParameters extension of the manifest library can be used to provide values for the parameters to the expand (such as activeOnly and system-version).

**Proposed new text:**
> In addition, if the operation specifically involves value set expansion, the cqf-expansionParameters extension of the manifest library SHOULD be used to provide values for the parameters to the expand (such as activeOnly and default-system-version). If no cqf-expansionParameters is present, expansion-parameters MAY be used to provide the "as-published" expansion parameters.
>
> Note: cqf-expansionParameters defines default values that SHOULD be used for $expand input parameters. expansion-parameters defines the parameter values that were used during the publication expansion process.

## Analysis

### Key Observations

1. **The `expansion-parameters` extension does not yet exist.** Both the workgroup comment (carmela_couderc, 2026-02-23) and Zulip discussions confirm this. Grahame Grieve acknowledged the definition was missing from the tools IG 1.0.0 release and indicated a fix was pending.

2. **The proposed text change is text-only (TEXTONLY label)** — it updates narrative guidance in the CRMI IG, not structural definitions. However, the guidance references an extension that does not yet exist, creating a forward dependency.

3. **Three distinct extension concepts are in play**, and the community has been actively working to consolidate around `cqf-expansionParameters` as the standard prescriptive extension (see FHIR-52764, FHIR-49700, FHIR-49781). This ticket adds the descriptive/historical counterpart.

4. **The distinction between "should use" and "were used" is semantically important** for tooling — `cqf-expansionParameters` provides input defaults, while `expansion-parameters` records what actually happened during publication. This supports reproducibility and auditability.

5. **Closely related ticket FHIR-51089** (same work group) proposes allowing manifest parameters directly in `$expand` and `$validate-code` operations, which shares the same underlying problem space.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (with dependency caveat)
Accept the resolution as written. Update the CRMI IG text to reference both `cqf-expansionParameters` (SHOULD) and `expansion-parameters` (MAY). Add an informative note that the `expansion-parameters` extension definition must be published before this guidance becomes actionable. This approach approves the semantic distinction and normative language now, trusting that the extension will be defined shortly.

### Disposition B: Defer Until Extension Exists (Deferred)
Defer this ticket until the `expansion-parameters` extension is formally defined in the tools IG or extensions pack. The workgroup comment and Zulip threads confirm this extension has not been published despite being referenced by the IG publisher tooling. Deferral avoids publishing guidance that references a non-existent artifact. Revisit once the extension definition is available.

### Disposition C: Accept as Persuasive with Modification — Remove expansion-parameters reference
Accept the first part of the text change (clarifying `cqf-expansionParameters` as the SHOULD-level extension with updated parameter names like `default-system-version`) but remove the `expansion-parameters` MAY clause entirely until that extension is defined. This delivers the cqf-expansionParameters clarification immediately without creating a forward reference to an undefined extension.

## Recommendation

**Disposition B: Defer Until Extension Exists.** The core value of this ticket is codifying the distinction between prescriptive (`cqf-expansionParameters`) and descriptive (`expansion-parameters`) expansion parameter extensions. However, the `expansion-parameters` extension does not yet exist — confirmed by both the workgroup comment and multiple Zulip threads as of late February 2026. Publishing guidance that references an undefined extension would create confusion and QA errors (as already seen in the IG publisher). Deferring ensures the text change is actionable when published. The workgroup should track the extension definition in the tools IG and reactivate this ticket once it is available.
