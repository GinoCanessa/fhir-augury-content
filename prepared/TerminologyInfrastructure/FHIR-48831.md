# FHIR-48831: Properly clarify ValueSet filter.value values

## Summary

**Ticket:** [FHIR-48831](https://jira.hl7.org/browse/FHIR-48831)
**Status:** Triaged | **Type:** Change Request | **Priority:** Medium
**Resolution:** Unresolved | **Assignee:** Unassigned
**Reporter:** Michael Lawley
**Work Group:** Terminology Infrastructure | **Specification:** FHIR Core (FHIR)
**Created:** 2024-11-14 | **Updated:** 2026-01-20

Michael Lawley reports that the prior clarification from FHIR-38104 did not properly address the set of valid values for `ValueSet.compose.include.filter.value`. The current spec text states that for most filter operations, the value "represents a code," but properties can be of many types (code, Coding, string, integer, boolean, dateTime, decimal). This means the only way to do equality matching on non-code types is to use the "regex" operator, which is unintuitive. The ticket requests three changes:

1. Clarify that for the "=" operator, the value can be the string representation of any supported property type (not just code).
2. Define how a Coding value should be represented in string form — suggesting token syntax (`[system]|[code]`).
3. Adopt search-style prefix syntax for numeric inequality testing (e.g., `"value": "lt10"` for "less than 10").

**Reporter's latest position (Dec 2025):** Michael Lawley states this issue can now be closed, as it has since been resolved by a different ticket (identified as FHIR-51117, with FHIR-44866 also covering related ground).

## Zulip Conversations

### [terminology] Defining Relationships Between Codes from Different CodeSyst
**URL:** https://chat.fhir.org/#narrow/stream/terminology/topic/Defining%20Relationships%20Between%20Codes%20from%20Different%20CodeSyst
**Period:** 2024-11-12 to 2025-07-09 | **Participants:** Tayeb merabti, Rutt Lindström, Michael Lawley, Gino Canessa, Elliot Silver, Grahame Grieve, Abdelali Boussadi

This thread originated from a question about filtering on Coding-typed properties in ValueSets and directly led to the creation of FHIR-48831. Key points:

- **Real-world problem:** Users filtering on Coding-typed properties encounter validator errors (`VALUESET_BAD_FILTER_VALUE_CODED_INVALID`) when using `system|code` syntax (token syntax). The filter works for `$expand` but fails `$validate`.
- **Syntax disagreement:** Michael Lawley advocated for token syntax (`system|code`), while Grahame Grieve disclosed the validator uses canonical-style syntax (`system(|version)#code`). Grahame acknowledged this syntax is not documented in the spec.
- **Resolution direction:** Grahame acknowledged the documentation gap and posted a [build branch for FHIR-44866](https://build.fhir.org/branches/2025-07-gg-fhir-44866/valueset.html#filter-values) to address the syntax definition. Michael Lawley proposed that `system|code` be deprecated in favor of `system#code` and `system|version#code` to avoid ambiguity with the pipe character's use in canonical URLs.
- **Validator impact:** Abdelali Boussadi raised that the validator behavior needs updating once the spec is clarified; Grahame assigned Reuben Daniels to track this.

## Proposed Dispositions

### Disposition A: Close as Duplicate / Superseded (Recommended)
**Action:** Close FHIR-48831 as resolved/superseded by FHIR-51117 and FHIR-44866.

**Rationale:** The reporter (Michael Lawley) himself stated in December 2025 that "this issue can be closed. It has since been resolved as a result of a different ticket." FHIR-51117 (Applied, Persuasive with Modification) has already addressed the filter.value definition for the 'in' operation and ValueSet canonicals. FHIR-44866 (Waiting for Input, Persuasive) is actively being worked on by Grahame Grieve to define the complete syntax for all property data types in filter.value, including the Coding representation format — which directly covers the remaining concerns in FHIR-48831. Closing this ticket avoids duplicate tracking and consolidates the work under FHIR-44866 where the syntax decisions are being actively made.

### Disposition B: Persuasive — Resolve Independently
**Action:** Accept the ticket and apply changes to clarify filter.value syntax for all property types.

**Rationale:** Although FHIR-51117 and FHIR-44866 overlap, FHIR-48831 raises a specific point about adopting search-style prefix syntax for numeric inequality (e.g., `"lt10"`), which is not explicitly addressed by either sibling ticket. Resolving this independently would ensure the numeric comparison use case is formally specified. However, this risks fragmenting the design decisions across multiple tickets.

### Disposition C: Defer to FHIR-44866
**Action:** Mark as "Waiting for Input" or "Deferred" pending completion of FHIR-44866.

**Rationale:** FHIR-44866 is the most comprehensive ticket covering this space and has active work underway (a build branch exists). Rather than closing FHIR-48831 outright, the workgroup could keep it open as a tracking reference until FHIR-44866 is fully applied, then verify that all concerns from FHIR-48831 are addressed before closing. This is more cautious but adds administrative overhead.

## Recommendation

**Disposition A: Close as Duplicate / Superseded.** The reporter confirms the issue is resolved. Both FHIR-51117 (applied) and FHIR-44866 (in progress with a build branch) substantively cover the concerns raised. The specific numeric inequality syntax point from FHIR-48831 can be tracked as a comment on FHIR-44866 if the workgroup considers it valuable. Closing this ticket reduces redundancy and consolidates design decisions under the tickets where active work is happening.
