## Description

In practice, HL7 (in both the core FHIR spec and THO) deprecates or retires concepts using the `structuredefinition-standards-status` extension, but there is no documentation stating that this extension provides a concept status value in the absence of an explicit `status` concept property.

The submitter proposes either:
1. Documenting that the `standards-status` extension provides a status value when no explicit `status` concept property is present, **or**
2. Explicitly documenting that it is an error to have a standards-status of "deprecated" etc. without an explicit `status` property, and fixing up existing examples that violate this rule.

## Analysis

### Core Issue
HL7 uses the `structuredefinition-standards-status` extension on CodeSystem concepts to indicate deprecation/retirement (in both core FHIR and THO), but there is no documented rule stating the relationship between this extension and the `status` concept property. This creates ambiguity for implementers and terminology servers about whether the extension implies a status value.

### Key Considerations
- **Precedent exists:** FHIR-41678 (Applied, Persuasive) already added guidance that defined concept property codes should be treated as reserved and assumed standard when not explicitly declared. This ticket extends that pattern to the relationship between the `standards-status` extension and concept properties.
- **Scope question:** The TI tracker call raised whether this rule applies only to HL7 code systems or universally. Grahame's response indicates it should apply to any code system using the extension.
- **Conflict handling:** The ticket and comments surface an unresolved question: what happens when the extension and the explicit `status` property conflict? Grahame stated the extension should **not** override an explicit status.
- **Text-only change:** Labeled TEXTONLY, so this is a documentation/spec wording change, not a structural resource change.
- **Related work already applied:** FHIR-38106 and FHIR-41678 have both been applied as Persuasive, establishing a pattern of documenting implicit behavior of concept properties. This ticket is a natural continuation.

## Proposed Dispositions

### Disposition A: Persuasive — Document the extension as providing an implicit status value
Add text to the CodeSystem resource page (in the Defined Concept Properties section or near the standards-status extension documentation) stating:
- When a concept has a `structuredefinition-standards-status` extension with a value of "deprecated" or "retired" and no explicit `status` concept property, the extension's value SHOULD be interpreted as providing the concept's status.
- An explicitly stated `status` concept property takes precedence over the extension value.
- Include a caution that conflicting values between the extension and an explicit `status` property create an inconsistent state that publishers should avoid.

**Pros:** Directly addresses the gap. Aligns with existing practice in FHIR core and THO. Consistent with the pattern established by FHIR-41678 and FHIR-38106. Non-breaking (SHOULD-level guidance).
**Cons:** May be seen as overreach for non-HL7 code systems. Requires careful wording to avoid implying that the extension is the preferred way to set status (the explicit property is preferred).

### Disposition B: Persuasive — Require explicit status property when standards-status indicates deprecation
Add text requiring that when the `structuredefinition-standards-status` extension indicates "deprecated" or "retired", an explicit `status` concept property MUST also be present with a consistent value. Existing examples in FHIR core and THO that violate this rule should be corrected.

**Pros:** Eliminates ambiguity entirely. Makes the status property the single source of truth. Clean data model.
**Cons:** Breaking change for existing content — requires remediation of all existing CodeSystem instances in FHIR core and THO that use the extension without an explicit status property. Higher implementation burden.

### Disposition C: Not Persuasive — Defer to existing guidance
The issues raised are already partially addressed by FHIR-41678 (which added SHOULD-level guidance on assuming standard concept properties) and FHIR-42956 (which clarifies inactive vs status). No additional documentation is needed beyond what has already been applied.

**Pros:** No spec change needed. Avoids scope creep.
**Cons:** Does not resolve the specific ambiguity about the `standards-status` extension's relationship to concept status. Leaves implementers guessing. Does not address the actual practice gap identified by the submitter.

## Recommendation

**Disposition A: Persuasive — Document the extension as providing an implicit status value.**

This is the most pragmatic approach. It documents existing HL7 practice, fills a clear gap that has caused real confusion (as evidenced by the Zulip discussions), and aligns with the established pattern from FHIR-41678 and FHIR-38106. Using SHOULD-level language keeps it non-breaking while providing clear guidance. The addition of a caution about conflicting extension/property values addresses the concern raised during the TI tracker call. The scope should be stated as applying to any code system that uses the `standards-status` extension, consistent with Grahame's stated intent.
