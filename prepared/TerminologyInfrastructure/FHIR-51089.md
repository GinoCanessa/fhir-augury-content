## Description

The submitter requests adding a new `manifestParameters` parameter (0..1 Parameters) to both `ValueSet/$expand` and `ValueSet/$validate-code` operations. This parameter would be mutually exclusive with the existing `manifest` parameter but carry the same semantics — allowing callers to pass a Parameters resource directly rather than referencing a persisted Library.

Additionally, the request proposes allowing the existing `manifest` parameter to accept a reference to a `Library | ImplementationGuide`, since both resource types can carry a `cqf-expansionParameters` extension containing the Parameters resource to be used for expansion.

## Resolution Description (Persuasive)

The resolution accepts the proposal with a slight refinement:

1. **Add `manifestParameters` (0..1 Parameters):** Specifies manifest parameters that provide expansion rules for the operation. Parameters specified directly in an `$expand` operation override behaviors specified by the manifest parameter. It is an error to supply both `manifest` and `manifestParameters`.
2. **Update existing `manifest` parameter:** Allow reference to an `ImplementationGuide` as well (dropping the originally proposed `Library` addition since `manifest` already supports `Library`).

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Action:** Accept the resolution as stated. Add a `manifestParameters` parameter (0..1 Parameters) to both `ValueSet/$expand` and `ValueSet/$validate-code` with the documented semantics, and update the `manifest` parameter to accept `ImplementationGuide` references.

**Rationale:** The proposal addresses a genuine usability gap — currently, callers who want to provide expansion parameters must create and persist a Library resource, even when they only need to pass a simple Parameters set. The resolution is well-defined: it specifies the parameter cardinality, mutual exclusivity with `manifest`, override behavior, and error conditions. This aligns with the pattern already established in CRMI (FHIR-51088, FHIR-51624) and simplifies the most common use case. The companion ticket FHIR-51088 has already been applied with overlapping changes in CRMI, making alignment in FHIR Core important.

### Disposition B: Accept as Persuasive with Modification — Scope to $expand Only

**Action:** Add `manifestParameters` to `ValueSet/$expand` only (not `$validate-code`), and update `manifest` to accept `ImplementationGuide` references.

**Rationale:** The `$expand` operation is the primary consumer of manifest parameters, and adding to `$validate-code` simultaneously increases the scope of change. The workgroup could accept the core proposal for `$expand` and defer `$validate-code` to a follow-up ticket, allowing implementers to gain experience with the pattern on the more commonly used operation first. This reduces risk while still delivering the primary value.

### Disposition C: Not Persuasive — Defer to CRMI Alignment

**Action:** Do not add `manifestParameters` to FHIR Core operations at this time; instead, let CRMI continue to define the manifest parameter patterns through operation profiles.

**Rationale:** CRMI already defines operation profiles that add manifest support (FHIR-51088 is Applied, FHIR-51624 is Resolved). Adding these parameters directly to FHIR Core may create maintenance burden and risk divergence between the Core definition and CRMI profiles. The workgroup could wait until CRMI patterns are more stable and then promote to Core. However, this risks continued fragmentation and delays standardization of a widely-needed capability.

## Recommendation

**Disposition A: Accept as Persuasive.** The resolution is clear, well-scoped, and addresses a real interoperability need. The companion CRMI tickets have already been accepted and applied, making FHIR Core alignment the logical next step. The direct Parameters passing pattern reduces unnecessary indirection and simplifies implementation for terminology service consumers. The mutual exclusivity constraint and override behavior are well-defined, minimizing ambiguity for implementers.
