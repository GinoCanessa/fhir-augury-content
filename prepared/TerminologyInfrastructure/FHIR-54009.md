# FHIR-54009: Clarifying exception guidance for the required binding for code elements

## Summary

**Status:** Triaged | **Priority:** Medium | **Type:** Change Request
**Specification:** FHIR Core (FHIR) | **Work Group:** Terminology Infrastructure
**Reporter:** KyleBennett | **Assignee:** Unassigned
**Labels:** Binding
**Created:** 2026-01-12 | **Updated:** 2026-01-28
**URL:** https://jira.hl7.org/browse/FHIR-54009

This ticket raises two issues with the required binding strength guidance in section 4.1.4.2.1 of the FHIR specification:

1. **Deprecated "maximum" binding reference:** The guidance states that required binding strength SHALL be used for `code` elements unless there is a context-free additional binding with `purpose = maximum`. However, per [FHIR-46350](https://jira.hl7.org/browse/FHIR-46350), the `maximum` binding purpose has been deprecated in favor of `required`. The specification text still references this deprecated binding, which is contradictory — the guidance discourages a combination that uses a now-deprecated concept.

2. **Missing exception documentation:** The text states that "Exceptions to this rule need to be approved by the FHIR-I work group which will document the patterns that qualify as exceptions," but no exceptions are linked or enumerated. Readers have no way to find what those approved exceptions are.

## Comments

### esilver — 2026-01-28
Suggests that internal HL7 processes (approval by FHIR-I work group) should not be part of the published standard. Instead, the specification should directly enumerate the exception patterns:

> "Exceptions to this rule are:
> - \<pattern A\>"

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Update the required binding guidance in section 4.1.4.2.1 to (a) remove or replace the reference to the deprecated `maximum` additional binding purpose with current terminology (e.g., reference the `required` additional binding purpose or simply remove the parenthetical about `maximum`), and (b) replace the procedural language about FHIR-I approval with an explicit enumeration of the known exception patterns, per esilver's suggestion. If no exceptions are currently approved, state that clearly (e.g., "No exceptions to this rule are currently defined.").

**Rationale:** Both of the reporter's concerns are valid. The text references a deprecated concept and defers to an undiscoverable process. Fixing both improves clarity without changing binding semantics. The modification acknowledges that the exact replacement wording needs editorial work, particularly to align with the outcomes of FHIR-54128 and FHIR-54589 which are refining additional binding guidance more broadly.

### Disposition B: Persuasive
**Action:** Accept the ticket as-is and update the specification text to remove the deprecated `maximum` reference and enumerate (or link to) approved exception patterns for the required binding rule on `code` elements.

**Rationale:** The reporter's request is straightforward and both issues are clear defects in the specification text. No modification to the request is needed.

### Disposition C: Not Persuasive with Modification
**Action:** Acknowledge the concern about the deprecated `maximum` reference and correct it, but defer the exception documentation to FHIR-54128, which is undertaking a comprehensive review of additional binding documentation.

**Rationale:** The `maximum` deprecation fix is straightforward, but the exception enumeration is better addressed as part of the broader additional bindings documentation effort in FHIR-54128 rather than piecemeal in this ticket. Adding a cross-reference to FHIR-54128 from the guidance text would address the submitter's discoverability concern without duplicating work.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended. Both concerns raised are valid and actionable. The deprecated `maximum` reference is a clear defect that should be corrected regardless. For the exception documentation, the modification gives the editors flexibility to align the wording with the broader additional binding clarification work happening in FHIR-54128 and FHIR-54589, rather than committing to a specific list now that may change. Adopting esilver's suggestion to enumerate patterns directly (rather than referencing an internal approval process) improves the specification's self-containedness and usability for implementers.
