## Problem Statement

The `TerminologyCapabilities` resource currently allows a terminology server to specify a default **version** for any given CodeSystem. However, for SNOMED CT, there is also the concept of a default **Edition** (e.g., International, Australian, Dutch, US, Canadian). This is a distinct axis of configuration: a server may default to the Australian Edition regardless of the specific release version.

Ontoserver (CSIRO's terminology server) already uses a custom extension on `TerminologyCapabilities.codeSystem` to declare its default SNOMED CT Edition:

```json
{
  "codeSystem": [{
    "extension": [{
      "url": "http://ontoserver.csiro.au/fhir/StructureDefinition/defaultsnomededition",
      "valueUri": "http://snomed.info/sct/32506021000036107"
    }],
    "uri": "http://snomed.info/sct",
    "version": [...]
  }]
}
```

The request asks for this extension to be formally adopted or an equivalent extension to be defined in the `fhir.org` namespace, so that all terminology servers can consistently advertise their default SNOMED CT Edition.

## Analysis

### Core Issue
SNOMED CT is unique among code systems in that it has multiple national **editions** that extend the International Edition. Today, `TerminologyCapabilities.codeSystem.version` can list supported versions, but there is no standard way for a server to declare which SNOMED CT edition it uses by default. This forces clients to either guess (typically assuming International) or require out-of-band configuration.

### Real-World Impact
- **Validation failures**: The Zulip discussion shows a Canadian implementer getting "unknown code" errors because the validator defaulted to the International Edition rather than the Canadian Edition.
- **Implementer confusion**: FHIR-39640 and FHIR-44904 document repeated confusion about when to use edition-specific URIs.
- **Interoperability gap**: Without a standard mechanism, each server vendor invents their own approach (e.g., Ontoserver's custom extension).

### Existing Prior Art
Ontoserver's `http://ontoserver.csiro.au/fhir/StructureDefinition/defaultsnomededition` extension is already in production use and demonstrates the pattern: a `valueUri` on `TerminologyCapabilities.codeSystem` carrying the SNOMED CT edition module URI.

### Design Considerations
- The extension should be specifically for SNOMED CT or generalized to any code system that has an analogous edition concept.
- The extension context should be `TerminologyCapabilities.codeSystem` (where the CodeSystem `uri` is `http://snomed.info/sct`).
- The value type (`valueUri`) carrying a SNOMED CT module URI (e.g., `http://snomed.info/sct/32506021000036107`) is well-established by the SNOMED CT URI specification.

## Proposed Dispositions

### Disposition A: Persuasive — Define a standard extension in the FHIR namespace

**Action:** Define a new extension `http://hl7.org/fhir/StructureDefinition/terminologycapabilities-default-snomed-edition` with context `TerminologyCapabilities.codeSystem` and value type `valueUri`. Publish the extension definition with guidance that the URI value should be a SNOMED CT module URI (e.g., `http://snomed.info/sct/32506021000036107` for Australian edition). Add a note to the SNOMED CT usage page referencing this extension.

**Rationale:** This directly addresses the gap identified by the reporter and by multiple related tickets. It standardizes existing practice (Ontoserver) and solves a documented interoperability problem.

### Disposition B: Persuasive with Modification — Generalize to a default-edition concept for any code system

**Action:** Instead of a SNOMED-specific extension, define a more general extension such as `http://hl7.org/fhir/StructureDefinition/terminologycapabilities-default-edition` on `TerminologyCapabilities.codeSystem` with `valueUri`. Document its use for SNOMED CT editions as the primary use case but allow it to be used for other code systems with edition semantics. Update the SNOMED CT usage page accordingly.

**Rationale:** While SNOMED CT is the primary driver, designing a general-purpose extension avoids proliferation of code-system-specific extensions and future-proofs the solution if other code systems adopt edition-like structures.

### Disposition C: Not Persuasive — No change to the specification

**Action:** No change. Recommend implementers use vendor-specific extensions or configure default edition behavior out-of-band. Document the Ontoserver extension as an implementation example but do not standardize.

**Rationale:** The problem may be too SNOMED-specific to warrant a core-specification extension. The TerminologyCapabilities resource already supports version listing, and the concept of a "default edition" could be addressed through server configuration or profiling in implementation guides rather than the base spec.

---

## Recommendation

**Disposition A: Persuasive** is recommended.

The need for a standard default-SNOMED-edition extension is well-documented across multiple Jira tickets (FHIR-13408, FHIR-39640, FHIR-44904) and real-world implementation struggles (Zulip conformance thread). Production prior art exists in Ontoserver. SNOMED CT is the only code system with this particular edition complexity at scale, so a SNOMED-focused extension is pragmatic and directly solves the problem. Generalization (Disposition B) is an option but adds design complexity without a demonstrated need for other code systems. A formal HL7/FHIR namespace extension ensures interoperability across terminology servers and gives clients a reliable discovery mechanism.
