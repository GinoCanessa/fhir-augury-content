## Problem Statement

The SNOMED CT Change Set profile in the Terminology Change Set Exchange IG defines explicit slices for approximately 75 of the 105 currently active SNOMED CT concept properties. This approach has two problems:

1. **Incomplete coverage:** ~30 active SNOMED CT properties are not covered by the explicit slices, leaving no defined mechanism for handling them.
2. **National extension inflexibility:** National extensions of SNOMED CT (e.g., SNOMED CT-AU) define additional properties (~20 extra for Australia). The rigid slicing approach provides no mechanism to accommodate these extension-specific properties.

## Resolution Description (from Jira)

The resolution was marked **Persuasive**. The agreed approach is to replace explicit property slicing with ValueSet-based bindings:

- Create a **CodeSystem of CodeSystem property values** with associated FHIR datatypes, indicating the applicable source CodeSystem.
- Create **per-CodeSystem ValueSets** for properties (e.g., a "SNOMED Properties ValueSet", a "LOINC Properties ValueSet").

This allows extensibility: national extensions can define their own property ValueSets without requiring profile changes.

## Commenter Concerns

**Michael Lawley (2025-03-26):** Raised a versioning concern with self-referential ValueSets — a ValueSet defined relative to SNOMED CT that also constrains updates to SNOMED CT creates a circular dependency:
- The ValueSet must be clear about which SNOMED CT version it references.
- If the change set itself adds a new property, the ValueSet expansion for old SNOMED versions won't include it, but the updated version can't be used because the ChangeSet *is* the update.

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Implement ValueSet-Based Property Bindings

Accept the existing resolution. Replace explicit property slicing in the SNOMED Change Set profile with ValueSet-based bindings as described:

- Define a CodeSystem enumerating all known CodeSystem property values with FHIR datatypes.
- Create per-CodeSystem ValueSets (SNOMED, LOINC, etc.) that reference this CodeSystem.
- Bind the property element in the Change Set profiles to these ValueSets using extensible binding strength, allowing national extensions to supplement with their own ValueSets.
- Address the versioning concern raised by Michael Lawley by specifying that property ValueSets should be versioned independently of the source CodeSystem and should use an "include all known properties" approach rather than referencing a specific CodeSystem version.

**Pros:** Resolves both the incomplete coverage and national extension problems. Extensible bindings allow future properties without profile changes. Aligns with FHIR binding patterns.
**Cons:** Requires creation and maintenance of new ValueSet and CodeSystem resources. The versioning concern for self-referential ValueSets needs careful specification work.

### Disposition B: Accept as Persuasive with Modification — Use Open Slicing Instead

Accept the ticket as persuasive but modify the resolution approach: instead of replacing slicing with ValueSet bindings, retain the existing slices for well-known SNOMED CT properties and add an open slicing discriminator that allows additional properties not covered by the named slices.

**Pros:** Simpler implementation — no new ValueSet/CodeSystem resources needed. Existing implementations using the named slices are unaffected. Allows national extensions to add properties via the open slice.
**Cons:** Does not provide validation or documentation of which properties are valid for a given CodeSystem. Doesn't address the gap of ~30 missing SNOMED properties without adding more slices. Less semantically rich than ValueSet bindings.

### Disposition C: Accept as Persuasive with Modification — Hybrid Approach

Accept as persuasive and combine both approaches: retain named slices for the most commonly used SNOMED CT properties (e.g., top 20–30), add extensible ValueSet bindings for property validation, and use open slicing for properties not in the ValueSet. This provides backward compatibility, extensibility, and validation.

**Pros:** Backward compatible with existing implementations. Provides validation via ValueSet binding while remaining flexible via open slicing. Pragmatic middle ground.
**Cons:** Most complex to specify and maintain. Dual mechanisms (slicing + binding) may confuse implementers. Maintenance burden of both slice definitions and ValueSet resources.

---

## Recommendation

**Disposition A: Accept as Persuasive — Implement ValueSet-Based Property Bindings.**

The original resolution is well-reasoned and directly addresses both problems (incomplete SNOMED coverage and national extension inflexibility). ValueSet-based bindings are a standard FHIR pattern that scales naturally — new properties are accommodated by updating the ValueSet rather than modifying profiles. The versioning concern raised by Michael Lawley is valid and should be addressed in the specification text, but it does not undermine the overall approach. This aligns with the existing resolution consensus and the related work in FHIR-48265 on change set metadata specificity.
