## Problem Statement

The `ValueSet-reference` search parameter has two issues:

1. **Expression/description mismatch:** The search parameter description states it searches both included and **excluded** code systems, but the FHIRPath expression only covers `ValueSet.compose.include.system` — it does not include `ValueSet.compose.exclude.system`. This means the search parameter cannot actually find ValueSets that reference a code system in an exclude statement, despite the description implying it can.

2. **Naming concern:** The name "reference" is overly generic and potentially confusing for a search parameter on ValueSet. The reporter suggests a more descriptive name if compatibility rules permit.

Reference: [ValueSet search parameters](https://build.fhir.org/valueset-search.html#hcValueSet-reference)

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Update Expression Only

**Action:** Update the FHIRPath expression for the `ValueSet-reference` search parameter to include `ValueSet.compose.exclude.system` alongside the existing `ValueSet.compose.include.system`. Do not change the parameter name.

**Rationale:** This directly fixes the documented mismatch between the description and the expression. The description already states the parameter covers excluded code systems, so the expression should match. Renaming the parameter would be a breaking change with compatibility concerns and can be addressed separately. The use case provided by the reporter (excluding codes from a specific system within an included value set) validates that searching on exclude.system is meaningful.

**Risk:** Low. This is a text-only change to the search parameter expression. No structural or behavioral changes to the resource. The name remains "reference" which may continue to cause confusion.

### Disposition B: Accept as Persuasive with Modification — Update Expression and Rename Parameter

**Action:** Update the FHIRPath expression to add `ValueSet.compose.exclude.system`, and rename the search parameter from "reference" to "composeSystem" (or similar descriptive name) if compatibility rules allow.

**Rationale:** Addresses both issues raised in the ticket. The reporter and comment discussion support a more descriptive name. Since this is labelled TEXTONLY and targets a future release (R6), a name change may be feasible without breaking existing implementations if timed appropriately. The suggested name "composeSystem" clearly communicates that the parameter searches the code systems referenced in the compose element.

**Risk:** Medium. Renaming a search parameter is a breaking change for clients that use the current name. Would require deprecation of the old name or an alias mechanism. The compatibility rules for search parameter renaming need to be evaluated.

### Disposition C: Defer — Request Further Analysis on Naming

**Action:** Accept the expression fix (add `ValueSet.compose.exclude.system`) as a straightforward correction, but defer the naming question to a separate ticket for broader community input.

**Rationale:** The expression mismatch is clearly a bug and should be fixed now. The naming discussion requires evaluation of search parameter compatibility rules and broader impact assessment, which may benefit from more community input. This was already flagged for WGM discussion in January 2026, and the TI group had questions about exclude use cases.

**Risk:** Low for the expression fix. The naming issue remains unresolved but is tracked separately.

## Recommendation

**Disposition C: Defer — Request Further Analysis on Naming**

The expression fix is unambiguous — the description says excluded systems are covered but the FHIRPath expression omits them. This should be corrected. However, the naming question is substantively different and introduces breaking-change concerns. The TI tracker call already identified questions about exclude semantics and added this to the WGM agenda. Splitting the work allows the clear fix to proceed while the more complex naming discussion gets proper deliberation. The reporter's note that the proposed resolution "doesn't address a name change" and the question of whether "compatibility rules permit" it further support handling the rename as a separate, carefully evaluated change.
