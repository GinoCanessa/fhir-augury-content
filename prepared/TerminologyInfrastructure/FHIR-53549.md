## Description Summary

The reporter identifies an inconsistency in the CodeSystem specification text regarding the datatype used for coded concept properties. When a CodeSystem defines coded properties, the datatype should be **Coding** (not **code**) so that the IG publisher knows which system the codes come from.

The specification text for CodeSystem supplements correctly recommends using the `Coding` type for property values:

> *"If a supplement needs to define new concepts/codes to use as property values, it can be paired with a new (possibly contained) Code System and use the **Coding** type for the property values."*

However, the immediately following text in the **Concept Properties** section refers to the **code** datatype instead:

> *"The type of the property value. The values for properties of type 'code' (CodeSystem.concept.property.valueCode) are drawn from the codes defined in the CodeSystem instance."*

The reporter argues this creates confusion: using `code` type means the property values are limited to codes within the same CodeSystem, whereas `Coding` allows referencing codes from external code systems — a critical distinction for IG Publisher validation and interoperability.

## Analysis

The core issue is straightforward: there is a textual inconsistency in the CodeSystem specification's "Concept Properties" section. The supplement text correctly advises using `Coding` for property values that reference external code systems, but the adjacent general properties text discusses the `code` type — which by definition constrains values to codes within the same CodeSystem.

This matters because:
1. **IG Publisher behavior** — The property type (`code` vs `Coding`) directly affects how the IG Publisher validates property values. With `code`, values must come from the same CodeSystem; with `Coding`, an external system can be specified.
2. **Implementation guidance** — Authors of CodeSystems with coded properties from external systems need clear guidance to use `Coding` rather than `code`.
3. **Prior work** — FHIR-38105 (Applied, Persuasive) already addressed part of this by clarifying *which* code system the code-type property values come from, but the broader question of when to use `code` vs `Coding` type remains unclear in the text.

The requested change is a **text clarification** — the spec text should explicitly distinguish when `code` type is appropriate (values from the same CodeSystem) vs when `Coding` type should be used (values from external code systems), and the language between the supplements section and the concept properties section should be consistent.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Revise the Concept Properties text in `codesystem-notes.xml` to clearly distinguish between `code`-type properties (values drawn from the same CodeSystem) and `Coding`-type properties (values that reference codes from external systems). Harmonize the language with the supplements section text.

**Rationale:** The inconsistency is real and could lead implementers to use `code` where `Coding` is required, causing IG Publisher validation failures. The prior resolution of FHIR-38105 improved the text but did not fully resolve this particular ambiguity. A clear, consistent explanation across both sections is warranted.

### Disposition B: Persuasive with Modification
**Action:** Accept the concern, but rather than only fixing the language inconsistency, expand the Concept Properties section to include a brief guidance table or paragraph explaining each property type (`code`, `Coding`, `string`, `integer`, `boolean`, `dateTime`, `decimal`) and when each is appropriate, with specific attention to the `code` vs `Coding` distinction.

**Rationale:** The underlying confusion stems from the subtlety of the code-vs-Coding distinction. A broader clarification that covers all property types would be more useful to implementers and would prevent similar confusion in the future.

### Disposition C: Not Persuasive
**Action:** No change. The current text accurately describes the behavior of the `code` property type — values for `code`-type properties *are* drawn from the same CodeSystem instance. The supplement text describes a different scenario (supplements needing external codes). The two passages are not inconsistent; they describe different use cases.

**Rationale:** The spec text is technically correct for each context. The `code` type is intentionally limited to the same CodeSystem, and the supplement text correctly advises using `Coding` when external codes are needed. The distinction is inherent to the data model, not a documentation error.

## Recommendation

**Disposition A: Persuasive**

The inconsistency in language is genuine and has practical consequences for implementers. The text in the Concept Properties section should be revised to explicitly acknowledge both `code` (same-system) and `Coding` (cross-system) property types, bringing it into alignment with the supplement guidance. This is a focused text change to `source/codesystem/codesystem-notes.xml` that improves clarity without requiring structural changes to the resource definition. The prior resolution of FHIR-38105 addressed a related but distinct ambiguity, and this ticket closes the remaining gap.
