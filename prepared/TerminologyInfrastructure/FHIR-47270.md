# FHIR-47270: Duplication of code systems and value sets with THO (using different URLs)

## Summary

**Status:** Triaged | **Priority:** Highest | **Type:** Change Request
**Specification:** FHIR Extensions Pack (FHIR) | **Work Group:** Terminology Infrastructure
**Reporter:** Bas van den Heuvel | **Assignee:** Unassigned
**Labels:** DupResources
**Created:** 2024-09-10 | **Updated:** 2026-01-26
**URL:** https://jira.hl7.org/browse/FHIR-47270

The reporter identifies that multiple CodeSystems and ValueSets exist in both the FHIR Extensions Pack and HL7 Terminology (THO), with **different canonical URLs**. For example, the `ImplantStatus` ValueSet uses `http://hl7.org/fhir/ValueSet/implantStatus` in the Extensions Pack but `http://terminology.hl7.org/ValueSet/implantStatus` in THO; similarly the CodeSystem uses `http://hl7.org/fhir/implantStatus` vs. `http://terminology.hl7.org/CodeSystem/implantStatus`. The reporter argues FHIR should standardize on a single location and URL, deprecating or retiring the other.

## Analysis

This ticket is part of a well-documented, systemic issue: terminology resources (CodeSystems and ValueSets) are duplicated between the FHIR Extensions Pack and THO, with different canonical URLs. This causes confusion, validator failures, and interoperability problems. Key context:

- **FHIR-47498** identified 14 CS/VS pairs duplicated by resource ID, and a subgroup was formed at Sep 2025 WGM. Lloyd McKenzie proposed: "Things bound as code should live in the extension pack. Things not bound as code should live in THO."
- **FHIR-55033** (Jan 2026) consolidates the analysis with a shared spreadsheet tracking resolutions. Jessica Bota noted many duplicates have already been removed from FHIR core, with remaining resolutions tied to FMG's updated Terminology Expectations policy.
- The FMG Extensions Pack Zulip thread (2023) highlights real implementation pain from duplicate canonicals in HAPI and Firely servers.
- The ticket was scheduled for joint TI/FHIR-I discussion at the Jan 2026 WGM, with expectation it may be resolved before or during that meeting.

## Proposed Dispositions

### Disposition A: Persuasive — Consolidate in THO with deprecation
**Resolution:** Accept the request. For each duplicated CS/VS pair, designate THO (`terminology.hl7.org`) as the authoritative canonical URL. Deprecate the Extensions Pack copies with a description note directing implementers to THO. Update Extension bindings to reference THO URLs. This aligns with the TSC-approved "Terminology Expectations for IG Developers" policy and the direction established in FHIR-47498/FHIR-55033.

**Pros:** Eliminates URL confusion, aligns with established policy, reduces validator errors.
**Cons:** Requires coordinated updates across Extensions Pack, core spec, and THO; may break existing references using `http://hl7.org/fhir/` URLs.

### Disposition B: Persuasive — Defer to FHIR-55033 umbrella resolution
**Resolution:** Accept as persuasive but resolve by marking this ticket as a duplicate of or subsumed by [FHIR-55033](https://jira.hl7.org/browse/FHIR-55033), which already tracks the systematic resolution of all duplicate terminology artefacts with a detailed spreadsheet and analysis. No separate action is needed beyond ensuring `ImplantStatus` is included in that tracking.

**Pros:** Avoids piecemeal resolution; leverages existing coordinated effort; reduces overhead.
**Cons:** Delays explicit resolution of this specific case until FHIR-55033 completes.

### Disposition C: Not Persuasive — Maintain current structure
**Resolution:** Reject the request. The duplication serves a purpose: the Extensions Pack versions use `http://hl7.org/fhir/` URLs for consistency within the FHIR core namespace, while THO versions use `http://terminology.hl7.org/` for standalone terminology use. Both are valid and serve different contexts.

**Pros:** No migration effort required; preserves backward compatibility.
**Cons:** Contradicts TSC terminology policy; perpetuates implementer confusion; contradicts the direction already established by TI (FHIR-47498, FHIR-55033).

---

## Recommendation

**Disposition B: Persuasive — Defer to FHIR-55033 umbrella resolution.**

This is the pragmatic choice. The issue raised in FHIR-47270 is valid and well-documented, but it is one instance of a broader systemic problem already being systematically addressed through FHIR-55033 and the tracking spreadsheet maintained by Reuben Daniels and Jessica Bota. Significant progress has already been made (many duplicates removed from FHIR core per Nov 2025 comments). Resolving this ticket by linking it to the umbrella effort ensures the `ImplantStatus` case is handled consistently with all other duplicates, avoids redundant work, and aligns with the subgroup's resolution approach.
