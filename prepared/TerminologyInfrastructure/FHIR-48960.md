## Description

The submitter requests clarification in the FHIR specification around the behavior of CodeSystem fragments and supplements. Specifically, the ticket asks for guidance on:

1. **Relationship between multiple fragments of the same code system:**
   - Whether a system may have multiple fragments of the same code system, or must synthesize a single combined code system
   - Whether fragments are expected to be disjoint, may overlap, or may be subsets of each other
   - Whether there is a way to indicate that a set of fragments represents the entire code system (enabling negative validation)
   - Expectations around fragments of different code system versions (e.g., can a ValueSet expansion combine fragments from different versions)

2. **Relationship between fragments and supplements:**
   - Whether supplements must align with a specific fragment or can cover concepts from across multiple fragments (which may not be present on the system)

3. **Conflict resolution** when multiple fragments (or a fragment and a complete/example code system) contain differing information on the same concept.

4. **ValueSet resource updates** to reflect the impact of any clarifications.

## Resolution: Persuasive with Modification

The resolution proposes a comprehensive 7-point plan:

1. **Conformance statement for versioning**: Add a conformance statement (with FHIRPath constraint at warning level) that if a fragment is drawn from a versioned code system, the fragment SHOULD provide a value for `CodeSystem.version`.

2. **New terminology service section**: Add a new section 4.7.4 "Fragments and Supplements" to the [FHIR Terminology Service page](https://build.fhir.org/terminology-service.html).

3. **Complete CS takes precedence**: If a complete code system and a fragment for the same code system exist on a server, the server SHALL use the complete code system. A fragment SHOULD be used only if it is the only code system available.

4. **Supplements with fragments**: Supplements MAY be used with fragments. Supplements SHOULD define the version of the code system they supplement if the code system is versioned.

5. **New `TerminologyCapabilities.implementation.fragmentSupport` element** (markdown, 0..*): Servers must populate this to describe how fragments are supported. Must be present and valued if the server supports fragments. Examples include whether the server supports multiple fragments, whether fragments may overlap, and how conflicting information is handled.

6. **New `TerminologyCapabilities.implementation.supplementSupport` element** (markdown, 0..*): Must be present and valued if the server supports supplements. Documents whether supplements are supported for complete code systems and/or fragments.

7. **Note to balloters**: Add a callout in the CodeSystem resource fragment section stating that certain behaviors are deliberately left to implementers, with a request for community feedback. Link to the `fragmentSupport` element definition.

**Note**: The 3rd sub-bullet of the original request (indicating that a set of fragments constitutes the entire code system) is explicitly **not addressed** in this resolution.

## Key Discussion Points from Comments

- **Michael Lawley** (2025-08-18): No implication for explicit combination of fragments; no expectation that fragments are disjoint; a terminology server MAY handle fragments by taking the union of their contents; a supplement referencing a concept not in any available fragment is problematic.
- **Carmela Couderc** (2025-09-29 through 2026-03-31): Evolved the proposal through multiple iterations—from initial straw man to the final 7-point resolution. Noted that FHIR-48959 was discussed at the September 2025 WGM, and that by definition a fragment cannot conflict with the complete code system.
- **Carmela Couderc** (2026-03-12): Some servers treat any conflict between fragments for the same CodeSystem version as rendering all fragments invalid. If a fragment doesn't assert the code system version, servers may refuse to support it.

The original discussion that led to this ticket. Elliot Silver raised questions about behavior when multiple fragments exist for the same code system—validation, synthesis, supplement alignment. Key takeaways:
- Michael Lawley confirmed there is no specific expected behavior in the FHIR spec for multiple fragments.
- Grahame Grieve stated he does not synthesize fragments in his server.
- The discussion revealed that the spec says "Fragments can only be published by the code system authority" but practical usage varies.
- Consensus: no defined behavior exists, and the spec needs clarity. Three Jira tickets were filed (FHIR-48958, FHIR-48959, FHIR-48960).

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification (as resolved)
Accept the 7-point resolution as documented. This adds conformance statements for fragment versioning, creates a new "Fragments and Supplements" section on the terminology service page, establishes precedence rules (complete CS over fragment), allows supplements with fragments, and introduces two new `TerminologyCapabilities.implementation` elements (`fragmentSupport` and `supplementSupport`) to let servers describe their behavior. A note to balloters solicits implementer feedback. This approach pragmatically leaves complex behaviors (like fragment merging semantics) to server discretion while establishing baseline rules and requiring transparency through TerminologyCapabilities.

### Disposition B: Request further refinement before acceptance
The resolution deliberately does not address the 3rd sub-bullet (indicating a complete set of fragments). Additionally, the proposed `fragmentSupport` and `supplementSupport` elements are free-text markdown, which limits interoperability—servers would describe behavior in prose rather than computable terms. Request that the resolution be refined to consider structured/coded values for fragment support levels (e.g., "merge", "first-match", "reject-conflicts") to enable programmatic discovery of server capabilities.

### Disposition C: Accept with additional scope
Accept the resolution as-is but also request that the note to balloters explicitly ask whether a mechanism to indicate "complete set of fragments" is needed (the unaddressed 3rd sub-bullet). This would ensure the community has an opportunity to weigh in on this gap during the ballot cycle, potentially leading to a follow-up ticket if there is demand.

## Recommendation

**Disposition A: Accept as Persuasive with Modification.** The resolution is well-developed through multiple iterations by the TI work group (with input from Michael Lawley, Grahame Grieve, and others). It addresses the core questions raised in the ticket with pragmatic conformance statements and new TerminologyCapabilities elements while appropriately deferring complex merging semantics to implementers. The note to balloters provides a mechanism for gathering further community feedback. The unaddressed sub-bullet (completeness indicator for fragment sets) is a niche concern that can be tracked separately if ballot feedback warrants it.
