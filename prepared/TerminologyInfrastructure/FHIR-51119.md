## Description

This ticket proposes adding support for **nested filters** within `ValueSet.compose.include.filter`. Currently, when building complex value sets using filters, authors must decompose logic across multiple ValueSet resources. This ticket would allow a filter to contain child filters in place of a value, enabling more expressive inline definitions.

### Specific Proposed Changes

1. **Add a recursive `@filter` element** (`0..*`) to `ValueSet.compose.include.filter`, to be used in place of `value` when the operator permits a list of codes and/or a value set.
2. **Change `filter.value` cardinality** from `1..1` to `0..1`.
3. **Add an invariant** requiring that either `value` or nested `filter` SHALL be present.
4. **Semantics**: When one or more nested filters are present, their **intersection** specifies the set of codes that the parent operator acts on.

### Backward Compatibility

This is **not a breaking change** — all existing value sets remain valid. The change only enables new, more complex inline filter definitions.

### Resolution Description

> Make the proposed changes to the structure of the value set resource and add the constraint as specified. This disposition will be elaborated on further when the ticket is voted on.

## Analysis

### Context

This ticket is part of a **trio of coordinated changes** (FHIR-51117, FHIR-51118, FHIR-51119) filed by Grahame Grieve to enhance `ValueSet.compose` expressiveness. The driving motivation is ensuring that the VCL (ValueSet Compose Language) proposal by Michael Lawley can remain a 1:1 syntactic mapping to compose. Without nested filters, complex VCL expressions must be decomposed into multiple ValueSet resources, which is verbose and harder to maintain.

### Strengths

- **Clear backward compatibility**: No existing value sets are invalidated; this is purely additive.
- **Addresses a real gap**: The inability to nest filters forces decomposition into separate ValueSet resources, which is a known pain point.
- **Coherent with companion tickets**: Together with FHIR-51117 (value set references in filter values) and FHIR-51118 (reverse property navigation), this creates a comprehensive enhancement to compose's expressiveness.
- **Well-defined semantics**: Nested filters use intersection semantics, consistent with existing filter composition behavior.

### Concerns

- **Implementation complexity**: Recursive/nested filter evaluation adds complexity to terminology server implementations. Servers must handle arbitrary nesting depth.
- **Validation complexity**: The new invariant (value XOR filter present) is straightforward, but nested filter validation becomes recursive.
- **Resolution is underspecified**: The resolution states "will be elaborated on further when the ticket is voted on," meaning the precise structural changes are not yet fully specified.
- **Dependency on FHIR-51117**: Nested filters only make sense "if the operator allows for a list of codes/and or a value set," which is functionality introduced by FHIR-51117 (now Applied).

## Proposed Dispositions

### Disposition A: Persuasive (Accept as proposed)
Accept the resolution as stated. Add the recursive `@filter` element (`0..*`) to `ValueSet.compose.include.filter`, change `filter.value` to `0..1`, and add the invariant requiring either `value` or nested `filter`. The structural changes and constraint should be elaborated in the specification text with clear examples showing nested filter usage and the intersection semantics.

**Rationale**: The proposal is well-motivated by the VCL alignment goal, is backward-compatible, and follows logically from the already-applied FHIR-51117. The resolution's acknowledgment that it will be "elaborated on further" is appropriate for a structural change that needs worked examples.

### Disposition B: Persuasive with Modification — Require depth limit and implementation guidance
Accept the proposal but add: (1) a recommended maximum nesting depth (e.g., 3 levels) to constrain implementation complexity, (2) explicit implementation guidance for terminology servers on how to evaluate nested filters, and (3) a worked example in the specification showing at least one non-trivial nested filter use case.

**Rationale**: While the concept is sound, recursive structures in data exchange standards can lead to interoperability issues if implementations handle nesting differently. Adding bounds and guidance reduces this risk.

### Disposition C: Not Persuasive — Defer until VCL alignment is more mature
Defer this change until the VCL specification is more fully developed and the community has clearer consensus on exactly which compose extensions are needed for VCL alignment. The current workaround of using multiple ValueSet resources, while verbose, is functional and well-understood.

**Rationale**: This is a significant structural change to a foundational resource. The resolution itself acknowledges it needs further elaboration. Deferring allows the VCL initiative to mature so that nested filters can be designed with full knowledge of requirements.

## Recommendation

**Disposition A: Persuasive** — Accept as proposed. The change is backward-compatible, directly motivated by the VCL alignment work that has community support, and logically completes the trio of compose enhancements. FHIR-51117 (which extended filter.value semantics) is already Applied, and FHIR-51118 is resolved Persuasive with Modification — this ticket naturally follows. The resolution's note about further elaboration during voting is appropriate and does not block acceptance. The work group should ensure the elaborated specification includes clear examples and implementation notes.
