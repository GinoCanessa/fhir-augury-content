## Summary

The submitter requests clarification on the terminology binding for the `CodeSystem.concept.property.code` element. The element definition references "CodeSystem.property.code or a FHIR defined concept-property" but does not include a link to a value set or a formal binding. The submitter notes that the FHIR-defined concept properties (e.g., status, parent, child, deprecationDate) are documented on a separate page but are not connected through a binding, and asks whether `CodeSystem.property.code` should have a required or extensible binding, or whether it is intentional that each code system may define its own property codes freely.

## Details

**Description:**

Please provide clarification on the terminology binding for this code element CodeSystem.property.code.

The definition for this element in the table: "Reference to CodeSystem.property.code or a FHIR defined concept-property" does not include a link to a value set, nor a binding.

In the detailed definition ([CodeSystem.concept.property.code](https://hl7.org/fhir/6.0.0-ballot4/codesystem-definitions.html#CodeSystem.concept.property.code)) at least there is a link to the FHIR-defined concept-property ([defined-props](https://hl7.org/fhir/6.0.0-ballot4/codesystem.html#defined-props)), though that table is not entirely clear on what the "code" actually looks like (in implementations will it be "status" or "http://hl7.org/fhir/concept-properties#status"?).

The table summarizing the terminology bindings ([tx](https://hl7.org/fhir/6.0.0-ballot4/codesystem.html#tx)) does not list "CodeSystem.property.code" or "FHIR defined concept-property" but does list "CodeSystem.property.type" ([concept-property-type value set](https://hl7.org/fhir/6.0.0-ballot4/valueset-concept-property-type.html)).

Since it is a code, so no code system is available, shouldn't this be a required or at least extensible binding, or is it ok for each code system to create their own codes here?

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Add an extensible binding on `CodeSystem.concept.property.code` to a new or existing value set containing the FHIR-defined concept properties (status, inactive, effectiveDate, deprecated, retirementDate, notSelectable, parent, child, partOf, root, synonym, comment). Update the element definition to include a clear link to this value set and clarify that the property code is the short name (e.g., "status") rather than the full URI. Add explanatory text noting that code systems may define additional custom property codes beyond those in the value set.
#### Justification
Multiple community members have independently identified the lack of a binding as a discoverability and consistency problem. An extensible binding would not break existing implementations (since any custom codes would still be valid) while providing formal documentation of the well-known properties. The existing value set at `http://hl7.org/fhir/codesystem-concept-properties` already enumerates these codes and could serve as the basis for the binding.

---

### Disposition B: Improve Documentation Without Adding a Binding
#### Proposal
Do not add a formal terminology binding, but improve the element definition for `CodeSystem.concept.property.code` by adding a direct hyperlink to the FHIR-defined concept properties table, adding clarifying text that the code value is the short property name (e.g., "status") that references `CodeSystem.property.code`, and noting that the URI in `CodeSystem.property.uri` gives the property a well-known meaning. Add an entry in the terminology bindings summary table that references the concept-properties code system as informational guidance.
#### Justification
As Grahame Grieve and Michael Lawley explained in Zulip discussions, the property code is intentionally an internal reference within the code system, and its meaning is grounded by `CodeSystem.property.uri`. A formal binding could be misleading because these codes are relative identifiers, not absolute codes from a shared code system. However, the current documentation clearly has a discoverability gap that can be addressed without a binding change, avoiding any risk of breaking existing implementations or introducing confusion about the nature of property codes.

---

### Disposition C: Decline
#### Proposal
Close the ticket as "no change required." The current design is intentional: `CodeSystem.property.code` is a local reference within a code system, not an externally bound coded element. The FHIR-defined concept properties are documented and linked from the detailed definitions, and the existing design supports extensibility.
#### Justification
Grahame Grieve explicitly stated in Zulip that `CodeSystem.property.code` should not have a binding. The element is by design a local code whose meaning is established by the associated `CodeSystem.property.uri`. Adding a binding would incorrectly imply that these codes have absolute meaning independent of the code system context. The existing documentation, while imperfect, already provides links to the defined properties in the detailed element definitions.

---

### Recommendation
**Recommended disposition:** B — Improve Documentation Without Adding a Binding

The ticket raises a legitimate usability concern that has been echoed by multiple community members across several years of Zulip discussions. The difficulty in discovering and correctly using the FHIR-defined concept properties is well-documented. However, adding a formal extensible binding (Disposition A) risks misrepresenting the nature of `CodeSystem.property.code`, which is fundamentally a local reference within a code system that derives its well-known meaning from the associated URI — not from membership in a shared code system. Michael Lawley's observation that these codes are "effectively relative and grounded by their association with CodeSystem.property.uri" is technically precise and argues against a binding. Disposition B addresses the submitter's core concern — lack of discoverability and clarity — through improved documentation and explicit links, without altering the semantic model. This approach respects the original design intent confirmed by Grahame Grieve while meaningfully improving the implementer experience.
