## Summary

The reporter raises concern that FMM (FHIR Maturity Model) levels assigned to terminology profiles in the CRMI (Canonical Resource Management Infrastructure) IG are significantly higher than those in R5 or R4B, without adequate justification. Specifically, ShareableCodeSystem is FMM 0 in R5 but FMM 4 in CRMI, and PublishableCodeSystem jumped from 0 to 3. The reporter requests that FMM levels be adjusted after discussion with TI.

The resolution (Not Persuasive) states the workgroup discussed and agreed the maturity levels as proposed in the ballot are appropriate, listing the following levels:

| Profile | FMM Level |
|---------|-----------|
| ShareableValueSet | 4 |
| ShareableCodeSystem | 4 |
| ShareableConceptMap | 1 |
| ShareableNamingSystem | 1 |
| ShareableTerminologyCapabilities | 2 |
| PublishableCodeSystem | 3 |
| PublishableConceptMap | 1 |
| PublishableNamingSystem | 1 |
| PublishableTerminologyCapabilities | 2 |
| PublishableValueSet | 3 |
| ComputableValueSet | 3 |
| ExpandedValueSet | 3 |

## Details

### Reporter's Concern
The core issue is that FMM levels were 0 in R5 for these profiles, but in the CRMI IG they are significantly elevated (e.g., 4 for ShareableCodeSystem, 3 for PublishableCodeSystem) without visible justification for the jump.

### IG Developer Response (bryn.rhodes, 2025-10-22)
The reason FMM levels were 0 in R4, R4B, and R5 is that HL7 was not tracking or publishing FMM levels for profiles in the base specification. ShareableValueSet has been in the base specification since DSTU2, and ShareableCodeSystem since STU3. Additionally, the Shareable and Publishable profiles for all other terminology resources have been in CRMI since STU1, so the maturity levels reflect actual use in connectathon and/or production scenarios.

### Process Concern (carmela_couderc, 2026-02-23)
The reporter objected to including this ticket in a block vote, suggesting it should be added to a main TI workgroup call agenda for proper in-person discussion.

### Independent Review (marc.duteau, 2026-03-09)
Marc Duteau conducted a review of the FMM levels against the FHIR Maturity Model confluence page criteria. His findings:
- **FMM 1 profiles** (ShareableConceptMap, ShareableNamingSystem, PublishableConceptMap, PublishableNamingSystem): Cannot go lower; would need to meet FMM 2 testing requirements to go higher.
- **FMM 2 profiles** (ShareableTerminologyCapabilities, PublishableTerminologyCapabilities): Appear to meet testing requirements but lack evidence of broad review needed for FMM 3.
- **FMM 3 profiles** (PublishableCodeSystem, PublishableValueSet, ComputableValueSet, ExpandedValueSet): Meet the requirements but are not yet at FMM 4 level.
- **FMM 4 profiles** (ShareableCodeSystem, ShareableValueSet): Definitely at least FMM 4.

He concluded the FMM levels make sense, noting that the IG developers believe they meet the testing requirements and that the shareable code system and value set profiles warrant protection against non-backward-compatible changes.

## Proposed Dispositions

### Disposition A: Accept (Uphold "Not Persuasive")
Uphold the current "Not Persuasive" resolution. The FMM levels as assigned are justified because:
1. The FMM 0 values in R5 were artifacts of not tracking FMM for profiles in the base spec—not an indication of immaturity.
2. ShareableValueSet and ShareableCodeSystem have existed since DSTU2 and STU3 respectively, and all terminology profiles have been in CRMI since STU1.
3. An independent review by Marc Duteau confirmed the assigned levels align with the FHIR Maturity Model criteria.
4. The FHIR Maturity Model explicitly states that artifacts being moved or refactored don't have their maturity reset unless FMG deems the change significant.

### Disposition B: Accept with Additional Documentation
Uphold the "Not Persuasive" resolution but request that the CRMI IG include a brief rationale section explaining why FMM levels differ from R5. This could reference the fact that FMM was not tracked for profiles in the base specification, and that the CRMI levels are based on actual usage in connectathon and production scenarios. This would address the reporter's concern about "without justification" by making the justification visible to future reviewers.

### Disposition C: Decline (Overturn to Persuasive)
Overturn the resolution and lower the FMM levels to align more closely with R5 values. This would require resetting FMM levels to 0 or 1 for profiles that were newly moved from the core spec to CRMI, treating the move as a significant refactoring event. This approach would prioritize consistency with the base specification over historical usage evidence.

## Recommendation

**Disposition A: Accept (Uphold "Not Persuasive")**

The evidence strongly supports that the current FMM levels are justified. The reporter's concern was based on comparing CRMI FMM levels to R5 FMM 0 values, but R5 simply did not track FMM for profiles. The profiles themselves have substantial history: ShareableValueSet dates to DSTU2 and ShareableCodeSystem to STU3. Both independent review by Marc Duteau and the IG developers' explanation provide clear rationale that aligns with the FHIR Maturity Model criteria. The workgroup has discussed this through multiple venues (TI tracker call, joint CQI/CDS agenda at the January WGM), and consensus supports the current levels. However, the workgroup may wish to note the reporter's process concern about block voting and ensure this receives adequate discussion at a main TI workgroup call.
