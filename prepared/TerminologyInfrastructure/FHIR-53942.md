# FHIR-53942: The out parameters list of issues contains invalid/unclear paths

## Summary

**Ticket:** [FHIR-53942](https://jira.hl7.org/browse/FHIR-53942)
**Status:** Triaged | **Type:** Change Request | **Priority:** Medium
**Reporter:** Brian Postlethwaite | **Assignee:** Unassigned
**Work Group:** Terminology Infrastructure | **Specification:** FHIR Core (FHIR)
**Created:** 2026-01-11 | **Updated:** 2026-01-17

The reporter identifies that the `$validate-code` operation's output `issues` parameter (which contains an OperationOutcome) has paths in the OperationOutcome.issue entries that are invalid or unclear. Specifically, three questions are raised:

1. **What are the issue paths based on?** — Are the `expression` values in the OperationOutcome.issue entries relative to the *input* Parameters object or the *output* Parameters object? The reporter assumes input (since code-only requests have no Coding in the output), but this is not documented.
2. **How do we report issues with the `code` form of the input?** — When the input is a bare `code` (not a Coding or CodeableConcept), there is no clear path convention to reference it in the OperationOutcome.issue.expression.
3. **Request for examples** — Please include one or two examples showing the expected usage of the `issues` OperationOutcome with proper `expression` property values.

## Analysis

### Core Problem

The `$validate-code` operation (on CodeSystem, ValueSet, and ConceptMap) returns an `issues` output parameter containing an OperationOutcome resource. The OperationOutcome.issue entries can include an `expression` property to indicate which part of the input caused the issue. However:

1. **The specification does not clearly state the base context for `expression` paths.** Are they relative to the input Parameters resource, the output Parameters resource, or something else entirely? This is critical for implementers building terminology clients that need to map validation errors back to the original request.

2. **The `code` input form creates a path ambiguity.** When a caller uses the simple `code` parameter (a bare code string) rather than `coding` (a Coding datatype) or `codeableConcept` (a CodeableConcept), there is no obvious FHIRPath expression to reference the input. A `coding` input has a clear path (e.g., `Parameters.parameter.where(name='coding').valueCoding`), but a bare code does not have sub-properties like `system` to distinguish.

3. **No examples exist in the specification** showing how the `expression` property should be populated in the `issues` OperationOutcome for `$validate-code`. This leaves implementers to guess at conventions, leading to interoperability issues between different terminology servers (as evidenced by the Zulip discussions showing different server implementations).

### Impact

- **Implementers** cannot reliably parse the `expression` field to provide meaningful error feedback to users.
- **Terminology servers** may implement inconsistent path conventions, reducing interoperability.
- **Tooling** (e.g., FHIR validators, IDE integrations) that attempts to map OperationOutcome issues back to input elements will produce unreliable results.

## Proposed Dispositions

### Disposition A: Persuasive — Clarify paths and add examples

**Action:** Accept the change request. Update the `$validate-code` operation definition to:
1. Explicitly state that `expression` values in the `issues` OperationOutcome are relative to the **input Parameters** resource.
2. Define a convention for referencing the `code` input parameter (e.g., `Parameters.parameter.where(name='code').value`).
3. Add at least two examples to the operation documentation:
   - One showing an issue with a `coding` input (e.g., invalid display value) with the corresponding expression path.
   - One showing an issue with a bare `code` input with the corresponding expression path.

**Rationale:** All three asks are reasonable documentation improvements that directly address implementer confusion. The question of path base context is a genuine gap in the specification. Adding examples follows the FHIR convention of illustrating operation behavior through worked examples.

### Disposition B: Persuasive with Modification — Clarify paths only, defer examples to implementer guidance

**Action:** Accept in part. Clarify the path base context (input Parameters) and the `code` form convention in the operation narrative, but provide examples in the implementer guidance / terminology services documentation rather than in the operation definition itself.

**Rationale:** The operation definition should remain concise. Detailed examples may be better placed in supplementary documentation or the terminology services page, where they can cover multiple operations and scenarios comprehensively.

### Disposition C: Not Persuasive — Covered by general OperationOutcome guidance

**Action:** Reject the change request on the basis that OperationOutcome.issue.expression is already defined in the OperationOutcome resource definition and general operation handling rules. The path base context is the operation input, which is implicit in how operations work.

**Rationale:** The OperationOutcome resource already states that `expression` is "A [simple subset of] FHIRPath limited to element names, repetition indicators and the default child accessor". Adding operation-specific guidance may conflict with or duplicate general guidance. However, this disposition does not address the legitimate `code` form ambiguity or the lack of examples.

---

## Recommendation

**Disposition A: Persuasive** — This is the strongest disposition. The reporter raises three clear, actionable documentation gaps. The question of what the `expression` paths are relative to is a genuine specification ambiguity that affects interoperability between terminology servers (as seen in real-world Zulip discussions). The `code` form path convention is a real gap. Adding examples is a standard FHIR practice for operations and would significantly improve implementer experience. All three changes are low-risk documentation clarifications that strengthen the specification.
