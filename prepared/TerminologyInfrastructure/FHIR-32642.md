This change request, originally submitted in May 2021, proposes adding parameters to the `$expand` operation to control how draft/unreleased content is handled during value set expansion. The scope has narrowed over time: the **manifest** parameter was split to a separate ticket ([FHIR-45986](https://jira.hl7.org/browse/FHIR-45986), now Applied), and the **expansion** parameter was also deferred. The remaining proposal focuses on a single boolean parameter—`includeUnreleasedContent`—that allows clients to explicitly request whether unreleased (e.g., draft) value set versions should be considered during expansion.

A previous resolution (voted 18-3-4 in May 2024) introduced a `considerStatus` parameter (0..* code bound to PublicationStatus), but this was reverted by a 9-0-0 vote in March 2025 due to concerns about naming clarity and definition ambiguity. The current resolution proposes the simpler `includeUnreleasedContent` boolean approach, refined after CRMI separated the concepts of "published" from "released" at the Madrid WGM.

## Details

### Original Request

Add three parameters (`manifest`, `expansion`, `includeDraft`) to the core `$expand` operation, mirroring those defined in the CRMI IG's extended `$expand` operation ([CRMI ValueSet/$expand](https://build.fhir.org/ig/HL7/crmi-ig/OperationDefinition-crmi-valueset-expand.html)).

### Current Proposed Resolution

Add the following parameter to `$expand`:

> **includeUnreleasedContent** `boolean` `0..1`
>
> True indicates that when resolving which version of a value set definition should be used during an expansion, the server SHALL consider unreleased content. False indicates that the server SHALL only consider released content during the expansion. If this parameter is not supplied, the behavior is at the server's discretion. What constitutes released content is at the server's discretion, such as publication status or workflow status. Servers MAY choose to throw an exception if asked to include unreleased content and they do not support this capability.

### Key Discussion Points

1. **`considerStatus` was rejected**: The previous resolution defined `considerStatus` as a repeating code parameter bound to PublicationStatus. Grahame Grieve objected that the name was confusing ("I considered the status, and hey, I liked it anyway"), the definition was unclear about what happens when a version is excluded, and the "transitive through the expand" language was ambiguous. This was reverted 9-0-0 in March 2025.

2. **Boolean vs. status list**: Michael Lawley preferred a parameter listing which statuses to consider (more flexible). However, committee discussion moved toward the simpler boolean as the core use case is distinguishing "released" from "unreleased" content, not fine-grained status filtering.

3. **Default behavior varies across servers**: VSAC only returns draft content when explicitly requested (`includeDraft=true`). Grahame's server and Michael Lawley's server include draft by default. The parameter is designed so that servers can maintain their current default behavior when the parameter is omitted.

4. **Content lifecycle clarity**: At Madrid WGM (May 2025), CRMI agreed to separate "publication status" from "authoring status," leading to the `includeUnreleasedContent` naming which avoids confusion with the FHIR `PublicationStatus` valueset.

5. **Scope narrowing**: The manifest parameter was broken out to [FHIR-45986](https://jira.hl7.org/browse/FHIR-45986) (now Applied). The expansion parameter is to be addressed separately. This ticket now focuses solely on the unreleased content parameter.

6. **Broader alignment**: [FHIR-54108](https://jira.hl7.org/browse/FHIR-54108) proposes aligning and distinguishing operation parameter descriptions across all terminology operations, and FHIR-32642 has been linked for review in that context.

## Proposed Dispositions

### Disposition A: Accept — Approve `includeUnreleasedContent` as Proposed

**Resolution:** Persuasive with Modification. Add the `includeUnreleasedContent` boolean parameter (`0..1`) to the `$expand` operation as defined in the current resolution description.

**Rationale:** The parameter addresses a real interoperability gap—servers currently differ in whether draft/unreleased content is included by default, and clients have no way to request consistent behavior. The boolean approach is simple, backward-compatible (omitting the parameter preserves current server behavior), and aligns with CRMI's separation of publication vs. authoring status agreed at Madrid. The scope has been appropriately narrowed (manifest already handled in FHIR-45986). The proposal has been refined through 4+ years of discussion and multiple rounds of committee review.

**Risks:** The boolean may be too coarse for servers that want finer-grained status filtering. "What constitutes released content" is left to server discretion, which could lead to inconsistent interpretation.

### Disposition B: Alternative — Use a Status-List Parameter Instead of Boolean

**Resolution:** Persuasive with Modification. Replace `includeUnreleasedContent` with a parameter like `publicationStatusFilter` (`0..*`, code, bound to PublicationStatus) that explicitly lists which statuses to include when resolving value set versions. If omitted, behavior is server-defined.

**Rationale:** Michael Lawley's concern that a boolean is too blunt is valid—some implementers may want to include `retired` or `unknown` status content, or exclude only `draft` while keeping `active`. A status-list parameter is more expressive and avoids future need for additional parameters. The previous `considerStatus` was rejected for naming and definition issues, not for the concept of status-based filtering itself. A better name (e.g., `publicationStatusFilter`) and clearer definition could address prior objections.

**Risks:** More complex for implementers. Requires binding to PublicationStatus which may not perfectly map to all server's internal content lifecycle concepts. The previous attempt at this approach was reverted, suggesting committee resistance to the complexity.

### Disposition C: Decline — Not Persuasive

**Resolution:** Not Persuasive. The use case can be addressed through architectural separation (authoring vs. production server instances) or through IG-level extensions like CRMI's, without adding parameters to the core `$expand` operation.

**Rationale:** As discussed in the Zulip thread, separating authoring and production content through server architecture (dev/staging/prod) is standard practice and avoids complicating the core specification. The CRMI IG already defines these parameters as extensions for those who need them. Adding to core imposes implementation burden on all servers. The 4+ years of discussion and multiple reverted resolutions suggest the community has not reached consensus on the right approach.

**Risks:** Leaves the interoperability gap unresolved for servers that serve both draft and released content from the same endpoint. CRMI-specific extensions are not universally adopted.

## Recommendation

**Disposition A: Accept `includeUnreleasedContent` as proposed.**

The parameter has been refined through extensive community discussion (4+ years, 22 comments, multiple WGM sessions). The scope has been appropriately narrowed from three parameters to one focused boolean. The CRMI alignment work at Madrid provides a solid conceptual foundation for distinguishing released from unreleased content. The boolean approach is simpler than the rejected `considerStatus` and directly addresses the core use case. It is backward-compatible (servers maintain current behavior when the parameter is omitted) and allows servers that don't support the capability to throw an exception. Given the ticket was recently requested for the 2026-02-23 TI tracker agenda and linked to the broader FHIR-54108 parameter alignment effort, this is a natural time to finalize.
