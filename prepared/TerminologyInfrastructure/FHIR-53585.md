## Description

This ticket proposes a new terminology server operation for comparing two value sets. The motivation is that profile comparison—an increasingly important activity—requires determining the relationship between value sets bound to profiled elements. The current approach of expanding both value sets and comparing membership is:

1. **Slow** — full expansion can be expensive
2. **Fails for infinite value sets** — some value sets cannot be fully expanded
3. **Fragile** — results may vary over time due to expansion parameter effects

The proposed operation accepts two value sets (by URL or as full ValueSet resources) and returns one of the following relationship outcomes:

- **identical** — same members
- **superset** — one value set contains all members of the other (and more)
- **subset** — one value set is contained entirely within the other
- **different** — members differ (optionally further classified as overlapping or disjoint)
- **unknown** — the server cannot determine the relationship

### Formal Definition

Given VS_A and VS_B:

- X = concepts in VS_A that are NOT in VS_B
- Y = concepts in VS_B that are NOT in VS_A

| X empty? | Y empty? | Result |
|----------|----------|--------|
| Yes | Yes | identical |
| Yes | No | VS_A is a subset of VS_B |
| No | Yes | VS_B is a subset of VS_A |
| No | No | different |

### Additional Constraints

- **Version resolution:** Versions must resolve the same, or `versionNeeded != false` on both code system versions.
- **Membership only:** The comparison considers only code membership — displays and properties are not relevant.
- **Expansion parameters:** Only parameters embedded in the value sets are considered, not externally supplied ones (including supplements).
- **Implementation freedom:** Servers choose how to perform the comparison; full expansion is not required.

## Proposed Dispositions

### Disposition A: Persuasive

**Rationale:** Accept the proposal as stated. The need for a server-side value set comparison operation is well-motivated by practical profile comparison requirements. The formal definition is clear and mathematically sound. The operation design appropriately gives servers implementation freedom (no requirement for full expansion) and handles edge cases (infinite value sets, unknown results). The real-world use case from Zulip (ICD-10 validation blocking) validates the practical need. The ticket refines the earlier FHIR-51053 with additional rigor on version resolution, parameter handling, and membership semantics.

**Considerations:** Would need a formal OperationDefinition resource to be authored. The relationship to FHIR-51053 should be explicitly addressed (supersede, merge, or link).

### Disposition B: Persuasive with Modification

**Rationale:** Accept the concept but require the following modifications before implementation:

1. **Reconcile with FHIR-51053** — FHIR-51053 is the original proposal for the same operation, currently "Waiting for Input." The two tickets should be formally linked, with one superseding the other. The outcome terminology differs between the two (FHIR-53585 uses "identical/superset/subset/different" while FHIR-51053 uses "equivalent/overlapping/includes/includedBy/disjoint").
2. **Define a formal OperationDefinition** — Specify whether this is a type-level operation on ValueSet (e.g., `ValueSet/$compare`) or a named system operation, along with exact parameter names, types, and cardinalities.
3. **Clarify "unknown" handling** — The proposal mentions "unknown" as an outcome but the formal definition only covers four cases. Specify when servers should return "unknown" versus an HTTP error (FHIR-51053 suggested an HTTP error with OperationOutcome).
4. **Align outcome codes** — Standardize the outcome vocabulary. Consider aligning with ConceptMap relationship codes or defining a dedicated code system.

**Considerations:** This approach ensures a complete, implementable specification while preserving the core proposal.

### Disposition C: Not Persuasive

**Rationale:** While the use case is valid, adding a new operation increases the implementation burden on terminology servers. Existing operations (`$expand` and `$validate-code`) can be combined client-side to approximate comparison for finite value sets. The "unknown" outcome for infinite or complex value sets means the operation may not reliably solve the stated problem. Additionally, a duplicate proposal already exists (FHIR-51053), suggesting the approach has not yet been sufficiently refined.

**Considerations:** This would leave the profile comparison tooling dependent on client-side expansion, which Grahame has identified as slow, incomplete, and fragile.

## Recommendation

**Disposition B: Persuasive with Modification**

The need for a dedicated value set comparison operation is clearly demonstrated — both by the formal analysis of current limitations (slow, fails on infinite sets, fragile) and by the real-world blocking issue in profile validation tooling (Zulip: ICD-10 validation). The proposal from Grahame Grieve is well-reasoned and includes a sound formal definition. However, FHIR-53585 must be reconciled with its predecessor FHIR-51053 (same reporter, same concept, currently "Waiting for Input" with workgroup feedback requesting more detail). The ticket also needs a formal OperationDefinition, clarified "unknown" vs error handling, and standardized outcome codes before it can be implemented. Accepting with these modifications ensures the specification is complete and consistent.
