## Problem Statement

The FHIR specification does not adequately address how ValueSet expansion should behave when the underlying CodeSystem is represented by one or more fragments. Specifically:

1. **Section 4.9.6.5 (Referencing Code System Fragments):** The current text states "The CodeSystem referenced in a `ValueSet.compose` may reference a CodeSystem fragment" but does not address the case where the canonical URL references a CodeSystem represented by *multiple* fragments on the server.
2. **Section 4.9.10 (Expansion):** The expansion algorithm makes no mention of fragments. An expansion based on one set of fragments may differ from an expansion based on a different set, yet nothing in the expansion parameters would identify this difference.
3. A "Request for Feedback" callout in the spec contains a placeholder link (`[link to be provided]`) that was never populated.

This thread solicited community input on the three proposed options for signaling that a fragment was used during a terminology operation:

- **Option 1 (output parameter):** Server SHALL include a new output parameter flagging which CodeSystems relied on fragments. *Favored by Grahame Grieve and Peter Jordan.*
- **Option 2 (TerminologyCapabilities check):** Clients check `TerminologyCapabilities` before invoking operations. *Universally disfavored* — places burden on clients.
- **Option 3 (extension on result):** Server includes an extension on the returned resource (e.g., expansion) flagging fragment usage. *Reuben Daniels noted this helps downstream consumers who didn't invoke the operation themselves.*

Key discussion points:
- **Elliot Silver** challenged the assumption that "fragment" always means incomplete — two fragments together could constitute the full CodeSystem.

- **Rutt Lindström** raised concerns about identifying different fragments sharing the same URL and version.

## Proposed Resolution Analysis

The resolution proposes two sets of changes:

### Change 1: Section 4.9.6.5 (Referencing Code System Fragments)
- **Remove** the sentence "The CodeSystem referenced in ValueSet.compose may reference a CodeSystem fragment."
- **Add** clarification that fragments MAY be used to generate expansions in the absence of the complete CodeSystem.
- **Add** that fragments cannot be referenced directly — ValueSet definitions reference the CodeSystem canonical URL as usual, and `CodeSystem.content = fragment` indicates the resource is a fragment.

### Change 2: Section 4.9.10 (Expansion Algorithm)
- **Add** a new requirement to the expansion algorithm to signal when fragments were used.
- Three signaling options were proposed, with community consensus favoring **Option 1** (output parameter) over Option 2 (TerminologyCapabilities) or Option 3 (extension).
- The resolution text includes placeholders ("to be defined") for the specific parameter/extension names.

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification (with refinement)
**Recommendation:** Accept the resolution's two-part approach but require the "to be defined" placeholders be resolved before application. Specifically:
- Accept the Section 4.9.6.5 text changes as proposed.
- For Section 4.9.10, adopt **Option 1** (output parameter) as the primary mechanism, defining a named output parameter (e.g., `used-codesystem-content`) that indicates the `CodeSystem.content` value for each CodeSystem used in the expansion.
- Add guidance that if the server determines the union of available fragments is functionally complete (e.g., SNOMED CT editions), the server MAY omit the fragment flag.
- Address Elliot Silver's concern that "fragment" does not always mean incomplete by refining the language.

**Rationale:** This aligns with community consensus from Zulip (Option 1 was preferred by Grahame Grieve, Peter Jordan, and others). An output parameter is consistent with existing patterns (cf. FHIR-15817 for supplements) and does not require clients to pre-check TerminologyCapabilities.

### Disposition B: Accept as Persuasive with Modification (minimal, defer signaling mechanism)
Accept only the Section 4.9.6.5 changes now. Defer the Section 4.9.10 signaling mechanism to a separate ticket or connectathon track, given:
- The "to be defined" placeholders indicate the design is not yet finalized.
- Michael Lawley's open questions about practical utility remain unanswered.
- The relationship with FHIR-48960 (fragment relationships) suggests both tickets should be resolved together.

**Rationale:** Reduces risk of premature normative-track changes. The descriptive changes in 4.9.6.5 are well-understood and uncontroversial, while the signaling mechanism needs more implementation experience.

### Disposition C: Not Persuasive
Reject the change request. The current spec intentionally leaves fragment behavior underspecified because:
- Grahame Grieve noted there is "no expected behavior" by design.
- Michael Lawley questioned whether clients can usefully act on fragment-used signals.
- The fragment concept may be better addressed through implementation guidance rather than normative spec changes.

**Rationale:** Avoids adding complexity to the expansion algorithm for a feature with limited real-world usage. However, this leaves the acknowledged spec gaps unaddressed and the broken "link to be provided" placeholder unfixed.

## Recommendation

**Disposition A: Accept as Persuasive with Modification (with refinement).**

The community has clearly identified that fragments are underspecified, and this ticket has been discussed extensively across multiple WGM sessions and Zulip threads over 16+ months. The Section 4.9.6.5 clarifications are straightforward and uncontroversial. For Section 4.9.10, Option 1 (output parameter) received the strongest community support and fits existing patterns. The resolution should be refined to: (1) name the specific output parameter, (2) incorporate Elliot Silver's feedback about fragments not always implying incompleteness, and (3) coordinate with FHIR-48960 for consistency on fragment relationship semantics.
