## Description

The submitter requests that since US Core has added a Coverage profile (as of US Core 6.1.0), the CARIN Digital Insurance Card (C4DIC) IG should base its Coverage profile on the US Core Coverage Profile rather than maintaining a custom profile. If this is not possible, an exception should be sought from Cross Group Projects.

## Resolution Description

In STU 2.0.0 the IG is pointing to US Core version 6.1.0 and 7.0.0.

Mark Roberts explained to Cross-Group Projects that the C4DIC IG developed a custom Coverage profile because STU 1.0 predated US Core 6.1.0. The plan was to keep the custom profile for the STU 1.1 update (which added SMART Health Cards/Links) and then adopt US Core Coverage and Patient profiles in STU 2.0, using extensions where needed. Brett Marquard asked why the switch couldn't happen immediately; Mark explained the urgency of publishing STU 1.1 with the SMART Health Cards/Links use case first.

### 2. [CARIN Benefit Check IG > CARIN Digital Insurance Card Experiment](https://chat.fhir.org/#narrow/stream/CARIN%20Benefit%20Check%20IG/topic/CARIN%20Digital%20Insurance%20Card%20Experiment) (2025-06-27 – 2025-08-10)
**Participants:** Joshua Kelly (Flexpa), Frank McKinney

Joshua Kelly demonstrated a real-world use case for the C4DIC Coverage profile: the US Core Coverage profile (and CARIN BB Coverage profile) lack pharmacy benefit fields (rxbin, rxpcn, rxgrp) needed for Real-Time Pharmacy Benefit Check (RTPBC) submissions. The C4DIC Coverage profile includes these fields and was used to build a reference implementation ([flexpa/rtpbc-demo](https://github.com/flexpa/rtpbc-demo)) that retrieves C4DIC Coverage and transforms it into an RTPBC-compatible Coverage. This reinforces the importance of the C4DIC Coverage profile's extensions surviving the rebase to US Core.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Confirm Resolution)
**Action:** Confirm the existing "Persuasive" resolution. The C4DIC Coverage profile in STU 2.0.0 has been rebased on US Core Coverage as provided in US Core 6.1.0 and 7.0.0, with extensions added for insurance-card-specific data elements (rxbin, rxpcn, rxgrp, etc.).

**Rationale:** The resolution description states that STU 2.0.0 already points to US Core 6.1.0 and 7.0.0, which directly addresses the submitter's request. The variance requests (FHIR-43300, FHIR-43303) were granted specifically to allow time for this transition. FHIR-53034 further confirms the move to US Core 6.1.0/7.0.0. This aligns with Cross Group Projects expectations and matches the approach taken by Da Vinci IGs.

**Remaining Work:** Verify that the STU 2.0.0 C4DIC Coverage profile structurally derives from US Core Coverage and that the pharmacy-specific extensions (needed by RTPBC and other consumers) are properly defined.

### Disposition B: Accept as Persuasive with Modification
**Action:** Accept as Persuasive with Modification, adding an explicit requirement that the C4DIC Coverage profile must formally declare `baseDefinition` pointing to the US Core Coverage profile URL, and that any C4DIC-specific data elements not in US Core Coverage be published as standalone extensions in a shared extension pack.

**Rationale:** While the resolution says STU 2.0.0 "points to" US Core 6.1.0/7.0.0, it is not entirely clear whether the Coverage profile's `baseDefinition` was changed or whether the IG merely references US Core. A modification would ensure the intent of the original request—structural derivation, not just reference—is verified and enforced.

**Remaining Work:** Confirm the StructureDefinition for C4DIC Coverage in STU 2.0.0 uses US Core Coverage as its `baseDefinition`. Document any extensions needed.

### Disposition C: Defer (Not-Persuasive)
**Action:** Mark as Not Persuasive / Considered - No Change, on the basis that the current implementation already addresses the concern.

**Rationale:** If the work group determines that STU 2.0.0 has already fully addressed this issue and no further specification changes are needed, the ticket could be closed without additional action. The resolution description indicates the change was made.

**Risk:** This may leave ambiguity about whether the profile truly derives from US Core Coverage or merely references it, and could lead to conformance confusion for implementers.

## Recommendation

**Disposition A: Accept as Persuasive (Confirm Resolution)**

The evidence strongly supports that this request has been addressed. The resolution description confirms STU 2.0.0 points to US Core 6.1.0 and 7.0.0. The companion variance tickets (FHIR-43300, FHIR-43303) laid out the explicit plan to adopt US Core profiles in STU 2.0, and FHIR-53034 confirms the formal switch. The Zulip discussion from Mark Roberts in the argonaut stream documented the clear two-phase plan (STU 1.1 with variances → STU 2.0 with US Core adoption), which has been executed. The work group should confirm the Persuasive resolution and verify the structural derivation in the published IG.
