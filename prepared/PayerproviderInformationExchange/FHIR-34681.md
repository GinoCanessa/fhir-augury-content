## Summary

The submitter (Craig Newman) identifies a gap in the CARIN Digital Insurance Card (DIC) IG: the Data Elements page distinguishes between Primary and Secondary networks (both ID and Name), but the Coverage profile uses a single `network` type for `Coverage.class` with cardinality constrained to a single occurrence. This makes it impossible to distinguish primary from secondary networks when transmitting insurance card data.

The ticket was originally dispositioned as **Considered for Future Use** in March 2022 (vote 11-0-1), noting that no insurer had provided a rationale for differentiating primary vs. secondary networks. It was reopened after STU publication in July 2022 and again after STU 1.1 publication in April 2024.

A new proposed resolution (**Persuasive with Modification**) introduces two distinct slices on `Coverage.class`:

1. **`Coverage.class:network`** — Preferred/Primary Network (0..1, Must Support)
2. **`Coverage.class:alternatenetwork`** — Alternate Network(s) for secondary or additional network tiers (0..*, Must Support)

The proposer (Mark Roberts) has requested feedback from the original submitter as of February 2026.

---

## Details

### Problem Statement
The DIC IG's Data Elements page lists separate data elements for Primary Network ID, Primary Network Name, Secondary Network ID, and Secondary Network Name. However, the Coverage profile only provides a single `network` slice on `Coverage.class` constrained to one occurrence. There is no mechanism to indicate which network is primary vs. secondary.

### Proposed Resolution
The resolution introduces a new `alternatenetwork` slice alongside the existing `network` slice:

- **`Coverage.class:network`**: Slice for the preferred/primary network; cardinality 0..1; Must Support = true
- **`Coverage.class:alternatenetwork`**: Slice for secondary/additional networks; cardinality 0..*; Must Support = true

This allows payers to transmit a single primary network and zero or more alternate networks, each clearly distinguished by slice name.

## Proposed Dispositions

### Disposition A: Accept — Persuasive with Modification (Recommended)

**Action:** Accept the proposed resolution as stated. Add two slices on `Coverage.class`:
- `Coverage.class:network` (Preferred/Primary Network, 0..1, Must Support)
- `Coverage.class:alternatenetwork` (Alternate Network(s), 0..*, Must Support)

**Rationale:** This directly addresses the original gap identified by the submitter. The Data Elements page already distinguishes primary and secondary networks, so the profile should support transmitting both. The two-slice approach is clean: a single primary network with optional secondary/alternate networks. The 0..* cardinality on `alternatenetwork` provides flexibility for multiple network tiers. This approach aligns with established patterns of slicing `Coverage.class` used in other IGs (e.g., C4BB, Da Vinci PCT per FHIR-34922). The ticket has been open since 2021 and reopened twice after STU publications, indicating persistent community need. Making both slices Must Support ensures implementers handle both scenarios.

**Impact:** Moderate. Requires updating the Coverage profile in the CARIN DIC IG to add the `alternatenetwork` slice and update the `network` slice definition. Existing implementations that only send a single network continue to work (cardinalities are 0..1 and 0..*). Receivers must be updated to recognize the new slice.

---

### Disposition B: Alternative — Persuasive with Modification (Use Extension Instead)

**Action:** Instead of adding a new `alternatenetwork` slice on `Coverage.class`, allow multiple occurrences of the existing `network` slice (change cardinality from 0..1 to 0..*) and add an extension (e.g., `network-tier` or `network-priority`) to distinguish primary from secondary.

**Rationale:** This approach avoids proliferating slices on `Coverage.class` and instead uses an extension to carry the primary/secondary designation. It keeps the slice structure simpler and could be more flexible if additional network tier classifications emerge in the future. However, it introduces an extension that implementers must understand, and the semantic meaning is less immediately obvious from the slice name alone.

**Impact:** Moderate. Requires defining a new extension, updating the Coverage profile cardinality, and updating the Data Elements mapping. More complex for implementers than the slice-based approach.

---

### Disposition C: Decline — Not Persuasive

**Action:** Decline the change request. Maintain the current single `network` slice on `Coverage.class` with 0..1 cardinality.

**Rationale:** The original 2022 disposition noted that no insurer provided a rationale for differentiating primary vs. secondary networks, and the IG already supports passing multiple networks. If the use case is truly not needed by payers, adding complexity to the profile is unwarranted. Implementers could use `Coverage.order` or other mechanisms outside the DIC profile for coordination of benefits scenarios.

**Impact:** Low (no change). However, this leaves the documented gap between Data Elements and the profile unresolved and ignores the fact that the ticket has been reopened twice, suggesting ongoing community interest.

---

## Recommendation

**Disposition A: Accept — Persuasive with Modification** is recommended.

The proposed two-slice approach (`network` + `alternatenetwork`) is a straightforward, well-precedented solution that directly resolves the gap between the Data Elements page and the Coverage profile. The ticket's history—reopened twice after STU publications over four years—demonstrates persistent community need. The resolution is backwards-compatible (existing single-network implementations remain valid), aligns with slicing patterns used in other FHIR IGs, and both slices are clearly named to convey their purpose without requiring extensions. The workgroup should adopt this resolution.
