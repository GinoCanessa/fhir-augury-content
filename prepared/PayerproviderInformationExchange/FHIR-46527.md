## Description

The CDex SDC QuestionnaireResponse Profile is derived from US Core, which in turn derives from SDC. However, this profile is **invalid** because there is no QuestionnaireResponse profile in US Core 3.1.1.

Meanwhile, the DTR QuestionnaireResponse profiles (including the adaptive form variant) derive directly from the SDC QuestionnaireResponse profile—a different derivation chain for the same Burden Reduction use case.

### Problems Identified

1. **Inconsistent dependencies** between DTR and CDex DaVinci IGs for the same Burden Reduction use case.
2. **Invalid profile** — CDex references a US Core QuestionnaireResponse that does not exist in US Core 3.1.1.
3. **Unnecessarily complex dependencies** across the IG landscape.

### Proposal

1. Change the CDex QuestionnaireResponse Profile dependency to derive from DTR QuestionnaireResponse Profiles instead.
2. DTR needs a variance request to US Core QuestionnaireResponse for versions 6.1 and 7.0.0 before publication.

> **Note:** The Signature datatype is in flux in FHIR R6 and may affect both SDC and CDex.

### Resolution Description

> "Change as proposed. This is a forward breaking change."

## Analysis

This ticket addresses a fundamental **profile derivation inconsistency** in the Da Vinci Burden Reduction ecosystem. The CDex SDC QuestionnaireResponse Profile claims derivation from a US Core QuestionnaireResponse that didn't exist in US Core 3.1.1, making the profile technically invalid. Meanwhile, DTR's QuestionnaireResponse profiles derive directly from SDC, creating two incompatible derivation chains for what should be a unified use case.

The resolution ("change as proposed") is classified as a **forward breaking change**, meaning implementations currently referencing the CDex QuestionnaireResponse profile will need to update their dependencies. The cross-IG nature of this change (CDex, DTR, SDC, US Core) requires coordination with the Burden Reduction team (Jeff, Lloyd) as noted in both Jira comments and Zulip discussions.

As of mid-2025, the disposition remains pending offline discussion with the Burden Reduction team, suggesting the technical direction is agreed upon but implementation coordination is still in progress.

## Proposed Dispositions

### Disposition A: Persuasive (Accept as Proposed)
Accept the resolution as stated: re-derive CDex QuestionnaireResponse profiles from DTR QuestionnaireResponse profiles, and file a DTR variance request against US Core QuestionnaireResponse 6.1 and 7.0.0. This aligns the dependency chains, eliminates the invalid US Core 3.1.1 derivation, and simplifies the IG relationship. The forward breaking change is acknowledged and accepted.

- **Pros:** Resolves the invalid profile issue cleanly; simplifies the dependency graph; aligns with the agreed Persuasive resolution.
- **Cons:** Forward breaking change for existing CDex implementers; requires cross-IG coordination with DTR/SDC teams; Signature datatype flux in R6 may require further changes.

### Disposition B: Persuasive with Modification — Phase the Change
Accept the direction but require a phased approach: (1) first publish CDex with a deprecation notice on the current QuestionnaireResponse profile, (2) coordinate DTR variance request and profile updates, (3) complete the switch in a subsequent CDex version. This gives implementers time to adapt.

- **Pros:** Reduces immediate disruption; allows coordinated timing with DTR publication; gives implementers a migration window.
- **Cons:** Extends the timeline during which the invalid profile exists; adds complexity to CDex publication planning; the "disposition pending" status already suggests delays.

### Disposition C: Not Persuasive — Defer Until R6 Signature Stabilizes
Defer this change until the FHIR R6 Signature datatype stabilizes, since the ticket itself notes Signature is in flux and may affect both SDC and CDex. Avoid making a breaking change now that may need to be revised again when R6 publishes.

- **Pros:** Avoids potential double-breaking-change; waits for stable R6 foundation.
- **Cons:** Leaves the invalid CDex profile in place indefinitely; does not resolve the inconsistent dependency chain; R6 timeline is uncertain and could delay resolution significantly.

## Recommendation

**Disposition A: Persuasive (Accept as Proposed)**

The profile derivation is technically invalid today, the resolution has already been agreed as Persuasive, and the proposed fix is well-defined. The forward breaking change is justified because the current state is incorrect. The Burden Reduction team coordination (Jeff, Lloyd) is already in progress per the Zulip threads. Waiting for R6 Signature stabilization (Disposition C) would leave a known-invalid profile in place for an indeterminate period, and phasing (Disposition B) adds unnecessary complexity to what is fundamentally a dependency correction. The work group should accept the change as proposed and prioritize the offline coordination with Burden Reduction to unblock implementation.
