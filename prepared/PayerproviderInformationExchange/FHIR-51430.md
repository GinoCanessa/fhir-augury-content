## Description

The LOINC-defined ValueSet URI `http://loinc.org/vs/valid-hl7-attachment-requests` does not resolve in the IG publisher. The LOINC terminology server hosts this ValueSet at `https://fhir.loinc.org/ValueSet/valid-hl7-attachment-requests`, but the IG publisher tooling cannot directly access LOINC's terminology server to expand and render the ValueSet.

The reporter proposes two possible paths:
1. **Preferred:** Follow up with the IG publisher to fix support for directly resolving LOINC ValueSets from LOINC's terminology server, then replace the CDex and PDex locally-maintained ValueSets with the canonical LOINC URI.
2. **Fallback:** If publisher support cannot be added, update the CDex and PDex ValueSets with the latest LOINC attachment codes from the most recent LOINC publication, or migrate the ValueSet to THO (terminology.hl7.org).

## Analysis

This is a long-standing interoperability gap between LOINC's terminology server and the FHIR IG publisher tooling. The core problem is that `http://loinc.org/vs/valid-hl7-attachment-requests` — the canonical LOINC ValueSet for attachment request codes — cannot be resolved, expanded, or validated by the IG publisher because:

1. LOINC does not distribute this ValueSet as a FHIR package.
2. The IG publisher does not natively query LOINC's terminology server (`fhir.loinc.org`) for ValueSet definitions.
3. The ValueSet uses a custom LOINC property (`ValidHL7AttachmentRequest`) that is not part of the standard LOINC filter set in the FHIR specification.

CDex and PAS have been working around this since 2024 by maintaining locally enumerated copies of the LOINC attachment codes (see FHIR-44871, FHIR-45742). This ticket seeks to either fix the root cause (publisher + LOINC interop) or formalize the workaround.

The PIE work group has explicitly deferred disposition pending IG publisher updates.

## Proposed Dispositions

### Disposition A: Persuasive — Accept as Correction, Conditional on Publisher Fix

**Resolution:** Accept the ticket and defer the code change until the IG publisher supports directly resolving LOINC-hosted ValueSets. Once the publisher is updated, replace the locally-maintained CDex (and PDex) LOINC attachment codes ValueSets with the canonical LOINC URI `http://loinc.org/vs/valid-hl7-attachment-requests`. Track the publisher dependency and revisit when tooling support is confirmed.

**Rationale:** This is the cleanest long-term solution — it eliminates the maintenance burden of keeping enumerated copies in sync with LOINC releases. However, it carries timeline risk as the publisher fix has been pending since 2022 and depends on coordination between HL7 tooling and LOINC.

### Disposition B: Persuasive with Modification — Update Local ValueSets Now, Plan for LOINC URI Later

**Resolution:** Immediately update the CDex and PDex locally-defined LOINC attachment codes ValueSets with the latest codes from the current LOINC publication. Add a comment in the ValueSet noting the intent to replace with the canonical LOINC URI once publisher support is available. File or link a separate tracker for the publisher fix.

**Rationale:** This is a pragmatic approach that ensures the attachment codes are current and correct without waiting on external tooling dependencies. It maintains the existing workaround pattern established in FHIR-44871 and FHIR-45742 while preserving the intent to migrate.

### Disposition C: Not Persuasive — Defer Entirely to Publisher/LOINC Coordination

**Resolution:** Close as Not Persuasive on the grounds that this is an IG publisher tooling issue, not a CDex specification issue. The existing locally-defined ValueSets are functional. Direct the reporter to file against the IG publisher or coordinate with LOINC to publish a FHIR package.

**Rationale:** The current workaround (locally enumerated codes) functions correctly. The root cause is not in the CDex specification but in the IG publisher's inability to resolve LOINC terminology server resources. However, this disposition risks the local ValueSets becoming stale as LOINC updates its attachment codes.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The work group has already established the pattern of maintaining local ValueSets as a workaround (FHIR-44871, FHIR-45742), and the IG publisher fix has been pending for over three years with no clear timeline. Updating the local ValueSets now ensures correctness for implementers while preserving the migration path to the canonical LOINC URI. This also aligns with the work group's current stance of "waiting for IG publisher updates" — by decoupling the immediate code update from the publisher dependency, the specification stays current regardless of when the tooling issue is resolved.
