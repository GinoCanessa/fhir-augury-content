## Description Summary

The ticket proposes introducing guidance in the UDAP SSRA Guide for **federated client registration** — the ability to share a client registration record across multiple authorization servers. The submitter identifies several advantages of federation:

- Consolidation of registration workflows and business logic across authorization servers
- Minimized discrepancies in client identities across servers running the same software
- Improved ability for trust communities to detect misconfigurations or malicious actors
- Reduced maintenance burden on client developers managing multiple client versions

Specific proposals include:
1. Adding metadata to the authorization server's metadata endpoint indicating whether federated client records are shared with other authorization servers
2. Encouraging trust communities to define minimum (and possibly maximum) scope sets for clients participating in data exchange

## Analysis

### Key Considerations

1. **Scope Evolution:** This ticket was originally deemed "Considered for Future Use" (15-0-0) during STU2 ballot but has now been re-opened post STU2 publication, signaling workgroup readiness to address it.

2. **Guidance vs. Technical Solution:** The workgroup consistently agreed that creating new technical mechanisms is out of scope, but **non-normative guidance** for trust communities and implementers is appropriate.

3. **Related Work Completed:** FHIR-48986 (now published) established guidance on the client-FHIR server-authorization server relationship, providing a foundation. Federated registration guidance would be a natural extension.

4. **Practical Need:** Multiple commenters acknowledged this pattern already exists in practice (e.g., shared authorization servers across FHIR versions). Formalizing guidance would reduce implementation confusion.

5. **Open Question:** John Moehrke's suggestion to explicitly mark federation as out-of-scope (with explanation) remains unaddressed and should be considered as an alternative disposition.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Add non-normative guidance to the UDAP SSRA Guide addressing federated client registration patterns. This guidance should:
- State that client registrations SHOULD NOT have unintended side effects on other authorization servers
- Recommend that authorization servers maintain independent client records per logical FHIR server when sharing infrastructure
- Encourage trust communities to define scope requirements for cross-server client participation
- Reference the relationship guidance already established via FHIR-48986

**Rationale:** The workgroup has already acknowledged the need (re-opened post STU2). The scope is limited to guidance (not new technical mechanisms), consistent with prior workgroup consensus. FHIR-48986's resolution provides a foundation to build upon.

### Disposition B: Considered for Future Use
**Action:** Defer to a future ballot cycle while the community gains more implementation experience with UDAP STU2, particularly with the FHIR-48986 guidance now in effect. Request that implementers report real-world federation challenges to inform future guidance.

**Rationale:** While the need is acknowledged, the ticket was re-opened only recently. Allowing the published STU2 guidance (especially FHIR-48986) to be adopted first would provide concrete implementation feedback to shape more targeted guidance.

### Disposition C: Not Persuasive
**Action:** Close without change. Explicitly note in the specification (per John Moehrke's suggestion) that federated client registration is a legitimate use-case but is out of scope for the UDAP IG, and point implementers to general OAuth 2.0 federation patterns.

**Rationale:** The UDAP IG's scope is intentionally focused on bilateral client-server registration. Federation introduces complexity (trust propagation, scope harmonization, conflict resolution) that may be better addressed by dedicated federation specifications or trust community policies rather than the UDAP IG itself.

## Recommendation

**Disposition A: Persuasive with Modification**

The workgroup's decision to re-open this ticket post STU2 publication signals clear intent to address it. The prior consensus (October–November 2024) already converged on adding guidance rather than technical mechanisms. With FHIR-48986 now published and establishing the client-server-authorization server relationship model, this ticket represents the natural next step: guidance on what happens when that relationship spans multiple authorization servers. The 15-0-0 prior vote for "future use" and the subsequent re-opening demonstrate unanimous interest. Non-normative guidance carries low risk and addresses a real-world pattern that implementers are already encountering.
