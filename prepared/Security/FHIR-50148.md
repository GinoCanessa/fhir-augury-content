## Problem Statement

During connectathon testing, excessively large JWTs were encountered. The root cause is that UDAP authentication currently requires embedding the full X.509 certificate chain via the `x5c` header parameter in every authentication JWT. Since certificates are relatively static content, this significantly bloats the JWT payload transmitted on every request.

The submitter proposes allowing clients to register a JKU (JSON Web Key Set URL) at registration time that points to their certificate. Servers could then trust the client-provided JKU and cache the JKU response per RFC requirements. Subsequent authentication requests would reference the JKU instead of embedding the full certificate chain, significantly reducing JWT sizes while maintaining UDAP's security and trust model.

## Analysis

### Core Technical Issue
UDAP currently mandates including the full X.509 certificate chain (`x5c`) in the JWT header for every authentication request. This is effective for establishing trust but results in large JWTs. The JKU mechanism (already defined in JWS RFC 7515) allows the certificate to be referenced by URL rather than embedded, with caching per HTTP cache-control headers.

### Key Design Question (Unresolved)
Joe Lamy's question remains central: **SHALL or MAY?** If JKU usage is optional (MAY), servers must still handle large `x5c`-bearing JWTs. If mandatory (SHALL), it's a breaking change for existing UDAP implementations.

### Precedent
FHIR-32184 already addressed unclear JKU text in the Bulk Data spec (resolved as Persuasive), indicating the community has worked with JKU semantics before. The CDS Hooks spec already supports `jku` in its JWT security model, providing a reference implementation pattern.

### Reopening Context
The ticket was deferred pre-STU2 due to time constraints and scope concerns, not technical objections. It was reopened post-STU2 publication, indicating the workgroup considers it a valid enhancement for the next cycle.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Summary:** Accept the request but specify JKU as MAY (optional), not SHALL (mandatory). Add guidance that servers SHOULD support JKU-based authentication in addition to x5c, and that clients MAY register a JKU at registration time. Include normative language that when a client has registered a JKU, the server SHALL accept authentication JWTs using either `jku` or `x5c`.

**Rationale:** This provides the performance benefit for implementations that choose to adopt it without breaking existing `x5c`-only implementations. It addresses Joe Lamy's concern by making it opt-in while still enabling the optimization. The STU2 publication window is now past, giving time for proper specification and connectathon testing.

**Trade-offs:** Servers must support both paths. The "problem" of large JWTs persists for clients that don't adopt JKU.

### Disposition B: Persuasive
**Summary:** Accept the request as stated — enable clients to register their JKU at registration time and use JKUs in subsequent authentication requests. Add a SHOULD-level requirement for clients to use JKU when registered, with servers required to support both mechanisms.

**Rationale:** The connectathon evidence of JWT size issues is concrete. JKU is already a well-defined mechanism in RFC 7515 and is used in CDS Hooks security. Aligning UDAP with this pattern reduces payload sizes and leverages standard caching behavior. The prior deferral was about timing, not merit.

**Trade-offs:** Stronger push toward JKU adoption may require more implementation effort from servers that currently only validate `x5c`.

### Disposition C: Deferred / Not Persuasive
**Summary:** Defer again or reject, noting that the current `x5c` mechanism works correctly and JWT size has not been shown to cause failures — only larger payloads. Wait for more connectathon data on the severity of the issue.

**Rationale:** The submitter was asked to assess criticality with their technical team (June 2025), and that assessment may not yet be complete. Changing the authentication mechanism has significant security implications that warrant careful analysis, including cache poisoning risks for JKU endpoints.

**Trade-offs:** Leaves the large JWT issue unaddressed. The ticket has already been deferred once.

---

## Recommendation

**Disposition A: Persuasive with Modification** — Accept with JKU as MAY/optional rather than mandatory. This pragmatically addresses the JWT size concern identified during connectathon testing without breaking existing implementations. The JKU mechanism is well-established in RFC 7515 and already used in CDS Hooks security. Making it optional resolves Joe Lamy's design question, allows incremental adoption, and gives the community a clear path to reduce authentication token sizes when both client and server support it. The post-STU2 timing provides an appropriate window for this enhancement.
