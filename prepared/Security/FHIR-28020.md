## Details

The submitter requests that concrete use-case examples be added to the DS4P IG's implementation section. The existing text states:

> *"By defining profiles and maturity models for the security labeling system, subsets of the security labeling systems can be selected in a way that is tailored to suit specific application areas and use-cases."*

The request asks that one or more of the "potential use cases" referenced by this text be explicitly included as illustrative examples.

## Proposed Dispositions

### Disposition A: Accept — Add Use-Case Examples from Known Implementations

**Resolution:** Applied — Persuasive

Add one or more concrete use-case examples to the DS4P IG's implementation section, drawing from documented real-world experience:

1. **Confidentiality labeling (N vs R):** An EHR system uses the DS4P security labeling vocabulary to distinguish Normal (N) from Restricted (R) data when sharing documents via C-CDA exchange. This is the most widely adopted DS4P use case.
2. **Security Labeling Service (SLS) pipeline:** A healthcare organization deploys an SLS that examines clinical documents for sensitive topics (e.g., substance abuse, HIV), assigns sensitivity labels, and then an Access Control enforcement engine determines whether to block, redact, or release the document with an appropriate confidentiality code.
3. **Context-aware labeling for FHIR resources:** A FHIR server integrates an SLS that accepts individual resources or bundles along with transaction context (client identity, purpose of use), producing labeled resources with confidentiality codes and handling instructions tailored to the recipient.

These examples are grounded in VA pilot implementations and FHIR Connectathon demonstrations discussed in community forums.

### Disposition B: Accept with Modification — Add a Placeholder Section with Pointers

**Resolution:** Applied — Persuasive with Modification

Rather than embedding full use-case narratives, add a new "Implementation Examples" subsection that:

1. Briefly describes the SLS processing model (label → decide → enforce) as an archetypal use case.
2. Provides hyperlinks to the existing HL7 SLS specification and relevant Connectathon track materials.
3. Includes an editorial note inviting implementers to submit additional use cases for future versions.

This approach addresses the submitter's request while keeping the IG concise and reducing maintenance burden.

### Disposition C: Decline — Reaffirm Considered for Future Use

**Resolution:** Not Applied — Considered for Future Use

Reaffirm the 2022 ballot resolution. The prior vote (3-0-1) concluded that no sufficiently mature, broadly deployed implementation examples exist to include in the normative text at this time. The IG can be updated in a future cycle when more implementers have published profiles based on the DS4P labeling system. Re-opening this ticket without new implementation evidence does not change the underlying rationale.

## Recommendation

**Disposition A: Accept — Add Use-Case Examples from Known Implementations**

Since the ticket was originally resolved as "Considered for Future Use" and then explicitly re-opened post-STU1 publication, the workgroup has signaled intent to revisit. The 2019 Zulip discussion and prior Connectathon demonstrations (VA pilot, Mohammad Jafari's FHIR demo) provide concrete, community-documented use-case material that did not exist at the time of the original ballot. Adding these examples would directly address the submitter's request, improve the IG's implementer guidance, and align with the workgroup's decision to reconsider. The reporter had already accepted the original deferral, so including examples now would exceed expectations and strengthen the specification.
