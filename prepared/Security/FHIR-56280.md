## Problem Statement

The UDAP Security IG's discovery section currently states that `{baseURL}` represents "the base FHIR URL for the FHIR server." However, UDAP security may be relevant for servers that are not FHIR servers. The ticket cites CDS Hooks as a concrete example: if CDS Hooks adopted UDAP in place of its current security process, the CDS service endpoint is not a FHIR server, yet UDAP discovery should still work. The reporter suspects there will be additional non-FHIR use cases.

The core issue is that the current specification language unnecessarily constrains the UDAP discovery mechanism to FHIR servers, limiting its applicability to other health IT systems that could benefit from UDAP trust establishment.

## Analysis

This ticket identifies a genuine limitation in the current UDAP Security IG specification language. The specification's assumption that `{baseURL}` is always a FHIR server URL creates an unnecessary coupling that could hinder UDAP adoption in broader health IT contexts.

**Evidence of real-world need:**
- Active prototyping of UDAP-enabled CDS Hooks (Joseph Shook, FAST Security)
- Extensive discussion of UDAP/CDS Hooks security gaps (Da Vinci CRD working group)
- Isaac Vetter's explicit statement that CDS Hooks must work both inside and outside PKI-based trust communities
- Precedent in FHIR-33291 where UDAP was broadened to support non-UDAP-aware OIDC providers

**Scope of change:**
The fix is likely a wording change in the discovery section, generalizing `{baseURL}` from "the base FHIR URL for the FHIR server" to a broader definition that accommodates any server supporting UDAP discovery. The `.well-known/udap` mechanism itself is already protocol-agnostic — the constraint is purely in the descriptive text.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Change:** Revise the discovery section to generalize `{baseURL}` from "the base FHIR URL for the FHIR server" to "the base URL of the server hosting the resource or service." Add a note clarifying that while the IG is primarily targeted at FHIR servers, the UDAP discovery mechanism can be used by any server that wishes to support UDAP workflows (e.g., CDS Hooks services, OIDC providers). Review the entire discovery section for other FHIR-server-specific language that could be similarly generalized.

**Rationale:** This directly addresses the reporter's concern while preserving backward compatibility. It aligns with the precedent set by FHIR-33291 (broadening tiered OAuth beyond UDAP-aware providers) and the real-world trajectory shown in Zulip discussions where implementers are already experimenting with UDAP in non-FHIR contexts. The `.well-known/udap` mechanism is inherently server-type-agnostic, so the specification language should reflect this.

### Disposition B: Persuasive
**Change:** Accept the concern as valid and acknowledge that `{baseURL}` should not be limited to FHIR server URLs. Leave the specific wording change to the editors, noting that the intent is to allow any server to host a `.well-known/udap` discovery endpoint.

**Rationale:** The issue is clearly valid but the exact wording may need careful consideration to ensure consistency across the entire IG. The editors may need to audit multiple sections for similar FHIR-server-specific assumptions, making a broad editorial pass more appropriate than a targeted fix.

### Disposition C: Not Persuasive
**Change:** No change. The UDAP Security IG is a FHIR IG and appropriately scoped to FHIR servers. Non-FHIR use cases should define their own discovery mechanisms or reference the underlying UDAP.org specifications directly.

**Rationale:** The IG is titled "US UDAP Security (FHIR)" and is published through the FHIR IG process. Expanding scope to non-FHIR servers could create confusion about which specification governs discovery for non-FHIR systems. CDS Hooks or other non-FHIR specifications could define their own profiles of UDAP discovery if needed.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The evidence strongly supports this change. The UDAP discovery mechanism (`.well-known/udap`) is already protocol-agnostic in practice — the FHIR-specific language is an artificial constraint in the descriptive text. Active implementation efforts (CDS Hooks + UDAP prototyping) and extensive community discussion demonstrate real-world demand. The change is low-risk (wording clarification, not architectural), maintains backward compatibility, and follows established precedent (FHIR-33291). Keeping the FHIR-specific language would force non-FHIR implementers to either ignore the specification text or create redundant parallel specifications — neither outcome serves the community well.
