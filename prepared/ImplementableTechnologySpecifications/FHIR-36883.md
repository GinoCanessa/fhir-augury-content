## Details

The reporter (Mohamed Omar) noted that some legend items in the resource schema views—specifically `R!` (required) and `C?` (conditional)—and the lock icon (in XML views) are not fully documented. The legend on the JSON representation page is missing an entry for `C?`.

A comment from **mckenzie** (2022-05-11) confirmed that `R!` and the lock icon are already documented, but agreed that `C?` needs to be added to the documentation on the JSON page. The ticket was transferred to the ITS work group as the page owner.

The fix is straightforward: add a `C?` entry to the legend/key on the JSON representation page (likely `json.html` or its source template) to match the existing XML page documentation.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Add C? to JSON legend)

**Action:** Add the `C?` (conditional) entry to the legend/key section on the JSON representation page in the FHIR specification.

**Rationale:** The commenter (mckenzie) already confirmed that `R!` and the lock icon are documented but `C?` is missing from the JSON page. This is a documentation gap—the symbol appears in resource schemas but is not explained on the JSON page. Prior tickets (FHIR-3516, FHIR-14520) show established precedent for fixing legend documentation issues, and both were resolved as Persuasive. The fix is minimal (adding one line to the legend) and improves clarity for implementers.

**Scope:** Add a legend entry for `C?` on the JSON representation page, with a definition such as: "*C? — This element is conditionally required (required if a condition is met)*", consistent with how it is documented elsewhere in the specification.

### Disposition B: Accept as Persuasive with Modification (Audit all format page legends for completeness)

**Action:** Add `C?` to the JSON legend, and also perform a broader audit of all format representation pages (JSON, XML, Turtle) to ensure all legend symbols are consistently documented across all pages.

**Rationale:** The pattern of incremental legend fixes (FHIR-3516 for lock icon, FHIR-14520 for `<D>` definition, now FHIR-36883 for `C?`) suggests the legends across format pages may have drifted out of sync. A one-time audit would prevent future tickets of the same kind.

**Scope:** Add `C?` to the JSON page and cross-check all legend entries across JSON, XML, and Turtle format pages for consistency.

### Disposition C: Decline as Not Persuasive

**Action:** Close without change.

**Rationale:** One could argue `C?` is documented elsewhere in the specification (e.g., in the general resource definitions or conformance pages) and that implementers can find the meaning through those references. However, given that other symbols are already in the JSON legend and `C?` is not, this argument is weak.

## Recommendation

**Disposition A: Accept as Persuasive** — This is a clear, uncontested documentation gap confirmed by the FHIR editorial team (mckenzie). The fix is trivial (one legend entry), has strong precedent from similar accepted tickets, and directly improves the implementer experience on the JSON representation page.
