## Summary

Both XML and JSON canonicalization algorithms defined in the FHIR specification lack a variant that excludes the `.signature` element from the canonicalized output. This makes Bundle.signature effectively unusable for digital signatures, since the signature must be calculated over the Bundle content *before* inserting the signature value — creating a circular dependency. The reporter requests a new canonicalization algorithm that excludes `.signature` to enable Bundle.signature to function correctly for any Bundle type, and potentially for the `#document` canonicalization variant as well.

A secondary question raised in comments is whether the exclusion should cover the entire `Signature` datatype or only `Signature.data`, leaving `Signature.type`, `Signature.when`, and `Signature.who` within the signed content.

## Analysis

### Core Problem
Bundle.signature is currently non-functional for digital signatures because no canonicalization algorithm in the FHIR specification excludes the `.signature` element. This creates a circular dependency: you cannot compute a signature over a Bundle that already contains the signature. The `#document` canonicalization variant also does not exclude `.signature`, making document signing equally broken.

### Key Open Question
The central design question is **scope of exclusion**: exclude only `Signature.data` (Grahame's position) or the entire `Signature` datatype (John's position). This affects whether `Signature.who`, `Signature.when`, and `Signature.type` are covered by the digital signature.

### Maturity & Impact
- **Priority is Highest** — John explicitly states this must be resolved for R6.
- Bundle.signature is currently Trial Use and targeted for normative in R6; this ticket is a prerequisite.
- No known implementations of Bundle.signature exist (per John's assessment), largely because this canonicalization gap makes it impossible.
- This applies to both JSON and XML formats; JSON canonicalization should reference RFC 8785.
- Multiple downstream use cases are blocked: IPS signing, CDex signatures, document bundles.

### Coordination
Grahame has proposed (in the "Signing and narrative" Zulip thread) that FHIR-45023 and ~11 related tickets be moved to FHIR-I for coordinated resolution, with dispositions proposed as a block and a collaborative PR to the base spec.

## Proposed Dispositions

### Disposition A: Persuasive — Define canonicalization excluding entire Signature datatype
**Resolution:** Define a new canonicalization URI (e.g., `http://hl7.org/fhir/canonicalization/json#bundle` and XML equivalent) that excludes the entire `Bundle.signature` element (the full `Signature` datatype). Apply the same pattern to the `#document` canonicalization variant. Document that `Signature.who`, `.when`, and `.type` are populated for FHIR convenience but are not covered by the digital signature; consumers wanting trustworthy metadata should inspect the signature payload directly.

**Rationale:** Aligns with John Moehrke's and Gino Canessa's positions. Simpler implementation — the entire `.signature` is removed before hashing. Avoids the complexity of partial Signature element inclusion, which creates ambiguity about what is trustworthy. Consistent with existing digital signature standards (JWS/JAdES) where signer identity and timestamp are carried inside the signature itself.

### Disposition B: Persuasive with Modification — Define canonicalization excluding only Signature.data
**Resolution:** Define a new canonicalization URI that excludes only `Bundle.signature.data` (the actual digital signature bytes), keeping `Signature.type`, `Signature.when`, `Signature.who`, and `Signature.onBehalfOf` within the signed content. Document that these elements are signed and therefore tamper-evident for consumers who do not wish to inspect the signature payload directly.

**Rationale:** Aligns with Grahame Grieve's position. Allows FHIR-processing applications to rely on Signature metadata without full signature verification. Provides defense-in-depth by signing the metadata about the signature alongside the resource content. May introduce complexity when `.who` references a value that differs from the `iss`/`sub` claims inside the signature.

### Disposition C: Persuasive with Modification — Define both canonicalizations
**Resolution:** Define two canonicalization URIs — one excluding the entire `Signature` datatype and one excluding only `Signature.data`. Recommend one as the default (e.g., exclude entire Signature) but allow implementations to choose based on their trust model. The canonicalization method is recorded in the JWS header and (per FHIR-51098) potentially in a new Signature element.

**Rationale:** Accommodates both use cases and avoids a winner-take-all decision. However, adds complexity to the specification and to consumers who must support both variants. Risk of interoperability issues if senders and receivers expect different canonicalization methods.

---

## Recommendation

**Disposition A: Persuasive — Define canonicalization excluding entire Signature datatype.**

This is the strongest option because:
1. It resolves the immediate blocker simply and completely — no partial-element-exclusion complexity.
2. It has majority support among the active discussants (John Moehrke as reporter, Gino Canessa as tiebreaker), with Grahame indicating willingness to accept either approach.
3. It aligns with digital signature best practices: signer identity, timestamp, and purpose should be trusted from within the signature itself, not from unsigned or partially-signed FHIR metadata.
4. It avoids creating ambiguity about the trustworthiness of `Signature.who`/`.when`/`.type` — these remain as FHIR convenience elements but are clearly documented as not covered by the signature.
5. It is simpler to implement and easier to explain, which is important for a feature (Bundle.signature) that has seen zero known implementations to date.

This disposition should be coordinated with the related cluster of canonicalization/signature tickets (FHIR-51098, FHIR-50934, FHIR-46789, etc.) as proposed by Grahame Grieve for joint FHIR-I resolution.
