## Summary

This ticket identifies two ambiguities in the FHIRPath specification's [quantity equivalence](https://build.fhir.org/ig/HL7/FHIRPath/#quantity-equivalence) section:

1. **Missing equivalence algorithm**: The quantity equivalence section describes how to handle different units but never explicitly states *how* equivalence is determined. The reporter proposes stating: "the two quantity values should be converted to the same unit if possible under UCUM, and then equivalence is determined by the equivalence of the numeric parts using the definition of decimal equivalence."

2. **Contradictory precision rules**: The section states comparisons use "the most granular unit of either input," but the decimal equivalence section says "comparison is done on values rounded to the precision of the least precise operand." These rules conflict — converting to the most granular unit and then rounding to the least precise operand effectively undoes the conversion. Example: `4 'g' ~ 4040 'mg'` converts to `4000 'mg' ~ 4040 'mg'`, but under decimal equivalence, `4000` has only 1 significant digit so they'd be equivalent — making the conversion pointless.

The reporter proposes either removing the "most granular unit" language or clarifying that quantity equivalence should not follow the same rules as decimal equivalence.

## Proposed Dispositions

### Disposition A: Persuasive — Accept as stated with clarifying spec text
**Action**: Add explicit language to the quantity equivalence section stating: (1) quantities should be converted to a common unit under UCUM before comparison, and (2) equivalence is then determined using decimal equivalence semantics. Remove or replace the ambiguous "most granular unit" language.

**Rationale**: The reporter correctly identifies that the current spec text is incomplete (no algorithm for determining equivalence) and contradictory (most granular unit vs. least precise operand). The community discussion confirms that multiple FHIRPath engines implement this differently. The Persuasive resolution is appropriate — the spec needs this clarification.

**Risk**: Low for the principle, but the specific algorithm details (base unit vs. granular unit, precision preservation) still need resolution and are being addressed in FHIR-56190.

---

### Disposition B: Persuasive — Defer to FHIR-56190 for comprehensive resolution
**Action**: Acknowledge FHIR-53249 as Persuasive but note that the comprehensive fix is being tracked in [FHIR-56190](https://jira.hl7.org/browse/FHIR-56190), which subsumes this ticket along with FHIR-49525 and FHIR-53250. The preview branch ([BP-2026-03-quantity-preview](https://build.fhir.org/ig/HL7/FHIRPath/branches/BP-2026-03-quantity-preview/)) already contains the proposed resolution including: removal of "most granular unit" language, explicit equivalence algorithm, unit conversion section, and extensive examples.

**Rationale**: Brian Postlethwaite's comprehensive rewrite in FHIR-56190 addresses all the concerns in this ticket plus related ambiguities. Reviewing this ticket in isolation risks creating partial fixes that conflict with the broader rewrite. The Zulip discussion shows active consensus-building around the preview branch.

**Risk**: FHIR-56190 is large and may take time to finalize. Some details (calendar unit implicit conversion, "least granular" definition) are still being debated.

---

### Disposition C: Persuasive with Modification — Adopt base-unit approach with explicit precision rules
**Action**: Modify the resolution to specify: (1) implementations that support UCUM SHALL convert arguments to common UCUM base units for equivalence comparison, (2) the equivalence comparison then uses decimal equivalence semantics, (3) precision of the original operands is preserved during unit conversion (i.e., converting `4 g` to `4000 mg` retains the original precision of 1 digit). Remove the "most granular unit" phrasing entirely.

**Rationale**: This aligns with Bryn Rhodes' proposed resolution for FHIR-49525 and addresses Paul Lynch's concern about inconsistencies (the base-unit approach makes `4 'g' ~ 4040 'mg'` and `4 'kg' ~ 4040 'g'` behave consistently). Adding explicit precision preservation rules resolves Jing Tang's core concern.

**Risk**: The base-unit approach still has edge cases (e.g., Brian Kaney's concern about `1 year ~ 11 months → true`). Calendar unit handling requires additional rules beyond pure UCUM conversion.

## Recommendation

**Disposition B: Persuasive — Defer to FHIR-56190 for comprehensive resolution.**

The community discussion clearly shows this ticket has been subsumed by FHIR-56190, which provides a comprehensive rewrite of quantity handling in the FHIRPath spec. Brian Postlethwaite's preview branch already addresses all the concerns raised in FHIR-53249, and the ticket was removed from the block vote in favor of the broader effort. Multiple implementers (fhirpath.js, .NET, Rust, Java/CQL) are actively reviewing the preview branch. Accepting this ticket individually risks creating partial or conflicting changes when the comprehensive solution is actively in progress. The workgroup should confirm the Persuasive resolution and note that the implementation is tracked in FHIR-56190.
