# Ticket Review: FHIR-20342

**Title:** ShEx representation of Extension.
**Status:** Triaged **Priority:** Medium **Type:** Change Request
**Work Group:** Implementable Technology Specifications **Specification:** FHIR Core (FHIR)
**Reporter:** Harold Solbrig **Assignee:** Unassigned
**Created:** 2019-02-03 **Updated:** 2019-02-04 **Labels:** None

---

## Summary

Harold Solbrig reports a mismatch between the ShEx (Shape Expressions) representation of the FHIR Extension element and its RDF rendering. The ShEx schema defines `Extension.url` as a direct `xsd:string`, while the RDF serialization treats it as an extensible value (wrapping the URL in a `fhir:value` property). This inconsistency causes FHIR Extension instances to fail ShEx validation. The reporter notes that either the ShEx schema must be updated to accommodate the extensible value pattern, or the RDF rendering must be corrected to use a plain string — but the current state is broken in either case.

## Details

**Description:**

The ShEx representation of the Extension element defines `Extension.url` as a direct URI typed as `xsd:string`:

```
<Extension> CLOSED {
  fhir:Element.id @<string>?
  fhir:Element.extension @<Extension>*
  fhir:Extension.url xsd:string;
  ( fhir:Extension.valueBase64Binary @ | ...
```

However, the RDF rendering treats the URL as an extensible value, wrapping it in a `fhir:value` structure:

```
fhir:Extension.url [ fhir:value "http://hl7.org/fhir/StructureDefinition/observation-geneticsGene" ];
fhir:Extension.valueCodeableConcept [
```

The reporter states: "IF the Extension url is extensible, the ShEx needs to change. If not, the RDF rendering needs fixed. In either case, extensions as they exist today do not pass the ShEx evaluator."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Align the ShEx schema for Extension with the actual RDF serialization by updating the ShEx definition of `Extension.url` to match the extensible value pattern used in RDF output (i.e., change from `fhir:Extension.url xsd:string` to a shape that accepts the `fhir:value` wrapper structure). Verify that the updated ShEx schema correctly validates existing FHIR Extension instances in RDF form.

#### Justification

The RDF serialization is the canonical representation that implementers produce and consume. Aligning the ShEx schema to match the RDF output ensures that conformant FHIR RDF data passes validation. This is consistent with the resolution of the closely related ticket FHIR-25619 (ShEx for FHIR primitive types is incorrect, resolved as Persuasive), which also addressed mismatches between ShEx schemas and actual FHIR data representations. The reporter (Harold Solbrig) is a recognized expert in ShEx and RDF for FHIR, lending credibility to the identified inconsistency.

---

### Disposition B: Alternative Approach — Fix the RDF Rendering

#### Proposal

Instead of modifying the ShEx schema, correct the RDF rendering of `Extension.url` to use a plain `xsd:string` value directly (without the `fhir:value` wrapper), matching the current ShEx definition. This would mean treating `Extension.url` consistently as a non-extensible simple value in both RDF output and ShEx validation.

#### Justification

The `Extension.url` is a fixed, canonical URL that identifies the extension definition — it is not a value that should be extensible or carry sub-extensions. Treating it as a plain string in RDF (matching the ShEx) is semantically cleaner and simpler. However, this approach requires changes to RDF serialization logic and may affect existing RDF consumers, so it would need careful impact analysis and potentially a breaking-change migration path.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that this ticket is from 2019 and the ShEx/RDF tooling landscape for FHIR has evolved significantly since then. The issue may have been addressed by subsequent work on ShEx generation (e.g., FHIR-25619) or by changes in the FHIR RDF specification.

#### Justification

The ticket has been in Triaged status since 2019 with no comments or further activity, suggesting it may no longer be a pressing concern. FHIR-25619 (ShEx for FHIR primitive types is incorrect), which was filed by the same reporter and addresses a closely related ShEx generation issue, was resolved as Persuasive and published. It is possible that the fix for FHIR-25619 or other subsequent ShEx generation improvements have addressed this inconsistency. Before acting, the work group could verify whether the current ShEx output for Extension still exhibits this mismatch.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested)

The core issue identified in this ticket — a mismatch between the ShEx schema and the RDF serialization of `Extension.url` — is a concrete, well-documented validation failure. The reporter, Harold Solbrig, is a leading authority on ShEx and FHIR RDF, and the problem statement is clear and actionable. The resolution of the closely related FHIR-25619 (which fixed ShEx for primitive types) established a precedent for correcting ShEx schemas to match actual FHIR data representations. Accepting this ticket and aligning the ShEx definition of Extension with the RDF output is the most consistent and lowest-risk approach, as it avoids breaking changes to the RDF serialization that downstream consumers depend on. Even if subsequent ShEx generation work has partially addressed this, formally resolving the ticket ensures the fix is tracked and verified.
