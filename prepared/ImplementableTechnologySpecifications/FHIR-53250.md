## Problem Statement

The FHIRPath specification has conflicting rules around calendar duration equivalence that break the transitivity of the equivalence operator (`~`). The issue is:

- `1 year ~ 1 'a' ~ 365.25 'd'` (UCUM-defined conversion)
- `1 year ~ 365 days ~ 365 'd'` (calendar duration conversion factor in `toQuantity`)
- But `365.25 'd' !~ 365 'd'`, so equivalence is not transitive

The root cause is that the spec says *"Implementations that do support units shall do so as specified by [UCUM] as well as the calendar durations as defined in the toQuantity function"* (§6.1.2.1 Quantity Equivalence), which allows calendar duration conversion factors to be used implicitly during equivalence checks. This creates ambiguity: `1 year ~ 365 days` can be either true or false depending on the conversion path chosen, making equivalence non-deterministic.

The reporter proposes that calendar duration conversion factors (years → days, months → days, etc.) should only be usable via explicit `toQuantity()` calls, not implicitly during equivalence/equality/comparison operations.

## Proposed Resolution (from Disposition)

The disposition proposes five changes to the FHIRPath spec, classified as **Persuasive with Modification**:

1. **§6.6 Math, §6.1.1.1 Quantity Equality, §6.1.2.1 Quantity Equivalence, §6.2 Comparison** — Change the unit support statement to: *"Implementations that do support units shall do so as specified by [UCUM]. Note that calendar unit conversions for months and years must be performed explicitly, as defined in the [toQuantity(unit)] function."*

2. **§6.7 Date/Time Arithmetic** — Update the threshold from "above seconds" to "above days (or weeks)" for when definite-quantity durations signal an error, and add a note that calendar durations below months are both equal and equivalent to UCUM definite-time duration units (ignoring leap seconds for practical reasons).

3. **§5.5.7.1 toQuantity(unit)** — Clarify that calendar duration conversion factors must be invoked explicitly via `toQuantity(unit)`. Attempts to perform comparisons or arithmetic requiring implicit calendar duration conversions result in empty (`{}`).

4. **Add illustrative examples** to §6.7, including:
   - `1 year ~ 1 'a'` → true (by definition in equivalence)
   - `1 year ~ 12 months` → false (unsupported implicit conversion)
   - `1 year + 12 months` → empty (calendar unit conversion must be explicit)
   - `@2025-11-29 + 1 'a'` → error (UCUM definite-time duration above days in date/time arithmetic)
   - `@2025-11-29 + 1 year` → `@2026-11-29`

5. **§4.1.8.1 Time-valued Quantities** — Add a note about equality/equivalence relationships between calendar and definite duration quantities.

**Late update (2026-02-23, Bryn Rhodes comment):** Proposed additional clarification that conversions not involving variable-length periods (years/months) can be transitive. Example: `1 year.toQuantity('minutes')` is invalid, but `1 day.toQuantity('minutes')` is valid.

**Note:** The resolution description now references **FHIR-56190** — a large consolidation ticket by Brian Postlethwaite that subsumes this disposition into an extensive set of quantity/unit/calendar clarifications across the FHIRPath spec.

### 1. [fhirpath > Transitivity and determinism of equivalence for durations](https://chat.fhir.org/#narrow/stream/fhirpath/topic/Transitivity%20and%20determinism%20of%20equivalence%20for%20durations)
- **Participants:** Jing Tang, Paul Lynch, Bryn Rhodes, Gino Canessa, Brian Postlethwaite, Richard Townley-O'Neill
- **Period:** 2025-09-24 to 2026-01-12 (44 messages)
- **Summary:** The originating discussion. Jing Tang raised that FHIRPath equivalence for calendar durations is neither transitive nor deterministic. Bryn Rhodes argued `1 year ~ 365 days` sneaks in an approximate conversion and should not hold. The community converged on requiring explicit conversion for years/months via `toQuantity()`, while allowing implicit conversion for days/weeks and below. Key consensus points:
  - `1 year ~ 12 months` → false (implicit conversion not allowed)
  - `1 year.toQuantity('months') ~ 12 months` → true (explicit conversion OK)
  - Calendar durations for days and below can implicitly convert to/from UCUM definite-time units
  - Brian Postlethwaite disagreed with disallowing comparison between calendar units of different types (e.g., years vs months) but accepted the calendar-to-UCUM restriction

## Proposed Dispositions

### Option A: Accept as Persuasive with Modification (per existing disposition + FHIR-56190 subsumption)
**Disposition:** Persuasive with Modification
**Rationale:** The core issue is valid — calendar duration equivalence is non-deterministic and non-transitive as currently specified. The proposed changes (requiring explicit `toQuantity()` for years/months conversions, allowing implicit conversion for days and below, updating the "above seconds" threshold to "above days") are well-reasoned and have broad community support from Bryn Rhodes, Jing Tang, and Brian Postlethwaite. The disposition has been refined through extensive Zulip discussion and Jira comments. However, since FHIR-56190 now consolidates and extends these changes into a comprehensive quantity handling overhaul, the specific text changes from this ticket's disposition should be deferred to FHIR-56190 to avoid duplication/conflict.
**Action:** Accept the issue as persuasive with modification. Note that the resolution is being applied through FHIR-56190 which provides the comprehensive spec text changes. Mark as addressed-by/subsumed-by FHIR-56190.

### Option B: Accept as Persuasive (original proposal)
**Disposition:** Persuasive
**Rationale:** The reporter's original proposal — remove the sentence allowing implicit calendar duration conversion in equivalence and clarify that `toQuantity` conversion factors are for explicit use only — is sound and addresses the core transitivity/determinism problem. This is a narrower acceptance that validates the problem statement without committing to all five specific text changes in the disposition.
**Action:** Accept as persuasive. Defer detailed spec text to FHIR-56190.

### Option C: Not Persuasive with Modification
**Disposition:** Not Persuasive with Modification
**Rationale:** While the transitivity concern is theoretically valid, calendar duration conversions being available implicitly may be what some implementers expect (Brian Postlethwaite expressed reluctance to disallow calendar-unit-to-calendar-unit comparison). The practical impact is limited since most implementations already handle this inconsistently. Instead of the sweeping changes proposed, add only a clarifying note that equivalence involving calendar durations with different units may not be transitive, and recommend explicit conversion via `toQuantity()`.
**Action:** Add a warning note to the spec about calendar duration equivalence limitations without changing the operational semantics.

## Recommendation

**Option A: Accept as Persuasive with Modification** is recommended.

The problem is real and well-documented — the spec as written produces non-deterministic and non-transitive equivalence results for calendar durations, which is a fundamental correctness issue. There is strong community consensus (across 100+ Zulip messages and 14 Jira comments) that explicit conversion via `toQuantity()` is the right approach for years and months, while days and below can safely convert implicitly. The disposition has been refined through multiple rounds of feedback. Since FHIR-56190 now subsumes this work into a comprehensive overhaul, accepting FHIR-53250 as persuasive with modification and deferring the detailed text changes to FHIR-56190 avoids duplication while properly crediting the original issue.
