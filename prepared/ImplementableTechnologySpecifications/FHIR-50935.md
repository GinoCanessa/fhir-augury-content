## Description

The submitter notes that the **Canonical JSON** section of the FHIR specification defines a set of canonicalization URIs (used to specify what elements are covered by a signature) presented as a plain table/list. The request asks whether these canonicalization URIs should instead be defined as a formal **ValueSet**, which would make them machine-processable, bindable to elements, and consistent with how other coded values are managed in FHIR.

## Zulip Discussions

### [Security and Privacy > Signature](https://chat.fhir.org/#narrow/stream/Security%20and%20Privacy/topic/Signature)

**Participants:** Grahame Grieve, John Moehrke, Gino Canessa
**Date:** 2025-06-13

Key discussion points relevant to FHIR-50935:

1. **Canonicalization in targetFormat as MIME parameter:** Grahame proposed embedding the canonicalization method as a MIME parameter on `targetFormat` (e.g., `application/fhir+json; canonicalization=http://hl7.org/fhir/canonicalization/json`). Gino Canessa initially supported this approach.

2. **Separate element preferred:** John Moehrke argued for keeping `targetFormat` purely as a MIME type, preferring a new separate element. Gino ultimately agreed, noting that a MIME parameter could be lost during reserialization across formats. The consensus shifted toward a **new element** on the Signature datatype (FHIR-51098 was filed for this).

3. **ValueSet for FHIR MIME types:** John Moehrke explicitly raised: *"do we have a valueSet of mime-types that are legitimate FHIR mime-types? the targetElement is bound to all possible mime-types, would be good to have that limited."* This directly aligns with the concern in FHIR-50935 about formalizing the canonicalization values.

4. **Cardinality discussion:** John noted that XML Signature and JOSE both support multiple canonicalization methods within a single signature, suggesting 0..* cardinality. However, the group settled on **0..1** for simplicity, with replication at the Signature datatype level for multiple signatures.

## Proposed Dispositions

### Disposition A: Persuasive — Create a ValueSet for Canonicalization URIs

**Action:** Define a formal ValueSet containing the canonicalization URIs currently listed as a table in the Canonical JSON section (e.g., `http://hl7.org/fhir/canonicalization/json`, `http://hl7.org/fhir/canonicalization/json#data`, `http://hl7.org/fhir/canonicalization/json#document`, etc.). This ValueSet would be bindable to the new canonicalization element proposed in FHIR-51098 or usable alongside existing Signature elements.

**Rationale:** Formalizing these values as a ValueSet is consistent with FHIR conventions for coded concepts, makes them machine-processable, and supports validation tooling. The Zulip discussion showed consensus that canonicalization scope is important metadata. A ValueSet would complement the new element proposed in FHIR-51098.

### Disposition B: Persuasive with Modification — Defer to FHIR-51098 and Define Binding There

**Action:** Acknowledge the concern as valid but defer the specific binding/ValueSet definition to be handled as part of FHIR-51098 (which proposes adding a new `canonicalization` element to Signature). When that element is added, its binding should reference a formal ValueSet of canonicalization URIs rather than a free-text or unbound code. Close FHIR-50935 as addressed by FHIR-51098.

**Rationale:** The Zulip discussion evolved beyond the original question and led to FHIR-51098, which covers the broader design question of how to represent canonicalization in Signature. Defining the ValueSet in isolation without the element to bind it to is premature. The two tickets should be resolved together to ensure a coherent design.

### Disposition C: Not Persuasive — The Current Presentation is Adequate

**Action:** Leave the canonicalization URIs as an informational table in the specification. The values are not codes in the traditional FHIR sense — they are URIs embedded in signature headers (e.g., JWS `canon` parameter). A ValueSet binding would not apply naturally since these values are not used in FHIR resource elements today.

**Rationale:** The canonicalization URIs are part of the signature process (embedded in JWS headers), not values bound to FHIR elements. Until a new element is added to Signature (per FHIR-51098), there is no element to bind a ValueSet to. The table presentation is adequate for the current specification structure.

## Recommendation

**Disposition B: Persuasive with Modification — Defer to FHIR-51098 and define the binding there.**

This is the most pragmatic approach. The submitter's underlying concern — that canonicalization URIs should be formalized — is valid and well-supported by the Zulip discussion. However, the Zulip conversation revealed that the community is moving toward adding a dedicated `canonicalization` element to the Signature datatype (FHIR-51098). Defining a ValueSet makes most sense when paired with that new element and its binding. Resolving this ticket independently risks creating a ValueSet without a clear home. The recommended path is to note the dependency on FHIR-51098, ensure that when the new element is added it includes a proper ValueSet binding for canonicalization URIs, and close FHIR-50935 as addressed by that work.
