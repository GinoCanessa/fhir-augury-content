## Details

The proposed changes touch virtually every section of the FHIRPath spec that deals with quantities. Key areas include:

### Quantity Literals & Time-Valued Quantities
- Highlight case-sensitivity earlier; simplify/refactor duplicated concepts
- Label UCUM units `'a'` and `'mo'` as "mean" values (mean Julian year/month)
- Include system values for the two code systems used in FHIR content
- Note that precision/rounding are part of specific operations, not conversion itself

### Conversions (toQuantity / convertsToQuantity)
- Add implicit conversion from `long` to `Quantity`
- When converting decimal/integer/long to Quantity, the resulting unit is the default unit `'1'`
- New `toQuantity` overload: e.g., `45.toQuantity('g')` to create a quantity with a specified unit
- **New dedicated section on unit conversions** referenced from all relevant sections, including:
  - Time-valued unit conversion sub-section with UCUM conversion factors from the UCUM spec
  - Cross-system conversion rules (UCUM ↔ calendar, with shortest-path for `a`/`mo`)
  - Clarification that conversion is not required for date/time operations except with partial dates

### Equality (`=`), Equivalence (`~`), and Comparison
- Quantities require commensurable UCUM units (same scale and comparable)
- **Removes** the "uses the most granular unit" language; replaced with clearer conversion rules
- Equality: quantities must be converted to the same unit; then decimal equality rules apply
- Equivalence: conversion to same or common unit, then decimal equivalence rules (trailing zeros ignored); precision/rounding are part of equivalence, not conversion
- Time-valued quantities at year/month level across code systems require explicit `toQuantity` conversion for equality; they can be compared via equivalence
- Extensive worked examples with reasoning added throughout

### Arithmetic (Addition, Subtraction, Multiplication, Division)
- Add `long` support across all arithmetic operations via implicit conversion
- Clarify implicit conversion rules (numeric + quantity interactions)
- Multiplication: empty result for calendar quantity multiplication (except unit `'1'`); explain quantity × quantity
- Division: result always decimal except with quantities; UCUM specification rules for resulting units; empty result for calendar quantity division (except unit `'1'`)
- Addition/Subtraction: resulting unit is that of the left argument; cross-system conversion rules; reference to date/time arithmetic section
- Systems not supporting UCUM shall return empty if they cannot convert correctly

### Date/Time Arithmetic
- Merge redundant bullet list and table into a single table
- Include all supported units (UCUM and calendar) and input datatypes
- Clarify behavior with partial dates (unit conversion may be required)
- Decimal portion ignored except for seconds/milliseconds precision (produces a warning)
- Note cyclic time wrapping for Time datatypes
- Many more worked examples for addition and subtraction, including partial dates

## Proposed Dispositions

### Disposition A: Accept as Persuasive
**Action:** Accept the ticket as Persuasive and apply the proposed changes from the `BP-2026-03-quantity-preview` branch to the FHIRPath specification.

**Rationale:** The proposal is extraordinarily thorough. Brian Postlethwaite has done the heavy lifting: a complete preview branch with every change highlighted, extensive worked examples with reasoning, cross-engine testing, and an AI-assisted review document. The changes address long-standing ambiguities in quantity handling that have led to engine divergence (documented via testing across all major FHIRPath engines). The proposal consolidates three existing Persuasive/Persuasive-with-Modification tickets (FHIR-49525, FHIR-53249, FHIR-53250), and community feedback from multiple implementers (Rust, Java, .NET perspectives) has been actively incorporated. Mark the three subsumed tickets as resolved by reference to FHIR-56190.

**Remaining work:** Address Brian Kaney's specific concerns during application — particularly (1) formally defining the unit granularity selection algorithm for edge cases, (2) deciding whether `Number / Quantity → reciprocal unit` should be optional conformance, and (3) confirming the whitespace normalization intent (collapse sequences vs. character-by-character).

### Disposition B: Persuasive with Modification — Scope the "shall" vs "should" conformance levels
**Action:** Accept as Persuasive with Modification, explicitly stratifying the new requirements into "shall support" (basic same-unit operations, calendar unit conversions) and "should support" (full UCUM algebra, reciprocal units, composed unit handling).

**Rationale:** The current FHIRPath spec already has a two-tier conformance model for UCUM support ("shall support" same-unit operations; "should support" full UCUM functionality with empty as fallback). This massive change set could inadvertently raise the minimum conformance bar, particularly for engines that do not have a full UCUM layer (e.g., the Rust engine noted it does not yet support Quantity × Quantity). By explicitly marking which new behaviors are required vs. recommended, the workgroup avoids creating a de facto requirement that all engines implement full derived-unit handling. This addresses Brian Kaney's concern that reciprocal units and composed unit algebra are a "high bar" while still capturing the full intent of the cleanup. Mark the three subsumed tickets as resolved by reference.

### Disposition C: Not Persuasive — Too large for a single ballot cycle
**Action:** Decline as Not Persuasive, requesting the submitter break the change into smaller, independently reviewable tickets.

**Rationale:** This is self-described as a "VERY large set of changes" touching nearly every quantity-related section of the FHIRPath spec. Reviewing this as a single unit makes it difficult for workgroup members to evaluate each change on its merits, and risks introducing unintended normative consequences. Bryn Rhodes has already noted that FHIR-49525's resolution may not be fully reflected in the branch. The community discussion shows several open technical questions (unit granularity algorithm, reciprocal units, whitespace normalization) that would benefit from focused, incremental resolution. Recommend the submitter create separate tickets for: (a) quantity literal/conversion cleanup, (b) equality/equivalence rules, (c) arithmetic rules, (d) date/time arithmetic, and re-submit for focused review.

## Recommendation

**Disposition A: Accept as Persuasive** — with the caveat that Brian Kaney's feedback items should be addressed during application.

This ticket represents months of careful implementation-informed analysis across all FHIRPath engines, with a complete ready-to-merge branch. The alternative of breaking it into smaller tickets (Disposition C) risks losing the coherence of the changes — the equality, equivalence, comparison, and arithmetic sections are deeply interdependent and benefit from being reviewed as a unified whole. The conformance tiering (Disposition B) is a valid concern but can be addressed as an editorial pass during application rather than requiring a separate ballot cycle. The three subsumed tickets already carry Persuasive or Persuasive-with-Modification resolutions, confirming community consensus on the direction. The workgroup should accept this and work with Brian Postlethwaite and Brian Kaney to resolve the remaining edge-case questions during the editorial process.
