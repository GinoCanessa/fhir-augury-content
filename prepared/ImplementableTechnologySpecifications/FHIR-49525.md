## Problem Statement

The FHIRPath specification does not clearly define what unit the result of a Quantity-based arithmetic operation should have. For example, `1 'kg' + 100 'g'` could yield either `1.100 'kg'` or `1100 'g'`. The submitter argues that the spec should be explicit about this behavior and recommends that when a specific unit is needed, `toQuantity(unit)` should be used to convert the result.

## Resolution Description

The resolution has been updated to **defer to [FHIR-56190](https://jira.hl7.org/browse/FHIR-56190)** ("Extensive clarifications of handling quantities, and specifically unit conversions and calendar/time valued quantities"), which is a comprehensive rewrite of the Quantity handling sections in FHIRPath.

The original resolution (prior to deferral) proposed:
1. Adding a cross-reference to the Date/Time Arithmetic section at the end of the first sentence in the [Math section](https://build.fhir.org/ig/HL7/FHIRPath/#math-1).
2. Retaining the existing rule: *"The unit of the result will be the most granular unit of either input."*
3. Adding a clarifying note: the most granular unit SHALL be determined by comparing the conversion factor of the unit to the UCUM base unit, selecting the largest factor.
4. Recommending `toQuantity(unit)` for explicit unit control, e.g., `(1 'cm' + 20 'mm').toQuantity('m')`.

## Zulip Discussions

### [fhirpath > FHIR-49525 Quantity Math](https://chat.fhir.org/#narrow/stream/fhirpath/topic/FHIR-49525%20Quantity%20Math)
**Participants:** Brian Postlethwaite, Vasilii Kupriakov, Paul Lynch
**Period:** 2026-02-26 to 2026-04-01 (37 messages)

Key discussion points:
- **Left-unit rule:** Brian Postlethwaite observed that almost all engines return the unit of the left argument for addition/subtraction (e.g., `20 'mm' + 1 'cm'` → result in `mm`), except Firely which returns UCUM base units. He initially favored documenting "most granular unit" but flipped to strongly preferring the left-unit rule to match existing implementations.
- **Granularity ambiguity:** Vasilii Kupriakov raised that choosing "most granular" is difficult for non-SI units (e.g., `[ft_i]` vs `[ft_us]`). Paul Lynch proposed converting to base units and comparing conversion factors to determine granularity.
- **Mixed type arithmetic:** Brian proposed allowing `1 'cm' + 1` (implicit conversion of integer to quantity), but Paul Lynch argued this should be an error since the units are not commensurate. Alternative workarounds discussed include `1 * 1 'cm'` and Quantity instance creation.
- **`toQuantity` for unit assignment:** Discussion about whether `1.toQuantity('cm')` should work to assign a unit to a dimensionless value. Paul argued it should only work for commensurable unit conversions, suggesting a separate "setUnit" function instead.
- **Multiplication/division units:** Agreed that multiplication/division don't need same-unit requirements; unit algebra follows UCUM rules.

### [fhirpath > Quanity equivalence unit granularity](https://chat.fhir.org/#narrow/stream/fhirpath/topic/Quanity%20equivalence%20unit%20granularity)
**Participants:** Brian Postlethwaite, Bryn Rhodes, Vasilii Kupriakov, Paul Lynch, Brian Kaney, Grahame Grieve, Yury Sedinkin, Jing Tang
**Period:** 2025-11-22 to 2026-04-06 (30 messages)

Key discussion points:
- **Comprehensive rewrite:** Brian Postlethwaite completed a large proposed cleanup of quantity handling at [BP-2026-03-quantity-preview branch](https://build.fhir.org/ig/HL7/FHIRPath/branches/BP-2026-03-quantity-preview/), resulting in [FHIR-56190](https://jira.hl7.org/browse/FHIR-56190).
- **Implementation divergence:** Testing across engines (Java, fhirpath.js, .NET, CQL, Rust) revealed significant divergence in quantity arithmetic, unit conversion, equivalence, and precision handling.
- **Rust engine feedback (Brian Kaney):** Raised concerns about `1 year ~ 11 months → true` being clinically dangerous, the lack of a formal algorithm for "least granular unit" selection, and the high implementation bar for reciprocal units (`2 / 2 'cm' → 1 '/cm'`).
- **Calendar vs UCUM durations:** Cross-system comparisons (e.g., `1 'wk' ~ 1 week`) should be valid per the conversion factor table.
- **Whitespace equivalence:** Debated whether whitespace normalization should collapse sequences; current spec treats each whitespace character individually, not collapsed.
- **UCUM conformance levels:** Spec currently has SHALL for same-unit operations and SHOULD for full UCUM functionality, with empty return permitted when conversion is unsupported.

## Proposed Dispositions

### Disposition A: Accept as Not Persuasive with Modification (Defer to FHIR-56190)

Affirm the current "Not Persuasive with Modification" resolution and formally defer the substantive changes to [FHIR-56190](https://jira.hl7.org/browse/FHIR-56190), which provides a comprehensive rewrite of all Quantity handling in FHIRPath — covering arithmetic result units, equivalence, comparison, calendar durations, and `toQuantity` overloads. The resolution for FHIR-49525 would simply note that the concern is fully addressed by the disposition of FHIR-56190.

- **Pros:** Avoids duplicating or conflicting with the comprehensive work already done in FHIR-56190. Brian Postlethwaite has explicitly stated (in Zulip on 2026-04-02) that FHIR-49525 should be "redundant" once FHIR-56190 is applied. The preview branch already addresses unit result rules, `toQuantity` guidance, and conversion factor algorithms.
- **Cons:** If FHIR-56190 stalls or is modified significantly, this ticket's concern could remain unaddressed. The submitter's original request was relatively narrow (just clarify result unit of arithmetic), but the resolution now depends on a much larger change.

### Disposition B: Persuasive with Modification — Apply Minimal Clarification Now

Accept the ticket as Persuasive with Modification and apply a focused, standalone change: strengthen the existing "most granular unit" rule in the Math section to use normative language (SHALL), add a note explaining how to determine granularity via UCUM base unit conversion factors, and add a recommendation to use `toQuantity(unit)` for explicit unit control. Do not wait for FHIR-56190.

- **Pros:** Addresses the submitter's core concern immediately without depending on the larger FHIR-56190 effort. The submitter (bvdh) accepted the "most granular unit" rule as sufficient (2025-09-23 comment). Provides a clear, implementable rule.
- **Cons:** May conflict with FHIR-56190's approach, which proposes the left-operand unit rule for addition/subtraction (matching what most engines actually do) rather than "most granular unit." Brian Postlethwaite's testing showed the "most granular unit" rule is not what implementations actually follow — most use the left argument's unit. Applying a different rule now would create inconsistency.

### Disposition C: Not Persuasive — Existing Spec is Sufficient

Close as Not Persuasive. The N1 ballot spec already contains the sentence *"The unit of the result will be the most granular unit of either input"* in the Math section, which directly answers the submitter's question. The `toQuantity` function already exists for explicit unit conversion. No change needed.

- **Pros:** Simplest path forward; the spec already addresses the concern.
- **Cons:** Does not address the ambiguity in how to determine "most granular" for non-SI units, the divergence between the spec rule and actual implementations (most use left-operand unit), or the lack of normative language. The submitter explicitly stated (2025-01-29) that the original resolution did not address the topic. Ignores the significant implementation divergence documented in Zulip discussions.

---

## Recommendation

**Disposition A: Accept as Not Persuasive with Modification (Defer to FHIR-56190)**

This is the most practical path forward. The concern raised in FHIR-49525 — unclear result units for Quantity arithmetic — is fully subsumed by the comprehensive Quantity rewrite in [FHIR-56190](https://jira.hl7.org/browse/FHIR-56190), which has already been accepted as Persuasive and has a complete preview branch with extensive clarifications, examples, and test expressions. Brian Postlethwaite's work in FHIR-56190 not only addresses the unit-of-result question but also resolves the related ambiguities in equivalence, comparison, calendar durations, and mixed-type arithmetic that emerged from this ticket's discussion. Applying a narrow fix now (Disposition B) risks conflicting with FHIR-56190's approach (left-operand unit vs most-granular unit), while doing nothing (Disposition C) ignores documented implementation divergence across engines. The workgroup should confirm that FHIR-56190's disposition explicitly covers the FHIR-49525 concern and update this ticket's resolution description to reference it clearly.
