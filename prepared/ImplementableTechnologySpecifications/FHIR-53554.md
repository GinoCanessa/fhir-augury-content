## Summary

FHIRPath does not precisely define what constitutes a "character" in its string operations. The specification uses terms like "length", "Unicode character", "index", and "case" without specifying whether these refer to code units, code points, Unicode scalar values, or grapheme clusters. Additionally, the `\uXXXX` escape syntax only covers the Basic Multilingual Plane (U+0000–U+FFFF) and includes the surrogate range (U+D800–U+DFFF), which is problematic for non-UTF-16-based implementations (C, C++, Python, Rust, Swift, Go).

The submitter proposes: (1) forbid surrogates in `\uXXXX` escapes, (2) define strings as sequences of Unicode scalar values, (3) specify that `length()`, `indexOf()`, `substring()`, etc. count Unicode scalar values, and (4) optionally add a `\u{hex}` escape syntax for non-BMP characters.

---

## Resolution Details

The resolution ("Persuasive with Modification") accepts the core concern but takes a different approach. Rather than forbidding surrogates in escapes, the disposition:

1. **Declares FHIRPath strings as UTF-8 encoded**, consistent with XML, JSON, and the base FHIR spec.
2. **Defines `\uXXXX` escapes as UTF-16 code units**, requiring valid surrogate pairs to be combined into a single Unicode scalar value (e.g., `'\uD83D\uDD25'` → U+1F525 🔥).
3. **Defines "character" as "Unicode scalar value"** throughout the spec, clarifying that `length()`, `indexOf()`, `substring()`, `toChars()`, and the indexer all operate on scalar values, not grapheme clusters or encoding-specific code units.
4. **Adds a new section "Unicode and String Operations"** with a table demonstrating precomposed vs. combining forms and surrogate pair behavior.
5. **Defers the `\u{hex}` extended escape syntax** to a future version of FHIRPath.

The submitter (Vasilii Kupriakov) has expressed ongoing concerns that requiring surrogate pair processing imposes UTF-16 complexity on non-UTF-16 languages, and has advocated for simply forbidding the surrogate range in `\uXXXX` escapes instead. Brian Postlethwaite notes that most existing engines already handle surrogate pairs, and the disposition aligns with how JSON and most modern languages process these escapes.

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification (current resolution)
Accept the current disposition text which defines FHIRPath strings as UTF-8 encoded sequences of Unicode scalar values, treats `\uXXXX` escapes as UTF-16 code units with mandatory surrogate pair processing, and clarifies that all string functions (`length()`, `indexOf()`, `substring()`, `toChars()`) operate on Unicode scalar values. This adds the "Unicode and String Operations" section with worked examples and defers extended escape syntax (`\u{hex}`) to a future version.

- **Pros:** Aligns with JSON string processing rules and how most current FHIRPath engines (Java, C#, JavaScript) already behave. Provides clear, detailed specification text. Consistent with the Unicode Standard's definition of UTF-16 encoding.
- **Cons:** Imposes UTF-16 surrogate pair handling on implementations in non-UTF-16 languages (Python, Go, Rust, Swift, C/C++). The original submitter has explicitly objected to this approach. Ongoing discussion on Zulip suggests the text is not yet finalized.

### Disposition B: Accept as Persuasive with Modification (forbid surrogates in escapes)
Accept the core changes (UTF-8 encoding, strings as sequences of Unicode scalar values, functions operate on scalar values) but modify the `\uXXXX` escape handling to **forbid** the surrogate range [U+D800–U+DFFF] entirely. Non-BMP characters can appear in strings directly (e.g., `'😀'`) but cannot be entered via `\u` escapes until a future syntax extension (such as `\u{hex}`) is introduced.

- **Pros:** Eliminates UTF-16 surrogate complexity from all implementations. Every `\uXXXX` escape maps directly to a single Unicode scalar value. Simpler for non-UTF-16 languages. Aligns with the submitter's original proposal.
- **Cons:** Cannot encode non-BMP characters in escape sequences (only via literal characters). Diverges from JSON's approach to `\u` escapes. May require engines that currently process surrogate pairs to add validation rejecting them.

### Disposition C: Not Persuasive
Decline the change. The current FHIRPath specification is adequate and implementers can interpret "character" based on their platform's conventions.

- **Pros:** No specification changes needed. No risk of breaking existing implementations.
- **Cons:** Leaves fundamental ambiguity in string function semantics. Different implementations may return different results for `length()`, `indexOf()`, etc. on the same input. Does not address the real interoperability concern raised. Active community discussion shows this is a genuine gap.

---

## Recommendation

**Disposition A: Accept as Persuasive with Modification (current resolution).**

The current disposition provides comprehensive, well-structured specification text that resolves the core ambiguity (what is a "character") by defining it as a Unicode scalar value. While the submitter's concern about UTF-16 surrogate complexity in non-UTF-16 languages is valid, Brian Postlethwaite has demonstrated that most existing engines already handle surrogate pairs, and the approach is consistent with JSON's string processing model which FHIRPath implementations must already support. The workgroup should note the ongoing refinement (Brian commented "we're still refining this much further" on 2026-03-18) and may want to confirm the final text incorporates any remaining feedback from the Zulip discussion before formally closing.
