## Description

Grahame Grieve reports that `MessageDefinition.event[x]` has a mismatch between its definition text and its allowed types. The current definition reads:

> "Event code or link to the EventDefinition."

The allowed type is a choice between `Coding` and `uri`. However, since EventDefinition is a canonical resource, the `uri` type should be `canonical` if the definition text is correct. Alternatively, if the definition is too narrow, the types should be extended by adding `canonical(EventDefinition)` and the definition text should be clarified.

## Analysis

### Core Issue
There is a type/definition mismatch on `MessageDefinition.event[x]`. The definition text says "Event code or link to the **EventDefinition**", implying a typed reference to the EventDefinition canonical resource. However, the type is `Coding | uri`, where `uri` is an untyped, non-version-specific reference — inconsistent with how canonical resources should be referenced in FHIR.

### Historical Context
This inconsistency dates back to FHIR-13849 (R4 era, 2017), which changed MessageDefinition.event from `Coding` to `Coding | uri`. Meanwhile, the sibling element `MessageHeader.event[x]` was updated via FHIR-39635 (2023) to `Coding | canonical(EventDefinition, SubscriptionTopic)`. Lloyd McKenzie explicitly recommended at that time that MessageDefinition should be harmonized to use the same type pattern, but MessageDefinition was not updated.

FHIR-32399 attempted to address a similar issue (changing EventDefinition references from uri to canonical) but was retracted. The underlying problem persists.

### Impact
- **MessageHeader.event[x]** now uses `Coding | canonical(EventDefinition, SubscriptionTopic)` — already fixed
- **MessageDefinition.event[x]** still uses `Coding | uri` — needs to be fixed to match
- This creates an inconsistency between two closely related resources in the messaging framework

## Proposed Dispositions

### Disposition A: Persuasive (Minimal Fix)
Change `MessageDefinition.event[x]` from `Coding | uri` to `Coding | canonical(EventDefinition)`. This is the narrowest fix: replace the untyped `uri` with `canonical(EventDefinition)` to match the definition text exactly. The definition text can remain as-is.

**Pros:** Simple, directly addresses the reported mismatch, low risk.
**Cons:** Does not harmonize with MessageHeader.event[x] which also includes SubscriptionTopic. May require a follow-up ticket.

### Disposition B: Persuasive with Modification (Full Harmonization)
Change `MessageDefinition.event[x]` from `Coding | uri` to `Coding | canonical(EventDefinition, SubscriptionTopic)` to fully align with the change already applied to `MessageHeader.event[x]` in FHIR-39635. Update the definition text to: "Event code or link to the EventDefinition or SubscriptionTopic."

**Pros:** Achieves the harmonization Lloyd McKenzie recommended in FHIR-39635. Eliminates the inconsistency between MessageHeader and MessageDefinition event types. Future-proofs the element.
**Cons:** Broader scope than the reporter requested. The work group should confirm SubscriptionTopic is appropriate for MessageDefinition (it was for MessageHeader).

### Disposition C: Not Persuasive
The current `Coding | uri` type is intentionally flexible. The `uri` type allows referencing EventDefinition (or any resource) by URL without constraining the reference target type. This preserves backward compatibility and the definition text is merely descriptive, not prescriptive about type constraints.

**Pros:** No breaking change. Preserves maximum flexibility.
**Cons:** Ignores a genuine inconsistency between the definition text and type. Leaves MessageDefinition out of alignment with MessageHeader. Does not follow the canonical reference pattern recommended for canonical resources.

## Recommendation

**Disposition B: Persuasive with Modification** — Change `MessageDefinition.event[x]` to `Coding | canonical(EventDefinition, SubscriptionTopic)` and update the definition text accordingly.

This is the recommended disposition because:
1. It directly addresses the reported type/definition mismatch
2. It harmonizes MessageDefinition with the changes already applied to MessageHeader.event[x] via FHIR-39635
3. Lloyd McKenzie explicitly recommended this harmonization in FHIR-39635 comments (2023-01-30): *"there should be harmonization between MessageHeader and MessageDefinition to allow the same list of types for event"*
4. It properly uses `canonical` instead of `uri` for references to canonical resources, consistent with FHIR design principles

The work group should confirm that `SubscriptionTopic` is appropriate as an event target for MessageDefinition, not just MessageHeader. If the group prefers the narrower fix (without SubscriptionTopic), Disposition A would still resolve the core issue.
