## Description

The reporter (Grahame Grieve) identified that the existing Digital Signatures page references XAdES-X-L conformance for long-term signatures, but does not address a critical interoperability issue with whitespace handling in the `SignedProperties` block. The XAdES signature digest is computed by dereferencing `#SignedProperties` and canonicalizing that XML fragment. However, canonicalization libraries differ in whether they treat whitespace within `SignedProperties` as significant, producing different digest values and making signatures unverifiable.

The reporter discovered this issue while adding XML DigSig support to the FHIR browser. The IG publisher and the validator already produce and verify signatures without whitespace in this block.

### Proposed Changes

1. **Primary:** Add a strong recommendation (SHOULD-level) to the Digital Signatures page that the XAdES `SignedProperties` block should not contain whitespace other than a single space character where required for attributes. Proposed wording:

   > *"Due to variations in whitespace handling when generating digests, applications SHOULD ensure that there is no whitespace in the XAdES `SignedProperties` block other than a single space character where required for attributes"*

2. **Secondary:** Register the digital signatures page properly (on the documentation page and in JIRA).

3. **Secondary:** Move all content relating to digital signatures including canonicalization from the datatypes, JSON, and XML pages to the Digital Signatures page, and cross-reference from those pages (reconciling content in the process).

## Analysis

### Core Problem
XML canonicalization (C14N) libraries handle whitespace inconsistently when canonicalizing the XAdES `SignedProperties` fragment. Since the digest of `SignedProperties` is included in the `SignedInfo` block and signed, any difference in whitespace treatment produces a different digest, making signature verification fail. This is a concrete interoperability barrier for implementations using XAdES signatures with FHIR.

### Impact
- **Signature Verification Failures:** Implementations that produce pretty-printed (indented) `SignedProperties` blocks may generate signatures that cannot be verified by implementations using different canonicalization libraries.
- **Tooling Alignment:** The IG publisher and validator already follow the no-whitespace approach, so the recommendation formalizes existing practice.
- **Scope:** The secondary changes (page registration and content consolidation) are broader housekeeping tasks that could be tracked separately.

### Considerations
- The primary change (whitespace recommendation) is narrow and well-justified by empirical evidence from the reporter.
- The secondary changes (page registration, content consolidation) overlap with FHIR-53835 and may be better addressed there or as a coordinated effort.
- The SHOULD-level language is appropriate — it allows exceptions while clearly communicating best practice.

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Apply All Three Changes
Accept the ticket in full. Add the whitespace SHOULD recommendation to the Digital Signatures page, register the page in documentation/JIRA, and consolidate all digital signature content from datatypes/JSON/XML pages to the Digital Signatures page.

**Pros:** Comprehensive approach addresses all identified issues in one pass. Consolidating content improves discoverability and consistency.
**Cons:** The secondary changes (page registration and content consolidation) are significant editorial work that overlaps with FHIR-53835. Bundling them risks delaying the critical whitespace fix.

### Disposition B: Accept as Persuasive-with-Modification — Apply Primary Change Only
Accept the whitespace recommendation as persuasive but limit scope to adding the SHOULD-level whitespace guidance to the `SignedProperties` section of the Digital Signatures page. Defer page registration and content consolidation to FHIR-53835 or a new follow-up ticket.

**Pros:** Delivers the critical interoperability fix promptly. Avoids scope creep and overlapping work with FHIR-53835. Clear, surgically scoped change.
**Cons:** Does not address the secondary housekeeping issues raised in the ticket.

### Disposition C: Not Persuasive — No Change Required
Reject the change on the grounds that canonicalization should inherently handle whitespace consistently per the W3C C14N specification, and that the issue lies with non-conformant canonicalization libraries rather than the FHIR specification.

**Pros:** Avoids adding implementation-specific workaround guidance to the spec.
**Cons:** Ignores a real-world interoperability problem documented by the reporter. The practical reality is that libraries do differ, and FHIR should guide implementers toward interoperable outcomes.

## Recommendation

**Disposition B: Accept as Persuasive-with-Modification — Apply Primary Change Only.**

The whitespace recommendation is well-justified by practical experience, aligns with existing tooling behavior (IG publisher, validator), and directly addresses a concrete interoperability problem. Using SHOULD-level language is appropriate. The secondary changes (page registration, content consolidation) overlap significantly with FHIR-53835 and should be coordinated with that ticket rather than bundled here, avoiding duplicated effort and scope creep.
