## Summary

The reporter identifies that OID `2.16.840.1.113883.4.642.4.1930` has the name "Transport Status" when it should be "Transport Status Reason". This OID corresponds to the `transport-status-reason` code system/value set, not the `transport-status` code system. The current name is identical to the name used by `transport-status`, creating a duplicate naming collision.

## Proposed Dispositions

### Disposition A: Persuasive (Correct the OID name)

**Action:** Change the name associated with OID `2.16.840.1.113883.4.642.4.1930` from "Transport Status" to "Transport Status Reason" in the OID registry and any publication metadata that references it.

**Rationale:** The source XML definitions in the HL7/fhir repository already use the correct name "TransportStatusReason". The OID name mismatch was independently identified by Jose Costa Teixeira in an October 2022 audit that found `transport-status-reason` and `transport-status` sharing the duplicate name "TransportStatus". This is a clear metadata error — the OID belongs to the transport-status-reason code system and should carry the corresponding name. The fix is low-risk and eliminates a naming collision that can cause confusion for implementers looking up OIDs.

### Disposition B: Persuasive with Modification (Broader duplicate-name cleanup)

**Action:** Correct the OID name for `2.16.840.1.113883.4.642.4.1930` to "Transport Status Reason", and additionally audit and fix other duplicate-name issues identified in the October 2022 Zulip thread (e.g., `BiologicallyDerivedProductCodes`, `DeviceUsageStatus`, `FHIRTypes`).

**Rationale:** The 2022 audit found at least 7 code system pairs and 10+ value set pairs with duplicate names. Fixing only this one without addressing the broader pattern leaves other naming collisions in place. However, this broader scope goes beyond the ticket as filed and may require coordination across multiple work groups.

### Disposition C: Not Persuasive (No change needed)

**Action:** Close the ticket without making changes.

**Rationale:** The OID name is a display label in the OID registry and does not affect the semantics or behavior of the code system or value set. The canonical URL and the source definitions already correctly identify the resource. Implementers should rely on the canonical URL rather than the OID name for disambiguation. However, this argument is weak because the OID registry is a reference document that should be accurate.

## Recommendation

**Disposition A: Persuasive** — This is a straightforward technical correction. The OID name is factually wrong; it says "Transport Status" when the OID identifies the "Transport Status Reason" code system. The source artifacts in the HL7/fhir repository already have the correct name, confirming this is purely a metadata/registry error. The fix is low-risk, requires no breaking changes, and eliminates a documented naming collision. The broader cleanup in Disposition B, while valuable, exceeds the scope of this ticket and should be tracked separately.
