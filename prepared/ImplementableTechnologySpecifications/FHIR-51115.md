## Summary

The current JSON canonicalization section in the FHIR spec defines a method consisting of: "No whitespace other than single spaces in property values and in the xhtml in the Narrative + Order properties alphabetically." Grahame Grieve reports this is insufficient and proposes replacing it with a reference to **RFC 8785** (IETF JSON Canonicalization Scheme / JCS). The proposed text would adopt RFC 8785 as the formal method and assign the existing FHIR URI `http://hl7.org/fhir/canonicalization/json` to identify this method.

**Key technical concern:** This is technically a breaking change in how JSON numbers are canonicalized. However, Grahame argues that reliable JSON signatures are impossible without this change and that all known implementations already follow RFC 8785.

## Comments on Ticket (3)

1. **Grahame Grieve (2025-06-17):** Acknowledges this is a breaking change to number canonicalization but argues it is necessary — JSON signatures cannot be implemented reliably without it, and all known implementations already do this.

2. **Eric Haas (2025-06-18):** Proposes refined wording: *"This specification uses the IETF's JSON Canonicalization Scheme (JCS, RFC 8785) to standardize JSON data for consistent processing. Since RFC 8785 does not specify a formal URL to identify the canonicalization method, this specification adopts the URI `http://hl7.org/fhir/canonicalization/json` to represent the JCS method within the FHIR context."*

3. **Eric Haas (2025-06-18):** Raises additional concern about the existing assertion: *"These canonicalization algorithms do not work for enveloped signatures."* Questions why this statement exists when FHIR already specifies detached JWS signatures. Suggests the enveloped signature caveat should be reconsidered.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Accept the change request as submitted. Rewrite the JSON canonicalization section to formally adopt RFC 8785 (JCS) as the canonicalization method and update the identifying URI text. Use Grahame's proposed wording as the basis.

**Rationale:** RFC 8785 is an established IETF standard with multiple open-source implementations. The current FHIR-defined canonicalization is under-specified (especially for number formatting) and has proven insufficient for reliable signature verification. All known implementations already follow RFC 8785. Two prior tickets (FHIR-31566 in 2021, FHIR-41522 in 2023) requested the same change, demonstrating longstanding community recognition of this gap. The spec source already appears to reference RFC 8785, indicating alignment.

**Risk:** This is a breaking change to number canonicalization. Signatures created under the old rules would fail verification under RFC 8785. However, as Grahame notes, this is a theoretical rather than practical concern since implementations already follow RFC 8785.

### Disposition B: Persuasive with Modification
**Action:** Accept the change with Eric Haas's refined wording. Additionally, address the enveloped signature assertion that Eric flagged — either remove or update the statement *"These canonicalization algorithms do not work for enveloped signatures"* given that FHIR already specifies detached JWS signatures.

**Rationale:** Eric's wording is clearer and more precisely describes the relationship between RFC 8785 and the FHIR canonicalization URI. Addressing the enveloped signature note avoids internal inconsistency since FHIR specifies detached signatures. This approach resolves the core issue while cleaning up adjacent confusion.

**Risk:** Same breaking change risk as Disposition A. The additional editorial change to the enveloped signature text is low-risk but should be coordinated with the broader signature guidance work (FHIR-51133, FHIR-51112).

### Disposition C: Not Persuasive
**Action:** Defer this change, pending resolution as part of the broader signature guidance coordination effort proposed by Grahame in the Zulip discussion (processing all ~12 signature-related tickets as a block).

**Rationale:** The scope of canonicalization and signature changes is broad, with many interrelated tickets. Making this change in isolation may create inconsistencies with other pending changes. The Zulip discussion showed consensus for handling these tickets as a coordinated set.

**Risk:** Continued use of the under-specified canonicalization method. Further delays a change that has been requested since 2021. The spec source may already be ahead of the formal ticket disposition.

---

## Recommendation

**Disposition B: Persuasive with Modification** — Accept the RFC 8785 adoption using Eric Haas's refined wording, and also address the outdated enveloped signature assertion. This ticket has been requested three times over four years (FHIR-31566, FHIR-41522, now FHIR-51115), implementations already follow RFC 8785, and the spec source already references it. The refined wording from Eric Haas is clearer than the original proposal. While coordination with the broader signature ticket block is desirable, this specific change is well-defined enough to proceed independently and doing so would unblock the related tickets rather than delay them.
