## Description

The reporter notes that AU PS Composition's `Composition.extension:InformationRecipient` references base FHIR resource types (`Practitioner | Device | Patient | RelatedPerson | PractitionerRole | Organization`) rather than the corresponding AU PS profiles (AU PS Patient, AU PS Practitioner, AU PS PractitionerRole, AU PS RelatedPerson, AU PS Organization, etc.) defined within the IG.

It was explained during ballot review that data elements **without** a "SO" (Send/Obligation) flag intentionally reference base FHIR resources rather than AU-specific profiles. However, the reporter argues this design principle is **not obvious to readers** unfamiliar with IG development guidelines, causing confusion — especially since the IG defines AU PS-specific profiles for many of these resource types.

**Change suggestion:** Add explanatory text under Section 4.1 (General Guidance) to clarify why non-Must Support reference elements point to base FHIR resources rather than AU PS profiles. The reporter notes this issue applies to other AU PS profiles as well, not just AU PS Composition.

## Resolution Description

Update [AU PS Profile Design Principles](https://build.fhir.org/ig/hl7au/au-fhir-ps/general-guidance.html#au-ps-profile-design-principles) to clearly indicate that reference constraints apply only to elements flagged as *Must Support*.

## Analysis

### Core Issue

This ticket addresses a **documentation gap** rather than a technical defect. The AU PS IG follows a deliberate design principle: reference constraints to AU-specific profiles are applied **only** to elements flagged as Must Support (with obligations). Elements without Must Support intentionally reference base FHIR resource types. This is a valid and well-reasoned design — but the principle is not documented in the IG, causing reader confusion.

### Context from Prior Ballot

A closely related ticket, FHIR-52333, raised the same underlying concern during the previous ballot cycle and was resolved as **Not Persuasive** — the design was considered correct. FHIR-56128 represents a refinement of that feedback: the reporter now accepts the design principle but argues it needs to be **explicitly stated** in the IG's guidance section so readers understand why certain references use base types rather than AU PS profiles.

### Impact Assessment

- **Scope:** Documentation-only change to Section 4.1 (General Guidance), specifically the AU PS Profile Design Principles sub-section.
- **Affected profiles:** All AU PS profiles with reference elements that lack Must Support (AU PS Composition is the primary example, but the reporter notes it applies broadly).
- **Risk:** Very low — this is guidance text, not a structural or constraint change.
- **Benefit:** Directly addresses a source of reader confusion that has been raised across multiple ballot cycles.

## Proposed Dispositions

### Disposition A: Persuasive (Accept as-is)

**Action:** Accept the ticket and update the [AU PS Profile Design Principles](https://build.fhir.org/ig/hl7au/au-fhir-ps/general-guidance.html#au-ps-profile-design-principles) section to add explanatory text stating that reference type constraints to AU-specific profiles apply only to elements flagged as Must Support, and that elements without Must Support intentionally reference base FHIR resource types.

**Rationale:** The resolution description already identifies the specific change needed. This is a low-risk documentation improvement that addresses a recurring source of reader confusion (see also FHIR-52333). Adding a clear statement of this design principle improves IG usability without changing any technical content.

### Disposition B: Persuasive with Modification

**Action:** Accept the intent but expand the scope slightly — in addition to updating Section 4.1 General Guidance, add a brief inline note or "info box" on the AU PS Composition profile page itself (and potentially other affected profile pages) pointing readers to the design principles explanation.

**Rationale:** Readers encountering the base FHIR references on individual profile pages may not navigate to General Guidance. Adding a contextual pointer directly on affected profile pages would maximize the clarity benefit. However, this increases implementation effort and may introduce maintenance burden across multiple profile pages.

### Disposition C: Not Persuasive

**Action:** Reject the change request. The existing IG structure and the profile design principles (as refined through the ballot process) are sufficient for the intended audience.

**Rationale:** FHIR IG development conventions are well-established, and implementers are expected to understand that unconstrained reference elements inherit base definitions. Adding explanatory text for every design convention could clutter the IG. The prior resolution of FHIR-52333 (Not Persuasive) established that the design is correct; additional documentation is optional.

## Recommendation

**Disposition A: Persuasive (Accept as-is)**

The resolution has already been determined as Persuasive with a clear, actionable change: update the AU PS Profile Design Principles section to state that reference constraints apply only to Must Support elements. This is a straightforward documentation addition with high benefit-to-effort ratio. It directly addresses a legitimate and recurring source of reader confusion (previously raised in FHIR-52333) without changing any technical content or profile constraints. The change is consistent with AU PS's maturation goals (see FHIR-53192) and aligns with the principle of making IG design decisions explicit rather than implicit.
