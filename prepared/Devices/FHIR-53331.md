## Summary

This ticket requests that the PhDDevice profile (Section 4.11.1 of the PHD IG) explicitly support reporting of EUI-64 identifiers. EUI-64 (Extended Unique Identifier, 64-bit) is an IEEE-defined hardware identifier format consisting of a 24-bit organizationally unique identifier (OUI) followed by a 40-bit manufacturer-defined ID. In the IEEE 11073-20601 standard, the system identifier (SYSID) must be an EUI-64, and this is the mandatory device identifier type. The newer IEEE 11073-10206 (ACOM) standard relaxes this slightly, allowing other identifier types while keeping the identifier mandatory.

The ticket was previously resolved as "Considered for Future Use" and deferred to the next revision of the PHD IG per Devices WG review on 2025-11-12. It was subsequently re-opened post STU2 publication (per lynn_laakso's comments on 2026-03-31) for reconsideration.

## Details

**Current State:** The PHD IG's PhdDevice profile already defines identifier slicing with the ContinuaDeviceIdentifiers code system, supporting identifier types such as SYSID (IEEE 11073 system ID), BTMAC (Bluetooth MAC), and Ethernet MAC. The SYSID identifiers are normally EUI-64 type strings. However, the profile does not explicitly represent EUI-64 as a distinct, reportable identifier type.

**Background from Related Work:**
- **FHIR-53351** (also by MalcolmClarke, same date) raises a similar request: Section 2.9 of the PHD IG should support EUI-64. This sibling ticket has the same history — deferred then re-opened.
- **FHIR-48236** (Ron G. Parker, Highest priority, resolved as Persuasive with Modification) addressed broader identifier profiling issues in PhdDevice and PhgDevice, noting that identifier profiling was too strict and should permit additional type codes. Work was partially pre-applied in the ACOM branch of the IG (per erikmoll, 2025-06-26).
- The Zulip thread on **ContinuaDeviceIdentifiers codesystem** (devices stream, 2025-01-11) confirms that SYSID-type identifiers are EUI-64 strings and explains the OID-based system identifier for "EUI-64 with OUI assigned by IEEE" (`1.2.840.10004.1.1.1.0.0.1.0.0.1.2680`).

**Relationship to ACOM:** The IEEE 11073-10206 (ACOM) standard allows multiple identifier types beyond EUI-64. The ongoing ACOM branch work (FHIR-48236) is already relaxing identifier constraints. Explicit EUI-64 support would align the profile with both the legacy 11073-20601 requirement and the ACOM flexibility.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

**Action:** Add explicit EUI-64 identifier support to the PhdDevice profile in Section 4.11.1.

**Rationale:** EUI-64 is the mandatory identifier type in IEEE 11073-20601 and a supported type in IEEE 11073-10206 (ACOM). The current profile handles EUI-64 implicitly through the SYSID slice, but does not surface EUI-64 as a distinct, first-class identifier type. Adding an explicit EUI-64 identifier slice (or extending the ContinuaDeviceIdentifiers code system to clearly represent EUI-64) would improve clarity and interoperability. This should be coordinated with FHIR-53351 (which requests the same for Section 2.9) and the ongoing ACOM profiling work from FHIR-48236.

**Impact:** Non-substantive to Substantive (depending on whether this adds a new required slice or merely extends documentation). Affects PhdDevice profile, potentially ContinuaDeviceIdentifiers code system.

### Disposition B: Persuasive with Modification — Subsume into FHIR-48236 / FHIR-53351

**Action:** Merge the intent of this ticket into the broader identifier profiling rework already underway (FHIR-48236) and coordinate with FHIR-53351, rather than making a standalone change.

**Rationale:** FHIR-48236 is already relaxing identifier constraints in PhdDevice/PhgDevice profiles and has active ACOM branch work. Adding EUI-64 support as part of that holistic rework avoids fragmented changes and ensures consistency across identifier types. The same reporter filed FHIR-53351 for a closely related section, suggesting these should be addressed together.

**Impact:** Non-substantive to Substantive. Deferred to the next PHD IG revision cycle where FHIR-48236 changes are being finalized.

### Disposition C: Decline (Not Persuasive)

**Action:** Close without change, noting that EUI-64 is already implicitly supported through the SYSID identifier slice.

**Rationale:** The current PhdDevice profile's SYSID slice with OID `1.2.840.10004.1.1.1.0.0.1.0.0.1.2680` already represents "EUI-64 with OUI assigned by IEEE." Implementers can report EUI-64 identifiers today using the existing SYSID mechanism. The Zulip discussion (2025-01-11) confirms this interpretation. Adding a separate EUI-64 identifier type may introduce confusion about whether SYSID and EUI-64 are distinct concepts.

**Impact:** None.

---

## Recommendation

**Disposition B: Persuasive with Modification — Subsume into FHIR-48236 / FHIR-53351**

The request is valid — explicit EUI-64 support improves clarity — but addressing it in isolation risks fragmentation. FHIR-48236 is already reworking identifier profiling in the ACOM branch with active contributions (erikmoll). Coordinating with FHIR-53351 (same reporter, overlapping scope) ensures both sections are updated consistently. This aligns with the WG's prior decision to defer to the next PHD IG revision and provides a structured path for inclusion.
