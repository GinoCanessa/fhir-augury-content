## Summary

This ticket requests that the Personal Healthcare Devices (PHD) IG review its internally-defined code systems and either migrate them to HL7 Terminology (THO / terminology.hl7.org) or obtain a formal exemption from the Terminology Services Management Group (TSMG), per the TSC-approved [Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers) policy.

The ticket has a complex disposition history:

1. **2024-09-16** — Original filing by Jessica Bota requesting code system review.
2. **2025-07-16** — Voted **Persuasive with Modification** (Elliot Silver / Erik Moll, 9-0-1, Non-compatible Enhancement): the Devices WG committed to engage TSMG before the next ballot with a proposal for all code systems.
3. **2025-07-23** — Resolution reverted; re-voted as **Considered for Future Use** (Michael Faughn / Erik Moll, 8-0-0): deferred until the PHD IG approaches FMM 3, with discussion about whether an exemption is strictly required before that milestone.
4. **2026-03-31** — Re-opened by lynn_laakso post STU2 publication, reverting the "Considered for Future Use" resolution.

The Devices WG produced a detailed analysis of all seven code systems in the PHD IG, categorizing each by intended final destination:

| Code System | Disposition |
|-------------|-------------|
| [ASN1 bits to HL7 enumerated values](https://build.fhir.org/ig/HL7/phd/CodeSystem-ASN1ToHL7.html) | **Move to THO** — growing code system; needs updates independent of IG releases |
| [Continua Device Identifiers](https://build.fhir.org/ig/HL7/phd/CodeSystem-ContinuaDeviceIdentifiers.html) | **Move to THO** — identifiers useful beyond the IG scope |
| [Continua PHD Interface Identifiers](https://build.fhir.org/ig/HL7/phd/CodeSystem-ContinuaPHDInterfaceIDs.html) | **Keep in IG** — tied to defunct PCHA/Continua certification programme |
| [Continua Health and Fitness Service Interfaces](https://build.fhir.org/ig/HL7/phd/CodeSystem-ContinuaHFS.html) | **Keep in IG** — tied to defunct PCHA/Continua certification programme |
| [PHD Observation Categories](https://build.fhir.org/ig/HL7/phd/CodeSystem-PhdObservationCategories.html) | **Keep in IG** — identifies Observation resources specific to this IG |
| [PHD Reference Range Meaning](https://build.fhir.org/ig/HL7/phd/CodeSystem-PhdReferenceRangeMeaning.html) | **Remove from IG** — redundant with SampledData LowerLimit/UpperLimit fields |
| [SimpleAlertOperationalStates](https://build.fhir.org/ig/HL7/phd/CodeSystem-SimpleAlertOperationalStatesCS.html) | **Keep in IG** for R4; will be superfluous in R5/R6 |

Key open question from the comment thread: whether a formal TSMG exemption is required before FMM 3 or only at FMM 3 (per Item 4 of the TSMG guidance). Jay Snell asked Marc Duteau to confirm whether FMG will require an exemption for publication, but no response is recorded.

## Details

The [Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers) policy requires IGs to either host their code systems in THO or obtain a documented TSMG exemption. The PHD IG currently defines seven code systems inline. The Devices WG has done significant analysis (documented in the 2025-07-23 comment by mfaughn) and plans to:

- **Migrate 2 code systems** (ASN1ToHL7, ContinuaDeviceIdentifiers) to THO because they grow independently and have utility beyond the IG.
- **Retain 4 code systems** in the IG (ContinuaPHDInterfaceIDs, ContinuaHFS, PhdObservationCategories, SimpleAlertOperationalStatesCS) — these are tightly bound to the IG or legacy certification programs.
- **Remove 1 code system** (PhdReferenceRangeMeaning) as redundant.

The WG also planned to update URLs for code systems destined for THO. The ticket was re-opened post STU2 publication (2026-03-31) indicating the work has not yet been completed.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Execute the Code System Migration Plan

**Vote:** Persuasive with Modification

Accept the ticket and require the Devices WG to execute the migration plan documented in the 2025-07-23 analysis before the next ballot cycle:

1. **Migrate ASN1ToHL7 and ContinuaDeviceIdentifiers** to THO, coordinating with TSMG for onboarding.
2. **Request formal TSMG exemptions** for the four code systems remaining in the IG (ContinuaPHDInterfaceIDs, ContinuaHFS, PhdObservationCategories, SimpleAlertOperationalStatesCS), documenting the rationale for each.
3. **Remove PhdReferenceRangeMeaning** from the IG as redundant.
4. **Update all code system URLs** in the IG for migrated systems.

**Impact:** Non-compatible  
**Category:** Enhancement  

*Rationale:* The WG has already done the analysis and identified a clear plan. STU2 has been published, and delaying further risks accumulating technical debt. Executing the plan now aligns with TSMG policy and resolves a Highest-priority ticket.

### Disposition B: Considered for Future Use — Defer Until FMM 3 with Immediate Exemption Request

**Vote:** Considered for Future Use

Acknowledge the WG's analysis but defer the actual code system migration until the PHD IG approaches FMM 3, consistent with Item 4 of the TSMG Terminology Expectations guidance. However, immediately file a blanket TSMG exemption request for all IG-defined code systems to satisfy any publication gating requirements, and commit to executing the migration plan as part of the FMM 3 advancement work.

**Impact:** None (deferred)  
**Category:** Enhancement  

*Rationale:* The TSMG guidance suggests this is not strictly required until FMM 3. A formal exemption request now would document the WG's intent and satisfy any FMG/publication requirements without forcing premature migration work. This is consistent with the prior 8-0-0 vote.

### Disposition C: Not Persuasive — Exemptions Not Required at Current Maturity

**Vote:** Not Persuasive

Close the ticket. The TSMG Terminology Expectations for IG Developers policy (Item 4) does not require code system migration or exemptions until FMM 3. The PHD IG is at a lower maturity level, and no action is needed at this time. If policy requirements change or the IG advances to FMM 3, a new ticket can be filed.

**Impact:** None  
**Category:** N/A  

*Rationale:* The WG's reading of the TSMG guidance suggests this is not yet mandatory. However, this disposition risks being overridden by FMG or TSMG if they interpret the policy differently, and ignores the detailed analysis already performed.

## Recommendation

**Disposition B: Considered for Future Use — Defer Until FMM 3 with Immediate Exemption Request**

This disposition balances pragmatism with policy compliance. The Devices WG has already produced a thorough code system analysis (2025-07-23) and demonstrated good-faith engagement with TSMG policy. The key uncertainty — whether a formal exemption is required before FMM 3 — remains unresolved (Jay Snell's question to Marc Duteau received no recorded answer). Filing an immediate exemption request resolves this ambiguity at low cost, while deferring the actual migration work to FMM 3 avoids premature effort on an IG that may still undergo significant changes. The prior WG vote (8-0-0) supports this approach. The ticket was re-opened post STU2 publication, making it an appropriate time to formalize the exemption request and document the planned migration path.
