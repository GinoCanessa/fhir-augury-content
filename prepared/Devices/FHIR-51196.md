## Description

The Point of Care Devices (PoCD) IG includes a guidance page on RESTful Transfer but contains no guidance on FHIR messaging. The submitter requests that the IG add content covering FHIR messaging, including at minimum:

- Examples of FHIR messaging in the PoCD context
- Guidance on persistence (or lack thereof) of references and resources across different message bundles

The submitter notes that the FHIR Core spec's [messaging page](https://build.fhir.org/messaging.html) does not make these points obvious, so IG-specific guidance would be valuable.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Action:** Accept the existing resolution. Add the `messaging.md` content (attached by Erik Moll) as a new Messaging guidance page in the PoCD IG.

**Rationale:**
- The gap is clear: the IG covers RESTful Transfer but is silent on FHIR messaging, a legitimate exchange paradigm for point-of-care device data.
- Draft content has already been produced, reviewed by the original submitter (Javier Espina), and confirmed as acceptable.
- A preview page already exists on the version-1.1.0 branch, demonstrating that the work is largely complete.
- This is low-risk additive content that does not alter existing profiles or conformance requirements.

### Disposition B: Persuasive with Modification — Expand Scope

**Action:** Accept but request additional content beyond the attached `messaging.md`, such as worked examples showing message bundle construction for common PoCD scenarios (e.g., periodic device observations, alarm notifications), and explicit guidance on how resource references should be handled across message boundaries.

**Rationale:**
- The original request specifically asked for examples and persistence guidance, which may not be fully covered in the AI-generated draft.
- The related ticket FHIR-39507 highlights similar gaps around resource handling (Patient, Location) that could be addressed together.
- However, this expands scope and may delay publication.

### Disposition C: Not Persuasive — Defer to Core Spec

**Action:** Reject, arguing that FHIR messaging guidance belongs in the FHIR Core spec or a cross-cutting messaging IG (cf. FHIR-26212), not in a device-specific IG.

**Rationale:**
- FHIR Core already defines the messaging paradigm. IG-specific guidance risks divergence or duplication.
- However, this argument is weak because the IG already provides RESTful Transfer guidance, establishing a precedent for exchange-pattern-specific content. The submitter and workgroup have consensus that PoCD-specific messaging guidance is needed.

## Recommendation

**Disposition A: Accept as Persuasive.** The content gap is real, the draft content has been produced and reviewed, and a working preview already exists. The workgroup should confirm that the `messaging.md` content on the version-1.1.0 branch adequately addresses the request and proceed with inclusion in the next publication.
