## Summary

This ticket identifies that Section 2.9.7 of the Personal Healthcare Devices (PHD) IG incorrectly associates time synchronisation properties with relative time. Since relative time is, by definition, not synchronised to any external clock, the time synchronisation method property should not apply to relative time entries. The reporter argues that including the synchronisation method property for relative time is semantically incorrect and should be removed or clarified.

This ticket was previously resolved as "Considered for Future Use" and deferred to the next revision of the PHD IG per Devices WG review (see 2025-11-12 Devices Main Call). It was subsequently re-opened post STU2 publication and reverted to Triaged status.

## Proposed Dispositions

### Disposition A: Accept as Technical Correction (Persuasive)

**Resolution:** Remove the time synchronisation method property from relative time entries in Section 2.9.7 of the PHD IG.

**Rationale:** Relative time is by definition not synchronised to any external time source — it is a free-running counter measuring elapsed time between events. Including a time synchronisation method property for relative time entries is semantically incorrect and could mislead implementers into attempting to apply synchronisation metadata where none is applicable. This is a straightforward technical correction that aligns the data model with the actual semantics of relative time as described elsewhere in the specification (Sections 2.8 and 3.3). The correction should be coordinated with the related cluster of tickets (FHIR-53354, FHIR-53349, FHIR-53344, FHIR-53355) which collectively address time-handling inconsistencies in the PHD IG.

**Spec Change:** In Section 2.9.7, remove or exclude the time synchronisation method property from the relative time case. Add a clarifying note that synchronisation method is only applicable to absolute time (e.g., UTC-based) entries.

### Disposition B: Accept with Broader Scope — Bundle with Related Time Tickets (Persuasive with Modification)

**Resolution:** Address this ticket as part of a broader refactoring of the time-handling sections (2.8, 2.9.7, 3.3) in coordination with FHIR-53354, FHIR-53349, FHIR-53344, and FHIR-53355.

**Rationale:** This ticket is one of six closely related issues filed by the same reporter, all targeting time semantics in the PHD IG. Rather than applying a narrow fix to Section 2.9.7 alone, the WG should consider a coordinated revision that: (1) clearly distinguishes relative vs. absolute time throughout, (2) makes synchronisation properties conditional on absolute time, and (3) resolves inconsistencies across sections. This avoids piecemeal changes that could introduce new inconsistencies.

**Spec Change:** Perform a holistic review and revision of Sections 2.8, 2.9.7, and 3.3 to consistently model relative time as an independent quantity without synchronisation properties, and absolute time with appropriate synchronisation metadata. Document the design rationale in an implementer note.

### Disposition C: Decline — Not Persuasive

**Resolution:** Close as Not Persuasive; retain current structure.

**Rationale:** While relative time is semantically unsynchronised, including the synchronisation method property in the data model does not cause implementation harm — an implementer would simply populate it with a "not synchronised" or null value. The property's presence ensures structural consistency across all time types in the PHD Device profile, simplifying parsing logic. Removing it would create a special case that adds complexity for implementers. The WG previously deferred this to a future revision, and there is no evidence of real-world implementation issues caused by the current approach.

**Spec Change:** No change. Optionally add a guidance note clarifying that for relative time entries, the synchronisation method should be set to indicate "not synchronised."

## Recommendation

**Disposition A: Accept as Technical Correction** is recommended. The ticket correctly identifies that the time synchronisation method property is semantically inapplicable to relative time. This is a clear technical error — relative time by definition lacks external synchronisation, and the data model should reflect that. The fix is narrowly scoped and low-risk. While Disposition B's broader approach has merit given the cluster of related tickets, each ticket should be individually dispositioned to maintain clear traceability; the WG can coordinate the actual spec changes during implementation. The ticket has already been reviewed once and deferred; now that it is re-opened post-STU2, it should be resolved definitively.
