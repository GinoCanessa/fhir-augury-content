## Description

The reporter identifies that **Section 3.2** of the PHD IG contains a passage that uses the terms "object", "attributes", and "types" in a confusing, seemingly synonymous manner. The specific passage reads:

> *"The attributes, like resource elements, have specified data types. All IEEE 11073-10206 ACOM PHD specializations from thermometers to home ventilators use the same set of objects and associated attributes. The simplest PHDs, like a thermometer, generate only one type of observation whereas more complex PHDs like a home ventilator generate several different types. The sleep apnea breathing therapy equipment PHD defines more than 80 observation types."*

The concern is that the sentence conflates IEEE 11073 concepts (objects, attributes) with FHIR concepts (types, observations) without clear distinction, making the text difficult to follow for readers unfamiliar with the IEEE 11073-10206 ACOM model.

## Proposed Dispositions

### Disposition A: Persuasive (Recommended)

**Action:** Accept the feedback and revise the text in Section 3.2 to use consistent, well-defined terminology.

**Rationale:** The reporter correctly identifies that the passage conflates IEEE 11073 domain terms ("objects", "attributes") with FHIR-centric concepts ("types", "observation types") without clearly distinguishing them. This is a straightforward editorial/technical correction that improves readability. The precedent set by FHIR-48010 (resolved as Persuasive for similar terminology confusion in the same IG) supports acceptance. The ticket has been reopened post-STU2 specifically for the next revision cycle, indicating WG intent to address it. A suggested revision could:

- Clearly define "objects" and "attributes" in the IEEE 11073-10206 ACOM context before using them.
- Distinguish "observation types" (FHIR concept) from "attribute types" (IEEE concept).
- Consider aligning with the rewrites proposed in the sibling tickets (FHIR-53338, FHIR-53340, FHIR-53341).

### Disposition B: Persuasive with Modification

**Action:** Accept the feedback but address it as part of a broader rewrite of Section 3.2 that also incorporates FHIR-53338, FHIR-53340, FHIR-53341, and FHIR-53342.

**Rationale:** Since five related tickets all target Section 3.2 terminology issues, it may be more efficient and produce better results to do a single coordinated rewrite rather than piecemeal edits. This disposition defers the specific fix to a consolidated editorial pass while acknowledging the validity of the feedback.

### Disposition C: Not Persuasive

**Action:** Reject the feedback; no change to the text.

**Rationale:** The terms "objects", "attributes", and "types" have specific meanings within the IEEE 11073-10206 ACOM model, and the paragraph is aimed at readers with some familiarity with that model. However, this argument is weakened by the fact that the IG is intended for a FHIR audience who may not have IEEE background, and similar terminology concerns have already been accepted (FHIR-48010).

## Recommendation

**Disposition A: Persuasive** — The feedback is valid and actionable. The text demonstrably conflates distinct IEEE and FHIR terminology in a way that hinders comprehension. The prior acceptance of FHIR-48010 for analogous terminology confusion sets clear precedent. The WG's decision to reopen this post-STU2 signals intent to address it. The fix is low-risk (editorial only, no structural or normative impact) and should be coordinated with sibling tickets targeting the same section.
