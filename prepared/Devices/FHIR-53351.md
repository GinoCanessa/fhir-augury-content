## Summary

The reporter requests that **Section 2.9** of the Personal Healthcare Devices (PHD) IG add support for **EUI-64** (Extended Unique Identifier, 64-bit) identifiers. EUI-64 is an IEEE-defined globally unique identifier format consisting of a 24-bit Organizationally Unique Identifier (OUI) plus a 40-bit manufacturer-defined ID. It is widely used in IEEE 11073-20601 personal health devices as the mandatory system identifier. The ticket was previously resolved as "Considered for Future Use" and deferred to the next PHD IG revision per Devices WG review on 2025-11-12, but has since been re-opened post-STU2 publication.

This ticket is closely related to [FHIR-53331](https://jira.hl7.org/browse/FHIR-53331), which requests the same EUI-64 support specifically for the **PhDDevice** profile (Section 4.11.1). Both tickets were filed by the same reporter, deferred together, and re-opened together.

---

## Details

### Description

> Section 2.9 It would be helpful to support EUI-64

The request targets Section 2.9 of the PHD IG, asking that the specification explicitly support EUI-64 identifiers. Currently, the PHD IG handles system identifiers via the `ContinuaDeviceIdentifiers` CodeSystem and the `PhdDevice` profile's system identifier section, but the reporter considers the EUI-64 support insufficient or unclear in this section.

## Related Zulip Discussions

### [devices > ContinuaDeviceIdentifiers codesystem](https://chat.fhir.org/#narrow/stream/devices/topic/ContinuaDeviceIdentifiers%20codesystem) (2025-01-11)
Discussion about the `ContinuaDeviceIdentifiers` CodeSystem used in PHD device examples. **Jacob Andersen** explains that SYSID-type identifiers are normally EUI-64 strings with the organizational part assigned by IEEE, per IEEE 11073-20601:2019. The identifier consists of a 24-bit OUI followed by a 40-bit manufacturer-defined ID. The system OID `1.2.840.10004.1.1.1.0.0.1.0.0.1.2680` means "EUI-64 with OUI assigned by IEEE." This confirms that EUI-64 is already in use within the PHD ecosystem but may lack explicit specification-level support in Section 2.9.

## Proposed Dispositions

### Disposition A: Accept — Add Explicit EUI-64 Support to Section 2.9

**Action:** Update Section 2.9 of the PHD IG to explicitly define and support EUI-64 as a recognized device identifier format. This should include:

1. A clear definition of EUI-64 (IEEE Extended Unique Identifier, 64-bit: 24-bit OUI + 40-bit manufacturer ID) in the section narrative.
2. Guidance on encoding EUI-64 values in `Device.identifier`, including the appropriate `identifier.system` URI (e.g., the existing OID `urn:oid:1.2.840.10004.1.1.1.0.0.1.0.0.1.2680` or a FHIR-defined URI like `http://hl7.org/fhir/sid/eui-64`).
3. An example resource showing EUI-64 usage in a PHD context.
4. Coordination with companion ticket [FHIR-53331](https://jira.hl7.org/browse/FHIR-53331) to ensure the PhDDevice profile (Section 4.11.1) also reflects this support.

**Rationale:** EUI-64 is the mandatory system identifier in IEEE 11073-20601 and is already in practical use (as confirmed by Zulip discussions). Making it explicit in the specification improves interoperability and reduces ambiguity.

### Disposition B: Accept with Modification — Consolidate with FHIR-53331

**Action:** Merge this ticket with [FHIR-53331](https://jira.hl7.org/browse/FHIR-53331) and address EUI-64 support holistically across the PHD IG rather than section-by-section. Create a single comprehensive update that:

1. Adds EUI-64 documentation in Section 2.9 (general identifier overview).
2. Updates the PhDDevice profile (Section 4.11.1) to include EUI-64 in its identifier slicing.
3. Adds or updates the `ContinuaDeviceIdentifiers` CodeSystem to explicitly include an EUI-64 code if not already present.
4. Ensures all relevant PHD examples use consistent EUI-64 encoding.

**Rationale:** Both tickets are from the same reporter, target the same specification, and address the same gap. A consolidated approach avoids inconsistencies between sections.

### Disposition C: Decline — EUI-64 Already Supported Implicitly

**Action:** Close as "Not a Problem." The PHD IG already implicitly supports EUI-64 through the SYSID identifier type in the `ContinuaDeviceIdentifiers` CodeSystem and the system OID `1.2.840.10004.1.1.1.0.0.1.0.0.1.2680`. Section 2.9 does not need modification because EUI-64 is the default format for the existing system identifier infrastructure.

**Rationale:** As Jacob Andersen noted in the Zulip discussion, SYSID-type identifiers are already "normally EUI-64 type strings." Adding explicit EUI-64 support may be redundant. However, this disposition risks continued implementer confusion about how to properly encode EUI-64 values.

---

## Recommendation

**Disposition B: Accept with Modification — Consolidate with FHIR-53331**

This ticket and FHIR-53331 are fundamentally the same request (EUI-64 support in the PHD IG) split across two sections. Addressing them together ensures a coherent, specification-wide treatment of EUI-64 identifiers. While EUI-64 is already implicitly supported through the SYSID mechanism, making it explicit improves implementer clarity — the fact that this ticket was filed at all demonstrates that the current specification is not sufficiently clear. The Devices WG previously deferred this to the next PHD IG revision (post-STU2), and it has now been re-opened, indicating readiness to address it. A consolidated approach is the most efficient path to resolution.
