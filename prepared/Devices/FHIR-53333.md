# FHIR-53333: Timestamp coding not strictly required

## Summary

**Ticket:** [FHIR-53333](https://jira.hl7.org/browse/FHIR-53333)
**Status:** Triaged | **Type:** Technical Correction | **Priority:** Medium
**Specification:** Personal Healthcare Devices (FHIR)
**Work Group:** Devices
**Reporter:** MalcolmClarke | **Assignee:** Unassigned
**Created:** 2025-11-05 | **Updated:** 2026-03-31

The reporter states that Section 4.10.1 of the PHD IG describes "timestamp coding" that is not strictly required in FHIR, because the FHIR `dateTime` format already inherently conveys the coding for local and UTC timestamps through its format structure (e.g., `+zz:zz` suffix for timezone, `Z` for UTC). The implication is that the PHD IG includes unnecessary or redundant guidance about timestamp coding that could be simplified or removed.

This ticket was previously resolved as "Considered for Future Use" and deferred to the next revision of the PHD IG per Devices WG review on 2025-11-12. It was subsequently re-opened post STU2 publication by lynn_laakso on 2026-03-31.

## Proposed Dispositions

### Disposition A: Applied — Remove Redundant Timestamp Coding Guidance
**Resolution:** Applied (Technical Correction)
**Action:** Update Section 4.10.1 of the PHD IG to remove or streamline the timestamp coding guidance that duplicates what FHIR's `dateTime` format already conveys. Add a brief note clarifying that FHIR `dateTime` format inherently indicates whether a timestamp is local (with offset) or UTC (with `Z` suffix), making explicit "timestamp coding" unnecessary.
**Rationale:** The FHIR `dateTime` datatype already encodes local vs. UTC via its format. Redundant guidance adds confusion and implementation burden for PHD implementers. This aligns with the reporter's observation and simplifies the specification.

### Disposition B: Applied — Clarify Rather Than Remove
**Resolution:** Applied (Technical Correction)
**Action:** Retain the timestamp coding section but add clarifying text explaining how PHD timestamp coding maps to FHIR `dateTime` format rules. Explicitly note that implementers should rely on the FHIR `dateTime` format conventions and that the PHD-specific coding is informational context from the 11073 domain rather than an additional requirement.
**Rationale:** The timestamp coding section may provide useful context for implementers coming from the IEEE 11073 PHD domain who need to understand the mapping from device-level timestamp types to FHIR representations. A clarification preserves this bridge while eliminating the impression that it imposes additional requirements.

### Disposition C: Not Persuasive — Retain Current Text
**Resolution:** Not Persuasive
**Action:** No change. The current timestamp coding guidance in Section 4.10.1 is retained as-is.
**Rationale:** The timestamp coding section serves as important domain context for the PHD-to-FHIR mapping, helping implementers understand how different IEEE 11073 timestamp types translate to FHIR representations. The guidance is complementary to, not duplicative of, FHIR's `dateTime` rules.

## Recommendation

**Disposition B: Applied — Clarify Rather Than Remove**

This ticket raises a valid point that the PHD IG's timestamp coding section can be confusing if it appears to impose requirements beyond what FHIR `dateTime` already handles. However, given the cluster of related timestamp tickets from this reporter (FHIR-53334, FHIR-53335, FHIR-53332, FHIR-53365), there is clearly a broader need to clarify the PHD IG's timestamp handling for implementers. Disposition B addresses this ticket's concern while preserving the 11073-to-FHIR mapping context that is valuable for the PHD domain. The section should be reviewed holistically alongside the related tickets to ensure consistent, clear guidance across all timestamp-related sections.
