## Description

The submitter requests that IEEE 11073-20601 be properly mentioned in the PHD IG, with notes on how mapping may differ between 20601 and the newer 10206 standard—specifically regarding power and clock reporting. The submitter notes that metric observations are the same between the two standards.

### [devices > derivedFrom relationships in PHD](https://chat.fhir.org/#narrow/stream/devices/topic/derivedFrom%20relationships%20in%20PHD)
- **Date range:** 2024-04-24 to 2024-04-25
- **Participants:** Josh Mandel, Erik Moll
- **Summary:** Discussion about how IEEE 11073-20601 observation references map to FHIR. Erik Moll noted that 20601 uses a generic reference between metrics, which the PHD IG maps to `Observation.derivedFrom`. With IEEE 11073-10206 (ACOM), there are now two explicit relationship types—`derivedFrom` and `hasMember`—that map better to FHIR R4. This directly illustrates how mapping semantics differ between 20601 and 10206, supporting the need for documentation requested in this ticket.

## Analysis

This ticket has meaningful support from implementers. The original commenter (erikmoll) initially dismissed the issue, but JIRAUSER29419 (likely an implementer working with PHGs) made a compelling case that IEEE 11073-20601 is still actively used, currently being revised, and underpins parts of the PHD IG. The WG initially deferred to the next PHD IG revision, but lynn_laakso re-opened it post-STU2 publication, indicating the WG considers it actionable now.

The core ask is modest: add proper mention of IEEE 11073-20601 and document where mapping differs from 10206 (specifically for power reporting, clock/time handling). The submitter acknowledges metric observations are the same, so the scope is bounded. Related tickets (FHIR-53350 on source-handle-reference, FHIR-53328 on proprietary device mapping) and the Zulip thread on derivedFrom all reinforce that 20601-to-10206 mapping differences are a practical concern for implementers.

FHIR-48010 (resolved Persuasive) already improved confusing text about 20601, suggesting the WG is receptive to better documentation of the 20601/10206 relationship.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Accept the request as submitted. Add documentation to the PHD IG that (1) properly references IEEE 11073-20601 as a current, actively-maintained standard used by existing PHD devices, (2) provides notes on mapping differences between 20601 and 10206 for power reporting and clock/time handling, and (3) clarifies that metric observation mapping is the same between the two standards.

**Rationale:** The request is well-supported by implementer feedback (JIRAUSER29419's comments), aligns with prior WG precedent (FHIR-48010 resolved Persuasive), and addresses a genuine documentation gap. The scope is bounded and the content is factual—20601 is still in active use and revision.

### Disposition B: Persuasive with Modification
**Action:** Accept the intent but expand the scope slightly. In addition to documenting 20601/10206 mapping differences for power and clock, also add a brief overview section explaining the relationship between the two standards (20601 as the transport-layer protocol, 10206/ACOM as the abstract content model), and cross-reference related mapping guidance where `derivedFrom` vs `hasMember` semantics differ (as noted in the Zulip discussion). Coordinate with FHIR-53350 and FHIR-53328 for consistent treatment.

**Rationale:** The ticket identifies a real gap, but a slightly broader treatment would provide better context for implementers working with both standards. Coordinating with the related open tickets (FHIR-53350, FHIR-53328) would avoid fragmented guidance.

### Disposition C: Not Persuasive
**Action:** Reject the request. The PHD IG is focused on IEEE 11073-10206 (ACOM) and already acknowledges 20601 as a predecessor. Detailed mapping guidance between the two IEEE standards is outside the scope of a FHIR IG.

**Rationale:** The FHIR IG should document FHIR-specific mappings, not serve as a guide for IEEE standard interrelationships. Implementers needing 20601-specific mapping guidance should consult the IEEE standards directly.

## Recommendation

**Disposition B: Persuasive with Modification** — The submitter's core request is valid and supported by implementer evidence, but the PHD IG would benefit from a slightly broader treatment that explains the 20601/10206 relationship in context rather than just noting differences for specific attributes. Coordinating with FHIR-53350 and FHIR-53328 will produce more coherent guidance. This aligns with the WG's demonstrated willingness to improve 20601 documentation (FHIR-48010 precedent) and reflects the reality that 20601 remains actively used by devices in the field.
