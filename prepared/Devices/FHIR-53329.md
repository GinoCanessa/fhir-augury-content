## Description

Section 3.4 of the PHD IG refers in several places to "generating the PHD FHIR resources." The submitter notes that strictly speaking, the "FHIR resource" is the definition (i.e., the StructureDefinition), and the IG should use more precise terminology—such as "FHIR resource instances" or "FHIR resource representations"—to distinguish between resource type definitions and the actual data instances that a PHG creates from PHD data.

## Analysis

This is a straightforward editorial/terminology correction. The submitter correctly identifies that "generating the PHD FHIR resources" is imprecise. In FHIR, a "resource" can mean the resource type definition (StructureDefinition) or a resource instance (the actual data). The PHD IG Section 3.4 uses "FHIR resources" when it means "FHIR resource instances" that a PHG creates from PHD measurements. This ambiguity could confuse implementers, particularly those new to FHIR.

This ticket is part of a larger batch of ~10 terminology clarification tickets filed simultaneously by MalcolmClarke, all targeting the PHD IG's introductory/overview sections. The WG previously deferred these to the next revision cycle and has now re-opened them post-STU2 publication.

The change has no impact on profiles, conformance, or behavior—it is purely editorial wording improvement.

## Proposed Dispositions

### Disposition A: Applied — Clarify "FHIR resource" to "FHIR resource instance"
**Action:** Replace occurrences of "generating the PHD FHIR resources" in Section 3.4 with "generating FHIR resource instances from PHD data" (or similar phrasing like "creating FHIR resource instances that represent PHD measurements"). Review Section 3.4 holistically for consistent use of "resource instance" vs. "resource definition."

**Rationale:** The submitter's observation is technically correct. FHIR distinguishes between resource definitions and resource instances. Precise terminology improves implementer understanding and aligns the IG with standard FHIR usage patterns. This is a low-risk editorial change with clear benefit.

**Impact:** Editorial only. No change to profiles, conformance criteria, or behavior.

### Disposition B: Applied — Broader terminology pass across Sections 3–4
**Action:** In conjunction with the related batch of tickets (FHIR-53330, FHIR-53339, FHIR-53340, FHIR-53341, FHIR-53342), perform a comprehensive terminology review of Sections 3–4. Clarify "FHIR resource" → "FHIR resource instance," and address the related terminology confusion (objects vs. observations, items, PHG vs. PHD references) in a single coordinated edit pass.

**Rationale:** The batch of tickets from MalcolmClarke all target the same sections with the same type of concern. A single coordinated pass would be more efficient than piecemeal changes, ensure consistency, and avoid introducing new inconsistencies.

**Impact:** Editorial only but broader in scope. Requires coordination across multiple tickets.

### Disposition C: Not Persuasive — Context makes meaning clear
**Action:** Close with no change. Add an editorial note or leave existing text as-is, arguing that the context of Section 3.4 (which discusses PHG implementation) makes it clear that "FHIR resources" refers to instances, not definitions.

**Rationale:** Experienced implementers reading Section 3.4 in context would understand that "generating the PHD FHIR resources" means creating data instances. The distinction, while technically valid, may be overly pedantic for this audience.

**Impact:** None.

## Recommendation

**Disposition B: Applied — Broader terminology pass across Sections 3–4**

This ticket is one of a coherent batch of ~8 terminology clarification tickets from the same reporter, all targeting the same PHD IG sections, all re-opened post-STU2 for the next revision. The submitter's point is technically valid, and the fix is purely editorial with zero risk. Addressing this ticket in isolation (Disposition A) would work, but coordinating with the related tickets (Disposition B) would be more efficient and produce a more consistent result. Disposition C is not recommended because clear terminology matters in a specification, and the cost of fixing it is minimal.
