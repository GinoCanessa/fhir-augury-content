## Summary

This ticket requests that the FHIR `dateTime` datatype definition explicitly support representation of UTC time when the local timezone is unknown. The reporter notes that Personal Health Devices (PHDs) and Personal Health Gateways (PHGs) frequently do not have timezone information—for example, when time is acquired via NTP, the device knows UTC but not the local offset. The current FHIR `dateTime` definition requires a timezone if hours and minutes are specified (`YYYY-MM-DDThh:mm:ss+zz:zz`), but does not clearly define a representation for "UTC known, local timezone unknown." The PHD IG has adopted `YYYY-MM-DDThh:mm:ss-00:00` for this purpose, and the ticket asks FHIR Core to formally document this convention.

The ticket was previously resolved as "Considered for Future Use" and deferred to the next revision of the PHD IG per Devices WG review (2025-11-12 call). It was subsequently reopened post-STU2 publication by lynn_laakso for renewed consideration.

## Details

### Problem Statement
The FHIR `dateTime` specification states: *"If hours and minutes are specified, a time zone SHALL be populated."* While the `Z` suffix indicates UTC, and `+00:00` indicates the zero offset timezone, there is no clearly documented convention in FHIR for representing "time is in UTC but the local timezone is unknown." This is a meaningful semantic distinction:

- `2024-01-15T10:30:00Z` — time is UTC (and the source knows it is in UTC timezone)
- `2024-01-15T10:30:00+00:00` — time is in the +00:00 offset (semantically identical to Z)
- `2024-01-15T10:30:00-00:00` — per RFC 3339 §4.3, this indicates "UTC time with unknown local offset"

### PHD/PHG Context
Personal health devices frequently synchronize their clocks using NTP or similar protocols, giving them accurate UTC time but no knowledge of the user's local timezone. The PHD IG has defined the convention of using `-00:00` to indicate this scenario, following the RFC 3339 precedent. However, this convention is not documented in the FHIR Core `dateTime` definition, creating potential interoperability ambiguity.

## Proposed Dispositions

### Disposition A: Accept — Add explicit `-00:00` convention to FHIR Core dateTime definition
**Resolution:** Persuasive

Add a note to the FHIR Core `dateTime` datatype definition that explicitly documents the semantic meaning of `-00:00` as "UTC time with unknown local offset," following the RFC 3339 §4.3 convention. This would:
1. Add a paragraph to the `dateTime` description clarifying the three offset conventions: `Z` (UTC), `+00:00` (zero offset), and `-00:00` (UTC with unknown local offset).
2. Confirm that `-00:00` is a valid timezone value in FHIR `dateTime`.
3. Note the use case of devices/gateways that know UTC but not local timezone.

**Rationale:** This directly addresses the reporter's request, provides clear guidance for PHD/PHG implementers, aligns with RFC 3339 semantics, and reduces interoperability ambiguity. The PHD IG already uses this convention; formalizing it in FHIR Core benefits all implementers.

### Disposition B: Accept with modification — Document in PHD IG only, with FHIR Core informative note
**Resolution:** Persuasive with Modification

Rather than adding the `-00:00` convention directly to the FHIR Core `dateTime` definition, add an informative note in FHIR Core acknowledging that IGs may define conventions for distinguishing "known UTC" from "UTC with unknown local offset." The detailed `-00:00` convention would be documented in the PHD IG, which is the primary consumer of this capability. This avoids changing core datatype semantics while providing the needed clarity.

**Rationale:** This is a more conservative approach that respects the scope of FHIR Core. The `-00:00` convention, while valid per RFC 3339, is a specialized semantic distinction primarily needed by device-oriented IGs. Adding it to FHIR Core may create confusion for implementers who don't need this distinction.

### Disposition C: Decline — Existing specification is sufficient
**Resolution:** Not Persuasive

The current FHIR `dateTime` definition already permits `Z` and `+00:00` for UTC timestamps, and `-00:00` is not prohibited. The semantic distinction between "UTC known" and "UTC with unknown local offset" is a profile-level concern best handled by individual IGs (as the PHD IG already does) rather than in the core specification. The existing requirement that "a time zone SHALL be populated" when time is present is satisfied by any of these representations.

**Rationale:** Adding this level of semantic nuance to the core `dateTime` definition may overcomplicate the specification for the majority of implementers who do not need this distinction. The PHD IG's existing convention is sufficient for its use case.

## Recommendation

**Disposition A: Accept** — The `-00:00` convention is well-established in RFC 3339, already in use by the PHD IG, and addresses a genuine interoperability gap. The distinction between "known UTC" and "UTC with unknown local offset" is not unique to devices—any system ingesting time from a source that lacks timezone info (NTP, legacy systems, IoT sensors) faces this ambiguity. Documenting it in FHIR Core provides clarity for all implementers, not just the PHD community. The ticket has been reopened after STU2 publication specifically for this purpose, and the related FHIR-52878 work on the dateTime regex (now Applied) demonstrates active investment in clarifying timezone handling in FHIR Core.
