## Description

The submitter argues that identifying all Personal Health Devices (PHDs) with a single fixed code (`MDC_MOC_VMS_MDS_SIMP`) in `Device.type` discards valuable device-type information. Since `Device.type` is bound to an example value set in FHIR, individual PHDs could specify more appropriate device types. Requiring a single fixed value also hampers interoperability when a device might be used in the context of other IGs.

This was originally imported as Comment 84 by Ron G. Parker.

## Analysis

### Core Issue
The PHD IG currently requires `Device.type` to be fixed to a single MDC code (`MDC_MOC_VMS_MDS_SIMP` — "Simple MDS") for all PHDs. The submitter argues this discards useful device-type information and hinders interoperability when devices are used across IGs.

### Key Technical Constraints
- **R4 limitation:** `Device.type` has cardinality 0..1, so it can only carry one CodeableConcept. Replacing the fixed PHD code with a more specific type means losing the ability to also flag the device as a PHD.
- **R6 opportunity:** `Device.type` cardinality and binding are changing in R6, potentially allowing multiple type codes.
- **Specializations overlap:** Device-specific codes (e.g., pulse oximeter) already appear in `Device.specialization`. Duplicating them in `Device.type` creates ambiguity.

### Workgroup Position
The workgroup initially found this Persuasive (7-0-0) with a plan to define a richer value set including Bluetooth appearance codes. This was then reversed to Considered for Future Use (8-0-1) to defer to R6 due to R4 cardinality constraints. Both resolutions were subsequently reverted, and the ticket is now reopened post-STU2 for fresh consideration.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Define extensible value set, keep PHD code as default
**Action:** Define a new value set for `Device.type` containing Bluetooth appearance category codes, relevant MDC/FHIR device type codes, and `MDC_MOC_VMS_MDS_SIMP`. Bind it extensibly. Update guidance to state that `MDC_MOC_VMS_MDS_SIMP` should be used when no more specific type is known from the protocol, but more specific codes are preferred when available. This aligns with Erik Moll's 2025-05-14 proposal and the original Persuasive vote.

**Pros:**
- Preserves backward compatibility (existing PHD-coded resources remain valid)
- Adds richer device type information when available
- Extensible binding allows implementers to use additional codes
- Addresses the submitter's core concern about lost information

**Cons:**
- In R4 (0..1 cardinality), choosing a specific type code means losing the explicit PHD flag
- May need revisiting when the IG moves to R6

### Disposition B: Considered for Future Use — Defer to R6 version of the IG
**Action:** Retain the current fixed `MDC_MOC_VMS_MDS_SIMP` requirement in the R4-based IG. Document the intent to address this in the R6 version where `Device.type` supports multiple codes, allowing both the PHD identifier and specific device type to coexist. Add an informational note to the IG acknowledging the limitation.

**Pros:**
- Avoids breaking changes in the current R4-based IG
- R6's expanded cardinality will cleanly solve the problem
- Preserves the PHD identification use case (distinguishing PHDs from PoCD devices)

**Cons:**
- Delays a useful improvement indefinitely
- Continues discarding available device type information
- The IG has already been through two ballot cycles on this issue without resolution

### Disposition C: Persuasive — Allow any appropriate device type code, remove fixed value requirement
**Action:** Remove the fixed value requirement for `Device.type`. Allow implementers to use any appropriate device type code. Optionally provide a recommended value set as guidance but do not mandate it. This directly implements the submitter's original request.

**Pros:**
- Maximum flexibility for implementers
- Fully addresses the interoperability concern for cross-IG device usage
- Simplest change to implement

**Cons:**
- Loss of the ability to reliably identify PHDs by `Device.type` alone
- May break existing implementations that rely on the fixed PHD code
- Less structured than Disposition A; could lead to inconsistent coding

---

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

This approach directly implements the well-analyzed proposal by the assignee (Erik Moll) and aligns with the original 7-0-0 Persuasive vote. It addresses the submitter's concern by enabling richer device type information while preserving backward compatibility through the existing PHD code as a default. The extensible binding provides a structured path forward without waiting for R6. The ticket has been reopened post-STU2 specifically for active resolution, and further deferral (Disposition B) risks perpetuating a pattern of unresolved cycling. The fully unconstrained approach (Disposition C) sacrifices too much structure for the PHD use case.
