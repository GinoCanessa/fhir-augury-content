## Summary

The ticket requests guidance on how to map the ISO/IEEE 11073-10207 SDC `CalibrationInfo` structure to FHIR resources within the Point of Care Devices (PoCD) Implementation Guide. The existing HL7 SDC-to-FHIR mapping guidelines provide value set mappings for **calibration state** and **calibration type** that point to `DeviceMetric` properties, but the current PoCD profiles only define `DeviceMetric.calibration` mappings at the **Channel level**. In SDC, `CalibrationInfo` is part of `AbstractDeviceComponentState` and can appear at the **MDS, VMD, or Channel** state level.

The primary use case is representing an anesthesia device's daily self-test, including results, errors, and required human interactions. After system boot, `ComponentCalibrationState` transitions from `Req` (required) to `Cal` (calibrated) following a successful self-test. VMDs may also have specific calibration requirements.

## Analysis

### Core Issue
The PoCD IG has a gap: SDC `CalibrationInfo` exists at all levels of the device containment tree (MDS, VMD, Channel), but the FHIR PoCD profiles only map calibration information at the Channel level via `DeviceMetric.calibration`. There is no documented guidance for representing MDS-level or VMD-level calibration states in FHIR.

### Complexity
This is not a trivial mapping problem. The use case involves:
- **State transitions** (e.g., `Req` → `Cal` after self-test completion)
- **Multi-observation scenarios** when human interaction is needed (error codes + interaction identifiers require multiple Observations)
- **Hierarchical placement** — calibration may apply at the device level (MDS/VMD), not just the metric/channel level

### Two Proposed Approaches
1. **Device Extension approach:** Add a calibration extension to MDS/VMD Device profiles. Aligns with the fact that MDS/VMD containment is now modeled in Device (not DeviceMetric). Simpler but may not handle complex calibration result data well.
2. **DeviceMetric + Observation approach:** Use DeviceMetric resources with `parent` referencing VMD/MDS Devices, plus Observation resources for calibration values. Richer but introduces DeviceMetric instances that aren't at the Channel level, which is a departure from current PoCD profile patterns.

## Proposed Dispositions

### Disposition A: Persuasive — Add Device Extension for MDS/VMD Calibration
Add a new extension to the MDS Device and VMD Device profiles carrying `calibration-state` (using the existing SDC CalibrationState value set mapping) and `calibration-type` (using the existing SDC CalibrationType value set mapping). Update the SDC mapping page to document that Channel-level calibration continues to use `DeviceMetric.calibration`, while MDS/VMD-level calibration uses this extension. For complex calibration results involving multiple values (error codes, interaction identifiers), recommend using associated Observation resources referencing the Device.

**Rationale:** Aligns with Todd Cooper's recommendation from the Zulip discussion. Since MDS/VMD containment is now represented via the Device resource (not DeviceMetric), adding a calibration extension to Device is the natural location. This keeps the existing `DeviceMetric.calibration` pattern intact for Channel-level metrics while filling the gap at higher containment levels.

### Disposition B: Persuasive with Modification — Add Guidance Text Only (Defer Extension)
Add the text proposed by erikmoll to the PoCD 1.1.0 SDC mapping page, documenting both approaches (Device extension and DeviceMetric with parent references) as possible strategies. Mark this as an open design question for a future ballot cycle, without committing to a specific normative approach. Include implementer guidance that either approach may be used for now.

**Rationale:** The problem is recognized but the best solution is not yet settled. The Zulip discussion and comments show ongoing debate about whether a Device extension or DeviceMetric approach is better. Publishing guidance text acknowledges the gap without prematurely constraining the solution. This is lower risk for PoCD 1.1.0.

### Disposition C: Not Persuasive — Existing DeviceMetric.calibration Is Sufficient
Close the ticket noting that `DeviceMetric.calibration` already supports calibration state and type, and that `DeviceMetric.parent` can reference any Device (including MDS/VMD). Implementers can create DeviceMetric instances at the VMD/MDS level by setting `DeviceMetric.parent` to the appropriate Device. No specification change needed.

**Rationale:** The existing FHIR resources technically support this use case without modification — DeviceMetric is not formally restricted to Channel-level only in the core spec, and `parent` can reference any Device. However, this contradicts the current PoCD profile patterns and lacks explicit IG guidance.

---

## Recommendation

**Disposition A: Persuasive — Add Device Extension for MDS/VMD Calibration**

This is the strongest approach because: (1) Todd Cooper, a domain expert and PoCD IG maintainer, recommended the Device extension direction in the Zulip discussion; (2) it follows the established architectural direction where MDS/VMD containment lives in Device rather than DeviceMetric; (3) erikmoll's recent comments confirm the workgroup has narrowed to two options and a decision is needed; and (4) the anesthesia device self-test use case is concrete and well-understood, providing sufficient requirements to design the extension now rather than deferring. The extension should be complemented by mapping page text explaining its usage for MDS/VMD-level `CalibrationInfo`, plus guidance on using Observations for complex calibration result values.
