## Summary

This change request asks the Point of Care Devices (PoCD) Implementation Guide to include use case examples demonstrating how device data modeled in IEEE 11073 standards is represented in FHIR resources. The ticket was originally filed in January 2018 and was resolved as "Considered for Future Use" in March 2018 with a 9-0-0 vote acknowledging the suggestion's merit. The work group intended to apply these examples in the STU2 version but the examples were never added. In April 2026 the ticket was re-opened by a 15-0-0 vote (Erik Moll / Elliot Silver motion) and reverted to Triaged status.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Add one or more use case examples to the Point of Care Devices Implementation Guide that demonstrate how IEEE 11073 device data (e.g., metric observations, device descriptions, and containment hierarchies) maps to FHIR resources such as Device, DeviceMetric, and Observation. The examples should walk through a representative IEEE 11073 device model and show the corresponding FHIR resource instances.
#### Justification
The work group has twice expressed consensus that this enhancement is valuable — first with a 9-0-0 vote in March 2018 and again with a 15-0-0 vote to re-open in April 2026. Concrete use case examples are a recognized best practice in implementation guides, and the IEEE-to-FHIR mapping is a core concern of the PoCD IG. Providing these examples would help implementers understand the intended usage patterns and reduce ambiguity in device data representation.

---

### Disposition B: Alternative Approach
#### Proposal
Instead of embedding full narrative use cases in the IG, create a structured mapping reference (e.g., a table or ConceptMap resource) that documents the correspondence between IEEE 11073 domain information model elements and FHIR PoCD resources. Supplement this with one concise worked example rather than multiple use cases.
#### Justification
A structured mapping reference would be easier to maintain as both the IEEE and FHIR standards evolve. A single worked example paired with a systematic mapping table may serve implementers better than multiple narrative use cases, which can become outdated. This approach also reduces the editorial burden on the work group while still addressing the core request to illustrate the IEEE-to-FHIR transition.

---

### Disposition C: Decline
#### Proposal
Decline the request on the basis that the PoCD IG has evolved substantially since 2018 and the scope and context of the original request may no longer align with the current IG direction. Recommend that IEEE-to-FHIR mapping guidance be addressed through external educational materials or a separate companion document rather than within the IG itself.
#### Justification
The ticket has been open for over eight years and the PoCD IG has undergone significant changes in that time. The lack of a detailed description in the original ticket makes it difficult to determine precisely what was envisioned. Declining would allow the work group to focus editorial resources on normative content and address educational needs through other channels. However, this disposition runs counter to two unanimous votes in favor of the enhancement.

---

### Recommendation
**Recommended disposition:** A

The work group has demonstrated clear and sustained consensus for this enhancement, with two unanimous votes spanning eight years — a 9-0-0 vote in 2018 acknowledging the suggestion's merit and a 15-0-0 vote in 2026 to re-open the ticket. The recent re-opening indicates that the work group considers these use cases still relevant and valuable for the current IG. Adding IEEE-to-FHIR use case examples directly addresses a core implementer need and aligns with the IG's purpose of guiding the representation of point-of-care device data in FHIR. Given this strong consensus and the practical value to implementers, accepting the request as stated is the most appropriate path forward.
