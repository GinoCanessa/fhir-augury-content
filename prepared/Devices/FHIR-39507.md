The reporter notes that the PoCD IG's RESTful transfer page describes FHIR-generic options for transferring device-sourced data, but lacks clear guidance on how to handle **Patient** and **Location** resources specifically. Key questions include: when to create and include these resources in a device-data bundle versus referencing existing resources on the destination FHIR server, and how to align with additional resource profiles that a specific hospital may require. The reporter suggests that deployment archetypes and use cases should be identified to provide this guidance, and notes a possible connection to workflow discussions in the Orders & Observations WG.

## Details

### Proposed Resolution

The resolution (marked **Persuasive**) proposes adding a new section to the PoCD IG profiles overview page titled **"Patient and Location Resource Handling"** with the following subsections:

1. **Reference Patterns** — Three patterns for including Patient and Location references in device data:
   - **Reference existing resources**: The device/gateway resolves references by searching with known identifiers before submitting observations (most common in enterprise deployments).
   - **Conditional create in the bundle**: Include Patient/Location in a transaction bundle with `ifNoneExist`; requires strong identifiers (e.g., MRN) to avoid duplicates.
   - **Include in the bundle**: The device is the source of truth for the resource (rare for Patient; possible for Location in mobile/temporary setups).

2. **Minimum Identity Requirements** — Patient references should include at least an MRN or facility-specific identifier from an ADT system. Location references should include at least a bed label or location identifier matchable against server resources.

3. **Conformance to Destination Profiles** — The IG does not profile Patient or Location for clinical use; implementers should conform to profiles required by their deployment context (e.g., US Core, IPA).

4. **Future Work** — Deployment archetypes and workflow integration patterns (including device observations and clinical orders) are deferred to future versions.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Accept the Resolution)

Accept the proposed resolution text as-is. The new "Patient and Location Resource Handling" section addresses the gap identified in the original ticket by:
- Enumerating three reference patterns (reference existing, conditional create, include in bundle) with clear deployment-context guidance for when each is appropriate.
- Specifying minimum identity requirements for Patient (MRN/facility identifier) and Location (bed label/location identifier).
- Clarifying that the IG does not profile Patient/Location for clinical use and deferring to deployment-context profiles (US Core, IPA).
- Scoping future work for deployment archetypes and workflow integration.

The original reporter has endorsed the text as sufficient for the current IG version. A preview build is available for review.

### Disposition B: Accept with Modification — Strengthen Cross-References

Accept the resolution but request the following modifications:
- Add explicit cross-references to FHIR-33695 (Observation Subject Guidance for when patient is unknown) to ensure the new text links to the existing handling of unknown subjects.
- Incorporate mcros's suggestion from WGM-Sept25 to include informative references to the **RTLS IG** and **IHE-MEM-LS** as examples of related approaches for location management.
- Verify that the "Future Work" section aligns with the project scope statement as noted by the original reporter.
- Fix the broken internal link to the RESTful Transfer page (currently points to a `file+.vscode-resource` URI in the proposed text).

### Disposition C: Decline — Request Broader Scope

Return as Not Persuasive with Modification, requesting a more comprehensive treatment that includes:
- Concrete deployment archetype definitions rather than deferring them entirely to future work.
- Alignment with O&O WG workflow patterns (as the reporter originally noted).
- Prescriptive recommendations rather than informative guidance, including conformance language (SHALL/SHOULD/MAY).

This would require significantly more work and would delay the current ballot cycle.

## Recommendation

**Disposition B: Accept with Modification** is recommended.

The proposed resolution is well-structured and addresses the core gap. The original reporter and the implementer both agree the content is appropriate for the current IG version. However, the modifications in Disposition B address real issues: the broken `file+.vscode-resource` URI in the conditional-create cross-reference link needs fixing before publication, the cross-reference to FHIR-33695's Observation Subject Guidance would strengthen coherence, and incorporating the WGM-Sept25 suggestion to reference RTLS IG and IHE-MEM-LS would add practical value at minimal cost. The future-work scope alignment check flagged by the reporter is also a low-effort, high-value verification step.
