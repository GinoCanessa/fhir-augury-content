## Description Summary

The `Device-alertDetection` extension (created by [FHIR-43708](https://jira.hl7.org/browse/FHIR-43708)) applies to both `Device` and `DeviceMetric` contexts. This ticket raises three concerns about the `limitRange` subextension:

1. **Missing guidance on when `limitRange` is appropriate:** During the original discussion (FHIR-43708), a comment noted that additional guidance may be needed to discourage the `limitRange` subextension when the extension is applied to `Device` without an `alertCode`. The intent was to discourage `limitRange` when no specific alert code is given (since it is meaningless to specify a limit range—e.g., 65–100°C—without identifying which alert has that limit). No text was added at the time.

2. **Language is too DeviceMetric-centric:** The current `limitRange` definition text assumes application to `DeviceMetric` specifically (referring to "the DeviceMetric" for unit matching). The language should be generalized to accommodate use on both `Device` and `DeviceMetric`.

3. **Possible formal constraint:** A FHIRPath constraint could enforce that `limitRange` is only present when `alertCode` is also present, regardless of the resource context.

## Comments Analysis

### esilver (2025-10-17) — Proposed Resolution
Elliot Silver (the reporter) provided a detailed self-authored proposal with three specific changes:

1. **Add a note** to the extension documentation:
   > Since the `limitRange` element of this extension records the values outside of which an active alert of the specified priority is detected, it may be populated only if an `alertCode` is specified. (It doesn't make sense to specify a limit of 65–100 degrees Celsius without identifying which alert has that limit.)

2. **Update the `limitRange` definition** to be more context-inclusive:
   > The range of values outside of which an alarm condition of the (single) specified priority would be triggered for the specified alert code, if the alert activation state were `active`. Limits may include upper or lower values, or both. **The range should be in the same units as when the associated value is reported. When this extension is applied to DeviceMetric, the units, if present, should match DeviceMetric.unit; if absent, the units are implied to be the same as DeviceMetric.unit.**

   (Replacing the current text: *"If units are present, they should be the same as the DeviceMetric. If absent, units are assumed to be the same as the DeviceMetric."*)

3. **Add constraint `alrtdet-2`:**
   - Human: Subextension limitRange applies only to a specific alert code.
   - Expression: `extension('limitRange').exists() implies extension('alertCode').exists()`

### javier_espina (2026-01-22)

## Proposed Dispositions

### Disposition A: Persuasive
**Accept the reporter's proposed changes as written.**

Apply all three changes from esilver's 2025-10-17 comment:
- Add the explanatory note that `limitRange` may only be populated if `alertCode` is specified.
- Update the `limitRange` definition to use more inclusive, context-aware language about unit matching.
- Add the `alrtdet-2` FHIRPath constraint: `extension('limitRange').exists() implies extension('alertCode').exists()`.

**Rationale:** The proposal is well-reasoned, addresses a real gap in the specification, and was authored by the reporter who has deep familiarity with the alertDetection extension design. The constraint is logically sound — specifying alert limits without identifying the alert they apply to is semantically meaningless. The updated text properly generalizes the unit guidance beyond DeviceMetric-only contexts.

### Disposition B: Persuasive with Modification
**Accept the intent but refine the constraint and text.**

Accept the updated `limitRange` definition text and the `alrtdet-2` constraint, but:
- Strengthen the note to also explain the relationship between `limitRange` and compound metrics (per FHIR-48895), since a compound metric may have component-specific limits.
- Consider whether the constraint should also account for the `priority` subextension (e.g., should `limitRange` require both `alertCode` and `priority`?).
- Review the Workshop 2 background material (section 1.e.iii, per Javier Espina's comment) to confirm the constraint aligns with the original design intent before finalizing.

**Rationale:** The core proposal is sound, but given the active evolution of the alertDetection extension (compound metrics, terminology migration), a slightly broader review of the constraint scope may prevent future follow-up tickets.

### Disposition C: Not Persuasive
**Defer until the extension has more implementation experience.**

The `Device-alertDetection` extension and `DeviceAlert` resource are still maturing (DeviceAlert recently reached FMM 2). Adding constraints now may prematurely restrict valid use cases that have not yet been encountered. The current text, while imperfect, does not prevent correct implementation.

**Rationale:** The extension is still evolving (see FHIR-48895 for compound metrics, FHIR-49183 for terminology changes, and ongoing Zulip discussions about binding types). Constraints added now may need to be relaxed later if new use cases emerge, which creates backward-compatibility concerns.

## Recommendation

**Disposition A: Persuasive.**

The proposed changes are specific, well-justified, and address a clear specification gap. The reporter (Elliot Silver) has been the primary driver of the alertDetection extension work and has provided complete, ready-to-apply text and a formal FHIRPath constraint. The constraint (`limitRange` requires `alertCode`) is logically necessary — specifying alert limits without identifying the alert is meaningless — and the updated definition text correctly generalizes the unit-matching guidance for both `Device` and `DeviceMetric` contexts. The workgroup should review the Workshop 2 background material (section 1.e.iii) referenced by Javier Espina to confirm alignment, but the proposal as written is sound and ready for application.
