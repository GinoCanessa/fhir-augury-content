## Description

The submitter observes that the Metric availability extension in the PoCD IG currently uses a value of type `code` with an `extensible` binding strength. According to FHIR core specification guidance ([FHIR R4 Terminologies](http://hl7.org/fhir/R4/terminologies.html#required)), elements of type `code` should generally use `required` binding strength so that values can be interpreted with confidence. The submitter requests the extension be updated to either:

1. Change the binding strength to **required** (if the code set is meant to be strictly controlled), or
2. Change the type from `code` to `Coding` (which makes `extensible` binding appropriate).

## Resolution Description

The proposed resolution is **Persuasive with Modification**. The resolution aligns the R4 PoCD extension with the approach taken in FHIR R6, where `DeviceMetric.availability` is a first-class element of type **CodeableConcept** with **Extensible** binding to the [metric-availability ValueSet](https://build.fhir.org/valueset-metric-availability.html).

Specifically, the resolution proposes:
- Change the extension's value type from `code` to `CodeableConcept` with `Extensible` binding (mirroring R6).
- Keep the extension in the R4 PoCD IG.
- Use a URL like `http://hl7.org/fhir/StructureDefinition/metric-availability` for the extension, anticipating it will become available as a "backport" extension from R6.
- Use corresponding CodeSystem/ValueSet in the IG with THO URLs, anticipating migration from FHIR Core to THO (see [FHIR-33760](https://jira.hl7.org/browse/FHIR-33760)).

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification (Align with R6 using CodeableConcept)
**Action:** Change the metric availability extension value type from `code` to `CodeableConcept` with `Extensible` binding, mirroring the R6 DeviceMetric.availability element. Use a FHIR Core-style extension URL (e.g., `http://hl7.org/fhir/StructureDefinition/metric-availability`) anticipating R6 backport availability. Include corresponding CS/VS in the IG using THO URLs per the approach in FHIR-33760/FHIR-33768.

**Rationale:** This follows the resolution already proposed by the commenter (erikmoll) and aligns the R4 PoCD IG with the direction established in R6. Changing to `CodeableConcept` with `Extensible` binding resolves the original concern about the mismatch between `code` type and `extensible` binding. It also provides forward compatibility, making migration to R6 smoother for implementers.

**Impact:** Breaking change for existing R4 PoCD implementers using the `code`-typed extension — they would need to update to `CodeableConcept`. However, this is a pre-normative IG change during ballot, which is the appropriate time for such corrections.

### Disposition B: Accept as Persuasive (Change binding to Required, keep type as code)
**Action:** Keep the value type as `code` but change the binding strength from `extensible` to `required`, consistent with FHIR core guidance for code-typed elements.

**Rationale:** This is the simpler of the two options proposed by the submitter. It directly addresses the spec inconsistency with minimal structural change. If the set of metric availability codes is considered complete and stable, a `required` binding is appropriate.

**Impact:** Minimal structural impact — only the binding strength changes. However, this diverges from the R6 approach (which uses `CodeableConcept`) and would require a different migration path when moving to R6. It also limits extensibility for implementers who may need additional availability codes.

### Disposition C: Not Persuasive
**Action:** No change. Keep the current `code` type with `extensible` binding.

**Rationale:** The current definition is technically valid even if it does not follow the general recommendation. The FHIR spec guidance for `code` + `required` is a recommendation ("generally used"), not an absolute rule. Implementers can still use the extension as defined.

**Impact:** No implementation changes needed, but the inconsistency with core guidance remains. This would leave the R4 IG out of alignment with the R6 approach, complicating future migration.

## Recommendation

**Disposition A: Accept as Persuasive with Modification** — Align the R4 PoCD metric availability extension with the R6 DeviceMetric approach by changing the value type to `CodeableConcept` with `Extensible` binding. This resolves the reported inconsistency, establishes forward compatibility with R6, and aligns with the resolution already proposed by the Devices WG commenter. The related THO migration work (FHIR-33760, FHIR-33768, FHIR-53238) supports this direction. Since this is a ballot-period change, now is the appropriate time to make this correction before the IG is normative.
