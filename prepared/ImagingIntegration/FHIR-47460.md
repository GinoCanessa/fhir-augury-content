## Description

The submitter notes that the FHIR specification proposes a modeling pattern for grouped observations using `Observation.hasMember`, in which only `Observation.code` is populated and `Observation.value` is left empty (see [Observation Grouping guidance](https://build.fhir.org/observation.html#gr-other)). The ImagingMeasurementGroup profile populates both `.code` AND `.value` while also using `.hasMember`, which the submitter argues goes against the FHIR modeling pattern.

## Comment History

1. **jwhitby** (2024-09-23): Agrees this goes against the described pattern, but notes the FHIR text says this is "not typically done" rather than forbidding it. Points out that for measurement groups, `code` is already mapped to Finding Category (describing the type of observation) while Finding specifies the observation value — providing a rationale for populating both.

2. **ana_kostadinovska** (2025-07-14): Acknowledges it is guidance, not a requirement. Suggests that if there are reasons to populate both code and value, the IG should add an explanation for why it deviates from the FHIR modeling pattern.

## Resolution Description

The resolution (Not Persuasive) cites additional guidance from the Observation resource that covers this exact scenario:

> "When linking to other Observations from which an Observation is derived. In this case both `Observation.code` and `Observation.value[x]` are present, and the linked observations are listed in `Observation.derivedFrom`."

The resolution argues this derived observation pattern (e.g., BMI derived from height and weight) fits the imaging measurement group use case, thereby justifying having both code and value populated.

## Proposed Dispositions

### Disposition A: Not Persuasive (Confirm Current Resolution)
Uphold the current "Not Persuasive" resolution. The FHIR specification's grouped observation guidance is not a hard constraint — it says observations with `hasMember` "typically" have no value, not that they must not. Furthermore, the resolution correctly identifies the "derived observation" pattern (code + value + derivedFrom) as applicable to imaging measurement groups where measurements are derived from DICOM SR data. No change to the IG is needed.

### Disposition B: Persuasive with Modification — Add Explanatory Text
Accept the submitter's underlying concern and add a brief note to the IG explaining why `ImagingMeasurementGroup` populates both `.code` and `.value` despite the general grouped-observation guidance. This follows the submitter's own suggestion (comment of 2025-07-14) and would improve implementer understanding without changing the technical mapping. The added text could reference the derived observation pattern cited in the resolution.

### Disposition C: Persuasive — Remove value from ImagingMeasurementGroup
Accept the change request as filed and remove the `.value` population from the ImagingMeasurementGroup profile to align with the standard grouped observation modeling pattern. This would require reworking how Finding is mapped — potentially moving it to a child observation or an extension — which could be a significant structural change to the IG.

## Recommendation

**Disposition B: Persuasive with Modification — Add Explanatory Text**

The technical justification in the current resolution is sound: the derived observation pattern does allow both code and value. However, both the submitter and a commenter (jwhitby) noted the apparent deviation from the grouped observation guidance, which suggests implementers will encounter the same confusion. Adding a brief explanatory note to the IG (as the submitter herself suggested in her July 2025 comment) would resolve this concern at minimal cost — no structural changes to mappings, no breaking changes, just improved documentation. This balances correctness with clarity and is consistent with the spirit of the original feedback.
