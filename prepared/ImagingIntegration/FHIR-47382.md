## Description

The submitter observes that the DICOM SR to FHIR mapping diagram maps only the **Preliminary flag** from DICOM SR. They ask why the **Completion flag** and **Verification flag** are not also mapped, and request clarification on this design decision.

## Analysis

The core question is whether DICOM SR **Completion** and **Verification** flags should be mapped to FHIR alongside the already-mapped **Preliminary** flag. In DICOM SR, the Preliminary flag maps naturally to `Observation.status = "preliminary"` in FHIR. However, as noted by commenter jwhitby, the DICOM Completion flag and Verification flag do not have immediately obvious FHIR target elements.

Potential FHIR targets that the workgroup could consider:
- **Observation.status** — could potentially accommodate the Completion flag (e.g., mapping to `final` or `amended`), but the Verification flag (which indicates author/verifier identity and timestamp) is semantically different.
- **Observation.note** or extensions — could capture verification metadata, but this would require profiling.
- The lack of a clear FHIR element for verification metadata is a genuine gap that the IG should at minimum document, even if no mapping is provided.

This ticket relates to a broader pattern across the SR Mapping ballot comments requesting improved clarity and completeness in the mapping diagrams.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Add mappings for the Completion flag to `Observation.status` (mapping DICOM "complete" to FHIR `final`) and document in the IG why the Verification flag is not mapped, noting the absence of a suitable FHIR target element. Add a brief explanatory note to the mapping diagram clarifying which DICOM SR flags are mapped and why.

**Rationale:** This partially addresses the submitter's concern by mapping the Completion flag where a natural FHIR target exists, while being transparent about the Verification flag gap. It improves the IG's completeness without requiring new FHIR extensions.

### Disposition B: Persuasive
**Action:** Map both the Completion flag and the Verification flag to FHIR. Map Completion flag to `Observation.status`. Define a FHIR extension or use `Observation.performer` and `Observation.issued` to capture the Verification flag's verifier identity and verification timestamp. Update the mapping diagrams accordingly.

**Rationale:** This fully addresses the submitter's request by providing mappings for all three flags. However, it may require defining extensions and adds implementation complexity that may not be justified if verification metadata is not commonly needed in the FHIR receiving system.

### Disposition C: Not Persuasive with Modification
**Action:** Retain the current mapping (Preliminary flag only) but add a clarifying note in the IG explaining that the Completion and Verification flags are intentionally not mapped because they lack clear FHIR target elements, and that implementers needing this information should refer to the original DICOM SR document.

**Rationale:** This preserves the current IG design while addressing the submitter's request for clarification. The "not persuasive" disposition acknowledges that the current design is intentional, while the modification improves documentation.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended. The Completion flag has a natural mapping to `Observation.status = final` that is straightforward to add and improves the IG's usefulness. The Verification flag lacks a clear FHIR target, and documenting this gap transparently is more valuable than either ignoring it or adding complex extensions. This approach balances completeness with pragmatism and aligns with the broader ballot feedback pattern requesting improved mapping clarity in the DICOM SR to FHIR IG.
