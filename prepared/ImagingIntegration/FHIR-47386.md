## Description

The submitter (Ignacio Jauregui, Philips) notes that the DICOM SR to FHIR mapping diagram is unclear about how to handle the two elements that map to BodyStructure resources. Specifically, the question is:

- Should implementers create **two separate BodyStructure instances** — one holding the Tracking Identifier mapping and another holding the Finding Site value?
- Or should implementers create **a single BodyStructure instance** that is populated from both sources (Tracking Identifier and Finding Site)?

The request asks for clarification in the IG to resolve this ambiguity.

## Analysis

This ticket is part of a cluster of September 2024 ballot feedback on the DICOM SR to FHIR Resource Mapping IG, specifically around how BodyStructure resources are used to represent anatomical information from DICOM SR documents. Several closely related tickets (FHIR-47388, FHIR-47392, FHIR-47477) raise overlapping concerns about:

1. **Whether one or two BodyStructure instances should be created** when both a Tracking Identifier and a Finding Site are present in the SR.
2. **Whether separate BodyStructure profiles are needed** for tracking identifiers vs. finding site (FHIR-47388, now Applied).
3. **Whether tracking identifiers belong on BodyStructure at all** vs. directly on Observation.identifier (FHIR-47477, resolved No Change — the WG kept them on BodyStructure).

The resolution is marked **Persuasive**, indicating the WG agrees the diagram and narrative need clarification. Related ticket FHIR-47388 (Applied) already merged the two profiles, and FHIR-47941 (Applied) clarified that Observation.bodySite should be used when only a code is present (no BodyStructure needed). These Applied changes likely address part of the underlying confusion, but this ticket (FHIR-47386) still needs its own explicit clarification applied to the IG text and diagrams.

## Proposed Dispositions

### Disposition A: Apply – Clarify diagram and narrative text (Recommended)

**Action:** Add explicit guidance in the IG explaining that when both a Tracking Identifier and a Finding Site map to BodyStructure, a **single BodyStructure instance** should be created that combines both the tracking identifier (in `BodyStructure.identifier`) and the finding site information (in `BodyStructure.includedStructure`). Update the mapping diagram to show a single BodyStructure resource receiving input from both DICOM SR elements, with annotations indicating which BodyStructure fields each source populates.

**Rationale:** The resolution is Persuasive and the ambiguity is real — implementers need unambiguous guidance on instance cardinality. Related Applied changes (FHIR-47388 merging profiles, FHIR-47941 clarifying when BodyStructure is not needed) have already shifted the model toward consolidation, and this clarification aligns with that direction. This is the most direct response to the submitter's request.

### Disposition B: Apply – Add worked example only

**Action:** Rather than modifying the diagram, add a worked example (e.g., a JSON BodyStructure instance) in the IG showing how both the Tracking Identifier and Finding Site are populated in a single resource instance, with commentary explaining the mapping from each DICOM SR element.

**Rationale:** This addresses the ambiguity through concrete demonstration without requiring diagram changes, which may be complex. However, it leaves the diagram itself potentially confusing and may not fully resolve the issue for implementers who rely on the visual mapping.

### Disposition C: Defer – Mark as addressed by related Applied tickets

**Action:** Close as already addressed by the Applied changes from FHIR-47388 (merged BodyStructure profiles) and FHIR-47941 (bodySite vs. BodyStructure guidance), with a comment noting that the consolidated profile and updated guidance resolve the ambiguity.

**Rationale:** If the WG believes the Applied changes from related tickets have already made the mapping sufficiently clear, no additional work is needed. However, this risks leaving residual ambiguity in the diagram if it was not updated as part of those other changes.

## Recommendation

**Disposition A: Apply – Clarify diagram and narrative text.** The resolution is Persuasive and the submitter's question is specific and actionable. The related Applied tickets addressed adjacent concerns (profile consolidation, when to use bodySite vs. BodyStructure) but none explicitly addressed the diagram clarity question raised here. A direct update to the diagram and narrative text showing that a single BodyStructure instance should be created from both sources is the clearest path to resolving this feedback.
