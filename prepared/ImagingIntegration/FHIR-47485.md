## Summary

The submitter reports that the Qualitative Evaluation profile in the DICOM SR to FHIR Resource Mapping IG does not appear to support free-text answers. According to DICOM, Qualitative Evaluations allow encoding a flat list of name-value pairs where questions are coded but answers may be either **coded** or **text**. The current value definition in the IG only appears to support coded data (via `valueCodeableConcept`), and the submitter requests that text-only answers be supported as well.

## Details

The resolution of "Not Persuasive with Modification" acknowledges the concern but clarifies that the existing `CodeableConcept` data type already supports text-only values. Specifically:

1. **`CodeableConcept` includes a `.text` element** — The FHIR R5 `CodeableConcept` datatype ([documentation](https://hl7.org/fhir/R5/datatypes.html#CodeableConcept)) allows encoding text without any coding entries. A text-only qualitative evaluation can be represented using `valueCodeableConcept.text` without any `valueCodeableConcept.coding`.

2. **The mapping table already covers both** — The existing mapping at [imaging-qualitative-evaluation-mappings](https://hl7.org/fhir/uv/dicom-sr/2024SEP/StructureDefinition-imaging-qualitative-evaluation-mappings.html) includes both text and coded values as sources for `valueCodeableConcept`.

3. **Suggested improvement** — The resolution suggests that the mapping line could be split to make the distinction clearer: `text → valueCodeableConcept.text` and `code → valueCodeableConcept.coding`. This would make it more obvious that text-only values are supported.

## Proposed Dispositions

### Disposition A: Accept the Existing Resolution (Not Persuasive with Modification)
Agree with the current resolution that `CodeableConcept` already supports text-only values and the mapping table already covers both text and coded sources. Accept the suggested modification to **split the mapping line** in the Qualitative Evaluation mapping table so that:
- `text → valueCodeableConcept.text`
- `code → valueCodeableConcept.coding`

This makes the existing capability explicit without any structural change to the profile. The IG editor should update the mapping table at `StructureDefinition-imaging-qualitative-evaluation-mappings` to show these two distinct mapping rows.

### Disposition B: Enhance with Additional Guidance
Accept the resolution but go further by adding an **informational note or usage guidance** to the Qualitative Evaluation profile narrative explaining that text-only qualitative evaluations (without a coded value) should populate `valueCodeableConcept.text` alone, and optionally include an example instance showing a text-only qualitative evaluation. This provides clearer implementer guidance beyond just splitting the mapping row.

### Disposition C: Decline (No Change Needed)
Decline the request on the basis that:
- The `CodeableConcept` datatype inherently supports text-only values, which is documented in the base FHIR specification.
- The mapping table already includes text as a source for `valueCodeableConcept`.
- Implementers familiar with FHIR datatypes should understand this capability without additional IG-level clarification.

No changes to the IG are necessary.

---

## Recommendation

**Disposition A: Accept the Existing Resolution (Not Persuasive with Modification)**

The submitter raised a legitimate usability concern — while text-only values *are* technically supported by `CodeableConcept`, the mapping table as written does not make this obvious to implementers coming from the DICOM SR side. Splitting the mapping line into two rows (`text → valueCodeableConcept.text` and `code → valueCodeableConcept.coding`) is a minimal, low-risk change that directly addresses the confusion without altering the profile structure. The existing resolution already proposes this modification, and it aligns with the spirit of the request. This is the most pragmatic path: it validates the reporter's concern while keeping the change scoped and non-breaking.
