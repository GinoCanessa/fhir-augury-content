## Description

The reporter notes that the example provided in the IG only shows DICOM information and asks whether the mapped FHIR representation was missed. The suggestion is to include both the DICOM and FHIR representations in the example.

## Resolution Description

The resolution states: "As noted in comment, the goal of the IG is **not** to produce a representation of the entire SR as a DiagnosticReport or Composition."

## Key Comments

- **Jonathan Whitby (2024-09-24):** Explains the intent was not to create a full FHIR DiagnosticReport representing the DICOM SR contents, but rather to use the FHIR Example mechanism to surface a DICOM JSON SR as a FHIR Example. Notes that the contents are mostly mapped in the main mapping section of the IG. Asks whether creating a DiagnosticReport would be confusing since it is not an intended output of the IG.

## Proposed Dispositions

### Disposition A: Not Persuasive (matches pre-applied resolution)

**Rationale:** The IG's stated scope is to define mappings from individual DICOM SR elements to FHIR resources—not to produce a complete FHIR representation (e.g., DiagnosticReport or Composition) of an entire SR document. The example was intentionally designed to surface a DICOM JSON SR using the FHIR Example mechanism so the source data could be inspected. The mapped FHIR resources are already demonstrated in the mapping sections of the IG. Adding a synthesized DiagnosticReport to the example could mislead implementers into thinking that producing such a resource is an intended output of this IG.

**Suggested response to commenter:** "Thank you for the suggestion. The purpose of this example is to display the source DICOM SR in JSON form as a FHIR example; the IG's goal is not to produce a full FHIR DiagnosticReport or Composition from the entire SR. The individual FHIR resource mappings are covered in the main mapping sections."

### Disposition B: Persuasive with Modification

**Rationale:** While the IG is not intended to produce a full DiagnosticReport, the current example could be enhanced with an explanatory note clarifying: (a) that the example shows only the source DICOM SR, (b) that the mapped FHIR resources are found in the main mapping sections, and (c) that the IG does not aim to produce a composite FHIR document. This would address the reporter's confusion without altering the IG's scope or adding a potentially misleading DiagnosticReport example.

**Suggested response to commenter:** "We agree this could be clearer. We will add a note to the example explaining that it shows the source DICOM SR data, and that the corresponding FHIR resource mappings are detailed in the mapping sections. No full DiagnosticReport will be added as it is outside this IG's scope."

### Disposition C: Not Persuasive with Modification

**Rationale:** The core request to include a full FHIR DiagnosticReport alongside the DICOM example is not persuasive for the reasons stated in Disposition A. However, the confusion suggests the example section could benefit from a brief inline comment or annotation indicating that this is the source DICOM data only, with a cross-link to where mapped FHIR resources are defined. This is a minor editorial improvement that does not change the IG's scope.

**Suggested response to commenter:** "The request for a full FHIR representation is not persuasive, as producing a DiagnosticReport is outside this IG's scope. However, we will consider adding a clarifying note to the example section pointing readers to the mapping sections for the corresponding FHIR resources."

## Recommendation

**Disposition A: Not Persuasive** — The pre-applied resolution is well-supported. The IG author (Jonathan Whitby) clearly explains that the example serves a specific technical purpose: displaying source DICOM JSON via the FHIR Example mechanism. The IG's scope is element-level mapping, not full-document transformation. Adding a synthetic DiagnosticReport would risk implying an output the IG does not intend to define. The related ticket cluster (FHIR-47223, FHIR-47487, etc.) shows broader feedback about example clarity, but those are addressed through their own tickets; this specific request goes beyond clarification into scope expansion that the WG has appropriately declined.
