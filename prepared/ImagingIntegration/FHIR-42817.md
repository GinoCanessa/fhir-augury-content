## Summary

DICOM-defined ValueSet bindings in FHIR resources (e.g., ImagingStudy) point to HTML pages on the DICOM website (e.g., `http://dicom.nema.org/medical/dicom/current/output/chtml/part16/sect_CID_244.html`) rather than directly resolvable ValueSet endpoints. The actual ValueSet JSON is only available via FTP links found on those HTML pages (e.g., `ftp://medical.nema.org/medical/dicom/resources/valuesets/fhir/json/ValueSet-dicom-cid-244-Laterality.json`). This makes it impossible for the FHIR IG Publisher and Validator to retrieve and validate ValueSet contents, producing warnings during validation of resources like ImagingStudy and FHIRcast profiles.

The reporter requests that these bindings be updated to point to versioned, directly resolvable ValueSet definitions so that standard FHIR tooling can resolve them properly.

---

## Details

### Problem

1. **Non-resolvable bindings**: ValueSet URIs in resource definitions point to DICOM HTML documentation pages, not to FHIR-conformant ValueSet endpoints.
2. **FTP-only access**: The actual ValueSet JSON files are only available via FTP, which is not supported by the FHIR Publisher/Validator.
3. **Missing versioning**: The ValueSet references lack version indicators (e.g., `|6.0.0-cibuild`), making it ambiguous which version of the ValueSet is intended.
4. **Validation warnings**: Resources using these bindings (ImagingStudy, FHIRcast profiles) generate validation warnings because the tooling cannot resolve the bound ValueSets.

### Discussion History

- **2024-01-30 (WGM)**: DICOM WG-06 stated they will **not** address this via a DICOM change. The working group discussed that changing FTP resources to HTTP alone would be an improvement. Bas van den Heuvel (bvdh) offered to help.
- **2024-03-12 (john_moehrke)**: Suggested the solution is to add a dependency on the `fhir.dicom` package (maintained by Grahame Grieve), which is periodically regenerated from DICOM releases. Adding this IG dependency resolves the URI issues:
  ```yaml
  dependencies:
    fhir.dicom:
      uri: http://fhir.org/packages/fhir.dicom
      version: 2024.1.20240120
  ```
- **2024-04-08 (bvdh)**: Agreed that adding the `fhir.dicom` dependency is the correct approach. Created a branch [`FHIR-42817`](https://github.com/HL7/fhircast-docs/tree/FHIR-42817) in the `HL7/fhircast-docs` repository to implement this fix.

### Proposed Fix

Add `fhir.dicom` as a dependency in the FHIRcast IG's `sushi-config.yaml` (or equivalent IG configuration). This package contains FHIR-conformant representations of DICOM ValueSets, allowing the IG Publisher and Validator to properly resolve and validate the DICOM-defined bindings without requiring changes on the DICOM side.

## Proposed Dispositions

### Disposition A: Accept — Persuasive (Add `fhir.dicom` dependency)

**Action**: Accept the ticket as **Persuasive**. Direct the FHIRcast IG to add the `fhir.dicom` package as a dependency in its IG configuration. This resolves the validator/publisher errors by providing locally resolvable DICOM ValueSet definitions without requiring any changes on the DICOM side.

**Rationale**: This is the consensus approach from the comment thread. DICOM WG-06 has explicitly declined to make changes on their end (e.g., switching from FTP to HTTP or providing resolvable ValueSet URIs). The `fhir.dicom` package already exists and is maintained by Grahame Grieve, regenerated with each DICOM release. Both john_moehrke and the reporter (bvdh) agree this is the correct fix. A branch already exists in the fhircast-docs repository with the implementation.

**Implementation**: Merge the work from the `FHIR-42817` branch in `HL7/fhircast-docs` after review, adding:
```yaml
dependencies:
  fhir.dicom:
    uri: http://fhir.org/packages/fhir.dicom
    version: 2024.1.20240120
```

### Disposition B: Accept — Persuasive with Modification (Add dependency + document guidance)

**Action**: Accept as **Persuasive with Modification**. In addition to adding the `fhir.dicom` dependency to FHIRcast, add a note in the FHIRcast IG (and/or the core spec's ImagingStudy page) documenting that any IG using DICOM-defined ValueSet bindings must include the `fhir.dicom` package dependency for proper validation. This ensures other IGs using DICOM codes don't encounter the same issue.

**Rationale**: The root problem affects all IGs that reference DICOM ValueSets, not just FHIRcast. While fixing FHIRcast is necessary, broader guidance prevents the same issue from recurring in other imaging-related IGs.

### Disposition C: Decline — Not Persuasive (Out of scope for FHIRcast)

**Action**: Decline as **Not Persuasive**. The issue originates in how DICOM publishes its ValueSets (HTML pages with FTP links), which is outside the control of the FHIRcast IG. The `fhir.dicom` package is a workaround, not a fix, and the proper resolution should come from DICOM updating their publication infrastructure.

**Rationale**: DICOM WG-06 has been notified. Adding the `fhir.dicom` dependency is a workaround that masks the underlying problem and creates a maintenance burden (the dependency version must be updated with each DICOM release). However, this disposition is weak since DICOM WG-06 has already stated they will not address the issue.

---

## Recommendation

**Disposition A: Accept — Persuasive** is recommended.

The consensus among commenters is clear: add the `fhir.dicom` package dependency. DICOM WG-06 has explicitly declined to make changes, so waiting for them is not viable. The `fhir.dicom` package is an established, maintained solution already used by other IGs and the core spec build process. The reporter has already created a branch with the implementation. This is a straightforward fix that eliminates validation warnings and aligns with the standard approach for using DICOM terminology in FHIR IGs.
