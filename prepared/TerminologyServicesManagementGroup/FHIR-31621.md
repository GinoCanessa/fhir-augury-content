## Problem Statement

The reporter (Alexander Henket) identifies a systemic governance problem: while HL7's OID Registry has been meticulously maintained and stable, URIs have been created in an ad hoc, decentralized fashion and changed over time. This results in:

1. **Multiple URIs for the same OID** — 17 OIDs in R4 and 54 OIDs in STU3 map to more than one URI, making it impossible to determine which is authoritative for a given FHIR version.
2. **Same OID assigned to different objects** — Some OIDs are shared between ValueSets and CodeSystems, or between V2 and V3 ValueSets (i.e., logically distinct objects).
3. **Inconsistent NamingSystem "preferred" markers** — Both an OID and a URI may be marked `preferred = true`, creating ambiguity about which identifier is authoritative for which context (V2, V3/CDA, FHIR R4, etc.).
4. **No centralized governance** — OID and URI administration is done per-working-group rather than centrally, leading to drift and conflicts.

The reporter argues that OID/URI governance must be centralized (potentially under TSMG) and that new code system URIs must also have OIDs assigned to maintain CDA compatibility. Switching the preferred identifier mid-FHIR-version is a breaking change and must be prohibited.

## Proposed Dispositions

### Disposition A: Persuasive — Establish Centralized OID/URI Governance

**Action:** Accept the ticket and formally establish TSMG as the governance body for centralized OID/URI identifier management. Create a policy document requiring:
1. All new code system URIs must have a corresponding OID assigned.
2. NamingSystem resources must designate a single preferred identifier per FHIR version (using a version-specific qualifier, not just a boolean `preferred`).
3. OID/URI changes are prohibited within a FHIR major version once published.
4. A systematic audit of existing OID/URI conflicts (starting with the 17 R4 and 54 STU3 cases documented in this ticket) with a remediation plan for the next THO release.

**Rationale:** The evidence is compelling — dozens of OIDs map to multiple URIs, creating interoperability risks. TSMG has been active since 2021 and is the natural home for this governance. The 2025-11-03 TSMG discussion confirms agreement with the problem. This addresses a real, long-standing issue that has already generated multiple related Jira tickets.

### Disposition B: Persuasive with Modification — Scope to THO NamingSystem Cleanup Only

**Action:** Accept the ticket but scope the resolution to improving NamingSystem resources in THO rather than establishing broad governance policy. Specifically:
1. Conduct an audit of all OIDs with multiple URIs and create/update NamingSystem resources to designate the correct preferred identifier for R5+.
2. Add guidance to the THO publishing process requiring NamingSystem review for all new or modified CodeSystem/ValueSet entries.
3. Defer the broader governance question (mandatory OID assignment for new URIs, cross-version identifier stability policy) to a future ticket or TSMG work item.

**Rationale:** The full governance proposal is ambitious and may stall without a clear owner and resources. The NamingSystem cleanup is a concrete, actionable deliverable that addresses the most visible symptoms. The broader policy work can proceed incrementally.

### Disposition C: Not Persuasive — Close with Commentary

**Action:** Close the ticket as not persuasive, acknowledging the problem but noting that:
1. THO/UTG NamingSystem resources are already tracking duplicate/changed identifiers (as noted by Ted Klein in 2021).
2. Significant cleanup work has occurred since the ticket was filed (multiple related tickets resolved).
3. The OID Registry covers more than terminology (as noted by Jessica Bota in 2025), so TSMG alone cannot own all OID governance.
4. The scope of the governance change requested exceeds what can be actioned through a single Jira ticket.

**Rationale:** While the analysis is valuable, the problem is already being addressed incrementally through THO maintenance. A governance-level change requires organizational action beyond the scope of a FHIR specification change request.

## Recommendation

**Disposition A: Persuasive — Establish Centralized OID/URI Governance**

This ticket documents a well-evidenced, systemic problem that has persisted for over four years and continues to generate new tickets (e.g., FHIR-50143 in 2025). The 2025-11-03 TSMG discussion shows the work group agrees with the problem statement. The incremental cleanup approach (Disposition B) has been the de facto strategy since 2021 and has not resolved the underlying governance gap. Establishing formal policy now — while TSMG is actively engaged — is the best path to preventing further identifier proliferation and ensuring V3/CDA compatibility during the ongoing FHIR transition. The reporter's detailed analysis of 17 R4 and 54 STU3 OID conflicts provides a ready-made starting point for systematic remediation.
