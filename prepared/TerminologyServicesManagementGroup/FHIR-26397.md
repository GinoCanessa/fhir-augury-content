## Issue Description

The reporter identifies that OID **2.16.840.1.113883.4.642.3.376** (listed in the summary section of the Medication Status value set) does not exist in the HL7 OID registry. Without a registry entry, there is no way to verify this is the correct OID for the value set.

**Request:** Add the OID to the HL7 OID Registry, or update the specification with the correct OID.

## Key Context for Workgroup Review

1. **Established precedent:** Two closely related sibling tickets (FHIR-26394, FHIR-26395) were resolved in January 2025 as **"Not Persuasive"** by cmacumber. The stated TSMG policy is:
   - There is **no expectation** that the HL7 OID registry is comprehensive of all known OIDs.
   - OIDs in HL7 specifications and IGs are known to not all be in the HL7 OID Registry.
   - Validation can be done via **published artifacts** (not the OID registry).
   - If contradicting publications exist, the OID present in a **normative standard** (e.g., FHIR R4) is considered authoritative.

2. **Earlier comment contradicts current policy position:** carmela_couderc's 2021 comment on this ticket stated "Vocab will register the OID," suggesting an intent to register. However, the 2025 resolutions of sibling tickets established that registration is not required. No subsequent registration appears to have occurred.

3. **Batch of ~15 identical tickets:** This is one of many nearly identical tickets filed by the same reporter. Consistent disposition across the batch is important for clarity.

## Proposed Dispositions

### Disposition A: Not Persuasive (Recommended)

**Resolution:** Not Persuasive — no specification change required.

**Rationale:** Apply the same TSMG policy established in the January 2025 resolution of sibling tickets FHIR-26394 and FHIR-26395. There is no current expectation that the HL7 OID registry must be comprehensive of all known OIDs. The OID 2.16.840.1.113883.4.642.3.376 for the Medication Status value set is present in the published FHIR specification and can be validated there. No specification change is needed. This disposition maintains consistency across the batch of similar tickets from the same reporter.

**Suggested comment:**
> At this time there is no expectation that the HL7 OID registry is comprehensive of all known OIDs. It is known that there are OIDs in HL7 specifications and IGs that are not in the HL7 OID Registry. Validation can be done via the published artifacts. Consistent with the resolution of FHIR-26394 and FHIR-26395.

---

### Disposition B: Persuasive with Modification — Register the OID

**Resolution:** Persuasive with Modification — register OID 2.16.840.1.113883.4.642.3.376 in the HL7 OID registry as a quality improvement, but no specification text change is required.

**Rationale:** While the current TSMG policy does not mandate registry completeness, registering this OID would fulfill the 2021 commitment from carmela_couderc and improve discoverability for implementers who use the OID registry for validation. This would be an administrative action, not a specification change. However, this would set a precedent requiring similar action across all sibling tickets, creating significant administrative burden.

**Suggested comment:**
> While there is no policy requirement, registering the OID improves discoverability. Vocab to register OID 2.16.840.1.113883.4.642.3.376 for the Medication Status value set. No spec text change needed.

---

### Disposition C: Duplicate — Refer to FHIR-26394

**Resolution:** Duplicate of FHIR-26394.

**Rationale:** This ticket raises the same fundamental question as FHIR-26394 (whether OIDs must be in the HL7 OID registry) and FHIR-26395, both of which have been resolved. The reporter on FHIR-26395 (same reporter) acknowledged that resolution of FHIR-26394 would also resolve that ticket. The same logic applies here. Close as duplicate and point to the established resolution.

**Suggested comment:**
> Closing as duplicate. The underlying policy question (whether FHIR value set OIDs must be registered in the HL7 OID Registry) was resolved in FHIR-26394 as Not Persuasive. That resolution applies equally to this ticket.

---

## Recommendation

**Disposition A: Not Persuasive** is recommended. This is the most consistent approach given the January 2025 TSMG policy resolution on the identical sibling tickets FHIR-26394 and FHIR-26395. It avoids creating administrative burden across the ~15 related open tickets and aligns with the stated policy that the OID registry need not be comprehensive. Disposition C (Duplicate) would also be reasonable but Not Persuasive more precisely reflects that the request was considered on its merits and found not to require action.
