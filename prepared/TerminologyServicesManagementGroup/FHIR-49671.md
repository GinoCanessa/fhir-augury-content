## Description Summary

Several code systems (especially older or communication-related ones) define ranges of "private codes" — codes whose meaning is not fixed by the code system authority, but left for implementers to assign locally. A key example is [DICOM 7.8 Private Data Elements](https://dicom.nema.org/dicom/2013/output/chtml/part05/sect_7.8.html). IEEE 11073 MDC is another prominent case where the upper 4K of each 64K partition is reserved for private/vendor codes.

The ticket requests that the FHIR specification provide guidance on:
1. Whether private codes can be expressed in FHIR
2. If so, how they should be represented

The reporter proposes:
- FHIR resources **can** use private codes from code systems that define them
- For **CodeableConcepts**: implementations SHOULD include both the original private code and a more descriptive code
- For **Codings**: implementations SHOULD add a more descriptive code alongside the private code
- An **extension on CodeSystem** should be defined to express that a range of codes shares the same definition (avoiding adding thousands of codes individually)

## Zulip Discussions

### 1. [#IG creation > supporting private codes in code system](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/supporting%20private%20codes%20in%20code%20system)
- **Dates:** 2024-11-18 to 2024-11-27 (50 messages)
- **Participants:** Erik Moll, Grahame Grieve, Michael Faughn, Michael Lawley, Elliot Silver, Lloyd McKenzie, David Pyke, John Moehrke, Richard Townley-O'Neill
- **Summary:** Triggered by IEEE 11073 MDC private codes causing validation errors in the PHD IG. Key discussion points:
  - **Regex extension on CodeSystem** proposed by Grahame Grieve: codes matching a regex would be treated as valid by validators without requiring display or property validation
  - **Lloyd McKenzie** emphasized that if the same code can mean different things from different vendors, they **cannot** be in the same code system (Coding.code + Coding.system must have unique stable meaning)
  - **Michael Faughn** argued each vendor's private codes should be in **separate code systems**, not the MDC code system, to preserve semantic integrity
  - **Elliot Silver** agreed private codes should not be part of MDC per Lloyd's rule, and MDC should represent only published codes
  - **Consensus direction:** Private codes where meaning varies by vendor belong in vendor-specific code systems; the regex extension approach is viable for code systems that truly define private spaces with consistent semantics

### 2. [#terminology > Use of private MDC codes in FHIR](https://chat.fhir.org/#narrow/stream/terminology/topic/Use%20of%20private%20MDC%20codes%20in%20FHIR)
- **Dates:** 2024-11-19 (4 messages)
- **Participants:** Paul Schluter, Grahame Grieve
- **Summary:** Paul Schluter (IEEE 11073 expert) cautioned against elevating private MDC codes to the same level as published terms. Key concerns: private codes haven't undergone open-consensus review; multiple vendors may reuse the same numeric codes with different meanings; encouraging private code proliferation could suppress innovation. He proposed a mandatory REFID naming convention (e.g., `MDCGE_ATTR_ABC`) to ensure vendor-unique identifiers if private codes proceed.

## Analysis

### Core Problem
FHIR currently provides no guidance on how to handle "private codes" — codes within a code system that are reserved for local/vendor-specific use rather than defined by the code system authority. This causes validation errors and ambiguity in implementations that use code systems with such reserved spaces (IEEE 11073 MDC, DICOM, ISO 3166-1).

### Key Tensions
1. **Code system integrity vs. practical use:** Lloyd McKenzie's principle that Coding.code + Coding.system must have unique stable meaning conflicts with private codes whose meaning varies by context/vendor
2. **Single code system vs. separate systems:** Community is split on whether private codes should stay in the parent code system (with a regex extension) or be moved to vendor/context-specific code systems
3. **Validator behavior:** Current tooling rejects private codes as unknown; any solution must define how validators should handle them

### Progress Made
FHIR-50851 (resolved, persuasive with modification) addresses the **CodeSystem extension** component of this ticket — defining a regex-based extension to mark private code spaces. FHIR-50922 (applied) extends this by allowing supplements to reference private codes to document their local meaning.

---

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Rationale:** Accept the request for guidance but narrow the scope, acknowledging that the specific CodeSystem extension mechanism has already been addressed by FHIR-50851. Add normative guidance to FHIR Core that:
1. Private codes from code systems that formally define them CAN be used in FHIR resources
2. When private codes have vendor/context-specific meanings, implementations SHOULD use a vendor-specific code system URI rather than the parent code system
3. When using private codes within the parent code system (where the code system authority permits this), the CodeSystem extension from FHIR-50851 SHOULD be used to define the private space
4. For CodeableConcepts, implementations SHOULD include a more descriptive coding alongside any private code
5. Reference the existing solutions in FHIR-50851 and FHIR-50922 for extension and supplement mechanisms

**Impact:** Provides comprehensive guidance while leveraging work already done. Clarifies the `system` value question raised by Elliot Silver.

### Disposition B: Persuasive
**Rationale:** Accept the ticket as-is. Add guidance and the proposed CodeSystem extension exactly as the reporter suggests, including defining an extension for expressing that a range of codes shares the same definition.

**Risk:** Largely duplicates the extension work already done in FHIR-50851. Does not address the critical point raised in Zulip discussions that private codes with different meanings per vendor/context cannot safely share a code system URI.

### Disposition C: Not Persuasive — Addressed by Existing Tickets
**Rationale:** The specific technical mechanisms requested have been substantially addressed by FHIR-50851 (CodeSystem extension for private code spaces, resolved) and FHIR-50922 (supplements for private codes, applied). The remaining need is general guidance text, which could be tracked under those tickets or as editorial work.

**Risk:** Leaves a gap — no unified guidance document for implementers on the overall approach to private codes. The existing tickets solve mechanics but not the overarching "what should I do?" question.

---

## Recommendation

**Disposition A: Persuasive with Modification** is recommended. The core request for FHIR specification guidance on private codes remains valid and unfulfilled — FHIR-50851 and FHIR-50922 solve the mechanical/extension issues but don't provide the broader guidance that implementers need. The modification narrows the extension proposal (already addressed) and adds the critical clarification from community discussions about when private codes can vs. cannot share a parent code system URI. This disposition best synthesizes the ticket's intent with the significant community discussion and progress that has occurred since filing.
