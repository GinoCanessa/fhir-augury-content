## Description

The reporter identifies that OID `2.16.840.1.113883.4.642.3.1321`, listed in the summary section of the Medication Request Intent value set, does not exist in the HL7 OID Registry. This means there is no way to verify it is the correct OID for the value set. The request is to either add the entry to the HL7 OID Registry or update the specification with the correct OID.

## Proposed Dispositions

### Disposition A: Not Persuasive

**Rationale:** The HL7 OID Registry is not expected to be comprehensive of all known OIDs. There is no approved TSMG or TSC policy requiring that FHIR value set OIDs be registered in the HL7 OID Registry. The published FHIR specification itself serves as the authoritative source of truth for OIDs assigned to its value sets. Validation of the OID can be done via the published artifacts (e.g., the FHIR specification page for the value set) without needing a separate registry entry. This aligns with the resolution of the closely related ticket FHIR-26394 (resolved Not Persuasive in January 2025) and the positions stated by Robert McClure and Ted Klein.

**Suggested comment:** *"There is no requirement that FHIR value set OIDs be listed in the HL7 OID Registry. The published FHIR specification is the authoritative source for OIDs assigned to its value sets. Consistent with the resolution of FHIR-26394 and TSMG's established position, the OID registry is not expected to be comprehensive of all known OIDs."*

### Disposition B: Persuasive with Modification

**Rationale:** While there is no formal policy requiring OID registration, registering the OID would improve discoverability and cross-specification verification. The OID `2.16.840.1.113883.4.642.3.1321` should be registered in the HL7 OID Registry as a courtesy action, consistent with carmela_couderc's earlier commitment to register it. However, this should not be treated as a specification defect — the modification is to register the OID administratively rather than change the specification itself. The value set should also be evaluated for migration to THO.

**Suggested comment:** *"The OID 2.16.840.1.113883.4.642.3.1321 will be registered in the HL7 OID Registry as a courtesy to improve discoverability. Note that this is not a specification defect — there is no requirement for FHIR OIDs to be in the registry. The value set's status regarding THO migration should be evaluated separately."*

### Disposition C: Retracted

**Rationale:** The request is based on the incorrect premise that OIDs must be listed in the HL7 OID Registry to be valid. As established by TSMG policy and precedent (FHIR-26413, FHIR-26416), the FHIR specification publication is the source of truth. The concern raised has been addressed through policy clarification across the related batch of tickets, and no specification change is needed.

**Suggested comment:** *"This ticket is based on the premise that OIDs must appear in the HL7 OID Registry to be valid. TSMG has clarified that there is no such requirement; the published FHIR specification is authoritative. Similar tickets (FHIR-26413, FHIR-26416) were retracted. Retracting per established precedent."*

## Recommendation

**Disposition A: Not Persuasive**

This is the most appropriate disposition because it directly aligns with TSMG's most recent and clearly articulated position from the resolution of FHIR-26394 (January 2025). Unlike "Retracted" (Disposition C), which implies the submitter should withdraw the request, "Not Persuasive" acknowledges the concern was legitimate but clarifies that the current policy does not require action. Unlike "Persuasive with Modification" (Disposition B), it avoids creating an expectation of bulk OID registration that TSMG has not committed to as a systematic practice. The batch of similar tickets from John Snyder has been progressively resolved using this approach, establishing a clear and consistent precedent.
