## Description Summary

The reporter identified that the summary section for HL7 Version 2 Table 0007 (section 4.3.15.7) in FHIR Core displays "??" for the OID value. The suggested solution is to populate the OID field using the OID registered by Ted Kline in the HL7 OID Registry (`2.16.840.1.113883.12.7`).

## Keywords

V2 Table 0007, OID, HL7 OID Registry, value set, summary rendering, THO, UTG sync, v2tables.xml

## Comment Analysis (2 comments)

1. **jsnell (2021-02-19):** Confirmed that terminology.hl7.org (THO) already has the correct data. The FHIR Core page is out of sync with UTG, and the sync issue will be addressed in a separate ticket.

2. **tedhl7geek (2021-12-20):** Stated the suggested solution is **wrong**. The OIDs under branch 12 in the OID registry are for PDF text printed objects of V2 tables — tables do not exist as individual FHIR resources. The correct TABLE OIDs are in THO's `v2tables.xml` CodeSystem. If THO content is incorrect, a UTG ticket should be filed instead. Also noted that some V2 tables were imported misleadingly as FHIR CodeSystem resources in R4.

## Analysis

This ticket is part of a batch of OID-related issues filed by John Snyder in February 2020, all reporting missing OID values in V2 table summaries within FHIR Core. Both commenters agree that the underlying terminology data in THO is correct. The core issue is a **rendering/sync problem** between FHIR Core and UTG/THO, not a missing OID in the registry.

Notably, the closely related broader ticket FHIR-17625 ("Fix OID rendering for V2 tables") was resolved as **Not Persuasive**, suggesting the community has already declined to fix this class of rendering issue in FHIR Core.

The reporter's suggested solution (using OIDs from the `2.16.840.1.113883.12.*` branch) was explicitly called out as incorrect by tedhl7geek, since those OIDs refer to PDF text printed objects, not FHIR resources.

## Proposed Dispositions

### Disposition A: Not Persuasive with Modification
**Rationale:** The OID data is already correct in THO. The suggested solution (using branch 12 OIDs) is incorrect as confirmed by subject matter experts. The broader issue of V2 table OID rendering was already resolved as Not Persuasive in FHIR-17625. Close this ticket, noting that THO is the authoritative source and FHIR Core V2 table rendering is a known limitation that does not warrant a change. Recommend the reporter file a UTG ticket if any specific THO content is incorrect.

### Disposition B: Persuasive with Modification
**Rationale:** While the suggested OID source is wrong, the underlying display issue ("??" shown for OID) is a legitimate usability concern. Accept the ticket with modification: instead of using the branch 12 OIDs, update the FHIR Core build to pull the correct OIDs from THO's `v2tables.xml` to populate the V2 table summary sections, resolving the rendering gap.

### Disposition C: Defer to UTG
**Rationale:** Since the authoritative terminology content lives in THO/UTG, and both commenters indicate this is a sync issue between FHIR Core and UTG, defer this ticket entirely to the UTG project. The FHIR Core spec should not independently manage V2 table OIDs; instead, the build pipeline should be updated to consume THO data correctly. Close this FHIR Jira ticket and recommend a corresponding UTG ticket if one does not already exist.

## Recommendation

**Disposition A: Not Persuasive with Modification.** The data is correct in THO, the suggested solution is wrong, and the broader rendering issue (FHIR-17625) was already resolved as Not Persuasive. This ticket does not present new information that would change that prior decision. The workgroup should close this ticket, directing the reporter to UTG if any specific THO content needs correction.
