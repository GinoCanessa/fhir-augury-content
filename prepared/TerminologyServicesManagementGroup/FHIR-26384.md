## Problem Statement

The reporter notes that in the FHIR R4 spec, the summary page for HL7 Version 2 Table 0004 (section 4.3.15.4) displays "??" in the OID field for the value set. The reporter requests that this OID be populated from the HL7 OID Registry, referencing an OID registered by Ted Kline (`2.16.840.1.113883.12.4`).

1. **jsnell (2021-02-19):** States that everything is updated correctly in terminology.hl7.org (THO) and that the FHIR spec page is simply out of sync with UTG. Notes a separate ticket will address the sync issue.

## Analysis

This ticket is part of a batch of OID-related tickets filed by John Snyder in February 2020. The core observation — that the FHIR R4 spec page shows "??" for the OID — is valid. However, both commenters confirm:

1. **The OIDs are correct in THO/UTG** — this is not a data gap.
2. **The suggested fix (using OID registry branch 12) is incorrect** — those OIDs are for printed V2 table PDF objects, not FHIR value sets.
3. **The real issue is a sync problem** between the FHIR spec pages and UTG/THO content.

Since the underlying data in THO is correct and V2 table content in FHIR R4 was an import artifact, this ticket does not require a content change. The display issue has likely been superseded by the UTG/THO migration and subsequent FHIR build improvements.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Redirect to UTG Sync
- **Action:** Acknowledge the display gap was real but note the fix belongs in the FHIR build/sync pipeline, not in OID registration. Close this ticket and reference any existing UTG sync ticket. If no sync ticket exists, create one.
- **Rationale:** The reporter identified a real problem (missing display data), but the root cause is a FHIR spec ↔ THO sync issue, not a missing OID. Closing here with a redirect respects the reporter's observation while routing the fix correctly.

### Disposition B: Not Persuasive — Already Resolved in THO
- **Action:** Close as "Not Persuasive" noting that the OID data is present in THO and the FHIR R4 display issue is a historical artifact of the V2 table import. The suggested solution was incorrect per tedhl7geek's analysis.
- **Rationale:** THO is now the authoritative source. The FHIR R4 pages for V2 tables are legacy rendered content. No action is needed on the FHIR Core spec.

### Disposition C: Not Persuasive — Retract
- **Action:** Request retraction similar to FHIR-26418. The reporter's premise (OID is missing) is incorrect — it exists in THO. The suggested OID source (branch 12) was wrong per expert comment.
- **Rationale:** The ticket is based on a misunderstanding of V2 table OID structure. The sibling ticket FHIR-26418 was retracted for similar reasons.

## Recommendation

**Disposition B: Not Persuasive — Already Resolved in THO.**

The OID data for V2 Table 0004 is confirmed present and correct in THO's `v2tables.xml`. The FHIR R4 display showing "??" was a sync/import artifact, not a missing OID. The suggested solution (using OID Registry branch 12) was explicitly identified as incorrect by Ted Klein (tedhl7geek). Since THO is the authoritative terminology source and no FHIR Core spec change is needed, "Not Persuasive" is the appropriate disposition. This aligns with the resolution pattern of the related ticket FHIR-26418 (retracted) and the commentary consensus.
