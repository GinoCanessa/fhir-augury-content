## Details

This ticket asks about the classification of HL7 v3 OIDs when used as ValueSet identifiers in FHIR. Specifically, the submitter is requesting clarification on which type of `ValueSet` identifier an HL7 v3 OID should be categorized as.

The question touches on a known area of ambiguity in the FHIR specification regarding how OIDs are represented within the `ValueSet.identifier` element. Per current spec guidance, when an OID is used as a "naturally globally unique URI," it should be placed in `ValueSet.identifier` with `system` = `"urn:ietf:rfc:3986"` and `value` = `"urn:oid:..."`. The `ValueSet.identifier` element is defined as a business identifier (type `Identifier`, cardinality `0..*`) used to identify the value set when represented in other formats or referenced in non-FHIR contexts.

The sole comment on the ticket, from tedhl7geek (2021-12-20), states: *"Do not understand the question."* — indicating the ticket's wording may be insufficiently clear to act upon.

The broader community has discussed related topics extensively, including when `Identifier.type` should be used to distinguish OIDs serving different purposes (e.g., DICOM UIDs vs. general OIDs), and whether `ValueSet.identifier` definitions align with the Value Set Definition Profile (VSDP) requirements.

## Proposed Dispositions

### Disposition A: Persuasive — Clarify OID guidance in ValueSet.identifier documentation

**Rationale:** The question, while imprecisely worded, points to a real gap in the specification. Accept the ticket and add clarifying text to the `ValueSet.identifier` documentation explaining that HL7 v3 OIDs for value sets are business identifiers carried in `ValueSet.identifier` with `system` = `"urn:ietf:rfc:3986"` and `value` = `"urn:oid:<OID>"`. Reference the existing guidance on "naturally globally unique" identifiers. This aligns with the resolution of FHIR-26549 and the broader community discussion on OID identifier patterns.

**Proposed change:** Add a clarifying sentence or example to the `ValueSet.identifier` comments showing that an HL7 v3 OID is represented as a business identifier using the standard OID-as-URI encoding pattern.

### Disposition B: Not Persuasive — Already addressed by existing specification text and FHIR-26549

**Rationale:** The existing specification already describes how OIDs are used in `ValueSet.identifier`. The comments on the `ValueSet.identifier` element state: *"Typically, this is used for identifiers that can go in an HL7 V3 II (instance identifier) data type, and can then identify this value set outside of FHIR."* Combined with the Identifier datatype guidance on OID encoding and the improvements made via FHIR-26549, the spec already answers the question. The ticket does not identify a specific deficiency or propose a concrete change.

### Disposition C: Not Persuasive with Modification — Request clarification from submitter

**Rationale:** The ticket question is ambiguous, as noted by the only commenter ("Do not understand the question"). The submitter may be asking about (a) which `Identifier.type` code to use, (b) whether the OID goes in `ValueSet.identifier` vs. `ValueSet.url`, or (c) how to encode the OID value. Without a clear problem statement or proposed change, the work group cannot act. Close as Not Persuasive but note that if the submitter can clarify the specific issue, a new ticket with a concrete change request would be welcome.

## Recommendation

**Disposition C: Not Persuasive with Modification** — The ticket's intent is unclear, as confirmed by the sole commenter. The existing specification text (especially after FHIR-26549 improvements) and community guidance in Zulip already address the common questions about OIDs in `ValueSet.identifier`. Without a clarified problem statement or proposed wording change from the submitter, there is insufficient basis to make a specification change. Closing as Not Persuasive with an invitation to re-submit with specifics is the most appropriate path.
