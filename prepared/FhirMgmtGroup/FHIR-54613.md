## Summary

This change request asks the FHIR Management Group to review the use of Required and Extensible bindings on elements in base FHIR resources, with the goal of identifying cases where such bindings create conflicts for legitimate profiling use cases. The ticket was originally submitted by Gay Dolin on behalf of US Core, IPS, and eCR profiling efforts, which have encountered situations where strong bindings in the base specification prevent profiles from using jurisdiction-appropriate value sets. Lloyd McKenzie moved the ticket to FMG as a governance question and requested that specific per-resource tracker tickets be filed. Eric Haas subsequently linked several related tickets and uploaded a spreadsheet cataloging the problematic bindings. The discussion also raised concerns about Required and Extensible bindings on non-code elements. A comment from John Moehrke highlights a counter-concern: some Required bindings (e.g., Provenance activity codes) enforce state-machine semantics from W3C PROV, and loosening them could break established behavioral contracts.

## Details

**Description:**
The ticket requests a systematic review of Required and Extensible bindings in base FHIR resources. The submitters argue that certain bindings in the base specification are overly restrictive for international or jurisdiction-specific profiling. Profiles that need to use alternative code systems or value sets are unable to do so when the base resource imposes Required or Extensible binding strength, leading to conformance conflicts. The ticket does not request a specific change to a single element; rather, it raises a governance-level concern and asks FMG to establish policy or guidance on when Required and Extensible bindings are appropriate in the base specification.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Direct all owning work groups to review Required and Extensible bindings on their base resources and loosen binding strength where there is no compelling semantic or safety rationale. Publish FMG guidance stating that Required bindings should only be used when code values enforce behavioral semantics (e.g., state machines) or safety constraints, and that Extensible bindings should include an explicit rationale. Request that QA tooling flag Required and Extensible bindings on non-code elements as warnings.
#### Justification
The submitters have demonstrated concrete cases where strong bindings in the base specification create real barriers for international and jurisdictional profiling. Loosening bindings where appropriate would make the base specification more globally usable without sacrificing interoperability where it genuinely matters. This aligns with Lloyd McKenzie's stated principle that Required and Extensible bindings should not appear on non-code elements, and individual tracker tickets have already been filed for specific instances.

---

### Disposition B: Alternative Approach
#### Proposal
Rather than directing a broad review of all bindings, issue FMG governance guidance on binding strength policy. The guidance would establish criteria for when Required and Extensible bindings are appropriate in base resources (e.g., state-machine enforcement, safety-critical coded values) and when they should be avoided (e.g., non-code elements, jurisdiction-specific code systems). Leave individual binding changes to be addressed through the already-filed per-resource tracker tickets, and add a QA rule to flag new Required or Extensible bindings that do not include a documented rationale.
#### Justification
This approach addresses the root governance concern without requiring a sweeping review that may disrupt existing work group processes. It provides clear, forward-looking policy while respecting John Moehrke's concern that some Required bindings (such as Provenance activity codes) serve important behavioral purposes. The per-resource tickets already filed provide the mechanism for addressing existing problematic bindings on a case-by-case basis.

---

### Disposition C: Decline
#### Proposal
Decline the request on the grounds that the ticket does not propose a specific actionable change, and the appropriate mechanism for addressing individual binding concerns is through per-resource tracker tickets, which have already been filed.
#### Justification
As Lloyd McKenzie noted, this ticket does not request a specific change and cannot be assigned to a single work group. The auto-filed tickets for Required bindings without escape valves and the manually filed tracker tickets already provide the vehicles for making concrete changes. Declining this governance-level request avoids setting a precedent for broad directives that may have unintended consequences, such as weakening state-machine bindings in Provenance.

---

### Recommendation
**Recommended disposition:** B — Alternative Approach

Disposition B strikes the best balance between addressing the legitimate concern and maintaining stability. The submitters have raised a real governance gap: there is no published FMG policy on when Required and Extensible bindings are appropriate in base resources, and the lack of such guidance has led to inconsistent binding decisions across work groups. Publishing clear criteria would prevent future problematic bindings and give profilers a basis for requesting changes. At the same time, this approach avoids the risks of a broad retroactive review, which could inadvertently weaken bindings that serve important behavioral contracts (as John Moehrke's Provenance example illustrates). The individual per-resource tracker tickets already filed provide the right granularity for evaluating existing bindings case by case, and a new QA rule would catch future violations of the policy. This approach is actionable, proportionate, and responsive to all stakeholders' concerns.
