## Description

This is a TSMG-initiated change request asking the Patient Empowerment work group to review the code systems defined in the Personal Health Record (PHR) IG against the TSC-approved [Terminology Guidance for IG Authors](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers) policy. The ticket asks the work group to either:

1. **Anchor/add** the code systems in [terminology.hl7.org (THO)](http://terminology.hl7.org/), or
2. **Request an exemption** from TSMG if the code systems are tightly bound to the IG, via the TSMG listserv, a TSMG meeting, or the Zulip thread.

The reporter (Jessica Bota) notes that the code systems in this IG may be candidates for an exemption given that they appear tightly bound to the IG.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Request TSMG Exemption

**Action:** Accept the ticket as persuasive and contact TSMG to request an exemption for the PHR IG's code systems. The modification is that the code systems remain in the IG rather than being moved to THO, with a documented exemption.

**Rationale:** The PHR IG code systems are strongly tied to Apple HealthKit concepts (workout types, sample types, biological characteristics, correlation types, etc.). These are proprietary concepts originating from Apple's framework that are specific to the IG's use case of mapping HealthKit data to FHIR. They are not broadly reusable terminology and are tightly bound to the IG. The ticket reporter herself suggests this may be the case. The TSC policy explicitly contemplates this scenario and provides the exemption path. This follows the pattern established by FHIR-44083 (Persuasive with Modification).

**Precedent:** FHIR-44083 (SMART on FHIR) resolved similarly — acknowledged the request and engaged TSMG.

### Disposition B: Persuasive with Modification — Migrate Code Systems to THO

**Action:** Accept the ticket and migrate the Apple HealthKit–related code systems from the PHR IG to THO.

**Rationale:** Even though the code systems reflect Apple-specific concepts, housing them in THO would make them centrally discoverable and governable. Other IGs that may need to reference HealthKit data types could reuse these code systems. Moving to THO ensures consistent lifecycle management and avoids fragmented terminology ownership.

**Risk:** This adds maintenance burden — the code systems would need to track changes in Apple's HealthKit API. THO governance may not be well-suited for code systems that mirror a vendor-specific external framework. The work group would lose direct control over updates.

### Disposition C: Not Persuasive with Modification — Code Systems Appropriately Placed

**Action:** Reject the request, arguing the code systems are correctly defined within the IG and do not need to be in THO or require a formal exemption.

**Rationale:** The code systems are inherently IG-specific since they model Apple HealthKit concepts for the PHR use case. They are not general-purpose terminology. The IG is the appropriate home for these definitions, and requiring an exemption for obviously IG-specific, vendor-derived code systems creates unnecessary process overhead.

**Risk:** This disposition does not comply with the TSC-approved terminology policy. Even if the outcome is the same (code systems stay in the IG), failing to formally request the exemption through TSMG may create compliance issues during IG balloting or publication.

---

## Recommendation

**Disposition A: Persuasive with Modification — Request TSMG Exemption** is recommended.

The PHR IG's code systems are clearly derived from Apple HealthKit — a proprietary, vendor-specific framework — making them tightly bound to the IG. The ticket reporter explicitly suggests this may be the case. The TSC policy provides the exemption path precisely for this scenario, and following it ensures policy compliance while keeping the code systems where they belong. The work group should contact TSMG (via listserv, meeting, or Zulip) to formally request the exemption. This approach aligns with the precedent set by FHIR-44083 and satisfies the TSMG review requirement with minimal disruption to the IG.
