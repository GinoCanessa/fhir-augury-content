The reporter requests clarification of the MUST requirement in Section 2.1 of the Personal Health Record IG regarding FHIR data schema usage. Three specific concerns are raised:

1. **Ambiguous scope** — It is unclear whether the MUST requirement applies to long-term storage, over-the-wire interfaces, or the creation/consumption of `.sphr` files, all of which are discussed in the same section.
2. **Conflicting conformance language** — The sentence immediately preceding the MUST statement uses "recommends," which is at odds with a mandatory requirement (RFC 2119 semantics).
3. **Inconsistency with IG scope** — The Getting Started page states the IG does not prescribe how to program software, which arguably includes storage implementation, yet the MUST requirement appears to impose a storage constraint.

## Proposed Dispositions

### Disposition A: Persuasive (Clarify scope of FHIR schema requirement)

Accept the feedback and revise Section 2.1 to explicitly state that the MUST requirement for FHIR data schemas applies specifically to data packaged in the `.sphr` exchange format — not to internal/long-term storage implementations. Resolve the conflicting "recommends" vs. "MUST" language by aligning the introductory sentence with the conformance statement (e.g., change "recommends" to "requires" for the MUST bullet, or downgrade the bullet to SHOULD). Add a clarifying note that the IG does not prescribe internal storage formats.

**Pros:** Directly addresses all three concerns. Aligns with the existing WG sentiment (awatson1978's comment) and with the direction of related resolved tickets (FHIR-49244, FHIR-50629). Concrete, actionable fix.
**Cons:** Requires careful rewording to avoid inadvertently weakening the interoperability conformance requirement.

### Disposition B: Persuasive with Modification (Narrow the MUST to exchange context only)

Accept the feedback but apply a narrower fix: retain the MUST language but explicitly scope it to the `.sphr` file exchange context only. Add a parenthetical or separate sentence clarifying that this requirement does not apply to how systems store data internally. Do not change the "recommends" framing in the introductory sentence (treat it as a general recommendation with a specific mandatory exception for conformance claims).

**Pros:** Preserves the strong conformance requirement for exchange interoperability. Minimal text change.
**Cons:** Leaves the "recommends" → "MUST" tension partially unresolved, which may continue to confuse readers. Does not address the broader scope clarity concerns raised by the reporter.

### Disposition C: Not Persuasive (Current language is sufficient)

Reject the request on the grounds that the Section 2.1 language is sufficiently clear in context: the MUST applies to systems that "claim to be compliant with this IG," which inherently scopes it to the exchange/packaging format. The introductory "recommends" is a general framing, while the MUST is a specific conformance gate — these are not in conflict.

**Pros:** No text changes required; avoids scope creep in revisions.
**Cons:** Does not address the documented confusion from multiple reviewers (this ticket and FHIR-49071). Contradicts the WG's own preliminary assessment ("leaning towards persuasive"). Ignores the pattern of related tickets all requesting better clarity.

## Recommendation

**Disposition A: Persuasive** — The WG should accept this feedback and clarify that the FHIR data schema MUST requirement in Section 2.1 applies to data packaged for exchange in the `.sphr` format, not to internal storage. The conflicting conformance language ("recommends" followed by "MUST") should be harmonized. This aligns with the WG's preliminary assessment, with the direction set by resolved companion tickets (FHIR-49244 and FHIR-50629 both resolved as Persuasive), and with the concrete rewording already proposed in FHIR-50737. The WG should consider resolving FHIR-49070 and FHIR-49071 together given their overlapping scope.
