## Summary

The reporter states that the Security page in the Personal Health Record (FHIR) IG is incomplete and must be updated. The ticket notes that the list of sections on the Security page appears to be derived from the PHR-S Functional Model, and if so, that source should be clearly called out.

## Details

The PHR IG's Security page currently contains section headings but lacks substantive content, marked only as "Work in Progress." The reporter's concern has two parts:

1. **Incomplete content**: The Security page exists but is not filled out, making the specification feel unfinished.
2. **Unclear provenance**: The section headings appear to be derived from the HL7 PHR-S Functional Model, but this is not explicitly stated. If the sections are indeed sourced from the Functional Model, this should be clearly attributed.

This ticket is closely related to **FHIR-50750** ("Security section is a shell – recommend removing until ready and needed," filed by Virginia Lorenzi, Priority: Highest), which goes further and recommends removing the Security page entirely until it has substantive content. FHIR-50750 argues that having an empty security page makes the spec feel like a draft and that security is often out of scope for an IG of this type.

A broader pattern exists across the PHR IG ballot where reviewers have flagged incomplete or shell-like pages derived from the Functional Model. **FHIR-50739** (Priority: Highest) raises a similar concern about the Record Life Cycle section, recommending removal and noting that maintenance burden to keep Functional Model alignments current is not justified without interoperability guidance.

## Proposed Dispositions

### Disposition A: Accept (Persuasive – complete the Security page)

Accept the ticket and commit to completing the Security page content. This would involve:
- Populating each section on the Security page with substantive guidance relevant to PHR FHIR interoperability.
- Explicitly attributing section headings to the PHR-S Functional Model where applicable.
- Referencing existing security/privacy resources (e.g., FHIR core security guidance, IHE ITI.PCF) to provide implementers with actionable content.

**Pros**: Addresses the reporter's concern directly; a completed security section adds value for implementers.
**Cons**: Significant authoring effort required; may delay publication. FHIR-50750 (Priority: Highest) argues that security may be out of scope for this IG.

### Disposition B: Persuasive with Modification – remove Security page (align with FHIR-50750)

Accept as persuasive with modification: rather than completing the page, remove the Security page from this version of the IG. Add a brief note on the Functional Model page referencing the PHR-S FM security functions and linking to existing FHIR security guidance. This consolidates FHIR-49072 with FHIR-50750.

**Pros**: Eliminates the incomplete/draft impression; reduces maintenance burden; consistent with FHIR-50739 disposition for the Record Life Cycle section; aligned with the higher-priority FHIR-50750.
**Cons**: Loses a placeholder for future security content; implementers lose a direct entry point for PHR security considerations.

### Disposition C: Decline (Not Persuasive)

Decline the change request, keeping the Security page as-is with its "Work in Progress" designation. The section headings serve as a roadmap for future work and signal to implementers that security is a recognized concern.

**Pros**: No effort required; preserves the intent to address security in a future ballot cycle.
**Cons**: Leaves the spec looking incomplete, undermining implementer confidence. Does not address the attribution concern regarding the Functional Model. Contradicts the higher-priority FHIR-50750 feedback.

## Recommendation

**Disposition B: Persuasive with Modification – remove Security page and consolidate with FHIR-50750.**

The strongest path forward is to remove the incomplete Security page for this version, consistent with the closely related and higher-priority FHIR-50750. Multiple ballot reviewers have flagged shell-like Functional Model sections as detrimental to the IG's readiness impression (see also FHIR-50739). Removing the page avoids projecting an unfinished state, while a brief reference to FHIR core security guidance and the PHR-S Functional Model on another page preserves the intent. Security content can be developed and added in a future ballot cycle when substantive PHR-specific guidance is ready. This approach should be applied in coordination with FHIR-50750 to avoid duplicate dispositions.
