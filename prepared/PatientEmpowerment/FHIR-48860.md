## Problem Statement

The submitter requests clearer guidance and examples for how to represent Healthcare Agent (HCA) appointments using FHIR Consent resources, specifically:

1. How to explicitly reference the "usual powers" granted to an HCA.
2. How to represent an HCA appointment that includes usual powers **plus** additional powers (e.g., the power to share mental health information).
3. How to express an HCA appointment with only the usual powers and nothing additional.

The current IG lacks concrete examples that demonstrate these common real-world scenarios, making it unclear how implementers should construct Consent resources for HCA appointments.

## Keywords

## Resolution Description

The resolution accepts the request with modifications. The agreed changes include:

1. **A dedicated narrative page** titled "Healthcare Agents" containing a section on "Healthcare Agent (HCA) Appointments" explaining that HCA appointments are consents, with links to examples for each scenario.
2. **Four FHIR Consent examples** covering distinct scenarios:
   - **Example 1:** No healthcare agent exists (demonstrating use of "data absent reason").
   - **Example 2:** Permit of "usual powers" granted to an HCA without additional actions (because usual powers are already sufficient).
   - **Example 3:** "Usual powers" granted to an HCA **plus** additional granted powers (example containing multiple actions).
   - **Example 4:** Like Example 2 but also includes deny actions (requiring multiple example resources).

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification (Affirm the Resolution)

**Rationale:** The approved resolution is comprehensive and well-structured. Adding a dedicated "Healthcare Agents" narrative page with four distinct Consent examples covering the key HCA appointment scenarios (no agent, usual powers only, usual powers plus additional powers, and permit with deny actions) directly addresses the submitter's request. The companion tickets FHIR-48856 (LOINC grounding) and FHIR-48900 (ADIPermitConsent/ADIDenyConsent profiles) have already been applied, providing the profile infrastructure these examples need. This resolution aligns with the broader ADI IG direction of clear, implementer-friendly guidance.

**Action:** Affirm the "Persuasive with Modification" resolution. Ensure the four examples are created conforming to the ADIPermitConsent and ADIDenyConsent profiles established by FHIR-48900, coded with LOINC 81377-4 per FHIR-48856, and linked from the new "Healthcare Agents" narrative page.

### Disposition B: Accept as Persuasive (Simplified Scope)

**Rationale:** The core ask is for examples showing "usual powers" and "usual powers plus additional powers." The resolution could be simplified to focus on just these two primary examples without requiring a full new narrative page and all four scenarios. This would reduce implementation effort while still addressing the submitter's specific confusion.

**Action:** Accept as Persuasive. Provide at minimum two examples — one for usual-powers-only HCA appointment and one for usual powers plus additional powers (e.g., sharing mental health information) — as standalone Consent examples without requiring a dedicated narrative page.

### Disposition C: Not Persuasive (Defer Until Profile Work Stabilizes)

**Rationale:** The ADI Consent profile landscape is actively evolving. FHIR-49014 (requesting combined permit/deny in a single Consent) was resolved as Not Persuasive, but the separate-profile approach from FHIR-48900 is still being applied. Adding examples now may require rework if the Consent modeling approach changes. It may be premature to lock in examples before the profile design fully stabilizes.

**Action:** Reject as Not Persuasive. Defer example creation until the ADIPermitConsent and ADIDenyConsent profiles are fully stable and tested, then create examples in a follow-up ticket.

## Recommendation

**Disposition A: Accept as Persuasive with Modification (Affirm the Resolution)**

The resolution is well-considered and directly addresses the implementer confusion raised in the ticket. The companion profile work (FHIR-48856, FHIR-48900) has already been applied, so the infrastructure for these examples is in place. The four-example approach with a narrative page provides comprehensive, implementer-friendly guidance that covers the full range of HCA appointment scenarios. This is a documentation and examples enhancement with clear value and low risk to the specification.
