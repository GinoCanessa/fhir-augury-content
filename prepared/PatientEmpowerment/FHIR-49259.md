## Description

The submitter asks whether FHIR resources contained within an `.sphr` file are required to conform to the PHR profiles defined in the IG. If so, the submitter states this should be explicitly stated in the IG — particularly on the [recordkeeping page](http://hl7.org/fhir/uv/phr/2025Jan/recordkeeping.html).

## Analysis

This ticket addresses a clear documentation gap in the PHR IG regarding whether FHIR resources inside `.sphr` files must conform to the IG's PHR profiles. The ballot comment originated from the January 2025 ballot cycle, and several other ballot comments (FHIR-49269, FHIR-49266, FHIR-49489) raise the same or closely overlapping concern — the `.sphr` format was insufficiently documented.

The WG has already:
- Resolved this as **Persuasive** with agreement to add documentation.
- Proposed new text for **Section 2.1.2 (File Extensions - .sphr)** clarifying that `.sphr` is a zip-format folder containing at least one `.phr` file plus supporting materials.
- Voted to add to Block Vote 1 (3-0-0).

However, the proposed text does not explicitly state whether resources within the `.sphr` file **must** conform to the PHR profiles — it states the `.sphr` contains FHIR resources and supporting materials, but the conformance requirement is addressed only in the earlier comment ("Generally, yes"). The IG should make this explicit.

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Apply Proposed Text with Conformance Statement
**Action:** Accept the WG's proposed Section 2.1.2 text and additionally add an explicit statement that FHIR resources within the `.sphr` file SHOULD conform to the applicable PHR profiles defined in the IG. Add this to the recordkeeping page as the submitter suggested.

**Rationale:** This directly addresses the submitter's question and closes the documentation gap. The "SHOULD" (rather than "SHALL") allows flexibility for resources not covered by PHR profiles while setting a clear expectation. Aligns with the WG comment that "generally, yes" resources should conform.

### Disposition B: Accept as Persuasive — Apply Proposed Text Only (No Additional Conformance Requirement)
**Action:** Accept the proposed Section 2.1.2 text as-is. Rely on the existing IG conformance language (which requires FHIR data schemas) without adding a new profile conformance statement specifically for `.sphr` contents.

**Rationale:** The proposed text already clarifies the file format. Adding a profile conformance requirement might create unintended constraints, especially since `.sphr` files may contain resources from external sources that don't have corresponding PHR profiles. The general requirement to "use FHIR data schemas" may be sufficient.

### Disposition C: Accept as Persuasive with Modification — Require SHALL Conformance
**Action:** Accept the proposed text and add a **SHALL** conformance requirement: all FHIR resources within `.sphr` files must conform to the applicable PHR profiles. Add explicit language on the recordkeeping page and in the conformance section of the IG.

**Rationale:** A stronger conformance requirement ensures interoperability and consistent expectations for PHR consumers. If the purpose of the IG is to define the PHR exchange format, then the profiles should be mandatory for resources within that format.

## Recommendation

**Disposition A: Accept as Persuasive — Apply Proposed Text with Conformance Statement**

This is the recommended disposition because it directly answers the submitter's question with an explicit conformance expectation (SHOULD) while maintaining the flexibility acknowledged by the WG. The proposed Section 2.1.2 text provides the needed .sphr documentation, and adding a SHOULD-level conformance statement to the recordkeeping page closes the gap without being overly restrictive. This approach is consistent with the cluster of related tickets (FHIR-49269, FHIR-49266, FHIR-49489) that all point to the same documentation need, and it aligns with the WG's own comment that resources "generally" should conform to the PHR profiles.
