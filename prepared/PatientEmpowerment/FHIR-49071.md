## Description

The reporter raises three issues with the Data Model page of the PHR IG:

1. **Assumption of FHIR format storage**: The Data Model page opens with an assumption that data is contained in FHIR format, implying long-term storage as FHIR records. The reporter argues this should be clarified to indicate that data held by a PHR will be FHIR *mapped/compatible* rather than necessarily stored natively in FHIR. The scope of a PHR may extend beyond current FHIR scope, and this assumption conflicts with the Getting Started page statement that the IG doesn't cover how to program software.

2. **Naming inconsistency**: Section 4 references "Standard Patient Health Record" while the IG is titled "Standard Personal Health Record."

3. **Argonaut R4 column**: It is unclear what the "Argonaut R4" column in the Data Model is referencing. The reporter requests clarification or removal.

## Proposed Dispositions

### Disposition A: Persuasive

**Rationale:** Accept all three points as stated. (1) Revise the Data Model page to remove the assumption that data is *stored* in FHIR format, instead clarifying that PHR data should be FHIR-mapped/compatible for interoperability. (2) Correct "Standard Patient Health Record" to "Standard Personal Health Record." (3) Either add a clear explanation of the Argonaut R4 column's purpose or remove it.

**Pros:**
- Directly resolves all three issues raised.
- Aligns with the IG's own Getting Started statement that it does not prescribe how to program software.
- Consistent with the direction indicated by awatson1978's comment and the related tickets (FHIR-50736, FHIR-50737).

**Cons:**
- Requires careful rewording to maintain the IG's goal of normalization to a standard format without losing the intent.

### Disposition B: Persuasive with Modification

**Rationale:** Accept points 2 (naming fix) and 3 (Argonaut R4 clarification) fully. For point 1 (FHIR format storage assumption), modify the Data Model page to distinguish between *storage format* (which the IG does not prescribe) and *exchange format* (which MUST be FHIR), rather than removing the FHIR format requirement entirely. This preserves the IG's core normalization goal while clarifying that internal storage is implementation-defined.

**Pros:**
- Preserves the IG's intent of enabling longitudinal records through FHIR normalization.
- Draws a clear boundary between exchange/interoperability requirements and internal storage.
- Addresses the reporter's concern without weakening conformance requirements for exchange.

**Cons:**
- Requires the workgroup to draft precise language distinguishing storage from exchange contexts.

### Disposition C: Not Persuasive

**Rationale:** The Data Model page's assumption of FHIR format is intentional and core to the IG's purpose of creating standardized longitudinal records. The naming issue (Patient vs. Personal) and Argonaut R4 column questions could be addressed editorially without changing the fundamental assumption.

**Pros:**
- Maintains the IG's current architectural position.

**Cons:**
- Does not address the real confusion documented by this ticket and multiple related tickets (FHIR-49070, FHIR-50736, FHIR-50737, FHIR-50763).
- awatson1978's own comment acknowledges the points are well taken, suggesting the current language does need clarification.
- The naming inconsistency and unexplained Argonaut R4 column still need resolution regardless.

## Recommendation

**Disposition B: Persuasive with Modification**

This disposition best balances the reporter's valid concerns with the IG's core purpose. The cluster of related tickets (FHIR-49070, FHIR-50736, FHIR-50737, FHIR-50763) demonstrates a consistent pattern of confusion around whether the IG prescribes internal PHR storage format vs. an interoperability/exchange format. The IG editor (awatson1978) has already signaled agreement that clarification is needed. Disposition B preserves the normalization goal while explicitly scoping the FHIR format requirement to exchange/interoperability contexts, and addresses the naming inconsistency and Argonaut R4 column issues directly. The workgroup should coordinate this clarification with the overlapping tickets to ensure consistent language across the IG.
