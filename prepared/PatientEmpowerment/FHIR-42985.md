## Description

The Contextual Value extension ties a context around a personal advance care plan preference (e.g., "If I am so sick or seriously injured that I cannot express my own medical treatment preferences, and if I am not expected to live without additional treatment for my illness, disease, condition or injury, then I want my medical care team to know that these are the things that are most important to me: Avoiding prolonged dependence on machines, Not being a physical burden to my family, Dying at home").

The current definition allows any value type to be specified for the context as well as the value. As a result, it is not very interoperable and too broadly defined to be adopted in US Core.

## Resolution Description

The US Core team recommended using the existing concept of a precondition extension to capture this information. The existing ServiceRequest-precondition extension provides the needed capability, but it must apply to both Observation- and Goal-based profiles. As a result, the resolution adopts the structure of the ServiceRequest-precondition but names it "enclosed-precondition" in the guide.

## Analysis

### Core Issue
The ContextValue extension in the PACIO ADI IG is too loosely typed â€” it allows any value type for both the context and the value. This flexibility makes it unsuitable for adoption in US Core, which requires tighter, more interoperable structures.

### Resolution Approach
Rather than tightening the ContextValue extension itself, the resolution takes a different approach: adopt the structure of the existing ServiceRequest-precondition extension and create a new "enclosed-precondition" extension. This allows preconditions to be applied to both Observation-based and Goal-based ADI profiles, not just ServiceRequests.

### Key Considerations
1. **ServiceRequest-precondition deprecation**: PR #184 in fhir-extensions is deprecating ServiceRequest-precondition and adjusting version scope for observation-precondition. The workgroup should confirm the "enclosed-precondition" approach remains viable given these upstream changes.
2. **Scope across resources**: The resolution explicitly needs to work for both Observation and Goal profiles, which the original ServiceRequest-scoped extension did not support.
3. **Alignment with related tickets**: FHIR-46926 and FHIR-43552 both explore similar precondition/conditionality patterns in ADI content, suggesting a coordinated approach may be needed.
4. **Status is "Triaged" with label "To_be_applied"**: The resolution was agreed in 2023 but has not yet been applied to the specification.

## Proposed Dispositions

### Disposition A: Apply as Persuasive â€” Create "enclosed-precondition" Extension
Accept the resolution as stated. Create the "enclosed-precondition" extension modeled on the ServiceRequest-precondition structure, applicable to Observation and Goal resources. Retire the overly-broad ContextValue extension in favor of this more constrained approach.

- **Pros**: Directly addresses the US Core interoperability concern; reuses proven precondition pattern; supports both Observation and Goal profiles as needed.
- **Cons**: The upstream ServiceRequest-precondition is being deprecated (PR #184); need to verify the "enclosed-precondition" design is independent of that deprecation. Coordination with FHIR-46926 and FHIR-43552 may be needed.

### Disposition B: Apply with Modification â€” Align with Current Extension Landscape
Accept the intent but modify the approach to account for the deprecation of ServiceRequest-precondition (fhir-extensions PR #184). Instead of modeling after the deprecated extension, align the "enclosed-precondition" with the current observation-precondition or any successor pattern emerging from the extensions package.

- **Pros**: Future-proofs the design against upstream deprecations; ensures alignment with the evolving FHIR extensions ecosystem.
- **Cons**: Requires additional analysis of the current state of precondition extensions; may delay application further.

### Disposition C: Defer â€” Coordinate with Related Precondition Tickets
Defer application until FHIR-46926 (Medical Orders preconditions) and the fhir-extensions PR #184 outcomes are finalized, ensuring a unified precondition approach across all ADI sections and resource types.

- **Pros**: Avoids creating a precondition extension that may need immediate revision; enables a holistic design across ADI content types.
- **Cons**: Further delays a ticket open since 2023; US Core adoption of ADI preferences remains blocked.

**Disposition A: Apply as Persuasive** â€” The resolution is well-reasoned and addresses a clear interoperability gap.
