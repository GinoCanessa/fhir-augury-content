## Description

The ticket requests that the ADI IG specify the form type for a particular PMO (Portable Medical Order) document type. Currently, there is no mechanism to distinguish which specific form a PMO document derives from. For example, multiple PMO documents could exist for a patient, but one might be an ePOLST while another might be a state-specific form (e.g., Louisiana LaPOST or Louisiana using National POLST).

The goal is to separate jurisdiction information from the document type—allowing systems to distinguish both *where* a form originates (e.g., Louisiana) and *what* the form is (e.g., LaPOST).

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Confirm Pre-Applied Fix)

**Recommendation:** Accept the current resolution of Persuasive and confirm the pre-applied fix in PR #138.

**Rationale:** The issue identifies a genuine gap — PMO documents lack a way to specify their source form type and distinguish jurisdiction from document type. The pre-applied fix (PR #138) introduces a minimal source form (MSF) profile using a Binary resource, which has already been merged into the master branch. The approach of creating a dedicated profile for source form representation is consistent with the related work in FHIR-41779 (adding source of advance directive) and FHIR-48857 (improving source artifact extension guidance). However, the PR author notes the MSF profile needs further requirements refinement, so the workgroup should confirm that the current implementation is sufficient or identify what further constraints are needed.

### Disposition B: Persuasive with Modification — Request Additional Constraints on MSF Profile

**Recommendation:** Change the resolution to Persuasive with Modification, requesting that the minimal source form (MSF) profile be enhanced with specific constraints.

**Rationale:** While PR #138 provides a starting point, the PR description explicitly states it "needs further requirements [for] what to constrain in the MSF profile" and that "right now, it's just a Binary resource with the name and description." The workgroup could direct that additional constraints be added—such as requiring metadata elements that capture the specific form type (e.g., ePOLST, LaPOST) and the jurisdiction (e.g., state)—to fully satisfy the ticket's goal of distinguishing jurisdiction from document type. This would align with the guidance improvements in FHIR-48857 regarding jurisdiction representation.

### Disposition C: Not Persuasive — Addressed by Related Tickets

**Recommendation:** Change the resolution to Not Persuasive, noting that the concern is addressed by the combination of FHIR-48857 (jurisdiction extension guidance) and FHIR-49914 (PMO categories value set corrections).

**Rationale:** One could argue that the existing jurisdiction extension (FHIR-48857) combined with the corrected PMO categories value set (FHIR-49914) already provides the mechanisms needed to distinguish source form type and jurisdiction without introducing a new MSF profile. The jurisdiction extension on Composition captures state and organization, while the corrected value set clarifies document type classification. However, this disposition is weaker because those tickets address related but not identical concerns — neither specifically enables capturing the original form identity at the Observation level.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The pre-applied fix in PR #138 is a reasonable first step, but the PR author's own note that further constraints are needed suggests the implementation is incomplete. The workgroup should confirm the MSF profile approach and specify what additional constraints are required — particularly around capturing the specific form type (e.g., ePOLST vs. LaPOST) and jurisdiction as distinct, queryable elements. This aligns with the ticket's core goal of separating jurisdiction from document type and ensures the profile is useful for interoperability rather than being an unconstrained Binary wrapper.
