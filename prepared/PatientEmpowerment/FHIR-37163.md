## Description

The ticket requests that the **ValueSet canonical URL** for `PatientCorrectionCommunicationTypesVS` be moved from `hl7.org/fhir` to `terminology.hl7.org` (THO — HL7 Terminology). A UTG ticket should be created to register the ValueSet in THO.

The ticket notes that only ValueSets with a **required binding to a 'code' data type** should remain anchored in `hl7.org/fhir`. All others should use THO.

**Current URL:**
`http://hl7.org/fhir/uv/patient-corrections/2022MAY/ValueSet-PatientCorrectionCommunicationTypesVS.html`

This was originally imported as **Comment 6** by Ron G. Parker from the 2022 May ballot cycle.

## Analysis

### Context
This is a vocabulary governance request from TSMG (Terminology Services Management Group) during the 2022 May ballot. The request follows HL7 best practice (documented in the R5 specification at `terminologies.html#system`) that CodeSystem and ValueSet canonical URLs should be anchored in THO (`terminology.hl7.org`) rather than in the IG's own namespace (`hl7.org/fhir/uv/patient-corrections/...`). The exception is ValueSets with required binding to a `code` data type, which may remain in the FHIR namespace.

### Key Considerations
1. **Standard practice:** Moving terminology resources to THO is established HL7 policy and aligns with the TSMG Code System Identifier Task Force guidance.
2. **Originally deferred:** The workgroup previously voted 7-0-0 to defer ("Considered for Future Use") in October 2022, wanting to wait until code changes stabilized.
3. **Re-opened post STU1:** Now that STU1 has been published, the ticket has been re-opened, suggesting the workgroup is ready to address this.
4. **Batch change:** This is one of 8 related tickets — all should ideally be addressed together as a coordinated vocabulary migration.
5. **Binding check needed:** The workgroup should verify whether `PatientCorrectionCommunicationTypesVS` has a required binding to a `code` data type (which would exempt it from needing to move to THO).
6. **UTG ticket required:** Implementation requires creating a UTG ticket to register the ValueSet in THO, which is a process dependency on the Vocabulary WG / UTG.

## Proposed Dispositions

### Disposition A: Applied — Migrate to THO
**Action:** Accept the change request. Re-anchor the `PatientCorrectionCommunicationTypesVS` ValueSet canonical URL in THO (`terminology.hl7.org`). Create a UTG ticket to register it. Coordinate with the other 7 sibling tickets for a batch migration.

**Rationale:** This aligns with established HL7 terminology best practice. STU1 is published and the deferral rationale ("wait for code changes to stabilize") no longer applies. The 7-0-0 prior vote shows no objection to the change itself.

### Disposition B: Deferred — Defer to Next Ballot Cycle
**Action:** Defer the change to a future ballot cycle. Keep the ticket open and triaged.

**Rationale:** If the workgroup has higher-priority functional changes for the next version and wants to minimize scope, the terminology migration could be deferred again. However, this risks accumulating technical debt and repeated TSMG feedback.

### Disposition C: Not Persuasive — Retain Current URL
**Action:** Close as not persuasive, keeping the current URL anchoring.

**Rationale:** If the workgroup determines that `PatientCorrectionCommunicationTypesVS` has a required binding to a `code` data type (meeting the exception criteria stated in the ticket itself), the current URL placement may be acceptable. This would require a binding analysis and TSMG consultation.

## Recommendation

**Disposition A: Applied — Migrate to THO**

This is the recommended disposition. The change aligns with HL7 terminology governance best practice, was already supported 7-0-0 by the workgroup (only deferred for timing), and the timing concern has been resolved with STU1 publication. The workgroup should coordinate all 8 sibling THO tickets (FHIR-37158 through FHIR-37165) as a single batch migration, creating the necessary UTG tickets. Before proceeding, confirm whether this ValueSet's binding qualifies for the `code` data type exception — if it does, Disposition C may apply instead.
