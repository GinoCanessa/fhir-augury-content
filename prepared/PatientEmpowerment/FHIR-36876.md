# Ticket Review: FHIR-36876

**Title:** Can a disagreement be persuasive?
**Status:** Triaged **Priority:** Medium **Type:** Change Request
**Work Group:** Patient Empowerment **Specification:** Patient Corrections (FHIR)
**Reporter:** Craig Newman **Assignee:** Unassigned
**Created:** 2022-04-26 **Updated:** 2025-07-10 **Labels:** Section-3

---

## Summary

This ticket asks whether the state machine in Section 3.4.1 of the Patient Corrections IG should allow a transition from `disagreement_logged` to `amendment_completed`. The scenario is: a patient's rebuttal to a denial is persuasive enough that the provider decides to update the record after all. Currently, the state machine does not model this path—once a disagreement is logged, there is no arrow leading to amendment completion.

The ticket was originally resolved as "Considered for Future Use" (voted 7-0-0) and deemed out of scope for STU1. It was reopened on 2025-07-10 by Lynn Laakso for reconsideration post-STU1 publication.

## Details

**Description:**

In the state machine in Section 3.4.1, is it possible that a patient's rebuttal to a denial is persuasive and the record is updated? If so, should there be an arrow from `disagreement_logged` to `amendment_completed`?

**Comments:**

1. **Virginia Lorenzi (2022-07-11):** Explained that under HIPAA, a patient can disagree and a provider can optionally provide a rebuttal, but the provider is only required to log the disagreement. In practice, across all sites and patients consulted, disagreements were not reconsidered and rebuttals did not happen. The feature was intentionally left out unless a need arises.

2. **Craig Newman (2022-07-11):** Acknowledged Virginia's explanation.

3. **Lynn Laakso (2025-07-10):** Reopened the ticket post-STU1 publication. Reverted the previous resolution of "Considered for Future Use" (voted 7-0-0 by Debi Willis / John Keyes, originally made 2022-12-05) which had noted "Out of scope for STU1."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a transition arrow from `disagreement_logged` to `amendment_completed` in the Section 3.4.1 state machine diagram. This would model the scenario where a provider, upon reviewing a patient's disagreement with a denial, is persuaded to accept the correction after all. Update the corresponding state table in Section 3.4 to include this new valid state transition, with a description noting that a provider may reconsider a denial after reviewing the patient's disagreement and optionally amend the record.

#### Justification

Although Virginia Lorenzi noted in 2022 that this scenario had not been observed in practice, the ticket has been reopened post-STU1, suggesting the workgroup now sees value in modeling this path. HIPAA explicitly permits the provider to reconsider after a disagreement. Omitting this transition creates a dead-end in the state machine that does not reflect the full legal and procedural reality. Related tickets (FHIR-36874, FHIR-37048) show a pattern of commenters questioning whether the state machine fully captures denial and disagreement workflows. Adding this transition would make the model more complete and better align with HIPAA's framework, which allows but does not require providers to reconsider.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a direct arrow from `disagreement_logged` to `amendment_completed`, model the reconsideration as a new correction request cycle. Add guidance text to Section 3.4.1 explaining that if a provider is persuaded by a patient's disagreement, the provider should initiate a new correction process (returning to the initial state) rather than transitioning directly from `disagreement_logged`. This keeps the existing state machine simple while acknowledging the possibility. Add a note or informative text box explaining this workflow.

#### Justification

This approach preserves the simplicity of the current state machine while addressing the gap Craig Newman identified. A direct transition from `disagreement_logged` to `amendment_completed` could imply that disagreements routinely lead to corrections, which contradicts the real-world evidence Virginia Lorenzi cited. By modeling the reconsideration as a new correction cycle, the IG acknowledges the possibility without complicating the state machine diagram. This is consistent with how HIPAA frames the process—the provider reconsiders and then takes a new action, rather than the disagreement itself triggering the amendment.

---

### Disposition C: Decline

#### Proposal

Decline the change request. The current state machine accurately reflects the observed workflow in practice: once a denial is issued and a disagreement is logged, the process terminates. No transition from `disagreement_logged` to `amendment_completed` is needed.

#### Justification

Virginia Lorenzi's comment established that across all sites and patients consulted during STU1 development, provider reconsideration after a disagreement did not occur in practice. HIPAA only requires logging the disagreement, not reconsidering the denial. Adding a transition for an unobserved scenario adds complexity without demonstrated implementer need. The ticket was previously resolved 7-0-0 as "Considered for Future Use." If real-world evidence of this transition path emerges, it can be addressed in a future cycle with concrete use cases to guide the design.

---

### Recommendation

**Recommended disposition:** A

The ticket was originally deferred as "Out of scope for STU1" rather than rejected on its merits, and has now been explicitly reopened for post-STU1 consideration—signaling that the workgroup sees this as a legitimate gap to address. The request is technically sound: the HIPAA framework does permit providers to reconsider after a patient disagrees with a denial, and the current state machine creates a dead-end that does not reflect this legal reality. Multiple related tickets (FHIR-36874, FHIR-36875, FHIR-37048) demonstrate a pattern of reviewers finding the state machine's handling of denials and disagreements incomplete. Adding the `disagreement_logged` → `amendment_completed` transition is a targeted, low-risk change that makes the model more accurate and complete. Now that the spec is past STU1 and gaining real-world implementation experience, this is the appropriate time to fill in gaps that were intentionally deferred.
