## Description

The submitter identified a case-sensitivity bug in the FHIRPath constraint with key `HCA-section-entries` on the `ADI-PACPComposition` StructureDefinition (`http://hl7.org/fhir/us/pacio-adi/StructureDefinition/ADI-PACPComposition`).

The constraint expression incorrectly references `relatedPerson` (lowercase 'r') instead of `RelatedPerson` (uppercase 'R'). Since [FHIRPath is a case-sensitive language](http://hl7.org/fhirpath/N1/#case-sensitivity), the type check `$this.resolve() is relatedPerson` would never match any resource, effectively breaking the invariant.

**Current (incorrect):**
```fhirpath
entry.exists().not() or (entry.where($this.resolve() is Consent).exists() and entry.where($this.resolve() is relatedPerson).exists())
```

**Proposed (corrected):**
```fhirpath
entry.exists().not() or (entry.where($this.resolve() is Consent).exists() and entry.where($this.resolve() is RelatedPerson).exists())
```

## Analysis

This is a straightforward bug fix. The FHIRPath type literal `relatedPerson` is invalid because FHIRPath is case-sensitive and the correct FHIR resource type name is `RelatedPerson`. The broken invariant means the `HCA-section-entries` constraint could never pass validation when entries are present, since the `relatedPerson` type check would always evaluate to false. The fix has already been implemented and merged via [PR #95](https://github.com/HL7/fhir-pacio-adi/pull/95) on the same day the ticket was filed.
