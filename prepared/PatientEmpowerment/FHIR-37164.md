## Description

The ticket requests that the canonical URL for the **PatientCorrectionOutputTypesVS** ValueSet be moved from the `hl7.org/fhir` namespace to `terminology.hl7.org` (THO — HL7 Terminology). A UTG ticket should be created to register the ValueSet in THO.

Per HL7 terminology governance policy, only Value Sets with a **required** binding to a `code` data type should remain anchored in `hl7.org/fhir`. All others should be registered in THO.

**Current canonical URL:**
`http://hl7.org/fhir/uv/patient-corrections/ValueSet/PatientCorrectionOutputTypesVS`

This ticket was originally imported as Comment 7 by Ron G. Parker.

## Proposed Dispositions

### Disposition A: Persuasive with Modification

**Action:** Accept the request. Change the canonical URL for `PatientCorrectionOutputTypesVS` from `http://hl7.org/fhir/uv/patient-corrections/ValueSet/PatientCorrectionOutputTypesVS` to a THO-anchored URL under `http://terminology.hl7.org/ValueSet/...`. Create a UTG ticket to register the ValueSet in THO. Coordinate with TSMG to ensure proper registration. Handle this as part of the full batch of eight sibling tickets (FHIR-37158 through FHIR-37165) to ensure consistent treatment of all Patient Corrections terminology resources.

**Rationale:** The request is well-founded per HL7 terminology governance policy. The ValueSet does not have a required binding to a `code` data type, so it should be in THO. However, modification is needed because: (a) the new canonical URL must be determined in coordination with UTG/TSMG, (b) the IG will need a migration strategy to avoid breaking existing implementations now that STU1 is published, and (c) all eight sibling tickets should be handled together for consistency.

### Disposition B: Not Persuasive

**Action:** Reject the request. Retain the current canonical URL.

**Rationale:** The ValueSet URL is functional and established post-STU1 publication. Changing canonical URLs after publication risks breaking existing implementations that reference the current URL. The cost of migration and implementation disruption may outweigh the governance compliance benefit. The previous 7-0-0 vote to defer ("Considered for Future Use") reflected a reasonable judgment that this change was not yet warranted.

### Disposition C: Persuasive

**Action:** Accept as-is. Move the canonical URL to `terminology.hl7.org` and create a UTG registration ticket.

**Rationale:** HL7 terminology governance policy is clear: Value Sets not required-bound to a `code` data type belong in THO. The ticket was re-opened specifically because STU1 has now been published and the specification is ready for further changes. The policy should be applied directly without modification.

## Recommendation

**Disposition A: Persuasive with Modification**

The request correctly identifies a terminology governance compliance issue. Now that STU1 is published and the ticket has been re-opened for action, the canonical URL for `PatientCorrectionOutputTypesVS` should be moved to THO. However, this should be done with the following modifications:

1. **Batch handling** — All eight sibling tickets (FHIR-37158 through FHIR-37165) should be addressed together to ensure consistent treatment of all Patient Corrections Code Systems and Value Sets.
2. **UTG coordination** — A UTG ticket must be created to register the ValueSet in THO, and the new canonical URL should be confirmed with TSMG before implementation.
3. **Migration strategy** — Since STU1 is published, the IG should document the URL change and provide guidance for implementers transitioning from the old to the new canonical URL.
