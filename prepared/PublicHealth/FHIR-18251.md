## Analysis

This ticket was filed in September 2018, early in BSeR's development. It raises a fundamental design question: since BSeR is a US-realm IG, its Patient profile should logically derive from US Core Patient rather than the base FHIR Patient resource. This aligns with standard US-realm IG practice, where specifications build on US Core profiles to ensure interoperability with the broader US FHIR ecosystem.

Notably, **FHIR-22518** — filed approximately 8 months later (May 2019) — raises the same underlying concern from the implementer's perspective: the BSeR IG text states it relies on US Core Patient, but the actual BSeR Patient profile does not reference US Core Patient. FHIR-22518 was voted on (2019-06-20), resolved as **Persuasive with Modification**, and has been **Applied**. This strongly suggests that the US Core Patient alignment issue identified in FHIR-18251 has already been addressed through the work done for FHIR-22518.

Given that FHIR-18251 remains in "Triaged" status despite the related fix having been applied, this ticket appears to be a stale duplicate or predecessor of the resolved work.

## Proposed Dispositions

### Disposition A: Persuasive (Already Applied via FHIR-22518)
- **Resolution:** Persuasive
- **Action:** Acknowledge that the concern was valid and has been addressed. The BSeR Patient profile was aligned with US Core Patient as part of the resolution of FHIR-22518 (Persuasive with Modification, Applied). Close this ticket noting the original concern has been resolved.
- **Rationale:** The submitter's question was answered affirmatively — yes, the BSeR Patient profile should be (and now is) based on US Core Patient. The applied changes from FHIR-22518 satisfy this request.

### Disposition B: Duplicate of FHIR-22518
- **Resolution:** Duplicate
- **Action:** Mark as duplicate of FHIR-22518, which addressed the same US Core Patient alignment concern and has been applied.
- **Rationale:** FHIR-22518 explicitly calls out the disconnect between the BSeR IG's stated reliance on US Core Patient and the actual profile definition — the same issue raised by this ticket. Marking as duplicate provides clear traceability.

### Disposition C: Not Persuasive (Overtaken by Events)
- **Resolution:** Not Persuasive
- **Action:** Close as not persuasive, noting the issue was independently resolved via FHIR-22518.
- **Rationale:** While the concern was valid, it was addressed through a separate tracker item. No further action is needed on this ticket.

## Recommendation

**Disposition B: Duplicate of FHIR-22518** is recommended. This is the most accurate characterization: FHIR-22518 directly addressed the same concern (BSeR Patient profile lacking US Core Patient reference), was voted on, resolved as Persuasive with Modification, and Applied. Marking FHIR-18251 as a duplicate of FHIR-22518 provides clear traceability, avoids redundant work, and accurately reflects the history of this issue.
