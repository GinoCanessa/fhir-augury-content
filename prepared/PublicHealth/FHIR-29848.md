## Summary

This ticket identifies a structural problem in the SANER Implementation Guide where actor-level requirements are misplaced in transaction definition sections. Specifically, the transaction sections contain statements like:

> *"The MeasureSource SHALL demonstrate the FHIR read operation on the Location, MeasureReport, Measure and Organization resources."*

The submitter argues these are **actor requirements** (capabilities of MeasureSource), not **transaction requirements** (constraints on a specific interaction), and should be relocated to the actor definition/requirements sections of the IG.

The approved resolution is **Persuasive**: replace instances of actor names (e.g., "MeasureSource") with generic terms "client" and "server" in the transaction sections, thereby decoupling actor identity from transaction-level capability statements.

## Proposed Dispositions

### Disposition A: Apply as Persuasive — Replace Actor Names with Client/Server
**Action:** Accept the Persuasive resolution as stated. In all SANER transaction requirement sections, replace specific actor names (MeasureSource, MeasureDefinitionSource, etc.) with the generic terms "client" and "server." This decouples actor identity from transaction-level requirements while preserving the capability statements.

**Rationale:** This is the approved resolution and is architecturally sound. Transactions should describe client/server interactions generically; specific actor capabilities belong in actor definitions. This aligns with IHE-style IG conventions where transactions are actor-neutral.

**Considerations:** Should be applied in coordination with FHIR-29854, which raises the identical issue for MeasureDefinitionSource. Both tickets should be resolved together for consistency.

---

### Disposition B: Apply as Persuasive with Modification — Move Requirements to Actor Sections
**Action:** Rather than just renaming actors to client/server in transactions, **move** the actor-specific capability requirements (e.g., "SHALL implement FHIR read on Location, MeasureReport, Measure and Organization") into the respective actor definition sections, and simplify the transaction sections to reference only the transaction-specific behavior.

**Rationale:** Simply replacing "MeasureSource" with "client" still leaves actor-level requirements in the wrong section. A more thorough fix would relocate these statements entirely, as the submitter originally requested. Also incorporates kwboone's suggestion to change "demonstrate" to "implement."

**Considerations:** More editorial work required. May need to coordinate with FHIR-31744 (conformance section completeness) to ensure relocated requirements don't create further gaps.

---

### Disposition C: Not Persuasive — Retain Current Structure
**Action:** Reject the change. Keep actor-specific capability statements in the transaction sections as-is.

**Rationale:** For a small IG like SANER, having actor requirements in transaction sections provides a self-contained reference — implementers can read one section to understand all requirements for a given interaction without cross-referencing actor definitions. The current structure prioritizes readability for implementers.

**Considerations:** Does not address the architectural concern. Leaves the IG inconsistent with common IG authoring patterns and could cause confusion if the IG grows or additional actors are added.

---

## Recommendation

**Disposition B: Apply as Persuasive with Modification** is recommended.

The approved Persuasive resolution (rename to client/server) is a reasonable minimal fix, but the submitter's core concern is that actor requirements are in the wrong section — not merely that actor names appear in transactions. Disposition B addresses the root issue by relocating actor-level capability statements to actor definition sections, incorporates kwboone's "demonstrate → implement" wording improvement, and produces a cleaner IG architecture. This should be applied in tandem with FHIR-29854 to ensure both MeasureSource and MeasureDefinitionSource actor requirements are treated consistently.
