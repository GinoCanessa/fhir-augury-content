## Description

The Tobacco Request Support Bundle only allows one each of SmokingStatus and TobaccoCessationReferralObs, neither of which seems to include a way to send data elements set out in the DAM such as `bestDayToCall`, `bestTimeToCall`, `leaveMessageIndicator`, etc. The reporter asks how these data elements are supposed to be sent in a referral request.

**Reporter comment:** Other subject matter areas such as Childhood Nutrition may have the same issue.

## Analysis

### Core Issue

The ticket identifies a gap in the BSeR Tobacco Request Support Bundle profile. The bundle constrains its content to only SmokingStatus and TobaccoCessationReferralObs, but the DAM (Domain Analysis Model) defines additional data elements (e.g., `bestDayToCall`, `bestTimeToCall`, `leaveMessageIndicator`) that are needed for tobacco cessation referrals. There is no clear mechanism in the profiled bundle to transmit these DAM-defined elements.

### Context

This ticket was filed during the initial BSeR ballot cycle (September 2018) and reflects early implementer confusion about how the FHIR profiles align with the DAM requirements. The reporter (Craig Newman) filed several related tickets (FHIR-18206, FHIR-20864, FHIR-20871) all raising similar concerns about the mapping between the DAM and the FHIR profiles, the specificity of observation profiles, and guidance for implementers.

Several of those related tickets (FHIR-20864, FHIR-20871) have since been resolved as "Persuasive" and applied, suggesting the BSeR specification has evolved to address some of these structural concerns. However, FHIR-18248 itself and the closely related FHIR-18206 remain in "Triaged" status and unresolved.

### Impact

- **Implementers** cannot determine from the specification alone how to send DAM-specified data elements like contact preferences in a tobacco cessation referral request.
- The issue may extend to **other BSeR use cases** (Childhood Nutrition, as noted by the reporter).
- Without resolution, implementers must make ad hoc decisions about how to represent these data elements, reducing interoperability.

---

## Proposed Dispositions

### Disposition A: Persuasive — Expand the Tobacco Request Support Bundle to include DAM-defined data elements

**Action:** Update the BSeR Tobacco Cessation Referral Request support bundle profile to accommodate additional observations or resources for DAM-defined data elements such as `bestDayToCall`, `bestTimeToCall`, and `leaveMessageIndicator`. Define specific Observation profiles or use appropriate FHIR resource types (e.g., ContactPoint extensions, communication preferences) to carry these data elements. Document the mapping from DAM elements to FHIR resources.

**Rationale:** The DAM defines data elements required for tobacco cessation referrals that the current FHIR profiles do not accommodate. Implementers need explicit guidance and profiled structures to transmit these elements consistently.

### Disposition B: Persuasive with Modification — Provide guidance without structural profile changes

**Action:** Add narrative guidance to the BSeR specification explaining how DAM-defined data elements (contact preferences, scheduling information) should be represented using existing FHIR resource mechanisms (e.g., Patient contact information, Task inputs, or extensions). Do not add new sliced entries to the support bundle but clarify that additional data can be included via existing base resource capabilities.

**Rationale:** Since the specification has evolved since this ticket was filed (e.g., FHIR-20864 and FHIR-22515 addressed bundle structure), some DAM elements may now be accommodable through existing mechanisms. Narrative guidance may be sufficient without adding new profile constraints.

### Disposition C: Not Persuasive — The issue has been overtaken by subsequent specification changes

**Action:** Close the ticket as not persuasive, noting that subsequent BSeR revisions (particularly resolutions to FHIR-20864, FHIR-20871, FHIR-22515, FHIR-22607) have substantially restructured the referral bundles, updated terminology, and clarified observation profiles. If the gap still exists in the current version, a new ticket should be filed against the current specification.

**Rationale:** The ticket was filed against the September 2018 ballot version. The specification has undergone multiple ballot cycles and structural changes since then. The specific profiles and bundle structures referenced may no longer exist in their original form.

---

## Recommendation

**Disposition C: Not Persuasive** — This ticket was filed in September 2018 against the initial BSeR ballot and has remained in "Triaged" status for over six years. The BSeR specification has undergone significant structural changes in that time, including resolution of closely related tickets (FHIR-20864, FHIR-20871, FHIR-22515, FHIR-22607) that addressed bundle structure, observation profile clarity, and terminology. The specific profiles referenced (SmokingStatus, TobaccoCessationReferralObs within a sliced Tobacco Request Support Bundle) may no longer reflect the current specification architecture. If the underlying gap persists in the current BSeR version, a new ticket filed against the current specification would be more actionable. Closing this ticket avoids resolving against an outdated specification baseline.
