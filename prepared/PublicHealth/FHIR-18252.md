## Summary

The submitter identifies that the BSeR Obesity Request bundle appears to allow only a single occurrence of PatientAllergy, which in turn permits documenting only a single allergy. This limitation is restrictive for real-world clinical referral scenarios where patients commonly have multiple allergies. The same issue is noted in the Arthritis Request bundle. The request is to allow the allergy entry to repeat so that multiple patient allergies can be conveyed in BSeR referral request bundles.

## Proposed Dispositions

### Disposition A: Persuasive — Allow PatientAllergy to Repeat
**Resolution:** Persuasive  
**Action:** Modify the BSeR Obesity Referral Supporting Information and Arthritis Referral Supporting Information profiles (or their containing Composition/Bundle profiles) to change the cardinality of the PatientAllergy entry (or slice) from `0..1` to `0..*`, allowing multiple allergy resources to be included in the referral request bundles.  
**Rationale:** Patients commonly have multiple allergies that are clinically relevant to a referral. Limiting the bundle to a single allergy is an artificial constraint that does not reflect real-world clinical data. Allowing repetition aligns with FHIR's general resource model where AllergyIntolerance resources are naturally multi-instance per patient.

### Disposition B: Persuasive with Modification — Broaden Allergy Support via AllergyIntolerance Reference
**Resolution:** Persuasive with Modification  
**Action:** Rather than simply increasing cardinality on the existing PatientAllergy slice, restructure the referral supporting information profiles to reference `0..*` AllergyIntolerance resources directly (or via a more general slice that accommodates multiple allergy entries). Review both the Obesity and Arthritis request bundle profiles and any other use-case-specific bundles that share the same pattern.  
**Rationale:** The submitter's concern is valid, but the fix should be evaluated in the context of the broader BSeR bundle architecture. A structural review ensures the solution is consistent across all BSeR referral use cases and aligns with any ongoing BSeR 2.0 redesign work (per FHIR-42491's discussion of bundle vs. allowed content).

### Disposition C: Not Persuasive — Existing Structure Is Sufficient
**Resolution:** Not Persuasive  
**Action:** No change. Document that implementers can use the `otherSupportingInformation` slice or an additional Composition section to convey additional allergies if needed.  
**Rationale:** The current profile may intentionally limit the structured allergy entry to the most clinically significant allergy for the referral. Additional allergies could be communicated through other mechanisms in the bundle or via existing clinical documents. Changing cardinality may have downstream effects on referral recipients who expect a single allergy entry.

---

## Recommendation

**Disposition A: Persuasive — Allow PatientAllergy to Repeat**

This is the most straightforward resolution that directly addresses the submitter's valid concern. Patients frequently have multiple allergies, and a referral bundle that artificially restricts this to a single entry is inconsistent with clinical reality and the FHIR AllergyIntolerance resource model. The same fix should be applied to both the Obesity and Arthritis request bundle profiles. This is a low-risk change — expanding cardinality from `0..1` to `0..*` is backward-compatible since existing single-allergy bundles remain valid. The ticket has been open since 2018 and the issue is clear-cut; a direct resolution is appropriate.
