## Summary

The current State/Province/Territory value set used by the Vital Records IGs (VRDR, BFDR, and the Common Profiles Library) lacks two special two-letter codes used in IJE (Inter-Jurisdictional Exchange) records to represent unknown or unclassifiable jurisdictions:

- **XX** — Country is Canada but the Canadian province is unknown, or the country is known but is not the U.S. or Canada.
- **ZZ** — Unknown or blank U.S. state/territory, or unknown/unclassifiable country.

Without these codes, faithful round-tripping of data between FHIR and IJE formats is impossible. The reporter notes that the analogous problem for country codes was already solved by including the ZZ value in the country code value set, and the same approach should be applied to the state/territory/province value set. Technical corrections to all three IGs (VRDR, BFDR, and VRCL) would be necessary.

## Details

### Description

Current submitted vital records have two distinct unknown values that function as two-letter state codes:

- **XX** = Country is Canada but unknown Canadian province, or any country that is known but not U.S. or Canada.
- **ZZ** = Unknown or blank U.S. state or territory, or unknown/unclassifiable country.

The current value set lacks codes for these two values, which prevents faithful round-tripping of data from FHIR↔IJE. The same problem was faced with country codes, and the ZZ value was correctly included there. The same treatment is needed for the State/Territory/Province codes. This change is needed for both VRDR and BFDR, so technical corrections to all 3 IGs are probably necessary.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

**Action:** Add codes XX and ZZ to the States/Territories/Provinces value set in the Vital Records Common Profiles Library (VRCL), and publish technical corrections to VRDR, BFDR, and VRCL IGs.

**Rationale:** The request is well-justified and technically sound. IJE records legitimately use XX and ZZ as sentinel values for unknown state/territory/province, and the current value set cannot represent them. This is a data fidelity issue — without these codes, FHIR↔IJE round-tripping loses information. There is clear precedent: ZZ was already added to the country code value set for the same reason (FHIR-33193), and the value set was previously expanded to add YC for New York City (FHIR-33152). Adding two well-defined codes is a minimal, low-risk change that restores complete IJE semantics.

**Changes Required:**
1. Define XX and ZZ codes in the appropriate code system (e.g., a VRCL-specific code system or the existing jurisdictions code system).
2. Add XX and ZZ to the `StatesTerritoriesAndProvinces` (or equivalent) value set in VRCL.
3. Publish technical corrections to VRDR, BFDR, and VRCL IGs to reference the updated value set.

### Disposition B: Persuasive with Modification — Add ZZ Only, Map XX Differently

**Action:** Add only ZZ to the value set for unknown/blank state or territory. For the XX case (known non-US/non-Canada country with unknown province), use a data-absent-reason extension or a null flavor instead.

**Rationale:** ZZ represents a truly unknown value and aligns with FHIR's approach to missing data. XX, however, conflates two different semantic concepts (known foreign country with unknown subdivision). Instead of propagating IJE-specific sentinel codes, the XX case could be handled with FHIR-native mechanisms such as the `data-absent-reason` extension with value "unknown" on the state element, paired with a populated country code. This approach would improve semantic clarity while still enabling IJE round-tripping through mapping logic.

**Changes Required:**
1. Add ZZ to the state/territory/province value set.
2. Define mapping guidance for XX scenarios using data-absent-reason or equivalent FHIR mechanism.
3. Update IJE↔FHIR mapping documentation to cover the XX transformation.

### Disposition C: Decline (Not Persuasive)

**Action:** Do not modify the value set. Instead, handle XX and ZZ at the IJE↔FHIR mapping layer rather than in the value set itself.

**Rationale:** XX and ZZ are IJE-specific sentinel values, not standard jurisdiction codes. Adding them to a FHIR value set conflates transport-layer encoding with clinical data modeling. FHIR has its own mechanisms for representing unknown or absent data (e.g., data-absent-reason extension, nullFlavor). The mapping between IJE and FHIR should translate these sentinel values into appropriate FHIR representations rather than importing IJE-specific codes into FHIR terminology. This preserves clean separation between the IJE and FHIR data models.

## Recommendation

**Disposition A: Accept (Persuasive)** is recommended.

The request addresses a concrete data fidelity gap that prevents faithful FHIR↔IJE round-tripping. The approach is consistent with how the same problem was already solved for country codes (ZZ was added there), and with how the same value set was previously expanded (YC for New York City in FHIR-33152). The reporter, Saul Kravitz, has deep domain expertise in vital records encoding and has successfully driven related improvements (FHIR-33193, FHIR-36404). XX and ZZ are well-established, well-defined codes in the IJE ecosystem, and adding them to the FHIR value set is a pragmatic, low-risk change that enables complete interoperability between the two systems. While FHIR-native mechanisms for representing unknown data exist (Disposition B/C), the vital records community has consistently preferred direct code representation for IJE compatibility, making Disposition A the most practical and consistent choice.
