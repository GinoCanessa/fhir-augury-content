## Problem Statement

The NHSN dQM IG currently defines a single submission profile — the NHSN MeasureReport Bundle — which packages all data into one monolithic bundle. The submitter notes that a single submission could reach gigabyte scale or larger. This single-resource design is incompatible with the Bulk Data Transport IG, which requires NDJSON with one resource per line. Transmitting very large single bundles risks interrupted transfers, and any chunking workaround would require non-FHIR mechanisms.

The ticket requests:
1. **New submission profiles** that enable bulk data transport (e.g., a Header Measure Bundle profile and an Individual Report Bundle profile for individual-level measure reports).
2. **Alignment with DEQM** guidance for organizing bundles by subject, where each bundle contains MeasureReports and evaluatedResources for all measures applying to a single subject, reducing data duplication.

## Proposed Dispositions

### Disposition A: Persuasive — Create New Submission Profiles for Bulk Data Support

**Action:** Accept the request. Create new submission profiles enabling bulk data transport:
- Define a **Header/Summary MeasureReport Bundle** profile containing subject-list MeasureReports and organizational metadata.
- Define an **Individual Report Bundle** profile containing individual-level MeasureReports with their evaluatedResources for a single subject.
- Add guidance on how these profiles can be used with the Bulk Data Transport IG (NDJSON-based export/import).
- Align bundle organization with DEQM guidance (per FHIR-46372) so each bundle is organized by subject.

**Rationale:** The DEQM ecosystem has matured since the original deferral — FHIR-46372 (subject-organized bundles) and FHIR-50487 ($submit-data alignment) are now published/applied. The Argonaut and HRSA efforts referenced in the original comment provide additional patterns for multi-file bulk data. Post-STU1, this is the appropriate time to incorporate these learnings. Gigabyte-scale single bundles are a practical barrier to reliable data transmission.

**Risk:** Significant design effort required. New profiles must be coordinated with DEQM and Bulk Data IG teams to ensure consistency. May break existing implementations using the monolithic bundle approach if not properly versioned.

---

### Disposition B: Persuasive with Modification — Add Guidance and Defer New Profiles

**Action:** Partially accept. Add normative guidance in the NHSN dQM IG describing how implementers SHOULD organize submissions for bulk data compatibility (referencing DEQM subject-organized bundle patterns), but defer creating new formal submission profiles until the Bulk Data IG's file organization work (Argonaut) reaches a more mature state.

**Rationale:** The underlying standards (Bulk Data output file organization, manifest patterns) are still evolving. Adding guidance now gives implementers direction without locking in profiles that may need revision. This balances the immediate need for large-submission guidance with the risk of premature standardization.

**Risk:** Without concrete profiles, implementations may diverge. Guidance-only approaches are less enforceable and may not sufficiently address the interoperability gap.

---

### Disposition C: Considered for Future Use — Re-defer Pending Ecosystem Maturity

**Action:** Re-defer this request. The ticket was previously resolved as "Considered for Future Use" with a 35-0-0 vote. Maintain that posture until the Argonaut Bulk Data file organization and HRSA UDS+ manifest work are finalized and published.

**Rationale:** The original rationale remains valid — bulk data multi-file transport patterns are still being developed across the ecosystem. Premature standardization in the NHSN IG could create misalignment with these community-wide efforts.

**Risk:** The ticket has been re-opened post-STU1 specifically to address this. Continued deferral may frustrate implementers dealing with gigabyte-scale submissions and delay needed interoperability improvements. The ecosystem has progressed since the original deferral (FHIR-46372 and FHIR-50487 are now resolved).

---

## Recommendation

**Disposition A: Persuasive — Create New Submission Profiles for Bulk Data Support**

This is the recommended disposition. The ticket was re-opened post-STU1 with the explicit intent to address bulk data transport for the next version. The DEQM alignment work that was cited as a reason for deferral has now been completed (FHIR-46372 published, FHIR-50487 applied). The practical problem — gigabyte-scale monolithic bundles being incompatible with standard FHIR transport mechanisms — is a real barrier for implementers. Creating Header/Summary and Individual Report bundle profiles would decompose submissions into NDJSON-compatible resources, enabling use of the Bulk Data Transport IG. Coordination with the CQI work group (which owns DEQM) will be essential to ensure profile consistency, but the foundational standards are now sufficiently mature to proceed.
