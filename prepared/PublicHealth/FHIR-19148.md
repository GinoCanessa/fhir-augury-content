## Summary

This ticket asks whether the ODH (Occupational Data for Health) RetirementDateModel has considered capturing a **planned or future retirement date**, in addition to past retirement dates. The submitter notes that retirement likely results in changes to income and healthcare, and questions whether the model should support patients who are preparing for or anticipating retirement. The ticket was originally resolved as "Considered for Future Use" by PHWG vote (20-0-6) on 2023-01-16, with the rationale that this is not part of considerations for the current model. However, it was subsequently **re-opened** post ODH 1.3 publication by lynn_laakso and reverted to Triaged status.

## Details

The core question is whether the ODH RetirementDateModel should be extended to support **prospective (future) retirement dates** — i.e., a date a patient plans to retire — rather than only recording that a person has already retired. The submitter argues that upcoming retirement impacts income and healthcare planning, and capturing this information could support clinical care.

## Proposed Dispositions

### Disposition A: Accept — Add Future Retirement Date Support to ODH
**Resolution: Persuasive**

Accept the request and extend the RetirementDateModel to support a planned/future retirement date. This would involve:
- Adding an optional `plannedRetirementDate` element (or equivalent) to the RetirementDateModel logical model.
- Updating the ODH profile to allow a dateTime value that may be in the future.
- Adding guidance clarifying the distinction between an actual retirement date (past) and a planned/anticipated retirement date (future).

**Rationale:** Capturing planned retirement supports proactive care planning, especially around transitions in healthcare coverage and income. The ticket was re-opened post 1.3 publication, indicating the work group sees value in reconsidering this for a future version. Related tickets (FHIR-19168 on multiple retirements, FHIR-19156 on RetirementDateModel definitions) have already been resolved, laying groundwork for model evolution.

### Disposition B: Alternative — Document as Guidance Without Model Change
**Resolution: Persuasive with Modification**

Rather than adding a new data element, add implementation guidance to ODH explaining how implementers can use the existing RetirementDateModel with a future date value to indicate planned retirement. This approach:
- Avoids structural changes to the logical model.
- Clarifies that the existing `retirementDate` element may contain a future date.
- Adds a note in the element description that a date in the future indicates anticipated/planned retirement.

**Rationale:** This is a lower-impact change that addresses the submitter's concern without expanding the model's scope. It preserves backward compatibility while enabling the use case through guidance rather than new elements.

### Disposition C: Decline — Not in Scope for Current ODH Model
**Resolution: Not Persuasive**

Decline the request, reaffirming the original 2023-01-16 vote (20-0-6) that future retirement date is not part of the considerations for the current model. The ODH RetirementDateModel is intended to capture whether a person self-identifies as retired (a past event), not to predict future employment transitions. Planned retirement is a social/financial planning concept outside the clinical scope of ODH.

**Rationale:** The original vote was decisive (20-0-6). While the ticket was re-opened for reconsideration post 1.3, the fundamental scope argument remains: ODH focuses on occupational data relevant to health, and prospective retirement dates are speculative and not directly actionable in clinical workflows.

---

## Recommendation

**Disposition B: Persuasive with Modification** — Document that the existing RetirementDateModel can accept a future date to indicate planned retirement, and add implementation guidance.

This balances the submitter's valid concern about retirement's impact on healthcare planning with the work group's earlier conclusion that a new model element is not warranted. By clarifying that the existing date element can hold a future value and adding guidance on interpretation, the use case is enabled without structural model changes. This approach is consistent with the decisive original vote against model expansion while still addressing the re-opened ticket constructively.
