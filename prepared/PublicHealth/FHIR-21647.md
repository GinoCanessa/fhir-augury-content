## Summary

This change request asks the Public Health work group to consider migrating the US Healthcare Associated Infections (HAI) FHIR Implementation Guide from FHIR STU3 (R3) to FHIR R4. The request was filed in May 2019 by David DeRoode and has remained in Triaged status since then with no comments or further activity recorded.

## Details

The ticket description is brief: "Consider updating HAI from FHIR R3 to FHIR R4." At the time of filing, FHIR R4 had just been released as the first normative edition (January 2019), making migration from STU3 a natural consideration for implementation guides that were still based on the earlier version.

The HAI FHIR IG defines profiles and resources for reporting healthcare-associated infections to the National Healthcare Safety Network (NHSN) at the CDC. Moving to R4 would align the HAI IG with the current normative base, improve interoperability with the broader US FHIR ecosystem (e.g., US Core, which is R4-based), and enable use of R4-specific features and improvements.

No comments have been recorded on this ticket. No cross-references to other Jira tickets, Zulip discussions, or GitHub issues were found directly linking to FHIR-21647.

A related ticket in the HAI ecosystem — [FHIR-24484](https://jira.hl7.org/browse/FHIR-24484) ("Update to published versions of profiles once the HAI FHIR IG is published") — was resolved as **Persuasive** under the HAI-LTCF specification, indicating the work group has previously accepted the direction of updating HAI-related specifications to align with published FHIR versions.

## Proposed Dispositions

### Disposition A: Persuasive (Accept)

Accept the change request and formally plan migration of the HAI FHIR IG from STU3 to FHIR R4 (or R4B/R5 as appropriate at time of implementation).

**Rationale:** FHIR R4 has been the normative release since January 2019 and is the established baseline for US FHIR implementation guides including US Core. Migrating HAI to R4 aligns with the broader HL7 ecosystem direction, improves interoperability, and enables use of normative resources. The related HAI-LTCF ticket (FHIR-24484) was already resolved as Persuasive, establishing precedent for this kind of update within the HAI family of specifications.

**Action:** Work group to schedule HAI IG R4 migration work, identifying resource mapping changes, profile updates, and testing requirements.

### Disposition B: Persuasive with Modification

Accept the request but broaden the scope to target the most current applicable FHIR version at the time work begins, rather than specifically R4, given that the ticket has been open since 2019 and FHIR R5 is now available.

**Rationale:** Given the elapsed time since the ticket was filed, a strict R3-to-R4 migration may not represent the best use of resources if the work group could instead target R4B or R5. The disposition acknowledges the valid request while allowing the work group flexibility to choose the most strategic target version.

**Action:** Work group to evaluate whether R4 or a later version is the appropriate migration target and plan accordingly.

### Disposition C: Not Persuasive (Decline)

Decline the request on the grounds that it is too vague to be actionable and lacks a specific technical justification or implementation plan.

**Rationale:** The ticket contains only a brief suggestion ("Consider updating HAI from FHIR R3 to FHIR R4") without detailing specific problems with the current R3-based IG, breaking changes that would be addressed, or stakeholder demand for the migration. The work group may prefer to address migration through a more detailed, scoped set of tracker items rather than a single open-ended request.

**Action:** Close as not persuasive; request submitter or interested parties to file specific, actionable tickets if migration work is desired.

## Recommendation

**Disposition A: Persuasive (Accept)**

The request to migrate from FHIR R3 to R4 is straightforward, well-aligned with the broader HL7 ecosystem trajectory, and consistent with the work group's prior acceptance of FHIR-24484 for the HAI-LTCF specification. FHIR R4 has been normative for over six years and is the baseline for US Core and virtually all active US realm implementation guides. Keeping the HAI IG on STU3 creates increasing interoperability friction. While the ticket text is brief, the intent is clear and the direction is uncontroversial. Accepting and planning the migration work is the most productive path forward.
