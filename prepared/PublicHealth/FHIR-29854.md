## Description

The submitter identifies that the SANER IG transactions are placing requirements on actors that belong in the actor definition/requirements sections rather than in the transaction requirements section. Specifically, the existing wording states:

> "The MeasureDefinitionSource SHALL demonstrate the FHIR read operation on the Measure, Library, ValueSet, CodeSystem and ConceptMap resources."

This is an **actor-level capability requirement** (what the MeasureDefinitionSource must support), not a **transaction-level requirement** (what must happen during a specific transaction exchange). The submitter argues these requirements should be relocated to the appropriate actor definition sections of the specification.

The approved resolution description states: "Adjust language to not be an actor requirement but a transaction requirement," meaning the wording should be reframed so that it describes what the transaction requires rather than what the actor must be capable of overall.

## Proposed Dispositions

### Disposition A: Apply as Persuasive â€” Relocate Requirements to Actor Sections

**Action:** Accept the ticket as Persuasive and move the MeasureDefinitionSource capability requirements (SHALL support FHIR read on Measure, Library, ValueSet, CodeSystem, ConceptMap) from the transaction requirements section to the MeasureDefinitionSource actor definition section. Update the transaction section to describe only the expected behavior during the transaction exchange itself (e.g., "The transaction SHALL include a FHIR read interaction on the relevant resource").

**Rationale:** The resolution was already voted as Persuasive. The distinction between actor capabilities and transaction behaviors is a well-established IHE/HL7 modeling principle. Relocating these requirements improves specification clarity and aligns with how other FHIR IGs structure actor vs. transaction conformance. This also aligns with the related fix in FHIR-29847 (changing "demonstrate" to "support").

**Impact:** Structural reorganization of the SANER conformance sections. No functional change to what implementers must support. Should be coordinated with FHIR-29848 (same issue for MeasureSource actor).

### Disposition B: Apply as Persuasive with Modification â€” Reword In-Place Without Relocating

**Action:** Accept as Persuasive with Modification. Instead of relocating the requirements, reword them in the transaction section to clearly frame them as transaction-scoped behaviors: e.g., "This transaction requires the MeasureDefinitionSource to respond to FHIR read operations on Measure, Library, ValueSet, CodeSystem, and ConceptMap resources."

**Rationale:** The resolution description says "Adjust language to not be an actor requirement but a transaction requirement," which could be interpreted as rewording rather than relocating. This minimizes structural changes to the IG while still addressing the conformance language concern.

**Impact:** Minimal editorial change. Lower risk of introducing inconsistencies but does not fully resolve the architectural concern about where actor capabilities should be specified.

### Disposition C: Defer â€” Coordinate with Broader SANER Conformance Cleanup

**Action:** Defer this ticket pending a coordinated review of all SANER conformance requirements. Group it with FHIR-29848, FHIR-29855, FHIR-29847, and FHIR-29889 for a single pass through the conformance sections.

**Rationale:** Multiple open tickets (FHIR-29848, FHIR-29855) address the same structural conformance issue across different actors. A piecemeal approach risks inconsistencies. A holistic review would ensure all actor/transaction requirements are properly placed and consistently worded across the entire specification.

**Impact:** Delays resolution but produces a more consistent outcome. Requires workgroup coordination to schedule the review.

## Recommendation

**Disposition A: Apply as Persuasive â€” Relocate Requirements to Actor Sections**

This is the strongest option because:

1. **The resolution is already approved as Persuasive** with clear direction to adjust the language so it is a transaction requirement rather than an actor requirement.
2. **The fix is straightforward** â€” move MeasureDefinitionSource capability statements to the actor definition section and reframe the transaction section language.
3. **FHIR-29847 (already published) sets precedent** â€” it already changed "demonstrate" to "support" in similar conformance language, validating the direction of this change.
4. **FHIR-29848 should be resolved in tandem** as it is a near-duplicate affecting the MeasureSource actor. Applying both together ensures consistency.

The workgroup should apply the Persuasive resolution, coordinate with FHIR-29848, and leverage the language patterns already established by the published FHIR-29847 fix.
