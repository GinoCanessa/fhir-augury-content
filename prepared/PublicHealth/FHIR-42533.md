## Description

The ticket poses a question about intermediary behavior in the BSeR IG. Sometimes the Intermediary (CBO Hub or Healthcare Information Exchange (HIE)) acts **on behalf of the CBO**, presenting the CBO with web access to data at the intermediary. Other times, the intermediary acts as a **"pass through"** and full functionality exists at the CBO. Currently, one of each option is represented in the interaction diagrams (with intermediary). The question is: **Should both of these possibilities be represented for both transaction use cases?**

- **Referral - Push (with intermediary)**
- **Referral - Notify and Pull (with intermediary)**

## Analysis

This is a **design question** about the completeness of interaction diagrams in the BSeR IG. The current IG shows one intermediary mode (act-on-behalf) for one use case and the other mode (pass-through) for the other use case, but does not show both modes for both use cases. The question is whether a full 2×2 matrix of diagrams is needed:

| | Referral - Push | Referral - Notify and Pull |
|---|---|---|
| **Intermediary acts on behalf of CBO** | Currently shown | Not shown |
| **Intermediary as pass-through** | Not shown | Currently shown |

The related ticket FHIR-42127 already contains a ballot-response-level answer of **"Yes"** — both possibilities should be represented. However, FHIR-42533 itself remains unresolved, suggesting the diagrams and narrative have not yet been updated to reflect this decision.

The topic also connects to broader ongoing work across BSeR and SDOH Clinical Care IGs about better defining intermediary roles (FHIR-53803, FHIR-53802) and detailing the Notify and Pull workflow (FHIR-42467).

## Proposed Dispositions

### Disposition A: Persuasive — Add Both Intermediary Modes for Both Use Cases

**Action:** Add interaction diagrams and narrative showing both intermediary modes (act-on-behalf and pass-through) for both the Referral - Push (with intermediary) and Referral - Notify and Pull (with intermediary) use cases, producing a total of four intermediary interaction diagrams.

**Rationale:** This directly answers the question posed in the ticket and aligns with the ballot response already captured in FHIR-42127 ("Yes, both of the approaches/possibilities should be represented in the IG"). Providing all four diagrams gives implementers a complete picture of the interaction patterns and removes ambiguity about which intermediary modes apply to which transaction types.

**Trade-offs:** Increases IG diagram count and maintenance burden; however, the gain in clarity for implementers justifies the effort.

### Disposition B: Persuasive with Modification — Answer the Question Without Prescribing Diagram Format

**Action:** Resolve the question by confirming that both intermediary modes are applicable to both use cases. Add narrative text explaining the two modes. Instead of four separate diagrams, update the existing two diagrams to annotate both intermediary behaviors (e.g., via notes or conditional paths) and reference the distinction in accompanying text.

**Rationale:** Acknowledges the answer is "yes" but avoids proliferating diagrams. This approach provides the information in a more compact format, reducing maintenance overhead while still informing implementers.

**Trade-offs:** Annotated diagrams may be harder to read than separate focused diagrams. May not be as clear for implementers new to the specification.

### Disposition C: Considered — No Change Required (Defer to FHIR-42127)

**Action:** Mark this ticket as a duplicate of or answered by FHIR-42127, which already contains the "Yes" response. Close with no independent change, relying on FHIR-42127 to drive the actual IG updates.

**Rationale:** FHIR-42127 already addresses this exact question with a clear response. Implementing the changes under the parent ticket avoids duplicated work and conflicting edits.

**Trade-offs:** If FHIR-42127 is not tracked carefully, the answer may never result in IG changes. The question remains visibly unresolved on this ticket, which may cause confusion.

## Recommendation

**Disposition C: Considered — No Change Required (Defer to FHIR-42127)** is recommended.

FHIR-42127 already contains the exact same question with an authoritative ballot response of "Yes, both approaches should be represented." Both tickets share the BSeR_INTERMEDIARIES label and target the same specification. Making IG changes under FHIR-42127 (which is a Change Request of Highest priority and broader scope covering three ballot questions) is the most efficient path. FHIR-42533 should be linked to FHIR-42127 and closed as answered, with a note pointing to the existing response. The work group should ensure that FHIR-42127's resolution includes the diagram updates that answer this question.
