## Description

The ticket asks whether `Observation.valueQuantity` in the BSeR Body Height profile should be constrained to use centimeters as the unit of measure, as indicated by the UML expression in the specification. The submitter is requesting clarification or a normative constraint on the unit for body height observations within BSeR.

## Analysis

This ticket has been in **Triaged** status since 2018 with no comments, no assignee, and no activity since February 2019. The core question is straightforward: should the BSeR Body Height profile constrain `Observation.valueQuantity` to centimeters (cm) as the UML diagram suggests?

Key considerations:
1. **FHIR Core Vital Signs profiles** (e.g., `observation-bodyheight`) already define expected UCUM units for body height as `cm` or `[in_i]` (inches). The base profile does not restrict to a single unit.
2. **BSeR specification context**: If the BSeR UML expression explicitly shows centimeters, the profile should either match that constraint or the UML should be corrected to reflect the intended flexibility.
3. **Interoperability**: Constraining to a single unit (cm) simplifies validation but may not accommodate US clinical workflows that commonly use inches. Allowing both cm and inches (as the core vital signs profile does) is more flexible.
4. **Precedent**: FHIR-33613 (resolved as Persuasive with Modification) addressed similar body height profiling questions and aligned with the base VitalSigns profile approach.

---

## Proposed Dispositions

### Disposition A: Persuasive — Constrain to centimeters only
Accept the request and constrain `Observation.valueQuantity` in the BSeR Body Height profile to centimeters (cm) only, matching the UML expression. This ensures consistency between the UML diagram and the profile, and simplifies unit handling for BSeR implementers who can standardize on metric units.

**Pros:** Consistency with UML; simpler validation.
**Cons:** Limits flexibility for US implementers who use inches; diverges from FHIR Core vital signs approach.

### Disposition B: Persuasive with Modification — Allow both cm and inches, update UML
Accept the concern but resolve by updating the BSeR UML expression to allow both centimeters (`cm`) and inches (`[in_i]`), aligning with the FHIR Core `observation-bodyheight` vital signs profile which permits either unit. Update the profile and UML to be consistent with each other.

**Pros:** Aligns with FHIR Core vital signs; accommodates US clinical practice; resolves UML/profile inconsistency.
**Cons:** Slightly more complex validation than a single-unit constraint.

### Disposition C: Not Persuasive — No change needed
Reject the request. The UML expression is informative/illustrative and does not necessarily mandate a constraint in the formal profile. Implementers can use any valid UCUM unit for body height as permitted by the base vital signs profile. No structural change to BSeR is required.

**Pros:** No specification change needed; preserves maximum flexibility.
**Cons:** Leaves the inconsistency between UML and profile unaddressed; may confuse implementers.

---

## Recommendation

**Disposition B: Persuasive with Modification** — This is the most balanced resolution. It addresses the submitter's concern about the UML/profile inconsistency while aligning BSeR with the FHIR Core vital signs approach that allows both centimeters and inches. This approach follows the precedent set by FHIR-33613 (aligning with base vital signs profiles) and accommodates real-world US clinical workflows where height is commonly recorded in inches. The UML expression should be updated to reflect that both `cm` and `[in_i]` are acceptable units.
