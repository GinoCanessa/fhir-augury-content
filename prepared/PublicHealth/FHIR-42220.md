## Summary

The reporter requests relaxing the minimum cardinality of `Task.for` from 1..1 to 0..1 in the BSeR Referral Task profile. The argument is that the subject (patient/beneficiary) details are already conveyed through the linked ServiceRequest, making the requirement on Task.for duplicative. Requiring systems to populate the same information in multiple places creates potential for inconsistency errors if values do not match. The ticket was resolved as **Not Persuasive** — the workgroup decided to keep `Task.for` as required based on stakeholder consensus (including alignment with Gravity SDOH) and the rationale that `Task.for` is a reference (pointer) to a beneficiary rather than duplicated data, consistent with how `subject` is treated in nearly all FHIR resources.

## Details

### Description

> Minimum of 1.. For Task.for appears overly constrained. Details on the subject of the serviceRequest are contained within — duplicating the requirement for systems to convey these details multiple places with potential for error if they do not is overly rigid.

### Resolution Rationale

The workgroup resolved this as Not Persuasive, stating:

- This was decided through stakeholder engagement and alignment with Gravity SDOH and various other stakeholders.
- `Task.for` is a reference to a beneficiary, so the data isn't being repeated but pointed to — the same pattern as the `subject` requirement in almost all FHIR resources.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

Relax the cardinality of `Task.for` from 1..1 to 0..1 in the BSeR Referral Task profile, making it optional. This acknowledges the reporter's concern that the patient subject is already conveyed through the mandatory linked ServiceRequest (`ServiceRequest.subject`). Systems that do not have a separate need for `Task.for` would no longer be required to populate it redundantly. The existing `Task.for` element would remain available and recommended for use, but not enforced. This approach aligns with the decision made on FHIR-42225 (relaxing `ServiceRequest.occurrence[x]` cardinality) and FHIR-42219 (relaxing `Task.statusReason` MustSupport), where similar "overly constrained" arguments from the same ballot cycle were found persuasive.

### Disposition B: Persuasive with Modification

Keep `Task.for` as required (1..1) but add explicit guidance in the profile narrative explaining *why* it is required despite the linked ServiceRequest also carrying subject information. The guidance should clarify that `Task.for` serves as a direct, first-class beneficiary reference enabling task-centric queries and workflow routing without requiring traversal to the ServiceRequest. This addresses the reporter's underlying concern (lack of rationale) without changing the cardinality. Additionally, consider adding an invariant or guidance noting that `Task.for` SHOULD reference the same patient as `ServiceRequest.subject` to address the data consistency concern.

### Disposition C: Decline (Not Persuasive)

Maintain the current resolution of Not Persuasive with no changes. The 1..1 cardinality on `Task.for` was established through stakeholder consensus involving Gravity SDOH and other cross-IG alignment efforts. `Task.for` is a reference (pointer), not duplicated data, and follows the universal FHIR pattern where resources carry a `subject` reference. Relaxing this cardinality would break alignment with Gravity SDOH profiles and potentially degrade interoperability for systems that rely on `Task.for` being present for patient-matching and workflow routing.

## Recommendation

**Disposition B: Persuasive with Modification.** The original resolution rationale is sound — `Task.for` as a reference is not truly duplicative data, and its presence enables important query and workflow patterns. However, the reporter's concern highlights a gap in the profile's explanatory content. The same ballot cycle saw similar "overly constrained" arguments accepted (FHIR-42225, FHIR-42219) and declined (FHIR-42214), suggesting the workgroup evaluates each element on its merits. Adding explicit rationale and consistency guidance to the profile would address the implementer confusion raised here without breaking stakeholder-agreed architecture. This balances the legitimate implementer concern with the interoperability requirements that drove the original decision.
