## Description

The CauseOfDeathPart1 profile currently passes cause-of-death information as a text string in the `.text` part of a CodeableConcept. Meanwhile, the EntityAxisCauseOfDeath profile provides ICD-10 codes for each line/position. The submitter proposes two alternatives to improve coded cause-of-death exchange:

1. **Bind to ICD-10 ValueSet:** Bind `valueCodeableConcept` in CauseOfDeathPart1 to the same ICD-10 valueset currently used for EntityAxisCauseOfDeath (EAC), and document its use for the coded version of cause of death. This would provide a more natural, "FHIR-like" way to exchange coded cause of death for both Part1 and Part2 records.

2. **Bind and eliminate EAC profile:** Same as option 1, but also formally map the EAC to `valueCodeableConcept.code` and eliminate the EntityAxisCauseOfDeath profile entirely.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Bind ICD-10 ValueSet to Part1 and Part2 (Option 1 only)

**Action:** Bind `valueCodeableConcept` in CauseOfDeathPart1 and CauseOfDeathPart2 to the same ICD-10 valueset used by EntityAxisCauseOfDeath, with an `example` or `preferred` binding strength. Document usage for coded cause-of-death exchange. Retain the EntityAxisCauseOfDeath profile.

**Rationale:** This provides a more FHIR-idiomatic way to exchange coded cause-of-death data while preserving the existing EntityAxisCauseOfDeath profile, which serves a distinct purpose in representing NCHS-produced axis coding. The EntityAxisCauseOfDeath profile conveys position-specific multi-axis coding produced by NCHS mortality classification, which is structurally different from a clinician-reported cause-of-death code. Retaining it avoids breaking existing implementations. The previous 25-0-0 vote to "consider for next ballot" signals strong workgroup interest in this direction.

**Impact:** Moderate. Adds an optional binding to existing profiles. Does not break existing implementations since text-only use remains valid. Aligns with the pattern established by FHIR-43599 and FHIR-41314 of making these profiles more flexible.

### Disposition B: Persuasive — Bind and Eliminate EAC Profile (Option 2)

**Action:** Bind `valueCodeableConcept` in CauseOfDeathPart1 and CauseOfDeathPart2 to the ICD-10 valueset, formally map EntityAxisCauseOfDeath data to `valueCodeableConcept.code`, and eliminate the EntityAxisCauseOfDeath profile.

**Rationale:** This achieves maximum simplification of the VRDR profile landscape by consolidating coded cause-of-death into the existing Part1/Part2 profiles. Reduces profile count and conceptual overhead.

**Impact:** High. Eliminating the EntityAxisCauseOfDeath profile is a breaking change. Implementations that currently produce or consume EntityAxisCauseOfDeath resources would require migration. The entity axis coding carries NCHS-specific multi-position axis data that may not map cleanly into the Part1/Part2 structure. This approach carries significant implementation risk.

### Disposition C: Not Persuasive — Maintain Current Separation

**Action:** No change. Keep CauseOfDeathPart1/Part2 as text-based and EntityAxisCauseOfDeath as the coded representation.

**Rationale:** The current design reflects a deliberate separation: clinician-reported text cause-of-death (Part1/Part2) vs. NCHS-produced coded axis data (EntityAxisCauseOfDeath). These represent different stages in the mortality coding pipeline and different semantic contexts. Adding ICD-10 bindings to Part1/Part2 could create ambiguity about which coded representation to use. The existing structure has been implemented and deployed.

**Impact:** None. Maintains status quo.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended. The prior 25-0-0 unanimous vote to consider this change for the next ballot version indicates clear workgroup consensus that coded cause-of-death support in Part1/Part2 profiles is desirable. Option 1 (bind without eliminating EAC) strikes the right balance: it makes the profiles more useful for broader exchange scenarios (as noted by FHIR-39470, which was a near-duplicate) while avoiding the breaking change risk of eliminating EntityAxisCauseOfDeath. The binding should use `example` or `preferred` strength to preserve flexibility. This change aligns with the ongoing trend of making VRDR profiles more flexible and reusable (cf. FHIR-43599 removing string limits, FHIR-41314 adding quantity support).
