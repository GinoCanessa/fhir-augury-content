## Description

The profile data elements for Component Building Blocks should have updated PQ-specific description and definition text, where appropriate, in place of the base resource descriptions. This request targets the Pharmaceutical Quality (Industry) IG, asking that the generic FHIR resource element descriptions be replaced with domain-specific language that more accurately reflects pharmaceutical quality concepts.

## Context & Analysis

This ticket is part of a cohort of post-STU1 improvement requests for the Pharmaceutical Quality (Industry) IG, all filed by Robert Hausam in January 2024 during the ballot cycle. During the initial ballot reconciliation (Feb 2024), the workgroup unanimously (15-0-0) deferred this item as "Considered for Future Use," acknowledging its value but deprioritizing it relative to STU1 publication. After STU1 was published, Lynn Laakso re-opened the ticket in May 2024 for consideration in the next cycle.

The companion ticket **FHIR-44042** (adding extensions to the same Building Block Component profiles) was already resolved as **Persuasive** and published, establishing precedent that these profiles are being actively enhanced. Additionally, **FHIR-43811** highlighted that the PQ profiles were sparse — containing little beyond Must Support flags and example value sets — further motivating the need for richer, domain-specific descriptions.

The request is straightforward in concept but may require significant effort: each Component Building Block profile element needs review to determine where generic FHIR descriptions should be replaced with PQ-specific language. The scope of work depends on the number of profiles and elements involved.

## Proposed Dispositions

### Disposition A: Persuasive

**Action:** Accept the request. Update all Component Building Block profile data elements with PQ-specific description and definition text where the base resource descriptions are insufficient or misleading in the pharmaceutical quality context.

**Rationale:** This ticket was deliberately re-opened post STU1 publication — the "future use" scenario envisioned by the original 15-0-0 deferral has arrived. The companion ticket FHIR-44042 (adding extensions to the same profiles) was already resolved as Persuasive, demonstrating the workgroup's commitment to enhancing these profiles. Domain-specific descriptions will improve implementer understanding and reduce ambiguity for pharmaceutical quality use cases. The original unanimous vote signals strong consensus on the value of this change.

### Disposition B: Persuasive with Modification

**Action:** Accept the request but scope the initial work to the highest-priority Component Building Block profiles (e.g., those most actively used by implementers or those flagged in FHIR-43811 as lacking content). Remaining profiles can be addressed in subsequent cycles.

**Rationale:** The overall intent is sound, but the effort to review and update every element across all Component Building Block profiles may be substantial. A phased approach ensures the most impactful profiles are addressed first while distributing the workload across ballot cycles. This also allows the workgroup to establish a consistent pattern for PQ-specific descriptions before applying it broadly.

### Disposition C: Not Persuasive

**Action:** Decline the request. Retain the base resource descriptions in the Component Building Block profiles.

**Rationale:** Base resource descriptions maintain consistency with the core FHIR specification and reduce maintenance burden when new FHIR versions are released. PQ-specific guidance could alternatively be provided in narrative IG pages or usage notes rather than overriding element definitions. However, this disposition conflicts with the workgroup's prior unanimous acknowledgment of the request's value and the precedent set by FHIR-44042.

## Recommendation

**Disposition A: Persuasive**

The workgroup previously voted 15-0-0 to acknowledge this request's value, deferring only for timing reasons. Now that STU1 is published and the companion profile enhancement (FHIR-44042) has been completed, there is no remaining reason to defer. Updating the Component Building Block profiles with PQ-specific description and definition text will directly address the content sparseness identified in FHIR-43811 and improve the IG's utility for pharmaceutical quality implementers. The request aligns with the workgroup's established direction for post-STU1 profile enrichment.
