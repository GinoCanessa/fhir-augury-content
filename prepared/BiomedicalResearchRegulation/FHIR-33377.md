## Description

The submitter asks whether any of the algorithms listed in the IG have been evaluated for accuracy (sensitivity/specificity) so that users can understand the margin of error for automated detection of adverse event cases. The submitter also asks whether there has been a comparison of reported instances to detected instances by the IG.

No existing or proposed wording was provided.

## Proposed Dispositions

### Disposition A: Considered for Future Use (Re-affirm Prior Decision)
**Rationale:** The prior ballot vote (16-0-0) already concluded that sensitivity and specificity cannot currently be evaluated because false negatives are unmeasurable in passive surveillance systems like VAERS and FAERS. The IG focuses on structuring and transmitting adverse event reports, not on validating detection algorithm performance. Positive predictive value could be added in a future iteration once large-scale validation studies are completed. This is an important research question but is outside the scope of the current IG's interoperability mandate.

**Wording Change:** Add a brief informative note to the IG's introduction or algorithm description section stating: *"Algorithm accuracy metrics (e.g., positive predictive value) are planned for inclusion in future versions pending completion of validation studies at scale. Sensitivity and specificity evaluation is limited by the inability to assess false negatives in passive surveillance systems."*

### Disposition B: Persuasive with Modification
**Rationale:** While full sensitivity/specificity analysis remains infeasible, the IG could be improved by documenting known limitations of the automated detection approach and referencing any existing published validation data. This would address the submitter's core concern—that users need to understand what the automated detection can and cannot do—without requiring the IG to conduct new validation research. Adding a "Limitations and Known Accuracy" section would enhance transparency and user trust.

**Wording Change:** Add a new section titled *"Detection Algorithm Limitations"* that describes: (1) why sensitivity/specificity cannot be fully assessed in passive surveillance; (2) any available positive predictive value data from pilot implementations; and (3) a reference to planned future validation work.

### Disposition C: Not Persuasive
**Rationale:** The IG is an interoperability specification for structuring and transmitting ICSR data, not a clinical decision support tool or algorithm validation study. Evaluating algorithm accuracy is the responsibility of implementers and researchers using the IG, not the IG itself. The question, while valid from a public health perspective, asks the specification to solve a research problem that is beyond its scope. The prior ballot already addressed this question and the re-opening was procedural (post-STU1 re-triage), not based on new information.

**Wording Change:** None.

## Recommendation

**Disposition A: Considered for Future Use (Re-affirm Prior Decision)** is recommended.

The prior ballot produced a unanimous vote recognizing that this is a valid concern but one that cannot be addressed in the current iteration due to the inherent limitations of passive surveillance systems. The ticket was re-opened procedurally after STU1 publication, not because of new data or arguments. Re-affirming the prior decision with the addition of a brief informative note about algorithm accuracy limitations and future validation plans would address the submitter's transparency concern while keeping the scope appropriate for an interoperability IG. This approach is consistent with the related ticket FHIR-33375 (root cause analysis of underreporting) being resolved as Not Persuasive, establishing that these broader public health research questions are outside the IG's remit.
