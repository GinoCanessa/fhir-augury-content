## Description

Determine how to refer to pre-defined NCI (National Cancer Institute) and NCI Meta-Thesaurus value sets instead of defining them locally in the IG.

The core issue is that the US ICSR IG uses value sets that originate from the NCI Thesaurus (NCIt) and NCI Meta-Thesaurus (NCIm), and the IG needs a way to reference these externally-maintained value sets rather than redefining them inline.

## Analysis

This ticket addresses a fundamental terminology infrastructure question: how should FHIR IGs reference value sets that are defined and maintained by the NCI rather than redefining them locally?

**Current state of the problem:**
1. The NCI Thesaurus is registered in FHIR (URI: `http://ncicb.nci.nih.gov/xml/owl/EVS/Thesaurus.owl`) but only as a fragment on tx.fhir.org due to size constraints (>50k terms exceeds limits).
2. No standard FHIR-compliant URI pattern exists for NCI-defined value sets.
3. IGs currently either redefine NCI value sets locally or add needed codes piecemeal to the fragment via PRs to FHIR/packages.
4. More recent tickets (FHIR-52157, resolved Persuasive) show progress — the community is now recommending pointing to the NIH NCI EVS Explore for terminology references.

**The ticket remains unresolved after 3+ years**, but the broader ecosystem has made progress on related issues (fragment expansion, EVS Explore links, CDISC terminology references).

## Proposed Dispositions

### Disposition A: Persuasive — Define a standard URI pattern for NCI value sets

Accept the request and define a canonical URI convention for referencing NCI and NCI Meta-Thesaurus value sets in FHIR IGs. Work with NCI/EVS to establish FHIR-compliant value set URLs (e.g., based on the EVS Explore API) and document the pattern in the US ICSR IG. This removes the need to redefine NCI value sets locally and aligns with the direction taken in FHIR-52157.

**Pros:** Provides a clean, reusable solution; aligns with community direction; reduces IG maintenance burden.
**Cons:** Requires coordination with NCI/EVS to agree on URI patterns; may depend on NCI publishing FHIR-native value set resources.

### Disposition B: Persuasive with Modification — Reference NCI EVS Explore and use fragment approach

Accept the request but scope the solution to what is currently feasible: (1) reference NCI value sets via links to the NIH NCI EVS Explore tool, (2) continue using the NCI Thesaurus fragment on tx.fhir.org with codes added as needed, and (3) document the approach for IG authors. Defer formal FHIR-compliant value set URIs until NCI provides native FHIR support.

**Pros:** Immediately actionable with existing infrastructure; follows the pattern already adopted by FHIR-52157.
**Cons:** Does not fully solve the referencability problem; still requires manual code additions to the fragment.

### Disposition C: Not Persuasive — Close as overtaken by events

Decline the request on the basis that subsequent work has addressed the core concern through other mechanisms. FHIR-52157 (Persuasive) established the pattern of pointing to NCI EVS Explore. The NCI Thesaurus fragment approach on tx.fhir.org provides a workable (if imperfect) mechanism. The US ICSR IG can adopt these approaches without a formal resolution on this ticket.

**Pros:** Avoids duplication of effort with already-resolved tickets.
**Cons:** Leaves the US ICSR IG without specific guidance; the fragment approach is a workaround rather than a solution.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. This ticket raises a valid concern that remains unresolved in the US ICSR context, but the full solution (formal NCI value set URIs in FHIR) depends on external coordination with NCI that is beyond the scope of this IG. The pragmatic path forward is to adopt the EVS Explore reference pattern established by FHIR-52157, continue leveraging the NCI Thesaurus fragment on tx.fhir.org, and document the approach for IG authors. The modification scopes the work to what is achievable now while acknowledging the longer-term goal of native NCI FHIR value set support.
