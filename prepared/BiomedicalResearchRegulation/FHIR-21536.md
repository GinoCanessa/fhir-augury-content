## Summary

This ticket requests that the CDM-to-BRIDG mapping tables in the US Common Data Model Harmonization (CDMH) Implementation Guide be represented as FHIR ConceptMap resources. The submitter observes that the existing tabular mappings — where the left-hand column contains a code representing a concept in a common data model (e.g., PCORnet, OMOP, i2b2, Sentinel) and the right-hand column contains the corresponding FHIR element name and profile — are well suited for ConceptMap representation. The ticket also notes a correction: in the OBSERVATION table, the mapping for `unit_concept_id` and `value_as_number` should use `Observation.valueQuantity` rather than the polymorphic `Observation.value[x]`. The resolution was "Persuasive with Modification," indicating agreement to use ConceptMap where applicable, and the implementer subsequently added sample ConceptMap resources to the IG.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Accept the ticket as resolved with modification. Confirm that the CDM-to-BRIDG mapping tables in the CDMH IG should be represented as FHIR ConceptMap resources where applicable, and that `Observation.value[x]` should be replaced with `Observation.valueQuantity` in the OBSERVATION table for `unit_concept_id` and `value_as_number`. The implementer has already added sample ConceptMap resources to the IG, fulfilling the request.
#### Justification
The original request is technically sound — ConceptMap is the appropriate FHIR resource for representing structured mappings between code systems or data models. The correction to use `Observation.valueQuantity` instead of the polymorphic `Observation.value[x]` for quantity-typed fields improves precision. The resolution ("Persuasive with Modification") was applied, and the implementer confirmed that sample ConceptMaps have been added to the IG. The work appears complete.

---

### Disposition B: Accept with Additional Guidance
#### Proposal
Accept the resolution but request that the IG include additional guidance on how to systematically produce ConceptMap resources for all CDM mapping tables (not only samples), and that the ConceptMap resources include explicit `sourceUri` and `targetUri` values referencing the relevant code systems and FHIR StructureDefinition profiles. Ensure all OBSERVATION table entries that reference quantity-typed fields consistently use `Observation.valueQuantity`.
#### Justification
While sample ConceptMaps demonstrate the approach, the IG would benefit from comprehensive, computable ConceptMap resources for all mapping tables, enabling automated validation and tooling support. Explicit source and target URIs improve interoperability and machine-readability. This disposition strengthens the original intent without contradicting it.

---

### Disposition C: Decline
#### Proposal
Decline the request on the basis that the current tabular mapping format is sufficient for the IG's informational purposes and that ConceptMap resources add implementation complexity without clear consumer demand.
#### Justification
ConceptMap resources require maintenance and validation tooling. If the CDMH IG is primarily informational and the mappings are not consumed programmatically, the overhead of producing and maintaining ConceptMap resources may not be warranted. However, this argument is weakened by the fact that the work has already been done and sample ConceptMaps are in the IG.

---

### Recommendation
**Recommended disposition:** A

The request has already been resolved as "Persuasive with Modification" and the implementer has confirmed that sample ConceptMap resources have been added to the IG. The technical approach is sound — ConceptMap is the standard FHIR mechanism for representing mappings between data models, and the correction to use `Observation.valueQuantity` for quantity-typed fields is appropriate. Since the work is complete and aligns with FHIR best practices, accepting the ticket as resolved is the most practical path forward.
