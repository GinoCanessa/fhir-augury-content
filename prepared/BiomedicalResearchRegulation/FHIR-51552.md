## Summary

The submitter identifies that the FHIR to OMOP IG's CodeableConcept mapping pattern page includes a sample SQL INSERT for the OMOP `observation` table that misuses the `qualifier_source_value` column. In the example (mapping a "No known allergy" CodeableConcept), the column is populated with the free text value `'NKA'` to preserve additional context from the FHIR source. However, per the [OMOP CDM v5.4 documentation](https://ohdsi.github.io/CommonDataModel/cdm54.html#observation), `qualifier_source_value` is intended to store the source value for a qualifier concept—not arbitrary metadata or free text tags. Since no `qualifier_concept_id` is populated, the `qualifier_source_value` should be NULL.

The submitter argues that the IG should not encourage implementors to repurpose CDM columns for unintended uses, and notes that full preservation of FHIR source data in OMOP is not achievable without extensions.

## Resolution Description (from Jira)

> This is correct. This usage is not supported by CDM documentation or Themis convention and should not have been prescribed here.
>
> We have removed qualifier_source_value from all SQL examples and field mapping tables where it was used as a metadata/tag column, updated the surrounding narrative accordingly, and added a callout note clarifying its CDM-defined purpose. Individual implementations remain free to use the field as they see fit, but the IG will no longer establish that convention.

---

## Proposed Dispositions

### Disposition A: Accept as Persuasive (affirm current resolution)
**Rationale:** The submitter is factually correct—`qualifier_source_value` in the OMOP CDM is defined to hold the source value corresponding to a `qualifier_concept_id`, not arbitrary metadata or free text. Using it to store `'NKA'` without a corresponding qualifier concept contradicts CDM documentation and Themis conventions. The resolution already applied removes this misuse from all SQL examples and field mapping tables, adds clarifying narrative, and appropriately leaves implementation-specific decisions to individual sites. This is the cleanest path.

### Disposition B: Accept as Persuasive with Modification — add explicit guidance on FHIR data loss
**Rationale:** Accept the correction but additionally require the IG to include a dedicated section or callout box explicitly acknowledging that certain FHIR data elements (e.g., free text, display values without coded equivalents) cannot be faithfully represented in OMOP CDM without extensions, and that implementors should document local strategies for handling data loss. This addresses the submitter's broader concern about the impossibility of full source preservation.

### Disposition C: Reject as Not Persuasive
**Rationale:** One could argue that using `qualifier_source_value` for contextual free text is a pragmatic approach that some implementations already follow, and that the IG should reflect real-world practice rather than strictly enforcing CDM column semantics. However, this position is weak because the CDM documentation is explicit about the column's purpose, and an IG published under HL7 should not contradict the data model it targets.

---

## Recommendation

**Disposition A: Accept as Persuasive (affirm current resolution)**

The applied resolution is well-scoped and correct. The CDM specification is unambiguous about `qualifier_source_value`'s purpose, the Themis conventions do not support the repurposing shown in the examples, and the resolution cleanly removes the offending patterns while preserving implementation freedom. The companion ticket FHIR-51583 addresses the same concern across the broader page. No additional modifications are needed beyond what has already been applied.
