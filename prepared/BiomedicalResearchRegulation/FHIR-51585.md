## Description

The submitter identifies that the IG page provides sample SQL directing implementers to find a substance concept ID via a display-name-based `LIKE` query:

```sql
SELECT concept_id, concept_name, domain_id, vocabulary_id, standard_concept
FROM concept
WHERE concept_name LIKE '%Penicillin G%'
  AND vocabulary_id IN ('RxNorm', 'SNOMED')
  AND standard_concept = 'S';
```

This approach is **fragile and non-deterministic** — dozens of OMOP concepts contain "Penicillin G" in the name, requiring an undocumented clinical judgment call that will produce inconsistent results across implementations.

The submitter proposes replacing this with the OMOP `concept_relationship` table's **"Maps to value"** relationships, which yield deterministic, clinically correct results validated by the OMOP vocabulary team.

## Resolution Description

The workgroup agreed with this correction (Persuasive). The resolution states:

1. The `concept_name LIKE '%Penicillin G%'` query will be **removed**.
2. The page will document a **three-step pattern** using the OMOP `concept_relationship` table:
   - **Step 1:** Resolve the source concept from the FHIR resource's `code.coding[].code` + `code.coding[].system` via `concept_code` and `vocabulary_id`.
   - **Step 2:** Derive `observation_concept_id` via the **"Maps to"** relationship.
   - **Step 3:** Derive `value_as_concept_id` via the **"Maps to value"** relationship.
3. The page will document correct handling of **vocabulary gaps** (`value_as_concept_id = 0` + report to OHDSI).
4. The original query will be retained as a **clearly marked anti-pattern**.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)
**Action:** Accept the existing Persuasive resolution as-is and implement the three-step `concept_relationship`-based pattern as described in the resolution.

**Rationale:** The submitter's concern is technically sound and clinically important. Display-name-based `LIKE` queries are inherently non-deterministic and will produce different results depending on OMOP vocabulary version, producing unreliable clinical data. The "Maps to value" relationship is the canonical OHDSI-endorsed mechanism for deriving `value_as_concept_id`. The resolution's three-step pattern (resolve source concept → derive via "Maps to" → derive via "Maps to value") is well-defined, deterministic, and consistent with OMOP best practices. Retaining the original query as an anti-pattern adds educational value.

### Disposition B: Accept with Extended Scope
**Action:** Accept the Persuasive resolution and additionally audit all other sample SQL in the IG for similar display-name-based lookups, replacing them with relationship-based patterns throughout.

**Rationale:** If the `LIKE` pattern exists in the value-as-concept section, it may appear in other IG pages as well. A broader sweep would ensure consistency across the entire IG and prevent similar fragility in other mapping directions. This aligns with the pattern of SQL quality issues raised by the same reporter in FHIR-51582 and related tickets.

### Disposition C: Accept with Narrower Scope
**Action:** Accept the Persuasive resolution but remove the anti-pattern retention (i.e., do not keep the original `LIKE`-based query in the IG at all).

**Rationale:** Retaining the incorrect query, even marked as an anti-pattern, risks copy-paste adoption by implementers who skip the surrounding context. A cleaner approach would be to simply replace the incorrect SQL with the correct pattern and add a brief text warning against name-based lookups without showing the problematic code.

## Recommendation

**Disposition A: Accept as Persuasive.** The existing resolution is well-crafted — it removes the fragile query, introduces a deterministic three-step vocabulary relationship pattern, documents gap handling, and retains the original as a marked anti-pattern for educational purposes. This balances correctness with implementer guidance. The broader audit suggested in Disposition B is valuable but should be tracked as a separate action item rather than blocking this ticket's resolution.
