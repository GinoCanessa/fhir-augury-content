The submitter questions whether "storage and handling conditions" should be children of the `ProductShelfLife` datatype and proposes renaming it to `StorageConditions`, incorporating `duration` as a child element. The rationale is that the current datatype is too narrowly named for shelf life alone; a broader name like `StorageConditions` would support simple shelf-life cases as well as more complex storage rules (e.g., "do not expose above 5Â°C for more than 50 minutes"). The submitter references the FDA GUDID database specification, SPL characteristics, and GS1 element structures as external models to consider.

## Details

### Reporter's Proposal

1. **Rename** the `ProductShelfLife` datatype to `StorageConditions`.
2. **Move `duration`** inside the renamed datatype as a child element.
3. This supports both simple shelf-life use cases and complex conditional storage rules.

## Proposed Dispositions

### Disposition A: Accept â€” Rename ProductShelfLife to StorageConditions

**Action:** Rename the `ProductShelfLife` datatype to `StorageConditions` and restructure it so that `duration` is a child element, enabling representation of both simple shelf-life and complex conditional storage/handling rules.

**Rationale:** The current `ProductShelfLife` name is misleadingly narrow. The datatype is already used for broader storage/handling context (e.g., in `DeviceDefinition`). Aligning with the FDA GUDID model of "storage and handling" makes the datatype more general-purpose. The restructuring would allow expressions like "do not exceed XÂ°C for more than Y minutes," which is a real-world regulatory requirement not easily represented today.

**Risks:** This is a breaking change to a datatype name, affecting all resources and profiles that reference `ProductShelfLife` (including `MedicinalProductDefinition`, `DeviceDefinition`, etc.). Given that the companion ticket FHIR-28255 was resolved as Not Persuasive, there may be resistance to further refactoring of this datatype. Additionally, the R4â†’R4B transition already caused compatibility concerns with `ProductShelfLife` changes.

### Disposition B: Alternative â€” Extend ProductShelfLife Without Renaming

**Action:** Keep the `ProductShelfLife` datatype name but add optional child elements for storage and handling conditions (e.g., `temperatureRange`, `humidityRange`, `handlingInstructions`) alongside the existing `period`/`type`/`specialPrecautionsForStorage` elements.

**Rationale:** This addresses the functional gap â€” the inability to express complex storage rules â€” without introducing a breaking rename. The existing `specialPrecautionsForStorage` element (a CodeableConcept) already acknowledges that storage conditions go beyond shelf life, but it lacks the structure to express quantitative constraints. Adding structured elements would be additive and non-breaking.

**Risks:** The datatype name would remain somewhat misleading (still called `ProductShelfLife` while covering broader storage concepts). This incremental approach may not fully satisfy the submitter's design goal of a clean, well-scoped abstraction.

### Disposition C: Decline â€” No Change

**Action:** Close as Not Persuasive. The `ProductShelfLife` datatype already includes `specialPrecautionsForStorage` (a repeating CodeableConcept with an example binding to `SpecialPrecautionsForStorage`) which can capture storage condition codes. No structural or naming changes are needed.

**Rationale:** The companion ticket (FHIR-28255) requesting element-level changes to this same datatype was already resolved as Not Persuasive. The existing datatype structure is functional for its intended use in medication and device definition contexts. Renaming a datatype at this stage (post-R4B, targeting R5) carries significant migration cost for limited benefit. The `specialPrecautionsForStorage` element provides an extensible mechanism for capturing storage conditions via coded values.

**Risks:** The datatype name remains semantically narrow relative to its actual use, which may confuse implementers. Complex quantitative storage rules (e.g., temperature-duration constraints) remain difficult to express without extensions.

---

## Recommendation

**Disposition C: Decline (No Change)** is recommended.

The companion ticket FHIR-28255 (same reporter, same datatype, same day) was already resolved as Not Persuasive, establishing a precedent against restructuring this datatype. The `ProductShelfLife` datatype has already been through disruptive changes between R4 and R4B that caused significant backward-compatibility concerns in the community. The existing `specialPrecautionsForStorage` CodeableConcept element provides an extensible mechanism for coded storage conditions. If structured quantitative storage rules are needed in the future, they could be addressed through a focused extension rather than a breaking datatype rename. The workgroup should consider whether the submitter's use case is adequately covered by existing elements before introducing further instability to this datatype.
