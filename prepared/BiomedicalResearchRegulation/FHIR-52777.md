## Summary

Anita Walden identified a gap in the FHIR-to-OMOP IG's Coding Field Mapping Principles page: when a source code resolves to a valid OMOP CONCEPT table entry but that concept has no "Maps to" relationship to any Standard Concept, the IG provides no guidance on what implementers should do. The submitter encountered this with RxNorm codes where the OMOP source concept ID existed but no standard mapping was available, and asked whether it is permissible to navigate other vocabulary relationships (e.g., RxNorm "Has ingredient") to reach a Standard Concept rather than defaulting to `concept_id = 0` (complete loss of meaning).

The ticket was resolved as **Persuasive** with a comprehensive set of changes planned for the Coding Field Mapping Principles page.

---

## Details

### Description (Anita Walden)

The submitter noted that the IG lacked guidance for the scenario where an external concept is present in the OMOP CONCEPT table but is not connected via a "Maps to" relationship. She described a real-world situation with RxNorm codes where a source concept ID existed but no standard mapping was available. The team debated whether to navigate other RxNorm relationships (beyond parent/child) to find the nearest standard code — preserving partial meaning — versus simply entering `0`, which results in complete loss of meaning. She referenced existing language in Section 5.7 about mapping to a parent code and suggested that the Code Prioritization Framework could be expanded to cover relationship navigation beyond parent/child.

### Comment — daverag (2026-02-12)

Daverag acknowledged the gap and outlined six specific changes to the Coding Field Mapping Principles page:

1. **New subsection "Handling Concepts Without Standard Mappings"** in the Code Prioritization Framework — establishes the OHDSI convention of `concept_id = 0` as default, while permitting optional navigation of vocabulary-native relationships (e.g., "Has ingredient," "Is a") under six explicit conditions. Includes a worked RxNorm example and decision logic summary.

2. **Amendment to "Mapping Complexity and Validation"** — adds explicit mention of the scenario where a concept exists in the CONCEPT table but lacks a "Maps to" relationship, with a cross-reference to the new subsection.

3. **Broadened "Multiple FHIR Reference Codings to OMOP"** — extends fallback navigation beyond "closest parent concept" to include lateral and compositional relationships such as "Has ingredient."

4. **New Step 4 in Standard OMOP Vocabulary API Lookup Methodology** — adds deterministic handling for when Step 3 ("Maps to" lookup) returns no result.

5. **New "Reporting Vocabulary Gaps and Missing Mappings" section** — documents the OHDSI community contribution process, forums, GitHub issue tracker, submission guidance, and biannual release timeline.

6. **Concrete RxNorm worked example** — demonstrates a branded drug formulation navigated to its Ingredient-level Standard Concept via "Has ingredient," clarifying what is gained (drug identity) and lost (dose form, strength, brand).

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

Accept the resolution as proposed. The six changes outlined by daverag comprehensively address the submitter's concern by:
- Establishing the default OHDSI convention (`concept_id = 0`) explicitly in the IG.
- Providing constrained, documented guidance for optional vocabulary relationship navigation beyond "Maps to."
- Adding a deterministic failure-path step (Step 4) to the API Lookup Methodology.
- Including a concrete RxNorm worked example that illustrates the trade-offs.
- Adding a section on reporting vocabulary gaps to OHDSI, closing the feedback loop.

This disposition preserves network consistency while giving implementers a principled path to retain partial clinical meaning rather than defaulting to zero information.

### Disposition B: Accept with Modifications (Persuasive with Mod)

Accept the overall direction but request modifications:
- **Tighten the six conditions** for permissible relationship navigation to prevent scope creep — consider requiring work-group-level approval or a published decision matrix for each vocabulary domain (Drug, Condition, Measurement, etc.) rather than leaving the decision entirely to individual implementers.
- **Add a conformance expectation** (e.g., SHOULD vs MAY) to clarify whether relationship navigation is merely permitted or actively encouraged when conditions are met.
- **Require a machine-readable audit trail** (e.g., a ConceptMap or Provenance resource) for any non-standard mapping derived through relationship navigation, not just human-readable documentation.

This would strengthen governance and interoperability while still addressing the submitter's core concern.

### Disposition C: Decline (Not Persuasive)

Decline the change on the grounds that:
- Permitting relationship navigation beyond "Maps to" could undermine OMOP network consistency, which depends on standardized vocabulary mappings maintained centrally by OHDSI.
- The OHDSI biannual vocabulary release cycle is the intended mechanism for resolving missing mappings; interim workarounds, even constrained ones, risk fragmenting the network.
- The IG should instead strengthen the recommendation to report gaps and wait for official resolution, keeping `concept_id = 0` as the sole fallback.

This would preserve strict adherence to OHDSI conventions but leaves implementers without actionable guidance for the interim period.

---

## Recommendation

**Disposition A: Accept (Persuasive)**

The proposed changes are well-researched, comprehensive, and directly responsive to the submitter's concern. The resolution balances clinical utility (preserving partial meaning through constrained relationship navigation) against network consistency (requiring documentation, reporting, and explicit conditions). The addition of a worked RxNorm example, a deterministic API lookup failure path, and a vocabulary-gap reporting section make the guidance immediately actionable. The six explicit conditions for navigation provide sufficient guardrails against ad-hoc workarounds. This resolution will meaningfully improve the IG's utility for real-world ETL implementers encountering missing "Maps to" relationships.
