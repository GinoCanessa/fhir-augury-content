## 2. Problem Statement

The submitter reports that throughout the **CodeableConcept Mapping Patterns** page in the FHIR-to-OMOP IG, the column `qualifier_source_value` is used as a generic text-based tag column to store additional contextual information (e.g., free-text labels, quality flags).

The submitter asserts that:

1. **No CDM documentation or Themis convention supports** using `qualifier_source_value` as a general-purpose tag field.
2. This usage was **not discussed during any WG calls** (to the submitter's knowledge).
3. While individual implementations may choose to use this column for site-specific metadata, the **IG should not establish this as a convention**.

## 3. Resolution Description

> *"We will fix our use of the \*\_source\_value so that they are used properly, i.e. the source of the code or id is placed there."*

The resolution commits to correcting the usage of `*_source_value` columns across the IG so they conform to their intended CDM purpose — storing the original source code or identifier value, not arbitrary metadata tags.

## 8. Proposed Dispositions

### Option A: Accept as Persuasive (Affirm Current Resolution)

**Rationale:** The submitter correctly identifies that the IG's use of `qualifier_source_value` as a generic text tag is inconsistent with CDM documentation and Themis conventions. The resolution to fix `*_source_value` columns to store actual source codes/identifiers aligns with OHDSI best practices. The companion tickets (FHIR-51552, FHIR-51582) reinforce that this is a systemic issue across multiple examples on the CodeableConcept Mapping Patterns page.

**Action:** Confirm the Applied status. Verify that all `*_source_value` columns in IG examples now contain only original source codes or identifiers, and that no examples use them as generic metadata tags.

### Option B: Persuasive with Modification — Add Explicit Guidance

**Rationale:** Beyond fixing the SQL examples, the IG would benefit from explicitly stating the intended purpose of `*_source_value` columns and cautioning against their use as general-purpose tag fields. This would prevent future contributors from reintroducing the same pattern.

**Action:** Accept as Persuasive with Modification. In addition to correcting the examples, add a normative note on the CodeableConcept Mapping Patterns page (or a general conventions section) stating that `*_source_value` columns must be used per CDM/Themis conventions and are not to be repurposed for storing arbitrary metadata.

### Option C: Not Persuasive

**Rationale:** The use of `qualifier_source_value` as a tag could be seen as a pragmatic IG convention to preserve FHIR source richness that would otherwise be lost in OMOP. CDM conventions allow some implementation flexibility, and the IG could document this as a deliberate design choice.

**Action:** Reject the change. Document the current usage as an intentional IG convention with appropriate caveats.

## 9. Recommendation

**Option A: Accept as Persuasive (Affirm Current Resolution)**

The resolution is well-aligned with the submitter's concern and OHDSI/CDM best practices. The three related tickets from the same reporter (FHIR-51552, FHIR-51582, FHIR-51583) collectively demonstrate that `qualifier_source_value` was misused across multiple examples — as a non-existent column (FHIR-51582), as a populated field without a corresponding concept (FHIR-51552), and as a generic tag (this ticket). The resolution to correct `*_source_value` usage across the IG is the appropriate and sufficient fix. The ticket is already Applied, so the WG should verify the changes are complete and move forward.
