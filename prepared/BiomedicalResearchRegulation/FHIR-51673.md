## Description Summary

The submitter identified incorrect FHIR data type usage in the **ConditionEra** logical model within the FHIR-to-OMOP Implementation Guide. Specifically:

- **`condition_era_id`** and **`condition_concept_id`** are typed as `code`, but represent integer identifiers and should use `integer`.
- **`condition_era_end_date`** is typed as `code`, but represents a date value and should use `date`.
- The `code` type should only be used for coded value sets or enumerations.

The submitter references the ConditionEra StructureDefinition profile: [ConditionEra.profile.json.html](https://build.fhir.org/ig/HL7/fhir-omop-ig/StructureDefinition-ConditionEra.profile.json.html)

A follow-up comment from the reporter expanded the scope, noting the same issue exists in other logical models — e.g., **`observation_id`** in the Observation model is also typed as `code` instead of `id`.

## Resolution Description

> The use of code for the end_date is a mistake. We will fix that.
>
> Although we could have used integer for the various concept_id attributes, we wanted to call out that those are special.

The resolution acknowledges the `code` type for date fields is an error and will be corrected, but deliberately retains `code` for concept_id attributes as an intentional design choice to distinguish OMOP concept identifiers.

## Proposed Dispositions

### Disposition A: Affirm (Persuasive with Modification)

**Rationale:** The applied resolution correctly identifies that using `code` for date fields (e.g., `condition_era_end_date`) is a clear error that must be fixed. The decision to retain `code` for concept_id attributes is a defensible design choice — OMOP concept IDs, while integer-valued, reference entries in the OMOP vocabulary system (Athena), and the `code` type signals this semantic distinction. This partial acceptance appropriately balances correctness with intentional modeling decisions. The resolution is already applied.

### Disposition B: Persuasive (Accept Fully)

**Rationale:** Accept the submitter's request in full — change date fields to `date` **and** change identifier fields (condition_era_id, concept_id attributes) to `integer`. The FHIR `code` type is defined for values drawn from a controlled set of strings, not for numeric identifiers. Even though OMOP concept IDs reference a vocabulary, they are integer-valued keys, and typing them as `code` misrepresents their data type. FHIR already provides `Coding` and `CodeableConcept` types for vocabulary-bound values, and `integer` with documentation or extensions could convey the OMOP concept semantics without type misuse. Note: this would reverse the already-applied resolution.

### Disposition C: Not Persuasive

**Rationale:** Reject the change request entirely. The OMOP-to-FHIR IG logical models are representational mappings, not executable FHIR profiles. The use of `code` across all OMOP-sourced fields is a consistent simplification for logical modeling purposes. Changing types selectively introduces inconsistency without meaningful validation benefit, since these logical models are not directly instantiated as FHIR resources. The submitter's concern is more about executable precision than logical representation.

## Recommendation

**Disposition A: Affirm (Persuasive with Modification)**

The applied resolution strikes the right balance. The `code` type for `condition_era_end_date` is objectively wrong — dates should be typed as `date`. This fix is unambiguous. However, the concept_id attributes are a reasonable design judgment: while `integer` would be technically more accurate for the raw value, using `code` signals that these fields carry vocabulary semantics (OMOP concept IDs from Athena). The related ticket FHIR-51675, which pushed harder on the concept_id typing and terminology binding, was resolved as Not Persuasive, reinforcing that the workgroup has deliberately chosen this approach. Affirming the existing resolution maintains consistency with the workgroup's prior decisions.
