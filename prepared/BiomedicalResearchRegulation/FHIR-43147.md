## Description

A number of the extensions defined in the US Structured Product Labeling (SPL) Mapping IG have a context of `Element`, which is too broad for the obviously intended usage of the extensions. The request is to define the extensions with contexts that reflect their actual intended use — i.e., narrow the context to the specific resource types or elements where each extension is expected to be applied.

## Analysis

This is a straightforward conformance/quality request. FHIR extension definitions include a `context` element that specifies where the extension may be used. Setting the context to `Element` (the most abstract base type) effectively allows the extension to appear anywhere in any resource, which is rarely the actual intent. Best practice — and IG publisher guidance — is to constrain extension contexts to the specific resource(s) or element path(s) where the extension is meaningful.

The request requires the SPL Mapping IG authors to review each extension definition in the IG, determine where it is actually used or intended to be used, and update the `context` from `Element` to those narrower paths (e.g., `MedicinalProductDefinition`, `Composition.section`, etc.).

The strong precedent from sibling tickets (4 of 5 resolved as Persuasive) makes this a low-risk acceptance.

## Proposed Dispositions

### Disposition A: Persuasive (Accept as-is)
**Action:** Accept the change request. The SPL Mapping IG team reviews all extension definitions in the IG, identifies those with an overly broad context of `Element`, and narrows each context to the specific resource type(s) or element path(s) where the extension is actually used or intended.

**Rationale:** This aligns with FHIR best practices for extension authoring. Overly broad contexts weaken the ability of validators, tooling, and implementers to correctly constrain and interpret extension usage. Four sibling tickets with the identical request have already been resolved as Persuasive by other work groups, establishing a clear community-wide pattern. No technical risk — narrowing contexts is additive safety.

### Disposition B: Persuasive with Modification
**Action:** Accept the request but scope the initial change to extensions that are demonstrably used in only one or two resource contexts. For extensions whose intended scope is genuinely broad or uncertain, retain the current context and add documentation explaining why.

**Rationale:** Some extensions in the SPL IG may legitimately apply across multiple resource types (e.g., general annotation-style extensions). Forcing narrow contexts on those could break valid usage. A phased approach lets the IG team tighten the clearly scoped extensions now while deferring judgment on ambiguous cases.

### Disposition C: Not Persuasive (Reject)
**Action:** Reject the request. Keep extension contexts as-is.

**Rationale:** The SPL Mapping IG is a specialized mapping IG that may intentionally use broad contexts for flexibility during early development. The IG team may prefer to defer context tightening to a future cycle when the extension usage landscape is more settled.

---

## Recommendation

**Disposition A: Persuasive** — The request is well-established FHIR best practice, carries no technical risk, and has already been accepted by four other work groups for identical sibling tickets. There is no indication that any of the SPL Mapping IG extensions require an intentionally broad `Element` context. The work group should accept this and task the IG authors with narrowing the extension contexts to reflect actual usage.
