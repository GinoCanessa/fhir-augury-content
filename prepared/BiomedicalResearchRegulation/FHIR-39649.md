## Description

Lloyd McKenzie questions the continued existence of the `researchstudy-siterecruitment` extension. The ticket observes that details about a study at a particular "site" are now handled by a new component instance of ResearchStudy — that component captures recruitment information, status, responsible individuals, protocol variations, etc. The implication is that the siterecruitment extension is redundant because the ResearchStudy resource's component structure already covers site-level recruitment details.

## Analysis

The siterecruitment extension was originally created to carry site-level recruitment data on a ResearchStudy. With the introduction of the component model for ResearchStudy (where a child ResearchStudy instance represents a site and natively carries recruitment status, responsible individuals, protocol variations, etc.), the extension became structurally redundant. Both the ticket reporter (Lloyd McKenzie) and the commenter independently identified this redundancy. A community member (Gustav Vella) subsequently confirmed on Zulip that the extension appears deprecated and its reference should be cleaned up from the ResearchStudy build page.

The core question is straightforward: should the siterecruitment extension be formally removed or deprecated, and should its references be cleaned up from the spec?

## Proposed Dispositions

### Disposition A: Persuasive — Remove the Extension and Clean Up References
**Action:** Formally deprecate and remove the `researchstudy-siterecruitment` extension from the FHIR Extensions Pack. Remove all references to it from the ResearchStudy resource page (including the "Extensions" section). Document that site-level recruitment details are handled via component ResearchStudy instances.

**Rationale:** The component ResearchStudy model for sites was specifically designed to replace this kind of extension-based approach. Both the reporter and commenter agree the extension is redundant. The Zulip discussion confirms the community expectation that it should be removed. Leaving a deprecated, unused extension referenced in the spec causes confusion.

### Disposition B: Persuasive with Modification — Deprecate but Retain with Warning
**Action:** Mark the extension as officially deprecated (status = "retired") rather than removing it entirely. Add a deprecation notice pointing users to the component ResearchStudy approach. Remove the extension reference from the ResearchStudy "Extensions" section but keep the extension definition available for backward compatibility.

**Rationale:** Some implementers may have adopted the extension before the component model was available. A soft deprecation preserves backward compatibility while clearly directing new implementations to the component approach.

### Disposition C: Not Persuasive — Retain the Extension
**Action:** Keep the siterecruitment extension as-is. No changes.

**Rationale:** The extension could serve edge cases where a full component ResearchStudy instance is too heavyweight for simple site recruitment tracking. However, no concrete use case for this has been identified.

## Recommendation

**Disposition A: Persuasive — Remove the Extension and Clean Up References**

The evidence is clear: the siterecruitment extension is structurally superseded by the component ResearchStudy model. Both the reporter and commenter agree, and the Zulip community has independently flagged the same issue. The extension adds no unique capability beyond what the component model provides. Removing it and cleaning up references will reduce confusion and align the spec with its current architectural direction.
