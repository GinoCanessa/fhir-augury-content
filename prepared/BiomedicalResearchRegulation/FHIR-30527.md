## Analysis

This ticket asks a straightforward scoping question: why was SDTMIG v3.2 used instead of the newer SDTMIG v3.3? The work group's answer was clear — CDASH v2.1 (the current version at project initiation) is based on SDTMIG v3.2, so the mapping naturally referenced that version. The original ballot vote (13-0-2) resolved this as "Considered for Future Use," acknowledging that future versions of the CDISC Mappings IG could adopt SDTMIG v3.3.

The ticket was re-opened post-STU1 publication by Lynn Laakso and then reverted back to the original disposition. The current status is "Triaged" with resolution "Considered for Future Use," meaning it remains open for future consideration.

The question is whether the work group should now plan to upgrade to SDTMIG 3.3 (or a newer version) in a future cycle, or whether this should remain deferred.

## Proposed Dispositions

### Disposition A: Considered for Future Use (reaffirm original)
**Rationale:** The original rationale remains valid. The CDISC Mappings IG was scoped to CDASH v2.1 / SDTMIG v3.2 at project initiation. Updating to SDTMIG v3.3 (or newer) should be planned as part of a future IG update cycle when resources and project scope allow. The 13-0-2 vote from the 2021-02-23 BR&R meeting already established strong consensus for this disposition.

**Suggested wording:** "The CDISC Mappings IG was scoped to CDASH v2.1 which references SDTMIG v3.2. Updating to SDTMIG v3.3 or a later version will be considered in a future update to the IG when the project scope is expanded."

### Disposition B: Persuasive with Modification
**Rationale:** If the work group is now actively planning a new cycle of the CDISC Mappings IG, they could accept this as persuasive and commit to using a current SDTMIG version (3.3 or later) in the next update. This would create a concrete action item rather than leaving it as a vague future consideration.

**Suggested wording:** "Accepted. The next version of the CDISC Mappings IG will reference the current SDTMIG version at the time of project initiation for that cycle."

### Disposition C: Not Persuasive
**Rationale:** The version choice was a deliberate project scoping decision, not an error. CDASH v2.1 maps to SDTMIG v3.2 by definition, so using v3.2 was correct for the stated scope. The question has already been thoroughly answered.

**Suggested wording:** "The mapping correctly uses SDTMIG v3.2 because CDASH v2.1 is based on that version. The version selection was intentional and appropriate for the project scope."

## Recommendation

**Disposition A: Considered for Future Use** — This reaffirms the original well-supported disposition (13-0-2 vote) and keeps the door open for version updates in future IG cycles without creating an immediate obligation. It accurately reflects the work group's position that version upgrades are desirable but need to be scoped into a future project.
