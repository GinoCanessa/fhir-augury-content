# FHIR-29401: ProductShelfLife - issues with content

## Summary

**Status:** Triaged | **Priority:** Medium | **Type:** Change Request
**Specification:** FHIR Core (FHIR) | **Work Group:** Biomedical Research & Regulation
**Reporter:** Melva Peters | **Assignee:** Unassigned
**Labels:** MedDef, R5
**Created:** 2020-10-21 | **Updated:** 2022-04-28
**URL:** https://jira.hl7.org/browse/FHIR-29401

The reporter identified that the `ProductShelfLife` data type page incorrectly indicates it belongs to the **Pharmacy Work Group** and contains links to "Dosage" and other content that was erroneously copied from the `Dosage` data type. The request is to clean up the page by correcting the owning work group to **BR&R (Biomedical Research & Regulation)** and removing all inappropriate content that does not belong to `ProductShelfLife`.

A comment from **Hugh Glover** at the **2021-01-25 WGM** acknowledged that there are known issues with the build in this area and that the work group would check and fix where necessary.

## Details

The `ProductShelfLife` data type describes the shelf life and storage conditions for a packaged product. It is primarily used in `PackagedProductDefinition`. The issue appears to be a build/template artifact where content from the `Dosage` type was incorrectly included in the `ProductShelfLife` page, including work group attribution and cross-links.

This is a content/metadata cleanup issue rather than a structural or semantic change to the data type itself. The source files for ProductShelfLife reside in the HL7/fhir repository under `source/productshelflife.html`, `source/productshelflife-definitions.html`, and `source/productshelflife-mappings.html`.

## Proposed Dispositions

### Disposition A: Persuasive (Accept and Fix)
**Action:** Correct the ProductShelfLife page to attribute ownership to BR&R and remove all content incorrectly copied from the Dosage data type.

Specifically:
1. Update the work group attribution in `source/productshelflife.html` from Pharmacy to Biomedical Research & Regulation.
2. Remove any links to Dosage-specific content (e.g., Dosage cross-references, Dosage-related descriptions) that were erroneously included.
3. Review the definitions page (`productshelflife-definitions.html`) for any Dosage-derived text and correct it.
4. Verify the fix in the build output to confirm clean rendering.

**Rationale:** The issue is clearly a copy-paste artifact. The content is factually incorrect and misleading. The fix is low-risk and improves specification quality.

### Disposition B: Persuasive with Modification (Accept with Broader Cleanup)
**Action:** Accept the ticket but expand scope to include a comprehensive review of all MedDef data types (ProductShelfLife, MarketingStatus, ProdCharacteristic, SubstanceAmount) for similar copy-paste issues, since they were all added around the same time by the same author.

**Rationale:** If ProductShelfLife had content copied from Dosage, other MedDef data types added in the same commit may have similar issues. A broader sweep would prevent future duplicate tickets. However, this increases scope beyond the original request.

### Disposition C: Not Persuasive (Decline)
**Action:** Close the ticket without changes, noting that the current build may have already corrected these issues through subsequent work on the ProductShelfLife data type.

**Rationale:** The ticket is from 2020 and the WGM comment from 2021 acknowledged the issue. Multiple other ProductShelfLife tickets have been processed since then (FHIR-28255, FHIR-31801, FHIR-35961), and the data type has undergone significant changes through R4B and R5 development. The issue may already be resolved. However, this disposition risks leaving incorrect content in place if the fix was never actually applied.

## Recommendation

**Disposition A: Persuasive** is recommended. This is a straightforward content cleanup issue with no semantic or structural impact on the data type. The fix is low-risk and clearly warranted — the page should accurately reflect BR&R ownership and not contain Dosage-related content. Given that the ticket has been triaged since 2020 with only an acknowledgment at the 2021 WGM, it should be resolved. If the issues have already been fixed in the current build, the ticket can be verified and closed as already addressed; if not, the fix should be applied.
