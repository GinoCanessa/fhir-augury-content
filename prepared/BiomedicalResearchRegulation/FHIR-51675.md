## Summary

This change request raises two concerns about the ConditionEra logical model in the FHIR to OMOP FHIR IG. First, the reporter questions the 0..* cardinality at the top level of ConditionEra, arguing that if the IG is intended to be an executable physical-level application, each instance should represent a single record (1..1) for cleaner validation. Second, the reporter notes that `condition_concept_id` uses a `code` type without a binding to any terminology system (e.g., SNOMED CT or the OMOP concept vocabulary defined in Athena) and requests that either a binding be defined or the element be clarified as an OMOP concept ID integer. A commenter (JIRAUSER16658) further asked how empty CDM tables should be handled under the current cardinality. The ticket was resolved as "Not Persuasive" with the explanation that the 0..* cardinality is a standard artifact of how the FHIR publisher displays logical models (all FHIR resources share this pattern), and that because the logical models represent OMOP tables directly, adding a terminology binding provides no practical value.

## Details

**Description:**

ConditionEra is set to 0..* which is fine for logical modeling, but if we intend this to be an executable physical-level application, then clarify whether each instance is expected to represent a single record (i.e., should be 1..1 per instance for cleaner validation).

Similarly, condition_concept_id is using code without a binding to a terminology (e.g., SNOMED CT, OMOP concept vocabulary). Again, similar to getting to executable substantiation, consider defining a binding to a terminology system or clarify it is an OMOP concept ID (as integer). This should reflect the standard terminology expected on the OMOP side (e.g., those defined in Athena).

See [ConditionEra StructureDefinition](https://build.fhir.org/ig/HL7/fhir-omop-ig/StructureDefinition-ConditionEra.profile.json.html)

**Comments:**

**JIRAUSER16658** (2025-08-07):
For the ConditionEra cardinality, how do we manage situations when a CDM table should be present but can be empty (i.e., no data to populate)?

> The comment included a JSON excerpt from the ConditionEra StructureDefinition showing `"min": 0, "max": "*"` with the definition of Condition Era as a span of time when a Person is assumed to have a given condition, used to aggregate chronic conditions and closely timed doctor visits.

**Resolution Description:**
All logical models (and FHIR resources for that matter) have 0..* as the top-level cardinality. That is just a figment of how the publisher displays those. These logical models are representing the OMOP tables, so there is no binding. A binding to the entire OMOP code system could be added but the resolver does not see value in that.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Constrain the top-level ConditionEra cardinality to 1..1 per instance and add a required terminology binding on `condition_concept_id` to the OMOP Concept vocabulary (as published via Athena), making the logical model executable and self-validating.
#### Justification
If the FHIR to OMOP IG is intended to serve as an executable mapping specification, tighter cardinalities and explicit terminology bindings would enable automated validation of conformance. Specifying that `condition_concept_id` maps to OMOP standard concept IDs (integers from Athena) would remove ambiguity for implementers and tooling authors who consume these logical models programmatically.

---

### Disposition B: Alternative Approach
#### Proposal
Retain the current 0..* cardinality (as it is inherent to how the FHIR publisher renders logical models) but add informative guidance text to the ConditionEra StructureDefinition clarifying that each instance represents a single CDM row. Additionally, add a comment or short description on `condition_concept_id` noting that its values correspond to OMOP standard concept IDs from the Athena vocabulary, without imposing a formal FHIR ValueSet binding.
#### Justification
This approach addresses the reporter's underlying concern—lack of clarity for implementers—without altering the structural representation that all FHIR logical models share. Adding narrative guidance and element-level documentation is lower-risk than changing cardinality or adding a binding to a non-FHIR-native vocabulary system, while still improving usability for those building executable transforms.

---

### Disposition C: Decline
#### Proposal
Decline the request as not persuasive, consistent with the existing resolution.
#### Justification
The 0..* cardinality is a universal characteristic of how the FHIR publisher displays all logical models and FHIR resources; changing it for ConditionEra alone would be inconsistent. The logical models intentionally mirror OMOP CDM tables without imposing FHIR-style terminology bindings, since the OMOP vocabulary system operates differently from FHIR ValueSets. Adding a binding would introduce maintenance burden and potential confusion without meaningful validation benefit, as OMOP concept IDs are integers managed outside FHIR's terminology infrastructure.

---

### Recommendation
**Recommended disposition:** C (Decline)

The resolution rationale provided is technically sound: the 0..* cardinality is an artifact of the FHIR StructureDefinition rendering, not a modeling choice that can be meaningfully changed for a single logical model. All FHIR resources and logical models share this display convention, and altering it for ConditionEra would introduce inconsistency across the IG. Similarly, the OMOP vocabulary system (Athena) is not structured as a FHIR-native terminology, and imposing a FHIR ValueSet binding on logical models that are intentionally representing OMOP tables would conflate two different paradigms without adding practical validation capability. The existing resolution correctly identifies that these are representational artifacts rather than substantive modeling deficiencies. However, if the work group feels the reporter's confusion highlights a documentation gap, Disposition B's suggestion of adding clarifying narrative text could be pursued as a separate, low-priority enhancement without reopening this ticket.
