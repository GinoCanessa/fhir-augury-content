## Description

The submitter requests that Device identification information be included in the Real World Data (RWD) FHIR IG. Specifically, the request is to add appropriate Device profiles to the IG to support retrieval of device-related data for clinical research purposes.

## Analysis

This ticket follows a recurring pattern in the RWD FHIR IG where requests to expand scope were initially deferred as "Considered for Future Use" during the first STU cycle, then re-opened after publication for consideration in subsequent iterations. The IPS Device profile (`Device-uv-ips`) was identified early in the comment thread as a viable foundation, and the incremental, use-case-driven approach of the IG suggests that Device support could be incorporated once a concrete use case requiring device data is identified.

The key question for the work group is whether the current or planned iteration of the RWD IG includes use cases that require device data retrieval (e.g., identifying patients who have been given a certain device, or capturing device information as part of clinical research data collection).

---

## Proposed Dispositions

### Disposition A: Persuasive — Add Device Profile
**Action:** Accept the request and add a Device profile to the RWD IG, potentially based on the IPS Device profile (`Device-uv-ips`).
**Rationale:** Device data is a legitimate component of real-world data for clinical research. Medical device identification is increasingly important for post-market surveillance, registries, and observational studies. The IPS already defines a Device profile that could serve as a starting point.
**Trade-off:** Requires identifying or creating a concrete use case to justify inclusion within the IG's use-case-driven approach. Adds scope to the next iteration.

### Disposition B: Considered for Future Use — Defer Until Use Case Identified
**Action:** Re-apply the original "Considered for Future Use" resolution, acknowledging the need but deferring until a specific use case requiring device data is formally developed.
**Rationale:** The IG's methodology is incremental and use-case-driven. Without a concrete use case motivating device data retrieval, adding a Device profile risks creating an untested, underspecified artifact. The prior 7-0-0 vote reflected consensus that this was not yet needed.
**Trade-off:** Further delays device support. The ticket has already been deferred once and re-opened; repeated deferral may frustrate stakeholders.

### Disposition C: Persuasive with Modification — Commit to Device Use Case in Next Cycle
**Action:** Accept the request in principle and commit to developing a use case involving device data in the next IG cycle, with the Device profile to be added once the use case is defined.
**Rationale:** This acknowledges the validity of the request while maintaining the IG's use-case-driven methodology. It provides a concrete commitment rather than an open-ended deferral, and allows the work group to scope the device-related requirements properly.
**Trade-off:** Commits the work group to future work without fully defining it yet.

## Recommendation

**Disposition C: Persuasive with Modification** — This balances the IG's incremental methodology with the legitimate need for device data in clinical research. Given that the ticket was already deferred once and re-opened post-STU publication, simply deferring again (Disposition B) would not advance the issue. A concrete commitment to develop a device-related use case in the next iteration (Disposition C) respects the IG's use-case-driven approach while giving the submitter a clear path forward. The IPS Device profile provides a ready foundation to build upon once the use case is established.
