# FHIR-55074: RequestOrchestration.intent has no escape valve

## Summary

**Status:** Triaged | **Priority:** Highest | **Type:** Change Request
**Work Group:** Clinical Decision Support | **Specification:** FHIR Core (FHIR)
**Reporter:** Lloyd McKenzie | **Assignee:** Unassigned
**Created:** 2026-01-20 | **Updated:** 2026-03-27
**URL:** https://jira.hl7.org/browse/FHIR-55074

`RequestOrchestration.intent` is a mandatory element with a required binding to the [request-intent](http://hl7.org/fhir/ValueSet/request-intent) value set. The value set currently lacks an `unknown`, `other`, or equivalent escape-valve concept, making the element impossible to populate in some cases (e.g., when the intent is not known or does not map to an existing code). The submitter requests that such an escape-valve concept be added to the value set.

2. **Lloyd McKenzie (2026-03-27):** Confirmed that a generic tracker has been raised against the Request pattern (FHIR-56142) and that consistency across resources would be desirable.

Lloyd McKenzie responded with concrete scenarios: legacy systems that do not distinguish proposal from order, and interconversion from a new FHIR version that introduces additional intent types (e.g., solicit-offer, directive).

## Proposed Dispositions

### Disposition A: Persuasive — Add `unknown` to the request-intent value set

**Action:** Add an `unknown` concept to the `request-intent` value set. This would provide the necessary escape valve for `RequestOrchestration.intent` and all other resources that bind to the same value set (MedicationRequest, CarePlan, ServiceRequest, DeviceRequest, CommunicationRequest, NutritionOrder).

**Rationale:** The FHIR design principle is that mandatory elements with required bindings should include escape-valve concepts to handle real-world data quality gaps. Lloyd McKenzie has identified concrete scenarios where the intent genuinely cannot be determined (legacy system mappings, cross-version interconversion). A pattern-level tracker (FHIR-56142) already exists in FHIR Infrastructure, supporting a coordinated fix. Adding `unknown` is the least disruptive change and follows precedent from other FHIR value sets.

**Trade-off:** Adding `unknown` could mask data quality issues if implementers use it as a convenience rather than reflecting a genuine gap. However, this risk is common to all escape-valve concepts and is accepted elsewhere in FHIR.

### Disposition B: Persuasive — Defer to FHIR-56142 (pattern-level resolution)

**Action:** Mark this ticket as a duplicate of or dependent on FHIR-56142 (Request.intent has no escape valve). The Clinical Decision Support workgroup would note agreement with the request but defer the actual value set change to FHIR Infrastructure for a coordinated, cross-resource resolution.

**Rationale:** Since the request-intent value set is shared across many resources and a pattern-level tracker already exists, a coordinated resolution ensures consistency. The CDS workgroup avoids making a unilateral change that may conflict with decisions made by FHIR Infrastructure or other affected workgroups (Pharmacy, Patient Care, etc.).

**Trade-off:** This defers resolution and does not directly fix the issue for RequestOrchestration. The fix timeline depends on FHIR Infrastructure's prioritization. However, the comments on this ticket explicitly call for consistency, making a coordinated approach appropriate.

### Disposition C: Not Persuasive — No change needed

**Action:** Resolve as Not Persuasive with no change to the value set.

**Rationale:** The sibling RequestOrchestration escape-valve tickets (FHIR-55075 for `action.condition.kind` and FHIR-55076 for `action.relatedAction.relationship`) were both resolved as Not Persuasive. One could argue that if a system is creating a RequestOrchestration, it should know the intent. The existing codes (`proposal`, `plan`, `directive`, `order`, `original-order`, `reflex-order`, `filler-order`, `instance-order`, `option`) cover the recognized categories. Adding `unknown` may weaken data quality expectations.

**Trade-off:** This position ignores the legitimate scenarios identified by Lloyd McKenzie (legacy systems, cross-version mapping) and is inconsistent with the direction of FHIR-56142 and the broader escape-valve review. It also contradicts the comment-thread consensus that a consistent resolution is desirable.

## Recommendation

**Disposition B: Persuasive — Defer to FHIR-56142** is recommended.

The core issue is valid — the request-intent value set lacks an escape valve, and real-world scenarios exist where intent cannot be determined. However, because (1) a dedicated pattern-level tracker (FHIR-56142) already exists in FHIR Infrastructure to address this across all affected resources, (2) both commenters on this ticket explicitly advocate for a consistent, cross-resource resolution, and (3) the value set is shared among many resources owned by different workgroups, the CDS workgroup should express agreement with the request and defer the mechanics of the value set change to FHIR-56142. This ensures a single, coordinated change rather than piecemeal modifications, and aligns with the consensus reflected in the ticket's own comment thread.
