## Description

This change request asks for support for **site-specific configuration of responses** in CDS Hooks within the Clinical Guidelines IG. The use case is writing client-specific extensions in a CDS Hooks response. The proposed mechanism involves:

1. A **`cdsHooksExtension`** extension on RequestGroup to carry site-specific CDS Hooks extensions in the response.
2. Use of **`dynamicValue`** in PlanDefinition to write the `cdsHooksExtension` extension value at apply time.
3. Use of **`moduleConfiguration`** to override the `dynamicValue` of the `cdsHooksExtension`, enabling client-specific customization without changing the PlanDefinition logic itself.

This would allow CDS Hooks service implementers to define a generic PlanDefinition that produces CDS Hooks responses, while allowing individual client sites to configure site-specific extension values through moduleConfiguration overrides.

## Analysis

### Context

This ticket was originally filed in January 2024 as part of the Clinical Guidelines IG work. It was initially resolved as "Considered for Future Use" in July 2024 with a unanimous 20-0-0 vote, with the note that more input was needed and it would be pursued after STU2 publication. Following STU2 publication, it was re-opened in November 2024 and is now tagged Ready-For-Vote.

The core problem is that CDS Hooks responses may need to include **site-specific extensions** (e.g., custom extensions required by a particular EHR vendor or deployment site). Currently, a PlanDefinition author must hardcode these extensions or create separate PlanDefinition instances per client. This ticket proposes a layered configuration approach using existing FHIR Clinical Reasoning infrastructure:

- **PlanDefinition.action.dynamicValue** defines how to compute extension values at apply time
- **moduleConfiguration** (a Clinical Reasoning concept) allows overriding those dynamic values per client/site without modifying the PlanDefinition

This builds on the foundation laid by FHIR-34265 (which added dynamicValue support to RequestGroup/RequestOrchestration actions) and complements FHIR-51662 (which addresses the broader mapping from RequestOrchestration to CDS Hooks responses).

### Open Questions

3. **Implementer feedback:** The original resolution noted "more input is needed." Has additional implementer input been gathered since STU2 publication?

## Proposed Dispositions

### Disposition A: Persuasive

**Rationale:** Accept the change request and add guidance and supporting extensions to the Clinical Guidelines IG for client-specific CDS Hooks response configuration using the `cdsHooksExtension`/`dynamicValue`/`moduleConfiguration` pattern.

- The use case is valid and well-motivated — real-world CDS Hooks deployments (e.g., Da Vinci CRD) already use client-specific extensions.
- The proposed mechanism leverages existing Clinical Reasoning infrastructure (dynamicValue, moduleConfiguration) rather than inventing new patterns.
- FHIR-34265 already laid the groundwork by adding dynamicValue to RequestGroup.
- The previous vote (20-0-0) showed no opposition, only a desire for more input and timing alignment with STU2.
- The ticket is labeled Ready-For-Vote and Highest priority.

**Suggested vote text:** "Add support for client-specific CDS Hooks response configuration in the Clinical Guidelines IG, including guidance on using the cdsHooksExtension extension on RequestGroup/RequestOrchestration, dynamicValue in PlanDefinition for computing extension values, and moduleConfiguration for site-specific overrides."

### Disposition B: Persuasive with Modification

**Rationale:** Accept the change request but require the submitter to provide a more detailed proposal with concrete examples before implementation.

- The use case is valid, but the description is brief and lacks specifics on the structural changes needed.
- Request that Bryn Rhodes (reporter) provide: (a) a worked example showing a PlanDefinition with dynamicValue writing a cdsHooksExtension, (b) an example of moduleConfiguration overriding that value for a specific client, and (c) a clear mapping to CDS Hooks response JSON.
- This addresses the original concern that "more input is needed" while still moving the ticket forward.

**Suggested vote text:** "Persuasive with modification: Accept the change request contingent on the submitter providing detailed examples of the proposed cdsHooksExtension/dynamicValue/moduleConfiguration pattern, including a worked example with PlanDefinition, RequestOrchestration, and the resulting CDS Hooks response JSON."

### Disposition C: Considered – No Change Required

**Rationale:** The existing Clinical Reasoning mechanisms (dynamicValue, extensions) are already sufficient to support this use case without additional IG guidance.

- PlanDefinition already supports dynamicValue on actions.
- RequestOrchestration (formerly RequestGroup) already supports dynamicValue per FHIR-34265.
- Extensions can already be defined and applied to any resource.
- The CDS Hooks configuration mechanism in CRD already addresses client-specific customization at the hooks layer.
- Additional guidance may be premature until more implementer experience is gathered.

**Suggested vote text:** "No change required — existing Clinical Reasoning infrastructure (dynamicValue, extensions, moduleConfiguration) already supports the described use case. Implementers can use these existing mechanisms without additional IG-level guidance at this time."

## Recommendation

**Disposition B: Persuasive with Modification** is recommended.

The use case is clearly valid and addresses a real need in CDS Hooks implementations. The previous unanimous vote (20-0-0) confirms workgroup support for the concept. However, the ticket description remains high-level, and the original deferral explicitly cited the need for "more input." Accepting with a modification requiring concrete examples and detailed structural proposals will ensure the resulting IG guidance is implementable and well-specified. This also aligns with concerns raised in related tickets like FHIR-36134, which highlighted the lack of practical examples for CDS Hooks configuration options.
