## Description

The submitter requests adding verbiage to the order-select hook section recognizing that this hook could increase physician alert fatigue and burden if it triggers ("pings") each time a physician adds an element to the order. The suggestion is to include implementer guidance about being mindful of alert fatigue when building CDS services that respond to order-select.

## Analysis

### Context

The order-select hook in CDS Hooks fires when a clinician selects (adds or modifies) orders during the ordering workflow. Because this hook can trigger repeatedly—each time an order element is added or changed—there is a well-documented concern about alert fatigue. This ticket asks for explicit implementer guidance within the order-select hook section of the CDS Hooks Library specification.

This concern is not new. The same reporter (Celine Lefebvre) previously filed FHIR-36627 arguing that order-sign should be preferred over order-select in the CRD context, specifically citing noise and burden concerns. Multiple other tickets (FHIR-28622, FHIR-22328, FHIR-29597) have addressed clinician burden and alert fatigue from various angles across CDS-related specifications, and several were resolved as Persuasive with Modification—indicating the community recognizes the validity of these concerns.

FHIR-42366 (Published, Persuasive with Modification) clarified the contradictory workflow descriptions for order-select, which is directly relevant because the frequency with which the hook fires is central to this ticket's concern.

Meanwhile, FHIR-51776 represents a countervailing perspective: it argues payers should be required to support order-select to provide earlier CDS engagement. This creates a tension—increasing payer support for order-select could amplify the very alert fatigue this ticket seeks to mitigate, making implementer guidance even more important.

### Key Considerations

1. **Established pattern**: Multiple resolved tickets acknowledge alert fatigue is a legitimate concern for CDS Hooks implementations.
2. **Guidance is non-breaking**: Adding implementer guidance does not change the hook's semantics or break existing implementations—it's purely informational.
3. **Balanced perspective needed**: The guidance should acknowledge that order-select provides valuable early intervention while warning about overuse.
4. **Existing specification language**: The order-select section in CDS Hooks already notes "The hook could fire multiple times" but lacks explicit alert fatigue guidance.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Add targeted implementer guidance to the order-select hook section.**

Add a paragraph to the order-select hook specification section advising implementers to be mindful of alert fatigue when designing CDS services that respond to this hook. The guidance should note that because order-select can fire multiple times during order composition, CDS service implementers should consider strategies such as:
- Limiting the frequency or conditions under which cards are returned
- Avoiding redundant alerts across successive order-select invocations
- Reserving certain alerts for the order-sign hook when the order is more complete

This addresses the submitter's concern directly, is consistent with resolutions on related tickets (FHIR-28622, FHIR-22328, FHIR-29597), and does not alter the hook's semantics. The modification qualifier acknowledges the specific wording will need to be drafted by the work group.

### Disposition B: Persuasive
**Accept the request as stated.**

Add verbiage to the order-select hook section acknowledging that this hook could increase physician alert fatigue if it fires each time a physician adds an element. Include a general statement that implementers should be mindful of this concern. This accepts the submitter's exact suggestion without further elaboration, deferring specific guidance details to future work.

### Disposition C: Not Persuasive
**Decline the request—existing specification text is sufficient.**

The order-select hook section already notes it can fire multiple times and that services should not warn about insufficient information. Alert fatigue is an implementation concern rather than a specification concern, and adding subjective guidance about implementer behavior could set an unwanted precedent. Implementers are expected to exercise professional judgment without explicit specification-level warnings.

## Recommendation

**Disposition A: Persuasive with Modification**

This is the strongest option because it addresses a genuine and well-documented concern (alert fatigue from repeated hook invocations) while providing actionable guidance rather than a vague admonition. Multiple related tickets have already been resolved favorably on the same theme, establishing community consensus that alert fatigue guidance belongs in the specification. The "with Modification" qualifier appropriately reserves the exact wording for work group deliberation while committing to the addition of substantive guidance. Given the growing push for payer support of order-select (FHIR-51776), proactively including alert fatigue guidance becomes even more important to balance the specification.
