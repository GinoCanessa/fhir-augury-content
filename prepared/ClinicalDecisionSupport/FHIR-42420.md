## Description

The submitter requests that the CDS Hooks Library specification add documentation showing which implementations (CDS Clients and CDS Services) support each hook, as evidence for the maturity level assigned to that hook. The CDS Hooks Maturity Model (http://cds-hooks.hl7.org/2.0/#hook-maturity-model) bases maturity levels on the number of client and service implementations. The ticket asks that this implementation evidence be explicitly documented alongside each hook's maturity level.

## Analysis

This ticket addresses a transparency and governance gap in the CDS Hooks maturity process. The CDS Hooks Maturity Model defines maturity levels based on implementation counts (e.g., maturity 4 requires "at least 2 CDS Clients and 3 independent CDS Services in production"), but the specification does not currently document which implementations satisfy those requirements. This was concretely demonstrated when FHIR-42045 challenged the maturity levels of appointment-book and encounter-discharge — there was no documentation to verify the claims, and the levels had to be lowered.

Adding implementation documentation would:
1. Provide verifiable evidence for each hook's maturity level
2. Allow the community to challenge or confirm maturity assessments
3. Prevent future disputes about whether maturity criteria have been met
4. Help implementers understand the adoption landscape

The main implementation question is **where and how** to document this information — whether inline in the hook specification pages, in a separate section, in the CDS Hooks wiki, or via external references (e.g., links to connectathon results, production attestations).

---

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Accept the request and add an implementation evidence section to each hook's page in the CDS Hooks Library specification. For each hook, document the known CDS Client and CDS Service implementations (with links where possible) that justify the current maturity level. Establish a template or convention for how this evidence is recorded going forward, so future maturity level changes include updated implementation documentation.

**Rationale:** This directly addresses a demonstrated governance gap. The Zulip discussion and FHIR-42045 showed that maturity levels were challenged because no documentation existed. Adding this is low-effort, high-value, and aligns with the maturity model's own criteria.

### Disposition B: Persuasive with Modification
**Action:** Accept the intent but instead of embedding implementation documentation directly in the spec, maintain it in the [CDS Hooks wiki](https://github.com/cds-hooks/docs/wiki) (which already contains some implementation information, e.g., "CDS-Hooks-Availability"). Add a reference from each hook's spec page to the wiki page. This keeps the specification focused on normative content while providing a living document for implementation evidence.

**Rationale:** The specification ballot cycle is slow; implementation evidence changes more frequently as new systems adopt hooks. A wiki-based approach allows community-maintained updates between ballot cycles while still linking from the spec.

### Disposition C: Not Persuasive
**Action:** Decline the request. The maturity model defines the criteria, and the CDS WG already tracks implementation evidence internally. Embedding this in the specification adds maintenance burden and risks becoming stale.

**Rationale:** The maturity model is a process tool for the WG, and the WG can maintain its own records. However, this contradicts the community's demonstrated need for transparency (per the Zulip thread and FHIR-42045).

---

## Recommendation

**Disposition A: Persuasive** — The ticket identifies a real and demonstrated gap. The community has already experienced confusion and disputes (FHIR-42045, Zulip discussion) due to the absence of documented implementation evidence supporting maturity claims. Adding this documentation directly to the CDS Hooks Library specification is the most transparent and authoritative approach. The CDS WG should define a standard format for recording implementation attestations alongside each hook and require updated evidence when maturity levels are proposed for change.
