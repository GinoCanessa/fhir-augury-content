## Description

The CQL specification currently states that "the value for the default of a parameter must be able to be calculated at compile-time." However, this language does not adequately clarify that the compile-time value must also be the same as the value that would be computed at run-time.

The specific concern involves functions like `Today()`, `Now()`, and `TimeOfDay()`. While these can technically be evaluated at compile-time, their values would generally differ from what would be computed at run-time (since the date/time changes between compilation and execution). The ticket requests that the specification language be tightened to exclude such time-dependent functions from parameter default values.

## Comment Analysis

Two comments provide substantive input:

1. **JIRAUSER23970 (2023-08-14):** Argues against fixing `Now()` at compile-time because the CQL spec requires `Now()` to remain constant only during a single execution period — not across compilations. Fixing it at compile-time would break rolling-window logic (e.g., "encounters during the last 2 days") and make ELM artifact distribution impractical. Proposes clarifying that "compile-time" means expressions whose value can be calculated at compile-time **and** would remain constant at runtime, thereby forbidding `Now()`, `Today()`, and `TimeOfDay()` in parameter defaults.

2. **JIRAUSER25154 (2023-10-31):** Suggests aligning with prior art from other programming languages (e.g., C# constants) and restricting parameter default values to **literal expressions only**, which would provide a clear, well-understood boundary.

## Proposed Dispositions

### Disposition A: Persuasive — Clarify with "constant at runtime" language

**Resolution:** Accept the ticket as a technical correction. Update the CQL specification to clarify that parameter default values must be expressions that can be evaluated at compile-time **and** whose value would be the same if evaluated at run-time. Explicitly state that time-dependent functions (`Now()`, `Today()`, `TimeOfDay()`) are not permitted as parameter default values because their values change between compile-time and run-time.

**Rationale:** This directly addresses the reported ambiguity with minimal disruption. It preserves the existing allowance for computed expressions (e.g., arithmetic on literals) while closing the loophole for time-varying functions. This aligns with the commenter's (JIRAUSER23970) suggested approach and the original reporter's intent.

### Disposition B: Persuasive — Restrict to literal expressions only

**Resolution:** Accept the ticket and tighten the requirement further: parameter default values must be **literal expressions only** (i.e., values directly representable as CQL literals — numbers, strings, dates, date/times, intervals of literals, etc.). No function calls or computed expressions would be permitted.

**Rationale:** This is the stricter approach suggested by JIRAUSER25154, aligned with how most mainstream programming languages handle constants. It eliminates all ambiguity about what constitutes a valid default value, at the cost of disallowing default values that involve simple computation (e.g., `1 + 1`). This may be overly restrictive for CQL's use cases.

### Disposition C: Not Persuasive — Current language is sufficient

**Resolution:** Close with no change. The existing language ("must be able to be calculated at compile-time") is deemed sufficient because implementers can reasonably infer that time-dependent functions violate the compile-time requirement in practice.

**Rationale:** Implementers who read the specification in context with the broader CQL execution model (where `Now()` is constant per execution, not per compilation) should understand the intent. However, this disposition risks continued ambiguity and divergent implementation behavior.

## Recommendation

**Disposition A: Persuasive — Clarify with "constant at runtime" language.**

This disposition directly resolves the reported ambiguity with precise, narrowly-scoped language. It forbids `Now()`, `Today()`, and `TimeOfDay()` in parameter defaults while still permitting other compile-time-computable expressions (e.g., computed intervals from literal date values). This strikes the best balance between clarity and flexibility, aligns with the reporter's intent and the most substantive comment, and is the least disruptive change to existing CQL content. The related ticket FHIR-48682 (parameter constraints) has already been applied, indicating active specification evolution in this area — clarifying parameter default semantics now will complement that work.
