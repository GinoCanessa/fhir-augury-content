## Description

The ticket requests adding support for using **ObservationDefinition** in CPG (Clinical Practice Guidelines) activities within the PlanDefinition and ActivityDefinition patterns. Currently, `PlanDefinition.action.definition` allows references to ActivityDefinition, PlanDefinition, Questionnaire, SpecimenDefinition, ObservationDefinition, MessageDefinition, and uri — but the behavior when referencing an ObservationDefinition is not explicitly specified.

The reporter (Bryn Rhodes) wants the CPG IG to formally define how ObservationDefinition should be used in activity definitions, enabling clinical guidelines to directly express observation-taking activities (e.g., "take blood pressure").

## Analysis

### Core Question

Should CPG profiles explicitly support ObservationDefinition as a `PlanDefinition.action.definition` target to express observation-taking activities directly, or should these use cases continue to be modeled through ServiceRequest/ActivityDefinition (or Task-based) patterns?

### Key Tensions

1. **FHIR-49988 conflict**: The already-applied FHIR-49988 (Persuasive) proposes *restricting* `PlanDefinition.action.definition` to only ActivityDefinition and PlanDefinition in FHIR Core, with other resources (including ObservationDefinition) handled via Task. If FHIR-49988 is implemented in a future FHIR release, adding ObservationDefinition support in CPG would diverge from core FHIR direction.

2. **Existing pattern adequacy**: The workgroup previously noted (20-0-0 vote) that taking an observation can already be expressed via ServiceRequest + ActivityDefinition. The question is whether a *direct* ObservationDefinition reference provides enough additional value to justify the complexity.

3. **WHO/SMART Guidelines demand**: WHO has concrete use cases (ANC DT.04 physical exams) where directly referencing ObservationDefinition would be more natural and expressive than wrapping it in a ServiceRequest/ActivityDefinition.

4. **Post-STU2 timing**: The ticket was deliberately re-opened after STU2 publication, signaling ongoing interest from the workgroup.

## Proposed Dispositions

### Disposition A: Persuasive — Define ObservationDefinition Activity Pattern in CPG

Accept the request and define how ObservationDefinition should be used within CPG activity patterns. Create a CPG profile or guidance that specifies the semantics when `PlanDefinition.action.definition` references an ObservationDefinition (e.g., it means "collect this observation"). Document the $apply behavior for ObservationDefinition targets.

- **Rationale**: WHO SMART Guidelines has demonstrated concrete use cases. Direct ObservationDefinition references are more expressive for observation-taking activities than indirect ServiceRequest wrapping. The CPG IG can define this pattern even if FHIR Core eventually restricts the base element (CPG can maintain its own profiles).
- **Risk**: May conflict with FHIR-49988 direction if Core restricts the base element. Would need to reconcile with the Task-based approach proposed there.
- **Vote text**: *"Define CPG-specific activity patterns for ObservationDefinition as a PlanDefinition.action.definition target, including $apply semantics, to support use cases such as WHO SMART Guidelines observation-taking activities."*

### Disposition B: Considered for Future Use — Defer Pending FHIR-49988 Resolution

Acknowledge the use case but defer until the FHIR-49988 changes are implemented and the Task-based approach for non-ActivityDefinition/PlanDefinition resources is fully specified. Re-evaluate whether ObservationDefinition support is still needed after those patterns are available.

- **Rationale**: FHIR-49988 (already Applied/Persuasive) is moving Core toward restricting `PlanDefinition.action.definition`. It would be premature to build CPG patterns around ObservationDefinition as a direct definition target if Core will remove that option. Better to wait and see if the Task-based alternative adequately addresses the WHO use cases.
- **Risk**: Delays WHO/SMART Guidelines implementers who need this capability now. May lose momentum if deferred again.
- **Vote text**: *"Defer until FHIR-49988 changes are implemented and the Task-based approach for ObservationDefinition is available. Re-evaluate whether direct ObservationDefinition support in CPG is still needed at that point."*

### Disposition C: Not Persuasive — Existing Patterns Are Sufficient

Reject the request. The existing ServiceRequest + ActivityDefinition pattern adequately covers observation-taking activities. An ActivityDefinition can already reference ObservationDefinition via `observationRequirement` and `observationResultRequirement` elements, providing the needed linkage without expanding `PlanDefinition.action.definition` semantics.

- **Rationale**: The 20-0-0 prior vote already established that the workgroup sees the existing pattern as sufficient. FHIR-49988 further reinforces that non-ActivityDefinition/PlanDefinition resources should not be direct definition targets. Adding complexity for marginal expressiveness gain is not justified.
- **Risk**: WHO/SMART Guidelines implementers may find the indirect pattern cumbersome and create non-standard workarounds.
- **Vote text**: *"The existing ServiceRequest/ActivityDefinition pattern with observationRequirement/observationResultRequirement elements adequately supports observation-taking activities in clinical guidelines. No change needed."*

---

## Recommendation

**Disposition B: Considered for Future Use** is recommended.

The fundamental tension is that FHIR-49988 (already Applied as Persuasive) is moving FHIR Core in the opposite direction — toward restricting `PlanDefinition.action.definition` to only ActivityDefinition and PlanDefinition. Building CPG-specific ObservationDefinition patterns on an element that Core intends to constrain would create a fragile foundation. However, the WHO use cases are legitimate and the ticket was deliberately re-opened, so outright rejection is premature. Deferring until FHIR-49988's Task-based alternative is specified allows the workgroup to make an informed decision about whether the Task approach satisfies the need or whether CPG-specific extensions are warranted.
