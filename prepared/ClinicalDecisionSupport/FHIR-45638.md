## Summary

The `cqf-knowledgeRepresentationLevel` extension references a code system and value set defined in FHIR R5. When CRMI shareable profiles (targeting R4) include this extension, it creates a broken cross-version dependency — the R5 terminology cannot be resolved from an R4 IG context. For ballot, CRMI duplicated the terminology locally. For publication, the `knowledgeRepresentationLevel` slice was removed from all shareable profiles as a temporary workaround. The ticket requests restoring the slice by moving the terminology into the FHIR Extensions Pack, mirroring the successful approach already used for the `knowledgeCapability` code system and value set.

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification (move terminology to Extensions Pack)
Accept the approved resolution. Move the `knowledgeRepresentationLevel` code system and value set into the FHIR Extensions Pack, then restore the `knowledgeRepresentationLevel` slice on all CRMI shareable profiles. This mirrors the proven approach already used for `knowledgeCapability` (as noted by Bryn Rhodes in the Zulip discussion). It eliminates the cross-version broken link without requiring new cross-version tooling. **Impact:** Requires changes to both the Extensions Pack (add terminology artifacts) and CRMI (restore slices on all shareable profiles).

### Disposition B: Defer pending cross-version resolution infrastructure
Defer this ticket until the FHIR cross-version dependency chain and publication tooling can natively support references from R4 IGs to R5 terminology. This would avoid duplicating terminology across packages but leaves the `knowledgeRepresentationLevel` slice absent from shareable profiles indefinitely. **Impact:** No immediate changes needed, but shareable profiles remain incomplete and implementers cannot formally declare knowledge representation levels. Depends on Grahame Grieve's cross-version tooling work, which has no committed timeline.

### Disposition C: Duplicate terminology within CRMI itself
Instead of moving terminology to the Extensions Pack, re-include the `knowledgeRepresentationLevel` code system and value set directly in the CRMI IG (the approach used during ballot). This keeps CRMI self-contained but creates terminology duplication across packages and was explicitly rejected during publication preparation. **Impact:** Simpler change scope (CRMI only) but introduces maintenance risk from duplicated definitions and conflicts with the direction established for `knowledgeCapability`.

---

## Recommendation

**Disposition A: Accept as Persuasive with Modification (move terminology to Extensions Pack)**

This is the recommended disposition. The resolution is well-grounded: it follows the exact pattern already successfully applied for `knowledgeCapability`, where the terminology was placed in the Extensions Pack so it could be referenced from any FHIR version context. The Zulip discussion confirms Bryn Rhodes (the reporter) identified this as the natural solution, and the approach was validated through the `knowledgeCapability` precedent. Moving the terminology to the Extensions Pack is the cleanest fix — it restores the missing slice on shareable profiles, avoids terminology duplication, and does not depend on unreleased cross-version tooling. The related tickets (FHIR-42942, FHIR-32636, FHIR-32634) show an established trajectory of refining these knowledge extensions across profiles, and this change is a logical continuation of that work.
