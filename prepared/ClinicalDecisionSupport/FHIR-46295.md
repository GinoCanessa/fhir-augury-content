## Description

The submitter proposes recommending that all resources related to testing a particular artifact be tagged with the canonical reference to that artifact. Specifically:

- At a minimum, a `Meta.tag` with a code that is the canonical reference (possibly versioned) to the artifact under test.
- Alternatively, consider an extension. The existing `cqf-testArtifact` extension identifies a test artifact but is currently only usable on Library and Group resources.

The core problem is that when test data for measures or other canonical artifacts is loaded onto a FHIR server, there is no standard way to identify which resources belong to a specific artifact's test suite, making it difficult to query, manage, or delete test data sets.

## Comment Discussion (3 comments)

1. **René Spronk (2024-07-04):** Advocates for `Meta.tag` because it is searchable on FHIR servers (e.g., "GET all example resources associated with a certain test or measure") and supports deletion of test data sets. Notes that extensions do not support searching or deletion.

2. **Bryn Rhodes (2024-07-10):** Raised during a CRMI call — the concern with `Meta.tag` is that since it is only a `Coding`, there is no way to distinguish that the canonical reference in the tag means "test data for this artifact" vs. some other meaning. Asked for ideas on how to differentiate.

3. **René Spronk (2024-07-10):** Suggested using a dedicated CodeSystem URI as the `system` in the tag's `Coding` to indicate "measure this resource serves as test data for." Also proposed an alternative approach using the `List` resource: `List.subject` = the Measure, `List.entry[*].item` = test resources, `List.code` = "test data". This supports `List.subject` search. Reiterated that extensions are unattractive due to lack of search support.

## Analysis

**Core Issue:** There is no standardized mechanism in CRMI to associate test data resources with the canonical artifact they test, which hampers test data lifecycle management (query, update, delete) on FHIR servers.

**Key Design Tension:** The discussion reveals a trade-off between three approaches:

1. **`Meta.tag` approach** — Searchable and supports bulk operations. However, since `tag` is only a `Coding` (system + code), it requires defining a specific CodeSystem to distinguish "test data for artifact X" from other uses of the same canonical URL in tags. This is workable but somewhat unconventional.

2. **Extension approach (`cqf-testArtifact`)** — Semantically precise (an extension explicitly means "this is test data for artifact X"). However, extensions are not searchable via standard FHIR search parameters, making server-side management of test data impractical. Currently limited to Library and Group.

3. **`List` resource approach** — A `List` with `subject` pointing to the Measure and `entry` items referencing test resources. Searchable via `List.subject`. However, requires maintaining a separate List resource and does not tag the individual test resources themselves.

**Maturity:** The ticket is in Triaged status with active discussion but no resolution. The conversation stalled after the July 2024 exchange and no formal proposal has been selected.

## Proposed Dispositions

### Disposition A: Persuasive — Recommend Meta.tag with a defined CodeSystem
**Action:** Add guidance in CRMI recommending that test data resources be tagged using `Meta.tag` with:
- `system` = a newly defined CodeSystem URI (e.g., `http://hl7.org/fhir/uv/crmi/CodeSystem/test-artifact-tag`) indicating "test data for artifact"
- `code` = the canonical URL (optionally versioned) of the artifact under test

**Rationale:** This addresses the searchability requirement (servers can search by `_tag`), supports bulk operations (query + delete), and uses an existing FHIR mechanism. The dedicated CodeSystem resolves Bryn's concern about distinguishing the purpose of the tag. This is the approach most aligned with the Zulip discussion consensus.

**Trade-off:** Requires defining and maintaining a new CodeSystem. The tag value (a canonical URL as a code) is somewhat unconventional but valid.

### Disposition B: Persuasive with Modification — Recommend both Meta.tag and List
**Action:** Recommend a two-pronged approach:
1. Tag individual test resources with `Meta.tag` (as in Disposition A) for searchability and deletion.
2. Additionally recommend grouping test cases in a `List` resource (`List.subject` = artifact, `List.code` = "test data") for structured organization and navigation.

**Rationale:** Combines the strengths of both approaches — `Meta.tag` for server-side resource management, `List` for structured grouping and discovery. This gives implementers the most flexibility while providing clear guidance.

**Trade-off:** More complex guidance; implementers must maintain both tags and List resources, increasing implementation burden.

### Disposition C: Not Persuasive — Defer until cqf-testArtifact extension is broadened
**Action:** Close as not persuasive, noting that the existing `cqf-testArtifact` extension (from FHIR-41874) should be expanded to cover additional resource types rather than introducing a parallel tagging mechanism. Revisit once FHIR search capabilities for extensions improve.

**Rationale:** The `cqf-testArtifact` extension already provides the semantic meaning needed. Adding a tagging convention creates a second mechanism for the same purpose. Broadening the extension's applicability (beyond Library and Group) and potentially adding search parameter support would be a cleaner solution.

**Trade-off:** Leaves the practical problem unsolved in the near term, since custom extensions are not searchable via standard FHIR search. This defers the problem rather than solving it.

## Recommendation

**Disposition A: Persuasive — Recommend Meta.tag with a defined CodeSystem**

This disposition best addresses the practical need identified in the originating Zulip discussion: the ability to search for, manage, and delete test data on FHIR servers. The `Meta.tag` approach uses existing FHIR infrastructure (`_tag` search), is immediately implementable by servers, and the dedicated CodeSystem resolves the semantic ambiguity concern raised by Bryn Rhodes. It is the lightest-weight solution that directly solves the stated problem. The List approach (Disposition B) could be added as supplementary guidance later if structured grouping is needed, but is not essential for the core use case.
