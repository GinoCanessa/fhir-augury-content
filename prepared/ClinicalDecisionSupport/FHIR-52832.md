## Summary

This ticket requests the creation of a new **CQLExpressionEvaluationService** capability statement within the "Using CQL With FHIR" implementation guide. Currently, the IG defines a single CQL Evaluation Service capability statement that covers both the `$cql` operation (ad-hoc CQL expression evaluation) and the `Library/$evaluate` operation (full library-based evaluation). The submitter (Bryn Rhodes) identifies that there is a useful, simpler subset of this capability — a service that supports **only** the `$cql` operation without needing to implement `Library/$evaluate`. Defining a separate capability statement for this subset would allow implementers to declare conformance to a lighter-weight CQL expression evaluation service.

---

## Details

### Description

> The current implementation guide defines a CQL Evaluation Service that supports `$cql` and `Library/$evaluate`, but there is a useful subset of this capability that is just the CQL Expression Evaluation Service, that only supports the `$cql` operation.

## Proposed Dispositions

### Disposition A: Accept — Define CQLExpressionEvaluationService capability statement (Persuasive)

Accept the request as stated. Add a new **CQLExpressionEvaluationService** CapabilityStatement to the "Using CQL With FHIR" IG that declares conformance to only the `$cql` operation (system-level `POST /$cql`), without requiring support for `Library/$evaluate`.

**Specific changes:**
1. Create a new CapabilityStatement resource `CQLExpressionEvaluationService` in the IG, modeled after the existing `CQLEvaluationService` but including only the `$cql` operation definition.
2. The new capability statement should declare `rest.resource` and `rest.operation` entries for only the `$cql` operation at the system level.
3. Update the IG narrative to describe the relationship between the full `CQLEvaluationService` (supports `$cql` + `Library/$evaluate`) and the new `CQLExpressionEvaluationService` (supports `$cql` only).
4. Clarify that a server conforming to `CQLEvaluationService` implicitly satisfies `CQLExpressionEvaluationService`.

**Rationale:** This is a clean decomposition of existing capabilities. Some implementers may want to offer ad-hoc CQL expression evaluation without the overhead of library management and evaluation infrastructure. The `$cql` operation is self-contained (it accepts inline CQL text), whereas `Library/$evaluate` requires library resolution, versioning, and dependency management.

### Disposition B: Alternative — Define a hierarchy of CQL service capability levels (Persuasive with Modification)

Instead of defining just one additional capability statement, define a tiered set of CQL service capability levels within the IG:

1. **CQLExpressionEvaluationService** — supports only the `$cql` operation for ad-hoc expression evaluation.
2. **CQLLibraryEvaluationService** — supports only `Library/$evaluate` for pre-authored library evaluation.
3. **CQLEvaluationService** — the existing full service, supports both `$cql` and `Library/$evaluate` (inherits from both above).

**Specific changes:**
1. Create two new CapabilityStatement resources (`CQLExpressionEvaluationService` and `CQLLibraryEvaluationService`).
2. Refactor the existing `CQLEvaluationService` to reference or compose the two sub-capabilities.
3. Incorporate relevant capability declaration guidance from FHIR-35921 (CQL versioning and capability representation) where applicable, since that ticket was not persuasive at the FHIR Core level but may be appropriate within the Using CQL With FHIR IG.

**Rationale:** Provides maximum flexibility for implementers to declare exactly which capabilities they support. However, this is a broader scope change than requested and should be validated against actual implementation patterns.

### Disposition C: Decline — Defer to implementer-defined capability statements (Not Persuasive)

Decline the request. The existing `CQLEvaluationService` capability statement is sufficient. Implementers who only support `$cql` can create their own CapabilityStatement or simply not declare conformance to the full service. Adding granular capability statements to the IG adds maintenance burden without clear interoperability benefit.

**Rationale:** FHIR-35921 (which also addressed CQL capability representation in CapabilityStatements) was resolved as Not Persuasive. There may not be sufficient implementation demand for a standardized expression-only service definition. Servers already have flexibility to declare partial conformance.

---

## Recommendation

**Disposition A (Accept)** is recommended.

The request is well-scoped, low-risk, and aligns with a clear architectural distinction. The `$cql` operation and `Library/$evaluate` operation serve different use cases: `$cql` is for lightweight, ad-hoc expression evaluation (accepting inline CQL text), while `Library/$evaluate` requires infrastructure for library storage, resolution, and dependency management. Defining a `CQLExpressionEvaluationService` capability statement allows implementers to declare conformance to the simpler use case without implying they support the more complex library evaluation workflow. This is a straightforward addition to the IG that parallels how FHIR capability statements are used elsewhere to distinguish service levels. The reporter (Bryn Rhodes) is the primary author of the CQL specification and Using CQL With FHIR IG, lending strong domain authority to the request.
