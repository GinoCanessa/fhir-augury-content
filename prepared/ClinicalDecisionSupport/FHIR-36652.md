## Description

The submitter asks for clarification on the semantic difference between two approaches to triggering a Questionnaire from a PlanDefinition:

1. **Collect Information ActivityDefinition** — An ActivityDefinition with kind `collect-information` generates a Task that references a Questionnaire. The implication is that the questionnaire may be completed later (deferred/asynchronous).
2. **Questionnaire directly in `PlanDefinition.action.definitionCanonical`** — A Questionnaire referenced directly in the `definitionCanonical` element. The implication is that the questionnaire should be completed immediately (synchronous).

The submitter asks whether this distinction (deferred vs. immediate) is the correct interpretation for handling Questionnaires in PlanDefinitions.

## Analysis

This ticket raises a legitimate question about the semantic implications of how a Questionnaire is referenced from a PlanDefinition. At the time of filing (2022), the spec was ambiguous about what should happen when a Questionnaire appeared directly in `definitionCanonical` versus being referenced through a collect-information ActivityDefinition.

**Key development since filing:** FHIR-49988 (filed 2025-03, resolved as Persuasive, now Applied) directly addresses this ambiguity. It restricts `definitionCanonical` to only ActivityDefinition and PlanDefinition, and documents that other resource types (including Questionnaire) should be handled through a Task. This means a Questionnaire should always be reached via a collect-information ActivityDefinition that produces a Task — the "direct Questionnaire in definitionCanonical" pattern will no longer be valid.

This effectively answers the question in FHIR-36652: there is no longer a distinction to make, because only the ActivityDefinition/Task path is permitted.

## Proposed Dispositions

### Disposition A: Persuasive — Add Clarifying Documentation
**Action:** Add narrative text to the Clinical Guidelines IG explaining the distinction between immediate and deferred execution of Questionnaires via PlanDefinition actions, and reference the FHIR-49988 constraint.
**Rationale:** Even though FHIR-49988 restricts the allowed types, implementers would benefit from explicit documentation explaining *why* Questionnaires are handled via Tasks and clarifying the execution semantics.

### Disposition B: Not Persuasive — Addressed by FHIR-49988
**Action:** Close with no change, noting that the underlying ambiguity has been resolved by FHIR-49988 (which restricts `definitionCanonical` to ActivityDefinition and PlanDefinition and documents that Questionnaires should be handled via Tasks).
**Rationale:** The applied change in FHIR-49988 removes the ambiguous scenario entirely. The Questionnaire-directly-in-definitionCanonical pattern is no longer valid, so there is no distinction to clarify.

### Disposition C: Considered — Question Answered
**Action:** Close as "Considered - Question Answered" with a response confirming that, per FHIR-49988, Questionnaires should be handled through a collect-information ActivityDefinition that creates a Task. The direct use of Questionnaire in `definitionCanonical` is no longer supported.
**Rationale:** The ticket was framed as a question. The answer now exists in the spec via FHIR-49988. Formally answering the question and closing provides a clear record.
