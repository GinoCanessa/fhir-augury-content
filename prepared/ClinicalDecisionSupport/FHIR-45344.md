## Summary

This ticket requests that standard `useContext` slices be added to **all Publishable profiles** defined in the Canonical Resource Management Infrastructure (CRMI) IG. The goal is to facilitate artifact discovery and searching by establishing a common, well-defined set of metadata categories across all publishable canonical resources.

The original request proposed three categories (focus, setting, age range). The resolution expanded this to **seven standard useContext slices**: gender, age, focus, user, workflow, task, and venue — each with specific cardinality (0..*) and extensible bindings to established value sets.

## Analysis

### Context
The CRMI IG defines Publishable profiles for canonical resources (Measure, Library, PlanDefinition, ActivityDefinition, Questionnaire, etc.). These profiles standardize how published artifacts are described. Currently, there is no consistent set of useContext slices across these profiles, making it difficult to search and filter artifacts by common categories like clinical focus, patient demographics, or care setting.

### Impact
- **Scope**: All Publishable profiles in CRMI (potentially 10+ profiles covering different canonical resource types).
- **Alignment with ecosystem**: The resolution aligns with existing patterns already adopted in downstream IGs (e.g., QM's qualityProgram slice in FHIR-43320, eCR's focus slice in FHIR-36776, QM's venue slice in FHIR-48533). Standardizing these at the CRMI level creates consistency.
- **FHIR Core alignment**: FHIR-48493 is pursuing a similar approach in FHIR Core (binding valueCodeableConcept to specific code domains). This ticket's resolution at the profile level is complementary.
- **Binding strength**: All bindings are extensible, allowing IGs to extend with additional codes while maintaining interoperability for the standard categories.

### Risks & Considerations
1. **Downstream IG impact**: IGs already defining their own useContext slices (e.g., QM, eCR) will need to reconcile their existing slices with the new standard set. The extensible binding strength minimizes breaking changes.
2. **Value set currency**: The focus binding references the R4 URL (`https://hl7.org/fhir/R4/valueset-condition-code.html`). The implementation should use the version-independent URI for forward compatibility.
3. **Overlap with FHIR-48533**: The venue slice proposed here and the venue slice in FHIR-48533 need to be coordinated to avoid conflicting definitions.

## Proposed Dispositions

### Disposition A: Apply as Resolved (Recommended)
**Action:** Implement all seven useContext slices (gender, age, focus, user, workflow, task, venue) across all Publishable profiles in CRMI as specified in the resolution.

**Rationale:** The resolution is well-defined, aligns with established patterns in multiple downstream IGs, and addresses a real gap in artifact discoverability. The seven categories map directly to the standard `usage-context-type` code system. The extensible bindings provide flexibility while maintaining a baseline for interoperability. Multiple related tickets (FHIR-43320, FHIR-48533, FHIR-36776) have already implemented similar patterns in individual IGs, validating the approach. Standardizing these at the CRMI level is the logical consolidation step.

### Disposition B: Apply with Modification — Phased Approach
**Action:** Implement the three originally requested slices (focus, age, setting/venue) first. Defer gender, user, workflow, and task to a follow-up ballot cycle after gathering implementer feedback.

**Rationale:** The original request identified three high-priority categories. Adding all seven at once increases the implementation burden for profile maintainers and downstream IGs. A phased approach validates the pattern with the most commonly needed slices before expanding. However, this delays full standardization and may cause additional reconciliation work later.

### Disposition C: Defer Pending FHIR Core Resolution
**Action:** Defer implementation until FHIR-48493 (useContext slicing with paired bindings in FHIR Core) is applied, then build the Publishable profile slices on top of the Core-level pattern.

**Rationale:** FHIR-48493 proposes the same code-to-value-binding pairing at the FHIR Core level. Implementing at the CRMI level first could create divergence if Core adopts a different structural approach. However, FHIR-48493 is still in "Resolved - change required" status and may take time to apply. Waiting could significantly delay CRMI improvements, and the extensible-binding approach used here is unlikely to conflict with whatever Core decides.

---

## Recommendation

**Disposition A: Apply as Resolved.** The resolution is specific, well-justified, and aligns with established ecosystem patterns. The seven useContext slices cover the standard categories already defined in the `usage-context-type` code system, and downstream IGs have proven the pattern works. Deferring for Core alignment (Disposition C) is unnecessary given the extensible binding approach, and phasing (Disposition B) adds complexity without proportional benefit.
