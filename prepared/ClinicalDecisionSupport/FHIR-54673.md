## Summary

This ticket requests a general-purpose mechanism in CDS Hooks for indicating which version of an implementation guide (e.g., CRD) a CDS service supports, and for clients to specify which version they wish to invoke. Currently, CRD handles this need via proprietary extensions (`davinci-crd.version` on the discovery response and `davinci-crd.requestedVersion` on the CDS Hooks request). The submitter argues this is a cross-cutting concern that would benefit from a standardized solution in CDS Hooks itself rather than requiring each IG to define its own extensions.

## Details

### Description

In CRD, a need has been identified to communicate to payers (who may support multiple CRD versions with differing expectations for allowed response types) what versions they support, and for clients to communicate what version they wish to invoke. This requirement is currently handled with CRD-specific extensions, but the submitter believes it is not CRD-specific and would benefit from a general-purpose solution in the CDS Hooks specification.

### Comment Discussion

**bryn.rhodes (2026-02-25):** Additionally, there should be the ability for a server to advertise its discovery endpoint via a FHIR CapabilityStatement, and for clients to advertise what hooks they support.

**isaac.vetter (2026-03-05):** Referenced FHIR-28684 as related. Detailed the CRD extension definitions:
- `davinci-crd.version`: Appears in each services.hook declaration in the discovery response. CRD servers SHALL declare at least one supported CRD version per hook.
- `davinci-crd.requestedVersion`: Appears in the extensions object at the root of a CDS Hooks request. SHALL be present if the service supports multiple versions for that hook.

Isaac noted that decorating the discovery response with version metadata is straightforward, but dynamically changing service behavior based on a requested version increases complexity. A more straightforward approach would be for each service definition to declare its supported version and create new services for new versions, reducing risk to existing integrations.

**isaac.vetter (2026-03-18):** Noted that the resolution of FHIR-28650 already added a service "version" element to the discovery endpoint as part of STU3. Currently, the version lives within per-service documentation, meaning a single base URL could host services of multiple CDS Hooks versions. Isaac suggested it would be cleaner for a single CDS Hooks service base URL to always be a single version (similar to how FHIR does it). Also clarified that the second feature request (a mandatory version element in the request) is not needed because the balloted version element holds only a single version — the expectation is that new versions get new services or are backward-compatible. A clarification of this could be helpful.

## Proposed Dispositions

### Disposition A: Accept — Extend the CDS Hooks spec with a standard IG-version mechanism

Accept this change request and define a general-purpose mechanism in CDS Hooks for indicating implementation guide versions. This would involve:
1. Adding an optional `igVersion` or `guideVersion` extension point in the discovery response per service, allowing services to declare which IG version(s) they conform to.
2. Defining a corresponding standard extension in the CDS Hooks request for clients to specify a requested IG version.
3. Documenting guidance that services supporting multiple IG versions SHOULD expose separate service entries per version (consistent with isaac.vetter's recommendation).

**Rationale:** This generalizes what CRD has already implemented with proprietary extensions, enabling any IG built on CDS Hooks to leverage a standard versioning mechanism without reinventing it. Aligns with the existing `version` element (for CDS Hooks spec version) added by FHIR-28650 but addresses the orthogonal need of IG-level versioning.

### Disposition B: Partially Address — Clarify existing version element usage and recommend service-per-version pattern

Rather than adding new elements, clarify in the CDS Hooks specification that:
1. The existing `version` element in the discovery response can be used alongside IG-specific metadata.
2. The recommended pattern is for services to expose separate service definitions for each supported IG version (i.e., one service per version) rather than dynamically switching behavior based on a request-time version parameter.
3. Add non-normative guidance on how IGs should handle version negotiation using extensions if the service-per-version pattern is insufficient.

**Rationale:** This leverages the infrastructure already in place from FHIR-28650's resolution without adding new specification elements. Isaac Vetter's comment supports this simpler approach, arguing that creating new services for new versions reduces implementation complexity and risk. However, it does not fully solve the general-purpose problem — IGs would still need their own extensions for version declaration.

### Disposition C: Decline — Defer to IG-specific extensions

Decline this request on the basis that version negotiation at the IG level is inherently IG-specific and best handled by individual implementation guides via the existing CDS Hooks extension mechanism. The CDS Hooks specification already supports:
- The `version` element (per FHIR-28650) for CDS Hooks spec-level versioning.
- Extension points for IGs to add custom metadata like CRD's `davinci-crd.version`.

**Rationale:** Each IG may have different versioning semantics, and a one-size-fits-all approach may not accommodate the diversity of use cases. CRD has demonstrated that the extension mechanism is workable. Adding general-purpose IG-version elements to CDS Hooks could create complexity for implementers who don't need multi-version support.

## Recommendation

**Disposition B: Partially Address — Clarify existing version element usage and recommend service-per-version pattern.**

This is the most pragmatic approach. The STU3 `version` element (from FHIR-28650) already addresses CDS Hooks spec-level versioning, and isaac.vetter's analysis in the comments strongly supports the service-per-version pattern over dynamic version switching. Adding clarifying guidance and a recommended pattern costs little but provides real value to IG authors. It avoids over-engineering a general-purpose solution before more IGs demonstrate the need, while still giving CRD and other IGs clear direction. If multiple IGs later converge on the same extension pattern, that empirical evidence would justify a future ticket to standardize it (Disposition A). Bryn Rhodes's suggestion about CapabilityStatement advertising could also be addressed as a separate, complementary enhancement.
