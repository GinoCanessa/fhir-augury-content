## Summary

The reporter requests that the `$clone` operation be updated to include an input parameter for specifying the URL of the newly created cloned resource. Currently, the `$clone` operation creates a new resource with the same contents as the source artifact but does not provide a mechanism to supply a different URL for the clone. Since canonical resources are identified by their URL, the cloned resource must have a distinct URL to avoid conflicts. A commenter (mitrep9g) further notes that because `$clone` recursively clones composed child artifacts, those child resources also need new URLs, requiring either additional input parameters or a convention for generating new URLs for the entire composition tree.

## Details

**Description:**

The new cloned resource created by the `$clone` operation needs to have a `url` element that has a different value than the resource being cloned, but the operation as currently defined does not provide an input parameter to provide the new url. Add an input parameter to add the url for the new resource created by the `$clone` operation.

**Comments:**

**mitrep9g** (2024-07-16):
Since the operation also clones all resources it is composed of, all of these other resources also need new urls passed in to the operation, or some way to handle the new urls.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Add a `url` input parameter (type `uri`, cardinality `1..1`) to the `$clone` OperationDefinition. When supplied, the server sets the `url` element of the newly created root resource to the provided value. For recursively cloned child artifacts, define either an additional repeating `childUrl` parameter (allowing callers to supply url mappings for composed resources) or specify a server-side convention for deriving child URLs from the root URL (e.g., by appending a suffix or namespace transformation).
#### Justification
This directly addresses the reporter's request and the commenter's concern about child resources. The `$clone` operation's purpose is to create a distinct copy of a canonical resource tree, and without explicit URL assignment the operation cannot produce valid, uniquely-identifiable resources. Adding explicit URL parameters gives callers full control and avoids ambiguity. This approach is consistent with how other CRMI operations (e.g., `$draft`, `$release`) accept identifying parameters, and aligns with the pattern established in FHIR-39489 which described $clone as creating a resource "with the new URL."

---

### Disposition B: Alternative Approach
#### Proposal
Add a `url` input parameter (type `uri`, cardinality `1..1`) for the root resource only, and define a normative server-side URL derivation convention for child artifacts. The convention would specify that child artifact URLs are constructed by replacing the base portion of the original root URL with the new root URL in all composed child artifacts. Document this convention in the operation's narrative description and add an output parameter listing the generated child URL mappings so callers can discover the new URLs.
#### Justification
This simplifies the caller's burden by requiring only a single URL input rather than a potentially complex mapping of URLs for an entire composition tree. It addresses the commenter's concern about child resources without requiring callers to know the full dependency graph in advance. The output mapping ensures transparency and allows callers to update any external references. This is more practical for deeply nested composition hierarchies where enumerating all child URLs would be cumbersome.

---

### Disposition C: Decline
#### Proposal
Decline the request on the grounds that the `$clone` operation's parameter design should be deferred to the CRMI IG rather than addressed at the base specification level. The current description in FHIR-39489 already implies a new URL is part of the clone semantics; the OperationDefinition for `$clone` has not yet been formally published, and parameter design should be handled as part of that formal definition effort rather than as an incremental patch.
#### Justification
The `$clone` operation is still maturing within the CRMI specification. FHIR-43943 already identified that many CRMI operations, including Clone, lack formal OperationDefinitions. Addressing the URL parameter in isolation risks creating an incomplete or inconsistent design before the full operation semantics are finalized. A holistic approach to defining the `$clone` OperationDefinition — including URL handling, child resource strategies, status management, and versioning — would produce a more coherent result.

---

### Recommendation

**Recommended disposition:** A

Disposition A is recommended because the need for a `url` parameter on `$clone` is well-established and non-controversial. The operation's core purpose — creating a distinct copy of a canonical resource — fundamentally requires that the clone have a different URL, and the current omission of this parameter is a clear gap in the operation's definition. The reporter's request is straightforward, the commenter's observation about child resources is valid and should be addressed concurrently, and precedent from FHIR-39489 already described the intended behavior as creating a resource "with the new URL." Adding explicit parameters (for both root and child resources) gives implementers clarity and aligns with the pattern of parameter refinements being applied across other CRMI operations (FHIR-46634, FHIR-48382, FHIR-48383). While Disposition B's server-side derivation approach has merit for simplicity, the additional complexity of specifying a URL derivation convention may introduce its own ambiguities and implementation variance. Accepting the request as stated allows the work group to define the parameter structure as part of the formal `$clone` OperationDefinition, which is already needed per FHIR-43943.
