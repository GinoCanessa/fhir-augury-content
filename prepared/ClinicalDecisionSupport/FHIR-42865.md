## Summary

This change request asks that the CQL (Clinical Quality Language) ModelInfo be enhanced to support automatic terminology filtering through CodeableReference-typed elements during retrieve operations. Currently, when a FHIR element uses CodeableReference (e.g., `MedicationRequest.medication`), CQL authors cannot use a simple retrieve with a terminology filter like `[MedicationRequest: "Aspirin"]` and have it automatically resolve against both the inline CodeableConcept path and the referenced resource's code. Authors are forced to manually write complex union queries that handle both paths. The proposal is for the CQL translator to automatically expand such retrieves.

## Details

**Problem:** FHIR's CodeableReference type allows a concept to be represented either inline as a CodeableConcept or by reference to another resource that carries the code. When CQL authors write a retrieve with a terminology filter against an element of this type, the retrieve can only filter on the direct code-valued path. It cannot "push through" the reference to filter on the referenced resource's code element.

**Example:** For `MedicationRequest.medication` (a CodeableReference to Medication), the simple CQL:

```cql
define TestChoiceMedicationRequest:
  [MedicationRequest: "Aspirin"]
```

should ideally be expanded by the translator into:

```cql
define TestChoiceMedicationRequestExpanded:
  [MedicationRequest: medication in "Aspirin"]
   union (
     [MedicationRequest] MR
      with [Medication] M
        such that M.id = Last(Split(MR.medication.reference, '/'))
          and M.code in "Aspirin"
  )
```

**Proposed Solution:** Enhance the CQL ModelInfo to be aware of CodeableReference patterns so the translator can automatically generate the expanded retrieve, relieving authors from managing this complexity manually.

**Impact:** This change would affect the CQL specification's ModelInfo definition for FHIR, the CQL-to-ELM translator, and potentially CQL engine implementations that need to support the expanded retrieve pattern. It simplifies CQL authoring for any resource using CodeableReference.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

**Action:** Accept the change request and enhance the CQL ModelInfo to natively support CodeableReference-typed elements so the CQL translator can automatically expand terminology-filtered retrieves into unions covering both the inline CodeableConcept and referenced resource paths.

**Rationale:** This is a genuine gap in the CQL-FHIR integration. The CodeableReference pattern is increasingly common in FHIR R5+ and will appear more frequently as resources migrate from choice types to CodeableReference. Requiring authors to manually write union expansions is error-prone and creates a barrier to correct CQL authoring. The translator already handles choice-type expansion for simpler cases; extending this to CodeableReference is a natural evolution. The reporter (Bryn Rhodes) is the primary CQL specification author and is well-positioned to assess feasibility.

**Specification Changes:**
- Update CQL ModelInfo for FHIR to annotate CodeableReference-typed elements with metadata linking the reference target type and its code property.
- Update CQL-to-ELM translator logic to generate the expanded retrieve pattern when a terminology filter targets a CodeableReference element.
- Document the expansion pattern and its behavior in the Using CQL with FHIR IG.

### Disposition B: Accept with Modification (Persuasive with Modification)

**Action:** Accept the concept but scope the initial implementation to a well-defined set of known CodeableReference elements (e.g., `MedicationRequest.medication`, `ServiceRequest.code`) and require an explicit opt-in annotation in ModelInfo rather than automatic expansion for all CodeableReference elements.

**Rationale:** Automatic expansion for all CodeableReference elements could have unintended performance consequences — each expanded retrieve generates additional queries against referenced resources. A phased approach allows the community to validate the pattern on high-value use cases (medication-related resources are the most common) before generalizing. This also gives CQL engine implementers time to optimize the join/union pattern.

**Specification Changes:**
- Add a new ModelInfo annotation (e.g., `codeableReferenceExpansion`) that identifies which elements support automatic terminology filter expansion.
- Populate the annotation for well-known elements in the FHIR ModelInfo.
- Document the annotation and the expansion behavior, including performance considerations.

### Disposition C: Decline (Not Persuasive)

**Action:** Decline the change request. CQL authors should continue to write explicit union queries when they need to filter across CodeableReference paths.

**Rationale:** The current CQL language is fully capable of expressing the expanded retrieve pattern; the issue is convenience, not capability. Adding implicit expansion to the translator increases complexity in both the translator and CQL engines, may introduce subtle behavioral differences across implementations, and could mask performance-critical decisions (each expansion implies an additional resource query). Authors writing production CQL for quality measures and CDS already handle complex data access patterns. Providing guidance or a reusable CQL library for the common medication case may be sufficient.

---

## Recommendation

**Disposition B: Accept with Modification (Persuasive with Modification)**

The underlying need is valid — CodeableReference expansion is a real authoring pain point that will grow as FHIR R5+ adoption increases. However, automatic expansion for every CodeableReference element introduces performance risk and implementation complexity that should be managed incrementally. Starting with an explicit opt-in annotation for high-value elements (particularly medication-related resources) lets the community validate the approach, gives engine implementers a bounded scope, and preserves author control over when the expansion occurs. This balances improving the authoring experience with practical implementation concerns.
