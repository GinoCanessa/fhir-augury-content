## Description

This ticket requests adding mappings between CDS Hooks 1.1 features and FHIR Clinical Reasoning resources. Specifically:

- **`isRecommended`** → `PlanDefinition.precheckBehavior` — Maps the CDS Hooks suggestion-level `isRecommended` flag to the existing FHIR element that indicates whether an action should be pre-checked (pre-selected) for the user.
- **`card.topic`** → `PlanDefinition.topic` / `ActivityDefinition.topic` — Maps the CDS Hooks card topic concept to existing FHIR definition-level topic elements. The ticket notes a gap: there is no Request-level equivalent, and suggests a possible request-extension for topic may be needed.

## History & Context

This ticket was originally resolved as **Persuasive** on 2021-03-24 (vote: Abdullah Rafiqi/Kensaku Kawamoto: 16-0-0) with an expanded scope covering five CDS Hooks 1.1 features:

1. System actions
2. Feedback support and override reasons
3. `isRecommended` and `topic` elements
4. Permit `UserId` as `PractitionerRole`
5. Add `Coding` type to simplify usage from object-oriented languages

That resolution was **reverted** on 2025-11-05, and the ticket was moved to the CDS-IG specification (label `CDS-IG`). A new vote was recorded: Gail Winters/Chris Schuler: 20-0-0.

The reversion and move to CDS-IG suggests the work group determined this mapping work belongs in the CDS Hooks IG rather than FHIR Core, and the ticket needs to be re-triaged under that context.

## Key Considerations

1. **Deprecation conflict**: The `PlanDefinition.topic` and `ActivityDefinition.topic` elements — which are proposed mapping targets for `card.topic` — are being deprecated/removed in R6 (FHIR-55564, FHIR-55443, FHIR-55431). The work group must decide whether to establish alternative mapping targets before approving this work.

2. **Scope expansion**: The previously reverted Persuasive resolution had a broader scope (5 CDS Hooks 1.1 features) than the original ticket description (2 features). The work group should clarify the intended scope for the CDS-IG.

3. **Specification move**: The ticket has been moved from FHIR Core to CDS-IG, which is appropriate since these mappings are specific to the CDS Hooks integration guide.

4. **CDS Hooks version alignment**: CDS Hooks has progressed from 1.1 to STU2 since this ticket was filed. The `Coding` type change for `card.source.topic` (FHIR-38317) should be considered when defining mappings.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Recommendation:** Accept the request but narrow the scope to `isRecommended` → `PlanDefinition.precheckBehavior` mapping only in the CDS-IG. Defer the `card.topic` mapping until the R6 deprecation of `PlanDefinition.topic` and `ActivityDefinition.topic` is resolved (FHIR-55564). The CDS-IG should document the `isRecommended` mapping and note the `topic` mapping as pending resolution of the deprecation question. Also update to reference CDS Hooks STU2 rather than 1.1.

- **Impact:** Compatible, substantive
- **Category:** Enhancement

### Disposition B: Persuasive
**Recommendation:** Accept the full scope from the previously reverted resolution (all 5 CDS Hooks 1.1/STU2 features) and implement the mappings in the CDS-IG. For `card.topic`, define new mapping targets (e.g., request-level extensions for topic) since `PlanDefinition.topic` and `ActivityDefinition.topic` are being deprecated. This aligns with the original ticket's suggestion of a "request-extension for topic."

- **Impact:** Compatible, substantive
- **Category:** Enhancement

### Disposition C: Not Persuasive
**Recommendation:** The ticket is too outdated to act on as-is. CDS Hooks has moved from 1.1 to STU2, the mapping target elements (`PlanDefinition.topic`, `ActivityDefinition.topic`) are being deprecated in R6, and the specification has been moved to CDS-IG. Request a new, comprehensive ticket filed against CDS-IG that accounts for the current state of both CDS Hooks STU2 and FHIR R6.

- **Impact:** N/A
- **Category:** N/A

## Recommendation

**Disposition A: Persuasive with Modification.** The `isRecommended` → `PlanDefinition.precheckBehavior` mapping is straightforward and non-controversial. The `card.topic` mapping is complicated by the R6 deprecation of both proposed target elements, making it premature to finalize without resolving that conflict. Splitting the work — implementing the clear mapping now and deferring the ambiguous one — is the most pragmatic path forward. The scope should also be updated to reference CDS Hooks STU2 rather than 1.1.
