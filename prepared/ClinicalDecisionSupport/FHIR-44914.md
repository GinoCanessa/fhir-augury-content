## Description

> Add support for commonly used case feature patterns

The ticket requests adding support for commonly used case feature patterns in the Clinical Guidelines (CPG) Implementation Guide. The description is terse — it does not enumerate which specific patterns are intended, nor does it provide examples or normative text.

## Keywords

`case feature`, `patterns`, `CPG`, `Clinical Guidelines`, `StructureDefinition`, `case feature definition`, `case feature group`

## Context & Background

The CPG IG already defines a framework for case features, established through a family of related tickets from the same reporter (Bryn Rhodes) that have all been resolved as **Persuasive** and **Published**:

- **[FHIR-23954](https://jira.hl7.org/browse/FHIR-23954)** — *Add profiles to support case features* (Published, Persuasive): Established the foundational case feature framework including:
  - `cpg-casefeaturedefinition` — a profile of StructureDefinition with optional formal computability semantics, supporting patient data primitives, assertions, inferences, and enrichments
  - `cpg-casefeature` — resource-specific profiles (or an extension) indicating instances of case feature definitions
  - `cpg-casefeaturegroupdefinition` — a profile of GraphDefinition specifying grouped case features
  - `cpg-casefeaturegroup` — a Bundle realized from the GraphDefinition
  - Note: case features represent **patient state** only, explicitly excluding planned/proposed actions (strategy/workflow state)

- **[FHIR-23957](https://jira.hl7.org/browse/FHIR-23957)** — *Define a cpg-casesummary* (Published, Persuasive): Defined case summary as a set of feature groups/features that completely represent case data for a particular pathway.

- **[FHIR-23958](https://jira.hl7.org/browse/FHIR-23958)** — *Add a case plan note* (Published, Persuasive): Added `cpg-caseplannote` profile of Composition tying to a specific case with features, clinician assessment, and planned actions.

- **[FHIR-23961](https://jira.hl7.org/browse/FHIR-23961)** — *Clarify the relationship to a case report* (Published, Persuasive): Discussed how case data and case features can generate a case report at end of pathway, aligned with eCaseReporting IG.

- **[FHIR-51859](https://jira.hl7.org/browse/FHIR-51859)** — *Define GraphDefinition/$apply* (Triaged): A newer related ticket proposing `GraphDefinition/$apply` to walk case feature relationships and produce FHIR Document Bundles. This ticket explicitly builds on case feature concepts.

A comment on FHIR-23954 by Bryn Rhodes notes that "representation of data elements as case features may vary across different functions" — decision support → ordering → documentation → quality measure — which provides motivation for standardized patterns.

## Analysis

This ticket is a continuation of the case feature work in the CPG IG. While the foundational profiles were added through FHIR-23954, this ticket appears to request **pre-built, commonly reused case feature patterns** — likely standard definitions for frequently encountered clinical data types (e.g., vital signs observations, conditions, medications, lab results) that implementers can adopt rather than building from scratch.

**Strengths:**
- Builds logically on already-published case feature infrastructure
- Filed by the original architect of the case feature framework (Bryn Rhodes)
- Addresses a practical need — without standard patterns, every implementer must define their own case features for common clinical data
- The "Fit-For-Teleconference" label indicates the work group considers it appropriate for discussion

**Weaknesses:**
- The description is extremely minimal — no specifics on which patterns, how many, or what form they should take
- No comments, no attached design documents, no linked PRs or prototypes
- No Zulip or community discussion to indicate broader demand or consensus
- The ticket has been open since March 2024 with no apparent progress

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Recommendation: Accept the intent but require the submitter to enumerate the specific case feature patterns to be added.**

Accept the change request in principle since standardized case feature patterns are a logical and valuable extension of the published case feature framework. However, modify the ticket to require Bryn Rhodes (or an assignee) to provide a concrete list of the "commonly used" patterns, with at least:
- Which resource types are covered (e.g., Observation, Condition, MedicationStatement, Procedure)
- Example StructureDefinition content for each pattern
- Criteria for what qualifies as "commonly used"

This ensures the work group can evaluate the scope and feasibility before implementation begins.

### Disposition B: Deferred
**Defer until FHIR-51859 (GraphDefinition/$apply) is resolved, then revisit case feature patterns in that context.**

The newer FHIR-51859 proposes operations that depend on case feature relationships. It may be more efficient to define commonly used patterns *after* the operational semantics (like `$apply`) are finalized, so that the patterns are designed with those operations in mind. Deferral avoids doing rework if the operational model changes the requirements for case feature patterns.

### Disposition C: Not Persuasive
**Reject due to insufficient specification of the request.**

The ticket description — a single sentence with no examples, no proposed design, and no acceptance criteria — does not provide enough information to act on. The existing case feature infrastructure (from FHIR-23954 and related tickets) already enables implementers to create their own patterns. Without evidence that a standardized set is needed (e.g., implementer feedback, ballot comments, or Zulip discussion), this does not meet the threshold for a change request.

## Recommendation

**Disposition A: Persuasive with Modification** — The request is directionally sound and aligns with the established case feature architecture. Standardized patterns would reduce implementer burden and improve interoperability across CPG-based implementations. However, the ticket needs concrete details before work can proceed. The work group should accept the request conditioned on the submitter providing a specific list of proposed patterns with scope, resource types, and example definitions.
