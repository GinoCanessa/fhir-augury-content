## Summary

This ticket requests that CQL implementations consider issuing a warning when a choice-type function resolution eliminates possible representations of argument types. The scenario arises when a function accepting `Choice<Integer, String>` is called with a `Choice<Boolean, String>` argument — the `Boolean` component is silently dropped during resolution, potentially causing unexpected `null` results at runtime with no indication to the author that data loss has occurred.

## Details

**Description:**

Consider the following definitions:

```cql
library SilentNullidity

define function Normalize(value Choice<Integer, String>):
  case
    when value is DateTime then ToString(value)
    when value is String then value as String
    else null
  end

define ChoiceResult:
  First(List<Choice<Boolean, String>> { true })

define SilentNullidity:
  Normalize(ChoiceResult)
```

Consider recommending that implementations provide a warning if a choice type function resolution eliminates choices of argument types.

In the example above, `ChoiceResult` has type `Choice<Boolean, String>` and when passed to `Normalize(value Choice<Integer, String>)`, the `Boolean` component has no matching type in the function's parameter type. The call is still valid because `String` overlaps, but the `Boolean` choice is silently eliminated. If the runtime value happens to be `Boolean`, it will resolve to `null` with no compiler warning.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Add a recommendation to the CQL specification (e.g., in the Function Resolution or Type Casting section) that implementations SHOULD produce a warning when resolving a function call where the actual argument's choice type includes component types that are not representable by the formal parameter's choice type, indicating which types are eliminated.
#### Justification
This is a low-cost, non-breaking change that improves developer experience. The scenario described is a genuine source of subtle bugs — a `Boolean` value silently becoming `null` with no diagnostic is counter to the principle of least surprise. A warning (not an error) preserves backward compatibility while alerting authors to potential data loss. The reporter (Bryn Rhodes) is the primary CQL spec author, and the request is narrowly scoped to a recommendation rather than a requirement, giving implementers flexibility.

---

### Disposition B: Alternative Approach
#### Proposal
Instead of merely recommending a warning, add a normative "SHOULD" requirement in the specification for implementations to emit a warning diagnostic when choice-type resolution eliminates argument type components. Additionally, define a standard diagnostic code (e.g., `CQL-0042`) and message format so that warnings are consistent across implementations.
#### Justification
A non-normative recommendation may be too weak to achieve meaningful adoption across implementations. Standardizing the diagnostic code and message ensures that tooling (IDEs, CI pipelines, quality measure testing frameworks) can reliably detect and act on these warnings. This is still non-breaking since it is a "SHOULD" rather than "SHALL", but provides stronger guidance and interoperability of diagnostics.

---

### Disposition C: Decline
#### Proposal
Decline the change request. The current behavior (silent resolution with type narrowing) is consistent with the existing CQL type system semantics and does not require additional specification text.
#### Justification
CQL already defines that choice-type resolution works by finding overlapping types, and authors are expected to understand the type system. Adding warnings for every type narrowing scenario could produce excessive noise in implementations that heavily use choice types. Furthermore, this is an implementation quality concern rather than a specification concern — implementers are already free to add such warnings without specification guidance.

---

### Recommendation

**Recommended disposition:** A — Accept As Requested

The request is well-motivated by a concrete example that demonstrates a real risk of silent data loss in CQL expressions. The proposed change is minimal — a recommendation rather than a requirement — which means it is non-breaking and gives implementers discretion in how (and whether) to surface the warning. The reporter is deeply familiar with the CQL specification and has identified a genuine gap in implementation guidance. Accepting as requested adds value to the spec without imposing undue burden, and implementers who already handle this case well are unaffected. Disposition B's stronger normative approach could be considered in a future cycle if adoption of warnings proves inconsistent, but for an initial step a recommendation is appropriately cautious.
