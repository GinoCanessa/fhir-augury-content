## Summary

This ticket requests the ability to model data flow within the CPGWorkflowDefinition profile. Specifically, it proposes adding extensions to the `input` element of a workflow definition so that data requirements can reference each other, enabling the representation of data flow between steps. The proposed extensions include:

- An `id` element on input to uniquely identify data requirements (e.g., `data_2`)
- A `relatedRequirement` element with a `requirementId` child that references another input's `id` (e.g., `data_1`)

This would allow authors to express that one data requirement (e.g., a QuestionnaireResponse filtered by a specific questionnaire) is related to or dependent on another data requirement, thereby modeling how data flows through a clinical guideline workflow.

## Details

The request provides an XML example showing the desired extension pattern on the `input` element of a PlanDefinition action (as used in CPGWorkflowDefinition):

```xml
<input>
    <id value="data_2" />
    <dataRequirement>
        <type value="QuestionnaireResponse"/>
        <codeFilter>
            <path value="questionnaire"/>
            <code>
                <code value="http://hl7.org/fhir/uv/cpg/Questionnaire/cpg-common-identity"/>
            </code>
        </codeFilter>
    </dataRequirement>
    <relatedRequirement>
        <requirementId value="data_1"/>
    </relatedRequirement>
</input>
```

The `relatedRequirement` and `requirementId` elements are proposed additions (marked with "added" comments in the original). The `id` element would also be added to allow data requirements to be referenced by other requirements within the same workflow.

This is closely related to the broader need for expressing data dependencies in clinical guidelines â€” knowing not just *what* data is needed, but *how* data flows from one step's output to another step's input.

## Proposed Dispositions

### Disposition A: Accept â€” Persuasive (Subsume Under FHIR-44779)

**Rationale:** The core need described in FHIR-27161 â€” relating data requirements to each other to model data flow â€” has been substantially addressed by FHIR-44779, which defines the `cqf-relatedRequirement` extension with `targetId` and `targetProperty` elements. Since FHIR-44779 has already been applied with "Persuasive with Modification" in the Using CQL With FHIR specification, the workgroup should accept FHIR-27161 by referencing the relatedRequirement extension work and applying it to the CPGWorkflowDefinition profile.

**Action:** Resolve as Persuasive. Update the CPG IG to incorporate the `cqf-relatedRequirement` extension (from FHIR-44779) into the CPGWorkflowDefinition profile's input elements, and add guidance on how to use it to model data flow between workflow steps.

### Disposition B: Accept â€” Persuasive with Modification (CPG-Specific Data Flow Extensions)

**Rationale:** While FHIR-44779 provides a general-purpose `relatedRequirement` extension, the CPGWorkflowDefinition may benefit from additional CPG-specific data flow modeling beyond simple requirement relationships â€” such as directionality (input vs. output flow), cardinality constraints, or transformation descriptions. The workgroup could accept the ticket but modify the approach to build on top of the FHIR-44779 foundation with CPG-specific extensions.

**Action:** Resolve as Persuasive with Modification. Leverage `cqf-relatedRequirement` as the base mechanism but define additional CPG-specific extensions or profiling constraints that address data flow directionality and workflow-specific semantics in CPGWorkflowDefinition.

### Disposition C: Decline â€” Duplicate of FHIR-44779

**Rationale:** FHIR-44779 (Support relatedRequirements) has already been resolved and applied, directly addressing the ability to relate data requirements to each other. The `cqf-relatedRequirement` extension with `targetId` and `targetProperty` provides the mechanism FHIR-27161 was requesting. No additional CPG-specific work is needed beyond what FHIR-44779 provides.

**Action:** Resolve as Duplicate of FHIR-44779. Close the ticket with a reference to the applied extension.

## Recommendation

**Disposition A: Persuasive (Subsume Under FHIR-44779)** is recommended.

FHIR-44779 has already been applied and provides the `cqf-relatedRequirement` extension that addresses the fundamental need described in FHIR-27161. However, FHIR-27161 specifically targets the CPGWorkflowDefinition profile, so there is still actionable work: incorporating the relatedRequirement extension into the CPG IG's workflow profile and providing guidance. Resolving as Persuasive (rather than Duplicate) acknowledges this CPG-specific work while avoiding redundant extension design. The ticket was re-opened post-STU2 specifically to pursue this, and the original 20-0-0 vote confirmed workgroup interest â€” the FHIR-44779 work now makes it actionable.
