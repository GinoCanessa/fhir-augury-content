## Description

Brian Postlethwaite requests that `DataRequirement.profile` (currently typed as `canonical(StructureDefinition)`) be expanded to also permit references to **Questionnaire** and possibly **ObservationDefinition**.

The motivating use case is a PlanDefinition-driven workflow where a completed QuestionnaireResponse for one Questionnaire triggers a requirement to complete a different Questionnaire. Because QuestionnaireResponse is based on a Questionnaire (not a StructureDefinition profile), the current DataRequirement type binding is too restrictive to express this requirement.

The reporter also suggests that ObservationDefinition could be useful as an additional permitted target.

## Zulip Discussions

### [implementers > ✔ Coded DataRequirement in PlanDefinition](https://chat.fhir.org/#narrow/stream/implementers/topic/%E2%9C%94%20Coded%20DataRequirement%20in%20PlanDefinition) (Jul–Nov 2022)

Directly relevant thread. Alexander Zautke asked about representing PlanDefinition action steps that require answering a Questionnaire, proposing DataRequirement input pointing to a Questionnaire with QuestionnaireResponse as output.

**Bryn Rhodes** responded that he would be "a bit surprised to see a data requirement used to reference a Questionnaire" directly, since the action doesn't require the Questionnaire itself — it requires the *answers*. He outlined three alternative patterns:
1. A separate pre-step pointing to the Questionnaire explicitly, with QuestionnaireResponse as output (then input to the next step).
2. A "collect information activity" modeled as a Task.
3. Using input DataRequirements to specify required data elements, then *deriving* a Questionnaire from unfulfilled requirements (so users are only asked questions the system cannot already answer).

Ward Weistra followed up asking how to specify `DataRequirement` type `Questionnaire` or `QuestionnaireResponse` with the canonical reference — highlighting the exact gap this ticket identifies.

## Analysis

### Core Question
Should `DataRequirement.profile` be broadened from `canonical(StructureDefinition)` to also accept `canonical(Questionnaire)` and/or `canonical(ObservationDefinition)`?

### Key Considerations
- **Semantic expansion:** This changes what DataRequirement fundamentally represents — from profiled FHIR resource types to definitional artifacts. Bryn Rhodes flagged this as significant.
- **Precedent:** ActivityDefinition already references definitional resources in a similar way, suggesting the pattern is not unprecedented but may need a cleaner home.
- **Practical need:** The use case (Questionnaire-driven CDS workflows via PlanDefinition) is common and well-understood. The Zulip thread shows implementers actively hitting this limitation.
- **Alternative patterns exist:** Bryn Rhodes outlined three workarounds that avoid changing DataRequirement's type binding, though they add complexity.
- **Stalled since August 2022:** The ticket has been waiting for a worked example and further CDS/CPG-on-FHIR discussion for nearly three years.

---

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Expand `DataRequirement.profile` to accept `canonical(StructureDefinition | Questionnaire)` and add non-normative guidance explaining when to use Questionnaire references versus StructureDefinition profiles. Defer ObservationDefinition to a separate tracker item pending further analysis.

**Rationale:** The Questionnaire use case is well-established and implementers are actively encountering this limitation. Expanding to Questionnaire addresses the primary ask with minimal disruption. ObservationDefinition and SpecimenDefinition can be evaluated separately since they raise additional scope questions. Adding guidance (not just a type change) ensures implementers understand the semantic implications.

### Disposition B: Deferred
**Action:** Defer to a future ballot cycle. Request that CDS and CPG-on-FHIR complete the worked example Bryn Rhodes requested in August 2022, and evaluate whether the alternative patterns (pre-step, Task-based, derived Questionnaire) are sufficient.

**Rationale:** The change is acknowledged as semantically significant. No concrete worked example has been produced. The alternative approaches described by Bryn Rhodes may adequately address the use case without changing the DataRequirement type system. A deferred disposition allows time for the community to evaluate whether the workarounds are sufficient or whether the type expansion is truly needed.

### Disposition C: Not Persuasive
**Action:** Close as not persuasive. Document the existing alternative patterns (pre-step with explicit Questionnaire reference, Task-based collection, derived Questionnaire from DataRequirements) as recommended approaches.

**Rationale:** The alternative patterns described by Bryn Rhodes in the Zulip thread already address the use case without fundamentally changing what DataRequirement represents. Expanding the type target risks blurring the distinction between resource-type-based data requirements and definitional-artifact requirements. The existing mechanisms, while requiring slightly more modeling effort, keep the DataRequirement semantics clean and well-bounded.

---

## Recommendation

**Disposition A: Persuasive with Modification** — Expand `DataRequirement.profile` to accept `canonical(StructureDefinition | Questionnaire)` with supporting guidance; handle ObservationDefinition separately.

The Questionnaire-driven CDS use case is well-documented, actively encountered by implementers, and already has precedent in ActivityDefinition's handling of definitional resources. The alternative patterns, while workable, add modeling complexity that is disproportionate to the simplicity of the underlying need. Limiting the initial expansion to Questionnaire (deferring ObservationDefinition) reduces scope and risk. The ticket has been stalled for nearly three years awaiting a worked example — the Zulip thread and related tickets (FHIR-32898, FHIR-50165) collectively provide sufficient evidence of need. Requesting a specific example as part of the resolution (not as a prerequisite) would move the work forward while ensuring quality.
