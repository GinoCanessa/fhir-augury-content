## Problem Statement

The Artifact Operation profile in CRMI defines a pattern for identifying the focus artifact for an operation, but it only supports a single artifact. Operations like `$evaluate-measure` need the ability to specify **multiple artifacts** (e.g., evaluating multiple measures in a single request). There is a tension between supporting multiple artifacts and maintaining the simplicity of GET invocation.

The ticket proposes three options:

1. **Option 1 (single-parameter):** Repeat existing parameter names (`measureId`, `measureUrl`, `measureIdentifier`, `measure`) with `0..*` cardinality. Supports GET invocation. Instance-level stays single; type-level supports multi.
2. **Option 2 (multi-part parameters by measure):** A `measure 0..*` parameter with sub-parts (`.id`, `.url`, `.version`, `.identifier`, `.resource`, `.topic`). More structured but requires POST.
3. **Option 3 (multi-part parameters by artifact):** A generic `artifact 0..*` parameter with sub-parts (`.reference`, `.url`, `.version`, `.identifier`, `.resource`, `.topic`). Most general; supports mixed resource types (Measure, Library).

## Resolution Description

Update the artifact operation profile pattern to allow specification of multiple artifacts using the **first pattern** (single-parameter with `0..*` cardinality):

- `artifactId` id `0..*`
- `artifactIdentifier` Identifier `0..*` (version not allowed; up to server)
- `artifactUrl` canonical `0..*` (version optional)
- `artifact` Resource `0..*`

## Analysis

### Context

This ticket addresses a fundamental limitation in the CRMI Artifact Operation profile pattern: it was designed for single-artifact focus, but real-world operations (particularly `$evaluate-measure` in quality reporting) frequently need to operate on multiple artifacts simultaneously. The performance case is compelling — evaluating dozens of measures against the same patient population should not require redundant data retrieval per measure.

The resolution chose **Option 1** (repeating single parameters with `0..*` cardinality) over the more structured multi-part parameter approaches. This preserves GET invocability and aligns with the pattern already established in DEQM's `$care-gaps` and the multi-measure `$evaluate` operation (FHIR-46287).

### Tensions

- **FHIR-49856** (filed later by the same reporter, Bryn Rhodes) explicitly states that DEQM's pattern for multi-artifact evaluation is "superior" and should be reconsidered for CRMI — suggesting the resolution here may not be the final word.
- **FHIR-51856** pushes in the opposite direction, arguing that too many identification parameters create interoperability risk and vendor lock-in, and advocating for a minimal parameter set.
- The `artifactIdentifier` parameter disallows version (leaving it "up to server"), which could create ambiguity in environments with multiple artifact versions.

### Impact

The change affects the CRMI specification's artifact operation profile pattern, which is used as the basis for multiple operation definitions across CRMI and downstream IGs like DEQM. Changing cardinality from `0..1` to `0..*` for the artifact identification parameters is additive and backward-compatible (existing single-artifact invocations remain valid).

## Proposed Dispositions

### Disposition A: Approve as Persuasive with Modification (Accept Resolution)

Accept the voted resolution as-is. Update the artifact operation profile pattern to use `0..*` cardinality for `artifactId`, `artifactIdentifier`, `artifactUrl`, and `artifact` parameters.

**Rationale:** The resolution addresses the stated need for multi-artifact support using the simplest approach (Option 1). It preserves GET invocability, is backward-compatible, and aligns with the pattern already successfully implemented in DEQM (FHIR-46287). The single-parameter approach has proven workable in practice.

**Risk:** FHIR-49856 suggests the community may revisit this pattern in favor of DEQM's multi-part parameter approach, potentially requiring a second round of changes.

### Disposition C: Request Modification — Adopt DEQM Multi-Part Parameter Pattern

Return as Persuasive with Modification but change the resolution to use a multi-part parameter pattern (closer to Option 2/3) that aligns with the DEQM `$evaluate` operation definition, as suggested in FHIR-49856.

Aligning CRMI with the proven DEQM approach avoids divergence between the base infrastructure IG and its most prominent consumer. **Risk:** More complex than Option 1. Requires POST for invocation (no GET support). May need more implementation experience to validate.

The voted resolution provides a practical, backward-compatible solution using the simplest approach. While FHIR-49856 raises valid questions about whether DEQM's pattern is superior, the Option 1 pattern is already working in DEQM for the `$care-gaps` operation and addresses the immediate need. The `0..*` cardinality change is additive and low-risk. If the community later converges on a different pattern through FHIR-49856, that can be handled as a separate evolution — implementing multi-artifact support now does not preclude refinement later.
