## Problem Statement

The submitter reports that `MonetaryComponent.factor` (used in ChargeItemDefinition and other resources) lacks a clear explanation of what numerical values should be used. Specifically, it is ambiguous whether a 25% tax should be represented as a factor of `0.25` or `1.25`. The existing informative examples use factors like `1.19` and `1.07` (German tax rates), which compound the confusion because those values represent the total multiplier (base + tax) rather than the component factor.

The original ticket description was accidentally submitted incomplete ("ChargeItemDefinition factor is a little bit unclear - while"), but the reporter immediately followed up in a comment clarifying the issue and linking to the Zulip discussion that prompted it.

## Keywords

`MonetaryComponent`, `ChargeItemDefinition`, `factor`, `tax`, `discount`, `surcharge`, `deduction`, `Invoice`, `ChargeItem`, `PriceComponentType`, `unitPriceComponent`, `totalPriceComponent`

## Resolution Description (from Jira)

The resolution is detailed and proposes multiple changes:

1. **MonetaryComponent.factor clarification** — Add a comment explaining that the factor is multiplied to the base value to produce the component amount (e.g., a 25% discount → factor of `0.25`, with `amount = base × 0.25`).

2. **MonetaryComponent narrative guidance** — Add guidance explaining that components are ordered, each type (surcharge, tax, discount) has a different impact on the total, and provide a worked example table showing base ($100), discounts (percentage-based and fixed), surcharge ($50), and taxes (state 10%, local 2%).

3. **Remove "deduction" type** — The `deduction` type from PriceComponentType should be removed as it is just a form of discount. The `code` property can differentiate deduction vs discount subtypes.

4. **Update PriceComponentType definitions** — Make the code system definitions more explicit about whether each type increases or decreases the amount applied to the base.

5. **Update ChargeItem cardinalities** — Change `ChargeItem.unitPriceComponent` and `ChargeItem.totalPriceComponent` to `0..*` (from current cardinality).

Active discussion from 2023-10-02 to 2024-02-01 with 14 messages involving Vadim Peretokin, Brian Postlethwaite, Frank Oemig, and Simone Heckmann.

- **Vadim Peretokin** asked whether 25% tax should be factor `0.25` or `1.25`, noting the confusing examples using `1.19` and `1.07`.
- **Frank Oemig** explained that `1.19`/`1.07` are real multiplication factors for German tax rates (19%/7%), but acknowledged the definition must be clear about whether it's a *component factor* (`0.25`) or a *total factor* (`1.25`).
- **Simone Heckmann** clarified that the factor should be `0.25` for a 25% tax, as the definition states the factor calculates the *component* amount, not the total. She also noted that ChargeItemDefinition only lists possible components — the total is calculated elsewhere (e.g., on an Invoice).

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Apply the Resolution as Written)

**Action:** Apply the full resolution description — add the factor comment and narrative guidance to MonetaryComponent, remove the `deduction` PriceComponentType value, update PriceComponentType code definitions, and change ChargeItem.unitPriceComponent / totalPriceComponent to `0..*`.

**Rationale:** The resolution was discussed at the January 2024 WGM and provides a comprehensive, well-structured clarification. The worked example table eliminates the ambiguity that sparked the original issue. Removing `deduction` simplifies the model and aligns with FHIR-44610 (already Applied). The Zulip discussion shows strong community consensus that factor represents the component multiplier (e.g., `0.25`), not the total multiplier (`1.25`).

**Risk:** This is a broad change touching MonetaryComponent (a data type used in multiple resources), ChargeItem cardinality, and PriceComponentType. Removing `deduction` is a breaking change for any implementations using that code. The resolution itself notes it needs coordination with Financial Management (joint call was proposed in Jan 2025). Some wording in the resolution is still marked as TODO.

### Disposition B: Accept as Persuasive with Modification — Narrow Scope to Factor Clarification Only

**Action:** Apply only the MonetaryComponent.factor comment and the narrative guidance (items 1 and 2 from the resolution). Defer the `deduction` removal, PriceComponentType definition updates, and ChargeItem cardinality changes to their respective dedicated tickets (FHIR-44610, FHIR-37946).

**Rationale:** The original submitter's request was specifically about factor semantics. The broader changes (removing deduction, cardinality updates) have their own tickets and should be tracked/applied independently. This avoids duplication of applied changes and reduces risk of scope creep. Brian Postlethwaite's recommendation to hold open for wider review supports a more cautious approach.

**Risk:** The narrative guidance references discount/deduction/surcharge semantics that may be inconsistent if the deduction removal (FHIR-44610) hasn't been applied yet. The example table would need adjustment to omit the "deduction (remove - feedback please)" row.

### Disposition C: Defer — Pending Joint PA/FM Review

**Action:** Keep the ticket in Triaged status. Schedule the joint PA/Financial Management call (as proposed by Brian Postlethwaite in January 2025) before applying any changes.

**Rationale:** The January 2025 Zulip message explicitly groups this ticket with several others needing FM coordination. The resolution description still contains TODO markers and "(remove - feedback please)" annotations, suggesting it is not fully finalized. Given the cross-resource impact of MonetaryComponent changes and the FM work group's stake in financial resources (Invoice, ChargeItem, ChargeItemDefinition), joint review would strengthen consensus.

**Risk:** Further delay on a ticket originally filed in October 2023. The core factor clarification is non-controversial and could be applied independently.

## Recommendation

**Disposition B: Accept as Persuasive with Modification — Narrow Scope to Factor Clarification Only.**

The factor ambiguity is the core issue and has clear community consensus (Zulip discussion, WGM notes). Applying the factor comment and narrative guidance directly addresses the submitter's request without entangling it with the broader PriceComponentType restructuring that is still under discussion. The broader changes (deduction removal, cardinality) are already tracked in FHIR-44610 and FHIR-37946 and should progress on their own timelines, especially after the planned joint PA/FM call. The example table in the narrative should be adjusted to remove the provisional "deduction (remove)" row to avoid confusion.
