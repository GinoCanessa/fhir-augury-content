## Description

The submitter requests that the `Consent.extension:restrictFhirPath` extension in the NDH (National Directory of Healthcare Providers and Services) Restriction profile be deprecated. The argument is that this use case—partial inline security labeling of specific data elements—has already been addressed more generally by the [DS4P (Data Segmentation for Privacy)](https://www.hl7.org/fhir/uv/security-label-ds4p/2021Sep/inline.html) IG. The submitter contends that NDH should not be the place to define and address security labeling guidance.

The extension in question is defined at:
- http://build.fhir.org/ig/HL7/fhir-us-ndh/branches/master/StructureDefinition-ndh-Restriction-definitions.html#diff_Consent.extension:restrictFhirPath

## Discussion History

### PA WGM — 2024-01-29 (brian.postlethwaite)
The workgroup reviewed the DS4P content and noted that the DS4P functionality **may not handle the details of the restrictions** at the level of identity that NDH requires. The group decided to refer to the Security WG to confirm whether DS4P covers NDH's requirements. The question was raised whether the [`extension-sec-label-related-artifact`](https://hl7.org/fhir/uv/security-label-ds4p/StructureDefinition-extension-sec-label-related-artifact.html) extension could cover the ability to refer to a Consent with all required restriction details.

### Security Perspective — 2024-01-29 (john_moehrke)
John Moehrke (Security WG) strongly discouraged having data point at policy, as that approach means policy changes require data changes. The preferred model is for policy (e.g., Consent, Permission) to point at the data. This has proven to be a more sustainable and scalable solution, consistent with broader security standards.

## Key Considerations

1. **Superseded by FHIR-48636**: The later ticket FHIR-48636 (status: Applied, resolution: Persuasive) explicitly requested and achieved the removal of the NDH Restriction profile and adoption of DS4P inline security labels—which is essentially what FHIR-41604 was asking for.

2. **Security architecture principle**: John Moehrke's comment reinforces that policy should reference data, not the other way around. DS4P's approach (inline security labels on data elements, with references to Consent/Permission resources) aligns with this principle, while NDH's `restrictFhirPath` approach inverted it.

3. **DS4P coverage confirmed**: The resolution of FHIR-48636 as Persuasive, with a concrete example of DS4P inline labels (including `extension-sec-label-related-artifact` linking to Consent), demonstrates that DS4P does cover NDH's use case.

4. **Long-stale ticket**: FHIR-41604 has been in Triaged status since 2023-07-19 with no movement since the Jan 2024 WGM discussion. The work has since been accomplished via FHIR-48636.

## Proposed Dispositions

### Option A: Persuasive (Recommended)
**Resolution:** Persuasive
**Action:** Close as Persuasive, noting that the requested change has been implemented via FHIR-48636 (which was resolved as Persuasive and Applied). The NDH Restriction profile based on Consent has been removed, and DS4P inline security labels have been adopted in its place. The `restrictFhirPath` extension is no longer part of the NDH IG. No further action is needed beyond closing this ticket with a reference to FHIR-48636.

**Rationale:** The submitter's request was correct—NDH should not define its own security labeling approach when DS4P provides a standardized mechanism. This has been validated by the resolution of FHIR-48636, and the Security WG's endorsement of the policy-points-at-data architecture.

### Option B: Persuasive with Modification
**Resolution:** Persuasive with Modification
**Action:** Close as Persuasive with Modification, acknowledging the submitter's concern was valid but noting the implementation was achieved through FHIR-48636 rather than direct deprecation of `restrictFhirPath`. The modification is that the entire Restriction profile was removed (not just the extension deprecated), which is a more complete solution than what was originally requested.

**Rationale:** The submitter asked only for deprecation of one extension and a reference to DS4P. The actual resolution went further—removing the entire Restriction profile. This is a broader change than requested but achieves the submitter's intent.

### Option C: Not Persuasive with Modification
**Resolution:** Not Persuasive with Modification
**Action:** Close as Not Persuasive with Modification, noting that while the specific request to "deprecate and reference DS4P" was not actioned through this ticket, the underlying concern was addressed through a separate tracker (FHIR-48636). Document the cross-reference.

**Rationale:** The ticket was never directly acted upon; the resolution came through a different path. However, the end result aligns with the submitter's intent.

## Recommendation

**→ Option A: Persuasive**

The submitter's core argument—that NDH should not define its own security labeling and should instead use DS4P—was correct and has been validated by the community. FHIR-48636 implemented exactly this change (resolved Persuasive, Applied). The `restrictFhirPath` extension and the entire NDH Restriction profile have been removed. Closing this ticket as Persuasive with a cross-reference to FHIR-48636 gives proper credit to the original submitter's insight while acknowledging the work was completed through the later ticket.
