## Description

The submitter compares the B2B with User Authorization Extension Object (`hl7-b2b-user`) to the B2B Authorization Extension Object (`hl7-b2b`) and raises three specific areas of confusion:

1. **Ambiguous cross-reference to requirements:** The spec states "The same requirements for use of hl7-b2b apply in the use of hl7-b2b-user" without specifying which requirements. The submitter believes this refers to section 5.2.1.2 of UDAP Security but seeks confirmation.

2. **Unclear "as per" language:** The spec says `hl7-b2b-user` is included "as per UDAP Security 5.2.1.1." The submitter asks whether this means "analogously to" or "in addition to following" that section.

3. **Additive vs. replacement semantics:** It is unclear whether `hl7-b2b-user` is intended to be included *in addition to* or *instead of* `hl7-b2b`. If it replaces `hl7-b2b`, then section 5.2.1 (which requires `hl7-b2b` for B2B client apps using `client_credentials` flow) would need to be rewritten. Additionally, the Identity IG has no equivalent SHALL requirement for when `hl7-b2b-user` must be included.

## Comment History

The ticket has 8 comments spanning from December 2023 to June 2025, with the primary participants being **jlamy** (reporter) and **juliemaas** (editor/IG author).

**Key discussion points:**

- **jlamy (2023-12-04):** Questioned why the `hl7-b2b-user` extension appears on the patient matching page, noting it has nothing to do with `$match`.

- **juliemaas (2024-03-28):** Clarified this extension is for person matching when a B2B use case has a user present and the requestor wants to provide details about the user (not an authorization code flow transaction). Noted that requirements throughout section 4 apply since demographics are specified.

- **juliemaas (2024-06-18):** Clarified the page is not unique to the FHIR patient match operation but to person matching more generally.

- **jlamy (2025-06-19):** Documented testing assumptions: (a) requirements refer to section 5.2.1.2 "Preferred format for identifiers and codes"; (b) "as per" means analogously; (c) `hl7-b2b-user` is used instead of `hl7-b2b`, and filed FHIR-51161 against FAST Security IG to permit alternate extensions.

- **juliemaas (2025-06-19):** Noted that there are use cases where *both* extensions could be needed (Facilitated FHIR for IAS or JWT-Based Auth B2B with a user present but not "on the wire").

- **jlamy (2025-06-20):** Summarized understanding: if only `hl7-b2b-user` is present, authenticate the identified user; if `hl7-b2b` is present, authenticate the org—`hl7-b2b-user` may narrow authorization scope but may also be ignored.

- **juliemaas (2025-06-20):** Expressed concern about the suggestion to ignore `hl7-b2b-user` info from a policy perspective, noting that adding user identity information implies data will flow back to an individual and should not be ignored in policy decisions/audit logs.

## Proposed Dispositions

### Disposition A: Not Persuasive (Affirm Current Resolution)

**Rationale:** The concerns raised by the submitter have been substantively addressed through parallel work. The reporter himself filed FHIR-51161 (resolved Persuasive, Published) against the FAST Security IG to clarify how B2B may be extended by other IGs. FHIR-45199 (also Persuasive, Published) addressed the broader inconsistency between patient authentication requirements and B2B AEO assertions in the same specification. The comment thread demonstrates that the reporter and IG author reached a working understanding of the extension's intended use. The resolution description—"Discussed and incorporated as part of other changes"—accurately reflects this outcome.

**Applied change:** None. Affirm the existing "Not Persuasive" resolution.

### Disposition B: Persuasive with Modification — Add Clarifying Text to Identity IG

**Rationale:** While the concerns have been discussed, the underlying specification text remains ambiguous. The three questions raised (which requirements apply, what "as per" means, additive vs. replacement semantics) are reasonable reader confusion points. A small clarifying note in the Identity IG's B2B with User Authorization Extension Object section could address these without structural changes. For example: explicitly reference section 5.2.1.2 for shared requirements, clarify that "as per" means "following the same pattern as," and state that `hl7-b2b-user` may be used alongside or in place of `hl7-b2b` depending on use case (with `hl7-b2b` still required for pure client_credentials flows without a user context).

**Applied change:** Add a brief clarifying paragraph to the `hl7-b2b-user` section of the Identity IG.

### Disposition C: Persuasive with Modification — Add Cross-Reference Note

**Rationale:** The resolution text "Discussed and incorporated as part of other changes" lacks traceability. At minimum, the resolution description should reference the specific tickets (FHIR-51161, FHIR-45199) where the related changes were made, so future readers can follow the thread. This does not change the Not Persuasive outcome but improves documentation.

**Applied change:** Update the resolution description to cite FHIR-51161 and FHIR-45199 as the tickets where related changes were incorporated.

## Recommendation

**Disposition A: Not Persuasive (Affirm Current Resolution)**

The submitter's concerns were legitimate but have been resolved through the natural progression of the work. The reporter filed FHIR-51161 (now Published/Persuasive) to address the UDAP Security side, and FHIR-45199 (also Published/Persuasive) addressed the Identity IG's internal inconsistencies.
