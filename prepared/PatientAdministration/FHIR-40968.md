## Description

The submitter suggests using `udiCarrier` in the RTLS Tag Device Profile. The request is to add UDI (Unique Device Identifier) carrier information to the Device profile used for RTLS tags/badges.

## Resolution Analysis

The existing resolution is **Not Persuasive** with the following rationale:

1. The RTLS Tag Device profile represents a **badge**, not a medical device. Badges are not considered medical devices and would not have a UDI.
2. An RTLS badge *could* be affixed to a medical device that has a UDI, but the badge itself does not.
3. The related ticket [FHIR-40970](https://jira.hl7.org/browse/FHIR-40970) was resolved as **Persuasive with Modification** and addresses adding UDI to the Location Update Event *example* — acknowledging that UDI is relevant when a tracked device is a medical device, just not at the RTLS Tag profile level.

## Proposed Dispositions

### Disposition A: Not Persuasive (affirm current resolution)

**Rationale:** The RTLS Tag Device profile models a location-tracking badge, which is not itself a medical device and therefore does not carry a UDI. Adding `udiCarrier` to this profile would incorrectly imply that all RTLS tags are UDI-bearing medical devices. The companion ticket FHIR-40970 already addresses the legitimate use case — adding UDI in the Location Update Event example where the tracked item *is* a medical device. The profile and the example serve different purposes: the profile defines the badge; the event example shows how UDI data flows when applicable.

**Changes required:** None.

### Disposition B: Persuasive with Modification — add udiCarrier as optional (0..1)

**Rationale:** While RTLS badges themselves are not medical devices, the Device resource can represent both the badge and the tracked device. Making `udiCarrier` optionally available (0..1 or 0..*) in the RTLS Tag Device profile would allow implementers flexibility to include UDI information when the tag is affixed to a medical device, without requiring it for non-medical badges. This keeps the profile inclusive of both scenarios.

**Changes required:** Update the RTLS Tag Device profile StructureDefinition to include `udiCarrier` as an optional element (cardinality 0..*), and add guidance clarifying when it should be populated.

### Disposition C: Persuasive — require udiCarrier support in the profile

**Rationale:** If RTLS tags can be placed on medical devices, the profile should natively support UDI to ensure interoperability of device identification in RTLS workflows. Requiring `udiCarrier` support (must-support, 0..*) ensures that systems processing RTLS data can always handle UDI when present.

**Changes required:** Update the RTLS Tag Device profile to include `udiCarrier` with must-support flag and cardinality 0..*, and update conformance documentation accordingly.

## Recommendation

**Disposition A: Not Persuasive** (affirm current resolution)

The existing resolution is well-reasoned. The RTLS Tag Device profile specifically models a location-tracking badge — not a medical device. The distinction between the badge (tag) and the tracked item (which may be a medical device) is important. FHIR-40970 already handles the case where UDI information needs to appear in RTLS workflows by adding it to the Location Update Event example. Adding `udiCarrier` to the Tag Device profile would blur the conceptual boundary between the tracking tag and the tracked device, and there is no community discussion or additional evidence suggesting the original resolution should be overturned.
