## Description

The Patient Administration work group has raised concerns about the `identifier-status` extension used in the Validated Healthcare Directory (VHDir) IG. Three specific issues are identified:

1. **Overlap with `Identifier.use`:** The concepts in the identifier-status extension overlap with existing values in `Identifier.use`. Specifically: `inactive` and `revoked` map to `old`, and `pending` maps to `temp`.
2. **Modifier concerns:** Some of the status concepts (e.g., `revoked`, `inactive`) may function as modifiers — they change the meaning of the identifier in ways that could affect processing if not understood.
3. **Scope:** If an identifier status concept is needed, it should be a generic (core) extension rather than an IG-specific one, to avoid proliferation and inconsistency.

The ticket requests discussion with FHIR Infrastructure (FHIR-I) at a Working Group Meeting (WGM).

## Analysis

This ticket raises a fundamental design question: should identifier lifecycle status be handled through `Identifier.use` (which already exists in core) or through a dedicated status extension? The VHDir IG introduced an `identifier-status` extension with concepts like `active`, `inactive`, `revoked`, and `pending`, but the ticket argues these overlap semantically with existing `Identifier.use` codes (`old`, `temp`).

The concern about modifier semantics is significant — a `revoked` identifier changes the fundamental meaning of the data, and consumers unaware of the extension could misinterpret the identifier as still valid. This is a classic case for either a modifier extension or a core element.

The ticket has been in "Triaged" status since December 2023 with no comments, no assignee, and no resolution — suggesting the WGM discussion has not yet occurred or no outcome has been recorded.

## Proposed Dispositions

### Disposition A: Persuasive — Promote to Core Extension with FHIR-I Coordination
Accept the ticket's concerns as valid. Coordinate with FHIR-I to define a standard (core-level) extension for `Identifier.status` that replaces IG-specific versions. Clarify the boundary between `Identifier.use` (intended purpose of the identifier) and identifier status (lifecycle state). Mark the extension as a modifier extension for concepts like `revoked` and `inactive`. Deprecate the VHDir-specific `identifier-status` extension in favor of the core version.

**Pros:** Addresses all three concerns (overlap, modifier, scope). Creates consistency across IGs. Proper modifier semantics protect consumers.
**Cons:** Requires cross-WG coordination with FHIR-I. May require deprecation path for existing VHDir implementations.

### Disposition B: Persuasive with Modification — Clarify Identifier.use and Defer Extension
Accept the concerns about overlap but resolve them by expanding the definition of `Identifier.use` codes to explicitly cover lifecycle states. Add guidance clarifying that `old` encompasses `inactive`/`revoked` and `temp` covers `pending`. Do not create a new extension unless a use case demonstrates that `Identifier.use` is insufficient. Request the VHDir IG align with this guidance.

**Pros:** Avoids introducing new extensions. Leverages existing infrastructure. Simpler for implementers.
**Cons:** `Identifier.use` may not capture all needed granularity (e.g., distinguishing `revoked` from `expired` from `inactive`). VHDir may have valid use cases not fully covered by `Identifier.use`.

### Disposition C: Not Persuasive — Leave as IG-Specific Extension
The `identifier-status` extension serves specific directory use cases that go beyond what `Identifier.use` was designed for. While there is overlap in some concepts, the extension provides finer granularity (e.g., `revoked` vs `inactive` vs `expired`) that directory implementations need. IGs are the appropriate place for domain-specific extensions until proven need for core adoption. The modifier concern can be addressed by marking the extension as a modifier extension within the IG.

**Pros:** No disruption to existing VHDir implementations. Allows the IG to evolve independently.
**Cons:** Risks proliferation of similar extensions across IGs. Does not address the modifier concern at the core level. Other IGs (e.g., Interoperable Digital Identity) may reinvent similar concepts.

## Recommendation

**Disposition A: Persuasive — Promote to Core Extension with FHIR-I Coordination**

The reporter (Lloyd McKenzie, FHIR co-chair) raises well-founded concerns about semantic overlap, modifier safety, and appropriate scope. The evidence from related tickets (FHIR-18142, FHIR-18144, FHIR-45680) shows that multiple IGs are independently creating identifier status concepts, confirming the need for a core-level solution. The rendering confusion noted in the Zulip discussion further highlights that this extension behaves like a first-class property. Coordinating with FHIR-I to define a core extension (or evaluate adding `status` to the Identifier data type itself) would provide consistency, proper modifier semantics, and avoid IG-level fragmentation.
