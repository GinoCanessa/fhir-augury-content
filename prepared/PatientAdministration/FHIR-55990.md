## Problem Statement

The reporter (FredTrotter) asks how to express two practitioner verification attributes in the NDH Implementation Guide:

1. **`is_medicare_validated`** — indicates the practitioner has an IAL2 login to NDH to verify their identity.
2. **`is_cms_enrolled`** — indicates the practitioner has enrolled as a practitioner in Medicare.

The reporter asks whether the existing [`extension:verification-status`](https://build.fhir.org/ig/HL7/fhir-us-ndh/StructureDefinition-ndh-Practitioner-definitions.html#Practitioner.extension:verification-status) is the appropriate mechanism and requests an example.

## Resolution Description (from Jira)

> Add extension **NDH CMS Enrollment In Good Standing** — Boolean – PractitionerRole, Organization, and Verification.
>
> Add extension **NDH Aligned with CMS Data Network** — Boolean – PractitionerRole, Organization.

## Proposed Dispositions

### Disposition A: Persuasive with Modification (Accept with Entity Placement Refinement)

**Action:** Accept the resolution to add two new Boolean extensions ("NDH CMS Enrollment In Good Standing" and "NDH Aligned with CMS Data Network"), but **modify the placement per the JIRAUSER20919 comment**: apply both extensions to **Practitioner, PractitionerRole, and Organization** rather than including Verification as a target. The Verification resource should capture provenance and trust metadata, not business/regulatory status attributes of the entity itself.

**Rationale:** The original resolution adds valuable CMS-related status indicators that real-world NDH implementations need. However, the comment from JIRAUSER20919 correctly identifies a design concern: these are entity attributes, not verification metadata. Adjusting placement maintains clean separation of concerns between entity attributes and verification provenance. Implementation is underway in [PR #10](https://github.com/HL7/fhir-us-ndh/pull/10).

### Disposition B: Persuasive (Accept Resolution As-Is)

**Action:** Accept the resolution exactly as stated — add "NDH CMS Enrollment In Good Standing" (Boolean) to PractitionerRole, Organization, and Verification; add "NDH Aligned with CMS Data Network" (Boolean) to PractitionerRole and Organization.

**Rationale:** The resolution was agreed upon by the work group. Including "CMS Enrollment in Good Standing" on Verification may be intentional — a verifier might need to record whether CMS enrollment was confirmed as part of the verification process. The comment suggesting otherwise is one perspective that the WG can weigh but need not act on if the original placement was deliberate.

### Disposition C: Not Persuasive (Defer to Existing Mechanisms)

**Action:** Reject the request. The existing `extension:verification-status` on Practitioner already provides a mechanism to express verification outcomes. CMS-specific enrollment status could be modeled as values within that existing extension's value set rather than adding new top-level Boolean extensions.

**Rationale:** Adding multiple CMS-specific Boolean extensions increases profile complexity and creates a pattern where each regulatory status gets its own extension. A more scalable approach would be to extend the existing verification-status value set to include CMS-specific codes, keeping the extension surface area smaller.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The need for CMS enrollment and data network alignment indicators is clearly justified by real-world NDH implementation requirements. However, the JIRAUSER20919 comment raises a valid architectural concern that the work group should address: the Verification resource is designed for provenance/trust metadata (who verified, when, how), while CMS enrollment status is a business attribute of the entity. Accepting the resolution with the refinement to exclude Verification as a target for these extensions maintains clean resource semantics. The implementation work is already underway in PR #10, and coordinating with the companion ticket FHIR-55989 (which addresses the related IAL2 verification extension) will ensure a consistent approach across both tickets.
