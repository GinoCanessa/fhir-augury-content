## Description

The submitter requests that the **enroll tag** and **unenroll tag** interactions in the RTLS IG be redesigned from FHIR Messaging to **RESTful interactions or operations**. The rationale is that RESTful approaches would be better defined and easier to implement than the current messaging-based pattern.

## Resolution Description

> We had several discussions. The current plan is to use messages for enroll and unenroll. There are no plans to adopt subscriptions.
>
> We will update some narrative sections that use subscriber-like language to clarify that this IG is using the messaging pattern.

The work group resolved this as **Not Persuasive with Modification**: the fundamental request to switch from messaging to REST was rejected, but the WG acknowledged that narrative language in the IG conflated messaging with subscription-like terminology, and committed to clarifying those sections.

## Analysis

This ticket reflects a fundamental architectural question about the RTLS IG: should tag enrollment/unenrollment use FHIR Messaging or RESTful operations? The submitter (Vassil Peytchev) argued that RESTful interactions are simpler and more widely understood. The WG had "several discussions" and decided to retain messaging, which aligns with their broader architecture for the RTLS event-driven workflow.

The modification component — clarifying narrative language that incorrectly used subscriber-like terminology — is a concrete, actionable change. FHIR-52436 further reinforces this direction by proposing formal MessageDefinitions for the enroll/unenroll/location-update events, and FHIR-52433 addresses similar narrative alignment issues.

The resolution represents a deliberate architectural choice: while REST would be simpler for point-to-point interactions, the WG views the messaging pattern as more appropriate for the event-driven nature of RTLS workflows where a tag management system broadcasts enrollment/unenrollment events.

## Proposed Dispositions

### Disposition A: Accept as Not Persuasive with Modification (Affirm Resolution)

Accept the WG's resolution as stated. The messaging pattern is a deliberate architectural decision for the RTLS IG's event-driven workflow. The modification to clarify narrative language that incorrectly uses subscriber-like terminology is appropriate and addresses the valid concern about confusing wording. Subsequent tickets (FHIR-52436, FHIR-52433) show the WG is actively investing in improving the messaging specification rather than replacing it.

- **Pros:** Respects the WG's informed architectural decision after multiple discussions; the narrative fix addresses the practical confusion; consistent with the direction shown in newer RTLS tickets.
- **Cons:** Does not address the submitter's core concern that RESTful interactions would be easier to implement.

### Disposition B: Persuasive — Switch to RESTful Operations

Agree with the submitter that enroll and unenroll should be modeled as RESTful operations (e.g., `$enroll-tag`, `$unenroll-tag`) rather than FHIR Messages. This would simplify implementation, align with the predominant FHIR interaction paradigm, and reduce the implementation burden on systems that already support REST but not FHIR Messaging infrastructure.

- **Pros:** Lower implementation barrier; aligns with mainstream FHIR REST paradigm; simplifies conformance requirements.
- **Cons:** Contradicts multiple rounds of WG deliberation; would require significant rework of the IG including profiles, examples, and the newly planned MessageDefinitions (FHIR-52436); may not fit the event-driven broadcast nature of RTLS.

### Disposition C: Not Persuasive — No Narrative Change Needed

Reject both the architectural change and the narrative modification. The current specification and language are adequate, and implementers should understand that FHIR Messaging is the intended paradigm.

- **Pros:** No work required; maintains the current specification as-is.
- **Cons:** Leaves confusing subscriber-like language in the narrative that the WG itself acknowledged was problematic; ignores a straightforward editorial improvement.

## Recommendation

**Disposition A: Accept as Not Persuasive with Modification (Affirm Resolution)**

The WG's resolution reflects a well-considered architectural decision after multiple discussions. The commitment to clarify subscriber-like language addresses the legitimate confusion without requiring a fundamental paradigm shift. The pattern of subsequent tickets (FHIR-52436 creating MessageDefinitions, FHIR-52433 fixing narrative) demonstrates the WG is actively improving the messaging approach, not ignoring the concerns about clarity and implementability. Affirming this resolution supports both the WG's design authority and the practical editorial improvement.
