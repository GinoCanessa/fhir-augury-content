## Description

The submitter requests that the RTLS specification be improved by increasing the number of profiles and examples. Specifically, the request is to:

1. Add an RTLS subscription notification profile showing the structure of an RTLS subscription notification.
2. Add profiles for each notification type defined in section 3, for both message-based and subscription-based notifications.
3. Add examples for each event type in both message and subscription format.

## Resolution Statement

> We are removing Subscriptions from the spec, so we are no longer planning to have additional events specific to Subscriptions.

The workgroup resolved this as **Not Persuasive** because the RTLS specification is being revised to remove Subscriptions entirely. Since the requested profiles and examples center on subscription-based notifications, the underlying mechanism is being eliminated from the spec, making the request moot.

## Analysis

This ticket was submitted during the STU 1 ballot of the RTLS IG (April 2023). At that time, the specification included both FHIR Messaging and Subscription-based notification mechanisms for RTLS events. The submitter reasonably identified a gap: the spec defined multiple notification types but lacked corresponding profiles and examples for each.

Since ballot, the RTLS workgroup has decided to **remove Subscriptions from the specification**, which fundamentally changes the scope. The resolution correctly notes that subscription-specific profiles and examples are no longer needed because the mechanism itself is being eliminated. However, the ticket also requested profiles/examples for **message-based** notifications, which the resolution does not explicitly address.

The related ticket FHIR-41167 (advocating simpler RESTful interactions over FHIR Messaging) suggests the spec's interaction patterns are undergoing broader architectural revision. FHIR-52566 raises overlap concerns with the Devices WG, adding further uncertainty about what notification patterns will ultimately remain.

## Proposed Dispositions

### Disposition A: Agree with "Not Persuasive"
**Rationale:** The removal of Subscriptions from the RTLS spec eliminates the primary mechanism the submitter requested profiles and examples for. The workgroup has taken a clear architectural direction away from Subscriptions. While the submitter also mentioned message-based notification profiles, the ongoing architectural revisions (see FHIR-41167) make it premature to mandate additional profiles until the interaction patterns stabilize. The workgroup can address profile/example coverage in a future ballot cycle once the spec's notification architecture is finalized.

### Disposition B: Persuasive with Modification — add message-based profiles/examples only
**Rationale:** While the Subscription portion of the request is correctly rejected, the submitter also asked for profiles and examples for **message-based** notifications, which remain in the spec. The resolution should acknowledge this portion and commit to adding message-based profiles and examples for each event type defined in section 3. This would improve implementer guidance for the notification mechanisms that are being retained.

### Disposition C: Not Persuasive with Modification — acknowledge gap and defer
**Rationale:** Accept the "Not Persuasive" outcome for subscription-based content, but add a tracker item or note acknowledging that comprehensive profile and example coverage for the retained notification mechanisms (messaging and/or RESTful) should be addressed in a future cycle. This balances the architectural uncertainty with the submitter's valid concern about specification completeness.

## Recommendation

**Disposition A: Agree with "Not Persuasive"**

The resolution is sound. The spec is undergoing significant architectural changes — removing Subscriptions (this ticket's primary concern) and potentially simplifying messaging patterns (FHIR-41167). Mandating additional profiles and examples now would be premature while the notification architecture is in flux. The workgroup should finalize the interaction model first and then ensure adequate profile and example coverage in a subsequent ballot cycle. The reporter's comment ("What kind of input do you require?") suggests they may be open to further engagement, but the current resolution correctly reflects the spec's direction.
