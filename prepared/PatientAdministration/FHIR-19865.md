Eric Haas proposes that the `contains` search parameter on Location (which determines whether a geographic point falls within a boundary defined by the `location-boundary-geojson` extension) should be redefined as an **operation** rather than a search parameter. The argument is that this functionality requires the server to perform a geometric calculation against base64-encoded or URL-linked GeoJSON attachment data, which is not normally searchable per the FHIR search specification. The submitter also questions the priority of this feature given that zip code lists or other address elements could achieve similar results.

The resolution ("Not Persuasive with Modification") notes that keeping this as a search parameter is intentional: search parameters support chaining and combination with other parameters, capabilities that operations lack. The resolution also points out that zip code searching cannot handle arbitrary geo-location queries or shape boundaries.

## Details

### Core Issue
The `contains` search parameter (defined in VhDir, later adopted by NDH and DaVinci PDex Plan Net) targets `Location.extension.where(url='http://hl7.org/fhir/StructureDefinition/location-boundary-geojson')`, which is an `Attachment` containing GeoJSON boundary data. The submitter argues this is semantically an operation because:

1. **Calculation required** — The server must determine whether a given coordinate point falls within a polygon boundary, which is a non-trivial geometric computation.
2. **Attachment not searchable** — The FHIR search specification does not define how to search Attachment-type elements directly.
3. **Encoding complexity** — The GeoJSON data is base64-encoded or URL-referenced, adding another layer of complexity for search indexing.

### Resolution Rationale
The resolution defends keeping `contains` as a search parameter because:
- Search parameters can be **chained** (e.g., `PractitionerRole?location.contains=-83.69|42.25`) and **combined** with other search criteria — operations cannot.
- The search parameter is intended to be rolled into the **core FHIR spec** (which happened in R5, documented in Location resource notes).
- Zip code searching is not a substitute, as it cannot express arbitrary geographic boundaries or handle incoming geo-coordinates.

## Proposed Dispositions

### Disposition A: Accept the Current Resolution (Not Persuasive with Modification)

Affirm the existing resolution. The `contains` functionality is correctly modeled as a search parameter, not an operation. The core advantages of search parameters — chaining, combination with other parameters, and use in `_include`/`_revinclude` — are critical for directory-style queries (e.g., "find practitioners at locations that contain this coordinate"). Since the ticket was resolved, this search parameter has been incorporated into the R5 core specification with proper documentation (see `location-notes.xml` and [Location §8.7.6.1.2](https://hl7.org/fhir/location.html#8.7.6.1.2)), addressing the original concern about lack of searchability guidance. The `special` search parameter type was specifically designed for cases like this where custom server-side logic is required.

**Outcome:** No spec change. Existing resolution stands.

### Disposition B: Accept with Additional Implementation Guidance

Affirm the resolution (keep as search parameter), but acknowledge the submitter's valid concern about implementability by ensuring the R5+ documentation explicitly addresses:
1. How coordinate values are passed in the search query (format, order, type).
2. How the server is expected to decode and evaluate base64/URL-referenced GeoJSON attachments.
3. A clear note that this is a `special` type search parameter requiring custom server logic.

This would align with the concerns raised in FHIR-41963, which specifically calls out insufficient implementation guidance for the same search parameter.

**Outcome:** No structural change, but a tracker item for documentation enhancement if not already covered in R5.

### Disposition C: Decline — Reopen and Redefine as Operation

Reopen the ticket and redefine `contains` as a FHIR operation (e.g., `$contains` on Location). This would align with the submitter's original request and the observation that geometric containment testing is fundamentally a computation, not a search. However, this would sacrifice chaining and parameter combination capabilities, break existing implementations in VhDir, NDH, and PDex Plan Net, and contradict the R5 core spec decision.

**Outcome:** Significant breaking change across multiple IGs and core spec.

## Recommendation

**Disposition A: Accept the Current Resolution.** The resolution is well-reasoned and has been validated by subsequent specification work. The `contains` search parameter has been successfully integrated into the R5 core spec with documentation, the `special` search type provides the necessary flexibility for server-side geometric computation, and the chaining/combination capabilities are essential for real-world directory queries. The concerns about implementability raised in this ticket and the related FHIR-41963 have been substantially addressed in R5 documentation. Reopening this to change the paradigm from search parameter to operation would be disruptive with no clear benefit.
