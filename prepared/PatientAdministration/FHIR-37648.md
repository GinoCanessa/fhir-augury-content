## Summary

Long-stay hospital admissions produce Encounter resources with very large `participant` lists. While a small number of participants are currently active, hundreds or thousands of historical participants (e.g., nurses from months ago) accumulate over time. The submitter requests a better approach for managing these large repeating backbone elements within Encounter resources.

The ticket was discussed at PA WGMs in 2022, 2023, and 2024, and was resolved as **Persuasive** at the January 2024 WGM. The resolution points to the "[Operations for Large Resources](https://hl7.org/fhir/operations-for-large-resources.html)" mechanism (`$filter`, `$add`, `$remove`) as the preferred direction, but identifies several unresolved gaps that require FHIR Infrastructure input. The ticket is labeled **MovingToGuide**, indicating content is intended to move to an implementation guide.

- **Key points:**
  - Aleksandr Kislitsyn raised the general problem of large FHIR resources across multiple interaction patterns (CRUD, search, subscriptions), affecting ValueSet, CodeSystem, ConceptMap, Group, List, and others.
  - Gino Canessa mentioned a Cascais WGM proposal for NDJSON-based streaming of large resources (first row = resource minus large arrays; subsequent rows = array values). Grahame Grieve expressed concern about nesting.
  - Jim Steel recommended defaulting search results to `_summary` resources and highlighted validation performance as a related concern.
  - Michael Lawley suggested `_elements`, GraphQL, and HTTP Range headers (e.g., `Range: CodeSystem.concept=0-1000`) as alternatives, with PATCH for updates.
  - **Cooper Thompson explicitly linked FHIR-37648**, noting PA is "stuck" on this Jira for R6 and asking whether FHIR-I will land FHIR-52912 on R6 timelines.
  - **Grahame Grieve responded** that "dealing with these issues is in the incubator space," implying it may not be on the R6 critical path.

### 2. [fhir/infrastructure-wg] "Large Resources" - FHIR-52912 (Oct 2025)
- **URL:** https://chat.fhir.org/#narrow/stream/fhir%2Finfrastructure-wg/topic/%22Large%20Resources%22%20-%20FHIR-52912
- **Participants:** Cooper Thompson, Grahame Grieve, Gino Canessa, Brian Postlethwaite
- **Key points:**
  - Gino Canessa confirmed the operations for large resources are **moving into the "API Incubator" project** for continued iteration.
  - Grahame Grieve stated the questions from PA "have answers now, though they have variable amounts of documentation": use `HEAD` to detect large resources, `_elements` or `_summary` to request partial content, and PATCH for updating core properties (not the large list).
  - Cooper Thompson pushed back: requiring every client to `HEAD` before `GET` is impractical since <1% of Encounters are large; auto-subsetting with server-initiated filtering would be preferable.
  - Brian Postlethwaite noted PATCH requires knowing where a value is in the list, which requires the entire resource—a circular dependency for large resources.
  - Grahame acknowledged the shift from current client behavior but argued the problem drives the change.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Add PA-Specific Guidance to Implementation Guide

**Action:** Accept the existing Persuasive resolution. Since the label is already "MovingToGuide," author PA-specific implementation guidance for handling large Encounter resources in the relevant IG. Guidance should cover: (1) servers MAY auto-subset Encounter.participant for resources exceeding a threshold, applying the SUBSETTED meta tag; (2) clients SHOULD use the `$filter` operation to retrieve historical participants with date-range filtering; (3) clients SHOULD use `$add`/`$remove` to modify participant lists on large Encounters; (4) document the interaction with PATCH for updating core Encounter properties independently of the large list. Coordinate with FHIR-I on the API Incubator work for alignment.

**Pros:** Provides actionable guidance for implementers now, without waiting for FHIR-I to finalize the infrastructure-level specification. Aligns with the MovingToGuide label. Addresses the R6 urgency raised by Cooper Thompson.

**Cons:** The underlying infrastructure mechanisms (`$filter`, `$add`, `$remove` for non-Group/List resources) are still in the API Incubator and may change, requiring IG updates. Auto-subsetting behavior is not yet standardized—PA guidance may diverge from what FHIR-I eventually specifies.

### Disposition B: Defer — Block on FHIR-52912 Resolution

**Action:** Keep the ticket in Triaged status. Do not produce PA-specific guidance until FHIR Infrastructure resolves FHIR-52912 and the "operations for large resources" mechanisms are stabilized (detection, complete retrieval, core element update, auto-subsetting). Re-evaluate when the API Incubator work matures.

**Pros:** Avoids publishing guidance that may conflict with future infrastructure decisions. Ensures PA's approach is fully aligned with the broader FHIR ecosystem solution. Reduces rework risk.

**Cons:** Leaves PA implementers without guidance for R6, potentially for an extended period given the incubator status. Cooper Thompson has flagged this as important for R6 and impacting normative resources. Long-stay Encounter scenarios remain problematic in production systems.

### Disposition C: Not Persuasive — Recommend Existing Mechanisms

**Action:** Close as Not Persuasive. Advise implementers to use existing FHIR mechanisms: `_summary` and `_elements` to limit response content; `_include`/`_revinclude` with separate participant-tracking resources (e.g., CareTeam); PATCH for targeted updates. No spec changes or new guidance needed—this is an implementation concern.

**Pros:** No specification changes required. Existing tools theoretically address the problem. Avoids adding complexity to the spec.

**Cons:** Does not address the real-world scalability issues reported. Existing mechanisms are acknowledged as insufficient by both PA and FHIR-I participants (e.g., `_elements` still returns the full large list if participants are requested; PATCH requires a prior complete GET). Contradicts the already-agreed Persuasive resolution and years of WG discussion.

## Recommendation

**Disposition A: Persuasive with Modification — Add PA-Specific Guidance to Implementation Guide.**

This ticket has been discussed for nearly four years across multiple WGMs and Zulip threads. The resolution was already agreed as Persuasive in January 2024, and the label MovingToGuide indicates the intended path. While the FHIR-I infrastructure work (FHIR-52912) is in the API Incubator and may not land on R6 timelines (per Grahame Grieve's December 2025 comment), PA implementers face real production challenges with large Encounter resources today. Authoring IG-level guidance that recommends the `$filter`/`$add`/`$remove` approach—while clearly noting its incubator status and dependency on FHIR-52912—gives implementers a practical path forward without requiring core spec changes. The guidance can be updated as the infrastructure work matures.
