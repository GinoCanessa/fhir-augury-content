## Summary

The submitter requests that the example Generated Narrative in the Validated Healthcare Directory (VHD) Implementation Guide be made more user-friendly. Specifically, the submitter suggests elevating the note text — "This care team was founded to optimize the management of complex cardiac cases" — to serve as the description of the example as a whole, rather than leaving it buried within the narrative. The ticket was originally resolved as "Considered for Future Use" in October 2018 (vote: 6-0-0, Alex Kontur/Iryna Roy) on the grounds that narrative generation is performed by the HL7 IG publishing tooling and the commenter should file a separate tracker for tooling changes. The ticket was subsequently re-opened in February 2025 post STU1 publication by lynn_laakso.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Modify the CareTeam example in the Validated Healthcare Directory IG so that the narrative includes a more descriptive, human-readable summary. Specifically, elevate the "note" text ("This care team was founded to optimize the management of complex cardiac cases") to appear prominently as the example's descriptive text within the narrative section, providing readers with immediate context about the purpose of the example CareTeam resource.
#### Justification
The original request is reasonable — example narratives should help readers understand what the example represents. Since the ticket has been re-opened post STU1 publication, the work group has an opportunity to improve the IG's usability. The change is low-risk and would improve readability for implementers reviewing the directory examples. The previous resolution deferred the issue to IG publishing tooling, but the IG authors can also manually author or augment example narratives to improve clarity, independent of the tooling.

---

### Disposition B: Alternative Approach
#### Proposal
Rather than modifying the generated narrative directly, add or enhance the `CareTeam.text` element in the example resource with a hand-authored narrative div that includes a user-friendly description. Additionally, consider adding a descriptive comment or introductory text in the IG's example section explaining the purpose of each example, rather than relying solely on the generated narrative.
#### Justification
This approach separates the concern of tooling-generated narratives from the IG content itself. By hand-authoring the narrative div in the example resource, the IG authors retain full control over the example's presentation without depending on changes to the IG publishing tooling. This also establishes a pattern that can be applied to other examples in the IG for consistency. It addresses the submitter's concern while respecting the original resolution's observation that the IG publisher controls narrative generation.

---

### Disposition C: Decline
#### Proposal
Close this ticket as "Not Persuasive" on the grounds that narrative generation is controlled by the HL7 IG publishing tooling, and the submitter should file a separate tracker against the IG publisher if tooling changes are desired.
#### Justification
The previous resolution (Considered for Future Use, 6-0-0) already determined that this is fundamentally a tooling concern rather than an IG content concern. The Generated Narrative is auto-produced by the IG publisher, and modifying it within the IG would create maintenance burden as future IG publisher updates could overwrite hand-authored narratives. The re-opening post STU1 does not introduce new information that would change the original analysis.

---

### Recommendation
**Recommended disposition:** B — Alternative Approach

The ticket has been re-opened post STU1 publication, signaling that the work group believes the issue merits reconsideration. The original resolution correctly identified that generated narratives come from IG publishing tooling, but the IG authors can still improve the example by hand-authoring the narrative div in the CareTeam example resource. This approach directly addresses the submitter's usability concern without requiring tooling changes or creating a dependency on IG publisher modifications. It is a pragmatic middle ground: it acknowledges the validity of the request while respecting the constraints of the auto-generation workflow. The change is minimal in scope, low-risk, and improves the IG's value to implementers reviewing directory examples.
