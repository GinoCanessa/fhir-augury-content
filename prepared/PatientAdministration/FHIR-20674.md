## Summary

The submitter requests removal of the `CareTeam.encounter` element from the VHDir (Validated Healthcare Directory) CareTeam profile. The `encounter` element was introduced to the base CareTeam resource in R4. Since the VHDir IG was originally built on STU3 (which did not have this element), it was not considered during the initial profile design. Now that VHDir is based on R4, the element appears in the continuous build. The submitter argues that a validated healthcare directory would not be expected to store local encounter-level data, so the element should be constrained to 0..0 in the VHDir CareTeam profile.

## Related Zulip Discussions

### [implementers > CareTeam without patient and encounter](https://chat.fhir.org/#narrow/stream/implementers/topic/CareTeam%20without%20patient%20and%20encounter) (2019-12-09)

A community discussion about using CareTeam when there is no specific patient or encounter context (e.g., standing code blue teams). Key points:
- **Lloyd McKenzie** confirmed that neither `subject` nor `encounter` are mandatory in the base resource, and CareTeam can represent standing teams without patient/encounter references.
- **Dave Carlson** pointed to the VHDir CareTeam profile as an example that already addresses this by constraining `subject` to 0..0.
- **Yunwei Wang** noted inconsistencies in the VHDir CareTeam profile between its background text and the snapshot table (e.g., `context` and `reasonReference` were described as removed but still appeared with non-zero cardinalities).

This thread reinforces the rationale behind FHIR-20674: the VHDir CareTeam profile is intended for directory-level use cases where encounter-specific data is out of scope.

## Proposed Dispositions

### Disposition A: Persuasive (Recommended)

**Action:** Constrain `CareTeam.encounter` to 0..0 in the VHDir CareTeam profile.

**Rationale:** The request is well-aligned with the stated design intent of the VHDir CareTeam profile. The profile already constrains `CareTeam.subject` to 0..0 on the grounds that patient-level data is out of scope for a validated healthcare directory. The `encounter` element represents even more granular, local clinical data than `subject`. The same rationale that justified removing `subject` applies with equal or greater force to `encounter`. The prior ticket FHIR-14468 (resolved Persuasive) already acknowledged contradictions in the profile's scope statements about element removal, and this change would bring the profile into closer alignment with its documented intent. Community discussion on Zulip also confirms that CareTeam without encounter context is a valid and intended use case.

---

### Disposition B: Persuasive with Modification

**Action:** Constrain `CareTeam.encounter` to 0..0, and also review other R4-introduced elements (e.g., `reasonReference`, `reasonCode`) for consistency with the profile's directory-level scope.

**Rationale:** While removing `encounter` is clearly appropriate, the VHDir CareTeam profile has known inconsistencies (per FHIR-14468 and the Zulip thread) between its scope description and actual constraints. A broader review of R4-added elements would ensure comprehensive alignment. However, this broader scope may be better addressed as a separate ticket to avoid delaying the straightforward `encounter` fix.

---

### Disposition C: Not Persuasive

**Action:** Leave `CareTeam.encounter` unconstrained in the VHDir CareTeam profile.

**Rationale:** The base CareTeam resource defines `encounter` with cardinality 0..1, meaning implementations are not required to populate it. The VHDir profile could simply document that the element is not expected to be used in directory contexts without formally constraining it to 0..0. This preserves flexibility for edge cases where a directory might optionally associate a care team with an encounter. However, this argument is weak given that the profile already constrains `subject` to 0..0 — leaving `encounter` unconstrained would be inconsistent.

---

## Recommendation

**Disposition A: Persuasive** — The request to constrain `CareTeam.encounter` to 0..0 is logically consistent with the existing VHDir CareTeam profile design, which already removes `subject`. A validated healthcare directory is not expected to store encounter-level clinical data. The change is straightforward, well-justified, and supported by prior ticket resolutions and community discussion.
