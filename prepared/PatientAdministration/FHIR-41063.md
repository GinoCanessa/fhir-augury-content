## Summary

This ticket requests that the official URL for RTLS-related terminology (likely event code systems or value sets) be anchored under `terminology.hl7.org` (HL7 Terminology / THO) rather than under `hl7.org/fhir`. The submitter (Ron G. Parker) questions the current URL anchoring practice for the RTLS specification's terminology artifacts.

The resolution description states that the work group agrees it makes sense for the list of "events" to eventually be published in THO, but proposes deferring this migration until after the upcoming RTLS 2.0 ballot, since the approach for using subscriptions vs. messaging has not fully stabilized yet.

## Details

This is one of a series of related tickets (FHIR-41063 through FHIR-41066) all raising the same question about URL anchoring for RTLS terminology in `terminology.hl7.org`. The broader pattern — requesting that value set and code system canonical URLs be moved from `hl7.org/fhir` to `terminology.hl7.org` (THO) — has been raised across multiple HL7 specifications by the same reporter.

Key context:
- **Resolution rationale:** The work group acknowledges the request has merit but defers action because the RTLS subscription/messaging approach is still evolving. Moving terminology to THO prematurely could create churn if event definitions change.
- **Sibling tickets:** FHIR-41064 (Not Persuasive), FHIR-41065 (Not Persuasive with Modification), and FHIR-41066 (Not Persuasive) address the same URL anchoring question for different RTLS terminology items.
- **Cross-specification precedent:** Similar requests against other IGs (FHIR-41050 for mCODE, FHIR-35454 for US Core) were resolved as "Not Persuasive," indicating that moving canonical URLs to THO is not automatically accepted across HL7 specifications.

## Proposed Dispositions

### Disposition A: Considered for Future Use ✅

Accept the current resolution. The work group acknowledges the request to move RTLS event terminology canonical URLs to `terminology.hl7.org` (THO) is reasonable in principle, but defers the migration until after the RTLS 2.0 ballot when the subscription/messaging approach has stabilized. This avoids premature URL changes that could cause breaking changes if event definitions are still evolving.

**Rationale:** This is a pragmatic approach that validates the submitter's concern while protecting implementers from churn. It sets a clear trigger point (post-2.0 ballot) for revisiting.

### Disposition B: Not Persuasive with Modification

Decline the request to move the URL now, but add a tracker item or informational note in the RTLS IG stating the intent to migrate event terminology to THO in a future version. This provides transparency without committing to a specific timeline tied to the 2.0 ballot.

**Rationale:** Sibling ticket FHIR-41065 was resolved as "Not Persuasive with Modification," and aligning this ticket's disposition would create consistency. However, this disposition provides less commitment to future action than "Considered for Future Use."

### Disposition C: Not Persuasive

Decline the request outright. The RTLS specification's terminology canonical URLs are appropriately anchored under `hl7.org/fhir` as an IG-defined artifact, consistent with how other IGs handle their terminology. Migration to THO is a separate process managed by the UTG team and should be requested through UTG channels if appropriate.

**Rationale:** This aligns with the precedent set by FHIR-41050 (mCODE) and FHIR-35454 (US Core), where identical requests were declined. However, this ignores the work group's own stated agreement that the move makes sense eventually.

## Recommendation

**Disposition A: Considered for Future Use** is recommended. The work group's own resolution description explicitly acknowledges the request has merit and commits to revisiting after the RTLS 2.0 ballot. This resolution honestly reflects the work group's intent, gives the submitter a meaningful path forward, and avoids premature changes to canonical URLs during an unstable specification phase. It is also the resolution already recorded on the ticket, suggesting prior work group consensus.
