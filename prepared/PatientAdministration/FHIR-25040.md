## Description

Many SearchParameters in the VhDir (Validated Healthcare Directory) IG that involve extensions have incorrectly coded FHIRPath expressions. The reporter requests that all extension-based SearchParameters be checked and tested using FHIRPath.

**Specific example cited:** The `endpoint-usecase-standard` SearchParameter
([source](http://build.fhir.org/ig/HL7/VhDir/SearchParameter-endpoint-usecase-standard.json.html))
uses the incorrect expression:

```
Endpoint.extension('http://hl7.org/fhir/uv/vhdir/StructureDefinition/endpoint-usecase').extension('standard')
```

The corrected expression should use the `.where(url='...')` syntax:

```
Endpoint.extension.where(url='http://hl7.org/fhir/uv/vhdir/StructureDefinition/endpoint-usecase').extension.where(url='standard')
```

The issue is that the `extension('url')` function-call syntax, while defined as shorthand in FHIRPath, may not be universally supported across implementations. The `.extension.where(url='...')` pattern is the more explicit and broadly compatible approach, especially for nested extension navigation.

## Proposed Dispositions

### Disposition 1: Persuasive

**Action:** Accept the change as described. Correct all extension-based SearchParameter FHIRPath expressions in the VhDir IG to use the `.extension.where(url='...')` syntax instead of the `extension('url')` function-call syntax.

**Rationale:** The reporter has identified a clear, reproducible issue. The `extension('url')` function-call syntax, while defined as shorthand in FHIRPath, is not reliably implemented across FHIR servers and tooling. Using `.extension.where(url='...')` is the explicit, universally understood pattern. Multiple related tickets (FHIR-19729, FHIR-23068) have established precedent for correcting SearchParameter expressions, and both were resolved as Persuasive. The fix is straightforward and improves interoperability.

### Disposition 2: Persuasive with Modification

**Action:** Accept the general issue, but require that the submitter or IG maintainers provide an explicit enumeration of all affected SearchParameters in VhDir with corrected expressions, and that each corrected expression be validated against a FHIRPath engine before publication.

**Rationale:** The ticket states "all of them should be checked and tested" but only provides one example. For a thorough fix, the workgroup should ensure every extension-based SearchParameter in VhDir is explicitly listed, corrected, and tested. This provides accountability and traceability. The modification adds rigor without rejecting the valid concern.

### Disposition 3: Not Persuasive

**Action:** Reject the change. The `extension('url')` syntax is valid FHIRPath per the FHIR specification and should be supported by conformant implementations.

**Rationale:** The FHIRPath specification defines `extension(url : string)` as a built-in function equivalent to `extension.where(url='...')`. If implementations do not support this function, the bug is in the implementation, not in the SearchParameter definition. Changing the IG to work around non-conformant implementations sets a poor precedent.

---

## Recommendation

**Disposition 1: Persuasive**

While the `extension('url')` shorthand is technically valid FHIRPath, practical interoperability demands that SearchParameter expressions work across the broadest range of implementations. The `.where(url='...')` syntax is unambiguous, widely supported, and consistent with how other IGs and core FHIR SearchParameters handle extension navigation. Prior precedent (FHIR-19729 resolved Persuasive, FHIR-23068 resolved Persuasive with Modification) supports correcting SearchParameter expressions for practical usability. The fix is low-risk and high-impact for implementers consuming the VhDir IG.
