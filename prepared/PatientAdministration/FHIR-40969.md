## Description

The submitter requests that the RTLS Device example include a `udiCarrier` element, to demonstrate how UDI (Unique Device Identifier) information can be represented on a Device resource within the RTLS (Real Time Location Services) specification.

## Resolution Description

The work group resolved this as **Not Persuasive** with the following rationale:

> The RTLS device is for a badge, which, as far as we know, isn't considered a medical device and wouldn't have a UDI. However, that RTLS badge could be affixed to a medical device that *would* have a UDI. We are updating the relevant example to add UDI as part of [FHIR-40970](https://jira.hl7.org/browse/FHIR-40970), but we don't believe the RTLS Tag Device example would have a UDI, so we find this ticket not persuasive.

In other words, the UDI use case is being addressed in the companion ticket FHIR-40970 (Location Update Event example) where a medical device with a UDI is more appropriate, rather than in the RTLS Tag Device example which represents a non-medical badge/tag.

## Proposed Dispositions

### Disposition A: Affirm "Not Persuasive"

**Rationale:** The work group's reasoning is sound. The RTLS Tag Device example represents a location-tracking badge/tag, not a medical device subject to UDI requirements. Adding `udiCarrier` to this example would be misleading, as it would imply that non-medical RTLS tags carry UDI identifiers. The appropriate UDI use case — a medical device being tracked via RTLS — is correctly addressed in the companion ticket FHIR-40970 (resolved as Persuasive with Modification). The submitter's underlying goal (showing UDI in RTLS examples) is being satisfied through that ticket.

### Disposition B: Change to "Persuasive with Modification" — Add Optional udiCarrier with Explanatory Text

**Rationale:** Even though an RTLS badge itself is not a UDI-bearing device, implementers may attach RTLS tags to medical devices that do have UDIs. Adding an optional `udiCarrier` element to the RTLS Device example (with a clear comment that this applies only when the tracked item is a UDI-bearing medical device, not the tag itself) could improve the example's educational value. This would complement FHIR-40970 rather than duplicate it.

### Disposition C: Change to "Persuasive" — Add udiCarrier to the RTLS Device Example

**Rationale:** Accept the submitter's request at face value and add `udiCarrier` to the RTLS Device example. This provides a complete representation of Device capabilities in the RTLS context. However, this risks conflating the RTLS tag (a non-medical tracking device) with a UDI-regulated medical device.

## Recommendation

**Disposition A: Affirm "Not Persuasive"**

The work group's original resolution is well-reasoned and correctly distinguishes between an RTLS badge/tag (non-medical, no UDI) and a medical device being tracked (which appropriately carries UDI). The submitter's intent is already addressed by FHIR-40970, which was accepted as Persuasive with Modification. Adding UDI to the RTLS Tag Device example would be semantically incorrect for the entity it represents. Affirming "Not Persuasive" maintains clarity in the examples while ensuring UDI coverage exists in the right place.
