## Description

The submitter argues that the Identity Matching Guide refers only to first and last name, implying the patient's current name. By not including previous names (including variations and aliases), the guide limits the number of good matches. Patient names change frequently (e.g., marriage) and can be stored differently across systems. If a patient recently changed their last name and one system has the new name while another has the old, the current matching framework would fail to find a match. Including previous names in match requests would help establish matches across systems with stale data.

## Analysis

### Core Issue
The submitter's concern is valid and well-documented: patient names change (marriage, divorce, legal name change), and different systems may hold different versions of the same patient's name at any given time. The current Identity Matching Guide discusses how to *store* previous names but does not explicitly assign a matching weight or provide specific algorithmic guidance for using previous names in the `$match` operation.

### Prior Resolution Context
This ticket was previously resolved as "Considered for Future Use" (July 2025, 2-0-0 vote) with the WG acknowledging the value of previous names but citing **patient privacy concerns** as the reason for not assigning a weight. The ticket was then re-opened post STU2 publication (December 2025) by Lynn Laakso, indicating the WG wants to reconsider this for a future version.

### Predecessor Ticket
FHIR-36802 (same reporter, same request) was resolved as "Not Persuasive with Modification" and published, meaning *some* guidance on previous names was already added. The current ticket is asking for the guide to go further—specifically to include previous names in matching weight recommendations.

### Privacy Concern
The WG's prior resolution explicitly noted the tension between the matching value of previous names and patient privacy. Some patients may not want their previous name disclosed (e.g., gender transition, domestic violence situations). Any guidance must balance matching effectiveness with privacy protections.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Add guidance in the Identity Matching IG recommending that implementers *consider* previous names (HumanName with `use: "old"`) in match requests and responses, without assigning a specific numeric weight. Include a privacy caveat noting that patients should be able to control whether previous names are shared, and that responders should be aware of the sensitivity of name history.

**Rationale:** This addresses the submitter's concern by making previous names an explicit part of matching guidance while respecting the WG's prior privacy concerns. It stops short of mandating a specific weight but encourages implementers to use available name history. Aligns with the existing guidance from FHIR-36802's resolution while going further for STU3.

### Disposition B: Considered for Future Use
**Action:** Retain the prior resolution. Acknowledge the request's validity but defer assigning any matching weight or specific guidance for previous names until privacy implications are more fully explored, potentially in coordination with patient consent frameworks.

**Rationale:** The WG already considered this in July 2025 and decided to defer. The privacy concerns remain unresolved—specifically, how to introduce previous-name matching value while protecting patients who have changed their names for sensitive reasons. Current guidance already mentions storing previous names. Wait for privacy framework maturity before acting.

### Disposition C: Not Persuasive with Modification
**Action:** Clarify in the IG that previous names stored in `HumanName(use: "old")` *should* be included in Patient resources submitted via `$match` (as Julie Maas noted is already implied), but explicitly state that the guide does not assign a weight to previous names due to privacy considerations. Add a brief note explaining why.

**Rationale:** The existing guidance already covers storing and transmitting previous names. Making this more explicit (without changing matching weights) addresses the submitter's concern about visibility while maintaining the WG's privacy-first position.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The ticket has been re-opened post STU2 publication, signaling WG intent to address it in a future version. The predecessor ticket (FHIR-36802) already established that some guidance was needed and was applied. The real-world problem is well-documented: name changes are common, cross-system matching fails when only current names are considered, and the FAST Identity Zulip discussions confirm implementer demand. The privacy concern is valid but can be addressed through guidance language (SHOULD-level recommendation with privacy caveats) rather than avoidance. Deferring indefinitely (Disposition B) risks perpetuating a known matching gap in a published standard, while Disposition C is too conservative given the WG's decision to re-open.
