## Summary

The submitter requests adding a `udiCarrier` element in the Location Update Event example within the RTLS specification. The resolution acknowledges this as **Persuasive with Modification**, clarifying that the Location Update message communicates information about the RTLS badge itself (which would not have a UDI), not the medical device it may be affixed to. Instead, the Enroll Tag message is the appropriate place to show UDI carrier information, since that message associates an RTLS badge with a tracked medical device (e.g., an infusion pump). The resolution commits to creating an example Enroll Tag message demonstrating this association.

## Details

Two types of devices are relevant in the RTLS context:

1. **RTLS Badge** — The tracking tag itself, which would not carry a UDI.
2. **Medical Device** (e.g., infusion pump) — Has the RTLS badge affixed so staff can track its location.

The Location Update message type only communicates RTLS badge positioning data, so including `udiCarrier` there is not appropriate. However, the **Enroll Tag** message type is where the RTLS badge is associated with a medical device, making it the correct place for a `udiCarrier` example.

The resolution proposes creating an Enroll Tag example that demonstrates how to associate an RTLS badge with a medical device like an infusion pump, including UDI information for the medical device.

## Proposed Dispositions

### Disposition A: Accept (Persuasive with Modification)
**Rationale:** Agree with the existing resolution. The Location Update Event is not the right place for `udiCarrier` since it only conveys RTLS badge position data, not medical device identity. However, the Enroll Tag message — which associates an RTLS badge with a tracked medical device — is the appropriate context. Create a new Enroll Tag example that shows an RTLS badge being associated with a medical device (e.g., infusion pump) and includes the `udiCarrier` element on the medical device's Device resource.

**Concrete Changes:**
- Add a new example Bundle for the Enroll Tag message in the RTLS IG.
- The example should include a Device resource for the medical device (infusion pump) with a populated `udiCarrier` element.
- The example should include a Device resource for the RTLS badge without `udiCarrier`.
- Include narrative text explaining the association pattern.

### Disposition B: Broaden to Cover Multiple Message Types
**Rationale:** Accept the modification but go further: in addition to the Enroll Tag example, add guidance text across all RTLS message type pages clarifying when and where UDI information is appropriate. This would prevent future confusion about which message types should carry device identification data.

**Concrete Changes:**
- Create the Enroll Tag example as in Disposition A.
- Add a guidance section (or an informational note) to the Location Update Event page explicitly stating that `udiCarrier` is not expected in this message type.
- Add similar guidance to other RTLS message type pages as appropriate.

### Disposition C: Decline (Not Persuasive)
**Rationale:** Decline the request entirely. The RTLS specification already distinguishes between badge tracking and device association messages. The existing profiles and message definitions are sufficient; adding UDI examples may imply that RTLS systems must support UDI tracking, which is beyond the core RTLS scope. Companion tickets FHIR-40968 and FHIR-40969 (requesting `udiCarrier` in the Tag Device Profile and RTLS Device example respectively) were both resolved as Not Persuasive, suggesting the work group has already expressed skepticism about adding UDI to RTLS artifacts.

## Recommendation

**Disposition A: Accept (Persuasive with Modification)**

The existing resolution is well-reasoned and appropriately scoped. The submitter's underlying need — showing how UDI information can be conveyed in RTLS workflows — is valid, and the Enroll Tag message is the correct place to demonstrate this. The companion tickets (FHIR-40968, FHIR-40969) were declined because they targeted the wrong artifacts (profile and device example), not because UDI in RTLS is inherently inappropriate. Creating an Enroll Tag example with `udiCarrier` directly addresses the use case of tracking UDI-bearing medical devices via RTLS badges without misrepresenting what the Location Update message conveys.
