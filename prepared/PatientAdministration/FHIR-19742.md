## Summary

This ticket reports an inconsistency in the Validated Healthcare Directory (VHDir) StructureDefinitions for the **NewPatients** extension on the **Location** and **PractitionerRole** profiles. The extension's narrative description already states its cardinality as `0..*`, but the StructureDefinition itself does not reflect this. The reporter (Eric Haas) identifies this as an error and requests that the SD be corrected to match the documented cardinality of `0..*`.

The ticket has been in **Triaged** status since 2018 and remains unresolved. A duplicate ticket ([FHIR-19743](https://jira.hl7.org/browse/FHIR-19743)) was filed simultaneously and marked as Duplicate.

---

## Details

The NewPatients extension is used across the VHDir profiles to indicate whether a location or practitioner role is accepting new patients. It was designed to support multiple entries (e.g., different accepting-patients statuses per network), which requires `0..*` cardinality. However, the StructureDefinition constrains it more tightly than the accompanying description states.

This discrepancy has been a recurring source of confusion in the FHIR community, as evidenced by subsequent tickets ([FHIR-43796](https://jira.hl7.org/browse/FHIR-43796)) and Zulip discussions questioning the cardinality semantics of the NewPatients extension. The downstream Da Vinci PDex Plan-Net IG inherited much of this extension design, and questions about `0..*` cardinality and its interaction with the network reference (`0..1`) have continued to surface.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

**Action:** Correct the StructureDefinitions for the NewPatients extension on both the Location and PractitionerRole VHDir profiles to set cardinality to `0..*`, matching the existing narrative description.

**Rationale:** The reporter correctly identifies a straightforward inconsistency — the description already documents `0..*` cardinality, but the SD does not match. This is an error correction, not a design change. The downstream PDex Plan-Net IG already uses `0..*` for its clone of this extension, and multiple subsequent tickets (FHIR-25355, FHIR-43796) assume and build upon `0..*` semantics. Correcting this aligns the SD with documented intent.

### Disposition B: Accept with Clarification (Persuasive with Modification)

**Action:** Correct the cardinality to `0..*` in the StructureDefinitions as requested, and additionally add usage guidance explaining when multiple repetitions are appropriate (e.g., different accepting-patients status per network or patient type). Reference the related NewPatientProfile sub-extension to clarify how repeated entries should be differentiated.

**Rationale:** While the cardinality fix itself is unambiguous, the ongoing community confusion (FHIR-43796, Zulip discussions) shows that simply setting `0..*` without guidance leaves implementers uncertain about when and how to use multiple entries. Adding short usage notes would prevent future questions.

### Disposition C: Decline (Not Persuasive)

**Action:** Leave the StructureDefinition cardinality as-is and instead update the narrative description to match the current SD constraint.

**Rationale:** If the original SD constraint was intentional (e.g., `0..1`), the error may be in the description rather than the SD. However, this is unlikely given that the downstream PDex Plan-Net IG uses `0..*`, FHIR-25355 was resolved as Persuasive with Modification to extend NewPatients usage, and the design intent supports multiple entries per resource.

---

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The cardinality correction from the current SD value to `0..*` is clearly warranted — it is an error fix aligning the SD with its own description. However, given the documented pattern of implementer confusion about the semantics of repeated NewPatients entries (as seen in FHIR-43796 and multiple Zulip threads), adding brief usage guidance alongside the fix would deliver more lasting value. The VHDir IG should also note that the duplicate ticket FHIR-19743 is addressed by this change.
