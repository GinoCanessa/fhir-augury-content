## Description

The submitter questions why the [messageheader-response-request](https://hl7.org/fhir/R5/valueset-messageheader-response-request.html) value set does not use the same codes as the V2 table [v2-0155](https://terminology.hl7.org/ValueSet-v2-0155.html), given that the extension is mimicking a V2 construct. The V2 codes are:

- **AL** = always
- **ER** = on-error
- **NE** = never
- **SU** = on-success

The current FHIR value set (`messageheader-response-request`) defines its own code system with codes such as `always`, `on-error`, `never`, and `on-success` rather than reusing the V2 code values.

## Analysis

### Core Issue

The request asks HL7 to replace the FHIR-native code values in the `messageheader-response-request` value set (`always`, `on-error`, `never`, `on-success`) with the equivalent V2 table 0155 codes (`AL`, `ER`, `NE`, `SU`). The rationale is that the extension mimics V2 behavior, so it should reuse V2 terminology for consistency.

### Key Considerations

1. **Breaking Change**: Grahame Grieve's comment explicitly notes this would be a breaking change. The current codes are already published and in use across FHIR R4 and R5 implementations.

2. **Historical Precedent**: FHIR-13851 (2017) aligned the extension with `MessageDefinition.responseRequired`, establishing the current FHIR-native codes. This was a deliberate design choice that has been stable for years.

3. **FHIR Design Philosophy**: FHIR generally defines its own code systems with human-readable codes rather than reusing terse V2 code values. The FHIR codes (`always`, `on-error`, `never`, `on-success`) are more self-documenting than the V2 equivalents (`AL`, `ER`, `NE`, `SU`).

4. **Concept Mapping Exists**: V2-to-FHIR concept maps can bridge the two code systems without requiring the FHIR value set to adopt V2 codes directly.

5. **Active Area of Spec**: FHIR-55247 and FHIR-55474 show that the MessageHeader response code space is under active evolution. Introducing additional disruptive changes could compound migration burden.

## Proposed Dispositions

### Disposition A: Not Persuasive

**Rationale:** While the submitter raises a reasonable philosophical point about code alignment with V2, changing the codes would be a breaking change to a value set that has been stable since at least R4. FHIR intentionally defines its own human-readable code systems rather than importing V2's terse codes. The existing concept mapping infrastructure (V2-to-FHIR maps) already provides the necessary interoperability bridge. No change is warranted.

**Wording:** "The committee appreciates the desire for alignment with V2 table 0155. However, the FHIR `messageheader-response-request` value set has been stable and in active use since R4. Changing these codes would constitute a breaking change. FHIR follows a design principle of using human-readable, self-documenting codes rather than importing V2 code values directly. Interoperability with V2 is addressed through V2-to-FHIR concept maps."

### Disposition B: Not Persuasive with Mod — Add ConceptMap

**Rationale:** Acknowledge the alignment concern without changing the codes. Instead, ensure a formal ConceptMap from V2 table 0155 to the FHIR `messageheader-response-request` value set is published (if not already available), making the mapping explicit for implementers working with both V2 and FHIR messaging.

**Wording:** "The committee agrees that clear mapping between V2 table 0155 and the FHIR `messageheader-response-request` codes is valuable, but changing the FHIR codes would be a breaking change. The committee will ensure an appropriate ConceptMap between V2-0155 and the FHIR response-request codes is available to implementers."

### Disposition C: Persuasive with Mod — Deprecate and Create New Code System

**Rationale:** Accept the submitter's point in principle by deprecating the current codes and introducing the V2 codes alongside them in a transitional approach. This would allow a gradual migration while preserving backward compatibility.

**Wording:** "The committee acknowledges the value of aligning with V2 table 0155. However, rather than directly replacing the existing codes (a breaking change), the V2 codes will be added as additional concepts in the value set with a mapping to the existing codes, and the existing FHIR-specific codes will be marked as deprecated over a transition period."

## Recommendation

**Disposition A: Not Persuasive**

The breaking change concern raised by Grahame Grieve is dispositive. The current FHIR codes are self-documenting, stable, semantically equivalent to the V2 codes, and in widespread use. FHIR's design philosophy intentionally favors readable codes over terse V2 legacy codes. Concept maps already provide the necessary interoperability bridge for V2-to-FHIR migration. Introducing a code change — whether immediate or gradual — would create migration burden for all current users without a proportionate benefit. The committee should decline the change while acknowledging the submitter's interoperability concern.
