## Description

The `DiagnosticReport-select` event currently uses `reference` as the type for its `select` context key (with cardinality `1..*`), where each selected resource is represented as a separate `select` entry in the context array. However, `resources` is not an allowed datatype in FHIRCast context types, nor is `references` (as a plural/collection type).

The reporter proposes two possible solutions:
1. **Extend allowed datatypes** to support multiples (i.e., allow `resources`/`references` as context types).
2. **Replace** the multiple `reference` entries with a single `resource` entry containing a FHIR **List** resource that holds references to the selected items.

A proof-of-concept implementation exists on the branch [`select-event-using-list`](https://github.com/HL7/fhircast-docs/tree/select-event-using-list) in the HL7/fhircast-docs repository, with a rendered preview at the [DiagnosticReport-select build page](https://build.fhir.org/ig/HL7/fhircast-docs/branches/select-event-using-list/3-6-4-DiagnosticReport-select.html).

## Proposed Changes (from branch)

The `select-event-using-list` branch (commit `ebef60a` by Bas van den Heuvel, 2024-10-22) modifies `3-6-4-DiagnosticReport-select.md`:

- Changes the `select` context key type from `reference` to `resource`
- Wraps selected items in a `List` resource (using a StructureDefinition `fhircast-select-list`)
- Removes the `patient` context key
- Updates the example to show a single `select` entry containing a `List` resource with `entry` items referencing selected/deselected `Observation` resources
- Adds support for indicating deselection via `"deleted": true` on list entries

**Current (master):**
```
select  | 1..*  | reference | Contains zero or more references to selected resources.
```

**Proposed (branch):**
```
select  | 1..*  | resource  | Contains zero or more references to selected resources,
                              stored in a List for Select Events.
```

## Proposed Dispositions

### Disposition A: Persuasive — Adopt the List-based approach
Accept the proposal to change the `select` context key from type `reference` (with multiple entries) to type `resource` containing a FHIR `List` resource. This aligns the select event with allowed FHIRCast context datatypes and provides a clean structure for expressing selections and deselections (via the `deleted` flag on List entries). Apply the changes from the `select-event-using-list` branch, creating the `fhircast-select-list` StructureDefinition and updating the DiagnosticReport-select event definition and examples. Consider whether the `patient` context key should be retained or removed as part of this change.

### Disposition B: Persuasive with Modification — Extend allowed datatypes instead
Rather than wrapping selections in a List resource, extend the FHIRCast context type system to explicitly allow `references` (plural) as a valid context type for select events. This preserves the current event structure (multiple `select` key entries) while formally resolving the datatype constraint violation. This approach is simpler but requires a broader specification change to the allowed context types.

### Disposition C: Not Persuasive — Current structure is sufficient
The current `reference` type with `1..*` cardinality for the `select` key is functionally adequate and already resolved by FHIR-39436. Implementers can work with the existing structure. The datatype constraint concern is a specification formalism issue rather than a practical interoperability problem. Defer any structural changes until there is demonstrated implementer need.

## Recommendation

**Disposition A: Persuasive — Adopt the List-based approach.**

The reporter has already provided a working proof-of-concept on the `select-event-using-list` branch that demonstrates the approach is feasible. Using a `List` resource resolves the datatype constraint issue cleanly within the existing FHIRCast context type system (no need to extend allowed types). It also introduces the ability to explicitly mark items as deselected via the `deleted` flag, which is a capability the current structure lacks. The related tickets FHIR-37305 and FHIR-39436 show ongoing evolution of the select event definition, and this change represents a natural next step in that progression. The work group should review the branch changes, decide on retention of the `patient` context key, and ensure consistency with the related `FHIR-45719-Advanced-Reference` pattern being applied to other events.
