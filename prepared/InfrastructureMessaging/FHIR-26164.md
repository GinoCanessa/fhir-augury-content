## Summary

This ticket requests adding a new section to the Da Vinci Unsolicited Notifications (formerly "Alerts") IG covering batch transactions and bulk data exchange. The reporter (Eric Haas) identified a gap: the IG lacks guidance on how to handle scenarios where large volumes of notification data need to be transmitted — for example, batching multiple notifications into a single transaction or leveraging bulk data mechanisms for high-volume exchanges.

The resolution is **Persuasive with Modification**: batching is on the Notifications roadmap for the next version, and the workgroup should determine the need for bulk data support. A comment from Eric Haas (2022-10-05) suggests that if the data payload is too large, an async `$import` operation could be a solution, referencing the [SMART Bulk Import manifest specification](https://github.com/smart-on-fhir/bulk-import/blob/master/import-manifest.md). The next step noted was to determine if this is an industry-wide need.

The ticket was previously resolved as "Considered for Future Use" in May 2020 (vote 3-0-1, Craig Newman/Paul Knapp), then reverted and retriaged. It was included in Block Vote 5 (2020-04-22) with the resolution "Considered for Future Use" and later updated to its current "Persuasive with Modification" resolution.

## Proposed Dispositions

### Disposition A: Accept (Persuasive with Modification)

**Action:** Accept the current resolution. Add a section to the next version of Da Vinci Notifications IG that addresses:
1. **Batching** — guidance on bundling multiple notifications into a single transaction Bundle, including size limits and error handling for partial failures.
2. **Bulk data** — reference the SMART Bulk Import (`$import`) pattern as an option for high-volume scenarios, with a link to the [bulk import manifest specification](https://github.com/smart-on-fhir/bulk-import/blob/master/import-manifest.md).

The section should note that batching is a priority for the next version, and bulk data support should be included if industry demand is confirmed. This aligns with the resolution description and Eric Haas's comments.

### Disposition B: Alternative — Batching Only, Defer Bulk

**Action:** Accept as Persuasive with Modification, but narrow the scope. Add guidance on **batching notifications** only (bundling multiple notification messages into a single transaction Bundle). Defer the bulk data / `$import` discussion to a separate tracker item, pending evidence of industry demand. This reduces scope risk and allows the batching feature to ship sooner while bulk data matures independently (e.g., through the SMART Bulk Import initiative).

### Disposition C: Decline — Not Persuasive

**Action:** Resolve as Not Persuasive. The Da Vinci Notifications IG focuses on individual event-driven notifications (admit, discharge, transfer). Batch and bulk patterns belong at the FHIR core or Subscriptions level, not in a use-case-specific IG. Related work in FHIR Subscriptions (e.g., FHIR-50622 on `maxCount` batching) already addresses batching at the infrastructure level. Adding this to the Notifications IG would duplicate effort and create maintenance burden.

---

## Recommendation

**Disposition A: Accept (Persuasive with Modification)**

The resolution is well-reasoned: batching is a practical need for notifications at scale and is already on the roadmap. The `$import` approach for bulk scenarios is a proven pattern with an existing specification. The ticket has been open since 2020 and was specifically retriaged (not left as "Considered for Future Use"), signaling that the workgroup considers this actionable. With related tickets (FHIR-26173 on bundle size, FHIR-50622 on batching in Subscriptions) confirming industry interest in both batching and bulk notification patterns, there is sufficient demand to proceed with both aspects in the next version.
