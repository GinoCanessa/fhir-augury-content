## Summary

Isaac Vetter filed this change request questioning the necessity and clarity of a new SHALL conformance requirement introduced in [HL7/fhircast-docs PR #587](https://github.com/HL7/fhircast-docs/pull/587) (which implemented FHIR-45719). The requirement appears at line 59 of `input/pagecontent/2-10-ContentSharing.md` in the FHIRCast specification.

The reporter raises two concerns:

1. **Clarity**: The new SHALL requirement is "difficult to understand" — it is unclear whether it is directed at implementers or at future event specification authors.
2. **Redundancy**: If the requirement is directed at implementers, it duplicates the existing 1..1 cardinality requirement for the context element already defined on the event definition page. If it is directed at spec authors, it is in the wrong location (mixed in with implementer requirements).

## Details

### Context: PR #587 and FHIR-45719

PR #587 was created by Eric Martin (EricOnFHIR) to address [FHIR-45719](https://jira.hl7.org/browse/FHIR-45719) ("cardinality of ImagingStudy should be changed"), which requested changing the cardinality to 0..* to allow multiple studies in update events. The PR corrected typos and elaborated the handling of context resource updates during `-update` events, including addition and removal of context-like resources. It was a substantial change (291 additions, 193 deletions across 24 files, 14 commits) and was merged by Isaac Vetter on 2024-12-17.

### The Disputed Requirement

During the PR review, Isaac Vetter left a [review comment](https://github.com/HL7/fhircast-docs/pull/587#discussion_r1810704998) on line 59 of `2-10-ContentSharing.md` asking:

> "Is this SHALL a requirement on implementers or authors of event specifications? The implementers SHALL include required elements in the context, right? If it's a SHALL on spec authors, do we really need it? If so, can we consolidate requirements for spec authors away from requirements for implementers?"

Bas van den Heuvel (bvdh) responded:

> "This is a SHALL on both. Spec authors of new events and users of events following this pattern not explicitly defined in the event library."

The PR was merged despite this unresolved review thread. Isaac subsequently filed FHIR-49078 thirteen days later (2024-12-30) to formally track the concern.

### Additional PR Review Context

Other unresolved review threads in PR #587 included:
- Concern about unclear "removal" terminology for context array resources
- Discussion about whether context values other than `updates` should be considered static during `-update` events
- Questions about references vs. slimmed-down profiles in update event definitions

## Proposed Dispositions

### Disposition A: Persuasive — Remove the Redundant SHALL

**Action:** Remove the disputed SHALL requirement from `2-10-ContentSharing.md` line 59, since the 1..1 cardinality constraint on the context element is already enforced on the event definition page. The existing cardinality requirement already mandates that implementers include the required context element, making the additional SHALL redundant.

**Rationale:** The reporter's concern is valid — the requirement duplicates existing cardinality constraints. Removing it eliminates confusion about audience (implementer vs. spec author) and reduces the risk of contradictory normative language. The FHIRCast spec should follow the principle of stating conformance requirements in exactly one canonical location.

### Disposition B: Persuasive with Modification — Rewrite and Relocate the SHALL

**Action:** Rewrite the SHALL requirement to clearly distinguish its two audiences: (1) for implementers, clarify that this is a restatement of the cardinality constraint for emphasis in the content sharing context; (2) for event specification authors, relocate the requirement to a dedicated "Event Specification Authoring Guidelines" section or page, clearly separated from implementer-facing requirements.

**Rationale:** Bas van den Heuvel's response confirms the requirement is intended to apply to both implementers and spec authors. Rather than removing it entirely, it could be preserved with clearer language and appropriate placement. This addresses the "difficult to understand" and "wrong location" concerns while retaining the spec authoring guidance that bvdh intended.

### Disposition C: Not Persuasive — Retain the Current SHALL

**Action:** Retain the requirement as-is. Add a brief editorial note clarifying that the SHALL applies to both implementers and future event specification authors, without restructuring the specification.

**Rationale:** The requirement serves a dual purpose as confirmed by the PR author. While the wording could be clearer, the requirement is not technically incorrect. Adding a clarifying note addresses the readability concern without requiring a structural change to the specification. The 1..1 cardinality constraint on the event definition page and this SHALL are complementary, not duplicative — the former defines the data model, the latter defines the behavioral expectation.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended.

The reporter's concern is well-founded — the current requirement is genuinely unclear about its audience and its relationship to existing cardinality constraints. However, completely removing it (Disposition A) would lose the valuable guidance for future event specification authors that bvdh explicitly intended. The best path forward is to rewrite the requirement with clear audience separation: keep a brief implementer-facing note in `2-10-ContentSharing.md` that cross-references the event definition cardinality, and move the spec authoring guidance to a dedicated section. This preserves the intent while resolving the clarity and placement concerns that prompted the ticket.
