## Summary

The submitter (Brian Pech) requests additional narrative in **FHIRCast section 3.6.3** to clarify how discrete data is shared between applications via FHIRCast content sharing events. The ticket specifically raises three questions:

1. **Consumability by downstream apps** — The existing example references an Impression Narrative with a "simple cyst" finding. The submitter asks whether such data would be consumable by a downstream application (e.g., an EHR/medical record system).
2. **Timing and event-driven updates** — Would the arrival of a content-sharing event update an existing report in the receiving system, and what governs that timing?
3. **General context** — More explanatory narrative is needed in section 3.6.3 to help implementers understand the practical flow of discrete data sharing.

This is filed as a **Technical Correction** with **Highest** priority, indicating the submitter sees this as an important gap in the current specification text.

## Analysis

### Context

FHIRCast section 3.6.3 covers content sharing scenarios where applications exchange discrete clinical data (e.g., findings, observations) through FHIRCast events. The current text includes examples (such as an Impression Narrative referencing a "simple cyst" finding) but does not adequately explain:

- How a receiving application should interpret and consume discrete data from these events
- Whether and how content sharing events trigger updates to existing resources in downstream systems
- The expected behavior when events arrive at different points in a clinical workflow

### Prior Work

Several related tickets have already addressed aspects of the content sharing narrative:

- **FHIR-36970** (Applied, Persuasive with Modification) resulted in spec clarifications around how anchor content and associated resources are exchanged in update events, including method operations and bundle semantics.
- **FHIR-36892** (Applied, Not Persuasive with Modification) led to improvements in distinguishing content sharing from context synchronization.
- **FHIR-45631** (Deferred) raised the broader concern that content sharing is too underspecified for interoperability — this was deferred as "Considered for Future Use," suggesting the work group acknowledges the gap but sees it as a larger initiative.

The submitter's request for "more context" in section 3.6.3 is narrower than these prior tickets — it asks for additional explanatory text rather than structural specification changes.

### Considerations

- The ticket is typed as a **Technical Correction**, which typically implies clarifying existing text rather than introducing new normative requirements.
- The questions raised (consumability, timing, event-driven updates) are legitimate implementation concerns that new adopters of FHIRCast content sharing would encounter.
- Some of the requested context may already be partially addressed by changes from FHIR-36970 and FHIR-36892. The work group should assess whether the current spec text (post those changes) already addresses the submitter's concerns.
- The deferral of FHIR-45631 suggests the work group may prefer to address broader content sharing clarity in a future cycle, which could encompass this ticket's concerns.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Add a brief explanatory paragraph or informative note to section 3.6.3 clarifying that:
1. Discrete data shared via FHIRCast content sharing events (e.g., findings in a DiagnosticReport) is intended to be consumable by subscribing applications, but the receiving application's behavior (e.g., persisting to a medical record) is implementation-specific and outside FHIRCast's scope.
2. The timing of event delivery is governed by the FHIRCast event model (real-time via WebSocket); whether an event updates an existing report depends on the subscriber's implementation and the event type (e.g., `DiagnosticReport-update`).
3. Add a forward reference to the content sharing scenario page for implementers seeking further detail.

**Rationale:** This addresses the submitter's request for additional context as a Technical Correction without expanding the normative scope of FHIRCast. It improves clarity for new implementers navigating section 3.6.3.

### Disposition B: Not Persuasive with Modification
**Action:** Acknowledge the concern but note that the questions raised are implementation-specific and outside the scope of the FHIRCast specification. Add a brief informative note to section 3.6.3 directing readers to the content sharing scenario pages and the event catalog for further context on how events convey discrete data.

**Rationale:** FHIRCast intentionally does not prescribe how receiving applications handle shared content. The existing spec text plus changes from FHIR-36970 and FHIR-36892 provide sufficient normative guidance. A minimal pointer to the relevant scenario documentation addresses the usability concern without expanding the section.

### Disposition C: Considered — Duplicate / Addressed by Existing Tickets
**Action:** Resolve as duplicate or partially addressed by FHIR-36970 (Applied) and note that the broader concern is tracked by FHIR-45631 (Deferred). No spec change needed if the work group determines the current text post-FHIR-36970 already provides adequate context.

**Rationale:** Prior ballot cycles have applied changes to clarify content sharing semantics. If the current spec text in section 3.6.3 already addresses the submitter's questions (post-FHIR-36970 changes), a separate change is unnecessary. The broader interoperability gap is deferred to a future cycle via FHIR-45631.

---

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The submitter raises valid questions that a new implementer would encounter when reading section 3.6.3. While the questions touch on implementation-specific behavior, the spec can improve its explanatory text without adding normative requirements. A short informative paragraph clarifying the intent and scope of discrete data sharing in the example — along with pointers to the content sharing scenario and event documentation — would materially improve the reader experience. This is consistent with the Technical Correction type and addresses the "Highest" priority without requiring major specification restructuring. The prior changes from FHIR-36970 and FHIR-36892 laid groundwork, but section 3.6.3 specifically could benefit from the additional narrative the submitter requests.
