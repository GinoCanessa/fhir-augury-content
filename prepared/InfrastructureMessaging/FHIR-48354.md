## Description

Raised during Connectathon 37 II testing track. The submitter asks whether an application that has connected to a FHIRcast hub and issued a `Patient-open` can subsequently bootstrap a content sharing session by sending a `DiagnosticReport-open` for a **newly created** DiagnosticReport (i.e., one that does not yet exist on a FHIR server). Specifically:

1. Can an application start a content sharing session on a newly created DiagnosticReport by sending a `DiagnosticReport-open`?
2. If so, is the DiagnosticReport resource implicitly present in the content linked to that anchor context?
3. Do the other currently open anchor resources (Patient, Encounter, etc.) appear in the content sharing context as well?

The submitter also notes that a hub **could** refuse creation of new anchor resources, whereas subscribers can only refuse to change context.

## Analysis

This ticket raises an important gap in the FHIRcast content sharing specification: the **bootstrapping sequence** for initiating a content sharing session. The current specification describes content sharing mechanics once a `DiagnosticReport-open` context is established, but does not clearly address:

- **Whether a subscriber can create a new anchor resource** (DiagnosticReport) by sending `DiagnosticReport-open` when no DiagnosticReport exists yet. The spec allows the hub to refuse, but doesn't clarify the expected behavior when it accepts.
- **Implicit content population**: If a new `DiagnosticReport-open` is accepted, it is unclear whether the DiagnosticReport resource itself is automatically part of the content sharing bundle, or if a separate `DiagnosticReport-update` must follow.
- **Relationship between nested anchor contexts**: When a `Patient-open` has already established patient context and then `DiagnosticReport-open` is sent, the spec does not clearly state whether the Patient (and possibly Encounter) resources appear in the DiagnosticReport's content sharing context.

These questions surfaced during real-world Connectathon testing (Connectathon 37 II), indicating they are practical interoperability concerns rather than theoretical edge cases. Several related tickets (FHIR-45629, FHIR-36910, FHIR-43080) have addressed adjacent aspects of content sharing lifecycle and context management, but the specific bootstrap scenario remains unaddressed.

## Proposed Dispositions

### Disposition A: Persuasive — Add Content Sharing Bootstrap Guidance

**Action:** Add a new section or subsection to the FHIRcast content sharing documentation that explicitly describes the bootstrapping sequence for initiating a content sharing session. This should:

1. Clarify that a subscriber MAY send a `DiagnosticReport-open` to create a new content sharing context, and that the hub MAY accept or refuse the request.
2. Specify that when the hub accepts a `DiagnosticReport-open`, the DiagnosticReport resource is implicitly the anchor of the content sharing context and is included in the initial content.
3. Define the relationship between the newly created DiagnosticReport context and any previously opened anchor contexts (e.g., Patient, Encounter), specifying whether they appear as part of the content sharing bundle.
4. Add an example showing the complete bootstrap flow: `Patient-open` → `DiagnosticReport-open` → initial content state.

**Rationale:** This addresses a documented gap discovered during interoperability testing. Clear bootstrapping guidance will improve implementer consistency and reduce ambiguity, especially for new adopters of content sharing.

### Disposition B: Persuasive with Modification — Clarify Existing Text Only

**Action:** Rather than adding a new section, modify existing `DiagnosticReport-open` and content sharing documentation to:

1. Add a note that `DiagnosticReport-open` can be used to initiate a content sharing session for a newly created DiagnosticReport.
2. Clarify that the hub's acceptance of the event establishes the content sharing context, with the DiagnosticReport as the anchor resource.
3. Add a cross-reference to the existing documentation on hub refusal behavior.

**Rationale:** The existing specification may already implicitly support this workflow. Targeted clarifications in existing sections may be more maintainable than adding new content and reduce the risk of contradictions.

### Disposition C: Not Persuasive with Modification — Defer Pending Related Resolution

**Action:** Acknowledge the gap but defer resolution until the outcomes of related tickets (particularly FHIR-45629 on multiple anchor resources and FHIR-45645 on resource identity during content sharing) are fully integrated. Add a tracker comment noting the dependency.

**Rationale:** The bootstrap question is tightly coupled with ongoing work on anchor context management and resource identity. Resolving this ticket prematurely could conflict with changes from those tickets. A coordinated approach would yield a more coherent specification.

## Recommendation

**Disposition A: Persuasive — Add Content Sharing Bootstrap Guidance** is recommended.

The question arose from practical Connectathon testing and represents a genuine specification gap that affects implementers today. The most closely related tickets (FHIR-45629, FHIR-43080) have already been resolved and applied, meaning the specification landscape is stable enough to address this bootstrapping scenario directly. Adding explicit guidance with a worked example would provide the most value to implementers and reduce ambiguity in a critical area of the FHIRcast content sharing workflow.
