## Description

The submitter argues that the FHIRCast [Hub conformance statement](http://hl7.org/fhir/uv/fhircast/2022May/2-7-Conformance.html#wellknown-endpoint) — which specifies which events a Hub supports — is problematic when event customization is a feature. If a Hub (e.g., an EMR/RIS) only reports support for its own known events, it blocks third-party applications (e.g., ImageViewers, Dictation systems) from exchanging custom events through that Hub.

The core question: **Should a Hub propagate events it does not itself recognize?** The submitter contends that an EMR/RIS acting as a Hub should pass through any events between subscribers, even if the EMR itself does not listen to or understand those events.

## Zulip Discussions

### 1. [FHIRcast > Hubs and Apps](https://chat.fhir.org/#narrow/stream/FHIRcast/topic/Hubs%20and%20Apps) (3 messages, Jun 2022)

**Participants:** Evan Schnell, Isaac Vetter

Evan Schnell (the ticket reporter) presented a scenario with an ImageViewer and Dictation system integrated through an EMR/RIS hub. The key problem: when each hub's conformance is limited to events its own vendor understands, "following" apps are blocked from expanding FHIRCast functionality. He illustrated how this forces multiple hubs and multiple WebSocket connections, creating unmanageable complexity. Isaac Vetter asked about the responsibilities of a "pass through hub" — specifically whether it would need to derive context events (e.g., gracefully degrading an ImagingStudy-open into a Patient-open). Evan proposed the hub maintain an abstract graph of patient/encounter references and simply pass events as "primitive middleware," only cascading on patient-close or encounter-close.

### 2. [FHIRcast > Agnostic hub](https://chat.fhir.org/#narrow/stream/FHIRcast/topic/Agnostic%20hub) (25 messages, Nov 2022 – Jan 2023)

**Participants:** Paulius Galinauskas, Bas van den Heuvel, Aris van Dijk, Elliot Silver, Eric Martin, Isaac Vetter, kamal gnanadesikan, Catie Ladd

This extended discussion directly continues the FHIR-36994 concern. Key points:

- **Bas van den Heuvel** explained the current design: the Hub is not agnostic — some events require specific hub behavior (context change, content exchange), and the conformance endpoint publishes supported events. Reasons include: (1) early implementers wanted control, (2) the security/OAuth model is event-specific, (3) most events require specific hub behavior.
- **Aris van Dijk** and **Paulius Galinauskas** argued this creates a "worst of both worlds" scenario — if hubs only support known events, third-party apps must build their own hubs for custom events, turning the hub-and-spokes model into point-to-point integrations between multiple hubs.
- **Bas van den Heuvel** acknowledged the idea has merit and proposed three solution directions: (a) an optional event registration mechanism, (b) a new `-simple` event type that is just forwarded, or (c) only applying scope restrictions to spec-defined events.
- **Eric Martin** proposed a concrete spec change: allow a `*` wildcard element in the `eventsSupported` array to indicate the Hub supports proxying unknown events, with explicit rules about subscription confirmation when unknown events are not supported.
- **Elliot Silver** suggested the hub could forward unrecognized events but add an "unrecognized" element so receivers know the server did no processing.
- **Isaac Vetter** noted a tension: proxying unknown events breaks the ability to derive implied events, and raised privacy concerns (would a hub developer need an NDA to proxy proprietary events?).

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Accept the concern and modify the FHIRCast conformance specification to support optional event pass-through.

Specifically, adopt Eric Martin's proposal from the Zulip discussion: allow a `*` wildcard element in the `eventsSupported` array of the conformance endpoint to indicate that the Hub supports forwarding of unknown/custom events. Add normative language:
- A `*` in `eventsSupported` indicates the Hub **WILL** forward unknown events to subscribers who subscribed to them.
- Hubs without `*` **MAY NOT** forward unknown events.
- Hubs that do not support forwarding **SHALL** remove unknown events from the `hub.events` array in the subscription confirmation.

This preserves backward compatibility (hubs that don't want pass-through can omit the wildcard) while enabling the use case described in the ticket. It also aligns with FHIR-37325 (already applied) which recognized the need for custom event extensibility.

**Rationale:** The community discussion shows broad agreement that the current model is too restrictive. A concrete proposal exists and has been discussed. The closely related FHIR-39545 was deferred (not rejected), indicating the workgroup sees merit but needed a specific mechanism — this provides one.

### Disposition B: Considered for Future Use (Defer)
**Action:** Acknowledge the concern as valid but defer to a future ballot cycle, pending further implementation experience and connectathon testing.

The concern is well-understood and the community has proposed multiple solution approaches, but none have been validated through implementation or connectathon testing. The related FHIR-39545 was already deferred with this same rationale. There are unresolved design questions around:
- Hub behavior for events requiring specific processing (e.g., `-open` events that require state tracking, `-update` events that require content management)
- Security/OAuth scope implications for unknown events
- Privacy concerns when proprietary events traverse third-party hubs

**Rationale:** Deferring allows time to explore the solution space at a connectathon and gather implementer feedback before committing to a specific mechanism.

### Disposition C: Not Persuasive
**Action:** Reject the change request, maintaining the current conformance model where hubs explicitly list supported events.

The current design is intentional: hubs need awareness of events they handle because many event types (context changes, content sharing) impose specific behavioral requirements on the hub. Making hubs agnostic undermines safety guarantees — for example, a hub that blindly forwards an `-open` event without tracking context state could leave subscribers in inconsistent states. The security model is also event-specific, and pass-through would bypass these controls. Implementers who need custom event exchange between specific apps can use point-to-point mechanisms outside FHIRCast.

**Rationale:** The current architecture was designed with safety and consistency in mind. The complexity of supporting pass-through across all event categories may outweigh the convenience benefit.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The community discussion (25+ messages across two Zulip threads) demonstrates a clear, practical need for event pass-through capability. Multiple implementers from different organizations (Evan Schnell, Paulius Galinauskas, Aris van Dijk) have independently identified the same limitation. A concrete, backward-compatible solution (the `*` wildcard in `eventsSupported`) has been proposed and discussed without fundamental objections. The related FHIR-37325 was already resolved as "Persuasive with Modification," establishing precedent that custom event extensibility is valued. Deferring again (as was done with FHIR-39545) risks losing implementer engagement on this important interoperability gap.
