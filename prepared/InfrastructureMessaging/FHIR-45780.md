## Description

The submitter argues that since the FHIR Encounter resource allows both Group and Patient as subject, Encounters can exist without a Patient reference. FHIRcast currently requires Patient to be present in certain event contexts, which is more restrictive than the base FHIR specification. The request is to:

1. **Make Patient optional** in FHIRcast event context requirements where it is currently mandatory.
2. **Add Group as a supported context element**, aligning FHIRcast with FHIR's allowance for Encounter.subject to reference a Group.

This would enable non-patient-centric deployments (e.g., veterinary, research, group-based care) to use FHIRcast without violating context requirements.

## Analysis

### Core Issue
FHIRcast event profiles (e.g., `Patient-open`, `Encounter-open`) currently require Patient as a mandatory context element. However, FHIR itself allows `Encounter.subject` to reference either a Patient or a Group. This means FHIRcast imposes a constraint not present in the base FHIR specification, preventing legitimate use cases where Patient is not applicable.

### Pattern of Related Requests
This ticket is part of a broader pattern from the same submitter (Bas van den Heuvel) requesting that FHIRcast relax constraints to better align with the base FHIR specification. Related tickets FHIR-45611 (event profiles too strict), FHIR-45786 (identifier requirements), and FHIR-37089 (secondary anchors) all follow a similar theme. Notably, FHIR-45611 and FHIR-45786 were both deferred with a "Considered for Future Use" resolution and labeled as block-vote items, suggesting the WG has previously been cautious about loosening constraints.

### Duplicate Concern
FHIR-45781 appears to be an exact duplicate of this ticket — same title, same reporter, same description, same creation date. The WG should consider consolidating these.

### Prior WG Sentiment
The comment on FHIR-45611 (the closely related "event profiles too strict" ticket) indicates at least one reviewer found the profiles "relatively relaxed" and recommended marking the request as unpersuasive, arguing that exceptions should be left to implementors. This may signal WG resistance to the general approach of loosening event profile requirements.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Rationale:** The submitter correctly identifies that FHIRcast's Patient requirement is stricter than base FHIR allows. Make Patient optional (0..1) in event context profiles where Encounter is the anchor, and add Group (0..1) as an alternative subject context element. This aligns FHIRcast with FHIR Encounter's subject cardinality and type. Constrain that at least one of Patient or Group SHOULD be present when an Encounter is in context, to maintain useful context for subscribers.

**Scope of change:** Modify FHIRcast event profiles (e.g., Encounter-open, Encounter-close) to change Patient cardinality from 1..1 to 0..1, add Group as an optional context element (0..1), and update conformance language. This is a **breaking change** for implementations that assume Patient is always present.

### Disposition B: Not Persuasive with Modification (Defer to Implementation Guides)
**Rationale:** FHIRcast's Patient requirement reflects the dominant healthcare deployment pattern where Patient is always known at the point of care. Making Patient optional in the base spec could cause interoperability problems — subscribers may assume Patient is present and fail silently when it is absent. Instead, add guidance that implementation guides for non-patient-centric deployments (veterinary, research, group care) may relax the Patient requirement. Note that FHIR-45611 received similar feedback suggesting the base profiles should not be loosened.

**Scope of change:** Add a paragraph of guidance text to the FHIRcast spec; no profile changes.

### Disposition C: Not Persuasive
**Rationale:** FHIRcast event profiles are intentionally prescriptive to ensure interoperability across subscribing applications. Making Patient optional would weaken the contract between publishers and subscribers, potentially causing failures when subscribers expect Patient context. The base FHIR specification's flexibility is appropriate for FHIR resources, but FHIRcast context events serve a different purpose — synchronized context sharing — where predictability is paramount. Implementors in non-standard deployments can already deviate from profiles as needed. The related tickets (FHIR-45611, FHIR-45786) were deferred for similar reasons.

## Recommendation

**Disposition B: Not Persuasive with Modification (Defer to Implementation Guides)** is recommended.

The submitter raises a valid alignment concern, but the practical impact of making Patient optional in the base FHIRcast specification could disrupt existing implementations that rely on Patient being present. The WG's prior handling of closely related tickets (FHIR-45611 deferred, FHIR-45786 deferred) shows a pattern of caution around loosening event profile requirements. Adding guidance that acknowledges the use case and directs non-patient-centric deployments to create appropriate implementation guides preserves interoperability for the majority of users while addressing the submitter's concern. The WG should also consolidate FHIR-45781 as a duplicate.
