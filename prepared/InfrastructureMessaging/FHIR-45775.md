## Description

The submitter notes that the current FHIRCast extension approach is event-context specific. The request is to either:

1. **Define a generic extension support mechanism** (e.g., adopting the FHIR extensibility model), or
2. **Move the extension section** to Section 2.3 (Event Format) so that extensions are defined at the event format level rather than per-context.

The core concern is that having context-specific extension definitions creates unnecessary duplication and inconsistency across different FHIRCast events. A generic, centralized approach would be more maintainable and aligned with FHIR's own extension patterns.

## Analysis

### Core Issue

The FHIRCast specification currently defines extension mechanisms at the event-context level (per-event type), which leads to fragmented and duplicated extension definitions. The submitter proposes consolidating this into a single, generic extension framework — either by adopting FHIR's existing extensibility model or by relocating extension definitions to the shared Event Format section (Section 2.3).

### Impact Assessment

- **Scope**: This is a structural/architectural change to how extensions are defined in the FHIRCast spec. It would affect all event types that currently define their own extension approaches.
- **Prior Art**: FHIR-43151 (Applied) already tightened the scope of the `fhircast-configuration-extensions`, showing the WG's willingness to improve extension definitions. FHIR-40658 (Resolved) dealt with Hub behavior around resource attributes and extensions in context, indicating active discussion in this area.
- **Alignment**: Adopting FHIR's standard extensibility model would improve consistency between FHIRCast and the broader FHIR ecosystem.
- **Risk**: Changing the extension architecture could affect existing implementations that rely on the current context-specific extension approach. A migration path or backwards-compatibility guidance would be needed.

## Proposed Dispositions

### Disposition A: Persuasive — Define a generic extension mechanism in Section 2.3

Accept the request and define a single, generic extension mechanism in Section 2.3 (Event Format) that applies uniformly to all FHIRCast events. This would adopt FHIR's extensibility patterns (e.g., using Extension elements with defined URLs) as the standard approach, replacing any event-context-specific extension definitions. Remove or deprecate duplicated per-context extension sections.

**Rationale**: Centralizing extension support in the Event Format section eliminates duplication, improves consistency, and aligns FHIRCast with FHIR's well-established extensibility model. The WG has already shown willingness to clean up extension scoping (FHIR-43151).

### Disposition B: Persuasive with Modification — Move extension section to Event Format but preserve context-specific overrides

Move the primary extension mechanism to Section 2.3 (Event Format) to establish a generic baseline, but allow individual events to define context-specific extension constraints where necessary (e.g., required extensions for particular clinical workflows). Document clear guidance on when context-specific extensions are appropriate vs. when the generic mechanism should be used.

**Rationale**: A purely generic approach may not account for event-specific constraints. A layered approach (generic base + context-specific refinements) provides flexibility while still reducing duplication.

### Disposition C: Deferred — Await broader FHIRCast extensibility review

Defer this change to a future ballot cycle, pending a broader review of FHIRCast's extensibility model across the entire specification. This would allow the WG to assess the full impact of restructuring extension support and coordinate with related open tickets (e.g., FHIR-37089 on secondary anchors, FHIR-40658 on Hub filtering behavior).

**Rationale**: The change is architecturally significant. Deferral allows for a comprehensive design that addresses extension support, Hub behavior, and context resource handling holistically rather than piecemeal.

## Recommendation

**Disposition A: Persuasive** — The request is well-motivated and aligns with standard FHIR extensibility patterns. The current context-specific approach creates unnecessary fragmentation. Defining a generic extension mechanism in Section 2.3 (Event Format) is the cleanest solution, and the WG's prior acceptance of FHIR-43151 (tightening extension scope) indicates support for this direction. The change should include migration guidance for existing implementations and explicit references to FHIR's extensibility framework.
