## Summary

The Da Vinci Notifications IG currently constrains `MessageHeader.response` to `0..0` in the base DaVinci Notifications MessageHeader profile. This prevents implementers from using the IG within the standard FHIR messaging paradigm, where a response message must include `MessageHeader.response` to link back to the original request message. The submitter (Ulrike Merrick, an IG author) proposes either:

1. **Removing** the `0..0` constraint on `.response` in the existing profile, or
2. **Creating a separate MessageHeader profile** specifically for response messages, where `.response` is mandatory.

## Details

### Background

The Da Vinci Notifications IG defines a MessageHeader profile for unsolicited event notifications (e.g., ADT admission/discharge/transfer alerts). The current profile constrains `MessageHeader.response` to `0..0`, which makes sense for outgoing unsolicited notifications that are not responses to a prior message. However, this constraint also prevents the IG from being used in a full messaging round-trip where a receiver acknowledges or responds to a notification.

### Real-World Use Case

A Zulip discussion thread ("Feedback loop for ADT notifications" in the Da Vinci Alerts stream) documents a concrete implementation need from athenahealth. Their workflow:

1. Payer sends ADT notification via Da Vinci IG
2. athenahealth receives and validates the notification, returning an initial OperationOutcome
3. athenahealth asynchronously processes/ingests the notification
4. **Gap**: The payer has no way to learn whether the notification was successfully ingested

The proposed solution involves a response message bundle containing a MessageHeader with `MessageHeader.response.identifier` pointing back to the original notification and `MessageHeader.response.details` referencing an OperationOutcome with ingestion status. This pattern is standard FHIR messaging but is blocked by the current `0..0` constraint on `.response`.

### Technical Considerations

- The `0..0` constraint was likely intentional for the original unsolicited notification use case, where `.response` is not applicable
- Relaxing the constraint to `0..*` on the base profile would make it applicable to both notification and response scenarios, but may reduce profile specificity
- Creating a separate response MessageHeader profile would preserve the original profile's semantics while enabling the messaging response pattern
- Related historical ticket [FHIR-26359](https://jira.hl7.org/browse/FHIR-26359) dealt with similar constraints (Bundle `entry.request`/`entry.response` being prohibited in messages), resolved as "Not Persuasive" for the core spec
- [FHIR-40264](https://jira.hl7.org/browse/FHIR-40264) addressed reconsideration of MessageHeader `sender`, `author`, `responsible` elements in this same IG

## Proposed Dispositions

### Disposition A: Accept — Create a Separate Response MessageHeader Profile
**Resolution:** Persuasive with Modification

Accept the request by creating a new MessageHeader profile (e.g., "DaVinci Notifications Response MessageHeader") specifically for response messages. In this profile:
- `MessageHeader.response` is `1..1` (mandatory)
- `MessageHeader.response.identifier` is `1..1` and must reference the original notification's `Bundle.id` or `MessageHeader.id`
- `MessageHeader.response.details` references an OperationOutcome

The existing base DaVinci Notifications MessageHeader retains its `0..0` constraint on `.response`, preserving its semantics for unsolicited notifications. This approach cleanly separates the two use cases and provides clear conformance expectations for implementers.

### Disposition B: Accept — Relax the Constraint on the Existing Profile
**Resolution:** Persuasive

Remove the `0..0` constraint on `MessageHeader.response` in the existing base profile, changing it to `0..1` (optional). This is simpler than creating a separate profile and allows the same MessageHeader profile to be used for both notifications and responses. Add guidance text explaining when `.response` should be populated (i.e., when the message is a response to a prior notification). This approach is less prescriptive but avoids profile proliferation.

### Disposition C: Decline — Maintain Current Constraint
**Resolution:** Not Persuasive

The Da Vinci Notifications IG was designed for unsolicited event notifications, not for bidirectional messaging. The `0..0` constraint on `.response` correctly reflects the IG's intended scope. Implementers who need response messaging should use the base FHIR MessageHeader resource (without the Da Vinci profile) or define their own profiles. The feedback loop use case described in Zulip may be better addressed through a separate IG or an extension to the existing one, rather than modifying the base notification profile.

## Recommendation

**Disposition A: Create a Separate Response MessageHeader Profile**

This is the strongest option because:

1. **Preserves existing semantics**: The base notification profile retains its `0..0` constraint, which correctly models unsolicited notifications and avoids breaking existing implementations.
2. **Addresses a real implementer need**: The Zulip discussion demonstrates concrete demand from athenahealth for a response messaging pattern within the Da Vinci ecosystem.
3. **Aligns with FHIR messaging patterns**: A dedicated response profile follows the standard messaging paradigm where response messages carry `MessageHeader.response` linking back to the original.
4. **Reporter is an IG author**: Ulrike Merrick co-authored this IG and explicitly identified both options, suggesting the separate profile approach has community support.
5. **Clean separation of concerns**: Two profiles (notification vs. response) provide clearer conformance expectations than a single permissive profile.

The quarterly Notifications IG call (planned for September 3, 2025) would be an appropriate venue to finalize the profile design with implementer input.
