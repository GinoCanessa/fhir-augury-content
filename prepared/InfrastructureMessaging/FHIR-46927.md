## Description

In Section 2.3.4.4 of the FHIRCast spec, selection events are called out as experimental. However, the DiagnosticReport-select event does not have a similar "experimental" callout. The submitter requests that the experimental label be added to DiagnosticReport-select for consistency with the other selection events documented in the spec.

The ticket explicitly references [FHIR-36981](https://jira.hl7.org/browse/FHIR-36981) ("Consider *-select event for other resources"), which introduced select events for additional resources including DiagnosticReport and was resolved as Persuasive with Modification.

## Resolution Description

> We will update the DxReport-select event to be labeled as experimental.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Action:** Add the "experimental" callout to the DiagnosticReport-select event documentation (Section 3.6.4) to match the experimental labeling already present on other selection events in Section 2.3.4.4.

**Rationale:** This is a straightforward consistency fix. The *-select events were introduced as experimental through FHIR-36981, and the other selection events are already marked experimental. The DiagnosticReport-select event was simply missed. The existing resolution description ("We will update the DxReport-select event to be labeled as experimental") is clear and appropriate. No technical or normative content changes are needed — only adding the experimental designation label.

### Disposition B: Accept as Persuasive with Modification

**Action:** Add the experimental callout to DiagnosticReport-select, and additionally audit all *-select events in the Event Catalog to confirm that every select event consistently carries the experimental label.

**Rationale:** While the ticket only identifies DiagnosticReport-select, there may be other select events that were similarly missed. A broader audit would prevent further inconsistency tickets. However, this may be out of scope for this specific change request.

### Disposition C: Not Persuasive

**Action:** No change.

**Rationale:** If the work group determines that DiagnosticReport-select has matured sufficiently through implementation and ballot feedback to remove its experimental status rather than add it, no labeling change would be needed. However, this would create an inconsistency in the opposite direction (one non-experimental select event among several experimental ones) and would need to be accompanied by a broader discussion about graduating all select events.

---

## Recommendation

**Disposition A: Accept as Persuasive.** The change is a minor editorial consistency fix — adding the missing experimental label to DiagnosticReport-select to match the other *-select events. The resolution is already clear and the implementation is trivial. No modification to the proposed resolution is needed.
