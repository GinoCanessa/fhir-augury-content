## Summary

The submitter requests that **"deprecated"** be added as a maturity title (level) to the FHIRCast Event Maturity Model's event definition lifecycle. Currently, the FHIRCast event maturity model defines maturity levels for events (e.g., levels 0 through 4+ indicating increasing stability and implementation readiness), but there is no formal "deprecated" status to indicate that an event has been superseded or is no longer recommended for use.

This gap has practical consequences: events like the Heartbeat event have already been deprecated in practice (see FHIR-45627), and context keys like `report` and `study` have been proposed for deprecation (FHIR-45613). Without a formal "deprecated" maturity level, there is no standardized way to signal to implementers that an event should no longer be used.

## Proposed Dispositions

### Disposition A: Persuasive

**Rationale:** The request is well-founded and fills a clear gap in the FHIRCast Event Maturity Model. The maturity model currently defines levels for events progressing toward maturity (0–4+) but lacks a formal mechanism for events that should no longer be used. The Heartbeat event (FHIR-45627) was effectively deprecated and removed, and the `report`/`study` keys were proposed for deprecation (FHIR-45613), showing real-world need.

Adding a "deprecated" maturity level would:
1. Provide implementers clear, standardized notice that an event is no longer recommended.
2. Allow a graceful transition period before full removal.
3. Align with common software lifecycle practices (e.g., FHIR's own resource maturity model includes deprecation concepts).
4. Reduce ambiguity when the WG needs to sunset events.

**Suggested implementation:** Add "Deprecated" as a formal maturity level in the event maturity model section of the FHIRCast specification. Define its meaning (event is no longer recommended, may be removed in a future version) and the criteria/process for transitioning an event to this level.

---

### Disposition B: Persuasive with Modification

**Rationale:** The concept is valid, but rather than adding "deprecated" as a full maturity *level* (which implies a position in the 0–4 progression), add it as a separate *status flag* or *annotation* that can be applied to an event at any maturity level. This avoids confusion about whether "deprecated" is a progression step or a terminal state.

**Suggested implementation:** Add a "deprecated" boolean flag or status marker to the event definition template, separate from the numeric maturity level. Include guidance on when and how the WG applies this flag, the expected timeline for removal, and communication requirements to implementers.

---

### Disposition C: Not Persuasive

**Rationale:** The current maturity model adequately handles event lifecycle through the existing levels and editorial process. When an event needs to be deprecated, the WG can simply remove it from the specification (as was done with the Heartbeat event in FHIR-45627) or mark it informatively in release notes. Adding a formal "deprecated" level introduces process overhead and normative complexity for a situation that arises rarely and can be handled editorially.

---

## Recommendation

**Disposition A: Persuasive**

The request addresses a genuine gap. The FHIRCast specification has already encountered multiple situations where events or context keys needed deprecation (Heartbeat removal in FHIR-45627, proposed deprecation in FHIR-45613), and the absence of a formal "deprecated" maturity level led to inconsistent handling. Adding "deprecated" as a maturity title is a small, well-scoped change that aligns with standard software lifecycle practices, provides clear signaling to implementers, and supports the WG's governance of the event catalog. The related ticket FHIR-45779 (resolved Persuasive with Modification) already strengthened WG authority over maturity decisions, making the addition of a "deprecated" level a natural complement.
