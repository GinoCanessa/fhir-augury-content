# Ticket Review: FHIR-26437

**Title:** Define urls for passports' identification
**Status:** Triaged **Priority:** Medium **Type:** Change Request
**Work Group:** HL7 Terminology Authority **Specification:** FHIR Core (FHIR)
**Reporter:** Giorgio Cangioli **Assignee:** Unassigned
**Created:** 2020-03-04 **Updated:** 2023-02-24 **Labels:** None

---

## Summary

Giorgio Cangioli requests that a standard URL pattern be defined for identifying passport numbers in FHIR, analogous to the existing OID hierarchy (2.16.840.1.113883.4.330.[country code]). The HL7 OID registry already assigns OIDs for passport identification by country, but no corresponding FHIR URI (system identifier) pattern exists. The submitter proposes a pattern such as `http://hl7.org/fhir/sid/passport-[ISO alpha-3 country code]` (e.g., `http://hl7.org/fhir/sid/passport-FRA` for French passports) and requests that this pattern be documented in the FHIR identifier registry. This ticket was reassigned to the HL7 Terminology Authority (HTA) as the appropriate body for establishing identifier namespaces.

## Details

**Description:**

The HL7 registry specifies OIDs for identifying passports (root 2.16.840.1.113883.4.330):

`2.16.840.1.113883.4.330.[country oid code]`

The submitter strongly suggests that a rule for building URLs for the identification of passports be specified as well. This pattern should be included in the identifier registry page.

A possible pattern could be `http://hl7.org/fhir/sid/passport/[ISO alpha-3 country code]`

Example: French passport → `http://hl7.org/fhir/sid/passport/fra`

(Unless someone else has already assigned a URI for this.)

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Define a standard URI pattern `http://hl7.org/fhir/sid/passport-{XXX}` where `{XXX}` is the ISO 3166-1 alpha-3 country code, and register this as an official NamingSystem in the FHIR identifier registry. The pattern would allow any valid ISO 3166-1 alpha-3 code as the suffix (e.g., `http://hl7.org/fhir/sid/passport-USA`, `http://hl7.org/fhir/sid/passport-FRA`). The existing OID hierarchy (2.16.840.1.113883.4.330) should be cross-referenced.
#### Justification
There is clear community demand for a standardized passport identifier URI, evidenced by multiple Zulip discussions, supporting comments from implementers in Israel and Germany, and an existing PR (HL7/fhir#1094). The OID hierarchy already exists but is cumbersome for FHIR usage. Lloyd McKenzie has endorsed creating such a SID. The HAPI FHIR validator has already been patched to accept this pattern, indicating de facto adoption. A template-based approach avoids the need to pre-register every individual country.

---

### Disposition B: Alternative Approach
#### Proposal
Define the pattern using a slash separator instead of a dash — `http://hl7.org/fhir/sid/passport/{XXX}` — which aligns with the original submitter's proposed format and is more consistent with hierarchical URI conventions. Register the NamingSystem with this pattern and update the validator accordingly.
#### Justification
The original request proposed `http://hl7.org/fhir/sid/passport/[country code]` using a slash separator, which follows standard URI path conventions and is arguably more natural for a hierarchical namespace. However, the community discussion has since converged on the dash-separated pattern (`passport-XXX`), and a validator fix has already been merged using that pattern. Changing to slash would require updating the validator regex and the existing PR, but would provide a cleaner long-term URI structure.

---

### Disposition C: Decline
#### Proposal
Decline the request on the grounds that the existing OID hierarchy (`urn:oid:2.16.840.1.113883.4.330.{country OID code}`) already provides a mechanism for identifying passport numbers by country, and implementers can use this OID-based system with the `urn:oid:` prefix in `Identifier.system`.
#### Justification
FHIR already supports OID-based system identifiers via the `urn:oid:` prefix. Introducing a parallel URI pattern creates maintenance burden and potential confusion about which system identifier to use. However, this disposition is weak because community consensus strongly favors URL-based identifiers over OIDs, the OID country codes differ from the ISO 3166 codes used in the proposed URL pattern, and multiple implementers have expressed concrete need for a standardized URI.

---

### Recommendation
**Recommended disposition:** A

The evidence strongly supports accepting this request. There is broad community consensus — from the original submitter, supporting commenters (José Costa Teixeira, Daniel Mechanik from Israel, Alexander Zautke and Simone Heckmann from Germany), and Lloyd McKenzie — that a standardized passport URI pattern is needed. The dash-separated pattern `http://hl7.org/fhir/sid/passport-{XXX}` has already achieved de facto adoption: a PR exists to add it to the FHIR registry (HL7/fhir#1094), the HAPI FHIR validator has been patched to accept it (hapifhir/org.hl7.fhir.core#512), and national affiliates like HL7 Germany have already built NamingSystems referencing this pattern. Passports are a common identifier in cross-border care, and the lack of a standardized URI has caused practical implementation difficulties including validator errors. The template-based approach using ISO 3166-1 alpha-3 country codes avoids the need to enumerate all countries individually while remaining predictable and interoperable.
