# FHIR-43438: Change message-reason-encounter from Draft to Informative

## Summary

**Ticket:** [FHIR-43438](https://jira.hl7.org/browse/FHIR-43438)
**Status:** Triaged | **Type:** Change Request | **Priority:** Medium
**Reporter:** Anthony Julian | **Assignee:** Unassigned
**Work Group:** HL7 Terminology Authority | **Specification:** FHIR Core (FHIR)
**Resolution:** Unresolved
**Created:** 2024-01-02 | **Updated:** 2024-01-02

The reporter requests that the `ValueSet/message-reason-encounter` (`ExampleMessageReasonCodes`) be changed from Draft status to Informative. This value set provides example message reason codes used when updating an encounter via admin-update, and is bound to `MessageHeader.reason`.

## Details

The value set `http://hl7.org/fhir/ValueSet/message-reason-encounter` includes codes from the `http://terminology.hl7.org/CodeSystem/message-reasons-encounter` code system. The reporter notes that its status is currently "draft" and proposes changing it to "informative" to better reflect its maturity and usage intent.

**Current state in the FHIR source repository (HL7/fhir):** The value set source file (`source/messageheader/valueset-message-reason-encounter.xml`) currently shows:
- `structuredefinition-standards-status` extension: **informative**
- `status`: **active**
- `experimental`: **true**

This suggests the standards-status may have already been updated to "informative" in the current build, though the ticket remains unresolved and has not been formally dispositioned.

**Context:** There is broader community discussion about the adequacy of message-reason-encounter. In a September 2025 Zulip thread (fhir-messages > MessageHeader.event), Anthony Julian noted that `MessageHeader.reason` is mapped to message-reason-encounter which "although optional, is inadequate for many use cases." Lloyd McKenzie confirmed that the reason binding can be removed without breaking backward compatibility since old instances would remain valid.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Already Applied)
**Recommendation: Accept the request and close the ticket as already applied.**

The value set source in the FHIR repository (`HL7/fhir`) already has `structuredefinition-standards-status` set to `informative` and `status` set to `active`. The requested change appears to have been implemented in the current build. The ticket should be resolved as **Persuasive** and marked as applied, noting that the standards-status is already set to informative in the current source.

**Verification needed:** Confirm that the UTG source-of-truth copy at `HL7/UTG` also reflects the informative status. If not, a corresponding UTG update should be tracked.

### Disposition B: Persuasive with Modification — Remove the Binding Entirely
**Instead of changing status, consider removing the message-reason-encounter binding from MessageHeader.reason.**

Given the broader community feedback (FHIR-20881, Zulip discussions) that message-reason-encounter is inadequate for many use cases, and Lloyd McKenzie's confirmation that removing the reason binding would not be a breaking change, the workgroup could resolve this ticket as **Persuasive with Modification** — acknowledging the status concern but recommending the binding be removed or loosened rather than merely re-classified. This would be a more comprehensive fix but goes beyond the original request and should be coordinated with Infrastructure & Messaging.

### Disposition C: Decline — Not Persuasive
**Decline the change request as unnecessary.**

The distinction between "draft" and "informative" for an example-bound value set with `experimental: true` is largely semantic. Since the value set is already marked as experimental and carries only an example binding, changing its standards-status provides minimal practical benefit to implementers. The workgroup could resolve as **Not Persuasive**, noting that the status accurately reflects the value set's nature as a draft example set.

## Recommendation

**Disposition A: Accept as Persuasive (Already Applied)** is recommended. The requested change to set the standards-status to "informative" has already been made in the FHIR source repository. The ticket should be formally resolved to reflect this. The broader question of whether the binding itself is adequate (Disposition B) is a separate concern better addressed through a dedicated ticket coordinated with Infrastructure & Messaging, and should not block closure of this straightforward status change request.
