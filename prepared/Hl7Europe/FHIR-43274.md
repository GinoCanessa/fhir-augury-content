## Description

The reporter asks why `Specimen.subject` has a cardinality of 0..1 in the [EU Laboratory Report Specimen profile](https://hl7.eu/fhir/laboratory/StructureDefinition-Specimen-eu-lab-definitions.html#Specimen.subject). The question is whether every specimen should have a link to its subject, or whether the intent is that the only link to the subject is via `DiagnosticReport.subject` (which has cardinality 1..1), with the assumption that any specimen in the report belongs to the same subject.

## Analysis

This is a **question** (not a change request) about a design decision in the EU Laboratory Report Specimen profile. The question touches on a well-established tension in FHIR:

1. **Base resource design**: The base FHIR Specimen resource intentionally keeps `subject` as 0..1 per FHIR methodology rules — elements are only made mandatory when the resource is unusable without them (see FHIR-17424, Lloyd McKenzie's rationale).

2. **Profile-level tightening**: Individual profiles (like US Core, At-Home In-Vitro Test) have chosen to tighten Specimen.subject to 1..1 when the use case demands it (see FHIR-38013, resolved as Persuasive).

3. **EU Lab Report context**: In a laboratory report context, the subject is always known (the patient who had the specimen collected). The EU Lab Report already requires `DiagnosticReport.subject` as 1..1. The question is whether redundancy at the Specimen level is necessary or whether the DiagnosticReport link is sufficient.

4. **Related EU Lab concerns**: FHIR-43128 raises a related issue — that specimen-to-observation linkage is also important, not just specimen-to-subject.

## Proposed Dispositions

### Disposition A: Considered — Question Answered (Explain the Design Rationale)

**Response:** The EU Laboratory Report Specimen profile inherits the base FHIR Specimen resource cardinality of 0..1 for `Specimen.subject`. This is consistent with the core FHIR design principle that elements are only required when the resource is unusable without them (see FHIR-17424). In the EU Lab Report context, the subject is always identifiable through `DiagnosticReport.subject` (1..1), so requiring it redundantly on Specimen is not strictly necessary. Implementers are encouraged to populate `Specimen.subject` when available, but the 0..1 cardinality allows flexibility for edge cases such as referenced specimens where the subject context is inherited from the containing report.

**Pros:** Addresses the question directly; aligns with established FHIR design rationale; does not require specification changes.
**Cons:** Does not resolve the underlying implementer expectation that subject should always be present on laboratory specimens.

### Disposition B: Persuasive — Tighten Cardinality to 1..1 in the EU Lab Specimen Profile

**Response:** In the EU Laboratory Report context, every specimen is associated with a known patient. The profile should be updated to constrain `Specimen.subject` to 1..1, ensuring explicit linkage. This is consistent with how other IGs (e.g., At-Home In-Vitro Test Report, per FHIR-38013) have addressed similar concerns. This does not affect the base FHIR resource cardinality.

**Pros:** Provides clarity for implementers; ensures data completeness; consistent with the mandatory subject on DiagnosticReport.
**Cons:** Reduces flexibility; may not account for edge cases where subject is inherited from context; requires a specification update.

### Disposition C: Not Persuasive — Maintain Current Cardinality

**Response:** The 0..1 cardinality for `Specimen.subject` in the EU Laboratory Report is an intentional design choice. The subject is already required at the DiagnosticReport level (1..1), and requiring it again on Specimen would create redundancy. The current design allows the specimen to inherit subject context from the report, which is a valid and common pattern in FHIR document bundles.

**Pros:** No specification change needed; avoids redundancy.
**Cons:** May leave implementers confused about where to find the subject linkage; does not align with the expectation that clinical specimens always have a known subject.

## Recommendation

**Disposition A: Considered — Question Answered** is recommended.

This is filed as a Question (not a Change Request), and the answer is well-established in FHIR design principles. The base resource keeps Specimen.subject as 0..1 intentionally (FHIR-17424), and the EU Lab Report's DiagnosticReport.subject already provides a mandatory subject link. If the work group later decides the EU Lab profile should tighten this constraint, that would be a separate change request. For now, answering the question with the rationale — and noting the precedent from FHIR-38013 where other IGs have chosen to tighten cardinality — provides the reporter with actionable guidance.
