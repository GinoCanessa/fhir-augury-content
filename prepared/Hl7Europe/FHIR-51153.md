## Description

The submitter questions the value of defining 17 required slices for MIME types in the EU Imaging Study Report IG. Specifically:

- It is unclear whether a WADO endpoint is actually required to support all 17 MIME types.
- The submitter asks the work group to clarify the intent behind the requirement and, if appropriate, express the rule differently (e.g., as a minimal value set rather than 17 required slices).

## Resolution Description

**Persuasive with Modification** — The resolution acknowledges that:

1. The list was intended to represent the required MIME types of a WADO-URI service.
2. The cardinalities are **wrong** and need to be corrected per [DICOM Part 18, Chapter 7 — Persistent Object Types](https://dicom.nema.org/dicom/2013/output/chtml/part18/chapter_7.html):
   - **WADO-URI** requires only `application/dicom` and `image/jpeg`.
   - **WADO-RS** requires only `application/dicom`, `application/octet-stream`, and `application/dicom+xml`.
3. The suggestion of creating a **ValueSet** for the optional and required MIME type lists is valid and will be adopted.

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification (Recommended)

**Action:** Accept the existing resolution. Fix the MIME type cardinalities in the EU Imaging Study Report IG to align with DICOM Part 18 Chapter 7 requirements (WADO-URI: `application/dicom` + `image/jpeg`; WADO-RS: `application/dicom` + `application/octet-stream` + `application/dicom+xml`). Create ValueSets distinguishing required from optional MIME types.

**Rationale:** The submitter's concern is valid — 17 required MIME type slices significantly over-constrains implementations without clear justification. The resolution correctly identifies the DICOM standard source of truth and proposes a practical solution (ValueSets for required vs optional). This aligns with related ticket FHIR-51334's concern about the scope of WADO-RS endpoint obligations and reduces implementer burden. The ValueSet approach provides extensibility while maintaining interoperability.

### Disposition B: Persuasive — Remove All Required MIME Type Slices

**Action:** Remove all required MIME type slices entirely, replacing them with a required binding to a single ValueSet of DICOM-standard MIME types, with no individual cardinality constraints on specific types.

**Rationale:** Rather than fixing individual cardinalities, this approach simplifies the profile by delegating MIME type requirements entirely to the DICOM standard and a referenced ValueSet. This avoids the maintenance burden of keeping the IG's slice cardinalities synchronized with evolving DICOM specifications. However, this may reduce the IG's value as a self-contained conformance specification.

### Disposition C: Not Persuasive — Retain Current Slices with Documentation

**Action:** Keep the 17 required slices but add extensive documentation explaining that these represent the complete MIME type repertoire expected of a conformant WADO endpoint per DICOM standards.

**Rationale:** The original intent was to express the complete MIME type requirements for WADO endpoints. If all 17 types are genuinely required by DICOM for full conformance, making them explicit in the profile is valuable for conformance testing. However, this contradicts the DICOM Part 18 standard cited in the resolution, which requires far fewer mandatory types, making this disposition factually unsupported.

## Recommendation

**Disposition A: Accept as Persuasive with Modification.** The resolution is well-grounded in the DICOM standard and directly addresses the submitter's concern. Reducing the required MIME type slices to match DICOM Part 18 Chapter 7 and introducing ValueSets for required vs optional types is the correct balance between interoperability and implementability. Coordination with FHIR-51334 (WADO-RS endpoint scope) and FHIR-51338 (WADO-URI vs WADO-RS references) should be considered to ensure consistent MIME type expectations across the IG.
