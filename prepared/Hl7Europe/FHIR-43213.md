## Summary

This ticket asks whether the EU Laboratory Report logical model element `LabReport.header.metadata.dateTime` (A.1.8.3 — "Report date and time") should be mapped to `DiagnosticReport.effectiveDateTime` or to `DiagnosticReport.issued`. The current mapping targets `effectiveDateTime`, but the reporter notes that the EU specification does not explicitly state this element represents the "clinically relevant" date/time. Since `effectiveDateTime` is defined in FHIR as the "clinically relevant time/time-period for report" and `issued` is "the date and time this version of the report was made available," the question is which semantics better match the EU model's "Report date and time."

## Details

In the FHIR DiagnosticReport resource:

- **`effectiveDateTime`** — The clinically relevant time or time-period for the report (e.g., when the specimen was collected or the observation was clinically relevant).
- **`issued`** — The instant at which this version of the report was released/made available to providers, typically after the report is verified or finalized.

The EU Laboratory Report logical model element A.1.8.3 is described as "Report date and time" under the `header.metadata` group, which suggests it represents administrative/metadata timing (when the report was produced or issued) rather than a clinically relevant measurement time. If this interpretation is correct, `DiagnosticReport.issued` would be a more semantically accurate mapping target.

This question has direct parallels to prior work in US Core (FHIR-34973) that sought to clarify the distinction between "time the measurement was taken" (`effective[x]`) vs. "time the measurement was reported" (`issued`), and to FHIR-31956 which requested more guidance on `DiagnosticReport.issued` usage. There is no existing resolution or comment thread on this ticket.

## Proposed Dispositions

### Disposition A: Persuasive — Change Mapping to `DiagnosticReport.issued`

**Rationale:** The EU logical model element A.1.8.3 is titled "Report date and time" and is located under `header.metadata`, strongly suggesting it represents an administrative/document metadata timestamp — when the report was produced or released — rather than a clinically relevant time. The FHIR definition of `issued` ("DateTime this version was made") aligns with this metadata semantics. Change the mapping from `DiagnosticReport.effectiveDateTime` to `DiagnosticReport.issued`.

**Action:** Update the EU Laboratory Report StructureDefinition mapping for A.1.8.3 (`LabReport.header.metadata.dateTime`) to target `DiagnosticReport.issued` instead of `DiagnosticReport.effectiveDateTime`. Add an explanatory comment noting the semantic distinction.

### Disposition B: Persuasive with Modification — Map to Both Elements with Guidance

**Rationale:** The EU specification's "Report date and time" may be ambiguous enough to serve as both a clinically relevant date and an administrative date in some implementations. Rather than choosing one, provide guidance that the element MAY map to `DiagnosticReport.issued` (preferred for metadata/administrative date) or to `DiagnosticReport.effectiveDateTime` (if implementers determine the date represents clinically relevant timing). Add a note clarifying the FHIR semantics of each element.

**Action:** Update the mapping documentation to include both potential targets with guidance text explaining when each is appropriate. Set `DiagnosticReport.issued` as the default/primary mapping and note `DiagnosticReport.effectiveDateTime` as an alternative when the report date is known to represent clinically relevant timing.

### Disposition C: Not Persuasive — Retain Current Mapping to `effectiveDateTime`

**Rationale:** The current mapping to `effectiveDateTime` may have been intentional to capture the date/time that is clinically relevant for the report context. Changing it could break existing implementations that depend on this mapping. Without a clearer statement from the EU specification authors about the intended semantics, the current mapping should be retained. Implementers can additionally populate `issued` if they have a separate administrative timestamp.

**Action:** No change. Optionally add a clarifying note in the mapping documentation explaining the rationale for the `effectiveDateTime` choice and advising implementers to also populate `issued` when a distinct administrative release date is available.

## Recommendation

**Disposition A: Persuasive — Change Mapping to `DiagnosticReport.issued`**

The placement of A.1.8.3 under `header.metadata` is a strong signal that this is an administrative/document timestamp rather than a clinically relevant measurement time. FHIR's definitions are clear: `effectiveDateTime` is for "clinically relevant time/time-period for report," while `issued` is for "DateTime this version was made." A "Report date and time" in a metadata header aligns squarely with `issued`. This interpretation is consistent with the clarifications made in FHIR-34973 (US Core) which drew the same distinction. Correcting this mapping now, while the EU Laboratory Report specification is still maturing, avoids embedding a semantic mismatch that would be harder to fix later.
