## Description

The reporter questions why the `Bundle.entry:specimen` element in the EU Laboratory Report Bundle profile has a cardinality of **0..*** instead of **1..***. The expectation is that at least one Specimen resource should be required in a laboratory report bundle. A commenter (rikim) extends the same question to `DiagnosticReport.specimen`, which is also optional in the EU Lab DiagnosticReport profile.

## Resolution Description

The resolution states that specimen is optional because **none of the defined profiles require specimen**. In some cases the specimen is **implicit in the test code** — meaning the test code itself identifies what type of specimen is expected, making an explicit Specimen resource unnecessary.

## Analysis

This ticket is part of a cluster of questions (FHIR-43271, FHIR-43272, FHIR-43273) all filed by Ulrike Merrick on 2023-11-27 about why various key resources in the EU Laboratory Report Bundle profile are optional (0..*) rather than required (1..*). The resolution explains that specimen optionality is intentional because some laboratory tests have the specimen type implicit in the test code itself, making an explicit Specimen resource unnecessary. This is consistent with the general FHIR design principle of keeping base constraints minimal and allowing further profiling at national/implementation levels.

The related ticket FHIR-43128 provides additional context: HL7 Germany raised a concern that specimen should be assigned at the Observation level (not just DiagnosticReport level) to clarify which specimen each observation came from — that ticket was resolved as "Not Persuasive with Modification," suggesting the work group acknowledged the concern but took a modified approach.

## Proposed Dispositions

### Disposition A: Confirm "Considered - Question Answered" (No Change)
Accept the existing resolution as-is. The question has been adequately answered: specimen is optional because some lab tests have the specimen implicit in the test code, and none of the defined profiles require it. The 0..* cardinality provides flexibility for implementations that do or do not include specimen. No specification change is needed.

### Disposition B: Considered — Question Answered with Additional Guidance
Affirm the resolution but recommend adding an **implementation note** or **comment** in the EU Laboratory Report profile explaining *why* specimen is optional (i.e., that some tests have implicit specimens). This would address future questions from implementers encountering the same concern, improving the specification's self-documentation without changing any cardinality constraints.

### Disposition C: Persuasive — Require Specimen with Must-Support
Change `Bundle.entry:specimen` cardinality to **1..*** and mark `DiagnosticReport.specimen` as must-support. This would enforce that at least one specimen is always included in the bundle, addressing the reporter's original expectation. However, this would break the existing design rationale (implicit specimens) and conflict with the resolved FHIR-43128 approach.

## Recommendation

**Disposition A: Confirm "Considered - Question Answered" (No Change)**

The existing resolution is well-reasoned and consistent with FHIR design principles. The specimen is intentionally optional because the test code can implicitly identify the specimen type, making an explicit Specimen resource unnecessary in some cases. The sibling tickets (FHIR-43272, FHIR-43273) and the related FHIR-43128 resolution collectively demonstrate that the work group has already deliberated on cardinality choices for the EU Lab Bundle. Requiring specimen (Disposition C) would impose unnecessary burden on implementations where the specimen is implicit, and while adding guidance (Disposition B) has merit, the resolution text already provides adequate explanation. The work group should confirm the existing resolution.
