## Description

The submitter questions why the cardinality for `Bundle.entry:serviceRequest` in the EU Laboratory Report bundle profile is **0..*** rather than **1..***. The core question is: **can a laboratory report exist without a ServiceRequest?**

The referenced profile element is at:
- [Bundle-eu-lab: Bundle.entry:serviceRequest](https://hl7.eu/fhir/laboratory/StructureDefinition-Bundle-eu-lab-definitions.html#Bundle.entry:serviceRequest)

A comment by **rikim** extends the same question to `DiagnosticReport.basedOn` in the EU Lab DiagnosticReport profile, which also references ServiceRequest with optional (0..*) cardinality:
- [DiagnosticReport-eu-lab: DiagnosticReport.basedOn](https://hl7.eu/fhir/laboratory/StructureDefinition-DiagnosticReport-eu-lab-definitions.html#DiagnosticReport.basedOn)

## Analysis

### Context

The EU Laboratory Report IG defines a document bundle profile (`Bundle-eu-lab`) that includes sliced entries for various resource types. The `serviceRequest` slice has cardinality **0..*** , meaning a lab report bundle can be valid without including any ServiceRequest resource. Similarly, `DiagnosticReport.basedOn` (which references the originating order/ServiceRequest) is optional at **0..*** .

### Key Considerations

1. **Clinical reality**: Not all laboratory results originate from an explicit electronic order. Walk-in/self-referred testing, point-of-care results, historical results, and results from external labs may not have a corresponding ServiceRequest available in the sending system.

2. **Pattern in the IG**: The sibling tickets (FHIR-43271 for specimen, FHIR-43273 for observation) show this is a broader design pattern in the EU Lab IG — multiple entry types are optional at 0..* . FHIR-43271 was resolved as "Question answered," suggesting the workgroup has already addressed this pattern for specimen.

3. **Related resolution**: FHIR-51567 (Persuasive with Modification) acknowledged that `DiagnosticReport.basedOn` is the primary mechanism for linking to ServiceRequest, and found Composition-level duplication unnecessary — reinforcing that the ServiceRequest linkage belongs at the DiagnosticReport level but need not always be present.

4. **Base FHIR alignment**: In the base FHIR specification, `DiagnosticReport.basedOn` is 0..* , acknowledging that diagnostic reports can exist independently of a formal order.

## Proposed Dispositions

### Disposition A: Considered — Question Answered (Recommended)

**Rationale:** The 0..* cardinality for `Bundle.entry:serviceRequest` is intentional. Not all laboratory reports are associated with an explicit, electronically-captured ServiceRequest. Common scenarios include walk-in/self-referred testing, results from external laboratories, historical/legacy results, and point-of-care testing where no formal order was placed. The base FHIR `DiagnosticReport.basedOn` element is also 0..* for the same reason. This is consistent with the design pattern applied to other optional entries in the EU Lab bundle (e.g., specimen in FHIR-43271, which was resolved with the same answer). Implementations that require a ServiceRequest can apply additional constraints in their derived profiles.

**Action:** No specification change. Provide the above explanation as the answer.

### Disposition B: Persuasive — Tighten to 1..*

**Rationale:** A laboratory report should always be traceable to an originating order for proper clinical context, accountability, and billing. Requiring at least one ServiceRequest in the bundle (1..*) and at least one `DiagnosticReport.basedOn` reference would enforce this traceability. Systems that cannot provide the full ServiceRequest could use a minimal resource with just an identifier.

**Action:** Change `Bundle.entry:serviceRequest` cardinality from 0..* to 1..* in the EU Lab bundle profile. Similarly update `DiagnosticReport.basedOn` cardinality to 1..* in the EU Lab DiagnosticReport profile.

### Disposition C: Persuasive with Modification — Add Guidance Without Changing Cardinality

**Rationale:** While requiring ServiceRequest universally is too restrictive, the IG should explain why it is optional and provide guidance on when it should be included. Add narrative documentation explaining the design rationale and recommend that implementations include ServiceRequest whenever the ordering context is available.

**Action:** Keep 0..* cardinality but add implementation guidance in the IG narrative explaining the design decision and recommending inclusion of ServiceRequest when available.

## Recommendation

**Disposition A: Considered — Question Answered**

This is a question (not a change request), and the 0..* cardinality is a deliberate, well-established design choice consistent with both the base FHIR specification and the broader EU Lab IG pattern. The sibling ticket FHIR-43271 (specimen optionality) was already resolved with the same "Question answered" disposition, providing clear precedent. Requiring ServiceRequest would break valid real-world scenarios where no electronic order exists.
