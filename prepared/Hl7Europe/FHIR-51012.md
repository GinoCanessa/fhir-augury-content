## Description

The reporter notes that the IM ImagingStudy Manifest Bundle profile does not constrain `Bundle.type` to a specific value. The current profile allows all Bundle types (document, message, transaction, batch, searchset, collection, etc.), which is under-specified for implementers. The reporter suggests `collection` as a possible fixed value.

## Analysis

This is a straightforward profile constraint request. The IM ImagingStudy Manifest Bundle profile currently inherits all allowed `Bundle.type` values from the base Bundle resource without constraining them. Since the manifest serves a specific purpose (grouping imaging study references for exchange), allowing all bundle types creates ambiguity for implementers and increases implementation burden unnecessarily.

The closely related FHIR-51388 explores this same gap in greater detail, discussing two perspectives:
1. **Single-resource perspective**: Using `searchset` bundles via FHIR search on ImagingStudy.
2. **Document perspective**: Using a `document` or `collection` bundle for manifest exchange.

FHIR-51388 suggests that a FHIR `document` with a full Composition may be overkill for the manifest use case (unlike the Imaging Report, which naturally fits the document model). The `collection` type appears to be the most appropriate since the manifest groups related resources without requiring transaction semantics or a Composition.

The already-applied FHIR-51165 (removing inapplicable elements like search, request, response from the manifest bundle) further supports constraining the bundle type, since those removed elements are only relevant for certain bundle types.

FHIR-51013 (rejected) confirms the manifest concept itself is staying, so properly constraining its bundle type is necessary.

## Proposed Dispositions

### Disposition A: Persuasive (Constrain to `collection`)
**Action:** Fix `Bundle.type` to `collection` on the IM ImagingStudy Manifest profile.

**Rationale:** The `collection` type is the most appropriate for the manifest use case. It groups related resources without requiring a Composition (unlike `document`), without implying transaction semantics (unlike `transaction`/`batch`), and without implying search results (unlike `searchset`). This aligns with the reporter's suggestion and the analysis in FHIR-51388. The already-applied FHIR-51165 (removing search/request/response elements) is consistent with a `collection` type.

### Disposition B: Persuasive with Modification (Constrain to `collection` or `document`, coordinate with FHIR-51388)
**Action:** Allow either `collection` or `document` as the constrained `Bundle.type`, and coordinate resolution with FHIR-51388 to ensure consistent guidance on when each type should be used.

**Rationale:** While `collection` is the simpler choice, some exchange workflows (particularly MHD-based document exchange) may benefit from the `document` type. Allowing both with clear guidance covers both workflow patterns. This should be resolved jointly with FHIR-51388 to avoid conflicting decisions.

### Disposition C: Persuasive with Modification (Defer to FHIR-51388)
**Action:** Acknowledge the issue as valid but defer the resolution to FHIR-51388, which provides a more comprehensive analysis of the manifest bundle exchange model. Apply the same fix to both tickets.

**Rationale:** FHIR-51388 covers the same ground with significantly more detail and context about workflow considerations. Resolving both tickets together avoids duplication and ensures a holistic approach to the manifest bundle design.

## Recommendation

**Disposition C: Persuasive with Modification (Defer to FHIR-51388)** — FHIR-51388 is the more comprehensive ticket covering this exact issue with detailed workflow analysis. The workgroup should acknowledge FHIR-51012 as valid, resolve it as Persuasive with Modification noting that the fix will be applied as part of FHIR-51388's resolution. This avoids duplicate work and ensures the bundle type decision accounts for all exchange workflow considerations raised across both tickets. The most likely outcome is constraining to `collection`, consistent with the reporter's suggestion and the manifest's purpose.
