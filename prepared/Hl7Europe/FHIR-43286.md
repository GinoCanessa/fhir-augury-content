## Summary

The submitter questions whether `ServiceRequest.intent` in the EU Laboratory Report IG should use a more constrained value set. The base FHIR `RequestIntent` value set includes codes such as `proposal`, `plan`, and `directive` that do not seem appropriate in the context of a laboratory report, where `order` (or possibly `original-order`, `reflex-order`, `filler-order`, `instance-order`) would be the expected intents.

The resolution text states: **"Remove the proposed concepts"** — indicating the work group agreed to constrain the value set by removing inappropriate intent codes from the EU Lab profile.

## Proposed Dispositions

### Disposition A: Confirm Resolution — Constrain the Value Set (Recommended)

**Action:** Accept the existing resolution ("Remove the proposed concepts"). The EU Laboratory Report profile should constrain `ServiceRequest.intent` to a subset of the `RequestIntent` value set appropriate for lab orders (e.g., `order`, `original-order`, `reflex-order`, `filler-order`, `instance-order`), excluding `proposal`, `plan`, and `directive`.

**Rationale:** The submitter's concern is valid and widely echoed across jurisdictions (AU eRequesting fixed to `order`, US Core debated similar constraints). In a lab report context, a service request represents an actual order — not a proposal or plan. The resolution text already directs removal of inappropriate concepts. This is a profile-level constraint and does not affect the base FHIR value set. Multiple related tickets (FHIR-52390, FHIR-54947, FHIR-46771) confirm this is a recurring community concern.

**Formal Disposition:** Considered - Question Answered (retain existing resolution)

---

### Disposition B: Constrain More Narrowly to Only `order`

**Action:** Go further than removing `proposal`/`plan`/`directive` — fix `ServiceRequest.intent` to the single value `order` in the EU Lab profile, similar to the AU eRequesting approach.

**Rationale:** In practice, lab reports reference orders that have already been fulfilled. Sub-types like `original-order`, `reflex-order`, and `filler-order` may add complexity without clear EU use cases. A fixed value of `order` simplifies implementation and aligns with AU eRequesting precedent (FHIR-52354, FHIR-52390).

**Formal Disposition:** Considered - Question Answered (with implementation note to fix to `order`)

---

### Disposition C: Defer to Base Spec Resolution

**Action:** Close the question and defer to ongoing base specification work on the `RequestIntent` value set (FHIR-56142, FHIR-52001). Do not make profile-level changes in the EU Lab IG at this time.

**Rationale:** The base FHIR spec is actively considering changes to the `RequestIntent` value set, including adding an escape valve (FHIR-56142) and reassessing `directive` (FHIR-52001). Making profile-level constraints now may conflict with future base spec changes. However, this risks leaving the EU Lab IG with an overly broad value set in the interim.

**Formal Disposition:** Considered - No Action (defer to base spec)

---

## Recommendation

**Disposition A: Confirm Resolution — Constrain the Value Set.** The resolution is already recorded as "Remove the proposed concepts," which aligns with the submitter's question and community consensus across multiple jurisdictions. The EU Lab profile should constrain `ServiceRequest.intent` to order-related codes, excluding `proposal`, `plan`, and `directive`. This is a sensible profile-level constraint that does not conflict with base spec evolution and addresses a widely recognized issue.
