## Summary

This ticket requests that generic (non-imaging-specific) artifacts currently housed in the EU Imaging Study Report IG be moved to the HL7 Europe Base or Core IG. The submitter identifies two categories of artifacts that should be relocated:

1. **EHDS mappings** from the EHDS Logical Model to profiles that already exist in EU Base (e.g., `EHDSPatientMap`) — these mappings are duplicated or misplaced in the Imaging IG.
2. **Generic EU profiles** that were created within the Imaging IG but are not imaging-specific (e.g., `ProcedureEU`, which serves as the parent profile for `IM Imaging Procedure`).

The rationale is that these generic components belong in the EU Base/Core IG where they can be properly validated, balloted, and reused across all EU IGs.

## Details

The EU Imaging Study Report IG (FHIR R5) was developed as one of the HL7 Europe EHDS-aligned Implementation Guides. During its development, several reusable artifacts were created within the Imaging IG for practical reasons, but they are domain-agnostic and applicable across multiple EU IGs (Laboratory, Hospital Discharge Report, Medication, Patient Summary, etc.).

Specific examples cited:
- **`EHDSPatientMap`** — a mapping from the EHDS Logical Model to EU Patient profiles; this belongs alongside the EU Base Patient profile.
- **`ProcedureEU`** — a generic EU Procedure profile that the imaging-specific `IM Imaging Procedure` is derived from. Since `ProcedureEU` is not imaging-specific, it should live in EU Base where other IGs (e.g., Lab, HDR) can also derive from it.

This ticket is part of a broader pattern of requests to rationalize the artifact placement across the HL7 Europe IG ecosystem. Multiple related tickets have been filed during the EU Imaging IG ballot addressing the same fundamental concern: generic models, mappings, and profiles should be centralized in EU Base/Core rather than scattered across domain-specific IGs.

The EU Imaging IG is being developed in FHIR R5 (while other EU IGs like Lab and HDR are in R4), which adds complexity to cross-IG profile sharing. Any migration of generic profiles to EU Base would need to account for multi-version support.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)
**Action:** Accept the request as persuasive and plan the migration of generic components from the Imaging IG to EU Base/Core.

**Rationale:** The submitter correctly identifies that domain-agnostic artifacts like `ProcedureEU` and EHDS mappings for non-imaging resources do not belong in a domain-specific IG. Centralizing them in EU Base/Core:
- Prevents duplication across IGs (Lab, HDR, Medication, etc. all need `ProcedureEU`).
- Ensures these shared artifacts receive proper cross-community validation and ballot.
- Is consistent with the resolution of FHIR-51124 (Applied, Persuasive with Modification), which already moved some mappings.
- Aligns with the architectural direction established by related tickets (FHIR-51123, FHIR-51398).

**Implementation:** Identify all generic profiles and EHDS mappings in the Imaging IG, coordinate with the EU Base/Core IG editors, migrate the artifacts, and update the Imaging IG to depend on the published EU Base/Core versions.

### Disposition B: Defer (Considered for Future Use)
**Action:** Acknowledge the request as valid but defer to a future ballot cycle, consistent with the disposition of the near-duplicate FHIR-51398.

**Rationale:** While the architectural direction is correct, the migration involves cross-IG coordination complexity:
- The Imaging IG is R5-only while EU Base currently targets R4 — profiles need multi-version alignment before migration.
- FHIR-51123 (the model-migration counterpart) is still "Waiting for Input," indicating the scope of what to move is not yet finalized.
- FHIR-51398 (the near-duplicate) was already deferred as "Considered for Future Use," suggesting the workgroup has already assessed that this cannot be resolved in the current ballot cycle.

**Implementation:** Mark as "Considered for Future Use" and consolidate with FHIR-51398 and FHIR-51123 into a single work item for the next ballot cycle.

### Disposition C: Decline (Not Persuasive)
**Action:** Decline the request on the grounds that it duplicates existing tracked work.

**Rationale:** The near-duplicate ticket FHIR-51398 already captures this exact request ("Move more generic EU Resource and Data Type Profiles to the HL7 Europe Base and Core FHIR IG") and has been formally dispositioned as "Considered for Future Use." Additionally, FHIR-51124 (mappings) has already been applied, and FHIR-51123 (models) is being actively tracked. Accepting another ticket for the same work creates tracking overhead without advancing the resolution.

**Implementation:** Close as duplicate of FHIR-51398, linking both tickets for traceability.

## Recommendation

**Disposition B: Defer (Considered for Future Use)** is recommended.

The request is architecturally sound and aligns with the community consensus that generic artifacts should live in EU Base/Core. However, the near-duplicate FHIR-51398 was already deferred, and the prerequisite work (FHIR-51123 scoping which models to move) remains unresolved. Deferring maintains consistency with prior workgroup decisions while keeping the request on the roadmap. The ticket should be linked to FHIR-51398 and FHIR-51123 so they can be addressed together in a future ballot cycle when the R4/R5 multi-version strategy for EU Base is clarified.
