## Description Summary

The submitter identifies that the EU Imaging Study Report Composition resource includes several authorship-related fields — **Composition.author** (1..\*), **Composition.attester** (0..\*), **Composition.legalAuthenticator** (0..\*), and **Composition.custodian** (0..1) — but lacks guidance on when each should be used. Key concerns raised:

1. **Multiple authors in imaging workflows**: Reports may have multiple authors from different specialties, or involve resident/attending collaboration, making authorship roles ambiguous.
2. **Legacy data**: Historical imaging data may not clearly document authorship roles, yet should still be shareable.
3. **System-generated Compositions**: Many Compositions will be assembled ad-hoc at query time (delayed-assembly), meaning no user is present to attest. Implementers should not assume the presence of Attester or legalAuthenticator, and should reference DiagnosticReport fields (performer, resultsInterpreter) instead.

## Resolution Summary

Resolution is **Persuasive**. The work group agreed to:

1. Update the Composition resource description to clearly differentiate the authorship fields (author, attester, legalAuthenticator, custodian).
2. Explain in the use case and imaging report sections that DiagnosticReport is the main entry point and Composition is a rendering of the report.
3. Flip the overview picture to place DiagnosticReport on the left and Composition on the right.
4. Change the unstructured report example to rely on just a DiagnosticReport (not DocumentReference plus an example).

A detailed proposed resolution was provided in comments by JIRAUSER28670 (2025-07-15), covering:
- **Composition.author**: Guidance for resident/attending models, collaborative reads, scribe workflows, and legacy data.
- **Composition.attester**: Result validation vs. legal authentication modes, with a note that dynamically generated Compositions may lack attesters.
- **Composition.custodian**: Organization responsible for report integrity and availability.

JIRAUSER28578 (2025-08-07) endorsed the guidance and suggested linking Composition fields directly to DiagnosticReport fields (e.g., Composition.author from DiagnosticReport.resultsInterpreter:author), noting a cardinality mismatch worth a separate issue.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Action**: Accept the resolution as stated and apply the proposed guidance text.

**Rationale**: The submitter raises a legitimate implementer confusion risk around four overlapping Composition authorship fields. The proposed resolution is thorough and well-constructed:
- It provides scenario-based guidance (resident/attending, collaborative reads, scribes, legacy data) that directly addresses the problem.
- It appropriately positions DiagnosticReport as the primary entry point, aligning with the practical reality that many systems store data as DiagnosticReports rather than Compositions.
- The suggestion to flip the overview diagram reinforces this conceptual shift.
- The comment from JIRAUSER28578 adds value by proposing explicit field-to-field mappings between DiagnosticReport and Composition.
- Sister ticket FHIR-51288 (overread workflows) has already been Applied with a similar approach, establishing precedent.

**Changes Required**: Add the proposed guidance text to the ImComposition page. Update the overview diagram. Revise the unstructured report example to use DiagnosticReport directly. Consider incorporating the DiagnosticReport field mapping suggestions from the JIRAUSER28578 comment.

### Disposition B: Persuasive with Modification — Require Explicit Field Mappings

**Action**: Accept the resolution but require formal mapping documentation between Composition and DiagnosticReport authorship fields.

**Rationale**: The proposed guidance text is good but stops short of explicitly specifying how each Composition authorship field maps to/from DiagnosticReport fields. JIRAUSER28578's comment highlights that Composition.author should come from DiagnosticReport.resultsInterpreter:author, but notes a cardinality mismatch (Composition.author is 1..\*, DiagnosticReport.resultsInterpreter is 0..\*). Without formal mappings, implementers building system-generated Compositions will still face ambiguity.

**Changes Required**: Everything in Disposition A, plus: add a mapping table showing Composition field → DiagnosticReport source field for author, attester, and custodian. File a separate ticket for the cardinality mismatch between Composition.author and DiagnosticReport.resultsInterpreter.

### Disposition C: Not Persuasive — Defer to Xt-EHR Alignment

**Action**: Defer action pending alignment with the Xt-EHR common model for attester and legalAuthenticator.

**Rationale**: JIRAUSER28670's comment (2025-07-16) flags that the proposed guidance should be checked against the Xt-EHR common model definitions for [EHDSImagingReport.header.attestation.attester](https://build.fhir.org/ig/Xt-EHR/xt-ehr-common/StructureDefinition-EHDSImagingReport-definitions.html#EHDSImagingReport.header.attestation.attester) and [EHDSImagingReport.header.legalAuthentication.legalAuthenticator](https://build.fhir.org/ig/Xt-EHR/xt-ehr-common/StructureDefinition-EHDSImagingReport-definitions.html#EHDSImagingReport.header.legalAuthentication.legalAuthenticator). If the IG's guidance diverges from the Xt-EHR model, it could create cross-specification inconsistencies within the EU ecosystem.

**Changes Required**: No changes until Xt-EHR alignment is confirmed. This may delay the guidance but reduces the risk of conflicting definitions.

## Recommendation

**Disposition A: Accept as Persuasive.** The proposed guidance comprehensively addresses the implementer confusion identified in the ticket. The resolution text covers the key imaging workflow scenarios (resident/attending, collaborative reads, scribes, legacy data, system-generated Compositions) and correctly positions DiagnosticReport as the authoritative source for authorship information. The approach is consistent with the already-Applied sister ticket FHIR-51288, and the overall direction of making DiagnosticReport the primary entry point has broad support in the comments. The Xt-EHR alignment concern from Disposition C is valid but can be addressed as a follow-up rather than blocking this guidance, which provides immediate implementer value.
