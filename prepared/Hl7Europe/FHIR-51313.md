## Summary

The submitter questions why `AppointmentResponse` is listed as a valid reference target in the manifest obligations for `ImImagingStudy.basedOn` in the EU Imaging Study Report IG. The title requests its removal, and the description asks whether there is an example workflow justifying this reference. This concern aligns with an already-resolved change in FHIR Core ([FHIR-49681](https://jira.hl7.org/browse/FHIR-49681)), where `AppointmentResponse` was removed from `ImagingStudy.basedOn` with a "Persuasive" resolution.

## Details

### Description

The submitter notes that in the manifest obligations for `ImImagingStudy`, `AppointmentResponse` is listed as a valid reference for `ImagingStudy.basedOn`. They ask whether there is an example workflow where this reference would actually be used—implying that no such workflow exists and the reference type should be removed.

### Comments

No comments have been recorded on this ticket.

### Context from Related Tickets

- **[FHIR-49681](https://jira.hl7.org/browse/FHIR-49681)** (Applied, Persuasive — Imaging Integration / FHIR Core): Directly addressed the same issue at the base resource level. The resolution states that `AppointmentResponse` was likely included due to the Event workflow pattern, but `ImagingStudy` is no longer part of that pattern. `AppointmentResponse` is not a request resource—it represents an individual participant's acceptance of an appointment—and therefore does not belong in `basedOn`. This change has already been applied to FHIR Core.

- **[FHIR-38722](https://jira.hl7.org/browse/FHIR-38722)** (Published, Persuasive — Structured Documents / FHIR Core): Raised the same conceptual issue for `DocumentReference.basedOn`, noting that `AppointmentResponse` is not a Request resource and should not appear in `basedOn` elements. This was also accepted and applied.

- **[FHIR-47658](https://jira.hl7.org/browse/FHIR-47658)** (Duplicate, Persuasive — Imaging Integration / FHIR Core): Questioned the detailed description of `ImagingStudy.basedOn`, including the presence of `AppointmentResponse`, and asked whether DICOM tags map to it. Marked as duplicate of the broader basedOn cleanup.

- **[FHIR-47681](https://jira.hl7.org/browse/FHIR-47681)** (Applied, Persuasive — Patient Administration / FHIR Core): Identified confusion between `AppointmentRequest` (a non-existent resource referenced in documentation) and `AppointmentResponse`, leading to editorial corrections.

- **[FHIR-14452](https://jira.hl7.org/browse/FHIR-14452)** (Published, Persuasive with Modification — Imaging Integration / FHIR Core): Flagged that `ImagingStudy` did not align with workflow patterns, contributing to the original inclusion of inappropriate reference types in `basedOn`.

## Proposed Dispositions

### Disposition A: Persuasive (Accept — Remove AppointmentResponse)

**Action:** Remove `AppointmentResponse` from the set of valid reference targets in the manifest obligations for `ImImagingStudy.basedOn` in the EU Imaging Study Report IG.

**Rationale:** FHIR Core has already resolved this issue via [FHIR-49681](https://jira.hl7.org/browse/FHIR-49681), removing `AppointmentResponse` from `ImagingStudy.basedOn` at the base resource level. The EU Imaging Study Report IG profile should align with this change. `AppointmentResponse` is not a Request resource—it represents a participant's acceptance of an appointment—and has no legitimate use case as a `basedOn` reference for imaging studies. The same principle was applied in [FHIR-38722](https://jira.hl7.org/browse/FHIR-38722) for `DocumentReference.basedOn`.

### Disposition B: Persuasive with Modification (Defer to Base Spec Alignment)

**Action:** Do not make an explicit profile-level change now. Instead, note that this will be automatically resolved when the IG is rebased onto a version of FHIR Core that includes the FHIR-49681 change, since the base resource will no longer include `AppointmentResponse` in `basedOn`.

**Rationale:** Since the base resource change is already applied, the profile will inherit the correction upon rebase. An explicit profile-level removal is unnecessary if the IG is tracking current FHIR Core snapshots. However, the ticket should be accepted as valid and tracked until the rebase is confirmed.

### Disposition C: Not Persuasive (Decline)

**Action:** Retain `AppointmentResponse` in the manifest obligations for `ImImagingStudy.basedOn`.

**Rationale:** This disposition would argue that the EU Imaging Study Report IG has region-specific workflows where `AppointmentResponse` might be referenced in `basedOn`—for example, linking an imaging study to the specific participant acceptance that triggered the study. However, no such workflow has been identified by the submitter or in any related discussions, making this rationale speculative.

## Recommendation

**Disposition A: Persuasive** is recommended. The removal of `AppointmentResponse` from `ImagingStudy.basedOn` is well-established precedent in FHIR Core (FHIR-49681, Applied/Persuasive) and in analogous resources (FHIR-38722). `AppointmentResponse` is semantically incorrect as a `basedOn` target because it is not a Request resource. The EU IG should proactively align with this change rather than waiting for a passive rebase, ensuring implementers are not misled by the current manifest obligations.
