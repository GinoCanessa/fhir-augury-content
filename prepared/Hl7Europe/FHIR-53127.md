## Summary

This ticket identifies multiple problems with how `Observation.accreditationStatus` is implemented in the EU Laboratory Report IG. The current mapping to `Observation.performer.qualification` via PractitionerRole is invalid because the Observation profile does not allow PractitionerRole as a performer type. The alternative mapping to `Organization.qualification` is also non-functional in R4 since Organization lacks a `qualification` element. Additionally, accreditation applies per laboratory test rather than to an organization as a whole, making the Organization-level mapping conceptually incorrect. The resolution proposes creating a simple EU accreditation extension modeled after the Norwegian NILAR extension, applicable to both Observation and ServiceRequest resources.

## Details

### Description

The reporter identifies two mapping issues with `Observation.accreditationStatus`:

1. **PractitionerRole mapping**: The element is mapped to `Observation.performer.qualification` (performer ofType PractitionerRole), but the Observation profile does not allow PractitionerRole as a performer type. This mapping should be removed.

2. **Organization mapping**: There is also a mapping to `Observation.performer.organization.qualification` (performer ofType Organization), but Organization does not have a `qualification` element in R4. Furthermore, this mapping is impractical because labs are accredited per laboratory test, not as a whole.

The reporter asks whether ObservationDefinition or a custom Observation extension would be a better approach.

### Resolution Description

For R4, create a simple accreditation EU extension based on the Norwegian NILAR extension ([StructureDefinition-nilar-accredited](https://hl7norway.github.io/NILAR/DiagnosticReportIG/CurrentBuild/StructureDefinition-nilar-accredited.html)). The extension should be attachable to:

- **Observation** — indicating whether the lab test within the lab was/is accredited
- **ServiceRequest** — indicating the request should only be fulfilled by an accredited lab

For this version the extension will be included in the guide using the canonical for EU extensions.

## Proposed Dispositions

### Disposition A: Accept (Persuasive with Modification)

**Recommendation**: Accept the resolution as proposed. Create a new EU accreditation extension modeled on the Norwegian NILAR `nilar-accredited` extension, applicable to both `Observation` and `ServiceRequest`. Remove the invalid PractitionerRole and Organization.qualification mappings. Include the extension in the EU Laboratory Report IG using the EU extensions canonical. This approach provides an immediate R4 solution while the lab semantic TF works with O&O on a potential core resource addition for R6.

**Rationale**: The resolution follows directly from the lab semantic TF's thorough analysis. The Norwegian extension provides a proven, real-world model. Applying accreditation at the Observation level (per test) correctly reflects that labs are accredited per test, not as a whole. The dual applicability to ServiceRequest addresses the ordering use case. The R6 proposal (adding accreditation to Observation core) is handled separately.

### Disposition B: Accept with Enhanced Extension Structure

**Recommendation**: Accept the proposal but request that the new EU extension include additional data elements beyond simple true/false accreditation status. Specifically, include: (1) accreditation status (CodeableConcept rather than boolean, to allow for "pending", "expired", etc.), (2) date/period of accreditation validity, and (3) optional reference to the issuing agency (Organization). This richer model was discussed in the lab semantic TF (2025-12-12 comment) and would better serve European use cases.

**Rationale**: The lab semantic TF identified these additional elements as important for the R6 proposal. Including them now in the R4 extension would avoid the need for a breaking change later, improve interoperability with national implementations that track accreditation validity periods, and provide a smoother migration path to R6. However, this adds complexity and may delay publication.

### Disposition C: Decline (Not Persuasive)

**Recommendation**: Decline the change request and defer to the R6 core resource proposal. Argue that adding a new EU-specific extension in R4 creates fragmentation, especially since the Norwegian extension already exists independently. Instead, recommend that implementers use the existing Norwegian extension directly or wait for the R6 core addition.

**Rationale**: Creating another extension (EU-specific) when a functionally equivalent one already exists (Norwegian NILAR) could lead to interoperability issues between European jurisdictions. The problem is real but may be better solved at the FHIR core level in R6 rather than through a proliferation of jurisdiction-specific extensions.

## Recommendation

**Disposition A: Accept (Persuasive with Modification)** is recommended. The resolution is well-grounded in the lab semantic TF's analysis, addresses the immediate R4 need with a proven approach (the Norwegian NILAR extension model), and correctly fixes the invalid PractitionerRole and Organization mappings. The dual applicability to Observation and ServiceRequest covers both the reporting and ordering use cases. The R6 core proposal is being handled separately via the Orders & Observations work group, so accepting this R4 extension does not preclude a more comprehensive solution in the future.
