## Summary

The reporter asks why the cardinality of `Bundle.entry:observation` in the EU Laboratory Report Bundle profile ([StructureDefinition-Bundle-eu-lab](https://hl7.eu/fhir/laboratory/StructureDefinition-Bundle-eu-lab-definitions.html#Bundle.entry:observation)) is `0..*` rather than `1..*`. The core question is: **can you have a laboratory report without any observation?**

A comment from **rikim** extends the question to `DiagnosticReport.result` in the [EU Lab DiagnosticReport profile](https://hl7.eu/fhir/laboratory/StructureDefinition-DiagnosticReport-eu-lab-definitions.html#DiagnosticReport.result), noting the same cardinality concern — if a DiagnosticReport must contain results, why is `result` optional?

## Analysis

This ticket is part of a series of questions (FHIR-43271, FHIR-43272, FHIR-43273) from the same reporter examining the cardinality choices in the EU Laboratory Report Bundle profile. The pattern across all three is the same: sliced `Bundle.entry` elements for key resources (Specimen, ServiceRequest, Observation) are all `0..*` rather than `1..*`.

The sibling ticket FHIR-43271 (Specimen) was already resolved as **"Considered - Question answered"**, suggesting the work group has previously addressed this class of question. The likely rationale is that the EU Lab Bundle profile is designed to accommodate diverse laboratory report use cases — including preliminary reports, reports with only narrative content, or reports where observations are carried in different structures. Mandating at least one observation entry at the bundle level would reduce this flexibility.

The comment from rikim adds an important dimension: if `DiagnosticReport.result` is also `0..*`, the optionality is consistent at both the bundle entry level and the resource content level. This suggests a deliberate design choice rather than an oversight.

The Zulip thread on "EU Lab DiagnosticReport - mandatory composition" provides additional context: the EU approach treats the DiagnosticReport as a holistic entity, and the design reflects a compromise between different implementation paradigms (FHIR Document, RESTful, messaging).

## Proposed Dispositions

### Disposition A: Considered — Question Answered (No Change)
**Rationale:** The `0..*` cardinality on `Bundle.entry:observation` is intentional. Not all laboratory reports require discrete observations at the bundle level — for example, preliminary reports, narrative-only reports, or reports where results are embedded in nested structures. The sibling ticket FHIR-43271 (Specimen optionality) was resolved with the same disposition. The `DiagnosticReport.result` element in the base FHIR specification is also `0..*`, and the EU Lab profile follows this convention. The work group should confirm this rationale and close the question.

**Suggested response text:** "The `0..*` cardinality is by design. The EU Laboratory Report Bundle must accommodate a range of use cases including preliminary reports and reports where observations may not yet be available or are conveyed through other mechanisms. This aligns with the base `DiagnosticReport.result` cardinality of `0..*` and the design choices documented in the IG."

### Disposition B: Persuasive — Tighten Cardinality to `1..*`
**Rationale:** A completed laboratory report should logically contain at least one observation. Making `Bundle.entry:observation` `1..*` would enforce a minimum data quality expectation and align with the clinical expectation that a lab report carries results. If preliminary or partial reports need accommodation, they could use a different profile or status-based constraints.

**Suggested response text:** "Agreed. A completed laboratory report should contain at least one observation. Change `Bundle.entry:observation` to `1..*` and consider adding an invariant that relaxes this for reports with `status` of `preliminary` or `partial`."

### Disposition C: Persuasive with Modification — Add Invariant Instead of Changing Base Cardinality
**Rationale:** Rather than changing the base cardinality, add a FHIRPath invariant requiring at least one observation entry when the DiagnosticReport status indicates a final or amended report. This preserves flexibility for preliminary reports while addressing the valid concern that a final lab report without observations is problematic.

**Suggested response text:** "The base cardinality remains `0..*` to support preliminary/partial reports, but a FHIRPath invariant will be added to require `Bundle.entry:observation` count >= 1 when the DiagnosticReport status is `final`, `amended`, or `corrected`."

## Recommendation

**Disposition A: Considered — Question Answered (No Change)**

This is the recommended disposition because:
1. The sibling ticket FHIR-43271 was already resolved with the same pattern ("Considered - Question answered"), establishing precedent for this class of question.
2. The `0..*` cardinality is consistent with the base FHIR `DiagnosticReport.result` element and reflects a deliberate design choice documented in the EU Lab IG's design rationale.
3. The EU Lab Report profile must accommodate multiple exchange paradigms (FHIR Document, RESTful, messaging) and report statuses (preliminary, partial, final), making a flexible cardinality appropriate.
4. Tightening the cardinality could break existing implementations and reduce the profile's applicability across the diverse European healthcare landscape.
