## Description

The reporter asks why `Bundle.type` in the [EU Laboratory Report Bundle profile](https://hl7.eu/fhir/laboratory/StructureDefinition-Bundle-eu-lab-definitions.html#Bundle.type) is not constrained. Since the stated goal of the IG is to produce a signable document, the reporter proposes that `Bundle.type` should be fixed to `document`.

## Zulip Discussions

### [europe > EU Lab DiagnosticReport - mandatory composition](https://chat.fhir.org/#narrow/stream/europe/topic/EU%20Lab%20DiagnosticReport%20-%20mandatory%20composition)
A 2026 discussion where **Giorgio Cangioli** (EU Lab IG author) stated:
> "the compromise achieved to take in account all the different needs has been that of having a Laboratory Report always represented by one and only one DiagnosticReport, but **always exchanged as a FHIR Document**"

He further clarified this is specific to the EU Lab Report; the UV Lab Report (universal realm) allows three exchange options including non-document bundles. This confirms the design intent that the EU Lab Report Bundle should always be of type `document`.

Notably, **Ralf Herzog** quoted this and compared the EU approach (holistic DiagnosticReport as a FHIR Document) with the US approach (multiple DiagnosticReports in a collection or document Bundle), reinforcing that the EU IG explicitly targets document Bundles.

## Analysis

The question is well-founded. The EU Laboratory Report IG is explicitly designed to exchange laboratory reports as FHIR Documents. A FHIR Document Bundle requires `Bundle.type = document` (per the FHIR spec, a Composition must be the first entry in a Bundle where `Bundle.type=document`). Leaving `Bundle.type` unconstrained in the profile is inconsistent with this design intent and could lead to implementers creating bundles of other types that would not be valid FHIR Documents.

The Zulip discussion provides strong evidence from the IG author (Giorgio Cangioli) that the EU Lab Report is "always exchanged as a FHIR Document," supporting the fix of `Bundle.type` to `document`. The related ticket FHIR-48278 (MeasureReport Bundle) demonstrates this is a common pattern — other IGs have also been asked to constrain their Bundle.type and resolved it with modification.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Fix `Bundle.type` to the value `document` in the EU Laboratory Report Bundle profile (`Bundle-eu-lab`).

**Rationale:** The IG's stated design choice is that EU Laboratory Reports are always exchanged as FHIR Documents. The IG author has confirmed this in Zulip. Constraining `Bundle.type` to `document` aligns the profile with its design intent, ensures Composition-first validation rules apply, and removes ambiguity for implementers. This is consistent with how FHIR-48278 was resolved for a similar issue.

### Disposition B: Considered — Question Answered
**Action:** Respond that the profile already implies document-type usage through its Composition and DiagnosticReport entry requirements, and that an explicit `Bundle.type` constraint is not strictly necessary because conformance tooling will enforce it.

**Rationale:** The profile already slices `Bundle.entry` with a required Composition as the first entry, which implicitly restricts valid instances to document Bundles. Adding an explicit `fixedCode` on `Bundle.type` would be redundant with the structural constraints. However, this disposition leaves a gap in profile clarity.

### Disposition C: Not Persuasive
**Action:** No change. Leave `Bundle.type` unconstrained.

**Rationale:** The profile may intentionally leave `Bundle.type` unconstrained to allow future flexibility for non-document exchange patterns (e.g., collection or message bundles). However, this contradicts the IG's current documented design choice and the author's public statements.

## Recommendation

**Disposition A: Persuasive** — Fix `Bundle.type` to `document`. The evidence from both the IG documentation and the IG author's Zulip statements is clear: the EU Laboratory Report is designed to always be exchanged as a FHIR Document. Explicitly constraining `Bundle.type` to `document` makes the profile self-documenting, prevents invalid bundle types, and aligns with the resolution pattern used in similar tickets (FHIR-48278). The question is valid and the fix is straightforward.
