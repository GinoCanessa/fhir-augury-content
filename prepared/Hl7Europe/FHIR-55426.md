# FHIR-55426: header.authorship mapping to Observation

## Summary

**Ticket:** [FHIR-55426](https://jira.hl7.org/browse/FHIR-55426)
**Status:** Triaged | **Type:** Question | **Priority:** Medium
**Resolution:** Unresolved
**Reporter:** Alexander Henket | **Assignee:** Unassigned
**Work Group:** HL7 Europe
**Specification:** EU Base and Core (R4) (FHIR)
**Labels:** NegativeVote, eu-ballot-jan-2026
**Created:** 2026-01-26 | **Updated:** 2026-02-13

The submitter notes that the FHIR Observation resource does not have a dedicated `author` element, and no relevant extension exists for that purpose. The EU model's `header.authorship` concept needs to map somewhere on Observation. The submitter argues that `Observation.performer` is the closest semantic fit and should suffice for the generic authorship requirement. The question asks whether the EU can formally confirm that `Observation.performer` is "equivalent enough" to `header.authorship`, allowing the matter to be closed rather than left as an open issue for individual IGs to solve.

## Analysis

This ticket is part of a cluster of NegativeVote ballot comments from Alexander Henket (eu-ballot-jan-2026) focusing on mapping clarity in the EU Base and Core MedicalTestResult Observation profile. The core question is straightforward: the EU model had a `header.authorship` concept that needed a FHIR mapping target, and `Observation.performer` is the pragmatic choice.

Giorgio's comment is significant — it indicates the EU model has already evolved (v0.3.0) to remove `header.authorship`, which substantially reduces the urgency of this question. The `author[x]` element remains unmapped but this is intentional since `performer` already captures the relevant semantics. Simone Heckmann's agreement reinforces the consensus that no additional extension or mapping complexity is warranted.

The resolution path is clear: the model change in v0.3.0 has effectively resolved the underlying concern by removing the problematic mapping source.

## Proposed Dispositions

### Disposition A: Considered – Question Answered
**Action:** Resolve as "Considered – Question answered" with no specification change.
**Rationale:** The question asks whether `Observation.performer` is equivalent enough to `header.authorship`. Giorgio's comment confirms that `header.authorship` has been removed from the model in v0.3.0, and `author[x]` was intentionally left unmapped because `performer` already covers the concept. The question is therefore answered by the model evolution itself — the mapping gap no longer exists. No specification change is needed since the underlying model has already been updated to align with this position.

### Disposition B: Persuasive – Add Mapping Documentation
**Action:** Resolve as "Persuasive" and add explicit mapping documentation clarifying that `Observation.performer` fulfils the authorship requirement and that `header.authorship` (removed in v0.3.0) maps to `Observation.performer`.
**Rationale:** While the model change resolves the structural issue, implementers consulting the mapping table may still wonder about the relationship between authorship and performer. Adding an explicit note in the mapping documentation would prevent future confusion and acknowledge the ballot comment constructively. This aligns with the pattern in FHIR-55428 (also resolved as Persuasive) where mapping table completeness was addressed.

### Disposition C: Not Persuasive
**Action:** Resolve as "Not Persuasive" — the mapping table is implementation guidance that does not need to document removed elements.
**Rationale:** Since `header.authorship` was removed from the model, there is no requirement to document its former mapping. The current specification is correct as-is. Documenting removed model elements could create confusion about whether they are still relevant.

## Recommendation

**Disposition A: Considered – Question Answered** is recommended.

The ticket is a Question, not a Change Request, and the core question has been definitively answered by the model's own evolution: `header.authorship` was removed in v0.3.0, and `performer` already exists in the model. Both commenters agree the matter is settled. Resolving as "question answered" accurately reflects the situation without requiring specification changes to document a concept that no longer exists in the model. This is the cleanest resolution consistent with the ticket type and the consensus in the comments.
