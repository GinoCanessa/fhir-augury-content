# FHIR-39244: Add example/guidance on associating device settings with clinical observations

## Summary

**Ticket:** [FHIR-39244](https://jira.hl7.org/browse/FHIR-39244)
**Status:** Triaged | **Priority:** Highest | **Type:** Change Request
**Specification:** FHIR Core (FHIR)
**Work Group:** Orders & Observations
**Reporter:** Ron G. Parker | **Assignee:** Unassigned
**Labels:** OO, component-qualifier
**Created:** 2022-10-14 | **Updated:** 2026-02-02

This ticket requests guidance and/or examples on how to associate clinical observations generated by devices with the device settings in effect at the time the observation was made. It is essentially a duplicate of FHIR-38917, filed by Ron G. Parker as "Comment 52" of the earlier ticket. The ticket title itself references FHIR-38917.

The core problem: there is no clear, agreed-upon FHIR pattern for linking a clinical Observation (e.g., a blood pressure reading) to the device settings (e.g., PEEP setting, target ranges) that were active when that observation was taken. Two main approaches have been discussed:

1. **Explicit association** — The clinical Observation uses `hasMember` to reference settings Observations (either directly or via a "settings set" grouping Observation). This provides unambiguous linkage but adds overhead to every clinical Observation.
2. **Temporal inference** — Settings Observations are recorded when they change (or at defined intervals), and consumers determine which settings applied to a clinical Observation by finding the most recent settings prior to the observation's timestamp. This is simpler but makes it harder to detect missing/deleted settings.

There is also confusion about the boundary between `Device.property` (static properties), `DeviceMetric` (setting/reading capabilities), and settings Observations (current values at a point in time).

## Analysis

This ticket has a long history spanning R5 and R6 cycles. It was resolved twice as "Considered for Future Use" and reopened both times — most recently on 2026-01-30 with strong support (11-0-1). This pattern indicates the workgroup considers the need real but has struggled to reach consensus on the approach.

The core challenge is architectural: FHIR currently has no first-class mechanism to link a clinical Observation to the device settings in effect when it was generated. The existing mechanisms (hasMember, derivedFrom, component) are semantic mismatches, and extensions or new resources add complexity.

**Key considerations:**
- FHIR-40288 (Device module landing page) is already **Applied/Persuasive**, creating a natural home for this guidance
- FHIR-39322 (Device.property overlap) is **Resolved - change required**, so the property/metric/observation boundaries are being clarified
- FHIR-40335 (Observation.subject/.device/.focus for devices) is already **Applied**, which partially addresses how devices are referenced from Observations
- Hans Buitendijk's July 2025 comment indicates intent to discuss this jointly with related JIRAs, suggesting a coordinated resolution is preferred
- The `observation-example-devicemetricfocus` example exists but is "not yet agreed to"

## Proposed Dispositions

### Disposition A: Persuasive — Add guidance to Device module page
Accept the request. Add a section to the now-applied Device module landing page (FHIR-40288) that documents how to associate device settings with clinical observations. Include:
- The recommended pattern using `Observation.hasMember` to a "settings set" grouping Observation
- The alternative temporal-inference approach with trade-offs
- Clarification of Device.property (static) vs. DeviceMetric (capability) vs. Observation (point-in-time value)
- One or more examples showing a clinical Observation linked to settings Observations via hasMember

**Pros:** Addresses the longstanding need; leverages the already-applied Device module page; provides concrete guidance the community has repeatedly requested.
**Cons:** Requires choosing a recommended pattern when community consensus has been fragile; may need coordination with PoCD/PHD IG teams.

### Disposition B: Considered for Future Use — Defer to post-R6-ballot with ballot feedback insertion
Defer again, inserting this as ballot feedback in the first R6 round for ballot reconciliation. This matches the pattern used for other device settings tickets in the October 2025 resolution.

**Pros:** Allows more time for the coordinated discussion Hans Buitendijk described; avoids premature commitment to a pattern.
**Cons:** This ticket has been deferred twice already and reopened both times (including most recently with 11-0-1 support); continued deferral risks losing community trust and leaving implementers without guidance.

### Disposition C: Persuasive with Modification — Provide guidance and example, but do not mandate a single pattern
Accept with modification: add guidance documenting both the explicit-association (hasMember) and temporal-inference approaches as valid patterns, with trade-off analysis, rather than mandating a single approach. Include an example using the hasMember pattern as the primary illustration. Coordinate with FHIR-38917 to resolve both tickets together.

**Pros:** Provides the requested guidance without forcing a single pattern; acknowledges legitimate use cases for both approaches; resolves the long-running ticket.
**Cons:** Not providing a single recommended pattern may perpetuate interoperability gaps between implementations using different approaches.

## Recommendation

**Disposition C: Persuasive with Modification** — This ticket has been open since 2022 and deferred twice. The January 2026 re-opening with 11-0-1 support signals clear workgroup intent to address it. Given that FHIR-40288 (Device module page) is already applied and FHIR-39322/FHIR-40335 are clarifying related boundaries, the foundation exists to provide meaningful guidance. Documenting both patterns with trade-offs (rather than mandating one) respects the legitimate diversity of use cases while giving implementers the clarity they have repeatedly requested. The resolution should be coordinated with FHIR-38917 since the tickets are effectively duplicates. The `observation-example-devicemetricfocus` example referenced by Hans Buitendijk should be finalized as part of this work.
