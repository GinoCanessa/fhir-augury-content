## Problem Statement

The submitter observes that both `DiagnosticReport.code` (1..1) and `ServiceRequest.code` (0..1) constrain each resource to a single code value. This forces the creation of a separate resource instance for each orderable panel code. The submitter proposes extending `DiagnosticReport.basedOn` to accept references to DiagnosticReport (i.e., allowing DiagnosticReports to be nested), similar to how `ServiceRequest` already allows self-referencing.

## Proposed Dispositions

### Disposition A: Not Persuasive with Modification (Clarify Guidance)

**Resolution:** Do not change the cardinality of `DiagnosticReport.code` or `ServiceRequest.code`. Instead, add explanatory text to the specification clarifying:
- Multiple panel orders should be grouped using a shared `ServiceRequest.requisition`.
- A single DiagnosticReport can aggregate multiple ServiceRequests using a higher-level grouping code in `DiagnosticReport.code`.
- The `.code` values in ServiceRequest and DiagnosticReport do not need to match.
- Defer internal DiagnosticReport grouping/nesting to FHIR-48255.

**Rationale:** This aligns with rhausam's suggested resolution and preserves the current cardinality model while addressing the submitter's practical concern through better documentation. It avoids a breaking change to `DiagnosticReport.code` cardinality and keeps the nesting discussion consolidated in FHIR-48255.

### Disposition B: Persuasive with Modification (Extend basedOn)

**Resolution:** Allow `DiagnosticReport.basedOn` to reference DiagnosticReport in addition to its current reference targets, enabling hierarchical nesting of diagnostic reports without changing `DiagnosticReport.code` cardinality.

**Rationale:** Directly addresses the submitter's proposal. However, this is a structural change to the DiagnosticReport resource that should be coordinated with FHIR-48255 and the broader international consensus effort currently underway. Adopting this independently of that process risks conflicting with the eventual consensus.

### Disposition C: Deferred (Pending FHIR-48255 Resolution)

**Resolution:** Defer disposition of FHIR-48258 until FHIR-48255 (nested DiagnosticReports) reaches resolution, as the two tickets are tightly coupled and the final approach to `DiagnosticReport.code` cardinality depends on the nesting model chosen.

**Rationale:** The comment history explicitly states that the cardinality decision depends on FHIR-48255. The active international discussion group (with meetings through 2026) is still working toward consensus. Deciding this ticket prematurely could conflict with that process.

## Recommendation

**Disposition A: Not Persuasive with Modification** is recommended. This aligns with the workgroup commenter's (rhausam) suggested resolution, provides immediate value through improved guidance text, and avoids premature structural changes while the broader nesting discussion (FHIR-48255) is actively being resolved by the international working group. The explanatory text about `ServiceRequest.requisition`-based grouping addresses the submitter's practical concern without requiring cardinality changes. The final cardinality decision can be revisited if FHIR-48255 ultimately requires it.
