## Description

The submitter raises two questions regarding the Laboratory Report IG's use of Obligations:

1. **Must Support prerequisite for Obligations:** In R5, Obligations can be defined when an element is marked as Must Support. The submitter asks whether elements should be explicitly marked as Must Support before applying Obligations, or whether the presence of an Obligation implies Must Support.

2. **R4/R5 pre-adoption guidance:** Since the Laboratory Report IG is R4-based but pre-adopts the R5 Obligation mechanism, the submitter asks how this works in practice. At minimum, the IG should reference the R5 Obligation section, and potentially provide additional guidance given the R4 context.

## Analysis

This ticket addresses a real gap in the Laboratory Report IG: the relationship between Must Support flags and the R5 Obligation mechanism when pre-adopted in an R4 context. The core tension is:

- **R5 defines Obligations as refinements of Must Support**, providing more granular actor-specific requirements (e.g., SHALL:populate, SHOULD:handle). In R5, Obligations are applied to Must Support elements.
- **The Laboratory Report IG is R4-based** and pre-adopts the R5 Obligation extension, but it is unclear whether elements carrying obligations are also (or should also be) marked Must Support, or whether the obligation alone conveys the Must Support semantics.
- **Implementers need clarity** on whether they should look for Must Support flags, Obligation extensions, or both when determining conformance requirements.

The Zulip discussions show that obligations in R4-based IGs are already causing practical tooling and comprehension issues, reinforcing the need for better documentation of the pre-adoption approach.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Add Must Support flags and R5 cross-reference documentation
Mark all elements that carry Obligations as Must Support, and add a section to the IG explaining the relationship between Must Support and Obligations in the R4 pre-adoption context. Include a reference to the [R5 Obligation documentation](https://hl7.org/fhir/R5/obligations.html) and explain that Obligations refine the meaning of Must Support for specific actors. This provides the clearest guidance for implementers.

### Disposition B: Persuasive — Add documentation only
Add a section to the IG that explains how the R5 Obligation mechanism is pre-adopted in this R4-based IG, referencing the R5 Obligation section. Document that the presence of an Obligation on an element implies Must Support semantics for the actors specified in the obligation, without necessarily adding the Must Support flag to every such element. This avoids structural profile changes while addressing the documentation gap.

### Disposition C: Not Persuasive — Current approach is sufficient
The R5 Obligation extension is a well-defined mechanism and its semantics are self-documenting. Implementers familiar with R5 concepts will understand that obligations define actor-specific requirements. Adding Must Support flags would be redundant since obligations already convey stronger, more specific requirements. No changes needed.

## Recommendation

**Disposition A: Persuasive with Modification** — This is the strongest approach because it addresses both of the submitter's concerns directly. Explicitly marking elements as Must Support before applying Obligations aligns with R5 semantics (where obligations refine Must Support), removes ambiguity for implementers working in the R4 context, and provides the necessary cross-references to R5 documentation. The Zulip discussions confirm that the relationship between obligations and existing R4 mechanisms is causing real confusion, making explicit documentation and consistent flagging the most implementer-friendly resolution.
