## Summary

The `Specimen.collection.device[x]` element describes the device used to perform specimen collection (e.g., needle, swab, scalpel, biopsy forceps), but its example binding currently points to the **Specimen Container** value set, which contains codes for specimen containers (e.g., tube, vial, cup). This is a semantic mismatch — collection devices and containers are fundamentally different concepts. The ticket requests that the binding be changed to a collection-device-appropriate value set. The resolution proposes creating a new "collection-device" value set with specific SNOMED CT codes for collection instruments, while also combining existing container-related value sets for `Specimen.container.device`.

## Details

**Description:**

**Element:** `collection.device[x]` — "Device used to perform collection"

**Binding:** Specimen Container (Example)

**Issue:** The element describes a **collection device** (e.g., needle, swab, scalpel, biopsy forceps), but the value set is bound to **Specimen Container** (e.g., tube, vial, cup). These are semantically different, even if it is an example binding.

- **Collection device:** The instrument used to obtain the specimen
- **Container:** The vessel that holds the specimen after collection

**Recommendation:** Rename the binding to a collection-device-appropriate value set.

**Resolution (Persuasive with Modification):**

The resolution involves two changes:

1. **For `Specimen.container.device`:** Combine the values from the existing [specimen-container-type](https://hl7.org/fhir/6.0.0-ballot4/valueset-specimen-container-type.html) and [containerdevice-code](https://hl7.org/fhir/6.0.0-ballot4/valueset-containerdevice-code.html) value sets into a new value set called "specimen-container-type" (removing the filter "where concept is-a 434711009 (Specimen container (physical object))"), with example binding.

2. **For `Specimen.collection.device`:** Create a new value set called "collection-device" with the following SNOMED CT codes (example binding):
   - 1003474008 — Oral specimen collection swab (physical object)
   - 20568009 — Urinary catheter, device (physical object)
   - 706681000 — Aspiration needle (physical object)
   - 700956008 — Blood collection needle, basic (physical object)
   - 22566001 — Cytology brush, device (physical object)

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Create the new "collection-device" value set containing the five specified SNOMED CT codes (oral specimen collection swab, urinary catheter, aspiration needle, blood collection needle, cytology brush) and bind it to `Specimen.collection.device[x]` with an example binding. Additionally, combine the existing specimen-container-type and containerdevice-code value sets into a unified "specimen-container-type" value set for `Specimen.container.device`, removing the "is-a 434711009" filter, retaining the example binding.
#### Justification
The resolution as voted is well-reasoned: it fixes the semantic mismatch by giving each element a binding that accurately reflects its purpose. Collection devices and containers are clearly distinct concepts in laboratory practice, and the current binding conflates them. The proposed SNOMED CT codes are representative of common collection instruments. Both bindings remain at example strength, allowing implementers flexibility to extend with additional codes as needed.

---

### Disposition B: Alternative Approach
#### Proposal
Create the "collection-device" value set for `Specimen.collection.device[x]` as proposed, but expand the initial seed codes to include additional common collection instruments such as biopsy forceps (SNOMED CT code for biopsy punch or forceps), venipuncture devices, and lancets. Keep the existing container value set arrangement for `Specimen.container.device` unchanged rather than merging two value sets, to minimize disruption.
#### Justification
While the core issue (binding mismatch) is valid and must be fixed, the proposed collection-device value set contains only five codes, which may not adequately represent the breadth of collection devices encountered in practice. Expanding the initial set provides better guidance to implementers. Leaving the container value sets separate avoids introducing changes to `Specimen.container.device` that were not part of the original request and could have unintended downstream effects on implementations already using those value sets.

---

### Disposition C: Decline
#### Proposal
Decline the change request and retain the current Specimen Container example binding on `Specimen.collection.device[x]`.
#### Justification
Since this is an example binding, implementations are not constrained to the bound value set and can already use any appropriate codes for collection devices. The mismatch, while semantically imprecise, does not create interoperability barriers or validation failures. Changing the binding and creating new value sets introduces churn in the specification. However, this position is weak because example bindings serve as guidance, and misleading guidance is arguably worse than no binding at all.

---

### Recommendation
**Recommended disposition:** A (Accept As Requested)

The resolution as voted directly and cleanly addresses the reported issue. The semantic mismatch between collection devices and specimen containers is real and potentially confusing to implementers who look to example bindings for guidance on appropriate coding. The proposed fix correctly separates the two concepts by creating a dedicated "collection-device" value set with representative SNOMED CT codes while also rationalizing the container value sets. Both bindings remain at example strength, preserving implementer flexibility. The changes are narrowly scoped, well-defined, and align with the intent of the original change request. There is no compelling reason to deviate from the workgroup's voted resolution.
