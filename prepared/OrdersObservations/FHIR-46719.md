## Description

The submitter asks how complex specimens with separate reports should be handled in the UV Laboratory Report IG. In HL7 V2.5.1, a gross section and pathologist report are transmitted as two separate OBR/ORC groups within a single message. In the current UV Lab IG, the Composition profile (`Composition-uv-lab`) constrains DiagnosticReport to a cardinality of `1..1`, which would require two separate FHIR Documents to convey both reports.

**Proposed Change:** Make DiagnosticReport `1..*` in the [Composition-uv-lab](https://build.fhir.org/ig/HL7/uv-lab-rep-ig/StructureDefinition-Composition-uv-lab.html) profile so that both DiagnosticReports (gross and pathologist report) can be included in the same Document Bundle.

The submitter notes that an alternative would be to change the cardinality at the [Bundle](https://build.fhir.org/ig/HL7/uv-lab-rep-ig/StructureDefinition-Bundle-uv-lab.html) level, but argues against it because the gross and pathologist reports SHOULD be in the same message for clinical context.

## Analysis

This ticket addresses a real structural limitation in the UV Lab Report IG. The current `1..1` cardinality of DiagnosticReport in the Composition profile forces implementers to create separate FHIR Documents for what is logically a single clinical report with multiple sections (e.g., gross examination + pathologist interpretation). This conflicts with V2.5.1 migration patterns where multiple OBR/ORC groups are naturally grouped in one message.

The ticket is part of a broader cluster of `results-grouping` issues (FHIR-48255, FHIR-48258, FHIR-47571, FHIR-46827, FHIR-47201) that collectively point to unresolved design questions about how to structure multi-panel, multi-report laboratory results in FHIR. The OO WG is actively discussing these issues in dedicated calls.

The US vs. EU philosophical divide (one DiagnosticReport per ServiceRequest vs. one holistic DiagnosticReport) adds complexity. The proposed change to `1..*` would better support the US pattern within a Document Bundle, but the workgroup needs to ensure this doesn't undermine the EU design choice or create interoperability friction.

## Proposed Dispositions

### Disposition A: Persuasive — Change DiagnosticReport cardinality to 1..* in Composition-uv-lab

**Action:** Update the Composition-uv-lab profile to allow `1..*` DiagnosticReport references.

**Rationale:** The use case is valid and well-documented. Complex specimens with multiple report sections (gross, pathologist interpretation) are common in pathology and require multiple DiagnosticReports to be bundled in a single clinical document. Changing from `1..1` to `1..*` is a relaxation of constraint (non-breaking), preserves backward compatibility, and directly supports V2.5.1 migration patterns. This aligns with the US approach where DiagnosticReport maps to individual ServiceRequests, while still allowing the EU pattern (a single DiagnosticReport remains valid under `1..*`).

**Risks:** May complicate consumer implementations that assume exactly one DiagnosticReport per Composition. Needs coordination with FHIR-48255 (nesting) and FHIR-48258 (basedOn references) to ensure a coherent overall design.

### Disposition B: Persuasive with Modification — Defer to the broader results-grouping design resolution

**Action:** Acknowledge the use case as valid but defer the specific cardinality change until the OO WG completes its ongoing "Grouping options for Observation/DiagnosticReport" design work. The cardinality decision should be part of the holistic resolution of FHIR-48255, FHIR-48258, and this ticket together.

**Rationale:** There are multiple competing approaches to solve this problem: relaxing cardinality at the Composition level (this ticket), nesting DiagnosticReports (FHIR-48255), or extending basedOn references (FHIR-48258). Making a piecemeal change risks creating an inconsistent design. The WG is actively holding dedicated calls on this topic and a coordinated solution would better serve implementers.

**Risks:** Delays resolution for implementers who need this capability now. The broader design discussion may take considerable time.

### Disposition C: Not Persuasive — Maintain 1..1 constraint and recommend alternative patterns

**Action:** Keep the current `1..1` DiagnosticReport cardinality in Composition-uv-lab. Recommend that implementers use a collection Bundle (rather than a document Bundle) to group multiple DiagnosticReports, or use DiagnosticReport nesting (per FHIR-48255) once available.

**Rationale:** The `1..1` constraint is an intentional design choice that maintains the principle of one Composition = one logical report. The FHIR specification already supports collection Bundles for grouping related resources. Relaxing the Composition constraint could blur the boundary between clinical documents and resource collections.

**Risks:** Does not address the submitter's specific use case of keeping related reports in a single FHIR Document with clinical context. May not align well with V2.5.1 migration patterns. Forces implementers into workarounds.

## Recommendation

**Disposition B: Persuasive with Modification** — The use case is clearly valid, but the change should not be made in isolation. FHIR-46719 is one of at least five interrelated `results-grouping` tickets, and the OO WG has active, ongoing design discussions about exactly this problem (including dedicated calls). The workgroup should acknowledge this use case, group it with FHIR-48255 and FHIR-48258 for coordinated resolution, and determine whether the answer is cardinality relaxation, DiagnosticReport nesting, extended basedOn references, or a combination. This avoids a piecemeal fix that might conflict with the eventual holistic design.
