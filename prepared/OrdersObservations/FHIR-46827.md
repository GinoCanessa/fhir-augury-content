## Description

The introduction and diagram in the Laboratory Report IG are not clearly conveying that one can share either a full FHIR Document **or** a DiagnosticReport with all relevant Observations only (without the Document Bundle/Composition wrapper).

The submitter is requesting that the specification more explicitly present these two approaches as distinct, valid options for exchanging laboratory report data.

## Analysis

This ticket identifies a documentation clarity gap in the Laboratory Report IG. The introduction and associated diagrams do not clearly communicate that there are **two valid exchange patterns**:

1. **FHIR Document approach:** A Bundle of type `document` containing a Composition that references a DiagnosticReport and its Observations — a formal, legally signed, self-contained package.
2. **DiagnosticReport-only approach:** A DiagnosticReport resource (possibly within a collection or searchset Bundle) referencing its Observations directly — a structured data exchange without the Composition/Document wrapper.

The Zulip discussions confirm real-world implementer confusion on this point. The EU Lab IG mandated the Document approach, while the UV Lab Report IG and US implementations often use the DiagnosticReport-only approach. The lack of clarity in the introduction leads implementers to assume one approach is required when both are valid.

This is consistent with related work in FHIR-40540 (diagrams and narrative updates for the diagnostics module), FHIR-54562 (grouping philosophy), and FHIR-19249 (DiagnosticReport boundary clarification).

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Update the Laboratory Report IG introduction text and diagram to explicitly present both exchange patterns (FHIR Document Bundle and DiagnosticReport-only) as distinct, supported options. Add a brief comparison table or decision guidance noting when each approach is appropriate (e.g., Document for formal/legal exchange, DiagnosticReport-only for RESTful API queries). Coordinate with the UV Lab Report IG's existing three-option design choice page as a model.

**Rationale:** The request is valid — the current introduction is ambiguous. The UV Lab Report IG already documents multiple options (see its design-choice page), providing a proven pattern to follow. This improves implementer clarity without changing the underlying resource model.

### Disposition B: Persuasive
**Action:** Accept as-is and update the introduction and diagram to clarify both approaches. Leave the specific editorial changes to the IG editors without prescribing the exact wording or diagram changes.

**Rationale:** The submitter's observation is correct and the fix is editorial. Trusting the editors to determine the best presentation gives flexibility while acknowledging the issue.

### Disposition C: Not Persuasive
**Action:** No change. The specification already supports both patterns; the introduction is sufficient as-is.

**Rationale:** The two approaches are inherent in the FHIR specification (any DiagnosticReport can be packaged as a Document or exchanged standalone). The Laboratory Report IG does not prohibit either. Additional clarification may over-constrain or create maintenance burden.

## Recommendation

**Disposition A: Persuasive with Modification** — The real-world implementer confusion documented in Zulip threads (particularly the EU Lab mandatory composition discussion) validates this request. The UV Lab Report IG's design-choice page already provides a model for clearly presenting the options. Updating the Laboratory Report IG introduction and diagram to explicitly describe both the FHIR Document and DiagnosticReport-only approaches, with brief guidance on when each is appropriate, would materially reduce implementer confusion and align with existing best practices in the UV Lab Report IG.
