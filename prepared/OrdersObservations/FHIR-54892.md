# FHIR-54892: Avoid background workflow status in the status value of data resources

## Summary

**Ticket:** [FHIR-54892](https://jira.hl7.org/browse/FHIR-54892)
**Status:** Triaged | **Type:** Change Request | **Priority:** Highest
**Resolution:** Unresolved | **Assignee:** Unassigned
**Reporter:** Kevin O'Donnell | **Work Group:** Orders & Observations
**Specification:** FHIR Core (FHIR)
**Created:** 2026-01-19 | **Updated:** 2026-01-20

The submitter requests removal of the `specimen-in-process` code from the `Observation.status` value set. The argument is that most `Observation.status` values describe the state of the Observation resource itself, but `specimen-in-process` instead describes an intermediate workflow state of a laboratory procedure. The submitter contends that `registered` ("The existence of the observation is registered, but there is no result yet available") already covers this situation, and that workflow/procedure tracking should be handled by Procedure or workflow-oriented resources rather than embedded in data resource status codes.

## Proposed Dispositions

### Disposition A: Persuasive — Remove `specimen-in-process`
**Action:** Remove `specimen-in-process` from the `ObservationStatus` code system and value set. Update the definition of `registered` to explicitly clarify it encompasses the "specimen received / results pending" workflow state (e.g., v2 OBX-11 = "I").

**Rationale:** The submitter makes a valid architectural argument: `Observation.status` should reflect the state of the Observation data resource, not intermediate workflow states of specimen processing. The `registered` code already semantically covers the "observation exists but no result yet" scenario. Workflow tracking for specimen processing belongs in Procedure, Task, or ServiceRequest resources. Removing `specimen-in-process` keeps the status value set focused and avoids setting a precedent for adding domain-specific workflow states to data resource status codes. Since `specimen-in-process` is new in R6 and not yet in production use, the migration cost is minimal.

### Disposition B: Not Persuasive — Retain `specimen-in-process`
**Action:** No change. Retain `specimen-in-process` in `Observation.status`.

**Rationale:** The `specimen-in-process` code was deliberately added through FHIR-33041 after extensive discussion by the O&O work group to address a real v2-to-FHIR mapping gap. Laboratory systems need to communicate that a specimen has been received and is being processed, which is a distinct state from merely `registered` (where the specimen may not have been collected yet). Removing it so soon after its intentional addition would undermine the prior ballot decision, disrupt v2-to-FHIR mapping guidance, and leave laboratory implementers without a precise status for this common workflow state. The argument that workflow belongs only in Procedure/Task resources does not reflect how laboratory observations are actually tracked in many production systems.

### Disposition C: Persuasive with Modification — Deprecate and clarify
**Action:** Deprecate `specimen-in-process` (do not remove) and enhance the definition of `registered` to clearly encompass the "specimen received, results pending" state. Add implementation guidance noting that detailed specimen/procedure workflow tracking should use Procedure, Task, or ServiceRequest resources rather than `Observation.status`.

**Rationale:** This is a compromise that acknowledges the architectural concern without breaking existing implementations or reversing the prior ballot decision. Deprecation signals the intent to move away from workflow-specific status codes in data resources while giving implementers time to transition. Enhancing the `registered` definition addresses the original FHIR-33041 concern about mapping clarity. Adding guidance about proper workflow tracking via Procedure/Task resources addresses the submitter's core point without an abrupt removal.

## Recommendation

**Disposition C: Persuasive with Modification** is recommended. It respects the submitter's valid architectural principle that data resource status codes should not encode background workflow states, while also honoring the prior deliberate decision in FHIR-33041. Since `specimen-in-process` was just added in R6 after a legitimate ballot process, outright removal (Disposition A) risks undermining work group credibility and disrupting v2 mapping guidance. However, the submitter's concern is well-founded — the code *is* architecturally out of place compared to other `Observation.status` values. Deprecation with enhanced `registered` semantics and explicit workflow guidance gives the community a clear path forward, acknowledges the design issue, and avoids breaking changes.
