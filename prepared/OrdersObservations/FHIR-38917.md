## Description

The ticket requests examples and/or guidance on how to associate clinical observations generated by devices with the device settings in effect at the time those observations were made. This is important for providing context and historical evidence for clinical observations.

Two approaches are identified:

1. **Explicit hasMember linking:** The clinical observation uses `hasMember` relationships to settings observations (directly or via a "settings set" observation), unambiguously tying the clinical observation to the settings in effect.

2. **Temporal inference:** Settings observations are recorded when they change or at defined intervals (daily, at procedure start, etc.), and clinical observations are recorded independently. The applicable settings are inferred by finding the most recent settings observations prior to the clinical observation. This is simpler but makes it harder to detect missing/deleted settings.

Additionally, there is confusion about the difference between device settings (recorded as Observations linked to DeviceMetrics linked to Devices) and device properties, and how to record device settings in general.

The request is for an example or guidance that ties these concepts together, ideally starting with a clinical Observation linking to settings Observations.

## Analysis

### Key Considerations

1. **Persistent need:** This ticket has been opened since October 2022 and has been deferred twice (for R5 and then R6), only to be reopened each time by strong votes (10-0-0 and 11-0-1). The workgroup clearly sees this as an important gap.

2. **Device Module page now exists:** FHIR-40288 (which was originally proposed as the vehicle for addressing this issue) has been Applied. The lconstab comment from July 2025 asks whether the updated module page already addresses this. This is a critical question for disposition — if the module page adequately covers settings-to-observation association, this ticket may already be satisfied.

3. **Related clarifications resolved:** Several related tickets have been resolved:
   - FHIR-39322 (Device.property overlap) resolved as Persuasive with Modification
   - FHIR-40335 (Observation.subject/.device/.focus for devices) Applied
   - These resolutions may have addressed parts of the confusion this ticket describes.

4. **Duplicate ticket:** FHIR-39244 is an identical duplicate at Highest priority and is also Triaged. These should be resolved together.

5. **Scope:** The request spans examples (concrete resource instances showing the pattern) and normative guidance (which approach to use). These are distinct deliverables.

## Proposed Dispositions

### Disposition A: Persuasive — Add guidance and examples to Device Module page
**Action:** Add a dedicated section to the Device module page (created via FHIR-40288) that:
- Explains the two approaches to associating device settings with clinical observations (hasMember linking vs. temporal inference)
- Provides a recommendation on when to use each approach
- Clarifies the distinction between Device.property (static characteristics) and DeviceMetric + Observation (dynamic settings)
- Includes a worked example with a clinical Observation linking to settings Observations via hasMember

**Rationale:** The ticket has been reopened twice by strong votes, indicating persistent unmet need. The Device module page is the natural home for this guidance. The related ticket resolutions (FHIR-39322, FHIR-40335) have clarified some boundaries but have not provided the specific settings-to-observation association guidance requested here.

### Disposition B: Considered - No Change (already addressed)
**Action:** Close as already addressed by the combination of FHIR-40288 (Device module page), FHIR-39322 (Device.property clarification), and FHIR-40335 (Observation reference clarification).

**Rationale:** If the current Device module page and the resolved related tickets collectively provide sufficient guidance for implementers, no additional change is needed. The lconstab comment suggests this should be verified.

### Disposition C: Considered for Future Use — Defer pending Device module review
**Action:** Defer until the workgroup reviews the updated Device module page to determine whether the existing content sufficiently addresses this issue, and if not, what specific gaps remain.

**Rationale:** The module page exists but it's unclear whether it covers the specific settings-to-observation association guidance. A targeted review would prevent duplicating existing content while ensuring gaps are identified.

## Recommendation

**Disposition A: Persuasive** — This ticket has been deferred twice and reopened both times with strong consensus votes (most recently 11-0-1 in Feb 2026). The community clearly wants this addressed. While the Device module page (FHIR-40288) and related ticket resolutions have improved the landscape, the specific guidance on how to associate device settings with clinical observations — including worked examples showing the hasMember pattern and the temporal inference pattern — remains a gap. The duplicate FHIR-39244 should be resolved as a duplicate of this ticket. Adding a focused section to the Device module page with concrete examples and a recommended approach would satisfy a long-standing implementer need with relatively contained effort.
