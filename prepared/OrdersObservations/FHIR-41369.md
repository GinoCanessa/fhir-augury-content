## Description

The original issue reported a broken link in the Order Catalog IG. In section 1.7 of the IG home page, the text describing how the catalog constrains the Composition resource with the "Catalog profile" linked to `http://hl7.org/fhir/composition-catalog.html`, which was incorrect. The correct URL should have been `http://hl7.org/fhir/catalog.html`.

However, the scope of this ticket expanded significantly. The 'Catalog' profile was renamed to 'ProfileForCatalog' in the R5 base specification, and SDWG planned to remove it from the base Composition resource in R6. It was agreed (in the 2024-10-10 OO Main call) to move the profile into the Order Catalog IG itself, rather than simply fixing the link. This broadened the change from a simple URL fix to migrating the profile and updating all references across multiple IG pages.

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification

**Action:** Accept the proposed resolution. The original link fix has been expanded to migrate the Catalog profile (formerly 'ProfileForCatalog') from the base FHIR spec into the Order Catalog IG, update all links on the home page, labservices, devices, and drugs pages, and align the profile with the ValidityPeriod extension and CatalogHeader conventions.

**Rationale:** The changes have already been pre-applied across three commits. The Zulip thread from February 2025 confirms the profile is showing up correctly in the CI build. The scope expansion from a simple link fix to a profile migration is well-justified: SDWG plans to remove the profile from base in R6, and the Order Catalog IG is the natural home for it. The work group discussed and agreed to this approach in the 2024-10-10 OO Main call.

**Risk:** Low. Changes are pre-applied, confirmed working in CI, and align with the broader R6 direction.

### Disposition B: Accept as Persuasive (Original Scope Only)

**Action:** Accept only the original link fix (correcting the URL from `composition-catalog.html` to `catalog.html`), and handle the profile migration as a separate tracker item.

**Rationale:** The original request was a simple link correction. The profile migration and extension changes are significant scope creep that could be tracked and reviewed independently for clearer traceability.

**Risk:** Medium. Since the changes are already pre-applied as a unit, splitting them retroactively would be disruptive and would not reflect the actual implementation.

### Disposition C: Not Persuasive

**Action:** Reject the change and revert the pre-applied commits.

**Rationale:** The scope grew well beyond the original request without a formal vote on the expanded scope before implementation.

**Risk:** High. The profile has already been migrated and is live in CI. Reverting would break the Order Catalog IG build and conflict with SDWG's R6 plans to remove the profile from base.

---

## Recommendation

**Disposition A: Accept as Persuasive with Modification.**

The expanded scope is well-justified by the R6 deprecation plans, was agreed upon in OO Main and FMG meetings, and has been fully implemented and verified in CI. The three pre-applied commits are clean and internally consistent. Accepting this resolution aligns the Order Catalog IG with the agreed-upon direction for the Catalog profile.
