## Details

This ticket proposes adding new status codes to `ServiceRequest.status` and reason concept codes to `ServiceRequest.reason.concept` to support re-interpretation workflows after a ServiceRequest has been completed (e.g., when final results need re-interpretation due to clinical concerns or demographic/AOE changes).

### Proposed Changes

1. **New `ServiceRequest.status` codes:**
   - `under-revision` — a change to AOEs or demographics yielded the re-interpretation
   - `under-review` — the results do not appear to fit the patient's clinical profile

2. **New `ServiceRequest.reason.concept` codes:**
   - `clinical-reinterpretation` — Re-interpretation (clinically questioned) + text (valid only with `under-review`)
   - `demographic-reinterpretation` — Re-interpretation (demographic change) (valid only with `under-revision`)
   - `ask-at-order-entry-response-reinterpretation` — Re-interpretation (AOE change) (valid only with `under-revision`). Note: for AOEs the `Observation.status` would have changed to corrected/amended.

3. **`ServiceRequest.reason` comments addition:**
   - When a re-interpretation is requested, the observation(s) to be re-interpreted should be included as a reference in `ServiceRequest.reason.reference`.

### Key Concern

The central unresolved tension is whether re-interpretation should be modeled as **new status codes on the existing ServiceRequest** (as originally proposed) or as a **new separate ServiceRequest** (as suggested by IHE LCC LAB-7 and regulatory requirements under CLIA §493.1291). The most recent comments (2025) strongly favor the new-order approach.

## Proposed Dispositions

### Disposition A: Accept with Modification — New Order Model

**Resolution:** Persuasive with Modification

Accept the intent of the ticket but **replace the proposed approach** (adding new status codes to an existing ServiceRequest) with a model based on creating a **new ServiceRequest** for re-interpretation, aligned with the IHE LCC LAB-7 fulfillment order pattern:

- Define guidance that when re-interpretation is needed, a new ServiceRequest is created referencing the original order and/or observations via `ServiceRequest.basedOn` or `ServiceRequest.supportingInfo`.
- Add new `ServiceRequest.reason.concept` codes (e.g., `clinical-reinterpretation`, `demographic-reinterpretation`, `aoe-reinterpretation`) to indicate the purpose of the re-interpretation request.
- Do **not** add `under-revision` / `under-review` status codes to the original ServiceRequest, as this conflicts with CLIA regulations requiring retention of original reports and the principle that completed orders should not be reopened.
- The response to the new order may result in an addendum or amendment to the original report, following established error correction processes.

**Rationale:** This aligns with IHE LCC LAB-7, CLIA §493.1291 regulatory requirements, global practice norms, and the consensus emerging from 2025 OO call discussions.

### Disposition B: Accept Original Proposal

**Resolution:** Persuasive

Accept the original proposal as written — add `under-revision` and `under-review` status codes to `ServiceRequest.status` and new `ServiceRequest.reason.concept` codes for re-interpretation scenarios.

**Rationale:** The original proposal was previously voted Persuasive (7-0-1) before reopening. It provides a simple mechanism to track re-interpretation within the existing ServiceRequest lifecycle. Some implementers may prefer this approach for simpler workflows where regulatory constraints do not apply (e.g., non-US jurisdictions without CLIA requirements).

**Risk:** Conflicts with CLIA regulations and IHE LCC guidance. Reopening a completed ServiceRequest status is not considered best practice per the 2025 discussion consensus.

### Disposition C: Decline — Not Persuasive

**Resolution:** Not Persuasive

Decline the proposal. The re-interpretation workflow is adequately addressed by creating a new ServiceRequest using existing mechanisms (`basedOn`, `replaces`, `supportingInfo`). No new status codes or reason concepts are needed.

**Rationale:** The existing FHIR model already supports creating follow-up orders. Adding new status codes to reopen completed orders introduces complexity and regulatory compliance risks. The IHE LCC approach demonstrates this can be handled without specification changes.

**Risk:** Leaves a gap in standardized reason codes for re-interpretation requests, requiring implementers to define their own codes.

## Recommendation

**Disposition A: Accept with Modification — New Order Model** is recommended.

The 2025 discussion thread shows clear consensus that the original approach of adding status codes to reopen completed ServiceRequests conflicts with CLIA regulations and global laboratory practice. The IHE LCC LAB-7 fulfillment order model provides a well-established pattern for this workflow. Accepting with modification preserves the ticket's valuable identification of the re-interpretation use case while implementing it through the regulatory-compliant new-order approach. Riki Merrick was already tasked (2025-07-11) with proposing the IHE-analogous approach, providing a natural path forward for this modified resolution.
