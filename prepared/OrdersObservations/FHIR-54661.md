## Summary

Lloyd McKenzie requests that `category` be added to **NutritionOrder**, **VisionPrescription**, and **DeviceRequest**, mirroring the existing `category` element on ServiceRequest and MedicationRequest. The `category` element enables grouping orders into dimensions such as acute/chronic, inpatient/outpatient, and others. The submitter prefers a core element but would accept a standard extension.

The subsequent discussion reveals significant workgroup hesitation. The Nutrition project group opposes adding `category` to NutritionOrder, arguing it would duplicate existing structure (oral diet, enteral, supplement types) and oversimplify complex instructions. For DeviceRequest, OO previously decided against adding `.category` in [FHIR-43444](https://jira.hl7.org/browse/FHIR-43444) due to lack of clear use cases, though potential categories (DME, HME, PriorAuth) were identified. The workgroup is struggling to identify suitable example vocabulary for any of these resources, noting that `category` is inconsistently defined and used across existing resources. McKenzie acknowledges that category can be multi-dimensional (multiple CodeableConcepts) and suggests even an experimental value set with scattered examples would suffice. Da Vinci has expressed a specific need for inpatient/outpatient categorization.

## Details

### Description

Just as ServiceRequest and MedicationRequest have `category`, it is also needed for NutritionOrder, VisionPrescription, and DeviceRequest. Categories allow grouping orders into acute/chronic, inpatient/outpatient, and a variety of other things. The submitter prefers a core element but is willing to accept an extension if preferred by the WG.

### Comments

1. **JIRAUSER19407** (2026-01-29): Asked Michelle Ashafa and Paul Knapp to review suitability of adding `category` from the Request pattern to NutritionOrder and VisionPrescription, noting that example codes for a binding would be needed.

2. **hbuitendijk** (2026-01-29): Regarding DeviceRequest — OO Thu Q1 discussed that while potential categories could be envisioned, no clear need has been established. Previously decided in [FHIR-43444](https://jira.hl7.org/browse/FHIR-43444) not to include `.category`. While potential categories (DME, HME, PriorAuth) were identified, they are not fully vetted. Recommends waiting for clear use cases and using an extension in the meantime.

3. **mckenzie** (2026-01-29): Da Vinci wants category for inpatient/outpatient. Reasonable to have as an extension, but a standard extension would be appropriate.

4. **JIRAUSER24962** (2026-02-26): Nutrition project group opposes adding Category to NutritionOrder. Main concerns: nutrition orders are dynamic and often span multiple categories; current structure already identifies types (oral diets, enteral, supplements); a single category would be misleading and oversimplify complex instructions.

5. **esilver** (2026-03-23): While category seems to make sense, OO is having trouble identifying suitable vocabulary. For DeviceRequest specifically, considered: therapeutic/diagnostic/assistive/life-sustaining, consumable/DME, fulfillment source, purchase model, delivery needs, and care setting — but unclear which are appropriate for a `category` vs. belonging on the device or in dedicated elements.

6. **mckenzie** (2026-03-23): Category is not single-dimensional; separate CodeableConcepts can categorize across multiple dimensions. An experimental value set with scattered examples is sufficient for now.

7. **esilver** (2026-03-24): Agrees that if added, cardinality should be 0..*. Still having trouble identifying expected category values. Asks whether the listed categorizations are appropriate and assumes category implies a corresponding search parameter.

## Proposed Dispositions

### Disposition A: Accept — Add `category` as a Core Element to All Three Resources

Add a `category` element with cardinality `0..*` and type `CodeableConcept` to **NutritionOrder**, **VisionPrescription**, and **DeviceRequest**, consistent with the existing `category` on ServiceRequest and MedicationRequest. Define an example-strength value set with initial codes drawn from existing usage patterns (e.g., inpatient/outpatient, acute/chronic). Add a corresponding `category` search parameter to each resource.

**Rationale:** Aligns all request resources with the Request workflow pattern. Da Vinci and other IGs have concrete use cases (inpatient/outpatient). Multi-dimensional categorization via 0..* CodeableConcept addresses the concern about oversimplification. An example-strength binding gives flexibility while providing starting vocabulary.

**Risk:** Vocabulary remains poorly defined; may lead to inconsistent usage across implementations if categories are not clearly scoped. The Nutrition project group and OO have expressed specific concerns about duplication with existing NutritionOrder structure and lack of vetted DeviceRequest categories.

### Disposition B: Accept with Modification — Add `category` to DeviceRequest and VisionPrescription Only; Define a Standard Extension for NutritionOrder

Add `category` (0..*, CodeableConcept, example binding) as a core element to **DeviceRequest** and **VisionPrescription** with corresponding search parameters. For **NutritionOrder**, define a standard extension (`http://hl7.org/fhir/StructureDefinition/nutritionorder-category`) rather than a core element, deferring promotion to a core element until clearer use cases and vocabulary emerge.

**Rationale:** Addresses the strongest stated need (Da Vinci inpatient/outpatient) while respecting the Nutrition project group's objection that NutritionOrder's existing type structure already provides categorization. A standard extension ensures interoperability for IGs that do need it without forcing the element on NutritionOrder's core. DeviceRequest and VisionPrescription have less structural overlap with category semantics, making a core element more appropriate.

**Risk:** Creates inconsistency — some request resources have core `category`, others use extensions. May require later migration if NutritionOrder eventually adopts a core element.

### Disposition C: Decline — Defer Pending Concrete Vocabulary and Use Cases

Do not add `category` to any of the three resources at this time. Recommend that IGs needing categorization define their own extensions. Revisit when concrete, vetted value sets are proposed and clear implementation demand beyond Da Vinci inpatient/outpatient is demonstrated.

**Rationale:** The workgroup has struggled to identify appropriate vocabulary for any of these resources. Prior review in [FHIR-43444](https://jira.hl7.org/browse/FHIR-43444) explicitly decided against `category` on DeviceRequest. The Nutrition project group opposes it. Adding a poorly-defined element risks inconsistent adoption and could mislead implementers. The Request pattern does not mandate `category` — ServiceRequest and MedicationRequest have it because their use cases clearly warranted it.

**Risk:** Leaves Da Vinci and other IGs without a standard mechanism, potentially leading to fragmented extension-based approaches. Diverges further from request pattern alignment goals.

## Recommendation

**Disposition B: Accept with Modification** — Add `category` to DeviceRequest and VisionPrescription; define a standard extension for NutritionOrder.

This approach best balances the competing concerns. There is concrete demand (Da Vinci inpatient/outpatient) and the submitter's core argument about request pattern alignment is sound for DeviceRequest and VisionPrescription, which lack alternative categorization mechanisms. The Nutrition project group's objection is well-founded — NutritionOrder already has structural type distinctions that partially serve the role of `category`, so a standard extension preserves interoperability without overriding the domain experts' judgment. The workgroup's vocabulary concerns are addressed by using an example-strength binding with initial codes (inpatient, outpatient, acute, chronic) that can be expanded over time. This also leaves the door open for promoting the NutritionOrder extension to a core element in a future cycle once use cases and vocabulary mature.
