## Description

Stephen Chu requested the ability to request device settings changes directly of a device. Currently, FHIR supports defining a Task or ServiceRequest to ask a **person** to change settings on a device, but there is no defined mechanism to request those changes **of the device itself**.

The ticket identifies three needs:

1. **Structured representation of requested settings** — a way to express which settings should be changed and to what values.
2. **Conveyance mechanism** — a way to send the request to the device, possibly by profiling or modifying Task, and/or defining a new operation on Device.
3. **Settings lifecycle tracking** — the ability to track requested settings, pending settings, and as-applied settings through their various states.

## Analysis

### Current Gap
FHIR currently models device settings as observations — DeviceMetric defines what a device can report, and Observation captures the actual values. This is inherently a **read/observe** pattern. There is no standardized write path: no operation, resource, or profile that allows a system to request a device to change its own settings. The only workaround is to create a Task or ServiceRequest directed at a human operator to manually adjust the device, which does not support automated or direct device-to-device communication.

### Scope and Complexity
This ticket is significant in scope. It requires:
- New or profiled resources/operations for the request itself
- A lifecycle model for settings states (requested → pending → applied / rejected)
- Coordination with the companion ticket FHIR-48394 (read/query side)
- Consideration of how this interacts with existing Device, DeviceMetric, and Observation patterns

### Workgroup Sentiment
The ticket has strong support. After initially being deferred as "Considered for Future Use" for R6, the workgroup reversed course with an 11-0-1 vote, indicating the community sees this as an important capability to address.

## Proposed Dispositions

### Disposition A: Persuasive — Define new Device operation(s) for settings management
Accept the change request and define one or more new FHIR operations on the Device resource (e.g., `$request-settings-change`) that accepts a structured parameter set describing desired settings values. Introduce a settings request lifecycle model (requested/pending/applied/rejected) using DeviceRequest or a profiled Task to track the state of each settings change request. Coordinate with FHIR-48394 to ensure a consistent approach for both querying and changing device settings.

- **Impact:** High — introduces new operations and potentially new profiles/resources
- **Pros:** Directly addresses the gap; provides a machine-actionable API for automated device configuration; enables IoT and point-of-care device management use cases
- **Cons:** Significant design work required; must consider security implications of remote device configuration; may need coordination across multiple workgroups (Devices, OO, Patient Care)

### Disposition B: Persuasive with Modification — Defer to OO Incubator for design exploration
Accept the request as valid but scope the initial work to the OO Incubator specification. Define requirements and candidate approaches (profiled Task, Device operation, or new resource) as incubator content, allowing iterative development and testing before promotion to FHIR Core. Pair with FHIR-48394 for a unified device settings management design.

- **Impact:** Medium — incubator-scoped; defers core spec changes
- **Pros:** Lower risk; allows experimentation with different approaches; aligns with the ticket's current OO Incubator specification assignment
- **Cons:** Delays availability of a standardized mechanism; implementations may diverge without core guidance

### Disposition C: Considered for Future Use — Revisit during R6 ballot reconciliation
Maintain the original disposition intent. Acknowledge the use case as valid but defer active design work, inserting it as ballot feedback for the first R6 normative round. This preserves the requirement without committing to a specific design approach now.

- **Impact:** Low (deferred)
- **Pros:** Avoids premature design decisions; allows the community to develop implementation experience first
- **Cons:** The workgroup already reversed this disposition with strong support for re-opening; re-deferring contradicts the 11-0-1 vote to revert

---

## Recommendation

**Disposition B: Persuasive with Modification — Defer to OO Incubator for design exploration.**

The workgroup has demonstrated strong support (11-0-1) for actively addressing this gap, ruling out further deferral (Disposition C). However, the design space is broad — multiple approaches exist (operations, profiled Task, new resource) and the work must be coordinated with companion ticket FHIR-48394. Starting in the OO Incubator (where the ticket is already assigned) allows the workgroup to explore and validate approaches before committing to core specification changes, while still signaling active progress on the request.
