## Description

The submitter (LabCorp, via Kathy Walsh) requests that a laboratory report should allow a high-level DiagnosticReport to contain multiple nested DiagnosticReports — one per ordered panel. They are currently sending a Bundle containing multiple DiagnosticReports but not as a FHIR document Bundle. Their key positions are:

1. **DiagnosticReports should be nestable** — a parent DiagnosticReport should be able to reference child DiagnosticReports for each panel.
2. **Observations should NOT be nested** — Observation is referenced by DiagnosticReport to provide atomic results and should not be repurposed as panel groupers.
3. **Observations should not include panel headers** — using Observations with empty `Observation.value` attributes as panel groupers is undesirable.

## Analysis

This ticket is at the center of a longstanding and active debate within the Orders & Observations work group about how to represent laboratory panel grouping in FHIR. The core tension involves several competing approaches:

1. **Nest DiagnosticReports** (this ticket's proposal): Allow a parent DiagnosticReport to contain child DiagnosticReports, each representing a panel. This preserves DiagnosticReport-level metadata (status, performer, timestamps) at the panel level.
2. **Use Observation.hasMember with grouper Observations**: The current common pattern where an Observation with no value acts as a panel header. The submitter strongly opposes this approach.
3. **Use a new/existing grouper resource (Group/List)**: Proposed in FHIR-54562 by Hans Buitendijk as an alternative that avoids overloading either DiagnosticReport or Observation.
4. **Flat panels in a Bundle**: Proposed in FHIR-47052 — keep panels one level deep in DiagnosticReport and use Bundles for multi-panel orders.

Key considerations:
- **Observation is normative** — changes to its nesting behavior are constrained.
- **DiagnosticReport.code is 1..1** — nesting requires addressing how parent vs child codes relate.
- **International scope** — Europe (Germany), Australia, and the US all have active interests with different implementation patterns.
- **Active discussion** — Multiple international meetings have been held and continue through 2026; no consensus reached yet.
- A prior similar request (FHIR-43256 for EU Lab) was resolved as **Not Persuasive**.

## Proposed Dispositions

### Disposition A: Not Persuasive with Modification
**Rationale:** Acknowledge the valid concern about using Observation as a panel grouper, but decline adding nesting to DiagnosticReport itself. Instead, provide enhanced guidance on using Bundles with flat DiagnosticReports (one per panel) sharing a common `ServiceRequest.requisition`, consistent with the approach described in FHIR-47052. Update the Laboratory Report specification to include clear examples of multi-panel reporting using Bundles. Reference the ongoing organizer/grouping work as the venue for a comprehensive solution.

**Pros:** Avoids structural changes to DiagnosticReport; aligns with the flat-panel approach; provides immediate actionable guidance.
**Cons:** Does not address the submitter's core request; the Bundle approach may not meet all use cases (e.g., sub-report-level signatures).

### Disposition B: Persuasive with Modification
**Rationale:** Add a reference element to DiagnosticReport (e.g., `DiagnosticReport.parent` or extend `DiagnosticReport.basedOn`) to allow a hierarchical relationship between DiagnosticReports. This would enable a high-level DiagnosticReport to reference child DiagnosticReports for individual panels, directly addressing the request. Implementation would need to be coordinated with FHIR-48258 and the broader grouping discussion.

**Pros:** Directly addresses the submitter's request; preserves DiagnosticReport-level metadata for panels; supported by real-world implementation patterns (LabCorp, German HiMed).
**Cons:** Adds structural complexity to DiagnosticReport; risks diluting the meaning of DiagnosticReport (per FHIR-54562 concern); a prior similar request was Not Persuasive; may conflict with the ongoing organizer/grouper resource discussion.

### Disposition C: Deferred — Considered for Future Use
**Rationale:** The topic is under active international discussion with multiple meetings scheduled through 2026. No consensus has yet been reached on the best approach for panel grouping. Defer this ticket pending the outcome of the broader "Grouping options for Observation/DiagnosticReport" discussions and coordinate with FHIR-48258, FHIR-54562, FHIR-54897, and FHIR-47052 for a unified resolution.

**Pros:** Allows the community to reach consensus before committing to a structural approach; avoids premature decisions that may conflict with the eventual solution; the grouping discussion is actively progressing.
**Cons:** Further delays resolution for implementers who need guidance now; ticket has been open since September 2024 with highest priority.

## Recommendation

**Disposition C: Deferred — Considered for Future Use**

This ticket sits at the nexus of a larger, actively evolving discussion about diagnostic report and observation grouping that has not yet reached consensus despite extensive international engagement. The work group has scheduled ongoing meetings through 2026 specifically on this topic (the "Grouping options for Observation/DiagnosticReport" Zulip thread and associated calls). Making a structural decision on DiagnosticReport nesting in isolation risks conflicting with the eventual consensus on grouping — especially given that alternatives like a dedicated grouper resource (FHIR-54562) and enhanced Bundle guidance (FHIR-47052) are also being actively considered. Deferring allows the coordinated resolution of this ticket alongside FHIR-48258, FHIR-50472, FHIR-54562, FHIR-54897, and FHIR-47052, ensuring a coherent approach to panel grouping across the specification.
