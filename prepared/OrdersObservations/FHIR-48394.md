## Description

A request originated from Stephen Chu to define a mechanism for requesting a device's current settings on demand. Currently, devices autonomously choose when and which settings to report — there is no standardized way to actively query a device for its configuration. The use case motivates the need to capture device settings at clinically significant moments, such as:

- Recording how a device is configured at the start of a procedure
- Querying an unconscious patient's insulin pump configuration

A suggestion was made that this could be addressed via a **new FHIR operation** on the Device resource, potentially accepting a parameter specifying "kinds" of settings to retrieve (e.g., end-user configurable settings, software/firmware version, calibration data).

## Analysis

### Core Problem
FHIR currently supports *passive* device settings reporting (device pushes settings as Observations when it chooses) but lacks a *pull/query* mechanism. There is no standardized way to ask a device "what are your current settings?" This gap is clinically significant for:
1. **Pre-procedure documentation** — confirming device configuration before surgery
2. **Emergency care** — querying an unconscious patient's implanted device (e.g., insulin pump)
3. **Audit/traceability** — ensuring a complete record of device state at critical moments

### Design Considerations
- The existing device settings model (DeviceMetric → Observation) is well-established and recently clarified (FHIR-39322, FHIR-46963). Any new mechanism should return data in this pattern.
- The companion ticket FHIR-48395 (requesting settings *changes*) should be considered together to ensure a coherent approach.
- A new operation on Device would need to define: input parameters (kinds of settings), output format (Bundle of Observations?), and whether it implies real-time device communication or just retrieval from a system of record.
- FHIR-39688 (Highest priority) provides extensive use case analysis for device function parameters that would benefit from this mechanism.

## Proposed Dispositions

### Disposition A: Persuasive — Define a new `$get-settings` operation on Device

Accept the request and define a new operation `$get-settings` (or similar) on the Device resource. The operation would accept an optional parameter specifying categories of settings (end-user configurable, firmware/software version, calibration, etc.) and return a Bundle of Observation resources following the established DeviceMetric/Observation pattern. This should be developed in coordination with FHIR-48395 to ensure a coherent device interaction model.

**Rationale:** The clinical need is clear and well-supported by the voting history (11-0-1 to reopen). The DeviceMetric/Observation pattern for representing settings is now well-established, and the missing piece is the query/pull mechanism. An operation is the natural FHIR pattern for this kind of on-demand request.

### Disposition B: Persuasive with Modification — Scope as a profile on Task rather than a new operation

Accept the need but address it by defining a Task profile for requesting device settings rather than a new operation. A Task-based approach would support asynchronous workflows where the device response is not immediate, and would integrate with existing Task-based device communication patterns. The Task would reference the target Device and specify the types of settings requested.

**Rationale:** A Task-based approach is more flexible for real-world device communication where responses may not be immediate. However, it is less discoverable and more complex to implement than a dedicated operation.

### Disposition C: Deferred — Coordinate with FHIR-48395 and FHIR-39688 before defining the mechanism

Defer action on this ticket until FHIR-48395 (device settings changes) and FHIR-39688 (device function parameter structure) are further developed. These three tickets form a cluster of related device settings requirements, and designing the query mechanism in isolation risks inconsistency with the broader device settings model being developed.

**Rationale:** The problem is real, but solving it piecemeal may lead to an incoherent device interaction model. A holistic approach considering read, write, and structural representation together would produce a better result.

---

## Recommendation

**Disposition A: Persuasive — Define a new `$get-settings` operation on Device.**

The clinical need is well-documented, the vote to reopen was strongly supported (11-0-1), and the underlying data model (DeviceMetric/Observation) is already clarified. A dedicated operation is the cleanest, most discoverable approach. The operation specification should be coordinated with FHIR-48395 to ensure the read and write mechanisms are consistent, but this coordination does not require deferral — both tickets can progress together within the OO Incubator. The operation's output as a Bundle of Observations aligns with the established pattern confirmed by FHIR-39322 and FHIR-46963.
