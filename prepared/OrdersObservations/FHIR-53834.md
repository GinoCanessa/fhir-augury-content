## Summary

The submitter requests a new dateTime element on the ServiceRequest resource to capture **when the service request was initially created**, as distinct from the existing `authoredOn` (date the request was signed) and `occurrence[x]` (when the service should occur). The use case originates from home health and hospice workflows where the creation-to-signing-to-rendering lifecycle of orders is clinically significant. The submitter argues that the gap between creation and signing can be substantial, and currently there is no way to track how long a request has remained unsigned.

---

## Details

### Description

ServiceRequest currently has two datetime-related elements:

- **`authoredOn`** — captures the date the request was signed.
- **`occurrence[x]`** — captures when the service should occur.

Neither captures the instant the service request was first created. The submitter notes that the time between creation and signing can differ greatly. This delay is important for:

- Identifying requests still under consideration.
- Detecting requests that were never signed.
- Understanding the full creation → sign → render lifecycle.

The submitter's perspective comes from **home health and hospice** settings where therapies are requested/ordered, reviewed by the attending physician or IDG, signed off, and then rendered. This pattern is described as broadly applicable across healthcare.

Alternative approaches considered and rejected by the submitter:

- **`Encounter` element**: Encounters can span multiple days and contain many requests, each with its own creation time.
- **`orderDetail.parameter.valuePeriod`**: Creation is a singular instant, not a period.

## Proposed Dispositions

### Disposition A: Persuasive — Add a `createdOn` element to ServiceRequest

**Rationale:** The request identifies a genuine gap in the ServiceRequest resource's temporal model. The existing `authoredOn` element is defined as "Date request signed" and `occurrence[x]` as "When service should occur" — neither captures the moment the request was first created. Adding a new element (e.g., `createdOn` of type `dateTime` with cardinality `0..1`) would complete the creation → authorization → occurrence lifecycle and support workflows where the delay between creation and signing is clinically significant.

**Action:** Add a new element `ServiceRequest.createdOn` (type `dateTime`, cardinality `0..1`) with a short description such as "When the request was initially created" and appropriate documentation distinguishing it from `authoredOn`.

---

### Disposition B: Persuasive with Modification — Recommend using existing extension or Provenance

**Rationale:** While the use case is valid, adding a new top-level element increases the surface area of the resource. The creation timestamp could be captured through:

1. A **standard extension** (e.g., based on the `Resource.meta.lastUpdated` pattern or a new creation-tracking extension).
2. The **Provenance** resource, which can record the creation event with a precise timestamp and link it to the ServiceRequest via `Provenance.target`.

This approach avoids modifying the core resource while still supporting the workflow.

**Action:** Document guidance in the ServiceRequest resource narrative recommending use of Provenance (activity = "create") to capture creation timestamps, or define a standard extension if a simpler mechanism is preferred.

---

### Disposition C: Not Persuasive — Existing mechanisms are sufficient

**Rationale:** The `authoredOn` element, while described as "Date request signed," is already used by many implementations to represent the date the order was authored/created, not necessarily the date of a formal signature. The `Resource.meta.lastUpdated` and server-managed timestamps may also serve this purpose. Additionally, `Provenance` already provides a mechanism for tracking creation events. Adding a new element could create confusion about the distinction between `createdOn` and `authoredOn`, especially given that FHIR-43010 already discussed `authoredOn` semantics in depth.

**Action:** No change to the resource. Optionally add clarifying language to the `authoredOn` definition or add implementation guidance.

---

## Recommendation

**Disposition B: Persuasive with Modification** is recommended.

The submitter identifies a real workflow gap — the distinction between initial creation and formal authorization of a ServiceRequest is clinically meaningful, particularly in home health and hospice settings. However, adding a new core element (`createdOn`) risks creating semantic overlap with `authoredOn`, which many implementations already treat as the creation date. The Provenance-based approach (or a standard extension) addresses the use case without enlarging the core resource or introducing ambiguity. The work group should provide explicit guidance on how to capture the creation timestamp using Provenance or define a standard extension, ensuring the creation → sign → render lifecycle is representable without modifying the ServiceRequest structure directly.
