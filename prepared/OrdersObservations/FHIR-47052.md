## Summary

This change request proposes that the Laboratory Report IG define explicit guidance on structuring DiagnosticReport and Observation resources when reporting panels and microbiology results. The core proposal is:

1. **Panels can only be one level deep** — the panel code goes in DiagnosticReport and individual test codes go in Observation.
2. **Multi-level panels** must be flattened into individual single-level panels, each represented as a separate DiagnosticReport within a Bundle (with contained or referenced Observations).
3. **If multi-level panel structure is clinically relevant**, DiagnosticReport should be enhanced to support nesting — rather than using Observation.hasMember for grouping.
4. **Observation.hasMember** should be reserved for relationships between actual observations (e.g., organism–sensitivity in microbiology), not for panel grouping. However, sensitivity results should use `.hasMember` rather than `.component`, even though the parent Observation has no value.
5. **Consistent representation**: panels always appear in DiagnosticReport; actual test results always appear in Observation. A single-test order would appear in both.

The proposal aims to eliminate ambiguity about where panel-level versus test-level information resides in the resource model.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Add IG Guidance with Group Resource as Grouper

**Action:** Accept the proposal's core principles (panels in DiagnosticReport, tests in Observation, hasMember for actual observation relationships only) and additionally adopt the Group resource approach suggested by ralf.herzog for panel grouping:

- Add "panel" to `Group.type`
- Add Observation to `Group.member` reference targets
- Add Group to `DiagnosticReport.result` reference targets
- Add guidance in the Laboratory Report IG describing single-level and multi-level panel representation
- Provide worked examples for simple panels, multi-level panels (flattened), and microbiology sensitivity reports

**Rationale:** The Group resource approach resolves the fundamental tension between FHIR-47052 (restrict Observation to actual observations) and the practical need for multi-level grouping (raised by apitkus). Group is semantically appropriate for grouping and avoids overloading either DiagnosticReport or Observation. This aligns with the related tickets FHIR-50472 and FHIR-54562 that also favor a separate grouper mechanism.

**Risk:** Requires core FHIR resource changes (Group, DiagnosticReport) beyond IG-level guidance, which increases scope and timeline. The Group resource was not designed for diagnostic reporting and may need additional attributes.

---

### Disposition B: Persuasive — Accept Proposal As-Is with Flat Panel Constraint

**Action:** Accept the proposal as written: panels are one level deep in DiagnosticReport, multi-level panels are flattened into separate DiagnosticReports in a Bundle, and Observation.hasMember is reserved for actual observation relationships (with the microbiology sensitivity exception).

- Add normative guidance in the Laboratory Report IG codifying this structure
- Provide examples for single panels, flattened multi-level panels, and microbiology sensitivity reports
- Document the sensitivity exception explicitly

**Rationale:** The flat-panel approach is the simplest to implement and provides a clear, unambiguous structure. It does not require core resource changes. Labs that currently report flat panels are unaffected.

**Risk:** As apitkus noted, reflex panels and tiered panels in clinical practice can be 1–5 levels deep. Flattening these loses structural information that may be clinically important, particularly for public health reporting. The interoperability concern about representing the same result in different ways is not fully addressed.

---

### Disposition C: Not Persuasive — Defer Pending Broader Grouping Resolution

**Action:** Defer this ticket until the broader Observation grouping/organizer discussion (FHIR-54562, FHIR-50472, FHIR-54887, FHIR-50534) reaches resolution in core FHIR. Do not add IG-level guidance that may conflict with an upcoming core design decision.

**Rationale:** This ticket touches on a fundamental modeling question (how to represent groupers in FHIR) that is actively being debated across multiple tickets in the R6 ballot cycle. The January Virtual WGM discussion (per ralf.herzog) may produce a core-level resolution. Premature IG-level guidance could conflict with that outcome and require rework.

**Risk:** Continued lack of guidance means implementers will continue to represent panels inconsistently, exacerbating the interoperability problem described by apitkus. The Laboratory Report IG ballot may need to proceed without this guidance.

---

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The fundamental problem — ambiguity in how panels and groupers are represented — is real and well-documented across multiple tickets. The flat-panel-only approach (Disposition B) is too restrictive for real-world use cases like reflex panels and tiered microbiology reporting. Deferral (Disposition C) perpetuates the current inconsistency.

Disposition A offers a path forward that:
- Preserves Observation for actual observations (the reporter's core goal)
- Accommodates multi-level panel structures without overloading DiagnosticReport or Observation
- Aligns with the emerging consensus in FHIR-50472 and FHIR-54562 toward a dedicated grouper mechanism
- Can be coordinated with the January Virtual WGM discussion to ensure IG-level and core-level guidance are aligned

The workgroup should coordinate with the core FHIR team on the Group resource changes and consider whether this ticket's scope should be expanded to propose those core changes or remain IG-level guidance that references a parallel core change request.
