# FHIR-53864: BodyStructure OriginCodes should be considered

## Summary

**Ticket:** [FHIR-53864](https://jira.hl7.org/browse/FHIR-53864)
**Status:** Triaged | **Type:** Change Request | **Priority:** Medium
**Reporter:** Lorraine Constable | **Assignee:** Unassigned
**Work Group:** Orders & Observations | **Specification:** FHIR Core (FHIR)
**Created:** 2026-01-08 | **Updated:** 2026-01-20

This change request asks for a new element on the BodyStructure resource to capture the **origin** of a body structure. The submitter cites organ transplant as a key use case where knowing the origin is clinically important. The Anatomical Inventory and Person Characteristics (AIPC) logical model already includes an OriginCodes attribute with five coded values: **autologous**, **non-autologous**, **allogenic**, **zooplastic**, and **synthetic**. Currently, the R6 BodyStructure resource has no element to represent this concept.

## Proposed Dispositions

### Disposition A: Persuasive

**Rationale:** The clinical use case for tracking body structure origin is well-established, particularly in organ transplantation. Knowing whether a body structure in a patient is autologous, allogenic, zooplastic, or synthetic is essential for clinical decision-making (e.g., immunosuppression management, graft monitoring, surgical planning). The AIPC logical model already defines this concept with a mature set of codes, and BodyStructure is the natural resource to represent "what structures exist in/on a patient and their characteristics." Add a new `origin` element (0..1, CodeableConcept) bound to a new value set containing the five proposed codes.

**Consequences:**
- Adds a new optional element `BodyStructure.origin` (0..1, CodeableConcept) to the R6 resource.
- Requires defining a new value set (e.g., `BodyStructureOrigin`) with codes: autologous, non-autologous, allogenic, zooplastic, synthetic.
- Should be coordinated with the companion ticket FHIR-53863 (wholeness) to ensure consistent modeling of AIPC concepts.
- Expands BodyStructure's scope beyond purely anatomical description to include provenance characteristics.

---

### Disposition B: Persuasive with Modification

**Rationale:** The concept of body structure origin is clinically valid and the organ transplant use case is compelling. However, the proposal needs further specification before implementation. The submitter should provide: (1) formal element definitions including short description and definition text, (2) clear scoping of whether this applies only to transplanted structures or more broadly, (3) a formal value set proposal with proper terminology binding (e.g., whether to use SNOMED CT concepts, a FHIR-defined code system, or the AIPC codes directly), and (4) coordination with the companion ticket FHIR-53863 to present a cohesive set of AIPC-derived enhancements. Accept the change request in principle but require the submitter to provide a concrete element proposal with value set definition for workgroup review.

**Consequences:**
- Ticket remains open pending additional specification work from the submitter.
- Submitter must propose element details (name, cardinality, type, binding strength, value set).
- Should be reviewed alongside FHIR-53863 for consistent modeling.
- Delays implementation but ensures a well-defined addition to the resource.

---

### Disposition C: Not Persuasive

**Rationale:** BodyStructure describes anatomical structures as they exist in or on a patient — their location, morphology, and qualifiers. The origin of a structure (where it came from) is a provenance/history concept that may be better captured through other mechanisms. For transplanted organs, the BiologicallyDerivedProduct resource (which already tracks collection source per FHIR-51586) and the Procedure resource (which captures what was implanted and from where) may be more appropriate homes for origin information. Adding origin to BodyStructure risks conflating structure description with provenance tracking. If the use case is specific to transplant scenarios, an extension may be more appropriate than a core element.

**Consequences:**
- No change to the BodyStructure resource.
- Submitters would need to use existing resources (BiologicallyDerivedProduct, Procedure) or propose an extension for this use case.
- Maintains BodyStructure's focused scope on anatomical description.

---

## Recommendation

**Disposition B: Persuasive with Modification**

The clinical use case is legitimate and the AIPC logical model provides a solid foundation. However, the current proposal is too high-level for direct implementation — it lists concept codes but does not define element semantics, binding strength, or how origin interacts with existing BodyStructure elements (e.g., `includedStructure`, `morphology`). Accepting with modification allows the workgroup to endorse the concept while requiring the submitter to develop a concrete proposal. This should be reviewed in conjunction with the companion ticket FHIR-53863 (wholeness) to ensure AIPC concepts are added to BodyStructure in a consistent and well-designed manner.
