## Description

This ticket asks how to model the situation where, during fulfillment of an order, the fulfiller identifies the need for **additional, alternative, or continuation procedures**. The ticket raises the following questions:

- What FHIR elements, resources, and references should link the new request to the original order?
- Should `ServiceRequest.basedOn` reference the original request?
- Should `Task.output` be used to capture the fulfiller's proposal?

Three scenarios are presented:
1. A nurse can directly authorize an order.
2. A nurse acts from protocol, implying an (implicitly approved) order.
3. A nurse identifies a need but requires an order to be authorized by a physician.

## Analysis

This ticket addresses a real gap in the Clinical Order Workflow IG: the pattern for a fulfiller to propose new, alternative, or continuation orders during fulfillment is not explicitly documented. The core question is one of **resource linkage and workflow signaling** — how does a nurse or other fulfiller create a new request that is traceable back to the original order, and how does the placer know it needs review?

The Zulip discussion on "Intent-driven interaction" provides significant design context. Lloyd McKenzie's guidance establishes two categories: (1) immediate actions handled by RESTful POST/PUT, and (2) workflow-initiating requests handled by Task. For the fulfiller-proposed scenario, the most likely pattern is a combination: the fulfiller creates a new `ServiceRequest` with `intent=proposal` (POST), linked back to the original order, and optionally creates a Task to request the placer's review/authorization.

FHIR-56051 (already Applied) directly supports this ticket by generalizing Task as the mechanism for requesting order status changes. The guidance from FHIR-55588 should build on that foundation and document how `basedOn`, `triggeredBy`, and `reason` link derived orders to originals.

## Proposed Dispositions

### Disposition A: Persuasive — Add Fulfiller Proposal Pattern with Example
**Action:** Add a new section to the COW IG documenting the "Fulfiller-Proposed Order" pattern. This would:
1. Define the linkage model: a new `ServiceRequest` with `intent=proposal`, using `basedOn` to reference the original authorization and `reason` to explain the clinical justification.
2. Describe the use of Task (building on FHIR-56051's Task profiles) for the fulfiller to request the placer's review.
3. Document the three scenarios from the ticket (nurse-authorized, protocol-implied, physician-authorization-required).
4. Include the wound care → infection → analgesic request example from the reporter's comment.
5. Clarify when `Task.output` is appropriate vs. when a new ServiceRequest is the correct approach.

**Rationale:** This is the most complete resolution. The pattern is a known gap, the Zulip discussion confirms community interest, and FHIR-56051 already lays the groundwork for Task-based status change requests.

### Disposition B: Persuasive with Modification — Add Guidance to Core Concepts Only
**Action:** Add guidance to the COW IG Core Concepts section explaining the general approach for fulfiller-proposed orders:
- Use `ServiceRequest` with `intent=proposal` linked via `basedOn`.
- Use Task to request placer review when authorization is needed.
- Reference existing patterns and defer detailed scenario documentation to a future cycle.

**Rationale:** A lighter-weight resolution that addresses the core question without committing to full scenario documentation. Appropriate if the work group wants to gather more implementation feedback before fully specifying the pattern.

### Disposition C: Deferred — Gather Implementation Experience First
**Action:** Defer this ticket to a future ballot cycle. Document the question and the emerging design direction (from Zulip and FHIR-56051) in a tracker note, but do not add normative content yet.

**Rationale:** The pattern involves complex interactions across multiple resources and roles. Deferring allows time to collect real-world implementation experience, particularly from the intent-driven interaction scenario raised in Zulip, before committing to a specific pattern.

## Recommendation

**Disposition A: Persuasive** is recommended. The fulfiller-proposed order pattern is a clearly identified gap in the COW IG, the reporter (who is also a key contributor to COW) has already outlined the resolution direction, and the closely related FHIR-56051 has been applied with supporting Task profiles. The Zulip discussion confirms active community need. Adding a documented pattern with an example will directly improve implementer guidance without requiring new resource types or breaking changes.
