# FHIR-39688: Device resource needs semantically accurate structure for setting device function parameters

## Summary

**Status:** Triaged | **Type:** Change Request | **Priority:** Highest | **Resolution:** Unresolved
**Reporter:** Stephen Chu | **Assignee:** Unassigned
**Work Group:** Orders & Observations | **Specification:** FHIR Core (FHIR)
**Labels:** DeviceSettings, HCP, Joint-OO-PC, Ready-to-Discuss, WGM-January2026
**Created:** 2022-12-23 | **Updated:** 2026-02-02
**URL:** https://jira.hl7.org/browse/FHIR-39688

Devices such as insulin infusion pumps and implantable cardioverter-defibrillators (ICDs) must be programmed with patient-specific parameters (e.g., basal/bolus insulin dosages, pacing modes). These settings may need to be adjusted over time based on changing clinical status. The reporter asserts that FHIR's current Device-related resources do not adequately support capturing initial device settings or subsequent adjustments, which are needed for decision traceability, clinical effectiveness evaluation, and future parameter-setting guidance. The ticket requests a detailed review of requirements and determination of optimal changes to the Device resource structure.

## Zulip Discussions

### implementers > FHIR resources for devices
**URL:** https://chat.fhir.org/#narrow/stream/implementers/topic/FHIR%20resources%20for%20devices
**Date range:** 2024-05-16 to 2024-05-23 | **Participants:** Olex, Lloyd McKenzie, Elliot Silver

A user (Olex) asked how to store device configuration settings (input preset and nurse-adjusted preset) using FHIR resources. Elliot Silver explained the current state:
- **DeviceMetric** defines a measurement or setting on a device but does not carry the current value.
- **Observation** (with device referencing DeviceMetric) records the current value of a setting.
- **ServiceRequest.orderDetail** with `parameterFocus` referencing the Device is the most appropriate current mechanism for requesting specific device settings during a procedure.
- **Task** was discussed as an alternative when no ServiceRequest exists — Lloyd McKenzie suggested using Task.input for each setting, or pointing to a contained Parameters instance for complex/nested parameters.
- FHIR deliberately stays away from sending settings directly to devices due to safety concerns — device command-and-control requires additional safety protocols not yet built into FHIR.

This thread was directly referenced in the FHIR-39688 comments by Elliot Silver.

## Proposed Dispositions

### Option A: Triaged — Defer to IG Project (Recommend for Future Use)
**Rationale:** The OO WG's own analysis (Hans Buitendijk, Jan 2026) concluded that while existing resources can convey actual device settings, the "instructing a device" use case is too large and insufficiently explored for R6 proposals. The appropriate path is an Implementation Guide project that can thoroughly analyze requirements and propose specific resource enhancements. This aligns with the Device WG's concern about safety protocols for device command-and-control.

**Resolution:** Considered for Future Use — Direct the reporter to initiate an IG project (possibly under OO or Devices WG sponsorship) to analyze device parameter setting use cases end-to-end, which may yield concrete proposals for a future FHIR release.

### Option B: Persuasive with Modification — Incremental Core Improvements
**Rationale:** The ticket identifies a real gap, and some incremental improvements can be made within R6 scope without tackling full device command-and-control:
1. Expand `ServiceRequest.parameterFocusReference` to include DeviceMetric.
2. Add guidance/examples showing how Device + DeviceMetric + Observation can represent device settings.
3. Coordinate with FHIR-38917 (device settings/observations guidance) and FHIR-48394 (mechanism for prompting device settings).

**Resolution:** Persuasive with Modification — Accept the need but scope R6 changes to incremental improvements (ServiceRequest reference expansion, guidance documentation) while deferring device command-and-control to an IG project.

### Option C: Not Persuasive — Existing Resources Suffice
**Rationale:** Per the OO analysis, Device, DeviceMetric, and Observation already support conveying actual device settings. ServiceRequest.orderDetail can capture desired settings. Task with inputs can represent setting change requests. The core specification already provides the building blocks; what is needed is guidance (covered by FHIR-38917) rather than structural changes to Device.

**Resolution:** Not Persuasive — The existing resource structure is sufficient for capturing device settings. Refer the reporter to FHIR-38917 for guidance improvements and FHIR-48394 for the prompting mechanism.

## Recommendation

**Option A: Considered for Future Use — Defer to IG Project**

This is the most appropriate disposition given:
1. The OO WG's own January 2026 analysis explicitly concluded this is "too big to address as a ballot comment to R6" and is best suited for an IG project.
2. The Device WG has emphasized that device command-and-control requires safety analysis not yet performed.
3. The most recent vote (2026-01-30, 11-0-1) reverted the previous "Considered for Future Use" resolution — the WG should confirm whether this reversion was intended to re-open for further discussion or to change the disposition.
4. The related tickets (FHIR-48394, FHIR-38917) can address the more tractable sub-problems (guidance, prompting mechanisms) independently.
5. An IG project would provide the structured environment needed to analyze the full device settings lifecycle — from reading current settings, to recording historical settings, to instructing setting changes — with appropriate safety considerations.
