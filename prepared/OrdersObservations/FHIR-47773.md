## Description

Eldon Metz (PCIM WG) requests the ability to convey and search by **location at the time of device association or disassociation** within `DeviceAssociation`. In HL7v2, `PRT.9` ("Participant Location") tracks this information for the IHE PCIM profile. Currently, `Device.location` only reflects the device's current location, and `_history` is neither required nor searchable, making it impossible to determine where a device was when it was associated with or disassociated from a patient.

**Use case:** A reporting system tracking device utilization or billing needs to filter by the location where a device was used (e.g., ER vs OR vs ICU). A patient monitor may be associated in the ER and move with the patient to the ICU; the association-start location is the most clinically and operationally valuable data point.

2. **Eldon Metz / PCIM WG (2024-10-16):** Proposed a backbone element structure for DeviceAssociation:
   - `location` BackboneElement 0..* — "List of locations recorded during device association"
     - `location` Reference(Location) 1..1
     - `reportedTime` dateTime 1..1
   - Initially captures start/end locations; extensible for future location tracking over time.

3. **Elliot Silver (2024-12-23):** Acknowledged the PCIM input; expressed concern about an unlimited list of locations.

4. **HCP (2025-09-29):** Elliot tasked with providing two options for the group to review.

5. **Elliot Silver (2025-10-01):** Presented three options:
   - **Option 1 — Add location elements:** `associationStartLocation` and `associationEndLocation` on DeviceAssociation, plus a search parameter. Simple, limited to the relevant time points.
   - **Option 2 — "Located in" associations:** Use DeviceAssociation to associate a device with a Location via a relationship type. More flexible but potentially confusing and indirect.
   - **Option 3 — List resource:** A List with subject=Device and location items. Reasonable but unclear where to describe it in core.

6. **Elliot Silver (2025-10-22):** Reported that the suggestion is to implement **Option 1 as extensions**. Dev WG supports the extension approach. Requested extension content be provided.

## Proposed Dispositions

### Disposition A: Persuasive — Implement as Extensions

Add `associationStartLocation` (Reference(Location)) and `associationEndLocation` (Reference(Location)) as **standard extensions** on DeviceAssociation, along with guidance on a corresponding search parameter. This follows the Dev WG's endorsed approach (Oct 2025) and keeps core DeviceAssociation unchanged while enabling implementation experience.

- **Pros:** Lower risk to core; allows real-world validation before promotion; Dev WG already supports this; aligns with the strong 11-0-1 vote to proceed.
- **Cons:** Extensions are less discoverable than core elements; implementers must know to look for them; may delay normative inclusion.

### Disposition B: Persuasive with Modification — Add Core Elements to DeviceAssociation

Add `associationStartLocation` and `associationEndLocation` directly as core elements on DeviceAssociation (Reference(Location), 0..1 each), plus a `location` search parameter that searches across both. This aligns with the PCIM WG's original structural proposal and the well-established HL7v2 PRT.9 precedent.

- **Pros:** Most direct and discoverable approach; avoids the two-step extension-then-promote path; the use case is well-validated by PCIM and IHE.
- **Cons:** Higher bar for change to core resource; may be premature without implementation experience; previous 4-0-0 vote initially deferred this.

### Disposition C: Not Persuasive — Defer to Implementation Guides

The use case is valid but niche. Recommend implementers use IG-specific extensions (e.g., in the PCIM IG) until broader community demand justifies core changes. Device.location combined with Provenance or List resources can partially address the need.

- **Pros:** No change to core spec; allows organic demand to drive future changes.
- **Cons:** Ignores the strong 11-0-1 vote and established cross-WG consensus; fragments the approach across IGs; HL7v2 PRT.9 precedent suggests this is a broadly needed capability.

## Recommendation

**Disposition A: Persuasive — Implement as Extensions**

The workgroup should approve adding location elements as standard extensions on DeviceAssociation. This approach has explicit Dev WG endorsement (Oct 2025), strong OO WG support (11-0-1 vote, Jan 2026), and a clear use case validated by the PCIM WG and IHE PCIM profile. The extension path provides the right balance: it unblocks implementers now while preserving the option to promote to core elements based on implementation experience before R6 normative publication. The previous "Considered for Future Use" resolution was intentionally reverted to allow forward progress, and the extension approach delivers that progress with minimal risk.
