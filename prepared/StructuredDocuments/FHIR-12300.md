## Description

The ticket requests that the C-CDA on FHIR specification consider using `Composition.section.emptyReason` for sections that in C-CDA require coded entries but sometimes simply have no data. The goal is to avoid the C-CDA pattern of populating sections with null flavor–coded entries when no actual data exists. The submitter notes this would be a cleaner approach than forcing implementers to produce placeholder entries with null flavors.

## Analysis

### Context

This ticket has been open since October 2016 and was re-opened after STU1.2.0 publication in March 2024. The core request is straightforward: C-CDA on FHIR should leverage `Composition.section.emptyReason` to handle sections that require coded entries but have no data, rather than forcing implementers to produce entries with null flavor codes (a long-standing pain point inherited from CDA).

### Ecosystem Developments Since Filing

Since this ticket was filed, the broader FHIR ecosystem has moved significantly toward adopting `emptyReason`:

1. **IPS (FHIR-41529, published):** The International Patient Summary specification explicitly adopted `emptyReason` (relaxed from 0..0 to 0..1) and loosened entry cardinality, with an invariant warning when a required section has no entries and no `emptyReason`. This was resolved as "Persuasive with Modification."

2. **FHIR Core invariant cmp-2:** The base Composition resource already defines the constraint that a section can only have an `emptyReason` if it has no entries (`emptyReason.empty() or entry.empty()`), providing a well-defined semantic framework.

3. **Active community discussion (2025):** The IPS Zulip thread on "emptyReason in Required sections" shows ongoing real-world need, particularly from Canada's Patient Summary implementations where different systems are authoritative for different data domains.

### Impact on C-CDA on FHIR

Adopting `emptyReason` in C-CDA on FHIR would:
- **Reduce implementer burden** by eliminating the need to create placeholder entries with null flavors.
- **Align with IPS precedent** (FHIR-41529), improving interoperability between C-CDA on FHIR and IPS documents.
- **Require profile changes** to the C-CDA on FHIR Composition profiles to allow `emptyReason` (currently likely constrained out) and potentially relax entry cardinalities in required sections.
- **Not conflict with FHIR Core** since the cmp-2 invariant already governs the relationship between `emptyReason` and `entry`.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Adopt `Composition.section.emptyReason` in C-CDA on FHIR profiles for sections that currently require coded entries but may legitimately have no data. Specifically:
- Allow `emptyReason` (0..1) on all C-CDA on FHIR Composition section slices where it is currently constrained out.
- Relax `entry` cardinality on those sections from 1..* to 0..* where appropriate.
- Add guidance text explaining when `emptyReason` should be used (e.g., no data available, system is not the authoritative source) and which codes from the `list-empty-reason` value set are appropriate.
- Add a warning-level invariant (modeled after IPS FHIR-41529) requiring `emptyReason` when a required section has no entries.

**Rationale:** This directly addresses the original request, aligns with the IPS precedent (FHIR-41529), eliminates the null-flavor antipattern, and is supported by active community discussion and real-world implementation need.

### Disposition B: Persuasive
**Action:** Accept the request as stated—consider using `Composition.section.emptyReason` for sections that require coded entries but have no data. Allow `emptyReason` on all section slices without additional constraints or guidance beyond what FHIR Core already provides via cmp-2.

**Rationale:** The request is valid and the mechanism exists in FHIR Core. A minimal change (just enabling `emptyReason`) reduces the risk of over-constraining and lets implementers use the element as defined by the base specification.

### Disposition C: Not Persuasive
**Action:** Reject the request. Maintain the current C-CDA on FHIR approach where sections that require coded entries must always include entry resources, potentially with absence/negation codes (e.g., using coded entries like "no known allergies" rather than `emptyReason`).

**Rationale:** The C-CDA on FHIR specification intentionally requires coded entries to ensure clinical completeness and facilitate clinical decision support. Using `emptyReason` instead of coded absence entries could reduce the quality of structured data available for downstream processing. The CDA tradition of using specific absence codes (rather than a generic "empty" flag) provides richer semantics.

---

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The request is well-founded, addresses a genuine implementer pain point, and is strongly supported by ecosystem precedent. The IPS specification has already adopted this pattern (FHIR-41529, published), and active community discussions confirm real-world need. Adopting `emptyReason` with appropriate guidance and invariants provides a clean, interoperable solution while maintaining the ability to distinguish *why* a section is empty. The modification to add explicit guidance and warning-level invariants ensures that the change is made responsibly, preventing misuse while eliminating the null-flavor antipattern that motivated the original request.
