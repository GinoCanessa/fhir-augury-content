## Details

The reporter (Corey Spears) identifies that the Family History section in the C-CDA on FHIR Implementation Guide does not include an entry reference to the DAF-FamilyMemberHistory profile (`http://hl7.org/fhir/StructureDefinition/daf-familymemberhistory`). The expectation is that `Composition.section.entry` for the Family History section should reference the U.S. DAF FamilyMemberHistory Profile with cardinality `[0..*]` (or `[1..*]`), but this constraint is missing from the profile definition.

The reporter provides the anticipated XML element definition that should appear, specifying a `Reference` type pointing to the `daf-familymemberhistory` StructureDefinition.

**Comments:**
- **lynn_laakso (2024-03-12):** Re-opened post STU1.2 publication. Also noted the original related URL: `http://build.fhir.org/ccda/ccda-ccd-composition.html`.

**Context:** This ticket is part of a cluster of issues filed by the same reporter around the same time, all identifying missing DAF profile references in various C-CDA on FHIR Composition sections. Since its original filing in 2016, the DAF profiles have been superseded by US Core profiles, and the C-CDA on FHIR IG has undergone significant evolution. The ticket was re-opened in 2024 post STU1.2 publication, suggesting the underlying issue may still be relevant in some form.

## Proposed Dispositions

### Disposition A: Accept — Update Family History Section Entry Reference to Use Current US Core Profile

**Rationale:** The core issue is valid — the Family History section of C-CDA on FHIR should constrain `Composition.section.entry` to reference the appropriate FamilyMemberHistory profile. However, since the original DAF profiles have been superseded, the fix should reference the current US Core FamilyMemberHistory profile (or the appropriate C-CDA on FHIR FamilyMemberHistory profile) rather than the now-deprecated `daf-familymemberhistory`. Update the Family History section's `Composition.section.entry` to include a typed reference to the current FamilyMemberHistory profile with appropriate cardinality.

**Scope:** Modify the C-CDA on FHIR CCD Composition profile to add the missing entry constraint for the Family History section.

### Disposition B: Accept as Duplicate — Consolidate with FHIR-10320

**Rationale:** FHIR-10320 ("Sections with DAF defined entries do not have a proper profile reference") appears to address the same underlying issue more broadly, covering multiple sections including Family History. Rather than resolving each section-specific ticket individually, consolidate this ticket into FHIR-10320 and address all missing section entry profile references in a single change. Close FHIR-10194 as a duplicate of FHIR-10320.

**Scope:** Close this ticket and track the fix under FHIR-10320.

### Disposition C: Decline — Not Applicable to Current IG Version

**Rationale:** This ticket was filed in 2016 against the original C-CDA on FHIR IG using DAF profiles. The specification has undergone substantial revision since then — DAF profiles were replaced by US Core, and the C-CDA on FHIR IG has been restructured. If the current version of the IG already correctly constrains the Family History section entry (or if the IG no longer uses a structure where this constraint applies), the ticket may no longer be relevant. Close as not applicable, noting that the IG has evolved significantly since the issue was reported.

**Scope:** No specification change required.

## Recommendation

**Disposition A: Accept — Update Family History Section Entry Reference to Use Current US Core Profile**

This is recommended because the underlying issue — missing profile constraints on section entries — represents a genuine gap in conformance guidance. The ticket was deliberately re-opened in 2024 post STU1.2 by lynn_laakso, indicating the issue persists in the current IG. Updating the entry reference to point to the current FamilyMemberHistory profile (replacing the deprecated DAF reference) would improve implementation consistency. The cluster of related tickets (FHIR-10193, FHIR-10195, FHIR-10320, etc.) should also be reviewed together to ensure all section entry constraints are addressed comprehensively.
