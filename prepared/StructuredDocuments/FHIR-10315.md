## Description

The C-CDA on FHIR composition profiles use `fixedCodeableConcept` to constrain codes. This approach is overly restrictive because it requires an exact match of **all** elements in the CodeableConcept, which disallows providing optional elements such as `displayName`. The reporter requests that codes instead be defined at the leaf level (i.e., constraining `code` and `system` individually), which would permit additional optional elements to be included.

This issue affects **all** composition structure definitions in C-CDA on FHIR **except** Transfer Summary. The original related URL referenced the Consultation Note Composition profile: http://build.fhir.org/ccda/ccda-consultation-note-composition.html

## Proposed Dispositions

### Disposition A: Persuasive

**Rationale:** Replace `fixedCodeableConcept` with `patternCodeableConcept` (or equivalent leaf-level constraints on `code` and `system`) across all affected C-CDA on FHIR composition profiles (all except Transfer Summary). This aligns with established FHIR profiling best practices: `pattern[x]` requires the specified values to be present while allowing additional optional elements like `display`. This is the same approach that was resolved as Persuasive in FHIR-33151 for the VRDR IG. Additionally, FHIR-10314 and FHIR-10316 should be resolved as duplicates of this ticket.

**Pros:**
- Consistent with FHIR best practices and the precedent set by FHIR-33151
- Enables implementers to include `displayName` and other optional elements, improving readability and interoperability
- Resolves a long-standing issue (open since 2016) that was deliberately re-opened post-STU1.2

**Cons:**
- Technically a breaking change: instances that were previously invalid (because they included extra elements like `display`) would now become valid, which could affect validation workflows
- Requires auditing all affected composition profiles to ensure no unintended loosening of constraints

### Disposition B: Persuasive with Modification

**Rationale:** Apply the change from `fixedCodeableConcept` to `patternCodeableConcept`, but scope the modification to only the `Composition.type` and `Composition.section.code` elements where the constraint is demonstrably too restrictive. Conduct a per-profile audit to determine whether any specific code constraints intentionally require exact matching (e.g., for safety-critical sections). Document the rationale for each change.

**Pros:**
- More cautious approach that avoids blanket changes
- Allows the work group to evaluate each constraint individually
- Still resolves the core issue for the most impactful elements

**Cons:**
- More labor-intensive than a blanket change
- May result in inconsistent constraint styles across profiles if some elements remain as `fixed[x]`

### Disposition C: Not Persuasive

**Rationale:** Reject the change on the grounds that the C-CDA on FHIR IG may have evolved since 2016 and newer versions may already address this issue, or that `fixedCodeableConcept` was intentionally chosen to enforce strict interoperability by requiring exact matches.

**Pros:**
- Preserves strict validation behavior
- No risk of unintended constraint loosening

**Cons:**
- Contradicts established FHIR profiling best practices and the Persuasive resolution of FHIR-33151
- Leaves the long-standing usability issue unresolved
- The ticket was explicitly re-opened post-STU1.2 publication, indicating the issue persists

## Recommendation

**Disposition A: Persuasive**

The request is well-founded and directly aligns with modern FHIR profiling best practices. The use of `fixedCodeableConcept` is widely recognized as overly restrictive when the intent is to require a specific code/system while permitting optional elements like `displayName`. The precedent set by FHIR-33151 (resolved Persuasive for the identical issue in VRDR) strongly supports this disposition. The ticket has been open since 2016 and was deliberately re-opened in 2024, indicating continued relevance. FHIR-10314 and FHIR-10316 should be resolved as duplicates.
