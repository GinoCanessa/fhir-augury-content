# FHIR-10320: Sections with DAF defined entries do not have a proper profile reference

## Summary

**Status:** Triaged | **Type:** Change Request | **Priority:** Medium | **Resolution:** Unresolved
**Specification:** US C-CDA on FHIR (FHIR)
**Work Group:** Structured Documents
**Reporter:** Corey Spears | **Assignee:** Unassigned
**Created:** 2016-07-14 | **Updated:** 2024-03-12
**URL:** https://jira.hl7.org/browse/FHIR-10320

Multiple `Composition.section.entry` elements in the C-CDA on FHIR StructureDefinitions describe DAF profile references in the `definition` text but are missing the corresponding `<type>` element with `<code value="Reference"/>` and `<profile value="..."/>`. Without the `<type>` block, validators and tooling cannot enforce or detect the intended profile constraint. The reporter identified eight affected sections: Allergies and Intolerances, Medications, Problems, Results, Family History, Immunizations, Vital Signs, and Procedures.

## Proposed Dispositions

### Disposition A: Applied — Add missing `<type>` blocks to all affected sections

**Resolution:** Add the missing `<type>` element with `<code value="Reference"/>` and `<profile value="..."/>` to each of the eight identified `Composition.section.entry` elements (Allergies and Intolerances, Medications, Problems, Results, Family History, Immunizations, Vital Signs, Procedures). This makes the StructureDefinition machine-enforceable and aligns the formal constraint with the narrative definition text already present.

**Rationale:** The definition text already states the intended DAF profile (e.g., "U.S. DAF AllergyIntolerance Profile"), but without the `<type>` block, tooling cannot validate conformance. This is a straightforward profile authoring fix. Note that DAF profiles have since been superseded by US Core profiles, so the fix should use the current equivalent US Core profile URLs if the IG is being updated to a current FHIR version. Consider consolidating with related tickets (FHIR-10193, FHIR-10194, FHIR-10195, FHIR-10324) that report the identical pattern for other sections.

### Disposition B: Applied — Consolidate with related tickets and resolve as a batch

**Resolution:** Merge FHIR-10320 with the cluster of related tickets (FHIR-10193, FHIR-10194, FHIR-10195, FHIR-10324/10325/10326, FHIR-10337) that all report the same missing `<type>` block pattern across different C-CDA on FHIR sections. Address them all in a single StructureDefinition update pass, closing the duplicates and tracking the work under one ticket.

**Rationale:** These tickets were all filed by the same reporter (Corey Spears) in mid-2016, all describe the same structural defect, and all remain in Triaged status. Consolidating reduces tracking overhead and ensures no section is missed. However, since DAF has been superseded by US Core, the workgroup should decide whether to fix against the current IG version with US Core profile references or note that this has been addressed in subsequent IG publications.

### Disposition C: Not Persuasive — Superseded by US Core migration

**Resolution:** Close as Not Persuasive with the rationale that the DAF profiles referenced in this ticket (e.g., `daf-allergyintolerance`, `daf-condition`) have been superseded by US Core profiles. The C-CDA on FHIR IG has undergone significant revisions since STU1, and the specific StructureDefinition structures described in the ticket no longer reflect the current IG. The defect may have been corrected as part of later IG updates.

**Rationale:** The ticket was filed in 2016 against STU1-era profiles. DAF has been replaced by US Core, and C-CDA on FHIR has been updated multiple times. If the current IG version already includes proper `<type>` blocks with US Core profile references, this ticket is moot. However, lynn_laakso's 2024 comment ("re-open post STU1.2 publication") suggests the issue may persist in the current version.

## Recommendation

**Disposition B: Applied — Consolidate with related tickets and resolve as a batch.**

This is the most efficient path. The cluster of related tickets (FHIR-10193, 10194, 10195, 10324–10326, 10337) all stem from the same systematic omission of `<type>` blocks in C-CDA on FHIR section entry definitions. Consolidating them under a single tracking ticket enables a comprehensive fix. The workgroup should verify whether the current C-CDA on FHIR IG version still exhibits this defect; if it does, the fix should use current US Core profile URLs. Lynn Laakso's 2024 comment reopening the ticket post-STU1.2 suggests the issue was not fully resolved in subsequent publications, making active resolution preferable to closing as superseded.
