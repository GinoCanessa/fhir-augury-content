## Description

The submitter reports that all codes defined in the C-CDA on FHIR Composition profiles use `fixedCodableConcept` constraints. The problem with `fixedCodableConcept` is that it **disallows** providing optional sub-elements such as `displayName` or other elements within the CodeableConcept. This issue affects all C-CDA on FHIR Composition structure definitions **except** Transfer Summary.

The request is to replace `fixedCodableConcept` with an approach that defines the code and coding system at the leaf level (e.g., using `fixedCode` and `fixedUri` on `Coding.code` and `Coding.system`, or using `patternCodableConcept`), allowing implementers to include additional optional elements like `display`.

## Analysis

This is a **long-standing profiling best-practice issue** originally filed in 2016 and re-opened in 2024 after STU1.2 publication. The core problem is well-understood in the FHIR profiling community:

- **`fixed[x]`** constrains every sub-element of a complex type. For `CodeableConcept`, this means that if a `fixedCodableConcept` is set, the instance must match *exactly* — no additional `Coding` entries, no `text`, and no `display` on the fixed coding unless explicitly included in the fixed value. This is overly restrictive for most use cases.
- **`pattern[x]`** (introduced as the recommended alternative) requires that the specified pattern elements are present but allows additional sub-elements. Using `patternCodableConcept` would allow implementers to include `display`, additional codings, or `text` while still requiring the specified code/system.
- Alternatively, constraining at the **leaf level** (e.g., `fixedCode` on `Coding.code` and `fixedUri` on `Coding.system`) achieves a similar effect by only fixing the specific sub-elements.

FHIR-10315 and FHIR-10316 are apparent duplicates filed within seconds of this ticket with identical content.

## Proposed Dispositions

### Disposition A: Persuasive — Replace fixedCodableConcept with patternCodableConcept

**Action:** Change all C-CDA on FHIR Composition profiles (except Transfer Summary, which reportedly does not have this issue) to use `patternCodableConcept` instead of `fixedCodableConcept` for type codes. This aligns with current FHIR profiling best practices where `pattern[x]` is the recommended approach for requiring a code while still allowing additional data elements like `display`.

**Rationale:** The FHIR community has broadly moved toward `patternCodableConcept` as the preferred mechanism for constraining CodeableConcept values in profiles. This change would bring C-CDA on FHIR profiles in line with current guidance, resolve the implementer pain point of not being able to include `display` values, and is a straightforward mechanical change across the affected profiles.

### Disposition B: Persuasive with Modification — Use leaf-level fixed constraints

**Action:** Instead of switching to `patternCodableConcept`, constrain at the leaf level by applying `fixedCode` on `Coding.code` and `fixedUri` on `Coding.system` within the relevant slices. Remove the `fixedCodableConcept` constraints. This allows `display` and other optional elements while still requiring the exact code and system.

**Rationale:** Leaf-level constraints provide more explicit control and may be clearer to implementers about exactly which elements are required vs. optional. However, this approach is more verbose and `patternCodableConcept` has become the community-preferred idiom.

### Disposition C: Not Persuasive — Defer to next major version

**Action:** Acknowledge the issue but defer resolution, noting that the current constraints are technically correct even if overly restrictive, and that changing them could affect existing implementations that depend on the exact fixed values for validation.

**Rationale:** Changing from `fixed[x]` to `pattern[x]` or leaf-level constraints is a breaking change for validators and implementations that rely on exact matching. However, this concern is mitigated by the fact that `pattern[x]` is strictly *less* restrictive — all instances valid under `fixed[x]` would also be valid under `pattern[x]`.

## Recommendation

**Disposition A: Persuasive — Replace fixedCodableConcept with patternCodableConcept.**

This is the most appropriate resolution because:
1. It directly addresses the reported problem (inability to include `display` and other optional elements).
2. It aligns with current FHIR profiling best practices and community consensus.
3. The change is backward-compatible in terms of instance validity (all previously valid instances remain valid).
4. FHIR-10315 and FHIR-10316 should be resolved as duplicates of this ticket.
5. The ticket has been open since 2016 and was deliberately re-opened in 2024, indicating ongoing relevance and intent to address it.
