## Description

The ticket requests an evaluation of how to make it more evident which IPS profiles are referenced by the `observation-member-laboratory-uv-ips` and `observation-member-pathology-uv-ips` profiles. The reporter (Giorgio Cangioli) points to the `specimen-uv-ips` profile as an example of how profile cross-references could be presented more clearly. The affected profiles are:

- [Observation-member-laboratory-uv-ips](http://hl7.org/fhir/uv/ips/StructureDefinition-observation-member-laboratory-uv-ips.html)
- [Observation-member-pathology-uv-ips](http://hl7.org/fhir/uv/ips/StructureDefinition-observation-member-pathology-uv-ips.html)

## Proposed Dispositions

### Disposition A: Persuasive — Improve profile cross-reference visibility

**Rationale:** The request is a valid usability improvement. Implementers working with the IPS laboratory and pathology observation profiles should be able to easily identify which other IPS profiles are referenced (e.g., Specimen, DiagnosticReport). The companion tickets (FHIR-16278, FHIR-16279) targeting the same profiles were both accepted as Persuasive. The `specimen-uv-ips` profile cited by the reporter demonstrates a workable pattern for making profile cross-references more evident. The ticket was deferred in 2018 and explicitly re-opened in 2022, indicating the work group recognized the issue still needed attention.

**Action:** Add clearer documentation or structured references in the profile pages for `observation-member-laboratory-uv-ips` and `observation-member-pathology-uv-ips` (or their current equivalents) to make related IPS profiles more discoverable. Follow the pattern used by `specimen-uv-ips` or adopt IG Publisher capabilities for profile dependency visualization.

### Disposition B: Not Persuasive — Profile structure has evolved

**Rationale:** The IPS IG has undergone substantial restructuring since 2018. The specific profiles referenced (`observation-member-laboratory-uv-ips` and `observation-member-pathology-uv-ips`) may have been refactored, consolidated, or renamed in more recent versions of the IG. If the original profiles no longer exist in their 2018 form, the request may no longer apply. Additionally, modern IG Publisher tooling now generates profile dependency views automatically, potentially making the manual enhancement requested here unnecessary.

**Action:** Close with no changes if the work group confirms the original profiles have been superseded and current tooling adequately addresses the concern.

### Disposition C: Considered for Future Use — Defer to broader IG tooling improvements

**Rationale:** The request is fundamentally about IG presentation quality — how profile pages display cross-references to other profiles. This is partly an IG Publisher tooling concern rather than an IPS-specific content issue. The work group could defer this to a broader initiative around IG Publisher profile page improvements that would benefit all IGs, not just IPS.

**Action:** Defer resolution pending assessment of whether current IG Publisher features already address the concern, or whether a cross-cutting improvement request should be filed with the FHIR tooling team.

## Recommendation

**Disposition B: Not Persuasive** is recommended.

The ticket is over 7 years old and targets profiles (`observation-member-laboratory-uv-ips` and `observation-member-pathology-uv-ips`) from the 2018 STU ballot of the IPS IG. The IPS specification has undergone major restructuring since then — laboratory and pathology observation profiles have been consolidated into `Observation-results-laboratory-uv-ips` and `Observation-results-pathology-uv-ips` in more recent versions. The modern IG Publisher also now provides improved profile rendering, including dependency graphs and type reference links, which partially addresses the original concern. The companion tickets (FHIR-16278, FHIR-16279) were resolved against the old profiles; this ticket's request for improved cross-reference visibility is effectively moot given the profile restructuring. If profile cross-reference clarity remains a concern for the current profiles, a new ticket against the current IPS profile structure would be more appropriate.
