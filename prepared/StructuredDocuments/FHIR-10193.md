## Summary

The reporter noted that the Encounters section of the C-CDA on FHIR Implementation Guide does not provide an entry reference to the DAF-Encounter profile in `Composition.section.entry`. The expected behavior was that the Encounters section would constrain its entry elements to reference `http://hl7.org/fhir/StructureDefinition/daf-encounter` (the U.S. DAF Encounter Profile), with cardinality `[1..*]`.

This ticket was previously resolved as "Considered for Future Use" in February 2017 (vote Rick Geimer/Calvin Beebe: 16-0-0) with the note that DAF-Encounter had been removed from US-Core and the section would link to an un-profiled Encounter resource. However, the ticket was re-opened by Lynn Laakso in March 2024 following STU1.2 publication.

## Details

- **Original context:** The C-CDA on FHIR CCD Composition profile ([http://build.fhir.org/ccda/ccda-ccd-composition.html](http://build.fhir.org/ccda/ccda-ccd-composition.html)) lacked a typed profile reference for Encounter entries in the Encounters section.
- **Prior resolution (2017):** The DAF-Encounter profile was removed from US-Core. The workgroup voted (16-0-0) to resolve by linking to an un-profiled Encounter resource, and the ticket was marked "Considered for Future Use."
- **Re-opening (2024-03-12):** Lynn Laakso re-opened this ticket after STU1.2 publication, reverting the previous resolution. The current landscape has evolved significantly since 2016 — US Core now includes the [US Core Encounter Profile](http://hl7.org/fhir/us/core/StructureDefinition-us-core-encounter.html), and C-CDA on FHIR has undergone multiple updates.
- **Core question:** Should the Encounters section of the C-CDA on FHIR Composition constrain `Composition.section.entry` to reference the current US Core Encounter profile (or another appropriate profile), rather than leaving it un-profiled?

## Proposed Dispositions

### Disposition A: Accept — Constrain entry to US Core Encounter profile

**Action:** Update the Encounters section of the C-CDA on FHIR CCD Composition profile to constrain `Composition.section.entry` to reference the [US Core Encounter Profile](http://hl7.org/fhir/us/core/StructureDefinition-us-core-encounter.html) with cardinality `[0..*]` (SHOULD).

**Rationale:** The original request was valid — the Encounters section should reference a profiled Encounter resource rather than leaving it un-profiled. Since 2017, the DAF-Encounter profile has been succeeded by the US Core Encounter profile, which is now the standard US Realm Encounter profile. Constraining the entry to reference US Core Encounter aligns with how other sections of C-CDA on FHIR reference US Core profiles for their entries. This also aligns with the broader issue raised in FHIR-10320.

**Impact:** Moderate. Existing implementations that already populate Encounters entries with US Core Encounter-conformant resources would not be affected. Implementations using un-profiled Encounter resources would need to ensure conformance to US Core Encounter. This should be coordinated with sibling tickets FHIR-10194, FHIR-10195, and FHIR-10320 for consistency.

### Disposition B: Accept with modification — Reference base Encounter with guidance

**Action:** Keep `Composition.section.entry` referencing the base Encounter resource but add implementation guidance (in the narrative/description) recommending the use of the US Core Encounter profile in US Realm implementations.

**Rationale:** C-CDA on FHIR may be used in contexts beyond US Realm where US Core profiles are not applicable. By keeping the base Encounter reference but adding guidance, the specification remains flexible while still directing US implementers toward the appropriate profile. This avoids a hard constraint that could limit international adoption.

**Impact:** Low. No breaking changes. Adds informative guidance only. However, this approach is weaker than Disposition A and may not fully address the reporter's concern about interoperability.

### Disposition C: Decline — Close as not applicable

**Action:** Close the ticket noting that the original concern about DAF-Encounter is obsolete, and the current specification already allows Encounter references.

**Rationale:** The original ticket was about the DAF-Encounter profile, which no longer exists. The 2017 resolution already addressed this by linking to an un-profiled Encounter. The re-opening rationale is unclear — STU1.2 publication alone does not necessarily warrant revisiting a previously resolved decision. The entry element already allows Encounter references, so no change is technically needed.

**Impact:** None. However, this disposition ignores the opportunity to improve the specification by adding profile constraints that improve interoperability, and leaves this ticket inconsistent with how other sections handle profiled entries.

---

## Recommendation

**Disposition A: Accept — Constrain entry to US Core Encounter profile.**

The US Core Encounter profile now exists as the successor to DAF-Encounter and is widely adopted. Constraining the Encounters section entry to reference it is consistent with the pattern used in other C-CDA on FHIR sections that reference US Core profiles. The ticket was appropriately re-opened post-STU1.2 to align with the current US Core landscape. This change should be coordinated with the sibling tickets (FHIR-10194, FHIR-10195, FHIR-10320) to ensure a consistent approach across all affected sections.
