## Summary

The reporter identifies that the `ccda-us-realm-header-composition` StructureDefinition (`http://hl7.org/fhir/ccda/StructureDefinition/ccda-us-realm-header-composition`) does not specify a referenced profile from the `ccda-authorization-extension` structure. The reporter requests clarification on the intended approach for this extension, as the lack of a referenced profile makes it unclear how implementers should use the authorization extension within C-CDA on FHIR documents.

## Details

**Description:**

The `ccda-us-realm-header-composition` specifies no referenced profile from the `ccda-authorization-extension` structure. The reporter states they are not clear on the intended approach for this extension and requests further guidance.

This is a conformance and profiling issue: the authorization extension exists in the C-CDA on FHIR specification but does not constrain its value to a specific profile (e.g., a Consent resource profile), leaving implementers without clear direction on how to populate the extension.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Add a referenced Consent profile (or other appropriate resource profile) to the `ccda-authorization-extension` within the `ccda-us-realm-header-composition` StructureDefinition. The extension should specify a target profile so that implementers know which resource type and constraints are expected when populating the authorization data in a C-CDA on FHIR Composition.
#### Justification
The reporter raises a valid conformance concern. An extension that references no profile provides no implementable guidance. Adding a referenced profile aligns the C-CDA on FHIR specification with the CDA Authorization consent model and gives implementers a clear contract for the extension's expected content. This is consistent with HL7 profiling best practices, where extensions should be as specific as practical.

---

### Disposition B: Alternative Approach
#### Proposal
Rather than adding a referenced profile, update the extension definition and its associated documentation to clarify the intended usage. Add narrative guidance explaining that the extension may reference any Consent resource (or that it is intentionally unconstrained) and provide an example demonstrating correct use. If the C-CDA on FHIR Consent profile does not yet exist, create one alongside this documentation update.
#### Justification
This approach addresses the reporter's confusion without immediately constraining the extension. If the workgroup has not yet finalized a Consent profile for C-CDA on FHIR, it may be premature to lock in a specific profile reference. Providing documentation and examples can unblock implementers while the workgroup determines the right level of constraint. This also avoids breaking changes for any existing implementations that may already use the extension with varying Consent representations.

---

### Disposition C: Decline
#### Proposal
Close the ticket without making changes to the extension definition, on the basis that the extension is intentionally unconstrained and no profile reference is required at this time.
#### Justification
It is defensible to decline if the workgroup considers the authorization extension to be a general-purpose hook that should remain flexible. Some extensions are deliberately left open to allow implementation-specific profiles. However, this leaves the original implementer's confusion unresolved and may lead to inconsistent implementations across C-CDA on FHIR adopters. Given the ticket's age (filed in 2016, last updated 2019) with no comments or resolution, declining would at least provide closure.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach)

Disposition B is recommended because it directly addresses the reporter's request for guidance while preserving flexibility for the workgroup. The ticket has been open since 2016 with no comments or progress, suggesting the workgroup has not yet converged on a specific Consent profile for C-CDA on FHIR. Rather than forcing a premature profile constraint (Disposition A) or leaving implementers without any guidance (Disposition C), updating the extension's documentation with clear narrative guidance and an example would provide immediate value to implementers. If and when a formal C-CDA Consent profile is developed, the extension can be further constrained at that time. This incremental approach resolves the reported confusion, avoids breaking changes, and gives the workgroup a clear path to tighten the specification in a future cycle.
