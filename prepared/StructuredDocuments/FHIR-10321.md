## Summary

The C-CDA on FHIR Referral Note Composition profile references a ValueSet by OID-based URL (`http://hl7.org/fhir/ValueSet/2.16.840.1.113883.1.11.20.2.3`) for `Composition.type`, but this ValueSet does not exist in the build. The reporter requests that the ValueSet be created and added. A comment by lynn_laakso (2024-03-12) notes that this ticket was reopened post STU1.2 publication, with the original related page being `http://build.fhir.org/ccda/ccda-referral-note-composition.html`.

## Analysis

This ticket is part of a cluster of five related issues (FHIR-10321, 10322, 10323, 10338, 10339) all filed by Corey Spears on 2016-07-14, each reporting missing or broken `Composition.type` ValueSet bindings across different C-CDA on FHIR document type profiles (Referral Note, Transfer Summary, etc.). These ValueSets were referenced by OID-based URLs that did not resolve to actual ValueSet resources in the FHIR build.

The ticket has been open for over 8 years and was explicitly reopened post STU1.2 publication (per lynn_laakso's March 2024 comment), indicating the issue persists. Since 2016, the Composition.type terminology landscape has evolved significantly — FHIR-28078 (Applied/Persuasive) moved DocumentReference.type to align with Composition.type using LOINC DOC type codes rather than OID-based C-CDA ValueSets. The current C-CDA on FHIR IG may have undergone restructuring that either resolves or changes the nature of this issue.

The workgroup should determine: (1) whether the referenced OID-based ValueSet URL is still the correct approach for the Referral Note profile, or whether the profile should be updated to use a LOINC-based ValueSet consistent with the direction established in FHIR-28078; and (2) whether this ticket and its siblings (10322, 10323, 10338, 10339) should be resolved collectively.

## Proposed Dispositions

### Disposition A: Persuasive — Add the Missing ValueSet
**Action:** Create the ValueSet at `http://hl7.org/fhir/ValueSet/2.16.840.1.113883.1.11.20.2.3` and include it in the C-CDA on FHIR build so the Referral Note Composition profile's `Composition.type` binding resolves correctly.
**Rationale:** This directly addresses the reporter's request and fixes a broken reference. The OID-based ValueSet approach may still be appropriate for C-CDA-specific document type codes that don't map cleanly to LOINC.

### Disposition B: Persuasive with Modification — Update Binding to LOINC-based ValueSet
**Action:** Instead of creating the OID-based ValueSet, update the Referral Note Composition profile to bind `Composition.type` to an appropriate LOINC-based ValueSet (e.g., a subset of `http://hl7.org/fhir/ValueSet/doc-typecodes`), aligning with the direction established in FHIR-28078. Coordinate resolution of sibling tickets FHIR-10322, 10323, 10338, and 10339 under the same approach.
**Rationale:** The terminology landscape has evolved since 2016. Aligning with LOINC-based document type codes provides consistency with FHIR Core direction, reduces maintenance burden of OID-based ValueSets, and supports harmonization across IGs.

### Disposition C: Not Persuasive — Defer to Current IG State
**Action:** Close as Not Persuasive if the current version of the C-CDA on FHIR IG has already addressed this binding (e.g., the Referral Note profile no longer references this OID-based URL, or the profile has been removed/restructured).
**Rationale:** Given the age of the ticket and multiple IG revisions since 2016, the issue may no longer be applicable. If the current IG already uses valid terminology bindings, no further action is needed.

---

## Recommendation

**Disposition B: Persuasive with Modification** — The workgroup should update the Referral Note profile's `Composition.type` binding to use a LOINC-based ValueSet consistent with the FHIR Core direction established in FHIR-28078, rather than attempting to create the now-outdated OID-based ValueSet. This also provides an opportunity to coordinate resolution of the entire cluster of related tickets (FHIR-10322, 10323, 10338, 10339) under a unified terminology approach. The reopening post STU1.2 confirms the issue is still relevant, and a modernized fix avoids perpetuating legacy OID-based patterns.
