## Summary

This change request asks for the removal of Trifolia-generated conformance (CONF) statement identifiers from the `definition` text of elements in C-CDA on FHIR StructureDefinition profiles. These identifiers (e.g., `(CONF:2219-146)`) are artifacts of the Trifolia authoring tool, which was originally built to produce conformance statements for CDA text/word documents and testing tools. They carry no meaning in the FHIR specification context and clutter the element descriptions across nearly all C-CDA StructureDefinitions.

The ticket was originally filed in June 2016. A commit by Rick Geimer in August 2017 to the [HL7/ccda-on-fhir](https://github.com/HL7/ccda-on-fhir) repository addressed this ("Removed Trifolia conf statements"). However, the ticket was re-opened by Lynn Laakso in March 2024 with the note "re-open post STU1.2 publication," suggesting that CONF statements have reappeared or were not fully removed in the current published version.

---

## Details

**Problem:** StructureDefinition profiles representing C-CDA templates that were exported from the Trifolia tool include tool-generated CONF statement identifiers appended to element definition text. For example, a definition value might end with `(CONF:2219-146)`. These identifiers were meaningful in the CDA conformance testing ecosystem but have no relevance in FHIR.

**Scope:** The issue affects nearly all C-CDA StructureDefinitions in the US C-CDA on FHIR implementation guide. The CONF text appears at the end of `StructureDefinition/differential/element/definition/@value`.

**History:**
- **2016-06-16:** Ticket filed by Corey Spears (javachance), noting this is an artifact of Trifolia.
- **2017-08-07:** Rick Geimer committed a fix to HL7/ccda-on-fhir that "Removed Trifolia conf statements" along with other cleanup (renamed bat file, cleaned up slice names, added section.entry profiles).
- **2024-03-12:** Lynn Laakso re-opened the ticket post STU1.2 publication, and noted the original related URL: http://build.fhir.org/ccda/ccda.html.

**Current State:** Despite the 2017 fix, the ticket remains in "Triaged" status, indicating CONF statements may persist in current or newer versions of the IG and need to be verified and cleaned again.

## Proposed Dispositions

### Disposition A: Accept — Remove All Remaining CONF Statements

**Action:** Audit all StructureDefinition profiles in the current C-CDA on FHIR IG for any remaining `(CONF:xxxx-xxx)` patterns in element `definition` values. Remove all such occurrences. Ensure the Trifolia export process (if still used) no longer injects CONF identifiers into FHIR profile definitions.

**Rationale:** CONF identifiers are CDA-era artifacts that add no value to FHIR profiles. They create noise in the specification text and could confuse implementers. The 2017 fix may have been incomplete or the CONF statements may have been reintroduced during subsequent IG updates or re-exports from Trifolia.

### Disposition B: Accept with Migration — Remove CONF Statements and Add Mapping

**Action:** Remove all `(CONF:xxxx-xxx)` patterns from element definition text, but preserve the CDA conformance statement mappings as structured data in `StructureDefinition.mapping` entries (using a CDA mapping identity). This retains the traceability between FHIR profile elements and their original CDA template conformance statements without polluting human-readable descriptions.

**Rationale:** Some implementers transitioning from CDA to FHIR may find value in knowing which CDA conformance statement corresponds to which FHIR element. A structured mapping preserves this information without cluttering definitions.

### Disposition C: Decline — Close as Previously Addressed

**Action:** Close the ticket as already resolved based on the 2017 commit by Rick Geimer. If CONF statements have genuinely reappeared, request a new ticket with specific examples from the current published IG version.

**Rationale:** The 2017 fix was applied and the IG built successfully at that time. If the issue has returned, it may warrant a fresh, scoped ticket rather than re-opening a 2016 item.

---

## Recommendation

**Disposition A: Accept — Remove All Remaining CONF Statements.**

The ticket was re-opened in 2024 specifically because CONF statements persist post STU1.2 publication. The 2017 fix was either incomplete or subsequent IG updates reintroduced the artifacts. A clean audit and removal of all `(CONF:xxxx-xxx)` patterns from element definitions is the most straightforward resolution. These identifiers serve no purpose in FHIR and should not be carried forward. The work is low-risk (editorial/descriptive text only, no structural or behavioral impact on the profiles) and directly improves readability for implementers.
