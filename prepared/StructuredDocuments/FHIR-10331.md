## Description

The ticket reports that certain complex C-CDA R2.1 conformance constraints for the **Referral Note** document type are not represented in the US C-CDA on FHIR implementation guide. Specifically, the following C-CDA conformance statements are cited as unaddressed:

- **CONF:1198-29102** — The structuredBody SHALL contain an Assessment and Plan Section (V2) (`2.16.840.1.113883.10.20.22.2.9:2014-06-09`), OR an Assessment Section (`2.16.840.1.113883.10.20.22.2.8`) AND a Plan of Treatment Section (V2) (`2.16.840.1.113883.10.20.22.2.10:2014-06-09`).
- **CONF:1198-29103** — The structuredBody SHALL NOT contain an Assessment and Plan Section (V2) when either an Assessment Section or a Plan of Treatment Section (V2) is present (mutual exclusion).

These are "either/or but not both" constraints governing which section combinations are permitted in the document body. The original related URL references the [C-CDA Referral Note Composition](http://build.fhir.org/ccda/ccda-referral-note-composition.html).

## Proposed Dispositions

### Disposition A: Apply as Submitted — Add FHIRPath Invariants to Composition Profiles

Add FHIRPath invariants to the relevant C-CDA on FHIR Composition profiles (e.g., the Referral Note Composition) that enforce the mutual exclusion constraints:

1. **Invariant for CONF:1198-29102:** The Composition must include either a section with code for Assessment and Plan, OR both a section with code for Assessment and a section with code for Plan of Treatment.
2. **Invariant for CONF:1198-29103:** The Composition must NOT include an Assessment and Plan section when either an Assessment section or a Plan of Treatment section is present.

These would be expressed as FHIRPath constraints on `Composition.section` slices. This directly addresses the reporter's concern and maintains fidelity with C-CDA R2.1 conformance.

**Pros:** Full parity with C-CDA constraints; catches non-conformant documents at validation time.
**Cons:** FHIRPath invariants expressing "either A, or (B and C), but not both" are moderately complex; requires careful testing. May also need coordination with sibling tickets (FHIR-10333, FHIR-10335) to handle similar constraints in other document types.

### Disposition B: Defer — Coordinate with Sibling Tickets for a Unified Approach

Defer this ticket to be resolved as part of a broader effort that addresses all the related "complex constraints not addressed" tickets (FHIR-10332 through FHIR-10335) together. This would allow the workgroup to design a consistent pattern for expressing mutual exclusion and conditional section requirements across all C-CDA on FHIR document types.

**Pros:** Ensures consistency across document types; avoids piecemeal fixes; FHIR-10332 and FHIR-10334 are already marked Duplicate suggesting some consolidation has begun.
**Cons:** Further delays resolution of a ticket originally filed in 2016; the constraint gap has persisted through STU1.2.

### Disposition C: Add Documentation / Guidance Only (No Profile Change)

Rather than adding formal invariants, add narrative guidance or a "differences from C-CDA" section to the IG noting that these mutual exclusion constraints are not formally enforced in the FHIR profiles and should be handled by implementers. This acknowledges the gap without adding complex FHIRPath invariants.

**Pros:** Low implementation effort; avoids complex invariant logic.
**Cons:** Does not actually enforce the constraints; may lead to non-conformant documents passing validation; does not fulfill the original change request.

## Recommendation

**Disposition A: Apply as Submitted** is recommended. The constraints cited (CONF:1198-29102 and CONF:1198-29103) are mandatory ("SHALL" / "SHALL NOT") in C-CDA R2.1, and the purpose of C-CDA on FHIR is to faithfully represent C-CDA document requirements in FHIR profiles. The ticket has been open since 2016 and was explicitly re-opened post-STU1.2 publication by lynn_laakso, indicating ongoing workgroup awareness. Implementing FHIRPath invariants on the Referral Note Composition profile's section slices is the most direct and standards-compliant resolution. The workgroup should also consider resolving the still-open sibling tickets (FHIR-10333, FHIR-10335) in the same cycle to avoid repeated revisits.
