## Description

The reporter identifies that C-CDA on FHIR Composition profiles use `fixedCodableConcept` to constrain code elements. This approach fixes the entire `CodeableConcept` datatype, which prevents implementers from including optional sub-elements such as `display`, `text`, or additional `coding` entries. The reporter argues that profiles should instead constrain at the leaf level—fixing `coding.system` and `coding.code` individually—to allow the rest of the CodeableConcept to remain open for optional content.

The issue affects all C-CDA on FHIR Composition structure definitions except Transfer Summary. This specific ticket references the **History and Physical** composition (`ccda-history-and-physical-composition`).

## Analysis

### Core Issue

The use of `fixedCodableConcept` in C-CDA on FHIR Composition profiles is overly restrictive. When a `CodeableConcept` is fixed, **every** sub-element of the datatype must match exactly—including `coding.display`, `text`, and any additional `coding` entries. This means:

1. **No display names**: Implementers cannot include human-readable `display` values alongside the fixed code/system, reducing interoperability and readability.
2. **No additional codings**: Implementers cannot provide translations or alternative code system representations.
3. **No text**: The `CodeableConcept.text` element cannot be populated.

The FHIR profiling best practice is to use **patternCodableConcept** (or fix individual leaf elements like `coding.system` and `coding.code`) rather than `fixedCodableConcept`. This constrains the required code/system while leaving optional elements open. This approach is now well-established in the FHIR profiling community and is the standard recommendation in FHIR R4+.

### Scope

This ticket specifically targets the History and Physical composition. The identical issue for Referral Note (FHIR-10314) and Consultation Note (FHIR-10315) should ideally be addressed together. The reporter notes that Transfer Summary is already exempt from this issue.

### Age and Re-opening

Originally filed in 2016, the ticket was re-opened in March 2024 post STU1.2, indicating it remains unresolved and relevant for future C-CDA on FHIR updates.

## Proposed Dispositions

### Disposition A: Persuasive (Recommended)

**Action**: Accept the change request. Replace `fixedCodableConcept` constraints with `patternCodableConcept` (or leaf-level `fixedUri`/`fixedCode` on `coding.system` and `coding.code`) across the History and Physical composition profile, and coordinate with FHIR-10314 and FHIR-10315 to apply the same change to the Referral Note and Consultation Note compositions.

**Rationale**: The request aligns with established FHIR profiling best practices. Using `pattern` instead of `fixed` for CodeableConcept elements is now the standard approach in FHIR R4+ profiling. The change improves implementability by allowing optional elements like `display` and `text` while still enforcing the required code and system. The ticket has been open since 2016 and was re-opened in 2024, demonstrating sustained relevance. All three sibling tickets should be resolved together for consistency.

### Disposition B: Persuasive with Modification

**Action**: Accept the change but apply it globally across **all** C-CDA on FHIR Composition profiles (not just History and Physical) as a single coordinated change, consolidating FHIR-10314, FHIR-10315, and FHIR-10316 into one resolution.

**Rationale**: Since the identical issue affects multiple compositions (all except Transfer Summary, per the reporter), it is more efficient to address this systematically across the entire C-CDA on FHIR specification rather than piecemeal. This avoids inconsistency and reduces the number of tickets to track. However, this broader scope may require additional review time.

### Disposition C: Not Persuasive

**Action**: Reject the change request, keeping `fixedCodableConcept` as-is.

**Rationale**: The `fixedCodableConcept` approach ensures strict validation and prevents any variation in the code elements. Some implementers may prefer this strictness to guarantee uniformity. However, this position contradicts current FHIR profiling guidance and would continue to prevent the use of `display` and other optional elements, which is a significant practical limitation.

## Recommendation

**Disposition A: Persuasive** — The request is well-founded, aligns with current FHIR profiling best practices (using `pattern` over `fixed` for CodeableConcept), and addresses a real implementability barrier. The work group should accept this change for the History and Physical composition and coordinate resolution of the sibling tickets (FHIR-10314, FHIR-10315) at the same time to ensure consistency across C-CDA on FHIR compositions.
