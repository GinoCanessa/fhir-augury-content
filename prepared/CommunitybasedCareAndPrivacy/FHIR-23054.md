## Summary

This change request questions whether the eLTSS Implementation Guide should use `ServiceRequest.occurrencePeriod` rather than `ServiceRequest.occurrenceTiming.boundsPeriod` to capture service start and end dates. The reporter observed that using `occurrenceTiming` for simple date ranges may be unnecessarily complex when `occurrencePeriod` could suffice, and that `occurrenceTiming` is only justified when additional scheduling details (interval, quantity, day-of-week, time-of-day) are also needed. The ticket was originally resolved as "Considered for Future Use" in 2020 with a vote of 12-0-1, noting that `ServiceRequest.occurrence` is 0..1 and `occurrenceTiming` was chosen to accommodate both date ranges and scheduling details in a single element. The ticket was subsequently re-opened after STU2 publication for further discussion. Two options have been proposed: (1) change the cardinality of `ServiceRequest.occurrence` from 0..1 to 0..* to allow both `occurrencePeriod` and `occurrenceTiming`, or (2) add a new element such as `ServiceRequest.period` for start/end dates. The cross-references point to the FHIR elements `ServiceRequest.occurrence`, `ServiceRequest.occurrencePeriod`, and `ServiceRequest.period`.

## Details

**Description:**

In reference to FHIR R4 Resource Elements:
- `ServiceRequest > occurrenceTiming > boundsPeriod -> start`

Question: Should this be using `occurrencePeriod` instead of `occurrenceTiming`? `occurrenceTiming` makes sense only if it is meant to be used for other service schedule and quantity information (separate comment).

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Adopt Option 1 from the comments: request that the base FHIR specification change the cardinality of `ServiceRequest.occurrence[x]` from 0..1 to 0..*, enabling the eLTSS profile to use `ServiceRequest.occurrencePeriod` for simple service start/end dates and `ServiceRequest.occurrenceTiming` for complex scheduling details (interval, quantity, day-of-week). Update the eLTSS profile mappings so that "Service Start Date" and "Service End Date" map to `ServiceRequest.occurrencePeriod.start` and `ServiceRequest.occurrencePeriod.end`, respectively, while retaining `occurrenceTiming` for scheduling information.
#### Justification
This approach cleanly separates the two distinct concepts — a date range for when a service is authorized versus a recurring schedule for how the service is delivered. Using `occurrencePeriod` for start/end dates is more intuitive for implementers who only need to express a simple date range and do not require the complexity of the Timing datatype. However, this requires a core specification change to `ServiceRequest.occurrence[x]` cardinality, which would need coordination with the Orders and Observations work group.

---

### Disposition B: Alternative Approach
#### Proposal
Retain the current STU2 approach of using `ServiceRequest.occurrenceTiming.boundsPeriod` for service start and end dates, and enhance the eLTSS IG guidance to more clearly document this pattern. Add explicit narrative and examples showing that when only start/end dates are needed without complex scheduling, implementers should populate only `occurrenceTiming.boundsPeriod.start` and `occurrenceTiming.boundsPeriod.end` and may leave other Timing sub-elements empty. This avoids any dependency on a core specification change.
#### Justification
The STU2 ballot already established this pattern with a strong vote (12-0-1), and `occurrenceTiming.boundsPeriod` is technically capable of representing service start and end dates. Improved documentation and examples would address implementer confusion without requiring a breaking change to the base specification or coordination across work groups. This is the lowest-risk path and can be accomplished entirely within the eLTSS IG.

---

### Disposition C: Decline
#### Proposal
Decline the change request on the grounds that the current modeling using `occurrenceTiming.boundsPeriod` is functionally correct and was already deliberated and voted on (12-0-1) during the STU2 cycle. The concern is an editorial/usability issue rather than a functional gap.
#### Justification
The existing approach does capture service start and end dates correctly via `occurrenceTiming.boundsPeriod`. Changing the cardinality of a core FHIR element (`ServiceRequest.occurrence[x]`) is a significant undertaking with broad impact beyond eLTSS. The ticket has been open since 2019 and was previously resolved as "Considered for Future Use," suggesting the community did not view it as critical. Declining preserves stability for existing implementations that have already adopted the STU2 pattern.

---

### Recommendation
**Recommended disposition:** B

The alternative approach of retaining `occurrenceTiming.boundsPeriod` with enhanced documentation strikes the best balance between addressing implementer concerns and maintaining stability. The current pattern is technically correct and was adopted through a strong consensus vote. The core issue raised — that using `occurrenceTiming` for simple date ranges feels unnecessarily complex — is best resolved through clearer guidance and examples in the IG rather than through a core specification change that would require cross-work-group coordination and could introduce breaking changes. Option 1 (changing `occurrence[x]` cardinality) would be a significant core spec change with broad implications, and Option 2 (adding a new element) would introduce redundancy. Better documentation within the eLTSS IG is achievable in the near term, addresses the usability concern, and preserves backward compatibility with STU2 implementations.
