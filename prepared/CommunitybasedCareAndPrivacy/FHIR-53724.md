## Problem Statement

The Consent resource's former `.policyRule` element has been removed but is still referenced in the narrative text (specifically in the "Optimal Searching" section). The reporter identifies that:

1. **Stale reference**: The `policyRule` element name appears in the Optimal Searching section but the element no longer exists. It should refer to `.policyBasis` instead.
2. **Unclear element migration**: The old `policyRule` was restructured into three new elements — `.regulatoryBasis`, `.policyBasis`, and `.policyText` — but the narrative does not clearly explain where the original `policyRule` functionality went.
3. **Missing capability for patient-specific external rules**: There is no clear way to point at patient-specific computable rules expressed in external standards languages like XACML or ODRL, for implementers who do not want to use the `.provision` element. The `.policyBasis` element is intended for the organization's non-patient-specific backing policy, not for patient-specific rules.

This ticket was filed while working on [FHIR-48988](https://jira.hl7.org/browse/FHIR-48988) (Consent using XACML example), which revealed these issues.

## Analysis

This ticket raises two distinct issues:

**Issue 1 — Stale narrative references (straightforward fix):** The Consent resource narrative still references `policyRule` which no longer exists as an element. The Optimal Searching section needs updating to reference `.policyBasis`. This is corroborated by FHIR-55983, which independently identified similar stale references in the `.provision` definition. This is an editorial correction.

**Issue 2 — Missing mechanism for patient-specific external rules (design question):** The reporter argues that `.policyBasis` is semantically intended for the organization-level backing policy (not patient-specific), yet there is no dedicated element for pointing at patient-specific computable rules in external standards (XACML, ODRL). The comment on FHIR-48988 elaborates: reusing `.policyBasis.url` for patient-specific XACML policies "breaks the usefulness of .policyBasis to find all consents based on 'last years backing policy'."

This second issue is partially addressed by FHIR-46021 (Applied/Persuasive), which added a Permission-based alternative to `.provision`. However, the reporter's concern is broader — covering non-FHIR standards like XACML and ODRL, not just FHIR Permission resources.

The cluster of related tickets (FHIR-53886, FHIR-54155, FHIR-55983) suggests widespread community confusion about the Consent resource's policy-related elements after the R4→R6 restructuring.

## Proposed Dispositions

### Disposition A: Persuasive — Fix narrative references and add guidance for external computable policies

**Action:**
1. Update all narrative references from `policyRule` to `.policyBasis` in the Consent resource text (Optimal Searching section and any other occurrences).
2. Add guidance in the Consent resource notes explaining how to reference patient-specific computable rules in external standards (XACML, ODRL) — e.g., via the Permission-based alternative from FHIR-46021, or by clarifying the intended usage pattern for external standards.
3. Consider whether a new element or extension is needed for patient-specific external policy references, or whether existing elements (policyBasis, the Permission alternative) are sufficient with better documentation.

**Rationale:** The stale narrative references are clear editorial errors. The request for external policy guidance aligns with FHIR-46021's direction and addresses the overlapping confusion seen in FHIR-53886, FHIR-54155, and FHIR-55983. Addressing both issues provides the most value.

### Disposition B: Persuasive with Modification — Fix narrative references only; defer external policy mechanism

**Action:**
1. Update all narrative references from `policyRule` to `.policyBasis` (and verify `provision` definition references per FHIR-55983).
2. For the external computable policy request, add a note referencing FHIR-46021's Permission-based approach and suggest that XACML/ODRL rules can be referenced via that mechanism. Defer any new element to a future cycle.

**Rationale:** The narrative fix is unambiguous. The external rules question is a broader design topic being addressed across multiple tickets and should be considered holistically rather than in isolation.

### Disposition C: Persuasive with Modification — Narrow to narrative correction; split out external policy as new tracker

**Action:**
1. Fix the stale `policyRule` references in the Consent narrative.
2. Create a new tracker item specifically for the "patient-specific external computable policy" mechanism question, allowing it to be discussed alongside FHIR-53886, FHIR-54155, and FHIR-48988.

**Rationale:** Separating the editorial fix from the design question allows the editorial change to proceed immediately while the design question gets proper analysis across the related ticket cluster.

## Recommendation

**Disposition A: Persuasive** — The stale `policyRule` narrative references are clearly incorrect and should be fixed. Additionally, the question of how to reference patient-specific external computable policies (XACML, ODRL) is a real gap that multiple implementers have encountered. With FHIR-46021 already applied, there is an established direction (Permission-based alternative) that can be documented and extended. Addressing both concerns in this ticket is efficient given the extensive overlap with the related ticket cluster (FHIR-53886, FHIR-54155, FHIR-55983), and the reporter's detailed analysis provides a strong foundation for the resolution. The "Highest" priority reflects the impact of incorrect narrative on implementers trying to understand the Consent resource's post-R4 structure.
