## Description

The submitter questions the need for the custom `patientId` search parameter defined in the Scalable Consent Management IG ([SearchParameter-fast-consent-patientId](https://hl7.org/fhir/us/consent-management/2026Jan/SearchParameter-fast-consent-patientId.html)). They propose using or enhancing the native FHIR chained search parameter `patient:identifier` (or `patient.identifier`) instead of creating a new search parameter.

The submitter also notes an inconsistency: the list of search parameters used for searching consents ([technical.html#searching-for-consents](https://hl7.org/fhir/us/consent-management/2026Jan/technical.html#searching-for-consents)) does not align with the list of search parameters for consent subscriptions ([technical.html#consent-subscriptions](https://hl7.org/fhir/us/consent-management/2026Jan/technical.html#consent-subscriptions)). Harmonizing these by using native FHIR search parameters would address both issues.

## Resolution Description

The resolution states that the organization ID and patient ID search parameters will be removed because FHIR-I has already indicated that defining search parameters that cross resource boundaries is not allowed. The IG will instead indicate that search for consents based on patient identifier and organization identifiers is required, without defining an improper search parameter.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Agree with Current Resolution)

**Action:** Accept the Persuasive resolution as stated. Remove the `patientId` (and `organizationId`) custom search parameters from the IG. Replace them with guidance that servers SHALL support chained search using native FHIR parameters (e.g., `patient:identifier`) and update the IG narrative to harmonize consent search and subscription parameter lists.

**Rationale:** The submitter's feedback is technically sound and aligns with FHIR-I guidance that search parameters should not cross resource boundaries. The native chained search parameter `patient:identifier` already provides the needed functionality. Removing the custom parameters simplifies implementation, reduces IG-specific complexity, and brings the specification into conformance with core FHIR search conventions. The resolution description already captures this intent. This also aligns with companion tickets FHIR-53778 and FHIR-53781, enabling a consistent approach across all three.

**Impact:** Requires removing the `patientId` and `organizationId` SearchParameter resources from the IG, updating the "Searching for Consents" and "Consent Subscriptions" narrative sections, and adding guidance on required chained search support. Implementers who already built against these custom parameters would need to migrate to chained search, but this is a simplification.

---

### Disposition B: Accept as Persuasive with Modification — Keep as Convenience Parameters with Deprecation Path

**Action:** Accept the comment as Persuasive with Modification. Mark the `patientId` search parameter as deprecated rather than immediately removing it. Add the chained search parameter `patient:identifier` as the primary recommended approach and set a deprecation timeline (e.g., removal in the next ballot cycle).

**Rationale:** While the submitter's technical argument is valid and the FHIR-I guidance is clear, an abrupt removal could disrupt implementers who have already adopted the custom search parameter from the January 2026 ballot. A deprecation period provides a migration path while still converging on the correct approach.

**Impact:** Requires marking the SearchParameter as deprecated in the IG, adding chained search guidance, and maintaining both approaches temporarily. Adds short-term maintenance burden but reduces implementer disruption.

---

### Disposition C: Accept as Not Persuasive — Retain Custom Search Parameter

**Action:** Reject the comment. Keep the `patientId` search parameter as defined.

**Rationale:** The custom `patientId` search parameter provides a simpler, more direct query interface than chained search. Chained search parameters impose a higher implementation burden on servers and may not be uniformly supported. The custom parameter provides a single, well-defined search interface that is easier to test and validate.

**Impact:** No changes to the IG. However, this would conflict with FHIR-I guidance on cross-resource-boundary search parameters and would leave the inconsistency between consent search and subscription parameter lists unresolved. It would also contradict the resolutions of companion tickets FHIR-53778 and FHIR-53781.

---

## Recommendation

**Disposition A: Accept as Persuasive** is recommended.

The submitter correctly identifies that the custom `patientId` search parameter is unnecessary given native FHIR chained search capabilities, and the FHIR-I ruling that cross-resource-boundary search parameters are not allowed makes this a compliance issue, not just a design preference. The resolution description already captures the correct approach. Accepting this disposition aligns with companion tickets FHIR-53778 and FHIR-53781, enabling a clean, consistent update across the IG. The change simplifies the specification and reduces the implementation surface for adopters.
