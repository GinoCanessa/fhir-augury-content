## Summary

John Moehrke raises an inconsistency in how the Scalable Consent Management IG profiles AuditEvent across its three cases. In Case 1, the IG hints that AuditEvent is the solution but does not include the profiling that IHE-BALP provides. In Case 2, IHE-BALP is simply referenced. In Case 3, an operation is defined AND AuditEvent is profiled from BALP. The reporter questions why Case 3 receives a US-Core AuditEvent profile when Cases 1 and 2 do not, and argues that Case 3 does not actually benefit from the additional profiling. The recommendation is to remove the AuditEvent profile from Case 3 for consistency.

## Details

### Description

The reporter identifies three cases in the Scalable Consent Management specification with inconsistent approaches to AuditEvent profiling:

- **Case 1:** Hints that AuditEvent is the solution, but does not include the profiling that IHE-BALP provides.
- **Case 2:** Simply references IHE-BALP.
- **Case 3:** Defines an operation AND profiles AuditEvent from BALP.

The core question is: why does Case 3 receive a US-Core AuditEvent profile when Cases 1 and 2 do not? The reporter asserts that Case 3 does not particularly benefit from this profiling and recommends removing it.

## Proposed Dispositions

### Disposition A: Persuasive — Remove Case 3 AuditEvent Profile

**Action:** Accept the recommendation to remove the US-Core AuditEvent profile from Case 3 and align all three cases with a consistent approach to IHE-BALP referencing.

**Rationale:** The reporter makes a valid point about inconsistency across the three cases. If Cases 1 and 2 rely on IHE-BALP without additional US-Core profiling, Case 3 should follow the same pattern. This aligns with the related ticket FHIR-54692 (Persuasive), which argues that AuditEvent compliance should be at the search/read boundary via IHE-BALP's ITI-81, not through additional profiling or operations. Removing the profile simplifies the IG and avoids implying that Case 3 has requirements that Cases 1 and 2 do not.

**Changes required:** Remove the FastConsentAuditEvent profile from Case 3. Ensure all three cases consistently reference IHE-BALP profiles (CRUD, Consent Authorized Decision, Privacy Disclosure) as the source for AuditEvent conformance.

### Disposition B: Persuasive with Modification — Extend Profiling to All Cases

**Action:** Rather than removing the Case 3 profile, add appropriate IHE-BALP-based AuditEvent profiles to Cases 1 and 2 as well, achieving consistency in the opposite direction.

**Rationale:** AuditEvent profiling adds value by making expectations explicit and testable. The problem the reporter identifies is inconsistency, which can be resolved by profiling all three cases rather than profiling none. Case 1 should reference the appropriate BALP CRUD AuditEvent profiles. Case 2 should reference the BALP Consent Authorized Decision and Privacy Disclosure profiles (per FHIR-54650's distinction). Case 3's existing profile can be retained or adjusted.

**Changes required:** Add AuditEvent profile references for Cases 1 and 2 based on appropriate IHE-BALP patterns. Review Case 3's profile to ensure it uses the correct BALP base profile. Update documentation to clearly explain which BALP profile applies to which case.

### Disposition C: Not Persuasive — Maintain Current Approach

**Action:** Keep the current structure where only Case 3 includes an AuditEvent profile.

**Rationale:** Case 3 defines a specific operation ($record-disclosure) that creates an AuditEvent, making profiling necessary to specify the expected output. Cases 1 and 2 describe scenarios where existing IHE-BALP patterns apply without the IG needing to re-profile them. The asymmetry reflects a genuine difference: Case 3 introduces new functionality (an operation) that requires explicit profiling, while Cases 1 and 2 reference existing patterns.

**Changes required:** None. Optionally, add explanatory text to the IG clarifying why Case 3 is profiled while Cases 1 and 2 are not.

## Recommendation

**Disposition A: Persuasive — Remove Case 3 AuditEvent Profile**

Disposition A is recommended because it directly addresses the reporter's valid observation about inconsistency and aligns with the emerging consensus across related tickets (FHIR-54692 resolved Persuasive, FHIR-54682 resolved Persuasive). The reporter, John Moehrke, is the author of IHE-BALP and a recognized authority on AuditEvent profiling patterns. His argument that Case 3 does not benefit from the additional profiling carries significant domain expertise. Furthermore, FHIR-54692 argues convincingly that AuditEvent compliance should be defined at the search/read boundary via ITI-81, not through creation-time profiling — removing the Case 3 profile is consistent with that approach. The related ticket FHIR-54682 also questions requiring the Record Disclosure Operation when access control is out of scope, further undermining the rationale for profiling in Case 3.
