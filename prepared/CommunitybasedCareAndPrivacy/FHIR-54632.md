# FHIR-54632: Consent.provision.documentType valueSet

## Summary

**Ticket:** [FHIR-54632](https://jira.hl7.org/browse/FHIR-54632)
**Status:** Triaged | **Type:** Change Request | **Priority:** Medium
**Reporter:** John Moehrke | **Assignee:** Unassigned
**Work Group:** Community-Based Care and Privacy
**Specification:** FHIR Core (FHIR)
**Created:** 2026-01-17 | **Updated:** 2026-01-17

The submitter asks why a special ValueSet (`consent-content-class` / `ConsentContentClass`) is defined specifically for `Consent.provision.documentType` (and Contract), and why it differs from the ValueSet used by `Composition.type` and `DocumentReference.type`. The implication is that document type vocabularies should be consistent across the specification, especially now that R5 split the old `Consent.provision.class` into separate `documentType` and `resourceType` elements.

## Analysis

The core question is straightforward: `Consent.provision.documentType` uses the `consent-content-class` ValueSet, while `Composition.type` and `DocumentReference.type` use `doc-typecodes`. The reporter (John Moehrke) previously drove the alignment of `DocumentReference.type` with `Composition.type` via FHIR-28078, and is now asking the logical follow-up — why doesn't Consent also align?

The `consent-content-class` ValueSet has a troubled history. It was designed to cover all possible content classes for consent scoping (format codes, resource types, document types), but this breadth has caused expansion failures (documented in the 2017 Zulip thread) and ongoing implementer confusion (FHIR-56065). Critically, the R5 restructuring that split `provision.class` into separate `documentType` and `resourceType` elements (per FHIR-39394) removed the original justification for the broad ValueSet — `documentType` now specifically targets document types, not all content classes.

Since `documentType` is semantically scoped to document types, binding it to the same ValueSet used for document typing elsewhere in the specification (`doc-typecodes`) is both logically consistent and practically beneficial (eliminates expansion issues, reduces implementer confusion).

---

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Align `Consent.provision.documentType` to use the `doc-typecodes` ValueSet (the same ValueSet bound to `Composition.type` and `DocumentReference.type`). Retire or repurpose the `consent-content-class` ValueSet.

**Rationale:** The R5 split of `provision.class` into `documentType` and `resourceType` means `documentType` is now specifically about document types. Using a different, broader ValueSet for the same semantic concept across the specification is inconsistent and confusing. The same reporter (John Moehrke) successfully drove this exact alignment for `DocumentReference.type` via FHIR-28078. The `consent-content-class` ValueSet has documented expansion problems and its breadth is no longer justified for the narrower `documentType` element.

### Disposition B: Persuasive with Modification
**Action:** Replace the `consent-content-class` binding on `Consent.provision.documentType` with `doc-typecodes`, but retain a broader binding strength (e.g., extensible) to allow consent-specific extensions. Review whether `Contract` elements referencing `consent-content-class` should also be updated. Document the rationale for the change.

**Rationale:** The alignment is warranted, but given that existing implementations may depend on the current `consent-content-class` binding, the transition should be handled carefully. An extensible binding to `doc-typecodes` allows implementers who need codes beyond the standard document type set to continue using them while guiding the ecosystem toward consistency. Contract should also be evaluated as part of this change.

### Disposition C: Not Persuasive
**Action:** No change. Retain the `consent-content-class` ValueSet binding on `Consent.provision.documentType`.

**Rationale:** The `consent-content-class` ValueSet, while overlapping with `doc-typecodes`, was intentionally designed for the consent context and may include content categories (e.g., IHE format codes) that are relevant for consent scoping but not for document typing in Composition or DocumentReference. Changing the binding could break existing implementations. The expansion issues are implementation problems, not specification problems.

---

## Recommendation

**Disposition A: Persuasive**

The case for alignment is compelling. The R5 restructuring into separate `documentType` and `resourceType` elements removed the original rationale for the broad `consent-content-class` ValueSet on this element. The `documentType` element is semantically identical to document type concepts in `Composition` and `DocumentReference` and should use the same vocabulary. This follows the precedent set by FHIR-28078 (same reporter), which aligned `DocumentReference.type` with `Composition.type`. The `consent-content-class` ValueSet has documented expansion problems dating back to 2017, and FHIR-56065 shows implementers are actively working around it. Aligning to `doc-typecodes` simplifies the specification, eliminates a long-standing pain point, and improves cross-resource consistency.
