## Description

The FAST Consent SubscriptionTopic defines the `resourceTrigger` `supportedInteraction` of `delete`. However, there is no guidance regarding a use case or implementation details for a consent management server deleting a Consent resource from its repository.

The reporter recommends either:

1. **Removing** the FAST Consent SubscriptionTopic `resourceTrigger` `supportedInteraction` of `delete`, or
2. **Adding** specific guidance via a defined use case and implementation notes on how the deletion of a Consent resource is handled by the consent management server, including updates to the [Implementation Notes - Exchanging Consent](https://build.fhir.org/ig/HL7/fhir-consent-management/implementation_notes.html#exchanging-consent) section related to Subscriptions.

## Resolution Description

> We will remove 'delete' as a supported interaction from the subscription topic.

## Analysis

This ticket identifies a gap in the Scalable Consent Management IG where the FAST Consent SubscriptionTopic declares `delete` as a `supportedInteraction` in its `resourceTrigger`, but the IG provides no use case, implementation guidance, or behavioral specification for what happens when a Consent resource is deleted on the consent management server.

The companion ticket **FHIR-53813** strengthens this concern by documenting an explicit contradiction: the IG's Technical Specification states "A revoked consent SHALL NOT be deleted from the consent management system," and the Consent Administrative Server Capabilities CapabilityStatement does not declare support for the delete interaction. Yet the SubscriptionTopic implies delete is a supported trigger.

The resolution description indicates the work group intends to remove `delete` as a supported interaction from the subscription topic, which is the simpler of the two remediation paths proposed by the reporter. This is consistent with the existing IG language that discourages or prohibits deletion of Consent resources.

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Remove `delete` from SubscriptionTopic (Recommended)

Accept the ticket as persuasive and remove the `delete` value from the FAST Consent SubscriptionTopic `resourceTrigger` `supportedInteraction`. This aligns with:
- The resolution description already stated by the work group.
- The existing IG language prohibiting deletion of revoked consents.
- The Consent Administrative Server Capabilities CapabilityStatement which does not declare delete support.
- The companion ticket FHIR-53813 which documents the same mismatch.

No additional use case or implementation guidance for delete would be needed since the interaction would no longer be declared.

### Disposition B: Accept as Persuasive with Modification — Add Delete Guidance Instead

Accept the ticket but choose the alternative path: retain `delete` in the SubscriptionTopic and add comprehensive guidance including:
- A defined use case for when deletion of a Consent resource is appropriate.
- Implementation notes in the Exchanging Consent section describing server behavior on delete.
- Updates to the CapabilityStatement to declare delete support.
- Reconciliation of the "SHALL NOT be deleted" language for revoked consents (perhaps scoping the prohibition to revoked consents only).

This approach is more complex and would require resolving the contradiction documented in FHIR-53813.

### Disposition C: Not Persuasive — No Change Needed

Reject the ticket on the grounds that the `delete` interaction in the SubscriptionTopic is aspirational or forward-looking, and implementers can ignore it if their server does not support delete. This disposition is weak because it leaves a documented contradiction in the IG and provides no implementer guidance.

## Recommendation

**Disposition A: Accept as Persuasive — Remove `delete` from SubscriptionTopic.** This is the simplest, most internally consistent resolution. It aligns with the work group's stated resolution, eliminates the contradiction with the existing prohibition on deleting revoked consents and the CapabilityStatement, and is consistent with the closely related FHIR-53813. Coordinate resolution with FHIR-53813 to avoid duplicate or conflicting changes.
