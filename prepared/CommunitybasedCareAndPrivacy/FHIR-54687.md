## Summary

The submitter (John Moehrke) reports that the Scalable Consent Management Implementation Guide uses inconsistent terminology when defining system actors. The IG's page on "Systems" defines two actor types—**Client** (systems that gather and file consents) and **Server** (consent administration services that store consents and handle consent requests)—but these definitions differ from actor terminology used elsewhere in the specification. The reporter notes confusion about which set of definitions to follow.

The proposed resolution (Persuasive) is to establish a canonical set of actor and system definitions on the Use Cases page and then audit the entire specification for consistent usage of those names.

---

## Details

### Description

The reporter identifies that the IG defines two system types on one page:

- **Client** systems — responsible for gathering and filing consents.
- **Server** systems — consent administration services that store consents and handle requests for consents.

However, these definitions are inconsistent with actor terminology used on other pages of the IG, creating ambiguity about which definitions are authoritative.

### Resolution Description

> "We are going to list the set of actors and systems and provide definitions of them on the Use Cases page. We will then scour the specification to make sure that we consistently use these names."

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

Accept the ticket as Persuasive. The work group should:

1. Define a canonical list of actors and systems on the **Use Cases** page of the Scalable Consent Management IG, including clear definitions for roles such as "Client" (consent gatherer/filer), "Server" (consent administration service), and any other actor roles referenced in the IG (e.g., data subject, data custodian, policy administrator).
2. Perform a comprehensive audit of all pages in the IG to replace ad-hoc or inconsistent actor/system terminology with the canonical names established in step 1.
3. Add a cross-reference or glossary section so readers can quickly locate the authoritative definitions.

**Rationale:** The resolution aligns with what was already proposed by the group—centralizing definitions and enforcing consistency. This is a straightforward editorial improvement that will materially reduce reader confusion.

### Disposition B: Alternative — Accept with Expanded Scope

Accept the ticket as Persuasive with an expanded scope:

1. In addition to the actions in Disposition A, align actor definitions with the broader FHIR ecosystem terminology (e.g., CapabilityStatement actor types, IHE actor naming conventions).
2. Consider adding a formal **ActorDefinition** resource for each actor defined in the IG, as supported by the FHIR R5 specification, to provide machine-readable actor definitions alongside the narrative.

**Rationale:** Aligning with standard FHIR actor-definition patterns would improve interoperability and conformance-testing support, though it increases the scope of work beyond the original request.

### Disposition C: Decline (Not Persuasive)

Decline the ticket as Not Persuasive. The current page-level definitions are sufficient context for implementers, and the minor terminology variations do not create genuine interoperability issues.

**Rationale:** This is a documentation-style concern rather than a technical defect. Implementers can resolve any ambiguity from context. However, this position is weak given that the reporter—an active specification contributor—found the inconsistency genuinely confusing.

---

## Recommendation

**Disposition A: Accept (Persuasive)**

The ticket identifies a real usability problem in the Scalable Consent Management IG: inconsistent actor terminology across pages confuses readers about authoritative definitions. The proposed resolution—centralizing definitions on the Use Cases page and auditing the IG for consistency—is well-scoped, actionable, and already accepted by the group. This is a low-risk editorial improvement that directly benefits implementers. Disposition B's expanded scope (ActorDefinition resources, IHE alignment) is valuable but can be addressed separately; Disposition C's argument that context suffices is undermined by the reporter's demonstrated confusion. Accepting as Persuasive is the appropriate path.
