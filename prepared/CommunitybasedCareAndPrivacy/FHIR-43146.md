## Description

The `clinicalImpression` extension (basedOnClinicalImpression) in the PACIO Re-Assessment Timepoints IG is defined with a context of `element`, meaning it is allowed on all elements in all resources. The reporter (Lloyd McKenzie) requests that the extension context be constrained to the specific resource(s) or element(s) where it is actually intended to be used, rather than being universally available.

## Analysis

This is a straightforward best-practice request. Extensions defined with a context of `element` (i.e., any element on any resource) are overly permissive and undermine the purpose of context declarations. The basedOnClinicalImpression extension is clearly intended for use on specific resources (likely Encounter or related assessment resources in the Re-Assessment Timepoints IG), not on arbitrary elements.

The reporter, Lloyd McKenzie, filed a series of similar tickets (FHIR-43147 through FHIR-43152) against multiple IGs in November 2023, all requesting tighter extension contexts. Notably, FHIR-43109 (CARIN DIC) was resolved as **Persuasive**, confirming community consensus that extensions should declare specific contexts. FHIR-43108 (US Core) was retracted, but only because the specific extension in question had different circumstances.

A prior ticket, FHIR-35123, already addressed the *purpose and clarity* of this same ClinicalImpression extension and was resolved as **Persuasive** during the STU1 ballot. That resolution focused on improving documentation and examples but did not address the overly broad extension context, which is the specific concern raised here.

## Proposed Dispositions

### Disposition A: Persuasive (Recommended)

**Rationale:** The request is technically sound and aligns with FHIR authoring best practices. Extension contexts should reflect actual intended usage. The basedOnClinicalImpression extension is designed for use on Encounter (or a small set of assessment-related resources) in the Re-Assessment Timepoints IG, not on arbitrary elements across all resources. Constraining the context to the specific resource type(s) and element(s) where the extension is profiled improves IG quality, tooling validation, and implementer clarity. This is consistent with FHIR-43109 (resolved Persuasive) and the broader pattern of community acceptance for these requests.

**Action:** Update the basedOnClinicalImpression extension definition to declare its context as the specific FHIR element(s) or resource(s) where it is intended (e.g., `Encounter` or the specific profiled resource), replacing the current overly broad `element` context.

### Disposition B: Persuasive with Modification

**Rationale:** Accept the request but define the context more broadly than a single resource type, in case the extension is intended for use across multiple resource types in future IG versions. For instance, the context could be set to a small enumerated list (e.g., `Encounter`, `Observation`, `ClinicalImpression`) rather than a single resource, preserving some flexibility while still being far more specific than `element`.

**Action:** Update the extension context to an enumerated list of resource types or elements where the extension is actually used or anticipated to be used in the IG.

### Disposition C: Not Persuasive

**Rationale:** The extension may intentionally use a broad context of `element` to allow future use in unforeseen contexts without requiring an IG update. Narrowing the context could limit extensibility for implementers who need the extension on resources not currently anticipated. The current broad context does not cause technical errors — it merely permits wider usage.

**Action:** No change. Retain the `element` context for maximum flexibility.

## Recommendation

**Disposition A: Persuasive.** The request is well-founded, consistent with FHIR best practices, and supported by precedent (FHIR-43109 resolved Persuasive). Overly broad extension contexts weaken validation and create confusion for implementers. The basedOnClinicalImpression extension has a clear, specific purpose within the Re-Assessment Timepoints IG and its context should reflect that.
