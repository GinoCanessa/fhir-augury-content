## Description

The submitter notes that the Consent resource model changed significantly between R4 and R5/R6. In R4, the `permit`/`deny` decision was placed on each **provision**, with rich contextual information (time, data, purpose, actor) defining the scope of each decision. In the current model, there is a single overall `decision` element at the root of Consent (`permit` or `deny`), with provisions serving as exceptions.

The submitter asks: **"If I put 'permit' here, what am I permitting?"** — questioning whether the root decision applies to the `policyBasis`, `policyText`, or `regulatoryBasis`, and whether there is additional context (data, treatment, time) that scopes the overall decision. They request clarification of the new model's rationale and suggest adding explanatory text to the resource description.

## Analysis

This ticket reflects a recurring theme across multiple Jira submissions and Zulip discussions: **the semantics of the root-level `decision` element in the restructured Consent resource are insufficiently documented**. The R4→R5 change moved the permit/deny from individual provisions to the resource root, but the specification does not adequately explain:

1. **What the root decision applies to** — Is it a blanket statement, or does it require additional context (e.g., scope, category, purpose)?
2. **How provisions relate to the root decision** — Are provisions exceptions to the root decision? Must there be at least one provision to make the consent meaningful?
3. **The rationale for the structural change** — Why was the model changed, and how does the new model achieve parity with R4's contextual richness?

The companion ticket FHIR-47213 (already resolved as Persuasive with Modification) addressed scope identification. This ticket focuses specifically on the decision context, which remains unresolved.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Add explanatory text to the Consent resource description clarifying the semantics of the root-level `decision` element. Specifically:
- Document that `decision` represents the base/default consent posture (the overall grant or denial).
- Explain that `provision` entries serve as exceptions or refinements to the root decision, and that contextual details (actor, purpose, data, time period) are expressed within provisions.
- Add a brief note on the rationale for the R4→R5 restructuring (moving from per-provision type to root decision with flat exception provisions).
- Optionally, add an example demonstrating a typical consent with a root `permit` decision and one or more `deny` provisions that carve out exceptions.

**Rationale:** The confusion is legitimate and shared by multiple community members (see comments and related tickets FHIR-53886, FHIR-43326, FHIR-43331). The model itself is sound — the issue is documentation, not design. Adding clarifying text addresses the submitter's request without structural changes.

### Disposition B: Not Persuasive with Modification
**Action:** Resolve as not persuasive (no structural change needed) but add a brief informational note or link in the resource description pointing to the existing narrative sections that explain the consent model (e.g., Section 6.2.1.1 on computable rules, Section 6.2.6 on policy language).

**Rationale:** The information already exists in the specification but is scattered. A lightweight cross-reference or "how to read this resource" note may suffice without rewriting the description.

### Disposition C: Not Persuasive
**Action:** Resolve as not persuasive. David Pyke's comment already explains the model: the root decision is the base posture, and provisions define exceptions. No spec change needed.

**Rationale:** The resource structure is intentional and documented. The submitter's confusion arises from unfamiliarity with the new model, which can be resolved through community education rather than spec changes.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The confusion expressed in this ticket is well-founded and echoed by multiple other tickets (FHIR-53886, FHIR-43326, FHIR-43331, FHIR-49014) and Zulip discussions spanning years. The root `decision` element's relationship to provisions and to elements like `policyBasis` and `regulatoryBasis` is genuinely unclear from the current spec text. David Pyke's comment provides the right answer but it should be in the specification itself, not just in a Jira comment. Adding a concise explanatory paragraph and/or an example to the resource description is a low-cost, high-value improvement that will reduce recurring confusion.
