## Description

The submitter raises two issues with the "Must Have" list in the Scalable Consent Management IG:

1. **Ambiguous "period" element:** The Must Have list references "period," but the required `.recorded` element is typed as an `instant` (not a `Period`). The submitter asks whether "period" refers to `.recorded` or something else.
2. **Missing `.source` element:** The `.source` element is required (1..1) in the profile but is not included in the Must Have list.

## Resolution Description

> We will update the Must Have list to ensure it corresponds to the profile.

## Proposed Dispositions

### Disposition A: Persuasive (Recommended)

**Rationale:** The submitter correctly identifies two clear discrepancies between the Must Have list and the profile:
- "period" is ambiguous and likely mislabeled (`.recorded` is an `instant`, not a `Period`).
- `.source` is required (1..1) in the profile but omitted from the Must Have list.

The existing resolution description—"We will update the Must Have list to ensure it corresponds to the profile"—is appropriate and directly addresses both issues. Accept as persuasive and update the Must Have list to:
1. Replace or clarify "period" (correct to `.recorded` if that is the intended element, or specify which `Period`-typed element is meant).
2. Add `.source` to the Must Have list.

### Disposition B: Persuasive with Modification

**Rationale:** Accept the two specific corrections requested, but also perform a comprehensive audit of the entire Must Have list against the current profile constraints. This ensures no other required elements are missing and no other ambiguous names remain. The resolution description would expand to: "We will audit the full Must Have list against the profile and correct all discrepancies, including the 'period' ambiguity and the missing .source element."

### Disposition C: Not Persuasive

**Rationale:** The Must Have list is intended as a human-readable summary for implementers and is not meant to be an exhaustive mirror of the profile's formal constraints. Implementers should rely on the profile itself for cardinality and datatype requirements. However, this disposition is weak because the list currently contains an ambiguous term ("period") that could mislead implementers, and omitting a required element undermines the list's utility.

## Recommendation

**Disposition A: Persuasive.** The request is well-founded—both issues are clear documentation errors where the Must Have list diverges from the profile. The existing resolution ("update the Must Have list to ensure it corresponds to the profile") directly and appropriately addresses both concerns. No modification is necessary beyond what the resolution already commits to.
