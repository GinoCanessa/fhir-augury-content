## Description

The submitter requests clarification on the meaning of "disclosure" and "information" in the ConsentAuditEvent section of the Scalable Consent Management IG. Specifically, it is unclear whether the audit event records:

1. That the **content of the Consent resource itself** was shared/disclosed, or
2. That **information covered by the Consent** (e.g., lab results authorized for sharing) was shared/disclosed.

This ambiguity impacts implementers' understanding of what the audit event outcome represents.

## Resolution Description

The workgroup resolved this as **Persuasive**. The stated intent is that a FASTAuditEvent would be created by a system whenever a Consent record was **consulted** and a decision made on whether information should be shared or not. Planned changes include:

- Adding a new actor in the business cases around the audit event
- Extensively revising the section on the audit event
- Removing operations in favor of discussing how patients can discover when Consent resources were consulted for information-sharing decisions

## Analysis

This ticket sits at the center of a cluster of related ballot comments (FHIR-54650, FHIR-54682, FHIR-54692, FHIR-55374) that collectively challenge the IG's approach to consent disclosure auditing. The core issue is a **terminological and conceptual ambiguity**: the IG uses "disclosure" loosely where the audit event actually captures an "authorization decision" (consulting a Consent to decide whether to share). IHE BALP distinguishes these as separate profiles with different semantics.

The resolution acknowledges this by committing to extensively revise the audit event section, remove the Record Disclosure Operation, add a new actor, and refocus on how patients can discover when their Consent resources were consulted. This resolution aligns with the feedback from related tickets (particularly FHIR-54692's argument that audit searchability, not creation operations, should be the compliance surface).

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Action:** Accept the existing Persuasive resolution and the proposed changes as described.

**Rationale:** The resolution directly addresses the ambiguity raised. The commitment to revise the audit event section, clarify terminology (distinguishing "consent consultation" from "disclosure"), introduce a new actor, and remove the Record Disclosure Operation resolves both this ticket and aligns with the broader feedback cluster (FHIR-54650, FHIR-54682, FHIR-54692). The revised approach — defining audit events as searchable/readable rather than requiring creation via a custom operation — is consistent with IHE BALP patterns and reduces implementer burden.

**Changes Required:**
- Revise the ConsentAuditEvent / FASTAuditEvent section to clearly define that the audit records when a Consent resource was consulted for an information-sharing decision (not a "disclosure" per se)
- Add a new actor to business cases for the audit event workflow
- Remove the Record Disclosure Operation
- Add guidance on how patients can discover when their Consent resources were consulted
- Align terminology with IHE BALP distinctions between authorization decisions and disclosures

### Disposition B: Persuasive with Modification — Require Explicit BALP Alignment

**Action:** Accept as Persuasive but add an explicit requirement to reference and align with IHE BALP profiles for Consent Authorized Decision and Privacy Disclosure audit events.

**Rationale:** While the resolution commits to revising the audit section, it does not explicitly commit to aligning with the existing IHE BALP profiles that already distinguish authorization decisions from disclosures (as identified in FHIR-54650). Mandating BALP alignment would prevent the IG from reinventing audit semantics and ensure interoperability with systems already implementing BALP. This strengthens the resolution by providing a concrete normative anchor.

**Changes Required:**
- All changes from Disposition A
- Add normative references to IHE BALP §3.5.7.6 (Consent Authorized Decision Audit Message) and §3.5.7.7 (Privacy Disclosure Audit Message)
- Define the IG's audit profiles as constraints on or companions to the BALP profiles
- Reference ITI-81 as the search mechanism for audit events

### Disposition C: Not Persuasive — Defer to Broader Audit Reconciliation

**Action:** Defer this ticket's resolution until the cluster of related audit tickets (FHIR-54650, FHIR-54682, FHIR-54692, FHIR-55374) are collectively reconciled.

**Rationale:** This ticket is one of at least five interrelated tickets challenging the IG's audit approach. Resolving this ticket in isolation risks creating inconsistencies with how the other tickets are resolved. A coordinated review would ensure the audit event architecture is coherent across all concerns (terminology, scope, operations, search, multi-system recording).

**Risk:** This delays resolution and could block IG progression if the broader reconciliation takes significant time.

## Recommendation

**→ Disposition A: Accept as Persuasive**

The existing resolution is well-crafted and directly addresses the submitter's concern. The commitment to revise terminology, add actors, remove the Record Disclosure Operation, and refocus on patient-facing discoverability resolves the core ambiguity. The related ticket cluster (FHIR-54650, FHIR-54682, FHIR-54692) raises complementary concerns that naturally align with this resolution direction, so coordinated review (Disposition C) is not strictly necessary. While BALP alignment (Disposition B) would be beneficial, the resolution already implies movement in this direction and overly prescriptive requirements at this stage could constrain the editorial team unnecessarily.
