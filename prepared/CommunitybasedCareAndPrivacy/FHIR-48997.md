## Description

The reporter requests that the existing `event-statusReason` extension (from the FHIR Extensions Pack) be allowed on the Consent resource. Currently, Consent is not listed in the extension's context, preventing its use. The reporter also asks whether other Event-pattern resources may have the same gap.

## Keywords

`event-statusReason`, `Consent`, `statusReason`, `extension`, `Event pattern`, `workflow alignment`, `revocation`, `status change reason`

## Background and Context

The Consent resource follows the FHIR Event workflow pattern but has never included a `statusReason` element. This gap was identified as early as 2017–2018 in workflow alignment reviews (see FHIR-14218, FHIR-14453). The `event-statusReason` standard extension was created (via FHIR-13907) to provide `statusReason` for Event-pattern resources that lack it as a core element, but Consent was not added to the extension's context list.

The immediate use case arose from a community discussion about recording the reason for revoking a Consent (e.g., termination of proxy access). Without `statusReason`, implementers must resort to less direct mechanisms like Provenance, Task, or embedding the reason in source attachments.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Add `Consent` to the context list of the `event-statusReason` extension in the FHIR Extensions Pack.

**Rationale:** This is the simplest and most direct resolution. The `event-statusReason` extension was designed precisely for Event-pattern resources that lack a core `statusReason` element. Consent follows the Event pattern and has a documented gap for `statusReason` dating back to 2017 (FHIR-14218, FHIR-14453). FHIR-I has already indicated support in principle. Adding Consent to the extension context requires no structural changes to the Consent resource itself and immediately enables implementers to record reasons for consent status changes (revocation, suspension, etc.).

### Disposition B: Persuasive with Modification
**Action:** Instead of (or in addition to) adding Consent to the `event-statusReason` extension context, add `statusReason` as a core element on the Consent resource in R6, with an inter-version extension for R4/R5 use.

**Rationale:** Lloyd McKenzie raised this as an alternative. Adding `statusReason` as a first-class element on Consent would provide better discoverability and align Consent more fully with the Event workflow pattern. However, this is a larger change that affects the Consent resource definition and would require coordination with the R6 timeline. It could be combined with Disposition A (extension context now, core element in R6).

### Disposition C: Not Persuasive
**Action:** No change. Document that Provenance, Task, or source attachments/references should be used to record consent status change reasons.

**Rationale:** FHIR already provides mechanisms to capture reasons for status changes: Provenance can record why any resource was modified, Task can carry a `statusReason` for the change request, and `Consent.sourceAttachment`/`sourceReference` can hold evidence of the ceremony. However, this disposition ignores the established Event pattern convention, the existing extension purpose, and FHIR-I's stated support for the change.

## Recommendation

**Disposition A: Persuasive** — Add Consent to the `event-statusReason` extension context. This is a minimal, well-scoped change that addresses a documented gap, aligns with the Event workflow pattern, has FHIR-I endorsement, and serves a concrete implementer need (recording reasons for consent revocation/termination). The work group may additionally consider requesting a core `statusReason` element for R6 as a follow-up, but the extension context change can proceed immediately.
