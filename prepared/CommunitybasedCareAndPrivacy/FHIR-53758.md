## Description

The submitter asks why the Subscription `.status` element is fixed to `active` in the Scalable Consent Management IG. Once a system subscribes to a Consent resource, it should be permitted to unsubscribe — but the current constraint prevents transitioning the Subscription to any other status (e.g., `off`, `requested`, `error`, `entered-in-error`).

## Resolution Description

> There is no reason why we fixed this to 'active'. We will remove this constraint.

The workgroup agreed the fixed-value constraint on Subscription.status was unintentional and will be removed to allow the full Subscription status lifecycle.

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Remove the fixed-value constraint

**Action:** Remove the fixed-value constraint that binds `Subscription.status` to `active` in the Scalable Consent Management IG subscription profile. Allow the element to use the full `SubscriptionStatus` value set as defined in the base FHIR Subscription resource.

**Rationale:** The constraint was unintentional per the workgroup's own resolution. Systems need the ability to unsubscribe (set status to `off`) and to represent other lifecycle states such as `requested`, `error`, and `entered-in-error`. Removing this constraint aligns the IG with standard FHIR Subscription behavior and is essential for practical implementation.

**Risk:** Low. This is a loosening change — existing conformant instances remain valid.

### Disposition B: Accept as Persuasive with Modification — Remove constraint and add guidance

**Action:** Remove the fixed-value constraint on `Subscription.status` *and* add normative or informational text explaining which status values are expected in typical Scalable Consent Management workflows (e.g., `active` for monitoring, `off` for unsubscribing).

**Rationale:** Simply removing the constraint may leave implementers uncertain about expected behavior. Brief guidance would improve implementability, especially given the related ticket FHIR-54101 which raises questions about subscription trust and lifecycle. This also complements FHIR-54676's request for documentation of intentional status constraints.

**Risk:** Low. Adds a small documentation burden but improves clarity.

### Disposition C: Not Persuasive — Keep the constraint with documentation

**Action:** Retain the fixed-value constraint on `Subscription.status` = `active` and add documentation explaining the design intent: that the IG expects subscriptions to be managed as create/delete rather than status transitions.

**Rationale:** If the IG's subscription model assumes that unsubscribing is accomplished by deleting the Subscription resource rather than changing its status, the constraint could be intentional. This would be a valid pattern and should be documented.

**Risk:** Medium. Deviates from standard FHIR Subscription lifecycle patterns and may confuse implementers who expect to manage subscriptions via status transitions.

## Recommendation

**Disposition B: Accept as Persuasive with Modification — Remove constraint and add guidance.**

The workgroup has already agreed the constraint was unintentional (resolution: Persuasive). Disposition B goes one step further than a bare removal by adding brief guidance on expected Subscription status usage within Scalable Consent Management workflows. This addresses the immediate issue while proactively reducing implementer confusion — especially relevant given the related tickets (FHIR-54676, FHIR-54101) that highlight a pattern of questions around status constraints and subscription lifecycle in this IG.
