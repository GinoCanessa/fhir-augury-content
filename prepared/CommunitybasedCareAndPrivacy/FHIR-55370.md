## Description

The submitter observes that it is unclear what connections exist between the steps in the use cases and the technical mechanisms defined in the IG. They request an informative mapping between use cases and their corresponding technical mechanisms, with specific examples:

- **Sign and File Consent / Revoke Consent**: Appear straightforward but could be done via the defined API or internal mechanisms — the distinction is not made clear.
- **Delegate Consent**: Has no API mapping. Is it a special case of Sign and File, or would it have additional behavior? Should there be normative requirements?
- **Propagate Consent Events**: How would events like "the consenter starting to review the consent form" be detected for notification purposes?

## Resolution Description

The resolution states: *"We are going to include a link to the relevant technical operation in each use case diagram and provide a table that shows what operations are required for each use case."*

## Analysis

This ticket addresses a usability gap in the Scalable Consent Management IG: the use cases describe workflows at a conceptual level but do not explicitly connect steps to the technical mechanisms (API operations, resources, profiles) that implement them. This disconnect makes it difficult for implementers to understand which FHIR operations are needed for each use case and whether certain steps require API interactions or can be handled through internal mechanisms.

The proposed resolution — adding links to relevant technical operations in each use case diagram and providing a mapping table — is practical and directly addresses the concern. This approach aligns well with the cluster of related tickets (FHIR-54677, FHIR-54047, FHIR-53752) that all express confusion about the relationship between use-case descriptions and normative/technical content.

The resolution is appropriately scoped as informative content (a mapping table and diagram links), avoiding the risk of over-constraining implementations while providing the guidance implementers need.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Action**: Accept the ticket as Persuasive and implement the proposed resolution.

**Rationale**: The request is well-formed and the proposed resolution is specific, achievable, and directly addresses the gap. Adding links to relevant technical operations in each use case diagram and providing a mapping table will significantly improve implementer comprehension without imposing new normative requirements. This approach also partially addresses the concerns raised in related tickets (FHIR-54677, FHIR-54047, FHIR-53752) about the unclear relationship between use cases and conformance/technical expectations. The resolution aligns with the informative nature of the request and is a straightforward editorial enhancement.

### Disposition B: Accept as Persuasive with Modification

**Action**: Accept as Persuasive with Modification, expanding the scope to also annotate each use-case step as normative or informative alongside the technical operation mapping.

**Rationale**: Several related tickets (FHIR-54047, FHIR-54677, FHIR-53752) ask for clarity on which use-case elements are normative vs. illustrative. Combining the operation mapping table with normative/informative annotations in a single pass would address multiple ballot comments efficiently and provide a more comprehensive improvement. However, this broadens scope beyond the submitter's request and may delay resolution.

### Disposition C: Not Persuasive

**Action**: Reject as Not Persuasive.

**Rationale**: The use cases are intended as informative context and implementers should refer to the conformance and operations sections for technical details. Adding a formal mapping may create a maintenance burden if use cases or operations change independently. However, this disposition ignores the consistent ballot feedback from multiple reviewers expressing the same concern and would leave a significant usability gap.

## Recommendation

**Disposition A: Accept as Persuasive.** The proposed resolution is well-defined, proportionate, and directly responsive to the feedback. The mapping table and diagram links will clearly connect use cases to their technical mechanisms without overcomplicating the specification. This approach addresses the core concern effectively while leaving room for the related normative/informative clarification tickets (FHIR-54047, FHIR-54677) to be handled through their own resolutions.
