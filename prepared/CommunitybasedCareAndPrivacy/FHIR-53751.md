## Description

The submitter raises two clarity issues regarding system/role terminology in the Scalable Consent Management IG:

1. **Must Support role ambiguity:** The Must Support expectations use the terms "data sources" and "data consumers," but Section 6.1 defines systems as "Client" and "Server." It is unclear whether either client or server could act as either data source or consumer, or whether each system is tied to a particular role.

2. **ConsentSystem identity confusion:** The Use Cases reference a "ConsentSystem" actor, while other pages define "Client" (usually the EHR) and "Server" (the Consent Management System). The relationship between these terms is unclear. For example, in Section 3.3 (Sign and File), it is ambiguous whether the ConsentSystem represents the EHR recording the Consenter's approval or the Consent Management System receiving it.

## Resolution Description

> We will update the use cases to have an explicit client system (generally representing an EHR) along with the current Consent System.
>
> e.g., Consenter → Consent Client System → Consent Server System
>
> Then we will change the capability statements to be ConsentClient and ConsentServer.

## Analysis

This ticket is part of a **cluster of ballot comments** identifying a systemic terminology problem in the Scalable Consent Management IG. Multiple reviewers (Craig Newman, Virginia Lorenzi, Abderrazek Boufahja, EssienGe) independently flagged that the specification uses inconsistent terms for actors and systems — "ConsentSystem," "Client," "Server," "data source," "data consumer," "EHR," "Consent Management System" — without clearly mapping them to each other or to human roles.

The proposed resolution introduces a three-tier model (Consenter → Consent Client System → Consent Server System) and renames capability statements to ConsentClient and ConsentServer. This directly addresses the submitter's concerns and aligns well with the resolution direction of the related tickets (FHIR-54922, FHIR-53887, FHIR-53775).

**Key considerations:**
- The resolution should be coordinated with FHIR-54922 (Highest priority) and FHIR-53887, which call for a comprehensive actor/terminology harmonization across the entire IG.
- The renamed capability statements (ConsentClient/ConsentServer) must be consistently propagated to all conformance-related pages (addressing FHIR-53775).
- The use case diagrams should reflect the updated system model (addressing FHIR-53773).

## Proposed Dispositions

### Disposition A: Accept as Persuasive with Modification (Recommended)

**Action:** Accept the resolution as stated — introduce the Consenter → Consent Client System → Consent Server System model, rename capability statements to ConsentClient and ConsentServer, and update use case diagrams accordingly.

**Rationale:** The resolution directly addresses both concerns raised (Must Support role ambiguity and ConsentSystem identity confusion) with a clear, implementable approach. The three-tier naming convention eliminates ambiguity about which system is which and how they relate to human actors. This aligns with the direction needed for the related cluster of terminology tickets (FHIR-54922, FHIR-53887, FHIR-53775) and can serve as the foundation for broader harmonization. The "with Modification" qualifier is appropriate because the resolution refines the request (the submitter asked for clarification; the resolution provides a specific renaming scheme).

### Disposition B: Accept as Persuasive

**Action:** Accept the comment as-is, requiring full actor/terminology harmonization across the entire specification before the next ballot cycle, going beyond the scope of just use cases and capability statements.

**Rationale:** While the proposed resolution addresses the immediate concern, the broader terminology problem flagged by FHIR-54922 (Highest priority) suggests that a piecemeal fix may be insufficient. A full harmonization — defining all actors (Consenter, Administrator, etc.) and all systems (Consent Client, Consent Server) upfront in a dedicated section, then using those terms consistently — would be more complete. However, this broader scope may be better tracked under FHIR-54922.

### Disposition C: Not Persuasive

**Action:** Reject the request, arguing that the existing terminology is adequate and the context makes system roles clear enough.

**Rationale:** The specification does define Client and Server in Section 6.1, and readers could infer the mapping from context. However, this disposition is not recommended because multiple independent reviewers raised the same concern, indicating a genuine usability problem that would hinder implementer adoption.

## Recommendation

**Disposition A: Accept as Persuasive with Modification.** The proposed resolution is well-scoped, directly addresses both concerns, and provides a clear naming convention (ConsentClient/ConsentServer) that can be consistently applied. The workgroup should coordinate this change with the broader terminology harmonization tracked under FHIR-54922 and FHIR-53887 to ensure a consistent result across the full specification.
