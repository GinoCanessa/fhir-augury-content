## Summary

The submitter requests that the names of CapabilityStatement resources in the Scalable Consent Management IG be reviewed and aligned with the actor names used elsewhere in the specification. Currently, terms such as "client system," "server system," "consent admin service," "consent administrative service," and "client admin service" are used inconsistently across pages—including in capability statement titles, conformance expectations, and the technical specification. The submitter notes that "client and server" alone are too vague and asks the authors to select clear, well-defined actor names and apply them uniformly throughout the IG. This is closely related to the submitter's companion ticket (FHIR-54922) requesting standardized actor/system definitions.

The resolution accepted the request as **Persuasive**: the workgroup will create "Consent Client" and "Consent Server" actors, use them in use cases, and update the CapabilityStatement resources to use those names.

## Proposed Dispositions

### Disposition A: Accept (Persuasive) — Rename CapabilityStatements to "Consent Client" and "Consent Server"

Accept the ticket as persuasive and implement the stated resolution. Specifically:

1. **Rename CapabilityStatement resources:**
   - Rename `ConsentClientCapabilities` to use the title **"Consent Client CapabilityStatement"** (or similar pattern such as `ConsentClientCapabilityStatement`).
   - Rename `ConsentAdministrativeServerCapabilities` to **"Consent Server CapabilityStatement"** (or `ConsentServerCapabilityStatement`).
2. **Define actors consistently:** Introduce formal "Consent Client" and "Consent Server" actor definitions in the Actors section and reference them uniformly in use cases, conformance expectations, and the technical specification page.
3. **Eliminate ambiguous variants:** Remove or replace all occurrences of "client system," "server system," "consent admin service," "consent administrative service," and "client admin service" with the two canonical actor names.
4. **Coordinate with related tickets:** Align this work with FHIR-54922, FHIR-54567, FHIR-53887, FHIR-53775, and FHIR-53751, which all request similar terminology cleanup.

### Disposition B: Accept with Modification — Use More Descriptive Actor Names

Accept the request to standardize but choose more descriptive actor names than "Consent Client" and "Consent Server" to avoid the vagueness the submitter originally flagged. For example:

1. **Use domain-specific names:** Adopt names like **"Consent Requestor"** (the system that creates/submits/revokes consents) and **"Consent Registry"** (the system that stores and manages consent records).
2. **Rename CapabilityStatements accordingly:** Update `ConsentClientCapabilities` → **"Consent Requestor CapabilityStatement"** and `ConsentAdministrativeServerCapabilities` → **"Consent Registry CapabilityStatement."**
3. **Update all references throughout the IG** to use these more self-describing terms.

This approach addresses the submitter's concern that "client" and "server" alone are too generic and could confuse readers about the functional roles each system plays.

### Disposition C: Decline (Not Persuasive)

Decline the request on the grounds that:

- "Client" and "server" are standard FHIR actor paradigms used across many implementation guides and are well understood by implementers.
- Renaming CapabilityStatement resources and actor definitions introduces breaking changes for implementers who have already built against the current ballot names.
- The existing definitions, while slightly inconsistent across pages, are understandable in context and can be addressed with minor editorial fixes (fixing "consent admin service" vs. "consent administrative service") rather than a full rename.

---

## Recommendation

**Disposition A: Accept (Persuasive)** is recommended.

There is overwhelming ballot feedback requesting terminology harmonization in the Scalable Consent Management IG—at least five related tickets (FHIR-54922, FHIR-54567, FHIR-53887, FHIR-53775, FHIR-53751) all identify the same class of problem. The workgroup has already resolved this ticket as Persuasive with a concrete plan to introduce "Consent Client" and "Consent Server" actors. This approach standardizes naming, directly addresses the inconsistency between CapabilityStatement titles and actor definitions, and is the simplest path that satisfies the cluster of related tickets. Coordinating this rename with the related tickets will produce a single coherent terminology pass rather than piecemeal fixes.
