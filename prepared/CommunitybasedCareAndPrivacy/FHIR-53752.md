## Description

The submitter raises concerns about Section 7.2.3 of the Scalable Consent Management IG, which requires that a "consent administration service" must support filing and revoking consent. Specifically:

1. **Terminology inconsistency**: The spec uses multiple terms—"consent administration service," "consent administration system," and "Consent Registry" (in the Subscription Topic profile)—without clarifying whether these refer to the same entity.
2. **Use case–conformance alignment**: It is unclear whether the "filing" and "revoking" requirements in Section 7.2.3 map directly to the similarly named use cases on the Use Case page. Does "filing consent" include the requesting and reviewing steps, or just the formal filing action?
3. **Optional vs. required use cases**: Only a single use case on the Use Case page is listed as optional, implying all others are required. This may create unintended mandatory conformance expectations that do not align with Section 7.2.3's scope.

## Resolution Description

The work group resolved this as **Persuasive**:

> We are going to provide an informative mapping from the business use cases to the technical operations to show when the operations are used. We will also ensure we are using consistent language of Consent Client Systems and Consent Server Systems.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Action**: Accept the existing Persuasive resolution as stated.

**Rationale**: The resolution directly addresses both concerns raised: (1) providing an informative mapping from business use cases to technical operations clarifies the relationship between the Use Case page and Section 7.2.3 conformance requirements, and (2) adopting consistent terminology ("Consent Client Systems" and "Consent Server Systems") resolves the naming confusion across the IG. This aligns with the related work on FHIR-55370 (which also requests use-case-to-technical mappings) and FHIR-54567 (which flags actor definition inconsistencies). The work group should ensure the mapping clearly distinguishes which use cases correspond to mandatory conformance requirements vs. informative guidance, directly addressing the submitter's concern about optional-vs-required use cases.

**Suggested implementation notes**:
- Add an informative table or section mapping each business use case to the corresponding FHIR operations and conformance statements.
- Standardize on "Consent Client System" and "Consent Server System" throughout the IG, replacing "consent administration service," "consent administration system," and "Consent Registry."
- Clarify in the mapping which use cases are required vs. optional for conformance.

### Disposition B: Persuasive with Modification — Expand to Include Normative Clarifications

**Action**: Accept as Persuasive but expand the resolution to also add explicit normative language distinguishing required from informative use cases.

**Rationale**: The current resolution promises an informative mapping and consistent terminology, but does not directly address the submitter's concern that nearly all use cases appear mandatory. In addition to the mapping, the IG should add explicit normative labels (e.g., "SHALL support" vs. "SHOULD support" vs. "informative") to each use case. This would more fully resolve the ambiguity and align with FHIR-54047's request for clarity on normative vs. illustrative content. The cost is additional editorial work, but it would reduce future ballot comments on the same theme.

### Disposition C: Not Persuasive — Informative Mapping Alone Is Sufficient

**Action**: Accept as Not Persuasive, arguing the existing IG structure is adequate with minor editorial fixes.

**Rationale**: One could argue the conformance expectations in Section 7.2.3 are already clear as written and that use cases are inherently informative context. However, this interpretation is undermined by the volume of related ballot comments (FHIR-55370, FHIR-54047, FHIR-54677, FHIR-53775) all raising similar confusion. Dismissing this ticket would not resolve the systematic clarity issue and would likely generate repeat comments in future ballots.

## Recommendation

**Disposition A: Accept as Persuasive** is recommended. The resolution already commits to the two key actions needed—an informative mapping and consistent terminology—which address the submitter's core concerns. The work group should coordinate implementation with the closely related FHIR-55370 and FHIR-54567 to ensure a consistent outcome across all three tickets. If the work group has capacity, elements of Disposition B (explicit normative labeling of use cases) would further strengthen the IG but are not strictly required to resolve this ticket.
