## Description

The submitter questions why `Consent.scope` is defined as both a subscription filter element and a search element in the Scalable Consent Management (FASTConsent) IG, given that the profile fixes `Consent.scope` to the single value `patient-privacy`. If the scope is always fixed, filtering and searching by scope provides no discriminating value. The submitter asks whether the IG assumes that consent administration systems will also manage Consent resources for other purposes (where scope would vary), and whether `.category` would be a more meaningful element for filtering and searching.

## Resolution Description

> We should not have scope as a subscription topic filter. We will change that to category as that could conceivably be a valid filter.

## Zulip Discussions

### [Security and Privacy > Consent.scope](https://chat.fhir.org/#narrow/stream/Security%20and%20Privacy/topic/Consent.scope) (May 2021)

A detailed discussion about a proposal to phase out `Consent.scope` and consolidate its function into `Consent.category`. Key points:

- **Mohammad Jafari** introduced the CBCP WG proposal to phase out `Consent.scope`, citing duplication with `Consent.category` (referencing FHIR-31903).
- **David Pyke** confirmed the motivation: "It's seen that Consent.scope and Consent.category are duplicative."
- **John Moehrke** disagreed, arguing that `Consent.scope` is analogous to `Observation.category` (high-level kind) while `Consent.category` is analogous to `Observation.code` (more refined detail). He felt having two separate elements is clearer than requiring slicing.
- **Lloyd McKenzie** argued that `Consent.scope` is fundamentally just a category — it does not drive provision behavior. He noted: "Consent.scope is just a category — it doesn't change what the consent does or doesn't do. And the current proposed codes for Consent.scope are not orthogonal or even necessarily complete." He supported consolidating into a single `category` element.
- The thread shows ongoing tension between those who see value in a dedicated scope element (cleaner separation, no slicing needed) and those who view it as redundant categorization.

## Analysis

This ticket identifies a genuine inconsistency in the Scalable Consent Management IG: the FASTConsent profile fixes `Consent.scope` to a single value (`patient-privacy`), yet the IG defines scope as both a subscription topic filter and search element. When a value is fixed, filtering or searching by it has no discriminating power — every matching resource will have the same scope value.

The resolution ("Persuasive") acknowledges this inconsistency and proposes replacing `Consent.scope` with `Consent.category` as the subscription topic filter and search element. This aligns with the broader community direction (seen in the 2021 Zulip discussion) where `Consent.scope` has been viewed as duplicative with `Consent.category`, and with Lloyd McKenzie's position that scope is effectively just another category.

The closely related FHIR-54056 (also Persuasive) addresses the same fixed-scope constraint from a rationale/documentation perspective, and both tickets should be considered together during implementation.

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Replace scope with category in subscription/search definitions

Accept the resolution as stated. Replace `Consent.scope` with `Consent.category` as the subscription topic filter element and search element in the Scalable Consent Management IG. Since scope is fixed to `patient-privacy`, it provides no filtering value; `Consent.category` can carry varying values that are meaningful for filtering and subscription matching.

- **Pros:** Directly addresses the logical inconsistency; aligns with the resolution text; consistent with the broader community view that scope is effectively a category.
- **Cons:** Implementers who have already built subscription filters on `Consent.scope` would need to update.

### Disposition B: Accept as Persuasive with Modification — Remove scope filter and add category filter with guidance

Accept the resolution but add explicit IG guidance explaining (1) why `Consent.scope` was removed as a filter (it is fixed and non-discriminating), (2) how `Consent.category` should be used for filtering and subscription, and (3) recommended category values for common consent filtering scenarios. Coordinate with FHIR-54056 to also document the rationale for the fixed scope constraint.

- **Pros:** Provides implementers with clear migration guidance; addresses both this ticket and FHIR-54056 concerns; reduces future confusion.
- **Cons:** Requires additional documentation effort beyond the minimal code change.

### Disposition C: Accept as Not Persuasive — Retain scope as filter for cross-IG interoperability

Retain `Consent.scope` as a subscription filter and search element on the basis that consent administration systems may manage Consent resources across multiple IGs with different scope values. The fixed value in FASTConsent does not preclude other profiles from using different scope values, and a system aggregating consents from multiple profiles may benefit from filtering by scope.

- **Pros:** Preserves current behavior; acknowledges multi-IG deployment scenarios.
- **Cons:** Does not resolve the fundamental inconsistency within this IG; filtering on a fixed value remains logically unnecessary within FASTConsent's own scope.

## Recommendation

**Disposition B: Accept as Persuasive with Modification.** The resolution correctly identifies the inconsistency and the right fix (swap scope for category in filters). However, given the closely related FHIR-54056 and the long community history of confusion around `Consent.scope` vs. `Consent.category`, adding implementation guidance alongside the technical change will maximize clarity and ease adoption. This approach resolves the immediate issue while proactively reducing future questions about the fixed scope constraint.
