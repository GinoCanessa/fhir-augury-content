## Description

The FHIR build emits an INFORMATION-level QA message:

> `INFORMATION:Consent.verification.verificationDate Name of child (verificationDate) overlaps with name of parent (verification)`

The submitter proposes renaming the child element from `verificationDate` to `date` to eliminate the naming overlap.

## Analysis

This is a straightforward naming cleanup. The FHIR build QA system flags cases where a child element name contains the parent element name (e.g., `verificationDate` under `verification`), producing redundancy in the fully-qualified path `Consent.verification.verificationDate`. The proposed rename to `Consent.verification.date` eliminates the overlap and improves readability.

**Key observations:**
1. **Well-established pattern**: FHIR-37578 (Published) set precedent for this exact type of rename in MedicinalProductDefinition, approved by a different work group.
2. **Already flagged multiple times**: Both FHIR-44632 and FHIR-33039 previously noted these naming overlaps in Consent build QA output.
3. **Implementation already in place**: The `fhir.ini` rename mapping `Consent.verification.verificationDate=@Consent.verification.date` is already present in the HL7/fhir repository.
4. **Companion ticket**: FHIR-50872 addresses the identical issue for the sibling element `verificationType` ‚Üí `type`.
5. **No controversy**: No Zulip discussion or dissenting opinions found.

## Proposed Dispositions

### Disposition A: Persuasive (Accept as-is)
**Action:** Rename `Consent.verification.verificationDate` to `Consent.verification.date`.

**Rationale:** The rename eliminates the QA naming overlap warning, follows established precedent (FHIR-37578), and is a minimal non-breaking naming change for a build-level information message. The `fhir.ini` mapping already reflects this change. The new name `date` is clear in context ‚Äî under `verification`, a `date` element unambiguously refers to the verification date.

### Disposition B: Persuasive with Modification ‚Äî Use a more descriptive name
**Action:** Rename `Consent.verification.verificationDate` to `Consent.verification.dateTime` instead of `date`.

**Rationale:** The element is typed as `dateTime`, so using `dateTime` as the name would both resolve the overlap and better reflect the data type. However, this deviates from the submitter's proposal and FHIR convention (element names don't typically mirror type names), so this may cause unnecessary confusion.

### Disposition C: Not Persuasive ‚Äî Defer to informational-only status
**Action:** Leave the element name as `Consent.verification.verificationDate`.

**Rationale:** The QA message is INFORMATION-level, not an error or warning. The current name is descriptive and unambiguous. Renaming introduces a breaking change for implementers who reference this path. However, this disposition conflicts with the existing `fhir.ini` mapping and the precedent set by FHIR-37578.

## Recommendation

**Disposition A: Persuasive** ‚Äî Accept the rename of `Consent.verification.verificationDate` to `Consent.verification.date`.

This is a well-supported, low-risk change that has already been applied in the FHIR source repository (`fhir.ini`), follows established precedent from FHIR-37578, and resolves a long-standing QA message flagged across multiple prior tickets. The workgroup should also consider processing companion ticket FHIR-50872 (`verificationType` ‚Üí `type`) in the same session for consistency.
