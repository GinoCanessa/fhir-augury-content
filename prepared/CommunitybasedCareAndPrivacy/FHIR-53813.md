## Summary

The FAST Consent SubscriptionTopic in the Scalable Consent Management IG declares support for Subscription notifications on the **delete** interaction of a Consent resource in the Admin Server. However, this conflicts with two other parts of the IG:

1. The **Technical Specification – Revoke Consent Operation** section states: *"A revoked consent SHALL NOT be deleted from the consent management system."*
2. The **Consent Administrative Server Capabilities** CapabilityStatement does **not** declare support for the delete interaction on the Consent resource.

The submitter requests clarification by either removing the delete interaction from the SubscriptionTopic, or adding guidance and CapabilityStatement support for how Consent deletes would work.

The resolution description states: **"We will remove 'delete' as a supported interaction on the Subscription Topic."**

---

## Details

### Description

There is a mismatch in declared support for the FHIR delete interaction on the Consent resource type. The FAST Consent SubscriptionTopic declares support for a Subscription notification on the delete of a Consent resource in the Admin Server. However, references in the IG indicate the delete of a Consent resource in the Admin Server is not supported:

- The [Technical Specification - Revoke Consent Operation](https://hl7.org/fhir/us/consent-management/2026Jan/technical.html#revoke-consent-operation) section states:
  > A revoked consent SHALL NOT be deleted from the consent management system.

- The [Consent Administrative Server Capabilities](https://hl7.org/fhir/us/consent-management/2026Jan/CapabilityStatement-ConsentAdministrativeServerCapabilities.html) CapabilityStatement does not declare support for delete on the Consent resource interaction.

Clarification can be defined by either:
- Removing the Consent delete interaction declaration in the FAST Consent SubscriptionTopic
- Updating the Technical Specification with language describing how Consent deletes can be supported (and if supported, adding the Consent delete interaction to the CapabilityStatement)

### Resolution Description

> We will remove 'delete' as a supported interaction on the Subscription Topic.

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Remove delete from SubscriptionTopic *(Recommended)*

Accept the ticket as **Persuasive** and remove the `delete` supportedInteraction from the FAST Consent SubscriptionTopic's resourceTrigger. This aligns with the stated resolution and is the simplest path to resolve the internal inconsistency.

**Rationale:** The IG already constrains consent lifecycle management through the Revoke Consent Operation (which changes Consent.status rather than deleting the resource) and the CapabilityStatement already omits delete. Removing delete from the SubscriptionTopic makes all three artifacts consistent: the SubscriptionTopic, the CapabilityStatement, and the Technical Specification all agree that Consent resources are not deleted. This is also consistent with the companion ticket FHIR-53703 (also resolved Persuasive).

**Actions required:**
- Remove `delete` from the `supportedInteraction` list in the FAST Consent SubscriptionTopic resource's `resourceTrigger`.
- Verify no other IG references assume delete notification support.

### Disposition B: Accept as Persuasive with Modification — Remove delete and add clarifying guidance

Accept as **Persuasive with Modification**. Remove the delete interaction from the SubscriptionTopic (as in Disposition A) but also add an informative note in the Technical Specification and/or Implementation Notes explaining why delete is intentionally excluded and that consent revocation (not deletion) is the supported workflow.

**Rationale:** While removing the delete interaction resolves the inconsistency, implementers may still wonder why delete is unsupported. Adding a brief explanatory note improves clarity and reduces future confusion, particularly for implementers accustomed to supporting delete on other resource types.

**Actions required:**
- Remove `delete` from the SubscriptionTopic `resourceTrigger.supportedInteraction`.
- Add a note in the Technical Specification (e.g., in the Revoke Consent Operation section) stating that Consent resources are managed through status transitions (revocation) rather than deletion, and that delete is intentionally not supported.

### Disposition C: Decline as Not Persuasive

Reject the ticket and retain the delete interaction in the SubscriptionTopic.

**Rationale:** One could argue the SubscriptionTopic should declare delete support to allow servers that optionally support deletion to notify subscribers. The CapabilityStatement could be treated as a minimum conformance requirement while the SubscriptionTopic covers a broader set of possible interactions. However, this interpretation conflicts with the SHALL NOT language in the Technical Specification and would require reconciling the normative conformance language, which is a larger change.

---

## Recommendation

**Disposition A: Accept as Persuasive — Remove delete from SubscriptionTopic.**

The evidence strongly supports this disposition. The three IG artifacts (SubscriptionTopic, CapabilityStatement, Technical Specification) are currently inconsistent, and the simplest and most internally coherent fix is to remove the delete interaction from the SubscriptionTopic. This aligns with the ticket's stated resolution, the existing conformance language ("SHALL NOT be deleted"), and the companion ticket FHIR-53703 which reached the same conclusion. Both tickets were filed by the same knowledgeable reviewer (Richard Ettema), and the resolution is straightforward with minimal risk of unintended consequences.
