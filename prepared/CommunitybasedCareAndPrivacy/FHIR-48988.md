## Description

The Consent resource's introduction and notes discuss the use of external standards (such as XACML) to record consent rules, but no example is provided. The ticket requests an example of a Consent resource that uses XACML (or another external policy language) to represent its rules, rather than using the `.provision` element.

The proposed example would be similar to existing Consent examples, except:
- `Consent.policyBasis.url` would be populated with a full URL pointing to an external XACML policy (e.g., `http://example.org/xacml/policy/set23456.xml`)
- No `.provision` elements would be populated

## Analysis

### Core Issue
The Consent resource narrative describes the ability to use external policy standards (XACML, ODRL) as an alternative to `.provision` for encoding consent rules, but no example demonstrates this pattern. This is a documentation/example gap.

### Complications Discovered
While creating the example, the reporter discovered a deeper structural issue: the old `.policyRule` element (which was the natural place to point at patient-specific external rules) has been removed. The current `policyBasis` element is semantically defined as the patient-independent backing policy, not a patient-specific rule set. This means there is currently **no clear element** for pointing at patient-specific XACML/ODRL rules when `.provision` is not used. This structural gap is tracked in FHIR-53724.

### Impact
- Adding an example alone may be insufficient if the structural gap (no element for patient-specific external rules) is not addressed first.
- Multiple related tickets (FHIR-53724, FHIR-54155, FHIR-54440) are converging on the same area of the Consent resource, suggesting a coordinated approach may be warranted.

## Proposed Dispositions

### Disposition A: Persuasive — Add example with editorial guidance
**Action:** Accept the request. Add a Consent example that uses `Consent.policyBasis.url` to reference an external XACML policy set, with no `.provision` elements. Include editorial notes clarifying the distinction between patient-independent backing policies and patient-specific external rules (acknowledging the gap identified in FHIR-53724). Leverage the work in the [JohnMoehrke/xacml-consent](https://github.com/JohnMoehrke/xacml-consent) repository as the basis for the example.

**Rationale:** The narrative already describes this pattern; an example is needed to make it concrete. The editorial notes bridge the gap until the structural issue is resolved.

### Disposition B: Persuasive with Modification — Defer until FHIR-53724 is resolved
**Action:** Accept the request in principle, but defer the example creation until FHIR-53724 (missing `policyRule` element) is resolved. The example should use whatever new or clarified element emerges from that resolution to properly reference patient-specific external rules.

**Rationale:** John Moehrke's own analysis (comment 2025-12-28) identified that `policyBasis.url` is semantically wrong for patient-specific external rules. Creating an example using the wrong element would be misleading and would need to be revised once FHIR-53724 is addressed.

### Disposition C: Not Persuasive — Existing narrative is sufficient
**Action:** Decline the request. The Consent resource narrative (enhanced by FHIR-33104) already explains that external policy languages can be used. The `policyBasis` element documentation is sufficient for implementers to understand the pattern without a formal example.

**Rationale:** The structural uncertainty around where patient-specific external rules belong makes any example premature. Adding an incorrect or ambiguous example could cause more confusion than clarity.

## Recommendation

**Disposition B: Persuasive with Modification** — The reporter's own subsequent analysis (leading to FHIR-53724) revealed that the straightforward approach originally proposed has a semantic conflict with `policyBasis`. The workgroup should acknowledge the need for an XACML example but coordinate it with the resolution of FHIR-53724, which addresses the missing element for patient-specific external policy rules. The draft work in the [JohnMoehrke/xacml-consent](https://github.com/JohnMoehrke/xacml-consent) repository can serve as the foundation once the structural question is settled.
