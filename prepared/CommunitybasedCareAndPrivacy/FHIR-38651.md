## Description

Josh Mandel (during FMG publication approval review) observes that `Encounter.basedOn` is semantically defined as "the request this encounter satisfies (e.g. incoming referral or procedure request)." However, the PACIO Re-Assessment Timepoints Encounter profile uses a `clinicalImpression` extension *within* `Encounter.basedOn` to reference things like "Assessment and impression at discharge from SNF" — which are not requests at all.

The recommendation is to move the [`clinicalImpression` extension](http://hl7.org/fhir/us/pacio-rt/StructureDefinition/clinicalImpression) to the root of the timepoint `Encounter` resource (i.e., `Encounter.extension`) rather than nesting it inside `Encounter.basedOn`.

## Proposed Dispositions

### Disposition A: Persuasive — Move the extension to Encounter.extension

**Rationale:** The submitter's argument is well-founded. The FHIR core definition of `Encounter.basedOn` is explicitly about fulfilling requests (referrals, procedure requests). Using it to anchor a ClinicalImpression extension misuses the semantics of `basedOn`. Moving the `clinicalImpression` extension to `Encounter.extension` aligns with standard FHIR extension placement practices and eliminates the semantic mismatch. This is a clean, non-breaking change for STU2 since it only affects the profile structure (consumers would look for the extension at the Encounter root instead of nested under `basedOn`).

**Applied Change:** Update the Re-Assessment Timepoints Encounter profile to place the `clinicalImpression` extension at `Encounter.extension` instead of `Encounter.basedOn.extension`. Update IG narrative, examples, and StructureDefinition context accordingly.

### Disposition B: Persuasive with Modification — Move the extension and deprecate basedOn usage

**Rationale:** Accept the core request to move the extension to `Encounter.extension`, but go further by removing or un-must-supporting `Encounter.basedOn` in the profile entirely if it is not being used to reference actual request resources. This would prevent future confusion about the role of `basedOn` in timepoint encounters.

**Applied Change:** Move the `clinicalImpression` extension to `Encounter.extension`. Additionally, remove `basedOn` from Must Support (or remove profiling constraints on it) unless the IG has a legitimate request-based use for it. Update examples and narrative.

### Disposition C: Not Persuasive — Retain current placement

**Rationale:** While the semantic concern is valid in the abstract, the current placement has been published in STU1 and implementers may have already built against it. Moving the extension in STU2 would be a breaking change for existing implementations. The `basedOn` element is being used as a grouping mechanism for clinical context, and the extension's presence under it is sufficiently documented in the IG. The FHIR core `basedOn` semantics, while suggestive, do not strictly prohibit extensions from carrying non-request data.

**Applied Change:** No change. Optionally add IG narrative clarifying why the extension is placed under `basedOn` despite the element's core definition.

## Recommendation

**Disposition A: Persuasive — Move the extension to Encounter.extension.**

The semantic argument is strong and was raised by a highly experienced FHIR reviewer during FMG publication approval. The PACIO team member (Karl Naden) acknowledged the point as valid on Zulip. The closely related ticket FHIR-35123 (resolved as Persuasive) already established that the ClinicalImpression extension's purpose and usage needed clarification — moving it to the correct location is a natural follow-on. Placing extensions on semantically inappropriate backbone elements creates confusion for implementers and tooling. STU2 is the appropriate time to correct this, and the change is straightforward.
