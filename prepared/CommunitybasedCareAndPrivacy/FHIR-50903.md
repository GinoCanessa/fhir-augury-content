## Summary

The reporter (from Nictiz / HL7 Netherlands) asks which FHIR core resource best represents a **TreatmentDirective** — a joint decision between a health professional and a patient (or representative) about the desirability of performing specific treatments (e.g., ICU admission, CPR, artificial respiration). The Dutch "zib" (health information building block) [TreatmentDirective2](https://zibs.nl/wiki/TreatmentDirective2-v1.0(2020EN)) models three outcomes: *Execute*, *Do not execute*, and *Other* (conditional execution).

The core question is whether **Consent** or **ServiceRequest** is the correct resource:
- **Consent** represents legal directives and agreements; IPS uses it for advance directives.
- **ServiceRequest** is actionable and supports `doNotPerform`; PACIO ADI IG uses it for DNR orders.
- Neither resource natively supports three-valued outcomes — both are binary (permit/deny or doNotPerform boolean), requiring an extension for the conditional "other" option.

The issue was discussed at the **May 2025 WGM in Madrid** (Patient Care session), where the resource landscape was mapped: Goal for informal wishes, ServiceRequest for DNR orders (PACIO), Consent for legal directives (IPS), and Observation for patient preferences (PACIO). A harmonization gap between PACIO (ServiceRequest) and IPS (Consent) was identified, and Rob Hausam was informed.

As of September 2025, the reporter's team chose **Consent for R4** (documented on [Simplifier](https://simplifier.net/nictiz-r4-zib2020/documentation-zib-treatmentdirective2)), noting it worked "well enough" in testing, but expressed hope for a better solution in R6.

## Details

### Background — The Dutch TreatmentDirective Model
The zib TreatmentDirective2 captures a clinical agreement between provider and patient about specific treatments. It goes beyond simple advance directives: it includes a three-valued `TreatmentPermitted` element (Yes / Yes-But / No), the specific treatment, verification status, and links to advance directives. The "Yes-But" (JA_MAAR) conditional scenario is the key challenge — neither Consent (permit/deny) nor ServiceRequest (doNotPerform boolean) can express it without an extension.

### Resource Evaluation from WGM Madrid (May 2025)
| Resource        | Use Case                            | IG Usage           |
|-----------------|-------------------------------------|--------------------|
| Goal            | Informal wishes ("die at home")     | —                  |
| ServiceRequest  | DNR orders, Do Not Perform          | PACIO ADI          |
| Consent         | Legal directives, advance directives| IPS                |
| Observation     | Patient preferences                 | PACIO ADI, US Core |

### Harmonization Issue
PACIO ADI uses ServiceRequest for portable medical orders (CPR, intubation, etc.) while IPS uses Consent for advance directives. The September 2024 WGM in Atlanta reached consensus (documented in [FHIR-10028](https://jira.hl7.org/browse/FHIR-10028)) that:
- **Person-authored** treatment preferences → Observation
- **Practitioner-authored** portable medical orders → ServiceRequest
- Administrative/documentation elements → Observation

This consensus does not directly address the Dutch TreatmentDirective use case, which is a *joint* decision (neither purely patient-authored nor purely practitioner-ordered).

### Reporter's Current Decision (Sept 2025)
Nictiz adopted Consent for R4, reasoning that while ServiceRequest is more actionable and PACIO uses it, the zib model is more elaborate than either resource can fully cover. The Consent resource's stated scope includes "Advance Care Directives" even though that use case is not yet fully modeled.

## Proposed Dispositions

### Disposition A: Provide Guidance in Consent Resource Documentation (Recommended)
**Action:** Add guidance to the Consent resource documentation acknowledging the TreatmentDirective use case and clarifying how Consent can model three-valued treatment decisions (permit, deny, permit-with-exceptions via `Consent.provision`). Reference the PACIO ADI IG's ServiceRequest approach as an alternative for practitioner-authored portable medical orders.

**Rationale:** This is a Question-type ticket seeking guidance. The reporter's team has already adopted Consent for R4 with a workable pattern. Documenting this pattern in the core spec would help other implementers facing the same challenge, especially given the demonstrated need from the Netherlands, and would begin to address the PACIO/IPS harmonization gap. This aligns with the Consent resource's stated (but unmodeled) scope of "Advance Care Directives."

### Disposition B: Create a Joint Cross-WG Harmonization Task
**Action:** Convert this ticket to a Change Request and coordinate with Patient Care, Orders & Observations, and PACIO to define a harmonized approach for treatment directives across Consent, ServiceRequest, and Observation. Target R6 for a unified modeling pattern that accommodates both person-authored and practitioner-authored directives, including three-valued outcomes.

**Rationale:** The ticket reveals a real harmonization gap: PACIO uses ServiceRequest, IPS uses Consent, and neither fully covers the joint-decision use case. The Sept 2024 WGM consensus (FHIR-10028) addressed person-authored vs practitioner-authored scenarios but not joint decisions. A cross-WG effort for R6 could produce a definitive answer.

### Disposition C: Decline — Out of Scope for Core
**Action:** Close as "Resolved — Not a Problem" with a pointer to the PACIO ADI IG and IPS for implementation guidance. Advise the reporter to work within the existing IG ecosystem.

**Rationale:** The question is about implementation guidance for a specific national use case (Dutch zibs). The reporter's team has found a workable R4 solution using Consent. Core spec changes may not be warranted when the issue can be addressed through IGs and national profiles. However, this leaves the PACIO/IPS harmonization gap unaddressed.

## Recommendation

**Disposition A: Provide Guidance in Consent Resource Documentation.**

This ticket is a well-documented, community-driven question that highlights a genuine gap in the FHIR core spec — the Consent resource claims to cover advance care directives but lacks concrete modeling guidance. The reporter and the Netherlands community have done substantial work analyzing the options, presenting at WGM, and documenting their decision. Adding guidance to the Consent resource would be a low-risk, high-value improvement that validates the reporter's approach, helps future implementers, and takes a first step toward the larger harmonization needed between PACIO and IPS. The cross-WG harmonization work (Disposition B) is valuable but should be pursued as a separate, broader initiative rather than blocking resolution of this specific question.
