## Description

The ballot commenter (Gary Dickinson) notes that it is not clear if or how provenance is represented in the US Breast Cancer Radiology Reporting FHIR Implementation Guide. The comment argues that provenance — who, what, when, where, why — is essential at two key capture points:

1. **Point of origination** — at the source, for newly captured content
2. **Point of update** — for newly updated content, while preserving previous content and its provenance

The comment enumerates specific provenance dimensions:

- **Who**: subject of care (patient), performer, author, organization
- **What**: action taken
- **When**: date/time of action, date/time of information capture or update
- **Where**: physical location of action, physical location of capture/update, network address / device ID
- **Why**: rationale or purpose of action, rationale or purpose of capture/update

The comment also notes that provenance granularity (e.g., binding authorship to content) may be at the dataset or element level.

## Resolution Description

> We agree that these elements are not well defined in this ballot, and intend to define them more thoroughly in the next ballot.

The work group agreed with the commenter that provenance elements were inadequately defined and committed to addressing them in a subsequent ballot cycle.

## Proposed Dispositions

### Disposition A: **Persuasive** (Confirm existing resolution)

Accept the existing "Persuasive" resolution as-is. The work group already agreed the provenance elements were insufficiently defined and committed to defining them more thoroughly in the next ballot. This disposition affirms that commitment and keeps the ticket in its current state, expecting the IG authors to add provenance guidance (e.g., referencing the FHIR Provenance resource, clarifying which resource elements carry provenance semantics, and/or profiling US Core Provenance) in the next published version.

**Pros:** Aligns with the work group's own prior assessment; acknowledges a genuine gap; consistent with how other IGs (PCDE, mCODE) handled the same comment.
**Cons:** The ticket has been in "Triaged" status since 2019 with no visible follow-through; confirming the resolution without verifying implementation may leave the gap unaddressed.

### Disposition B: **Persuasive with Modification** — Scope the provenance requirements

Accept the comment as persuasive but modify the scope of the response. Rather than committing to define all provenance dimensions enumerated by the commenter (which is a broad, cross-cutting concern), the IG should:
1. Reference the FHIR Provenance resource and US Core Provenance profile
2. Clarify which key clinical events in breast cancer radiology reporting require provenance tracking (e.g., diagnostic report authorship, observation performers)
3. Note that infrastructure-level provenance (network addresses, device IDs) is outside the IG's scope

**Pros:** Provides actionable, bounded guidance; avoids over-burdening a domain-specific IG with infrastructure concerns; gives implementers concrete direction.
**Cons:** May not fully satisfy the commenter's expansive provenance framework; requires additional specification work.

### Disposition C: **Not Persuasive** — Provenance is a cross-cutting concern

Reject the comment on the grounds that provenance is a cross-cutting infrastructure concern addressed by the base FHIR Provenance resource and US Core Provenance profile, and does not need to be re-specified in every domain IG. The Breast Cancer Radiology Reporting IG can rely on these foundational resources without duplicating provenance guidance. This is consistent with how Da Vinci PAS (FHIR-24431) and SDOH Clinical Care (FHIR-31088) handled the same comment.

**Pros:** Avoids specification bloat; consistent with the principle that domain IGs should not re-specify base infrastructure; matches "Not Persuasive" outcomes for the same comment on other IGs.
**Cons:** Contradicts the work group's own prior "Persuasive" resolution on this ticket; the commenter's concern about unclear provenance representation may remain valid for implementers of this IG specifically.

---

## Recommendation

**Disposition B: Persuasive with Modification**

This is the most balanced approach. The original "Persuasive" resolution acknowledged a real gap, but the comment's provenance framework is extremely broad and includes infrastructure-level elements (network addresses, device IDs) that are outside the scope of a domain-specific radiology reporting IG. Disposition B honors the valid concern by directing the IG to reference established provenance mechanisms (FHIR Provenance resource, US Core Provenance profile) and clarify provenance expectations for the key clinical events in breast cancer radiology reporting, without imposing an unbounded specification burden. This approach is also consistent with the "Persuasive with Modification" outcomes on the closely related mCODE (FHIR-24457) and PCDE (FHIR-24590) tickets from the same commenter.
