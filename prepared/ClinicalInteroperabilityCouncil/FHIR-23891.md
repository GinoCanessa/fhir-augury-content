## Summary

The reporter (Craig Newman) identifies multiple issues with the profiles page in the US Breast Cancer Radiology Reporting IG (2019SEP ballot). Key problems:

1. **Missing profiles on the profiles page**: The Condition (`obf-Condition`), ActionRequest (`obf-ActionRequest`), Procedure (`obf-Procedure`), and MedicationStatement (`obf-MedicationStatement`) profiles are not listed on the profiles page, even though they can be part of the Composition and related resources.
2. **Broken/unclear references**: `Composition.relatesTo.targetReference` references an `obf-Composition` profile whose hyperlink does not work. It is unclear whether this should instead be `breastrad-BreastRadiologyDocument`.
3. **Poorly defined profiles**: The `obf-Observation` profile (used for Patient Risk observations) lacks adequate narrative text or constraints to explain its usage.

A follow-up comment by the reporter adds that many profiles have overly generic descriptions (e.g., the Condition profile listing an exhaustive range of conditions without clarifying scope for this use case), and that constraints lack usage guidance (e.g., `Condition.stage` constrained to 0..1 without explaining how staging should be used).

## Resolution Description

> This is an artifact of the CIMPL logical model. For the next ballot we will not be using these logical models and instead adhere more closely to the base FHIR resources.

## Details

The ticket targets the September 2019 ballot of the US Breast Cancer Radiology Reporting IG. The IG at that time used the Clinical Information Modeling Profiling Language (CIMPL) to generate logical models and FHIR profiles. This approach produced profiles with the `obf-` prefix (Onco-FHIR Base) that were often confusingly documented — mixing logical model artifacts with actual FHIR resource profiles. The resolution acknowledges this was a problem caused by the CIMPL tooling and commits to abandoning that approach in favor of profiles that more closely follow base FHIR resources.

### FHIR Resources Mentioned
- `Composition.relatesTo.targetReference`
- `Condition` (including `Condition.stage`)
- `Observation`
- `Procedure`
- `MedicationStatement`
- `ActionRequest` (non-standard resource from CIMPL)

## Proposed Dispositions

### Disposition A: Accept (Applied — Persuasive)

**Action:** Accept the existing resolution as Persuasive and confirm the ticket is applied.

The resolution acknowledges the problem — the CIMPL-generated logical model artifacts produced missing, poorly linked, and generically defined profiles. The commitment to abandon CIMPL logical models and adhere more closely to base FHIR resources directly addresses all issues raised:

- Missing profiles will be resolved by restructuring around standard FHIR resource profiles
- Broken `obf-` prefix references will be eliminated
- Generic descriptions will be replaced with use-case-specific profile documentation

This aligns with the resolutions of related tickets FHIR-23885, FHIR-25077, and FHIR-25078, all of which were resolved persuasively with the same directional change. Mark as Applied.

### Disposition B: Accept with Additional Guidance (Persuasive with Modification)

**Action:** Accept as Persuasive with Modification, adding a requirement that the next ballot explicitly include a profiles index page listing all profiles referenced by the Composition, with working hyperlinks and usage narrative for each.

While the resolution to abandon CIMPL is correct, it does not specifically commit to resolving the documentation quality issues (generic descriptions, missing usage guidance for constraints like `Condition.stage` 0..1). A modification could require that:
1. All profiles referenced from Composition sections must appear on the profiles listing page
2. Each profile must include use-case-specific description text
3. Constrained elements must include usage guidance

### Disposition C: Decline (Not Persuasive)

**Action:** Decline the ticket as Not Persuasive on the grounds that the issues are ballot-specific artifacts that were inherently resolved by the IG's subsequent restructuring.

The problems described are specific to the September 2019 ballot's CIMPL-generated content. Since the IG has since been fundamentally restructured (the `obf-` prefix profiles no longer exist), these issues are moot. No formal resolution tracking is necessary for problems that disappeared with the architectural change.

## Recommendation

**Disposition A: Accept (Applied — Persuasive)**

The existing resolution is appropriate. The core issue — CIMPL logical model artifacts creating confusing, missing, and poorly documented profiles — was clearly acknowledged and the remedy (abandoning CIMPL in favor of standard FHIR resource profiles) directly addresses every concern raised. All three closely related tickets (FHIR-23885, FHIR-25077, FHIR-25078) reached the same conclusion. The resolution is clear, actionable, and has already been implemented in subsequent ballots of the IG. The ticket should be marked as Applied.
