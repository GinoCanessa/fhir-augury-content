## Summary

This ticket requests that Da Vinci implementation guides explicitly state the conformance expectations that are implicit in data element definitions. The submitter (Lloyd McKenzie) observes that most Da Vinci guides define data structures with data element definitions that are not framed as formal SHALL statements, yet there is an implicit expectation that systems must populate these structures with data matching the definitions. The request is to make this implicit expectation explicit and to clarify that different levels of responsibility apply (e.g., data sources vs. consumers).

The work group resolved this as **Not Persuasive**, stating: *"We do not feel the need to add language that states these sort of implicit statements."*

## Details

### Description

Most Da Vinci guides leverage data structures that define data elements. These definitions typically aren't framed as "SHALL" statements, however there's an implicit expectation that systems need to populate the data structures with data that matches the definitions. The ticket asks to make that implicit expectation explicit and to make clear that there are different levels of responsibility.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

Add a paragraph to the HRex Conformance Expectations page stating that data element definitions in Da Vinci profiles carry implicit conformance expectations: data sources are expected to populate elements with data matching the definitions when that data is available and relevant, while data consumers are expected to be capable of processing any defined element. Include a note clarifying the levels of responsibility (e.g., data source SHALL be capable of populating defined elements; data consumer SHALL NOT reject or fail when encountering defined elements).

**Rationale:** Making implicit expectations explicit reduces ambiguity for implementers. FHIR-48653 established precedent by successfully adding similar clarifying conformance language to the same page. The concern raised in FHIR-51085 about Must-Support being too restrictive further demonstrates that implementers struggle with what is and isn't expected. Explicit language would reduce misinterpretation.

**Trade-off:** Risks over-specifying behavior that is already understood by experienced implementers. Could create unintended conformance obligations if the language is not carefully scoped.

### Disposition B: Not Persuasive with Modification

Decline the specific request to add blanket conformance language about data element definitions, but add a brief informational note to the HRex Conformance Expectations page acknowledging that data element definitions convey semantic expectations about how data should be structured and populated, without elevating those expectations to formal conformance requirements. Refer implementers to the individual profile definitions for specific conformance obligations.

**Rationale:** This approach acknowledges the concern without creating new SHALL-level requirements that could be problematic across the diverse range of Da Vinci profiles. As the work group noted, the current implicit understanding may be sufficient, but a clarifying note could help implementers new to Da Vinci guides without creating compliance burdens.

**Trade-off:** May not go far enough to address the ambiguity that prompted this ticket, but avoids the risk of unintended conformance implications.

### Disposition C: Decline (Not Persuasive)

Affirm the current resolution. The existing data element definitions, combined with profile-level Must-Support flags and existing conformance language, are sufficient. Adding additional conformance language about implicit expectations risks over-specification and could create unintended compliance obligations across different Da Vinci guides with different use cases.

**Rationale:** The work group has already considered this and concluded that such language is unnecessary. Da Vinci profiles already use Must-Support and formal SHALL statements where specific conformance is required. Adding blanket statements about implicit expectations could conflict with ongoing work to refine Must-Support definitions (FHIR-51085) and actor-specific conformance (FHIR-53227).

**Trade-off:** Leaves the ambiguity that the submitter identified unaddressed, potentially leading to inconsistent implementation interpretations across Da Vinci guides.

## Recommendation

**Disposition C: Decline (Not Persuasive)**

The work group's original resolution is sound. The request addresses a real concern about implicit expectations, but the proposed solution — adding blanket conformance language about data element definitions — is too broad and risks creating unintended compliance obligations. The Da Vinci ecosystem already has mechanisms for expressing conformance expectations (Must-Support, SHALL statements, profile definitions), and the related ticket FHIR-51085 is actively working to refine Must-Support semantics in HRex. Adding generalized language now could conflict with or complicate that effort. The existing resolution appropriately declines to add language for what implementers should already understand from the data element definitions themselves.
