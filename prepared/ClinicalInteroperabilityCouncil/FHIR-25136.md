## Description

Ballot comment #87 on the US Breast Cancer Radiology Reporting specification. The commenter (Mitra Rocca) identified that the specification mentions "ANTLR4" without providing a reference or citation. The existing wording simply states "ANTLR4" without explaining what it is or where to find more information about it.

**Existing Wording:** ANTLR4

**Comment:** A reference is needed.

## Resolution Description

> We agree. In the next cycle we will make sure that all references to existing tooling are included.

The resolution is **Persuasive** — the work group agreed that a reference for ANTLR4 (and all tooling references) should be included in the specification.

## Analysis

This is a straightforward documentation quality issue from the 2019 ballot cycle. The commenter correctly identified that the specification references ANTLR4 (a parser generator tool used in the specification tooling) without providing a citation or explanation. The work group agreed with the comment and resolved it as Persuasive, committing to include all tooling references in the next cycle.

Notably, FHIR-27486 addressed a nearly identical issue — adding an acronym description and reference for ANTLR4 — in the Clinical Quality Language specification, and that ticket has been successfully published. This demonstrates that the community recognizes this as a valid documentation gap.

The ticket remains in "Triaged" status despite being resolved as Persuasive in 2019, suggesting the actual implementation (adding the ANTLR4 reference to the specification) may still be pending.

## Proposed Dispositions

### Disposition A: Apply as Resolved (Persuasive)
**Action:** Add a reference and brief description for ANTLR4 (e.g., "ANTLR4 — ANother Tool for Language Recognition, version 4; see https://www.antlr.org/") wherever it is mentioned in the specification. Follow the precedent set by FHIR-27486 in the CQL specification.
**Rationale:** The work group already agreed with the ballot comment. The fix is minimal (adding a citation/reference) and improves specification clarity. FHIR-27486 demonstrates a successful resolution pattern.
**Risk:** Very low. This is a documentation-only change.

### Disposition B: Apply with Broader Scope
**Action:** Conduct a full audit of the specification for all unreferenced tooling mentions (ANTLR4, CIMPL, etc.) and add references for each, addressing this ticket alongside FHIR-25135 and similar documentation gaps in a single pass.
**Rationale:** Multiple ballot comments from the same cycle (FHIR-25135, FHIR-25132) identify similar reference/link issues. A comprehensive pass would resolve multiple outstanding items efficiently.
**Risk:** Low, but requires more effort and could delay resolution if scope grows.

### Disposition C: Close as Resolved – No Further Action
**Action:** Verify whether the ANTLR4 reference has already been added in subsequent publication cycles. If the reference is now present, close the ticket as already addressed.
**Rationale:** The ticket was resolved Persuasive in 2019 and may have been addressed in intervening publication cycles without the ticket status being updated.
**Risk:** If the reference was not actually added, this would leave the gap unresolved.

## Recommendation

**Disposition A: Apply as Resolved (Persuasive).** This is a clear, agreed-upon documentation fix. The work group already accepted the comment as Persuasive. Add an ANTLR4 reference and description following the pattern established by FHIR-27486 in the CQL specification. The change is low-risk, well-precedented, and directly addresses the ballot commenter's concern.
