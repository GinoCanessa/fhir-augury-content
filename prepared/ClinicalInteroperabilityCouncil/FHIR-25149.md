## Summary

Susan Matney submitted a change request against the US Breast Cancer Radiology Reporting (FHIR) specification (ballot comment BC #100) requesting that the resource reference type be constrained to include at least a breast radiology report rather than allowing an open `Reference(Resource)`. The ticket was triaged and labeled for discussion at a Working Group Meeting but remains unresolved. The resolution description notes "We will discuss this."

## Details

**Description:**

The submitter notes that the specification should include at least a breast radiology report reference type rather than an unconstrained `Reference(Resource)`. The original ballot comment states:

> Should include at least breast radiology report not Reference(Resource)

This indicates that a profile or element within the US Breast Cancer Radiology Reporting IG currently allows a reference to any resource type, where it should be constrained to at minimum reference a breast radiology report profile or DiagnosticReport resource.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Constrain the `Reference(Resource)` element in the relevant profile(s) of the US Breast Cancer Radiology Reporting IG to require at minimum a reference to the breast radiology report profile (e.g., `Reference(BreastRadiologyReport)` or an appropriate DiagnosticReport profile), rather than allowing an open reference to any resource.

#### Justification

An unconstrained `Reference(Resource)` in a breast cancer radiology reporting context undermines interoperability by allowing any resource type in a slot that is clearly intended for a radiology report. Tightening the reference type is consistent with profiling best practices and ballot comment intent. This is a straightforward constraint that improves data quality and implementer guidance.

---

### Disposition B: Alternative Approach

#### Proposal

Constrain the reference to a choice of relevant report profiles (e.g., `Reference(BreastRadiologyReport | DiagnosticReport)`) to permit both the specialized breast radiology report profile and the base DiagnosticReport resource, providing flexibility while still narrowing the reference from any resource.

#### Justification

A broader but still constrained reference type allows implementations that have not yet adopted the full breast radiology report profile to remain conformant while still preventing nonsensical references (e.g., to a Patient or Medication resource). This balances the submitter's concern with practical implementability and backward compatibility.

---

### Disposition C: Decline

#### Proposal

Decline the change request and retain the current `Reference(Resource)` type, noting that the specification has not been actively updated since 2021 and the US Breast Cancer Radiology Reporting IG may need broader review before individual constraint changes are applied.

#### Justification

The ticket has been open since 2019 and remains unresolved with no substantive discussion. Making isolated profiling changes without a broader review of the IG's current status and alignment with the evolving FHIR ecosystem may introduce inconsistencies. If the IG is no longer being actively maintained, the change may have no practical effect.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because it addresses the submitter's valid concern — that an unconstrained `Reference(Resource)` is too permissive for a breast radiology reporting context — while providing a pragmatic path forward. Constraining the reference to `Reference(BreastRadiologyReport | DiagnosticReport)` narrows the type to clinically appropriate resources without being so restrictive that it breaks existing implementations. Given that the ticket has been open for several years without resolution, a balanced approach that acknowledges both the need for tighter profiling and the practical reality of implementation variability is most appropriate. This disposition honors the original ballot comment intent while being defensible from an interoperability standpoint.
