## Summary

This change request proposes that the Prostate Gleason Grade Group profile in mCODE should derive from a new "StageCategory" parent profile rather than the existing CancerStage profile. The submitter argues that per AJCC staging definitions, the Gleason Grade Group is a **non-anatomical category** used as input to determine the overall Prostate Cancer Stage Group, rather than being a stage group itself. Additional non-TNM stage categories (PSA for prostate; ER, PR, HER2, OncotypeDx for breast; HPV for oropharynx) would also benefit from a dedicated StageCategory parent profile that could reference the underlying lab or genomic observations.

The ticket was previously resolved as "Considered for Future Use" in March 2024 (vote 6-0-0, Jimmy Tcheng/Bruce Bray) with a note that it should be evaluated as part of the mCODE STU4 reconciliation process alongside broader cancer staging design feedback. It was subsequently re-opened in February 2025 post-STU4.0.0 publication for further consideration.

## Details

### Description

The submitter identifies that determining the Prostate Cancer Stage Group requires two additional categories beyond TNM: the **Gleason Grade Group** and the **PSA categorical value** (not the actual lab value). Per AJCC definitions, the Gleason Grade Group is a non-anatomical category rather than a stage group — a distinction already partially reflected in the current profile modeling (it does not contain reference Observations).

The proposal is to create a **StageCategory** parent profile to capture non-TNM categories of StageGroup, including:
- **Prostate**: G (Gleason Grade Group), PSA categorical value
- **Breast**: ER, PR, HER2, G, OncotypeDx
- **Oropharynx**: HPV

This StageCategory profile could reference the lab/genomic profiles from which the raw data originates, while keeping those as separate resources (recognizing that oncology-specific EMRs may have access to stage categories without the underlying lab values).

Reference: [AJCC Staging Rules 8th Edition (PDF)](https://www.facs.org/media/2zvlrdmi/ajcc_staging_rules_8th_ed.pdf)

## Proposed Dispositions

### Disposition A: Accept — Create a StageCategory Parent Profile

**Action:** Create a new `StageCategory` abstract parent profile (based on Observation) in mCODE to represent non-anatomical stage input categories. Re-parent the `Prostate Gleason Grade Group` profile from `CancerStage` to `StageCategory`. Design `StageCategory` to:
- Allow `hasMember` references to lab and genomic observation profiles providing the underlying raw data.
- Include value set bindings appropriate for categorical staging inputs (as distinct from stage group results).
- Serve as the parent for future non-TNM category profiles (e.g., PSA categorical value, ER/PR/HER2 status, HPV status, OncotypeDx score).

**Rationale:** This aligns the profile hierarchy with AJCC staging semantics, where Gleason Grade Group is a category input to staging rather than a stage group itself. The existing model conflates stage categories with stage groups. A dedicated parent profile creates a clearer and more extensible modeling pattern for the many cancer-type-specific non-anatomical staging inputs that exist across oncology.

**Impact:** Breaking change to the Gleason Grade Group profile's parent; implementers currently validating against CancerStage would need to update. New abstract profile adds complexity. Requires coordination with downstream IGs (e.g., Central Cancer Registry Reporting IG).

### Disposition B: Partial Adoption — Document the Distinction Without Re-parenting

**Action:** Keep `Prostate Gleason Grade Group` under the existing `CancerStage` parent profile hierarchy, but add explicit documentation and invariants distinguishing "stage category inputs" (like Gleason Grade Group) from "stage group results" (like TNM Stage Group). Add a `stage-role` extension or code to `CancerStage` to differentiate category inputs from stage group results.

**Rationale:** This addresses the semantic concern without introducing a breaking profile hierarchy change. Implementers can programmatically distinguish stage categories from stage groups using the extension/code. It preserves backward compatibility while acknowledging the AJCC distinction between categories and groups.

**Impact:** Non-breaking. Adds an optional extension or coded element. Less disruptive but does not achieve the clean separation the submitter envisions.

### Disposition C: Decline — Maintain Current Structure

**Action:** Resolve as Not Persuasive. Keep the current profile hierarchy where `Prostate Gleason Grade Group` is a child of `CancerStage`.

**Rationale:** The current structure is functional and already supported by implementers. mCODE's design philosophy is "minimal" — adding a new abstract parent profile adds complexity. The distinction between stage categories and stage groups, while semantically valid in AJCC terminology, does not materially affect interoperability or data exchange. The STU4 ballot cycle already received community input on staging design without this change being identified as a priority. FHIR-41003 already addressed the broader need for non-TNM staging in mCODE STU3.

**Impact:** No change. Maintains backward compatibility and simplicity.

---

## Recommendation

**Disposition B: Partial Adoption — Document the Distinction Without Re-parenting**

The submitter raises a valid semantic point grounded in AJCC staging definitions — the Gleason Grade Group is indeed a stage *category input* rather than a stage *group result*. However, introducing a new abstract parent profile (Disposition A) would be a breaking change with significant downstream impact, and the mCODE community has already deferred this twice (initially to STU4 reconciliation, then re-opened post-STU4 publication). Disposition B addresses the core semantic concern by making the distinction explicit through documentation and an optional coded element, without disrupting the existing profile hierarchy. This pragmatic approach acknowledges the AJCC-based classification while preserving implementer stability, and it could serve as a stepping stone toward the fuller refactoring proposed in Disposition A if community feedback in a future ballot cycle supports it.
