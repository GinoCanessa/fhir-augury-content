## Description

The ballot comment raises a naming/cardinality inconsistency in the US Breast Cancer Radiology Reporting IG. The element `breastrad-PriorReports` uses a plural name ("Reports"), yet its cardinality is constrained to `0..1`, which supports only a single entry. The commenter asks whether the cardinality should be changed to `0..*` to match the plural naming convention.

## Resolution Description

The resolution was marked **Persuasive**. The ballot committee agreed that `breastrad-PriorReports` should have cardinality `0..*` (rather than `0..1`) to align with the plural naming. The change was committed to be made in the next ballot.

## Analysis

This is a straightforward naming-consistency issue. The plural element name `breastrad-PriorReports` implies it can hold multiple prior report references, but the cardinality of `0..1` contradicts that expectation. The original ballot committee already agreed (Persuasive resolution) that the cardinality should be corrected to `0..*`. The fix is small, well-defined, and aligns with standard FHIR profiling conventions around naming and cardinality coherence.

The ticket status remains **Triaged** despite being resolved as Persuasive in November 2019, which suggests the change may not yet have been formally applied in a published ballot.

## Proposed Dispositions

### Disposition A: Apply as Resolved (Persuasive) — Update cardinality to `0..*`
**Action:** Change the cardinality of `breastrad-PriorReports` from `0..1` to `0..*` to match its plural naming.
- **Rationale:** The original ballot committee already voted this Persuasive and agreed the fix is appropriate. The plural name clearly implies multiple entries should be allowed. This is a correctness fix that improves clarity for implementers.
- **Risk:** Low. This widens a constraint (from single to multiple), so existing conformant instances remain valid.

### Disposition B: Apply as Persuasive with Modification — Rename to singular instead
**Action:** Keep cardinality at `0..1` but rename the element from `breastrad-PriorReports` to `breastrad-PriorReport` (singular).
- **Rationale:** If the clinical intent is truly to reference only one prior report, the naming should be corrected rather than the cardinality. This preserves the original constraint.
- **Risk:** Medium. Renaming breaks existing implementations that reference the plural name. Requires assessment of whether a single prior report reference is clinically sufficient.

### Disposition C: Close as Already Addressed
**Action:** Verify the change has already been applied in a subsequent ballot and close the ticket.
- **Rationale:** The ticket was resolved Persuasive in November 2019 and may have been applied in later ballot cycles (the spec has had subsequent ballot activity through 2025). If the fix is already in place, no further action is needed.
- **Risk:** Low, but requires verification against the current published IG.

## Recommendation

**Disposition A: Apply as Resolved (Persuasive) — Update cardinality to `0..*`**

This is the disposition already agreed upon by the ballot committee. The plural name `breastrad-PriorReports` clearly signals multiple entries, and expanding the cardinality from `0..1` to `0..*` is the correct fix. It is non-breaking (existing single-entry instances remain valid under `0..*`) and aligns with FHIR naming conventions. If the workgroup confirms the change has already been applied in a later ballot, Disposition C would apply instead.
