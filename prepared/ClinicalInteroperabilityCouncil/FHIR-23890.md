## Summary

This ticket raises concerns about the lack of clarity regarding Patient and RelatedPerson references in author, attester, asserter, recorder, and performer elements within the US Breast Cancer Radiology Reporting profiles. The reporter notes that several resources (Composition, Condition, Observation) permit Patient or RelatedPerson references, but the implementation guide does not explain whether these are valid scenarios—such as self-reported breast exam data—or whether the profiles should be further constrained to exclude them. The resolution was marked Persuasive, with the authors acknowledging the elements were not well defined and committing to more thorough definitions in a subsequent ballot.

## Details

**Description:**

There are several resource author or attester elements which allow Patient or RelatedPerson references. It's not clear from the introduction what sources of data are expected to be included in a breast radiology report. Should the authors and attesters in Composition allow Patient and RelatedPerson? Should the asserter or recorder in Condition allow Patient and RelatedPerson? Should the performer in the breast abnormality Observation allow Patient and RelatedPerson? If these aren't valid scenarios, the profiles should be further constrained. If they are valid scenarios (eg. Self breast exam data), then a description of the valid scenarios should be included for implementers.

**Resolution Description:**

We agree that these elements are not well defined in this ballot, and intend to define them more thoroughly in the next ballot.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Constrain the Composition.author, Composition.attester, Condition.asserter, Condition.recorder, and Observation.performer elements in the breast radiology profiles to exclude Patient and RelatedPerson reference types where they are not clinically appropriate. For any elements where patient- or related-person-reported data is a valid scenario (e.g., self breast exam findings), add explicit guidance in the implementation guide narrative describing the expected use cases and how implementers should handle such references.
#### Justification
The original request is well-reasoned: profiles that permit reference types without explaining their intended use create ambiguity for implementers. Constraining unused reference types and documenting valid scenarios is consistent with FHIR profiling best practices and improves interoperability by reducing optionality.

---

### Disposition B: Alternative Approach
#### Proposal
Rather than constraining the reference types at the profile level, add an informative section to the implementation guide that enumerates which elements permit Patient and RelatedPerson references, explains the clinical scenarios where these are expected (e.g., patient-reported breast self-exam data), and notes which elements are not expected to carry such references in typical radiology workflows. Profiles would remain unchanged, preserving flexibility for edge cases.
#### Justification
This approach avoids breaking changes for implementers who may already be using the broader reference types, while still addressing the clarity concern. It provides implementer guidance without imposing hard constraints that could limit future extensibility or require additional profiling work.

---

### Disposition C: Decline
#### Proposal
Decline on the basis that this is already resolved. The resolution was marked Persuasive, the authors acknowledged the gap, and the work was deferred to the next ballot cycle.
#### Justification
The ticket was resolved as Persuasive in November 2019 with a commitment to address the definitions in a future ballot. Since the work group already accepted the feedback and planned corrective action, no further disposition is necessary at this time. Reopening the ticket would duplicate effort that was already triaged and acknowledged.

---

### Recommendation
**Recommended disposition:** C — Decline (Already Resolved)

This ticket was resolved as Persuasive over five years ago, with the authors explicitly agreeing that the elements were insufficiently defined and committing to improve them in the next ballot. The core concern—lack of clarity around Patient and RelatedPerson references—was accepted in full. Reopening the ticket for additional disposition would duplicate an already-acknowledged resolution. If the subsequent ballot did not adequately address the concern, a new ticket referencing this one would be the appropriate mechanism to track remaining gaps.
