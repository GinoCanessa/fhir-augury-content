This ticket requests alignment of the Human Services Directory FHIR IG with backwards-incompatible changes in the upcoming HSDS (Human Services Data Specification) version 3.0. The reporter (Greg Bloom, Open Referral) enumerates specific HSDS 3.0 changes and assesses their impact on the IG:

1. **Address merging** — `postal_address` and `physical_address` merged into `address`. Minimal IG impact since FHIR already merges these; only mappings need updating.
2. **Attribute renaming** — `service_attributes` and `other_attributes` renamed to `attributes`. Mapping-only change.
3. **Eligibility table removal** — Removed in favor of attributes. Currently unmapped in the IG, so very little effect.
4. **Organization identifier changes** — `Organization.tax_id` deprecated, replaced by new `organization_identifier` table. This could supply additional fields to FHIR `Organization.identifier`; mapping-only impact.
5. **Minor field name changes** — Mapping-only updates.

Additionally, HSDS 3.0 introduces a standardized JSON representation and API specification, which could simplify mappings by eliminating join conditions. Open Referral plans to produce an HSDS 3.0 profile to improve FHIR alignment by constraining HSDS fields and taxonomies to match FHIR value sets.

The ticket was previously resolved as "Considered for Future Use" (vote: 5-0-1, 2023-02-06), but was reopened post-STU1 publication by lynn_laakso on 2023-10-04 with the note that once HSDS 3.0 is published, analysis should surface necessary mapping and profile changes. HSDS Organization table changes were identified as the most significant impact.

## Details

### Reporter's Assessment of Impact

The reporter characterizes most changes as mapping-only updates that would not require changes to the IG's profiles or definitions. The most significant structural change is the `organization_identifier` table replacing `Organization.tax_id`, which could enrich FHIR `Organization.identifier` with additional fields.

## Proposed Dispositions

### Disposition A: Accept — Persuasive (Update Mappings for HSDS 3.0)

**Action:** Accept the change request and update the IG's HSDS-to-FHIR mappings to reflect the published HSDS 3.0 specification.

**Rationale:** HSDS 3.0 has been published since this ticket was created. The reporter's own analysis indicates most changes are mapping-only and would not alter the IG's profiles or definitions. The most significant change (`organization_identifier` replacing `Organization.tax_id`) could enrich `Organization.identifier` mappings. Updating mappings maintains the IG's value as a bridge between HSDS and FHIR data models. The ticket was previously deferred specifically pending HSDS 3.0 publication, which has now occurred.

**Scope:** Update all HSDS mapping tables and narrative text to reference HSDS 3.0 field names and structure (address, attributes, organization_identifier). Evaluate whether the new JSON representation and API specification should be referenced or used as the primary mapping target. Consider coordination with FHIR-40040 (contact alignment) which addresses another HSDS alignment gap.

### Disposition B: Persuasive with Modification — Phased Alignment with HSDS 3.0

**Action:** Accept the change request but scope the work to two phases: (1) update mappings for non-breaking HSDS 3.0 changes (address merge, attribute rename, field name changes) in the next ballot cycle, and (2) defer the `organization_identifier` enrichment and JSON API mapping alignment to a subsequent cycle for deeper analysis.

**Rationale:** While most changes are straightforward mapping updates, the `organization_identifier` table replacement and the new standardized JSON representation introduce opportunities for deeper structural alignment that may warrant additional community input. A phased approach ensures the IG stays current on non-controversial changes while allowing deliberate evaluation of more impactful enhancements.

**Scope:** Phase 1: Update mapping tables for address, attributes, eligibility, and minor field name changes. Phase 2: Analyze `organization_identifier` mapping to `Organization.identifier`, evaluate JSON API representation as a mapping target, and coordinate with Open Referral's planned FHIR profile of HSDS 3.0.

### Disposition C: Decline — Not Persuasive (Defer Pending HSDS FHIR Profile)

**Action:** Decline the change request and defer until Open Referral publishes their planned HSDS 3.0 FHIR profile.

**Rationale:** The reporter notes that Open Referral "aims to produce a profile of HSDS 3.0 that would help alignment with FHIR and make the mappings simpler to implement." Implementing mapping updates now, before that profile exists, risks double work if the HSDS FHIR profile introduces different mapping strategies or constrained field sets. Waiting for the HSDS FHIR profile would provide a clearer, more authoritative mapping target.

**Risk:** Further delay on a "Highest" priority ticket that has already been deferred once. The IG's mappings remain based on HSDS 2.x while implementers may already be adopting HSDS 3.0.

## Recommendation

**Disposition A: Accept — Persuasive (Update Mappings for HSDS 3.0)** is recommended.

The ticket was previously deferred specifically until HSDS 3.0 publication, which has now occurred. The reporter's detailed analysis demonstrates that most changes are mapping-only updates with minimal risk to the IG's profiles. This is a "Highest" priority ticket from a key stakeholder (Greg Bloom / Open Referral), and continuing to defer undermines the IG's stated purpose of bridging HSDS and FHIR. The companion ticket FHIR-40040 (contact alignment) is also Triaged/Unresolved and should be considered alongside this work. Waiting for the HSDS FHIR profile (Disposition C) introduces indefinite delay with no published timeline.
