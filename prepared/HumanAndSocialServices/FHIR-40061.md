## Summary

This change request proposes adding a new use case to the Human Services Directory IG's use case section: **analysis of aggregated services directory information** for purposes such as community asset mapping, evaluation of population-level health and well-being, and assessing the effectiveness of resource allocation for addressing social determinants of health.

The reporter (Greg Bloom) notes that this use case involves a different type of user—a researcher or analyst performing intelligence gathering, advocacy, or policy-making—and acknowledges that this use case was not originally specified, though the existing read-only query functions established for service discovery may serve it in a minimally viable capacity.

The proposed text for a new third bullet in the use case section reads:
> **Analysis of aggregated services directory information (for community asset mapping, etc) to assess the effectiveness of resource allocation for meeting communities' needs and addressing social determinants of health at a population-level.**

### History

This ticket was originally resolved as **"Considered for Future Use"** on 2023-03-06 with a vote of **6-0-0** (Serafina Versaggi / Chirag Bhatt). The rationale was that this use case was not introduced during original development, and the IG was scoped to read-only queries for service discovery. The workgroup noted this use case would be introduced during ballot reconciliation and marked for future consideration.

On 2023-10-04, **lynn_laakso** reverted that resolution and reopened the ticket post-STU1 publication for reconsideration.

## Details

This ticket is part of a coordinated set of change requests from Greg Bloom targeting revisions to the use case section of the Human Services Directory IG:

- **FHIR-40060** (Published/Persuasive): Clarification on primary use case(s)—consolidating use cases around "service discovery" and removing references to "closed-loop referrals."
- **FHIR-40061** (this ticket): Addition of aggregate data analysis use case.
- **FHIR-40062** (Published/Persuasive): Declaring additional functionalities out of scope (eligibility assessment, user feedback, federation).

The companion tickets (FHIR-40060 and FHIR-40062) were both resolved as **Persuasive** and published. This ticket was the only one in the group deferred as "Considered for Future Use" before being reopened.

A separate ticket, **FHIR-39917** (Published/Persuasive, reporter: Himali Saitwal), also called for revisions to the use case section, noting it was "poorly constructed" and needed further elaboration and workflow diagrams.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

**Action:** Accept the proposed text and add a third bullet to the use case section describing aggregate data analysis for community asset mapping, population-level health assessment, and resource allocation effectiveness.

**Rationale:** The companion tickets (FHIR-40060, FHIR-40062) were both accepted as Persuasive, and the use case section has already been revised in STU1. The aggregate analysis use case is a legitimate and increasingly important application of human services directory data. The proposed text accurately characterizes this use case as distinct from service discovery while acknowledging that existing read-only query capabilities may already support it minimally. Adding this use case does not require new technical capabilities—it documents an existing valid application of the IG's functionality. The ticket was specifically reopened post-STU1 for inclusion, indicating workgroup intent to address it.

### Disposition B: Accept with Modification (Persuasive with Mod)

**Action:** Accept the use case but add qualifying language indicating this is an acknowledged but not fully specified use case, and note that future versions may include enhanced query capabilities (e.g., aggregation operations, bulk data access) to better support this scenario.

**Rationale:** The reporter acknowledges that this use case "may not have been carefully considered on its own terms." While documenting it in the use case section is appropriate, the workgroup may want to temper expectations by noting that the current read-only query API is designed primarily for individual service discovery and may not be optimized for population-level aggregate analysis. Adding a note about potential future enhancements (such as aggregate query operations or bulk data export) would set appropriate expectations while still acknowledging the use case's validity.

### Disposition C: Decline (Not Persuasive)

**Action:** Maintain the original "Considered for Future Use" disposition and defer this use case to a future ballot cycle.

**Rationale:** The original 6-0-0 vote to defer was well-reasoned: the IG was designed for read-only service discovery queries, and aggregate analysis represents a fundamentally different interaction pattern. Simply listing this use case without specifying how the IG supports it may create confusion about the IG's intended scope. The workgroup could revisit this when the IG evolves to include capabilities more suited to population-level analysis (e.g., bulk data operations, aggregation endpoints), at which point the use case could be properly specified rather than merely listed.

---

## Recommendation

**Disposition B: Accept with Modification (Persuasive with Mod)**

The aggregate analysis use case is valid and important—documenting it acknowledges a real-world application of human services directory data. However, since the current IG's read-only query capabilities were not designed with this use case in mind (as the reporter acknowledges), adding qualifying language ensures the workgroup communicates clearly about what the IG does and does not support today. This approach aligns with the companion tickets' acceptance while maintaining the precision requested in FHIR-39917's call for better use case construction. It also positions future IG versions to expand support for this use case with appropriate technical specifications.
