## Summary

The `veteran-status` extension (defined in the US Military Service IG, STU1) currently has an extension context of `Element`, which means it can be applied to **any** resource or data type element in FHIR. This is overly permissive and semantically incorrect — veteran status is a property of a person (typically a Patient), not of arbitrary elements like `Condition.status` or `Address.postalCode`. The ticket requests that the context be constrained to `Patient` (and possibly `Practitioner` and `RelatedPerson`).

## Proposed Dispositions

### Disposition A: Persuasive — Constrain context to Patient only
- **Action:** Update the `veteran-status` extension StructureDefinition to set the context to `Patient` (i.e., `context.type = "element"`, `context.expression = "Patient"`).
- **Rationale:** Veteran status is inherently a patient-level concept. The original context of `Element` was an authoring error that allows nonsensical usage (e.g., on `Address.postalCode`). Constraining to `Patient` reflects the intended semantics and aligns with the origin of this extension from PID-27 (a Patient segment field). Most known usage in practice is on Patient resources.
- **Risk:** Users who have applied this extension to `Person` (as discussed in the Zulip thread) or other resources would be technically non-conformant. However, `Person` is rarely used in practice and this can be addressed via a separate extension or by expanding context in a future ballot if needed.

### Disposition B: Persuasive with Modification — Constrain context to Patient, Practitioner, RelatedPerson, and Person
- **Action:** Update the `veteran-status` extension StructureDefinition to set the context to `Patient`, `Practitioner`, `RelatedPerson`, and `Person`.
- **Rationale:** Veteran status is a person-level attribute that could reasonably apply to any person-type resource. The Zulip discussion shows a real-world request to use it on `Person`. Including `Practitioner` and `RelatedPerson` (as the reporter suggested) plus `Person` covers all reasonable person-type resources without opening the door to nonsensical usage on non-person elements.
- **Risk:** Slightly broader than strictly necessary, but still semantically sound. All four resources represent individual people.

### Disposition C: Not Persuasive — Leave the context as Element
- **Action:** No change.
- **Rationale:** The overly broad context is permissive but not harmful — validators will not reject the extension on Patient, and implementers can self-restrict. Changing the context in a published STU could break existing implementations that use the extension on non-Patient resources (such as the Person usage noted in Zulip). The IG could address this in the next major version instead.
- **Risk:** Continues to allow semantically meaningless usage of the extension (e.g., on `Encounter.dischargeDisposition`), which undermines interoperability and could confuse implementers.

---

## Recommendation

**Disposition B: Persuasive with Modification** — Constrain context to `Patient`, `Practitioner`, `RelatedPerson`, and `Person`.

This disposition best balances correctness with practical usage. The reporter (Lloyd McKenzie) explicitly suggested including `Practitioner` and `RelatedPerson`, and Zulip evidence shows real-world usage on `Person`. Constraining to all four person-type resources eliminates nonsensical usage while accommodating all reasonable use cases. This is a low-risk, high-clarity fix that aligns with the original intent of the extension and addresses the confirmed authoring error.
