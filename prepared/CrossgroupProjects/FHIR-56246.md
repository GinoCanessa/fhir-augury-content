## Summary

This ticket proposes a minor text clarification in the US Core Must Support guidance for complex elements. Specifically, the phrase "is represented in the structure" would be changed to "is present in an instance" in the `must-support.html#must-support---complex-elements` section. The goal is to make the language more precise: the current phrasing ("is represented in the structure") could be misread as referring to the profile structure definition itself, whereas the intended meaning is about actual FHIR resource instances. The proposed resolution is "Persuasive" — update as proposed. The change impact is classified as non-substantive (Clarification).

## Details

### Description

The ticket proposes clarifying one sentence in the Must Support – Complex Elements guidance:

**Current text:**
> If any sub-element is marked as Must Support [for a complex element] and the parent element is not… [and] the parent element ~~is represented in the structure~~, Servers **SHALL** support the sub-element(s) marked as Must Support.

**Proposed text:**
> If any sub-element is marked as Must Support [for a complex element] and the parent element is not… [and] the parent element ***is present in an instance***, Servers **SHALL** support the sub-element(s) marked as Must Support.

## Proposed Dispositions

### Disposition A: Accept (Persuasive — Update as Proposed)

Accept the proposed resolution as-is. Change "is represented in the structure" to "is present in an instance" in the Must Support – Complex Elements section (`must-support.html#must-support---complex-elements`).

**Rationale:** The change is a straightforward text clarification with no normative impact. The phrase "is represented in the structure" is ambiguous — it could refer to the StructureDefinition (profile) rather than actual resource data. Replacing it with "is present in an instance" makes the intent unambiguous: the Must Support obligation for sub-elements only applies when the parent element actually appears in a resource instance. This aligns with the broader Block Vote 3 effort to tighten conformance language throughout US Core v9.

### Disposition B: Accept with Broader Clarification

Accept the text change and additionally add a short parenthetical or note clarifying that "instance" refers to FHIR resource instances returned by a Server (e.g., "is present in a resource instance returned by the Server"), to further disambiguate for implementers who might confuse "instance" with "instance of the profile."

**Rationale:** While "is present in an instance" is an improvement, some readers may still find "instance" slightly ambiguous. A more explicit phrasing would be maximally clear, especially for implementers new to FHIR. However, this goes slightly beyond the original proposal and may introduce scope creep.

### Disposition C: Decline (Not Persuasive)

Reject the change, keeping the existing text "is represented in the structure" as-is.

**Rationale:** The existing text has been in place through prior US Core versions without reported confusion or implementation errors. The conformance behavior is well understood by the community. Changing text, even in a non-substantive way, introduces publication risk and diff noise without addressing a demonstrated problem.

## Recommendation

**Disposition A: Accept (Persuasive — Update as Proposed)**

This is a minimal, low-risk text clarification that improves precision. The phrase "is present in an instance" is unambiguously correct, aligns with how Must Support is described elsewhere in the spec, and fits naturally within the broader Block Vote 3 effort to standardize conformance language in US Core v9. The proposed resolution has already been triaged as Persuasive and is included in a block vote, indicating consensus. There is no reason to deviate from the proposed change.
