## Summary

This ticket requests that the US Vital Signs with Qualifying Extensions implementation guide consider specifying that `Observation.subject` should reference a Patient resource rather than allowing only display text. The reporter notes that some examples in the IG (e.g., the AverageBloodPressure example) include only a `display` value in `Observation.subject` without an actual resource reference. The suggestion is to add guidance or a constraint requiring a proper Patient reference, which would improve interoperability and data quality for vital signs observations.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Add a constraint to the US Vital Signs with Qualifying Extensions profiles specifying that `Observation.subject` must include a reference to a Patient resource (i.e., `Observation.subject` should have a `reference` element pointing to a Patient resource, not only a `display` value). Update all examples in the IG to include a proper Patient reference in `Observation.subject`.
#### Justification
Vital signs observations are inherently patient-centric. Requiring a Patient reference in `Observation.subject` aligns with US Core expectations (US Core Vital Signs profiles already require `Observation.subject` to reference US Core Patient). Ensuring examples consistently demonstrate proper references improves implementer guidance and prevents ambiguity about whether display-only subjects are acceptable.

---

### Disposition B: Alternative Approach
#### Proposal
Rather than mandating a Patient reference at the profile level, update only the examples to include proper Patient references and add non-normative guidance recommending that implementers populate `Observation.subject.reference` with a Patient resource reference. The profile constraint on `Observation.subject` would remain unchanged, preserving flexibility for edge cases (e.g., Group or Location subjects).
#### Justification
This approach addresses the immediate concern about misleading examples without introducing a breaking constraint. The base Observation resource allows `Observation.subject` to reference Patient, Group, Device, or Location. Adding guidance rather than a hard constraint respects this flexibility while steering implementers toward the expected pattern for vital signs.

---

### Disposition C: Decline
#### Proposal
Decline the request as out of scope for this ballot cycle, noting that `Observation.subject` constraints are appropriately managed by US Core profiles which already require Patient references for vital signs.
#### Justification
The US Core Vital Signs profile, which this IG builds upon, already requires `Observation.subject` to reference a US Core Patient. Adding a redundant constraint in this IG would be duplicative. The example issue (display-only subject) is an editorial matter that can be addressed separately without a formal change request. The existing profile inheritance chain already enforces the desired behavior.

---

### Recommendation
**Recommended disposition:** C — Decline
The US Core Vital Signs profiles, upon which the US Vital Signs with Qualifying Extensions IG is built, already require `Observation.subject` to reference a Patient resource. Adding this constraint again in the derived profiles would be redundant. The substantive concern raised — that examples show only display text without a proper reference — is valid but is an editorial/example quality issue rather than a profile constraint gap. The examples should be corrected as part of routine editorial cleanup, which does not require a formal change request to the profile itself. Declining the formal constraint change while acknowledging the example issue as an editorial fix provides the cleanest resolution.
