## Description

All of the value set bindings for the `.component` slices in the Military Service Episode profile are set to **Required** binding strength. The submitter argues this may render the profile unusable for non-US settings, since the value sets are US-specific. A binding strength of **Extensible** or **Preferred** would make the IG more implementable outside of the US.

## Resolution Description

> Will update binding

The resolution was marked **Persuasive**, indicating the work group agreed the bindings should be loosened.

## Analysis

### Context

This ticket is part of a cluster of ballot comments (all from Craig Newman, labeled Block-Vote-2) filed against the US Military Service (FHIR) IG during its September 2021 ballot cycle. The common theme across FHIR-33438, FHIR-33440, FHIR-33441, and FHIR-33442 is that the IG's profiles use **Required** binding strength on US-specific value sets, which prevents international adoption and limits extensibility.

The FHIR binding strength hierarchy matters here:
- **Required** = only codes from the specified value set are permitted (no exceptions)
- **Extensible** = codes from the value set should be used if applicable, but other codes are permitted if no suitable code exists
- **Preferred** = the value set is recommended but not enforced

For `.component` slices on the Military Service Episode (an Observation-based profile), Required bindings on US-centric code systems (e.g., PHINVADS occupation codes, US-specific discharge statuses) make the profile fundamentally unusable in other countries' military health systems.

### Implementation Note

The resolution ("Will update binding") is terse. The ticket status is **Triaged** (not Published), which means this change may not yet have been applied to the published IG. Sibling tickets FHIR-33440, FHIR-33441, and FHIR-33438 have already reached **Published** status, suggesting those binding/cardinality changes were applied. ## Proposed Dispositions

### Disposition A: Apply as Persuasive — Relax all `.component` slice bindings to Extensible

**Action:** Change all Required bindings on `.component` slice value elements in the Military Service Episode profile to **Extensible** binding strength.

**Rationale:** Extensible is the standard FHIR practice for profiles that want to encourage use of a specific value set while allowing codes from other systems when the value set doesn't cover the needed concept. This aligns with the resolution already applied to sibling tickets (FHIR-33440, FHIR-33438, FHIR-34109) and maintains interoperability while enabling international use. The work group already agreed this was Persuasive.

**Risk:** Low. Extensible is the most common binding strength for profiled value sets in FHIR IGs and is well-understood by implementers.

### Disposition B: Apply as Persuasive — Relax bindings to Preferred

**Action:** Change all Required bindings on `.component` slice value elements in the Military Service Episode profile to **Preferred** binding strength.

**Rationale:** Preferred is even more permissive than Extensible and places no conformance obligation on implementers to use codes from the specified value set. This would maximize flexibility for international adoption.

**Risk:** Medium. Preferred bindings provide less interoperability assurance than Extensible. Systems may use entirely different code systems with no overlap, reducing data comparability. This goes further than what the sibling tickets applied.

### Disposition C: Verify Already Applied — Confirm and close

**Action:** Verify whether the binding changes were already applied as part of the broader set of ballot reconciliation changes (since siblings FHIR-33438, FHIR-33440, FHIR-33441 are all Published). If applied, update status to Published. If not, apply the Extensible binding change and publish.

**Rationale:** Given the resolution date (2022-04-25) and the terse resolution description, it's possible the change was applied but the ticket status was not updated to Published. The work group should confirm the current state of the IG before taking further action.

**Risk:** Low. This is primarily an administrative/verification action.
