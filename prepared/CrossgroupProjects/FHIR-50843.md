## Description Summary

The submitter raises concerns about the semantic clarity of medication mappings in the US C-CDA on FHIR IG, specifically regarding bi-directional (round-trip) transformations between CDA `substanceAdministration` and FHIR Medication resources. Two use cases are identified:

1. **Round-trip fidelity loss:** Starting from a FHIR `MedicationAdministration` resource, the expected CDA mapping would be `substanceAdministration` with `moodCode=EVN`. However, per the IG, CDA `substanceAdministration` with `moodCode=EVN` maps back to FHIR `MedicationRequest` with `intent=planned` — not to `MedicationAdministration`. The round-trip yields a different FHIR resource than the original source.

2. **No path from CDA "consumed" medications to FHIR:** There is no clear guidance for how a CDA `substanceAdministration` representing a medication that was actually consumed/administered can populate a FHIR `MedicationAdministration` resource.

## Analysis

This ticket represents a **well-known, long-standing issue** in the C-CDA ↔ FHIR medication mapping space. The core problem is a semantic mismatch:

- **CDA** uses `substanceAdministration` with `moodCode` to differentiate between ordered (INT) and occurred/consumed (EVN) medications within a single template.
- **FHIR** separates these concepts into distinct resources: `MedicationRequest` (orders/plans), `MedicationAdministration` (administered events), and `MedicationStatement` (patient-reported usage).
- **US Core** constrains the available profiles to `Medication`, `MedicationRequest`, and `MedicationDispense` — excluding `MedicationAdministration` and `MedicationStatement`.

The current IG mapping of `moodCode=EVN` → `MedicationRequest` with `intent=planned` is a pragmatic compromise driven by US Core's limited profile set, but it loses the semantic distinction between ordered and administered/consumed medications. This breaks round-trip fidelity as the submitter correctly identifies.

The closely related FHIR-41147 (Deferred) proposed the same fundamental correction — mapping `moodCode=EVN` to `MedicationStatement` — but was deferred as "Considered for Future Use." FHIR-40897 made a similar argument for `MedicationAdministration` and was closed with no action.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Revise the medication mapping guidance in the US C-CDA on FHIR IG to:
1. Clarify that the current `moodCode=EVN` → `MedicationRequest` (`intent=planned`) mapping is a **lossy mapping** constrained by US Core's profile set, and explicitly document that round-trip fidelity is not achievable for administered medications under this mapping.
2. Add supplementary guidance for implementers who need to preserve administered/consumed semantics: when base FHIR resources are acceptable (outside US Core constraints), map `moodCode=EVN` to `MedicationAdministration` (for medications administered by clinicians) or `MedicationStatement` (for patient-reported medication usage).
3. Document the round-trip limitation as a known issue with a forward reference to future US Core updates.

**Rationale:** This is a pragmatic middle ground that acknowledges the US Core constraint while providing actionable guidance for implementers who need semantic precision. It aligns with the community discussion (Zulip Option 2 variant) and addresses the submitter's concerns without requiring US Core profile changes.

### Disposition B: Deferred — Considered for Future Use
**Action:** Defer this ticket pending broader resolution of the medication resource mapping strategy, specifically:
- Alignment with future US Core versions that may add `MedicationAdministration` or `MedicationStatement` profiles.
- Coordination with the related deferred tickets (FHIR-41147, FHIR-40831, FHIR-49765).

**Rationale:** The underlying constraint (US Core's limited medication profile set) cannot be resolved within the C-CDA on FHIR IG alone. Multiple related tickets have already been deferred for the same reason. Deferring maintains consistency with prior workgroup decisions and avoids piecemeal solutions.

### Disposition C: Persuasive
**Action:** Accept the submitter's concern and revise the mapping to align `moodCode=EVN` with `MedicationStatement` (for Medications Section context) and `MedicationAdministration` (for Medications Administered Section context), as proposed in FHIR-41147. Update the mapping tables accordingly, even though these resources may not have US Core profiles.

**Rationale:** The current mapping is semantically incorrect — `moodCode=EVN` represents an event that occurred, not a planned request. FHIR's resource model provides the correct targets. Implementers need accurate semantic mappings; US Core profile availability is a separate concern. This aligns with FHIR-41147's proposal and the community discussion Option 4.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

This disposition addresses the submitter's valid concerns about semantic clarity and round-trip fidelity while respecting the practical constraints of US Core. It provides a layered approach: maintaining the current US Core–compatible mapping as the default while adding explicit guidance for semantic-preserving mappings using base FHIR resources. This avoids the pitfall of deferring indefinitely (as happened with FHIR-41147 in 2023) while stopping short of a breaking change to the primary mapping tables. The documentation of round-trip limitations also directly addresses the submitter's Use Case 1.
