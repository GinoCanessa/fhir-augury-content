## Description

The veteran-status extension in the US Military Service IG currently defines a context scope of `Element`, which allows it to appear on any data type or resource element in FHIR. The reporter (Lloyd McKenzie) argues this scope is too broadŌĆöa veteran status extension is semantically relevant only to specific resources (most logically `Patient`) and should not be permitted on arbitrary elements.

## Analysis

### Context

The FHIR extension mechanism allows IG authors to specify a `context` that declares where an extension may legally appear. A context of `Element` is the broadest possible scopeŌĆöit permits the extension on every element of every resource and data type. For the veteran-status extension (`military-service-veteran-status`), this is semantically incorrect: veteran status is a property of a person (a `Patient`, and potentially `Practitioner` or `RelatedPerson`), not something meaningful on an `Observation.code` or a `Quantity` value.

### Precedent

Lloyd McKenzie has filed a series of similar tickets across multiple IGs (FHIR-43110, FHIR-43143, FHIR-43335) all requesting tighter extension contexts. The ones against other IGs (PACIO ADI, CodeX RT) have been resolved as **Persuasive** and published, establishing a clear community consensus that overly broad extension contexts should be constrained.

### Relationship to FHIR-43110

FHIR-43110 is a near-duplicate filed by the same reporter one week earlier. That ticket is more specificŌĆöit names `Patient` (and optionally `Practitioner`/`RelatedPerson`) as the appropriate context. FHIR-43142 makes the same argument more generally. The work group should consider whether to resolve both together or merge them.

## Proposed Dispositions

### Disposition A: Persuasive (Recommended)
**Action:** Constrain the veteran-status extension context from `Element` to `Patient` (and optionally `Practitioner`, `RelatedPerson`).

**Rationale:** The request is well-foundedŌĆöa veteran-status boolean has no semantic meaning on arbitrary elements. Precedent from FHIR-43143 (PACIO ADI) and FHIR-43335 (CodeX RT) confirms the community has consistently accepted this type of correction. Real-world usage (Zulip threads) shows the extension is used exclusively on `Patient`. Coordinate resolution with FHIR-43110 since they address the same concern.

### Disposition B: Persuasive with Modification
**Action:** Constrain the context to `Patient` only (not `Practitioner` or `RelatedPerson`).

**Rationale:** A narrower fix than Disposition A. Veteran status from the VA perspective specifically pertains to patients. Including `Practitioner`/`RelatedPerson` could be deferred until a use case is demonstrated, keeping the context as tight as possible per the principle described in FHIR-43335.

### Disposition C: Not Persuasive
**Action:** Leave the extension context as `Element`.

**Rationale:** Some IGs intentionally use broad contexts to maximize reuse flexibility. However, this argument is weak for veteran status, which is intrinsically person-specific. This disposition would contradict the precedent set by FHIR-43143 and FHIR-43335.

## Recommendation

**Disposition A: Persuasive** ŌĆö Constrain the veteran-status extension context to `Patient` (with optional inclusion of `Practitioner` and `RelatedPerson`). This aligns with established precedent, the reporter's well-reasoned request, and observed real-world usage patterns. The work group should resolve this ticket in coordination with FHIR-43110 to avoid duplicate or conflicting changes.
