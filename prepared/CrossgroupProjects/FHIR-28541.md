# FHIR-28541: Profile XML Format: StructureDefinition xml and JSON

## Summary

**Reporter:** Gay Dolin
**Status:** Triaged | **Type:** Change Request | **Priority:** Medium
**Work Group:** Cross-Group Projects
**Specification:** US Vital Signs with Qualifying Extensions (FHIR)
**Resolution:** Unresolved
**Created:** 2020-09-14 | **Updated:** 2020-09-16
**URL:** https://jira.hl7.org/browse/FHIR-28541

The reporter identified that several profiles in the US Vital Signs with Qualifying Extensions IG have their StructureDefinition `kind` element set to `"complex-type"` instead of `"resource"`. Since these profiles are constrained from Observation (which is a Resource), the `kind` value should be `"resource"`. The reporter checked BMI, Body Weight, and Blood Pressure panel profiles and confirmed the issue, asking whether this was intentional.

In the XML representations, the profiles contain `<kind value="complex-type"/>` when they should contain `<kind value="resource"/>`. In JSON, they show `"kind": "complex-type"` instead of `"kind": "resource"`.

## Analysis

This ticket reports a clear technical error in the US Vital Signs with Qualifying Extensions IG. Profiles on Observation (a FHIR resource) must have `kind` set to `"resource"` in their StructureDefinition. The value `"complex-type"` is only appropriate for profiles on complex datatypes (e.g., Address, HumanName). Using the wrong `kind` value can cause:

1. **Validation failures** — Validators may reject instances or profiles that claim to be complex-types but reference resource elements.
2. **Tooling confusion** — FHIR tooling (e.g., profile renderers, code generators) relies on `kind` to determine how to process a StructureDefinition.
3. **Interoperability issues** — Implementers consuming the downloadable definitions may encounter unexpected behavior.

The precedent set by FHIR-13654 (which corrected a similar `kind` value error in FHIR Core and was resolved as Persuasive) supports treating this as a straightforward bug fix.

The ticket has been in "Triaged" status since September 2020 with no comments or resolution, suggesting it may have been overlooked.

## Proposed Dispositions

### Disposition A: Persuasive (Recommended)
**Action:** Correct the `kind` value from `"complex-type"` to `"resource"` in all affected StructureDefinition profiles in the US Vital Signs with Qualifying Extensions IG.

**Rationale:** This is a clear technical error. Profiles constraining Observation (a resource) must use `kind = "resource"`. The FHIR specification defines that `kind` must match the base type being profiled. FHIR-13654 established precedent for correcting this exact category of error. All affected profiles (at minimum BMI, Body Weight, and Blood Pressure panel, and likely others) should be audited and corrected. This is a non-substantive fix that does not change the intent or constraints of the profiles.

### Disposition B: Persuasive with Modification
**Action:** Correct the `kind` values as in Disposition A, and additionally perform a comprehensive audit of all profiles in the IG to identify any other metadata errors (e.g., incorrect `type`, `baseDefinition`, or `derivation` values).

**Rationale:** If the `kind` error exists in multiple profiles, there may be a systemic issue in the profile authoring or publishing toolchain. A broader audit would catch additional problems and improve overall quality. The modification expands scope beyond the reporter's specific finding to ensure completeness.

### Disposition C: Not Persuasive
**Action:** No change.

**Rationale:** The ticket was filed in 2020 and the IG may have been updated or superseded since then. If the profiles have already been corrected in a newer version, or if the IG is no longer actively maintained, the issue may be moot. However, this disposition is not recommended without first verifying the current state of the profiles.

## Recommendation

**Disposition A: Persuasive** — This is a straightforward technical error with clear precedent (FHIR-13654). The `kind` value in StructureDefinition profiles on Observation must be `"resource"`, not `"complex-type"`. The fix is non-substantive and improves correctness for validators, tooling, and implementers. The work group should confirm the fix applies to all profiles in the IG, not just the three the reporter checked.
