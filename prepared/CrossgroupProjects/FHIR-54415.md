**FHIR Element:** Procedure.code

## Description

The submitter requests that the US Core Procedure Code value set Content Logical Definition (CLD) be updated to include only appropriate LOINC concepts representing procedures, rather than the current inclusion of all LOINC concepts.

## Resolution Description

> Currently, there is no way to create intrinsic definitions of procedures for LOINC and SNOMED that can be easily maintained over the long term. Once these criteria are met, we will update the value set.
>
> Note that we have the following narrative guidance:
> - Procedure codes can be taken from SNOMED-CT, CPT, HCPCS II, ICD-10-PCS, CDT, or LOINC.
>   - *Only LOINC concepts that reflect actual procedures **SHOULD** be used.*

5. **Carmela Couderc** (2026-03-10) — Proposed removing LOINC from the CLD entirely. Since the binding strength is "preferred," health IT developers could still use LOINC codes. Proposed adding guidance: "LOINC may be used to represent Procedure.code however this is not common." Noted LOINC has been in Procedure value sets since 2007 (CCD CONF-434) and that Epic uses LOINC for the procedure aspect of lab draws in C-CDAs.

6. **Gay Dolin** (2026-03-13) — Agreed with Carmela's proposal. Suggested first checking with vendors at CGP/US Core discussions to ensure none depend on LOINC for procedure ordering in any common way. Guessed vendors use CPT for documenting procedures and possibly SNOMED CT.

7. **Cooper Thompson** (2026-03-13) — Noted that large value sets (LOINC, SNOMED, CPT) are generally not used directly by implementers. Health systems import and map those value sets across multiple standards (HL7v2, X12, C-CDA, FHIR, etc.), so specific CLD inclusion rules for large value sets are not very relevant in practice.

## Proposed Dispositions

### Disposition A: Not Persuasive (Affirm Current Resolution)

**Rationale:** The workgroup has already resolved this as "Not Persuasive" and the Argonaut/CGP call voted to reject the request as "unfeasible at this time." There is no practical, maintainable way to create an intrinsic LOINC-only subset for procedures. The two candidate subsets explored comprise ~94,570 of ~104,000 LOINC codes, offering negligible reduction. Existing narrative guidance already states that "Only LOINC concepts that reflect actual procedures SHOULD be used," which provides adequate direction to implementers. Revisiting the value set definition once LOINC provides better procedure classification mechanisms is the appropriate path forward.

**Change Impact:** None. No changes to the specification.

### Disposition B: Persuasive with Modification — Remove LOINC from the CLD Entirely

**Rationale:** Accept the intent of the request but implement it by removing LOINC from the Procedure Code value set CLD entirely, as proposed by the reporter (Carmela Couderc) and supported by Riki Merrick's observation that USCDI does not list LOINC for Procedures. Since the binding strength is "preferred," implementers who need LOINC (e.g., Epic for lab draw procedures) can still use it. Add profile-specific guidance: "LOINC may be used to represent Procedure.code; however, this is not common." This approach avoids the unsolvable subsetting problem while aligning with USCDI.

**Change Impact:** Compatible, substantive (Enhancement). The value set CLD would be narrowed, and guidance text would be updated.

### Disposition C: Persuasive with Modification — Narrow LOINC Inclusion Using Properties

**Rationale:** Rather than removing LOINC entirely, narrow the CLD to include only LOINC codes that plausibly represent procedures, using LOINC properties such as Type=2 (Clinical) and filtering by descendants of LP7787-7 or by the ~392 LOINC codes containing "procedure" (per apitkus's analysis). This preserves LOINC in the formal value set for known use cases while dramatically reducing the scope.

**Change Impact:** Compatible, substantive (Enhancement). The value set CLD would be modified; ongoing maintenance required as LOINC evolves.

## Recommendation

**Disposition A: Not Persuasive (Affirm Current Resolution)**

The workgroup already voted on this at the 2026-03-26 CGP call and rejected it as unfeasible. The fundamental problem — that LOINC lacks an intrinsic, maintainable classification of "procedures" — has not been solved. The two subsets explored (94,570 of 104,000 codes) offer no meaningful reduction. Existing narrative guidance ("Only LOINC concepts that reflect actual procedures SHOULD be used") adequately addresses the concern in practice. Cooper Thompson's comment further confirms that large value set CLDs have minimal practical impact on implementers, who map content independently. If LOINC introduces better procedure classification in the future, a new tracker can revisit this. Affirming the current resolution respects the workgroup's deliberation and avoids creating a maintenance burden with no clear benefit.
