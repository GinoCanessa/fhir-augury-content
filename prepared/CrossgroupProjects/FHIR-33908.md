## Description

The submitter (Genevieve Luensman) requests that slicing be added to the Occupation element in the US Military Service IG to include the Occupational Data for Health (ODH) vocabulary. Specifically, the request is to add a slice for the **Occupation ONETSOC Detail (ODH)** value set (OID: `2.16.840.1.114222.4.11.7901`, available at [PHIN VADS](https://phinvads.cdc.gov/vads/ViewValueSet.action?oid=2.16.840.1.114222.4.11.7901)).

The rationale is that ODH uses a different vocabulary for occupation that is "collectable and more complete" than what is currently specified.

## Analysis

### Context
This ticket is part of a batch of changes filed by Genevieve Luensman in September 2021 against the US Military Service IG (Block-Vote-2). The request aligns with a broader effort to incorporate ODH vocabularies into FHIR IGs — similar requests have appeared in QICore (FHIR-37755, published), VRDR (FHIR-42745, not persuasive), and US Core (FHIR-40181, published).

### Key Considerations
1. **Resolution is Persuasive** — The ticket was already resolved as Persuasive on 2022-04-21, and the reporter acknowledged the update ("ok will update"). The status remains "Triaged" rather than "Published," suggesting the change may not yet be applied.
2. **Companion ticket divergence** — The near-identical companion ticket FHIR-33909 (Industry slicing with NAICS Detail) was resolved as **Not Persuasive**, creating an asymmetry where occupation gets the ODH slice but industry does not.
3. **Value set availability** — The ONETSOC Detail value set (OID `2.16.840.1.114222.4.11.7901`) is hosted on PHIN VADS and is used in the ODH IG. Its inclusion via slicing would allow systems to send more granular occupation codes alongside existing CDC Census-based codes.
4. **Implementation pattern** — The change involves adding a slice on the Occupation CodeableConcept to bind an additional value set. This is a non-breaking, additive change.

## Proposed Dispositions

### Disposition A: Persuasive (Affirm Resolution)
**Action:** Affirm the existing Persuasive resolution. Add a slice to the Occupation element in the US Military Service IG to include binding to the Occupation ONETSOC Detail (ODH) value set (`2.16.840.1.114222.4.11.7901`).

**Rationale:** The ODH ONETSOC vocabulary provides more granular and complete occupation coding than CDC Census codes alone. The additive slicing approach is non-breaking and follows precedent set by the ODH IG itself. The reporter (who is the ODH subject matter expert) has confirmed she will apply the change. Similar ODH integration has been accepted in QICore (FHIR-37755) and US Core (FHIR-40181).

### Disposition B: Not Persuasive (Reverse Resolution)
**Action:** Reverse the Persuasive resolution. Do not add ODH ONETSOC slicing to the US Military Service IG.

**Rationale:** The companion Industry ticket (FHIR-33909) was resolved as Not Persuasive, suggesting the workgroup may have concerns about adding ODH vocabulary slicing to this IG. The US Military Service IG may not be the appropriate venue for ODH vocabulary expansion, as the ODH IG itself already defines these bindings. Adding slices here could create maintenance burden and duplicative bindings across IGs.

### Disposition C: Persuasive with Modification
**Action:** Accept the request but constrain the binding strength and/or add guidance that the ONETSOC slice is optional and intended for systems that also conform to the ODH IG.

**Rationale:** A middle path that acknowledges the value of the ONETSOC vocabulary while keeping the US Military Service IG focused on its core use case. Adding an explicit "preferred" or "extensible" binding with documentation that this slice aligns with ODH would reduce confusion and maintain interoperability without mandating ODH adoption.

## Recommendation

**Disposition A: Persuasive (Affirm Resolution)**

The ticket already has a Persuasive resolution, and the change is straightforward and non-breaking. The ONETSOC vocabulary is widely recognized as the more granular occupation coding system, and its inclusion via slicing is consistent with ODH integration patterns accepted in other IGs (QICore, US Core). The reporter is an established ODH expert and has acknowledged responsibility for applying the change. The primary action needed is to ensure the change is actually published — the ticket appears stalled at "Triaged" despite being resolved over two years ago.
