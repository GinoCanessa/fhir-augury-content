## Summary

Hans Buitendijk requests clarification of the AI Data profile in the AI Transparency on FHIR IG. The profile is defined on DomainResource and its documentation states "it can't be used in an IG," yet it appears within the IG. The description also characterizes it as "a resource that captures the input and output data used by an AI model" when it is actually metadata on existing resources, not a standalone resource. The reporter raises three specific concerns: (1) the profile's intended usage pattern needs better explanation, (2) the ProvenanceVS ValueSet contains only output-related codes despite the description implying input tagging, and (3) tagging input resources could lead to significant resource bloating over time if data is used by many AI processes. John Moehrke commented that the tag method is only used for AI output and that using meta.security to indicate data was consumed by AI would be inappropriate.

## Details

### Description

The reporter identifies a disconnect between the AI Data profile's documentation and its actual usage:

1. **Profile usage contradiction:** The profile on DomainResource states "it can't be used in an IG," yet it is used within the AI Transparency IG itself. More explanation is needed about how this profile is intended to be used.

2. **Misleading resource description:** The profile is described as "A resource that captures the input and output data used by an AI model in generating or enhancing FHIR resources," but it is not a resource—it is metadata applied to existing resources. The reporter asks whether the intent is that every resource serving as input or output of AI should include these tags in addition to a Provenance resource.

3. **ProvenanceVS gap:** The [Recommended provenance codes ValueSet](https://hl7.org/fhir/uv/aitransparency/2026Jan/ValueSet-ProvenanceVS.html) contains only output-related values. If the profile is supposed to also tag input data, why are there no input-related codes?

4. **Resource bloating concern:** If input data is also tagged, resources used as input to many AI processes would accumulate substantial tags over time, bloating the resource. The reporter questions whether input tracking should be on the resource itself or handled elsewhere (e.g., via an audit mechanism).

### Comments

- **John Moehrke (2026-01-26):** Clarified that the tag method is only used to tag output from AI. There is no way to use meta.security to indicate that data was used by AI; that would be an inappropriate use of meta.security.

## Proposed Dispositions

### Disposition A: Accept — Clarify AI Data Profile Description and Usage

**Resolution:** Persuasive with Modification

Accept the request and make the following changes to the AI Data profile:

1. **Revise the profile description** to accurately state that it defines metadata constraints applied to existing DomainResource instances (not a standalone resource), and that it captures output data produced or enhanced by AI.
2. **Clarify the "can't be used in an IG" statement** by explaining that DomainResource profiles serve as abstract base constraints for validation tooling but cannot be directly instantiated or referenced as a concrete resource type in IG conformance statements.
3. **Explicitly state that tagging applies only to AI output resources**, consistent with John Moehrke's comment that the tag method is not used for input data.
4. **Add a note** directing implementers to the AI Provenance profile for tracking input data relationships, rather than tagging input resources.

This disposition aligns with John Moehrke's comment and addresses all four of the reporter's concerns directly.

### Disposition B: Accept Partially — Clarify Description Only, Defer Input/Output Scope

**Resolution:** Persuasive with Modification

Accept the documentation clarity concerns (items 1 and 2 from the description) but defer the input vs. output tagging scope questions to be resolved alongside related tickets FHIR-53973 and FHIR-54725:

1. **Fix the profile description** to correctly characterize it as metadata constraints rather than a standalone resource.
2. **Clarify the DomainResource profile usage** with an explanatory note.
3. **Defer** the input tagging, ProvenanceVS gap, and resource bloating concerns to a coordinated resolution with FHIR-53973 (Clarification on input data) and FHIR-54725 (AI Data profile missing examples), since these tickets share overlapping scope.

This approach avoids piecemeal changes to input/output semantics that may conflict with decisions made on related tickets.

### Disposition C: Decline — Already Addressed by Comment Clarification

**Resolution:** Not Persuasive with Modification

Decline the substantive change request. John Moehrke's comment already clarifies that tags are only for AI output, which resolves the input tagging and bloating concerns. Add a brief clarifying note to the profile description stating that the AI Data profile applies to AI-generated or AI-enhanced output resources only, and that input data tracking is handled through the AI Provenance profile's entity references rather than resource tagging.

This is the minimal-change option that relies on the existing design being correct and only needing a documentation touch-up.

## Recommendation

**Disposition A: Accept — Clarify AI Data Profile Description and Usage**

Disposition A is recommended because it comprehensively addresses all four concerns raised by the reporter while remaining consistent with the existing IG design intent as clarified by John Moehrke. The AI Data profile description is demonstrably misleading — calling metadata constraints a "resource" and implying input tagging that the IG does not actually support — and the "can't be used in an IG" statement is confusing when the profile appears within the IG. These documentation deficiencies are significant enough to cause implementer confusion and should be fixed directly rather than deferred (as in Disposition B) or minimally patched (as in Disposition C). Disposition A also proactively addresses the input/output scope question, reducing the surface area for related tickets FHIR-53973 and FHIR-54725.
