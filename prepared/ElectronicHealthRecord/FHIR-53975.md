## Description

The submitter raises a concern about the `extension:modelCardDescription` element on the [AI Device profile](https://hl7.org/fhir/uv/aitransparency/2026Jan/StructureDefinition-AI-Device.html). Currently, the AI Device references the Model-Card documents it supports via this extension. The submitter argues this creates a maintenance burden: every time a new Model-Card is created, the Device resource must be updated to add the new reference. The submitter proposes inverting the relationship direction so that the Model-Card references the Device(s) it applies to, rather than the Device enumerating all its Model-Cards. This would allow new Model-Cards to be added without modifying existing Device resources.

## Analysis

The core question is whether the Device→Model-Card reference direction is correct, or whether Model-Card→Device would be better. The submitter's argument is valid from a maintenance perspective: if the Device must enumerate all its Model-Cards, then adding a new Model-Card requires updating the Device. An inverse reference (Model-Card references the Device) would follow a pattern common in FHIR where dependent/derived resources point to their parent rather than the parent listing all dependents.

However, several considerations apply:
1. **The closely related FHIR-53974 was resolved as Not Persuasive** — that ticket from the same reporter also questioned the Device↔Model-Card relationship, suggesting the work group has already considered this linkage direction.
2. **FHIR-54527 proposes strengthening the current direction** by making `extension:modelCardDescription` the only mechanism for the Device-to-Model-Card link, implying the community intends to keep the current reference direction.
3. **Device as the anchor resource** — in AI transparency use cases, the Device represents the AI system, and knowing which Model-Cards describe a device is a primary query need. Having the Device reference its Model-Cards supports direct read access without reverse-include searches.
4. **Both directions could coexist** — the Model-Card (DocumentReference) already has `subject` and `context` elements that could reference the Device, enabling bidirectional navigation if needed.

## Proposed Dispositions

### Disposition A: Not Persuasive
**Rationale:** The current design where Device references its Model-Cards via `extension:modelCardDescription` is intentional and aligns with the primary use case of retrieving AI transparency information from a Device. The closely related FHIR-53974 was already resolved as Not Persuasive, indicating the work group has considered the relationship direction. Requiring consumers to search Model-Cards by Device reference (reverse lookup) would be less efficient for the most common query pattern. The existing direction also aligns with FHIR-54527's proposal to make this extension the canonical linkage mechanism.

### Disposition B: Persuasive with Modification
**Rationale:** Acknowledge the valid maintenance concern but rather than inverting the relationship, add guidance clarifying that the `extension:modelCardDescription` extension lists the known Model-Cards at Device registration time and does not need to be exhaustive. Additionally, recommend that the Model-Card profile include a reference back to the Device (e.g., via `DocumentReference.subject` or a dedicated extension), enabling bidirectional navigation. This addresses the submitter's concern about update burden while preserving the current Device-centric query pattern.

### Disposition C: Persuasive
**Rationale:** Accept the submitter's proposal to invert the relationship. Remove `extension:modelCardDescription` from the AI Device profile and instead add a required reference from the Model-Card (DocumentReference) to the applicable Device(s). This follows the FHIR pattern of dependent resources referencing their parent and eliminates the need to update Device resources when new Model-Cards are created.

## Recommendation

**Disposition A: Not Persuasive**

The companion ticket FHIR-53974 from the same reporter — which directly questioned the Device↔Model-Card relationship — was already resolved as Not Persuasive, establishing that the work group supports the current reference direction. Additionally, FHIR-54527 (Highest priority) proposes strengthening the current approach by making `extension:modelCardDescription` the sole mechanism for this linkage. Inverting the relationship would conflict with that direction and would make the most common query pattern (retrieving Model-Cards for a Device) less efficient. The maintenance concern, while valid, is typical of FHIR resource management and can be addressed through implementation guidance rather than structural redesign.
