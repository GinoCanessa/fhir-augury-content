## Description

The submitter notes that the AI Transparency IG's observability factors table, under the **Process** category, mentions "multiple actors, including the human role." This implies that human-AI interaction (e.g., a clinician accepting, rejecting, or modifying an AI inference) is a recognized observability factor. However, the IG provides **no guidance on how to model or encode these interactions** using FHIR resources.

**The submitter requests:**

1. Describe how to encode the "human in the loop" element of AI usage in the IG model — i.e., how to represent a human actor's interaction with an AI inference (accept, reject, modify).
2. Describe a generic workflow for human-AI collaboration.
3. Provide example FHIR instances demonstrating this encoding.

## Analysis

This ticket identifies a genuine **modeling gap** in the AI Transparency IG. The IG's observability factors table acknowledges the importance of human-AI interaction under the "Process" category, but the current specification does not provide:

- A FHIR resource mapping for human review actions (accept/reject/modify)
- Guidance on using Provenance, Task, or other resources to capture the human actor's role and decision
- A workflow pattern showing how human-in-the-loop fits into the overall AI transparency data flow
- Example instances demonstrating the encoding

This is a substantive enhancement request that would require new normative content (resource usage guidance, workflow diagrams, and example instances). It is closely related to several other tickets from the same reporter (FHIR-54210, FHIR-54215, FHIR-54118) that collectively highlight gaps between the IG's conceptual framework (observability factors) and its concrete FHIR modeling guidance.

The "Process" observability factor is one of four top-level categories in the IG, and human-in-the-loop is arguably its central concern. Addressing this gap would significantly strengthen the IG's practical applicability.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Resolution:** Accept the request in principle but scope it to the current ballot cycle's capacity. Add a new subsection under the Process observability factor that:
1. Defines the concept of "human in the loop" in the context of AI transparency (accept, reject, modify actions).
2. Provides guidance on using **Provenance** (with agent roles and activity codes) to capture human review decisions on AI-generated content.
3. Includes at least one normative example instance showing a human clinician accepting an AI inference with Provenance recording.
4. Defers the full generic workflow diagram and additional examples to a future ballot cycle.

**Rationale:** This addresses the core gap with concrete, implementable guidance while keeping scope manageable. Provenance is already referenced in the IG for AI-related tagging, making it a natural fit.

### Disposition B: Persuasive
**Resolution:** Fully accept the request. Add comprehensive human-in-the-loop modeling guidance including:
1. A new section describing how to encode human review interactions using Provenance (agent, activity, role) and potentially Task (for workflow state).
2. A generic workflow diagram showing AI inference → human review → recorded decision.
3. Multiple example instances covering accept, reject, and modify scenarios.
4. Binding to appropriate terminology for human review action codes.

**Rationale:** The Process observability factor is central to the IG's value proposition, and human-in-the-loop is its defining characteristic. Full treatment is warranted given the Highest priority and the IG's stated goal of enabling downstream use cases.

### Disposition C: Not Persuasive
**Resolution:** Acknowledge the observation but decline to add detailed modeling guidance in this cycle. The IG intentionally presents the Process observability factor at a conceptual level, and concrete FHIR modeling patterns for human-AI interaction should emerge from implementation experience before being standardized. Add an informative note acknowledging this as a future work item.

**Rationale:** Premature standardization of human-in-the-loop patterns could constrain implementers. The IG's current approach of identifying observability factors without prescribing specific encodings allows flexibility.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended. The gap between the conceptual framework and FHIR modeling guidance is real and well-articulated. The human-in-the-loop concept is fundamental to the Process observability factor and is already partially addressed by the IG's use of Provenance for AI tagging. Adding focused guidance on using Provenance to capture human review actions (with one normative example) is achievable within a ballot cycle and directly addresses the submitter's concern without requiring the full scope of workflow diagrams and multiple examples that Disposition B would entail. This also aligns with the parallel work needed for related tickets (FHIR-54210, FHIR-54215) that address other observability factor modeling gaps.
