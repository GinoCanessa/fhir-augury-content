## Description

The submitter observes that the [AI Device profile](https://hl7.org/fhir/uv/aitransparency/2026Jan/StructureDefinition-AI-Device.html) references "AI Model-Card DocumentReference," but that reference is not visible in the [AI Model-Card profile](https://hl7.org/fhir/uv/aitransparency/2026Jan/StructureDefinition-AI-ModelCard.html) itself. The submitter expected the Model-Card profile to explicitly show a link back to the Device, and finds the current relationship unclear.

## Resolution Description

> The Device references the Model Card. The Model Card does not reference the Device. One can navigate FHIR References in the reverse direction. The Linkage from Device to the DocumentReference is by an Extension in Device, the linkage the publisher shows is to the extension, not to Device.

## Analysis

### Core Issue
The submitter is confused by the apparent lack of a visible relationship between the AI Device profile and the AI Model-Card profile. The AI Device has an extension (`modelCardDescription`) that references a DocumentReference (the Model-Card), but when looking at the Model-Card profile, no back-reference to Device is shown.

### Root Cause
The linkage from Device to Model-Card is implemented via an **extension** on Device. The IG publisher displays the extension relationship on the extension definition, not on the Device or Model-Card profile pages. This is a known limitation of how FHIR IG publisher renders extension-based references — the relationship exists but is not immediately visible in the profile's reference graph.

### Context from Related Tickets
The broader ballot feedback shows significant community confusion around the Device/Model-Card relationship:
- **FHIR-54582** (Highest priority) argues the model card should be referenced **by** Device rather than through Provenance, since it's typically 1:1 with an algorithm+version.
- **FHIR-53975** (same reporter) proposes inverting the reference direction entirely.
- **FHIR-54421** (Persuasive) corrects the IG's misleading statement that "Model-Card represents the request made of the AI" — it should describe the AI model/system.
- **FHIR-54735** (Persuasive with Modification) calls for a clearer definition of what a Model Card is.

John Moehrke's comment on FHIR-54582 clarifies that the IG uses "model-card" for three distinct purposes: (1) defining the Device, (2) defining regularly used prompts, and (3) customized prompts — only use (1) relates to Device. This triple-use pattern is a significant source of the confusion reported here.

## Proposed Dispositions

### Disposition A: Not Persuasive (uphold existing resolution)
The current resolution correctly explains that the Device→Model-Card relationship exists via an extension. The reverse navigation is possible using standard FHIR reference search. The IG publisher's rendering limitation does not indicate a modeling error. The submitter's concern stems from a display issue, not a structural one.

**Rationale:** The architecture is sound. FHIR extensions are by design not shown as direct references in profiles. Adding a back-reference from Model-Card to Device would create a circular dependency and violate the design principle that Model-Card is a general-purpose DocumentReference usable independent of Device.

### Disposition B: Persuasive with Modification — Add clarifying text
Accept the concern about clarity but address it through documentation rather than structural changes. Add explanatory text to the AI Device profile and/or the Model-Card usage section explaining: (1) the Device references the Model-Card via the `modelCardDescription` extension, (2) this link is not shown in the Model-Card profile because it is an extension-based reference, and (3) reverse navigation is possible via FHIR search (`Device?_has:extension:url=modelCardDescription&_has:extension:value-reference=DocumentReference/[id]`).

**Rationale:** The confusion is legitimate and shared by multiple ballot reviewers (FHIR-53975, FHIR-54582). Documentation improvements would reduce future confusion without changing the information model.

### Disposition C: Persuasive — Add explicit cross-reference
Accept the concern and add an explicit element or invariant in the Model-Card profile that clarifies its relationship with Device. This could be a "Usage Note" section in the profile, a SearchParameter enabling `ModelCard:device` reverse lookups, or a formal must-support reverse-include directive.

**Rationale:** Multiple high-priority tickets (FHIR-54582, FHIR-53975) confirm that the Device/Model-Card relationship modeling needs improvement. A structural enhancement would provide a permanent solution rather than relying on documentation.

## Recommendation

**Disposition A: Not Persuasive** (uphold existing resolution)

The existing resolution is technically correct: the relationship exists via extension, and FHIR's reference model supports reverse navigation. The visibility issue is a known IG publisher rendering limitation, not a modeling defect. Related tickets FHIR-54582 and FHIR-53975 (both still unresolved) more directly address whether the reference direction should be changed, and those are the appropriate venues for structural changes. Accepting this ticket could create conflicting guidance with those open tickets. The workgroup should note, however, that the broader pattern of confusion suggests the IG would benefit from improved documentation of the Device/Model-Card relationship (which can be tracked under FHIR-54582 or FHIR-54735).
