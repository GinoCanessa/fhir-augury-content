## Description

The submitter questions why the [AI Data profile](https://hl7.org/fhir/uv/aitransparency/2026Jan/StructureDefinition-AI-data.html) extends `DomainResource` instead of `Resource`. Extending `Resource` would allow the profile to be applied to Bundles (which extend `Resource`, not `DomainResource`), enabling AI-generated Bundles to be tagged with AI transparency metadata.

## Resolution Description

> Yes, this should be on Resource. although unclear how useful this profile is as it can not be applied.

The resolution agrees the profile's base type should be changed from `DomainResource` to `Resource`, while noting uncertainty about the practical applicability of the profile.

## Analysis

The core ask is straightforward: change the base type of the AI Data profile from `DomainResource` to `Resource`. This would broaden the profile's applicability to all resource types, including `Bundle`, `Parameters`, and `Binary` — resources that inherit from `Resource` but not from `DomainResource`.

The resolution already agrees with this change ("Yes, this should be on Resource"), but adds a caveat about the profile's general usefulness since it "can not be applied" in the traditional IG sense. This caveat is echoed in FHIR-54260 and FHIR-54725, which both question the AI Data profile's practical usage model. The related ticket FHIR-53967 independently identifies the same gap — a Bundle in an example cannot be tagged because the profile doesn't extend Resource.

Changing the base type is technically simple but has implications: profiles on `Resource` have access to fewer elements than profiles on `DomainResource` (notably no `text`, `contained`, `extension`, or `modifierExtension`). However, the AI Data profile primarily uses `meta.security` and `meta.tag` for tagging, which are available on `Resource`, so this change should be compatible.

## Proposed Dispositions

### Disposition A: Persuasive (Accept as proposed)
Change the AI Data profile's base type from `DomainResource` to `Resource`, allowing AI transparency tags to be applied to all resource types including Bundle, Parameters, and Binary.

- **Pros:** Directly addresses the gap; technically sound since the profile uses `meta` elements available on `Resource`; aligns with the IG's own examples that show Bundles needing AI tagging; resolution already agrees.
- **Cons:** Broader scope may introduce unforeseen validation or tooling issues; the resolution notes the profile's limited practical applicability regardless of base type.

### Disposition B: Persuasive with Modification (Accept with scope refinement)
Change the base type to `Resource` as requested, but also add guidance text clarifying which resource types are expected to benefit from this profile (e.g., Bundle, DomainResource subtypes) and how the profile should be used for validation purposes given the IG tooling limitations for Resource-level profiles.

- **Pros:** Addresses the gap while mitigating the confusion raised in FHIR-54260 and FHIR-54725 about how to use the profile; provides implementer guidance.
- **Cons:** Requires additional editorial work beyond the simple base type change; may overlap with the scope of FHIR-54260 and FHIR-54725.

### Disposition C: Not Persuasive (Reject)
Keep the AI Data profile on `DomainResource` and address Bundle-level tagging through a separate mechanism (e.g., Provenance resources that reference the Bundle, or a distinct Bundle-level profile).

- **Pros:** Avoids the tooling and validation complications of Resource-level profiles; DomainResource provides richer profiling capabilities.
- **Cons:** Does not solve the stated problem; leaves a gap for Bundle tagging; contradicts the existing resolution which already agrees the change should be made.

## Recommendation

**Disposition A: Persuasive** — The resolution already agrees this should be on `Resource`, and the technical analysis supports it since the AI Data profile's tagging elements (`meta.security`, `meta.tag`) are available on `Resource`. The concerns about the profile's applicability model (raised in FHIR-54260 and FHIR-54725) are real but should be addressed in those separate tickets rather than blocking this straightforward base type change.
