## Description Summary

The reporter notes that the [AI Provenance profile](https://hl7.org/fhir/uv/aitransparency/2026Jan/StructureDefinition-AI-Provenance.html) clearly specifies `entity:AIInputData` for capturing input data to an AI model. However, the profile does not explicitly document that `Provenance.entity` also serves to link **source data** (i.e., existing FHIR resources used to generate or influence AI-produced target resources) back to those target resources.

The request is for the profile narrative to **explicitly state** that it establishes the provenance link between:
- **Target resources** — the AI-generated or AI-influenced FHIR resources, and
- **Source/input resources** — the data used to create or influence those targets.

Even though this relationship may be implicitly understood, the reporter argues it should be made explicit in the profile documentation.

## Proposed Dispositions

### Disposition A: Persuasive — Enhance AI Provenance Profile Narrative

**Action:** Add explicit narrative text to the AI Provenance profile (`StructureDefinition-AI-Provenance`) clarifying that `Provenance.entity` with role `source` is used to reference the input/source FHIR resources that were consumed by or influenced the AI model, linking them to the `Provenance.target` AI-generated resources.

**Rationale:** The request is reasonable and well-supported by John Moehrke's comment. The `Provenance.entity` element's role in connecting source data to AI-generated outputs is a core semantic of the profile. Making this explicit improves implementer understanding without changing the profile's structure or constraints. This aligns with the IG's goal of transparency and would reduce ambiguity for ballot reviewers and implementers.

**Scope:** Documentation/narrative change only. No structural or constraint changes to the profile.

### Disposition B: Persuasive with Modification — Clarify and Add Examples

**Action:** In addition to adding narrative clarification (as in Disposition A), add a worked example showing how `Provenance.entity` entries with different roles (`source` for input data vs. the existing `AIInputData` slice) are used together. Coordinate with FHIR-54260 (AI Data profile clarification) and FHIR-53967 (general clarifications) to ensure a consistent explanation of input data semantics across the IG.

**Rationale:** The concern is valid but benefits from concrete illustration. The distinction between `entity:AIInputData` (specific input data slice) and `Provenance.entity` with `source` role (general provenance semantics) is subtle. An example would serve implementers better than narrative alone. Coordinating with related tickets avoids contradictory or duplicative clarifications.

**Scope:** Narrative and example additions. No structural changes. Cross-ticket coordination recommended.

### Disposition C: Not Persuasive — Current Profile Is Sufficient

**Action:** No change. Add a comment explaining that the standard FHIR `Provenance.entity` semantics (including entity roles like `source`, `derivation`, and `revision`) already cover the requested link between source and target resources, and that the AI Provenance profile inherits these semantics from the base Provenance resource.

**Rationale:** The base FHIR Provenance resource already defines `entity.role` with values that cover source data linking. The AI Provenance profile's `entity:AIInputData` slice explicitly addresses input data. Restating base resource semantics in every profile would be redundant and could create maintenance burden. Implementers are expected to understand the base Provenance resource when implementing profiles on it.

**Scope:** No change to the IG.

## Recommendation

**Disposition A: Persuasive — Enhance AI Provenance Profile Narrative**

This is a straightforward documentation improvement request with clear consensus (John Moehrke's comment supports the interpretation). The AI Transparency IG is in its first ballot cycle and explicitly aims to make AI usage in FHIR transparent — leaving implicit the very provenance links the profile is designed to capture undermines that goal. A focused narrative addition to the profile page is low-risk, addresses the reporter's concern directly, and improves clarity for all implementers. The more expansive Disposition B could be pursued if the workgroup wants to coordinate with FHIR-54260 and FHIR-53967, but the core ask here is self-contained and can be resolved independently.
