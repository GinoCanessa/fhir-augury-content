## Description

The ticket identifies a structural gap in the AI Transparency IG: the "Context" observability factor category lists five subcategories — **Request input (to AI)**, **Reference input**, **Operations**, **Data quality**, and **Data qualification** — but only section 2.7 ("Data sources") provides a FHIR model and encoding examples (specifically for "input prompt"). The remaining subcategories lack corresponding model representation.

Additionally, the reporter raises two related concerns:

1. **Ambiguity of "input prompt":** An input prompt could be classified as either a Request input or a Reference input, depending on how the system implements user and system prompts. The distinction is unclear.
2. **Naming inconsistency:** The spec uses "Context" in the observability factors table but "Data sources" in section 2.1.3, and alternates between "input prompt" and "request input" without clear differentiation.

The reporter suggests either refactoring the Context subcategories to align with the current model, or expanding the model to cover all subcategories, and harmonizing terminology throughout the specification.

## Analysis

This ticket is the **umbrella issue** for a cluster of five related tickets (FHIR-54239, FHIR-54233, FHIR-54235, FHIR-54211, and to some extent FHIR-53967), all filed during the January 2026 ballot cycle against the AI Transparency IG. While the companion tickets each drill into a specific Context subcategory, FHIR-54210 identifies the overarching structural problem: the observability factors table promises five Context subcategories, but the FHIR model only supports one of them (input prompt / data sources).

Key considerations for the workgroup:

1. **The gap is real and well-documented.** Multiple independent reviewers (IgnacioJauregui, Abderrazek Boufahja) identified that the observability factors table sets expectations the model does not fulfill.
2. **The companion tickets provide specific feedback.** FHIR-54239 (request vs reference input), FHIR-54233 (operations), and FHIR-54235 (data quality/qualification) each propose concrete changes for their respective subcategories.
3. **Naming inconsistency is a separate, straightforward fix.** "Context" vs "Data sources" and "input prompt" vs "request input" can be harmonized regardless of the model alignment decision.
4. **Scope risk:** Fully modeling all five subcategories could significantly expand the IG scope for this cycle. FHIR-54233 explicitly suggests *removing* the Operations section rather than modeling it, given the fast-moving nature of the technology.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Full Refactor

Accept the suggestion to refactor the Context subcategories to align with the FHIR model. For each subcategory (Request input, Reference input, Operations, Data quality, Data qualification), either provide explicit FHIR encoding patterns and examples, or remove the subcategory from the observability factors table if it is not yet modeled. Harmonize naming throughout ("Data Sources" consistently, with clear definitions for "input prompt" vs "request input"). Coordinate resolution with FHIR-54239, FHIR-54233, and FHIR-54235.

**Pros:** Fully addresses the structural gap; ensures the table and model are consistent.
**Cons:** Large scope; may delay publication. Risk of premature modeling for areas like Operations (MCP, A2A) that are still evolving.

### Disposition B: Persuasive with Modification — Harmonize Naming, Defer Model Expansion

Address the naming inconsistency immediately (standardize on "Data Sources" or "Context" throughout the spec; define "input prompt" vs "request input" clearly). Add clarifying text to the observability factors table indicating which subcategories have full model support and which are identified for future development. Defer detailed model expansion for individual subcategories to the companion tickets (FHIR-54239, FHIR-54233, FHIR-54235).

**Pros:** Fixes the most actionable issues (naming, reader expectations) without blocking progress. Leverages the existing companion tickets for subcategory-specific work.
**Cons:** Does not close the full model-vs-table gap in this ticket alone.

### Disposition C: Not Persuasive with Modification — Clarify Intent

The observability factors table is intentionally a broader conceptual framework than the current FHIR model supports, representing the full scope of AI transparency concerns to be addressed iteratively over multiple ballot cycles. Add a note explaining that not all subcategories have model support in this version and that the table represents the IG's development roadmap. Fix the naming inconsistency between "Context" and "Data Sources."

**Pros:** Preserves the aspirational scope of the observability framework; minimal change.
**Cons:** May not satisfy reviewers who expect the table and model to be aligned. Does not address the ambiguity of input prompt classification.

## Recommendation

**Disposition B: Persuasive with Modification — Harmonize Naming, Defer Model Expansion**

This is the most pragmatic path forward. The naming inconsistency ("Context" vs "Data Sources," "input prompt" vs "request input") is a clear editorial defect that should be fixed regardless of how the broader structural question is resolved. Adding a note to the observability factors table about which subcategories have full model support sets appropriate reader expectations without over-promising or blocking progress. The companion tickets (FHIR-54239, FHIR-54233, FHIR-54235) already provide the specific feedback needed for subcategory-level decisions, so it is efficient to let those tickets drive the detailed model work. Given that FHIR-53967 independently flagged the same naming issue, there is corroborating evidence that the harmonization is needed. The workgroup should consider addressing all six "Data Sources & Context" tickets as a coordinated cluster.
