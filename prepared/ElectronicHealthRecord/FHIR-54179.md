## Description

The submitter notes that the AI Transparency IG states: *"Some computer-based logic that touches health data, such as simple calculations and data transformations, may not be considered to be AI algorithms **but observability of such events should also be supported by this implementation guide.**"*

The concern is that while the IG explicitly claims rule-based system observability is in scope, the actual generated artifacts (profiles, examples) appear to be directed exclusively toward machine learning algorithms. No examples or guidance are provided for representing rule-based systems' output using the IG's model.

The submitter requests that the IG either:
1. Include examples and rationale for how to represent rule-based system outputs, **or**
2. Remove rule-based systems from the IG's stated scope.

## Analysis

This ticket is part of a significant cluster of ballot feedback (at least 4–5 tickets) questioning the AI Transparency IG's scope definition. The core tension is:

- The IG defines "AI algorithm" very broadly (any computer-based logic that touches health data), and explicitly states that rule-based system observability should be supported.
- However, the profiles, examples, and use cases in the IG are clearly oriented toward machine learning / LLM scenarios.
- This creates a gap: implementers of rule-based clinical decision support systems have no guidance on how to use the IG's artifacts for their systems.

The submitter (IgnacioJauregui) also filed the closely related FHIR-54112, which frames the issue from the definition side. Together, these tickets present a coherent request: make the IG's scope and artifacts consistent with each other.

This is a substantive scope question that the workgroup needs to decide: should the IG genuinely cover non-ML clinical logic (requiring new examples and potentially new profiles), or should it narrow its stated scope to ML/AI systems only?

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Add rule-based system examples
**Resolution:** Accept the request in part. Add at least one example demonstrating how a rule-based system output (e.g., a clinical decision support alert based on deterministic rules) can be represented using the existing AI Transparency profiles. Add a brief section to the IG explaining how rule-based system observability maps to the model, acknowledging that some profile elements (e.g., training data, model performance metrics) may not apply. Coordinate with the CDS WG as suggested in FHIR-54112.

**Rationale:** The IG explicitly claims rule-based systems are in scope, so the artifacts should demonstrate this. Adding examples is less disruptive than restructuring the entire IG. This approach validates the submitter's concern while preserving the broad scope.

### Disposition B: Persuasive with Modification — Narrow the scope to exclude rule-based systems
**Resolution:** Accept the request by choosing the second option. Remove the statement that rule-based system observability should be supported by this IG. Narrow the scope to AI/ML algorithms only, and add a note that rule-based clinical logic observability may be addressed by other IGs (e.g., CDS Hooks, CPG). Adjust the scope definition to explicitly exclude deterministic, non-ML computer logic.

**Rationale:** The IG's profiles and examples are inherently ML-focused. Attempting to cover rule-based systems adequately would require significant additional work and risks diluting the IG's focus. Narrowing scope provides clarity for implementers and avoids the inconsistency the submitter identified.

### Disposition C: Not Persuasive
**Resolution:** Acknowledge that the current artifacts emphasize ML scenarios but maintain that the broad scope is intentional and appropriate for a first ballot. Rule-based system examples may be added in a future cycle as the IG matures. No changes to the current ballot.

**Rationale:** The IG is in its first ballot cycle, and the broad scope signals the community's intent to eventually address all forms of clinical computing observability. Providing ML examples first is a reasonable starting point.

## Recommendation

**Disposition A: Persuasive with Modification — Add rule-based system examples.** This is the strongest option because it directly resolves the inconsistency the submitter identified (scope says rule-based is included, but artifacts don't support it) without requiring the contentious decision to narrow scope. Given the cluster of related tickets (FHIR-54893, FHIR-54112, FHIR-54274) all raising scope concerns, demonstrating that the broad scope is actionable — by providing at least one concrete rule-based example — would address multiple ballot comments simultaneously. It also opens a path for CDS WG coordination as recommended in FHIR-54112.
