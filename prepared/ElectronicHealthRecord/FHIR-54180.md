## Description

The AI Transparency IG currently targets FHIR R4 (with an R4B package also published). The IG's landing page states: *"This is an R4 IG. None of the features it uses are changed in R4B, so it can be used as is with R4B systems."* Packages are provided for both R4 (`hl7.fhir.uv.aitransparency.r4`) and R4B (`hl7.fhir.uv.aitransparency.r4b`).

The submitter requests that the IG include a statement of **intention and ideally timelines** for developing FHIR R5 and R6 versions of the AI Transparency IG. This is a forward-looking roadmap request, not a technical change to current content.

## Analysis

### Context
The AI Transparency IG (January 2026 ballot) is authored against FHIR R4. The HL7 FHIR ecosystem is moving toward R5 (current published release) and R6 (in development). The submitter is asking the work group to articulate a version migration strategy — when and how the IG will provide native R5 and R6 support rather than relying solely on R4/R4B packages.

### Key Considerations
1. **Low technical barrier today**: The IG states it uses no features changed between R4 and R4B, suggesting the profile and extension content may port to R5 with modest effort. However, R5 introduced structural changes to resources like Provenance, which is central to this IG.
2. **Roadmap transparency**: Ballot reviewers and implementers benefit from knowing whether R5/R6 support is planned, in progress, or deferred — even if timelines are approximate.
3. **Scope**: This is a documentation/roadmap request, not a technical change. It could be addressed by adding a short section (e.g., "Version Support and Roadmap") to the IG's landing page or a dedicated planning page.
4. **Work group capacity**: Committing to specific R5/R6 timelines may overextend a work group in early ballot phases. A statement of intent without hard dates is a reasonable middle ground.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Add a "Version Support and Roadmap" section to the IG landing page or a dedicated planning page. The section should state that R5 and R6 versions are planned, describe the anticipated approach (e.g., multi-version package publication, assessment of R5 Provenance changes), and note that timelines will be communicated as the IG matures. Do not commit to specific release dates at this stage.

**Rationale:** Addresses the submitter's core concern — transparency about version plans — without overcommitting the work group. A general statement of intent with technical considerations is informative for implementers and ballot reviewers while remaining achievable within the current ballot cycle.

### Disposition B: Persuasive
**Action:** Accept the request as stated — add intentions *and* timelines for R5 and R6 versions of the IG.

**Rationale:** Full acceptance signals strong commitment to multi-version support and gives implementers concrete planning information. However, providing timelines at this stage of IG maturity (first ballot) carries risk of missing stated dates, which could erode credibility.

### Disposition C: Not Persuasive
**Action:** Decline the request. Note that the IG is in early ballot and version planning is premature. The work group will address multi-version support in future ballot cycles as the IG stabilizes.

**Rationale:** The IG is still undergoing significant structural feedback (90 ballot issues across 9 categories). Focusing on R4 stabilization before committing to R5/R6 timelines is pragmatic. However, this disposition risks the appearance of not engaging with version-forward planning, which is a reasonable concern given the FHIR ecosystem trajectory.

## Recommendation

**Disposition A: Persuasive with Modification** — Add a roadmap section stating R5/R6 support is planned without committing to specific timelines. This directly addresses the submitter's request for transparency while being realistic about the IG's maturity level. The work group can update the timelines as the IG stabilizes through subsequent ballots.
