## Description

The submitter questions the "Operations" subsection under Context in the AI Transparency IG, which currently lists only:

- Model Context Protocol (MCP)
- Agent to Agent (A2A)

Two concerns are raised:

1. **Ambiguity of purpose:** If MCP/A2A are only means to fetch data, then the resulting data can be encoded as input prompts (already modeled in the IG). If the intent is to represent the operation itself, the model lacks dedicated objects for operation attributes (name, endpoint, protocol, arguments, retries, etc.).

2. **Rapidly evolving landscape:** Given that skills are now state of the art, A2A is in its infancy, and MCP is beginning to reveal pitfalls, the submitter suggests **removing the Operations section entirely** and modeling only the data that comes out of these operations as context using existing input prompt mechanisms.

## Analysis

### Core Issue

The Operations subsection under Context in the AI Transparency IG is underspecified. It lists MCP and A2A as bullet points without explaining:
- What aspect of these protocols/operations the IG intends to capture
- How to encode operation metadata (endpoint, arguments, retries, protocol) in the FHIR model
- Why protocol-level detail matters for AI transparency (vs. simply capturing the data that was provided as context)

### Ballot Context

This is one of several tightly coupled "Data Sources & Context" tickets (FHIR-54233, FHIR-54438, FHIR-54210, FHIR-54239) from the January ballot that collectively argue the Context observability factor subcategories are poorly defined and not adequately represented in the FHIR model. FHIR-54438 in particular raises nearly identical concerns about the Operations section.

### Key Considerations

1. **FHIR-54438 overlap:** These two tickets should be resolved together or one should reference the other. FHIR-54438 provides a more structured analysis of the same problem, suggesting either renaming to "Protocols / Interaction Protocols" or broadening the scope with clear categories.

2. **Removal vs. elaboration:** The submitter proposes removing Operations entirely. FHIR-54438's submitter proposes clarifying scope. Both agree the current text is insufficient.

3. **Rapidly evolving technology:** The submitter's point about the fast-moving nature of MCP, A2A, skills, and similar technologies is valid — anchoring the IG to specific protocol names risks rapid obsolescence.

4. **Model gap:** If Operations is retained, the IG needs to define FHIR resource patterns to encode operation attributes. No such model currently exists in the IG.

## Proposed Dispositions

### Disposition A: Persuasive (Remove Operations subsection)
Accept the submitter's suggestion to remove the Operations subsection entirely. Model the data returned by operations as context input prompts using existing mechanisms (DocumentReference-based input prompt pattern). Add a brief informative note that context data may be retrieved via various protocols (MCP, A2A, REST APIs, etc.) without attempting to enumerate or model them. This avoids coupling the IG to specific rapidly-evolving protocols.

**Pros:** Simplest resolution; avoids modeling something the IG has no resources for; aligns with the submitter's intent; reduces risk of obsolescence.
**Cons:** Loses the ability to trace *how* context was obtained (protocol/operation metadata), which some implementers may want for auditability.

### Disposition B: Persuasive with Modification (Clarify and generalize, don't remove)
Acknowledge the concern but retain a revised Operations subsection. Rename to "Interaction Protocols" or "Context Retrieval Methods" to avoid confusion with FHIR operations. Replace the MCP/A2A bullet list with a general statement that context may be obtained via various interaction protocols (with MCP and A2A as non-exhaustive examples). Add a note that the IG does not currently define a FHIR model for operation/protocol metadata, but this may be addressed in future iterations. Coordinate with FHIR-54438.

**Pros:** Preserves the concept while addressing ambiguity; leaves room for future elaboration; acknowledges the submitter's valid concerns.
**Cons:** Still lacks a concrete FHIR model; may remain underspecified.

### Disposition C: Not Persuasive (Keep as-is with minor clarification)
Keep the Operations subsection as informative context about the technology landscape. Add a brief clarifying sentence that MCP and A2A are listed as examples of protocols used in AI agent interactions, and that the IG does not currently model operation-level metadata. The submitter's concern about rapid technology evolution is acknowledged but considered acceptable for an informative section.

**Pros:** Minimal change; preserves informational value.
**Cons:** Doesn't address the core modeling gap; the submitter and FHIR-54438 submitter both identify real confusion that would persist.

## Recommendation

**Disposition B: Persuasive with Modification.** The Operations subsection addresses a real concept (how AI systems retrieve context) but is too narrowly framed around two specific protocols. Renaming the subsection, generalizing the content to be protocol-agnostic, and explicitly stating that protocol-level metadata modeling is deferred to a future IG cycle addresses the submitter's concerns without losing useful conceptual framing. This should be coordinated with FHIR-54438 to ensure consistent resolution across both tickets.
