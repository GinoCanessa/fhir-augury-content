## Summary

This ticket requests the addition of a new FHIR extension to AU Base for recording tribe/mob (community) affiliation for indigenous Australians. The request originates from a Zulip discussion during the 2023-11 Melbourne Connectathon, subsequently captured as GitHub issue [hl7au/au-fhir-base#813](https://github.com/hl7au/au-fhir-base/issues/813).

Grahame Grieve raised the question of whether AU Base should define an extension for tribe/mob. Brett Esler confirmed no prior discussion existed. Community feedback established that:

- There is no formal terminology or code list for tribe/mob â€” it is typically "local knowledge" and rarely captured in structured form in clinical systems.
- Gavin Morris confirmed Communicare does not record this data in a structured way; the closest concept is "skin group," which is distinct.
- Grahame proposed a strawman extension named `community-affiliation` using `valueString`.
- Michael Lawley suggested using `CodeableConcept.text` to leave room for future coded values.
- Danielle Tavares noted tradeoffs: String mitigates risk of inappropriate terminology and preserves patient ability to self-describe; CodeableConcept sets an expectation of future terminology support.
- Grahame expressed a preference for `valueString`, arguing the data "isn't ever codeable," especially when considering immigrant communities.

## Proposed Dispositions

### Disposition A: Accept as Submitted â€” Define `community-affiliation` extension with `valueString`

Accept the change request and define a new extension `http://hl7.org.au/fhir/StructureDefinition/community-affiliation` on the Patient resource using `valueString` as the data type, per Grahame Grieve's strawman proposal.

**Rationale:** There is no existing terminology for tribe/mob, the data is inherently free-text and patient-reported, and using a simple string type avoids setting an expectation of future coded values that may never materialise. This aligns with Grahame's strong preference and the practical reality reported by Communicare (Gavin Morris) that this data is not captured in a structured or codeable way.

**Risks:** Using `valueString` forecloses the option of later adding coded values without a breaking change or a new extension. The extension name `community-affiliation` may be too broad or ambiguous.

### Disposition B: Accept with Modification â€” Use `CodeableConcept` data type

Accept the change request but define the extension using `valueCodeableConcept` instead of `valueString`, allowing free text via `CodeableConcept.text` while leaving room for future terminology bindings.

**Rationale:** This follows Michael Lawley's suggestion and Danielle Tavares' observation that CodeableConcept sets a pathway for future terminology support. If an authoritative list of First Nations groups or community names is ever developed, it could be bound without changing the extension's data type.

**Risks:** May set an expectation that a coded terminology will be developed, which may not be appropriate or achievable. Adds complexity for implementers who only need to capture a simple string value. Grahame explicitly argued against this approach.

### Disposition C: Defer â€” Request further community consultation

Defer the ticket pending broader consultation with indigenous community stakeholders, clinical informaticists, and potentially the Australian Digital Health Agency (ADHA). Request a formal requirements analysis covering the scope of the extension (Patient only vs. broader applicability), naming, data type, and cultural appropriateness.

**Rationale:** The discussion so far has been limited to a small number of participants at one connectathon. Given the cultural sensitivity of recording indigenous identity information and the absence of any formal requirements or use cases, a more deliberate community engagement process may be warranted before committing to a design.

**Risks:** Further delays an extension that has been informally discussed since late 2023 without progress. May lose momentum if no one champions the consultation effort.

## Recommendation

**Disposition A: Accept as Submitted** â€” Define the `community-affiliation` extension with `valueString`.

The community discussion, though informal, reached a clear consensus that this data is free-text and patient-reported with no suitable terminology available. The retraction of the related coded ethnicity effort (FHIR-46325) reinforces the difficulty of establishing coded terminologies in this space. A simple `valueString` extension is pragmatic, low-risk, and immediately useful for implementers who need to capture this information. Naming and scope details can be refined during the ballot process.
