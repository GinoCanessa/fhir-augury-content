# FHIR-48853: AUCDI Encounter – Clinical Context Modality Mapped to Encounter.class Will Cause Conflict with Existing Usage

## Summary

This Change Request (CR), raised by Heath Frankel, identifies a conflict between the AUCDI (Australian Clinical Data for Interoperability) mapping of "Encounter – clinical context Modality" to the AU Core `Encounter.class` element and the established FHIR semantics and existing system implementations of that element.

The AUCDI defines "modality" as "the type of communication or method used to conduct the encounter" (e.g., Face-to-face, Telephone, Video, SMS, Email). However, `Encounter.class` is defined in FHIR as "Classification of patient encounter" (e.g., inpatient, ambulatory, emergency, home health, virtual) and is constrained to 1..1 Coding — meaning only one value can be provided. Multiple existing systems already use the standard FHIR encounter classification values.

The CR makes two recommendations:
1. **Remove the AUCDI Encounter modality mapping** from AU Core `Encounter.class` and find an alternative element to represent modality separately.
2. **Remove the modality codes** (PHONE, VIDEO, EMAIL, SMS) from the AU Base ActEncounterCode – AU Extended value set.

**Recommendation #2 has already been resolved** via the related CR [FHIR-47120](https://jira.hl7.org/browse/FHIR-47120) (Applied, Persuasive with Modification) which removed PHONE, VIDEO, EMAIL, and SMS from the value set. The remaining open question is **Recommendation #1**: removing the AUCDI modality mapping from `Encounter.class`.

## FHIR Path Context

- **Encounter.class** — Classification of patient encounter context (1..1, Coding). Standard values: inpatient, ambulatory, emergency, home health, virtual. The AU Extended value set previously included PHONE, VIDEO, EMAIL, SMS (now removed via FHIR-47120).

## Analysis

The core issue is a semantic mismatch: AUCDI's "modality" concept (how the encounter was conducted) is fundamentally different from FHIR's `Encounter.class` concept (what type of patient encounter it is). Modality values like phone or video are qualifiers that can apply across multiple encounter classes (e.g., an ambulatory visit could be conducted via phone or video). Since `Encounter.class` is 1..1, mapping modality there forces a choice between classification and communication method.

With FHIR-47120 having already removed the problematic codes from the value set, the AUCDI modality mapping to `Encounter.class` is now broken — the example values (Face-to-face, Telephone, Video, SMS, Email) no longer exist in the bound value set. The remaining action is to formally remove this mapping from the AU Core AUCDI mapping table and identify an alternative element or extension for modality.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Accept the CR as-is. Remove the AUCDI "Encounter – clinical context Modality" mapping from `Encounter.class` in the AU Core AUCDI mapping table. Document that modality representation requires a separate element or extension (to be addressed in a future CR).

**Rationale:** The semantic mismatch between modality and encounter classification is well-established. The value set codes have already been removed (FHIR-47120). The mapping is now factually incorrect since the example values no longer exist. This is a clean-up action that aligns AU Core with the already-applied AU Base change.

### Disposition B: Persuasive with Modification
**Action:** Remove the AUCDI modality mapping from `Encounter.class` AND explicitly specify the alternative mapping target. For example, map modality to a new AU Core extension (e.g., `Encounter.extension:encounterModality`) or to an existing FHIR element such as `Encounter.serviceType` with an appropriate value set, ensuring the AUCDI concept is not left unmapped.

**Rationale:** Simply removing the mapping without identifying an alternative leaves the AUCDI modality concept unrepresented in AU Core, which could be a gap for clinical data capture. Addressing both the removal and the replacement in one CR provides a complete solution and avoids a dangling AUCDI requirement.

### Disposition C: Not Persuasive
**Action:** Reject the CR. Retain the AUCDI modality mapping to `Encounter.class` and instead restore or revise the modality codes in the value set to coexist with encounter classification codes.

**Rationale:** Some GP systems historically used these more specific encounter type codes (PHONE, VIDEO) as noted by Brett Esler. Removing the mapping without a ready alternative could leave implementations without guidance for modality capture. However, this conflicts with the already-applied FHIR-47120 resolution and the FHIR-defined semantics of `Encounter.class`.

## Recommendation

**Disposition A: Persuasive** is recommended.

The value set changes have already been applied via FHIR-47120, making the current AUCDI mapping to `Encounter.class` factually incorrect — the example modality values no longer exist in the bound value set. The semantic mismatch between modality (communication method) and encounter class (patient encounter classification) is clear, and this CR correctly identifies the need to remove the broken mapping. The question of where modality *should* be mapped can be addressed in a separate, focused CR, which is appropriate given its complexity and the urgency of correcting the current mapping before AU Core R1 publication.
