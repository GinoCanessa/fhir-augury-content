## Description

Request to create a new AU Base profile of Identifier for **Healthcare Administration Entity (HAE)** identifiers. HAEs are issued by Services Australia under amendments to the Healthcare Identifiers Act 2010. They are organisational identifiers similar to HPI-Os, referred to as *Healthcare Administration Entity Organisation* identifiers.

The proposed profile would follow the existing pattern established by the [AU HPI-O](https://hl7.org.au/fhir/6.0.0/StructureDefinition-au-hpio.html) identifier profile, using the namespace `http://ns.electronichealth.net.au/id/hi/hae/1.0`.

A draft CI build of the profile is available at: [AU HAE - AU Base IG v6.0.1-ci-build](https://build.fhir.org/ig/hl7au/au-fhir-base/branches/cr-FHIR-54928-HAE/StructureDefinition-au-hae.html)

### Key Open Issue

The primary open question is **what Identifier.type code to assign to HAE**. The initial draft used `NOI` (National Organisation Identifier), but AU Core Organization constrains `NOI` to HPI-O and PAI-O only. This has been actively discussed and a subsequent ticket ([FHIR-56102](https://jira.hl7.org/browse/FHIR-56102)) proposes creating a dedicated `HAE` code in the IdentifierType AU code system. A Connectathon breakout session in March 2026 developed proposals around identifier types for national identifiers that will be brought through the AU FHIR WG for decision.

## Proposed Dispositions

### Disposition A: Approve as Profile Creation with Deferred Type Resolution

**Action:** Accept the request to create the AU HAE Identifier profile in AU Base, modelled on AU HPI-O. Defer the choice of `Identifier.type` code to the outcome of [FHIR-56102](https://jira.hl7.org/browse/FHIR-56102) (which proposes a dedicated `HAE` type code).

**Rationale:** The HAE identifier is legislated under the Healthcare Identifiers Act 2010, has a registered namespace and OID, and a draft profile already exists. Creating the structural profile is well-justified. The identifier type question is being actively resolved through FHIR-56102 and the Connectathon breakout outcomes. Approving the profile creation now while linking the type code decision to FHIR-56102 avoids blocking progress.

**Impact:** New `StructureDefinition-au-hae` profile added to AU Base. Downstream impact on AU Core Organization will depend on FHIR-56102 outcome. No breaking changes.

### Disposition B: Approve with NOI as Interim Type Code

**Action:** Accept the profile using `NOI` as the `Identifier.type` and coordinate with [FHIR-55636](https://jira.hl7.org/browse/FHIR-55636) to redefine `NOI` to encompass all HI Service national organisation identifiers (HPI-O, PAI-O, HSP-O, HAE).

**Rationale:** `NOI` already covers HPI-O and PAI-O which are structurally similar identifiers. Expanding its definition to include HAE (and HSP-O) is consistent with the original intent of "National Organisation Identifier." This avoids code system proliferation and maintains backward compatibility for existing systems using `NOI`.

**Impact:** Requires updating the `NOI` definition in the AU Extended code system and relaxing the AU Core Organization `au-core-org-01` constraint. Existing systems using `NOI` for HPI-O/PAI-O would not break, but their queries might return additional identifier types.

### Disposition C: Defer Until Identifier Type Strategy is Finalized

**Action:** Defer the ticket until the AU FHIR WG has finalized its proposals on identifier types arising from the March 2026 Connectathon breakout session and the Confluence investigation.

**Rationale:** The Connectathon breakout session produced a set of proposals that are still being worked through the WG. Creating the profile before these decisions are made risks rework if the type code, naming, or validation approach changes. The Zulip discussions indicate decisions are expected "over the coming months."

**Impact:** Delays availability of the HAE profile in AU Base. No impact to existing profiles or code systems. Risk that implementers building systems for Services Australia HAE identifiers lack a standard profile in the interim.

## Recommendation

**Disposition A: Approve as Profile Creation with Deferred Type Resolution** is recommended.

The HAE identifier is legislated and already has a draft CI build. The profile structure (namespace, system URI, value format) is well-defined and follows the established AU HPI-O pattern. The only open question — which `Identifier.type` code to use — is being actively addressed by the dedicated follow-up ticket FHIR-56102 and the Connectathon breakout proposals. Approving the profile creation now, with an explicit dependency on FHIR-56102 for the type code, enables progress without prejudging the type decision. Deferral (Disposition C) is unnecessarily cautious given that the structural profile is uncontroversial and the type code can be updated before publication.
