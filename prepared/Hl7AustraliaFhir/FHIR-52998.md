Stephen Chu reports that the AU Base `au-location-physical-type-extended` ValueSet includes a "vi" (virtual) code for `Location.physicalType`, but this is semantically inappropriate. The FHIR definition of `physicalType` is "physical form of the location, e.g. building, room, vehicle, road" — a virtual location has no physical form. Meanwhile, `Location.type` already appropriately includes virtual location codes via the `ServiceDeliveryLocationRoleType - AU Extended` ValueSet, making the "vi" code in `physicalType` redundant and conceptually misplaced.

## 5. Proposed Dispositions

### Disposition A: Persuasive — Remove "vi" code from `au-location-physical-type-extended`

**Action:** Remove the "vi" (virtual) code from the `au-location-physical-type-extended` ValueSet bound to `Location.physicalType`. Virtual location classification is already available via `Location.type` using the `ServiceDeliveryLocationRoleType - AU Extended` ValueSet, which is the semantically correct element for this concept.

**Rationale:**
- The FHIR definition of `physicalType` explicitly describes "physical form" (building, room, vehicle, road). A virtual location contradicts this definition.
- `Location.type` already captures the virtual location concept appropriately in AU Base/AU Core.
- PCA (the original introducer of the code) has explicitly stated they have no objection to removal.
- International practice (e.g., HL7 Norway) chose `Location.type` rather than `physicalType` for virtual codes.
- The WG co-chairs have already triaged and accepted this ticket.

**Risk:** Implementers currently using the "vi" code in `physicalType` would need to migrate to using `Location.type` instead. This is a breaking change for systems that rely on the "vi" code in this specific ValueSet.

### Disposition B: Persuasive with Modification — Deprecate "vi" in `physicalType` and provide migration guidance

**Action:** Mark the "vi" code as deprecated in `au-location-physical-type-extended` (rather than immediate removal), add guidance directing implementers to use `Location.type` for virtual locations, and plan full removal in a future AU Base release.

**Rationale:**
- Achieves the same semantic correction as Disposition A but provides a gentler transition path.
- The "vi" code has been in AU Base since release 4.0.0 (2022) and may be in active use.
- A deprecation period gives implementers time to adjust.

**Risk:** Extends the period during which the semantically incorrect code is available, potentially leading to continued adoption by new implementers who don't read deprecation notices.

### Disposition C: Not Persuasive — Retain "vi" code in `physicalType`

**Action:** No change. Keep the "vi" code in `au-location-physical-type-extended`.

**Rationale:**
- The code has been in AU Base since 2022 and was referenced as prior art in international FHIR core discussions (FHIR-36815).
- FHIR core itself considered adding a "vi" code to `physicalType` (FHIR-36815 was resolved as Persuasive with Modification).
- Removing it could break existing implementations that depend on it.
- The semantic argument, while valid in a strict reading, is somewhat academic — implementers may find it useful to have virtual as a physical type for filtering and classification.

**Risk:** Perpetuates a semantic inconsistency where `physicalType` contains a non-physical concept, which may confuse implementers and conflict with the FHIR core definition of the element.

## 6. Recommendation

**Disposition A: Persuasive** is recommended.

The semantic argument is clear and well-founded: `physicalType` describes physical form, and "virtual" is explicitly not physical. The virtual location concept is already properly represented in `Location.type`, making the "vi" code in `physicalType` both redundant and misleading. Critically, the PCA project that originally introduced the code has confirmed they support its removal, and the WG co-chairs have already triaged and accepted the ticket. International community practice (particularly HL7 Norway's approach) corroborates using `Location.type` rather than `physicalType` for virtual classification. The change aligns AU Base with the intended semantics of the FHIR core specification.
