## Summary

Grahame Grieve raises that the AU Base specification does not make clear how the Samoan gender identity tradition of [Fa'afafine](https://en.wikipedia.org/wiki/Fa%CA%BBafafine) should be handled. The reporter suggests the spec should at minimum provide clear guidance and ideally add Fa'afafine as a specific gender identity code in the extension's value set.

Brett Esler (brettesler) responded noting that the existing AU Base [sex and gender guidance](https://build.fhir.org/ig/hl7au/au-fhir-base/sexgender.html#gender-identity-gi) already supports the "[I/They] use a different term (please specify)" concept by sending only `text` with no `coding`. The NCTS value set bound to the gender identity extension is aligned with the Australian Bureau of Statistics (ABS) standard and the AIHW metadata standard. No specific Fa'afafine code currently exists in the NCTS value set; adding one would require a formal request to NCTS to change the [gender-identity-response-1](https://www.healthterminologies.gov.au/integration/R4/fhir/ValueSet/gender-identity-response-1) value set.

The ticket was referred from the AU Core TDG and accepted by the FHIR WG co-chairs on 2025-09-22.

## Analysis

### Core Question
Should AU Base add a specific coded concept for Fa'afafine to its gender identity value set, or is the existing text-only guidance sufficient?

### Key Considerations
1. **Current mechanism exists**: AU Base already supports uncoded gender identities via the "[I/They] use a different term" guidance, allowing `text`-only representation without a `coding`. This means Fa'afafine *can* be represented today.
2. **Value set governance**: The NCTS gender identity value set is externally governed (aligned with ABS and AIHW standards). Adding codes requires a formal NCTS change request, which is outside the direct control of HL7 Australia FHIR.
3. **Scope of change**: Adding a specific code for one cultural gender identity raises questions about which other cultural identities (e.g., Hijra, Two-Spirit, Māhū) should also be coded, and whether that belongs in AU Base or in a broader international terminology effort.
4. **Documentation gap**: Even if no new code is added, the spec could benefit from clearer documentation showing how to represent culturally specific gender identities using the existing text mechanism.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Add Documentation Only
**Action**: Enhance the AU Base sex and gender guidance page to explicitly describe how culturally specific gender identities (including Fa'afafine) should be represented using the existing text-only mechanism. Add an example showing a gender identity observation with `text` = "Fa'afafine" and no `coding`. Do not add a new code to the value set.

**Rationale**: The existing extensible binding and text-only mechanism already support this use case. Clear documentation with a worked example addresses the reporter's concern that the specification "doesn't make clear" how to handle Fa'afafine, without requiring an NCTS value set change request or opening the broader question of which cultural gender identities merit specific codes.

### Disposition B: Persuasive — Add Code and Documentation
**Action**: Submit an NCTS change request to add a Fa'afafine concept to the gender-identity-response-1 value set. In parallel, update AU Base guidance to document the text-only approach as an interim measure and add narrative explaining the intent to support culturally specific gender identities as coded concepts.

**Rationale**: Grahame Grieve specifically suggested Fa'afafine "should be yet another gender in the codes for the extension." A discrete code enables consistent analytics and decision support. Given AU Base's Pacific regional context, first-class coded support for Fa'afafine is appropriate.

### Disposition C: Not Persuasive
**Action**: No change. The existing guidance and extensible binding are sufficient.

**Rationale**: The text-only representation is already documented. The value set is aligned with national standards (ABS, AIHW), and adding culturally specific codes is a terminology governance matter for NCTS, not an AU Base specification issue. Implementers can already represent Fa'afafine today.

---

## Recommendation

**Disposition A: Persuasive with Modification — Add Documentation Only**

This disposition best balances the reporter's legitimate concern with practical constraints. The specification already *supports* Fa'afafine representation but fails to make this clear — which is exactly what the ticket title states. Adding explicit guidance and a worked example directly resolves the reported issue. Attempting to add a specific code (Disposition B) depends on an external NCTS process and raises scope questions about other cultural identities, while doing nothing (Disposition C) leaves the documentation gap the reporter identified.
