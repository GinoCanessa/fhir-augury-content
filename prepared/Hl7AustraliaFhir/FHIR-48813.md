## Description Summary

Following the resolution of [FHIR-47141](https://jira.hl7.org/browse/FHIR-47141) (which added a new eRequesting Responder Actor and Capability Statement), the AU eRequesting Server CapabilityStatement definition needs revision. The current definition no longer accurately reflects the scenario where a Placer system implements the AU eRequesting Server CapabilityStatement.

The reporter (Heath Frankel) offers two alternative definitions:

1. **Shorter/alignment-oriented** (acknowledged as vague): _"This CapabilityStatement describes the basic rules for the AU eRequesting Server actor that is a system that makes diagnostic service requests accessible and can accept diagnostic service requests."_

2. **More descriptive** (reporter's preference): _"This CapabilityStatement describes the basic rules for the AU eRequesting Server actor that is a system that makes diagnostic service requests accessible to filler and patient record systems and can accept diagnostic service requests from placer systems."_

The reporter notes a tension between keeping definitions vague to align with existing actor definition patterns versus providing more useful, specific definitions that clarify the target audience of the system's capabilities.

## Analysis

### Context

FHIR-47141 was a significant architectural change to AU eRequesting: it introduced a Responder Actor that allows a Placer system to make its own ServiceRequests available via read/search, rather than only creating them on a central server. This change means the Server CapabilityStatement can now be implemented by both:

- **Traditional server/repository systems** that receive and store requests from placers
- **Placer systems** that expose their own originated requests to fillers and patient access systems

The current Server CapabilityStatement definition was written assuming only the first scenario. With the Responder Actor in place, the definition needs to accommodate both.

### Key Considerations

1. **Companion tickets**: FHIR-48812 (Actor definition) and FHIR-48811 (obligation changes) form a tightly coupled set with this ticket. Ideally they should be resolved together to maintain consistency across Actor, CapabilityStatement, and obligation definitions.

2. **Newer overlapping tickets**: FHIR-52184 and FHIR-52425 also modify the same Server CapabilityStatement text. FHIR-52425 has already been applied with modifications. The workgroup should consider whether this ticket's changes have already been partially addressed or superseded.

3. **Specificity vs. alignment**: The reporter acknowledges the tension between vague-but-aligned definitions and more useful, specific definitions. The more descriptive option explicitly names the audiences (filler systems, patient record systems, placer systems), which adds clarity but could be seen as limiting.

4. **Scope broadening**: FHIR-52180 and FHIR-52184 propose removing "diagnostic" from actor and capability definitions. If the workgroup intends to broaden scope, the definition adopted here should be compatible with that direction.

## Proposed Dispositions

### Disposition A: Persuasive — Accept the more descriptive definition

**Action:** Update the AU eRequesting Server CapabilityStatement definition to: _"This CapabilityStatement describes the basic rules for the AU eRequesting Server actor that is a system that makes diagnostic service requests accessible to filler and patient record systems and can accept diagnostic service requests from placer systems."_

**Rationale:** This is the reporter's preferred option and provides the most clarity about the Server's role in the eRequesting ecosystem. By naming the specific system types that interact with the Server, implementers gain a clearer understanding of what the CapabilityStatement covers. This aligns with the intent of FHIR-47141 by explicitly acknowledging the Placer-as-server scenario. Note: the typo "is is" in the original suggestion should be corrected to "is".

**Trade-off:** Less aligned with the terse style of existing actor definitions; may need revision if FHIR-52184 (broadening scope) is accepted.

### Disposition B: Persuasive with Modification — Accept the shorter definition, pending scope decisions

**Action:** Update the AU eRequesting Server CapabilityStatement definition to: _"This CapabilityStatement describes the basic rules for the AU eRequesting Server actor that is a system that makes diagnostic service requests accessible and can accept diagnostic service requests."_ Additionally, coordinate with FHIR-48812 (Actor definition) and FHIR-48811 (obligations) to ensure consistent updates across all three tickets.

**Rationale:** This option addresses the immediate need to update the definition following FHIR-47141, while maintaining alignment with the existing pattern of concise actor definitions (as established by FHIR-46802 and FHIR-52413). It avoids committing to specific audience types, leaving flexibility for the scope broadening proposed in FHIR-52184. The workgroup can add more specificity later if needed.

**Trade-off:** The reporter himself acknowledges this option is "fairly vague and unhelpful." It does not differentiate the Server's role from other actors.

### Disposition C: Not Persuasive — Defer to scope clarification tickets

**Action:** No change to the CapabilityStatement definition at this time.

**Rationale:** Given that FHIR-52425 (Applied, Persuasive with Modification) has already updated the Server CapabilityStatement to clarify scope, and FHIR-52184 proposes further changes to broaden scope beyond diagnostics, this ticket's concern may already be partially addressed or may be better resolved as part of the broader scope clarification effort. Resolving this ticket now risks needing to re-edit the definition when FHIR-52184 is resolved.

**Trade-off:** Leaves the current definition potentially inaccurate following FHIR-47141 until the scope tickets are resolved. The companion tickets (FHIR-48811, FHIR-48812) would also remain unaddressed.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The CapabilityStatement definition must be updated to reflect the FHIR-47141 change—the current definition is inaccurate for the Placer-as-server scenario. The shorter definition maintains consistency with established actor definition patterns and does not conflict with the scope broadening under consideration in FHIR-52184. The workgroup should resolve this ticket together with its companions (FHIR-48812, FHIR-48811) to ensure a consistent set of changes, and should note the existing "is is" typo in the suggested text needs correction to "is".
