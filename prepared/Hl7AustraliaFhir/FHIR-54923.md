## Description

This ticket requests creation of a new AU Base profile of `Identifier` for **HSP-O** (Healthcare Support Service Provider for Organisation) identifiers. HSP-Os are a new category of identifier issued by Services Australia under amendments to the Healthcare Identifiers Act 2010. They are structurally similar to existing HPI-O identifiers.

The reporter has provided extensive reference material:
- **Namespace:** `http://ns.electronichealth.net.au/id/hi/hspo/1.0` ([ADHA namespace assertion](https://developer.digitalhealth.gov.au/namespaces/id/hi/hspo/1.0/index.html))
- **OID Register:** [Healthcare Identifiers (HI) OID Register](https://developer.digitalhealth.gov.au/resources/1003-healthcare-identifiers-hi-oid-register)
- **Legislation:** [Healthcare Identifiers Act 2010](https://www.legislation.gov.au/C2010A00072/latest/text) and [DoHA FAQ on amendments](https://www.health.gov.au/sites/default/files/2025-12/amendments-to-the-healthcare-identifiers-act-2010-frequently-asked-questions.pdf)
- **Model profile:** [AU HPI-O (v6.0.0)](https://hl7.org.au/fhir/6.0.0/StructureDefinition-au-hpio.html)
- **CI build draft:** [AU HSP-O v6.0.1-ci-build](https://build.fhir.org/ig/hl7au/au-fhir-base/branches/cr-FHIR-54923-HSP-O/StructureDefinition-au-hspo.html)

A superseded GitHub issue exists at [hl7au/au-fhir-base#1037](https://github.com/hl7au/au-fhir-base/issues/1037).

## Key Issues Identified in Comments

1. **Identifier type code:** The initial draft uses `NOI` (National Organisation Identifier), but this creates a conflict with the AU Core Organization constraint `au-core-org-01` which restricts NOI to HPI-O and PAI-O only. A decision was made at the 2026-03 HL7 Australia Sydney FHIR Connectathon that a **new identifier type code** will likely be allocated specifically to HSP-O.
2. **Example values:** Awaiting guidance from Services Australia on permitted example values for HSP-O before an example can be added to the profile.
3. **Prefix validation:** The HSP-O identifier has a distinctive prefix (`800364100[0-9]`) that can be used for validation, similar to HPI-O and PAI-O prefixes.

## Proposed Dispositions

### Disposition A: Accept — Create the HSP-O profile with new HSPO type code

**Action:** Accept the request. Create the AU HSP-O identifier profile modelled on AU HPI-O, using the namespace `http://ns.electronichealth.net.au/id/hi/hspo/1.0` and a new `HSPO` identifier type code (per FHIR-56101). Include a prefix validation invariant for `800364100[0-9]`. Defer example values until Services Australia provides guidance.

**Rationale:** The Connectathon breakout session concluded that a dedicated type code should be created for HSP-O rather than overloading NOI. A CI build already exists demonstrating the profile. This approach avoids disrupting the existing NOI/HPI-O/PAI-O ecosystem and AU Core constraints.

**Considerations:** Depends on FHIR-56101 (new type code) being accepted in parallel. The prefix invariant alignment (FHIR-56104) should also be resolved. This approach is the cleanest path forward and aligns with community consensus from the Connectathon.

### Disposition B: Accept — Create the HSP-O profile reusing the NOI type code

**Action:** Accept the request. Create the AU HSP-O identifier profile using the existing `NOI` type code and require relaxation of the AU Core Organization constraint `au-core-org-01` (per FHIR-55738) to permit HSP-O alongside HPI-O and PAI-O.

**Rationale:** NOI is defined as "National Organisation Identifier" and HSP-O is a national organisation identifier, so reusing NOI is semantically defensible. This avoids proliferating type codes.

**Considerations:** Requires changes to AU Core (FHIR-55738) and the NOI definition (FHIR-55636). Andy Bond's Zulip feedback questions whether continuing to aggregate identifiers under NOI is sound practice. The Connectathon consensus favours a distinct code, making this approach less aligned with community direction.

### Disposition C: Defer — Wait for identifier type decisions before creating the profile

**Action:** Defer the ticket until the AU FHIR WG formally decides on identifier type handling for new national organisation identifiers (NOI scope, new codes, etc.).

**Rationale:** The Connectathon breakout session produced proposals that are still pending formal WG decision. Creating the profile now risks rework if the identifier type approach changes.

**Considerations:** A CI build already exists and the legislation is in effect — Services Australia may begin issuing HSP-Os before the WG completes its deliberations. Deferral creates a gap where implementers have no official profile to reference.

---

## Recommendation

**Disposition A: Accept — Create the HSP-O profile with new HSPO type code.**

This is the strongest option because: (1) a CI build draft already demonstrates feasibility, (2) the Connectathon breakout session reached consensus on creating distinct type codes for new identifiers, (3) it avoids disrupting existing AU Core constraints, and (4) the companion ticket FHIR-56101 already proposes the specific `HSPO` code. The profile should be accepted in coordination with FHIR-56101 and FHIR-56104 to ensure the type code and invariant expression are correct from the outset.
