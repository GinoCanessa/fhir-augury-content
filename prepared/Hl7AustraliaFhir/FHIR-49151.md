## Description

[FHIR-47120](https://jira.hl7.org/browse/FHIR-47120) introduced a breaking change that removed the codes EMAIL, SMS, PHONE, and VIDEO from the ValueSet bound to `Encounter.class` in AU Base. QA feedback from Danielle (dt-r) in [hl7au/au-fhir-base#906](https://github.com/hl7au/au-fhir-base/issues/906) recommended that additional guidance be added to the AU Base Encounter profile for implementers who had adopted the now-removed codes.

## Analysis

### Context
FHIR-47120 (Applied, Persuasive with Modification) removed four encounter modality codes (PHONE, VIDEO, EMAIL, SMS) from the ActEncounterCode - AU Extended value set bound to `Encounter.class`. These codes were originally added to represent more specific virtual encounter types but were deemed inappropriate as encounter *class* codes — they are modalities applicable across multiple encounter classes (ambulatory, home health, etc.). The recommended replacement is the existing `VR` (virtual) code.

This was acknowledged as a breaking change. A Zulip announcement and survey were posted to identify affected implementations. GitHub issue #906 raised QA concerns about the change log quality and recommended adding guidance for affected implementers. GitHub issue #925 subsequently addressed STU Notes for breaking changes more broadly. FHIR-49151 was created specifically to track the addition of implementation/migration guidance to the AU Base Encounter profile.

### Impact
- Implementers who adopted PHONE, VIDEO, EMAIL, or SMS codes for `Encounter.class` need clear direction on how to migrate.
- The codes were available in AU Base for some time; production systems may have used them.
- Without guidance, implementers may be unaware their data is non-conformant or may not know the recommended migration path (use `VR` instead).

### Open Questions
1. Has the work in GitHub issue #925 (STU Notes for breaking changes) already addressed this need, making FHIR-49151 redundant?
2. What form should the guidance take — a STU Note, a change log entry, or narrative implementation guidance on the profile page?
3. Should the guidance address data migration (existing stored data with removed codes) or only prospective usage?

---

## Proposed Dispositions

### Disposition A: Persuasive — Add an STU Note to AU Base Encounter

**Action:** Add a STU Note to the AU Base Encounter profile page stating that codes PHONE, VIDEO, EMAIL, and SMS have been removed from the `Encounter.class` value set as of this release. The note should advise implementers to use `VR` (virtual) instead and indicate that these modality codes were removed because they represent the mode of encounter rather than the class.

**Rationale:** Aligns with the commenter's suggestion that a STU Note is sufficient. STU Notes are a well-understood mechanism in FHIR IGs for highlighting important changes. This is a focused, low-effort change that directly addresses the need without over-engineering the solution.

**Pros:** Minimal effort; uses established convention; directly visible on the profile page.
**Cons:** Does not provide detailed migration guidance for data already in production.

### Disposition B: Persuasive with Modification — Add implementation guidance and change log improvements

**Action:** Add a brief implementation guidance section (or STU Note with expanded content) to the AU Base Encounter profile that:
1. Lists the removed codes (PHONE, VIDEO, EMAIL, SMS) and the replacement (`VR`).
2. Explains *why* they were removed (modality vs. class distinction).
3. Recommends implementers review existing data and map removed codes to `VR`.
4. Update the change log to explicitly list the removed codes (if not already done per GitHub issue #906).

**Rationale:** The original QA feedback in GitHub #906 specifically called for implementation guidance. Given this was a breaking change affecting production systems, a brief but substantive guidance note provides more value than a minimal STU Note alone. The change log improvement may already be done via #925 but should be verified.

**Pros:** Comprehensive; addresses both prospective and retrospective migration; fulfils the original QA recommendation.
**Cons:** Slightly more work; some of the change log work may already be complete.

### Disposition C: Not Persuasive — Already addressed by existing work

**Action:** Close as not persuasive, noting that the breaking change has already been communicated via the Zulip announcement, the implementation survey, and the STU Notes work in GitHub issue #925.

**Rationale:** The community has already been notified through multiple channels. The change log and STU Notes work in #925 may already cover the specific guidance needed. Adding further guidance may be redundant.

**Pros:** No additional work required.
**Cons:** If #925 did not specifically add Encounter-level guidance, implementers consulting the AU Base Encounter profile page would still lack direction. The survey suggests there may be affected production systems that need clear on-profile guidance.

---

## Recommendation

**Disposition B: Persuasive with Modification** — This is a confirmed breaking change with known production impact (a survey was conducted to identify affected implementations). The original QA feedback specifically requested implementation guidance on the AU Base Encounter profile. A STU Note combined with brief migration guidance (listing removed codes, the replacement `VR` code, and the rationale) provides the most value to implementers at modest effort. The work group should verify whether the change log improvements from GitHub #925 are already in place and ensure the guidance is consistent with the broader STU Notes approach adopted for this release.
