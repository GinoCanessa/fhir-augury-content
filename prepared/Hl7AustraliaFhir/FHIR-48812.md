## Description Summary

Following the resolution of [FHIR-47141](https://jira.hl7.org/browse/FHIR-47141) (which added the eRequesting Responder Actor and Capability Statement), the AU eRequesting Server Actor definition needs to be reviewed. The current definition is no longer representative of the scenario where a Placer system implements the AU eRequesting Server CapabilityStatement.

The reporter proposes two alternative definitions:

1. **Minimal (aligned with existing actor definitions):** "The AU eRequesting Server is a system that makes diagnostic service requests accessible and can accept diagnostic service requests."
2. **More descriptive (preferred by reporter):** "The AU eRequesting Server is a system that makes diagnostic service requests accessible to filler and patient record systems and can accept diagnostic service requests from placer systems."

The reporter notes that option 1 is "fairly vague and unhelpful" but was suggested to align with existing actor definitions, while option 2 provides clearer context about which systems interact with the Server.

## Analysis

### Context
FHIR-47141 introduced a new **Responder Actor** that allows a placer system to serve its own service requests without creating them on a separate server. This changed the role dynamics: previously, the Server Actor was the primary system hosting service requests; now, a placer system implementing the Responder can also make requests accessible. This means the Server Actor definition needs updating to accurately distinguish it from the Responder.

### Key Considerations
1. **Consistency vs. Clarity:** Option 1 aligns with the pattern used for other actor definitions but sacrifices specificity. Option 2 explicitly names the interacting systems (filler, patient record, placer) providing implementers with clearer guidance.
2. **Companion ticket FHIR-48813** addresses the identical issue for the CapabilityStatement text — these should be resolved in tandem to maintain consistency.
3. **FHIR-52180** suggests removing "diagnostic" entirely to future-proof the definition. If that ticket is accepted, any wording agreed here would need further revision.
4. **FHIR-52608** (Not Persuasive) attempted to restructure the actor model (renaming Placer to Initiator) — the workgroup rejected this, indicating the current Placer/Server/Filler/Responder/Patient actor model is settled.
5. **FHIR-46815 and FHIR-46814** (both Applied) consolidated repeated actor descriptions — the definition text agreed here will now be the single source of truth.

## Proposed Dispositions

### Disposition A: Persuasive — Accept the more descriptive definition (Option 2)
**Action:** Update the AU eRequesting Server Actor definition to: *"The AU eRequesting Server is a system that makes diagnostic service requests accessible to filler and patient record systems and can accept diagnostic service requests from placer systems."*

**Rationale:** This definition clearly communicates the Server Actor's role in the eRequesting ecosystem by naming the interacting system types. With FHIR-46815/FHIR-46814 having consolidated actor definitions to a single location, precision matters more than brevity. The reporter and community have expressed that vague definitions are unhelpful to implementers. This wording distinguishes the Server from the Responder Actor introduced by FHIR-47141.

### Disposition B: Persuasive with Modification — Accept Option 2 but broaden scope
**Action:** Update the definition to: *"The AU eRequesting Server is a system that makes service requests accessible to filler and patient record systems and can accept service requests from placer systems."* (removing "diagnostic" to align with FHIR-52180's direction).

**Rationale:** Given that FHIR-52180 (still Submitted) proposes removing the diagnostic qualifier from the Server Actor definition, and the IG title itself is scope-agnostic ("AU eRequesting"), it may be more efficient to address both concerns simultaneously. This avoids a second definition change if FHIR-52180 is later accepted.

### Disposition C: Not Persuasive — Retain the minimal definition (Option 1)
**Action:** Update the definition to: *"The AU eRequesting Server is a system that makes diagnostic service requests accessible and can accept diagnostic service requests."* (the minimal option from the ticket).

**Rationale:** This aligns with the existing pattern used across other HL7 actor definitions. Naming specific interacting systems (filler, placer, patient record) in the definition may over-constrain the actor and create maintenance burden if the actor model evolves. The specific interactions are already documented in the CapabilityStatement and conformance pages.

## Recommendation

**Disposition A: Persuasive — Accept the more descriptive definition (Option 2).**

The more descriptive definition directly addresses the problem: after FHIR-47141, the Server Actor's role must be clearly distinguished from the Responder Actor. Naming the interacting systems (filler, patient record, placer) provides implementers with actionable clarity about what the Server Actor does and for whom. The workgroup has already committed to a single definition location (FHIR-46815), making precision important. The "diagnostic" qualifier should be retained for now since FHIR-52180 has not yet been triaged, and changing scope prematurely could create confusion. FHIR-48813 (the companion CapabilityStatement ticket) should be resolved consistently with whatever wording is agreed here.
