## Summary

This ticket requests that a package dependency on the NCTS (National Clinical Terminology Service) profiles be added to the AU Base IG once NCTS publishes a distributable FHIR package containing those profiles. This is a follow-on from two previously applied change requests:

- **[FHIR-47148](https://jira.hl7.org/browse/FHIR-47148)** (Applied, Persuasive with Modification) — Changed AU Base CodeSystem resources to claim conformance to the NCTS Complete Code System profile instead of the HL7 International `shareablecodesystem` profile.
- **[FHIR-47149](https://jira.hl7.org/browse/FHIR-47149)** (Applied, Persuasive with Modification) — Changed AU Base ValueSet resources to claim conformance to the NCTS Composed Value Set profile instead of the HL7 International `shareablevalueset` profile.

While those tickets changed the conformance claims, the IG Publisher currently **cannot validate** CodeSystem and ValueSet resources against the NCTS profiles because it cannot resolve them — NCTS has not yet published a FHIR package containing those profile definitions. This ticket requests adding the NCTS package as a dependency to AU Base once it becomes available, enabling proper validation.

## Details

### Background
FHIR-47148 and FHIR-47149 moved AU Base CodeSystem and ValueSet resources from claiming conformance to the international `shareablecodesystem` / `shareablevalueset` profiles to the Australian NCTS profiles:
- `https://healthterminologies.gov.au/fhir/StructureDefinition/complete-code-system-4|4.0.0`
- `https://healthterminologies.gov.au/fhir/StructureDefinition/composed-value-set-4|4.1.0`

### Current Problem
The IG Publisher cannot see or validate against these NCTS profiles because they are not available in a published FHIR package. Until NCTS publishes such a package, the validation gap remains.

### External Dependency
A GitHub issue has been logged for NCTS to action the publication of the profile package: [aehrc/fhirpit#120](https://github.com/aehrc/fhirpit/issues/120).

### Comment Discussion
- A commenter noted this should be classified as a **change request** (not a technical correction), and flagged that the workgroup needs to **discuss whether AU Base should be dependent on NCTS** at all — this is a governance/architectural question, not purely technical.

## Proposed Dispositions

### Disposition A: Persuasive — Accept and add NCTS dependency when package is available
**Rationale:** The predecessor tickets (FHIR-47148 and FHIR-47149) have already been applied, meaning AU Base CodeSystem and ValueSet resources now claim conformance to NCTS profiles. Without the dependency, the IG Publisher cannot validate those conformance claims, leaving a validation gap. Adding the dependency is the logical completion of the changes already approved. Once NCTS publishes the package (tracked via aehrc/fhirpit#120), AU Base should add it as a dependency.

**Action:** Accept the change request. Add the NCTS profile package as a dependency to AU Base once NCTS publishes it. Monitor aehrc/fhirpit#120 for availability.

### Disposition B: Persuasive with Modification — Accept with conditions on dependency management
**Rationale:** While the dependency is necessary for validation, introducing a dependency on an external (non-HL7) package raises governance and maintenance concerns. The workgroup should accept the change but establish conditions: (1) pin the NCTS package to a specific version, (2) document the dependency relationship and its rationale, and (3) define a review process for updating the NCTS dependency version in future AU Base releases. This addresses the concern raised in comments about whether AU Base should be dependent on NCTS.

**Action:** Accept the change request with modifications: require version pinning of the NCTS package dependency, add documentation explaining the dependency rationale, and establish a governance process for NCTS dependency version updates in AU Base.

### Disposition C: Not Persuasive — Decline and reconsider the NCTS profile approach
**Rationale:** Adding a dependency on an externally managed NCTS package introduces a coupling that may complicate AU Base's release cycle and creates a dependency on a non-HL7 organization. The comment on the ticket raises a valid point: the workgroup should first resolve whether AU Base should depend on NCTS at all. If the governance question is resolved against the dependency, the conformance claims from FHIR-47148/FHIR-47149 may need to be revisited. The policy and conventions working party (being formed per the Zulip thread) may be the appropriate body to decide this.

**Action:** Decline the change request pending a broader governance decision on AU Base's relationship to NCTS profiles. Defer to the HL7 AU FHIR IG policy and conventions working party to establish a position on external package dependencies before proceeding.

## Recommendation

**Disposition B: Persuasive with Modification.** The predecessor tickets have already been applied and AU Base resources already claim conformance to NCTS profiles, so the validation gap needs to be closed. However, the governance concern raised in the comments is valid — adding an external dependency deserves deliberate conditions. Accepting with version pinning, documentation, and a review process provides a balanced path that completes the validation chain while establishing responsible dependency management practices. This approach also aligns with the broader policy and conventions work being initiated by the working party.
