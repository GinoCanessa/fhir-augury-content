## Description

This ticket requests the addition of a `ServiceRequest.focus` extension to the AU Diagnostic Request profile (`au-diagnosticrequest`). Originally raised as ballot issue FHIRIG-227 for AU Base 4.1.0, it proposes pre-adoption of the R5 `ServiceRequest.focus` element via an extension.

**Use case:** Support imaging requests to determine tumour margins of a surgical pathology specimen. This is a common clinical scenario, particularly in breast-conserving surgeries for removal of breast tumours, where a separate imaging study (e.g., specimen radiography) is performed on the excised tissue rather than on the patient directly.

The ticket proposes that the AU Diagnostic Request profile explicitly include guidance on using the R5 pre-adoption `ServiceRequest.focus` extension, so that the focus/target of the service request can be distinguished from the subject (the patient).

## Analysis

### Key Context

1. **`ServiceRequest.focus` is now a core R5/R6 element.** FHIR-37266 added it to the base specification and it has been published. This means pre-adoption via extension is a well-established pattern with a clear path forward.

2. **The ticket has not progressed to a specific proposal.** Despite being raised in early 2023 (via the GitHub issue) and transferred to Jira in mid-2024, no concrete StructureDefinition or IG change has been drafted.

3. **Transfer to eRequesting TDG agreed.** The most recent comments (April–May 2025) indicate the FHIR WG co-chairs and the Agency have agreed this should be handled by the eRequesting Technical Design Group rather than AU Base.

4. **The specification field already reflects eRequesting.** The ticket's specification is listed as "AU eRequesting (FHIR)", consistent with the agreed transfer.

5. **The use case is clinically valid.** Specimen-focused imaging in surgical pathology (e.g., tumour margin assessment in breast-conserving surgery) is well-documented and widely practiced.

6. **Core FHIR evolution supports this direction.** FHIR-54620 (accepted) will mark `ServiceRequest.focus` as a modifier, reinforcing it as a significant semantic element. The element is mature and stable in core.

## Proposed Dispositions

### Disposition A: Transfer to eRequesting TDG (Persuasive with Modification)

**Resolution:** Accept the request but redirect it to the AU eRequesting IG scope as agreed by the Agency and FHIR WG co-chairs.

**Rationale:** The use case is valid and `ServiceRequest.focus` is a stable R5 core element suitable for pre-adoption. However, this belongs in the eRequesting IG where diagnostic request workflows are being actively designed, not in AU Base. The eRequesting TDG should evaluate the requirement in the context of their broader diagnostic request profiling work and add guidance on the R5 pre-adoption extension (`http://hl7.org/fhir/5.0/StructureDefinition/extension-ServiceRequest.focus`) to their AU eRequesting Diagnostic Request profile as appropriate.

**Action:** Change specification to AU eRequesting, assign to eRequesting TDG backlog, update status to reflect the transfer.

### Disposition B: Defer — Consider for Future Use

**Resolution:** Defer the request until AU eRequesting IG reaches a maturity level where specimen-focused imaging workflows are in scope.

**Rationale:** While the use case is valid, no specific proposal or profile change has been drafted despite the ticket being open since 2023. The eRequesting IG may not yet be at a stage where this extension adds immediate value. Deferring allows the TDG to prioritize core workflow requirements first and revisit this when their diagnostic request profiles are more mature.

**Action:** Set status to "Consider for Future Use" and tag for eRequesting TDG follow-up.

### Disposition C: Withdraw — Not Persuasive

**Resolution:** Close the ticket as the request has not been substantiated with a specific proposal.

**Rationale:** The ticket lacks a concrete StructureDefinition change or IG diff. The `ServiceRequest.focus` extension is already available as a standard R5 pre-adoption extension and implementers can use it without explicit profiling in AU specifications. Adding it to a profile without demonstrated implementation need may add unnecessary complexity.

**Action:** Close as withdrawn / not persuasive.

## Recommendation

**Disposition A: Transfer to eRequesting TDG (Persuasive with Modification)** is recommended.

This aligns with the explicit agreement between the FHIR WG co-chairs and the Agency (May 2025 comments). The use case is clinically valid and well-supported by the FHIR core specification evolution. The eRequesting TDG is the appropriate home for this work, as they are actively defining diagnostic request workflows and can evaluate this extension requirement in context. The ticket's specification field already reflects this target. A formal transfer ensures the requirement is tracked and not lost.
