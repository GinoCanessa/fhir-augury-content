## Description

The submitter requests that the `doNotPerform` element be disabled (constrained out) from the AU eRequesting ServiceRequest profiles. The rationale is that `doNotPerform` is not relevant to AU eRequesting use cases.

`ServiceRequest.doNotPerform` is a boolean modifier element in the base FHIR ServiceRequest resource that, when set to `true`, indicates that the service/procedure should **not** be performed. The submitter's position is that this concept does not apply within the Australian eRequesting context, where service requests represent positive orders for diagnostic services.

## Analysis

### Context
- `ServiceRequest.doNotPerform` is a **modifier element** (isModifier = true) in the base FHIR R4/R5 ServiceRequest resource. When `true`, it negates the meaning of the request â€” turning "perform this test" into "do not perform this test."
- AU eRequesting focuses on positive diagnostic service orders (pathology and imaging requests) within the Australian healthcare context. The submitter argues that negative orders ("do not perform this test") are not part of this workflow.
- The related ticket FHIR-52405 (now Applied) addressed the ambiguity of what `doNotPerform = false` means, suggesting the WG has already recognized this element creates confusion in the eRequesting context.

### Considerations
1. **Constraining out a modifier element** is a well-established profiling practice when the element is not meaningful for the profile's use case. It simplifies implementation and prevents misuse.
2. **However**, completely disabling the element is a stronger action than setting a fixed value of `false`. The WG should consider whether to: (a) constrain `doNotPerform` to 0..0 (prohibited), or (b) fix the value to `false` (must be false if present).
3. **FHIR-52405's resolution** (Persuasive with Modification) suggests the WG already took action on `doNotPerform` semantics â€” the disposition of this ticket should be consistent with that resolution.
4. **Broader ecosystem impact**: If other Australian IGs or downstream consumers ever need to express "do not perform" scenarios, constraining this out in eRequesting would not prevent them from using their own profiles.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Constrain `ServiceRequest.doNotPerform` to `0..0` (prohibited) in AU eRequesting ServiceRequest profiles.

**Rationale:** The AU eRequesting use case is exclusively about positive diagnostic orders. Negative orders are out of scope. Prohibiting `doNotPerform` simplifies the profile, eliminates a potential source of confusion (as evidenced by FHIR-52405), and aligns with the profiling principle that elements irrelevant to the use case should be constrained out. This is the cleanest approach and sends a clear signal to implementers.

### Disposition B: Persuasive with Modification
**Action:** Instead of fully prohibiting the element, fix `ServiceRequest.doNotPerform` to a value of `false` (i.e., constrain cardinality to `0..1` with a fixed value of `false`), effectively communicating that all eRequesting service requests are positive orders.

**Rationale:** This achieves the submitter's goal of making `doNotPerform` irrelevant while preserving the element for explicit documentation purposes. Systems that populate the field with `false` would still be conformant. This is a more conservative approach that may be preferable given that FHIR-52405 already addressed the semantics when `false`. However, it does add marginal complexity compared to `0..0`.

### Disposition C: Not Persuasive
**Action:** No change. Leave `doNotPerform` unconstrained in AU eRequesting profiles.

**Rationale:** While `doNotPerform` may not be needed for current eRequesting use cases, constraining it out could limit future flexibility if negative order scenarios emerge (e.g., clinical decision support recommending against a test). The base FHIR resource already handles semantics, and implementation guides can provide guidance without structural constraints. The WG may prefer to address confusion through documentation (as partly done via FHIR-52405) rather than structural prohibition.

## Recommendation

**Disposition A: Persuasive** â€” Constrain `doNotPerform` to `0..0`.

The AU eRequesting specification is focused on positive diagnostic orders, and there is no identified use case for negative orders in this context. The WG has already engaged with the confusion this element causes (FHIR-52405), and prohibiting it entirely is the simplest resolution. If future Australian IGs need negative order capability, they can create separate profiles. This approach is consistent with the profiling principle of constraining elements that are not relevant to the use case, and eliminates ambiguity for implementers.
