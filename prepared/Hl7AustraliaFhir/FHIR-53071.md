## Description

The existing AU Base extension `closing-the-gap-registration` (`http://hl7.org.au/fhir/StructureDefinition/closing-the-gap-registration`) currently supports only the Patient resource in its usage context. This ticket requests extending the ability to capture Closing the Gap (CTG) Pharmaceutical Benefits Scheme registration status at the **Organization** level — specifically to record whether a practice or service is eligible to dispense under the CTG PBS program.

Two options are proposed:

- **Option 1:** Define a new, separate extension for Organization to record CTG program participation.
- **Option 2:** Extend the scope of the existing `closing-the-gap-registration` extension to include Organization in its context.

## Proposed Dispositions

### Disposition A: Persuasive (Option 2 — Extend Existing Extension)

**Resolution:** Extend the context of the existing `closing-the-gap-registration` extension to include Organization.

**Rationale:** The concept of "Closing the Gap registration" applies to both patients (registered for CTG benefits) and organizations (eligible to dispense under CTG). Reusing the existing extension avoids proliferation of near-identical extensions and keeps the CTG registration concept consolidated. The extension is a simple indicator; the resource context (Patient vs. Organization) naturally differentiates the two use cases. Update the extension description to document both contexts.

**Actions required:**
- Add `Organization` to the extension's `context` in the StructureDefinition.
- Update the extension description and usage notes to cover the organizational eligibility use case.
- Add examples demonstrating Organization usage.

### Disposition B: Persuasive with Modification (Option 1 — New Extension)

**Resolution:** Create a new, separate extension for Organization CTG program eligibility rather than broadening the existing patient-focused extension.

**Rationale:** The semantics are meaningfully different — patient CTG registration (an individual registered for benefits) vs. organizational CTG eligibility (a practice approved to dispense). A separate extension allows independent evolution, distinct documentation, and avoids ambiguity. The existing patient extension's definition ("indicates whether the patient is registered for the Closing the Gap program") would need rewording if Organisation is added, potentially creating confusion.

**Actions required:**
- Define a new extension (e.g., `closing-the-gap-eligibility`) with context limited to Organization.
- Document the distinction between patient registration and organizational eligibility.
- Add examples and usage notes.

### Disposition C: Not Persuasive

**Resolution:** Decline the change; the use case can be addressed through existing mechanisms.

**Rationale:** An organization's eligibility to dispense under CTG may be better represented through `HealthcareService.program` (which was introduced via FHIR-19357 and already lists CTG as an example program), or through Organization tags/identifiers specific to CTG registration. Adding CTG to the Organization via an extension may not be the most appropriate modeling approach when program-level participation is a service-level concern.

## Recommendation

**Disposition A: Persuasive (Option 2 — Extend Existing Extension)** is recommended.

The Zulip discussion shows community consensus that extending the existing extension is a reasonable path, and Heath Frankel explicitly offered this as one of two valid approaches. The extension is a straightforward boolean-like indicator, and broadening its context to include Organization is a minimal, non-breaking change. The resource type in which the extension appears (Patient vs. Organization) provides sufficient semantic differentiation without requiring a separate extension. This approach is simpler, avoids extension proliferation, and aligns with the reporter's stated need to record CTG eligibility at the practice level. The extension description should be updated to encompass both patient registration and organizational eligibility use cases.
