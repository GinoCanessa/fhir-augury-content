## Description Summary

The reporter identifies confusion around the `referencePolicy` element (value `resolves`) in both the AU Core Responder and AU Core Requester CapabilityStatement resources. Key points:

1. **Applicability concern:** The FHIR specification defines `referencePolicy` values in server-centric terms ("The server will attempt to..."), suggesting it is only meaningful for the AU Core Responder (server), not the AU Core Requester (client).
2. **Implementation gap:** Testing with Smile CDR (HAPI JPA Server) revealed it does not support resolving `Reference.identifier` to `Reference.reference`. Google Cloud FHIR Server also appears to lack this capability.
3. **Expectation too strong:** Even though conformance is only SHOULD, the reporter argues that `resolves` sets an overly ambitious expectation given major reference implementations do not support it.

**Recommendation from reporter:**
1. Remove `referencePolicy` from each resource in AU Core Requester CapabilityStatement.
2. Remove `referencePolicy` from each resource in AU Core Responder CapabilityStatement, or alternatively change the value to `logical`.

## Proposed Dispositions

### Disposition A: Persuasive — Remove referencePolicy from both CapabilityStatements

**Action:** Remove the `referencePolicy` element from all resources in both the AU Core Responder and AU Core Requester CapabilityStatements. AU Core would remain silent on reference resolution policy.

**Rationale:**
- Aligns with the resolution of FHIR-48446, where US Core (the originator of the `resolves` pattern) has itself decided to remove all `referencePolicy` guidance as contradictory and unnecessary.
- The `referencePolicy` element is server-centric by definition; its inclusion in the client (Requester) CapabilityStatement is semantically incorrect.
- Major FHIR server implementations (Smile CDR/HAPI, Google Cloud FHIR) do not support `Reference.identifier` resolution, making the SHOULD expectation unachievable in practice.
- AU Core's Create/Update interactions are only MAY, further reducing the relevance of reference resolution policy.
- Removes a source of implementer confusion without losing meaningful interoperability capability.

**Risk:** Low. Removing a SHOULD that is not implementable in practice has no practical interoperability impact.

### Disposition B: Persuasive with Modification — Remove from Requester, change to `logical` on Responder

**Action:** Remove `referencePolicy` from the AU Core Requester CapabilityStatement entirely. On the AU Core Responder CapabilityStatement, change the value from `resolves` to `logical` (indicating the server accepts logical references but does not commit to resolving them).

**Rationale:**
- Addresses the reporter's secondary recommendation.
- Acknowledges that some server-side reference handling guidance may be useful while lowering the bar to a more realistic expectation.
- `logical` simply means the server can store and return logical references, which is a much more achievable expectation.

**Risk:** Moderate. Retaining `referencePolicy` on the Responder may perpetuate confusion about what the element means and what implementations should test. The value `logical` may itself be poorly understood given FHIR-17624's findings about unclear definitions.

### Disposition C: Not Persuasive — Retain current referencePolicy

**Action:** Keep the current `referencePolicy` of `resolves` on both CapabilityStatements. Optionally add clarifying guidance.

**Rationale:**
- Maintains alignment with the original US Core and IPA approach.
- The conformance level is only SHOULD, giving implementations flexibility.
- Removing it could be seen as weakening interoperability expectations.

**Risk:** High. US Core itself has moved to remove this guidance (FHIR-48446, Persuasive). Retaining it in AU Core would create a divergence from US Core's direction. Additionally, the Zulip discussions received zero responses, suggesting the community lacks clarity on how to implement or test this requirement, making it an ineffective conformance expectation.

## Recommendation

**Disposition A: Persuasive — Remove referencePolicy from both CapabilityStatements.**

This is the strongest option because:
1. It directly mirrors the resolution already accepted by US Core (FHIR-48446), ensuring AU Core stays aligned with the broader FHIR community direction rather than retaining guidance the originating IG has itself abandoned.
2. The technical evidence is clear — major server implementations do not support `resolves`, and the element is semantically inappropriate for client CapabilityStatements.
3. It follows AU Core's own precedent (FHIR-46060) of removing conformance expectations that are unlikely to be implemented.
4. It eliminates implementer confusion without any practical interoperability loss.
