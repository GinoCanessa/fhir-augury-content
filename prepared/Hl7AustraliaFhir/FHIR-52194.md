## Description

The reporter (Grahame Grieve) raises a concern about the gender identity value set binding in AU Base. The value set contains codes for **male**, **female**, and **nonbinary** and is bound with **extensible** strength. The argument is that "extensible" means other codes cannot be used unless they do not overlap in meaning with the existing codes — but these three codes already cover the full logical space (male, female, and not-male-or-female). This creates ambiguity about when implementers may legitimately use additional codes. The reporter suggests two alternatives:

1. **Use different codes with clearer definitions** so that the scope of "extensible" is well-understood, or
2. **Change the binding strength to required**, thereby closing the value set.

## Analysis

### Core Issue

The fundamental question is whether the codes **male**, **female**, and **nonbinary** in the Gender Identity Response value set create a logically exhaustive partition when bound with extensible strength. The reporter argues they do: any gender identity is either male, female, or something else (covered by "nonbinary"). If that interpretation is correct, extensible is functionally equivalent to required, making the choice of extensible misleading.

### Counterargument

Brett Esler's comment provides the AU FHIR community's rationale: the value set is based on Australian national standards (ABS and AIHW METeOR 741825). The intent is that "nonbinary" does not serve as a catch-all for every identity outside male/female. Some individuals use terms that they do not consider equivalent to "nonbinary" (e.g., "Different term" in the METeOR standard). The extensible binding allows these cases to be represented via text or alternative code systems, aligning with the ABS approach of "[I/They] use a different term (please specify)."

### Precedent

- **FHIR-43718** (Applied) established the current extensible binding to the Gender Identity Response value set as part of AU Base's Gender Harmony adoption.
- **FHIR-39208** explored the same conceptual tension at the core FHIR level and was resolved with no change.
- **FHIR-35677** addressed similar binding-strength concerns in the US Core context.

## Proposed Dispositions

### Disposition A: Persuasive — Change binding to required

**Action:** Change the binding strength from extensible to required for the Gender Identity Response value set in AU Base profiles.

**Rationale:** The reporter's argument has logical merit — if the three codes are interpreted as exhaustive, extensible is misleading. A required binding eliminates ambiguity for implementers and aligns with a strict reading of FHIR binding semantics. Systems that need more granularity would use the Gender Identity extension's text sub-element or the Recorded Sex or Gender (RSG) extension for coded alternatives.

**Risk:** This would break alignment with the ABS/AIHW standards that envision a "Different term" option. It may restrict implementers who need to send locally-defined gender identity codes.

### Disposition B: Persuasive with Modification — Improve value set coverage and retain extensible

**Action:** Keep the extensible binding but request that the Gender Identity Response value set be expanded (via NCTS) to include concepts such as "Different term" or "Prefer not to answer" (aligned with METeOR 741842), making the extensible semantics clearer by demonstrating that the code list is not meant to be exhaustive.

**Rationale:** This addresses the reporter's concern that the three-code value set appears exhaustive while preserving extensibility. Adding explicit "Different term" and "Prefer not to answer" concepts signals to implementers that the value set is intentionally open-ended, consistent with Australian national standards. FHIR-50111 commentary already identified METeOR 741842 as including these additional concepts.

**Risk:** Requires coordination with NCTS for value set updates. The "Different term" concept may be unusual in FHIR value sets (typically handled by the extensible mechanism itself).

### Disposition C: Not Persuasive — Retain current extensible binding as-is

**Action:** No change. The existing extensible binding with three codes (male, female, nonbinary) is retained.

**Rationale:** The extensible binding is working as intended. As explained in the comments, "nonbinary" is not semantically a catch-all for every non-male/non-female identity. The Australian standards explicitly acknowledge that some people use different terms. The extensible mechanism correctly handles this — implementers can use text or alternative code systems when none of the three codes is appropriate. FHIR-39208 was resolved with no change at the core FHIR level for the same conceptual question. The current approach was deliberately established in FHIR-43718.

**Risk:** The reporter's concern about implementer confusion remains unaddressed. Some implementers may incorrectly interpret the three codes as exhaustive and struggle to understand when extensible allows alternatives.

---

## Recommendation

**Disposition B: Persuasive with Modification** — Expand the value set to include "Different term" and/or "Prefer not to answer" while retaining the extensible binding.

This disposition best balances the competing concerns. The reporter raises a valid usability issue: a three-code value set that appears logically complete creates implementer confusion when combined with extensible binding. However, changing to required (Disposition A) would break alignment with Australian national standards and the intent of FHIR-43718. Doing nothing (Disposition C) leaves the ambiguity in place. Expanding the value set to include concepts like "Different term" and "Prefer not to answer" — already present in the aligned METeOR standard 741842 — makes the extensible semantics self-evident and resolves the confusion without constraining implementers.
