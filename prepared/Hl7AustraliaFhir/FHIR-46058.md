# FHIR-46058: AU Core Responder CapabilityStatement Support Patch Formats conformance specified in narrative

## Summary

**Status:** Triaged | **Type:** Change Request | **Priority:** Medium | **Resolution:** Unresolved
**Reporter:** Heath Frankel | **Assignee:** Unassigned
**Specification:** AU Core (FHIR) | **Work Group:** HL7 Australia FHIR
**Created:** 2024-05-31 | **Updated:** 2025-04-03
**URL:** https://jira.hl7.org/browse/FHIR-46058

During a Conformance Friday session on 31 May 2024, reviewers found that the AU Core Responder CapabilityStatement's structured "Support Patch Formats" section (specifying SHOULD-level conformance) was not reflected in the narrative, unlike the Support Formats requirements for JSON and XML. Discussion also questioned whether SHOULD was the correct conformance level, with the group leaning toward MAY given that PATCH is complex and not commonly implemented.

## Details

### Core Issue
The ticket raises two distinct concerns:

1. **Narrative/structured mismatch:** The structured CapabilityStatement included a "Support Patch Formats" conformance requirement (SHOULD), but this was not described in the handcrafted narrative section, creating an inconsistency between structured and human-readable content.

2. **Conformance level question:** Whether SHOULD is appropriate for patch format support, given the community consensus that PATCH operations are complex and unlikely to be widely implemented.

### Comment Analysis

- **JIRAUSER22800 (2024-05-31):** Clarified that the conformance requirement is about the *format* (e.g., JSON Patch) to use when PATCH is implemented, not about whether to implement PATCH itself. Suggested the format requirement should remain SHOULD, or be removed if patch interaction is removed from the CapabilityStatement. Noted a parallel change would be needed for AU Core Requester.

- **JIRAUSER22800 (2024-10-28):** Flagged related work on a new "Resource Produce Actor" with associated CapabilityStatements. Proposed either (a) downgrading patch conformance to MAY in existing Requester/Responder CapabilityStatements and moving SHOULD patch format requirements to new CapabilityStatements, or (b) leaving write capabilities as SHOULD in existing CapabilityStatements with no new Actors.

- **JIRAUSER25986 (2025-04-03):** Noted that the IG Publisher now supports proper narrative generation from structured content, meaning the original narrative mismatch problem is resolved by tooling improvements. The narrative is now auto-generated rather than handcrafted.

### Relationship to Other Tickets
- **FHIR-46060** (Applied — Persuasive with Modification): Proposed removing MAY conformance requirements for patch, vread, and instance-history interactions from AU Core Responder and Requester CapabilityStatements. This was accepted and applied, which directly affects the scope of patch-related conformance in FHIR-46058.
- **FHIR-48720** (Resolved — Retracted): Proposed removing all MAY interaction conformance requirements not tied to SHALL/SHOULD testable aspects, including patch. Was retracted.
- **FHIR-45081** (Applied — Persuasive with Modification): Added informative guidance on AU Core FHIR RESTful interactions, related to overall CapabilityStatement clarity.
- **FHIR-53614** (Resolved — Persuasive): Addresses removing narrative-only resource-level conformance requirements from CapabilityStatements in a different specification (Da Vinci CRD), but reflects a similar pattern of narrative/structured alignment concerns.

## Proposed Dispositions

### Disposition A: Accept — Close as Resolved by Tooling and Related Changes
**Rationale:** Both original concerns have been addressed:
1. The narrative/structured mismatch is resolved because the IG Publisher now auto-generates narratives from structured content (per the 2025-04-03 comment), eliminating the handcrafted narrative problem.
2. The patch interaction conformance question was addressed by **FHIR-46060** (Applied), which removed MAY conformance requirements for patch, vread, and instance-history from the CapabilityStatements.

**Action:** Resolve as "Already Applied" or "Not Persuasive — Already Resolved." Note in the resolution that the narrative generation is now handled by tooling and the patch interaction conformance was addressed by FHIR-46060.

### Disposition B: Accept with Modification — Verify Patch Format Conformance Alignment
**Rationale:** While the narrative rendering and patch interaction removal have been addressed, there may be residual "Support Patch Formats" conformance requirements in the structured CapabilityStatement that should be reviewed for consistency. If FHIR-46060 removed the patch *interaction* but the patch *format* conformance (SHOULD level) remains, this should be explicitly addressed — either retained at SHOULD (as the original commenter suggested, since it governs *how* to patch if implemented) or removed to align with the removal of patch interactions.

**Action:** Resolve as "Persuasive with Modification." Verify whether a "Support Patch Formats" requirement persists in the structured CapabilityStatements. If so, decide whether to retain it (as guidance for implementers who do support patch) or remove it (to fully align with the FHIR-46060 removal of patch interactions). Apply any needed change to both Responder and Requester CapabilityStatements.

### Disposition C: Decline — Not Persuasive
**Rationale:** The original narrative mismatch was a tooling limitation that has since been resolved by IG Publisher improvements. The conformance level question was addressed separately in FHIR-46060. This ticket does not introduce a new actionable change beyond what has already been applied.

**Action:** Resolve as "Not Persuasive." Document that the concerns raised have been independently resolved through tooling improvements (IG Publisher narrative auto-generation) and the applied resolution of FHIR-46060.

## Recommendation

**Disposition A: Accept — Close as Resolved by Tooling and Related Changes**

Both concerns in this ticket have been substantively addressed. The narrative mismatch is no longer possible since the IG Publisher now auto-generates narratives from structured content, and the patch interaction conformance was resolved by FHIR-46060 (Applied). However, it would be prudent to verify during resolution that no orphaned "Support Patch Formats" structured requirement remains. If it does, Disposition B should be preferred to clean up the residual requirement. Otherwise, Disposition A provides a clean close with clear justification.
