## Description Summary

The ticket raises two related issues regarding Australian PBS and MBS code systems in AU Base:

1. **PBS CodeSystem (partially addressed):** The current PBS CodeSystem published in AU Base explicitly references OID `1.2.36.1.2001.1005.22`, which contains codes *modified* by Services Australia (typically padded with leading zeros for consistent character length). The reporter argues this is not easily discoverable and impractical for systems that use unpadded codes. The suggestion is to detach the OID identifier from the PBS CodeSystem and clarify that either padded or unpadded codes may be used.

2. **MBS CodeSystem (remaining scope):** AU Base has no published MBS CodeSystem or ValueSet. Existing MBS-related resources (e.g., in ADHA's ExplanationOfBenefit profile) reference OID `1.2.36.1.2001.1005.21`, which also refers to modified/padded Services Australia codes. The suggestion is for HL7 AU to publish a new MBS CodeSystem and ValueSet that does *not* reference the padded system, with clarification that either padded or unpadded codes may be used.

**Note (per comment from brettesler, 2025-03-03):** The PBS portion of this ticket has been split into [FHIR-49933](https://jira.hl7.org/browse/FHIR-49933) (status: Applied, resolution: Persuasive). **This ticket now only requires resolution of the MBS-related material.**

## Proposed Dispositions

### Disposition A: Persuasive â€” Create MBS CodeSystem and ValueSet in AU Base
**Action:** Create a new metadata-only MBS CodeSystem resource (similar to the existing PBS one) and a corresponding ValueSet in AU Base, using a canonical URL under HL7 AU's control (e.g., `http://terminology.hl7.org.au/CodeSystem/mbs-item`). The CodeSystem should explicitly state that codes are unpadded MBS item numbers as published on MBS Online. Include guidance text clarifying the relationship to the Services Australia modified OID (`1.2.36.1.2001.1005.21`) and note that padded codes belong to a different system.

**Rationale:** This directly addresses the remaining scope of the ticket. The community discussion shows clear consensus that unpadded MBS codes need a formal CodeSystem. The PBS portion was already resolved via FHIR-49933. The approach mirrors the existing PBS artefact pattern in AU Base and is consistent with Brett Esler's stated willingness to create MBS placeholder resources.

### Disposition B: Persuasive with Modification â€” Create MBS CodeSystem with alternateCode guidance
**Action:** Create the MBS CodeSystem and ValueSet as in Disposition A, but additionally include implementation guidance about the `alternateCode` extension (or equivalent mapping mechanism) to document the relationship between unpadded MBS codes and their padded equivalents under `1.2.36.1.2001.1005.21`. This would help systems that need to interoperate with Services Australia's modified code representations.

**Rationale:** Michael Lawley raised the `alternateCode` property in the Zulip discussion. While this adds complexity, it would address interoperability concerns for systems that must exchange data with Services Australia's modified code systems while primarily using unpadded codes.

### Disposition C: Deferred â€” Wait for THO migration decision
**Action:** Defer the MBS CodeSystem creation until the broader question of where metadata-only CodeSystems should be published (AU Base vs. THO) is resolved per FHIR-47173.

**Rationale:** FHIR-47173 proposes retiring metadata-only CodeSystems from AU Base in favour of THO publication. Creating a new metadata-only CodeSystem in AU Base now may create additional migration work later. However, FHIR-47173 is itself deferred with no clear timeline, and the community has an immediate practical need.

## Recommendation

**Disposition A: Persuasive â€” Create MBS CodeSystem and ValueSet in AU Base.**

This is the most pragmatic resolution. The PBS portion is already resolved (FHIR-49933 applied). Community discussion demonstrates clear consensus and practical need for a formal MBS CodeSystem. The FHIRWG has already demonstrated willingness to act (the PBS OID was removed per the 2025-01-29 meeting decision). Waiting for THO migration (Disposition C) blocks a real implementation need on an indefinitely deferred ticket. The alternateCode guidance (Disposition B) can be added as a follow-up if needed, without delaying the core deliverable. Polls conducted in Zulip confirm implementers are actively using unpadded MBS codes and need a formal system URI.
