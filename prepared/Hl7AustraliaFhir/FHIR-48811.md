## Description

Following the application of [FHIR-47141](https://jira.hl7.org/browse/FHIR-47141) (Provide eRequesting Responder Actor and Capability Statement), the AU eRequester Server CapabilityStatement can now be implemented by a Placer system that provides Read and Search interactions. This creates a mismatch in the AU eRequesting Server Actor obligations on the AU eRequesting ServiceRequest profile:

**Problem 1 — Obligations that are too strict for Placer-as-Server:** Elements such as `identifier`, `priority`, `encounter`, `reasonCode`, `insurance`, `bodySite`, and `note` currently have the AU eRequesting Server obligation set to **SHALL:able-to-populate**. Since a Placer system implementing the Server CapabilityStatement may not have known values for these elements, the obligation should be changed to **SHALL:populate-if-known** to align with the AU eRequesting Placer Actor obligations.

**Problem 2 — Obligations that are too weak for mandatory elements:** Elements such as `requisition`, `status`, `intent`, `category`, `code`, `subject`, `authoredOn`, and `requester` should have the AU eRequesting Server obligations changed from **SHALL:able-to-populate** to **SHALL:populate** to align with AU eRequesting Placer Actor obligations, since these are elements the Placer will always have.

## Analysis

This ticket is a logical consequence of FHIR-47141, which expanded the AU eRequesting Server CapabilityStatement to allow Placer systems to implement it with Read and Search interactions. The original Server obligations on ServiceRequest elements were written assuming only a centralized server (like a broker/intermediary) would implement them — a system that could be expected to store and serve back all elements it received. With Placer systems now also acting as Servers, the obligations must reflect that a Placer may not have values for optional elements (like `bodySite` or `insurance`) but will always have values for core elements (like `status`, `intent`, `code`, `subject`).

The proposed changes are consistent with:
- The default obligation conventions established in FHIR-51637 (SHALL:populate for mandatory elements, SHALL:populate-if-known for optional)
- The corrections already applied in FHIR-53090 (fixing obligation mismatches against TDG defaults)
- The Placer Actor obligations already in place

The companion tickets FHIR-48812 and FHIR-48813 address the Server Actor and CapabilityStatement *definitions* respectively, and should ideally be reviewed together with this ticket as a coherent set.

## Proposed Dispositions

### Disposition A: Persuasive
**Accept as-is.** Change the AU eRequesting Server Actor obligations on the AU eRequesting ServiceRequest profile as described:
- Change `identifier`, `priority`, `encounter`, `reasonCode`, `insurance`, `bodySite`, and `note` from **SHALL:able-to-populate** to **SHALL:populate-if-known**.
- Change `requisition`, `status`, `intent`, `category`, `code`, `subject`, `authoredOn`, and `requester` from **SHALL:able-to-populate** to **SHALL:populate**.

This directly aligns Server obligations with Placer obligations for the elements listed, which is appropriate now that a Placer system can implement the Server CapabilityStatement.

### Disposition B: Persuasive with Modification
**Accept with scoping to the agreed default obligation framework.** Apply the proposed obligation changes but verify each element against the default obligation rules established in FHIR-51637 and the TDG-agreed conventions (i.e., SHALL:populate for Must Support elements with min cardinality ≥ 1, SHALL:populate-if-known for Must Support elements with min cardinality = 0) rather than mechanically copying the Placer obligations. This ensures the changes are principled and consistent with the broader obligation framework, not just the specific elements listed. Additionally, coordinate with companion tickets FHIR-48812 and FHIR-48813 to ensure the Server Actor definition and CapabilityStatement description are updated coherently.

### Disposition C: Not Persuasive
**Reject.** Maintain the existing SHALL:able-to-populate obligations for the AU eRequesting Server actor on the grounds that the Server role (regardless of whether implemented by a Placer or intermediary) should demonstrate the *capability* to populate all Must Support elements, even if a particular instance doesn't have a known value. This interpretation treats "able-to-populate" as a capability declaration rather than a data availability statement, and keeps the Server obligations independent of the Placer obligations.

## Recommendation

**Disposition B: Persuasive with Modification.** The core request is well-founded — aligning Server obligations with what a Placer system can realistically fulfill is necessary after FHIR-47141. However, applying changes through the lens of the agreed default obligation framework (FHIR-51637) rather than purely mirroring Placer obligations provides a more principled and maintainable approach. This also avoids any edge cases where an element's cardinality might not match the assumed obligation pattern. Coordinating with the companion tickets (FHIR-48812 and FHIR-48813) will ensure a coherent outcome across the Server actor definition, CapabilityStatement, and profile obligations.
