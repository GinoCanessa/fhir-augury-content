## Description

This ticket requests that the Diagnosis Related Group (DRG) be promoted from a `supportingInfo` element to a **backbone element** on the Claim, ClaimResponse, and ExplanationOfBenefit resources.

A DRG is a payment category used to classify patients—especially Medicare patients—for the purpose of reimbursing hospitals with a fixed fee per case category, regardless of actual costs incurred. DRGs are used on all inpatient facility claims.

- **Existing Wording:** The IG defines the Diagnosis Related Group (DRG) as `supportingInfo`.
- **Proposed Wording:** Define the Diagnosis Related Group (DRG) as a backbone element.

Originally submitted for the CARIN Blue Button IG, this is a FHIR Core request (per Corey Spears, 2025-10-30).

## Key Discussion Points (from Comments)

1. **Misplacement of DRG under `diagnosis.packageCode`** (Corey Spears, 2023-09-06): In R4, `ExplanationOfBenefit.diagnosis.packageCode` mentions DRG, but this is structurally incorrect. DRG is not a collection of diagnosis codes—it depends on multiple factors including procedure codes, diagnosis codes, age, gender, and discharge status. It should not be placed under `EoB.diagnosis`.

2. **History:** Initially resolved as "Considered for Future Use" in 2020 (vote 18-0-5, Mark Roberts/Pat Taylor), then re-opened post-STU2 publication (Lynn Laakso, 2022-11-28).

3. **Scope clarification** (Corey Spears, 2025-10-30): Confirmed this is a FHIR Core request, not just a CARIN BB IG request.

## Analysis

### Core Issue
The fundamental problem is that DRG is currently modeled in R4 under `ExplanationOfBenefit.diagnosis.packageCode`, which is semantically incorrect. A DRG is not a package of diagnoses—it is a complex classification derived from multiple factors (principal/secondary diagnoses, procedures, comorbidities, patient age/sex, and discharge status). Placing it under `diagnosis` implies a relationship that does not exist.

### Evolution of the DRG Modeling Discussion
1. **2016 (FHIR-11391):** `packageCode` introduced as a placeholder term.
2. **2020 (FHIR-26712):** This ticket requests DRG as a backbone element, originally via CARIN BB.
3. **2020 (FHIR-29688):** A more specific request to add `diagnosisRelatedGroup` at the root of Claim/ClaimResponse/EOB — **approved as Persuasive**.
4. **2021 (FHIR-34666):** PCT IG takes the interim approach of mapping DRG to `supportingInfo` while awaiting the core fix.

### Key Consideration
**FHIR-29688 has already been resolved as Persuasive**, adding `diagnosisRelatedGroup 0..1 CodeableConcept` at the root level of the financial resources and removing `diagnosis.packageCode`. This substantially addresses the concern raised in FHIR-26712, though the approaches differ slightly:
- **FHIR-26712** requests a **backbone element** (complex structure with sub-elements).
- **FHIR-29688** adds a **simple CodeableConcept** at the root level.

The workgroup should consider whether FHIR-29688's resolution (root-level CodeableConcept) fully satisfies the intent of this ticket, or whether additional backbone structure (e.g., DRG type, DRG version, weighting factor) is needed.

## Proposed Dispositions

### Disposition A: Persuasive — Implement DRG as Backbone Element
**Action:** Create a new backbone element (e.g., `diagnosisRelatedGroup`) on Claim, ClaimResponse, and ExplanationOfBenefit that contains sub-elements such as `type` (CodeableConcept), `code` (CodeableConcept), `version` (string), and optionally `weight` (decimal).

**Rationale:** A backbone element provides richer structure than a simple CodeableConcept and better supports the complexity of DRG classification (multiple DRG systems like MS-DRG, AP-DRG, APR-DRG each with versions and weighting factors). This goes beyond FHIR-29688's resolution.

**Risk:** May be seen as over-engineering if implementers only need the DRG code. Conflicts with or supersedes the already-applied FHIR-29688 resolution.

### Disposition B: Duplicate/Subset of FHIR-29688 — Close as Already Addressed
**Action:** Close this ticket as a duplicate or subset of FHIR-29688, which was resolved as Persuasive and added `diagnosisRelatedGroup 0..1 CodeableConcept` at the root of the financial resources.

**Rationale:** FHIR-29688 addresses the core concern—removing DRG from `diagnosis.packageCode` and placing it correctly at the resource root. The simple CodeableConcept approach is consistent with typical FHIR modeling and allows DRG type/version to be conveyed via the coding system and version fields within CodeableConcept. No additional backbone structure is needed.

**Risk:** If implementers later need structured sub-elements (e.g., explicit weight or severity), they would need a new change request.

### Disposition C: Persuasive with Modification — Acknowledge FHIR-29688 and Extend if Needed
**Action:** Acknowledge that FHIR-29688 addresses the primary concern (correct placement of DRG). If the workgroup identifies specific sub-elements beyond CodeableConcept that are needed (e.g., weighting factor, severity level), file a new focused ticket for those additions. Otherwise, close as addressed by FHIR-29688.

**Rationale:** This pragmatic approach validates the submitter's concern while recognizing the existing resolution. It defers additional complexity only if real-world implementation demonstrates the need for backbone structure beyond what CodeableConcept provides.

**Risk:** Defers potential structural improvements, which may be harder to introduce later.

---

## Recommendation

**Disposition B: Duplicate/Subset of FHIR-29688 — Close as Already Addressed**

FHIR-29688 was resolved as Persuasive and directly addresses the core issue raised by FHIR-26712: removing DRG from the incorrect `diagnosis.packageCode` location and adding it as a properly named `diagnosisRelatedGroup` element at the root of the financial resources. The CodeableConcept type can accommodate DRG code, system, version, and display name through standard coding fields. No evidence from the discussion or related tickets suggests that additional backbone sub-elements are required. The workgroup should close this ticket with a reference to FHIR-29688 as the implementing resolution.
