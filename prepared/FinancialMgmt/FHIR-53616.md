# FHIR-53616: Add more details to the Bulk Provider Request for Data showing the Asynchronous Response Pattern

## Summary

**Reporter:** Linda Michaelsen | **Status:** Triaged | **Type:** Technical Correction | **Priority:** Medium
**Specification:** US Da Vinci PDex (FHIR) | **Work Group:** Financial Mgmt
**Labels:** PDex-Vote-1, Pre-Applied, TechnicalCorrection
**Created:** 2025-12-11 | **Updated:** 2026-03-25 | **Resolution:** Persuasive
**URL:** https://jira.hl7.org/browse/FHIR-53616

The submitter identifies that section 6.2.1 (Provider Bulk Access API) contains a diagram that incorrectly implies the payer "sends" a response to the provider when a Bulk query completes. In reality, the FHIR Asynchronous Request Pattern requires the requester to poll a content-location URL until the operation completes, then retrieve the NDJSON results. The ticket requests updating the Provider Access API page to accurately depict this three-step pattern: kick-off → poll → retrieve NDJSON.

## Resolution Details

The resolution states that prior work from FHIR-49943 and FHIR-53256 had already clarified the async pattern for the payer-to-payer and bulk-member-match pages. This ticket extends that same treatment to the **provider access page** (`provider-access-api.md`), which previously only mentioned the `Prefer: respond-async` header requirement without describing the actual async flow.

The applied change adds sections §pdex-304a and §pdex-304b to `provider-access-api.md` with:
- The explicit three-step pattern: **kick-off → poll → retrieve NDJSON**
- An explicit statement that the payer does **not** push data to the provider

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Rationale:** The ticket correctly identifies a gap where the Provider Access API page only specified the `Prefer: respond-async` header but lacked any description of the actual FHIR Asynchronous Request Pattern. This is a straightforward technical correction that aligns the provider access documentation with the same treatment already applied to the payer-to-payer and bulk-member-match pages via FHIR-49943 and FHIR-53256. The resolution adds well-defined sections (§pdex-304a/304b) documenting the kick-off → poll → retrieve NDJSON flow and clarifies that the payer does not push data. This is consistent, accurate, and reduces implementer confusion.

**Trade-offs:** None significant — this is a documentation improvement that makes an implicit requirement explicit without changing behavior.

### Disposition B: Persuasive with Modification — Consolidate Async Documentation

**Rationale:** While the fix is correct, the async pattern is now documented in three separate places (payer-to-payer, bulk-member-match, and provider access pages). A modification could consolidate the async pattern description into a single shared section with cross-references from each page, reducing duplication and maintenance burden. This would go beyond the original ask but improve long-term spec quality.

**Trade-offs:** Larger scope than the original technical correction; would require more review effort and coordination with the resolutions from FHIR-49943 and FHIR-53256. Risk of scope creep.

### Disposition C: Not Persuasive — Defer to Existing Specifications

**Rationale:** The FHIR R4 Asynchronous Request Pattern is already a well-defined specification. The `Prefer: respond-async` header reference in the existing text, combined with the normative reference to the FHIR async spec, may be considered sufficient for implementers familiar with the standard. Adding redundant descriptions of the async flow in each IG page could introduce maintenance risk if the base pattern changes.

**Trade-offs:** Leaves the Provider Access API page less explicit than its sibling pages (payer-to-payer, bulk-member-match), which have already received the detailed async flow documentation. Implementers new to the FHIR async pattern would still need to navigate to the base spec to understand the full flow, creating an inconsistency with other PDex pages.

## Recommendation

**Accept as Persuasive (Disposition A).** The resolution is a clean, well-scoped technical correction that addresses a genuine documentation gap. The Provider Access API page now accurately describes the async pattern consistently with the payer-to-payer and bulk-member-match pages. The change is already pre-applied and labeled as a technical correction, requiring no behavioral or structural changes to the IG.
