## Summary

This ticket requests that the **Claim Received Date** be promoted from a `supportingInfo` element to a first-class backbone element on the Claim, ClaimResponse, and ExplanationOfBenefit (EOB) resources, modeled similarly to the existing `created` date element. The ticket was originally submitted in the context of the CARIN Blue Button IG but has been identified as a FHIR Core request (per comment by Corey Spears, 2025-10-30).

The ticket was initially resolved as "Considered for Future Use" in July 2020 with a vote of 18-0-5, with the direction to consider in the next FHIR version. It was subsequently re-opened post-STU2 publication in November 2022 by Lynn Laakso.

## Details

### Current State
The CARIN Blue Button IG currently maps the Claim Received Date using `ExplanationOfBenefit.supportingInfo`, which is a general-purpose backbone element designed for additional clinical or administrative information. While functional, this approach treats a common and important business date as ancillary data rather than a structured, named element.

### Proposed Change
Define the Claim Received Date as a dedicated element on the Claim, ClaimResponse, and ExplanationOfBenefit resources, analogous to the existing `created` element (which records the date the resource was created). This would give the received date first-class status with a defined path, making it directly searchable, constrainable, and semantically clear.

## Proposed Dispositions

### Disposition A: Accept — Add a `receivedDate` element to Claim, ClaimResponse, and EOB

Add a new `dateTime` element (e.g., `receivedDate` or `received`) to the Claim, ClaimResponse, and ExplanationOfBenefit resources, parallel to the existing `created` element. This element would record the date the claim was received by the payer/processor.

**Rationale:** The claim received date is a universally important business date in claims processing, used across professional, institutional, and pharmacy claim types. Promoting it from `supportingInfo` to a named element improves semantic clarity, enables direct search parameter definition, and aligns with how other key dates (e.g., `created`, `billablePeriod`) are modeled. The original vote (18-0-5) showed strong support, and the ticket was explicitly re-opened for consideration in a future FHIR version.

### Disposition B: Accept with Modification — Add `receivedDate` to EOB only, not Claim/ClaimResponse

Add the `receivedDate` element only to the ExplanationOfBenefit resource, where it is most commonly needed (payer-side reporting), rather than to all three financial resources. Claim and ClaimResponse could continue using `supportingInfo` for this date since those resources are typically provider-authored.

**Rationale:** The primary use case for claim received date is in payer adjudication reporting via the EOB. Adding the element to the Claim resource (which is provider-submitted) is less clear since the submitter may not know when the payer received it. A narrower scope reduces the change footprint while addressing the core need.

### Disposition C: Decline — Retain `supportingInfo` approach

Decline the request and continue using `supportingInfo` for the claim received date. Document recommended usage patterns in the implementation guidance to standardize how implementers encode this date.

**Rationale:** The `supportingInfo` backbone element is designed for exactly this type of additional administrative data. Adding a dedicated element for every date used in claims processing risks unbounded growth of the resource structure. IGs like CARIN BB can define slices on `supportingInfo` with specific category codes and cardinality constraints to achieve the same level of interoperability without modifying the base resource.

## Recommendation

**Disposition A: Accept** — Add a `receivedDate` element to Claim, ClaimResponse, and ExplanationOfBenefit.

The claim received date is a fundamental business concept in claims processing that is universally needed across claim types. The original 18-0-5 vote demonstrated strong consensus for this change. The ticket was deliberately re-opened post-STU2 for inclusion in a future FHIR version, and the 2025 comment confirms it has been triaged as a FHIR Core change. Promoting this date to a named element follows the established pattern used by `created` and other key dates, improving discoverability and enabling straightforward search parameter definition without relying on category-coded `supportingInfo` slicing.
