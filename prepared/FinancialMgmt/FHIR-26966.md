## Description

The submitter (from the Workflow / FHIR-I group) asserts that the Contract resource properly belongs to the **Event pattern**, but its current definition attempts to track activities related to or leading to contract activation within the Contract resource itself. This conflates multiple concepts that are at odds with the general **Request/Event/Definition** workflow patterns.

The submitter requests discussion of the intent of the Contract resource and how it can work with other resources (e.g., **Task**) to manage the associated lifecycle of a contract, rather than embedding lifecycle state management within Contract directly.

## Analysis

### Core Issue

The Contract resource currently embeds multiple overlapping state machines and lifecycle tracking within a single resource, conflating:
1. The **contract document** itself (the legal artifact)
2. The **contract definition** (template/terms)
3. The **contract execution** (negotiation, activation, lifecycle)

This design deviates from FHIR's standard Request/Event/Definition workflow patterns, where these concerns would typically be separated across different resources or managed via Task-based workflow orchestration.

### Evidence of Consensus

- **FHIR-26947** (sister ticket to FHIR Infrastructure) was resolved as **Persuasive**, confirming that Contract belongs to the Event pattern.
- **FHIR-55563** (2026) was resolved as **Persuasive with Modification**, explicitly stating Contract is "not ready to go normative" due to its state machine incoherence and unimplemented refactoring plans.
- Multiple related tickets (FHIR-22704, FHIR-20152) raise the same fundamental concern from different angles.

### Open Questions

1. What is the concrete refactoring plan identified in the workflow calls referenced in FHIR-55563? Has any of this work been captured in a design document?
2. Should the Contract lifecycle activities be managed exclusively via Task resources, or is a hybrid approach more practical?
3. How should existing implementations be migrated if the Contract state machine is significantly restructured?

---

## Proposed Dispositions

### Disposition A: Persuasive (Refactor Contract to align with Event pattern)
- **Action:** Accept the ticket and commit to refactoring the Contract resource to align with the Event workflow pattern. Separate the contract document, definition, and execution concepts. Delegate lifecycle orchestration to Task resources as recommended by the Workflow group.
- **Rationale:** The sister ticket FHIR-26947 was already resolved as Persuasive by FHIR-I, establishing that Contract belongs to the Event pattern. FHIR-55563 confirms the refactoring need. Alignment with standard workflow patterns is essential for consistency, implementability, and normative readiness.
- **Risk:** Major refactoring effort; breaking change for existing implementations.

### Disposition B: Persuasive with Modification (Accept in principle, defer to broader Contract refactoring effort)
- **Action:** Accept the concern as valid but scope this ticket to documenting the intended alignment rather than performing the full refactoring. Link this ticket to FHIR-55563 and the broader Contract normative-readiness effort so the state machine refactoring is tracked as part of that consolidated work.
- **Rationale:** The concern is well-founded and already validated by multiple resolved tickets, but the refactoring is substantial and should be coordinated under a single effort (FHIR-55563) rather than addressed piecemeal. This avoids duplicate or conflicting changes.
- **Risk:** Further delay if the broader effort stalls; may appear as inaction on a 5-year-old ticket.

### Disposition C: Not Persuasive (Maintain current design)
- **Action:** Reject the change, keeping Contract's current state machine and internal lifecycle tracking as-is.
- **Rationale:** The Contract resource's complexity reflects the real-world complexity of contract management. Separating concerns across multiple resources could increase implementation burden and reduce usability for systems that need a self-contained contract representation.
- **Risk:** Contradicts the already-published FHIR-26947 (Persuasive) resolution and the FHIR-55563 (Persuasive with Modification) resolution. Would leave Contract misaligned with FHIR workflow patterns and block normative advancement.

## Recommendation

**Disposition B: Persuasive with Modification** — The concern is valid and well-supported by related resolutions (FHIR-26947 Persuasive, FHIR-55563 Persuasive with Modification). However, since FHIR-55563 already tracks the broader Contract normative-readiness refactoring effort, this ticket should be resolved by acknowledging the workflow alignment need and linking the actual refactoring work to the consolidated Contract effort. This avoids fragmented tracking of what is fundamentally one large design problem while validating the submitter's concern.
