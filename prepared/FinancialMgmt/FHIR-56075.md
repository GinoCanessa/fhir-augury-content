## Summary

The reporter identified that the IN parameters for the `$provider-member-match` operation differ between two authoritative sources in the Da Vinci PDex IG:

1. **IG Parameters profile** (`provider-parameters-multi-member-match-bundle-in` StructureDefinition page) lists IN params as: `MemberPatient`, `CoverageToMatch`, `Consent`, `CoverageToLink`.
2. **Package OperationDefinition** (`OperationDefinition-ProviderMemberMatch.json` in `definitions.json.zip`) lists IN params as: `MemberPatient`, `CoverageToMatch`, `CoverageToLink`, `TreatmentAttestation`.

The key discrepancies are:
- **`Consent` vs `TreatmentAttestation`**: The IG profile uses `Consent` while the OperationDefinition uses `TreatmentAttestation` for the same parameter.
- **Missing profile links**: The OperationDefinition page in the IG does not link to the IN/OUT Parameters profiles, making it hard for readers to cross-reference.

The ticket requests that the two artifacts be aligned and that the OperationDefinition page add links to the Parameters profiles.

### Resolution Applied

The resolution (Persuasive) applied changes to two FSH source files:

**`ProviderMultiMemberMatchRequestProfile.fsh`:**
- Renamed container slice from `MemberBundle` → `MembersToMatch` (to align wire-format name with OperationDefinition)
- Renamed part slice from `Consent` → `TreatmentAttestation` (to match OperationDefinition and avoid confusion with patient consent in payer-to-payer exchange)

**`Operation-provider-member-match.fsh`:**
- Added links to the IN (`provider-parameters-multi-member-match-bundle-in`) and OUT (`provider-parameters-multi-member-match-bundle-out`) profiles in the operation's Description
- Added profile reference in `MembersToMatch` parameter documentation
- Added clarifying note in `TreatmentAttestation` part documentation explaining the naming rationale

> **Note:** A subsequent ticket [FHIR-56242](https://jira.hl7.org/browse/FHIR-56242) reversed the container slice rename, changing `MembersToMatch` back to `MemberBundle` to align with the Payer-to-Payer Bulk Member Match naming. The workgroup should ensure these two resolutions are reconciled.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Affirm Pre-Applied Resolution)

Accept the pre-applied resolution as-is. The changes align the Parameters profile with the OperationDefinition by renaming `Consent` → `TreatmentAttestation` and `MemberBundle` → `MembersToMatch`, and add profile links to the OperationDefinition page.

- **Pros:** Fixes the documented discrepancy. The `TreatmentAttestation` name is more semantically accurate and avoids confusion with the `Consent` resource used in payer-to-payer workflows. Adding profile links improves IG navigability.
- **Cons:** The `MemberBundle` → `MembersToMatch` rename conflicts with FHIR-56242, which reverses it. Accepting both tickets as-is creates contradictory changes. The workgroup must reconcile the container slice name with FHIR-56242's resolution.
- **Risk:** Medium — if FHIR-56242 is also accepted, the final published IG will depend on which change is applied last, creating potential implementer confusion.

### Disposition B: Accept as Persuasive with Modification

Accept the `Consent` → `TreatmentAttestation` rename and the addition of profile links to the OperationDefinition page, but **modify** the container slice name to use `MemberBundle` (instead of `MembersToMatch`) to align with FHIR-56242 and the Payer-to-Payer Bulk Member Match convention.

- **Pros:** Resolves the parameter naming discrepancy (`TreatmentAttestation`), adds missing profile links, and proactively reconciles with FHIR-56242 to ensure a consistent container name across provider and payer operations. Reflects Connectathon implementer feedback favoring `MemberBundle`.
- **Cons:** Partially diverges from the pre-applied change, requiring an update to the already-applied FSH source.
- **Risk:** Low — both tickets agree on `TreatmentAttestation`; only the container slice name needs coordination.

### Disposition C: Not Persuasive — Defer to FHIR-56242

Reject this ticket in favor of FHIR-56242, which addresses overlapping naming changes (`MembersToMatch` → `MemberBundle`, `Consent` → `TreatmentAttestation`) and was filed based on Connectathon implementer experience. The profile link additions could be tracked as a separate enhancement.

- **Pros:** Avoids conflicting resolutions across two tickets. FHIR-56242's naming convention (`MemberBundle`) aligns with other PDex operations.
- **Cons:** Loses the profile link additions from FHIR-56075's resolution. The `Consent` → `TreatmentAttestation` rename would need to be credited to FHIR-56242 instead. Also, FHIR-56075 was filed first and its reporter's observation is valid.
- **Risk:** Low technical risk, but discards useful improvements (profile links) that would need to be re-raised separately.

## Recommendation

**Disposition B: Accept as Persuasive with Modification.**

The core issue — the `Consent` vs `TreatmentAttestation` discrepancy — is a genuine technical correction that improves clarity and avoids confusion with the FHIR `Consent` resource used in payer-to-payer workflows. The addition of profile links to the OperationDefinition page is a clear navigability improvement. However, the container slice rename (`MemberBundle` → `MembersToMatch`) conflicts with FHIR-56242, which has Connectathon-driven justification for reverting to `MemberBundle`. Accepting with modification to use `MemberBundle` as the container name reconciles both tickets, maintains consistency across PDex operations, and preserves all the valuable improvements from this ticket's resolution. The workgroup should also note the related unresolved ticket FHIR-56107 (incorrect `MatchedMembers` OUT param reference) for concurrent review.
