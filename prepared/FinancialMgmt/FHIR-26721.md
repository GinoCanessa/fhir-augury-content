## Summary

This ticket requests that the various payer adjudication amount categories currently defined via `supportingInfo` be formally defined as data elements within the `adjudication` section of the Claim, ClaimResponse, and ExplanationOfBenefit (EOB) resources. The specific amount fields are: **submitted**, **eligible**, **deductible**, **coinsurance**, **copay**, **noncovered**, **priorpayerpaid**, **benefit**, **paidbypatient**, **paidtoprovider**, **paidtopatient**, **memberliability**, **discount**, and **drugcost**.

Originally submitted against the CARIN Blue Button IG, this was reclassified as a FHIR Core request (per comment by corey_spears, 2025-10-30). It was previously resolved as "Considered for Future Use" with a vote of 18-0-5 (Mark Roberts / Pat Taylor, 2020-07-21), then reopened post-STU2 publication (lynn_laakso, 2022-11-28) for reconsideration in a future FHIR version.

## Details

### Problem

The FHIR Core ExplanationOfBenefit resource does not formally define the set of adjudication amount categories within the `adjudication` backbone element. Instead, these amount types are modeled as `supportingInfo` entries, requiring IG-level profiling (e.g., CARIN Blue Button) to establish adjudication category slices. This leads to:

1. **Inconsistent implementation** — Different IGs define their own value sets and slicing approaches for the same fundamental adjudication concepts.
2. **Ambiguity in calculations** — As documented in FHIR-51208, the relationships between amounts (e.g., how noncovered interacts with eligible and memberliability) are unclear without formal definitions.
3. **Repeated profiling effort** — Every financial IG must independently profile the same adjudication amount concepts.

### Existing Wording

The IG defines the Adjudication Amount Category as `supportingInfo`.

### Proposed Wording

Define these data elements within the adjudication section of the Claim, ClaimResponse, and ExplanationOfBenefit resources.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

**Action:** Define the 14 adjudication amount categories (submitted, eligible, deductible, coinsurance, copay, noncovered, priorpayerpaid, benefit, paidbypatient, paidtoprovider, paidtopatient, memberliability, discount, drugcost) as named concepts in the FHIR Core adjudication code system with formal definitions. Bind these to `ExplanationOfBenefit.item.adjudication.category`, `ExplanationOfBenefit.adjudication.category`, and `ExplanationOfBenefit.total.category` via an expanded Core value set with extensible binding. Add narrative guidance in the EOB notes explaining the semantic relationships between amounts (e.g., eligible = submitted − noncovered − discount).

**Rationale:** These categories are universally needed across financial IGs (CARIN BB, PCT, DaVinci) and are already defined at the IG level. Elevating them to Core would reduce duplicated profiling effort, resolve semantic confusion (as evidenced by the Zulip discussion and FHIR-51208), and establish a single authoritative definition for each amount type. The original vote (18-0-5) showed strong support, and the ticket was reopened specifically for inclusion in a future FHIR version.

### Disposition B: Accept with Modification (Persuasive with Modification)

**Action:** Rather than defining all 14 categories in Core, add a subset of the most universally applicable adjudication amounts (submitted, eligible, deductible, copay, coinsurance, noncovered, benefit) to the Core adjudication code system with formal definitions. Leave domain-specific categories (drugcost, priorpayerpaid, memberliability, discount, paidbypatient, paidtoprovider, paidtopatient) to be defined by realm-specific or domain-specific IGs. Add guidance in the EOB resource notes explaining the adjudication backbone's intended use for financial amounts and the role of the category element.

**Rationale:** Some of the 14 categories (e.g., drugcost) are domain-specific and may not apply universally across all claim types or jurisdictions. A tiered approach keeps Core focused on broadly applicable concepts while still addressing the core issue of missing definitions. This mirrors how FHIR-32776 (already published) added guidance without fully restructuring the value set.

### Disposition C: Decline (Not Persuasive)

**Action:** Close as not persuasive. The existing FHIR Core adjudication code system already includes foundational codes (submitted, eligible, deductible, copay, benefit, tax). The remaining categories have been successfully defined at the IG level (CARIN BB, PCT) where domain-specific context can inform precise definitions. No changes to FHIR Core.

**Rationale:** The CARIN BB IG (FHIR-26992) has already resolved the practical need by defining the full adjudication amount value set at the IG level. FHIR-32776 added additional guidance on adjudication usage to Core. The remaining categories (e.g., drugcost, memberliability) are US-payer-specific concepts that may not translate to international contexts, making Core inclusion inappropriate. IG-level definition provides the flexibility needed for different jurisdictions and business domains.

## Recommendation

**Disposition A: Accept (Persuasive)**

This ticket has strong historical support (18-0-5 vote), was deliberately reopened for future FHIR versions, and addresses a well-documented gap. The Zulip thread on CPCDS EOB Data Fields demonstrates real implementer confusion about adjudication amount semantics — confusion that formal Core definitions would resolve. The existence of FHIR-51208 (noncovered amount inconsistency) and the multiple IGs that have independently profiled the same adjudication categories confirms that this is a universal need, not an IG-specific concern. The 14 amount types are already present in the Core adjudication code system; what is missing is formal definitions and guidance on their relationships. Given that this is now confirmed as a FHIR Core request and has been waiting since 2020, full acceptance is warranted.
