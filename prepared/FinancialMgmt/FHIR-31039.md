## Description

The `Claim / EOB.procedure.type` element is currently bound to an example Code System ([ex-procedure-type](https://www.hl7.org/fhir/codesystem-ex-procedure-type.html)) that defines values of **"primary"** and **"secondary"**. The reporter notes that neither of these values applies to the values defined in claim submission standards. The request is to:

1. **Delete** the existing example Value Set and Code System (`ex-procedure-type`).
2. **Replace** with an ICD-aligned Code System / Value Set using values of **"principal"** and **"other"**.

This aligns the FHIR Core terminology with US institutional claim submission standards (e.g., UB-04), which distinguish between a "principal" procedure (the procedure performed for definitive treatment) and "other" procedures.

## Analysis

### Core Issue
The current FHIR Core example Code System `ex-procedure-type` uses the terms **"primary"** and **"secondary"** for `Claim.procedure.type` and `ExplanationOfBenefit.procedure.type`. These terms do not align with US institutional claim submission standards (UB-04), which use **"principal"** (the procedure performed for definitive treatment) and **"other"** (all remaining procedures). This misalignment creates confusion and requires implementers in CARIN Blue Button, Da Vinci PCT, and other US-realm IGs to define their own replacement terminology.

### Precedent
- **FHIR-28603** (Persuasive, Published) already established "Principal" and "Other" as procedure code types in the CARIN Blue Button IG.
- **FHIR-34902** (Persuasive, Published) renamed the PCT institutional profile procedure slice from "primary" to "principal."
- **FHIR-20527** provides extensive background confirming that "principal procedure" is the established clinical coding/billing concept.

These precedents strongly support updating the FHIR Core example Code System to match.

### Scope of Impact
- **Resources affected:** Claim, ExplanationOfBenefit (both reference `procedure.type`)
- **Artifacts affected:** `CodeSystem/ex-procedure-type`, `ValueSet/ex-procedure-type`
- **Binding strength:** Example (change would not break existing implementations since example bindings are non-normative)
- **Downstream IGs:** CARIN Blue Button and Da Vinci PCT have already moved to "principal"/"other" in their own value sets; aligning FHIR Core would reduce divergence.

### Open Question
The comment from pknapp suggests consulting with Mary Kay about the X12 equivalent for this concept. Resolving whether X12 defines specific procedure type codes could inform whether the replacement values should be "principal"/"other" or should reference an external code system.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Accept the change request. Delete the existing `ex-procedure-type` Code System and Value Set (with values "primary" and "secondary") and replace with a new Code System / Value Set defining values **"principal"** and **"other"**, aligned with institutional claim submission standards (UB-04).

**Rationale:** The current "primary"/"secondary" terminology does not match established claim submission standards. CARIN Blue Button (FHIR-28603) and Da Vinci PCT (FHIR-34902) have already adopted "principal"/"other" via persuasive resolutions. Aligning the FHIR Core example code system reduces fragmentation and provides consistent base terminology for downstream IGs. Since the binding is example-strength, the change is non-breaking.

### Disposition B: Persuasive with Modification
**Action:** Rather than deleting the example Code System and creating a new one, **update the existing `ex-procedure-type` Code System** to replace "primary" with "principal" and "secondary" with "other". Retain the Code System as an example binding. Additionally, add a note in the element definition clarifying that the values align with UB-04 institutional claim procedure type classification.

**Rationale:** This achieves the same alignment goal while preserving the existing Code System URL and structure, minimizing disruption to implementers who may reference the URL. The modification also adds clarifying documentation to help international implementers understand the US-realm origin of these values.

### Disposition C: Not Persuasive
**Action:** Reject the change. Keep the existing `ex-procedure-type` Code System with "primary" and "secondary" values.

**Rationale:** The binding is example-strength, meaning IGs are free to define their own value sets (as CARIN BB and PCT have done). Changing the FHIR Core example values to US-specific claim terminology ("principal"/"other") may not serve international implementers where different procedure classification schemes apply. The open question about X12 equivalents (per pknapp's comment) remains unresolved and should be investigated before making changes.

---

## Recommendation

**Disposition B: Persuasive with Modification**

The strongest path forward is to update the existing Code System in place (rather than delete and recreate) to use "principal" and "other", which aligns with the well-established UB-04 convention and the precedent set by FHIR-28603 and FHIR-34902. Since the binding is example-strength, this change is non-breaking but provides better guidance to implementers. However, the workgroup should first address the open question raised by pknapp regarding X12 equivalents—confirming or ruling out an external standard for procedure type classification—before finalizing the specific code values. Adding documentation noting the US-realm origin of these values would help international implementers evaluate applicability to their context.
