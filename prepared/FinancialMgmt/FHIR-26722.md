**Request:** Move the "allowed units" data element from `supportingInfo` to the `detail` section of the ExplanationOfBenefit (EOB) resource. The allowed units represent the quantity of units, times, days, visits, services, or treatments allowed for a service described by an HCPCS code, revenue code, or procedure code as submitted by the provider. This data element resides in all medical claims.

**Existing Wording:** The IG defines allowed number of units as `supportingInfo`.

**Proposed Wording:** Define allowed units in the detail section.

**Affected Resources:** Claim, ClaimResponse, ExplanationOfBenefit.

## 7. Analysis

### Context
This request originated from the CARIN Blue Button (C4BB) IG development effort (confirmed by corey_spears in 2025-10). The C4BB IG had to model "allowed units" using `ExplanationOfBenefit.supportingInfo` because there was no dedicated field in the EOB `detail` backbone element for this concept. The submitter argues that since allowed units are a fundamental property of a specific service line item, they belong in the `detail` section rather than as loosely-typed supportingInfo entries.

### Technical Considerations
- **Current state:** `ExplanationOfBenefit.item.detail` has a `quantity` element (SimpleQuantity) that represents the count of products or services. However, the "allowed" quantity (adjudicated/approved units) is distinct from the submitted quantity.
- **Structural fit:** Placing allowed units at the detail level associates them directly with the service line item, which is semantically correct. The current `supportingInfo` approach creates an indirect link via category codes.
- **Scope:** The description mentions Claim, ClaimResponse, and EOB, suggesting the change could affect all three resources. However, "allowed units" is an adjudication concept most relevant to EOB and ClaimResponse.
- **Prior vote:** Was originally resolved as "Considered for Future Use" in July 2020 (18-0-5) but was re-opened post STU2 publication in November 2022.
- **Clarity concern:** saul_kravitz noted in 2020 that it was unclear what specific text or field the request refers to. The proposal could benefit from a more precise specification of exactly where in `detail` the element should go and what its datatype should be.

### Risk
- **Breaking change:** Adding new elements to backbone structures is generally additive and non-breaking.
- **Cross-resource consistency:** Changes should be considered across Claim, ClaimResponse, and EOB together.
- **Cohort impact:** This is one of ~10 similar tickets; a piecemeal approach risks inconsistency, while a batch approach could be more efficient.

## 8. Proposed Dispositions

### Disposition A: Persuasive — Add an `allowedQuantity` element to the detail backbone
**Action:** Add a new optional `SimpleQuantity` element (e.g., `allowedQuantity` or `adjudicatedQuantity`) to `ExplanationOfBenefit.item.detail` (and corresponding structures in ClaimResponse) to represent the payer-allowed quantity for a service line. Update definitions and provide guidance distinguishing submitted `quantity` from allowed quantity.

**Rationale:** The allowed units concept is intrinsically tied to a specific service line item and should be co-located with other line-item details rather than indirectly referenced via `supportingInfo`. This aligns with the broader CARIN BB community's experience and reduces implementation complexity for payer data exchange.

### Disposition B: Persuasive with Modification — Address via adjudication quantity rather than detail
**Action:** Instead of adding a new element to `detail`, leverage the existing `ExplanationOfBenefit.item.adjudication` structure which already has a `quantity` element in R5. Provide guidance that the allowed quantity should be represented as an adjudication entry with an appropriate category code (e.g., "allowed-units"). This avoids adding new elements and uses existing structures.

**Rationale:** Allowed units are fundamentally an adjudication concept (what the payer allows), not a submitted claim detail. The adjudication backbone already supports quantity, and using a category code keeps the model extensible without structural changes.

### Disposition C: Not Persuasive — Retain supportingInfo approach
**Action:** Close the ticket. The current `supportingInfo` mechanism is sufficient for conveying allowed units. IGs like CARIN BB can define specific `supportingInfo` category codes and slices to make the data computable and interoperable.

**Rationale:** The `supportingInfo` pattern is a deliberate design choice for extensibility. Promoting every IG-specific data element to a backbone element risks unbounded growth of the core resource. The 80/20 rule suggests that allowed units, while important for US payer workflows, may not be universal enough for the core spec.

## 9. Recommendation

**Disposition A: Persuasive — Add an `allowedQuantity` element to the detail backbone.**

This ticket has been open for five years, was re-opened after an initial "future use" resolution, and represents a real interoperability pain point identified by the CARIN BB community. The request is part of a cohort of similar structural improvement tickets, all pointing to the same fundamental issue: `supportingInfo` is being used as a catch-all for data that has a natural home in more specific backbone elements. Adding an optional `allowedQuantity` element to the detail section is additive, non-breaking, and directly addresses the semantic gap. However, the work group should coordinate this change with the sibling tickets (FHIR-26717 through FHIR-26723) to ensure a consistent approach to promoting supportingInfo data elements across the EOB resource.
