## Summary

This change request asks that the Da Vinci PDex specification remove the requirement to store bulk export operation details (download timestamps, `_type` values, `_typeFilter` values) as extensions on Patient and Group resources. The submitter argues the current approach is fundamentally flawed because: (1) the same Patient or Group may be used by multiple export operations from different clients, causing data collision; (2) a read-only bulk export operation would improperly modify the resources it reads; and (3) a client with only read permission would effectively be triggering writes. The submitter proposes that audit trail requirements be handled internally by the server or client rather than through resource extensions.

The resolution (Persuasive) removes three extension definitions (`LastTransmission`, `LastTypes`, `LastFilters`) and replaces the associated SHALL conformance statements with a SHOULD recommendation to maintain an internal audit trail.

## Details

### Description

The current PDex specification mandates that several bulk export details be recorded as extensions on Patient and Group resources:

- **Provider-download tracking:** The Data Provider/Payer SHALL update an extension on the Patient with the latest download date/time for each provider-requested export.
- **`_type` recording:** When `_type` is used, the export operation SHALL record the `_type` parameter content in the `lastResources` extension and update `lastTransmitted` with the export timestamp or `_until` value.
- **`_typeFilter` recording:** When `_typeFilter` is used, the export operation SHALL record the `_typeFilter` parameter content in the `lastFilters` extension and update `lastTransmitted` similarly.

The submitter identifies three concrete problems:
1. The same Patient can be exported by different operations from different clients, causing conflicting extension values.
2. The same Group can be used by different export operations, similarly causing collisions.
3. A client with only read permission ends up triggering resource modifications — a violation of expected read semantics.

The submitter recommends that if the server needs to track export history, it should use an internal audit trail rather than externally visible extensions. Similarly, clients that need this data can maintain their own audit records.

### Resolution Applied

The resolution removes the problematic extensions and conformance statements across multiple FSH and markdown files:

- **Extensions.fsh** — Removed `LastTransmission` (`base-ext-last-transmission`), `LastTypes` (`base-ext-last-types`), and `LastFilters` (`base-ext-last-typefilter`) extension definitions; replaced with a comment referencing FHIR-55987.
- **ProfilePDexProviderGroup.fsh** — Removed the `member.extension` slice block for all three extensions; updated profile description.
- **ExamplePDexProviderGroup.fsh** — Removed the three extension value lines from the example member.
- **provider-access-api-v2.md** — Removed §pdex-269 (SHALL update extension on provider-download), §pdex-276/277 (SHALL record `_type`), §pdex-280/281 (SHALL record `_typeFilter`); replaced with SHOULD audit trail recommendation.
- **provider-access-api.md** — Removed parallel requirements (§pdex-313, §pdex-329, §pdex-336/337, §pdex-340/341); replaced with SHOULD audit trail statement.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)
**Action:** Accept the pre-applied resolution as-is — remove the three bulk export tracking extensions (`LastTransmission`, `LastTypes`, `LastFilters`) and their associated SHALL conformance statements from both provider-access-api.md and provider-access-api-v2.md, replacing them with a SHOULD recommendation for internal audit trails.

**Rationale:** The current requirement is architecturally unsound. Storing per-request export metadata as extensions on shared Patient/Group resources creates data collisions when multiple clients export the same resources, leaks one client's query parameters to another, and violates the principle that read operations should not modify the data they read. The Zulip discussions show clear community consensus (Michele Mottini, Cooper Thompson, Paul Church) that internal audit trails or client-side tracking are the correct approach. The `transactionTime` value already returned by bulk export responses provides the `_since` hint that may have originally motivated these extensions. The pre-applied changes are well-scoped and correctly downgrade the tracking requirement from SHALL to SHOULD.

### Disposition B: Accept as Persuasive with Modification — Add AuditEvent Guidance
**Action:** Accept the extension removal as proposed, but additionally add non-normative guidance recommending that servers use FHIR AuditEvent resources to record bulk export operations, including client identity, timestamps, and query parameters used.

**Rationale:** While removing the extensions is clearly correct, some implementers may want standardized audit trail guidance. Adding a brief section recommending AuditEvent as the mechanism for server-side export audit logging would preserve the intent of the original requirement (trackability) without the architectural problems. This goes slightly beyond the submitted change but addresses Paul Church's speculation about the original intent.

### Disposition C: Decline — Not Persuasive
**Action:** Reject the change and retain the current extension-based tracking requirements.

**Rationale:** The extensions provide a standardized, interoperable way for servers to expose export history. Removing them eliminates a mechanism that downstream systems might rely upon for operational monitoring or compliance. However, this disposition is weak because the Zulip discussion clearly demonstrates the current approach causes data leakage between clients and violates read-only semantics, problems that outweigh the interoperability benefit.

## Recommendation

**Disposition A: Accept as Persuasive.** The architectural problems identified — data collision across clients, cross-client information leakage, and read operations triggering writes — are genuine and well-documented in both the ticket and the Zulip discussions. The pre-applied resolution is clean, well-scoped, and already has community consensus. The existing `transactionTime` in bulk export responses adequately addresses the likely original motivation for these extensions.
