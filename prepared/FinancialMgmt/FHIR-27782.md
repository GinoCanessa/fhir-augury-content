## Description

The submitter requests that `CarePlan.goal` and `CarePlan.careTeam` be constrained to 0..0 in the PCDE (Payer Coverage Decision Exchange) profile. The rationale is that both elements create confusion by implying the profile represents a clinical care plan, when in fact the PCDE IG uses the CarePlan resource as a retrofit of PCDE ActiveTreatments. The submitter states these attributes are never expected to be populated in the PCDE context and should therefore be prohibited.

## FHIR Elements Referenced

- **CarePlan.goal** — Reference to goals the care plan addresses (base: 0..*)
- **CarePlan.careTeam** — Who is involved in the care plan (base: 0..*)

## Comments

### mckenzie (2020-11-23)
Expressed concern about prohibiting these elements. Key points:
- A payer **could** have goals for a particular therapy or know the set of providers relevant to ongoing therapy.
- Sending this information would not be strictly **wrong** — which is the usual bar for prohibiting something in FHIR.
- Does not want to constrain these out in the initial release.
- Recommends group discussion.

## Proposed Dispositions

### Disposition A: Not Persuasive

**Rationale:** The standard FHIR design principle is that elements should only be prohibited (constrained to 0..0) when sending the data would be *incorrect*, not merely because the data is unexpected. As noted by mckenzie, a payer could legitimately have goals for a therapy or know relevant providers. Prohibiting these elements would prevent valid data exchange. Leaving them as 0..* (the base cardinality) allows flexibility without requiring population. Other IGs (US Core via FHIR-20036) have moved in the opposite direction, arguing these elements should be *required*. Constraining them out of PCDE would diverge from community direction.

**Action:** Close with no change to the PCDE profile. CarePlan.goal and CarePlan.careTeam remain 0..* as inherited from the base resource.

### Disposition B: Persuasive

**Rationale:** The PCDE IG specifically uses CarePlan as a vehicle for ActiveTreatments data, not as a clinical care plan. Leaving clinical-oriented elements like `goal` and `careTeam` unconstrained sends a misleading signal to implementers about the profile's intent. Zeroing them out makes the profile's scope explicit, reduces implementer confusion, and aligns the profile more tightly with its actual use case. Since PCDE is an IG-level profile (not the base resource), constraining to 0..0 does not affect other uses of CarePlan.

**Action:** Constrain both `CarePlan.goal` and `CarePlan.careTeam` to 0..0 in the PCDE CarePlan profile. Add a descriptive note explaining that PCDE uses CarePlan for ActiveTreatments representation and these clinical elements are out of scope.

### Disposition C: Persuasive with Modification

**Rationale:** Rather than prohibiting these elements entirely, add IG-level guidance clarifying that `CarePlan.goal` and `CarePlan.careTeam` are not expected to be populated in the PCDE context but are not prohibited. This addresses the submitter's concern about implementer confusion while respecting the FHIR principle that data should not be prohibited unless sending it would be wrong. The elements could be explicitly marked as mustSupport = false with narrative explaining the PCDE scope.

**Action:** Do not constrain to 0..0. Instead, add an "Implementation Note" or "Usage" section in the PCDE IG explaining that these elements are out of scope for PCDE's ActiveTreatments use of CarePlan. Optionally set mustSupport = false to signal they are not expected. Consider adding an informational invariant or short description override to reinforce the profile's scope.

---

## Recommendation

**Disposition C: Persuasive with Modification**

This disposition best balances both concerns. The submitter's core issue — that implementers are confused about the profile's scope — is valid and should be addressed. However, mckenzie's point that payers *could* legitimately provide goal or care team data is also well-founded, and the FHIR design principle against prohibiting valid data is important. Adding clear guidance and marking these elements as not expected (without prohibiting them) resolves the confusion without closing the door on legitimate data exchange. This approach is also consistent with how other IGs have handled elements that are out of scope but not invalid — through documentation and mustSupport flags rather than 0..0 constraints.
