## Summary

This ticket requests a new **Member Change** operation for the Da Vinci ATR (Member Attribution) specification. Currently, only Member Add and Member Remove operations exist. When a consumer organization agrees that a member belongs in a group but disagrees with the attributed provider, it must issue a separate Member Remove followed by a Member Add — a non-atomic two-step process. The submitter argues that this pair of operations introduces two problems: (1) unnecessary web traffic and (2) risk that one operation succeeds while the other fails, leading to inconsistent state between producer and consumer. An atomic Member Change operation would resolve both issues.

This concern was reinforced during a PDex Community Call (2027-03-27), where implementers confirmed real-world risk of partial failure when using Remove + Add. A commenter also linked this to the earlier ticket FHIR-39667, which asked whether the Member Add operation could be used to modify an existing member's attributes.

## Details

### Description

The Da Vinci ATR specification defines Member Add (`$member-add`) and Member Remove (`$member-remove`) operations that allow a consumer (e.g., a provider organization) to manage members on a Member Attribution List maintained by a producer (e.g., a payer). However, there is no operation to **change** a member's attributed provider while keeping the member on the list.

The current workaround — issuing a Remove followed by an Add — is non-atomic. If the Remove succeeds but the Add fails (or vice versa), the producer and consumer end up with mismatched views of the attribution list. The submitter proposes a new `$member-change` operation that would atomically update a member's attributed provider, reducing both network overhead and the risk of inconsistency.

## Proposed Dispositions

### Disposition A: Persuasive (Accept — Define a new `$member-change` operation)

**Action:** Add a new `$member-change` operation to the Da Vinci ATR specification that atomically updates a member's attributed provider on the Member Attribution List. The operation should accept the member identifier, the current provider reference, and the new provider reference, and return an appropriate OperationOutcome. Define a corresponding OperationDefinition resource and a MemberChange profile if needed.

**Rationale:** The request identifies a genuine gap in the specification. The existing workflow figure in Section 2.5 already references "add/modify or delete" (per FHIR-39739), but no modify/change operation is defined. An atomic change operation eliminates the partial-failure risk confirmed by implementers on the PDex Community Call, aligns the spec with its own workflow diagrams, and reduces unnecessary network traffic. The predecessor ticket FHIR-39667 was resolved as Persuasive with Modification but did not fully address the need for a dedicated change operation.

### Disposition B: Persuasive with Modification (Extend `$member-add` to support updates)

**Action:** Instead of creating a new operation, modify the existing `$member-add` operation to support update semantics — i.e., if the member already exists on the attribution list, `$member-add` replaces the attributed provider rather than returning an error. Update the operation documentation to clarify this upsert behavior and ensure the operation is idempotent.

**Rationale:** This avoids introducing a third member management operation and keeps the API surface smaller. FHIR-39667 already raised the question of whether `$member-add` could serve as a modify, and it was resolved as Persuasive with Modification. Extending `$member-add` with upsert semantics achieves the atomicity goal without requiring consumers to learn a new operation. However, this approach conflates "add" and "change" semantics, which may reduce clarity.

### Disposition C: Not Persuasive (Decline)

**Action:** No change to the specification. Document that consumers should use sequential `$member-remove` and `$member-add` operations to change attributed providers.

**Rationale:** The existing operations are sufficient for the use case. The risk of partial failure can be mitigated through implementation guidance (e.g., retry logic, idempotency requirements on the producer side). Adding a new operation increases specification complexity and the implementation burden on both producers and consumers. The current two-step approach is well-understood and already in use.

## Recommendation

**Disposition A: Persuasive** — Define a new `$member-change` operation.

The case for a dedicated change operation is strong. Implementers have confirmed real-world concerns about partial failure with the Remove+Add workaround (per ekivemark's comment from the PDex Community Call). The ATR specification's own workflow diagram already references "modify" as a consumer action, yet no corresponding operation exists. The predecessor ticket FHIR-39667 partially addressed this but left the gap open. A dedicated `$member-change` operation provides clear semantics, atomicity guarantees, and aligns the specification with its documented workflows. While Disposition B (extending `$member-add`) is a reasonable alternative, a dedicated operation is clearer and avoids overloading the semantics of the add operation.
