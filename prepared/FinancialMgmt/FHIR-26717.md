## Description

The ticket requests that five NCPDP (National Council for Prescription Drug Programs) data elements currently defined as `supportingInfo` slices in IGs (such as CARIN Blue Button) be promoted to first-class backbone elements on the Claim, ClaimResponse, and ExplanationOfBenefit resources in FHIR Core.

**Affected Resources:** Claim, ClaimResponse, ExplanationOfBenefit

**Existing Wording:** The IG defines several NCPDP data elements as `supportingInfo`.

**Proposed Wording:** Define the following as Claim / ClaimResponse / EOB backbone data elements:
1. **Days Supply** — the number of days the dispensed medication is expected to last
2. **DAW (Dispensed as Written) Product Selection Code** — indicates whether the pharmacist dispensed as written or substituted
3. **Refill Number** — identifies which fill this is (00 = original, 01 = first refill, etc.)
4. **Prescription Origin Code** — identifies the origin of the prescription (written, phone, electronic, fax)
5. **Plan Reported Brand-Generic Code** — indicates whether the drug was classified as brand or generic by the plan

These data elements are required on all retail pharmacy claims per NCPDP standards.

## Analysis

### Context
This ticket is part of a batch of approximately 10 change requests submitted by Patricia Taylor in April 2020, all originating from CARIN Blue Button IG implementation experience. The common theme is promoting data elements from `supportingInfo` slices (an IG-level mechanism) to dedicated backbone elements in FHIR Core resources. The ticket was initially resolved as "Considered for Future Use" in July 2020 with a vote of 18-0-5, then re-opened post-STU2 publication in November 2022 for reconsideration.

### Impact Assessment
- **Resources affected:** Claim, ClaimResponse, ExplanationOfBenefit (3 resources)
- **Scope:** Adding up to 5 new backbone elements or a new backbone structure to accommodate pharmacy-specific data
- **Breaking change risk:** Additive change — new optional elements would not break existing implementations
- **IG impact:** CARIN Blue Button and other pharmacy-related IGs currently using `supportingInfo` slices would need to migrate to the new backbone elements, though `supportingInfo` could remain supported during a transition period
- **Terminology dependency:** FHIR-26718 (defining NCPDP code systems) has already been resolved as Persuasive, so the terminology foundation is in place

### Considerations
1. **Precedent:** The `total` backbone element was added to EOB in R4 based on similar reasoning (promoting data from extensions/workarounds to first-class elements). This ticket follows the same pattern.
2. **Scope question:** Should these elements be added individually to each resource, or grouped into a pharmacy-specific backbone (e.g., `pharmacyDetail`)? A grouped backbone would be cleaner but larger in scope.
3. **Cross-SDO coordination:** NCPDP data elements carry licensing requirements. The code system side (FHIR-26718) is resolved, but the structural side needs to ensure proper binding.
4. **Batch coordination:** This ticket should be considered alongside FHIR-26710 (NUBC backbone elements) and FHIR-26720 (Claim Received Date backbone) as they represent the same architectural decision applied to different claim types.

## Proposed Dispositions

### Disposition A: Persuasive — Add pharmacy backbone elements to Claim/ClaimResponse/EOB

**Action:** Add a new `pharmacyDetail` backbone element (or individual elements) to Claim, ClaimResponse, and ExplanationOfBenefit containing: daysSupply (SimpleQuantity), dawCode (CodeableConcept), refillNumber (positiveInt), prescriptionOriginCode (CodeableConcept), and brandGenericCode (CodeableConcept). Bind code elements to NCPDP-defined value sets per FHIR-26718.

**Rationale:** These data elements are universally required on retail pharmacy claims per NCPDP standards and are currently awkwardly represented as `supportingInfo` slices. Promoting them to backbone elements improves discoverability, enables proper validation, and aligns with the precedent set by the `total` backbone addition in R4. The 18-0-5 vote in 2020 showed consensus for this direction ("consider in next FHIR version"), and the re-opening post-STU2 signals readiness. The companion ticket FHIR-26718 for NCPDP code systems has already been resolved.

**Pros:**
- Establishes proper structural representation for pharmacy claims in FHIR Core
- Improves validation, discoverability, and interoperability
- Follows established precedent (R4 `total` backbone)
- Strong prior vote support (18-0-5)

**Cons:**
- Adds complexity to three already large resources
- Requires migration effort for existing IGs using `supportingInfo` slices
- Pharmacy-specific elements in a general resource may not be universally welcomed

---

### Disposition B: Persuasive with Modification — Add as EOB-only backbone, defer Claim/ClaimResponse

**Action:** Add the pharmacy data elements as a backbone element on ExplanationOfBenefit only (the primary consumer-facing resource). Defer Claim and ClaimResponse changes to a subsequent ballot cycle pending implementation experience.

**Rationale:** The primary use case (CARIN Blue Button) is consumer-facing EOB access. Starting with EOB reduces the blast radius while validating the structural approach. If successful, the pattern can extend to Claim and ClaimResponse in a future cycle. This is a pragmatic middle ground that delivers the most impactful change first.

**Pros:**
- Smaller, more manageable change
- Focuses on the highest-value resource (EOB)
- Allows learning from implementation before expanding

**Cons:**
- Inconsistency across Claim/ClaimResponse/EOB
- May require rework when extending to the other resources
- Does not fully address the submitter's request

---

### Disposition C: Not Persuasive — Maintain supportingInfo approach

**Action:** Close as not persuasive. The current `supportingInfo` mechanism is adequate for pharmacy-specific data elements and can be constrained via IG profiling.

**Rationale:** The `supportingInfo` backbone with category-based slicing was specifically designed as an extensible mechanism for domain-specific data. Adding pharmacy-specific elements directly to core resources sets a precedent for other domains (dental, vision, etc.) to request similar treatment, potentially leading to resource bloat. IGs like CARIN Blue Button can and do effectively profile `supportingInfo` for these elements.

**Pros:**
- Avoids resource bloat
- No migration required for existing implementations
- `supportingInfo` was designed for this purpose

**Cons:**
- `supportingInfo` is less discoverable and harder to validate than backbone elements
- Pharmacy claims are a major use case that arguably warrants first-class support
- Goes against the prior 18-0-5 vote direction
- Does not address the consistent implementation feedback about `supportingInfo` limitations

---

## Recommendation

**Disposition A: Persuasive** is recommended.

The case for promoting these NCPDP data elements to backbone elements is strong:
- Prior vote of 18-0-5 already indicated consensus for this direction
- The ticket was deliberately re-opened post-STU2 for action, not just review
- The companion terminology ticket (FHIR-26718) has already been resolved, laying the groundwork
- The pattern of promoting frequently-used `supportingInfo` items to backbone elements has precedent in R4's `total` backbone
- Retail pharmacy claims are a major segment of the claims ecosystem, and these five elements are universally required per NCPDP standards
- This should be coordinated with companion tickets FHIR-26710 (NUBC) and FHIR-26720 (Claim Received Date) for a consistent approach across claim types

The workgroup should decide whether to add elements individually or as a grouped `pharmacyDetail` backbone, and should coordinate with NCPDP on value set bindings.
