## Description

The submitter notes that the `purchasedService` concept in the Care Team Role code system does not clearly represent a care team role compared to other concepts in the code system. The original definition—"A purchased service occurs when one provider purchases a service from another provider and then provides it to the patient, e.g. a diagnostic exam"—describes a service transaction rather than a provider role. The submitter requests either removing the concept from the code system or redefining it to clarify why it represents a care team role.

## Resolution Description

The resolution states that this concept was better defined in the HL7 Terminology (THO) code system. The ticket directs to the THO CodeSystem at:
<https://terminology.hl7.org/CodeSystem-claimcareteamrole.html>

The proposed updated definition for `purchasedService` is: **"The provider from which a service was purchased by another provider."** This reframes the concept from describing a transaction to identifying a provider role on the care team.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Accept Updated Definition via THO

**Action:** Accept the current resolution. Update the `purchasedService` concept definition to **"The provider from which a service was purchased by another provider"** and align with the THO CodeSystem at `https://terminology.hl7.org/CodeSystem-claimcareteamrole.html`.

**Rationale:** The original definition described a transaction, not a role. The revised definition reframes `purchasedService` as a provider role (the provider from whom the service was purchased), which is consistent with how other concepts in the CareTeamRole code system identify functional roles on a claim. THO has already published this improved definition. The prior block vote (13-0-0) endorsed moving to THO, and reopening was procedural (post-STU2 publication cleanup), not a substantive objection.

**Impact:** Low. The code concept is retained with a clearer definition. Existing implementations using `purchasedService` are unaffected—only the human-readable definition text changes. The CodeSystem URL aligns with THO.

### Disposition B: Persuasive — Remove the `purchasedService` Concept

**Action:** Remove the `purchasedService` concept from the CareTeamRole code system entirely, as it does not represent a care team role.

**Rationale:** Even with a revised definition, `purchasedService` describes a commercial relationship between providers rather than a functional clinical or administrative role on the care team. Other concepts (attending, rendering, referring, supervising, etc.) clearly identify a provider's function in delivering care. Removing the concept would improve code system coherence.

**Impact:** Medium-High. Any implementations currently using `purchasedService` as a `careTeam.role` value would break. This could affect payer systems that track purchased services. A deprecation period or migration guidance would be needed. This is a breaking change for CARIN BB implementations.

### Disposition C: Not Persuasive — Retain Original Definition

**Action:** Close with no change. The original definition is adequate in context.

**Rationale:** The `purchasedService` concept has existed since the CareTeamRole code system was defined and is used in production claim data. The definition, while describing a transaction, implicitly identifies the care team member's role as the purchased-service provider. Changing definitions mid-cycle risks confusion.

**Impact:** None operationally, but leaves the submitter's concern about definitional clarity unaddressed.

## Recommendation

**Disposition A: Persuasive with Modification — Accept Updated Definition via THO**

This is the strongest option because it directly addresses the submitter's concern (unclear definition) without breaking existing implementations. The revised definition—"The provider from which a service was purchased by another provider"—correctly reframes the concept as a provider role rather than a transaction description. THO has already published this improvement, the prior block vote endorsed the THO migration unanimously (13-0-0), and the reopening was procedural rather than substantive. Disposition B (removal) would be a breaking change with no clear benefit over a definition fix, and Disposition C leaves a legitimate terminology quality issue unresolved.
