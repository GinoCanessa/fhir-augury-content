## Description

The ticket requests that Provider Network Contracting Status be defined as a data element within the adjudication section of the ExplanationOfBenefit (EOB), Claim, and ClaimResponse resources in FHIR Core, rather than as `supportingInfo`.

**Current State:** The CARIN Blue Button IG defines Provider Network Contracting Status for various provider types (billing, attending, referring, performing, PCP, site) as `supportingInfo` elements.

**Proposed Change:** Move these data elements into the adjudication section of the EOB resource. The contracting status indicates whether a provider has a contract with the payer plan (regardless of network) as of the effective date of service or admission, with values: `contracted` | `non-contracted`.

## Analysis

### Context
This ticket is part of a family of change requests (FHIR-26710, FHIR-26717, FHIR-26719–26723) submitted in April 2020 by the CARIN Blue Button team to promote various EOB data elements from `supportingInfo` into first-class backbone elements or adjudication categories in FHIR Core. The original resolution in July 2020 was "Considered for Future Use" (vote 18-0-5), indicating broad agreement with the direction but deferral to a future FHIR version. It was re-opened in November 2022 post-STU2 publication.

### Key Observations
1. **IG-level work already done:** FHIR-27002 (resolved as Persuasive with Modification in the CARIN BB IG) created a code system that includes `contracted` | `non-contracted` codes and a "Payer Provider Contracting Status" value set. This demonstrates the concept has been vetted and implemented at the IG level.
2. **Scope distinction:** This ticket asks for the change in FHIR Core (the base spec), not just the IG. The comment from corey_spears (2025-10-30) confirms this is intended as a FHIR Core request.
3. **Pattern consistency:** Multiple sibling tickets follow the same pattern of moving data from `supportingInfo` to adjudication/backbone elements. Disposition of this ticket should be consistent with the approach taken for the family.
4. **Staleness risk:** The ticket is over 5 years old and has been through one deferral cycle. The EOB resource may have evolved since the original submission.

## Proposed Dispositions

### Disposition A: Persuasive — Accept as Core Adjudication Element
**Action:** Add a provider network contracting status category to the adjudication section of EOB (and optionally ClaimResponse) in FHIR Core, with a value set containing `contracted` | `non-contracted`, aligning with the code system already defined in the CARIN BB IG (FHIR-27002).

**Rationale:** The concept has been proven in the CARIN BB IG, the original vote showed strong support (18-0-5), and promoting it to FHIR Core would reduce the need for IG-specific extensions. This is consistent with the overall direction of making the EOB resource richer in the base spec. The comment from 2025 reaffirms it as a FHIR Core request.

**Risk:** Adds complexity to the base EOB resource. May need coordination with how the CARIN BB IG currently handles this (to avoid breaking existing implementations).

### Disposition B: Persuasive with Modification — Define as Extension in Core, Not Backbone
**Action:** Instead of adding contracting status directly into the adjudication backbone, define a standard FHIR Core extension for provider contracting status that can be used on the adjudication element. Also provide a recommended value set (`contracted` | `non-contracted`).

**Rationale:** This preserves the base resource structure while giving a standardized, discoverable mechanism for implementers. It allows IGs (like CARIN BB) to require the extension while keeping the core resource stable for communities that don't need this concept.

**Risk:** Extensions are less visible and may be ignored by generic consumers. Doesn't fully address the original ask of making this a first-class adjudication element.

### Disposition C: Not Persuasive — Defer to IG-Level Solutions
**Action:** Close the ticket as not persuasive for FHIR Core. The CARIN BB IG (via FHIR-27002) already defines a code system and value set for provider contracting status. Other IGs can adopt the same approach.

**Rationale:** The base EOB resource serves a broad international audience, and provider network contracting status is primarily a US payer concept. Adding US-centric adjudication categories to the core spec may not be appropriate. The IG-level solution is working and does not require core changes.

**Risk:** Fragments the ecosystem — different IGs may define contracting status differently without a core anchor. Does not address the stated goal of FHIR Core enrichment.

---

## Recommendation

**Disposition A: Persuasive — Accept as Core Adjudication Element**

The concept has been validated through the CARIN BB IG (FHIR-27002), the original workgroup vote was strongly supportive (18-0-5), and the 2025 comment explicitly reaffirms this as a FHIR Core request. Provider contracting status is a fundamental adjudication concept that affects benefit calculation globally, not only in the US. Promoting it from `supportingInfo` to a proper adjudication category in the core resource would improve interoperability and reduce extension proliferation. This should be coordinated with the broader family of sibling tickets (FHIR-26710, FHIR-26720–26722) to ensure a consistent approach to EOB resource enrichment.
