## Description

The submitter requests adding a new code **`replacedby`** to the Related Claim Relationship code system (`ex-relatedclaimrelationship`). The proposed code:

| Code | Display | Definition |
|------|---------|------------|
| `replacedby` | Replaced by claim number | The current claim has been adjusted; i.e., replaced by or merged to another claim number. |

The stated use case is that payers use this relationship in their adjudication systems to indicate that a claim has been replaced by or merged into another claim.

## Analysis

The core question is whether a `replacedby` code is needed in addition to the existing `prior` code. These represent **opposite directions** of the same relationship:

- **`prior`**: Used on the *new/replacing* claim pointing back to the *old* claim it replaces.
- **`replacedby`**: Would be used on the *old/original* claim pointing forward to the *new* claim that replaced it.

FHIR methodology generally favors the `prior` direction (newer resource references older) to avoid circular references. However, payer adjudication systems commonly track both directions, and there are real-world use cases (e.g., M3P pharmacy claims) where the reverse pointer is needed.

Critically, **FHIR-27119 already resolved this same request** as "Persuasive with Modification" and is tagged `FM-R5+_ToBeApplied`, indicating the change was approved but not yet applied.

This ticket requests the identical change against the same specification. Close FHIR-31035 as a duplicate of FHIR-27119 and ensure FHIR-27119 is applied in the next appropriate release cycle.

**Applied Change:** None to this ticket. Ensure FHIR-27119 is tracked for application.

### Disposition B: Persuasive (Apply Directly)
**Rationale:** Rather than treating this as a duplicate, use this ticket as the vehicle to apply the change if FHIR-27119 has stalled. Add code `replacedby` (Display: "Replaced by claim number"; Definition: "The current claim has been adjusted; i.e., replaced by or merged to another claim number.") to the `ex-relatedclaimrelationship` code system. The business need is well-established across CARIN Blue Button, Da Vinci, and Medicare M3P implementations.

**Applied Change:** Add `replacedby` code to `http://terminology.hl7.org/CodeSystem/ex-relatedclaimrelationship` and update the `related-claim-relationship` value set.

### Disposition C: Defer Pending Reporter Input
**Rationale:** Per pknapp's January 2025 comment, Patricia Taylor has not yet confirmed whether the existing `prior` code fulfills the business intent. Defer until the reporter presents a concrete use case to the FM work group demonstrating why both `prior` and `replacedby` are needed, especially given FHIR's guidance against circular references.

**Applied Change:** None. Return to Triaged status pending reporter follow-up.
