## Summary

The reporter asks for guidance on how to retrieve inactive or terminated provider data records via the Bulk FHIR API, referencing "PDex PG 1.1.0." They load data incrementally into a FHIR store on a daily basis and expose a Bulk Export API. The challenge is communicating records that have become inactive or terminated to consumers performing incremental bulk exports. The resolution notes that this question should be directed to Plan-Net if it concerns Provider Data. A comment from ekivemark also notes that PDex did not publish a version 1.1.0, questioning which IG is actually being referenced.

## Details

### Description

The reporter's organization loads data into a FHIR store incrementally on a daily basis and exposes a Bulk Export feature. They need to return details about inactive/terminated records to consumers performing bulk exports per HL7 specs. The question asks: "What is the recommendation to pull Inactive Records through BULK FHIR APIs based on current PDEX PG 1.1.0 version Guidelines?"

This is a near-duplicate filed by the same reporter on the same date, but as a Change Request against the Plan-Net specification rather than PDex. It includes substantially more discussion:
- The active flag of `true` is intentionally required on all Plan-Net profiles.
- `OrganizationAffiliation.period` can indicate when a record is no longer active, but `Organization` and `Practitioner` profiles lack a `period` element and may need extensions.
- The Subscriptions Backport IG could help track deletes.
- The National Directory of Healthcare Providers (NDH) has a [proposal for a "deletions" bundle](https://hl7.org/fhir/us/ndh/2023Sep/exchange-ig.html#ndh-exchange-bulk-data) to handle this scenario.
- Resolution was "Not Persuasive with Modification" — the WG acknowledged the concern but determined existing/planned mechanisms address it.

## Proposed Dispositions

### Disposition A: Accept (Considered – Question Answered)

Affirm the existing resolution. The question was filed against PDex, but the actual concern relates to Provider Data managed under the Plan-Net IG. The companion ticket FHIR-45300 (filed against Plan-Net by the same reporter) received substantive review and was resolved as "Not Persuasive with Modification." The question has been answered: inactive/terminated provider data handling is a Plan-Net concern, and mechanisms such as `OrganizationAffiliation.period`, the Subscriptions Backport IG, and the NDH "deletions" bundle approach are the recommended paths. No further action is needed on this question ticket.

### Disposition B: Redirect to Plan-Net with Cross-Reference

Accept the resolution but add an explicit cross-reference to FHIR-45300 in the resolution description so that future readers can find the detailed discussion. Update the resolution text to: "This question pertains to Provider Data and should be addressed under the Plan-Net IG. See FHIR-45300 for the detailed discussion and resolution regarding inactive/terminated records in Bulk Export." This improves traceability without reopening the question.

### Disposition C: Decline (Not Persuasive)

Mark as "Not Persuasive" on the grounds that the question references a non-existent version ("PDex PG 1.1.0"), is filed against the wrong specification, and is a duplicate of FHIR-45300 which was already fully addressed. No guidance beyond what was provided in FHIR-45300 is warranted.

## Recommendation

**Disposition B: Redirect to Plan-Net with Cross-Reference.**

The existing resolution is substantively correct — this is a Plan-Net concern, not a PDex concern — but it lacks a pointer to the companion ticket FHIR-45300 where the real discussion and resolution occurred. Adding an explicit cross-reference improves the findability of the detailed guidance for future implementers encountering the same question. This is a low-effort improvement that meaningfully enhances the ticket's usefulness as a reference.
