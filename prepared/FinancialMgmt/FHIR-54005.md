## Summary

This ticket requests that the Financial Management (FM) work group standardize the spelling of "payer" (over the alternative "payor") across all FM-owned content in the FHIR specification. FM has previously adopted "payer" as the canonical spelling, and this change request aims to enforce that decision consistently throughout FM resources including Coverage, CoverageEligibilityRequest, CoverageEligibilityResponse, EnrollmentRequest, PaymentNotice, and PaymentReconciliation.

This is part of a broader cross-workgroup effort led by Chris Cioffi to standardize "payer" spelling throughout all of FHIR R6. Companion tickets have been filed for other work groups (FHIR-54020 for Patient Empowerment/FHIR Mgmt Group, FHIR-54022 for FHIR Infrastructure), and an earlier, broader ticket (FHIR-53190) was also submitted.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

**Resolution:** Persuasive
**Action:** Accept the change request and update all FM-owned FHIR content to use "payer" consistently instead of "payor."

**Rationale:** FM has already formally standardized on "payer" as the preferred spelling. This is a straightforward editorial consistency fix. The companion ticket FHIR-54020 (PE/FHIR Mgmt Group) has already been resolved as Persuasive, establishing precedent for this change across work groups. "Payer" is the more widely used modern English spelling and aligns with US healthcare industry conventions (e.g., CMS uses "payer"). Making this consistent reduces confusion for implementers. The affected resources (Coverage, CoverageEligibilityRequest, CoverageEligibilityResponse, EnrollmentRequest, PaymentNotice, PaymentReconciliation) should be reviewed for occurrences of "payor" in narrative text, descriptions, and definitions — noting that element names and code values may require separate backward-compatibility considerations.

### Disposition B: Accept as Technical Correction (Persuasive with Modification)

**Resolution:** Persuasive with Modification
**Action:** Reclassify as a Technical Correction (consistent with FHIR-54020 and FHIR-54022) and apply the spelling change only to narrative/descriptive text in FM-owned resources, explicitly excluding any changes to element names, code values, or value set codes to avoid breaking changes.

**Rationale:** While the intent is correct, the ticket is filed as a Change Request with Highest priority, which may overstate the scope. The companion tickets FHIR-54020 and FHIR-54022 were both filed as Technical Corrections at Medium priority. Reclassifying to align with the sibling tickets ensures consistent handling. Limiting the change to narrative text avoids any risk of backward-incompatible changes to structured data elements. Additionally, this ticket may partially overlap with FHIR-53190 (also FM, Submitted); the relationship should be clarified — either link them or close the earlier ticket as a duplicate.

### Disposition C: Decline (Not Persuasive)

**Resolution:** Not Persuasive
**Action:** Decline the change request.

**Rationale:** Both "payer" and "payor" are accepted English spellings, and "payor" has legal/financial usage precedent. Changing established terminology risks breaking implementations that search or match on specific text strings. The effort to audit and update all FM content for a purely cosmetic spelling preference may not justify the risk, especially if element names or code system displays are involved. Furthermore, this ticket appears to duplicate FHIR-53190 which is already submitted to the same work group; processing both creates unnecessary overhead. If the change is desired, it should be consolidated into a single ticket.

## Recommendation

**Disposition A: Accept (Persuasive)** is recommended.

FM has already made the policy decision to standardize on "payer." The companion ticket FHIR-54020 has been accepted as Persuasive by the FHIR Mgmt Group, establishing clear cross-workgroup precedent. This is a low-risk editorial consistency change for FM's own content. The work group should ensure the scope is limited to narrative and descriptive text, flag any element-name or code-value implications for separate consideration, and clarify the relationship with the earlier ticket FHIR-53190 (either link or close as duplicate).
