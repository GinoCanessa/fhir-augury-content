## Summary

This change request asks the PCT IG to add an **optional synchronous response mode** to the `gfe-submit` operation, allowing payers to return the AEOB Packet directly in the operation response rather than requiring the current asynchronous polling workflow. The motivation comes from PCT Pilot experience: when payers can adjudicate GFEs in near real-time, a synchronous path would let providers and patients receive cost estimates before the patient leaves the office or hospital.

The ticket was previously resolved as **"Considered for Future Use"** during the Block-2025-Aug-2 vote (10-0-2, Corey Spears / Chris Cioffi) on 2025-08-26, with the note that it "will need to be reviewed and designed for a future version" and that "some testing would also be good to do before inclusion in the specification." However, this resolution was **reverted post STU2 publication** by Lynn Laakso on 2025-10-08, and the ticket was re-opened for further consideration.

## Details

The PCT IG currently defines only an asynchronous workflow for the `gfe-submit` operation (see [Payer Perspective End-to-End Workflow](https://build.fhir.org/ig/HL7/davinci-pct/gfe_submission_and_aeob_overview.html#payer-perspective-end-to-end-workflow), step 5). In this workflow:

1. A provider submits a [GFE Packet](https://build.fhir.org/ig/HL7/davinci-pct/StructureDefinition-davinci-pct-gfe-packet.html) via the `gfe-submit` operation.
2. The payer begins asynchronous processing and returns an async response.
3. The submitter polls for results until the AEOB is ready.

The existing IG text already acknowledges that "some payers may process GFEs in near real time, which this guide can support," but does not define how a synchronous response would work.

This ticket proposes adding an **optional** synchronous path where:
- The payer returns the AEOB Packet directly in the operation outcome (or response body) when processing completes quickly enough.
- The asynchronous path remains the default for payers that require longer processing times.

### Context from Prior Vote

The previous "Considered for Future Use" resolution indicated workgroup consensus that the concept has merit but needs further design and pilot testing before being included in the specification. The re-opening post-STU2 suggests the workgroup wants to revisit this for a future ballot cycle.

### Relationship to FHIR-49778

[FHIR-49778](https://jira.hl7.org/browse/FHIR-49778) (Published, Persuasive with Modification) already introduced a synchronous error-reporting mechanism into the `gfe-submit` operation response. That ticket addressed the problem of business-requirement errors being delayed until the polling step. The pattern established in FHIR-49778 — returning synchronous content in the `gfe-submit` response — could serve as a foundation for this ticket's broader request to return the full AEOB Packet synchronously.

## Proposed Dispositions

### Disposition A: Considered for Future Use (Re-affirm Prior Resolution)

Re-affirm the previous resolution of "Considered for Future Use." The concept of synchronous AEOB responses has merit and is supported by pilot experience, but requires additional design work and testing before inclusion.

**Rationale:** The original vote (10-0-2) reflected workgroup consensus that this feature is not ready for the current specification version. Key open questions remain:
- How to signal that the client supports/prefers synchronous responses (e.g., via the `Prefer: respond-sync` header or an operation parameter).
- How to define the synchronous response payload (full AEOB Packet Bundle vs. an OperationOutcome containing a reference).
- Timeout and fallback semantics — what happens when payer processing exceeds a synchronous window.
- Interplay with the async polling mechanism already defined.

This defers the work to a future ballot cycle with an expectation that pilot implementations will inform the design.

### Disposition B: Persuasive — Define Optional Synchronous Response Mode

Accept the request and define an optional synchronous response path in the `gfe-submit` operation for the next ballot.

**Proposed changes:**
1. Add an optional input parameter `prefer-sync` (boolean) to the `gfe-submit` OperationDefinition, allowing clients to indicate preference for a synchronous response.
2. Define that when `prefer-sync` is true and the payer can produce the AEOB within a reasonable timeframe, the payer MAY return the AEOB Packet Bundle directly in the `gfe-submit` response (HTTP 200) instead of initiating the async flow (HTTP 202).
3. If the payer cannot fulfill the synchronous request, it MUST fall back to the existing async workflow (HTTP 202) — making this backward-compatible.
4. Add guidance on the `gfe_submission_and_aeob_overview` page describing the synchronous option, expected behavior, and timeout considerations.
5. Leverage the synchronous response pattern already established by FHIR-49778 for error reporting.

### Disposition C: Not Persuasive

Decline the request. The existing asynchronous workflow is sufficient and aligns with the regulatory expectation that AEOB processing may take a substantial amount of time. Adding a synchronous path increases implementation complexity for both payers and providers without a clear regulatory mandate.

**Rationale:** The IG already acknowledges near-real-time processing is possible within the async framework. Payers that process quickly will simply have their async result available sooner for polling. A synchronous mode adds optional complexity that may lead to inconsistent implementations and interoperability challenges across payers with different processing capabilities.

## Recommendation

**Disposition A: Considered for Future Use** is recommended.

The workgroup previously voted 10-0-2 for this resolution, indicating strong consensus. While the concept is valuable and pilot feedback supports it, the ticket was re-opened post-STU2 to keep it on the radar — not because the design is ready. The synchronous response pattern introduced by FHIR-49778 for error reporting provides a useful foundation, but extending it to full AEOB response requires careful design of client signaling, timeout semantics, and fallback behavior. Deferring to a future version allows time for additional pilot testing and a well-designed proposal, while the re-opened status ensures the work is tracked and prioritized.
