## Description

The submitter asks how the Medicaid ID **identifier type** should be coded in FHIR. Three alternatives are posed:

1. Use identifier type code **"SB"** with LOINC ID **45400-9**
2. Designate the Medicaid ID via the **Identifier.system** URI
3. Use a **separate identifier system** analogous to Medicare's OID `2.16.840.1.113883.4.927` (from the [FHIR Identifier Registry](https://www.hl7.org/fhir/identifier-registry.html))

The submitter further notes that Medicaid IDs are **state-specific** — each U.S. state issues its own Medicaid numbers — so any solution must accommodate per-state differentiation, similar to how driver's license identifiers are handled in the Identifier Registry.

## Related Zulip Conversations

### 1. "Medicaid Identifier Type" — implementers stream (Feb 2021)

- **URL:** https://chat.fhir.org/#narrow/stream/implementers/topic/Medicaid%20Identifier%20Type
- **Participants:** Kimberly Kenna, Lloyd McKenzie, Eric Haas, Richard Townley-O'Neill
- **Key points:**
  - **Eric Haas:** US Core is silent on `Patient.identifier.type`; it requires both a `value` and a `system`. The system URI tells you what the identifier is — e.g., `http://hl7.org/fhir/sid/us-ssn` for SSN. Questions whether `.type` adds value when `.system` is present.
  - **Lloyd McKenzie:** `.type` is useful in two contexts: (1) when system is omitted, type gives a clue (e.g., `type.code` = "provider license number"); (2) when system is present, `type.text` provides a human-readable label (e.g., "Alaska Driver's License") — critical for UI display.
  - **Implication for FHIR-23045:** The system URI is the primary computational identifier; `.type` is supplemental for human readability.

### 2. "Medicare & Medicaid OIDs" — implementers stream (Oct 2021)

## Proposed Dispositions

### Disposition A: Provide Guidance and Request NamingSystem Definitions *(Persuasive with Modification)*

**Rationale:** The question is valid and reflects a real interoperability gap. Medicaid identifiers are state-specific and have no registered system URIs.

**Action:**
1. Answer the question: Medicaid IDs should be identified primarily by `Identifier.system` (a state-specific URI), not by LOINC codes. The identifier type can supplement with `type.text` for display (e.g., "Pennsylvania Medicaid Number").
2. File a request with the HL7 Terminology Authority (HTA) to define state-specific Medicaid NamingSystems following the pattern used for driver's licenses (e.g., `http://hl7.org/fhir/sid/us-medicaid-{state}`).
3. Update the eLTSS IG mapping guidance to reflect this approach and remove any suggestion to use LOINC for identifier type coding.
4. Close FHIR-23045 once the HTA request is filed and eLTSS guidance is updated.

**Pros:** Provides a definitive answer, addresses the root gap, and establishes a reusable pattern for all IGs.
**Cons:** Requires HTA coordination and 50-state NamingSystem definitions — significant effort.

### Disposition B: Answer the Question with Existing Guidance *(Not Persuasive with Modification)*

**Rationale:** Sufficient guidance already exists in the FHIR spec and community discussions to answer this question without new infrastructure.

**Action:**
1. Answer: Use `Identifier.system` with a locally defined or state-managed URI for the Medicaid ID. Use `Identifier.type` with an appropriate code from the [v2-0203 code system](http://terminology.hl7.org/CodeSystem/v2-0203) (e.g., `MA` for Patient Medicaid Number or `MC` for Patient's Medicare Number as a parallel). Use `type.text` for human-readable state context.
2. Note that there is no HL7-registered NamingSystem for state Medicaid IDs and recommend implementers coordinate with their state Medicaid agency for the appropriate system URI.
3. Close the ticket with this guidance in the resolution comments.

**Pros:** Immediately actionable, no dependency on HTA or NamingSystem creation.
**Cons:** Does not resolve the underlying gap. Different implementers may adopt different state URI patterns, fragmenting interoperability.

### Disposition C: Redirect to HTA / US Realm *(Not Persuasive)*

**Rationale:** This is fundamentally a terminology/identifier-registry question, not a Financial Management or eLTSS question. The FM WG does not own the Identifier Registry or NamingSystem definitions.

**Action:**
1. Acknowledge the question is valid but out of scope for FM WG disposition.
2. Redirect the submitter to the HL7 Terminology Authority (HTA) and/or the US Realm steering committee to request state-specific Medicaid NamingSystems.
3. Close FHIR-23045 as redirected, with a pointer to the relevant resources (identifier registry, THO, existing Medicare NamingSystem as a model).

**Pros:** Directs the question to the appropriate authority.
**Cons:** The ticket has already been open for 6+ years and bounced between groups; another redirect risks further inaction. Does not provide the submitter with any immediate practical guidance.

## Recommendation

**Disposition B: Answer the Question with Existing Guidance *(Not Persuasive with Modification)***

This ticket has been open since August 2019 and has bounced from the HSS WG to FM without resolution. Community discussions (Zulip threads from 2021) confirm that Medicaid identifiers are universally understood to be state-specific, that no HL7-registered NamingSystem exists for them, and that `Identifier.system` is the correct computational mechanism (not LOINC codes).

The FM WG should close this with a concrete answer:
- **Do not use LOINC** for identifier type coding (FHIR-20887 already challenged this language).
- **Use `Identifier.system`** with a state-specific URI (implementer-defined until HTA registers standard ones).
- **Use `Identifier.type`** with an appropriate v2-0203 code (e.g., `MA`) and `type.text` for human-readable state context (per Lloyd McKenzie's guidance).
- **Optionally** file a separate HTA request for standardized state Medicaid NamingSystems, but do not block this ticket's closure on that.

This balances providing immediate, practical guidance to the submitter while acknowledging that formal standardization of state Medicaid identifier systems is a broader effort beyond this ticket's scope.
