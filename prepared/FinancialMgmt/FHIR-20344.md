## Description

The reporter proposes that `Contract.signer` should be simplified to directly use the `Signature` datatype rather than its current backbone structure. Currently, `Contract.signer` contains:

- `Contract.signer.type` — duplicates `Signature.type`
- `Contract.signer.party` — duplicates `Signature.who`
- `Contract.signer.signature` — the actual `Signature` datatype

The argument is that since the `Signature` datatype has matured and is now stable (including `Signature.type`, `Signature.who`, and `Signature.onBehalfOf`), the extra wrapper elements `type` and `party` are redundant leftovers from an earlier version of the `Signature` datatype. The proposal would flatten the structure to:

```
Contract.signature   0..*   Signature
```

## Analysis

### Arguments For Simplification
1. **Redundancy**: `Contract.signer.type` and `Contract.signer.party` overlap semantically with `Signature.type` and `Signature.who`, respectively.
2. **Signature maturity**: The Signature datatype has been enhanced (e.g., addition of `onBehalfOf` in FHIR-9563) and is now considered stable, capable of representing signer identity, type, and delegation without wrapper elements.
3. **Implementer confusion**: The current structure forces implementers to populate both wrapper and Signature sub-elements with potentially conflicting data (e.g., `signer.party` vs `signature.who`).
4. **Consistency**: Other FHIR resources that use signatures do not wrap them in a backbone with duplicative identity/type fields.

### Arguments Against Simplification
1. **Semantic distinction**: FHIR-9567 established detailed semantics for the signer backbone, distinguishing between the "signer" role (legal standing, type of participation) and the "signature" act. The `Contract.signer.type` uses `ContractSignerTypeCodes` which is explicitly noted as "related but different" from the Signature type.
2. **Breaking change**: This would be a significant structural change to a resource that has been through multiple ballot cycles. Existing implementations, profiles (e.g., eLTSS — FHIR-20896, FHIR-20897), and tooling depend on the current structure.
3. **Value set orphaning**: The `ContractSignerTypeCodes` value set and code system would need to be retired or re-mapped to `Signature.type`, which may not be semantically appropriate.
4. **Multiple signers per contract**: The current backbone allows associating a specific party and their role with one or more signatures. Flattening to `Contract.signature` would lose the grouping of "this party in this role produced these signatures."

## Proposed Dispositions

### Disposition A: Persuasive — Simplify to Signature Datatype
Accept the proposal and replace `Contract.signer` backbone with `Contract.signature 0..* Signature`. Retire or deprecate the `ContractSignerTypeCodes` value set. Map `signer.type` semantics into `Signature.type` bindings specific to Contract context. Document migration guidance for implementers.

- **Pros**: Eliminates redundancy, simplifies the model, aligns with Signature datatype maturity.
- **Cons**: Major breaking change, orphans existing value sets, loses explicit signer-to-signature grouping, invalidates existing profiles.

### Disposition B: Not Persuasive — Retain Current Structure
Reject the proposal. Acknowledge that while the elements may appear duplicative, the Contract.signer backbone serves a distinct purpose: associating a legal party and their contractual role (via `signer.type`) with one or more digital signatures. The `ContractSignerTypeCodes` captures contract-specific signer roles that `Signature.type` does not address. Add clarifying documentation to explain the distinction between `signer.type`/`signer.party` and `signature.type`/`signature.who`.

- **Pros**: No breaking change, preserves existing semantics, protects downstream profiles and implementations.
- **Cons**: Does not address the redundancy concern, continues to confuse implementers about the difference between signer-level and signature-level elements.

### Disposition C: Persuasive with Modification — Add Guidance Without Structural Change
Keep the current `Contract.signer` structure but add documentation that:
1. Clarifies the semantic distinction between `signer.type` (contractual role) and `signature.type` (digital signature mechanism).
2. Clarifies the relationship between `signer.party` (legal party) and `signature.who` (signing entity/device).
3. Adds invariants or guidance requiring consistency between `signer.party` and `signature.who` / `signature.onBehalfOf`.
4. Considers deprecating `signer.type` and `signer.party` for a future major version if the community agrees they are truly redundant.

- **Pros**: Addresses implementer confusion without breaking changes, establishes a path toward future simplification, preserves backward compatibility.
- **Cons**: Does not fully resolve the redundancy, defers the structural question.

## Recommendation

**Disposition C: Persuasive with Modification** — The redundancy concern raised by the reporter is valid, and the maturation of the Signature datatype (especially the addition of `onBehalfOf`) has reduced the need for the wrapper elements. However, the current `Contract.signer` backbone has been refined through multiple ballot cycles (FHIR-9533, FHIR-9567) and carries distinct semantics around contractual legal standing and signer roles that go beyond what the Signature datatype captures. A full structural change would be a breaking change affecting profiles like eLTSS and would orphan the `ContractSignerTypeCodes` value set. The best path is to clarify the existing distinctions in documentation, add consistency guidance between the signer and signature sub-elements, and consider deprecation of the duplicative elements in a future major release when the breaking change can be properly managed.
