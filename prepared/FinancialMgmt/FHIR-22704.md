## Description

The reporter questions why the Contract resource's state model does not map to the standard FHIR state model pattern that all FHIR resources are expected to follow. The concern is that `Contract.status` (with its required binding) deviates from the common workflow patterns used across other FHIR resources.

## Keywords

Contract, state model, workflow pattern, Contract.status, Contract.state, Contract.legalState, required binding, event pattern, state machine

## Comments Summary

**costateixeira (2019-06-15):** Agrees with the concern. Distinguishes between `Contract.legalState` (extensible binding, which seems reasonable) and `Contract.status` (required binding, which is problematic). Raises several points:
- Questions whether the Contract state diagram is supposed to be normative, noting it could render the resource unusable in some jurisdictions.
- Suggests **not** assuming a fixed set of states, because contract states differ significantly across countries and legal systems.
- Recommends aligning `Contract.status` with the workflow pattern so it reflects the status of the *request or action*, not the resulting legal state of the contract itself.

## Analysis

This ticket is part of a broader, long-standing cluster of issues with the Contract resource's state model. The core problem identified across FHIR-22704, FHIR-26966, FHIR-20152, and FHIR-55563 is that:

1. **Contract.status uses a required binding** that doesn't align with the standard FHIR workflow event pattern, where status should reflect the lifecycle of the resource instance (draft, active, completed, etc.), not the legal state of the contract.
2. **The resource conflates multiple concepts** — contract document, contract definition, and contract execution — leading to multiple overlapping state machines.
3. **The state diagram and value sets are inconsistent** (per FHIR-20152).
4. **International usability is compromised** because contract states vary by jurisdiction, and a required binding to a specific state set is too restrictive.

The most recent related ticket, FHIR-55563 (2026), was resolved as "Persuasive with Modification," indicating the community acknowledges the Contract resource needs significant refactoring before it can advance to normative status.

## Proposed Dispositions

### Disposition A: Persuasive — Align Contract.status with Workflow Event Pattern
**Action:** Refactor `Contract.status` to use the standard FHIR workflow event status codes (e.g., `draft | active | on-hold | revoked | completed | entered-in-error | unknown`) with a required binding to the common event status value set. Move jurisdiction-specific contract lifecycle states entirely into `Contract.legalState` with an extensible binding. Update the state diagram to reflect this separation.

**Rationale:** This directly addresses the reporter's concern and aligns with the workflow pattern requirements. It is consistent with the direction acknowledged in FHIR-26966 and FHIR-55563. The separation of workflow status from legal state preserves international flexibility while ensuring conformance with FHIR-wide patterns.

### Disposition B: Persuasive with Modification — Defer to Contract Refactoring Effort
**Action:** Acknowledge the issue as valid but defer specific changes to the broader Contract resource refactoring effort already underway (per FHIR-55563 and FHIR-26966). Mark this ticket as a sub-task or related issue to be addressed as part of that comprehensive refactoring. Add a tracker note that state model alignment must be included in the refactoring scope.

**Rationale:** Given the scale of Contract resource issues (conflated concepts, multiple state machines, inconsistent value sets), fixing the state model in isolation risks creating new inconsistencies. A coordinated approach as part of the planned refactoring ensures all related issues are addressed holistically.

### Disposition C: Not Persuasive — Contract Requires Domain-Specific State Model
**Action:** Document that the Contract resource intentionally uses a domain-specific state model that diverges from the common FHIR workflow pattern due to the unique lifecycle requirements of legal contracts. Clarify in the resource documentation how `Contract.status` and `Contract.legalState` relate to each other and to the workflow patterns.

**Rationale:** Legal contracts have inherently different lifecycle semantics than clinical events. Forcing the common workflow pattern onto Contract could oversimplify the model and lose important domain-specific distinctions. However, this position conflicts with the community consensus reflected in FHIR-55563.

## Recommendation

**Disposition B: Persuasive with Modification — Defer to Contract Refactoring Effort**

The concern raised is clearly valid and well-supported by community consensus (FHIR-26966, FHIR-55563, costateixeira's comment). However, the state model issue is deeply intertwined with the broader structural problems of the Contract resource. FHIR-55563 has already been resolved as "Persuasive with Modification," establishing that the Contract resource needs comprehensive refactoring. The most effective path is to acknowledge this ticket's validity, link it to the refactoring effort, and ensure state model alignment with the workflow event pattern is an explicit requirement in that refactoring scope. This avoids piecemeal changes while ensuring the concern is not lost.
