## Summary

The reporter (Lisa R. Nelson) states that the subject of a Contract resource should be able to reference a **PractitionerRole** — representing a provider at a specific location — rather than being limited to Organization. The use case is contracts where the contracted party is a specific provider in a specific role/location, not just an organization.

In a comment dated 2020-02-06, **pknapp** pointed out that `Contract.subject` is currently typed as `Reference(Any)`, which already permits PractitionerRole and is not limited to Organization. pknapp asked whether the actual issue is with a **sub-element** below `Contract.subject`. The reporter has not responded to this clarifying question since 2020. A follow-up invitation to discuss on the Thursday call (2021-02-18) also did not produce visible resolution.

## Analysis

The central question is whether there is actually a problem to solve. The commenter pknapp identified that `Contract.subject` is already `Reference(Any)`, which permits PractitionerRole without any change. The reporter's original concern — that "allowing only Organization to be referenced is too narrow" — appears to be based on a misunderstanding of the current constraint.

However, two possibilities remain:
1. The reporter may have been referring to a **sub-element** (e.g., `Contract.term.offer.party`, `Contract.term.action.subject`) rather than the top-level `Contract.subject`. Some sub-elements do have constrained reference types.
2. The reporter never confirmed or denied pknapp's clarification, and the invitations to discuss (2020, 2021) did not produce visible outcomes.

Given that this ticket has been open since January 2020 with no substantive progress and the core premise appears to be addressed by the current spec, the workgroup should determine whether there is a remaining real issue or whether the question has been answered.

---

## Proposed Dispositions

### Disposition A: Not Persuasive (Recommended)
**Rationale:** `Contract.subject` is already defined as `Reference(Any)` in the current specification, which permits referencing PractitionerRole, Practitioner, Organization, Location, or any other resource type. The reporter's stated concern — that the reference is limited to Organization — is not accurate for `Contract.subject`. The clarifying question from pknapp (2020-02-06) received no response, and no specific sub-element has been identified as the actual problem. Close as the question has been answered: PractitionerRole is already permitted in `Contract.subject`.

### Disposition B: Persuasive with Modification
**Rationale:** While `Contract.subject` itself is `Reference(Any)`, treat this as an opportunity to review whether any **sub-elements** of Contract (e.g., `Contract.term.offer.party.reference`, `Contract.term.action.subject.reference`) inappropriately exclude PractitionerRole. If any such sub-elements are found to be overly constrained, add PractitionerRole to their allowed reference types. This aligns with the broader effort documented in the "Adding PractitionerRole to resource references" thread.

### Disposition C: Deferred / Request for Information
**Rationale:** The ticket cannot be fully evaluated without clarification from the reporter. The original premise about `Contract.subject` appears to be incorrect, but the reporter may have intended a sub-element. Defer and request that the reporter (Lisa R. Nelson) or the named collaborator (Lenel James) specify exactly which Contract element is too narrowly constrained for their use case.

---

## Recommendation

**Disposition A: Not Persuasive** — The stated issue (`Contract.subject` limited to Organization) does not match the current specification, where `Contract.subject` is `Reference(Any)`. The clarifying question has gone unanswered for over five years. If the workgroup believes there may be a sub-element issue worth investigating, Disposition B is a reasonable alternative, but the ticket as written does not identify a specific problem requiring a change.
