## Summary

The submitter questions how closely the FHIR value sets used in `CoverageEligibilityResponse` should align with the corresponding X12 270/271 (eligibility inquiry/response) code sets. A comparison table is provided showing that FHIR value sets contain significantly fewer codes than their X12 counterparts across five key elements:

| FHIR Element                     | X12 271 Segment | FHIR Code Count | X12 271 Code Count |
|----------------------------------|-----------------|:----------------:|:-------------------:|
| `insurance.item.benefit.type`    | EB01            | 12               | 35                  |
| `insurance.item.unit`            | EB02            | 2                | 9                   |
| `insurance.item.category`        | EB03            | 28               | 100+                |
| N/A                              | EB04            | N/A              | 45                  |
| `insurance.item.term`            | EB06            | 3                | 19                  |

The implicit request is for FHIR to expand its value sets to match the X12 271 scope more closely, or to clarify the relationship between FHIR's smaller value sets and the broader X12 code sets.

---

## Details

### Description

The submitter asks: *"How closely should the value sets for CoverageEligibilityResponse follow the ones used in X12 270/271?"* They note that FHIR value sets are considerably smaller than their X12 counterparts, with specific gaps identified across five EB segments (EB01–EB06). The element EB04 (Insurance Type) has no apparent FHIR equivalent, and the FHIR `insurance.item.term` (EB06) has only 3 codes compared to X12's 19.

### Resolution Description

The resolution states: *"All of the FHIR Elements referenced (insurance.item.benefit.type, insurance.item.unit, insurance.item.term) use example or preferred codeSystems. Therefore the codeSystems can be replaced with X12 code systems if required."*

This indicates that the existing binding strengths (example/preferred) already permit implementers to substitute X12 code systems without requiring changes to the FHIR specification.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

Expand the FHIR-defined value sets for `insurance.item.benefit.type`, `insurance.item.unit`, `insurance.item.category`, and `insurance.item.term` to include additional codes that map to X12 271 EB01–EB06 segments. Add a FHIR element mapping to EB04 (Insurance Type Code) if no equivalent currently exists. This would reduce the burden on implementers performing X12-to-FHIR translations by providing out-of-the-box alignment.

**Pros:** Improves interoperability for US-based implementations that rely on X12; reduces custom mapping work; provides richer default code sets.
**Cons:** Adds US-specific X12 codes to an international standard; increases maintenance burden; may not reflect non-US eligibility workflows.

### Disposition B: Accept with Modification (Not Persuasive with Note)

Retain the current value sets and binding strengths but add explicit guidance in the CoverageEligibilityResponse documentation noting that the example/preferred bindings intentionally allow implementers to use X12 code systems (e.g., ASC X12 EB01–EB06 code lists) as replacements or supplements. Include a mapping table in the resource documentation showing the correspondence between FHIR elements and X12 271 EB segments.

**Pros:** Provides clarity without expanding core value sets; maintains international applicability; addresses the submitter's underlying concern about how to handle the gap.
**Cons:** Does not expand the default value sets; implementers must still source and bind X12 code systems themselves.

### Disposition C: Decline (Not Persuasive)

Affirm the current resolution that the existing example and preferred binding strengths already allow implementers to substitute X12 code systems where needed. The FHIR value sets are intentionally smaller as they represent an international baseline, and X12-specific codes are jurisdiction-specific. No specification change is required.

**Pros:** No spec change needed; maintains clean separation between international standard and jurisdiction-specific code systems; binding strength already supports the use case.
**Cons:** Does not improve discoverability of X12 alignment options for implementers unfamiliar with FHIR binding mechanics.

---

## Recommendation

**Disposition C: Decline (Not Persuasive)**

The existing resolution is well-founded. The FHIR elements in question (`insurance.item.benefit.type`, `insurance.item.unit`, `insurance.item.category`, `insurance.item.term`) all use **example** or **preferred** binding strengths, which by definition allow implementers to replace the bound code systems with X12 code systems (or any other code system) without violating conformance. FHIR intentionally maintains smaller, international-baseline value sets and delegates jurisdiction-specific expansions to implementation guides. The Zulip discussion on `CoverageEligibilityResponse` (Oct 2023) confirms this flexibility is already understood by implementers. Expanding core FHIR value sets to mirror X12 271 would introduce US-specific content into an international standard, which is better addressed through US-realm IGs such as CARIN or Da Vinci. The workgroup should affirm the "Not Persuasive" resolution.
