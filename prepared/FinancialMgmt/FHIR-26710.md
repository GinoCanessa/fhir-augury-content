## Description

This ticket requests that specific NUBC (National Uniform Billing Committee) data elements currently represented as `supportingInfo` slices be promoted to first-class **backbone data elements** on the **Claim**, **ClaimResponse**, and **ExplanationOfBenefits** resources in FHIR Core.

The NUBC data elements in scope are:

1. **Point of Origin for Admission or Visit**
2. **Priority (Type) of Admission or Visit**
3. **Type of Bill**
4. **Patient Discharge Status**
5. **Present on Admission**

These data elements are required on all facility (institutional) claims. The rationale is that encoding these as `supportingInfo` categories in IGs (originally CARIN Blue Button) is insufficient for elements that are fundamental to institutional claim processing. Promoting them to backbone elements in the base resource would provide clearer semantics, better validation, and improved interoperability.

## Analysis

### Core Issue

The fundamental question is whether NUBC institutional claim data elements (Point of Origin, Admission Type, Type of Bill, Discharge Status, Present on Admission) should be elevated from `supportingInfo` category slices to dedicated backbone elements in the FHIR Core Claim, ClaimResponse, and ExplanationOfBenefits resources.

### Arguments For

- **Ubiquity on institutional claims:** These NUBC elements are required on all facility/institutional claims in the US. They are not optional supplementary information — they are core structural data.
- **Precedent in IGs:** The companion ticket FHIR-26711 was resolved as Persuasive, establishing NUBC code systems as recognized external code systems. FHIR-35319 was resolved as Persuasive with Modification in the Da Vinci PCT IG, demonstrating that IGs are already treating these elements as first-class structural data.
- **Strong initial support:** The original 2020 vote was 18-0-5 in favor (deferred to future version), indicating broad agreement on the merits.
- **Clearer semantics:** Backbone elements provide explicit element names, stronger typing, and better validation compared to `supportingInfo` slicing, which relies on category codes for disambiguation.
- **Part of a cohort:** This ticket is part of a series of related requests (FHIR-26712, FHIR-26717, FHIR-26719, FHIR-26720) all seeking to promote commonly-used `supportingInfo` data to backbone elements, suggesting a systemic gap.

### Arguments Against

- **Resource complexity:** Adding multiple backbone elements to Claim/ClaimResponse/EOB increases the base resource size and complexity for all implementers, even those not processing US institutional claims.
- **Jurisdiction-specific concern:** NUBC is a US-specific standard. FHIR Core resources serve an international community, and encoding US institutional billing concepts as backbone elements may not generalize globally.
- **supportingInfo extensibility:** The `supportingInfo` mechanism was designed precisely for this use case — carrying additional claim-specific data without bloating the base resource. Profiles and IGs can constrain `supportingInfo` slices effectively.
- **Scope of change:** Five new backbone elements across three resources represents a significant structural change requiring careful design, migration guidance, and coordination with existing IGs.

## Proposed Dispositions

### Disposition A: Persuasive — Promote All Five NUBC Elements to Backbone Elements

Accept the request as stated. Add Point of Origin, Admission Type, Type of Bill, Patient Discharge Status, and Present on Admission as backbone data elements to Claim, ClaimResponse, and ExplanationOfBenefits.

**Rationale:** These elements are universally required on institutional claims and have already been recognized via FHIR-26711 (code systems) and FHIR-35319 (PCT IG profile). The 18-0-5 vote in 2020 endorsed the concept — only the timing was deferred. This improves structural clarity for the most widely-used claim type.

**Trade-off:** Increases base resource complexity and may set a precedent for additional jurisdiction-specific backbone elements.

### Disposition B: Persuasive with Modification — Promote a Subset and Use Extensions for the Rest

Promote the two most universally applicable elements — **Type of Bill** and **Patient Discharge Status** — to backbone elements, and recommend the remaining three (Point of Origin, Admission Type, Present on Admission) be modeled as standard extensions or handled via `supportingInfo` constraints in IGs.

**Rationale:** Balances the need for structural clarity on the most critical elements with the goal of keeping the base resource manageable. Type of Bill and Discharge Status are arguably the most fundamental identifiers for institutional claims.

**Trade-off:** Splits the NUBC data elements across two modeling patterns, which may cause confusion and inconsistency in IG profiles.

### Disposition C: Not Persuasive — Retain supportingInfo Approach

Decline the request. The `supportingInfo` mechanism is the intended approach for carrying these elements. IGs such as CARIN BB and Da Vinci PCT can continue to define constrained `supportingInfo` slices for these NUBC data elements.

**Rationale:** FHIR Core resources should remain jurisdiction-neutral. The `supportingInfo` pattern is working in practice (IGs are using it), and adding backbone elements for US-specific billing concepts would increase resource complexity for all implementers globally.

**Trade-off:** Perpetuates the semantic gap where important institutional claim data lacks explicit naming in the base resource, and IGs must independently re-profile the same `supportingInfo` slices.

## Recommendation

**Disposition A: Persuasive** is recommended.

The 2020 vote (18-0-5) already endorsed this change in principle, deferring only on timing. The ticket has been re-opened for the next FHIR version, the companion code system ticket (FHIR-26711) has been resolved, and the Da Vinci PCT IG (FHIR-35319) has independently addressed these same elements at the IG level — all pointing to readiness. These five NUBC elements are not niche; they are required on every institutional claim in the US, the largest single-country FHIR implementation market. Promoting them to backbone elements improves validation, discoverability, and consistency across the growing number of IGs that profile these resources. The workgroup should address the full cohort of related tickets (FHIR-26712, FHIR-26717, FHIR-26719, FHIR-26720) together to ensure a coherent approach to `supportingInfo`-to-backbone promotion.
