## Description

The submitter requests that the canonical URL for **CodeSystem-C4BBAdjudication** be moved from `hl7.org/fhir` to `terminology.hl7.org` (THO). A UTG ticket should be created to register the CodeSystem in THO. This aligns with the HL7 principle that terminology artifacts with broad community reuse value should be anchored in the Unified Terminology Governance (UTG) managed THO rather than within individual IGs or the base FHIR specification.

## Proposed Dispositions

### Disposition A: Persuasive — Migrate CodeSystem-C4BBAdjudication to THO

**Action:** Accept the request. Create a UTG ticket to register CodeSystem-C4BBAdjudication in terminology.hl7.org. Update the canonical URL in the CARIN BB IG to reference the THO-hosted CodeSystem. Coordinate the migration timeline with the next IG publication cycle.

**Rationale:** The C4BBAdjudication CodeSystem has been in use since CARIN BB STU1 and is referenced by multiple implementation guides. The original deferral (CFFU in 2022) cited the need for EoB resource model re-evaluation, but now that STU2 has been published and the ticket was explicitly re-opened for reconsideration, the CodeSystem has demonstrated sufficient maturity for THO registration. Anchoring in THO supports cross-IG interoperability and aligns with HL7 terminology governance best practices.

**Vote text:** _"Persuasive. Migrate CodeSystem-C4BBAdjudication canonical URL to terminology.hl7.org (THO). Create UTG registration ticket and update IG canonical references in the next publication cycle."_

### Disposition B: Considered for Future Use — Defer pending coordinated C4BB terminology migration

**Action:** Defer this ticket until the broader C4BB terminology migration effort is ready to proceed. Keep the CodeSystem in the IG for now but track this alongside the full set of sibling tickets (FHIR-35390 through FHIR-35399, FHIR-35410, etc.).

**Rationale:** The same reasoning that led to the 2022 CFFU may still partially apply — migrating a single CodeSystem to THO in isolation risks inconsistency with the remaining C4BB CodeSystems and ValueSets still anchored in the IG. A coordinated migration of all mature C4BB terminology artifacts would be more coherent and reduce implementer disruption. The work group may wish to define clear maturity criteria before proceeding.

**Vote text:** _"Considered for Future Use. Defer pending a coordinated plan to migrate all qualifying C4BB CodeSystems and ValueSets to THO. Track alongside sibling tickets FHIR-35390–35399, FHIR-35410."_

### Disposition C: Persuasive with Modification — Approve in principle, batch with sibling tickets

**Action:** Approve the migration in principle but require that it be executed as part of a batched effort covering all re-opened C4BB THO anchoring tickets. Assign an owner to coordinate the UTG registration process for the full set.

**Rationale:** This balances the submitter's valid request with practical implementation concerns. The C4BBAdjudication CodeSystem is mature enough for THO, but migrating it alone without its siblings would create an inconsistent terminology landscape in the IG. Batching ensures a single coordinated transition, reducing the burden on implementers who need to update their systems.

**Vote text:** _"Persuasive with Modification. Approve THO migration for CodeSystem-C4BBAdjudication as part of a batched effort with all re-opened C4BB THO anchoring tickets (FHIR-35390–35399, FHIR-35410, etc.). Assign coordinator for UTG registration."_

## Recommendation

**Disposition C: Persuasive with Modification** is recommended.

The request is fundamentally sound — C4BBAdjudication has demonstrated maturity through multiple STU cycles and should be in THO. However, the existence of 15+ sibling tickets in the same state (all re-opened post STU2) makes a batched, coordinated approach the most practical path. This avoids piecemeal migration that could confuse implementers and ensures the FM WG addresses the full scope of the THO anchoring issue in one deliberate effort. The previous CFFU resolution's concern about EoB model re-evaluation has been partially addressed by STU2 publication, and further deferral without a concrete plan risks this ticket remaining unresolved indefinitely.
