## Details

Bryn Rhodes reports that the current mechanism for providing examples for `OperationDefinition` resources appears to only allow a single example. The [CodeSystem operations page](https://hl7-fhir.github.io/codesystem-operations.html) suggests that multiple examples should be possible, but in practice (specifically for the `$evaluate` operation definition in `DecisionSupportServiceModule`), the separation between examples does not come through.

Additionally, even when multiple examples are present, the current approach does not maintain continuity between request and response for each example — all requests are listed first, then all responses, rather than pairing each request with its corresponding response.

**Two distinct issues raised:**

1. **Multiple examples not rendering correctly** — The tooling or specification doesn't properly support or display multiple distinct examples for a single OperationDefinition.
2. **No request/response pairing** — When multiple examples exist, there is no mechanism to associate a specific request example with its corresponding response example, making it difficult for implementers to understand the expected input/output relationship.

## Proposed Dispositions

### Disposition A: Accept — Add Structured Example Grouping to OperationDefinition

Accept the change request and enhance the OperationDefinition resource (or the publication tooling) to support multiple named example groups, each containing a paired request and response. This could be done by:

- Extending the example mechanism to allow multiple example sets, each with a label/name.
- Ensuring each example set explicitly pairs a request with its corresponding response.
- Updating the publishing toolchain to render these example groups with clear visual separation and request/response pairing.

**Rationale:** This improves implementer experience by showing multiple usage scenarios with clear input/output relationships, which is especially valuable for complex operations like `$evaluate`.

### Disposition B: Resolve via Documentation and Tooling Guidance

Accept the request as a tooling/documentation issue rather than a specification change. Provide clear documentation on how to properly structure multiple examples within the existing OperationDefinition framework, and fix any rendering issues in the publication toolchain so that example separation works correctly.

**Rationale:** If the specification already allows multiple examples (as the CodeSystem operations page suggests), the issue may be purely in tooling and documentation rather than the resource definition itself. This avoids a spec change while still addressing the reporter's concern.

### Disposition C: Decline — Defer to Implementation Guides

Decline the change request, noting that operation examples can be documented in Implementation Guides or supplemental pages rather than within the OperationDefinition resource itself. Individual IGs can structure their examples with request/response pairing as needed.

**Rationale:** OperationDefinition is a conformance resource, and rich example documentation may be better suited to narrative IG pages. However, this leaves the core spec without a standardized mechanism for pairing operation examples.

## Recommendation

**Disposition A: Accept — Add Structured Example Grouping to OperationDefinition.**

This is a longstanding usability gap (filed in 2016, still unresolved) that affects how implementers understand operations. The inability to pair request/response examples and to clearly separate multiple examples is a meaningful limitation for operations with complex parameter sets. A structured example grouping mechanism would benefit the entire FHIR ecosystem, particularly for operations like `$evaluate` that have multiple valid usage patterns. The fact that the CodeSystem operations page already appears to support this pattern suggests the specification intent is there but the implementation is incomplete.
