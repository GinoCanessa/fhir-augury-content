**Title:** Please tighten your extension contexts
**Status:** Triaged **Priority:** Medium **Type:** Change Request

**Reporter:** Lloyd McKenzie **Assignee:** Unassigned
**Created:** 2023-11-20 **Updated:** 2023-11-29 **Labels:** None

## Summary

Lloyd McKenzie requests that several extensions defined in the FHIR for FAIR Implementation Guide have their extension contexts tightened. Currently, a number of these extensions declare a context of "Element," which is overly broad and does not reflect the actual intended usage of those extensions. The ticket asks that narrower, more specific contexts be asserted to accurately describe where each extension is expected to be applied.

## Details

**Description:**

A number of the extensions defined in this IG have a scope of "Element," which is broader than the obviously intended scope of the extensions. Please assert contexts that are narrower and reflect where the extensions are actually expected to be used.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Accept the change request and narrow the extension contexts in the FHIR for FAIR IG. Each extension currently scoped to "Element" should be reviewed and re-scoped to the specific resource types, data types, or element paths where they are intended to be used (e.g., restricting to specific resources such as "Patient," "Observation," or particular backbone elements within those resources).

#### Justification
Using "Element" as a context is effectively unconstrained and provides no guidance to implementers about where an extension is appropriate. Narrowing contexts is a best practice recommended by the FHIR specification itself, as it improves validation, tooling support, and implementer understanding. The reporter (Lloyd McKenzie) is a FHIR core specification editor and the request aligns with established IG authoring standards.

---

### Disposition B: Alternative Approach
#### Proposal
Partially accept the request by narrowing contexts for extensions where the intended scope is clearly understood, while retaining the broader "Element" context for any extensions whose applicability is genuinely broad or not yet fully determined. Document the rationale for each extension's chosen context in the IG's narrative text.

#### Justification
Some extensions may intentionally apply to a wide range of elements (e.g., metadata annotations), and forcing an overly narrow context could limit valid use cases. A phased approach allows the work group to refine contexts incrementally as real-world usage patterns emerge, while still addressing the core concern for extensions with obviously narrower intent.

---

### Disposition C: Decline
#### Proposal
Decline the change request and retain the current "Element" context for all extensions in the FHIR for FAIR IG.

#### Justification
The FHIR for FAIR IG is designed for broad interoperability across diverse research data, and the extensions may legitimately need to apply across many element types. Narrowing contexts prematurely could prevent valid usage scenarios that have not yet been fully explored. However, this position is weak given that overly broad contexts undermine tooling validation and implementer guidance.

---

### Recommendation

**Recommended disposition:** A

Disposition A is recommended because tightening extension contexts is an established FHIR best practice that directly improves the quality and usability of the implementation guide. Extensions scoped to "Element" provide no meaningful constraint, which undermines profile validation tooling and makes it harder for implementers to understand where extensions are intended to be used. The request comes from a core FHIR specification editor and reflects a standard expectation for IG quality. The work group should review each extension in the FHIR for FAIR IG, determine its intended usage scope, and assert appropriately narrow contexts accordingly.
