## Summary

This ticket requests editorial corrections to the PDMP section of the US Medication FHIR Implementation Guide. The submitter identifies three issues:

1. **Inaccurate language**: The IG text describes the API call as "the exact API," but it is actually an example (it contains the undefined placeholder `[base]`). The wording should be changed from "exact" to "an example."
2. **Minor wording improvements**: Adding "call" after "API," changing "MedicationDispense" to "MedicationDispense resources," and adding "resources" after the list of included resource types for improved clarity.
3. **Readability suggestion**: The submitter suggests breaking the example API call into a table with columns for "call parts" and "explanation" to match the code snippet to the explanatory text.

The ticket was originally resolved as "Considered for Future Use" in January 2023 (voted 4-0-0 by John Hatem / Tim McNeil) because the IG had been withdrawn. It was reopened in November 2023 when the PDMP on FHIR project (PSS-2225) was reactivated.

## Details

### Existing Wording (from IG)

> The following is the exact API that uses multiple search parameters to access PDMP data from a PDMP Responder
> …
> The above API will fetch all MedicationDispense .... The bundle returned will include MedicationDispense, MedicationRequest, Practitioner, Organization, Patient and Medication.

### Proposed Wording

> The following is **an example** API **call** that uses multiple search parameters to access PDMP data from a PDMP Responder
> …
> The above API **call** will fetch all MedicationDispense .... The bundle returned will include MedicationDispense **resources, and associated** MedicationRequest, Practitioner, Organization, Patient and Medication **resources**.

### Additional Suggestion

Break the example API call into a table with columns "call parts" and "explanation" to help implementers understand each component of the query.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

Accept the ticket as submitted. Apply the proposed wording changes:
- Change "the exact API" → "an example API call"
- Change "The above API will fetch" → "The above API call will fetch"
- Change "include MedicationDispense, MedicationRequest, Practitioner, Organization, Patient and Medication" → "include MedicationDispense resources, and associated MedicationRequest, Practitioner, Organization, Patient and Medication resources"

Additionally, add a table breaking down the API call components with columns "Call Part" and "Explanation" as suggested by the submitter. This is a straightforward editorial improvement that corrects an inaccuracy (the `[base]` placeholder makes the call an example, not "exact") and improves implementer comprehension.

### Disposition B: Accept with Modification (Persuasive with Modification)

Accept the wording corrections ("exact" → "an example," adding "call," adding "resources") but defer the table suggestion. The wording fixes are clearly needed and low-risk. The table structure is a larger editorial change that may benefit from coordination with FHIR-29487 (which requests formatting improvements to the same API example with line breaks and indentation). The workgroup could address both formatting requests together in a single pass.

### Disposition C: Decline (Not Persuasive)

Decline the ticket on the basis that the PDMP IG has already been published as STU1 (October 2024) and these editorial changes, if not already incorporated, should be tracked against the next version of the published IG rather than the pre-publication tracker. The workgroup should verify whether these changes were already applied during the STU1 publication process before taking further action.

## Recommendation

**Disposition B: Accept with Modification (Persuasive with Modification)**

The wording corrections are clearly valid — the API call contains `[base]` and is demonstrably an example, not an "exact" specification. The additional "call" and "resources" edits improve clarity with no risk of changing meaning. These should be accepted.

However, the table suggestion should be coordinated with FHIR-29487, which requests complementary formatting improvements (line breaks, indentation) to the same API example. Addressing both tickets together would produce a more coherent result than applying them independently. The workgroup should also verify whether the STU1 publication (announced October 2024) already incorporated any of these changes, and if so, close this ticket as already applied.
