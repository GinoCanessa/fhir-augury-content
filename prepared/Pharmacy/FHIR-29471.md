## Summary

This ticket identifies a mapping mismatch between the NCPDP `<FillStatus>` element and the FHIR `MedicationDispense.type` element in the US PDMP FHIR IG. The NCPDP `<FillStatus>` is a **numeric value** representing which fill in a sequence this dispense represents (e.g., fill 1, fill 2, fill 3). However, `MedicationDispense.type` is a **CodeableConcept** that categorizes the *kind* of dispense (e.g., "First Fill", "Refill", "Partial Fill") — it does not carry a numeric refill sequence number. Therefore, `MedicationDispense.type` is not an appropriate mapping target for the NCPDP refill number concept.

The ticket was originally resolved as "Considered for Future Use" in January 2023 (vote: 4-0-0) because the IG had been withdrawn. It was reopened in November 2023 when the PDMP on FHIR project (PSS-2225) was re-initiated.

## Details

### Existing Wording

The IG maps the NCPDP Data Element for Refill Number to `MedicationDispense.type`.

### Problem

- The NCPDP `<FillStatus>` element is a numeric field indicating the fill sequence number (e.g., "this is the 3rd fill").
- FHIR's `MedicationDispense.type` is a coded value (CodeableConcept) from the [v3-ActPharmacySupplyType](http://terminology.hl7.org/CodeSystem/v3-ActPharmacySupplyType) value set, with codes like `FF` (First Fill), `RF` (Refill), `DF` (Daily Fill), etc.
- `MedicationDispense.type` conveys the *category* of the dispense but has no mechanism to carry the ordinal fill number.
- This means the IG's mapping loses information: implementers cannot determine *which* refill (e.g., refill 2 of 5) from `MedicationDispense.type` alone.

## Proposed Dispositions

### Disposition A: Persuasive (Accept)

**Action:** Add a numeric extension or dedicated element to carry the fill sequence number in MedicationDispense.

- Acknowledge that `MedicationDispense.type` is not an appropriate mapping for the NCPDP `<FillStatus>` refill number.
- Define a PDMP IG-specific extension (e.g., `pdmp-fill-number`) on `MedicationDispense` to carry the ordinal fill sequence number as an integer.
- Update the NCPDP-to-FHIR mapping table in the IG to map `<FillStatus>` (refill number) to this new extension instead of `MedicationDispense.type`.
- Retain `MedicationDispense.type` for its intended purpose: categorizing the dispense as first fill, refill, partial fill, etc.
- Coordinate with FHIR-29472 (partial fill indicator) to ensure both FillStatus sub-concepts are properly addressed.

### Disposition B: Persuasive with Modification

**Action:** Map the refill number to an existing FHIR element rather than creating a new extension.

- Acknowledge the mapping issue.
- Instead of an extension, use an existing mechanism such as a coded `MedicationDispense.type` value **combined** with a note or `MedicationDispense.extension` for the ordinal number, or map to `MedicationDispense.note` with a structured convention.
- Alternatively, explore whether the fill sequence can be inferred from the `MedicationDispense.authorizingPrescription` reference and the count of prior dispenses against the same prescription.
- Update the mapping table to clarify that `MedicationDispense.type` conveys the fill category and that the numeric fill sequence requires supplementary representation.
- This avoids proliferating IG-specific extensions but may not perfectly capture the discrete numeric value.

### Disposition C: Not Persuasive (Decline)

**Action:** Retain the current mapping with clarified documentation.

- Acknowledge that `MedicationDispense.type` does not carry a numeric fill number but argue that the coded type values (`FF`, `RF`, etc.) are sufficient for PDMP use cases.
- Note that PDMP queries primarily need to know *whether* a dispense is a refill, not necessarily the exact ordinal number.
- Add clarifying text to the IG explaining that the fill sequence number is not directly mapped and that systems needing the exact refill number should derive it from the sequence of `MedicationDispense` records linked to the same `MedicationRequest`.
- Risk: This may not satisfy NCPDP data fidelity requirements and could reduce interoperability with systems that depend on the numeric fill number.

## Recommendation

**Disposition A: Persuasive (Accept)** is recommended. The ticket correctly identifies a genuine data loss in the current mapping — the NCPDP `<FillStatus>` refill number is a discrete numeric value that cannot be represented by the coded `MedicationDispense.type` element. With the PDMP STU1 IG now published and the project actively maintained, this is an appropriate time to introduce a targeted extension to carry the fill sequence number. This approach preserves data fidelity from NCPDP SCRIPT, aligns with the IG's dispense-focused scope, and should be coordinated with the closely related FHIR-29472 (partial fill indicator). The Zulip discussion on "Medication Refills" confirms that even FHIR experts recognize the complexity of tracking fill counts, reinforcing the need for an explicit representation.
