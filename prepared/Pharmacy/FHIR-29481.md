## Summary

The submitter identifies a gap in the PDMP IG's search specification: when a PDMP Responder receives a query using the required chained search parameters (patient given name, family name, and birthdate) and cannot resolve a unique patient match, there is no defined behavior or guidance for what should be returned. The limited set of mandatory search parameters (only first name, last name, and DOB) does not provide the requester with any mechanism to refine the query to disambiguate between multiple potential matches. The submitter requests that the IG address this scenario, at minimum with implementer guidance.

The existing wording specifies that the PDMP Responder SHALL support chained search parameters (`subject:Patient.name.given`, `subject:Patient.name.family`, `subject:Patient.birthdate`) and `_include` parameters for MedicationDispense search — but says nothing about non-unique or zero match outcomes.

## Proposed Dispositions

### Disposition A: Persuasive — Add Implementer Guidance for Non-Unique Match Scenarios

**Resolution:** Persuasive

**Rationale:** The ticket correctly identifies a specification gap. The PDMP IG mandates chained search parameters for MedicationDispense queries but provides no guidance on expected behavior when the patient demographics do not resolve to a unique match. This is a real implementation concern — PDMP systems commonly encounter ambiguous patient identities given the limited demographics (name + DOB).

**Suggested Change:** Add a section or implementer note to the PDMP search documentation describing:
1. When a PDMP Responder cannot resolve a unique patient from the provided search parameters, it SHOULD return an empty Bundle (no MedicationDispense results) and MAY include an OperationOutcome indicating a non-unique or no-match result.
2. Implementers should be aware that the limited search parameter set (given name, family name, birthdate) may not uniquely identify patients, and additional out-of-band disambiguation mechanisms may be needed.
3. PDMP Responders MAY support additional optional search parameters (e.g., address, SSN last-4) to aid disambiguation where permitted by state PDMP regulations.

**Impact:** Low — adds guidance text only; no structural change to the API.

---

### Disposition B: Persuasive with Modification — Restructure to Two-Step Patient Resolution

**Resolution:** Persuasive with Modification

**Rationale:** The underlying problem is architectural: the chained search pattern bundles patient identification and dispense retrieval into a single request, leaving no room for the requester to resolve ambiguity. As noted in the closely related FHIR-29493 (Josh Mandel), a two-step approach — first resolve the patient identity, then query dispenses for the resolved patient — would natively support disambiguation and allow more flexible patient matching strategies over time.

**Suggested Change:** Rather than only adding guidance text, modify the PDMP search approach to support (or prefer) a two-step workflow:
1. Step 1: Resolve the patient identity (via Patient search, `$match`, or other mechanism), allowing the requester to handle multiple matches.
2. Step 2: Query MedicationDispense by resolved patient reference.

Coordinate this change with FHIR-29493 which proposes the same architectural shift. Retain the existing chained search as an alternative for simple cases.

**Impact:** High — significant restructuring of the PDMP search specification; should be coordinated with FHIR-29493 and FHIR-29474.

---

### Disposition C: Not Persuasive — Defer to General FHIR Search Behavior

**Resolution:** Not Persuasive

**Rationale:** The FHIR specification already defines general behavior for search: when chained search parameters do not uniquely match a target resource, the server returns results matching all candidates or an empty result set depending on implementation. The PDMP IG does not need to restate general FHIR search semantics. Non-unique patient matching is a universal challenge across FHIR implementations and is not specific to the PDMP use case. State PDMP systems have their own matching logic that operates beneath the FHIR API layer.

**Impact:** None — no specification change.

---

## Recommendation

**Disposition A: Persuasive** is recommended.

The specification has a clear gap — implementers need guidance on how to handle non-unique patient matches in PDMP queries. Disposition A directly addresses the submitter's request with practical, implementable guidance at low risk and minimal specification disruption. While Disposition B (two-step resolution) is architecturally superior, it represents a significant change that should be evaluated holistically alongside FHIR-29493 and FHIR-29474 rather than resolved through this single ticket. Disposition C is insufficient because the PDMP context imposes specific constraints (limited demographics, regulatory requirements) that make generic FHIR search behavior inadequate for implementer guidance.
