## Summary

Eric Haas submitted this change request asserting that the CARIN RTPBC request bundle profile (as published in the 2020FEB ballot) incompletely defines its message bundle. Specifically, he argues that profiling `Bundle` alone is insufficient — the entries should be profiled using the `aggregate` element to define the resource graph, or a `GraphDefinition` resource should be used in addition to (or instead of) the Bundle profile. The referenced profile is: http://hl7.org/fhir/us/carin-rtpbc/2020FEB/StructureDefinition-carin-rtpbc-request-bundle.html

The ticket was initially resolved as **"Considered for Future Use"** during ballot reconciliation (2020-04-20, vote: Frank McKinney / Pooja Babbrah, 11-0-1) with the note that the workgroup would consider using GraphDefinition in a future version. However, Eric Haas disagreed with that resolution, requesting the ticket be pulled for further discussion on why the profiles cannot be updated or a GraphDefinition used. Subsequently, Lynn Laakso re-opened the ticket after STU1 publication (2020-08-27), reverting the previous resolution. The ticket has remained in **Triaged** status since then.

---

## Details

### Core Issue

The fundamental concern is about the completeness and correctness of the CARIN RTPBC request bundle profile. When defining a message bundle via a Bundle profile, simply profiling the Bundle resource type is insufficient to fully constrain the expected entries. The issue identifies two approaches to improve this:

1. **Profile entries with the `aggregate` element** — Use Bundle.entry slicing with discriminators to fully define what resource types (and their profiles) are expected in each entry slot.
2. **Use `GraphDefinition`** — Define a GraphDefinition resource that formally specifies the graph of resources expected in the bundle, including their relationships and cardinalities.

## Proposed Dispositions

### Disposition A: Accept — Require GraphDefinition or Enhanced Bundle Slicing in RTPBC

**Action:** Update the CARIN RTPBC specification to either (a) add a `GraphDefinition` resource that formally defines the expected resource graph in the request bundle, or (b) enhance the Bundle profile with proper entry slicing using discriminators to fully constrain entry types, profiles, and cardinalities.

**Rationale:** The original concern is technically valid — a Bundle profile without entry constraints is incomplete and does not guide implementers on what resources to include. Since GraphDefinition's maturity and tooling support remain limited (as evidenced by FHIR-26203 and FHIR-26204), the pragmatic approach is enhanced Bundle entry slicing, which is well-supported by existing tooling. A GraphDefinition could be added as supplementary documentation.

**Impact:** Moderate. Requires updates to the RTPBC Bundle StructureDefinition. Existing implementations that already conform to the expected pattern would not break, but the profile would become more precise.

### Disposition B: Accept with Modification — Add Bundle Entry Slicing Only, Defer GraphDefinition

**Action:** Update the CARIN RTPBC Bundle profile to include proper entry slicing with type and profile discriminators. Document the decision to defer GraphDefinition adoption until the resource reaches higher maturity and better tooling support in the FHIR ecosystem.

**Rationale:** The core issue is the incomplete Bundle profiling. Bundle entry slicing is a well-established pattern in FHIR IGs (e.g., US Core, Da Vinci). GraphDefinition, while conceptually appropriate, has limited adoption and tooling support. The workgroup's original 2020 vote acknowledged the idea had merit but deferred it. This disposition addresses the substantive concern (incomplete profiling) while being pragmatic about GraphDefinition readiness.

**Impact:** Low-to-moderate. The Bundle profile changes are straightforward and align with current best practices for Bundle profiling.

### Disposition C: Decline — Not Persuasive

**Action:** Close as Not Persuasive. The current RTPBC Bundle profile is sufficient for implementers, and the specification's narrative documentation adequately describes the expected bundle contents.

**Rationale:** The RTPBC IG has been published and implemented as STU1 without reported interoperability issues stemming from insufficient bundle profiling. Adding GraphDefinition or complex slicing introduces specification complexity without demonstrated implementation need. The original 11-0-1 vote to defer was reasonable, and no new evidence of implementation problems has been presented since the ticket was re-opened.

**Impact:** None. No specification changes required.

---

## Recommendation

**Disposition B: Accept with Modification — Add Bundle Entry Slicing Only, Defer GraphDefinition**

This disposition directly addresses the valid technical concern (incomplete Bundle profiling) using a proven, well-supported approach (entry slicing) while avoiding the risks of mandating a resource type (GraphDefinition) that still has limited ecosystem adoption and tooling. It respects the workgroup's earlier judgment to defer GraphDefinition while acknowledging that the underlying profiling gap should be fixed. The ticket has been open for over five years and was re-opened specifically because the reporter felt the concern was not adequately addressed — applying the pragmatic fix of proper Bundle entry slicing resolves the substantive issue.
