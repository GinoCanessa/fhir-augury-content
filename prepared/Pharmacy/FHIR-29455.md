## Description

The submitter requests clarification on a statement in the PDMP Responder section of the Security requirements. The existing wording states:

> "The PDMP Responder is expected to allow registration of a PDMP Requestor for each individual Provider or a Provider Organization as required by policies."

The submitter notes this statement may have been intended to call out that providers need to get registered first, but it does not seem closely related to the problem the IG is trying to address in that section.

## Resolution Description

> Agree. Move this statement about provider registration out of Security/Responder Requirements and into the Use Case page as background.

## Analysis

This ticket addresses a **content placement** issue in the PDMP IG. The statement about provider registration is informational background — it describes a prerequisite (providers must register with the PDMP) rather than a technical security requirement. Placing it in the Security/Responder Requirements section conflates policy context with implementation conformance requirements.

The approved resolution (Persuasive with Modification) — moving the statement to the Use Case page as background — is well-aligned with the concern. This approach:
1. Preserves the information about provider registration requirements
2. Removes non-technical content from the security section
3. Keeps the security section focused on actual conformance requirements

This ticket is closely related to a cluster of PDMP security section issues (FHIR-29490, FHIR-29492, FHIR-29477, FHIR-29527) that collectively identify structural and clarity problems in the same IG section. The IG was previously withdrawn (leading to a 2023-01-18 "Considered for Future Use" resolution), then reopened in November 2023 when PSS-2225 relaunched the PDMP on FHIR project.

## Proposed Dispositions

### Disposition A: Persuasive with Modification (affirm existing resolution)
**Action:** Accept the existing resolution. Move the provider registration statement from the Security/Responder Requirements section to the Use Case page as background context.

**Rationale:** The resolution correctly identifies that the registration statement is policy background, not a security requirement. Moving it to the Use Case page preserves the information while keeping the security section focused on technical conformance. This aligns with similar feedback in FHIR-29490 and FHIR-29492 about the same text. The prior vote (4-0-0) also endorsed this direction before the IG was withdrawn.

### Disposition B: Persuasive
**Action:** Remove the registration statement from the Security/Responder Requirements section entirely, as registration processes are outside the scope of the FHIR-based PDMP exchange IG.

**Rationale:** The registration process for PDMP access is governed by state-level PDMP programs and is an operational concern, not a FHIR implementation concern. Removing it entirely avoids placing normative-sounding text in any section of the IG. However, this loses context that may help implementers understand the broader workflow.

### Disposition C: Not Persuasive
**Action:** Retain the statement in its current location in the Security/Responder Requirements section.

**Rationale:** Provider registration is a prerequisite to PDMP data access and is therefore relevant to the security context. Implementers of PDMP Responder systems need to understand that registration is expected. However, this does not address the submitter's valid concern that the statement is not a technical security requirement.

## Recommendation

**Disposition A: Persuasive with Modification** — Affirm the existing resolution. The statement is informational context about provider registration policy, not a technical security requirement. Moving it to the Use Case page is the correct structural fix. This disposition was previously endorsed by a 4-0-0 vote and is consistent with the cluster of related feedback on the PDMP security section. Coordinate with FHIR-29490 and FHIR-29492, which raise overlapping concerns about the same text.
