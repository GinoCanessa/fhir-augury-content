## Description & Context

The ticket reports that `<RequestingState>` is not a valid field in any version of the NCPDP SCRIPT standard. The existing wording in the IG references an "NCPDP Data Element for State code of Requesting Facility," but this element does not exist in NCPDP SCRIPT as named.

The submitter notes that in NCPDP SCRIPT version 2017071, the correct element path would be `<Facility>/<Address>/<State>` for transmitting the state code of the requesting facility.

This is one of a large set of NCPDP SCRIPT mapping correction issues (labeled `PDMP-old-issues`) originally submitted by Daniel Zhang against the US PDMP FHIR IG and imported from a spreadsheet in October 2020.

## Proposed Dispositions

### Disposition A: Applied — Correct the element path to `<Facility>/<Address>/<State>`

**Rationale:** The submitter's correction is technically accurate — `<RequestingState>` does not exist in NCPDP SCRIPT standards, and the correct path in NCPDP SCRIPT 2017071 is `<Facility>/<Address>/<State>`. The PDMP IG should reference valid NCPDP SCRIPT element paths to ensure implementers can correctly map between standards. This should be applied as a straightforward technical correction in the next version of the IG.

**Action:** Update the NCPDP-to-FHIR mapping table in the US PDMP FHIR IG to replace the reference to `<RequestingState>` with the correct NCPDP SCRIPT 2017071 element path `<Facility>/<Address>/<State>` for the state code of the requesting facility.

### Disposition B: Considered — Verify against published PDMP STU1 before applying

**Rationale:** The PDMP STU1 IG was published in October 2024, and the NCPDP SCRIPT mapping tables may have already been reworked during that publication cycle. Before applying a change, the work group should verify whether the published STU1 already corrected this element path. If it has been addressed, this ticket can be closed as already resolved. If not, it should be applied in the next errata or STU update.

**Action:** Review the published PDMP STU1 IG (`http://hl7.org/fhir/us/pdmp/STU1`) to determine if the `<RequestingState>` reference has already been corrected. Disposition accordingly.

### Disposition C: Not Persuasive — Resolve as part of a broader NCPDP mapping overhaul

**Rationale:** This is one of many NCPDP SCRIPT mapping corrections in the PDMP-old-issues batch. Rather than addressing each element-level correction individually, the work group could defer this ticket and address all NCPDP mapping issues holistically in a future IG revision, ensuring consistency across all element paths.

**Action:** Defer in favor of a consolidated review of all PDMP-old-issues tickets as a group effort.

## Recommendation

**Disposition B: Considered — Verify against published PDMP STU1 before applying.**

Given that the PDMP STU1 IG was published in October 2024 — well after this ticket was reopened — the most prudent first step is to verify whether this correction was already incorporated. The technical correction itself (replacing `<RequestingState>` with `<Facility>/<Address>/<State>`) is straightforward and valid, but applying a change that has potentially already been made would be redundant. If PDMP STU1 still contains the incorrect element name, the correction should be applied as described in Disposition A. The closely related ticket FHIR-29463 (requesting facility identification) should be reviewed at the same time, as both address the `<Facility>` element's sub-paths.
