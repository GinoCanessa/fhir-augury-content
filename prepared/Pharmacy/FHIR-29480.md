## Description

The submitter requests that the PDMP IG mention the distinction between **workflow-based integration** (e.g., web portal launch, SMART on FHIR app launch where data is viewed but not stored) and **discrete data integration** (e.g., structured FHIR resources or NCPDP exchanges where data is incorporated into the local system). The submitter acknowledges these intricacies may be out of scope for the IG but suggests the distinction would be useful for implementers evaluating integration approaches.

## Analysis

### Context

This ticket originates from the original PDMP IG ballot (2020) and requests that the IG explain the difference between two PDMP integration approaches:

1. **Workflow integration** — The user launches a web portal or SMART on FHIR app to view PDMP data in context, but the data is not stored locally. This is the dominant pattern today, particularly in states that prohibit discrete data transmission or local storage.

2. **Discrete data integration** — Structured data (via NCPDP RxHistory, FHIR resources, etc.) is returned and can be stored in the local system, used for CDS, etc. This is only permitted in certain states.

### Significant Development Since Filing

The most significant related development is **FHIR-46046** (resolved Persuasive with Modification, now Published), which directly tackled the question of how FHIR fits into the existing PDMP ecosystem. The disposition of FHIR-46046 resulted in a new section titled **"Adding FHIR to the PDMP Ecosystem"** in the IG's home page. The extensive comment thread on FHIR-46046 (between Scott Robertson and Cooper Thompson) explicitly discusses:

- Most PDMPs currently operate via web portals (workflow integration) because they are **not permitted to provide discrete data** or allow local storage.
- NCPDP is the established standard where discrete data is allowed, though its usage is reportedly declining.
- The FHIR IG is positioned as a **future option** for when PDMPs allow discrete data exchange via FHIR infrastructure.
- PDMPs are often housed in state Departments of Justice (legal entities), not health departments, which drives the "read-only" restriction.

This context is highly relevant because FHIR-46046 already partially addresses the concern in FHIR-29480 — the new "Adding FHIR to the PDMP Ecosystem" section implicitly describes the discrete data vs. workflow distinction by explaining when FHIR integration is and isn't beneficial.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Add a brief paragraph to the IG's introduction or background section explicitly defining and contrasting "workflow integration" (portal/app launch, view-only) and "discrete data integration" (structured data returned and stored locally). Reference the existing "Adding FHIR to the PDMP Ecosystem" section (added via FHIR-46046) and note that this IG primarily targets the discrete data integration pattern.

**Rationale:** The request is valid — implementers benefit from understanding these two distinct integration patterns when evaluating the IG. While FHIR-46046 added relevant context, it does not use the explicit "workflow vs discrete data" terminology that would help readers quickly understand the landscape. A short, explicit definition would close this gap without significant effort.

### Disposition B: Not Persuasive with Modification
**Action:** Close the ticket with a note pointing to the content added under FHIR-46046 ("Adding FHIR to the PDMP Ecosystem" section) as substantially addressing the concern. No further IG changes required. Optionally add a brief clarifying sentence to that section noting the two integration patterns.

**Rationale:** The concern has been largely addressed by the work done for FHIR-46046, which explains the current PDMP ecosystem and where FHIR fits. The new section implicitly covers the workflow vs. discrete data distinction through its discussion of web portals, SMART on FHIR, and discrete data limitations. Adding further content risks over-complicating the IG with background material that is tangential to its core technical scope.

### Disposition C: Considered for Future Use
**Action:** Defer this ticket for potential inclusion in a future ballot cycle. The IG is still evolving and the distinction between workflow and discrete data integration may become more important as state policies change.

**Rationale:** The IG is currently in early adoption. As more states permit discrete data exchange and real-world FHIR-based PDMP implementations emerge, clearer guidance on integration patterns will be more actionable. Deferring allows the IG authors to incorporate real-world implementation experience rather than speculative guidance.

---

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The commenter's request is reasonable and actionable. While FHIR-46046 addressed the broader "why FHIR?" question, it did not explicitly name and contrast the two integration paradigms (workflow vs. discrete data) that implementers encounter. Adding a brief, clearly labeled explanation of these two patterns — and noting that this IG is primarily relevant to the discrete data pattern — would orient implementers quickly and cost very little editorial effort. The extensive FHIR-46046 comment thread (particularly Scott Robertson's detailed explanations of PDMP legal constraints, web portal dominance, and discrete data limitations) provides ample source material for this addition. Given the ticket has been open since 2020 and the IG now has sufficient content to contextualize this distinction, resolving it with a targeted modification is the most efficient path forward.
