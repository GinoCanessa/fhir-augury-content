## Description

The submitter requests that the PDMP FHIR IG make it clearer that flexibility in standards usage is key to connecting more EHRs to PDMPs. Specifically, while the message from an intermediary to a PDMP requires NCPDP, the message from an EHR to an intermediary can use a variety of standards including FHIR, PMIX/NIEM, and NCPDP. The submitter states this flexibility may be suggested in the document but is not clearly stated.

**Proposed Wording (from submitter):**
> While the message from the intermediary to the PDMP requires NCPDP, often, the message from the EHR to the intermediary can be in a variety of standards including FHIR, PMIX/NIEM standard, and NCPDP

## Resolution Description

> The Use Case page now includes info about the participants, standards used, and where FHIR fits in the IG

## Analysis

This ticket is a ballot comment from the original PDMP FHIR IG ballot cycle (2020). The core request is straightforward: make the IG clearer about the fact that EHR-to-intermediary communications can use multiple standards (FHIR, PMIX/NIEM, NCPDP), even though the intermediary-to-PDMP leg is NCPDP-only. This is an important architectural point for implementers evaluating how to connect their EHRs to PDMP systems.

The resolution description indicates the Use Case page was updated to include information about participants, standards used, and where FHIR fits—addressing the spirit of the request, though not adopting the specific proposed wording verbatim (hence "Not Persuasive with Modification" rather than "Persuasive").

The ticket was originally deferred (Considered for Future Use) when the IG was withdrawn, then reopened under PSS-2225 when the PDMP on FHIR project resumed.

## Proposed Dispositions

### Option A: Agree with Current Resolution (Not Persuasive with Modification)
Accept the "Not Persuasive with Modification" resolution as-is. The resolution description confirms the Use Case page now addresses participants, standards, and FHIR's role. The submitter's core concern—that the IG should explain the standards flexibility in the EHR-to-intermediary segment—has been addressed in substance, even if the exact proposed wording was not used. The workgroup has already incorporated the needed clarifications.

**Pros:** Aligns with work already done; avoids re-litigating a resolved editorial improvement.
**Cons:** Without reviewing the current Use Case page text, there is some risk the clarification is insufficient.

### Option B: Upgrade to Persuasive with Modification
Change the resolution to "Persuasive with Modification" to acknowledge that the comment had merit and was substantively addressed. The distinction matters because the submitter's point was valid—the IG's Use Case page was updated to include the requested information. "Persuasive with Modification" more accurately reflects that the feedback was incorporated in a modified form.

**Pros:** More accurately characterizes the outcome; acknowledges the comment's value.
**Cons:** Requires changing the existing resolution status.

### Option C: Reopen for Further Clarification
Reopen the ticket to verify whether the current Use Case page text adequately addresses the standards flexibility point, particularly given the IG was substantially reworked under PSS-2225. Ensure the description of FHIR, PMIX/NIEM, and NCPDP roles in the EHR→intermediary→PDMP chain is explicit and accurate in the current IG draft.

**Pros:** Ensures the revived IG properly addresses this important implementer concern.
**Cons:** Adds review burden for what may already be adequately addressed.

## Recommendation

**Option A: Agree with Current Resolution (Not Persuasive with Modification).** The resolution description confirms the Use Case page was updated to include information about participants, standards used, and where FHIR fits. This addresses the submitter's core concern about making standards flexibility visible to implementers. The "Not Persuasive with Modification" resolution appropriately indicates the feedback was considered and the IG was updated, though not with the exact proposed wording. Given the volume of sibling PDMP-old-issues tickets being processed, accepting the existing resolution is efficient and consistent.
