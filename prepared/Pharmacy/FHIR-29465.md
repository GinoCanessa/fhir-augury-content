## Description

The submitter requests that additional patient data elements beyond first name, last name, and date of birth be mapped in the NCPDP Data Elements for Patient section of the PDMP portion of the US Medication FHIR IG. The existing wording references "NCPDP Data Elements for Patient" but the current mapping table is limited to only those three basic demographic fields. NCPDP SCRIPT standards include additional patient-identifying data elements (e.g., address, phone, gender, identifiers such as SSN or Medicare Number) that are commonly used in PDMP queries and responses and should be represented in the FHIR mapping.

## Analysis

This ticket addresses a genuine gap in the PDMP section of the US Medication FHIR IG. The NCPDP SCRIPT standard includes numerous patient data elements beyond the three currently mapped (first name, last name, DOB). Elements such as patient address, phone number, gender, SSN, Medicare Number, and state-specific identifiers are commonly transmitted in PDMP request and response transactions. The limited mapping reduces the IG's utility for implementers who need to exchange these data elements.

The ticket was originally deferred when the IG was withdrawn but has been reopened now that the PDMP on FHIR project (PSS-2225) is actively developing the IG. Several sibling tickets (FHIR-29497 on patient identifiers, FHIR-29451 on SSN mandating, FHIR-29459 on role-dependent query elements) address overlapping concerns and should be coordinated together.

The NCPDP-to-FHIR mapping thread on Zulip confirms that comprehensive mapping work has not yet been completed, supporting the need for this enhancement.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Accept the request and expand the NCPDP-to-FHIR patient data element mapping table to include all commonly used PDMP patient fields from NCPDP SCRIPT 2017071 (e.g., address, phone, gender, identifiers, state-issued ID). Map each to appropriate FHIR Patient resource elements.

**Rationale:** The current mapping is incomplete and does not serve implementers who need to transmit the full set of patient demographics in PDMP transactions. NCPDP SCRIPT 2017071 defines these elements and they have clear FHIR Patient mappings. This aligns with the reopened PSS-2225 project scope.

**Trade-offs:** Requires identifying the complete set of patient elements from NCPDP SCRIPT 2017071 and determining FHIR mappings. Should be coordinated with FHIR-29497 (identifiers) and FHIR-29451 (SSN optionality) to avoid conflicts.

### Disposition B: Persuasive with Modification
**Action:** Expand the patient data element mapping, but limit scope to elements that are commonly required or must-support across state PDMP programs, rather than mapping every possible NCPDP patient field. Add a note that state-specific elements may vary and provide guidance on extensibility.

**Rationale:** Different states require different patient data elements for PDMP queries. A pragmatic approach maps the most commonly used elements and provides guidance for state-specific extensions, rather than attempting an exhaustive mapping that may not apply universally.

**Trade-offs:** More manageable scope but may require future updates as state requirements evolve. Still requires coordination with sibling tickets.

### Disposition C: Considered – No Action
**Action:** Acknowledge the gap but defer action until the PDMP on FHIR IG (PSS-2225) completes its comprehensive NCPDP SCRIPT 2017071 mapping effort, which would naturally address this as part of a broader data element review.

**Rationale:** The PDMP on FHIR project is actively re-developing the IG. A piecemeal expansion of the patient mapping table may be superseded by the project's comprehensive mapping work. This avoids duplicating effort.

**Trade-offs:** Leaves the gap unaddressed in the interim. Implementers currently lack guidance on mapping additional patient elements.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The gap is real and well-documented across multiple sibling tickets. Expanding the mapping to cover the most commonly used PDMP patient data elements (address, phone, gender, key identifiers) provides immediate value to implementers without attempting an exhaustive mapping that could conflict with the broader PSS-2225 effort. The workgroup should coordinate resolution with FHIR-29497 and FHIR-29451 to ensure consistent handling of patient identifiers and SSN optionality.
