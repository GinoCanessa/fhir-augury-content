## Description

The submitter asks two questions about PDMP FHIR IG error handling:

1. **Error mapping**: Conformance requirements define how the PDMP responder should respond in error scenarios. Should the IG map FHIR errors to error handling solutions for each standard (e.g., NCPDP SCRIPT, PMIX/NIEM)?
2. **No patient match**: When the PDMP responder has no dispense information to return because there is no patient match, how should that be mapped in the response?

Both questions address a gap in the IG regarding what FHIR responses (e.g., OperationOutcome codes, empty Bundles) should be returned in failure or no-data scenarios, and how those map to equivalent concepts in other PDMP standards.

## Proposed Dispositions

### Disposition A: Persuasive — Add Error Handling and No-Match Guidance to the PDMP IG

**Rationale**: The submitter raises a valid gap. The PDMP FHIR IG should define how the PDMP responder communicates error conditions and no-match results in FHIR terms. Specifically:
- Define standard OperationOutcome codes for common PDMP error scenarios (authentication failure, invalid request parameters, system unavailable, etc.).
- Specify that a no-patient-match should return an HTTP 200 with an empty searchset Bundle (consistent with FHIR search semantics) rather than an error, and include an OperationOutcome entry with an informational severity code indicating no match.
- Optionally include a mapping table showing how FHIR error responses correspond to error codes in NCPDP SCRIPT and PMIX/NIEM for implementers bridging standards.

**Impact**: Moderate — adds a new error handling section and conformance language to the IG; would need review alongside FHIR-29481 (non-unique match) to ensure consistent guidance.

### Disposition B: Persuasive with Modification — Add No-Match Guidance Only; Defer Cross-Standard Error Mapping

**Rationale**: The no-match scenario is the most pressing gap, as implementers need clear guidance on what to return when a patient is not found. However, mapping FHIR errors to every other PDMP standard's error model is a larger undertaking that could delay the IG. Instead:
- Add guidance that a no-patient-match returns an empty searchset Bundle with an informational OperationOutcome.
- Add a general note that implementers should consult each standard's error handling documentation for cross-standard mapping.
- Defer the full error mapping table to a future version or companion document.

**Impact**: Low-to-moderate — adds focused guidance without the complexity of cross-standard mapping.

### Disposition C: Not Persuasive — Existing FHIR Semantics Are Sufficient

**Rationale**: FHIR already defines how search operations handle errors (OperationOutcome) and no-results (empty searchset Bundle). The base specification's error handling is well-documented and applies to all FHIR search interactions, including PDMP. Adding IG-specific error mapping to other standards is out of scope for a FHIR IG. Implementers familiar with FHIR should already know how to interpret these responses.

**Impact**: None — no change to the IG.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended.

The no-match scenario is a real implementation gap — implementers need to know whether a PDMP with no data for a patient returns an error or an empty result set, and how to distinguish "no match" from "system error." Adding this focused guidance is high-value and low-effort. However, the request to map FHIR errors to error codes in every other PDMP standard (NCPDP SCRIPT, PMIX/NIEM, etc.) is a significant scope expansion that should be addressed separately. This approach also aligns well with FHIR-29481, which raises the complementary question of non-unique match handling and should be resolved together for consistency. The newer PDMP IG (US Prescription Drug Monitoring Program FHIR IG) — revived under PSS-2225 — is the appropriate vehicle for this guidance.
