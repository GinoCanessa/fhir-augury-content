## Description

The submitter reports that each NCPDP Data Element mapping in the PDMP IG only shows the leaf (child) element name without the parent path context. This makes it unclear exactly which XML element is intended, especially when the same element name exists under multiple parents.

The request is to include relative path information in the NCPDP Data Element column. For example:

- When a **prescriber** is the PDMP requestor, "Requestor Name" should show:
  `<Prescriber>/<Name>/<LastName>` and `<Prescriber>/<Name>/<FirstName>`

- When a **pharmacist** is the PDMP requestor, it should show:
  `<Pharmacy>/<Pharmacist>/<LastName>` and `<Pharmacy>/<Pharmacist>/<FirstName>`

### Existing Wording
NCPDP Data Element definitions (leaf element names only, without parent path context).

## Proposed Dispositions

### Disposition A: Persuasive

**Action:** Add full relative path context to all NCPDP Data Element entries in the mapping tables.

Update every NCPDP Data Element cell to include the parent element hierarchy from the NCPDP SCRIPT message structure. Where the same leaf element appears under different parents depending on requestor type (prescriber vs. pharmacist), provide separate qualified paths for each case — for example:

- Prescriber requestor: `<Prescriber>/<Name>/<LastName>`
- Pharmacist requestor: `<Pharmacy>/<Pharmacist>/<LastName>`

This directly addresses the submitter's request and brings the NCPDP mapping table in line with the level of specificity expected by implementers working with the XML-based SCRIPT standard. Given the large cluster of related tickets (FHIR-29459, -29462, -29463, -29464, -29466) raising similar path-clarity concerns, a comprehensive pass over all mappings is warranted.

### Disposition B: Persuasive with Modification

**Action:** Add relative path information only where ambiguity exists, and add an introductory section explaining NCPDP path conventions.

Rather than fully qualifying every mapping entry, add relative paths where a leaf element name is shared across multiple parent elements or where the requestor type changes the applicable path. Additionally, add an introductory note or legend to the NCPDP mapping section explaining the SCRIPT XML structure and how to interpret element references. This addresses the core clarity concern while limiting the maintenance burden of the mapping table.

### Disposition C: Not Persuasive

**Action:** No change.

The NCPDP Data Element column is intended to identify the data element by name; implementers are expected to reference the NCPDP SCRIPT standard for full structural context. Adding XML paths would increase the maintenance burden of the mapping table and risk becoming outdated as NCPDP standards evolve. Reject the request and instead add a reference/link to the NCPDP SCRIPT standard for path details.

## Recommendation

**Disposition A: Persuasive.**

The request is well-justified and directly supported by multiple related tickets from the same submitter (FHIR-29459, FHIR-29462, FHIR-29463, FHIR-29464, FHIR-29466) that all point to confusion caused by ambiguous leaf-only NCPDP element references. The PDMP IG has been actively restarted under PSS-2225 and has since reached STU publication (Edition 1, October 2024), making this the appropriate time to address accumulated mapping clarity issues. The Zulip discussion in "NCPDP 2017 to FHIR 4.0" confirms that the NCPDP-to-FHIR mapping space has historically been under-specified, reinforcing the need for explicit path information. A comprehensive update to include relative paths would benefit implementers and should be coordinated with the related batch of PDMP-old-issues tickets.
