## Description

The `<Quantity>` field within MedicationDispensed of NCPDP SCRIPT (versions 2017071 and 10.6) has multiple required subfields. The ticket requests more granular guidance for mapping from those individual NCPDP SCRIPT Quantity subfields to FHIR resources. The existing wording simply references "NCPDP Data Element for Quantity" without specifying how each subfield maps.

## Proposed Dispositions

### Disposition A: Applied — Add Granular Quantity Subfield Mapping Table

**Action:** Add a detailed mapping table to the US Medication FHIR IG that enumerates each NCPDP SCRIPT 2017071 `<Quantity>` subfield (e.g., `<Value>`, `<CodeListQualifier>`, `<UnitOfMeasure>`, `<QuantityUnitOfMeasure>`) and maps each to the corresponding FHIR MedicationDispense elements (e.g., `MedicationDispense.quantity.value`, `MedicationDispense.quantity.unit`, `MedicationDispense.quantity.system`, `MedicationDispense.quantity.code`).

**Rationale:** The request is valid — NCPDP SCRIPT Quantity is a composite field with multiple required subfields, and the current IG only provides a top-level "Quantity" mapping without guidance on individual subfields. This creates ambiguity for implementers. Adding explicit subfield-level mappings aligns with the approach being taken for other NCPDP elements in the same IG (see FHIR-29464, FHIR-29466, FHIR-29463). The mappings should reference NCPDP SCRIPT 2017071 per FHIR-29460.

**Vote Suggestion:** Applied (with editorial discretion on exact table format).

---

### Disposition B: Deferred — Defer Until PDMP IG Reaches Active Development

**Action:** Defer this ticket until the PDMP on FHIR IG (PSS-2225) is actively under development with an assigned editor and established timeline. Track as part of the batch of PDMP-old-issues tickets for coordinated resolution.

**Rationale:** While the request is technically valid, the IG was previously withdrawn and all ~30 PDMP-old-issues tickets were deferred in January 2023 by a 4-0-0 vote. The IG has been reopened under PSS-2225, but the ticket remains unassigned and no active editing work is underway. Addressing this mapping in isolation without the broader IG context (which version of NCPDP SCRIPT, which profiles are in scope, etc.) risks producing guidance that will need rework. It may be more efficient to address the entire batch of NCPDP mapping tickets (FHIR-29459, 29460, 29463, 29464, 29466, 29471, 29472, 29473) together when the IG editor is ready.

**Vote Suggestion:** Deferred.

---

### Disposition C: Considered for Future Use — Revisit When NCPDP SCRIPT Version Is Finalized in IG

**Action:** Resolve as "Considered for Future Use" with a note that this should be addressed once the PDMP IG finalizes which NCPDP SCRIPT version(s) are in scope (per FHIR-29460) and the overall mapping approach is established.

**Rationale:** The Quantity subfield mapping depends on decisions not yet made — specifically which NCPDP SCRIPT version the IG will target (FHIR-29460) and how granular the overall NCPDP-to-FHIR mapping tables will be. This ticket was previously resolved this way in 2023 and the underlying dependency (IG development status) has not materially changed. However, this risks a repeat cycle of deferral and reopening.

**Vote Suggestion:** Considered for Future Use.

## Recommendation

**Disposition A: Applied** is recommended. The request is straightforward and technically sound — the NCPDP SCRIPT `<Quantity>` field has well-known subfields that map naturally to FHIR's `SimpleQuantity` datatype used in `MedicationDispense.quantity`. Adding a granular mapping table is low-risk editorial work that would benefit implementers regardless of other IG-level decisions. This ticket has already been deferred once (2023) and reopened; applying it now demonstrates progress on the batch of PDMP mapping issues and provides a template for resolving the sibling tickets (FHIR-29471, 29464, 29466, etc.) in the same pattern.
