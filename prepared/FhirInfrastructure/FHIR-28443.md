## Description

Keith W. Boone requests that FHIRPath be extended with new operations (or a family of functions) to compare `date`, `dateTime`, `instant`, and `Period` values against each other, replicating the comparison logic described in [FHIR Search prefixes](https://www.hl7.org/fhir/search.html#prefix) (eq, ne, ge, gt, le, lt, sa, eb, ap). The rationale is that getting these comparisons right in FHIRPath expressions today is challenging, and utility functions would simplify the task for implementers.

**Proposed syntax options:**
1. `sourceCollection.compare(String code, Expression targetCollection)` — where `code` is a search prefix (eq, ne, ge, gt, le, lt, sa, eb, ap) and the function returns true if any source element matches any target element per the code.
2. A family of named functions: `compareEq()`, `compareNe()`, `compareGe()`, etc.

The submitter notes this approach could also generalize to numbers and ranges.

## Resolution: Not Persuasive

The resolution states that FHIR search uses FHIRPath expressions to extract properties from data for creating native indexes, not for evaluating comparisons at search time. The resolution notes that a clear use case would be needed to justify adding this functionality above what FHIRPath already provides with its native `=`, `>`, `>=`, `<`, etc. operators.

## Proposed Dispositions

### Disposition A: Not Persuasive (Uphold Current Resolution)

**Rationale:** The current resolution is well-reasoned. FHIRPath already provides native comparison operators (`=`, `>`, `>=`, `<`, `<=`, `!=`) for date, dateTime, and instant values. The proposed functions duplicate FHIR search prefix semantics in a context (FHIRPath evaluation) where they don't naturally belong — FHIR search works by extracting data into indexes, not by evaluating prefix-style comparisons at runtime. Two separate Zulip threads over four years failed to generate community support or even a single concrete use case from the submitter or others. The lack of any real-world example where the existing FHIRPath operators are insufficient undermines the case for adding new functions.

### Disposition B: Persuasive with Modification — Request Concrete Use Cases First

**Rationale:** The underlying idea has merit — comparing date/dateTime/instant values against Period boundaries using search-like semantics could be valuable in scenarios such as decision support rules, Questionnaire enableWhen logic, or StructureDefinition invariants. However, the proposal is too abstract without demonstrated need. Accept the ticket with modification: request that the submitter (or community) provide at least two concrete FHIRPath use cases where existing operators are insufficient or overly complex, and then design targeted functions addressing those specific needs rather than a broad `compare()` family.

### Disposition C: Not Persuasive — Defer to FHIRPath Community

**Rationale:** Since ITS/FHIRPath owns the FHIRPath specification, and since no community engagement materialized despite two Zulip threads over multiple years, close as Not Persuasive. If future need arises, a new ticket with concrete use cases can be submitted directly to the FHIRPath community, which would be the appropriate venue.

## Recommendation

**Disposition A: Not Persuasive (Uphold Current Resolution)**

The evidence strongly supports upholding the current resolution. After nearly six years, neither the submitter nor the community has provided a single concrete use case where FHIRPath's existing comparison operators are inadequate. Two Zulip outreach efforts (2022 and 2025–2026) generated no community support. Brian Postlethwaite's analysis correctly identifies that FHIR search prefix semantics operate at the indexing layer, not the FHIRPath expression evaluation layer, making the conceptual mapping inappropriate. The absence of demonstrated need, combined with the architectural mismatch between search-time indexing and FHIRPath evaluation, makes this a clear Not Persuasive.
