## Summary

This ticket requests clarification of two gaps in the FHIR search specification regarding the multi-value separator and its escaping:

1. **Normative language for the comma separator**: The [search multi-value section](https://build.fhir.org/search.html#multivalue) states that OR-values **can** be separated by commas, but does not say they **shall** be. This makes the comma separator's role ambiguous—it reads as optional rather than normative.

2. **Escape mechanism for literal commas in values**: There is no documented way to include a literal comma within a search parameter value when the comma is also the OR-separator. The existing [escaping section](https://build.fhir.org/search.html#escaping) covers `$`, `,`, and `|` with backslash escaping, but the multi-value section does not cross-reference this or explain how to disambiguate.

The ticket was raised as a follow-up to [FHIR-54778](https://jira.hl7.org/browse/FHIR-54778) and [FHIR-54806](https://jira.hl7.org/browse/FHIR-54806), which identified these same issues in the ImagingStudy and ImagingSelection search parameter "Multiples" rows. Both of those tickets were resolved as "Not Persuasive" by the Imaging Integration work group, with a note that the issue is really about the core search specification text rather than the per-resource generated documentation—leading to this new ticket being filed against FHIR Infrastructure.

## Details

### Description

The submitter notes that the auto-generated "Multiples" rows for search parameters (e.g., in [ImagingStudy](https://build.fhir.org/imagingstudy-search.html) and [ImagingSelection](https://build.fhir.org/imagingselection-search.html)) contain two bullets:

- The **first bullet** describes AND behavior (repeating the parameter with `&`), but does not explicitly state the separator mechanism.
- The **second bullet** describes OR behavior and says values are "separated by commas," but does not explain how to escape a comma when it appears as part of a value.

When directed to the core search spec at [search.html#multivalue](https://build.fhir.org/search.html#multivalue), the submitter found:

- The language uses "can" rather than "shall" for comma-separated OR values, making it unclear whether the comma is the **required** separator or merely one possible approach.
- There is no mention of how to escape the comma when a search value legitimately contains one.

The submitter also notes that since these Multiples rows are auto-generated, any fix to the core search specification text and/or the generation tooling would propagate to all resources.

## Proposed Dispositions

### Disposition A: Accept — Strengthen Normative Language and Add Cross-Reference to Escaping

**Resolution: Persuasive**

Accept the request and make two targeted changes to the core search specification:

1. **In [search.html#multivalue](https://build.fhir.org/search.html#multivalue)**: Change the language from "values **can** be separated by commas" to "values **shall** be separated by commas" (or equivalent normative phrasing such as "are separated by commas") to make the comma the unambiguous OR separator.

2. **In [search.html#multivalue](https://build.fhir.org/search.html#multivalue)**: Add an explicit cross-reference to the [escaping section](https://build.fhir.org/search.html#escaping), noting that when a search parameter value contains a literal comma, it must be escaped with a backslash (`\,`) as described in the escaping rules. This makes the existing escape mechanism discoverable from the multi-value section.

3. **Review the auto-generated Multiples text** for all resources to ensure the generated bullets reference or are consistent with the updated core search text.

**Rationale**: The escaping mechanism already exists (backslash escaping documented in search.html#escaping, previously clarified by FHIR-17464 and FHIR-42834). The gap is that the multi-value section does not use normative language for the comma separator and does not point readers to the escaping section. This is a documentation/cross-referencing fix, not a behavioral change.

### Disposition B: Accept with Modification — Clarify Without Changing Normative Force

**Resolution: Persuasive with Modification**

Address the discoverability concern without changing the normative force of "can" to "shall":

1. **In [search.html#multivalue](https://build.fhir.org/search.html#multivalue)**: Retain the current "can" language but add a note clarifying that the comma is the **standard** OR separator in FHIR search, and that when a value contains a literal comma, it must be escaped per the [escaping rules](https://build.fhir.org/search.html#escaping).

2. **In the auto-generated Multiples rows**: Add a brief note or link to the escaping section so that readers encountering the per-resource documentation can find the relevant rules.

**Rationale**: Some implementers may intentionally avoid comma-separated OR syntax in favor of repeating the parameter (e.g., `?status=active&status=resolved`), which is also valid. Keeping "can" preserves flexibility while the added cross-reference addresses the escaping gap. This is a more conservative change that resolves the practical confusion without altering the semantics of "can" vs. "shall."

### Disposition C: Decline — Existing Specification Is Sufficient

**Resolution: Not Persuasive**

Decline the request on the grounds that the existing specification already addresses both concerns:

- The comma as OR separator is described in [search.html#multivalue](https://build.fhir.org/search.html#multivalue).
- The backslash escaping mechanism for commas, pipes, and dollar signs is documented in [search.html#escaping](https://build.fhir.org/search.html#escaping).
- Previous tickets (FHIR-17464, FHIR-42834) have already clarified the escaping rules.

The predecessor tickets FHIR-54778 and FHIR-54806 were resolved as Not Persuasive for similar reasons.

**Rationale**: The information is present in the specification, albeit across two sections. Adding further cross-references may be helpful but is not strictly necessary. However, this disposition risks continued confusion among implementers who encounter the Multiples rows without navigating to the escaping section.

## Recommendation

**Disposition A: Accept — Strengthen Normative Language and Add Cross-Reference to Escaping**

The submitter raises a legitimate readability and discoverability issue. The predecessor tickets (FHIR-54778, FHIR-54806) were correctly redirected from Imaging Integration to FHIR Infrastructure since the problem lies in the core search specification text, not in any resource-specific content. The escaping mechanism for commas already exists via backslash escaping (as clarified by FHIR-17464 and FHIR-42834), but the multi-value section neither uses normative language for the comma separator nor cross-references the escaping section. Strengthening the wording to "shall" and adding an explicit link to the escaping rules would close this gap with minimal specification impact and no behavioral change for conformant implementations.
