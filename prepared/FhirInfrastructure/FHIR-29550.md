## Summary

This ticket requests that the FSH (FHIR Shorthand) specification document the behavior whereby SUSHI automatically populates required fixed (or pattern) values in generated instances. Currently, when a profile constrains an element to be required (e.g., min cardinality ≥ 1) and also fixes or patterns its value, SUSHI will automatically include that value in any instance declared as `InstanceOf` that profile — without the author needing to explicitly assign it. For example, instances of `USCoreLaboratoryResultObservationProfile` automatically receive the "laboratory" category code because the profile requires the category slice and fixes its value. The submitter (who is a SUSHI maintainer) wants this useful behavior to be explicitly documented in the FSH specification so authors are aware of it and can take advantage of it when writing examples.

## Details

The behavior in question is a SUSHI implementation feature where instance generation inspects the profile constraints:

1. **Required + Fixed**: If an element has a minimum cardinality of 1 or more and a `fixedValue`, SUSHI automatically assigns that value in the generated instance.
2. **Required + Pattern**: Similarly, if an element is required and has a `patternValue`, SUSHI applies the pattern as a default in the generated instance.
3. **Optional elements are excluded**: Elements that are 0..* or 0..1 with fixed/pattern values are NOT automatically populated — only required elements get this treatment.

This behavior is confirmed by Chris Moesel (SUSHI maintainer) in a December 2024 Zulip discussion where he explains: *"SUSHI only auto-populates elements that have a fixed/pattern value and are required (e.g., min = 1). Since they're not required, SUSHI does not add them to instances unless that element path is used in the FSH Instance's rules."*

The request is purely a documentation change to the FSH specification — no change to SUSHI's runtime behavior is proposed.

## Proposed Dispositions

### Disposition A: Persuasive (Accept — Document the Behavior)

**Recommendation:** Add a section or paragraph to the FSH specification (in the Instances chapter) that explicitly documents SUSHI's behavior of automatically applying required fixed and pattern values to generated instances.

**Rationale:** This is a purely additive documentation change proposed by the SUSHI maintainer himself. The behavior already exists and is relied upon by authors, but is not documented in the specification. Documenting it would:
- Help FSH authors understand why certain values appear in generated instances without explicit assignment
- Clarify the conditions under which auto-population occurs (element must be required AND have a fixed/pattern value)
- Reduce confusion when optional fixed/pattern values are NOT auto-populated

**Suggested wording (for the Instances section):**
> When SUSHI generates an instance from a profile, it automatically assigns values for elements that are both required (minimum cardinality ≥ 1) and constrained with a fixed or pattern value in the profile. Authors do not need to explicitly assign these values in their FSH Instance rules. Elements that are optional (minimum cardinality = 0) with fixed or pattern values are not automatically populated unless explicitly referenced in the instance's rules.

### Disposition B: Persuasive with Modification (Accept with Expanded Scope)

**Recommendation:** Accept the request to document auto-population behavior, but expand the documentation to also include:
1. A worked example (e.g., the USCoreLaboratoryResultObservationProfile category example from the ticket)
2. An explanation of the distinction between `fixedValue` and `patternValue` behavior in instance generation
3. A note about what happens when a profile inherits required fixed values from a parent profile

**Rationale:** Since this is a frequently misunderstood behavior (as evidenced by the 2024 Zulip thread where an author was confused about inconsistent auto-population), a more thorough documentation treatment with examples would be more valuable than a simple paragraph.

### Disposition C: Not Persuasive (Decline)

**Recommendation:** Decline the request, noting that this is a SUSHI implementation detail rather than a FSH language specification concern.

**Rationale:** The FSH specification defines the language syntax and semantics, while SUSHI is a specific implementation (compiler) of that language. Auto-population of required fixed values during instance generation is a compiler behavior decision, not a language feature. Other FSH implementations might handle instance generation differently. Documenting SUSHI-specific behavior in the FSH spec could create normative expectations that constrain future implementations.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The auto-population behavior is important for FSH authors to understand, and the Zulip evidence shows it is a recurring source of confusion. However, a simple statement is insufficient — the documentation should include a concrete example and clearly distinguish the conditions (required + fixed/pattern = auto-populated; optional + fixed/pattern = not auto-populated). This aligns with the approach taken in FHIR-42335, which also expanded FSH spec documentation about fixed value semantics. The concern about SUSHI-specific behavior (Disposition C) is mitigated by the fact that the FSH spec already describes instance generation semantics, so this is a natural extension of existing specification content.
