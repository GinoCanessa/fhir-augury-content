## Summary

The FHIR [formats page](https://build.fhir.org/formats.html#table) mentions an "NE" flag in its descriptive text — specifically in the statement: *"Extensions are not always shown, but may appear except where the flag NE appears."* However, this "NE" flag is **not listed** in the "Key to Flags" legend on the same page, and it does not appear to be actively used on any element in the specification. The reporter requests either removing the reference to the NE flag from the text or adding it to the Key to Flags legend.

## Details

### Description

The reporter identified an inconsistency on the resource formats documentation page. The page text states:

> Any of the elements may have an `id` attribute to serve as the target of an internal reference. The `id` attribute is not shown in this format. Extensions are not always shown, but may appear except where the flag `NE` appears.

The "NE" flag (meaning "This element cannot have extensions") is referenced in this paragraph but is absent from the "Key to Flags" section above it. Additionally, the flag does not appear to be applied to any element in the current specification.

The reporter suggests two options:
1. **Remove** the text about the "NE" flag, or
2. **Add** the "NE" flag to the "Key to Flags" legend.

## Proposed Dispositions

### Disposition A: Accept — Add NE flag to Key to Flags and apply it

**Action:** Add the "NE" flag (meaning "This element cannot have extensions") to the "Key to Flags" legend on the formats page, and apply the NE flag to the three elements that cannot have extensions: `Element.id`, `Resource.id`, and `Extension.url`.

**Rationale:** The NE flag concept is valid and serves a useful purpose — certain elements genuinely cannot have extensions. Adding the flag to the legend and applying it to the appropriate elements makes the documentation complete and consistent. This aligns with Grahame Grieve's confirmation (in the Zulip thread) that these three elements exist but were never flagged.

### Disposition B: Accept — Remove NE flag reference from text

**Action:** Remove the sentence fragment *"except where the flag NE appears"* from the descriptive text on the formats page, since the flag is not in use and is not defined in the legend.

**Rationale:** This is the simpler fix. Since the NE flag has never been applied to any element in the specification (as confirmed by Grahame Grieve in the Zulip discussion), the reference to it in the descriptive text is dead documentation. Removing it eliminates confusion without requiring any structural changes to element definitions. If the flag is needed in the future, it can be re-introduced with proper documentation.

### Disposition C: Decline — No action needed

**Action:** Leave the documentation as-is.

**Rationale:** The NE flag text is informational and does not cause functional issues. Since no elements currently carry the flag, the reference is harmless. However, this leaves a documented feature that is never used and not defined in the legend, which is inconsistent and has already caused confusion (as evidenced by both this ticket and the 2022 Zulip thread).

## Recommendation

**Disposition B: Accept — Remove NE flag reference from text.**

This is the most appropriate resolution because:
1. The NE flag has never been applied to any element in the specification, making the text reference orphaned documentation.
2. Grahame Grieve confirmed in 2022 that the three elements that cannot have extensions (`Element.id`, `Resource.id`, `Extension.url`) are not marked with NE, suggesting this flag was never operationalized.
3. As a Technical Correction, removing dead text is the cleanest fix with zero risk of breaking changes.
4. If there is future desire to mark extension-prohibited elements, the flag can be properly introduced at that time with both legend entry and actual usage.

Disposition A (adding and applying NE) would also be valid but is a larger change requiring updates to element definitions across the spec, which goes beyond a simple technical correction.
