Keith W. Boone asks whether aliases should expand inside other alias definitions in FHIR Shorthand (FSH). Currently, if you define `Alias: $Base = http://hl7.org/fhir/us/saner`, you cannot reference `$Base` inside a subsequent alias such as `Alias: Populations = $Base/ValueSet/Populations`—the user must spell out the full URL. The ticket links to [FHIR/sushi#356](https://github.com/FHIR/sushi/issues/356) as proposed wording.

## Proposed Dispositions

### Disposition A: Persuasive — Specify Alias Expansion Rules

**Action**: Add a specification section defining that aliases may be expanded inside other alias definitions under the following constraints:
1. The alias reference must appear at the **start** of the alias value.
2. It must be followed by `/`, `?`, or whitespace (or be the entire value).
3. Alias processing order must be deterministic (aliases are expanded in declaration order or require forward-declaration rules).

**Rationale**: This feature was previously approved in concept (14-0-0) and deferred only for timing. The community has provided a concrete, safe expansion algorithm (proposed by Chris Moesel) that avoids accidental substring matching. The `$` prefix convention (strengthened by FHIR-33561) further reduces ambiguity risk. The use case is practical—IG authors frequently define base URLs and then derive child URLs from them, and currently must duplicate long URL strings.

**Risks**: Adds complexity to FSH preprocessing; edge cases like the SNOMED version URL pattern (`SCT|SCTIntlMar2020`) would not be addressed by these rules and might cause user confusion about what is and isn't expanded.

### Disposition B: Considered for Future Use

**Action**: Acknowledge the feature request as valid but defer to a future FSH specification cycle. No specification changes at this time.

**Rationale**: While the concept has merit and prior support, the ticket has been open for over 4 years without implementation pressure. The FSH ecosystem has matured with FHIR-42298 and FHIR-33561 clarifying alias semantics, and the community may benefit from a more comprehensive alias redesign (covering the SNOMED version URL case and other edge cases) rather than an incremental fix. Deferring allows more implementation experience to accumulate.

**Risks**: Continues the status quo where authors must duplicate long URLs. The feature was already deferred once and re-opened; repeated deferral may signal lack of progress.

### Disposition C: Not Persuasive

**Action**: Close with no change. Alias expansion inside aliases is not needed in the specification.

**Rationale**: The current alias mechanism is sufficient for its intended purpose (providing shorthand for full URLs/OIDs). Authors can define each alias independently with its full URL. The substring-matching risks and processing-order complexities introduced by nested alias expansion outweigh the convenience benefit. The GitHub issue was closed as "not planned" by the SUSHI team.

**Risks**: May frustrate users who expect this natural behavior. The GitHub closure was procedural (redirecting to the spec ballot), not a technical rejection. The prior 14-0-0 vote in favor suggests community desire.

## Recommendation

**Disposition A: Persuasive — Specify Alias Expansion Rules**

This feature had unanimous conceptual support (14-0-0) and was deferred only because of STU1 timing. A safe, well-defined expansion algorithm has been proposed by the SUSHI team lead (Chris Moesel) and endorsed by Mark Kramer. The `$` prefix convention (now strengthened by FHIR-33561) provides additional safety. The use case is practical and common among IG authors who define hierarchical URL namespaces. Four years of deferral is sufficient; the feature should be formally specified with the proposed start-of-value expansion rules.
