## Summary

This change request proposes adding syntax to FHIR Shorthand (FSH) that allows authors to create new items at the end of an array and/or reference the last item in an array, without needing to know or manage explicit numeric indices. The current approach requires manually numbered indices (e.g., `name[0]`, `name[1]`), which is brittle: inserting or removing an element forces renumbering of all subsequent indices. The ticket was originally filed by Chris Moesel (a core FSH author) and was initially resolved as "Considered for Future Use" with a 13-0-0 vote (Kramer/Rhodes) during STU1 ballot reconciliation. It was subsequently re-opened after STU1 publication for further consideration.

The proposed approach introduces two special index markers:
- `[+]` — add a new item to the array (append)
- `[ ]` or `[=]` — refer to the last/current item in the array

This would allow authors to build arrays without explicit numbering, making FSH more maintainable, especially in mixins/RuleSets where the array length is unknown.

---

## Details

### Problem Statement

FSH's current array indexing requires explicit numeric indices. This creates several pain points:

1. **Brittleness**: Inserting or removing an element in the middle of an array requires renumbering all subsequent elements.
2. **Mixin/RuleSet incompatibility**: RuleSets that add array elements cannot know the correct index to use because it depends on where/how the RuleSet is applied.
3. **Error-prone**: Large arrays (e.g., 50 items) are prone to index errors (duplicate or skipped indices) that may not be caught by tooling.

### History

- **2020-05-19**: Ticket created by Chris Moesel proposing `[+]` syntax for appending to arrays.
- **2020-06-05**: Chris Moesel noted openness to alternative syntax beyond `[+]`.
- **2020-07-15**: Extended Zulip discussion on array syntax. Proposed `[+]` (new item) and `[ ]` (last item). Community raised concerns about procedural semantics in a declarative language, and potential for silent errors when lines are reordered.
- **2020-08-03**: Resolved as "Considered for Future Use" (13-0-0 vote, Kramer/Rhodes) — path forward was unclear despite significant user input.
- **2020-08-27**: Updated proposal settled on `[+]` (next/append) and `[=]` (same element) syntax. This was folded into a broader proposal including parameterized RuleSets.
- **2020-10-12**: Re-opened by Lynn Laakso after STU1 publication for reconsideration.
- **2021-07**: Related Zulip discussion on caret rules overwriting array elements reinforced the need. Community participants (Mark Kramer, Martin Höcker) supported redefining `[+]` to always mean "append."

### Current State

The ticket remains **Triaged** and **Unresolved**. The "soft indexing" feature using `[+]` and `[=]` was ultimately implemented in a later version of the FSH specification (visible in FHIR-30708 referencing soft indexing examples), but this ticket tracks the original proposal and may still have open design questions about edge cases (e.g., behavior of `[+]` after an explicit index).

## Proposed Dispositions

### Disposition A: Accept — Confirm Soft Indexing as Specified

**Resolution:** Applied — Resolved with no further spec change needed.

The soft indexing feature (`[+]` for next/append, `[=]` for current element) has already been incorporated into the FHIR Shorthand specification and implemented in SUSHI. The ticket's original request has been substantively addressed. Close this ticket as resolved, confirming the feature is in place. If any remaining edge cases exist (e.g., behavior of `[+]` after an explicit numeric index), they should be filed as separate, targeted tickets.

### Disposition B: Accept with Further Refinement — Clarify Edge Case Semantics

**Resolution:** Applied — Resolved with additional clarifying language.

Accept the soft indexing feature as implemented, but add normative clarification in the spec regarding:
1. The precise behavior of `[+]` when used after an explicit numeric index (e.g., after `array[2]`, does `array[+]` mean `array[3]` or `array[end+1]`?).
2. The interaction of soft indexing with nested arrays (index reset semantics when a parent index is bumped).
3. Document that `[+]` in caret rules appends to existing arrays (not starting from 0).

This addresses Chris Moesel's 2021 caution that the semantics need careful definition.

### Disposition C: Decline — Defer Pending Broader Syntax Redesign

**Resolution:** Deferred — Consider for future use.

Defer this ticket until a broader array/nesting syntax proposal (including the `with` syntax suggested by Jose Costa Teixeira) is developed. The current `[+]`/`[=]` approach has known usability concerns (silent errors from line reordering, confusion about procedural semantics in a declarative language). A more comprehensive solution that addresses both array population and deeply nested paths would provide a more coherent user experience.

---

## Recommendation

**Disposition A: Accept — Confirm Soft Indexing as Specified.**

The soft indexing feature (`[+]` and `[=]`) has been implemented in both the FSH specification and SUSHI tooling, and is actively used by the community (as evidenced by FHIR-30708 referencing soft indexing examples and ongoing Zulip discussions about its behavior). The original ballot vote was 13-0-0 in favor of considering it, and community feedback has been broadly supportive. The ticket's core request — a syntax to add to or reference the end of an array without explicit numeric indices — has been fulfilled. Closing this ticket acknowledges the work done and keeps the backlog clean. Any remaining edge-case clarifications (per Disposition B) can be tracked as separate, focused tickets.
