## Summary

This is a tracking ticket filed by Bas van den Heuvel noting that a series of 11 previously submitted signature-related ballot comments (FHIR-52810 through FHIR-52820) were **not addressed** in the FHIR R6 ballot. The reporter reviewed the R6 ballot version and found that none of the agreed resolutions from these earlier tickets had been applied to the specification text. These tickets cover critical aspects of FHIR digital signature infrastructure, including Bundle signing references, Provenance target semantics, canonicalization rules, terminology clarifications, serialization precision, and advanced signature profile recommendations (JAdES, XAdES).

Notably, four of the referenced tickets (FHIR-52810, FHIR-52813, FHIR-52818, FHIR-52819) were resolved by the Security WG with "Persuasive" or "Persuasive with Modification" dispositions and marked as Applied, yet the reporter found no corresponding changes in the ballot text. The remaining seven tickets under FHIR Infrastructure remain Triaged/Unresolved.

### Key Issues by Category

**Bundle Signing References (FHIR-52810, FHIR-52811):** The current spec uses `Bundle/[id]` in `Provenance.target` to reference the Bundle for signing. Since Bundle ids can change during transmission, the agreed resolution for FHIR-52810 was to use pointer-based fragment `#/` instead. FHIR-52811 notes the meaning of `#` as a Provenance target is ambiguous.

**Canonicalization Rules (FHIR-52812, FHIR-52814, FHIR-52815):** Several tickets request clarification or expansion of canonicalization rules for document signing — specifically what elements are excluded (signature data, meta elements) and whether additional canonical rules are needed for signing the whole Bundle including the Provenance.

**Countersigning (FHIR-52813):** The specification text for countersigning (section 6.1.2.2.9, points 4-6) is unclear about whether adding a target to a referred Provenance is allowed during countersigning.

**Terminology & Serialization (FHIR-52816, FHIR-52817):** The term "electronic signature" is confusing (could mean public-key signing), and the JWS serialization format (compact vs. JSON) needs to be specified explicitly rather than just saying "base64 encoded."

**Advanced Signature Profiles (FHIR-52818, FHIR-52819, FHIR-52820):** Recommendations to add JAdES-B-T/XAdES-B-T for non-repudiation and JAdES-B-LTA/XAdES-B-LTA for long-term signatures, along with corresponding worked examples.

## Proposed Dispositions

### Disposition A: Accept — Comprehensive Resolution of All Referenced Tickets

**Action:** Accept this tracking ticket and commit to resolving all 11 referenced tickets (FHIR-52810 through FHIR-52820) as part of a coordinated signature specification update in R6. Specifically:

1. **FHIR-52810:** Replace `Bundle/[id]` references in `Provenance.target` with pointer-based fragment `#/` throughout the signing specification and worked examples.
2. **FHIR-52811:** Clarify that `Provenance.target` value `#` means the Provenance signs itself in addition to the Bundle, and rewrite the ambiguous sentence in section 6.1.2.2.9.
3. **FHIR-52812:** Determine whether `#` scope includes the entire Bundle with Provenance; if not, define additional canonicalization rule URIs (e.g., `{fmt}#documentNoSignInfo`).
4. **FHIR-52813:** Clarify countersigning rules in section 6.1.2.2.9 points 4–6, explicitly stating how a countersigning Provenance references the original signing Provenance.
5. **FHIR-52814:** Update canonicalization rules for `{fmt}#document` to explicitly state which elements are excluded (Bundle.id, Bundle.meta.lastUpdated, Bundle.meta.versionId, Bundle.signature, and signature.data in the current Provenance.signature).
6. **FHIR-52815:** For `{fmt}#static`, change to only remove `resource.id`, `resource.meta.versionId`, and `resource.meta.lastUpdated`, preserving other meta elements such as security tags.
7. **FHIR-52816:** Replace "electronic signature" terminology with "image of an ink signature" or similar non-ambiguous phrasing.
8. **FHIR-52817:** Specify that JWS uses compact serialization per RFC 7515.
9. **FHIR-52818/52819:** Add MAY recommendations for JAdES-B-T and XAdES-B-T for non-repudiation, and SHOULD recommendations for JAdES-B-LTA/XAdES-B-LTA for long-term signatures.
10. **FHIR-52820:** Add worked examples for JAdES-B-T and JAdES-B-LTA signed Bundles.

Coordinate with Security WG to ensure the four Applied tickets (FHIR-52810, 52813, 52818, 52819) have their agreed resolutions actually published.

**Rationale:** The reporter has documented a genuine gap — previously agreed changes were not applied. This is a process issue that should be corrected.

### Disposition B: Accept with Partial Scope — Prioritize Applied Tickets

**Action:** Accept this ticket with a narrower scope: immediately apply the four Security WG tickets that already have "Applied" status (FHIR-52810, FHIR-52813, FHIR-52818, FHIR-52819) since their resolutions were previously agreed. For the seven remaining FHIR Infrastructure tickets (FHIR-52811, 52812, 52814, 52815, 52816, 52817, 52820), schedule them for workgroup discussion and resolution in a subsequent ballot cycle, as they represent substantive specification changes requiring further deliberation.

Specific immediate actions:
1. Apply pointer-based fragment `#/` for Bundle signing references (per FHIR-52810 resolution).
2. Clarify countersigning rules (per FHIR-52813 resolution).
3. Add JAdES-B-T/XAdES-B-T MAY recommendations (per FHIR-52818/52819 resolutions).

**Rationale:** The Applied tickets represent agreed work that was simply not published — this is low-risk to apply. The remaining tickets require substantive design decisions and would benefit from dedicated workgroup review.

### Disposition C: Decline — Defer to Individual Tickets

**Action:** Decline this tracking ticket as a duplicate/meta-tracker. The individual referenced tickets (FHIR-52810 through FHIR-52820) already exist with their own status tracking. Close this ticket and ensure each referenced ticket is tracked independently through its assigned work group. Add a note to each unresolved ticket to indicate continued review is needed for R6.

**Rationale:** Tracking tickets can create confusion about which ticket to action. Each issue is distinct enough to be managed individually, and four have already been resolved by Security WG. Closing this meta-ticket avoids double-tracking.

## Recommendation

**Disposition A: Accept — Comprehensive Resolution of All Referenced Tickets.**

The reporter has identified a clear process failure: multiple ballot comment resolutions (including four marked as "Applied") were never published in the R6 ballot text. This affects fundamental digital signature infrastructure that implementers depend on. The issues are well-documented, specific, and in most cases already have agreed resolutions. A coordinated effort to address all 11 tickets ensures consistency across the interrelated signature specification sections. FHIR Infrastructure should coordinate with Security WG to publish the already-agreed changes and work through the remaining open items, which are largely clarifications and improvements rather than controversial changes.
