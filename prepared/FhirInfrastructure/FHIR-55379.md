## Summary

Brian Postlethwaite requests extending the FML (FHIR Mapping Language) tutorial (`mapping-tutorial.html`) to cover more of FML's features. The current tutorial steps only cover the very basics, leaving users without guidance on more advanced features such as explicit rule naming, `resolve()` usage, complex nested rules, FHIRPath expressions in conditions, and other capabilities of the language.

This ticket is part of a broader set of FML-related issues filed by the same reporter during the R6 ballot cycle, including a companion ticket to remove the tutorial from the core specification entirely ([FHIR-54486](https://jira.hl7.org/browse/FHIR-54486)) and a ticket to fix invalid FHIRPath syntax in existing tutorial steps ([FHIR-54919](https://jira.hl7.org/browse/FHIR-54919)). These tickets reflect a strategic view: if the tutorial remains in the spec, it should be comprehensive; alternatively, FML content (including the tutorial) could move to its own specification.

### Context

The FML tutorial is currently located at `https://build.fhir.org/mapping-tutorial.html` and consists of approximately 14 steps covering basic mapping concepts. Community discussions reveal several areas where the tutorial falls short:

1. **Rule naming**: Users encounter "Complex rules must have an explicit name" errors without understanding why names are required or where to place them (Zulip: "FHIR Mapping Language > ✔ Explicit name").
2. **`resolve()` behavior**: Step 14 of the tutorial suggests following a reference is "implicit," but the actual behavior with `resolve()` in FML contexts is poorly documented and inconsistently implemented across engines (Zulip: "FHIR Mapping Language > resolve() references within a Bundle").
3. **FHIRPath expressions**: Existing tutorial examples contain invalid FHIRPath syntax, such as missing required brackets around embedded expressions ([FHIR-54919](https://jira.hl7.org/browse/FHIR-54919)).
4. **Advanced features**: Nested group calls, `then` blocks, `where` clauses, type checks (`ofType`), string concatenation, logging, and `evaluate()` are all used in practice but not covered by the tutorial.

## Proposed Dispositions

### Disposition A: Accept as Persuasive — Extend the Tutorial In-Place

Accept the request and extend the existing FML tutorial at `mapping-tutorial.html` with additional steps covering:
- Explicit rule naming (when required, syntax, purpose)
- Nested `then` blocks and complex rule structures
- `where` clauses with proper FHIRPath bracket syntax
- `evaluate()` and `resolve()` usage
- Working with Bundles and references
- Logging and debugging techniques
- Type checking with `ofType`

**Rationale**: The tutorial is already in the spec and users demonstrably struggle with undocumented features. Extending it addresses immediate usability gaps. This is compatible with a future move to a separate spec — the content would simply migrate.

**Coordination**: Should be resolved in conjunction with [FHIR-54919](https://jira.hl7.org/browse/FHIR-54919) (fix invalid FHIRPath syntax) and considered alongside [FHIR-54486](https://jira.hl7.org/browse/FHIR-54486) (tutorial removal/relocation).

### Disposition B: Accept as Persuasive with Modification — Defer Pending FML Spec Decision

Accept the intent but defer execution until the work group resolves whether FML will remain in core or move to its own specification ([FHIR-54486](https://jira.hl7.org/browse/FHIR-54486), [FHIR-54473](https://jira.hl7.org/browse/FHIR-54473)). If FML moves to its own spec, the extended tutorial should be authored there rather than in core FHIR.

**Rationale**: Investing significant effort in extending the tutorial in core FHIR may be wasted if the content is subsequently relocated. The "Move FML to a new spec?" Zulip discussion shows this decision is actively being debated with key community members (Grahame Grieve, Lloyd McKenzie, Brian Postlethwaite, Oliver Egger). Waiting avoids duplicate work.

**Risk**: Delays improvement of documentation that users need now.

### Disposition C: Decline as Not Persuasive

Decline on the grounds that tutorials do not belong in the core FHIR specification. The reporter himself filed [FHIR-54486](https://jira.hl7.org/browse/FHIR-54486) arguing that the tutorial should be removed from core, as it is the only tutorial in the entire spec. Rather than extending it, the tutorial should be maintained externally (e.g., in an IG, Confluence, or a dedicated FML specification).

**Rationale**: The core specification should focus on normative content, not educational material. Community-maintained tutorials (e.g., Oliver Egger's implementations, FHIRPath Lab) already exist outside the spec.

**Risk**: Leaves the existing incomplete tutorial in place without improvement until an alternative home is established.

## Recommendation

**Disposition A: Accept as Persuasive** — Extend the tutorial in-place.

The FML tutorial is actively used by implementers and the gaps are well-documented through Zulip discussions showing real user confusion around rule naming, `resolve()`, and FHIRPath expressions. Regardless of whether FML ultimately moves to its own specification, the tutorial content needs to be written and any investment in extending it transfers directly to a new location. Deferring (Disposition B) risks leaving users without adequate documentation for an indefinite period while the spec-separation debate continues. Declining (Disposition C) is internally inconsistent — even if the tutorial moves, _someone_ needs to extend it; the question is only where it lives. The work group should accept this ticket, fix the existing content ([FHIR-54919](https://jira.hl7.org/browse/FHIR-54919)), and extend the tutorial with practical examples drawn from real community questions. If FML moves to its own spec, the improved tutorial migrates with it.
