## Summary

This ticket raises a concern about SMART App State scopes and their interaction with the Basic resource. Specifically, when a Basic resource uses the `author` field to refer to a patient, it also matches the `patient/code=...` scope — meaning the App State scopes behave differently from standard SMART scopes. The reporter asks whether this is intentional and, if the behavior is specific to App State, suggests creating App State-specific scope rules (e.g., using the profile canonical in the scope).

The ticket was previously resolved as "Considered for Future Use" with a 13-0-0 vote (2023-02-13), with the rationale that existing semantics for `patient/Basic.rs?code=` are being leveraged and App State scopes are optional. The resolution was subsequently reverted by Lynn Laakso for reconsideration post-STU2.1 publication.

## Details

### Description

When interpreting the SMART app state scopes according to the SMART on FHIR guidance, a Basic resource using the `author` field to refer to a patient also complies to the `patient/code=...` scope. The reporter questions whether this is intended behavior, noting that if it is not, the App State scopes behave differently from other SMART scopes, which could cause issues when other Basic resources are also in use. If the behavior is App State-specific, the suggestion is to create App State-specific rules, for example using the profile canonical in the scope.

## Proposed Dispositions

### Disposition A: Accept — Define App State-Specific Scope Rules

**Action:** Add guidance or normative text in the SMART on FHIR specification clarifying that App State scopes have distinct behavior from standard resource scopes when using Basic resources. Introduce App State-specific scope syntax (e.g., incorporating the profile canonical URL in the scope) to disambiguate App State Basic resources from other Basic resources that may use `Basic.author` to reference a patient.

**Rationale:** The current scope semantics create an unintended overlap where `patient/Basic.rs?code=...` can match Basic resources beyond App State entries. Since the ticket was re-opened post-STU2.1 specifically for refinement, defining explicit App State scope rules would improve clarity and prevent servers from inadvertently exposing non-App-State Basic resources through the patient compartment.

### Disposition B: Accept with Clarifying Guidance Only

**Action:** Add non-normative guidance in the SMART App State documentation explicitly noting the interaction between `Basic.author`, the Patient Compartment, and `patient/Basic.rs?code=...` scopes. Clarify that servers MAY apply additional restrictions beyond compartment-based scope enforcement to limit App State access. Do not introduce new scope syntax at this time.

**Rationale:** The prior resolution (13-0-0 vote) established that the current semantics are intentional and that servers can apply additional restrictions. Adding clear guidance acknowledges the concern without requiring breaking changes to scope syntax. This aligns with the original committee position while making the behavior explicit for implementers.

### Disposition C: Decline — Defer to Future Work

**Action:** Close as "Considered for Future Use." The current semantics are intentional and documented; servers already have flexibility to restrict scope behavior. Revisit if implementer feedback demonstrates concrete interoperability issues.

**Rationale:** The 13-0-0 vote in 2023 confirmed the design intent. App State scopes are optional, and servers are free to apply additional restrictions. Without evidence of real-world interoperability problems caused by this overlap, introducing new scope syntax or additional normative text may add unnecessary complexity.

## Recommendation

**Disposition B: Accept with Clarifying Guidance Only**

This is the most pragmatic path forward. The prior committee vote confirmed the current semantics are by design, but the ticket was deliberately re-opened for post-publication refinement. Adding explicit non-normative guidance about the `Basic.author` / Patient Compartment interaction gives implementers the clarity they need without introducing breaking changes to scope syntax. It respects the original design decision while addressing the reporter's valid concern that the behavior is non-obvious and could lead to unintended data exposure. If implementer experience later reveals concrete problems, the committee can revisit with a stronger case for Disposition A.
