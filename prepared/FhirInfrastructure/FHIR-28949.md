## Description

The submitter requests that FHIR Shorthand (FSH) support the inclusion and reuse of rules (specifically RuleSets) defined in other FSH projects. The stated motivation is to enable reuse of common conventions and patterns across implementation guides (IGs) without having to manually copy FSH files between projects.

One comment from **litlfred** expressed support (+1) for the feature.

FHIR-42620 is essentially a more developed articulation of the same feature request. Its previous disposition acknowledged the desire and raised open design questions:
- Are RuleSets the only shareable entity, or should other FSH definitions be included?
- How should FSH libraries be packaged (e.g., FHIR `Library` resource, CRMI)?
- How should they be distributed (NPM packages, GitHub repos)?
- How should they be scoped when used in a consuming project?

## Zulip Discussions

### [shorthand > Reusable RuleSets](https://chat.fhir.org/#narrow/stream/shorthand/topic/Reusable%20RuleSets)
**Date range:** 2022-01-17 to 2022-02-07  
**Participants:** Jose Costa Teixeira, David Pyke, Chris Moesel, Elliot Silver, Ward Weistra, Richard Stanley

Key points from the discussion:
- **Current workaround:** Manually copying FSH files into the `/fsh` directory; symlinks and git submodules were suggested but found to have significant limitations (submodules require explicit management, don't integrate with IG Publisher tooling).
- **Packaging options explored:**
  1. Add FSH files to standard FHIR packages (would require IGP changes to include FSH source in packages).
  2. Create FSH-only packages published to a registry (NPM?).
  3. Specify GitHub project URLs as dependencies so SUSHI can find FSH files.
  4. Simple include directories (favored by the original reporter, Elliot Silver, as the least complex solution).
- **Chris Moesel** (SUSHI maintainer) raised design considerations: should the dependency be listed in the generated IG JSON? Should authors opt-in to consuming FSH from dependencies? What FSH entities are shareable — just RuleSets and Aliases, or everything?
- **Elliot Silver** argued the package route is over-engineering the problem and recommended simple include directories.
- **Jose Costa Teixeira** noted a FHIR-only IG for common functions would be awkward (would it even build without artifacts?).

## Analysis

This is a long-standing feature request (filed October 2020) for cross-project RuleSet reuse in FSH. It has clear community support — evidenced by the +1 comment, the parallel FHIR-42620 (filed independently in 2023 with Highest priority), and the active Zulip discussion involving key FSH community members and the SUSHI maintainer.

The core request is straightforward: enable FSH authors to reference and insert RuleSets (and potentially other FSH entities) from external projects. However, the design space is non-trivial, with open questions about packaging, distribution, scoping, and which entity types to support.

Since FHIR-42620 covers the same ground with more specificity and has already been through a ballot vote (12-0-0 for future consideration, now re-opened), FHIR-28949 is effectively a precursor/duplicate.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Merge into FHIR-42620
Mark as **Persuasive with Modification**. Acknowledge the validity of the request but note that FHIR-42620 already captures this feature request with more detail and has undergone formal ballot consideration. Close FHIR-28949 as addressed by FHIR-42620 and consolidate future work under that ticket.

**Rationale:** Avoids duplicate tracking of the same feature. FHIR-42620 is actively being considered and has the richer context including ballot vote history and design questions.

### Disposition B: Deferred — Await FHIR-42620 resolution
Mark as **Deferred**. Acknowledge the request is valid and note it will be addressed as part of the broader shareable FSH effort tracked under FHIR-42620. No immediate specification change needed.

**Rationale:** Keeps the ticket open as a reference point for the original feature request while signaling that it will be handled through the existing, more detailed ticket.

### Disposition C: Considered for Future Use
Mark as **Considered for Future Use**. The WG agrees that cross-project rule inclusion would be valuable but significant design work remains (packaging, distribution, scoping). This will be considered in a future version of the FSH specification.

**Rationale:** Consistent with the 12-0-0 disposition previously applied to the closely related FHIR-42620. Keeps the ticket visible for future planning cycles.

## Recommendation

**Disposition A: Persuasive with Modification — Merge into FHIR-42620**

This is the cleanest resolution. FHIR-28949 and FHIR-42620 request the same capability. FHIR-42620 has richer detail, a ballot vote history, and is already re-opened for active consideration. Consolidating under FHIR-42620 reduces duplicate tracking overhead and ensures all community input is gathered in one place. The work group should reference FHIR-28949 in FHIR-42620 to preserve the original request history.
