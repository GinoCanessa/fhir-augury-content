## Description

Keith W. Boone requests a simplified FSH syntax for defining simple value set restrictions inline, without needing to craft a full ValueSet definition. The proposed shorthand would allow authors to write something like:

```
status = #complete | #pending
```

instead of defining a separate ValueSet with its name, id, title, and description. The submitter argues that the required metadata (name, id, description) could be auto-generated from context — for example, deriving the name from a provided title, and generating a description like "used to restrict values of fieldname in profilename." The submitter notes that having to define a full ValueSet is 3–5× more lines of FSH code for what are conceptually very simple restrictions.

The ticket references [FHIR/sushi#321](https://github.com/FHIR/sushi/issues/321) as the proposed wording / original issue.

## Analysis

This is a usability enhancement request for the FSH language. The core idea — allowing inline code restrictions without explicit ValueSet definitions — is well-motivated for simple use cases where authors only need to bind an element to a small set of codes.

**Key considerations:**

- **FHIR constraint:** FHIR profiles require a ValueSet to restrict coded elements to specific values. Any shorthand syntax would still need to produce a ValueSet artifact. The question is whether this can be automated transparently.
- **Auto-generation challenges:** Generating unique, meaningful names/ids/URLs for auto-created ValueSets is non-trivial. Naming collisions, canonical URL generation, and discoverability of implicit ValueSets are open design questions.
- **Prior assessment:** The linked GitHub issue (FHIR/sushi#321) was closed in September 2021 because the team "couldn't find a clear path to a solution." The ballot vote (13-0-0) acknowledged the idea favorably but deferred it.
- **Age and activity:** The ticket has been open since May 2020 with no activity since October 2020. The corresponding GitHub issue was closed without resolution in 2021.

## Proposed Dispositions

### Disposition A: Persuasive — Add Inline ValueSet Shorthand Syntax
Accept the request and define a new FSH syntax for inline code restrictions (e.g., `status = #complete | #pending`) that auto-generates a ValueSet artifact. Specify conventions for deriving the ValueSet name, id, canonical URL, and description from the binding context (profile name, element path). This directly addresses the usability concern and reduces boilerplate for a common authoring pattern.

**Pros:** Directly addresses a real authoring pain point; strong ballot support (13-0-0); consistent with FSH's goal of being concise.
**Cons:** Significant design work needed for auto-generation conventions (naming, URLs, collision avoidance); implicit artifacts may reduce transparency and discoverability; FHIR/sushi#321 was closed without a clear path forward.

### Disposition B: Not Persuasive — Decline Due to Unresolved Design Challenges
Reject the request on the grounds that auto-generating ValueSet artifacts introduces significant complexity and ambiguity that the FSH tooling team has been unable to resolve. The SUSHI team explicitly noted in FHIR/sushi#321 that no clear solution path was found. Authors can continue to define explicit ValueSets, which is verbose but unambiguous and transparent.

**Pros:** Avoids introducing implicit behavior and hidden artifact generation; preserves simplicity of the FSH-to-FHIR mapping.
**Cons:** Ignores a real usability problem acknowledged by ballot voters; leaves a 3–5× verbosity penalty for a very common pattern.

### Disposition C: Considered for Future Use — Defer Pending Design Proposal
Acknowledge the request as a valuable future enhancement but defer resolution until a concrete design proposal addresses the open questions: ValueSet naming conventions, canonical URL generation, collision avoidance, and discoverability of auto-generated artifacts. Encourage community input on a design document or FSH specification proposal.

**Pros:** Validates the idea without committing to an incomplete design; allows the community to develop a robust proposal; consistent with the prior 13-0-0 vote sentiment.
**Cons:** This ticket has already been deferred once (2020) and re-opened; further deferral risks indefinite stalling.

## Recommendation

**Disposition C: Considered for Future Use.** The request has clear merit and strong prior support (13-0-0 vote), but the fundamental design challenges — particularly around auto-generating ValueSet identity (name, id, canonical URL) and ensuring discoverability of implicit artifacts — remain unresolved. The SUSHI team's closure of FHIR/sushi#321 in 2021 without a solution path confirms these are non-trivial problems. Deferring with an explicit call for a design proposal gives the community a concrete next step while acknowledging the value of the request. If a design proposal emerges that addresses the naming/URL/discoverability concerns, the ticket can be re-evaluated as Persuasive.
