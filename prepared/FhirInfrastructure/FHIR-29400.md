## Description

The submitter requests updating the version/artifact file for the FHIR Shorthand specification to add version 1.0.0 to Jira. This refers to the Jira Spec Artifacts XML file (maintained in the [HL7/JIRA-Spec-Artifacts](https://github.com/HL7/JIRA-Spec-Artifacts) repository) that controls which specification versions appear in Jira dropdowns when filing issues. Without the 1.0.0 version entry, users cannot properly tag issues against the published FHIR Shorthand 1.0.0 release.

## Analysis

This ticket was filed in October 2020, shortly after the FHIR Shorthand 1.0.0 specification was published. The request is straightforward administrative maintenance: adding a version entry to the Jira Spec Artifacts XML so that Jira trackers can reference the 1.0.0 release.

**Current state context:** The 2024 Zulip discussion by Chris Moesel (a key Shorthand maintainer) indicates that the Jira spec file for Shorthand *does* currently contain version entries for 1.0.0 and 2.0.0, since the IG Publisher was proposing to *remove* them. This strongly suggests the underlying issue has already been addressed at some point in the years since this ticket was filed, even though the ticket was never formally resolved. The Shorthand specification is now at version 3.0.0-ballot, and the JIRA-Spec-Artifacts file appears to include historical versions.

**Precedent:** FHIR-40964 (Clinical Guidelines) was resolved as Persuasive for an identical issue — the fix was to run a new IG Publisher build and submit the updated XML to the JIRA-Spec-Artifacts repo.

## Proposed Dispositions

### Disposition 1: Persuasive (Applied)
**Rationale:** The request is valid and the fix (adding 1.0.0 to the Jira spec artifact XML) has effectively already been applied. Evidence from the 2024 Zulip discussion confirms that the Shorthand Jira XML file currently contains version 1.0.0. Accept the ticket as Persuasive and note it as already addressed. This aligns with the resolution of the analogous FHIR-40964 ticket.

### Disposition 2: Not Persuasive (Stale)
**Rationale:** The ticket is nearly 5 years old and the underlying issue appears to have been resolved organically through normal spec maintenance processes without formal tracking back to this ticket. The work group could close it as not persuasive on the grounds that it is stale and the issue no longer exists, though this would be somewhat unusual for a request that was clearly valid.

### Disposition 3: Persuasive with Modification
**Rationale:** Accept the ticket but broaden the scope: not only should 1.0.0 be present, but the work group should verify that *all* published Shorthand versions (1.0.0, 2.0.0, 3.0.0-ballot) are correctly represented in the Jira Spec Artifacts file and that the IG Publisher's auto-generated XML does not regress them (per the 2024 Zulip discussion). This addresses the root issue and the ongoing maintenance concern.

## Recommendation

**Disposition 1: Persuasive (Applied)** is recommended. The request was valid and straightforward, and the evidence strongly indicates the fix has already been applied. Marking it Persuasive (Applied) accurately reflects that the change was needed and has been implemented, cleanly closing out a long-standing triaged item. The broader Jira XML regression concern raised in the 2024 Zulip thread is a separate tooling issue and should be tracked independently if not already addressed.
