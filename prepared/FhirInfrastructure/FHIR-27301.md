## Summary

Rick Geimer requests that FHIR Shorthand (FSH) define a **native slicing rule** instead of requiring caret (`^`) syntax for the common task of setting up slicing discriminators. The submitter notes that slicing is one of the most frequent profiling operations, yet FSH requires authors to manually set `^slicing.discriminator.type`, `^slicing.discriminator.path`, and `^slicing.rules` via caret escape syntax — which is verbose and error-prone. The ticket was originally filed during the STU1 ballot period and was resolved as "Considered for Future Use" (voted 11-0-1, Moesel/Rhodes), then **re-opened** by Lynn Laakso after STU1 publication for continued consideration.

### Existing Wording (cited by submitter)
> "Slicing rules…Caret syntax is used to set these parameters directly"

### Core Request
Define a dedicated FSH grammar construct for slicing setup (discriminator type, path, and rules) so that authors do not need to resort to caret syntax for this very common operation.

## Proposed Dispositions

### Disposition A: Persuasive — Define Native Slicing Syntax

**Action:** Add a dedicated FSH grammar construct for defining slicing parameters (discriminator type, discriminator path, and slicing rules) that replaces the need for caret syntax in the common case.

**Rationale:** Slicing is one of the most frequent profiling tasks, and the current caret syntax is verbose, error-prone, and a significant source of community confusion (as evidenced by repeated Zulip questions and the prior 11-0-1 vote acknowledging this need). A native syntax would improve readability, reduce errors, and align FSH's expressiveness with the frequency of the operation. Mark Kramer's comment acknowledged two viable approaches — either a dedicated grammar rule or SUSHI inference — and this disposition favors the grammar approach for explicitness and author control.

**Risk:** Requires grammar design work and spec updates; may need coordination with SUSHI's existing auto-slicing behavior to avoid conflicts.

---

### Disposition B: Persuasive with Modification — Rely on SUSHI Auto-Inference Instead of New Syntax

**Action:** Rather than adding new FSH grammar, enhance SUSHI's auto-slicing capabilities so that discriminator type, path, and rules can be inferred from context (e.g., from the slice definitions and their constraints). Update the spec to document that caret syntax for slicing setup is optional when SUSHI can infer the values.

**Rationale:** This is the second approach Mark Kramer identified in his comment. SUSHI has already made progress on auto-slicing inference since STU1. Formalizing and expanding this capability avoids grammar complexity while still solving the underlying usability problem. The spec would document when inference applies and when caret syntax is still needed as a fallback.

**Risk:** Inference may not cover all slicing scenarios, leaving some cases still requiring caret syntax. Implicit behavior can be harder to debug when the inferred values are wrong.

---

### Disposition C: Not Persuasive — Maintain Current Caret Syntax for Slicing

**Action:** Close with no change. Slicing setup via caret syntax works correctly and is well-documented.

**Rationale:** Caret syntax provides a general-purpose escape mechanism that can set any StructureDefinition element. Adding special-case syntax for slicing increases the language surface area and creates a precedent for adding native syntax for every common operation. The caret syntax, while verbose, is consistent and predictable.

**Risk:** Continued user friction and support questions around slicing setup. Community sentiment (11-0-1 prior vote) clearly favors improvement.

---

## Recommendation

**Disposition A: Persuasive — Define Native Slicing Syntax**

The prior workgroup vote of 11-0-1 in favor of considering this feature, combined with persistent community confusion documented across multiple Zulip threads and related tickets, strongly supports adding a native slicing construct to FSH. Slicing is arguably the most common profiling operation that still requires caret syntax, and a dedicated grammar would bring significant usability improvement. The ticket has been open since 2020 and was explicitly re-opened after STU1 publication, signaling workgroup intent to address it. Even if SUSHI auto-inference (Disposition B) is also pursued as a complementary enhancement, having an explicit grammar option gives authors control and transparency over discriminator configuration.
