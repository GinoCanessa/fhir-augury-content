## Summary

Mark Kramer requests that the FHIR Shorthand (FSH) documentation be updated to clarify the rules around extension definition scope. Specifically, the documentation should state that **all top-level extensions must be defined as Standalone extensions**, and that **only complex extensions may use inline (local) extension definitions**. The current documentation does not make this distinction sufficiently clear, which could lead to confusion for FSH authors attempting to define extensions.

## Proposed Dispositions

### Disposition A: Persuasive

**Rationale:** Accept the request as stated. Add explicit language to the FHIR Shorthand specification documentation stating that all top-level extensions must be defined as Standalone, and only complex extensions may contain inline (local) sub-extension definitions. This directly addresses the reporter's concern with minimal specification impact.

**Suggested action:** Add a clarifying note or paragraph in the FSH extension definition section (e.g., near the Standalone extension grammar) stating:
> "All top-level extensions SHALL be defined as Standalone extensions. Inline extension definitions are only permitted as sub-extensions within complex extensions."

---

### Disposition B: Persuasive with Modification

**Rationale:** Accept the intent of the request but broaden the clarification to include additional context. In addition to stating the standalone vs. inline rule, update the documentation to include a brief example illustrating a standalone top-level extension and a complex extension with inline sub-extensions, making the distinction concrete for FSH authors.

**Suggested action:** Add both the clarifying text from Disposition A and a paired example set (one standalone top-level extension, one complex extension with inline children) in the relevant section of the FSH specification.

---

### Disposition C: Not Persuasive

**Rationale:** The existing FSH specification grammar and examples already imply the rule that top-level extensions must be standalone. The FSH compiler enforces this behavior, and experienced users understand the distinction. Additional documentation may add clutter without significant benefit.

**Suggested action:** No change. Close the ticket as not persuasive.

## Recommendation

**Disposition B: Persuasive with Modification**

This is a documentation-only change request with low risk and clear benefit. The reporter (Mark Kramer, a co-creator of FHIR Shorthand) identifies a genuine gap in clarity. The related ticket FHIR-27196 was already resolved as "Persuasive with Modification" for similar FSH documentation clarity concerns, establishing precedent. Adding both explicit text and a supporting example will make the specification more accessible to new FSH authors without altering any normative behavior. The fact that FHIR-53088 recently tightened language around complex extension profiling in FHIR Core further supports making the FSH documentation explicit about which extension types can contain inline definitions.
