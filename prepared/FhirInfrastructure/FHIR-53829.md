# FHIR-53829: Add Formal Definition and examples for SearchParameters

## Summary

**Reporter:** Eric Haas
**Status:** Triaged | **Type:** Change Request | **Priority:** Medium
**Work Group:** FHIR Infrastructure | **Specification:** FHIR Core (FHIR)
**Resolution:** Unresolved | **Assignee:** Unassigned
**Created:** 2026-01-06 | **Updated:** 2026-01-13
**URL:** https://jira.hl7.org/browse/FHIR-53829

Eric Haas requests that SearchParameter rendering in the FHIR specification be enhanced to include two additions for each SearchParameter: (1) a link to the formal definition (i.e., the JSON/XML conformance artifact), and (2) an example search URL demonstrating usage. Currently, to access a formal SearchParameter definition, a reader must download the entire FHIR package or source from the [searchparameter-registry](https://hl7.org/fhir/6.0.0-ballot4/searchparameter-registry.html). The proposal follows the existing style used for OperationDefinitions, which already provide inline links to their formal definitions within the published specification. An attached screenshot (not reproduced here) illustrates the proposed rendering changes.

## Analysis

**Core Request:** Add formal definition links and example URLs to SearchParameter pages in the published specification, matching the existing pattern used by OperationDefinitions.

**Strengths:**
- The request is straightforward and well-scoped—it asks for rendering parity with an existing pattern (OperationDefinitions).
- The rationale is sound: requiring users to download the full FHIR package just to view a SearchParameter definition is a usability barrier.
- Consistency across conformance artifacts (StructureDefinition, OperationDefinition, SearchParameter) is a longstanding community goal, as evidenced by FHIR-8363 from 2015.
- The Zulip thread on comparator rendering shows active community friction with current SearchParameter rendering limitations.

**Considerations:**
- The implementation primarily affects the FHIR publisher/build tooling rather than the FHIR specification content itself.
- The attachment showing the proposed rendering is an image and not reproduced here; reviewers should examine it on the Jira ticket directly.
- The ticket title contains a typo ("Defnition" → "Definition") which should be corrected if accepted.

## Proposed Dispositions

### Disposition 1: Persuasive
**Action:** Accept the proposal as submitted. Update SearchParameter rendering in the FHIR specification build tooling to include a link to the formal definition (JSON/XML) and an example search URL for each SearchParameter, following the OperationDefinition rendering pattern.

**Rationale:** The request is well-motivated, improves usability, and achieves consistency with OperationDefinition rendering that is already in place. The change is low-risk—it is a rendering/publishing change that does not alter the SearchParameter resource definition or any normative content. The proposal includes a clear visual mockup. This aligns with the community's long history of requesting SearchParameter parity with other conformance artifacts (cf. FHIR-8363).

### Disposition 2: Persuasive with Modification
**Action:** Accept the formal definition link but defer the example URL component. Add a link to the formal SearchParameter definition (JSON/XML artifact) on each SearchParameter page, consistent with OperationDefinition rendering. Defer the example URL portion pending further discussion on what constitutes a good example URL for different SearchParameter types (e.g., composite, reference, token).

**Rationale:** The formal definition link is unambiguously valuable and straightforward to implement. Example URLs, however, vary significantly by search parameter type and may require editorial decisions about which examples best serve readers. Splitting the work allows the high-value, low-risk portion to proceed immediately while the example URL component can be refined with community input.

### Disposition 3: Not Persuasive
**Action:** Decline the request. The formal definitions are accessible via the FHIR package and the searchparameter-registry page; adding redundant links to individual SearchParameter pages adds maintenance burden without sufficient benefit.

**Rationale:** The formal definitions are already available to those who need them (via the downloadable package or the registry). Adding links to individual pages increases the rendering and maintenance surface. IG authors who need custom SearchParameter rendering already have mechanisms (liquid templates) to generate it, as demonstrated in the Zulip thread.

---

## Recommendation

**Disposition 1: Persuasive** is recommended. The request is clear, well-scoped, and follows an established pattern already used by OperationDefinitions. Adding formal definition links and example URLs to SearchParameter pages directly improves usability and consistency with no risk to normative content. The community has repeatedly raised SearchParameter rendering as a pain point (FHIR-8363, Zulip threads), and this ticket addresses a specific, actionable gap. The implementation is primarily a build-tooling change with minimal specification impact, making it low-risk to accept in full.
