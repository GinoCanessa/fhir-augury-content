## Description

The submitter asks whether the FHIR Subscriptions Backport IG is effectively defining a WebSocket sub-protocol, and if so, why it is not formally registered as one. Per the WebSocket specification (RFC 6455), sub-protocols should be declared so that clients (e.g., the JavaScript `WebSocket` constructor) can pass the sub-protocol identifier during the handshake. The submitter's position is that the IG *is* defining a sub-protocol (message formats, binding tokens, connection semantics) and should acknowledge and formalize this.

This ticket was originally imported from a ballot spreadsheet (comment #18 by Gino Canessa). It was previously resolved as **"Considered for Future Use"** with a 41-0-1 vote (Gino Canessa / Michael Donnelly), with the rationale that several implementers had created working clients and servers without a formal sub-protocol definition, and that formalizing one lacked sufficient real-world feedback. However, the ticket was **re-opened post STU1 publication** by Lynn Laakso on 2022-07-18.

## Analysis

### Context

The FHIR R5 Subscriptions Backport IG defines how clients connect to servers via WebSocket to receive subscription notifications. The IG specifies message formats, binding token operations, and connection semantics — all elements that constitute a WebSocket sub-protocol per RFC 6455. However, the IG does not formally declare or register a sub-protocol identifier.

Several related tickets (FHIR-31438, FHIR-31471) have already been resolved as Persuasive, resulting in improved documentation of message formats and interaction descriptions. FHIR-40924 also pushed for better definition of the websocket extension. This suggests the community recognizes the need for clearer websocket specifications, but the question of formally defining a sub-protocol remains open.

### Key Considerations

1. **RFC 6455 Compliance**: The WebSocket protocol specification expects sub-protocols to be declared during the handshake. Formalizing a sub-protocol would improve interoperability and allow clients to negotiate capabilities.
2. **Implementation Experience**: The previous vote noted that implementers had working clients/servers without a formal sub-protocol, suggesting it is not a hard blocker. However, the lack of a formal sub-protocol may lead to implementation divergence as adoption grows.
3. **Evolution Since STU1**: The ticket was re-opened post STU1, indicating the workgroup wanted to reconsider. Related tickets have since been resolved, improving message format and interaction descriptions — providing a better foundation for a formal sub-protocol definition.
4. **Effort vs. Benefit**: Defining and registering a formal IANA WebSocket sub-protocol requires significant specification work and ongoing maintenance.

## Proposed Dispositions

### Disposition A: Persuasive with Modification

**Action**: Define a formal WebSocket sub-protocol identifier (e.g., `fhir.subscription.r5`) for the Subscriptions Backport IG. Document the sub-protocol's message format, handshake requirements, and binding semantics. Register the sub-protocol with IANA per RFC 6455 Section 11.5. Update the IG to instruct clients to pass the sub-protocol identifier in the WebSocket constructor.

**Rationale**: The IG already defines the elements of a sub-protocol. Formalizing it improves interoperability, enables proper client-server negotiation, and aligns with WebSocket standards. The groundwork from resolved tickets (FHIR-31438, FHIR-31471) makes this more achievable now than at STU1.

**Trade-offs**: Requires significant specification and IANA registration effort. May introduce backward compatibility concerns for existing implementations that don't negotiate a sub-protocol.

### Disposition B: Considered for Future Use

**Action**: Acknowledge the validity of the request but defer formal sub-protocol definition to a future version. Add a non-normative note to the IG explaining that a formal WebSocket sub-protocol is under consideration and may be defined in a future release. Encourage implementers to provide feedback on their sub-protocol needs.

**Rationale**: Existing implementations work without a formal sub-protocol. Deferring avoids premature standardization and allows more real-world feedback to accumulate, particularly as the Subscriptions framework matures. This was the original resolution rationale and the situation may not have changed sufficiently.

**Trade-offs**: Continued lack of a formal sub-protocol could lead to implementation divergence. May be perceived as ignoring a legitimate standards compliance concern.

### Disposition C: Persuasive

**Action**: Accept the submitter's feedback and add normative guidance acknowledging that the WebSocket communication pattern constitutes a sub-protocol. Require implementations to support sub-protocol negotiation during the WebSocket handshake using a designated identifier, but defer IANA registration until broader adoption. Update JavaScript and other client examples to demonstrate passing the sub-protocol identifier.

**Rationale**: A middle-ground approach that addresses the core concern (acknowledging and using a sub-protocol identifier) without requiring the full overhead of IANA registration. Provides immediate interoperability benefits while leaving formal registration for later.

**Trade-offs**: Using an unregistered sub-protocol identifier is technically non-compliant with the full IANA registration process in RFC 6455, though it is common practice in early-stage protocols.

## Recommendation

**Disposition C: Persuasive** is recommended.

The submitter correctly identifies that the IG is defining a sub-protocol. The most practical path forward is to acknowledge this, define a sub-protocol identifier, and require its use during WebSocket handshake negotiation — without immediately taking on the burden of IANA registration. This addresses the standards compliance concern, improves interoperability between client and server implementations, and builds on the improved message format/interaction descriptions from previously resolved related tickets (FHIR-31438, FHIR-31471). IANA registration can follow once the sub-protocol definition stabilizes through real-world use.
