## Summary

The submitter asks whether it is valid to have more than one of `queryCriteria`, `fhirPathCriteria`, and `event` populated on the same `SubscriptionTopic.trigger` element, and if so, what the semantic relationship is (AND, OR, or translations of each other). The submitter suggests that if only one should be used at a time, a constraint should be added to enforce mutual exclusivity.

## Details

### Description

The reporter notes that the SubscriptionTopic narrative and Topic-Based Subscription documentation clearly describe how to convey trigger details via `queryCriteria`, `fhirPathCriteria`, or `event` (along with `resource` and optionally `supportedInteraction`). However, the specification does not clarify what it means when more than one of these three criteria types is populated on the same trigger:

- Is it an **AND** relationship (all must be satisfied)?
- Is it an **OR** relationship (any can trigger)?
- Are they **translations** of each other (equivalent definitions in different formalisms)?

If only one should be used at a time, the reporter suggests adding a constraint to enforce this.

## Related Zulip Discussions

### 1. [implementers > subscribe to specific resource properties changes](https://chat.fhir.org/#narrow/stream/implementers/topic/subscribe%20to%20specific%20resource%20properties%20changes)
- **Key exchange (2024-05-27):** Jens Villadsen directly asked: *"Is the use of fhirPathCriteria thought as being mutually exclusive to using queryCriteria on SubscriptionTopic?"*
- **Gino Canessa's answer (2024-05-28):** *"No, the criteria is not exclusive. In fact, we are discussing collapsing the event structure into the same generic trigger definition for R6. Overall, there is no way of knowing how a system will actually want to implement a topic, so we want as many types of definitions as authors and communities can provide."*
- **Significance:** This is the authoritative answer from the subscriptions domain expert — queryCriteria and fhirPathCriteria are intentionally **not mutually exclusive** and are meant as **equivalent alternative representations** of the same trigger logic.

### 2. [subscriptions > resourceTrigger](https://chat.fhir.org/#narrow/stream/subscriptions/topic/resourceTrigger)
- **Key exchange (2023-08-08):** Stefano Contento asked about the difference between queryCriteria and fhirPathCriteria.
- **Gino Canessa's answer:** *"Yes, the two rules are equivalent. There are some things that are quite complicated to describe in one or the other, so it is allowed to just have one. However, it is expected that some implementations will prefer to execute based on one or the other depending on their architecture, so it is recommended to have both."*
- **Significance:** Confirms the design intent — both criteria types express the same trigger semantics and having both is **recommended** (not prohibited).

## Proposed Dispositions

### Disposition A: Persuasive — Add Clarifying Text

**Action:** Add clarifying text to the `SubscriptionTopic.trigger` documentation explaining the relationship between `queryCriteria`, `fhirPathCriteria`, and `event` when multiple are populated on the same trigger.

Specifically, add language such as:

> When more than one of `queryCriteria`, `fhirPathCriteria`, or `event` is populated on the same trigger, they represent equivalent alternative definitions of the trigger criteria expressed in different formalisms. Implementations may choose which representation to evaluate based on their architecture. They are not combined with AND or OR logic; rather, they describe the same trigger condition so that servers with different capabilities can each find a usable definition.

This addresses the submitter's core question without adding a structural constraint, and aligns with the established design intent documented in Zulip discussions by the subscriptions domain expert (Gino Canessa).

### Disposition B: Persuasive with Modification — Add Clarifying Text and a Short Description

**Action:** In addition to Disposition A's clarifying text, add a `short` description or note on each of the three sub-elements (`queryCriteria`, `fhirPathCriteria`, `event`) indicating they are alternative representations and that having multiple populated is valid and recommended. This reinforces the intent at the element definition level rather than only in narrative.

### Disposition C: Not Persuasive — No Change Required

**Action:** Leave the specification as-is.

**Rationale:** The answer to the submitter's question is already available in community discussion (Zulip) and is consistent with the general FHIR design principle that specifications should not over-constrain implementations. Adding a mutual exclusivity constraint would be incorrect (contradicting the design intent), and the existing documentation is sufficiently clear when read alongside the definitions of each sub-element. The question type suggests this is informational rather than requiring a spec change.

## Recommendation

**Disposition A: Persuasive — Add Clarifying Text**

The submitter raises a legitimate question that has been asked multiple times in different contexts (Zulip threads from 2023 and 2024, plus this Jira ticket in 2025). The fact that implementers repeatedly ask the same question indicates the specification text is insufficiently clear on the relationship between the three criteria types. Gino Canessa has provided a clear and consistent answer in Zulip — that the criteria are equivalent alternative representations, not mutually exclusive, and having multiple is recommended — but this intent is not reflected in the specification text. A brief clarifying statement in the narrative would prevent future confusion at minimal cost, without introducing any breaking change.
