## Summary

The reporter (Elliot Silver) identifies a bug in the FHIR Shorthand (FSH) language reference documentation's example of soft indexing. The current example for adding a second given (middle) name reads:

```fsh
* name[+].given[+] = "Robert"
* name[+].given[+] = "David"
* name[=].family = "Smith"
```

The second line uses `name[+]` which increments the name array index, creating an unintended *new* HumanName entry. The reporter asserts it should instead use `name[=]` (reuse the current index) so that "David" is added as a second given name on the *same* HumanName element as "Robert":

```fsh
* name[+].given[+] = "Robert"
* name[=].given[+] = "David"    // corrected: [=] instead of [+] for name
* name[=].family = "Smith"
```

This is a documentation correction to an example in the normative FSH specification.

## Proposed Dispositions

### Disposition A: Persuasive

**Action:** Accept the correction as-is. Change the second line of the soft indexing example from `name[+].given[+] = "David"` to `name[=].given[+] = "David"`.

**Rationale:** The reporter's analysis is correct. In FSH soft indexing, `[+]` increments the array index (creating a new element), while `[=]` reuses the current index. The intent of the example is to add a second given name ("David") to the *same* HumanName that already has "Robert" as the first given name, and then assign "Smith" as the family name. Using `[+]` on the `name` path in the second line incorrectly creates a second HumanName entry, which contradicts the stated purpose of the example. The fix is a single-character change (`+` → `=`) that makes the example behave as documented.

**Scope:** One-line change in the FSH language reference, section on soft indexing examples.

---

### Disposition B: Persuasive with Modification

**Action:** Accept the correction and additionally expand the example with inline comments clarifying what each soft index operator does at each step.

**Rationale:** The example is used to teach a subtle feature. In addition to fixing the bug, adding inline comments showing the resolved paths (e.g., `// → name[0].given[1]`) would make the example significantly clearer for readers. The Zulip discussion ("Soft Indexing downside") shows that even experienced users find soft indexing confusing to debug. An annotated example would help prevent similar misunderstandings.

**Scope:** One-line fix plus a small enhancement to the surrounding documentation in the soft indexing section.

---

### Disposition C: Not Persuasive

**Action:** Reject the change request; the example is correct as written.

**Rationale:** This would only apply if the intent of the example was to create *two* separate HumanName entries — one with given "Robert" and another with given "David" and family "Smith". However, the surrounding prose states the purpose is to "add a second given (middle) name to the example above," which clearly indicates a single HumanName. This disposition is not supportable given the stated intent.

---

## Recommendation

**→ Disposition A: Persuasive**

The reporter's correction is straightforward and clearly correct. The current example contains a single-character error (`[+]` instead of `[=]`) that causes the example to not match its stated purpose. The fix is minimal, unambiguous, and non-controversial. While Disposition B's suggestion to add comments has merit, it expands the scope beyond the reported issue and can be tracked separately if desired.
