## Summary

The submitter reports that the description text for the Bundle constraint **bdl-17** is unclear about what it actually constrains. The current description reads: *"Use and meaning of issues for documents has not been validated because the content will not be rendered in the document."* This text does not clearly state what behavior is expected or prohibited. The submitter suggests that if this is informational guidance, it should be moved to the [2.4.4.3 Notes about Bundle](https://build.fhir.org/bundle.html#notes) section. If it is intended to be a rule, the wording should be rewritten to clearly state the expected behavior.

## Details

### Description

The `Bundle.issues` element allows an OperationOutcome to be included directly in a Bundle (not as an entry). Constraint bdl-17 applies to document-type Bundles, but its description text is ambiguous — it reads more like an explanatory note about a limitation ("has not been validated") rather than a testable constraint.

The submitter asks:
1. Should this statement be reclassified as informational guidance and relocated to the Bundle notes section?
2. If it is intended as a rule, what is the expected behavior — should `Bundle.issues` be prohibited in document Bundles, or are there specific conditions under which it is valid?

## Proposed Dispositions

### Disposition A: Accept — Reword bdl-17 as informational guidance

**Action:** Remove the ambiguous description from the bdl-17 constraint and relocate it as a note in the [2.4.4.3 Notes about Bundle](https://build.fhir.org/bundle.html#notes) section. The note would explain that `Bundle.issues` in document Bundles has limited utility because issues content is not rendered as part of the document narrative, and implementers should be aware that the semantics of issues in document Bundles are not well-defined.

**Rationale:** The current text reads as an explanation of a limitation rather than a testable constraint. Making it a note removes the ambiguity about whether it is a rule. This aligns with the submitter's first suggestion and the fact that the Zulip discussion revealed this element was poorly exercised and not fully validated.

### Disposition B: Accept — Rewrite bdl-17 description as a clear constraint

**Action:** Rewrite the bdl-17 description to clearly state the intended rule. For example, if the intent is that `Bundle.issues` SHOULD NOT be populated in document Bundles, the description should say: *"Document Bundles SHOULD NOT include issues, because the content of issues is not rendered as part of the document."* Alternatively, if issues should be allowed with conditions, state those conditions explicitly.

**Rationale:** If there is a valid reason to constrain (or warn about) the use of `Bundle.issues` in document Bundles, the constraint should be expressed in clear, testable language using RFC 2119 keywords. This preserves the constraint while addressing the clarity concern.

### Disposition C: Decline — No change needed

**Action:** Leave bdl-17 description as-is. The text is sufficient to inform implementers about the limitation, even if the wording is unconventional.

**Rationale:** The current text, while awkwardly worded, does communicate that issues in document Bundles have not been validated and may not be rendered. Implementers can interpret this as a warning. However, this leaves the ambiguity unresolved and does not address the submitter's valid concern that the wording is unclear.

## Recommendation

**Disposition B: Accept — Rewrite bdl-17 description as a clear constraint.**

The evidence strongly supports clarifying the constraint text. The Zulip thread on "Testing Invariants" demonstrates that bdl-17 was poorly understood even by core committers — Brian Postlethwaite's testing revealed that no spec examples exercised `Bundle.issues`, and the Java parser had never been tested against it. The current description text fails the basic test of a constraint: it does not clearly state what is required or prohibited. Rewriting it with clear RFC 2119 language (e.g., SHOULD NOT or SHALL NOT) would make the constraint testable and unambiguous, while preserving the intent. If the workgroup determines that the text is purely informational, Disposition A is also acceptable, but given that bdl-17 exists as a named constraint with a severity level, it should either be expressed as a proper rule or removed as a constraint entirely. A half-measure of leaving ambiguous text (Disposition C) perpetuates the confusion that led to this ticket.
