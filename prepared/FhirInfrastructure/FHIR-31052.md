## Description Summary

The reporter proposes two major changes to the SMART Web Messaging specification:

1. **Remove the "scratchpad" concept entirely** — The scratchpad assumes an EHR-specific implementation detail (an in-memory holding area for draft resources). Whether an EHR uses a scratchpad, shopping cart, or persists draft resources directly to a FHIR server should not be dictated by the specification. The reporter argues the specification should remain silent on this, much as the base FHIR spec does.

2. **Consolidate message groups into three roles** — The current spec defines multiple message groups (`scratchpad.*`, `ui.*`, `fhir.*`, `status.*`) that the reporter sees as arbitrary and unnecessarily complex. The proposed replacement is:
   - **`ui.*` (or `host.*`)** — Messages from the App invoking the EHR host (e.g., `ui.create`, `ui.delete`, `ui.update`, `ui.handshake`)
   - **`app.*` (or `guest.*`)** — Messages from the EHR host invoking the App
   - **`fhir.*`** — Special case where the EHR host acts as a FHIR proxy on behalf of the App

The reporter acknowledges this is a significant breaking change and suggests it may not be part of an immediate ballot.

## Analysis

### Core Question
Should the SMART Web Messaging specification remove the "scratchpad" concept and consolidate its message groups into three role-based categories (`ui.*`, `app.*`, `fhir.*`)?

### Arguments For
- The scratchpad concept is poorly defined (as acknowledged in FHIR-29349, which was resolved as Persuasive with Modification).
- It bakes an EHR implementation detail into the specification — some EHRs may not have a scratchpad.
- Consolidating message groups around roles (host, app, FHIR proxy) is cleaner and more extensible.
- A PR was created with concrete proposed changes demonstrating feasibility.

### Arguments Against
- The prior related ticket FHIR-29368 (same reporter) was resolved as **Not Persuasive**, indicating the work group has already considered and rejected the core argument about removing the scratchpad.
- The scratchpad serves a real purpose: session-scoped ephemeral data that may not exist on the FHIR server (Josh Mandel, Lloyd McKenzie, Dennis Patterson, Carl Anderson, Chuck Feltner all defended this).
- This aligns with CDS Hooks' existing interaction model for draft orders.
- The PR was **closed without being merged**, suggesting the community/editors did not adopt the proposed changes.
- The renaming portion may have been addressed through FHIR-29349's resolution (Persuasive with Modification), which was published.

### Key Observations
- This ticket is now over 4 years old and remains Triaged/Unresolved.
- The associated PR was closed without merge in August 2021.
- The closely related FHIR-29349 (scratchpad undefined) was resolved with modifications and published, suggesting the definitional concerns were addressed through that path.
- The community discussion showed some convergence around renaming but not around full removal.

## Proposed Dispositions

### Disposition A: Not Persuasive
**Rationale:** The work group previously considered the core argument about removing the scratchpad in FHIR-29368 and resolved it as Not Persuasive. The scratchpad concept serves a distinct purpose — providing a session-scoped abstraction for ephemeral resources that may not yet exist on the FHIR server. Key community members (Josh Mandel, Lloyd McKenzie, Dennis Patterson, Carl Anderson) defended this concept in both Zulip threads. The associated PR (#41) was closed without merge. The definitional concerns about the scratchpad were addressed through FHIR-29349 (resolved Persuasive with Modification, now Published). The message group consolidation proposal is a breaking change without demonstrated consensus.

### Disposition B: Persuasive with Modification
**Rationale:** While the full removal of the scratchpad is not supported by community consensus, the Zulip discussions revealed agreement that the naming could be improved. Accept the principle that message group naming should better reflect the roles involved (host/app/proxy) but retain the scratchpad as a concept. The modification would be to review the message group taxonomy for clarity and alignment with role-based naming in a future ballot cycle, building on what was already done for FHIR-29349.

### Disposition C: Considered — Resolved by FHIR-29349
**Rationale:** The primary concerns raised in this ticket — that the scratchpad is undefined and that message groups are confusing — have been substantially addressed through the resolution of FHIR-29349 (Persuasive with Modification, Published). The remaining request for full scratchpad removal was previously rejected in FHIR-29368. This ticket can be considered resolved by the combination of those two prior resolutions.

## Recommendation

**Disposition A: Not Persuasive** — The fundamental proposal to remove the scratchpad and restructure message groups has been thoroughly debated. The work group rejected the core argument in FHIR-29368, and the associated PR was closed without merge. The definitional gap was addressed via FHIR-29349. The scratchpad abstraction serves a recognized need for session-scoped ephemeral data interaction, as supported by multiple community members. The breaking change proposed here lacks the consensus needed to proceed.
