## 1. Summary

This ticket requests adding a new event type to the SMART Web Messaging specification that would allow a SMART client (embedded app) to request the EHR host to launch another application. The ticket description references a Google Doc discussion on a "handshake protocol" ([link](https://docs.google.com/document/d/1dsVVnSAZpYCplpNYtWp6OO1rQNQ9zeglA086pZfIw2M)) that explored this concept. Essentially, the submitter envisions a messaging event where an embedded SMART app can tell the EHR: "please launch app X" — extending the existing set of SMART Web Messaging event types (which currently cover scratchpad operations, UI signaling, and status handshakes) to include cross-app launch coordination.

## 6. Proposed Dispositions

### Disposition A: Persuasive — Add an app-launch-request event
**Action:** Add a new message type (e.g., `ui.launchApp` or `app.launch`) to SMART Web Messaging that allows a SMART client to request the EHR to launch a specified application. Define required payload fields (target app identifier, launch context, optional parameters) and expected EHR response (success/failure with launched app handle or error).  
**Rationale:** This fills a genuine gap in the messaging spec. Current events handle scratchpad CRUD, UI lifecycle (`ui.done`), and handshakes, but there is no mechanism for cross-app launch coordination. Clinical workflows frequently require chaining between apps (e.g., a CDS app requesting launch of a specialized ordering app). The handshake infrastructure is mature enough to support this extension.  
**Risk:** Scope creep; the spec is still maturing and adding new event categories before stabilizing existing ones (see FHIR-31052 on message group restructuring) could complicate adoption.

### Disposition B: Persuasive with Modification — Defer until message group restructuring is resolved
**Action:** Accept the concept as a valid future addition but defer formal specification until the message group taxonomy is finalized (pending resolution of FHIR-31052). Document the use case in a "future considerations" section of the IG.  
**Rationale:** The message group restructuring proposed in FHIR-31052 (consolidating to `ui.*`/`app.*`/`fhir.*` groups) would directly affect how this new event is named and categorized. Defining `ui.launchApp` now under the current taxonomy risks a breaking change later. Documenting the use case ensures it isn't lost while the architecture stabilizes.  
**Risk:** The use case remains unaddressed for an indefinite period; FHIR-31052 has been Triaged since 2021 with no resolution.

### Disposition C: Not Persuasive — Out of scope for SMART Web Messaging
**Action:** Close as not persuasive. Cross-app launch coordination is better handled through existing mechanisms such as CDS Hooks (which already supports app launch via `Link` cards) or FHIRCast (which handles context synchronization across apps).  
**Rationale:** SMART Web Messaging focuses on communication between an embedded app and its EHR host within an iFrame. Requesting the EHR to launch a *different* app crosses the boundary from in-context messaging to application orchestration, which is the domain of CDS Hooks and FHIRCast. Adding this would blur the separation of concerns between these specifications.  
**Risk:** Real-world implementers may need this capability within the iFrame context specifically, where CDS Hooks and FHIRCast do not apply (as shown by the questionnaire/forms integration discussions).

---

## 7. Recommendation

**Disposition B: Persuasive with Modification** — Defer until message group restructuring is resolved.

This is the most pragmatic path. The use case is valid — clinical workflows where an embedded SMART app needs to request the EHR to launch another application are real and growing (as evidenced by the 2025 questionnaire/forms discussion where implementers are already creating custom message types). However, the foundational question of how SMART Web Messaging event types are organized (FHIR-31052) remains unresolved since 2021. Defining a new `ui.launchApp` or `app.launch` event now, only to rename or restructure it later, would create unnecessary churn for implementers. The recommended approach is to formally acknowledge the use case, document it in the IG as a planned future addition, and specify it once the message taxonomy is stable.
