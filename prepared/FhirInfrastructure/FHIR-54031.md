## Summary

The reporter asks for clarification on how the new `limited-filter` OperationOutcome.issue code relates to and differs from the existing Self Link mechanism described in the [search page](https://build.fhir.org/search.html#selflink). Both appear to provide mechanisms for a server to communicate to the client that the search actually performed was different from what was requested, but their relationship and intended usage is not documented. The ticket requests guidance on which mechanism should be preferred and how they differ.

## Details

### Description

The new `limited-filter` OperationOutcome.issue code was introduced to allow servers to indicate that search filters were limited or modified. However, the FHIR specification already defines a Self Link mechanism (section 3.2.1.9) requiring servers to return the parameters actually used to process a search via the Bundle's `self` link. Both mechanisms address the same fundamental question: "Was the search performed different from what the client requested?"

The reporter observes that without clarification, implementers face ambiguity:
- The **Self Link** approach modifies the URL in `Bundle.link` where `relation = "self"` to reflect only the parameters that were actually applied.
- The **`limited-filter` OperationOutcome** approach adds an OperationOutcome entry to the search bundle indicating that filtering was limited.

The ticket asks: Which should be preferred, and/or how are they different?

## Proposed Dispositions

### Disposition A: Persuasive — Add clarification distinguishing self link and limited-filter mechanisms

Add a paragraph or note to the search page (near the `limited-filter` code definition and/or the Self Link section) that explicitly explains the relationship between the two mechanisms:

- **Self Link** reflects the search parameters the server *actually used* to process the search. It is the canonical, machine-readable record of "what search was performed." If a server drops or modifies parameters, the self link should reflect the parameters as actually applied.
- **`limited-filter` OperationOutcome** is a *human/client notification* mechanism that alerts the client to the fact that some requested filters were limited, dropped, or not applied as expected. It provides an explanation of *why* the search was different, not just *what* was different.
- Both mechanisms are complementary and SHOULD be used together: the self link records what was done, and the OperationOutcome explains what changed and why.

This directly addresses the reporter's question and aligns with the existing resolution of FHIR-37993 and FHIR-51506.

### Disposition B: Persuasive with Modification — Clarify and add normative guidance for when each mechanism is required

In addition to clarifying the distinction (as in Disposition A), add normative or best-practice language:

- Servers **SHALL** use the self link to reflect the parameters actually used in processing the search.
- Servers **SHOULD** include a `limited-filter` OperationOutcome.issue when any client-requested search parameter was dropped, ignored, or applied only partially (e.g., to some categories but not others).
- Add an example showing both mechanisms used together in a search response bundle.

This provides stronger implementation guidance and reduces ambiguity for implementers building both servers and clients.

### Disposition C: Not Persuasive — Existing spec text is sufficient

The existing specification already defines both mechanisms independently. The self link is defined in the Server Conformance section (3.2.1.9), and the `limited-filter` code is defined in the OperationOutcome issue type value set. Implementers can infer their complementary nature from context, and no additional clarification is needed.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The reporter raises a valid interoperability concern — two mechanisms addressing overlapping scenarios without documented guidance creates ambiguity. The resolution of FHIR-37993 (which proposed using OperationOutcome warnings for partial parameter application) and FHIR-51506 (which clarified self link behavior for server-added parameters) both support the idea that these mechanisms are complementary. Adding explicit guidance with normative language and an example will reduce implementation divergence and improve client/server interoperability. This is particularly important as the `limited-filter` code is new and implementers need clear direction on how it fits with the established self link pattern.
