## Summary

The SMART on FHIR specification's Client Authentication section currently references [RFC 7519](https://www.rfc-editor.org/rfc/rfc7519) (JSON Web Token). However, RFC 7519 has since been updated by [RFC 8725 / BCP 225](https://www.rfc-editor.org/rfc/rfc8725) (JSON Web Token Best Current Practices), which provides important security guidance for JWT usage. The submitter requests that FHIR adopt RFC 8725 as a directly referenced and required standard for both front-end and back-end SMART launches, ensuring SMART on FHIR client authentication aligns with current JWT security best practices.

---

## Details

### Description

The SMART on FHIR Client Authentication currently references RFC 7519 for JWT handling. RFC 8725 (BCP 225) was published as an update to RFC 7519, providing security best practices for JWT implementations including:

- Explicit algorithm validation to prevent algorithm confusion attacks
- Restrictions on the use of the `"none"` algorithm
- Guidance on token lifetime and expiration validation
- Recommendations against embedding sensitive data in unencrypted JWTs
- Best practices for audience (`aud`) claim validation

The submitter argues that the SMART on FHIR specification should normatively reference RFC 8725 to ensure implementations follow current security best practices when handling JWTs in both EHR launch (front-end) and standalone launch (back-end) flows.

## Proposed Dispositions

### Disposition A: Accept — Add RFC 8725 as a Normative Reference

**Action:** Update the SMART on FHIR Client Authentication section to normatively reference RFC 8725 (BCP 225) alongside the existing RFC 7519 reference. Add a conformance requirement such as: *"Implementations SHALL conform to the security best practices defined in [RFC 8725](https://www.rfc-editor.org/rfc/rfc8725) when creating and validating JSON Web Tokens for SMART on FHIR client authentication in both EHR launch and standalone launch flows."*

**Rationale:** RFC 8725 is the IETF's own best-current-practice update to RFC 7519. Its guidance (algorithm validation, `"none"` algorithm restrictions, audience claim checks) directly addresses known JWT attack vectors relevant to healthcare data exchange. The precedent set by FHIR-31966 (updating RFC 2119 → RFC 8174 reference in Bulk Data, resolved Persuasive) supports this approach. Making this normative ensures consistent security posture across SMART implementations.

### Disposition B: Alternative — Reference RFC 8725 as Informative Guidance

**Action:** Add an informative (non-normative) reference to RFC 8725 in the SMART on FHIR Client Authentication section with a SHOULD-level recommendation. For example: *"Implementations SHOULD follow the JWT best practices described in [RFC 8725](https://www.rfc-editor.org/rfc/rfc8725) for enhanced security."* Keep the existing normative reference to RFC 7519 unchanged.

**Rationale:** This provides visibility of the updated practices without imposing new normative requirements on existing implementations. It allows implementers to adopt RFC 8725 guidance at their own pace while signaling the community's direction. This is a lower-impact change that avoids potential backward compatibility concerns for implementations that are technically compliant with RFC 7519 but have not yet adopted all RFC 8725 recommendations.

### Disposition C: Decline — No Change Needed

**Action:** No changes to the SMART on FHIR specification. Close the ticket as Not Persuasive.

**Rationale:** RFC 7519 remains a valid and current standard; RFC 8725 updates it but does not obsolete it. Implementations are free to follow RFC 8725 best practices without a spec mandate. Individual implementation guides (such as UDAP Security) can add RFC 8725 requirements as needed for their specific security contexts. Additionally, the SMART App Launch specification is maintained by the SMART Health IT project and may prefer to coordinate such changes through their own process rather than through a FHIR Infrastructure ballot item.

---

## Recommendation

**Disposition A: Accept** — Add RFC 8725 as a normative reference in SMART on FHIR Client Authentication.

RFC 8725 addresses well-documented JWT security vulnerabilities that are directly applicable to SMART on FHIR authentication flows. The IETF published BCP 225 specifically because RFC 7519 alone is insufficient to prevent common attack vectors. The precedent from FHIR-31966 (adding updated RFC 8174 reference alongside RFC 2119, resolved Persuasive) demonstrates that HL7 supports updating RFC references when superseding best-practice documents are available. Healthcare data exchange demands the highest security standards, and normatively requiring JWT best practices is the appropriate level of conformance for this context.
