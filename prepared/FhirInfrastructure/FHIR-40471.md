## Description

When profiles are presented in the FHIR specification, the default view is the **Differential table**, which does not show modifier elements. The submitter argues that modifier elements are critical because they can change the behavior of the resource. To prevent implementers from overlooking them, the default representation of profiles in Implementation Guides and the core spec should be changed to the **Key Elements table**.

## Related Zulip Discussions

### [IG creation > Make KeyElements the default tab](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/Make%20KeyElements%20the%20default%20tab)
**Dates:** 2024-03-05 to 2024-10-18 | **Messages:** 55 | **Participants:** Jean Duteau, Lloyd McKenzie, Jose Costa Teixeira, Eric Haas, Grahame Grieve, Richard Townley-O'Neill, Gino Canessa, Ádám Z. Kövér

**Key Discussion Points:**

- **Jean Duteau** initiated the discussion, noting that profiles inheriting from other profiles cause confusion when the Differential is the default tab because readers miss inherited constraints.
- **Lloyd McKenzie** framed the core tension as "most useful for balloters" (differential) vs. "most useful for implementers" (key elements). He advocated for consistent behavior across all HL7 IGs and noted that FHIR aims for ease for implementers.
- **Eric Haas** had a prior parameter (`default_profile_view`) in old templates that could control the default tab. He pushed back on Key Elements as default, arguing it "omits and contradicts the diff" and that the concept of "key" is subjective and not controllable by the IG editor.
- **Lloyd McKenzie** responded that Key Elements should never contradict or omit anything from the diff — if it does, that's a bug. Key Elements lists all must-supports, mandatories, constrained elements, etc.
- **Richard Townley-O'Neill** agreed Key Elements is more useful for implementers but noted it needs work. He identified a specific bug: AU Base Patient constraints on `Patient.contact` not appearing in the Key Elements table (filed as [#888](https://github.com/HL7/fhir-ig-publisher/issues/888)).
- **Gino Canessa** preferred Key Elements over Differential (and actually prefers Snapshot). He also identified missing flags in the Differential view (filed as [#887](https://github.com/HL7/fhir-ig-publisher/issues/887)).
- **Jose Costa Teixeira** suggested this could be a template-level setting but acknowledged consensus may be hard.
- **Ádám Z. Kövér** (2024-10-18) proposed a new "summary diff" view showing all customizations from the base spec — distinct from this request but related to the broader discussion of profile views.
- **Consensus (per lmckenzi on GitHub #879, 2024-04-30):** "General agreement that we should move to key elements as the default, but we need to fix the previously reported issues with 'key elements' first."

## Analysis

### Current State
The underlying tooling change has been **implemented and completed**. GitHub issue HL7/fhir-ig-publisher#879 was closed as "Done" on 2025-07-22 by lmckenzi. The IG publisher now defaults to the Key Elements view. However, the Jira ticket FHIR-40471 remains in "Triaged" status with no resolution set, even though the request has been fulfilled by the tooling change.

Two related enhancement issues remain open:
- **#887** — Differential tab should show unchanged flags (Approved, not yet implemented)
- **#888** — Key Elements tab missing some elements from the differential (Approved, not yet implemented)

### Impact
This change affects all FHIR IGs and the core specification. The Key Elements view as default improves visibility of modifier elements, must-support flags, and inherited constraints for implementers — the primary audience for FHIR specifications.

## Proposed Dispositions

### Disposition A: Persuasive — Close as Applied
**Rationale:** The requested change — making the Key Elements view the default profile representation — has been implemented in the IG publisher (GitHub issue #879 closed as "Done" on 2025-07-22). The community reached general agreement on this change through Zulip discussion and tooling group review. The Jira ticket should be closed as applied/persuasive since the tooling now reflects the requested behavior.

**Resolution Text:** *The Key Elements view has been made the default tab for profile representation in the IG publisher (see HL7/fhir-ig-publisher#879, completed 2025-07-22). This was implemented after community discussion and agreement that Key Elements is more useful for the target implementer audience.*

### Disposition B: Persuasive with Modification — Close with Note on Remaining Work
**Rationale:** The primary request has been implemented, but note that related issues (#887, #888) improving both the Differential and Key Elements views remain open. Close the ticket as persuasive while acknowledging the broader improvements are still in progress.

**Resolution Text:** *The Key Elements view has been made the default tab (HL7/fhir-ig-publisher#879, completed 2025-07-22). Related enhancements to the Key Elements table (#888) and Differential table (#887) are tracked separately and remain in progress.*

### Disposition C: Not Persuasive — Revert to Differential Default
**Rationale:** Some community members (notably Eric Haas) argued that the Key Elements view is subjective, not fully under editor control, and may omit or misrepresent constraints. The Differential gives editors direct control over what readers see. Reverting to Differential as default would restore editor agency.

**Resolution Text:** *After review, the workgroup prefers the Differential view as the default because it provides IG editors with direct control over the displayed constraints, while the Key Elements view remains available as an alternative tab.*

## Recommendation

**Disposition A: Persuasive — Close as Applied** is recommended. The change has already been implemented and deployed in the IG publisher. Community consensus was reached through the Zulip discussion and tooling group review. The remaining issues (#887, #888) are separate enhancements tracked in their own GitHub issues and do not block closing this ticket. The Jira ticket should be updated to reflect the completed work.
