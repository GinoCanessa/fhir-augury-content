## Summary

The Resource Type page in the FHIR Core specification contains a link to the FHIR Testing incubator IG (where TestScript and TestPlan resources were moved), but the URL `https://build.fhir.org/ig/hl7/fhir-testing-ig/index.html` returns an HTTP 404 error. This is a straightforward broken-link issue. The correct URL appears to be `https://build.fhir.org/ig/HL7/fhir-testing-ig/` (note the uppercase `HL7` in the path, and the `/en/` landing page for the published IG).

## Details

### Description
The Resource Type Page in the FHIR Core specification contains a link to the new FHIR Testing incubator where the TestScript and TestPlan resources were moved as part of the "Additional Resources" initiative (moving certain resources out of FHIR Core R6 into dedicated IGs). The link points to `https://build.fhir.org/ig/hl7/fhir-testing-ig/index.html`, which results in an HTTP 404 error.

### Comments
No comments on this ticket.

### Analysis
The broken link is likely caused by one or both of the following issues:
1. **Case sensitivity**: The link uses lowercase `hl7` in the path, but the GitHub organization (and FHIR CI build URLs) use uppercase `HL7`. The correct base path is `https://build.fhir.org/ig/HL7/fhir-testing-ig/`.
2. **Path structure**: The published IG may use a language-prefixed path (e.g., `/en/index.html` rather than `/index.html` at the root). A Zulip participant (Jorn Duwel) successfully referenced `https://build.fhir.org/ig/HL7/fhir-testing-ig/en/` in February 2026.

A duplicate ticket [FHIR-54820](https://jira.hl7.org/browse/FHIR-54820) (reported by Chris Melo, Priority: Highest) reports the same broken link with the same URL.

## Proposed Dispositions

### Disposition A: Accept as Technical Correction — Fix the URL
**Resolution:** Applied  
**Action:** Update the broken link on the Resource Type page from `https://build.fhir.org/ig/hl7/fhir-testing-ig/index.html` to the correct URL `https://build.fhir.org/ig/HL7/fhir-testing-ig/` (with uppercase `HL7`). Also audit other pages in the FHIR Core specification (notably `source/ballot-intro.html`) for the same incorrect link and correct those as well. Mark FHIR-54820 as a duplicate of this ticket.

**Rationale:** This is a clear technical correction — the URL path has incorrect casing and/or structure. Fixing it resolves the 404 and restores navigation to the Testing IG. Correcting all instances at once prevents future duplicate reports.

### Disposition B: Accept with Broader Link Audit
**Resolution:** Applied  
**Action:** Fix the broken link as described in Disposition A, and additionally perform a comprehensive audit of all links in the FHIR Core specification that reference any "additional resource" IGs (not just fhir-testing-ig). Establish a link-checking step in the CI build process for the FHIR Core specification to catch future broken links to external IGs automatically.

**Rationale:** Since the "Additional Resources" initiative is moving multiple resources to external IGs, there may be other broken or soon-to-be-broken links. A systematic audit and automated validation would prevent recurring issues of this type.

### Disposition C: Decline — Defer Until Testing IG is Stable
**Resolution:** Not Persuasive with Modification  
**Action:** No change to the current specification. Wait until the fhir-testing-ig has a stable, officially published URL before updating links in the core specification.

**Rationale:** The Testing IG is still in development (incubator status), and its URL structure may change again. However, this leaves a broken link in a published specification, which harms usability and credibility.

## Recommendation

**Disposition A: Accept as Technical Correction — Fix the URL**

This is a clear, low-risk technical correction. The broken link is straightforward to fix (case sensitivity in the URL path), and a duplicate ticket (FHIR-54820, marked Highest priority) confirms the impact. The Testing IG CI build appears to be active based on recent Zulip references, so the corrected URL should resolve. Deferring the fix (Disposition C) leaves a published specification with a broken link unnecessarily, while the broader audit (Disposition B) adds scope beyond what this ticket requires and can be tracked separately.
