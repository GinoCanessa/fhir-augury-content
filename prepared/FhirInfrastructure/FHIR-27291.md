## Summary

This change request proposes adding a shorthand list syntax to FHIR Shorthand (FSH) so that array-valued fields can be assigned multiple values on a single line using comma-separated notation instead of requiring individual indexed assignments.

Currently, assigning multiple values to an array field in FSH requires verbose indexed syntax:

```fsh
* modifier[0] = #missing
* modifier[1] = #text
* modifier[2] = #not
...
```

The submitter proposes allowing a compact comma-separated form:

```fsh
* modifier = #missing, #text, #not, #in, #not-in, #below, #above, #ofType
```

The ticket references the FSH ANTLR grammar file (`FSH.g4` in `FHIR/sushi`) as the location where such a change would be implemented.

## Proposed Dispositions

### Disposition A: Persuasive (Accept)

**Action:** Accept the change request and add comma-separated list assignment syntax to the FSH grammar.

**Rationale:** The request addresses a genuine verbosity problem in FSH. The 2024 Zulip discussion shows continued community interest four years after the original submission. The original ballot vote (13-0-0) showed unanimous support for the concept, deferring only on timing. Arrays of codes, CodeableConcepts, and Codings are common enough in FHIR profiles that a compact list syntax would materially reduce boilerplate. The grammar change would be additive and backward-compatible—existing indexed syntax would remain valid.

**Implementation:** Update the FSH ANTLR grammar (`FSH.g4`) in the FHIR/sushi repository to allow comma-separated value lists on the right-hand side of assignment rules for array-typed elements. Update the FSH specification text to document the new syntax. SUSHI tooling would need corresponding parser and code-generation updates.

### Disposition B: Persuasive with Modification (Scoped Approach)

**Action:** Accept but scope the feature to simple/primitive value types and FSH coding assignments only, deferring complex type list syntax to a future cycle.

**Rationale:** Elliot Silver's concern in the Zulip discussion is valid—many array fields hold complex types that cannot be meaningfully expressed in a single comma-separated line. Scoping the initial implementation to codes, booleans, strings, integers, and FSH coding literals (which cover CodeableConcept and Coding arrays) provides the highest-value use cases while avoiding ambiguity in the grammar for complex inline assignments. This reduces implementation risk and allows the community to evaluate the feature before extending it further.

**Implementation:** Same grammar and tooling changes as Disposition A, but with the parser restricted to simple value types and FSH coding expressions. The specification should note the scope limitation and indicate that complex type list syntax may be considered in a future ballot cycle.

### Disposition C: Not Persuasive (Decline)

**Action:** Decline the change request.

**Rationale:** The existing indexed and `[+]` soft-indexing syntax already provides a clear, unambiguous way to populate arrays. Adding comma-separated syntax introduces potential parsing ambiguity (e.g., commas within string values, distinguishing list separators from other comma uses). The 2024 Zulip discussion raised questions about how frequently this would be used in practice, given that most array fields hold complex types. The FSH language is already expressive, and the additional syntactic sugar may not justify the grammar complexity and tooling maintenance burden.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The request addresses a real ergonomic pain point that has persisted for four years with continued community interest. The scoped approach (simple types and FSH codings only) balances usability improvement with implementation pragmatism, addresses the concern about complex type applicability, and provides a safe path to evaluate the feature before broader extension. The unanimous original ballot support and 2024 community re-engagement indicate this is a worthwhile enhancement.
