## Summary

The reporter requests that the **General Service API** section of the FHIRPath page (`fhirpath.html#srvr-api`) be removed from the FHIR core specification and moved into a separate Implementation Guide (IG) to mature before being considered for normative status. The reporter considers the General Service API "powerful, useful and under used," and argues it is not ready for normative ballot.

This ticket is part of a broader set of ballot comments (many from Brian Postlethwaite and Bas van den Heuvel) questioning the normative readiness of several FHIRPath extension sections that were maturity level 0 (draft) in R5, including the `%server` factory, `%factory` type factory, and Terminology Service API.

## Details

## Proposed Dispositions

### Disposition A: Persuasive (Accept — Move to Incubator IG)

Accept the request and move the General Service API section from `fhirpath.html` in FHIR core to a separate incubator IG (e.g., a "FHIRPath Extensions" or "FHIRPath Service API" IG). This is consistent with the resolution of the closely related FHIR-53954 (already marked Persuasive) and FHIR-54745 (resolved Persuasive). The content was maturity level 0 in R5 and has minimal known implementation experience. Moving it to an incubator allows the API to mature with community feedback while keeping the normative FHIRPath page focused on well-established functionality.

**Applied Change:** Remove the General Service API section from `source/fhirpath.html` and establish it in an incubator IG. Add a brief note in the FHIRPath page referencing the incubator IG for implementers interested in the General Service API.

### Disposition B: Persuasive with Modification (Retain as Informative/Trial-Use)

Rather than fully removing the General Service API from core, downgrade the section's maturity to Trial-Use (STU) or Informative within the FHIRPath page, clearly marking it as not yet normative. This keeps the content discoverable within the core spec while acknowledging it is not ready for normative status. This avoids fragmenting FHIRPath documentation across multiple publications.

**Applied Change:** Add explicit maturity labeling (e.g., "Trial-Use" or "Maturity Level 0") to the General Service API section in `fhirpath.html`. Add warnings about limited implementation experience and outstanding issues (security, error handling, authorization) per feedback in FHIR-54074 and FHIR-53953.

### Disposition C: Not Persuasive (Decline — Retain in Core for Normative Ballot)

Decline the request on the basis that the General Service API provides important functionality that should be balloted as part of the normative FHIRPath package. Separating it into a different IG could reduce visibility and slow adoption. The community should address maturity concerns through the ballot process rather than deferring the work.

**Applied Change:** No structural change. Address specific technical concerns (security, error handling, documentation) raised in related tickets (FHIR-54074, FHIR-53953) as separate corrections within the normative content.

## Recommendation

**Disposition A: Persuasive** — Move the General Service API to an incubator IG.

The evidence strongly supports this disposition. The General Service API was maturity level 0 (draft) in R5 and there is minimal known implementation experience. The closely related ticket FHIR-53954 (`%factory` and `%server` not mature enough for normative) has already been resolved as Persuasive, and FHIR-54745 (Type Factory) was also resolved Persuasive — establishing a clear precedent. Multiple ballot commenters (Brian Postlethwaite via FHIR-54074 and FHIR-53953) have raised substantive concerns about missing security documentation, error handling, authorization models, and practical usability that would need to be resolved before normative status. Moving to an incubator IG allows this content to mature with proper implementation feedback while keeping the normative FHIRPath specification focused on proven, widely-implemented functionality.
