## Description

How does a client know what events (message types) are supported by the host EHR in SMART Web Messaging?

## Keywords

`discover`, `event`, `host`, `client`, `support`

## History & Prior Discussions

This ticket was originally filed during the initial ballot of SMART Web Messaging (Oct 2020). During ballot reconciliation on 2021-02-19, it was voted **8-0-0** (Gino Canessa / Bas van den Heuvel) to defer as **"Considered for Future Use"** after lengthy discussion ([Confluence: Capabilities and Discovery](https://confluence.hl7.org/pages/viewpage.action?pageId=104565559#SMARTWebMessagingBallotReconciliationCC20210219-CapabilitiesandDiscovery)).

The reconciliation produced two concrete proposals:

1. **Handshake note:** Add to the handshake documentation a note that the WG is looking to standardize discovery behavior in a future release, based on implementation experience.
2. **`.well-known` capability:** Define a `"smart-web-messaging"` value that servers MAY include in their `.well-known/smart-configuration` file within the `capabilities[]` array.

Carl Anderson (Microsoft) created a [shared design document](https://docs.google.com/document/d/1CUoQbsXv_gcWrqIFF-hme8qrqztRUQqp1yArVNkVWzE/edit?usp=sharing) to collect design considerations for this topic (Feb 2021).

After publication of STU1 (May 2022), Lynn Laakso re-opened this ticket to be addressed in a future cycle.

## Analysis

This ticket addresses a fundamental gap in the SMART Web Messaging specification: **there is no standardized mechanism for a client to discover which message types a host EHR supports.** The problem was recognized during the original ballot but deliberately deferred pending implementation experience.

Since the deferral, significant new evidence has emerged:

1. **Real implementations now exist** — Axel Vanraes (SDC/EHR integration) and Pete O'Toole (digital pathology) have both built SMART Web Messaging implementations, providing the real-world experience the WG wanted before specifying discovery.
2. **Implementers are hitting this exact problem** — The Feb 2025 Zulip discussion shows implementers asking how to handle unsupported message types, exactly as predicted.
3. **Two discovery mechanisms were already proposed** — the handshake-based approach and the `.well-known/smart-configuration` approach from the 2021 reconciliation.
4. **The broader FHIR ecosystem has parallel patterns** — FHIR-47954 (Application Feature Framework) and FHIR-31463 (Subscriptions Backport) show this is a recurring capability-discovery challenge across specifications.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Specify handshake-based discovery

**Action:** Add normative guidance for the handshake protocol to include a machine-readable enumeration of supported message types. During the initial `smart_web_messaging_connect` handshake, the host SHOULD include a `supportedMessageTypes` array listing the message types it supports. Define error response semantics for unsupported message types.

**Rationale:** The handshake approach was already identified in the 2021 reconciliation, is contextually aware (can vary per session/user), and multiple implementers (Josh Mandel, Cooper Thompson) have pointed to this as the natural place for discovery. Implementation experience from Axel Vanraes confirms the need. This is the most granular and dynamic approach.

### Disposition B: Persuasive with Modification — Specify `.well-known` discovery plus handshake guidance

**Action:** (1) Define a `smart-web-messaging` entry in `.well-known/smart-configuration` `capabilities[]` to indicate general SWM support. (2) Add an optional `smart_web_messaging_supported_types` field in the SMART configuration listing supported message types. (3) Also add informative guidance for the handshake to refine at runtime.

**Rationale:** This is a two-layer approach: static discovery via `.well-known` for pre-integration planning (as Cooper Thompson advocated), plus handshake refinement for runtime. Addresses both the "weeks before go-live" and "at runtime" discovery needs.

### Disposition C: Deferred — Gather more implementation experience

**Action:** Keep the ticket open but defer again. Request that implementers (Axel Vanraes, Pete O'Toole, others) document their current discovery workarounds and report back to the WG.

**Rationale:** While implementations now exist, the specification may benefit from more diverse implementation experience before standardizing. The risk is premature commitment to a discovery mechanism that doesn't align with real-world needs.

## Recommendation

**Disposition B: Persuasive with Modification — Specify `.well-known` discovery plus handshake guidance.**

The original deferral was justified by lack of implementation experience, but that gap has now been filled. The two-layer approach addresses both pre-integration planning (Cooper Thompson's concern) and runtime discovery (the handshake). The `.well-known` capability flag was already proposed in the 2021 reconciliation with 8-0-0 consensus, so there is prior agreement on this direction. Further deferral risks losing the momentum from active implementers who are hitting this problem today.
