## Summary

The reporter identifies an inconsistency in the FHIR search documentation. In section 3.2.1.5.1 (Search Parameter Types summary), the text states that a "Search Prefix" is "allowed on the `number` search type," but it omits `date` and `quantity` ÔÇö two other search types that also support prefixes. This contradicts the dedicated [Search Prefix section](https://hl7.org/fhir/6.0.0-ballot4/search.html#prefix), which clearly states that prefixes apply to "the ordered parameter types of number, date, and quantity." The reporter requests that `date` and `quantity` be added to the description in section 3.2.1.5.1 so the documentation is internally consistent.

A follow-up comment from the reporter (JIRAUSER27471, same person) clarifies formatting issues in the original description and pinpoints the exact sentence beginning with "Finally, a Search Prefix, allowed on the number search typeÔÇª"

## Proposed Dispositions

### Disposition A: Persuasive

**Rationale:** The text in section 3.2.1.5.1 is factually incomplete. It states that search prefixes are "allowed on the `number` search type" without mentioning `date` or `quantity`, which also support prefixes as documented in the [Search Prefix section](https://hl7.org/fhir/search.html#prefix). This creates a misleading impression that only `number` parameters accept prefixes. Adding `date` and `quantity` to this sentence is a low-risk, high-clarity documentation fix that brings section 3.2.1.5.1 into alignment with the authoritative prefix section. The Zulip discussion from 2021 confirms this inconsistency has already confused implementers.

**Action:** Update the sentence in section 3.2.1.5.1 from "allowed on the `number` search type" to "allowed on the `number`, `date`, and `quantity` search types." Review other sections for similar omissions and correct them as well.

---

### Disposition B: Not Persuasive

**Rationale:** The authoritative definition of prefix-supporting search types is in the [Search Prefix section](https://hl7.org/fhir/search.html#prefix), which already correctly lists all three types (`number`, `date`, `quantity`). Section 3.2.1.5.1 is an introductory overview that uses `number` as a representative example, not an exhaustive enumeration. Readers are expected to follow the hyperlink to the prefix section for complete details. Duplicating the full list in every mention creates a maintenance burden and risks future divergence if the set of prefix-supporting types ever changes.

**Action:** No change to the specification.

---

### Disposition C: Persuasive with Modification

**Rationale:** The reporter's observation is valid ÔÇö the incomplete mention of only `number` is misleading. However, rather than enumerating all prefix-supporting types in section 3.2.1.5.1 (and potentially other locations), a more maintainable approach is to reword the sentence to reference the prefix section without listing specific types. This avoids hardcoding the list in multiple locations while still directing readers to accurate information.

**Action:** Reword the sentence in section 3.2.1.5.1 to something like: "Finally, a [Search Prefix](search.html#prefix), allowed on ordered search types (number, date, and quantity), can be used to change the test to something like a 'greater than'ÔÇª" and add an explicit cross-reference. Additionally, audit the `number`, `date`, and `quantity` search type sections for consistent language about prefix optionality (currently only `quantity` states "the prefix is optional").

---

## Recommendation

**Disposition C: Persuasive with Modification**

The documentation inconsistency is real and has previously confused implementers (as evidenced by the 2021 Zulip thread). Disposition A addresses the immediate issue but risks the same problem recurring elsewhere. Disposition C takes the stronger approach: it corrects the specific sentence, adds all three types with a cross-reference to the authoritative prefix section, and recommends an audit of the individual type sections for consistent language about prefix optionality. This produces a more robust and maintainable result with minimal additional effort.
