**Description:** The submitter asks whether FSH (FHIR Shorthand) should auto-populate the `display` element when only `system` and `code` are provided for a Coding. The submitter recommends that auto-population should **not** be the default behavior (to allow intentional exclusion), but proposes adding a simple language construct — e.g., a `+` suffix like `http://snomed.info/sct#363346000 +` — that would trigger lookup and insertion of the default display value from the relevant code system.

## 3. Prior Vote History

- **2021-09-24:** FHIR-I voted **10-0-1** to resolve as **"Considered for Future Use"**. Rationale: auto-populating display would require the FSH interpreter (SUSHI) to have knowledge of every code system used, necessitating terminology server access for all code systems — presenting significant practical difficulty.
- **2021-10-25 to 2021-10-27:** Mark Kramer and Lloyd McKenzie attempted to correct the Jira status from "Resolved - Change Required" to "Deferred" to match the actual vote outcome. Administrative Jira issues prevented clean correction.
- **2022-02-17:** Lynn Laakso re-opened the ticket post-publication of the Normative Shorthand specification for reconsideration.
- **Current state:** Triaged / Unresolved — the ticket remains open for re-evaluation.

## 5. Key Discussion Points from Comments

1. **Submitter's rebuttal (Bas van den Heuvel, 2021-09-24):** Argued the feature should not be dismissed — users could be warned when code systems are unavailable. The feature would prevent copy/paste errors and incorrect display values, saving significant work. Suggested deferring to a future version rather than rejecting outright.

2. **Original resolution rationale (2021-09-24, vote 10-0-1):** FSH auto-populating display would require the interpreter to access a terminology server for every code system referenced — an impractical requirement given the large number of code systems in use. The concern is architectural: FSH is designed to be interpretable without external service dependencies.

3. **Administrative complexity (Mark Kramer / McKenzie, 2021-10-25 to 2021-10-27):** The ticket was reopened and re-closed multiple times to correct Jira status, not to revisit the technical decision.

4. **Post-normative reopening (Lynn Laakso, 2022-02-17):** The ticket was explicitly re-opened for reconsideration after the Shorthand specification achieved normative status, suggesting the workgroup intended to revisit this feature.

## 6. Proposed Dispositions

### Disposition A: Deferred (Recommend for Future Consideration)

**Action:** Formally defer with specific criteria for reconsideration.

**Rationale:** The original concern — that FSH interpreters would need terminology server access — remains valid but is increasingly mitigated. Modern SUSHI already supports some terminology server interactions (e.g., for value set expansion). The ecosystem has matured since the 2021 vote, and terminology server availability is more common in IG development workflows. However, the feature still requires careful design to avoid making terminology server access a hard requirement.

**Specific changes:** No specification changes now. Document the feature as a candidate for a future FSH version, contingent on:
- A design that makes display auto-population opt-in (as the submitter originally proposed)
- Graceful degradation when terminology servers are unavailable (warning, not failure)
- Community input on the proposed syntax (e.g., the `+` suffix)

---

### Disposition B: Not Persuasive

**Action:** Close without change.

**Rationale:** The fundamental architectural concern has not changed. FSH is designed to be a self-contained authoring language that does not require external service dependencies. Introducing terminology server lookups — even as opt-in — creates a category of FSH files that cannot be fully processed without network access. Authors who want display values can simply include them in their FSH source, which is the current expected practice. The benefit (convenience) does not outweigh the cost (architectural complexity and potential for inconsistent behavior depending on server availability).

**Specific changes:** None.

---

### Disposition C: Persuasive with Modification

**Action:** Accept the concept but scope it to SUSHI tooling guidance rather than FSH language syntax.

**Rationale:** Rather than adding new FSH syntax, recommend that SUSHI (the FSH compiler) offer an optional flag or configuration setting to auto-populate missing display values during compilation when a terminology server is available. This keeps FSH syntax clean and self-contained while providing the convenience benefit the submitter seeks. The behavior would be entirely tool-side, not language-side.

**Specific changes:**
- Add a non-normative note in the FSH specification acknowledging that tooling MAY offer display auto-population as an optional feature
- Recommend that such tooling emit a warning when display values cannot be resolved
- No new FSH syntax is introduced

---

## 7. Recommendation

**Recommended disposition: A — Deferred (Recommend for Future Consideration)**

**Justification:** This ticket has a clear prior vote history (10-0-1 for "Considered for Future Use") and was intentionally reopened for reconsideration after the normative publication of the Shorthand spec. The submitter's original request is reasonable and well-scoped (opt-in, not default), and the broader FHIR ecosystem trend toward requiring display values (see FHIR-40361, FHIR-56066) strengthens the case for eventually supporting this. However, the architectural concern about terminology server dependency remains valid and requires careful design work. A formal deferral with documented criteria for reconsideration respects both the original vote outcome and the evolving ecosystem needs, while avoiding premature specification changes.
