## Description Summary

The ticket responds "Yes" to a question posed in the SMART Web Messaging specification:

> **Existing Wording (question in the spec):** "FHIRCast and CDS Hooks specify their events (or hooks) in separate specifications, using their own maturity models and lifecycles. Should the SMART Web Messaging adopt similar patterns for its message types and activities?"

The submitter (Paul Knapp) answers that it is preferable to have the message exchange specification as one specification, with message definitions as separate documents. This provides separation of concerns and allows messages to evolve without requiring reballoting of the core infrastructure specification each time a new message is defined.

## Analysis

This ticket was originally filed during the STU1 ballot cycle (October 2020) as ballot feedback answering the specification's own posed question about governance structure. The workgroup initially deferred it as "Considered for Future Use" (7-0-0, March 2021), acknowledging the suggestion but deeming it premature during STU1. After STU1 publication, the ticket was re-opened (May 2022) for consideration in a future version.

The core question is an architectural/governance one: should SMART Web Messaging follow the same pattern as FHIRCast (which defines events in separate specifications) and CDS Hooks (which defines hooks separately), allowing message type definitions to evolve independently of the transport/exchange specification?

This question has become more pressing as the Zulip discussions show real-world implementers (Brian Postlethwaite, Axel Vanraes) already creating domain-specific message types (e.g., SDC/Questionnaire-specific messages) outside the core spec. The experience reported by Axel Vanraes — "we started trying to fit everything in the existing message types but ended up with custom ones" — directly validates the need for the separation this ticket recommends.

FHIR-31052 also illustrates the tension: that ticket proposes significant restructuring of existing message groups, which would be easier to manage if message definitions were governed separately from the core protocol.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Accept the recommendation and adopt a governance model where SMART Web Messaging message type definitions are maintained in separate, independently-balloted specification(s), distinct from the core message exchange protocol.

**Rationale:** The submitter's answer aligns with established HL7 precedent (FHIRCast events, CDS Hooks hook definitions). Real-world implementation experience (SDC SMART Web Messaging work in 2025) confirms that domain-specific message types are being defined outside the core spec already. Formalizing this separation provides a clear governance path, allows message types to mature at their own pace, and avoids forcing reballoting of the core infrastructure for each new message definition.

### Disposition B: Considered for Future Use
**Action:** Defer the decision to a future ballot cycle, acknowledging the recommendation but not acting on it yet.

**Rationale:** While the recommendation has merit, the specification may not yet have enough mature message type definitions to justify the overhead of establishing a separate governance framework. The workgroup could wait until there are more concrete message type proposals (e.g., from the SDC SMART Web Messaging work) before formalizing the separation.

### Disposition C: Persuasive with Modification
**Action:** Accept the recommendation in principle, but instead of mandating fully separate specifications, add normative language to the SMART Web Messaging spec that (1) defines a clear extension mechanism for domain-specific message types, (2) establishes maturity criteria for message types, and (3) explicitly allows message type definitions to be published in companion IGs without requiring changes to the core spec.

**Rationale:** This achieves the submitter's goal of separation of concerns without the full overhead of a separate specification governance track. It provides a lighter-weight path that formalizes what implementers are already doing (defining custom message types) while keeping a single authoritative specification for the core protocol and extension framework.

---

## Recommendation

**Disposition C: Persuasive with Modification** is recommended.

The submitter's core insight — that message definitions should evolve independently from the transport protocol — is validated by both HL7 precedent and current implementation experience. However, mandating fully separate specifications (Disposition A) may be premature given the specification's maturity level. A pragmatic middle ground (Disposition C) formalizes the extension/separation pattern that implementers are already following, establishes the governance framework for future message type IGs, and avoids the overhead of splitting existing content into separate ballot tracks before there is sufficient volume to justify it. This approach also aligns with the direction evidenced by the SDC SMART Web Messaging work, which defines domain-specific messages in a companion specification while relying on the core protocol.
