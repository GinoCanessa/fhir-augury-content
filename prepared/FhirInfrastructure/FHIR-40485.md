## Summary

This ticket requests adding an optional `aud` (audience) parameter to the SMART on FHIR token introspection endpoint. In systems with multiple FHIR endpoints—particularly "facade" servers where internal IDs are converted to FHIR IDs in an endpoint-specific manner—the patient ID returned by token introspection may differ depending on which FHIR endpoint is being targeted. By including an `aud` parameter specifying the target FHIR endpoint, the authorization server can perform the correct ID conversion and return the appropriate patient FHIR ID for that endpoint.

## Details

### Description

Token introspection returns the patient ID for patient-scoped tokens, but in systems that have multiple FHIR endpoints, the correct FHIR patient ID can differ depending on the endpoint. This occurs in "facade" servers where internal IDs are converted (in an endpoint-specific way) to FHIR IDs. Adding an optional audience parameter—specifying which FHIR endpoint the introspection is for—would enable the system to perform the correct conversion and return the right patient ID.

## Proposed Dispositions

### Disposition A: Persuasive (Accept)
Accept the request and add an optional `aud` parameter to the token introspection request in the SMART on FHIR specification. When the `aud` parameter is provided, the authorization server SHOULD return context values (e.g., `patient`) appropriate for the specified FHIR endpoint. When omitted, the authorization server returns context values for its default/primary endpoint (preserving backward compatibility).

**Rationale:** The problem is real and documented by implementers operating facade servers with multiple FHIR endpoints. The related ticket FHIR-36179 (adding `aud` to the `/token` response) was already accepted with modification, establishing the pattern. Josh Mandel (SMART spec editor) acknowledged in Zulip that this information is worth putting in introspection responses. Adding `aud` as an optional input to introspection is a natural, low-risk complement to the already-accepted output-side change.

### Disposition B: Persuasive with Modification
Accept the concept but expand the scope: instead of only adding `aud` to token introspection, define a general pattern where the `aud` parameter can be included in any context-returning interaction (introspection, token response, userinfo) so the authorization server can return endpoint-specific context values consistently. This would align with FHIR-36179 and FHIR-33130 to provide a unified solution.

**Rationale:** The underlying problem (context values varying by endpoint) affects multiple touchpoints, not just introspection. A holistic approach ensures consistency and avoids piecemeal specification changes.

### Disposition C: Not Persuasive (Decline)
Decline the request on the grounds that the `aud` is already implicitly known to the resource server performing introspection (since it is introspecting a token it received), and that the related FHIR-36179 change (adding `aud` to the token response) already provides sufficient audience-awareness in the SMART ecosystem.

**Rationale:** The resource server calling introspection already knows which endpoint it is—it can include its own identity via other mechanisms (e.g., client authentication to the introspection endpoint). Adding another optional parameter increases specification surface area for a use case that may already be adequately addressed by FHIR-36179 and FHIR-43623.

## Recommendation

**Disposition A: Persuasive (Accept)**

The request is well-motivated by real-world implementer experience with facade architectures, has SMART spec editor endorsement (Josh Mandel's Zulip comment), and aligns with the already-accepted FHIR-36179 pattern of audience-aware responses. The change is additive and optional, posing no backward-compatibility risk. The specific, focused scope of adding `aud` to introspection is preferable to Disposition B's broader expansion, which could delay resolution. Disposition C's argument that the resource server already knows its own identity is valid but overlooks the key insight: the *authorization server* needs to know which endpoint-specific patient ID to return, and the `aud` parameter is the cleanest way to communicate that.
