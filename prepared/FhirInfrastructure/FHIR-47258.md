## Summary

The submitter asks whether the exact values for **context** parameters in the Feature Query operation are fully defined by the Application Feature Framework IG, or whether they are intended to be specified by downstream IGs that depend on this framework (e.g., the context for `ValueSet.$expand`). The IG defines *how* to structure feature + context + value combinations in the operation input/output, but the submitter found no clear specification of *what* the actual context values should be.

The resolution states that context for any specific feature will be described using the **FeatureDefinition** logical model, pointing to: https://build.fhir.org/ig/HL7/capstmt/StructureDefinition-FeatureDefinition.html

---

## Details

### Description

> While this IG has clearly defined how the context value should be specified (i.e. how to specify feature + context + value combos on the feature query operation input/output), I'm not sure what the exact value of context should be. Is the exact value for context supposed to be defined by subsequent IGs that will be dependent on this IG (like the context for ValueSet.$expand)?

### Resolution Description

> The context for any specific feature will be described using the FeatureDefinition logical model:
> https://build.fhir.org/ig/HL7/capstmt/StructureDefinition-FeatureDefinition.html

## Proposed Dispositions

### Disposition A: Accept (Considered - Question Answered)

**Rationale:** The resolution accurately answers the question. The FeatureDefinition logical model at `https://build.fhir.org/ig/HL7/capstmt/StructureDefinition-FeatureDefinition.html` is the mechanism by which context values for any specific feature are defined. The framework IG intentionally defines the *structure* for feature + context + value combinations, while individual FeatureDefinition instances (created in this or downstream IGs) specify the *actual* context values. This is consistent with the extensible design philosophy — the framework provides the grammar, and FeatureDefinition instances provide the vocabulary.

**Action:** Accept the existing resolution as-is. No specification changes required.

### Disposition B: Accept with Clarification Enhancement

**Rationale:** While the resolution is correct, the IG itself could benefit from a brief informative note or paragraph in the Feature Query operation section explicitly stating that context values are defined per-feature via FeatureDefinition instances, and that downstream IGs are expected to define their own FeatureDefinition instances for their specific features. This would prevent future readers from having the same question.

**Action:** Accept the resolution and additionally request that a brief clarifying statement be added to the Feature Query operation documentation in the IG, pointing implementers to FeatureDefinition as the source of context value definitions. This could be tracked as a separate enhancement ticket.

### Disposition C: Decline (Not Persuasive)

**Rationale:** The question is valid but the answer is already implicit in the IG's design. The FeatureDefinition logical model is published and linked. Implementers familiar with FHIR's general pattern of canonical-URL-identified definition resources should be able to discover this mechanism. No further action is warranted.

**Action:** Mark as Not Persuasive. No changes to the specification.

---

## Recommendation

**Disposition A: Accept (Considered - Question Answered)**

The existing resolution is accurate, complete, and sufficient. The FeatureDefinition logical model is the correct and designated mechanism for defining context values per feature. The question has been properly answered by pointing to this structure. The related Zulip discussions and sibling tickets (FHIR-47255, FHIR-47963) confirm that the community understands and is actively using FeatureDefinition as the vehicle for specifying feature contexts in downstream IGs (e.g., AU PS, tx-ecosystem). No specification change is needed beyond the resolution already provided.
