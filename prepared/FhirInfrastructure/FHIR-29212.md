# FHIR-29212: CapabilityStatement state support of async

> **Prepared for:** FHIR Infrastructure Work Group Review
> **Prepared on:** 2025-07-17
> **Ticket URL:** https://jira.hl7.org/browse/FHIR-29212

## 1. Summary

**Status:** Triaged | **Type:** Change Request | **Priority:** Medium
**Specification:** FHIR Core (FHIR) | **Labels:** CapabilityStatement2
**Reporter:** Bas van den Heuvel (2020-10-12) | **Assignee:** Corey Spears
**Resolution:** Unresolved

The submitter asks how a server can declare in its CapabilityStatement that it supports asynchronous (async) request patterns. The request specifies that async support should be expressible at the server, resource, and operation levels.

### History

1. **2020-10** — Ticket filed by Bas van den Heuvel.
2. **2021-01** — Lloyd McKenzie suggested experimenting with the new "feature" approach in CapabilityStatement2.
3. **2021-06** — Bas confirmed the feature mechanism would work and proposed adding `synchronous` and `asynchronous` codes to the `capability-features` CodeSystem, applicable at `rest`, `rest.resource`, `rest.interaction`, `rest.operation`, `rest.resource.interaction`, and `rest.resource.operation` levels, with deeper declarations overriding higher ones.
4. **2021-06-28** — Originally resolved as **Not Persuasive with Modification** (vote: Ewout Kramer / Josh Mandel, 11-0-0). The plan was to add the proposed codes to the CapabilityStatement2 `capability-features` CodeSystem.
5. **2022-08** — Grahame Grieve raised on Zulip whether FHIR-I wanted to revisit or defer to a future CS2 IG.
6. **2023-02** — Listed in Grahame's status report as "CapabilityStatement2 stuff — deferred for an implementation guide."
7. **2025-10-27** — Josh Mandel **reverted** the previous resolution, reopening the ticket. He noted that after 4+ years the community now has a better understanding: the **App Feature IG** should define a general feature mechanism, and the **API Incubator IG** will define async-related features. He asked Corey Spears to bring this back to FHIR-I once the App Feature IG is published and the externally defined features mechanism is clear.

## 6. Proposed Dispositions

### Option A: Defer — Wait for App Feature IG (Recommended)

**Disposition:** Not Persuasive with Modification
**Rationale:** The most recent comment from Josh Mandel (2025-10-27) explicitly states the community's current direction: the **App Feature IG** will define a general feature-declaration mechanism, and the **API Incubator IG** will define specific features for async discovery. Acting on this ticket before those IGs are published risks creating a mechanism in FHIR Core that conflicts with or duplicates the IG-based approach. The ticket should remain triaged and be brought back to FHIR-I once the App Feature IG publishes its externally defined features mechanism, as Josh requested of the assignee Corey Spears.

**Action:** Keep status as Triaged. Add a note documenting that resolution depends on the App Feature IG and API Incubator IG. Corey Spears to re-raise when the feature mechanism is available.

---

### Option B: Resolve with CapabilityStatement Feature Codes

**Disposition:** Persuasive with Modification
**Rationale:** The original 2021 resolution (11-0-0 vote) directed adding `synchronous` and `asynchronous` codes to the `capability-features` CodeSystem in CapabilityStatement2. Bas van den Heuvel's proposal was well-defined — boolean-valued features at multiple REST hierarchy levels with deeper declarations overriding higher ones. If the work group believes the feature mechanism is sufficiently mature, the original resolution could be re-applied.

**Action:** Add `synchronous` and `asynchronous` feature codes to the `capability-features` CodeSystem. Define their applicability at `rest`, `rest.resource`, `rest.interaction`, `rest.operation`, `rest.resource.interaction`, and `rest.resource.operation` levels with override semantics.

---

### Option C: Not Persuasive — Out of Scope for FHIR Core

**Disposition:** Not Persuasive
**Rationale:** Async support discovery is an implementation-level concern that may be better addressed entirely within IGs (App Feature IG, API Incubator IG, Bulk Data IG) rather than FHIR Core. Given the repeated deferrals over 4+ years and the explicit plan to handle this in IGs, the work group could close this as not appropriate for FHIR Core and let the IG community own it.

**Action:** Close ticket. Document that async feature discovery should be tracked in the App Feature IG and/or API Incubator IG.

---

## 7. Recommendation

**Option A: Defer — Wait for App Feature IG** is recommended.

This aligns with the most recent guidance from Josh Mandel (October 2025) and the community consensus that the App Feature IG + API Incubator IG is the right home for async discovery features. The original 2021 proposal from Bas van den Heuvel remains sound and can be applied once the feature mechanism is formalized. Premature action in FHIR Core risks duplication or conflict with the IG-based approach. The assignee (Corey Spears) has a clear action item to bring this back when the prerequisite IGs are ready.
