## Description

The reporter requests that SMART on FHIR add support for operation-level scopes. Currently, the SMART scope syntax (`cruds`) covers CRUD interactions and search, but does not address authorization for FHIR operations (e.g., `$everything`, `$export`). The ticket proposes adding an `o` permission character for operations and a suffix syntax to target specific operations:

- `user/Patient.crudso` → permission to use all Patient-based instance and resource operations
- `user/Patient.crudso.everything` → permission to use the `$everything` operation on Patient
- `user/Resource.o` → permission to use all system operations
- `user/Resource.o.export` → permission to use the `$export` operation

## Prior Resolution History

This ticket was previously resolved as **"Considered for Future Use"** on 2021-08-02 with an **8-0-0 vote** (Eric Haas / Carl Anderson). The resolution noted:

> *"We were not able to drive testing for operation-based scopes at connectathons. We explored syntaxes, and raised concerns with several (e.g., the need to bind or fix certain parameters when delegating permission to invoke an operation). The syntax proposed here is an interesting direction, and may be worth testing at connectathons – though it's unclear how ".export" and other suffixes would interact with our existing search parameter based suffixes. Most options we discussed for operation-based scopes used dedicated operation-centric scopes like `user/$export.type(Patient)`; but again, we could not drive consensus on this."*

The ticket was **re-opened** post-publication of STU2 by Lynn Laakso on 2021-11-29.

## Zulip Discussions

### 1. [smart > SMART scopes for operations](https://chat.fhir.org/#narrow/stream/smart/topic/SMART%20scopes%20for%20operations) (Sep 2022 – Jun 2023)

**Participants:** Alexander Zautke, Josh Mandel, Igor Bossenko

Key points:
- **Josh Mandel** confirmed: *"We didn't manage to land on a general-purpose set of scopes for FHIR operations. It's tricky because you sometimes want to say 'you can call this operation, but only on this resource, or this type, or with this or that parameter.'"*
- SMART App State explored a simpler case (system-level operations with required string parameters) but did not mandate any particular scope syntax.
- **Igor Bossenko** raised the practical question of how to scope custom operations (e.g., `MpiPatientDisability`), preferring a separate "execute" permission like `user/MpiPatientDisability.e`.
- Josh Mandel suggested modeling permissions through data scopes (e.g., `patient/Observation.rs?category=...`) rather than operation-specific scopes, letting operation logic make authorization decisions based on data scopes.

### 2. [smart > $everything operation and referenced patients](https://chat.fhir.org/#narrow/stream/smart/topic/%24everything%20operation%20and%20referenced%20patients) (Apr 2023)

**Participants:** Craig McClendon, Josh Mandel

- Josh Mandel stated: *"We don't even have operation-specific scopes in SMART today."*
- Highlighted that `$everything` authorization falls back to general data scopes (`patient/*.read` or `user/*.read`), with no operation-level refinement.

### 3. [smart > SMART scopes for delete-history?](https://chat.fhir.org/#narrow/stream/smart/topic/SMART%20scopes%20for%20delete-history%3F) (Feb 2024)

**Participants:** Michael Lawley, Josh Mandel, Gino Canessa

- Josh Mandel acknowledged the need to unify operations and new R6 interactions (e.g., `delete-history`) from a permissions perspective.
- Noted that operations and interactions *"both occupy the same semantic space (i.e., specifying 'what action') where `.cruds` sits today."*
- Suggested that **IETF Rich Authorization Requests (RAR)** data model may be needed for parameter-level authorization, as the granular scope syntax alone may be insufficient.

## Analysis

### Core Problem
The SMART on FHIR v2 scope syntax (`cruds`) covers standard FHIR interactions but provides no mechanism to authorize FHIR operations. This gap affects:
1. **Custom operations** — implementers have no standard way to express "this app can invoke operation X"
2. **Bulk Data operations** (`$export`) — authorization is implicit rather than explicitly scoped
3. **System operations** — no way to distinguish permission for system-level vs. type-level vs. instance-level operations
4. **New R6 interactions** (e.g., `delete-history`) — similarly unaddressed by current syntax

### Complicating Factors
- **Parameter binding**: Operations often require authorizing not just invocation but specific parameter values (e.g., "you can call `$export` but only for `Patient` resources"). The proposed `.o.export` suffix doesn't capture this.
- **Suffix conflicts**: The proposed syntax (e.g., `.crudso.everything`) could conflict with existing search-parameter-based suffixes in the SMART v2 scope grammar.
- **Diverse use cases**: Approaches that work for simple operations (like App State) break down for parameter-rich or multi-resource operations.
- **Alternative models**: The WG previously discussed dedicated operation-centric scopes (e.g., `user/$export.type(Patient)`) and more recently IETF RAR, but consensus has not been reached on any approach.

### Community Sentiment
As of the most recent Zulip discussions (2024), the SMART maintainers (Josh Mandel) acknowledge the gap but emphasize complexity. The prevailing approach has been to defer to data-level scopes and let operation logic make authorization decisions, rather than defining a new operation-scope syntax.

## Proposed Dispositions

### Disposition A: Deferred — Re-affirm "Considered for Future Use"

**Rationale:** The previous 8-0-0 vote to defer was well-reasoned. Since then, community discussions (Zulip 2022–2024) confirm that no consensus on syntax has emerged. Josh Mandel has indicated IETF RAR may be required for proper parameter-level authorization. Re-affirming the deferral acknowledges the gap while avoiding premature standardization of a syntax that may conflict with granular scope suffixes or prove inadequate for parameter binding.

**Applied Change:** No specification change. Add a tracker note documenting the ongoing community interest and linking to the Zulip discussions for future reference.

### Disposition B: Persuasive — Define Operation Scope Syntax

**Rationale:** Accept the need and define a new permission character (e.g., `o` or `e`) in the SMART scope grammar for operation invocation. Specify syntax like `user/Patient.o$everything` or `user/Resource.o$export` with clear rules for system/type/instance-level operations. This addresses the well-documented gap and gives implementers a standard to test at connectathons.

**Applied Change:** Add an "Operation Scopes" section to the SMART on FHIR specification defining the `o` permission, its syntax, and its interaction with existing `cruds` permissions and search-parameter suffixes. Include examples for common operations (`$everything`, `$export`, `$validate`).

### Disposition C: Persuasive with Modification — Acknowledge Gap, Define Extensibility Point

**Rationale:** Accept the problem statement but modify the approach. Rather than defining a full operation-scope syntax now, add specification language that (1) explicitly acknowledges operations are not covered by `cruds`, (2) defines an extensibility point where operation-specific scopes can be expressed (e.g., using a reserved scope prefix or IETF RAR integration), and (3) provides non-normative guidance for implementers who need operation authorization today (e.g., modeling through data scopes as Josh Mandel suggests).

**Applied Change:** Add a section titled "Operations and Authorization" that documents the current limitation, provides implementer guidance for workarounds, and establishes a framework for future normative operation-scope syntax. Reference IETF RAR as a potential direction.

---

## Recommendation

**Disposition C: Persuasive with Modification** is recommended.

The gap in operation-level authorization is real, well-documented, and increasingly pressing as FHIR operations grow (Bulk Data, App State, R6 interactions). However, the prior 8-0-0 vote to defer and the continued lack of community consensus on syntax (confirmed through 2024 Zulip discussions) indicate that a full normative syntax is premature. Disposition C threads the needle: it formally acknowledges the gap in the specification (which Disposition A does not), provides actionable guidance for current implementers, and establishes a clear extensibility framework without locking in a syntax that may prove inadequate — particularly given Josh Mandel's observation that IETF Rich Authorization Requests may be needed for parameter-level authorization. This positions the WG to iterate toward a normative solution in future cycles while giving implementers something to work with now.
