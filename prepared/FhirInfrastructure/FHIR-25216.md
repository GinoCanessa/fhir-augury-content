# Ticket Review: FHIR-25216

**Title:** What happens if FHIRPath compares FHIR quantities with non-ucum codes?
**Status:** Triaged | **Priority:** Medium | **Type:** Change Request
**Work Group:** FHIR Infrastructure | **Specification:** FHIR Core (FHIR)
**Reporter:** Lee Surprenant | **Assignee:** Unassigned
**Created:** 2019-11-13 | **Updated:** 2026-04-02
**Labels:** None

## Summary

The submitter raises the question of how FHIRPath should handle comparisons of FHIR Quantity values that use non-UCUM unit codes. FHIRPath currently defines quantity comparison semantics primarily around UCUM units, leaving behavior undefined or ambiguous when quantities use coded units from other systems (e.g., proprietary or locally defined unit codes). The ticket requests clarification or specification changes to address equality, ordering, and arithmetic operations (addition, subtraction) for such quantities. Discussion in the comments explores extending FHIRPath with overloads for FHIR Quantity comparison, supporting arithmetic on non-UCUM quantities with matching coded units, and the relationship between quantity equality and general object equality semantics.

## Details

**Description:**

The ticket title poses the core question: what should FHIRPath do when comparing FHIR Quantity values that carry non-UCUM unit codes? The FHIR specification defines Quantity with a system and code for units, and while UCUM is the preferred unit system, quantities may use codes from other systems. FHIRPath's comparison and arithmetic operations for quantities are defined with UCUM conversion in mind, but the behavior for non-UCUM coded quantities is not clearly specified.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Clarify the FHIRPath specification to define comparison and arithmetic behavior for FHIR Quantity values with non-UCUM unit codes. Specifically:
- For equality (`=`, `!=`): two Quantity values with non-UCUM codes are equal if and only if their value, system, and code are all identical.
- For ordering (`<`, `>`, `<=`, `>=`): ordering is defined only when both quantities share the same system and code; otherwise the result is empty (`{}`).
- For addition and subtraction: operations are permitted when both operands share the same system and code, with the result retaining that unit.
- For multiplication/division: operations are permitted when one operand has the UCUM default unit `'1'`.

Add these rules as FHIR-specific overloads in the FHIRPath specification, analogous to the existing overloads for `ofType()`, `toString()`, and equality.
#### Justification
This directly addresses the gap identified by the submitter and aligns with the consensus expressed in the comments. Defining explicit behavior for non-UCUM quantities eliminates ambiguity for implementers and ensures consistent behavior across FHIRPath engines. The approach of requiring identical system and code for comparison and arithmetic is conservative and avoids the complexity of cross-system unit conversion.

---

### Disposition B: Alternative Approach
#### Proposal
Rather than adding full comparison and arithmetic overloads, limit the change to equality semantics only. Define that two non-UCUM Quantity values are equal when their value, system, and code match. For ordering and arithmetic on non-UCUM quantities, return empty (`{}`) to indicate the operation is not supported, and document this as a known limitation. Encourage implementers who need arithmetic on non-UCUM quantities to convert them to UCUM units before performing operations.
#### Justification
This approach addresses the most common use case (equality checks in invariants and search) while avoiding the complexity of defining arithmetic semantics for arbitrary coded units. Since ordering and arithmetic across non-UCUM units could produce misleading results if units are not truly commensurable, restricting these operations to UCUM quantities is a safer default. The equality semantics follow naturally from the existing definition of object equality, as noted by Bryn Rhodes in the comments.

---

### Disposition C: Decline
#### Proposal
Decline the change request on the basis that the current FHIRPath specification already provides sufficient guidance. Quantity comparison and arithmetic are intentionally defined in terms of UCUM because UCUM provides the canonical, unambiguous unit framework needed for safe computation. Non-UCUM quantities should be treated as opaque values for comparison purposes, with equality following from general object equality rules.
#### Justification
The existing specification is intentionally conservative about quantity operations to prevent incorrect calculations from incompatible unit systems. Extending arithmetic and comparison to non-UCUM quantities introduces risk that implementers will perform computations on quantities whose units are not truly comparable. The current behavior—where non-UCUM quantity comparisons return empty—is a safe default that prevents incorrect results. Implementers who need to compare or compute with non-UCUM quantities can do so in application logic outside of FHIRPath.

---

### Recommendation

**Recommended disposition:** A

Disposition A is recommended because the ticket identifies a genuine gap in the FHIRPath specification that affects real-world implementations. FHIR Quantity values frequently use non-UCUM codes (e.g., for scores, counts, or domain-specific measures), and FHIRPath expressions that involve these quantities currently produce undefined or inconsistent behavior across engines. The proposed approach—defining equality, ordering, and arithmetic for non-UCUM quantities when system and code match—is both intuitive and conservative. It does not attempt cross-system conversion, which would be genuinely dangerous, but instead requires exact unit identity. The comment discussion shows broad agreement on this direction, including support for arithmetic on matching units and the `'1'` unit exception for multiplication. The ticket has been open since 2019 and the recent comment from Brian Postlethwaite in 2026 provides a concrete implementation path through FHIR-specific overloads, making this an appropriate time to resolve it.
