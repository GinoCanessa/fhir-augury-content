## Summary

The `Subscription` resource contains four optional coded elementsâ€”`Subscription.filterBy.comparator`, `Subscription.filterBy.modifier`, `Subscription.contentType`, and `Subscription.content`â€”that lack `meaningWhenMissing` documentation. Without specifying what default behavior applies when these elements are absent, implementations may diverge in their handling of omitted values, leading to interoperability issues. The ticket requests that `meaningWhenMissing` be populated for each of these elements so that default behavior is consistent across implementations.

## Proposed Dispositions

### Disposition A: Accept â€” Add `meaningWhenMissing` to All Four Elements

Add `meaningWhenMissing` documentation to each of the four optional coded elements with the following proposed defaults:

- **`Subscription.filterBy.comparator`**: "When not present, the comparator defaults to `eq` (equals), consistent with the default behavior of FHIR search comparators."
- **`Subscription.filterBy.modifier`**: "When not present, no modifier is applied to the filter parameter (exact match on the filter definition)."
- **`Subscription.contentType`**: "When not present, the server determines the content type based on its default serialization format (typically `application/fhir+json`)."
- **`Subscription.content`**: "When not present, the server determines the notification payload level; servers SHOULD default to `id-only` to minimize data exposure."

**Rationale:** This is a low-risk documentation/metadata improvement that directly addresses a real interoperability gap. Implementers currently must guess at default behavior, and different assumptions lead to inconsistent implementations. The proposed defaults align with existing search parameter conventions and security best practices.

### Disposition B: Partial Accept â€” Add `meaningWhenMissing` with Coordination on FHIR-54070

Add `meaningWhenMissing` to `Subscription.filterBy.modifier`, `Subscription.contentType`, and `Subscription.content` now, but defer `Subscription.filterBy.comparator` pending resolution of [FHIR-54070](https://jira.hl7.org/browse/FHIR-54070), which questions whether that element should exist at all. If FHIR-54070 results in removing or deprecating `comparator`, adding `meaningWhenMissing` would be unnecessary for that element.

**Rationale:** Avoids duplicating or conflicting work. The three non-controversial elements can be addressed immediately, while `comparator` is deferred until its future in the resource model is settled.

### Disposition C: Decline â€” Default Behavior Is Implementation-Specific

Decline the change on the grounds that default behavior for optional elements is inherently implementation-specific and should not be prescribed by the specification. Implementations are expected to document their own defaults, and `meaningWhenMissing` could create false expectations of uniformity that servers may not actually honor.

**Rationale:** Adding `meaningWhenMissing` imposes normative expectations about behavior that may not match reality across diverse server implementations. The specification already allows servers to reject subscriptions with missing elements, and prescribing defaults could conflict with server-specific policies.

---

## Recommendation

**Disposition A: Accept** is recommended.

This is a straightforward documentation improvement with clear interoperability benefits and minimal risk. The four elements in question are optional coded values where implementers genuinely need guidance on default behaviorâ€”this is exactly the use case `meaningWhenMissing` was designed for. The proposed defaults align with existing FHIR conventions (e.g., `eq` as the default search comparator) and security best practices (e.g., `id-only` as a safe default for notification content). While FHIR-54070 may eventually affect `filterBy.comparator`, documenting its current default behavior does not conflict with a potential future removalâ€”if the element is removed, the `meaningWhenMissing` goes with it. Addressing all four elements together ensures comprehensive and consistent documentation.
