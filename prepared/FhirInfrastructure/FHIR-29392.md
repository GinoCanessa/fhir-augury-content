## Description

The submitter reports that the SMART Web Messaging IG lacks a section describing data provenance. No specific guidance on how provenance should be captured or communicated is provided within the specification.

## Analysis

This ticket was originally filed during the SMART Web Messaging STU1 ballot in October 2020, identifying the absence of data provenance guidance in the IG. During ballot reconciliation in February 2021, the workgroup voted 7-0-0 to resolve as "Considered for Future Use," and Carl Anderson submitted PR #45 to add a provenance mention in the Limitations section. However:

- **PR #45 was closed without being merged** (dirty mergeable state as of May 2021).
- **The ticket was reopened** by Lynn Laakso in May 2022 after STU1 publication, reverting the prior resolution.

The core issue remains: the SMART Web Messaging IG defines no new FHIR Resources, making traditional Provenance resource guidance arguably inapplicable. However, the specification does facilitate data exchange via messaging between EHR and embedded applications, which raises legitimate questions about how provenance of exchanged data should be tracked.

The ticket is currently **Triaged** and **Unassigned** with no active work since the 2022 reopening.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Add provenance guidance to the Limitations section
**Action:** Merge a revised version of the content from the unmerged PR #45 (or create a new PR) that adds an explicit subsection to the IG's Limitations section acknowledging the lack of provenance specification and directing implementers to the FHIR Provenance resource and relevant US Core Provenance guidance where applicable. Mark the ticket as resolved.

**Rationale:** This is the minimal viable resolution. The workgroup already voted 7-0-0 in favor of this approach during the original ballot reconciliation. Reopening after STU1 means the content from PR #45 was never actually incorporated. Re-applying this approach addresses the ballot comment while keeping scope appropriate for an IG that defines no new Resources.

### Disposition B: Deferred — Defer provenance specification to a future STU cycle
**Action:** Resolve as "Deferred" with a note that provenance guidance should be addressed in a future version of the SMART Web Messaging IG, potentially in coordination with broader FHIR provenance efforts (e.g., FHIR-54528 on AI provenance, Basic Provenance guidance in US Core).

**Rationale:** Since the IG defines no new Resources and the messaging pattern is relatively lightweight, detailed provenance guidance may be premature until the broader FHIR ecosystem has more mature provenance patterns. This preserves the intent of the original 7-0-0 vote but defers without adding any content.

### Disposition C: Not Persuasive — No provenance section needed
**Action:** Resolve as "Not Persuasive" with an explanation that the SMART Web Messaging IG is a messaging framework specification that does not define new FHIR Resources, and therefore data provenance is the responsibility of the applications exchanging data via the messaging channel, not the messaging specification itself.

**Rationale:** Provenance tracking for data exchanged via SMART Web Messaging is an application-level concern. The IG specifies how messages are structured and transmitted, not how the data within those messages should be attributed. Adding provenance guidance here could create confusion about where responsibility lies.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended. The workgroup already reached consensus (7-0-0) on adding a provenance mention to the Limitations section, and the only reason this remains open is that the implementing PR was never merged. Reviving and merging that content (or an updated version) is the most straightforward path to resolution and honors the prior workgroup decision. The ticket has been open for nearly five years and this approach has the lowest risk of further delay.
