## Summary

This ticket requests that FHIR Shorthand (FSH) support autopopulating the formal (system) and display elements of a Quantity when only the UCUM code is provided, since UCUM already defines both. The reporter argues this would make FSH more concise and less error-prone. The ticket was previously resolved as "Considered for Future Use" during FSH Block Vote #1 (vote: 11-2-3), with the rationale that requiring knowledge of code systems crosses into validation territory, which is outside FSH's scope. It was subsequently re-opened post-publication of the Normative Shorthand spec. The reporter notes that this request is closely related to FHIR-33559 ("Autopopulate display"), which addresses auto-populating the display element when only system and code are provided, and suggests they should be discussed together. Both tickets remain unresolved and triaged.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Enhance FHIR Shorthand tooling (e.g., SUSHI) to support autopopulation of the `Quantity.system` (http://unitsofmeasure.org) and `Quantity.unit` (display) elements when only the UCUM code is provided. This could be implemented by bundling or referencing a UCUM code-to-display mapping within the FSH compiler, allowing authors to write a UCUM code alone and have the remaining Quantity fields filled in automatically.
#### Justification
UCUM is a well-defined, stable code system with a finite set of codes and corresponding display values. Unlike arbitrary code systems, UCUM's system URI is always `http://unitsofmeasure.org`, making auto-resolution straightforward. This would reduce repetitive boilerplate in FSH files, decrease transcription errors, and improve authoring efficiency — particularly for implementation guides that are quantity-heavy.

---

### Disposition B: Alternative Approach
#### Proposal
Rather than embedding UCUM knowledge directly into the FSH language or compiler, provide an optional external mechanism — such as a UCUM import file or a lookup configuration — that FSH authors can include to enable autopopulation. This would keep the FSH specification itself code-system-agnostic while allowing tooling (e.g., SUSHI) to support the feature as an opt-in extension. This approach should be coordinated with FHIR-33559 ("Autopopulate display"), which addresses the same underlying need for general code system display resolution.
#### Justification
This preserves the FSH design principle of not requiring code system knowledge within the language itself, while still addressing the author's need. An opt-in mechanism avoids scope creep and keeps the core FSH specification simple. Coordinating with FHIR-33559 ensures a consistent approach across both UCUM-specific and general code system display autopopulation.

---

### Disposition C: Decline
#### Proposal
Decline the request. FHIR Shorthand should remain code-system-agnostic. Autopopulating UCUM elements crosses into validation and terminology resolution territory, which is outside FSH's scope.
#### Justification
The original block vote resolution (11-2-3) already established that requiring knowledge of code systems is outside FSH's scope. Keeping FSH focused on structural authoring without terminology dependencies maintains its simplicity, avoids coupling to external terminology services or data files, and ensures that all FSH implementations can remain lightweight. Authors can explicitly specify the system and display values when needed.

---

### Recommendation

**Recommended disposition:** B

Disposition B strikes the best balance between addressing the real authoring pain point and preserving FSH's design philosophy. The reporter and the previous work group discussion both acknowledge that the core challenge is code system availability — not the FSH language itself. An opt-in mechanism (such as an importable UCUM mapping or a tooling-level configuration) would allow FSH authors who want this convenience to use it without imposing terminology dependencies on the FSH specification or all implementations. Coordinating this with FHIR-33559 ("Autopopulate display") would yield a coherent, general-purpose solution rather than a UCUM-specific special case. This approach also aligns with the work group's earlier stated intent to "discuss how best to meet this requirement — possibly incorporating other layers of the publishing process."
