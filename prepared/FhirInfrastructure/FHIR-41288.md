## Summary

The "Compare to R4" and "Compare to R4B" links at the bottom of FHIR specification pages (e.g., the base resource page) returned 404 errors. The comparison links pointed to the W3C HTML diff service (`services.w3.org/htmldiff`), which was failing — either because the target page did not exist in the earlier FHIR version (e.g., a page added in R5 has no R4B counterpart), or because the W3C service itself became unavailable.

The issue has been resolved by hosting the comparison service directly within HL7 infrastructure, removing the dependency on the external W3C service. The resolution also notes this is not strictly a specification issue, but since it has been addressed it is being treated as **Persuasive**.

## Details

### Description

An error (404 Not Found) occurred when trying to access `http://hl7.org/fhir/R4B/types.html` via the comparison link. The reporter noted that the link may not be appropriate for pages added in R5, since those pages have no R4B equivalent.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Confirm Existing Resolution)

Accept the ticket's existing Persuasive resolution. The underlying issue — broken "Compare to R4/R4B" links caused by reliance on the external W3C HTML diff service — has been fully resolved by hosting the comparison service within HL7 infrastructure. No specification text changes are required since the links are generated by tooling, not authored in the spec itself. The fix also benefits the related ticket FHIR-46684.

**Actionability:** No further action required; the fix is already deployed.

### Disposition B: Accept as Persuasive with Additional Tooling Improvement

Accept as Persuasive and additionally recommend that the FHIR build tooling be updated to suppress comparison links for pages that are new to the current version (i.e., have no counterpart in the comparison target version). As Grahame McKenzie noted, the tooling could check whether the target page exists and either suppress the link or display "New to R6" instead. This would prevent 404 errors for pages introduced in later versions even with the self-hosted service.

**Actionability:** File a separate tooling issue or enhancement request for the FHIR publisher to detect and suppress invalid comparison links.

### Disposition C: Decline — Not a Specification Issue

Decline the ticket on the grounds that this is a tooling/infrastructure issue rather than a specification change. The comparison links are generated by the FHIR publisher tooling and are not part of the normative specification content. The fix was an infrastructure change (re-hosting the diff service) and did not involve any specification text modifications.

**Actionability:** Close with no specification action.

---

## Recommendation

**Disposition A: Accept as Persuasive (Confirm Existing Resolution)**

The issue has been definitively resolved — the comparison service is now hosted by HL7, eliminating the W3C dependency that caused the failures. The existing Persuasive resolution accurately reflects that the problem was real, the reporter's concern was valid, and the fix has been applied. Since no specification text was involved and the fix is already deployed, confirming the existing resolution is the cleanest path forward. The additional tooling improvement suggested in Disposition B is worthwhile but should be tracked separately rather than holding up this ticket's disposition.
