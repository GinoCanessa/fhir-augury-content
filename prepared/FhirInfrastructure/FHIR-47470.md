## Summary

The submitter requests that a preferred ValueSet be added to `Bundle.link.relation` via `binding.additional` to surface the most commonly used navigation link relation codes — specifically `first`, `last`, `self`, `prev`, and `next`. The current binding references the IANA Link Relations ValueSet, which contains approximately 120 entries, most of which are rarely used in FHIR Bundle contexts. The resolution is "Persuasive," indicating agreement to create the preferred additional binding as proposed.

## Details

**Description:**

Bundle.link.relation points to valueset-iana-link-relations which has 120 entries.

The navigation links (first, last, self, prev, next) are used in the vast majority of the cases. Add a preferred ValueSet to binding.additional that indicates these as preferred.

Will create the preferred additional binding as proposed.

## Proposed Dispositions

### Disposition A: Accept As Requested
#### Proposal
Create a new preferred ValueSet (e.g., `bundle-link-relation-preferred`) containing the five commonly used navigation link relation codes: `first`, `last`, `self`, `prev`, and `next`. Add this ValueSet as a preferred `binding.additional` on `Bundle.link.relation`, leaving the existing required binding to the full IANA Link Relations ValueSet unchanged.
#### Justification
The five navigation codes represent the vast majority of real-world usage of `Bundle.link.relation`. Surfacing them via a preferred additional binding improves developer experience by making the most relevant codes immediately visible without scrolling through 120+ IANA entries. This approach mirrors the pattern already being applied to `Patient.language`, and the resolution has already been marked as Persuasive by the work group.

---

### Disposition B: Alternative Approach
#### Proposal
Instead of creating a formal additional binding, add a short description or comment to `Bundle.link.relation` explicitly listing the five most common navigation codes (`first`, `last`, `self`, `prev`, `next`) and noting their predominant use in FHIR Bundles. This avoids any dependency on tooling support for `binding.additional` in the core spec publication pipeline.
#### Justification
If core spec tooling does not yet fully support publishing additional bindings (as raised in the 2024-09-16 comment), a descriptive comment provides equivalent guidance to implementers without requiring tooling changes. This is a lower-risk approach that still addresses the usability concern. However, it is less machine-readable and does not provide the same validation benefits as a formal binding.

---

### Disposition C: Decline
#### Proposal
Decline the change on the basis that the existing required binding to the IANA Link Relations ValueSet is sufficient and that additional bindings add specification complexity.
#### Justification
The current binding already includes all five navigation codes. Implementers familiar with FHIR pagination are generally aware of these codes. Adding a preferred additional binding increases the maintenance surface of the specification and may set a precedent for similar requests on other elements with large ValueSets. However, given the work group's Persuasive resolution and the parallel work on `Patient.language`, this disposition is difficult to defend.

---

### Recommendation
**Recommended disposition:** A

Disposition A is the strongest choice because the work group has already resolved this ticket as Persuasive with an explicit plan to create the preferred additional binding. The precedent set by the analogous change to `Patient.language` demonstrates that the pattern is accepted. The five navigation codes (`first`, `last`, `self`, `prev`, `next`) are clearly the dominant use case for `Bundle.link.relation`, and a formal preferred binding improves both developer experience and tooling support for validation. Any tooling concerns raised in 2024 appear to have been resolved, as the 2026 comment confirms feasibility by referencing the `Patient.language` implementation.
