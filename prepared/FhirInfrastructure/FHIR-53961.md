## Description

Brian Postlethwaite requests that the `elementDefinition()` FHIRPath function documentation be clarified in several ways:

1. **Return type ambiguity:** The function presumably returns a collection of `FHIR.ElementDefinition` objects, but the spec does not make this explicit. For a path like `Patient.identifier`, it is unclear whether the returned ElementDefinition is from the Patient resource (the identifier element) or from the Identifier datatype (the root element).
2. **Context of use:** Since the function references `%profile`, the reporter assumes it is only available in invariant definitions — the spec should state this explicitly.
3. **Completeness of returned data:** Is a full ElementDefinition returned, or only a subset? If a subset, which fields are required (e.g., name, type, cardinality)?
4. **Reflection acknowledgment:** The function could be considered a form of reflection that is partially supported in FHIRPath — should this be noted?
5. **Examples needed:** The ticket requests concrete examples be added (e.g., for `Patient.identifier` and `Patient.identifier.system`).

## Analysis

This ticket is part of a cluster of issues from Brian Postlethwaite (filed 2026-01-12) aimed at improving the FHIRPath specification's documentation around introspection and remote data capabilities. The `elementDefinition()` function currently has ambiguous documentation regarding:

- **What it returns** (full vs. partial ElementDefinition, from the element's own type or the parent resource)
- **Where it can be used** (invariant context only, or broader)
- **How it relates to the reflection section** (which is itself under active review for possible reduction)

The Zulip discussion reveals that the broader reflection capability has limited implementation support, with most engines only supporting `type()` metadata. The ITS call discussed stripping the elements detail from the reflection section. This directly impacts the value of clarifying `elementDefinition()` — if the reflection section is significantly reduced, the function's scope and documentation needs change accordingly.

## Proposed Dispositions

### Disposition A: Persuasive — Clarify Documentation as Requested

**Action:** Accept the change request. Add explicit documentation to the `elementDefinition()` function specifying:
- It returns `FHIR.ElementDefinition` objects from the applicable StructureDefinition via `%profile`.
- For `Patient.identifier`, it returns the ElementDefinition for the `identifier` element as defined in the Patient StructureDefinition (not the root of the Identifier datatype).
- For `Patient.identifier.system`, it returns the ElementDefinition for `system` as defined in the Identifier datatype.
- The function is available in invariant definition contexts where `%profile` is in scope.
- Note that only a subset of ElementDefinition fields may be populated (specify the minimum: path, type, min, max).
- Add the requested examples.
- Acknowledge the function as a form of FHIRPath reflection.

**Rationale:** The questions raised are legitimate documentation gaps. Implementers need clarity on return values and scope to implement correctly.

### Disposition B: Not Persuasive with Modification — Defer Pending Reflection Review

**Action:** Acknowledge the documentation gaps but defer detailed clarification until the broader reflection section review (discussed on the ITS call 2026-01-07, see Zulip thread) is resolved. Add a brief STU note indicating that `elementDefinition()` semantics are under active review and may change.

**Rationale:** The Zulip discussion indicates the reflection section may be significantly reduced or restructured. Investing in detailed documentation for `elementDefinition()` now risks wasted effort if the function's scope changes. Better to stabilize the overall approach first.

### Disposition C: Persuasive with Modification — Minimal Clarification Now, Full Update After Reflection Decision

**Action:** Add minimal clarifications now (context of use limited to invariant definitions, return type is `FHIR.ElementDefinition`, add the two requested examples), but mark the full semantics as STU content pending the reflection section review. Coordinate with FHIR-53953 (remote data handling) for consistent documentation of when ElementDefinition data might require remote access.

**Rationale:** Balances immediate implementer needs with the ongoing reflection review. Provides enough clarity for current implementers without over-committing to semantics that may shift.

## Recommendation

**Disposition C: Persuasive with Modification** is recommended. The ticket raises valid documentation gaps that impede implementers today, but the active discussion about potentially reducing the reflection section (Zulip: "Removal of reflection section," ITS call 2026-01-07) means the full scope of `elementDefinition()` may change. A minimal clarification — stating the context of use, return type, and adding examples — addresses the immediate need without conflicting with the ongoing reflection review. This should be coordinated with FHIR-53953 to ensure consistent treatment of profile-related functions that may involve remote data access.
