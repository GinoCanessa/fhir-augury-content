## Summary

Rene Spronk requests improved documentation for the new `resource` search parameter type. The current description in the FHIR specification is minimal and may be insufficient for implementers unfamiliar with this parameter type. The ticket proposes two specific textual changes:

1. **Add a cross-reference** to section 3.2.1.5.16, making it easier for readers to find the detailed definition of the `resource` search parameter type.
2. **Clarify the purpose** by appending "embedded within an overarching resource (e.g. Bundle)" to explain the relationship between `Bundle.entry.resource` and the enclosing Bundle resource.

This is a documentation-only change with no functional or structural impact on the specification.

## Details

### Description

The reporter notes that the current specification text reads:

> "A `resource` type parameter is a special case parameter type that is used to enable chaining through elements that are actual resources (e.g., Bundle.entry.resource)."

The suggested replacement is:

> "A `resource` type parameter is a special case parameter type [see section 3.2.1.5.16] that is used to enable chaining through elements that are actual resources (e.g., Bundle.entry.resource) embedded within an overarching resource (e.g. Bundle)"

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Accept with suggested text)

Accept the ticket and apply the reporter's suggested wording changes:
- Add the cross-reference "[see section 3.2.1.5.16]" to the `resource` type description.
- Append "embedded within an overarching resource (e.g. Bundle)" to clarify the context.

**Rationale:** The existing text is genuinely sparse for a new parameter type that implementers need to understand. The proposed changes are small, purely editorial, and directly improve discoverability and comprehension. The cross-reference to section 3.2.1.5.16 helps readers navigate to the detailed definition quickly. This change should be coordinated with FHIR-50493 (removing stale text about `reference` covering Resource elements) and FHIR-53764 (updating the spd-2 invariant) to ensure consistent documentation of the `resource` parameter type.

### Disposition B: Accept as Persuasive with Modification (Broader documentation improvement)

Accept the intent of the ticket but expand the scope of the documentation improvement beyond the two specific text changes. In addition to adding the cross-reference and clarifying text, also:
- Add one or two concrete usage examples showing how to chain through `Bundle.entry.resource` using the `resource` search parameter type.
- Clarify the distinction between `resource` and `reference` parameter types inline, since implementers may confuse the two.
- Reference the Bundle search parameters (`composition`, `message`) as concrete examples of this type in use.

**Rationale:** Since this is a new and unfamiliar parameter type, minimal text edits alone may not be sufficient. Adding examples and a brief comparison with `reference` type would provide substantially more value to implementers. This approach addresses the root concern (documentation is "rather minimal") more comprehensively.

### Disposition C: Decline as Not Persuasive

Decline the ticket on the grounds that the existing documentation is adequate when read in conjunction with section 3.2.1.5.16, and that adding inline cross-references and clarifying text creates a maintenance burden. Readers can follow existing navigation to find the relevant section.

**Rationale:** The specification already contains the detailed definition in section 3.2.1.5.16. Adding redundant cross-references and explanatory text could lead to inconsistencies if the section numbering changes. However, this position is weak given that the `resource` parameter type is new and the documentation gap is real.

## Recommendation

**Disposition B: Accept as Persuasive with Modification** is recommended.

The reporter correctly identifies that the documentation for the `resource` search parameter type is insufficient. While Disposition A would address the immediate request, Disposition B better serves the community by providing a more thorough treatment of this new concept. The `resource` type is a novel addition to FHIR search that emerged from FHIR-48363, and implementers will benefit from examples and clearer differentiation from the `reference` type. The work should be coordinated with the related tickets (FHIR-53764 for the spd-2 invariant fix and FHIR-50493 for removing stale `reference` text) to deliver a cohesive documentation update for the `resource` search parameter type.
