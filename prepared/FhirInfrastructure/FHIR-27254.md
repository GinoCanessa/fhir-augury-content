## Summary

This change request proposes allowing **inline invariant definitions** in FHIR Shorthand (FSH). Currently, invariants must be defined as separate standalone entities using the `Invariant` keyword, then referenced via `obeys` rules on profiles or elements. The submitter notes that invariants are frequently used in only one location, and allowing them to be defined inline where they are applied would make profiles more concise and readable.

The ticket was originally resolved as "Considered for Future Use" during STU1 ballot with a 14-0-0 vote (Kramer/Rhodes), noting it was "an interesting proposal, although not essential for the 1.0 specification." After STU1 publication, the ticket was re-opened by Lynn Laakso and reverted to Triaged status. The original reporter (bvdh) agreed with the revised resolution to reconsider.

---

## Details

### Current Behavior

In FSH, invariants are defined as separate top-level entities:

```fsh
Invariant: my-invariant-1
Description: "Some constraint description"
Expression: "some.fhirpath.expression"
Severity: #error
```

They are then referenced on profiles or elements:

```fsh
* obeys my-invariant-1
* someElement obeys my-invariant-1
```

If an invariant is only relevant to a single profile element, the user must still define it as a separate block, potentially far from its point of use. Alternatively, caret syntax can be used to define constraints inline, but this is verbose and not ergonomic.

### Proposed Change

Allow invariant definitions to be provided inline at the point of use, eliminating the need for a separate `Invariant` block when the invariant is only applied in one place. This would condense profiles that have single-use invariants.

## Proposed Dispositions

### Disposition A: Persuasive (Accept — Define Inline Invariant Syntax)

**Resolution:** Persuasive

Accept the proposal and define a new inline invariant syntax for FSH. For example, an inline invariant could be expressed directly within an `obeys` rule:

```fsh
* someElement obeys "Description of constraint" : "fhirpath.expression" #error
```

Or via an inline block attached to the obeys rule:

```fsh
* someElement obeys
    Description: "Some constraint"
    Expression: "some.fhirpath.expression"
    Severity: #error
```

**Rationale:** The workgroup previously acknowledged this as "interesting" with a unanimous 14-0-0 vote. Post-STU1, it is appropriate to add syntactic sugar that improves authoring ergonomics. Single-use invariants are common, and requiring a separate named block adds unnecessary verbosity. The exact syntax would need to be designed by the FSH specification editors.

**Applied to:** Shorthand (FHIR) specification — add new grammar rule for inline invariant definition.

### Disposition B: Persuasive with Modification (Accept — Limited to Caret Syntax Improvement)

**Resolution:** Persuasive with Modification

Instead of introducing a fully new inline syntax, improve the existing caret syntax approach to make inline invariant definition more ergonomic. Document a recommended pattern for single-use invariants using caret syntax with clearer examples and possibly a shorthand helper.

**Rationale:** This addresses the underlying concern (verbosity for single-use invariants) without requiring a new grammar construct. It reduces specification complexity while still improving the authoring experience. The caret syntax workaround was already noted as viable in the original STU1 disposition.

**Applied to:** Shorthand (FHIR) specification — enhance caret syntax documentation and examples for inline invariant patterns.

### Disposition C: Not Persuasive (Decline)

**Resolution:** Not Persuasive

Decline the proposal. The current approach of separate `Invariant` blocks with `obeys` references provides clear separation of concerns and avoids syntax ambiguity. Caret syntax already provides a workaround for users who want inline definitions.

**Rationale:** Adding inline invariant syntax increases the complexity of the FSH grammar and tooling (SUSHI compiler). The benefit of slightly more concise profiles does not outweigh the cost of maintaining two parallel ways to define invariants. The existing workaround via caret syntax is sufficient.

---

## Recommendation

**Disposition A: Persuasive** is recommended.

The workgroup already expressed unanimous support (14-0-0) for the concept during STU1, deferring only due to timing. Now that FSH is post-STU1, this is an appropriate enhancement. Inline invariant definitions for single-use constraints are a natural ergonomic improvement that aligns with FSH's design philosophy of concise, readable profiling. The exact syntax should be defined by the FSH editors, but the direction should be approved.
