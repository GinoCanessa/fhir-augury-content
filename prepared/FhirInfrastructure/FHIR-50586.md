## Summary

On the main Datatypes page (`datatypes.html`) in the R6 Ballot 3 build, the constraints (invariants) box is missing for the **SampledData** and **RelativeTime** data types, even though these types have constraints defined. Specifically:

- **SampledData** is missing the `sdd-1` constraint box
- **RelativeTime** is missing the `rlt-1` and `rlt-2` constraint boxes

The constraints still appear correctly on the detailed definitions pages (e.g., `datatypes-definitions.html#SampledData.interval`), indicating the invariants are defined in the source but are not being rendered on the summary datatypes page.

## Details

### Description

The reporter identified that the main Datatypes page at `http://hl7.org/fhir/6.0.0-ballot3/datatypes.html` omits the constraints box for certain data types. Other data types on the same page correctly display their constraints in a summary box, but SampledData and RelativeTime do not. This is a rendering/publishing issue — the constraint definitions exist in the underlying data and appear in the detailed definitions view, but they are not surfaced in the summary constraints box on the main page.

## Proposed Dispositions

### Disposition A: Accept as Technical Correction (Persuasive)

**Resolution:** Persuasive  
**Action:** Fix the FHIR publisher tooling or the source StructureDefinition markup so that the constraints box is rendered for SampledData (showing `sdd-1`) and RelativeTime (showing `rlt-1`, `rlt-2`) on the main `datatypes.html` page, consistent with how other datatypes display their constraints.

**Rationale:** This is a clear rendering omission. The constraints are defined and visible in the detailed definitions pages but are missing from the summary view. The fix is straightforward — either the publisher template needs to be corrected to include constraints for these types, or the source data needs a minor markup adjustment so the publisher picks them up. This is aligned with the "Technical Correction" classification.

### Disposition B: Accept as Persuasive with Modification — Audit All Datatypes

**Resolution:** Persuasive with Modification  
**Action:** Fix the missing constraints for SampledData and RelativeTime, and additionally audit all datatypes on the `datatypes.html` page to verify that every type with defined constraints has a constraints box rendered. Document any additional types found to be missing constraint boxes.

**Rationale:** If two datatypes are missing constraints boxes, others may be missing as well. A broader audit ensures completeness and prevents recurrence. Given that FHIR-50467 (same reporter) also identifies a classification issue with RelativeTime on the datatypes page, there may be a systemic rendering issue affecting newer or restructured types.

### Disposition C: Decline — Not Persuasive

**Resolution:** Not Persuasive  
**Action:** No change. The constraints are available on the detailed definitions pages, so the information is not lost.

**Rationale:** The detailed definitions pages already display these constraints. Users who need to see constraint details can navigate to those pages. However, this disposition is weak because the inconsistency between types reduces usability and could cause implementers to miss important invariants.

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The reported issue is clearly valid — the constraints exist but are not rendered consistently. Accepting as a simple fix (Disposition A) addresses the immediate problem, but the broader audit (Disposition B) is prudent because: (1) the same reporter found another rendering issue with RelativeTime (FHIR-50467), suggesting a pattern; (2) newer datatypes like RelativeTime and recently-modified types like SampledData may share a common rendering gap; and (3) verifying all types have correct constraints boxes is a low-cost, high-value quality check for the specification. Declining (Disposition C) is not advisable since the inconsistency undermines trust in the specification's completeness.
