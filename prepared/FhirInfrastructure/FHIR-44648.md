## Summary

This ticket proposes deprecating the SMART `authorize-post` capability due to evolving browser cross-origin cookie policies. The `authorize-post` mechanism performs a cross-origin form POST to the authorization server, which causes modern browsers to withhold critical cookies (SSO session, "remember me", device tracking) unless the authorization server has set its cookies to `SameSite=None`. The reporter (Dan Cinnamon, Okta) noted that the OpenID working group recommends RFC-9126 (Pushed Authorization Requests / PAR) as a successor to solve the same problems with additional security benefits.

However, subsequent investigation on Zulip by both the reporter and Josh Mandel revealed that the urgency is lower than initially believed: the `authorize-post` flow involves a top-level browser navigation and is therefore **not** classified as a "third-party request" by current browser deprecation efforts. The main concern is that authorization servers must use the least secure cookie setting (`SameSite=None`), but current-generation browsers still support this. Both Dan Cinnamon and Josh Mandel agreed that deprecation need not be rushed and could be handled as part of a broader discussion about aligning SMART launch with current identity standards (including PAR and DPoP).

## Details

### Description

Since SMART launch v2 was originally published, browser vendors have introduced significant cross-origin policy changes to combat cross-site request forgery and reduce marketing tracker cookies. The `authorize-post` capability causes a SMART client to perform a cross-origin form POST to the authorization server. Depending on cookie security flags, browsers will not send critical cookies (SSO session cookies, "remember me" cookies, device tracking cookies), breaking commonly implemented authorization server features.

The impact is most significant for the practitioner community where SSO expectations exist across multiple applications. Cross-origin form POSTs are no longer considered best practice; the OpenID working group recommends RFC-9126 (Pushed Authorization Requests) as the replacement. A companion ticket was planned to propose PAR adoption for the SMART/FHIR community.

## Proposed Dispositions

### Disposition A: Accept — Deprecate `authorize-post` with a migration path to PAR

Mark the `authorize-post` capability as deprecated in the next SMART App Launch release (e.g., STU 2.2 or STU 3). Add informative guidance recommending RFC-9126 Pushed Authorization Requests (PAR) as the replacement mechanism. Include a transition period (e.g., two ballot cycles) during which both capabilities are available, with `authorize-post` carrying a deprecation warning.

**Rationale:** Aligns SMART with current OpenID best practices. Even though the cookie crisis is less urgent than initially feared, requiring `SameSite=None` is the least secure cookie configuration and represents a long-term security posture concern. PAR provides additional security benefits (e.g., request integrity, client authentication at the authorization endpoint).

### Disposition B: Defer — Address as part of a broader SMART security modernization effort

Do not deprecate `authorize-post` in isolation. Instead, create a coordinated work item that addresses PAR adoption, DPoP token binding, and client-held key binding together. Mark this ticket as "Deferred" with a note linking it to the broader modernization effort. Add an informative note to the current SMART specification acknowledging that PAR is the recommended direction for new implementations.

**Rationale:** The Zulip discussion concluded that there is no urgent need to deprecate. Deprecating `authorize-post` without PAR being formally specified in SMART would remove functionality without providing a replacement. A holistic approach ensures the community transitions to a complete modern security model rather than piecemeal changes. This also avoids the pushback seen in the OpenID community when POST deprecation was attempted in isolation.

### Disposition C: Decline — Retain `authorize-post` as-is

Reject the deprecation proposal. Keep `authorize-post` as a supported capability. The cross-origin cookie concerns that motivated this request have been investigated and found to be manageable (authorize-post uses top-level navigation and is not impacted by third-party cookie deprecation). Implementers who prefer PAR can adopt it independently without requiring deprecation of the existing mechanism.

**Rationale:** The reporter and key community members confirmed on Zulip that the feared cookie breakage is not materializing. Cooper Thompson (Epic) confirmed active use and real-world URL-length challenges that authorize-post solves. Deprecation without a formally specified replacement would remove functionality some implementers depend on. The OpenID community also faced pushback on POST deprecation.

## Recommendation

**Disposition B: Defer — Address as part of a broader SMART security modernization effort.**

The Zulip discussion — including the ticket reporter (Dan Cinnamon) and SMART lead (Josh Mandel) — reached consensus that the urgency has dissipated and that this work should be folded into a broader effort covering PAR, DPoP, and client-held keys. Deprecating `authorize-post` in isolation would remove functionality that some implementers rely on (Epic confirmed real usage) without providing a formally specified alternative. Deferring allows the community to design a complete migration path while adding an informative note to guide new implementers toward PAR.
