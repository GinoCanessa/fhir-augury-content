## Description

The ticket requests the addition of an "in-progress" response pattern to SMART Web Messaging activities. Currently, activities such as `ui.done` can only respond with success or failure. There is no mechanism for an application to communicate an intermediate state such as "received, but not yet completed" — for example, when waiting for a user to respond to a 'save' dialog.

Today, each activity must independently define any such interim-response functionality. The submitter proposes that a cross-cutting, specification-level pattern be established so that any activity can optionally indicate an "in-progress" state. Two possible approaches are suggested:

1. **By default** — make an in-progress response always valid for any activity.
2. **Opt-in** — add a parameter on the request indicating whether a pending/in-progress response is allowed.

## Analysis

### Core Issue

SMART Web Messaging's current response model is synchronous and binary: an activity responds with either success or failure. For activities that involve user interaction or asynchronous processing (e.g., a save dialog, a multi-step workflow), there is no standardized way to signal that the request was received and is being processed. This forces each activity to independently invent interim-response semantics, leading to inconsistency and potential interoperability issues.

### Precedent and Context

- **FHIR-29336** (Published) already raised the limited expressiveness of the `success`/`details` response for `ui.done`, and the workgroup answered the question — but the underlying limitation remains relevant here.
- **FHIR-32948** attempted to add "in-progress" to the core FHIR Request Status ValueSet, which was resolved as "Not Persuasive with Modification." The resolution suggests the community is cautious about adding intermediate states broadly but open to targeted modifications.
- **FHIR-29365** (Persuasive with Modification) shows the workgroup is willing to refine `ui.done` semantics.

### Impact

Adding an in-progress response pattern would affect:
- The SMART Web Messaging response message schema (new status field or response type).
- All existing activity definitions (which would need to specify whether they support in-progress responses).
- Client applications (which would need to handle a new response state).
- The capability/discovery mechanism (if opt-in, hosts must advertise support).

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Recommendation: Add an opt-in in-progress response pattern**

Accept the need for an in-progress response mechanism, but implement it as an opt-in pattern rather than a universal default. Specifically:
- Define a new `status` field in the SMART Web Messaging response message schema with values such as `completed`, `in-progress`, and `error` (replacing or augmenting the current boolean `success` field).
- Activities opt into supporting in-progress responses by declaring it in their definition.
- Requesters may include a parameter (e.g., `acceptInProgress: true`) to indicate they can handle intermediate responses.
- If a requester does not indicate support, the responder must wait until completion before replying (preserving backward compatibility).

**Rationale:** This addresses the real use case (e.g., save dialogs, multi-step processes) while avoiding breaking changes. The opt-in approach aligns with the workgroup's cautious stance seen in FHIR-32948 and avoids imposing complexity on simple activities.

### Disposition B: Persuasive
**Accept as-is and define a universal in-progress pattern**

Add an in-progress response as a standard option available to all SMART Web Messaging activities by default. Any activity may return an in-progress response at any time, and clients must be prepared to handle it. This would involve:
- Adding a `status` field to the response message schema with `in-progress` as a recognized value.
- Defining expected client behavior when receiving an in-progress response (e.g., continue waiting, display progress indication).
- No opt-in required — all activities support it inherently.

**Rationale:** The simplest approach from a specification perspective. Avoids the complexity of per-activity opt-in declarations. However, it imposes a new requirement on all clients to handle in-progress responses, which may be disruptive.

### Disposition C: Not Persuasive
**Decline — existing mechanisms are sufficient**

The current specification already allows activities to respond with multiple messages and to define activity-specific response semantics. Individual activities (like `ui.done`) can be modified on a case-by-case basis to include intermediate states if needed. A cross-cutting in-progress pattern is premature given the specification's current maturity level.

**Rationale:** The SMART Web Messaging spec is still relatively early-stage, and adding cross-cutting response patterns may over-engineer the protocol before real-world implementations have fully demonstrated the need. Per-activity solutions (as the ticket itself acknowledges exist) may be more appropriate until usage patterns stabilize.

---

## Recommendation

**Disposition A: Persuasive with Modification** — The use case is legitimate and well-described. Real-world SMART App scenarios (save dialogs, multi-step clinical workflows) benefit from a standardized in-progress signal. However, making it universal (Disposition B) risks backward-compatibility issues and imposes requirements on simple activities that don't need it. An opt-in pattern balances the need with pragmatism and aligns with the workgroup's established approach to incremental, targeted modifications (as seen in resolutions for FHIR-29365 and FHIR-32948).
