## Summary

This ticket requests an update to the "Signing Bundles" section of the FHIR Core specification (signatures.html) to improve clarity and correctness. The reporter identifies two specific technical issues with the current text: (1) the term "enveloping signatures" is incorrectly used—it has specific meaning only in XMLDSig and does not apply to JWS or CMS signatures; and (2) whether a signature is attached or detached is irrelevant in the context described, since either approach works regardless of whether the Provenance resource is in the Bundle. The ticket includes complete proposed replacement text for the Signing Bundles section, sourced from a GitHub branch at [Healthedata1/fhir](https://github.com/Healthedata1/fhir/blob/master/source/signatures.html).

## Details

### Current Text (Problematic)

The current specification text states:

> Note that it is not necessary to sign Bundles with embedded signatures like this - systems and/or ecosystems may choose to use enveloping signatures (JSON, XML), where the signed data is included within the signature itself, or make the Provenance resources available separately.

### Issues Identified

1. **Incorrect terminology**: "Enveloping signatures" has a specific meaning in digital signature standards (XMLDSig), where the signed content is contained within the signature element. This term does not apply to JWS or CMS, which have fundamentally different structures.
2. **Irrelevant distinction**: Whether the signature is attached or detached is irrelevant in this context—either approach can be used regardless of whether the Provenance resource is in the Bundle.

### Proposed Replacement Text

The proposed replacement provides a restructured "Signing Bundles" section with clear numbered rules:

1. Provenance is used to carry a signature.
2. `Provenance.target.reference` = `"Bundle/[id]"` where id is the Bundle's id (i.e., the Bundle itself is being signed).
3. Other Provenance resources in the Bundle that don't target the Bundle id are not embedded signatures and not subject to these rules.
4. The Provenance resource SHALL have a `Provenance.signature` element for the Bundle signature.
5. The Provenance resource is either added after signature creation, or removed/re-added after signature creation.
6. `Provenance.target.reference` can reference other resources in addition to Bundle contents.
   - Special target value `"#"` signs itself minus `Provenance.signature.data`.
   - Target = `fullUrl` of another Provenance entry signs that entry (including `Provenance.signature.data`).
   - Targets not in the Bundle are allowed but resolution is undefined.

The proposal also includes a note deprecating `Bundle.signature` due to unresolved issues, recommending the Provenance-based method instead.

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

Accept the proposed text as submitted. The replacement Signing Bundles section corrects the incorrect use of "enveloping signatures" terminology, provides a clear numbered list of rules for embedding signatures via Provenance, and includes the deprecation notice for `Bundle.signature`. This aligns with FHIR-51112's resolution (which acknowledged the "enveloping" terminology was confusing), the direction discussed on Zulip (move to Provenance, deprecate `Bundle.signature`), and the parallel work in FHIR-52813 to clarify countersigning rules.

**Rationale**: The proposed text directly addresses known terminology errors that have caused confusion across multiple tickets and workgroups. The Provenance-based approach has active community support and is already being implemented in the IG publisher.

### Disposition B: Persuasive with Modification

Accept the direction of the change but request modifications:
1. **Coordinate with FHIR-53835**: Since the same reporter has a companion ticket to rewrite the entire Signatures introduction, these two changes should be reviewed together to ensure consistent structure and terminology.
2. **Clarify the deprecation language**: The current note says `Bundle.signature` has "several unresolved issues"—these should be enumerated or cross-referenced so implementers understand the rationale.
3. **Normalize list formatting**: The proposed text has inconsistent list numbering (each item is a separate `<ol>` element); this should be cleaned up into a single properly nested ordered list.
4. **Add guidance on migration**: Include a brief note on how systems currently using `Bundle.signature` should transition to the Provenance-based approach.

**Rationale**: The technical direction is correct and well-supported, but coordinating with the companion ticket and improving the deprecation guidance will produce a more cohesive and implementer-friendly result.

### Disposition C: Decline (Not Persuasive)

Decline the change on the grounds that the Signing Bundles section is undergoing a broader rewrite (FHIR-53835) and accepting piecemeal edits risks inconsistency. The terminology issue ("enveloping") was already addressed in FHIR-51112 for CDex. The deprecation of `Bundle.signature` is a significant architectural decision that should go through a formal community process rather than being introduced as part of a text clarity edit.

**Rationale**: While the concerns are valid, accepting individual section rewrites before the broader page restructuring is complete may create duplicate work and inconsistencies.

## Recommendation

**Disposition B: Persuasive with Modification**

The technical substance of this ticket is sound—the "enveloping signatures" terminology is demonstrably incorrect (as confirmed by FHIR-51112's resolution), and the proposed Provenance-based rules are clearer and more precise than the current text. However, the companion ticket FHIR-53835 from the same reporter proposes rewriting the entire Signatures introduction, and these changes should be coordinated to avoid fragmentation. The deprecation note for `Bundle.signature` is an important architectural signal that deserves slightly more explanation. Accepting with modifications to coordinate with FHIR-53835, clarify the deprecation rationale, and clean up formatting will produce the best outcome for implementers.
