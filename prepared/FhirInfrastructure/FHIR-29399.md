## Description

The ticket requests that the FHIR Shorthand (FSH) documentation be updated to explain how to constrain a datatype choice element to `Reference(Any)` in a profile. The reporter (Mark Kramer) notes that the syntax `* value[x] only Reference` achieves this, but the approach is not documented. The ticket links to a Zulip discussion where the question originally arose.

- **Chris Moesel** suggested using `* value[x] only Reference` (without parenthetical target types), which constrains the choice to the Reference type without affecting what the reference can point to.

## Proposed Dispositions

### Disposition A: Persuasive — Add documentation for Reference(Any) constraint syntax

**Action:** Update the FHIR Shorthand specification/documentation to include an explicit example and explanation of how to constrain a datatype choice element to `Reference(Any)`. Specifically, document that:
- `* value[x] only Reference` constrains a choice to only allow Reference without restricting the target resource types.
- This applies to both profiling existing elements (e.g., `Claim.supportingInfo.value[x]`) and extension definitions (e.g., `Extension.value[x]`).
- A brief note should explain why no parenthetical type list is needed — omitting the target types preserves the existing "any" reference scope.

**Rationale:** The Zulip discussion demonstrates a real user pain point where an experienced FSH author could not find this information. The syntax is non-obvious (using bare `Reference` instead of `Reference(Any)`), and adding a short example to the docs is low-effort and high-value.

### Disposition B: Persuasive with Modification — Add documentation and clarify Reference vs Reference(Any) distinction

**Action:** In addition to documenting the `* value[x] only Reference` syntax, add a broader clarification section to the FSH documentation that distinguishes between:
- `Reference` (bare) — constrains the choice type to Reference without restricting targets (equivalent to Reference(Any)).
- `Reference(Patient)` — constrains to Reference targeting a specific resource type.
- `Reference(Patient or Practitioner)` — constrains to Reference targeting multiple specific types.

Include a note that `Reference(Any)` is **not** valid FSH syntax and that the bare `Reference` form should be used instead.

**Rationale:** This provides a more complete treatment of Reference constraints in FSH, preventing future confusion. Users familiar with the StructureDefinition `Reference(Any)` notation may naturally try to use it in FSH, so an explicit note about this difference is helpful.

### Disposition C: Not Persuasive — Decline as already inferable from existing documentation

**Action:** Close without change, reasoning that the FSH specification already defines the `only` keyword for constraining types, and `Reference` is a listed type — so the behavior is inferable from existing rules.

**Rationale:** The existing FSH grammar rules technically support this use case. However, this disposition is weak because the Zulip discussion shows that even experienced users could not figure it out from the existing docs.

## Recommendation

**Disposition B: Persuasive with Modification** — This addresses the immediate documentation gap while also providing a more comprehensive treatment of Reference constraint variants in FSH. The additional clarity about bare `Reference` vs `Reference(TargetType)` vs the invalid `Reference(Any)` syntax will prevent recurring confusion, and the effort to add this is modest. The Zulip discussion demonstrates clear user need, and the fact that the ticket has been open since 2020 suggests this documentation gap persists.
