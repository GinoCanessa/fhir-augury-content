## Summary

The IPS specification includes informal comments on three status elements stating "In the scope of the IPS the entered-in-error concept is not allowed," but the profiles do not enforce this constraint — the bound value sets still include `entered-in-error` and no invariants exist. The reporter asks HL7 to clarify the intent and align the specification accordingly.

The ticket was discussed on the IPS Project call on 2025-11-25 and resolved as **Persuasive**: the intent is confirmed — `entered-in-error` should not be permitted for these elements. The agreed resolution is to create new IPS-specific ValueSets that exclude `entered-in-error` for each affected element, bind those ValueSets to the elements, and update the short descriptions.

### Affected Elements

- `AllergyIntolerance.verificationStatus`
- `MedicationRequest.status`
- `MedicationStatement.status`

## Details

### Description

The IPS specification contains the comment "In the scope of the IPS the entered-in-error concept is not allowed" on:

- `AllergyIntolerance.status`
- `MedicationRequest.status`
- `MedicationStatement.status`

This reads like a normative constraint but is not enforced in the profiles — the bound value sets still include `entered-in-error`, and there are no invariants. The reporter requests that the intent be clarified and the spec updated accordingly.

### Resolution Description

The intent is confirmed: `entered-in-error` should not be allowed for `AllergyIntolerance.verificationStatus`, `MedicationStatement.status`, and `MedicationRequest.status`. The agreed actions are:

1. **Create new IPS ValueSets** — subsets of the existing FHIR value sets that exclude `entered-in-error`, one for each affected element.
2. **Bind the new ValueSets** to the respective elements in the IPS profiles.
3. **Adjust short descriptions** to reflect the change — exact wording to be supplied before vote.

## Proposed Dispositions

### Disposition A: Accept as Persuasive (Recommended)

**Action:** Accept the resolution as stated — create three new IPS-specific ValueSets (subsets of the existing FHIR ValueSets excluding `entered-in-error`) for `AllergyIntolerance.verificationStatus`, `MedicationRequest.status`, and `MedicationStatement.status`, bind these new ValueSets to the respective profile elements, and update the short descriptions.

**Rationale:** The current specification text already communicates the intent to prohibit `entered-in-error` in IPS context, but fails to enforce it through the profile mechanics. This creates ambiguity for implementers. Creating dedicated IPS ValueSets is the cleanest approach: it makes the constraint machine-enforceable via validation, aligns the normative intent with the computable profile, and follows established FHIR profiling patterns for constraining value sets. The resolution was discussed and agreed on the IPS Project call (2025-11-25) with input from both international and AU stakeholders.

### Disposition B: Accept with Modification — Use Invariants Instead of ValueSets

**Action:** Instead of creating new ValueSets, add FHIRPath invariants on the three affected elements that reject `entered-in-error` as a valid value.

**Rationale:** Invariants can enforce the constraint without introducing additional ValueSet resources, reducing the maintenance burden. However, this approach is less discoverable for tooling that relies on terminology bindings for validation and code generation, and it doesn't simplify the value domain visible in the binding. This is a viable alternative but less aligned with typical profiling conventions.

### Disposition C: Decline — Not Persuasive

**Action:** Reject the change request. Keep the existing informal comment and do not add computable constraints.

**Rationale:** The text comment already signals the intent. Implementers constructing an IPS document are expected to understand that `entered-in-error` records should be excluded at the business logic level (such records should never appear in a summary document). Adding profile-level constraints may be overly restrictive for edge cases and increases ValueSet maintenance overhead. However, this disposition leaves the specification inconsistent — the text says one thing while the profile allows another — which has already caused confusion as evidenced by this ticket.

## Recommendation

**Disposition A: Accept as Persuasive.** The existing specification text clearly intends to prohibit `entered-in-error` but the profiles fail to enforce it, creating implementer confusion. Creating new IPS ValueSets that exclude `entered-in-error` and binding them to the three affected elements is the standard FHIR profiling approach, makes the constraint machine-enforceable, and was already agreed upon during the IPS call on 2025-11-25. The resolution is well-scoped and actionable, with specific steps (new ValueSets, updated bindings, revised short descriptions) ready for implementation.
