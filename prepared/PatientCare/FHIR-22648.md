## Issue Description

The documentation for `CommunicationRequest.payload` says it contains "Text, attachment(s), or resource(s) to be communicated to the **recipient**." However, the only existing example uses `payload` with a CodeableConcept whose text says "Please provide the accident report and any associated pictures to support your Claim# DEF5647." — which is clearly instructions directed at the **sender** (the information provider), not content intended for the eventual recipient.

Josh Mandel (reporter) notes this is not just an example issue — there is a fundamental conceptual ambiguity in whether `CommunicationRequest.payload` represents:
- The actual content to be communicated to the ultimate recipient, or
- Instructions/description for the sender (information provider) about what to communicate.

This ambiguity suggests `payload` may need to be refactored rather than serving double-duty.

## Workgroup Discussion Record

**Patient Care FHIR Conference Call — September 5, 2019** (from comment by michelle.m.miller):

The workgroup identified three options:

1. **Option 1:** `CommunicationRequest.payload.content` is defined **differently** from `Communication.payload.content`. The CommunicationRequest payload is a communication from the requester to the informationProvider, while the Communication payload is from the sender to the recipient. This would require cleaning up the resource definition and potentially using different element names.

2. **Option 2:** `CommunicationRequest.payload.content` is defined **similarly** to `Communication.payload.content`. Under this interpretation, the example would be wrong.

3. **Option 3:** `CommunicationRequest.payload.content` is **serving two masters** (both the informationProvider and the recipient).

**Lloyd McKenzie's clarifying comments:**
- CommunicationRequest is created by A for B to communicate with C.
- `CommunicationRequest.payload` is better thought of as "payload description" — describing what should be communicated once communication happens.
- It is less common to send the actual content itself; the requester may not have authority to communicate directly, or it may need to be communicated in person.
- The `Communication` resource could say "I communicated pamphlet on smoking cessation" without actually transporting the specific pamphlet. The resource documents that communication occurred.
- It is not wrong to use Communication as the delivery mechanism, but that is not its primary or sole purpose.

Discussion was scheduled to resume at the September 2019 WGM (Tuesday Q3). No subsequent resolution is recorded.

## Proposed Dispositions

### Disposition A: Persuasive — Clarify payload as "payload description" and fix the example
Accept the ticket. Update the definition of `CommunicationRequest.payload` to clarify that it describes what should be communicated (a "payload description") rather than being the literal content for the recipient. Specifically:
- Change the element short description from "Text, attachment(s), or resource(s) to be communicated to the recipient" to something like "Description or content of what should be communicated."
- Update the definition text to explain that payload may contain either the actual content to send or a description/categorization of the type of content to send.
- Fix the existing example to align with the clarified definition, or add a note explaining that the example illustrates a request-to-provider use case where the payload describes what is being requested.

**Rationale:** This aligns with Lloyd McKenzie's interpretation (payload as "payload description") and is the least disruptive change. It preserves backward compatibility while resolving the documented ambiguity.

### Disposition B: Persuasive with Modification — Add a separate element for request instructions
Accept the ticket with modification. In addition to clarifying the definition (as in Disposition A), add a new element (e.g., `CommunicationRequest.instruction`) to carry instructions directed at the sender/information provider, and restrict `CommunicationRequest.payload` to describe only the content intended for the eventual recipient.

**Rationale:** This fully resolves the "serving two masters" problem (Option 3 from the WG discussion) by separating the two distinct concepts. However, it introduces a breaking structural change and increases resource complexity.

### Disposition C: Not Persuasive — Current design is adequate
Reject the ticket. The current design is intentionally flexible: `CommunicationRequest.payload` can represent either actual content or a description of requested content, and the existing definitions and examples, while imperfect, are adequate. Implementers can determine intent from context.

**Rationale:** The ambiguity is inherent in the flexible design of CommunicationRequest. Adding more structure or redefining elements risks breaking existing implementations. The resource has been in use since STU3 and implementers have worked with the current model.

## Recommendation

**Disposition A: Persuasive — Clarify payload as "payload description" and fix the example.**

This is the most practical resolution. The core issue is a documentation/definition gap, not a structural deficiency. The WG discussion (Sept 2019) converged on Lloyd McKenzie's framing that `payload` is best understood as a "payload description," but this understanding was never codified back into the element definitions or the example. Disposition A addresses the root cause (ambiguous documentation) without the disruption of adding new elements (Disposition B) and without dismissing a legitimate source of implementer confusion (Disposition C). The related ticket FHIR-22791, which addressed similar sender/recipient confusion in CDex, was resolved as Persuasive with Modification — establishing precedent that this class of definitional ambiguity warrants correction.
