## Summary

This ticket requests changing the `bodySite` element (CodeableConcept) to a `bodyStructure` element of type `CodeableReference(BodyStructure)` — following the pattern already applied in Observation — and deprecating the existing `bodySite` element. The ticket does not specify which resource(s) should be changed, but the discussion context (comments, related tickets, and WGM minutes) indicates this applies broadly to Patient Care resources such as **Condition** and **ConditionDefinition**, and potentially others like **Procedure**.

This is part of an ongoing cross-workgroup effort (Patient Care + Orders & Observations) to align body site representation across FHIR R6 resources. The topic has been actively discussed at the January 2026 WGM and in Patient Care calls, with significant debate about cardinality (0..1 vs 0..*), data type (CodeableReference vs separate elements), and naming conventions.

## Proposed Dispositions

### Disposition A: Accept — Adopt CodeableReference pattern with two-element approach (Option 3)

Apply the WGM-preferred pattern (Option 3 from FHIR-50481 discussion) to resources under Patient Care scope. For Condition and ConditionDefinition, replace the existing `bodySite` element with:
- `bodySite : CodeableConcept 0..*` (simple coded body sites, retained for backward compatibility)
- `bodyStructure : Reference(BodyStructure) 0..1` (complex anatomical detail via BodyStructure resource)
- Add an invariant preventing simultaneous use of both elements.

Deprecate the old `bodySite` element name where it conflicts. This aligns with the cross-WG consensus from the Jan 2026 WGM and follows the precedent set by Procedure and ServiceRequest. Coordinate with OO for their parallel resource changes.

### Disposition B: Accept with Modification — Use single CodeableReference 0..1

Change the body site element to a single `bodyStructure : CodeableReference(BodyStructure) 0..1` as done in Observation and DocumentReference. This is the simplest approach and matches the reporter's original request. Simple coded body sites would be conveyed via the CodeableConcept portion of CodeableReference, while complex cases use the Reference portion pointing to a BodyStructure resource.

However, this limits cardinality to `0..1`, which means multiple simple coded body sites (e.g., multi-site abdominal pain) would require creating a BodyStructure resource with multiple `includedStructure` entries. This trade-off was debated at WGM — Rob Hausam supports this approach, while Emma and Stephen preferred `0..*`.

### Disposition C: Decline — Defer pending cross-WG alignment

Decline this ticket as premature. The Jan 2026 WGM produced an action item (assigned to JD) to capture a coordinated ticket for changes aligned with Option 3 across both OO and PC resources. That work is still in progress. Making piecemeal changes to individual resources before the cross-WG pattern is finalized risks inconsistency and potential rework. The ticket also lacks specificity about which resources to change. Recommend the reporter wait for the coordinated proposal and refile against specific resources once the pattern is agreed.

## Recommendation

**Disposition C: Decline — Defer pending cross-WG alignment.**

The strongest rationale for deferral is that the Jan 2026 WGM (PC/OO joint session) already produced a preferred direction (Option 3: two elements with invariant) and assigned an action item to JD to create a coordinated ticket. Accepting FHIR-55388 independently risks making changes that conflict with the forthcoming coordinated proposal. Additionally, the ticket as filed is vague — it does not specify which resources to change, uses imprecise language ("bodyStructrure," "depricated"), and the requested pattern (`CodeableReference` as in Observation) is only one of three options actively debated. The work group should wait for JD's coordinated ticket, which will provide a well-scoped, cross-WG-aligned proposal for body site representation changes across all affected resources.
