## Description

The reporter identifies that the `ObservationVisualAcuity.method` element in the Ophthalmology Retinal (Eyes on FHIR) IG is bound to the [Visual acuity chart valueset](http://build.fhir.org/ig/HL7/fhir-eyecare-ig/ValueSet-observation-visual-acuity-methods.html), which contains **devices** (types of visual acuity charts) rather than **methods** of observation. The request is to replace this binding with a value set that actually contains methods (i.e., how the visual acuity observation was performed), not the devices used.

## Analysis

This ticket raises a valid terminological concern: the `Observation.method` element is semantically intended to capture *how* an observation was performed (the technique or process), yet in the Ophthalmology Retinal IG it is bound to a value set containing types of visual acuity *charts* (devices used during the test). This conflates the method with the device, which are distinct FHIR concepts (`Observation.method` vs. `Observation.device`).

The reporter, Richard Townley-O'Neill, has filed a cluster of related tickets (FHIR-36111, FHIR-36130) against the same IG, all identifying value set and binding issues in the Visual Acuity profile. These collectively suggest the VA profile's terminology bindings need a systematic review.

The ongoing Zulip discussion in the Eye Care stream shows active work on a revised VA Observation Profile where the `method` element is being used more appropriately — to capture the general testing method — suggesting that the community is already moving in the direction this ticket proposes.

Multiple other IGs (SDOH, PACIO, MDI) have had their own Observation.method binding issues and resolutions, confirming this is a recurring pattern in FHIR profiling.

## Proposed Dispositions

### Disposition A: Persuasive
**Action:** Accept the change request as-is. Replace the current `Observation.method` binding on the ObservationVisualAcuity profile with a value set containing actual methods of visual acuity testing (e.g., chart-based testing, counting fingers, hand motion detection, light perception) rather than the current value set of visual acuity chart types (devices).

**Rationale:** The FHIR Observation resource defines `method` as "Indicates the mechanism used to perform the observation" — this clearly refers to the technique, not the device. Binding it to a value set of devices is semantically incorrect and may cause interoperability issues. The fix aligns the profile with FHIR's intended semantics.

### Disposition B: Persuasive with Modification
**Action:** Accept the change request, but additionally recommend that the current device-oriented value set be re-bound to an appropriate element (e.g., `Observation.device` or a component) to preserve the information about which chart type was used, while creating a new methods-oriented value set for `Observation.method`.

**Rationale:** Simply removing the chart-type binding from `method` would lose useful information about the testing device. A more complete fix would both correct the `method` binding AND relocate the device information to the semantically correct element. This aligns with the ongoing community work in the Eye Care stream where Lars Fuhrmann's revised profile uses components and extensions for device/lens information and reserves `method` for the general testing method.

### Disposition C: Not Persuasive
**Action:** Reject the change request. The current binding, while not perfectly aligned with the general FHIR semantics of `method`, is understood within the ophthalmology domain and changing it could break existing implementations.

**Rationale:** The Ophthalmology Retinal IG has a specific domain context where the "method" of VA testing is closely tied to the chart/device used. Changing the binding could disrupt implementations that rely on the current value set. The IG is still in early development and may be revised through the ongoing community work without needing a formal tracker disposition.

---

## Recommendation

**Disposition B: Persuasive with Modification** is recommended. The ticket correctly identifies that `Observation.method` is bound to devices rather than methods, which is a genuine semantic mismatch with FHIR's definition of the element. However, simply swapping the value set would lose the chart-type information. The more robust fix is to create a proper methods value set for `Observation.method` and relocate the chart-type concepts to the semantically appropriate element (such as `Observation.device` or a component). This approach is well-supported by the active community work on the revised Visual Acuity profile in the Eye Care Zulip stream, where the method element is already being used for general testing method while device information is handled separately. The cluster of related tickets from the same reporter (FHIR-36111, FHIR-36130) further suggests a systematic review of the VA profile's bindings would be beneficial.
