## Summary

This ticket flags a QA 5a violation: bidirectional resource references exist between the **Condition** and **ClinicalImpression** resources. Specifically:

- **Condition.stage.assessment** → references ClinicalImpression
- **ClinicalImpression.problem** → references Condition
- **ClinicalImpression.finding.item[x]** → references Condition

Per the DSTU QA Guidelines (rule 5a), a given relationship should only be represented in one resource, not both, and should generally live on the resource that is created second. The ticket asks Patient Care to resolve which direction the reference should go and remove the other.

The ticket was originally filed in September 2016. It was **deferred** in January 2017 (vote: 7-0-1) because ClinicalImpression was not mature enough (FMM=0) to make a definitive decision. A note was added to the spec acknowledging the known circular reference issue. In July 2018, the workgroup voted **Not Persuasive** (8-0-0), but the ticket was never formally resolved and remains in **Triaged** status.

Lloyd McKenzie noted in comments that the situation is nuanced — it is `Condition.stage` (not Condition as a whole) that points to ClinicalImpression, and both the Condition and its stage could plausibly be created as part of a ClinicalImpression workflow.

---

## Details

### The QA 5a Rule

The DSTU QA Guidelines state: *"Elements with a type of Resource Reference should be reviewed such that a given relationship shall only be represented in one resource, not both (and should generally be present on the resource that is normally created second)."*

### Affected Elements

| Element | Target | Direction |
|---------|--------|-----------|
| `Condition.stage.assessment` | ClinicalImpression | Condition → ClinicalImpression |
| `ClinicalImpression.problem` | Condition | ClinicalImpression → Condition |
| `ClinicalImpression.finding.item[x]` | Condition | ClinicalImpression → Condition |

### Vote History

- **2017-01-31** — Deferred (Rob/Elaine: 7-0-1). ClinicalImpression not mature enough; requested MnM exemption on QA checklist.
- **2017-02-01** — Note applied to spec acknowledging circular reference and low maturity of ClinicalImpression.
- **2018-07-05** — Voted Not Persuasive (George/Emma: 8-0-0). Discussion about whether to defer again due to ClinicalImpression maturity.

### Resolution Description (from prior deferral)

> *"ClinicalImpression is not mature enough to resolve this issue. Ask MnM for an exemption on the QA checklist (re: Condition having a circular reference with ClinicalImpression). Add note: 'A known issue exists with circular references between Condition and ClinicalImpression, which is due to the low maturity level of ClinicalImpression. The Patient Care work group intends to address this issue when ClinicalImpression is considered substantially complete and ready for implementation.'"*

## Proposed Dispositions

### Disposition A: Not Persuasive with Modification (Accept the Bidirectional References)

**Rationale:** The bidirectional references between Condition and ClinicalImpression serve distinct semantic purposes. `Condition.stage.assessment` links a specific staging to the clinical assessment that produced it, while `ClinicalImpression.problem` and `ClinicalImpression.finding.item[x]` link the assessment to the conditions it evaluates. These are not redundant — they represent different clinical workflows (staging a condition vs. documenting a clinical impression). The sibling tickets FHIR-10636 and FHIR-10634 were both resolved as "Not Persuasive with Modification," accepting similar bidirectional references with justification.

**Action:** Close as Not Persuasive with Modification. Add or update documentation on both resources explaining the rationale for bidirectional references: `Condition.stage.assessment` captures which assessment determined the stage, while `ClinicalImpression.problem`/`finding` captures the scope of the clinical assessment. Remove any existing spec notes about this being a "known issue" if the references are now accepted as intentional.

### Disposition B: Persuasive with Modification (Remove Condition.stage.assessment → ClinicalImpression)

**Rationale:** Per QA 5a, the reference should live on the resource created second. In typical clinical workflows, a ClinicalImpression documents an assessment of existing conditions — meaning ClinicalImpression is created after Condition. Therefore, the references should live on ClinicalImpression (via `.problem` and `.finding.item[x]`), and the back-reference from `Condition.stage.assessment` to ClinicalImpression should be removed or replaced with an extension.

**Action:** Remove ClinicalImpression as a target type from `Condition.stage.assessment` (or deprecate it). Retain `ClinicalImpression.problem` and `ClinicalImpression.finding.item[x]` as the canonical direction. Update search parameters and documentation accordingly.

### Disposition C: Decline — Continue to Defer

**Rationale:** ClinicalImpression remains at low maturity. Making structural changes to either resource's reference topology before ClinicalImpression is stable risks breaking changes to a resource that may itself be substantially revised. The existing spec note adequately communicates the known issue to implementers.

**Action:** Keep the ticket in Triaged/Deferred status. Revisit when ClinicalImpression reaches FMM 2 or higher. Maintain the existing spec note about the circular reference.

---

## Recommendation

**Disposition A: Not Persuasive with Modification** is recommended.

This ticket has been open since 2016 and was already voted Not Persuasive (8-0-0) in July 2018, indicating the workgroup does not consider this a blocking issue. The sibling QA 5a tickets (FHIR-10634, FHIR-10636) were both resolved by accepting the bidirectional references with justification. The Condition/ClinicalImpression case is analogous: the two directions serve semantically distinct purposes (staging provenance vs. assessment scope). Rather than continuing to defer indefinitely or making potentially disruptive structural changes to low-maturity resources, the workgroup should formally accept the bidirectional references, document the rationale, and close the ticket.
