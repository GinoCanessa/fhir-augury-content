## Summary

This ticket requests the creation of an Observation profile for capturing strength-based exercise measurements (e.g., weight lifting) in the US Physical Activity IG. The submitter (Lloyd McKenzie) notes that strength-based exercises require capturing exercise type (via `Observation.code`), body site (via `Observation.bodySite`), weight/resistance, and number of reps — and that weight and reps must be recorded together in a single Observation (likely as components) to be clinically meaningful. The ticket asks for guidance on how to structure components vs. value, and which standard codes to use.

The proposed disposition is **Not Persuasive with Modification**: rather than defining the profile now, the IG will add an open issue acknowledging the gap and solicit implementer feedback for future iterations.

## Details

### Description

When representing strength-based exercises, there is typically a need to capture:

- **Exercise type** — e.g., lateral press (`Observation.code`)
- **Body part** — e.g., left leg (`Observation.bodySite`)
- **Weight/resistance** — how much load
- **Number of reps** — repetition count

Weight and reps need to be captured as part of a single Observation because knowing one without the other is not clinically useful. The ticket asks for a defined structure specifying how to use Observation components for this, which element should be `value[x]` vs. `component`, and which standard codes to use. The submitter originally suggested seeking Orders & Observations (OO) feedback.

### Resolution Description

Several concerns were raised about defining this profile now:

1. Many providers are not yet ready to handle this sort of data (formal evidence cited).
2. There is limited evidence around the practical use of detailed strength training data.
3. The IG should remain "simple" to encourage broader adoption first; complexity can be added once demand exists.
4. There are no standardized, reliable assessment instruments for gathering this data.
5. Concern about increasing software development costs.

**Actions decided:**

- Add an **open issue** in the IG noting that this release does not include specific guidance for detailed strength-based training, along with reasons why, and actively solicit community feedback.
- Add an issue marked **"considered for future use"** for revisiting when implementer feedback can be evaluated.
- Open to **testing at connectathons**.

## Proposed Dispositions

### Disposition A: Accept Current Resolution (Not Persuasive with Modification)

**Action:** Affirm the existing "Not Persuasive with Modification" disposition.

The IG will:
1. Add an open issue documenting that detailed strength-based training observation guidance is out of scope for this release, along with the rationale (lack of standardized instruments, limited adoption, cost concerns).
2. Mark the topic as "considered for future use" for re-evaluation in a subsequent ballot cycle.
3. Actively solicit implementer feedback and welcome connectathon testing.

**Rationale:** This preserves the IG's simplicity during its adoption phase while explicitly leaving the door open. The concerns about readiness, cost, and lack of standardized assessment tools are well-founded and were discussed on the April 2024 PA call. The September 2025 OO WGM comment confirms that PC owns this space and can revisit when appropriate.

### Disposition B: Accept with Informative Guidance

**Action:** In addition to the open issue described in Disposition A, add a non-normative appendix or informative section to the IG that provides a *recommended* (but not required) pattern for representing strength-based exercise data using `Observation.component`.

Specifically:
- Recommend two components: one for weight/resistance (with a LOINC code if available, or an IG-defined code) and one for rep count.
- Use `Observation.code` for exercise type and `Observation.bodySite` for targeted body part.
- Mark the section as "informative / draft" and subject to change based on implementer feedback.

**Rationale:** This provides early guidance to implementers who want to experiment without imposing normative requirements. It addresses the core design question (value vs. component) without committing the IG to a final answer, and it gives connectathon participants a concrete structure to test against.

### Disposition C: Decline (Not Persuasive)

**Action:** Close as Not Persuasive with no further action in the IG.

**Rationale:** Strength-based exercise tracking is a niche use case with no demonstrated implementer demand. The Physical Activity IG should remain focused on the Exercise Vital Sign and basic activity measures. Defining component structures and codes for strength training adds complexity disproportionate to current community need. Implementers who need this data can define their own profiles or submit a new ticket when there is evidence of demand.

## Recommendation

**Disposition A: Accept Current Resolution (Not Persuasive with Modification)**

The existing disposition strikes the right balance. It acknowledges the valid use case, documents the gap transparently in the IG, and commits to revisiting when the ecosystem is more ready. The concerns raised (no standardized instruments, limited provider readiness, cost) remain valid. The September 2025 OO WGM comment further clarifies ownership — PC is the right group to address this when the time comes. Adding an open issue and a "considered for future use" marker provides a clear path forward without prematurely committing to a design that may need to evolve based on real-world implementer feedback and connectathon testing.
