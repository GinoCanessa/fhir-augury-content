## Summary

Martin Grundberg requests that ClinicalImpression be aligned with the FHIR event pattern by adding `partOf` and `basedOn` elements (and potentially other missing event features). ClinicalImpression is classified as an event-type resource but lacks these standard event pattern elements. The reporter's use case was A&E (Accident & Emergency) triage assessment. The resource remains at a low maturity level (FMM 0–1) with limited implementer adoption.

In a 2023 follow-up, Michelle Miller asked the reporter whether ClinicalImpression had been implemented. Martin Grundberg responded that he is no longer on the project but linked to a Cambio implementation guide for triage (RETTS). He noted this is not a blocker for him and was primarily a request for cross-resource pattern consistency, suggesting it could be closed if needed.

## Details

### Background

ClinicalImpression is intended to represent a clinician's assessment ("the A in SOAP") and has been in the FHIR specification since DSTU2 (~2015). Despite this long history, it remains at FMM level 0–1 with minimal implementer engagement. The FHIR event workflow pattern defines standard elements such as `basedOn`, `partOf`, `status`, `code`, `subject`, `encounter`, and `performer` that event-type resources are expected to carry for workflow consistency and cross-resource querying.

### Prior Work

[FHIR-10293](https://jira.hl7.org/browse/FHIR-10293) (resolved **Persuasive**, 2018) specifically discussed applying FHIR workflow standardization to Patient Care resources. That ticket recommended adding `partOf` and `basedOn` (as "in response to") to ClinicalImpression as extensions. This was approved, but ClinicalImpression's low maturity and limited uptake may mean these changes were not fully realized in subsequent releases.

### Broader ClinicalImpression Landscape

The Patient Care WG has been actively evaluating ClinicalImpression's future. In a June 2023 Zulip thread, Michelle Miller noted that Patient Care is "discussing whether to progress the maturity or remove this resource altogether" due to minimal feedback and implementation. Several related tickets reflect ongoing structural concerns:

- **[FHIR-46597](https://jira.hl7.org/browse/FHIR-46597)** (Triaged, Highest priority): Requests adding a `category` element to distinguish assessment types (comprehensive, focused, triage, nursing, etc.).
- **[FHIR-39832](https://jira.hl7.org/browse/FHIR-39832)** (Published, Persuasive with Modification): MCC Care Plan found ClinicalImpression lacked structure for condition status assessment.
- **[FHIR-35123](https://jira.hl7.org/browse/FHIR-35123)** (Published, Persuasive): Noted that the purpose and use of a ClinicalImpression extension was unclear, reflecting broader confusion about the resource's role.
- **[FHIR-8463](https://jira.hl7.org/browse/FHIR-8463)** (Published, Persuasive with Modification): Removed `ClinicalImpression.Trigger` due to ambiguity, an earlier instance of event-pattern cleanup.

### Reporter's Use Case

The reporter's original use case was implementing ClinicalImpression for A&E triage using the RETTS (Rapid Emergency Triage and Treatment System) protocol. An implementation guide was published at: https://simplifier.net/guide/CambioImplementationGuide2/ClinicalImpressionTriageRetts

## Proposed Dispositions

### Disposition A: Accept (Persuasive)

**Action:** Add `basedOn` and `partOf` elements to ClinicalImpression to align it with the FHIR event workflow pattern.

**Rationale:** ClinicalImpression is classified as an event-type resource and should conform to the event pattern for consistency. FHIR-10293 already approved this direction in 2018. Adding these elements would improve workflow interoperability, enable cross-resource searching (as addressed by FHIR-49437), and support use cases like triage assessments that naturally chain via `basedOn` (e.g., from a ServiceRequest) or `partOf` (e.g., a sub-assessment within a larger evaluation). This is a straightforward pattern alignment with established FHIR design principles.

### Disposition B: Defer pending ClinicalImpression maturity decision (Not Persuasive)

**Action:** Defer this ticket until Patient Care WG resolves the broader question of ClinicalImpression's future (progress maturity vs. retire the resource).

**Rationale:** The 2023 Zulip discussion indicates Patient Care is actively debating whether ClinicalImpression should be advanced or removed entirely. Investing effort in adding event pattern elements to a resource that may be deprecated is premature. The reporter himself indicated this is not a blocker and suggested the ticket could be closed. Once the WG decides the resource's fate, event pattern alignment can be addressed holistically alongside other structural requests (FHIR-46597 category, FHIR-39563 markdown description).

### Disposition C: Decline (Not Persuasive)

**Action:** Close as "Not Persuasive" given the reporter's own statement that this is no longer needed.

**Rationale:** The reporter stated in July 2023: "there is nothing blocking me/us at the moment" and "If you want, you can close it." ClinicalImpression has extremely low implementer adoption, and the community has largely moved to DocumentReference for clinical notes. With Patient Care considering retirement of the resource, adding pattern elements would add maintenance burden for minimal practical benefit. If ClinicalImpression is retained and matured in the future, event pattern alignment can be re-requested at that time.

## Recommendation

**Disposition B: Defer pending ClinicalImpression maturity decision.**

The request is technically sound — ClinicalImpression should carry event pattern elements if it remains in the specification. However, the resource is at FMM 0–1 with very limited implementation, and Patient Care is actively debating its future. The reporter has indicated this is not blocking work. Deferring avoids investing effort in a resource that may be retired, while preserving the option to apply the change if ClinicalImpression is advanced. This ticket should be revisited once the WG reaches a decision on the resource's maturity path, potentially in conjunction with FHIR-46597 (category element) for a comprehensive structural update.
