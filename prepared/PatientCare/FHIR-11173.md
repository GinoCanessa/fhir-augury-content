## Summary

This ticket requests that the CarePlan resource support expressing reviews — both at the component level (concerns, goals, activities) and at the whole-of-plan level. The minimum requirement proposed is a "next review date" element on CarePlan.

The ticket has an extensive history dating back to 2016. It was originally deferred (Nov 2016, vote 5-0-1) with the rationale that review tracking applies broadly across many resources (protocols, standing orders, long-term care admissions, etc.) and would best be handled via the Task resource with cross-work-group coordination. It was voted "Not Persuasive" (8-0-0) in July 2018, then reopened for further analysis with PA and OO work groups. As of the most recent comment (July 2025), it still has not been discussed on the CarePlan call and needs more discussion before voting.

## Details

### Original Request
CarePlan is missing support for expressing reviews. Care plan reviews are important activities/requirements that can occur at the component level (concerns, goals, activities) or at the whole-of-plan level. The submitter proposes at minimum a "next review date" element.

### Resolution Description (from prior deferral)
Tracking of reviews and plans for reviews applies to many resources, not just CarePlan. This was considered something best handled by Task, requiring analysis and discussion with other work groups. Deferred to R4.

The `careplan-notes.xml` file in HL7/fhir already states: "The Provenance resource can be used for detailed review information, such as when the care plan was last reviewed." This provides a pattern but does not add a dedicated element to CarePlan.

## Proposed Dispositions

### Disposition A: Accept — Add Review Extension to CarePlan

**Resolution:** Persuasive with Modification

Add a standardized extension (or first-class backbone element) to CarePlan capturing review metadata:
- `lastReviewDate` (dateTime) — when the care plan was last reviewed
- `nextReviewDate` (dateTime) — when the next review is due
- `reviewer` (Reference to Practitioner/PractitionerRole) — who performed or is responsible for the review

This addresses the minimum "next review date" requirement from the original request. Rather than waiting for a cross-resource review pattern (which has not materialized in 9 years), provide a CarePlan-specific solution that can be generalized later. The existing `artifact-reviewer` extension in fhir-extensions provides a precedent for reviewer metadata on FHIR resources.

**Rationale:** The ticket has been open since 2016 with no cross-resource review pattern emerging. A focused CarePlan extension is pragmatic and meets the stated clinical need. The existing Provenance-based guidance is insufficient because it requires separate resource creation and does not provide a simple queryable "next review date."

### Disposition B: Accept with Provenance Pattern — Formalize Provenance-Based Guidance

**Resolution:** Persuasive with Modification

Rather than adding new elements to CarePlan, formalize and strengthen the existing Provenance-based guidance in the CarePlan resource documentation:
1. Add explicit documentation with examples showing how to use Provenance to record review events (who, when, what was reviewed).
2. Define a recommended Provenance.activity code for "care plan review."
3. For the "next review date" need, recommend using a Task resource with `Task.code` = "review" and `Task.restriction.period` indicating the due date, referencing the CarePlan via `Task.focus`.

This approach avoids adding CarePlan-specific elements for a cross-cutting concern, leverages existing resources, and provides implementers with concrete actionable guidance.

**Rationale:** Aligns with the WG's long-standing position that reviews are a cross-resource concern. Avoids adding elements that may need to be deprecated if a broader pattern emerges. Provides immediate implementer value via better documentation and examples.

### Disposition C: Decline — Close as Not Persuasive

**Resolution:** Not Persuasive

The existing Provenance-based guidance documented in `careplan-notes.xml` already addresses this need. Adding dedicated review elements to CarePlan would set a precedent requiring similar additions to many other resources (protocols, standing orders, EpisodeOfCare, etc.). The concept of review tracking is a workflow concern best addressed through Task and Provenance, both of which already exist and are sufficient. No concrete implementation evidence has been presented showing the current approach is inadequate.

**Rationale:** The ticket was previously voted Not Persuasive (8-0-0 in July 2018). Despite multiple discussions since, no new technical requirements or implementation evidence has been brought forward. The current Provenance/Task approach is architecturally sound, and the ticket does not justify adding CarePlan-specific elements.

## Recommendation

**Disposition B: Accept with Provenance Pattern** is recommended.

This disposition best balances the clinical need with FHIR's architectural principles. The ticket has legitimate merit — care plan reviews are clinically important and the current documentation is thin — but the 9-year discussion history consistently concludes that reviews are a cross-resource concern. Rather than adding CarePlan-specific elements (Disposition A) or dismissing the need entirely (Disposition C), Disposition B provides immediate practical value by formalizing the Provenance + Task pattern with concrete examples and recommended codes. This gives implementers clear guidance while preserving architectural consistency. If a dedicated review pattern eventually emerges from cross-WG work, it will complement rather than conflict with this approach.
