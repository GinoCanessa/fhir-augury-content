## Description

The request proposes adding **status** and **status reason** elements to `CareTeam.participant` to track the lifecycle of individual participant membership on a care team. The original proposal listed 12 statuses modeled after task/request workflows: Proposed, Requested, Received, Accepted, Rejected, Ready, Cancelled, Active, On Hold, Not Started/Aborted, Completed, and Entered-in-Error.

The core problem: currently `CareTeam.status` applies only to the team as a whole, and `CareTeam.participant.period` (coveragePeriod) cannot distinguish between states like "suspended," "proposed," or "entered-in-error." There is no way to express that a participant has been proposed but not yet accepted, or that their participation has been temporarily paused.

## Discussion History (17 Comments, Dec 2019 – Jun 2024)

This ticket has been discussed extensively across **10+ WGM sessions and conference calls** over nearly five years:

### Dec 2019 – Initial Discussion
- Emma Jones raised this from a Clinicians on FHIR discussion.
- Rob questioned whether CareTeam should only include accepted participants.
- Debate on whether CareTeam is part of the workflow pattern (Lloyd McKenzie indicated it is).
- If status isn't on participant, which resource handles proposing/requesting a member?

### Jan 2020 – Scope Refinement
- Emma agreed CareTeam is **not** part of workflow.
- Consensus that individual members need "simple" statuses (active, inactive) to convey team membership.
- Emma tasked with revising the proposed value set with Learning Health.

### Apr 2020 – Proposal Requirements
- Michelle Miller requested a full formal proposal per FHIR spreadsheet authoring guidelines (element name, cardinality, data type, invariants, binding, value set).

### Sep 2020 – Formal Element Proposal (Emma Jones)
- Proposed element: `careTeam.participant.status` (careTeamParticipantStatus)
- Cardinality: 0..1
- Data Type: CodeableConcept
- Binding: Extensible
- Is Modifier: true (due to entered-in-error)
- Value set: proposed | pending | active | suspended | inactive | permanently inactive | entered-in-error

### Mar 2022 – Status vs. CoveragePeriod Tensions
- Key question: What does status convey that coveragePeriod cannot?
  - Active/inactive can be inferred from period, but **suspended cannot**.
  - Proposed and entered-in-error have no period equivalent.
- Use cases: PCP changes due to insurance; seasonal caregivers (Florida daughter vs. Michigan son); rehab that is planned but not started.
- Stephen Chu suggested using an invariant to relate status and coveragePeriod.
- Emma compared to CarePlan.activity.plannedActivityDetail statuses.
- TODO noted: Clarify the definition of "coverage" (disambiguate "active" vs "covering").

### Sep 2022 – Structural Restructuring Proposal
- **Major pivot**: Discussion shifted to restructuring `CareTeam.participant` into a two-level backbone:
  - `CareTeam.role` (backbone 0..*) — renamed from participant
    - `code` (CodeableConcept) — the role itself
    - `period` (Period | Timing) — when the role is needed
    - `status` — open question whether redundant with period
    - `member` (backbone 0..*) — individual actors filling the role
      - `actor` — Reference(Practitioner | PractitionerRole | RelatedPerson | Patient | Organization | CareTeam)
      - `coverage[x]` (Period | Timing) — member availability
      - `onBehalfOf` — still needed?
- PA/PC joint session: PA recommended using the Availability data type; had reservations about "coverage" naming.
- Lisa Nelson recommended moving reasonCode/reasonReference into the backbone element.

### Sep 2023 – Continuing Debate
- Scope question: Does CareTeam include past members or only go forward?
- Gay suggested "participation degree" as alternative to status.
- Grahame Grieve reviewed CareTeam extensions across IGs — most treat it as a snapshot of current state. Suggested building a comprehensive profile first.

### Jan 2024 – Concerns Raised
- **Grahame Grieve (mckenzie)** raised significant concerns:
  - If something needs a status, it may warrant its own **independent resource**.
  - The accept/reject statuses overlap heavily with **Task** resource.
  - "Team membership isn't something that can be 'completed' because there's no 'action' involved."
  - Called for "pretty careful consideration before approving."
- Noted relation to PA Jira [FHIR-34379](https://jira.hl7.org/browse/FHIR-34379) (Add employment category in PractitionerRole) — resolved as Not Persuasive.

### May/Jun 2024 – Most Recent (WGM LHS Quarter)
- Multiple attendees agreed a status element **is needed** at careTeam.participant.
- Additional suggestions: privileges/functions for hospital-based team members.
- Suggestion for a **trial implementation** — potential Connectathon/CoF topic.

## Proposed Dispositions

### Disposition A: Persuasive with Modification — Add Simple Participant Status

**Resolution:** Add a `CareTeam.participant.status` element with a **simplified** value set focused on membership state rather than task-like workflow states.

**Rationale:** The workgroup has consistently agreed over five years that a status element is needed. Grahame Grieve's concern about overlap with Task is valid — the solution is to scope the value set to membership lifecycle only (proposed, active, suspended, inactive, entered-in-error), dropping request/accept/reject/completed semantics. This avoids reinventing Task while addressing the real gap: coveragePeriod alone cannot express "suspended" or "proposed" states. The element should be:
- Cardinality: 0..1
- Data type: code (bound to a required value set, not CodeableConcept)
- Is Modifier: true
- Value set: proposed | active | suspended | inactive | entered-in-error

Defer the larger structural restructuring (role/member backbone hierarchy) to a separate R6 tracker.

### Disposition B: Deferred — Restructure CareTeam for R6

**Resolution:** Defer this ticket until R6 to incorporate the broader `CareTeam.participant` restructuring into `CareTeam.role` and `CareTeam.role.member` backbone elements, with status at the appropriate level(s).

**Rationale:** The Sept 2022 WGM discussions revealed that the real need may be a deeper structural change — separating the concept of a "role on the team" from "who fills that role." Adding a simple status now may create technical debt if the structure changes fundamentally in R6. Grahame's suggestion to build a comprehensive profile from existing extensions first would inform the right design. However, this means the gap persists for another release cycle.

### Disposition C: Not Persuasive — Use Existing Mechanisms

**Resolution:** Reject the change. The existing `CareTeam.participant.period` (coveragePeriod), combined with `CareTeam.status` and extensions, is sufficient. For workflow-like statuses (proposed, accepted, rejected), use the Task resource to manage participant onboarding.

**Rationale:** As Grahame noted, if participation needs a status lifecycle, it may warrant an independent resource or Task-based workflow. Adding status to participant risks conflicting information with period and duplicating Task semantics. Extensions already exist for implementations that need this. However, this position contradicts the repeated workgroup consensus that a gap exists and ignores that "suspended" genuinely cannot be represented today.

## Recommendation

**Disposition A: Persuasive with Modification** — Add a simple, membership-focused `CareTeam.participant.status` element.

This is recommended because:
1. **Five years of consistent workgroup agreement** that a status element is needed, including the most recent May 2024 WGM.
2. **Genuine representational gap**: "suspended," "proposed," and "entered-in-error" states cannot be expressed with coveragePeriod alone.
3. **Scoping to membership states** (not task-like workflow) addresses Grahame's valid concern about Task overlap — this is about *whether someone is on the team*, not about *requesting/accepting/completing work*.
4. **Deferring the structural restructuring** to a separate ticket avoids blocking a needed improvement on a larger, still-unresolved design question.
5. A **trial implementation** (suggested at May 2024 WGM) could validate the design before full normative inclusion.

The workgroup should also consider adding an optional `CareTeam.participant.statusReason` (CodeableConcept, 0..1) to capture why a participant was suspended or inactivated (e.g., insurance change, relocation, retirement), as several use cases in the discussion history depend on this context.
