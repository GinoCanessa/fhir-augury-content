## Summary

This change request asks for the ability to record detailed surgical timestamps (e.g., incision, suture, anesthesia start/end, OR entry/exit) directly within the Procedure resource. The request originates from German hospital requirements where precise perioperative process timestamps are a **legal requirement** and a key driver for hospital process optimization and transparency.

The submitter (René Störmer, via Vassil Peytchev) argues that a single Procedure tracks the full scope of `Procedure.code` and that splitting one procedure into 18+ sub-procedures for timestamps is impractical, confusing, and bad for query performance. Using extensions is also considered suboptimal for a core feature due to readability and query performance concerns.

The ticket has been discussed in multiple Patient Care calls (Nov 2020, Apr 2021, Sep 2025) and a joint OO/PC session without reaching a final resolution. The most recent discussion (Sept 2025) identified three candidate approaches.

---

## Details

### Original Use Case

German hospitals must document timestamps as surgical procedures unfold in real time. A typical procedure involves ~14–36 timestamps such as:
- Patient in OR area / OR room
- Anesthesia start/end
- Incision (cut) / suture
- Patient out of OR room / OR area
- Room cleaning start/end

The submitter referenced a German standard document (BDA/DGAI, BDC/DGCH, VOPM glossary of perioperative process times, 2016) that defines recommended timestamps on pages 674+.

### Three Options Identified (Joint OO/PC, September 2025)

1. **Observation-based**: Use Observation resources pointing back to the Procedure via `Observation.partOf` or `Observation.subject`. Open question: what makes each Observation unique?
2. **List of Periods**: Add a repeating element or core extension on Procedure to capture coded timestamps/periods — simpler but potentially limited.
3. **Sub-Procedures**: Use contained or referenced Procedure resources linked via `partOf` — the well-understood existing pattern. Each sub-procedure carries `status`, `code`, and `occurrence[x]`.

### Key Discussion Points

- **Jose Costa Teixeira** (multiple comments): The requirement is already met by sub-procedures (contained or referenced). Adding timestamps alone to the core is a "design preference," not an unmet requirement. Scope will inevitably grow to include performers, locations, and sub-statuses — exactly what sub-procedures already support.
- **Lloyd McKenzie**: Procedure has not been well-exercised for real-time capture. Contained sub-procedures are a viable approach. Formal change requests are the path to improvements.
- **René Störmer**: Sub-procedures create 18+ resources for a single procedure code, complicating status management and queries. Extensions are unsuitable for a core feature used in performance-sensitive queries.
- **Michelle Miller / Patient Care WG**: Agreed that parent-child Procedures (via `partOf`) are appropriate for one overarching procedure with components. Multiple independent procedures during a single OR visit need different guidance.
- **Marti Velezis (2025)**: Action items assigned — Grant/Marti to contribute use cases; Michelle to reach out to original reporter.

## Proposed Dispositions

### Disposition A: Accept — Add a repeating backbone element for coded timestamps on Procedure

Add a new repeating backbone element to the Procedure resource (e.g., `Procedure.event`) containing:
- `code` (CodeableConcept) — identifies the event type (incision, suture, anesthesia start, etc.)
- `occurrence[x]` (dateTime | Period) — when the event occurred
- `performer` (Reference) — optional, who performed/observed the event

**Rationale:** This directly addresses the request and the 2025 "list of Periods" option. It provides a first-class, queryable mechanism for surgical timestamps without the overhead of dozens of sub-resources. It aligns with the legal requirements in Germany and similar jurisdictions and supports the analytics/process-optimization use case efficiently.

**Risk:** Scope creep — once timestamps are a core element, requests to add device, location, and notes per event are likely, potentially duplicating what sub-procedures already model. May also set a precedent for other resources wanting inline event logs.

### Disposition B: Provide formal guidance recommending sub-procedures with a standard extension

Publish implementation guidance (and optionally a standard extension) that documents the recommended pattern for capturing surgical timestamps using contained sub-procedures linked via `partOf`. The guidance would cover:
- How to code sub-procedure events (incision, anesthesia, etc.)
- Using `occurrence[x]` on sub-procedures for timestamps
- When to use contained vs. referenced sub-procedures
- A worked example similar to the JSON provided by Jose Costa Teixeira in the Jira comments

**Rationale:** This leverages existing FHIR capabilities without core changes. The sub-procedure pattern is well understood, supports future extensibility (adding performers, devices, locations per step), and was endorsed by multiple commenters. A formal IG or guidance page removes ambiguity for implementers.

**Risk:** Implementers like the original submitter find the sub-procedure approach verbose and difficult to query efficiently, especially for 30+ timestamps. Without a core element, implementations may diverge.

### Disposition C: Decline — No change needed; existing mechanisms are sufficient

Close the ticket as not requiring a core change. Document in Procedure's usage notes that sub-procedures (via `partOf`, potentially contained) are the standard mechanism for capturing surgical sub-events and timestamps. Point implementers to existing profiling capabilities and extensions.

**Rationale:** As argued by Jose Costa Teixeira: capturing sub-statuses and timestamps is what sub-procedures are for. Adding a flat timestamp list to Procedure would hide the hierarchical nature of surgical workflows and would inevitably need to grow to include performers, locations, and other context — ending up as a duplicate of the existing sub-procedure model.

**Risk:** The ticket has been open since 2020 with no resolution, and declining without guidance may frustrate implementers facing real regulatory requirements. Related ticket FHIR-48449 raises the same gap.

---

## Recommendation

**Disposition B: Provide formal guidance recommending sub-procedures with a standard extension.**

This ticket has been discussed for five years across multiple Patient Care calls without consensus on a core change. The sub-procedure pattern using `partOf` (with contained resources) is a viable, well-understood solution that already exists in FHIR and was concretely demonstrated in the Jira comments. What is missing is **official guidance** — implementers are confused about the recommended approach, as evidenced by repeated Zulip threads and the related FHIR-48449 ticket.

Publishing clear implementation guidance with worked examples, and optionally defining a standard extension for simple timestamp-only use cases, would resolve the practical gap without the risks of adding a new core element. This also positions the community to evaluate whether a core change is warranted based on adoption experience, while immediately helping implementers who need to meet regulatory deadlines. If adoption experience later shows that the sub-procedure pattern is genuinely insufficient, the guidance can be revisited with concrete evidence for a Disposition A-style core change.
