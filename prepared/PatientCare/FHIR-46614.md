## Summary

The submitter requests changing the cardinality of `ClinicalImpression.previous` from `0..1` to `0..*`. The rationale is that more than one previous assessment may be relevant to a given ClinicalImpression — assessments may have been conducted in the same time frame by different authors while still relating to the same clinical problem(s). The current `0..1` cardinality only allows linking to a single prior assessment, which does not support use cases where multiple prior assessments must be directly referenced rather than navigated via a chain.

## Analysis

This ticket requests a cardinality change to `ClinicalImpression.previous` from `0..1` to `0..*`. The core clinical argument is that previous assessments are not always a linear chain — multiple prior assessments from different authors or time frames may all be relevant comparators for the current impression.

The most significant development is that a closely related ticket, **FHIR-50413**, has already been resolved as **Persuasive with Modification**. That resolution goes further than just changing the cardinality: it replaces `previous` entirely with a new element `comparesTo` at `0..*`, referencing `RelatedArtifact | ClinicalAssessment`. The Patient Care WG discussed this at length, with clinical use cases including:
- **Oncology:** Progressively monitoring lesions/tumors across multiple assessment time points
- **Cardiology:** Tracking heart function deterioration over time
- **Orthopedics:** Progression of lesions across visits

The WG specifically preferred `comparesTo` over `previous` or `relatesTo` because it more precisely conveys the intent of comparison rather than simple temporal succession.

A comment from michelle.m.miller on FHIR-46614 (Sept 18, 2025) explicitly links it to FHIR-50413 and asked the submitter to elaborate on use cases for multiple previous assessments and why walking the chain doesn't suffice. No further response from the submitter has been recorded.

## Proposed Dispositions

### Disposition A: Persuasive — Change cardinality to 0..*
Change `ClinicalImpression.previous` cardinality from `0..1` to `0..*` as requested.

- **Rationale:** The submitter's request is clinically valid — multiple prior assessments may be relevant. The cardinality change is the minimal change needed to address the requirement.
- **Pros:** Simple, backward-compatible change; existing implementations with `0..1` still work. Directly addresses the submitter's request.
- **Cons:** Does not address the broader semantic issues identified in FHIR-50413 (the name "previous" implies linear succession, not a set of comparators). Would be partially redundant with / potentially conflicting with the FHIR-50413 resolution.

### Disposition B: Persuasive, Applied via FHIR-50413 — Subsume into related ticket
Mark as Persuasive but note that the underlying requirement is addressed by the resolution of FHIR-50413, which replaces `previous` with `comparesTo` at `0..*` and broadens the reference types to include `RelatedArtifact | ClinicalAssessment`.

- **Rationale:** FHIR-50413 already encompasses and extends this request. The `comparesTo` element at `0..*` satisfies the cardinality requirement while also fixing the semantic mismatch of the `previous` name. Avoids duplicate or conflicting changes.
- **Pros:** Clean resolution that avoids making the cardinality change twice; aligns with the WG's considered direction. The submitter's core need (referencing multiple prior assessments) is fully met.
- **Cons:** The submitter may not agree with the rename from `previous` to `comparesTo`.

### Disposition C: Not Persuasive — Maintain current cardinality
Retain `ClinicalImpression.previous` as `0..1`.

- **Rationale:** The current design supports walking a chain of previous assessments (each `ClinicalImpression.previous` can point to an earlier one which itself has a `previous`). This chain model is sufficient for most use cases.
- **Pros:** No breaking changes; preserves the intended linear-chain semantics.
- **Cons:** Does not address the valid clinical scenario where multiple independent prior assessments (not in a chain) need to be referenced. Already contradicted by the WG's resolution of FHIR-50413.
