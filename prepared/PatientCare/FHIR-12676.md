## Problem Statement

The reporter seeks guidance on how to represent the **SOAP (Subjective, Objective, Assessment, Plan)** documentation model used by GPs (general practitioners) in FHIR. In a typical GP encounter, the patient presents with one or more complaints, each generating a set of S-O-A-P journal entries:

- **Subjective (S):** Plain text, patient's description of complaint; may be associated with an ICPC-1-NL code.
- **Objective (O):** May be text or formal Observations (length, weight, lab tests).
- **Assessment (A):** Plain text, clinician's assessment; most likely to carry an ICPC code.
- **Plan (P):** May be text, or references to MedicationRequest, CarePlan, Referral, Procedure, etc.

Multiple S-O-A-P groups can exist within a single Encounter (one per complaint/problem), and they may be associated with an EpisodeOfCare for longitudinal problem tracking. The reporter feels FHIR lacks an "EncounterReport" resource and asks whether Composition or ClinicalImpression is the correct approach.

## Zulip Discussions

### 1. [implementers > GP contact report](https://chat.fhir.org/#narrow/stream/implementers/topic/GP%20contact%20report) (Jan–May 2017, 27 messages)

The original discussion cross-posted to Jira. Key participants: Alexander Henket, Lloyd McKenzie, Ben Schrijver, Aurelius Baier, Oliver Egger.

**Key points:**
- Lloyd McKenzie initially suggested **ClinicalImpression + CarePlan** was intended to cover SOAP but acknowledged it should be better documented.
- Alexander Henket found ClinicalImpression awkward for the Subjective component and explored alternatives.
- Dutch implementers (Ben Schrijver) tried Observation then CarePlan for the P component; both felt "a bit off."
- Swiss implementers (Aurelius Baier / in4medicine/elexis) shared their approach: S→Condition (category=complaint), O→Observation, A→ClinicalImpression, P→ProcedureRequest, all linked via Encounter.
- Lloyd McKenzie agreed that splitting a single visit into multiple Encounters per problem is not viable (encounters have billing/administrative significance).
- Alexander Henket emphasized the Dutch concept of "Deelcontact" (encounter part) for grouping SOAP tuples within a single Encounter around different problems, linked to Episodes.
- No consensus was reached; more implementation experience was deemed necessary.

### 3. [implementers > SOAP Notes](https://chat.fhir.org/#narrow/stream/implementers/topic/SOAP%20Notes) (Jan 2018, 50 messages)

Broader discussion on SOAP note representation. Key participants: Lloyd McKenzie, Rob Hausam, Alexander Henket, Eric Haas, Michelle Miller, Brett Marquard, Michael Donnelly, Josh Mandel, Grahame Grieve.

**Key points:**
- Rob Hausam confirmed Patient Care did **not** want to use ClinicalImpression for SOAP notes.
- Michelle Miller stated the leading candidates were **DocumentReference and Composition**.
- Lloyd McKenzie argued Composition is for packaging data — all clinical data should live in semantic resources, not solely in Composition.section.text. He raised concerns about exempting SOAP notes from this principle.
- Eric Haas pushed back, arguing Composition.section.text is used in practice for narrative-only content and that the Composition ≠ Document conflation causes confusion.
- Michael Donnelly and Josh Mandel emphasized the practical need: plaintext notes are often the only representation, and notes need to flow both directions (read and write).
- Grahame Grieve acknowledged the spectrum from "a few paragraphs in the record" to "carefully constructed documents" and that different solutions suit different ends.
- Brett Marquard noted the emerging approach of using **DocumentReference** to discover notes, with the DocRef pointing to Binary (PDF/RTF) or Composition for structured content.
- No final resolution in this thread.

## Analysis

This ticket has been open since **January 2017** (over 7 years) and was auto-deferred. It represents a real and recurring implementation challenge — how to represent the GP SOAP documentation workflow in FHIR — that has generated substantial community discussion. Despite multiple WGM sessions and Zulip threads, no definitive guidance has been published in the FHIR specification.

**Core tensions identified:**
1. **ClinicalImpression** was the original intended resource but Patient Care explicitly moved away from it for this use case. It maps poorly to the full SOAP model (only the "A" fits well).
2. **Composition** is widely used in practice but architecturally is meant to package resources, not be the primary home for clinical data.
3. **No single resource** maps to the full SOAP model — each letter maps to different resource types (Condition/Observation for S, Observation for O, ClinicalImpression for A, CarePlan/MedicationRequest/etc. for P).
4. The need to **group SOAP entries within an Encounter by problem** has no clean mechanism in FHIR (the Dutch "Deelcontact" concept).

**Since this ticket was filed**, FHIR-23958 established a Composition-based SOAP-like pattern (cpg-caseplannote), and FHIR-46597 proposes adding Category to ClinicalImpression, which could partially address distinguishing assessment types. The US Core Clinical Notes guidance has also matured, using DocumentReference + Composition.

## Proposed Dispositions

### Disposition A: Persuasive — Add SOAP Guidance to Specification
Add a section to the ClinicalImpression and/or Composition resource pages (Boundaries & Relationships) providing explicit guidance on representing SOAP notes in FHIR. Document the recommended pattern:
- **S** → Condition (category=encounter-diagnosis or complaint) or narrative in Composition.section
- **O** → Observation resources referenced from Encounter
- **A** → ClinicalImpression or narrative in Composition.section
- **P** → CarePlan, MedicationRequest, ServiceRequest, etc.
- Use Composition (type=LOINC 67781-5) to bundle the SOAP structure with sections coded per SOAP component.

Include a worked example showing a GP encounter with multiple complaints. Note that grouping by problem within an encounter can be achieved via EpisodeOfCare references or Composition sections. This would directly address the original request for guidance without requiring new resources.

### Disposition B: Persuasive with Modification — Defer to Implementation Guide
Acknowledge the need but redirect to an Implementation Guide rather than core spec changes. The SOAP documentation pattern is a profiling concern that varies by jurisdiction (Dutch GP vs. Swiss GP vs. US ambulatory). Recommend:
- Creating a FHIR IG for primary care encounter documentation patterns
- Reference existing work (Alexander Henket's Simplifier profiles, cpg-caseplannote from Clinical Guidelines IG)
- Add a brief mention in ClinicalImpression and Composition B&R sections pointing to the IG

This keeps the core spec clean while providing needed guidance.

### Disposition C: Not Persuasive — Close as Out of Scope for Core
The SOAP note documentation pattern is a profiling and implementation concern, not a gap in the core specification. The necessary resources already exist (Composition, ClinicalImpression, Condition, Observation, CarePlan, etc.). National/regional IGs (such as the Dutch GP profiles on Simplifier) are the appropriate venue for this guidance. The auto-deferred status over 7+ years and the emergence of Clinical Notes guidance in US Core suggest the community has organically addressed this without core spec changes.

## Recommendation

**Disposition A: Persuasive — Add SOAP Guidance to Specification**

This ticket identifies a genuine gap in the FHIR specification: the lack of guidance on a universally used clinical documentation pattern. After 7+ years, the extensive Zulip discussions demonstrate that implementers repeatedly struggle with the same questions and arrive at inconsistent solutions. The resources exist but the guidance on how to combine them for SOAP documentation does not. Adding a guidance section with a worked example is low-cost, high-value, and would prevent future implementers from having to re-discover the same patterns. The cpg-caseplannote precedent (FHIR-23958) validates Composition as the anchoring resource for this pattern.
