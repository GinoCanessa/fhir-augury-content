## Description

The submitter requests adding `dosageInstructions 0..*` to the Procedure resource to allow blood products to be administered with multiple procedures tied to a ProcedureRequest.

## Keywords

Procedure, DosageInstructions, dosage, blood products, blood transfusion, BiologicallyDerivedProduct, MedicationAdministration, ServiceRequest, non-medication administration, cellular therapy, surgical procedure, perioperative medications, extension, backbone element

## Proposed Dispositions

### Disposition A: Persuasive — Add a dosage extension to Procedure

**Action:** Create a FHIR extension on Procedure based on the MedicationAdministration.dosage backbone element structure, with cardinality `0..*`. Publish in the FHIR Extensions Pack. This aligns with the approach already taken for ServiceRequest in FHIR-55923 (using DosageDetails on the request side) and the Oct 2025 WGM action item for PC to create a dosage extension on Procedure.

**Rationale:** The extension approach was explicitly endorsed by Patient Care in the Oct 2025 discussion as the preferred short-term path. Extensions are non-normative and can be modified more easily as the broader Non-MedicationProduct resource family matures. This provides immediate value for blood transfusion, cellular therapy, and perioperative medication use cases without constraining R6 normative decisions.

**Risk:** The extension may become redundant if new resources (e.g., Non-MedicationProductAdministration) are created that subsume these use cases.

### Disposition B: Deferred — Wait for Non-MedicationProduct resource proposals from OO

**Action:** Defer this ticket until OO completes resource proposals for Non-MedicationProductAdministration and related resources. Reassess whether dosage on Procedure is still needed once the new resource family is defined.

**Rationale:** Stephen Chu's April 2026 comments highlight that the modeling question is broader than just dosage — it involves the full lifecycle of non-medication products (request, dispense, administration, statement). Adding dosage to Procedure without resolving the resource-level questions risks creating parallel, inconsistent paths. OO is planning to revisit this at the Rotterdam WGM.

**Risk:** Further delays a 9-year-old request. Implementers continue to rely on ad-hoc extensions without standardization.

### Disposition C: Persuasive with Modification — Add dosage as a backbone element within Procedure.used

**Action:** Extend the existing `Procedure.used` backbone element to include a `dosage` child element (e.g., `Procedure.used.dosage 0..1`) modeled after MedicationAdministration.dosage. This leverages the existing `Procedure.used.item` which already references Medication, Substance, BiologicallyDerivedProduct, and Device.

**Rationale:** This was specifically discussed in the Oct 2025 PC call and appears in the Procedure.used structure discussion. It ties dosage directly to the substance/product being administered, which is clinically meaningful. It avoids a free-floating extension and integrates naturally with the existing Procedure model.

**Risk:** Procedure is going normative in R6, so backbone element changes are harder to revise later. Stephen Chu and others have questioned whether "used" semantically covers "administered" (e.g., "you don't say you 'use' IV fluid in a procedure"). The Apr 2026 call asked for 2–3 weeks of further use case analysis before deciding.

## Recommendation

**Disposition A: Persuasive — Add a dosage extension to Procedure.**

This is the most pragmatic path forward given the current state of discussion. It directly addresses the original request, aligns with the parallel work on FHIR-55923 (ServiceRequest dosage extension), and follows the approach explicitly endorsed by Patient Care in October 2025. The extension mechanism provides flexibility — it can be promoted to a backbone element later or deprecated if new Non-MedicationProduct resources render it unnecessary. After 9 years in the backlog, an actionable, non-breaking incremental step is warranted. The workgroup should coordinate with OO to ensure the extension's structure aligns with the emerging DosageDetails data type used on the request side.
