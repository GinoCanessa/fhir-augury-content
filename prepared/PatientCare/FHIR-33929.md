## Summary

This change request proposes making `subject` and `focus` into choice elements within the Ophthalmology Retinal profiles (and potentially other profiles) to prevent both from being populated simultaneously. The rationale is that the description for `focus` states it should be used when the observation is not about the subject, so allowing both to be populated creates ambiguity. The reporter notes this pattern appears in other profiles as well.

## Details

### Description

The submitter suggests making `subject` and `focus` into choice elements to prevent the use of both simultaneously. The reasoning is that the FHIR specification description for `focus` states it should be used when the observation is not about the subject — therefore, if `focus` is populated, `subject` should not also be populated, and vice versa. The reporter indicates this issue appears across multiple profiles in the Ophthalmology Retinal IG.

## Proposed Dispositions

### Disposition A: Persuasive (Accept with Profile Constraint)

Add a FHIR profile invariant (constraint) to the Ophthalmology Retinal Observation profiles that enforces mutual exclusivity between `subject` and `focus`. For example, an invariant such as `subject.empty() or focus.empty()` would ensure that only one of the two elements is populated at a time. This addresses the submitter's concern without altering the underlying element structure and can be applied consistently across affected profiles in the IG.

**Rationale:** This is the most practical approach. Converting two distinct elements into a single choice element would require a fundamental restructuring that is not how FHIR choice elements work (choice elements represent alternative *types* for a single logical element, not alternative *elements*). A profile-level invariant achieves the same mutual exclusivity goal using standard FHIR profiling mechanisms, and precedent exists in FHIR-28946 and FHIR-37736 for clarifying subject/focus usage through constraints.

### Disposition B: Persuasive with Modification (Clarify Documentation Only)

Rather than adding structural constraints, update the Ophthalmology Retinal profile documentation to clearly state the intended usage of `subject` versus `focus`, explaining when each should be used and that they should not both be populated when focus represents a non-subject entity. This provides guidance without adding technical enforcement.

**Rationale:** The underlying concern may stem from unclear documentation rather than a structural deficiency. FHIR-37736 (same reporter) was resolved with clarification to Observation.focus semantics. A documentation-first approach avoids over-constraining profiles while addressing the ambiguity.

### Disposition C: Not Persuasive (Decline)

Decline the request. The base FHIR Observation resource intentionally supports both `subject` and `focus` simultaneously for valid use cases (e.g., an observation about a RelatedPerson in the context of a patient's record, as discussed in FHIR-37736). Making these elements mutually exclusive — even at the profile level — could prevent legitimate clinical scenarios. The existing element definitions are sufficient, and implementers can apply local constraints as needed.

**Rationale:** The FHIR core design intentionally allows both `subject` and `focus` to coexist. The `focus` element documentation states it is used "when the observation is about something (or someone) associated with the patient, but is not about the patient themselves," which does not preclude `subject` also being present (the subject provides the patient context). Restricting this at the IG level may be overly restrictive.

## Recommendation

**Disposition A: Persuasive (Accept with Profile Constraint)** is recommended.

The submitter raises a valid modeling concern: within the Ophthalmology Retinal profiles, allowing both `subject` and `focus` without guidance creates ambiguity for implementers. However, the specific mechanism requested (making them a "choice element") is not technically feasible in FHIR — `subject` and `focus` are distinct elements, not alternative types of one element. The most appropriate resolution is to add a FHIRPath invariant at the profile level enforcing mutual exclusivity (e.g., `subject.empty() or focus.empty()`), which achieves the submitter's intent using standard FHIR profiling. This approach is consistent with how FHIR-28946 resolved a similar ambiguity with Task.focus, and aligns with the broader community discussion in FHIR-37736 about clarifying focus semantics. The companion tickets FHIR-33930 and FHIR-33931 from the same reporter should also be reviewed in the same session, as they address related choice element modeling issues in the same specification.
