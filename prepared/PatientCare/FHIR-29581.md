## Description

The submitter raises two questions about the FunctionalStatus and BundledFunctionalStatus profiles in the PACIO Functional Status IG:

1. **Why is Observation.value constrained to CodeableConcept?** There is no specific bound value set, so it is unclear what codes are appropriate to use.
2. **What is the meaning of `.value` in a bundled set of observations?** In BundledFunctionalStatus, individual observations are contained in `.hasMember`, so the purpose of `.value` on the parent observation is unclear.

The request asks the IG authors to clarify the usage of `Observation.value` in both profiles.

## Proposed Dispositions

### Disposition A: Persuasive — Clarify value usage and bind a value set

**Action:** Add clarifying narrative to the IG explaining the purpose of `Observation.value[x]` on both FunctionalStatus and BundledFunctionalStatus profiles. Bind `Observation.value[x]` to an appropriate value set (e.g., drawing from ICF or LOINC answer lists for functional status assessments). For BundledFunctionalStatus, clarify that `.value` may carry an overall/summary assessment while `.hasMember` carries individual observations.

**Rationale:** The submitter's concern is valid — constraining to CodeableConcept without a bound value set creates ambiguity for implementers. Since the original deferral noted that "work is underway by a HL7 group to define a value set for Functional Status," and the IG has since evolved into the PFE IG, this work should now be incorporated. The closely related FHIR-30047 was resolved as Persuasive with Modification, indicating the workgroup agreed these clarifications were needed.

### Disposition B: Applied — Duplicate of FHIR-30047

**Action:** Close as duplicate of FHIR-30047, which was resolved as "Persuasive with Modification" and published. Confirm that the changes made in response to FHIR-30047 (clarifying the purpose of `.code` and `.value` on BundledFunctionalStatus) also address the concerns raised here.

**Rationale:** FHIR-30047 asks substantially the same question about the purpose of `Observation.value` in BundledFunctionalStatus and was resolved. If the modifications made for FHIR-30047 also clarify the FunctionalStatus profile's value usage, this ticket may already be addressed. This avoids duplicating effort across the successor PFE IG.

### Disposition C: Not Persuasive — Superseded by PFE IG redesign

**Action:** Close as not persuasive, noting that the PACIO Functional Status IG has been superseded by the PACIO Personal Functioning and Engagement (PFE) IG, which restructures the observation profiles and may no longer have the same value constraint pattern. Direct the submitter to file new issues against the PFE IG if concerns persist.

**Rationale:** The profiles referenced (FunctionalStatus, BundledFunctionalStatus) belong to the STU1 PACIO FS IG which has been replaced. The successor PFE IG has undergone its own ballot cycles and may have already resolved these modeling questions through its redesign. Reworking constraints on a deprecated specification provides little value.

---

## Recommendation

**Disposition B: Applied — Duplicate of FHIR-30047** is recommended.

FHIR-30047 directly addresses the same core question about the purpose of `Observation.value` in BundledFunctionalStatus and was resolved as "Persuasive with Modification." The PACIO Functional Status IG has been superseded by the PFE IG, meaning any remaining value set binding concerns should be addressed against the new specification rather than the deprecated one. Closing as a duplicate acknowledges the validity of the concern while avoiding redundant work on a superseded IG. If the PFE IG still has unclear `.value` semantics, a new ticket against that IG would be more appropriate.
