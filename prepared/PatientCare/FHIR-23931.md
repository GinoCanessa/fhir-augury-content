## Description

This ticket raises the question of distinguishing between two conceptual "IPS products":

1. **The IPS Document** — a FHIR Bundle document with specific constraints for the self-contained International Patient Summary use case.
2. **The IPS Library / Toolbox** — the set of minimal, non-exhaustive "building blocks" (profiles) that compose the IPS document and could be reused across domains and jurisdictions beyond the document scope.

The ticket references a [Zulip discussion (IPS: Document Vs Library)](https://chat.fhir.org/#narrow/stream/207835-IPS/topic/IPS.3A.20Document.20Vs.20Library) where Giorgio Cangioli articulated this ambiguity and proposed creating two layers of profiles: those directly used in the IPS FHIR document, and those representing minimal reusable building blocks.

The ticket was initially resolved as "Considered for Future Use" in Feb 2020 (vote 12-0-1, Rob Hausam/Alexander Zautke) with the action: *"Identify a set of examples/cases that justifies the need of creating a new set of less constrained profiles to be used beyond the IPS document scope."* It was re-opened post-STU update publication in Nov 2022.

## Analysis

### Core Issue
FHIR-23931 identifies a fundamental architectural tension in the IPS specification: the IPS profiles serve dual purposes — constraining content within the IPS Bundle document **and** providing reusable building blocks for broader interoperability. This dual role creates problems:

- **Overly tight constraints** (e.g., requiring `subject.reference` as 1..1) break non-document use-cases.
- **Profile consumers** outside the document context (e.g., EU cross-border projects, IPA) cannot directly reuse IPS profiles without hitting document-specific constraints.
- **The IPA/IPS relationship** remains unresolved — both specs maintain separate profile sets for overlapping clinical content.

### Evolution Since Filing
- The ticket was initially "Considered for Future Use" (2020) with a clear action item to gather justifying examples.
- Subsequent Zulip discussions (especially the 2022 Must Support Review) provided concrete examples: an EU cross-border project could not use IPS profiles as-is due to document-specific constraints.
- IPA has since matured, creating a de facto parallel profile set.
- The ticket was re-opened in Nov 2022, acknowledging the issue remains relevant.

### Impact
- **Specification clarity:** The IPS IG landing page already describes profiles as "potentially reusable beyond the IPS scope," but the profiles themselves carry document-specific constraints that prevent this reuse.
- **Implementer confusion:** Implementers expecting reusable "toolbox" profiles encounter document constraints.
- **IPA/IPS alignment:** Without resolution, IPA and IPS maintain parallel, slightly different profiles for the same clinical concepts.

## Proposed Dispositions

### Disposition A: Persuasive — Create Two Profile Layers
**Action:** Restructure the IPS IG to define two explicit profile layers:
1. **IPS Core profiles** — minimal, reusable building blocks without document-specific constraints (e.g., no requirement for `subject.reference` to be a literal reference, relaxed cardinality where appropriate for non-bundle use).
2. **IPS Document profiles** — derived from Core profiles with additional document-specific constraints (e.g., `subject.reference` 1..1, Bundle membership requirements).

Update the IG narrative to clearly distinguish these layers and their intended use. Coordinate with the IPA project to evaluate whether IPS Core profiles could serve as the shared baseline.

**Rationale:** This directly addresses the reported ambiguity, has been discussed and endorsed in multiple WGM sessions and Zulip threads, and addresses concrete implementer pain points (e.g., EU cross-border project). The 2020 vote action item (gather justifying examples) has now been fulfilled through community feedback.

### Disposition B: Persuasive with Modification — Coordinate with IPA First
**Action:** Before restructuring IPS profiles, conduct a joint analysis with the IPA project team to:
1. Map IPS profile constraints against IPA profile constraints.
2. Identify which constraints are document-specific vs. clinically relevant.
3. Determine whether IPA's existing profiles can serve as the "core" layer, with IPS document profiles derived from them.
4. Produce a joint proposal for profile alignment.

Keep the ticket open and assign it as a work item for the IPS/IPA coordination effort.

**Rationale:** Both IPA and IPS have matured since the ticket was filed. A unilateral IPS restructuring risks creating a third parallel set of profiles. Joint coordination ensures alignment and avoids duplication. The 2022 Zulip discussion already identified this convergence path (Mikael Rinnetmäki: "IPS building on top of IPA").

### Disposition C: Considered for Future Use — Defer Pending IPS 2.0 / IPA Maturity
**Action:** Acknowledge the issue is valid but defer structural changes until the next major IPS version. In the interim:
1. Add IG narrative clarifying that IPS profiles carry document-specific constraints and may not be directly reusable outside a Bundle context.
2. Document the known tension and point implementers to IPA for non-document use-cases.
3. Track IPA maturity and revisit when both specs are ready for alignment.

**Rationale:** The IPS IG is in active use and a structural profile split is a breaking change for current implementers. Deferral to a major version allows proper planning, broader community input, and coordination with IPA's development timeline. Interim narrative improvements reduce implementer confusion at low cost.

---

## Recommendation

**Disposition B: Persuasive with Modification — Coordinate with IPA First**

This is recommended because:
1. **The problem is real and well-documented.** Multiple Zulip threads, WGM discussions, and concrete implementer experiences confirm the document-vs-library tension causes practical problems.
2. **Unilateral action risks fragmentation.** IPA already maintains parallel profiles. Restructuring IPS independently could worsen the situation.
3. **The convergence path is clear.** The 2022 Zulip discussion established broad consensus that IPA could serve as the "toolbox" baseline with IPS document constraints layered on top. A joint analysis is the logical next step.
4. **Actionable and scoped.** Unlike full deferral (Disposition C), this disposition moves the ticket forward with a concrete deliverable (joint IPA/IPS profile analysis) while avoiding the risk of a premature IPS-only restructuring (Disposition A).

The re-opening of this ticket in 2022, combined with the maturation of both IPA and IPS, makes this the right time to pursue coordinated alignment rather than further deferral.
