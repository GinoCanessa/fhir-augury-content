## Summary

This ticket requests that the CarePlan profile in the US Physical Activity IG be elevated to a "cornerstone" role by supporting references to all other profiles in the IG. The reporter (Gay Dolin) proposes that while referencing all profiles should not be required, each should be Must Support from the CarePlan. The ticket also asks for clarity on where physical activity outcomes are tracked, specifically via `CarePlan.activity.outcomeReference`.

The ticket was originally resolved as **Considered for Future Use** (vote: Lloyd McKenzie / Jay Lyle, 6-0-1) on 2023-06-15. The rationale was that the then-current version of US Core did not require systems to support these relationships, and not all workflows involve a formal CarePlan (e.g., direct referral to an exercise program). The workgroup agreed to revisit after initial implementation experience.

The ticket was **re-opened** post-STU publication on 2023-08-25 by Lynn Laakso, reverting the previous resolution for further consideration.

## Details

### Description

Care Plan by its definition should support referencing all of the other profiles in the US Physical Activity IG to "bring them all together," creating an aggregated comprehensive view of the patient's physical activity Care Plan. The reporter does not suggest requiring all references but recommends they all be Must Support from the CarePlan. The ticket also asks for clarity on where the outcomes (results of the Physical Activity Plan) are tracked within the CarePlan, specifically via `outcomeReference`.

Gay Dolin provided a reference to the MCC CarePlan profile as an example of this pattern: [MCC CarePlan StructureDefinition](https://build.fhir.org/ig/HL7/fhir-us-mcc/StructureDefinition-mccCarePlan.html).

## Proposed Dispositions

### Disposition A: Accept — Persuasive with Modification

**Action:** Add Must Support references from the PA CarePlan profile to all other profiles in the US Physical Activity IG (e.g., PA-related Observation profiles, ServiceRequest, Goal, Task, Condition). Add documentation clarifying how `CarePlan.activity.outcomeReference` should be used to track physical activity outcomes. Do not make populating all references required (i.e., keep cardinality at 0..*) but mark them Must Support so that systems claiming conformance must be able to handle them when present.

**Rationale:** The original vote (6-0-1) acknowledged the value of these relationships but deferred to a future release pending implementation experience. The ticket was explicitly re-opened post-STU publication, suggesting the workgroup believes it is now time to revisit. The MCC CarePlan profile already demonstrates this pattern, providing a proven model. Making references Must Support (not required) strikes a balance between encouraging comprehensive care plan documentation and not over-constraining simpler workflows (e.g., direct referral without a formal plan).

### Disposition B: Alternative — Considered for Future Use (Re-affirm Deferral)

**Action:** Re-affirm the original "Considered for Future Use" resolution. Document the intention to add Must Support references and outcome tracking in a future STU cycle, contingent on gathering implementation feedback from STU1 adopters.

**Rationale:** The original rationale for deferral remains valid: not all workflows involve a CarePlan (some clinicians go straight to referral), and US Core still has limited expectations for CarePlan relationship support. Without concrete implementation experience demonstrating demand for these cross-references, adding them as Must Support may increase implementer burden without proven benefit. A future cycle with real-world data would provide a stronger evidence base.

### Disposition C: Decline — Not Persuasive

**Action:** Decline the request. Keep the PA CarePlan profile focused on its current scope without adding Must Support references to all other IG profiles.

**Rationale:** Adding Must Support for all profile references from CarePlan could impose significant implementation burden on systems that use CarePlan in simpler contexts. The related ticket FHIR-39852 demonstrates that outcomeReference complexity in the MCC CarePlan already presents implementability challenges. Adding similar breadth to the PA IG may discourage adoption. Systems can already reference any resource from CarePlan.supportingInfo without profile-level constraints.

## Recommendation

**Disposition A: Persuasive with Modification** is recommended.

The ticket was explicitly re-opened post-STU publication, indicating the workgroup has determined sufficient time has passed to revisit. The original vote was strongly supportive (6-0-1) of the concept but deferred purely on timing grounds, not on merit. The MCC CarePlan profile provides a demonstrated pattern for this approach. Making all IG profile references Must Support (but not required) appropriately balances comprehensiveness with flexibility — implementers who do not use CarePlan are unaffected, and those who do gain a clear, documented set of linkages. Adding documentation for `outcomeReference` usage addresses a genuine gap in the IG's guidance.
