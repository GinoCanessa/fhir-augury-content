## Description

Projects are starting to use the Communication resource as a functional equivalent to MessageHeader, which creates confusion about the intended scope of each resource. The ticket requests that the FHIR specification provide clear guidance on the boundaries between Communication and MessageHeader (messaging), so implementers know when to use each.

**Examples cited:**
- **Da Vinci PDex** uses CommunicationRequest to ask for documentation to be shared, then Communication to record that the information was shared. Lloyd considers this acceptable because it operates at the provider level.
- **Da Vinci Alerts** uses Communication within a Bundle to send event-driven notifications (admission, discharge, lab results, prescription cancellation) — essentially replicating v2 ADT messages. This appears to overlap directly with the messaging paradigm.

The ticket observes that no guidance currently exists in MessageHeader, Communication, or the messaging section of the spec to differentiate these use cases.

## Analysis

### What Has Been Done

Significant progress has been made since this ticket was filed in 2019:

1. **WGM Discussion (Sept 2019):** Patient Care held a detailed working session establishing core principles — Communication is a paradigm-independent record of information disclosure; messaging is a framework for event-driven resource transport.
2. **Use Case Document (Feb 2020):** Lloyd drafted a Google Doc with use cases distinguishing Communication from messaging.
3. **FHIR-13389 (Published):** The earlier "Clarify Communication Scope and Usage" ticket was resolved with persuasive modification and published.
4. **FHIR-40214 (Applied):** Updated the Communication intro text from "does not represent the actual flow" to acknowledge it can represent the actual flow in some use cases (e.g., secure messaging).
5. **FHIR-28235 (Applied):** Further refined the intro text: "This resource is a record of a communication that has occurred or where the communication process has at least initiated."
6. **Communication wrapping section** added to the spec at `communication.html#wrapping`.

### What Remains

Despite these improvements, the original request for explicit **boundary guidance between Communication and MessageHeader** in the spec's Boundaries & Relationships sections has not been fully addressed. The intro text changes clarify Communication's scope, but there is no structured guidance telling implementers "use Communication when X; use MessageHeader/messaging when Y." The Zulip discussion and WGM notes identified this as the critical gap, and while a Google Doc was drafted, it's unclear whether that content has been incorporated into the normative spec.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Add explicit boundary guidance text to the Boundaries & Relationships sections of both Communication and MessageHeader, and optionally a note in the messaging infrastructure page. The text should:
- State that Communication is a clinical record of information disclosure, independent of transport mechanism.
- State that MessageHeader is part of the messaging infrastructure for event-driven, system-to-system transport of bundles of resources.
- Provide a brief decision table or criteria (e.g., "If the goal is to record that information was shared with a party, use Communication. If the goal is to route a bundle of resources to a system endpoint in response to an event, use messaging with MessageHeader.").
- Reference the existing Communication wrapping section for cases where Communication carries payload.

**Rationale:** The core request — add boundary language to the spec — is valid and partially but not fully met. The intro text changes (FHIR-28235, FHIR-40214) clarify Communication's nature but do not address MessageHeader's side or provide comparative guidance. Implementers continue to need explicit direction.

### Disposition B: Persuasive (Already Applied)
**Action:** Consider this ticket substantially addressed by the combination of FHIR-13389 (published), FHIR-28235 (applied), FHIR-40214 (applied), and the new Communication wrapping section. Close as resolved.

**Rationale:** The spirit of the request has been addressed incrementally over multiple tickets and spec updates. The Communication intro now acknowledges both record and flow use cases, and the wrapping section provides additional guidance. While explicit B&R comparison text could be better, the existing changes significantly reduce the ambiguity.

### Disposition C: Not Persuasive
**Action:** Close without additional changes. The current spec text, combined with the resolved related tickets, provides adequate guidance.

**Rationale:** The overlap between Communication and MessageHeader is inherent in the resource model and cannot be fully resolved by text alone. The spec already states Communication is a "record of communication" and the messaging section describes its infrastructure purpose. Adding further prescriptive text risks over-constraining legitimate use cases (e.g., Da Vinci PDex).

## Recommendation

**Disposition A: Persuasive with Modification.**

The original ticket's central ask — explicit boundary guidance between Communication and MessageHeader in the spec — remains unmet despite significant related progress. The intro text changes from FHIR-28235 and FHIR-40214 clarify Communication's scope but do not provide the comparative "when to use which" guidance that implementers need. The Zulip thread and WGM notes show strong community consensus that this guidance is needed. Adding concise B&R text to both Communication and MessageHeader, potentially drawing from Lloyd's 2020 Google Doc and the 2019 WGM discussion outcomes, would close the remaining gap with minimal effort.
