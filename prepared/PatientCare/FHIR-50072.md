## Summary

The SDOHCC Coordination Platform CapabilityStatement contains internal contradictions in conformance expectations. Multiple resources (CareTeam, Location, Questionnaire, QuestionnaireResponse, RelatedPerson) are listed as **SHALL support** with search parameters also marked **SHALL**, yet the `search-type` interaction is only **MAY** or **SHOULD**. This creates an unresolvable conformance gap: a system cannot meaningfully claim it SHALL support a search parameter when the search interaction itself is not required. The submitter asks what the expected behavior is for systems claiming conformance and how they can demonstrate support for mandatory search parameters when search-type is optional. A follow-up comment confirms the issue persists in STU 2.3.

---

## Details

### Description

Within the Coordination Platform CapabilityStatement, there are resources that are listed as SHALL support, with a search parameter listed as SHALL support, but an interaction summary where `search-type` is less than SHALL (MAY or SHOULD).

**Example — CareTeam:**
- Resource Conformance: **SHALL**
- Interaction Summary: **MAY** support read, search-type
- Search Parameter `_id` Conformance: **SHALL**

The submitter raises three questions:
1. What is the expected behavior of a system that claims conformance?
2. How does a system demonstrate that they support a resource without supporting read or search-type?
3. How does a system demonstrate that they support the search by `_id` without supporting the search-type interaction?

The same inconsistencies exist for **Location**, **Questionnaire**, **QuestionnaireResponse**, and **RelatedPerson** in this CapabilityStatement.

## Proposed Dispositions

### Disposition A: Persuasive — Align Interaction Conformance Upward to Match Search Parameters

**Resolution:** Upgrade the `search-type` and `read` interaction conformance to **SHALL** for CareTeam, Location, Questionnaire, QuestionnaireResponse, and RelatedPerson in the Coordination Platform CapabilityStatement, so that interaction expectations match the resource-level and search-parameter-level SHALL requirements.

**Rationale:** If a resource and its search parameters are SHALL, the interactions needed to exercise those search parameters must also be SHALL. This is the most internally consistent resolution and aligns with how the sibling tickets (FHIR-51138, FHIR-51139, FHIR-51140) were resolved as Persuasive. This approach makes the CapabilityStatement actionable for implementers and testable for conformance programs.

**Impact:** Updates the Coordination Platform CapabilityStatement interaction summaries for five resources. No structural changes to profiles or search parameter definitions. Implementers already supporting these search parameters would already support the interactions.

### Disposition B: Persuasive with Modification — Downgrade Search Parameter Conformance to Match Interactions

**Resolution:** Downgrade the search parameter conformance from **SHALL** to **MAY** or **SHOULD** for the affected resources, matching the current interaction conformance levels.

**Rationale:** If the workgroup intentionally set `search-type` to MAY/SHOULD (perhaps because these resources are primarily accessed through other mechanisms like references or includes), then the search parameters should be relaxed to match. This preserves the original interaction-level intent.

**Impact:** Reduces the conformance burden on implementers for these resources. May weaken interoperability expectations around searchability of these resources on the Coordination Platform.

### Disposition C: Not Persuasive — No Change Needed

**Resolution:** Decline the change request. Document that conformance expectations at different levels (resource, interaction, search parameter) are intentionally independent, and that a SHALL search parameter indicates the parameter must be supported *if* the search-type interaction is implemented.

**Rationale:** The FHIR conformance framework technically allows different conformance levels at each layer. A system that supports search-type (even at MAY) must use the SHALL search parameters. The CapabilityStatement is internally consistent under this interpretation.

**Impact:** No specification changes. However, this leaves an implementer-facing ambiguity that has been raised by multiple submitters, and conflicts with the Persuasive resolution already applied to the closely related sibling tickets.

## Recommendation

**Disposition A: Persuasive — Align Interaction Conformance Upward to Match Search Parameters**

This is the recommended disposition because:

1. **Consistency with precedent:** The closely related tickets FHIR-51138, FHIR-51139, and FHIR-51140 — all addressing the same class of conformance inconsistencies in the same CapabilityStatement — have already been resolved as Persuasive.
2. **Implementer clarity:** The current state is confusing and untestable. Aligning interaction conformance with resource and search parameter conformance provides clear, actionable expectations.
3. **Logical coherence:** A SHALL search parameter is meaningless without a corresponding SHALL interaction to exercise it. Aligning upward resolves this logical contradiction.
4. **Scope alignment:** The affected resources (CareTeam, Location, Questionnaire, QuestionnaireResponse, RelatedPerson) are already marked SHALL at the resource level, suggesting they are considered essential. Making their interactions also SHALL is the natural conclusion.
