## Description

The submitter requests that the PACIO Functional Status IG create a specific value set containing the functional domains identified in section GG (CMS Minimum Data Set section for Functional Abilities and Goals), rather than binding `Observation.code` to the entire LOINC code system. The proposed binding strength would be example or extensible.

**Existing Wording:** Code | A required field with 'extensible' binding to the LOINC code system (http://loinc.org/).

## Analysis

This ticket was originally filed in December 2020 as part of a batch of PACIO Functional Status STU1 ballot comments. It was initially resolved as "Considered for Future Use" with a 6-0-1 vote in August 2021, because no suitable value set existed for Functional Status `Observation.code` at that time. The ticket was re-opened in November 2021 for post-STU1 reconsideration and has remained in "Triaged" status since.

The core concern is valid: binding `Observation.code` to the entire LOINC code system provides no meaningful constraint for implementers. A curated value set of LOINC codes corresponding to section GG functional domains (self-care, mobility, etc.) would improve interoperability and validation.

The companion tickets (FHIR-30038, -30040, -30042) addressing the `Observation.category` binding were successfully resolved, suggesting the work group has engaged with the broader PACIO Functional Status terminology needs. However, the `Observation.code` value set work that was being deferred to an HL7 group has not visibly progressed based on available data.

This ticket has been open for over 4 years with no assignee and no recent activity since November 2021.

## Proposed Dispositions

### Disposition A: Persuasive with Modification
**Action:** Create a curated value set of LOINC codes representing the section GG functional domains (e.g., self-care, mobility, eating, oral hygiene) and bind `Observation.code` to it with **extensible** binding strength in the PACIO Functional Status profiles.

**Rationale:** This directly addresses the submitter's request and improves interoperability. Extensible binding allows implementers to use codes outside the set when needed while providing a clear starting point. The companion category tickets were already resolved with modifications, and this aligns with those changes.

**Trade-offs:** Requires terminology work to identify and curate the appropriate LOINC codes. The value set will need ongoing maintenance as section GG evolves.

### Disposition B: Considered for Future Use
**Action:** Defer the ticket again, pending completion of the HL7 Functional Status value set work mentioned in the 2021 comments.

**Rationale:** The original deferral rationale (no suitable value set exists yet) may still apply. If the HL7 terminology group is actively working on a Functional Status value set, it would be better to wait for that authoritative set rather than create an interim one.

**Trade-offs:** The ticket has already been deferred once and has been open for over 4 years. Continued deferral without clear progress on the upstream value set risks indefinite stalling. There is no visible evidence of progress on the referenced HL7 group's work.

### Disposition C: Not Persuasive
**Action:** Close the ticket as not persuasive, keeping the current binding to the full LOINC code system.

**Rationale:** LOINC is already the correct code system for these observations, and extensible binding to the full system gives implementers maximum flexibility. Profile-level constraints can be documented in narrative guidance without a formal value set.

**Trade-offs:** This contradicts the community consensus that the binding is too broad. Narrative guidance alone does not enable automated validation, and implementers would continue to lack clear direction on which LOINC codes are appropriate for functional status observations.

## Recommendation

**Disposition A: Persuasive with Modification** — The request is well-founded and aligns with resolved companion tickets. The 4+ year deferral without visible progress on an upstream value set suggests the PACIO community should take ownership of curating a value set of section GG LOINC codes with extensible binding. This provides concrete interoperability improvement while still allowing flexibility through the extensible binding strength.
