## Summary

The reporter identifies that the word **"limiting"** in the element definition for `MedicationRequest.medication[x]` and `MedicationStatement.medication[x]` in the IPS profiles is misleading. The current wording — *"limiting the usage of the medicationCodeableConcept only to the cases in which no other information than a simple code is available"* — reads as a conformance constraint rather than a recommendation. The request is to reword the text to make clear this is guidance, not a rule.

## Details

### Description

The current element definition text for `medication[x]` in both MedicationRequest(IPS) and MedicationStatement(IPS) reads:

> Identifies the medication being administered or the reason for absent or unknown Medication. This is either a link to a resource representing the details of the medication or a simple attribute carrying a code. To improve global interoperability, it is strongly encouraged that the reference to a medication resource is used, **limiting** the usage of the medicationCodeableConcept only to the cases in which no other information than a simple code is available.

The reporter notes that "limiting" implies a conformance requirement or constraint, and suggests rewording to clarify this is a recommendation — e.g., changing to *"medicationCodeableConcept is only suggested for cases"*.

### Comments

**JIRAUSER22800 (2025-11-25):** Discussed on the IPS Call Nov-25. The group confirmed the intent is **not** to constrain the use of `medicationCodeableConcept`. While the term "limiting" was not universally concerning, the group agreed "recommended" may be better wording. Discussion also clarified what "simple code" means — the intent relates to when there is no structured medication definition (form, ingredients, strength) as per ISO 27269:2021 Table 17.

Two alternative rewordings were proposed:

1. *"To improve global interoperability, it is strongly encouraged that the reference to a medication resource is used, **with the use** of medicationCodeableConcept **recommended** only in cases in which no other information than a simple code is available."*

2. *"To improve global interoperability, it is strongly encouraged that the reference to a medication resource is used, it is **recommended** that the usage of the medicationCodeableConcept is **only used in** cases **where** no information **other** than a **medication name and** code is available."*

**gdolin (2025-12-03):** Noted that the ISO spec (ISO 27269) must be reviewed to ensure the clarifying wording does not conflict, and to check if any changes are needed in ISO.

## Proposed Dispositions

### Disposition A: Accept — Reword using "recommended" and clarify "simple code"

Accept the change request and adopt revised wording that replaces "limiting" with "recommended" and clarifies what "simple code" means. The second alternative proposed on the Nov 25 IPS call provides the clearest language:

> *"To improve global interoperability, it is strongly encouraged that the reference to a medication resource is used. It is recommended that the usage of the medicationCodeableConcept is only used in cases where no information other than a medication name and code is available."*

This wording:
- Removes the confusing "limiting" language
- Replaces it with clearly advisory "recommended" phrasing
- Clarifies "simple code" to mean "medication name and code"
- Applies to both `MedicationRequest.medication[x]` and `MedicationStatement.medication[x]` in IPS profiles

**Precondition:** The revised wording must be checked for compatibility with ISO 27269:2021, as flagged by gdolin.

### Disposition B: Accept with minimal change — Replace only "limiting" with "with the use...recommended"

Accept with narrower scope, using the first alternative proposed on the Nov 25 call:

> *"To improve global interoperability, it is strongly encouraged that the reference to a medication resource is used, with the use of medicationCodeableConcept recommended only in cases in which no other information than a simple code is available."*

This preserves the existing "simple code" phrasing (avoiding scope creep into defining what constitutes a simple code) while removing the misleading "limiting" language. This is a more conservative edit that reduces the risk of unintended conflicts with ISO 27269.

### Disposition C: Decline — Current wording is adequate

Decline the change on the basis that "limiting" in context is understood as guidance rather than a conformance constraint. The preceding phrase "it is strongly encouraged" already establishes the advisory tone. No conformance language (e.g., SHALL, SHOULD, MUST) is used. The IPS call participants noted the term was not universally concerning. Changing the wording risks introducing inconsistency with the ISO specification without a clear conformance issue.

## Recommendation

**Disposition A: Accept — Reword using "recommended" and clarify "simple code"**

The ticket identifies a genuine ambiguity: "limiting" can reasonably be read as imposing a constraint, which is not the intent. The IPS call discussion confirmed the group's intent is advisory, and there is consensus that "recommended" better captures this. Disposition A's wording also addresses the secondary concern raised in discussion — that "simple code" is unclear — by replacing it with "medication name and code." This provides the most complete fix. The ISO 27269 compatibility check flagged by gdolin should be completed before applying the change, but should not block acceptance of the disposition direction.
