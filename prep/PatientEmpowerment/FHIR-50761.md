## Summary

The reporter (Virginia Lorenzi) requests a wording change in the Personal
Health Record (PHR) FHIR Implementation Guide to remove ambiguity in a
normative ("MUST") statement about declaring API endpoints. The current
sentence — *"Systems MUST post the API endpoints they use in the system's
CapabilityStatement."* — is ambiguous about which systems the requirement
applies to and what "post" / "use" mean in context. The proposal is to
reword the requirement to make the actor explicit (PHR systems following
the PHR FHIR IG) and to clarify both the action ("include") and the
target artifact ("the PHR's FHIR server's CapabilityStatement"), while
narrowing the scope of endpoints to those the PHR is *exposing* (rather
than the broader and looser "use").

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items recorded for FHIR-50761 in the local index. Keyword-similar PHR
clarity tickets (e.g., FHIR-50763 "Use of word record confusing — recommend
clarity" and FHIR-50737 "Clarify the purpose of phr format") suggest this
ticket is part of a broader editorial pass on the PHR IG by the same
reporter, but they are not formally linked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the sentence

> "Systems MUST post the API endpoints they use in the system's
> CapabilityStatement."

in the PHR FHIR Implementation Guide with

> "PHR systems following the PHR FHIR Implementation Guide MUST include
> the API endpoints they are exposing in the PHR's FHIR server's
> CapabilityStatement."

Concretely this is an editorial change to the PHR IG narrative (the
section that currently carries the quoted sentence — likely the
"Endpoints" / "Conformance" or PHR server actor section). No structural
change is required: no new resources, profiles, extensions, or
CapabilityStatement constraints are introduced. The normative force
("MUST") is preserved; only the actor, the verb, and the target artifact
are tightened. The reviewer/editor should also confirm any cross-page
references to the original sentence are updated for consistency.

#### Justification

The reporter's wording removes three concrete ambiguities in the original
sentence:

1. **Actor.** "Systems" is unscoped; the IG only governs PHR systems
   conforming to this IG. Naming the actor explicitly aligns with FHIR
   conformance-language conventions, where normative statements should
   identify the actor that bears the obligation.
2. **Action.** "Post" is overloaded in FHIR (the HTTP verb, the act of
   publishing, etc.). "Include" is unambiguous for what is being asked:
   the endpoints appear *in* the CapabilityStatement instance.
3. **Scope of endpoints.** "Endpoints they use" can be read to include
   third-party endpoints the PHR consumes; "endpoints they are exposing"
   correctly limits the requirement to the PHR's own server-side
   endpoints, which is what a CapabilityStatement is intended to
   describe.

This is a low-risk, editorial change. It does not alter conformance
behavior — a system that already meets the original intent will meet the
revised text — and it reduces the risk of inconsistent implementation
based on differing readings of "post" and "use".

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the spirit of the request but tighten the wording further and tie
the sentence to a machine-checkable conformance hook rather than relying
solely on prose. Suggested text:

> "A PHR server conforming to this Implementation Guide **SHALL** publish
> a `CapabilityStatement` (via `GET [base]/metadata`) that declares every
> FHIR API endpoint it exposes. The `CapabilityStatement.rest.resource`
> and `CapabilityStatement.rest.operation` entries SHALL describe the
> interactions and operations available on those endpoints."

Pair this prose change with one or both of:

- An explicit reference to the PHR server CapabilityStatement artifact
  defined (or to be defined) in the IG, so the requirement is anchored
  to a named conformance resource rather than a free-form sentence.
- A brief note clarifying that endpoints the PHR *consumes* (e.g.,
  external EHR endpoints it queries) are out of scope for this
  requirement and, if needed, are described elsewhere (e.g., via
  `Endpoint` resources or implementation notes).

#### Justification

Disposition A fixes the stated ambiguity but leaves two latent issues
the reviewer should weigh:

- "MUST" vs **SHALL**: HL7 conformance language convention is to use
  "SHALL" (RFC 2119 with HL7 profile) in normative IG text. If the rest
  of the PHR IG already uses MUST consistently, keep MUST; if it uses
  SHALL, this is a good moment to align.
- The original sentence does not explicitly bind the requirement to
  `CapabilityStatement.rest.resource` / `.rest.operation`, which is
  where API endpoints are actually conveyed. Tying the prose to those
  elements (and to the `metadata` endpoint that serves the
  CapabilityStatement) gives implementers and testers a concrete
  conformance target, which matches how Patient Empowerment IGs (and
  related US Core / SMART work) typically frame server obligations.

Trade-off vs. A: B is a slightly larger editorial change and may
overlap with adjacent sections of the IG (so it costs more reviewer
time), but it produces a sentence that is both unambiguous *and*
testable, which has more long-term value than a minimal rewording.

---

### Disposition C: Decline

#### Proposal

Decline the change as a not-substantive editorial-only request and
defer it to a routine editorial pass of the PHR IG, on the grounds that
the existing sentence — read in the context of the surrounding PHR IG
chapter on conformance — already conveys the intent (a PHR server
declares its own endpoints in its CapabilityStatement), and that
reopening the wording risks unintended scope changes (e.g., implying
that endpoints the PHR *consumes* are out of scope of any declaration
mechanism, when in fact the IG may want to address those separately).

#### Justification

A decline would rest on three points:

1. **Risk of unintended narrowing.** Replacing "endpoints they use"
   with "endpoints they are exposing" is a meaningful semantic change,
   not purely editorial. If the PHR IG ever wants to require declaration
   of endpoints the PHR *consumes* (e.g., for trust / discovery), the
   new wording forecloses that reading.
2. **Existing FHIR conformance machinery.** `CapabilityStatement` is
   already, by definition, a description of a server's exposed
   capabilities; a reader familiar with FHIR will read the original
   sentence correctly. Editorial drift can be addressed in a periodic
   cleanup rather than per-sentence tickets.
3. **Cost / value.** Changing one sentence triggers re-publication of
   the IG and reviewer time across the WG; declining preserves
   bandwidth for substantive PHR issues.

This justification is included for completeness; it is the weakest of
the three because the reporter's underlying complaint (ambiguity of
"systems" and "post") is real.

---

### Recommendation

**Recommended disposition:** A (with a friendly nudge toward B's
SHALL/elements refinement during editorial review)

The ticket is a small, well-scoped editorial fix from a Patient
Empowerment co-chair / regular contributor (Virginia Lorenzi), filed
against an IG (PHR FHIR) that is still maturing and where conformance
language matters disproportionately for early implementers. The
ambiguities the reporter flags — unspecified actor, overloaded verb
"post", and the looser "endpoints they use" — are real and have a
non-trivial chance of being misread by implementers who are not deeply
embedded in FHIR conventions. Disposition A removes those ambiguities
with essentially zero implementation risk and no structural change to
the IG.

Disposition B is materially better in the long run because it ties the
requirement to specific CapabilityStatement elements and aligns with
HL7's "SHALL" convention, but it expands the scope of the change beyond
what the reporter asked for and may need a quick check that the
surrounding PHR IG chapter is consistent. The pragmatic path is:
**accept A as the disposition for this ticket**, and ask the editor to
consider B's refinements opportunistically when they touch the
surrounding section. Disposition C is not recommended — the reporter's
complaint is legitimate and the cost of the fix is trivial.
