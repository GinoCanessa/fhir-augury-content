## Summary

The reporter offers a set of editorial / completeness suggestions on
table 5.1.1 of the Personal Health Record (PHR) FHIR IG, asking whether
several FHIR resources and a hooks specification that appear to be
missing should be added to specific rows of that table:

- In **PH.2.5.2**, whether **MedicationRequest** should be listed.
- In **PH 5.1**, whether **PlanDefinition** and **CarePlan** should be listed.
- In **PH 5.2**, whether **CDS-Hooks**, **MedicationRequest**, and
  **MedicationAdministration** should be listed.

The single comment on the ticket (from awatson1978, the PHR IG's
maintainer) finds the suggestions persuasive and indicates a
willingness to incorporate them as technical corrections, also noting
ongoing work to merge in the **mCode Data Element Dictionary**, which
the commenter considers aligned with this request. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were found by the augury
indexes for `FHIR-49271`.

### Disposition A: Accept As Requested

#### Proposal

Apply the suggested additions to table 5.1.1 of the Personal Health
Record (FHIR) IG as a technical correction:

- **PH.2.5.2:** add **MedicationRequest** to the listed FHIR resources.
- **PH 5.1:** add **PlanDefinition** and **CarePlan** to the listed
  FHIR resources.
- **PH 5.2:** add **CDS-Hooks** (as a referenced specification),
  **MedicationRequest**, and **MedicationAdministration** to the
  listed FHIR resources.

Update any introductory prose that enumerates the resources covered by
each section to match the new table contents, and confirm the cells in
adjacent columns (e.g., cardinality / must-support / notes columns, if
any) follow the table's existing convention for newly-added rows.

#### Justification

The change is scoped as a Technical Correction by the reporter,
acknowledged as persuasive by the IG maintainer, and is internally
consistent with FHIR's medication and care-planning resource families:
Medication-related sections that already cover medication
representation are incomplete without `MedicationRequest` /
`MedicationAdministration`, and a care-planning section that omits
`PlanDefinition` and `CarePlan` is missing the two resources most
directly responsible for representing planned care. Adding CDS-Hooks
to PH 5.2 aligns the table with the broader decision-support tooling
that PHR consumers are likely to encounter.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substantive additions but defer the table edit to the same
release cycle in which the **mCode Data Element Dictionary** merge
mentioned by the maintainer lands. In the interim:

1. File a tracking note on the PHR IG project board referencing this
   ticket.
2. Land the `MedicationRequest`, `MedicationAdministration`,
   `PlanDefinition`, `CarePlan`, and `CDS-Hooks` additions as part of
   a single, larger restructure of section 5.1.1 driven by the mCode
   data-element merge, so the table is normalized once instead of
   churned twice.
3. Treat this ticket as "Persuasive — Will Be Addressed at a Later
   Date" against that larger work item.

#### Justification

The maintainer has explicitly flagged that merging the mCode Data
Element Dictionary is in flight and "aligns with this ticket". A
larger structural pass is likely to re-shape table 5.1.1 anyway;
performing two edits — first the targeted additions here, then a
broader restructuring — risks merge churn, redundant review, and
inconsistent row formatting if the mCode merge introduces additional
columns or row groupings. Bundling the change reduces total review
overhead and keeps the table's evolution coherent.

---

### Disposition C: Decline

#### Proposal

Decline the request as written, on the grounds that table 5.1.1 in the
PHR IG is intended as a curated, opinionated summary of the
**minimum** resource set a PHR is expected to surface, not an
exhaustive enumeration of every FHIR resource that *could* appear in
the corresponding section. Direct the reporter to existing FHIR
resource-relationship guidance (e.g., the Medications module overview
and the Care Provision module overview) for the broader resource
landscape, and close as "Not Persuasive".

#### Justification

If table 5.1.1 is intentionally minimal, expanding its rows blurs the
distinction between "core PHR-relevant resources" and "all
medication-/care-related resources", which can mislead implementers
about what is actually required. Routing readers to the upstream
module overviews avoids duplicating that information in the PHR IG
and reduces ongoing maintenance burden as new resources are added in
future FHIR versions.

---

### Recommendation

**Recommended disposition:** B

The substance of Disposition A is correct — the resources called out
by the reporter genuinely belong in their respective rows, and the IG
maintainer has already signaled agreement, which makes Disposition C
hard to defend on the merits. However, the maintainer's comment also
makes clear that a larger mCode Data Element Dictionary merge is
imminent and is expected to touch this same table. Bundling the
targeted additions into that merge (Disposition B) delivers the same
outcome the reporter is asking for, avoids two rounds of review and
publication churn on table 5.1.1, and lets the workgroup normalize
row formatting once. If the mCode merge slips significantly, the
workgroup can fall back to Disposition A and apply the edits as a
standalone technical correction.
