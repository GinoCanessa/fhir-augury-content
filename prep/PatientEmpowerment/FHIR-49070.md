## Summary

The reporter (Craig Newman) is asking the Patient Empowerment work group to clarify a normative-style statement in **Section 2.1** of the Personal Health Record (FHIR) IG. The text in question reads: *"Systems MUST use FHIR data schemas to claim to be compliant with this IG."* The reporter finds this confusing for three reasons: (1) the surrounding paragraph mixes long-term storage, over-the-wire transfer, and `.sphr` file creation/consumption without saying which the MUST applies to; (2) the immediately preceding sentence says the IG *recommends* implementors treat storage like over-the-wire transfer, which conflicts with a MUST; and (3) the Getting Started page explicitly says the IG does not prescribe how to program software, which would seem to include storage. The reporter is asking for a clarifying edit, not necessarily a substantive policy change. No related Jira tickets, Zulip threads, or GitHub items were surfaced by cross-reference or search.

## Details

**Description:**

Section 2.1 states:

> As such, this implementation guide recommends that implementors treat storage in much the same way as over-the-wire data transfers.
>
> - Systems MUST use FHIR data schemas to claim to be compliant with this IG.
> - Systems SHOULD use the same MIME types when possible.
> - Systems MAY treat directories as Bundle entries or NDJSON lines by default.

The MUST use of FHIR is confusing given a couple of things:

1. It's not clear if this is in reference to long term storage of data, over the wire interfaces or simply the creation/consumption of `.sphr` files all of which are mentioned in the immediate vicinity.
2. The statement immediately before includes "recommends" which is at odds with any MUST requirement.
3. The Getting Started page clearly states that the IG doesn't prescribe how to program software which would seem to include a "storage" aspect.

Please clarify the intent of using FHIR data schemas.

## Proposed Dispositions

The dispositions below are scoped to a narrative/conformance-language clarification in **Section 2.1** of the Personal Health Record (FHIR) IG (and, where relevant, the Getting Started page). No GitHub repo briefing is available, so no specific authoring path is named; the workgroup should map the change to the IG source repo it maintains.

### Disposition A: Accept As Requested

#### Proposal

Persuasive. Rewrite Section 2.1's storage-vs.-wire paragraph and bullet list to:

1. Replace the conflicting "recommends … MUST" framing with a single, scoped conformance statement. Concretely, change the lead sentence to something like: *"This implementation guide treats persisted artifacts (including `.sphr` files) as semantically equivalent to over-the-wire FHIR exchanges. The following conformance rules apply to the FHIR resources a system produces, persists, or consumes when claiming conformance to this IG; they do not constrain a system's internal data model or storage engine."*
2. Tighten the bullets so each is unambiguous about *what* is being constrained:
   - "Systems **SHALL** represent FHIR resources, whether transmitted over-the-wire or persisted in `.sphr` files, using the FHIR data schemas (XML, JSON, or RDF as defined by FHIR R4/R5) when claiming conformance to this IG." (Use SHALL rather than MUST to align with FHIR's house style.)
   - "Systems **SHOULD** use the FHIR-defined MIME types (`application/fhir+json`, `application/fhir+xml`) when serializing those representations."
   - "Systems **MAY** treat directories as `Bundle` entries or NDJSON lines by default when packaging multiple resources."
3. Add a short cross-reference / footnote on the Getting Started page noting that "we do not prescribe how to program software" applies to internal implementation choices, not to the externally-observable resource representations governed by Section 2.1.

#### Justification

The reporter's three concerns are factually correct: the paragraph as written mixes scopes (storage, wire, `.sphr` files), uses "recommends" and "MUST" in adjacent sentences, and contradicts the Getting Started promise about not prescribing how software is programmed. The fix is purely editorial — it does not change any normative requirement, only the scope and verbiage. Aligning the bullets with FHIR's standard SHALL/SHOULD/MAY terminology (per the FHIR conformance rules and RFC 2119) is consistent with the rest of the FHIR specification and with Patient Empowerment's other IGs. The assignee's existing comment ("leaning towards persuasive … will clarify in upcoming versions") signals the work group already views this direction as plausible.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with modification. Instead of rewriting Section 2.1 in place, restructure the conformance content into two clearly-labelled subsections:

- **2.1.1 Conformance scope** — a one-paragraph statement that conformance to this IG is asserted at the level of FHIR resource *representations* (on the wire and in `.sphr` files), and that the IG does not constrain internal storage formats, databases, or programming approaches.
- **2.1.2 Representation rules** — the SHALL/SHOULD/MAY bullets, each prefaced by "When producing, transmitting, or persisting resources for conformance purposes, …".

Additionally, define `.sphr` (the file extension referenced in the surrounding text) in a glossary entry and link to it from both Section 2.1 and Getting Started, so readers don't have to infer the relationship between files, wire, and storage from context.

#### Justification

The same underlying confusion is addressed, but by partitioning *scope* from *rules* rather than splicing both into one paragraph. This makes future tickets ("does this MUST apply to my database?") less likely, and it gives implementers a stable anchor (`#conformance-scope`) to cite when explaining their compliance posture. The trade-off vs. Disposition A is more editorial work and a slight reorganization of section anchors (which may invalidate existing inbound links). Given the IG is still pre-publication, the link-stability cost is low and the long-term clarity benefit is high.

---

### Disposition C: Decline

#### Proposal

Not Persuasive. Leave Section 2.1 as written, noting in the resolution that "MUST use FHIR data schemas" is a long-standing conformance idiom across HL7 IGs and that the surrounding `.sphr` / wire / storage discussion is intentionally inclusive — the MUST applies whenever the system is *claiming conformance*, regardless of channel.

#### Justification

This disposition is defensible only if the work group concludes (a) that no real implementer has been confused by the wording in practice, and (b) that adding scope language risks weakening a deliberate normative requirement. It is the weakest of the three options here because the assignee has already publicly acknowledged the feedback as constructive, and because the cited contradiction with the Getting Started page is real and on the page — not speculative. Declining without addressing that contradiction would leave a self-inconsistent specification.

---

### Recommendation

**Recommended disposition:** A

The change requested is a narrative clarification, not a normative shift. Disposition A delivers exactly what the reporter asked for, fixes a real internal contradiction (the "recommends … MUST" pairing and the Getting Started conflict), and aligns the bullets with FHIR's standard SHALL/SHOULD/MAY conformance vocabulary — all with minimal risk to implementers. The assignee's existing comment already signals receptiveness to a persuasive resolution. Disposition B is a reasonable upgrade if the work group has appetite for a small structural reorganization, and can be folded into A as a stretch goal during the actual edit; if the work group prefers to keep the change tight and ship it in the next publication, A alone is the cleanest path.
