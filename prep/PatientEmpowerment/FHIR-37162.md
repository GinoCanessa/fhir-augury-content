## Summary

The ticket asserts that the canonical URL of the Patient Corrections IG
ValueSet `PatientCorrectionBusinessStatusVS` is anchored under
`hl7.org/fhir/uv/patient-corrections/...` when it should instead be anchored
under `terminology.hl7.org` (THO). The reporter (importing Comment 5 from
Ron G. Parker) cites a Terminology Authority (TSMG) policy: only value sets
that are bound as `required` to a `code` data type should keep a canonical
under `hl7.fhir.org`; everything else should live in THO, with a UTG ticket
filed to register the value set there. The change request is targeted at the
Patient Corrections IG, owned by Patient Empowerment, but the action it
requests is a TSMG/UTG-pattern fix rather than a content change to the IG's
clinical model.

The ticket was previously dispositioned **Considered for Future Use**
(2022-10-17, vote 7-0-0) on the rationale that further code changes should
settle before re-anchoring. Lynn Laakso reverted that resolution on
2025-07-10 and explicitly **re-opened it post STU1 publication**, so it now
needs a fresh disposition for the next Patient Corrections cycle.

No cross-referenced Jira tickets, Zulip threads, Confluence pages, or
GitHub items were found in the FHIR Augury index for `FHIR-37162`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor `PatientCorrectionBusinessStatusVS` (and any other Patient
Corrections value sets that do not satisfy the "required binding to `code`"
exception) under THO:

1. File a companion **UTG ticket** to register
   `PatientCorrectionBusinessStatusVS` in THO at
   `http://terminology.hl7.org/ValueSet/patient-correction-business-status`
   (or the current TSMG-approved naming convention).
2. In the Patient Corrections IG, change the value set's canonical `url`
   element from
   `http://hl7.org/fhir/uv/patient-corrections/ValueSet/PatientCorrectionBusinessStatusVS`
   to the new THO canonical, retire the IG-local definition, and import
   the THO value set instead.
3. Update every `binding.valueSet` reference in profiles/extensions of
   the Patient Corrections IG that points at the old canonical to point
   at the new THO canonical.
4. Sweep all other value sets owned by the IG and apply the same rule:
   keep IG-local only those that are `required`-bound to a primitive
   `code`; move everything else (including `extensible`/`preferred`/
   `example`-bound and CodeableConcept-bound sets) to THO.
5. Add `IG.ini` / dependency entries pulling in the appropriate `hl7.terminology`
   package version and add an errata/changelog note explaining the canonical
   change for downstream implementers.

#### Justification

This is the literal request and aligns with TSMG/Vocabulary policy as
quoted in the ticket: only `required`-bound `code` value sets should live
under `hl7.org/fhir`. Re-anchoring in THO improves cross-IG reuse,
prevents the IG from "owning" terminology that should be shared, and
brings Patient Corrections in line with current HL7 publishing practice
(the same pattern most active IGs have adopted since 2022). Lynn Laakso's
2025-07-10 re-open explicitly framed this as a post-STU1 cleanup, so the
prior "wait until code changes settle" objection no longer applies.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the THO-anchoring **rule** for the IG, but stage it: leave the
current canonical in place for STU1 errata to avoid breaking deployed
implementations, and require all *new or changed* value sets in
post-STU1 / STU2 to be authored in THO from the start. Specifically:

1. Mark `PatientCorrectionBusinessStatusVS` as **deprecated in place** in
   the next errata; add a `replacedBy` extension/note pointing to the
   future THO canonical.
2. File the UTG ticket now so the THO canonical is registered ahead of
   STU2 ballot.
3. In STU2, retire the IG-local canonical and switch all bindings to the
   THO canonical (i.e., perform the Disposition-A change set then).
4. Add a project-level rule (IG QA / sushi-config note) that any new
   value set added after this disposition must be created in THO unless
   it qualifies for the `required`+`code` exception.

#### Justification

A canonical URL change is a breaking change for every implementer that
has hard-coded the existing canonical, validated against it, or cached
the expansion. Patient Corrections published STU1 with the current
canonical, so silently replacing it (Disposition A) risks breaking
external validators, terminology servers, and FHIR clients mid-cycle.
Staging the change across an errata + STU2 boundary preserves
implementer trust while still satisfying the underlying TSMG policy.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Mod / Not Persuasive — keep the IG-local
canonical** on the grounds that `PatientCorrectionBusinessStatusVS` is a
Patient-Corrections-specific workflow vocabulary (business status of a
correction request) with no demonstrated reuse outside this IG, and that
the TSMG rule cited is a *guideline* rather than a normative
constraint. Document the decision in the IG's terminology section and
move on.

#### Justification

Defensible only if (a) the value set is genuinely IG-internal with no
prospect of reuse, (b) the cost of breaking the published STU1 canonical
outweighs the consistency benefit, and (c) the workgroup is willing to
accept a TSMG QA finding against the IG. Given TSMG's explicit guidance
in the ticket and Lynn Laakso's 2025-07-10 re-open, this position is
hard to defend; it would essentially be telling TSMG "we know, but not
this one."

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach — staged
re-anchor across errata + STU2).**

The technical merit of A is sound and matches TSMG policy verbatim, but
A executed as a single STU1-errata change would silently break every
implementer that has bound to the published canonical. C is essentially
unavailable given the ticket's explicit TSMG citation and the 2025
re-open. B captures the right *outcome* (the value set ends up in THO
where it belongs) while sequencing it sanely: file the UTG ticket now so
THO registration is in flight, mark the existing canonical deprecated in
the next errata, and complete the swap at the STU2 boundary where
canonical-URL changes are expected. This is also consistent with how
most active HL7 IGs have handled the same transition since the
2022-2024 TSMG sweep.

If the workgroup wants a path-level implementation plan (specific files,
sushi-config entries, dependency versions), please run `repo-analysis`
against the Patient Corrections IG GitHub repo first — it is not
currently present in the FHIR Augury cache, so this prep could only
reason at the policy level.
