## Summary

Ricardo Quintano asks whether the FHIR resources packaged inside a `*.sphr`
file (the Personal Health Record IG's "supporting PHR" container — a `.zip`
folder that bundles one or more `.phr` files alongside supporting materials
such as images, documents, and other media) are required to conform to the
PHR profiles defined by the IG. The reporter notes that, if conformance is
required, the IG should say so explicitly — the natural location being the
recordkeeping page at
`http://hl7.org/fhir/uv/phr/2025Jan/recordkeeping.html`. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced by FhirAugury for
this ticket, so the only available context is the ticket itself and the two
clarifying comments from `awatson1978` (one confirming "generally, yes" with
a promise to add documentation, and a later one quoting the IG's existing
section 2.1.2 description of the `.sphr` extension). The work group has
already moved this onto Block Vote 1 with a Persuasive resolution
(Motion: Hasahi; Second: Sarah Tonn, 3-0-0), so the disposition discussion
is essentially about how the editorial change should be framed.

## Details

**Description:**

> Does the *.sphr file resources need to conform to the PHR profiles?
>
> If so, this should be stated in the IG (e.g., http://hl7.org/fhir/uv/phr/2025Jan/recordkeeping.html)

**Comments:**

- **awatson1978 — 2025-01-30:** "Generally, yes. The idea is that the
  `.sphr` file is essentially a modified `.zip` file, and would contain a
  `.phr` file along with supporting artifacts. We will add additional
  guidance and documentation to make this more clear."
- **awatson1978 — 2025-07-10:** Quotes the IG's existing section 2.1.2 on
  file extensions:
  > **2.1.2 File Extensions - .sphr**
  > The `.sphr` file extension is introduced for when there are additional
  > supporting materials, which cannot be fully expressed in FHIR format.
  > The `.sphr` is a zip folder, which contains one or more FHIR resources,
  > as well as additional supporting materials.
  >
  > The `.sphr` file extension is intended to be used for data exports that
  > are saved to a CD, DVD, thumbdrive, etc. Each `.sphr` folder should
  > contain at least one `.phr` file within it, as well as additional
  > supporting materials such as images, documents, and other media.
  >
  > Per the `.zip` specification, the `.sphr` folder may be cryptographically signed.

**Resolution:** Persuasive — "Motion to add to Block Vote 1 (Motion: Hasahi;
Second: Sarah Tonn, 3-0-0)".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative guidance to the PHR IG stating that all FHIR resources
contained within a `.sphr` package **MUST** conform to the corresponding
PHR profiles defined in this IG. Specifically:

- In `recordkeeping.html` (section 2.1.2 "File Extensions - .sphr"),
  append a paragraph along the lines of:
  > "Any FHIR resources contained within a `.sphr` package — whether
  > inside the embedded `.phr` file(s) or as standalone resource
  > instances — SHALL conform to the profiles defined in this
  > Implementation Guide, in the same manner as resources packaged in a
  > standalone `.phr` file. Non-FHIR supporting materials (images,
  > documents, other media) are out of scope of profile conformance."
- Cross-link this paragraph to the existing PHR profile index so
  implementers can locate the applicable profiles.
- Mirror the same language in the `.phr` section so the two extensions
  are described consistently.

#### Justification

This is exactly what the reporter asked for, and it matches the
clarification awatson1978 already gave on the ticket ("Generally, yes").
Stating conformance explicitly removes the ambiguity that prompted the
ticket: today, section 2.1.2 only describes `.sphr` as a packaging
container, never tying its contents back to PHR profile conformance.
Codifying this in the recordkeeping page — which is the page the
reporter pointed at — keeps the guidance discoverable in the place
implementers will look. The work group has already voted Persuasive
(3-0-0, Block Vote 1), so this disposition aligns with the established
direction.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the conformance statement, but frame it as **SHOULD** rather than
**SHALL**, and split the guidance into two narrative bullets:

1. FHIR resources inside a `.sphr` package SHOULD conform to PHR
   profiles, and conformance SHALL be claimed via each resource's
   `meta.profile` so consumers can validate selectively.
2. Where a `.sphr` package contains resources that intentionally fall
   outside PHR profile scope (e.g., resources carried verbatim from an
   upstream source for traceability), the package SHALL include a
   manifest entry flagging those resources as
   non-conformant-by-design.

Apply the same wording to the `.phr` section for consistency, and add a
short example showing a manifest entry.

#### Justification

`.sphr` is positioned as a transport/archival container (CD, DVD,
thumbdrive). In real exports, implementers frequently need to carry
"as-received" resources that pre-date or post-date the profile set —
forcing strict conformance can push implementers to silently strip or
mutate data, which is worse for record-keeping than allowing a
clearly-labelled non-conformant payload. A `SHOULD` plus a manifest
mechanism gets the IG to the same place for the common case while
preserving fidelity for edge cases. The trade-off vs. Disposition A is
weaker default conformance and a small new manifest construct to
specify; the upside is fewer implementer escape hatches and better
provenance.

---

### Disposition C: Decline

#### Proposal

Decline the request and instead add only a non-normative editorial note
clarifying that `.sphr` is purely a packaging convention and that
profile conformance of the contained resources is governed by the
existing per-resource profile statements elsewhere in the IG, not by
the `.sphr` section. No new conformance rule is introduced.

#### Justification

The IG already defines PHR profiles and the conformance expectations
for the resources to which they apply; restating that inside the
file-extension section risks duplication and divergence. If `.sphr` is
merely a container, then container-level wording shouldn't introduce
new resource-level rules. However, this reading conflicts with the
work group's Persuasive vote on the ticket, so this disposition is
listed mainly to capture the "do nothing normative" option for the
record.

---

### Recommendation

**Recommended disposition:** A

The work group has already resolved the ticket Persuasive and moved it
onto Block Vote 1, so the substantive direction is settled: the IG
should say what `.sphr` resources are expected to conform to. Of the
three options, Disposition A is the cleanest match for both the
reporter's ask and awatson1978's stated intent ("generally, yes"). It
is a small, contained editorial change to a single page
(`recordkeeping.html`, section 2.1.2) plus a parallel update to the
`.phr` section, with no new normative machinery to specify. Disposition
B's manifest-and-`SHOULD` model is defensible on real-world grounds but
introduces new structure that the work group has not signalled appetite
for and would likely re-open the ticket. Disposition C contradicts the
existing vote. Recommend adopting A as part of Block Vote 1, with the
exact wording finalised by the IG editor.
