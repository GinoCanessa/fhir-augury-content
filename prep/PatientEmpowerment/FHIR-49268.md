## Summary

The reporter is reading section 5.1.1 of the Personal Health Record (FHIR)
specification and finds a table whose column headers and row identifiers
(e.g., "section", "header name", "PH 1", "Manage Problem Lists") are not
defined anywhere visible on the page. They cannot tell where the canonical
definitions of these columns and item codes live, and request that the spec
make this discoverable. A comment from awatson1978 confirms the underlying
issue: the definitions exist in a long (100+ page) PDF that is linked from
the page, but readers must download it to learn what the table is even
saying. The work group has marked this as Triaged and acknowledges the
need to better align the published HTML spec with the source PDF, but the
specific approach is still TBD. No related Jira tickets, Zulip threads, or
GitHub items were surfaced by the cross-reference index for this ticket.

## Details

**Description:**

Where are the definitions of the columns/items from table 5.1.1?

For example: section, header name, PH 1, Manage Problem Lists, etc.

I think it is important to indicate where the reader can find these
definitions.

**Comments:**

- **awatson1978 — 2025-01-30:** Persuasive that we need better

  to on that page, but it requires a download and is 100+ pages long; and

  approach, but we recognize the need.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Personal Health Record (FHIR) specification page that hosts
table 5.1.1 to make the column and row definitions discoverable directly
from the HTML, without requiring readers to download the source PDF.
Concretely:

- Add a short narrative block immediately above table 5.1.1 that defines
  every column header in use ("section", "header name", and any
  conformance/criticality columns), with each definition tied back to the
  corresponding concept in the underlying PHR-S Functional Model.
- Add a definition (or a link to a definition) for the row-identifier
  scheme (e.g., "PH" prefix + numeric ID such as "PH 1") and explain that
  these IDs come from the PHR-S FM source document, including which
  version/edition is the source of truth.
- For each row-identifier appearing in the table, hyperlink the ID to
  either (a) an inline glossary entry on the same IG page, or (b) a new
  "PHR-S FM Function Index" page in the IG that lists every referenced
  function (PH 1, PH 2, …) with its full name and short definition drawn
  from the source PDF.
- Keep the link to the source PDF, but reframe it as "authoritative
  source" rather than the only place the definitions live.

This work is purely narrative/IG-content; it does not touch FHIR
resource definitions, profiles, terminology, or conformance artifacts.

#### Justification

The reporter's complaint is straightforwardly correct: a normative-looking
table whose columns and row codes are undefined on the page is unusable
without out-of-band knowledge, and the work group's own comment concedes
the documentation is misaligned. FHIR's general IG authoring conventions
favour self-contained HTML pages over PDF dependencies, and other HL7
work groups routinely inline glossaries or function indexes when
referencing external functional models (e.g., the way EHR-S FM functions
are cross-referenced in related IGs). Doing the inline definitions plus a
function index page is the lowest-risk, highest-value option: it costs
only authoring effort, it is fully backwards compatible (no
machine-readable artifact changes), and it directly removes the barrier
the reporter encountered.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than inlining definitions on the section 5.1.1 page, restructure
the way the PHR IG references the PHR-S Functional Model so that the
table itself becomes a generated view over a structured source:

- Capture the PHR-S FM functions (ID, name, short description, source
  reference) as structured data inside the IG — for example, as a
  CodeSystem and ValueSet pair (CodeSystem of PHR-S FM function codes
  with display = function name and definition = the short description),
  or as a CSV/JSON data file under the IG's `input/` tree that the
  publisher renders into table 5.1.1 at build time.
- Replace the static table on the page with a generated rendering that
  pulls from this structured source, so column headers are defined in
  one place and every row link resolves to the CodeSystem concept page
  the IG publisher already produces.
- Keep the source PDF link as a provenance pointer on the CodeSystem,
  not as the primary lookup mechanism.

#### Justification

Disposition A fixes the immediate readability problem but leaves the
table as hand-maintained narrative that can drift from the source PDF
again the next time PHR-S FM is updated. Promoting the function list to
a CodeSystem (or at minimum a structured input file) gives the IG a
single source of truth, makes the PHR-S FM functions referenceable from
other FHIR artifacts (profiles, extensions, instances), and lets the IG
publisher do the page-level definition rendering for free. The trade-off
is up-front modelling effort and a discussion with the PHR-S FM owners
about whether HL7 can publish the function codes as a CodeSystem (vs.
only referencing the PDF), so this is a larger change than A but more
durable.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the definitions are already
available via the linked PHR-S Functional Model PDF on the same page,
and direct the reporter there. Close as "Not Persuasive" or "Already
Addressed" with a comment pointing at the PDF link.

#### Justification

This is defensible only as a strict reading: the page does link to the
authoritative source, and a determined reader can in principle find the
definitions. However, the work group's own triage comment ("persuasive
that we need better documentation") explicitly concedes the opposite, so
this disposition is included for completeness rather than as a serious
contender. It would also leave the table effectively undocumented for
casual readers and IG implementers who do not download multi-hundred-page
companion PDFs, which is at odds with how the rest of the FHIR ecosystem
publishes normative content.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers the reporter's complaint, is consistent
with the work group's own acknowledgement that the documentation needs
to be better aligned, and can be delivered as pure IG narrative work
without coordinating a CodeSystem publication with the PHR-S FM
maintainers. Disposition B is the better long-term answer and should be
captured as a follow-up ticket against the PHR IG, but treating it as
the response to FHIR-49268 risks letting the immediate readability
problem sit unresolved while the structured-source design is debated.
Disposition C contradicts the work group's own triage note and should
not be adopted. The recommended next step is to assign FHIR-49268 to a
PHR IG editor to add (a) inline column-header definitions above table
5.1.1 and (b) a "PHR-S FM Function Index" page, and to open a separate
ticket to track the eventual migration of the function list into a
structured source per Disposition B.
