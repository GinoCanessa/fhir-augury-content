### Disposition A: Accept As Requested

#### Proposal

Add at least one — preferably two or three — new complete DICOM example
sets to the `HL7/personal-health-record-format-ig` IG, demonstrating a
realistic PHR-scoped imaging record. Each example set should include:

- A `Patient` example reused from the IG's existing example patient(s).
- An `ImagingStudy` instance with a non-trivial `identifier` (DICOM
  Study Instance UID), `started`, `modality`, `numberOfSeries`,
  `numberOfInstances`, and a populated `series` element containing at
  least one `series.instance` with `uid`, `sopClass`, and `number`
  populated to spec.
- An `ImagingSelection` (R5/R6) or equivalent reference instance that
  highlights specific frames/instances of the study, showing how a PHR
  consumer would pin a clinically relevant slice.
- One or more `DocumentReference` (or `Binary`) examples that reference
  the DICOM payload via WADO-RS / DICOMweb URLs, illustrating retrieval
  rather than embedding the pixel data.
- A wrapping `Bundle` (transaction or collection per the IG's existing
  example pattern) tying the resources together so the example is
  ingestible end-to-end.

The new examples should be added under the IG's existing example
authoring root (location to be confirmed against the briefing) and
referenced from the relevant narrative page(s) so they appear in the
rendered IG.

#### Justification

The ticket text and the upstream issue both ask for "complete DICOM
records," which is a documentation/example request rather than a
profile change. Adding worked examples is the lowest-risk way to
satisfy implementer demand: it does not change conformance, does not
alter profiles, and slots into the IG Publisher build the IG already
runs. It also matches a common Patient-Empowerment pattern of using
PHR-shaped bundles to show how non-clinician users would assemble
imaging records they receive from providers. The main open question
(whether the IG already has a DICOM example skeleton to extend, or
whether one must be authored from scratch) needs the missing briefing
to answer; either way the change is additive.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) authoring net-new example bundles
inside this IG, reuse and link to the canonical DICOM-on-FHIR example
set maintained in the imaging IG ecosystem (e.g. the DICOMweb /
imaging-related FHIR examples), and have this IG add a short
"Imaging examples" page that:

- Embeds one minimal end-to-end PHR-flavored DICOM example authored
  here (Patient + ImagingStudy + ImagingSelection + DocumentReference +
  Bundle), and
- Links out, with explicit guidance, to the upstream imaging examples
  for the deeper variations (multi-modality studies, multi-series,
  WADO-RS retrieve flows), so the PHR IG does not duplicate and then
  drift from the imaging community's canonical examples.

#### Justification

DICOM examples are notoriously verbose and easy to author incorrectly
(UIDs, SOP class codings, modality codes, series ordering). Owning a
large set of bespoke DICOM examples inside a PHR IG creates a long-tail
maintenance burden that the Patient Empowerment work group is unlikely
to want to carry, and risks the PHR examples drifting from whatever the
imaging community considers current. A "one canonical example here,
link to the deep set elsewhere" approach still answers the implementer
question ("what does a complete DICOM record look like inside a PHR
bundle?") while keeping the maintenance surface small. Trade-off vs. A:
slightly less self-contained, but materially less ongoing work and less
risk of stale codings.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the request is purely an
example/documentation enhancement with no normative impact, and that
DICOM example coverage belongs in the imaging-focused FHIR IGs rather
than in the Personal Health Record IG. Close as "out of scope for the
PHR IG; consult the imaging IG examples" and update the relevant
narrative page in the PHR IG with a single pointer to the upstream
imaging examples instead of authoring new ones here.

#### Justification

The PHR IG's value proposition is patient-curated, longitudinal,
multi-domain records — not deep modality-specific worked examples.
Imaging-on-FHIR has its own active community with curated DICOM
examples; duplicating them inside the PHR IG risks divergence and adds
review burden to a work group whose bandwidth is better spent on
PHR-specific shape and provenance concerns. If the underlying need is
just "show me how DICOM data fits in a PHR bundle," a single narrative
paragraph plus a link is defensible.

---

### Recommendation

**Recommended disposition:** A

The ticket is high-priority, the request is concrete and additive, and
the upstream GitHub issue ([#57](https://github.com/HL7/personal-health-record-format-ig/issues/57))
indicates real implementer demand inside the PHR community
specifically. Disposition B's "link out" model is appealing on
maintenance grounds but does not actually answer the question
implementers are asking ("what does a complete DICOM record look like
*inside a PHR bundle*?"), and Disposition C punts a high-priority
Patient Empowerment ask onto a different work group, which is hard to
defend given the ticket's priority. Accepting the request and adding a
small, well-scoped set of complete DICOM example bundles (Patient +
ImagingStudy + ImagingSelection + DocumentReference + Bundle, with
WADO-RS-style references) gives implementers exactly what they asked
for at low conformance risk. Before authoring, the assignee should
generate the missing repo briefing (run `repo-analysis HL7/personal-health-record-format-ig`)
to confirm the example authoring root and any IG-Publisher gotchas,
and should consider Disposition B's "link to upstream imaging
examples" as a complementary follow-up rather than a substitute.
