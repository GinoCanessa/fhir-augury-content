## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the broken external link "File extension associations and default
apps in Windows 10, how to handle this using commandline" from the
Personal Health Record (FHIR) specification text. Leave the surrounding
narrative intact, or lightly reword the sentence so it still reads
naturally without the citation. This matches the action already recorded
in the ticket's resolution description ("Removed link").

#### Justification

The link is dead and the cited resource is a third-party Windows-platform
how-to that is tangential to the FHIR PHR specification's normative
content. Removing it eliminates a documentation defect with no impact on
conformance, profiles, or implementations. There is no community
disagreement on Zulip and no cross-referenced ticket suggesting a
different path; the assigned editor (`awatson1978`) has already signalled
intent to correct it.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of deleting the reference outright, replace it with either (a) an
Internet Archive (Wayback Machine) snapshot of the original page, or
(b) an equivalent current Microsoft Learn / docs.microsoft.com article on
file-type associations and default apps in Windows. Update the link text
if necessary to reflect the new target, and keep the surrounding
explanatory sentence so readers understand why the reference is being
made.

#### Justification

If the original citation was load-bearing — i.e., the PHR text leans on
that page to justify a platform-level behaviour around file handlers — a
silent removal can leave the prose dangling or weaker than intended.
Substituting an archived or current-equivalent URL preserves the editorial
intent and is consistent with how broken external references are commonly
handled in long-lived specifications. The cost is small (one editor
checks a replacement) and avoids a follow-up ticket if a future reader
asks why the citation is missing.

---

### Disposition C: Decline

#### Proposal

Take no action: leave the broken link in the specification text as-is and
close the ticket as `Won't Do`.

#### Justification

This is defensible only if the workgroup believes the link is purely
incidental and that touching the PHR text is not worth a publication
cycle. In practice, a dead external reference in a published HL7
specification is a documentation defect, and the cost of fixing it is
trivial; declining sets a poor precedent for editorial hygiene and would
contradict the assignee's already-recorded intent to correct it. This
disposition is included only for completeness.

---

### Recommendation

**Recommended disposition:** A

The recorded resolution ("Removed link") and the assignee's comment
already converge on Disposition A, and the change is purely editorial
with no normative impact. Disposition B is reasonable if the editor
judges that the citation was meaningful, but it adds review effort
without changing conformance, and there is no signal in the ticket or in
related discussion that the link was load-bearing. Disposition C should
be rejected — leaving a known dead link in a published FHIR
specification is a documentation defect the workgroup should not
ratify. Recommend accepting and confirming the link has been removed in
the next PHR build.
