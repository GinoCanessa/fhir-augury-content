## Summary

This Change Request asks the Patient Empowerment work group to update the
Personal Health Record (PHR) FHIR Implementation Guide so that the section
headers within its **Functional Model** narrative link out to the
corresponding sections of the EHR System Functional Model (EHR-FM) IG, using
the FHIR R5 build hosted at
`https://vdzel.home.xs4all.nl/ehrsfm-fhir-r5/` as the link target. The
ticket title references the "PHR-S IG" while the body refers to the "EHR-FM
IG"; both are part of the closely-related HL7 Functional Model family
(EHR-S FM and PHR-S FM share a common section structure), and the
reporter's intent is that PHR Functional Model section headers should
become hyperlinks pointing into the corresponding sections of the EHR-FM
FHIR rendering. No cross-referenced Jira tickets, Zulip threads, GitHub
issues/PRs, or Confluence pages were surfaced for FHIR-49462 in the
augury index, so context comes solely from the ticket body.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the Personal Health Record FHIR IG, update the narrative pages that
present the **Functional Model** so that each section header (e.g., the
top-level functional categories such as DC – Direct Care, S – Supportive,
IN – Information Infrastructure, and the per-function sub-headers under
them) becomes a hyperlink. Each link should target the corresponding
section anchor in the EHR-FM FHIR R5 IG hosted at
`https://vdzel.home.xs4all.nl/ehrsfm-fhir-r5/`. Mechanically, this is a
narrative/HTML edit in the IG's intro/section pages (the Markdown or
XHTML files that drive the Functional Model TOC), wrapping each header in
an `<a href="https://vdzel.home.xs4all.nl/ehrsfm-fhir-r5/{section}.html">…</a>`
or equivalent IG Publisher link macro. No StructureDefinition,
CapabilityStatement, or other conformance resource is changed — this is a
documentation/navigation-only change.

#### Justification

The PHR-S Functional Model is derived from, and aligned section-by-section
with, the EHR-S Functional Model. Linking section headers directly to the
EHR-FM IG gives readers a one-click path to the canonical definition of
each function and helps reviewers verify alignment between the two
functional models. The change is low-risk because it touches only
narrative content, does not affect conformance, and matches existing IG
practice of cross-linking related HL7 publications.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hard-coding links to the developer-hosted preview at
`https://vdzel.home.xs4all.nl/ehrsfm-fhir-r5/`, link the PHR-S Functional
Model section headers to the **canonical HL7-published EHR-FM FHIR IG
URL** (e.g., the `hl7.org/fhir/uv/ehrs-fm` or equivalent ci-build /
published location once available). In the PHR-S IG, introduce a single
"Related specification" panel at the top of the Functional Model page
linking to the EHR-FM IG as a whole, plus per-section "See EHR-FM
§{section}" links rendered next to each header rather than wrapping the
header text itself. Use the IG Publisher's dependency mechanism (declare
the EHR-FM IG as a dependency in `sushi-config.yaml` / `ig.ini`) so the
links can be generated/validated rather than authored as raw URLs.

#### Justification

The URL in the ticket (`vdzel.home.xs4all.nl`) is a personal/working
build, not a stable HL7 publication endpoint, and is likely to break or
move. Linking PHR-S to a non-canonical host would create dead links the
moment the EHR-FM IG is officially published or the working build moves.
Wrapping the header text in a link can also impair accessibility (header
text becomes a link, screen-reader semantics shift) and conflicts with
typical IG styling; placing a sibling "See EHR-FM §X" link is gentler.
Declaring the EHR-FM IG as a formal dependency lets the IG Publisher
validate the cross-references and lets the link targets follow the
canonical URL when the EHR-FM IG is republished.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that linking the PHR-S Functional Model
section headers out to a third-party / personal-build URL of another HL7
specification is **out of scope for this ticket as written**, and should
be deferred until (a) the EHR-FM FHIR IG has a stable canonical
publication URL and (b) the PHR-S work group has agreed on a
cross-IG-linking convention. Close the ticket as "Not Persuasive" with a
note inviting the reporter to reopen once the EHR-FM IG is officially
balloted/published, at which point a revised request can specify the
canonical URL.

#### Justification

Embedding links to a non-HL7-hosted preview build commits the PHR-S IG to
URLs the work group does not control, creating a near-certain future
maintenance burden when the build is renamed, moved, or replaced by an
official HL7 publication. Until the EHR-FM FHIR IG has a canonical URL,
adding the links produces churn for marginal short-term reader benefit.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need — making it trivial for readers of the
PHR-S Functional Model to jump to the corresponding section in the EHR-FM
Functional Model — is legitimate and aligns with how HL7 typically
cross-links related specifications. However, accepting the request
verbatim (Disposition A) would bake links to a personal build URL
(`vdzel.home.xs4all.nl`) into a published IG, which is fragile and not in
keeping with HL7 publication practice. Outright declining (Disposition C)
forfeits a real navigation improvement. Disposition B preserves the
reporter's intent while routing the links through the canonical EHR-FM
FHIR IG URL, declaring the EHR-FM IG as a formal IG dependency so the
links are validated by the IG Publisher, and using sibling "See EHR-FM
§X" links rather than wrapping the header text itself (better for
accessibility and consistent with the IG's existing styling). Because no
cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced, the work group should also confirm the PHR-S FHIR IG
authoring repository and run the `repo-analysis` skill against it before
the change is implemented.
