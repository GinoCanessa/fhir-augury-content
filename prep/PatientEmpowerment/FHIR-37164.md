## Summary

FHIR-37164 asks the Patient Empowerment work group to re-anchor the canonical
URL of the `PatientCorrectionOutputTypesVS` ValueSet (currently
`http://hl7.org/fhir/uv/patient-corrections/ValueSet/PatientCorrectionOutputTypesVS`)
to the HL7 terminology server — `terminology.hl7.org` (THO) — and to register
the value set in THO via a corresponding UTG ticket. The submitter (Ron G.
Parker, imported as comment 7 from a batch review) cites TSMG guidance that
only value sets bound as `required` to a `code` data type should remain under
`hl7.org/fhir`; everything else belongs in THO.

This ticket is one of a tightly related family of eight tickets filed against
the Patient Corrections IG that all apply the same THO-anchoring rule to
different artifacts: [FHIR-37158](https://jira.hl7.org/browse/FHIR-37158) and
[FHIR-37159](https://jira.hl7.org/browse/FHIR-37159) /
[FHIR-37160](https://jira.hl7.org/browse/FHIR-37160) /
[FHIR-37161](https://jira.hl7.org/browse/FHIR-37161) cover the four
`PatientCorrection*` CodeSystems, while
[FHIR-37162](https://jira.hl7.org/browse/FHIR-37162) /
[FHIR-37163](https://jira.hl7.org/browse/FHIR-37163) /
[FHIR-37165](https://jira.hl7.org/browse/FHIR-37165) cover the sibling
ValueSets (`PatientCorrectionBusinessStatusVS`,
`PatientCorrectionCommunicationTypesVS`, `PatientCorrectionTaskTypesVS`).
All eight were originally voted "Considered for Future Use" on 2022-10-17,
and Lynn Laakso re-opened them on 2025-07-10 ("re-open post STU1
publication") so they can be acted on now. Treating FHIR-37164 in isolation
without coordinating with the other seven would re-introduce the same
inconsistency they collectively try to fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the `PatientCorrectionOutputTypesVS` ValueSet out of the Patient
Corrections IG namespace and re-anchor it under the HL7 terminology
server. Concretely:

1. File a UTG ticket (HTA / TSMG-tracked) to register
   `PatientCorrectionOutputTypesVS` at the canonical URL
   `http://terminology.hl7.org/ValueSet/PatientCorrectionOutputTypesVS`
   (paired with the underlying `PatientCorrectionOutputTypes` code system
   from FHIR-37160).
2. In the Patient Corrections IG source, replace the local definition of
   `ValueSet/PatientCorrectionOutputTypesVS` with a reference (`compose`
   with `system` pointing at the THO-hosted code system, or — once the
   ValueSet is published in THO — drop the local copy and update every
   `binding.valueSet` to the new THO canonical).
3. Coordinate with the seven sibling tickets (FHIR-37158 through
   FHIR-37163, FHIR-37165) so all four CodeSystems and four ValueSets
   move together in one publication cycle, avoiding a half-migrated state.
4. Add an upgrade note in the IG history page explaining the canonical
   URL change for implementers.

#### Justification

This is exactly what TSMG asks for. Per the description and the broader
HL7 vocabulary policy referenced in it, only value sets that are
`required`-bound to a FHIR `code` data type belong in the
`hl7.org/fhir/...` namespace; reusable value sets — including the four
`PatientCorrection*` value sets in this IG — should live under
`terminology.hl7.org`. Doing this for all eight sibling artifacts in
lockstep keeps the IG internally consistent and keeps tooling
(validators, the FHIR registry, terminology servers) able to resolve
both the CodeSystem and the ValueSet without re-pointing to a transient
IG publication URL. Lynn Laakso's 2025-07-10 re-open ("post STU1
publication") explicitly clears the way to do this work now.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `PatientCorrectionOutputTypesVS` defined in the IG (no canonical
change), but **publish a THO mirror** under
`http://terminology.hl7.org/ValueSet/PatientCorrectionOutputTypesVS` and
mark the IG copy as deprecated in favour of the THO mirror starting from
the next publication. Concretely:

1. File the UTG ticket to publish the value set in THO, but with a
   `relatedArtifact` / `useContext` note that the IG copy remains the
   normative source for the current STU1 cycle.
2. Annotate the IG ValueSet page with a banner pointing implementers at
   the THO canonical for new use, and schedule the canonical-URL flip
   for the *next* IG ballot (rather than as a publication-cycle errata).
3. Apply the same staged plan to the sibling ValueSets (FHIR-37162,
   FHIR-37163, FHIR-37165). The four CodeSystem tickets (FHIR-37158
   through FHIR-37161) can move directly to THO since downstream impact
   is bounded.

#### Justification

The eight tickets re-open a publication-cycle correction on an IG that
has already been balloted (STU1). A canonical-URL change is technically
breaking for any implementer that has hard-coded the
`hl7.org/fhir/uv/patient-corrections/ValueSet/...` URLs into validators
or terminology servers; staging the migration over two publication
cycles with a deprecation banner gives those implementers a window to
update without surprise. The THO entry still gets created promptly (so
TSMG's intent is honoured), but the IG doesn't break existing tooling
the moment the next errata is published.

---

### Disposition C: Decline

#### Proposal

Close FHIR-37164 (and, by extension, the seven siblings) as **Persuasive
with mod / Not Persuasive** on the grounds that the
`PatientCorrection*` value sets are IG-specific (used only by Patient
Corrections artifacts) and that re-anchoring them in THO offers limited
practical reuse benefit while introducing breaking canonical-URL
changes. Document the decision and direct future re-users of the same
codes to import the IG-defined ValueSets directly.

#### Justification

The eight value sets/code systems were authored specifically for the
Patient Corrections workflow and have no recorded reuse outside this IG.
Per pure-cost reasoning, moving them to THO costs a UTG submission, an
IG edit, and a backwards-incompatible canonical URL change for every
binding. If reuse never materialises, the migration delivers no
implementer-visible value. This disposition is the right one **only**
if the work group concludes — explicitly — that TSMG's vocabulary policy
should not apply to this IG. Note that this conflicts with the policy
quoted in the ticket, so the work group would need to coordinate with
TSMG before adopting it.

---

### Recommendation

**Recommended disposition:** A

The submitter is quoting standing TSMG vocabulary policy, not opinion:
non-`code`-bound value sets belong in THO. All eight sibling tickets
(FHIR-37158–FHIR-37165) carry the same template and the same 2025-07-10
re-open by Lynn Laakso explicitly so the work can be done on the
post-STU1 cycle, which is the sane time to break the canonical URLs at
once rather than twice. Disposition B's staged approach reduces the
short-term shock for implementers, but it doubles the publication-cycle
overhead and leaves the IG inconsistent (CodeSystems in THO, ValueSets
still in the IG) for an entire ballot cycle — which is the exact state
TSMG is trying to avoid. Disposition C contradicts an HL7-wide policy
quoted directly by TSMG and would need to be escalated before being
defensible.

The work group should adopt Disposition A and coordinate the eight
tickets (and the corresponding UTG submissions) as a single batch so
that the post-STU1 publication ships the full migration in one move.
