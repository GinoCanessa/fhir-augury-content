## Summary

The reporter notes an inconsistency in the Personal Health Record (PHR) FHIR
Implementation Guide: although the IG is built on FHIR R4, the resource
hyperlinks shown in table 5.1.1 of the guide resolve to R5 versions of those
resources. The reporter is asking the Patient Empowerment work group to
explain (and presumably correct) the mismatch so that readers following the
links land on R4 documentation that matches the IG's normative base. The
ticket has been resolved as Persuasive: the maintainer agrees the R5 links
were not intentional and has committed to updating both the links and the
tooling that produced them. No cross-referenced Jira tickets, Zulip threads,
or GitHub items were surfaced via FhirAugury.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Personal Health Record (FHIR) IG so that every resource hyperlink
in table 5.1.1 (and any other table generated by the same template/macro)
targets the R4 publication of the resource (`http://hl7.org/fhir/R4/<resource>.html`)
rather than the R5 publication. This includes both:

1. A one-time content fix for the existing rendered table 5.1.1 to repoint
   each resource link to its R4 URL.
2. A fix to the link-generation tooling / template that produced those links
   so that future builds default to the IG's declared FHIR version (R4) and
   do not regress to R5 on republish.

#### Justification

This matches the maintainer's own assessment in the resolution comment
("Persuasive, and was not entirely intended to point to R5. Will update
links and tooling accordingly."). It is also the only outcome that aligns
the IG's documentation with its declared normative base: an R4 IG whose
worked-example links point to R5 is actively misleading to implementers,
who may inadvertently pick up R5-only elements, removed elements, or
changed cardinalities and assume they apply to their R4 implementation.
Fixing the tooling, not just the rendered page, prevents the same defect
from re-appearing on the next IG build.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hard-coding R4 links, change the IG tooling to derive resource
link targets from the IG's declared `fhirVersion` in its NPM package /
ImplementationGuide resource. The link macro used by table 5.1.1 (and any
similar tables) would emit `http://hl7.org/fhir/<version>/<resource>.html`
where `<version>` is taken from the IG's own metadata. Apply the resulting
correction to the existing publication so table 5.1.1 currently renders R4
links, but leave the mechanism version-aware for any future PHR re-base
(e.g., to R5 or R6).

#### Justification

This addresses the same defect as Disposition A and produces the same
visible result for the current publication, but it removes the underlying
class of bug instead of patching one instance. It also reduces maintenance
cost the next time the PHR IG (or any downstream IG using the same
template) is re-based to a different FHIR version: contributors will not
need to remember to globally search-and-replace `/R4/` ↔ `/R5/`. The
trade-off is a slightly larger tooling change up front; the benefit is
that the link target can never silently disagree with the IG's declared
FHIR version again.

---

### Disposition C: Decline

#### Proposal

Reject the change request, leaving the R5 links in table 5.1.1 in place,
on the grounds that the table is intended as a forward-looking reference
to the latest FHIR publication of each resource and that readers of an R4
IG are expected to translate to their target version.

#### Justification

This position is defensible only if the work group has an explicit policy
that "resource type" cells in IG tables always point to the most-current
FHIR publication regardless of the IG's normative base, and that policy
is documented somewhere a reader can find it. In practice no such policy
appears to exist for the PHR IG, the maintainer has already accepted the
ticket as Persuasive, and silently linking R4 implementers to R5
documentation creates a real risk of misuse. Decline is therefore weak.

---

### Recommendation

**Recommended disposition:** B

Disposition B delivers exactly the visible fix the reporter asked for
(table 5.1.1 links resolve to R4 in the published PHR IG) while also
satisfying the second half of the maintainer's commitment ("update links
and tooling accordingly") in a way that is durable. Hard-coding R4 in
the template (Disposition A) would close this ticket but leave the same
defect waiting to re-appear the next time the IG is rebuilt or rebased,
and would likely be filed again. Disposition C contradicts the
maintainer's own resolution and would mislead R4 implementers. The
incremental cost of making the link macro read the IG's declared
`fhirVersion` is small, the change is local to the PHR IG's publishing
template, and it leaves the work group with a cleaner pattern to reuse
in other Patient Empowerment guides.
