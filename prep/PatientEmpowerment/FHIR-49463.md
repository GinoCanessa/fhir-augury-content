## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level narrative page titled **"FAQ"** to the Personal
Health Record (FHIR) IG. Concretely, in the PHR IG repository:

1. Create `input/pagecontent/faq.md` (markdown) following the IG's
   existing page template / heading style.
2. Register the page in the IG's table of contents — either via
   `sushi-config.yaml` (`pages:` / `menu:` entries) or, if the IG uses a
   hand-edited `input/data/menu.xml` / `ig.ini`, add a "FAQ" menu entry
   pointing at `faq.html`.
3. Seed the page with a small starter set of question/answer entries
   covering the most common implementer and patient-facing questions
   already implicit in the IG (e.g., "What is a PHR?", "How does this
   IG relate to International Patient Summary / IPA / US Core?", "Is the
   patient the author of these resources?", "Where does authentication /
   authorisation come from?", "What is the conformance posture of the
   IG?"). The work group should curate and approve the actual content
   in a follow-up.
4. Link the new FAQ from the IG home page so it is discoverable.

The disposition would be marked *Persuasive* and resolved once a PR
adding the page lands and a new IG build is published.

#### Justification

The ask is small, low-risk, and consistent with how peer HL7 IGs handle
navigational documentation — many publish an FAQ alongside their
"Background" / "Use cases" / "Conformance" pages. Adding a new narrative
page does not change any conformance artifact (no profile, extension,
value set, or capability statement is touched), so there is no
implementer breakage to weigh. It also gives the Patient Empowerment WG
a low-cost, durable place to capture the patient-empowerment-specific
clarifications (consent, patient-as-author, reconciliation with US Core
/ IPA) that otherwise accumulate in Zulip and Jira.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the request as one for *better navigational guidance* rather than
specifically a "FAQ" page, and address it by:

1. Auditing the existing PHR IG narrative pages for gaps the reporter
   may be perceiving (Overview, Background, Scope, Use Cases, Patient
   Workflow, Implementer Guidance, Security & Privacy, Glossary).
2. Adding the **most underserved** of those pages — typically a
   "Implementer Guidance" or "Onboarding" page — and *also* a small
   FAQ-style "Common Questions" subsection on the IG home page (rather
   than a standalone `faq.md`).
3. Routing future questions into a tracked Zulip topic
   ("#patient empowerment > PHR IG questions") whose distilled answers
   are then promoted into the IG's Implementer Guidance page on each
   ballot cycle, so the IG does not accumulate stale FAQ entries.

#### Justification

Standalone FAQ pages in IGs are notoriously prone to bit-rot: they get
seeded on day one, then fall out of sync with the actual normative
content as profiles and extensions evolve, and they end up confusing
implementers more than they help. Embedding the most common questions
in the relevant sections (Implementer Guidance, Security & Privacy,
etc.) keeps the answer next to the artifact it explains, and a
lightweight "Common Questions" block on the home page still satisfies
the reporter's underlying need for a discoverable Q&A landing spot
without committing the WG to maintaining a parallel, drift-prone FAQ
page. Patient-empowerment topics in particular (consent, identity,
patient-as-author) tend to evolve quickly with adjacent IGs (US Core,
IPA, SMART Health Cards), which makes a frozen FAQ page especially
risky.

---

### Disposition C: Decline

#### Proposal

Mark the change request as **Not Persuasive** with the rationale
"insufficient detail to act on / authoring guidance, not a spec change"
and close it. Suggest the reporter re-file as either (a) a concrete
list of questions the FAQ should answer, or (b) a pull request against
the PHR IG repository proposing the page content directly. No change
to the IG is made on the basis of this ticket alone.

#### Justification

As filed, the ticket is a single sentence with no proposed questions,
no target audience, and no link to discussion that would tell the WG
*what* the FAQ should contain. A FAQ page is editorial / navigational
content, not a normative specification change, so it sits in a grey
area for Jira tracking — the standard HL7 mechanism for narrative-page
edits is a PR against the IG repo, optionally linked back to a more
specific Jira ticket. Declining (without prejudice) avoids committing
WG cycles to authoring content the reporter has not actually scoped,
while keeping the door open for a re-file with concrete material.

---

### Recommendation

**Recommended disposition:** A

The cost of accepting as requested is genuinely small — a single new
markdown page plus a one-line menu entry — and it gives the Patient
Empowerment WG a sanctioned, IG-resident landing spot for the
patient-empowerment clarifications that otherwise live only in Zulip
and prior Jira tickets. Disposition B's drift concern is real but is
better solved by editorial discipline (review the FAQ each ballot
cycle) than by refusing to create the page; an FAQ that lives under
`input/pagecontent/` participates in the same review and CI pipeline as
every other IG page, so keeping it current is a process question, not
an architectural one. Disposition C is technically defensible given the
sparse ticket but feels disproportionate for a one-page navigational
addition from a recognised PE community member; declining a
"please add an FAQ" request risks signalling the WG is hostile to
low-cost documentation improvements. Suggested next step: ask Abigail
Watson (or a PE co-chair) to seed the initial 5–10 Q&A entries, then
land them in a PR against the PHR IG repository (repo name to be
confirmed once the PHR IG is registered in FhirAugury's GitHub source).
