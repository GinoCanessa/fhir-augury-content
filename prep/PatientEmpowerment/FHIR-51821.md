## Summary

The reporter raises a wording / conformance-strength question against a
narrative passage in the US PACIO Advance Directive Interoperability (ADI)
FHIR IG. The passage currently reads that, when an advance directive is
replaced, "the original document then *should* be marked as replaced by
updating the `DocumentReference.status` to the code `superseded`." The
reporter asks whether this conformance verb should instead be **SHALL**,
i.e. whether marking the prior `DocumentReference` as `superseded` is a
mandatory step of the replacement workflow rather than a recommendation.

The reporter also notes that, if any legitimate use case exists where the
replaced `DocumentReference` does *not* end up with status `superseded`,
that exception should be called out in the IG so that future readers do
not raise the same question.

The cross-reference index surfaces only the FHIR artifact this affects
(`DocumentReference.status`); no other Jira tickets, Zulip threads, or
GitHub issues/PRs are linked to this item. The ticket has been resolved
"Persuasive with Modification" and approved as part of ADI Block Vote 4
(9-0-0); the workgroup agreed to strengthen the wording to **SHALL**, with
slightly adjusted final phrasing.

## Proposed Dispositions

> Note: The workgroup has already recorded a resolution of "Persuasive
> with Modification" (Block Vote 4, 9-0-0). Dispositions A and B below
> reflect that recorded outcome and a close alternative; Disposition C
> is included for completeness as the skill requires three options.

### Disposition A: Accept As Requested (with modification — matches recorded resolution)

#### Proposal

In the US PACIO ADI IG narrative section that describes replacing an
advance directive document, change the conformance verb on the prior
`DocumentReference` from **should** to **SHALL**, using the wording
already agreed in the resolution:

> "The original document then SHALL be marked as replaced by updating
> the `DocumentReference.status` to the code `superseded`."

This is a narrative-only change; no `StructureDefinition`, `ValueSet`,
or example resource needs to be modified. Concretely, the IG authoring
source page that contains this sentence (the page in the PACIO ADI IG
that documents the document-replacement / supersession workflow) is
edited so that the verb is upper-case **SHALL** and the surrounding
sentence is updated to the resolution text. No new exception list is
added, on the basis that the workgroup is asserting that there is no
legitimate replacement scenario in ADI in which the prior
`DocumentReference` should remain in any other status.

#### Justification

This matches the disposition the ADI community already approved (Block
Vote 4, 9-0-0) and addresses the reporter's primary concern: the IG was
ambiguous about whether updating the prior `DocumentReference.status` to
`superseded` is mandatory. Using **SHALL** makes the obligation testable
and aligns the narrative with the FHIR convention that conformance verbs
in upper case carry normative weight. It also closes a real
interoperability risk: if implementers treat the step as optional, two
valid `DocumentReference`s for the same logical advance directive can
remain in `current` status simultaneously, and downstream consumers
(patients, providers, registries) cannot tell which one is in force.

---

### Disposition B: Alternative Approach — SHALL plus an explicit exception clause

#### Proposal

Apply the same SHALL wording change as Disposition A, but additionally
add a short exception paragraph immediately after the SHALL sentence
that enumerates the (intentionally narrow) cases in which the prior
`DocumentReference` may legitimately *not* be moved to `superseded` —
for example, when the prior document has already been moved to
`entered-in-error` (per the `DocumentReference.status` value set), or
when the prior record is owned by a system that the replacing actor
cannot update. The paragraph would explicitly say that, except for
those listed cases, the prior `DocumentReference` SHALL be set to
`superseded`.

This directly answers the second half of the reporter's comment ("If
there is use case that the replaced DocumentReference does not have
'superceded' status, please add it here as an exception so readers do
not raise the same question.").

#### Justification

The reporter raised two concerns: (1) the verb is too weak, and (2) any
legitimate exceptions should be documented so the same question does not
recur. Disposition A satisfies (1) but silently leaves (2) unresolved.
Disposition B satisfies both. The trade-off is editorial weight: it
asks the workgroup to enumerate exceptions, and any list that is
incomplete may itself draw future tickets. It is preferable to A only
if the workgroup believes there *are* real exceptions worth naming
(e.g. `entered-in-error`, externally-owned prior records). If the WG
believes there are none, Disposition A is cleaner.

---

### Disposition C: Decline

#### Proposal

Leave the narrative as-is with the lower-case "should". Reject the
request on the grounds that the IG already references the standard
`DocumentReference.status` value set, and supersession behaviour is
adequately conveyed by the existing FHIR `DocumentReference` resource
documentation; tightening to SHALL is therefore unnecessary
restatement.

#### Justification

Defensible only if the workgroup believes that mandating
`superseded` would over-constrain implementers who have legitimate
reasons to leave the prior record in another state (e.g.
`entered-in-error` or unmodifiable external records) and that adding
an exception list (Disposition B) is more editorial burden than the
narrative warrants. This rationale is weak in practice for ADI,
because the IG explicitly describes a replacement workflow, and a
replacement workflow that is silent about the prior record's status
is the very ambiguity the reporter flagged. The Block Vote 4 outcome
(9-0-0) indicates the WG has already concluded that decline is not
the right answer.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already approved Disposition A by block vote (9-0-0),
and the change is narrative-only with clear interoperability benefit:
moving from a permissive "should" to a normative **SHALL** removes
ambiguity about who is responsible for retiring the prior
`DocumentReference` and prevents two `current` records from coexisting
for the same logical advance directive. Disposition B is a reasonable
upgrade if reviewers believe documented exceptions exist (notably
`entered-in-error` or externally-owned prior records), in which case
the WG can fold an exception clause into the same narrative edit
without re-balloting the core change. Disposition C should not be
adopted; it leaves the original ambiguity in place and is inconsistent
with the recorded WG vote.
