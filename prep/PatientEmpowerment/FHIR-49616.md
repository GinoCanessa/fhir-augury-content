## Summary

Gay Dolin raises an intellectual-property concern about the Personal Health
Record (PHR) FHIR Implementation Guide's use of non-standard
**AppleHealthKit** vocabularies. She has logged separate issues challenging
the use of these terminologies and has reached out to HL7 HQ to confirm
whether HQ shares those concerns. As a contingent recommendation — *if* the
IG continues through ballot with the Apple terminologies intact — she asks
the work group to do two things:

1. Add an **IP / trademark disclaimer** to the IG stating that "Apple
   Health" and "HealthKit" are trademarks of Apple Inc. and that the IG is
   neither affiliated with nor endorsed by Apple.
2. Add **explanatory guidance** about the use of Apple Health terminology
   — ideally on a dedicated **Terminology** page (or at minimum the home
   page) — explaining the rationale for using Apple Health codes, how they
   are applied within the IG, and how they complement standard
   vocabularies that are inherited.

The cross-reference index returned no related Jira tickets, Zulip threads,
or GitHub items for this key, so the request stands on its own at the
data layer; however, the reporter explicitly notes that this issue is
part of a broader set of tickets she filed about the same Apple-vocabulary
concern in the PHR IG, and the single comment on the ticket indicates the
project team intends to coordinate with the Terminology work group and
contact Pascal at Apple to determine the path forward (ConceptMap-based
mapping into standard FHIR codings rather than treating Apple HealthKit
as a standard).

The ticket has been **Triaged** with a resolution of *Persuasive with
Modification*. The resolution note records that the Apple-vocabulary
content was **removed per HL7 leadership**, with a follow-on plan to
adjust the approach with help from the Vocabulary work group — which
moots the disclaimer text but keeps the request's "explain the
terminology approach" intent live for whatever replacement strategy
lands.

## Proposed Dispositions

Because no GitHub repository was surfaced for this ticket, the
dispositions below reference the IG at the *content* level (home page,
terminology page, code-system narrative) rather than naming a specific
authoring root or file path. If the work group wants file-level
proposals, a `repo-analysis` run against the PHR IG repository would be
needed to anchor the dispositions in concrete authoring paths.

### Disposition A: Accept As Requested

#### Proposal

Adopt both parts of Gay Dolin's request *as written*, conditional on the
IG continuing to ship Apple HealthKit content:

1. Add the proposed trademark disclaimer verbatim ("Apple Health and
   HealthKit are trademarks of Apple Inc., registered in the U.S. and
   other countries and regions. This Implementation Guide is not
   affiliated with or endorsed by Apple.") to the IG's home page footer
   and to any page that renders Apple HealthKit codes (CodeSystem and
   ValueSet pages, and the new Terminology page below).
2. Add a dedicated **Terminology** page to the IG that (a) lists every
   Apple HealthKit-derived code system and value set used by the IG,
   (b) explains *why* Apple HealthKit terms are referenced (real-world
   device coverage / patient-generated data), (c) describes how those
   terms are intended to be mapped into standard FHIR codings via
   ConceptMaps, and (d) calls out which standard vocabularies (LOINC,
   SNOMED CT, UCUM, etc.) the IG inherits and recommends as the
   normative target.

#### Justification

This is a low-cost, high-value change: it does not alter any
StructureDefinition, ValueSet binding, or example, so it has no
implementer-facing breaking-change cost. The disclaimer language is
narrowly worded, factually correct, and consistent with how other HL7
IGs handle third-party trademarks. A standalone Terminology page is also
already a recommended HL7 IG pattern, so the change aligns the PHR IG
with broader publishing convention. The single ticket comment from
awatson1978 explicitly endorses the underlying intent (use Apple
HealthKit only as a *source* for ConceptMap-based mapping, not as a
standard), which means an explanatory Terminology page is the natural
written form of that plan.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the underlying concern as a **terminology-strategy** problem
rather than a documentation problem. Specifically:

1. Move every Apple HealthKit code reference out of the IG's normative
   bindings. Where the IG previously bound to Apple codes, switch the
   binding to a HL7-owned ValueSet (LOINC / SNOMED CT / UCUM-based
   where possible) and provide a published **ConceptMap** from the
   Apple HealthKit code system to that HL7-owned ValueSet.
2. Drop the IP disclaimer requirement, because the IG would no longer
   *carry* Apple terminologies in its normative content; the ConceptMap
   would reference the Apple code system by canonical URL only.
3. Add a short Terminology-page narrative explaining the
   ConceptMap-only relationship and how implementers consuming
   Apple-Health-sourced data should translate codes before persisting
   them as FHIR resources.

#### Justification

This alternative attacks the *root cause* the reporter identified
(non-standard vocabularies embedded in normative IG content) instead of
just adding a disclaimer around it. It's also tightly aligned with the
direction awatson1978 described in the ticket comment ("the goal isn't
to use the Apple HealthKit code system as a standard per se… but to
provide a ConceptMap"). The trade-off is significantly higher
implementation cost: every binding that currently references Apple codes
has to be re-pointed, ConceptMaps have to be authored and balloted, and
implementers who were relying on Apple codes flowing through the IG
unchanged will need to update. However, this matches what the
**resolution note already describes** ("Removed, per HL7 leadership.
Will adjust approach with vocab's assistance.") — i.e., the work
group has, in effect, already chosen this path; Disposition B simply
formalises the plan and gives the Vocabulary WG the change-shape it
needs to engage.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-49616 as not actionable on its own, on the basis that:

1. The IP-disclaimer text is contingent ("If this IG proceeds through
   ballot with the Apple terminologies intact…"). The PHR IG has, per
   the ticket's own resolution note, already removed the Apple
   vocabulary content per HL7 leadership, so the precondition no longer
   holds and the disclaimer is moot.
2. The "explain the terminology approach" guidance is better tracked
   against whichever follow-on ticket captures the *replacement*
   terminology strategy (ConceptMap-based mapping, etc.), so the
   Terminology-page narrative would be authored against that ticket
   rather than this one.

#### Justification

A decline here is defensible only because the underlying condition that
triggered the request has been overtaken by HL7 leadership's decision to
remove the Apple content. Accepting the disclaimer text now would commit
the IG to language about Apple trademarks that the IG no longer needs to
carry, which is mildly confusing for readers. The risk in declining is
that the *guidance* half of the request — the Terminology page — gets
lost; that risk is mitigated only if the work group has another live
ticket explicitly tracking the replacement terminology narrative.

---

### Recommendation

**Recommended disposition:** B

The ticket's own resolution note ("Removed, per HL7 leadership. Will
adjust approach with vocab's assistance.") and the awatson1978 comment
together describe a path that is materially Disposition B: stop carrying
Apple HealthKit terms as normative content, and shift to a
ConceptMap-based mapping coordinated with the Vocabulary work group.
Closing this ticket as straight Accept (Disposition A) would commit the
IG to a disclaimer for content it has already agreed to remove, and a
straight Decline (Disposition C) loses the *guidance* half of Gay
Dolin's request — readers and implementers still need the IG to
*explain* what happened to the Apple terms and how to translate Apple
HealthKit-sourced data into the IG's standard bindings. Disposition B
preserves the spirit of the reporter's concern, formalises what
leadership has already decided, gives the Vocabulary WG a concrete
deliverable (the ConceptMaps and the Terminology-page narrative), and
avoids publishing trademark language for content that will not appear
in the balloted IG.
