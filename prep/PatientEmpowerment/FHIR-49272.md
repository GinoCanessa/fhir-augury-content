## Summary

The reporter, Ricardo Quintano, asks how the Record Lifecycle Events
(RLE) listed in Table 6.0.1 of the Personal Health Record (PHR) FHIR
Implementation Guide are intended to be used. He reports that the
table presents a list of items but does not contain enough guidance
for an implementer to understand when, where, or how each lifecycle
event should be applied in a PHR context. The ticket is filed as a
Technical Correction against the PHR (FHIR) specification, with High
priority, against the Patient Empowerment work group.

The single triage comment from `awatson1978` indicates that the RLE
content "is likely going into the Record Lifecycle IG or CDS Hooks
IG, and will likely be removed from PHR IG", and notes the workgroup
is still considering how RLE harmonizes with CDS Hooks and where it
eventually lands. The triager characterized the request as "generally
persuasive". No related Jira tickets were surfaced via cross-references,
but adjacent community discussion in the `implementers` Zulip stream
("InitialRegistrationDateTime, MutationDateTime and Provenance",
October 2025) explicitly points implementers at the standalone EHR/PHR
Systems Functional Model — Record Lifecycle Events IG
(`hl7.org/fhir/uv/ehrs-rle/Informative1/`) as the canonical home for
this material, supporting the triage direction.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket literally as a request to clarify how the items in
PHR (FHIR) IG Table 6.0.1 should be used. Add narrative to Section 6
of the PHR FHIR IG that, for each Record Lifecycle Event listed in
Table 6.0.1, states:

1. The intended trigger (what action in the PHR causes the event),
2. The expected FHIR representation (typically a `Provenance`
   instance with `Provenance.activity` populated from the RLE
   vocabulary, optionally accompanied by an `AuditEvent`),
3. Whether the event is expected of conformant PHR systems or is
   purely informative,
4. A worked example linking back to the EHR/PHR Systems Functional
   Model — Record Lifecycle Events IG
   (`hl7.org/fhir/uv/ehrs-rle/Informative1/`) for the canonical
   profile.

Keep Table 6.0.1 in place; add a leading paragraph and per-row usage
notes (or a parallel column) so that the table is self-explanatory.

#### Justification

This directly answers the reporter's confusion without restructuring
the IG. It is consistent with the FHIR design principle that
normative tables in an IG should be accompanied by enough narrative
for an implementer to act on them, and it aligns with the community
guidance already given on the `implementers` stream
(InitialRegistrationDateTime / MutationDateTime / Provenance), where
John Moehrke pointed implementers at the same RLE IG as the canonical
profiling for these events. It also avoids any near-term disruption
to downstream consumers who already reference Table 6.0.1.

---

### Disposition B: Alternative Approach

#### Proposal

Replace Table 6.0.1 in the PHR (FHIR) IG with a short narrative
section that (a) explains *why* Record Lifecycle Events matter for a
PHR, (b) defers normative content to the EHR/PHR Systems Functional
Model — Record Lifecycle Events IG
(`hl7.org/fhir/uv/ehrs-rle/Informative1/`), and (c) links out to the
relevant pages of that IG for the actual event list, vocabulary
binding, and Provenance/AuditEvent profiles. Optionally retain a
small "PHR-specific considerations" subsection describing any
patient-facing nuances (e.g., patient-authored events, consent-driven
suppression) that the RLE IG does not already cover.

This is the structural form of the triage comment's suggestion that
the content "will likely be removed from PHR IG" and live in the RLE
IG (or, where event signalling is in scope, the CDS Hooks IG).

#### Justification

Maintaining a duplicate, less-detailed copy of the RLE catalogue in
the PHR IG is the root cause of the reporter's confusion: the table
on its own cannot carry enough context to be usable, and any
clarification added under Disposition A risks drifting from the
authoritative RLE IG over time. Centralizing the normative content
in the RLE IG is consistent with how the community already treats
this material (see the `terminology > ISO lifecycle events` and
`patient empowerment > HL7 PHR-S Functional Model` Zulip threads,
both of which point at the RLE IG as the canonical home) and is
backward-compatible for implementers who follow the link. The
trade-off vs. Disposition A is the one-time editorial cost of pruning
Table 6.0.1 and the need to keep the cross-reference URL stable.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed by an existing mechanism**: close the
ticket with a pointer to the EHR/PHR Systems Functional Model —
Record Lifecycle Events IG (`hl7.org/fhir/uv/ehrs-rle/Informative1/`)
and to the `Provenance` resource (with `Provenance.activity` bound
to the RLE vocabulary) as the canonical way to use Record Lifecycle
Events, and make no change to Table 6.0.1 or to the surrounding
narrative in the PHR (FHIR) IG.

#### Justification

The mechanism for *using* Record Lifecycle Events — populate a
`Provenance` (and optionally an `AuditEvent`) with the appropriate
RLE-vocabulary activity code — is already fully specified in the RLE
IG and in the base `Provenance` resource. A reasonable case can be
made that Table 6.0.1 is a pointer/index, not a normative
specification, and that implementer confusion should be answered with
documentation links rather than IG edits. Declining avoids spending
PHR work-group cycles on content that the same triage comment
acknowledges is on a path to be removed from the PHR IG anyway.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the cleanest match to both the triage comment ("this
is likely going into the Record Lifecycle IG ... and will likely be
removed from PHR IG") and the surrounding community signal from
Zulip, where implementers are consistently directed to the RLE IG as
the authoritative source for these events. Disposition A solves the
immediate confusion but locks the PHR IG into maintaining a parallel
copy of normative RLE content that the work group has already
indicated should not live there long-term, and risks the table
drifting from the RLE IG. Disposition C is defensible on principle
but leaves the documented confusion in place and ignores that the
reporter's complaint is against the PHR IG, not against the RLE IG —
shipping no edit at all is a poor experience for the next implementer
who hits Table 6.0.1. B captures the long-term direction while still
producing a concrete deliverable in the PHR IG (a clear narrative +
outbound link), and it keeps the door open for a future PHR/CDS Hooks
harmonization without re-doing the work.
