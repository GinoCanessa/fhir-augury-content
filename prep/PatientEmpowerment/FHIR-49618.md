## Summary

The reporter objects to the inclusion of vendor-specific Apple Health
(HealthKit) terminologies as code systems and value sets in the Personal
Health Record (PHR) FHIR IG. They argue that vendor-specific terminologies
are inconsistent with historical HL7 practice and with the IG's own
"medical grade" design philosophy, and ask the work group to (a) replace
the Apple Health terminologies with standard / HL7 / FHIR-based
terminologies where possible, or (b) if Apple Health terminologies are
retained, publish mappings to standard terminologies, expose those
mappings as `additionalBindings` on the affected profile elements, and
bind the standard codes at the element level with Apple Health codes
exposed only as additional bindings.

The ticket is tightly coupled to the reporter's companion ticket
[FHIR-49616](https://jira.hl7.org/browse/FHIR-49616) ("If Apple
vocabularies continue to be used in the IG. Please add an IP
disclaimer"), which raises the IP / trademark concern around the same
content and proposes a disclaimer plus a dedicated Terminology page
explaining the rationale and usage of Apple Health terminology. The
authoring response on FHIR-49616 (awatson1978, 2025-01-30) clarifies
that the project's intent is not to elevate HealthKit codes to a
standard, but to provide a `ConceptMap` from Apple HealthKit (and,
prospectively, Google / Common Health) into standard FHIR formats; the
exact mechanism was deferred for discussion with the Vocabulary work
group. The reporter also references the project scope statement
[PSS-2125](https://jira.hl7.org/browse/PSS-2125) and notes that the PSS
did not surface the planned use of external vendor terminologies.

The ticket has been resolved **Persuasive with Modification** with the
disposition note: *"Removed, per HL7 leadership. Will adjust approach
with vocab's assistance. 5-0-0"* тАФ i.e. the work group accepted the
substance of the request (the vendor terminologies are being removed
from the IG and the approach is being reworked with the Vocabulary work
group), without committing in this ticket to the precise
`additionalBindings` mechanic the reporter proposed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's literal request in two parts:

1. **Element-level bindings flip.** In every PHR IG profile that
   currently binds an element to an Apple HealthKit code system or
   value set, change the primary binding (whatever its strength) to a
   standard terminology (LOINC for lab/observation/quantitative
   measures, SNOMED CT for clinical findings/procedures, UCUM for
   units, RxNorm for medications, etc.). Where a one-to-one mapping
   exists, name the standard concept directly; where it does not,
   bind to a value set composed from the appropriate standard
   terminology.
2. **Apple HealthKit as `additionalBindings`.** For each affected
   element, retain the Apple HealthKit code system / value set as an
   `additionalBindings` entry on the `ElementDefinition.binding`,
   typically with `purpose = candidate` (or `preferred` / `ui`,
   depending on intent), so implementations consuming Apple Health
   data can still recognize the original codes.
3. **ConceptMap artifacts.** Publish a `ConceptMap` resource in the
   IG for each Apple HealthKit code system whose codes are referenced
   by the IG, mapping Apple HealthKit concepts to the chosen standard
   concept(s). Reference these `ConceptMap`s from a new "Terminology"
   narrative page (this aligns with the companion request in
   FHIR-49616).

#### Justification

This is exactly the structure FHIR's binding model was designed to
support: a standard primary binding for downstream interoperability
plus `additionalBindings` to record other relevant code systems an
implementation may encounter. It directly addresses the reporter's
"medical grade" concern by ensuring conformant resources can be
exchanged with non-Apple systems using HL7-recognized vocabularies,
while still giving Apple-Health-fed implementations a documented path
for the source codes. It also dovetails with the authoring team's
stated plan (FHIR-49616 comment) to provide a `ConceptMap` between
Apple HealthKit and standard FHIR terminologies тАФ Disposition A simply
makes that ConceptMap normative-shaped (additionalBindings) rather
than a side artifact.

---

### Disposition B: Alternative Approach

#### Proposal

Do not bind Apple HealthKit terminologies anywhere in the published
IG profiles at all. Instead:

1. **Remove all Apple HealthKit `CodeSystem` and `ValueSet` artifacts
   from the IG.** Profile bindings reference only standard
   terminologies (LOINC, SNOMED CT, UCUM, RxNorm, FHIR-defined
   terminologies, etc.).
2. **Publish a stand-alone `ConceptMap` companion artifact** (still
   inside the IG, but not referenced by any profile binding) that
   maps Apple HealthKit codes to the standard codes used by the IG.
   This is the artifact implementations doing Apple-Health ingest
   would consult.
3. **Add a Terminology narrative page** that explains: the IG binds to
   standard terminologies; the `ConceptMap` exists to help
   implementers translate Apple HealthKit (and, in the future, Google
   / Common Health) data into the IG's profiles; and Apple HealthKit
   codes are not part of the IG's conformance surface.

This is, in effect, the "Removed, per HL7 leadership" path captured
in the resolution note, formalized: vendor codes leave the
conformance model entirely and live only as a translation aid.

#### Justification

This approach is the cleanest fit with historical HL7 practice and
with the resolution that has already been agreed (5-0-0 on
*Persuasive with Modification* тАФ "Removed, per HL7 leadership"). It
also fully sidesteps the IP / trademark concern raised in
[FHIR-49616](https://jira.hl7.org/browse/FHIR-49616): if HealthKit
codes appear only as the *source* side of a translation `ConceptMap`,
the IG is not asserting any conformance dependency on an Apple
artifact, which is far easier to defend than `additionalBindings`
inside conformance resources. The trade-off vs. Disposition A is that
implementers ingesting Apple Health data lose the
machine-discoverable signal on each `ElementDefinition` that an Apple
HealthKit equivalent exists; they have to consult the `ConceptMap`
out-of-band. For an IG whose stated philosophy is "medical grade"
that is a defensible trade.

---

### Disposition C: Decline

#### Proposal

Reject the request to remove or de-prioritize Apple HealthKit
terminologies, and keep them as the primary binding (with strength
`preferred` or `extensible`) on the affected PHR profile elements.
Publish mappings as informative companions only.

#### Justification

A defensible argument exists that Apple HealthKit (and Common Health)
are de-facto the largest sources of consumer-collected health data
today, and an IG explicitly scoped to *Personal* Health Records may
serve implementers better by binding to the vocabulary their data
actually arrives in. Standard terminologies (LOINC / SNOMED CT) often
do not have a coded concept that exactly matches consumer-grade
HealthKit categories, and forcing an extensible binding to a
standard value set risks pushing implementers into ad-hoc
text-only `Coding`s. This option preserves implementer ergonomics at
the cost of HL7 convention.

This option is offered for completeness and is **not** recommended:
it directly contradicts the work group's recorded 5-0-0 vote and HL7
leadership's directive to remove the vendor terminologies, and it
does not address the IP / trademark concern in
[FHIR-49616](https://jira.hl7.org/browse/FHIR-49616).

---

### Recommendation

**Recommended disposition: B (Alternative Approach)**

Disposition B is the closest faithful expression of what the work
group has already voted to do (*Persuasive with Modification:
"Removed, per HL7 leadership. Will adjust approach with vocab's
assistance. 5-0-0"*). The reporter's underlying need тАФ a PHR IG whose
conformance surface uses HL7-recognized vocabularies тАФ is fully met
by removing Apple HealthKit `CodeSystem` and `ValueSet` artifacts
from profile bindings and confining them to a translation
`ConceptMap`. It also disposes of the companion ticket
[FHIR-49616](https://jira.hl7.org/browse/FHIR-49616) cleanly: with no
HealthKit artifacts in the conformance model, the trademark / IP
concern is materially reduced and the requested Terminology
explanation page becomes the natural home for both the rationale and
the `ConceptMap` reference.

Disposition A is a reasonable secondary option and could be revisited
if implementer feedback after publication shows that an
out-of-conformance `ConceptMap` is being ignored in practice;
`additionalBindings` would then be the right tool to re-introduce
the Apple HealthKit signal on a per-element basis without
compromising the standard primary bindings. Disposition C should be
declined for the reasons recorded in its own justification and is
inconsistent with the existing vote.

The work group should coordinate the implementation of Disposition B
with the Vocabulary work group (as the resolution note already
commits) and should plan to apply the same pattern to any future
Common Health (Google / Android) terminologies that enter the IG, so
the precedent is set once.
