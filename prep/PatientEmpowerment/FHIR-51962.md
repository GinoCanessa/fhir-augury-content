## Summary

The reporter notes that the "How to read this guide" section on the
home page of the **US PACIO Advance Directive Interoperability (ADI)**
implementation guide is out of date and no longer reflects the IG's
current contents. Specifically, the **General Guidance** bullet omits
several pages that have since been added to the guide — Signatures and
Search Parameters are called out as concrete examples. The request is a
narrative-only documentation fix: re-align the home page index with
what the menu / table of contents actually contains and improve the
descriptions so implementers can find the right page for their task.

The ticket has already been **resolved as Persuasive** by the work
group, and the resolution explicitly agrees on two actions: (1) better
align the home page "How to Read" section with the menu format, and
(2) make the menu section descriptions more descriptive so they guide
the implementer. The ticket was approved as part of ADI Block Vote 4
(May Terry / Yolanda Liu, 9-0-0).

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with this ticket in Augury, so this review is based
solely on the ticket itself and its resolution.

## Proposed Dispositions

> Note: The work group has already resolved this ticket as
> **Persuasive** with a clear scope (re-align the home page section
> with the menu and add more descriptive section blurbs). The
> dispositions below are framed to support the workgroup discussion of
> how that fix should land in the IG narrative.

### Disposition A: Accept As Requested

#### Proposal

Update the **"How to read this guide"** section on the ADI IG home
page so its outline matches the IG's current top-level menu, and
expand the per-section descriptions so an implementer can decide
which page to read first.

Concrete authoring actions for the IG narrative (home page intro
file in the ADI IG, typically `input/pagecontent/index.md` or the
equivalent home-page narrative for the ADI publication):

- Restructure the section as a one-to-one mirror of the published
  menu (Background / General Guidance / Use Cases / Artifacts /
  Downloads, etc., as currently rendered).
- Under **General Guidance**, enumerate every child page that
  currently appears in the menu, including (at minimum) the pages
  called out as missing in the ticket — **Signatures** and **Search
  Parameters** — plus any other pages added since the previous
  revision.
- For each enumerated section / page, add a 1-2 sentence description
  that tells the reader what they will find on that page and which
  audience it's aimed at (e.g., "Signatures — describes how
  Provenance and DocumentReference are used to convey witness and
  notary signatures on advance directive documents.").
- Re-validate links so each bullet resolves to the actual page in
  the published IG.

#### Justification

This matches the work group's resolution verbatim ("better align the
home page 'How to Read' section with the menu format" and "be more
descriptive on the menu sections to guide the implementer") and was
approved 9-0-0 in Block Vote 4. The change is narrative-only — it
touches only the home-page intro narrative, not StructureDefinitions,
ValueSets, or examples — so it carries no conformance impact and no
risk of breaking implementations. It directly addresses the reported
defect (missing pages such as Signatures and Search Parameters under
General Guidance).

---

### Disposition B: Alternative Approach

#### Proposal

Instead of curating the "How to read this guide" outline by hand,
**generate it from the IG menu definition** so that it cannot drift
again. Specifically:

- Replace the static bullet list with an IG Publisher
  include/template (e.g., a `{% include menu-overview.md %}` style
  fragment, or a small narrative block generated from the ImplementationGuide
  resource's `definition.page` tree) so the home-page outline is
  always in sync with the menu.
- Keep authored, hand-written descriptions per section (those still
  need to be human-written), but render the *list* of sections /
  pages from the same source as the menu.
- As part of the same change, still backfill the missing
  descriptions for **Signatures** and **Search Parameters** (and any
  other pages currently lacking a description), as called out in the
  ticket.

#### Justification

The root cause described in the ticket is *drift* — the home-page
outline lagged behind newly added pages like Signatures and Search
Parameters. A purely manual fix (Disposition A) re-aligns the
outline today but leaves the same drift mechanism in place, so the
same defect is likely to recur the next time a page is added. Driving
the outline from the IG menu / `ImplementationGuide.definition.page`
removes that class of defect entirely while still satisfying the
work group's two stated goals. The trade-off is a small amount of
extra authoring complexity in the IG build (one include or templated
section instead of a static list), which is well-understood
territory for the IG Publisher.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave the home-page "How to read this guide"
section as-is, on the grounds that the live menu / table of contents
already gives implementers a complete and authoritative listing of
the IG's pages, and the home-page section is intended as a curated
reading order rather than an exhaustive index.

#### Justification

This option is included for completeness, but it is **not a
defensible disposition for this ticket**. The reporter has identified
a concrete factual inaccuracy (pages that exist in the IG are not
listed in the home-page section that purports to enumerate them),
and the work group has already agreed that the home-page section
should align with the menu and be more descriptive. Declining would
leave a known incorrect/incomplete narrative in a published HL7 IG,
which is hard to justify against the standard editorial bar for
implementation guides.

---

### Recommendation

**Recommended disposition:** A

The work group has already resolved this ticket as Persuasive and
approved it 9-0-0 in Block Vote 4 with a scope that exactly matches
Disposition A: realign the home-page section with the menu and make
the section descriptions more useful to implementers. Disposition A
is low-risk, narrative-only, and directly closes the reported defect
(Signatures, Search Parameters, and any other missing pages get
added with descriptions). Disposition B is genuinely attractive as a
follow-up — generating the outline from the ImplementationGuide menu
would prevent this class of drift from recurring — but it expands
the scope beyond what the work group voted on and is better tracked
as a separate IG-tooling improvement ticket. Disposition C is not
recommended; it would leave a known inaccuracy in a published IG.
The suggested path is therefore to land the editorial fix now under
Disposition A, and optionally open a follow-up ticket to consider
the menu-driven approach from Disposition B.
