## Summary

The reporter (Yunwei Wang) observes that the **ADI Facilitator** profile in
the US PACIO Advance Directive Interoperability (ADI) IG is incorrectly
categorized in the IG's narrative. ADI Facilitator inherits from
`USCorePractitionerRoleProfile`, so it is conceptually and structurally a
`PractitionerRole` profile, but the IG's "Profile by Resource Type"
narrative page lists it under `Practitioner`. The reporter additionally
notes that the ADI Facilitator profile is not listed at all on the IG's
`CapabilityStatement` page. The work group has accepted the report as
persuasive — the categorization is acknowledged as an error in the
narrative content of the IG, not a structural issue with the profile
itself. A fix has already been pre-applied via PR
[HL7/fhir-pacio-adi#192](https://github.com/HL7/fhir-pacio-adi/pull/192),
and the disposition was approved as part of ADI Block Vote 4 (9-0-0).
There are no related Jira tickets surfaced via cross-references and no
Zulip discussions reference this ticket key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already recorded on the ticket and pre-applied in
[HL7/fhir-pacio-adi#192](https://github.com/HL7/fhir-pacio-adi/pull/192):

1. In the ADI IG's "Profile by Resource Type" narrative page (markdown
   under the IG's input/pages content), move the **ADI Facilitator**
   profile entry out of the `Practitioner` section and into the
   `PractitionerRole` section, matching the actual `baseDefinition`
   (`USCorePractitionerRoleProfile`).
2. Add the **ADI Facilitator** profile to the IG's `CapabilityStatement`
   page so the profile is discoverable via the published
   CapabilityStatement listing alongside other ADI profiles.
3. Confirm the published TOC / left-nav reflects the corrected
   categorization after IG publisher run.

No StructureDefinition change is required — this is a narrative and
CapabilityStatement-listing fix only; the profile's `baseDefinition`
already correctly points at `USCorePractitionerRoleProfile`.

#### Justification

The reporter's observation is factually correct: a profile derived from
`USCorePractitionerRoleProfile` is, by FHIR's profile mechanics, a
`PractitionerRole` profile. The "Profile by Resource Type" narrative is
an aid for implementers navigating the IG; mis-categorizing the profile
under `Practitioner` actively misleads readers and breaks the implicit
contract between the narrative grouping and the profile's resource
type. Likewise, omitting the profile from the CapabilityStatement page
hides a conformance-relevant artifact from implementers building or
validating ADI exchanges. The correction is low-risk (narrative + page
listing only), preserves all existing canonical URLs and conformance
content, and has already been applied in PR#192 and ratified by ADI
Block Vote 4 (9-0-0). FHIR design principles favor publishing accurate
navigational metadata that matches the underlying conformance graph.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the categorization fix from Disposition A, and additionally treat
this as a trigger to **automate the "Profile by Resource Type" narrative
page** rather than maintain it by hand. Specifically:

1. Replace (or back) the hand-curated markdown grouping with a generated
   list driven by each profile's `baseDefinition` (or its ultimate base
   resource), produced as part of the IG publisher pipeline (e.g. a
   page-list / fragment generator, or a `{% include %}` over a
   generated index).
2. Generate the CapabilityStatement profile list from the same
   resolved-base index so future profiles cannot silently be omitted.
3. Land the immediate ADI Facilitator fix from Disposition A as the
   first user of the generated index, so the narrative goes live
   correctly even before the generator is fully adopted.

#### Justification

The root cause here is not a one-off authoring slip — it is that two
discoverability surfaces (the resource-type narrative and the
CapabilityStatement page) are maintained by hand and can drift from the
actual `baseDefinition` graph. Since IG-wide automation is out of scope
for the ADI work group's immediate ballot cycle and would touch IG
tooling shared across HL7, treating it as an alternative rather than the
default keeps the per-ticket fix small while flagging the systemic
issue. Trade-off vs. Disposition A: higher implementation burden and
broader review surface, but eliminates a recurring class of defects.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the actual conformance
artifacts are already correct — `ADI Facilitator`'s `StructureDefinition`
correctly inherits from `USCorePractitionerRoleProfile`, so any
implementer relying on machine-readable conformance content is unaffected.
Treat the narrative mis-grouping and CapabilityStatement omission as
non-normative rendering issues that do not warrant a ballot-tracked
change.

#### Justification

A defensible decline could be argued only if the work group held that
narrative pages and CapabilityStatement listings are advisory rendering
and not part of the IG's conformance surface. Under that view, a
machine-readable consumer of the IG would already correctly resolve
ADI Facilitator as a `PractitionerRole` profile via its
`baseDefinition`, and the issue is cosmetic.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as **Persuasive**, the fix has been
pre-applied in PR
[HL7/fhir-pacio-adi#192](https://github.com/HL7/fhir-pacio-adi/pull/192),
and ADI Block Vote 4 ratified the disposition 9-0-0. Disposition A
formalizes that outcome. It is a narrative-only correction with no
StructureDefinition impact, no breaking change for implementers, and no
canonical URL churn — the cost is essentially zero and the benefit is a
faithful, navigable IG. Disposition B's automation idea is sound and
worth raising as a separate follow-up against the IG tooling, but
holding this ticket for it would block a ratified, low-risk fix.
Disposition C understates the role of the "Profile by Resource Type"
narrative and CapabilityStatement page as primary discoverability
surfaces for implementers and conflicts with the work group's already
recorded position. Confirm Disposition A and close.
