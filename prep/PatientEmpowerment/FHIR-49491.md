## Summary

Rob McClure (TSMG) flagged that the Personal Health Record (FHIR) IG ships
many code systems and value sets marked as `experimental` whose names
explicitly reference proprietary platforms — most notably **Apple
HealthKit**, but the same critique applies to the Android Common Health
mappings the IG was building out. The objection is twofold: (1) naming
artifacts after a vendor implies a level of vendor endorsement that has
not been obtained ("did Apple approve this?"), and (2) "experimental"
artifacts are intended for spec testing, not for normative-looking,
proprietary-named terminology shipped inside an IG. The reporter asked
the Patient Empowerment WG to bring this to TSMG to discuss the right
home for "just for testing" terminology.

The ticket has already been worked. Comments document a TSMG meeting on
2025-07-24 where the recommendation was to send the device-mapping
crosswalk to LOINC (contact: Eza Hafeza, Regenstrief), with Pascal
Pfiffner identified for Apple-side guidance. A staged sequence of pull
requests against `HL7/personal-health-record-format-ig` then implemented
the resolution: PR #71 began the rework, PR #78 removed the
`experimental` flag from the remaining codes, and PR #80 removed the
proprietary-named code systems entirely (handing them to LOINC) while
preserving the FHIR profiles without the experimental codes. The
proposed disposition is **Persuasive with Modification** (Hisashi moves,
Sarah seconds, 0-0-3) on the basis that the substantive concern was
addressed by deletion + LOINC handoff rather than by renaming in place.

No directly-linked Jira tickets were found in cross-references, and no
Zulip threads mention this ticket key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the workgroup's already-staged plan as-is: in
`HL7/personal-health-record-format-ig`, (1) delete every code system
and value set whose name embeds a vendor / product identifier (Apple
HealthKit, Android Common Health) — these have been forwarded to LOINC
via the attached `PGHD_codemapping.xlsx`; (2) clear `experimental =
true` on the remaining PGHD value sets that should ship in the IG
(notably `pcd-sleep-observation-code`); (3) keep the FHIR profiles that
referenced those codes but rebind them to LOINC codes (or to the
narrowed, non-vendor-named value sets) once LOINC publishes; (4) update
IG narrative pages so they no longer advertise vendor-branded
"experimental" terminology. This is what PRs
[#71](https://github.com/HL7/personal-health-record-format-ig/pull/71),
[#78](https://github.com/HL7/personal-health-record-format-ig/pull/78),
and [#80](https://github.com/HL7/personal-health-record-format-ig/pull/80)
together implement, and matches the resolution text already attached to
the ticket. Mark the ticket **Persuasive with Modification** and place
on Block Vote 1.

#### Justification

This is the path the WG has already executed and that TSMG (via the
2025-07-24 meeting) recommended. It honors two FHIR principles
simultaneously: (a) `experimental` artifacts must not look like
shippable, vendor-endorsed terminology, and (b) terminology that *does*
have implementer value should live with the appropriate steward —
LOINC for clinical observations, including device-derived ones. The
PR sequence demonstrates the WG can execute the change without
collateral damage to the surrounding profiles, and it cleanly closes
the reporter's two stated concerns (vendor branding + misuse of
`experimental`).

---

### Disposition B: Alternative Approach

#### Proposal

Same end-state for the IG (delete vendor-named code systems, clear
`experimental` on the remainder), **plus** preserve the
device-mapping content as a *companion* artifact rather than only as
a LOINC submission. Specifically: publish the
`PGHD_codemapping.xlsx` crosswalk as a non-normative supporting page
inside the IG (e.g., `mappings.html`) that links proprietary platform
identifiers to the LOINC codes that replace them, with explicit
"informative — not for conformance" framing. This gives
implementers a discoverable migration aid from the previous
experimental code systems to the LOINC-stewarded successors without
re-introducing vendor-named `CodeSystem` resources.

#### Justification

The reporter's substantive worry — that vendor-named experimental
artifacts read as endorsement and as shippable terminology — is fully
resolved by deletion. But the *implementer need* that motivated those
artifacts (mapping Apple HealthKit / Android Common Health payloads
into FHIR observations) does not go away when the codes are deleted.
A non-normative mappings page keeps the migration path visible
without misusing `experimental` and without standing up vendor-named
FHIR resources. It also gives the WG a place to maintain the
crosswalk over time as the LOINC-side codes evolve, and avoids
orphaning early adopters who were already binding to the deleted
code systems.

---

### Disposition C: Decline

#### Proposal

Decline the change request and retain the existing Apple HealthKit /
Android Common Health code systems and value sets in their current
`experimental` form, on the grounds that `experimental = true` is
itself the standards-defined signal that an artifact is provisional
and not for production binding, and that vendor-name strings inside
artifact identifiers are descriptive (the data really *is* shaped by
those platforms) rather than endorsement claims.

#### Justification

There is a defensible reading of the FHIR `experimental` flag where
it already does the job the reporter is asking the names to do —
i.e., consumers who honor `experimental` will not bind production
systems to these codes, and the names are useful precisely because
they tell implementers which proprietary data shape is being
illustrated. Declining would also avoid the breaking-change cost of
deleting code systems that some early adopters may already reference.
However, this reading is weak in the face of the actual TSMG
feedback and the LOINC channel that has already been opened.

---

### Recommendation

**Recommended disposition: A (Accept As Requested)** — with the
optional carrying of B's mappings page as a follow-on enhancement.

The WG has already done the work, TSMG has already weighed in, LOINC
has agreed to receive the crosswalk, and three pull requests
(#71, #78, #80) have been staged or merged that together implement
exactly what the reporter asked for. The resolution text on the
ticket already records this as Persuasive with Modification with a
clean motion (Hisashi/Sarah, 0-0-3). Disposition A simply ratifies
that work and lets it go to Block Vote 1. Disposition C is hard to
defend now that LOINC has opened a path for the substantive content.
Disposition B is appealing as a migration aid for early adopters of
the deleted code systems, and the WG should consider opening a
follow-up ticket to add a non-normative mappings page once the LOINC
codes are published — but that work should not block ratifying the
present resolution.
