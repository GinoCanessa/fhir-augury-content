## Summary

Virginia Lorenzi proposes two coupled edits to the Personal Health Record
(FHIR) implementation guide. First, she asks for a rewording of the
explanatory paragraph that currently uses the term "health record" when
discussing whether a PHR file must contain a medication reconciliation,
arguing that "record" has an explicit clinical meaning that makes the
current text confusing — the section should make clear that this IG is an
*interoperability / packaging* specification, not a definition of what a
PHR must contain. Second, she asks that the PHR file-format extension be
renamed from `.sphr` to `.phr`. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were located in FHIR Augury for this ticket, so
the request stands on its own without surrounding community discussion.

## Details

**Description:**

Recommend changing:

> But whether or not the .sphr health record contains a medication
> reconcilitation or not is left completely up to the implementor. The
> only requirement for conformance is that if the data is included in
> the .sphr file, it use FHIR data schemas.
>
> In effect, this implementation guide does not specify what
> functionality should be included the PHR, but it does specify medical
> charting format that the record uses. It's about the folder and
> record keeping, not the specific contents of the records themselves.

to:

> But whether or not the .phr FHIR format contains a medication
> reconciliation or not is left completely up to the PHR implementer.
> The only requirement for conformance is that if the data is included
> in the .phr file, it use FHIR data schemas as described in this
> specification.
>
> In effect, this implementation guide does not specify what
> functionality should be included the PHR, but it does specify a
> format that can be used for the PHR as it interoperates with other
> systems.  It's also more about the packaging of the content for
> exchange and not the content itself.

Rationale: the current wording talks about "records" which has an
explicit meaning in healthcare and can be confusing here. The reporter
wants to be clear that this is an interoperability spec, not a PHR
definition spec. Although that is the intent of the section, the
current wording in the spec is confusing.

Also — the format should be changed to `.phr` instead of `.sphr`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply both changes verbatim in the Personal Health Record (FHIR) IG:

1. Replace the two-paragraph block in the relevant narrative section
   with the reporter's suggested rewording, swapping "implementor" →
   "PHR implementer", "the .sphr health record" → "the .phr FHIR
   format", "the .sphr file" → "the .phr file", "medical charting
   format that the record uses … folder and record keeping … contents
   of the records themselves" → "a format that can be used for the
   PHR as it interoperates with other systems … packaging of the
   content for exchange and not the content itself", and adding "as
   described in this specification" to the conformance statement.
2. Rename the file extension everywhere it appears in the IG —
   narrative pages, examples, capability statements, and any
   normative text — from `.sphr` to `.phr`. Update the IG glossary /
   format section accordingly so the canonical extension is `.phr`.

#### Justification

The reporter is the WG's principal author of this content and has
provided concrete, drop-in replacement text. The rewording removes a
genuine ambiguity: "record" carries clinical-record connotations that
clash with the IG's actual scope (a packaging/interop format). The new
text is consistent with how the IG positions itself elsewhere
("packaging of the content for exchange") and does not introduce new
normative obligations. The `.sphr` → `.phr` rename is a cosmetic
identifier change in a Patient-Empowerment IG that has not yet
established broad implementation footprint outside HL7; doing it now,
together with the wording fix, avoids a second churn cycle later.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the wording rewrite (item 1) as written, but split off the
extension rename (item 2) into a follow-on ticket and make the rename
explicit and traceable:

- Apply the narrative replacement now to remove the "record"
  ambiguity.
- For the extension change, file a separate tracker that (a) lists
  every IG page, example, profile, capability statement, and figure
  that mentions `.sphr`, (b) defines the migration story for any
  existing implementer prototypes (dual-accept `.sphr` and `.phr` for
  one publication cycle, or a one-time clean cut with a release-note
  callout), and (c) updates any media-type / IANA registration intent
  the WG has discussed for the format.

#### Justification

The two requests have very different blast radii. The rewording is
purely narrative and risk-free. The extension rename touches every
mention of the format and may also affect any media-type the WG plans
to register, plus prototype implementations and downstream training
material. Bundling them risks an incomplete rename (some `.sphr`
mentions left behind) and obscures the rename in the changelog.
Splitting the work makes the rename auditable and lets the WG decide
on a transition policy explicitly rather than implicitly.

---

### Disposition C: Decline

#### Proposal

Decline the change as filed. Specifically, decline the `.sphr` → `.phr`
rename on the grounds that `.sphr` was chosen deliberately to avoid
collision with non-FHIR uses of `.phr` (a generic, vendor-overloaded
extension), and decline the wording rewrite as unnecessary because the
existing paragraph, read in context with the surrounding section,
already conveys that the IG defines a packaging format rather than PHR
contents.

#### Justification

`.phr` is a widely-overloaded extension used by various proprietary
PHR products and by unrelated tooling; reusing it for a FHIR-defined
package risks confusing implementers and end-users about whether a
given `.phr` file is "the FHIR PHR format" or one of several legacy
products. Keeping `.sphr` preserves a clear, FHIR-specific identifier.
On the wording, "record" is a perfectly ordinary English term and the
surrounding section already frames the IG as packaging-oriented;
patching one paragraph in isolation invites whack-a-mole editing
without resolving any concrete interoperability problem.

---

### Recommendation

**Recommended disposition:** B

This ticket bundles a clearly-good editorial fix with a
spec-identifier change that has wider consequences than its one-line
description suggests. The narrative rewrite is straightforward,
non-normative, authored by the WG's content lead, and removes a real
"record vs. format" ambiguity that confuses readers — there is no
reason to delay it. The extension rename, however, deserves its own
disposition: it touches every page, every example, any capability
statement, and any media-type registration the WG plans, and it is the
kind of change that is painful to back out if `.phr` later turns out
to collide badly with implementer expectations (the concern raised in
Disposition C). Separating the two lets the WG ship the editorial fix
this cycle while giving the rename the deliberate, repo-wide
treatment (and explicit transition note) it warrants. Disposition A
is acceptable if the WG is confident the rename is safe and is willing
to do the sweep in one PR; Disposition C under-serves the reporter's
valid editorial concern and should only be chosen if the WG concludes
both halves of the request are wrong.
