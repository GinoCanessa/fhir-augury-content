## Summary

Reporter Svitlana Vedmed identifies a typo in the FHIRPath expression of
the `HCA-section-entries` invariant on the `ADI-PACPComposition`
StructureDefinition (`http://hl7.org/fhir/us/pacio-adi/StructureDefinition/ADI-PACPComposition`)
in the US PACIO Advance Directive Interoperability IG. The expression
references the resource type as `relatedPerson` (lowercase initial), but
FHIRPath is case-sensitive and the actual FHIR resource type is
`RelatedPerson`. As written, the `is relatedPerson` test cannot match
any resource, so the `entry.where($this.resolve() is RelatedPerson)`
branch of the constraint is effectively dead code, defeating the
constraint's intent of requiring at least one `Consent` and at least one
`RelatedPerson` entry when entries are present. The reporter proposes
the obvious fix: change `relatedPerson` to `RelatedPerson` in the
invariant expression. The ticket is marked Resolution = Persuasive, and
the resolution comment from May Terry indicates the change has already
been applied to the CI Build via `HL7/fhir-pacio-adi#95`. No related
Jira tickets and no Zulip discussions surfaced in cross-references.

## Details

**Description:**

> Constraint with key = `HCA-section-entries` that is used for
> StructureDefinition with url = `http://hl7.org/fhir/us/pacio-adi/StructureDefinition/ADI-PACPComposition`
> looks like:
>
> ```
>       "expression": "entry.exists().not() or (entry.where($this.resolve() is Consent).exists() and entry.where($this.resolve() is relatedPerson).exists())",
> ```
>
> As [FHIRPath is a case-sensitive language](http://hl7.org/fhirpath/N1/#case-sensitivity)
> there is no such type `relatedPerson` in FHIR, only a type `RelatedPerson`.
> That is why the constraint should be change to:
>
> ```
>       "expression": "entry.exists().not() or (entry.where($this.resolve() is Consent).exists() and entry.where($this.resolve() is RelatedPerson).exists())",
> ```

**Resolution Description:**

> Agreed. will fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US PACIO ADI IG (`HL7/fhir-pacio-adi`), update the
`HCA-section-entries` invariant on the `ADI-PACPComposition`
StructureDefinition (canonical
`http://hl7.org/fhir/us/pacio-adi/StructureDefinition/ADI-PACPComposition`)
so the FHIRPath expression uses the correctly-cased resource type name:

```
entry.exists().not() or (entry.where($this.resolve() is Consent).exists() and entry.where($this.resolve() is RelatedPerson).exists())
```

i.e., change `is relatedPerson` to `is RelatedPerson`. Leave the
invariant key, severity, and human description unchanged. Regenerate
snapshots and re-run the IG publisher so the change propagates to
derived StructureDefinition snapshots, validator packages, and the
published IG. Confirm via the publisher's QA that the invariant now
parses against the FHIRPath grammar with a defined type test, and add
(or refresh) example bundles that cover both branches of the constraint
so the fix is regression-tested.

#### Justification

This is a straightforward defect: FHIRPath is case-sensitive
(http://hl7.org/fhirpath/N1/#case-sensitivity) and `relatedPerson` is
not a defined FHIR resource type. As written, the `is relatedPerson`
type test silently fails to match anything, which means the
`entry.where(... is relatedPerson).exists()` branch is always false,
and the overall constraint reduces to "no entries OR at least one
Consent AND false" — i.e., no Bundle with entries can satisfy it
unless `entry.exists().not()` holds. That is clearly not the authors'
intent, and the requested correction restores the invariant to the
behavior described by its accompanying human text. The fix has already
been applied to the CI Build via `HL7/fhir-pacio-adi#95` per the
resolution comment, and the ticket is already marked Resolution =
Persuasive, so accepting as requested simply ratifies what has been
implemented.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the case-correction (Disposition A's edit) and additionally
harden the invariant's robustness and clarity in the same change:

1. Replace the `$this.resolve() is RelatedPerson` style with
   `resolve() is RelatedPerson` (drop redundant `$this.`) and consider
   constraining over `entry.resource` directly when slicing by resource
   type is intended; this matches the more common idiom used elsewhere
   in HL7 IGs and tends to round-trip more cleanly through validators.
2. Tighten the human description and severity statement of
   `HCA-section-entries` so the requirement ("when present, entries
   must include at least one Consent and at least one RelatedPerson")
   is unambiguous to implementers reading the IG.
3. If the section is intended to require *exactly one* Consent and
   *exactly one* RelatedPerson (a question worth confirming with the
   author), state that and reflect it in the FHIRPath using `count()`.
4. Add explicit slicing on the `entry` element by resolved resource
   type for `Consent` and `RelatedPerson` so the cardinality is
   expressed structurally, with the invariant remaining as a belt-and-
   braces FHIRPath check.

#### Justification

The literal fix in Disposition A is correct but does not address the
fact that this invariant was published with a typo that existed for a
non-trivial period — which suggests the constraint is not being
exercised by examples or by automated review. Using more conventional
FHIRPath, clarifying the human-readable description, and (optionally)
expressing the cardinality through entry slicing makes the
requirement visible at the model level, where implementers and
validators are more likely to enforce it correctly. The trade-off is
slightly more authoring work and a larger diff (and possibly a
substantive change to the structural model if slicing is added),
which carries higher review and regression cost than a one-character
case fix.

---

### Disposition C: Decline

#### Proposal

Decline the requested change and leave the `HCA-section-entries`
invariant on `ADI-PACPComposition` unchanged.

#### Justification

There is no defensible basis for declining: the reporter is correct on
the FHIRPath specification (case-sensitivity is normative), the
resource type `relatedPerson` does not exist in FHIR, and the current
expression therefore cannot enforce the requirement its human
description states. Declining would knowingly leave a defective
invariant in a published IG. This option is included for completeness
only.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified an unambiguous defect in a published
invariant, the fix is a single-character case correction, the change
has already been implemented in the CI Build via
`HL7/fhir-pacio-adi#95`, and the ticket is already resolved as
Persuasive. Disposition A ratifies the in-flight fix with the smallest
possible diff and the lowest regression risk, which is appropriate for
a "groom-for-vote" item where the working group's job is to confirm
the resolution rather than reopen design. Disposition B's hardening
ideas (idiomatic FHIRPath, clearer description, optional slicing) are
worth tracking as separate enhancements against `ADI-PACPComposition`
but should not block ratifying this typo fix; if the working group
wants any of B's items, they should be raised as new tickets so they
can be scoped and reviewed independently. Disposition C is not
defensible and is listed only to make the trade-off explicit.
