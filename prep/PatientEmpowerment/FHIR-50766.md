## Summary

The ticket reports that the Personal Health Record (PHR) FHIR IG defines
numerous profiles whose StructureDefinitions contain no differential
content relative to the resource they profile (PHR Patient and PHR
Provenance are cited as examples). The reporter argues that empty,
zero-differential profiles are redundant and confusing for implementers
and recommends that all such profiles be removed from the PHR IG unless
there is an explicit reason to retain them.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the FHIR Augury cross-reference and search indexes for this
ticket key, so the request is being reviewed in isolation. The Jira
record itself shows the ticket already carries a recorded resolution of
"Persuasive with Modification": the workgroup chose to delete the
`Phr{ResourceType}` profiles except for `PhrBundle`, removed the
`NutrientOuttake` and `Sleep` logical models, and re-pointed the
experimental Sleep logical model at the PGHD Sleep profiles. This review
is therefore primarily about confirming and documenting that disposition
in a workgroup-friendly form.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove every `Phr{ResourceType}` profile in the Personal Health Record
IG whose StructureDefinition has an empty `differential` (i.e., does not
add, constrain, slice, fix, or document anything beyond the base
resource). At minimum this includes PHR Patient and PHR Provenance, and
any other PHR profile audit shows to be a pure pass-through of the base
resource. Update the IG's profile index, capability statements,
examples, and narrative pages to reference the base resource directly
in place of each removed profile, and add a brief migration note in the
IG change log explaining the removal. Retain only those profiles where
either (a) a meaningful constraint is intended and just hasn't been
authored yet (in which case file follow-up tickets to add the
differential) or (b) the profile is structurally required by the IG's
contract (e.g., `PhrBundle`, which scopes the document/bundle shape).

#### Justification

A profile that adds nothing to the base resource provides no
conformance value and actively misleads implementers, who reasonably
assume a named profile exists because it constrains something. FHIR's
profiling guidance is that profiles should narrow the base resource;
empty differentials violate that intent. Removing them simplifies the
IG, shrinks the StructureDefinition surface, and makes the remaining
profiles' purpose clearer. The reporter's framing matches what the
workgroup ultimately did in the recorded resolution (delete
`Phr{ResourceType}` except `PhrBundle`), so this disposition is also
the closest match to the existing resolution on file.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of deleting every empty PHR profile, keep the named profiles
but give each one a minimal, deliberate differential that documents
PHR-specific expectations: a `description`, `purpose`, and
`mustSupport` flags on the elements PHR systems are expected to
populate (e.g., `Patient.identifier`, `Patient.name`,
`Patient.birthDate`, `Provenance.recorded`, `Provenance.agent`).
Profiles that cannot be justified even at this level (no PHR-specific
"must support" set, no narrative worth documenting) are then removed,
matching the resolution's treatment of `PhrBundle`. The result is a
smaller set of profiles, but the surviving ones earn their place by
carrying real implementer guidance.

#### Justification

Named profiles per resource are a useful anchor for IG narrative,
example linkage, and capability-statement references; deleting them
outright forces implementers to navigate back to base FHIR for every
PHR-specific note. Adding even modest `mustSupport` and narrative makes
the profiles useful without changing their cardinality or structural
meaning, preserving backward compatibility for any consumer that has
already referenced the profile canonical URL. The trade-off vs.
Disposition A is editorial cost (someone must author the PHR-specific
"must support" set per resource) and the risk that this becomes a
larger spec change than the ticket asks for; the upside is keeping
stable canonical URLs and a clearer per-resource conformance story.

---

### Disposition C: Decline

#### Proposal

Leave the existing `Phr{ResourceType}` profiles in place. Document in
the IG that PHR-named profiles intentionally exist as identity markers
even when empty, so that implementers, registries, and validators can
refer to a stable canonical URL per PHR resource type and so that future
versions can add constraints without renaming.

#### Justification

This would only be defensible if the workgroup had already promised
stable PHR-specific canonical URLs to downstream implementers (for
example as part of a registry, validator configuration, or external
profile reference) and breaking those URLs would cause real harm. The
recorded resolution shows the workgroup did not find that argument
compelling — it deleted the `Phr{ResourceType}` profiles other than
`PhrBundle`. Declining is therefore the weakest option here and is
listed only for completeness.

---

### Recommendation

**Recommended disposition:** A

The reporter's observation is correct: profiles with empty
differentials add nothing to conformance, complicate the IG, and mislead
implementers who expect a named profile to mean something. The
workgroup's own recorded resolution already lands here — removing
`Phr{ResourceType}` profiles except for `PhrBundle` and trimming the
unused `NutrientOuttake` and `Sleep` logical models, while
re-pointing the experimental Sleep model at the PGHD Sleep profiles.
Disposition A formalises that decision and is the cleanest answer to
the ticket as written. Disposition B is a reasonable fallback if the
workgroup later decides PHR needs explicit `mustSupport` per resource,
but that work can be tracked separately on a per-resource basis rather
than holding up the cleanup.
