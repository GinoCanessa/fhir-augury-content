## Summary

Virginia Lorenzi requests that the Security page in the Personal Health
Record (PHR) FHIR Implementation Guide be removed until it is ready and
needed. The reporter notes that the page currently exists as a shell:
its sections are intended to align with the PHR functional model but
have no substantive content, and the page carries a "Work in Progress"
notice. The reporter argues that an empty, work-in-progress security
page makes the entire IG read like a draft that is not ready for
implementation. Additionally, security is often considered out of scope
for IGs of this kind, and the ticket states it is unclear how security
would be in scope for the PHR guide as currently scoped. The
recommendation is to remove the Security page from the PHR IG until
real, scoped content is available to publish there.

There were no cross-referenced Jira tickets, Zulip threads, or GitHub
items returned for this ticket, so the request stands on its own.

## Details

**Description:**

Having a whole page devoted to security with sections that need to align
with the current PHR functional model but with nothing on it and the
note "Work in Progress" makes the spec feel like a draft not ready for
implementation. Security is also often out of scope and it is unclear
of how it would be in scope for the purposes of this guide. Recommend
removing until ready and needed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the Security page from the Personal Health Record (FHIR) IG in
the next publication. Concretely: delete the Security narrative page
from the IG's pages collection, remove its entry from the IG menu / table
of contents, and remove any "see Security" cross-references from sibling
pages (e.g., Home, Conformance, Use Cases). Replace any inbound link with
either nothing or a one-sentence statement on the Home page that says
security guidance is out of scope for the current version of this IG and
will be addressed in a future version once aligned with the PHR
functional model. Do not retain the "Work in Progress" stub.

#### Justification

A published IG page with no substantive content and a "Work in Progress"
banner signals to implementers that the specification is not ready,
which undermines the rest of the guide's credibility and ballot
readiness. Removing the page (rather than leaving an empty placeholder)
is consistent with HL7 publication practice that ballot/STU material
should be substantive. It also matches the reporter's observation that
security is, at present, not clearly in scope for the PHR IG, so
shipping no page is more honest than shipping an empty one. The change
is editorial and low-risk: nothing normative is being removed because the
page contains no normative content today.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the Security page in the IG but rewrite it to be a short, explicit
**scope statement** rather than a structural shell. The replacement
content would: (1) state that comprehensive security guidance is out of
scope for this version of the PHR IG, (2) point readers to the relevant
external sources they should consult for now (e.g., the FHIR core
Security and Privacy Module, SMART App Launch where applicable, and the
PHR functional model's security expectations), and (3) note that future
versions of this IG intend to align security guidance with the PHR
functional model once that work is ready. Remove the empty section
headings and the "Work in Progress" banner.

#### Justification

This preserves a discoverable anchor for implementers who will reasonably
look for "Security" in a PHR IG and would otherwise be confused by its
absence, while still removing the impression that the IG is a draft.
A scoped "what we don't cover and where to look" page is a recognized
and accepted pattern in HL7 IGs and addresses the reporter's underlying
concern (the page reads as unfinished) without losing the navigational
landmark. Trade-off vs. Disposition A: it requires authoring effort and
ongoing maintenance of the external pointers, but it lowers the risk
that future ballot reviewers ask "where is the security page?" and that
implementers assume the IG silently endorses no security guidance at
all.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the existing Security page with its
"Work in Progress" notice, on the grounds that the page is a known
placeholder for forthcoming alignment with the PHR functional model and
that removing it would lose the structural commitment to address security
in a future version.

#### Justification

This is defensible only if the work group has an active, near-term
commitment to populate the page before the next ballot or publication
window, and wants the placeholder to remain as a visible reminder of
that commitment. In that case, the cost of leaving the shell in place
is short-lived and the structural slot is preserved. However, this
disposition does not address the reporter's substantive concern that the
page makes the IG look unfinished, so it should only be chosen if the
work group can commit to filling the page in the very next cycle.

---

### Recommendation

**Recommended disposition:** B

Disposition B (replace the shell with a short, explicit scope statement)
best balances the reporter's concern with the practical reality that
implementers will look for security guidance in a PHR IG. Disposition A
fully resolves the "looks unfinished" problem but leaves a discoverability
gap that ballot reviewers and implementers are likely to flag again,
whereas Disposition C does not address the substantive concern at all
and only makes sense if the work group is ready to commit content
immediately. A short scope page costs little to author, removes the
"Work in Progress" signal that the reporter objected to, gives implementers
a clear pointer to existing security guidance (FHIR Security and Privacy
Module, SMART App Launch, PHR functional model), and preserves a clean
slot to expand once the alignment work with the PHR functional model is
ready. If the work group decides authoring even a scope page is not
feasible for the current cycle, falling back to Disposition A is
preferable to leaving the shell in place.
