## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Complete the Security page of the Personal Health Record (FHIR) IG and add
explicit Functional Model provenance to it. Concretely:

1. Replace the current incomplete narrative on the IG's `security.html`
   (authoring source: the corresponding `input/pagecontent/security.md` —
   exact path to be confirmed once `HL7/phr` is analyzed) with substantive
   content covering each topic the page is supposed to enumerate
   (authentication / authorization, consent, access control, audit, data
   segmentation, breach handling, etc.).
2. Add a short attribution block at the top of the page in the form:
   *"The structure and section list on this page are derived from the
   Personal Health Record System Functional Model (PHR-S FM); see [link]."*
   with a stable hyperlink to the PHR-S FM.
3. For any section whose FHIR-side guidance is not yet ready, leave a
   visible, dated "TODO — see PHR-S FM §X" stub rather than an empty
   heading, so the provenance is preserved even where prose is incomplete.

#### Justification

This is the literal ask: the page is incomplete and the reporter wants the
Functional-Model origin called out. Both halves are low-risk, additive,
narrative-only IG changes — no resources, profiles, or constraints are
affected. Naming the PHR-S FM as the source matches established HL7
practice of cross-referencing the Functional Model from companion FHIR IGs
(this is also reflected in the older PHRS FM R2 wiki references that turned
up in Zulip), and it directly addresses the documentation gap Craig
identified.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than rewriting the Security page in-place, **restructure the IG's
Security guidance into a thin landing page that delegates to authoritative
sources**:

- Replace `security.md` with a short page that (a) states explicitly that
  the section structure is derived from the PHR-S FM and links to it, and
  (b) defers normative security guidance to existing HL7 specs — primarily
  the FHIR core *Security and Privacy Module* and the relevant PHR-S FM
  chapters — rather than restating them.
- Keep only PHR-specific deviations or profile-level notes (e.g., consent
  scoping for patient-curated records, audit expectations for patient-owned
  endpoints) in the IG itself, clearly marked as PHR-specific.
- Add a "What this page is / is not" callout to set reader expectations.

#### Justification

The reporter's underlying need is "this page should not be a confusing,
half-built duplicate of work that lives elsewhere." A delegation page
satisfies that need with substantially less authoring burden, avoids
divergence between the PHR FHIR IG and the FHIR core security guidance, and
removes the risk that an incompletely-written section reads as normative
when it isn't. It still satisfies the reporter's second request — the
Functional Model provenance is called out at the top — and is cheaper to
keep current as the PHR-S FM and FHIR core security guidance evolve.

Trade-off vs. A: readers who expect a self-contained security chapter in
the IG won't find one; they will need to follow links. For a Patient
Empowerment IG focused on PHR mechanics rather than on security
re-specification, that trade is generally favorable.

---

### Disposition C: Decline

#### Proposal

Decline to take action on FHIR-49072 as a standalone editorial ticket and
instead **fold the Security page rewrite into the next planned substantive
revision of the PHR FHIR IG**, tracking it as a documentation-debt item on
the IG rather than as a ballot-affecting change request. Close the ticket
as "Persuasive with Mod" / "Not Persuasive" with a pointer to the IG's
issue tracker.

#### Justification

The ticket carries no concrete content proposal, no comments, no
cross-references, and no Zulip discussion specific to it. Treating an
"incomplete page" report as a Change Request risks consuming a
disposition slot for what is really a backlog item the IG editor can pick
up in normal authoring. Declining keeps the workgroup's CR queue focused
on substantive design questions while still ensuring the work happens via
the IG's own backlog.

This option should only be preferred if the workgroup is confident the IG
editor will in fact pick the page up in the near term; otherwise the ticket
serves as the only durable record of the gap.

---

### Recommendation

**Recommended disposition:** **A**

Disposition A is recommended. The ticket is small, well-scoped, and
literally actionable: write the page, and add a sentence of provenance to
the Functional Model. The reporter is a known HL7 contributor and the
priority is High, which argues against deferral via Disposition C. Option B
is attractive for long-term maintainability and the workgroup may want to
adopt its "delegate to FHIR core security and to PHR-S FM" stance as the
*shape* of the rewritten page, but that is an editorial choice that can be
made while doing A — it does not require declining the request as
literally stated. Either way, the workgroup should also schedule a
`repo-analysis` pass on the PHR FHIR IG repo before authoring begins so
that the implementer touching the page has a current briefing (authoring
roots, generated areas, gotchas) to work from; the absence of that
briefing today is the main reason this prep cannot pin down an exact
source-file path.
