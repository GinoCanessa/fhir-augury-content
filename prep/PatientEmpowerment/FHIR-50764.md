## Summary

The reporter, Virginia Lorenzi, raises a concern that the Personal Health
Record (PHR) FHIR IG contains a "Domain analysis table" that appears to have
been generated by AI and not reviewed by appropriate clinical experts. She
considers the table out of scope for the IG, observes that it uses substantial
medical jargon (covering what looks like anatomy and possibly physiology), and
argues that an unreviewed AI-generated analysis of medical domains should not
appear in a normative HL7 specification. The ticket requests removal of the
Domain analysis table and a careful expert review of the surrounding
narrative before any similar content is published in the spec. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned for this ticket, so the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the AI-generated Domain analysis table from the Personal Health
Record (FHIR) IG narrative entirely. Specifically:

1. Delete the table and its surrounding narrative from the IG page where
   it currently appears (the "domain analysis" / scope section of the PHR
   IG).
2. Replace it with either (a) nothing, or (b) a short, human-authored
   paragraph that describes the IG's clinical scope at a high level
   without enumerating anatomical or physiological domains.
3. Add an editorial note (or ballot-cycle changelog entry) recording that
   the table was removed because it had not been clinically reviewed.
4. Add a working-group editorial guideline that AI-generated content
   describing clinical domains must be expert-reviewed and signed off
   before inclusion in any PHR IG publication.

#### Justification

The reporter's objection is fundamentally an editorial-quality and
governance issue, not a technical-modeling issue. HL7 specifications are
expected to be authoritative; publishing an unreviewed AI-generated
analysis of medical domains in a normative artifact creates real risk of
errors being treated as standard guidance and undermines confidence in
the IG. Removing the table is low-cost (it is narrative, not conformance
material), reversible (a vetted version can be reintroduced later), and
directly addresses the concern raised. Because the ticket priority is
"Highest", a clean removal before the next ballot cycle is the most
defensible action.

---

### Disposition B: Alternative Approach

#### Proposal

Retain the Domain analysis table, but treat it as a non-normative,
clearly-labelled informative appendix rather than part of the IG's main
scope narrative. Specifically:

1. Move the table out of the main scope section into an appendix titled
   something like "Informative: Candidate clinical domains (under
   review)".
2. Add a prominent editorial banner above the table stating that the
   content was machine-generated, has not yet been clinically reviewed,
   and must not be relied upon as authoritative guidance on clinical
   scope.
3. Strip or rewrite obviously jargon-heavy entries so the table reads at
   the same register as the rest of the IG.
4. Open a follow-up tracker for clinician/SME review of each row, with a
   commitment to either promote reviewed rows back into the main
   narrative or remove them in a subsequent ballot.

#### Justification

If the WG believes the table captured useful structure (even imperfectly)
about the clinical scope of a PHR, throwing it away loses that work.
Reframing it as an explicitly informative, clearly-flagged appendix
preserves the analysis as a starting point for SME review while
neutralising the reporter's core concern that unreviewed AI content is
being presented as standard guidance. The trade-off versus Disposition A
is that some readers will still encounter the jargon and may
mis-attribute authority to it; the editorial banner and appendix
placement mitigate but do not fully eliminate that risk.

---

### Disposition C: Decline

#### Proposal

Reject the request to remove the Domain analysis table on the grounds
that it provides useful scoping context, and instead resolve the ticket
by tightening the surrounding narrative to clarify that the table is
illustrative.

#### Justification

This option would only be defensible if the WG concluded that (a) the
table content has already been reviewed by appropriate SMEs (which the
ticket explicitly disputes and which we have no evidence of), and (b)
removing it would meaningfully degrade reader understanding of the IG's
scope. Neither is established here, and the reporter is a long-standing
WG participant raising a clear governance concern at "Highest"
priority. Declining without remediation is therefore the weakest
option.

---

### Recommendation

**Recommended disposition:** A

Remove the AI-generated Domain analysis table from the PHR IG. The
ticket is filed at the highest priority by an experienced WG member and
identifies a governance problem (unreviewed AI content in a normative
HL7 specification describing medical domains) rather than a modelling
disagreement. The cost of removal is low — the table is narrative
content, not conformance material — and removal is reversible: if SME
review later produces a vetted scope description, it can be reintroduced
either as Disposition B's informative appendix or as part of the main
narrative. Disposition B is a reasonable fallback if the WG wants to
preserve the table as a working artifact, but it leaves the same content
in front of readers behind a banner, which only partially answers the
reporter's concern. Disposition C is not recommended; it does not
address the underlying issue and is likely to provoke a re-file.
