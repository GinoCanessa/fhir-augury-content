## Summary

The reporter observes that the PACIO Advance Directive Interoperability (ADI)
IG defines two Observation profiles — `ADIAutopsyObservation` and
`ADIOrganDonationObservation` — whose `Observation.code` element is fixed to
specific LOINC codes (`75782-3` *Thoughts regarding autopsy Narrative -
Reported* and `75781-5` *Thoughts on organ and tissue donations Narrative -
Reported*, respectively). Those same two LOINC codes are also members of the
`UponDeathPreferences` value set
(OID `2.16.840.1.113762.1.4.1115.15`) that is bound to the
`ADIUponDeathPreferences` profile. As a result, a given `Observation` carrying
either LOINC code is simultaneously a legal instance of *both* the
autopsy/organ-donation profile *and* the upon-death-preferences profile, which
creates ambiguity for validators (IG Publisher and others) because there is no
unique `StructureDefinition` to select when validating the resource. The
reporter proposes two resolution options: **Option 1** — keep the
autopsy/organ-donation profiles but remove the two overlapping LOINC codes
from the `UponDeathPreferences` value set; or **Option 2** — retire the two
narrow profiles and use only `ADIUponDeathPreferences` for these
concepts. No cross-referenced Jira tickets, Zulip discussions, or GitHub
items were surfaced for this ticket.

## Details

**Description:**

> Autopsy Observation has Observation.code fixed to LOINC#75782-3 (Thoughts
> regarding autopsy Narrative - Reported), which also happens to be included
> in one of the concept codes in the UponDeathPreferences Value Set
> (http://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1115.15).
>
> There is a similar issue with the ADI Organ Donation Observation profile,
> which is fixed to code LOINC#75781-5 (Thoughts on organ and tissue donations
> Narrative - Reported) which also happens to be included in one of the
> concept codes in the UponDeathPreferences Value Set
> (http://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1115.15).
>
> **Problem / Implications:** This raises an issue of ambiguity where an
> Observation resource where a validator (IG Publisher or other) cannot
> identify which StructureDefinition to use for validating that resource.
>
> **Resolution options:**
>
> - **Option 1:** Keep the ADI Autopsy Observation and ADI Organ Donation
>   profiles and remove those 2 LOINC codes from the Upon Death Preferences
>   Value set.
> - **Option 2:** Remove ADI Autopsy Observation and ADI Organ Donation
>   profiles and only use the Upon Death Preferences profile to identify
>   one's autopsy and organ donation preferences.

## Proposed Dispositions

### Disposition A: Accept As Requested (Option 1 — shrink the value set)

#### Proposal

Apply Option 1 from the ticket: keep `ADIAutopsyObservation` and
`ADIOrganDonationObservation` unchanged (each with its fixed
`Observation.code`), and remove the two overlapping LOINC codes
(`75782-3` and `75781-5`) from the `UponDeathPreferences` value set
(VSAC OID `2.16.840.1.113762.1.4.1115.15`) that is bound to
`ADIUponDeathPreferences.code`. Because the value set is VSAC-hosted, the
change is made in VSAC by the value-set steward (PACIO Project / NLM VSAC
author of record) and a new expansion version is then referenced from the
ADI IG. In the IG, document in the narrative for `ADIUponDeathPreferences`
that autopsy and organ-donation preferences are represented using the
dedicated profiles, and add a profile-selection rule (e.g., in the IG
narrative / validator guidance) that `Observation.code = 75782-3` → use
`ADIAutopsyObservation` and `Observation.code = 75781-5` → use
`ADIOrganDonationObservation`.

#### Justification

This resolves the validator-disambiguation problem at its root: once the
two LOINC codes are no longer members of the `UponDeathPreferences` value
set, an `Observation` carrying either code can match only one profile, and
tools such as the IG Publisher can deterministically pick the right
`StructureDefinition`. It preserves the existing profile hierarchy that
implementers may already be building against, and it is the smallest
published-artifact change that removes the overlap. The cost is borne
almost entirely on the value-set side (a VSAC revision), rather than
requiring profile retirement or instance migration.

---

### Disposition B: Alternative Approach (Option 2 — collapse onto UponDeathPreferences, OR differentiate by category/slicing)

#### Proposal

Two sub-variants, both distinct from literal Option 1:

- **B1 (ticket's Option 2):** Retire `ADIAutopsyObservation` and
  `ADIOrganDonationObservation` and require all autopsy/organ-donation
  preferences to be expressed as `ADIUponDeathPreferences` instances (with
  `Observation.code` drawn from the `UponDeathPreferences` value set).
  Mark the two profiles as *Deprecated* in the current ADI release and
  remove them in the next substantive version.
- **B2 (differentiate without shrinking the value set):** Keep all three
  profiles and the current value set, but add a discriminating constraint
  other than `Observation.code` so validators can pick a unique profile —
  for example, introduce a required `Observation.category` slice
  (e.g., a PACIO-specific category code for "autopsy preference" /
  "organ-donation preference" vs. "general upon-death preference"), and
  update the three profiles' discriminators / `obeys` rules / publisher
  `profile:` slicing accordingly so that the autopsy and organ-donation
  LOINC codes, when accompanied by the appropriate category, match only
  their dedicated profile.

#### Justification

B1 is the simplest conceptual model — one profile, one value set, no
overlap — and it reduces the maintenance surface going forward. It is,
however, a breaking change for any implementer who has already built to
`ADIAutopsyObservation` / `ADIOrganDonationObservation`, and it erases
the profile-level documentation hooks those profiles provide (invariants,
examples, search-parameter guidance).

B2 preserves all three profiles and the current value-set membership
(no VSAC change required) while still eliminating the ambiguity, at the
cost of additional modelling complexity (a new category slice and
updated discriminators) and a requirement that instances now carry a
specific `Observation.category` to be valid. B2 may be attractive if the
workgroup ultimately wants `UponDeathPreferences` to continue to cover
autopsy/organ-donation as a superset concept for systems that do not
distinguish sub-types.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the apparent ambiguity is not a
FHIR validation defect: `Observation.meta.profile` is the normative,
implementer-controlled mechanism for asserting which profile(s) an
instance claims to conform to, and the IG Publisher only validates
against profiles the instance claims (or profiles explicitly supplied to
the validator). If PACIO publishes clear authoring guidance that
`Observation.meta.profile` must be populated with the intended ADI
profile URL, no structural change to profiles or value sets is required.

#### Justification

The ticket frames this as a validator problem, but in FHIR the validator
selects profiles from `meta.profile` (and from command-line/configuration
inputs), not by reverse-lookup from `Observation.code`. Overlap between a
fixed `code` in one profile and a bound value set in another is a
normal, tolerated pattern in FHIR IGs. Declining avoids a VSAC value-set
revision (Option 1), a breaking profile retirement (Option 2/B1), and
the added complexity of a new category slice (B2), in exchange for a
documentation-only clarification.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the minimum-surface change that directly removes the
ambiguity the ticket describes, and it keeps the existing ADI profile
set — which implementers are most likely already building against —
intact. Disposition C's argument that `meta.profile` disambiguates is
technically correct but is fragile in practice: validators run against
IG bundles frequently include every published profile and will flag an
instance as conforming to *both* `ADIUponDeathPreferences` and
`ADIAutopsyObservation` unless `meta.profile` is populated exactly
right, which places the burden on every producer of the resource.
Disposition B1 solves the problem but is unnecessarily destructive for
a technical-correction-class ticket, and B2 introduces modelling
complexity (a required category slice) that the current ADI IG has not
otherwise signalled a need for. Option 1 / Disposition A is the
cleanest fit: it is proportional to a Medium / Technical Correction
issue, owned by a body (the PACIO value-set steward in VSAC) that can
act on it without a normative FHIR-core change, and it leaves existing
profile consumers unaffected. The workgroup should confirm with the
PACIO value-set steward that removing the two LOINC codes from the
`UponDeathPreferences` value set does not break any other profile
binding before proceeding.
