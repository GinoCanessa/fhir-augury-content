## Summary

The ticket asks the Patient Corrections (FHIR) IG to move the canonical
`url` for the `PatientCorrectionBusinessStatus` CodeSystem off of the
`hl7.org/fhir/...` base and onto the HL7 Terminology (THO) base
`terminology.hl7.org/...`, in line with the TSMG Code System Identifier
Task Force guidance (January 2022 TSMG deck) and the R5 Terminologies
section best practice. Because moving the canonical implies the code
system should also be governed by THO/UTG, the reporter notes that a
companion UTG ticket should be filed to register the code system in THO.
The ticket was originally resolved as "Considered for Future Use" in
October 2022 (vote 7-0-0, deferred until further code changes were no
longer expected) and was re-opened by Lynn Laakso on 2025-07-10 for
post-STU1 reconsideration. No related Jira tickets, Zulip threads, or
GitHub items were surfaced through the FHIR Augury cross-reference index;
the only cross-reference is a low-signal mention of the Confluence
attachments URL hosting the TSMG deck.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the `PatientCorrectionBusinessStatus` CodeSystem canonical to
the THO base. Concretely:

1. Change `CodeSystem.url` from
   `http://hl7.org/fhir/uv/patient-corrections/CodeSystem/PatientCorrectionBusinessStatus`
   to
   `http://terminology.hl7.org/CodeSystem/PatientCorrectionBusinessStatus`
   in the Patient Corrections IG source for that CodeSystem (and update
   `CodeSystem.identifier`/OID metadata if present).
2. Update every `ValueSet.compose.include.system` and any
   `Coding.system` literal in the IG (profiles, examples, narrative,
   FSH/JSON sources) that currently references the old `hl7.org/fhir`
   URL to the new THO URL.
3. File the companion UTG ticket against `HL7/UTG` to register the
   code system in `terminology.hl7.org`, following the TSMG Code System
   Identifier Task Force guidance (Jan 2022 deck) and the R5
   Terminologies §system best-practice text.
4. Coordinate publication so the Patient Corrections IG ballot/STU
   update referencing the new canonical lands at-or-after the THO
   release that publishes the new code system, to avoid a window where
   the canonical is unresolvable.

#### Justification

This is exactly the pattern TSMG asks IG authors to follow: code systems
that are not specific to the FHIR core spec belong at
`terminology.hl7.org`, not `hl7.org/fhir`. The Patient Corrections IG's
business-status code system is a domain code system, not a FHIR-spec
artifact, so it falls squarely inside that policy. Doing the move now —
post-STU1 — is also the lowest-pain time: subsequent normative or STU
ballots can carry the corrected canonical forward, and downstream
implementers won't have to migrate twice. The reporter (Joan Harper) and
TSMG guidance both call for this; Lynn's 2025-07-10 reopen explicitly
revisits the previous "defer" vote now that further code-system churn is
expected to be limited.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a phased approach that minimises implementer breakage:

1. In the next Patient Corrections IG publication, add the new THO
   canonical
   (`http://terminology.hl7.org/CodeSystem/PatientCorrectionBusinessStatus`)
   as the *primary* canonical and keep the existing
   `http://hl7.org/fhir/uv/patient-corrections/CodeSystem/PatientCorrectionBusinessStatus`
   URL as a documented alias / `CodeSystem.identifier` so existing
   instances and ValueSets remain resolvable for one publication cycle.
2. Register the code system in THO (UTG ticket) as in Disposition A.
3. Mark the `hl7.org/fhir/uv/...` URL as deprecated in the IG narrative
   with a clear migration note pointing to the THO canonical, and
   schedule its removal for the publication after next.
4. Provide a ConceptMap or release-note table mapping old↔new system
   URIs for tooling that pins to the previous canonical.

#### Justification

Disposition A is correct in direction but a hard cut-over silently
invalidates every `Coding.system` literal in deployed Patient Corrections
data and in any external profile/ValueSet that imports the old URL.
Because the IG just published STU1, there is a non-trivial population of
existing instances and downstream artifacts that pin the old URL.
Carrying the old URL as an alias for one cycle preserves resolvability
while still moving governance to THO and signalling the canonical move
to implementers. This is consistent with how other HL7 IGs (e.g., parts
of US Core) have handled CodeSystem URL re-anchoring.

---

### Disposition C: Decline

#### Proposal

Re-affirm the original 2022-10-17 disposition ("Considered for Future
Use", 7-0-0): leave the CodeSystem canonical at
`http://hl7.org/fhir/uv/patient-corrections/CodeSystem/PatientCorrectionBusinessStatus`
for the current STU cycle, and revisit only when the IG is next opened
for substantive content changes. Persistent identifier policy alone is
not a sufficient reason to break STU1 implementers.

#### Justification

The change is governance/cosmetic rather than behavioural — codes,
display strings, and meaning are unchanged — and the cost falls almost
entirely on implementers who must rewrite `Coding.system` literals and
re-validate instances. The original workgroup vote was unanimous in
deferring this exact change; nothing in the cross-reference data
(no Zulip discussion, no GitHub activity, no linked tickets) shows new
implementer demand or new TSMG enforcement that would reverse that
calculus. Declining preserves STU1 stability and lets the workgroup
batch URL re-anchoring with the next substantive Patient Corrections
release.

---

### Recommendation

**Recommended disposition:** B

The TSMG / R5 best-practice direction is unambiguous — domain code
systems belong at `terminology.hl7.org` and need a UTG/THO registration
— so a flat decline (C) leaves a known governance gap that TSMG will
keep raising every cycle. A hard cut-over (A) is technically the
cleanest end state, but post-STU1 there is a real population of
`Coding.system` literals and downstream ValueSet imports in the wild,
and a single-step rewrite invalidates them silently. Disposition B
captures the correct end state (THO canonical, UTG-registered code
system) while giving implementers a one-cycle deprecation window and an
explicit old-URL alias, which is consistent with how comparable
re-anchoring tickets have been handled in other HL7 IGs. The Patient
Empowerment WG should pair this with the companion UTG ticket the
description already calls for, so the THO registration and the IG
re-anchor land in the same publication wave.
