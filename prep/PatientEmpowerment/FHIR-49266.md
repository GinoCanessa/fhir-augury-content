## Summary

Ricardo Quintano, reviewing the Personal Health Record (FHIR) IG, asks
whether the `.sphr` file format referenced in the IG is *created by* this
specification — i.e., whether the IG itself is the normative source for
`.sphr`, or whether `.sphr` is defined elsewhere and the IG is merely
consuming it. The reporter's request is editorial: if the IG is in fact
introducing `.sphr`, that origin should be stated explicitly; otherwise,
the IG should add a citation/reference to the external source so readers
can find a definition. The ticket is closely paired with
[FHIR-49269](https://jira.hl7.org/browse/FHIR-49269) ("Indicate what is
.sphr format and where to get additional info about it"), which the
ticket's resolver (awatson1978) flagged as a "semi-duplicate" in the
comments. Both tickets converge on the same fix — clarify what `.sphr`
is and where it is defined — and both have already been resolved as
Persuasive with a commitment to add documentation describing the format
(a zip container holding one or more `.phr` FHIR resource files plus
supporting non-FHIR materials, optionally cryptographically signed per
the zip specification).

## Details

**Description:**

> Is the .sphr format created by this IG?
>
> If so, it should be stated, or create a reference to the source of information.

**Comments:**

- **awatson1978 (2025-07-10)** — Posted draft replacement text for the
  IG covering the `.sphr` extension:
  > **2.1.2 File Extensions - .sphr**
  > The `.sphr` file extension is introduced for when there are
  > additional supporting materials, which cannot be fully expressed in
  > FHIR format. The `.sphr` is a zip folder, which contains one or
  > more FHIR resources, as well as additional supporting materials.
  >
  > The `.sphr` file extension is intended to be used for data exports
  > that are saved to a CD, DVD, thumbdrive, etc. Each `.sphr` folder
  > should contain at least one `.phr` file within it, as well as
  > additional supporting materials such as images, documents, and
  > other media.
  >
  > Per the `.zip` specification, the `.sphr` folder may be
  > cryptographically signed.
- **awatson1978 (2025-01-30)** — Notes this is a semi-duplicate of
  [FHIR-49269](https://jira.hl7.org/browse/FHIR-49269); marked
  Persuasive with a commitment to add additional documentation on the
  `.sphr` file format ("It's basically a `.zip` format; but we can do
  better in explaining it.").

**Resolution Description:** Motion to add to Block Vote 1 (Motion:
Hasahi; Second: Sarah Tonn; 3-0-0). Resolved 2025-03-13.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add explicit documentation in the Personal Health Record (FHIR) IG
stating that `.sphr` is **defined by this IG** and add the file-format
section that the resolver already drafted. Concretely, insert the
proposed §2.1.2 "File Extensions — .sphr" section with the wording
already supplied in the ticket comments:

- `.sphr` is introduced by this IG as a portable container.
- It is a `.zip` folder containing one or more FHIR resources plus
  additional supporting materials that cannot be expressed in FHIR.
- Intended for media-bound exports (CD/DVD/thumbdrive); each `.sphr`
  must contain at least one `.phr` file plus supporting media.
- May be cryptographically signed per the `.zip` specification.

At the first occurrence of `.sphr` in the IG narrative (the medication
reconciliation / CCD example called out in FHIR-49269), add a forward
link to §2.1.2. Also add a cross-reference back to `.phr` so readers
know the relationship between the two extensions.

Because FHIR-49269 is the broader formulation of the same request,
resolve both tickets with the same edit and note the disposition on
each.

#### Justification

This matches the resolver's own committed plan ("Persuasive; will add
additional documentation on .sphr file format") and the resolution
already voted in (Block Vote 1, 3-0-0, 2025-03-13). The Zulip history
in the *January 2024 SPHR Track* and *Bring Your Own Health Record*
topics establishes that `.sphr` was developed inside Patient
Empowerment connectathons, so attributing its definition to this IG is
factually correct rather than a forward-looking promise. The ticket is
editorial — there is no behavioural change, no schema change, and no
risk to existing implementations — so accepting as requested is the
lowest-cost path that closes the actual gap the reporter identified.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the documentation update, but instead of embedding the full
`.sphr` definition in the PHR IG narrative, **carve the file-format
specification into a dedicated, short companion page** ("PHR Container
Formats: `.phr` and `.sphr`") and have the IG narrative link to it.
Coordinate with the IPS workgroup (per the *Draft Wapper Specification*
thread) to ensure the wording does not collide with any IPS wrapper /
container that may be standardised later, and add a section explicitly
delineating which parts of the format are owned here vs. inherited
(`.zip`) vs. potentially shared with IPS.

The reporter's literal ask — "state it, or reference the source" — is
satisfied by the dedicated page plus the link from the narrative;
FHIR-49269 is satisfied by the same edit.

#### Justification

The substantive content is identical to Disposition A, but factoring
the format spec into its own page makes it (a) easier to cite from
other IGs that might consume `.sphr` in future, (b) easier to evolve
without churning the main IG narrative, and (c) easier to harmonise
with IPS if the cross-IG discussion in the *Draft Wapper Specification*
thread converges on a shared container. Trade-off vs. A: more editorial
overhead now (a new page, additional cross-links) for less coupling
and clearer ownership later.

---

### Disposition C: Decline

#### Proposal

Decline as already-addressed: the resolver has accepted the request,
voted it through Block Vote 1 with a Persuasive resolution, and posted
the replacement narrative inline in the comment thread. Mark the ticket
**Resolved – Persuasive** (already done) and **close as duplicate of
FHIR-49269** without further IG-level discussion.

#### Justification

This disposition is only defensible as a procedural housekeeping move,
not as a rejection of the request itself. The reporter's concern is
legitimate (the IG references `.sphr` without ever defining it), so
"decline" cannot mean "do nothing"; it can only mean "we have already
agreed to do this under FHIR-49269, so don't double-track the work
here." If the workgroup wants to keep both tickets open as paired audit
trails for the same edit, this disposition should be set aside.

---

### Recommendation

**Recommended disposition:** **A**

The ticket has already been resolved Persuasive with a 3-0-0 block vote
and the resolver has produced ready-to-paste section text in the
comment thread. The Zulip and connectathon history is unambiguous that
`.sphr` originates in this workgroup, so the cleanest, lowest-risk
close is to land the §2.1.2 wording exactly as drafted, fix the
forward-reference at the medication-reconciliation example called out
in FHIR-49269, and resolve both tickets together. Disposition B's
"split into a separate page" is attractive for long-term reuse and IPS
harmonisation, but it expands the scope beyond what the reporter asked
for and beyond what the workgroup has already voted to do; it would be
a sensible follow-up ticket once the basic definition is in the IG.
Disposition C is only a re-statement of the existing resolution and
shouldn't displace the actual edit.
