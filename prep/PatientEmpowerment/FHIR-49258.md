## Summary

The reporter asks how the profiles listed under "Structures: Logical Models"
in the Personal Health Record (PHR) FHIR IG are connected to the rest of
the IG. They observe that many of those logical models appear to derive
from Apple HealthKit, and request that the IG explicitly state the
relationship of those logical models to the IG and provide guidance on
when and how implementers are expected to use them. The ticket is a
documentation-clarity request against the PHR IG rather than a request
to change any conformance artifact. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced by `fhir-augury-cli`, so
the ticket is being reviewed on its own merits.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change already implemented by the assignee (awatson1978) and
recorded in the resolution: add a dedicated `logical-models.html`
narrative page to the Personal Health Record (FHIR) IG that:

1. Lists every `StructureDefinition` of `kind = logical` published by
   the IG (the "Structures: Logical Models" set the reporter referred
   to).
2. For each logical model, states (a) its provenance — explicitly
   calling out those derived from Apple HealthKit — and (b) its
   intended role inside the IG (e.g., source-system data shape,
   bridge to FHIR profiles, conceptual reference only, etc.).
3. Describes the relationship between each logical model and the FHIR
   resource profiles in the IG: which profile(s) (if any) it maps to,
   whether the mapping is normative or informative, and whether
   conforming implementations are expected to consume the logical
   model directly or only the derived profile(s).
4. Provides "when and how to use" guidance: for what use cases an
   implementer should look at the logical model versus the profile,
   and any caveats when the source vocabulary (e.g., HealthKit) does
   not align cleanly with FHIR.
5. Is linked from the IG table of contents and from each individual
   logical-model artifact page so a reader landing on a logical model
   can navigate to the explanation.

#### Justification

This is exactly what the reporter asked for and what the assignee has
already begun delivering (comment of 2025-08-28 announcing the
documentation work, comment of 2026-03-26 confirming the page was
added). The change is documentation-only — no conformance artifact is
modified, so there is no risk of breaking existing implementers. It
directly addresses the IG-publication good practice that any artifact
listed in the IG should have an explicit, documented role; logical
models with unclear provenance and unclear relationship to the
conformance set are a known source of implementer confusion. Marking
the ticket "Persuasive" and adopting the implemented page is
consistent with the Patient Empowerment work group's typical handling
of IG-clarity requests.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent of the request, but instead of (or in addition to) a
single `logical-models.html` overview page, push the explanatory
content **onto each logical model's own artifact page** by populating
the `StructureDefinition.description`, `purpose`, and where applicable
`StructureDefinition.mapping` elements. Specifically:

1. For each HealthKit-derived logical model, set `purpose` to a 2–4
   sentence statement of why this logical model exists in the PHR IG
   and how implementers should treat it.
2. Use `StructureDefinition.mapping` (with a uri identifying Apple
   HealthKit as the source system) to record the provenance, so the
   relationship surfaces in the auto-generated artifact page rather
   than only in a hand-written narrative.
3. Where a logical model corresponds to a FHIR resource profile in the
   IG, add a cross-link in `description` (and ideally a derived
   `targetProfile` reference in mappings) so navigation works in both
   directions.
4. Keep `logical-models.html` as a short index/landing page that links
   into the per-artifact pages, rather than as the primary location of
   the explanatory content.

#### Justification

Putting the explanation on the artifact itself means the guidance
travels with the StructureDefinition wherever it is consumed — IG
build, package install, registry browsing, downstream tooling — not
only via the IG website. It also reduces the risk of the narrative
page drifting out of sync with the artifacts it describes (a common
maintenance issue for hand-curated IG pages). The trade-off vs.
Disposition A is more authoring effort up front and a slight delay
relative to the page that has already been added; this disposition
would naturally be layered on top of Disposition A rather than
replacing it.

---

### Disposition C: Decline

#### Proposal

Decline to make any further documentation change beyond what is
already published, on the grounds that the IG already provides
auto-generated artifact pages for each logical model and that
StructureDefinition metadata is sufficient for implementers to
determine provenance and intended use.

#### Justification

This is the weakest disposition. The reporter's observation that the
relationship is *not* clear from the current IG, together with the
assignee's own acknowledgement that documentation needed to be added,
indicates that the auto-generated artifact pages were not in fact
sufficient. Declining would also be inconsistent with the Persuasive
resolution already recorded on the ticket. Included here only as the
explicit "do nothing further" option.

---

### Recommendation

**Recommended disposition:** A

The work has effectively already been done — the assignee added
`logical-models.html` and the ticket has been resolved Persuasive.
Disposition A formally ratifies that work and pins down the minimum
content the page should cover (provenance, role, profile relationship,
"when/how to use" guidance, navigability) so a reviewer can confirm
the implemented page meets the reporter's actual ask before the ticket
is closed. Disposition B is a sensible follow-up improvement that the
work group may want to schedule as a separate ticket, but it should
not block adoption of the page that already exists. Disposition C is
not defensible given the resolution already on the ticket.
