## Summary

Jessica Bota requests that the Personal Health Record (FHIR) IG provide
proper, human‑readable `Display` values for every concept in the Apple
HealthKit family of CodeSystems published by the Patient Empowerment work
group. She points to the Apple HealthKit Workout Event Type CodeSystem as
a positive example, and calls out the Apple HealthKit Category Type,
Characteristic Type, Correlation Type, and Workout Activity Type
CodeSystems as needing displays added. She also flags that the Apple
HealthKit Quantity Type and Apple HealthKit Workout Activity Type
CodeSystems are misusing `Display` to encode out‑of‑band information
(physical quantity types, deprecation status), and recommends moving
that information into proper concept `property` values so that `Display`
can be reserved for actual user-facing labels.

The Augury cross-reference index returned **no linked Jira tickets, no
linked Zulip threads, no Confluence pages, and no GitHub items** for
`FHIR-49409`, so the request stands on its own. The current resolution
on the ticket is **Persuasive with Modification**; the workgroup
disposition note from awatson1978 records that HL7 leadership voted 5‑0‑0
to remove these CodeSystems from the current PHR publication and to
re‑approach the work with the Terminology Infrastructure (Vocab) work
group's assistance — potentially extracting them into a separate IG
and/or reshaping some of them as `ConceptMap`s.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the in-scope Apple HealthKit CodeSystems published by the PHR IG
so that:

1. Every `concept` in `apple-healthkit-category-type`,
   `apple-healthkit-characteristic-type`,
   `apple-healthkit-correlation-type`, and the display-incomplete
   `apple-healthkit-workout-activity-type` CodeSystems carries a
   human‑readable `display` value, modelled after the
   `apple-healthkit-workout-event-type` CodeSystem already cited as a
   good example by the reporter.
2. In `apple-healthkit-quantity-type` and
   `apple-healthkit-workout-activity-type`, any information currently
   smuggled into `display` to convey physical-quantity type or
   deprecation status is migrated to explicit `concept.property` values
   — minimally a `quantity-type` (Coding/CodeableConcept) property and
   the standard MetadataResource-style `status`/`deprecated` (boolean)
   property, with matching `CodeSystem.property` declarations on the
   resource. `display` is then rewritten to be a clean, human-friendly
   label only.

#### Justification

This is the literal change the reporter asked for and aligns directly
with the FHIR CodeSystem design rules: `Concept.display` is meant to be
a user-facing label, while structured per‑concept facts belong in
`Concept.property` with declared `CodeSystem.property` definitions
(`http://hl7.org/fhir/codesystem-concept-properties` and the standard
`deprecated` property are the canonical mechanism). The work group
already has an internal positive example (Workout Event Type), so the
change is bounded, mechanical, and consistent with existing PHR
authoring patterns. The disposition is also responsive to the comment
from awatson1978 acknowledging that "we'll make sure to update the
Display values".

---

### Disposition B: Alternative Approach (Persuasive with Modification — extract & reshape with Vocab)

#### Proposal

Adopt the modified path already signalled in the workgroup comment and
in the resolution note ("Removed, per HL7 leadership. Will adjust
approach with vocab's assistance. 5‑0‑0"):

1. Remove the Apple HealthKit CodeSystems from the current PHR IG
   publication scope.
2. Stand up a dedicated terminology artifact (either a separate Apple
   HealthKit terminology IG, or a contribution to an existing
   vendor‑terminology container) co-stewarded by Patient Empowerment
   and Terminology Infrastructure (Vocab).
3. In that new home, re-author each CodeSystem with proper
   `Concept.display` values and explicit `CodeSystem.property` slots
   for `quantity-type` and `deprecated`/`status`, exactly as in
   Disposition A.
4. Where Apple HealthKit identifiers are better expressed as a mapping
   from a canonical FHIR/clinical concept (e.g. LOINC, UCUM,
   SNOMED CT) to an Apple HealthKit code, replace those CodeSystems
   with `ConceptMap` resources between the canonical system and the
   Apple HealthKit identifier set, again preserving sensible displays
   on each side.

#### Justification

This matches both the work group's recorded triage decision and the
guidance from HL7 leadership captured in the resolution. It addresses
the underlying defect the reporter raised (broken `display` semantics),
but does so in the right structural location: vendor-specific terminology
should generally not live inside an IG that profiles patient‑facing
PHR resources, and several of these "CodeSystems" are in practice
crosswalks to canonical FHIR terminologies and are better modelled as
`ConceptMap`s. Co-stewardship with Vocab also gives the artifacts
proper terminology governance (HTA review, versioning, deprecation
policy), which is precisely the gap that allowed `display` and
`property` to be conflated in the first place. The trade-off vs.
Disposition A is added scope (a new IG / artifact home and a Vocab
consultation cycle) in exchange for a durable fix.

---

### Disposition C: Decline (close as out-of-scope of base PHR IG; superseded by removal)

#### Proposal

Decline the ticket as a change against the current Personal Health
Record (FHIR) IG on the grounds that HL7 leadership has already
directed that these CodeSystems be **removed** from the IG, so there is
no stable artifact in PHR against which to apply the requested
`display` and `property` changes. Close `FHIR-49409` with a pointer to
the follow-on Vocab-led workstream, and request that any future
work on these CodeSystems be tracked under new tickets filed against
that new home.

#### Justification

The resolution note is unambiguous ("Removed, per HL7 leadership.
Will adjust approach with vocab's assistance. 5‑0‑0"), so editing the
PHR IG to add displays and properties to artifacts that are about to be
removed would be wasted effort and could be misleading to implementers
about the IG's intended terminology surface. Declining here also keeps
the Patient Empowerment ballot record clean: the underlying feedback is
preserved as the rationale for the leadership-directed removal, and the
substantive terminology work is reissued where it belongs.

---

### Recommendation

**Recommended disposition:** B

The reporter is plainly correct on the substance — `Concept.display`
should not be carrying physical-quantity type or deprecation
information, and the Workout Event Type CodeSystem demonstrates that
the work group can produce these artifacts properly. However, simply
applying that fix in place (Disposition A) ignores the work group's
own triage decision and the 5‑0‑0 leadership vote, which together say
the right answer is to move this terminology out of the base PHR IG
and rework it with Vocab. Disposition B captures that direction
explicitly, commits to the structural fix Disposition A would have
made (proper `display` values; quantity-type and deprecation modelled
as `concept.property` with declared `CodeSystem.property`), and
additionally permits the parts of these "CodeSystems" that are really
crosswalks to be re-expressed as `ConceptMap`s — which is the more
honest FHIR modelling. Disposition C preserves the leadership
decision but throws away the reporter's actionable, well‑targeted
feedback, which would be a poor outcome for implementers who do need
displayable Apple HealthKit terminology.
