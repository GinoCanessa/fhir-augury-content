## Summary

The submitter (Marti Velezis) asks that, when an `Observation` is recorded
by a device or app, the PHR / PGHD profiles treat `Observation.device` as a
**Must Support** element with device identifiers supported. As an
alternative, when the recording device is unknown or is not a true medical
device, the patient-contributed observation should explicitly indicate
that it was patient-reported. The intent is to make the provenance of
device- or app-sourced observations unambiguous to downstream consumers.

The cross-reference graph surfaces only one related artifact — the FHIR
element `Observation.device` — confirming this ticket is element-scoped
within the Personal Health Record IG. No related Jira tickets were
returned by the cross-reference query, but ticket comments explicitly
link to **FHIR-49491** ("All the 'Experimental' code systems and value
sets are overly detailed"), which led to splitting the original PGHD
pull request: terminologies were sent to LOINC and the PHR IG retains
the FHIR profiles. That split is the context in which this ticket's
profile-level change (mustSupport on `Observation.device`) is to be
applied.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In every PGHD `Observation` profile in the Personal Health Record IG —
at minimum `pghd-blood-glucose`, `pghd-sleep`, and any sibling
`pghd-*` profiles, plus `pcd-sleep-observation` — change the
`Observation.device` differential entry so that:

- `mustSupport = true`
- The narrative requires the referenced `Device` to support a device
  identifier (i.e., `Device.identifier` slice / mustSupport on the
  identifier system, particularly UDI where applicable).
- Cardinality remains `0..1` (do **not** raise to `1..1`), to match the
  PHR subgroup's 2025-06-26 conclusion.
- Add invariant or guidance text: when `Observation.device` is absent on
  a patient-recorded observation, the Observation must indicate that it
  is patient-reported (e.g., via `Observation.performer` referencing the
  Patient/RelatedPerson, or via an existing PHR provenance pattern).

#### Justification

This is precisely what the reporter asked for and matches the consensus
captured in the PHR subgroup discussion (mustSupport, not 1..1, because
the recording device may be unknown or non-medical). It piggy-backs on
the FHIR-49491 split, where the FHIR profiles were retained in the PHR
IG specifically so profile-level changes of this kind could be applied.
Mandating device-identifier support on the referenced `Device` is
consistent with how device-sourced data is treated in adjacent PCWG /
device guides and is the minimum change needed to make device-vs-patient
provenance unambiguous in PGHD data.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the resolution already noted on the ticket — replace the original
patchwork of changes by introducing the **PGHD Sleep Profile**
(`StructureDefinition-pghd-sleep`) as the canonical exemplar that
includes `Observation.device` — and generalise it:

- Define a single PGHD-wide pattern (e.g., a `pghd-observation-base`
  profile or a re-usable `extension-recordedBy-device` extension) that
  carries `Observation.device` as mustSupport plus a mustSupport
  `Observation.performer` slot for the patient/related-person case.
- Have each concrete PGHD profile (sleep, blood-glucose, etc.) derive
  from / reference that base instead of repeating the mustSupport flag
  and prose in every profile.
- Document a single conformance rule: "exactly one of
  `Observation.device` or a patient-performer must be present" using a
  FHIRPath invariant rather than raising cardinality.

#### Justification

This is the direction the ticket's own resolution narrative is already
pointing at (the PGHD Sleep Profile is cited as the replacement
exemplar). A shared base or extension is cheaper to maintain than
N parallel mustSupport flags across all `pghd-*` profiles, makes the
"device-or-patient" provenance rule machine-checkable, and avoids the
risk of profiles drifting out of sync as the IG grows. It also addresses
awatson1978's 2025-08-21 question about scope (does this apply to
`pcd-sleep-observation` too?) by making the rule explicit in one place
rather than each profile maintainer having to re-derive it. Trade-off
vs. A: more design work up front and a slight learning curve for
implementers, but materially better long-term consistency.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed by the FHIR-49491 resolution**: the PGHD
Sleep Profile (`StructureDefinition-pghd-sleep`) now carries
`Observation.device` and is the IG's canonical example of a
device-sourced PGHD observation. No further profile-wide change is
required; consumers needing device identification on other
patient-recorded observations should profile against `pghd-sleep` or
constrain `Observation.device` themselves.

#### Justification

The ticket's own Resolution Description already says the original change
was "Removed and replaced with the PGHD Sleep Profile, which contains
the `Observation.device` field." If the workgroup considers the
exemplar profile sufficient — and given the IG is still relatively
narrow in scope — adding mustSupport to every PGHD profile may exceed
the scope the workgroup is willing to commit to before the IG matures.
This option is the most defensive and the least likely to cause
downstream churn, but it leaves the original gap (device-vs-patient
provenance is not enforced across PGHD profiles generally) unresolved.

---

### Recommendation

**Recommended disposition:** A

The reporter's request is narrow, the PHR subgroup has already debated
and endorsed it (mustSupport, not `1..1`), and the FHIR-49491 split was
performed precisely so that profile-level changes like this one can land
in the PHR IG. Disposition B is attractive longer-term but is a larger
design lift that risks delaying a change the workgroup has already
agreed to in principle; it can be taken as a follow-on once a second or
third PGHD profile needs the same treatment. Disposition C
under-delivers — the resolution as written only updates a single profile
(`pghd-sleep`), which doesn't satisfy the reporter's broader concern
that *any* device- or app-sourced PGHD observation make its provenance
explicit. Accept as requested across the existing `pghd-*` profiles
(and `pcd-sleep-observation` as awatson1978 flagged), keep cardinality
at `0..1`, and revisit Disposition B if the PGHD profile family grows.
