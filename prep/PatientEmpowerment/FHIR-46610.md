## Summary

The reporter notes that the PACIO Advance Directive Interoperability (ADI)
Implementation Guide's Portable Medical Order (PMO) Observation does not
currently capture the *source form* of the underlying ADI document. In
practice a PMO can be backed by very different forms — for example a
national ePOLST, a state-specific instrument such as Louisiana's LaPOST,
or a state's local use of the National POLST. The request is to add a
representation that distinguishes the form/document type from the
jurisdiction so consumers can tell, for instance, "Louisiana, LaPOST"
apart from "Louisiana, National POLST." The ticket has been resolved
**Persuasive** and the reporter has noted in the comments that a fix
has been pre-applied via [HL7/fhir-pacio-adi PR #138](https://github.com/HL7/fhir-pacio-adi/pull/138);
the disposition still needs to be voted by the work group.

No other Jira tickets cross-reference FHIR-46610, and the
`fhir-augury-cli` cross-reference index returned only the GitHub PR
above. A `search` against Zulip for the ticket key returned zero hits.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the PMO Observation profile in the PACIO ADI IG so that the
*source form* of the underlying ADI document is represented separately
from its jurisdiction. Concretely, add (or bind) two distinct
slots on the PMO Observation:

1. A **form/document-type** element bound to a value set that enumerates
   the recognised ADI form types (e.g., National POLST, ePOLST, LaPOST,
   MOLST, MOST, POST, etc.) — most naturally modelled as an extension on
   the PMO Observation (or on the underlying DocumentReference) carrying
   a `CodeableConcept` from a `pmo-form-type` value set, drawn from a
   new code system maintained by the IG.
2. A **jurisdiction** element bound to an existing jurisdiction value
   set (e.g., `urn:iso:std:iso:3166:-2` for US states) — again
   represented as an extension carrying a `CodeableConcept`, separate
   from the form-type extension so that "Louisiana + National POLST"
   and "Louisiana + LaPOST" are unambiguously distinguishable.

Land these changes in `HL7/fhir-pacio-adi` (path to be confirmed from the
repo briefing once produced) and align them with what is already in
[PR #138](https://github.com/HL7/fhir-pacio-adi/pull/138).

#### Justification

This is exactly what the reporter asked for, and the ticket has already
been resolved **Persuasive** with a pre-applied PR, indicating the work
group has implicitly accepted the change. Splitting form-type and
jurisdiction into two coded slots:

- Lets implementers query / filter by either dimension independently
  (all LaPOSTs in Louisiana vs. all LaPOSTs anywhere they are exchanged,
  or all National POLST instances regardless of state).
- Avoids combinatorial value-set bloat that would otherwise be required
  if jurisdiction and form-type were fused into a single concept.
- Is consistent with the FHIR pattern of preferring orthogonal coded
  axes over composite codes, and with existing US-Realm IGs that bind
  jurisdiction separately from clinical concept.

The pre-applied PR provides a concrete artifact for the work group to
inspect and ratify rather than re-litigate the design from scratch.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding new extensions on the PMO Observation itself, model
the *form*-vs.-*jurisdiction* distinction on the **DocumentReference**
that bears the source form (the ADI document), and have the PMO
Observation continue to reference that DocumentReference. Specifically:

- Profile the source `DocumentReference` to require:
  - `type` bound to the new `pmo-form-type` value set (National POLST,
    ePOLST, LaPOST, MOLST, MOST, POST, …).
  - `context.facilityType` *or* a new `jurisdiction` extension bound to
    a US-state value set, carrying the legal jurisdiction the form was
    executed under.
- Leave the PMO Observation's element shape unchanged; consumers
  needing the form/jurisdiction split follow the reference to the
  governing DocumentReference.

#### Justification

The form type and jurisdiction are properties of the *document* that
backs the PMO, not strictly of the clinical observation derived from it.
Carrying them on `DocumentReference` keeps the PMO Observation focused
on the order itself and avoids duplicating document metadata in two
places (which can drift). It also re-uses existing FHIR document
infrastructure (`DocumentReference.type`, document context) and keeps
PMO Observation backwards compatible for consumers that don't yet need
the form/jurisdiction discrimination. Trade-off vs. Disposition A:
consumers must traverse one extra reference to discover the form type,
and tooling that surfaces PMO data without resolving the underlying
document loses the new information.

---

### Disposition C: Decline

#### Proposal

Decline the change to the PMO Observation profile on the grounds that
form-type and jurisdiction can already be conveyed by existing
mechanisms — specifically the underlying `DocumentReference.type` (for
the form) and the standard US-Realm jurisdiction extensions / the
patient's address (for the state) — and that adding new bespoke slots
on PMO Observation duplicates information that consumers can already
recover from the document graph.

#### Justification

A pure decline avoids growing the PMO Observation profile with
extensions that overlap data already available through the source
document, and it sidesteps the need to ballot and maintain a new
`pmo-form-type` value set inside the PACIO ADI IG. However, this
position is hard to defend given that the ticket has already been
resolved **Persuasive** by the work group and a pre-applied PR exists;
declining now would require reverting that PR and explicitly asserting
that the existing mechanisms are sufficient — which has not been
demonstrated by the implementer community in the ticket.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been adjudicated **Persuasive** by the Patient
Empowerment work group, and a concrete fix is pre-applied in
[HL7/fhir-pacio-adi PR #138](https://github.com/HL7/fhir-pacio-adi/pull/138).
The reporter's use case — distinguishing "Louisiana + LaPOST" from
"Louisiana + National POLST" — is real, jurisdiction-sensitive, and not
cleanly addressable today without the orthogonal split they describe.
Disposition A formalises that split directly on the PMO Observation
where consumers expect to find it, and ratifies the work that has
already been done. Disposition B is a credible alternative if the work
group decides during the vote that the metadata belongs on the
underlying `DocumentReference` instead, and the proposal in A can be
re-targeted to that location without changing the conceptual model. **A
new repo-analysis briefing for `HL7/fhir-pacio-adi` should be produced
before final balloting** so the change recipe and exact file paths can
be confirmed and any IG-specific gotchas surfaced.
