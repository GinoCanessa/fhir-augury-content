## Summary

Craig Newman is asking the Patient Empowerment WG to clarify (or remove) an
implicit assumption on the Personal Health Record (PHR) IG's **Data Model**
page that data held by a PHR is stored in FHIR format. The reporter notes
that this assumption suggests long-term storage *as* FHIR records, whereas
the IG's intended scope is broader — data held by a PHR may extend beyond
current FHIR scope and the IG itself (per the Getting Started page) does not
prescribe how to program software, which would normally include storage.
The same Data Model page also has two smaller editorial issues: Section 4
refers to a "Standard Patient Health Record" while the IG is titled
"Standard Personal Health Record", and the meaning of the "Argonaut R4"
column on that page is unclear and should be clarified or removed.

This ticket is a direct companion to
[FHIR-49070](https://jira.hl7.org/browse/FHIR-49070) (also by Craig Newman,
same WG, same IG, same day), which raises the parallel concern about
Section 2.1's "Systems MUST use FHIR data schemas" language being
inconsistent with the surrounding "recommends" wording and with the
Getting Started page's stance on not prescribing storage. Together the
two tickets ask the WG to draw a clear line between *interoperability
schemas* (where FHIR is normative) and *internal storage* (where FHIR
should not be assumed or required). The lone comment from awatson1978
(Abigail Watson) acknowledges that part of the IG's goal is producing
longitudinal records, which involves normalizing to a standard format,
but agrees the points are well taken and need WG discussion.

## Details

**Description:**

Similar to [FHIR-49070](https://jira.hl7.org/browse/FHIR-49070) ("Clarify
use of FHIR data schemes"), the Data Model page starts with an assumption
that data is contained in FHIR format. This suggests long-term storage as
FHIR records rather than the expectation that data held by the PHR will be
FHIR mapped/compatible. Given that the scope of the PHR may extend beyond
current FHIR scope, this assumption should be clarified or removed. This
is also incompatible with prior statements (Getting Started) that the IG
doesn't cover how to program software.

As well, Section 4 references "Standard Patient Health Record" while the
IG is called the "Standard Personal Health Record".

As well, it's not clear what the Argonaut R4 column is referencing.
Please clarify or remove.

**Comments:**

- **awatson1978 — 2025-01-30:** "Part of the goal of this IG is creating
  longitudinal records, which involves normalizing records to a standard
  format. But your points are well taken. Will need further discussion
  with workgroup."

## Proposed Dispositions

(No GitHub repo briefings apply — the PHR IG's source repository is not
surfaced as a cross-referenced repo for this ticket. Dispositions below
name the conceptual edit targets on the IG's Data Model page; the WG
will translate these to the IG repo's authoring root when staging the
change.)

### Disposition A: Accept As Requested

#### Proposal

Edit the PHR IG's **Data Model** page to:

1. Remove (or rewrite) the opening framing that presumes data held by a
   PHR is stored "in FHIR format." Replace with language that the Data
   Model describes the **logical / interoperability** model, and that
   conformant systems must be able to **emit and consume** data in FHIR
   schemas but are **not required** to use FHIR as their internal
   persistence format. Cross-link to the Getting Started page's
   "doesn't prescribe how to program software" statement.
2. Fix the Section 4 heading/body wording from "Standard Patient Health
   Record" to "Standard Personal Health Record" (and search the page
   for any other Patient/Personal swap).
3. Either define what the "Argonaut R4" column on the Data Model table
   is referencing (presumably a mapping to Argonaut Data Query / US
   Core R4 profiles) or remove the column.

Coordinate this with FHIR-49070 so the Section 2.1 MUST language and
this Data Model framing are reconciled in a single editorial pass.

#### Justification

The reporter's three points are factually correct: (a) the IG elsewhere
declines to prescribe storage, so the Data Model page should not
implicitly require it; (b) the typo is a clear defect; and (c) an
unexplained column header in a normative-looking table is a known
documentation smell. The first commenter has already signalled
agreement in spirit. The change is editorial / clarifying — no
StructureDefinition or capability statement is altered — so the cost
is low and the consistency win across FHIR-49070 + FHIR-49071 is
significant. Per the awatson1978 comment, the longitudinal-record
intent is preserved by saying systems must be able to *exchange* data
in FHIR, which is what Section 2.1 actually needs to mean.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only softening the Data Model page text, restructure the
page to **explicitly distinguish three layers**:

1. **Wire format** (FHIR resources / Bundles / NDJSON) — normative.
2. **`.sphr` package format** — normative for portability.
3. **Internal persistence** — out of scope; explicitly stated as
   implementer choice.

Add a short table at the top of the Data Model page mapping each layer
to its conformance status, then move the existing detailed model under
"Wire format." Apply the same Patient → Personal correction and either
populate the "Argonaut R4" column with a documented mapping table (with
US Core / Argonaut profile URLs) or excise it. Where appropriate,
reference IPS / US Core for prior art on layered conformance language.

#### Justification

Disposition A treats the symptoms; the underlying confusion is that the
IG conflates three different things called "the data" (over-the-wire,
package on disk, persisted store). A reader who only reads the Data
Model page cannot tell which layer a given MUST applies to — that's the
real defect both FHIR-49070 and FHIR-49071 are pointing at. A layered
re-framing fixes both tickets with one structural change and prevents
future tickets of the same shape. The trade-off is more authoring
effort than the minimal edit in A and a slightly larger ballot
footprint, but the IG is still pre-publication / actively iterating, so
this is a good moment to absorb that cost.

---

### Disposition C: Decline

#### Proposal

Decline the request. Retain the current Data Model framing on the basis
that the PHR IG's longitudinal-record goal *requires* implementors to
hold data in FHIR-shaped form (even if not literally as FHIR JSON), so
the assumption is intentional rather than accidental. Limit the
response to fixing the "Patient" / "Personal" typo and adding a
footnote defining the "Argonaut R4" column.

#### Justification

This position is defensible only if the WG concludes that loosening the
Data Model framing would undermine the IG's core value proposition —
that two PHRs can merge longitudinal records without bespoke mapping —
and that the Getting Started "doesn't prescribe software" statement was
itself overstated and should be tightened to "doesn't prescribe UI or
deployment" instead. It is a coherent stance but requires owning the
contradiction the reporter has highlighted; the awatson1978 comment
already concedes the points are "well taken," which weakens this path.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the minimum viable response and would close both
tickets, but FHIR-49070 and FHIR-49071 are two symptoms of the same
underlying defect: the IG uses one word ("data") for three distinct
artifacts (wire, package, store) and then attaches normative MUSTs
without saying which layer they bind. A targeted edit will likely
produce another wave of clarification tickets the next ballot cycle.
Disposition B fixes the framing once, picks up the Patient/Personal
and Argonaut-R4-column corrections along the way, and aligns the IG
with how comparable IGs (US Core, IPS) talk about wire vs. storage
conformance. The IG is still in a phase where structural editorial
work is cheap. Disposition C is not recommended — the lone comment
already acknowledges the reporter's points, and standing pat would
require weakening the Getting Started page in the opposite direction.
