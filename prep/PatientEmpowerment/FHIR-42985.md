## Summary

The PACIO Advance Directive Interoperability (ADI) IG defines a `ContextValue`
extension that attaches a contextual qualifier to a personal advance care plan
preference — e.g., "If I am so sick or seriously injured that I cannot express
my own medical treatment preferences … then these are the things most important
to me: avoiding prolonged dependence on machines, not being a physical burden
to my family, dying at home." As currently defined, both the context and its
value can be any FHIR data type, which David Hill reports is too loosely
constrained to be interoperable and blocks adoption of the construct in US
Core. The ticket asks Patient Empowerment / PACIO to tighten the extension so
it is fit for US Core adoption. The resolution captured on the ticket
(Persuasive) records the agreed direction: rather than continue evolving
`ContextValue`, PACIO ADI will adopt the structure of the existing FHIR core
`ServiceRequest-precondition` extension and republish it under a new name —
`enclosed-precondition` — applying it to both the Observation-based and
Goal-based ADI profiles. The ticket carries the `To_be_applied` label,
indicating the editorial change has been agreed but not yet rolled into the
guide. No cross-referenced Jira, Zulip, or GitHub items were surfaced by the
augury indices for this ticket.

## Details

**Description:**

> The Contextual Value extension ties a context around a personal advance care
> plan preference (e.g., "If I am so sick or seriously injured that I cannot
> express my own medical treatment preferences, and if I am not expected to
> live without additional treatment for my illness, disease, condition or
> injury, then I want my medical care team to know that these are the things
> that are most important to me: Avoiding prolonged dependence on machines,
> Not being a physical burden to my family, Dying at home")
>
> The current definition allows any value type to be specified for the context
> as well as the value. As a result, it is not very interoperable and too
> broadly defined to be adopted in US Core.

**Resolution Description:**

> The US Core team recommended we use the existing concept of a precondition
> extension to capture this information. The existing ServiceRequest-precondition
> extension provides the capability we need, but we want to apply it to both
> Observation- and Goal-based profiles. As a result, we will adopt the
> structure of the ServiceRequest-precondition, but name it as
> "enclosed-precondition" in our guide.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution recorded on the ticket. In the PACIO Advance Directive
Interoperability IG:

1. Define a new extension `enclosed-precondition` whose structural definition
   mirrors the FHIR core `ServiceRequest-precondition` extension (i.e., a
   complex extension carrying a `code` (CodeableConcept) for the precondition
   and a `value[x]` constrained to the small set of types used by core
   `ServiceRequest-precondition`).
2. Apply `enclosed-precondition` to both the Observation-based and
   Goal-based ADI profiles where `ContextValue` is currently used.
3. Mark the existing `ContextValue` extension as deprecated/retired in this
   IG version, with a migration note pointing implementers at
   `enclosed-precondition`.
4. Update narrative, examples, and any IG snapshots/profiles that currently
   reference `ContextValue` so the personal advance-care-plan preference
   examples in the ticket render via the new extension on Observation and
   Goal.

#### Justification

The reporter and the US Core reviewers have already converged on this design,
and the ticket is recorded as Persuasive with the `To_be_applied` label —
i.e., the workgroup has effectively accepted it; what remains is the
editorial change. Reusing the structure of an existing core extension
(`ServiceRequest-precondition`) gives PACIO a vetted shape with a constrained
`value[x]` set, which is precisely what the ticket identifies as the
interoperability gap in `ContextValue`. Naming the PACIO version
`enclosed-precondition` (rather than reusing the core extension verbatim)
acknowledges that PACIO is binding the precondition semantically to a
patient-authored advance-care-plan preference rather than a clinician-ordered
ServiceRequest, and lets PACIO scope the bindings/usage rules independently
without coupling its IG to changes in the core extension.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of minting a new `enclosed-precondition` extension, **directly profile
and reuse the core `ServiceRequest-precondition` extension** on the
Observation- and Goal-based ADI profiles. PACIO would:

1. Reference the canonical core extension URL on both profiles.
2. Add an ADI-specific profile/binding layer that constrains the
   `value[x]` types to the subset that supports advance-care-plan
   preference contexts (e.g., CodeableConcept and Quantity), and binds the
   `code` element to the appropriate ADI value set(s).
3. Update narrative and examples accordingly, retiring `ContextValue` with
   a migration note.

#### Justification

This achieves the same interoperability tightening the ticket requests, but
without introducing a parallel extension whose definition has to be kept in
lockstep with the core one. The principal trade-off vs. Disposition A is
semantic fit: `ServiceRequest-precondition` is named for a different clinical
intent, which can confuse implementers reading an Observation or Goal
instance and may invite incorrect cross-resource reasoning. Disposition B is
preferable if PACIO is willing to accept the naming friction in exchange for
strictly narrower IG surface area and zero extension drift; it is less
attractive if the workgroup believes the patient-preference precondition
warrants its own identifier so it can evolve (e.g., bindings, modifier rules)
independently from the core ServiceRequest one.

---

### Disposition C: Decline

#### Proposal

Leave `ContextValue` as defined and decline to tighten or rename it,
documenting that adoption in US Core is out of scope for the current PACIO
ADI publication cycle.

#### Justification

This is defensible only on the narrow grounds that `ContextValue`'s open
value-type was an intentional design to let advance-care-plan preferences
carry arbitrary contextual qualifiers, and that constraining it now would
break existing PACIO implementers. In practice the reporter, US Core
reviewers, and the assigned PACIO editor have already agreed the current
shape is too loose for interoperable exchange (the ticket is Persuasive with
`To_be_applied`), so declining would override a settled cross-IG
recommendation and leave US Core unable to adopt the construct. There is no
cited implementer base depending on the current open typing, so the
breaking-change argument is weak.

---

### Recommendation

**Recommended disposition:** A

The ticket is already at `Persuasive` / `To_be_applied`, the resolution text
specifies the agreed design (mirror `ServiceRequest-precondition`, publish as
`enclosed-precondition`, apply to Observation- and Goal-based profiles), and
the open work is editorial application rather than fresh debate. Disposition
A simply executes that agreement. Disposition B is a credible alternative on
purity grounds — it avoids a parallel extension definition — but it forces
implementers to consume an extension whose URL and name advertise a
ServiceRequest-centric intent on Observation and Goal instances; the
workgroup explicitly chose to avoid that by minting `enclosed-precondition`.
Disposition C contradicts the recorded consensus and would block US Core
adoption, which is the explicit motivation for the ticket. Workgroup
discussion should focus on confirming the final `value[x]` constraint set
and the `code` binding for `enclosed-precondition`, not on revisiting the
direction.
