## Summary

The Personal Health Record (FHIR) IG includes a "Section 9" page that
presents a view of the history of medical records to the patient/user.
The reporter (Rebecca Baker) asks the Patient Empowerment work group to
augment that view with explicit references — i.e., to surface the
provenance / source(s) of each historical entry shown to the user, so
that the rendered history is traceable back to the underlying records
(and, ultimately, to the FHIR resources or external sources from which
each entry was derived).

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with this ticket in the augury index, so the request
stands on its own and has not been visibly discussed elsewhere.

## Proposed Dispositions

> Note: Because no GitHub cross-references exist, the dispositions below
> are framed against the `Personal Health Record (FHIR)` IG named in the
> ticket metadata. Once a repo briefing for the PHR IG is available, the
> work group should re-anchor the chosen disposition to the specific
> authoring root and page source for Section 9 in that IG.

### Disposition A: Accept As Requested

#### Proposal

In the PHR IG's Section 9 ("history of medical records") rendering,
require each row/entry of the history view to display a reference back
to its source. Concretely:

- For every entry shown in the Section 9 history, add a "Source" /
  "Reference" column (or inline citation) populated from a
  `Provenance` resource that targets the entry's underlying resource(s)
  (e.g., `Provenance.target` → the `Encounter`, `Observation`,
  `Condition`, `MedicationStatement`, etc. that contributed the row).
- Where a `Provenance` is not available, fall back to
  `Resource.meta.source` and/or a linked `DocumentReference` that
  represents the original source document (lab report, discharge
  summary, claim, etc.).
- Update the Section 9 narrative guidance and the page's example/IG
  page source to show this rendering, and add a "shall include source
  reference" expectation to the Section 9 conformance text.

#### Justification

Provenance and `meta.source` are the FHIR-native mechanisms for
"where did this come from" and are already mandated or recommended in
several patient-facing contexts (e.g., US Core Provenance guidance,
International Patient Summary). For a patient-facing history view, the
absence of source attribution is a real usability and trust problem:
patients cannot tell whether a row came from their primary-care record,
a hospital encounter, or a payer claim. Accepting the request as written
aligns the PHR IG with existing FHIR provenance patterns and addresses
the reporter's concern directly.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than requiring an inline source reference on every Section 9
row, add a normative SHOULD-level requirement that each entry be
*traceable* to a source, but let implementations choose how to surface
it. Specifically:

- Section 9 SHALL be backed by resources that either (a) carry a
  `Provenance` whose `target` includes the resource, or (b) populate
  `Resource.meta.source` with a resolvable URI.
- The IG provides a recommended UI pattern (an expandable "Sources"
  disclosure per entry, plus a per-section "Sources used" summary),
  but does not mandate the visual treatment.
- Add a worked example page demonstrating both the data-side
  conformance (Provenance + meta.source) and one rendering option, so
  implementers have a concrete template.

#### Justification

A literal "every row shows a citation" mandate (Disposition A) can be
visually noisy in a consumer-facing app and may push implementers to
fabricate weak `Provenance` entries just to satisfy the conformance
text. Disposition B keeps the underlying data-quality requirement —
which is what the ticket actually needs — but lets PHR vendors render
it in a way that fits their UX. It also avoids over-specifying the
patient-facing UI in a FHIR IG, which historically the work group has
been cautious about.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that source attribution for a
patient-facing history view is already addressable with existing FHIR
mechanisms (`Provenance`, `Resource.meta.source`,
`DocumentReference.context`, `AuditEvent`) and does not require
PHR-IG-specific text. Close as "Not Persuasive — Existing Mechanism"
and, if helpful, add a one-line cross-reference from Section 9 to the
core Provenance guidance.

#### Justification

The FHIR core already provides robust ways to express where a
resource came from, and IGs that need patient-facing provenance
(IPS, US Core, International Patient Access) layer their requirements
on top of those primitives rather than re-defining them. Adding
PHR-specific conformance text risks divergence from those existing
patterns. If the PHR IG already references the core provenance
machinery, the request can be addressed editorially without a
substantive change.

---

### Recommendation

**Recommended disposition:** B

The ticket's underlying need — that Section 9's history view be
traceable to its sources — is real and worth addressing; pure
decline (C) under-serves the reporter and a patient-facing IG that
*shows* a history without showing where it came from is a credible
trust problem. At the same time, mandating a specific inline rendering
(A) over-specifies UI in a FHIR IG and is likely to draw pushback from
implementers. Disposition B threads the needle: it makes
Provenance/`meta.source` a first-class, conformance-level expectation
for everything Section 9 displays, gives implementers a recommended
rendering pattern and a worked example, and stays consistent with how
the broader FHIR ecosystem (US Core Provenance, IPS) handles the same
problem. The work group should confirm B with the reporter and then
scope the actual edits against the PHR IG's Section 9 page source once
that repo's briefing is available.
