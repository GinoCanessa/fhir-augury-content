## Summary

Eric Haas requests that the PACIO Advance Directive Interoperability (ADI)
Implementation Guide clarify its Signature narrative. If the IG chooses not
to follow the FHIR R6 proposal to move to a Provenance-based signature
model, then the existing Signature section should at least be updated to
link readers to authoritative, standard definitions of the terms
"detached", "enveloped", and "embedded" so that readers do not have to
infer the meanings from context. The ticket further asks the IG to
distinguish between the XML-based uses of these terms (as defined in
XML-DSig) and the JWS-based uses (RFC 7515), because the JWS notion of
"detached content" (RFC 7515 Appendix F) is not the same as an
XML-DSig "detached signature", and "enveloped" appears to be an
XML-only concept with no common JWS analogue. No related Jira tickets,
Zulip threads, or GitHub items were surfaced by cross-reference or
search against the FHIR Augury data sources.

## Details

**Description:**

> If the Guide does not update its Signature section to adopt the FHIR R6
> Proposal (Provenance-based or other decisions), then update the
> discussion to provide links to standard definitions of "detached",
> "enveloped", "embedded" to avoid confusion.
>
> Also, distinguish between these XML-based concepts and JWS-based
> concepts.
>
> The JWS specification defines detached content:
> [RFC 7515 Appendix F](https://datatracker.ietf.org/doc/html/rfc7515#appendix-F),
> which should not be conflated with detached signature.
>
> I am not aware of an example of a JWS-based enveloped signature and
> think it is only an XML concept.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the PACIO ADI IG's Signature narrative page so that the terms
"detached", "enveloped", and "embedded" are each linked to an
authoritative external definition, and so that the XML-DSig and JWS
contexts are explicitly separated. Concretely:

- In the Signature section of the ADI IG, add a short table or
  definition list with one row per term:
  - **Detached (XML-DSig):** link to
    [W3C XML-DSig §2 Signature over Detached Data](https://www.w3.org/TR/xmldsig-core/).
  - **Enveloped (XML-DSig):** link to the XML-DSig "Enveloped Signature"
    definition / transform in the same spec.
  - **Embedded / Enveloping (XML-DSig):** link to the XML-DSig
    "Enveloping Signature" definition (clarifying that "embedded" as
    used in FHIR's `Signature` datatype corresponds to XML-DSig's
    "enveloping" form, or name the difference).
  - **Detached (JWS):** link to
    [RFC 7515 Appendix F — Detached Content](https://datatracker.ietf.org/doc/html/rfc7515#appendix-F),
    with a one-sentence note that JWS "detached content" is *not* the
    same concept as an XML-DSig "detached signature".
- Add a paragraph explicitly stating which of these forms the ADI IG
  expects implementers to produce/consume, and which forms are
  out-of-scope (notably: whether JWS-based enveloped signatures are
  considered supported in the IG at all — per the reporter, there is
  no established JWS equivalent).
- Leave `Signature.type`, `Signature.sigFormat`, and the underlying
  base-FHIR signature model unchanged; this is a narrative/editorial
  fix only.

#### Justification

This is exactly what the ticket asks for, and it is a low-risk,
narrative-only change: no profile, extension, ValueSet, or structure
definition is touched, and no conformance behavior changes. It
directly addresses the reporter's concern that readers currently have
to infer what "detached", "enveloped", and "embedded" mean in context.
Pointing to W3C XML-DSig and RFC 7515 is consistent with FHIR's general
editorial practice of deferring to external standards bodies for
well-defined terms rather than re-stating them. It also preserves the
option to adopt the FHIR R6 Provenance-based signature direction
later, because the clarification is scoped to the *existing* Signature
narrative rather than pre-committing to a model.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding inline links in the PACIO ADI IG, refactor the
Signature discussion to align with (and cross-link) the base FHIR R6
signature guidance, and push the glossary work upstream. Specifically:

- Reduce the PACIO ADI IG's own Signature narrative to a short,
  IG-specific section covering **only** what ADI constrains (e.g.,
  which signature formats a conformant ADI Bundle must carry, who
  must sign, and when Provenance is required).
- For the definitional material on "detached / enveloped / embedded"
  and the XML-DSig vs. JWS distinction, link out to a single shared
  narrative — preferably:
  - the base FHIR `Signature` datatype page (in
    `HL7/fhir`), and/or
  - the FHIR R6 Provenance-based signature proposal page once it
    lands.
- If that base-spec material does not yet contain the XML-DSig vs. JWS
  clarification, open a companion ticket against FHIR Core (Work Group:
  FHIR-I or Security) so the glossary is fixed in one place rather
  than duplicated in each IG.
- In the ADI IG, add a one-line note that the IG intentionally *does
  not* adopt the R6 Provenance-based signature proposal (if that is
  the current decision), so readers are not confused about why the
  narrative differs from R6.

#### Justification

The underlying issue — that FHIR readers conflate XML-DSig
"detached/enveloped/embedded" with JWS "detached content" — is not
PACIO-specific. Every IG that discusses the `Signature` datatype is
exposed to the same confusion. Fixing it only in the ADI IG means the
next IG rediscovers the same problem and re-writes the same glossary,
possibly inconsistently. Centralizing the glossary in base FHIR (or
the R6 Provenance signature proposal page) and having IGs link to it
is consistent with FHIR's layered authoring model, where IGs
constrain rather than re-define base concepts. The trade-off vs.
Disposition A is that B is higher effort and depends on coordination
with another work group; if that coordination is not achievable in
the current PACIO ballot window, A remains a safe local fallback.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — already adequately addressed**:
the PACIO ADI IG's Signature section already relies on the base FHIR
`Signature` datatype, which in turn references XML-DSig and JWS. No
IG-level change is required; readers needing formal definitions of
"detached / enveloped / embedded" can follow the existing links from
the base `Signature` page. The request is essentially an editorial
preference and does not identify an incorrect statement in the IG.

#### Justification

A decline would be defensible only if the base FHIR `Signature` page
already contains the explicit XML-DSig vs. JWS disambiguation the
reporter is asking for. In practice that distinction is subtle and
not all readers follow out-links; the reporter is a very experienced
FHIR implementer (Eric Haas) and still flagged it as confusing, which
is evidence that "just follow the base spec" is not sufficient in
this IG's audience. Decline therefore only makes sense if the work
group has an explicit policy of not carrying editorial glossaries in
IGs, or if effort is genuinely unavailable in this ballot cycle. The
justification is weak on substance; it is primarily a
scope/effort-management rationale.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly resolves the reporter's concern with minimal
risk: it is a narrative edit in the PACIO ADI IG's existing Signature
section, it does not change any `StructureDefinition`, profile, or
conformance requirement, and it preserves the option for the IG to
later adopt the FHIR R6 Provenance-based signature direction. The
change can be made in time for the current ballot without coordinating
across work groups. Disposition B describes the *right long-term
shape* of the fix (centralize the glossary in base FHIR), and the
work group should seriously consider filing that follow-up ticket
against FHIR Core — but B should not block this ticket, because the
PACIO reader is confused *today*. Disposition C is not recommended: a
senior implementer explicitly finding the narrative confusing is
prima facie evidence that the current text is not sufficient, and a
decline would leave that confusion in the published IG.
