## Summary

Lisa R. Nelson (Patient Empowerment) is asking the PACIO Advance
Directive Interoperability (ADI) IG team to add a task that establishes
and requests the IHE/HL7 `formatCode` value(s) needed for documents that
conform to this IG. CDA and FHIR document exchange (notably XDS / XDS-on-FHIR
and similar document-sharing infrastructure) rely on `formatCode` slot values
to let consumers know which document profile they are receiving; without a
registered `formatCode` for ADI-conformant documents, downstream document
registries and consumers cannot reliably advertise or filter for these
documents.

The triaged resolution is **Persuasive with Modification**: the Work Group
agreed with the reporter and committed to clarifying Section 8.8.2 of the
ADI formal specification (Advance Directive CDA Document Structure
Requirements) by adding `formatCode` clarification language pointing at the
build of `HL7/fhir-pacio-adi`. No related Jira tickets, Zulip threads, or
GitHub items were surfaced by the cross-reference index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Open and track a dedicated narrative-only change against the ADI IG
(`HL7/fhir-pacio-adi`) Section 8.8.2 ("Advance Directive CDA Document
Structure Requirements") that:

1. Adds an explicit "formatCode" subsection enumerating, for each ADI
   CDA document type defined by the IG (e.g., Personal Advance Care Plan
   Document, Healthcare Agent Authorization Document, Patient-Provider
   Conversation Document), the LOINC `formatCode` the IG expects
   implementers to advertise in document-sharing metadata (XDS
   `formatCode` slot, `DocumentReference.content.format`,
   `DocumentManifest`/Bundle equivalents).
2. For any ADI document type that does **not** yet have a published
   `formatCode`, records that gap and commits the WG to file the
   `formatCode` request with the LOINC Document Ontology / IHE process
   on behalf of the IG, with the registered code(s) to be backfilled
   into the IG before publication.
3. Adds conformance language clarifying that producers SHALL populate
   the appropriate `formatCode` and consumers SHOULD use it for
   document-type discrimination.

This matches the triaged Resolution Description verbatim and turns the
"task" the reporter requested into a concrete, trackable narrative
change plus an external `formatCode` registration action item.

#### Justification

The reporter's request is operationally necessary: `formatCode` is the
canonical mechanism by which document-sharing infrastructure
distinguishes one CDA document profile from another, and IGs that
constrain CDA document structure are expected to either reuse an
existing `formatCode` or sponsor a new one. The ticket has already been
triaged "Persuasive with Modification" with the WG agreeing in
principle, so Disposition A simply executes that decision. Doing this in
Section 8.8.2 keeps the change scoped to narrative, which is
low-risk for the rest of the IG (no profile, value set, or example
churn) and aligns with the `Narrative` label on the ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same narrative clarification in Section 8.8.2, **but** rather
than enumerating concrete `formatCode` values inline, add a short
normative paragraph that:

1. States that ADI-conformant CDA documents SHALL carry the IHE/LOINC
   `formatCode` registered for the corresponding document type,
2. Points to a single living artifact (e.g., a dedicated "Document
   Identification & formatCodes" page in the IG, or a small
   FHIR `ConceptMap`/`CodeSystem` artifact in the IG) that lists the
   per-document `formatCode` values, and
3. Maintains that artifact (rather than Section 8.8.2 prose) as the
   source of truth that gets updated as new `formatCode` values are
   registered.

Pair this with the same external action item (file the missing
`formatCode` request(s) with LOINC/IHE).

#### Justification

`formatCode` registrations are out-of-band and slow — they happen on the
LOINC Document Ontology / IHE cadence, not the IG publication cadence.
Embedding the full table inline in Section 8.8.2 risks the IG narrative
going stale the moment a new code is granted or an existing one is
deprecated. Centralizing the values in a dedicated page or a tiny FHIR
terminology artifact keeps Section 8.8.2 stable, gives implementers a
single dereferenceable URL to bookmark, and lets the IG be
re-published with only that artifact changing when codes change.
Trade-off vs. Disposition A: slightly more authoring work up front and
one extra page/artifact to maintain, in exchange for a much smaller
footprint for future updates.

---

### Disposition C: Decline

#### Proposal

Mark the request as Not Persuasive on the grounds that `formatCode` is
infrastructure metadata governed by the document-sharing profile (IHE
XDS / MHD) and the LOINC Document Ontology rather than by the ADI IG
itself, and that ADI implementers can discover the appropriate code(s)
through the existing LOINC/IHE registries without IG-level narrative.

#### Justification

This rationale is technically defensible — `formatCode` is indeed
maintained externally — but it's weak in this case. The IG already
constrains CDA document structure (Section 8.8.2 exists precisely to
describe those constraints), so it is the natural and expected place to
tell implementers which `formatCode` to advertise for those constrained
documents. Declining would leave a real, reporter-identified
interoperability gap unaddressed and contradicts the WG's already-
recorded agreement to clarify the section. Included for completeness
only.

---

### Recommendation

**Recommended disposition:** A

The Work Group has already triaged this ticket as Persuasive with
Modification and committed, in the resolution text, to exactly the
Section 8.8.2 clarification described in Disposition A. The change is
narrative-only (consistent with the `Narrative` label and the
`Block-Vote-5` block-vote grouping), scoped to a single section of the
ADI IG, and resolves a concrete document-sharing interoperability gap
that the reporter — a long-standing PACIO/ADI implementer — has
identified. Disposition B is a reasonable refinement worth raising in
the discussion if the WG anticipates frequent `formatCode` churn, but it
expands the scope beyond what was triaged and shouldn't block adopting
A. Disposition C should be declined: the IG is the right place to tell
implementers which externally-registered `formatCode` to use for
documents the IG itself defines.
