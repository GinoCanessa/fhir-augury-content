## Summary

The PACIO Advance Directive Interoperability (ADI) IG includes two
Composition profiles — PACPComposition (Patient Advance Care Plan) and
PMOComposition (Personal Medical Orders) — that allow attesters to sign
the document, but they do not specify which `Composition.attester.mode`
(`personal`, `professional`, `legal`, `official`) should be used for
each kind of attester role: Witness, Notary, Signing Provider, and
Signing Patient. Implementers therefore have no consistent way to
populate or interpret `attester.mode`, which undermines the document
authenticity guarantees the IG is meant to provide. The ticket asks the
Composition profiles to describe the appropriate Attester Mode for each
of those roles. A secondary concern raised in comments is preserving
authenticity through electronic (non-digital) signatures and treating a
document as "complete" only when all expected attester signatures are
present.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the FHIR Augury indexes for this ticket. A broader Zulip
search surfaced only an unrelated 2022 PACIO ADI block-vote topic.

The Patient Empowerment WG has already taken a `Persuasive with
Modification` block vote (Block Vote 3, 8-0-0) on the ticket, with a
proposed disposition that:

1. Adds explicit references to attester party of `ADI Witness`, `ADI
   Notary`, and (PMO Composition only) `US Core PractitionerRole`.
2. Adds invariants enforcing the party-to-mode pairings:
   - `Patient` → `#personal`
   - `RelatedPerson` → `#personal`
   - `ADI Witness` → `#official`
   - `ADI Notary` → `#professional`
   - `US Core PractitionerRole` → `#legal`
3. Updates the PACP and PMO diagrams to include the attester in the
   Header.
4. Adds narrative guidance in the PACPComposition and PMOComposition
   descriptions (or a dedicated narrative section) describing the
   attester / mode associations.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the workgroup's already-voted `Persuasive with Modification`
language verbatim in the US PACIO ADI IG (likely repo
`HL7/fhir-us-pacio-adi`; authoring root not verified because no
briefing is cached):

1. In the `PACPComposition` and `PMOComposition` StructureDefinitions,
   slice/constrain `Composition.attester.party` to allow references to
   `Patient`, `RelatedPerson`, `ADI Witness`, `ADI Notary`, and (PMO
   only) `US Core PractitionerRole`.
2. Add FHIRPath invariants on `Composition.attester` enforcing:
   - `party.resolve() is Patient implies mode = 'personal'`
   - `party.resolve() is RelatedPerson implies mode = 'personal'`
   - `party.resolve() conformsTo('<ADI Witness canonical>') implies mode = 'official'`
   - `party.resolve() conformsTo('<ADI Notary canonical>') implies mode = 'professional'`
   - `party.resolve() conformsTo('<US Core PractitionerRole canonical>') implies mode = 'legal'`
   (PMO Composition includes the PractitionerRole rule; PACP omits it.)
3. Update the PACP and PMO Composition diagrams in the IG narrative to
   show `attester` in the Composition Header.
4. Add a dedicated "Attester / Mode associations" narrative section to
   each Composition profile page (or expand the profile description),
   listing the role → mode mapping and the rationale.

#### Justification

This is the disposition the workgroup has already block-voted 8-0-0 and
labeled `Compatible, substantive` / `Correction`, so it is the lowest-
risk path through ballot reconciliation. It directly resolves the
ambiguity reported by Maria D. Moen by giving each attester role exactly
one valid `attester.mode`, and it does so using mechanisms the
Composition resource already supports (`attester.party` references plus
`attester.mode` value-set membership), so no extensions or core-spec
changes are required. The invariants make the party→mode pairing
machine-checkable, which is what implementers actually need to
populate signed documents consistently.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substance of the resolution, but instead of expressing the
party→mode constraints as profile invariants, express them as a
**purpose-built `attester` slicing** on each Composition profile, with
one named slice per role (`witnessAttester`, `notaryAttester`,
`patientAttester`, `relatedPersonAttester`, and PMO-only
`providerAttester`). Each slice fixes `attester.mode` to its allowed
value, constrains `attester.party` to the matching profile reference,
and uses cardinality (e.g., `0..1` or `0..*`) appropriate to the role.
Replace the broad invariants with per-slice constraints, and keep the
narrative + diagram updates from the original proposal. Optionally,
publish a small CapabilityStatement-style narrative table on each
Composition profile page documenting the slices.

#### Justification

Slicing-based authoring is more discoverable to implementers reading the
profile in IG Publisher output than a list of FHIRPath invariants, and
profile validators surface slice violations with clearer error messages
than generic invariant failures. It also makes the role→mode coupling
visible in the differential view rather than buried in an invariant
expression, reducing the chance that downstream profile authors silently
break the pairing. Trade-off vs. Disposition A: slicing is slightly
more verbose in the StructureDefinition and changes the published
profile shape more than invariants would, but it does not change wire
format and is fully backward-compatible. It also gives a natural place
to add the "Signing Provider" role (PMO-only) without overloading the
`PractitionerRole` reference in PACP.

---

### Disposition C: Decline

#### Proposal

Decline the requested change to the Composition profiles and instead
add only a non-normative narrative paragraph to the PACPComposition and
PMOComposition profile descriptions noting that attester mode SHOULD be
chosen by the implementer based on local jurisdictional practice, with
no formal binding between attester role and `Composition.attester.mode`.

#### Justification

A "decline" is defensible only on the grounds that
`Composition.attester.mode` is already defined by the FHIR core spec
with a required value set, and that legal interpretation of "official"
vs. "legal" vs. "professional" varies by jurisdiction (advance
directives are heavily state/locale-dependent in the US). Hard-coding
the mappings could conflict with local statute and over-constrain
international or future regional reuse of these profiles. This option
is included for completeness only; the ticket's reporter and the
workgroup vote both indicate that the ambiguity is causing real
implementation problems, so declining without addressing the underlying
need is unlikely to satisfy the requester.

---

### Recommendation

**Recommended disposition:** A

The Patient Empowerment WG has already block-voted 8-0-0 in favor of
the modification described in Disposition A, and the change is
labeled `Compatible, substantive` / `Correction`. Honoring the existing
vote keeps reconciliation simple and lets the IG ship with the
clarification implementers asked for. Disposition B is a reasonable
authoring refinement and the WG may want to consider it during
implementation (slicing vs. invariants is largely a style choice with
the same conformance effect), but it should not block adoption of the
voted disposition. Disposition C does not address the reported
ambiguity and contradicts the WG's own vote, so it is not recommended.
