## Summary

The reporter (Virginia Lorenzi) is reviewing the Personal Health Record
(PHR) FHIR Implementation Guide and flags that one of its tables — which
catalogs implementation guides for specific data types — currently
includes a column devoted to "Argonaut R4". She raises three concerns:
(1) "Argonaut R4" is not defined in-line or linked, so non-US readers
won't know what it refers to; (2) the column is sparsely populated, so
it carries little informational weight; and (3) because the PHR IG is
intended to be an international specification, leaning on a
US-specific program creates an unbalanced US slant. She asks the work
group to either define / link Argonaut, balance the table by referencing
guides from other jurisdictions (for example IPS and IPA), or otherwise
rework the section so it does not bias the IG toward the US ecosystem.
No related Jira tickets were surfaced via cross-references.

## Details

**Description:**

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rework the affected table in the PHR (FHIR) IG narrative so that:

1. The "Argonaut R4" column header is expanded to a self-describing
   label (for example "Argonaut R4 (US — HL7 Argonaut Project)") and
   linked to the Argonaut Project landing page so non-US readers can
   follow up.
2. New columns or rows are added for international IGs already
   recognized by HL7 — at minimum **IPS** (International Patient
   Summary, `hl7.fhir.uv.ips`) and **IPA** (International Patient
   Access, `hl7.fhir.uv.ipa`) — populated for the same data-type rows.
3. An introductory paragraph is added above the table explaining its
   intent (mapping PHR-relevant data types to existing IGs that profile
   them) and explicitly stating that the listed IGs are illustrative,
   not exhaustive, and that other jurisdictional IGs may apply.

The change is purely narrative / IG-page content; no resource, profile,
or value-set artifacts are added or modified.

#### Justification

The reporter's three concerns (undefined acronym, sparse column,
US-only framing) are factual and verifiable from the page itself, and
the PHR IG is positioned by Patient Empowerment as a generally
applicable specification rather than a US-realm IG. Adding IPS / IPA
columns aligns the table with HL7's flagship international content and
follows the FHIR-wide convention of cross-linking related IGs from
narrative pages. Defining or linking "Argonaut R4" is a low-cost
editorial fix that improves accessibility for international readers
without removing useful US context.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of expanding the table in place, restructure the section as
follows:

1. Remove the per-IG matrix columns (including Argonaut R4) and replace
   the table with a single "Implementation guide(s) commonly used for
   this data type" column whose cells contain a short bulleted list of
   IG references (with package id and link). Each cell carries whatever
   IGs are genuinely relevant for that data type — Argonaut, US Core,
   IPS, IPA, AU Core, NL Core, etc. — without privileging one
   jurisdiction in the column structure.
2. Move the IG list to a small data file (e.g., a CSV / JSON included
   by the page template) so contributors from other realms can add
   their IGs via PR without touching narrative HTML.
3. Add a short "Jurisdictional scope" note clarifying that the PHR IG
   itself is jurisdiction-agnostic and that listed IGs are pointers to
   profiles a deploying realm may choose to conform to.

#### Justification

A wide matrix with one column per IG scales poorly: every additional
realm IG either gets its own sparse column or is silently excluded,
which is exactly the failure mode the reporter is flagging with the
Argonaut R4 column today. Collapsing the IG dimension into a list per
data-type row keeps the useful information (which IGs profile which
PHR-relevant data types) while removing the implicit ranking of
jurisdictions. Externalizing the data also lowers the maintenance bar
for international contributors, which is consistent with the IG's
international positioning and reduces future "US slant" tickets of the
same shape.

---

### Disposition C: Decline

#### Proposal

Decline the change request on the grounds that the PHR (FHIR) IG was
authored against the HL7 IG matrix as it existed at publication time,
that the Argonaut R4 column is intentionally retained for traceability
to the existing US implementer base, and that internationalization of
content-type mappings is out of scope for the current PHR IG ballot
cycle. Direct the reporter to raise jurisdiction-specific mappings in
the corresponding realm IGs (IPS / IPA / national cores) rather than
in PHR.

#### Justification

The PHR (FHIR) IG today does not define new profiles for the data types
in question; the table is a navigational aid pointing at existing
profiling work. Maintaining a column-per-IG matrix indefinitely is a
governance burden, and accepting the request without a maintenance plan
risks setting an expectation that PHR will track every realm IG for
every data type. If the broader concern is the IG's international
posture, that is better addressed as a single editorial pass on the
IG's scope statement than as table reshaping.

---

### Recommendation

**Recommended disposition:** A

The reporter's concerns are concrete, low-cost to address, and entirely
within the editorial scope of the PHR IG narrative. Disposition A
delivers the requested fix (define / link Argonaut, add IPS / IPA,
soften the US framing) without restructuring the page or committing the
work group to ongoing matrix maintenance, which makes it the lowest-risk
path to closing the ticket. Disposition B is the right longer-term
direction but is a larger editorial refactor and should be raised as a
separate ticket if Patient Empowerment wants to pursue it; doing both at
once risks scope creep on a "Highest" priority triage item. Disposition
C is hard to defend given that the PHR IG is explicitly positioned as
internationally applicable and the missing definition / link for
"Argonaut R4" is a documentation defect on its own merits.
