## Summary

The ticket asks the Patient Empowerment work group to re-anchor the
`PatientCorrectionTaskTypesVS` ValueSet defining canonical URL under
`terminology.hl7.org` (THO) rather than `hl7.org/fhir/uv/patient-corrections`,
and to file a corresponding UTG ticket so the ValueSet is registered in
THO. The Terminology Authority (TSMG) policy referenced here is that only
ValueSets used by **required** bindings on a `code` data type may remain
anchored under `hl7.org/fhir`; everything else (including the Patient
Corrections task-type ValueSet at issue, which is not bound to a `code`)
must live in THO. The ticket originally received a "Considered for Future
Use" disposition (7-0-0 vote, 2022-10-17) on the basis that further code
changes were still expected, but Lynn Laakso reverted that resolution and
re-opened it on 2025-07-10 "post STU1 publication", so the work group now
needs to dispose of it for the next publication cycle. No explicit Jira
links or cross-references to other artefacts were found, but Augury's
keyword-similarity index shows this ticket is one of an entire series of
TSMG-driven "Defining URL should be anchored in THO" change requests
(FHIR-37137, FHIR-37141, FHIR-37142, FHIR-37143, FHIR-37152, FHIR-37156,
FHIR-37162, FHIR-37163, FHIR-37164, FHIR-37193, …) raised against
multiple Patient Corrections ValueSets, and the workgroup will likely
want to dispose of them consistently.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the `PatientCorrectionTaskTypesVS` ValueSet's canonical URL
from `http://hl7.org/fhir/uv/patient-corrections/ValueSet/PatientCorrectionTaskTypesVS`
to a THO-hosted canonical, e.g.
`http://terminology.hl7.org/ValueSet/patient-correction-task-types`
(exact slug TBD with TSMG/UTG). Concretely:

1. File a UTG ticket against `Unified Terminology Governance` requesting
   creation of the ValueSet (and, if not already present, the underlying
   CodeSystem of task-type codes) in `terminology.hl7.org`, citing this
   FHIR ticket and the TSMG anchoring policy.
2. Once the THO artefacts exist, update the Patient Corrections IG
   sources so the ValueSet definition is removed from the IG and any
   binding that references it now points at the THO canonical URL.
3. Add a publication-time redirect / errata note from the legacy
   `hl7.org/fhir/uv/patient-corrections/ValueSet/...` URL to the new
   THO URL so existing implementer references do not silently break.
4. Confirm with TSMG that the binding strength does not qualify the
   ValueSet for an `hl7.org/fhir` exception (the policy reserves
   `hl7.org/fhir` for ValueSets bound `required` to a `code` data type;
   `PatientCorrectionTaskTypesVS` is bound to a `CodeableConcept`-typed
   element, so it does not qualify).

#### Justification

This is exactly the disposition TSMG asked for and exactly the
disposition the Vocabulary work group has been applying to the rest of
this ticket series. The ValueSet is not bound to a `code` data type, so
it falls squarely into the THO bucket under current policy. Doing this
post-STU1 (which is why Lynn re-opened the ticket) is the right time:
the canonical change is breaking, and bundling it with the next
substantive publication minimises the number of times implementers have
to re-pin URLs. Filing the UTG ticket is non-optional — without it, THO
will not host the ValueSet and the IG cannot legally reference it.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the ValueSet's defining URL inside the Patient Corrections IG
namespace (no THO move), but tighten the ValueSet's metadata to match
TSMG's intent:

- Mark the ValueSet `experimental = false`, set `jurisdiction` and
  `copyright`, and add an explicit `useContext` so that consumers can
  see the ValueSet is intended for cross-IG reuse even though it
  physically lives under `hl7.org/fhir/uv/patient-corrections/`.
- Document, in the IG's vocabulary section, that the URL will move to
  THO in a future version, and provide a published mapping table so
  early adopters can prepare.
- Defer the actual canonical change (and the UTG ticket) to the next
  major version of the IG (i.e., not the next STU update).

#### Justification

The literal request is a breaking canonical-URL change, and breaking
URL changes against an STU1-published IG carry real implementer cost —
every server that has bound to the existing canonical URL has to be
updated, which is non-trivial in the Patient Corrections deployer
community (small but production). Disposition B preserves the existing
URL for one more publication cycle while still acknowledging TSMG's
direction, and gives downstream implementers an explicit warning window.
The trade-off vs A is that the IG is technically out of compliance with
the THO-anchoring policy for one more cycle, and the work group will
have to revisit the same ticket again later.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the basis that `PatientCorrectionTaskTypesVS` is
intentionally part of the Patient Corrections IG's domain model and is
not, in practice, reused outside the IG. Close as
**Persuasive with Mod / Not Persuasive — IG-local ValueSet**, with a
note back to TSMG that the IG considers this artefact IG-scoped and
therefore appropriately anchored under
`hl7.org/fhir/uv/patient-corrections/`.

#### Justification

Defensible only if the work group is prepared to argue with TSMG that
the policy ("ValueSets must be anchored in THO unless bound `required`
to a `code`") is over-broad and should not apply to ValueSets that are
purely IG-local. The Vocabulary label, the `Highest` priority, the
TSMG-authored description, and the existing 7-0-0 community direction
on sibling tickets all argue against this; a Decline here would also
make the IG inconsistent with the rest of the THO-anchoring series.
Recommend only as a "do nothing" placeholder if the work group decides
the cost of the move outweighs compliance.

---

### Recommendation

**Recommended disposition:** A

The ticket was filed by TSMG, sits at `Highest` priority, was explicitly
re-opened "post STU1 publication" by Lynn Laakso so it could be acted
on in the next cycle, and is one of an entire numbered series of
identical change requests against multiple Patient Corrections
ValueSets — the workgroup gets the most leverage by disposing of all of
them the same way and the same way TSMG has asked. Disposition B trades
short-term implementer convenience for longer-term policy debt and
guarantees the workgroup will see the ticket again; Disposition C
relitigates a settled TSMG policy and would put Patient Corrections out
of step with the rest of the FHIR ecosystem on terminology hosting.
Accept As Requested, file the UTG ticket immediately so THO work can
start in parallel with IG-source updates, and bundle the breaking
canonical-URL change with the next substantive Patient Corrections
publication along with a published URL-mapping note for implementers.
