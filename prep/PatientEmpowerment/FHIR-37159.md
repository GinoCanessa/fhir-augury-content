## Summary

The ticket asserts that the canonical defining URL for the
`PatientCorrectionCommunicationTypes` CodeSystem in the HL7 Patient
Corrections IG is anchored under `hl7.org/fhir/uv/patient-corrections/...`
when, per TSMG best practice, HL7-published CodeSystems should be anchored
under `terminology.hl7.org` (THO). The reporter requests that the existing
URL `http://hl7.org/fhir/uv/patient-corrections/CodeSystem/PatientCorrectionCommunicationTypes`
be changed so the CodeSystem is defined in THO, with a corresponding UTG
ticket opened to register the CodeSystem in the THO repository. The
reporter cites the January 2022 TSMG CodeSystem Identifier Task Force
deck and the R5 `terminologies.html#system` guidance as the authoritative
best-practice references. The ticket was originally dispositioned in
October 2022 as "Considered for Future Use" (vote 7-0-0, Vogt/Richesson),
but was re-opened by lynn_laakso on 2025-07-10 "post STU1 publication"
so the work group can revisit the change now that STU1 is shipped and
further code changes are no longer being deferred. There are no
cross-referenced Jira tickets, Zulip threads, or GitHub items in the
Augury index — the only outbound reference is to the supporting
Confluence attachment listed above.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the `PatientCorrectionCommunicationTypes` CodeSystem out of the
Patient Corrections IG and into THO, then update the IG to reference
the THO-hosted CodeSystem. Concretely:

1. **In `HL7/UTG`:** open the companion UTG ticket called for in the
   description and add a new CodeSystem resource with
   `CodeSystem.url = http://terminology.hl7.org/CodeSystem/PatientCorrectionCommunicationTypes`,
   carrying the existing concept set, definitions, and properties from
   the IG copy. Set appropriate
   `caseSensitive`, `content`, `experimental`, `status`, and
   `jurisdiction` values per THO authoring conventions. Seed
   `identifier` and `meta.versionAlgorithm` per current UTG patterns.
2. **In `HL7/fhir-patient-corrections`:** delete the
   `CodeSystem-PatientCorrectionCommunicationTypes.json` (and any
   sibling `.fsh`) that defines the CodeSystem in the IG, and replace
   in-IG references with the new THO canonical
   `http://terminology.hl7.org/CodeSystem/PatientCorrectionCommunicationTypes`.
   Update any ValueSet that composes from the old canonical, including
   `ValueSet.compose.include[].system`, and update narrative pages
   that cite the URL.
3. Add a `ConceptMap` / errata entry (or at minimum release-note text)
   describing the URL change so existing implementations can migrate.
4. Coordinate publication so the THO release containing the new
   CodeSystem is available before (or no later than) the next IG
   publication that points at it.

#### Justification

This is exactly the policy TSMG codified in the January 2022 CodeSystem
Identifier Task Force guidance and that the R5 `terminologies.html#system`
section enshrines as "best practice" for HL7-defined code systems. The
ticket was only previously deferred because the IG was still actively
churning code; the 2025-07-10 re-open note explicitly states STU1 has
shipped and further code changes are no longer being deferred, which
removes the original objection. Anchoring HL7-authored CodeSystems in
THO also gives the artifact a stable, IG-version-independent identity
that downstream terminology servers and other IGs can reuse without
having to mirror IG-local URLs. The ticket's `Highest` priority and the
`Vocabulary` label are consistent with treating this as a vocabulary
hygiene fix rather than a behavioural change.

---

### Disposition B: Alternative Approach

#### Proposal

Re-host the CodeSystem in THO with the new canonical
`http://terminology.hl7.org/CodeSystem/PatientCorrectionCommunicationTypes`
**and** keep the existing IG-local canonical
`http://hl7.org/fhir/uv/patient-corrections/CodeSystem/PatientCorrectionCommunicationTypes`
alive as a deprecated alias for at least one publication cycle. Mechanically:

- Publish the THO CodeSystem as in Disposition A.
- In the IG, retain the old CodeSystem resource but mark it
  `status = retired`, set `CodeSystem.experimental` / a
  deprecation extension, and add a narrative + `CodeSystem.identifier`
  pointing at the THO canonical so tooling can resolve forward.
- Add a `ConceptMap` mapping every code 1:1 from the old system URL to
  the THO system URL, and document the migration in the IG release
  notes and a "Vocabulary Migration" page.
- Schedule removal of the deprecated IG-local CodeSystem for the
  publication after next.

#### Justification

This achieves the same end state TSMG/R5 ask for, but recognises that
STU1 has already been published with the IG-local URL. Real
implementers in the wild may have persisted instances or profile
assertions against the old canonical; flipping the URL hard in a single
release breaks those silently. A deprecation window plus a
machine-readable `ConceptMap` lets terminology servers and validators
resolve both URLs during the migration period, which is consistent with
how HL7 has handled comparable URL relocations elsewhere (e.g., several
v3-era code systems moved to THO with overlap periods). The trade-off
versus Disposition A is more authoring work and a slightly noisier IG,
in exchange for safer downstream migration.

---

### Disposition C: Decline

#### Proposal

Re-resolve the ticket as **Persuasive with Modification — no URL
change**, on the grounds that the CodeSystem is narrow in scope, only
used inside the Patient Corrections IG, and changing a published
canonical URL post-STU1 imposes migration cost on implementers that
outweighs the conformance benefit. Instead, document in the IG that the
CodeSystem is IG-local, add an explicit note in the CodeSystem narrative
that future publications may relocate it to THO, and close this ticket.

#### Justification

The TSMG/R5 guidance is a *should*, not a normative *shall*, and the
work group has already invested a published STU1 cycle in the existing
URL. The CodeSystem is small and IG-scoped, so the cost of forcing every
downstream consumer to re-bind is non-trivial relative to the benefit.
The original 2022 disposition (Considered for Future Use, 7-0-0) was
defensible on these grounds; absent evidence of concrete reuse demand
from outside this IG, declining the relocation remains defensible. This
disposition is the weakest of the three, however, because the ticket
was explicitly re-opened by the work group after STU1, signalling that
the prior deferral is no longer the position the WG wants to hold.

---

### Recommendation

**Recommended disposition:** B

Disposition B best matches the signal in the ticket history. The
`Vocabulary`-labeled, `Highest`-priority ticket was re-opened by the
work group specifically because STU1 has shipped and the prior
"defer until code changes settle" rationale no longer applies, so
Disposition C is hard to defend — the WG has already moved off that
position. Disposition A is the cleanest answer in principle and is
exactly what the TSMG / R5 best-practice references demand, but it
ignores the fact that STU1 has been published under the IG-local
canonical. Disposition B delivers the same policy outcome (the
authoritative CodeSystem moves to THO under the
`terminology.hl7.org/CodeSystem/...` canonical, and a UTG ticket is
opened to register it) while protecting any STU1 implementer that has
already bound to the old URL through an explicit deprecation window
and a `ConceptMap`. The extra authoring cost is modest relative to the
implementer-protection win, and the migration story (deprecate, alias,
remove next cycle) gives the work group a defensible answer if asked
"why did you change a published canonical URL?" at the next ballot. If
the WG concludes there are zero known external consumers of the
STU1-published URL, falling back to Disposition A is reasonable.
