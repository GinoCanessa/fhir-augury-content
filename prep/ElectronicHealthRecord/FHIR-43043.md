## Summary

John Moehrke (HL7 Security WG co-chair, owner of Provenance/AuditEvent) is
questioning whether the EHRS-RLE (Records Lifecycle Events) IG correctly
classifies the **Archive** activity as a **Create** event. His concern arises
from in-flight FHIR Core work on a `$purge` operation (tracked in
[FHIR-16198](https://jira.hl7.org/browse/FHIR-16198), "Add delete/purge REST
operation", FHIR-I, *Persuasive with Modification / Applied*), which led to a
Zulip discussion in `implementers > Purging a Version of a Resource`.

The EHRS-RLE definition appears to assume that *Purge implies Archive* — that
backup is created at purge time, justifying a Create lifecycle event. Moehrke
counters that backups are usually maintained continuously and independently of
purge, and that at purge time the trimmed resources are simply removed against
a pre-existing backup. He therefore asks the EHR WG either to (a) record Purge
through some lifecycle activity other than Archive, or (b) introduce a
distinct **Purge** lifecycle activity in EHRS-RLE if Purge and Archive are not
in fact the same event. The Zulip thread also surfaces related but distinct
concepts (redaction, tombstones, retention policy) that EHR WG should be
mindful not to conflate with Archive or Purge.

## Details

**Description:**

> There is discussion about a need in FHIR for a $purge operation. This would
> trim out old data that are not needed online anymore. It would seem this is
> consistent with the Archive activity in the EHRS-RLE.
>
> The EHRS-RLE defines "Archive" as a Create event.  I guess presuming that
> if you Purge you do have a backup (archive).
>
> Yet, it is not always at Purge time that the backup is created. The backup
> is continuously updated. At Purge time, the trimmed resources are just
> removed, having presumed that the backup is available.
>
> Is this the right alignment between Purge and Archive? If so, then I think
> we need a different way to record the Purge activity.
>
> If not, then should EHRS-RLE have a Purge activity?

## Related Zulip Discussions

### [implementers > Purging a Version of a Resource](https://chat.fhir.org/#narrow/stream/implementers/topic/Purging%20a%20Version%20of%20a%20Resource)

The thread (39 messages, Oct–Nov 2023) was started by Sonat Suer (Blaze) asking
how to implement `DELETE /:type/:id/_history/:vid` from FHIR-16198, and
specifically how to handle the resulting *gap* in the version history (their
use case: redacting accidentally-pushed sensitive data).

Key positions:

- **Gino Canessa (FHIR-I):** Delete-history-version will leave a gap;
  traceability (Provenance / Audit) is an implementation detail that the spec
  intentionally does not mandate. FHIR Core should remain devoid of policy.
- **John Moehrke (Security/EHR-RLE):** Purge is *similar but distinct from*
  Delete because it is policy-driven (e.g., "remove patient records 3+ years
  after death", "drop audit older than 1 year"). It deserves its own
  Provenance/AuditEvent profile and an exemplar pair when `$purge` lands. He
  explicitly notes EHRS-RLE currently treats Archive as a Create event and
  questions that ("I don't think this IG is correct on this topic… or my
  presumption that Purge is an EHRS-RLE Archive activity; which would mean
  that Purge is not Archive."). EHRS already separates Delete from Archive.
- **Brian Postlethwaite:** Notes the audit record for a purge could exceed
  what was purged, especially when purge is driven by storage cost.
- **Alexander Kiel & Alexander Zautke (Blaze):** Concerned with database
  consistency; `$purge` is the first FHIR operation that *changes* history
  rather than appending. Suggest a trial-run parameter on `$purge`.
- **Grahame Grieve:** `$purge` is by definition wiping; whether it is
  appropriate is a policy decision — implementations that can't accept that
  semantics should not support it. Provided historical context on paper
  retention (10–75+ years).
- **Kinson & Elliot Silver (Imaging):** Imaging retention is jurisdiction-
  and pathology-specific; in practice retention policies are rarely fully
  implemented because litigation cost dominates storage cost. Proposes
  **tombstones** as a way to keep purge as an append-only operation while
  signalling the version is gone — bridging the FHIR-I "gap" concern and the
  EHRS lifecycle classification question.
- **John Moehrke (closing positions):** Redaction is *not* purging; the three
  use cases (storage trimming, sensitive-data scrubbing, archival migration)
  are distinct and should not all be lumped under one EHRS lifecycle
  activity.

Consensus: (1) FHIR Core `$purge` will not mandate Provenance/Audit but
exemplars are wanted; (2) Purge, Delete, Archive, and Redaction are
**different activities** that should be kept distinct in EHRS-RLE; (3) work on
the IG profile for the purge AuditEvent/Provenance should proceed in parallel
with FHIR-I.

Participants: Sonat Suer, Gino Canessa, John Moehrke, Brian Postlethwaite,
Alexander Kiel, Alexander Zautke, Grahame Grieve, Kinson, Elliot Silver.

## Proposed Dispositions

### Disposition A: Accept As Requested — Add a distinct Purge lifecycle activity to EHRS-RLE

#### Proposal

Update the EHRS-RLE (FHIR) IG so that **Purge** is modelled as a first-class
lifecycle activity, separate from **Archive** and from **Delete**:

1. Add a new EHRS-RLE activity code (e.g., `purge`) with definition:
   *"Removal of resources from the live record store under a stated retention
   or compliance policy, where any backup/archive is presumed to exist
   independently of this event."*
2. Re-classify **Archive** so it no longer carries the implication that it is
   triggered by Purge. Archive remains a Create-style lifecycle event tied to
   producing an archival copy, but is decoupled from removal.
3. Bind the new Purge activity to a Provenance/AuditEvent exemplar pair (in
   the IG, not normative): a Provenance whose `.activity` = `purge` and
   `.policy` = the retention rule, and an AuditEvent whose
   `action`/`subtype` records the bulk removal. Coordinate exemplars with
   FHIR-16198's `$purge` operation so the IG ships with at least one example
   per `$purge` invocation pattern (per-Patient compartment, per-Group,
   per-version `DELETE _history/:vid`).
4. Cross-reference FHIR-16198 from the IG narrative so implementers see the
   linkage between the REST operation and the lifecycle event.

#### Justification

This matches what FHIR-43043 literally asks ("then should EHRS-RLE have a
Purge activity?") and is consistent with the Zulip consensus that Purge,
Delete, and Archive are *similar but materially different* activities driven
by different triggers (policy vs. user action vs. archival copy creation).
EHRS-RLE already distinguishes Delete from Archive; adding Purge completes
the set rather than overloading Archive. Aligning IG examples with the
FHIR-I `$purge` operation gives implementers (e.g., Blaze) a clear pattern
without forcing FHIR Core to take on policy text — addressing Gino Canessa's
concern that core stay policy-free while letting the IG carry the policy
guidance Moehrke wants.

---

### Disposition B: Alternative Approach — Re-scope Archive and add IG guidance, without a new activity code

#### Proposal

Rather than adding a new lifecycle activity, **clarify the existing Archive
activity definition** in EHRS-RLE and add narrative guidance for `$purge`:

1. Reword Archive in the IG so it explicitly describes the act of producing
   an archival copy (a Create), and explicitly states that it is *not* the
   same as removing data from live storage.
2. State in narrative that Purge (as defined by FHIR-16198) is a composite
   operation that may, depending on system policy, be recorded as
   **Delete** lifecycle events for the affected resources, optionally
   accompanied by an Archive lifecycle event when (and only when) an archival
   copy is created at purge time.
3. Provide an IG-level Provenance/AuditEvent profile or example for the
   composite case but reuse existing lifecycle activity codes.

#### Justification

This avoids growing the EHRS-RLE activity vocabulary, which carries downstream
ballot and binding cost (vocabulary changes ripple into the EHRS-FM Functional
Model and any CDS or conformance tooling that consumes it). It preserves
backward compatibility for existing EHRS-FM implementers and still resolves
the originally-raised issue (Archive should not silently mean "purge
happened"). The trade-off vs. Disposition A is that policy-driven bulk
removal is recorded as a fan-out of Delete events plus optional Archive
events, which some reviewers in the Zulip thread (Moehrke, Kinson) would
argue obscures the *policy* nature of the activity that distinguishes Purge
from a user-initiated Delete.

---

### Disposition C: Decline — Defer until FHIR-I `$purge` operation is finalised

#### Proposal

Mark FHIR-43043 as *Considered — No action at this time*, with a note that
the EHR WG will revisit once FHIR-I's `$purge` work (FHIR-16198) is closed
for R6 and the operation's normative definition (parameters, async behaviour,
audit expectations) is stable. Track FHIR-43043 against the R6 EHRS-RLE
publication cycle.

#### Justification

FHIR-16198 was last updated 2023-12-18 and is `Persuasive with Modification`
but still labelled `AwaitingMerge` / `ReOpen`. Defining EHRS-RLE lifecycle
semantics for Purge before the operation itself is normatively defined risks
having to revise the IG twice — once on speculation, once after FHIR-I
finalises the operation. Declining (with deferral) avoids that churn. The
weakness is that EHRS-RLE then publishes one more revision with the
ambiguous Archive-as-Create wording that triggered this ticket, leaving
implementers without IG-level guidance on how to record purge today.

---

### Recommendation

**Recommended disposition:** A.

The substantive content of the Zulip thread, the differing policy character
of Purge vs. Delete, the existence of the EHRS-RLE separation between Delete
and Archive (cited by Moehrke himself), and the practical need expressed by
the Blaze implementers all point to Purge being a *distinct* lifecycle
activity rather than a re-spelling of Archive. Disposition A resolves
FHIR-43043's literal question ("should EHRS-RLE have a Purge activity?")
while keeping the policy detail in the IG layer, which is exactly where
FHIR-I has insisted it should live. Disposition B is the lower-cost
alternative if EHR WG concludes the vocabulary cost is unacceptable, and
Disposition C remains available as a fallback if R6 timing forces deferral —
but neither addresses the concern that EHRS-RLE is currently *wrong* about
Archive, which is the actionable defect Moehrke reported.
