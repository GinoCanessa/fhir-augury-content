## Summary

FHIR-54580 is a Jan-2026 ballot comment against the **AI Transparency on
FHIR** IG (Electronic Health Record WG) reporting that an example resource
in the IG illustrates a `Provenance` that records AI involvement but does
**not** carry the corresponding AI security tag on `Resource.meta.security`.
The reporter (IgnacioJauregui) frames this as a defect — from a consumer's
perspective, AI-generated data should be discoverable via the tag without
having to walk every Provenance — and argues that since the IG's stated
goal is "simple, clear, and unambiguous" identification of AI-influenced
data, examples should consistently demonstrate the tag.

The comment thread reveals that the tag-vs-Provenance question is in
fact the underlying design point, not just an example error. John Moehrke
explains the IG's current position that **either** the tag, **or**
Provenance, **or both** are acceptable, so an untagged example is a
deliberate illustration. Workgroup discussion (Sam, Brian) recorded in
later comments shows live disagreement: Sam would tolerate Provenance
without `meta.security` but strongly recommend the tag; Brian holds that
the tag must always be present on AI-influenced data. Moehrke's last
comment notes consensus is still needed on whether the tag is mandatory
or whether the two methods remain independent. The ticket therefore has
two coupled dispositions: (a) a narrative/example clarification, and
(b) an underlying normative decision in `general_guidance.md`/the
Provenance profile about whether the tag is REQUIRED whenever Provenance
records AI authorship.

There are no other Jira tickets cross-referenced to FHIR-54580. The only
external mention is the Jan-ballot triage table on Zulip, which buckets
this issue under "Profile & Model Structure".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add the AI security tag to the offending example in
`input/fsh/EX_Provenance.fsh` (or `EX_Tagging.fsh`, depending on which
example the reporter encountered) so that the targeted resource carries
`meta.security = http://terminology.hl7.org/CodeSystem/v3-ObservationValue#AIAST`
(or `#DICTAST` as appropriate). Concretely: extend the relevant
`Instance:` with a `* meta.security = …` line and regenerate via SUSHI.
At the same time, harden the normative wording in
`input/pagecontent/general_guidance.md` so that any FHIR resource whose
`Provenance` records AI authorship is **REQUIRED** to carry the matching
AI security tag, and add an `obeys`-style invariant to
`input/fsh/SD_Provenance.fsh` that requires `Provenance.target.resolve().
meta.security` to include the AI tag whenever the Provenance asserts AI
agency. This eliminates the consumer-side ambiguity the reporter raises
and matches Brian's stated position in the 2026-02-16 comment.

#### Justification

The reporter's argument — that consumers cannot reliably identify
AI-generated data without dereferencing every Provenance — is a real
implementability problem and is consistent with how `meta.security`
labels are intended to work in FHIR (a queryable, machine-actionable
hint surfaced on the data itself). Sam's position ("strongly recommend")
and Brian's ("must exist on data that is influenced by AI") together
indicate that the WG is leaning toward the tag being normative. Making
the tag REQUIRED whenever AI-Provenance exists also keeps the IG's
stated primary goal — simple, clear, unambiguous identification of
AI-generated data — first-class. The change is scoped to one IG repo
(`HL7/aitransparency-ig`), uses already-existing terminology
(`AIAST`/`DICTAST` from `v3-ObservationValue`), and does not require
any change in HL7/fhir or HL7/UTG.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the IG's current "tag, Provenance, or both" two-method model
unchanged, but **(1)** annotate the existing example so its omission of
the tag is clearly intentional, and **(2)** add a sibling example that
shows the tagged form so the reader sees both shapes side by side.
Specifically:

- In `input/fsh/EX_Provenance.fsh` (or `EX_Tagging.fsh`), add a `*
  ^description = …` (or a leading FSH `// …` comment that survives into
  narrative) on the existing instance explaining that this example
  deliberately demonstrates the Provenance-only form.
- Add a new `Instance:` in the same FSH file showing the same scenario
  with `meta.security` populated, and reference both from the Tagging /
  AI-Observability sections of `input/pagecontent/general_guidance.md`.
- Tighten the narrative in `general_guidance.md` to **strongly
  RECOMMEND** the tag (SHOULD) whenever Provenance records AI agency,
  without making it a SHALL, and call out the consumer-side cost the
  reporter raises so implementers see it.

This matches Moehrke's published position (the IG already permits
either) and Sam's "willing to allow Provenance without `meta.security`,
but would like to strongly recommend".

#### Justification

This addresses the reporter's underlying concern (consumer
discoverability) without taking the breaking-change step of making the
tag REQUIRED. It preserves backward compatibility for systems that, per
Moehrke's comment, do not implement `meta.security` at all, while still
nudging future implementations toward the tagged form via a SHOULD.
Adding the paired example is low-risk (purely additive), keeps the IG
honest about both shapes, and makes the existing example
self-explanatory rather than confusing — directly resolving the
reporter's concrete observation. Trade-off vs. Disposition A: it leaves
the `Provenance`-only loophole open, so a strict consumer still cannot
rely on `meta.security` alone to find all AI-generated data; that cost
is acknowledged in the narrative rather than removed.

---

### Disposition C: Decline

#### Proposal

Decline the change. Treat FHIR-54580 as **Not Persuasive**: keep the
example as-is, do not change the IG's two-method model, and respond
that the example was deliberately chosen to show the
Provenance-only form because that form is explicitly permitted by the
IG (per Moehrke's 2026-01-18 comment).

#### Justification

The IG already documents that either the tag, Provenance, or both are
acceptable, so an untagged example is illustrative, not defective. The
tag-vs-Provenance question is itself an open normative discussion (see
the Provenance & Tagging bucket on the Jan-ballot triage Zulip thread)
and arguably should be resolved by the dedicated tickets in that bucket
rather than by an example-level change request. Declining keeps the
two-method model consistent with the IG's `1.0.0-ballot` design and
avoids a breaking change for implementers that have already aligned to
the Provenance-only form.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the smallest change that actually answers the
reporter's complaint while respecting the live disagreement in the
comment thread. The reporter is correct that an example showing
Provenance without the tag is genuinely confusing absent a label, and
Moehrke's own 2026-02-16 comment ("Example should explain that it is an
example of data that has Provenance without the tag") concedes that
point. At the same time, Moehrke's earlier comment about systems that
do not implement `meta.security` is a legitimate constraint and makes
Disposition A's SHALL premature — that normative decision should be
made by the WG once consensus is reached (per Moehrke's "need consensus
on if the tag is always needed" comment) and is better tracked in the
Provenance & Tagging cluster of tickets than via this example-level
issue. Disposition C ignores the reporter's observation that the
example, as currently rendered, is misleading even if the IG text
permits it. B threads the needle: it fixes the example so it is no
longer ambiguous, adds a tagged counterpart so both shapes are visible,
and strengthens the narrative to SHOULD without committing the WG to a
SHALL it has not yet ratified.
