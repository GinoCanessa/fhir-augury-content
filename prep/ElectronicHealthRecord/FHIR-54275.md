## Summary

FHIR-54275 is a January 2026 ballot comment against the **AI Transparency
(FHIR) IG**, raised by Mitra Rocca against Section 2.1, "Observability
Factors for iterative IG development." The reporter finds the table in
that section confusing: it contains four columns, but the values that
would logically belong to those columns have instead been split across
four rows, and there are no row/column headings explaining what kind of
information each axis represents. The requested change is to either add
row and column titles that clarify the semantics of the table, or
otherwise restructure the presentation so the relationships between the
four pieces of information are self-evident. The ticket has been
resolved **Persuasive with Modification** with a change applied via
[HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22),
which removed the table entirely and replaced it with a list of
observability factors. No other Jira tickets are cross-referenced; the
ticket does appear alongside many sibling January-ballot items in the
Zulip triage summary "IG - Jan ballot" on the *AI Transparency on FHIR*
stream, where it was grouped under "Documentation & Clarity."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Keep the four-column table in Section 2.1 but add explicit column
headings that name each of the four kinds of information being captured
(one heading per column) and, if the table is split into rows by factor,
add a leftmost column (or row header) that names each factor. In other
words, preserve the tabular structure but make the row/column semantics
self-describing, exactly as the reporter asks. Changes are confined to
the Section 2.1 narrative page of `HL7/aitransparency-ig`.

#### Justification

This is the narrowest, most literal read of the ballot comment: the
reporter explicitly offered "if some titles for the rows and columns
could be added to indicate the kinds of information for the table, that
would clarify the ideas better" as an acceptable fix. Adding headings
preserves the original authors' intent to present factors as a
structured comparison rather than a flat list, and it gives
implementers a table they can scan by column. In HL7 IG narrative
authoring this is a low-risk edit: a single narrative page, no profile
or terminology changes, no cross-repo impact.

---

### Disposition B: Alternative Approach — Replace the table with a list (this is what was actually done)

#### Proposal

Drop the Section 2.1 table entirely and replace it with a bulleted (or
numbered) list of observability factors for iterative IG development,
with each factor named and described inline. This is the approach taken
in [HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22),
which is already merged and forms the basis of the "Persuasive with
Modification" resolution. The disposition is therefore: **accept the
comment as Persuasive with Modification and adopt PR #22 as the
response**, with the modification being "restructure as a list of
factors rather than headings-augmented table."

#### Justification

The underlying complaint is that the table's structure obscures rather
than clarifies the relationships between its cells. When cell content
is itself narrative (a factor and its description), a list often
communicates the same information more directly and survives small
screens and accessibility tooling better than a 4×N table without
headers. This alternative addresses the reporter's core need
("clarify the ideas better") without committing to the specific
tabular-headings fix the reporter suggested. Because PR #22 is already
merged, this disposition is also the lowest-cost path: the change has
been authored, reviewed, and verified to render in the IG build.

---

### Disposition C: Decline

#### Proposal

Decline the ballot comment on the basis that Section 2.1 is
introductory/educational prose whose current layout conveys the four
observability factors adequately to the IG's intended audience, and
that further structural rework would add editorial churn without
changing normative content.

#### Justification

A decline would be defensible only if the work-group believed the
table as-written was already clear, or if they had evidence that
readers do not struggle with it. Given the comment was filed by a
named reporter, was carried into the "Documentation & Clarity" bucket
in the Zulip triage roll-up, and the work-group chose to resolve it
Persuasive with Modification by actually merging a fix, the
community signal clearly does not support decline. Recording it here
for completeness only.

### Recommendation

**Recommended disposition:** B

Disposition B matches the resolution already applied on the ticket
(Persuasive with Modification, resolved via merged PR
[HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22),

fallback if the work-group later concludes that the tabular

argued for preserving the table. Disposition C is not supported by the
