## Summary

The ticket flags a semantic mismatch in the **AI Transparency on FHIR** IG:
the `meta.security` slice `AItags` is bound with binding strength **`required`**
to a ValueSet whose human-readable name is **"Recommended provenance codes"**.
The word "Recommended" in the ValueSet name implies guidance-only usage, which
contradicts the conformance obligation a `required` binding actually imposes
(implementers MUST use a code from the ValueSet). The reporter offers two
remediations: (1) weaken the binding strength to **`extensible`** so the
naming and the conformance contract align, or (2) **rename the ValueSet** to
strip "Recommended" from its title while keeping the `required` binding.

No other Jira tickets directly cross-reference FHIR-54141. Its only inbound
reference is a roll-up post in the AI Transparency on FHIR Zulip stream that
groups it with nine other "Provenance & Tagging" issues being triaged for
the January ballot (e.g., FHIR-54707, FHIR-54281, FHIR-54280, FHIR-54279,
FHIR-54255, FHIR-54245, FHIR-54244, FHIR-54242, FHIR-53973). Those siblings
suggest the slice and its binding are an active area of ballot churn, so the
disposition chosen here may be coordinated with the broader provenance-tagging
clean-up rather than handled in isolation.

## Proposed Dispositions

### Disposition A: Accept As Requested — Option 1 (weaken to `extensible`)

#### Proposal

In the AI Transparency on FHIR IG, change the binding strength on the
`meta.security` slice `AItags` from **`required`** to **`extensible`**, leaving
the bound ValueSet (currently titled "Recommended provenance codes")
unchanged. Update any downstream conformance text that asserts a closed code
list for the slice. Add a brief note in the slice description stating that
codes from the ValueSet are the recommended set but that locally-defined
codes MAY be used when no listed code applies (the standard `extensible`
semantics).

#### Justification

This option directly resolves the semantic mismatch the reporter raised:
the ValueSet name communicates "recommended", and `extensible` is the FHIR
binding strength whose definition matches that intent ("if a code applies,
use one of these; otherwise a more specific code MAY be used"). It is also
the lowest-risk path for ballot consumers who may have built against the
current ValueSet content but cannot guarantee complete coverage of every AI
provenance scenario the IG ultimately targets. It does, however, reduce
machine-checkable conformance — validators will no longer reject codes
outside the ValueSet — which the workgroup should weigh against the ballot's
maturity goals.

---

### Disposition B: Accept As Requested — Option 2 (rename the ValueSet)

#### Proposal

Keep the binding strength on `meta.security:AItags` at **`required`** and
rename the bound ValueSet so its `title` (and, if practical, its `name`/`id`)
no longer contains the word "Recommended". A natural target is something
like "AI Provenance Codes" or "AI Transparency Provenance Tags". Update all
narrative references to the old name across the IG (slice description, any
guidance pages, glossary). Because `ValueSet.url` SHOULD remain stable for
already-published content, prefer changing only `title` and `name` rather
than the canonical URL; if the canonical URL also embeds "recommended",
introduce a new canonical and deprecate the old one.

#### Justification

If the workgroup intends the slice to be a **closed, machine-enforced** code
list (which a `required` binding asserts), then the right fix is to remove
the misleading word from the artifact label rather than weaken the
conformance contract. This preserves implementer expectations that systems
exchanging AI-tag metadata will produce only known codes, which matters for
provenance and auditability — the very purpose of `meta.security`. The main
cost is the ripple of name/URL changes through the IG and any external
material that already references the old name.

---

### Disposition C: Decline / No Change

#### Proposal

Close FHIR-54141 as "Not Persuasive" (or "Already Addressed" if the WG
considers the existing description sufficient) on the grounds that the
ValueSet name is descriptive rather than normative, and the binding
strength is the only conformance-bearing element. Optionally add a short
clarification to the slice's narrative explaining that the ValueSet name
reflects HL7's recommendation that implementers use these specific codes,
not the FHIR binding-strength keyword.

#### Justification

FHIR binding semantics are defined by the `strength` element, not by the
ValueSet's human-readable name. A reader who consults the StructureDefinition
sees `required`, which is unambiguous. The collision is a documentation /
naming nit rather than a substantive conformance defect, and either of the
two "accept" options carries non-trivial cost (B churns identifiers; A
relaxes machine checking). Declining preserves the current ballot artifact
and avoids re-balloting the slice over a cosmetic concern.

---

### Recommendation

**Recommended disposition: B (rename the ValueSet).**

The slice is `meta.security:AItags`, which exists to make AI-derived content
machine-detectable in security/audit pipelines. That use case benefits from
the strongest conformance guarantee FHIR offers — a `required` binding —
because downstream consumers (audit, access-control, provenance evaluators)
need to recognise the tags without bespoke handling of unknown codes. The
reporter raised a legitimate naming bug, but the right fix is to align the
artifact label with the conformance the IG already wants, not to weaken the
conformance to match a label. Disposition B (rename) preserves the audit /
provenance value of the slice, while Disposition A (extensible) trades that
value away to fix a wording problem. Disposition C is defensible if the WG
truly considers this cosmetic, but the ticket's appearance in the
"Provenance & Tagging" Jan-ballot bucket alongside nine related issues
suggests the area is being actively cleaned up — folding a rename into that
sweep is cheaper now than it will be once external material has accreted
around the misleading name.
