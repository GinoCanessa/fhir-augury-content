## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new UML class diagram to the **inline tagging use case** section of
the AI Transparency IG, modelled after the existing root-level tagging
diagram. The diagram should depict, at minimum:

- The host resource (e.g., `Observation`, `DiagnosticReport`, `Composition`
  section, etc.) with the specific element being tagged.
- The AI-transparency extension applied to that element (showing the
  extension URL and its key sub-extensions/values).
- The reference from the extension to the supporting provenance / AI
  context (e.g., `Provenance` referencing `AIModel` / instrument
  documentation), parallel to how the root-level diagram depicts the
  resource-wide tag.

Adopt the proposed image attached by JIRAUSER28670 in comment 64161 (or a
cleaned-up redraw of it) as the starting artwork, and add 1–2 sentences
of caption/narrative tying it back to the surrounding text. Source the
diagram in the IG's standard diagram format (PlantUML or SVG, matching
the root-level diagram's source) so it can be regenerated.

#### Justification

The request is a low-risk documentation enhancement that brings symmetry
to the IG: the root-level tagging section already has a UML diagram, and
inline tagging is the more nuanced of the two cases (because the tag
attaches at element level rather than resource level), so a diagram
arguably has *more* explanatory value here than at the root level. The
reporter has identified a real readability gap, JIRAUSER28670 has already
produced a candidate diagram, and there is no impact on conformance,
schemas, terminology, or tooling. This is a typical "Persuasive with mod
— add diagram" disposition.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a free-standing UML class diagram, **reuse the root-level
tagging diagram** and add an annotated variant beside it (or a single
combined diagram with two columns/lanes labelled "root-level" and
"inline") that highlights the single delta: the extension is anchored to
a specific element rather than the resource root. Pair the diagram with a
short comparative paragraph that explicitly says, "Inline tagging uses
the same extension mechanism as root-level tagging; the only structural
difference is the anchor point."

Optionally, supplement with a small **example snippet** (JSON or XML)
showing the extension placed on an element, since john_moehrke's comment
notes this is "fundamental to extensions, not specific to our use" and
implementers may benefit more from a concrete example than from another
class diagram.

#### Justification

Addresses john_moehrke's concern (raised in comment 64162) that a UML
class diagram of "extension applied at an element" risks restating
generic FHIR extension mechanics rather than teaching anything
AI-transparency-specific. A side-by-side or comparative presentation
keeps the symmetry the reporter wants while making the *delta* between
root-level and inline tagging — the only thing that's actually new — the
focal point. It also avoids visual duplication in the IG and reduces the
maintenance surface (one source diagram with two views, rather than two
independent diagrams that can drift).

---

### Disposition C: Decline

#### Proposal

Mark as **Not Persuasive** on the grounds that inline tagging is simply
the standard FHIR extension mechanism applied at element scope, and the
generic FHIR specification's extension documentation already covers the
structural model. Resolve the inline-tagging section's clarity gap with a
short prose paragraph and an example instance instead, without adding a
new UML diagram.

#### Justification

This is essentially the position John Moehrke takes in comment 64162:
inline tagging is "fundamental to extensions, not specific to our use".
A UML class diagram of "an extension on an element" risks re-teaching
generic FHIR extension semantics inside an IG that should focus on its
domain-specific contributions. Declining keeps the IG lean and points
implementers to the FHIR core spec for extension mechanics. The cost of
declining is modest because the reporter's underlying concern (clarity
of the inline tagging section) can be addressed via narrative and an
example rather than diagrammatic content.

---

### Recommendation

**Recommended disposition:** B

The reporter's concern is legitimate — the inline tagging section is
asymmetric with the root-level section and harder to follow without a
visual — and JIRAUSER28670's candidate diagram shows the workgroup is
willing to invest the effort. At the same time, john_moehrke's pushback
is well-founded: a standalone class diagram of an extension on an
element risks duplicating generic FHIR documentation. Disposition B
captures both signals: it adds the visual the reporter wants, but frames
it as a *comparative* artefact whose teaching value is the contrast with
root-level tagging, not the extension mechanism itself. It also opens
the door to pairing the diagram with a worked example, which is what
implementers will actually copy when they apply inline tagging in
practice. Disposition A is acceptable as a fallback if the workgroup
finds B too much work for a January-ballot technical correction;
Disposition C should only be chosen if the workgroup believes
implementers are unlikely to read the inline-tagging section without
already understanding FHIR extensions, which seems unlikely for an
AI-Transparency audience that may include non-FHIR-native readers.
