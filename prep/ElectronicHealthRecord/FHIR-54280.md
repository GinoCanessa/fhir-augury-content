## Summary

The ticket asserts that, in the AI Transparency on FHIR IG, the `meta.security`
tag/label mechanism is being treated more strongly than it should be. Because
`Resource.meta.security` is optional and may be applied inconsistently (or not
at all), its presence — or absence — cannot be used as authoritative evidence
that AI did, or did not, influence a resource. The reporter asks the IG to make
this explicit so that readers understand that querying `Provenance` is the
critical step for discovering authoritative evidence of AI authorship/influence,
with security labels serving only as discovery hints.

No other Jira tickets were surfaced via cross-references; the only related
discussion is the AI Transparency work group's January-ballot triage thread on
Zulip (which lists FHIR-54280 alongside nine other "Provenance & Tagging"
ballot items, including the closely related FHIR-54281 and FHIR-54279). There
are no GitHub cross-references, so no repository changes are implied directly
by Augury's link graph; any change would land in the AI Transparency IG source
(HL7/ai-transparency-ig).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the AI Transparency IG narrative (the section(s) that introduce the
two discovery mechanisms — `Resource.meta.security` AI-related labels and
`Provenance`) to add an explicit normative-style statement of the form:

> Because `Resource.meta.security` is optional and may not be applied
> uniformly across systems, the presence — or absence — of an AI-related
> security label SHALL NOT be relied upon as authoritative evidence that AI
> did or did not influence a resource. Security labels are discovery hints
> only. To obtain authoritative evidence of AI authorship/influence,
> consumers MUST query `Provenance` resources whose `target` references the
> resource of interest and inspect `Provenance.agent` (and, where present,
> `Provenance.entity`) for AI actors.

Apply the same framing wherever the IG enumerates discovery options
(landing/index page, the "labelling" section, and any conformance/usage
guidance), and update the corresponding example narratives so that no
example implies a label-only check is sufficient.

#### Justification

This is precisely what the reporter asks for, it is consistent with
Moehrke's own clarifying comment that the IG was always intended to offer
two methods (with an implied combined third), and it aligns with the
broader FHIR pattern: `Resource.meta.security` is optional metadata and
core FHIR already cautions against treating optional metadata as
guaranteed. Tightening the wording is low-cost, low-risk, and directly
addresses the ballot comment without expanding scope. It also coheres
with the rest of the "Provenance & Tagging" cluster on the work group's
January-ballot triage (FHIR-54281, FHIR-54279, FHIR-54707), which all
push in the same direction.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only sharpening narrative wording, restructure the IG's
"discovery" guidance into an explicit, layered model:

1. **Authoritative layer — Provenance.** Define a conformance expectation
   that systems claiming to expose AI-authorship information SHALL
   produce `Provenance` resources for AI-influenced resources, and that
   consumers SHALL use `Provenance` queries (e.g. `Provenance?target=...`)
   as the authoritative source of truth.
2. **Hint layer — security labels.** Keep the AI-related codes in
   `Resource.meta.security` as a non-authoritative discovery hint, with
   wording that they MAY be used to pre-filter or prioritise resources
   for deeper Provenance inspection, but MUST NOT be used as the sole
   basis for an authorship claim.
3. Optionally introduce — as Moehrke suggests in the comments — a light
   Actor / Requirements / use-case section that names the "AI Producer",
   "AI Consumer", and "Auditor" actors and states which mechanism is
   authoritative for each, so the asymmetry between the two mechanisms is
   anchored in actor obligations rather than only in prose.

#### Justification

Disposition A fixes the immediate wording but leaves the underlying
asymmetry between the two mechanisms only implicit. A layered model makes
the relative authority structural: implementers reading any one section
get the same answer about which mechanism is authoritative, and
auditors/AI consumers have a clear contract. The trade-off is scope:
Moehrke's own comment explicitly flags that the formal Actor /
Requirements / use-case work may exceed available resources for the
January cycle. Splitting the change into "narrative tightening now" (A)
and "layered model + actors next cycle" lets the work group capture the
ballot-cycle benefit of A while still committing to B as follow-on.

---

### Disposition C: Decline

#### Proposal

Decline the request as already-addressed: argue that the IG already
identifies Provenance as a discovery method alongside security labels,
and that the optional nature of `Resource.meta.security` is inherent to
core FHIR, so no IG-level change is needed beyond the existing text.
Mark FHIR-54280 as "Not Persuasive" and rely on existing core FHIR
guidance about optional metadata.

#### Justification

There is a defensible case that the request is restating a property of
core FHIR (`meta.security` is optional, therefore not authoritative) and
that adding more text about it risks redundancy with core. Declining
also conserves work-group bandwidth in a cycle that already has 90
remaining ballot items. However, this position is hard to sustain given
that Moehrke's own ballot-comment response acknowledges the asymmetry
"was not clear" — i.e. an actual reader of the IG was misled — which is
exactly the failure mode a clarification ballot comment is designed to
catch.

---

### Recommendation

**Recommended disposition:** A (with B captured as a follow-on for a
later cycle)

Disposition A is the right call for this ballot cycle. The reporter has
identified a specific, real ambiguity — and Moehrke's own response
("It seems this was not clear") concedes the point — so Disposition C
("Not Persuasive") is not credible. Disposition B is the right
*structural* answer but, as Moehrke notes in the second comment,
introducing formal Actors / Requirements / use-case tracking is a
larger piece of work than the January ballot can absorb. Accepting A
delivers the clarification the reporter asked for at low cost and with
no behavioural break for existing implementers (security labels remain
valid; their evidentiary weight is just correctly characterised), while
leaving the door open to adopt the layered/actor model in a later
cycle as part of the broader "Provenance & Tagging" cluster
(FHIR-54707, FHIR-54281, FHIR-54279, FHIR-54245, FHIR-54244, etc.).
