## Summary

Brian Postlethwaite (EHR WG, against the AI Transparency on FHIR IG, Jan 2026
ballot) raises four related concerns about the **AI Data profile**:

1. The narrative sentence *"Given that this is a profile on DomainResource,
   it can't be used in an IG, but it can be used for validation purposes."*
   is unclear to readers and should be rewritten so the intent is obvious.
2. The profile is missing **examples**. Reviewers want to see concrete
   instances of the kinds of resources an AI might typically produce or
   consume so the profile's intended use is grounded.
3. It is ambiguous whether the profile's tagging behaviour belongs on
   `Resource.meta.security` or `Resource.meta.tag`, and the IG should
   spell out the security implications.
4. The IG does not clearly distinguish between *"this resource was used
   as input to an AI"* and *"this resource was generated by an AI"* — a
   significant semantic difference that a single tag value cannot
   capture. The current AI Data definition (*"A resource that captures
   the input and output data used by an AI model in generating or
   enhancing FHIR resources"*) is called out as feeling wrong.

There are no Jira cross-references; the only related discussion is the
ballot-triage Zulip thread that grouped this ticket under **"Profile &
Model Structure"** (one of nine Jan-ballot categories). John Moehrke has
already partially responded in the ticket: profiles on `Resource` /
`DomainResource` can't be bound to a real resource (they exist only for
validation), and the tagging vocabulary is intentionally a *security*
vocabulary recommended for `meta.security`. The remaining items
(examples, "used vs. created" semantics, definition text) are still
open.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a bundle of four documentation/profile fixes and
apply them to the AI Data profile in the AI Transparency IG:

1. **Rewrite the "profile on DomainResource" sentence** in the AI Data
   profile narrative to plainly state: *"`AIData` is a validation-only
   profile defined on `DomainResource`. Because every concrete FHIR
   resource is itself derived from `DomainResource`, this profile is
   not intended to be claimed via `meta.profile` on an instance; it
   exists so that validators can check the security tag and related
   constraints against any DomainResource-derived resource."*
2. **Add at least two examples** showing the profile applied (for
   validation) to representative AI-produced resources — e.g. a
   DocumentReference of an AI-generated discharge summary draft and an
   Observation interpreted by an AI assistant — each carrying the
   relevant `meta.security` tag.
3. **Add a short "Security label vs. workflow tag" subsection** that
   explicitly states the AI marker is intended to live on
   `Resource.meta.security` (not `meta.tag`), names the value set used,
   and explains the access-control / handling implication (i.e. why a
   security-classification vocabulary was chosen instead of a workflow
   tag).
4. **Disambiguate "used by AI" vs. "created by AI"** by either
   (a) splitting the existing single value into two distinct security
   label codes (e.g. `AIINPUT` and `AIOUTPUT` in the IG's code system) or
   (b) keeping a single AI marker and pairing it with a `Provenance`
   resource whose `agent.role` distinguishes input vs. output. Update
   the profile definition text accordingly so the wording Brian flagged
   ("input and output data used by an AI model in generating or
   enhancing FHIR resources") is replaced with a precise statement of
   what a single tagged resource means.

#### Justification

All four items are scoped to the AI Data profile page and its
companion examples — no breaking model changes are required. John
Moehrke's comment already concedes the underlying intent on items 1
and 3 (the wording is the issue, not the design), so accepting them
is essentially editorial. The examples requested in item 2 are a
standard ballot expectation for any IG profile and improve adoption.
Item 4 is the only substantive design clarification, and it directly
addresses a real semantic gap that implementers will otherwise resolve
inconsistently; pairing the AI marker with `Provenance` (option b)
aligns with how FHIR already separates "what something is" (security
tag) from "how it came to be" (Provenance), and is consistent with
EHR-WG patterns elsewhere.

---

### Disposition B: Alternative Approach

#### Proposal

Split the ticket into two and treat the items differently:

- **Editorial sub-bundle** (items 1, 2, 3): accept as a technical
  correction. Rewrite the DomainResource sentence, add 2–3 validation
  examples, and add the `meta.security` rationale subsection. Roll
  these into the same editorial pass already covering the 32
  pre-approved technical corrections.
- **Semantic sub-bundle** (item 4 — "used by AI" vs. "created by AI"):
  defer to a follow-up substantive ticket rather than resolving in
  this ballot. Resolve FHIR-54725 itself with a narrative note that
  acknowledges the ambiguity, points to the new ticket, and recommends
  that early adopters use `Provenance` to disambiguate input vs.
  output until the WG adopts a normative approach. Cross-link with
  **FHIR-54757** (paired in the same triage row) so any model-level
  rework is considered together.

#### Justification

Item 4 is genuinely a design question (split the code, add a
companion `Provenance` pattern, or normatively model both roles in the
profile) and warrants its own discussion, possibly with security WG
input. Bundling a model decision into what is otherwise a documentation
ticket risks either rushing the decision or stalling the editorial
fixes that everyone agrees on. Splitting also matches the WG's own
triage signal — the ballot grouping flagged this ticket as
multi-category, and the bolded companion FHIR-54757 hints at related
profile-structure work that should drive the substantive decision.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive*. Keep the current AI Data narrative
and definition text, retain `meta.security` as the binding target, do
not add new examples, and do not introduce additional codes or a
Provenance pattern for input vs. output. Reply with John Moehrke's
existing explanation: the profile is intentionally validation-only on
`DomainResource`, and the security vocabulary on `meta.security` is
deliberate.

#### Justification

This is defensible only as a narrow read of the ticket — Moehrke has
already answered items 1 and 3 in his comment, and one could argue
items 2 and 4 are out of scope for a "missing example" change request.
However, declining outright leaves the narrative wording Brian flagged
unchanged in a balloted IG, and leaves the "used by AI" vs.
"created by AI" ambiguity unresolved at publication, which will
generate downstream tickets. The strongest argument for declining is
purely scope discipline (one ballot ticket, one fix); against it is
that Brian's points are substantively correct.

---

### Recommendation

**Recommended disposition:** B

The four items in the ticket are not equivalent. Items 1, 2, and 3 are
documentation fixes that John Moehrke has already conceptually agreed
with in-thread, and they fit the editorial pass already covering the
32 approved technical corrections — there is no reason to delay them.
Item 4, however, is a real semantic-design choice (single tag vs. split
codes vs. tag-plus-Provenance) that materially affects how
implementations will represent AI involvement. Forcing that decision
into this ticket either rushes it (Disposition A) or under-serves the
editorial fixes by lumping them with a contested redesign. Declining
outright (Disposition C) leaves balloted text Brian and Moehrke both
agree is unclear. Splitting — accept the editorial portion now,
spawn a follow-up tied to FHIR-54757 for the input/output semantics —
gives the WG a clean win on the parts that are uncontroversial while
preserving the design space for the part that isn't.
