## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the IG diagram (the image attached as `26538_image-2026-01-14-12-29-50-504.png`,
sourced from a diagram asset in the AI Transparency IG's
narrative/figures) so that the label that currently reads
`meta.security` (or `meta.security.<other>`) reads `meta.security.code`.
Concretely: edit the source diagram (e.g. the PlantUML / drawio / SVG
asset that produced this figure) so the `meta.security` arrow/label is
re-rendered as `meta.security.code`, regenerate the image, and replace
the published asset. No profile, value-set or example content changes
are required.

#### Justification

The reporter's correction is factually right: a security label on a
resource is carried in `Resource.meta.security`, which is typed as
`Coding`, and the actual code value lives at `Resource.meta.security.code`.
For a diagram that names the field implementers will populate, naming
the leaf (`.code`) is the precise, non-ambiguous reference and is
consistent with how other FHIR IG diagrams call out `Coding.code` /
`Coding.system` when the level of detail matters. Accepting as
requested closes a Highest-priority Technical Correction with a
trivially low implementation cost and no downstream impact on profiles
or examples.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the diagram label at `meta.security` (i.e., name the element rather
than the leaf code), but add a small caption / footnote under the
diagram such as: *"`meta.security` is a `Coding`; the AI-transparency
security label values populate `meta.security.code` (with
`.system` and `.display` set per the bound value set)."* This preserves
the diagram's current abstraction level while addressing the
correctness concern raised by the reporter.

#### Justification

This directly addresses John Moehrke's comment that adding `.code` at
"this abstract level" is "not helpful". The diagram's purpose is to
communicate which element on the resource carries the security tag, and
at that level `meta.security` is the more recognisable handle (it is
how the spec, search parameters and most narrative refer to the slot).
A caption keeps the visual unchanged — avoiding a re-render and any
risk of regressing the figure — while making the precise data path
discoverable for implementers who will actually populate the field.
Trade-off vs. Disposition A: it costs a bit of explanatory prose but
preserves diagram readability and keeps the IG editorially consistent
with how `meta.security` is named elsewhere in the document.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive* on the grounds that the diagram is an
intentionally abstract conceptual figure where naming the
`Coding`-level leaf would over-specify the intent and is not required
for reader comprehension. Leave the diagram unchanged.

#### Justification

This is essentially the position John Moehrke staked in his comment.
For a high-level conceptual figure, FHIR style across the core spec and
several IGs frequently labels the named element (`meta.security`,
`meta.tag`, `code`) rather than spelling out `.code` / `.system` /
`.display`, on the understanding that the reader knows a `Coding` has
those parts. Declining is defensible because (a) `meta.security` is
not *wrong*, only less specific, and (b) editorial consistency with the
rest of the IG is a legitimate reason to resist a per-figure
correction. Risk: it leaves a visible inconsistency that a future
balloter is likely to flag again.

---

### Recommendation

**Recommended disposition:** B

The reporter is technically correct (Disposition A) and the
co-chair-style comment from John Moehrke is also defensible
(Disposition C); both are partly right. Disposition B threads that
needle: it acknowledges the correctness concern by making
`meta.security.code` explicit *in the narrative*, while preserving the
diagram's chosen abstraction level so the figure remains readable and
consistent with how `meta.security` is named elsewhere in the IG. It
also avoids the cost and regression risk of re-rendering a diagram
asset for what is, ultimately, a one-word clarification. If the
workgroup decides the diagram itself must change, Disposition A is the
clean fallback and is a near-zero-effort edit; Disposition C should
only be selected if the WG is confident no future balloter will
re-raise the same point.
