## Summary

The reporter asks the AI Transparency on FHIR IG to drop the option,
described in section 2.6.4, of conveying a Model Card by simply pasting
its Markdown into `Device.note.text`. The IG currently presents this as
one of two ways to associate Model‑Card content with a `Device` resource
representing an AI model; the other approach uses the
`extension:modelCardDescription` extension. The reporter argues that
allowing both encodings increases variability and harms downstream
interoperability: `Device.note` is an untyped "bag of stuff" that can
carry any kind of free‑text annotation, so consumers cannot reliably
detect that a particular note is in fact a Model Card, and the inline
option only works for Markdown‑formatted cards anyway. The proposal is
to remove the quoted sentence from 2.6.4 and make the
`modelCardDescription` extension the only sanctioned mechanism for
referencing a Model Card on a `Device`. There are no related Jira
tickets cross‑referenced from the FhirAugury index for this issue; the
only surfaced discussion is the Electronic Health Record / AI
Transparency IG January ballot triage thread on Zulip, where the ticket
appears in the "Profile & Model Structure" bucket.

## Details

**Description:**

The text under 2.6.4 reads: "

#### R4 Simply put the Model-Card markdown into the note.text of the Device

One choice is to just put that Markdown Model-Card into the
Device.note.text element. This is not wrong from the definition of that
element, but it may not be obvious to one looking at the Device
resource that there is meaning to the markdown given."

This allowed extra modeling option would increase variability and make
interoperability more complex downstream. Strong suggestion to make the
extension `extension:modelCardDescription` the only allowed element to
encode the reference to the model card. Also, `Device.note` is a "bag
of stuff" that might encode many other types of data, and being untyped
makes it hard for consumers to parse the information.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the entire `R4 Simply put the Model-Card markdown into the
note.text of the Device` sub‑section (the H4 heading and the paragraph
beneath it) from section 2.6.4 of the AI Transparency on FHIR IG. After
the change, the IG will document only the `extension:modelCardDescription`
extension as the way to associate Model‑Card content with a `Device`
representing an AI model under R4. Add a short sentence in the
remaining 2.6.4 prose explicitly stating that
`extension:modelCardDescription` is the **only** sanctioned mechanism in
R4 and that implementations MUST NOT rely on plain Markdown placed in
`Device.note.text` to convey a Model Card. (No StructureDefinition or
profile invariants change is required because the IG already defines
the extension; this is purely IG narrative tightening.)

#### Justification

- The reporter, the commenter (john_moehrke), and the underlying
  argument all converge: the inline‑in‑`Device.note.text` option only
  works for Markdown cards, and `Device.note` is intentionally a free
  annotation slot with no semantic typing — consumers cannot
  distinguish a Model Card from any other clinical or operational note
  there.
- One sanctioned representation per concept is a long‑standing FHIR
  guidance for IGs ("don't offer two equally‑valid ways to do the same
  thing"), particularly for profiles whose primary value is reducing
  implementer choice.
- The change is editorial in scope (delete an option, add one
  prohibitive sentence); it does not break any existing conformant
  instance, because anything that was already using the extension stays
  conformant. Instances that placed the Model Card only in
  `Device.note.text` were already not detectable as Model Cards by
  conformant consumers, so removing the option has no real downstream
  cost.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both representations but stop presenting them as equivalent. Edit
2.6.4 so that the IG:

1. States that `extension:modelCardDescription` is the **normative,
   required** way to convey a Model Card on a `Device` resource, and
2. Reframes the `Device.note.text` option as a non‑normative *fallback*
   for legacy systems that cannot author the extension, with explicit
   guidance that producers using the fallback SHOULD also populate
   `Device.note.author`/`time` and SHOULD use a leading marker (e.g.
   "Model Card:") in `note.text` so that consumers have any chance of
   detecting it.

Optionally, define a small marker — e.g. a code on `Device.note`'s
extension or a `Device.note` `coding` (using a standard tag) — so that
the legacy fallback is at least discoverable.

#### Justification

This approach acknowledges the reporter's interoperability concern but
preserves a documented escape hatch for implementers whose base R4
servers do not yet author HL7 extensions on `Device`. Trade‑off versus
A: it keeps more text in the IG and asks consumers to handle two paths,
which works against the reporter's "reduce variability" intent. It is
defensible only if the workgroup has direct evidence that real
implementations cannot adopt the extension on the timeline of this
ballot cycle. Without such evidence, A is cleaner.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed** / **out of scope for this ballot**.
Leave the 2.6.4 text as published, on the basis that the IG narrative
already states the extension is the recommended mechanism and that
`Device.note.text` is offered only as a permissive option, not as
guidance to be followed. Mark the ticket "Persuasive with mod" only to
the extent of clarifying which option the IG prefers, without removing
the inline option.

#### Justification

Defensible only if the workgroup believes that removing a documented
option in a Standard‑for‑Trial‑Use IG would surprise implementers who
have already deployed against the published text, and that an editorial
preference statement is sufficient. The weakness — and the reason this
is the least attractive option — is that the reporter's substantive
point is correct: an untyped Markdown blob in `Device.note.text` is not
recoverable by a conformant consumer. Leaving the option in place
locks a known interoperability hazard into the spec.

---

### Recommendation

**Recommended disposition:** A

The ticket is small, the request is specific, the only commenter on the
ticket agrees with the reporter, and there is no Zulip dissent on
record — the only Zulip surface for this ticket is a triage catalogue,
not a debate. The substantive argument (`Device.note` is untyped and
the inline option only works for Markdown) is unanswerable on its own
terms, and the cost of accepting is essentially zero: no
StructureDefinition changes, no breakage of instances that already use
the extension, and no breakage of instances that used the inline path
because those instances were never machine‑detectable as Model Cards in
the first place. Disposition B's "documented fallback" only buys value
if the workgroup has evidence of implementers who cannot author the
extension; absent that evidence it just re‑introduces the variability
the reporter wants removed. Disposition C asks the workgroup to defend
keeping a known interoperability hazard in the IG and is the hardest
position to justify on the ballot floor.
