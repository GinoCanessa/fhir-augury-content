## Summary

The reporter notes that on the AI Transparency IG (January 2026 ballot
snapshot), the "usages in the FHIR IG Statistics" cross-reference link
shown on profile pages is broken. Two concrete examples are given:
`StructureDefinition-AI-data.html` and
`StructureDefinition-AI-ModelCard.html`. Following the link from either
page lands on the FHIR IG Statistics (qa.fhir.org) site with the error
`Unknown Resource: StructureDefinition/AI-data in package
hl7.fhir.uv.aitransparency|current` (and the analogous message for
`AI-ModelCard`). The defect is a publication/tooling issue: the link the
IG publisher emits for "usages in the FHIR IG Statistics" does not match
how the AI Transparency IG's StructureDefinitions are registered in the
statistics service, so the cross-IG usage lookup cannot resolve the
resource. A second commenter (JIRAUSER31049) has confirmed the same
behaviour. There are no linked Jira tickets and no GitHub
cross-references; the only cross-reference is the Electronic Health
Record / AI Transparency triage Zulip thread that catalogs the January
ballot issues and places this ticket in the "Technical Errors & Links"
bucket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a technical correction to the AI Transparency IG and fix
the "usages in the FHIR IG Statistics" links so they resolve correctly
for every profile page in the IG. Concretely, on the next IG publisher
build of `hl7.fhir.uv.aitransparency`:

1. Verify the published `package.json` / `ImplementationGuide` resource
   uses the canonical `http://hl7.org/fhir/uv/aitransparency` and the
   package id `hl7.fhir.uv.aitransparency` consistently.
2. Confirm each StructureDefinition (notably `AI-data` and
   `AI-ModelCard`) has a canonical URL of the form
   `http://hl7.org/fhir/uv/aitransparency/StructureDefinition/<Id>` and
   that the `id` matches the file name used by the publisher.
3. Re-run the IG publisher and re-publish the package to the FHIR
   package registry so qa.fhir.org's statistics index can pick up the
   resources under `hl7.fhir.uv.aitransparency|current` (and the
   ballot-tagged version).
4. After republication, click through both example links to confirm
   qa.fhir.org returns a usages page rather than the "Unknown Resource"
   error, and spot-check at least one other profile in the IG.

No editable spec content needs to change; the fix is in build / publish
metadata and re-indexing on qa.fhir.org.

#### Justification

The reporter and a second commenter both reproduce a concrete broken
link on the published ballot site, which the IG triage thread has
classified as a "Technical Error & Link". Fixing publication metadata
(or re-publishing so qa.fhir.org's stats index matches the canonical /
package id the page links to) is exactly the kind of low-risk, no-spec-
change correction that "Technical Correction" tickets are intended for,
and it directly restores the navigation behaviour the reader expects.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) re-publishing this single IG, address
the defect at the source: have the IG publisher omit or guard the
"usages in the FHIR IG Statistics" link when the target package /
canonical pair is not yet known to qa.fhir.org's statistics index, and
have qa.fhir.org return a friendly "no usages indexed yet" page instead
of an `Unknown Resource` error. The AI Transparency IG would still be
re-published as in Disposition A, but the fix would be paired with a
cross-IG-tooling change request (filed against the IG publisher and/or
the FHIR Tooling work group) so this class of broken link does not
recur for other IGs in pre-publication or `current` state.

#### Justification

This is the same symptom Abderrazek and the second commenter are
hitting, but it is almost certainly not unique to AI Transparency: any
IG whose `current` package has not yet been indexed by qa.fhir.org will
exhibit the same "Unknown Resource" error from this link. A tooling-
level fix prevents a steady stream of analogous "Technical Correction"
tickets across other work groups' IGs, and degrades gracefully when the
stats index is simply behind the publication. The downside is that the
fix lives outside this work group's direct control (FHIR Tooling /
qa.fhir.org maintainers), so the AI Transparency-specific
republication should still happen so this ballot's links work for
reviewers in the meantime.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" / "Not in this IG" on the grounds that the
broken behaviour is a property of the qa.fhir.org statistics service
(an external tooling component), not of the AI Transparency IG content
itself, and the IG's own profile pages render correctly. The reporter
would be redirected to file the issue against the FHIR IG Publisher /
qa.fhir.org tooling tracker.

#### Justification

The qa.fhir.org "FHIR IG Statistics" page is generated by shared
tooling, not by this IG's authors. Strictly speaking the AI
Transparency profile pages are correct; the failure happens on a
different site that this work group does not maintain. Declining keeps
the AI Transparency ballot scope tight and avoids the work group
taking ownership of a defect it cannot directly fix.

---

### Recommendation

**Recommended disposition:** A (with a follow-up tracking item in the
spirit of B).

The reporter's complaint is concrete, reproducible, and trivially
verifiable from the two URLs in the description, and a second voice has
already concurred. The triage thread has explicitly categorized this as
a "Technical Error & Link", which is the lightest-weight bucket the
work group has and signals that no design discussion is needed — the
appropriate response is to make the link work. The simplest, lowest-
risk path is to ensure the AI Transparency IG is published in a way
qa.fhir.org's statistics index can resolve (Disposition A), so that
ballot reviewers stop hitting the dead end. Disposition C is
unattractive because it leaves the published ballot in a visibly broken
state for reviewers and pushes the burden back onto the reporter, even
though the AI Transparency authors are the ones who can most easily
republish the package. Disposition B identifies a real systemic gap,
but it is best handled as a follow-up tooling request rather than a
reason to delay this ticket's correction; the work group should adopt
A now and consider opening a separate FHIR Tooling ticket to harden
the publisher / qa.fhir.org behaviour for other IGs.
