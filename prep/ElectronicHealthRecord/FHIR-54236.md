## Summary

The ticket is a January-2026 ballot comment against the AI Transparency on
FHIR IG (`hl7.fhir.uv.aitransparency`, owned by the Electronic Health
Record work group). The submitter flags a paragraph in the General
Guidance narrative that is grammatically and conceptually unclear: it
opens with "In this use-case need…", repeats the phrase "In this
Observability Factor…" in two consecutive sentences, mixes the terms
"use-case" and "Observability Factor" without clarifying whether they are
the same thing, and contains the typo "distinguishes" (should be
"distinguish"). The ticket asks the authors to (a) name the specific use
case being referred to, (b) avoid the duplicated sentence opener, and
(c) harmonise terminology between "use-case" and "Observability Factor".
The ticket has already been triaged as **Persuasive with Modification**
and resolved (per the resolution note and the GitHub cross-reference)
via PR **HL7/aitransparency-ig#22**, in which the offending text was
removed entirely as part of a broader restructure of the General
Guidance page. No related Jira tickets were surfaced by the
cross-reference index; the only Zulip context is John Moehrke's January
ballot triage roll-up, which lists FHIR-54236 in the "Documentation &
Clarity" bucket alongside the other 30 narrative-clarity tickets in this
ballot.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Confirm the existing resolution: the offending paragraph in
`input/pagecontent/general_guidance.md` is removed in its entirety as
part of the General Guidance restructure already merged in
**HL7/aitransparency-ig#22**. The block-vote disposition for
FHIR-54236 is recorded as **Persuasive with Modification**, with the
modification being deletion (rather than rewording) of the unclear
paragraph; the typo flagged in the comment ("distinguishes" →
"distinguish") is moot because the sentence no longer exists. No
further authoring change is required in this repo.

#### Justification

This matches what was actually done. The ticket reporter's three
concerns (ambiguous "use-case", duplicated sentence opener, conflated
"use-case" / "Observability Factor" terminology) and the commenter's
typo are all resolved by removal, and the restructure performed in
PR #22 simultaneously addresses several other Documentation & Clarity
tickets in the same Block-Vote-2 grouping — consistent with the Zulip
triage that bucketed this ticket alongside the other 30 narrative
cleanups. Because the change is purely narrative inside
`input/pagecontent/general_guidance.md`, the FSH artifacts
(`SD_Provenance.fsh`, `EX_Provenance.fsh`, `EX_Tagging.fsh`), the IG
identity in `sushi-config.yaml`, and the cross-repo touch points
(HL7/fhir, HL7/UTG, `hl7.fhir.uv.security-label-ds4p`) are unaffected,
so the impact is contained.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of deleting the paragraph outright, **rewrite** it inside the
post-restructure General Guidance page so the IG retains an explicit,
narrative description of why a Provenance/Tagging consumer needs only
"AI was used vs. not used" granularity in some scenarios. Concretely,
in `input/pagecontent/general_guidance.md` under the heading that now
covers the relevant Observability Factor (post-PR #22 this is one of
the `###` headings under "AI Usage Implementation Methods"):

1. Name the use case the paragraph was about (e.g., "Tagging-only use
   case" or whichever use case in `use_cases.md` it referenced) with
   an explicit cross-link to the corresponding entry in
   `input/pagecontent/use_cases.md`.
2. Replace the duplicated "In this Observability Factor…" sentences
   with a single sentence enumerating what is *not* required (model
   identity, prompts, influence on the data) and one sentence
   explaining *why* (the consumer only needs the "AI was used / not
   used" distinction).
3. Pick one term — "Observability Factor" — and stop using "use-case"
   as a synonym in this paragraph; reserve "use case" for the entries
   in `use_cases.md`.
4. Fix the "distinguishes" → "distinguish" typo at source.

Audit any `general_guidance.html#…` anchors in `index.md`,
`use_cases.md`, and `explainer.md` after the heading or wording change,
and update `input/ignoreWarnings.txt` if the IG-publisher emits new
intentional warnings.

#### Justification

Removing the paragraph (Disposition A) is the lowest-risk fix but
deletes information that the original authors thought worth stating —
namely, that "AI used / not used" is a deliberately coarse granularity
for one of the Observability Factors. Reviewers reading the published
IG after PR #22 lose that rationale. This alternative addresses every
concrete defect raised in the ticket (unnamed use case, repeated
opener, mixed terminology, typo) while preserving design intent, and
it stays within the same authoring file the briefing already flags as
the only narrative source touched. The trade-off vs. A is more
authoring work and a small risk of needing follow-up wordsmithing in a
later ballot cycle; vs. C, it actually gives the reporter the
clarification they asked for.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** on the grounds that the underlying
text no longer exists in the repo (already removed by the
unrelated restructure in PR #22), so the requested wording fixes have
no source to apply to and the ticket is effectively moot. Close
without a Persuasive-with-Modification annotation tying it to PR #22.

#### Justification

A formal "Not Persuasive / moot" close is defensible because the
ticket asks for changes to a paragraph that the implementer has
unilaterally deleted; there is no longer any narrative defect for the
reader to encounter. Recording the disposition this way makes the
audit trail precise: PR #22 was driven by a different set of
restructure tickets (most notably FHIR-54365 in the same
Documentation & Clarity bucket), and FHIR-54236's specific asks
(name the use case, harmonise terminology) were never explicitly
incorporated into the restructured prose.

---

### Recommendation

**Recommended disposition:** A

The ticket is already resolved on the implementation side and the
current Jira state (`Persuasive with Modification`, with the
resolution note pointing at PR #22) accurately describes what
happened: the unclear paragraph was removed as part of the General
Guidance restructure. Disposition A simply confirms that mapping for
the workgroup vote and lets the Block-Vote-2 group close cleanly,
which is consistent with the Zulip triage placement and the briefing's
explicit note that "any further disposition proposal should be framed
as confirming, refining, or revisiting that restructure rather than as
net-new spec authoring." Disposition B is the right call only if the
workgroup, on review of the post-PR-#22 narrative, judges that losing
the "AI used / not used granularity" rationale is a real readability
regression — in which case a small follow-on ticket against
`input/pagecontent/general_guidance.md` is more honest than retroactively
broadening this one. Disposition C is technically defensible but
needlessly pedantic: it discards the link between the ticket and the PR
that actually addressed the reader's complaint, making future audits of
the AI Transparency IG ballot harder.
