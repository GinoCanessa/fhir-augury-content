## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add CPT (Current Procedural Terminology, owner: American Medical
Association) to the IP / Cross-Version Analysis section of the AI
Transparency on FHIR Implementation Guide. Concretely, this would mean
either (a) adding `http://www.ama-assn.org/go/cpt` to the IG's declared
dependent code systems / terminology dependencies (e.g., adding a
ValueSet, profile binding, or example that references CPT codes so the IG
publisher emits the system into the auto-generated IP section), or (b)
manually overriding the IP page in the IG to enumerate CPT and its AMA
licensing notice even though no artifact references it.

#### Justification

If any forthcoming AI-Transparency artifact will plausibly reference
procedure codes — for example, an example `Procedure` resource used as
input to an AI inference whose transparency must be documented — then
having the AMA CPT licensing notice present up front protects implementers
from surprise licensing exposure. This reading treats the IP section as
documentation about *potential* terminology consumers of the IG, which is
the most permissive interpretation of the submitter's request.

---

### Disposition B: Alternative Approach

#### Proposal

Decline to add CPT to the IP section directly, but acknowledge the
underlying concern by adding a short paragraph elsewhere in the IG (e.g.,
in the "Terminology" or "Use of Codes" narrative page) that explicitly
notes: (1) the IP section is generated automatically from code systems
that the IG actually uses; (2) implementers who reference licensed code
systems such as CPT, LOINC, SNOMED CT, ICD-10-CM, etc. in their
AI-Transparency payloads are responsible for complying with those code
systems' licensing terms; and (3) point to standard HL7 guidance on
licensed terminology use. No change to the IP page itself.

#### Justification

This addresses the *plausible* underlying concern — that consumers of the
IG might reference CPT in real deployments and need to be made aware of
licensing obligations — without distorting the IG-publisher-generated IP
page or implying that the IG itself depends on CPT. It avoids a
hand-edited override that future publisher runs would either drop or
fight with, and it stays consistent with how cross-version / IP pages are
generated across the HL7 ecosystem.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Mark the ticket **Not Persuasive** and close it, with the resolution
already authored on the ticket: *"we don't use any CPT. The section is
auto generated by the IG publisher based on actual use."* No spec change.

#### Justification

The IP / Cross-Version Analysis section of an HL7 IG is produced
mechanically by the IG publisher from the code systems and value sets
that the IG actually binds, references, or includes examples for. The AI
Transparency IG does not currently use CPT in any artifact, so injecting
CPT into the IP section would (a) misrepresent the IG's actual
terminology surface, (b) create a maintenance hazard the next time the
publisher regenerates the page, and (c) imply an AMA licensing
relationship that does not exist. The submitter has not provided a use
case that would cause CPT to legitimately enter the IG's terminology set.
The current ticket Resolution and the reviewer's comment ("why? We don't
use any CPT.") already converge on this position.

---

### Recommendation

**Recommended disposition:** **C (Decline / Not Persuasive)**

The ticket as written is a one-line request with no use case, no example,
and no proposed binding that would justify CPT entering the AI
Transparency IG's terminology surface. The IP section of an HL7 IG is an
*output* of the publisher, not an editorial page — adding a code system
there for which the IG has zero references is both technically incorrect
(the next publisher run will either drop it or have to be specially
coerced) and substantively misleading (it would suggest an AMA-licensed
dependency that does not exist). The reviewer (John Moehrke) and the
ticket's existing Resolution already converge on this answer; the
work group should ratify "Not Persuasive" on the record, with the
resolution wording already on the ticket. If a future use case emerges
that legitimately needs CPT in this IG (e.g., an example or profile
binding), it can be raised as a fresh, scoped change request and CPT
will then appear in the IP section automatically. Disposition B remains
a reasonable fallback if the work group wants to pre-empt licensing
confusion among implementers, but it is not strictly responsive to what
this ticket asked for.
