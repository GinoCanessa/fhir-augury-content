## Summary

Hans Buitendijk asks the Electronic Health Record / AI Transparency on FHIR
work group to clarify the purpose, scope, and intended use of the **AIdata**
profile defined in the AI Transparency IG (v1.0.0-ballot, January 2026
ballot). The reporter raises three concrete concerns:

1. **Profile-of-DomainResource paradox.** The IG describes AIdata as a
   profile on `DomainResource` and notes "it can't be used in an IG," yet
   the IG ships it. The reporter wants explicit guidance on how the profile
   is meant to be applied and why a profile on `DomainResource` was chosen
   at all.
2. **Resource vs. metadata framing.** The description calls AIdata "a
   resource that captures the input and output data used by an AI model,"
   but the actual mechanism is `meta.security` tagging on existing
   resources, not a standalone resource. The reporter asks whether the
   intent really is for every resource that was input or output of AI to
   carry these tags (in addition to a Provenance resource).
3. **Input-vs-output asymmetry in `ProvenanceVS`.** If AIdata is supposed
   to cover both AI inputs and AI outputs, why does the recommended
   `ProvenanceVS` (AIRPT, CLINAST_AIRPT, AIAST_CLINRPT, AIAST) only carry
   output-oriented codes? And if input data is also to be tagged, does
   that not bloat resources used as inputs to many AI processes — and
   wouldn't audit-style mechanisms be a more appropriate place for that?

John Moehrke replied (2026-01-26) clarifying that "the tag method is only
used to tag output from AI. There is no way to use meta.security to
indicate that the data was used by AI; that would be an inappropriate use
of meta.security." That answer resolves part of (3) directly but leaves
(1) and (2) — and the description text that conflates "input and output"
— unaddressed in the IG itself.

There are no linked or cross-referenced Jira tickets. The only known
discussion artifact is the IG's January-ballot triage table (see Zulip),
which categorises this ticket under "Documentation & Clarity."

## Details

**Description:**

As this is a profile on DomainResource and the description states "it
can't be used in an IG," yet it is, more explanation is needed how this
profile is supposed to be used.

Furthermore it states "A resource that captures the input and output data
used by an AI model in generating or enhancing FHIR resources." yet it is
not a resource in itself, but metadata on a resource effectively. Is the
essence that every resource that was input or output of AI includes these
tags in addition to a Provenance resource? If so, why does
[Recommended provenance codes - AI Transparency on FHIR v1.0.0-ballot](https://hl7.org/fhir/uv/aitransparency/2026Jan/ValueSet-ProvenanceVS.html)
not have input related values as all seem to output related.

If the intent is to also tag data used as input it seems that data being
used for many AI processes will end up with substantial tags over time,
bloating the resource. Should that really be on the resource that was
input? Or elsewhere from an audit perspective as needed?

**Comments:**

- **john_moehrke (2026-01-26):** "the tag method is only used to tag
  output from AI. There is no way to use meta.security to indicate that
  the data was used by AI, that would be an inappropriate use of
  meta.security."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation-only clarification of the **AIdata** profile
in `input/fsh/EX_Tagging.fsh` and in `input/pagecontent/general_guidance.md`
(specifically the `### Tagging` and `### Defining the AI Model` sections)
of `HL7/aitransparency-ig`. Concretely:

1. Rewrite the `Description` of `Profile: AIdata` (currently:
   *"A resource that captures the input and output data used by an AI
   model in generating or enhancing FHIR resources. Given that this is a
   profile on DomainResource, it can't be used in an IG, but it can be
   used for validation purposes."*) so it no longer says "resource" or
   "input and output." Replace with wording like: *"A validator-only
   profile on `DomainResource` that constrains `meta.security` to require
   at least one slice from `ProvenanceVS`. It is intended solely for tools
   that validate AI-output tagging on any resource type; it cannot be
   referenced from `meta.profile` and is not used at runtime."*
2. Add a short "How to use the AIdata profile" subsection to
   `general_guidance.md` that explains: (a) AIdata exists only as a
   validator construct, (b) it can never appear in `meta.profile` because
   `DomainResource` is abstract (the comment in `EX_Provenance.fsh` lines
   104–112 already documents this), (c) the actual run-time mechanism is
   to apply `meta.security` codes from `ProvenanceVS` directly to the
   AI-output resource, and (d) inputs to AI are out of scope for
   `meta.security` tagging — provenance/audit of inputs is a Provenance /
   AuditEvent concern.
3. Update `ProvenanceVS` documentation (not the value set itself) to
   state explicitly that the codes intentionally describe AI **outputs**;
   point readers to Provenance / AuditEvent for input-side traceability.

#### Justification

This is exactly the request the reporter made — clarify how the profile
is supposed to be used and resolve the "resource vs. metadata" wording.
John Moehrke's comment already settles the substantive point (tagging is
output-only); accepting that as written, and pushing the clarification
into the IG narrative + the FSH `Description`, costs no design change.
The briefing confirms the affected files are authoring sources
(`input/fsh/`, `input/pagecontent/`), so this is a low-risk edit that
respects the "do not touch generated areas" gotcha. It also closes off
the third concern by making the input-side audit boundary explicit
without changing `ProvenanceVS`.

---

### Disposition B: Alternative Approach

#### Proposal

Same intent as A (preserve the output-only semantics), but additionally
restructure the artifact so it stops looking like a resource:

1. **Retire the `Profile: AIdata` artifact** from
   `input/fsh/EX_Tagging.fsh`.
2. Replace its role with a **FHIRPath invariant / `Invariant:` rule**
   (or, alternatively, a `Logical` model under `LM_AI.fsh`) named e.g.
   `ai-output-tagging`, expressed as: *"Any resource designated as AI
   output SHALL carry at least one `meta.security` code from
   `ProvenanceVS`."* Reference this invariant from
   `general_guidance.md` as the conformance hook for validators, and
   remove the AI-resource framing entirely.
3. Update `EX_Provenance.fsh` to drop the now-stale comment about
   `Canonical(AIdata)` not working.

#### Justification

The reporter's first two concerns ("profile on DomainResource that can't
be used in an IG", "describes itself as a resource but is metadata")
both stem from the artifact's *kind*, not just its description. Rewriting
the description (Disposition A) papers over the symptom; replacing the
profile with an invariant or logical-model rule eliminates it. Trade-off:
this is a larger change in the January ballot window and removes a
construct some implementers may already have wired into their validator
pipelines — the briefing flags the IG dependency on
`hl7.fhir.uv.security-label-ds4p` but does not flag downstream consumers
of `Profile: AIdata` specifically, so the blast radius is plausibly
small. Still a heavier lift than A, and it does require the WG to agree
that the validator-only profile pattern is itself a smell.

---

### Disposition C: Decline (Not Persuasive / Already Addressed)

#### Proposal

Decline as **already addressed** and close with a pointer to John

and the comment block in `EX_Provenance.fsh` lines 104–112. Make no
narrative or FSH change for the ballot.

#### Justification

The specific question — "is input data also tagged?" — has been answered
in-thread ("the tag method is only used to tag output from AI"), and the
IG source already carries inline notes explaining the validator-only
nature of the profile. One could argue no spec change is required.

This is the weakest of the three. The reporter is a senior implementer
(EHR WG co-chair); if the answer is not discoverable from the rendered
IG narrative alone, the documentation is in fact unclear, and "read the
FSH source comments" is not an acceptable resolution for a published
ballot.

---

### Recommendation

**Recommended disposition:** A

Disposition A delivers exactly what the reporter asks for, fits the
"Documentation & Clarity" bucket the IG editor (John Moehrke) has
already placed it in, and is achievable inside the current ballot
window without changing artifact identities or value-set membership.
Disposition B is the more *correct* long-term move — a profile on
`DomainResource` that "can't be used in an IG" is genuinely awkward —
but it is a structural change that should be staged for a later ballot
cycle once the WG has agreed in principle and downstream validator
behaviour is understood; trying to land it as a triage-time edit for
the January ballot is risky. Disposition C should be rejected because
the IG itself does not surface the "output-only" rule clearly to
readers, and a Jira comment is not a substitute for narrative.
Recommend the WG adopt **A** for this ballot and open a follow-up
ticket to track **B** as a structural cleanup for the next cycle.
