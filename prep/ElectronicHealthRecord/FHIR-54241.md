## Summary

The submitter, IgnacioJauregui, is reading section **2.3 Tagging** of the
AI Transparency on FHIR IG, which currently scopes the lightweight AI
tagging methodology to "the creation or updating of the given resource."
The ticket asks the workgroup to clarify whether **AI-driven deletion of
resources** is in or out of scope for this IG, and — if in scope —
whether the same tagging mechanism can be reused (perhaps with a
variation) to indicate that a deletion was performed by, or recommended
by, AI.

There are no other Jira tickets cross-referenced to FHIR-54241, so the
question stands on its own. The only Zulip cross-reference is the
ballot triage roll-up posted by John Moehrke, where this issue is
listed (in bold) under the **"Profile & Model Structure"** category for
the January ballot — i.e., it is being treated as a structural /
modelling question rather than a pure documentation fix. John Moehrke
has also commented directly on the ticket, raising two substantive
points: (1) it is not obvious that an AI would *recommend* removing a
resource (corrections via update seem more plausible), and (2) tagging
fundamentally cannot decorate a resource that no longer exists, whereas
**Provenance** is already designed to describe a deletion activity.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend section **2.3 Tagging** of the AI Transparency on FHIR IG to
explicitly cover AI-driven deletion. Concretely:

- Add narrative to 2.3 stating that AI involvement in the *deletion*
  of a resource is in scope.
- Define a tagging variation that can travel with the **DELETE**
  interaction. Two implementation options to specify:
  1. A `meta.tag` (or dedicated security label) carried on the
     **request** of the DELETE (e.g., via an HTTP header or the
     transaction-bundle entry's `request.extension`), since the
     resource itself will not exist post-delete; and/or
  2. A tag carried on the resource's prior version that is preserved
     in the server's history (`vread`) so the AI involvement is
     discoverable from the deleted resource's version history.
- Add at least one example illustrating each option.
- Update the IG's "AI Transparency Tag" value-set / code system page
  with any new code(s) needed (e.g., `ai-deleted` or a modifier on the
  existing tag) and document interaction with `_history`.

#### Justification

The submitter is asking a real gap question: 2.3 is silent on a CRUD
operation that the IG otherwise scopes. Tagging the request /
historical version keeps the existing "lightweight" philosophy of the
section instead of forcing every implementer to stand up Provenance.
It also gives parity with the create/update story, which makes the IG
easier to teach and conformance easier to test.

---

### Disposition B: Alternative Approach

#### Proposal

Keep section 2.3 focused on create/update tagging, and instead add a
short new subsection (e.g., **2.3.1 "AI-driven deletion"**) that
*defers deletion to Provenance*:

- State explicitly that AI-driven deletion is in scope for the IG, but
  that the lightweight tagging mechanism does not apply (tags cannot
  ride on a non-existent resource).
- Require / recommend a `Provenance` instance whose `activity` =
  "delete" (or equivalent code) and whose `target` references the
  deleted resource (by logical id / version), with the AI agent
  represented in `Provenance.agent` using the same agent profile the
  IG already defines for AI-assisted create/update.
- Provide one worked example (Provenance for a DELETE) and a
  cross-reference from 2.3 so readers find it.

#### Justification

This directly addresses John Moehrke's two objections: it acknowledges
that tags cannot decorate a deleted resource, and it leverages
Provenance, which is already the FHIR-native mechanism for describing
a deletion activity. Reusing the IG's existing AI-agent profile inside
Provenance preserves transparency without inventing new tag semantics
or new request-header conventions, and avoids the
implementer-burden / interop risk of carrying tags on the DELETE
request itself (an area with weaker tooling support).

---

### Disposition C: Decline

#### Proposal

Mark AI-driven deletion as **out of scope** for this release of the AI
Transparency on FHIR IG. Add a single sentence to section 2.3
clarifying that the tagging methodology covers create and update only,
and that deletion-related transparency is left to existing FHIR
mechanisms (Provenance, AuditEvent, server `_history`) without IG-level
constraint at this time. Park a follow-up tracker for a future
ballot.

#### Proposal — sub-rationale category

Out of scope / insufficient use cases — addressed by existing
mechanisms.

#### Justification

John Moehrke questions whether AI realistically *recommends deletion*
at all (corrections are the more common pattern, and those flow through
update tagging that the IG already covers). Provenance and AuditEvent
already provide a complete, normative way to attribute a deletion to
an actor — including an AI agent — so the IG would be layering new
constraints on top of mechanisms that already work. Declining keeps
the January ballot tight and avoids introducing a deletion-tagging
variation that may have very few real implementers.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the strongest balance of the three. The submitter
deserves a substantive answer rather than a "out of scope" deferral
(Disposition C), because section 2.3 today reads as if the IG has
considered all CRUD operations when it has not — that ambiguity is
exactly what FHIR-54241 surfaces. At the same time, John Moehrke is
correct on the technical point that `meta.tag` cannot ride on a
resource that has been deleted, which makes Disposition A's "extend
tagging to delete" approach awkward in practice (it forces
implementers into request-header tags or `vread`-only discoverability,
both weaker than the create/update story). Disposition B keeps the
lightweight tagging story intact for create/update, explicitly admits
deletion into scope, and routes it to **Provenance**, which is the
mechanism FHIR already provides for describing a delete activity and
its agent. It also lets the IG reuse the AI-agent profile it already
defines, giving consistent attribution semantics across all three
verbs without inventing new artifacts.
