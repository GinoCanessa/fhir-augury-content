filed by Karrie Schwencer against the *EHRS Functional Model — Record

## Proposed Dispositions

This is a **General Comment** (Jira `Type = Comment`) that does not
request a specific spec change. The dispositions below are framed
accordingly: each names a concrete WG action, since "modify the spec"
is not meaningful when nothing was asked to be modified.

### Disposition A: Accept As Requested

#### Proposal

Resolve FHIR-50701 as **Not Persuasive with Modification** in the sense
of "no change required — comment acknowledged". Concretely:

- Mark the ticket *Resolved* with resolution **Not Persuasive with
  Modification** (HL7 ballot terminology for "comment acknowledged, no
  spec change") and add a disposition comment from the EHR WG
  thanking the submitter and confirming that the EHRS Functional Model
  — Record Lifecycle Events (FHIR) deliverable is now produced with
  the IG Publisher and aligned to FHIR R5, per the work tracked by
  PSS-1956.
- Optionally add a one-line **Publication History / Changes** entry
  to the EHRS-FM RLE IG noting "publication format migrated from PDF
  to FHIR IG (R5)".
- No artifact, profile, value set, or narrative-content change is
  required.

#### Justification

The submitter is observing, not requesting. The substance of the
observation — PDF → FHIR IG tooling, alignment with R5 — is already
true of the deliverable, so accepting the comment "as requested"
amounts to recording that fact and closing the ticket. This matches
the precedent in [FHIR-49312](https://jira.hl7.org/browse/FHIR-49312),
where Public Health/Helios reached the same PDF → FHIR IG conclusion
and resolved *Persuasive / Applied* by committing to re-ballot in IG
form. Closing this ticket cleanly also keeps the WG's ballot disposition
record accurate for any auditor who later asks "did the WG hear the
format-change comments?".

---

### Disposition B: Accept With Follow-Up — Adopt the Common IG Navigation Template

#### Proposal

Treat FHIR-50701 as the prompt to take the migration one step further
and **align the EHRS-FM RLE IG navigation with the common FHIR IG
template** that other HL7 product families have converged on
(see [FHIR-51126](https://jira.hl7.org/browse/FHIR-51126)):

- Top-level pages: **Home, Introduction, Functional, Implementation,
  About, Artifacts**.
- Move existing front-matter (dependencies, change log, references)
  under `Introduction` / `About`.
- Keep RLE-specific functional content (events, lifecycle, actors)
  under `Functional`.
- File a child Change Request against the EHRS-FM RLE IG to track the
  navigation reshuffle and any IG-Publisher configuration changes
  needed (e.g., `sushi-config.yaml` `menu:` block, `pagecontent/`
  reorganisation), so that this Comment ticket can be closed but the
  follow-up work is tracked.

#### Justification

The submitter's underlying point is that the deliverable now "looks
more like an IG". The most concrete way to honour that is to make it
*behave* like the other HL7 IGs from a navigation/UX standpoint, which
is exactly the convergence FHIR-51126 codified for HL7 Europe. Doing
this as a follow-up CR (rather than under this Comment) keeps ballot
hygiene clean — Comment tickets are typically closed without
substantive spec changes — while still capitalising on the momentum of
the PDF → IG migration. The cost is modest (template / menu changes)
and the benefit is consistency with the rest of the HL7 IG portfolio.

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-50701 as **Not Persuasive** with a disposition note that
no action is required: the ticket is an unsolicited positive
observation, contains no specific change request, and the WG does not
need to take action to accept it.

#### Justification

Defensible on the narrow grounds that the ticket's `Type` is *Comment*
and its body contains no actionable proposal — strictly speaking, there
is nothing to "do". However, ballot-process best practice is to
acknowledge format/process comments rather than dismiss them outright,
because doing so can read as the WG having ignored ballot feedback.
Decline is therefore a defensible but suboptimal closure path; it
should be reserved for the case where the WG explicitly does *not*
want to publish a disposition note acknowledging the migration.

---

### Recommendation

**Recommended disposition:** A — *Accept As Requested* (acknowledge and
close), with the WG keeping Disposition B's template-alignment work on
its backlog as a separate, voluntary follow-up.

The ticket is a courtesy observation, not a change request, and the
underlying fact (PDF → FHIR IG tooling, R5 alignment) is already the
state of the deliverable — accepting and closing is the
lowest-friction, highest-fidelity response and is fully consistent with
how the broader community has resolved similar PDF → IG migrations
(FHIR-49312). Disposition B is attractive because aligning the EHRS-FM
RLE IG navigation with the common HL7 IG template (FHIR-51126) would
genuinely improve implementer experience, but bundling that work into
this Comment ticket conflates "acknowledge the comment" with "execute
a non-trivial restructure" and makes the disposition harder to defend.
The cleaner path is to close FHIR-50701 under A and, if the WG agrees
during review, open a fresh Change Request to track the
template-alignment work proposed under B. Disposition C is not
recommended because it gives no public acknowledgement of the
submitter's feedback and risks looking dismissive in the ballot
record.
