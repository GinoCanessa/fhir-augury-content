## Summary

FHIR-54240 requests a restructuring of the **General Guidance** page of the
AI Transparency IG. The reporter observes that §2.1 of the page defines
all observability factors up front, and §2.2 then describes their
implementation separately — forcing readers to scroll back and forth to
correlate the definition of each factor with its implementation guidance.
The request is to refactor the page into per-factor sections where each
observability factor's definition and implementation are co-located.

No other Jira tickets are cross-referenced, but the ticket surfaced in
the workgroup's January-ballot triage Zulip thread under the
"Technical & Formatting" category. The resolution recorded in Jira is
**Persuasive with Modification**, and the actual reorganisation was
delivered via PR
[HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22),
so the ticket is effectively awaiting workgroup sign-off on the already-merged
reorganisation rather than a green-field design decision.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reorganisation already delivered in
[HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22):
refactor `input/pagecontent/general_guidance.md` so that the page has one
section per observability factor, with each section containing that
factor's **definition** (previously in §2.1) immediately followed by its
**implementation guidance** (previously in §2.2). Remove the separate
§2.1/§2.2 split and re-title the top-level grouping to reflect the new
per-factor structure. Update any intra-page anchors referenced from
sibling pagecontent files, the IG ToC entry, and (if present) the
`sushi-config.yaml` menu so no navigation links are broken by the
restructure.

#### Justification

The reporter's argument is straightforward UX: co-locating the
definition and implementation of each observability factor eliminates
the back-and-forth scrolling that the current split requires, and it is
a pattern commonly used in HL7 IGs for parallel concepts (one heading
per concept, sub-sections for "definition" and "implementation"). The
WG chair concurred in-ticket ("agree some organization improvement is
needed") and the change has already been implemented and merged in
PR #22, which indicates consensus on the underlying need. The scope is
purely editorial — it does not change any profile, extension, code
system, binding, or example — so the risk of downstream breakage is
confined to anchor/link rot, which is mitigable at PR-review time.
**Caveat:** this justification should be re-checked against the
`repo-analysis` briefing once generated, because the briefing may flag
anchor/link conventions or cross-repo touch points (e.g., ig-registry,
sibling IGs citing this page) that would be invisible to this report.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the *definition* catalogue of observability factors as a single
introductory section (§2.1 today) — because a consolidated list is
useful for quick reference and cross-linking — **and** add, within that
single-list section, a short anchor/link per factor pointing to its
implementation subsection below. Simultaneously, break §2.2 into one
implementation subsection per factor (each deep-linkable), and add a
"Back to definition" anchor from each implementation subsection to its
definition entry. The page thereby offers both the consolidated
overview the current layout provides *and* the side-by-side navigation
the reporter asks for, without fully merging the two.

#### Justification

Full merging (Disposition A) loses the "cheat-sheet" value of a
consolidated factor list, which is useful for implementers scanning
what observability factors exist before drilling into any one. A
bidirectional-anchor approach addresses the reporter's real pain —
scrolling back and forth to correlate definition with implementation —
while preserving both presentations. It also reduces the risk of
breaking external links into §2.1 as a whole, since that section
survives. The trade-off is a slightly more complex page to maintain
(two blocks that must stay in sync when a factor is added or renamed)
and slightly heavier author tooling (named anchors). Given
PR #22 has already landed, this disposition would mean requesting a
follow-up editorial pass rather than adopting the merged PR as-is.

---

### Disposition C: Decline

#### Proposal

Decline the reorganisation and keep the current §2.1 / §2.2 structure,
treating the reorganisation in PR #22 as out of scope for this ballot
cycle. Close FHIR-54240 as "Not Persuasive — current structure
provides clearer separation of concerns between definitions and
implementation guidance," and — if PR #22 has not yet been published in
a ballot snapshot — consider reverting it.

#### Justification

This is the weakest option. The only defensible rationale for
declining would be that separating "what each factor is" from "how to
implement it" is a deliberate teaching sequence (define everything,
then apply), and that merging them obscures the conceptual hierarchy.
The WG chair has already stated agreement with the need for
reorganisation, and the reorganisation is already merged, so declining
would require reversing an in-flight editorial decision without any
substantive community objection on record.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the natural close-out for this ticket: the reporter's
UX concern is a straightforward editorial complaint, the WG chair
agreed organisation improvement was needed, PR #22 is already merged
against the IG, and the change carries no normative risk. Disposition B
is a reasonable future refinement (and worth noting during ballot
reconciliation if other reviewers ask for a "factor index"), but
unwinding PR #22 to stage it now is disproportionate to the benefit.
Disposition C has no support in the captured evidence and would
reverse an already-accepted improvement. The only follow-up owed under
Disposition A is a mechanical check that no anchor-based links into the
old §2.1/§2.2 structure have been broken by the merge — which the
`repo-analysis` briefing (once generated) should help confirm.
