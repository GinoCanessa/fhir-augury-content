## Summary

The reporter asks the AI Transparency on FHIR IG to add a use case covering
**traceability of FHIR API calls made autonomously by LLM-based agents**, with
or without prior human approval. The request acknowledges that fully agentic
behavior may be out of scope for the current ballot cycle and explicitly
positions the addition as something to consider for a future version of the
IG. The submitter suggests `AuditEvent` as the natural mechanism for capturing
those agent-initiated API interactions.

In the single comment to date, John Moehrke (EHR WG / IG editor) agrees with
the framing: **data-changing** agent actions are already covered by the IG's
Provenance profile, but the broader pattern of *"the agent was asked to do
something and then carried it out"* is not a Provenance event and would
indeed be best captured as `AuditEvent`. He flags it as a good topic for
discussion rather than a clear-cut accept/reject.

No related Jira tickets were surfaced by the cross-reference index. The only
cross-reference is the work group's own January-ballot triage roll-up on
Zulip, where this ticket is grouped under "Use Cases & Examples" and the bold
formatting marks it as one of the items the chair flagged for substantive
discussion (as opposed to the 32 technical corrections that were approved
en bloc).

## Details

**Description:**

> Although this may fall outside the current IG's version if agentic systems
> are not considered, there is a need to have traceability of FHIR API calls
> made autonomously (with/without human approval) by LLM agents. This could
> be achieved through AuditEvent resources.
>
> Please consider adding this use case for future versions where agentic
> workflows are tackled.

**Comments:**

- **john_moehrke (2026-01-26):** "If the agent does create or update data;
  then it would be tracked using the given profile on Provenance. However,
  it is true that if agent is simply asked to do something, that ask and the
  carrying out of the act is not a Provenance activity. That would indeed be
  best recorded as an AuditEvent. I wonder if it would be good to expand to
  this. Good topic for discussion."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new **use-case section** to the AI Transparency on FHIR IG describing
traceability of FHIR API calls made by LLM-based agents (with or without
human approval), and explicitly call out `AuditEvent` as the recommended
mechanism for recording those agent-initiated interactions. The use case
should:

1. Describe at least two scenarios — (a) an agent acting under direct
   human approval per call, and (b) an agent acting autonomously under a
   standing authorization — and show how each is recorded.
2. Specify which `AuditEvent` elements identify the LLM agent
   (`agent.who` / `agent.role` / `agent.type`), the human authorizer when
   present (a second `agent` entry with the appropriate role), and the
   prompt or instruction that triggered the call (e.g., `entity` of type
   `Plan` / `Job` referencing a stored prompt, or a `Provenance` reference
   when one was created).
3. Cross-reference the existing IG Provenance profile so readers
   understand that *data-changing* agent actions remain Provenance events
   and that `AuditEvent` covers the *call/ask* itself, per John Moehrke's
   comment.
4. Stay narrative-only for this ballot cycle (no new profile, extension,
   or value-set commitments) so it does not slip the IG's scope; mark it
   "informative" and tag follow-up tickets for any concrete profile work
   in the next version.

#### Justification

The reporter asked specifically for the *use case* to be added, not for new
constraints, and the IG editor has already conceded on Zulip and in the
ticket comment that `AuditEvent` is the right fit for the
"agent-was-asked-and-acted" pattern that Provenance does not cover.
Documenting the use case now (a) establishes the IG's position before
agentic systems become widespread, (b) gives implementers a normative
narrative they can point to, and (c) keeps the change purely additive and
informative, which avoids re-balloting concerns. The `agent.role` / second-
agent pattern is already established in core `AuditEvent` guidance, so the
IG is documenting an application of existing FHIR machinery rather than
inventing new structure.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding the use case to the current ballot cycle, **persist the
request as a tracked future-scope item**: file (or link) a follow-up issue
on the IG project board titled "Agentic / LLM call traceability use case",
add a single forward-looking sentence to the IG's "Future Considerations"
section noting that agent-initiated API call traceability is recognized as
an open need to be addressed when the WG takes up agentic workflows, and
defer the substantive use-case authoring (scenario walk-throughs, agent
identification pattern, prompt-as-entity guidance) to the next version
where it can be developed alongside other agentic-AI guidance (e.g., model
provenance, prompt-injection auditing, multi-agent orchestration).

#### Justification

The reporter explicitly acknowledges this "may fall outside the current
IG's version" and frames the request for a future version. Treating it as
a placeholder rather than an in-cycle addition (a) keeps the January
ballot focused on the changes that already have community consensus, (b)
avoids publishing an underspecified use case that implementers might read
as normative when the surrounding agentic-AI guidance does not yet exist,
and (c) preserves the option to design the use case coherently with other
agent-related tickets that are likely to arrive in the next cycle.
Trade-off vs. A: less is published now and implementers get no immediate
guidance, but the published material will be more coherent when it does
land.

---

### Disposition C: Decline (Not Persuasive — Already Addressed)

#### Proposal

Decline the request on the basis that **the underlying need is already
covered by existing FHIR mechanisms** outside the IG's scope:
`AuditEvent` (R5+) already supports recording an agent of any type
(human, system, software) with arbitrary `role` codings, including
software acting on behalf of a human; the IG's existing Provenance
profile covers data-changing agent actions; and the core FHIR
`AuditEvent` examples and security/privacy module already discuss
recording calls made by automated actors. Add a one-line note to the
ticket pointing the reporter at base `AuditEvent` guidance and close as
"Not Persuasive — Already Addressed" without modifying the IG.

#### Justification

The IG is intentionally narrow (AI Transparency for current AI uses in
healthcare) and adding a forward-looking agentic use case risks scope
creep into territory that other groups (Security WG for `AuditEvent`,
the broader FHIR-AI initiatives) are better positioned to own. The
reporter's stated mechanism — `AuditEvent` — already exists and works for
this scenario without IG changes; documenting it again in the IG could
create the false impression of an IG-specific obligation. Trade-off:
implementers get no IG-level pointer to the pattern, which slightly
weakens the IG's claim to comprehensive coverage of AI transparency
concerns.

---

### Recommendation

**Recommended disposition:** B (with a soft lean toward A if the editors
have bandwidth in this cycle).

The reporter themselves framed the request as future-scope and the only
substantive comment from the IG editor agrees the topic is "good for
discussion" rather than ready-to-publish. Pushing a fully-developed
agentic-traceability use case into the January ballot in isolation —
without companion guidance on agent identification, prompt-as-entity, and
the boundary with Provenance — is likely to land as an awkward, partial
section that will need to be rewritten when agentic workflows are tackled
in earnest. Disposition B captures the request, signals the WG has heard
it, and protects the integrity of the current ballot. If the editors do
have appetite to write the section now (Disposition A's narrative-only,
informative scope), that is also acceptable because it is purely additive
and aligns with John Moehrke's comment; in that case the WG should commit
to the four-point structure listed under A so the section is not left
under-specified. Disposition C is not recommended: closing as "already
addressed" ignores both the reporter's and the editor's recognition that
the *ask + carry-out* pattern is genuinely a gap in current IG narrative
coverage even if the underlying resource exists.
