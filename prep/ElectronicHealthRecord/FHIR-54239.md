## Summary

The reporter argues that the AI Transparency IG's distinction between
"Request input" and "Reference input" is arbitrary and not operationally
useful: large language models receive a prompt as a single, undifferentiated
context, and the same artifact (e.g., a laboratory result) could be classified
as either category depending on how the prompt is constructed and what
question is asked. The ticket asks the work group either to (a) provide a
crisp, non-ambiguous definition of what falls into each category, or (b)
abandon the split and treat all data fed to the LLM as prompt input. It also
flags that the current two-bucket scheme says nothing about non-textual
modalities (images, audio, biosignals) and asks whether those need their own
categories. The ticket was raised against the January ballot of the
AI Transparency IG and was triaged into the "Data Sources & Context" bucket
along with related items (FHIR-54433, FHIR-54233, FHIR-54211, FHIR-54210,
FHIR-53967), which collectively re-examine how input data is described.
The ticket has been resolved Persuasive with Modification: per the Jira
resolution description and PR
[HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22)
("Rewrite of General Guidance Page"), the section was rewritten so that all
input is now categorized as prompts, collapsing the request/reference
distinction.

## Details

**Description:**

> Request input and reference input distinction seem to be arbitrary. LLMs
> are fed a prompt where there is no distinction of the source or type of
> data that is passed. Systems could classify the data into these two
> proposed categories but there is no uniformity of criteria of what is
> reference and what is request. E.g. a laboratory test can be interpreted
> as a reference or as a request depending of how the prompt is crafted and
> what is the question that is asked to the LLM.
>
> Suggestion to provide a very clear and non-ambiguous definition of what
> kind of data falls in each category, or else treat all data input as a
> prompt to the LLM.
>
> Also consider other data modalities that can be passed, like images,
> audio, biosignals, etc. Where do these kinds of data fall in the proposed
> categories, or are new categories required.

**Resolution:**

> We rewrote the section and now categorize all input as prompts.
>
> This was part of PR: https://github.com/HL7/aitransparency-ig/pull/22

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the rewriter's chosen path — the second branch the reporter
explicitly offered ("treat all data input as a prompt to the LLM") — and
keep the merged change in
[HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22)
as the resolution. Concretely, in
`input/pagecontent/general_guidance.md` of `HL7/aitransparency-ig`:

- Remove the prior `### Request Input` and `### Reference Input`
  subsections and any associated definitions/criteria.
- Replace them with a single normative treatment under a heading such as
  `### Prompt (Inputs to the AI)` that defines "prompt" as the entire
  bundle of content the system passes to the model, regardless of source
  or modality.
- Note explicitly that modality (text, image, audio, biosignal) does not
  define category — all are "prompt" — and that downstream provenance and
  tagging (existing `EX_Provenance`, `EX_Tagging` mechanisms) are the
  intended way to distinguish source, role, and trustworthiness of each
  input element.
- Audit and update intra-IG links (`index.md`, `use_cases.md`,
  `explainer.md`, `sushi-config.yaml` menu, `EX_Doc_ingest.fsh`-driven
  pages) for now-stale `#request-input` / `#reference-input` anchors and
  re-point them at the new heading.

#### Justification

This matches the reporter's own stated alternative ("treat all data input
as a prompt to the LLM") and is the disposition already implemented in
PR #22, which was merged against the `Persuasive with Modification`
resolution. The rewrite removes a categorization the reporter showed to be
ambiguous (the same lab result can be either category depending on prompt
phrasing) and avoids inventing modality-specific buckets that the IG has
no test cases for. It also keeps the IG aligned with how LLM APIs
actually receive input — a single prompt context — and pushes
source/role/trust distinctions to mechanisms the IG already defines
(`SD_Provenance`, Tagging via `v3-ObservationValue` `AIAST` / `DICTAST`),
which are the right place for that information per the General Guidance
structure noted in the repo briefing.

---

### Disposition B: Alternative Approach

#### Proposal

Keep input categorization as a useful concept but make it explicit and
non-binary. In
`input/pagecontent/general_guidance.md`, replace the prior
Request/Reference split with a single "Prompt" section that further
describes a small, role-based **vocabulary of prompt parts** — for
example: `instruction`, `question`, `context`, `evidence`, `example`,
`system_message` — and state explicitly that any of these may carry any
modality (text, image, audio, biosignal). Document that the role is a
property of how the implementer assembles the prompt, not an intrinsic
property of the data, and recommend recording the role per-input via
extension or via `Provenance.activity`/`agent` patterns already
described in the IG. No FSH or `sushi-config.yaml` changes are required;
the change remains scoped to `input/pagecontent/general_guidance.md`
(plus anchor/link audits in the other `pagecontent/*.md` files).

#### Justification

The reporter's first option asked for "a very clear and non-ambiguous
definition" of categories. Disposition A satisfies the second option but
loses any structured way to talk about what role a given input plays in
the prompt — which is information that observability tools, audit, and
reviewers do want. A small role vocabulary preserves that structure
without re-introducing the binary the reporter objected to, and the
gotchas section of the briefing makes clear that this is still a single-
file narrative edit (no FSH, no IG identity changes), so the
implementation cost is comparable to A. The downside is that it adds
work after a resolution has already been merged: the rewrite in PR #22
would have to be revisited and the role vocabulary added on top, and the
work group would need to commit to maintaining the vocabulary in lock-step
with the existing Provenance/Tagging guidance.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the original Request Input / Reference
Input distinction as it stood at the January ballot snapshot of
`input/pagecontent/general_guidance.md`. Cite scope ("ballot-stage IG,
distinction is informative not normative") and rely on the existing
Tagging and Provenance sections of General Guidance to disambiguate
specific cases. No code or narrative changes; PR #22 would be reverted in
the part that collapses the categories.

#### Justification

This is the weakest of the three options. The only defensible argument
for declining is continuity for any early implementer who built against
the Request/Reference framing in the January ballot draft, and the IG is
explicitly `1.0.0-ballot` / `STU1 Ballot` / FMM 2 — i.e., a stage at
which breaking informative changes are expected and tolerated. The
reporter's example (a lab result classifiable either way depending on
prompt phrasing) is not addressed by any clarification short of a
formal, testable definition that nobody on the thread has proposed, and
the gotchas in the briefing flag that arbitrary categorization without
clear criteria leads to inconsistent IG implementations, which is exactly
the harm to avoid at ballot.

---

### Recommendation

**Recommended disposition:** A

The Persuasive-with-Modification resolution and the merged rewrite in
[HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22)
already implement Disposition A, and that change addresses the reporter's
core complaint without inventing structure the IG has not validated.
The reporter's submission itself offered "treat all data input as a
prompt to the LLM" as an acceptable resolution, so accepting A is
faithful to the ticket. Disposition B is attractive on paper but
introduces a new role vocabulary on top of an already-merged ballot
rewrite and would create maintenance overlap with the existing Tagging
and Provenance guidance — best deferred to a follow-up ticket if
implementer feedback during STU1 surfaces concrete need for it.
Disposition C should not be carried forward: at FMM 2 / STU1 Ballot the
work group can absorb the change, and retaining a categorization that
the reporter (and the IG editors, by merging PR #22) have agreed is
ambiguous would propagate the very inconsistency the ticket calls out.
The work group should confirm A and, if desired, log a follow-up to
revisit B during STU1 implementer feedback.
