## Summary

The reporter raises a clarity/scope concern with the **Operations**
subsection of the *Context* chapter in the AI Transparency on FHIR
implementation guide (January 2026 ballot). As balloted, that
subsection lists only two bullets — *Model Context Protocol (MCP)* and
*Agent to Agent (A2A)* — without explaining how an "operation" enters
the model the IG defines for AI context. The submitter argues that the
IG is ambiguous about whether these protocols are merely transports for
fetching data (in which case the resulting payloads should be encoded
as another type of input prompt) or first-class objects in their own
right (in which case the IG is missing the attributes needed to
describe an operation: name, endpoint, protocol, arguments, retry
behaviour, etc.). Citing the rapid churn in this technology space
(skills overtaking traditional MCP usage, A2A still nascent, MCP's own
rough edges becoming visible) they recommend **removing the Operations
section entirely** and instead modelling only the *data* that flows out
of these protocols, surfaced as additional varieties of input prompt.

The ticket has no explicit Jira links and no GitHub cross-references.
Its sole cross-reference is the AI Transparency WG triage roster
(Zulip), where the AI Transparency IG project lead (John Moehrke)
groups it into the **Data Sources & Context** category alongside five
sibling tickets (FHIR-54433, FHIR-54239, FHIR-54211, FHIR-54210,
FHIR-53967) that collectively review how the IG handles inputs to the
LLM. The bold formatting in the triage list flags FHIR-54233 as one of
the items that could plausibly belong in more than one category.

## Details

**Description:**

> The Operation section under context reads as follows:
>
> **Operations**
> • Model Context Protocol (MCP)
> • Agent to Agent (A2A)
>
> Is not entirely clear to us how those enter the model. If they are only
> a means to fetch data, then the resulting data that is passed as
> context to the LLM can be encoded just as another input prompt as
> described in the IG. On the other hand, if the intent is to represent
> the operation itself, then the model lacks dedicated objects to encode
> it's attributes (e.g. name of the operation, endpoint, protocol,
> arguments passed, retries, etc).
>
> Taking into account the fast-moving pace of this technology (skills
> are now the state of art, A2A is in its infancy, and MCP is starting
> to show it's pitfalls) we suggest refraind from including an
> operations section and model only the data that comes out of them as
> coxtext with different types of input prompts.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the **Operations** subsection from the *Context* chapter of the
AI Transparency IG and refactor the surrounding narrative so that any
data obtained via MCP, A2A, agent skills, or similar protocols is
described as a *kind of input prompt* (or as additional context input
material) rather than as an "operation". Concretely, in the next
ballot revision of `input/pagecontent` for the *Context* page:

- Delete the bullets `Model Context Protocol (MCP)` and `Agent to
  Agent (A2A)` from the Operations subsection, and remove the
  subsection heading itself.
- Update the Input Prompt typology (the section the description refers
  to with "another input prompt as described in the IG") to call out
  that retrieved tool/agent output is one of the recognised input
  prompt varieties, with examples for MCP- and A2A-sourced payloads.
- Add a brief non-normative note explaining why the IG does **not**
  introduce dedicated structures for invoked tools/operations at this
  ballot cycle (rapid evolution of MCP/A2A/skills, no stable
  industry-wide attribute set yet).

#### Justification

This is the action the reporter literally requests, and it removes a
genuine ambiguity in the balloted text: today's two-bullet list does
not say whether MCP/A2A are *transports*, *artefacts*, or
*first-class context items*. Folding their outputs into the existing
Input Prompt typology keeps the model surface small and avoids
committing the IG to a tool/operation schema that the reporter (and
arguably the wider ecosystem) does not believe is stable enough to
standardise. It is also consistent with the ballot's "Documentation &
Clarity" theme — the bulk of the EHR triage in Moehrke's Zulip post is
about clarifying scope and removing ambiguity, not adding new
structures.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the *Operations* subsection but rewrite it so that:

1. The subsection is renamed to something neutral such as **Tool /
   Agent Invocations** to avoid confusion with FHIR Operations.
2. The narrative explicitly states the **scope of the IG**: at this
   ballot cycle the IG *records that an operation occurred and the
   payload it returned* (which is then represented as an input
   prompt), but does **not** define a normative structure for the
   operation itself.
3. A small non-normative `informative` extension or sub-element is
   sketched (name, protocol, endpoint, request, response — all
   `0..1`, all marked `MustSupport=false`) so that implementers who
   need to capture invocation metadata have a documented pattern,
   without the IG mandating it.
4. The text flags MCP, A2A, and "agent skills" as the **examples** of
   protocols in scope, not as an exhaustive list, with a forward
   pointer to a future ballot for normative work.

#### Justification

This preserves the option for implementers who genuinely need to
capture *how* a piece of context was obtained (a real concern for
provenance, replayability, and audit — themes that surface elsewhere
in the EHR ballot triage under the Provenance & Tagging category)
without committing the IG to a schema for fast-moving protocols. It
addresses the reporter's "the model lacks dedicated objects" concern
by acknowledging it openly, while addressing the "this technology is
moving too fast to standardise" concern by keeping the structure
informative and minimal. It also keeps the IG's narrative coverage of
MCP/A2A — useful for reader orientation — rather than removing all
mention of them.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as *Persuasive — No Action* (or *Not Persuasive*) and
keep the Operations subsection as balloted, on the basis that the
ambiguity the reporter identifies is best resolved by the upcoming
companion guidance (or by the ticket-author's own implementation
experience) rather than by removing balloted content during
reconciliation.

#### Justification

Defensible only on narrow procedural grounds: the AI Transparency IG
is at ballot, the EHR workgroup has 90 outstanding tickets to
reconcile (per Moehrke's Zulip triage), and FHIR-54233 — uniquely in
the *Data Sources & Context* bucket — proposes a structural deletion
rather than a clarification. Declining preserves the scope agreed
during ballot preparation and avoids opening a connected re-design of
the Input Prompt typology mid-reconciliation. This rationale is weak
on substance because the ticket does flag a real ambiguity, so this
disposition should only be selected if the workgroup believes the
fix-up belongs in a future ballot and not the current one.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern is sound: the balloted Operations
subsection is two bullets without a definition, and the IG does not
say whether a tool invocation is a transport, an event, or an
artefact. Disposition A (delete the subsection outright) cleanly
resolves the ambiguity but throws away useful reader orientation and
discards a hook implementers will eventually need for
provenance/audit — a theme already prominent in the EHR ballot
triage's *Provenance & Tagging* bucket. Disposition C is procedurally
tidy but leaves a documented ambiguity in a balloted IG, which is
exactly the kind of reconciliation item ballot is meant to fix.
Disposition B threads the needle: it acknowledges the reporter's
ambiguity (renaming the subsection, scoping it explicitly), defers the
hard normative work on MCP/A2A/skills to a later ballot when the
underlying protocols stabilise, and gives implementers a documented —
but informative-only — pattern for invocation metadata. It is the
disposition most consistent with both the literal text of the request
and the wider direction of the January ballot reconciliation as
captured in the AI Transparency on FHIR Zulip stream.
