## Summary

The reporter argues that the codes in the AI Transparency IG value set
[`AIdeviceTypeVS`](http://hl7.org/fhir/uv/aitransparency/ValueSet/AIdeviceTypeVS)
have a high degree of conceptual overlap — for example, an LLM is simultaneously
a deep-learning algorithm, a machine-learning algorithm, and an AI — which
makes the codes ambiguous when used to classify a device. They strongly
recommend revising the value set with input from data scientists to produce
a cleaner classification, and additionally suggest introducing a clear
distinction between deterministic and probabilistic systems so that
trustworthiness considerations can be expressed through the value set.
A commenter (JIRAUSER28670) further asks that codes be added for
summarization and Q&A AI systems, which they say are the most rapidly
implemented industry use cases and do not currently fit any of the proposed
functional codes.

No formally linked Jira tickets were returned by the cross-reference query;
the ticket appears in the AI Transparency on FHIR Zulip triage list under
the "Profile & Model Structure" category alongside several other ballot
comments touching the same value set / type axis (e.g., FHIR-54371,
FHIR-54267, FHIR-54264).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revise `ValueSet/AIdeviceTypeVS` in the AI Transparency IG to remove the
overlapping technology-tier codes (e.g., AI vs. ML vs. deep learning vs.
LLM, which form a strict generalization hierarchy) and replace them with
a flat, mutually-exclusive classification informed by a data-science
SME. As part of the same revision:

- Add codes for the functional categories called out in the ticket
  comments (summarization, Q&A / question-answering), and any obvious
  peers (classification, generation, retrieval, prediction/forecasting,
  recommendation).
- Introduce a separate axis (a second value set / extension on
  `Device.type` or a `Device.property`/extension carrying a
  `system-behavior` code) for **deterministic vs. probabilistic** so the
  trustworthiness distinction can be expressed without conflating it
  with the technology axis.

The change applies to the IG's authoring root for terminology
(typically `input/vocabulary/` or the equivalent ValueSet/CodeSystem
sources in the AI Transparency IG repo) plus any narrative pages that
describe how to populate `Device.type` for AI devices.

#### Justification

This is what the reporter and the supporting commenter explicitly
asked for, and the WG lead has already conceded the value set "needs
improvement (it was generated by AI)." Splitting the technology axis
from a behavior/trustworthiness axis is consistent with general FHIR
terminology practice (one value set = one orthogonal concept) and is
the only way to make the codes mutually exclusive — which is the
underlying complaint. Deferring on the deterministic/probabilistic axis
(per John Moehrke's pushback) is addressed by putting it on a *separate*
axis rather than baking it into device type, which avoids forcing a
hard binary on a blurring distinction while still letting implementers
record it when they can.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `AIdeviceTypeVS` as a single value set but restructure it as a
**hierarchical CodeSystem** (or adopt an external one such as a curated
subset from a data-science ontology) where the generalization
relationships (LLM `is-a` deep-learning model `is-a` ML model `is-a`
AI) are explicit. Bind `Device.type` `extensible` to the leaf concepts
and let implementers pick the most specific code they can. Add the
missing functional codes (summarization, Q&A) requested in the
comments. Do **not** add a deterministic/probabilistic axis in this
ballot cycle; instead, document the open question and defer it to the
next cycle pending more concrete use cases — directly addressing John
Moehrke's concern that the distinction is itself becoming blurry.

#### Justification

This preserves the WG's original intent that `DeviceType` describes
"the kind of AI" while resolving the practical problem (overlap) by
making the overlap *structural* and machine-readable rather than
ambiguous. Hierarchical bindings are a well-understood FHIR pattern and
require less downstream re-tooling than splitting into two value sets.
It also ducks the most contested part of the request (the
deterministic/probabilistic split) without rejecting it outright,
keeping the change set tractable for a publication-cycle revision. The
risk is that implementers still have to decide *how* specific to be,
which only partially solves the reporter's complaint.

---

### Disposition C: Decline (with note)

#### Proposal

Decline the broad request to revise the entire value set in this ballot
cycle, on the grounds that (a) the reporter has not supplied a
concrete proposed code list — the WG explicitly asked for one and none
has been provided, and (b) the deterministic-vs-probabilistic axis is
not yet well-defined enough to standardize. Accept only the narrow
sub-request from the comment: add codes for `summarization` and
`question-answering` to `AIdeviceTypeVS` (a non-breaking addition).
Carry the larger restructuring forward as a tracker for the next
publication cycle, with an action item to engage data-science SMEs.

#### Justification

The triage comment from the WG lead ("please recommend concrete
changes") makes clear that without a proposed taxonomy this ticket
cannot be acted on in its current form. Replacing a value set during
ballot reconciliation without a vetted alternative risks introducing a
worse classification than the one being replaced. The narrow additive
fix unblocks the rapidly-implemented summarization/Q&A use cases
called out in the comments without committing the WG to a contested
ontology. The trustworthiness axis can be reconsidered when there is a
concrete proposal and use cases.

---

### Recommendation

**Recommended disposition:** B

The ticket has clear merit — overlap among AI/ML/DL/LLM codes is real
and the WG already agreed the value set needs work — so an outright
decline is hard to defend even though the reporter has not produced a
draft taxonomy. Disposition A (full split into two axes) is the most
correct end-state but is large for a reconciliation cycle and runs into
John Moehrke's well-founded concern that deterministic-vs-probabilistic
is not crisp enough to standardize today. Disposition B threads the
needle: it makes the existing axis usable by giving it explicit
hierarchy (resolving the "an LLM is also ML" complaint), folds in the
concrete additive ask for summarization and Q&A codes, and explicitly
defers the trustworthiness axis with a documented open question rather
than declining it. That keeps the change scoped to something a data-
science SME can review during reconciliation while leaving a clean
path to the fuller two-axis model in a later cycle.
