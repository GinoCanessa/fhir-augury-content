## Summary

The reporter recommends that the AI Transparency IG change how the AI
"model card" is referenced in instances. Today the model card is carried as a
`DocumentReference` linked from `Provenance` at the instance level. The
reporter argues the model card is conceptually 1:1 with an algorithm + version,
which is what `Device` represents, so the model card `DocumentReference` should
be referenced from `Device` rather than from each `Provenance` instance — avoiding
redundant per-instance linkage. In comments, John Moehrke clarifies that the IG
actually uses model cards in three distinct ways: (1) to define the `Device`
(algorithm/version) itself, (2) to carry regularly used prompts, and (3) to
carry customized prompts; only use case (1) is what the reporter is targeting.
The reporter then asks for clarification on whether use cases (2) and (3) are
system or user prompts and whether they really belong in a "model card" at all,
suggesting that system prompts may also be model/agent-level (and therefore
better referenced from `Device`) while user prompts may not fit the model-card
concept. No cross-referenced Jira change-requests were surfaced; the only
external touch point is the AI Transparency on FHIR Zulip ballot-triage thread,
where this ticket is filed under "Profile & Model Structure".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the model-card `DocumentReference` reference off `Provenance` and put it
on `Device` for the use case of describing the algorithm/version itself
(Moehrke's use case 1). Concretely, in the AI Transparency IG: drop (or
deprecate) the model-card slice on the `Provenance`-side reference, and add a
slice/extension on the AI-Transparency `Device` profile that references the
model-card `DocumentReference` (e.g., `Device.extension:modelCard` →
`Reference(DocumentReference)` constrained to the existing model-card
`DocumentReference` profile). Update the IG narrative, examples, and any
associated value sets so that consumers walk
`Provenance.agent.who → Device → modelCard → DocumentReference` rather than
linking the model card from each Provenance instance. Use cases (2) and (3)
(regularly-used and customized prompts) are explicitly *not* reframed under
this disposition — they remain wherever the IG currently carries them.

#### Justification

A model card describes an algorithm + version, which is exactly the concept
`Device` (with `Device.definition` / `DeviceDefinition`) represents in FHIR.
Linking the model card from `Provenance` forces every instance to repeat a
reference to information that is invariant per algorithm version, which is
redundant and creates drift risk if different `Provenance` instances point at
different model-card revisions for the same `Device`. Anchoring the model card
on `Device` also matches how implementers already discover algorithm metadata
(via the device the provenance points to) and is consistent with the IG's
own framing in Moehrke's comment: "use of model-card to define the Device".
Keeping `DocumentReference` as the carrier preserves the metadata wrapper that
Moehrke called out as important for supporting multiple model-card formats
(XML, markdown, future standards).

---

### Disposition B: Alternative Approach

#### Proposal

Allow the model-card `DocumentReference` to be referenced from **both**
`Device` (for the algorithm/version-level model card — use case 1) **and**
`Provenance` (for use cases 2 and 3, where the prompt content is
instance-specific). Make the `Device`-side reference the SHALL/preferred
location for use case 1, and constrain the `Provenance`-side reference so that
it is only populated when the carried artifact is a per-invocation prompt
(system or user) rather than the algorithm description. Add IG narrative that
explicitly distinguishes the three uses Moehrke enumerated, and split the
model-card `DocumentReference` profile (or add a `type`/`category` slice) so
consumers can tell at a glance whether a given model-card document is
"algorithm definition", "system prompt", or "user prompt".

#### Justification

This addresses the reporter's underlying concern (don't repeat invariant
algorithm metadata on every instance) while preserving the IG's current
ability to carry per-invocation prompt content where it actually belongs —
on the `Provenance` of the specific inference event. It also responds directly
to the reporter's follow-up question about whether prompts are really "model
cards" by forcing the IG to label the artifact's role explicitly. Compared
to Disposition A, this avoids breaking the existing prompt-carrying use cases
that currently rely on the `Provenance`-side reference, and it surfaces the
ambiguity the reporter highlighted (system vs. user prompt) as a normative
distinction in the profile rather than leaving it implicit.

---

### Disposition C: Decline

#### Proposal

Decline the request to move the model-card reference from `Provenance` to
`Device`. Keep the current model where the model-card `DocumentReference` is
referenced from `Provenance`, and address the reporter's concern with IG
guidance only — explicitly documenting that for the algorithm-definition use
case implementers MAY share/reuse the same `DocumentReference` across many
`Provenance` instances and SHOULD treat it as immutable per
`Device`/algorithm version.

#### Justification

The current structure already supports the reporter's efficiency goal (a single
`DocumentReference` instance can be referenced from any number of `Provenance`
records) without any profile change, and `Provenance` is the resource that
already carries the audit/inference trail in this IG. Moving the link to
`Device` would be a breaking change for early implementers, would split
model-card linkage across two resources (since use cases 2 and 3 must remain
on `Provenance`), and would force consumers to traverse an additional hop
(`Provenance → Device → DocumentReference`) for a piece of metadata they
already get directly from `Provenance`. A narrative clarification is the
lowest-risk way to resolve the confusion Moehrke identified.

---

### Recommendation

**Recommended disposition:** B

The reporter's diagnosis is correct for use case (1): the algorithm-level
model card is genuinely a property of the `Device`, and forcing it onto every
`Provenance` is redundant. But Moehrke's clarification that the IG also uses
model cards for system and customized prompts means a wholesale move
(Disposition A) would either break or quietly drop those use cases, and the
reporter's own follow-up shows the prompt-vs-algorithm distinction is exactly
what's currently muddled. Disposition C preserves backward compatibility but
leaves the structural ambiguity intact and pushes implementers to discover the
right modeling by reading prose. Disposition B addresses both halves of the
problem: the algorithm-level model card moves to `Device` (resolving the
redundancy the reporter raised), per-invocation prompts stay on `Provenance`
(preserving the existing prompt use cases), and the IG is forced to label the
artifact's role so the three uses Moehrke enumerated stop colliding under a
single name. That makes B the strongest starting position for the workgroup
discussion.
