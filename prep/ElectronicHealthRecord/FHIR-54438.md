## Summary

In the AI Transparency on FHIR IG (January 2026 ballot), the "Data Sources"
section defines **Operations** as "the technical protocols used for AI
interactions" and lists **MCP** and **A2A** as examples. The reporter
(Dmitry Maslennikov) argues that this definition is both incomplete (it omits
other current/emerging protocols such as ACP, ANP, AGP, AG-UI) and misleading,
because the term "Operations" suggests something broader than transport
protocols (e.g., workflow steps such as PII masking, guardrails,
retrieval/tool use, post-processing).

The ticket asks the workgroup to clarify the intended scope of **Operations**:

- If the intent is *interaction protocols only*, the IG should explain why
  protocol metadata matters for AI Transparency (since e.g. REST vs. MCP
  typically does not change inputs or results), mark it as optional/minor
  audit-trail metadata, and consider renaming the concept to
  **Protocols / Interaction Protocols** to avoid confusion.
- If the intent is *broader* than protocols, the IG should explicitly list the
  intended categories (protocols **plus** workflow/processing actions) and
  clarify that any named protocols are examples only, with room for future
  protocols.

This ticket was triaged together with the rest of the AI Transparency IG
January-ballot batch (see the Zulip "IG - Jan ballot" thread, where it is
filed under the *Profile & Model Structure* category alongside related items
such as FHIR-54527, FHIR-54421, FHIR-54371, FHIR-54267, FHIR-54264,
FHIR-54259, FHIR-54241). It has no other Jira cross-references in the
Augury index.

## Details

**Description:**

In section "Data Sources", the guide defines **Operations** as "the technical
protocols used for AI interactions" and gives **MCP** and **A2A** as examples.
This looks like an incomplete and potentially misleading list (why MCP/A2A,
but not other current or future protocols such as ACP, ANP, AGP, AG-UI, etc.).
This is also confusing because it makes **Operations** sound like "protocols",
even though the term itself suggests something broader.

Please clarify what **Operations** means:

- Is it only **interaction protocols**?
- Or does it also include **workflow steps** such as PII masking, guardrails,
  retrieval/tool-use, post-processing, and other processing actions?

If Operations is meant to cover **protocols only**, it would help to explain
**why protocol information matters for AI Transparency**. In many cases, using
different protocols (e.g., REST vs MCP) does not change the input data or the
result, so this may be **optional, minor metadata**. If you keep it, please
clarify the intended use (traceability/audit) and consider naming it
**Protocols / Interaction Protocols** to avoid confusion.

If Operations is **broader than protocols**, please update the text to clearly
list the intended categories (protocols + workflow/processing actions) and
add a note that protocol names are **examples only** and other protocols may
exist now or in the future.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the AI Transparency on FHIR IG "Data Sources" page to:

1. Rename the **Operations** concept (in the narrative and, where it appears,
   in element labels/short descriptions) to **Interaction Protocols** (or
   **Protocols**).
2. Rewrite the definition to: "Identifier(s) for the interaction protocol(s)
   used between the AI consumer and the AI service (e.g., REST, MCP, A2A).
   This is informational/audit metadata only; the protocol does not normally
   change the inputs or outputs of the interaction."
3. Replace the closed-looking "MCP and A2A" example list with an explicit
   "examples only, not exhaustive" note that mentions ACP, ANP, AGP, and
   AG-UI alongside MCP and A2A, and points implementers to a value set
   (extensible binding) rather than a fixed list.
4. Add a short paragraph under Data Sources explaining *why* protocol
   metadata matters for AI Transparency (traceability/audit, reproducibility
   of an interaction, and detection of protocol-specific failure modes), and
   mark the element as optional (0..*) if it is not already.

#### Justification

This is purely a documentation/profile-labelling fix in a draft IG that is
explicitly inviting ballot feedback. Renaming "Operations" to
"Interaction Protocols" matches the semantics actually documented and
removes the collision with the much broader colloquial meaning of
"operation" (which in FHIR also overlaps with `OperationDefinition`).
Treating the example list as extensible is consistent with how FHIR
typically handles fast-moving terminology (extensible value sets,
example-only lists), and explicitly stating the audit/traceability rationale
addresses the reporter's "why does this matter" question without changing
the underlying data model.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the **Operations** label, but split the concept into two distinct,
clearly named sub-elements (or two coded slices on a single element) on the
relevant Data Sources profile:

- **`interactionProtocol`** — coded, extensible binding (REST, MCP, A2A,
  ACP, ANP, AGP, AG-UI, …); informational/audit metadata.
- **`processingStep`** — coded, extensible binding for workflow operations
  the AI service performed on or around the interaction (PII masking,
  guardrails, retrieval / tool use, post-processing, etc.).

Update the narrative to define **Operations** as the umbrella term covering
both, and add the "examples only, future protocols allowed" note that
Disposition A also calls for. Provide a worked example that uses both
sub-elements together.

#### Justification

The reporter raises a legitimate ambiguity that a pure rename does not
fully resolve: a number of stakeholders genuinely want to convey workflow
steps (guardrails, PII masking, retrieval, post-processing) in the same
place, because those are exactly the kinds of facts that AI Transparency
exists to surface. Splitting the concept rather than narrowing it preserves
the broader scope while removing the "is it protocols or workflow?"
ambiguity, and using extensible bindings keeps the IG forward-compatible
with new protocols and processing patterns. The trade-off vs. Disposition
A is a small structural change (one element becomes two, or one element
gains a slice discriminator), which is acceptable while the IG is still in
ballot.

---

### Disposition C: Decline

#### Proposal

Mark as **Not Persuasive — already covered**. Retain the current
"Operations" element and definition, optionally adding one sentence to the
narrative clarifying that the listed protocols are examples and that the
binding is extensible.

#### Justification

If the workgroup's design intent really is "interaction protocol only" and
the binding is already extensible, then the substantive issue is just the
example wording. A one-line "examples only" clarification would address
that without renaming the element (which has downstream cost: profile
changes, re-binding, example updates, and any implementer code that already
hard-codes the name). Workflow steps (PII masking, guardrails, etc.) are
arguably out of scope for a "Data Sources" section and may belong in a
separate Provenance/Processing area — several other January-ballot tickets
(e.g., FHIR-54707, FHIR-54281, FHIR-54244 in the Provenance & Tagging
bucket) are already addressing that space.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers the reporter's two questions ("what does
Operations mean?" and "why does it matter?") with the smallest, lowest-risk
change to a ballot-stage IG: a rename to a self-describing term, an explicit
audit/traceability rationale, and an extensible "examples only" treatment of
the protocol list. Disposition B is technically attractive but materially
expands scope by formally pulling workflow/processing semantics into Data
Sources, which other January-ballot tickets in the *Provenance & Tagging*
and *Data Sources & Context* buckets are already grappling with — that work
is better done as a coordinated cross-ticket design pass than as a
unilateral structural change driven by this single comment. Disposition C
under-delivers: the reporter is explicitly objecting to the name
"Operations," so an example-only fix without addressing the label would
likely be re-raised. Accepting A also leaves the door open for the workgroup
to layer in B-style structural changes later if the broader Provenance /
Processing-step discussion converges.
