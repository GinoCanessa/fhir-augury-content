## Summary

FHIR-53967 is an omnibus clarification ticket against the AI Transparency on
FHIR IG's `general_guidance.html` page. The reporter (Abderrazek Boufahja)
walks through several sections and asks for editorial improvements: explain
what each row of the "Observability Factors for iterative IG development"
table means; sharpen the "AI Usage Representation" wording so it states that
the use case is to distinguish where AI was used "to generate or influence";
harmonize the table column "Context" with the section heading "Data Sources";
cite any external sources used to derive the observability factors; rephrase
the "Tagging" examples (notably "Use when an example is completely authored
by an AI" and the AIAST/clinician-involvement paragraph that reads more like
a working-group note than guidance) and tag the "components of an example"
sentence as a Tip for consistency; make the conformance expectation explicit
that whenever the `inline-sec-label` extension is used, `meta.security` SHALL
include the DS4P PROCESSINLINE label; turn the Hugging Face fixtures URL
into a proper external clickable link; and improve the
"PDF interpreted by AI into FHIR" example, which currently targets a
transaction Bundle that is neither tagged AIAST nor able to apply the
AI Data profile (Bundle does not extend DomainResource). The ticket has
already been resolved as **Persuasive with Modification**: most points were
addressed in [HL7/aitransparency-ig#22](https://github.com/HL7/aitransparency-ig/pull/22),
with three follow-on commitments (replace "influenced/involved" with
"produced or manipulated" everywhere, reword the §2.2 "Use examples", and
make the Hugging Face URL clickable), and the PDF/Bundle example deferred
to [FHIR-56340](https://jira.hl7.org/browse/FHIR-56340). Cross-references
include the implementing PR `HL7/aitransparency-ig#22` and the AI
Transparency Zulip ballot-triage thread that scopes this ticket under
"Data Sources & Context".

## Details

**Description:**

The reporter raises eight separate clarification points against
`general_guidance.html` of the AI Transparency IG (2026Jan render):

1. *Observability Factors for iterative IG development* — the four-row
   table is opaque; the columns need an explanation of what each row
   differentiates.
2. *AI Usage Representation* — the sentence "In this use-case need, there
   is simply a need to distinguishes where AI was used and where it was
   not used." is unclear. Suggested rewrite: "…where AI was used **to
   generate or influence** and where it was not used."
3. *AI Usage Representation table* — column 3 is labelled "Context" but
   the corresponding section 2.1.3 is titled "Data Sources". Harmonize to
   "Data Sources".
4. *Observability Factors* — if external references were used to build
   the list, cite them.
5. *Tagging* — "Use when an example is completely authored by an AI."
   reads awkwardly; suggested rewrite: "An example of use in a completely
   authored FHIR resource by an AI."
6. *Tagging — AIAST paragraph* — the paragraph beginning "Discussion has
   indicated that a few more codes might be useful…" reads like a WG
   discussion artefact. Suggested rephrase: "AIAST code does not provide
   contextual indications, like for example if a clinician was involved
   in the use of the AI, or reviewed the output of the AI. This IG
   defined a new code system, [Added Provenance Codes](https://hl7.org/fhir/uv/aitransparency/2026Jan/CodeSystem-AddedProvenanceCS.html),
   with codes providing better contextual tagging."
7. *Tagging* — "Use when components of an example were authored by AI,
   but not the whole Resource." reads like a Tip but is not tagged as
   such — make it consistent with the surrounding markup.
8. *Tagging* — the sentence describing the relationship between
   `meta.security` (DS4P PROCESSINLINE) and the `inline-sec-label`
   extension is ambiguous. Make it explicit: when `inline-sec-label` is
   used, `meta.security` **SHALL** carry PROCESSINLINE.
9. *Hugging Face Markdown* — the bare URL
   `https://github.com/huggingface/huggingface_hub/tree/main/tests/fixtures/cards`
   should be turned into a clickable link.
10. *PDF interpreted by AI into FHIR* — the example targets a transaction
    Bundle, but neither the Bundle nor the contained resources are tagged
    AIAST, and Bundle does not extend DomainResource so the AI Data
    profile cannot apply. The example needs to be reworked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Confirm the existing **Persuasive with Modification** resolution and land
the three named follow-ups on top of merged PR `HL7/aitransparency-ig#22`,
all in `input/pagecontent/general_guidance.md`:

1. Global text replacement of `influenced` and `involved` with `produced
   or manipulated` across `general_guidance.md` (and a sweep of
   `index.md`, `use_cases.md`, `explainer.md` for the same pair).
2. Rewrite the "Use examples" subsection currently under the "Tagging"
   guidance (the §2.2 the resolution refers to in render numbering),
   normalising the three "Use when …" sentences to the same voice and
   tagging the "components of an example" sentence as a `{:.tip}` (or
   the IG template's equivalent Tip block) so it matches the other Tip
   in the same section.
3. Convert the bare Hugging Face fixtures URL in the "Hugging Face
   Markdown" subsection to an inline Markdown link
   (`[Hugging Face card fixtures](https://github.com/huggingface/huggingface_hub/tree/main/tests/fixtures/cards)`).

In addition, fold in the editorial points already covered by PR #22:
the "to generate or influence" sharpening of the AI Usage Representation
sentence, harmonising the "Context" column header to "Data Sources" in
the AI Usage Representation table, an explanatory lead-in for the
Observability Factors table (one sentence stating what each row
differentiates), the AIAST-paragraph rephrase, and the explicit
conformance statement that `meta.security` SHALL include DS4P
`PROCESSINLINE` whenever `inline-sec-label` is used. Defer the PDF/Bundle
example overhaul to [FHIR-56340](https://jira.hl7.org/browse/FHIR-56340)
as already agreed. After the edits, re-run `_genonce.sh`, sanity-check
that no `general_guidance.html#…` anchors elsewhere in the IG broke, and
update `input/ignoreWarnings.txt` only if intentionally-renamed anchors
trigger publisher warnings.

#### Justification

The ticket is a bundle of editorial clarifications, not a technical
correction (per Moehrke's own re-classification comment). PR #22 is
already merged and the WG has explicitly committed to the three remaining
refinements; this disposition simply executes that commitment cleanly in
the one file that owns all of the cited sections. The conformance
upgrade in point 8 (a `SHALL` tying `inline-sec-label` to PROCESSINLINE)
matches how DS4P security labels are described elsewhere in the IG, and
keeping it in narrative (rather than promoting to a profile invariant)
keeps the diff inside `general_guidance.md` and avoids re-balloting the
Provenance profile mid-cycle. Deferring the PDF/Bundle rework to
FHIR-56340 is appropriate because that point is genuinely about
example/profile design, not wording — exactly the boundary the resolver
already chose.

### Disposition B: Alternative Approach

#### Proposal

Land the editorial follow-ups from Disposition A, but **strengthen point
8 from narrative to a profile-level invariant**: add a SUSHI invariant in
`input/fsh/SD_Provenance.fsh` (or the appropriate AI-Data/AIAST-tagged
resource profile) of the form

```
* obeys ai-inline-sec-label-implies-processinline
Invariant: ai-inline-sec-label-implies-processinline
Description: "If any element carries the inline-sec-label extension indicating AI influence, meta.security SHALL include the DS4P PROCESSINLINE security label."
Severity: #error
Expression: "extension('http://hl7.org/fhir/uv/security-label-ds4p/StructureDefinition/inline-sec-label').exists().not() or %resource.meta.security.where(system='http://hl7.org/fhir/v3/ActCode' and code='PROCESSINLINE').exists()"
```

(exact path to be tightened to the IG's actual `inline-sec-label`
canonical and the correct ActCode coding-system URL once verified against
the DS4P dependency). Mirror the rule in narrative in
`general_guidance.md`. Keep the rest of Disposition A — the
"produced or manipulated" rename, Tip-block tagging, Use-examples
rewrite, Hugging Face link, AI Usage Representation sharpening, column
rename, Observability Factors lead-in, AIAST paragraph rephrase — exactly
as in A, and continue to defer the PDF/Bundle example to FHIR-56340.

#### Justification

Point 8 is the only sub-point in FHIR-53967 that is genuinely a
conformance question, not an editorial one: the reporter explicitly says
"It is not clear if this is a requirement, or a description of a usage. I
think the IG should be explicit: when inline-sec-label extension is used,
meta.security SHALL have PROCESSINLINE." A `SHALL` written only in prose
in `general_guidance.md` is unenforceable by validators and tends to be
missed by implementers; expressing it as a FHIRPath invariant on the
Provenance / AI Data profile makes it testable and machine-checkable
against the IG's existing examples. The DS4P dependency is already in
`sushi-config.yaml`, so no new cross-repo wiring is needed. Trade-off:
this is a behaviour change on the profile, which means any existing
example that uses `inline-sec-label` without a matching PROCESSINLINE
security label will start failing validation and need a fix-up — a
slightly larger blast radius than Disposition A, but correctly so for a
SHALL-level claim.

---

### Disposition C: Decline (the SHALL upgrade) and ship only the named editorial follow-ups

#### Proposal

Adopt the editorial portion of the resolution exactly as written in PR
#22 and the three follow-ups (produced/manipulated rename, Use-examples
rewrite, clickable Hugging Face URL). For point 8, **keep the existing
descriptive prose** rather than introducing new normative `SHALL`
language — explain in the narrative that PROCESSINLINE is the security
label DS4P uses to flag inline-tagged content and reference the DS4P
PROCESSINLINE definition, but do not make it conformance-binding in this
ballot cycle. Continue to defer the PDF/Bundle example to FHIR-56340.

#### Justification

This IG is at FMM 2 / `trial-use` / 1.0.0-ballot; the
`inline-sec-label` mechanism comes from the DS4P IG, not from this one.
Asserting a normative `SHALL` here that constrains how the DS4P extension
must co-occur with `meta.security` is, arguably, asserting conformance on
behalf of an upstream IG. Until DS4P itself either makes that pairing
mandatory or this IG ballots a profile that owns both elements, leaving
the relationship as descriptive narrative is the safer path — and the
existing PR #22 plus the three named follow-ups are sufficient to close
the reporter's other nine points. Down-side: implementers can interpret
the relationship loosely, which is exactly what the reporter flagged as a
problem; this disposition is the most conservative and the least
responsive.

---

### Recommendation

**Recommended disposition:** A

The WG has already balloted, voted, and merged PR #22 with a "Persuasive
with Modification" resolution and explicitly committed to three
remaining editorial follow-ups. Disposition A simply executes that
commitment in `input/pagecontent/general_guidance.md` and confirms the
deferral of the PDF/Bundle example to FHIR-56340 — it has the smallest
blast radius, no profile churn, and matches what the resolver already
told the reporter would happen. Disposition B is technically the more
correct answer for point 8 (a `SHALL` belongs in an invariant, not in
prose) and is worth tabling explicitly so the WG can record whether they
want to upgrade the conformance claim now or in the next ballot cycle;
if the WG does want it, doing so as part of A is mechanically cheap
because the DS4P dependency is already wired in. Disposition C is listed
mainly to make the upstream-conformance concern visible, but it walks
back exactly the clarification the reporter most wanted, so it should be
adopted only if the WG concludes that the SHALL belongs in DS4P, not
here. Net: ship A; if there is appetite in the room, fold in B's
invariant addition; otherwise leave it as narrative and note that point
8 may be revisited.
