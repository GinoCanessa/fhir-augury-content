## Summary

The reporter (Cooper Thompson) observes that the AI Transparency (FHIR) IG
models AI-created data well but provides little guidance on how that
information is actually exchanged over the RESTful API. Specifically, the
ticket asks the IG to add exchange guidance for the case where a client
POSTs a clinical resource via the RESTful `create` interaction and the
server must simultaneously know the AI provenance of that resource â€” i.e.,
the clinical resource and its `Provenance` need to be processed atomically.

Candidate approaches enumerated in the ticket comments include: (1) submit
a Bundle (or Parameters) containing both the clinical resource and the
Provenance; (2) embed the Provenance as a `contained` resource inside the
clinical resource; (3) use the `X-Provenance` HTTP header on the create;
or (4) perform two separate transactions (create the content first, then
create the Provenance), possibly with a "held until Provenance arrives"
protocol on the server. The reporter notes that option 4 is impractical
for most real-world servers and option 1 is incompatible with a single
`create` interaction, so the realistic standardization choices are
option 2 (contained) or option 3 (X-Provenance header). The reporter
leans toward contained, acknowledging it slightly bends the formal
meaning of `contained`.

No other Jira tickets are directly cross-referenced. The only surfaced
Zulip thread is a high-level AI Transparency WG triage index for the
January 2026 ballot, which groups this ticket in the "Exchange &
Implementation" category alongside FHIR-54351 and FHIR-54180 but does
not contain substantive discussion of this specific request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new "Exchange" page (or a top-level section on an existing
implementation page) to the AI Transparency (FHIR) IG that describes how
to exchange AI `Provenance` alongside the clinical resource it describes,
and **standardize on option 2: Provenance as a `contained` resource
inside the clinical resource** for the RESTful `create` interaction.
Specifically, the IG would:

1. State that on a RESTful `create`, clients that need atomic submission
   of AI provenance SHALL include the `Provenance` as a `contained`
   resource on the target clinical resource, with the containing
   resource's `meta.tag` (or existing AI-marker tag defined by this IG)
   set to indicate AI involvement.
2. Require that the `contained` Provenance reference the containing
   resource via the usual internal-fragment reference pattern
   (`Provenance.target = "#"` / the container's `id`), and reiterate the
   existing FHIR rule that contained resources inherit the container's
   identity.
3. Provide conformance language for servers: a conformant AI-aware
   server SHALL, on create, extract the `contained` Provenance, persist
   it as an independent resource with its own logical id, rewrite
   `Provenance.target` to reference the newly-assigned clinical
   resource id, and surface it through normal Provenance search
   (`_history`, `Provenance?target=...`).
4. Explicitly address alternative approaches in a non-normative
   subsection: note that Bundle/transaction (option 1) is the preferred
   pattern when multiple resources are being created at once; document
   that option 3 (`X-Provenance` header) is allowed but not required;
   and rule out option 4 (two independent creates) as non-atomic and
   therefore not recommended by this IG.
5. Provide at least one worked example (JSON) of an AI-authored
   `Observation` or `DocumentReference` carrying a contained
   AI-`Provenance`, plus a companion example of the same exchange via
   a Bundle for comparison.

#### Justification

The reporter and the sole commenter converge on "contained or
X-Provenance" as the only two options realistic for a `create`
interaction, and the reporter's stated preference is `contained`. The
community's objection to `contained` â€” loss of searchability â€” is
addressed by the explicit server-side conformance requirement that
contained Provenance be persisted as an independent, searchable
resource. Standardizing on a single mechanism (per Cooper's "pick one"
framing) reduces implementation divergence, which is the IG's purpose.
The approach also mirrors an increasingly common practice already
observed in other FHIR submissions where related metadata rides with a
`create` via `contained`.

---

### Disposition B: Alternative Approach

#### Proposal

Add the exchange section, but standardize on the **`X-Provenance` HTTP
header** (option 3) as the normative mechanism for a RESTful `create`
that needs atomic AI provenance, and document `contained`, Bundle, and
`transaction` only as non-normative alternatives. Specifically:

1. Point to the existing FHIR core definition of `X-Provenance` (the
   mechanism is already part of the base REST API) and require that on
   a `create` of an AI-authored resource, clients that wish to convey
   AI provenance atomically SHOULD send `X-Provenance` with a
   serialized `Provenance` resource whose `target` will be bound by the
   server to the newly-created resource.
2. Require servers to process the header atomically with the create
   and to fail the create (4xx) if the `Provenance` is malformed or
   inconsistent with the IG's AI profiles.
3. Make `Bundle`/`transaction` (option 1) the recommended mechanism
   whenever more than one clinical resource is being created at once,
   or when the client prefers a wire format that self-describes both
   resources.
4. Document `contained` as permitted but discouraged, explicitly
   because it alters what the clinical resource looks like on the wire
   and creates an implementation asymmetry between "create" and
   "update" (where `contained` would not normally appear).
5. Rule out option 4 (two separate, non-atomic transactions) as not
   recommended.

#### Justification

The `X-Provenance` header is the mechanism the FHIR core REST API
already defines for precisely this problem: attaching provenance to a
single RESTful write. Standardizing on it aligns the AI Transparency
IG with base-FHIR idiom rather than reinterpreting `contained` (which,
as Cooper acknowledges, "bends the meaning and intent"). Trade-offs vs.
Disposition A: (i) no need to persuade servers that contained resources
must be re-projected into searchable Provenance records; (ii) the
clinical resource on the wire is unchanged, which is friendlier to
generic FHIR clients and profiles that don't anticipate `contained`;
(iii) the cost is that `X-Provenance` is sometimes stripped by
intermediaries and some client libraries surface custom headers
awkwardly, so adoption friction is a real risk. Those risks are the
main reason Disposition A may still be preferable in practice.

---

### Disposition C: Decline

#### Proposal

Do not add normative exchange guidance for the RESTful `create`
interaction in this ballot cycle. Close the ticket as "Persuasive with
Mod" only to the extent of adding a short non-normative note pointing
implementers to (a) base-FHIR `transaction`/Bundle for multi-resource
atomic writes, (b) the base-FHIR `X-Provenance` header, and (c) the
existing two-step "create, then create Provenance" pattern, with the
explicit statement that the IG takes no position on which to prefer
for AI provenance in the current release.

#### Justification

The base FHIR specification already defines multiple adequate
mechanisms (Bundle/transaction, `X-Provenance`, sequential creates)
for attaching Provenance to a create. Picking one and making it
normative in this IG risks (i) conflicting with future base-FHIR
direction, (ii) excluding deployed implementations that already use a
different mechanism, and (iii) committing to behavior â€” e.g., the
"extract `contained` Provenance, rewrite `target`, persist as
independent resource" requirement in Disposition A â€” that goes beyond
what a typical FHIR server does today. With only two participants on
the ticket so far and no broader ballot feedback surfaced in Zulip, the
body of community input may be too thin to justify a normative choice
this cycle; the WG can revisit once more implementer feedback arrives.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly responds to the ticket: it adds the exchange
section the reporter asked for, picks one option (per Cooper's
"pick one" framing), and picks the option Cooper explicitly prefers.
The main technical objection to `contained` â€” searchability â€” is
answered by making server-side re-projection a conformance requirement,
which both aligns with what "real-world systems" (per Cooper's
comment) already do and gives implementers a crisp, testable rule.
Disposition B is the credible runner-up and should be seriously
considered if the WG weighs "stay close to base-FHIR idiom" more
heavily than "keep the clinical resource on the wire simple"; the
choice between A and B is genuinely close and the WG's judgment on
header-stripping by intermediaries should drive it. Disposition C is
the weakest of the three: the reporter's central complaint â€” that the
IG provides no exchange guidance â€” is legitimate and is not remedied by
a non-committal pointer to three different base-FHIR mechanisms, which
is exactly the "full menu" Cooper warned leads to divergence.
