## Summary

The ticket, filed against the January 2026 ballot of the AI Transparency
on FHIR IG, raises an ambiguity in the `AIDevice` profile: it is not
clear whether `AIDevice` is meant to represent an AI *algorithm/model*
or a *system that embeds one or more AI algorithms*. The reporter
suggests that, since real-world deployed systems often compose multiple
AI models, `AIDevice` should represent AI-enabled *systems*, while the
`ModelCard` profile should carry the per-model metadata. The reporter
further points out that many `Device` attributes (`manufacturer`,
`version`, etc.) are also present on the Model Card, and if `AIDevice`
is ultimately meant to represent an algorithm itself, the overlap
between the two profiles needs to be reconciled to prevent conflicting
or redundant information about the same AI artefact. The ticket has
been triaged "Persuasive with Modification" with the resolution note
that the specification text around this scoping needs to be clarified.
There are no cross-referenced Jira tickets; the ticket appears in a
single Zulip ballot-triage table for the AI Transparency IG under the
"Profile & Model Structure" category.

## Details

**Description:**

> Is not entirely clear if the AI Device profile is meant to represent
> an AI algorithm or a system that utilizes an AI algorithm. As many
> systems that are being developed are able to utilize an array of AI
> models, our suggestion is to represent AI-based systems, and the
> Model Card profile represents the AI model/s behind those systems.
>
> On the other hand, many of the attributes of the Device
> (manufacturer, version, etc) resource are also present in the model
> card. So, if the intention is to represent AI algorithms as Device,
> there should be a tight relation between the attributes of those two
> objects to mitigate inconsistencies.

**Resolution Description:** agreed this needs to be made more clear.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Redefine the `AIDevice` profile (AI Transparency IG) so that it
explicitly represents an **AI-enabled system** (the deployed
product/component that a clinician or workflow interacts with), and
delegate per-model metadata to the `ModelCard` profile. Concretely:

- Update the `AIDevice` profile's short description and narrative
  introduction to state that it profiles `Device` to describe the
  *system* that consumes one or more AI models, not the models
  themselves.
- Constrain or repurpose `AIDevice.definition` / a model-linking
  element (e.g., a typed extension
  `aidevice-model` pointing at the `ModelCard`-profiled
  `DeviceDefinition`/`Device` resources) so one `AIDevice` can
  reference *N* `ModelCard` instances.
- On `ModelCard`, scope elements like `manufacturer` / `version` to
  the **model** (training organisation, model version string) and
  add narrative clearly distinguishing them from `AIDevice`'s
  system-level `manufacturer` / `version`.
- Add a short "AIDevice vs. ModelCard" section to the IG narrative
  explaining the system-vs-model split and pointing implementers at
  which profile to use for each attribute.

#### Justification

This matches the reporter's stated real-world need — modern AI
offerings bundle multiple models behind a single product — and it
aligns with how `Device` is already used elsewhere in FHIR (physical
or logical deployed artefact) while keeping model-training metadata
on a purpose-built `ModelCard` profile. It also directly addresses
the "tight relation … to mitigate inconsistencies" concern by making
each overlapping attribute live on exactly one profile with
unambiguous semantics. The resolution ("agreed this needs to be made
more clear") already signals that the workgroup accepts the
underlying problem; this disposition is the most explicit resolution.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `AIDevice` as a single profile that **may** represent either an
AI algorithm *or* a system hosting algorithms, but disambiguate by:

- Adding a required `AIDevice.type` or `AIDevice.category` binding
  (new value set with at least `ai-model` and `ai-enabled-system`)
  so every instance declares which it is.
- Clarifying in the profile narrative that when
  `category = ai-enabled-system`, the instance SHALL reference one
  or more `ModelCard`-profiled resources via an existing/new
  reference element; when `category = ai-model`, the `ModelCard`
  carries the authoritative training/model metadata and the
  `AIDevice` instance MAY be omitted or be a minimal stub.
- Updating cross-references in the IG (examples, use cases,
  provenance guidance) to use the new category consistently.

#### Justification

This preserves backward-compatibility with the current ballot
snapshot — existing implementers who modelled an individual model as
`AIDevice` aren't forced to refactor — while still resolving the
ambiguity by making the scope explicit per-instance. It is a smaller
normative change than Disposition A and acknowledges that both
modelling choices exist in the field. The trade-off is that two
legitimate shapes of `AIDevice` persist, which keeps some of the
attribute-overlap pressure with `ModelCard` that the reporter flagged;
this is mitigated but not eliminated by the category-driven narrative
guidance.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that the current `AIDevice` profile
already scopes itself to AI *systems* (per existing narrative) and
that the overlap with `ModelCard` is intentional: `ModelCard`
describes the model qua artefact (training data, performance,
intended use), while `AIDevice` describes the deployed system's
identity (manufacturer, version). Close as *Not Persuasive* with a
note pointing to the existing narrative.

#### Justification

This is only defensible if the workgroup believes the current
narrative is unambiguous to implementers and that the reporter
misread it. It would keep ballot churn to a minimum. However, the
ticket's own resolution ("agreed this needs to be made more clear")
contradicts this stance, so this disposition is included for
completeness rather than as a serious contender — choosing it would
require the workgroup to walk back the triaged resolution.

---

### Recommendation

**Recommended disposition:** A

The resolution already records agreement that clarification is
needed, so Disposition C is off the table in practice. Between A and
B, Disposition A is the cleaner long-term fix: it puts a single,
well-defined semantic on each profile (`AIDevice` = deployed
system; `ModelCard` = per-model metadata) and lets the IG describe
an ensemble system as one `AIDevice` referencing many `ModelCard`s,
which mirrors how vendors actually ship AI today. Disposition B's
category-flag approach avoids a breaking change but preserves the
root-cause ambiguity — implementers can still put conflicting
`manufacturer` / `version` values on both profiles for the same
real-world artefact. Because the AI Transparency IG is still at
ballot, the cost of the cleaner A-style split is relatively low
compared to locking in dual semantics for the normative lifetime of
the IG. Recommend the workgroup adopt A and explicitly call out the
`AIDevice` → `ModelCard` reference cardinality (1..*) in the IG
narrative.
