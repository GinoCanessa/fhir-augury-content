## Summary

The AI Transparency on FHIR Implementation Guide (January 2026 ballot)
does not currently expose the IG Publisher's built-in next/previous
navigation chrome on its rendered pages. The reporter, John Moehrke, is
asking the IG authors to enable that navigation aid so that readers can
move sequentially through the IG without having to return to the table
of contents. The fix proposed in the ticket itself is concrete and
mechanical: add the IG Publisher parameter `shownav: 'true'` to the
`parameters:` block of `sushi-config.yaml`. There are no Jira cross-
references to other tickets, but the EHR work group's January-ballot
triage thread on Zulip groups this item under the "Exchange &
Implementation" bucket together with [FHIR-54528](https://jira.hl7.org/browse/FHIR-54528)
and [FHIR-54180](https://jira.hl7.org/browse/FHIR-54180), and the ticket
has been marked **Persuasive**, indicating EHR is inclined to adopt the
fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the change exactly as the reporter described. In the AI
Transparency on FHIR IG source (the IG project that publishes
`https://hl7.org/fhir/uv/aitransparency/`), edit `sushi-config.yaml` and
add the IG Publisher parameter under `parameters:`:

```yaml
parameters:
  shownav: 'true'
```

Re-run the IG build; the publisher will then render its next/previous
navigation chrome on each generated page. Verify on the rendered ballot
output that the navigation links appear and traverse the intended page
order.

#### Justification

`shownav` is a standard, supported IG Publisher parameter whose sole
effect is to enable the publisher's built-in sequential-navigation
chrome. The change is a one-line addition to `sushi-config.yaml`, has no
impact on the normative content of the IG, and addresses a real
readability gap that ballot reviewers (including the reporter) are
hitting. There is no observable downside, no dependency on other
tickets, and the EHR triage has already marked the ticket Persuasive.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than turning on the publisher's generic next/previous chrome
unconditionally, define an explicit page ordering for the IG and rely on
that ordering to drive navigation. Concretely: confirm/curate the page
sequence in the IG's menu / table of contents (e.g., the menu structure
declared in the IG resource or `sushi-config.yaml` `menu:` block, plus
any `pages` ordering), and only then enable `shownav: 'true'`. As a
follow-on, audit the resulting first/previous/next/last links on a
sample of pages to ensure the order is meaningful (background → use
cases → profiles → examples → downloads).

#### Justification

`shownav` simply surfaces whatever ordering the IG already declares; if
that ordering is incidental, enabling it can produce confusing
navigation (e.g., jumping from a use-case page to an unrelated profile).
For a ballot-stage IG it is cheap to verify the sequence before exposing
it to readers, and doing so prevents a follow-up ticket from being
opened against the navigation order itself. The trade-off vs.
Disposition A is a small amount of editorial work in exchange for
guaranteed-coherent navigation.

---

### Disposition C: Decline

#### Proposal

Decline the request on the basis that page-to-page navigation is
already available through the IG's table of contents and the
publisher-generated breadcrumbs, and that adding the `shownav` chrome
duplicates that capability without solving a normative problem with the
specification.

#### Justification

The ticket does not point at a defect in the AI Transparency content
itself; it is a publication-style preference. If the IG's TOC and menu
already let readers reach every page, the next/previous chrome is
redundant. Declining keeps the ballot output visually consistent with
prior publications of this IG and avoids any risk of misleading
sequential navigation if the page order has not been explicitly curated.

---

### Recommendation

**Recommended disposition:** A

The change requested is a one-line, low-risk edit to a publication
parameter in `sushi-config.yaml`, it is fully supported by the IG
Publisher tooling the work group already uses, and EHR has already
indicated it is Persuasive. Disposition B's underlying concern — that
the navigation order should be coherent — is real but is better handled
as a follow-up editorial pass once `shownav` is on and reviewers can
actually see the order; turning the chrome on first surfaces any
ordering problems for ballot reviewers to flag concretely. Disposition
C understates the readability benefit: ballot reviewers in particular
benefit from a linear walk through the IG, and the cost of providing it
is negligible.
