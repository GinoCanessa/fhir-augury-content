## Summary

The reporter, reviewing the AI Transparency on FHIR IG (January 2026 ballot,
[`general_guidance.html`](https://hl7.org/fhir/uv/aitransparency/2026Jan/index.html)),
flags that the meaning of each row in the table presented above the rows is
unclear. The reporter conjectures that each row represents one of the
"levels" of AI observability mentioned in the surrounding prose, and asks the
authors to clarify in the IG itself. John Moehrke reclassified the issue from
a technical correction to a Change Request because the requested clarification
involves authored-narrative restructuring rather than a typo / link fix. The
ticket has since been triaged with resolution **Persuasive with Modification**
and the work was carried out in
[`HL7/aitransparency-ig#22`](https://github.com/HL7/aitransparency-ig/pull/22),
which restructured the page to remove the table rows and present the same
content as named "factors" — the same PR that resolves the closely-related
[FHIR-54365](https://jira.hl7.org/browse/FHIR-54365) restructure of General
Guidance. The reviewer's job at ballot is to confirm that the restructured
narrative now answers the question the ticket originally asked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Confirm the already-merged resolution
([`HL7/aitransparency-ig#22`](https://github.com/HL7/aitransparency-ig/pull/22))
as **Persuasive with Modification** and close FHIR-53770 against the next
ballot snapshot. Concretely, in `input/pagecontent/general_guidance.md` the
authored content under the "AI Observability Factors" area no longer relies
on a row-keyed table whose key column was unlabelled; instead each AI
observability factor is presented as its own labelled subsection (`####
Defining the AI Model`, `#### Context of AI Usage`, `#### Process Utilizing
AI`, plus the output/utilisation factor) directly under the
`### AI Observability Factors` parent. This makes "what does each row
represent?" answer itself — the rows became named factors. As part of
acceptance, verify that any `general_guidance.html#…` links in
`index.md`, `use_cases.md`, and `explainer.md` resolve against the new
anchors and that `input/ignoreWarnings.txt` doesn't suppress a now-broken
intra-page link.

#### Justification

The ticket asks for clarification, not for a new artifact or constraint.
Rewriting unlabelled table rows as named subheadings is the lowest-risk way
to satisfy the exact request, is consistent with how the rest of the General
Guidance page is structured (everything else is `###`/`####` prose), and
avoids the table-rendering ambiguities the IG-publisher introduces. PR #22
is already in `main`, so this disposition is the cheapest path: confirm the
fix, validate anchors, ship in the next ballot.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the four-row table in `input/pagecontent/general_guidance.md` (rolling
back the table-to-headings part of PR #22 that pertains to this ticket) but
make the table self-describing by:

1. Adding an explicit first column header — e.g., `| Observability Factor |
   Description | Example | Mandatory at | …` — naming the column whose
   absence triggered the ticket.
2. Setting each row's first cell to the factor name (`AI Model`, `Context of
   AI Usage`, `Process Utilizing AI`, output factor), so the row is
   self-identifying.
3. Adding a one-sentence lead-in immediately above the table that states "Each
   row below describes one AI observability factor; the factors are the same
   four levels introduced in the preceding paragraph", with anchored links
   from each row label back to the corresponding deeper section that already
   exists below the table.

#### Justification

A table is genuinely useful when the same set of dimensions
(description / example / where-required / data carrier) must be compared
across factors at a glance — which is precisely the case here. The original
ticket complaint is *labelling*, not *table-vs-prose*, so a minimally
invasive fix is to label the rows and the key column. Compared to A this
preserves the side-by-side comparison view some readers prefer, and avoids
churn in `general_guidance.html` anchors that other IG pages may already
link to. Trade-off: it conflicts with PR #22 as merged and would need a
follow-up PR; it also duplicates the information that the per-factor
subsections below the table already convey.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive — already addressed**: take the position that the
prose surrounding the table on the source page already explicitly enumerates
the four observability factors, and that the rendered table ordering matches
that enumeration, so a reader following the page top-to-bottom is not
ambiguous. Close without further edits and without merging a row/heading
restructure for this ticket.

#### Justification

This is defensible only if the editors believe the existing narrative makes
the row-to-factor mapping clear by adjacency, and that adding labels would
clutter the table. It avoids further churn on a page that has already been
restructured for FHIR-54365 and others. The downside is that the reporter
explicitly said "It is not clear what the row represents", which is direct
evidence that an attentive ballot reviewer found the layout ambiguous —
declining a clarification request that was raised by a reader is a hard
position to hold against a ballot vote, especially with `Block-Vote-2`
applied.

---

### Recommendation

**Recommended disposition:** A

The reporter asked for clarification; the WG already produced a clarification
in PR #22 that restructures the rows into named factors and merged it as
`Persuasive with Modification`. The simplest, most honest action at this
point is to confirm that disposition, treat the ticket as the validation
checkpoint for the merged change, and verify the secondary effects in the
briefing's Gotchas (intra-page anchors, `ignoreWarnings.txt`) before the next
ballot snapshot. Disposition B is reasonable as a fallback if the WG wants
to preserve a comparison table, but it conflicts with the already-merged
restructure and would force a revert + redo cycle for marginal benefit.
Disposition C is hard to justify when a ballot reviewer has explicitly
flagged the layout as unclear and a low-risk fix is already in `main`.
