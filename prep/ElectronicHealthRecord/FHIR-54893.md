## Summary

The reporter (originally imported as a comment from Ron G. Parker) observes
that the AI Transparency IG defines "AI algorithm" intentionally broadly —
specifically: *"any computer-based logic that touches health data in a way
that might change the understanding of the data downstream."* Because that
definition explicitly stretches across the AI / non-AI boundary (e.g.,
deterministic calculators, simple transformations), the IG never draws a
crisp line between AI algorithms and ordinary computational logic. The
ticket argues that this ambiguity will lead implementers to interpret the
scope inconsistently, producing uneven observability practices: some
implementations will instrument every transformation as if it were AI,
while others will instrument only what they subjectively label "AI". The
ticket asks the work group to clarify the scope so observability can be
applied uniformly.

The single existing comment, from John Moehrke, takes the opposite view:
the definition is *deliberately* broad precisely so that non-AI
computational logic falls inside scope, and therefore the request is
already satisfied by the current wording.

No other Jira tickets are cross-referenced to FHIR-54893. The only related
discussion is the AI Transparency Zulip thread that lists this ticket
alongside ~30 other "Documentation & Clarity" January-ballot items for
triage.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the AI Transparency IG narrative page that defines "AI algorithm"
(the page reachable from the Jan-ballot index at
`hl7.org/fhir/uv/aitransparency/2026Jan/`, typically `index.html` or a
"Scope" / "Glossary" subpage) to add an explicit *"What is in scope"*
clarifier immediately after the existing definition. The clarifier
should:

1. State plainly that the IG's scope is **deliberately** wider than
   "AI" in the colloquial / ML sense, and that deterministic
   calculations, simple transformations, decision rules, and other
   non-AI computational logic that change the downstream
   understanding of health data are **all in scope** for the
   observability requirements.
2. Provide a short non-normative example list illustrating the
   boundary: e.g., a unit-conversion calculator (in scope), a
   risk-score regression (in scope), an LLM-generated summary (in
   scope), versus pass-through transport / serialisation that does
   not alter clinical meaning (out of scope).
3. Add a "Scope test" sentence implementers can apply: *"If the
   logic could change a downstream reader's understanding of the
   data, observability requirements apply, regardless of whether
   the logic is conventionally labelled 'AI'."*

No artefact (StructureDefinition, ValueSet, CodeSystem) changes are
required — this is purely a narrative-page change in the AI
Transparency IG repository.

#### Justification

This is the minimum-risk path: it preserves the existing definition
that John Moehrke (and the EHR WG by extension) has already endorsed
as intentionally broad, while directly addressing the reporter's
observation that implementers will not infer that intent from the
current wording. Adding worked examples and a single, applicable
"scope test" is the conventional FHIR pattern for resolving "the
definition is correct but readers don't realise its breadth"
clarity tickets, and keeps the change inside the Documentation &
Clarity bucket the WG has already assigned to FHIR-54893.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than patching the definition of "AI algorithm", reframe the
IG so the in-scope concept is named by what it does, not by what it
is. Concretely:

1. Rename the central scoping term from *"AI algorithm"* to
   something effect-based — e.g., *"data-affecting computational
   process"* or *"observable computational step"* — and define the
   AI Transparency observability requirements against that term.
2. Demote *"AI algorithm"* to an informative subset of the new
   term, with a one-line note that AI/ML implementations are the
   motivating use case but not the boundary.
3. Sweep the IG's narrative pages to update wording, and check
   whether any Coding / ConceptMap / extension element labels carry
   the literal phrase "AI algorithm" that would also need updating
   (this is not yet known without inspecting the IG sources).

#### Justification

Disposition A solves the symptom; Disposition B solves the cause.
The reporter's underlying concern is that the human label "AI"
fights the technical scope, and any future reader of the IG will
re-encounter that mismatch. An effect-based name makes the broad
scope self-evident and removes the need for repeated clarifying
prose throughout the IG. The trade-off is real ballot impact:
renaming a foundational term during a ballot window risks cascading
edits, may invalidate existing implementer mental models, and could
require co-ordinated changes to any extensions or value-set display
names that embed the phrase "AI". Because no GitHub repo briefing
is currently loaded for the AI Transparency IG, the exact scope of
that sweep cannot be quantified here, which is itself a reason to
prefer A unless the WG explicitly wants to invest in the rename.

---

### Disposition C: Decline (Not Persuasive / Already Addressed)

#### Proposal

Resolve the ticket as **Not Persuasive** (or **Already Addressed**)
on the grounds that:

- The current definition *already* explicitly includes non-AI
  computational logic ("any computer-based logic that touches
  health data…"), as called out in John Moehrke's comment.
- The reporter has not provided a concrete example where two
  reasonable implementers would land on opposite sides of the
  boundary under the current wording.
- The IG's observability requirements are written against the
  defined scope, so an implementer who follows the definition will
  apply observability uniformly.

Close with a short "Not Persuasive" disposition note that quotes
the existing definition and points to it as the authoritative
scope statement.

#### Justification

This is the position taken in the only existing comment on the
ticket and is internally consistent: if the definition is already
broad enough to cover the "non-AI" cases the reporter is worried
about, then the request is for clarification that has, on its face,
already been written. Declining preserves ballot velocity in a
window where the WG is processing 90+ comments. The risk is that
the reporter's underlying concern — that implementers will not
*read* the definition as broadly as the WG intends — goes
unaddressed, and the same comment is likely to return on a future
ballot.

---

### Recommendation

**Recommended disposition:** A

The current definition is technically correct (vindicating
Disposition C's premise) but the reporter's concern is empirically
plausible: the phrase "AI algorithm" carries a strong colloquial
meaning that most implementers will pattern-match before they
read the definition, so observability behaviour will diverge in
practice even when the spec is unambiguous on paper. A targeted
narrative clarification with worked examples and a one-sentence
"scope test" closes that gap at very low ballot cost, keeps the
ticket inside the Documentation & Clarity bucket the WG has
already triaged it into, and avoids the wider rename risk of
Disposition B. Disposition C should be reserved as a fallback if
the WG decides the existing wording is, in fact, prominent enough
to be self-explanatory; in that case the WG should still capture
the "scope test" wording in the Not Persuasive disposition so the
clarification lives somewhere implementers will find it.
