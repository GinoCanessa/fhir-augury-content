## Summary

The reporter observes that the AI Transparency IG defines a "Context"
observability factor with the subcategories *Request input (to AI)*,
*Reference input*, *Operations*, *Data quality*, and *Data
qualification*, but the IG only provides a model and worked examples for
encoding an input prompt in section 2.7 *Data sources*. The other
Context subcategories have no corresponding model artifacts. The ticket
also points out a structural ambiguity: an "input prompt" can be either
a *Request input* or a *Reference input* depending on whether the system
distinguishes user prompts from system prompts. Finally, naming is
inconsistent across the spec — the same idea is referred to as both
"Context" and "Data sources", and the same payload is referred to as
both "input prompt" and "request input". The reporter asks the workgroup
to either refactor the Context subcategory list to match the existing
model, or expand the model so every subcategory has a representation,
and to harmonise the terminology used across the spec. No related Jira
tickets are explicitly cross-referenced; the only inbound reference is
the Zulip triage table for the January ballot, which places this issue
in the "Data Sources & Context" cluster alongside FHIR-53967, FHIR-54211,
FHIR-54233, FHIR-54239, and FHIR-54433.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the model in the AI Transparency IG so that every Context
subcategory (*Request input*, *Reference input*, *Operations*, *Data
quality*, *Data qualification*) has a first-class representation
parallel to what section 2.7 *Data sources* already provides for input
prompts. Concretely:

1. Promote section 2.7 from "Data sources" to "Context" (or add a new
   parent "Context" page) so the page hierarchy mirrors the
   subcategory list.
2. Under that parent, add a sub-page per subcategory, each defining: a
   short narrative, the FHIR resources/profiles used to represent it,
   and at least one worked example. Reuse the existing input-prompt
   model for *Request input* / *Reference input* and add new sections
   for *Operations*, *Data quality*, and *Data qualification*.
3. Resolve the input-prompt ambiguity by giving it an explicit
   classification rule: a prompt that conveys the end-user's
   in-the-moment ask is *Request input*; a prompt that supplies
   reusable framing/system instructions is *Reference input*. Document
   this distinction on both subcategory pages and in the input-prompt
   example.
4. Rename "Data sources" everywhere it currently refers to the Context
   factor (page titles, section headings, navigation, cross-references)
   to "Context", and rename "input prompt" to "request input" /
   "reference input" as appropriate. Leave "Data sources" reserved for
   the broader concept of inbound content if the WG still wants that
   term, otherwise retire it.

#### Justification

This is what the reporter literally asked for and it is the option
that produces the most internally consistent IG. The Context factor
is currently described in one vocabulary (the subcategory list) and
modelled in another (Data sources / input prompt), which is exactly
the kind of inconsistency that ballot reviewers and downstream
implementers complain about. Filling out the missing subcategories
also addresses the structural ambiguity in the ticket: once *Request
input* and *Reference input* each have their own example, there is no
ambiguity about where an "input prompt" goes. The Zulip triage groups
this ticket with five other "Data Sources & Context" tickets, which
suggests the WG already expects to do non-trivial work in this area
for the next ballot — folding this in costs less than handling the
related tickets one-off.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the model, narrow the Context subcategory list
to match what the IG actually defines today, and address the naming
mismatch as a renaming-only change:

1. Reduce the Context subcategories to the ones for which the IG
   provides a model — most likely *Request input* and *Reference input*
   only, both encoded via the existing section 2.7 mechanism.
2. Move *Operations*, *Data quality*, and *Data qualification* out of
   the Context factor. Either reassign them to existing observability
   factors (e.g., model *Operations* under the Operations factor if one
   exists, *Data quality* / *Data qualification* under a Data factor),
   or move them into a "future work" / non-normative appendix until
   modelled.
3. Rename section 2.7 and its examples consistently — pick either
   "Context" or "Data sources" as the canonical term and replace the
   other throughout. Disambiguate "input prompt" by retiring the term
   in favour of "request input" / "reference input" with a one-line
   note that maps the legacy terminology.
4. Add a one-paragraph classification rule for prompts (user prompt =
   request input, system prompt = reference input) so the
   user/system-prompt ambiguity is resolved without adding new model
   artifacts.

#### Justification

The IG is at ballot, not pre-ballot, and adding three new modelled
subcategories is a substantial scope expansion that would invalidate
existing review and likely require a re-ballot. Trimming the
subcategory list to what is already modelled is a much smaller change
that still removes the inconsistency the reporter is complaining
about, and it can land in the disposition cycle without new ballot
content. It also forces the WG to be honest about which factors
actually have FHIR mappings today, which is useful information for
implementers. The trade-off is that the IG ends up with a less
ambitious Context taxonomy than its authors originally intended — but
the missing subcategories can still be re-introduced in a later
publication once their models exist.

---

### Disposition C: Decline

#### Proposal

Reject the request as out of scope for this ballot cycle and close
with a "Persuasive with Modification" pointing the reporter at the
narrower naming-only fixes already in flight (e.g., the related
"Data Sources & Context" cluster: FHIR-53967, FHIR-54211, FHIR-54233,
FHIR-54239, FHIR-54433). Acknowledge the input-prompt ambiguity in a
short clarifying note in section 2.7 — "an input prompt is *Request
input* when it carries the end-user ask and *Reference input* when it
provides system framing" — but do not refactor the subcategory list or
rename the Context section.

#### Justification

The ticket bundles three quite different asks (refactor the model,
expand the model, harmonise naming) into a single Highest-priority
Change Request and does not propose a concrete target structure. That
makes it expensive to action in a single disposition without
pre-empting decisions that the related tickets will force anyway. A
bare clarifying note resolves the only concrete interoperability risk
(prompt classification) at near-zero cost, and the WG can revisit a
proper refactor in the next cycle once the related tickets have been
worked through and the WG has a clearer view of the taxonomy it wants.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible answer for a ticket that is
filed at Highest priority but is essentially a "the model and the
narrative don't match" complaint. Disposition A is intellectually the
right end-state, but inventing first-class models for *Operations*,
*Data quality*, and *Data qualification* in a disposition cycle is
realistically several ballots of work and would balloon the scope of a
single Change Request beyond what the WG can absorb before
publication. Disposition C leaves the inconsistency in the published
IG and is hard to justify when the reporter has correctly identified
that the subcategory list and the model are out of sync. Disposition B
fixes the inconsistency immediately by aligning the subcategory list
to what is actually modelled, resolves the input-prompt ambiguity with
an explicit user-prompt / system-prompt rule, and harmonises the
"Context" vs. "Data sources" / "input prompt" vs. "request input"
naming — all of which can be done within disposition without
expanding ballot content. The unmodelled subcategories can be
reintroduced in a later cycle, ideally driven by the other "Data
Sources & Context" tickets (FHIR-53967, FHIR-54211, FHIR-54233,
FHIR-54239, FHIR-54433) so the WG addresses the cluster as a coherent
piece rather than ticket-by-ticket.
