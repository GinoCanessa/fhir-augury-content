## Summary

The submitter (Moritz Kähler) is asking the AI Transparency (FHIR) IG
authors to clarify the intended scope of the **AIRPT** code, defined as
"Indicates that the content was reported by an AI system." The ambiguity
is whether AIRPT applies to *any* AI-generated content, or only when the
AI output is explicitly part of a "report" artifact (e.g., a diagnostic
or narrative report). The reporter wants this distinction made explicit
to support consistent implementation. During triage, John Moehrke (EHR
WG / IG author) noted that in the underlying Provenance vocabulary,
"Reported" is specific to an agent that is merely *passing through*
content, which makes the concept a poor fit for AI generation; he
indicated on the ticket that the AIRPT code should go away. The ticket
has been converted from a question to a Change Request and is currently
Triaged. There are no other Jira or GitHub cross-references; the only
related discussion is the IG's January-ballot triage roll-up on Zulip,
which lists this ticket under "Documentation & Clarity" (i.e., not yet
discussed on its own merits).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Clarify the definition of `AIRPT` in the AI Transparency IG's
ParticipationType supplement / extension code system so that the
intended scope is explicit. Specifically, update the `definition` of
`AIRPT` (and any narrative usage notes in the IG that introduce the
code) to state unambiguously which of the two interpretations applies —
e.g., either:

- "Indicates that an AI system produced or generated the content
  (regardless of whether the content is delivered as a formal report
  artifact)"; **or**
- "Indicates that an AI system produced content that is being delivered
  as a report artifact (e.g., a DiagnosticReport or narrative report);
  use a different code for non-report AI-generated content."

Add a "Usage Note" paragraph to the IG page that introduces the AI
participation/role codes, explicitly contrasting AIRPT with the other
AI-related codes (e.g., AI author / AI informant) and giving one
positive and one negative example. Do not change the code value or
remove it.

#### Justification

This is the literal request: the reporter asked for clarification, not
removal. Keeping AIRPT and tightening its definition is the
lowest-impact change for implementers who have already started to
consume the January ballot draft, and it preserves any conformance
artifacts (profiles, examples, ValueSets) that already reference the
code. The clarification is also a documentation-only change, suitable
for a "Persuasive with Mod" disposition that does not affect the
substantive ballot.

---

### Disposition B: Alternative Approach — Retire AIRPT

#### Proposal

Remove `AIRPT` from the AI Transparency IG's code system in the
post-ballot revision and rely on the remaining AI participation/role
codes (e.g., the codes that designate an AI system as author /
generator / informant) plus standard `Provenance` modeling to convey
"this content came from an AI system." Specifically:

1. Mark `AIRPT` as `deprecated` (or remove it outright if the IG is
   pre-normative and no published artifact depends on it) in the IG's
   ParticipationType extension code system.
2. In the IG narrative, document that "reported by" semantics for an AI
   agent are not meaningful — Provenance's existing "Reported" concept
   is reserved for an agent that passes through pre-existing content
   without modifying it, which is not a useful distinction for an AI
   system that is, by construction, generating content.
3. Where examples or profiles in the IG currently use AIRPT, migrate
   them to the appropriate alternative AI role/author code.
4. Add a migration note to the IG change log calling out the removal
   and the recommended substitute(s).

#### Justification

This is what John Moehrke (the IG author who triaged the ticket)
actually proposed in comments: in the Provenance vocabulary "Reported"
is specific to pass-through agents, so applying it to AI generation is
semantically incorrect and will cause inconsistent tagging in the
field. The reporter's underlying confusion is itself evidence that the
code is mis-named relative to its intended meaning; clarifying the
definition (Disposition A) leaves a code in the IG whose name still
conflicts with Provenance semantics for any reader who later looks up
"Reported" in the base spec. Retiring the code aligns the IG with the
base FHIR Provenance vocabulary and removes a long-term source of
confusion. It is a more disruptive change than A, but the IG is at
ballot — this is the right window to take it.

The reporter's follow-up question ("will there be new codes to better
tag unconfirmed AI-generated data?") is addressed separately: per
Moehrke's response, the IG's design is that *absence* of an AI tag
indicates "not influenced by AI," and negative-meaning tags are not
introduced. That answer should be folded into the IG narrative as part
of this disposition.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Not Persuasive" on the grounds that the existing
definition of AIRPT, read in the context of the full code system and
the IG's introductory narrative, is sufficient and that the
clarification the reporter is asking for is implementer education
rather than a specification defect. Direct the reporter to the IG's
existing usage notes and the base Provenance specification's definition
of "Reported."

#### Justification

This is the defensible "do nothing" path. It is the weakest of the
three because (a) the IG author has explicitly agreed on the ticket
that the concept is problematic, and (b) the reporter's confusion is a
real-world signal that the current text is insufficient. Decline would
only be appropriate if the WG concludes that the ballot is too late to
absorb either a definition change or a code retirement, and that
implementer guidance can be issued out of band. It would leave the
underlying Provenance-vocabulary mismatch in place, which is likely to
generate repeat tickets in the next ballot cycle.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — Retire AIRPT)

The IG author has already stated on the ticket that "Reported" in the
Provenance vocabulary means pass-through agency and that AIRPT is
therefore not a useful concept for AI generation. That diagnosis is
correct — keeping the code (Disposition A) only patches over a naming
collision with the base spec, and any clarification text the WG writes
will have to explain *why* AIRPT does not mean what its name suggests.
Disposition B fixes the root cause, aligns the IG with the base
Provenance semantics, and lets the EHR WG take the simultaneous
opportunity to document the IG's intentional "absence-of-tag means
not-AI-influenced" design (per Moehrke's response to the reporter's
follow-up question), which several other Provenance & Tagging tickets
in the same January-ballot triage roll-up will likely also benefit
from. Disposition A is the acceptable fallback if the WG decides that
removing a balloted code is too disruptive for this cycle, in which
case the definition rewrite must explicitly call out the divergence
from Provenance's "Reported" semantics. Disposition C is not
recommended; the reporter has identified a real defect that the IG
author has already acknowledged.
