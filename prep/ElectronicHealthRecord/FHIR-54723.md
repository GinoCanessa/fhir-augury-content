## Summary

Brian Postlethwaite reports a design conflict in the AI Transparency IG's
AI Device profile. The profile leaves `Device.type` with its default example
binding to the base FHIR device-type value set and then layers a `pattern`
on top to try to constrain it to codes from `AIdeviceTypeCS`. The reporter
argues that this combination is awkward and would be better expressed by
binding `Device.type` directly to `AIdeviceTypeVS` as **required**, which
removes the need for the pattern.

The ticket also flags two structural problems with `AIdeviceTypeCS`
itself:

1. The code system mixes distinct concept categories — a top-level
   "Artificial-Intelligence" marker, model-type concepts, and scenario
   concepts — and the umbrella "Artificial-Intelligence" code overlaps in
   meaning with the more specific codes. That overlap breaks the
   semantics of `extensible` binding (the umbrella code "always applies",
   so implementers can never legitimately reach for a code outside the
   set).
2. The profile also defines an extension that re-uses the same code
   system extensibly, compounding the binding-strength problem.

The reporter proposes splitting `AIdeviceTypeCS` into three code
systems / value sets / extensions: a single-code system holding only
"Artificial-Intelligence" (required-bound on `Device.type`), an
`ai-model-type` system/extension (e.g., AI-By-Model), and an
`ai-scenario` system/extension (AI-By-Scenario). Reviewer John Moehrke
has acknowledged in a comment that the code system needs improvement and
defended the use of slicing/extensions on the basis of forward
compatibility with later FHIR versions where `Device.type` is `0..*`.

No directly-linked tickets are recorded in Jira. The only cross-reference
is a triage roll-up post in the AI Transparency on FHIR Zulip stream
that categorises this ticket under "Profile & Model Structure" alongside
~18 other ballot items, but does not contain a substantive discussion of
the issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure the AI Device profile and its supporting terminology along
the lines proposed by the reporter:

1. Split `AIdeviceTypeCS` into three new code systems (each with a
   matching value set):
   - `AIdeviceCS` / `AIdeviceVS` — contains only the umbrella
     `Artificial-Intelligence` code.
   - `AIModelTypeCS` / `AIModelTypeVS` — contains today's "AI-By-Model"
     family of codes.
   - `AIScenarioCS` / `AIScenarioVS` — contains today's "AI-By-Scenario"
     family of codes.
2. On the AI Device profile, change `Device.type` to a **required**
   binding to `AIdeviceVS` and remove the `pattern` constraint — the
   required binding alone enforces that `Device.type` is the
   `Artificial-Intelligence` concept.
3. Replace the existing single AI-type extension with two extensions —
   `ai-model-type` (extensibly bound to `AIModelTypeVS`) and
   `ai-scenario` (extensibly bound to `AIScenarioVS`). Each extension
   carries a single `CodeableConcept` and may repeat as appropriate.
4. Mark the original `AIdeviceTypeCS` / `AIdeviceTypeVS` and combined
   extension as deprecated/retired in this ballot cycle, with
   migration notes mapping old codes to the new systems.

#### Justification

This directly resolves both problems the ticket calls out:

- **Binding correctness.** A required binding plus removal of the
  pattern is the FHIR-idiomatic way to constrain `Device.type` to a
  single concept; layering a pattern on top of an example binding is a
  documented anti-pattern (the example binding does not enforce
  anything, and the pattern duplicates work the binding is meant to
  do).
- **Extensible-binding semantics.** Splitting the umbrella concept out
  of the model/scenario value sets removes the overlap that currently
  defeats `extensible` semantics. Once removed, implementers can
  legitimately fall back to local codes when the IG-published codes
  don't fit, which is the intent of `extensible`.
- **Concept hygiene.** Separate code systems for separate concept
  axes (presence-of-AI vs. model-type vs. scenario) is consistent with
  HL7 vocabulary guidance and makes downstream profiling, terminology
  binding, and value-set composition far cleaner.

The reviewer's R4-vs-later-versions concern (preserving slicing-friendly
shape) is addressed by keeping the structural pattern on the profile
(extensions for repeating axes) while fixing the bindings.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing single `AIdeviceTypeCS` (and its value set), but:

1. Tighten the value-set composition so that `AIdeviceTypeVS` excludes
   the umbrella `Artificial-Intelligence` code and contains only the
   model-type and scenario codes; bind the new
   "AI-flag" / `Device.type` slot via a separate small required value
   set carrying just `Artificial-Intelligence`, and drop the
   `pattern` constraint on `Device.type`.
2. Bind the existing AI-type extension to the trimmed `AIdeviceTypeVS`
   with strength **`preferred`** rather than `extensible`, removing the
   semantic problem the umbrella code created without splitting the
   code system into three.
3. Add explicit guidance text on the profile page and the code-system
   page explaining the axis distinctions (umbrella vs. model vs.
   scenario) and how to record them.

#### Justification

This addresses the binding-strength conflict (no more
"required-on-`Device.type`-via-pattern" plus "extensible-on-extension-
with-overlapping-codes") with a smaller surface change than
Disposition A. It preserves continuity for early implementers of the
Jan 2026 ballot draft who have already wired against `AIdeviceTypeCS`,
and it avoids deprecating any artifacts. The trade-off is that it does
not fully separate the model-type and scenario axes — implementers can
still mix them in a single extension — and `preferred` binding is
weaker than what the reporter would like, so future tightening would
itself be a breaking change.

---

### Disposition C: Decline

#### Proposal

Decline the structural changes; retain the current `AIdeviceTypeCS`,
`AIdeviceTypeVS`, single AI-type extension, and the
`example`-binding-plus-`pattern` shape on `Device.type`. Resolve only
the documentation gap by adding clarifying notes to the code-system
page about how the umbrella `Artificial-Intelligence` code relates to
the more specific codes and how implementers should choose among them.

#### Justification

Defensible only on grounds of ballot-cycle scope: the IG is mid-ballot
and has many other open items. The reviewer's comment notes that the
profiling shape was chosen deliberately for forward-compatibility with
later FHIR versions where `Device.type` is `0..*`. Declining preserves
that shape and avoids breaking implementations already coded against
the Jan 2026 draft. However, declining leaves the
`extensible`-binding-with-overlapping-umbrella-code problem in place,
which is a real conformance hazard, so this disposition is hard to
justify on technical merit.

---

### Recommendation

**Recommended disposition:** A

The ticket identifies two genuine, independent problems: (1) an
`example`-binding-plus-`pattern` construction that is not the right
FHIR mechanism for "this `Device.type` must be `Artificial-Intelligence`",
and (2) an `extensible`-bound value set that contains an umbrella code
which logically subsumes every other code in the set, defeating the
semantics of extensibility. The reviewer's own comment already concedes
that "the codeSystem is indeed needing improvement", and the reviewer's
concern about preserving slicing-friendly shape for future FHIR
versions is orthogonal to the binding/terminology fix and can be kept.
Disposition B is a defensible compromise but only partially solves the
concept-axis confusion and leaves the IG carrying weaker bindings than
needed; Disposition C is hard to justify because it knowingly ships a
broken extensible binding. Disposition A is the cleanest long-term
shape, the change is bounded to one IG and one profile, and it gives
implementers an unambiguous, axis-by-axis terminology surface to bind
against.
