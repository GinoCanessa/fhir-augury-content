## Summary

The ticket asks the AI Transparency on FHIR work group to reconsider the
direction of the relationship between the `AI-Device` profile and the
`AI-ModelCard` artifact. As currently published in the January 2026 ballot
of the AI Transparency IG, `AI-Device` carries an extension
`modelCardDescription` that points from the Device to the Model Cards it
supports. The reporter (Abderrazek Boufahja) argues that this orientation
is operationally awkward: every time a new Model Card is created, every
applicable `Device` instance must be re-published to add the new
reference. He proposes inverting the link so that the `Model-Card`
artifact references the list of `AI-Device` instances to which it
applies, eliminating the need to re-issue Device records each time the
Model Card catalogue grows.

There are no other Jira tickets cross-referenced to this one, but the
ticket is part of the broader Jan-ballot triage tracked in the
"AI Transparency on FHIR > IG - Jan ballot" Zulip topic, where it is
classified by the IG editors under **"Profile & Model Structure"** —
i.e., a structural change to the IG profiles rather than a documentation
fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Invert the direction of the `Device` ↔ `ModelCard` linkage in the AI
Transparency IG. Specifically, in
`StructureDefinition-AI-Device`:

- Remove (or deprecate) the `extension:modelCardDescription` slice that
  currently lets a `Device` enumerate the Model Cards it supports.

In the `AI-ModelCard` profile (the Model Card resource/profile that
backs the IG):

- Add a repeating element/extension (e.g.,
  `appliesToDevice : Reference(AI-Device)` 0..*) that lists the
  `AI-Device` instances that may apply this Model Card.

Update narrative, example instances and the IG's "Relationship between
Device and Model Card" page to describe the Card-to-Device direction.

#### Justification

The reporter's operational argument is sound: in publication-style
catalogues the artifact that changes most often (new Model Cards being
authored or revised) should carry the cross-reference, not the artifact
that is comparatively stable (an installed `Device`). Putting the
reference on the `Model-Card` side avoids forcing every existing
`Device` instance to be re-issued each time the Model Card catalogue
grows, and it aligns with the general FHIR pattern of "the new/changing
resource references the existing one" (e.g., `Observation` →
`Patient`/`Device`, not the reverse).

---

### Disposition B: Alternative Approach

#### Proposal

Keep both orientations but make them complementary, and clarify that the
authoritative link lives on the **Model Card** side:

- In `AI-ModelCard`, add `appliesToDevice : Reference(AI-Device) 0..*`
  as the **normative** linkage element.
- Re-purpose the existing `extension:modelCardDescription` on
  `AI-Device` as an **optional, descriptive** pointer (cardinality 0..*,
  flagged as informational / "snapshot at publish time") — i.e., a
  Device MAY advertise the Model Cards it shipped with, but the
  authoritative current set is determined by querying Model Cards that
  reference the Device.
- Add a clear sentence in the IG's narrative on
  `StructureDefinition-AI-Device.html` that explains the dual
  representation, when each is appropriate, and that the Model-Card-side
  reference is the one consumers should rely on.
- Add an `invariant` or guidance note discouraging mutation of the
  Device-side list once the Device is in production.

#### Justification

This addresses the reporter's lifecycle concern (no need to re-issue
Devices every time a Model Card is added) without breaking implementers
who have already wired up the Device-side extension during the January
ballot cycle. It also acknowledges a legitimate use case for the
Device-side representation: at deployment / commissioning time, an
operator may want a static, signed snapshot of which Model Cards a
particular AI-enabled Device shipped with — even if the long-term
authoritative linkage lives on the Card. The trade-off vs Disposition A
is added complexity (two ways to express the same relationship), which
is mitigated by clearly designating one as authoritative.

---

### Disposition C: Decline (Persuasive — keep current direction with
clarification only)

#### Proposal

Keep the relationship on the `AI-Device` side as currently specified.
Address only the reporter's secondary point — that the
`extension:modelCardDescription` element is under-described — by adding
a clarifying definition / short description and an example explaining
that the list is intended to be the set of Model Cards the device was
**designed to operate with at publication time**, not a live registry.

#### Justification

Declining is defensible if the work group concludes that the Device
profile is intentionally modelling an installed-base concept: a
particular AI-enabled `Device` is built around a known, finite set of
Model Cards, and adding a new Model Card to the world genuinely should
trigger a new Device version (or at least a new Device instance), since
the underlying software/firmware has changed. In that framing, the
"every new Model Card forces a Device update" concern is not a bug but
a deliberate signal that the Device's behaviour has changed. Inverting
the relationship would also mean that, to know what a given Device
supports, a consumer must perform a `_revinclude`-style query against
all Model Cards — which is more expensive than reading a single
extension on the Device. If the work group prefers the simpler
read-side ergonomics and the Device-as-versioned-software framing, only
the description text needs to change.

---

### Recommendation

**Recommended disposition: B (Alternative Approach).**

Disposition B captures the operational benefit the reporter is asking
for — Model Cards can be added without re-issuing Devices — while
preserving the read-side ergonomics that motivated the original
Device-side extension and avoiding a hard breakage of January-ballot
implementations. The Zulip triage explicitly classifies this ticket
under "Profile & Model Structure" rather than "Documentation &
Clarity", which signals that the work group already expects a
structural change rather than a description-only fix; that argues
against Disposition C. Disposition A is cleanly correct in principle
but is a breaking change relative to the January ballot artifact and
loses the legitimate "shipped-with" snapshot use case. Designating the
Model-Card-side reference as authoritative (B) gives implementers a
clear migration target while letting the Device-side extension survive
as an informational, point-in-time list.
