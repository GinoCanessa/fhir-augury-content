## Summary

The reporter (Abderrazek Boufahja) is reviewing the January 2026 ballot of the
AI Transparency IG and has filed a clarification request against the
`AI-Provenance` profile
(`https://hl7.org/fhir/uv/aitransparency/2026Jan/StructureDefinition-AI-Provenance.html`).
The profile currently defines an explicit slice `entity:AIInputData` for the
data items fed directly into an AI model, but the reporter notes that there
are *other* input data which are "source data" — resources that were used to
generate or to influence the target resources of the Provenance (rather than
being raw model inputs). The request is to make it explicit in the profile —
"even if that seems implicitly the case" — that `AI-Provenance` is the
mechanism that links the target AI-based resources back to the source data
used to create or influence them, so implementers do not have to infer that
intent. John Moehrke commented (2026-01-26) that this reads as a change
request rather than a passing comment, paraphrasing the ask as "improve the
explanation of the use of `Provenance.entity` to hold the data given to the
AI." The cross-reference search surfaced no other Jira tickets directly
linked to FHIR-53973, but the AI Transparency IG ballot triage thread on
Zulip groups it with nine other tickets in a "Provenance & Tagging" bucket,
indicating the WG is treating it alongside related Provenance modelling
items.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the AI Transparency IG narrative for the `AI-Provenance` profile
(authoring source for `StructureDefinition-AI-Provenance.html`, typically the
profile's intro/notes file in the IG sources alongside the StructureDefinition
itself) to add an explicit, top-level statement that `AI-Provenance` serves as
the link between the AI-generated/AI-influenced **target** resources
(`Provenance.target`) and the **source** data used to create or influence
them, expressed via `Provenance.entity`. Specifically:

1. In the profile's introductory description, add a paragraph along the lines
   of: *"This profile is the canonical mechanism in this IG for recording the
   relationship between an AI-produced or AI-influenced resource (referenced
   by `Provenance.target`) and the data inputs used by the AI system. Inputs
   that were directly fed to the model are recorded using the
   `entity:AIInputData` slice; other resources that were used to generate or
   to influence the target — for example reference data, retrieval-augmented
   sources, or contextual records — are recorded as additional
   `Provenance.entity` occurrences using the appropriate `entity.role` (e.g.
   `source`, `derivation`)."*
2. Tighten the slice documentation on `entity:AIInputData` to clarify that it
   is **not** the only allowed `entity` slice and that other entities with
   roles such as `source` / `derivation` remain valid and expected for
   non-model-input source data.
3. Optionally, add a short "How to record source vs. model-input data"
   bullet list or a tiny example snippet in the profile's "Notes" or "Usage"
   section to make the distinction concrete for implementers.

No StructureDefinition cardinality, slice, binding, or invariant changes are
required — this is a documentation/clarification change to the profile's
narrative.

#### Justification

The reporter explicitly says the intent "seems implicitly the case" and is
asking only that it be made explicit. John Moehrke's triage comment confirms
the same reading. FHIR's general modelling pattern for `Provenance.entity`
already supports this distinction via `entity.role` (R5 Provenance roles
include `source`, `derivation`, etc.), so the IG can lean on the base
resource semantics without inventing new structure. Improving narrative for
a sliced profile element is consistent with how other HL7 IGs treat
slice-vs-base-element clarifications, and it removes a real implementer
ambiguity (a reader could reasonably conclude that `entity:AIInputData` is
the *only* sanctioned way to populate `Provenance.entity` in this profile).
The change is low-risk, non-breaking, and squarely within "documentation &
clarity" — the largest bucket in the ballot triage.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) narrative-only clarification, formalise the
distinction in the StructureDefinition by adding a second named slice on
`Provenance.entity` for non-input source data — e.g.
`entity:AISourceData` — with `role` fixed or pattern-bound to `source` (and
optionally `derivation`), and update the profile narrative as in
Disposition A. The existing `entity:AIInputData` slice would keep its
current semantics (data fed directly to the model). The slicing
discriminator on `Provenance.entity` would be widened to accommodate both
slices (typically discriminating on `role` and/or a tag on `what`).

#### Justification

A second explicit slice makes the two roles discoverable through tooling
(IG publisher, profile renderers, validators) rather than relying on prose
alone, and it gives implementers a structural target to bind against —
useful for downstream profiling, validation, and quality-measure logic that
needs to distinguish "data the AI saw" from "data that influenced the AI's
behaviour but was not an input to the model." It is, however, a
**structural** change: it touches slicing on a profile that has already
been balloted in January 2026, which carries higher review burden, may
break existing examples and downstream profiles, and crosses into the
"Profile & Model Structure" bucket of the triage rather than staying in
"Documentation & Clarity." It also raises a follow-on question (which the
ticket does not ask) about whether the IG should constrain `entity.role`
codes for AI-source vs. AI-input — which is a separate design discussion.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Persuasive with Mod" → no change to the profile, on the
basis that base R5 `Provenance.entity` already allows multiple entities with
distinct `role` codes (`source`, `derivation`, etc.) and that the
`entity:AIInputData` slice is additive — not exclusive. A short response
would point implementers at the R5 `Provenance` definition and the
`http://hl7.org/fhir/provenance-entity-role` value set as the existing
mechanism for recording source data, and close the ticket without
modifying the profile.

#### Justification

Strictly speaking, the requested behaviour is already supported: nothing in
the AI-Provenance profile forbids additional `Provenance.entity` entries
with `role = source` or `role = derivation`. A reviewer could reasonably
argue this is implementer education rather than a profile defect, and the
WG could choose not to expand the IG's scope into "explain base FHIR
semantics." Declining preserves narrative footprint and avoids precedent
for re-stating base-resource behaviour inside every profile.

---

### Recommendation

**Recommended disposition:** A

The reporter is not asking the WG to add structure or change semantics; they
are asking for a single explicit sentence so implementers don't have to
infer the profile's purpose. John Moehrke's own triage comment frames the
ask the same way ("improving the explanation"). Disposition A delivers
exactly that with no breaking change, no slicing rework, and no impact on
existing examples or downstream profiles, and it cleanly fits the
"Documentation & Clarity" treatment the ballot triage thread already
contemplates for this kind of issue. Disposition B is a defensible *future*
change — and the WG may want to track it as a separate ticket — but it is
disproportionate to what was asked and would shift the work into the
structural bucket mid-cycle. Disposition C technically holds up against
base FHIR, but it leaves the documented ambiguity in the IG and effectively
tells the reporter "you should have known"; that is not a good outcome for
an IG whose entire premise is *transparency*. Accept and clarify.
