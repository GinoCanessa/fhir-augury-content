## Summary

The reporter raises a mapping question for the EU Laboratory Report (FHIR)
specification. The CDA-style element `LabReport.header.metadata.dateTime`
(section A.1.8.3 "Report date and time") is currently mapped to FHIR
`DiagnosticReport.effectiveDateTime`. The reporter notes that the EU narrative
specification does not assert that this date/time represents the *clinically
relevant* date/time (which is what `DiagnosticReport.effectiveDateTime` is
meant to convey); it more naturally reads as the date/time the report itself
was produced/released. The ticket asks whether the mapping should instead
target `DiagnosticReport.issued`. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced; the only cross-reference is the
in-text mention of the FHIR element `DiagnosticReport.effectiveDateTime`.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the EU Laboratory Report (FHIR) specification, change the mapping of
`LabReport.header.metadata.dateTime` (section A.1.8.3 "Report date and time")
from `DiagnosticReport.effectiveDateTime` to `DiagnosticReport.issued`.
Update both the narrative mapping table in the IG and any computable
StructureDefinition mappings (e.g., `mapping` entries on the EU Lab Report
DiagnosticReport profile that reference the CDA `LabReport` element) so the
A.1.8.3 source element points at `DiagnosticReport.issued`. Add a short note
in the mapping prose clarifying that `effectiveDateTime` is reserved for the
clinically-relevant date/time of the diagnostic activity (e.g., specimen
collection or observation), and that the "report date and time" represents
when the report itself was made available, which aligns with the FHIR R4
definition of `DiagnosticReport.issued` ("The date and time that this version
of the report was made available").

#### Justification

The base FHIR R4 `DiagnosticReport` definitions distinguish two semantically
different date/times: `effectiveDateTime[x]` is "The time or time-period the
observed values are related to" — the *clinically relevant* time — while
`issued` is "The date and time that this version of the report was made
available". The reporter correctly observes that the EU narrative for A.1.8.3
"Report date and time" does not assert clinical relevance; the natural
reading is the report-production timestamp. Mapping a report-production
timestamp to `effectiveDateTime` therefore mis-uses the FHIR element and can
mislead downstream consumers (decision support, trending) that reasonably
treat `effectiveDateTime` as the clinically meaningful time. Re-mapping to
`issued` aligns the EU IG with the FHIR semantic contract.

---

### Disposition B: Alternative Approach

#### Proposal

Map `LabReport.header.metadata.dateTime` (A.1.8.3) to **both**
`DiagnosticReport.issued` (primary) and, when systems cannot populate a
distinct clinically-relevant time, allow it to also satisfy
`DiagnosticReport.effectiveDateTime[x]` as a fallback. Concretely:

1. In the EU Lab Report IG, change the primary mapping for A.1.8.3 to
   `DiagnosticReport.issued`.
2. Add narrative guidance that, if no separate clinically-relevant date/time
   is available from the source (e.g., specimen collection time, observation
   time), implementers MAY also use the same value to populate
   `effectiveDateTime`, but SHOULD prefer a more clinically meaningful time
   when one is known (e.g., from the contained `Observation.effectiveDateTime`
   or `Specimen.collection.collectedDateTime`).
3. Optionally add an `invariant` or guidance note that
   `DiagnosticReport.effectiveDateTime` SHOULD be derived from specimen
   collection / observation effective times when present, with the report
   date as a last-resort default.

#### Justification

In real-world lab feeds, the only date/time guaranteed to be present is
often the report date/time itself; specimen collection and observation
effective times may be missing. Pure Disposition A could leave
`effectiveDateTime` empty in many instances, even though base FHIR
`DiagnosticReport` strongly encourages it (1..1 in many profiles for
trending/decision support). Disposition B preserves the semantic correctness
the reporter is asking for — `issued` is the *primary* target — while giving
implementers a documented fallback so the resulting `DiagnosticReport`
remains useful and interoperable. This is consistent with how several other
HL7 lab IGs (e.g., US Core / IPS DiagnosticReport for laboratory) handle
the same source ambiguity.

---

### Disposition C: Decline

#### Proposal

Keep the existing mapping of `LabReport.header.metadata.dateTime` (A.1.8.3)
to `DiagnosticReport.effectiveDateTime`. Add a single clarifying sentence to
the EU IG mapping prose stating that, in the EU Laboratory Report context,
the "report date and time" is treated as the effective date/time of the
report when no more granular clinically-relevant time is available, and
that `DiagnosticReport.issued` is reserved for the timestamp of the FHIR
resource version being exchanged.

#### Justification

A defensible reason to decline is implementation continuity: existing
EU Lab Report producers and consumers have been built against the current
mapping, and a mapping change touches both narrative tables and computable
`mapping` elements in the published StructureDefinitions. If the workgroup
considers `DiagnosticReport.issued` to mean "the timestamp at which *this
FHIR resource version* was issued" (per the FHIR definition's emphasis on
"this version of the report"), then the source element — which is about the
*clinical report* date, not the FHIR transport timestamp — is arguably
closer to `effectiveDateTime` than to `issued`. Under this reading the
existing mapping is acceptable and the fix is purely editorial.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern is correct: per base FHIR R4,
`DiagnosticReport.effectiveDateTime` is the *clinically relevant* time and
not the report-production time, so mapping A.1.8.3 ("Report date and time")
solely to `effectiveDateTime` is semantically off. Disposition A fixes that,
but in laboratory feeds the report date is often the only date/time
available, and pure A risks producing `DiagnosticReport` instances with no
`effectiveDateTime` — which downstream consumers (and several profile
constraints) will not handle gracefully. Disposition B keeps the semantically
correct primary mapping (`issued`) that the reporter is asking for, while
explicitly handling the common real-world case where the report date is also
the best available proxy for the clinically relevant time. Disposition C
preserves a known-incorrect semantic in exchange for editorial convenience
and should only be chosen if the workgroup has direct evidence that
implementers are relying on the current mapping in a way that a re-mapping
would break.
