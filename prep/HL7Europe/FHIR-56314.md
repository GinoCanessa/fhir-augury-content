## Summary

The reporter (RyanMay, NHSE) asks the EU Laboratory Report (FHIR) IG to relax
the cardinality of `Observation.effective[x]` from the IG-mandated `1..1` back
to the FHIR base `0..1`. The reporter argues that the natural source of the
clinically relevant time for a lab result is the specimen-side data
(`Specimen.collection.collected[x]`), and that mandating `Observation.effective[x]`
duplicates information that should live on the `Specimen`. A follow-up comment
extends the concern to dynamic function tests (e.g., glucose tolerance test)
where the grouping `Observation` would inherit multiple distinct
`Specimen.collection.collected[x]` datetimes from member specimens, making a
single `Observation.effective[x]` artificial. There are no cross-referenced
Jira tickets, Zulip threads, or GitHub items associated with this ticket; the
only cross-references are concept mentions of `Observation.effective`,
`Specimen.collection.collected`, `Specimen.receivedTime`, and
`Specimen.processing.time`.

## Details

**Description:**

> The Observation.effective[x] element has been mandated, which I'd expect to
> be the date/time the specimen was collected.
>
> Mandating this element creates duplication as we (NHSE) would expect this
> value to be stored within Specimen.collected[x].
>
> Is there a reason this was set as mandatory, otherwise it would be best to
> relax the cardinality back to base 0..1.

**Proposed Resolution Description (already entered on the ticket, status
"Not Persuasive"):**

> Observation.effective[x] is one of the critical information used for
> filtering results.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the EU Laboratory Report (FHIR) IG profile of `Observation` used for lab
results, relax `Observation.effective[x]` cardinality from `1..1` back to the
FHIR base `0..1` (keeping `mustSupport = true`). Add narrative guidance to
the profile stating that, when `Observation.effective[x]` is absent, the
clinically relevant time SHOULD be derived from the linked `Specimen`
following a defined fallback chain
(`Specimen.collection.collected[x]` → `Specimen.receivedTime` →
`Specimen.processing.time` → `Specimen.meta.lastUpdated` /
creation datetime). Document that for grouper `Observation`s referencing
multiple specimens (e.g., glucose tolerance test), `Observation.effective[x]`
MAY be omitted in favour of per-component / per-member specimen times.

#### Justification

Aligns the IG with the FHIR base cardinality and removes a structural
duplication objection raised by NHSE. The ticket's later comments show the
reporter is willing to populate `Observation.effective[x]` defensively when a
specimen time is available, so loosening cardinality does not preclude
populating it — it only avoids forcing a `data-absent-reason` extension or a
fabricated value. This is also a better fit for grouper Observations whose
constituent specimens may have multiple collection times.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Observation.effective[x]` at `1..1` `mustSupport`, but add an explicit,
normative population rule to the IG: producers MUST set
`Observation.effective[x]` from the linked `Specimen` using the fallback
chain `Specimen.collection.collected[x]` → `Specimen.receivedTime` →
`Specimen.processing.time` → `Specimen` creation datetime, and SHOULD use the
earliest applicable specimen collection time when a grouper Observation
references multiple specimens. Constrain the
`data-absent-reason` extension on `Observation.effective[x]` to `0..0` (as
the reporter himself proposed in the 2026-04-20 comment), so that a real
value is always sent. Optionally type-slice the `effective[x]` choice down to
`dateTime` and `Period` to give grouper observations a way to express a
range.

#### Justification

This preserves the IG's ability to filter / search results by date — which
the proposed resolution cites as the reason for the mandate — while
addressing the reporter's underlying concern that the field is sometimes
hard to populate. Codifying the derivation rule moves the work from each
implementer's local convention into the IG, removing the duplication concern
without weakening cardinality. It also closes the "no value at all"
escape-hatch via the data-absent-reason extension, giving consumers a
stronger guarantee than the FHIR base spec. The `Period` typing
accommodates the dynamic function test case raised in the 2026-04-09 comment
without changing cardinality.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Maintain `Observation.effective[x]` at `1..1` `mustSupport` in the EU
Laboratory Report (FHIR) IG and resolve the ticket as **Not Persuasive**, as
already drafted. Keep the resolution rationale that
`Observation.effective[x]` is a critical search/filter axis for lab results
and that the Observation-side time is the canonical "when did this result
apply" answer for downstream consumers, regardless of whether the source
data lives on the `Specimen`. Note in the resolution that producers can use
specimen-side data to populate it.

#### Justification

`Observation.effective[x]` is a base FHIR search parameter (`date`) and the
primary axis on which clinical systems query and timeline-render lab
results. Forcing every conformant resource to carry it makes consumer
behaviour predictable across all profiles in the IG and removes the need for
each consumer to walk to `Specimen` and apply a per-system fallback chain.
The reporter himself accepted the rationale in the 2026-04-14 comment
("happy with the resolution description, thanks"), which weakens the case
for change. Duplication of data between `Observation.effective[x]` and
`Specimen.collection.collected[x]` is intentional in the FHIR result/specimen
pattern and is not, by itself, a reason to relax a cardinality constraint
that drives query behaviour.

---

### Recommendation

**Recommended disposition:** C

The reporter explicitly accepted the proposed "Not Persuasive" resolution on
2026-04-14 once Giorgio's reasoning was clarified, and the only outstanding
substantive concern in the thread is whether `data-absent-reason` should be
locked out — which is a separate, narrower change that can be tracked as a
follow-up if HL7 Europe wants to harden the constraint. `Observation.effective[x]`
is the canonical search/filter axis for lab results, and the EU Laboratory
Report IG's mandate aligns with how downstream consumers actually query the
data. The duplication concern is real but inherent to the FHIR
Observation/Specimen pattern, not specific to this IG. If the workgroup
wants to give the reporter additional comfort beyond decline, the cleanest
follow-up is to fold the population-rule guidance from Disposition B (and the
optional `data-absent-reason 0..0` constraint) into the profile narrative
without changing cardinality — but the ticket itself should resolve as Not
Persuasive.
