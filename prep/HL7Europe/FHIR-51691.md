## Summary

The ticket reports a broken/obsolete binding on `Device.type` in the HL7
Europe Hospital Discharge Report (HDR) IG. `Device.type` is currently
bound (preferred) to the HDR-local ValueSet
`medical-devices-snomed-absent-unknown-eu-hdr`, which composes "all
SNOMED Devices" with the IPS ValueSet
`absent-or-unknown-devices-uv-ips`. The reporter notes that the
`absent-or-unknown-devices-uv-ips` URL 404s (the IPS team renamed /
restructured the absent/unknown content for IPS 2.0), so the HDR
ValueSet expansion is effectively broken.

The reporter proposes that HDR re-bind `Device.type` directly to the
IPS 2.0 ValueSet `medical-devices-uv-ips`
(`https://build.fhir.org/ig/HL7/fhir-ips/ValueSet-medical-devices-uv-ips.html`),
which is the canonical IPS binding for `Device.type` and which now
already includes the SNOMED Devices hierarchy plus
`787483001 |No known device use|`. Adopting that ValueSet would let
HDR drop its bespoke wrapper ValueSet and stay aligned with IPS going
forward.

No cross-referenced Jira tickets were returned by the FHIR Augury
cross-reference index, so the request is being treated as a stand-alone
HDR technical correction.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the EU Hospital Discharge Report IG, change the binding on
`Device.type` (in the HDR Device profile) from the HDR-local ValueSet
`medical-devices-snomed-absent-unknown-eu-hdr` to the IPS 2.0
ValueSet `medical-devices-uv-ips`
(`http://hl7.org/fhir/uv/ips/ValueSet/medical-devices-uv-ips`),
keeping strength = `preferred`. Retire (mark `status = retired`) the
now-unused HDR ValueSet `medical-devices-snomed-absent-unknown-eu-hdr`
and remove the dangling reference to the non-existent
`absent-or-unknown-devices-uv-ips` ValueSet. Update the IG narrative
on the Device page to point at the IPS ValueSet, and re-run the IG
publisher / terminology validation to confirm the expansion now
resolves and includes `787483001 |No known device use|`.

#### Justification

This is exactly what the reporter asks for and what IPS 2.0 already
defines: `medical-devices-uv-ips` already composes the SNOMED Devices
hierarchy plus the absent/unknown sentinel `787483001`, so HDR no
longer needs its own wrapper ValueSet to add the absent-unknown code.
Re-binding directly to the IPS ValueSet keeps HDR aligned with IPS 2.0
(an explicit goal of the EU base / discharge report family), removes a
broken outbound link (the 404 on
`absent-or-unknown-devices-uv-ips`), and reduces the number of
HDR-specific terminology artifacts that EU implementers and the
HL7 Europe terminology team have to maintain. As a Technical
Correction against an HDR publication this is in-scope for HL7 Europe
and does not require coordination with IPS beyond pinning the IPS
version.

---

### Disposition B: Alternative Approach

#### Proposal

Keep an HDR-local wrapper ValueSet but rebuild it correctly: rename /
republish `medical-devices-snomed-absent-unknown-eu-hdr` so that it
composes (a) the SNOMED CT Devices hierarchy and (b) the IPS
`medical-devices-uv-ips` ValueSet (or, equivalently, the explicit code
`787483001 |No known device use|`), eliminating the reference to the
non-existent `absent-or-unknown-devices-uv-ips`. `Device.type` in HDR
remains bound to the HDR-local ValueSet (preferred). Optionally pin
the IPS dependency to IPS 2.0 in the IG manifest so the composition is
reproducible.

#### Justification

This preserves HDR's existing canonical URL for the binding, which
means existing HDR implementations and downstream profiles that
already reference
`medical-devices-snomed-absent-unknown-eu-hdr` do not have to update.
It also gives HDR a stable insertion point if HL7 Europe later wants
to add EU-specific device codes (e.g., regional medical-device
identifiers) without changing the binding again. The trade-off is one
extra HDR-maintained ValueSet whose content essentially duplicates
`medical-devices-uv-ips` today, plus an ongoing maintenance cost to
track IPS revisions.

---

### Disposition C: Decline

#### Proposal

Decline the change. Leave the `Device.type` binding as-is and instead
fix only the broken outbound link in the narrative (e.g., point the
narrative at the IPS 2.0 documentation), on the grounds that the
binding strength is `preferred` and implementers are not strictly
required to use the bound ValueSet.

#### Justification

A `preferred` binding is non-normative, so a 404 in the *composition*
of the bound ValueSet does not block conformance; implementations are
free to use any code system. Declining avoids any republication churn
on HDR. However, this is hard to defend: the bound ValueSet's
expansion is genuinely broken (the include points at a ValueSet that
no longer exists), which means tooling that auto-expands HDR's
terminology will fail and HDR's published terminology page is
misleading. Decline is listed for completeness, not as a
recommendation.

---

### Recommendation

**Recommended disposition:** A

The reporter's analysis is correct: IPS 2.0's `medical-devices-uv-ips`
already covers HDR's intended value set (SNOMED Devices plus the
absent/unknown sentinel `787483001`), so HDR's wrapper ValueSet is now
strictly redundant *and* contains a broken include. Disposition A
removes the breakage, deletes a redundant artifact, and keeps HDR
aligned with the IPS family that HDR is explicitly built on top of —
all desirable outcomes for a Technical Correction. Disposition B is a
reasonable fallback if the workgroup wants to preserve the existing
HDR canonical URL for backwards compatibility with downstream EU
profiles, but it pays an ongoing maintenance cost for no current
content benefit. Disposition C is not defensible because the bound
ValueSet's expansion is materially broken today; the workgroup should
not ship an HDR errata that leaves a 404 in a published binding.
