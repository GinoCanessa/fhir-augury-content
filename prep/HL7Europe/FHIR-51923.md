## Summary

The ticket asks the HL7 Europe Hospital Discharge Report (HDR) IG to swap
its underlying logical model: replace the current eHN guidelines-based
model with the Xt-EHR Hospital Discharge Report logical model produced
under the EU Xt-EHR project (deliverable D7.3). Concretely, the
specification should (a) substitute the eHN model with the Xt-EHR model
as the normative source for HDR content, (b) retain a page that
describes the eHN guidelines and how they relate to the Xt-EHR model,
(c) add a new page introducing the Xt-EHR model, (d) revise the
model-to-FHIR map so it derives from the Xt-EHR model, and (e) update
the existing profiles to align with the new model where needed. The
reporter (Giorgio Cangioli) acknowledges in the original comment that
the change may need to be split into more granular change requests
depending on the size/shape of the resulting diffs. The ticket has
since been resolved as **Persuasive**, with the team adopting Xt-EHR
v0.3.0 and dropping a set of sections that are no longer used in the
Xt-EHR model (see Details).

No related Jira tickets surface through cross-references; the related
Zulip discussion in *EHDS - European Health Data Space › D7.3
(Hospital) Discharge Report* corroborates the rationale and reflects
the broader Xt-EHR / EHDS context that motivates the swap.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the eHN guidelines logical model with the Xt-EHR Hospital
Discharge Report logical model (v0.3.0) as the normative model
underlying the EU HDR IG, and apply the four documentation/structural
follow-ons in the same change:

1. Add a new IG page titled (e.g.) "Xt-EHR Hospital Discharge Report
   logical model" that introduces the Xt-EHR model, its provenance
   (Xt-EHR project, D7.3 deliverable), and its scope.
2. Retain (and rename if needed) the existing eHN guidelines page so
   that it (a) describes the eHN guidelines and (b) explicitly maps
   each eHN concept to its Xt-EHR counterpart, including the dropped
   sections listed in the 2026-02-09 comment
   (`sectionInfectiousContacts`, `sectionTravelHx`,
   `sectionFamilyHistory`, `sectionSocialHistory`,
   `sectionSubstanceUse`, `sectionAlcoholUse`, `sectionTobaccoUse`,
   `sectionDrugUse`, `sectionAdvanceDirectives`,
   `sectionCareTeamRules`, `sectionPayers`, `sectionPastIllnessHx`,
   `sectionHistoryMedicalDevices`, `sectionProceduresHx`).
3. Rebuild the "model map" (logical-model → FHIR profile mapping)
   against the Xt-EHR model so that every Xt-EHR data element either
   maps to a slice/element on `EHDSDischargeReport` (or its referenced
   profiles) or is explicitly marked as out of scope.
4. Revise the existing Composition profile (`EHDSDischargeReport`) and
   its referenced section/entry profiles to:
   - remove the 14 unused sections enumerated above from the section
     slicing,
   - re-align element cardinality/binding/definition text to the
     Xt-EHR model wording,
   - update Composition section codes only where Xt-EHR mandates a
     different code system or LOINC code.

#### Justification

The ticket is already resolved Persuasive with "proceed as requested"
and has been partially executed (Xt-EHR v0.3.0 adopted, section list
pruned). The Zulip thread shows the underlying numbering/labelling
inconsistencies between the IG and the Xt-EHR D7.3 deliverable were
real and confusing reviewers, and that no participant pushed back on
moving to Xt-EHR. Carrying the eHN-vs-Xt-EHR mapping page forward
preserves traceability for implementers who built against the eHN
view. Doing all four sub-tasks in one change keeps the IG, model, and
profiles internally consistent; doing them piecemeal would leave the
build in a half-migrated state.

---

### Disposition B: Alternative Approach — Adopt the Model, Split the Profile/Map Work into Sub-Tickets

#### Proposal

Accept the model substitution and the two documentation pages (intro
to Xt-EHR; eHN-relationship page) under FHIR-51923 itself, but split
the "revise the model map" and "revise the profiles as needed" steps
into separate, scoped child change requests — one per affected section
profile (or per logically grouped cluster of sections), filed against
the same EU HDR specification. Each child ticket would cite the
relevant Xt-EHR element(s) and call out exactly which slice / element
/ binding changes. The 14 dropped sections would be handled by a
single dedicated removal ticket that documents the removal rationale
and any migration notes for downstream implementers.

#### Justification

This is exactly the path the reporter himself flagged in his Aug 2025
comment ("Depending on the numbers and types of changes this item
should be split into several more specific change requests"). It
trades a slightly heavier ticket-management overhead for far better
diff readability, easier balloter review, and cleaner per-change audit
trails — important because the IG is in the EHDS / HL7 Europe ballot
pipeline and reviewers (per the Zulip thread) are actively comparing
the IG and the D7.3 PDF section-by-section. The HDR repo is not in
the local repo-analysis cache, so the workgroup is in any case better
served by smaller, repo-confirmed changes than by one sweeping commit.

---

### Disposition C: Decline

#### Proposal

Decline the model substitution; keep the eHN guidelines as the
underlying logical model for the EU HDR IG and instead publish only an
informative page describing how the Xt-EHR model relates to the
existing eHN-derived content.

#### Justification

A defensible-but-weak rationale would be: the IG is already in
ballot/use against the eHN model, and switching the underlying
logical model is a conceptually breaking change to documentation,
mappings, and profile derivation that risks invalidating reviewer
work mid-cycle. However, this is undermined by the actual evidence:
(a) the ticket is already resolved Persuasive and partially
implemented, (b) the EU policy direction (Xt-EHR D7.3 / EHDS) is
moving toward Xt-EHR as the canonical model, and (c) the Zulip thread
shows that not switching is what is currently *causing* reviewer
confusion. Decline is therefore difficult to justify on its merits
and is included primarily for completeness.

---

### Recommendation

**Recommended disposition:** **B — Adopt the model, split profile/map work into sub-tickets.**

Disposition B aligns with where the work has actually landed (Xt-EHR
v0.3.0 adopted, 14 sections dropped) while honouring the reporter's
own up-front guidance that profile/map revisions should be tracked as
discrete change requests. Splitting the long tail of profile edits
makes each diff reviewable in isolation, gives EHDS / HL7 Europe
balloters a clear per-section audit trail (which the Zulip thread
shows they want — they are already comparing the IG against the
Xt-EHR PDF section-by-section), and avoids one oversized PR landing
in a repo whose authoring conventions cannot be verified here against
a `repo-analysis` briefing. Disposition A would close out the work in
a single sweep but at the cost of review clarity; Disposition C is
not defensible given the resolution and the EU policy direction. If
the workgroup prefers A for scheduling reasons, the documentation
pages and the section removals should still be staged as separate
commits within that single ticket so that the model swap and the
profile changes remain individually reviewable.
