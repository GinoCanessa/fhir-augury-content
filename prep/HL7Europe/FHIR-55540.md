## Summary

The ticket is a ballot-time **Question** raised against the EU Base and Core
(R4) profile of `DiagnosticReport`. The reporter notes that on the German
national level there have been requests to introduce a *secondary status*
mechanism on `DiagnosticReport`, so that practitioner EHR systems whose
internal state machines do not align with FHIR's nested `DiagnosticReport`
status codes (e.g. `partial`, `preliminary`, `final`, `amended`,
`corrected`, `appended`, `cancelled`, `entered-in-error`) can still emit
those codes without losing information from their own coarser-grained
workflow states. The German MIO lab profile already exposes this as the
`SekundaerStatus` extension on `KBV_PR_MIO_LAB_DiagnosticReport` (see
referenced Simplifier link).

The reporter asks whether (a) similar discussions have been observed on the
European or other member-state levels, and (b) whether reviewing the
`DiagnosticReport` status logic and value set should be taken up at the
European (EU Base & Core) level for the next revision.

This ticket has **no formal cross-references** in Augury (Jira links,
Zulip pointers, GitHub commits/PRs that mention `FHIR-55540`). It does,
however, sit in a tight cluster of EU-ballot-Jan-2026 `DiagnosticReport`
tickets from the same reporter — see
[FHIR-55539](https://jira.hl7.org/browse/FHIR-55539) (top-level extensions
for "pseudonymized" / "critical result" flags),
[FHIR-55541](https://jira.hl7.org/browse/FHIR-55541) (max-cardinality on
`DiagnosticReport.category` slice, Applied / Persuasive with Modification),
[FHIR-55542](https://jira.hl7.org/browse/FHIR-55542) (referenced-profile

fine-grained workflow state German systems struggle to map cleanly today.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level extension to the EU Base and Core (R4)
`DiagnosticReport` profile, named (working title) `secondaryStatus`, with
the following shape:

- `url`: `http://hl7.eu/fhir/base/StructureDefinition/diagnosticreport-secondaryStatus`
- `value[x]`: `Coding` (or `CodeableConcept`) bound to a new EU value set
  `DiagnosticReportSecondaryStatus` with binding strength **example**.
- Cardinality `0..1` on `DiagnosticReport`.
- Definition: "Implementer-/jurisdiction-defined coarse workflow state
  from the source EHR. Provided alongside `DiagnosticReport.status` so
  that systems whose internal state machine does not map 1-to-1 onto
  the FHIR nested status codes can still emit a precise nested
  `DiagnosticReport.status` without losing the source workflow state."

The example value set seeds entries from the German MIO Lab
`SekundaerStatus` (so existing KBV implementations remain conformant by
re-pointing the URL) and is explicitly open for member-state extension.
Add narrative guidance to the `DiagnosticReport` profile page describing
when to populate the extension and reaffirming that
`DiagnosticReport.status` remains the normative source of truth.

#### Justification

This addresses the implementer pain reported on the German level (and
plausibly elsewhere — see FHIR-53859 for the lab-retraction example)
without changing the `DiagnosticReport.status` value set itself, which
is FMM-mature, normative-tracked, and used pan-FHIR. Layering an
extension keeps the change **compatible** and **enhancement-only**.
The pattern is well established (extensions hold jurisdiction-specific
workflow nuance; core elements stay portable). Standardizing the
extension at EU Base and Core — rather than letting each member state
mint its own — also enables cross-border interpretation, which is one of
the EU Base and Core IG's stated goals.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add the extension to EU Base and Core. Instead:

1. In EU Base and Core, add **narrative guidance only** under the
   `DiagnosticReport.status` element documenting the known
   EHR-state-machine gap, listing recommended mappings from common
   coarse-grained EHR workflow states (draft/in-progress/done/withdrawn)
   to the FHIR nested codes, and pointing implementers at the existing
   FHIR R4 mechanism `Provenance` (with `Provenance.activity` /
   `.reason` / `.signature`) for capturing source-system workflow
   semantics.
2. Acknowledge `KBV_PR_MIO_LAB_DiagnosticReport.SekundaerStatus` as a
   conformant member-state extension (link from EU narrative) but do
   not redefine it at the EU level.
3. Refer the `secondaryStatus` extension itself to the **EU Laboratory
   Report IG** (where FHIR-53859 already lives) so that the extension is
   defined in the domain that demonstrably needs it, rather than in EU
   Base and Core.

#### Justification

This mirrors the precedent set by **FHIR-55457 / FHIR-51308**: HL7
Europe explicitly chose *not* to add the `AnatomicalRegion` extension
to the core `DiagnosticReport` profile, deferring to the imaging IG
(IHE RAD MADO). Keeping EU Base and Core "thin" — extensions live in
domain IGs — reduces the surface area of pan-EU profiles and avoids
EU Base and Core implicitly endorsing one EHR's workflow vocabulary.
Existing R4 mechanisms (`Provenance`, `DiagnosticReport.extension`
within member-state profiles) already cover the use case for
implementers who need it today, so the cost of "no core change" is low.
Risk: the extension proliferates unharmonized across member states; the
narrative guidance is the mitigation.

---

### Disposition C: Decline

#### Proposal

Close this ticket as a **Question – Answered, No Change** to EU Base
and Core. Document the answer:

- Yes, similar pain has been reported (FHIR-53859 on lab retraction).
- No, EU Base and Core will not introduce a `secondaryStatus` extension
  in this ballot cycle.
- Reasoning: the `DiagnosticReport.status` value set is sufficient for
  exchange; secondary internal workflow states are an EHR-internal
  concern and should be captured in jurisdiction- or organization-level
  profiles, not in pan-EU normative-track artifacts.

Encourage the German community to keep `KBV_PR_MIO_LAB_DiagnosticReport.SekundaerStatus`
as a national extension and to surface the use case to the international
Orders & Observations WG for FHIR R6 if it is judged to be a global
problem.

#### Justification

The ticket is filed as a Question, not as a change request, and the
reporter explicitly asks whether reviewing the status logic *should* be
considered. A defensible answer is "not at this level". Adding an
extension to EU Base and Core for a use case that has only been
demonstrated in one member state risks (a) standardizing a vocabulary
prematurely, (b) inflating the EU Base and Core surface area, and
(c) confusing implementers about which status field is authoritative.
FHIR-55457 / FHIR-51308 established that HL7 Europe is willing to say
"this belongs in the domain IG, not in core". The same logic applies
here.

---

### Recommendation

**Recommended disposition: B (Alternative Approach)**

Disposition B threads the needle between the reporter's legitimate
implementer concern and HL7 Europe's recently demonstrated reluctance
to absorb domain-specific extensions into EU Base and Core (FHIR-55457,
FHIR-51308). It validates the use case, gives implementers concrete
guidance today (narrative + Provenance pattern + acknowledgement of the
KBV extension), and parks the actual extension definition in the EU
Laboratory Report IG — which is the domain where the use case
demonstrably bites (see FHIR-53859) and where the same workgroup can
own the value set without committing pan-EU `DiagnosticReport`
implementers to it. Disposition A is also reasonable but creates a new
EU-wide extension on a thinly-evidenced single-member-state need;
Disposition C is the most "correct" against precedent but provides no
constructive path for the German implementers who raised the question.
B captures most of the value of A at most of the cost of C, and is the
disposition the workgroup should debate first.
