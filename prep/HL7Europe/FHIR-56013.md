## Summary

The reporter (HynekKruzik) asks the HL7 Europe Laboratory Report
project to tighten the LOINC value set used for the laboratory test
code element. As specified today, the value set is built solely from
the LOINC `CLASSTYPE = 1` filter (laboratory codes), but that filter
still admits two groups of LOINC codes that are not appropriate for
reporting an observed lab result:

1. LOINC codes whose `order_observation` attribute is `ORDER` (i.e.,
   they describe an order/request, not an observable result), and
2. LOINC codes in `CLASS = LP94892-4 (LABORDERS)`.

The reporter recommends excluding both categories so that the value
set bound to the lab result code element only contains codes meant
to carry observed results. The ticket has already been triaged with
a resolution of *Persuasive with Modification*: the project will
restrict the value set to `CLASSTYPE = 1` minus `CLASS = LABORDERS`,
but will **not** filter on `order_observation`, because that LOINC
attribute is not usable in a FHIR `ValueSet` filter expression.

## Details

**Description:**

Current specification of the value set is broad: where
`CLASSTYPE = 1`, this includes also test codes with
`order_observation` element = `ORDER` and those where
`CLASS = LP94892-4` (LABORDERS). I recommend to exclude those two
categories from the value set.

**Resolution Description (already triaged):**

Refine the ValueSet by keeping `CLASSTYPE = 1` and excluding codes
with `CLASS = LABORDERS`. The `order_observation` attribute cannot
be used for filtering in FHIR.

## Proposed Dispositions

Note: the cross-reference index returned no GitHub items for this
ticket, so no per-repo briefing was loaded. The dispositions below
name the affected ValueSet conceptually rather than a specific
authoring path; the work group will know the project's resource
file under the `hl7.fhir.eu.laboratory` IG.

### Disposition A: Accept As Requested

#### Proposal

Modify the lab test code ValueSet in the EU Laboratory Report IG
so that its `compose` includes only LOINC codes meeting **both** of
the following filters:

- `CLASSTYPE = 1` (laboratory class), AND
- `order_observation` ≠ `ORDER` (i.e., keep `OBSERVATION` and
  `BOTH`),

and additionally **excludes** any code where
`CLASS = LP94892-4` (LABORDERS), e.g. via a second `include` /
`exclude` block.

Rebind the lab result code element (the `Observation.code` slice
in the EU Lab Report Observation profile) to this refined
ValueSet, regenerate expansion snapshots, and update the IG
narrative to call out that the value set now scopes to observable
lab results only.

#### Justification

This is exactly what the reporter asked for and aligns with the
intent of `Observation.code` in a result-bearing resource: the
codes should describe what was observed, not what was ordered. The
LABORDERS class explicitly groups order-only LOINCs, and the
`order_observation = ORDER` partition is LOINC's own marker for
"this code is intended for orders, not results." Excluding both
removes a class of false positives where implementers could
legitimately bind an order-only code to a result.

### Disposition B: Alternative Approach (the triaged resolution)

#### Proposal

Refine the ValueSet by keeping `CLASSTYPE = 1` and **excluding
codes with `CLASS = LABORDERS`**, but do **not** add an
`order_observation` filter. Instead:

- In the IG narrative on the lab result code element, add usage
  guidance that implementers SHOULD select LOINC codes whose
  `order_observation` attribute is `OBSERVATION` or `BOTH`, not
  `ORDER`.
- Optionally, raise a LOINC-side ask (or a FHIR terminology
  ticket) to expose `order_observation` as a queryable filter
  property on the LOINC `CodeSystem` resource, so a future version
  of the IG can tighten the binding.

#### Justification

This matches the disposition already on the ticket
(*Persuasive with Modification*). The `order_observation`
attribute is part of the LOINC distribution but is not exposed as
a `filter` property in the FHIR LOINC `CodeSystem` resource, which
means a `ValueSet.compose.include.filter` of
`property = order_observation, op = ≠, value = ORDER` will not
expand reliably across terminology servers — an `$expand` against
a server that does not recognize the property would either fail
or silently return the unfiltered set. Dropping LABORDERS via
`CLASS` (which **is** a recognized LOINC property) gives most of
the value the reporter is after, while narrative guidance covers
the residual `ORDER`-only codes without producing a value set
that breaks at expansion time.

### Disposition C: Decline

#### Proposal

Leave the value set as currently specified (`CLASSTYPE = 1`
only) and add no narrative restriction; close the ticket as
*Not Persuasive*.

#### Justification

The current binding is the simplest expression of "all LOINC
laboratory codes," and any tightening risks excluding codes that
some implementers legitimately use today. If the work group
believes that (a) order-only LOINC codes appearing in a result
slot is an acceptable implementer responsibility rather than a
spec defect, and (b) the cost of expanding a narrower value set
(extra filter complexity, risk of expansion mismatches across
terminology servers — especially given LOINC properties like
`order_observation` not being uniformly exposed) outweighs the
benefit, then declining is defensible. This option is included
for completeness; it is not the recommended path.

---

### Recommendation

**Recommended disposition:** B (the triaged resolution).

Disposition B is the right balance and matches what the work
group has already triaged for this ticket. The reporter's
underlying concern is real — order-only LOINC codes do not
belong in a value set bound to an *observed* lab result code —
but Disposition A overstates what FHIR's `ValueSet.compose`
machinery can actually enforce against the LOINC `CodeSystem` as
published. The `order_observation` attribute is not a recognized
LOINC `CodeSystem.property`, so a filter referencing it will not
expand consistently across terminology servers and is likely to
produce surprising or empty expansions in implementer tooling.
Removing the LABORDERS class via the standard `CLASS` property
delivers the bulk of the cleanup safely, and pairing it with
narrative usage guidance (and, optionally, a terminology-side
ask to expose `order_observation`) preserves a clear migration
path if LOINC/THO later expose the attribute as a usable filter.
Disposition C is not recommended because it ignores a concrete
and easily-fixable defect in the current binding.
