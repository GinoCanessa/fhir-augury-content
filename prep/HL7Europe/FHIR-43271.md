## Summary

The reporter (Ulrike Merrick) is asking why the `specimen` slice on `Bundle.entry`
in the EU Laboratory Report Bundle profile (`Bundle-eu-lab-definitions`) is
defined with cardinality `0..*` rather than `1..*`. Her intuition is that a
laboratory report should always carry at least one Specimen resource, so a
mandatory minimum of one would seem natural. A follow-up comment (rikim) raises
the same question for `DiagnosticReport.specimen` in the EU Lab
`DiagnosticReport-eu-lab-definitions` profile, which is also left optional.

The ticket has already been resolved by HL7 Europe as **"Considered - Question
answered"**, with the rationale that none of the constituent EU Lab profiles
require a Specimen, and in some workflows the specimen is implicit in the test
code (e.g., the analyte/method code itself implies the specimen type, so a
distinct Specimen resource is not always needed).

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items —
only artifact mentions of `Bundle.entry` and `DiagnosticReport.specimen`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the cardinality of the `specimen` slice on `Bundle.entry` in
`Bundle-eu-lab-definitions` from `0..*` to `1..*`, and correspondingly tighten
`DiagnosticReport.specimen` in `DiagnosticReport-eu-lab-definitions` from `0..*`
to `1..*`. Where the slice tightens the entry, also consider adding an
invariant requiring that any referenced `DiagnosticReport.specimen` resolve
to a Specimen resource present as a `Bundle.entry:specimen`. Update the IG
narrative to state that every EU Lab Report SHALL carry at least one Specimen.

#### Justification

This matches the reporter's clinical intuition that a laboratory report is
fundamentally about analysing a specimen, so making Specimen mandatory enforces
internal consistency and improves data quality for downstream consumers
(analytics, registries, cross-border exchange). It also aligns the EU Lab Bundle
with patterns used in other lab IGs that profile Specimen as required.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the base slice cardinalities at `0..*` on the generic
`Bundle-eu-lab-definitions` and `DiagnosticReport-eu-lab-definitions` profiles,
but introduce (or strengthen) more specific derived profiles / use-case bindings
where Specimen genuinely is required (e.g., histopathology, microbiology,
specimen-driven workflows) and constrain those derived profiles to `1..*`.
Document the rule in the IG: when the test code does not imply the specimen,
the derived profile or implementation guide built on top of EU Lab MUST require
at least one Specimen entry. Optionally add a `dom-`/IG-level invariant that
checks "if `DiagnosticReport.specimen` is populated, the referenced Specimen
SHALL be present in the Bundle".

#### Justification

This preserves the answered rationale — "in some cases the specimen is implicit
in the test code" (e.g., point-of-care, capillary glucose, whole-body imaging
codes that intrinsically imply the specimen) — while still giving stricter
realms and use cases a path to mandate Specimen. It avoids breaking existing
implementers whose lab reports legitimately omit a separate Specimen resource,
and pushes the constraint to the layer where the requirement actually applies.
This is also more consistent with how generic IG-layer profiles in HL7 Europe
typically defer realm- or use-case-specific tightening to derived profiles.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the existing `0..*` cardinality on both
`Bundle.entry:specimen` and `DiagnosticReport.specimen`. Treat the ticket as
closed via "Considered - Question answered" (its current resolution) and, if
helpful, add a short clarifying note to the IG narrative explaining when a
Specimen resource is and is not expected (e.g., "Specimen MAY be omitted when
the specimen is implicit in the test code or workflow").

#### Justification

The question has already been resolved in this sense. None of the constituent
EU Lab profiles require Specimen, and there are well-known lab scenarios where
forcing a Specimen resource adds no information (the specimen type is fully
encoded in the test code, e.g., capillary blood glucose, whole-blood point-of-care
tests, breath tests, in-vivo imaging-style measurements). Tightening to `1..*`
would be a breaking change for current implementers without a documented
clinical safety driver and would force creation of placeholder Specimen
resources that carry no useful additional structured data.

---

### Recommendation

**Recommended disposition:** C

The work group has already resolved this ticket as "Considered - Question
answered", and the published rationale is sound: at the generic EU Lab IG
layer, none of the in-scope profiles require Specimen, and there are real
laboratory workflows where Specimen is intentionally implicit in the test
code. Tightening cardinality to `1..*` at the base profile would be a breaking
change against current conformant content and would not actually improve data
quality in those implicit-specimen cases — it would just encourage stub
Specimen resources. The cleanest action is to confirm the existing closure
under Disposition C, optionally adding a short narrative paragraph in the EU
Lab IG that explains *when* a Specimen entry is expected. If a specific
sub-domain (e.g., histopathology, microbiology) needs Specimen mandated, that
is best handled as Disposition B in a derived profile rather than at the
generic Bundle/DiagnosticReport profile level.
