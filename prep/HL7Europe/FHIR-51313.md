## Summary

The reporter is reviewing the manifest obligations for `ImagingStudy` in the
HL7 Europe **EU Imaging Study Report (R5)** implementation guide and asks
whether `AppointmentResponse` (the ticket title) — or `AppointmentRequest`
(the term used in the description) — should remain a valid target type for
`ImagingStudy.basedOn`. The reporter is looking for a concrete example
workflow that would justify the reference; absent one, the implication is
that the target type should be removed from the IG's profile of
`ImagingStudy`. Note the title and description disagree on which type is
in question, which the workgroup will need to resolve before disposition.
No cross-referenced Jira tickets, GitHub items, or Zulip threads are
linked to this ticket in the index.

## Proposed Dispositions

Because no GitHub cross-references were surfaced, the dispositions below
are framed in terms of the EU Imaging Study Report (R5) IG profile of
`ImagingStudy.basedOn`. Any of them would be implemented in the HL7
Europe IG repository for the EU Imaging Study Report; the workgroup
should confirm the exact repo/path against the IG's source before
acting.

### Disposition A: Accept As Requested

#### Proposal

In the EU Imaging Study Report (R5) IG, tighten the profile on
`ImagingStudy.basedOn` to remove the disputed target type. Concretely:
edit the `ImagingStudy` profile (StructureDefinition) so that the
allowed `targetProfile` values on `ImagingStudy.basedOn` no longer
include `AppointmentResponse` (per the ticket title) — and, pending
clarification with the reporter, also `AppointmentRequest` if that is
what was actually meant in the description. Update the manifest /
obligations narrative for `ImagingStudy` so the removed target is not
listed. Bump the IG version per its usual change cadence and add a
release-note entry naming the removed target type.

#### Justification

`AppointmentResponse` is the *response* to a proposed appointment, not
an order or request that can drive an imaging procedure; in the EU IG's
constrained workflow, the imaging act is initiated by a `ServiceRequest`
(and sometimes an `Appointment` or `Task`), so listing
`AppointmentResponse` as a `basedOn` target invites confusion without a
demonstrated use case. Profiles in IGs are explicitly allowed to narrow
the target set of a reference relative to base FHIR (R5 Conformance
rules on `Reference.targetProfile`), so this is a low-risk, in-scope
constraint for an HL7 Europe IG. Removing an unused, confusing
reference target reduces implementer surface area and lines up with the
reporter's premise that no example workflow exists.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the target type in the profile, but add explicit guidance to the
EU Imaging Study Report IG that documents *when* an
`AppointmentResponse` (or `AppointmentRequest`) reference would apply.
Specifically: add a short usage note on the `ImagingStudy` profile page
describing the workflow (e.g., a confirmed appointment slot for the
imaging encounter being recorded as `basedOn`), and — if the workgroup
considers it weak but not wrong — mark the relevant `basedOn` slice as
`mustSupport = false` and add a `comment` on the element constraint to
discourage casual use without removing it.

#### Justification

Removing a reference target is technically a breaking change for any
implementer that already populates it (even if rare), and the EU IG
inherits the broad target list from base R5 ImagingStudy. If there is
*any* European workflow — e.g., scheduling-driven imaging where the
slot confirmation is the closest thing to a "request" the system has —
preserving the option while documenting it avoids interop regressions
and respects the principle of conservative profile narrowing. This
disposition also addresses the reporter's underlying ask (lack of
example) by supplying one rather than removing capability.

---

### Disposition C: Decline

#### Proposal

Decline the change and close as **Not Persuasive**. Leave the EU
Imaging Study Report (R5) profile of `ImagingStudy.basedOn` aligned
with the base R5 target list, on the grounds that (a) the ticket
conflates `AppointmentResponse` (title) with `AppointmentRequest`
(description) and so does not actually identify a concrete defect, and
(b) an IG removing a base-FHIR reference target without a documented
implementer complaint is a profile narrowing that should clear a higher
bar than "no example was offered."

#### Justification

The ticket as filed is a *question* ("Is there an example workflow…?"),
not a documented interoperability problem. Base R5 `ImagingStudy.basedOn`
intentionally lists a broad set of admin/workflow types, and the EU IG
inheriting that list is the default behavior; there is no reporter
evidence that any implementer is harmed. Declining preserves alignment
with core FHIR and avoids a (small) breaking change to the IG schema
for what may be a documentation gap rather than a profile bug. The
workgroup can invite the reporter to reopen with a concrete
implementer scenario if the question is more than rhetorical.

---

### Recommendation

**Recommended disposition:** A

The reporter's question is the kind of "is this target type doing any
work?" review that profile authors should welcome — IGs exist precisely
to narrow base-FHIR optionality to the workflows the IG actually
covers. For an *EU Imaging Study Report* IG, the imaging act is
overwhelmingly initiated by `ServiceRequest` (and, where scheduling
matters, `Appointment`), and `AppointmentResponse` does not represent
an order at all; carrying it forward unexamined is a footgun for
implementers parsing the manifest obligations. Disposition B is
attractive only if the workgroup can produce the missing example;
absent that, B amounts to documenting a target type nobody uses.
Disposition C is defensible procedurally (the title/description
mismatch is real) but should be paired with a request for the reporter
to clarify which type they meant — and the workgroup will likely end
up at A regardless once that mismatch is resolved. Recommend the
workgroup confirm with the reporter that the intended target is
`AppointmentResponse` (per the title), then narrow the profile per
Disposition A.
