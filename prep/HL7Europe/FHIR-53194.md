## Summary

Bas van den Heuvel asks the HL7 Europe Imaging Study Report project to refresh
the XtEHR logical-model mappings carried in its FHIR IG so they line up with the
QA-preview release of the XtEHR models (version 0.2.1) that has been published
on the new canonical `https://www.xt-ehr.eu/fhir/models/`. The request applies
to both the R4 and R5 variants of the EU Imaging Study Report IG. In addition
to refreshing the mapping content, the ticket asks the project to **stop
shipping local copies of the XtEHR models** in the IG repository and instead
reference the externally published canonical artifacts directly. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced for
this issue, so the context is limited to what the reporter wrote. The ticket
has already been triaged with a resolution of **Persuasive**, with a resolution
note stating the change "Is covered by the ballot release."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

For both the R4 and R5 EU Imaging Study Report IGs, update every XtEHR mapping
reference (mapping declarations on profiles such as `ImagingStudy`,
`DiagnosticReport`, and any associated `Observation`/`Composition` profiles, as
well as any narrative pages that reproduce the XtEHR model) to point to the
**XtEHR 0.2.1 QA-preview** logical models published under
`https://www.xt-ehr.eu/fhir/models/`. As part of the same change:

- Add an explicit `dependsOn` entry in `sushi-config.yaml` (or the equivalent
  IG manifest) for the XtEHR QA-preview package.
- Replace each in-repo copy of an XtEHR `StructureDefinition` /
  `Logical`/`LogicalModel` resource with a reference to the canonical URL on
  `xt-ehr.eu`, deleting the duplicate authoring file from the IG sources.
- Update mapping `targets` (e.g., `^mapping[xtehr].uri`) and any narrative
  cross-walk tables to use the new canonical and version 0.2.1 identifiers.
- Re-run the IG build for both R4 and R5 branches and confirm that QA does not
  flag any unresolved canonical references.

#### Justification

This is exactly the change the reporter (and the published resolution) asks
for, and it is consistent with HL7 IG hygiene: an IG should depend on the
authoritative publication of an external model rather than maintain a local
fork that can drift. The ticket is already resolved as **Persuasive** with the
note "Is covered by the ballot release," indicating that this work has been
folded into the current ballot cycle, so accepting "as requested" is the
disposition the project has effectively already chosen. Switching to a real
package dependency also gives downstream implementers a deterministic version
pin (0.2.1 QA preview) and makes the next XtEHR refresh a single
`sushi-config.yaml` edit.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the canonical-URL refresh to XtEHR 0.2.1 in the mapping declarations,
**but keep a thin, clearly-marked snapshot copy** of the XtEHR logical models
inside the EU Imaging Study Report IG repository (e.g., under
`input/resources/xtehr-snapshot/` with a README pointing at the upstream
canonical and the exact upstream commit/version). The snapshot would be marked
as informative-only and excluded from the published artifact list, with CI
that fails if the local copy diverges from the upstream package without an
accompanying version bump.

#### Justification

XtEHR 0.2.1 is explicitly a **QA preview**, not a normative or trial-use
publication. Depending on a moving QA preview as the IG's only source of truth
risks build-breaks every time XtEHR republishes the same canonical at a new
QA-preview revision, and it leaves the IG vulnerable to the upstream site
being temporarily unreachable during build. Keeping a pinned local snapshot
behind the canonical reference gives the project reproducible builds and an
audit trail of exactly what XtEHR content was current when each ballot was
published, while still presenting the canonical URL to implementers. This
trades a small amount of repo bloat for ballot-cycle stability.

---

### Disposition C: Decline

#### Proposal

Defer the canonical refresh until XtEHR publishes a non–QA-preview release
(i.e., a stable 0.2.x or 1.0 line at `https://www.xt-ehr.eu/fhir/models/`).
In the interim, leave the existing in-repo XtEHR mapping copies untouched and
mark the ticket as *Persuasive — Deferred*, reopening once a non-preview XtEHR
release is available.

#### Justification

QA-preview artifacts are, by definition, subject to breaking change without
notice. Pointing the EU Imaging Study Report IG at a QA-preview canonical for
its mapping targets means every IG ballot becomes contingent on the stability
of an unreleased XtEHR build, and any retraction or rename on the XtEHR side
would silently break the published mappings. Declining (with a deferral) keeps
the IG on known, reviewed mapping content until the upstream is on a stable
release line.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as **Persuasive** and the resolution note
states the work "Is covered by the ballot release," so the project has
effectively committed to this path; the workgroup's remaining job is to
validate that the change actually lands in both the R4 and R5 IGs. Accepting
as requested also matches standard HL7 practice — IGs should declare external
logical models as proper package dependencies rather than carrying drift-prone
copies. Disposition B's stability concern is real but is better mitigated by a
**version-pinned** dependency on the 0.2.1 QA-preview package (which
Disposition A already specifies) plus a short note in the IG that the
dependency tracks a QA-preview line, rather than by perpetuating a local fork.
Disposition C would leave the IG out of sync with the only currently published
XtEHR artifacts and contradicts the ballot-release decision already recorded
on this ticket.
