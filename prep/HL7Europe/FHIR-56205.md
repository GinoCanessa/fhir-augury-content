## Summary

The ticket, raised by HynekKruzik against the EU Laboratory Report (FHIR)
specification, proposes that the value set used for specimen container type
in the EU Laboratory Report IG be re-anchored on the SNOMED CT hierarchy
**434711009 |Specimen container (physical object)|** instead of the
currently-used **706041008 |Device for body fluid and tissue
collection/transfer/processing (physical object)|** hierarchy, in line with
guidance from SNOMED International. The reporter notes that the 434711009
hierarchy is currently incomplete, so the proposal also entails submitting
the missing concepts to SNOMED International for inclusion in a future
SNOMED CT release. The ticket was filed primarily as evidence of the
proposal; the reporter explicitly states that the change still needs to be
discussed with SNOMED International to align approaches before any concrete
spec change is made. The ticket has been triaged with a resolution of
"Considered for Future Use", reflecting the fact that the work depends on
external (SCT) coordination.

There are no cross-referenced Jira tickets associated with this issue.

### Disposition A: Accept As Requested

#### Proposal

Re-base the EU Laboratory Report IG's specimen container type value set on
the SNOMED CT **434711009 |Specimen container (physical object)|**
hierarchy, replacing the current binding to the **706041008 |Device for
body fluid and tissue collection/transfer/processing (physical object)|**
hierarchy. Concretely:

1. In the EU Laboratory Report IG, locate the value set bound to
   `Specimen.container.type` (and any container-type-related extensions or
   profiles in the IG) and update its compose definition to include
   descendants of `434711009` instead of (or in addition to, during a
   transition period) descendants of `706041008`.
2. Adjust the corresponding profile binding strength / documentation as
   needed so that implementers know the recommended hierarchy comes from
   SNOMED International guidance.
3. In parallel, prepare a SNOMED International content request enumerating
   the specific container concepts that are presently missing from the
   `434711009` hierarchy but are required for laboratory specimen
   container reporting in Europe, and submit it through the appropriate
   SCT submission channel.
4. Document the dependency on SCT acceptance in the IG (e.g., release
   note / known issue) so adopters understand which concepts are
   provisional.

#### Justification

This is what SNOMED International is recommending and what the reporter is
explicitly asking for. `434711009 |Specimen container (physical object)|`
is the semantically correct anchor for "what physical container the
specimen is in", whereas `706041008` describes devices used in
collection/transfer/processing more broadly and therefore mixes container
and non-container concepts. Aligning with SCT guidance also reduces the
risk that downstream tooling (terminology services, mapping projects)
treats the IG's binding as inconsistent with SNOMED's intended modelling.
The submission step closes the loop on the gap the reporter explicitly
calls out — that the hierarchy is currently incomplete — so accepting
"as requested" is only meaningful if the IG change and the SCT submission
are tracked together.

---

### Disposition B: Alternative Approach

#### Proposal

Stage the change rather than swap hierarchies in a single ballot cycle:

1. Define a new value set in the EU Laboratory Report IG that is composed
   from the union of descendants of `434711009 |Specimen container
   (physical object)|` **plus** an explicitly-enumerated set of currently
   used `706041008` descendants that have no equivalent under `434711009`
   yet. Bind `Specimen.container.type` to this transitional value set
   (extensible binding).
2. Open (or link to) the SCT submission for the missing container
   concepts and record the submission identifiers in the IG's value set
   description.
3. Once SCT has accepted and published the missing concepts, retire the
   `706041008` carve-outs from the value set in a subsequent IG release,
   converging on a pure-`434711009` binding.
4. Optionally, raise a sibling ticket on UV Core / international Specimen
   guidance to ensure the wider FHIR community's container-type value
   sets follow the same SCT direction.

#### Justification

The reporter explicitly flags that the `434711009` hierarchy is incomplete
today, and the resolution status ("Considered for Future Use") indicates
the workgroup is uncomfortable making the change without SCT alignment. A
hard cut to `434711009` (Disposition A) risks regressing implementer
coverage: containers that exist in production data but not yet in the
target hierarchy would lose their binding. Staging the change preserves
implementer coverage during the transition, gives the SCT submission time
to land, and avoids forcing the IG to ship a known-incomplete binding.
The trade-off is a slightly more complex value set in the interim and an
additional release cycle to clean up — both of which are acceptable in an
IG that is still maturing.

---

### Disposition C: Decline

#### Proposal

Decline to make any normative spec change at this time, on the grounds
that the request is contingent on an external (SNOMED International)
process. Close the ticket as "Considered for Future Use" (which is
effectively where it sits today) and instead:

- Add an editorial note in the EU Laboratory Report IG that the
  workgroup is aware of the SCT recommendation to prefer
  `434711009 |Specimen container (physical object)|` and intends to
  align once the hierarchy is sufficiently complete.
- Track the SCT submission and revisit the binding in a future ballot
  cycle when the prerequisites are met.

#### Justification

The reporter themselves frames the ticket as evidence of the proposal
and notes that "this proposal needs to be discussed with SCT to align
approach". Acting on it before SCT has agreed an approach risks the IG
diverging from SNOMED International's eventual model — exactly the
situation the original `706041008` binding is being criticised for.
Declining now (with an editorial note that records the intent) keeps
the IG terminologically defensible and avoids burning a normative
change on a target that is admittedly incomplete. This is consistent
with the existing resolution of "Considered for Future Use" and avoids
implementer churn.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the reporter's intent — moving toward
`434711009 |Specimen container (physical object)|` — without ignoring
the hard constraint they themselves identify, namely that the target
hierarchy is incomplete. A staged, transitional value set (B) preserves
coverage for implementers today, makes the alignment with SNOMED
International's direction visible in the IG, and creates a concrete
hook (the SCT submission) for converging on a clean binding in a later
release. Disposition A is correct in direction but premature given the
acknowledged content gap; Disposition C is honest about the SCT
dependency but leaves implementers with no signal in the spec that a
re-anchoring is coming, which makes the eventual transition more
disruptive. The workgroup should still confirm — before merging the
transitional binding — that the EU Laboratory Report IG's profile
binding strength for `Specimen.container.type` allows an extensible
binding, and that the SCT submission can be sponsored (likely via HL7
Europe / the ticket reporter) so that the convergence path in step 3
of Disposition B is realistic.
