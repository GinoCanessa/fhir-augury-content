## Summary

FHIR-51333 questions the obligation rules on the
`Manifest-ImImagingStudy` profile in the HL7 Europe Imaging Study Report
IG (v0.1.0-ballot). The profile currently makes both the IID viewer
endpoint and the WADO-RS retrieve endpoint optional. The reporter
(JoshPriebe, Epic) asks the working group to confirm that an
`ImagingStudy` SHOULD still appear in the manifest when no usable
endpoint is available â€” for example when a study exists but images are
not retrievable, or when PACS instances have been archived to nearline /
offline storage and so are temporarily unreachable. The argument is
clinical: even endpoint-less metadata that "the imaging happened" alerts
consumers to follow up out of band, and improves the completeness of the
manifest as a clinical record.

The first comment (the IG editor) confirms that no profile change is
strictly required because endpoints are already optional, but raises
three handling cases the IG should make explicit: (1) study done /
images unavailable, (2) external/cross-site studies a manifest provider
is aware of, and (3) images in long-term storage where DICOM uses
instance-availability `NEARLINE`/`OFFLINE` â€” possibly representable via
an `ImagingStudy.series.instance`-level status. Input from PACS vendors
and the FHIR Imaging WG is solicited. The second comment asks the
reporter to confirm the request is a documentation/clarification ask
rather than a structural profile change.

There are no recorded cross-references to this ticket in Augury (no
linked Jira issues, no Zulip threads, no GitHub PRs/issues/commits).
However, several closely-related Triaged HL7 Europe Imaging tickets
reported by the same author share the `manifest` label and form a
coherent ballot cluster around the same profile and endpoint mechanics:
[FHIR-51296](https://jira.hl7.org/browse/FHIR-51296) (clarify IID vs
WADO-RS expectations), [FHIR-51334](https://jira.hl7.org/browse/FHIR-51334)
(scope of WADO-RS endpoint compliance),
[FHIR-51338](https://jira.hl7.org/browse/FHIR-51338) (Key Image
DocumentReference using WADO-URI vs WADO-RS),
[FHIR-51388](https://jira.hl7.org/browse/FHIR-51388) (Manifest Bundle
type underspecified) and the already-applied
[FHIR-51164](https://jira.hl7.org/browse/FHIR-51164) (resource name /
title harmonisation). Disposition should be coordinated with these.

## Proposed Dispositions

The two comment threads on the ticket already converge on a key fact:
the profile **does not** need a structural change to *allow*
endpoint-less `ImagingStudy` entries â€” the cardinalities are already
optional. The disposition decision is therefore primarily about
**guidance, signalling and obligation language** in the
`Manifest-ImImagingStudy` profile and its companion narrative pages.

### Disposition A: Accept As Requested

#### Proposal

Update the HL7 Europe Imaging Study Report IG (`hl7-eu/imaging`) to:

1. **`Manifest-ImImagingStudy` profile** â€” add a SHALL-style obligation
   that `ImagingStudy` instances MUST be included in the manifest when
   the imaging event is known to the manifest provider, *even when*
   `ImagingStudy.endpoint` is empty or all referenced `Endpoint`
   resources have a non-active `Endpoint.status` (e.g., `suspended`,
   `error`, `off`, `entered-in-error`). Document this as an obligation
   on the Manifest Provider actor.
2. **Narrative page (Manifest obligations for ImImagingStudy)** â€” add a
   new section "Studies without retrievable images" that enumerates the
   three cases from the reporter's comment:
   - study performed, images permanently unavailable â†’ include
     `ImagingStudy` without `endpoint`;
   - manifest provider aware of an external study at another site â†’
     OPTIONAL re-expose with a note on provenance;
   - images temporarily nearline/offline â†’ include `ImagingStudy` and
     signal availability via DICOM-aligned status (see #3).
3. **Instance-level availability signalling** â€” adopt the DICOM
   `Instance Availability` concept (`ONLINE` / `NEARLINE` / `OFFLINE` /
   `UNAVAILABLE`) and bind it through one of:
   - a small extension `imagingstudy-instance-availability` on
     `ImagingStudy.series.instance`, with a CodeSystem mirroring the
     DICOM tag (0008,0056) values; **or**
   - reuse `Endpoint.status` semantics on the linked `Endpoint` and
     document the mapping (less precise â€” `Endpoint.status` is endpoint
     state, not per-instance availability).
4. **Examples** â€” add at least one example bundle showing an
   `ImagingStudy` with no `endpoint` and a NEARLINE instance.

Implementation lives entirely in the `hl7-eu/imaging` IG: under
`input/fsh/` for the `Manifest-ImImagingStudy` profile and any new
extension/CodeSystem, and under `input/pagecontent/` (or equivalent IG
Publisher source) for the new narrative section and examples. Do not
hand-edit anything generated under `output/`.

#### Justification

This addresses every concrete handling case the reporter raised, and
matches the comment thread's own conclusion that the gap is one of
explicit guidance rather than profile cardinality. The DICOM Instance
Availability vocabulary (0008,0056) is the established, vendor-neutral
way PACS communicate "the bytes aren't here right now" â€” borrowing it
through a small extension keeps the IG aligned with the source-system
behaviour PACS vendors already implement, which is exactly the audience
the reporter wants to hear from. Constraining at the
`ImagingStudy.series.instance` level (rather than reusing
`Endpoint.status`) is more faithful to where the availability state
actually lives. The change is additive, optional to populate, and does
not break v0.1.0-ballot conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-51333 as a **documentation-and-conformance-statement-only**
change in the same IG, **without** introducing a new extension or
CodeSystem:

1. Leave `Manifest-ImImagingStudy` cardinalities unchanged.
2. Add a single normative paragraph in the Manifest obligations page
   stating that the Manifest Provider SHALL include
   `ImagingStudy` for any imaging event known to the provider regardless
   of endpoint availability, and SHOULD omit `ImagingStudy.endpoint`
   when no retrieval mechanism is currently usable.
3. For temporarily-unavailable images, recommend reuse of the existing
   `Endpoint.status` value set (`active` | `suspended` | `error` | `off`
   | `entered-in-error` | `test`) on the linked `Endpoint` resource, and
   document the mapping `NEARLINE â†’ suspended`, `OFFLINE â†’ off`,
   `UNAVAILABLE â†’ error`.
4. Defer per-instance availability signalling to a future EU Imaging
   release, pending a base-spec change request (in the FHIR Imaging WG)
   for an `ImagingStudy.series.instance.availability` element â€” and
   raise that as a separate Jira ticket cross-linked from FHIR-51333.

#### Justification

Coordinating IID/WADO endpoint expectations is already the subject of
FHIR-51296 and FHIR-51334; layering a new extension and CodeSystem on
top of an IG that is still at v0.1.0-ballot risks shipping a parallel
vocabulary that the FHIR Imaging WG may later supersede in the base
spec. Reusing `Endpoint.status` keeps the surface area small and uses
mechanisms every implementer already supports, while the documentation
text fully resolves the original "should it still be in the manifest?"
question. The trade-off vs Disposition A is precision: `Endpoint.status`
operates at endpoint granularity, not per-`instance`, so it cannot
distinguish a study where some series are NEARLINE and others ONLINE.
This is acceptable for a ballot release if a follow-up base-spec ask is
filed.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-51333 as **Not Persuasive â€” Already Supported** with a
resolution note that:

- the profile already permits `ImagingStudy.endpoint` to be empty;
- the obligation framework already requires the provider to populate
  what is known;
- the three edge cases raised in the reporter's follow-up comment are
  out of scope for the current v0.1.0 ballot cycle and will be
  re-evaluated when concrete vendor input from the FHIR Imaging WG
  arrives.

No changes to FSH, narrative, or examples.

#### Justification

The IG editor's own comment on the ticket (JIRAUSER28670, 2025-07-09)
already establishes that no profile change is required to support the
core use case; the original reporter agrees ("no changes are needed
there"). The reporter's expanded scenarios are useful framing but
explicitly invite further discussion from PACS vendors and the FHIR
Imaging WG that has not yet materialised in either Jira or Zulip. A
ballot disposition that adds new vocabulary or normative obligations on
the strength of one stakeholder's hypothetical scenarios risks locking
the IG into a model the broader Imaging community may reject. Declining
now keeps the door open without baking in a premature design.

---

### Recommendation

**Recommended disposition:** B

The ticket's own comment exchange has already resolved the literal
question (no cardinality change is required), so Disposition C
under-delivers: the IG genuinely lacks a written statement that
endpoint-less `ImagingStudy` entries are not only allowed but
**expected**, and reviewers will keep filing this ticket against future
ballots until that text exists. Disposition A goes further by inventing
an instance-availability extension, but doing that inside an HL7 Europe
IG â€” when the right home for instance-availability semantics is
arguably the base FHIR `ImagingStudy` resource maintained by the FHIR
Imaging WG â€” risks a parallel vocabulary that would later need to be
deprecated. Disposition B captures the documentation win immediately,
reuses `Endpoint.status` for the only signalling that is genuinely
needed at ballot time, and explicitly hands the per-instance
availability question to the right working group via a follow-up
ticket. It also composes cleanly with the sibling tickets FHIR-51296
(IID vs WADO guidance), FHIR-51334 (WADO-RS scope) and FHIR-51388
(Manifest Bundle scope), which together should be dispositioned as a
coordinated narrative-and-obligation refresh of the
`Manifest-ImImagingStudy` profile and its surrounding pages.
