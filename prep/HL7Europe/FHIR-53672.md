## Summary

The reporter (VictoriaAyvasky, HL7 Europe Laboratory project) requests that the
EU Laboratory Report IG add explicit narrative guidance on how implementers
should populate the `Specimen` resource — in particular distinguishing the
**primary sample** (what the laboratory actually receives, captured via
`Specimen.collection`, `Specimen.type`, `Specimen.container.type`, and
`Specimen.condition`) from a **secondary sample** that results from
post-collection processing (captured via a child `Specimen` whose
`Specimen.parent` references the primary specimen, and whose
`Specimen.processing.procedure` and `Specimen.processing.additive` describe the
processing step). The ticket is grounded in conventions reportedly endorsed by
laboratory experts and includes a fictive worked example: a whole-blood primary
specimen in an EDTA tube, centrifuged with a buffer additive into a serum
secondary specimen.

The ticket also explicitly cross-links its sibling change request
[FHIR-53671](https://jira.hl7.org/browse/FHIR-53671) ("Extend terminology for
specimen container type"), which proposes adding precoordinated container codes
(e.g. *Blood collection tube with sodium citrate 3.8%*, *Blood collection tube
with EGTA*) to the `lab-specimenContainer-eu-lab` value set bound at
`Specimen.container.type`. Together the two tickets articulate the same
underlying claim: container-side additives belong in `Specimen.container.type`
(precoordinated) — *not* in `Specimen.processing.additive`, which is reserved
for substances added during post-collection processing of the sample.

The current resolution is **Considered for Future Use**: the WG agrees the
guidance is valuable but believes it needs further development before being
adopted into the IG.

## Details

**Description:**

To mitigate misunderstandings regarding the conventions used in laboratory
report documentation, it is crucial to add more guidance especially regarding
additives in the context of `Specimen.container.type` and `Specimen.processing`.

The following topics were recently discussed with a laboratory expert and could
provide a good basis for additional guidance in the HL7 Europe Laboratory Report
IG:

- `Specimen.type` = body fluid (in microbiology also smear material, cells, etc.)
- Convention for `Specimen.collection`:
  - What a laboratory receives is (by definition) the primary sample.
  - This is obtained through `Specimen.collection`.
  - What the laboratory receives here is considered as 'unprocessed', possibly
    in a state "xy" (`Specimen.condition`, e.g. 'frozen').
  - Sample collection must not be confused with processing activities (which
    would result in a post-processing sample).
- `Specimen.container.type` = tube/container including (optional) additives
  - The specimen container type defines the vehicle for the primary sample.
  - The additives in a container are part of the code for the container type
    (precoordination).
  - See ticket requesting extension of the VS for specimen container types:
    [FHIR-53671](https://jira.hl7.org/browse/FHIR-53671).
- Convention for `Specimen.processing`:
  - Processing means post-processing of a primary sample that the laboratory
    has received.
  - Processing is applied to the entire sample, potentially resulting in a
    secondary sample.
  - In laboratory medicine this must be distinguished from sample changes that
    occur during analysis (e.g. inside the measuring device); those are not
    documented as processing.
  - Separation of samples is not equivalent to transforming a sample.
- `Specimen.processing.additive`:
  - Documented when a substance is added during the processing step.
  - Example: buffer solution.
- Processing information should be included in the FHIR instance of the
  *secondary* sample.

Worked example:

*Primary Sample*
- `Specimen.type` = Whole blood specimen
- `Specimen.collection.method` = Collection of blood specimen
- `Specimen.container.type` = EDTA blood collection tube

*Secondary Sample*
- `Specimen.parent` = Reference to primary sample (whole blood specimen)
- `Specimen.type` = Serum specimen from blood
- `Specimen.processing.procedure` = Centrifugation of blood specimen
- `Specimen.processing.additive` = Buffer solution
- `Specimen.condition` = centrifuged

**Comments:**

- **patrick-werner (2026-04-19)** — Agrees additional `Specimen` guidance,
  especially around processing, is valuable, but recommends *deferring* to allow
  time to develop well-established guidance. Points to the MII Biobank glossary
  as a similar reference that could be reused
  (`https://simplifier.net/guide/mii-ig-biobank-de-v2026/MIIIGModulBiobank/AnwendungsfaelleInformationsmodell/glossar.md`).
  Adds inline reactions to the proposed bullets:
  - On `Specimen.type` = body fluid: core already defines `Specimen.type` as
    "Kind of material that forms the specimen" — questions whether more guidance
    is needed beyond what core says.
  - On the "primary sample" rule: notes `Observation.specimen` may legitimately
    point to a *derived* specimen, so the "what the lab receives is the primary
    sample" framing is not absolute.
  - On `Specimen.container.type`: container defines the vehicle for *every*
    sample (not only the primary one).
  - On precoordinated additives in container.type: since
    `Specimen.processing.additive` is already constrained to `0..0` in the
    profile, the precoordination rule could be expressed as a "short" on the
    element itself rather than only narrative.
  - On the processing definition: clarifies that processing expresses what was
    done *to this specimen*, which can be either the primary or a derived one.
- **JIRAUSER26730 (2026-04-20)** — Side note: the Netherlands (Feikje) have
  proposed a *specimen container model in SNOMED CT*; suggests aligning with it
  to avoid duplicating the model (e.g. additives) in FHIR.

## Proposed Dispositions

The dispositions below address narrative/IG content in the **EU Laboratory
Report IG** (`hl7-eu/laboratory`). No briefing is loaded for that repo because
no GitHub cross-references exist, so file paths are described in conventional
HL7 IG terms (the IG's `input/pagecontent/` markdown pages and the
`StructureDefinition` for the lab-specimen profile) rather than asserted against
a specific clone layout.

### Disposition A: Accept As Requested

#### Proposal

Add a new "Specimen conventions" section to the EU Laboratory Report IG
narrative (a dedicated page under the IG's specimen / profile documentation, or
a sub-section on the existing lab-specimen profile page) that codifies the
reporter's bullets verbatim:

1. Define **primary sample** as what the laboratory receives, populated via
   `Specimen.collection` (with `Specimen.condition` capturing the as-received
   state, e.g. *frozen*).
2. State that `Specimen.container.type` is the vehicle for that primary sample
   and that **additives in the container are precoordinated into the container
   code** — implementers MUST NOT use `Specimen.processing.additive` for
   container additives. Reinforce this by adding a profile-level `short`
   ("Container including any precoordinated additives") on
   `Specimen.container.type` and, since the lab-specimen profile already
   constrains `Specimen.processing.additive` to `0..0` on the *primary*
   specimen, add a `short`/comment on that element explaining the rationale.
3. Define **secondary sample** as the result of post-collection processing,
   represented as a separate `Specimen` instance with `Specimen.parent`
   pointing to the primary specimen and the processing step expressed via
   `Specimen.processing.procedure` and `Specimen.processing.additive` on the
   secondary specimen.
4. Include the worked EDTA-tube → centrifuged-serum example as a normative
   example (FSH instances in `input/fsh/examples/` plus rendering on the
   guidance page).
5. Cross-link [FHIR-53671](https://jira.hl7.org/browse/FHIR-53671) so readers
   know the precoordinated container value set is being expanded to support the
   convention.

#### Justification

This is the lowest-cost, highest-leverage change: the request is for *guidance*,
not a structural model change, and the worked example is already concrete enough
to drop into the IG. Codifying the primary/secondary split matches established
laboratory practice and directly mitigates the misuse pattern the reporter and
the FHIR-53671 commenter (Martina) both flagged — implementers conflating
container additives with processing additives. Anchoring the rule in the
profile via `short`/comment text (in addition to narrative) is consistent with
how the IG already carries semantic constraints (e.g. the existing
`Specimen.processing.additive 0..0` slice on the primary-specimen profile),
and exploits a constraint that is already in place rather than introducing new
ones.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the *intent* of the request but refactor it before publication, as
patrick-werner suggested:

- Author the guidance as a "Specimen modelling" page that **reuses (and
  attributes) the MII Biobank glossary**
  (`mii-ig-biobank-de-v2026/.../glossar.md`) for the
  primary/secondary/processing terminology, rather than inventing parallel
  EU-specific definitions.
- **Soften absolute statements** so they survive scrutiny against FHIR core:
  - Frame "what the laboratory receives is the primary sample" as the *typical*
    case and explicitly call out that `Observation.specimen` may legitimately
    reference a derived specimen.
  - State that `Specimen.container.type` describes the vehicle for *the*
    specimen instance (primary or derived), not only the primary one.
- Keep the precoordinated-container-additive rule, but express it as a
  *normative* short on `Specimen.container.type` (Disposition A item 2) plus a
  pointer to the SNOMED CT specimen container model that the Netherlands
  (Feikje) is proposing (per JIRAUSER26730's comment), so the EU IG aligns with
  emerging SNOMED structure rather than locking in an EU-only convention.
- Defer the worked example into a *non-normative* "Examples" appendix until at
  least one round of review with the lab experts who contributed to FHIR-53671.
- Make landing this guidance **dependent on FHIR-53671** so the value set can
  actually express the codes the convention demands.

#### Justification

patrick-werner's review surfaced two real problems with accepting the text
verbatim: (a) the "primary sample" framing is too absolute given core's
`Observation.specimen` semantics, and (b) the container/vehicle statement is
narrower than what the data model actually allows. Publishing the text as-is
would create EU-specific guidance that contradicts core, which is worse than
publishing nothing. Reusing the MII Biobank glossary avoids reinventing
terminology that another HL7 affiliate has already developed and reduces the
chance of EU vs. DACH guidance drifting. Tying the convention to FHIR-53671 and
to the proposed SNOMED CT specimen container model addresses the substantive
risk that "use precoordinated container codes" turns into "but the codes don't
exist yet."

---

### Disposition C: Decline

#### Proposal

Close FHIR-53672 as *Considered for Future Use* (the current resolution) and
take no IG change at this time. Document in the resolution comment that the
guidance is **deferred pending** (a) progress on FHIR-53671 / the SNOMED CT
specimen container model, and (b) reconciliation of the proposed text with FHIR
core's `Specimen` definitions and the MII Biobank glossary, with the
expectation that the topic returns as a fresh, scoped guidance ticket once
those inputs land.

#### Justification

This is the resolution the WG has already reached (the ticket is currently
*Triaged* / *Considered for Future Use* and patrick-werner explicitly leaned
toward deferral). The guidance, while clearly useful, is not yet
self-consistent with FHIR core (the `Observation.specimen`-derived-specimen
case, the "container vehicle for the primary sample only" overreach), and a
core piece of its mechanism — the precoordinated container codes — is gated on
FHIR-53671, which is itself *Deferred*. Shipping guidance whose recommended
codes do not exist would actively harm implementers. Declining now and
re-opening when the dependencies are ready is the lowest-risk path; nothing
about deferral is irreversible because no spec artifact is being touched.

---

### Recommendation

**Recommended disposition:** B

The underlying request is sound and the worked example is unusually concrete for
a guidance ticket — declining outright (Disposition C) leaves a known
implementer-confusion problem in place, and the WG has the raw material to fix
it now. But Disposition A would publish text that patrick-werner has already
flagged as inconsistent with FHIR core in two specific places (the absolute
"primary sample" framing and the "vehicle for the primary sample" overreach),
and it would lean on precoordinated container codes whose value set is still
being extended via the *Deferred* FHIR-53671. Disposition B keeps the parts of
the request that are unambiguously correct (the primary/secondary distinction,
the rule that container additives are precoordinated and *not*
`Specimen.processing.additive`, the worked example), pulls in the MII Biobank
glossary and the proposed SNOMED CT specimen container model so the EU IG does
not invent parallel terminology, softens the statements that conflict with
core, and explicitly couples publication to FHIR-53671 so the recommended codes
exist when implementers reach for them. That gives the WG a credible path from
"Considered for Future Use" to a publishable narrative section without
shipping guidance that would have to be re-issued.
