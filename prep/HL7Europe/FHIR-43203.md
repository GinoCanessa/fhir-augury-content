## Summary

The reporter observes that the EU Laboratory Report IG ValueSet
[`lab-orderCodes-eu-lab`](http://hl7.eu/fhir/laboratory/ValueSet/lab-orderCodes-eu-lab)
expands to **0 concepts**. They cannot tell whether the empty expansion
is the intended result of the ValueSet's filter expression, or whether
the underlying terminology server cannot interpret (or does not support)
the filter as authored. The ticket is filed as a `Question` against the
EU Laboratory Report (FHIR) specification and asks the work group to
clarify which of the two it is — and, by implication, to fix the
ValueSet so that conforming order codes can actually be selected from
it. No cross-referenced Jira tickets, Zulip threads, or GitHub items
were found by FhirAugury for this ticket, so the request stands on its
own at this point.

### Disposition A: Accept As Requested

#### Proposal

Treat the empty expansion as a defect in the ValueSet definition and
fix it in the EU Laboratory Report IG sources.

Concretely:

1. Open the ValueSet resource for `lab-orderCodes-eu-lab` in the EU
   Laboratory Report IG (typically under the IG's `input/resources/`
   or `input/vocabulary/` authoring root, file
   `ValueSet-lab-orderCodes-eu-lab.json` / `.xml`).
2. Inspect the existing `compose.include[*].filter` entries. Verify
   that:
   - The `system` references a code system the publisher's terminology
     server can resolve (typically LOINC, `http://loinc.org`).
   - Each filter `property` and `op` pair is one of the operators the
     target code system actually advertises (e.g., for LOINC, the
     `LP*` ancestor / part hierarchy filters and `copyright` / `STATUS`
     properties — not arbitrary property names).
   - The `value` matches the expected lexical form (e.g., a LOINC part
     code such as `LP29693-6` for "Laboratory order").
3. Replace any unsupported filter with an equivalent expression that
   the IG's terminology server (typically tx.fhir.org or the HL7
   Europe tx server) can evaluate, or fall back to an explicit
   `concept` enumeration of the intended order codes.
4. Re-run the IG publisher and confirm that the
   `ValueSet/$expand` output is non-empty in the published expansion
   tab.
5. Add a regression test (publisher expansion check or an explicit
   expected count comment in the ValueSet narrative) so a future
   accidental empty expansion is caught at build time.

#### Justification

The reporter's observation is verifiable from the published expansion
and is, by FHIR convention, a defect: a ValueSet that an IG references
for binding should resolve to a non-empty set in the publisher's chosen
expansion context, otherwise instances cannot be authored against it
and validation degrades to a no-op. Fixing the filter (or replacing it
with explicit concepts) restores the binding's intent. This also aligns
with HL7 publishing practice that every IG-defined ValueSet must
expand cleanly against the IG's declared terminology server before
release.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only fixing the filter in place, restructure the ValueSet
to make its intent explicit and reviewable:

1. Replace the single filter-based `compose.include` with a layered
   composition:
   - One `compose.include` per source code system (LOINC, SNOMED CT,
     and any national catalogue HL7 Europe wishes to permit), each
     with a tightly-scoped filter or an explicit `concept` list.
   - Add `compose.include.valueSet` references to upstream IPS or
     LOINC order-panel ValueSets where they already capture the
     intent, instead of re-deriving the set with custom filters.
2. Document the inclusion criteria in the ValueSet's `description` and
   `purpose` so reviewers (and downstream IGs) can audit *why* a code
   is in the set, not just *that* it is.
3. If the underlying need is "any laboratory order code accepted by
   the receiver", consider declaring the binding as `example` or
   `preferred` against a broad ValueSet (e.g., a LOINC order-codes
   reference set) and create a separate, narrower ValueSet only for
   the concepts HL7 Europe wants to *recommend*.

#### Justification

A filter-only ValueSet that expands to zero is a symptom of a fragile
authoring pattern: the set's contents depend entirely on whether the
target terminology server implements the chosen filter operator the
same way the author intended. Splitting the ValueSet into explicit,
auditable inclusions reduces server-portability risk, makes review by
the work group tractable, and lets HL7 Europe state binding strength
honestly. The trade-off vs. Disposition A is more authoring effort up
front and a small loss of "automatic" coverage when LOINC adds new
order codes — but the gain in reviewability and cross-server
reproducibility is significant for a published European IG.

---

### Disposition C: Decline

#### Proposal

Close the ticket as `Not Persuasive` / `Works As Designed` on the
grounds that the empty expansion is an artifact of the *expanding*
terminology server's capability set, not of the IG content, and direct
the reporter to use a server (e.g., tx.fhir.org with the appropriate
LOINC license) that supports the filter operator the ValueSet relies
on.

#### Justification

This is defensible only if the work group can demonstrate that the
ValueSet's filter is well-formed per the FHIR Terminology specification
and that *some* conformant server does produce a non-empty expansion.
In that case the issue is environmental rather than normative.

---

### Recommendation

**Recommended disposition:** A (with elements of B folded into the fix)

A published IG ValueSet that expands to zero concepts is, in practice,
indistinguishable from a broken binding for downstream implementers —
profile validators will silently accept anything, and authors have no
machine-readable list to choose from. That alone justifies treating
this as a defect (Disposition A) rather than as an environmental issue
(Disposition C). Disposition B's structural cleanup is desirable but
need not block the immediate fix; the work group can land Disposition
A in the next IG patch release and schedule the layered restructuring
of `lab-orderCodes-eu-lab` for the next minor cycle. Decline
(Disposition C) is only safe if HL7 Europe can point to a reference
terminology server on which the current filter expands non-trivially —
absent that evidence in this ticket, the conservative move is to
correct the ValueSet.
