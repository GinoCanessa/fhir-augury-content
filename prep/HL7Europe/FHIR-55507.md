## Summary

The ticket asks a clarifying question about the EU Core MedicationRequest
profile ([medicationRequest-eu-core](https://hl7.eu/fhir/base/2.0.0-ballot/StructureDefinition-medicationRequest-eu-core.html)):
is `MedicationRequest.intent` truly limited to the three codes "order",
"proposal", and "plan", as the rendered "Description & Constraints" view
suggests? The reporter (Francois Peverali, gematik GmbH) flagged this during
the gematik internal review of the HL7 Europe Base & Core STU2 January 2026
ballot. A follow-up comment on the ticket points out that the source of the
confusion is the [profile snapshot's Description & Constraints rendering](https://build.fhir.org/ig/FHIR/ig-guidance/readingIgs.html#table-views),
which shows "Order, proposal or plan" — wording that is *not* coherent with
the underlying ValueSet bound to `intent`. No related Jira tickets were
explicitly linked. The only related discussion is the
`german (d-a-ch)` Zulip thread "HL7 EU Base & Core Ballot", in which
Joshua Wiedekopf confirms that the EU Core profile keeps the standard
required binding to `http://hl7.org/fhir/ValueSet/medicationrequest-intent`
and adds no invariants or patterns restricting the value, so other codes
(e.g. `original-order`, `reflex-order`, `filler-order`, `instance-order`,
`option`) are in fact still allowed; the issue is therefore purely a
documentation/short-text problem in the profile, and the ticket text should
be reworded as a request to fix that wording.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a documentation / rendering defect against the EU Core
MedicationRequest profile and update the profile's short description on
`MedicationRequest.intent` so that it accurately reflects the underlying
required binding. Concretely, replace the current "Order, proposal or
plan" `short` (which is what drives the "Description & Constraints"
column shown in the IG tooling) with wording that is consistent with the
full `medicationrequest-intent` ValueSet — e.g. a short along the lines of
"proposal | plan | order | original-order | reflex-order | filler-order |
instance-order | option" (matching the base FHIR R4 short for the
element), or, if HL7 Europe really does want to express a preference for
the three codes, change the wording to something like "Typically order,
proposal or plan" and add an explanatory sentence in `definition` /
`comment` making clear that the binding remains the full base ValueSet
and that other codes are not prohibited.

#### Justification

This is the literal fix for the incoherence the reporter and the
follow-up comment identified: the rendered short text contradicts the
required binding. Fixing the `short` (and, if needed, `definition` /
`comment`) is a low-risk, non-breaking change that brings the profile's
human-readable description in line with its computable behaviour, which
is exactly what the FHIR IG-rendering guidance ([reading IGs — table
views](https://build.fhir.org/ig/FHIR/ig-guidance/readingIgs.html#table-views))
expects readers to rely on. Joshua Wiedekopf's analysis on Zulip
explicitly endorses this framing, and the reporter has accepted that
re-wording is the right outcome.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the underlying required binding to
`http://hl7.org/fhir/ValueSet/medicationrequest-intent` unchanged, but
formalise the apparent EU intent: rather than only fixing the prose,
introduce an `additionalBinding` on `MedicationRequest.intent` of
`purpose = preferred` (or `usage`) pointing at a *new* EU-Core-defined
ValueSet that contains exactly `proposal`, `plan`, and `order`, and
update the `short` to read along the lines of "Typically order, proposal
or plan (see EU Core preferred ValueSet); other codes from the base
ValueSet remain allowed." This documents the EU recommendation in a
machine-readable way without restricting the element.

#### Justification

If HL7 Europe genuinely wants to nudge implementers toward the three
codes (which the existing short suggests), an `additionalBinding` is the
mechanism FHIR R4 / R4B / R5 provides for exactly this case — it
preserves backward compatibility with the base ValueSet, does not
invalidate any existing instances using `original-order`, `reflex-order`
etc., and survives validation cleanly because `additionalBinding` is
advisory. It also makes the EU recommendation discoverable by tooling,
not just by reading the prose. The trade-off vs. Disposition A is more
artefacts to maintain (a new ValueSet) and a slightly heavier change for
what is currently classified as a Question.

---

### Disposition C: Decline

#### Proposal

Decline as "Not Persuasive — Already Addressed by Existing Mechanism".
Close with a comment that `MedicationRequest.intent` in the EU Core
profile binds to the standard required ValueSet
`http://hl7.org/fhir/ValueSet/medicationrequest-intent` and no
EU-specific constraint, invariant, or pattern restricts that binding;
all base codes remain allowed. Refer the reporter to the snapshot view
and the binding statement in the formal definition rather than the
rendered "Description & Constraints" column.

#### Justification

Strictly speaking, the question has a clear answer in the artefacts as
they stand: the binding is unchanged from base FHIR. No spec change is
*required* to make the profile behave correctly, and the ticket itself
is logged as a Question, not a Change Request. The German affiliate's
own summary on Zulip excluded FHIR-55507 from the issues it would attach
to a negative vote, which signals the workgroup did not view it as a
defect requiring action.

---

### Recommendation

**Recommended disposition:** A

The reporter's confusion is real and traceable to a concrete artefact
defect: the `short` description on `MedicationRequest.intent` in the EU
Core profile says "Order, proposal or plan", which directly contradicts
the required binding to `medicationrequest-intent`. Joshua Wiedekopf's
on-thread analysis (the only substantive technical reply in the d-a-ch
Zulip discussion) reaches the same conclusion and explicitly recommends
re-wording. Disposition C technically resolves the *question* but leaves
the misleading short in place, guaranteeing the next implementer asks
the same thing — that's why it should be rejected. Disposition B is
defensible if HL7 Europe really does want to express a preference for the
three codes, but no such preference is documented anywhere outside the
short text, so introducing a new ValueSet plus an `additionalBinding`
adds maintenance burden without clear evidence of intent. Disposition A
is the minimal, correct, non-breaking fix and matches both the
reporter's and the responding community member's expectation; if during
reconciliation HL7 Europe surfaces an actual preference for the three
codes, the discussion can naturally upgrade to Disposition B.
