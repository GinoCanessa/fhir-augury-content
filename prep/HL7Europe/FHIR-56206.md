## Summary

The ticket asks the HL7 Europe Laboratory Report (FHIR) IG to update three
value sets used by `Observation` so that they align with the recommendations
recently published by the **Xt-EHR** project. Specifically, Xt-EHR has
proposed updated terminology for (1) coded laboratory results
(`Observation.value[x]` when bound to a coded value set) and (2) the
`Observation.method` element. The reporter additionally requests that the
working group develop a properly worked-out value set for
`Observation.dataAbsentReason`, which is currently either unbound or bound
to a value set that the reporter considers inadequate for EU laboratory
exchange. The ticket has been triaged with the resolution "Considered for
Future Use", with a note that it should be reviewed by the HL7 EU
Laboratory Semantic group for the next version of the IG. There are no
cross-referenced Jira tickets, no related Zulip threads, and no related
GitHub PRs/issues recorded for this item.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the EU Laboratory Report (FHIR) IG, replace or extend the current
bindings on the relevant `Observation` profile(s) so that:

1. The value set bound to coded laboratory result values
   (`Observation.value[x]` when typed as `CodeableConcept`, e.g. on
   the `ObservationResultsLaboratory` / `ObservationResultLaboratory`
   profile) is updated to the Xt-EHR-recommended value set for coded
   results.
2. The value set bound to `Observation.method` on the same profile(s)
   is updated to the Xt-EHR-recommended method value set.
3. A new EU-Lab-specific value set for `Observation.dataAbsentReason`
   is authored (drawing on, but possibly narrower than, the FHIR core
   `data-absent-reason` value set), and `Observation.dataAbsentReason`
   on the lab profile(s) is bound to it (binding strength `extensible`
   or `preferred`, to be decided by the Semantic group).

For each of the three value sets, define them in the IG's value-set
folder, cite the upstream Xt-EHR artifact (URL + version) as the source,
and document in the profile narrative which Xt-EHR release the binding
tracks. Add a versioning note so future Xt-EHR updates can be picked up
in a controlled way.

#### Justification

The HL7 Europe Laboratory Report (FHIR) IG is explicitly aligned with
Xt-EHR's cross-border laboratory work; tracking Xt-EHR's terminology
recommendations is part of the IG's value proposition. The reporter is
the IG's primary semantic contributor, and the resolution already
defers the work to the HL7 EU Laboratory Semantic group "for the next
version", which signals organisational acceptance in principle. Adding
a worked-out `dataAbsentReason` value set closes a known gap that
implementers hit when laboratory results are partially missing (no
specimen, indeterminate, etc.) and is consistent with how other EU
clinical IGs constrain `dataAbsentReason`.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-defining the three value sets inside the EU Laboratory
Report IG, **reference the Xt-EHR-published value sets directly** (by
canonical URL) and constrain only where EU-Lab needs to diverge. For
`Observation.dataAbsentReason`, instead of authoring a brand-new value
set, define an EU-Lab profile that **narrows the FHIR core
`data-absent-reason` value set** to the subset relevant to laboratory
reporting (e.g. `unknown`, `not-performed`, `not-applicable`,
`unsupported`, `error`, `masked`, `temp-unknown`) and bind with strength
`required`. Document the rationale per excluded code.

#### Justification

Re-publishing Xt-EHR value sets inside the IG creates a maintenance
burden and a divergence risk: every Xt-EHR refresh would require an EU
IG republication, and consumers would have two canonical URLs for what
is meant to be the same concept set. Pointing at Xt-EHR's canonical
value sets keeps a single source of truth and makes the alignment
visible in the artifact graph. For `dataAbsentReason`, narrowing the
existing FHIR core value set is lighter-weight than authoring a new
ValueSet resource and avoids fragmenting the terminology landscape; a
`required` binding on a narrowed set also gives implementers stronger
guarantees than an `extensible`/`preferred` binding to a new IG-local
set.

---

### Disposition C: Decline (defer pending Xt-EHR finalisation)

#### Proposal

Decline to update the bindings in the current ballot cycle. Mark the
ticket as **Persuasive with Modification** in spirit but defer the
actual binding changes until Xt-EHR formally publishes a versioned,
URL-stable release of the recommended value sets, and until the HL7 EU
Laboratory Semantic group has reviewed and signed off on the
`dataAbsentReason` set. Document the intent to track Xt-EHR in the IG
narrative without changing any bindings now.

#### Justification

The ticket description references Xt-EHR "proposals", and the resolution
text already says the work is to be considered for the next version.
Changing terminology bindings mid-cycle without a stable, versioned
upstream invites churn for implementers and breaks reproducibility of
prior IG snapshots. Declining now (with a clear deferral rationale)
preserves implementer stability while keeping the alignment commitment
on record.

---

### Recommendation

**Recommended disposition:** B

The HL7 EU Laboratory Report IG's value to implementers comes from
**aligning with**, not **forking from**, the Xt-EHR terminology work.
Disposition B keeps a single canonical source for the coded-results and
method value sets (Xt-EHR) and only does new authoring where EU-Lab
genuinely needs something the FHIR core does not already provide
(`dataAbsentReason` narrowed to a lab-relevant subset). Disposition A
solves the user-visible problem but risks the IG drifting from Xt-EHR
between releases. Disposition C is defensible only if Xt-EHR's
recommendations are not yet stable enough to cite by canonical URL —
the Semantic group should confirm that before accepting C. Because no
GitHub PR, issue, or Zulip thread was found for this ticket, the
Semantic group will need to confirm the canonical URLs and exact
binding strengths as part of accepting B.
