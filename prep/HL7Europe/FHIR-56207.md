## Summary

The ticket carries forward feedback received on the Xt-EHR D7.1 deliverable and asks the HL7 Europe Laboratory Report (FHIR) IG to expand the value set bound to the laboratory `Observation.value[x]` (referred to in the ticket as `observation.codedValue`) used for antimicrobial susceptibility test (AST) reporting. Specifically, SNOMED International recommends including the descendants-or-self of SNOMED CT `419241000 |Substance with antibacterial mechanism of action (substance)|` (`<<419241000`) so AST results can identify the antibacterial substance under test. The ticket also asks the working group to consider the analogous SNOMED CT "mechanism of action" grouper concepts for other anti-infective classes — acaricide, anthelmintic, antifungal, antiprotozoal, and antiviral substances — so that susceptibility reporting is not limited to antibacterials. The reporter notes the proposal should be analysed by the semantic working group. No cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next version of the HL7 Europe **EU Laboratory Report (FHIR)** IG, extend the value set bound to `Observation.value[x]` (the "coded value" used by AST result observations) so that it includes, in addition to the codes already permitted, the SNOMED CT subsumption expansions:

- `<< 419241000 |Substance with antibacterial mechanism of action (substance)|`
- `<< |Substance with acaricide mechanism of action (substance)|`
- `<< |Substance with anthelmintic mechanism of action (substance)|`
- `<< |Substance with antifungal mechanism of action (substance)|`
- `<< |Substance with antiprotozoal mechanism of action (substance)|`
- `<< |Substance with antiviral mechanism of action (substance)|`

Implement this by either (a) adding these `concept`/`filter` includes to the existing AST substance value set, or (b) creating a dedicated "Anti-infective substances for susceptibility testing" value set composed from these six SNOMED CT mechanism-of-action groupers and binding it (preferred binding strength: `extensible`) to the appropriate `Observation.value[x]` / "codedValue" slice on the AST observation profile. Look up the exact SCTIDs for the five non-antibacterial groupers during authoring (the ticket only supplies the SCTID for antibacterial), and document the binding rationale by citing the Xt-EHR D7.1 / SNOMED International recommendation.

#### Justification

The change is exactly what the SNOMED International review of Xt-EHR D7.1 recommended, keeping the EU Lab Report IG aligned with international SNOMED guidance for AST reporting. Using SNOMED's mechanism-of-action groupers with the `<<` (descendants-or-self) operator is idiomatic SNOMED CT value-set authoring and avoids enumerating individual substances, so the value set stays maintainable as SNOMED CT evolves. Covering all six anti-infective classes (antibacterial, acaricide, anthelmintic, antifungal, antiprotozoal, antiviral) ensures susceptibility reporting is not silently restricted to antibacterials, which matches real European laboratory practice (mycology, virology, parasitology). An `extensible` binding preserves the ability of jurisdictions to add locally required substances while still steering implementers toward the shared SNOMED groupers.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the single "coded value" value set on the AST observation, model the change as a **profile-level slice** plus a **dedicated, separately-published value set**:

1. Create a new value set, e.g. `ValueSet/anti-infective-substance-eu-lab`, composed of the six SNOMED CT mechanism-of-action groupers (`<< 419241000` and the acaricide/anthelmintic/antifungal/antiprotozoal/antiviral analogues).
2. On the EU Lab Report AST observation profile, slice `Observation.value[x]` (or the AST "substance under test" element, depending on the IG's current modelling — `Observation.code` for the test, or a dedicated component such as `component:antimicrobial.code`) so there is an explicit slice for "anti-infective substance under test" bound to that new value set with `extensible` strength.
3. Leave the existing broader binding intact for any non-AST coded values, so the new constraint only applies in the AST context.

This approach also opens the door to publishing the anti-infective value set as a reusable HL7 Europe terminology artifact that other European IGs (e.g. AMR surveillance) can reuse without depending on the EU Lab Report IG.

#### Justification

The ticket conflates two questions: *which SNOMED concepts are appropriate for AST substance reporting* and *where in the Observation structure they belong*. Disposition B answers both explicitly and avoids quietly broadening a value set that may also be reused by non-AST observations. A dedicated, named value set is easier to reference from external IGs, easier to version, and easier to align with future SNOMED International releases. Slicing the AST observation profile makes the constraint discoverable to implementers (a slice with a clear name and binding) rather than relying on them to know that one of many codes in a large value set happens to be the right one. The trade-off vs. Disposition A is more authoring effort and a slightly larger artifact surface, but the resulting model is more precise and more reusable.

---

### Disposition C: Decline

#### Proposal

Decline to incorporate the proposal into the EU Laboratory Report (FHIR) IG at this time, on the grounds that (a) the request originates from external (Xt-EHR D7.1 / SNOMED International) feedback that has not yet been ratified by the HL7 Europe Laboratory Semantic group, and (b) AST substance terminology is arguably the responsibility of a dedicated AMR / antimicrobial profile or of the SNOMED CT international edition itself, not of the EU Lab Report base IG. Recommend instead that the requester raise the change against a future "EU AMR Reporting" companion IG, or wait for the Semantic group's analysis to produce a concrete value-set proposal before opening a binding-level change.

#### Justification

Declining is defensible because the ticket as written is a *direction-setting* request ("we should analyse this in the semantic WG") rather than a precise, ready-to-implement change. The five non-antibacterial groupers are listed only by display name without SCTIDs, which means accepting the ticket as-is would commit the IG to codes the ticket did not actually identify. There is also a scope question: AST substance value sets are a candidate for cross-jurisdiction reuse, and locking them into the EU Lab Report IG could pre-empt a more coordinated AMR terminology effort. None of these arguments say the proposal is wrong on the merits; they say the ticket is not yet a concrete change. Declining (or deferring) until the Semantic group produces a reviewed value-set composition is the cautious option.

---

### Recommendation

**Recommended disposition:** B

The Semantic group's stated direction (and SNOMED International's recommendation) is sound on the merits, so outright declining (Disposition C) under-serves the AST use case the IG is meant to support. However, simply expanding the existing coded-value binding (Disposition A) hides a non-trivial modelling decision inside a value-set edit and risks broadening codes for non-AST observations as a side effect. Disposition B keeps the SNOMED-aligned content from Disposition A while making the AST constraint explicit on the profile (a named slice with a dedicated, named value set), which is easier to review, easier to version, and easier for other HL7 Europe IGs to reuse. Given the ticket's resolution is already "Considered for Future Use" with an explicit hand-off to the Semantic group, the next-version work-package is a natural place to do the slightly larger authoring task that Disposition B requires.
