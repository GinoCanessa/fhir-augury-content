## Summary

Alexander Zautke (on behalf of the German imaging community, informed by the German DIN Norm) proposes that the EU Imaging Study Report IG distinguish between two conceptually different questions captured around an imaging order:

- The **medical question** authored by the referring physician (e.g. "coughing and fever, pneumonia?"), and
- The **radiological question** — the specific findings the radiologist looks for (e.g. "Evaluate for pulmonary infiltrates, pleural effusion, and other signs suggestive of pneumonia").

Today the ImOrder profile (a `ServiceRequest` profile at `https://build.fhir.org/ig/hl7-eu/imaging/StructureDefinition-ImOrder.html`) maps only the medical question to `ServiceRequest.reason`. The reporter proposes letting the radiologist author a new *version* of the ServiceRequest with `requester` and `reason` updated to capture the radiological question, and asks whether this is the right pattern. A second, related question asks whether multiple ImagingOrders (for different medical questions) that collapse into a single procedure should be linked by referencing multiple ImagingOrders from the order section.

The ticket is cross-referenced only to two FHIR targets — `Composition.section` and `ServiceRequest.reason` — both mentioned in the discussion. No Jira, Zulip, or GitHub cross-references were surfaced by the orchestrator. However, the ticket comments explicitly link to **hl7-eu/imaging issue #14** ("Is the order section needed?"), where the same community has been modelling the medical question and study indication.

## Details

**Description:**

> Resource profile: https://build.fhir.org/ig/hl7-eu/imaging/StructureDefinition-ImOrder.html
>
> In the German DIN Norm, there is a differentiation between the medical question and the radiological question.
>
> The medical question is the one the referring physician is asking, like "coughing and fever, pneumonia?"
>
> The radiological question is the specific features the radiologist would be then looking for, like "Evaluate for pulmonary infiltrates, pleural effusion, and other signs suggestive of pneumonia."
>
> We mapped the medical question to `servicerequest.reason`.
>
> 1. This would also be a reasonable place to map the radiological question to, however, we then run into issues of unclear authorship as the `ServiceRequest` is usually authored by the referring physician.
>
> Therefore, we see a need to be able to create a new version of the order by the radiologist, which is indeed possible to our knowledge. The fields `servicerequest.requester` would have to be updated to the radiologist's details and `servicerequest.reason` to the radiological question. Would this be the right way?
>
> 1. Sometimes, multiple medical questions from different service requests result in the same procedure. For example, "Cough and fever, pneumonia?" and "History of malignancy, rule out pulmonary metastases?" would both result in one single chest CT, while "Cough and fever, pneumonia?" alone would more probably result in an X-Ray. In this case, we would reference multiple ImagingOrder resources in the order-section. Would this be the right way?

**Comments:**

### JIRAUSER28670 — 2025-07-07

Points Alexander to the German community's parallel work in [hl7-eu/imaging#14](https://github.com/hl7-eu/imaging/issues/14) ("Is the order section needed?"). Notes that issue #14 only discussed the *medical question* and an *indication*, not a separate *radiological question*. Agrees that the medical question (e.g. "rule out pneumonia") must be distinguished from the indication (e.g. "fever and shortness of breath"). Observes that the XtEHR [EHDSImagingReportBody](https://build.fhir.org/ig/Xt-EHR/xt-ehr-common/StructureDefinition-EHDSImagingReportBody.html) logical model exposes both `.orderReason` / `.orderReasonText` and `.clinicalQuestion` but the current EHDS Imaging mapping sends both to `ServiceRequest.reason`, which is incorrect because the medical question is usually free text. Suggests encoding the clinical question in `ServiceRequest.reason.extension[note]` (already used in the IG for free-text capture) and replicating it in `Composition.section[order].extension[note]`. Asks Alexander three scoping questions about the radiological question:
- Is it a common use case that radiologists "overwrite" the order to add radiological questions?
- Would that be a new version of the same `ServiceRequest` or a new one with extra requests from the radiologist?
- Is a separate `ServiceRequest` profile required?

### JIRAUSER28670 — 2025-07-15

Reports a tentative resolution from the HL7 EU Imaging meeting of 15 July 2025:
- Change the mapping of XtEHR `orderInformation.clinicalQuestion` from `ImOrder.reason.concept` to `ImOrder.reason.extension[note]`.
- Add a mapping from `orderInformation.clinicalQuestion` → `Composition.section[order].extension[note]`.

Rationale: the clinical question (the medical question, free text entered by the requester, e.g. "rule out pneumonia") would live in `ImOrder.reason.extension[note]` for the RESTful representation and be replicated in `Composition.section[order].extension[note]` for the document representation. The coded *indication* (e.g. "thoracic pain and fever") remains in `ImOrder.reason.concept` and `CodeableConcept.text` is replicated into the composition section for the human-readable document.

### JIRAUSER28670 — 2025-07-22

Floats "option 2": add a custom extension `ImOrder.reason.extension[clinical-question]` to encode the clinical question primarily, and render it into `Composition.section[order].extension[note]`. This distinguishes the clinical question from any generic `note` and avoids overloading the existing `[note]` slice.

### JIRAUSER29565 — 2025-07-22

Alexander answers the three scoping questions from the 2025-07-07 comment:
- **Common use case?** *"Yes, from the information we gathered this is a common case."*
- **New version vs. new ServiceRequest?** *"A new version. The old one is outdated once the radiologist edited it."*
- **Separate ServiceRequest profile?** *"No, only a new version."*

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's pattern verbatim in the EU Imaging Study Report IG:

1. Document in the `ImOrder` profile narrative that when a radiologist refines a request with a *radiological question*, implementers **update the existing `ServiceRequest` resource** (create a new version via `PUT`), setting `ServiceRequest.requester` to the radiologist and `ServiceRequest.reason` to the radiological question. The original order becomes an historical version retrievable through the FHIR history interaction. No new profile, no new extension.
2. Clarify in the same narrative that when **multiple distinct medical questions** (from multiple referring-physician `ServiceRequest` instances) collapse into a single imaging procedure, the IG supports referencing **multiple `ImagingOrder` resources** from the ImComposition's order `section.entry`, one per originating `ServiceRequest`.
3. Leave the existing mapping of `orderInformation.clinicalQuestion` → `ServiceRequest.reason` unchanged.

#### Justification

This is what the reporter asked for, is mechanically supported by FHIR (resource versioning plus `Composition.section.entry` cardinality `0..*`), and requires no structural change to the IG. The 2025-07-22 comment from the reporter confirms that radiologists overwriting the order is a common case, that a new *version* (not a new resource or a new profile) is the intended pattern, and that no separate profile is needed — which aligns exactly with this disposition.

However, the disposition inherits the problems the 2025-07-07 and 2025-07-15 comments raise: (a) `ServiceRequest.reason` is a `CodeableReference`, so free-text radiological questions end up in `reason.concept.text`, which conflates the *coded indication* with the *free-text clinical question*; (b) mutating `requester` on a new version erases the referring physician from the current view of the resource, forcing consumers to follow history to recover authorship — a non-trivial requirement for documents where provenance matters and for receivers that do not store history. These caveats should be called out explicitly in the disposition text.

---

### Disposition B: Alternative Approach — Separate extensions for medical and radiological questions; keep `reason.concept` for the coded indication

#### Proposal

Adopt a structural split in the `ImOrder` profile:

1. Continue to use `ImOrder.reason.concept` for the coded **indication** (e.g. "fever, shortness of breath").
2. Introduce **two dedicated extensions** on `ImOrder.reason` (or re-use the existing `[note]` slice plus one new slice):
   - `ImOrder.reason.extension[clinical-question]` — free-text **medical question** from the referring physician (e.g. "rule out pneumonia"). This is the renamed/typed form of today's `[note]` use.
   - `ImOrder.reason.extension[radiological-question]` — free-text **radiological question** authored by the radiologist (e.g. "evaluate for pulmonary infiltrates, pleural effusion...").
3. Update the XtEHR logical-model mapping:
   - `orderInformation.orderReason` / `.orderReasonText` → `ImOrder.reason.concept` / `.concept.text`.
   - `orderInformation.clinicalQuestion` → `ImOrder.reason.extension[clinical-question]`.
   - (New) `radiologicalQuestion` → `ImOrder.reason.extension[radiological-question]` (add to the XtEHR model or document as HL7-EU-only until XtEHR catches up).
4. Replicate both question extensions into `Composition.section[order].extension[note]` (or introduce parallel `[clinical-question]` / `[radiological-question]` slices on the composition section) so the documentary form preserves the same distinction.
5. Record authorship of the radiological question via `Extension.author[x]` on the `radiological-question` extension (typed as `Reference(Practitioner|PractitionerRole)`), avoiding the need to mutate `ServiceRequest.requester`. This keeps the original referring-physician authorship intact on the canonical `ServiceRequest`.
6. Versioning the `ServiceRequest` when the radiologist contributes becomes optional rather than the primary mechanism.

#### Justification

This alternative directly addresses the two weaknesses of Disposition A. It preserves the referring physician as `requester` (the predominant real-world and legal expectation), which was flagged by the reporter themselves as the authorship concern. It gives the radiological question a first-class, typed slot that consumers can render and validate independently of the indication and the medical question, matching the XtEHR `.orderReason` / `.clinicalQuestion` split and extending it naturally. It also aligns with the discussion's 2025-07-15 tentative resolution (map `clinicalQuestion` to `ImOrder.reason.extension[note]`) and the 2025-07-22 "option 2" suggestion (a named `[clinical-question]` extension), generalised to cover the radiological case. Because `Composition.section[order].extension[note]` is already used in the IG, the document-side change is additive.

The trade-off is extension proliferation and a required coordination step with XtEHR (or an HL7-EU-local extension until the logical model absorbs `radiologicalQuestion`). The IG team will also need to decide whether to add `Extension.author[x]` or to rely on Provenance for authorship — the former is lighter-weight for consumers.

---

### Disposition C: Decline

#### Proposal

Decline the request as worded (addition of a dedicated *radiological question* concept on the ImOrder profile) on the grounds that the requirement is implementation-specific (German DIN Norm) and is already expressible with existing mechanisms:

- The free-text radiological question can be carried today in `ServiceRequest.reason.extension[note]` (already in the IG) or in a generic `ServiceRequest.note` entry authored by the radiologist.
- FHIR `ServiceRequest` versioning plus `Provenance` already covers "radiologist refined the order" cases for jurisdictions that need it.
- Add only a non-normative narrative paragraph to the `ImOrder` profile documenting how implementers can use these existing mechanisms, and point back to the XtEHR logical model for the indication / clinical-question distinction.

#### Justification

The EU Imaging IG is meant to be a pan-European baseline and should avoid encoding one national standard's vocabulary (DIN) into normative profile structure when no cross-jurisdictional consensus yet exists that the *radiological question* is a distinct, widely-used concept. The only supporting evidence in the ticket is one reporter's input and a single HL7 EU Imaging meeting discussion; no Zulip threads, no other Jira tickets, and no other national IGs have been cited. Adding an extension now risks locking in a model that other EU jurisdictions will need to rework. A narrative-only change keeps the door open for a future, evidence-backed change request.

The cost of declining is that German implementers either carry the distinction out-of-band or lose it on round-trip through the EU IG.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is the cleanest fit to the community's own discussion trajectory. The 2025-07-07 and 2025-07-15 comments already converged on encoding the medical (clinical) question as a typed extension on `ImOrder.reason` and replicating it into `Composition.section[order].extension[note]`; Disposition B simply extends that pattern symmetrically to the radiological question and fixes the authorship problem the reporter themselves raised. Disposition A is mechanically valid but forces consumers to chase `ServiceRequest` history to recover the referring physician and conflates coded indication with free-text question text — both real interop liabilities for an IG that flows into a `Composition`. Disposition C is defensible as a conservative move but discards signal from the very meeting that produced the current draft mappings and leaves German implementers with no normative home for the radiological question.

A practical path for the workgroup: adopt B's extension shape, but keep scope tight — (i) name the two extensions `clinical-question` and `radiological-question` (avoid reusing the generic `[note]` slice for semantically distinct content), (ii) add `Extension.author[x]` so authorship travels with the radiological question rather than via a versioned `ServiceRequest.requester` swap, and (iii) open a follow-up ticket against XtEHR's `EHDSImagingReportBody` to add a `radiologicalQuestion` attribute so the cross-IG mapping stays coherent. A concrete implementation would live in the `hl7-eu/imaging` repo; before PR work begins, run `repo-analysis hl7-eu/imaging` so the change can be grounded in the repo's authoring roots and change recipes.
