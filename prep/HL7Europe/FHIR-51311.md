## Summary

The reporter asks the EU Imaging Study Report (R5) guide to add
narrative describing when a `Device` is expected to appear as an
author on the report Composition (and, by extension, on downstream
artifacts such as `ImagingStudy` performers). The reporter's hypothesis
is that the intent of allowing a `Device` author was to accommodate
AI-generated or AI-assisted **preliminary reads / machine-generated
interpretations**, *not* the imaging modality itself (i.e., not the
scanner that acquired the study). The ticket therefore requests that
the guide make this distinction explicit so implementers do not
misuse `Device` authorship as a pointer to the acquisition device.

The ticket is informally linked (via a comment) to
[FHIR-51407](https://jira.hl7.org/browse/FHIR-51407), which addressed a
closely related gap: insufficient documentation of the performer slices
in `ImagingStudy.series.performer` and misalignment between the chosen
performer code and the restricted set of allowed performer types.
FHIR-51407 has already been applied (Resolution: Persuasive) with PR
[hl7-eu/imaging#92](https://github.com/hl7-eu/imaging/pull/92) clarifying
the custodian vs. imaging device vs. clinician distinction and allowing
`EuDevice`/`EuOrganization` in the performer slice. FHIR-51311 extends
the same clarity request — one level up — to Composition-level
authorship semantics, specifically calling out AI preliminary reads as
the intended use case for a `Device` author.

A triage comment from 2025-08-21 states the ticket will be addressed in
the current iteration.

### Disposition A: Accept As Requested

#### Proposal

Add narrative to the EU Imaging Study Report (R5) guide clarifying the
intended semantics of a `Device` appearing as `Composition.author`
(and, consistently, in the `ImagingStudy.series.performer:device`
slice). Specifically:

- State that `Device` as author is intended to represent **software
  that authored interpretive content** — typically an AI / CAD /
  machine-learning algorithm that produced a preliminary read or an
  automated interpretation that forms (part of) the report.
- Explicitly say that the acquisition modality (the scanner/imaging
  device that produced the pixel data) is **not** the intended
  referent of `Composition.author` and should instead be represented
  as the `ImagingStudy.series.performer:device` (per FHIR-51407's
  resolution) or via the relevant `ImagingStudy` device references.
- Provide at least one example (e.g., an AI preliminary-read
  Composition whose `author` is a `Device` profiled as an AI software
  device, with the human radiologist who signs off appearing as an
  additional author or as `attester`).
- Cross-link this narrative from both the Composition profile page and
  the ImagingStudy profile page so readers arrive at the same
  explanation from either direction.

#### Justification

This is exactly what the reporter asked for and it aligns with the
direction already set by FHIR-51407 (Persuasive / Applied), whose
resolution distinguished the custodian, the imaging device, and the
clinician on `ImagingStudy.series.performer`. Extending that same
distinction to Composition-level authorship keeps the guide internally
consistent and answers a concrete implementer question — "what do I
put in `author` for an AI-generated preliminary read?" — that is
currently left to guesswork. Core FHIR already permits `Device` in
`Composition.author`; this change is purely narrative and cannot break
instance conformance.

---

### Disposition B: Alternative Approach — Clarify *and* constrain with an AI-device profile + attester guidance

#### Proposal

In addition to the narrative in Disposition A, strengthen the guide so
that the AI-preliminary-read use case is not just described but
*supported* by machine-readable guidance:

- Introduce (or reuse, if one already exists in the EU Imaging Study
  Report suite) an `EuDevice`-derived profile or example tagged for
  "software/AI that produces interpretive content" (e.g., a profile or
  an example with `Device.type` from an appropriate SNOMED
  CT / device-type value set for AI/CAD software), and reference it
  from the `Composition.author` documentation.
- Add explicit guidance on the expected interaction between a `Device`
  author (the AI) and the human attester/author: the human radiologist
  who reviews and signs the report should be represented — the guide
  should specify whether that is via a second `Composition.author`
  reference or via `Composition.attester` (recommending one).
- Retain all of Disposition A's narrative and examples.

#### Justification

A narrative-only fix (Disposition A) answers the immediate question
but leaves a second ambiguity in place: what happens when an AI
pre-read is subsequently reviewed by a human. Implementers routinely
need to represent both actors. Resolving both questions together
avoids a predictable follow-up ticket and is consistent with how
FHIR-51407 chose to both *describe* the performer slices and
*allow* `EuDevice`/`EuOrganization` in them. The trade-off is scope:
this adds profile/example work beyond the reporter's literal ask and
is slightly larger than a typical "documentation clarification"
change.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive* on the basis that core FHIR's
definitions for `Composition.author` and `Device` already cover the
case: `Composition.author` permits `Device`, and `Device` can be an
AI/software device. No IG-level clarification is strictly required;
implementers can infer the intent from base FHIR.

#### Justification

Defensible only on minimalist grounds: the change is purely editorial
and adds no normative content that is not already implicit. Any
implementer who reads base FHIR carefully would reach the same
conclusion the reporter did.

---

### Recommendation

**Recommended disposition:** A (with an optional path toward B in a
later iteration).

The reporter is not asking for a semantic change — they are asking the
IG to say out loud what the design already assumes. The 2025-08-21
triage comment ("included as to be addressed in this iteration")
indicates the WG already considers this in-scope, and the parallel
resolution of FHIR-51407 has set the pattern for how the guide
discusses device-vs-human-vs-organization actors. Disposition A
matches that pattern exactly and is low risk: no profile change, no
breaking change, no impact on conformant instances.

Disposition C is hard to defend given the explicit implementer
confusion reported here and on FHIR-51407 — declining would leave the
same question to recur. Disposition B is attractive but should be
deferred: the workgroup can ship the clarification now (Disposition A)
and, if the AI-vs-human authoring pattern generates further questions
during the current iteration, open a follow-up to add the profile and
attester guidance.
