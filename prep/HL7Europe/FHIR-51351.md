## Summary

The reporter asks the EU Imaging Study Report work group to close out three
DICOM KOS (Key Object Selection) manifest mapping topics that are still open as
GitHub issues in the `hl7-eu/imaging` repository (issues #51, #49, and #54),
and to ensure their resolutions are reflected in the EU Imaging Study Report
(R5) specification. The two open issues on Referenced SOP Class UID (#51) and
Referenced SOP Instance UID (#49) appear to need clarifying language that the
manifest's `Referenced SOP Class UID` / `Referenced SOP Instance UID` carry the
same values as the underlying DICOM instance's `SOP Class UID` / `SOP Instance
UID` (the manifest just references the instances in the underlying study).
The third issue (#54) covers DICOM endpoint-information attributes — `Retrieve
AE Title` (0008,0054), `Retrieve URL` (0008,1190) and `Retrieve Location UID`
(0040,E011) — and asks the spec to reconcile its mapping against IHE MADO and
the cross-community/cross-border reality, where AE Title is largely
meaningless, the preformatted Retrieve URL is only valid inside the creating
domain, and the URL/Location UID need a clear declaration that they point to a
WADO-RS endpoint. Augury reports no formal cross-references (Jira, Zulip, or
GitHub) for this ticket; the GitHub items above are extracted from the ticket
description.

**Comments:**

### JIRAUSER30266 (2025-07-21)

Issues #51 and #49 just need some clarification: the Referenced SOP Class UID /
Referenced SOP Instance UID carry the same values as the SOP Class UID and SOP
Instance UID of the referenced instance (image) respectively — the manifest
simply references the instances in the underlying DICOM study.

For issue #54, the (0008,0054) "Retrieve AET" is a local imaging source
(PACS/VNA) identifier with no real meaning in a national/regional or
cross-border setting; it exists to assist PACS/VNA addressing when using the
classic DICOM C-services (FIND, MOVE, STORE) inside an enterprise. The
(0008,1190) "Retrieve URL" is a preformatted URL for the referenced series
that can be used directly to retrieve the series instances, but the
retrieval-endpoint type is not conveyed — the consumer cannot tell whether the
URL is WADO-URI, WADO-RS, or other. EHDS DICOM KOS defines this URL as
WADO-RS, but that may not always be the case; the URL is also only valid
within the domain where it was created and is of little use cross-domain. The
(0040,E011) "Retrieve Location UID" can be looked up via an addressing
service, but again the service type is not conveyed; EHDS DICOM KOS defines it
as WADO-RS.

### JIRAUSER30266 (2025-12-16)

The PC version of the IHE MADO Profile clearly defines the attributes
described in #51 and #49.

For #54, "Retrieve AE Title" is not specified by MADO (no use in either Intra
or Cross Community settings) — Retrieve AE Title is only of use in tightly
integrated enterprise solutions.

For #54, "Retrieve URL" is defined in IHE MADO as the Base URI of the WADO-RS
service endpoint. This aligns with its definition in XC-WADO and also in the
WADO-RS Option of XDS-I.b.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Resolve all three issues directly in the EU Imaging Study Report (R5)
specification's KOS mapping section and then close `hl7-eu/imaging` issues
#51, #49 and #54:

- For #51 / #49, add an explicit note in the KOS-to-ImagingStudy mapping
  table stating that the manifest's `Referenced SOP Class UID` and `Referenced
  SOP Instance UID` are required to carry the same values as the
  `SOP Class UID` and `SOP Instance UID` of the referenced DICOM instance, and
  that the manifest references (does not redefine) those instances.
- For #54, in the same mapping table:
  - Mark `Retrieve AE Title` (0008,0054) as **not used** in
    cross-community / cross-border (EHDS) settings, with a note that it
    remains optional only for tightly integrated enterprise solutions, in
    line with the comment that IHE MADO does not specify it.
  - Constrain `Retrieve URL` (0008,1190) to the **WADO-RS Base URI** of the
    service endpoint, citing IHE MADO and the WADO-RS Option of XDS-I.b /
    XC-WADO as the alignment basis, and add an implementer note that the URL
    is only valid within the creating domain.
  - State that `Retrieve Location UID` (0040,E011), when used, designates a
    **WADO-RS** endpoint that is to be looked up via an addressing service.

#### Justification

This is the disposition the most recent comment (2025-12-16) effectively
proposes: align the EU Imaging Study Report KOS mapping with IHE MADO,
XC-WADO and the WADO-RS Option of XDS-I.b, which is consistent with the EHDS
DICOM KOS specification cited in the earlier comment. The clarifications for
#51/#49 are not behavioural changes — they make explicit a constraint that is
already implied by the DICOM model and avoid implementer ambiguity. Pinning
the endpoint-type semantics for #54 closes a real interoperability gap in
cross-community use that the comment thread already documents.

---

### Disposition B: Alternative Approach

#### Proposal

Same substantive content as Disposition A for #51 and #49 (clarifying notes
in the mapping table), but for #54 split the change into two layers:

1. In the EU Imaging Study Report (R5) base specification, *describe* the
   three endpoint attributes neutrally and *recommend* WADO-RS without
   forbidding other endpoint types, so the base spec stays aligned with the
   DICOM standard's broader semantics.
2. Define an EHDS-flavoured profile / IG layer (or extend the existing one)
   that **constrains** Retrieve URL and Retrieve Location UID to WADO-RS, and
   marks Retrieve AE Title as not used cross-community.

#### Justification

The endpoint-type ambiguity is a real-world DICOM modelling problem, not just
an EU-Imaging-Study-Report problem; constraining it only inside the EU Imaging
Study Report base risks creating a divergence with other downstream
specifications that may legitimately use WADO-URI or other endpoint flavours.
Layering the constraint into the EHDS profile keeps the base spec faithful to
DICOM/IHE MADO while still giving EHDS implementers the unambiguous, WADO-RS-
only contract that the comment thread argues for. This follows the FHIR
practice of using profiles to tighten cardinality / semantics for a
jurisdictional use case rather than baking the constraint into a re-usable
core artefact.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the three GitHub issues are themselves
the primary record of the discussion and that any resulting spec change should
be raised as separate, narrowly scoped tickets per attribute rather than as a
single "resolve all of these" change request.

#### Justification

This is defensible only as a process decision, not on technical grounds:
issues #51, #49 and #54 raise distinct questions (two clarifications and one
multi-attribute interoperability change) that would benefit from being tracked
and balloted separately. The ticket as-written does not propose a specific
edit, so any decision the work group makes here would still need follow-on
tickets. The substantive issues themselves should not be declined — the
endpoint-type ambiguity in #54 is a genuine interoperability gap.

---

### Recommendation

**Recommended disposition:** A

The two most recent comments converge on a concrete, IHE-MADO-aligned
resolution and there is no evidence of dissent in either Jira or Zulip. The
clarifications for #51/#49 are essentially editorial and carry no
implementation cost. For #54, accepting the WADO-RS-only constraint inside
the EU Imaging Study Report (R5) is consistent with how the EHDS DICOM KOS
specification, IHE MADO, XC-WADO and the WADO-RS Option of XDS-I.b already
treat these attributes, and it gives EHDS implementers the unambiguous
contract they need. Disposition B is reasonable if the work group decides
the base spec must remain endpoint-agnostic, but absent that constraint the
extra IG/profile layer is unnecessary overhead. Disposition C should only be
selected as a process clean-up if the work group wants per-attribute tickets;
the underlying technical changes still need to land.
