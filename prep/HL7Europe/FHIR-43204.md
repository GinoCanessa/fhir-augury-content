## Summary

The reporter asks how the **XD-Lab "Notification Organizer"** construct
(from the IHE/CEN XD-LAB laboratory report model) should be represented
within the HL7 Europe **EU Laboratory Report (FHIR)** IG. In XD-LAB, the
Notification Organizer is a grouping that may carry information about a
**Notifiable Condition**, **Case Identification**, and **Outbreak
Identification** — concepts that arise when a laboratory result triggers
public-health reporting. The ticket is a Question (not a defect or
change request) and seeks guidance on whether the EU Lab IG already
covers this organizer, and if not, which FHIR resource(s) and profile(s)
should be used to model it. No cross-references to other Jira tickets,
Zulip threads, or GitHub items were found in the FhirAugury index for
this ticket.

### Disposition A: Accept As Requested

#### Proposal

Treat this Question as a request for **explicit IG guidance** on the
XD-LAB Notification Organizer. Add a narrative section (and, if
applicable, a worked example) to the EU Laboratory Report (FHIR) IG
documenting that the Notification Organizer maps to a dedicated
**Composition section** in the EU Lab Report Composition profile, whose
`section.entry` references:

- a **Condition** instance for the Notifiable Condition (bound to the
  EU/ECDC notifiable-condition value set or a placeholder extensible
  binding until one is agreed),
- an **Observation** (or **DetectedIssue**) for Case Identification
  (case identifier, classification: suspect / probable / confirmed),
- an **Observation** for Outbreak Identification (outbreak identifier,
  linkage to the cluster).

Also add a profile-level `section:notification` slice on the EU Lab
Report Composition with a fixed `code` drawn from LOINC (or the LOINC
section code already used by XD-LAB for this organizer) so implementers
can locate the section deterministically.

#### Justification

XD-LAB is the source model the EU Lab IG aligns to, so silence on the
Notification Organizer is itself a defect: implementers serving public
health workflows have no canonical way to round-trip the organizer.
Modelling it as a Composition section rather than inventing a new
resource keeps the IG consistent with how every other XD-LAB organizer
is already represented in document-style FHIR IGs (e.g., IPS, the IHE
PaLM IG). Using `Condition` + `Observation` for the contained items
reuses base resources rather than introducing custom extensions, which
keeps the IG lightweight and interoperable with existing public-health
FHIR work (e.g., eCR / NotifiableConditionList in US Public Health).

---

### Disposition B: Alternative Approach

#### Proposal

Decline to embed notification semantics inside the EU Lab Report itself,
and instead **delegate** to a separate, companion HL7 Europe profile (or
a future "EU Public Health Reporting" IG). The EU Lab Report would
expose only the laboratory findings; a generator would then assemble a
**MessageHeader**- or **Bundle**-based notification payload that
`focus`-references the EU Lab Report and adds:

- a `Condition` for the notifiable condition,
- an `Encounter` or `EpisodeOfCare` to anchor case identification,
- a `Group` or `DetectedIssue` to anchor outbreak identification.

The IG change is therefore limited to (a) a short "Out of scope: see
companion notification profile" note in the EU Lab Report IG, and (b)
opening a tracker against HL7 Europe to scope the companion artifact.

#### Justification

The Notification Organizer is fundamentally a **public-health workflow**
artefact, not a laboratory-result artefact. Embedding it in the EU Lab
Report mixes two governance domains (lab vs. public health reporting)
and risks scope creep that the EU Lab WG cannot fully satisfy alone
(notifiable-condition value sets, case-classification rules, and
outbreak identifiers are all set by national public-health authorities,
not labs). Keeping the EU Lab Report focused and pointing at a
purpose-built notification artefact mirrors how the US realm separated
**eICR** (the trigger document) from the lab report itself, and avoids
imposing notification structures on labs that do not perform statutory
reporting.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — Out of Scope**, with the
explanation that the EU Laboratory Report (FHIR) IG models the
laboratory analytical report only, and that XD-LAB's Notification
Organizer is a public-health-reporting concept addressed by separate IGs
(e.g., a future EU realm equivalent of eCR / Trigger Code, or national
public-health reporting profiles). Direct the reporter to raise the
question against the appropriate public-health workgroup or IG.

#### Justification

The EU Lab Report IG's stated scope is the structured laboratory report;
it deliberately does not cover downstream consumers (clinical pathways,
billing, or public-health surveillance). The XD-LAB document contains
several organizers (e.g., billing, prescription, notification) that
analogous FHIR IGs routinely choose to leave out of scope rather than
re-specify. Declining keeps the IG cohesive and avoids creating partial,
under-specified notification structures that public-health implementers
would then have to work around.

---

### Recommendation

**Recommended disposition:** A

The ticket is a **Question**, not a change request, and the reporter is
explicitly asking how to model an XD-LAB construct that the EU Lab
Report IG is expected to align with. Silently leaving it unaddressed
(Disposition C) would force every implementer to reinvent the same
mapping, and Disposition B's "go build a separate IG" answer is
correct in the long run but does not actually answer the question the
reporter asked today. Disposition A — adding a normative
`section:notification` slice plus narrative guidance, while keeping the
contained content in base `Condition`/`Observation` resources — gives
implementers a deterministic answer now, stays inside the EU Lab WG's
remit (it only constrains how lab-side authors emit the section), and
leaves the door open for a future public-health IG to layer richer
semantics on top without breaking changes. The recommendation should be
revisited if HL7 Europe's public-health track confirms a companion IG
is already imminent, in which case Disposition B becomes the cleaner
long-term answer.
