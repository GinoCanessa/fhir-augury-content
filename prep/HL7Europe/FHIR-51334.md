## Summary

The reporter (JoshPriebe) asks the EU Imaging Study Report (R5) IG authors to
clarify the scope of WADO‑RS support that the `ImWadoEndpoint` profile
implies. The profile's `mimeType` element today reads as if it requires a
fully WADO‑RS‑compliant endpoint, which would oblige servers to support
Study/Series rendered resources, `/metadata`, `/bulkdata`, `/thumbnail`, and
frame‑level retrieval. The reporter argues that the immediate EHDS use cases
only need access to the underlying DICOM objects, and that requiring the full
WADO‑RS surface raises the bar unnecessarily — though some optional features
(thumbnails, rendered images at non‑diagnostic quality) could later be useful
for patient‑facing applications and lightweight EHR previews. The request is
to scope the IG explicitly to the Instance‑level subset of WADO‑RS, in line
with the Multi‑Country Working Group on Imaging Information Sharing's
direction for XC‑WADO.

The two existing comments on the ticket (from JIRAUSER30266) confirm IHE's
direction: the new IHE "Manifest‑based Access to DICOM Objects" (MADO)
profile and the related "Intra‑Community WADO‑RS Retrieve" / "Cross‑Community
WADO‑RS Retrieve" transactions both **mandate Instance Retrieval**
(DICOM Part‑10) and make Rendered Instance Retrieval **optional**, with no
defined support for `/metadata`, `/bulkdata`, etc. Retrieval is supported at
STUDY, SERIES, and INSTANCE levels. The second comment commits to addressing
this in the IG by aligning with the MADO profile so that the WADO requirements
become "a clear subset (supporting the EHDS use cases) of the DICOMweb WADO‑RS
capabilities."

This ticket sits inside a tight cluster of related EU Imaging tickets from the
same reporter — all asking for clearer expectations around the IG's endpoint
profiles ([FHIR-51273](https://jira.hl7.org/browse/FHIR-51273) on access
control across NCPs, [FHIR-51296](https://jira.hl7.org/browse/FHIR-51296) on
IID vs. WADO endpoint expectations, and
[FHIR-51315](https://jira.hl7.org/browse/FHIR-51315) on preferring
ImagingStudy over DICOM KOS for the manifest) plus
[FHIR-51351](https://jira.hl7.org/browse/FHIR-51351) which collates KOS‑related
GitHub‑issue feedback for resolution. Together they describe a coordinated
review of the IG's endpoint and manifest model that should be dispositioned
together.

**Description:**

> mimeType within the ImWadoEndpoint profile suggests full WADO‑RS compliant
> endpoints, which carries additional obligations:
>
> - can support Study/Series rendered resources, in addition to access to DICOM objects.
> - /metadata
> - /bulkdata
> - /thumbnail
>
> The immediate use cases require only access to the DICOM objects, but other
> EHDS use cases could benefit from these features:
>
> - Patient applications (EHDS patient access of data)
> - Non‑dicom apps (Simple GP browser system, EHR imaging overview) could
>   benefit from thumbnails or rendered images for reference quality (not
>   diagnostic quality)
>

> that there are current use‑cases for it in the present IG.
>
> We would suggest that this be clarified so that the community can focus on
> the subset of features that are most important for the immediate workflow
> goals. Focusing only on Instance resources is in line with work the
> Multi‑Country Working Group on Imaging Information Sharing is currently
> considering for XC‑WADO ([XC‑WADO – Google Drive](https://drive.google.com/drive/u/0/folders/1VIy0PMgQY4b9TeVzN1CVKo2hRlIo3ilo)).

## Proposed Dispositions

The dispositions below address the IG narrative + the `ImWadoEndpoint`
profile only; specific file paths cannot be cited because no
`repo-analysis` briefing exists for the EU Imaging IG repository (see "Repo
Context" gap).

### Disposition A: Accept As Requested

#### Proposal

Update the `ImWadoEndpoint` profile and the surrounding IG narrative in the
EU Imaging Study Report (R5) IG to scope WADO‑RS conformance to the
**Instance Retrieval** subset only, mirroring the IHE MADO / XC‑WADO
profiling that is already noted in the ticket comments:

- Restrict the `mimeType` (and any associated conformance text) on
  `ImWadoEndpoint` so that compliance does **not** require support for
  Study/Series Rendered, `/metadata`, `/bulkdata`, `/thumbnail`, or
  frame‑level retrieval.
- State explicitly that **Instance Retrieval (DICOM Part‑10) is mandatory**
  at STUDY, SERIES, and INSTANCE levels, and that **Rendered Instance
  Retrieval is optional**, with all other WADO‑RS options out of scope for
  this IG.
- Reference the IHE "Manifest‑based Access to DICOM Objects" profile and the
  Intra‑Community WADO‑RS Retrieve [RAD‑1xy] / Cross‑Community WADO‑RS
  Retrieve [RAD‑160] transactions as the conformance baseline.

#### Justification

This is what the reporter actually asked for, and it matches the direction
already publicly committed to in the ticket comments (the 2025‑07‑21 comment
promises alignment with MADO; the 2025‑12‑16 comment confirms MADO mandates
exactly this Instance‑only subset). It also keeps the IG aligned with the
Multi‑Country Working Group on Imaging Information Sharing's XC‑WADO
direction, which reduces fragmentation for cross‑border EHDS deployments.
Tightening the profile lowers the implementation bar for NCP‑proxied
deployments — directly easing the deployment concerns raised in the closely
related [FHIR-51273](https://jira.hl7.org/browse/FHIR-51273).

---

### Disposition B: Alternative Approach

#### Proposal

Keep `ImWadoEndpoint` as a single profile but make the WADO‑RS surface
**explicitly tiered via capability declarations** rather than collapsing the
profile to Instance‑only. Concretely:

- Define a **Core conformance level** ("EHDS Instance Access") that mandates
  Instance Retrieval at STUDY/SERIES/INSTANCE levels, aligned with MADO /
  XC‑WADO. This is the only level required for IG conformance.
- Define one or more **optional capability flags** (e.g., extension or coded
  value on `ImWadoEndpoint` such as `supports-rendered-instance`,
  `supports-thumbnail`) that an endpoint MAY advertise to indicate support
  for Rendered Instance, `/thumbnail`, etc.
- Update the narrative to call out the patient‑access and non‑DICOM consumer
  use cases (called out by the reporter) as the motivation for keeping the
  optional capability flags discoverable rather than dropping them entirely.

#### Justification

Disposition A satisfies the immediate use cases but discards the secondary
use cases the reporter himself flagged as valuable (patient applications,
"Simple GP browser" thumbnails, EHR imaging overviews at reference quality).
Removing those capabilities from the profile altogether means each future
IG that wants them has to re‑introduce its own extension. A tiered model
keeps the IG conformance bar low (matching MADO) while leaving a
discoverable, machine‑readable path for richer endpoints — which is more
future‑proof for EHDS' broader patient‑access ambitions. The trade‑off is
that it adds one extension/codeable mechanism that implementers must
understand; the WG should weigh that complexity against the long‑term value
of capability discovery.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave `ImWadoEndpoint` as a reference to fully
WADO‑RS‑compliant endpoints, but add a short **non‑normative narrative
clarification** stating that the IG's immediate use cases only exercise the
Instance Retrieval subset, and that other WADO‑RS capabilities are
out‑of‑scope for the IG even though servers may legitimately support them.

#### Justification

Defensible only on the grounds that (a) WADO‑RS is itself a stable DICOM
specification, (b) the IG profiling layer is intentionally thin and is not
the right place to redefine WADO‑RS conformance, and (c) the existing IHE
MADO / XC‑WADO profiles already constrain the surface for cross‑community
deployments, so the IG can simply *reference* them instead of *redefining*
them. The cost is real: implementers reading the IG today are left to infer
that "WADO‑RS compliant" means the full surface, which contradicts both the
reporter's reading and the WG's own stated direction in the ticket
comments. This option is included for completeness but is hard to defend
given the maintainer's existing on‑record commitment to align with MADO.

---

### Recommendation

**Recommended disposition:** A

The two existing comments on the ticket already commit the IG editors to
exactly this outcome: align `ImWadoEndpoint` with the IHE MADO / XC‑WADO
profiling, which mandates Instance Retrieval and treats the rest of WADO‑RS
as out of scope. Disposition A simply executes that commitment in the IG
artifacts. It also delivers immediate, concrete relief for the cluster of
related tickets ([FHIR-51273](https://jira.hl7.org/browse/FHIR-51273),
[FHIR-51296](https://jira.hl7.org/browse/FHIR-51296),
[FHIR-51315](https://jira.hl7.org/browse/FHIR-51315)) by making the IG's
endpoint conformance footprint as small as the EHDS use cases actually
require. Disposition B is more ambitious and arguably more future‑proof, but
introduces a capability‑discovery mechanism that the WG has not yet
committed to and that the related tickets do not call for; it is a
reasonable follow‑up once the core scope is locked. Disposition C
contradicts what the IG editors have already publicly told the reporter and
should only be revisited if the MADO alignment plan is itself withdrawn.
