## Summary

The reporter (Ulrike Merrick) is asking whether the `Bundle.type` element on
the EU Laboratory Report `Bundle` profile
(`StructureDefinition-Bundle-eu-lab-definitions`) should be constrained to a
fixed value of `document`. The observation is that the current profile leaves
`Bundle.type` unconstrained, even though the stated intent of the artifact is
to convey a **signable laboratory report document**. If the bundle is meant to
function as a clinical document (with a `Composition` as the first entry, a
single subject, etc.), then leaving `Bundle.type` open to other values
(`message`, `transaction`, `collection`, …) is inconsistent with that intent
and would allow conformant instances that cannot actually be processed,
signed, or exchanged as documents. The ticket is logged as a
`Question`/`Triaged` item with the `no-ballot` label, suggesting the reporter
is asking the work group to confirm whether this is an intentional design
choice or a profiling gap that should be tightened.

The cross-reference index surfaced a single mention link from this ticket to
the FHIR element `Bundle.type` and no related Jira tickets, Zulip threads, or
GitHub items.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Constrain `Bundle.type` on the `Bundle-eu-lab` profile to a fixed value of
`document` in the EU Laboratory Report IG. Concretely, on the
`StructureDefinition` for the EU Lab Report Bundle:

- Set `Bundle.type` to `min = 1`, `max = 1` (already implicit in the base
  resource).
- Add `fixedCode = "document"` (or, equivalently,
  `patternCode = "document"`) on the `Bundle.type` element.
- Update the element short/definition to state that this profile only
  represents document Bundles and may not be reused for `message`,
  `transaction`, etc.
- Re-run the IG publisher to regenerate the rendered StructureDefinition
  page and refresh any narrative that currently lists alternative `type`
  values.
- Update example instances and any test fixtures so `Bundle.type =
  "document"`, and adjust validator expectations accordingly.

#### Justification

The IG already shapes the artifact like an IPS-style document bundle: a
`Composition` as the first entry conveys the laboratory report, the bundle
has a single patient subject, and the design intent (per the ticket) is to
support digital signing. Of the values in the `BundleType` value set, only
`document` aligns with that contract — `message`, `transaction`,
`transaction-response`, `batch`, `batch-response`, `history`, `collection`,
and `subscription-notification` all imply different processing semantics,
none of which are exercised by a signable lab report. Fixing
`Bundle.type = "document"` makes the profile self-describing, lets validators
catch malformed instances at conformance time rather than at signing time,
and is consistent with how IPS, the International Patient Summary, and other
"document" profiles in the FHIR ecosystem handle the same element.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than fixing `Bundle.type` directly on `Bundle-eu-lab`, restructure the
profile family so that the constraint comes from a shared base profile:

- Introduce (or reuse, if one already exists) a `Bundle-eu-document` base
  profile that fixes `Bundle.type = "document"` and asserts the
  document-bundle invariants (`bdl-9`, single Composition first entry, single
  subject, etc.).
- Re-derive `Bundle-eu-lab` from that base profile, leaving any
  laboratory-specific slicing (Composition profile, mandatory entries such as
  `DiagnosticReport`, `Observation`, `Specimen`, `Patient`) on the
  laboratory-specific profile only.
- Update narrative on the laboratory profile to point readers at the base
  document profile for the generic document-bundle rules.

#### Justification

The HL7 Europe Laboratory Report is one of several HL7 Europe document-style
artifacts (Patient Summary, Discharge Summary, Imaging Report, etc.).
Distributing the "is a document Bundle" constraint across each leaf profile
duplicates rules and risks divergence — for example, one profile fixing
`type = "document"` while another only marks it `mustSupport`. Centralising
the constraint in a shared base aligns the European IGs with how the
international IPS guide layers `Bundle-document → Bundle-ips`, makes future
cross-IG reuse cheaper, and still produces the same downstream effect for
laboratory consumers (the rendered EU Lab Bundle still requires
`type = "document"`). The trade-off versus Disposition A is a larger,
cross-IG editorial change and a coordination cost with other HL7 Europe
projects, which is why Disposition A may be preferred if no shared base
profile is currently in scope.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** / **Already Addressed**, on the basis
that the EU Laboratory Report Bundle is intentionally permissive at
`Bundle.type` because (a) the `BundleType` value set is already a
`required`-binding controlled vocabulary in the base FHIR specification, so
non-document values are semantically distinct and unambiguous, and (b) some
EU jurisdictions exchange the same logical lab-report content as a
`message` (for HL7 v2-style choreographies) or a `collection` (for
back-end persistence), and a hard `fixed = "document"` would break those
flows. Document the expectation that `type = "document"` for signed
exchange in narrative guidance instead of in the StructureDefinition.

#### Justification

If the work group has explicit evidence of non-document usage of the EU Lab
Report content (messaging, batch import, persistence-only collections), then
fixing `Bundle.type` removes a legitimate degree of freedom for implementers
and forces them to define a parallel profile to convey the same payload in
a non-document context. Declining is also defensible if signing is performed
at the `Composition` / `provenance` level rather than over the bundle
envelope, in which case the `type` value is not what guarantees
"signability".

---

### Recommendation

**Recommended disposition:** A

The ticket reads as a straightforward profiling gap rather than a design
debate. The IG's stated goal — a signable laboratory report — only makes
sense for `Bundle.type = "document"`; the other values in the `BundleType`
value set carry processing semantics (message routing, transactional
side-effects, batch responses, history, subscription notifications) that the
laboratory report profile does not exercise and that downstream signers and
validators cannot meaningfully act on. There is no Zulip or Jira evidence in
the cross-reference index that EU implementers are exchanging this content
as anything other than a document, and the cost of the change is small
(adding `fixedCode = "document"` on a single element and refreshing
examples). Disposition B is the right move **if** HL7 Europe is willing to
introduce or adopt a shared `Bundle-eu-document` base profile in this
ballot cycle; if that is out of scope, fold the constraint into
`Bundle-eu-lab` directly per Disposition A and leave the refactor as a
follow-up tracker. Disposition C should only be selected if the work group
can point to a concrete, supported non-document use of this profile, which
the current ticket and surrounding evidence do not surface.
