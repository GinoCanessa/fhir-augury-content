## Summary

The ticket is a January-2026 EU ballot negative-vote question against the
**EU Base and Core (R4)** specification. The reporter (Alexander Henket)
observes that the abstract logical model element `header.authorship` has no
direct counterpart on `Observation` — `Observation` has neither an `author`
element nor an established extension for authorship. He proposes that
`Observation.performer` is the closest analogue and is "equivalent enough"
to satisfy the generic authorship requirement, and asks the EU work group
to confirm so the matter can be closed rather than being left as a
follow-up problem for downstream IGs.

The cross-reference index surfaces only the FHIR element
`Observation.performer` (extracted from the description); there are no
linked or related Jira tickets, no GitHub items, and no related Zulip
threads in the indexed corpus.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Formally close the ticket by adopting the position that, in the EU Base
and Core (R4) logical model, the abstract `header.authorship` concept is
mapped to **`Observation.performer`** for the `Observation`-shaped
artifacts, and that `author[x]` is intentionally left unmapped on
`Observation` because `performer` already covers the requirement.
Reflect this in the EU Base and Core mapping documentation / narrative
for `Observation` (and in any logical-model → FHIR mapping table that
currently shows an unmapped row), citing `Observation.performer`
(`Reference(Practitioner | PractitionerRole | Organization | CareTeam |
Patient | RelatedPerson)`) as the authorship analogue. No structural
change to the profile is required; this is a documentation/mapping
clarification.

#### Justification

Per giorgio's comment, the underlying logical model already moved in
this direction in v0.3.0 (the abstract `header.authorship` element was
removed; `author[x]` on `Observation` was deliberately left unmapped
because `performer` is in the model). simone heckmann's comment ("Keep
it simple") and the reporter's own framing both endorse this resolution.
Treating `Observation.performer` as the authorship target is consistent
with FHIR R4 practice — `Observation` exposes `performer` precisely to
identify the actor responsible for the observation, which is the
operational meaning of "author" in this context. Closing the ticket on
this basis discharges the ballot question without adding a new element
or extension and without leaving a TODO for downstream IGs.

---

### Disposition B: Alternative Approach

#### Proposal

Accept that `Observation.performer` is the *primary* mapping for
`header.authorship`, but explicitly carve out the case where authorship
and performance need to be distinguished (e.g., a clinician *authoring*
a recorded observation that was *performed* by a device or another
party). For that case, point implementers to the existing core
extension `http://hl7.org/fhir/StructureDefinition/event-recorder` (or
an equivalent EU extension if one is preferred) on `Observation` to
carry the recorder/author when it differs from the performer. Document
both halves of the mapping in the EU Base and Core narrative for
`Observation`: `header.authorship` → `Observation.performer` (default),
with `event-recorder` available when the two roles diverge.

#### Justification

This preserves the simplicity of Disposition A for the common case
while acknowledging the small but real set of scenarios where author
≠ performer (a frequent point of confusion that has surfaced in other
HL7 contexts and is the usual reason core resources without `author`
attract negative votes). It avoids profiling a new element onto
`Observation` and reuses an extension that already ships with FHIR R4,
so it imposes no additional maintenance burden on EU Base and Core. The
trade-off vs. A is one extra paragraph in the mapping narrative in
exchange for a defensible answer if a reviewer follows up with "what
about the recorder?".

---

### Disposition C: Decline

#### Proposal

Decline the request to formally map `header.authorship` to
`Observation.performer` and instead leave the row explicitly *unmapped*
in the EU Base and Core logical-model → FHIR mapping, on the grounds
that `performer` and `author` are not semantically identical (performer
= who carried out the observation; author = who is responsible for the
recorded statement) and that conflating them in a normative mapping
risks misimplementation.

#### Justification

A defensible "decline" would rest on the FHIR R4 modelling stance that
`Observation` has no `author` element by design and that pretending
`performer` is equivalent could mislead implementers in jurisdictions
where authorship has legal weight distinct from performance. However,
this position is contradicted by the reporter, by both commenters, and
by the model change already made in v0.3.0, so it would amount to
re-opening a settled question.

---

### Recommendation

**Recommended disposition:** A

The author-of-record on this ticket, both commenters, and the v0.3.0
model change all converge on the same answer: `Observation.performer`
is the authorship mapping for `Observation`, and no new element or
extension is needed. Accepting as requested simply documents that
existing decision in the EU Base and Core narrative and closes the
ballot item cleanly. Disposition B is a reasonable upgrade if the
work group wants to pre-empt a future "what about recorder vs. author?"
question, and could be folded into A as a non-normative note without
changing the disposition. Disposition C is included for completeness
but is not defensible on this record — the v0.3.0 change has already
committed the model in the direction of A.
