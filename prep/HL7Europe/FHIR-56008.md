## Summary

The reporter (WHO-UMC) asks the EU Medication Prescription and Dispense
(MPD-R5) IG editors to add concrete worked examples for `MedicationRequest`
and `MedicationDispense` that include a global Pharmaceutical Product
Identifier (PhPID). Two example PhPIDs are supplied — `UTPHPID3X3H0DLA27Y1BE`
(Simvastatin 40mg tablets) and `UTPHPID5HCC15YE637XRB` (Enoxaparin sodium
10000IU/ml solution for injection) — together with their dose-form attributes
and the GSID substance definitions they reference. Because the IG has not yet
decided whether a PhPID should be carried as an `Identifier`, a coded
`CodeableConcept` (`Medication.code`), or as a higher-level classification,
the reporter requests that examples be produced for *all three* usage
scenarios so implementers can see them side-by-side. The ticket also asks
that the IDMP / PhPID story be explained more fully in the descriptive
text on the [medicationconcepts](https://build.fhir.org/ig/hl7-eu/mpd-r5/medicationconcepts.html)
and [terminology](https://build.fhir.org/ig/hl7-eu/mpd-r5/terminology.html)
pages so readers understand what the new examples illustrate.

There are no cross-referenced Jira tickets, GitHub items, or Zulip threads
linked from the ticket itself, but historical Zulip discussion about how
to use PhPID in FHIR (notably *BRR – Pharmacy model* and the older *IPS –
PHPID* topic) is directly relevant background and is cited below.

## Details

**Description:**

Create examples that include global PhPIDs to illustrate how they are used
in prescriptions and dispenses. It has not been decided exactly how the
PhPID should be included — it could act as either identifier, code, or
classification — so examples for all scenarios would be good.

PhPIDs that could be used in examples:

- **PhPID 1:** `UTPHPID3X3H0DLA27Y1BE` (Simvastatin 40mg tablets) — usable
  for generic products available with many different trade names in
  European countries.
- **PhPID 2:** `UTPHPID5HCC15YE637XRB` (Enoxaparin sodium 10000IU/ml
  solution for injection) — usable for the product *Clexane* prefilled
  syringes, available in many European countries.

Attached are example files with all details for the PhPIDs above (e.g.,
dose-form attributes), and substance details for the GSIDs referenced from
the PhPIDs. IDMP and PhPID should also be included and described better in
the descriptive texts so it is clear what the examples with PhPIDs mean.
References:

- https://build.fhir.org/ig/hl7-eu/mpd-r5/medicationconcepts.html
- https://build.fhir.org/ig/hl7-eu/mpd-r5/terminology.html

**Comments:**

- **Rutt Lindström — 2026-03-05.** Reports that the topic was discussed at
  length in the 05.03.2026 MPD call and tables a structured set of
  questions to WHO-UMC (ÅsaPärnaste, MagnusWallberg). (1) To use PhPID as
  a *classification* it must be a `CodeableConcept` — what code-system
  URL should be used, and do PhPIDs have display names? (2) When the
  PhPID *is* the medication, it could be carried as `CodeableConcept` or
  `Identifier`; if as identifier, defining information (substance, dose
  form, strength) probably still needs to be present in `Medication`. How
  should dose form be modeled — extensions on `doseForm`, plain dose
  form, or a mapping to a higher-granularity terminology? The substance
  is currently a `SubstanceDefinition` reference but a `CodeableConcept`
  would be needed — what code/code-system? Or is bare PhPID-as-identifier
  enough? (3) Are there IP / licensing concerns and is a copyright
  statement needed in the IG?

- **ÅsaPärnaste — 2026-03-06.** Answers the easy points: (1.2) PhPIDs do
  not currently have display names. (2.1) Dose-form mapping has been
  flagged before (WGM December) and is genuinely hard to prescribe; the
  global IDMP intent is that the PhPID dose-form attributes are the
  harmonization layer and there is no plan to map back to e.g. EDQM PDF.
  WHO-UMC currently uses extensions for the dose-form attributes because
  APD does not carry that granularity, and the same may need to be done
  in MPD-R5. (2.3) Carrying just the PhPID as identifier is technically
  possible but would force prescribing/dispensing systems to look up the
  PhPID externally to learn what product it represents, so EMA's plans
  for PhPID adoption in PMS are highly relevant.

- **ÅsaPärnaste — 2026-03-09.** Provides the missing URLs: (1.1) the PhPID
  code-system URL will be `http://umcterminologies.org/phpid` (live in
  about a month, but safe to reference now); (2.2) the substance/GSID
  code-system URL will be `http://umcterminologies.org/gsid`. Asks
  whether display names are wanted from the IG side — for GSIDs the
  preferred substance name would be straightforward, but PhPID display
  names would be long and structured (built from the four dose-form
  attributes plus, for combinations, multiple substance + strength
  fragments).

- **JIRAUSER29546 — 2026-03-20.** Agrees with Åsa on dose-form handling
  (2.1): use the dose-form *extensions* (as WHO-UMC does) for now.
  Highlights a concrete failure mode if extensions are not used: products
  with the same substance + base dose form but differing release
  characteristics (e.g., regular vs. prolonged-release oxycodone /
  morphine tablets) would otherwise look identical, and only the PhPID
  plus the dose-form extensions would distinguish them.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Author and ship a triplet of new example bundles in the MPD-R5 IG that
demonstrate every plausible PhPID placement, using the two PhPIDs the
reporter supplied:

1. **PhPID-as-classification** — a `Medication` resource whose
   `Medication.code` carries the PhPID as a `CodeableConcept`
   (`system = http://umcterminologies.org/phpid`, `code =
   UTPHPID3X3H0DLA27Y1BE`), with no display name (per Åsa's 2026-03-06
   comment), referenced from a `MedicationRequest` and a
   `MedicationDispense`. Substance is provided via a
   `SubstanceDefinition`-style `CodeableConcept` on `Medication.ingredient`
   using `system = http://umcterminologies.org/gsid`.
2. **PhPID-as-Medication-identifier** — a `Medication` resource carrying
   the PhPID in `Medication.identifier` (same system URI), still
   populating `Medication.code`, `doseForm`, and `ingredient` with the
   defining attributes (per Åsa 2.3 / Rutt 2-summary) so a system that
   cannot resolve the identifier still has the clinical meaning.
3. **PhPID-as-product-classification** — analogous to #1 but where the
   PhPID is the *grouping* code on `Medication.code` and a separate,
   trade-name-bearing `Medication` (e.g., "Clexane prefilled syringe")
   references it (e.g., via a contained or referenced second
   `Medication`, or via the existing classification mechanism the IG
   uses for branded vs. generic).

In all three cases, model the four IDMP dose-form attributes as
*extensions* on `Medication.doseForm` (per the 2026-03-20 comment) so
that release-characteristic variants such as regular vs. prolonged-release
tablets remain distinguishable. Also extend the
[medicationconcepts](https://build.fhir.org/ig/hl7-eu/mpd-r5/medicationconcepts.html)
and [terminology](https://build.fhir.org/ig/hl7-eu/mpd-r5/terminology.html)
narrative pages with: (a) a short IDMP/PhPID primer; (b) a register of
the two new code systems (`http://umcterminologies.org/phpid`,
`http://umcterminologies.org/gsid`) with a note that PhPIDs do not
currently have display names; (c) cross-links to each of the three new
examples; and (d) a copyright/IP placeholder pending WHO-UMC's answer
to question (3).

#### Justification

This is exactly what the reporter and the 05.03.2026 MPD call asked
for. Providing all three examples is the only way the work group can
make the *usage* decision honestly, because today the choice is being
debated in the abstract (see the 2023 BRR – Pharmacy model thread, where
PhPID-as-classification vs. PhPID-as-identifier was unresolved). The
code-system URLs are now confirmed and stable per the 2021 IPS thread
and Åsa's 2026-03-09 comment, so referencing them is safe. Modeling
dose-form attributes as extensions follows the WHO-UMC reference
implementation and is the only proposal in the thread that survives the
oxycodone / morphine prolonged-release counter-example raised on
2026-03-20.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt **PhPID-as-classification on `Medication.code`** as the IG's
*recommended* pattern and ship a single, fully-worked example for each
of the two supplied PhPIDs (Simvastatin 40 mg tablet; Enoxaparin
10000 IU/ml solution for injection), used end-to-end through
`MedicationRequest` → `MedicationDispense`. The `Medication.code` carries
the PhPID `CodeableConcept` (`system = http://umcterminologies.org/phpid`),
ingredient codes use GSIDs (`system = http://umcterminologies.org/gsid`),
and the four IDMP dose-form attributes are conveyed via a named extension
on `Medication.doseForm` defined in the IG. A short narrative section
explicitly lists "PhPID as `Medication.identifier`" and "PhPID as a
higher-level classification of a branded `Medication`" as *alternative,
non-recommended* patterns, with a one-paragraph rationale for each, but
does not provide running examples for them. Update the
medicationconcepts / terminology pages as in Disposition A.

#### Justification

The reporter's underlying need is "implementers can see how to use a
PhPID"; this is satisfied by one canonical example per PhPID and is
cheaper to maintain than three parallel example sets. Picking
classification-on-`Medication.code` aligns with the dominant view in the
2023 BRR – Pharmacy model thread (PhPID = classification in the IDMP
sense) and with the WHO-UMC reference solution (extensions on dose
form). It also avoids the well-known pitfall of identifier-only
modelling raised by Åsa (2.3): downstream prescribing and dispensing
systems would otherwise need an external lookup just to know what the
product is. The trade-off vs. Disposition A is reduced optionality and a
slight risk of locking in a pattern before EMA's PMS plans for PhPID
are public; this is mitigated by keeping the alternatives documented as
non-recommended patterns.

---

### Disposition C: Decline

#### Proposal

Defer the addition of PhPID examples until (a) the WHO-UMC PhPID code
system is actually live at `http://umcterminologies.org/phpid` (Åsa
notes "in about a month" as of 2026-03-09), (b) the IP / licensing
question (Rutt's question 3) has a definitive answer, and (c) EMA's
plans for PhPID in PMS are clear enough to know whether identifier-style
or classification-style use is normative. In the meantime, do not change
the IG; close the ticket as "Deferred / Persuasive with mod" with a
pointer to the next ballot cycle.

#### Justification

A defensible — but, on balance, weak — case can be made that publishing
examples that reference a code system that is not yet live, has no
display names, and whose modelling pattern (identifier vs. code vs.
classification) is still openly under debate may bake unfortunate
assumptions into the IG narrative. Declining now also avoids any
copyright risk while the licensing answer is outstanding. The
counter-arguments are substantial: the URL is already reserved, Åsa has
confirmed it is safe to reference, and the WG explicitly *cannot* make
the modelling decision until it sees worked examples — so deferral
mostly perpetuates the very paralysis the ticket exists to break.

---

### Recommendation

**Recommended disposition:** A

The MPD-R5 work group is currently stuck deciding between three plausible
PhPID placements (identifier, code, classification) without a concrete
artifact to argue from — that is precisely why this ticket was raised
after the 05.03.2026 MPD call, and the same impasse is visible in the
2023 BRR – Pharmacy model Zulip thread. Disposition B is attractive for
maintenance reasons but it requires the work group to make the modelling
choice *first*, which is exactly what they have been unable to do.
Disposition C concedes defeat for procedural reasons that are now thin:
the code-system URLs are confirmed stable (per the 2021 IPS thread), the
WHO-UMC reference data is in hand, and Åsa has explicitly signed off on
referencing the URLs ahead of go-live. Disposition A is the only path
that *unblocks the design conversation*; once the WG sees the three
examples side-by-side it can pick a recommended pattern and demote the
others (effectively converging on Disposition B) in a follow-up ticket
without losing any work. The example authoring should follow the
2026-03-20 guidance to model the four IDMP dose-form attributes as
extensions, since failing to do so produces the indistinguishable
oxycodone/morphine counter-example and would invalidate the examples
the moment a real implementer tried to use them.
