## Summary

Alexander Henket (HL7 Europe ballot reviewer) asks how the EU Base and Core
(R4) IG documents the mapping strategy when an Xt-EHR / EHDS logical model
could legitimately map to either a **Base** profile or a **Core** profile.
From the IG's mapping landing page, a reader is funnelled into a single
mapping page per model — there is no obvious way to choose between the
"map to Core" view and the "map to Base" view, and the IG does not state
what to do when that distinction matters. The ticket was filed as a
NegativeVote against the January 2026 EU ballot and is a documentation /
strategy clarification rather than a request for a structural change.

The ticket's own resolution (recorded by HL7 Europe) commits to three
points: (1) Xt-EHR models are represented by **core** profiles;
(2) the mapping specified for a core profile is **also valid for the
corresponding base profile** when one exists; and (3) when other models
are defined (e.g., a Common Cancer model), a **dedicated** map table will
be specified, with the core or the base chosen as the target according to
context.

No direct cross-references to other Jira tickets, Zulip threads, or
GitHub items were recorded in Augury for FHIR-55421. However, several
HL7 Europe ballot tickets in the same January 2026 cycle ask closely
related questions about the mapping-table design and the relationship
between Base and Core profiles — most notably
[FHIR-55538](https://jira.hl7.org/browse/FHIR-55538) (Improve the
mapping tables design),
[FHIR-55461](https://jira.hl7.org/browse/FHIR-55461) (Page text
revision, including mapping pages),
[FHIR-55534](https://jira.hl7.org/browse/FHIR-55534) (Remove
MedicationRequest Core profile and/or mappings), and
[FHIR-51124](https://jira.hl7.org/browse/FHIR-51124) (Move some of the
mapping to the base/core profiles in the EU Imaging guide). FHIR-55421
sits at the documentation-strategy level above these more concrete
proposals.

## Proposed Dispositions

The reporter (HL7 Europe) has already recorded a resolution of
*Considered - Question answered* with a three-point answer. The
dispositions below frame how the ballot workgroup might formally close
the comment.

### Disposition A: Accept As Requested

#### Proposal

Codify the three-point answer already in the ticket resolution as a
new short subsection in the EU Base and Core (R4) IG mapping-strategy
narrative. Specifically:

1. State explicitly that **Xt-EHR (EHDS) logical models are
   represented by Core profiles** in this IG; the mapping table on
   any model page therefore targets the Core profile.
2. State that **mappings specified against a Core profile also apply
   to the corresponding Base profile**, where a Base profile exists,
   so a separate "map to Base" page is unnecessary by default.
3. State that **future models that do not follow this rule** (e.g., a
   Common Cancer model) will get their **own dedicated map tables**,
   and each such table will explicitly identify whether its target is
   a Base or a Core profile.

Add a paragraph on the mapping landing page that links the reader to
this subsection so the question Alexander raised ("which mapping page
am I on?") is answered before the reader clicks through to a single
mapping page. Mark the comment **Considered - Question answered** in
the ballot reconciliation.

#### Justification

This is exactly the answer HL7 Europe has already converged on, as
recorded in the Jira resolution. Writing it into the IG narrative
turns a private answer into a public, ballotable rule and prevents the
same question recurring on the next ballot. It also gives a stable
hook that the more concrete tickets — FHIR-55538 (table design),
FHIR-55461 (page text revision), and FHIR-55534 (MedicationRequest
Core scope) — can reference without each having to re-argue the
strategy. Because the change is documentation-only on the EU Base &
Core IG narrative and does not touch any StructureDefinition,
ConceptMap, or example, it carries no implementer or downstream-IG
risk.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a single narrative paragraph, restructure the mapping
landing page so each model entry **declares its mapping target
explicitly** (one of: *Core only*, *Core + Base (mapping is shared)*,
*Dedicated table — target Base*, *Dedicated table — target Core*) and
links to the appropriate page(s). Where a Core mapping is also valid
for the Base profile (the rule from point 2 of the resolution),
generate a stub Base mapping page automatically that simply
transcludes/redirects to the Core mapping. This is closer in spirit to
FHIR-55538's "Improve the mapping tables design" and FHIR-51124's
"Move some of the mapping to the base/core profiles" proposals, and
also addresses FHIR-25704's class of complaint (mapping tables that
don't tell the reader what they represent).

#### Justification

Disposition A answers the question but leaves the navigation problem
intact: the reader still lands on a single page and has to remember
the rule. Disposition B makes the rule visible at the point where the
reader is making the decision, and it lines FHIR-55421 up with the
already-Applied tickets FHIR-55538, FHIR-55461, FHIR-51124 so they
deliver a consistent reader experience together rather than each
fixing a slice of the same problem. Trade-off: more authoring effort
and at least one new generated artifact per shared-mapping model, plus
co-ordination with the IG publisher (or template) so the
transclusion / redirect behaves on every render.

---

### Disposition C: Decline

#### Proposal

Decline to take any further action on FHIR-55421 on the grounds that
the ticket is a *Question* (not a change request), it is already
marked **Considered - Question answered** with a substantive
three-point answer in the Jira record, and the structural improvements
the question hints at are tracked elsewhere — primarily by FHIR-55538
(table design), FHIR-55461 (mapping page text revisions), and
FHIR-51124 (move generic mappings into Base/Core).

#### Justification

The ticket is technically already resolved, and HL7 Europe explicitly
chose the *question answered* resolution rather than *persuasive*. If
the workgroup considers the answer adequate as a reviewer-direct
reply, no IG change is strictly required. Declining keeps the ballot
queue lean and avoids two tickets writing competing narrative for the
same paragraph (this ticket vs. FHIR-55461). The risk is real,
though: the comment came in as a NegativeVote, the reporter clearly
*didn't* find the answer in the published material, and leaving the
narrative untouched almost guarantees the same question on the next
ballot.

---

### Recommendation

**Recommended disposition:** A

The reporter, an experienced ballot reviewer, could not infer the
Core-vs-Base mapping policy from the published IG even though HL7
Europe already had a settled answer. That is the failure FHIR-55421 is
really pointing at, and Disposition A fixes it directly with a
documentation change of well-bounded cost: write the agreed three-point
rule into the EU Base and Core (R4) IG mapping-strategy narrative and
link to it from the mapping landing page. It uses the resolution that
HL7 Europe has already endorsed, it converts the NegativeVote into a
*Considered - Question answered* with a public artifact the reviewer
can be pointed at, and it gives FHIR-55538, FHIR-55461, FHIR-55534,
and FHIR-51124 a single canonical statement of the rule to align on.
Disposition B is the right *next* step once those concrete tickets
land — but doing it inside FHIR-55421 conflates a documentation
clarification with a structural redesign and risks blocking the
reconciliation of an already-answered Question on the larger debate.
Disposition C is not recommended: the NegativeVote signal indicates
the existing material does not in fact answer the question on its own.
