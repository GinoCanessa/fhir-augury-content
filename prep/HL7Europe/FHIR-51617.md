## Summary

The reporter (Hynek Kružík) observes that, in the EU Imaging Study Report
(R5) IG, `Composition.type` is bound (extensible) to the locally-defined
`HL7EuDocumentTypes` code system, which currently contains only a single
code — `imaging-report-v1-0.0` (definition: "ImagingManifest v 1.0.0").
That code does not actually describe an imaging study/document and so the
binding does not convey the intended meaning of an imaging study report.
The reporter proposes binding `Composition.type` to a more meaningful
external concept, suggesting LOINC `18748-4` ("Diagnostic imaging study")
and/or SNOMED CT `4201000179104` ("Imaging report") and its descendants.

The same conceptual issue is being chewed on in several sibling tickets
in the same IG. [FHIR-51171](https://jira.hl7.org/browse/FHIR-51171)
("Composition: remove fixed code for type") asks to remove the same
hardcoded `imaging-report-v0-0-1` value and its extensible binding to
`ImImagingReportTypesEuVS`, arguing it functions like a `formatCode`
and should not be embedded in the document instance. That ticket has
been resolved Persuasive with Modification and is now Applied; the
agreed approach is captured on the HL7 Europe Confluence page "The use
of category and type in the IG" and aligns `Composition.type` with
`DiagnosticReport.code` (LOINC document-type slice such as 18748-4) and
`Composition.category` with `DiagnosticReport.category`.
[FHIR-51388](https://jira.hl7.org/browse/FHIR-51388) is the parallel
manifest concern. The proposed resolution on FHIR-51617 itself does
**not** change the binding as the reporter requested; instead it (a)
documents the choice in the design decisions, (b) removes the manifest
codes from the code system, and (c) renames `HL7EuDocumentTypes` to
`Hl7EuImagingDocumentTypes`. It also notes that the manifest has been
split into a separate IG, so manifest-related codes no longer belong in
this code system. Comment from `bvdh` (2025-08-12) flags that this was
pulled from the block-vote because the proposed resolution does not
match the discussion on Composition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-bind `ImComposition.type` (in the EU Imaging Study Report (R5) IG,
`hl7-eu/imaging` repo) to a value set whose primary code is
LOINC `18748-4` "Diagnostic imaging study", with optional inclusion of
SNOMED CT `4201000179104` "Imaging report" and its descendants. Drop
the local `imaging-report-v1-0.0` code from `HL7EuDocumentTypes` for
this binding. Keep the binding strength **extensible** (matching the
current strength) so jurisdictions can add national codes if needed.

#### Justification

The reporter is correct that the present binding does not describe an
imaging study/document; the single local code in `HL7EuDocumentTypes`
literally points at "ImagingManifest v 1.0.0", which is a profile/format
artifact, not a clinical document type. LOINC 18748-4 is the
standards-community default for "diagnostic imaging study" and is
already the value the discussion in FHIR-51171 (comments by
JoshPriebe, IgnacioJauregui) converged on as the document-type code.
Using a real terminology binding rather than a self-referential local
code is also consistent with Giorgio Cangioli's observation in
FHIR-51171 that "the spec itself should define standard codes, and not
just be self-referential."

---

### Disposition B: Alternative Approach (recommended pattern from FHIR-51171)

#### Proposal

Adopt the resolution already applied for FHIR-51171 in the same IG:
align `Composition.type` with `DiagnosticReport.code` and
`Composition.category` with `DiagnosticReport.category`, modelled on the
EU Laboratory Report IG. Concretely, in `hl7-eu/imaging`:

1. On `ImComposition` (the EU Imaging Study Report Composition profile),
   slice `Composition.type` so a 1..1 high-level slice carries
   LOINC `18748-4` (or, per the FHIR-51171 discussion, LOINC `81220-6`
   with the more specific codes drawn from `$loinc where ancestor =
   #18726-0`), with slicing rules **open** so implementers can add
   detailed procedure codes.
2. Replace the binding of `Composition.type` to `HL7EuDocumentTypes`
   with a binding to the resulting value set (extensible).
3. Apply the bookkeeping changes from the current proposed resolution:
   document the design decision on the "Design considerations" page
   (mirroring the rationale captured in the HL7 Europe Confluence page
   "The use of category and type in the IG"), remove the now-orphan
   manifest codes from the code system, and rename
   `HL7EuDocumentTypes` → `Hl7EuImagingDocumentTypes`.
4. Coordinate with FHIR-51388 / the split-off Manifest IG so the
   equivalent code-system rename and code removal are done once and
   consistently across both IGs.

#### Justification

This is precisely why `bvdh` pulled FHIR-51617 out of Block-Vote-2: the
proposed "Not Persuasive with Modification" resolution only does the
janitorial work (remove manifest codes, rename the code system) and
silently leaves the underlying binding-meaning problem unaddressed,
while the parallel discussion on FHIR-51171 has already produced a
concrete, consensus-backed redesign of `Composition.type` that fully
absorbs FHIR-51617's request. Re-using that resolution avoids two
divergent fixes for the same element, keeps the EU Imaging IG aligned
with the EU Laboratory IG (a stated design goal in FHIR-51157 and
FHIR-51171), and gives the reporter's LOINC 18748-4 suggestion a
principled home (as the high-level document-type slice). It also
satisfies the housekeeping the current proposed resolution intends
(rename + manifest-code removal) without losing the substantive
binding fix.

---

### Disposition C: Decline (close as duplicate of FHIR-51171)

#### Proposal

Close FHIR-51617 as **Not Persuasive — Already Resolved**, citing
FHIR-51171 (Applied, PR #84 / commit `2f60ce3`) as the disposition of
record for `Composition.type`. Track only the residual cleanup
(rename `HL7EuDocumentTypes` → `Hl7EuImagingDocumentTypes`, remove
manifest codes) under FHIR-51171's follow-up commits or a small new
ticket, rather than under FHIR-51617.

#### Justification

The substantive binding decision for `ImComposition.type` has already
been made in FHIR-51171: align with `DiagnosticReport.code` via a sliced
binding that includes LOINC `18748-4` (the same code the FHIR-51617
reporter proposed). Keeping FHIR-51617 open risks a divergent or
contradictory disposition for the same element. A duplicate-style
closure preserves the audit trail (the reporter's request was heard and
is satisfied by FHIR-51171's outcome) without re-litigating the design.

---

### Recommendation

**Recommended disposition: B (Alternative Approach — adopt the
FHIR-51171 pattern).**

The current draft resolution on FHIR-51617 ("Not Persuasive with
Modification") is exactly the wrong shape: it does the housekeeping
(rename code system, drop manifest codes) but does not change the
binding the reporter complained about, which is why `bvdh` pulled it
from the block-vote. Outright accepting the request as written
(Disposition A) gets the right code in but creates a parallel,
inconsistent design path with the already-Applied FHIR-51171.
Outright declining as a duplicate (Disposition C) is defensible but
loses the housekeeping work (code-system rename, manifest-code removal,
manifest IG coordination) that this ticket has been used to track and
that FHIR-51171's resolution does not by itself cover. Disposition B
explicitly folds the FHIR-51171 design into FHIR-51617's resolution,
keeps the EU Imaging IG aligned with EU Laboratory and IHE IDR
expectations, and gives implementers (and the reporter) a single
coherent answer about what `Composition.type` will carry going forward.
