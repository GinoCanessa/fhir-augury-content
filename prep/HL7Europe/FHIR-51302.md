## Summary

The ticket asks the EU Imaging Study Report (R5) authors to revisit how
"impressions" are represented in the imaging report Composition. The
reporter argues that whether a piece of clinical content is a *finding*
or an *impression* should not be determined by its position in the
Composition (since the Composition is essentially a rendering of the
content); instead, the nature/status of findings should be derivable
from the underlying clinical resources themselves. As a concrete
alternative, the reporter suggests that the set of clinical
impressions might be better represented using the dedicated
`ClinicalImpression` resource rather than as free-form Observations
slotted into a particular Composition section. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were found by the
augury index, so the request currently stands on its own.

## Details

**Description:**

The current specification represents impressions as a set of findings
and impressions, whether they are a finding or an impression should not
be defined by the position in the composition resource (which is a
rendering of the content). The status of findings should be derivable
from the clinical content.

Also the set of clinical impressions might be better represented by
using the ClinicalImpression resource.

## Proposed Dispositions

> Note: Because no `repo-analysis` briefing is available for the EU
> Imaging Study Report (R5) source repo, the dispositions below name
> the spec artifacts conceptually (Composition profile, section
> definitions, and the resources that populate the report) rather than
> concrete file paths. The HL7 Europe imaging team should map these to
> the corresponding StructureDefinitions / FSH files in their IG
> source.

### Disposition A: Accept As Requested

#### Proposal

Refactor the EU Imaging Study Report Composition profile so that the
distinction between "findings" and "impressions" is no longer carried
solely by the section the content is filed under. Specifically:

1. In the report Composition profile, allow a single
   "Findings & Impressions" section (or keep separate sections but
   relax their `entry` slicing so that classification is not implied
   by section membership).
2. Introduce / adopt a dedicated profile for impressions based on the
   `ClinicalImpression` resource (e.g.,
   `EUImagingStudyReportClinicalImpression`), and require that
   `Composition.section.entry` for impressions reference instances of
   that profile rather than generic Observations.
3. For findings, require that the typing be carried on the underlying
   resource (e.g., `Observation.code` from a value set of imaging
   finding concepts, or a profile such as
   `EUImagingStudyReportImagingFinding`) so that a consumer can
   classify content from the resource alone, without re-deriving it
   from the section path.
4. Update the IG narrative, examples, and any rendering rules to
   reflect that the section structure is presentation-only.

#### Justification

This directly addresses the reporter's concern that classification is
currently coupled to Composition layout, which is a presentation
artifact. FHIR's general guidance is that the meaning of content
should be self-describing on the resource (via `code`, profile, or
category), and `ClinicalImpression` is the resource explicitly
designed to capture an interpreter's synthesis of evidence — a much
better semantic fit for a radiology "impression" than a generic
Observation in a particular section. It also lets non-document
consumers (e.g., a UI that renders only impressions, or a measure
calculation) work off resource type/profile/code without parsing
section paths.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing Composition section structure (separate Findings
and Impressions sections) for backward compatibility and rendering
ergonomics, but layer **resource-level self-classification** on top
so that section membership is no longer the source of truth:

1. Define an extension or require use of `Observation.category` (or
   `meta.profile`) so that any Observation acting as an imaging
   finding or impression carries an explicit classifier, drawn from a
   small EU-bound value set (`finding | impression`).
2. Make the Composition section `entry` slicing accept Observations
   that match the corresponding category/profile, so the section
   becomes a *view* over self-classified resources rather than the
   classifier.
3. Permit (but do not yet require) `ClinicalImpression` instances in
   the impressions section, so implementers can migrate to
   `ClinicalImpression` over time without breaking current consumers.
4. Add IG guidance that consumers MUST classify by
   category/profile/resource type, not by section path.

#### Justification

This preserves the current document layout — useful for human
rendering and for implementers already building against the published
profile — while still removing the structural-position-as-semantics
problem the reporter raised. It is a smaller, less disruptive change
than Disposition A: existing instances remain valid, and adopters can
migrate to `ClinicalImpression` opportunistically. The trade-off is
that the IG carries two ways of expressing impressions (Observation
with classifier, and `ClinicalImpression`) until a future major
version cleans it up.

---

### Disposition C: Decline

#### Proposal

Decline the request for this version of EU Imaging Study Report (R5)
on the grounds that (a) using Composition sections to group findings
vs. impressions is consistent with how other imaging report IGs (and
HL7 CDA-derived imaging reports) are structured, (b) the reporter has
not provided a concrete implementation problem caused by the current
modelling, and (c) `ClinicalImpression` use in production radiology
reporting is currently rare, so mandating it would risk
interoperability without clear benefit. Mark the ticket as "Persuasive
With Mod" only if a future version (R6 or a major revision of the EU
Imaging IG) revisits the data model holistically.

#### Justification

The current model is workable and aligned with prevailing imaging
reporting practice. A change of this size mid-cycle would invalidate
existing example instances and any downstream profiles, with no
documented consumer that is blocked by the current design. The IG can
revisit the structure when there is concrete implementer feedback or
when a broader R6-era refresh is on the table.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance for HL7 Europe at this stage. It
takes the reporter's concern seriously — section position should not
be the semantic carrier — and fixes it where it actually matters
(resource-level self-classification), while avoiding the disruption
and migration cost of Disposition A in an already-published EU IG.
It also keeps the door open to `ClinicalImpression` adoption without
forcing it before there is concrete implementer demand. Disposition C
should only be chosen if the EU Imaging Study Report WG can confirm,
on the call, that no current implementer is parsing impressions vs.
findings programmatically; that is unlikely once the IG is in real
use, which is exactly when this ambiguity bites.
