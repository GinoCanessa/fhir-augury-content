## Summary

The reporter (Josh Priebe) flags that the HL7 Europe Imaging Study Report
IG's top-level "Manifest Bundle" profile (`ImImagingStudyManifest`) is
under-specified: the underlying `Bundle.type` is left open to all FHIR
bundle types — `document | message | transaction | transaction-response |
batch | batch-response | history | searchset | collection |
subscription-notification`, etc. Requiring manifest producers and
consumers to support every bundle type imposes a large and unnecessary
implementation burden. The ticket asks the IG authors to (a) constrain
`Bundle.type` to the specific kind(s) actually intended and (b) clarify
which bundle type is the primary mechanism for exchanging the manifest.
The reporter discusses two perspectives: a single-resource/REST flavour
where consumers `search` `ImagingStudy` and receive a `searchset` Bundle
(their preferred direction, supported by `_include` or follow-up calls),
versus a "document"-style exchange where it would need to be clear which
resources are inlined vs. referenced and how it fits into MHD-style
document exchange. The reporter suggests that a true FHIR Document with
human-readable `Composition` is a poor fit for a manifest, and a
`collection` (or possibly an operation result) bundle may be more
appropriate. A follow-up comment on 2025-07-15 notes the same problem on
the sibling profile `ImReport` ("Document Bundle for Imaging Report"),
which likewise appears to allow every bundle type when the intent is a
`document` bundle.

No cross-referenced Jira tickets were returned by the augury index for
this key. However, the ticket sits inside a wider EU Imaging ballot
cluster with several adjacent manifest tickets (e.g., FHIR-51387 on
separating Manifest from Report, FHIR-51011 on a better definition of
"manifest", FHIR-51013 "don't use imaging manifest", FHIR-51299 on
recommendations for communicating key images, FHIR-51163 on Bundle
invariants for DR/Composition consistency) that should be considered
together when dispositioning this one.

## Details

**Description:**

[IM ImagingStudy Manifest - HL7 Europe Imaging Study Report v0.1.0-ballot](https://build.fhir.org/ig/hl7-eu/imaging/branches/0.1.0-ballot/StructureDefinition-ImImagingStudyManifest.html)

Related to the top-level Manifest Bundle - Which type of bundle is being
used here seems under-specified. At first look it appears to not just
include a Document-Type bundle with Composition (as Imaging Report and
others have used), but also any other bundle type (document | message |
transaction | transaction-response | batch | batch-response | history |
searchset | collection | subscription-notification, etc).

Requiring manifest providers/consumers to support all of these bundle
types is a lot of scope and greatly increases effort. It should be
constrained to simplify deployment, and clarified which bundle type
should be primarily used to communicate the manifest Bundle (see
Document approach below).

This ties in with the use cases and transactions need discussion noted
elsewhere in tickets.

Workflow goal here is to offer an interface to search over manifests:

From the **single-resource perspective** (which is my preference), I
would want to do a FHIR native search on `ImagingStudy` and get back a
SearchSet Bundle as the core FHIR spec defines. I can determine which
resources as a client I want with `_include`, or follow links in
subsequent calls.

From the "**document**" perspective, how the Manifest should be
communicated in a Bundle is a good question. It should be clear (A)
which resources should be included in the bundle vs referenced (there's
some business logic here) and (B) fit into something like MHD or similar
for document exchange.

However, there's less of a need in the manifest for doing a FHIR
document with the human-readable composition. Maybe `collection` or
operation type bundle, not sure.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the EU Imaging Study Report IG (`hl7-eu/imaging`,
`0.1.0-ballot`), tighten the two affected Bundle profiles:

1. **`ImImagingStudyManifest`** (Manifest Bundle):
   - Constrain `Bundle.type` to a single value (`fixed[type] = "collection"`),
     reflecting the reporter's preference that the manifest is not a
     human-readable FHIR Document and the entries are simply a
     server-side roll-up of `ImagingStudy` plus references.
   - Slice `Bundle.entry` with closed slicing to require at least one
     `ImagingStudy` entry and explicitly enumerate the permitted
     resource types (e.g., `Endpoint`, `ImagingStudy`,
     `DocumentReference` for key images) so consumers know exactly what
     to expect.
   - Add a narrative section to the profile page describing the
     primary exchange pattern: server exposes a
     `GET [base]/ImagingStudy?...&_include=...` REST search returning a
     `searchset` Bundle, with the constrained `collection` Manifest
     used only when the manifest is *persisted* and exchanged as a
     standalone artefact (e.g., across an MHD boundary).
2. **`ImReport`** (Document Bundle for Imaging Report):
   - Apply `fixed[type] = "document"` and add the standard FHIR
     document-bundle invariants (`bdl-1`/`bdl-9`-style), since the
     reporter's follow-up comment confirms that the intent there *is*
     a FHIR Document.

#### Justification

The reporter and the comment author both note that the current open
`Bundle.type` is unintended: every IG that has built an exchange
artefact on top of `Bundle` (Imaging Report, IPS, MHD, US Core
documents, etc.) constrains `Bundle.type`, and FHIR-51163's resolution
in Block-Vote-2 (Persuasive — adding invariants on this IG's report
bundle for DR/Composition consistency) sets the precedent that the WG
is willing to tighten bundle-level rules. Constraining `ImReport` to
`document` is non-controversial — it matches the profile name. The
`collection` choice for `ImImagingStudyManifest` matches the
reporter's stated preference and keeps the door open for
search/REST-driven retrieval (which is unconstrained by the manifest
profile because a `searchset` Bundle is generated server-side and
isn't a profile target).

---

### Disposition B: Alternative Approach

#### Proposal

Split the response into two complementary changes rather than picking
one bundle type for the manifest:

1. **Re-frame `ImImagingStudyManifest` as a query result, not a
   profile.** Remove the standalone Manifest Bundle profile and replace
   it with a documented REST/search interaction: an
   `ImagingStudy`-rooted search (`GET
   [base]/ImagingStudy?subject=...&_include=ImagingStudy:endpoint&_include=ImagingStudy:basedOn`)
   that returns the standard `searchset` Bundle. Document the required
   `_include`/`_revinclude` parameters and the SearchParameters that
   must be supported.
2. **Keep a narrowly-scoped persisted Manifest profile only for
   document-exchange contexts.** Where a "manifest as artefact" really
   is required (e.g., MHD payload, asynchronous handoff), define it
   as a `Bundle.type = collection` profile, with closed-slice entries
   as in Disposition A, and mark it as optional / use-case-conditional
   in the IG's CapabilityStatement / actor definitions.
3. Apply the `Bundle.type = document` fix to `ImReport` regardless,
   since the comment thread already agrees on that.

#### Justification

The reporter explicitly states that the *single-resource* (search +
`_include`) view is their preference and that a FHIR Document is a
poor fit for a manifest. Disposition A still leaves an open question
of *when* a client should expect the persisted `collection` Manifest
vs. a server-side `searchset`. Disposition B answers that by making
the search interaction the primary mechanism and reserving the
profile for the narrower document-exchange use case. It also
dovetails with the related ballot tickets: FHIR-51387 (separation of
Manifest from Report), FHIR-51011 (better definition of "manifest"),
and FHIR-51299 (per-mechanism guidance for key images). The
trade-off is more rewriting in the IG and a heavier impact on
implementers (e.g., MADO in Australia per the
`australia > Adapting MADO` Zulip thread) who have already adopted
the v0.1.0-ballot Manifest profile shape. A short migration note in
the IG release notes mitigates that.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** on the grounds that constraining
`Bundle.type` is implicitly handled by the IG narrative and by the
related Block-Vote-1/2 tickets already in flight (FHIR-51387,
FHIR-51011, FHIR-51163, FHIR-51299), and re-issuing this requirement
would conflict with those resolutions.

#### Justification

This is defensible only as a procedural disposition: FHIR-51011 (better
definition of manifest) and FHIR-51163 (Bundle invariants for
DR/Composition consistency) are both already Persuasive in the EU
Imaging block votes, and an editor could argue their resolutions
collectively cover the under-specification noted here. Substantively,
however, the request is well-founded — leaving `Bundle.type`
unconstrained is a real and confirmed issue (the same author re-raised
it on `ImReport` in the comment thread) — so a pure decline would leave
implementers like the Australian MADO adopters without an answer and
should be paired with a clear pointer to the ticket(s) that the WG
believes resolve the issue.

---

### Recommendation

**Recommended disposition:** **A (Accept As Requested), with the
`ImReport` half folded in.**

Disposition A directly addresses both halves of the reporter's request
(the original `ImImagingStudyManifest` issue and the follow-up comment
on `ImReport`) and stays inside the existing v0.1.0-ballot profile
shape, which minimises churn for early adopters such as the Australian
MADO project. It also aligns with the precedent set by FHIR-51163 in
Block-Vote-2: this IG is willing to tighten Bundle profiles via
fixed types and invariants. Disposition B is the more architecturally
ambitious option and is worth raising as a follow-up ticket once
FHIR-51387's "separate Manifest from Report" framing is locked in,
but rolling that scope into FHIR-51388 risks expanding the change
beyond what the reporter asked for and beyond what a single ballot
disposition can carry safely. Disposition C should only be considered
if the WG concludes that FHIR-51011 + FHIR-51163 already cover the
ground — and even then the resolution comment should explicitly
direct implementers to the constraint(s) that fix the open
`Bundle.type`, so the issue does not surface again in the next
ballot.
