## Summary

The reporter (Ulrike Merrick) is asking whether the value set bound to
`ServiceRequest.intent` in the EU Laboratory Report IG's
`ServiceRequest-eu-lab-definitions` profile should be further constrained.
The base FHIR `request-intent` value set includes concepts (e.g.,
`proposal`, `plan`, `directive`) that the reporter believes are not
appropriate for a laboratory report context. The ticket was filed as a
Question against the EU Laboratory Report (FHIR) IG and was resolved as
"Considered - Question answered" with the resolution direction "Remove
the proposed concepts" — i.e., the workgroup agreed to tighten the
allowed `intent` codes by removing those that do not apply to the
laboratory ordering workflow. There are no other Jira tickets,
GitHub items, or Zulip threads cross-referenced to this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the EU Laboratory Report IG profile
`ServiceRequest-eu-lab-definitions`
(`https://hl7.eu/fhir/laboratory/StructureDefinition/ServiceRequest-eu-lab-definitions`),
constrain `ServiceRequest.intent` away from the base FHIR
`request-intent` value set to a smaller IG-defined value set that
excludes `proposal`, `plan`, and `directive`. Concretely, define a new
required-bound value set (e.g.,
`ValueSet/serviceRequest-eu-lab-intent`) that includes only the codes
appropriate to a lab order context — typically `order` and its
specialisations (`original-order`, `reflex-order`, `filler-order`,
`instance-order`) — and update the profile's element binding for
`ServiceRequest.intent` to that value set with `strength = required`.
This matches the resolution note "Remove the proposed concepts."

#### Justification

`ServiceRequest.intent` carries workflow semantics that determine how a
request should be acted on. In a laboratory ordering / reporting IG,
the artefact being represented is an actual order placed against a
laboratory, so codes that denote pre-order workflow stages (`proposal`,
`plan`) or advance directives (`directive`) are not meaningful for
conformant instances. Tightening the binding to only the order-family
codes prevents producers from sending instances that other EU lab
implementers cannot meaningfully process, and aligns the profile with
the IG's stated scope. A required binding to a curated subset is the
standard FHIR mechanism for this kind of profile-level narrowing and
matches the resolver's stated direction on the ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the base FHIR `request-intent` binding on
`ServiceRequest.intent`, but add a profile-level invariant (or change
the binding strength to `extensible` against an IG-preferred value set)
that *recommends* the order-family codes while still permitting
`proposal`, `plan`, and `directive` for edge cases (e.g., regional
workflows where a lab service request may be pre-staged as a plan
before it becomes an order). Document the expected use of `order` /
`original-order` in the profile's narrative and add a `Must Support` +
guidance note rather than removing the codes outright.

#### Justification

A required binding to a narrow set is a breaking constraint for any
implementer that already produces instances with `plan` or `proposal`
(for example, decision-support or care-planning systems that hand off
to a lab). Using an extensible binding plus narrative guidance
preserves interoperability with neighbouring workflows while still
steering EU lab implementers towards the canonical `order` codes. This
is a lower-risk evolution path if there is any uncertainty about
whether the excluded codes are used in any current EU member-state
deployment.

---

### Disposition C: Decline

#### Proposal

Leave `ServiceRequest.intent` bound to the base FHIR `request-intent`
value set unchanged, and close the question by pointing the reporter
at the base resource's intent semantics. Add (at most) a non-normative
note in the profile narrative clarifying that EU lab implementations
typically use `order`-family codes.

#### Justification

The base FHIR `request-intent` value set is already the curated set of
intent codes that the FHIR community agreed on for `ServiceRequest`,
and any code in it is technically valid. Profiling away widely-used
base codes risks fragmenting `ServiceRequest` semantics across
jurisdictional IGs and creates a maintenance burden if base FHIR adds
or revises intent codes. If implementations are sending inappropriate
intents in practice, that is arguably an implementation-conformance
problem, not a profile-binding problem.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as "Considered - Question
answered" with the explicit direction "Remove the proposed concepts,"
so the workgroup has effectively endorsed Disposition A. Substantively
it is also the right call: `ServiceRequest.intent` directly drives
downstream workflow handling, and an EU Laboratory Report instance
that arrives with `intent = proposal`, `plan`, or `directive` is not
actionable as a lab order in the IG's intended use cases. A required
binding to a narrowed IG-specific value set is the standard FHIR
mechanism for that kind of profile-level scoping and is preferable to
relying on narrative guidance (Disposition B) or leaving the base
binding untouched (Disposition C). Disposition B's interoperability
concern is largely theoretical for an IG focused on laboratory
*reports* rather than upstream care-planning hand-offs; if a concrete
implementer surfaces a need for `plan` later, the value set can be
expanded in a subsequent version with a clear deprecation/expansion
note.
