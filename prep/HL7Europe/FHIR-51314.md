## Summary

The reporter is reviewing the `ImAdverseEvent` profile in the EU Imaging Study
Report (R5) IG and asks the work group to clarify or justify why the profile
inherits/permits `AdverseEvent.subject` references to `Practitioner`, `Group`,
and `RelatedPerson` (i.e., parties other than the imaging `Patient`). They
note that the base FHIR `AdverseEvent` resource cites broader use cases such
as biohazard exposures (e.g., legionella) but cannot identify a concrete
imaging-context workflow in a system of clinical record where an adverse event
would be reported against a non-patient subject. They tentatively float
"practitioner radiation exposure event" as the only plausible imaging
scenario. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items in Augury for this ticket, so the request stands on its own
within the EU Imaging IG; the work group is being asked either to (a) add
documentation/examples justifying the non-Patient subject options, or (b)
constrain `subject` so the profile only allows `Patient`.

## Details

**Description:**

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket literally as a documentation request: in the EU Imaging
Study Report (R5) IG, update the `ImAdverseEvent` profile page to add a
narrative section ("Subject scope") that explains why
`AdverseEvent.subject` permits `Practitioner`, `Group`, and `RelatedPerson`
in addition to `Patient`. Provide one concrete imaging-relevant example per
allowed type — e.g., a `Practitioner` subject for an interventional
radiologist's cumulative radiation exposure incident, a `Group` subject for
a contrast-media batch event affecting a cohort scanned on the same day, and
a `RelatedPerson` subject for a parent/guardian present in the room during
a paediatric study who suffers an exposure or reaction. Add at least one
worked Example instance per non-Patient subject type so implementers see the
pattern in machine-readable form. Leave the underlying cardinality and
target profile list on `subject` unchanged.

#### Justification

Inheriting the full `subject` choice set from base `AdverseEvent` keeps the
profile aligned with R5 and avoids a deliberately-narrower constraint that
might block legitimate but uncommon cross-border imaging incidents (the EU
profile is meant to be broadly reusable across member states). The reporter
explicitly invites this kind of clarifying example ("Maybe Practitioner
radiation exposure event?"), and addressing the concern via documentation +
examples respects existing implementations without changing conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Tighten the profile rather than only documenting it: in the EU Imaging Study
Report (R5) IG, constrain `ImAdverseEvent.subject` to `Reference(Patient)`
only (drop `Practitioner`, `Group`, and `RelatedPerson` from the allowed
target types) and add an invariant/comment stating that staff exposure
events and cohort-level safety signals are out of scope for this IG and
should be carried in a separate `AdverseEvent` instance conforming to the
base resource (or to a future occupational/safety profile). If a future
imaging-specific occupational use case is identified, layer it as a
sibling profile (e.g., `ImStaffAdverseEvent`) rather than re-widening
`ImAdverseEvent`.

#### Justification

The reporter's underlying concern is that the inherited target list has no
demonstrated imaging-report use case, which makes the profile harder to
implement and validate (every server must accept and resolve four reference
types when in practice only `Patient` is ever populated). Profiles exist
precisely to narrow base resources to the subset a domain actually uses;
over-permissive `subject` choices can also leak PHI in unexpected ways
(e.g., a `RelatedPerson` reference in an imaging report). The trade-off vs.
Disposition A is loss of forward flexibility, but that flexibility is
hypothetical today and any future need can be addressed with an additive
profile rather than by leaving the door open in the core IG profile.

---

### Disposition C: Decline

#### Proposal

Decline as "not persuasive": close the ticket with the rationale that
`ImAdverseEvent` deliberately inherits the base R5 `AdverseEvent.subject`
choice set without further constraint, that the base specification already
documents the rationale for non-Patient subjects (occupational exposure,
cohort-level biohazard events), and that no change to the EU Imaging Study
Report (R5) profile or its narrative is required. The ticket can be marked
"Persuasive with Mod" only to the extent of adding a one-sentence pointer
in the profile narrative back to the base `AdverseEvent` resource's subject
discussion.

#### Justification

This is defensible if the work group's position is that EU-level imaging
profiles should not re-explain base-resource semantics and that profile
inheritance of `subject` is the correct default. The base FHIR
`AdverseEvent` resource is the authoritative explainer for who can be a
subject; duplicating that text in the IG risks drift across versions. The
risk is that the ticket will simply be re-raised by the next implementer
who reads the profile and asks the same question — declining without at
least the pointer-sentence trades short-term editorial savings for repeated
review burden.

---

### Recommendation

**Recommended disposition:** A

Disposition A best fits the actual ask. The reporter is not challenging the
profile's conformance — they are asking for an explanation, and the question
is reasonable enough that future implementers will hit it too. Adding a
short "Subject scope" narrative plus concrete, imaging-flavoured examples
(staff radiation exposure, contrast-batch cohort, paediatric guardian) costs
little, preserves alignment with base R5 `AdverseEvent`, and avoids the
forward-compatibility risk of Disposition B's hard constraint. Disposition B
is a reasonable fallback if HL7 Europe wants to take an explicit "Patient
only" stance, but should not be chosen without canvassing imaging-safety and
occupational-health stakeholders, since stripping target types from a
profile is harder to reverse than to add. Disposition C should be avoided:
the ticket's existence is itself evidence that the inherited semantics are
not self-evident in the imaging context.
