## Summary

The reporter (JoshPriebe) asks the EU Imaging Study Report (R5) IG to clarify
the relationship between the two image-access endpoint mechanisms it currently
exposes as optional alternatives: WADO-RS and IID (Invoke Image Display).
Because both are allowed and neither is required, image consumers may have to
implement and operate both transports to achieve full coverage across source
systems, doubling implementation cost. The ticket asks the work group to (a)
state explicit guidance for when each endpoint type should be used and
(b) decide whether one of them should be the minimum-mandatory endpoint, with
the reporter advocating that WADO-RS be preferred where available because it
better supports radiologist over-read, comparison studies, key images, and
locally deployed viewers, and because IID raises specific cross-border problems
around viewer localization, NCP-routed client⇄server traffic for viewer
sessions, and consumer-side usability of dealing with up to 26 different
country-specific viewers. No cross-referenced Jira tickets, Zulip threads, or
GitHub items were found in FHIR Augury for this ticket.

## Details

**Description:**

This guide specifies both WADO-RS and IID endpoints (although both are
optional). Is there guidance for when each should be used, and is one expected
as the minimum to support? Allowing both has the potential to double consumer
costs to get complete coverage of source implementations.

WADO-RS and IID have potentially substantial differences in workflow,
auth/access, provenance, etc. Creating clearer expectations here could save the
implementer community significant development, and helping to create alignment
could save implementation time and cost.

For most of the workflows described, and depending on the wider architectural
question of how references may be resolved across affiliate domains, we believe
WADO-RS endpoints should be preferred if available. This can support both the
Radiologist Overread workflow (by providing full diagnostic quality images),
comparison studies, and key images. It can also support viewers deployed at the
local level, which we expect to be the preference for many providers.

Additionally, deploying IID Endpoints in a cross-border setting introduces
additional challenges:
1. **Localization:** If I'm deploying a Spanish-language viewer cross-border,
   either all receivers need to understand Spanish or I need to support ALL
   possible destination locales.
2. **Access and authorization:** Sending all the client ⇄ server viewer traffic
   through the NCP connections is more challenging compared to an API such as
   WADO-RS. Going direct could be better but multiplies the access
   possibilities for servers.
3. **Usability.** From an image access consumer perspective, I don't want 26
   different viewers from 26 countries. I probably want the locally deployed
   viewer I'm familiar with and use for other workflows.

**Comments:**

- **JIRAUSER28578 (2025-06-27)** — Triage note: "Need to tag with HL7 Europe
  Workgroup." (housekeeping; explains the work-group assignment).
- **JIRAUSER30266 (2025-12-16)** — Substantive response from an IHE/MADO
  perspective:
  - The IHE MADO profile supports identification of both WADO-RS and IID
    endpoints in the imaging study manifest specification.
  - WADO-RS endpoint attributes live at the imaging-study **SERIES** level and
    provide either a UID Base URI endpoint lookup or the Base URI directly; the
    consumer composes STUDY/SERIES/INSTANCE retrieval URLs from there.
  - The manifest also defines a complete STUDY-level server-side viewer URL
    suitable for launching a server-based DICOM viewer keyed by Study Instance
    UID.
  - WADO-RS retrieval is supported both intra-community and cross-community via
    the MADO / XC_WADO profiles.
  - **IID-based viewing is only supported intra-community by MADO; no
    cross-community variant exists yet.** The server-side viewer URL in the
    manifest may use the IID Profile or be a proprietary viewer URL, but should
    contain the Study Instance UID to allow correct consumer- and server-side
    auditing.
  - "The choice of using either the WADO or IID based approach will be defined
    by the use case being supported."

## Proposed Dispositions

> Note: FHIR Augury surfaced no cross-referenced Jira/Zulip/GitHub items and no
> repo briefings for the EU Imaging Study Report (R5) IG, so the dispositions
> below name the conceptual targets in the IG (the `Endpoint` profile(s) and the
> implementation-guidance pages on image access) rather than concrete file
> paths. The work group should map them onto the IG's authoring sources before
> implementation.

### Disposition A: Accept As Requested

#### Proposal

Add a new normative-style "Image Access Endpoints" section to the IG (and a
matching narrative paragraph on the relevant ImagingStudy / Endpoint profile
pages) that:

1. **Sets WADO-RS as the minimum-mandatory image access mechanism** for
   conformant servers in the EU Imaging Study Report (R5) IG. Update the
   relevant `Endpoint` profile binding/slicing (and the cardinality of any
   `endpoint` reference on `ImagingStudy` / its series) so that at least one
   `Endpoint` slice with `connectionType = dicom-wado-rs` is required and the
   `IID`-typed slice is explicitly optional.
2. **Documents IID as a complementary, intra-community-only viewer launch
   mechanism**, drawing the wording from the JIRAUSER30266 comment (MADO
   intra-community only; no XC variant; server-side viewer URL must carry the
   Study Instance UID for auditing).
3. **Adds explicit cross-border deployment guidance** covering the three issues
   the reporter raised: viewer localization, NCP-vs-direct routing of viewer
   traffic, and consumer-side usability of dealing with many country-specific
   viewers — all leading to the recommendation that cross-border consumers
   SHOULD prefer WADO-RS and that publishers offering only IID SHOULD declare
   that limitation.

#### Justification

The reporter's analysis aligns with the substantive triage comment: WADO-RS is
the only mechanism with a defined cross-community/cross-border story
(MADO / XC_WADO), while IID is intra-community-only today. Making WADO-RS the
minimum closes the "double-implementation" cost the reporter calls out and
gives implementers a single, predictable retrieval contract. The change is
additive in the sense that current WADO-RS-supporting publishers stay
conformant, and it directly leverages the existing IHE MADO / XC_WADO design
the IG already aligns to. It also makes the IG's conformance position
defensible to European NCP operators, where the auth/access concerns the
reporter flagged are concrete operational risks.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both endpoint types optional in the base profile, but address the
reporter's underlying need with **decision guidance** rather than a new
mandatory constraint:

1. Add a non-normative "Choosing Between WADO-RS and IID" page that codifies
   the use-case-driven framing from JIRAUSER30266's comment: WADO-RS for
   diagnostic-quality retrieval, comparison studies, key-image workflows, and
   any cross-community / cross-border scenario; IID for viewer-launch
   intra-community scenarios where a server-side rendered view is acceptable
   and a Study Instance UID can be carried in the URL for audit.
2. Tighten the `Endpoint` profile's documentation (description / definition
   text on the `connectionType` and `payloadType` elements) so that the
   semantics, expected `Study Instance UID` carriage, and audit obligations are
   spelled out for each mechanism, without changing cardinalities.
3. Add a CapabilityStatement-level expectation that publishers SHALL declare
   which of the two mechanisms they support, so consumers can plan accordingly
   even if the IG itself does not mandate WADO-RS.

#### Justification

This addresses the reporter's "implementers don't know which to build" pain
without breaking publishers that today only expose IID for legitimate
intra-community viewer-launch use cases. It honours the JIRAUSER30266 position
that "the choice will be defined by the use case being supported," but converts
that statement into IG text rather than leaving it as tribal knowledge. It is
also a lower-risk change for an IG that is already published — it does not
re-cardinalise existing profiles and so avoids breaking existing claims of
conformance — while still measurably improving implementer experience. The
declared-mechanism CapabilityStatement requirement gives consumers the
predictability they want without forcing every publisher to implement WADO-RS.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for this IG and refer the question to IHE / the
underlying MADO profile maintainers, on the basis that the IG already aligns
to MADO and MADO already documents which transport applies to which
intra/cross-community scenario. Close as "Not Persuasive — already addressed by
referenced standard."

#### Justification

The substantive triage comment notes that MADO and XC_WADO already define when
WADO-RS vs IID is applicable, including the explicit fact that IID has no
cross-community variant. One could argue the IG should simply point at those
profiles and let IHE evolve the multi-community story for IID, rather than the
HL7 Europe IG carrying duplicate guidance that may drift out of sync with IHE.
Declining keeps the IG narrow and avoids HL7 Europe taking on an authoring
burden for image-access decision logic that does not originate in FHIR.

---

### Recommendation

**Recommended disposition:** B

Disposition B best matches the actual ask in the ticket — the reporter is
asking for *guidance and a stated minimum expectation*, not necessarily for a
profile-cardinality change — and it is consistent with the substantive triage
comment's framing that the right mechanism depends on the use case. Disposition
A is more decisive but risks invalidating existing IID-only publishers
(particularly intra-community deployments where IID is exactly the right tool
per MADO) and would need a deprecation path the IG isn't obviously set up for.
Disposition C under-serves the reporter: even if MADO is the source of truth,
EU IG implementers reasonably expect the EU IG itself to tell them which
endpoint to build first, especially given the cross-border concerns
(localization, NCP routing, viewer usability) that are specific to the EU
deployment context and not addressed by IHE. B captures the WADO-RS preference
the reporter argues for, the use-case nuance JIRAUSER30266 raised, and the
cross-border specifics — without breaking existing conformance claims. If the
work group later decides to mandate WADO-RS, B sets up a clean migration path
to A in a future version.
