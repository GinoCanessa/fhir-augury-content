## Summary

The reporter is asking the HL7 Europe authors of the *EU Imaging Study
Report (R5)* implementation guide to clarify a statement on the IG's
**Obligations** page for `ImagingStudy`. That page documents that the
`ImagingStudy.subject` could be a `Device`, but it does not explain
when that situation would arise. The reporter wants the IG to spell out
the concrete use case(s) in which a `Device` (rather than a `Patient`
or a `Group`) is the subject of an `ImagingStudy` in the European
imaging context, so implementers know whether and how to support that
target type.

There are no cross-referenced Jira tickets, no related Zulip threads
discussing this specific question, and no related GitHub PRs/issues
recorded for FHIR-51312 in the augury index.

## Details

**Description:**

## Proposed Dispositions

All three dispositions below address the same underlying request:
explain — in the EU Imaging Study Report (R5) IG — when
`ImagingStudy.subject` would point at a `Device`. Because no GitHub
repo is currently indexed for this IG, none of the dispositions cite a
specific source file; the work group should map the change onto the
IG's actual narrative source (typically the ImagingStudy profile's
intro page and/or the Obligations page in the IG's `input/pagecontent`
folder) when implementing.

### Disposition A: Accept As Requested

#### Proposal

In the EU Imaging Study Report (R5) IG, add a short narrative block to
the Obligations page (and ideally also to the ImagingStudy profile
intro page) that explicitly enumerates the use cases in which
`ImagingStudy.subject` is a `Device`. Concretely, list at least:

1. **Quality assurance / calibration imaging** — acquisitions of an
   imaging phantom or test object (modeled as `Device`) for routine QA,
   acceptance testing, or constancy testing of imaging equipment, where
   no patient is involved but the study still needs to be archived,
   reported, and tracked alongside clinical studies.
2. **Imaging of an explanted or stand-alone device** — e.g., specimen
   radiography of an explanted implant, hardware fragment, or other
   device-as-object, where the subject of the acquisition is the device
   itself rather than the patient it came from.
3. **Equipment self-imaging / service imaging** — service or
   engineering acquisitions performed against an imaging device or
   component, captured in the same archive as clinical studies.

For each, include one sentence on how the receiving system should
interpret the study (e.g., that patient-context elements may be absent
and that downstream reporting/billing pipelines should branch on
`subject` target type).

#### Justification

The reporter's question is a documentation gap, not a model gap: the
core FHIR R5 `ImagingStudy.subject` element already allows
`Patient | Device | Group`, and the EU IG inherits that. Documenting
the intended Device use cases inline in the IG is the lowest-risk,
highest-value response — it directly answers the reporter's question,
keeps implementers aligned on whether they must support
Device-subject studies in the EU profile, and does not require any
change to constraints, cardinalities, or value sets. It is also
consistent with how the broader FHIR community (see the *implementers*
stream threads above) has historically explained these target sets:
informally, by listing the use cases.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding free-text justification for a permitted-but-unused
target type, **constrain `ImagingStudy.subject` in the EU Imaging
Study Report profile to `Reference(Patient | Group)`** and remove
`Device` from the allowed targets. If the work group later identifies
a concrete EU use case for Device-as-subject (QA imaging,
specimen/explant imaging), reintroduce it deliberately with an
accompanying obligation that says how to populate it. Update the
Obligations page narrative accordingly so it no longer mentions Device
as a permitted subject type.

#### Justification

If, in practice, no EU implementer of this IG produces or consumes
Device-subject ImagingStudy resources, then leaving the option open in
the profile (and merely documenting it) imposes a small but real cost
on every consumer: receivers have to either implement handling for a
target type they will never see, or defensively reject it. Tightening
the profile to the cases the EU community actually uses removes that
ambiguity, simplifies conformance, and is fully backward-compatible
with the base R5 resource (a profile may narrow target types). The
trade-off vs. Disposition A is loss of forward-flexibility: any future
EU QA-imaging or explant-imaging use case would require a profile
revision. The work group should only choose this path if it is
confident those scenarios are out of scope for the IG.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — Documentation Sufficient",
on the basis that `ImagingStudy.subject`'s allowed target types are
already documented in the base FHIR R5 ImagingStudy resource and that
the EU IG's Obligations page is intentionally a normative obligations
table, not a tutorial on every permitted reference target. Direct the
reporter to the base FHIR R5 ImagingStudy page for the rationale of
the `Device` target type.

#### Justification

A defensible "decline" rests on the position that the EU IG inherits
its subject-type semantics from base R5 and has no obligation to
re-document them. This is the lowest-effort response and avoids
expanding the IG's narrative surface. The weakness — and the reason
this is the least attractive option — is that the base FHIR R5 spec
itself does not give a crisp use-case-driven explanation of the Device
target either, so pointing the reporter "upstream" does not actually
answer the question. Declining is only justified if the work group has
already decided that EU implementers do not need this clarified.

---

### Recommendation

**Recommended disposition:** A

This is fundamentally a documentation question from an implementer who
is trying to figure out whether to support `Device`-subject studies in
an EU deployment. Disposition A answers the question directly, costs
the IG only a few sentences of narrative, and does not commit the work
group to any model-level decision. Disposition B (profile narrowing)
is a credible second choice and worth explicit consideration during
review — but it should only be taken on the back of an affirmative
statement from the EU imaging community that QA/phantom and
specimen/explant imaging are out of scope for this IG. Without that
evidence, narrowing the profile risks excluding a small but real set
of legitimate radiology workflows. Disposition C is the weakest
option: pushing the question back to the base spec leaves the
reporter no better informed and invites the same ticket to be filed
again by the next implementer.
