## Summary

The reporter notes that the EU Laboratory Report (FHIR) IG appears to bind a
`Location.type`-style element to the HL7 v3 value set
[ServiceDeliveryLocationRoleType](https://terminology.hl7.org/ValueSet-v3-ServiceDeliveryLocationRoleType.html),
which is limited to healthcare-delivery settings. Because the IG also allows
`Direct subject of the observation` to reference a `Location`, the referenced
location may legitimately be a non-healthcare environment (e.g., a home, a
workplace, an outdoor site, a body of water sampled for laboratory analysis).
The current binding cannot describe these cases. The reporter asks the work
group to either (a) loosen the binding strength to (at most) extensible so
implementers can add codes for non-healthcare settings, or (b) replace/extend
the value set — for example by including SNOMED CT descendants of
`276339004 | Environment (environment)` — or (c) compose a mixed value set
that combines the v3 ServiceDeliveryLocationRoleType codes with an
environment-oriented SNOMED CT subset. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were found for this ticket.

## Proposed Dispositions

> Note: With no GitHub cross-references available, the dispositions below
> reference the IG by name (EU Laboratory Report (FHIR), HL7 Europe) rather
> than to specific authoring paths. The work group should map any accepted
> change to the relevant `input/` ValueSet / profile authoring file in the
> HL7-Europe laboratory report IG repository when scheduling the work.

### Disposition A: Accept As Requested

#### Proposal

In the EU Laboratory Report (FHIR) IG, locate the profile/element on
`Observation` (and/or `Specimen` / contained `Location` profiles) that
constrains the type of `Location` referenced as the *Direct subject of the
observation*. Change the binding on the relevant `Location.type` element so
that:

1. The binding strength is reduced from `required` (or `preferred`) to
   `extensible`; **and**
2. The value set is replaced with a *mixed* IG-defined value set that
   composes:
   - all codes currently in
     `http://terminology.hl7.org/ValueSet/v3-ServiceDeliveryLocationRoleType`,
     plus
   - SNOMED CT descendants of `276339004 | Environment (environment) |`
     (using a SNOMED CT `concept is-a` filter).

Publish the new value set under the IG's canonical (e.g.
`http://hl7.eu/fhir/laboratory/ValueSet/location-type-mixed`) and update all
profile bindings that previously pointed at the v3 value set to point at the
new IG value set.

#### Justification

The reporter's underlying concern is correct: an `Observation` whose subject
is a `Location` can describe environmental sampling (water, air, surfaces,
fields, residences) that the v3 `ServiceDeliveryLocationRoleType` value set
cannot represent. FHIR's binding-strength guidance explicitly recommends
`extensible` for situations where the listed concepts cover the common cases
but the model must allow additional codes for legitimate edge cases — exactly
this scenario. Composing the existing v3 codes with the SNOMED CT
`Environment` hierarchy preserves backward compatibility with implementers
already using the v3 codes while opening the door to environmental
laboratory reports that are squarely in scope for the EU laboratory program.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing `Location.type` binding as-is for healthcare-setting
locations, and instead address non-healthcare subjects by:

1. Splitting the use cases at the *subject* level rather than the
   *Location.type* level — e.g., recommend that environmental observations
   reference an environment-oriented profile (a small new
   `EnvironmentLocation` profile, or `Group` for sampled populations) rather
   than the existing healthcare `Location` profile; and
2. Documenting in IG narrative that the existing `Location.type` binding is
   intentionally healthcare-scoped, and pointing implementers at the
   environment profile for non-HC subjects.

If a single `Location` profile must remain, scope the change more narrowly:
relax the binding strength to `extensible` but **do not** replace the value
set — leaving implementers free to add their own codes (including SNOMED CT
`Environment` codes) without the IG taking on maintenance of a curated
mixed value set.

#### Justification

A mixed v3 + SNOMED CT value set introduces real maintenance cost (SNOMED CT
license obligations for downstream IGs, version pinning of the SNOMED filter,
overlap/duplication between the two code systems, and ambiguity when a
healthcare setting is also describable in SNOMED CT). Separating the use
cases — healthcare service-delivery locations vs. environmental sampling
sites — keeps each binding tight and intent-revealing, and is consistent
with how other HL7 IGs handle environment vs. service-delivery distinctions.
The narrower "extensible only, no value-set change" variant is the lowest-
risk way to unblock the reporter's use case without committing the work
group to curating new terminology.

---

### Disposition C: Decline (track as Considered for Future Use)

#### Proposal

Confirm the existing resolution (`Considered for Future Use`) and close
without an immediate normative change. Record in the IG that the current
release intentionally targets healthcare-setting laboratory reporting, and
defer environmental / non-HC subject support to a future ballot cycle once
(a) concrete implementer use cases for environmental laboratory reports in
the EU profile are documented and (b) a value-set strategy (pure SNOMED CT,
mixed, or extension-driven) has been agreed with the relevant terminology
stakeholders.

#### Justification

The ticket itself is exploratory ("might be better to … or create a mixed
value set?") and the current resolution is already `Considered for Future
Use` with a note that *further analysis is needed*. There are no
cross-referenced Jira tickets, no Zulip discussion, and no GitHub
implementation work indicating a present, blocked use case in the EU
laboratory report community. Making a binding-strength change or
introducing a SNOMED CT-derived mixed value set without that analysis risks
churn for current implementers (who would face a value-set canonical
change) and locks the IG into a SNOMED CT dependency footprint it has not
yet evaluated. Deferring is the conservative path that preserves the
reporter's concern in the backlog without prematurely committing the
terminology.

---

### Recommendation

**Recommended disposition:** A (with the binding strength change as the
must-have, and the mixed value set composed from existing v3 codes plus
`276339004 | Environment | descendants` as the recommended shape).

The reporter has identified a concrete inconsistency: the IG permits
`Location` as the subject of an `Observation` while binding the location's
type element to a value set that cannot describe the very cases that
permission is meant to enable. That is a defect, not a wishlist item, and
"Considered for Future Use" leaves implementers either misusing the v3
codes or dropping the binding locally — both worse than a deliberate IG-level
fix. Disposition A resolves the defect with the minimum surface area: an
`extensible` binding plus a published, named IG value set that explicitly
composes the existing v3 codes (preserving backward compatibility) with the
SNOMED CT environment hierarchy the reporter suggests. Disposition B's
"split the profiles" variant is cleaner architecturally but is a much larger
change and pushes complexity onto implementers; Disposition C punts on a
defect the reporter has already framed clearly. The work group should still
do the SNOMED CT licensing/versioning analysis the resolution calls out, but
that analysis belongs *inside* the change, not as a reason to defer it.
