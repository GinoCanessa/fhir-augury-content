## Summary

The ticket asks the EU Laboratory Report (FHIR) IG to stop relying on the
plain R4 shape of `DiagnosticReport.media.link` and instead adopt the
R5-derived cross-version extension
`http://hl7.org/fhir/StructureDefinition/alternate-reference` (published
in the `hl7.fhir.uv.xver-r5.r4` package), so that `media.link` can point
at a `DocumentReference` (the R5 type) while the IG remains R4-based.
The reporter's motivation is that R4 only allows `media.link` to
reference `Media`, which is too narrow for the kinds of attachments
labs need to associate with a report. The request is closely related to
[FHIR-53138](https://jira.hl7.org/browse/FHIR-53138) ("Add attachments
section"), which broadened the description and short text of
`DiagnosticReport.media` to cover images/PDFs and documents — that
ticket is already Applied. A community discussion on Zulip (see below)
confirmed with FHIR-I that filling `Reference.extension` is a
spec-valid way to carry the R5 reference target inside an R4 element,
which is the technical foundation for this proposal.

## Details

**Description:**
Instead of using `DiagnosticReport.media.link` as it is in R4, we
should use the cross-version extension for the element
`DiagnosticReport.media.link`
(`https://hl7.org/fhir/uv/xver-r5.r4/0.0.1-snapshot-2/StructureDefinition-ext-R5-DiagnosticReport.me.link.html`).
That way we can already use `DocumentReference` as the reference for
attachments of the lab report.

**Resolution Description (Persuasive):**

- Adopt the cross-version extension for `media.link` on
  `DiagnosticReport`: add the reference extension
  (`http://hl7.org/fhir/StructureDefinition/alternate-reference`,
  published in `hl7.fhir.uv.xver-r5.r4` 0.1.0).
  - Slicename `link`.
- Add a fixed display value: `"Media is a DocumentReference linked in the link extension"`.
- Set `media.link.reference`, `media.link.type`, and `media.link.identifier`
  to `0..0`; instead, the `valueReference` in the extension SHALL be used.

## Related Jira Tickets

### [FHIR-53138: Add attachments section](https://jira.hl7.org/browse/FHIR-53138)

Reporter HynekKruzik proposed adding a dedicated `attachments` section
to the EU Laboratory Report so that images, PDFs, and documents could
be carried as `Binary` or `DocumentReference` entries instead of
overloading `DiagnosticReport.presentedForm`. The disposition committee
landed on a "Persuasive with Modification" resolution that did **not**
add a new section, but instead broadened `DiagnosticReport.media`'s
short and definition text to cover additional data such as images and
diagrams (see PR
[hl7-eu/laboratory#93](https://github.com/hl7-eu/laboratory/pull/93)).
This is the direct predecessor to FHIR-55560: it normalised that
`media` is the carrier for attachments, and FHIR-55560 now closes the
remaining gap that R4 `media.link` can only reference `Media`.

Participants: HynekKruzik (reporter), giorgio (commenter / PR author).

Victoria Ayvasky opened the topic on 2026-02-09 explaining the EU

(FHIR-I) replied that the situation is delicate because an R4-only
consumer must still be able to make sense of the resource: filling out
`Reference.extension` produces a "valid" `Reference`, and the published

question to FHIR-I and reported back that filling

that generic guidance for cross-version use will be added. The thread
was marked resolved by Victoria on 2026-02-10.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Implement the resolution exactly as written. In the EU Laboratory
Report `DiagnosticReport` profile, on the `media.link` element:

1. Add a new extension slice on `media.link.extension` named `link`
   bound to
   `http://hl7.org/fhir/StructureDefinition/alternate-reference`
   (from `hl7.fhir.uv.xver-r5.r4` 0.1.0), allowing it to carry a
   `valueReference` of type `DocumentReference` (and `Media`, per the
   cross-version extension's target types).
2. Constrain `media.link.reference`, `media.link.type`, and
   `media.link.identifier` to `0..0`, so the only legitimate carrier of
   the link target is `extension[link].valueReference`.
3. Fix `media.link.display` to the literal string
   `"Media is a DocumentReference linked in the link extension"` so
   that R4-only consumers see a human-readable hint about where the
   real reference lives.
4. Apply the same change in the base EU laboratory IG where it
   propagates from (per patrick-werner's "also relevant for base ig"
   comment), and re-publish PR-style changes against `hl7-eu/laboratory`.

#### Justification

This is the option the workgroup has already converged on: the
resolution is recorded as Persuasive, the Zulip thread with FHIR-I
confirmed `Reference.extension` is the spec-valid mechanism, and the
cross-version extension is now published in
`hl7.fhir.uv.xver-r5.r4` 0.1.0, eliminating the original blocker
("snapshot-2"). Pinning `reference`/`type`/`identifier` to `0..0`
prevents the ambiguous case where both the R4 reference and the R5
extension are populated and disagree, and the fixed display gives
naïve R4 clients a discoverable signal. UK Core has used an equivalent
pattern in production, so there is implementation precedent for both
the slicing and the discoverability concern.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the cross-version extension as in Disposition A, but **soften**
the constraints on the underlying R4 element so that R4-only consumers
get a usable degenerate value:

1. Add the same `extension[link]` slice bound to
   `alternate-reference`.
2. Leave `media.link.reference` at its default cardinality (`1..1` per
   R4) and require it to point at a contained or referenced `Media`
   stub that wraps the real document; or
3. Alternatively, follow the **UK Core pattern** literally: keep
   `media.link.reference` at `1..1` but require an extension
   `data-absent-reason = unsupported` on `media.link` whenever the
   actual payload is carried via the cross-version extension; only set
   `display` to the fixed string.

#### Justification

The constraint `reference = 0..0` in Disposition A is technically
clean but produces a `Reference` with no R4-visible target, which an
R4-only consumer may treat as an error or silently drop. Disposition B
preserves "valid R4" semantics on the surface and uses the
cross-version extension as the additive R5-aware payload, which is
closer to the spirit of cross-version extensions (additive, not
substitutive). The trade-off is more verbosity and the need to
generate a `Media` stub or a `data-absent-reason` marker on every
instance. Victoria explicitly raised the UK Core variant in the Zulip
thread, so the workgroup has already considered it.

---

### Disposition C: Decline

#### Proposal

Decline the change for this ballot cycle and instead document, in the
IG narrative, that implementers who need to attach a
`DocumentReference` to a lab report should use the new
`DiagnosticReport.media` semantics introduced by FHIR-53138 (broader
short/definition text covering "additional data … like images,
diagrams"), pointing them at `presentedForm` for whole-report
documents and leaving `media.link` as plain R4 until the EU IG moves
to R5.

#### Justification

A pure-prose disposition would be defensible **only** if the workgroup
judged that the cross-version extension is too immature to ship in a
ballot — but the package
`hl7.fhir.uv.xver-r5.r4` is now published at 0.1.0 (the snapshot
problem from the original report no longer applies), FHIR-I has
explicitly blessed this pattern on Zulip, and a peer IG (UK Core) is
already shipping an equivalent. The remaining "decline" rationale is
mostly conservatism and slipping the change to a future R5-based
revision of the EU laboratory IG; the cost is that implementers
continue to have no spec-conformant way to point `media.link` at a
`DocumentReference` in the meantime, which is exactly the gap the
ticket was filed to close.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already recorded a Persuasive resolution with a
detailed mechanical specification, and FHIR-I has publicly endorsed
the cross-version-extension pattern in the linked Zulip thread. The
only substantive open question is whether to constrain
`media.link.reference` to `0..0` (Disposition A) or leave it filled
with an R4-valid stub plus a `data-absent-reason` (Disposition B).
Disposition A is preferable because it is unambiguous (only one place
ever holds the link target), it matches the resolution text already
agreed, and the fixed display string mitigates the discoverability
concern that motivates Disposition B. If implementer feedback during
the block-1 vote shows that R4-only consumers genuinely break on
`reference = 0..0`, the workgroup can fall back to the UK Core variant
(Disposition B) without invalidating any of the slicing or extension
work done for Disposition A.
