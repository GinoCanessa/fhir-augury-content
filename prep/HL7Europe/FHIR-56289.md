## Summary

The ticket asks HL7 Europe to refine the EU Imaging Study Report (R4)
specification so that imaging order metadata can be represented in line
with the MADO (Medical Acquisition Diagnostic Order) specification.
Specifically, an imaging study must be able to carry multiple orders,
each with its own accession number; each accession number must be able
to carry its own globally scoped issuer (an OID or DICOM
`CodingSchemeDesignator`); and each order must be able to carry one or
more order / placer identifiers, each with its own issuer. The ticket
also notes that order numbers are treated as global numbers (with an
associated system / coding system) and asks for clearer guidance on
how to model the multi-order / multi-accession case, using MADO's
coding wherever possible.

No other Jira tickets, Zulip discussions, or GitHub items were surfaced
by the cross-reference service for `FHIR-56289`, so the context above is
drawn entirely from the ticket itself.

## Details

**Description:**

We need to be able to support:

- multiple orders with each its own accession number
- each accession number needs its own (global) issuer — OID or `CodingSchemeDesignator`
- orders need to be able to have one or more order and placer identifier/number, each can have its own issuer

Further explanation is needed how to deal with multiple order and accession numbers.

Order numbers are global numbers (can have a system / codingSystem …).

Further description of the use cases is in the MADO spec.

Where possible, the coding of this in the specification needs to be the
same / aligned with MADO.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the EU Imaging Study Report (R4) IG so that the imaging order
model permits, at the data level, exactly what the ticket requests:

1. Allow a study to reference **one or more** orders. Model each order
   as a `ServiceRequest` (or equivalent profile in the IG) whose
   `identifier` element is `0..*` and can carry an `accessionNumber`
   slice plus one or more `placer` / `filler` order-number slices.
2. On every accession / order / placer identifier slice:
   - keep `Identifier.value` as the number itself, and
   - require `Identifier.system` (or `Identifier.assigner`) to carry
     the issuer. Document that the issuer may be an OID URI
     (`urn:oid:…`) or a DICOM `CodingSchemeDesignator` mapped through
     the IG's existing OID/DICOM conventions.
3. Slice `ImagingStudy.identifier` so that `accessionNumber` is
   `0..*` (not `0..1`), matching the "multiple orders, each with its
   own accession number" requirement.
4. Add narrative in the IG describing the mapping to MADO's order /
   accession-number model, plus an example Bundle showing a study with
   two orders, two accession numbers, and different issuers per
   identifier.

#### Justification

This directly satisfies every bullet in the ticket and reflects how
DICOM / MADO already model the problem: accession numbers and placer /
filler order numbers each have their own issuing authority, so they
must be represented as full `Identifier` instances with `system` (or
`assigner`) populated, not as bare strings. Raising the cardinality of
`accessionNumber` and of the order/placer identifiers to `0..*` is a
profile-level tightening/loosening inside the EU IG and does not
require changes to the base FHIR spec. It is also consistent with the
reporter's stated need to use the same coding as MADO wherever
possible.

---

### Disposition B: Alternative Approach (FAQ + example, no profile change)

#### Proposal

Adopt the approach recorded in the ticket's own Resolution Description:
do **not** change the profiles' cardinalities or slicing. Instead:

1. Add a dedicated FAQ entry to the EU Imaging Study Report IG that
   explains how to represent multiple orders, multiple accession
   numbers, and per-identifier issuers using the existing
   `Identifier` data type (`value` + `system` / `assigner`) and the
   existing `ServiceRequest.identifier` / `ImagingStudy.identifier`
   elements.
2. Add a worked example (a Bundle or a pair of `ServiceRequest` +
   `ImagingStudy` instances) that demonstrates two orders, each with
   its own accession number and its own issuer — one expressed as an
   OID, one as a DICOM `CodingSchemeDesignator` — and reference it
   from the FAQ entry.
3. Cross-link the FAQ to the relevant sections of the MADO spec so
   implementers can confirm the alignment.

#### Justification

The base FHIR data types and the existing IG profiles already permit
multiple identifiers with per-identifier `system` / `assigner`, so the
gap identified in the ticket is primarily a **guidance** gap, not a
modelling gap. A FAQ entry plus a concrete example is much lower risk
than re-slicing `accessionNumber` or changing cardinalities on a
published R4 IG: it avoids any chance of invalidating existing
instances, avoids new `inv-*` invariants, and still gives implementers
an unambiguous, MADO-aligned recipe. This is also exactly what the
reporter's co-authored resolution text proposes, which suggests the
submitter is comfortable with the guidance-level fix.

---

### Disposition C: Decline

#### Proposal

Decline the request as a change to the EU Imaging Study Report
specification. Close the ticket with a statement that the base FHIR
`Identifier` data type and the existing `ServiceRequest` /
`ImagingStudy` `identifier` elements are already sufficient to carry
multiple order numbers, multiple accession numbers, and per-identifier
issuers (OID via `Identifier.system` / `urn:oid:…`, or DICOM
`CodingSchemeDesignator` via the established OID mapping), and that
alignment with MADO is a documentation concern for the MADO authors,
not a normative change to the EU IG.

#### Justification

A strict "nothing is missing" reading is defensible: the mechanisms
requested (multiple identifiers, per-identifier issuer) already exist
in the base spec, so there is no modelling defect. However, the ticket
explicitly calls out that "further explanation is needed", i.e., the
submitter is reporting a documentation / alignment gap, not claiming
the data model is broken. Declining outright would leave that gap in
place and would not serve EU imaging implementers who need MADO-aligned
guidance.

---

### Recommendation

**Recommended disposition:** B

Disposition B is exactly the path the ticket's Resolution Description
already captures ("We will add descriptive text in the FAQ section on
this topic and add an example on this."), and it is the lowest-risk
way to close the gap the reporter is actually pointing at: a
**guidance** gap about how to express multi-order / multi-accession /
multi-issuer imaging orders in the EU IG, aligned with MADO. The base
`Identifier` type and existing `ServiceRequest` / `ImagingStudy`
elements already permit the required cardinalities and per-identifier
issuers, so the real deliverable is a clear FAQ entry plus a worked
example that implementers can copy. Disposition A (changing profile
cardinalities and slicing) is stronger medicine than this ticket
warrants for an already-published R4 IG and risks invalidating
existing instances; Disposition C ignores the documentation need the
reporter explicitly flagged. If, during authoring of the FAQ and
example, the team discovers that an existing slice actually prevents
the MADO-aligned pattern, that finding can be raised as a follow-up
ticket and escalated to Disposition A scope.
