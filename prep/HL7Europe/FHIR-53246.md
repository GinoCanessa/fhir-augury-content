## Summary

The reporter asks how the **EU Laboratory Report (FHIR) IG** should
represent the **result of a blood cross-match** test performed by a
hematology / blood-bank laboratory. A cross-match compares a unit of
**blood product** (the donor unit) against a **specimen from the
intended recipient**, and yields a compatibility result that determines
whether that specific unit may be transfused into that specific patient.
Two distinct modelling questions are raised:

1. **How is the cross-match result itself reported in a lab report?**
   (i.e., what does the `Observation` look like — code, value,
   interpretation, related observations.)
2. **How is the blood product identified in the context of that
   `Observation`?** (i.e., which element references which resource so
   that the donor unit being cross-matched is unambiguously identified.)

The reporter explicitly suggests that **guidance should be added to the
EU Laboratory Report IG**.

There are **no cross-referenced Jira tickets, Zulip threads, or GitHub
items** in the Augury index for this ticket, so the request is
self-contained as filed.

## Proposed Dispositions

> The ticket has already been disposed by the work group as
> **"Considered for Future Use"** with the `Semantic-Subgroup` and
> `block-vote-1` labels (i.e., parked for the Semantic Subgroup to
> develop the underlying model). The dispositions below are framed as
> options the work group could choose from when this ticket comes back
> off the parking lot.

### Disposition A: Accept As Requested

#### Proposal

Add a new **"Blood compatibility testing"** narrative section and a
worked example to the EU Laboratory Report IG that shows how to report
a cross-match result. Concretely:

- Define a recommended `Observation` shape for a cross-match result:
  - `Observation.code`: bind to a LOINC term for crossmatch
    interpretation (e.g., LOINC `91138-6` "Crossmatch interpretation
    in Blood product unit by Immune antiglobulin technique" or the
    appropriate parent code as decided by the Semantic Subgroup).
  - `Observation.value[x]`: `CodeableConcept` bound to a value set
    containing at minimum `Compatible` / `Incompatible` /
    `Indeterminate` (SNOMED CT codes such as `30714006` "Compatible",
    `62325002` "Incompatible").
  - `Observation.interpretation`: optional, using the
    `observation-interpretation` value set.
  - `Observation.specimen`: reference the recipient `Specimen`.
  - `Observation.focus` (or a dedicated extension on the EU Lab
    Report's `Observation` profile): reference the donor unit as a
    `BiologicallyDerivedProduct` (R5) or, in R4 IGs, a `Substance` /
    `Device` profile carrying the unit's DIN/ISBT-128 identifier.
- Define (or reference) a `Specimen` profile slice for "donor unit"
  vs. "recipient blood specimen", so the two inputs to the cross-match
  are unambiguous.
- Provide a complete example bundle (cross-match Observation +
  recipient Specimen + donor unit resource + DiagnosticReport) under
  `input/examples/` of the IG.

#### Justification

The reporter is asking a real, recurrent modelling question that the IG
does not currently answer; transfusion labs are part of the IG's stated
scope. Solving it once in the IG (with a worked example) is consistent
with how the IG handles other specialised lab domains and avoids each
implementer inventing their own pattern. The Semantic Subgroup label
already on the ticket signals that the work group accepts the question
is in scope — Disposition A simply commits to delivering the guidance
once the semantic model is settled.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than baking blood-bank guidance into the **EU Laboratory Report**
IG itself, **defer to (or create) a dedicated transfusion / blood-bank
companion IG** and only add a short **"out of scope — see X"** note to
the EU Lab Report IG that points to it. Concretely:

- Add a one-paragraph section to the EU Lab Report IG narrative noting
  that blood-bank workflows (cross-match, antibody screen, type &
  screen, transfusion reaction) require richer modelling than a plain
  lab `Observation` (donor unit identity, ABO/Rh, antibody panels,
  reservation/issue events) and are intentionally out of scope.
- Identify the target home for the work — e.g., an HL7 Europe
  Transfusion IG, the IHE LAB BBT profile, or the existing
  `BiologicallyDerivedProduct` / `BiologicallyDerivedProductDispense`
  resources in R5 — and link to it.
- File a follow-up Jira against the chosen target IG.

#### Justification

A correct cross-match model is more than one `Observation`: it bundles
donor-unit identity (DIN / ISBT-128), the recipient specimen, the
testing method, the antibody panel that motivated the test, and the
issue/reservation of the unit. Stretching the EU Lab Report IG to
cover this risks shallow guidance that doesn't compose with the
transfusion lifecycle, and locks decisions about donor-unit
representation (`BiologicallyDerivedProduct` vs `Substance` vs `Device`)
into a lab-report IG whose remit is general clinical lab. Pointing
implementers at a dedicated transfusion specification keeps the EU Lab
Report IG focused while still answering the reporter's question
("where do I look?").

---

### Disposition C: Decline

#### Proposal

Close as **"No change to the EU Laboratory Report IG"** on the basis
that the existing IG profiles (the generic `Observation` profile for a
laboratory result, plus `Specimen` and `DiagnosticReport`) are already
sufficient to carry a cross-match result: the lab picks an appropriate
LOINC code, puts the compatibility verdict in `Observation.value`,
references the recipient `Specimen`, and identifies the donor unit
either via an existing identifier slot or a project-local extension.
Direct the reporter to existing examples and the FHIR
`BiologicallyDerivedProduct` resource for donor-unit identity.

#### Justification

The IG is intentionally generic about which lab tests it covers; it
does not enumerate guidance per discipline (chemistry, microbiology,
hematology, blood bank). The cross-match question is answerable today
with the existing profiles plus correct terminology. Adding a
discipline-specific section sets a precedent that every sub-domain
(microbiology susceptibility, anatomic pathology synoptic reports, …)
deserves the same treatment, which would inflate the IG without adding
conformance value. The current resolution
("Considered for Future Use") effectively expresses this position —
Disposition C makes it explicit.

---

### Recommendation

**Recommended disposition:** **B**

The ticket exposes a real gap, but the gap is broader than a lab-report
question: cross-match is one observation in a larger transfusion
workflow that also has to model donor-unit identity, ABO/Rh typing,
antibody screening, reservation, issue, and transfusion reactions. The
Semantic Subgroup label that the work group already applied is
implicitly signalling the same concern — the underlying model has to
be settled before the IG can usefully document it. Disposition A risks
freezing a half-model into the EU Lab Report IG before that semantic
work is done, and Disposition C leaves implementers with no concrete
pointer at all. Disposition B keeps the EU Lab Report IG's scope clean,
explicitly acknowledges blood-bank workflows in the narrative, and
hands the substantive modelling to the right venue (a dedicated
transfusion / blood-bank IG, or an existing one such as the OO
Incubator / `BiologicallyDerivedProduct`-based work) — which is the
most defensible answer to bring back from the Semantic Subgroup when
this ticket is un-parked.
