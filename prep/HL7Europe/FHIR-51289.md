## Summary

Josh Priebe (Epic) requests that the EU Imaging Study Report (R5) IG provide
explicit, implementer-facing guidance on how to populate the
authorship-related elements of `Composition` — specifically `author`,
`attester`, `legalAuthenticator`, and `custodian` — in the context of
imaging report workflows. The submitter accepts the existing cardinalities
but anticipates implementer confusion in two recurring scenarios:

1. **Multi-author / dual-read / resident workflows**, where multiple
   radiologists (across subspecialties) co-author or hand off the report,
   or where legacy data does not cleanly distinguish author vs. attester
   roles.
2. **System-generated Compositions**, where the IG's `Composition` is
   assembled on demand by a server (e.g., from a `DiagnosticReport`
   stored in a non-FHIR or DiagnosticReport-only EHR), with no human user
   present to "send" or attest to the document.

The ticket asks the WG to (a) describe when each authorship element is
appropriate, and (b) caution implementers against assuming
`Composition.attester` / `legalAuthenticator` are always populated,
directing them instead to `DiagnosticReport.performer` /
`DiagnosticReport.resultsInterpreter` for the underlying provenance.

The FHIR cross-reference index links this ticket to five FHIR artifacts
that are directly named in the discussion: `Composition.author`,
`Composition.attester`, `Composition.custodian`,
`DiagnosticReport.performer`, and `DiagnosticReport.resultsInterpreter`.
No related Jira tickets, Zulip threads, or GitHub items were surfaced by
the cross-reference index. The ticket has already been disposed
**Persuasive** (resolved 2025-11-24); the recorded resolution is to
update the `Composition` description, clarify in the use-case / report
overview that `DiagnosticReport` is the primary entry point and the
`Composition` is a rendering of it (including flipping the overview
diagram so `DiagnosticReport` sits on the left and `Composition` on the
right), and rework the unstructured-report example to lean on a
`DiagnosticReport` rather than a `DocumentReference`.

## Details

**Description:**

The Composition currently includes several fields related to the originator
of the Result and/or Composition. These include:

- Author 1..*
- Attester 0..*
- legalAuthenticator 0..*
- Custodian 0..1

We would suggest providing at least informational guidance on when each of
these is used. The cardinalities as specified seem correct, but we expect
confusion from implementers.

Note that an Imaging Results Report may have multiple authors – a set of
imaging may have been read by authors from different imaging specialties,
who then collaborated on a report or each wrote separate sections, or a
resident may have been involved. Especially for legacy data (which we
believe should be shared if possible to improve care outcomes), the roles
of multiple providers on a result may not have been well specified.

We also note that many Compositions will be generated ad-hoc by the system
at the time of query, similar to delayed-assembly documents. We should be
careful in baking in assumptions that a user is present to send and attest
to the document. This is because an EHR may store the data in a format
besides FHIR. Even for groups that do have data stored in a FHIR format
at rest, they may only have DiagnosticReports today. As such, we suggest
guidance for implementers that they should not rely on the presence of
an Attester or legalAuthenticator, and direct to DiagnosticReport fields
instead.

**Resolution Description (already recorded on the ticket):**

We will update the description in the `Composition` resource to more clearly
state the difference of these fields. We will explain in the use case and
imaging report section that the `DiagnosticReport` is the main entry point
and the `Composition` is a rendering of the report. This includes flipping
the overview picture, placing the `DiagnosticReport` to the left and the
`Composition` to the right. We will also change the unstructured-report
example to rely on just a `DiagnosticReport` and not on `DocumentReference`,
and an example.

**Comments:**

- **JIRAUSER28670 — Rutt Lindstrom (2025-07-15):** Proposes concrete draft
  text to be added to the **ImComposition page**, structured as guidance
  per element:
  - **Composition.author** — radiologist(s) producing clinical content,
    represented as `Practitioner` / `PractitionerRole`. Walks through
    common scenarios: resident-and-attending (resident in `author`,
    attending in `attester` with `attester.mode = professional`),
    collaborative reads (all contributing radiologists in `author`),
    scribe workflows (radiologist is author, scribe is not), and legacy
    data (record all known contributors as best as possible).
  - **Composition.attester** — distinct from authorship; `attester.mode`
    drives the role: *resultValidator* for non-legal data verification
    (e.g., quality reviewer or automated checker), *legalAuthenticator*
    when a clinician takes legal responsibility. Adds an explicit
    italicised caveat: attesters should only be populated when the role
    is clearly represented in the workflow; for dynamically generated
    Compositions, implementers must not assume `attester` /
    `legalAuthenticator` and should fall back to
    `DiagnosticReport.performer` / `DiagnosticReport.resultsInterpreter`.
  - **Composition.custodian** — the organization responsible for the
    report's integrity and availability (typically the imaging system /
    EHR host), noting it may differ from where the report was authored.
- **JIRAUSER28670 — Rutt Lindstrom (2025-07-16):** Asks Rutt Lindstrom
  to confirm the proposed wording covers the intent of the corresponding
  EHDS Imaging Report header elements
  (`EHDSImagingReport.header.attestation.attester` and
  `EHDSImagingReport.header.legalAuthentication.legalAuthenticator`),
  to keep the EU Imaging IG aligned with the Xt-EHR / EHDS common header
  model.
- **JIRAUSER28578 — Ignacio Jauregui (2025-08-07):** Endorses the proposed
  guidance and asks the WG to go further by linking the Composition
  authorship elements directly to the corresponding `DiagnosticReport`
  elements. Specific points:
  - `Composition.author` should be derived from
    `DiagnosticReport.resultsInterpreter` (slice `:author`), and flags a
    cardinality mismatch (required on `Composition`, optional on
    `DiagnosticReport`) as worth a separate ticket.
  - The `DiagnosticReport` linkage for `Composition.attester` is already
    captured.
  - No clear `DiagnosticReport` analogue for `custodian`; questions
    whether `custodian` is meaningful in a RESTful (non-document) view at
    all, but agrees the proposed guidance is fine as-is.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the draft guidance proposed by Rutt Lindstrom in the 2025-07-15
comment essentially as-is, by adding a new "Guidance on how to populate
Composition authorship elements in imaging" section to the **ImComposition**
page of the EU Imaging Study Report (R5) IG. Concretely:

- Add three subsections — `Composition.author`, `Composition.attester`,
  `Composition.custodian` — each with a definition tailored to imaging
  reporting and the worked scenarios captured in the comment
  (resident-and-attending; collaborative / multi-subspecialty reads;
  scribe; legacy data).
- For `Composition.attester`, document the two `attester.mode` codes the
  IG cares about (`resultValidator` vs. `legalAuthenticator`) and include
  the italicised caveat that systems generating `Composition` resources
  dynamically should not be assumed to populate `attester` /
  `legalAuthenticator`, and that consumers should fall back to
  `DiagnosticReport.performer` / `DiagnosticReport.resultsInterpreter`
  for provenance.
- Keep `Composition.custodian` guidance as the short organizational note
  in the draft, without forcing a one-to-one map to `DiagnosticReport`.
- Apply the resolution already recorded on the ticket: refresh the
  `Composition` element descriptions in the ImComposition profile to
  emphasise the distinction between author, attester, legalAuthenticator
  and custodian; in the use-case / overview narrative, state explicitly
  that `DiagnosticReport` is the primary entry point and `Composition`
  is a rendering of it, and flip the overview diagram so
  `DiagnosticReport` is on the left, `Composition` on the right; rework
  the unstructured-report example so it is anchored on a
  `DiagnosticReport` rather than a `DocumentReference`.

No cardinality changes. No new must-support flags. No element additions
or removals.

#### Justification

This matches the ticket's actual ask (clarifying guidance, not a
structural change), the submitter has already validated the shape of the
proposed text by commenting on it, and the resolution recorded on the
ticket (Persuasive) commits HL7 Europe to exactly this scope. Imaging
reporting in Europe genuinely has the workflows the ticket calls out
(dual read, resident, system-assembled documents driven by EHDS
queries), so codifying the expected mapping reduces the risk of
inconsistent implementations between Member States and between the EU
Imaging IG and the Xt-EHR `EHDSImagingReport` header (which the
2025-07-16 comment explicitly asks to be aligned). It is a low-risk
change because it is informative-only and does not affect conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of Disposition A, but tighten the binding to
`DiagnosticReport` in line with Ignacio Jauregui's 2025-08-07 comment.
In addition to the narrative guidance on the ImComposition page:

- Add a normative table (or mapping section) on ImComposition explicitly
  stating the source of each authorship element when the `Composition`
  is rendered from an `ImDiagnosticReport`:
  - `Composition.author` ⇐ `DiagnosticReport.resultsInterpreter` (slice
    `:author` on the imaging profile).
  - `Composition.attester` (mode `professional` / `legal`) ⇐
    `DiagnosticReport.performer` (or `.resultsInterpreter` where the
    interpreting clinician also legally signs).
  - `Composition.custodian` ⇐ `DiagnosticReport.performer.organization`
    (or, if absent, the issuing system's managing organization), with a
    note that it is informative only.
- File a follow-up ticket against the ImComposition / ImDiagnosticReport
  profiles to address the cardinality mismatch flagged by Ignacio
  (`Composition.author` is 1..* but `DiagnosticReport.resultsInterpreter`
  is 0..* — a renderer cannot guarantee a populated `author` without an
  IG-level constraint on `ImDiagnosticReport`).
- Otherwise apply the same description / overview / example clean-ups as
  in Disposition A.

#### Justification

This goes beyond pure guidance and gives implementers a concrete
algorithm for populating `Composition` authorship elements from a
`DiagnosticReport`-only source — which is the exact scenario the ticket
worries about. It also takes Ignacio's feedback seriously rather than
deferring it: the cardinality mismatch is real and will bite the very
"system-generated Composition" workflows this ticket is about, so
addressing it in a sibling ticket is better than letting it be discovered
in CI later. The trade-off vs. Disposition A is more authoring effort and
slightly more constraint surface (the new mapping is more prescriptive),
but it is still informative on `Composition` and only adds a constraint
on `ImDiagnosticReport`, where the IG already shapes the profile.

---

### Disposition C: Decline

#### Proposal

Decline to add ImComposition-specific authorship guidance and instead
rely on (a) the base R5 `Composition` resource definitions for `author`,
`attester`, `legalAuthenticator`, and `custodian`, and (b) the existing
EHDS `EHDSImagingReport.header.attestation` / `legalAuthentication`
guidance in the Xt-EHR common module. Mark the ticket as "Not Persuasive
— Already Addressed" with a pointer to those resources.

#### Justification

The base `Composition` element definitions already describe each of the
four elements in general terms, and the EHDS Imaging Report header
already models `attestation` and `legalAuthentication` in a way the EU
Imaging IG aligns to. Duplicating that guidance specifically for imaging
risks drift between the EU IG, base FHIR R5, and Xt-EHR if any of those
upstream sources change.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved Persuasive, and Disposition A is
what that resolution commits to: it is informative-only, it directly
addresses the implementer confusion the submitter (and Ignacio's
follow-up) flag around dual-read, resident, and system-generated
workflows, and it does not constrain conformance for implementations
that legitimately cannot populate `attester` / `legalAuthenticator`.
Disposition C understates a real gap — base `Composition` text is
generic and the EHDS header is structured differently enough from
`Composition` that implementers really do ask "where does my radiologist
go?" — and Disposition B, while more rigorous, reaches into
`ImDiagnosticReport` cardinality and should be tracked as the separate
follow-up ticket Ignacio explicitly suggested rather than bundled here.
The WG should adopt Disposition A as the primary action and open a
sibling ticket capturing Ignacio's `Composition.author` /
`DiagnosticReport.resultsInterpreter` cardinality concern so it does not
get lost.
