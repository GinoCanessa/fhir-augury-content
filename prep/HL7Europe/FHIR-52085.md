## Summary

The ticket asks the HL7 Europe Hospital Discharge Report (HDR) work group to
revise the `administeredProduct` extension currently used on the
`immunization-eu-hdr` profile so that it is built "the same way as the
extension regarding `basedOn`". The reporter is pointing to the EU HDR
Immunization profile (`StructureDefinition-immunization-eu-hdr`) and asking
that the cross-version mechanism used to surface the R5 element
`Immunization.administeredProduct` in an R4-based IG be aligned with whatever
pattern HDR is already using for `Immunization.basedOn` (a different R5-only
element that HDR also pre-adopts). A comment on the ticket points
implementers at the cross-version-source candidate definition published at
`StructureDefinition-ext-R5-Immunization.administeredProduct`, signalling that
the desired shape is the R5 cross-version extension rather than a
locally-authored EU extension. Cross-reference search surfaced no other Jira
tickets, but did surface a recent (Nov 2025) "IG creation" Zulip thread in
which Giorgio Cangioli hits a snapshot-generation failure on exactly this
extension when pulling it via the `hl7.fhir.extensions.r5` package, plus
older `implementers` and `fhir/infrastructure-wg` discussions that frame
`Immunization.administeredProduct` as a `CodeableReference(Medication)` R5
addition that R4-based IGs (IPS, EU HDR) want to pre-adopt.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the EU Hospital Discharge Report IG, refactor the `administeredProduct`
slice on the `Immunization` profile (`immunization-eu-hdr`) so that it is
declared in the **same shape used by the existing `basedOn` slice** in the
same profile. Concretely, that means: (1) drop any locally-authored EU
extension URL or any reference to the `hl7.fhir.extensions.r5` package, and
(2) bring the canonical R5 cross-version extension for
`Immunization.administeredProduct` —
`http://hl7.org/fhir/5.0/StructureDefinition/extension-Immunization.administeredProduct`
— in via the cross-version package
`hl7.fhir.uv.xver-r5.r4` (currently `0.0.1-snapshot-2`), wired into the EU
HDR `sushi-config.yaml` / IG dependencies. The FSH should mirror the
`basedOn` block, e.g. an alias to the R5 cross-version StructureDefinition,
a `* extension contains $immunization-administeredProduct-r5 named
administeredProduct 0..1`, and any
slicing/discriminator/short/definition cosmetics that the `basedOn` slice
uses.

#### Justification

This is what the reporter literally asked for ("the same way as the
extension regarding basedOn"), and the comment from JIRAUSER17796 explicitly
nominates the cross-version-source candidate at
`StructureDefinition-ext-R5-Immunization.administeredProduct` as the
intended source. The recent IG-creation Zulip thread is direct evidence that
the cross-version package (`hl7.fhir.uv.xver-r5.r4`) is the working
mechanism for pulling R5-only elements like
`Immunization.administeredProduct` (CodeableReference(Medication)) into an
R4-based IG, and that the extensions-pack route fails snapshot generation.
Aligning `administeredProduct` with `basedOn` keeps the EU HDR profile
internally consistent (one mechanism for all pre-adopted R5 elements),
keeps authors looking in one place, and does not invent any
HL7-Europe-specific URL.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than alias the cross-version extension manually in HDR FSH at all,
adopt the **tooling-generated pattern-based cross-version extensions** for
all R5 pre-adoptions in EU HDR (Immunization.administeredProduct,
Immunization.basedOn, and any other R5-only elements currently surfaced as
extensions). Operationally: remove explicit
`hl7.fhir.extensions.r5` / `hl7.fhir.uv.xver-r5.r4` dependencies for these
elements from `sushi-config.yaml`, drop the explicit `Alias:` /
`extension contains` lines, and let the IG publisher emit the pattern-based
cross-version extensions automatically (the path Gino Canessa described in
the Zulip thread as "if you include no relevant xver package … the tooling
does that"). Where slicing/cardinality differs from the R5 default, keep a
thin EU HDR profile-level constraint, but don't author the URL itself.

#### Justification

This addresses the **underlying need** — "make administeredProduct work the
same way as basedOn, and don't get snapshot errors" — without coupling EU
HDR releases to a specific snapshot-N of `hl7.fhir.uv.xver-r5.r4`. It
sidesteps the recurring "wrong xver package version" trap visible in the
Zulip thread (and in Lloyd McKenzie's infrastructure-wg observation that the
canonical URLs for these extensions have themselves been in flux between R4
and R5 namespaces). It also keeps EU HDR forward-compatible: when the
cross-version tooling stabilises, HDR inherits the fix without re-editing
FSH. The trade-off vs. Disposition A is loss of explicit visibility — the
extension URL no longer appears verbatim in HDR sources — and a stronger
dependency on publisher behaviour, which is a mild risk given that the
publisher's pattern-based generation is comparatively new.

---

### Disposition C: Decline

#### Proposal

Decline the change as scoped. Rationale: the existing
`administeredProduct` declaration in `immunization-eu-hdr` is already
rendering on `hl7.eu/fhir/hdr/StructureDefinition-immunization-eu-hdr.html`
and the issue is purely cosmetic alignment with how `basedOn` is authored.
There is no published implementer report against EU HDR (only a separate
IG-creation reproducer in `hl7-eu/base`), no terminology or conformance
defect, and no cross-referenced FHIR core ticket. The HDR work group
should defer until either (a) a normative R4→R5 cross-version pattern is
published by FHIR-I, or (b) an HDR implementer files a concrete
interoperability problem caused by the current shape.

#### Justification

Defensible on scope-and-timing grounds: the ticket explicitly asks for
parity with `basedOn`, but does not assert that the current shape is
*broken* in EU HDR — only that it should be authored differently. With
`hl7.fhir.uv.xver-r5.r4` still at `0.0.1-snapshot-2` and Lloyd McKenzie's
infrastructure-wg observation about R4-vs-R5 canonical namespacing for
backport extensions still in play, refactoring HDR now risks a second
churn when the cross-version mechanism settles. Declining keeps the EU HDR
release stable and avoids importing a moving target.

---

### Recommendation

**Recommended disposition:** A

Disposition A is what the reporter asked for, what the comment on the
ticket nominated, and what the very recent (Nov 2025) IG-creation Zulip
thread independently validated as the working mechanism: pull
`Immunization.administeredProduct` in via the
`hl7.fhir.uv.xver-r5.r4` cross-version package, mirroring whatever pattern
EU HDR already uses for `Immunization.basedOn`. It is concrete, low-risk,
and produces a profile that is internally consistent — the same mechanism
for every R5 pre-adoption in HDR, the same URL the cross-version source
publishes, and no bespoke EU extension to maintain. Disposition B is
attractive longer-term but depends on publisher behaviour that is still
maturing and would be best raised as a separate technical-correction once
the pattern-based cross-version tooling is declared stable. Disposition C
is hard to defend given the reporter is a HDR implementer and the Zulip
evidence shows the current shape actively trips up downstream IGs.
