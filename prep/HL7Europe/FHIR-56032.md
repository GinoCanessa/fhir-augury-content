## Summary

Ralf Herzog reports that the EU Laboratory Report (FHIR) logical model's
"A.5 - Results data elements" section lists **EMDN** as a preferred code
system for *Device (analyser), laboratory test kit and used calibrator
information* (identifier, type, name, model, manufacturer). The reporter
argues this is the wrong choice because EMDN identifies only the *kind*
of device (e.g., `W0201010101` "CHEMISTRY ANALYSERS - LOW ROUTINE
(Random access without ISE mode test/h ≤ 400)") and not a concrete
device instance. They propose that **EUDAMED** (the EU device database)
should be used instead to identify the specific device. The ticket has
already been resolved as **Not Persuasive** with the rationale that the
underlying EHN logical models have been replaced by EHDS models that
sit outside this group's jurisdiction. No cross-referenced Jira, Zulip,
or GitHub items were located for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the EU Laboratory Report (FHIR) logical model section *A.5 -
Results data elements*, for the "Device (analyser), laboratory test
kit and used calibrator information" element, by removing **EMDN** as
a preferred code system for the device *identifier* and replacing it
with **EUDAMED** (the EU device registry, fulfilling the UDI-DI role).
Retain SNOMED CT where it is used to classify the *type* of device,
and clarify in the narrative that EMDN (if kept anywhere) is a
device-type nomenclature and must not be used as a device instance
identifier.

#### Justification

The reporter's core argument is structurally correct: EMDN is a
European device *nomenclature* (type codes) maintained as part of
EUDAMED, not an identifier system for concrete device instances. Where
the data element is the device *identifier* (as the element name
states), a registry-based instance identifier such as EUDAMED/UDI-DI
is the right binding; SNOMED CT and EMDN are appropriate only for
*type* classification. This aligns with FHIR's general pattern of
separating `Device.identifier` (instance) from `Device.type` (type
coding).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than swapping EMDN for EUDAMED wholesale, **split the binding
by the sub-attribute it governs**:

- For the device **identifier** sub-attribute, recommend **EUDAMED /
  UDI** (UDI-DI for the device model, UDI-PI for production
  instance-level identifiers where available).
- For the device **type** sub-attribute, retain **SNOMED CT** and
  **EMDN** as preferred type-classification systems.
- Update the A.5 narrative to make the type-vs-instance distinction
  explicit, addressing the ambiguity raised in rikim's comment.

#### Justification

This preserves existing implementer investment in EMDN/SNOMED CT
type coding while fixing the conceptual error of listing a type
nomenclature under an element labelled "identifier". It also directly
answers rikim's question about type vs. instance: the logical model
becomes explicit that different preferred systems apply to different
sub-attributes, which is a common pattern in FHIR logical models and
avoids a lossy either/or choice. Lower implementer burden than
Disposition A because EMDN guidance is retained where it is genuinely
appropriate.

---

### Disposition C: Decline (as the ticket was already resolved)

#### Proposal

Decline the change on the grounds already recorded in the resolution:
the EHN logical models that contain this A.5 section have been
superseded by **EHDS** models, which are not maintained by this
HL7 Europe group. The correct venue for this feedback is the body
governing the EHDS logical models; no change will be made to the EU
Laboratory Report (FHIR) artifact on this basis.

#### Justification

This matches the existing **Not Persuasive** resolution. Making a
binding change to a logical-model section that has already been
retired/replaced would introduce churn in a deprecated artifact and
would not propagate to the successor EHDS models where the issue
actually needs to be addressed. It is also consistent with standard
HL7 practice of declining requests that fall outside a work group's
jurisdiction, and preserves reviewer bandwidth for artifacts this
group actively maintains.

---

### Recommendation

**Recommended disposition:** C

The ticket raises a genuinely valid modelling concern — EMDN is a
type nomenclature and is a poor fit for a device *identifier* binding
— and under different circumstances Disposition B would be the right
answer. However, the resolution text is unambiguous that the EHN
logical models carrying this A.5 section have been replaced by EHDS
models outside this work group's remit. Accepting a change (A or B)
to a superseded artifact would deliver no implementer value and
would not reach the EHDS successor, where the fix actually needs to
land. The workgroup should confirm Disposition C, record the
type-vs-instance analysis from Dispositions A/B in the ticket
comments so it is preserved for the EHDS maintainers, and, if
desired, forward the feedback to the appropriate EHDS governance
channel.
