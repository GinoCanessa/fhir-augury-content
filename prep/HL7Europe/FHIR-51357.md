## Summary

The reporter (Alexander Zautke) is providing feedback on the EU Imaging
Study Report (R5) IG concerning the coding system used for imaging
procedures. The submitter expresses support for the IG's choice of
SNOMED CT as the preferred semantic standard, noting alignment with both
German national policy (SNOMED CT as the basic semantic standard for
clinical applications) and the eHN strategy for the cross-border
exchange of medical imaging studies and reports under Directive
2011/24/EU (which references SNOMED CT and LOINC). At the same time the
submitter asks the work group to ensure that existing standards already
in use in the imaging field — explicitly DICOM — are also considered
when selecting coding systems for imaging procedures. In comments,
Alexander Zautke specifically asks whether the binding description of
`Procedure.code` could be extended so that it covers more than just
SNOMED CT, and Bas van den Heuvel (HL7 Europe) responds that the binding
is already extensible (not required), so other code systems may be
used. The ticket is currently `Triaged` with a proposed resolution of
"Considered - No action required", with the resolution comment pointing
at the existing `im-procedure-type` value set in the IG as already
satisfying the request.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items for this submission; the only auto-detected reference is to the
FHIR core element `Procedure.code` mentioned in the ballot comment.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `Procedure.code` element in the IG's imaging procedure
profile (and the corresponding narrative in the EU Imaging Study Report
R5 IG) so that the **binding description** explicitly enumerates the
acceptable / encouraged code systems for imaging procedures: SNOMED CT
(preferred), LOINC (for procedure types where LOINC has appropriate
coverage), and DICOM (e.g. CIDs / DICOM-controlled terminology where
DICOM-defined procedure-type codes are already in operational use). The
binding strength remains `extensible` against the existing
`im-procedure-type` value set, but the description text is broadened so
implementers reading the IG see that DICOM-coded procedures are
explicitly in scope, and the value set itself can be extended (or
companion value sets added) to enumerate the allowed DICOM-derived
codes.

#### Justification

This directly answers the submitter's clarifying request ("extend the
binding description of `Procedure.code` … to cover more than just SCT")
and explicitly recognises DICOM, which is the dominant existing
standard in radiology departments. Because the binding is already
extensible, expanding the descriptive guidance and (optionally) the
value-set membership does not break the conformance contract for
existing implementers — it only widens what is sanctioned. It also
keeps the IG aligned with the eHN cross-border guideline, which calls
for both SNOMED CT and LOINC, while honouring the German national
policy preference for SNOMED CT as the primary semantic standard.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the current `Procedure.code` extensible binding to
`im-procedure-type` exactly as-is, but address the submitter's concern
with a documentation-only change: add a short "Coding-system guidance"
section to the IG narrative for the imaging procedure profile that (a)
restates that the binding is extensible and therefore non-SNOMED codes
(notably DICOM and LOINC) may be supplied, (b) gives a concrete example
of a DICOM-coded `Procedure.code`, and (c) cross-links to the eHN
imaging guideline. No structural / value-set changes; the text simply
makes the existing flexibility discoverable.

#### Justification

The technical complaint ("only SCT is allowed") is, per Bas van den
Heuvel's reply, a misunderstanding — the binding is already extensible.
A targeted narrative clarification is the lowest-risk way to close the
gap between the submitter's perception and the IG's actual conformance
behaviour, without re-opening value-set scope discussions or committing
the work group to curate a DICOM-derived value set. It also matches the
proposed resolution already on the ticket ("Considered - No action
required", pointing at the published value set), but adds a concrete
documentation deliverable so future implementers do not raise the same
question.

---

### Disposition C: Decline

#### Proposal

Decline to make any change. Resolve the ticket as **Considered - No
action required** (the resolution currently proposed in the snapshot),
on the grounds that the IG already specifies an *extensible* binding on
`Procedure.code` to the `im-procedure-type` value set, which by
FHIR-binding semantics permits other code systems (including DICOM and
LOINC) to be used where the value set does not contain a suitable
concept. Refer the submitter to the published [Imaging Procedure
Type](https://build.fhir.org/ig/hl7-eu/imaging-r5/en/ValueSet-im-procedure-type.html)
value set and to the FHIR rules for extensible bindings.

#### Justification

The submitter's concern is fully addressed by existing FHIR semantics:
extensible bindings explicitly allow additional codings outside the
named value set when no suitable concept is present. No spec defect has
been identified, no concrete missing concept has been named, and no
implementation has been blocked. Adding extra descriptive prose or
broadening the value set without a specific use case risks scope creep
and inconsistency with how `Procedure.code` is described elsewhere in
the IG family.

---

### Recommendation

**Recommended disposition:** B

The technical state of the IG is already correct (Disposition C is
defensible on pure conformance grounds), and the submitter has not
proposed a specific code or use case that the value set fails to cover
(making Disposition A heavier than the evidence warrants). However, the
fact that a national affiliate raised the question at all — and that
the chat exchange ended with the work group asking the submitter to
clarify whether the change should happen at European or country level —
shows that the *binding description* and surrounding narrative are not
making the existing flexibility obvious. A small, documentation-only
clarification on the imaging-procedure profile (per Disposition B) puts
DICOM and LOINC explicitly on the page, satisfies the spirit of the
submission and the eHN guideline, and keeps the door open for a future
ticket to add specific DICOM-derived codes to the value set if a
concrete use case is brought forward. It also lets the work group keep
the proposed "Considered - No action required" resolution honest by
attaching one tangible IG-narrative change instead of relying purely on
"the binding is extensible — read the spec."
