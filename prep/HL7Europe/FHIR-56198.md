## Summary

The ticket asks the HL7 Europe **EU Imaging Study Report (R4)** IG to stop
duplicating terminology and structure definitions that already exist in the
IHE **Manifest-based Access to DICOM Objects (MADO)** specification, and
instead reuse the MADO artifacts directly. The reporter (Bas van den Heuvel)
identifies four specific overlap areas: (1) the accession number and study
instance UID identifier definitions, (2) the requested procedure and
imaging study references, (3) the anatomical region value set, and
(4) the DocumentReference search parameters for bodysite and modality.
The ticket has already been triaged with resolution **Persuasive** and a
resolution note stating "We will update the spec accordingly," so this
review is to confirm the disposition and frame the implementation. There
are no cross-referenced Jira tickets and no comments on the ticket itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **EU Imaging Study Report (R4)** IG so that, wherever it
currently restates artifacts that already exist in the IHE MADO
specification, it instead **references the MADO artifacts** as the
authoritative source. Concretely:

1. **Identifiers.** Replace the local Identifier definitions for
   *accession number* and *study instance UID* with references to (or
   profiles derived from) the MADO identifier system/type definitions,
   keeping the same `system` URIs MADO uses so existing instances stay
   valid.
2. **References.** Replace the local definitions of *requested
   procedure* and *imaging study* references (e.g., on
   `ServiceRequest`, `DiagnosticReport`, or the IG's report profile)
   with references typed to the MADO `ServiceRequest`/`ImagingStudy`
   profiles.
3. **Anatomical region value set.** Retire the IG-local anatomical
   region `ValueSet`; bind the corresponding element(s) to MADO's
   anatomical region value set (by canonical URL), at the existing
   binding strength.
4. **DocumentReference search parameters.** Remove the IG-defined
   `bodysite` and `modality` SearchParameter resources for
   `DocumentReference` and instead reference the MADO-defined
   SearchParameter canonicals.

For each change, leave a deprecation note in the IG narrative pointing
implementers to the MADO canonicals, and add a "Relationship to MADO"
section listing the reused artifacts.

#### Justification

This matches both the literal request and the recorded resolution
("Persuasive — we will update the spec accordingly"). Reusing upstream
canonicals is the standard FHIR-IG approach to avoid divergent
definitions of the same concept across IHE and HL7 specs in the imaging
space, and aligns with the Australian Zulip thread's observation that
the EU/MADO artifacts should be a single source of truth that other
jurisdictions point at. It also reduces the maintenance surface for the
EU Imaging Study Report IG — every future MADO terminology fix
propagates automatically rather than requiring a synchronized errata in
two specs.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of reusing MADO canonicals one-for-one, **publish the shared
artifacts in a separate "EU Imaging Common" companion module** (or
existing HL7 EU imaging-r5 module if scope allows) and have **both** the
EU Imaging Study Report IG and MADO depend on that module. The shared
module would own:

- the accession number / study instance UID Identifier definitions,
- the ImagingStudy/requested-procedure reference profiles,
- the anatomical region ValueSet,
- the DocumentReference `bodysite` and `modality` SearchParameters.

The EU Imaging Study Report IG would then drop its local copies and
import the shared module; MADO would be invited (via IHE liaison) to do
the same in a future revision. Until MADO migrates, the EU IG can still
reference the MADO canonicals as equivalents.

#### Justification

This addresses the same duplication problem but acknowledges the
governance asymmetry: MADO is an IHE deliverable while the EU IG is an
HL7 Europe deliverable, and pointing the HL7 IG at IHE-owned canonicals
ties the HL7 release cadence to IHE's. A neutral shared module keeps
both specs aligned without either becoming a hard dependency of the
other, which the Zulip discussion implied is a real concern for
downstream adopters (e.g., AU jurisdictions) that want to pick up MADO
without inheriting an EU-specific Bundle definition or vice-versa. The
trade-off is that the shared module needs a publication path and IHE
buy-in, which Disposition A does not require.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the EU Imaging Study Report IG's local
definitions of accession number, study instance UID, requested
procedure / imaging study references, the anatomical region value set,
and the DocumentReference `bodysite` / `modality` search parameters.
Document the equivalence to MADO in a non-normative "Relationship to
MADO" appendix so implementers can map between the two without the IG
taking a normative dependency on MADO.

#### Justification

The defensible reason to decline would be **governance independence**:
MADO is an IHE-published artifact at `build.fhir.org/ig/IHE/RAD.MADO`
and is currently `0.0.1-current`, i.e., not yet on a stable, versioned
publication. Taking a normative dependency on a pre-release
external-SDO canonical would expose the EU Imaging Study Report IG to
breaking changes outside HL7's control. Keeping local definitions plus
an equivalence appendix preserves implementer clarity without coupling
two release schedules.

---

### Recommendation

**Recommended disposition:** A

The work group has already resolved this as **Persuasive**, the
duplication is concrete and well-scoped (four specific artifact
families), and reusing MADO canonicals is the cleanest way to keep the
two specifications from drifting. Disposition B (shared companion
module) is architecturally cleaner long-term but is materially more
work and depends on IHE coordination that the current ticket does not
fund; it is a good candidate for a follow-up ticket once the immediate
duplication is removed. Disposition C is hard to defend given the
existing resolution and the explicit reporter request. Recommend
proceeding with Disposition A, with the caveat from C noted: pin the
MADO references to a specific MADO version once MADO publishes a stable
release, and track MADO's versioning explicitly in the EU IG's
dependency manifest so a MADO breaking change is a visible event rather
than a silent regression.
