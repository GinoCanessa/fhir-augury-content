## Summary

The ticket asks the EU Laboratory Report (FHIR) IG to broaden the SNOMED CT
value set used to express morphology of a body structure. Today the IG's
value set is bound to the SNOMED CT hierarchy `< 49755003 |Abnormal tissue
appearance|`. SNOMED International, in feedback to the Xt-EHR D7.1
deliverable, recommends instead binding to the wider hierarchy
`< 118956008 |Body structure, altered from its original anatomical
structure (morphologic abnormality)|`, which more completely covers the
range of morphological abnormalities (tissue changes, malformations,
inflammatory features, neoplasms, etc.) that lab and pathology reports
need to communicate. The ticket is already resolved as **Persuasive**
with the resolution direction "Use `< 118956008 |Body structure, altered
from its original anatomical structure (morphologic abnormality)|`" — so
the workgroup vote is essentially endorsing that change.

The request sits next to several adjacent tickets that the workgroup will
likely want to weigh together: [FHIR-56207](https://jira.hl7.org/browse/FHIR-56207)
and [FHIR-56205](https://jira.hl7.org/browse/FHIR-56205) capture two
other Xt-EHR D7.1 SNOMED feedback items against the same EU Laboratory
Report IG (susceptibility-test substances and specimen container
hierarchy). On the FHIR core side, [FHIR-47616](https://jira.hl7.org/browse/FHIR-47616)
(Persuasive, Applied) cleaned up the core `BodyStructure.morphology`
example value set, and [FHIR-56262](https://jira.hl7.org/browse/FHIR-56262)
asks O&O to allow `0..*` cardinality on
`BodyStructure.includedStructure.morphology` — both are the upstream
"core" companions to this EU IG binding change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution as written: rebind the EU Laboratory Report (FHIR)
IG's morphology value set from
`< 49755003 |Abnormal tissue appearance|` to
`< 118956008 |Body structure, altered from its original anatomical
structure (morphologic abnormality)|`. Concretely, update the
intensional value set definition (the `compose.include` ECL filter that
today targets `< 49755003`) to target `< 118956008` instead, regenerate
the expansion against the SNOMED CT International edition version
declared by the IG, and update any narrative / mapping pages that
reference the previous hierarchy. Add a short note to the IG's value-set
narrative attributing the change to SNOMED International's Xt-EHR D7.1
review.

#### Justification

This is exactly the change SNOMED International recommended in the
Xt-EHR D7.1 review and matches the resolution the WG already marked
**Persuasive**. The `118956008` hierarchy is the SNOMED-curated parent
for "morphologic abnormality" and is a strict superset of `49755003`,
which means the change is non-breaking for any system that was already
posting codes from the narrower hierarchy — those codes remain valid —
while opening the binding up to malformations, neoplastic morphologies,
inflammatory features, and other morphology types that lab reports
legitimately need. Aligning the IG with SNOMED's own recommended
hierarchy also reduces the risk of future SCT releases moving concepts
out of `49755003` and silently shrinking the IG's expansion.

---

### Disposition B: Alternative Approach

#### Proposal

Rebind to a curated subset of `< 118956008` rather than the full
hierarchy. Define the value set as `< 118956008` **excluding**
disease/pathology-flavored branches that historically caused trouble in
the core `BodyStructure.morphology` binding (the kinds of codes
[FHIR-47616](https://jira.hl7.org/browse/FHIR-47616) had to remove —
e.g., specific carcinomas and named tumors). Express this as
`compose.include` with `< 118956008` plus `compose.exclude` blocks for
the unwanted descendant subtrees, and document the exclusion rationale
in the IG. Optionally pair this with raising
`BodyStructure.includedStructure.morphology` cardinality to `0..*` if
[FHIR-56262](https://jira.hl7.org/browse/FHIR-56262) lands in core.

#### Justification

The core spec has already been through a cleanup cycle on this exact
binding ([FHIR-47616](https://jira.hl7.org/browse/FHIR-47616)) precisely
because `118956008` includes a broad set of pathology-style codes that
clinicians and modelers consider "diagnoses", not "morphology". Adopting
the full hierarchy in the EU IG re-imports that ambiguity and may invite
similar QA rework later. A curated subset preserves the intent of
SNOMED's recommendation (a wider, more complete morphology coverage)
while keeping the binding consistent with the trimmed core example value
set. The trade-off vs Disposition A is more authoring/maintenance burden
(the exclusions must be reviewed each SCT release) and a small risk that
the WG's exclusion list disagrees with SNOMED's own intent.

---

### Disposition C: Decline

#### Proposal

Decline the rebinding and keep the existing `< 49755003 |Abnormal
tissue appearance|` value set. Document the decision in the IG and the
ticket, citing alignment with the (recently-curated) FHIR core
`BodyStructure.morphology` value set and the desire to avoid mixing
"morphology" with diagnostic/pathology concepts. Refer the SNOMED
International recommendation back to the Xt-EHR liaison and to the O&O
WG for upstream consideration.

#### Justification

Defensible on the grounds that `49755003` is the historical "morphology
proper" subtree and that broadening it imports pathology codes the core
spec just finished removing ([FHIR-47616](https://jira.hl7.org/browse/FHIR-47616)).
However, this stance contradicts the WG's already-recorded **Persuasive**
resolution and the explicit SNOMED International recommendation, so it
would only be appropriate if the WG has substantively new information
about the downstream cost of the broader binding.

---

### Recommendation

**Recommended disposition:** A

The ticket is already resolved Persuasive and the resolution text
matches Disposition A verbatim, so the WG's main job at the block vote
is to confirm that the implementation will follow through. The
SNOMED-International recommendation is authoritative on SNOMED
hierarchies, the change is additive (every code in the current
expansion remains valid), and it directly closes a piece of Xt-EHR D7.1
feedback that Europe is committed to addressing. Disposition B is a
reasonable engineering refinement but should be handled as a follow-up
QA pass once the broader binding is in place — not as a reason to delay
the rebinding itself; if the WG wants to pre-empt the pathology-vs-
morphology drift that bit core in [FHIR-47616](https://jira.hl7.org/browse/FHIR-47616),
file a companion ticket to QA the expansion after the rebind. Disposition
C should only be considered if new information has emerged since the
Persuasive vote.
