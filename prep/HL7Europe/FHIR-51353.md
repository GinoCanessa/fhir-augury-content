## Summary

The ticket reports a binding/value-set inconsistency in the HL7 Europe
*EU Imaging Study Report (R5)* IG. The `ImagingStudy` profile slices
`ImagingStudy.series.performer` into three named slices — `performer`,
`device`, and `custodian` — and binds `performer.function` (extensibly)
to the value set **`ImImagingStudyPerformerTypeVS`**. The slice
discriminator pattern fixes `function` for each slice to a code from
`http://terminology.hl7.org/CodeSystem/v3-ParticipationType`:
`PRF` (performer), `CST` (custodian), and `DEV` (device). The reporter
observes that the `DEV` code is **not** included in the current contents
of `ImImagingStudyPerformerTypeVS` (per the connecthaton39 branch
build), so a conformant device-performer slice cannot satisfy the
binding without flipping to `extensible`'s escape hatch — and even then
fails the spirit of the constraint. The fix is therefore either to add
`v3-ParticipationType#DEV` (and ideally also `CST`) to the value set
so that all three slice-fixed codes are members, or to relax/realign the
binding so that the slice patterns and the bound value set agree.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the value set `ImImagingStudyPerformerTypeVS` (canonical
under `http://hl7.eu/fhir/imaging/...`) so that **all three codes
fixed by the slice patterns** are explicit members:

- `http://terminology.hl7.org/CodeSystem/v3-ParticipationType#PRF` (performer)
- `http://terminology.hl7.org/CodeSystem/v3-ParticipationType#CST` (custodian)
- `http://terminology.hl7.org/CodeSystem/v3-ParticipationType#DEV` (device) ← currently missing

Add `DEV` (and verify `CST` is present; if not, add it). Keep the
binding on `ImagingStudy.series.performer.function` at **extensible**.
Re-publish the value set on the `connecthaton39` branch and any other
active branches of `hl7-eu/imaging`, and regenerate the IG so the
ValueSet page lists `DEV` as a concept.

#### Justification

The slice discriminator pattern in the profile fixes
`performer.function` to a specific code per slice. Even though the
binding strength is `extensible` (which technically permits codes
outside the value set when no suitable code exists), it is poor
practice for a profile to *itself* fix a code that lies outside its
own bound value set: it produces noisy validator warnings, confuses
implementers about which codes are "officially" expected, and
contradicts the intent of declaring the value set in the first place.
Adding `DEV` (and `CST` if missing) costs essentially nothing — the
codes already exist in `v3-ParticipationType` — and makes the
profile's slicing self-consistent with its binding. This is the
narrowest, most conservative fix that preserves all existing
slice/cardinality decisions and reference targets.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the existing value set, **align the binding to a
value set that already covers the three required codes** — or
introduce a single, purpose-built value set that contains exactly
`{PRF, CST, DEV}` from `v3-ParticipationType` — and bind
`ImagingStudy.series.performer.function` to it as **required** (since
the profile's own slicing leaves no genuine choice for conformant
instances). Concretely, define
`ImImagingStudyPerformerTypeVS` (or a successor with a clearer name,
e.g. `ImImagingStudyPerformerFunctionVS`) as the closed enumeration
`{PRF, CST, DEV}` and tighten the binding strength from `extensible`
to `required`.

#### Justification

The current profile is, in practice, *closed* on
`performer.function`: every slice fixes a specific code, and the
slicing rule (`type` discriminator on `actor` resolving via
`function`) leaves no room for an instance to invent a new code
without falling outside one of the three named slices. An
`extensible` binding therefore advertises flexibility that the
slicing immediately removes. A `required` binding to a tightly-scoped
value set is more honest about the profile's actual conformance
contract and gives implementers and validators a sharper signal.
Trade-off vs. Disposition A: this is a (mild) breaking change for any
implementer that has been emitting custom `function` codes outside
`{PRF, CST, DEV}`, and it forecloses the workgroup's future ability
to add a fourth performer slice without simultaneously revising the
value set. Disposition A keeps the door open; Disposition B closes it
deliberately.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing **extensible**
binding strength on `ImagingStudy.series.performer.function`
*already* permits the use of `v3-ParticipationType#DEV` even though
that code is not enumerated in `ImImagingStudyPerformerTypeVS`. Per
the FHIR binding-strength rules, an extensible binding allows codes
outside the value set when no suitable code exists in the value set —
so instances conforming to the `device` slice are technically valid
today, and no spec change is required.

#### Justification

This disposition would only be defensible if the workgroup explicitly
prefers the value set to enumerate only the *most common* performer
function codes and intends `extensible` to absorb edge cases like
`DEV`. It minimises churn — no value-set rev, no IG re-publish — but
leaves the profile in a state where its own slice patterns reference a
code outside its own bound value set, which most reviewers and
validators will (correctly) flag as a smell. Declining is the weakest
option because nothing about `DEV` makes it an "edge case": it is one
of the three first-class slices the profile defines.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cheapest, least risky, and most defensible fix.
The reporter is pointing at a genuine internal inconsistency in the
HL7-EU Imaging IG: the profile fixes `function = DEV` for one of
its own first-class slices but binds the same element to a value set
that does not include `DEV`. Adding `DEV` (and verifying `CST`) to
`ImImagingStudyPerformerTypeVS` removes the inconsistency without
breaking any existing instances and without committing the workgroup
to the tighter, harder-to-revisit binding-strength change in
Disposition B. Disposition B remains a sensible follow-up if HL7
Europe later wants to lock the contract down with a `required`
binding, but it is not required to resolve the bug being reported here.
Disposition C should be rejected because it relies on the
extensible-binding escape hatch to paper over a self-inflicted
inconsistency in a profile the workgroup itself authors and controls.
