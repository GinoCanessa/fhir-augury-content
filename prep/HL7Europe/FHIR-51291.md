## Summary

The reporter, raising input from the IHE Vienna Connectathon, asks the
HL7 EU *Imaging Study Report (R5)* IG to require Endpoint resources to
expose **WADO-URI** in addition to the currently-required **WADO-RS**.
The motivation is to lower the bar for "simple viewers and patient-facing
apps" that today can consume a single rendered image via a WADO-URI
query string but cannot easily integrate with the multipart, byte-range,
DICOM-aware WADO-RS API. No directly-linked Jira tickets were found, and
the only cross-reference is the HL7 EU Imaging block-vote announcement
on Zulip ([europe > HL7 Imaging Studies and Report FHIR IG](https://chat.fhir.org/#narrow/stream/europe/topic/HL7%20Imaging%20Studies%20and%20Report%20FHIR%20IG/near/531791062))
where this ticket was listed in **Block-Vote-1** with the proposed
resolution **Not Persuasive** ("Support for WADO-URI is not required.").

## Proposed Dispositions

(There are no GitHub cross-references and therefore no repo briefings
to anchor against; the dispositions below are scoped to the
*EU Imaging Study Report (R5) (FHIR)* IG specification artifacts in the
HL7 EU work group's authoring repository — typically `HL7/imaging-eu`
or the equivalent under the `hl7-eu` org — and would touch the
`Endpoint` profile(s) and the conformance / narrative pages that
describe the required DICOMweb retrieval transactions for imaging
manifests. Implementers should confirm the exact authoring root before
applying any change.)

### Disposition A: Accept As Requested

#### Proposal

Update the EU Imaging Study Report (R5) IG so that Endpoints used by
the imaging manifest **MUST** (or at minimum **SHOULD**) advertise a
WADO-URI capability in addition to WADO-RS. Concretely:

- Add a `WADO-URI` connection type to the IG's `Endpoint.connectionType`
  binding (or extend the value set the IG uses for DICOMweb endpoints,
  e.g. via a new code such as `dicom-wado-uri` to complement the
  existing `dicom-wado-rs`).
- Update the imaging manifest profile's narrative / conformance section
  to require that any `ImagingStudy.endpoint` (and any equivalent
  Endpoint reference attached at the series/instance level) provide
  *both* a WADO-RS and a WADO-URI base URL.
- Add an example Endpoint instance demonstrating a WADO-URI base URL,
  and update the IG's "DICOMweb retrieval" page to document
  query-parameter usage (`requestType=WADO`, `studyUID`, `seriesUID`,
  `objectUID`, `contentType`).
- Add an invariant or capability statement entry asserting the
  combined requirement.

#### Justification

The reporter's use case — simple browser-based viewers and
patient-facing apps — is real: WADO-URI returns a single rendered image
via a plain HTTP GET with query parameters, which any HTML `<img>`
element or lightweight mobile client can consume, whereas WADO-RS
typically returns multipart `application/dicom` payloads that require
DICOM tooling. The IG already standardises one DICOMweb retrieval
transaction (WADO-RS); standardising the second known retrieval form
brings the EU IG into closer alignment with the *DICOM PS3.18* family
and IHE Connectathon expectations and removes ambiguity for server
implementers about whether they are allowed to advertise WADO-URI at
all. It also keeps a single, interoperable answer to "where is the
rendered image?" rather than leaving each deployment to invent its own
extension.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** make WADO-URI a hard requirement. Instead, keep WADO-RS as
the only **required** DICOMweb retrieval transaction, and add
**optional, normatively-described** support for WADO-URI in the IG:

- Define an additional, optional `Endpoint.connectionType` code
  (`dicom-wado-uri`) and document its semantics.
- Add a "Capability Profile: Patient-facing / Lightweight Viewer" page
  to the IG that **recommends** servers also expose a WADO-URI Endpoint
  and shows how to advertise it alongside WADO-RS.
- Leave the imaging manifest profile's cardinality on
  `ImagingStudy.endpoint` unchanged; just clarify that multiple
  Endpoints with different connection types are expected in this case.
- Optionally, document an alternative path: many WADO-RS servers
  already support `RetrieveRenderedTransaction` (`/rendered`), which
  serves the same patient-facing / simple-viewer need over the WADO-RS
  API surface. Recommend that servers expose `/rendered` either
  instead of, or in addition to, WADO-URI.

#### Justification

The patient-facing use case is legitimate but does not, on its own,
warrant raising WADO-URI to a *required* endpoint for *every* server
that implements the EU Imaging IG. WADO-URI is increasingly considered
a legacy transaction in the DICOM community (DICOM PS3.18 retains it
but most new deployments standardise on WADO-RS, often plus
`/rendered` for thumbnails and JPEG rendering), and forcing it on
every conformant server would impose implementation cost on
modern PACS that have moved past it. An *optional, well-documented*
WADO-URI profile, plus explicit acknowledgement of the WADO-RS
`/rendered` alternative, satisfies the simple-viewer / patient-app
need without breaking servers that have only WADO-RS.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Not Persuasive** (the resolution already
recorded in Jira and carried into Block-Vote-1). Keep WADO-RS as the
single required DICOMweb retrieval mechanism in the EU Imaging Study
Report (R5) IG. Do not introduce a `dicom-wado-uri` connection type,
do not change `ImagingStudy.endpoint` cardinality or constraints, and
do not add WADO-URI examples or conformance text.

Rationale category: *insufficient use case to justify a normative
requirement; need is already addressable through existing mechanisms*.

#### Justification

WADO-URI is a legacy DICOMweb transaction whose primary capability —
"give me a rendered image at a URL" — is already addressed by the
WADO-RS `RetrieveRenderedTransaction` (`/rendered`), which the IG
already permits via WADO-RS. Mandating WADO-URI in addition would
impose a real implementation burden on conformant servers (most modern
PACS expose only WADO-RS) without delivering capability that is not
otherwise reachable. The reporter cites a single source — discussion at
the IHE Vienna Connectathon — and no implementer has come forward on
the EU Imaging Zulip topic asking for this; in particular the
Block-Vote-1 thread did not surface anyone willing to pull the ticket
to argue against the Not Persuasive resolution. Patient-facing app
authors who genuinely need a single-shot rendered image can rely on
WADO-RS `/rendered` or on a deployment-specific Endpoint with an
extended connection type, neither of which requires a normative IG
change.

---

### Recommendation

**Recommended disposition:** C (with one carve-out drawn from B).

The current Jira resolution (Not Persuasive) is well-founded.
WADO-URI is legacy, the patient-facing rendering use case is already
covered by WADO-RS `/rendered`, and the request lacks any second voice
from implementers — the Block-Vote-1 thread on Zulip did not produce a
single objection to the Not Persuasive disposition. Accepting (A)
would force every EU-IG-conformant server to implement a transaction
the wider DICOMweb community is moving away from, which is a
disproportionate response to a single Connectathon comment. The work
group should confirm Disposition C in Block-Vote-1 as written, but
should consider folding in the small documentation gesture from
Disposition B: a one-paragraph note in the IG that explicitly tells
implementers of "simple viewers / patient-facing apps" to use the
WADO-RS `/rendered` retrieval and points at the relevant DICOM PS3.18
section. That keeps the surface area unchanged while removing the
ambiguity that prompted the ticket in the first place.
