## Summary

The reporter (Dennis Kipping) is asking the HL7 Europe work group to consider
standardising two top-level extensions on `DiagnosticReport` in the EU Base and
Core (R4) profile, motivated by repeated requests from the German national
context. The first extension would mark a `DiagnosticReport` as
**pseudonymized** (German: `KennzeichenPseudonymisierterBefund`); the second
would carry a **critical-result flag**
(`KennzeichenKritischerBefund`). The German laboratory MIO draft profile at
`https://simplifier.net/lab1x0x0/kbv_pr_mio_lab_diagnosticreport` is offered as
a concrete prior-art example. The ticket is one of a cluster of EU-ballot
DiagnosticReport tickets filed by the same reporter for the
`eu-ballot-jan-2026` round (see Related Jira Tickets).

No cross-references to other Jira tickets, Zulip threads, or GitHub items were
recorded for this ticket in FHIR Augury at the time of preparation; the
neighbouring tickets below were surfaced via keyword similarity rather than
explicit linking.

## Proposed Dispositions

Because no GitHub cross-references exist, no per-repo briefing facts are
available; dispositions therefore name the EU Base and Core (R4) artifact
(`DiagnosticReport`) at the spec level rather than a specific authoring path.
Implementation paths would be confirmed once a `repo-analysis` briefing exists
for the EU Base and Core repository.

### Disposition A: Accept As Requested

#### Proposal

Add two new top-level extensions to the `DiagnosticReport` profile in the EU
Base and Core (R4) IG:

1. `diagnosticReport-pseudonymized` (working name) — `boolean`, 0..1, on
   `DiagnosticReport`, indicating that the report's content has been
   pseudonymized (subject identifying information replaced by a pseudonym).
   Description should reference the German `KennzeichenPseudonymisierterBefund`
   precedent and clarify that `true` means the report is pseudonymized as
   distributed (it does not describe the pseudonymisation algorithm).
2. `diagnosticReport-criticalResult` (working name) — `boolean`, 0..1, on
   `DiagnosticReport`, set to `true` when the issuer is flagging the report as
   containing a clinically critical finding requiring expedited handling.
   Bind tightly in the description to the use case (it is *not* a substitute
   for `Observation.interpretation` of individual results).

Both extensions defined in the EU Base and Core (R4) IG, with German MIO
mapping documented in the extension's `mapping` element.

#### Justification

The reporter cites repeated demand in the German national context and a
concrete, already-deployed precedent in the KBV MIO laboratory profile.
Standardising at the EU level (rather than each member state inventing its own
flag) is the explicit motivation for an EU Base and Core profile and is the
kind of harmonisation the EU work group exists to broker. Top-level extensions
on `DiagnosticReport` are non-breaking (extensions are always optional in
FHIR), so the cost to other implementers is minimal.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add EU-Core-defined extensions. Instead:

1. For **pseudonymisation**, recommend the existing security-labels mechanism:
   apply `Resource.meta.security` with the v3 `ActCode` value
   `PSEUDED` (Pseudonymized) — already a normative HL7 value in
   `http://terminology.hl7.org/CodeSystem/v3-ActCode` — and add EU Core
   guidance text describing this as the canonical way to flag a pseudonymized
   `DiagnosticReport`. This is consistent with how confidentiality and other
   handling caveats are conveyed in FHIR.
2. For **critical result**, prefer encoding criticality on the underlying
   `Observation` instances via `Observation.interpretation` (with
   `http://terminology.hl7.org/CodeSystem/v3-ObservationInterpretation` codes
   such as `HH`/`LL`/`AA`) and surface the report-level signal via
   `DiagnosticReport.conclusionCode` or by setting
   `DiagnosticReport.priority` (in R5; in R4 via the existing
   `workflow-supportingInfo` / `flag-detail` patterns) rather than a new
   boolean. If a single boolean is essential for the German use case, define
   it in the *German national* extension pack and cross-link it from EU Core
   as a recognised national variant.

#### Justification

Both needs may already be expressible with existing FHIR machinery:
pseudonymisation has a normative security label, and clinical criticality is
traditionally expressed at the observation level (where the criticality
actually applies). Introducing two new boolean extensions risks (a) duplicating
existing mechanisms, (b) creating semantic drift between EU implementers who
adopt the extension and those who use the standard mechanisms, and (c) baking
into EU Core a flag whose meaning ("pseudonymized" — partially? fully? per
which scheme?) is hard to pin down with a `boolean`. This disposition
preserves the German use case while keeping EU Core minimal and aligned with
existing FHIR/HL7 patterns.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** for EU Core scope, on the rationale that:

* the `pseudonymized` semantics are already expressible via
  `Resource.meta.security` `PSEUDED`, so a new extension is duplicative;
* the `criticalResult` flag is a clinical-interpretation concept that belongs
  on `Observation.interpretation` rather than as a boolean on the report
  envelope;
* recommend that both flags continue to live in the German national
  (KBV MIO) profile pack rather than in EU Core, and reopen only when at
  least one additional EU member-state realm presents a documented use case
  that cannot be met by the existing mechanisms.

#### Justification

The EU work group has, in this same ballot cycle, repeatedly chosen to defer
or decline EU-Core additions when an existing FHIR/HL7 mechanism already
covers the need or when a global / external extension is expected (see
**FHIR-55457**, deferred pending a global extension, and **FHIR-55543**,
declined in favour of inheritance from Composition core). Single-realm demand
("numerous requests in Germany") without a stated second-realm signal is a
weak basis for adding to EU Core, especially where standard HL7 vocabulary
(`v3-ActCode PSEUDED`) and existing observation-level interpretation codes are
available.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible starting point for the work group. The
underlying needs (signal pseudonymisation; surface clinical criticality) are
real and recurring, so a flat decline (C) under-serves the reporter and the
German implementers who already have the requirement in production MIOs.
However, accepting two new EU-Core booleans verbatim (A) duplicates a
normative HL7 security-label code (`PSEUDED`) and locates a clinical-criticality
signal at the wrong layer of the model (the report envelope rather than the
underlying observations). Disposition B threads that needle: it satisfies the
use case using mechanisms FHIR already provides, gives EU Core a small piece
of *guidance* (which is exactly what an EU base profile should contribute) and
leaves the door open to introducing an extension later only if a concrete
second realm requires it. It is also consistent with this work group's recent
ballot pattern of preferring reuse and harmonisation over net-new EU-only
extensions.
