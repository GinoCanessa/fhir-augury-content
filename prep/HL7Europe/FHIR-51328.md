## Summary

The reporter observes that the EU Imaging Study Report (R5) IG defines a Patient
profile whose only material difference from the base Patient is the addition of
a `birthTime` extension. They request that this extension be moved out of the
imaging-specific profile and into the (EU) base Patient profile, so that no
imaging-specific Patient profile is needed at all. A subsequent comment from
Giorgio Cangioli notes that `birthTime` is in fact already part of the EU base
Patient profile (`patient-eu`), which—if confirmed against the current EU base
release—means the imaging profile's added constraint is redundant and can be
removed in favour of inheriting from EU base Patient. The reporter's earlier
comment (bvdh) explains the original motivation: DICOM supports a precise
patient birth time, and `Patient.birthDate` only carries a date, so an
extension was needed; the question for the workgroup is whether to rely on the
EU base patient extension or keep an imaging-local one.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the imaging-local `birthTime` extension and the dedicated EU Imaging
Patient profile from the EU Imaging Study Report (R5) IG, and update every
reference inside the IG (e.g., `ImagingStudy.subject`, `DiagnosticReport.subject`,
example instances, narrative) to point at the EU base Patient profile
(`https://hl7.eu/fhir/base/StructureDefinition/patient-eu`), which already
defines the `birthTime` extension per Giorgio Cangioli's comment. Mark the
removed Imaging Patient profile and any removed extension URL as deprecated
(or as a redirect/alias) for one cycle so that existing implementations are
not silently broken.

#### Justification

This matches the reporter's request literally and eliminates a profile whose
only delta is an extension already provided one layer up. It reduces the
EU-IG profile surface, removes a parallel definition of the same concept, and
keeps the imaging IG aligned with the EU realm baseline. Bart van den Herik's
own rationale for the imaging extension—DICOM round-tripping of a precise
birth time—remains satisfied, because the EU base Patient already carries
that same extension. It is also consistent with general FHIR IG practice:
profiles should add value over their base, and a profile whose sole
constraint is duplicated upstream is a maintenance hazard.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the EU Imaging Patient profile but redefine it to extend the EU base
Patient profile (`patient-eu`) instead of base FHIR `Patient`, and drop the
locally-declared `birthTime` extension slice in favour of the inherited one.
The imaging profile then only carries imaging-specific constraints (e.g.,
must-support flags driven by DICOM C-FIND/C-MOVE behaviour, identifier
slicing for MRN vs. accession-derived identifiers, narrative requirements);
if no such delta exists today, leave the profile in place but empty as a
named anchor for future imaging-specific tightening.

#### Justification

This preserves a stable canonical URL for the imaging "Patient" target type,
which is convenient for downstream profiles, search-parameter scoping, and
implementer documentation, while still removing the duplicated extension
declaration. It avoids a breaking change for any implementer that has
already validated against the imaging profile's canonical URL, and it gives
HL7 Europe a clean place to layer imaging-specific constraints later
without re-introducing a profile URL. The cost is a profile that is, today,
"just an alias" — which some reviewers will reasonably consider clutter,
and which is the very thing the reporter is pushing back on.

---

### Disposition C: Decline

#### Proposal

Persuaded that the EU base Patient may evolve independently of imaging
needs, retain the imaging-specific `birthTime` extension and the dedicated
EU Imaging Patient profile as currently published. Document in the profile
narrative why the imaging IG carries its own copy (DICOM round-trip,
decoupling from EU base Patient release cadence).

#### Justification

This is defensible only if (a) the EU base Patient's `birthTime` extension
URL or semantics differ from the imaging IG's version, (b) the EU base
release cadence is materially slower than imaging needs, or (c) the
imaging WG wants to be insulated from future EU base changes. Giorgio's
comment makes (a) look unlikely, but it has not been verified element-by-
element in this prep. If the URLs/cardinalities are in fact identical, this
disposition is hard to justify and amounts to maintaining a duplicate.

---

### Recommendation

**Recommended disposition:** A

Giorgio Cangioli has confirmed on-ticket that `birthTime` is already a
first-class part of the EU base Patient profile, and the reporter's
request is the obvious cleanup that follows from that confirmation. The
DICOM-driven need for sub-day precision is preserved by relying on the EU
base extension, so no implementer capability is lost. The principal risk
is a breaking change for any downstream that referenced the imaging
profile's canonical URL or the imaging-local extension URL; this is
mitigated by deprecating (rather than hard-deleting) the old artifacts for
one publication cycle. Disposition B is a reasonable fallback if the WG
wants to keep an "imaging Patient" anchor URL for future use, but adopting
B without any imaging-specific constraint to add is essentially deferring
the cleanup the reporter asked for. Before the ballot vote, the WG should
verify that the extension URL on the EU base Patient is exactly the one
the imaging IG would need (canonical URL, value type `dateTime`,
cardinality `0..1`); if it is not, that mismatch should be raised with
the EU base maintainers as a separate item rather than used to justify
keeping a duplicate in imaging.
