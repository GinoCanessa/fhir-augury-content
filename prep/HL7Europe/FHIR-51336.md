## Summary

The reporter, reviewing the EU Imaging Study Report (R5) IG, observes that the
`Manifest_*` profiles all carry the publisher-generated note "not used by any
profiles in this Implementation Guide". They have interpreted these profiles as
data models that describe the shape of content delivered inside a bundle,
rather than as resources that a server is expected to expose for direct query.
They ask the IG to clearly distinguish, for each profile, whether it is an
individually query-able artifact (reachable via a standard FHIR `GET` /
SearchSet interaction) or whether it only appears as a component of a larger
document/manifest bundle.

The ticket also flags an internal tension in the manifest page: one passage
describes the manifest as a FHIR bundle containing `ImagingStudy` plus at least
one `Endpoint`, while another describes a document-based deployment where a
"document" (Composition-rooted bundle) is searched for and provides URLs to
the imaging content from a PACS. The reporter asks whether a FHIR document is
mandatory, or whether returning `ImagingStudy` (which already carries
`Endpoint` references) in a normal SearchSet bundle is also permitted. Their
preference is to allow standard SearchSet bundles to keep the IG aligned with
typical FHIR workflows and minimise implementation cost. They suggest the IG
make this split explicit per profile (their hypothesis: "deployable" vs "data
model" could be expressed via `StructureDefinition.kind = resource` vs
`logical`) and provide worked examples for both a Document Approach
(`GET {Document URL indexed by MHD DocumentReference}` returning a
document-type bundle conforming to the manifest/report bundle) and a
Single-Resource Search (`GET ImagingStudy` / `GET DiagnosticReport` returning a
SearchSet bundle conforming to FHIR core).

The single cross-reference surfaced for this ticket is a textual mention of
the FHIR core element `StructureDefinition.kind`; no other Jira tickets, Zulip
threads, or GitHub items were linked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the EU Imaging Study Report (R5) IG to explicitly classify every
profile as either **query-able** (a server SHALL/MAY expose it via a standard
FHIR REST interaction) or **manifest-internal data model** (only valid as a
component inside a manifest/report bundle). Concretely:

1. Add a "Conformance & Query Surface" section to the IG narrative that
   enumerates each profile and labels it `Queryable` or `Bundle-internal`.
2. For every `Manifest_*` profile that is intended as a data model only,
   change `StructureDefinition.kind` from `resource` to `logical` (or, if
   tooling/MHD constraints prevent that, keep `kind = resource` and document
   the bundle-internal status prominently in the profile narrative and
   remove/replace the publisher's "not used by any profiles" boilerplate
   with an explicit "used only inside the EU Imaging Manifest bundle"
   statement).
3. Define a CapabilityStatement (one per actor: PACS source, consumer) that
   lists exactly the query-able resources and SearchParameters, and reference
   it from each query-able profile.
4. Allow **both** delivery styles for `ImagingStudy` results:
   - a **Document Approach** — `GET` the URL indexed by an MHD
     `DocumentReference` and receive a document-type Bundle conforming to the
     EU Imaging Manifest profile, and
   - a **Single-Resource Search** — `GET ImagingStudy` (and
     `GET DiagnosticReport`) returning a standard SearchSet Bundle whose
     entries conform to the EU `ImagingStudy`/`DiagnosticReport` profiles,
     with `Endpoint` reachable via `_include`.
   Provide a worked example for each.

#### Justification

The reporter's confusion is a direct consequence of the IG silently mixing
two FHIR delivery paradigms (document vs REST search) on the same set of
profiles, with no per-profile statement of which interactions are conformant.
This is a documentation / conformance-surface gap, not a model gap, so
explicit labelling plus a CapabilityStatement (the FHIR-native mechanism for
declaring query-ability) is the most direct fix. Permitting both delivery
styles aligns the IG with normal FHIR REST behaviour for `ImagingStudy`
(which already carries `Endpoint` in core) and lowers the barrier for
implementers who do not have an MHD/document infrastructure, while preserving
the document-centric path for deployments that need it. Using
`StructureDefinition.kind = logical` for purely manifest-internal shapes is
the mechanism the reporter himself suggests and is consistent with how other
HL7 IGs distinguish data-model profiles from queryable resource profiles.

---

### Disposition B: Alternative Approach

#### Proposal

Keep all existing `Manifest_*` profiles as `StructureDefinition.kind =
resource` (so existing tooling, validators, and downstream IGs that reference
them continue to work unchanged), and instead address the ticket purely
through **conformance + narrative**:

1. Publish a single normative CapabilityStatement that enumerates the
   query-able resources (`ImagingStudy`, `DiagnosticReport`, `Endpoint`,
   `Patient`, `DocumentReference`) and their SearchParameters, and declare
   every other profile as "bundle-internal — not exposed via REST".
2. Add a top-of-page "How this IG is consumed" section to the IG with two
   diagrams: the Document/MHD path and the SearchSet/REST path, each labelled
   with the exact `GET` URL shape and the expected Bundle type.
3. Replace the IG-publisher boilerplate "not used by any profiles in this
   Implementation Guide" on each manifest-internal profile with an explicit
   "Used as an entry inside the EU Imaging Manifest bundle. Not intended for
   direct REST query." note (an IG-publisher narrative override, not a
   structural change).
4. Make support for the SearchSet path **optional** for servers (a server MAY
   expose `ImagingStudy` directly; if it does, the response SHALL conform to
   the EU `ImagingStudy` profile and SHALL `_include` `Endpoint`), so the
   document-based approach remains the canonical path the IG was designed
   around.

#### Justification

Changing `StructureDefinition.kind` on already-published profiles is a
non-trivial structural change with downstream impact on validators, IG
publisher output, derived profiles in other EU IGs, and any existing
implementations. This alternative resolves the actual reviewer pain — "I can't
tell which profiles I'm supposed to be able to GET" — using only narrative
plus a CapabilityStatement, both of which are the FHIR-native mechanism for
declaring query surface. Making the SearchSet path optional rather than
co-equal preserves the IG's original document-centric design intent (which
matters for MHD-based deployments and for ordering/audit semantics that the
manifest bundle encodes) while still giving non-document implementers an
explicitly supported path.

---

### Disposition C: Decline

#### Proposal

Decline the change request on the basis that the IG's intended consumption
model is the document/manifest path indexed by MHD `DocumentReference`, and
that the per-profile query-ability question is already answerable by reading
the IG's CapabilityStatement (or, if none exists, by adding one — which is a
separate editorial task already implicit in IG conformance requirements).
Close as **Not Persuasive** with a pointer to the existing manifest-bundle
narrative and a note that "not used by any profiles in this Implementation
Guide" is auto-generated IG-publisher text that can safely be ignored by
implementers.

#### Justification

Defensible only if the workgroup's position is that mixing the document and
SearchSet paths in one IG is out of scope for this revision and would dilute
the manifest-centric design. In practice this is hard to defend: the reporter
is not asking for a new modelling primitive, only for the IG to state clearly
which profiles a server is expected to expose — a baseline conformance
expectation for any FHIR IG. Declining without at least adding a
CapabilityStatement leaves the same ambiguity in the IG and is likely to
re-surface as repeat tickets from other implementers.

---

### Recommendation

**Recommended disposition:** B

Disposition B fixes the real problem the reporter ran into — the IG does not
clearly tell an implementer which profiles are query-able versus
bundle-internal — using the FHIR-native mechanism for that purpose
(CapabilityStatement plus per-profile narrative), without taking on the
structural and downstream-tooling cost of flipping `StructureDefinition.kind`
on already-published manifest profiles. It still grants the reporter's core
ask (a SearchSet path for `ImagingStudy`), but as an optional server
capability, which preserves the IG's document-centric design and MHD
integration story. Disposition A is a fine end state but is heavier than
necessary for this ballot cycle and risks breaking derived profiles; if the
workgroup decides query-ability really should be encoded structurally, A can
be adopted in a future revision once the impact on dependent IGs has been
assessed. Disposition C is not recommended — the ambiguity the reporter
identified is real and will recur until the IG declares its query surface
explicitly.
