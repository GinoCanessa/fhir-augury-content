## Summary

HynekKruzik proposes updating the EU Laboratory Report `Bundle` profile so
that it (a) aligns its entry slices with the HL7 Europe base profiles by
adding a `LocationEu` entry, (b) adds the resources required to support
`ServiceRequest.supportingInformation` — specifically `ConditionEu`,
`ProcedureEu`, `MedicationEu` and `MedicationStatementEu`, (c) broadens the
existing observation entry slice so that plain `Observation` (not only
`LaboratoryObservation`) is allowed, because `ServiceRequest.supportingInformation`
may point at non-laboratory observations, and (d) considers constraining
the target resource types of `ServiceRequest.supportingInformation` instead
of leaving it fully open.

The fourth item — tightening `ServiceRequest.supportingInformation` — was
explicitly removed from this ticket by Patrick Werner on 2026-04-19 and
moved to [FHIR-56397](https://jira.hl7.org/browse/FHIR-56397) for
separate tracking. This ticket therefore covers only the Bundle alignment
and the broadening of the Observation slice. The resolution recorded on
the ticket is Persuasive with Modification: the `location` slice is changed
to `LocationEuCore`, new slices for `ConditionEu`, `ProcedureEu`,
`MedicationEu` and `MedicationStatementEu` are added, and the `observation`
slice is broadened to `Observation`. The change has already been pre-applied
to the IG repository (commit `dbae738` by Giorgio Cangioli on 2026-03-27),
which in practice uses the `*EuCore` profile names (`LocationEuCore`,
`ConditionEuCore`, `ProcedureEuCore`, `MedicationEuCore`,
`MedicationStatementEuCore`) and also introduces a `medicationAdministration`
slice bound to the core `MedicationAdministration` resource.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution as pre-applied in commit `dbae738` on
`hl7-eu/laboratory`, updating `input/fsh/profiles/bundle-lab.fsh` in the
EU Laboratory Report Bundle profile:

- Change `entry[location].resource only Location` → `only LocationEuCore`.
- Change `entry[observation].resource only ObservationResultsLaboratoryEu` → `only Observation` (broaden, to allow non-laboratory observations referenced from `ServiceRequest.supportingInformation`).
- Add new entry slices (each `0..*`) bound respectively to `ConditionEuCore`, `ProcedureEuCore`, `MedicationEuCore`, `MedicationStatementEuCore`, and `MedicationAdministration`.
- Leave the `ServiceRequest.supportingInformation` target-type question to [FHIR-56397](https://jira.hl7.org/browse/FHIR-56397), as already split out by patrick-werner.

#### Justification

This is exactly what the resolution records and what has already been
committed to the IG source of truth; no additional design work is
required. It aligns the Bundle with the HL7 Europe base profiles (the
stated goal of the ticket) and closes the gap that prevented
`ServiceRequest.supportingInformation` from dereferencing Condition /
Procedure / Medication[Statement] entries inside the same document. The
commit uses the `*EuCore` naming that is already consumed elsewhere in
the project, and the split-out of item 4 into FHIR-56397 keeps this
ticket focused on document composition rather than reference
constraints.

---

### Disposition B: Alternative Approach (accept with narrowing of the Observation slice)

#### Proposal

Apply all of Disposition A except the Observation change. Instead of
broadening the existing slice to plain `Observation`, keep
`ObservationResultsLaboratoryEu` on the existing `observation` slice and
add a **second** `observation`-type slice — e.g.
`entry contains observationSupporting 0..*` with
`entry[observationSupporting].resource only Observation` — intended
specifically for observations referenced from
`ServiceRequest.supportingInformation`. Optionally constrain that new
slice to profile on the HL7 Europe base Observation (if one exists) to
preserve some minimum conformance rather than dropping to the base FHIR
`Observation` resource.

#### Justification

Broadening the existing `observation` slice from
`ObservationResultsLaboratoryEu` to `Observation` is a loosening of
conformance for every existing consumer of lab-result observations in
the Bundle, not just for `ServiceRequest.supportingInformation` use. A
two-slice approach preserves the strong laboratory-observation
conformance that readers of the EU Laboratory Report rely on, while
still admitting the non-laboratory observations the reporter legitimately
needs. It also matches the slicing style already used in the profile
(separate slices for different semantic roles — patient, specimen,
encounter, device, etc.).

---

### Disposition C: Decline (this ticket, not the underlying need)

#### Proposal

Do not merge the Bundle-entry additions in this release. Revert commit
`dbae738` on `hl7-eu/laboratory` and defer all four items to the next
EU Laboratory Report cycle, after (a) the HL7 Europe base profile
naming (`*Eu` vs `*EuCore`) is frozen, and (b) FHIR-56397 has decided
whether `ServiceRequest.supportingInformation` is constrained at all —
because the whole reason items 1–3 are needed is to satisfy references
from that element.

#### Justification

Items 1–3 only matter because clients want to reach
Condition/Procedure/Medication/MedicationStatement/Observation from
`ServiceRequest.supportingInformation`. If FHIR-56397 ultimately decides
to *constrain* that element to a narrower set of target types, some of
the newly-added Bundle slices may be unnecessary (or conversely, may
need different profile bindings). Ordering the work — decide 56397 first,
then materialize the Bundle — avoids two rounds of churn on the Bundle
profile and the downstream validator caches. This disposition is only
defensible if the block-vote schedule allows the delay.

---

### Recommendation

**Recommended disposition:** A

The ticket is already resolved Persuasive with Modification, the change
has been pre-applied to `hl7-eu/laboratory` in `input/fsh/profiles/bundle-lab.fsh`,
and the only substantively contentious part (constraining
`ServiceRequest.supportingInformation`) has cleanly been moved to
[FHIR-56397](https://jira.hl7.org/browse/FHIR-56397). Disposition B's
concern about loosening the Observation slice is real and worth raising
at the call, but it can be handled as a follow-up refinement (either on
this ticket before ballot close or as a small new ticket) without
blocking the core alignment fix; Disposition A should be confirmed and,
if the workgroup agrees, B's two-slice refinement can be layered on top
rather than replacing A. Disposition C is not recommended: holding the
entire alignment for 56397 creates more rework than it avoids, because
the target-type question is orthogonal to whether Bundle can *carry* the
resources at all.
