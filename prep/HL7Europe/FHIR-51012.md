## Summary

The reporter asks the HL7 Europe Imaging Study Report (R5) IG to constrain
`Bundle.type` on the imaging Manifest profile (`ImImagingStudyManifest`) to a
single, predefined value rather than leaving it open to all `BundleType`
codes. The suggested value is `collection`. Today the profile inherits the
unconstrained `Bundle.type` binding, which means any of `document | message |
transaction | transaction-response | batch | batch-response | history |
searchset | collection | subscription-notification` is allowed — confusing
implementers about what the manifest actually is and what processing rules
apply.

The same concern is raised, in more depth, by the cross-referenced ticket
[FHIR-51388](https://jira.hl7.org/browse/FHIR-51388) ("Exchange of Manifest
Bundle is Unclear, Perhaps Underscoped"), filed by JoshPriebe a month later
and explicitly linked from this ticket's comments. FHIR-51388 makes the same
observation against both `ImImagingStudyManifest` and `ImReport`, and frames
the question more broadly — whether the manifest should be a
`collection`/`document`/`searchset` Bundle, or whether the manifest pattern
is needed at all on top of a `_include`-driven `ImagingStudy` search.

The reporter previously filed [FHIR-51013](https://jira.hl7.org/browse/FHIR-51013)
("don't use imaging manifest") which was resolved **Not Persuasive** in HL7
Europe Imaging Block-Vote-2 (12 Aug 2025). With the manifest pattern now
retained, FHIR-51012 is the narrower follow-up: if the manifest stays, fix
its `Bundle.type` so it has one well-defined meaning.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the EU Imaging Study Report (R5) IG, constrain `Bundle.type` on the
`ImImagingStudyManifest` profile to a fixed value of `collection`
(pattern/fixed binding on `Bundle.type`, with the existing required
binding to `BundleType` retained). Update the profile narrative to say
that the manifest is a `collection` Bundle whose entries are the
`ImagingStudy` (and any related referenced resources) being manifested,
and that no Bundle-level processing semantics (transaction/document/etc.)
apply. Apply the same constraint to `ImReport` only if the work group
agrees with FHIR-51388's parallel observation; otherwise keep that as a
separate decision so this ticket's scope stays on the manifest.

#### Justification

`collection` is the only `BundleType` value whose definition explicitly
matches what the manifest does — group a set of resources that have no
processing semantics at the Bundle level — and the reporter explicitly
nominated it. Fixing the value (rather than just slicing or documenting
it) gives validators something to enforce, removes the implementer
ambiguity called out in both FHIR-51012 and FHIR-51388, and is consistent
with how other HL7 Europe IGs constrain Bundle profiles in this space
(e.g., the document-bundle pattern used elsewhere fixes
`Bundle.type = document`). It is a minimally invasive change to a
profile that will already be republished as part of the ballot
disposition cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Constrain `Bundle.type` on `ImImagingStudyManifest` to `searchset` and
re-express the manifest profile as a constraint on the response Bundle
returned by an `ImagingStudy` search with `_include` (or an
`$everything`-style operation). Document the manifest as the canonical
shape of the searchset response rather than a free-standing artifact a
producer assembles by hand. Keep `collection` as the fixed value only if
the work group concludes the manifest must be exchangeable outside a
search response.

#### Justification

This addresses the deeper concern raised in FHIR-51388: that a
free-standing manifest Bundle duplicates what a FHIR-native
`ImagingStudy?_include=...` search already returns, and that implementers
would be better served by a profile on the search response. It avoids
inventing a new exchange pattern, leverages mechanisms every FHIR server
already implements, and lines up with the reporter's stated preference in
FHIR-51388 for "a FHIR native search on ImagingStudy and get back a
SearchSet Bundle". The trade-off is scope: this is a larger redesign than
FHIR-51012 literally asks for, and would interact with how the IG models
its actors and transactions (FHIR-51128, currently in Block-Vote-2).

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-51012 as **Not Persuasive — Resolved - No Change**, leaving
`Bundle.type` on `ImImagingStudyManifest` bound to the full `BundleType`
value set as inherited from the base `Bundle` resource. Address the
implementer-clarity concern with narrative text in the profile
description rather than a binding/fixed-value constraint.

#### Justification

This is the same path the work group took for the reporter's adjacent
ticket FHIR-51013 and would be defensible if the work group considers
the manifest a future-extensible container that may legitimately need to
appear in different exchange contexts (e.g., wrapped in a `document`
Bundle for archival, or a `transaction` Bundle when a producer
materialises it on demand). Locking `Bundle.type` to a single value would
be a breaking constraint for any implementer already producing a
non-`collection` manifest. The downside is that it leaves both FHIR-51012
and FHIR-51388 effectively unanswered and pushes the ambiguity onto
implementers and reviewers.

---

### Recommendation

**Recommended disposition:** A

The work group has already decided in Block-Vote-2 to keep the imaging
manifest pattern (FHIR-51013 → Not Persuasive). Given that decision, the
reporter's actual ask in FHIR-51012 is small, concrete, and squarely in
the IG's authority: pick a `Bundle.type` and fix it. `collection` is the
only `BundleType` value whose semantics match what the manifest does, the
reporter named it, and FHIR-51388's commenter independently arrived at
the same shortlist. Disposition B is a more honest answer to FHIR-51388
but is out of scope for this ticket and should be tracked there;
treating FHIR-51012 as the narrow profile-level fix lets the work group
close real implementer ambiguity now without prejudicing the larger
"manifest vs. searchset" discussion. Disposition C repeats the
non-answer pattern from FHIR-51013 and would leave both tickets in the
same unresolved state.
