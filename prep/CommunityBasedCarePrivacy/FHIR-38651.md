## Summary

Josh Mandel observes that the PACIO Re-Assessment Timepoints IG's
"Re-Assessment Timepoints Encounter" profile slices and constrains
`Encounter.basedOn` (with the `clinicalImpression` extension nested
inside it) to convey items such as "Assessment and impression at
discharge from SNF". This is at odds with the base FHIR semantics of
`Encounter.basedOn`, which the core spec defines as "the request this
encounter satisfies (e.g. incoming referral or procedure request)" —
i.e., a pointer to a `Request`-pattern resource. The reassessment
timepoint payloads are not requests, so using `basedOn` as the carrier
is a semantic mismatch.

The ticket recommends that the IG instead place the
`http://hl7.org/fhir/us/pacio-rt/StructureDefinition/clinicalImpression`
extension at the **root** of the timepoint `Encounter` (i.e., on
`Encounter.extension` rather than on `Encounter.basedOn.extension`),
preserving the same data carriage while aligning with the base FHIR
definition of `basedOn`. The ticket grew out of a chat conversation in
the PACIO Re-assessment Timepoints stream where Karl Naden
(co-author/steward) acknowledged the concern as valid and explicitly
asked Josh to file an off-cycle JIRA ticket so the WG could consider it
for a possible STU2.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify the **Re-Assessment Timepoints Encounter** profile in the US
PACIO Re-Assessment Timepoints IG (`HL7/fhir-pacio-rt`,
`StructureDefinition-reassessment-timepoints-encounter`) so that the
`http://hl7.org/fhir/us/pacio-rt/StructureDefinition/clinicalImpression`
extension is sliced/required on `Encounter.extension` (the root of the
Encounter resource) rather than on `Encounter.basedOn.extension`. As
part of the same change:

- Remove (or relax to `0..*` with no profile binding) the slicing on
  `Encounter.basedOn` that currently carries the clinical-impression
  payload.
- Update the `clinicalImpression` extension's `context` element so its
  allowed context is `Encounter` instead of `Encounter.basedOn`.
- Update IG narrative, the StructureDefinition examples, and any
  example instances under the IG to put the extension at
  `Encounter.extension`.
- Note this as a breaking change in the IG's change log and call it
  out for the next STU release (STU2 candidate, per Karl Naden's
  suggestion on Zulip).

#### Justification

This aligns the profile with the core FHIR semantics of
`Encounter.basedOn`, which is defined as "the request this encounter
satisfies (e.g. incoming referral or procedure request)" — i.e., a
reference to a `Request`-pattern resource. A clinical impression at
discharge from SNF is not a request, so the current slicing is a
semantic misuse that can confuse implementers, validators, and any
cross-IG consumer that reasons over `basedOn` as a request pointer.
Moving the extension to `Encounter.extension` is the conventional FHIR
mechanism for attaching an additional concept to a resource when no
existing element fits, and it matches the reporter's recommendation
and the steward's on-thread acknowledgement that the concern is valid.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the **conceptual** scope of the change (carry the clinical
impression somewhere semantically appropriate on `Encounter`), but
instead of attaching `clinicalImpression` directly at
`Encounter.extension`, model the link from the timepoint Encounter to
the assessment via an existing element rather than a new extension
slot. Specifically:

- Profile `Encounter.reasonReference` (R4) /
  `Encounter.reason.value[x] : Reference(ClinicalImpression)` (R5) to
  point at the `ClinicalImpression` (or a profile thereof) that the
  timepoint represents.
- Retire the `clinicalImpression` extension over time (mark deprecated
  in the next STU; remove in a subsequent major version), or retain it
  only as a transitional alias while implementers migrate.
- Leave `Encounter.basedOn` alone, used only in its base-spec sense
  (e.g., a `ServiceRequest` that prompted the encounter, when one
  exists).

#### Justification

Replacing an IG-specific extension with a base-FHIR element where one
already fits is generally preferred: it removes a custom artifact
implementers must learn, improves interoperability with other
encounter-consuming systems, and keeps the semantics ("why the
encounter happened" / "what clinical impression it captures")
expressible in vanilla FHIR. The trade-off vs. Disposition A is more
work for implementers (a model migration, not just a path change) and
a longer deprecation horizon, but it leaves the IG cleaner long-term
and avoids the pattern of "extension at root" becoming a permanent
fixture for what is fundamentally a reference relationship. This also
addresses the underlying concern that Josh raised — the misuse of
`basedOn` — without locking PACIO into a different ad-hoc carrier.

---

### Disposition C: Decline

#### Proposal

Decline the change for the current ballot/publication cycle on the
grounds that the IG is already published with `Encounter.basedOn` as
the carrier, that no implementer has reported breakage, and that
moving the extension would be a breaking change to existing instances,
profile slices, validators, and downstream tooling. Document the
semantic mismatch in the IG narrative as a known design decision and
revisit only if a future STU is opened for unrelated reasons.

#### Justification

The PACIO Re-Assessment Timepoints IG is published; any path change to
the `clinicalImpression` extension carrier rewrites every conformant
instance and forces implementers to update servers, clients, and
validators. There is no evidence in the ticket or the linked Zulip
thread that this mismatch has caused real-world ambiguity, and the WG
("Community-Based Care and Privacy [deprecated]") has been retired,
which itself raises a process question about who would own and
shepherd the change. Absent a concrete implementer report or a broader
STU2 effort already in motion, the cost of breaking conformance
exceeds the benefit of tightening semantics that humans can read
correctly from the profile narrative.

---

### Recommendation

**Recommended disposition:** A

The reporter's analysis is correct on the merits — `Encounter.basedOn`
is defined by core FHIR as a request pointer, and the IG's use of it
to nest a `clinicalImpression` extension is a semantic misuse that
will trip up implementers and any consumer that reasons over `basedOn`
generically. The IG steward (Karl Naden) explicitly agreed on Zulip
that the concern is valid and invited the JIRA ticket precisely so it
could be considered for STU2. Disposition B is attractive in
principle, but it conflates two changes (path move + model
re-architecture) and is materially harder to ship in one ballot
cycle; it can be sequenced as a follow-up if the WG (or its
successor) wants to retire the extension entirely. Disposition C
preserves a known semantic bug to avoid breakage, which is hard to
defend once a steward has acknowledged the issue. Accepting the
move to `Encounter.extension` in the next STU, and flagging it
clearly as a breaking change in the IG change log, is the cleanest
path forward. The work group should also confirm the current
ownership/disposition path given that "Community-Based Care and
Privacy" is marked deprecated.
