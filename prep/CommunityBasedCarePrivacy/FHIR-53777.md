## Summary

The submitter (Abderrazek Boufahja) questions the FAST Scalable Consent
Management IG's introduction of a custom `fast-consent-patientId`
SearchParameter on Consent. He argues that, rather than defining a new
parameter that reaches into a referenced Patient's identifier, the IG
could instead use (or enhance) the native chained search form
`patient:identifier` / `patient.identifier`. He also notes that doing so
would harmonize the parameters listed in the IG's "Searching for
Consents" section with those listed for "Consent Subscriptions".

This ticket is closely related to two sibling change requests filed the
same day against the same IG:

- [FHIR-53778](https://jira.hl7.org/browse/FHIR-53778) — raises the
  parallel concern about `fast-consent-organizationId`, where the
  defined expression does not always resolve to an Organization
  (grantee may also be a HealthcareService) and recommends using
  `controller:identifier` / `manager:identifier` instead.
- [FHIR-53781](https://jira.hl7.org/browse/FHIR-53781) — applies the
  same "use the native search parameter" pattern to the AuditEvent side,
  asking whether `fast-auditevent-consent` can be replaced by reuse of
  the standard `entity` search parameter.

All three were resolved Persuasive on 2026-01-26 with a single,
consistent resolution: the IG will remove the cross-resource custom
search parameters (because FHIR-I has indicated such parameters are not
allowed) and will instead mandate, in narrative, that servers support
search by patient identifier and by the various organization
identifiers using the existing chained-identifier mechanism.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the IG-defined `SearchParameter/fast-consent-patientId` (and
its companion `fast-consent-organizationId` per FHIR-53778) from the
Scalable Consent Management IG. Replace its role in the IG's narrative
("Searching for Consents" and "Consent Subscriptions" sections) with a
**SHALL**-level requirement that conformant servers support search for
Consent by patient identifier using the existing chained-identifier
form `Consent?patient:identifier=<system>|<value>` (and the analogous
`controller:identifier` / `manager:identifier` chains for
organizations). Update the CapabilityStatement(s) that currently
reference the removed SearchParameter to instead document the chained-
identifier requirement. This matches the existing Persuasive
resolution verbatim.

#### Justification

FHIR-I has consistently held that SearchParameters whose `expression`
crosses a resource boundary (e.g., reads `Consent.patient.resolve()
.identifier`) are not permitted, because the parameter cannot be
satisfied without dereferencing another resource. Honoring that
guidance — which the resolution already cites — keeps the IG aligned
with core FHIR rules and removes a divergence between the "Searching
for Consents" and "Consent Subscriptions" lists, which the submitter
correctly flags as inconsistent. It also reduces implementation
burden: servers already implement chained-identifier search on
references, so no new machinery is required.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the *capability* (search Consent by patient identifier) but
implement it differently from a strict deletion: define an
**OperationDefinition** (e.g., `$consent-find-by-patient-identifier`)
on Consent that takes `(system, value)` and returns the matching
Consents, and retire the SearchParameter. Continue to document the
`patient:identifier` chained-search recipe in narrative as the
preferred RESTful path, but provide the operation as a
profile-friendly alternative for servers that cannot rely on chained
search performance.

#### Justification

A subset of consent-management deployments (notably consent registries
and legal/audit servers) index by patient identifier as a primary key
and may not have efficient chained-identifier search. An operation
gives those implementers a stable, named integration point without
re-introducing the cross-resource SearchParameter that FHIR-I
prohibits. Trade-off vs. Disposition A: more spec surface area to
maintain, and operations are less universally implemented than search;
some implementers will see this as scope creep beyond what the
submitter asked for.

---

### Disposition C: Decline

#### Proposal

Keep `SearchParameter/fast-consent-patientId` (and its companions) as
defined and clarify in narrative why a custom parameter is necessary
for this IG's use cases. Reject the request on the grounds that the
parameter is already published in the 2026Jan ballot snapshot and that
removing it is a breaking change for early implementers.

#### Justification

This is the weakest of the three options. While stability for
2026Jan-era implementers is a real concern, FHIR-I has explicitly
indicated that cross-resource SearchParameters are not allowed, so
defending the parameter would put the IG out of conformance with core
FHIR rules. The "breaking change" cost is therefore better paid now,
during the same review cycle that produced FHIR-53778 and FHIR-53781,
than carried into a later normative edition.

---

### Recommendation

**Recommended disposition:** A

Disposition A matches the existing Persuasive resolution text exactly,
honors explicit FHIR-I guidance against cross-resource
SearchParameters, fixes the narrative inconsistency between
"Searching for Consents" and "Consent Subscriptions" that the
submitter highlighted, and is consistent with the parallel resolutions
on FHIR-53778 (organizationId) and FHIR-53781 (AuditEvent consent).
Disposition B preserves a niche capability but expands scope and
adds maintenance burden the submitter did not ask for; Disposition C
is hard to defend against the FHIR-I rule cited in the resolution.
The workgroup should adopt A and coordinate the edit so that all
three sibling tickets land in the same IG publication.
