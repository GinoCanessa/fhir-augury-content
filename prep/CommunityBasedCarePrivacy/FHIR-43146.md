## Summary

Lloyd McKenzie has filed FHIR-43146 against the **US PACIO Re-Assessment
Timepoints (FHIR)** IG asking the authors to tighten the
`Extension.context` for the IG's `clinicalImpression` extension
(`http://hl7.org/fhir/us/pacio-rt/StructureDefinition/clinicalImpression`).
Today the extension declares its context as `element`, which means it is
nominally allowed on any element in any resource. The reporter argues
that the extension's stated purpose — pointing to the `ClinicalImpression`
resource(s) that drive a re-assessment timepoint — only makes sense in a
narrow context (in practice, on `Encounter` and/or `Encounter.basedOn`),
and should be constrained accordingly.

This is one of a clearly-templated wave of "Please tighten your extension
contexts" tickets that Lloyd opened on 2023-11-20 against multiple IGs
(see FHIR-43141, FHIR-43147, FHIR-43148, FHIR-43149, FHIR-43150,
FHIR-43152, FHIR-43108) and is methodologically equivalent to FHIR-44032
("Fix the context for extensions with 'Element' context") which the
Biomedical Research & Regulation WG accepted as a Technical Correction
(Persuasive). There is also strongly related prior history specific to
this extension: **FHIR-35123** (Persuasive, Published) clarified the
purpose and structural placement of the same `clinicalImpression`
extension, and **FHIR-38651** (Triaged, Unresolved) — filed by Josh
Mandel after a Zulip discussion — proposes moving it from
`Encounter.basedOn` to the root of `Encounter`. Together these establish
that the *real* usage of the extension is on `Encounter` (currently
`Encounter.basedOn`), which is exactly the constraint FHIR-43146 is
asking the IG to express.

## Details

**Description:**

> The clinicalImpression extension is defined as being on 'element', but
> this extension doesn't make sense to be allowed on all elements in all

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat as a **Technical Correction** to the US PACIO Re-Assessment
Timepoints IG. In `HL7/fhir-pacio-rt`, edit the
`StructureDefinition/clinicalImpression` extension so that its
`Extension.context` is replaced with a single context entry of `type =
element` and `expression = "Encounter.basedOn"` — matching the only
place the extension is actually used in the IG's profiles and examples
(see FHIR-35123). Re-publish the IG once the next ballot/technical
correction window allows.

#### Justification

This is exactly what the reporter asked for and what the FHIR
methodology calls for: `Extension.context` should describe the actual
intended usage, not be left at the universal `element` default.
FHIR-35123 (Persuasive, Published) already nailed down that the
extension lives on `Encounter.basedOn` in this IG, so the "appropriate
context" Lloyd is asking for is unambiguous. FHIR-44032 demonstrates
that tightening an over-broad `Element` context is routinely accepted
by WGs as a Technical Correction. Cost is essentially zero for
implementers because no one could conformantly rely on the broader
context for a PACIO-specific extension anyway.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve FHIR-43146 jointly with FHIR-38651. Set
`Extension.context` to **two** entries — `Encounter` and
`Encounter.basedOn` — and update the IG narrative + examples so that the
extension may sit either at the root of the timepoint `Encounter`
(preferred forward-looking placement, per Josh Mandel's analysis) or
inside `Encounter.basedOn` (preserving any existing implementer usage).
Document the migration intent so a future major version can drop the
`Encounter.basedOn` context if no usage materialises.

#### Justification

The Zulip thread and FHIR-38651 surfaced a real semantic concern with
the current `basedOn` placement: `basedOn` is defined as
"the request this encounter satisfies", which doesn't fit
"assessment and impression at discharge from SNF". Constraining only to
`Encounter.basedOn` (Disposition A) freezes that semantic mismatch into
the spec. Allowing both contexts lets the WG fix the real underlying
issue without breaking any current implementations and avoids having
the WG re-open the same extension a second time when FHIR-38651 is
finally dispositioned. The cost is modest: a slightly broader context
list and an extra paragraph of guidance.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-43146 as **Not Persuasive** on the grounds that the IG is
on the deprecated "Community-Based Care and Privacy" work group's
docket, the IG is currently in a stable/published state, and the cost
of re-publishing for a non-functional context tightening exceeds the
benefit; instructive guidance about extension contexts can be added to
the IG's narrative without modifying the StructureDefinition.

#### Justification

The work group has been deprecated, the IG has not had recent activity
on this extension, and `Extension.context` is informational rather than
strictly enforced by most validators in IG-specific extensions. Any
implementer using a PACIO-named extension is implicitly working inside
the PACIO IG's domain, so the practical risk of misuse is low. If a
correction is made it triggers a re-publish that the deprecated WG may
not have the capacity to shepherd.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B addresses the literal request (tightening the context off
`element`) **and** absorbs the still-open FHIR-38651 by allowing the
extension on the root of `Encounter`, which the community discussion
has already agreed is the more semantically correct placement. Doing
this in one technical correction is cheaper than dispositioning
FHIR-43146 narrowly now (Disposition A) and then revisiting the same
StructureDefinition again when FHIR-38651 is processed; it also avoids
freezing the known `basedOn` semantic mismatch into the spec. Disposition
C is hard to defend given that an essentially-identical class of
ticket (FHIR-44032) has already been treated as a Persuasive Technical
Correction by another work group, and the change carries effectively
no implementer risk.
