## Summary

The ticket asks the Scalable Consent Management (FHIR) IG authors to justify
why recording a disclosure has been modeled as a custom FHIR operation
(`$record-disclosure`) instead of a plain RESTful `create` (HTTP POST) of an
`AuditEvent` resource. The reporter (John Moehrke) sees no benefit to wrapping
the act of recording an audit event in an operation when the underlying FHIR
RESTful API already provides the same capability with broader interoperability.
The ticket has been resolved **Persuasive**, with a resolution that the IG
will remove the `RecordDisclosure` operation in favor of plain RESTful
interactions and rewrite the surrounding text to describe how AuditEvents
should be recorded and retrieved.

The request is one of a closely related cluster of Persuasive tickets against
the same IG section, all converging on the same outcome:

- **[FHIR-53779](https://jira.hl7.org/browse/FHIR-53779)** (Abderrazek
  Boufahja) asks the same rationale question, and was resolved with the same
  intent: remove the operation and use REST for create/retrieve.
- **[FHIR-54692](https://jira.hl7.org/browse/FHIR-54692)** (Moehrke) argues
  more broadly that an IG should not require an actor to *create* AuditEvents
  in any specific FHIR shape (compare US-Core, which only constrains
  search/read), and that IHE-BALP + ITI-81 already cover the search side.
  Resolution: remove the operation, replace the section with FAST
  AuditEvent + searchability guidance.
- **[FHIR-55374](https://jira.hl7.org/browse/FHIR-55374)** (Joseph M. Lamy)
  questions which system is the disclosing system when consent instances are
  propagated across multiple systems; resolved Persuasive with Modification by
  removing the operation and rewriting the section.
- **[FHIR-54682](https://jira.hl7.org/browse/FHIR-54682)** (Moehrke) argues
  that if access-control decisioning is out of scope, then the disclosure
  audit derived from it cannot be required either; resolution splits the
  Consent Decision System into in-scope (search/retrieve audit events) and
  out-of-scope (decisioning + recording).
- **[FHIR-54649](https://jira.hl7.org/browse/FHIR-54649)** (Moehrke) flags
  inconsistent AuditEvent profiling across the three IG cases; the resolution
  explicitly defers to the outcome of FHIR-54648/54692/55374/53779 — the
  custom AuditEvent profile attached to the operation will be removed with
  the operation, and remaining cases will use IHE-BALP profiles.
- **[FHIR-54650](https://jira.hl7.org/browse/FHIR-54650)** (Moehrke) is a
  related but distinct concern: "disclosure" and "authorization decision" are
  conflated in the same section and should be split, each pointing at the
  appropriate IHE-BALP profile.

Together these tickets indicate a Workgroup direction to retire the
`RecordDisclosure` operation entirely and rewrite the IG's audit/disclosure
section around RESTful AuditEvent + IHE-BALP profiling.

### [FHIR-54692: AuditEvent Search/Read would not require Create](https://jira.hl7.org/browse/FHIR-54692)

Argues that an IG can constrain AuditEvent on the *search/read* side (à la
US-Core's pattern for Observation/Blood-Pressure) without mandating any
specific FHIR-shaped *create*, since most production audit logs are not
FHIR-native. Recommends leaning on IHE-BALP + ITI-81 rather than defining a
Record Disclosure operation. This ticket carries the strongest design
rationale for removing the operation and replacing it with searchability
requirements; it is the one FHIR-54648's resolution most directly follows.

Reporter: John Moehrke. Assignee: Unassigned.

### [FHIR-55374: Which system(s) do disclosures need to be recorded against?](https://jira.hl7.org/browse/FHIR-55374)

Highlights an unresolved scoping question in the current text: when Consent
instances are replicated across multiple systems, the IG never says which
system bears the disclosure-recording obligation. Resolved Persuasive with
Modification — the resolution removes the operation and rewrites the section
to describe REST-based recording/retrieval, sidestepping the ambiguity.

Reporter: Joseph M. Lamy. Assignee: Unassigned.

### [FHIR-54682: If access control decision is not in-scope, then neither can disclosure audit](https://jira.hl7.org/browse/FHIR-54682)

Points out that §OP12 (and similarly §OP8) currently mandates use of the
Record Disclosure Operation when a system accesses a Consent instance to
decide whether information may be released — but access-control
decisioning/enforcement is explicitly out of scope. The resolution splits the
Consent Decision System into out-of-scope (decisioning + audit recording) and
in-scope (querying audit events for CRUD against Consent, constrained to
IHE-BALP profiles). Directly affects how §OP12/§OP8 must be rewritten when
the operation is removed.

Reporter: John Moehrke. Commenters: David Pyke. Assignee: Unassigned.

### [FHIR-54649: AuditEvent profiling](https://jira.hl7.org/browse/FHIR-54649)

Notes the IG inconsistently profiles AuditEvent across three cases (only
case 3 — the Record Disclosure operation — defines a custom AuditEvent
profile). Resolution explicitly chains off FHIR-54648/54692/55374/53779:
removing the operation also removes its bespoke AuditEvent profile, and the
remaining cases will be aligned on IHE-BALP via the broader rewrite tracked
under FHIR-53755 ("Usage of ConsentAuditEvent").

Reporter: John Moehrke. Assignee: Unassigned.

### [FHIR-54650: Disclosure is not the same as Authorized Decision](https://jira.hl7.org/browse/FHIR-54650)

A sibling concern in the same section: the IG conflates "authorization
decision" (consulting a Consent to decide on access) with "disclosure" (the
actual sharing of information). Asks for separation and explicit references
to the two distinct IHE-BALP profiles
([3.5.7.6 Consent Authorized Decision Audit Message](https://profiles.ihe.net/ITI/BALP/content.html#3576-consent-authorized-decision-audit-message)
and
[3.5.7.7 Privacy Disclosure Audit Message](https://profiles.ihe.net/ITI/BALP/content.html#3577-privacy-disclosure-audit-message)).
The rewrite triggered by FHIR-54648 must respect this distinction.

Reporter: John Moehrke. Assignee: Unassigned.

## Proposed Dispositions

The Workgroup has already marked this ticket **Persuasive** and supplied a
resolution. The dispositions below frame the choices the editorial pass
still has to make when implementing that resolution in the IG narrative.

### Disposition A: Accept As Requested

#### Proposal

Implement the recorded resolution exactly as written:

1. **Remove** `OperationDefinition-record-disclosure` from the Scalable
   Consent Management IG (the artifact at
   `OperationDefinition-record-disclosure.html`), and remove the
   `$record-disclosure` reference from the CapabilityStatement and any
   conformance bullets in `technical.html`.
2. **Remove** the bespoke `AuditEvent` profile that was attached to the
   operation (per the chained resolution on FHIR-54649).
3. **Rewrite** `technical.html#record-disclosure-operation` to describe
   recording as a plain FHIR `create` (HTTP POST `AuditEvent`) and retrieval
   as a plain `search` (HTTP GET `AuditEvent?…`), with a normative reference
   to IHE-BALP profiles for the AuditEvent payload (Privacy Disclosure Audit
   Message; Consent Authorized Decision Audit Message — keeping the
   distinction asked for in FHIR-54650).
4. Update §OP8 and §OP12 (per FHIR-54682) so that recording is described as
   an out-of-scope obligation of the decisioning system, while in-scope
   conformance is reduced to *being able to return* IHE-BALP-profiled
   AuditEvents on search.
5. Add ApplyTo notes recording that this disposition jointly resolves
   FHIR-54648 / FHIR-53779 / FHIR-54692 / FHIR-55374 / FHIR-54682 / FHIR-54649.

#### Justification

This is the disposition already endorsed on the ticket and on every related
Persuasive ticket in the cluster. It aligns the IG with general FHIR REST
conventions (avoid bespoke operations when CRUD suffices) and with the
US-Core precedent (constrain AuditEvent on the *retrieve* side, not the
*create* side) explicitly cited in FHIR-54692. Reusing IHE-BALP profiles
removes a divergent custom profile and keeps the IG in step with existing
audit ecosystems (ATNA / ITI-81). The chained resolution language on
FHIR-54649 makes clear the editors expect this exact bundle of changes.

---

### Disposition B: Alternative Approach — Deprecate, don't delete

#### Proposal

Keep `OperationDefinition-record-disclosure` present in the IG but mark it
**deprecated** (status `retired` on the OperationDefinition, plus a
narrative banner pointing readers at the new RESTful recording pattern), and
in parallel add the RESTful + IHE-BALP guidance prescribed by Disposition A.
Remove the SHALL-level requirement to use `$record-disclosure` from §OP8
and §OP12 immediately, but leave the artifact discoverable for one ballot
cycle. Schedule outright removal for the next ballot.

#### Justification

If any early implementer or reference implementation has already wired up
`$record-disclosure`, an immediate hard-delete is a breaking change against a
SHALL-level conformance rule. A deprecate-then-remove path gives implementers
a documented migration window while still accomplishing the intent of the
ticket (no longer requiring the operation, removing the bespoke profile from
*new* conformance). The cost is one ballot cycle of carrying a retired
artifact and the editorial overhead of two narrative passes instead of one.
This disposition would weaken the chain note on FHIR-54649 (the bespoke
AuditEvent profile would have to linger alongside the retired operation), so
the editors should be willing to accept that.

---

### Disposition C: Decline

#### Proposal

Keep the `RecordDisclosure` operation as a normative artifact and the SHALL
requirement in §OP12 / §OP8 unchanged. Add a non-normative note to
`technical.html#record-disclosure-operation` explaining the design choice
(e.g., that the operation exists to bundle policy context with the
AuditEvent payload, or to give servers a single conformance target).

#### Justification

Defensible only if the Workgroup can articulate a concrete capability the
operation provides over a plain `POST AuditEvent` — for example, an input
parameter that carries decisioning context the server cannot reconstruct
from the AuditEvent payload alone, or a server-side side-effect (notification,
ledger entry) that a bare REST create does not trigger. None of the related
tickets surface such a capability, and the existing resolution text on this
ticket and on FHIR-53779/54692/55374 explicitly endorses removal, so this
disposition would require the Workgroup to overturn its own already-recorded
Persuasive resolution and answer the rationale question the reporter posed.

---

### Recommendation

**Recommended disposition:** A

The Workgroup has already concluded — across five sibling Persuasive tickets
from three different reporters (Moehrke, Boufahja, Lamy) — that the operation
adds no capability over a RESTful `create` and that constraining only the
*retrieve* side (IHE-BALP via ITI-81, US-Core-style) is the right fit for an
IG that does not own audit storage. Disposition A simply executes that
consensus and lets a single editorial pass close out the entire cluster
(54648 / 53779 / 54692 / 55374 / 54682, with knock-on effects on 54649 and
54650). Disposition B (deprecate-then-delete) is only worth the extra
ballot-cycle overhead if implementer evidence of in-flight use of
`$record-disclosure` is presented during review; in the absence of that
evidence, the IG is still pre-publication enough that immediate removal is
clean. Disposition C contradicts the Workgroup's own recorded resolution and
would need to overcome the rationale arguments in FHIR-54692 — which it
currently has no answer for.
