## Summary

FHIR-46334 is a small, scoped editorial change request against the
`CoverageEligibilityRequest` resource page in FHIR Core. The reporter
asks that the resource's "Boundaries and Relationships" section be
extended with a paragraph that draws an explicit line between the
authorization captured in `CoverageEligibilityRequest` and the
fulfillment activity tracked in `Task`, and that it link out to the
"Fulfillment/Execution" section of the Request pattern for the rationale.

The ticket is a downstream child of [FHIR-45872](https://jira.hl7.org/browse/FHIR-45872)
("Need something on the various 'Request' resources that explain the
use of Task"), which was resolved Persuasive with Modification by FHIR
Infrastructure and is now Applied. That parent ticket extended the
Request pattern's Fulfillment/Execution section with a multi-bullet
rationale for keeping authorization and fulfillment status in separate
resources, and explicitly stated that change requests would be raised
against each individual Request resource asking them to add the
following sentence (templated as `[MyRequest]`) to their Boundaries and
Relationships section:

> "The `[MyRequest]` resource represents an authorization for a service
> to be provided. Details about the fulfillment of the authorization
> are handled by the `[Task]` resource. For further information about
> this separation of responsibilities, refer to the
> `[Fulfillment/Execution]` section of the Request pattern."

FHIR-46334 is exactly that downstream change request, scoped to
`CoverageEligibilityRequest`, with a note from the reporter that the
text may need slight rewording for this resource (since
`CoverageEligibilityRequest` authorizes an *eligibility check*, not a
clinical service, and its fulfillment is `CoverageEligibilityResponse`
rather than the kind of service-delivery `Task` chain that the parent
text presupposes).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the change as requested and extend the
`CoverageEligibilityRequest` resource's "Boundaries and Relationships"
section with the templated sentence from FHIR-45872, with only the
minor wording tweaks needed to make it accurate for an eligibility
check. Suggested text:

> The `CoverageEligibilityRequest` resource represents an authorization
> for an eligibility check to be performed. Details about the
> fulfillment of that request — including in-progress, rejected, and
> retried attempts — are handled by the `Task` resource (with
> `CoverageEligibilityResponse` carrying the eventual eligibility
> determination payload). For further information about this
> separation of responsibilities, refer to the
> [Fulfillment/Execution](workflow.html#fulfillment) section of the
> Request pattern.

The edit lives on the `CoverageEligibilityRequest` resource page in the
FHIR Core specification's "Boundaries and Relationships" narrative
(the same page that already cross-references
`CoverageEligibilityResponse`, `Coverage`, and `Claim`). No changes to
the StructureDefinition, search parameters, examples, invariants, or
any other Financial Mgmt artifact are required.

#### Justification

This is the disposition the parent ticket FHIR-45872 explicitly
contemplated when it noted "We may want some re-wording for some
requests (CarePlan, Appointment, Claim, etc.)." Adopting it keeps
`CoverageEligibilityRequest` in line with every other Request resource
that will receive the same paragraph, gives readers a one-click path
into the Request-pattern rationale that already shipped, and removes a
known confusion point — implementers who reach for an extension on
`CoverageEligibilityRequest.status` to track payer-side processing
state instead of using `Task` (and ultimately
`CoverageEligibilityResponse`). The wording tweaks above respect the
fact that the "fulfillment" of an eligibility request is not service
delivery but the production of a `CoverageEligibilityResponse`, while
still pointing at the same shared Fulfillment/Execution prose. As a
narrative-only change with no schema impact it is FMM-safe and
backward compatible.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent but consolidate, rather than duplicate, the
boilerplate. Instead of pasting a near-identical paragraph onto each
Request resource page, add a **single** short sentence plus link on
`CoverageEligibilityRequest`'s Boundaries and Relationships section,
e.g.:

> Like other Request resources, `CoverageEligibilityRequest` captures
> an authorization; the *fulfillment* of that authorization (including
> retries, partial fulfillment, and per-attempt status) is tracked
> separately on `Task`. See the Request pattern's
> [Fulfillment/Execution](workflow.html#fulfillment) section for the
> rationale, and `CoverageEligibilityResponse` for the eligibility
> determination payload returned to the requester.

Pair this with a Request-pattern–level table (or a single shared
include) that lists, for each Request resource, the corresponding
fulfillment artifact (e.g., `MedicationRequest` →
`MedicationDispense`/`MedicationAdministration`,
`CoverageEligibilityRequest` → `CoverageEligibilityResponse`, etc.),
so the per-resource Boundaries section stays compact and the
authoritative explanation lives in exactly one place.

#### Justification

The literal parent-ticket boilerplate is written for service-delivery
Requests where `Task` is the fulfillment carrier. For
`CoverageEligibilityRequest`, the *primary* fulfillment artifact in
practice is `CoverageEligibilityResponse`; `Task` is the workflow
wrapper around the request/response exchange but is rarely the thing
implementers reach for first. Pasting the unmodified template risks
overstating Task's role for this specific resource and understating
the role of `CoverageEligibilityResponse`, which could confuse Da
Vinci / payer implementers more than it helps. Centralising the
"authorization vs fulfillment" explanation in the Request pattern and
keeping per-resource pages to a one-liner also reduces the maintenance
cost across the ~14 Request-pattern resources and avoids drift between
their Boundaries sections. Trade-off vs Disposition A: it does not
exactly mirror the wording the other Request resources will get, which
slightly hurts cross-resource consistency.

---

### Disposition C: Decline

#### Proposal

Decline the change against `CoverageEligibilityRequest` on the grounds
that the fulfillment story for an eligibility check is already
adequately covered by the existing Boundaries and Relationships
discussion of `CoverageEligibilityResponse` (and, more broadly, by
the Financial module's narrative about eligibility request/response
exchanges and the `Claim`/`ClaimResponse` parallel). Note in the
resolution that the generic Request-pattern guidance from FHIR-45872
already exists for readers who want it, and that no additional
per-resource paragraph is needed here.

#### Justification

Unlike `MedicationRequest` or `ServiceRequest` — where there is a
real and recurring problem of implementers using extensions to track
fulfillment status on the Request itself — there is little evidence in
the trackers or on Zulip (no related Zulip threads were found for this
ticket) of payer/provider implementers misusing
`CoverageEligibilityRequest.status` to track payer-side processing
state. The eligibility request/response pair, plus optional Task
wrapping, is already a well-understood pattern in Da Vinci PAS, CRD,
and DTR. Adding the standard Request-pattern paragraph might therefore
add words without changing implementer behaviour, and risks confusing
readers by foregrounding `Task` over the much more commonly used
`CoverageEligibilityResponse`.

---

### Recommendation

**Recommended disposition:** A (with the wording tweaks shown).

The whole point of FHIR-45872 was cross-resource consistency: every
Request resource's Boundaries and Relationships section gets the same
paragraph, pointing back to one shared Fulfillment/Execution rationale.
Declining for `CoverageEligibilityRequest` (Disposition C) breaks that
pattern with no concrete harm avoided, and Disposition B's "let's
redesign the boilerplate" path is a much larger piece of work that
properly belongs on the parent ticket in FHIR Infrastructure rather
than as a side effect of a single Financial Mgmt narrative tweak.
Disposition A delivers exactly what the parent ticket asked the
downstream work groups to deliver, while honouring the
"updating text if needed" caveat the reporter explicitly invited:
the tweak names `CoverageEligibilityResponse` as the eligibility
payload carrier so readers are not left thinking `Task` is the only
fulfillment artifact. It is a low-risk, narrative-only edit on a
single resource page with no schema, FMM, or backwards-compatibility
consequences, and it can be applied immediately without further
cross-work-group coordination.
