## Summary

The PDex IG ships several Provider example resources (Organization and
Practitioner) whose `Identifier.type` is coded with
`http://terminology.hl7.org/CodeSystem/v2-0203#NPI`. Because
`Identifier.type` carries an extensible binding to the FHIR R4
`identifier-type` ValueSet, and that value set already contains the code
`PRN` ("Provider Number") which covers NPI, using the v2-0203 `NPI` code
alone is non-conformant. The reporter requests that the affected examples
either add the `PRN` identifier type, or replace the v2-0203 `NPI` code
with `PRN`.

The ticket is the PDex follow-on to
[FHIR-48320](https://jira.hl7.org/browse/FHIR-48320) (CARIN BB), which
raised the same conformance issue against the C4BB profiles. CARIN BB
required profile (StructureDefinition + invariant) changes; PDex only
carries the issue in **examples**, so it is filed as a Technical
Correction. The resolution (already pre-applied) replaces
`$IdentifierTypeCS#NPI` with `$IdentifierTypeCS#PRN` in 7 instances in
`input/fsh/SupportingExampleProfiles.fsh` (4 payer Organizations, 2
provider Organizations, and 1 Practitioner). The `$IdentifierTypeCS`
alias (= `http://terminology.hl7.org/CodeSystem/v2-0203`) is kept; only
the code is changed.

## Details

**Description:**

There are several examples of Provider resources
(Organization/Practitioner) where `identifier.type =
http://terminology.hl7.org/CodeSystem/v2-0203#NPI`.

However, `identifier.type` has an extensible binding to
`https://hl7.org/fhir/R4/valueset-identifier-type.html`, which has the
code `PRN`, which is what an NPI is. So, the examples should either add
this identifier type `PRN`, and/or remove the v2-0203 `NPI` code.

See original ticket submitted against CARIN BB (CARIN BB had this in the
profile and will require more changes; PDex only has these in examples
and therefore is a TC):
[FHIR-48320](https://jira.hl7.org/browse/FHIR-48320) — NPI isn't a valid
identifier type.

**Resolution Description (Pre-Applied):**

- 7 instances in `input/fsh/SupportingExampleProfiles.fsh` — all
  Organization (4 payer, 2 provider) and Practitioner examples changed
  from `$IdentifierTypeCS#NPI` → `$IdentifierTypeCS#PRN`.
- `PRN` ("Provider Number") is the correct code in the FHIR R4
  identifier-type ValueSet that covers NPIs; `NPI` does not exist in
  that extensible binding set.
- The `$IdentifierTypeCS` alias
  (`http://terminology.hl7.org/CodeSystem/v2-0203`) itself is correct —
  only the code needed fixing.

**Comments:**

No comments are recorded on FHIR-50458.

## Linked Jira Tickets

### [FHIR-48320: NPI isn't a valid identifier type](https://jira.hl7.org/browse/FHIR-48320)

The original CARIN Blue Button ticket from Lloyd McKenzie (Sept 2024)
that raised the conformance problem with using `v2-0203#NPI` for
`Identifier.type`. It argues that (1) the extensible binding on
`Identifier.type` requires a code from the FHIR identifier-type ValueSet
when one applies, and `PRN` does, and (2) `NPI` is really an identifier
*system*, not type — which is already conveyed by
`Identifier.system = http://hl7.org/fhir/sid/us-npi`. CARIN BB resolved
"Persuasive with Modification" by removing the fixed `^patternIdentifier.type`
NPI binding from C4BBPractitioner / C4BBOrganization and rewording the
"must include an NPI or Tax ID" invariant to be system-based. FHIR-50458
is the PDex companion: same root cause, but PDex only manifests it in
examples, so the fix is far smaller in scope.

Users involved: Lloyd McKenzie (mckenzie, reporter), Corey Spears
(corey_spears), Mark Roberts (mark_roberts), Pat Knapp (pknapp), Clem
McDonald (cmc), JIRAUSER22800 (Australia commenter).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the pre-applied technical correction as-is: in
`HL7/davinci-epdx` at `input/fsh/SupportingExampleProfiles.fsh`,
replace all 7 occurrences of `$IdentifierTypeCS#NPI` with
`$IdentifierTypeCS#PRN` across the 4 payer Organization examples, the 2
provider Organization examples, and the 1 Practitioner example. Leave
the `$IdentifierTypeCS` alias
(= `http://terminology.hl7.org/CodeSystem/v2-0203`) and each example's
`Identifier.system = http://hl7.org/fhir/sid/us-npi` unchanged. Update
`input/pagecontent/changehistory.md` to reference FHIR-50458 (already
done) and regenerate `output/` and `template/` artifacts via the
standard sushi + IG-publisher build (`./_genonce.sh`). Target the
current pre/post-publish branch per the repo's branch convention.

#### Justification

`PRN` ("Provider Number") is the only code in the FHIR R4
`identifier-type` ValueSet that semantically covers an NPI; the
extensible binding on `Identifier.type` therefore requires that PRN (or
another applicable code from the value set) be used when a provider
identifier type is asserted. The v2-0203 `NPI` code is **not** in that
extensible value set, so the current examples are non-conformant —
exactly the conformance argument that FHIR-48320 established and that
the CARIN BB ("Persuasive with Modification") and PDex ("Persuasive")
resolutions both ratify. The change is scoped purely to examples
(no profile / invariant / ValueSet / dependency change), so blast
radius is minimal: it touches only the flat `input/fsh/` authoring
root, and the briefing flags no gotchas that would block it. The fix
keeps `Identifier.system = http://hl7.org/fhir/sid/us-npi`, which is
the canonical and unambiguous way to convey "this identifier is an
NPI", aligning with Lloyd McKenzie's argument in FHIR-48320.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of replacing `NPI` with `PRN`, **add** a second
`Identifier.type.coding` to each affected example so that the type
slice carries both `PRN` (from the FHIR `identifier-type` ValueSet,
satisfying the extensible binding) and the existing v2-0203 `NPI`
coding (preserved for downstream readers that have come to expect it,
including non-US jurisdictions such as Australia where v2-0203 `NPI`
is in active use per the FHIR-48320 discussion). Apply this only in
`input/fsh/SupportingExampleProfiles.fsh`; no profile or alias change
is required. Optionally add a one-line note in
`input/pagecontent/changehistory.md` explaining the dual-coding choice.

#### Justification

This still resolves the conformance defect (because `PRN` from the
extensible value set is now present), while preserving signal that
existing readers and non-US implementers may key off of.
FHIR-48320's discussion explicitly noted that v2-0203 `NPI` is used by
Australia's national Healthcare Provider Identifier profile, and Lloyd
McKenzie's reply acknowledged that sending `NPI` in addition to `PRN`
is permissible. Trade-offs vs. Disposition A: examples become slightly
more verbose, and PDex examples diverge stylistically from the CARIN BB
resolution (which dropped the type binding entirely). Backward
compatibility for any consumer that was matching on
`Identifier.type.coding` containing v2-0203 `NPI` is preserved, at the
cost of suggesting to PDex implementers that dual-coding is the
recommended pattern when the IG itself doesn't actually require it.

---

### Disposition C: Decline

#### Proposal

Decline the technical correction on the grounds that the affected
artifacts are **examples**, not normative profiles or invariants, and
the canonical NPI signal in each example
(`Identifier.system = http://hl7.org/fhir/sid/us-npi`) is already
unambiguous. Leave the v2-0203 `NPI` coding in `Identifier.type` in
`input/fsh/SupportingExampleProfiles.fsh` and rely on PDex consumers
to interpret `Identifier.system` for the authoritative NPI assertion.

#### Justification

This disposition is the weakest of the three. It can be defended on
narrow grounds — that examples carry no conformance weight in the
publisher's validation report, and that real-world consumers tend to
key off `Identifier.system = http://hl7.org/fhir/sid/us-npi` rather
than `Identifier.type` — but it leaves the IG publishing examples that
the FHIR R4 extensible binding rule deems non-conformant, which is the
exact problem FHIR-48320 surfaced and which the CARIN BB workgroup
already accepted as worth fixing. Declining would also be
inconsistent with the parent Da Vinci/CARIN ruling and with the
pre-applied commit already on the repository.

---

### Recommendation

**Recommended disposition:** A

This is a low-risk, example-only Technical Correction that is already
pre-applied (commit `0adecac8` on `HL7/davinci-epdx`) and whose
resolution has been recorded as Persuasive. It is consistent with the
companion CARIN BB ruling on FHIR-48320 (Persuasive with Modification),
respects the FHIR R4 extensible binding on `Identifier.type`, and
touches only `input/fsh/SupportingExampleProfiles.fsh` (plus the
already-updated change-history page) — no profile, ValueSet, alias,
dependency pin, or cross-IG coordination is required. Disposition B's
dual-coding compromise is technically defensible and accommodates the
Australian use case raised on FHIR-48320, but doing so in PDex
*examples* (which exist primarily to teach the canonical PDex pattern)
risks signalling to US implementers that dual-coding is expected when
PDex's normative profiles do not require it. Disposition C should be
declined because the reporter, the parent CARIN ruling, and the FHIR
core extensible-binding rule are aligned on the conformance defect
being real. The workgroup should ratify the pre-applied change as-is.
