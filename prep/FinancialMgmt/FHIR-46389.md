## Summary

The ticket asks the CARIN Blue Button (C4BB) IG to retire its locally-defined
`AHANUBCPresentOnAdmission` CodeSystem (with the external URL
`https://www.nubc.org/CodeSystem/PresentOnAdmissionIndicator`) and instead
adopt the Present On Admission Indicator CodeSystem now hosted in
Terminology.HL7.org (THO) at
`https://terminology.hl7.org/5.5.0/CodeSystem-presentOnAdmission.html`,
which uses the canonical
`https://www.cms.gov/Medicare/Medicare-Fee-for-Service-Payment/HospitalAcqCond/Coding`.
The reporter argues that, now that THO carries an authoritative version of
this code system, the IG-local stub should be removed and bindings should
point to the THO-hosted resource.

This request is a near-duplicate of the older Financial Management ticket
[FHIR-38278](https://jira.hl7.org/browse/FHIR-38278) ("CARIN BB Present on
Admission Code System URL different than THO"), which was resolved
**Persuasive** and Published by the workgroup on 2024-11-21 with a concrete
action: (1) remove the NUBC Present On Admission Indicator CodeSystem stub
from C4BB; (2) repoint the NUBC Present On Admission Indicator Codes
ValueSet from
`https://www.nubc.org/CodeSystem/PresentOnAdmissionIndicator` to
`https://www.cms.gov/Medicare/Medicare-Fee-for-Service-Payment/HospitalAcqCond/Coding`;
and (3) rename the ValueSet to "CMS Present On Admission Indicator Codes
Value Set." Related ticket [FHIR-35404](https://jira.hl7.org/browse/FHIR-35404)
(2022) raised the same canonical-URL anchoring issue at the ValueSet level,
and [FHIR-48340](https://jira.hl7.org/browse/FHIR-48340) (2024) bundles the
POA value set submission into a broader request to move several C4BB value
sets to THO. The historical Zulip thread (#terminology > "nubc present on
admission code system v2-0895") confirms that the v2-0895 stub in THO is a
distinct artifact from the NUBC POA code system Marc Duteau migrated into
THO and is not the URL CARIN BB should adopt.

## Details

**Description:**
The Present on Admission indicator is currently an IG-defined CodeSystem
resource (with an external URL):
`https://hl7.org/fhir/us/carin-bb/CodeSystem-AHANUBCPresentOnAdmission.html`
(external URL: `https://www.nubc.org/CodeSystem/PresentOnAdmissionIndicator`).

This Code System has been added to THO with a different URL. In THO it can
be found at `https://terminology.hl7.org/5.5.0/CodeSystem-presentOnAdmission.html`
with the URL of
`https://www.cms.gov/Medicare/Medicare-Fee-for-Service-Payment/HospitalAcqCond/Coding`.

Remove the CARIN BB CodeSystem and adopt the THO-identified CodeSystem URL.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the same change agreed for FHIR-38278 (Persuasive) directly to the
CARIN Blue Button IG:

1. **Retire** the IG-local `CodeSystem/AHANUBCPresentOnAdmission` stub in
   C4BB (canonical `https://www.nubc.org/CodeSystem/PresentOnAdmissionIndicator`).
2. **Rebind** the existing `ValueSet/AHANUBCPresentOnAdmission` so its
   single include points at the THO CodeSystem
   `https://www.cms.gov/Medicare/Medicare-Fee-for-Service-Payment/HospitalAcqCond/Coding`
   (THO `presentOnAdmission`).
3. **Rename** the value set to "CMS Present On Admission Indicator Codes
   Value Set" (per the FHIR-38278 resolution) and update its display label
   in any narrative tables.
4. **Update bindings** on `ExplanationOfBenefit` profiles (and any other
   C4BB profile referencing the old VS canonical) to keep using the
   ValueSet canonical URL (no element-binding change is required if the
   ValueSet URL is preserved; a new VS canonical would require a profile
   touch).
5. **Add a migration note** to the IG's "Changes since last version"
   section calling out the CodeSystem URL change for implementers, and
   leave the old CodeSystem instance in the IG's `history.json` with a
   "retired" status entry so existing Coding instances remain
   discoverable.

#### Justification

This is the disposition the workgroup already accepted on the predecessor
ticket FHIR-38278 (Persuasive, Published). Implementing it on FHIR-46389
brings this ticket into alignment with existing precedent, reduces
duplicate IG-local code systems, and matches the consensus from the
#terminology Zulip thread that the THO `presentOnAdmission` CodeSystem
(content-bearing, CMS-anchored) is the canonical home. It also unblocks
related tickets FHIR-35404 (URL anchoring) and the POA bullet of
FHIR-48340.

---

### Disposition B: Alternative Approach — Close as Duplicate of FHIR-38278

#### Proposal

Close FHIR-46389 with resolution **Duplicate** of
[FHIR-38278](https://jira.hl7.org/browse/FHIR-38278), rather than carrying
two parallel tickets describing the same change. Add a comment on
FHIR-46389 cross-linking to FHIR-38278's Resolution Description so the
reporter can track the change there. Optionally, also link FHIR-35404 and
the POA bullet of FHIR-48340 to FHIR-38278 so all four issues converge on
a single implementation ticket.

#### Justification

FHIR-38278 covers the identical request (same reporter, same artifact,
same target THO URL) and already carries a workgroup-approved resolution
that is more specific than FHIR-46389's request (it also addresses the
ValueSet rename and the URL change inside the include). Maintaining two
open tickets for the same change risks divergent implementations and
makes it harder for the workgroup to track ballot impact. Closing as
Duplicate preserves the audit trail without obligating the workgroup to
re-deliberate. Trade-off vs. Disposition A: this resolution does not by
itself produce the IG change — it relies on FHIR-38278 already being
substantively complete (Status: Published).

---

### Disposition C: Decline

#### Proposal

Mark FHIR-46389 as **Not Persuasive** with the rationale that the change
is already adopted under FHIR-38278 and no separate action is needed.
Direct the reporter to FHIR-38278 (and to the next CARIN BB STU
publication that carries the change) for the implementation status.

#### Justification

If the workgroup considers FHIR-38278 fully resolved and published, then
FHIR-46389 is not persuasive on its own merits — there is nothing left to
change as a result of accepting it. This is a defensible "no-op"
disposition when the issue has been substantively addressed elsewhere.
The downside compared to Disposition B is that "Not Persuasive" is
slightly misleading (the request itself is persuasive — it just arrived
late) and may confuse the reporter, who will see his own previously-
accepted ask declined here.

---

### Recommendation

**Recommended disposition:** B (close as Duplicate of FHIR-38278).

The substance of FHIR-46389 is identical to FHIR-38278, which the Financial
Management work group already resolved **Persuasive** and Published with a
specific implementation plan (retire the stub CodeSystem, repoint and
rename the ValueSet to the CMS POA URL). Re-deliberating the same change
under a second key adds workgroup overhead with no additional outcome, and
"Not Persuasive" (Disposition C) misrepresents the merits — the request
*is* persuasive, it has simply already been accepted. Disposition A is
only appropriate if FHIR-38278's resolution has **not** in fact been
applied to the IG sources; the ticket-prep tooling does not have a
`HL7/carin-bb` repo briefing to confirm the actual code state, so the
workgroup should verify the C4BB sources during the call before defaulting
to B. If the change has not yet been merged into the C4BB IG, fall back
to Disposition A and treat both tickets as one work item against
FHIR-38278.
