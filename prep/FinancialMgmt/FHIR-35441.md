## Summary

The ticket requests that the canonical (defining) URL for
`ValueSet-NCPDPBrandGenericIndicator` in the US CARIN Blue Button IG be
re-anchored under `terminology.hl7.org` (THO) rather than under
`hl7.org/fhir`. The reporter (Joan Harper, comment imported by Ron G.
Parker) further notes that a UTG ticket should be opened to register the
ValueSet in THO. The underlying principle stated in the parent
ballot-comment guidance is that only artifacts with a required binding
to a `code` data type should remain anchored under `hl7.fhir.org`; all
others (including this NCPDP-backed ValueSet) belong in THO.

This ticket is one of ~25 sibling tickets (FHIR-35390 through FHIR-35447,
plus FHIR-36486/36487) created against CARIN BB STU2 to apply the same
URL-anchoring rule across the IG's CodeSystem and ValueSet artifacts.
The Financial Management WG handled them as Block-Vote-6 on 2022-07-26
and originally dispositioned this item as **Considered for Future Use**,
with the resolution comment: "This is a ValueSet including all codes in
a complete external Code system. There is a ticket to add 'all codes'
ValueSets ([UP-316](https://jira.hl7.org/browse/UP-316)). When this is
complete, the ValueSet in this IG can be removed and the bindings point
to the THO ValueSet." On 2022-11-28 lynn_laakso reverted that resolution
and re-opened the ticket post-STU2 publication, returning it to
**Triaged** status and putting it back in front of the work group. The
ticket's fate is therefore now tied to the status of UP-316 and to
whether THO has a registered "all-codes" ValueSet for the NCPDP Brand
Generic Indicator code system.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the ValueSet `NCPDPBrandGenericIndicator` so its canonical
URL becomes
`http://terminology.hl7.org/ValueSet/NCPDPBrandGenericIndicator` (or the
appropriate THO-issued URL) instead of the current
`http://hl7.org/fhir/us/carin-bb/ValueSet/NCPDPBrandGenericIndicator`.
Concretely:

1. File a UTG ticket against THO to publish an "all codes from NCPDP
   Brand/Generic Indicator" ValueSet (i.e., the dependency that UP-316
   was tracking generically) and obtain the canonical URL.
2. In the next CARIN BB normative/STU update, remove the local
   `ValueSet-NCPDPBrandGenericIndicator` definition.
3. Repoint every binding currently referencing the carin-bb canonical
   (e.g., the EOB Pharmacy slice referencing brand/generic indicator)
   to the new THO canonical.
4. Add a deprecation/redirect note in the IG's terminology page
   describing the move.

#### Justification

This is precisely the rule the IG editors agreed to in the original
ticket-class rationale: ValueSets that are simply "all codes from an
external CodeSystem" do not belong under `hl7.org/fhir/us/carin-bb`;
they belong in THO so they can be reused by other IGs (notably the
sibling pharmacy-using IGs flagged in the WG's harmonization
discussions). NCPDP Brand/Generic Indicator is a complete external code
system with a small, stable code set, which makes it the textbook case
for a THO "all-codes" ValueSet. Re-opening this ticket post-STU2
publication (lynn_laakso's 2022-11-28 action) signals that the WG no
longer wants to wait on the generic UP-316 work and is willing to move
ahead per-ValueSet.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the canonical URL anchored in `hl7.org/fhir/us/carin-bb` for the
current STU edition, but mark the ValueSet as **deprecated** with a
`replacedBy` extension pointing to the eventual THO canonical, and add
guidance in the IG narrative that implementers SHOULD use the THO
ValueSet once it is published. Schedule the actual canonical move and
binding repoint for the next major (R5/STU3) revision once UP-316 (or
the targeted UTG ticket from Disposition A, step 1) has actually
landed and been published in a THO release.

#### Justification

CARIN BB STU2 is already published. A canonical URL change is a
breaking change for any consumer that has hard-coded the binding URL
(profile validators, terminology-server caches, payer/provider
implementations). Doing the move in a transition release — with the
deprecation marker giving implementers a clear migration target —
preserves backward compatibility while still recording the WG's
intent. It also avoids creating a hard dependency on a THO publishing
cadence that the FM WG does not control: if the THO-side ValueSet
publishes late, the IG isn't blocked.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive**, mirroring the resolution wording used on
the sibling tickets where the WG chose to keep the ValueSet local
(e.g., FHIR-35403, FHIR-35413, FHIR-35414, FHIR-35415, FHIR-35439,
FHIR-35446): "ValueSet is not complex, fairly specific to this IG, and
does not warrant the maintenance of moving to THO."

#### Justification

NCPDPBrandGenericIndicator is used in exactly one place in CARIN BB
(EOB Pharmacy supportingInfo / brand/generic indicator slice). Moving
it to THO adds cross-org maintenance overhead (UTG ticketing,
versioning, release coordination) for a ValueSet with no demonstrated
reuse outside this IG. The WG has already declined to move several
similarly-scoped ValueSets in the same Block-Vote-6 batch on exactly
this rationale, so declining here is consistent with that pattern.
Implementers who need a THO-anchored equivalent can still bind to the
underlying NCPDP code system directly without the IG owning a redirect.

---

### Recommendation

**Recommended disposition:** B

The ticket was re-opened by lynn_laakso specifically because STU2 had
already published, which means the constraints around breaking
changes are now real, not theoretical. Disposition A is the correct
end state — the underlying principle that "all-codes" ValueSets over
external code systems belong in THO is sound, and the WG already
agreed with it 13-0-0 — but executing the move in the current
publication line breaks every consumer that has resolved the
canonical. Disposition C is inconsistent with the WG's own stated
rule and with the reporter's intent, and would re-litigate a question
the WG already conceded on principle. Disposition B captures the
intent (deprecate, point at THO, commit to the move), avoids the
breaking change in the published edition, and decouples the IG's
release schedule from the THO publishing cadence — which is precisely
the friction that caused this ticket to sit through STU2 in the first
place. The work-item that should accompany the disposition is the
THO/UTG ticket called out in the original description; without it,
even Disposition A cannot actually be executed.
