## Summary

Becky Gradl requests that the `Invoice.status` value set (`InvoiceStatus`) be
expanded beyond its current "happy path" lifecycle codes to better represent
the historical and accounting reality of invoices over time. The submitter
proposes adding four new codes — **Refunded in Full**, **Refunded in Part**,
**Suspended**, and **Written-Off** — so that an `Invoice` instance can carry
states such as full/partial refund, suspension of further payments, and
write-off to collections. Her use case is a U.S. Department of Veterans
Affairs–style scenario in which `Invoice` is used to present veterans with the
charges and payments rolled up on a monthly statement, and the source systems
expose lifecycle states that today have to be captured via the third-party
`alternate-codes` extension on `Invoice.status`.

The ticket is currently in **Triaged** status with active discussion. Mark
Scrimshire (`ekivemark`) has suggested a structural alternative: keep
`Invoice.status` aligned with the standard "common status" pattern (instance
maturity) and introduce a new dedicated element/value set for the accounting
lifecycle. Paul Knapp (`pknapp`) reinforced this distinction, observing that
the requested codes are really an *accounting / settlement* status (e.g.,
"paid in full", "partial payment", "unpaid", and the requester's additions),
not the *record* lifecycle that `.status` is defined to convey, and asked
whether the request would also be met by a new extensible value set on a new
element. He further raised whether the Invoice should additionally carry an
outstanding balance, last-payment date, or a payment-tracking sub-list (date,
amount, type code: payment / refund / adjustment). The reporter has
responded that she has no preference between (a) adding an "accounting
status" / "settlement status" element, or (b) changing the existing
`Invoice.status` value set — provided the lifecycle states can be expressed
natively. She also notes that balance information today is being shoe-horned
into `Invoice.totalPriceComponent.code.text`, and that last-payment date is
derivable via `PaymentReconciliation.allocation.target` in R5.

There are no Jira, Zulip, or GitHub cross-references attached to this ticket
in Augury; the only cross-references are FHIR artifact mentions
(`Invoice.status`, `PaymentReconciliation.allocation.target`).

## Details

**Description:**

The status values for Invoice seem to be "happy path" and from the present.
In looking at the data from a historical perspective and with what could
happen to an invoice, there needs to be more status values to represent an
invoice lifecycle. Some suggested values to add to `Invoice.status` (value
set `InvoiceStatus`):

| Display | Definition |
|-|-|
| Refunded in Full | The total invoice has been refunded |
| Refunded in Part | One or more line item has been refunded or credited to the bill |
| Suspended | The invoice can no longer have payments applied |
| Written-Off | The invoice has not been paid and has been sent to collections |

**Comments:**

- **ekivemark (Mark Scrimshire) — 2025-08-19:** Suggests `Invoice.status`
  should adopt the typical/common status value set (record lifecycle), and
  that a new element `InvoicesStatus` be introduced on `Invoice` whose
  starting code system combines the present "happy path" values with those
  proposed in this ticket, with the value set bound **extensible** so
  implementers can express additional invoice-lifecycle states.

- **pknapp (Paul Knapp) — 2025-08-26:** Notes that the requested change does
  not match the semantics of the common `.status` element, which conveys the
  *maturity / lifecycle status of the instance*. The proposal looks instead
  like an **accounting / settlement** status (payment lifecycle). Asks
  whether the request would be met by adding a new element backed by a new
  extensible value set / code system covering the proposed concepts plus
  others such as "paid in full", "partial payment", "unpaid".

- **pknapp (Paul Knapp) — 2025-08-26:** Follow-up — if payment status is
  added, should `Invoice` also carry an optional **outstanding balance**, a
  **last payment date**, or a **payment tracking list** (date, amount, type
  code: payment / refund / adjustment)?

- **bgradl (Becky Gradl) — 2025-10-03:** Acknowledges Paul's point about
  `.status` being instance maturity; has no preference among (a) a new
  "accounting status" element, (b) a "settlement status" element, or
  (c) widening the existing `Invoice.status` value set. Use case is
  presenting VA veterans with monthly statements that combine billed
  charges and payments. Today they use the public `alternate-codes`
  extension on `Invoice.status` to map source-system lifecycle codes.
  For balance information they currently overload
  `Invoice.totalPriceComponent.code.text`; the underlying types could be
  modified to carry balance information natively. Notes that
  *last payment date* could be derived from
  `PaymentReconciliation.allocation.target`, which references `Invoice`
  in R5.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive Considered. Add the four proposed codes to the `InvoiceStatus`
code system and value set bound to `Invoice.status`:

| Code | Display | Definition |
|-|-|-|
| `refunded` | Refunded in Full | The total invoice has been refunded. |
| `partially-refunded` | Refunded in Part | One or more line items have been refunded or credited to the bill. |
| `suspended` | Suspended | The invoice can no longer have payments applied. |
| `written-off` | Written-Off | The invoice has not been paid and has been sent to collections. |

The existing `Invoice.status` binding remains **required** (per the common
"status" pattern), but the value set is widened. Existing codes (`draft`,
`issued`, `balanced`, `cancelled`, `entered-in-error`) are preserved for
backward compatibility. Update the `Invoice` resource narrative and the
status definitions table to clarify the lifecycle ordering (e.g.,
`balanced → refunded`, `balanced → partially-refunded`, `issued → suspended`,
`issued → written-off`).

#### Justification

Smallest delta to the existing artifact and the smallest implementer impact
— no new element, no new structural definitions, no new extensions to
re-bind. It directly addresses the reporter's stated need (the VA monthly
statement use case), eliminates the workaround of using the
`alternate-codes` extension on `Invoice.status`, and aligns with how
implementers are already (mis)using the field — the Zulip
*implementers > Payments* thread shows people setting
`Invoice.status = balanced` as a proxy for "paid", which a richer value set
would correct. **Caveat that must be addressed in the Justification per
Paul Knapp's comment:** the common-status pattern defines `.status` as
*record/instance maturity*, and adding settlement-oriented codes blurs that
contract. This disposition accepts that semantic compromise on the grounds
that `Invoice.status` is already not a pure record-lifecycle element
(`balanced` is a settlement state) and the resource is at low maturity, so
the precedent has effectively been set.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Mod. Adopt the structural change suggested by Mark
Scrimshire and Paul Knapp:

1. Re-align `Invoice.status` with the standard/common status pattern
   (record-lifecycle: `draft`, `active`, `retired`, `unknown`,
   `entered-in-error`), keeping the binding **required**.
2. Add a new element on `Invoice` carrying the **payment / settlement
   lifecycle**. Suggested name `Invoice.paymentStatus` (0..1, code), bound
   **extensible** to a new value set `InvoicePaymentStatus` whose code
   system seeds with: `unpaid`, `partial-payment`, `paid-in-full` /
   `balanced`, `refunded`, `partially-refunded`, `suspended`, `written-off`,
   plus `unknown`.
3. Optionally (per Paul Knapp's second comment) add `Invoice.balance`
   (0..1, Money) for outstanding balance and `Invoice.lastPaymentDate`
   (0..1, dateTime), or — more conservatively — defer those to a separate
   ticket and rely on `PaymentReconciliation.allocation.target` for last
   payment date as the reporter noted.
4. Mark the migration explicitly: codes that move from `Invoice.status` to
   `Invoice.paymentStatus` (today's `balanced`) get a deprecation note in
   the legacy code system and a ConceptMap from R5 → R6.

#### Justification

This is the cleaner long-term model: it preserves the common-status
contract that other Financial Management resources (`Claim`, `ClaimResponse`,
`PaymentNotice`, `PaymentReconciliation`) follow, makes the payment
lifecycle a first-class concept that consumers can query and constrain
independently of record state, and the **extensible** binding gives
implementers (such as the VA) the headroom they're trying to obtain today
via the `alternate-codes` extension. It also addresses Paul Knapp's open
question about balance / last-payment-date in the same go, rather than
bolting them on later. The cost is real: this is a **breaking change** to
`Invoice.status` (existing instances using `balanced` would need migration),
adds a new element and a new code system / value set, and requires a
ConceptMap. Mitigated because `Invoice` carries low FMM and the change is
proposed for the next major release.

---

### Disposition C: Decline

#### Proposal

Not Persuasive. Decline the request and direct implementers to the existing
extensibility mechanisms:

- Use the standard `Invoice.status` codes (`draft`, `issued`, `balanced`,
  `cancelled`, `entered-in-error`) for record state.
- Use the existing public `alternate-codes` / a project-defined extension
  on `Invoice.status` to convey local accounting/settlement detail (which
  is what the VA project already does).
- Use `PaymentReconciliation` (with `allocation.target` referencing the
  `Invoice`) and `PaymentNotice` to convey refund, write-off, suspension
  semantics and last-payment date.

Rationale category: **already addressed by existing mechanisms (extensions
+ PaymentReconciliation / PaymentNotice).**

#### Justification

`Invoice.status` is defined to follow the FHIR common-status pattern
(record/instance maturity) and the proposed codes are properly settlement
semantics. Refunds, partial refunds, write-offs, and suspensions are all
expressible today through the financial choreography: `PaymentNotice` for
payment events, `PaymentReconciliation.allocation` (with `type` such as
`payment`, `adjustment`, `credit`) for refunds and write-offs, and the
`Account` resource for collections handling. Widening `Invoice.status` to
mix instance state with settlement state would entrench an existing design
flaw rather than fix it, and would make `Invoice.status` even less
interoperable across the FM resources. The reporter's own use case is
already being served via the `alternate-codes` extension; that is the
architecturally correct place for source-system codes that don't map
cleanly into the standard.

---

### Recommendation

**Recommended disposition:** **B — Alternative Approach (Persuasive with
Mod).**

Disposition B is the recommendation because it is the only option that
honors *both* sides of the active discussion in the ticket. The reporter's
need — being able to express refunded / partially-refunded / suspended /
written-off natively, without an extension — is real and corroborated by
the Zulip *implementers > Payments* thread, where users default to
`Invoice.status = balanced` because the value set has nothing better.
Disposition C effectively tells those implementers to keep using
extensions, which is already the status quo and is what produced this
ticket; it also leaves the systemic mismatch between `Invoice.status`'s
nominal "record-lifecycle" semantics and its actual "balanced"/settlement
semantics in place. Disposition A removes the immediate friction but
doubles down on that mismatch, and Paul Knapp's comment makes clear that
Financial Management has a defensible principle (common-status pattern
alignment) it would be giving up. Disposition B realigns `Invoice.status`
with the common pattern, introduces a properly-scoped, **extensible**
`Invoice.paymentStatus` element that can hold both the requested codes and
Paul's proposed `paid-in-full` / `partial-payment` / `unpaid`, and provides
a single hook to address Paul's follow-up about outstanding balance and
last-payment date if the workgroup wants to scope them in. The trade-off
is a breaking change to `Invoice` and a required ConceptMap, but `Invoice`
is at low FMM and the next major version is the right window for this kind
of correction. If the workgroup decides the breaking change is unaffordable
this cycle, the fallback is Disposition A, treated explicitly as an interim
compromise on the way to B.
