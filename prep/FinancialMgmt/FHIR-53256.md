## Summary

Casey Trauer reports that the Da Vinci PDex specification's description of
how the `$bulk-member-match` (a.k.a. `$multi-member-match`) operation behaves
asynchronously is internally inconsistent. The narrative in
`payertopayerbulkexchange.md` says the operation `SHALL` follow the FHIR R4
Asynchronous Request Pattern (with the Bulk Data Status Request protocol),
which means the operation kick-off cannot return the response Parameters
resource directly — it must return a status URL whose final manifest points
at file(s) containing the response. However, the current `BulkMemberMatch`
OperationDefinition lists the response groups (Matched / NotMatched /
ConsentConstrained) inline as `out` parameters, the Bulk Member Match with
Consent diagram does not show that an ndjson file is what is ultimately
delivered, and the surrounding prose still implies an inline Parameters
response. The reporter asks for three concrete clarifications: (1) update
the diagram to show ndjson delivery, (2) remove the inline `out` parameters
from the OperationDefinition (mirroring how
[`$davinci-data-export`](https://build.fhir.org/ig/HL7/davinci-atr/OperationDefinition-davinci-data-export.html)
omits its response because it is defined in the Bulk Data spec), and (3)
make explicit that the server writes the multi-member-match response to an
ndjson file referenced by the async completion manifest, so that the same
client code path used for `$export` and `$davinci-data-export` works here.

This ticket is tightly coupled to
[FHIR-49943](https://jira.hl7.org/browse/FHIR-49943) ("Clarify usage of
`$multi-member-match`"), which raised the same async-vs-sync inconsistency
from the OperationDefinition side; that ticket's resolution explicitly calls
FHIR-53256 the deeper fix that comes when the OperationDefinition's inline
`out` parameters are removed and the response is documented as an
async-manifest delivery. The Zulip thread
"Bulk member match async operation specifications" in the **Da Vinci PDex**
stream is the implementer-side debate that motivated the ticket — one
position interprets the current spec as a single-file ndjson Parameters
delivered through the R4 async manifest; another argues that R4 async was
not designed to return a FHIR resource and that R5's `async-bundle`
(tracked separately as [FHIR-55981](https://jira.hl7.org/browse/FHIR-55981))
would be cleaner. The ticket resolution status is `Persuasive with
Modification` and the ticket is labelled `Pre-Applied`, indicating that the
narrative-only fix has already been implemented on the working branch.

## Details

**Description:**

In describing how `$bulk-member-match` works, we have the following guidance:

> The [PDex multi-member-match operation](https://build.fhir.org/ig/HL7/davinci-epdx/OperationDefinition-BulkMemberMatch.html) SHALL be performed as an Asynchronous operation. This should follow the methods identified in the [Asynchronous Request Pattern](https://hl7.org/fhir/R4/async.html) defined in FHIR R4. Implementers SHALL follow the guidance provided in the [Bulk Data Status Request section](https://hl7.org/fhir/R4/async.html#3.1.6.4) of the Async Request Pattern.

In this pattern, an operation would return a URL to the generated Parameters
resource, not the `$multi-member-match` response Parameters resource itself.

I think that there is still a lot of ambiguity in the PDex specification
about how this works.

1. The diagram in
   [Bulk Member Match with Consent](https://build.fhir.org/ig/HL7/davinci-epdx/payertopayerbulkexchange.html#bulk-member-match-with-consent)
   should be more specific that an ndjson file is returned. It is that file
   that contains the `$multi-member-match` response.
2. The operation definition here should not list those groups as out
   parameters, in much the same way that the
   [`$davinci-data-export` operation definition](https://build.fhir.org/ig/HL7/davinci-atr/OperationDefinition-davinci-data-export.html)
   excludes out response because it is defined in the Bulk Export
   specification.

In essence, to fit in the async request pattern, the operation would have to
write the whole `$multi-member-match` response to an ndjson file and store it
somewhere. The URL to that resource would be returned by the operation. I
feel we can and should make it clearer in the specification.

The point of this is that we can use the same code to make all types of
async queries — `$bulk-member-match`, `$export`, `$davinci-data-export`.

**Comments:** No comments on the ticket itself (the relevant discussion lives
on the linked FHIR-49943 ticket and on Zulip — see below).

**Resolution Description (already applied on the working branch):**

Three narrative fixes in `input/pagecontent/payertopayerbulkexchange.md`:

1. **"Out:" bullet** — now reads: *"delivered asynchronously — upon
   completion the server writes Group resources to ndjson files referenced
   by the async completion manifest"*.
2. **Bulk Member Match description** (line ~153) — replaced with a full
   async flow description: HTTP 202 → `Content-Location` → poll → ndjson
   manifest; explicitly notes the OperationDefinition intentionally has no
   inline `out` parameters, consistent with `$davinci-data-export`.
3. **Consent flow summary** (lines ~163–165) — *"resulting Groups are
   returned in the Operation response"* → *"serialized to ndjson and
   delivered via the async completion manifest"*.

The OperationDefinition itself was already correct from the FHIR-49943 work
— this ticket only needed narrative alignment.

## Related Jira Tickets

### [FHIR-49943: Clarify usage of $multi-member-match](https://jira.hl7.org/browse/FHIR-49943)

Reported by Daniel Venton against US Da Vinci PDex (FHIR), this ticket
flagged that the spec sentence requiring `Prefer: respond-async` for both
`$multi-member-match` and `$davinci-data-export` reads as "both are async"
while the `$multi-member-match` OperationDefinition is described as
synchronous. Its resolution (also `Persuasive with Modification`) splits
the conflated sentence into two unambiguous statements, removes the inline
`out` parameters (Matched / NotMatched / ConsentConstrained groups) from
the `BulkMemberMatch` OperationDefinition so the response is delivered via
an async manifest URL, and adds a paragraph at the top of section 6.4.1
describing the two-step async flow. The resolution explicitly identifies
FHIR-53256 as the narrative companion that completes the same fix.

Daniel Venton (reporter), Michele Mottini (commenter), JIRAUSER17326 (commenter).

### [FHIR-55981: (Async pattern, R5) — referenced from Zulip](https://jira.hl7.org/browse/FHIR-55981)

Filed by Michele Mottini at the end of the Zulip discussion as a proposal
to switch the async pattern to R5's `async-bundle`
(https://hl7.org/fhir/R5/async-bundle.html) instead of the R4 manifest

## Related Zulip Discussions

### [Da Vinci PDex > Bulk member match async operation specifications](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PDex/topic/Bulk%20member%20match%20async%20operation%20specifications/near/568026263)

Michele Mottini opened the thread asking whether the PDex bulk
member-match async spec is meant to follow R5's `async-bundle` pattern,
since R4's async pattern is documented for non-FHIR bulk export manifests
and does not naturally describe returning a FHIR Parameters resource.
Nathaniel Tomtishen answered that the spec text already says R4 async, and
that the intended interpretation is exactly what FHIR-53256 asks the spec
to make explicit: the status response is an R4 async-status complete
response whose `output` field points to an ndjson file containing the
`Parameters` payload, with errors handled either via HTTP 5XX +
OperationOutcome or via OperationOutcome files referenced from the
manifest's `error` field. Michele initially read the proposal as
returning the Parameters file directly in place of the manifest and pushed
back on the overhead of "manifest pointing at a single ndjson file with a
single Parameters resource", but on clarification accepted that this is
how the spec is written. The thread closed with Michele filing
[FHIR-55981](https://jira.hl7.org/browse/FHIR-55981) as a separate
proposal to use the R5 `async-bundle` pattern instead. No formal
consensus was reached on R4-vs-R5; the working group's chosen path (per
this ticket and FHIR-49943) is to clarify the existing R4 pattern rather
than switch to R5.

Michele Mottini, Nathaniel Tomtishen, (Mark Scrimshire mentioned).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the three narrative clarifications in
`input/pagecontent/payertopayerbulkexchange.md` on `HL7/davinci-epdx`,
exactly as already implemented on commit
[`2068cfcc`](https://github.com/HL7/davinci-epdx/commit/2068cfcc9379bdc61153eaba8681eb09726fc1be):

1. Rewrite the "Out:" bullet to state that the response is delivered
   asynchronously and that the server writes the response Group resources
   to ndjson file(s) referenced by the async completion manifest.
2. Replace the Bulk Member Match description (~line 153) with the full
   async flow: HTTP 202 + `Content-Location` → poll the status endpoint →
   read the ndjson manifest, and explicitly note that the
   `OperationDefinition-BulkMemberMatch` intentionally has **no** inline
   `out` parameters, mirroring `$davinci-data-export`.
3. Update the consent-flow summary (~lines 163–165) so that "resulting
   Groups are returned in the Operation response" becomes "serialized to
   ndjson and delivered via the async completion manifest".

Do **not** modify `input/fsh/Operation-bulk-member-match.fsh` — the
OperationDefinition was already corrected under FHIR-49943's
`Pre-Applied` resolution (the inline `out` parameters were removed there).
Update `input/pagecontent/changehistory.md` to record FHIR-53256 (already
done by the same commit). Targeted branch: the active
`pre-publish_stu2` / `post-publish_stu2` branch per the README's branch
conventions, not `master` blindly.

#### Justification

This is a narrative-only fix that brings the prose into alignment with the
operation definition that FHIR-49943 already corrected. It addresses every
specific clarification the reporter asked for (ndjson delivery in the
"Out:" bullet, no inline response groups, consent-flow wording), it makes
the operation behave as the FHIR R4 Asynchronous Request Pattern + Bulk
Data Status Request actually require, and it keeps the implementer
contract identical to `$davinci-data-export` and `$export` so the same
client polling code works. It is also consistent with the resolution the
working group has already labelled `Pre-Applied` and `Persuasive with
Modification`. The only file edited under this disposition lives under the
authoring root (`input/pagecontent/`); the touched diagram in the Bulk
Member Match with Consent section can remain unchanged because the
narrative now explicitly names ndjson — the briefing's "Generated areas
(do not edit)" gotcha is honoured because we do not touch
`output/changehistory.html`/`.md` or `template/changehistory.html`
(they will be regenerated on the next IG-publisher run).

### Disposition B: Alternative Approach

#### Proposal

In addition to the three narrative edits in Disposition A, also update the
Bulk Member Match with Consent **sequence diagram** so the visual flow
shows the async kick-off (HTTP 202 + `Content-Location`), the status poll,
the ndjson manifest, and the ndjson response file as discrete steps —
matching what the resolution prose now describes. Concretely: redraw the
diagram in `input/images-source/` (and apply the README's
`image-sources/pdex` workaround), regenerate `input/images/`, and ensure
the regenerated PNG/SVG is referenced from
`payertopayerbulkexchange.md`'s Bulk Member Match with Consent section.

#### Justification

Reporter explicitly asked that "the diagram … should be more specific
that an ndjson file is returned"; the Pre-Applied resolution clarifies the
prose but does not redraw the diagram. Implementers (per the Zulip
thread) are demonstrably reading the diagram and reaching different
conclusions than the prose intends — Michele Mottini's "manifest
pointing at a single ndjson file with a single Parameters resource"
pushback shows the picture matters. Trade-off vs. A: extra diagram-
authoring effort and the documented `image-sources/pdex` workaround
gotcha must be applied; the change does not affect the build other than
producing a refreshed image. Vs. C: this preserves the R4 async pattern
the WG has already endorsed (rather than reopening R4-vs-R5, which is
tracked separately under FHIR-55981) while doing more than the minimum
to remove the misreading risk that motivated the ticket in the first
place.

### Disposition C: Decline

#### Proposal

Close FHIR-53256 as already addressed by FHIR-49943's resolution.
Argument: FHIR-49943's resolution removed the inline `out` parameters from
the `BulkMemberMatch` OperationDefinition and added the two-statement
clarification of which operations are async; with that change, the
existing R4 Asynchronous Request Pattern reference in
`payertopayerbulkexchange.md` is unambiguous, and no additional narrative
edits in PDex are required. Direct implementers to the FHIR R4 async
specification and to `$davinci-data-export` for the canonical pattern.

#### Justification

This disposition is defensible only if the working group judges the
remaining ambiguity (the "Out:" bullet, the consent-flow wording, and the
diagram) as already covered by FHIR-49943's split-sentence rewrite plus
the OperationDefinition correction. It would minimise PDex edit churn
right before publication and avoid any risk of editorial drift between
the two ticket fixes. The downside is significant: the Zulip thread
documents that experienced implementers were misreading the spec **before
FHIR-49943's resolution was applied**, and FHIR-49943's prose changes are
about *which operations are async*, not about *how the response is
delivered* — the latter is exactly what FHIR-53256 targets, and the
"Out:" bullet still reads as inline-Parameters delivery without the
FHIR-53256 fix. Decline therefore leaves a real, demonstrated reader-
confusion gap on the page.

### Recommendation

**Recommended disposition:** A

This matches what has already been applied (the `Pre-Applied` label and
commit `2068cfcc`) and exactly addresses the reporter's three asks
without over-scoping. FHIR-49943 corrected the OperationDefinition;
FHIR-53256 corrects the surrounding narrative — both fixes are needed and
both have already landed. The changes are confined to a single file under
the authoring root (`input/pagecontent/payertopayerbulkexchange.md`),
they do not touch any of the briefing's "do-not-edit" generated areas,
they do not interact with the triple US Core pinning or HRex 1.1.0
dependency, and they leave the R4-vs-R5 async-pattern question (the
counter-proposal in [FHIR-55981](https://jira.hl7.org/browse/FHIR-55981))
where it belongs — as a separate, larger-scope ticket. Disposition B is
the right follow-up if the working group has appetite to also redraw the
sequence diagram in this cycle, but the diagram refresh is not on the
critical path for resolving the reported ambiguity now that the prose is
explicit about ndjson delivery. Disposition C should be rejected because
the Zulip discussion shows the prose-level ambiguity exists independently
of the OperationDefinition fix.
