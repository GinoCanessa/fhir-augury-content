## Summary

The ticket requests that the Da Vinci Patient Cost Transparency (PCT) IG add
support for a **real-time, synchronous** response to the `$gfe-submit`
operation as an option alongside the currently-defined asynchronous flow.
Today the IG describes payer processing of the GFE Packet as occurring
asynchronously (with the AEOB returned later via a separate channel). PCT
pilot participants have surfaced interest in allowing payers that can
process a GFE in real time to return the **AEOB Packet directly in the
`$gfe-submit` response**, so providers and patients can see the estimate
before the patient leaves the office. The reporter is not asking to
replace the asynchronous pattern, only to add the synchronous response as
a permitted alternative. No related Jira tickets were surfaced through
cross-references, so the request stands on its own scope-wise; it was
included in the August 2025 PCT block vote (Block-2025-Aug-2) where it was
initially dispositioned **Considered for Future Use** by a 10-0-2 vote
(Spears/Cioffi), and was subsequently reverted and re-opened post STU2
publication for re-evaluation.

## Details

**Description:**

The IG currently supports an asynchronous response to receiving a GFE.
Through PCT Pilots, there is interest in supporting a real-time
synchronous response to the `gfe-submit` operation, e.g. returning the
AEOB Packet in the operation outcome. This is not defined in the IG, but
could be added as an option in the future, which would be excellent for
providers and patients to have the information before the patient leaves
the provider office/hospital.

Currently the PCT IG has under step 5 on
<https://build.fhir.org/ig/HL7/davinci-pct/gfe_submission_and_aeob_overview.html#payer-perspective-end-to-end-workflow>:

> 5. The payer system begins processing the [GFE Packet](https://build.fhir.org/ig/HL7/davinci-pct/StructureDefinition-davinci-pct-gfe-packet.html)
> asynchronously to produce the AEOB. Some payers may process GFEs in
> near real time, which this guide can support, but more likely this
> process will take a substantial amount of time, up to the limits
> allowed by regulation. If a payer receives a [GFE Packet](https://build.fhir.org/ig/HL7/davinci-pct/StructureDefinition-davinci-pct-gfe-packet.html)
> with a `replaces` `relatesTo` identifier that matches a previous GFE
> Submission identifier by the same submitter and the AEOB is not yet
> complete, the payer may assume that an AEOB response is no longer
> needed for the previous request and process the new request instead.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an **optional synchronous response mode** to the `$gfe-submit`
operation in the PCT IG. Concretely:

- Update the `$gfe-submit` OperationDefinition to make the response
  parameter polymorphic: a payer MAY return either (a) the existing
  asynchronous acknowledgement (typically a `Bundle` or `OperationOutcome`
  with a polling/notification pointer), or (b) a fully-populated **AEOB
  Packet `Bundle`** when the AEOB is available at submission time.
- Add a new section to `gfe_submission_and_aeob_overview.html` describing
  the synchronous variant of the payer-perspective end-to-end workflow,
  rewriting Step 5 to call out the two permitted modes ("payer SHALL
  return either an acknowledgement for later retrieval *or* the AEOB
  Packet directly").
- Define how a client signals willingness to receive a synchronous
  response (e.g., `Prefer: respond-async` defaulting to async, with the
  client opting in to synchronous via header negotiation), and define how
  a payer signals "not synchronously available" so the client falls back
  to polling.
- Add CapabilityStatement expectations and at least one example
  illustrating the synchronous response.

#### Justification

The existing asynchronous pattern is preserved, so this is purely
additive and does not break current implementations. Returning the AEOB
Packet directly in the `$gfe-submit` response is a natural extension of
the operation pattern already used elsewhere in FHIR, and the IG already
acknowledges that "some payers may process GFEs in near real time" — the
ticket simply asks the IG to give those payers a normative way to return
the result. Pilot interest cited by the reporter provides a real
implementer use case, and delivering an estimate before the patient leaves
the provider office is a clear consumer-facing benefit aligned with the
intent of the No Surprises Act work that PCT supports.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than altering `$gfe-submit` itself, address the synchronous
use-case through **content negotiation / HTTP Prefer semantics already
defined by FHIR**, plus IG-level guidance:

- Document that a client MAY send `Prefer: respond-async=false` (or the
  equivalent FHIR `handling=` preference) on `$gfe-submit`, and that a
  payer that has the AEOB Packet available at submission time MAY return
  it directly as the operation `Parameters`/`Bundle` response in lieu of
  the async acknowledgement.
- Keep the OperationDefinition response untouched in terms of cardinality
  changes; instead, clarify in narrative + add a new "Synchronous
  Response (Optional)" subsection under the payer-perspective workflow
  page describing when a payer is permitted to return the AEOB Packet
  inline.
- Add a `MustSupport`-free profile note that conformant clients SHALL
  handle both response shapes.

#### Justification

This avoids any change to the `$gfe-submit` OperationDefinition signature
(which carries higher implementer impact and risks intersecting with
other in-flight tickets in the same block), and leverages mechanisms FHIR
implementers already understand. It also keeps the synchronous mode
explicitly opt-in by the requester, which matches the ticket's framing
("an option in the future") and avoids surprising existing async-only
payer implementations. Trade-off vs. A: less prescriptive, so two
implementers may interpret "synchronous OK" slightly differently — but
that risk is contained because the *content* returned is the same AEOB
Packet that's already fully profiled.

---

### Disposition C: Decline

#### Proposal

Do not adopt synchronous `$gfe-submit` response in this version of the
PCT IG. Resolve as **Not Persuasive – Considered for Future Use** with a
note that the workgroup will revisit once: (1) at least two PCT pilot
implementations have demonstrated end-to-end synchronous AEOB generation
in production-like conditions, and (2) the workgroup has connectathon
test scripts for the synchronous variant.

#### Justification

The August 2025 block vote already reached this conclusion (10-0-2:
"This will need to be reviewed and designed for a future version. Some
testing would also be good to do before inclusion in the specification.")
The reverting comment on 2025-10-08 was procedural ("re-open post STU2
publication") rather than reflecting new substantive evidence. AEOB
generation is a non-trivial payer workflow that typically involves
adjudication-adjacent processing; mandating a normative synchronous
response shape now — without connectathon evidence and without two
independent implementations — risks specifying a pattern that pilots
later find unworkable, which is harder to walk back than to add later.
Declining for this cycle is consistent with FHIR's general "prove it,
then publish it" pattern for new operation behaviors.

---

### Recommendation

**Recommended disposition:** B

The reporter's ask is a real implementer pain point and the asynchronous
pattern was never intended to forbid synchronous AEOB delivery — Step 5
already acknowledges that some payers process in near real time. That
makes pure decline (C) hard to defend now that the previous "future use"
disposition has been reverted and the ticket is back on the workgroup's
table. At the same time, fully reshaping the `$gfe-submit`
OperationDefinition response (A) is a larger surface change that
overlaps with other in-flight PCT tickets touching CapabilityStatements
and AEOB delivery (e.g., FHIR-51684), and the original block-vote
discussion explicitly flagged the need for testing before specification.
Disposition B threads that needle: it gives pilots a clearly-permitted
synchronous path using mechanisms FHIR already defines (`Prefer` /
`handling`), keeps the change additive and opt-in, and leaves room to
formalize the OperationDefinition response shape in a later cycle once
connectathon evidence accrues. The workgroup should adopt B for this
cycle and explicitly schedule a connectathon track for synchronous
`$gfe-submit` to gather the evidence that would justify upgrading to A
in a subsequent version.
