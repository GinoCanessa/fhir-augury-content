## Summary

This ticket asks that the standard set of payer adjudication amount
categories used when payers receive and adjudicate a claim be defined
directly within the `adjudication` section of `Claim`,
`ClaimResponse`, and `ExplanationOfBenefit` in FHIR Core. The
specific values called out are:

`submitted | eligible | deductible | coinsurance | copay | noncovered |
priorpayerpaid | benefit | paidbypatient | paidtoprovider | paidtopatient |
memberliability | discount | drugcost`

Today, FHIR Core's `adjudication.category` uses the small
`adjudication` value set (submitted, copay, eligible, deductible,
unallocdeduct, eligpercent, tax, benefit), and the CARIN Blue Button IG
expresses the broader set of payer amount categories on
`ExplanationOfBenefit.supportingInfo` rather than on `adjudication`.
The reporter argues these data elements logically belong inside the
adjudication structure (where the `Money` amount lives) rather than as
supporting info, so that downstream consumers can treat them
consistently across submitter, payer, and member-facing payloads.

The ticket originated against the CARIN BB IG but, per the most recent
comment from corey_spears (2025-10-30), is being treated as a FHIR Core
request. It was previously dispositioned "Considered for Future Use"
in 2020 and re-opened post STU2 (lynn_laakso, 2022-11-28). It is
labelled `Future-EOB.resource`, indicating Financial Mgmt has parked it
for the next major EOB revision.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the FHIR Core `adjudication` value set
(`http://hl7.org/fhir/ValueSet/adjudication`, backed by
`CodeSystem/adjudication`) to add the missing payer-amount categories
called out in the ticket:

- `coinsurance`
- `noncovered`
- `priorpayerpaid`
- `paidbypatient`
- `paidtoprovider`
- `paidtopatient`
- `memberliability`
- `discount`
- `drugcost`

(`submitted`, `eligible`, `deductible`, `copay`, and `benefit` are
already in the core value set.)

The codes would continue to be used on
`Claim.item.adjudication.category`,
`ClaimResponse.item.adjudication.category`, and
`ExplanationOfBenefit.item.adjudication.category` (and the equivalent
`addItem` / `total` siblings), so that each payer-amount field is
expressed as an `(category, amount)` pair on the existing
`adjudication` BackboneElement rather than via
`ExplanationOfBenefit.supportingInfo`. Update the resource narratives
on `claim.html`, `claimresponse.html`, and `explanationofbenefit.html`
to enumerate the expected categories and to deprecate the
`supportingInfo` "adjudication amount category" pattern that CARIN BB
currently uses.

#### Justification

This is the smallest-surface change that satisfies the request: the
codes are added to a value set that the resources already bind to via
`example` strength, no element is added or removed, and existing
instances continue to validate. Putting amounts on
`adjudication.category + adjudication.amount` is also the structurally
correct location — it co-locates the semantic label with its `Money`
value (and `quantity`/`value`), avoids the cross-element correlation
that `supportingInfo` requires, and matches how `submitted`,
`eligible`, `deductible`, `copay`, and `benefit` are already modelled.
CARIN BB and similar IGs can then re-bind to the enriched core code
system with a stronger binding instead of carrying a parallel set of
codes.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the core `adjudication` value set with US-payer-
oriented codes, keep the core value set as `example`-strength and
"intent" oriented (submitted/eligible/deductible/copay/benefit/etc.)
and instead:

1. Publish the additional payer amount categories
   (`coinsurance`, `noncovered`, `priorpayerpaid`, `paidbypatient`,
   `paidtoprovider`, `paidtopatient`, `memberliability`, `discount`,
   `drugcost`) in a dedicated CodeSystem owned by Financial Mgmt
   (e.g. `http://hl7.org/fhir/CodeSystem/payer-adjudication`) and a
   companion ValueSet that is referenced from
   `Claim`/`ClaimResponse`/`ExplanationOfBenefit` documentation as a
   recommended additional binding.
2. In the resource narratives, add a note that
   implementations modelling payer-side adjudicated amounts SHOULD use
   `adjudication.category` with codes drawn from that additional value
   set, and SHOULD NOT use `supportingInfo` for amount categories.
3. Leave the existing core `adjudication` value set untouched so
   existing implementers and IGs are not disturbed.

#### Justification

The list in the ticket (`paidtopatient`, `memberliability`,
`drugcost`, …) is strongly US-payer / pharmacy oriented and largely
mirrors choices CARIN BB has already made. Pulling those codes into
the base FHIR `adjudication` code system risks committing the core
spec to a US-centric vocabulary that other realms (UK NHS, AU MBS,
European national payers) may not share. A separate
`payer-adjudication` code system keeps the modelling guidance ("use
`adjudication.*` not `supportingInfo`") that the ticket really wants,
without polluting the core code system or forcing every implementer to
recognize US-only codes. It also keeps the door open for CARIN BB to
take ownership of the code list and version it on its own cadence.

---

### Disposition C: Decline

#### Proposal

Do not change FHIR Core. Mark the ticket "Not Persuasive" with the
following rationale: the binding on `adjudication.category` is already
`example` strength against
`http://hl7.org/fhir/ValueSet/adjudication`, so implementers and IGs
(including CARIN BB) are explicitly permitted to use any code system —
including the broader payer-amount list called out in the ticket — on
that element today. The choice to model these categories under
`ExplanationOfBenefit.supportingInfo` in CARIN BB is an IG-level
decision that should be re-litigated in the CARIN BB IG, not in core
FHIR.

#### Justification

The ticket conflates two questions: (a) "where in the resource should
payer amount categories live?" and (b) "should FHIR Core enumerate
those categories?". Question (a) is already answered by the existing
structure — `adjudication.category` + `adjudication.amount` is exactly
the pair-shaped element the ticket asks for, and nothing in core
prevents using the requested codes there today. Question (b) does not
require a core change: with an `example` binding, any code system can
be used. Re-modelling the CARIN BB IG to drop the `supportingInfo`
pattern in favour of `adjudication.category` is achievable entirely at
the IG layer and would deliver the interoperability benefit the
reporter is after, without the cost of a core value-set churn that
would ripple through every realm and IG that consumes the
`adjudication` value set.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying complaint is sound: amount categories
*should* be on `adjudication.category`/`amount` rather than smuggled
through `supportingInfo`, because that is the element specifically
designed to carry `(category, Money)` pairs, and locating them there
is what makes payer EOBs cross-payer-comparable. That argues against
declining (Disposition C) — even though core technically permits the
right modelling today, the ecosystem has demonstrably drifted, and
core needs to provide explicit guidance and a code system to anchor
that guidance. At the same time, the specific code list in the ticket
is unmistakably US-payer oriented (`paidtopatient`,
`memberliability`, `drugcost`, `priorpayerpaid`, …); adopting it
wholesale into the core `adjudication` value set (Disposition A)
commits FHIR Core to a US-flavoured vocabulary that other realms will
have to either ignore or reinterpret. Disposition B threads the
needle: it gives Financial Mgmt a place to publish the payer-amount
categories with HL7 governance, lets core's narrative make the
"`adjudication.*` not `supportingInfo`" rule explicit, and leaves room
for CARIN BB and other realm IGs to bind more strongly without forcing
a global core change. The label `Future-EOB.resource` further suggests
this work should be staged into the next EOB revision rather than
patched into the existing core value set in isolation.
