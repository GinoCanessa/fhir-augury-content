## Summary

This ticket asks the Financial Management work group to evaluate four
value sets currently defined inside the FHIR core specification and
bound on `ExplanationOfBenefit` for possible migration to Terminology.HL7.org
(THO). The value sets and their bindings are:

- `datestype` (CodeSystem `DatesEventTypeCodes`) bound on
  `ExplanationOfBenefit.event.type`,
- `claim-decision` (CodeSystem `ClaimAdjudicationDecisionCodes`) bound
  on `ExplanationOfBenefit.decision`,
- `icd-10-procedures` (CodeSystem `ICD10ProcedureCodes`) bound on
  `ExplanationOfBenefit.procedure.procedure[x]`, and
- `claim-decision-reason` (CodeSystem `ClaimAdjudicationDecisionReasonCodes`)
  bound on `ExplanationOfBenefit.item.reviewOutcome.reason`.

The reporter (Lloyd McKenzie) explicitly notes that migration is **not
required**, but should be considered when (a) the value-set/code-system
content is likely to need updates between FHIR releases or (b) the
content might be reusable by CDA, V2, or other HL7 specs. The ticket is
one of a series of similarly-shaped FM tickets covering the financial
resources — see "Related Jira Tickets" below — so the workgroup's
disposition here should be consistent with how those companion tickets
are resolved.

## Proposed Dispositions

The four value sets in scope have meaningfully different profiles, so a
"one-size-fits-all" answer is unlikely to be ideal; nevertheless the
ticket asks for a single disposition. Each disposition below addresses
the four value sets together and notes per-value-set nuances.

### Disposition A: Accept As Requested

#### Proposal

Migrate all four value sets and their backing code systems out of FHIR
core and into THO, then re-bind the `ExplanationOfBenefit` elements to
the new THO canonical URLs:

- Move `CodeSystem/datestype` and `ValueSet/datestype` to THO; re-bind
  `ExplanationOfBenefit.event.type` to the THO ValueSet canonical.
- Move `CodeSystem/claim-decision` and `ValueSet/claim-decision` to
  THO; re-bind `ExplanationOfBenefit.decision`.
- Move `CodeSystem/icd-10-procedures` and `ValueSet/icd-10-procedures`
  to THO (recognising that ICD-10 PCS is itself an external classification
  governed by CMS — the THO artifact remains a pointer/wrapper); re-bind
  `ExplanationOfBenefit.procedure.procedure[x]`.
- Move `CodeSystem/claim-decision-reason` and
  `ValueSet/claim-decision-reason` to THO; re-bind
  `ExplanationOfBenefit.item.reviewOutcome.reason`.

The original core canonical URLs would be retired (or aliased) per the
standing THO-migration playbook used for prior tickets in this series
(see FHIR-55302, FHIR-55266 as precedent).

#### Justification

Three of the four value sets (`datestype`, `claim-decision`,
`claim-decision-reason`) describe administrative/business semantics
that can plausibly evolve faster than the FHIR core publication cadence
and are clear candidates for cross-paradigm reuse — adjudication
decisions and reasons in particular are concepts that V2 financial
messages and CDA-based payer/clinical-financial documents could
legitimately need to reference. Migrating to THO lets FM tune
content between FHIR releases without forcing a core publication, and
matches the disposition already taken on companion tickets such as
FHIR-55302 and FHIR-55266. For `icd-10-procedures`, THO is the right
home for the wrapper artifact because the underlying classification is
externally maintained and version-volatile; keeping the wrapper inside
core forces version churn into core publications.

---

### Disposition B: Alternative Approach

#### Proposal

Migrate selectively rather than wholesale. Specifically:

1. Migrate `claim-decision`, `claim-decision-reason`, and
   `icd-10-procedures` (CodeSystem + ValueSet) to THO and re-bind
   `ExplanationOfBenefit.decision`,
   `ExplanationOfBenefit.item.reviewOutcome.reason`, and
   `ExplanationOfBenefit.procedure.procedure[x]` to the THO canonicals.
2. Leave `datestype` (and its ValueSet `datestype`) in FHIR core for
   now, on the basis that the binding is currently `Example` (or
   otherwise weak) and the small set of date-event types is closely
   tied to the FM resource model itself.
3. Track the residual `datestype` work separately under a new ticket
   if FM later concludes it should also move.

#### Justification

This honours the reporter's framing — migration is "not required" but
should be considered where reuse or independent evolution is realistic.
The two adjudication-related value sets have the strongest
cross-paradigm reuse case (V2 financial messages, CDA payer documents)
and are most likely to evolve out-of-band. The ICD-10-PCS wrapper
clearly belongs in THO for governance reasons. `datestype`, by
contrast, is narrowly EOB-shaped and gains little from THO co-location.
A selective migration also reduces the migration churn and
implementer-facing canonical-URL changes per ticket and matches FHIR's
general preference for minimum-necessary normative change.

---

### Disposition C: Decline

#### Proposal

Persuasive – Not Persuasive: keep all four value sets and code systems
in FHIR core for this cycle. No bindings change. Optionally re-file as
a tracker-only item to be revisited when (a) THO has confirmed
governance for FM-curated content, and (b) at least one of the four
value sets has had a real-world request for inter-cycle change or
cross-paradigm reuse.

#### Justification

The ticket itself notes migration is not required. None of the four
value sets has, to the data available here, an outstanding change
request or documented external (CDA/V2) reuse case. Migrating
canonical URLs is a non-trivial implementer cost (every published IG,
test artifact, and downstream profile that references the current
canonicals must update), and that cost is hard to justify on the
strength of "potentially reusable". This stance is consistent with FHIR's
general principle that normative artifacts should not be moved without a
concrete driver.

---

### Recommendation

**Recommended disposition:** B

The reporter's own framing — migration is *considered*, not required —
maps most cleanly onto a selective migration. Disposition A treats all
four value sets as equivalent and incurs the maximum implementer churn
even where the case for migration is weak (`datestype`). Disposition C
is defensible but ignores the genuine cross-paradigm reuse argument
for the adjudication-decision and ICD-10-PCS value sets, and would put
this ticket out of step with the precedents already set by FHIR-55302
and FHIR-55266 in the same review wave. Disposition B migrates the
three value sets with the strongest reuse / independent-evolution case
(`claim-decision`, `claim-decision-reason`, `icd-10-procedures`) and
defers `datestype` until a concrete driver appears, giving FM a
proportionate, defensible answer that the workgroup can apply
symmetrically across the companion tickets (FHIR-55199, FHIR-55216,
FHIR-55220, FHIR-55237) where the same value sets recur.
