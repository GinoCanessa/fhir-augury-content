## Summary

The reporter (Bhanu Vemuri) flags that the Da Vinci PDex narrative for both
the **Provider Access API** and the **Payer-to-Payer API** mentions that the
`$davinci-data-export` operation (and, on the Payer-to-Payer side, the
`$member-match` operation) "notifies" the requestor when the bulk response is
ready, but does not say *how* that notification works. The ticket asks whether
PDex relies on polling or subscriptions, and requests that the IG be explicit
— in particular, that if the intent is to follow the FHIR Bulk Data IG's
async pattern, the narrative should say so.

The ticket has been **triaged and resolved as Persuasive with Modification**
(resolved 2025-12-17). The recorded resolution states: *"Bulk Member Match,
Payer-to-Payer data export, Provider Access API (v1) data export and Provider
Access (v2) Provider-member-match and data export follow the Bulk FHIR async
protocol and use polling to determine completion of an export operation. The
narratives are updated to reflect this."* Workgroup review is therefore
about confirming the disposition language and the narrative wording the
resolution implies.

No other Jira tickets were surfaced via cross-references; the only
cross-references are mentions of FHIR-50184 in the
`HL7/davinci-epdx` change-history files (i.e., the resolution has already
been logged in the IG change log).

## Proposed Dispositions

The ticket has already been resolved on Jira as **Persuasive with
Modification**; the dispositions below are framed against that recorded
resolution. The workgroup is being asked to confirm the resolution
language and the corresponding narrative edits.

### Disposition A: Accept As Requested

#### Proposal

In `HL7/davinci-epdx` (`input/pagecontent/`), update the four narrative
pages so that every place currently using the words "notify" / "notifies"
in the context of `$davinci-data-export`, `$member-match`, and
`$provider-member-match` instead **explicitly** states that completion is
detected via the **FHIR Bulk Data Access (Flat FHIR) IG** *async request
pattern* — i.e., the requestor receives a `202 Accepted` with a
`Content-Location` polling URL, and polls that status endpoint until a
`200 OK` (or terminal error) is returned, at which point the NDJSON file
URLs are retrieved. Specifically:

- `provider-access-api.md` — replace the "$DaVinci-data-export notifies
  Requestor" and "FHIR Store … notifies requestor" sentences with a
  short paragraph stating that Provider Access v1 follows the Bulk
  Data async protocol, citing the Bulk Data IG status-polling section.
- `provider-access-api-v2.md` — same treatment for the v2 narrative,
  also covering `$provider-member-match`.
- `payertopayerbulkexchange.md` — replace the `$MemberMatch` "Notify
  Payer that parameter Response bundle is available" wording, and add
  the missing notification/polling description for
  `$davinci-data-export`, again citing the Bulk Data IG.
- `payertopayerexchange.md` — align any cross-reference text.
- `changehistory.md` — already updated (per cross-references); confirm
  the entry reads accurately.

No FSH, profile, ValueSet, CodeSystem, CapabilityStatement, or
SearchParameter changes are required. No `sushi-config.yaml` change is
required. No update to `FHIR-us-davinci-pdex.xml` or `package-list.yml`
is required.

#### Justification

This matches the reporter's request word-for-word ("If the intent is to
follow Bulk IG then including that statement will help") and the
recorded resolution. The Bulk Data async pattern with `Content-Location`
polling is the well-known, implementer-validated mechanism that Da
Vinci has used consistently for bulk member match and bulk export
across PDex, ePDex, and HRex; spelling that out in the narrative
reduces implementer ambiguity without introducing new conformance
obligations. The change is low-risk (narrative only) and respects the
"do not edit `output/` / `template/`" gotcha — the generated change-
history pages will refresh on the next IG publisher run.

---

### Disposition B: Alternative Approach — Accept Polling and Add an Optional Subscription Pathway

#### Proposal

Adopt Disposition A's narrative clarification (so the Bulk Data
async/polling pattern is explicit), **and** additionally add a SHOULD-
level discussion in `provider-access-api-v2.md` and
`payertopayerbulkexchange.md` describing an *optional* notification
mechanism using **FHIR R4 Subscriptions** (or, for forward-looking
Da Vinci pilots, a pointer to R5 `Subscription` / `SubscriptionTopic`
backports). Frame this as an out-of-band signal that a Server **MAY**
expose to a Requestor as a complement to — not a replacement for —
status-endpoint polling.

#### Justification

Polling at scale (thousands of providers × millions of members) has been
a recurring implementer pain point; several Da Vinci Connectathon tracks
have explored Subscription-based "ready" signals. Acknowledging that
pathway in the narrative gives implementers room to experiment without
fragmenting the core conformance story. The cost is narrative complexity
and the risk that implementers read the SHOULD as a MUST. There is no
new FSH artifact required — this remains a `input/pagecontent/` edit —
but it does expand scope beyond what the reporter literally asked for.

---

### Disposition C: Decline

#### Proposal

Close the ticket without any narrative change, on the basis that the
existing references to "$davinci-data-export" and "$member-match"
already imply the FHIR Bulk Data Access async protocol by virtue of
being bulk operations, and that the IG already cites the Bulk Data IG
elsewhere.

#### Justification

This is the weakest path. The ticket is explicit that the reporter
could not find the polling/subscription answer in the current text;
that alone is evidence that "implied by context" is not sufficient for
implementers. Declining contradicts the recorded resolution
(*Persuasive with Modification*) and would leave the change-history
cross-references (already merged into `input/pagecontent/changehistory.md`)
pointing to a non-change. Listed only for completeness.

---

### Recommendation

**Recommended disposition:** A

The recorded resolution is already *Persuasive with Modification* and the
PDex change-history file has been updated to reference FHIR-50184, so the
workgroup has effectively pre-committed to a narrative fix. Disposition A
delivers exactly what the reporter asked for — make the Bulk Data IG
async/polling protocol explicit in the four affected `input/pagecontent/`
pages — at the lowest risk, with no FSH or version-registration churn,
and respects the IG's "do not edit generated areas" rule. Disposition B
is attractive long-term but expands scope into Subscription pilots that
do not yet have consensus across the Da Vinci stack and that the
reporter did not request; it would be better tracked as a separate
forward-looking change request once R5 Subscription support across the
Da Vinci suite stabilises. Disposition C is not defensible given the
existing resolution.
