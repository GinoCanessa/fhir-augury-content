## Summary

The reporter (Linda Michaelsen) flags multiple typographical errors in the
`EligibilityRequestPurpose` CodeSystem (`http://hl7.org/fhir/eligibilityrequest-purpose`),
which backs the same-named ValueSet used by `CoverageEligibilityRequest.purpose`
(and the legacy R3 `EligibilityRequest.purpose`). Specifically: the word
"coverages" is misspelled as "converages" in the `definition` text of both
the `auth-requirements` and `benefits` concepts, and "billing codes" is
misspelled as "biling codes" in the `auth-requirements` definition. This is
a pure technical correction — no code values, displays, URLs, or OIDs change,
only narrative `definition` strings. The CodeSystem is `normative` and the
ValueSet is `immutable`, so the correction is constrained to non-normative
prose only. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items in the FhirAugury index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply pure-typo corrections to the `definition` element of the affected
concepts in the `EligibilityRequestPurpose` CodeSystem. Concretely, in
**both**:

- `source/coverageeligibilityrequest/codesystem-eligibilityrequest-purpose.xml`
- `source/eligibilityrequest/codesystem-eligibilityrequest-purpose.xml`

make the following replacements inside `<definition value="…"/>`:

1. `auth-requirements` concept — replace
   `"…discovered if specified, converages for the categories of service and/or specifed biling codes are requested."`
   with
   `"…discovered if specified, coverages for the categories of service and/or specified billing codes are requested."`
2. `benefits` concept — replace `converages` with `coverages` (single
   substitution).

Do not change `code`, `display`, `url`, `identifier`, `version`, or
concept ordering. Do not modify the ValueSet
(`valueset-eligibilityrequest-purpose.xml`) — it composes by `system`
only and contains no concept-level text.

#### Justification

The CodeSystem is marked `normative` (FMM 2) and the ValueSet is
`immutable`. A normative CodeSystem's *codes*, *displays*, and *meaning*
must not change, but corrections to non-normative `definition` prose are
explicitly permitted as technical corrections — which is exactly the
ticket type the reporter chose. The change is fully backward-compatible:
no instance data, validator behaviour, or terminology server expansion
output is altered. Both copies of the file (the new
`coverageeligibilityrequest/` location and the legacy `eligibilityrequest/`
copy carried for R3 compatibility) must be updated in lockstep so the
two serializations of the same canonical CodeSystem do not diverge.

---

### Disposition B: Alternative Approach (Accept and Expand Scope)

#### Proposal

Adopt all of Disposition A, **plus** sweep the same two CodeSystem files
for the other typo of the same root that the reporter did not enumerate
but that is visible in the file:

- `discovery` concept definition — replace
  `"…in addition to any specifed."` with `"…in addition to any specified."`
- `benefits` concept definition — collapse the double space in
  `"benefits consumed  for the listed"` to a single space.

Optionally, also run a regex sweep across `source/coverageeligibilityrequest/`,
`source/eligibilityrequest/`, `source/coverageeligibilityresponse/`, and
`source/eligibilityresponse/` for the strings `converages`, `specifed`,
and `biling` to catch any sibling typos in adjacent FM CodeSystems /
ValueSets and fold them into the same Technical Correction.

#### Justification

The reporter narrowed scope to the two definitions they noticed, but the
underlying root-cause is a single drafting pass that left the same
misspelling pattern in at least one further concept (`discovery`) inside
the very same file. Fixing only the two flagged definitions leaves a
known third typo in the published spec, which will almost certainly
generate a near-duplicate FM-Term ticket in the next ballot cycle.
Because every change is still narrative-only on a `normative` CodeSystem,
the risk profile is identical to Disposition A; the marginal cost is one
extra commit hunk and a regex sweep. The trade-off versus A is purely
that the workgroup is asked to authorise edits the reporter did not
explicitly request.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54443 as "Persuasive with Modification — No Action" on the
basis that `definition` text on a `normative` CodeSystem is editorial
and can be folded into the next routine editorial pass rather than a
tracked Technical Correction.

#### Justification

This disposition is included for completeness only; it is hard to
defend. The errors are unambiguous misspellings ("converages",
"biling codes") that affect the human-readable definition of codes used
by `CoverageEligibilityRequest.purpose`. Declining a Medium-priority,
zero-risk Technical Correction submitted with FM-Term labelling would be
inconsistent with how the workgroup has historically handled similar
typo tickets, and would leave incorrect English in a normative artifact
of the FHIR core specification.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is recommended. The change is the lowest-possible-risk
class of edit — narrative-only `definition` text on a `normative`,
`immutable` artifact — so there is no marginal cost in fixing the third
"specifed" instance and the cosmetic double-space at the same time. The
reporter's intent is plainly "clean up the typos in this CodeSystem",
and shipping a partial fix that knowingly leaves "specifed" in the
adjacent `discovery` concept would invite a duplicate ticket. Both file
copies (`source/coverageeligibilityrequest/codesystem-eligibilityrequest-purpose.xml`
and `source/eligibilityrequest/codesystem-eligibilityrequest-purpose.xml`)
must be updated together to keep the two serialisations of the same
canonical CodeSystem in sync. If the workgroup prefers to stay strictly
inside the reporter's wording, fall back to Disposition A; Disposition C
should be used only if the workgroup decides to bundle this into a
broader editorial pass rather than a tracked TC.
