## Summary

The submitter (Dan Murphy) asks the Financial Management work group to add
clarifying language to the Da Vinci PAS (Prior Authorization Support)
specification's Bundle construction section. Today the spec instructs
implementers to populate the PAS Request Bundle by "fully traversing all
references" from the `Claim` resource and any resources it points to.
Implementers reading this literally have concluded that the rule transitively
descends into resources reached via `Claim.supportingInfo.valueReference`
(i.e., supporting-information attachments, including `DocumentReference`,
`QuestionnaireResponse`-pointed evidence, prior CRD `ServiceRequest`s, etc.).
A Zulip discussion on the *Da Vinci CRD* stream (topic *CRD/PAS
ServiceRequest.occurrence inconsistencies*) confirmed with Lloyd McKenzie
that the transitive-resolution rule was **never intended** to extend into
supporting-information attachments — those are submitted as evidence and
should not drag in their entire reference graph. The ticket therefore
requests targeted prose in the Bundle construction section explicitly
exempting `Claim.supportingInfo.valueReference` (and the resources reached
through it) from the "fully traversing all references" requirement. There
are no other linked Jira tickets and no GitHub cross-references; the only
substantive supporting context is the Zulip thread cited above.

## Details

**Description:**

The PAS spec requires that the PAS Request Bundle be populated by
["fully traversing all references"](https://build.fhir.org/ig/HL7/davinci-pas/en/specification.html#:~:text=Additional%20Bundle%20entries%20SHALL%20be%20populated%20with%20any%20resources%20referenced%20by%20the%20Claim%20resource%20(and%20any%20resources%20referenced%20by%20those%20resources%2C%20fully%20traversing%20all%20references%20and%20complying%20with%20all%20identified%20profiles))
from the `Claim` resource and its referenced resources.

Per [the Zulip discussion](https://chat.fhir.org/#narrow/channel/180803-Da-Vinci-CRD/topic/CRD.2FPAS.20ServiceRequest.2Eoccurrence.20inconsistencies),
it was confirmed that the transitive resolution rule is not intended to
apply to attachments; those resources located in
`Claim.supportingInfo.valueReference`.

Add language to the PAS bundle construction section clarifying that the
"fully traversing all references" requirement does not extend into
resources included as supporting information.

## Related Zulip Discussions

### [Da Vinci CRD > CRD/PAS ServiceRequest.occurrence inconsistencies](https://chat.fhir.org/#narrow/channel/180803-Da-Vinci-CRD/topic/CRD.2FPAS.20ServiceRequest.2Eoccurrence.20inconsistencies)

Brian Lind opened the thread (2026-02-27) noting that the CRD and PAS
`ServiceRequest` profiles diverge on `occurrence[x]` — PAS allows only
`occurrenceTiming`, while CRD allows three choices — and asked whether the
PAS bundle is expected to "downgrade" a CRD-style `ServiceRequest` when it
flows through. Lloyd McKenzie clarified that the PAS `ServiceRequest` is
intentionally narrow because it only needs to satisfy X12 requirements and
is not meant to be the same instance as the EHR's clinical
`ServiceRequest`. Dan Murphy then surfaced the deeper implementer pain:
when a `QuestionnaireResponse` is transported in the PAS Request Bundle,
the resources it answers/references (including a CRD-style
`ServiceRequest` reached via supporting information) get pulled in by a
literal reading of the "fully traversing all references" rule, producing
profile-conformance conflicts. Lloyd confirmed that supporting-information
attachments are *not* meant to be traversed transitively — the submitter
includes whatever the payer plausibly needs, but the bundle-traversal SHALL
does not recurse through `supportingInfo`. Dan agreed and proposed adding
clarifying prose; Lloyd welcomed a change request, which became this
ticket. Consensus was reached and there were no dissenting opinions.

Participants: Brian Lind, Lloyd McKenzie, Dan Murphy.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the *Da Vinci PAS* IG's *Bundle Construction* narrative (the same
section quoted in the ticket from
`build.fhir.org/ig/HL7/davinci-pas/en/specification.html`) to add a
normative clarification immediately after the existing "fully traversing
all references" sentence. Suggested wording:

> The transitive reference-resolution requirement above SHALL NOT be
> applied to resources referenced from `Claim.supportingInfo.valueReference`.
> Resources included as supporting information are submitted as
> evidence/attachments only; the submitter is not required to additionally
> include resources they reference. The submitter SHOULD include any
> additional resources the payer is likely to need, but is not REQUIRED to
> recursively resolve the reference graph below `supportingInfo`.

Apply the same exemption to any companion language in the *PAS Request
Bundle* profile description and, if present, in the
`profiles-and-extensions` narrative for the PAS Request Bundle.

#### Justification

This is exactly the clarification the reporter (and the Zulip thread)
asked for. It matches the design intent that Lloyd McKenzie articulated
publicly: `Claim.supportingInfo` carries attachment-style evidence and was
never meant to drag the rest of an EHR graph into the prior-auth
submission. A literal reading of the existing prose has already caused
real implementer confusion (CRD-vs-PAS `ServiceRequest` profile
conflicts), so an explicit carve-out has clear interoperability value. The
change is text-only, fully backwards-compatible, and does not alter any
StructureDefinition, slice, or invariant — it only narrows a SHALL that
was being read more broadly than intended. As a Highest-priority
clarification on a balloted IG, an As-Requested edit is the lowest-risk,
highest-value path.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt Disposition A's textual carve-out, **and** add a machine-checkable
companion: a new FHIRPath invariant (e.g., `pas-bundle-traversal-1`) on
the PAS Request Bundle profile that flags Bundle entries which are
*only* reachable through a `Claim.supportingInfo.valueReference` chain
and are not themselves directly referenced by `Claim` (or by a non-`supportingInfo`
path). The invariant would be `warning` severity, not `error`, so legacy
submitters that already over-include do not break, but new submitters get
guidance. Pair this with an updated example in the IG showing a
supporting `DocumentReference` whose internal references are *not*
expanded into Bundle entries.

#### Justification

The pure-prose fix in Disposition A relies on every implementer reading
the new sentence the same way the work group means it; that is exactly
the failure mode that produced this ticket in the first place. A
warning-level invariant plus a worked example converts the clarification
into something the IG publisher and validators can surface, which is more
robust against the next implementer who only reads the profile, not the
narrative. The trade-off is added authoring/validation cost and the risk
that the invariant misclassifies edge cases (e.g., a resource reachable
via *both* `supportingInfo` and a direct `Claim` reference). Keeping the
severity at `warning` and shipping a clarifying example mitigates that
risk while still giving implementers something machine-readable to lean
on.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Persuasive — Not Persuasive with Mod* / *Not Persuasive*
on the basis that the existing FHIR core semantics of "reference" plus
the existing PAS prose are already sufficient: implementers who include
unwanted resources from `supportingInfo` are over-applying the SHALL, but
the PAS narrative does not actually mandate that behaviour, and the
narrative around `Claim.supportingInfo` already frames it as
attachment-style evidence. Close the ticket with a comment pointing to
the Zulip clarification rather than amending the IG.

#### Justification

The strongest argument for declining is scope discipline: the IG is
balloted, every textual change is a non-zero implementer cost, and the
"fix" is essentially documenting what Lloyd already documented in chat.
However, this argument is weak in practice — the very fact that an
EHR-side implementer (Dan Murphy) and a payer-side implementer (Brian
Lind) independently reached the wrong reading is concrete evidence the
prose is ambiguous, and a Highest-priority change request is not the
right vehicle to defer with "the spec is fine." Decline is documented
here for completeness and to bound the recommendation.

---

### Recommendation

**Recommended disposition:** A

The ticket is a narrowly-scoped, text-only clarification that codifies
the design intent the responsible work-group lead already confirmed in
public chat. It eliminates a documented implementer foot-gun (CRD-style
`ServiceRequest` instances being pulled into PAS bundles via
`supportingInfo` and then failing PAS profile conformance) without
breaking any existing valid sender or receiver. Disposition B is
attractive but adds a new normative invariant on a balloted profile,
which is a meaningfully larger change to argue through ballot; it can
reasonably be deferred to a follow-up ticket once the prose carve-out is
in place and implementer feedback shows whether a machine-checkable
guard is actually needed. Disposition C is hard to defend given a
Highest-priority report, two implementers reaching the same wrong
reading, and a clean, low-cost fix on the table. Recommend the work
group accept FHIR-56022 as requested, with the wording in Disposition A
(or equivalent) added to the PAS Bundle Construction section and any
parallel narrative in the PAS Request Bundle profile.
