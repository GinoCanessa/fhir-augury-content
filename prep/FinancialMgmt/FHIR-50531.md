## Summary

The reporter argues that `InsuranceProduct.coverage` (the backbone element
listing the benefits offered by an insurance product) reuses the term
"coverage" in a way that conflicts with the meaning carried by the
`Coverage` resource. `Coverage` represents an actual relationship between
a beneficiary and an insurance plan, while `InsuranceProduct.coverage`
describes the categories of benefits an insurance product offers — no
real-world coverage relationship exists at that point. The ticket asks
that the element be renamed (e.g., `benefitList`, `benefitGroup`, or
simply `benefit`) so that "coverage" remains tied to the relationship
semantics of the `Coverage` resource. The ticket has been triaged
"Persuasive with Modification" with a draft Resolution that proposes
renaming `coverage → benefit`, `coverage.type → benefit.category`,
`coverage.benefit → benefit.item`, and `coverage.benefit.type →
benefit.item.type`. No related Jira tickets, Zulip threads, or GitHub
items were surfaced by the cross-reference index for this key.

## Details

**Description:**

> Note that this is about InsuranceProduct, but Related Artifacts
> drop-down did not recognize that.
>
> The attribute name for InsuranceProduct.coverage create unnecessary
> conflict with the Coverage resource name. The Coverage resource
> reflects an actual relationship between a patient/beneficiary and an
> InsurancePlan. Therefore, when using the term "coverage" elsewhere I
> would expect it to mean the same. However, InsuranceProduct.coverage
> is not reflecting an actual instance/relationship with a
> patient/beneficiary. Had the Coverage resource been called
> CoveragePurchased and InsuranceProduct used
> InsuranceProduct.coverageAvailable (both terrible names to actually
> use) it would use "coverage" consistently.
>
> Suggest to find a replacement attribute name for
> InsuranceProduct.coverage that at least is not just "coverage" (i.e.,
> add something to it) or is different and reflects the benefit concept
> it represents. Consider the replacements to use benefitList or
> benefitGroup and the list backbone element to use benefits. Note that
> at times the backbone element is the same as an attribute within it,
> i.e., "benefit" could be used for both as well.

**Resolution Description (draft, "Persuasive with Modification"):**

> *Note: this was drafted before things moved over to InsuranceProduct.*
>
> Suggestion A — Rename:
>
> - `coverage` → `benefit` (List of benefits) *// don't think
>   offer/available is appropriate (as this is the set of benefits
>   included in this product)*
> - `coverage.type` → `benefit.category` (Benefit category —
>   classification of included benefit items; e.g., dental benefits,
>   medical benefits, physio benefits …)
> - `coverage.benefit` → `benefit.item` (details of a benefit)
> - `coverage.benefit.type` → `benefit.item.type` (e.g., consultation,
>   deductible, cosmetic surgery)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the rename already drafted in the ticket's Resolution Description
on the `InsuranceProduct` resource:

- Rename the backbone element `InsuranceProduct.coverage` →
  `InsuranceProduct.benefit` (description: "List of benefits offered
  under this insurance product").
- Rename `InsuranceProduct.coverage.type` →
  `InsuranceProduct.benefit.category` (classification of the included
  benefit items, e.g., medical, dental, physio).
- Rename `InsuranceProduct.coverage.benefit` →
  `InsuranceProduct.benefit.item` (details of an individual benefit).
- Rename `InsuranceProduct.coverage.benefit.type` →
  `InsuranceProduct.benefit.item.type` (e.g., consultation, deductible).
- Update the resource's narrative, examples, search parameters that
  reference these paths, and any maintained mappings (e.g., to legacy
  `InsurancePlan`) so that "coverage" remains reserved for the
  beneficiary-to-plan relationship modelled by the `Coverage` resource.

#### Justification

This is exactly what the reporter asks for and matches the FMG-drafted
"Persuasive with Modification" resolution already on the ticket, so it
carries the lowest interpretive risk. The rename eliminates a real
semantic collision: in FHIR, `Coverage` carries strong instance-level
"who is covered, by which plan, when" semantics, while
`InsuranceProduct.coverage` is a catalogue of benefit categories with no
beneficiary attached. Reusing the same word for both invites
implementer error in mappings, search-parameter design, and
documentation. `benefit / benefit.category / benefit.item /
benefit.item.type` is also internally consistent (the backbone uses the
same root term as the contained items, which the reporter explicitly
calls out as acceptable) and aligns with terminology already used
elsewhere in `Coverage.class` / `ExplanationOfBenefit.benefitBalance`.

---

### Disposition B: Alternative Approach

#### Proposal

Rename the outer backbone element to disambiguate it from the `Coverage`
resource, but keep the contained `benefit` element name to minimise
churn for implementers who have already begun working against the new
`InsuranceProduct` resource. Concretely:

- `InsuranceProduct.coverage` → `InsuranceProduct.benefitGroup` (or
  `coveredBenefit` if FMG prefers to keep "coverage" wording in the
  product-catalogue sense).
- `InsuranceProduct.coverage.type` → `InsuranceProduct.benefitGroup.category`.
- Leave `InsuranceProduct.coverage.benefit` and
  `InsuranceProduct.coverage.benefit.type` as
  `InsuranceProduct.benefitGroup.benefit` /
  `InsuranceProduct.benefitGroup.benefit.type` (no rename of the inner
  items).

Document the rationale in the `InsuranceProduct` resource notes,
explicitly contrasting `InsuranceProduct.benefitGroup` (catalogue of
benefits offered) vs. the `Coverage` resource (an actual coverage
instance for a beneficiary).

#### Justification

This addresses the reporter's core concern — eliminating the "coverage"
collision with the `Coverage` resource — while introducing a smaller
delta than Disposition A. The backbone-vs-item naming asymmetry is
mildly less elegant, but it preserves the inner element names that
already match the `benefit / benefit.type` pattern used in
`ExplanationOfBenefit` and `CoverageEligibilityResponse`, which reduces
re-mapping work for trading partners that have already begun migrating
from `InsurancePlan`. It is a defensible middle ground if FMG does not
want to chain four renames at once.

---

### Disposition C: Decline

#### Proposal

Keep `InsuranceProduct.coverage` and its sub-elements as currently
named. Address the reported confusion by tightening the element
descriptions and the resource narrative on `InsuranceProduct` so they
explicitly contrast `InsuranceProduct.coverage` ("the set of benefit
categories offered under this product, independent of any beneficiary")
with the `Coverage` resource ("an actual coverage relationship between a
beneficiary and an insurance plan"). Add a "Boundaries and
Relationships" note linking the two.

#### Justification

`coverage` has historical precedent on this resource (carried over from
`InsurancePlan.coverage`), and consumers migrating from `InsurancePlan`
already understand the term in the catalogue sense. A pure
documentation fix avoids breaking changes for early movers and
preserves the existing FHIR-shop tooling that maps `InsurancePlan` to
`InsuranceProduct`. The argument here is essentially "the cost of the
rename outweighs the clarity benefit, given that the two uses live on
different resources and should already be distinguishable from
context".

---

### Recommendation

**Recommended disposition:** A

The ticket has already been triaged "Persuasive with Modification" with
the rename outlined in detail, so Disposition A is the path that
matches the work group's prior direction and the reporter's intent.
The collision between `InsuranceProduct.coverage` (a benefit catalogue)
and the `Coverage` resource (a beneficiary relationship) is a real
semantic clash that documentation alone (Disposition C) is unlikely to
fix at scale — implementers routinely lean on element names rather
than narrative when wiring up systems. Disposition B is a reasonable
fallback if FMG wants to limit churn, but the consistent
`benefit / benefit.category / benefit.item / benefit.item.type` chain
in Disposition A is cleaner and lines up with existing benefit
terminology in `ExplanationOfBenefit` and
`CoverageEligibilityResponse`. `InsuranceProduct` is still relatively
new and pre-normative, so the cost of the rename is at its lowest now
and grows the longer it is deferred.
