## Summary

FHIR-53240 asks the FHIR Core spec maintainers to expand the example
CodeSystem behind `ExplanationOfBenefit.related.relationship`
(`http://hl7.org/fhir/ex-relatedclaimrelationship`, mirrored at
[terminology.hl7.org](https://terminology.hl7.org/6.0.2/CodeSystem-ex-relatedclaimrelationship.html))
by adding a single new code, **`prioreob`**. The intent is to let an
ExplanationOfBenefit point back at the EOB it replaces, so consumers can
recognise that the new EOB supersedes a prior one.

The ticket explicitly carves out the inverse direction: a `replacedbyeob`
code is **not** being requested, because the EOB is treated as an
immutable document and the prior EOB must not be mutated to record that
it has been replaced. Only the forward "this EOB is a successor of that
EOB" relationship needs a code.

This work was split out of the resolved-and-applied
[FHIR-48304](https://jira.hl7.org/browse/FHIR-48304) ("Expand resources
types the EOB.related.claim element can reference to include
ExplanationOfBenefit"), whose three-part resolution covered (1) updating
the description of `related.claim`, (2) widening its target types to
include `ExplanationOfBenefit`, and (3) adding `prioreob` to the
relationship CodeSystem. Per Paul Knapp's 2025-10-24 comment on
FHIR-48304, the terminology piece (#3) was moved out into FHIR-53240 so
the binding change can be tracked and applied independently. There is
also a parallel, broader Financial Mgmt effort —
[FHIR-50885](https://jira.hl7.org/browse/FHIR-50885) — to convert the
CodeSystem from "example" to a regular extensible CodeSystem and add
`replacedby`, `extend`, and `appealing`; FHIR-53240 should be reconciled
with that work so the two changes do not collide.

## Details

**Description:**

1. Expand the example CodeSystem for `related.relationship` to add
   "prioreob".
   <https://terminology.hl7.org/6.0.2/CodeSystem-ex-relatedclaimrelationship.html>

We are not supporting "replacedbyeob" since the EOB is considered an
immutable document. Therefore we are enabling a new EOB to point back to
the EOB it is replacing but not requiring the original EOB to be
modified to reflect that it has been replaced.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a single new concept to the `ex-relatedclaimrelationship` example
CodeSystem (canonical
`http://terminology.hl7.org/CodeSystem/ex-relatedclaimrelationship` /
mirrored at `http://hl7.org/fhir/ex-relatedclaimrelationship`):

- **code:** `prioreob`
- **display:** "Prior ExplanationOfBenefit"
- **definition:** "The referenced ExplanationOfBenefit is a prior version
  of this ExplanationOfBenefit, which adjusts and replaces it."

Do **not** add `replacedbyeob` or any inverse code, consistent with the
ticket's stated rationale that the prior EOB is immutable. The change
is purely terminological and pairs with the already-applied FHIR-48304
work that widened `EOB.related.claim` to permit referencing an
ExplanationOfBenefit; with this code added, an adjusting EOB can carry
`related.claim → ExplanationOfBenefit/<prior>` together with
`related.relationship.coding = ex-relatedclaimrelationship#prioreob`.

#### Justification

This is exactly the residual scope explicitly carved out of FHIR-48304 by
the original resolution and the 2025-10-24 split comment from Paul
Knapp; the WG already agreed to this change in principle when FHIR-48304
was applied. Adding the inverse code was deliberately rejected because
the EOB is treated as an immutable document and the prior EOB must not be
required to mutate. The change is low-risk: the CodeSystem is still an
example/non-binding CodeSystem (in current FHIR Core), no instances are
invalidated, and no constraints are tightened. It also leaves
FHIR-50885's broader conversion of the same CodeSystem to a regular,
extensible CodeSystem free to land independently — `prioreob` simply
becomes one more concept in the converted CodeSystem.

---

### Disposition B: Alternative Approach

#### Proposal

Roll FHIR-53240 into FHIR-50885 and apply both changes as a single
CodeSystem revision: convert `ex-relatedclaimrelationship` to a regular
CodeSystem, set its binding on `EOB.related.relationship` to
*extensible*, and include all of `replacedby`, `extend`, `appealing`
(from FHIR-50885) **plus** `prioreob` (from FHIR-53240) in the same
publication. Keep the immutable-EOB rationale: `replacedby` (FHIR-50885)
is the forward-looking concept on the new resource, `prioreob`
(FHIR-53240) is the back-pointer on the replacing EOB, and no inverse
`replacedbyeob` is added.

Alternatively, if the WG decides `replacedby` from FHIR-50885 already
covers the "this EOB has a prior" semantic, close FHIR-53240 against
FHIR-50885 with a note that `prioreob`'s intent is satisfied by
`replacedby`.

#### Justification

FHIR-50885 is already touching the same CodeSystem to convert it from
example to regular and to add three new codes. Landing FHIR-53240
piecemeal first risks two CodeSystem revisions in quick succession and
diverging publication artifacts on terminology.hl7.org. Bundling the
changes also forces the WG to make an explicit semantic choice between
`replacedby` and `prioreob` — they are easy to confuse, and consumers
will benefit from a single coherent vocabulary rather than two near-
duplicates added in different tickets.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-53240 on the grounds that FHIR-50885 already adds a
`replacedby` code to the same CodeSystem and converts it to a regular,
extensible CodeSystem; jurisdictions or IGs that need a more specific
"prior EOB" semantic can add it locally via the extensible binding
without changing the FHIR Core CodeSystem.

#### Justification

The terminology gap is small and partly covered by the in-flight
FHIR-50885 change. Keeping the FHIR Core example CodeSystem minimal and
letting IGs (e.g., C4BB, mentioned by DavidRiddle on FHIR-48304#60341)
add `prioreob`-style codes preserves Core's neutrality and avoids
encoding a single business interpretation into Core terminology.

---

### Recommendation

**Recommended disposition:** B

The ticket is clearly in scope and was already pre-agreed when
FHIR-48304 was applied, so outright Decline (C) is hard to justify — the
WG has already signalled it wants this concept in the CodeSystem.
Disposition A is the safe minimum, but it ignores that FHIR-50885 is
about to revise exactly the same CodeSystem (convert to regular, change
binding to extensible, add three more codes). Applying FHIR-53240 first
in isolation creates churn and leaves the WG to deconflict
`prioreob` vs. `replacedby` after the fact. Disposition B keeps the
agreed change but lands it as one coherent CodeSystem revision: convert
the CodeSystem, add `replacedby`/`extend`/`appealing` (FHIR-50885) and
`prioreob` (FHIR-53240) together, and explicitly document — in the
CodeSystem's description and in the EOB.related.relationship binding
guidance — the immutable-EOB rationale that distinguishes `prioreob`
(back-pointer on the replacing EOB) from any not-added inverse code on
the prior EOB. The WG should confirm whether `replacedby` and `prioreob`
are intended as semantically distinct concepts or whether one of the two
(most likely `prioreob`) should be the canonical name.
