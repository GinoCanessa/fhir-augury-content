## Summary

The ticket asks the Financial Management work group to anchor the
`USClaimMedicalProductOrServiceCodes` ValueSet — currently defined inside
the US Da Vinci Patient Cost Transparency (PCT) IG — in
`terminology.hl7.org` (THO) instead of leaving it scoped to the PCT IG's
own canonical namespace. The intent is reuse: the same set of US claim
medical product/service billing codes is potentially useful to any IG that
exchanges Claim, ClaimResponse, CoverageEligibilityRequest/Response, or
ExplanationOfBenefit instances in the US (not just PCT). Comments on the
ticket confirm general community acceptance for moving the ValueSet to
THO, with a caveat that the THO version must clarify the set represents
billing codes (not strictly "medical" codes). The ticket is one of a
sibling cluster (FHIR-45815, 45816, 45817, 45818, 35529–35550, etc.) that
collectively asks PCT to migrate code systems and value sets from the IG
namespace to THO. It was put on the PCT `Block-2025-Jul` block vote with
a proposed "Considered for Future Use" resolution ("Resolve across IGs
continuing to add to THO"), the block vote passed 9-0-2 on 2025-07-08,
and the resolution was then reverted on 2025-10-08 (`re-open post STU2
publication`) so it is once again Triaged and awaiting disposition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-home the `USClaimMedicalProductOrServiceCodes` ValueSet from the PCT
IG namespace (`http://hl7.org/fhir/us/davinci-pct/ValueSet/USClaimMedicalProductOrServiceCodes`)
to a THO canonical (e.g.,
`http://terminology.hl7.org/ValueSet/USClaimMedicalProductOrServiceCodes`).
Concretely, in the next PCT release: (1) submit the ValueSet definition
(name, description, compose rules, included code systems) to UTG/THO
under the proposed canonical, with the description updated per Corey
Spears' note to clarify it represents *US claim billing* codes covering
medical product or service items, not exclusively clinical "medical"
codes; (2) once published in THO, replace the in-IG ValueSet resource
with a stub or remove it from the PCT package and update every PCT
profile / binding reference (e.g., the PCT GFE Item product-or-service
binding, the PCT Adjudication item bindings) to point at the new THO
canonical; (3) keep the old PCT canonical resolvable via a deprecation
note pointing at the THO canonical to preserve binding integrity for
already-deployed PCT instances. Coordinate with the other Da Vinci
financial IGs (CRD/DTR/PAS/CARIN BB) so the same THO ValueSet is also
adopted there, satisfying the "coordinate across IGs" condition the FM
WG attached to the original block-vote disposition.

#### Justification

The ticket's request is straightforward and the community has already
expressed general acceptance on the ticket and (implicitly) on the
2025-07-08 block vote. The set is a US-realm, cross-IG terminology
artifact; FHIR best practice is for such artifacts to live in THO so
they can be governed by Vocabulary and reused without forcing
implementers to import the PCT IG just for a billing-code value set.
Doing this work now (post-STU2) avoids freezing a non-reusable canonical
into yet another published version. Corey Spears' caveat — that the THO
description must call out the "billing codes" scope — is addressed
explicitly in the proposal so the THO publication does not silently
expand the apparent semantics.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the unilateral move and instead open a coordinated
cross-IG/UTG work item that jointly migrates *all* PCT-class
billing-code ValueSets (this ticket plus the FHIR-45815/16/17/18,
35529–35550 sibling cluster) to THO in a single batch, after agreement
with the other Da Vinci financial IGs (CRD, DTR, PAS, CARIN BB,
HRex/PDex where applicable) on naming, scope, and bindings. PCT itself
would continue to ship `USClaimMedicalProductOrServiceCodes` under its
current PCT canonical for the current STU release; the disposition on
this ticket would be "Considered for Future Use" with a tracker pointing
at the cross-IG migration item (matching the "Resolve across IGs
continuing to add to THO" wording the WG already used in the July
block vote). The migration item, not this ticket, is what would actually
land the change in a future PCT version once UTG approves the THO
artifact.

#### Justification

This is exactly the conditional the FM WG attached to the 2025-07-08
block-vote disposition ("should be coordinated with other Implementation
Guides that address the same scope") and matches Lynn Laakso's reason
for reopening: the WG wants coordination, not piecemeal moves. Migrating
one ValueSet in isolation risks two parallel canonicals (one in PCT,
one in THO) that other IGs may bind to inconsistently. Batching also
amortises the UTG submission overhead and lets the renamed/clarified
description ("billing codes" rather than "medical codes") be applied
uniformly across the cluster. The downside vs. Disposition A is that the
underlying improvement is delayed by another release cycle, but the
ticket can stay open against that tracker so it is not lost.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" and keep the ValueSet in the PCT IG
namespace, with the rationale that
`USClaimMedicalProductOrServiceCodes` was authored to support PCT's
specific GFE / AEOB use of `Claim.item.productOrService` and that PCT is
the maintainer/owner of that scope. Implementers needing the same set in
other contexts can either reference the PCT canonical directly (it is a
public, resolvable URL) or define their own ValueSet.

#### Justification

Defensible only if the WG concludes (a) no other IG is in fact ready to
adopt the ValueSet, so a THO migration would be premature; (b) the
churn cost (re-binding every PCT profile, deprecating the existing
canonical, double-publication risk against deployed STU2 servers)
outweighs the reuse benefit; and/or (c) the THO Vocabulary committee
prefers not to take on US-realm billing-code value sets that are not
governed by an external US authority. None of these arguments has been
made on the ticket or in the related Zulip thread, so this disposition
is mostly a placeholder for completeness.

---

### Recommendation

**Recommended disposition:** B

The community on both the ticket and the 2025-07-08 PCT block vote
agrees the ValueSet *belongs* in THO, so a hard decline (C) ignores
clearly stated consensus. But the same community attached an explicit
condition — coordinate with the other US-realm financial IGs that share
the same scope — and Lynn Laakso's October 2025 reopening is best read
as the WG saying "yes, do it, but as part of a coordinated migration,
not a unilateral PCT-only edit." Disposition B operationalises that:
keep this ticket aligned with the sibling cluster (FHIR-45815/16/17/18
and the FHIR-35529–35550 series), open a single coordinated UTG/THO
work item covering all of them, and have PCT carry the change once the
THO canonicals are published with the clarified "billing codes" wording
Corey Spears called out. Disposition A is the right *outcome*, but
delivering it via Disposition B respects the cross-IG governance the WG
explicitly demanded and avoids landing a one-off canonical change that
would have to be undone or reconciled when the rest of the cluster
moves. Implementers lose nothing in the interim because the existing
PCT canonical remains valid until the coordinated migration ships.
