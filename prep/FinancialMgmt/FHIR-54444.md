## Summary

The reporter notes an inconsistency in the FHIR Core specification between
two module-level indexes. On the Resource List page
(`resourcelist.html`), the resources `InsurancePlan` and `InsuranceProduct`
are included under the Financial section, but the dedicated Financial
Module page (`financial.html`) does not list or describe them alongside
the other financial resources (Coverage, EOB, Claim, etc.). This is
flagged as a Technical Correction — the expected behaviour is that the
two module landing pages should be consistent so readers who enter the
Financial Module via either page see the same complete set of
resources.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced by the augury index for this ticket, so the summary is based
solely on the ticket description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Financial Module landing page so that every resource that
the Resource List page categorises under "Financial" is represented
there. Concretely, add `InsurancePlan` and `InsuranceProduct` to the
module's resource roster: include them in the resource inventory
table/list on the Financial Module page, give each a short one-line
description consistent with the existing entries (Coverage, Claim,
ClaimResponse, ExplanationOfBenefit, etc.), and link each to its
resource page. If the Financial Module page groups resources by
sub-topic (e.g., "Coverage", "Billing"), place `InsurancePlan` and
`InsuranceProduct` in the sub-topic that already owns Coverage-like
plan/product metadata (typically under a "Coverage / Plan" grouping).

No schema, StructureDefinition, or normative text changes are
required — this is a documentation/index correction on the module's
narrative page. The change should be made on the active working
branch of the specification source that generates `financial.html`.

#### Justification

The two module index pages are intended to describe the same scope,
and any asymmetry between them is a defect of the published
specification. Accepting the ticket as requested is the minimum
change that resolves the inconsistency the reporter observed and
does not affect any conformance material. The type ("Technical
Correction"), Medium priority, and the absence of any community
debate (no Zulip threads, no linked tickets) all indicate this is a
straightforward editorial fix that the Financial Management work
group can apply without re-litigating scope.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than editing the Financial Module page by hand, fix the
underlying cause of the drift: make the Financial Module page derive
its resource roster from the same categorization source that drives
the Resource List page. If the specification build already has a
machine-readable resource-to-module mapping (the source behind
`resourcelist.html`'s Financial section), change the Financial
Module page template so the resource list is generated from that
mapping at build time rather than maintained as a hand-authored list.
As part of this change, also audit the other module pages
(Administration, Clinical, Diagnostics, Workflow, Medications, etc.)
for the same class of drift and fix any additional mismatches found.

#### Justification

The reported symptom — `InsurancePlan` and `InsuranceProduct` missing
on one page but present on the other — is exactly the failure mode
that two parallel hand-maintained lists produce. Patching only the
Financial Module page (Disposition A) leaves the same trap in place
for the next resource added to the Financial section, and plausibly
in other modules too. A generated roster eliminates the whole class
of defect and is consistent with how much of the FHIR build already
treats categorization metadata as the source of truth. The trade-off
is higher scope and touching the build/tooling rather than just
prose, which is more work than the ticket literally asks for.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" on the grounds that the Resource List page
is the authoritative categorized index of all resources, and the
Financial Module page is a narrative overview that is not required
to enumerate every resource categorized to the module. Readers who
want a complete list are directed to `resourcelist.html`.

#### Justification

Declining would be defensible only if the work group formally
positions module pages as curated narratives rather than exhaustive
indexes. In that framing, omitting `InsurancePlan` and
`InsuranceProduct` from the Financial Module page is an editorial
choice, not a defect, and the reporter's expectation of symmetry
between the two pages is incorrect. This rationale is weak in
practice because the other financial resources *are* enumerated on
the Financial Module page, so the page is already functioning as an
index — which makes the omission of these two resources look like
oversight rather than curation.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly resolves the reported defect with the smallest
change that is clearly in the Financial Management work group's
purview — an editorial addition of `InsurancePlan` and
`InsuranceProduct` (and their short descriptions/links) to the
Financial Module page. It requires no normative changes, no tooling
work, and no coordination with other work groups. Disposition B is a
better long-term fix but expands scope into build tooling and an
audit of every module page, which is more appropriately tracked as a
separate infrastructure ticket; it should not block correcting the
specific asymmetry the reporter identified. Disposition C is not
recommended because the Financial Module page already lists the
other financial resources, so omitting these two is inconsistent
with the page's own established pattern and is best treated as the
technical correction the reporter filed it as.
