## Summary

`MonetaryComponent.code` is documented in the metadata-types section of the
FHIR Core specification with an *example* terminology binding, but no value
set is actually attached to that binding. The reporter (Reinhard Egelkraut)
flagged this gap so that implementers have a concrete starting set of codes
to reference and so the binding statement matches reality.

The ticket has already been triaged with a draft resolution of **Persuasive
with Modification**: a new example CodeSystem/ValueSet is to be created for
`MonetaryComponent.code`, seeded with codes such as `state-tax`, `city-tax`,
`staff-disc`, `pre-pay`, `lf-30`, `lf-90`, and `ah`. Alongside the new
binding, a clarifying comment is to be added to the element explaining the
display precedence (`code.text` → `code.coding.display` → `type`) and the
typical semantics of a "base" component.

The disposition was originally drafted in PA (1 Feb 2024 WGM, Sonja Ziegler /
Reinhard Egelkraut, 4-0-0). Subsequently, FHIR-I motioned to transfer
ownership of `MonetaryComponent` to FM (13 May 2024, Eric Haas / Gino
Canessa, 11-0-0), and FM accepted the transfer on 4 Jun 2024 (Paul Knapp /
Carie Hammond, 11-0-0). FM is therefore now the disposition owner.

The cross-reference index returned no direct cross-refs to or from this
ticket, but keyword similarity surfaces several closely-related Financial /
PA tickets touching the same `MonetaryComponent` datatype (see *Related Jira
Tickets* below); these provide useful neighborhood context, particularly
because `MonetaryComponent.code` interacts with the
`price-component-type` value set discussed in
[FHIR-44610](https://jira.hl7.org/browse/FHIR-44610).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already drafted on the ticket (Persuasive with
Modification):

1. Create a new example `CodeSystem` and example `ValueSet` for
   `MonetaryComponent.code`, seeded with the codes:
   - `state-tax` — state tax
   - `city-tax` — city tax
   - `staff-disc` — Staff Discount
   - `pre-pay` — Discount for pre-payment
   - `lf-30` — 30 day late surcharge
   - `lf-90` — 90 day late surcharge
   - `ah` — After hours surcharge
   plus a `base` code (implied by the proposed comment text) so that the
   "base" component referenced in the comment is actually present in the
   example value set.
2. Bind that ValueSet to `MonetaryComponent.code` with strength `example`,
   matching the existing binding statement in the metadata-types page.
3. Add the new element comment on `MonetaryComponent.code`:
   *"When systems show the monetary component they are likely to display
   the first of: code.text, code.coding.display, type. The component with
   the 'base' code will typically display the description of the actual
   item associated with the components."*
4. Mark the ticket Applied under FM ownership and reflect the FHIR-I → FM
   transfer in the datatype's stewardship metadata.

#### Justification

This is exactly what the reporter asked for and what PA already drafted
and FM accepted by motion. The binding statement on
`MonetaryComponent.code` is currently incomplete (it advertises an example
binding with no value set), which is a documented FHIR methodology defect
— example bindings are expected to *exemplify* a set of codes, even if
non-normative. Providing concrete starter codes lowers the barrier for
implementers and is consistent with how other PA/FM metadata-type bindings
are handled (cf. `price-component-type` for `MonetaryComponent.type`).
Including `base` in the value set closes a small inconsistency in the
draft (the comment text references a "base" code that is not in the
proposed table). The change is non-breaking: an example binding does not
constrain implementers, so adding the value set affects no existing
conformant content.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the request but split the resolution into two parts:

1. **Now (this ticket):** add the proposed *comment* clarifying display
   precedence and the role of the "base" component, and replace the empty
   example binding on `MonetaryComponent.code` with a pointer to
   `price-component-type` (the existing value set already bound to
   `MonetaryComponent.type`) downgraded to strength `example` — re-using
   it as the example value set for `code` as well, since its members
   (`base`, `surcharge`, `discount`, `deduction`, `tax`, `informational`,
   etc.) already cover the conceptual space.
2. **Defer (new follow-up ticket):** if a *distinct* example value set
   for `code` is genuinely needed (e.g., narrower codes like `lf-30`,
   `staff-disc`), create a separate ticket targeting R6 with explicit use
   cases and implementer demand, and design the CodeSystem in concert
   with the FM Invoice / ChargeItem work rather than as a one-off.

#### Justification

`MonetaryComponent.type` is already bound to `price-component-type`, which
contains a `base` code — exactly the code the proposed comment text relies
on. Introducing a *second* CodeSystem with overlapping but inconsistent
codes for `code` risks confusion: implementers will have to choose between
`price-component-type#discount` (on `type`) and a new `monetary-
component-code#staff-disc` (on `code`) for closely related concepts, and
the relationship between the two axes is not yet specified. FHIR-44610,
which is **Applied** in R6, has just refined `price-component-type`'s
definitions; piggy-backing on that work is cheaper to maintain and is
consistent with how other "code" elements that mirror "type" elements are
handled in the spec. This option still resolves the reporter's primary
defect (no value set bound) while avoiding premature codification of
ad-hoc surcharge/discount codes.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** on the grounds that an example binding without
an attached value set is acceptable for a non-required binding, and add
*only* a one-line note in the metadata-types page clarifying that the code
is intentionally open and that implementers are expected to supply their
own example codes from local fee schedules.

#### Justification

Defensible chiefly on minimum-change / scope grounds: an example binding
is the weakest binding strength in FHIR and imposes no conformance
requirement, so the absence of an explicit ValueSet is not a defect that
breaks any implementation. Publishing a thin example value set risks
*overstating* HL7's recommendation — readers tend to treat anything HL7
publishes as preferred — and the seven codes proposed are arbitrary
shop-floor labels (`lf-30`, `ah`, `staff-disc`) that no implementer is
likely to adopt verbatim. Declining keeps the spec minimal and avoids the
maintenance burden of a CodeSystem that will likely never grow.

---

### Recommendation

**Recommended disposition:** A (with the small "include `base` in the
value set" tightening already folded into the proposal).

The ticket has already been through PA's drafting, FHIR-I's transfer
motion, and FM's acceptance motion — three workgroup votes have endorsed
the substance of Disposition A (resolution = *Persuasive with
Modification*). Re-opening the substance in a Disposition B-style split
would discard that prior consensus without strong cause; while the
overlap with `price-component-type` is real, the FHIR-I/PA discussion
explicitly chose to give `code` its own example set so that the `type`
axis (kind of component) and the `code` axis (specific reason) remain
semantically distinct. Disposition C is hard to defend given that the
spec currently advertises a binding it does not actually provide — that
*is* a defect, even on an example binding, because tooling and reviewers
expect the link target to resolve. Adding `base` to the value set (the
single tightening over the as-drafted resolution) costs nothing and
closes the only internal inconsistency in the draft. FM should apply
Disposition A and treat any future expansion of the codes as a separate
R6 ticket.
