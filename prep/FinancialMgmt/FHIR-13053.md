## Summary

Wes Rishel raises five separate clarification/cleanup issues against the
Contract Signer Type value set (the value set bound to `Contract.signer.type`
in FHIR Core):

1. Inconsistent spellings of "ad litem" — *ad Lidem*, *ad lidem*, and
   *ad litem* — appear across the *Dependent* and *Guardian ad litem* codes.
   *Ad litem* is the standard legal spelling and the others appear to be
   typographical variants.
2. The definitions of **Grantee** and **Grantor** look swapped; if not a
   simple typo this should go through committee.
3. The **Interpreter** definition only mentions written and spoken language
   and does not cover sign-language interpreters.
4. **Healthcare Power of Attorney** and **Special Power of Attorney** are
   defined as *persons*, but in normal legal usage a "power of attorney"
   is the document; the person who has been granted the power needs a
   different label (commonly "attorney-in-fact" or "agent").
5. **Named Insured** vs **Covered Party** appear to denote the same concept;
   if a real semantic difference exists it should be made explicit in the
   definitions, otherwise one should be deprecated.

A 2021 comment from `mkmcdaniel` suggests pairing this with
[FHIR-20344](https://jira.hl7.org/browse/FHIR-20344), which proposes
collapsing `Contract.signer` to just the `Signature` datatype. That ticket,
if accepted, would remove `Contract.signer.type` (and therefore the binding
to this value set) from the resource — directly affecting the scope of any
fix proposed here.

## Details

**Description:**

1) The spellings *ad Lidem*, *ad lidem*, and *ad litem* are used at
*Dependent* and *Guardian ad litem*. They all seem to mean the same thing
and should be represented consistently. Googling the terms I find that
*ad litem* is used far more frequently than the others. *Ad Lidem* seems to
be used primarily on law-practice Web sites in a few states, and *ad lidem*
without upper case seems not to be used very much.

2) It appears that the definitions of Grantee and Grantor are switched.
If this has not been corrected as a typo it should be handled through the
committee process.

3) The definition of Interpreter does not seem to include sign language
interpreters as it only addresses written and spoken language.

4) Healthcare Power of Attorney and Special Power of Attorney are defined
as *persons*, whereas powers of attorney are commonly defined as the legal
document that grants the power. There should be another term for the person
that has been granted the power.

5) Named Insured vs. Covered Party. These two concepts seem to be identical.
It is unusual to have two codes that represent exactly the same concept.
If there is a semantic distinction, this should be explicitly identified in
their definitions.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply all five fixes to the Contract Signer Type code system / value set
(`http://hl7.org/fhir/contract-signer-type`) in FHIR Core for the next
release:

1. Normalise every occurrence of *ad Lidem* / *ad lidem* in the display
   strings and definitions of the *Dependent* and *Guardian ad litem*
   codes to *ad litem*.
2. Investigate the **Grantee**/**Grantor** definitions; if they are
   transposed, swap the definition text (keeping the codes themselves
   unchanged so existing instances are preserved).
3. Extend the **Interpreter** definition to explicitly include
   sign-language interpreters (and any other non-spoken/non-written
   modality) — e.g., "A person who renders communication between
   parties speaking, writing, or signing different languages."
4. Either (a) rename **Healthcare Power of Attorney** and **Special
   Power of Attorney** to *Healthcare Attorney-in-Fact* / *Special
   Attorney-in-Fact* with definitions clarifying they describe the
   appointed person, or (b) keep the displays but rewrite the
   definitions to explicitly say "the natural person granted authority
   under a [healthcare|special] power-of-attorney instrument."
5. Compare the **Named Insured** and **Covered Party** definitions; if
   genuinely identical, deprecate one (likely *Named Insured*, since
   *Covered Party* aligns with insurance/eligibility vocabulary used
   elsewhere in FHIR Financial Mgmt) and add a `replacedBy` to the
   survivor; otherwise, rewrite both definitions to spell out the
   distinction.

All five are pure terminology / definition changes; no element-level
cardinality or structure changes are required. Code values themselves
are preserved (only displays/definitions change, plus one possible
deprecation), keeping existing Contract instances valid.

#### Justification

The reporter's observations are factually correct: *ad litem* is the
standard legal spelling, sign-language interpretation is plainly within
the everyday meaning of "interpreter," and conflating the legal
*instrument* with the appointed *person* is a real definitional bug that
will lead to misuse. These are exactly the kind of terminology hygiene
fixes that benefit every implementer using `Contract.signer.type`
without breaking conformance, since codes remain stable. The Vocabulary
label on the ticket signals that VocabWG can co-sign the changes.

### Disposition B: Alternative Approach

#### Proposal

Defer the value-set clean-up and resolve FHIR-13053 jointly with
[FHIR-20344](https://jira.hl7.org/browse/FHIR-20344). Specifically:

1. Adopt FHIR-20344 first — replace `Contract.signer 0..* BackboneElement`
   with `Contract.signature 0..* Signature` (or simply expose
   `Contract.signer 0..* Signature`), so that signer typing is taken from
   `Signature.type` (bound to the existing IHE/HL7 signature-purpose value
   set), not from the Contract-local value set.
2. As part of the same change, **deprecate the `contract-signer-type`
   value set / code system** in FHIR Core, mark all of its codes as
   `status = retired`, and add a NarrativeLink pointing implementers to
   `Signature.type` (DocumentSignaturePurpose / IHE signature-purpose
   codes) and to other FHIR resources for role-modelling
   (e.g., `RelatedPerson`, `PractitionerRole`, `Coverage.subscriber`/
   `beneficiary` for the insured/covered-party distinction).
3. Apply only the spelling fix in (1) of the original ticket as a
   stop-gap erratum on the current published value set, so existing
   R4/R5 implementations are not stuck with *ad Lidem*; skip the
   structural definition rewrites for Grantee/Grantor, Interpreter, the
   POA codes, and Named Insured/Covered Party because the value set is
   on a path to retirement.

#### Justification

If FHIR-20344 is accepted, every code in the Contract Signer Type value
set becomes orphaned: no element in `Contract` would bind to it. Doing
the heavy definition rewrites now (POA semantics, sign-language
interpreter wording, Grantee/Grantor swap, Named Insured/Covered Party
deduplication) would be wasted effort that future implementers might
mistake for a forward-looking signal. Replacing
`Contract.signer.type` with `Signature.type` also fixes Wes's underlying
confusion — many of the offending codes (Grantee, Grantor, Healthcare
POA, Named Insured) describe a *role/relationship*, not the *purpose of
a signature*, and arguably should never have been on a "signer type"
value set to begin with. Pairing the two tickets is what the 2021
comment from `mkmcdaniel` already recommended.

### Disposition C: Decline

#### Proposal

Mark FHIR-13053 as *Persuasive with Mod — Will Not Fix in Current Form*
and close. Do not modify the Contract Signer Type value set. Encourage
the reporter to file targeted, single-issue tickets if any individual
clean-up is still needed against the published spec.

#### Justification

The ticket bundles five logically independent issues (spelling,
swapped definitions, missing modality, person-vs-instrument, possible
duplicate codes), some of which are clean errata (1) and some of which
are substantive design changes (4 and 5) that would require migration
guidance. Workgroup process generally prefers one ticket per change so
that ballot disposition, voting, and tracking are coherent. The ticket
has been Triaged and untouched (other than a 2021 cross-reference
comment) for eight years, suggesting low community pressure. Declining
preserves backward compatibility for the many implementations that
have already mapped to these codes by display string, while leaving the
door open for follow-up tickets that are scoped narrowly enough to
disposition cleanly.

---

### Recommendation

**Recommended disposition:** B

The 2021 comment on this very ticket already points the workgroup at
FHIR-20344, and the two issues are deeply entangled: most of the
"code-quality" complaints in FHIR-13053 (POA-as-person, Named Insured
vs Covered Party, Grantee/Grantor) are symptoms of a value set that is
trying to mix *signature purpose* (which `Signature.type` already
covers cleanly) with *party role* (which belongs on resources like
`RelatedPerson`, `Coverage`, or a properly bound role code system).
Solving FHIR-20344 dissolves most of FHIR-13053; the *ad litem*
spelling fix is the only piece worth pulling forward as a standalone
erratum. Disposition A spends committee time polishing a value set
that is on borrowed time, and Disposition C ignores genuine
definitional bugs the reporter is correct about. B keeps the small
hygiene win, prevents wasted vocabulary work, and aligns with the
existing FM_R5_Contract redesign track.
