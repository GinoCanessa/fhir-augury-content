## Summary

The reporter, working with the CARIN IG for Blue Button (C4BB), observed real
claims where two or more payees are warranted on a single claim (for example,
when both a provider and the member are payees on the same claim). The C4BB
profile inherits a `0..1` cardinality on `ExplanationOfBenefit.payee` and
therefore cannot represent these multi-payee cases. The reporter asks that
`ExplanationOfBenefit.payee` cardinality be relaxed from `0..1` to `0..*` so
that multi-payee claims can be mapped without breaking single-payee mappings.

The ticket was originally filed against the CARIN BB IG, but reviewers noted
(comments by `corey_spears`, 2026-01-08/09) that the constraint is inherited
from the base FHIR `ExplanationOfBenefit` resource and that the IG cannot
relax its parent's cardinality. The ticket has therefore been moved to FHIR
Core / Financial Mgmt for consideration; the suggested IG-side mitigation
(should the base resource not change in time) is a pre-adopted extension.

This request is closely related — by keyword similarity — to a small cluster
of prior FM tickets that have already loosened other aspects of `EOB.payee`
on the C4BB side: [FHIR-31699](https://jira.hl7.org/browse/FHIR-31699)
relaxed `payee.party` to `0..1` and added a constraint tying it to
`payee.type=other`, and [FHIR-37740](https://jira.hl7.org/browse/FHIR-37740)
broadened the allowed reference targets on `payee.party` (adding
`RelatedPerson`). FHIR-46469 is the next step in the same family — it asks
the underlying base resource to permit *more than one* payee per EOB.

No formal cross-references (Jira issue links, GitHub items, Confluence
pages) were found for this ticket. There are no GitHub PRs, branches, or
commits associated with FHIR-46469.

## Details

**Description:**

> Currently in the CARIN IG for Blue Button, the Payee field within a claim
> is a must support attribute limited in cardinality to a max of one (0...1).
> It has been observed on multiple claims that there are situations that
> warrant two or more payees (e.g., claims with both a provider and member
> listed as payees). This has caused issues in our mapping, as we are not
> able to properly display the payee(s) on the impacted claims. If possible,
> could the Payee field cardinality be changed from 0...1 to 0...* to allow
> additional payees when necessary? Doing this would enable us to correctly
> map these claims to the CARIN IG, and would not impact any claims with
> only one payee, as they would still be able to proceed as they were.

## Related Jira Tickets

The following tickets were not formally cross-referenced but were surfaced

### [FHIR-31699: Base EOB payee.party has a 1..1 cardinality but the reference type available are not sufficient](https://jira.hl7.org/browse/FHIR-31699)

Filed against C4BB by Corey Spears, this Change Request raised a parallel
concern on the *child* element `payee.party`: its `1..1` cardinality and
the limited list of reference target types (Organization, Practitioner,
Patient) were insufficient when `payee.type` is `subscriber` or `other`. It
was resolved **Persuasive with Modification** with a published outcome that
relaxed `payee.party` to `0..1`, added `RelatedPerson` as a permitted
reference target, and added an invariant requiring `payee.party` when
`payee.type=other`. This is the most direct precedent for FHIR-46469: it
shows the work group is willing to relax payee constraints, but did so on
the IG layer (because the base spec couldn't be moved fast enough) rather
than in core FHIR. Participants: Corey Spears (reporter & primary
commenter).

### [FHIR-37740: EOB.payee needs additional Referenced Resource type](https://jira.hl7.org/browse/FHIR-37740)

Filed by Lisa R. Nelson, this asked that `claim.payee` allow a
`RelatedPerson` reference (e.g., when a parent or other family member pays
on behalf of a patient). It was resolved **Persuasive** (Block-Vote-7) and
includes a comment from Corey Spears noting "this was agreed to before the
ballot for the base ExplanationOfBenefit profile, but the change did not
make the ballot version" — i.e., the broader base-spec adjustment was
intended but slipped. FHIR-46469 is the cardinality counterpart to this
reference-type change. Participants: Lisa R. Nelson (reporter), Corey
Spears (commenter).

### [FHIR-26627: EOB.provider change](https://jira.hl7.org/browse/FHIR-26627)

Filed by Mark Roberts, asking to relax `EOB.provider` to allow Organization
*or* Practitioner references. Resolved **Not Persuasive with Modification**
(Block-Vote-2a) — instead of relaxing the base, the IG narrowed
Inpatient/Outpatient Institutional EOB profiles to constrain `provider` to
`CarinBBOrganization`. Useful as a counter-example: the work group has
also chosen *not* to broaden `EOB` slots in the past and has used profile
constraints to handle the specific use case instead. Participants: Mark
Roberts (reporter), Patricia Taylor, Saul Kravitz (CarinWG approver).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the base FHIR `ExplanationOfBenefit` resource, change the cardinality of
the `payee` BackboneElement from `0..1` to `0..*`. This is a base-spec
change owned by FM (as the work group of record for `ExplanationOfBenefit`)
and would propagate naturally to every downstream profile, including the
C4BB `ExplanationOfBenefit` family. No change to the inner shape of
`payee` (`payee.type`, `payee.party`) is required — each repetition would
carry its own type/party pair, which already accommodates the
"provider + member" case the reporter describes. CARIN BB and any other
profiling IG would be free to re-tighten cardinality (e.g., back to `0..1`)
on profiles where multi-payee is not expected.

#### Justification

This directly addresses the reporter's mapping problem and is consistent
with the trajectory the work group has already taken on the sibling
elements: `payee.party` was loosened in FHIR-31699 and the reference target
list was broadened in FHIR-37740. Relaxing `0..1` → `0..*` is
*non-breaking for existing instances* (every conformant single-payee
instance remains conformant) and is a well-understood pattern in FHIR for
backbone elements that turned out to be more list-like than originally
modelled. The most recent comment thread on this very ticket (Corey
Spears, 2026-01-08/09) explicitly identifies the base resource as the right
place for the fix.

---

### Disposition B: Alternative Approach

#### Proposal

Leave base `ExplanationOfBenefit.payee` at `0..1` and instead define a
**pre-adopted extension** on `ExplanationOfBenefit` (e.g.,
`http://hl7.org/fhir/StructureDefinition/explanationOfBenefit-additionalPayee`,
typed as `BackboneElement` matching the shape of `payee`, cardinality
`0..*`). C4BB would mark the extension as Must Support on its
`ExplanationOfBenefit` profiles so that implementers can carry the
additional payees the reporter describes without waiting on a base-spec
change. The extension would be slated for absorption into `payee` itself
in a future normative release.

#### Justification

This matches Corey Spears' fallback proposal in the 2026-01-09 comment
("preferably a preadoption") and is the same pattern FM has used in the
past when a needed base change couldn't make a ballot in time (see
FHIR-37740's comment that the agreed change "did not make the ballot
version for some reason"). It unblocks CARIN BB implementers immediately,
keeps base FHIR conformance unchanged for tools that don't yet handle
multi-payee, and front-loads the design conversation about whether the
extra payees should share the same shape as the existing `payee`
backbone.

---

### Disposition C: Decline

#### Proposal

Decline the cardinality change. Take the position that
`ExplanationOfBenefit.payee` is a *single* logical payee per EOB, and that
"provider + member as payees on one claim" should be modelled as **two
EOB instances** (or as one EOB with `payee.type = other` plus a separate
mechanism — e.g., adjudication line items or `processNote` — describing
the second recipient of funds). Direct the reporter to the existing
modelling pattern and close as Not Persuasive.

#### Justification

A defensible reading of the existing `payee` design is that one EOB
represents one adjudication outcome with one payee of record, and that
splitting payment between multiple recipients is a *payment* concern, not
an *adjudication* concern. This was the framing used in FHIR-26627, where
the work group declined to relax `EOB.provider` and instead handled the
specific scenario via profiling. Relaxing cardinality on a Must-Support
backbone has downstream cost for every implementer, validator, and
mapping tool that has assumed the singleton — and the ticket only
documents one real-world scenario without quantifying frequency.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested), with B as the
fallback if FHIR R6 ballot timing makes A infeasible.

The work group has already accepted the broader argument that `EOB.payee`
was modelled too narrowly: `payee.party` cardinality was relaxed
(FHIR-31699) and its reference targets were broadened (FHIR-37740). The
current ticket completes that arc by relaxing the outer cardinality. The
change is non-breaking for existing single-payee instances, requires no
new structures or extensions to maintain, and was explicitly identified
as a base-spec issue by FM in the 2026-01-08 comment on this ticket.
Disposition B is functionally equivalent for implementers but adds an
extension that will need to be deprecated later; it is the right answer
*only* if R6 timing or stewardship constraints prevent the base change.
Disposition C is the weakest path — it pushes implementers toward
multi-EOB workarounds that don't match how payers actually adjudicate
multi-payee claims, and it is inconsistent with the work group's recent
dispositions on neighbouring `payee` elements.
