## Summary

Grahame Grieve reports that the resource list permitted as the target of
`Contract.term.action.performer` (which "indicates who or what is being asked
to perform (or not perform) the action") incorrectly includes `Substance` and
`Location`, neither of which can sensibly *perform* an action. He asks that
`Substance` and `Location` be removed from the target list and that
`HealthcareService` be added in their place. The ticket also flags a typo in
the element's definition: "ction" should read "action". A 2021 comment from
a_stechishin notes the issue was deferred for review pending other
Contract changes; it has remained in **Triaged / Unresolved** status since,
carrying the `FM_R5_Contract` label, indicating Financial Management intends
to address it in an R5/R6 Contract revision cycle. There are no
cross-referenced Jira tickets — the only cross-reference of substance is a
single Zulip post that batch-listed this tracker number alongside several
others without further discussion.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the change literally to `Contract.term.action.performer` in
`HL7/fhir` (FHIR Core, R6 ballot cycle) as part of FM's
`FM_R5_Contract` work:

1. **Target list edit** on `Contract.term.action.performer` (in the
   Contract resource StructureDefinition / `contract.fsh` or
   equivalent authoring source under the Contract source root):
   - **Remove** `Substance` from the allowed target types.
   - **Remove** `Location` from the allowed target types.
   - **Add** `HealthcareService` to the allowed target types.
   - Leave the existing person/organization/device/group targets in
     place (Practitioner, PractitionerRole, Organization, CareTeam,
     Patient, RelatedPerson, Device, Group, etc.).
2. **Definition wording fix** on the same element: change the element
   definition string from "Indicates who or what is being asked to
   perform (or not perform) the ction" to "...the action". Update any
   identical wording on `Contract.term.action.performer.reference`
   description if it carries the same typo.
3. **Resource maturity / ballot impact:** flag as a
   non-backward-compatible change to the type list (drops two
   previously-allowed targets); record under the R6 substantive
   change log for Contract; coordinate with FM's `FM_R5_Contract`
   tracker bucket so it ships with the rest of the Contract revision.
4. **Test/example sweep:** scan Contract examples and IGs that
   instantiate `term.action.performer` to confirm none currently use
   `Substance` or `Location` references; if any do, migrate them
   (most likely to `Device`, `Organization`, or the new
   `HealthcareService`) before publishing.

#### Justification

The core argument is sound on FHIR semantics: `performer` denotes an
*actor* asked to perform (or not perform) an action. `Substance` and
`Location` are not actors — they have no agency and nothing in their
resource scope models "performing." Other `*.performer` slots in FHIR
(e.g., `Procedure.performer.actor`, `MedicationAdministration.performer`,
`Task.owner`) consistently restrict targets to agentive resources
(Practitioner, PractitionerRole, Organization, Patient, RelatedPerson,
Device, CareTeam, HealthcareService, Group). Adding `HealthcareService`
aligns Contract with that established pattern and gives implementers a
realistic way to nominate a service line (e.g., "the imaging service")
as the obligated party. The typo fix is uncontroversial and zero-risk.
The breaking nature of the target removals is acceptable in an R6
substantive cycle and consistent with how `FM_R5_Contract` work has been
batched.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the typo fix and the addition of `HealthcareService`, but
**deprecate rather than remove** `Substance` and `Location` from the
target list:

1. Edit `Contract.term.action.performer` to **add** `HealthcareService`.
2. Fix the "ction" → "action" typo in the element definition.
3. Leave `Substance` and `Location` in the target list for one ballot
   cycle, but add a normative comment / mapping note flagging them as
   "deprecated targets — to be removed in a subsequent release; use
   `HealthcareService`, `Organization`, or `Device` instead." Add a
   warning to the Contract resource notes page describing the
   deprecation.
4. Open a follow-up FM ticket scheduled against the *next* Contract
   revision after R6 to actually drop the two targets, giving
   implementers a transition window.

#### Justification

The literal Disposition A removes resources from a target list that
may be referenced (even if incorrectly) by existing implementations,
profiles, or test suites; that is a hard breaking change. A
deprecation-then-remove path preserves the semantic correction
(implementers are explicitly steered away from `Substance` /
`Location`) while letting existing instances and validators continue
to function during the transition. It also matches HL7's general
preference for staged removals in widely-used resources. Trade-off:
the spec carries a known wart for an extra cycle, and the
"Substance/Location as performer" combination remains technically
valid for that cycle. Given Contract's growing but still
not-fully-mature use, the extra cycle is a modest cost.

---

### Disposition C: Decline

#### Proposal

Decline the target-list changes; apply only the typo fix.

Specifically: change "ction" to "action" in the
`Contract.term.action.performer` element definition, and take **no**
action on the resource target list (keep `Substance` and `Location`,
do not add `HealthcareService`). Justify the decline as: the existing
target list is open enough that profiles can constrain it locally;
implementers who don't want `Substance` or `Location` simply don't use
them; and since the ticket has been **Triaged / Unresolved** for ~6
years with no implementer escalations and only a single 2021
"deferred" comment, there is insufficient evidence of real-world harm
to justify a breaking change.

#### Justification

The defensible decline rationale is "lack of demonstrated implementer
pain." There is no Zulip discussion, no GitHub PR, no second Jira
ticket, and no IG-level complaint in the FhirAugury index pointing at
this specific issue — only Grahame's original 2019 review note. FHIR's
permissive base resource references can always be tightened by
profiles (e.g., a payer's Contract profile can `Reference(Practitioner
| Organization | HealthcareService)`), so the conformance cost of the
existing wording is bounded. Removing two target types is a
non-backward-compatible change that breaks any existing instance,
validator, or profile that even mentions those types. On a cost/benefit
basis with the typo fixed, declining the structural change is
defensible.

---

### Recommendation

**Recommended disposition:** A

The original observation is correct on FHIR semantics: `performer`
should reference *actors*, and `Substance` / `Location` are not
actors. Every comparable `*.performer` element in FHIR restricts
itself to agentive resources, and `HealthcareService` is the standard
addition for "a service line is the obligated party," which is a real
Contract use case (e.g., obligating a hospital's imaging service or a
health plan's case-management service). Disposition B's
deprecation-first approach is defensible and would normally be
preferred for a widely-deployed resource, but Contract is still
maturing, the offending targets (`Substance`, `Location`) are
semantically unusable in this slot — implementations that depend on
them are almost certainly modeling something incorrectly — and the
ticket is already labelled `FM_R5_Contract`, indicating FM has scoped
it into a substantive revision cycle where breaking changes are
expected and announced. Disposition C should be rejected: leaving a
known semantic error in the spec because no one has complained loudly
is the wrong default for a resource at Contract's maturity level, and
the typo-only fix understates the work FM has already queued under
`FM_R5_Contract`. Recommend FM accept as requested, schedule against
the active R6 Contract substantive change set, and run the example /
IG sweep noted in Disposition A's step 4 to confirm no in-the-wild
usages need migration before publication.
