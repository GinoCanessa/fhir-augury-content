## Summary

The reporter (Charles Willcutt, Epic) asserts that "Property & Casualty"
is a standard field on a claim in current US healthcare billing practice
but that the FHIR `Claim` resource has no identifiable place to carry
this information. They are asking the Financial Management work group to
add such a place. The ticket is short and does not specify whether the
intended representation is a coded element on `Claim` itself, a code on
the related `Coverage` / `Account`, or a property of the encounter that
generated the claim.

No cross-referenced Jira tickets, GitHub items, or Zulip threads were
returned by the FhirAugury cross-reference index for this ticket. A
broader Zulip search for "property casualty claim" surfaces a single
older `#implementers > Billing data in Encounter` thread (2018) in which
Paul Knapp — the FM co-chair who left the lone clarifying comment on
this ticket — sketches the same conceptual list of payer/responsibility
categories ("selfpay; auto accident related; work accident related;
Property/Casualty accident related (home, office etc); none of the
above"). That thread is not directly tied to this ticket but provides
useful prior art for how FM has previously thought about the
classification.

## Related Zulip Discussions

### [#implementers > Billing data in Encounter](https://chat.fhir.org/#narrow/stream/implementers/topic/Billing%20data%20in%20Encounter/near/153964581)

Originally posted in 2018, this thread discusses where billing /
coverage information should live for an `Encounter`. Varvara asks
whether `Encounter` should be extended to carry per-visit billing
context. Paul Knapp argues that `Account` typically holds the list of
applicable `Coverage` references and that the system filters them by
event nature and date of service; he then sketches a code list that
`Encounter` (or a related artifact) might carry to drive processing
decisions: "selfpay; auto accident related; work accident related;
Property/Casualty accident related (home, office etc); none of the
above." Lloyd McKenzie counters that a single code cannot capture the
flexibility patients need (e.g., excluding a specific coverage,
patient-directed funding) and recommends a separate `Account` per
distinct payment context. The thread did not reach a normative
conclusion and is the closest existing community discussion to the P&C
classification this ticket is asking about.

Participants: Varvara, Paul Knapp, Lloyd McKenzie, Brian Postlethwaite,

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a representation of "Property & Casualty" to the `Claim` resource on
FHIR Core. The most surgical approach is to extend the value set bound
to `Claim.type` (or `Claim.subType`) so that "professional", "pharmacy",
"oral", "vision", "institutional" are joined by an explicit
"property-casualty" concept (and likely peer concepts such as
"workers-compensation" and "auto" if not already present), and to
clarify in the `Claim` narrative that P&C claims are in scope. Source
under `source/claim/` in `HL7/fhir` would change in three places: (a)
the `claim-type` (or `example-claim-type`) CodeSystem/ValueSet
definition, (b) the `Claim` resource intro/page text noting P&C
applicability, and (c) example instances showing a P&C claim. No new
element on `Claim` is added.

#### Justification

`Claim.type`/`subType` is the FHIR mechanism explicitly intended to
discriminate categories of claim for routing and adjudication, and
several real-world payer ecosystems already submit P&C claims
electronically (e.g., X12 837 with Claim Filing Indicator code "AM"
auto, "WC" workers comp, and analogous P&C indicators). Extending the
existing binding keeps the change additive and non-breaking, leaves the
resource shape stable for current implementers, and aligns with how
other claim categories are already modeled. It also directly answers
the reporter's literal ask ("there is not an identifiable place for it
on the FHIR Claim today").

---

### Disposition B: Alternative Approach

#### Proposal

Treat "Property & Casualty" as a property of the **payer/coverage**
rather than the claim, and address the reporter's need by ensuring it
can be expressed on `Coverage` (and surfaced indirectly on `Claim` via
`Claim.insurance.coverage`). Concretely: confirm or extend the value
set bound to `Coverage.type` to include a "property-casualty" concept,
and — if the claim-side categorization is also wanted — recommend an
extension on `Claim` (e.g., `claim-relatedCondition` or a new
`claim-causeCategory` extension carrying values such as
`auto | workers-comp | property-casualty | none`) rather than a core
element. This mirrors Paul Knapp's 2018 Zulip framing, in which the
P&C/auto/work classification was suggested as a small code on the
encounter/account side, with `Coverage` carrying the actual payer
identity.

#### Justification

In US billing practice, "Property & Casualty" is fundamentally an
attribute of the payer/policy (a P&C carrier, often paying under a
homeowner/auto/commercial policy rather than a health policy), not of
the clinical claim line itself. Modelling it on `Coverage` keeps the
type system aligned with how the financial responsibility is actually
organized and avoids overloading `Claim.type`, which is more often used
to drive form selection (professional vs. institutional vs. pharmacy
vs. vision vs. oral). Doing it via an extension on `Claim` rather than
a new core element preserves backward compatibility for existing
implementers and lets jurisdictions opt in via profiles. It also
addresses Lloyd McKenzie's concern in the related thread that a single
top-level code cannot capture the flexibility implementers need.

---

### Disposition C: Decline

#### Proposal

Decline the change as written and ask the reporter to either (a)
identify the gap in the existing `Claim.type` / `Claim.subType` /
`Claim.related` / `Claim.accident` / `Coverage.type` machinery that
prevents them from representing a P&C claim today, or (b) supply a
concrete profile and example showing what they cannot express. Close
the ticket as "Not Persuasive" / "Need More Information" pending the
reporter attending an FM call as already requested by Paul Knapp.

#### Justification

The ticket as filed is a single sentence and does not identify which
aspect of P&C the existing `Claim` resource cannot carry. `Claim`
already exposes `type`, `subType`, `use`, `accident` (with location and
type), `related` (other related claims), and `insurance.coverage`
(linking to `Coverage`, which itself has `type`); for many P&C
workflows these are sufficient. The lone FM-side comment on the ticket
is an explicit request for clarification, and there is no evidence in
Jira, GitHub, or Zulip of follow-up from the reporter. Declining (with
a clear path to reopen) avoids speculative spec changes and matches the
work group's stated need to "understand and evaluate" the comment
before acting.

---

### Recommendation

**Recommended disposition:** B

Property & Casualty is in real US production use as a claim category,
so the workgroup should not simply decline (Disposition C is defensible
only if the reporter never engages, in which case it becomes the
fallback). At the same time, modelling P&C as a top-level
`Claim.type` (Disposition A) conflates "what kind of payer is paying"
with "what kind of claim form / clinical context this is", which is
already how `Claim.type` is otherwise used and is the cleavage Paul
Knapp drew in the 2018 Zulip thread. Disposition B — extending
`Coverage.type` (and, if needed, providing a small extension on
`Claim`) — gives implementers an unambiguous place to identify a P&C
payer without disturbing the existing `Claim.type` semantics, is
additive and non-breaking, and is consistent with how FM has historically
discussed this classification. The work group should still take Paul
Knapp's offer and bring the reporter onto an FM call to confirm the
exact use case before drafting the change.
