## Summary

FHIR-35405 asks the CARIN IG for Blue Button (C4BB) work group to move
the canonical (defining) URL of the `AHANUBCPriorityTypeOfAdmissionOrVisit`
ValueSet from the IG namespace (`hl7.org/fhir/...`) into terminology.hl7.org
(THO), and to file a corresponding UTG ticket so that the ValueSet is
registered and maintained in THO. The underlying rationale (per the
imported Comment 17 from Ron G. Parker) is that ValueSets — particularly
those that simply expose an external code system (here, the AHA NUBC
Priority/Type of Admission or Visit codes) — should not be anchored in
the IG's own canonical space; only artifacts with a *required* binding to
a `code` data type belong under `hl7.org/fhir/...`. The ticket was
originally part of the Block-Vote-6 batch reviewed on 2022-07-26 and was
voted "Considered for Future Use" (Corey Spears / Kathleen Conner, 13-0-0)
with the disposition note "This is a ValueSet including all codes in a
complete external Code system. There is a ticket to add 'all codes'
ValueSets ([UP-316](https://jira.hl7.org/browse/UP-316)). When this is
complete, the ValueSet in this IG can be removed and the bindings point
to the THO ValueSet." That resolution was reverted on 2022-11-28 by
lynn_laakso to re-open post STU2 publication, leaving the ticket in
Triaged status pending a new Financial Mgmt disposition. The ticket has
no Jira cross-references and no GitHub cross-references; the only
external mention is the Block-Vote-6 announcement on Zulip.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the originally-requested change in CARIN BB: in the next published
version, remove the in-IG `ValueSet-AHANUBCPriorityTypeOfAdmissionOrVisit`
(canonical
`http://hl7.org/fhir/us/carin-bb/ValueSet/AHANUBCPriorityTypeOfAdmissionOrVisit`)
and re-anchor the binding to a THO-hosted ValueSet (e.g.
`http://terminology.hl7.org/ValueSet/AHANUBCPriorityTypeOfAdmissionOrVisit`)
that simply enumerates the AHA NUBC Priority (Type) of Admission or
Visit code system. File the corresponding UTG ticket to register the
ValueSet in THO (the work the ticket explicitly asks for), then update
every C4BB profile that currently binds to the IG ValueSet (notably the
institutional EOB profile) to bind to the THO canonical instead.

#### Justification

This is the literal request from Joan Harper / Ron G. Parker and aligns
with HL7 terminology stewardship guidance: a ValueSet that is just
"all codes from an external CodeSystem" belongs in THO, not in an IG's
canonical namespace, so that other IGs can reuse it and so that
maintenance follows the same cadence as the underlying NUBC release.
Doing it now (rather than deferring) closes the ticket cleanly without
relying on the open-ended UP-316 work, and keeps FHIR-35405 from being
re-opened a third time.

---

### Disposition B: Alternative Approach — Considered for Future Use, gated on UP-316

#### Proposal

Re-apply the original Block-Vote-6 disposition with an explicit gating
condition: mark FHIR-35405 "Considered for Future Use", note that the
ValueSet will be retired from CARIN BB and the bindings re-pointed to a
THO ValueSet **once UP-316 ("all codes" ValueSets in THO) lands**, and
add the ticket to the C4BB tracking list for the next ballot cycle so
it is acted on automatically when UP-316 publishes. No file change in
this ballot; the IG retains
`http://hl7.org/fhir/us/carin-bb/ValueSet/AHANUBCPriorityTypeOfAdmissionOrVisit`
as-is for STU2.

#### Justification

This is exactly what the WG voted on 2022-07-26 (13-0-0) before
lynn_laakso reopened the ticket post-STU2; the underlying rationale —
that the canonical place for "all codes" enumerations of an external
code system is the THO "all codes" ValueSet pattern — is still valid
and is shared with ~12 sibling FHIR-3540x/-3544x tickets in the same
block. Acting unilaterally now (Disposition A) duplicates work that
UP-316 is explicitly designed to deliver and risks creating a CARIN-
specific THO ValueSet that has to be deprecated when the generic
pattern arrives. Documenting the gating dependency explicitly addresses
the reason the original resolution was reverted (lack of follow-through
visibility post-STU2).

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Decline the migration for this specific ValueSet. The C4BB IG keeps
`AHANUBCPriorityTypeOfAdmissionOrVisit` in its own canonical namespace,
matching the pattern already applied in Block-Vote-6 to the "not
complex, fairly specific to this IG" ValueSets such as FHIR-35403
(`AHANUBCPointOfOriginForAdmissionOrVisit`), FHIR-35409, FHIR-35413,
FHIR-35414, FHIR-35415, FHIR-35432, FHIR-35434–35436, FHIR-35439,
FHIR-35446, and FHIR-35447, all of which were voted "Not Persuasive"
with the rationale that the ValueSet is small, IG-specific, and does
not warrant THO maintenance.

#### Justification

A consistent "Not Persuasive" line is defensible when the ValueSet is
a narrow enumeration that no other IG currently consumes; it avoids
adding a THO maintenance burden for a ValueSet whose only known
consumer is C4BB. This treats FHIR-35405 the same way the WG already
treated several siblings in the same block.

---

### Recommendation

**Recommended disposition:** B

The 2022-07-26 vote already settled the substantive question 13-0-0:
this ValueSet *is* an "all codes from an external CodeSystem" pattern
(AHA NUBC Priority/Type of Admission or Visit), which is exactly the
class of ValueSet UP-316 is being built to host generically in THO.
The only reason the ticket is back in front of the WG is procedural —
the resolution was reverted to surface the unfinished follow-up after
STU2 publication, not because the technical decision changed. Disposition
A would force CARIN to do UP-316's job ahead of UP-316 and risks a
churn-then-deprecate cycle once the generic pattern lands; Disposition C
contradicts the WG's own classification of which C4BB ValueSets are
"specific to this IG" vs. "wraps an external code system" (this one is
clearly the latter — NUBC). Disposition B reaffirms the original vote,
keeps FHIR-35405 aligned with the rest of its Block-Vote-6 cluster, and
explicitly attaches a tracking dependency on UP-316 so the ticket
actually moves the next time the WG cycles through deferred items.
