## Summary

The ticket flags two related sources of confusion in the PDex single
`$member-match` flow. First, the narrative under "The $member-match
operation" (process diagram step 2b, item 15) uses the FHIR-loaded word
**Identifier** to describe what is actually a **Patient FHIR ID** — a
reference rather than a business identifier — which conflicts with FHIR
terminology conventions. Second, single `$member-match` is layered on the
HRex `hrex-parameters-member-match-out` profile, which defines two output
parameters — `MemberIdentifier` (a business `Identifier`, e.g. UMB) and
`MemberId` (a `Reference(Patient)`) — but only `MemberIdentifier` is
mandatory / Must Support per HRex. PDex's prose, by contrast, treats the
Patient FHIR ID as the thing that is returned, leaving implementers
unsure whether to plan for one parameter, the other, or both, and how
that interacts with an OAuth2 authorization flow that expects a Patient
ID for `sub`.

The reporter explicitly asks (a) why "Identifier" is used in PDex prose
when "Patient FHIR ID" is meant, and (b) how implementations should be
written when the HRex contract makes the Identifier the only must-support
return.

The ticket has already been resolved as **Persuasive** (`Pre-Applied`) on
2026-03-17. The applied resolution adds a new "**$member-match Response
Parameters**" subsection under "The $member-match operation", explicitly
defining both HRex parameters and their cardinality, introduces two new
PDex conformance statements (§pdex-197a — server SHALL return
`MemberIdentifier` and SHOULD return `MemberId`; §pdex-197b — clients
SHALL handle either parameter alone or both, and SHOULD use `MemberId` as
the OAuth2 `sub` when present), and corrects the loose "Identifier"
terminology at three call-sites in the narrative (lines 43, 120, 321).

For historical context: a closely-related precursor ticket
[FHIR-38701](https://jira.hl7.org/browse/FHIR-38701) ("PDex and HRex
identify different profiles for $member-match", Persuasive, Published)
addressed the *input* profile mismatch in 2022 by changing the HRex
member-match request profile to reference the HRex demographics profile.
FHIR-50227 is the analogous cleanup on the **response** side.

## Details

**Description:**

> I wanted to raise two issues related to single $member-match that at
> the very least are causing a bit of dissonance in our requirements
> analysis.
>
> 1. In the process diagram under step 2b, item 15, we read:
>
>    > **Return Member Identifier (Member ID):** This process assumes
>    > Payer1 provides a unique ID for the member that crosses all
>    > relevant contract arrangements. .... Payer2 receives the Member
>    > Identifier (Patient FHIR ID) from Payer1. The Member Identifier
>    > (Patient FHIR ID) is only returned if member is matched AND
>    > consent can be complied with per date and policy constraints.
>
>    The issue with this wording is that it uses the word Identifier,
>    which in FHIR terminology generally refers to a business identifier,
>    not a FHIR Patient Reference. Is there a reason we don't simply use
>    Patient FHIR ID instead of a catch-all term?
>
>    This leads into the second question ...
>
> 2. Single $member-match leverages the HRex profile. The profile for
>    [$member-match response](https://hl7.org/fhir/us/davinci-hrex/StructureDefinition-hrex-parameters-member-match-out.html)
>    describes two possible returned parameters: Member Identifier (UMB)
>    and Member ID (a Patient ID). However, only the *identifier* is
>    considered mandatory and must support. That means there appears to
>    be a conflict between the PDex specification for $member-match and
>    the HRex response profile.
>
>    As an implementer, I am wondering: Should I game plan for a
>    scenario where a $member-match operation can return one or the
>    other or both? And if an identifier is the only must-support, how
>    does that affect the authorization flow if the auth server expects
>    a Patient ID?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution that has already been merged on the
`pre-publish_stu2` track for PDex 2.x:

1. In `input/pagecontent/PayerToPayerExchange.md`, under "The
   `$member-match` operation", add a new subsection
   **"$member-match Response Parameters"** that names both HRex
   parameters and their cardinalities verbatim:
   - `MemberIdentifier` — `Identifier` (e.g. UMB) — *1..1 Must Support
     per HRex* — represents the cross-payer business identifier.
   - `MemberId` — `Reference(Patient)` — *0..1 per HRex* — represents the
     Patient FHIR ID at Payer1.
2. Add two PDex conformance statements alongside the existing §pdex-19x
   block:
   - **§pdex-197a** — "Responding payers **SHALL** return
     `MemberIdentifier` in the `$member-match` response when a member is
     matched and consent permits, and **SHOULD** return `MemberId`."
   - **§pdex-197b** — "Implementers **SHALL** be able to process
     `$member-match` responses containing `MemberIdentifier` alone,
     `MemberId` alone, or both. When `MemberId` is present, it
     **SHOULD** be used as the OAuth2 `sub` in the subsequent
     authorization flow."
3. Replace the loose "unique Member ID" / "Unique Patient Identifier
   (Patient FHIR ID)" wording at lines 43, 120, and 321 of the same
   narrative file with the precise parameter names
   (`MemberIdentifier` / `MemberId`), and amend the process diagram
   caption for step 2b item 15 to read "Return `MemberIdentifier` (and,
   when available, `MemberId`)" rather than "Return Member Identifier
   (Member ID)".
4. Refresh `Requirements-fromNarrative.json` and
   `conformance_statements.csv` so the new §pdex-197a / §pdex-197b
   entries are picked up by the conformance-language table.
5. Add a `changehistory.md` row referencing FHIR-50227.

No FSH or capability-statement change is required: the cardinalities
flow from HRex.

#### Justification

This is exactly what the reporter asked for and what has already been
applied (`Pre-Applied` label, Resolution = Persuasive,
[commit `7b459ff`](https://github.com/HL7/davinci-epdx/commit/7b459ff5c9d662431c044cfd9dae0ac48c68b781)).
It (a) repairs the FHIR terminology mismatch (an `Identifier` is a
business identifier, not a `Reference(Patient)`); (b) makes the
multiplicity story explicit so an implementer knows to handle both
parameters; and (c) writes down the previously-undocumented assumption
that the Patient FHIR ID, when present, is what feeds the OAuth2 `sub` —
which is the actual operational concern the reporter raised. It honours
the PDex/HRex layering pattern established by the precursor ticket
[FHIR-38701](https://jira.hl7.org/browse/FHIR-38701): HRex owns the
profile contract, PDex layers usage and conformance language on top.
The change is narrative and conformance-language only, so it stays
inside the "Add narrative" recipe in the briefing and does not engage
the FSH layer, the capability statements, or the triple US Core pins.
The one repo gotcha that bites — the manual
`process_conformance.py` / `conformance_statements.csv` step for the
new §pdex-197a/b — is called out in step 4.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same narrative + conformance-statement clarification as
Disposition A, **but** push the multiplicity/Must Support change to the
HRex profile itself rather than carrying it as PDex-side conformance
language. Specifically:

1. In PDex 2.x, apply only the terminology fixes (lines 43, 120, 321
   and the process-diagram caption) and a short "see HRex" pointer
   subsection. Do *not* introduce §pdex-197a / §pdex-197b.
2. Open a companion HRex change request asking HRex to either (a)
   tighten `hrex-parameters-member-match-out` so that *both*
   `MemberIdentifier` and `MemberId` are Must Support when a payer
   participates in payer-to-payer exchange, or (b) add an HRex
   conformance statement to the same effect. The OAuth2-`sub`-on-
   `MemberId` guidance would also be hoisted to HRex (or to a
   shared HRex narrative), since `$member-match` is also consumed by
   PCDE/PCT and the auth-flow expectation is not PDex-unique.
3. PDex would then re-issue the affected paragraphs as references to
   the new HRex contract.

#### Justification

The strongest argument for B is *consistency*: PDex is one of several
Da Vinci IGs that consume HRex `$member-match`. If the operational
expectation ("you must always be able to return both, and `MemberId` is
what fills OAuth2 `sub`") is correct in PDex, it is almost certainly
also correct in PCDE / PCT / any other consumer. Encoding it in HRex
spares each consumer IG from redefining the same expectation in slightly
different language and forecloses the kind of drift that produced
FHIR-38701 and FHIR-50227 in the first place.

The trade-offs vs. A: it requires HRex Work Group cycles
(Clinical Interoperability Council on the precursor), it lengthens the
turnaround for PDex implementers, and §pdex-197b's OAuth2 guidance is
arguably PDex-specific (only PDex makes the SMART/OAuth2 assumption
that explicit in its trust framework), so hoisting *all* of it to HRex
may overshoot. It also loses the in-repo testability the briefing
flags: PDex's `process_conformance.py` / `conformance_statements.csv`
machinery exists precisely to track PDex-numbered conformance
statements, and §pdex-197a/b plug into that.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive — already addressed by the underlying HRex
profile**. Rationale category: *already addressed by existing
mechanism*. The HRex `hrex-parameters-member-match-out` profile is
normative; it already defines `MemberIdentifier` as Must Support 1..1
and `MemberId` as 0..1. Implementers should follow the HRex profile
contract rather than the loose PDex prose. PDex would emit only an
errata-style note pointing readers at the HRex profile.

#### Justification

A strict-constructionist reading: HRex *is* the source of truth for
`$member-match`; if PDex prose contradicts HRex, the reader is supposed
to defer to HRex anyway. Adding new PDex conformance statements
(§pdex-197a/b) on top of an already-Must-Support HRex element risks
double-specification (the same SHALL appears in two specs with subtly
different wording, and at the next HRex revision they could diverge
again — exactly the failure mode FHIR-50227 reports).

The argument against: this is a *narrative reconciliation* ticket, not a
profile change. The reporter is explicit that PDex prose is misleading
implementers in practice, and "RTFM-the-other-IG" is not a defensible
disposition for a pre-publication clarification ticket — particularly
when the PDex narrative is the document an implementer reads first to
understand payer-to-payer exchange. The OAuth2-`sub`-on-`MemberId`
guidance is also genuinely missing from HRex today, so "already
addressed" overstates the existing coverage.

### Recommendation

**Recommended disposition:** **A** (which is also what has been
applied — the ticket is `Pre-Applied`, Resolution = Persuasive).

The reporter's two questions are concrete and operationally important:
"why does the prose say `Identifier` when it means `Reference(Patient)`"
and "as an implementer, how do I plan for both parameters and where does
the OAuth2 `sub` come from?" Disposition A answers both directly, in
the document the implementer is reading, with conformance numbers that
plug into PDex's existing conformance-language tracking. Disposition B
is structurally cleaner but pushes a near-term implementer pain point
through a second Work Group's ballot cycle, and the OAuth2-`sub`
guidance is genuinely PDex-flavoured — it belongs in PDex even if HRex
later strengthens the profile-side cardinality. Disposition C reads as
a procedural refusal to fix prose that is, by the reporter's own
account, already misleading implementations. The cleanest follow-up to
A is to file a *non-blocking* note against HRex suggesting it consider
making `MemberId` Must Support in the next HRex cycle — but that should
not gate this PDex fix.
