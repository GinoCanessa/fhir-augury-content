## Summary

The reporter (ZacharyZemanovic) flags that `PaymentReconciliation.method`
is bound to the HL7 v2 value set `hl7VS-paymentMethodCode`
(v2 table 0570), and that this value set does not include a `NON` code.
In current U.S. industry practice (notably the X12 835 transaction's
BPR04 element), `NON` is routinely used to convey a zero-dollar /
"non-payment" remittance — e.g., a remittance advice with no
accompanying funds transfer. Because there is no equivalent code in
the bound value set, implementers conveying $0 remittances via
`PaymentReconciliation` cannot represent the payment method faithfully
within the current binding.

## Proposed Dispositions

> Note: every disposition below touches terminology that lives in
> repos for which no `repo-analysis/briefing.md` was loaded. The
> implementing workgroup should re-validate the specific authoring
> roots and gotchas (especially around v2 table editorial ownership
> and THO version pinning) before committing to a path.

### Disposition A: Accept As Requested

#### Proposal

Coordinate with the Vocabulary / v2 management group to add a `NON`
concept (display: "Non-Payment Data" or equivalent X12-aligned wording;
definition: "Remittance advice with no associated payment, e.g., a
zero-dollar (`$0`) remittance") to HL7 v2 table 0570 and republish
`hl7VS-paymentMethodCode` in a future THO release. Once published,
re-pin (or rely on the unversioned canonical of)
`PaymentReconciliation.method` so implementers automatically gain
access to the new code. No structural change to
`PaymentReconciliation` itself is required; the change is purely
terminological.

#### Justification

The reporter's underlying need — distinguishing a $0 / informational
remittance from a real payment — is real and routinely encountered in
U.S. revenue-cycle workflows that mirror X12 835 BPR04. Keeping the
existing binding to `hl7VS-paymentMethodCode` preserves alignment with
the v2-to-FHIR mapping work and avoids fragmenting payment-method
vocabulary across HL7 standards. The downside is editorial: v2 table
0570 is not owned by Financial Management, so this disposition
requires a Vocab/INM hand-off and will land on a THO release cadence
rather than the next FHIR ballot. FHIR-56268 also illustrates that
consumers of this value set are sensitive to the version pin, so any
republish should be coordinated with downstream IGs (NDH and others)
to avoid the "binding pinned to 2.0 vs IG declares 7.0.1" trap.

---

### Disposition B: Alternative Approach

#### Proposal

Leave v2 table 0570 alone and instead change
`PaymentReconciliation.method` to bind to a small FHIR-managed,
*extensible* value set that unions `hl7VS-paymentMethodCode` with the
specific X12-835 BPR04 codes the industry actually needs — minimally a
`NON` (Non-Payment Data) concept, sourced either from a new
HL7-managed CodeSystem (e.g.,
`http://hl7.org/fhir/CodeSystem/payment-method-supplement`) or directly
from the X12 BPR04 code system if licensing allows. Confirm the
binding strength is `extensible` (and weaken from `required` if it is
not already) so implementers can supply `NON` from the supplemental
system without breaking conformance for systems that still emit only
v2-0570 codes.

#### Justification

This decouples the FM workgroup's release cadence from the v2 table
editorial process and lets FM ship a fix in the next FHIR Core ballot
without waiting for a THO publication. It also reflects the reality
that `PaymentReconciliation` is being used to convey X12-835 semantics
and that the v2 payment-method code list pre-dates the
zero-dollar-remittance use case. The trade-off is fragmentation: a
second value set/code system to maintain, and the burden of explaining
to implementers when to use `hl7VS-paymentMethodCode` codes vs the
supplemental `NON` code. If the binding is already `extensible`, this
disposition collapses to "publish the supplemental code system and
update guidance" — a strictly additive change.

---

### Disposition C: Decline

#### Proposal

Decline the request as already addressable through the existing
binding mechanism. Update the FHIR specification text on
`PaymentReconciliation.method` (or add a usage note) clarifying that
the binding is extensible and that implementers needing to convey
X12-835 BPR04 codes such as `NON` for zero-dollar remittances should
supply the appropriate X12 code via the extensible binding, rather
than relying on a v2-0570 equivalent. Mark the resolution as
"Persuasive with Mod" or "Not Persuasive" depending on whether the
guidance text is added.

#### Justification

If `PaymentReconciliation.method` is in fact bound `extensible` (the
typical FHIR pattern for cross-walked v2 value sets), implementers
already have the latitude to supply `NON` from the X12 system without
any spec change. Declining avoids touching v2 table 0570 (out of FM's
editorial control) and avoids creating a parallel HL7 code system that
duplicates X12 content. The risk is that implementers (and validators
configured against the bound value set) treat the v2 list as
exhaustive — exactly the assumption that drove this ticket — so a pure
decline without a documentation update is likely to surface the same
complaint again. This disposition is weakest if the binding turns out
to be `required`, in which case it is not actually viable and the
workgroup should fall back to A or B.

---

### Recommendation

**Recommended disposition:** B

The reporter is describing a real, common U.S. industry workflow — $0
remittance advice — that the current binding does not let implementers
express cleanly. Disposition A is correct in principle but puts FM on
the v2/THO editorial critical path and bakes in the version-pin
fragility that FHIR-56268 already illustrates for this same value set.
Disposition C is only viable if the binding is already extensible *and*
the workgroup is willing to settle for a documentation note; even
then, validator behaviour against the named value set will keep
producing the same friction. Disposition B threads the needle: FM
keeps editorial control, ships in a normal FHIR ballot cycle, and gives
implementers a sanctioned, discoverable code for `NON` while leaving
the v2-0570 codes intact for everyone who already uses them. The
workgroup should confirm the current binding strength on
`PaymentReconciliation.method` before finalising — if it is already
extensible, B reduces to publishing the supplemental code and a usage
note, which is a low-risk additive change.
