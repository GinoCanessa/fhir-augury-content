## Summary

The reporter (Celine Lefebvre) is alerting the CARIN Blue Button work group
that the American Medical Association (AMA) is revamping the way CPT codes
are distributed and licensed to developers, and that a new link will be
available for that program. The ticket itself does not request a specific
spec change — it is an informational heads-up so that any URL or licensing
guidance the CARIN BB IG points implementers to (for obtaining CPT codes /
the CPT license required to publish CPT-coded data) can be refreshed when
the AMA's new landing page is live. No cross-referenced Jira tickets,
Zulip discussions, or GitHub items were surfaced, so the scope is
limited to whatever CPT-related references exist in the CARIN BB IG
narrative (terminology / licensing pages).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a narrative-only technical correction in the CARIN
Consumer-Directed Payer Data Exchange (CARIN BB) IG. In the next
republication, audit every place the IG references the AMA / CPT —
typically the terminology, "external code systems", or licensing
sections that accompany the CPT code system (`http://www.ama-assn.org/go/cpt`)
and any narrative on how implementers obtain a CPT license — and replace
the legacy URL(s) with the AMA's new developer-distribution landing
page. No StructureDefinition, ValueSet, or CodeSystem resource changes
are required; only narrative pages (and any informative links in
profile descriptions) are touched. Confirm the canonical CPT system URI
itself is unchanged.

#### Justification

The ticket is explicitly informational ("we just wanted to note…") and
the early-2022 comment thread shows the WG already accepted it as a
link / narrative fix ("done. thanks"). The 2022-11-28 note from
Lynn Laakso to "re-open post STU2 publication" makes this exactly the
kind of housekeeping change that should be batched into the next CARIN
BB technical-correction cycle, since the AMA's new developer program
will have stabilised by then. Keeping the canonical CPT system URI
unchanged preserves backward compatibility with all existing
CPT-coded EOB / Claim instances in the wild.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) hard-coding a new AMA URL into the IG
narrative, generalise the CPT licensing guidance: replace the specific
link with a short paragraph that (1) names the AMA as the licensor,
(2) states that implementers must obtain a current CPT distribution
license before publishing CPT-coded data via CARIN BB, and (3) points
to a single "for current AMA developer program details, see ama-assn.org"
soft reference rather than a deep link that is liable to rot again.
Optionally add a small "External terminology licensing" section that
covers CPT, HCPCS, ICD, and other licensed code systems uniformly so
future AMA / CMS / WHO URL changes only need to be touched in one
place.

#### Justification

The underlying problem the reporter is flagging — AMA changing how it
distributes CPT — is recurring; this is at least the second time the
licensing path has shifted during the IG's lifetime. A
single-source-of-truth section reduces the maintenance burden on the FM
WG, lowers the chance of broken links between technical-correction
cycles, and matches how other HL7 IGs handle licensed code systems.
Trade-off vs. Disposition A: implementers lose a one-click deep link
into the AMA's developer flow, but they gain a stable narrative that
will not need another ticket the next time the AMA rebrands the
program.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Modification — already addressed"
on the basis that (a) the in-comment record shows an edit was made in
February 2022 ("done. thanks"), (b) no concrete broken link or
incorrect URL has been supplied since, and (c) no cross-referenced
Zulip or GitHub activity has accumulated against it in the intervening
years. Any future AMA URL change can be filed as a fresh, specific
technical correction with the new URL in hand.

#### Justification

The ticket is, as filed, a forward-looking heads-up rather than a
defect report. With no current evidence of a stale link in the
published IG and no community follow-up after the 2022-11-28 "re-open
post STU2" note, holding the ticket open indefinitely adds backlog
noise without giving the WG a concrete change to make. Declining (or
closing as "Already Addressed") is defensible because the original
correction was applied and any new AMA URL change is better tracked as
its own actionable ticket.

---

### Recommendation

**Recommended disposition:** A

The ticket has been sitting "Triaged / future" since 2022 specifically
because Lynn Laakso asked for it to be revisited after STU2 publication —
that condition is now satisfied, so the right move is to do exactly what
the reporter asked: sweep the IG narrative for AMA / CPT references and
refresh them against the AMA's current developer-distribution page. It
is a low-risk, narrative-only edit, it honours the reporter's intent
and the WG's earlier acceptance, and it clears a long-lived "future"
ticket. Disposition B is appealing as a follow-up hardening step but
should not block this fix; Disposition C understates the open
commitment Lynn Laakso recorded in 2022 to come back to this after
STU2.
