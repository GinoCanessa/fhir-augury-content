## Summary

Lloyd McKenzie (the submitter) argues that the `Claim` resource has no
clean way to point at a `MedicationDispense` as the actual billable
event. Today the only path is `Claim.supportingInfo`, but a dispense is
the primary action being billed (analogous to `Procedure`), not
"supporting information," and none of the existing
`supportingInfo.category` example codes are a natural fit. The submitter
proposes a structural rework of how `Claim.item` links to authorizations
and to clinical events:

1. **Replace `prescription` and `referral` with a single generic
   `request` reference** that can point to any authorising request
   resource ã `ServiceRequest`, `MedicationRequest`,
   `VisionPrescription`, `DeviceRequest`, `RequestOrchestration`
   (originally written as "OrchestrationGroup"), etc. ã and **drop
   `originalPrescription`**, relying on `Request.basedOn` chaining
   instead, since "original" is observer-dependent.
2. **Replace `Claim.item.procedure` with a generic `eventReference`**
   that can point to any billable event resource ã `Procedure`,
   `MedicationDispense`, `MedicationAdministration`, `DeviceDispense`,
   `Communication`, `ImagingStudy`, `NutritionIntake`, `SupplyDelivery`,
   `Transport`, etc.

The Financial Mgmt review (pknapp) was largely persuaded on the request
side: rename `prescription` ã `request` with the expanded reference set
including `RequestOrchestration` and `NutritionOrder`, add `basedOn` to
`VisionPrescription`, and remove `originalPrescription`. On the event
side FM was more cautious ã recommending that `Procedure` stay in place
and that the broader "billable event" idea be explored as a separate new
resource, with a follow-up suggestion to instead add an
`eventDetails 0..* Reference(MedicationDispense + ãÎ)` element at the
`.item`, `.detail`, and `.subDetail` levels. The ticket has been
dispositioned **Persuasive with Modification** but is still in **Triaged**
status pending the concrete spec edits.

There were no cross-referenced Jira tickets, Zulip threads, or GitHub
items returned for FHIR-43308; the only cross-reference detected is the
FHIR artifact path `Claim.item.request` mentioned inside the discussion
itself.

## Details

**Description:**

> Right now, the only way to reference a MedicationDispense is via
> 'supportingInformation', but it's no more supporting information than
> Procedure is ã it's the primary action. None of the supportingInformation
> categories really make sense. While these are example and I can invent a
> new code, I think the proper solution is to re-jig the resource.
>
> Specifically, notions like prescription and referral are all the same
> thing. There should be a generic 0..* 'Request' resource that can point
> to ServiceRequest, MedicationRequest, VisionPrescription, DeviceRequest,
> OrchestrationGroup and any other Request resource that can realistically
> act as an 'authorization'. This would replace 'prescription' and
> 'referral'. You might want an 'originalRequest' as well, but my
> recommendation would be to ditch that and rely on the chaining of
> 'basedOn' for the requests ã because what constitutes 'original' might
> vary depending on who's asking.
>
> And then replace Procedure with eventReference which could be Procedure,
> DeviceDispense, Communication, ImagingStudy, MedicationAdministration,
> MedicationDispense, NutritionIntake, SupplyDelivery and Transport (and
> any other 'event' resources you think might be reasonable billable
> actions) so that the detail can be pointed to.

**Resolution Description (Persuasive with Modification):**

- *On reusing supportingInfo for MedicationDispense:* The proposed path
  forward is to allow MedicationDispense to be referenced at the
  line-item level via an extension; the open question is whether to
  define a formal extension for it.
- *On the generic `request` element:* Adopt in modified form. Concretely:
  (1) rename `Claim.item.prescription` to `Claim.item.request`;
  (2) change the allowed reference targets to
  `DeviceRequest | MedicationRequest | ServiceRequest |
  VisionPrescription | RequestOrchestration | NutritionOrder`;
  (3) add `basedOn 0..* Reference(Any)` to `VisionPrescription` so
  chaining works across all request types; and
  (4) remove `Claim.item.originalPrescription`, since that role is now
  covered by walking `request.basedOn`.
- *On the generic `eventReference`:* Keep `Claim.item.procedure` as-is
  for now and consider creating a new dedicated resource that captures
  the broader notion of "billable event" with a wider community review
  before changing Claim.

  (b) noted that `Claim.item.request 0..* Reference(...)` already exists

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure `Claim.item` (and mirror at `.detail` / `.subDetail`) per
Lloyd's original proposal, in two coordinated changes to the FHIR Core
`Claim` resource:

1. **Replace authorisation references with a single generic element.**
   - Remove `Claim.item.prescription`, `Claim.item.originalPrescription`,
     and `Claim.item.referral` (and their `.detail` / `.subDetail`
     analogues if present).
   - Add `Claim.item.request 0..* Reference(DeviceRequest |
     MedicationRequest | NutritionOrder | ServiceRequest | SupplyRequest
     | VisionPrescription | RequestOrchestration)`.
   - Add `VisionPrescription.basedOn 0..* Reference(Any)` so chains of
     `basedOn` work uniformly for every authorising request type.
   - Document that "original request" is reconstructed by walking
     `request.basedOn`, and that what counts as "original" is
     observer-dependent.
2. **Replace event references with a single generic element.**
   - Remove `Claim.item.procedure` and its `.detail` / `.subDetail`
     analogues.
   - Add `Claim.item.event 0..* Reference(Procedure | MedicationDispense
     | MedicationAdministration | DeviceDispense | Communication |
     ImagingStudy | NutritionIntake | SupplyDelivery | Transport)`,
     mirrored at `.detail` and `.subDetail`.

#### Justification

This is the cleanest model and directly addresses the original concern:
a `MedicationDispense` becomes a first-class billable event, exactly
parallel to `Procedure`, with no extension gymnastics. It also aligns
`Claim` with the `request`/`event` framing already used elsewhere in
FHIR (the Workflow pattern), which improves consistency for implementers
and makes future event types trivial to add. Dropping
`originalPrescription` removes a long-standing conceptual wart that
forces senders to pick a "canonical" original even when none exists.

### Disposition B: Alternative Approach (Persuasive with Modification ã current FM direction)

#### Proposal

Adopt the Financial Mgmt resolution as written, in two
backward-friendlier passes:

1. **Authorisation side (renaming + reference-list expansion):**
   - Rename `Claim.item.prescription` ã `Claim.item.request`.
   - Change the allowed targets of `Claim.item.request` to
     `DeviceRequest | MedicationRequest | ServiceRequest |
     VisionPrescription | RequestOrchestration | NutritionOrder`.
   - Add `VisionPrescription.basedOn 0..* Reference(Any)` so the
     `basedOn` chain is usable across all request types.
   - Remove `Claim.item.originalPrescription`, because the same
     information is now derivable by walking `request.basedOn`.
2. **Event side (additive, no rename of `procedure`):**
   - Leave `Claim.item.procedure` in place to avoid breaking existing
     pharmacy/medical claim implementations.
   - Add a new element `Claim.item.eventDetails 0..* Reference(
     MedicationDispense | ãÎ additional event resources TBD by FM)` at
     `Claim.item`, `Claim.item.detail`, and `Claim.item.subDetail`, per
     pknapp's 2025-04-01 comment.
   - Defer the broader "generic billable event" model to a separate
     ticket / a possibly-new resource, with a wider community review
     before any further reshaping of `Claim`.
3. **Interim path for early adopters:** publish a formal HL7 extension
   for `MedicationDispense` references on `Claim.item` so that current
   implementers can converge on a shared shape before the rename ships
   in a future release.

#### Justification

This is what the ticket's own Resolution Description and pknapp's
follow-up comment have effectively converged on. It solves the original
problem ã `MedicationDispense` becomes a first-class line-item event
target via `eventDetails` ã without having to rename `procedure` or
relitigate the breadth of "what counts as a billable event" for every
existing pharmacy/medical claim implementation. Renaming
`prescription` ã `request` and dropping `originalPrescription` brings
the authorisation side into line with the Workflow pattern with minimal
surprise for senders, and adding `VisionPrescription.basedOn` plugs the
one gap that would otherwise prevent uniform `basedOn` chaining. The
formal extension gives implementers a way to converge before the next
normative-impact release without forcing the rename early.

### Disposition C: Decline

#### Proposal

Decline the structural rework. Instead, address the immediate need by
**defining (and registering) a `MedicationDispense`-typed reference via
an HL7-published extension on `Claim.item` / `.detail` / `.subDetail`,
plus a new example code in the `Claim.supportingInfo.category` value set
that explicitly covers "dispense as billed event."** Keep
`prescription`, `originalPrescription`, and `procedure` unchanged.

#### Justification

`Claim.item.request` already exists and lists the realistic
authorisation request resources, and pknapp's first review noted that
claims rarely need the full clinical content of a `MedicationDispense` ã
typically only the drug code, quantity, days-supply, and timing, all of
which are already representable on the line item. Dropping
`originalPrescription` and renaming `prescription` are breaking changes
that ripple into downstream IGs (CARIN, Da Vinci, national pharmacy
claim guides) for limited net benefit if the underlying need can be met
with an extension plus a clearer example category code. A "decline +
extension" path also matches the resolution text's own first option
("the way to reference a MedicationDispense at the line item level
would be to use an extension").

### Recommendation

**Recommended disposition:** **B (Alternative Approach ã adopt the FM
resolution as written).**

This is the disposition the ticket has already partially landed on
(`Persuasive with Modification`), and it threads the needle between the
two real concerns. The submitter's underlying request ã make
`MedicationDispense` a first-class line-item link ã is solved by the
new `eventDetails` element at `.item` / `.detail` / `.subDetail`. The
authorisation half (`prescription` ã `request`, drop
`originalPrescription`, add `VisionPrescription.basedOn`) tidies an
inconsistency that has bothered reviewers for releases without
introducing a brand-new mental model. Crucially, it leaves
`Claim.item.procedure` intact, so the much larger debate about what
qualifies as a "billable event" can happen on its own ticket / future
resource without delaying the dispense-linkage fix that motivated this
ticket. Disposition A is conceptually cleanest but trades a real,
shippable change for a much wider impact on existing pharmacy and
medical claim implementations and re-opens scope FM has explicitly
deferred. Disposition C is defensible but contradicts the FM team's
own review and leaves Lloyd's original semantic complaint unresolved
(an extension still says "this is supporting info"). FM should
proceed with B; the workgroup may want to confirm exactly which event
resources go into the initial `eventDetails` reference set before
balloting.
