## Summary

The ticket asks the Financial Management work group to clarify the
definitions used in the **Contract Term Subtype Codes** value set. The
codes in that value set are described using the contract-law phrases
"go to the very root of a contract" and "go to the root of a contract"
without any in-spec explanation of what "going to the root" means. The
reporter (Wes Rishel) argues that, when a reader encounters the value
set as part of the FHIR specification, those phrases are not
self-explanatory — for example, it is not clear whether "going to the
root" means a breach voids the contract, nor what distinguishes the
plain "root" from the "very root" variant. The reporter requests that
either the definitions be rewritten in plain, self-explanatory language
or that supporting explanatory text be added so a non-lawyer
implementer can understand the codes' intent.

The ticket is the sibling of [FHIR-13053](https://jira.hl7.org/browse/FHIR-13053)
(Clarifications on Value Set Contract Signer Type Codes), filed by the
same reporter on the same day, which raises the same class of concern
— that several Contract-related value sets reuse legal jargon without
adapting it to FHIR's implementer audience. Both are labelled
`FM_R5_Contract`, indicating Financial Management intends to address
them as part of a Contract resource refresh. The ticket has been in
"Triaged" status since 2017 and remains Unresolved.

## Details

**Description:**

> The terms "go to the very root of a contract" and "go to the root of
> a contract" are used here without any explanation of what going to
> the root means. Unless this is clarified in related text when the
> value set is viewed as part of the complete FHIR document the term
> should be explained or replaced with a self-explanatory term.
>
> For example, does going to the root mean something that, if it is
> not true, voids the contract? What is the difference between going
> to the root and going to the *very* root?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rewrite the code definitions in `ValueSet/contract-term-subtype` (and
the underlying `CodeSystem/contract-term-subtype`, currently three
codes — `condition`, `warranty`, `innominate`) so each definition is
self-contained and free of unexplained "go to the root" jargon. For
example, replace the definitions with plain-language wording such as:

- **condition** — "A fundamental term of the contract; if it is
  breached the wronged party is entitled to treat the contract as
  repudiated (historically described in common law as a term that
  'goes to the root of the contract')."
- **warranty** — "A subsidiary term of the contract; breach entitles
  the wronged party to damages but not to treat the contract as
  repudiated."
- **innominate** — "A term whose remedy on breach (damages versus
  repudiation) depends on the seriousness of the actual breach rather
  than being fixed in advance."

The original phrases can be retained parenthetically as "historical
common-law usage" so existing readers are not disoriented. No code
identifiers are changed; only the human-readable `definition`
elements are modified, so this is a non-breaking change at the
terminology binding level.

#### Justification

FHIR's terminology design guidance is that code system definitions be
intelligible to implementers, who are predominantly software engineers
rather than contract lawyers. The reporter's complaint is precisely
that the existing wording is opaque outside the contract-law
community, which matches that guidance. Because the change is to
`definition` text only (not to `code`, URI, or system), it is a
documentation-grade, non-breaking change suitable for an R5/R6
Contract refresh — consistent with the `FM_R5_Contract` label
Financial Management has applied to this ticket and to its sibling
FHIR-13053.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing code definitions essentially as-is, but add a
single explanatory paragraph to the value set's narrative
(`ValueSet.text` and the rendered `valueset-contract-term-subtype.html`
page) that introduces the common-law "goes to the root of the
contract" concept once, in plain language, and then notes that the
three codes (`condition`, `warranty`, `innominate`) reflect the
classical English-law tripartite classification of contract terms.
Optionally, link to that explanation from each code's `designation`
or from a `display` annotation. This addresses the reporter's
"explained or replaced" request via the "explained" branch.

#### Justification

The "go to the root" wording is anchored in long-established contract
law and is recognisable to legal reviewers; rewriting it loses that
provenance. Hoisting the explanation up to the value set narrative
preserves the canonical legal language for legal reviewers while
giving software implementers a one-time orientation. This is also a
narrower change with less risk of subtly altering the legal meaning
the existing wording is meant to convey.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **No Action** on the grounds that (a) the existing
code definitions, while legalistic, are accurate and well-established
contract-law terms, and (b) the `Contract` resource and its supporting
value sets have undergone substantial restructuring since the ticket
was filed in 2017, so the specific wording the reporter quoted may no
longer be the dominant point of confusion. Implementers needing
deeper guidance can consult standard references on contract law.

#### Justification

The Contract resource is currently maturity level 1 / Trial Use and
has limited deployed implementations using `Contract.term.subType`;
investing review cycles in re-wording a low-traffic value set may not
be the best use of Financial Management's R5/R6 effort. The "goes to
the root" phrasing, while jargon, is not factually incorrect.

---

### Recommendation

**Recommended disposition:** A

The reporter's complaint is well-founded and squarely within FHIR's
own terminology-authoring guidance: code definitions should be
intelligible to FHIR implementers without requiring outside legal
training. Disposition A is a low-risk, non-breaking documentation
change (only `definition` text moves; codes, URIs, and bindings are
untouched), it directly answers the reporter's "explained or
replaced" ask, and it dovetails with the `FM_R5_Contract` work
already implied by the label and by the parallel sibling ticket
FHIR-13053. Disposition B is a reasonable compromise but leaves the
opaque wording on each code's primary `definition`, which is what
most tooling surfaces to end users — so it only partially solves the
reported problem. Disposition C is hard to defend given the ticket
has sat Triaged for eight years; the more honest outcome is either
to fix the wording or to formally state the value set is being
deprecated in the next release. Financial Management should adopt
A, optionally combining it with B's narrative paragraph for legal
reviewers.
