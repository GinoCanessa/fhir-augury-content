## Summary

The reporter observes that the FHIR `Invoice` resource currently models
only the *charges* side of a billable encounter — its `lineItem`
elements reference `ChargeItem` (or inline `chargeItemCodeableConcept`)
plus `priceComponent` adjustments local to a line — but provides no
first-class way to attach *payments* or post-adjudication
*adjustments* applied against the invoice. They cite the Account
resource's narrative (section 8.12.2, "Payment and adjustment
resources have not yet been developed.") as evidence that this gap is
already acknowledged in the spec, and they request that `Invoice` be
extended (or that companion resources be made referenceable from
`Invoice`) so a single invoice can carry the full
charges + payments + adjustments picture rather than only the charges
view.

No related Jira tickets, Zulip discussions, or GitHub items were
surfaced by the cross-reference, search, or content-search calls
against `fhir-augury-cli`, so the ticket is being reviewed purely on
its own merits and against the existing spec.

## Details

**Description:**

> The Invoice content contains references to ChargeItems but not to
> Payments or Adjustments. In 8.12.2 of the Account spec, it mentions
> that "Payment and adjustment resources have not yet been developed."
> We would expect an option for the Invoice to contain
> payment/adjustment information as well as charges.

## Proposed Dispositions

> Note: because no GitHub repositories were surfaced as related, the
> proposals below name the canonical FHIR Core authoring areas
> (`source/invoice/`, `source/account/`, `source/paymentnotice/`,
> `source/paymentreconciliation/`) by convention rather than by an
> authoritative briefing. A reviewer should confirm those paths against
> the working `HL7/fhir` clone before acting on a disposition.

### Disposition A: Accept As Requested

#### Proposal

Extend the `Invoice` resource so a single instance can carry payments
and adjustments alongside its existing charges. Concretely:

1. Add `Invoice.payment` (0..*, BackboneElement) with at minimum:
   - `payment.reference` (Reference(`PaymentNotice` |
     `PaymentReconciliation`)) — link to the financial event,
   - `payment.amount` (Money) — amount applied against this invoice,
   - `payment.date` (dateTime),
   - `payment.type` (CodeableConcept, bound preferred to a new value
     set `InvoicePaymentType` covering patient-pay, payer-remit,
     refund, write-off, etc.).
2. Add `Invoice.adjustment` (0..*, BackboneElement) with:
   - `adjustment.type` (CodeableConcept; reuse / extend
     `AdjudicationCodes` where possible),
   - `adjustment.amount` (Money, may be negative),
   - `adjustment.reason` (CodeableConcept),
   - `adjustment.date` (dateTime).
3. Recompute `Invoice.totalNet` / `Invoice.totalGross` semantics so the
   resource also exposes an `Invoice.balance` (Money, 0..1) =
   totalGross − Σpayment.amount − Σadjustment.amount, and document
   that derivation in the resource narrative.
4. Update Account §8.12.2 to remove the "have not yet been developed"
   statement and instead point at `PaymentNotice` /
   `PaymentReconciliation` plus the new `Invoice.payment` /
   `Invoice.adjustment` slots.

Authoring touch points (FHIR Core convention): `source/invoice/`
(structure + introduction + examples), `source/account/account.html`
(narrative §8.12.2), `source/valueset/` for any new value sets, and
the financial module index.

#### Justification

This is the most direct interpretation of the request and removes a
self-acknowledged gap in the spec. Putting the payment/adjustment
edges *on* `Invoice` keeps the "what does this invoice still owe?"
question answerable from a single resource, which is how most billing
systems present statements to end users. It also replaces the dangling
text in the Account spec, which has been a known TODO across multiple
ballot cycles. Because `PaymentNotice` and `PaymentReconciliation` do
exist today, the new elements are mostly *links* and a balance roll-up
— the Financial module already carries the underlying primitives.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the `Invoice` resource shape. Instead, address the
underlying need by:

1. Adding (or formalising) reverse-pointing search parameters and
   narrative guidance so payments and adjustments *find their way back
   to* the invoice rather than being embedded in it:
   - `PaymentNotice.request` and `PaymentNotice.response` already
     accept `Reference(Any)`; constrain the documentation to recommend
     `Invoice` as a target, and add a `payment-for-invoice` search
     parameter on `PaymentNotice` keyed off `request`.
   - `PaymentReconciliation.allocation.target` already accepts
     `Reference(Any)`; add explicit `Invoice` to the target type list
     and add an `invoice` search parameter on
     `PaymentReconciliation.allocation`.
2. Updating `Account` §8.12.2 to *replace* the
   "not-yet-developed" sentence with a cross-reference to
   `PaymentNotice` / `PaymentReconciliation` and to the new search
   parameters above, plus a worked example showing
   Account → Invoice → ChargeItem on the charges side and
   PaymentReconciliation.allocation.target → Invoice on the
   settlement side.
3. Optionally publishing a small companion **`InvoiceBalance`
   operation** (`$balance` on `Invoice`) that returns a computed
   net/paid/adjusted/balance Money set, so clients get the same
   single-resource answer Disposition A offers without changing the
   resource.

Authoring touch points (FHIR Core convention): `source/paymentnotice/`,
`source/paymentreconciliation/` (search parameter + type-list +
narrative), `source/account/account.html` (narrative §8.12.2),
`source/invoice/` (operation definition + narrative cross-links).

#### Justification

`Invoice` is meant to be a *bill* (a request for payment), not a
ledger. FHIR already has `PaymentNotice` (the act of paying) and
`PaymentReconciliation` (the allocation of payments against
billed items) — embedding the same concepts again on `Invoice` risks
two sources of truth for "what's been paid?" and creates ambiguity
when a single payment settles several invoices. The alternative
keeps each resource focused on its lifecycle stage, fixes the
discoverability problem with explicit search parameters, and lets the
Account spec retire its stale TODO. The downside vs. Disposition A is
that a consumer must traverse 1–2 references to compute a balance,
which the optional `$balance` operation mitigates.

---

### Disposition C: Decline

#### Proposal

Decline the change to `Invoice`. Update only the Account narrative
(§8.12.2) to remove the obsolete "Payment and adjustment resources
have not yet been developed." sentence and to point at the existing
`PaymentNotice` and `PaymentReconciliation` resources, with a worked
example. No structural change to `Invoice`, `PaymentNotice`, or
`PaymentReconciliation`.

#### Justification

The premise of the ticket — that payment and adjustment resources do
not exist — is no longer true; the cited Account-spec sentence is the
artefact of an earlier ballot cycle and the gap has been filled by
`PaymentNotice` and `PaymentReconciliation` (and adjudication-bearing
resources such as `ClaimResponse` / `ExplanationOfBenefit`). Adding
overlapping payment/adjustment slots on `Invoice` would be a
non-trivial breaking-shape change for a resource that several
implementers have already deployed, and would create two ways to say
the same thing. The right fix is editorial: correct the misleading
note in Account so readers are pointed at the existing financial
resources.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between honouring the reporter's
real concern — "I cannot get the full charges + payments + adjustments
picture from `Invoice` today" — and preserving the deliberate
separation-of-concerns that already exists between `Invoice`,
`PaymentNotice`, and `PaymentReconciliation`. It removes the
misleading Account §8.12.2 note (which is what triggered the report),
makes the existing payment/adjustment resources discoverable from an
`Invoice` via explicit search parameters, and offers an optional
`Invoice/$balance` operation so consumers that need a one-call answer
get one without forcing every server to carry duplicated payment
state. Disposition A solves the problem most fully but at the cost of
introducing a second, redundant home for payment data and a
breaking-shape change to a published resource; Disposition C is
defensible as a pure editorial fix but leaves the discoverability and
balance-roll-up problem unsolved. Given that no Zulip / GitHub
discussion has yet weighed in, B is the proposal most likely to
survive workgroup review without re-litigating prior Financial-module
design decisions.
