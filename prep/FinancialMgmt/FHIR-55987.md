## Summary

Michele Mottini reports that the Da Vinci PDex Provider Access specification
(STU 2.1) requires data providers to record bulk-export bookkeeping (latest
download date/time, the `_type` parameter, and the `_typeFilter` parameter) by
mutating extensions (`LastTransmission`, `LastTypes`, `LastFilters`) on the
exported Patient and Group resources. The reporter argues this is
problematic because (1) the same Patient or Group can participate in exports
from multiple clients, so per-Patient/Group extensions cannot honestly
represent "the last" anything; (2) different clients would observe each
other's parameter history, which is inappropriate; and (3) a read operation
(`$export`) is mandated to mutate the resources being read, which means a
client with read-only permission would have to be granted write permission
to comply. The proposal is to drop the SHALL requirements that store these
values as extensions and instead address any audit need with an internal
audit trail (server-side and/or client-side, e.g., AuditEvent), since the
bulk-export response already returns `transactionTime` for the client to
persist its own state. The ticket has been **resolved Persuasive** and is
labelled **Pre-Applied** — the change has already been implemented in the
PDex IG (commit `96e9f2ec`, "Resolve FHIR-55987: remove export tracking
extensions from Patient/Group") for the v2.2.0 publication.

## Details

**Description:**

The current specifications call for various details of the executed bulk
exports to be recorded as extensions on the export Patient and Group:

> From the Data Provider's perspective the
> `hl7.fhir.us.davinci-pdex#provider-download` exportType parameter will
> require the Data Provider/Payer to track the latest download date/time
> for the Patients that the provider requests data for. These values
> **SHALL** be updated in an extension associated with the Patient for
> which a download was requested.

> When `_type` is used the export operation **SHALL** record the content
> of the `_type` parameter in the
> [`lastResources`](https://hl7.org/fhir/us/davinci-pdex/STU2.1/StructureDefinition-base-ext-last-types.html)
> element for each Member for which data is retrieved. The
> [`lastTransmitted`](https://hl7.org/fhir/us/davinci-pdex/STU2.1/StructureDefinition-base-ext-last-transmission.html)
> **SHALL** be recorded with either the Date/Time of the Export
> Transaction or the value of the `_until` parameter, if it is earlier.

> When `_typeFilter` is used the export operation **SHALL** record the
> content of the `_typeFilter` parameter in the
> [`lastFilters`](https://hl7.org/fhir/us/davinci-pdex/STU2.1/StructureDefinition-base-ext-last-typefilter.html)
> element for each Member for which data is retrieved. The
> [`lastTransmitted`](https://hl7.org/fhir/us/davinci-pdex/STU2.1/StructureDefinition-base-ext-last-transmission.html)
> **SHALL** be recorded with either the Date/Time of the Export
> Transaction or the value of the `_until` parameter, if it is earlier.

This is problematic:

- the same Patient be used by different export operations from different
  clients,
- the same Group can be used by different export operations,
- a client with only read permission would end up modifying resources.

If the requirement is that the server (payer) must record which export
operations were executed and by which client, that can be better done with
an internal audit trail. If the requirement is that the client must have
such a record, that can again be done with an internal audit trail.

**Resolution (already applied — `Pre-Applied` label):**

- **Extensions.fsh** — Removed three extension definitions
  (`LastTransmission` / `base-ext-last-transmission`, `LastTypes` /
  `base-ext-last-types`, `LastFilters` / `base-ext-last-typefilter`)
  and replaced them with an inline comment explaining the rationale and
  citing FHIR-55987.
- **ProfilePDexProviderGroup.fsh** — Removed the `member.extension`
  slice block that bound the three extensions; updated the profile
  description.
- **ExamplePDexProviderGroup.fsh** — Removed the three extension value
  lines from the example member.
- **provider-access-api-v2.md** — Removed §pdex-269 (SHALL update
  extension on `provider-download`), §pdex-276/277 (SHALL record
  `_type` in `lastResources`/`lastTransmitted`), and §pdex-280/281
  (SHALL record `_typeFilter` in `lastFilters`/`lastTransmitted`);
  replaced the `provider-download` paragraph with a SHOULD audit-trail
  recommendation.
- **provider-access-api.md** — Same treatment for the parallel
  requirements (§pdex-313, §pdex-329, §pdex-336/337, §pdex-340/341);
  replaced the entire extension-description block with a single SHOULD
  audit-trail statement.

## Related Zulip Discussions

### [Da Vinci PDex > Provider access - record bulk export latest download time](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PDex/topic/Provider%20access%20-%20record%20bulk%20export%20latest%20download%20time/near/576801067)

Michele Mottini opened the topic asking which extension is meant by the
SHALL requirement to record the latest download date/time on the exported
Patient and noting two structural problems: a Patient resource may be
exported to multiple providers (so a single per-Patient extension cannot
honestly represent "the last download") and an operation that reads data
should not mutate the data being read. Cooper Thompson agreed it was
inappropriate and asked for a Jira; Michele questioned what to actually
propose, suggesting either an internal audit system (if only the payer
needs the data) or AuditEvent (if clients need it). Cooper countered that
clients can keep their own audit, observed that the current spec language
is effectively "the server tells the client what *some other* client did
in the past," and judged that inappropriate. Paul Church speculated the
intent might have been to give clients a durable hint for the next
`_since` value; Michele replied that the bulk-export response already
returns `transactionTime`, which the client can persist itself. Consensus
landed on removing the SHALL and pointing to audit instead.

Participants: Michele Mottini, Cooper Thompson, Paul Church.

### [Da Vinci PDex > Provider access - record bulk export resources and filters](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PDex/topic/Provider%20access%20-%20record%20bulk%20export%20resources%20and%20filters/near/576801079)

The companion topic raised the same concern about the SHALL requirements
that the export operation record `_type` in `lastResources` and
`_typeFilter` in `lastFilters`, with `lastTransmitted` updated on the
exported Patient/Group: an operation reading data is being required to
mutate it, and that seems wrong. Cooper Thompson agreed, pointing out
that a Group can be shared across many requestors but `_type` and the
specific request are caller-specific, so two systems doing different
`_type` queries would clobber each other's recorded values. The thread
ended with both participants agreeing a Jira was warranted and Cooper
offering to file it if Michele didn't; Michele subsequently filed
FHIR-55987 covering both topics.

Participants: Michele Mottini, Cooper Thompson.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change exactly as already applied in commit
[`96e9f2ec`](https://github.com/HL7/davinci-epdx/commit/96e9f2ecea24ac823ea8633af823304360994d26)
on `HL7/davinci-epdx` for v2.2.0:

- Delete the three extensions `LastTransmission`
  (`base-ext-last-transmission`), `LastTypes` (`base-ext-last-types`), and
  `LastFilters` (`base-ext-last-typefilter`) from
  `input/fsh/Extensions.fsh`, replacing them with a comment that cites
  FHIR-55987.
- Remove the `member.extension` slice block that bound those extensions
  from `input/fsh/ProfilePDexProviderGroup.fsh`, and update the profile
  description accordingly.
- Strip the corresponding extension value lines from the example member
  in `input/fsh/ExamplePDexProviderGroup.fsh` (and the example marker
  in `input/fsh/ExampleExtensions.fsh`).
- In `input/pagecontent/provider-access-api-v2.md`, drop §pdex-269,
  §pdex-276/277 and §pdex-280/281, replacing the `provider-download`
  paragraph with a single SHOULD audit-trail recommendation.
- In `input/pagecontent/provider-access-api.md`, give §pdex-313,
  §pdex-329, §pdex-336/337 and §pdex-340/341 the same treatment, with a
  single SHOULD audit-trail statement replacing the extension block.
- Record the change in `input/pagecontent/changehistory.md` under v2.2.0.

#### Justification

The reporter's three objections are all structurally sound: (1) Patient
and Group resources can be exported to multiple providers, so storing
"the last" parameter values per resource cannot honestly represent
multiple concurrent consumers — Cooper Thompson explicitly demonstrated
the multi-client clobber scenario on Zulip; (2) requiring `$export` (a
read operation) to write back into the exported resources forces a
read-only client to be granted write permission, which is a real
deployment problem; and (3) the bulk-export response already returns
`transactionTime`, giving any client that genuinely needs to track
"when did I last fetch this" a self-managed mechanism that does not
require server-side per-Patient state. The Zulip discussion converged on
this disposition with no dissent, and the change has already been
applied (`Pre-Applied` label), giving reviewers concrete code to
inspect. Per the briefing, all touched files live in the standard
`input/fsh/` and `input/pagecontent/` authoring roots, no US Core /
HRex derivation is affected, and no cross-IG coordination is needed —
the deleted extensions are PDex-local. Ratifying the applied change
requires only confirming it for v2.2.0 publication.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the removal of the SHALL conformance language and the
extension-based mechanism, but tighten the replacement narrative beyond
"a SHOULD audit-trail recommendation":

- Keep the deletions in `Extensions.fsh`, `ProfilePDexProviderGroup.fsh`
  and `ExamplePDexProviderGroup.fsh` exactly as applied.
- Replace the SHOULD audit paragraphs in `provider-access-api.md` and
  `provider-access-api-v2.md` with a normative reference to FHIR R4
  `AuditEvent` (and/or the IHE-style `Provenance`-on-`$export` pattern
  already in use elsewhere in Da Vinci): name the resource, identify
  which fields capture the requestor (`AuditEvent.agent.who`), the
  operation (`AuditEvent.action = R`, `AuditEvent.entity.what` for the
  exported Group/Patient set), the parameters (`AuditEvent.entity.detail`
  for `_type`, `_typeFilter`, `_since`, `_until`), and the transmission
  time (`AuditEvent.recorded` aligned with the response's
  `transactionTime`).
- Optionally introduce a small PDex `AuditEvent` profile under
  `input/fsh/ProfilePDexProviderAccessAuditEvent.fsh` (per the
  briefing's "Add a profile (FSH)" recipe) and reference it from both
  narrative pages. Capability statements
  (`pdex-provider-access-server-capability-statement.fsh`) would be
  amended to advertise the `AuditEvent` resource as supported.

#### Justification

Disposition A removes a documented (if flawed) audit mechanism and
replaces it with a vague SHOULD, which risks silently dropping the
audit obligation rather than relocating it. Implementers who relied on
the extension-based bookkeeping to satisfy their own internal
governance now have no spec-level guidance on what "audit trail" means.
A normative `AuditEvent` mapping (or a thin PDex `AuditEvent` profile)
addresses Michele's underlying question on the Zulip thread ("use
`AuditEvent`?") without re-introducing the read-modifies-data anti-pattern
or the multi-client clobber problem. It also gives downstream payers a
testable target for connectathon. The trade-off is scope: this is a
larger change than the applied resolution and would need
`pdex-provider-access-server-capability-statement.fsh` updates, which
the v2.2.0 publication may not have time to absorb cleanly given the
flat-FSH layout flagged in the briefing. The triple US-Core
dependency does not affect this approach — `AuditEvent` is base FHIR R4
and does not derive from US Core.

### Disposition C: Decline

#### Proposal

Reject FHIR-55987 and retain the existing STU 2.1 SHALL requirements that
the export operation update `LastTransmission` / `LastTypes` /
`LastFilters` extensions on the exported Patient and Group resources.
Treat the read-modifies-data concern as an implementation matter (the
server is free to write internally without exposing the extensions on the
read API surface) and the multi-client concern as out-of-scope for v2.x
(the existing language presumes a single Provider Access consumer per
Group). Document the multi-consumer semantics as a known limitation if
needed.

#### Justification

Declining is defensible on continuity grounds: the extensions and
SHALL requirements have shipped in STU 2.1 and presumably some payer
implementations have already built against them; removing them now
breaks those implementations. The "operation that reads data mutates
the data" concern, while real, is a property of the implementation
choice (the server could maintain the extensions out-of-band), not of
the contract; and the multi-consumer problem could be addressed by
clarifying that `LastTransmission` is per-(Patient, requesting client)
rather than removing the mechanism. **However**, this rationale is
weak in practice: the Zulip threads show no defender of the existing
language, the reporter's analysis went unchallenged, the extensions
have no canonical-id consumers outside this IG, and the workgroup has
already stamped the ticket `Pre-Applied` — declining now would amount
to reverting an applied change without a concrete advocate.

---

### Recommendation

**Recommended disposition:** A

The Zulip discussion converged with no dissent on exactly the change
described in Disposition A, the reporter's three objections are
structurally correct rather than stylistic, and the work is already done
and labelled `Pre-Applied` — commit `96e9f2ec` cleanly removes the
extensions, the slice, the example, and the SHALL conformance statements
in the two narrative pages, all within the standard `input/fsh/` and
`input/pagecontent/` authoring roots called out in the repo briefing,
with no cross-repo (US Core, HRex, JIRA-Spec-Artifacts, UTG) impact.
Disposition B's normative `AuditEvent` mapping is genuinely better
specification design and the workgroup should consider it as a
**follow-up ticket against v2.3.x or v3.0** — but bundling it into
FHIR-55987 would expand scope past what was discussed on Zulip and
require capability-statement edits the v2.2.0 cycle is unlikely to
accommodate. Disposition C lacks an advocate in the gathered evidence
and would un-apply a change the workgroup has already endorsed.
Recommend the workgroup ratify the applied change as Persuasive and,
separately, file a v2.3.x ticket to add an `AuditEvent` mapping for
provider-access bulk exports.
