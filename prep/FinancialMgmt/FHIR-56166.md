## Summary

The reporter notes that the Da Vinci CRD ServiceRequest profile contains a
slice on `locationCode` for the CMS place-of-service (POS) code, but the
parallel CRD MedicationRequest profile does not. They argue that the CMS
POS code is just as important when requesting coverage information for a
medication as it is for a service, because payer coverage rules and prior
authorization frequently depend on where the medication will be
administered (e.g., pharmacy `01`, office `11`, outpatient hospital `22`,
home `12`). The request is therefore to add an equivalent CMS
place-of-service `locationCode` slice to the CRD MedicationRequest
profile so that coverage requirements queries for medications can carry
the same place-of-service signal that ServiceRequest already supports.

The request sits in a long-running thread of CRD work around CMS POS
codes that has shaped the current ServiceRequest slice:
[FHIR-43327](https://jira.hl7.org/browse/FHIR-43327) (Persuasive)
introduced the need to expand Location.type to cover all CMS POS
concepts; [FHIR-53648](https://jira.hl7.org/browse/FHIR-53648)
(Persuasive with Modification) established that CRD Location should use
the CMS Place of Service code system, which is the work that produced
the slice on ServiceRequest the reporter is referencing;
[FHIR-46120](https://jira.hl7.org/browse/FHIR-46120) (Persuasive with
Modification) tightened the CMS POS binding on the CRD Location profile;
and [FHIR-52057](https://jira.hl7.org/browse/FHIR-52057) /
[FHIR-51958](https://jira.hl7.org/browse/FHIR-51958) /
[FHIR-55582](https://jira.hl7.org/browse/FHIR-55582) addressed
formatting and value-set composition issues for the same CMS Location
Codes ValueSet that any new MedicationRequest slice would also bind to.
[FHIR-40885](https://jira.hl7.org/browse/FHIR-40885) (Persuasive)
previously added guidance to CRD MedicationRequest for HCPCS J-codes,
showing the workgroup has been actively layering coverage-relevant
information onto the MedicationRequest profile.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a CMS place-of-service `locationCode` slice to the CRD
MedicationRequest profile (`StructureDefinition/profile-medicationrequest`
in `HL7/davinci-crd`) that mirrors the existing slice on the CRD
ServiceRequest profile. Concretely:

- Slice `MedicationRequest.supportingInformation` (the existing path used
  by CRD to convey associated `Location` resources) such that one slice
  references a `Location` whose `locationCode` is bound to the
  **CMS Location Codes** value set
  (`http://hl7.org/fhir/us/davinci-crd/ValueSet/cmslocationcodes`) that
  the ServiceRequest slice already uses.
- Use the same slicing discriminator, cardinality (`0..*`), strength
  (`required` for the CMS code system, allowing additional codings such
  as US Core), and naming as the ServiceRequest slice, so implementers
  see a single, parallel pattern.
- Update the profile narrative and any "Coverage Requirements
  Discovery" guidance pages that describe how to convey
  place-of-service for service requests to also describe the equivalent
  for medication requests, mentioning HCPCS J-code / clinic-administered
  medication scenarios (per FHIR-40885).
- Add at least one example MedicationRequest in the IG that carries a
  CMS POS code (e.g., `11` Office or `22` Outpatient Hospital) on the
  associated Location.

#### Justification

This is the minimum change needed to give CRD MedicationRequest parity
with CRD ServiceRequest for a coverage signal that payers already rely
on in production (FHIR-46120 documents Cambia's requirement). The
underlying value set, code system, and formatting issues have already
been resolved by the FHIR-53648 / FHIR-43327 / FHIR-52057 chain, so the
slice can be added with no new terminology work. Mirroring the
ServiceRequest slice exactly minimises implementer surprise and avoids
introducing a second, divergent way of expressing place-of-service in
the same IG.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of slicing `supportingInformation`, define (or reuse) a
**CRD-wide `place-of-service` extension** that can be applied uniformly
to both `ServiceRequest` and `MedicationRequest` (and any future
request-style profile in CRD), with the extension typed as
`CodeableConcept` and bound `required` to the CMS Location Codes value
set. Then:

- Add the extension to the CRD MedicationRequest profile at
  `MedicationRequest.extension` with cardinality `0..1`.
- Re-document the CRD ServiceRequest `locationCode` slice as the
  legacy/canonical mechanism for ServiceRequest, but recommend the new
  extension for any new request-style profile, with a note that the two
  conveyances are semantically equivalent.
- Provide a mapping in the IG narrative between the extension value and
  the existing `supportingInformation → Location.locationCode` shape so
  implementers can translate.

#### Justification

The current ServiceRequest pattern requires the sender to construct a
full `Location` resource solely to carry a single CMS POS code, which
many implementers have noted is heavier than the use case warrants
(echoed by the FHIR-40674 discussion of lightweight modifiers, and by
FHIR-52357's preference for not synthesising "dummy" resources). A
single, request-level extension would be cheaper to author and easier to
validate, and would let CRD evolve toward a single place-of-service
mechanism. The trade-off is a temporary period in which CRD has two
ways to convey the same signal, and an extension may be perceived as
diverging from the ServiceRequest pattern the reporter is explicitly
asking to mirror.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that CRD MedicationRequest can already carry
CMS place-of-service information today by referencing a `Location`
resource via `MedicationRequest.supportingInformation`, where that
`Location` already binds to the CMS Location Codes value set per the
work done under FHIR-53648 / FHIR-46120. Document this pattern in the
CRD MedicationRequest profile narrative and add an example, but do not
add a new slice or extension.

#### Justification

The mechanism the reporter wants is technically already available
through the generic `supportingInformation` reference to a CMS-coded
`Location` resource. Declining the slice avoids profile churn,
keeps MedicationRequest simpler, and avoids creating a third
maintenance surface for the same CMS POS binding (the value set itself
is still seeing churn — see FHIR-55582). If the practical issue is
discoverability, that can be addressed with documentation alone.

---

### Recommendation

**Recommended disposition:** A

The reporter's framing is correct: CMS place-of-service is a first-class
coverage signal for medications administered under the medical benefit
(see FHIR-40885's HCPCS J-code guidance), not an incidental piece of
"supporting information." The CRD ServiceRequest profile already treats
it that way with an explicit slice, and the workgroup has invested
substantial effort (FHIR-43327, FHIR-53648, FHIR-46120, FHIR-52057) in
making that slice's value set fit-for-purpose. Disposition A propagates
that investment to MedicationRequest with the smallest possible
profile change, preserves a single, consistent CRD pattern for
implementers, and is directly supported by an existing payer
requirement. Disposition B is worth raising in the workgroup discussion
as a longer-term direction, but adopting it for this ticket would mean
diverging the two profiles right now, which is the opposite of what
the reporter is asking for. Disposition C is the weakest option:
relying on un-sliced `supportingInformation` leaves payers with no
deterministic way to find the CMS POS code, which is exactly the
problem the ServiceRequest slice was created to solve.
