## Summary

The ticket questions why `Consent.provision.period` is marked as required
on the HRex Consent profile that Da Vinci PDex Member Match relies on.
The reporter notes that the CMS/ONC Final Rule (Advancing
Interoperability and Improving Prior Authorization Processes, 89 FR
8758) states that a member's opt-in election is valid indefinitely
unless explicitly revoked. Because indefinite opt-in is allowed by
regulation, the reporter asks whether the validity period should be
optional rather than required, so that Member Match exchanges can
faithfully represent an opt-in with no defined end date. No other Jira
tickets were cross-referenced; the only cross-reference returned is the
FHIR element `Consent.provision.period` itself, mentioned in the ticket
description.

## Details

**Description:**

The Final Rule indicates that a member's opt in election is valid
indefinitely unless it is revoked (see comment and
[response](https://www.federalregister.gov/documents/2024/02/08/2024-00895/medicare-and-medicaid-programs-patient-protection-and-affordable-care-act-advancing-interoperability#p-907)).
However, the HRex Consent resource used by Member Match has the Consent
validity period (`Consent.provision.period`) as required. Can/should
this be optional instead?

**Resolution Description:**

PDex aims to stay in line with HRex.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the HRex Consent profile referenced by PDex Member Match, relax the
cardinality of `Consent.provision.period` from required (1..1) to
optional (0..1). Because the profile lives in HRex (not PDex), the
change would need to be coordinated with the HRex specification owners;
PDex would either (a) wait for HRex to relax the constraint and then
re-reference the updated profile, or (b) define a PDex-local derived
profile that loosens the cardinality on `provision.period` to 0..1 while
keeping the rest of the HRex constraints intact. Update the Member Match
narrative to state that an absent `provision.period` indicates an
indefinite opt-in that remains in force until explicitly revoked via a
later Consent instance.

#### Justification

This directly aligns the Consent profile with the CMS Final Rule
language that an opt-in is valid indefinitely unless revoked. Forcing a
period today either compels payers to fabricate an arbitrary end date
(undermining data integrity) or to repeatedly re-issue Consent records
to "extend" a period that the regulation says does not need to end. FHIR
already treats `Consent.provision.period` as 0..1 in the base resource,
so loosening the profile constraint is consistent with base-spec design
and does not break implementers who still choose to send a period.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Consent.provision.period` required in the HRex profile but change
its semantics so that the `start` is required and the `end` is
explicitly optional, with profile-level guidance stating that an absent
`end` denotes an indefinite opt-in that persists until revoked. If HRex
already permits an open-ended `Period` (only `start` populated), update
PDex Member Match documentation to call this pattern out explicitly,
add an example bundle showing an indefinite opt-in, and add an
invariant/guidance that revocation MUST be expressed via a superseding
Consent with `status = inactive` (or equivalent) rather than by setting
`provision.period.end` retroactively.

#### Justification

This preserves the existing HRex constraint surface (no profile
cardinality change, no downstream re-conformance work for existing
implementers) while still satisfying the Final Rule's "indefinite
unless revoked" semantics. It also avoids the ambiguity of a fully
absent `provision.period`, which today carries no shared interpretation
across PDex implementers — some may treat a missing period as "no
consent on file" rather than "indefinite consent." Trade-off vs.
Disposition A: implementers must understand and uniformly apply the
"open-ended Period = indefinite" convention, but no profile churn is
required.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Considered - Question answered**, with the
response that PDex intentionally tracks the HRex Consent profile as-is
and does not redefine `Consent.provision.period` cardinality on its own.
Implementers needing to express an indefinite opt-in should follow HRex
guidance on Consent.provision.period (e.g., supply a far-future `end`
or use the open-ended Period pattern HRex supports), and any change to
the cardinality itself should be raised against HRex rather than PDex.

#### Justification

This is the disposition the workgroup actually applied: the resolution
recorded on the ticket is *"PDex aims to stay in line with HRex."*
Because the constrained element lives in the HRex profile, declining at
the PDex level is procedurally correct — PDex changing the cardinality
unilaterally would fork from HRex and create two divergent Consent
profiles for overlapping use cases, which is exactly the situation HRex
was created to prevent. The substantive question (does `period` need to
be required?) belongs in an HRex ticket.

---

### Recommendation

**Recommended disposition:** C

The ticket has already been resolved as "Considered - Question
answered" with the rationale that PDex stays aligned with HRex, and
that is the right outcome for this ticket *as scoped to PDex*. The
constraint in question is owned by HRex, so any change to
`Consent.provision.period` cardinality must be made there to avoid PDex
forking the HRex Consent profile. The substantive concern raised by the
reporter — that the Final Rule allows indefinite opt-in and the current
required period is awkward to represent — is real and worth pursuing,
but the appropriate next step is a parallel HRex ticket (which the
workgroup should encourage the reporter to file). Disposition A is the
right *substantive* fix but is out of scope for PDex alone; Disposition
B is a reasonable interim documentation pattern that PDex could adopt
even without an HRex profile change, and the workgroup may want to add
an implementation note to that effect when closing this ticket.
