## Summary

This ticket asks Financial Management to promote five National Uniform Billing
Committee (NUBC) data elements that are required on every UB-04 institutional
(facility) claim from the generic `supportingInfo` slot into first-class
*backbone* elements on the Claim, ClaimResponse, and ExplanationOfBenefit
resources. The five elements named are **Point of Origin for Admission or
Visit**, **Priority (Type) of Admission or Visit**, **Type of Bill**,
**Patient Discharge Status**, and **Present on Admission**. The reporter's
position is that, because these values are mandatory on all institutional
claims and have well-defined NUBC code semantics, expressing them via the
generic `supportingInfo` repeating extension point under-specifies them and
forces every implementer (and every payer-facing IG) to re-invent the same
slicing.

No formal Jira links, Zulip threads, or GitHub items are cross-referenced
from this ticket in the FHIR Augury index. However, the ticket is one of a
large batch filed the same day by Patricia Taylor proposing the same
"promote from supportingInfo to backbone element" treatment for other
billing data points (DRG, Claim Received Date, NCPDP retail-pharmacy
elements, payer adjudication amounts, network status, etc.) — those
sibling tickets are listed below under *Related Jira Tickets*. There is
also a later, narrower ticket [FHIR-35319](https://jira.hl7.org/browse/FHIR-35319)
(Da Vinci PCT, Persuasive with Modification) that addresses three of the
same five elements (Point of Origin, Admission Type, Discharge Status) by
profiling them on an Institutional profile rather than by changing the
core resources — that prior outcome is directly relevant to the
disposition discussion here. A 2025-10-30 comment by Corey Spears notes
the ticket was originally filed against CARIN BB but is in fact a FHIR
Core request, and Lynn Laakso's notes record that the original 2020
disposition ("Considered for Future Use") was reverted post-CARIN BB STU2
publication so the request is once again live.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add five new optional backbone elements (cardinality `0..1` in the base
resources) to **Claim**, **ClaimResponse**, and **ExplanationOfBenefit**
to carry NUBC institutional billing data:

- `admission` (BackboneElement, `0..1`) on each of Claim, ClaimResponse, and
  ExplanationOfBenefit, with child elements:
  - `pointOfOrigin : CodeableConcept` bound (extensible) to a value set
    drawn from NUBC Point of Origin for Admission or Visit (FL 15);
  - `priority : CodeableConcept` bound (extensible) to NUBC Priority
    (Type) of Admission or Visit (FL 14);
  - `dischargeDisposition : CodeableConcept` bound (extensible) to NUBC
    Patient Discharge Status (FL 17).
- `billType : CodeableConcept` (`0..1`) bound (extensible) to NUBC Type
  of Bill (FL 04), present at the resource root of all three resources.
- A `presentOnAdmission` modifier on each `diagnosis` element
  (`Claim.diagnosis.onAdmission`, `ExplanationOfBenefit.diagnosis.onAdmission`),
  bound (extensible) to the NUBC Present on Admission code system. This
  modeling already exists today on `ExplanationOfBenefit.diagnosis.onAdmission`
  (a `CodeableConcept`) and should be mirrored on `Claim.diagnosis` and
  `ClaimResponse.diagnosis` to close the gap.

All five bindings should reference the value sets established by
[FHIR-26711](https://jira.hl7.org/browse/FHIR-26711) so that the
modeling change and the terminology change land coherently.

#### Justification

These five elements are not optional pieces of supplementary metadata —
they are required on every UB-04 institutional claim by the NUBC
specification, and US payers reject claims that omit them. Carrying
them as untyped `supportingInfo` slices forces every IG (CARIN BB, Da
Vinci PCT, US Core, payer-specific guides) to re-define the same five
slices, with the same NUBC bindings, in subtly different ways. Promoting
them to first-class backbone elements gives implementers a single,
canonically-typed location, enables search parameters and operations to
reference them by element path, and is consistent with how comparable
high-frequency clinical data (e.g. `ExplanationOfBenefit.diagnosis`,
`ExplanationOfBenefit.procedure`) is already modeled. The presence of an
existing `ExplanationOfBenefit.diagnosis.onAdmission` element is direct
precedent that NUBC concepts already belong in the core resources, not
only in `supportingInfo`.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** modify the core Claim / ClaimResponse / ExplanationOfBenefit
resources. Instead, publish a normative **HL7-maintained Institutional
Claim profile set** (one profile per resource) under the Financial
Management work group that:

1. Slices `supportingInfo` into named, fixed-`category` slices for each of
   the five NUBC elements (`pointOfOrigin`, `admissionType`, `typeOfBill`,
   `dischargeStatus`, `presentOnAdmission`), each with cardinality `1..1`
   on institutional claims.
2. Binds those slices (extensible) to the NUBC-derived value sets
   delivered by FHIR-26711.
3. Adds `SearchParameter` definitions on the slice paths so payers can
   filter on them.
4. Is referenced (rather than re-defined) by CARIN BB, Da Vinci PCT, and
   US Core, so the bindings are authored exactly once.

This is essentially the approach already taken by Da Vinci PCT in
[FHIR-35319](https://jira.hl7.org/browse/FHIR-35319) (Persuasive with
Modification), generalised and lifted up to a base-spec-adjacent profile.

#### Justification

A core-resource change costs every existing implementation of Claim /
ClaimResponse / ExplanationOfBenefit — including every non-US
implementation that has no NUBC obligation — a backwards-compatibility
break, even if the new elements are `0..1`. Profile-based modeling is
the FHIR-design-principle answer to "this concept is required in *some*
realms but not all": carry it as a slice with a meaningful, conformance-
checkable name. Da Vinci PCT (FHIR-35319) demonstrates that this works
for three of the five elements; extending it to all five and centralising
the profile under FM avoids both the breaking-change cost and the
fragmentation that the reporter is correctly worried about.

---

### Disposition C: Decline

#### Proposal

Decline to add new backbone elements. Resolve the ticket as **Not
Persuasive** on the basis that (a) the existing `supportingInfo`
mechanism already supports these data elements, with `category` codes
already defined in the `claim-information` and `claiminformationcategory`
value sets for several of them, (b) `ExplanationOfBenefit.diagnosis.onAdmission`
already covers Present on Admission for the most-used resource of the
three, and (c) the realm-specific modeling concern is best handled in
the relevant US realm IGs (CARIN BB, Da Vinci PCT, US Core), not in the
international base resource.

#### Justification

Claim / ClaimResponse / ExplanationOfBenefit are deliberately
international and mode-of-payment-agnostic resources. NUBC is a
US-specific, institutional-claim-specific code authority; baking five
new backbone elements into the international resource to satisfy a
single national billing form is hard to defend against equivalent
requests from other realms (e.g. Australian DRG forms, UK PbR, German
G-DRG/DKG). The original 2020-07-21 FM vote (18-0-5) explicitly chose
"Considered for Future Use" precisely to avoid that precedent; while
that resolution was administratively reverted in 2022, the underlying
design rationale still holds. Implementers who need the elements
typed and required *can* get them today via the existing
`supportingInfo` slot plus a realm profile.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying problem is real and well-documented: NUBC
elements that are mandatory on every US institutional claim should not
require every IG to re-invent the same `supportingInfo` slice. But
Disposition A pays for that fix with a base-resource modeling change
that other realms do not need and that breaks the international /
realm-neutral character of the Claim family — a cost the FM workgroup
has historically been unwilling to accept (the 2020-07-21 18-0-5 vote
on this same ticket, and the parallel Persuasive-with-Modification
outcome on [FHIR-35319](https://jira.hl7.org/browse/FHIR-35319), both
point that direction). Disposition C dismisses a genuine implementer
pain point and ignores that the workgroup has already, in PCT, agreed
that the data elements deserve a typed home. Disposition B threads the
needle: it gives implementers a single, canonical, conformance-checkable
location for all five NUBC elements (and binds them to the value sets
delivered by [FHIR-26711](https://jira.hl7.org/browse/FHIR-26711)),
generalises the precedent already set in PCT, and avoids the
international-vs-realm modeling tension entirely. As a side benefit, it
gives the FM workgroup a reusable Institutional Claim profile asset
that can absorb the related sibling tickets ([FHIR-26712](https://jira.hl7.org/browse/FHIR-26712)
DRG, [FHIR-26720](https://jira.hl7.org/browse/FHIR-26720) Claim
Received Date) under the same disposition without further base-resource
churn.
