## Summary

The ticket asks the Da Vinci PDex IG to rename the input Bundle on the
Provider Member Match v2 operation (`$provider-member-match`) so that it
matches the Bundle name already used by the Payer-to-Payer Bulk Member
Match operation. Specifically, the Bundle parameter currently named
`MembersToMatch` should be renamed to `MemberBundle`, and the `Consent`
parameter inside that bundle should be renamed to `TreatmentAttestation`.
The reporter notes that the inconsistency was caught at the US Realm
Connectathon and that aligning the names will improve implementer
experience across the two member-match operations. The ticket carries the
`Pre-Applied` label and is resolved Persuasive, indicating the change has
already landed in the repo and is being formally balloted (Block-Vote-1 /
PDex-Vote-1). No related Jira tickets were surfaced via cross-references.

## Details

**Description:**

> The change aligns the member bundle in the Provider member Match v2 to
> use the same bundle name as the Payer-to-Payer Bulk member Match.
> The Bundle was MembersToMatch and was changed to MemberBundle.
> Consent within that parameter was changed to TreatmentAttestation.
>
> This was made to align with implementer experience and was caught
> during the US Realm Connectathon.

**Resolution Description:**

> Change Bundle name to match Payer-to-Payer Input Parameter.
> Changed from MembersToMatch to MemberBundle.
> Also change the Consent Parameter to TreatmentAttestation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the parameter rename as written by the reporter (the change the
`Pre-Applied` label already reflects):

- In `HL7/davinci-epdx` `input/fsh/Operation-provider-member-match.fsh`,
  rename the input `parameter` whose `name` is `MembersToMatch` (type
  `Bundle`) to `MemberBundle`.
- Within that same parameter (or its sibling part / nested parameter,
  per the operation's current FSH structure), rename `Consent` to
  `TreatmentAttestation`.
- Update any references to the old names in narrative pages under
  `input/pagecontent/` and in any FSH examples under `input/fsh/`
  (e.g. `Example*.fsh`) so the IG renders cleanly.
- Add a change-history entry citing FHIR-56242 to
  `input/pagecontent/changehistory.md` under STU 2.2.0 (already
  present per the cross-references).
- Do not touch `output/`, `template/`, or `fsh-generated/` — those are
  publisher-generated.
- Target the active publication branch (`pre-publish_stu2` /
  `post-publish_stu2`) per the repo's documented branch convention;
  no version-registration change in `FHIR-us-davinci-pdex.xml` /
  `package-list.yml` is required because PDex stays at 2.2.0.

#### Justification

The change is a pure naming alignment between two sibling operations
(`$provider-member-match` and the Payer-to-Payer Bulk Member Match) in
the same IG, both of which conceptually pass the same artifact (a
Bundle of patient candidates plus a treatment-relationship attestation).
Calling the same input two different things across the two operations
forced implementers at the US Realm Connectathon to maintain parallel
serialization paths for no semantic reason. Renaming `Consent` to
`TreatmentAttestation` is also more accurate: PDex provider access is
predicated on a treatment-relationship attestation, not on FHIR
`Consent`-resource semantics, so the previous name was misleading. The
rename is local to one OperationDefinition (no profile derivations,
search parameters, or value sets are affected), is already implemented
in the repo, and the resolution is recorded as Persuasive — Disposition
A simply ratifies the in-tree change for the ballot.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the rename, but additionally retain the previous
`MembersToMatch` and `Consent` parameter names as **deprecated
aliases** for one publication cycle. Mechanically: in
`Operation-provider-member-match.fsh`, define both parameter names with
identical type/cardinality, mark the old names with a deprecation note
(via `^extension[$standards-status]` set to `deprecated`, or an
inline `^comment` flagging them as deprecated and to be removed in the
next major version), and document the migration in
`input/pagecontent/changehistory.md`. Servers SHOULD accept either
name; clients SHOULD send the new names.

#### Justification

The original ticket framing (caught at Connectathon, alignment with
implementer experience) implies that some implementers had already
written code against `MembersToMatch` / `Consent`. A hard rename in a
2.x point release breaks those implementations on the wire. A
deprecation cycle preserves the alignment win that motivates the
ticket while giving early adopters a documented migration path. The
trade-off is real: an OperationDefinition with two synonymous input
parameters is awkward, and PDex would carry the cruft until a 3.x
release. The branch / pin context (PDex 2.2.0, HRex 1.1.0) makes a
single point release the natural deprecation window.

---

### Disposition C: Decline

#### Proposal

Reject the rename and keep `MembersToMatch` and `Consent` as the input
parameter names of `$provider-member-match`. Revert the in-tree change
on the publication branch and document the decision in
`input/pagecontent/changehistory.md`.

#### Justification

A rename of an input parameter on a published OperationDefinition is a
breaking change for any client or server already implementing the v2
operation. If consistency with the bulk operation is the only driver,
that consistency could be communicated through narrative ("Provider
Member Match's `MembersToMatch` is the same artifact as Bulk Member
Match's `MemberBundle`") without changing the wire contract. This is
the most defensive option and would only be appropriate if the
workgroup judged the implementer disruption to outweigh the alignment
benefit — which the Persuasive resolution and `Pre-Applied` label
suggest is not the workgroup's current read.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved Persuasive, the change carries
the `Pre-Applied` label, the corresponding commit
(`0fcdfea4…: FHIR-56242: Rename MembersToMatch to MemberBundle in
Provider Member Match`) is already in `HL7/davinci-epdx`, and the
cross-references show the STU 2.2.0 change-history already cites the
ticket. The change is local to one OperationDefinition, has no
profile, value-set, or search-parameter fan-out, and aligns
`$provider-member-match` with the sibling Bulk Member Match operation
so implementers stop having to special-case the parameter names.
Disposition B (deprecation aliases) is reasonable in principle, but
overweighs the implementer-disruption risk for a v2 operation that is
explicitly being aligned at the same Connectathon where the
inconsistency was reported, and an OperationDefinition with two
synonymous input parameters is itself a footgun that drags into the
next major version. Disposition C should be declined: the alignment
benefit is exactly what was validated at the Connectathon and the
rename also corrects a misleading `Consent` name that did not match
PDex's treatment-relationship-attestation semantics.
