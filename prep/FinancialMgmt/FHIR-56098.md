## Summary

The reporter asks the Da Vinci Risk Based Contracts Member Attribution
(ATR) IG to adopt the new Bulk Submit operation that the Bulk Data
Access IG is adding (currently drafted at
`https://hackmd.io/@argonaut/rJoqHZrPle` and tracked in the Bulk Data
"Future" changes page). The submitter's argument is that an attribution
list is fundamentally a curated, payer-published artifact that is
periodically pushed to providers (or providers/EHRs pushing changes
back), so a Submit-style workflow fits attribution exchange better than
the on-the-fly `Group/$export` pattern that ATR uses today, and Submit
should "become the primary exchange mechanism for ATR over time."

The same Bulk Submit alignment work is happening in adjacent IGs: DEQM
already has [FHIR-51762](https://jira.hl7.org/browse/FHIR-51762)
("Align DEQM bulk submit options with upcoming Bulk Data IG changes",
Applied/Persuasive) which retargets DEQM's `$bulk-submit-data` to the
new core Bulk Submit; the underlying core operations are being
introduced into the Bulk Data IG itself by
[FHIR-56171](https://jira.hl7.org/browse/FHIR-56171) (`$bulk-submit`
and `$bulk-submit-status`, FHIR Infrastructure, Resolved – change
required) and
[FHIR-56172](https://jira.hl7.org/browse/FHIR-56172) (`$bulk-publish`,
FHIR Infrastructure, Resolved – change required). Earlier DEQM
narrative work in [FHIR-39901](https://jira.hl7.org/browse/FHIR-39901)
established the editorial pattern of pointing readers from a domain
IG's data-exchange section to a `$bulk-submit-data` operation, and
[FHIR-47449](https://jira.hl7.org/browse/FHIR-47449) shows another
specification (NHSN dQM) doing the same alignment. ATR is currently the
notable Da Vinci attribution IG that has *not* yet adopted Bulk Submit;
this ticket asks Financial Mgmt to close that gap, leveraging the work
above instead of inventing an ATR-specific submission pattern.

### Disposition A: Accept As Requested

#### Proposal

Add first-class support for the new core Bulk Submit operations to the
Da Vinci ATR IG. Concretely:

1. In the ATR IG's "Bulk Data" / data-exchange narrative, add a section
   stating that an ATR roster MAY be exchanged using
   `Bulk Data Access IG` `$bulk-submit` (with `$bulk-submit-status`)
   from FHIR-56171, in addition to the existing `Group/$export`
   pattern. Reference the Bulk Data IG operations rather than redefining
   them.
2. Define an ATR-specific manifest contract: the input `inputFormat` is
   `application/fhir+ndjson`, the input is the ATR `Group` plus the
   `Patient` / `Coverage` / `Practitioner` / `PractitionerRole` /
   `Organization` / `Location` resources that make up the roster, and
   the receiver MUST validate against the ATR profiles.
3. Update the ATR `CapabilityStatement`(s) for Member Attribution
   List Source and Consumer servers to declare support for `$bulk-submit`
   and `$bulk-submit-status` (analogous to how DEQM declares
   `$bulk-submit-data`). Mark Submit as `SHOULD` for the first
   normative-track release that targets this, and signal intent to make
   it `SHALL` once Bulk Data IG STU is final.
4. Add a "Future direction" note matching the reporter's intent:
   `$bulk-submit` is expected to become the primary exchange mechanism
   for ATR over time, with `Group/$export` retained for ad-hoc/on-demand
   pulls.

#### Justification

The upstream operations are no longer hypothetical — they were Resolved
as Persuasive on the Bulk Data IG side (FHIR-56171, FHIR-56172) on
2026-03-30, and DEQM has already taken the same step (FHIR-51762,
Applied). Doing the same alignment in ATR keeps Da Vinci IGs
internally consistent and follows the editorial pattern established by
FHIR-39901: introduce the operation in the data-exchange narrative,
update the CapabilityStatement, and link out to the core
OperationDefinition rather than redefining it. Attribution rosters are
periodic, payer-curated artifacts where push semantics are a better fit
than the polling/export semantics ATR currently relies on, which is the
substantive design argument the reporter makes and which the Argonaut
draft was explicitly designed to support.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than directly adopting `$bulk-submit` for the full roster
exchange, take a two-step approach in this STU cycle:

1. **Narrative-only update now.** Add a short "Bulk Submit (future)"
   subsection to the ATR data-exchange page that (a) acknowledges the
   new core operations from FHIR-56171/FHIR-56172, (b) cites the
   Argonaut draft, and (c) states ATR's intent to adopt Submit as the
   primary mechanism in a future release. Do not yet change the
   CapabilityStatement or define an ATR-specific manifest binding.
2. **Profiled `$bulk-publish` for the master roster.** Where ATR is
   used to publish a periodically refreshed master attribution list to
   downstream consumers, prefer the `$bulk-publish` operation (FHIR-56172)
   – which is explicitly designed for static / snapshot-with-deltas
   publishing without requiring a full FHIR server – and reserve
   `$bulk-submit` (FHIR-56171) for provider→payer corrections /
   acknowledgements where a true submission semantics applies.
3. Open a follow-up ticket against the next ATR ballot cycle to do the
   full Submit binding once Bulk Data IG reaches STU.

#### Justification

Both new operations are still very fresh (Resolved on 2026-03-30, days
after this ticket was filed), and ATR's roster exchange use case is not
a uniform fit for "Submit": rosters are predominantly payer-published
snapshots that downstream consumers pull periodically, which is much
closer to `$bulk-publish` semantics than to `$bulk-submit` semantics.
Splitting the editorial work from the normative binding lets Financial
Mgmt signal direction-of-travel in the next ATR release without
locking in a manifest contract that may shift as the Bulk Data IG goes
through ballot. It also protects implementers from a churn-y normative
change in the ATR `CapabilityStatement` immediately followed by another
when the upstream operations stabilize.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as **premature** for this ATR cycle. ATR retains
`Group/$export` as its bulk mechanism. Re-file once the Bulk Data IG
release containing `$bulk-submit` / `$bulk-submit-status` /
`$bulk-publish` is at STU and there is implementer demand specifically
for ATR Submit (e.g. connectathon tracks, CMS RFI feedback, or payer
implementer requests).

#### Justification

The ticket is brief, has no profile/operation design attached, no
specific use cases or implementers cited, and was raised within days of
the upstream FHIR Infrastructure tickets (FHIR-56171/FHIR-56172) being
resolved. The only WG discussion captured (Plex Community Call,
2026-03-27, comment from `ekivemark`) is a single line saying "needs
further review." Adding a normative binding to a pre-STU operation
risks ATR having to re-issue breaking guidance once the Bulk Data IG
ballot consolidates the manifest and status semantics. Declining now
avoids that churn without precluding the change in a subsequent cycle.

---

### Recommendation

**Recommended disposition:** B

The substantive direction the reporter argues for — that Submit-style
push semantics fit attribution roster exchange better than on-the-fly
export — is correct, and the upstream Bulk Data work is now Resolved
rather than speculative, so a flat Decline (C) sends the wrong
direction-of-travel signal and would have to be reversed in the next
cycle (mirroring DEQM's path via FHIR-39901 → FHIR-51762). At the same
time, fully binding ATR to `$bulk-submit` *and* updating the ATR
`CapabilityStatement` *and* defining an ATR-specific manifest contract
in this cycle (Disposition A) is more normative weight than a
six-sentence ticket can justify, especially while the Bulk Data IG is
still pre-STU and while ATR's actual use case (payer-published
periodic rosters) is arguably a better fit for `$bulk-publish` than for
`$bulk-submit`. Disposition B captures direction now via a narrative
"future" note, splits Submit vs. Publish along the right semantic line,
and defers the normative binding to a follow-up where Financial Mgmt
can co-design the manifest contract with the Bulk Data IG editors after
ballot. That gives implementers a clear roadmap without locking ATR
into a contract that is likely to shift.
