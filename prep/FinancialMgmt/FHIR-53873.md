## Summary

The reporter (Charles Willcutt, Epic) requests that the base FHIR
`Organization` resource gain a way to express the organization's
specialty. The motivating use case is claim billing: when an
organization (e.g., a billing provider, facility, or group practice) is
referenced on a `Claim`, some payers require an organization-level
specialty (typically a NUCC/Healthcare Provider Taxonomy code) in order
to adjudicate and reimburse correctly. The reporter notes that the
linked `Organization` resource has no `specialty` element today and
suggests extending `Organization` to include a `specialty` if
applicable. No related Jira tickets, GitHub items, or Zulip threads
were surfaced via the cross-reference index for this ticket; the only
recorded activity is a single FM-Workgroup comment asking the reporter
to attend an FM conference call to walk through the requirement.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new optional, repeating element `Organization.specialty` of type
`CodeableConcept` (cardinality `0..*`), bound to the same value set
already used by `PractitionerRole.specialty` and
`HealthcareService.specialty` — the
`http://hl7.org/fhir/ValueSet/c80-practice-codes` value set (preferred
binding), which surfaces the SNOMED practice-setting / NUCC taxonomy
codes implementers already use for billing. Definition: "The kind(s)
of services or specialties provided by this organization (e.g.,
Cardiology, Independent Diagnostic Testing Facility). Used when a
payer or downstream consumer needs to know the organization's
specialty independently of any `PractitionerRole` or
`HealthcareService` it owns."

#### Justification

The change is minimally invasive (additive, optional, repeating
`CodeableConcept`), aligns with the binding pattern already used on
`PractitionerRole.specialty` and `HealthcareService.specialty`, and
gives `Claim`/`ExplanationOfBenefit` producers a direct path to
satisfy payers that require a billing-organization specialty
without having to fabricate a `PractitionerRole` or
`HealthcareService` solely as a carrier. The reporter is from a
major EHR vendor and cites a concrete reimbursement requirement, so
the use case is real and recurring.

---

### Disposition B: Alternative Approach

#### Proposal

Do not modify base `Organization`. Instead, do two things:

1. Publish a standard extension
   `http://hl7.org/fhir/StructureDefinition/organization-specialty`
   (type `CodeableConcept`, `0..*`, preferred binding to
   `c80-practice-codes`) so that organization-level specialty can be
   expressed today on any `Organization` instance, including in
   profiles such as US Core Organization and Da Vinci PDex
   Plan-Net Organization.
2. In the Financial Management guidance for `Claim`, document the
   recommended pattern for conveying a billing organization's
   specialty: either (a) reference a `PractitionerRole` that has the
   organization in `.organization` and the specialty in `.specialty`
   (no practitioner required — already supported per Lloyd's note in
   the *Resource for Provider* Zulip thread), or (b) populate the new
   `organization-specialty` extension on the referenced
   `Organization`.

#### Justification

Existing FHIR semantics already allow organization-level specialty
via `PractitionerRole` with no `Practitioner` (per the implementers
Zulip thread linked above) and via `HealthcareService.specialty`.
Adding specialty to base `Organization` introduces a third place
the same concept can live, which fragments queries (provider
directories already have to look in `PractitionerRole.specialty`
and `HealthcareService.specialty`) and risks contradictory data on
the same organization. A standard extension preserves the option
for implementers who genuinely need it without committing the core
spec to a fourth specialty-bearing element, and it can be promoted
to a core element later if uptake justifies it. This approach is
also non-breaking for R4/R5 implementers and can ship in a future
US Core / FM IG without waiting for an R6 ballot cycle.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed by existing mechanisms**. Recommend
the reporter convey organization specialty by either:

- Referencing a `PractitionerRole` from the `Claim`'s
  `provider`/`careTeam` whose `.organization` is the billing
  organization and whose `.specialty` carries the required taxonomy
  code (no `Practitioner` required), or
- Using `HealthcareService.specialty` on the `HealthcareService`
  delivered by the organization, or
- For payer-specific requirements, defining the specialty as a
  payer-specific or IG-specific extension on `Organization` rather
  than changing the base resource.

#### Justification

The base `Organization` resource is intentionally a thin
administrative resource; specialty is a *role* concept (what an
organization *does* in a particular relationship), which is why
FHIR locates it on `PractitionerRole` and `HealthcareService`. The
historical Zulip discussions (Argonaut, US Core, QI-Core,
provider-directory) consistently treat `PractitionerRole.specialty`
as the canonical search axis for "find me a provider with this
specialty." Adding `Organization.specialty` would create
data-locality ambiguity (which of three places is authoritative
when they disagree?) for a use case that is already expressible.
Payer-specific billing requirements are a classic extension use
case rather than a base-spec change.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — standard
extension plus FM billing guidance).

The reporter's underlying need is real and well-grounded in claim
adjudication: payers do require an organization-level specialty in
some scenarios, and asking implementers to instantiate a
practitioner-less `PractitionerRole` purely as a carrier is awkward,
under-documented, and easy to get wrong (as the implementers thread
shows, even experienced contributors have to clarify that this is
allowed). Disposition C is technically correct but ignores the
documentation gap that drove the ticket. Disposition A solves the
reporter's problem directly but introduces a third place specialty
can live on an organization, which the Financial Management and
Patient Administration work groups have historically resisted because
it complicates provider-directory queries and US Core constraints.

Disposition B threads the needle: a standard extension immediately
unblocks billing implementers (including the reporter) without a
breaking-change ballot cycle, preserves `PractitionerRole.specialty`
as the canonical search axis, and gives the Financial Management
work group a concrete artifact (the extension definition + Claim
guidance) to review. If real-world uptake of the extension is high
across multiple IGs, FM and Patient Administration can revisit
promoting it to a core `Organization.specialty` element in a future
release. The single existing comment on this ticket — pknapp asking
the reporter to attend an FM call — is consistent with this path:
the reporter can confirm the use cases on the call, and FM can
sponsor the extension while looping in Patient Administration (the
work group that owns `Organization`).
