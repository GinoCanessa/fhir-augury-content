## Summary

This ticket asks Financial Management (FM), in coordination with Patient
Administration (PA), to harmonize the `type` elements that appear across the
insurance-related resources so that they no longer use mismatched cardinalities
or unrelated value sets for what is effectively the same concept. Specifically:
`Coverage.type` (FM) is currently `0..1` and bound to `valueset-coverage-type`
(HMO, PPO, dental, LTC, vision); `InsurancePlan.type` (PA) is `0..*` and bound to
`valueset-insuranceplan-type` (medical, dental); and `InsurancePlan.coverage.type`
(PA) has no binding at all but, by definition, overlaps the other two. The
reporter recommends raising `Coverage.type` to `0..*` in FM's resource and
realigning the InsurancePlan-side bindings to share the same value set
(`valueset-coverage-type`), while clarifying *why* both `InsurancePlan.type` and
`InsurancePlan.coverage.type` exist if they end up bound to the same vocabulary.

The companion PA-side ticket [FHIR-24915](https://jira.hl7.org/browse/FHIR-24915)
("Harmonize Coverage and InsurancePlan type", *Persuasive with Modification*)
captures the InsurancePlan-side half of the same problem. PA's draft resolution
on FHIR-24915 already proposes a concrete realignment: bind `Coverage.type`,
`InsuranceProduct.type`, and `InsuranceProduct.coverage.type` to
`http://terminology.hl7.org/ValueSet/insurance-coverage-type` (the value set
already used by the US National Directory IG / NDH), with `Coverage.type` at
`Preferred` and the `InsuranceProduct` bindings at `Example`. Because R6 splits
R4's `InsurancePlan` into `InsuranceProduct` + `InsurancePlan`, the original
`InsurancePlan.coverage.type` concern is now scoped to
`InsuranceProduct.coverage.type` (per FM's June 2024 comment), and the immediate
FM priority is finishing the cleanup of the `Coverage.type` code set before
locking in any cross-resource binding alignment.

## Details

**Description:**

> Variances to be harmonized: value set bindings, cardinality, etc.
>
> For example:
> * `Coverage.type` is `0..1` and binds to `valueset-coverage-type` (e.g. HMO, PPO, dental, LTC, vision)
> * `InsurancePlan.type` (product type) is `0..*` and binds to `valueset-insuranceplan-type` (e.g. medical, dental)

> Recommendation is to:
> * Change `Coverage.type` to `0..*` (FM)
> * Change `InsurancePlan.type` and/or `InsurancePlan.coverage.type` to bind to the same value set (`valueset-coverage-type`) as well as differentiate why both of these types are needed (logged to PA as tracker 24915 — now [FHIR-24915](https://jira.hl7.org/browse/FHIR-24915))

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the original two-part change. In **HL7/fhir** (FHIR Core, FM-owned
artifacts), change `Coverage.type` cardinality from `0..1` to `0..*`. In
coordination with PA via [FHIR-24915](https://jira.hl7.org/browse/FHIR-24915),
rebind `InsurancePlan.type` (R4 / R5) and `InsuranceProduct.type` /
`InsuranceProduct.coverage.type` (R6) to the same value set FM uses on
`Coverage.type` — the cleaned-up successor to `valueset-coverage-type`,
expected to be `http://terminology.hl7.org/ValueSet/insurance-coverage-type`
once FM finishes its in-flight code set work — and document in the resource
narrative why both a product-level `type` and a per-coverage `type` continue
to exist.

#### Justification

This is the literal request and matches the direction PA has already drafted on
FHIR-24915 (shared `insurance-coverage-type` value set; `Coverage.type` at
*Preferred*, the InsuranceProduct-side bindings at *Example*). Raising
`Coverage.type` to `0..*` is consistent with how implementers actually use
Coverage today — a single insurance coverage commonly carries more than one
applicable "type" classifier (e.g., medical + vision rider) — and aligning the
vocabulary across resources removes a long-standing source of confusion when
data flows between Coverage, InsurancePlan, and InsuranceProduct.

---

### Disposition B: Alternative Approach

#### Proposal

Decouple the two halves. **Now (current cycle):** keep `Coverage.type` at
`0..1` but tighten its definition and finish FM's `Coverage.type` value-set
cleanup so it can serve as the shared insurance-type vocabulary. **Next
cycle:** once FM publishes the cleaned value set and PA closes its open TODOs
on FHIR-24915 (specifically: do `InsuranceProduct.type` and
`InsuranceProduct.coverage.type` share a value set, and what value set is used
for `InsuranceProduct.benefit.type`), perform the cross-resource binding
alignment in a single coordinated PR. If implementers genuinely need to express
multiple insurance "types" on one coverage, expose that through a
`Coverage.classification` or a standard extension rather than by widening
`Coverage.type` cardinality, since `Coverage.type` is heavily relied on as a
single discriminator in profiles (e.g., US Core, NDH).

#### Justification

The June 2024 FM comment explicitly states the priority is to clean up
`Coverage.type` *first* and only then coordinate with `InsuranceProduct.type`
in R6, and FHIR-24915's resolution still has unresolved TODOs that block any
clean shared-binding decision today. Cardinality changes on a base resource
element that downstream IGs profile heavily are also a meaningful
implementer-impact event — splitting the work lets FM ship the high-value
vocabulary harmonization without forcing IGs to react to a `0..1 → 0..*` widening
in the same release.

---

### Disposition C: Decline

#### Proposal

Close FHIR-24916 as **Not Persuasive** on the cardinality change and as a
**Duplicate of FHIR-24915** for the binding-harmonization piece. The substantive
binding work is already tracked and resolved (Persuasive with Modification) on
the PA-side ticket; carrying a parallel FM ticket adds coordination overhead
without changing the outcome. If implementers need to express multiple
insurance type classifiers on a single Coverage, they can use repeating
`Coverage.class` or a profile-level extension rather than a base-spec
cardinality change.

#### Justification

`Coverage.type` widening is a backward-incompatible change for every consumer
that has assumed a single value, including widely deployed US Core and payer-IG
profiles. The FHIR-24915 resolution already commits to the shared
`insurance-coverage-type` value set on the PA side, which delivers the
"harmonized vocabulary" outcome this ticket asks for, so the FM-side ticket is
largely redundant once that resolution lands. Declining keeps base-resource
churn low and concentrates the work where it has already been triaged.

---

### Recommendation

**Recommended disposition:** B

The harmonization is real and worth doing, and FHIR-24915's draft resolution
gives the workgroups a clear direction (`insurance-coverage-type` as the shared
value set). But the most recent substantive comment on this ticket, from FM
itself, explicitly sequences the work: clean up `Coverage.type` first, then
align with the R6 `InsuranceProduct.type`. PA's resolution still carries
unresolved TODOs about whether `InsuranceProduct.type` and
`InsuranceProduct.coverage.type` should share a value set, so committing today
to the full Disposition A package (including the `0..1 → 0..*` cardinality
change on `Coverage.type`) risks pushing a breaking change into a base resource
ahead of a vocabulary decision that is not yet final. Disposition B preserves
the spirit of the original request, lines up cleanly with the joint FM/PA work
already in flight, and avoids forcing implementers to absorb a cardinality
widening and a binding swap in the same cycle. Disposition C understates how
much value FM still adds here — the cardinality question and the FM-owned
resource changes genuinely need an FM ticket — so it is the weakest option.
