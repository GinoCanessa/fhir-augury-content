## Summary

The ticket reports that the legal state machine diagram on the Contract resource page does not align with the codes defined in the bound `contract-legalstate` value set. The reporter has identified two distinct sets of mismatches: (1) states that appear in the diagram but are absent from the value set (`addendum`, `conditional`, `counter`, `draft`, `effective`, `exchange of contracts`, `modified`, `offer`, `pended`, `Request for Information`); and (2) codes that exist in the value set but are not represented in the diagram (`amended`, `appended`, `entered-in-error`, `offered`, `policy`). The reporter proposes three possible remedies: update the diagram to match the value set, update the value set to match the diagram, or revise the surrounding documentation to clarify how the two artifacts relate. No FHIR Augury cross-references to other Jira tickets, GitHub items, or Confluence pages were returned for this ticket. A loosely related committers Zulip thread ("Contract state machine") raised the broader question of whether the Contract state machinery should be normative, but did not address this specific diagram/value-set mismatch.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reconcile the Contract resource's legal state diagram and the `contract-legalstate` value set so that they describe the same set of states. Concretely:

1. Decide which artifact is canonical. Recommended canonical: the `contract-legalstate` CodeSystem/ValueSet (since `Contract.legalState` is extensibly bound to it and validators consume it directly).
2. Update the **diagram** on the Contract resource page (the legal state machine SVG/PlantUML in the Contract source under the FHIR Core spec) so that every state node is one of the codes in `contract-legalstate` (`amended`, `appended`, `cancelled`, `disputed`, `entered-in-error`, `executable`, `executed`, `negotiable`, `offered`, `policy`, `rejected`, `renewed`, `resolved`, `revoked`, `resolved`, `terminated`, plus current additions). For each diagram-only label called out by the reporter (`addendum`, `conditional`, `counter`, `draft`, `effective`, `exchange of contracts`, `modified`, `offer`, `pended`, `Request for Information`), either rename it to the closest existing code (e.g., `addendum` → `appended`, `offer` → `offered`, `effective` → `executed`/`executable`, `modified` → `amended`) or add a new code to the CodeSystem/ValueSet with a clear definition and transition rules.
3. Add the value-set-only codes that have no diagram representation (`amended`, `appended`, `entered-in-error`, `offered`, `policy`) as nodes/transitions in the diagram so the diagram is exhaustive over the value set.
4. Update the Contract narrative on the resource page to assert explicitly that the diagram is normative-with-respect-to the value set (i.e., a state in the diagram MUST correspond to a code in `contract-legalstate`), and document each transition.

#### Justification

`Contract.legalState` is extensibly bound to `contract-legalstate`, so the value set is the authoritative vocabulary for the element. A diagram that visualises that vocabulary should therefore be a faithful rendering of it; otherwise it is actively misleading to implementers, who may write code against labels that have no corresponding code. The reporter's enumeration of missing states on both sides shows the divergence is large enough that a "documentation-only" tweak cannot bridge it. Lloyd McKenzie's general validator stance (in the related HAPI thread) — that invalid codes should be flagged regardless of binding strength — reinforces that implementers expect the value set to be a real source of truth. Aligning the artifacts also unblocks any future move to make the legal state machine normative.

---

### Disposition B: Alternative Approach

#### Proposal

Demote the diagram to a non-normative usage example and add explicit prose disclaiming that the diagram illustrates one possible workflow rather than constraining `Contract.legalState`. Specifically:

1. Leave the `contract-legalstate` value set unchanged (preserves backwards compatibility for implementers).
2. Re-label the diagram heading from "legal state machine" to something like "Example contract lifecycle (informative)" and add a caption stating that the labelled states are illustrative business-process states, some of which map 1:1 to `contract-legalstate` codes and some of which represent finer-grained workflow concepts that implementers may model with extensions, `Contract.status`, or other elements.
3. Add a small mapping table on the Contract page that shows, for each diagram label, the closest `contract-legalstate` code (or "no direct code — model via extension/`Contract.status`").
4. Optionally, file a follow-up ticket to redesign a smaller, formally-aligned state diagram for a future release.

#### Justification

This approach addresses the underlying user complaint — that the two artifacts appear to disagree — without committing the workgroup to either expanding the value set (which could destabilise implementations that already rely on the current code list) or rewriting the long-standing diagram (which has had implementer mindshare since R3). Treating the diagram as informative is consistent with how several other FHIR resources document business-process flows. The mapping table preserves the diagram's explanatory value while making the formal binding unambiguous. The trade-off vs. Disposition A is reduced normative force: implementers cannot rely on the diagram as a state contract, only as guidance.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — Persistent Issue Without Concrete Implementer Demand." Document on the ticket that the diagram has long been understood as illustrative rather than normative, that no implementer has reported being misled in production, and that both `Contract` and `contract-legalstate` are extensively used as-is. Add a single sentence to the Contract page noting that the diagram is informative.

#### Justification

The ticket has been open since January 2019 and last commented in March 2021 with "Being reviewed in light of recent changes," with no further action — indicating the workgroup has not seen sufficient implementer pressure to act. The HAPI Zulip thread confirms `Contract.legalState` is in active use without anyone raising the diagram mismatch. The Contract resource's overall maturity and roadmap (FM_R5_Contract label suggests R5-era review) means there is risk in making code-system or diagram changes that ripple into existing implementations and IGs. Declining preserves stability; the inconsistency, while real, has not produced documented harm.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The reporter's underlying concern — that the diagram and value set appear to be authoritative on the same concept but disagree — is real and deserves a response, so Disposition C (decline) understates the issue and leaves a documented inconsistency on the spec. Disposition A (full reconciliation) is the cleanest long-term outcome but is expensive: it requires the workgroup to either rename diagram nodes (a breaking change for readers and IGs that quote them) or expand the `contract-legalstate` CodeSystem (a terminology change that affects validators and downstream tooling, and one Lloyd's stance in the HAPI thread suggests should be done deliberately). Disposition B resolves the user-visible confusion immediately, with low risk, by clearly designating the diagram as informative and providing a mapping table that points implementers at the formally bound vocabulary. It also preserves the option of executing Disposition A in a future cycle if Financial Management decides to make the legal state machine normative — at which point the alignment work would be done with intent rather than as a documentation cleanup. The workgroup should still decide whether to spawn a follow-up ticket for a properly normative state machine in a later release.
