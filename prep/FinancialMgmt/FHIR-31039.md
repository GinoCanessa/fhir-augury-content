## Summary

The reporter (Patricia Taylor) observes that `Claim.procedure.type` and
`ExplanationOfBenefit.procedure.type` are bound (as an example binding) to the
FHIR-defined CodeSystem / ValueSet `ex-procedure-type`, whose only codes are
`primary` and `secondary`. She asserts that those two codes do not match the
values used by the real-world claim submission standards (e.g., X12 837),
which distinguish a *principal* procedure from *other* procedures (terminology
that aligns with ICD's `principal procedure` concept). The ticket asks the
Financial Management work group to **delete the existing example CodeSystem
and ValueSet and replace them with an ICD-aligned CodeSystem / ValueSet whose
codes are `principal` and `other`** (or otherwise reflect the standards
actually used for claim procedure ordering).

The most recent comment (pknapp, 2025-01-30) defers the substantive decision,
suggesting Mary Kay be consulted because she may know the source of this
element or whether X12 has an explicit equivalent concept. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced
by the FHIR Augury indexes for this ticket, so this Summary relies entirely
on the ticket body and the single comment.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the FHIR-defined example CodeSystem and ValueSet
`http://hl7.org/fhir/ex-procedure-type` with an ICD-aligned ValueSet whose
codes are `principal` and `other` (matching the reporter's request and
ICD-10-PCS / X12 conventions for distinguishing the principal procedure from
the remaining procedures on a claim). Concretely:

1. In `HL7/fhir`, retire the existing
   `CodeSystem/ex-procedure-type` (codes `primary`, `secondary`) — mark
   `status = retired` for back-compat rather than physically deleting, so
   existing instances do not become invalid against earlier publications.
2. Author a new CodeSystem (e.g., `procedure-type` with codes `principal` and
   `other`, plus definitional text grounded in ICD/X12 837 usage) and a
   matching ValueSet binding it.
3. Update `Claim.procedure.type` and `ExplanationOfBenefit.procedure.type` in
   the StructureDefinition `<differential>` blocks to bind (still as
   `example`) to the new ValueSet's canonical URL.
4. Add a release note / change-log entry describing the replacement and
   reminding implementers that the binding strength remains `example`, so
   alternate code systems (e.g., the actual X12 837 procedure-qualifier
   values) remain permitted.

#### Justification

The reporter's complaint is substantive: the current `primary`/`secondary`
codes do not appear in the canonical claim submission standards that
implementers map to/from, so the example offers no real implementation
guidance and is arguably misleading. ICD's `principal procedure` is a
well-established concept in claims and inpatient coding, and `principal` /
`other` is the same partitioning used by X12 837I (HI segment qualifiers
distinguish the principal procedure from other procedures). Because the
binding strength is `example`, replacing the contents of the example does
not break conformant implementations — they remain free to use any other
CodeSystem — so the disruption is limited to documentation, narrative, and
any tooling that hard-coded the old codes. Retiring (rather than deleting)
the old CodeSystem preserves resolvability for historical instances.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting a brand-new "principal/other" CodeSystem inside FHIR
Core, **withdraw the FHIR-authored example CodeSystem entirely** and
re-bind `Claim.procedure.type` and `ExplanationOfBenefit.procedure.type`
to a single ValueSet that *enumerates real-world sources*:

- The actual ICD-10-PCS section/qualifier concept for principal procedure
  (or an external CodeSystem hosted by UTG that mirrors the ICD principal
  vs. non-principal partition), **plus**
- The X12 external code list (or a named concept set drawn from it) for the
  837 procedure qualifiers, included via `compose.include.valueSet` /
  external CodeSystem references rather than an HL7-invented enumeration.

Binding strength remains `example`. The deprecated `ex-procedure-type`
CodeSystem is retired as in Disposition A. New terminology lives in
`HL7/UTG` (not inline in `HL7/fhir`) so that updates can ship on the UTG
release cadence, independent of FHIR Core.

#### Justification

The underlying complaint is not really "the codes are wrong" — it is "the
example does not point implementers at the standards they actually use."
Inventing a second small HL7-authored vocabulary (`principal`/`other`)
addresses the cosmetic complaint but repeats the original mistake: a
two-code FHIR-invented enumeration that any real claim system will still
have to map to/from X12 and ICD. Pointing the example binding at *external*
code systems (or UTG-hosted mirrors of them) tells implementers exactly
where the authoritative values live, removes the temptation to treat the
HL7 example as normative, and avoids future churn when X12 or ICD evolve.
The trade-off vs. Disposition A is implementation cost (UTG coordination,
licensing review for ICD / X12 content) and a slightly less self-contained
example, but the result is more defensible for FM's R5 review of this
binding.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep `CodeSystem/ex-procedure-type` (`primary`,
`secondary`) as-is. Rationale category: *example bindings are
non-normative and the existing mechanism already addresses the need.*
Optionally, sharpen the narrative around `Claim.procedure.type` and
`ExplanationOfBenefit.procedure.type` to remind readers that the binding
strength is `example` and that real implementations should bind to whatever
local/regional code system (ICD, X12, or jurisdictional equivalent) they
already use.

#### Justification

The binding is `example`, which by definition is illustrative only and
imposes no conformance requirement. Implementers in claim-submission
contexts are already expected to bind to the appropriate
jurisdiction-specific code system in their profiles (US Da Vinci /
PAS-style IGs, regional payer IGs, etc.), and they routinely do. Replacing
the example with a different invented enumeration risks giving the *new*
codes unwarranted normative weight and may collide with downstream IGs that
have already chosen their own bindings. A documentation-only clarification
is cheaper, lower-risk, and avoids breaking any tooling that has the
existing CodeSystem URL embedded.

---

### Recommendation

**Recommended disposition:** B

The reporter's substantive point — that `primary`/`secondary` does not
match any real claim-submission vocabulary — is correct, so Disposition C
(decline) understates the problem; an example that nobody would
realistically copy is not a useful example, even if it is technically
non-binding. Disposition A fixes the names but repeats the structural
problem (an HL7-invented two-code enumeration parallel to ICD/X12), which
is exactly the criticism pknapp's comment hints at when it suggests
checking with Mary Kay for the X12 equivalent. Disposition B addresses the
underlying need by pointing the example at the standards implementers
actually use (ICD principal-procedure semantics and the X12 837 procedure
qualifiers, hosted via UTG), keeps the binding strength at `example` so no
existing conformance is broken, and aligns with how Financial Management
has historically handled claim-domain vocabulary (external code systems,
not FHIR-invented enumerations). It does cost more — UTG coordination and
a licensing/IP review for ICD/X12 content — and the workgroup may
reasonably fall back to A as a tactical fix if those external dependencies
cannot be cleared in the R5 timeframe.
