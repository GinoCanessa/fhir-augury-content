## Summary

This CARIN IG for Blue Button (C4BB) ticket asks that the canonical URL for the
`AHANUBCPresentOnAdmission` ValueSet be anchored in `terminology.hl7.org`
(THO) rather than under `hl7.org/fhir/us/carin-bb`. The reporter notes that
only ValueSets bound as `required` to a `code` data type should remain
anchored in the IG; everything else — and in particular ValueSets that simply
enumerate an external code system such as the AHA NUBC Present-on-Admission
indicator codes — belongs in THO. The ticket also asks that a corresponding
UTG ticket be opened to register the ValueSet in THO.

The ticket was originally part of **Block-Vote-6** in the CARIN BB STU2
ballot cycle and was dispositioned **Considered for Future Use** on
2022-07-26 (vote 13-0-0) with the rationale that this is a ValueSet
including all codes from a complete external code system, and that the
broader fix depends on UTG ticket [UP-316](https://jira.hl7.org/browse/UP-316)
("add 'all codes' ValueSets" in THO). Once UP-316 is delivered the IG-local
ValueSet can be removed and bindings can point directly at the THO ValueSet.
That resolution was subsequently reverted by Lynn Laakso to be re-opened
post STU2 publication, leaving the ticket in **Triaged** status today.
There are no explicit Jira links and no GitHub cross-references; the only
related discussion is the FM workgroup Block-Vote-6 announcement on Zulip.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor `ValueSet-AHANUBCPresentOnAdmission` in THO. Concretely:

- Open (or attach to the existing) UTG ticket to register a
  `terminology.hl7.org`-hosted ValueSet that enumerates the AHA NUBC
  Present-on-Admission code system (`https://www.nubc.org/CodeSystem/PointOfOrigin`
  family — exact NUBC URI as already used in C4BB), keyed off the
  external code system rather than off the IG.
- Once published in THO, change the C4BB ValueSet's canonical URL from
  `http://hl7.org/fhir/us/carin-bb/ValueSet/AHANUBCPresentOnAdmission`
  to a `http://terminology.hl7.org/ValueSet/...` URL.
- Update every binding in C4BB profiles (notably the institutional EOB
  profile slices on `supportingInfo`/`item` that reference Present on
  Admission) to point at the new THO URL.
- Retire the IG-local ValueSet (leave a redirect/retired note so existing
  implementers are not silently broken).

#### Justification

This is exactly the architectural pattern HL7 has been steering ValueSets
that wrap external code systems toward — IGs should not own the canonical
URL for a generic, reusable enumeration of a third-party code system.
Moving it to THO makes the ValueSet reusable by Da Vinci, US Core, and
other Financial Management IGs, and aligns with the rationale recorded in
the Block-Vote-6 disposition ("plan on making change when external
CodeSystem 'all codes' ValueSet is published"). The reporter's framing —
"only those with required binding to a `code` data type should be
anchored in `hl7.org/fhir`" — is consistent with FHIR's general guidance
that IG-local ValueSets are reserved for IG-specific constraints, not for
mirroring external terminologies.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the canonical-URL move but address the underlying need now by
**replacing the explicit IG-local ValueSet with a THO "all codes"
ValueSet binding** as soon as [UP-316](https://jira.hl7.org/browse/UP-316)
delivers the generic mechanism. In the interim:

- Mark the C4BB `AHANUBCPresentOnAdmission` ValueSet as deprecated /
  transitional in its description, pointing readers at the planned THO
  ValueSet.
- Keep the existing canonical URL stable for STU2 implementers (no
  breaking change mid-cycle), and schedule the actual canonical-URL
  switch for the next substantive C4BB version that follows UP-316.
- Coordinate with Vocabulary / UTG so the AHA NUBC PoA enumeration is
  registered as a THO "all codes" ValueSet rather than as a hand-curated
  copy.

#### Justification

This honours both halves of the original Block-Vote-6 reasoning: the
change is the right thing to do, but it is **gated on UP-316**, and
shipping a new THO canonical now (before the "all codes" pattern is
published) risks creating a one-off ValueSet that itself has to be
re-anchored later. It also avoids breaking existing STU2 conformance
URLs and validators mid-stream, which is a real concern for a published
US Realm IG with payer implementers in production.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the IG-local
ValueSet is a thin wrapper around a complete external (AHA NUBC) code
system, is fairly specific to C4BB's institutional EOB use case, and
"does not warrant the maintenance of moving to THO" — the same rationale
the FM workgroup applied to several sibling Block-Vote-6 tickets
(e.g. FHIR-35403, FHIR-35409, FHIR-35413, FHIR-35414, FHIR-35415,
FHIR-35432, FHIR-35434, FHIR-35435, FHIR-35436, FHIR-35439, FHIR-35446,
FHIR-35447). Leave the canonical URL anchored in
`http://hl7.org/fhir/us/carin-bb/...`.

#### Justification

The AHA NUBC Present-on-Admission enumeration is small and overwhelmingly
used by C4BB's institutional EOB profile; there is little observed reuse
demand from other IGs. Moving it to THO incurs ongoing UTG/governance
overhead, breaks existing canonical URLs that payers and validators have
been using since STU1, and creates churn that is not justified by clear
implementer benefit. The workgroup has already used this exact rationale
to decline structurally identical sibling tickets in the same block vote.

---

### Recommendation

**Recommended disposition:** B

The ticket is being re-opened post-STU2 precisely because the original
"Considered for Future Use" disposition was correct in principle but
needs to be re-validated now that some time has passed. Disposition B
preserves that intent: the workgroup commits to moving
`AHANUBCPresentOnAdmission` (and its Block-Vote-6 siblings that share
the "Plan on making change when … UP-316 is published" rationale) to a
THO "all codes" ValueSet on the next substantive C4BB version, while
avoiding a mid-STU2 breaking change to canonical URLs that payers are
already conforming to. Disposition A is the right end state but is
risky to execute as a one-off ahead of UP-316 — it would create a
bespoke THO ValueSet that may itself be re-anchored once the "all codes"
pattern lands. Disposition C is defensible and consistent with how
several sibling Block-Vote-6 tickets were handled, but it contradicts
the reporter's explicit architectural argument and the workgroup's own
acknowledgement (in the reverted resolution comment) that the move is
the desired direction once tooling supports it. Pairing this ticket
with a status check on [UP-316](https://jira.hl7.org/browse/UP-316) at
disposition time would let the workgroup pick a concrete target version.
