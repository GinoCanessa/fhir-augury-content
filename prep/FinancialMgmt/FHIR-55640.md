## Summary

The PDex IG ships two parallel sets of examples for the
`$provider-member-match` operation, and they disagree on naming and on
the operation's intended cardinality. `Parameters-provider-member-match-request-001`
calls itself "Provider Member Match", lists multiple members, and uses
the top-level parameter name `MembersToMatch`. `Parameters-provider-bulk-member-match-in`
calls itself "Provider Bulk Member Match", contains only a single member,
and uses the top-level parameter name `MemberBundle`. The reporter notes
that, to their knowledge, there is only one provider member match
operation and it is bulk-style — so two examples that disagree on both
the operation's identity and its parameter names are confusing for
implementers and reviewers alike.

This sits directly on top of [FHIR-56075](https://jira.hl7.org/browse/FHIR-56075)
("Discrepancies in between operation for $provider-member-match between
IG and package operation definition"), which renamed the wire-format
parameters in the underlying profile so that the multi-member-match
bundle uses `MembersToMatch` (container) and `TreatmentAttestation` (part)
rather than `MemberBundle` / `Consent`. The older example files were
authored before that rename and were never reconciled, leaving the two
inconsistent variants the reporter observed.

## Proposed Dispositions

The Persuasive resolution already on the ticket corresponds to
Disposition A below. Dispositions B and C are reconstructed alternatives
the workgroup could weigh against it.

### Disposition A: Accept As Requested

#### Proposal

Treat the duplicate examples as accumulated cruft from before
FHIR-56075's rename and delete the stale set, keeping a single
canonical pair under the renamed parameter names. Concretely, in
`HL7/davinci-epdx` under `input/fsh/`:

- Delete the request/response example pair built around the old
  `MemberBundle` / `Consent` slice names — i.e. the FSH instances that
  produce `Parameters-provider-bulk-member-match-in` and
  `Parameters-provider-bulk-member-match-out`, together with the six
  patient example instances those examples reference.
- Retain `Parameters-provider-member-match-request-001` (using
  `MembersToMatch` + `TreatmentAttestation`, two members) and
  `Parameters-provider-member-match-response-001` as the single
  canonical pair for `$provider-member-match`.
- Update `input/fsh/Operation-provider-member-match.fsh` /
  `input/pagecontent/`-narrative so that any prose that still points at
  the removed `provider-bulk-member-match-*` pages is rewritten to
  reference the surviving canonical examples.
- Add a `changehistory.md` entry crediting FHIR-55640 (already done in
  commit `5c834d9`).

#### Justification

This is exactly what the reporter asked for ("will the real provider
member match parameters example stand up?") and it is the minimum
change that resolves the visible inconsistency for implementers
browsing the IG. It also keeps the example wire-format aligned with the
profile rename made in FHIR-56075, so the OperationDefinition,
StructureDefinition, and Examples all tell the same story. The IG
playbook's "Add an example" recipe explicitly co-locates examples with
their profile in `input/fsh/`; deleting the stale FSH instances is the
clean way to apply that recipe in reverse without leaving orphan JSON
in `fsh-generated/` or `output/`. The 379-line delta in the resolution
commit confirms the change is contained to example artifacts and
does not touch the operation's normative contract.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both example pairs but recast them as deliberately distinct
fixtures: rename the surviving `Parameters-provider-bulk-member-match-*`
files to make clear they exemplify the *single-member* call shape
(e.g. `Parameters-provider-member-match-request-single` /
`-response-single`), update their top-level parameter names from
`MemberBundle` / `Consent` to the post-FHIR-56075 names
`MembersToMatch` / `TreatmentAttestation`, and reduce them to one
member by design. Then reposition the existing `…request-001` /
`…response-001` pair as the multi-member exemplar. Update
`Operation-provider-member-match.fsh` to link both examples from its
Description and call out which scenario each one illustrates.

#### Justification

The operation is genuinely capable of being invoked with one or many
members; a reader landing on the IG learns more from a curated
single-member example *plus* a curated multi-member example than from
a single canonical pair. This addresses the reporter's confusion
about the "bulk" label by making the difference explicit (count of
members) rather than nominal (different parameter names). The cost is
two example files to maintain instead of one and a small amount of
narrative coordination on the Operation page; the gotcha to weigh is
the flat `input/fsh/` layout, which means both examples will sit as
siblings and must be discoverable by filename prefix — manageable, but
slightly noisier than Disposition A.

---

### Disposition C: Decline

#### Proposal

Leave the two example variants in place and instead address the
confusion with narrative only — add a paragraph to
`Operation-provider-member-match.fsh` (and/or
`input/pagecontent/`) explaining that the older
`Parameters-provider-bulk-member-match-*` examples are retained for
historical continuity and that the `…request-001` / `…response-001`
examples are the current canonical form. Mark the older examples as
"deprecated example" via a note rather than removing them.

#### Justification

Defensible only if there is a concrete consumer (test suite, package
dependent, or implementer reference) that pins to the literal
filenames `Parameters-provider-bulk-member-match-in/out` and would
break on removal. PDex publishes versioned packages and the README is
explicit that recorded versions cannot be removed once registered, so
there is some precedent for preferring deprecation over deletion.
Outside that narrow case the rationale is weak: the deprecated
examples use parameter names (`MemberBundle`, `Consent`) that no
longer match the profile after FHIR-56075, so retaining them keeps
publishing artifacts that contradict the normative OperationDefinition.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right call and matches what was already applied.
The two example sets do not represent two real operation flavours —
they are a single operation captured at two different points in the
profile's naming history, and FHIR-56075 has already chosen the names
(`MembersToMatch`, `TreatmentAttestation`). Keeping the stale pair
either as-is (Disposition C) or rebadged (Disposition B) preserves
content that contradicts the very profile and OperationDefinition the
examples are supposed to illustrate, which is the failure mode the
reporter flagged in the first place. Disposition B's "single vs.
multi-member" framing is appealing in the abstract, but the operation
is intrinsically multi-member (the renamed container is literally
`MembersToMatch`) — a one-member instance is just a degenerate case
and does not need its own canonical example to be understandable.
Disposition A is also the smallest defensible change: it is contained
to `input/fsh/` example files plus a `changehistory.md` entry, leaves
the OperationDefinition and profiles untouched, and removes 379 lines
of confusing wire-format from the published IG. The workgroup should
ratify the Persuasive resolution as applied.
