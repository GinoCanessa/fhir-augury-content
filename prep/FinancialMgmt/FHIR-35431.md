## Summary

The ticket requests that the canonical defining URL for the CARIN Blue
Button ValueSet `C4BBRelatedClaimRelationshipCodes` be anchored in
terminology.hl7.org (THO) rather than hl7.org/fhir. The reporter
(Joan Harper) notes that a companion UTG ticket should be opened to
register the ValueSet in THO so that CARIN BB can reference the
terminology from its canonical home. The ticket was originally
dispositioned "Considered for Future Use" during Block-Vote-6 (2022-07-26,
13-0-0) with a plan to make the change once the external CodeSystem
"all codes" ValueSet mechanism (UP-316) was available, but that
resolution was reverted by Lynn Laakso to be re-opened post STU2
publication. The ticket is tightly coupled with the related CARIN BB
ticket [FHIR-36487](https://jira.hl7.org/browse/FHIR-36487) (change
the related claim relationship type from an example code system to a
non-example THO CodeSystem) and the THO/UTG tickets
[UP-313](https://jira.hl7.org/browse/UP-313) (create the non-example
Related Claim Relationship CodeSystem/ValueSet in THO) and
[UP-316](https://jira.hl7.org/browse/UP-316) (add "all codes"
ValueSets for external CodeSystems in THO). Once those upstream THO
changes land, the CARIN BB IG-local ValueSet can be removed and
bindings redirected to the THO-anchored ValueSet.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rehome the ValueSet `C4BBRelatedClaimRelationshipCodes` under
terminology.hl7.org. Concretely:

1. File (or confirm) a UTG ticket to register
   `ValueSet-C4BBRelatedClaimRelationshipCodes` in THO — coordinated
   with UP-313/UP-316 which cover the underlying non-example
   CodeSystem and the "all codes" ValueSet pattern.
2. In the CARIN BB IG, change the ValueSet's canonical URL from
   `http://hl7.org/fhir/us/carin-bb/ValueSet/C4BBRelatedClaimRelationshipCodes`
   to a `http://terminology.hl7.org/ValueSet/...` canonical aligned
   with the THO registration from UP-313.
3. Retarget every binding that currently points at the CARIN BB
   canonical (notably the `Claim.related.relationship` / C4BB
   ExplanationOfBenefit profile bindings) to the new THO canonical.
4. Remove the IG-local ValueSet definition once the THO ValueSet is
   published, leaving only the binding reference.

Bundle this change with FHIR-36487 so the CodeSystem move (example →
non-example THO CS) and the ValueSet rehoming ship atomically.

#### Justification

This matches the explicit request in the ticket, Ron Parker's
comment-43 guidance, and HL7's long-standing rule that terminology
artifacts with cross-IG reusability belong in THO rather than in an
IG namespace. Lynn Laakso's re-open comment makes clear the original
"future use" deferral is no longer the WG position now that STU2 has
published. Keeping the binding target in THO also resolves the
mismatch with FHIR-36487 (example CodeSystem) in a single coherent
terminology refactor.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the CARIN BB IG ValueSet URL as-is for STU2.x and instead
publish the THO ValueSet (via UP-313 / UP-316) as the forward-looking
binding target, then flip the CARIN BB binding to the THO canonical
in the next substantive CARIN BB release (STU3). In the interim:

- Add an informative note to the CARIN BB IG narrative stating the
  IG-local ValueSet is deprecated-on-arrival and identifying the THO
  successor canonical.
- Leave the current `hl7.org/fhir/us/carin-bb/...` canonical in
  place so already-deployed CARIN BB consumers are not broken by a
  canonical URL change mid-STU2 cycle.

#### Justification

Changing a ValueSet canonical URL is a breaking change for any
validator or CapabilityStatement already configured against the old
URL. The sibling tickets (FHIR-35426 / -35427 / -35428) were
originally dispositioned "Considered for Future Use" precisely
because the UTG-side enablers (UP-313, UP-316) were not yet done.
Deferring the binding flip to the next major CARIN BB release keeps
the STU2 publication stable while still committing to THO as the
long-term home — which is what Lynn Laakso's revert effectively asks
for ("re-open post STU2 publication"). Trade-off vs. A: slower
cleanup and a temporary duplicated definition, but lower
implementer disruption.

---

### Disposition C: Decline

#### Proposal

Persistent Disposition: **Not Persuasive — Already Addressed**.
Close FHIR-35431 without further CARIN BB spec action on the grounds
that the underlying concern (terminology belongs in THO) is being
handled entirely by the UTG workstream (UP-313, UP-316) and the
CARIN BB IG will simply pick up the THO canonical once that
workstream delivers. Redirect any remaining CARIN BB-side work into
FHIR-36487.

#### Justification

Defensible only if the WG judges that the canonical-URL rehoming is
a pure consequence of the UTG work and introduces no design
decisions on the CARIN BB side. However, this stance is weak because
the CARIN BB IG must still actively change its binding canonicals
and remove the IG-local ValueSet definition when THO lands — that is
spec work FHIR-35431 uniquely tracks. Declining risks the change
being forgotten between the UTG delivery and the next CARIN BB
release.

---

### Recommendation

**Recommended disposition:** A

The ticket has been explicitly re-opened by the WG co-chair (Lynn
Laakso) after STU2 publication, which signals that the earlier
"future use" deferral (Disposition B's posture) is no longer the
WG's preferred path. The underlying THO enablers referenced in the
2022 comment thread (UP-313 for the non-example CodeSystem, UP-316
for the "all codes" ValueSet pattern) have had substantial time to
mature, and this ticket is tightly coupled with FHIR-36487, which
requires a coordinated terminology change in the same artifact — so
rolling both tickets into a single THO rehoming is the lowest-risk
path for implementers (one binding flip, not two). Disposition B's
backward-compatibility concern is real but is better handled via a
standard `ValueSet.identifier` / deprecation note on the old
CARIN BB canonical, not by deferring the fix again. Disposition C
understates the CARIN BB-side spec work still required. The WG
should accept the request and schedule the change for the next
CARIN BB publication window, explicitly paired with FHIR-36487.
