## Summary

The reporter argues that the `Contract` resource conceptually belongs to the
**Event** pattern within the Workflow module, but its current definition tries
to carry the activities that lead up to contract activation (negotiation,
authoring, agreement, signing) inside the Contract instance itself. That mixes
multiple concepts that, under the FHIR Workflow patterns, would normally be
split across `Definition` (the template / boilerplate), `Request` (the
proposed/ordered agreement), and `Event` (the executed/active contract). The
Workflow / FHIR-I community asked Financial Management to discuss the intent of
`Contract` and how its life-cycle should be managed — particularly how `Task`
should drive contract preparation, signing, amendment, and termination instead
of being collapsed into Contract's own state machine.

This ticket is the carryover of the second item from
[FHIR-26947](https://jira.hl7.org/browse/FHIR-26947) ("Contract should be
listed under Event in the Workflow module") — the first item (relisting
Contract under Event) was applied as Persuasive, but the deeper conversation
about the resource's intent and its relationship to `Task` was deferred here.
The conversation has since matured into
[FHIR-55563](https://jira.hl7.org/browse/FHIR-55563) ("Contract is not ready
to go normative", February 2026), which was Resolved Persuasive with
Modification with a decision to pull `Contract` out of the R6 normative
specification and host it in an "in development" specification while the
refactor work is completed.

## Details

**Description:**

> Contract properly belongs to the Event pattern. The current definition of
> the Contract resource seems to attempt to track activities related/leading
> to the Contract activation within the contract itself. This seems to be
> conflating multiple concept that are in discord with the general
> Request/Event/Definition patterns.
>
> The Workflow FHIR-I group would like to discuss the intent of the Contract
> resource, and how it can work with other resources like Task to manage the
> associated life-cycle of the contract.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Hold the requested joint Financial Management × FHIR-I Workflow discussion and
**refactor `Contract` to fit cleanly into the Event pattern**, externalising
its preparatory life-cycle to `Task`:

- Strip out (or deprecate) the elements of `Contract` that model
  pre-activation activities — for example, the inner negotiation/term-status
  bookkeeping currently embedded in `Contract.term` and the auxiliary status
  flags that drive its second state machine.
- Reduce `Contract.status` to an Event-pattern status (drawing from
  `event-status` / a constrained subset, e.g., `preparation | in-progress |
  completed | entered-in-error | …`) so the remaining state machine is the
  one inherited from Event.
- Use `Task` (with `Task.focus` or `Task.basedOn` pointing at the Contract,
  following the FHIR-22572 / FHIR-22578 precedent) to represent the
  authoring, negotiation, signing, amendment, and termination steps. Each
  Task carries its own Request-pattern state.
- For pure template/boilerplate content, lean on the existing
  `Contract.contentDefinition` (or a sibling Definition-pattern resource) so
  the Definition / Request / Event split is explicit.
- Update the workflow.html / contract.html spec narrative and Workflow module
  diagrams so that `Contract` appears unambiguously under Event with a single
  state machine.

#### Justification

This is the change the reporter asked for, and it directly aligns the
resource with the Workflow patterns FHIR has documented as normative
guidance. It is also consistent with how the workgroup is already handling
analogous cases (FHIR-22572/22578 use `Task` for referral state; FHIR-38870
is hardening the rule that pattern membership implies pattern-shaped
fields). Critically, **this approach is now compatible with the FHIR-55563
resolution**: Contract is being pulled from the R6 normative track into an
"in development" spec specifically so that this kind of structural rework
can be done without breaking a normative artifact. That removes the usual
"too disruptive for normative" objection.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Contract`'s existing structure largely intact for backwards
compatibility, but **document a usage pattern** — without changing the
resource definition itself — that says:

- Implementers SHOULD treat the `Contract` resource as the Event-pattern
  artifact (the active, in-force agreement) and use `Task` to drive any
  authoring, negotiation, signing, or amendment workflow that surrounds it.
- The pre-activation elements currently on `Contract` are **retained but
  marked as informative / legacy** in the spec narrative, and the Workflow
  module page calls them out as "do not use these to drive workflow; use
  `Task` instead."
- Add a non-normative "Contract Workflow" sub-page (sibling to
  `contract-examples.html`) that walks through a worked example: a
  `Contract` in `Contract.status = draft`, surrounded by a `Task` chain that
  authors, reviews, signs, and activates it; on activation, the Contract
  flips to an Event-aligned status.

This would also tie naturally into FHIR-20370 — if the Event pattern needs
additional states to honestly express the contract life-cycle, that work
happens once at the pattern level instead of being duplicated inside
Contract.

#### Justification

A documentation-and-guidance disposition avoids breaking every existing
implementer that has built against the current Contract elements (and there
are several profile families that depend on the existing `term`/`signer`/
status structure). It still answers the conceptual concern — "Contract
conflates multiple patterns" — by drawing a clear Workflow-module line
between what `Contract` is *for* (Event) and what `Task` is *for*
(Request-pattern lifecycle around the Contract). Trade-off vs. Disposition
A: the inner conflation persists in the resource definition itself, so
tooling that walks the StructureDefinition will still see the messy mix;
implementers have to read the narrative to know what's "real" workflow and
what's vestigial.

---

### Disposition C: Decline (Close as Superseded)

#### Proposal

Close FHIR-26966 with a resolution of **Not Persuasive / Superseded** on the
grounds that the underlying issue is now being handled by
[FHIR-55563](https://jira.hl7.org/browse/FHIR-55563): "The Contract resource
will be pulled from the R6 normative specification and included in an 'in
development' specification." Any concrete refactor work — including the
Event-pattern alignment, the externalisation of authoring/signing to `Task`,
and the cleanup of multiple state machines — should be tracked against the
in-development specification under FHIR-55563's `FM-R5+_ToBeApplied` work
stream rather than continuing to age in this 2020-vintage Triaged ticket.
The closure comment should explicitly cite FHIR-55563 and FHIR-26947 as the
disposition trail.

#### Justification

FHIR-26966 has been Triaged with no substantive movement since 2021, and the
later FHIR-55563 effectively answers it at the spec-status level: the
workgroup has already acknowledged that Contract is not workflow-coherent
enough for normative and has chosen the structural remedy (pull it out of
normative and rework it). Keeping FHIR-26966 open as an independent
discussion thread risks divergent dispositions on the same underlying
problem and obscures the fact that real work now belongs in the
`FM-R5+_ToBeApplied` queue. Declining here is *not* declining the underlying
concern — it's redirecting it to the live ticket.

---

### Recommendation

**Recommended disposition: A (Accept As Requested), with FHIR-55563 cited as
the enabling context.**

The original 2020 objection has only strengthened with time: FHIR-55563
(February 2026) is essentially the workgroup re-discovering and validating
exactly what FHIR-26966 raised — that `Contract` carries multiple state
machines and conflates Definition / Request / Event concerns. With FHIR-55563
already removing Contract from the R6 normative track and parking it in an
"in development" specification, the usual blocker for a deep refactor (don't
break a normative resource) is gone, so this is the right moment to do the
work properly rather than paper over it with narrative. Disposition B is the
fallback if the joint FM × FHIR-I conversation reveals there is no appetite
for breaking changes even in the in-development spec; Disposition C is
reasonable bookkeeping but loses the specific framing (Event pattern + `Task`
externalisation, by analogy to FHIR-22572/22578) that this ticket
contributes to FHIR-55563. The cleanest outcome is to accept FHIR-26966,
explicitly link it as a child of FHIR-55563's `FM-R5+_ToBeApplied` work, and
let the technical proposal in Disposition A be the design brief for the
refactor.
