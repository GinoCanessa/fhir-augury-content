## Summary

The submitter (Casey Trauer) requests that the Da Vinci PDex IG provide a
**coded distinction between Consent resources that govern the three
CMS-mandated APIs**: Provider Access, Payer-to-Payer (P2P) and Patient
Access. The ticket reports that today the existing Consent coded
elements (`category`, `scope`, `policy`) are too vague to answer three
operational questions:

1. When a Patient submits a Consent through the Patient Access API that
   actually authorises a third-party app's *Provider Access*-style use,
   how should it be coded so the receiving system knows it applies to
   Provider Access?
2. When a member changes their Consent for a given API, how does an
   implementer locate the correct Consent to update?
3. At request-time, how does a system look up the Consent that
   determines sharing rules for a given API, given that for global
   opt-in/opt-out the request client cannot reliably be matched to
   `Consent.actor`?

The submitter also asks a side question about why P2P (HRex) and
Provider Access (PDex) are defined as separate Consent profiles. The
ticket has been discussed in the June 27 PDex workgroup meeting and is
flagged Fit-For-Teleconference and PDex-Vote-1. There are no
explicitly-linked Jira tickets; the only FHIR-element cross-reference is
to `Consent.actor`. The ticket is marked **Pre-Applied** and
**Persuasive**, with a resolution captured in commit
[`1c31c7e4`](https://github.com/HL7/davinci-epdx/commit/1c31c7e4a6ffaa1aee8eab2bd15d5e84837cfc56)
on `HL7/davinci-epdx`.

## Details

**Description:**

> We would like to have a coded distinction between Consent resources
> that apply to the different CMS-mandated APIs: Provider Access, Payer
> to Payer and Patient Access.
>
> Right now, it is difficult to address several issues when there are
> multiple Consent policies for a given member:
>
> 1. If a Patient submits a Consent through Patient Access API for
>    Provider Access by a third-party application, how would that need
>    to be coded so we know it is applying to Provider Access?
> 2. When a member changes their Consent for a given API in any
>    context, how can I locate the correct Consent to update?
> 3. If my system wants to look up the Consent to determine sharing
>    rules at the time of request, how would it know which Consent to
>    reference? (For global opt-in/opt-out, we cannot rely on
>    cross-referencing the client making the request to the
>    `Consent.actor`. Plus, that path would just complicate
>    implementations.)
>
> We are finding the existing coding elements – category, scope, policy
> – too vague for these use cases.
>
> Side note: Why do we have different Consent profiles for P2P (HRex)
> and Provider Access (PDex)?
>
> *(Discussed in June 27 PDex workgroup meeting.)*

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change already pre-applied in commit
[`1c31c7e4`](https://github.com/HL7/davinci-epdx/commit/1c31c7e4a6ffaa1aee8eab2bd15d5e84837cfc56)
on `HL7/davinci-epdx`:

1. Add `PDexConsentApiPurposeCS` CodeSystem and `PDexConsentApiPurposeVS`
   ValueSet in `input/fsh/CodeSystemOptOutScope.fsh` with the three codes
   `provider-access`, `payer-to-payer`, `patient-access`.
2. Add a 1..1 MS `category[apiPurpose]` slice on
   `ProfilePDexConsentProviderSharingOptOut.fsh:18`, fixed to
   `provider-access`, so that any conformant Provider Access Consent
   carries an explicit, lookup-friendly category discriminator.
3. Add the "Differentiating Consent Resources by API" section to
   `input/pagecontent/consent.md` documenting:
   - The mapping API → category code → profile.
   - The recommended request-time / update-time search pattern
     `GET /Consent?patient=[id]&category=provider-access` (and parallel
     queries for the other two codes).
   - The opt-in (P2P) vs opt-out (Provider Access) semantic rationale
     for keeping P2P (HRex) and Provider Access (PDex) Consent profiles
     separate.

To fully close the submitter's three questions, **also extend the same
`category[apiPurpose]` slice pattern (fixed to `patient-access` and
`payer-to-payer` respectively) to PDex's Patient-Access Consent profile
and to the HRex Payer-to-Payer Consent profile** (the latter
coordinated with the HRex workgroup), and add an `_search_` example for
each in `consent.md`.

#### Justification

The ticket asks for a coded distinction usable in standard FHIR search;
adding a fixed `category` slice directly satisfies that need without
introducing a new extension or repurposing `scope`/`policy`. It uses an
existing FHIR Consent search parameter (`category`), which means
implementers get a request-time, update-time and opt-in/out lookup path
without any out-of-band convention. The repo briefing confirms that
ValueSet / CodeSystem changes belong in
`CodeSystemOptOutScope.fsh` and that profile slicing belongs in
`Profile…OptOut.fsh` under `input/fsh/`, so the change sits naturally in
the existing authoring layout. The labels **Pre-Applied** and
**Persuasive**, plus the recorded resolution, indicate that the
workgroup has effectively already accepted this approach. Extending the
same slice symmetrically to the Patient Access and P2P Consent profiles
is what closes question (1) — a Consent submitted via Patient Access
API but authorising Provider Access carries
`category = provider-access`, regardless of submission channel.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a fixed `category[apiPurpose]` slice on each Consent profile,
define a **PDex extension** (e.g. `Extension: PDexConsentApiPurpose`,
`Context: Consent`, `value[x]` bound to `PDexConsentApiPurposeVS`,
0..1 MS) and add a **PDex SearchParameter** indexing that extension
(e.g. `pdex-consent-api-purpose`). Document the
`GET /Consent?patient=[id]&api-purpose=provider-access` search pattern
in `consent.md`. Optionally, on the existing
`ProfilePDexConsentProviderSharingOptOut` the extension could be
constrained to `1..1 MS` and `valueCode` fixed to `provider-access`, but
the discriminator would not require the `category` slice machinery.

#### Justification

An extension + custom SearchParameter is a more loosely-coupled
mechanism: it does not constrain `Consent.category` (which is already
used for opt-out scope encoding via `OptOutScope` in PDex) and avoids
forcing every PDex Consent profile to carry a fixed-value category
slice. It also gives the workgroup a single canonical
`api-purpose` SearchParameter to point implementers at, regardless of
profile. Trade-offs vs Disposition A: extensions require implementers to
adopt a new search parameter and a new extension URL (both PDex-local),
while Disposition A reuses the standard `category` element and standard
`Consent?category=` search, which most FHIR servers already index out of
the box. The HRex P2P profile would still need to opt in to the same
extension to make the discriminator work uniformly across the three
APIs.

---

### Disposition C: Decline

#### Proposal

Decline the request as already addressed by existing mechanisms:
implementers should use **the existing `Consent.policyRule` /
`Consent.policy.uri`** (or, depending on Consent version, a profile-level
`meta.profile` filter combined with the existing PDex `OptOutScope`
codes) to differentiate Consents per API. The narrative in
`consent.md` would be updated to make the recommended search pattern
explicit (e.g. `GET /Consent?patient=[id]&_profile=…`), but no new
CodeSystem, ValueSet or profile slice would be introduced.

#### Justification

Declining is defensible only if the workgroup believes the existing
elements are sufficient and the perceived ambiguity is documentation-
shaped rather than model-shaped. The grounds would be: (a) the
`PDexConsentProviderSharingOptOut` profile already implies the API
context via its profile identity; (b) `policyRule` / `policy` already
exist for distinguishing the legal/regulatory basis; (c) introducing
a fourth coded discriminator risks encoding the same fact twice. The
weakness, which the submitter explicitly names, is that
`_profile=` search support is uneven across servers and that
`policy`/`scope` are too vague for the day-to-day
locate-and-update-the-right-Consent flow. Choosing this disposition
would require the workgroup to publish very prescriptive narrative
guidance in `consent.md` to compensate.

---

### Recommendation

**Recommended disposition:** A

The ticket is already labelled **Pre-Applied** and **Persuasive**, the
resolution is recorded on the ticket snapshot, and commit
[`1c31c7e4`](https://github.com/HL7/davinci-epdx/commit/1c31c7e4a6ffaa1aee8eab2bd15d5e84837cfc56)
implements exactly the shape of change Disposition A describes. The
resolution leverages a standard FHIR Consent search parameter
(`category`), which gives implementers the request-time, update-time
and locate-the-right-Consent lookup path the submitter asked for —
without needing a PDex-local SearchParameter (Disposition B) or relying
on uneven `_profile` support (Disposition C). The one gap to flag for
the teleconference is that the pre-applied commit only adds the slice
to the **Provider Access** profile; the workgroup should explicitly
confirm it also wants the symmetric slices on the Patient Access PDex
Consent profile and on the HRex P2P Consent profile (with HRex
coordination), so that question (1) — a Patient Access submission that
authorises Provider Access — is unambiguously coded as
`category = provider-access` no matter how it was submitted.
