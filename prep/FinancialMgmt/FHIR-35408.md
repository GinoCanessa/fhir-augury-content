## Summary

FHIR-35408 asks the CARIN IG for Blue Button® (C4BB) workgroup to relocate the
canonical (defining) URL of the `AMACPTCMSHCPCSModifiers` ValueSet from the
`hl7.org/fhir` namespace into the terminology.hl7.org (THO) namespace, so that
the value set is governed and published as a HL7 terminology artifact rather
than as IG-local content. The reporter explicitly notes that a companion UTG
ticket should be created to register the value set in THO. The change is one of
a coordinated cluster of "Defining URL should be anchored in THO not
hl7.org/fhir" tickets that Joan Harper filed against the C4BB STU2 ballot — the
related items in the same block (e.g., FHIR-35389/35394/35397/35400/35410/35420
…) target the analogous CodeSystem and ValueSet artifacts in C4BB. The
workgroup originally bundled FHIR-35408 into Block-Vote-5 (voted 2022-07-12,
7-0-2) with a resolution of *Considered For Future Use* — "Currently in
development for a future version" — but Lynn Laakso re-opened it on 2022-11-28
post-STU2 publication, so it is now back in Triaged state awaiting a fresh
disposition for the next C4BB release.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the request and schedule it for the next C4BB publication. Concretely:

1. File a UTG ticket against `HL7/UTG` to register the
   `AMACPTCMSHCPCSModifiers` ValueSet in THO. The ValueSet content (CPT/HCPCS
   modifier composition) and its naming should be carried over verbatim so
   semantics are preserved.
2. Once the UTG ticket is approved and the THO publication URL is reserved
   (under `http://terminology.hl7.org/ValueSet/...`), update the C4BB IG to
   either (a) remove the locally-defined ValueSet and re-bind every reference
   to the new THO canonical, or (b) leave a same-name shim ValueSet that
   `compose.include`s the THO ValueSet for one publication cycle to ease
   implementer migration.
3. Mark the original CARIN-namespaced canonical as a deprecated alias in the
   C4BB IG `package-list.json` / migration guide so existing implementations
   are not silently broken.

#### Justification

This is what the reporter literally asked for and is the standard HL7
governance pattern: terminology artifacts that are not IG-specific belong in
THO so they can be reused across realms and IGs without each IG re-defining
them. Lynn Laakso's 2022-11-28 reopen comment ("Working on putting in THO for
future version") indicates that the workgroup already intends to move in this
direction; formalizing it as Accept aligns the disposition with the actual
plan and lets the UTG work proceed without another round of triage. The
Block-Vote-5 thread shows no substantive opposition — only the procedural
deferral that has since been reverted.

---

### Disposition B: Persuasive with Modification — Move only when reuse is established

#### Proposal

Adopt the *intent* of the ticket (THO is the right long-term home) but only
migrate the canonical when the ValueSet has demonstrable reuse outside C4BB.
Specifically:

1. Keep the current `AMACPTCMSHCPCSModifiers` defining URL in the C4BB IG for
   the next publication so existing CARIN BB implementers experience zero URL
   churn.
2. File the UTG ticket now to *reserve* the THO canonical and mirror the
   ValueSet definition in THO, but mark the C4BB-local copy as "draft for
   migration" and keep both in sync until at least one other IG (e.g., Da Vinci
   PDex, US Drug Formulary, or Da Vinci HRex) consumes the THO version.
3. Once external reuse is observed, retire the C4BB-local URL with a
   deprecation cycle and re-point all C4BB bindings at the THO canonical.

#### Justification

The C4BB STU2 publication has already shipped with the `hl7.org/fhir`-anchored
URL, and a hard URL change is a breaking change for every consumer that
hard-codes canonicals (terminology servers, validators, payer mapping
spreadsheets). Several sibling tickets in the same Block-Vote-5 batch were
resolved *Not Persuasive* with the rationale "Simple ValueSet with little
re-use value" (e.g., FHIR-35412/35416/35418/35419/35421/35422), indicating the
WG has previously rejected URL migration when reuse was speculative.
AMACPTCMSHCPCSModifiers is plausibly more reusable than those (CPT/HCPCS
modifiers are claims-domain-wide) but that reuse is not yet demonstrated; a
phased "reserve in THO, migrate when reused" approach captures the governance
benefit without imposing breakage on STU2 implementers, and keeps the
disposition consistent with how the WG handled similar tickets in 2022.

---

### Disposition C: Decline / Not Persuasive

#### Proposal

Close FHIR-35408 as *Not Persuasive* with the rationale that the ValueSet is a
C4BB-specific composition of pre-existing external code systems (AMA CPT, CMS
HCPCS) and does not warrant a separate THO registration. Keep the ValueSet's
current `hl7.org/fhir/...` (CARIN-namespaced) canonical and instead invest the
WG's terminology effort in the sibling tickets where the underlying *code
systems* (FHIR-37334 HCPCS, FHIR-37335 ADA CDT) were judged Persuasive.

#### Justification

A defensible decline rests on three points: (1) the underlying code systems
already have authoritative external owners (AMA, CMS), so a THO ValueSet would
not be the source of truth for the codes — only for the composition rule;
(2) several near-identical sibling tickets in Block-Vote-5 were closed *Not
Persuasive* on the "little re-use value" basis, so declining here is
consistent with prior workgroup precedent; and (3) the URL change is a
breaking change for STU2 implementers and the WG would be incurring
implementer cost for an artifact whose practical reuse outside C4BB has not
been demonstrated. This option is weakest on governance grounds — it leaves a
non-IG-specific terminology composition inside an IG namespace — but it is
internally consistent with how the WG resolved comparable Block-Vote-5 items.

---

### Recommendation

**Recommended disposition:** **A — Accept As Requested**

Lynn Laakso's reopen comment is decisive: the WG has *already* signaled intent
to move this ValueSet to THO ("Working on putting in THO for future version").
Accepting is therefore the disposition that matches reality, removes the
ticket from the perpetual-deferral cycle that produced the 2022-11-28 reopen,
and unblocks the UTG registration. AMACPTCMSHCPCSModifiers is a composition
over CPT/HCPCS — both broadly used across US claims-oriented IGs (Da Vinci
PDex, US Core for billing-related profiles, future C4BB extensions) — so the
reuse case is materially stronger than for the simple single-purpose
ValueSets that the WG rejected as *Not Persuasive* in the same block. The
breaking-change concern that motivates Disposition B is real but manageable
with a one-publication-cycle deprecation alias plus a clear note in the C4BB
migration guide; that is cheaper than carrying parallel definitions
indefinitely. Disposition C is hard to justify given the WG's own stated
direction in the reopen comment. The workgroup should adopt A, file the
companion UTG ticket immediately, and treat the cluster of sibling
"anchor-in-THO" tickets (FHIR-35389/35394/35397/35400/35420/35423) as
candidates for the same treatment to avoid one-off migrations.
