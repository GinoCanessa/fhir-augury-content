## Summary

The ticket asks the CARIN IG for Blue Button® (C4BB) editors to relocate the
canonical defining URL for `ValueSet-C4BBPayerClaimPaymentStatusCode` from the
`hl7.org/fhir` namespace into the HL7 Terminology (THO,
`terminology.hl7.org`) namespace. The reporter (Joan Harper) further notes
that only ValueSets with a *required* binding to a `code` data type should
remain anchored under `hl7.fhir.org`, implying this ValueSet does not meet
that criterion. A companion UTG ticket would be required to register the
ValueSet in THO. The ticket was originally part of the C4BB **Block-Vote-6**
slate (FM call 2022-07-19, vote 2022-07-26) where it was dispositioned
*Considered for Future Use* with the note "May move this ValueSet to THO for
a future version based on harmonization work with other IGs." Per Lynn
Laakso's 2022-11-28 comments the prior resolution was reverted and the
ticket re-opened post STU2 publication, returning it to **Triaged** status
for fresh disposition. No directly cross-linked Jira tickets were surfaced;
the only cross-reference is the Block-Vote-6 announcement thread on Zulip,
which puts FHIR-35426 alongside ~25 sibling tickets making the same THO
re-anchoring request for other C4BB CodeSystems and ValueSets.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the canonical URL of `ValueSet-C4BBPayerClaimPaymentStatusCode` from
`http://hl7.org/fhir/us/carin-bb/ValueSet/C4BBPayerClaimPaymentStatusCode`
to a `http://terminology.hl7.org/ValueSet/...` canonical anchored in THO.
File a companion UTG ticket to register the ValueSet (and, if not already
present, its underlying CodeSystem `C4BBPayerBenefitPaymentStatus`) under
THO governance. Update the C4BB IG ValueSet definition to point at the new
THO canonical, fix all in-IG references (profile bindings on
`ExplanationOfBenefit` slices that bind to this ValueSet, examples,
narrative, and IG-level dependency declarations on `hl7.terminology`), and
add a documented alias from the old C4BB-namespaced canonical so existing
implementers can resolve the prior URL.

#### Justification

This is exactly the pattern HL7 has been promoting: terminology artifacts
that are not strictly bound to a `code` primitive — and are reusable across
IGs — should live in THO so they can be versioned, governed, and consumed
without dragging the host IG along. The reporter's framing matches the
HL7 cross-paradigm guidance, the FM workgroup explicitly previewed exactly
this disposition path in the Block-Vote-6 note ("May move this ValueSet to
THO for a future version based on harmonization work with other IGs"), and
the ticket was re-opened post-STU2 specifically so it could be revisited
for the next version. Doing it now positions C4BB consistently with the
other "Considered for Future Use" siblings in the same block vote that
share the same end state.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the move until the harmonization work referenced in the original
disposition note has produced a target ValueSet/CodeSystem in THO that
C4BB can adopt directly (rather than C4BB unilaterally publishing a
standalone THO artifact). Concretely: (1) keep the current
`hl7.org/fhir/us/carin-bb/ValueSet/C4BBPayerClaimPaymentStatusCode`
canonical for the next published version of C4BB; (2) coordinate with the
Vocab WG and the other IGs that carry similar payer/payment-status
ValueSets (e.g., DaVinci PDex, US Drug Formulary) to converge on one
shared THO artifact through UTG; (3) once that shared artifact is
published, retire the C4BB-local ValueSet, redirect the binding on
`ExplanationOfBenefit.adjudication[benefitpaymentstatus]` (and any other
slice using it) to the THO canonical, and document the migration in the
IG change log.

#### Justification

Several sibling tickets in the same Block-Vote-6 slate were dispositioned
*Not Persuasive* with the rationale that "the ValueSet is not complex,
fairly specific to this IG, and does not warrant the maintenance of moving
to THO." `C4BBPayerClaimPaymentStatusCode` is similarly small and
C4BB-specific today; publishing it standalone in THO simply to move the
URL trades one maintenance burden (C4BB editors) for another (THO/UTG
editors) without yielding the harmonization benefit that justifies a THO
home. Waiting for the cross-IG harmonization work delivers the *real*
value (one canonical that multiple IGs reuse), avoids creating a "C4BB-only
ValueSet that happens to live in THO" anti-pattern, and is consistent with
the editors' written intent in the original Block-Vote-6 note ("based on
harmonization work with other IGs").

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive* and keep the ValueSet's canonical anchored
under the C4BB IG namespace
(`http://hl7.org/fhir/us/carin-bb/ValueSet/C4BBPayerClaimPaymentStatusCode`).
Cite the same rationale used for the structurally identical sibling
tickets dispositioned *Not Persuasive* in Block-Vote-6: the ValueSet is
small, IG-specific, has no demonstrated cross-IG reuse, and the cost of
the move and ongoing dual maintenance outweighs the symbolic benefit of a
THO canonical. Update the ticket comment to point at the consistent
treatment of FHIR-35403, FHIR-35409, FHIR-35413, FHIR-35414, FHIR-35415,
FHIR-35432, FHIR-35434, FHIR-35435, FHIR-35436, FHIR-35439, FHIR-35446,
and FHIR-35447 as precedent.

#### Justification

The reporter's framing relies on a heuristic ("only ValueSets with required
binding to a `code` data type should be anchored in hl7.fhir.org") that
HL7 has not adopted as a normative rule; many IG-specific ValueSets that
do not meet that criterion legitimately live in their host IG. There is no
evidence in the ticket, comments, or Zulip thread of an external consumer
asking to reuse this ValueSet, no UTG ticket has actually been filed in
the four years since the request was opened, and the FM WG already voted
13-0-0 to defer. Declining now closes the loop cleanly and preserves
implementer stability — every existing C4BB consumer already resolves the
current canonical, and a move would invalidate stored references in
production payer systems.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern is legitimate and was endorsed by FM in
2022, which is why the ticket was re-opened post-STU2 — Disposition C
ignores that history and the explicit "may move… for a future version"
language the WG already agreed to. But Disposition A treats the move as a
mechanical URL change, when the value of being in THO comes from
*harmonization*, not from the namespace itself. Publishing a C4BB-only
ValueSet under THO would create exactly the kind of artifact THO/Vocab
governance has been trying to discourage: one with a single consumer and
no shared definition. Disposition B keeps the commitment the WG already
made (the ValueSet *will* migrate), ties the migration to the
harmonization work the original note pointed to, and gives implementers
predictable continuity for the next published version. It also lets FM
batch this resolution with the many sibling tickets in the same Block-Vote-6
slate that are likewise waiting on the external CodeSystem "all codes"
ValueSet (UP-316) to be published — yielding a consistent, defensible
group decision rather than 25 individually negotiated outcomes.
