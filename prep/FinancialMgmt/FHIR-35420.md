## Summary

The ticket asks the CARIN IG for Blue Button (CARIN BB) work group to
re-anchor the canonical URL of the
`C4BBEOBInstitutionalProcedureCodes` ValueSet in
`terminology.hl7.org` (THO) instead of `hl7.org/fhir/us/carin-bb`, and
to create a corresponding UTG ticket to register the value set in THO.
The reporter argues that, per the broader convention now applied across
HL7 specifications, only value sets that are the target of a `required`
binding to a `code`-typed element should be anchored in the IG's own
canonical space; all other value sets — including this one, which binds
a `CodeableConcept` (`ExplanationOfBenefit.procedure.procedureCodeableConcept`)
— belong in THO so they can be re-used across IGs without duplication.

This ticket is one of a Block-Vote-5 cluster for CARIN BB STU2 (the
related Zulip thread lists ~15 sibling FHIR-3540x/FHIR-3541x/FHIR-3542x
tickets all making the same THO-anchoring request against different
CARIN BB value sets). At STU2 publication those siblings were
collectively dispositioned as "Considered For Future Use", and this
ticket was re-opened post-STU2 publication with the comment
*"Working on putting into THO for a future version"*. The augury index
surfaced no other Jira cross-references and no related GitHub items.

## Proposed Dispositions

Note: the augury index surfaced no GitHub cross-references and no cached
repo-analysis briefing. The dispositions below therefore name the
authoritative spec sources by their canonical IG paths
(`HL7/carin-bb` for the IG itself and the HL7 UTG/THO repository for the
THO migration) without binding to specific clone paths. If the work
group adopts Disposition A or B, a follow-up `repo-analysis` run
against `HL7/carin-bb` (and `HL7/UTG`/THO) should be performed before
implementation so the change can be anchored to concrete authoring
files.

### Disposition A: Accept As Requested

#### Proposal

In a future CARIN BB version (STU 2.1 or STU 3):

1. File a UTG ticket against the THO project to migrate
   `C4BBEOBInstitutionalProcedureCodes` from
   `http://hl7.org/fhir/us/carin-bb/ValueSet/C4BBEOBInstitutionalProcedureCodes`
   to a THO-anchored canonical such as
   `http://terminology.hl7.org/ValueSet/C4BBEOBInstitutionalProcedureCodes`.
2. After the THO publication carrying the new canonical is released,
   update the CARIN BB IG (in the `HL7/carin-bb` IG repository — the
   `input/resources` ValueSet definition and any profile binding sites,
   notably the Institutional EOB profile's binding on
   `ExplanationOfBenefit.procedure.procedureCodeableConcept`) to point
   the binding at the THO canonical and remove the old IG-anchored
   definition.
3. Document the canonical change in the IG's change-log/migration notes
   so implementers know about the URL change.
4. Coordinate the same migration for the still-open sibling tickets
   (FHIR-35400, FHIR-35408, FHIR-35423, FHIR-35389, etc.) as a single
   THO batch to avoid repeated implementer churn.

#### Justification

This is the path the reporter requested and the path the work group has
already signalled in the comment trail ("Working on putting into THO
for a future version"). It aligns with the HL7-wide convention that
re-usable value sets — particularly those bound to `CodeableConcept`
slots — belong in THO so they can be referenced by multiple IGs without
duplication, and it removes a CARIN-specific definition that
HRex/Da Vinci/other claims-adjacent IGs could plausibly re-use. The
2022 Financial Management vote on the parent block (7-0-2) showed no
substantive opposition to the THO migration; the only reason it was
deferred was schedule, not technical objection.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than migrate the canonical (which is a breaking change for any
implementer that has hard-coded the current IG-anchored URL), publish
the value set in THO under its new canonical AND retain a *deprecated
alias* in CARIN BB that resolves to (and is kept logically equivalent
with) the THO definition for the remainder of STU 2.x. Specifically:

1. File the UTG/THO ticket as in Disposition A and publish
   `http://terminology.hl7.org/ValueSet/C4BBEOBInstitutionalProcedureCodes`.
2. In the CARIN BB IG repository, mark the existing
   `http://hl7.org/fhir/us/carin-bb/ValueSet/C4BBEOBInstitutionalProcedureCodes`
   ValueSet as deprecated, set its `compose` to a single `import` of
   the THO canonical (so the two definitions are guaranteed to match),
   and add narrative pointing implementers at the THO URL.
3. Switch the Institutional EOB profile binding to the THO canonical in
   the same release.
4. Schedule removal of the deprecated IG-anchored ValueSet for a clearly
   announced future major release.

#### Justification

This achieves the policy goal (the value set lives in THO and is re-usable
by other IGs) while protecting STU2 implementers — payers and apps that
have bound to the IG-anchored URL keep working until they migrate. It
also avoids the worst failure mode of a pure rename, which is silent
divergence between two value sets sharing a name but living at different
URLs. The cost is one additional release cycle of dual-publication; for
a published, regulator-referenced IG (CMS Interoperability and Patient
Access), this transitional gentleness is usually worth it.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive — Considered for Future Use** (the same
disposition the work group originally voted in 2022) and close the
ticket without migrating the canonical. Document that
`C4BBEOBInstitutionalProcedureCodes` is intentionally retained at its
current `hl7.org/fhir/us/carin-bb` canonical because the value set is
a CARIN-BB-specific composition with limited demonstrated re-use
outside the IG, and that any future cross-IG re-use need can be met by
having the consuming IG `import` the existing canonical directly.

#### Justification

Several of the sibling tickets in the same Block-Vote-5 cluster
(FHIR-35412/35416/35418/35419/35421/35422) were dispositioned as
"Not Persuasive — Simple ValueSet with little re-use value" on exactly
this rationale. Migrating canonical URLs for a published, regulator-
referenced IG is a real cost to implementers and to CMS-facing
documentation; if the work group has not seen concrete re-use demand
for this value set in the three years since the ticket was filed, the
cost-benefit may not justify the migration. Declining also keeps the
IG self-contained (one fewer external THO dependency to track at
publication time).

---

### Recommendation

**Recommended disposition:** B

The original work-group sentiment (and the 2022-11-28 re-open comment)
is clearly that the THO migration *should* happen — Disposition C goes
against the work group's own stated direction and would leave the
sibling tickets in an inconsistent state. Disposition A is the right
end-state but, for a regulator-referenced IG that has been in
production since STU2, a hard canonical swap risks silently breaking
implementers who have hard-coded the IG URL. Disposition B reaches the
same end-state as A while giving implementers an explicit, narrated
deprecation window and a guaranteed-equivalent alias, which is the
standard HL7 pattern for canonical migrations of published artifacts.
The work group should also use this ticket as the trigger to batch the
still-open siblings (FHIR-35400, FHIR-35408, FHIR-35423, FHIR-35389)
into a single THO submission so implementers face one migration, not
five.
