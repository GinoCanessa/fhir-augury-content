## Summary

This ticket asks the CARIN IG for Blue Button® (US CARIN BB) work group to
re-anchor the canonical URL of `ValueSet-NCPDPDispensedAsWrittenOrProductSelectionCode`
under `terminology.hl7.org` (THO) instead of `hl7.org/fhir`, with a follow-up
UTG ticket to formally register the value set in THO. The reporter notes the
broader rule the ticket is enforcing: only value sets that are bound as
`required` to a `code` data type belong under `hl7.org/fhir`; everything
else should live in THO. The value set in question is one of ~26 CARIN BB
value sets that were swept up in the same review (Block-Vote-6) — see the
Zulip block-vote announcement under "Related Zulip Discussions" for the
full set and their per-item resolutions. The ticket was originally
dispositioned "Considered for Future Use" in 2022 (block vote 13-0-0,
contingent on the UTG "all codes" value set work tracked as
[UP-316](https://jira.hl7.org/browse/UP-316)) and was then **re-opened
post STU2 publication**, which is its current state.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor `ValueSet-NCPDPDispensedAsWrittenOrProductSelectionCode` under
`http://terminology.hl7.org/ValueSet/...` (THO), and remove the
IG-local copy from CARIN BB. Concretely:

1. File a UTG ticket against THO requesting publication of an "all
   codes from NCPDP DAW Code System" value set (or, if UP-316 has
   landed, declare the value set using the "all codes" mechanism so
   it does not need hand-curation in THO).
2. Once the THO value set is live, change every binding in CARIN BB
   that currently references the IG-local
   `ValueSet-NCPDPDispensedAsWrittenOrProductSelectionCode` (and its
   `http://hl7.org/fhir/us/carin-bb/...` canonical) to point to the
   new THO canonical.
3. Delete the local ValueSet resource from the IG and add a redirect /
   migration note to release notes.

#### Justification

This is what the reporter asked for, what the work group's broader
"only `required`-bound `code` value sets stay under `hl7.org/fhir`"
rule prescribes, and what the comment trail on the ticket already
acknowledges as the eventual end state ("When [UP-316] is complete,
the ValueSet in this IG can be removed and the bindings point to the
THO ValueSet"). It also keeps FHIR-35443 consistent with its sibling
NCPDP-anchored tickets in Block-Vote-6 (FHIR-35441/35442/35444/35445),
all of which received the same proposed resolution. The hard
prerequisite is that UP-316 (or an equivalent "all codes" value set
publication path) is actually available — without that, hand-curating
the NCPDP DAW codes in THO is not viable.

---

### Disposition B: Alternative Approach

#### Proposal

Re-affirm the original 2022 disposition: **Considered for Future Use**,
contingent on UP-316. Specifically:

1. Keep the value set definition where it currently lives (under
   `hl7.org/fhir/us/carin-bb`) in the next CARIN BB publication.
2. Add an explicit forward-looking note in the value set's narrative
   stating that the canonical will move to THO once UP-316 ships and
   the "all codes" mechanism can host it without manual code
   enumeration.
3. Tie this ticket (and its NCPDP siblings 35441/35442/35444/35445) to
   UP-316 via Jira "is blocked by" links so the migration is not lost
   when UP-316 lands.

#### Justification

The work group has already voted 13-0-0 on this exact resolution. The
re-open in November 2022 was procedural (post-STU2 hygiene), not driven
by new substantive feedback in Zulip or the ticket itself. The blocking
external dependency (UP-316) has not changed: until THO can publish an
"all codes from NCPDP DAW" value set without HL7 manually mirroring
NCPDP's licensed code system, moving the canonical only relocates the
problem. Disposition B is the lowest-risk choice that preserves the
intent of Disposition A and keeps consumers' bindings stable in the
interim.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-35443 **Not Persuasive** with the rationale that
`ValueSet-NCPDPDispensedAsWrittenOrProductSelectionCode` is sufficiently
specific to CARIN BB pharmacy claim usage that it does not warrant the
ongoing THO maintenance burden, and close the ticket. (This is the
exact rationale the work group used in Block-Vote-6 for several sibling
value sets — e.g., FHIR-35432, 35434, 35435, 35436, 35439, 35446,
35447 — that were declined "Not Persuasive — ValueSet is not complex,
fairly specific to this IG, and does not warrant the maintenance of
moving to THO".)

#### Justification

The DAW value set in CARIN BB is in practice a thin wrapper over an
external NCPDP code system; the IG already cannot publish the codes
themselves due to NCPDP licensing, so the IG-local value set's only
real content is the binding statement and narrative, which is cheap to
maintain in-IG. If the work group's harmonization roadmap with other
pharmacy/IG efforts is not actively converging on a shared NCPDP DAW
value set, paying the THO migration cost for this one value set is
hard to justify. Declining keeps the canonical stable and avoids
churning every implementer's bindings.

---

### Recommendation

**Recommended disposition:** B

The work group has already deliberated this exact question and voted on
exactly the same answer (13-0-0). The November 2022 re-open looks like a
post-publication housekeeping action rather than a substantive change
of position, and there is no new Zulip discussion, GitHub work, or
fresh comment trail challenging the original outcome. Disposition A is
the right end-state, but it is gated by UP-316; committing to A now
without UP-316 in hand risks either (i) hand-curating NCPDP codes in
THO under license, or (ii) shipping a relocated-but-still-incomplete
value set. Disposition C is defensible for many of the IG-specific
sibling value sets in Block-Vote-6, but it is a poor fit here: the DAW
value set is explicitly described in the comment trail as "all codes
in a complete external Code system", which is exactly the case UP-316
is designed to handle and exactly the case THO is intended to host.

Disposition B re-affirms the 13-0-0 outcome, makes the UP-316
dependency explicit on the ticket and on the artifact, and lines this
ticket up with FHIR-35441/35442/35444/35445 so the five NCPDP value
sets move together when UP-316 ships.
