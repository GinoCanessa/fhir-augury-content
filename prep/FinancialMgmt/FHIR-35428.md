## Summary

This ticket asks that the canonical defining URL for the
`C4BBPractitionerIdentifierType` ValueSet (currently in the CARIN IG for Blue
Button under `hl7.org/fhir/us/carin-bb`) be re-anchored in
terminology.hl7.org (THO) rather than under `hl7.org/fhir`. The reporter notes
that a UTG (Unified Terminology Governance) ticket should be opened to register
the ValueSet in THO. The ticket is one of a large family of analogous
"defining URL should be anchored in THO" requests filed by Joan Harper against
CARIN BB code systems and value sets (e.g., FHIR-35426, FHIR-35427,
FHIR-35431–35447) that were processed together as Block-Vote-6 on 2022-07-26.
At that vote (Corey Spears / Kathleen Conner, 13-0-0) FHIR-35428 was given the
disposition **Considered for Future Use**, with the rationale that the change
would be made once an external CodeSystem "all codes" ValueSet is published
via [UP-316](https://jira.hl7.org/browse/UP-316); when that lands, the
in-IG ValueSet can be removed and bindings can point at the THO ValueSet
(after the NPI code is dropped from this ValueSet through a separate ticket).
Lynn Laakso re-opened the ticket on 2022-11-28 ("re-open post STU2
publication"), reverting the prior resolution so that Financial Management can
re-evaluate now that STU2 is out.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the `C4BBPractitionerIdentifierType` ValueSet out of the CARIN BB IG and
into THO. Concretely:

- File a UTG/THO change request to register the ValueSet at
  `http://terminology.hl7.org/ValueSet/C4BBPractitionerIdentifierType` (or
  another appropriately scoped THO canonical), preserving the current
  composition (Identifier Type Codes plus the NPI code from
  `http://hl7.org/fhir/sid/us-npi`, minus any codes removed by the companion
  "drop NPI" ticket the comment thread references).
- In the next CARIN BB release, retire the in-IG ValueSet
  `http://hl7.org/fhir/us/carin-bb/ValueSet/C4BBPractitionerIdentifierType`
  (or mark it deprecated with a `replacedBy` pointer) and re-point every
  binding currently using the IG-local URL — most notably the
  `Practitioner.identifier.type` binding on `C4BBPractitioner` — to the new
  THO canonical.
- Coordinate timing with [UP-316](https://jira.hl7.org/browse/UP-316) so the
  THO "all codes" ValueSet pattern this ValueSet depends on is in place
  before the binding swap ships.

#### Justification

This is what the ticket literally asks for and is consistent with the FHIR
terminology governance principle that reusable, cross-IG terminology should
live in THO rather than in IG-local namespaces. The 2022-07-26 Financial
Management vote (13-0-0) already endorsed this direction — it just deferred
execution pending UP-316 and the NPI cleanup. Lynn Laakso's 2022-11-28
re-open ("re-open post STU2 publication") signals that the deferral window has
closed and the work group is ready to act. Doing the move centrally in THO
also avoids each downstream IG having to redefine or shadow the same
identifier-type concepts.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than relocating the ValueSet itself, change the CARIN BB binding to
point directly at an existing THO ValueSet for identifier types and retire
`C4BBPractitionerIdentifierType` as a named artifact. Concretely, evaluate
whether `Practitioner.identifier.type` on `C4BBPractitioner` can bind to
`http://terminology.hl7.org/ValueSet/v2-0203` (Identifier Type Codes) extended
locally only with NPI — and, if so, replace the in-IG ValueSet with either
(a) a thin profile-level binding to the THO ValueSet plus a documented
expectation that NPI is conveyed via `Practitioner.identifier` with the
`http://hl7.org/fhir/sid/us-npi` system (no `type.coding` needed), or (b) a
small THO ValueSet that unions v2-0203 with the NPI sentinel code, owned by
THO from day one. If neither is acceptable, fall back to Disposition A.

#### Justification

The disposition comment notes this ValueSet is essentially "all codes in a
complete external Code system" plus the NPI carve-out, which is precisely the
shape v2-0203 already has in THO. Eliminating the IG-local ValueSet entirely
(rather than just relocating it) reduces long-term maintenance for Financial
Management, removes the dependency on UP-316's "all codes" pattern, and keeps
identifier-type semantics consistent with the broader HL7 ecosystem. It also
avoids creating a second THO artifact whose only differentiator is the NPI
inclusion.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** (or restore the prior **Considered for Future
Use** disposition with no commitment). Keep the ValueSet defined in the CARIN
BB IG at its current `hl7.org/fhir/us/carin-bb` canonical, and document that
relocation to THO is not planned.

#### Justification

A defensible "decline" rests on two arguments. First, the
`C4BBPractitionerIdentifierType` ValueSet is, in practice, used only by CARIN
BB profiles; the sibling tickets in the same Block-Vote-6 that were judged
**Not Persuasive** (FHIR-35432, 35434, 35439, 35447, etc.) used exactly that
rationale: "ValueSet is not complex, fairly specific to this IG, and does not
warrant the maintenance of moving to THO." Second, moving terminology to THO
is a breaking canonical-URL change for implementers who already validate
against the IG-local URL; if no other consuming IG has materialised since
2022, the migration cost may exceed the benefit. This option also avoids
blocking on UP-316.

---

### Recommendation

**Recommended disposition:** A

The Financial Management work group has already voted unanimously (13-0-0) to
move this ValueSet to THO once the supporting infrastructure is in place, and
the ticket was explicitly re-opened after STU2 publication to revisit
execution rather than to revisit the underlying decision. Disposition B is
attractive in principle but introduces a new design discussion (whether
v2-0203 truly suffices for the CARIN BB identifier-type binding, and how to
carry NPI) that the work group has not had and that risks pushing the change
out another ballot cycle. Disposition C contradicts the work group's prior
vote and would leave a piece of broadly-applicable identifier terminology in
an IG-local namespace, which is exactly the anti-pattern the reporter (and
the wider terminology governance program) are trying to retire. Accepting as
requested — execute the THO/UTG move now that STU2 is out, gated on UP-316
and the companion NPI-removal ticket — is the cleanest path and matches the
direction the work group already chose.
