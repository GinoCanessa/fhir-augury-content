## Summary

FHIR-45815 asks the Da Vinci Patient Cost Transparency (PCT) IG editors to
move ("anchor") one of the IG-defined value sets into HL7 Terminology
(THO, `terminology.hl7.org`) on the basis that the value set is likely
useful beyond PCT. The ticket is one of a family of effectively
identical change requests filed by Ron G. Parker against PCT
(FHIR-45812, FHIR-45816, FHIR-45817, FHIR-45818 — all "Consider
anchoring the valuesets in THO. The Value Set could be useful beyond
this IG. N") and sits alongside a broader cluster of older tickets in
the same vein (e.g., FHIR-35529 through FHIR-35550, requesting that
specific PCT CodeSystems and ValueSets be moved out of
`hl7.org/fhir/...` and into THO). The narrow text of this individual
ticket — "Please consider anchoring the valueset in THO" — does **not**
name a specific ValueSet, which is the central practical issue for
disposition.

The ticket was originally batched into the **Block-2025-Jul** PCT block
vote and resolved on 2025-07-08 as **Considered for Future Use** (vote
9-0-2; "Resolve across IGs continuing to add to THO… planned in the
coming months in larger FM/THO effort that has already gotten underway
in earnest"). On 2025-10-08, Lynn Laakso reverted that resolution and
re-opened the ticket post STU2 publication, returning it to **Triaged**
status; that is its current state.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-45815 as **Persuasive** but with the explicit precondition
that the **target ValueSet must be identified by the submitter (or by
FM editors) before any move occurs**. Because the ticket text does not
name a value set, FM should:

1. Ask Ron G. Parker (or his successor on PCT) to identify which PCT
   ValueSet the "3" in this ticket refers to (the sibling structure
   suggests it is one of the unnamed "useful beyond this IG" series
   alongside `PCTAdjudication` (FHIR-45818) and
   `USClaimMedicalProductOrServiceCodes` (FHIR-45812)).
2. Once identified, migrate that ValueSet's canonical from
   `http://hl7.org/fhir/us/davinci-pct/ValueSet/<Name>` to
   `http://terminology.hl7.org/ValueSet/<Name>` in THO, retire the
   PCT-local copy with a redirection note, and update every PCT
   profile/element binding to reference the new THO canonical.
3. Coordinate the move through the **FM/THO joint effort already
   underway** that Lynn Laakso referenced in the 2025-07-08 vote
   notes, so the move is sequenced with the related tickets
   (FHIR-45812, 45816, 45817, 45818, and the FHIR-35529–35550 family).

#### Justification

The request is consistent with HL7's standing terminology-governance
direction (value sets that are reusable across IGs belong in THO, not
under an IG-specific canonical), and FM has already opened a workstream
with Vocabulary to do exactly this. Approving as Persuasive aligns
FHIR-45815 with the disposition style FM already gave to the older
sibling tickets that *did* name a value set (e.g., FHIR-35549 / 35545 /
35540 — "Persuasive, Move to THO"). Doing the migration as part of the
broader FM/THO batch keeps canonical-URL churn for implementers
contained to a single coordinated cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve FHIR-45815 as **Persuasive with Modification**: do not move
this individual value set in isolation, and instead **fold FHIR-45815
into the FM/THO joint THO-migration tracking ticket** (or create one
if it does not yet exist) covering the entire PCT THO-anchoring family
(FHIR-45812, 45815, 45816, 45817, 45818, plus the FHIR-35529–35550
cluster). The modification text should:

- Acknowledge that the underlying intent (move PCT-defined,
  reusable value sets to THO) is accepted.
- Note that the work will be executed *as a single coordinated PR /
  set of PRs* across PCT and THO, rather than per-ticket, to avoid
  partial migrations and binding/canonical drift.
- Require that the ValueSet target for FHIR-45815 be named in the
  tracking ticket before the migration PR is opened.
- Specify that PCT will retain a deprecated stub canonical pointing to
  the THO canonical for backward compatibility for one publication
  cycle.

#### Justification

The ticket as written is too underspecified to action on its own — it
does not name a ValueSet, and acting on it in isolation would risk
moving the *wrong* value set or duplicating work already in flight in
the FM/THO joint effort. Bundling resolves the ambiguity, gives
implementers a single migration window to react to, and matches the
operational reality that Lynn Laakso flagged in the 2025-07-08 vote
notes ("…planned in the coming months in larger FM/THO effort that has
already gotten underway"). It also avoids re-living the
"Considered for Future Use → reverted" loop, which is the failure mode
that put this ticket back in front of FM.

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-45815 as **Not Persuasive — Insufficient Specification**.
The rationale: the ticket text is "Please consider anchoring the
valueset in THO" with no value set named, no canonical URL, no element
binding cited, and no concrete reuse case described. There is no
artifact in the PCT IG that can be unambiguously identified from the
ticket alone, and the duplicate-form sibling tickets
(FHIR-45816 / 45817) suffer the same defect. Close the ticket and
direct the requester to file a focused, value-set-specific request
against the FM/THO joint workstream, modelled on the older Joan Harper
tickets (e.g., FHIR-35549) which *do* name the artifact.

#### Justification

FM already attempted to dispose of this ticket as "Considered for
Future Use" in July 2025 and that resolution was reverted post-STU2,
which strongly suggests the ticket's lack of specificity is itself the
problem. Declining on insufficient-specification grounds is defensible
because the broader policy ("move reusable PCT value sets to THO")
is **already being acted on** through the FM/THO joint effort, so no
governance signal is lost by closing this particular ticket. It also
prevents a vague ticket from being silently interpreted as approval
to move an unspecified value set.

---

### Recommendation

**Recommended disposition:** B

Disposition B (Persuasive with Modification — fold into the FM/THO
joint THO-migration tracking ticket) is the right balance for
FM. The community signal across the PCT THO-anchoring family is
clear: FM is already willing to move reusable PCT value sets to THO,
and the FM/THO joint effort that Lynn Laakso cited is the operational
home for that work. Disposition A is essentially correct but
under-protects against the very ambiguity that caused this ticket to
be reverted in October — accepting a ticket that does not name its
target ValueSet invites another round of churn. Disposition C is
defensible on process grounds but would discard a legitimate request
that FM agrees with in principle and would force the same
conversation to recur on the next ballot. Disposition B captures the
intent (move it to THO), forces the missing detail (which ValueSet)
to be pinned down before the migration PR is cut, and aligns the
work with the cross-IG batch FM has already committed to — minimising
canonical-URL churn for PCT implementers and keeping FHIR-45815
synchronised with FHIR-45812 / 45816 / 45817 / 45818 and the
FHIR-35529–35550 cluster.
