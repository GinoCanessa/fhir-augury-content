## Summary

The reporter notes that the Claim resource recently received a new
`supportingInfo.subCategory` element (introduced via
[FHIR-51523](https://jira.hl7.org/browse/FHIR-51523)) but the analogous
element is missing on `ClaimResponse.supportingInfo` — itself a recent
addition driven by [FHIR-51522](https://jira.hl7.org/browse/FHIR-51522).
The ticket asks that `subCategory` be added to
`ClaimResponse.supportingInfo` so that the Claim and ClaimResponse
`supportingInfo` backbones remain structurally aligned. The Financial
Management workgroup has triaged the change as **Persuasive with
Modification** and broadened the scope in the resolution to also cover
ExplanationOfBenefit, so that all three claiming resources keep
`supportingInfo` in lockstep. The companion ticket
[FHIR-51518](https://jira.hl7.org/browse/FHIR-51518) (open-typing
`supportingInfo.value[x]`) already established that ClaimResponse and
EoB are tracked as the affected siblings whenever `Claim.supportingInfo`
changes shape, so the precedent for cross-resource alignment is in
place.

## Related Jira Tickets

### [FHIR-51523: Introduce subcategory for supportingInfo](https://jira.hl7.org/browse/FHIR-51523)

This is the ticket that originally added `subCategory 0..*
CodeableConcept` to `Claim.supportingInfo` (and EoB) to give finer-grained
classification analogous to the `Claim.type` / `Claim.subtype` pattern.
FHIR-54102 is the direct follow-on: ClaimResponse was not in the original
"affected resources" list, leaving the structures out of alignment, which
is exactly what FHIR-54102 corrects.

Reporter: AnjaSchwab. Assignee: Unassigned. Status: Applied (FM-R5+-Applied).

### [FHIR-51522: Add supportingInfo to ClaimResponse](https://jira.hl7.org/browse/FHIR-51522)

This ticket introduced `supportingInfo` as a BackboneElement on
ClaimResponse, modeled on `Claim.supportingInfo`. Its resolution
explicitly anticipated FHIR-54102 by stating that the new
ClaimResponse.supportingInfo should "mirror that already in Claim and
EOB with the addition of any changes to that structure approved for R6
such as the addition of the subCategory element," so adding subCategory
on ClaimResponse is the completion of work this ticket already scoped.

Reporter: AnjaSchwab. Assignee: Unassigned. Status: Applied (FM-R5+-Applied).

### [FHIR-51518: Change restriction on supportingInfo.value[x] to Open Type Element *](https://jira.hl7.org/browse/FHIR-51518)

This ticket relaxed the datatype restriction on
`Claim.supportingInfo.value[x]` (and, per its resolution, the equivalent
on ClaimResponse and ExplanationOfBenefit) to an open type. It is
relevant here as precedent: it is the most recent example of a
`supportingInfo` change that the workgroup explicitly extended across
Claim, ClaimResponse and EoB, and its resolution notes that "profiles
will then be free to determine the datatype choices for each category
and subCategory slice" — i.e., it already references `subCategory` as a
slicing axis, reinforcing why ClaimResponse must carry it too.

Reporter: AnjaSchwab. Assignee: Unassigned. Status: Applied (FM-R5+-Applied).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new element `ClaimResponse.supportingInfo.subCategory` typed as
`0..* CodeableConcept` with the same definition used on
`Claim.supportingInfo.subCategory` ("Finer-grained categorization of the
information"), bound to the same value set / binding strength as the
Claim equivalent (no binding initially; example/preferred binding
inherited from FHIR-51523's outcome). No other ClaimResponse changes are
required. Mark this as an FM-R6 breaking-aware change and ship it under
the existing `FM-R5+_ToBeApplied` workflow alongside the EoB twin.

#### Justification

This is the literal request and matches the resolution already recorded
on the ticket. Aligning ClaimResponse.supportingInfo with
Claim.supportingInfo is the long-standing convention for the claiming
trio (Claim / ClaimResponse / EoB), reinforced by FHIR-51518 and
FHIR-51522 — both of which were resolved by extending the same shape
across all three resources. Failing to add `subCategory` here would
re-introduce the structural drift that those earlier tickets explicitly
removed and would force implementers to extend ClaimResponse with a
custom `subCategory` extension that exactly duplicates the Claim
element.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding a first-class `subCategory` element to
`ClaimResponse.supportingInfo`, define a standard FHIR extension
(`http://hl7.org/fhir/StructureDefinition/claimresponse-supportingInfo-subCategory`,
type `CodeableConcept`, cardinality `0..*`) and ship it in the
extensions pack. ClaimResponse.supportingInfo would remain unchanged at
the resource level; profiles needing the finer classification would slot
the extension in.

#### Justification

This avoids an R6 breaking change to ClaimResponse (the ticket carries
the `FM-R6-BreakingChange` label) and lets jurisdictions that want
subCategory adopt it via the extension without forcing the change on all
implementers. Trade-off: it permanently breaks the symmetry with
`Claim.supportingInfo` and `EoB.supportingInfo` — both of which now
carry `subCategory` natively per FHIR-51523 — and pushes complexity onto
profile authors and consumers, who must check both the core element and
the extension. Given that the workgroup has already accepted breaking
changes on the sibling resources, the symmetry argument is strong;
Disposition B should only be preferred if the workgroup decides the R6
breaking-change cost on ClaimResponse outweighs alignment.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54102 as "Not Persuasive" and leave
`ClaimResponse.supportingInfo` without a `subCategory` element. Any
finer-grained classification on the response side would need to be
carried in `category` (with a richer value set) or via implementer
extensions.

#### Justification

This would normally be defensible only if `subCategory` had not already
been added to `Claim.supportingInfo` and `EoB.supportingInfo`. Since it
has, declining here would cement an avoidable inconsistency across the
three claiming resources and contradict the resolution already recorded
on FHIR-51522, which explicitly anticipated `subCategory` arriving on
ClaimResponse. The only remaining defensible rationale would be a
complete reversal of FHIR-51523 — i.e., removing `subCategory` from
Claim and EoB as well — which is out of scope for this ticket.

---

### Recommendation

**Recommended disposition:** A

The ticket is a routine alignment change that closes a structural
asymmetry the workgroup has already committed to closing: FHIR-51522's
resolution explicitly named `subCategory` as the kind of R6 change
ClaimResponse should mirror, and FHIR-51518's resolution already names
`subCategory` as a slicing axis on the shared `supportingInfo` shape.
Accepting the ticket as written (and as already amended in the workgroup
resolution to also touch ExplanationOfBenefit) keeps Claim,
ClaimResponse, and EoB on a single `supportingInfo` model, avoids
proliferating an extension that would exactly duplicate the Claim
element, and matches the precedent set by the two prior `supportingInfo`
tickets from the same reporter and workgroup. The R6 breaking-change
label is correct and should be retained, but it is the same kind of
breaking change that FHIR-51523 already imposed on Claim/EoB and is the
expected cost of keeping the trio aligned.
