## Summary

This PCT ticket asserts that the canonical (defining) URL for the
`PCTGFEItemRevenueVS` ValueSet should be anchored in
`terminology.hl7.org` (THO) rather than `hl7.org/fhir/us/davinci-pct`,
and that a UTG ticket should be opened to register the value set in
THO. The reporter further notes that only value sets that have a
required binding to a `code` data type should remain anchored under
`hl7.fhir.org`; everything else (including this revenue-code value
set, which binds to non-`code` data) belongs in THO so it can be
shared across IGs. The ticket is one of a tightly related family of
PCT "anchor in THO" tickets ([FHIR-35529](https://jira.hl7.org/browse/FHIR-35529),
[FHIR-35532](https://jira.hl7.org/browse/FHIR-35532),
[FHIR-35533](https://jira.hl7.org/browse/FHIR-35533),
[FHIR-35535](https://jira.hl7.org/browse/FHIR-35535),
[FHIR-35536](https://jira.hl7.org/browse/FHIR-35536), and the sibling
CodeSystem tickets FHIR-35537, 35540, 35545, 35548-35550) that the
work group has been processing as a block. A 2022-06 comment from
Corey Spears connects this work to [UP-316](https://jira.hl7.org/browse/UP-316)
("Create 'All Codes' ValueSets for all External CodeSystems"), which
would supply the THO-side all-codes value set the PCT IG could then
reference. The ticket has twice been resolved as "Considered for
Future Use" and twice re-opened — most recently on 2025-10-08 after
STU2 publication — because the work group still intends to migrate
this terminology to THO as part of the broader FM/THO effort, but has
not yet executed the move on this particular value set.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive. Migrate the `PCTGFEItemRevenueVS` value set from the PCT
IG to THO:

1. File a UTG ticket against the THO project requesting that
   `PCTGFEItemRevenueVS` (or an equivalently-named THO value set, e.g.
   one expressing the NUBC UB-04 revenue-code subset the PCT GFE
   uses) be defined under `http://terminology.hl7.org/ValueSet/...`
   with the same intensional definition currently in PCT.
2. In the next PCT publication, replace the current canonical URL
   (`http://hl7.org/fhir/us/davinci-pct/ValueSet/PCTGFEItemRevenueVS`)
   with the new THO canonical wherever the value set is referenced
   (StructureDefinitions on GFE profiles, narrative pages, and the IG
   ImplementationGuide resource). Per the 2023-03-30 commitment in
   this ticket, also drop the example-only `PCTGFEItemRevenueCS`
   from the value set binding once an "all codes" THO value set for
   the external revenue-code system (per [UP-316](https://jira.hl7.org/browse/UP-316))
   is available.
3. Add a `ConceptMap`/redirect or release note in PCT explaining
   the canonical change for downstream implementers, since the
   current canonical has shipped in STU1 and STU2.

#### Justification

This is what the reporter requested in 2022, what the work group
itself committed to on the 2023-03-30 and 2025-10-08 reverts, and
what the broader FM/THO migration effort is already chartered to
do. Anchoring shareable revenue-code terminology in THO is consistent
with the Sept-2022 THO guidance referenced in the comment trail and
with the Vocabulary work group's UP-316 program. Doing the migration
under a UTG ticket (rather than as a PCT-internal rename) ensures the
value set becomes reusable by sibling IGs (DaVinci PDex, CARIN BB,
etc.) that already need the same revenue codes.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification. Address this ticket as part of a single,
explicitly-scoped FM/THO migration package rather than as an
independent change:

- Group FHIR-35534 with its Block-2025-Jul siblings that bind to
  non-`code` data types (FHIR-35529, FHIR-35532, FHIR-35533,
  FHIR-35535, FHIR-35536, FHIR-45812, FHIR-45815-45818) into one
  cross-IG migration ticket owned jointly by FM and Vocab.
- Coordinate with UP-316 so the THO-side "all codes" value sets for
  external code systems (revenue codes, NDC, CPT/HCPCS, type-of-bill,
  etc.) are stood up in a single UTG release.
- Defer the PCT canonical URL flip until the THO targets exist and
  can be referenced atomically, then make the flip in a single PCT
  point release with a documented breaking-change note.
- Keep the current PCT-anchored canonical alive (deprecated) for at
  least one publication cycle to give implementers time to update.

#### Justification

The 2025-10-08 revert comment from Lynn Laakso explicitly says this
value set "will require a greater level of review across multiple
IGs, and is planned in the coming months in larger FM/THO effort that
has already gotten underway." Doing FHIR-35534 in isolation risks
introducing a one-off canonical that has to be re-shaped again when
the cross-IG migration lands; bundling avoids two breaking changes in
short succession and keeps PCT, CARIN BB, and other consumers in
lock-step. It also matches Corey Spears's 2022-06-21 dependency on
UP-316: a clean migration needs the THO target value set to exist
first.

---

### Disposition C: Decline

#### Proposal

Considered for Future Use (resolve elsewhere). Decline to make any
PCT IG change for FHIR-35534 in the current ballot cycle and direct
the migration entirely to the FM/THO program of work, leaving the
existing `http://hl7.org/fhir/us/davinci-pct/ValueSet/PCTGFEItemRevenueVS`
canonical in place until that program publishes the THO replacement.

#### Justification

Defensible because the PCT IG has already shipped this canonical in
STU1 and STU2 and external implementers (payers and providers
generating GFEs) are bound to it; flipping the canonical
unilaterally without the THO target ready is a breaking change with
no immediate implementer benefit. The work group already chose this
disposition in two prior block votes (2022-11-29 9-0-1 and 2025-07-08
9-0-2). The risk is that this is the third "Considered for Future
Use" cycle on the same ticket — Lynn Laakso re-opened it on
2025-10-08 specifically because the work group does not want it to
remain dormant — so picking C again without a concrete dated plan
would invite another revert.

---

### Recommendation

**Recommended disposition:** B

The literal request in Disposition A is correct on the merits and
matches the work group's stated intent across three years of
comments, but executing it in isolation has been tried twice and
twice deferred, because the dependency on UP-316 / the THO targets
is real. Disposition C is what the group has actually voted for
twice and has been forced to revert both times — repeating it now
ignores the explicit signal from the 2025-10-08 re-open that the
work group wants this on a concrete plan, not in a holding pattern.
Disposition B threads that needle: it gives FHIR-35534 a real home
inside the larger FM/THO effort already under way, lets UTG line up
the THO value sets through UP-316, batches all sibling "anchor in
THO" PCT tickets so implementers see one coordinated canonical
flip rather than a trickle of breaking changes, and produces a
deliverable the work group can point to the next time this ticket
is reviewed.
