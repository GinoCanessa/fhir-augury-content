## Summary

This ticket asks the Financial Management (FM) work group to take ownership
of a spelling-standardization sweep across all FM-owned content in FHIR
Core, replacing every occurrence of "payor" with the previously agreed
preferred spelling "payer". It is the FM-scoped sibling of a small family
of related tickets that together cover the same standardization across
different specification surfaces: [FHIR-53190](https://jira.hl7.org/browse/FHIR-53190)
("Standardize spelling of 'payer' in core") is the original FM-on-core
request; [FHIR-54022](https://jira.hl7.org/browse/FHIR-54022) extends the
ask to FHIR Infrastructure-owned content for R6; and
[FHIR-54020](https://jira.hl7.org/browse/FHIR-54020) is the FHIR Management
Group ticket where Grahame Grieve already accepted the principle as
"Persuasive" with the resolution "Agreed, will correct spelling of payer
throughout." There is also long-standing precedent for the same
normalization in IGs — see the published
[FHIR-18921](https://jira.hl7.org/browse/FHIR-18921) ("Payor should be
payer - CRD #62"), which was applied auto-approved in 2019 to Da Vinci
CRD. FHIR-54005 is the FM-side action item that operationalizes that
agreed direction within content the FM work group owns (Account, Claim,
ClaimResponse, Coverage, CoverageEligibilityRequest/Response,
EnrollmentRequest/Response, ExplanationOfBenefit, InsurancePlan,
PaymentNotice, PaymentReconciliation, plus FM-owned value sets, code
systems, naming systems, financial-module narrative pages, and any FM
example resources).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept as Persuasive and execute a mechanical, FM-scoped rename of
"payor" → "payer" across all FM-owned core artifacts and supporting
content. Concretely:

1. **Resource definitions** (definitions, short labels, comments,
   requirements, mapping notes, invariant human text) for all FM-owned
   resources: Account, Claim, ClaimResponse, Coverage,
   CoverageEligibilityRequest, CoverageEligibilityResponse,
   EnrollmentRequest, EnrollmentResponse, ExplanationOfBenefit,
   InsurancePlan, PaymentNotice, PaymentReconciliation.
2. **FM-owned terminology**: value sets, code systems (display and
   definition text), and naming systems where "payor" appears in
   human-readable strings. Codes themselves (machine identifiers) are
   **not** to be renamed if doing so would break existing implementer
   bindings — only the display/definition text.
3. **Financial Module narrative pages** (`financial.html`,
   module-overview, security/privacy guidance, workflow narratives) and
   any FM-owned glossary entries.
4. **FM-authored examples and search-parameter descriptions** where the
   word appears in prose.
5. Out of scope for this ticket: external-org IG content, non-FM core
   content (covered by FHIR-54022 / FHIR-54020), and any code-level
   identifiers (element names, code values, slice names) — those would
   be breaking and require a separate, scoped change request.

The change should be planned as a single editorial PR per FM-owned repo
to keep diffs reviewable and to make the ballot/QA story trivial. Apply
in R6 (FHIR Core) and forward-port to FM IGs as separate per-IG tickets
if those IGs still contain "payor".

#### Justification

The FHIR Management Group has already accepted the same direction as
Persuasive in [FHIR-54020](https://jira.hl7.org/browse/FHIR-54020)
("Agreed, will correct spelling of payer throughout"), and FM's own IG
portfolio has been applying this exact rule since
[FHIR-18921](https://jira.hl7.org/browse/FHIR-18921) (Published, 2020).
Accepting here is consistent with both that cross-WG agreement and FM's
established editorial practice. The change is non-breaking because it
only touches human-readable text (definitions, descriptions, narrative)
— machine identifiers and bindings are explicitly out of scope.
Editorial-only changes don't require ballot reconsideration and have
near-zero implementer impact.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the standardization in principle but execute it via a single
cross-cutting "spelling normalization" tracker rather than the current
per-WG ticket fan-out. Specifically:

1. Close FHIR-54005 (and the duplicate
   [FHIR-53190](https://jira.hl7.org/browse/FHIR-53190)) as duplicates
   of the FMG umbrella ticket
   [FHIR-54020](https://jira.hl7.org/browse/FHIR-54020), which already
   has a Persuasive resolution from Grahame Grieve.
2. Have the publisher / build tooling team run a single
   case-preserving find-and-replace across all narrative and definition
   text (with an allow-list of any legitimate "Payor" tokens that must
   be preserved verbatim — e.g., quoted external terminology, X12
   payor-id legacy references, citation titles).
3. Capture the allow-list and the regex/replacement rule in a
   publication-tooling lint so future "payor" reintroductions are
   blocked at PR time.

#### Justification

There are at least four near-identical tickets (FHIR-53190, FHIR-54005,
FHIR-54020, FHIR-54022) plus the historical IG ticket FHIR-18921, all
asking for the same normalization on different content slices. Tracking
this work through one tooling-driven sweep is more reliable than asking
each WG to manually scrub their content and risks fewer misses,
inconsistent application, or new "payor" creep between releases. The
allow-list concern is real — "payor" appears in some externally-defined
identifiers (e.g., X12 "Payor Identification") and in cited paper
titles, where preserving the original spelling is correct. A
linter-backed sweep handles that more safely than per-WG manual edits.

The trade-off vs. Disposition A is governance: FM gives up direct
control over when its content is touched, and the change rides on
publisher-tooling timing rather than FM's editorial cycle.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54005 (and recommend the same for FHIR-53190) on the
ground that it is a duplicate of the already-resolved FMG umbrella
ticket [FHIR-54020](https://jira.hl7.org/browse/FHIR-54020). Mark
resolution as "Not Persuasive — Duplicate" and direct any execution
work to be tracked under FHIR-54020's resolution rather than spawning
parallel per-WG tickets that all describe the same edit.

#### Justification

FHIR-54020 has already been resolved Persuasive by FMG with the
direction "will correct spelling of payer throughout." Carrying
FHIR-54005 as an independent open ticket adds bookkeeping load, risks
divergent application, and clutters the FM tracker without changing the
outcome. Declining as a duplicate keeps the single source of truth on
FHIR-54020 and avoids three or four WGs each independently re-litigating
the same agreed-on editorial change.

The trade-off is operational: FMG umbrella tickets historically don't
generate per-WG action; without a separate FM ticket the FM-owned
content slice can be quietly missed. That risk is real, which is why
this disposition is the weakest of the three.

---

### Recommendation

**Recommended disposition:** A

Accept as Persuasive and execute a scoped, FM-owned editorial sweep.
The substantive question — should FM content say "payer" or "payor"? —
is already settled, both by FM's own decade-long practice
(FHIR-18921 / Da Vinci CRD) and by FMG's recent Persuasive resolution
on FHIR-54020. The remaining question is purely operational, and the
per-WG ticket pattern (FHIR-54005 for FM, FHIR-54022 for FHIRI) is the
mechanism HL7 has effectively chosen to make that operational work
visible and assignable. Disposition A is a near-zero-risk editorial
change, fully aligned with the cross-WG decision, and yields the
outcome the reporter is asking for. Disposition B is appealing if
publisher tooling can realistically run the sweep, but it requires
infrastructure investment that should not block FM from cleaning its
own content now. Disposition C is operationally weak: an FMG umbrella
"yes" without a per-WG ticket has historically not produced the actual
edits. Recommend Accept, scoped strictly to human-readable strings in
FM-owned core content, with machine identifiers explicitly excluded and
any genuinely external "Payor" tokens (e.g., X12 identifiers, citation
titles) called out in an allow-list during the PR review.
