## Summary

The ticket asks the Financial Management work group to **remove the single
`$member-match` operation from the Da Vinci PDex implementation guide**,
leaving payer-to-payer member matching to the bulk (`$bulk-member-match`)
flow only. The reporter (Michele Mottini, CareEvolution) argues that single
member match is (a) redundant with bulk member match when invoked for one
member, (b) under-specified — it relies on the New Payer obtaining an OAuth
2.0 access token scoped to a single specific patient, a mechanism that is
not defined in SMART Backend Services or in PDex itself, and (c) a
significant additional implementation burden on top of bulk member match
(mTLS, OAuth 2.0 dynamic registration, the operation itself, an unspecified
patient-scoping extension, and `$everything` and/or patient-level bulk
export on the data side).

The ticket originated from the Zulip thread *"Payer to Payer Single Member -
Scoping to one Patient"* on **#Da Vinci PDex**, where an implementer
(Artem Sopin) hit exactly the missing patient-scoping spec gap while trying
to follow Step 3 (Token Request) of the Single Member flow. That same
thread surfaces several closely related items that are *mentioned* but not
formally cross-linked: **FHIR-55928** (proposing to delete a stale
"SHALL support `$member-match`" sentence elsewhere on the same IG page),
**FHIR-52121** (`$everything` problems noted by Cooper Thompson),
**FHIR-52063** (CMS-0057 `Group/{id}/$export` vs. `$davinci-data-export`
conflict, resolved Not Persuasive), and **FHIR-49980** (cited only as a
voting-procedure example).

The work group has already triaged the ticket with a proposed resolution of
**Not Persuasive**, on the basis that "Single Member Match is retained
since it is in use by payers in certain regions, such as California."

## Details

**Description:**

Single member match

- is somewhat redundant: bulk member match can be used for a single member
- is underspecified: depends on requesting / granting an access token
  scoped to a specific patient that is not currently supported by SMART
  and not clearly specified (see
  https://chat.fhir.org/#narrow/channel/235286-Da-Vinci-PDex/topic/Payer.20to.20Payer.20Single.20Member.20-.20Scoping.20to.20one.20Patient)
- significantly increases implementation burden

so it should be removed from the implementation guide.

**Resolution Description (proposed by work group):**

> Single Member Match is retained since it is in use by payers in certain
> regions, such as California.

**Comments:**

### michelemottini — 2026-03-08

As currently written single member match requires implementing:

- Mutual TLS authentication
- OAuth 2.0 Dynamic Registration
- The member match operation itself
- An (unspecified) extension to OAuth 2.0 to request an access token scoped
  to a specific patient
- `$everything` and / or patient-level bulk export

All these things are in addition to what is required for bulk member match.

The objection that has been raised to removing the whole thing is that
single member match can be useful for live verification of the information
entered by a member when it consents to get data from their old payer(s).
If this is really a widespread requirement (the objection so far has been
raised only by a single payer) then it can be satisfied by maintaining just
the operation itself — and still get rid of everything else.

The dissent comes primarily from **Daniel Venton**, who wants single

should perform comparably to a single match. Artem Sopin suggests two
fixes if single match is kept — drop the patient-scoping requirement

adopt a UDAP extension to carry a `patient` claim — but acknowledges the

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the entire single `$member-match` flow from the Da Vinci PDex IG.
Concretely, in **HL7/davinci-epdx**:

- Delete the *Member Match with Consent* / Step 1–3 single-member
  description and sequence diagram from `payertopayerexchange.html` (or
  reduce it to a "removed in this version; use bulk member match" note).
- Remove the SHALL/SHOULD conformance statements that require Health Plans
  to support `$member-match` (including the stale sentence already
  flagged by FHIR-55928).
- Remove or mark as withdrawn the `OperationDefinition/MemberMatch` and any
  PDex-side examples and parameter profiles tied to single member match.
- Update the Capability Statements (Payer-to-Payer client / server) so
  that `$member-match` is no longer listed.
- Add a migration paragraph telling existing implementers to use
  `$bulk-member-match` (which already accepts a single-Patient
  `$multi-member-match` input bundle) plus `Group/{id}/$export`.
- File a downstream HRex ticket (or coordinate with HRex) to also retire
  / deprecate the underlying `$member-match` operation in HRex if PDex is
  the only consumer; otherwise mark PDex as no longer using it.

#### Justification

This matches the dominant implementer position in the Zulip thread (Epic,
CareEvolution, and Alexander Zautke all stated they are bulk-only) and
removes a documented spec gap — there is no defined way to obtain an
OAuth 2.0 access token scoped to a specific Patient under SMART Backend
Services, so the single-member flow as written cannot be implemented
interoperably. Eliminating the operation also removes the entire
mTLS + Dynamic Registration + patient-scoping + `$everything`/Patient
export stack from the conformance surface, materially reducing
implementation burden in line with the FM goal of getting PDex CMS-ready.
The concern about interactive verification at member sign-up (Daniel
Venton) is addressed by Mottini's observation that
`$bulk-member-match` can be invoked synchronously with a one-element
`$multi-member-match` bundle and complete in comparable time.

---

### Disposition B: Alternative Approach

#### Proposal

Do not remove `$member-match`, but **strip it down** so the spec gap is
closed and the implementation burden is dramatically reduced:

- **Drop the requirement that the access token be scoped to one Patient.**
  Specify instead that the New Payer authenticates with the same
  `client_credentials` token used for `$bulk-member-match` and that
  authorisation to retrieve data for the matched Patient is enforced via
  the Consent attached to the request, exactly as `$bulk-member-match`
  already does.
- **Remove the IG-level requirement for `$everything` / patient-level
  bulk export** on the response side; require only `Group/{id}/$export`
  against a server-generated single-member Group whose ID is returned by
  the operation (parallel structure to `$bulk-member-match`).
- **Mark the operation Trial-Use / optional**, with a clear note that
  `$bulk-member-match` is the recommended path and that
  `$member-match` is retained only for interactive single-member
  verification.
- Update `payertopayerexchange.html` accordingly; align Capability
  Statements; close FHIR-55928 (stale SHALL) at the same time.

#### Justification

This addresses the substantive complaint in the ticket — the
patient-scoping spec hole and the stack of dependent requirements —
without breaking implementations in the regions (e.g. California, per the
proposed resolution) that are already using single member match. It also
preserves Daniel Venton's interactive-verification use case in a way that
fits SMART Backend Services as it actually exists today, rather than
relying on an unspecified OAuth extension or a UDAP-only mechanism (which
Sopin himself flagged as a heavy burden on auth servers). The downside is
that the IG continues to carry two near-equivalent operations, slightly
more conformance text, and the risk that some implementers choose only
the smaller one and break interoperability with bulk-only deployments.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Do not remove `$member-match`. Confirm the work group's already-proposed
resolution: *"Single Member Match is retained since it is in use by payers
in certain regions, such as California."* Make no normative changes, and
direct implementers experiencing the patient-scoping issue to file a
separate, narrower ticket against the auth flow once a concrete proposal
exists (UDAP extension, profile of SMART Backend Services, etc.).

#### Justification

Removing a published normative operation between STU 2.1 and the next
PDex release is a breaking change for implementers who already deployed
it, particularly in markets where it is in production use. The reporter's
"only one payer objected" claim is not formally substantiated, and the
Zulip thread does include at least one strong retention voice (Daniel
Venton) plus an open mention of Mark Scrimshire who has not weighed in.
The patient-scoping gap, while real, is fundamentally an authorisation-
profile issue and could be closed by a future PDex/HRex/UDAP profile
without retracting the operation. This is the lowest-risk path for the
ballot cycle and is consistent with the proposed resolution already on
the ticket.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is the strongest fit because it acknowledges both sides of
the actual disagreement without papering over the spec gap. The reporter
and the majority of Zulip respondents are right that the *current*
single-member flow is not implementable as written — there is no
SMART-conformant way to obtain a single-Patient–scoped token, and the
ticket's enumeration of dependent requirements (mTLS + Dynamic
Registration + patient scoping + `$everything`/Patient export) is
accurate. Disposition C ("retain as-is") leaves that gap in normative
text, which is hard to defend at ballot. Disposition A (full removal) is
clean and matches what Epic, CareEvolution, and others say they will
actually deploy, but it deletes capability that is reportedly in
production in at least one region and forecloses Daniel Venton's
interactive-verification use case before a replacement pattern has been
agreed.

By keeping a **stripped-down, consent-gated, single-token** form of
`$member-match` and pointing implementers to `$bulk-member-match` as the
recommended path, the IG closes the spec hole, eliminates the most
expensive parts of the implementation burden (the unspecified OAuth
extension and the `$everything`/Patient-export side), preserves the
interactive use case, and stays backward-compatible enough that the
California deployments mentioned in the proposed resolution can continue
to operate. It also aligns naturally with the cleanup tracked by
FHIR-55928 and avoids re-opening the unrelated CMS-0057 /
`$davinci-data-export` argument tracked by FHIR-52063. The work group
should also note that this rewrite still belongs in the same ballot cycle
as PDex-Vote-1 to avoid leaving the contradictory text in published
normative content.
