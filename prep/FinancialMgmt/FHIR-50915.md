## Summary

FHIR-50915 is a small, already-triaged change request from Mark Scrimshire
(Financial Management) against FHIR R5+ asking for two coordinated edits to
the example code system `ex-providerqualification` (used to bind
`Claim.careTeam.qualification` and `ExplanationOfBenefit.careTeam.qualification`):

1. Rename the code system's *display name* from `ex-providerqualification`
   to "Provider Qualification" (i.e., a human-readable name; the
   technical id/url is presumably preserved or transitioned per the
   normal FM rebind pattern).
2. Strengthen the binding strength on the `careTeam.qualification`
   element from `example` (the current `ex-` prefix convention) to
   `extensible`.

The ticket has resolution **Persuasive** and carries the
`FM-R5+_ToBeApplied` label, indicating Financial Management has already
accepted it in principle and queued it for application in R5+. The
ticket itself has no Jira links, no GitHub cross-references, and no
direct Zulip cross-references in the Augury index. However, it is part
of an obvious cluster of FM "rebind / promote example code systems"
tickets (FHIR-50882, 50883, 50885, 50886, 50906, 50907, 50908, 50910,
50913) that share the same `tobeapplied / extensible / claim /
explanationofbenefit / r5` keyword set and the same intent: move
provider-side terminology away from `example` bindings rooted in
`ex-*` placeholder code systems and toward properly-named,
`extensible`-bound vocabularies.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the ticket literally in the FM-owned R5+ source artifacts:

1. **Rename the code system display name.** Change the `name` (and
   matching `title`) on the `CodeSystem/ex-providerqualification`
   resource from a value derived from the id (e.g.,
   `EX-PROVIDERQUALIFICATION` / `Example Provider Qualification Codes`)
   to `ProviderQualification` (name) / `Provider Qualification`
   (title). Leave the `id`, `url`
   (`http://hl7.org/fhir/ex-providerqualification`), and existing
   concept set unchanged so existing instances and IGs that already
   reference the canonical URL keep validating. Mark the rename in the
   resource's `meta` / changes log per the FM workgroup convention for
   `FM-R5+_ToBeApplied` items.
2. **Strengthen the binding.** On `Claim.careTeam.qualification` and
   `ExplanationOfBenefit.careTeam.qualification` (and any
   profile-level usage in core), change the binding `strength` from
   `example` to `extensible`, retaining the same `valueSet` reference
   (the value set derived from `ex-providerqualification`, or a
   sibling value set named `provider-qualification` if FM prefers to
   match the new display name).

#### Justification

This is the most literal reading of a Persuasive-resolved,
`FM-R5+_ToBeApplied`-labelled ticket: FM has already decided the
change should ship in R5+. The rename is purely cosmetic at the
canonical-URL level, so it does not break existing references; it
brings the code system in line with how every other "non-example"
FHIR core code system is labelled (no `ex-`/`example` prefix in the
display name). Strengthening the binding to `extensible` matches the
parallel decisions captured in FHIR-50886 and FHIR-50913 for
`careTeam.role`, keeping the `careTeam` substructure internally
consistent. The Da Vinci PCT and NUCC/VSAC Zulip threads above
demonstrate that real implementers already source provider
qualifications from concrete value sets (NUCC Provider Taxonomy);
`extensible` codifies that expectation without forcing a single
vocabulary.

---

### Disposition B: Alternative Approach

#### Proposal

Replace the `ex-providerqualification` code system rather than just
rename it, mirroring the approach taken by FHIR-50881 / FHIR-50882
(`Ex-DiagnosisType`, `Ex-ProcedureType`):

1. **Create a new CodeSystem** at
   `http://hl7.org/fhir/CodeSystem/provider-qualification` with
   `name = ProviderQualification`, `title = "Provider Qualification"`,
   `status = active`, `experimental = false`, and the same (or a
   curated) set of concepts.
2. **Create a matching ValueSet** at
   `http://hl7.org/fhir/ValueSet/provider-qualification` that includes
   the new code system.
3. **Retire the old artifact.** Mark
   `CodeSystem/ex-providerqualification` and its derived ValueSet as
   `status = retired` with a clear `purpose`/`description` note
   pointing to the new canonical, but keep them in the build for
   backwards reference.
4. **Re-bind both elements.** Point
   `Claim.careTeam.qualification` and
   `ExplanationOfBenefit.careTeam.qualification` at the new
   `provider-qualification` value set with `strength = extensible`.

#### Justification

The `ex-` URL prefix is widely understood (in FHIR Core and tooling)
as a marker that the code system is *example only*. Promoting an
`ex-*` artifact to an `extensible` binding without changing its URL
sends a mixed signal: the metadata says "example", the binding says
"normative-ish". Replacing it with a properly-named canonical aligns
the artifact's identity with its new role and matches the precedent
already set by sibling tickets in this same wave (FHIR-50881,
FHIR-50882, FHIR-50883, FHIR-50885). The cost is a one-time migration
burden for implementers using the old URL, mitigated by retaining the
retired artifact for traceability.

---

### Disposition C: Decline

#### Proposal

Decline the binding-strength portion of the request and apply only the
display-name rename (or decline both). Rationale: the
`careTeam.qualification` element is intentionally bound `example`
because there is no single internationally-applicable provider
qualification vocabulary; jurisdictions use NUCC (US), GMC (UK),
SNOMED CT subsets, regional registries, etc. Forcing `extensible`
implies a "you should use this value set unless you have a good
reason" stance that core FHIR cannot honestly take here.

#### Justification

Defensible only if the WG concludes the underlying value set cannot
be made representative enough to justify an `extensible` binding. The
PCT / NUCC Zulip context shows that even Da Vinci had to add NUCC
explicitly in a profile rather than rely on the core binding, which
is mild evidence that the core value set is not strong enough to
carry `extensible` semantics. This disposition would, however,
contradict the existing Persuasive resolution and the
`FM-R5+_ToBeApplied` label, so it should only be revisited if FM has
new information since 2025-05-15.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — full replacement).

The ticket is small and Persuasive, so doing nothing (C) is hard to
justify against FM's own already-recorded decision. Between A and B,
B is preferable because it removes the contradiction at the heart of
A: an `ex-*` canonical URL bound `extensible`. FM has already
established (FHIR-50881, FHIR-50882, FHIR-50883, FHIR-50885) that the
right move when promoting an example artifact is to mint a properly
named CodeSystem/ValueSet pair and retire the `ex-*` original.
Applying the same pattern here keeps the FM R5+ rebind wave
internally consistent, removes the "example" signalling from a
binding that is now meant to be normative-leaning, and gives
implementers a clean URL to migrate to. The mitigations are
straightforward: retain the retired artifact, document the
replacement, and (per FHIR-50910's precedent) consider whether the
new code system should live under terminology.hl7.org rather than in
core. The workgroup should confirm (a) which canonical URL the new
artifact takes, (b) whether THO is the right home, and (c) the
concept set for the replacement value set, ideally aligning with —
or at least being friendly to extension via — NUCC Provider Taxonomy
as already discussed in the IG-creation Zulip thread.
