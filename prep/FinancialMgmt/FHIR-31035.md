## Summary

The ticket asks the Financial Management work group to add a new concept,
`replacedby` ("Replaced by claim number" â€” *"The current claim has been
adjusted; i.e., replaced by or merged to another claim number."*), to the
core HL7 Related Claim Relationship code system
(`http://terminology.hl7.org/CodeSystem/ex-relatedclaimrelationship`).
The reporter notes that payer adjudication systems already use this
concept and that the existing core codes (`prior`, `associated`) do not
clearly express the "replaced or merged" relationship between an
amended/successor claim and the original.

This request is a **near-duplicate of [FHIR-27119](https://jira.hl7.org/browse/FHIR-27119)**
("Update Claim-Relationship CodeSystem to Reflect Industry Practice"),
which Saul Kravitz raised in 2020 with the same proposal and which
Financial Management resolved on 2024-02-02 as **Persuasive with
Modification**: instead of adding `replacedBy`, the WG agreed that
`prior` already covers the same concept and that the *definition* of
`prior` should be amended to "*A prior Claim instance for the same suite
of services. This is typically used when an amended Claim (prior
authorization, predetermination or claim) is submitted.*" That
disposition is captured in the most recent comment on the present
ticket from `pknapp` (2025-01-30) asking the reporter to confirm that
`prior` fulfils the business intent, or to bring a use case to FM that
demonstrates the need for both codes.

[FHIR-26709](https://jira.hl7.org/browse/FHIR-26709) provides the other
half of the context: in the CARIN BB IG (US Realm), Financial Management
took the alternate path of defining an IG-local code system,
`http://hl7.org/fhir/us/carin-bb/CodeSystem/C4BBRelatedClaimRelationshipCodes`,
which adds `replacedby` ("Replaced by claim number" â€” same definition as
this ticket) on top of `prior`, and binds it Required to
ExplanationOfBenefit.related.relationship in the C4BB profiles. So the
exact code the reporter wants already exists in the FHIR ecosystem â€” it
just lives in a CARIN-BB CodeSystem rather than in core terminology.

**Comments:**

- **pknapp (2025-01-30):** Asks Patricia Taylor to confirm that `prior`
  fulfils the business intent of `replacedby`, or to come to FM and
  present a use case illustrating the need for both codes. (This is the
  follow-up consistent with the FHIR-27119 disposition.)
- **pknapp (2022-05-11):** Notes a methodology consideration â€” while it
  would be efficient for the replaced claim to point to the claim that
  replaced it, the FHIR methodology guides that (1) the *successor*
  resource should point to the prior, and (2) circular references should
  be avoided. Implication: a system looking for "what replaced this
  claim?" must search for the successor whose `related.relationship =
  prior` references the given claim, rather than reading a forward
  pointer on the claim itself. Marks the ticket "hold for larger
  committee review."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new concept `replacedby` to the core CodeSystem
`http://terminology.hl7.org/CodeSystem/ex-relatedclaimrelationship`
(authoring source maintained by THO / FM-owned terminology), with:

- **Code:** `replacedby`
- **Display:** `Replaced by claim number`
- **Definition:** *"The current claim has been adjusted; i.e., replaced
  by or merged to another claim number."*

Include the new code in the bound ValueSet
`http://hl7.org/fhir/ValueSet/related-claim-relationship`, ensure both
Claim.related.relationship and ExplanationOfBenefit.related.relationship
pick it up automatically (they bind to that ValueSet at *example*
strength in core), and add a usage note clarifying that `replacedby` is
a *forward* pointer (the original claim points to its successor) while
`prior` is a *backward* pointer (the successor points to the claim it
replaces). Reverse and override the FHIR-27119 disposition explicitly,
recording in the change log that FM reconsidered after CARIN-BB
implementation experience.

#### Justification

The `replacedby` concept is already in production in the C4BB
implementation guide (FHIR-26709) with the identical code, display, and
definition; promoting it to core simply harmonises core terminology with
de facto industry practice and removes a redundant IG-local CodeSystem.
Payer adjudication systems track both directions of the
"prior/successor" relationship and expressing only one side in core
forces every payer-facing IG to either redefine the code (as C4BB did)
or to abuse `prior` semantically. Mona O.'s 2024 M3P discussion shows
that real implementers continue to be frustrated by the limited core
vocabulary.

### Disposition B: Alternative Approach â€” Clarify `prior` and document the directional convention

#### Proposal

Do not add a new code. Instead, formally apply the FHIR-27119
resolution: amend the definition of `prior` in the core CodeSystem to
*"A prior Claim instance for the same suite of services. This is
typically used when an amended Claim (prior authorization,
predetermination or claim) is submitted (including the case where the
prior claim has been replaced by or merged into the current claim)."*
Add a non-normative note to both Claim.related.relationship and
ExplanationOfBenefit.related.relationship explaining the directional
convention captured in pknapp's 2022-05-11 comment: per FHIR
methodology, the *successor* claim references the *prior* claim with
`relationship = prior`, and queries for "what replaced this claim?" are
satisfied by reverse-chained search rather than by a forward
`replacedby` pointer. Optionally cross-reference C4BB's
`C4BBRelatedClaimRelationshipCodes` CodeSystem in the comments as the
US-Realm example for callers who must use `replacedby` literally.

#### Justification

This is exactly the disposition Financial Management already reached on
the identical request in FHIR-27119 (Persuasive with Modification,
2024-02-02), and it is explicitly the path pknapp invited the reporter
toward in the 2025-01-30 comment. It is consistent with FHIR's
preference for *backward* references over circular forward pointers
(pknapp 2022-05-11), avoids two near-synonymous codes (`prior` /
`replacedby`) that implementers would have to disambiguate, and keeps
core terminology stable. C4BB implementers who *want* a forward
"replaced by" code already have one in their IG-local CodeSystem â€” the
mechanism FHIR uses precisely so that core does not have to absorb
every realm-specific concept.

### Disposition C: Decline (close as not persuasive)

#### Proposal

Decline the request and close as duplicate of FHIR-27119, on the basis
that FM has already considered and resolved this exact change. The
existing core code `prior` already expresses the "this claim has been
adjusted / replaced / merged" relationship via the FHIR-preferred
direction (successor â†’ prior), and US-Realm payer needs are already
served by `C4BBRelatedClaimRelationshipCodes`. No spec change is made.
Add a comment to the ticket pointing the reporter to FHIR-27119's
resolution and to the C4BB CodeSystem.

#### Justification

The request was previously made and resolved with a directed change
(amend `prior`'s definition); re-adopting it as a separate change risks
two competing dispositions on the same code system. The methodology
gotcha pknapp documented (avoid circular forward pointers) is a
real design constraint in FHIR â€” adding a forward `replacedby` invites
producers to populate *both* sides of the relationship and creates
exactly the cycle FHIR's reference rules try to avoid. Implementers who
need a forward pointer have an unobstructed path via IG-local
terminology, as C4BB has demonstrated.

### Recommendation

**Recommended disposition: B â€” Alternative Approach (clarify `prior`
and document the directional convention).**

Disposition B is what FM has already resolved to do (FHIR-27119,
*Resolved - change required*, label `FM-R5+_ToBeApplied`), and it
honours the explicit invitation from pknapp's 2025-01-30 comment on
this ticket: confirm that `prior` fulfils the intent. It addresses the
underlying implementer pain â€” that `prior`'s current definition is
narrow enough to *seem* not to cover the replaced/merged case â€” without
introducing a near-duplicate code that competes with `prior` and that
violates the FHIR methodology preference for backward references over
forward ones. It also leaves the C4BB `replacedby` code in place where
it is genuinely useful (US-Realm payer adjudication with a Required
binding) without forcing its semantics into every other realm's claim
data.

If, however, the reporter brings a concrete use case to FM that
*requires* expressing both directions in the same dataset (e.g.,
historical adjudication records where the original claim must self-
identify its successor and the successor cannot be reliably reverse-
chained), the WG should pivot to Disposition A and accept the code,
treating FHIR-27119's disposition as superseded by new evidence. Plain
Decline (C) should be avoided â€” closing without action would leave
FHIR-27119's "to be applied" definition change in limbo and the 2025
comment thread unresolved.
