## Summary

The reporter (Ron G. Parker) asks the Da Vinci Patient Cost Transparency
(PCT) IG to anchor one of its value sets in HL7 Terminology
(terminology.hl7.org / THO) rather than publishing it inside the IG,
noting that the value set has utility beyond PCT and that its underlying
code systems are already published in THO. The ticket is the fifth in a
sibling series filed by Parker against PCT (companions are
[FHIR-45812](https://jira.hl7.org/browse/FHIR-45812) for
`USClaimMedicalProductOrServiceCodes`,
[FHIR-45815](https://jira.hl7.org/browse/FHIR-45815),
[FHIR-45816](https://jira.hl7.org/browse/FHIR-45816), and
[FHIR-45818](https://jira.hl7.org/browse/FHIR-45818) for
`PCTAdjudication`); FHIR-45817 is labelled only "5" in the title, and
neither the description nor the imported comment 6 names the specific
value set being challenged, so the workgroup will need to recover the
intended VS from the original ballot comment context before acting.
The PCT WG previously dispositioned this ticket as **Considered for
Future Use** in the Block-2025-Jul block vote (Corey Spears / Rick
Geimer 9-0-2), with the intent to resolve it across IGs as part of the
broader FM/THO migration effort already underway, but Lynn Laakso
reverted that resolution on 2025-10-08 to re-open it post-STU2
publication, putting it back in front of the workgroup.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the PCT value set referenced by this ticket (the unnamed "5" VS;
WG must confirm which VS the original ballot comment 6 targeted — the
sibling tickets pin theirs to `USClaimMedicalProductOrServiceCodes` and
`PCTAdjudication`, so the candidate set is the remaining
PCT-IG-defined value sets whose code systems are already in THO) out of
the PCT IG and publish it in HL7 Terminology (terminology.hl7.org). In
the PCT IG, replace the inline `ValueSet` resource with a reference to
the new THO canonical URL (e.g.
`http://terminology.hl7.org/ValueSet/<name>`), retire the
`hl7.fhir.us.davinci-pct`-scoped canonical with a deprecation note, and
update every PCT profile binding (`ElementDefinition.binding.valueSet`)
that previously pointed at the IG-local canonical to point at the THO
canonical. Coordinate the THO publication with the Vocabulary WG and
schedule the URL switch with the next PCT publication post-STU2.

#### Justification

The reporter's stated rationale — the underlying code systems already
live in THO and the value set has utility beyond PCT — aligns with HL7's
standing guidance that reusable terminology artifacts belong in THO so
they can be referenced consistently across IGs. The PCT WG itself
acknowledged this on 2025-07-08 by accepting the disposition direction
("Resolve across IGs continuing to add to THO"); Lynn Laakso's
2025-10-08 revert reopens the ticket precisely so the WG can act now
that STU2 has shipped. Acting in PCT first establishes the canonical and
unblocks the sibling tickets (FHIR-45812/-45815/-45816/-45818).

---

### Disposition B: Alternative Approach — Defer to the FM/THO Migration Track

#### Proposal

Keep the value set in the PCT IG for STU2.x but formally enroll
FHIR-45817 (and its sibling tickets) in the cross-IG FM/THO migration
work the WG referenced in the 2025-07-08 disposition note. Concretely:
(1) tag the ticket with the FM/THO migration tracking label / epic;
(2) record an explicit "future canonical" comment on the PCT VS page
indicating intended relocation to THO and the expected canonical URL;
(3) commit to the migration in a coordinated batch with Vocabulary WG so
that PCT, the other FM IGs, and any downstream consumers can switch
canonicals together in a single STU release rather than piecemeal.

#### Justification

A unilateral move of a single value set out of PCT introduces a
canonical-URL break for implementers already on STU2, and the WG note
from Spears (Block-2025-Jul) explicitly warned that the VS "will require
a greater level of review across multiple IGs." Doing the relocation as
part of a coordinated FM/THO batch limits the number of breaking
canonical changes implementers absorb, lets Vocabulary WG own the THO
governance review once, and keeps PCT's STU2.x publication stable. The
trade-off vs. Disposition A is timing only — the end-state is the same,
but the migration risk is amortised.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Close FHIR-45817 as **Not Persuasive** on the grounds that (a) ballot
comment 6 did not name the specific value set being challenged, leaving
the request under-specified, and (b) where the VS is genuinely
PCT-specific (binding semantics, intensional definition tied to PCT
workflows), THO is not the appropriate home and IG-local publication is
the standing pattern. Ask the reporter to file a fresh, VS-specific
ticket if a particular PCT value set still warrants relocation.

#### Justification

This is defensible only if the WG concludes, after recovering the
original comment-6 context, that the targeted VS is materially
PCT-specific and not reusable beyond PCT — in which case the IG-local
location is correct under HL7's "publish where you bind" guidance. The
under-specified description ("anchor the valueset in THO" without
naming the VS) is also a legitimate procedural reason to decline, since
the WG cannot act precisely without identifying the artifact. The cost
of declining is community signal: the reporter has filed five sibling
tickets in the same vein, and a blanket decline would conflict with the
WG's own 2025-07-08 statement that THO consolidation is in progress.

---

### Recommendation

**Recommended disposition:** B

The PCT WG has already publicly committed (in the Block-2025-Jul thread
and its 2025-07-08 vote) to resolving these THO-anchoring tickets as
part of a larger, cross-IG FM/THO migration effort that is "already
gotten underway in earnest." Lynn Laakso's 2025-10-08 revert reopens the
ticket post-STU2 not because the strategic answer changed but because
the WG now has the bandwidth to act; the right action is therefore to
commit to executing that pre-existing plan rather than either rushing a
unilateral PCT-only relocation (Disposition A — risks an uncoordinated
canonical break for STU2 implementers and pre-empts Vocabulary WG
governance review) or declining the request (Disposition C —
contradicts the WG's own stated direction and ignores the reporter's
rationale that the underlying code systems are already in THO).
Disposition B preserves implementer stability through STU2.x, keeps the
ticket actionable rather than parking it, and forces the prerequisite
step the WG must do anyway: identify which specific PCT value set
"comment 6 / ticket 5" refers to so the migration backlog has a
concrete entry. If, during that identification step, the WG discovers
the targeted VS is actually PCT-specific, the disposition can be
amended toward C with a documented rationale.
