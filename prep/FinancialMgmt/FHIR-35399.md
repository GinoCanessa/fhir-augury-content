## Summary

The ticket asks that the canonical (defining) URL of CARIN BB's
`CodeSystem/C4BBSupportingInfoType` be anchored in **terminology.hl7.org
(THO)** rather than under `hl7.org/fhir/us/carin-bb/...`, and that a
companion UTG ticket be opened to register the CodeSystem in THO. It is
one of a tightly related family of "Defining URL should be anchored in
THO" change requests filed by Joan Harper against CARIN BB STU2
(siblings include FHIR-35390 `C4BBAdjudicationDiscriminator`,
FHIR-35391 `C4BBClaimCareTeamRole`, FHIR-35398
`C4BBRelatedClaimRelationshipCodes`, FHIR-35401/35402 ValueSet
counterparts, and FHIR-33202 ADA tooth code system). FhirAugury surfaces
no formal cross-referenced Jira tickets â€” the only cross-reference is
the CARIN BB Block-Vote-6 announcement on Zulip, where the workgroup
disposed of FHIR-35399 as **"Considered for Future Use"** with the note
"May move this CodeSystem to THO for a future version based on
harmonization work with other IGs." That disposition (Corey Spears /
Kathleen Conner, 13-0-0, 2022-07-26) was later reverted by Lynn Laakso
on 2022-11-28 to **re-open post STU2 publication**, leaving the ticket
in `Triaged`/`Unresolved` and ready for a fresh disposition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move `C4BBSupportingInfoType` out of the CARIN BB IG's local code
systems and re-anchor its canonical URL at THO, e.g.
`http://terminology.hl7.org/CodeSystem/C4BBSupportingInfoType` (or an
equivalent THO-conformant identifier agreed with the UTG WG). Concrete
work items:

1. File the companion **UTG** ticket the description calls for, asking
   that `C4BBSupportingInfoType` be registered in THO with the same
   codes/display values currently published under
   `http://hl7.org/fhir/us/carin-bb/CodeSystem/C4BBSupportingInfoType`.
2. In `HL7/carin-bb`, remove the local CodeSystem definition once UTG
   publishes it, replace the `ValueSet-C4BBSupportingInfoType` includes
   with the new THO system URL, and update every profile (notably the
   `ExplanationOfBenefit` C4BB profiles' `supportingInfo.category`
   bindings) that references the old canonical.
3. Publish a release note + migration guidance in the CARIN BB IG
   identifying the canonical URL change, since `system` values appear in
   conformant instances.

#### Justification

This is the literal request and is consistent with HL7's general policy
that reusable CodeSystems live in THO, not in IG-local namespaces. It
matches the disposition already adopted for the sibling tickets
(FHIR-35390/35391/35392/35398) that the FM WG cleared via the UP-X UTG
work. Doing the same for `C4BBSupportingInfoType` keeps the CARIN BB
terminology surface consistent and lets other IGs reuse the system
without depending on `hl7.org/fhir/us/carin-bb/...`.

---

### Disposition B: Alternative Approach

#### Proposal

Treat `C4BBSupportingInfoType` differently from its siblings: keep the
existing IG-local canonical
`http://hl7.org/fhir/us/carin-bb/CodeSystem/C4BBSupportingInfoType` for
all currently-published codes, but begin **harmonization** with adjacent
IGs (Da Vinci PDex, Da Vinci CARIN-RTPBC, etc.) under a new THO
CodeSystem (e.g. `SupportingInfoType`) into which CARIN-specific codes
migrate over time. In CARIN BB, retain the local CodeSystem as a
deprecated alias that `compose.include`s the THO CodeSystem plus any
CARIN-only residual codes; mark the local CodeSystem `status =
deprecated` with a `replacedBy` extension pointing at the THO system.

#### Justification

The Zulip search shows `C4BBSupportingInfoType` is broadly cited in
implementer instances (CareTeam validation thread, BenefitPaymentStatus
thread, the fhirpath subsetOf example) â€” a hard re-anchor of the
canonical URL is a breaking change for already-deployed payer
extracts. The original 2022-07-26 disposition explicitly tied the THO
move to "harmonization work with other IGs", and Lynn Laakso reverted
the closure precisely because that work hadn't landed. Disposition B
honors that intent: it commits to harmonization (the underlying need)
without invalidating instances built against STU2's published canonical
during the interim.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** (or **Considered for Future Use** with no
commitment) on the grounds that `C4BBSupportingInfoType` is a CARIN
BB-specific slicing/categorization vocabulary (it carries codes such as
`admissionperiod`, `billingnetworkcontractingstatus`,
`additionalbodysite` that are scoped to CARIN BB's
`ExplanationOfBenefit.supportingInfo` design) and does not warrant a THO
canonical until cross-IG harmonization actually occurs. Document the
rationale on the ticket; leave the CodeSystem where it is.

#### Justification

A defensible decline on three grounds: (i) the CodeSystem's content is
IG-specific and doesn't currently have known reusers outside CARIN BB
(unlike Care Team Role / ADA tooth codes, which had concrete UP-X
UTG work in flight); (ii) re-anchoring the canonical breaks every
CARIN BB instance currently in production at payer organizations, with
no offsetting interoperability win until a harmonized THO CodeSystem
exists; and (iii) the workgroup already considered and applied this
disposition once (2022-07-26, 13-0-0) â€” the only reason it was reverted
was procedural (post-STU2 publication housekeeping), not substantive.
Closing as Not Persuasive with a pointer to the harmonization
prerequisite is consistent with that prior vote.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach).

Disposition A is the cleanest in principle but materially different
from FHIR-35399's siblings in one respect: the comments record explicit
WG concern that this particular CodeSystem's THO migration is gated by
*harmonization with other IGs* â€” there is no in-flight UP-X UTG ticket
named on this issue (unlike FHIR-35390/35391/35392/35398), and Lynn
Laakso re-opened the ticket precisely because that harmonization wasn't
done. Disposition C is defensible but throws away the reporter's
underlying concern (canonical URL governance) and will likely re-surface
the next ballot cycle. Disposition B captures both signals: it
acknowledges the harmonization gating the WG already articulated, while
committing to a deprecation path that keeps current implementers
non-broken (consistent with the active implementer use observed on
Zulip) and gives the next CARIN BB version a clear THO target. The
workgroup should pair this with a freshly filed UP-X / UTG ticket â€” the
one the original ticket asked for â€” naming the harmonization partners
explicitly so this doesn't drift again.
