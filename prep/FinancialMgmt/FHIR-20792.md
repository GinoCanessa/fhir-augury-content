## Summary

The reporter (John Moehrke) observed that
`http://build.fhir.org/codesystem-provenance-agent-type.html` is rendered
in the FHIR build as being used by two identical value sets. The root
cause appears to be that the Contract resource's source folder carries
its own duplicate copies of CodeSystem and ValueSet definitions that
are owned by other workgroups (e.g., Provenance, Consent), so the build
ends up registering two instances of the same canonical definition.
The resolution narrative on the ticket lists six specific duplicates
that should be removed from the Contract folder
(`codesystem-provenance-agent-type`, `codesystem-provenance-agent-role`,
`valueset-provenance-agent-role`, `valueset-provenance-agent-type`,
`valueset-consent-content-class`, `valueset-consent-content-code`),
notes that there may be others, and a 2021 follow-up comment from John
Moehrke narrows the still-outstanding pair to
`valueset-consent-content-class` and `valueset-consent-content-code`.
Lloyd McKenzie's comment supports simply deleting the unused duplicates
since "value set definitions should only exist in one place." There
are no cross-referenced Jira tickets; the only related discussion is
the originating Zulip post in `#fmg > Duplicated codesystem and
valuesets` that opened the GForge tracker item.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the duplicated terminology files from the Contract source
folder so that each canonical URL is defined exactly once, by its
owning workgroup. Specifically, delete from the Contract resource
folder any of the following that still resolve there as duplicates of
their canonical owners:

- `codesystem-provenance-agent-type` (canonical owner: Provenance / SD WG)
- `codesystem-provenance-agent-role` (canonical owner: Provenance / SD WG)
- `valueset-provenance-agent-role` (canonical owner: Provenance / SD WG)
- `valueset-provenance-agent-type` (canonical owner: Provenance / SD WG)
- `valueset-consent-content-class` (canonical owner: CBCC / Security)
- `valueset-consent-content-code` (canonical owner: CBCC / Security)

Per John Moehrke's 2021-12-01 comment, only
`valueset-consent-content-class` and `valueset-consent-content-code`
remain duplicated in the Contract folder today; remove those at a
minimum. Sweep the Contract folder for any additional CS/VS files
whose `url` matches a definition published from another resource's
folder and remove those as well. Confirm Contract's bindings still
reference the surviving canonical URLs (no rewrites of Contract
element bindings should be needed; the URLs are unchanged).

#### Justification

This is exactly the action requested in the ticket and endorsed by
both commenters. ValueSet and CodeSystem canonical URLs are required
to resolve to a single definition; duplicate source files in a
non-owning resource's folder produce build-time non-determinism (the
"luck-of-the-draw" the reporter flagged) and incorrect
"used-by-two-identical-valuesets" reports against owning workgroups'
artifacts. Deleting the duplicates restores single ownership without
any normative change to Contract or to the affected terminologies.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than delete in a single change, treat the cleanup as a
spec-hygiene work item with two staged steps:

1. Add an IG-publisher / build-tool guard that fails the build (or
   emits a high-severity QA error) when two source files declare the
   same `CodeSystem.url` or `ValueSet.url`. This makes the underlying
   class of bug self-policing for every workgroup, not just Contract.
2. As a follow-up under that guard, remove the still-duplicated
   `valueset-consent-content-class` and `valueset-consent-content-code`
   from the Contract folder, plus any further duplicates the new
   guard surfaces across the spec.

#### Justification

Disposition A fixes the immediate Contract symptom but leaves the
class of error open: any future resource that copies a terminology
file into its own folder will reintroduce the same
non-deterministic-build problem. A build-time uniqueness check would
prevent regression spec-wide and would surface other latent
duplicates the reporter explicitly suspected ("There may be others
that I am not an authority on that are also duplicated."). The
trade-off versus A is scope: this requires tooling work outside FM's
direct purview (likely FHIR-I / FMG / build-tooling) and so takes
longer to land, while the underlying Contract files remain
duplicated until step 2 ships.

---

### Disposition C: Decline

#### Proposal

Close as "Persuasive with Mod" only insofar as the Contract folder
should never have owned these definitions, but decline to take any
further normative action because: (a) the canonical URLs already
point to the owning workgroups' definitions, (b) consumers reading the
spec see a single rendered ValueSet/CodeSystem at the canonical URL,
and (c) the visible artifact of the duplication is a build-side QA
report rather than an interoperability defect.

#### Justification

This rationale would only be defensible if the duplicate files were
already inert (i.e., not picked up by the build at all) and the
"used by two identical value sets" rendering were a stale-cache
artifact. The reporter's evidence and Lloyd McKenzie's comment both
contradict that: the files are still in the resource folder, the
build still picks them up, and the duplication still resolves
non-deterministically. Declining therefore leaves a known
non-deterministic-build hazard in place and is not recommended; this
disposition is included for completeness so the workgroup can
explicitly reject it.

---

### Recommendation

**Recommended disposition:** A

The ticket has been open since 2019, both substantive comments agree
the duplicate files should simply be removed, and the reporter's 2021
follow-up has already narrowed the remaining work to two specific
ValueSets in the Contract folder (`valueset-consent-content-class`
and `valueset-consent-content-code`). The change is mechanical —
delete the duplicate source files; no Contract bindings change
because the canonical URLs are unchanged — and it removes a
documented build non-determinism that affects how Provenance- and
Consent-owned terminologies render. Disposition B (a build-time
uniqueness guard) is a worthwhile follow-up but should not block this
narrow cleanup; FM can take A now and request the broader guard as a
separate ticket against the build tooling. Disposition C is included
only to document why FM should not leave the duplicates in place.
