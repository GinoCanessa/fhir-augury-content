## Summary

FHIR-50910 asks the Financial Management work group to (1) move the
`datestype` CodeSystem out of the FHIR Core specification and into HL7
Terminology (THO, terminology.hl7.org), and (2) change the binding
strength on the element(s) currently bound to that CodeSystem from its
present strength to **Extensible**. The ticket is part of the FM R5+
"ToBeApplied" tranche of housekeeping changes that relocate
financial-domain code systems from FHIR Core to THO and relax the
matching bindings so implementers may extend the value set with local
or jurisdictional codes.

There are no explicit Jira links and no recorded cross-references in
Zulip, GitHub, or Confluence. However, keyword similarity surfaces a
large family of sibling FM tickets that perform the same pattern of
change against neighbouring FM CodeSystems
([FHIR-50907](https://jira.hl7.org/browse/FHIR-50907) — funds-reserve;
[FHIR-50906](https://jira.hl7.org/browse/FHIR-50906) — process-priority;
[FHIR-50908](https://jira.hl7.org/browse/FHIR-50908) — payeetype;
[FHIR-50886](https://jira.hl7.org/browse/FHIR-50886) — claim-careteamrole;
[FHIR-50883](https://jira.hl7.org/browse/FHIR-50883) — Ex-DiagnosisOnAdmission;
[FHIR-50885](https://jira.hl7.org/browse/FHIR-50885) — related-claim-relationship;
[FHIR-50882](https://jira.hl7.org/browse/FHIR-50882) — Ex-ProcedureType;

## Proposed Dispositions

The ticket has already been marked **Persuasive** by FM and tagged
`FM-R5+_ToBeApplied`, so Disposition A reflects the standing FM
decision; B and C are presented to make the trade-offs explicit for the
review record.

### Disposition A: Accept As Requested

#### Proposal

Apply the requested change in two coordinated edits:

1. **Move the CodeSystem to THO.**
   - Author the new resource at `terminology.hl7.org` as
     `CodeSystem/datestype`, preserving the existing canonical URL
     `http://hl7.org/fhir/datestype` if FM wants binding URLs in
     existing implementations to remain valid, OR mint a new THO
     canonical (e.g., `http://terminology.hl7.org/CodeSystem/datestype`)
     and add a deprecation note on the FHIR Core copy. The sibling FM
     tickets (FHIR-50907, FHIR-50906, FHIR-50908) should be
     cross-checked to keep the canonical-URL strategy consistent across
     the tranche.
   - Mark the FHIR Core copy as deprecated/retired in R6 with a
     pointer to the THO location.

2. **Change the binding strength to Extensible.**
   - Locate the ValueSet that wraps `datestype` (typically
     `http://hl7.org/fhir/ValueSet/datestype`) and ensure it points at
     the THO CodeSystem.
   - Update every `binding` block referencing that ValueSet on
     Claim, ClaimResponse, and ExplanationOfBenefit (the FM family
     where date-type codes appear on `*.diagnosis`, `*.procedure`,
     and similar elements) to `strength = extensible`.
   - Regenerate StructureDefinition snapshots, JSON/XML schema, and
     the resource HTML so the binding tables reflect the new strength.

#### Justification

This is precisely what the reporter requested and matches the pattern
already accepted for at least nine sibling FM CodeSystems in the same
"FM-R5+_ToBeApplied" tranche (FHIR-50906/50907/50908/50886/50883/50885,
etc.). FM has already resolved the ticket Persuasive, so accepting the
change preserves the consistency of the FM terminology cleanup
programme. Relocating financial code systems to THO aligns with HL7's
broader policy that terminology assets live in THO rather than in core
specifications, and Extensible is the established FM default for
domain code systems where jurisdictions or payers may need to extend
the list (as ANSI X12-aligned and payer-specific date type codes
typically do).

---

### Disposition B: Alternative Approach

#### Proposal

Accept the THO relocation, but bind to **Preferred** rather than
**Extensible**, OR keep the binding **Required** if FM concludes that
allowing implementer extension would compromise downstream EOB/Claim
processing semantics. If a stronger guarantee is desired without
blocking extension, an alternative is to publish the value set as
Extensible while also publishing a tightly-scoped "must-support" subset
(e.g., a `datestype-core` ValueSet) and binding the FM resources to
that subset with strength Required, leaving room for jurisdictional
profiles to widen the binding to the full Extensible value set.

#### Justification

`datestype` codes drive payer/clearing-house logic (e.g., date of
service vs. admission vs. discharge). Switching from a tighter binding
to Extensible without a controlled subset risks fragmentation across
implementations and complicates conformance testing. Disposition B
preserves the relocation benefits (governance, lifecycle, reuse) while
giving FM a tool to defend interoperability of the most critical date
types. The cost is additional editorial work (a second ValueSet, a
profile, and clearer guidance) and a divergence from the simpler
pattern adopted in the sibling tickets.

---

### Disposition C: Decline

#### Proposal

Decline the change for R5+: leave `datestype` in FHIR Core and leave
the binding at its current strength. Reasons would be (a) the
disruption of moving a CodeSystem whose canonical URL is widely
referenced, (b) the absence of a documented use case in the ticket
that requires Extensible, and (c) the lack of recorded community
discussion in Zulip / GitHub justifying the relaxation.

#### Justification

Defensible only on conservatism grounds: the ticket itself contains
two short sentences and no rationale, no Zulip thread was found, and
there are no recorded cross-references. A reviewer who weighed
backward-compatibility risk above the FM cleanup programme could argue
the change should not proceed without an explicit migration plan for
existing implementations that hard-code the FHIR Core canonical URL.
However, this is contradicted by FM's existing Persuasive resolution
and by the consistent acceptance of the same pattern across the
neighbouring FM CodeSystem tickets, so this disposition is included
mainly for completeness.

---

### Recommendation

**Recommended disposition:** A

FM has already resolved this ticket **Persuasive** and labelled it
`FM-R5+_ToBeApplied`, and the same pattern (relocate to THO + Extensible
binding) has been accepted across at least nine sibling FM CodeSystem
tickets in the same tranche. Accepting as requested keeps the FM
terminology clean-up internally consistent, aligns with HL7's THO
governance policy, and gives implementers — particularly U.S. payers
whose date-type vocabularies extend beyond the FHIR Core list — the
extensibility they need without forcing a profile. Disposition B's
"controlled subset + Required" idea is worth raising in the meeting
only if FM has concrete evidence that current `datestype` users rely
on the binding being closed; absent that evidence, the simpler,
consistent approach in A should be applied during the FM-R5+
"ToBeApplied" sweep.
