## Summary

The ticket requests that the Da Vinci PAS IG tighten the `Item Authorized
Provider` extension so that its nested `providerType` sub-extension is
required whenever the parent extension is present. The reporter grounds
the request in X12 Loop 2010EA, which is the source loop the extension
models: in X12, the authorized-provider context is keyed by provider
type, so leaving `providerType` optional on the FHIR side allows the
extension to be populated in ways that cannot round-trip to X12. The
reporter also notes, as a weaker secondary point, that the `provider`
sub-extension is arguably required too — but since 2010EA treats the
provider identity details themselves as optional at the loop's base
level, they stop short of formally requesting that change.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the Augury search (see sections below), so this ticket is
being reviewed in isolation.

## Details

**Description:**

Currently no sub-extensions in the Item Authorized Provider extension are
required. Based on the context of this data and the corresponding X12
Loop (2010EA), the nested providerType extension should be required when
the Item Authorized Provider extension is present. Arguably the nested
provider extension should be required as well, since the extension as a
whole seems pretty reliant on that context, but the line to X12 is a
little less clear since basically all of the provider details are
optional in the loop at a base level.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Da Vinci PAS IG, tighten the `Item Authorized Provider`
extension (applied on `Claim.item` in the PAS profiles) so that its
`providerType` sub-extension has minimum cardinality `1..1` while the
parent extension itself remains `0..*`. Concretely: locate the
StructureDefinition for the Item Authorized Provider extension (the
sliced extension with `providerType`, `provider`, and any other nested
slices), change the `providerType` slice's `min` from `0` to `1`, and
update the narrative / table in the extension's spec page to state that
`providerType` is required whenever the parent extension is used. Leave
`provider` and any other sub-extensions at their current optional
cardinalities. Add or amend an invariant/comment on the parent extension
explicitly documenting the link to X12 Loop 2010EA so implementers
understand why the constraint exists.

#### Justification

Requiring `providerType` when the parent extension is present directly
mirrors X12 Loop 2010EA, where the provider's role/type is the keying
context for the loop. PAS's explicit design goal is bidirectional
fidelity with X12 278, so a nested element whose absence makes the X12
mapping ambiguous should be required at the FHIR layer. The change is
minimally invasive: only the nested slice's `min` is tightened; the
parent extension stays optional, so producers that do not need an
authorized provider are unaffected. It also aligns with the general
FHIR profiling principle that when a data element is only meaningful
in combination with a co-occurring element, that dependency should be
expressed as a cardinality or invariant rather than left to prose.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of bumping `providerType`'s minimum cardinality, express the
constraint as a FHIRPath invariant on the parent `Item Authorized
Provider` extension — e.g.
`extension.where(url = 'providerType').exists()` with human text
"When Item Authorized Provider is present, providerType SHALL be
present." Keep the slice-level cardinalities unchanged and add the
invariant both to the StructureDefinition and to the extension's
narrative page. Optionally, add a secondary softer invariant
(severity `warning`) flagging that `provider` is expected in most
realistic exchanges, to capture the reporter's secondary concern
without mandating it.

#### Justification

An invariant expresses the same requirement as a cardinality bump but
is easier to evolve: if a future revision of the X12 mapping, or a
realm/trading-partner profile, needs to relax or re-shape the rule, an
invariant can be overridden or retired more cleanly than a cardinality
already baked into the base IG's extension. It also lets the IG
capture the *conditional* nature of the rule ("required **when the
parent extension is present**") explicitly, rather than relying on
readers to understand that a sub-extension's `min=1` is only evaluated
when the parent is instantiated. The trade-off is that some validators
surface invariant failures less loudly than cardinality failures, and
authoring tools cannot drive UI "required field" affordances from an
invariant the way they can from cardinality.

---

### Disposition C: Decline

#### Proposal

Decline the change and document the expectation in the extension's
narrative only: add text explaining that when the Item Authorized
Provider extension is populated, `providerType` is strongly expected
and SHOULD be present to support the X12 Loop 2010EA round-trip, but
leave all sub-extension cardinalities as-is. Treat the ask as a
documentation / conformance-guidance issue rather than a normative
constraint.

#### Justification

Two arguments support declining. First, PAS profiles layer on top of
core Claim/ClaimResponse resources, and PAS implementations are
already gated by rigorous 278 round-trip testing; adding a normative
`min=1` on a nested slice could break existing instances that were
otherwise acceptable under PAS's current guidance and that consumers
have already accommodated. Second, the reporter themselves concede
the X12 line is "less clear" for the sibling `provider` sub-extension
— tightening only one slice may over-specify relative to what 2010EA
actually mandates. A narrative-only clarification preserves the
current interoperability surface while still improving implementer
guidance.

---

### Recommendation

**Recommended disposition:** A

The requested change is small, well-motivated by the X12 Loop 2010EA
mapping that PAS is explicitly trying to preserve, and localized to a
single nested extension slice. Disposition B (invariant) is
defensible and arguably cleaner semantically, but in practice most
PAS implementers consume the StructureDefinition via tooling that
renders `min=1` as a first-class "required" affordance and surfaces
missing-element failures more prominently than invariant failures,
which matters for an IG whose whole value proposition is rigorous
payer/provider conformance. Disposition C under-delivers: PAS exists
precisely because prose-only guidance in payer/provider exchange has
historically produced ambiguous implementations. The workgroup
should adopt A, clearly scope the change to the `providerType` slice
only (leaving `provider` optional per the reporter's own caveat), and
add a pointer to X12 Loop 2010EA in the extension narrative so the
rationale is discoverable by future maintainers.
