## Summary

The reporter asks the Da Vinci PAS workgroup to clarify the relationship between
CPT and HCPCS coding in Prior Authorization Support (PAS) exchanges, and to
provide explicit guidance — and worked examples — for round-trip preservation of
the original codes a clinician/client supplied. Specifically, when a client
sends a CPT code, PAS must internally normalize to what X12 278 expects, but
the IG today gives no recommended pattern for retaining the original
clinician-facing CPT representation alongside the normalized form. The reporter
also wants a recommended strategy for distinguishing "original code" from
"normalized code" — either as documentation only or, if the workgroup prefers,
via an optional extension pattern — and a tightened set of service-type
examples that show CPT (professional services) and HCPCS Level II (DME / drugs)
being expressed in PAS in a way that remains conceptually consistent with the
matching CRD/DTR exchanges that fed the request. No Jira, Zulip, or GitHub
cross-references were surfaced for this ticket, so the request stands on its
own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next post-STU2.2 release of the US Da Vinci PAS IG, add a dedicated
narrative section (e.g., under the "Coding" / "Use of Terminologies" page)
that:

1. Describes the CPT vs HCPCS Level II split for the `Claim.item.productOrService`
   element and the corresponding `ClaimResponse` items, including which code
   system URI to use for each (`http://www.ama-assn.org/go/cpt` for CPT,
   `https://bluebutton.cms.gov/resources/codesystem/hcpcs` /
   `urn:oid:2.16.840.1.113883.6.285` for HCPCS Level II as appropriate).
2. Documents the X12 278 normalization rules PAS performs server-side so
   implementers know what gets rewritten on the wire.
3. Specifies the round-trip pattern as **multiple `coding` entries inside the
   same `CodeableConcept`** — preserve the clinician-supplied CPT/HCPCS
   `Coding` and add the X12-normalized `Coding` alongside it, with the
   original-vs-normalized distinction carried via the existing
   `coding-conversion` / "translation" extension (or, if the WG prefers, a
   newly minted PAS-specific extension `pas-original-code`) flagged on the
   normalized entry.
4. Adds at least three new fully worked PAS request/response example
   instances: (a) a professional service requested with CPT, (b) a DME item
   requested with HCPCS Level II, and (c) a drug requested with HCPCS Level
   II J-code, each showing the CRD recommendation and DTR questionnaire
   payload that fed the PAS request so reviewers can see the codes round-trip
   consistently across the three IGs.

#### Justification

This is exactly what the reporter asks for and what the WG previously
acknowledged was needed when it voted 10-0-0 to defer ("We will consider
adding examples and guidance in a future version"). The change is additive —
no profile tightening, no new required elements — so it is non-breaking for
existing PAS implementers. Using a multi-`coding` `CodeableConcept` is the
standard FHIR idiom for "same concept, multiple representations" and is
already idiomatic in claims, so the bar for adopting it is low. Pulling
matched CRD/DTR/PAS examples through the IG also discharges a long-standing
implementer complaint that the three Da Vinci IGs are hard to read together.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the documentation-only half of the request and skip a dedicated
extension. Specifically:

1. Add the same narrative + worked examples described in Disposition A.
2. State explicitly that **no new extension is required**: clinician-facing
   CPT/HCPCS codes are preserved as additional `Coding` entries on the same
   `CodeableConcept`, with `userSelected = true` set on the original
   clinician-supplied `Coding`. The X12-normalized `Coding` is just another
   entry without `userSelected`.
3. Cross-reference the FHIR core "Using Codes in resources" guidance and the
   existing translation extension
   (`http://hl7.org/fhir/StructureDefinition/coding-translation`) for
   implementers who want a stronger linkage between the two `Coding` entries.
4. Update the CRD and DTR companion guidance to mirror the same convention so
   the round-trip story is consistent across Da Vinci.

#### Justification

This still solves the underlying problem — implementers know how to preserve
the original clinician code and still meet X12 278 expectations — but it does
not introduce a new Da Vinci-specific extension to maintain, version, and
publish. `userSelected` is already part of the `Coding` data type and is the
designated FHIR signal for "this is the code the human picked," which is
exactly the original-vs-normalized distinction the reporter wants. Avoiding a
new extension reduces the migration burden on existing PAS implementers,
keeps the IG closer to vanilla FHIR, and is easier to align with CRD and DTR
because all three IGs already use `CodeableConcept`. The trade-off vs A is
that the original/normalized distinction is conveyed by a boolean flag rather
than a dedicated extension, which is slightly weaker semantically but is
sufficient for every use case the ticket actually cites.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — already addressed by existing
mechanisms." Take the position that PAS implementers can already preserve the
original clinician code by populating multiple `Coding` entries on
`Claim.item.productOrService.coding`, that `Coding.userSelected` is the
established FHIR mechanism for marking the human-chosen code, and that the
X12 normalization behavior is X12 turf, not FHIR turf, so the IG should not
codify it. Decline to add new examples on the basis that the existing PAS
examples already exercise both CPT and HCPCS service-type codings.

#### Justification

A defensible decline rests on the fact that nothing in the ticket actually
identifies a missing FHIR mechanism — the data type already supports
multi-coding, and `userSelected` already tags the original. The WG could
reasonably argue that publishing prescriptive guidance about X12 278
normalization is out of scope for a FHIR IG and risks conflicting with future
X12 errata. Declining also avoids spending publication cycles on examples
when the underlying profiles do not change.

---

### Recommendation

**Recommended disposition:** B

The original 10-0-0 vote acknowledged the request is reasonable but
mistimed for STU2.2; the only real question for the post-STU2.2 cycle is how
heavy a hammer to bring. Disposition B delivers everything the reporter
explicitly asked for — round-trip guidance, an "original vs normalized"
strategy, and tightened CPT/HCPCS examples — without minting a Da
Vinci-specific extension that other IGs would then have to learn. Leaning on
the already-existing `Coding.userSelected` flag and the FHIR-core
`coding-translation` extension keeps the solution composable with CRD, DTR,
PCT, and other Da Vinci IGs that touch the same service-type codings.
Disposition A is a fine fallback if the WG decides during review that the
boolean `userSelected` signal is too implicit and wants a named extension;
Disposition C ignores the implementer pain the reporter and the prior 10-0-0
vote both acknowledged and is hard to defend given the ticket's
`Ready-For-Vote` label.
