## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir`, edit `source/financial-module.html` and replace the
single occurrence of the contraction `doesn't` (currently on line 250,
in the sentence "A simple Poll request, one which doesn't specify
additional .input parameters: origresponse, include, exclude, period or
count; would return …") with the fully spelled-out form `does not`,
yielding "A simple Poll request, one which does not specify additional
.input parameters …". No other content, anchors, links, or sibling
pages are affected. Mark the ticket Resolved as "Persuasive" once the
PR lands.

#### Justification

This is exactly what the ticket asks for and matches HL7's long-standing
editorial rule that normative-track specification pages spell out
abbreviations and avoid contractions. The fix is a one-token textual
change on a hand-authored HTML page (the briefing confirms `source/*.html`
pages are authored, not generated), it cannot break any link or anchor,
and it carries zero implementation impact for any FHIR consumer. Highest
priority is reasonable only insofar as editorial backlog cleanup is
cheap; treating the request literally is the lowest-risk path.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the ticket but bundle it with a single editorial
sweep of `source/financial-module.html` that expands every contraction
and unexpanded abbreviation on the page in one PR (e.g., `won't`,
`can't`, `it's`, `we'll`, undefined acronyms on first use, etc.), rather
than chasing each contraction with its own one-line ticket. The PR would
still cite FHIR-55198 in the commit/PR description, and the resolution
note would record that the broader sweep was performed.

#### Justification

The auto-imported "Comment 164" tickets are known to be a long tail of
single-string editorial nits. Resolving them one token at a time
generates churn (one PR, one review, one publisher rebuild per
contraction) for changes that are individually trivial and collectively
small. Doing a single page-scoped sweep keeps the diff reviewable, lets
Financial Mgmt verify the page reads consistently after the changes,
and reduces the per-ticket overhead. The trade-off is that the PR's
diff is no longer "exactly what the ticket asked for", which the
briefing's gotcha about not silently widening scope warns against —
hence the explicit resolution note documenting the broader sweep.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-55198 as "Not Persuasive" on the grounds that the contraction
`doesn't` appears in informal narrative prose ("A simple Poll request,
one which doesn't specify…"), is unambiguous, and does not impair
implementer understanding. No spec content, conformance rule, schema, or
example would change.

#### Justification

A defensible argument can be made that HL7's "no abbreviations" guidance
is aimed at undefined acronyms and domain shorthand (e.g., `EOB`, `PA`,
`DTR`) that genuinely confuse implementers, not at common English
contractions in surrounding prose, which every reader parses without
effort. Declining would also signal that Highest priority on a single
contraction is not a sustainable bar for editorial review. The downside
is that "doesn't" really is on the published list of forms HL7 editors
have historically flagged, so declining sets up an inconsistent
precedent across the spec.

---

### Recommendation

**Recommended disposition:** A

This is a one-token, zero-risk prose change on a hand-authored page in
`source/`. The briefing confirms there is exactly one occurrence to
change, no generated artifact is involved, and there are no
cross-references, Zulip discussion, or linked GitHub work suggesting
hidden complexity. Disposition B is tempting given the auto-import
backlog, but expanding scope without a separate ticket conflicts with
the briefing's "do not silently roll in" guidance and would muddy the
resolution record for FHIR-55198. Disposition C is unlikely to survive
review given that the ticket cites the project's own editorial rule.
Take the literal fix, resolve as Persuasive, and if Financial Mgmt
wants a broader editorial sweep of `financial-module.html`, file it as
a separate ticket so the audit trail stays clean.
