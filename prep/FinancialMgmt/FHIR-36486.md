## Summary

The reporter (Corey Spears) is asking the CARIN IG for Blue Button to stop using
the locally-defined `C4BBClaimDiagnosisType` CodeSystem, which was published as
an "example" code system, and instead bind its diagnosis-type ValueSets to a
properly-published terminology hosted in HL7 Terminology (THO). The companion
UTG ticket [UP-309](https://jira.hl7.org/browse/UP-309) (raised by the same
reporter) created `http://terminology.hl7.org/CodeSystem/diagnosistype` in THO,
so the THO-side artifact now exists. The same pattern is being addressed in
parallel by Da Vinci PCT under
[FHIR-35540](https://jira.hl7.org/browse/FHIR-35540) ("Defining URL should be
anchored in THO not hl7.org/fhir for CodeSystem-PCTDiagnosisType"), which has
been resolved Persuasive and Published — establishing precedent within Financial
Mgmt for moving these "example" diagnosis-type code systems into THO. The
remaining wrinkle, captured in the most recent comment, is that the local
`C4BBClaimDiagnosisType` CodeSystem also defines an `other` code that is not
present in THO's `diagnosistype` CodeSystem, and that `other` code is currently
used by two ValueSets and four published examples; the workgroup must decide
whether to drop `other` (breaking change), retain a slimmed local CodeSystem
that contributes only `other`, or request that `other` be added to THO.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the CARIN BB diagnosis-type binding off the local "example"
`C4BBClaimDiagnosisType` CodeSystem entirely, in favour of the now-published
THO CodeSystem `http://terminology.hl7.org/CodeSystem/diagnosistype`.
Concretely, in the CARIN IG for Blue Button source IG:

- **Remove** the local `CodeSystem/C4BBClaimDiagnosisType` artifact (it was an
  example CodeSystem and is now superseded by THO).
- **Rebind** the three affected ValueSets to compose THO codes only:
  - `ValueSet/C4BBClaimInpatientInstitutionalDiagnosisType` →
    `$DiagnosisType#principal`, `$DiagnosisType#externalcauseofinjury`,
    `$DiagnosisType#admitting` (drop `C4BBClaimDiagnosisType#other`; replace
    `$HL7DiagnosisType#*` references with the THO `diagnosistype` canonical).
  - `ValueSet/C4BBClaimOutpatientInstitutionalDiagnosisType` →
    `$DiagnosisType#principal`, `$DiagnosisType#externalcauseofinjury`,
    `$DiagnosisType#patientreasonforvisit` (drop `#other`).
  - `ValueSet/C4BBClaimProfessionalAndNonClinicianDiagnosisType` →
    `$DiagnosisType#principal`, `$DiagnosisType#secondary`.
- **Update the four examples** that currently use
  `C4BBClaimDiagnosisType#other` (`EOBInpatient2`, `EOBInpatient2-nonfinancial`,
  `EOBOutpatient2`, `EOBOutpatient2-nonfinancial`) to either omit the diagnosis
  type slice or use a different in-binding code (e.g.,
  `$DiagnosisType#externalcauseofinjury` where clinically appropriate).
- Note the change in the IG's "Differences from previous version" section as a
  **breaking change** for any implementer using
  `C4BBClaimDiagnosisType#other`.

This is option (1) from the 2025-12-22 comment, executed verbatim against the
artifact set the comment enumerates.

#### Justification

This is what the ticket (and the 2022 block-vote disposition) literally asks
for: stop publishing an example CodeSystem and move to a non-example, THO-hosted
CodeSystem. The precedent is already set in this same workgroup by
[FHIR-35540](https://jira.hl7.org/browse/FHIR-35540) for PCT, and the THO
artifact is now in place per [UP-309](https://jira.hl7.org/browse/UP-309).
Carrying a local "example" CodeSystem after a normative THO equivalent exists
is exactly the anti-pattern the FHIR Terminology Authority discourages — it
fragments terminology and complicates downstream harmonization with PCT and
other Financial Mgmt IGs. The cost is real (loss of `#other` is a breaking
change for any implementer relying on it) but bounded: only two ValueSets and
four examples reference `#other`, and the IG can mitigate by calling the
removal out clearly in the version-differences page and by leaving the
diagnosis-type slice optional so that consumers who have nothing better than
"other" can omit it.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt option (2) from the 2025-12-22 comment: **bind primarily to THO
`diagnosistype` but retain a slimmed-down local `C4BBClaimDiagnosisType`
CodeSystem containing only the `other` code**, then compose both into the
affected ValueSets. Concretely:

- **Retain** `CodeSystem/C4BBClaimDiagnosisType` but reduce it to a single
  concept: `other` (re-publish it explicitly as a non-example,
  `content = complete` CodeSystem with a stable canonical and copyright/scope
  text that makes the bridging role explicit).
- **Rebind** the affected ValueSets to compose THO `diagnosistype` plus
  `C4BBClaimDiagnosisType#other`:
  - `C4BBClaimInpatientInstitutionalDiagnosisType` → `$DiagnosisType#principal`,
    `$DiagnosisType#externalcauseofinjury`, `$DiagnosisType#admitting`,
    `C4BBClaimDiagnosisType#other`.
  - `C4BBClaimOutpatientInstitutionalDiagnosisType` →
    `$DiagnosisType#principal`, `$DiagnosisType#externalcauseofinjury`,
    `$DiagnosisType#patientreasonforvisit`,
    `C4BBClaimDiagnosisType#other`.
  - `C4BBClaimProfessionalAndNonClinicianDiagnosisType` →
    `$DiagnosisType#principal`, `$DiagnosisType#secondary` (no `#other`
    needed).
- **Leave the four `#other` examples unchanged**.
- **Open a follow-up UTG ticket** asking THO to add `other` to the
  `diagnosistype` CodeSystem; once accepted, schedule a future-version ticket
  to drop the local CodeSystem entirely.

#### Justification

This option directly addresses the gotcha called out in the 2025-12-22 comment:
the THO `diagnosistype` CodeSystem does not currently include `other`, and
unilaterally removing `other` from the CARIN BB ValueSets would be a breaking
change for implementers who today have nothing else to send. Keeping a tiny
local CodeSystem solely as a bridging artifact is a well-precedented pattern
in HL7 IGs (e.g., US Core's "supplemental" CodeSystems used to carry codes that
are pending UTG adoption). It preserves implementer behaviour, removes the
"example CodeSystem" anti-pattern (the trimmed local CodeSystem would now be a
properly-published, single-concept extension), and lays an explicit migration
path to a fully-THO binding once UTG accepts `other`. The downside is that a
local CodeSystem persists in the IG and a follow-up UTG ticket has to be
tracked; the workgroup has shown willingness to do exactly this kind of
two-step migration in the FHIR-35540 / UP-309 history.

---

### Disposition C: Decline

#### Proposal

Re-resolve the ticket as **Not Persuasive** on the grounds that the 2022 block
vote already resolved it as "Considered for Future Use" pending THO
publication, and that the only outstanding work is now scoped under the UTG
ticket [UP-309](https://jira.hl7.org/browse/UP-309) (and any follow-up UTG
work to add `other`). Close FHIR-36486 with a comment directing implementers
and editors to the UTG-side migration and a future CARIN BB version-planning
ticket.

#### Justification

This is the weakest of the three options but is defensible only as a process
disposition: the ticket was already dispositioned, the substantive THO work
has shifted to UTG, and re-opening the ticket without committing to either
breaking-change removal or carrying a bridging CodeSystem leaves CARIN BB in
exactly the state it was in. Declining keeps STU2 stable and avoids any
breaking change in the current publication cycle. It does, however, leave the
"example CodeSystem" anti-pattern in the published IG indefinitely and ignores
the fact that the THO artifact now exists, so it should only be picked if the
workgroup is unwilling to make any change at this point in the publication
cycle.

---

### Recommendation

**Recommended disposition:** B

The factual record makes Disposition C untenable: the THO CodeSystem is now
published, [FHIR-35540](https://jira.hl7.org/browse/FHIR-35540) has set the
precedent in this same workgroup, and Lynn Laakso explicitly re-opened the
ticket post-STU2 to execute this migration — the ticket exists precisely
because the workgroup has decided not to leave things alone. Between A and B,
Disposition A is cleaner long-term but introduces a real breaking change
purely because THO does not yet carry `other`; that is an avoidable cost.
Disposition B preserves implementer behaviour, immediately removes the
"example CodeSystem" anti-pattern (the retained local CodeSystem becomes a
properly-published, single-code bridging artifact rather than an example), and
sequences the remaining work into UTG where it belongs. The follow-up UTG
ticket to add `other` to THO `diagnosistype` is small, has a clear precedent
in UP-309 itself, and can be tracked as a normal future-version cleanup. The
workgroup should adopt B for the next CARIN BB publication and open the
UTG-side ticket as part of the same disposition action.
