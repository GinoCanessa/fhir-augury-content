## Summary

The reporter (SudhansuMishra) raises an interoperability gap in the Da
Vinci Prior Authorization Support (PAS) flow when a Payer acts as the
single point of contact for downstream "delegated vendors" (entities
that the Payer has contracted to make some or all of the actual prior
authorization decisions). In their environment, delegated vendors today
submit X12 278 (specifically the X12N 278 X216 update transaction) back
to the Payer whenever a PA decision is rendered or additional
information is needed; the Payer then turns that X216 into the event
that is routed to its trading partners. With PAS moving to a
FHIR-based exchange (and the Subscription/notification work building on
top of it), the vendor would instead be sending the Payer a PAS
`ClaimResponse` Bundle. The reporter asserts that a PAS
`ClaimResponse` Bundle cannot be cleanly mapped back into an X216
transaction because it is missing several data elements the X216
requires, and asks the work group for a viable solution so that
delegated-vendor-to-Payer notifications can be implemented without
significant friction.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with this ticket in the indexed sources, so the
context above is drawn solely from the ticket description.

## Details

**Description:**

> As the Payer acts as the single point of contact and delegates
> transactions to its affiliated delegated vendors, the Payer must
> receive vendor-supplied information in a format that can be easily
> accommodated within the Payer's internal systems. Today, for example,
> our delegated vendors submit X12 278 (X216) transactions to our system
> for PA updates whether a PA decision is made or need more inforamtion,
> and from there the event is generated and routed to Trading Partners.
> However, a ClaimResponse Bundle cannot be converted directly into an
> X216 transaction because it lacks several required data elements.
> Because of this gap, we need a viable solution that enables smooth
> processing. Otherwise, we anticipate significant challenges in
> implementing the Subscription process through delegated vendors with
> Payer be the POC.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a gap analysis against the PAS `ClaimResponse` Bundle and
extend the PAS IG (HL7/davinci-pas) so that every X12 278 (X216)
required data element has a corresponding, addressable location in the
PAS `ClaimResponse` Bundle. Concretely:

1. Compile the authoritative list of X12N 278 X216 required loops /
   segments / elements (control numbers, payer/provider/subscriber
   identifiers, certification / review identification, HCR review
   action, request categories, decision reason codes, dates, AAA error
   information, etc.).
2. For each required X216 element that is not currently representable
   on the PAS `ClaimResponse` (or accompanying Bundle entries such as
   `Patient`, `Coverage`, `Organization`, `Practitioner`,
   `Communication`, `CommunicationRequest`), add an extension on the
   PAS `ClaimResponse` profile (or the relevant supporting profile)
   and bind code values to the existing X12 / Da Vinci value sets
   already used in PAS.
3. Publish a normative X216 ↔ PAS `ClaimResponse` Bundle mapping table
   in the IG (mirroring the existing 278 ↔ `Claim` mapping that PAS
   already provides for the request side) so that delegated-vendor
   implementations have a single authoritative crosswalk.
4. Reference the new mapping from the Da Vinci PAS Subscription
   notification guidance so that a Payer receiving a
   notification/ClaimResponse from a delegated vendor can deterministically
   produce the X216 it forwards to trading partners.

#### Justification

PAS already exists specifically to be a FHIR-on-the-wire equivalent of
the X12 278; an explicit goal of the IG is round-trippability with
X12. If `ClaimResponse` cannot reliably round-trip to X216 in the
update direction, that is a defect in the IG's primary value
proposition rather than an implementer-local concern. Closing the gap
through extensions plus a published mapping is consistent with how the
request-side 278 mapping is already handled in PAS, and it keeps the
delegated-vendor model the reporter describes (Payer as POC) viable
without forcing every Payer to invent its own private extensions.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than enlarging the PAS `ClaimResponse` profile itself, address
the delegated-vendor scenario as a companion mechanism:

1. Define a small **PAS Delegated-Vendor Notification** profile (or a
   set of extensions) that wraps a PAS `ClaimResponse` Bundle with the
   additional administrative envelope data the X216 needs but that is
   not clinically/decision-relevant (e.g., trading-partner control
   numbers, transaction set identifiers, sender/receiver ISA/GS
   identifiers, batch context).
2. Keep the PAS `ClaimResponse` Bundle itself unchanged so that
   non-delegated PAS exchanges are not burdened with X12-envelope
   concerns.
3. Document the delegated-vendor → Payer → trading-partner flow as a
   distinct use case in the PAS IG, with the new wrapper profile as
   the contract, and reference it from the Subscriptions / notification
   guidance.
4. Provide guidance (and ideally a reference mapping) showing how the
   Payer assembles the X216 from the wrapper plus the underlying
   `ClaimResponse` Bundle.

#### Justification

The reporter's description suggests the missing data elements are
largely envelope/administrative (needed so the Payer can re-emit an
X216 to *its* trading partners), not clinical decision content that
belongs on `ClaimResponse` itself. Layering a delegated-vendor
notification profile on top keeps the core PAS exchange focused on the
PA decision and avoids loading every PAS implementation with
X12-envelope extensions that only matter when a delegated-vendor model
is in play. It also constrains the change to a clearly scoped,
optional profile, which is easier to ballot than broad changes to
`ClaimResponse`.

---

### Disposition C: Decline

#### Proposal

Decline the change request with the rationale that the gap described
is implementer-specific business integration rather than a defect in
the PAS IG. Direct the reporter to:

- Confirm exactly which X216 elements are missing (with segment-level
  detail) and re-open if any of them are decision-content elements
  that genuinely cannot be represented in a `ClaimResponse` Bundle.
- Handle Payer-internal X12 envelope construction (control numbers,
  trading-partner routing, etc.) inside the Payer's own
  integration/translation layer, since those values are not part of
  the PA decision the delegated vendor is communicating.
- Use the existing FHIR Subscription R5 backport mechanisms in PAS as
  defined, without IG-level changes.

#### Justification

The ticket as filed does not enumerate the specific missing X216
elements, and many X12 envelope/control fields (ISA/GS/ST control
numbers, trading-partner identifiers, batch metadata) are intentionally
*not* modeled in PAS because they are properties of the
Payer-to-trading-partner exchange rather than of the prior
authorization itself. Adding them to `ClaimResponse` (or to a PAS
profile) risks spec creep into transport/envelope concerns and could
set a precedent that every implementer's downstream X12 obligations
must be reflected in the FHIR payload. Without a concrete element list
and broader implementer demand, the IG should not absorb this
complexity.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible compromise. The reporter has
identified a real implementer pain point — a Payer using delegated
vendors needs a deterministic way to reconstruct an X216 from what the
vendor sends — and PAS's stated goal of FHIR/X12 interoperability
makes "your IG broke our X12 pipeline" a legitimate concern, so a flat
decline (C) is too dismissive without first seeing the missing-element
list. At the same time, accepting the change as filed (A) would extend
the core `ClaimResponse` profile with envelope/administrative data
that most PAS implementations do not need and that arguably belongs to
the Payer-to-trading-partner exchange rather than to the PA decision
itself. A scoped delegated-vendor notification profile keeps the core
PAS contract clean, gives Payers and delegated vendors a normative
target to build against, and aligns naturally with the Subscription
notification flow the reporter is trying to enable. The work group
should ask the reporter for the explicit X216 element list as part of
accepting B so the new profile is grounded in concrete requirements.
