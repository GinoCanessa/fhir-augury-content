## Summary

The reporter (Hans Buitendijk) observes that the Da Vinci Coverage Requirements Discovery (CRD) IG, version 2.2.1, pins references to FHIR US Core at the R7 release across multiple profiles. In practice, deployed implementations are not aligned to US Core R7: ONC certification is currently anchored on US Core R6, and US Core R8 is in SVAP — leaving R7 as a version many implementers neither use nor are required to support. The ticket asks that the CRD IG (and, by extension, DTR and PAS, which share the same pattern) make it explicit that an implementation only needs to support the US Core version that aligns to ONC's Certification Criteria or SVAP, rather than the specific version literally cited in the profile references. The reporter suggests the [Conformance Expectations page](https://hl7.org/fhir/us/davinci-crd/2.2.1/en/conformance.html#us-core) as the natural location for the clarification, and in the comment offers the [FHIR Artifacts page](https://hl7.org/fhir/us/davinci-crd/2.2.1/en/allartifacts.html) as an alternative or additional location. The author flags the change as potentially a technical correction but raises it as a Change Request to allow workgroup discussion. No cross-referenced Jira tickets, Zulip threads, or GitHub items were found in the augury index for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative clarifying statement to the CRD IG's [Conformance Expectations → US Core](https://hl7.org/fhir/us/davinci-crd/2.2.1/en/conformance.html#us-core) section stating that, while the profile references in this IG name a specific US Core release (currently R7), implementers SHALL be considered conformant if they support the US Core version that aligns to either (a) the version named in the current ONC Certification Criteria, or (b) any version currently in SVAP. Update the same wording on the [FHIR Artifacts](https://hl7.org/fhir/us/davinci-crd/2.2.1/en/allartifacts.html) page (or cross-link from there to Conformance Expectations). Mirror the change in the corresponding pages of the Da Vinci DTR and Da Vinci PAS IGs so all three IGs convey the same expectation. Treat as a technical correction if the workgroup agrees no normative profile changes are needed.

#### Justification

The ticket is fundamentally about aligning IG conformance language with the regulatory reality faced by US implementers: ONC certification and SVAP set the de-facto US Core version, and pinning the CRD/DTR/PAS profiles to a single version that is "between" the certified and SVAP versions creates a conformance gap with no implementer benefit. The clarification is non-breaking — it only widens the set of conformant configurations — and keeps the CRD/DTR/PAS family aligned with each other. Documenting it on Conformance Expectations is the conventional location for cross-IG version-compatibility statements in HL7 US Realm IGs.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than treating the version reference as flexible in prose, restructure the US Core dependency itself: replace the hard `us-core` profile references in CRD (and, in lock-step, in DTR and PAS) with a "US Core version family" abstraction — either (i) a CRD-defined wrapper profile that derives from the R7 US Core profile but documents that derivations from R6 / R8 of the same logical profile are also acceptable, or (ii) a concise dependency-and-conformance section that enumerates the supported US Core releases (R6, R7, R8) and lists, per profile, the minimum element expectations CRD relies on. This makes the version flexibility machine-checkable to the extent possible and avoids ambiguity for testers.

#### Justification

Disposition A solves the documentation gap but leaves implementations and testing tools looking at profile references that still name R7 only. Implementers who lean on automated conformance tooling (IG publisher validation, Inferno, etc.) may still see warnings or failures even when their implementation is "conformant" by the prose. Defining the dependency once, in a structured form, gives downstream tools and testers a deterministic set of acceptable US Core versions and avoids per-profile drift between CRD, DTR, and PAS. The trade-off is that it is a heavier change — potentially impacting profile differential URLs and validation behaviour — and may not land as a technical correction.

---

### Disposition C: Decline

#### Proposal

Decline the request on the basis that US Core version selection is the implementer's responsibility per the existing US Core dependency-and-version guidance in the FHIR ecosystem, and that CRD profiles intentionally pin to a specific tested US Core release for testability and ballot integrity. Direct the reporter to the existing US Core "Must Support and Missing Data" / version compatibility guidance, and close the ticket as Not Persuasive.

#### Justification

There is a defensible argument that pinning IG profile dependencies to a single, tested US Core release is what makes CRD's conformance claim meaningful: an "any of R6/R7/R8 is fine" policy effectively shifts the conformance burden from the IG to each implementer pair, and may mask real interop gaps where CRD relies on elements introduced or constrained differently across US Core releases. If the workgroup is not prepared to enumerate exactly which CRD-relevant elements are stable across R6/R7/R8, opening the door without that enumeration risks producing implementations that claim CRD conformance but cannot interoperate over Patient/Coverage/Encounter the way the IG intends.

---

### Recommendation

**Recommended disposition:** A

The reporter is identifying a real and immediate misalignment between deployed US Core (R6 certified, R8 in SVAP) and the version named in CRD 2.2.1 (R7), and the same problem exists in DTR and PAS. Disposition A is the lowest-risk, highest-value response: it codifies what implementers are already doing in practice, is non-breaking, and can be applied consistently across the three Da Vinci IGs that share the pattern. Disposition B is conceptually cleaner but is a much larger structural change, almost certainly not a technical correction, and risks delaying a fix that workgroup-aligned implementers need now. Disposition C is the weakest option — it preserves a version pin that no implementer is actually following, and would leave CRD/DTR/PAS chronically out of step with ONC's certification cycle. The workgroup should adopt A as a technical correction (mirroring the wording into DTR and PAS) and, separately, consider B as a future ballot item if it wants a structured, tooling-checkable version dependency.
