## Summary

The ticket flags an editorial / accuracy bug in the US Da Vinci PDex IG.
Section **6.4.4 "Da Vinci Data Export Payload"** lives inside the
**Payer-to-Payer Bulk Exchange** chapter of the IG, but its narrative
was lifted from the Provider Access API description — it talks about
"in-network providers ... retrieving information about ... patients
attributed to them via an existing or impending treatment relationship",
which is the Provider Access use case, not the Payer-to-Payer use case.
The reporter asks that the section be rewritten to describe the
**payer-to-payer** workflow (a member joining a new payer, the new
payer retrieving the member's prior data from the previous payer)
rather than the provider-access workflow.

The ticket has already been triaged with resolution **Persuasive with
Modification**; the recorded resolution text is "Section updated to
correctly refer to Payer-to-Payer data exchange." No related Jira
tickets, Zulip threads, or upstream/downstream tickets were surfaced
by the cross-reference index — the only cross-references are
mentions of `FHIR-50183` in the IG's own change-history files
(`input/pagecontent/changehistory.md`, `output/changehistory.html`,
`output/changehistory.md`, `template/changehistory.html`) in
`HL7/davinci-epdx`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rewrite section **6.4.4 "Da Vinci Data Export Payload"** in
`HL7/davinci-epdx` at
`input/pagecontent/payertopayerbulkexchange.md` so its descriptive
paragraph reflects the **Payer-to-Payer** workflow rather than the
Provider Access workflow. Concretely, replace the existing
"Provider Access is meant to enable in-network providers to retrieve
information ... about patients attributed to them via an existing,
or impending treatment relationship" sentence with text along the
lines of: "The Payer-to-Payer data export operation is meant to
retrieve the information requested by a member when they join a new
payer. The new payer retrieves the member's prior coverage data
from the member's previous payer. Under the requirements of the
CMS Prior Authorization Rule (CMS-0057) the data available through
the API **SHOULD** include:" — followed by the existing data-payload
list. Add a one-line entry to `input/pagecontent/changehistory.md`
referencing **FHIR-50183**: "Section 6.4.4: corrected narrative to
describe the Payer-to-Payer Data Export Payload (was incorrectly
copied from Provider Access)." Do not edit the generated mirrors
under `output/` or `template/changehistory.html` — they will
regenerate. Target the active pre-publish branch per the repo's
README.

#### Justification

The ticket is narrative-only and the reporter's diagnosis is
factually correct: the existing prose describes attribution-driven
provider retrieval, which belongs to Provider Access (chapter 6.3 of
PDex), not to Payer-to-Payer Bulk Exchange (chapter 6.4). The
corrected text aligns the section heading, the chapter context, and
the linked CMS-0057 requirement (which calls out *both* Provider
Access **and** Payer-to-Payer obligations, but with distinct
trigger semantics). The change is local to a single Markdown file
in `input/pagecontent/`, follows the briefing's "Add narrative"
recipe exactly, touches no profiles/extensions/value sets, and has
no cross-repo or version-registration impact. It also matches the
workgroup's recorded resolution ("Section updated to correctly
refer to Payer-to-Payer data exchange"); the corrected wording is
already present in the cached clone (`HL7/davinci-epdx@8bc29a47`,
line 191 of `payertopayerbulkexchange.md`), so this disposition is
effectively confirming and ratifying work already in flight.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same factual correction as Disposition A, but additionally
**factor the shared "CMS-0057 data payload SHOULD include" list**
out of `payertopayerbulkexchange.md` and the corresponding Provider
Access narrative into a single included snippet under
`input/includes/` (e.g., `_cms0057-data-payload.md`), and reference
it from both the Provider Access page and the Payer-to-Payer page
via the IG-publisher `{% include %}` mechanism. Each consumer page
keeps its own one-paragraph workflow lead-in (provider-access
attribution wording on the Provider Access page; member-driven
payer-to-payer wording on the Payer-to-Payer page), while the
data-payload bullet list is authored in a single place. Record the
restructure in `changehistory.md` against FHIR-50183 with a note
that the original narrative bug was a literal copy-paste of the
Provider Access section, and that the include-based approach
removes the recurrence risk.

#### Justification

The root cause of the original bug is duplication: the
data-payload list is the same in both flows (CMS-0057 says "make
the same dataset available"), so the Provider Access narrative was
copy-pasted in full and the surrounding workflow paragraph was
never adapted. A pure text-edit (Disposition A) fixes the symptom;
extracting the shared list eliminates the failure mode. The
briefing confirms `input/includes/` is already part of the
authoring layout, so this fits the repo's existing pattern without
introducing new infrastructure. The trade-off is moderate scope
creep — touching at least two pagecontent files plus a new include
— and a small risk of diff-noise during ballot review. It is
also harder to attribute purely to FHIR-50183 if subsequent
ballot comments target one of the touched flows.

---

### Disposition C: Decline

#### Proposal

Close FHIR-50183 as "Not Persuasive" on the grounds that the
current PDex 2.2.0 narrative already states the issue is editorial
and bounded to one paragraph, and that the surrounding chapter
title ("Payer-to-Payer Bulk Exchange") plus the section heading
("Da Vinci Data Export Payload") already convey the correct
context to a reader; the misleading paragraph would be left in
place until an unrelated narrative refresh sweeps it.

#### Justification

This is the defensible-but-weak option. It avoids any IG churn
inside the current ballot window and treats the section's
prose as cosmetic relative to the normative profiles, operations,
and capability statements (which are correctly Payer-to-Payer
throughout). The case against this disposition is strong, however:
the offending paragraph directly contradicts the chapter it sits
in (it tells the reader the API is for in-network providers
retrieving attributed-patient data, which is the *opposite* trigger
model from Payer-to-Payer), and CMS-0057 conformance reviewers are
specifically reading this section to confirm payer-to-payer scope.
Decline is included for completeness only.

---

### Recommendation

**Recommended disposition:** A

This is an unambiguous editorial bug in a single Markdown file with
no normative or cross-repo impact. The reporter's diagnosis is
correct, the workgroup has already resolved it as "Persuasive with
Modification" with matching resolution text, and the cached PDex
clone (`HL7/davinci-epdx@8bc29a47`) shows the corrected wording is
already in place at
`input/pagecontent/payertopayerbulkexchange.md` line 191.
Disposition A simply ratifies that change and adds the
`changehistory.md` entry referencing FHIR-50183, which is what the
existing change-history cross-references suggest has either been
done or is being prepared. Disposition B is attractive on
maintenance grounds but expands scope beyond what the ticket asked
for and risks ballot-review noise; it would be a reasonable
follow-up issue rather than the response to this ticket.
Disposition C is not defensible — the misleading paragraph is
materially misleading to CMS-0057 conformance reviewers, who are
exactly the audience for this chapter.
