## Summary

The reporter (Dan Murphy, Meditech) asks that the CRD `DeviceRequest`
profile be relaxed so that an `Organization` reference is permitted in
`DeviceRequest.performer`, mirroring the change made by
[FHIR-52357](https://jira.hl7.org/browse/FHIR-52357) for the CRD
`ServiceRequest` profile. Today the CRD `DeviceRequest` profile
constrains `performer` to a narrower set of references that excludes
plain `Organization`, while the parallel CRD `ServiceRequest` profile
already allows an HRex `Organization` for its `performer` element. This
inconsistency was first surfaced on the `Da Vinci CRD > DeviceRequest
performer` Zulip thread, where Lloyd McKenzie agreed he could not
recall a reason for the constraint and explicitly invited a change
request, which became this ticket. Resolving FHIR-56349 effectively
extends the FHIR-52357 fix from `ServiceRequest.performer` to the
matching `DeviceRequest.performer` slot in the same IG.

## Linked Jira Tickets

### [FHIR-52357: CRD Should Allow Organization for Provider Fields](https://jira.hl7.org/browse/FHIR-52357)

The reporter explicitly cites this ticket as the precedent. FHIR-52357
asked CRD's `ServiceRequest` profile to permit a standalone
`Organization` for both `requester` and `performer`, arguing parity
with PAS (which already allows `Organization` for the requesting and
performing provider). It was resolved **Persuasive with Modification**
(Nov 20 BR call) — the requester remained Practitioner-only, but
support for `Organization` was added to `ServiceRequest.performer` as
**mustSupport** (committed at master `e03207c`). FHIR-56349 is the
direct DeviceRequest analogue of that performer change.

Note: A Dec 10 2025 comment from Lloyd records that "Gail asked for
this to be pulled" because **OrganizationAffiliation** is needed
alongside `Organization` for facility/group providers (the
Organization⇄OrganizationAffiliation relationship parallels
Practitioner⇄PractitionerRole). Whatever resolution is reached on that
follow-up for ServiceRequest will likely need to apply identically to
DeviceRequest under FHIR-56349.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CRD `DeviceRequest` profile
(`StructureDefinition/profile-devicerequest`, served at
`https://hl7.org/fhir/us/davinci-crd/StructureDefinition/profile-devicerequest`)
so that `DeviceRequest.performer` permits a target of `Organization` in
addition to its current allowed types. Concretely, add `Organization`
(or, to match the FHIR-52357 pattern, the HRex `Organization` profile
`https://hl7.org/fhir/us/davinci-hrex/StructureDefinition/hrex-organization`)
to the `targetProfile` list on `DeviceRequest.performer`, and mark the
new target as **mustSupport** to mirror exactly what FHIR-52357 did for
`ServiceRequest.performer` at master `e03207c`. No change to
`requester`. Update the narrative on the profile page and any
DeviceRequest examples that previously could not show an
Organization-only performer.

#### Justification

This is the smallest, most consistent change: it brings
`DeviceRequest.performer` into alignment with the already-published
treatment of `ServiceRequest.performer` in the same IG (FHIR-52357,
Persuasive with Modification, approved by Burden Reduction on
2025-11-20), and it directly answers the question posed on Zulip,
where Lloyd McKenzie explicitly agreed there is no recalled rationale
for the current constraint and invited a change request. It also
preserves cross-IG parity with PAS, which already allows a standalone
Organization for performing providers — exactly the alignment argument
that carried FHIR-52357.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the change *and* extend it to **OrganizationAffiliation** in the
same pass — i.e., add both `Organization` and `OrganizationAffiliation`
(per the HRex profiles) as `targetProfile` entries on
`DeviceRequest.performer`, both flagged mustSupport. At the same time,
revisit `ServiceRequest.performer` so the two profiles stay in lock
step with whatever resolution is reached on the FHIR-52357 "pulled"
follow-up that Gail raised on 2025-12-10. If the workgroup ultimately
decides Organization-only is insufficient and OrganizationAffiliation
is required, update both profiles together rather than landing
DeviceRequest with the now-known-incomplete Organization-only shape.

#### Justification

Lloyd's 2025-12-10 comment on FHIR-52357 records that Gail asked the
ServiceRequest change be pulled because "a facility or group provider
record is incomplete without OrganizationAffiliation." That concern is
a property of the *performer* concept in CRD, not of ServiceRequest
specifically, so it will recur on DeviceRequest.performer the moment
this ticket is implemented as drafted in Disposition A. Bundling the
fix avoids shipping FHIR-56349 only to immediately re-ballot it under
the same critique, and keeps the ServiceRequest and DeviceRequest
performer slots structurally identical — which was the original
motivation for the cross-reference to FHIR-52357.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-56349 on the grounds that performer-as-Organization for
device orders is **already addressable via PractitionerRole/HRex**:
implementers can convey the performing organization through the
existing target types (e.g., a PractitionerRole that points to the
Organization, or by using `DeviceRequest.subject`/related contained
context), and adding a standalone Organization target risks the same
"muddies the waters" concern raised on FHIR-52357 (JIRAUSER20510,
2025-09-17: "service request.performer ... will muddy the waters and
add variety to how implementers create service requests").

#### Justification

There is a defensible argument that performer should remain a
*person* (or person-in-role) so that payer adjudication has an
identifiable ordering/performing clinician — the position Lloyd
McKenzie repeatedly took on FHIR-52357 ("Knowing exactly who signed
the order is a requirement for any legal order"). Declining keeps the
DeviceRequest profile narrower and avoids implementer divergence.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already adjudicated this exact question for the
sibling element on the same IG: FHIR-52357 was approved Persuasive
with Modification and added Organization (mustSupport) to
`ServiceRequest.performer`. The asymmetry between
`ServiceRequest.performer` and `DeviceRequest.performer` is what
prompted the Zulip thread in the first place, and the ticket reporter
is asking only for parity — not for new semantics. Lloyd McKenzie's
Zulip reply ("I don't remember and can't think of a particularly good
reason. Feel free to raise a change request.") is effectively a
pre-disposition in favor of A, and Disposition C would re-litigate
arguments the workgroup already weighed on FHIR-52357.

Disposition B is tempting because of Gail's 2025-12-10 pull request on
FHIR-52357 around OrganizationAffiliation, but that conversation is
still open on the ServiceRequest side; coupling FHIR-56349 to it
risks blocking a clean parity fix on an unresolved upstream debate.
The cleaner path is to land A now (so the two profiles stop diverging)
and let whatever OrganizationAffiliation resolution emerges from the
FHIR-52357 follow-up flow into both profiles together as a separate,
explicit change.
