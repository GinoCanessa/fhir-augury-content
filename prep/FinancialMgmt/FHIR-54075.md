## Summary

The reporter (ZackStuebs) asks how closely the value sets used by
`CoverageEligibilityResponse` should follow the corresponding code lists
in the X12 270/271 (Eligibility, Coverage or Benefit Inquiry/Response)
transactions. Their analysis shows that several FHIR-bound value sets
are materially smaller than the X12 271 lists they ostensibly cover — for
example the FHIR value set for `insurance.item.benefit.type` carries 12
codes versus 35 in EB01, `insurance.item.category` carries 28 codes
versus 100+ in EB03, and `insurance.item.term` carries 3 versus 19 in
EB06. EB04 is flagged as having no clear FHIR equivalent. The implicit
question is whether FHIR Core should expand or replace these value sets
to be 1:1 with X12, or whether the existing bindings are sufficient.

No Jira tickets are linked or cross-referenced to FHIR-54075. The most
substantively related historical discussion is a single Zulip post in
`#financial > CoverageEligibilityResponse loop category` (Jay Lyle,
2021), which raises an adjacent concern: that
`CoverageEligibilityResponse.insurance.item.category` looks like the
right place to carry an X12-style "kind of loop" indicator but is
documented and bound for "categories of service", which is confusing
when used to mirror EB01-style metamodel semantics. A separate ticket,
[FHIR-44875](https://jira.hl7.org/browse/FHIR-44875) ("Correct X12 278
Data Element Mappings"), is keyword-similar but addresses prior
authorization (X12 278) under US Da Vinci CDex and is therefore only
loosely related.

The ticket has already been triaged with a resolution of
**Not Persuasive**, on the grounds that the affected element bindings
(`insurance.item.benefit.type`, `insurance.item.unit`,
`insurance.item.term`) use example/preferred strength, so implementers
that need exact X12 alignment may substitute the X12 code systems
without a spec change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Bring the FHIR Core value sets bound to
`CoverageEligibilityResponse.insurance.item.benefit.type` (EB01),
`insurance.item.unit` (EB02), `insurance.item.category` (EB03), and
`insurance.item.term` (EB06) into 1:1 alignment with the corresponding
X12 271 code lists. Concretely: republish each value set with the full
X12 enumeration (35 codes for benefit.type, 9 for unit, 100+ for
category, 19 for term), and add a value set or extension to carry the
EB04 concept (insurance type, e.g. HMO/PPO/POS) which currently has no
clear `CoverageEligibilityResponse` home — likely on `insurance.item` or
on `Coverage` as an extension if the work group prefers to keep
`CoverageEligibilityResponse` minimal.

#### Justification

For US implementers whose upstream and downstream systems are X12-bound
(which is the regulated norm for eligibility today), exact code
parity removes a class of mapping defects and round-tripping losses
between 270/271 and `CoverageEligibility{Request,Response}`. Aligning
explicitly is consistent with the FHIR design principle of preferring
existing standard code systems where they exist, and with the precedent
set by X12-aligned bindings in the claims/adjudication area
(e.g. CARC/RARC work tracked in
[FHIR-26907](https://jira.hl7.org/browse/FHIR-26907)). It also addresses
the unresolved confusion raised in the 2021 Zulip thread above about
what "category" actually means on `insurance.item`.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the FHIR Core value sets as-is (small, illustrative) but formalize
the X12-alignment story out-of-band: (1) clarify on the
`CoverageEligibilityResponse` page and the affected element definitions
that the bindings on `insurance.item.benefit.type`,
`insurance.item.unit`, `insurance.item.category`, and
`insurance.item.term` are example/preferred and that implementations
needing X12 271 fidelity SHOULD bind to the corresponding X12 code
systems; (2) publish (or reference) a non-normative mapping table from
each FHIR element to its EB01–EB06 counterpart; (3) document EB04
specifically — either by recommending a standard extension on
`insurance.item` for "insurance type" (e.g. HMO/PPO) or by directing
implementers to carry it on the referenced `Coverage` resource. No
required value sets are widened in FHIR Core.

#### Justification

This preserves the resolver's stated rationale ("the codeSystems can be
replaced with X12 code systems if required") but turns it from tribal
knowledge into a documented, discoverable contract. It also responds to
the 2021 Zulip concern that `insurance.item.category`'s intent is
ambiguous: a clarifying note plus a referenced mapping is cheaper and
lower-risk than re-binding the element. Realistically the X12 lists
churn on a different cadence than FHIR Core, so keeping them outside the
normative value sets reduces version-coupling. EB04 is the one genuine
gap (no FHIR target identified); calling it out explicitly is more
useful than leaving it as "N/A?".

---

### Disposition C: Decline

#### Proposal

Maintain the current resolution of **Not Persuasive** and close the
ticket without further spec change. Rationale: the affected bindings
are example/preferred strength, so implementers that need X12 parity
can already substitute the X12 code systems locally (e.g. inside an
IG or profile) without any change to FHIR Core.

#### Justification

`CoverageEligibilityResponse` is intentionally a generalized,
international resource, not a literal projection of X12 271. Forcing
1:1 alignment in FHIR Core would (a) impose a US-centric, X12-versioned
code list on every realm, (b) couple FHIR Core releases to X12 code
maintenance, and (c) duplicate work that is already, or could be,
expressed in US-realm IGs (CARIN BB, Da Vinci PAS / CRD / DTR). The
existing example/preferred bindings explicitly authorize the
substitution being requested, which is what the resolver pointed at.
This is consistent with how FHIR has historically handled X12 alignment
in adjacent areas (e.g. allowing X12 CARC/RARC via implementer choice
rather than baking them into the base value sets).

---

### Recommendation

**Recommended disposition:** B (Alternative Approach).

Disposition C matches the current "Not Persuasive" resolution and is
defensible, but it leaves the substantive issue — that today a reader of
`CoverageEligibilityResponse` cannot tell, from the spec alone, that the
small built-in value sets are not meant to be exhaustive vs. X12 271 —
unaddressed. The 2021 Zulip thread shows this confusion is not
hypothetical. Disposition A is the most ambitious answer but
over-commits FHIR Core to tracking X12 code maintenance and pulls
US-mandated lists into an international resource, which is structurally
the wrong place for them. Disposition B preserves the existing binding
strengths (and therefore the resolver's reasoning), explicitly
documents that X12 substitution is intended, points implementers at a
mapping table for EB01/EB02/EB03/EB06, and specifically addresses the
EB04 gap that the reporter flagged. It is the minimum change that
genuinely answers the reporter's question and improves downstream
implementation without re-opening normative value sets, and it leaves
realm-specific X12 enforcement to US IGs where it belongs.
