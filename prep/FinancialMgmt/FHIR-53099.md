## Summary

FHIR-53099 is a follow-on change request to [FHIR-53057](https://jira.hl7.org/browse/FHIR-53057) ("Change codeSystems used in FM Resources to example/extensible to Preferred"). FHIR-53057 was resolved Persuasive with Modification and is now in the Applied state, and its resolution explicitly notes that the changes made to Claim and ClaimResponse "also apply to ExplanationOfBenefit." This ticket asks the workgroup to (a) propagate the FHIR-53057 binding-strength changes from Claim and ClaimResponse into the equivalent elements on ClaimResponse and ExplanationOfBenefit (so all three resources stay aligned), and (b) make one additional, previously-missed change in Claim itself: `Claim.payeeType` should have its binding strength changed to **Preferred**. Together the two tickets keep the Financial Management resource family internally consistent in how it binds to coded value sets for the R5+ ballot cycle.

## Proposed Dispositions

> Note: No GitHub cross-references were surfaced for this ticket, so no repo-analysis briefings were loaded per the `ticket-prep` skill. Implementation paths below reference the conventional FM-resource source locations in the HL7 FHIR core build but are **not** sourced from a current briefing; the workgroup should confirm exact paths against the HL7/fhir repository at apply time.

### Disposition A: Accept As Requested

#### Proposal

Apply two coordinated sets of changes to the FM resource definitions for the R5+ ballot:

1. **Claim** — change the binding strength of `Claim.payeeType` from its current strength to **`preferred`** (keeping the existing value set / code system reference unchanged unless the workgroup also flags a content issue).
2. **ClaimResponse and ExplanationOfBenefit** — propagate every binding-strength change that FHIR-53057 applied to Claim and ClaimResponse into the equivalent elements on ClaimResponse and ExplanationOfBenefit, so the three resources remain aligned. Concretely, for each element pair listed in FHIR-53057 (e.g., `Claim.priority` → `ClaimResponse`/`EOB` analogues; `Claim.careTeam.role` → EOB `careTeam.role`; `Claim.diagnosis.type`/`.onAdmission`, `Claim.procedure.type`, `Claim.supportingInfo.category`; `ClaimResponse.decision`, `.item.reviewOutcome.decision`, `.item.adjudication.category`, `.payment.type`, `.payment.adjustmentReason`), set the corresponding EOB element binding to `preferred` (and apply any code-system / value-set draft/experimental flag flips that FHIR-53057 already mandated).

Tag the resulting commits with the same `FM-R5+-Applied` label used on FHIR-53057 and note in the resolution that this ticket subsumes the EOB propagation referenced in the FHIR-53057 resolution text.

#### Justification

This is exactly what the ticket asks for and is consistent with the workgroup's stated intent on FHIR-53057: that the Claim/ClaimResponse changes "also apply to ExplanationOfBenefit." Keeping ClaimResponse and EOB binding-strengths in lockstep with Claim is a long-standing FM convention because the three resources share most coded slots and are routinely round-tripped in adjudication workflows; divergent binding strengths cause real implementer pain (validators flag conformance differently for the same logical concept). Adding `Claim.payeeType` to the Preferred set also matches the broader FHIR-53057 pattern of moving away from `example` / `extensible` toward `preferred` for FM coded elements, which loosens overly-prescriptive constraints while still nominating a recommended value set.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the EOB propagation from FHIR-53057 as part of this ticket, but **handle `Claim.payeeType` separately**: leave its binding strength unchanged for the R5+ ballot and instead open a follow-on ticket to (a) review the `payeeType` value set's content and code-system status (draft/experimental flags, code coverage for common payees such as subscriber, provider, party, etc.) and (b) only then move the binding to `preferred` once the value set is curated. The EOB-propagation portion is applied as in Disposition A.

#### Justification

FHIR-53057 generally pairs each "move to preferred" with either an implicit or explicit cleanup of the underlying code system / value set (e.g., "make codesystem/valueset not draft, preferred", "needs to be changed from ex-code to a proper code system then made preferred"). `Claim.payeeType`'s binding was added to FHIR-53099 after the fact, with no accompanying review of the value set's quality or status. Bumping a binding to `preferred` carries an implicit endorsement of the value set, so it is defensible to insist that the value set first meet the same bar (non-draft, well-scoped concepts) that FHIR-53057 imposed on the others. This avoids a situation where `payeeType` is `preferred`-bound to a value set the workgroup hasn't recently looked at.

---

### Disposition C: Decline

#### Proposal

Decline this ticket as **redundant with FHIR-53057**. The EOB-propagation requirement is already captured in FHIR-53057's resolution text ("The changes to Claim and ClaimResponse also apply to ExplanationOfBenefit."), and `Claim.payeeType` can be folded into FHIR-53057's apply work or addressed via a small clarifying amendment to FHIR-53057 rather than carrying it as a separate change request.

#### Justification

FHIR-53057 is already in the Applied state with the `FM-R5+-Applied` label, and its resolution explicitly extends the Claim/ClaimResponse changes to EOB. If the EOB changes have not yet actually been committed, the cleaner project-management move is to reopen / amend FHIR-53057 (which carries the full element-by-element change list and prior workgroup discussion) rather than maintain a parallel ticket whose scope is "the rest of FHIR-53057." A separate ticket increases the risk of drift between the two lists and of one being applied without the other.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest path for the workgroup. The EOB propagation is a known gap that FHIR-53057's resolution acknowledged but did not formally enumerate, and `Claim.payeeType` is a small, targeted addition that is fully consistent with the Preferred-binding pattern FHIR-53057 already established across the rest of the FM resource family. Disposition C (decline as redundant) is tempting because FHIR-53057 is "Applied," but in practice the workgroup needs a discrete, trackable ticket for the EOB work and the `payeeType` addition — otherwise those changes risk being lost or applied inconsistently. Disposition B's caution about the `payeeType` value set is reasonable but probably out of scope for a binding-strength change request: if the value set turns out to need cleanup, that is best handled as a separate, narrowly-scoped follow-on ticket rather than blocking the binding change, which is itself a *loosening* (Preferred is less restrictive than extensible/example-with-content-controls in practice for implementers).
