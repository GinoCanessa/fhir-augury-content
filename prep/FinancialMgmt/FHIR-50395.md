## Summary

Casey Trauer (reporter) flags what they describe as a minor inconsistency in
the Da Vinci PDex *Member Match Group* profile: the IG narrative directs
implementers to convey the "match" / "non-match" classification of a returned
Group via `Group.characteristic`, but the profile additionally constrains
`Group.code` (the mandatory element) to carry the same coding. The result is
that the same `PdexMultiMemberMatchResultCS` code (e.g., `match`) appears
twice in every returned Group — once at `Group.code` and once at
`Group.characteristic.code` — and it is unclear from the IG whether this
duplication is required, recommended, or accidental. The ticket asks the
work group to confirm intent and clarify whether implementers should populate
both locations.

The cross-reference index surfaces only the two FHIR element mentions
(`Group.code`, `Group.characteristic`); no related Jira tickets or GitHub
items are linked to FHIR-50395. A Zulip search did surface a March
Connectathon thread (CareEvolution server) where an example Member Match
Group is shown in the wild with the `match` coding present in both `code`
and `characteristic` — direct evidence that implementers are in fact
populating both locations as the profile currently demands.

The current Jira resolution is **Not Persuasive**, with the proposed
resolution text "The duplication is intentional. Characteristics are
searchable and therefore enable searching to be utilized to find relevant
groups." The ticket is labeled `Fit-For-Teleconference` and `PDex-Vote-1`,
indicating Financial Mgmt has triaged it and is bringing it forward for a
ballot-disposition vote.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the duplication of the match-result code at both `Group.code` and
`Group.characteristic.code` as an editorial defect in the
*PDex Member Match Group* profile and remove one of the two locations.
Concretely: drop the slice / fixed-coding obligation on `Group.code` from
the profile, leave `Group.code` as the base R4 `0..1` element, and require
the match-result coding (`PdexMultiMemberMatchResultCS`) only on the
`Group.characteristic` slice that already carries the matched
`Organization` reference, period, and exclude flag. Update the IG narrative
to describe a single canonical location for the match-result classification
and republish the example Group payloads accordingly.

#### Justification

This is the most literal reading of the ticket: the reporter views the
two-location coding as confusing and asks whether it is intentional. R4
`Group.code` is "Kind of Group members" — i.e., a description of what the
members *are* (people, animals, devices) — not a slot for an
implementation-defined classification of the Group instance itself. Using
`characteristic` for the match/non-match classification is a more idiomatic
use of the Group resource because each `characteristic` already pairs the
classifying code with the value (the matched Organization reference), the
period, and `exclude`. A single canonical location also eliminates the
"what if the two codings disagree?" failure mode that the current profile
tacitly creates.

---

### Disposition B: Alternative Approach (Keep the duplication, fix the docs)

#### Proposal

Keep the existing dual-coding profile constraint (the current Jira
resolution path) but treat the ticket as evidence that the IG documentation
needs to make the duplication explicit and justified. Specifically:

1. Add an IG-narrative paragraph adjacent to the *PDex Member Match Group*
   profile page that explicitly states the two locations are intentional,
   identifies `Group.characteristic` as the *searchable* surface (so
   clients can `?characteristic=...` against the FHIR search parameter)
   and `Group.code` as the *mandatory header* surface for clients that
   only inspect the Group envelope, and clarifies that the two SHALL carry
   the same `PdexMultiMemberMatchResultCS` code.
2. Add a constraint (FHIRPath invariant) to the profile asserting that
   `Group.code.coding` and the matching `Group.characteristic.code.coding`
   are equal for the `PdexMultiMemberMatchResultCS` system, so
   implementations cannot drift the two values apart.
3. Update the example instances and the Member Match operation
   walkthrough to call the duplication out the first time it appears.

#### Justification

This honours the work group's stated rationale ("characteristics are
searchable, therefore enable searching to be utilized to find relevant
groups") without leaving the original ticket's confusion unaddressed.
Removing `Group.code` (Disposition A) is technically a breaking change
for any payer that has already implemented against the published profile
and against connectathon traffic — and the Connectathon Zulip thread above
shows at least one server already returning the dual coding in production
test payloads. Adding the invariant closes the "what if they disagree?"
hole that motivates Disposition A while preserving wire-compatibility, and
the narrative change directly resolves the reporter's "is this intended?"
question.

---

### Disposition C: Decline (Not Persuasive — current Jira posture)

#### Proposal

Close the ticket as **Not Persuasive** with the existing resolution text:
"The duplication is intentional. Characteristics are searchable and
therefore enable searching to be utilized to find relevant groups." Make
no profile or narrative changes.

#### Justification

The reporter explicitly calls the issue "a minor thing" and frames it as
a clarification request rather than a defect. The work group has already
articulated a coherent design intent — `Group.code` is the mandatory,
inspectable header value; `Group.characteristic` carries the same code in
a form that participates in FHIR search — and existing implementations
(per the CareEvolution Connectathon thread) appear to be coping with the
duplication without any reported interoperability harm. Spending a
ballot-disposition slot on either a breaking profile change (A) or a
documentation/invariant tightening (B) may not be warranted at this stage
of the IG.

---

### Recommendation

**Recommended disposition:** B

The ticket is resolvable without a breaking profile change, but the
reporter's question is real and is *not* answered by the IG as it stands —
which is exactly the failure mode the `Fit-For-Teleconference` label is
meant to surface for vote. Closing the ticket as Not Persuasive
(Disposition C) leaves the next implementer to file the same ticket again.
Removing `Group.code` (Disposition A) is the cleaner long-term shape but
is a wire-breaking change against payloads already exchanged at
connectathon, which is hard to justify late in the PDex ballot cycle.
Disposition B keeps the wire format stable, locks down the
"what if the two codings disagree?" failure mode with an invariant, and
turns the reporter's confusion into a documented, defensible design
choice — which is the outcome the work group has effectively already
decided on, just with the IG updated to match.
