## Summary

FHIR-35529 is a Da Vinci PCT change request asking that the canonical
defining URL for a PCT ValueSet (the description and 2023-03-30 reopen
comment identify an `All-ICD-10-CM` value set drawing all codes from
`http://hl7.org/fhir/sid/icd-10-cm`) be anchored in
terminology.hl7.org (THO) rather than `hl7.org/fhir`, and that a UTG
ticket be opened to register the ValueSet in THO. The ticket title
adds the qualifying principle: "Only those with required binding to a
`code` data type should be anchored in hl7.fhir.org" — i.e., everything
else (including this ValueSet) belongs in THO. The ticket has now been
through three publication cycles: it was first dispositioned
"Not Persuasive with Modification" in 2022, reverted and reopened in
2023 (because the underlying CodeSystem is in THO but the ValueSet
itself is not), reopened again post-STU1, dispositioned "Considered for
Future Use" in the 2025-07-08 PCT Block Vote (Block-2025-Jul) with the
note that PCT will resolve this "across IGs continuing to add to THO" as
part of the larger FM/THO effort already underway, and most recently
re-opened on 2025-10-08 post-STU2 publication. There are no Jira
cross-references and no GitHub cross-references; the only surfaced
cross-reference is the PCT Block-2025-Jul block-vote announcement on
Zulip.

## Proposed Dispositions

> Note: Dispositions cannot be grounded in a specific authoring root or
> file path because no GitHub cross-references were surfaced for this
> ticket and therefore no `repo-analysis` briefing was loaded. The
> proposals below name the canonical-URL spaces and the conventional
> PCT IG / THO authoring locations implied by HL7's public terminology
> governance, and any concrete edits will need to be re-grounded
> against `HL7/davinci-pct` and `HL7/UTG` once those briefings are
> available.

### Disposition A: Accept As Requested

#### Proposal

Move the PCT ValueSet identified by this ticket — described in the
2023-03-30 reopen comment as `All-ICD-10-CM`, drawing all codes from
`http://hl7.org/fhir/sid/icd-10-cm` — out of the PCT IG and into THO.
Concretely:

1. File a UTG (HL7 terminology) ticket against the corresponding THO
   work area to define a new ValueSet (proposed name `All-ICD-10-CM`,
   proposed canonical e.g.
   `http://terminology.hl7.org/ValueSet/all-icd-10-cm`) whose
   `compose.include[0]` is `system =
   http://hl7.org/fhir/sid/icd-10-cm` with no concept filter (include
   all codes).
2. In the next PCT release, remove the IG-anchored ValueSet definition
   (canonical under `http://hl7.org/fhir/us/davinci-pct/...`) and
   replace any internal binding references with the new THO canonical
   URL.
3. Apply the same pattern to the sibling tickets in the Joan Harper
   cohort (FHIR-35532, 35533, 35534, 35535, 35536, 35537, 35540, 35545,
   35548, 35549, 35550) and the Ron G. Parker cohort (FHIR-45812,
   45815, 45816, 45817, 45818) so that PCT only retains
   IG-canonical ValueSets that have a required binding to a `code`
   data type, per the principle stated in this ticket's title.

#### Justification

This is exactly what the reporter and the 2023-03-30 reopen comment
ask for, and it matches HL7's stated terminology-governance principle
that ValueSets reusable beyond a single IG belong in THO. The
underlying CodeSystem (`http://hl7.org/fhir/sid/icd-10-cm`) is already
hosted by HL7 outside the IG, so the ValueSet wrapper has no
PCT-specific semantics to justify keeping it IG-local. The 2025-10-08
comment from lynn_laakso explicitly frames this as the direction of
travel for the whole FM/THO effort; doing it now (rather than
deferring) keeps PCT aligned with that effort.

---

### Disposition B: Alternative Approach

#### Proposal

Address the request as a coordinated **batch migration** rather than a
single-ticket fix:

1. Treat FHIR-35529 as the lead ticket for the Joan Harper "Defining
   URL should be anchored in THO" cohort (FHIR-35532, 35533, 35534,
   35535, 35536, 35537, 35540, 35545, 35548, 35549, 35550) and the Ron
   G. Parker "Consider anchoring the valuesets in THO" cohort
   (FHIR-45812, 45815, 45816, 45817, 45818).
2. File a single, scoped THO/UTG submission that defines all of the
   migrated ValueSets together, plus a published canonical-URL
   redirect map (`http://hl7.org/fhir/us/davinci-pct/ValueSet/<X>` →
   `http://terminology.hl7.org/ValueSet/<X>`).
3. In the next PCT major version, retire the IG-anchored ValueSets in
   one cut, keeping only those with required binding to a `code` data
   type IG-local (per the ticket's title rule).
4. Coordinate with the FM/THO effort already underway (referenced in
   the 2025-10-08 reopen comment) so this batch lands as part of that
   workstream rather than as one-off edits.

#### Justification

The block-vote and 2025-10-08 reopen note already frame this as a
cross-IG concern that "requires a greater level of review across
multiple IGs" and is being driven by a larger FM/THO effort. Migrating
one ValueSet at a time creates 16+ small THO submissions, splits the
canonical-URL redirect work across releases, and risks mixed states
where some PCT bindings point at THO while sibling bindings still
point at the IG. A coordinated batch contains the disruption to a
single PCT major-version transition and lets THO review the full set
once. The trade-off vs. Disposition A is a longer wall-clock to
resolution, in exchange for less churn and a defensible cross-IG
story.

---

### Disposition C: Decline

#### Proposal

Resolve as **Considered for Future Use** (the disposition the 2025-07
PCT Block Vote actually adopted before the post-STU2 reopen) with the
explicit rationale that the migration is being subsumed by the larger
FM/THO terminology effort already underway and should be tracked
there, not as an STU2-blocking PCT change. Leave the IG-anchored
canonical URL in place for STU2 and document the intent in the IG's
terminology section.

#### Justification

This was the workgroup's most recent positive disposition (2025-07-08,
9-0-2). The reopen on 2025-10-08 was procedural — to keep the issue
visible after STU2 publication — not a substantive rejection of the
deferral rationale. Changing canonical URLs of published ValueSets is
a known compatibility hazard for implementers (existing instances and
profiles still cite the old URL); doing it inside a maintenance window
of an already-published STU is more disruptive than coordinating it as
part of the cross-IG THO effort. Declining-as-deferred preserves the
governance principle in the title without creating a new published
canonical ahead of the broader plan.

---

### Recommendation

**Recommended disposition:** B

The ticket's underlying request (Disposition A) is correct on
principle: a generic `All-ICD-10-CM` ValueSet over an HL7-owned
CodeSystem clearly belongs in THO, not inside one IG. But the
operational reality on the ground — visible in both the 2025-07
block-vote disposition and the 2025-10-08 reopen note — is that this
is one of ~16 sibling PCT ValueSets in the same situation, and the
FM/THO effort is already organising a coordinated migration. Doing
FHIR-35529 in isolation (Disposition A) creates a one-off THO
submission and a one-off canonical-URL redirect, then forces the
workgroup to repeat that work 15 more times for the cohort. Pure
deferral (Disposition C) is what the workgroup just tried and what
was reopened. Disposition B explicitly anchors FHIR-35529 to the
already-underway FM/THO effort, commits to a single coordinated batch
migration of the whole cohort with a published redirect map at the
next PCT major version, and is the most defensible answer to the
implementer-impact concern that has driven the multiple
revert-and-reopen cycles. Concrete edits should be re-grounded against
`HL7/davinci-pct` and `HL7/UTG` once `repo-analysis` briefings for
those repos are available.
