## Summary

The submitter requests that the Da Vinci PAS Implementation Guide be updated
to mandate that payers include a human‑readable denial reason description in
the ClaimResponse returned for prior‑authorization (PAS) requests, in addition
to the X12 reason code currently carried in the `reviewAction.reasonCode`
element. The motivation is the CMS Interoperability and Prior Authorization
Final Rule (CMS‑0057‑F), which the submitter reads as obligating payers to
"clearly communicate" the specific reason for a denial. From an EHR/end‑user
perspective, the X12 code alone is not actionable: displays show only a generic
descriptor, X12 does not publish a normative value‑set, and each payer
maintains its own list, making consistent UI rendering difficult. The
requested change is for the IG to require that the descriptive denial reason
text (as appears in the payer's denial letter) accompany the response — most
naturally as a `ClaimResponse.processNote` populated from the X12 MSG segment.
No directly linked Jira tickets were surfaced by cross‑references; the request
originates from, and is being actively debated in, the Da Vinci PAS Zulip
thread referenced below.

## Details

**Description:**

> Creating this ticket based on Zulip conversation:
>
> 1. Currently, payers only send x12 code in reasonCode element without sending the Reason description.
> 2. As per CMS, it is mandatory that payers should communicate specific reason for denied prior authorization decisions.
>
> See: https://www.cms.gov/newsroom/fact-sheets/cms-interoperability-prior-authorization-final-rule-cms-0057-f
>
> 3. IG should mandate sending a denial reason description in the ClaimResponse to ensure clearer communication of denial outcomes.

Source Zulip thread: [Da Vinci PAS > PAS Claim Response -> Denial reason description](https://chat.fhir.org/#narrow/channel/208874-Da-Vinci-PAS/topic/PAS.20Claim.20Response.20-.3E.20Denial.20reason.20description).

**Resolution (already applied — "Not Persuasive with Modification"):**

> The specification only mandates what X12 currently mandates. If you want
> payers to be forced to send a denial reason, then you need to petition X12
> to make it mandatory. The best we can do is provide guidance in the
> specification that payers should provide the reason and it would go in the
> process note if it was included in the MSG segment in the X12 message.

## Related Zulip Discussions

### [Da Vinci PAS > PAS Claim Response -> Denial reason description](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PAS/topic/PAS%20Claim%20Response%20-%3E%20Denial%20reason%20description/near/583914396)

Indhuja Narasimhan opened the thread asking whether payers are expected to use
`processNote` to convey the X12 denial reason description (e.g., the text for
code "0F – Not Medically Necessary"), and pointed at the CMS‑0057‑F final rule
as a driver. Jean Duteau (PAS) responded that PAS imposes no rules beyond X12:
the reason code maps from X12 HCR03 to `reviewAction.reasonCode`, and any free
text in the X12 MSG segment maps to `ClaimResponse.processNote`, but X12 does
not require payers to send MSG, so PAS cannot mandate it either. Lloyd
McKenzie reinforced that PAS / the Da Vinci BR guides are not bound to CMS
regulations (they have international/non‑CMS uses), that any system exchanging
PAS already requires an X12 license and can therefore look up the code's
display locally, and that if CMS believes PAS must be tightened beyond X12 the
appropriate path is a change request — which produced this ticket. The
submitter pushed back that X12 has no normative value‑set and each payer
maintains its own list, so the X12 code alone does not give EHRs an actionable
human‑readable reason; the ask was refined to "mandate that payers send the
exact denial reason description (the same text that appears in the denial
letter to members/providers)." The thread reached a clear, but not unanimous,
conclusion: the PAS authors are willing to add **guidance** that payers
SHOULD populate `processNote` with the denial reason text when available, but
will not make it a SHALL because the underlying X12 transaction does not.
Jean also noted there is an existing Jira issue concerning denial letters
(unspecified key).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the Da Vinci PAS IG (US Da Vinci PAS, STU 2.x) so that the
ClaimResponse returned for a denied prior‑authorization SHALL carry a
human‑readable denial reason description in addition to the X12 reason code.
Concretely:

- In the PAS profile of `ClaimResponse`, add a `SHALL` constraint on
  `ClaimResponse.processNote` (or, alternatively, on the
  `reviewAction.reasonCode.text`) for cases where `outcome` is `denied` /
  `partial`: implementations must populate it with the textual denial reason
  that the payer issues to members and providers.
- Update the PAS X12↔FHIR mapping page to require population of the X12 MSG
  segment whenever HCR03 carries a denial code, and to map that MSG text into
  `processNote`.
- Update the narrative in the ClaimResponse profile page and the relevant
  "Sending a Response" section to call out the new SHALL with examples.

#### Justification

This is the literal request and aligns most directly with CMS‑0057‑F's intent
that payers communicate a specific reason for denied prior‑authorization
decisions in a way downstream EHR systems can surface to clinicians without an
out‑of‑band lookup. It also addresses the submitter's usability concern that
X12 codes alone, lacking a normative display, are not actionable in EHR UIs.

### Disposition B: Alternative Approach (RECOMMENDED — matches the applied resolution)

#### Proposal

Do not introduce a new SHALL in PAS, but add **normative guidance** to the
PAS IG stating that:

1. Payers SHOULD populate `ClaimResponse.processNote` with the denial reason
   description that appears in the denial letter sent to the member/provider,
   whenever a denial or partial‑denial outcome is returned.
2. When the X12 278 response includes an MSG segment in the HCR (HCR04+), that
   text SHALL be mapped to `ClaimResponse.processNote` (i.e., make the
   mapping itself explicit and required, even though sourcing the MSG remains
   optional in X12).
3. Add an explanatory paragraph on the ClaimResponse page (and/or the X12
   mapping page) clarifying that the X12 reason code conveys the categorical
   reason, while the `processNote` carries the payer‑specific human description,
   and that consumers needing display text for the X12 code should resolve it
   via their X12 license and the X12 Service Review Decision Reason Codes
   listing (https://x12.org/codes/service-review-decision-reason-codes).

#### Justification

This is exactly the path the PAS authors articulated on Zulip and recorded as
the ticket's applied resolution ("Not Persuasive with Modification"). It
respects the design constraint that PAS faithfully mirrors X12 — Lloyd and
Jean both noted that PAS cannot require payers to send data X12 itself does
not require — while still giving implementers a clear, IG‑level expectation
that the human‑readable denial reason belongs in `processNote`. It also
preserves the use of PAS outside of the CMS‑0057‑F regulatory perimeter
(international and non‑CMS deployments), which Lloyd flagged as a hard
constraint. The added "MSG → processNote" mapping requirement is a real,
verifiable improvement over the status quo without overreaching.

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive": the PAS IG is a faithful FHIR projection of the
X12 278 transaction set, and it cannot impose conformance obligations
(SHALLs) that the underlying X12 standard does not. CMS‑0057‑F binds payers
to communicate denial reasons; that is a regulatory obligation on the payer,
not a structural defect in PAS. Implementers wishing to enforce richer
reason‑text exchange should petition X12 to mandate the MSG segment, or use
companion guides and trading‑partner agreements.

#### Justification

The PAS profile cannot enforce content the source X12 transaction is silent
on without breaking the IG's fundamental "FHIR mirror of X12" contract.
Receivers already have X12 licenses and can resolve the X12 code's display
themselves. Adding a SHALL would make compliant payers non‑conformant on day
one. However, this disposition ignores the legitimate usability gap raised
by the submitter and is unlikely to satisfy a workgroup that has already
chosen a "with modification" outcome.

---

### Recommendation

**Recommended disposition:** B

Disposition B mirrors the path the PAS authors already converged on in the
Zulip thread and the applied Jira resolution, and it threads the needle
between the submitter's real usability concern and the hard architectural
constraint that PAS cannot require X12 content X12 itself does not. Pure
acceptance (A) would create a SHALL that today's compliant payers cannot
meet without a corresponding change at X12, putting the IG ahead of the
underlying transaction; pure decline (C) leaves implementers with the same
ambiguity that produced this ticket. B keeps the SHALL where the PAS IG can
legitimately enforce it (the X12→FHIR mapping: if MSG is present, it goes in
`processNote`), adds a clear SHOULD that the payer populate `processNote`
with the denial‑letter text, and points implementers at the X12 Service
Review Decision Reason Codes listing for display fallback. The workgroup
should also pick up Jean Duteau's Zulip mention of an existing "denial
letters" Jira issue and decide whether this ticket should be merged with or
cross‑linked to it before closing.
