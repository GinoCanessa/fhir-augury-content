## Summary

The reporter, Cooper Thompson, observes that the Da Vinci PDex
`bulk-member-match` operation accepts a list of `MemberBundle` inputs
(each pairing a `Patient` with a `Coverage`) but the response only
echoes the matched `Patient` and discards the `Coverage` context. When a
client submits two `MemberBundle`s for the same patient under different
coverages — e.g., the same Cooper Thompson under "Employer Plan 123" and
under "Some other plan 456" — the responder may match one bundle and
not the other, since coverage details are commonly used in matching.
Because the `Member Match Group` only carries the patient, the same
`Patient` reference ends up in both `MatchedMembers` and
`NonMatchedMembers`, with no way to tell the two bundles apart on
review.

The reporter notes this does not break downstream `bulk-export` (which
just consumes the `MatchedMembers` group), but it makes the operation's
output ambiguous when a human reviews it. The recommendation in the
ticket is to provide an option to return the original
(`Patient` + `Coverage`) pair in each of `MatchedMembers`,
`NonMatchedMembers`, and `ConsentConstrainedMembers`.

There are no cross-referenced Jira tickets — the only Jira
cross-reference is the ticket's self-reference. All other cross
references point at `HL7/davinci-epdx` source files where the change
has already been applied (label `Pre-Applied`, resolution
"Persuasive with Modification").

## Details

**Description:**

For the bulk-member-match operation, the input is a list of
MemberBundles. I'm going to ignore consent for this ticket, since it
isn't relevant, but the MemberBundle includes the pair of (patient +
coverage).

However, the response only includes the member. No details about the
matched coverage are included.

The problem arises when if you do a member match for the same member
and multiple coverages. For example:

- MemberBundle #1 (Patient: Cooper Thompson, Coverage: Employer Plan 123)
- MemberBundle #1 (Patient: Cooper Thompson, Coverage: Some other plan 456)

In this case, if coverage details are use as part of matching (which
they often are) the first member bundle might be a match, but the
second one might not be a match.

However, in the response, the Member Match Group only includes the
matched patients. This means that the "Cooper Thompson" patient would
show up in both MatchedMembers AND NonMatchedMembers.

Note: while this doesn't really impact the actual overall exchange
across the wire, since usually you'd just do a bulk export for the
MatchedMembers group, the duplicate patient with no coverage context is
confusing if the output of the member match operation is displayed to a
user for review, as that UI would just show the same member in both
groups. So this issue is about user confusion during review processes,
not about how the output of this operation impacts a subsequent bulk
export.

**Recommendation:** provide for an option for the (patient+coverage)
pair to be returned in the bulk member match response sections
(MatchedMembers, NonMatchedMembers, and ConsentConstrainedMembers).

**Resolution Description (already applied per `Pre-Applied` label):**

1. **`input/fsh/Extensions.fsh`** — new `MatchCoverage` extension
   (`base-ext-match-coverage`), `Reference(Coverage)`, cardinality
   0..1, with context `Group.member.entity`.
2. **`input/fsh/ProfilePDexMemberMatchGroup.fsh`** —
   `matchedCoverage` (0..1 MS) added on `PDexMemberMatchGroup`;
   `nonMatchedCoverage` (0..1 MS) added on `PDexMemberNoMatchGroup`;
   both `contained` comments updated to note Coverage MAY be present.
3. **`input/fsh/ExamplePDexMemberMatchGroup.fsh`** —
   `input-coverage-1` inline Coverage added; `member[0]` now shows
   both `matchedMember` (Patient) and `matchedCoverage` (Coverage)
   extensions.
4. **`input/pagecontent/payertopayerbulkexchange.md`** — §pdex-139d:
   when the same patient is submitted with multiple coverages, the
   responder **SHALL** include the Coverage and populate the
   extension.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change exactly as drafted in the ticket's resolution and
already committed in `HL7/davinci-epdx` (commit
`afe3888c`):

- In `input/fsh/Extensions.fsh`, define a new PDex-namespaced
  extension `MatchCoverage` (`Id: base-ext-match-coverage`,
  `Reference(Coverage)`, cardinality 0..1, `Context:
  Group.member.entity`) so a `Coverage` reference can be carried
  alongside each member entity.
- In `input/fsh/ProfilePDexMemberMatchGroup.fsh`, add `matchedCoverage`
  (0..1 MS) to `PDexMemberMatchGroup` and `nonMatchedCoverage` (0..1
  MS) to `PDexMemberNoMatchGroup`, both bound to the new extension;
  update the `contained` comments to note that Coverage MAY be
  present.
- In `input/fsh/ExamplePDexMemberMatchGroup.fsh`, add
  `input-coverage-1` and wire `member[0]` to carry both
  `matchedMember` (Patient) and `matchedCoverage` (Coverage).
- In `input/pagecontent/payertopayerbulkexchange.md`, add §pdex-139d:
  when the same patient is submitted with multiple coverages, the
  responder **SHALL** include the original Coverage and populate the
  `MatchCoverage` extension on the corresponding group entity. Add a
  row in `input/pagecontent/changehistory.md` referencing FHIR-55927.

#### Justification

This addresses the root problem — loss of (Patient + Coverage)
provenance in the `bulk-member-match` response — without changing the
operation's invocation contract or the overall bulk-export wiring. It
uses a PDex-scoped extension on `Group.member.entity`, which is a
standard FHIR mechanism for adding optional context and cleanly
matches the briefing's "Add an extension (FSH)" recipe. The 0..1
cardinality preserves backward compatibility for responders that
choose not to carry the Coverage. The PDex-only scope avoids touching
HRex's base `bulk-member-match` definition, which keeps the change
self-contained and consistent with how the Da Vinci suite typically
layers IG-specific context on top of HRex operations. The label
`Pre-Applied` and the existing commit indicate the WG and authoring
team have already converged on this design, so a formal vote to
accept ratifies what is in the source tree.

---

### Disposition B: Alternative Approach

#### Proposal

Same end result, but tighten the conformance and clarify scope:

- Keep the new `MatchCoverage` extension and the
  `matchedCoverage` / `nonMatchedCoverage` slices.
- Strengthen §pdex-139d so that the responder **SHALL** populate the
  `MatchCoverage` extension on every member entity in
  `MatchedMembers`, `NonMatchedMembers`, and (where applicable)
  `ConsentConstrainedMembers` whenever the corresponding input
  `MemberBundle` carried a `Coverage` — not only when the same patient
  appears under multiple coverages. Edge-cases where the same patient
  appears once are still ambiguous in narrative-driven UI review and
  benefit from the same provenance.
- Add the `MatchCoverage` extension as a **must-support** element on
  the PDex member-match groups so server implementations are required
  to recognize it on round-trip parsing even if a particular response
  doesn't include it.
- Optionally raise the `MatchCoverage` extension to HRex (so other
  Da Vinci IGs reusing `bulk-member-match` can adopt the same
  pattern) — but only if HRex's next ballot window is open; otherwise
  defer the HRex move to a follow-up ticket and keep the extension
  PDex-namespaced for STU 2.x.

#### Justification

The ticket itself frames the issue as "user confusion during review."
That confusion is not limited to the duplicate-patient case — any
review UI that lays out the response next to the input bundles will
benefit from explicit Coverage provenance on every entity. Making
population unconditional (when input had Coverage) costs almost
nothing on the wire and removes a class of "is this the same case as
the input?" questions for reviewers. Promoting to must-support on the
profile slices is consistent with how PDex marks other contextual
elements. Considering HRex is honest about the cross-repo touch point
the briefing flags: `bulk-member-match` is HRex-defined, and an
HRex-level extension would let other IGs (e.g., PCT) reuse the
pattern without re-inventing it.

---

### Disposition C: Decline

#### Proposal

Decline the request. Document in §pdex-139d that, when a client needs
to disambiguate two `MemberBundle`s for the same patient, it must
either (a) submit them in separate `bulk-member-match` invocations,
or (b) correlate the response by relying on the order of input
bundles and the implicit `Group.member.entity` indexing. No change
to the response payload, no new extension, no new profile slices.

#### Justification

The reporter explicitly notes that the issue does not affect the
downstream `bulk-export` flow — only review UIs. One could argue the
appropriate place to solve a UI/review problem is the client review
tool (which already has the input bundles in hand and can correlate
on its own using `Patient` reference + invocation order), not the
operation's response payload. Adding a new extension introduces
modeling surface area, requires implementations to update parsers
and rendering, and risks divergence with the HRex-defined
`bulk-member-match` operation if HRex later chooses a different
mechanism.

---

### Recommendation

**Recommended disposition:** A

The change has already been implemented in `HL7/davinci-epdx`
(commit `afe3888c`) under the `Pre-Applied` label, with the
resolution flagged "Persuasive with Modification" — meaning the WG
authoring team has reviewed and accepted the design. The change is
small, additive, backward-compatible (0..1 cardinality, new optional
extension, no constraint tightening on existing elements), and uses
the briefing's standard "Add an extension (FSH)" recipe inside the
flat `input/fsh/` layout. The dispositional risk is correspondingly
low.

Disposition B's instinct to make population unconditional and
must-support is reasonable, and the WG should consider it as a
discussion-time amendment if there is appetite to broaden scope — but
folding that into this ticket risks expanding scope beyond what the
reporter actually raised and what has already been implemented.
Disposition C is not credible: the reporter's "review-tool can fix
it" workaround forces every implementer of the review tooling to
re-derive correlation from invocation order, which is brittle (e.g.,
async / chunked responses) and is exactly the kind of in-payload
ambiguity FHIR extensions exist to remove. Recommend ratifying
Disposition A and inviting WG members to raise B as an in-meeting
amendment if they want to tighten the SHALL.
