## Summary

The reporter asks that "Claim Received Date" be promoted from an ad-hoc
representation (currently carried in the CARIN BB IG via
`supportingInfo`) to a first-class backbone/top-level data element on
the FHIR Core financial resources — `Claim`, `ClaimResponse`, and
`ExplanationOfBenefit` — defined in a manner analogous to the existing
`created` element. A late comment from Corey Spears (2025-10-30)
clarifies that although the ticket originated against the CARIN Blue
Button IG, the requested change is properly a **FHIR Core** change to
the financial resources. The ticket was previously closed as
"Considered for Future Use" in 2020 and re-opened post-CARIN BB STU2
publication in 2022 so that it can be considered against the next
version of FHIR. No related Jira tickets surfaced via cross-reference
search, and no GitHub PRs/issues are linked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level `dateTime` element named `received` (cardinality
`0..1`) to **Claim**, **ClaimResponse**, and **ExplanationOfBenefit**
in FHIR Core, defined parallel to the existing `created` element. The
definition would be:

- **Path:** `Claim.received`, `ClaimResponse.received`,
  `ExplanationOfBenefit.received`.
- **Type:** `dateTime`.
- **Cardinality:** `0..1`.
- **Short:** "Date the resource was received".
- **Definition:** "The date this resource was received by the payer
  or processing system. Distinct from `created`, which records when
  the resource itself was instantiated/sent."
- **Comments:** Note relationship to `created` and to interchange
  patterns used by CARIN BB and CMS reporting.

Update the resource intro narratives on `claim.html`,
`claimresponse.html`, and `explanationofbenefit.html` to describe
`received` and its relationship to `created`. Mark the change in the
resource changelog. Coordinate with the CARIN BB IG so that the
existing `supportingInfo` slice for "claim received date" can be
deprecated in the next IG version in favour of the new core element.

#### Justification

This is what the reporter asked for and what Mark Roberts/Pat Taylor
voted to defer to a future FHIR version (18-0-5). Modelling
`received` as a peer of `created` is internally consistent — the two
dates carry distinct semantics (instantiation vs. receipt) and both
are needed to reason about adjudication timeliness. Promoting it from
an IG `supportingInfo` slice to a core element removes the magic
string / coding lookup that implementers must do today and gives the
element a stable canonical path that other IGs (CARIN BB, Da Vinci
PDex, CMS reporting) can rely on instead of re-defining their own
slices.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding `received` to all three resources, define a single
**standard FHIR extension** (e.g.,
`http://hl7.org/fhir/StructureDefinition/claim-receivedDate`,
`dateTime`, `0..1`, marked `Applicable to: Claim, ClaimResponse,
ExplanationOfBenefit`) published in FHIR Core. Update the CARIN BB IG
to reference this canonical extension in place of its current
`supportingInfo` slice. Cross-link the extension from each resource's
"Extensions" section.

#### Justification

`received` is meaningful primarily in payer-facing exchanges; many
non-claims-processing implementations of `Claim`/`ExplanationOfBenefit`
will never set it. A standard extension gives every implementer that
needs the element a stable, canonical place to put it without bloating
the core resource definitions or forcing FMG to commit to the precise
semantics in the base spec. It is also a smaller, lower-risk change
that can be balloted independently of the next major resource
revision, and it preserves backward compatibility with existing
instances. The trade-off versus Disposition A is discoverability and
ergonomics — extensions are less prominent than top-level elements —
but for a single optional date this is usually acceptable.

---

### Disposition C: Decline

#### Proposal

Decline the request and direct implementers to continue representing
"claim received date" via the existing `supportingInfo` mechanism
(with an appropriate code from a Financial Management value set), as
CARIN BB does today. Close the ticket as "Persuasive with Mod – use
existing mechanism" and document the rationale in the resource
narrative if needed.

#### Justification

`Claim.supportingInfo` (and the equivalent on `ClaimResponse` /
`ExplanationOfBenefit`) is designed precisely to carry typed,
date-bearing supporting facts about a claim, and the CARIN BB IG has
already proven that the pattern works in production. Adding a new
top-level element costs ballot effort, reviewer attention, and
implementer migration for a value that today only one IG actively
exchanges; if a second IG needs the same concept, it can adopt the
existing CARIN slice or the eventual standard extension proposed in
Disposition B without a base-resource change. Declining preserves
backward compatibility and avoids a precedent of promoting every IG
`supportingInfo` slice into the base spec.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need is real and is now reinforced by Corey
Spears's clarification that this is properly a Core concern rather
than a CARIN-only one — declining outright (Disposition C) leaves a
canonical concept fragmented across IGs and per-payer slices, which
is exactly the situation the original ballot vote (18-0-5) said
should be revisited in a future FHIR version. At the same time,
adding `received` as a new top-level element on three resources
(Disposition A) is a sizeable surface change with cross-IG migration
implications and should not be done without explicit FMG consensus
and CARIN BB coordination. A standard FHIR-Core extension
(Disposition B) gives the concept a single canonical home that CARIN
BB, Da Vinci, and CMS-facing IGs can all bind to immediately, defers
the harder "is this a peer of `created`?" modelling debate without
blocking implementers, and leaves the door open to promote the
extension to a top-level element in a later release if adoption
warrants it. FMG should confirm with the CARIN BB editors that the
extension URL and semantics meet their needs before publication.
