## Summary

The ticket asserts that the `ValueSet-NCPDPRejectCode` defined in the CARIN
Consumer Directed Payer Data Exchange (CARIN BB) IG should not have its
canonical URL anchored under `hl7.org/fhir`. Because the ValueSet is a
terminology artifact — and (per the surrounding "Block Vote 6" framing) is
an "all-codes" ValueSet over an external code system (NCPDP Reject Code) —
its defining URL should instead be anchored under `terminology.hl7.org`
(THO). The reporter calls for a companion UTG ticket to register the
ValueSet in THO so the IG can drop its locally-defined copy and re-bind to
the THO canonical. This ticket is gated on UTG ticket
[UP-316](https://jira.hl7.org/browse/UP-316), which would introduce the
"all codes from a complete external CodeSystem" ValueSet pattern in THO. No
explicitly-linked Jira tickets were returned by cross-reference, but the
comment thread points at UP-316 as the dependency. There are no related
GitHub items or repository cross-references on file.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat `ValueSet-NCPDPRejectCode` in the CARIN BB IG as a terminology
artifact whose canonical defining URL must live under
`terminology.hl7.org`, not `hl7.org/fhir`. Concretely:

1. File / link a UTG ticket (the Description explicitly asks for one;
   UP-316 is the natural parent) to register an "all codes" ValueSet
   over the external NCPDP Reject Code system at a THO canonical such
   as `http://terminology.hl7.org/ValueSet/NCPDPRejectCode`.
2. Once the THO ValueSet exists, retire the IG-local
   `ValueSet/NCPDPRejectCode` definition in the CARIN BB IG and
   re-point every binding (current and historical) at the THO
   canonical, preserving any version pin needed for STU2
   conformance.
3. Update narrative pages in the IG that reference the old
   `hl7.org/fhir/...` canonical to use the THO canonical, and add
   a release note to the IG's history page.

#### Justification

This is the standard FHIR / HL7 terminology layering: ValueSets and
CodeSystems are owned by THO, and IGs bind to them rather than
defining their own copies. The reporter is asking for nothing more
than that the IG conform to that pattern for an artifact that is
clearly terminology, not IG-specific content. The 2022-07-26
Financial Mgmt WG vote (13-0-0) already endorsed the underlying
direction; the only reason it was parked as "Considered for Future
Use" was the absence of an "all codes" pattern in THO, which UP-316
is meant to address. With the ticket re-opened post-STU2, the
cleanest disposition is to do exactly what the ticket asks once
UP-316 lands.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same end-state (THO-anchored canonical, IG bindings pointing
at THO) but stage it explicitly so the CARIN BB IG is not blocked on
UP-316. Specifically:

1. In the *next* CARIN BB publication, keep the IG-local ValueSet but
   change its canonical URL to a THO-style address
   (`http://terminology.hl7.org/ValueSet/NCPDPRejectCode`) under a
   formal MoU/coordination note with UTG so the URL is reserved.
2. Keep the existing IG-local ValueSet definition in place
   *transiently* (with a status of `draft` and a note that it will be
   retired) so implementers continue to have a resolvable definition.
3. When UP-316 (or its successor) introduces the "all codes" pattern
   in THO and the THO-hosted ValueSet exists, delete the IG-local copy
   and rely solely on the THO canonical with no further binding
   change required (because the canonical already matches).

#### Justification

This avoids a hard dependency on UTG schedule while still giving
implementers a stable, future-proof canonical URL. It also avoids two
disruptive binding changes (one to a new IG-anchored URL, then a
later one to a THO URL) — bindings change exactly once. The trade-off
is that for some interim period the canonical and the publishing
location diverge, which is irregular but not unprecedented across
HL7 IGs. If UP-316 progress remains uncertain, this is the lower-risk
path for CARIN BB consumers.

---

### Disposition C: Decline

#### Proposal

Persist with the previous resolution: "Considered for Future Use",
keep `ValueSet-NCPDPRejectCode` defined in the CARIN BB IG under its
current `hl7.org/fhir/...`-anchored canonical, and do not re-open the
issue until UP-316 has landed in THO and a THO-hosted NCPDP Reject
Code ValueSet exists. Track the ticket against UP-316 rather than
acting on it inside the CARIN BB IG.

#### Justification

The original FM WG vote (13-0-0) consciously deferred this work
because the "all codes from external CodeSystem" pattern did not yet
exist in THO. Re-opening without that prerequisite forces CARIN BB to
either (a) define a one-off THO-anchored canonical that THO does not
yet host (a layering inversion), or (b) churn the IG-local ValueSet
URL twice — once now, once when UP-316 lands. Declining (and waiting
on UP-316) avoids both anti-patterns and matches the prior consensus.
The reporter's underlying concern is preserved: the eventual end
state is identical; only the timing differs.

---

### Recommendation

**Recommended disposition:** A

The ticket is narrowly scoped, is standards-hygiene work the WG has
already endorsed in principle (13-0-0), and the only blocker — the
absence of an "all codes" pattern in THO — is itself an actively
tracked UTG item ([UP-316](https://jira.hl7.org/browse/UP-316)).
Accepting as requested keeps the CARIN BB IG aligned with FHIR's
terminology-layering convention (terminology artifacts live in THO;
IGs bind to them), produces a single binding change for implementers
once UP-316 lands, and explicitly creates the companion UTG work the
ticket calls for. Disposition B's "reserve the THO URL but keep an
IG-local definition" is defensible if the WG judges UP-316 unlikely
to land in a usable timeframe, but it leaves the IG publishing
content under a canonical it does not own — a worse long-term
posture than waiting. Disposition C re-litigates a vote that was
already taken and offers no improvement over A beyond schedule
optionality.
