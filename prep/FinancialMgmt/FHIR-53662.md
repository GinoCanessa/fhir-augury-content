## Summary

The ticket reports that the Da Vinci PDex `BulkMemberMatch` operation is
declared as a *type-level* operation in its `OperationDefinition`
([OperationDefinition-BulkMemberMatch](https://hl7.org/fhir/us/davinci-pdex/STU2.1/OperationDefinition-BulkMemberMatch.html))
but the page does not state which resource type the operation applies to
(i.e., the `OperationDefinition.resource` element is not surfaced on the
rendered page). Without that information, server implementers cannot
unambiguously determine the URL the operation must be mounted on
(`[base]/[ResourceType]/$bulk-member-match`). The PDex
`CapabilityStatement-pdex-payer-access-server` already declares the
operation as a `Group`-level operation, so the gap is between the
operation definition and the capability statement, not in the underlying
intent of the IG. The ticket has been closed as **Duplicate** with a
resolution note of "Fixed by earlier ticket"; no specific duplicate key
is recorded in the ticket metadata or in the surfaced cross-references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `OperationDefinition-BulkMemberMatch` resource in the Da
Vinci PDex IG so that the rendered page states the resource type the
type-level operation applies to. Concretely:

- Set `OperationDefinition.resource = ["Group"]` (matching the
  `pdex-payer-access-server` CapabilityStatement, which already declares
  `$bulk-member-match` as a `Group`-level operation).
- Keep `OperationDefinition.type = true` and `OperationDefinition.system
  = false`, `OperationDefinition.instance = false`.
- Republish `OperationDefinition-BulkMemberMatch.html` so the
  "Resource" / "URL" sections show `[base]/Group/$bulk-member-match`.
- Optionally, add a short narrative sentence to the operation page
  explicitly stating the mount point ("This type-level operation is
  invoked on the `Group` resource as
  `[base]/Group/$bulk-member-match`.") to avoid any future ambiguity for
  implementers reading only the operation page.

#### Justification

This is the literal fix the reporter asked for and aligns the
OperationDefinition with the existing (already-published) PDex
CapabilityStatement, which is the source of truth implementers used to
resolve the ambiguity in the connectathon. It restores the invariant
that any type-level FHIR `OperationDefinition` enumerates its target
resource(s) via `OperationDefinition.resource`, which is required for
clients and tooling that introspect the OperationDefinition without
also reading the CapabilityStatement. Because the CapabilityStatement
already says `Group`, this is a documentation-/metadata-only change with
no behavioural impact on conformant servers.

---

### Disposition B: Alternative Approach

#### Proposal

Close as a duplicate of the earlier PDex ticket that the resolution note
references and instead use FHIR-53662 to drive a cross-cutting
**operation-page consistency sweep** for PDex STU2.1: write an IG
publisher / QA check (or a simple script under the PDex IG repo) that
fails the build when an `OperationDefinition` with `type = true` (or
`instance = true`) does not enumerate at least one `resource`, and apply
the same `resource = ["Group"]` fix to `BulkMemberMatch` as part of the
sweep. Track the sweep under a new (or the existing duplicate) ticket;
mark FHIR-53662 itself as closed/duplicate as the resolution note
already records.

#### Justification

The reporter's underlying need — that operation definitions should be
self-describing about where they are mounted — is broader than this one
operation, and the PDex IG plausibly has other type/instance
operations with the same gap. A QA-level fix prevents the same class of
issue from recurring after future revisions and keeps the
OperationDefinition the single source of truth implementers expect.
This disposition also honours the existing "Duplicate / Fixed by
earlier ticket" resolution rather than re-opening the original ticket
just to make the same edit again, while still ensuring the underlying
content fix lands.

---

### Disposition C: Decline

#### Proposal

Leave the resolution as **Duplicate / Fixed by earlier ticket** and
make no further change under FHIR-53662. The implementer-facing
ambiguity is resolvable by reading the
`pdex-payer-access-server` CapabilityStatement, which authoritatively
declares `$bulk-member-match` as a `Group`-level operation, and the
referenced earlier ticket is already tracking the fix.

#### Justification

The Jira record is already closed as Duplicate; reopening or relabelling
it risks divergence from the canonical ticket that is actually
delivering the change. FHIR's CapabilityStatement is the normative
declaration of where an operation is mounted on a given server, and it
already conveys the `Group` mount point. A second editorial change
under a duplicate ticket adds no specification value and complicates
ballot-comment accounting.

---

### Recommendation

**Recommended disposition:** A

The resolution note "Fixed by earlier ticket" is too sparse to verify
that the duplicate actually amends `OperationDefinition.resource` on
`BulkMemberMatch` (no duplicate key is recorded on FHIR-53662 and no
GitHub PR is cross-referenced in the index). Disposition A makes the
intended state explicit and trivially verifiable: the rendered
operation page will list `Group` as the target resource and align with
the existing CapabilityStatement, which is what implementers in the
January 2026 PDex connectathon expected. The change is documentation-
only, has no breaking behavioural effect on servers that already mount
the operation on `Group`, and it closes the gap that caused at least
one connectathon participant to mount the operation on the wrong
resource. Disposition B is a reasonable upgrade if Financial
Management wants to invest in a publisher-level guard, but it should be
in addition to A rather than instead of it; Disposition C relies on the
duplicate ticket actually delivering this fix, which the available
evidence does not confirm.
