## Summary

This ticket flags two value sets bound from the `Claim` resource that are
candidates for migration to Terminology.HL7.org (THO) so they can be
reused (and independently versioned) outside the FHIR Core specification.
The two bindings called out are `Claim.event.type` (bound to ValueSet
`datestype`, backed by CodeSystem `datestype` / DatesEventTypeCodes) and
`Claim.procedure.procedure[x]` (bound to ValueSet `icd-10-procedures`,
backed by CodeSystem `icd-10-procedures` / ICD10ProcedureCodes). The
ticket explicitly notes that THO migration is **not required** — the
question Financial Management needs to answer is whether either value
set is realistic to need to evolve in the coming years and/or might be
useful to other product families (CDA, V2, other FHIR specs). The text
was imported from Comment 165 of a parent grooming ticket by Lloyd
McKenzie. No directly-linked Jira tickets were surfaced via cross-
references, but a tranche of nine sister tickets following the exact
same template was raised against other resources in the same window —
notably the Financial Management family ([FHIR-55204](https://jira.hl7.org/browse/FHIR-55204)
ClaimResponse, [FHIR-55216](https://jira.hl7.org/browse/FHIR-55216)
CoverageEligibilityRequest, [FHIR-55220](https://jira.hl7.org/browse/FHIR-55220)
CoverageEligibilityResponse, [FHIR-55228](https://jira.hl7.org/browse/FHIR-55228)
ExplanationOfBenefit) — so any disposition reached here will set
precedent for those companion tickets.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate **both** value sets and their backing code systems out of FHIR
Core and into Terminology.HL7.org:

- Move `CodeSystem/datestype` (DatesEventTypeCodes) and `ValueSet/datestype`
  to THO under HL7-wide ownership, retire the FHIR Core copies, and
  re-point `Claim.event.type` (and the equivalent bindings on
  `ClaimResponse.event.type`, `ExplanationOfBenefit.event.type`, and
  any other `*.event.type` slices flagged in the sister tickets) to the
  THO canonical URL.
- Move `CodeSystem/icd-10-procedures` (ICD10ProcedureCodes) and
  `ValueSet/icd-10-procedures` to THO and re-point
  `Claim.procedure.procedure[x]` (and the equivalent EOB binding) to
  the THO canonical URL. Treat the FHIR Core CodeSystem definition as a
  thin pointer to the externally-maintained ICD-10-PCS standard, with
  THO as the steward of the FHIR-side wrapper.

In both cases, the binding strength on the affected `Claim` elements is
preserved as-is; only the canonical URL of the bound value set changes.

#### Justification

Both value sets meet the explicit criteria in the ticket. `datestype`
already appears on `event.type` for several Financial Management
resources (Claim, ClaimResponse, CoverageEligibilityRequest/Response,
ExplanationOfBenefit per the sister tickets), so it is *de facto* a
cross-resource terminology that should not be governed by a single
resource's release cycle — exactly the case THO exists to serve. ICD-10
procedure codes are an externally-maintained classification whose
revision cadence is decoupled from FHIR releases; pinning them in FHIR
Core forces unnecessary FHIR ballot churn whenever ICD-10-PCS updates,
and a THO-hosted wrapper is also directly consumable from CDA and V2
implementation guides — another criterion the ticket calls out.
Aligning all FM `*.event.type` and `*.procedure.procedure[x]` bindings
to the THO copies in a single coordinated change keeps the FM family
consistent and avoids partial migrations.

---

### Disposition B: Alternative Approach

#### Proposal

Split the two value sets and treat them differently:

- **Migrate `datestype` to THO now.** It is a small, FHIR-defined
  enumeration that is already shared across multiple FM resources and
  is a clean candidate for HL7-wide stewardship. Re-point all FM
  `*.event.type` bindings to the THO canonical URL.
- **Leave `icd-10-procedures` in FHIR Core for this ballot cycle**,
  but re-classify it from a `complete` CodeSystem definition to a
  `not-present`/`fragment` CodeSystem with an `external` content mode,
  if it isn't already. Defer the THO migration until Vocabulary and FM
  have agreed on stewardship of FHIR's externally-sourced classification
  wrappers (ICD-10-CM, ICD-10-PCS, CPT, etc.) as a group, so all such
  code systems move together rather than one binding at a time.

#### Justification

The two value sets have very different characters and conflating them
into a single migration risks holding up the easy win (`datestype`)
behind a much larger policy question (how HL7 stewards FHIR's wrappers
around third-party classifications). Migrating `datestype` alone
delivers most of the cross-resource benefit Lloyd is asking for and
sets up the FM family for the matching dispositions on FHIR-55204 /
-55216 / -55220 / -55228. Holding `icd-10-procedures` for a coordinated
classification-wrapper effort avoids creating a one-off pattern that
later has to be revisited when the broader policy lands, and avoids
implementer churn from changing an ICD-10-PCS canonical twice in close
succession.

---

### Disposition C: Decline

#### Proposal

Decline THO migration for both value sets at this time. Document on
both `ValueSet/datestype` and `ValueSet/icd-10-procedures` that they
were reviewed for THO candidacy and intentionally retained in FHIR
Core, and close the ticket as "Persuasive with Modification — no
change" with a note pointing back to this disposition.

#### Justification

The ticket itself states migration is not required. `datestype` is a
small, stable enumeration whose definition has not changed in recent
releases; migrating it imposes implementer churn (every consumer must
update their canonical references and re-validate bindings) for a
benefit that is largely theoretical until another product family
actually asks to consume it. `icd-10-procedures` is already understood
by implementers to be an external classification wrapper, and moving
its canonical URL to THO does not change the underlying obligation to
license and download ICD-10-PCS from CMS — i.e., it doesn't materially
help implementers. If a concrete cross-spec consumer appears (a CDA or
V2 IG that wants to reuse these value sets), the migration can be
revisited at that point with real requirements driving the canonical
choice.

---

### Recommendation

**Recommended disposition:** B

Disposition B best matches the realities Lloyd's ticket implicitly
acknowledges: the two value sets are not equivalent migration
candidates, and FM should not bundle a quick, low-risk move
(`datestype`) with a much larger HL7-wide policy decision about
externally-maintained classifications (ICD-10-PCS, and by extension
ICD-10-CM, CPT, LOINC subsets, etc.). Migrating `datestype` to THO
*now* gives the FM family a single coordinated answer for FHIR-55199,
-55204, -55216, -55220, and -55228 in this ballot cycle and starts
paying off the cross-resource reuse benefit Lloyd is pointing at.
Holding `icd-10-procedures` for a coordinated FHIR-wide
classification-wrapper effort (ideally jointly with Vocabulary)
prevents an isolated change that we'd likely have to revisit, and
keeps implementer churn for that binding to a single future event
rather than two. Disposition A is defensible if FM and Vocabulary
already have an agreed plan for the classification wrappers; if they
don't, B is the safer recommendation to bring to the workgroup.
