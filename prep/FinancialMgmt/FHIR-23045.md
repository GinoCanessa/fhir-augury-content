## Summary

The reporter (Chirag Bhatt) is asking the Financial Management work group
how a **Medicaid identifier** should be represented on an ELTSS
Beneficiary's `Patient.identifier`. Specifically, the ticket questions
how `Identifier.type` (a CodeableConcept bound to the extensible
`identifier-type` value set) should be coded for a Medicaid ID, given
that the documented value set surfaces things like Medical Record
Number but not an obvious "Medicaid number" code, while LOINC carries
codes such as `45400-9` (Medicaid number) and the v2-0203 table carries
`SB` (Social Beneficiary Identifier).

The reporter also notes that Medicaid IDs are administered per state,
so any solution needs to allow a state-specific `Identifier.system`
(analogous to the per-state Driver License entries in the FHIR
Identifier Registry) rather than relying on a single national system
URL.

The ticket has been routed between work groups: Johnathan Coleman noted
in 2020 that Kathleen Connor recommended bringing this to FM, and Bob
Heale (HSS) escalated again in 2024 asking FM to respond. Three more
recent comments from `goradatta` (2026) are administrative ("not sure
why I am here / assigned") and add no substance.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items
were surfaced via `cross-referenced` or a Zulip `search`, so the
question has effectively sat unanswered since 2019.

### Disposition A: Accept As Requested

#### Proposal

Adopt the pattern the reporter sketched: distinguish Medicaid IDs
**primarily by `Identifier.system`** (one URL per state Medicaid
program), and **secondarily by `Identifier.type`** using the
HL7 v2-0203 code `SB` ("Social Beneficiary Identifier") from
`http://terminology.hl7.org/CodeSystem/v2-0203` (which is the source of
the `identifier-type` value set's `SB` concept, not LOINC).

Concretely:

1. In the **FHIR Identifier Registry**
   (`hl7.org/fhir` → `identifier-registry.html`) add a new section for
   **Medicaid Beneficiary Identifiers**, modeled on the existing
   **Driver's License** section: a base/parent URI plus a per-state
   table (e.g.,
   `http://hl7.org/fhir/sid/us-medicaid/{state}` where `{state}` is the
   USPS two-letter code), and document that the per-state URI is the
   canonical `Identifier.system` for that state's Medicaid program.
2. In **HL7 Terminology (THO)**, register a `NamingSystem` for the
   parent Medicaid identifier (and optionally one per state) parallel
   to `medicareHIC`, with a stable OID and the URI from step 1.
3. In the **ELTSS IG**, update the Beneficiary Demographics "Additional
   Mapping Details" item 3 to instruct implementers to:
   - set `Identifier.system` to the per-state Medicaid URI from the
     Registry,
   - set `Identifier.type.coding` to
     `{system: "http://terminology.hl7.org/CodeSystem/v2-0203",
     code: "SB", display: "Social Beneficiary Identifier"}`,
   - and *optionally* add a second `coding` with LOINC `45400-9`
     ("Medicaid number") for systems that prefer LOINC.

#### Justification

This mirrors the pattern already used for driver's licenses (per-state
identifier systems registered centrally) and for Medicare (a single
`NamingSystem` with both OID and URI), which Bob Heale's 2023 comment
explicitly offered as the precedent. It honours FHIR's standing
guidance that **identity is carried by `Identifier.system` and type is
carried by `Identifier.type`** — the type code does not need to encode
"which Medicaid program," because the system URL already does so. Using
v2-0203 `SB` keeps the `Identifier.type` binding inside the existing
extensible value set without requiring an extension, and allowing an
additional LOINC coding accommodates implementers (e.g., labs)
already using LOINC for the same concept. It also gives HSS — who
escalated in 2024 — a concrete answer they can publish.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** create a per-state branch in the FHIR core Identifier
Registry. Instead:

1. Publish a **single national Medicaid `NamingSystem`** in HL7
   Terminology (e.g., URI `http://hl7.org/fhir/sid/us-medicaid`)
   modeled on `medicareHIC`, and document that the actual per-state
   Medicaid issuer is conveyed via `Identifier.assigner` (a
   `Reference(Organization)`) and/or `Identifier.extension` carrying
   the state code (USPS two-letter), rather than by minting 50+
   identifier system URIs.
2. For `Identifier.type`, recommend a single `coding` of v2-0203 `SB`
   and discourage LOINC `45400-9` to avoid two competing canonical
   type codes.
3. Surface this guidance in the **ELTSS IG** Beneficiary Demographics
   section, plus a short worked example.

#### Justification

A per-state registry section (Disposition A, step 1) creates ongoing
maintenance for the FHIR core editors (50 + DC + territories) and a
risk of drift as states change Medicaid program names or URLs. A
single national system + `assigner`/extension keeps the registry small,
aligns with how some other US national identifiers (e.g., `us-ssn`,
`us-medicare`) are published, and pushes per-state knowledge to the
Organization resource where it more naturally lives. The trade-off is
that consumers have to look at two fields (`system` + `assigner`) to
disambiguate two beneficiaries with the same Medicaid number from
different states, but in practice ELTSS payloads already carry the
issuing organisation.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **"Already addressed by existing mechanism — no
spec change."** Respond to HSS that:

- `Identifier.system` already exists precisely to disambiguate per-payer
  identifiers; ELTSS implementers should mint and publish a state
  Medicaid `Identifier.system` URI (or use one their state already
  publishes) without a registry entry being mandatory.
- `Identifier.type` should use v2-0203 `SB`
  (`http://terminology.hl7.org/CodeSystem/v2-0203#SB`), which is
  already in the extensible `identifier-type` value set; LOINC
  `45400-9` may be added as an additional coding if local policy
  requires it.
- No change to the FHIR core spec, the Identifier Registry, or THO is
  required to support this; this is implementation guidance for the
  ELTSS IG only.

#### Justification

This is a 2019 **Question**-type ticket that has been Triaged but never
escalated to a CR or Change Request. The mechanics already exist in
FHIR (extensible `Identifier.type`, free-form `Identifier.system`,
optional `Identifier.assigner`). Declining the spec change keeps the
core registry small, avoids THO churn, and unblocks HSS by giving them
a definitive interpretation they can document in their own IG. It
also matches the most recent FM signal — the ticket has sat unassigned
for years, and the only inbound activity is administrative reassignment
noise.

---

### Recommendation

**Recommended disposition:** **A**

Disposition A directly answers the question HSS has been waiting on
since 2024 and gives ELTSS (and any other US realm IG handling
Medicaid) a stable, registered home for state Medicaid identifiers,
mirroring the well-understood Driver's License precedent and the
Medicare `NamingSystem` precedent that bheale already cited in the
ticket. Disposition B avoids registry maintenance but pushes
state-disambiguation into `assigner`/extension, which is harder for
implementers to consume consistently and weaker for cross-payer
matching. Disposition C is defensible as a strict reading of "no spec
change needed," but after seven years of the question being open,
silently telling HSS "you already had everything you need" is unlikely
to satisfy the originating work group; even if FM ultimately declines
the registry entry, the ticket should at minimum yield a written FM
ruling in the ELTSS IG, which is exactly what A produces.
