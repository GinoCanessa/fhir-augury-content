## Summary

The ticket asks the Da Vinci Patient Cost Transparency (PCT) IG to move
the canonical (defining) URL of `ValueSet-PCTProcedureSurgicalCodes` out
of the IG's `hl7.org/fhir/...` namespace and anchor it in
`terminology.hl7.org` (THO) instead, on the rationale that only value
sets bound `required` to a `code` data type belong in the IG's own
namespace. The reporter explicitly calls for a companion UTG ticket to
register the value set in THO. The ticket is one of a large family of
sibling PCT tickets raised in the same review pass
(FHIR-35529/35532/35533/35534/35535/35537/35540/35545/35548/35549/35550,
plus FHIR-45812/45815/45816/45817/45818) that all make the same THO
re-anchoring request against different PCT value sets and code systems;
its disposition should therefore be coordinated with the rest of that
family. No directly-linked Jira tickets are surfaced by cross-references,
but the Zulip Block-2025-Jul vote thread groups all of them together.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the canonical URL of `ValueSet-PCTProcedureSurgicalCodes` in
THO. Concretely:

1. Open a companion UTG ticket against the THO project to register the
   value set under
   `http://terminology.hl7.org/ValueSet/PCTProcedureSurgicalCodes`
   (final slug to be agreed with TSMG/UTG), preserving the existing
   compose (codes drawn from the relevant procedure code systems —
   CPT/HCPCS/ICD-10-PCS as currently composed).
2. In `HL7/davinci-pct`, change the `ValueSet.url` of
   `PCTProcedureSurgicalCodes` to the new THO canonical, retire the
   PCT-namespaced URL (with a `replacedBy` / "moved to" note in the
   PCT VS page during a transition window), and update every binding
   that references it (e.g., on `ExplanationOfBenefit`/`Claim`-derived
   PCT profiles where this VS is bound `extensible` or `preferred`).
3. Coordinate the change as part of the FM/THO effort referenced in
   Corey Spears' 2025-07-01 comment so the URL move lands in the next
   PCT publication after the THO registration.

#### Justification

This is what the reporter (Joan Harper) and the Comment-8 author (Ron
G. Parker) explicitly requested, and it is consistent with the broader
HL7 terminology governance pattern: value sets that are not bound
`required` to a `code`-typed element (and therefore are not part of the
IG's normative computable surface) belong in THO so they can be
maintained, versioned, and reused across IGs. Corey Spears' 2025-04-16
comment confirms `PCTProcedureSurgicalCodes` is not PCT-specific; that
non-IG-specific scope is precisely the criterion THO is meant to
absorb. The 2025-10-08 re-open of the ticket post-STU2 indicates the FM
WG no longer wants to defer indefinitely.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the per-ticket re-anchoring and instead resolve FHIR-35536 (and
its siblings FHIR-35529/35532/35533/35534/35535/35537/35540/35548/35550
and the FHIR-458xx family) as **Persuasive with Modification, batched**
under a single coordinated FM/THO project:

1. Treat the THO migration as one work item, not 20 individual ticket
   edits. Stand up (or reuse) a tracking ticket in the FM-THO effort
   that owns the migration plan, the THO slug list, and the deprecation
   timeline.
2. For each affected PCT VS/CS, agree the THO destination, raise the
   matching UTG ticket(s), and stage the PCT IG changes in a single PR
   so URL moves land atomically with the THO content going live.
3. Resolve FHIR-35536 with a comment pointing to the umbrella tracking
   ticket and the agreed THO target, then close it once the umbrella
   work merges.

#### Justification

The Block-2025-Jul resolution text ("Resolve across IGs continuing to
add to THO") and Corey Spears' 2025-04-16 / 2025-07-01 comments both
already acknowledge that this is a cross-IG terminology problem, not a
single-VS bug. Migrating canonical URLs piecemeal risks (a) inconsistent
naming across the family, (b) implementer churn from staggered
deprecation windows on related VS/CS, and (c) divergence between PCT and
sibling FM IGs that import the same procedure-code value sets. Batching
keeps the implementer impact to one breaking-change window, lets
TSMG/UTG approve the slugs as a set, and is less risk than re-opening
each of ~20 tickets independently and risking partial completion if
priorities shift mid-cycle. It still gives the reporter what they
asked for; it just ties FHIR-35536 to a project plan rather than a
standalone edit.

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-35536 as **Considered for Future Use** (or **Not
Persuasive**) and keep the existing
`http://hl7.org/fhir/us/davinci-pct/...` canonical URL for
`ValueSet-PCTProcedureSurgicalCodes`. Document in the resolution that
the FM WG, in consultation with TSMG, has elected to leave PCT-published
value sets in the IG's own namespace because (a) the VS is shipped and
versioned with the IG and (b) URL churn imposes a real implementer cost
that is not justified by the abstract "belongs in THO" argument alone.

#### Justification

A canonical URL is a contract with implementers. Once an IG has
published — and PCT has now published STU1 and STU2 — moving the URL is
a breaking change for any consumer who has bound to it, validated
against it, or stored references to it in `meta.profile`/canonical
fields. The Block-2025-Jul vote already chose this disposition (9-0-2)
on 2025-07-08, and the 2022-11-29 vote (9-0-1) chose the same
disposition for the same reason. Two successive WG votes landing on
"defer" suggests the cost/benefit of a blanket re-anchoring has not
clearly tipped in favour of the change. Declining (with a clear
rationale) is defensible, especially if the FM/THO umbrella work
referenced by Corey Spears does not in fact materialise.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — batched, coordinated migration).

The reporter's underlying concern is sound and has been re-affirmed
twice by re-opens (2023-03-30 post-STU1, 2025-10-08 post-STU2): a
not-PCT-specific procedure value set should live in THO so other IGs
(and other FM specifications) can reuse it without taking a dependency
on `hl7.fhir.org/us/davinci-pct`. Disposition A delivers that, but the
ticket history shows the WG has twice voted to defer because doing it
ticket-by-ticket is operationally expensive and breaking-change-noisy
for implementers; that pattern is unlikely to change if FHIR-35536 is
treated in isolation. Disposition C preserves implementer stability but
ignores the WG's own re-open signal and leaves the cross-IG reuse problem
unresolved. Disposition B threads that needle: it commits to the THO
destination the reporter asked for, but binds the work to the FM/THO
umbrella effort Corey Spears already named on 2025-07-01, so the URL
move lands once, atomically, with the THO content live and a single
deprecation window. It also lets the WG resolve the whole
FHIR-355xx/FHIR-458xx family with a consistent rationale instead of
re-litigating the same question twenty times.
