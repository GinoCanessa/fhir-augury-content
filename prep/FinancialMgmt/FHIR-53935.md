## Summary

The reporter notes that the `ClaimResponse` resource's *Scope and Usage*
section explicitly mentions that a `ClaimResponse` may be returned for
**re-adjudication** and **reversal** workflows, but the resource provides no
direct way to identify which prior `ClaimResponse` is being reversed or
re-adjudicated. Today only `ClaimResponse.request` (a reference to the
originating `Claim`) is available, which is insufficient when a single
`Claim` has produced multiple historical `ClaimResponse` payments — for
example when several payments and reversals have already been linked to one
`Claim`, or when duplicate payments were issued in error and the payer needs
to tell the provider exactly which payment is being reversed. The ticket
asks Financial Mgmt to add a `ClaimResponse` → original `ClaimResponse`
link so reversal/re-adjudication transactions are unambiguous.

No Jira links, no Zulip cross-references, and no GitHub cross-references
were surfaced for this ticket by the cross-reference index. The closest
prior FM tickets in the same problem space are FHIR-53934 (a sibling
ticket from the same reporter on the same day arguing that PLB-style
non-claim payment reductions should live outside the
`Claim`/`ClaimResponse` flow), and the PAS-side tickets FHIR-47769 and
FHIR-49021, which both wrestle with how a `ClaimResponse` reliably points
back to the originating `Claim`. None of those address the
ClaimResponse-to-ClaimResponse linkage that this ticket is asking for.

## Details

**Description:**

> In the Scope and Usage section, the spec mentions this resource can be
> returned for Re-adjudication and Reversals. In that case, it would be
> helpful if the ClaimResponse referenced the original ClaimReponse used
> for payment to illustrate which ClaimResponse is being reversed.
>
> This could be useful in a scenario where multiple payments and reversals
> are already linked with a single Claim resource. It could also be useful
> if multiple payments were issued by mistake to instruct providers which
> payment is being reversed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir`, add an optional self-reference to `ClaimResponse` that
points at the prior `ClaimResponse` being reversed or re-adjudicated.
Concretely, in
`source/claimresponse/claimresponse.xml` (`<differential>` only), add
either:

- `ClaimResponse.priorResponse : Reference(ClaimResponse) [0..1]` —
  populated only when `ClaimResponse.outcome` is a reversal /
  re-adjudication outcome and the receiver knows the prior
  `ClaimResponse`'s identity; or
- a backbone element
  `ClaimResponse.related : BackboneElement [0..*]` containing
  `claimResponse : Reference(ClaimResponse) [1..1]` and
  `relationship : CodeableConcept [0..1]` (preferred binding to a new
  `ClaimResponseRelationship` ValueSet under `HL7/UTG` with at least
  `reversal-of` and `readjudication-of` codes), mirroring the existing
  `Claim.related` pattern.

Update `source/claimresponse/claimresponse.html`'s *Scope and Usage*
section to explain when the new element is expected, and add a
constraint (`if outcome in {reversal, readjudication} then prior
linkage SHOULD be present`) using SHOULD-level guidance to avoid
breaking existing implementations.

#### Justification

This directly satisfies the reporter's use cases (multiple payments and
reversals on a single `Claim`; duplicate-payment correction). Modeling
the link via a `related` backbone with a coded relationship mirrors how
`Claim.related` already works and is consistent with how
adjudication-related metadata has historically been added to FM
resources (see FHIR-26723 / FHIR-26719). Keeping the new element
optional preserves backward compatibility for senders that do not need
the link, while the SHOULD-level constraint on reversal/re-adjudication
outcomes gives implementers like the reporter the deterministic linkage
they need without forcing every sender to populate it.

---

### Disposition B: Alternative Approach — Standard Extension instead of a Core Element

#### Proposal

Do not change the `ClaimResponse` `<differential>`. Instead, define a
standard FHIR extension (e.g., `http://hl7.org/fhir/StructureDefinition/
claimresponse-priorResponse`) of type `Reference(ClaimResponse)`,
optionally with a companion `claimresponse-priorResponseRelationship`
extension of type `CodeableConcept` bound to a new
`ClaimResponseRelationship` ValueSet (`reversal-of`,
`readjudication-of`). The extension would live in
`HL7/fhir-extensions` and would be registered against `ClaimResponse`.
Update the `ClaimResponse` *Scope and Usage* prose in `HL7/fhir` to
mention the extension and recommend its use for reversal /
re-adjudication transactions.

#### Justification

This keeps the change out of the FHIR Core resource `<differential>`
and out of the publisher pipeline for `HL7/fhir`, which is the
lower-risk path between ballots. It is consistent with FM/Da Vinci
practice of layering payer-specific semantics on top of
`ClaimResponse`/`ExplanationOfBenefit` as extensions before promoting
them to the core (see FHIR-26719). Implementers who need the linkage
can adopt the extension immediately without waiting for the next FHIR
Core ballot, and downstream IGs (PAS, PCT, EHRx) can mandate it in
their own profiles. The trade-off vs. Disposition A is discoverability:
implementers may overlook a non-core extension, and a future promotion
to core element is non-trivial because it requires a follow-on Jira
ticket and another ballot cycle.

---

### Disposition C: Decline — Use Existing Mechanisms (Identifier + outcome + request)

#### Proposal

Decline the requested change. Document in the *Scope and Usage* section
of `ClaimResponse` (in `source/claimresponse/claimresponse.html`) the
existing pattern for correlating reversal / re-adjudication
`ClaimResponse` instances:

- `ClaimResponse.request` already references the originating `Claim`
  (and PAS has tightened this to `1..1` per FHIR-47769 / FHIR-49021).
- `ClaimResponse.identifier` and the payer's payment/EOB identifiers
  carried in `Claim.identifier` and `ClaimResponse.identifier` /
  `ClaimResponse.payment.identifier` are already sufficient to
  correlate corrective transactions when both endpoints share the same
  identifier system.
- Sequencing across multiple `ClaimResponse` instances on a single
  `Claim` can be derived from `ClaimResponse.created` and
  `ClaimResponse.outcome`.

Add explicit narrative guidance describing this correlation pattern
rather than introducing a new element.

#### Justification

Defensible if the workgroup considers the existing identifier-based
correlation pattern adequate and prefers to avoid expanding the
`ClaimResponse` element list. Adding a `ClaimResponse.priorResponse`
reference creates a new optional path that downstream IGs (PAS, PCT,
EHRx) and X12 837/835 mappings would all need to consider, and the
underlying linkage can in principle be reconstructed from
`ClaimResponse.request` plus identifiers and timestamps. The cost of
this disposition is real: the reporter's scenarios — duplicate
payments, multiple reversals on one `Claim` — show that identifier
heuristics alone are fragile, and the narrative-only fix does not
fully close the gap.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a concrete, recurring implementer pain
point (multiple `ClaimResponse` instances against one `Claim`,
including duplicate-payment corrections) that is *not* solved by the
current `ClaimResponse.request` → `Claim` reference, and the related FM
ticket FHIR-53934 from the same reporter on the same day reinforces
that the existing `ClaimResponse` model under-specifies corrective /
reversal flows. PAS has already had to mandate the
`ClaimResponse → Claim` link (FHIR-47769, FHIR-49021), and FHIR-55394
shows that downstream consumers are now routinely receiving Bundles
with multiple `ClaimResponse` entries with no defined linkage. A
first-class, optional `ClaimResponse.related` (or `priorResponse`)
element with a coded relationship is a small, backward-compatible core
change that mirrors the established `Claim.related` pattern and gives
every downstream IG a single, discoverable hook to constrain. The
extension-only path (Disposition B) is a reasonable interim if the WG
wants to ship before the next Core ballot, and could be promoted to
core later — but starting with the core change avoids a duplication
cycle. Decline (Disposition C) is hard to defend given that the
reporter's duplicate-payment scenario is exactly where identifier-only
correlation breaks down.
