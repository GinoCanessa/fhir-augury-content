## Summary

The reporter identified a discrepancy between two authoritative artifacts
for the Da Vinci PDex `$provider-member-match` operation: the
`OperationDefinition-ProviderMemberMatch.json` shipped in the IG download
package lists the IN parameters as `MemberPatient`, `CoverageToMatch`,
`CoverageToLink`, and `TreatmentAttestation`, while the
`provider-parameters-multi-member-match-bundle-in` StructureDefinition
(the Parameters profile linked from Artifacts) lists them as
`MemberPatient`, `CoverageToMatch`, `Consent`, and `CoverageToLink`. The
ticket asks the work group to (a) decide which set of parameter names is
correct, (b) republish both artifacts so they agree, and (c) add explicit
links from the Provider Member Match operation page to the IN and OUT
Parameters profiles so readers can navigate between them.

The Resolution Description on the ticket (status `Triaged`,
resolution `Persuasive`, label `Pre-Applied`) indicates the issue has
already been worked: the Parameters profile was renamed (container slice
`MemberBundle` → `MembersToMatch`; part slice `Consent` →
`TreatmentAttestation`) to align the wire format with the
OperationDefinition, and the Operation FSH was updated with profile
links and clarifying notes. Two commits in `HL7/davinci-epdx`
(`eaf19a6f…` "Resolve FHIR-56075: align ProviderMemberMatch operation
and Parameters profile" and `4b33a8e7…` "Add FHIR-56075 to STU 2.2.0
change history") implement and record the fix. No related Jira tickets
or Zulip threads were surfaced by cross-references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the OperationDefinition (`OperationDefinition-ProviderMemberMatch`,
sourced from `input/fsh/Operation-provider-member-match.fsh`) as
authoritative and reshape the Parameters IN profile
(`provider-parameters-multi-member-match-bundle-in`, sourced from
`input/fsh/ProviderMultiMemberMatchRequestProfile.fsh` in
`HL7/davinci-epdx`) to match it: rename the container slice
`MemberBundle` → `MembersToMatch`, rename the part slice/name
`Consent` → `TreatmentAttestation`, and verify the OUT-side profile
slices match the OperationDefinition's OUT parameters. In
`Operation-provider-member-match.fsh`, add explicit profile links to
the IN (`provider-parameters-multi-member-match-bundle-in`) and OUT
(`provider-parameters-multi-member-match-bundle-out`) profiles in the
operation Description and in the relevant parameter docs, and add a
clarifying note on the `TreatmentAttestation` part explaining that the
name was chosen to avoid confusion with patient consent used in
payer-to-payer exchange. Add a STU 2.2.0 change-history entry in
`input/pagecontent/changehistory.md`. Rebuild with `_updatePublisher` +
`_genonce` so the download-package `OperationDefinition-ProviderMemberMatch.json`
and the Artifacts-page Parameters profile are republished from a single
consistent source.

#### Justification

This is exactly what the ticket requests — pick a single authoritative
shape and republish both artifacts so they agree, and surface the
Parameters profiles from the OperationDefinition page. Choosing the
OperationDefinition's parameter names as the anchor is the lower-impact
direction: the OperationDefinition is what implementations bind to
(parameter names appear on the wire), so changing the profile slice
names re-aligns documentation without forcing implementers to change
request payloads. The `Consent` → `TreatmentAttestation` rename is
itself a clarification — `Consent` collides conceptually with the
payer-to-payer Consent flow used elsewhere in PDex, while
`TreatmentAttestation` reflects what the parameter actually carries.
The ticket carries the `Pre-Applied` label and resolution
`Persuasive`, and the two `HL7/davinci-epdx` commits (`eaf19a6f` and
`4b33a8e7`) already implement this shape, so accepting as requested is
also the lowest-friction outcome for the work group. The flat
`input/fsh/` layout and the regenerated `output/changehistory.*` /
`template/changehistory.html` mean the source-level edits in the two
FSH files plus the changehistory entry are sufficient to update both
the download package and the Artifacts page in one publisher run.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the Parameters profile's original shape as authoritative instead:
keep the IN profile slices as `MemberBundle` / `Consent` /
`CoverageToMatch` / `CoverageToLink` / `MemberPatient`, and instead edit
`Operation-provider-member-match.fsh` so the OperationDefinition's
parameter names (and OUT-profile alignment) match the profile. Under
this alternative the `Consent` parameter would be retained but its
`Description` would be expanded to disambiguate it from the
payer-to-payer Consent flow (e.g. "treatment-relationship attestation,
recorded as a Consent resource"). Profile links would still be added
from the operation page to both IN and OUT profiles, and a change-history
entry would still be added to `input/pagecontent/changehistory.md`. All
edits remain in `input/fsh/` and `input/pagecontent/` in
`HL7/davinci-epdx`; no cross-repo changes are needed.

#### Justification

This alternative inverts the direction of alignment, which is worth
considering only if the work group believes the previously-published
Parameters profile is the artifact implementers have actually built
against. It avoids any rename to the profile's slice names — which can
matter if downstream IGs or test tooling reference those slice ids — but
it pushes the wire-format rename (`Consent` ↔ `TreatmentAttestation`)
onto the OperationDefinition side instead, which is a heavier change
because the OperationDefinition's parameter names appear directly in
the `parameters` element of request bundles. The repo briefing also
flags HRex (`davinci-hrex` 1.1.0) as the source of the member-match
patterns PDex re-uses; keeping `Consent` here perpetuates the naming
collision with the payer-to-payer Consent profile family in PDex
itself, which is precisely the confusion the ticket is trying to
remove. This option is therefore feasible but trades a documentation
fix for an implementer-visible regression.

---

### Disposition C: Decline

#### Proposal

Decline to make a normative change. Add narrative-only clarification in
`input/pagecontent/` (e.g. an inline note on the operation page and on
the `provider-parameters-multi-member-match-bundle-in` Artifacts page)
saying the canonical wire shape is the one in
`OperationDefinition-ProviderMemberMatch.json`, and that the Parameters
profile is informative scaffolding whose slice names are not normative.
Add cross-links between the operation page and the two Parameters
profiles (the navigation half of the request) without renaming any
slices or parameters. Categorize the rationale as
"already addressed by existing mechanism" — the OperationDefinition
itself is the normative artifact for parameter names.

#### Justification

A pure-decline (no change at all) is hard to defend because the IG
genuinely ships two artifacts with conflicting parameter sets, which is
a publication defect. The narrative-only variant is defensible in
principle — the OperationDefinition is the normative source for
parameter names, and a profile exists to constrain element typing /
cardinality, not to re-define operation parameter identifiers. Under
this disposition the discrepancy is reframed as a documentation bug
rather than a model bug. However, this leaves
`provider-parameters-multi-member-match-bundle-in` advertising slices
that don't appear on the wire, which validators and example
generators in the Da Vinci suite (and in HRex-derived tooling) will
continue to trip over. The disposition is therefore weak unless the
work group has a specific reason to avoid renaming the profile slices.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been worked under this disposition (label
`Pre-Applied`, resolution `Persuasive`, two implementing commits in
`HL7/davinci-epdx`), and the substantive trade-offs favour aligning the
Parameters profile to the OperationDefinition: the OperationDefinition
is the artifact implementers bind to, the `TreatmentAttestation` rename
removes a real terminology collision with the PDex payer-to-payer
Consent flow, and the change is fully contained in two FSH files plus a
change-history entry under `input/fsh/` and `input/pagecontent/` —
matching the briefing's "Add a profile (FSH)" and "Add narrative"
recipes and avoiding any of the flagged cross-repo / triple-US-Core
hazards. Disposition B is a coherent alternative but trades a
profile-only documentation fix for an implementer-visible parameter-name
change on the wire, which is the more disruptive direction. Disposition
C does not on its own resolve the underlying discrepancy. The work
group should ratify the already-applied fix and confirm publication on
the active STU 2.2.0 pre-/post-publish branch.
