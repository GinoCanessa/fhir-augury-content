## Summary

The submitter requests that the PDex Provider Access Consent Profile
(`ProfilePDexConsentProviderSharingOptOut`) allow `Consent.performer` to
reference `RelatedPerson` in addition to `Patient`. The motivation is
regulatory: under CMS-0057-F (and consistent with HIPAA), a patient may
exercise a Provider Access opt-out either personally or through a
legally recognized personal representative such as a parent, guardian, or
healthcare proxy. The current profile constrains `Consent.performer` to
`Patient` references only, which prevents faithfully representing
opt-out actions taken by a personal representative — even though such
actions are explicitly allowed by the underlying regulation. The
submitter notes that while a personal representative could occasionally
be modeled as a `Patient`, `RelatedPerson` is the more appropriate
resource and the reference target should be expanded to include it.

The ticket has already been **Resolved as Persuasive** and carries the
`Pre-Applied` label; commit
[`7360463b`](https://github.com/HL7/davinci-epdx/commit/7360463bab8bbba7a1e37ab621ac6f58b0b7b96c)
in `HL7/davinci-epdx` implements the requested change by widening
`performer` to `Reference(USCorePatientProfile | $USCoreRelatedPerson)`,
adding a `$USCoreRelatedPerson` alias, and updating the profile
description and performer comment to cite CMS-0057-F and HIPAA. There
are no cross-referenced Jira tickets giving additional context.

## Details

**Description:**

Under CMS-0057-F, patients must be able to opt out of Provider Access
API data sharing, and that opt-out may be exercised by the patient or
the patient's legally recognized personal representative (e.g., parent,
guardian, healthcare proxy).

The current PDex Provider Access Consent Profile constrains
`Consent.performer` to `Patient` references only. This prevents
compliant representation of opt-out actions when they are executed by a
personal representative acting on the patient's behalf, even though this
is explicitly permitted under CMS-0057 and HIPAA. While a personal
representative could be represented with a `Patient` resource sometimes,
a `RelatedPerson` resource would be more appropriate, and the reference
should be expanded to allow for that.

**Resolution (already applied — `Pre-Applied` label):**

- `input/fsh/aliases.fsh`: added `$USCoreRelatedPerson` alias.
- `input/fsh/ProfilePDexConsentProviderSharingOptOut.fsh`:
  `performer only Reference(USCorePatientProfile)` →
  `Reference(USCorePatientProfile | $USCoreRelatedPerson)`, with a
  comment citing CMS-0057-F and HIPAA basis.
- Profile description updated to note that personal-representative
  opt-outs are explicitly supported.
- `input/pagecontent/changehistory.md` updated.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/davinci-epdx`, expand the `Consent.performer` reference target
on the PDex Provider Access Consent (Opt-Out) profile to include
`RelatedPerson`. Concretely, in
`input/fsh/ProfilePDexConsentProviderSharingOptOut.fsh` change

```
* performer only Reference(USCorePatientProfile)
```

to

```
* performer only Reference(USCorePatientProfile | $USCoreRelatedPerson)
```

and add `Alias: $USCoreRelatedPerson = http://hl7.org/fhir/us/core/StructureDefinition/us-core-relatedperson`
to `input/fsh/aliases.fsh` alongside the existing
`USCorePatientProfile` alias. Update the profile `Description` and
`performer` comment to call out CMS-0057-F and HIPAA as the basis for
allowing a personal representative (parent, guardian, healthcare proxy)
to record the opt-out, and add a corresponding entry to
`input/pagecontent/changehistory.md`. This is exactly the change carried
in commit
[`7360463b`](https://github.com/HL7/davinci-epdx/commit/7360463bab8bbba7a1e37ab621ac6f58b0b7b96c)
and reflected by the ticket's `Pre-Applied` label and *Persuasive*
resolution.

#### Justification

The submitter's regulatory rationale is sound: CMS-0057-F explicitly
allows the opt-out to be exercised by a legally recognized personal
representative, and HIPAA's personal-representative concept makes
`RelatedPerson` the natural FHIR resource for that role.
`Consent.performer` in base FHIR R4 already permits a `RelatedPerson`
target; PDex's narrower constraint is what blocks compliant
representation. Widening the constraint (rather than removing it) keeps
the profile expressive about who *can* legitimately record the opt-out
while still excluding unrelated `Practitioner` / `Organization`
performers. Anchoring the expanded target on the US Core
`RelatedPerson` profile is consistent with the rest of the PDex profile
set, which derives from US Core for patient-facing resources. The
commit has already been written, reviewed by the WG path, and applied
in the working clone, so this disposition simply formalises the
existing implementation.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than enumerating `RelatedPerson` directly on
`ProfilePDexConsentProviderSharingOptOut`, define a small extension
(e.g. `Extension-pdex-opt-out-personal-representative`) carrying a
`Reference(USCoreRelatedPerson)` and recording the representative
relationship type, and leave `Consent.performer` constrained to
`USCorePatientProfile`. The extension would be sliced onto
`Consent.extension` with cardinality `0..1` and a binding to a
relationship-type value set (parent, guardian, healthcare proxy). The
`Consent.performer` would continue to identify the patient whose data
is being opted-out, while the extension records who actually exercised
the opt-out on their behalf.

#### Justification

This approach keeps `performer` semantically about "the subject whose
opt-out is being recorded" and uses an extension to capture the
*acting* representative — which some implementers find easier to
process than disambiguating between two different `Reference` targets
on the same element. It also avoids the cross-version question raised
by the briefing's triple US Core pin (`uscore3`/`uscore6`/`uscore7`) by
pinning the extension to a single US Core `RelatedPerson` canonical.
The cost is real, however: it diverges from base FHIR's existing
`Consent.performer` semantics, requires an additional artifact and
binding, and complicates implementer guidance compared with simply
expanding the reference target. It would also be a bigger lift for
servers that already implement `Consent.performer` per the base
resource definition.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that base FHIR R4 `Consent.performer`
already permits `RelatedPerson` and implementers can simply rely on the
base resource's allowed targets when the PDex profile's slice does not
match. Direct submitters to record the personal-representative opt-out
under the base `Consent.performer` rather than through a profile-level
constraint change.

#### Justification

The defensible angle for declining is "no spec change required — the
allowance already exists in base FHIR." However, this rationale is weak
in practice: a published US Realm IG with a profile that *narrows*
`Consent.performer` to `Patient` only will be enforced by validators
and downstream tooling against that profile, regardless of what base
FHIR allows. Implementers building to the PDex Provider Access Consent
profile cannot lawfully carry a `RelatedPerson` performer without the
profile permitting it. Declining therefore leaves a regulatory
compliance gap (CMS-0057-F) that the IG is meant to address, and it
contradicts the WG's existing *Persuasive* resolution and applied
commit.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been adjudicated *Persuasive*, the `Pre-Applied`
label is set, and the corresponding commit
([`7360463b`](https://github.com/HL7/davinci-epdx/commit/7360463bab8bbba7a1e37ab621ac6f58b0b7b96c))
is in the working clone making exactly the change requested. The
regulatory basis (CMS-0057-F + HIPAA personal representatives) is
strong, the change is minimal and surgical (one alias, one constraint
relaxation, a description tweak, and a changehistory entry), and it
does not break any existing valid `Consent` instance — it only widens
the set of valid `performer` targets. Disposition B's extension-based
approach has merit in the abstract but creates a parallel mechanism for
something base FHIR already models on `Consent.performer`, and it would
have to be re-litigated downstream by every implementer; B should be
held in reserve only if the WG identifies a concrete reason `performer`
cannot carry the representative directly. Disposition C cannot be
defended given the IG profile's existing narrowing of `performer`. The
WG should confirm A, with one small follow-up to verify the
`$USCoreRelatedPerson` alias resolves cleanly against the IG's triple
US Core pin (`uscore7` / `uscore6` / `uscore3`).
