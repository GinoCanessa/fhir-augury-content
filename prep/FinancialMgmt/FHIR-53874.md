## Summary

The reporter, on behalf of provider organizations, observes that providers
routinely submit "claims" to state agencies whose purpose is *reporting* of
clinical / encounter activity rather than seeking reimbursement. Today the
FHIR `claim-type` code system (used by `Claim.type` and
`ExplanationOfBenefit.type`) does not have a code that conveys this intent —
the existing codes (`institutional`, `oral`, `pharmacy`, `professional`,
`vision`) are all payment-oriented. The ticket therefore asks the Financial
Management work group to extend `claim-type` with an additional code (or
codes) to represent a *reporting* claim, so that downstream systems can
distinguish reporting submissions from payment submissions on the wire.

There are no cross-references to other Jira tickets, Zulip threads, or
GitHub items in Augury for this ticket. The closest semantically-adjacent
ticket surfaced by keyword similarity is
[FHIR-42876](https://jira.hl7.org/browse/FHIR-42876) (CQI / Da Vinci VBPR),
which addressed the same conceptual need — distinguishing *payment* vs.
*reporting-only* — but in the context of `Measure.relatedArtifact` rather
than `Claim.type`. That ticket was resolved Persuasive by adding a
`contractualStatus` extension with an extensible value set
(`contractual`, `reporting-only`, `contractual-alternate`); the prior art
is relevant when weighing whether to extend the core code system or model
the distinction as an extension here.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new code — e.g. `reporting` (display: "Reporting", definition:
"A submission of clinical/encounter information to a payer or regulatory
agency that is not intended to result in adjudication or payment") — to
the `http://terminology.hl7.org/CodeSystem/claim-type` code system, and
include it in the `claim-type` value set bound to `Claim.type` and
`ExplanationOfBenefit.type` (extensible binding). Update the narrative
on the `Claim` and `EOB` resource pages, and the `claim-type` value
set page, to describe the reporting use case (provider-to-state-agency
non-payment submissions). Provide an example `Claim` instance with
`type = reporting` to illustrate the pattern.

#### Justification

This is the most direct response to the reporter's request and aligns
with how `Claim.type` already partitions the resource by domain. Because
the binding is extensible, downstream profiles (state Medicaid IGs,
public-health-reporting IGs) could already use a local code; promoting
"reporting" to the core code system makes the distinction interoperable
across implementers without each jurisdiction re-inventing it. The
existing codes (`institutional`, `professional`, etc.) describe *what
kind of services were rendered*; a `reporting` code at the same level
describes *the intent of the submission*, which is a different axis but
is the axis the reporter is actually trying to communicate.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** alter `claim-type`. Instead, model the
payment-vs-reporting distinction as a separate concept on `Claim`
itself — preferred form: a new standard extension
`http://hl7.org/fhir/StructureDefinition/claim-submissionPurpose`
(0..1 `CodeableConcept`) bound extensibly to a new value set
`http://hl7.org/fhir/ValueSet/claim-submission-purpose` containing
codes such as `payment`, `reporting`, `pre-authorization`,
`information-only`. Keep `Claim.type` reserved for the
*service-domain* axis it currently describes, and use the new
extension to convey *intent*. Mirror the precedent from
[FHIR-42876](https://jira.hl7.org/browse/FHIR-42876), which added a
`contractualStatus` extension rather than reshaping a core element.

#### Justification

This responds to pknapp's pushback: the reporter is describing a
property of the *submission*, not a *kind of claim*, and conflating
the two on a single element makes both queries and validation harder.
Two axes (service-domain × submission-intent) are cleanly separable,
and an extension is a lower-risk vehicle than expanding a code system
that is already widely implemented and referenced from US-Core / Da
Vinci PAS / CARIN BB. It also matches the precedent the work group
already accepted for the analogous problem on `Measure`. The
extension can be promoted into the core resource at a future
normative cycle if implementer uptake warrants it.

---

### Disposition C: Decline

#### Proposal

Decline the change as proposed (extending `claim-type` with a
"reporting" code). Rationale: out of scope for `Claim.type`, which
classifies the *service domain* of the claim, not the *intent of the
submission*. Direct the reporter to (a) one of the existing
purpose-of-submission mechanisms — e.g. `Claim.use` (which already
distinguishes `claim`, `preauthorization`, `predetermination`) and
profile-level constraints — or (b) the Public Health Reporting work
group, since provider-to-state-agency reporting flows are typically
handled by reporting-specific resources (`MeasureReport`, custom
reporting profiles) rather than `Claim`.

#### Justification

`Claim.use` already exists precisely to indicate the purpose of a
submission, and the reporter's narrative could be served by adding a
`reporting` code there — but even that may be inappropriate, because a
"reporting submission to a state agency" is arguably not a `Claim`
resource at all and forcing it into `Claim` may push implementers
toward semantically-wrong models. The comment thread already raises
this concern. Declining (with a clear pointer to `Claim.use` and to
the reporting resource family) avoids polluting `claim-type` while
still giving the reporter a path forward.

---

### Recommendation

**Recommended disposition:** B

The reporter has identified a real interoperability gap — provider
organizations *are* submitting reporting payloads through claim-shaped
flows today, and there is no on-the-wire signal that distinguishes
those from payment claims. Disposition C is too dismissive: pointing
implementers at `Claim.use` does not actually solve the problem
(`claim`/`preauthorization`/`predetermination` are all
payment-pathway uses), and telling them "use a different resource"
ignores the operational reality surfaced in the ticket. Disposition A
solves the problem but does so on the wrong axis: `claim-type`
classifies *what kind of services* the claim is about, and conflating
that with *why the claim is being submitted* will create downstream
ambiguity for every consumer that already filters or routes on
`Claim.type`. Disposition B preserves the existing semantics of
`claim-type`, follows the precedent the workgroup community already
accepted for the analogous CQI ticket
([FHIR-42876](https://jira.hl7.org/browse/FHIR-42876)), and gives
state-agency and public-health profiles a stable, extensible point
to bind to without breaking existing implementers. The reporter
should still attend the FM conference call (per pknapp's comment) to
confirm the value-set membership and to scope whether
`pre-authorization` and `information-only` belong in the same
extension.
