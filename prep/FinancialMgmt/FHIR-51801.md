## Summary

The submitter requests an enhancement to how the Da Vinci ATR (Attribution
Response) IG handles `Group` resources that carry very large member sets.
Specifically, they propose that when a `Group` resource has more than 2,000
members, the server should transmit it using HTTP `Content-Encoding: gzip`,
in addition to (and complementary with) the existing use of the
`_summary=true` parameter. The intent is to reduce payload size and improve
transmission efficiency for high-volume attribution exchanges between payers
and providers.

No related Jira tickets were surfaced via cross-references. The only
non-Jira context discovered is a Confluence-hosted set of meeting minutes
for the Da Vinci PDex/ATR Implementer Support call on 2025-08-22, where
this ticket was discussed (referenced from a comment by `yanheras`). The
disposition reached on that call — captured in the ticket's only comment —
is that no normative IG change is required, and that the recommendation
should instead be documented as best practice on the IG's Implementer
Support pages, with Nagesh Bashyam to draft content.

## Details

**Description:**

> **Description:**
>
> We propose an enhancement to the ATR Group resource handling mechanism to
> improve performance and efficiency when dealing with large member sets.
> Specifically, we recommend that when the total number of members in a
> Group exceeds **2000**, the resource should be shared using **GZIP
> compression** (`Content-Encoding: gzip`). This change is intended to
> reduce payload size and improve transmission efficiency, especially in
> high-volume data exchange scenarios.
>
> This enhancement is **in addition to** the existing use of the
> `_summary=true` parameter, which provides a concise representation of the
> resource. The combination of `_summary=true` and GZIP encoding will
> ensure that large Group resources are both lightweight and performant.
>
> **Proposed Change:**
> - Introduce logic to apply `Content-Encoding: gzip` when
>   `Group.member.count > 2000`.
> - Ensure compatibility with existing `_summary=true` usage.
> - Update relevant documentation and implementation guides to reflect
>   this behavior.
>
> **Rationale:**
> - Large Group resources can significantly impact performance and
>   bandwidth.
> - GZIP compression is a widely supported and efficient method for
>   reducing payload size.
> - This change aligns with best practices for scalable FHIR API design.
>
> **Impact:**
> - Improves performance for clients consuming large ATR Group resources.
> - Reduces network load and latency.
> - Enhances scalability of implementations using the Da Vinci ATR IG.
>
> Please advise on the appropriate next steps for submitting this
> enhancement for consideration, including any required documentation or
> review processes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the requester's proposal at the IG level: amend the Da Vinci ATR
specification (US Da Vinci ATR FHIR IG) to require that ATR servers, when
returning a `Group` resource whose `Group.member.count` (the count of
member entries — i.e., size of `Group.member`) exceeds 2,000, MUST respond
with `Content-Encoding: gzip`. Document the requirement on the existing
`Group` profile / capability statement page, and clarify that this
behaviour is independent of, and additive to, `_summary=true`. Add
conformance language ("SHALL accept `Accept-Encoding: gzip` … SHALL
respond with `Content-Encoding: gzip` when the threshold is exceeded") and
note client expectations in the IG's "Must Support" / capability section.

#### Justification

This is the most literal answer to the request and gives implementers a
hard, testable conformance bar instead of a soft recommendation. Servers
shipping very large attribution rosters (the scenario the ticket describes
— >2,000 members per `Group`) genuinely benefit from compression, and a
2,000-member threshold is a reasonable cutoff that maps cleanly onto the
real-world payer/provider attribution lists that drive ATR usage.

### Disposition B: Alternative Approach

#### Proposal

Document GZIP usage as **non-normative best-practice guidance** on the
Da Vinci ATR Implementer Support pages (Confluence) rather than as
normative IG conformance. The page should:

- Explain that `Content-Encoding: gzip` is a transport-layer optimisation
  available to any conformant FHIR server and is fully orthogonal to the
  ATR resource model.
- Recommend (not require) that ATR servers negotiate and use gzip when the
  client advertises `Accept-Encoding: gzip`, especially for `Group`
  resources with large `Group.member` arrays (suggested guidance
  threshold: ≳2,000 members).
- Reinforce that `_summary=true` remains the primary IG-level mechanism
  for trimming payload, and that gzip is layered on top.
- Cross-link the guidance from the ATR `Group` profile page so
  implementers can find it.

This matches the disposition recorded on the 2025-08-22 Da Vinci PDex/ATR
call ("no changes are needed to the IG itself, but this should be
documented as best practices in Implementer Support pages") and assigns
Nagesh Bashyam (per that comment) as the drafter.

#### Justification

`Content-Encoding: gzip` is a standard HTTP transport mechanism, not a
FHIR-modelled concept. The FHIR core spec already permits HTTP-level
compression and does not constrain when servers may apply it; mandating it
inside an IG creates a normative obligation about HTTP behaviour that is
better expressed (and tested) at the deployment / operations layer. The
Implementer Support route preserves the practical advice the submitter is
seeking, avoids creating an IG conformance rule that would need a STU
update and FHIR-level testability, and matches what the workgroup actually
agreed to on the 2025-08-22 call.

### Disposition C: Decline

#### Proposal

Mark the ticket as "Not Persuasive" / "Persuasive with Modification — no
spec change". Rationale to be entered in the resolution: HTTP transport
encoding is out of scope for a FHIR Implementation Guide, is already
permitted by the underlying FHIR HTTP layer, and does not require any
artefact change in the Da Vinci ATR IG. Close without spec change.

#### Justification

The literal request — "Introduce logic to apply `Content-Encoding: gzip`
when `Group.member.count > 2000`" — describes server runtime behaviour at
the HTTP layer, not a FHIR resource, profile, operation, or search
parameter. The base FHIR specification already allows servers and clients
to negotiate `gzip` via standard HTTP `Accept-Encoding` /
`Content-Encoding` headers; nothing in the ATR IG forbids this today, and
no implementer is blocked from doing it now. A spec change would therefore
add no new capability and risks confusing readers about what the IG is
actually scoping.

### Recommendation

**Recommended disposition:** B

Disposition B aligns with the explicit outcome already recorded on the
2025-08-22 Da Vinci PDex/ATR Implementer Support call ("no changes are
needed to the IG itself, but this should be documented as best practices
in Implementer Support pages"), with a named drafter (Nagesh Bashyam)
already volunteered. It honours the legitimate operational concern the
submitter raises — large `Group` payloads in real-world attribution
rosters — without inventing a new normative HTTP-layer conformance rule
that the IG is poorly placed to enforce or test (Disposition A's main
weakness). At the same time, it avoids the dismissiveness of a flat
decline (Disposition C), which would leave the submitter without the
implementer-facing guidance they're effectively asking for. The workgroup
should ratify the 2025-08-22 outcome, close this ticket as "Persuasive
with Modification — guidance to be published on Implementer Support
pages", and track the drafting work outside the ballot stream.
