## Summary

The reporter (Lloyd McKenzie) is asking the Financial Management work
group to evaluate whether one or more value sets currently referenced
inline by the CoverageEligibilityResponse resource should be migrated to
THO (terminology.hl7.org) so they can be maintained independently of the
FHIR core release cycle and re-used by other HL7 product families
(CDA, V2, etc.). The ticket calls out one specific binding — the
`datestype` value set (backed by the `DatesEventTypeCodes` code system)
bound to `CoverageEligibilityResponse.event.type` — as a candidate.
Migration is not required, but is recommended whenever the value set is
likely to evolve between FHIR releases or could be useful outside FHIR.
This is part of a larger sweep of analogous "FM-Term" tickets Lloyd has
opened against Financial Management resources (the body is tagged
"Comment 186 - imported by: Lloyd McKenzie", indicating the same

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the `datestype` CodeSystem (`DatesEventTypeCodes`,
canonical currently `http://hl7.org/fhir/datestype`) and its
companion `datestype` ValueSet out of the FHIR core specification and
into THO (`http://terminology.hl7.org/CodeSystem/datestype` and
`http://terminology.hl7.org/ValueSet/datestype`). Update the binding on
`CoverageEligibilityResponse.event.type` (and any sibling references on
other FM resources, e.g. `Claim.event.type`,
`ClaimResponse.event.type`, `ExplanationOfBenefit.event.type`) to point
at the THO ValueSet canonical. Retain a `targetSystem`-style
mapping/redirect in FHIR core for one cycle so existing implementers
aren't broken, and coordinate with Vocabulary and the THO maintainers
on the actual move (PR against the THO build, deprecation note in the
FHIR core resource history).

#### Justification

The reporter's heuristic applies cleanly here: the `event.type` codes
(start/end of coverage, eligibility request period, etc.) are
intrinsically about administrative dates and are not FHIR-specific —
they are exactly the sort of small, slowly-evolving terminology that
CDA and V2 administrative-finance content could re-use, and they are
likely to need additions independently of FHIR R-cycles. THO is the
canonical home for cross-product, slowly-evolving HL7-maintained code
systems, and FM has a standing pattern (FM-Term tickets) of moving
this class of value set out of the core spec.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `datestype` / `DatesEventTypeCodes` in the FHIR core spec for
now, but (a) tighten the binding strength documentation on
`CoverageEligibilityResponse.event.type` (and the matching elements on
`Claim`, `ClaimResponse`, `ExplanationOfBenefit`) so the value set is
clearly identified as the single shared binding across all FM
"event.type" elements, and (b) open a follow-up vocabulary task to
revisit migration once there is concrete demand — for example, a CDA
or V2 work product asking to re-use these codes, or a request to add a
new code that would otherwise have to wait for a FHIR ballot. Track
the deferral on the ticket so it isn't lost.

#### Justification

THO migration is explicitly described in the ticket as
"recommended, not required." The current `datestype` value set is small
and stable, has no known external consumer asking for it, and has been
shipping in FHIR core long enough that moving its canonical URLs is a
non-trivial implementer-facing churn (every conformance resource that
references the binding has to be re-baselined). Deferring keeps that
churn off the table until there is a documented benefit, while still
leaving a paper trail so the migration can be picked up the moment a
real cross-product or rapid-evolution use case appears.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Mod / Not Persuasive" with the
position that `datestype` should remain in FHIR core. The codes are
tightly coupled to the FHIR financial-eligibility workflow
(`CoverageEligibilityResponse.event`), there is no demonstrated CDA / V2
re-use, and the value set is small enough that release-cycle additions
are not a realistic problem. Document the rationale on the ticket so
the question doesn't recur.

#### Justification

A migration is only worth the disruption if (a) the value set is
actually likely to evolve faster than FHIR or (b) other product
families actually want to consume it. Neither has been demonstrated for
`datestype`. Moving canonicals also breaks every IG and conformance
resource that already binds the FHIR-core URL, which is a real cost
borne by implementers. Absent a concrete trigger, "leave it where it
is" is the lowest-risk answer and is consistent with FHIR's general
preference for stability of published canonicals.

---

### Recommendation

**Recommended disposition:** A

The reporter's framing is correct on the merits: `event.type` codes are
generic administrative-date semantics with obvious re-use potential
outside FHIR, and FM already has a workstream (FM-Term) doing exactly
this kind of cleanup. Disposition B's "wait for concrete demand"
position is defensible but tends to leave THO-eligible content stuck in
core indefinitely, and Disposition C understates the cross-product
value of an HL7-maintained, slowly-evolving terminology. The
implementer-churn concern is real but bounded — it can be mitigated by
keeping the FHIR-core canonical alive as a deprecated redirect for one
release cycle, which is the standard playbook FM and Vocabulary have
used for prior THO migrations. Recommend FM agree in principle, hand
the actual move off to Vocabulary/THO, and bundle it with the parallel
FM-Term migrations against `Claim`, `ClaimResponse`, and
`ExplanationOfBenefit` so the binding update lands once across the
financial resources.
