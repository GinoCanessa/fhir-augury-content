## Summary

The ticket questions why the `Contract` resource's state model does not align
with the common workflow / state model pattern that the reporter understood
all FHIR resources should follow. Lisa R. Nelson (reporter) observes that the
Contract resource appears to use its own state vocabulary rather than mapping
to the standard FHIR resource state pattern, and asks the Financial Management
work group to explain or correct the divergence. The single comment from
costateixeira agrees with the concern, distinguishes `Contract.legalState`
(extensible binding, acceptable) from `Contract.state` (required binding,
problematic), and suggests that `Contract.state` should reflect the workflow
status of the request or action — consistent with the FHIR workflow pattern —
rather than enumerating jurisdiction-specific contract lifecycle states. No
other Jira tickets are cross-linked; the only structured cross-reference is to
the FHIR artifact `Contract.state` itself.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Realign `Contract.state` with the FHIR resource workflow / request pattern.
Concretely:

- Change the binding strength on `Contract.state` from **required** to a
  workflow-aligned value set drawn from the common state machinery used by
  request-pattern resources (`fm-status` / `request-status` family — the same
  pattern used by `Claim`, `CoverageEligibilityRequest`, etc. in Financial
  Mgmt), so that `Contract.state` describes the lifecycle of the
  request/instance (e.g., `draft | active | cancelled | entered-in-error`)
  rather than substantive contract lifecycle terminology.
- Move any existing required-binding contract-lifecycle codes (`executed`,
  `negotiable`, `policy`, etc.) into `Contract.legalState` (already
  extensible) where jurisdiction-specific extension is expected.
- Update the Contract resource's state diagram and prose in the FM Contract
  module to point readers at the standard FHIR workflow state machine, and
  document the mapping from the prior `Contract.state` codes to either the
  new workflow status codes or to `Contract.legalState`.

#### Justification

This is what the reporter and the only commenter explicitly asked for:
"`Contract.state` should show what is the status of the request or action, not
the resulting state of the contract." Aligning with the workflow pattern
brings Contract into consistency with other request-pattern resources that
implementers already know, removes the implementer burden of supporting a
required binding whose codes are jurisdiction-sensitive, and reserves
substantive contract-lifecycle modeling for `Contract.legalState`, whose
extensible binding is appropriate for that variability. The normative
expectation that "all FHIR resources map to a standard state model" is the
core motivation — non-alignment makes Contract an outlier in FHIR Core.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Contract.state` as a contract-domain status, but **relax the binding
strength** from required to **preferred** (or extensible), and **add a
separate** `Contract.status` element that carries the standard request /
workflow status codes used elsewhere in FHIR. Specifically:

- Introduce `Contract.status` (1..1, required binding to the canonical
  request status / fm-status value set) as the workflow alignment point.
- Reduce `Contract.state` to 0..1 with a relaxed binding so existing
  domain-specific contract lifecycle terminology can still be expressed but
  is not gated by a fixed required value set.
- Document `Contract.status` as the element that fulfills the "common state
  model" expectation, and `Contract.state` / `Contract.legalState` as
  domain-flavor descriptors.

#### Justification

This preserves backward compatibility for implementers already populating
`Contract.state` (no codes are removed or relocated, only the binding is
loosened), while still answering the alignment concern by introducing a
properly-bound `Contract.status`. It mirrors the dual-status pattern used in
several other resources where a workflow status coexists with a
domain-specific status. The trade-off vs. Disposition A is that the resource
gains an additional element and implementers must populate two status-like
fields, which can be confusing; however, it minimizes breaking change risk
for the ~6+ years that `Contract.state` has existed and avoids the migration
cost of relocating codes to `legalState`.

---

### Disposition C: Decline

#### Proposal

Decline the request to realign `Contract.state` with the common FHIR
workflow state model. Document in the Contract resource that the state model
is intentionally domain-specific (legal/contractual lifecycle) and is not
intended to participate in the request/workflow pattern, citing the
heterogeneity of contract lifecycles across jurisdictions and the explicit
modeling separation between operational status and legal effect.

#### Justification

This disposition is defensible only if the FM work group concludes that the
Contract resource is not a request-pattern resource and that forcing
alignment with the workflow state machine would distort the resource's
semantics. The community discussion already pushes back against this
position (costateixeira: required binding "may render the resource unusable
in some cases"), so a Decline would need to be paired with strong design
rationale and likely a documentation overhaul to explain the divergence.
This is the weakest of the three options given the available evidence.

---

### Recommendation

**Recommended disposition:** A

Both the reporter and the only commenter converge on the same diagnosis: the
required binding on `Contract.state` does not align with the FHIR workflow
state pattern and creates real implementer pain, especially across
jurisdictions. Disposition A directly fixes the divergence the ticket calls
out and preserves a place (`Contract.legalState`, already extensible) for
domain-specific lifecycle terminology, which is the right home for the kind
of jurisdictional variability costateixeira describes. Disposition B is a
defensible compromise if the FM work group is uncomfortable with the
migration impact of relocating codes, but it leaves implementers with two
overlapping status elements and only partially answers the original concern.
Disposition C should only be selected if FM has design rationale (not
present in this ticket's discussion) for treating Contract as outside the
workflow pattern. Given the ticket has been open since 2019 with no
counter-argument recorded, A is the cleanest path to closure and brings
Contract into line with the rest of FHIR Core.
