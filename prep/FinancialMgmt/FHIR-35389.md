## Summary

The reporter requests that the C4BB IG (US CARIN Blue Button) stop minting
its local CodeSystem `C4BBAdjudication` under the `hl7.org/fhir` canonical
namespace, and instead register the code system at `terminology.hl7.org`
(THO). The ticket explicitly asks that a companion UTG ticket be filed so
the code system is registered in THO and then referenced from C4BB rather
than defined inline. The ticket carries the `Terminology` label and was
included in CARIN BB STU2 Block-Vote-5 (2022-07-12), where it was
originally dispositioned **Considered for Future Use** (7-0-2: Corey
Spears / Rachel Foerster) on the rationale that the change is entangled
with broader rework of `ExplanationOfBenefit` and its associated binding
types. That resolution was subsequently **reverted post‑STU2 publication**
(2022-11-28, lynn_laakso) and the ticket re-opened to `Triaged` so the
work group can revisit it. There are no other Jira tickets cross-linked
to this issue in Augury, but it is one of a family of sibling C4BB
"Defining URL should be anchored in THO" tickets included in the same
block vote (e.g., FHIR-35394, FHIR-35400, FHIR-35408, FHIR-35420,
FHIR-35423) — all reported by Joan Harper with similar rationale.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move `CodeSystem-C4BBAdjudication` out of the C4BB IG and register it in
THO. Concretely:

1. File a companion UTG ticket against `HL7/UTG` to add the
   `C4BBAdjudication` code system at a THO-anchored canonical (e.g.,
   `http://terminology.hl7.org/CodeSystem/C4BBAdjudication`), preserving
   the existing concept set, codes, and display values.
2. In the next C4BB publication, remove the locally-defined
   `CodeSystem/C4BBAdjudication` resource (and its `hl7.org/fhir/us/carin-bb`
   canonical), and update every ValueSet, profile binding, and example in
   C4BB that currently refers to the IG-local canonical to point at the
   THO canonical instead.
3. Add a release-note entry flagging the canonical change as a breaking
   change for downstream implementers and provide a brief migration
   note (old → new canonical mapping).

#### Justification

This is the literal request and aligns with the standing HL7 terminology
governance pattern: shared, cross-IG code systems belong in THO so they
can be versioned, governed, and reused outside the IG that first needed
them. The `Terminology` label and Joan Harper's broader sweep of
sibling tickets (FHIR-35394, -35400, -35408, -35420, -35423, …) make
clear this is part of a programme-level cleanup, not a one-off. Doing it
correctly via a UTG ticket means the C4BB IG ends up consuming
terminology rather than redefining it, which is what THO exists to
support.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the canonical move until the broader CARIN/FM rework of
`ExplanationOfBenefit` and its adjudication-related bindings completes,
but in the interim:

1. **Do not** change the canonical of `CodeSystem-C4BBAdjudication` in
   the published C4BB IG (avoid a breaking-canonical change that would
   need to be repeated when the EoB rework lands).
2. Open the UTG ticket *now* to reserve the THO canonical
   (`http://terminology.hl7.org/CodeSystem/C4BBAdjudication`) and seed
   THO with the current concept set, marked as the future home.
3. Add a non-normative note to the C4BB IG page for
   `CodeSystem-C4BBAdjudication` stating that the canonical is planned
   to move to THO and citing the reserved THO URL, so implementers are
   forewarned.
4. Re-evaluate the actual canonical migration in the C4BB release that
   follows the EoB binding-type rework, where it can be done once,
   coordinated with related ValueSet binding changes.

#### Justification

This is essentially the rationale the work group adopted in the original
2022-07-12 vote ("re-evaluation of the model of the underlying resource
(EoB) and the associated binding types is necessary"). The advantage is
that it avoids two churn cycles for implementers: a canonical change
now, followed by binding changes later. It still does the right thing on
the THO side (registers the code system) so the cleanup is teed up,
without forcing a breaking change to a published IG before the broader
work converges. The trade-off vs. A is timing, not direction.

---

### Disposition C: Decline

#### Proposal

Re-close as **Considered for Future Use** (the original 2022-07-12
disposition) on the basis that the C4BB IG has already published STU2
with the IG-local `CodeSystem-C4BBAdjudication` canonical, and the
broader EoB / adjudication binding work that motivated keeping it
in‑IG has not yet completed. The change should not be adopted as a
standalone fix in the next minor release of C4BB.

#### Justification

A canonical change is a breaking change for every implementer that has
already coded against the published `hl7.org/fhir/us/carin-bb`
canonical for STU2. Doing it as an isolated fix — separate from the
ExplanationOfBenefit binding rework that the work group itself called
out — risks two breaking changes back-to-back without delivering any
new functional value. The original 7-0-2 vote already weighed this and
concluded the cost was not justified at that time.

---

### Recommendation

**Recommended disposition:** B

The literal request in Disposition A is correct in principle and aligns
with HL7 terminology governance, but executing it standalone before the
underlying EoB / adjudication binding rework completes risks subjecting
implementers to back-to-back breaking changes — exactly the concern
that drove the original "Considered for Future Use" vote. Disposition C
preserves that protection but defers all action and leaves the THO
registration unowned, which is fragile (the work tends to never
restart). Disposition B captures the best of both: it commits the work
group to the right end-state (THO-anchored canonical, sourced via UTG)
and reserves the canonical now, but sequences the actual canonical flip
in C4BB to land alongside the EoB binding rework so implementers
absorb a single coordinated change. Because FHIR-35389 is one of a
family of sibling tickets in Block-Vote-5, the work group should
consider applying the same B-style treatment to all of them so the
future C4BB release that accepts the EoB rework can also retire the
IG-local code systems and value sets in one batch.
