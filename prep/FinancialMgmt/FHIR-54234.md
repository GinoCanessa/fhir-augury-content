## Summary

The reporter requests that `Coverage.class.value` be allowed as a
`CodeableConcept` in addition to `Identifier`, by changing the element to a
choice type. Today `Coverage.class.value` is mandatory and constrained to
`Identifier`, which suits administrative identifiers but is unnecessarily
restrictive when the classification is governed by a shared, nationally
managed code system. The Austrian "insured category" is offered as a concrete
example: it is defined by a nationwide social-insurance code system intended
to be used consistently across all payers and systems, and modeling it as an
`Identifier` (a) prevents validation of `identifier.value` against the
underlying code system, (b) forces implementations to misuse
`identifier.system` as a pseudo-code system, and (c) prevents standard
terminology services from validating the value. The ticket has been triaged
**Persuasive with Modification**: rather than introducing a `value[x]` choice,
the proposed resolution is to make `class.value` optional (`0..1`), add a new
sibling `class.code` (`0..1` `Coding` with an example binding such as Gold /
Silver / Bronze), and add a constraint that at least one of `class.value` or
`class.code` is present. No other Jira tickets are cross-referenced. The
Augury cross-reference index links the ticket only to the FHIR artifact
`Coverage.class.value` itself.

## Details

**Description:**

Allow `Coverage.class.value` to be expressed as a CodeableConcept in addition
to Identifier. Change the type of `Coverage.class.value` from Identifier to a
choice type.

`Coverage.class.value` is mandatory and constrained to the data type
Identifier. While this is suitable for some administrative identifiers, it is
unnecessarily restrictive for use cases where the classification is based on
a shared, nationally governed code system rather than an identifier.

In Austria, for example, the insured category is defined by a nationwide,
social-insurance-spanning code system that is intended to be used
consistently across all payers and systems.

Modeling such a classification as an Identifier leads to the following
issues:

- The `identifier.value` cannot be meaningfully validated against the
  underlying code system.
- Implementations are forced to misuse `identifier.system` as a pseudo-code
  system.
- Tooling and terminology services cannot apply standard code validation.

**Resolution Description (Persuasive with Modification):**

In order to preserve forward compatibility:

- Change `class.value` cardinality from `1..1` to `0..` (i.e., `0..1`).
- Add `class.code` `0..1` `Coding` with an example binding (e.g., Gold,
  Silver, Bronze).
- Add a constraint that either `.value` or `.code` is required.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change `Coverage.class.value` from a single `Identifier` to a choice type:
`Coverage.class.value[x]` of type `Identifier | CodeableConcept`, keeping
cardinality `1..1`. The Austrian "insured category" use case (and any other
nationally governed classification) would then bind to a `CodeableConcept`
form with the appropriate value set, while existing payer / group identifier
use cases continue to use the `Identifier` form unchanged.

#### Justification

This is the literal request and it is FHIR-idiomatic: choice types are the
standard mechanism in FHIR when an element legitimately needs to carry either
a coded concept or an identifier (cf. `Observation.value[x]`,
`MedicationRequest.medication[x]`). It addresses every issue the reporter
raised — `CodeableConcept` allows terminology-service validation against a
governed code system, removes the need to misuse `identifier.system` as a
pseudo-code system, and makes the modeling intent explicit in the resource
itself rather than in implementer convention.

---

### Disposition B: Alternative Approach (the "Persuasive with Modification" resolution)

#### Proposal

Rather than introducing a `value[x]` choice on the existing element, evolve
`Coverage.class` to carry both an identifier-style value and a coded value
side-by-side, in a way that preserves R4/R5 forward compatibility:

- Relax `Coverage.class.value` cardinality from `1..1` to `0..1` (type
  unchanged: `Identifier`).
- Add a new sibling element `Coverage.class.code` `0..1` of type `Coding`,
  with an example binding (e.g., a "Coverage Class Code" example value set
  including Gold / Silver / Bronze and similar tiers).
- Add an invariant on `Coverage.class` requiring that at least one of
  `value` or `code` be present (e.g.,
  `value.exists() or code.exists()`).

This is exactly the resolution recorded on the ticket and matches the
direction Paul Knapp committed to in the implementer Zulip thread.

#### Justification

This avoids the `value[x]` rename, which is a renaming/breaking change for
existing R4 instances and tooling that already serialize `Coverage.class.value`
as an `Identifier`. By keeping `value` in place and adding `code` as a
sibling, both R4-shaped instances and new code-based instances remain
representable, and the new invariant ensures `class` is never empty — which
also incidentally cleans up the long-standing implementer complaint (see the
Zulip thread) about `1..1` on `class.value` being too strict for cases where
only a name is known. Using `Coding` (rather than `CodeableConcept`) for
`class.code` is consistent with other coded administrative classifiers in
Coverage and keeps the element size bounded; the example binding signals that
the value set is a starting point for jurisdictions to refine.

---

### Disposition C: Decline

#### Proposal

Decline the request and instruct implementers to continue using
`Coverage.class.value` as an `Identifier`, optionally setting
`identifier.system` to the canonical URL of the governing code system and
`identifier.value` to the code. National classifications that need richer
terminology semantics can be carried via a profile-defined extension on
`Coverage.class` (or by promoting the classification to a separate element
such as `Coverage.type` if it represents the coverage as a whole rather than
a sub-class).

#### Justification

The defensible grounds for declining would be (a) `Coverage.type` and
`Coverage.class.type` already provide a coded classification of the coverage,
so adding another coded slot risks overlap, and (b) treating a code as an
`Identifier` with `system` + `value` is an established FHIR pattern and does
not strictly *prevent* terminology validation if implementers agree on a
convention. However, both points are weak: `Coverage.type` and
`Coverage.class.type` classify the kind of class (e.g., "group", "plan",
"subgroup"), not the specific value of that class, and using `Identifier` to
smuggle codes is precisely the misuse the reporter calls out and is widely
recognized as an anti-pattern.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the resolution already recorded on the ticket
("Persuasive with Modification") and is the correct trade-off. It addresses
the reporter's underlying need — letting `Coverage.class.value` carry a
governed code rather than only an identifier — while sidestepping the
forward-compatibility cost of a `value[x]` rename (Disposition A), which
would break every R4 instance and downstream tool that currently serializes
`Coverage.class.value` as an `Identifier`. It also cleans up a separate,
long-standing implementer pain point captured in the
`#implementers > Should Coverage.class.value be 0..1?` thread, where the
`1..1` cardinality on `class.value` was already flagged as too strict and
where Paul Knapp had previously committed to relaxing it in R5. Disposition C
is not defensible: the reporter's critique that modeling a governed
classification as `Identifier` blocks terminology validation and forces
misuse of `identifier.system` is correct, and "use an extension" is a poor
answer when the use case is common enough to deserve a first-class element.
The workgroup should adopt the three-part resolution (relax `value` to
`0..1`, add `class.code` `0..1` `Coding` with an example binding, add the
"value or code" invariant) and apply it under the existing
`FM-R5+_ToBeApplied` label.
