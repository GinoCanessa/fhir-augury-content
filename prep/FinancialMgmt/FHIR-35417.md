## Summary

The reporter asks that the canonical defining URL for the
`C4BBClaimProcedureType` ValueSet (currently published under the CARIN BB
IG at `hl7.org/fhir/...`) be anchored in `terminology.hl7.org` (THO)
instead. The ticket also asks the workgroup to file a corresponding UTG
ticket so the value set is registered in THO. The broader rule the
reporter is invoking is that only value sets with a required binding to
a `code` data type should remain anchored in `hl7.fhir.org`; everything
else (including this CARIN BB value set) belongs in THO. No related
Jira tickets surface from cross-references; the single Zulip reference
is the CARIN BB STU2 Block-Vote-6 announcement that bundled this ticket
with 25 others. The previous disposition (Considered for Future Use,
vote 13-0-0 on 2022-07-26) tied resolution to UTG ticket
[UP-316](https://jira.hl7.org/browse/UP-316), which proposes a
mechanism for "all codes" value sets in THO; that disposition was
reverted post-STU2 and the ticket re-opened.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive with Modification: move the value set
`C4BBClaimProcedureType` out of the CARIN BB IG and re-anchor it under
THO. Concretely:

1. File (or link) a UTG ticket to register the value set in THO at the
   canonical URL `http://terminology.hl7.org/ValueSet/C4BBClaimProcedureType`,
   carrying forward the same intensional definition (all codes in the
   external HIPAA Claim Procedure Type code system).
2. In the next CARIN BB IG release, remove the IG-local
   `ValueSet/C4BBClaimProcedureType` definition and update every
   binding (e.g., on the C4BB EOB profiles' procedure-type element)
   to reference the THO canonical URL.
3. Coordinate the IG publication so the THO registration lands first
   (so binding consumers don't dangle), and add a release note that
   the canonical URL has changed.

#### Justification

This is exactly what the reporter requested and what the workgroup
already endorsed in the 2022-07-26 block vote (13-0-0). It also aligns
with the standing FHIR/HL7 policy that value sets — other than those
required-bound to a `code` datatype in core FHIR — should live in THO,
not under `hl7.org/fhir/...`. Doing the move now (rather than waiting
on UP-316's generic "all codes value set" mechanism) avoids a
multi-year wait and removes a maintenance footgun where two copies of
the same value set could drift.

---

### Disposition B: Alternative Approach

#### Proposal

Considered for Future Use, gated on UP-316: keep the IG-local
`C4BBClaimProcedureType` value set in place for the current STU2.x
maintenance cycle, but mark the ticket as dependent on
[UP-316](https://jira.hl7.org/browse/UP-316). When UTG delivers the
generic "all codes value set" mechanism, the IG-local copy is removed
and bindings are retargeted to the THO value set produced by that
mechanism — not a hand-authored THO value set. Add a note in the IG
narrative that the value set is provisional pending THO uptake.

#### Justification

This is the disposition the workgroup originally adopted on
2022-07-26. It avoids hand-creating a THO value set that will later be
superseded by an automatically-generated one (which would create a
canonical URL churn for implementers). It also respects the boundary
between FM-WG and Vocab/UTG: value-set authoring in THO is owned by
Vocab, and this approach lets that infrastructure drive the change
rather than CARIN BB doing a one-off. The trade-off is timing — UP-316
has been open for years with no firm delivery date, so implementers
continue to consume the `hl7.org/fhir/...` URL in the meantime.

---

### Disposition C: Decline

#### Proposal

Not Persuasive: keep `C4BBClaimProcedureType` defined in the CARIN BB
IG at its current `hl7.org/fhir/...` canonical URL. Close as Not
Persuasive on the grounds that the canonical URL is part of the
already-published STU2 normative surface, that implementers are
relying on it, and that no functional defect is alleged — the request
is purely about where the artifact is hosted.

#### Justification

A canonical URL change is a breaking change for any conformance
artifact, profile, or implementation that references the value set by
URL. Without a concrete implementer complaint or a coordinated UTG
landing path, moving the URL imposes migration cost on the CARIN BB
ecosystem in exchange for a hosting/policy improvement that has no
runtime effect. The IG can document, non-normatively, that the value
set will eventually move to THO, without taking the breaking change
in this cycle.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already once decided this should happen (the 13-0-0
block vote), and the only reason it did not land is that the
resolution was deferred to UP-316 and then reverted post-STU2.
UP-316 has not progressed, so Disposition B amounts to leaving the
ticket open indefinitely while implementers continue to consume a
canonical URL the workgroup itself agrees is in the wrong place.
Disposition C is hard to defend on policy grounds: the
THO-vs-`hl7.org/fhir` rule is well established, and this value set
(an "all codes" wrapper around an external HIPAA code system) is the
textbook case for THO hosting. Disposition A is the directionally
correct change; the migration burden is real but bounded (one value
set, a known set of bindings on C4BB EOB profiles), and can be
mitigated with a clear release note and a deprecation period in which
both URLs are documented. The workgroup should pair this with a UTG
ticket so Vocab is on the hook for the THO registration before the
next CARIN BB publication.
