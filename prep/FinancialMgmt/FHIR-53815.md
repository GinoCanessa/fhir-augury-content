## Summary

A Patient/Member opt-out from Provider Access data sharing through the Da
Vinci PDex Provider Access Bulk API is currently expressed by a `Consent`
resource carrying a `provision.type = "deny"`. Discovering those opt-outs
today requires servers (and downstream Provider Access bulk-export
implementers) to evaluate `Consent.provision` semantics with no
purpose-built search axis, which is awkward for both Payers and Provider
Access actors that need to filter members in/out of a bulk job
membership group.

The ticket asks PDex (US Da Vinci PDex, sponsored by Financial
Management) to (1) define a new extension — provisionally named
`ProviderAccessUseCase` — that explicitly tags a `Consent` (or related
artifact) as governing Provider Access data sharing, and (2) publish a
companion `SearchParameter` so that servers can return a member's
Provider Access opt-out via a single, well-known query. The Jira
resolution ("Persuasive") and resolution description ("Add Extension and
search parameter as proposed.") indicate the work group has already
agreed in principle; this prep note frames the disposition for ballot
discussion. No related Jira tickets, Zulip threads, or substantive
GitHub PRs were surfaced — the only GitHub mentions are PDex
change-history bookkeeping entries.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/davinci-epdx`, on the active pre-publish STU 2.1 branch:

1. Define a new extension in `input/fsh/Extensions.fsh` (or in a new
   `Extensions-ProviderAccess.fsh` if a topical split is preferred):
   - `Extension: ProviderAccessUseCase`
   - `Id: provider-access-use-case`
   - `Title: "Provider Access Use Case"`
   - `Description`: identifies that a `Consent` (or `Consent.provision`)
     governs the Da Vinci PDex Provider Access Bulk data-sharing use case.
   - `Context: Consent` (consider also
     `Context: Consent.provision` if granular per-provision tagging is
     in scope).
   - `* value[x] only Coding` (or `CodeableConcept`) bound to a new
     PDex `ProviderAccessUseCase` ValueSet (PDex-defined CodeSystem)
     covering at minimum a `provider-access` concept; binding strength
     `extensible` to allow future use cases.
2. Add a `SearchParameter` for `Consent` so a member's Provider Access
   opt-out can be retrieved in a single query. Follow the existing PDex
   precedent and add it as a hand-authored
   `SearchParameter-pdex-consent-provider-access-optout.json` under
   `input/resources/` (matching the
   `SearchParameter-pdex-medicationdispense-*.json` pattern), with:
   - `code = provider-access-optout`
   - `base = [Consent]`
   - `type = token`
   - `expression =
     Consent.extension('http://hl7.org/fhir/us/davinci-pdex/StructureDefinition/provider-access-use-case').value
     where Consent.provision.type = 'deny'`
     (or split into a token-on-extension + composite if simultaneous
     filtering on `provision.type` is desired).
   - Mark `_stu = 2.1` / appropriate maturity.
   *Do not also* register the same canonical from
   `SearchParameters.fsh` — that would trip the duplicate-canonical
   gotcha called out in the briefing.
3. Reference the new SearchParameter from
   `pdex-payer-access-server-capability-statement.fsh` and
   `pdex-provider-access-server-capability-statement.fsh` so that
   conformance is explicit on both server roles.
4. Add a `Consent` example (`ExampleConsentProviderAccessOptOut.fsh`)
   tagged with the extension and `provision.type = #deny`, plus a
   companion narrative section in `input/pagecontent/` describing the
   tagging convention and the search query, and roll FHIR-53815 into
   the next entry of `input/pagecontent/changehistory.md`.

#### Justification

The work group already marked the ticket **Persuasive** with the
resolution "Add Extension and search parameter as proposed", so an
"Accept As Requested" disposition is aligned with the recorded
disposition. The PDex repo's flat `input/fsh/` layout and the
`Extensions.fsh` + `SearchParameters.fsh` (or
`input/resources/SearchParameter-*.json`) recipes from the briefing
make this an additive STU 2.1 change with no breaking impact on the
three pinned US Core baselines. Tagging the consent explicitly avoids
forcing every server to encode "Provider Access opt-out" semantics into
ad-hoc combinations of `Consent.scope`, `category`, and
`provision.type`, and a dedicated SearchParameter gives Provider Access
bulk-export implementers a single, predictable filter for building
the export Group membership.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a PDex-defined `ProviderAccessUseCase` extension, lean on
existing Consent metadata and ship a SearchParameter only:

1. Recommend in narrative (new section in `input/pagecontent/`) that
   PDex Provider Access opt-out `Consent` resources be expressed using:
   - `Consent.scope` bound to an existing patient-privacy code
     (`http://terminology.hl7.org/CodeSystem/consentscope#patient-privacy`),
   - `Consent.category` populated with a PDex-defined concept (in
     `CodeSystems.fsh`/`ValueSets.fsh`) named e.g.
     `pdex-provider-access-optout`, and
   - `provision.type = #deny`.
2. Add a single `SearchParameter` in `input/resources/`
   (`SearchParameter-pdex-consent-provider-access-optout.json`) of
   `type = token` over `Consent.category`, named
   `provider-access-optout`, that filters by the new PDex category
   code.
3. Update both Provider/Payer Access CapabilityStatements to require
   the new SearchParameter, and add an `ExampleConsentProviderAccessOptOut`
   plus a `changehistory.md` row.

#### Justification

This keeps the PDex profile surface narrower: no new
`StructureDefinition/Extension` to maintain, version, or cross-reference
from HRex / Plan-Net, and it leverages `Consent.category` — the FHIR
core element specifically designed to classify what a consent is
about. It also avoids the briefing's mixed-SearchParameter gotcha by
working with a stock element path (`Consent.category`) rather than an
extension expression, which historically expands index support across
servers and makes the SearchParameter implementable by IG-naive
endpoints. The trade-off versus Disposition A is loss of the explicit
`use-case` value-set seam — future PDex use cases (e.g. payer-to-payer
opt-out) would need additional category codes rather than an additional
extension `value[x]` coding, but the Da Vinci suite already follows
that pattern in several other IGs.

---

### Disposition C: Decline

#### Proposal

Decline the new extension and SearchParameter. Document in narrative
that Provider Access opt-out is already discoverable today via
`Consent?status=active&category=<patient-privacy>&_filter=provision.type
eq deny` (or the equivalent chained search using existing PDex
categories), and direct implementers to filter on the existing
`Consent.provision.type = deny` semantics during bulk-job Group
construction. No new artifacts are added; only the narrative is
clarified.

#### Justification

A "decline" position is defensible only on the grounds that
`Consent.provision.type = deny` plus existing `category` / `scope`
codes are technically sufficient and that the cost of minting a
PDex-specific extension + SearchParameter (long-term maintenance,
duplicate-canonical risk highlighted in the briefing, additional
USCDI-version validation across the three pinned US Core baselines)
outweighs the implementer convenience. This rationale is weak in this
case because the Jira record already shows a "Persuasive" disposition
and explicit author intent; the decline disposition is included for
completeness so the work group can articulate why they did **not**
choose it.

---

### Recommendation

**Recommended disposition:** A

The Jira ticket is already resolved as **Persuasive** with a resolution
description ("Add Extension and search parameter as proposed.") that
matches Disposition A almost verbatim, and the original commenter
(`ekivemark`) explicitly named the `ProviderAccessUseCase` extension
and the companion search parameter. Disposition A delivers exactly
that: an extension that names the Provider Access use case unambiguously
and a token SearchParameter on `Consent` that lets Provider Access bulk
implementers filter member opt-outs in one round-trip. The PDex repo
already has the recipes (briefing's "Add an extension (FSH)" and "Add a
SearchParameter") and follows a flat `input/fsh/` layout, so the
change is mechanically straightforward and additive against the
STU 2.1 stream (label `STU2.1-upate`). The only meaningful execution
risk — duplicate-canonical SearchParameter registration across
`SearchParameters.fsh` and `input/resources/` — is fully mitigated by
choosing a single home, which Disposition A does (JSON under
`input/resources/`, matching existing PDex precedent). Disposition B is
the natural fallback if the work group wants to avoid minting another
PDex extension, but it would require deviating from the recorded Jira
resolution and forces all future Provider Access use-case distinctions
through ValueSet expansion rather than a typed extension seam.
