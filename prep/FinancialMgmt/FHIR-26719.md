## Summary

The ticket asks the Financial Management work group to define a "Provider
Network Contracting Status" data element — indicating whether the
provider has a contract with the plan (values `contracted | non-contracted`)
as of the effective date of service or admission — within the **adjudication**
section of the claims-family resources (Claim, ClaimResponse, and
ExplanationOfBenefit), rather than carrying it as `supportingInfo`. The
reporter notes that today the CARIN Blue Button IG conveys network
contracting status for several provider roles (billing, attending,
referring, performing, PCP, site) via `supportingInfo` slices, and is
asking that the concept be elevated into the adjudication structure where
financial outcomes are reported.

The ticket was originally filed against the CARIN BB IG but was
reclassified by Corey Spears (2025-10-30) as a FHIR Core request, which is
why it now sits under Financial Mgmt against the FHIR Core specification.
A previous resolution of "Considered for Future Use" (2020-07-21, vote
18-0-5) was reverted by Lynn Laakso post-STU2 publication and the ticket
re-opened. No cross-referenced Jira tickets, Zulip threads, or GitHub
items were surfaced for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new coded data element to the **adjudication** backbone of
`ExplanationOfBenefit`, `ClaimResponse`, and (where appropriate) `Claim`
that captures provider network contracting status. Concretely:

- Define a new `adjudication.category` concept code in the
  `adjudication` value set / code system (e.g.,
  `http://terminology.hl7.org/CodeSystem/adjudication#contracting-status`)
  with display "Provider Contracting Status".
- Define a companion `ValueSet` of allowed reason codes
  `contracted | non-contracted` (e.g.,
  `http://terminology.hl7.org/CodeSystem/contracting-status`).
- When the status applies to a specific provider role rather than the
  claim as a whole (billing, attending, referring, performing, PCP,
  site), use `adjudication.reason` to carry the role and
  `adjudication.category` to carry the contracting-status concept, so
  one `adjudication` entry per role is emitted at the item level (or at
  the header for billing).
- Update the `EOB.adjudication`, `ClaimResponse.item.adjudication`, and
  (if scope agrees) `Claim.item` definitions and examples to show the
  new code in use, and add narrative on the resource pages
  (`claim.html`, `claimresponse.html`, `explanationofbenefit.html`)
  pointing to the new value set.

#### Justification

This honors the reporter's literal request: contracting status is a
financial outcome that drives member liability, network steerage, and
benefit calculation, so it belongs alongside other adjudication
outcomes (allowed amount, deductible, coinsurance) rather than being
buried in `supportingInfo`. Modeling it via `adjudication.category` +
`reason` reuses existing FHIR machinery, keeps the change additive
(new concept codes, no structural break), and lets implementers who
already populate per-role provider information continue to do so by
emitting one adjudication line per role. It also gives the CARIN BB
IG a clean upgrade path: today's `supportingInfo`-based slices can be
re-anchored to the new core element in a future BB version.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new core adjudication concept. Instead, define a single
HL7-published **standard extension**
`http://hl7.org/fhir/StructureDefinition/eob-provider-contracting-status`
of type `CodeableConcept` (bound `required` to a
`contracted | non-contracted` value set), with the following anchor
points:

- On `ExplanationOfBenefit.careTeam` and
  `ExplanationOfBenefit.provider` (and the equivalents on
  `Claim.careTeam`/`Claim.provider`) so the status is attached to the
  specific provider role it describes.
- Optionally on `EOB.item.careTeamSequence`-referenced careTeam entries
  for line-level granularity.

Publish guidance (in the EOB resource page and the Financial Mgmt
section of the IG ecosystem) that this extension is the canonical way to
express network contracting status, deprecating the `supportingInfo`
pattern in CARIN BB.

#### Justification

The contracting-status concept is fundamentally a property *of a
provider role on a claim*, not of an adjudication line item. Anchoring
it to `careTeam`/`provider` keeps the data model semantically clean and
avoids overloading the `adjudication` backbone — which is shaped around
monetary amounts and benefit categories — with a non-monetary
classification. A standard extension is also a lower-risk change than
modifying the core adjudication value set: it can be normatively
published, tooling already understands it, and Da Vinci / CARIN
implementations can adopt it without a breaking change. The trade-off
versus Disposition A is reduced visibility (extensions are easier for
consumers to miss than first-class adjudication categories) and the
need for IG authors to coordinate retirement of the existing
`supportingInfo` slices.

---

### Disposition C: Decline

#### Proposal

Decline the change. Close as "Persuasive with Mod / No Change" with the
position that **the current `supportingInfo` mechanism in CARIN BB
already meets the requirement** at the IG layer where it belongs. FHIR
Core's claims resources are intentionally generic and US-payer-specific
concepts (in-network contracting status with a binary contracted /
non-contracted vocabulary) should remain in the implementation guides
that scope them — primarily CARIN BB and Da Vinci PDex — rather than
being promoted into core adjudication.

#### Justification

The original 2020 disposition ("Considered for Future Use", 18-0-5) was
already a soft decline, and the reasons still stand: contracting status
is jurisdiction- and business-arrangement-specific, the
contracted/non-contracted dichotomy is a US-payer simplification (other
markets have tiered networks, capitation, etc.), and `supportingInfo`
with a slice per provider role is exactly the extensibility surface FHIR
Core provides for IG-specific data. Adding a core adjudication category
risks codifying a US-centric concept into a global resource, and
publishing a HL7-level extension (Disposition B) duplicates work CARIN
BB has already done. If the underlying complaint is discoverability,
the right fix is to improve the CARIN BB documentation, not to
restructure core EOB.

---

### Recommendation

**Recommended disposition:** B

The reporter's actual pain point — that network contracting status is
hard to find when it sits inside a `supportingInfo` slice — is real,
and Disposition C's "leave it in the IG" answer doesn't address it.
However, Disposition A overloads `adjudication` with a non-monetary
classification and forces a value-set / code-system change that ripples
into every `EOB.adjudication`-aware tool. Disposition B threads the
needle: it gives the workgroup a single, normatively-published HL7
extension on `careTeam`/`provider` that is unambiguously *about a
provider role*, it lets CARIN BB collapse its current per-role
`supportingInfo` slices onto one canonical element, and it leaves the
core adjudication backbone untouched. The Financial Mgmt work group
should also confirm with CARIN BB / Da Vinci that the
`contracted | non-contracted` binding is sufficient or whether a richer
tiered-network value set is needed before binding strength is set to
`required`.
