## Summary

The reporter asks the Da Vinci PCDE (Post-Acute Care Data Exchange) IG to
constrain `CarePlan.goal` and `CarePlan.careTeam` out (cardinality `0..0`)
on the PCDE profile of `CarePlan`. The argument is scope-based: PCDE uses
`CarePlan` as a retrofit container for "ActiveTreatments" rather than as
a clinical care plan, and the reporter believes the presence of these
two elements invites confusion that the resource is being used
clinically. There are no linked or cross-referenced Jira tickets and no
Zulip or GitHub discussions associated with this ticket in the indexed
data; the only cross-references found are the FHIR element mentions
(`CarePlan.goal`, `CarePlan.careTeam`) drawn from the description text
itself.

## Details

**Description:**

> Both CarePlan.goal and CarePlan.careTeam cause more confusion in
> thinking it is a clinical care plan. Per our scope and use of this
> resource as a retrofit of PCDE ActiveTreatments, we never expect these
> attributes to be filled and therefore it's best that we just zero them
> out.

**Comments:**

- **mckenzie (2020-11-23):** "It's not clear that a payer *couldn't*
  have goals for a particular therapy or know the set of providers
  relevant to a particular ongoing therapy. As such, it wouldn't
  strictly be *wrong* to send this information — which is the usual bar
  for prohibiting something in FHIR. I don't want to constrain this out
  in the initial release. We should discuss as a group."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the Da Vinci PCDE IG's `CarePlan` profile, set the cardinality of
`CarePlan.goal` and `CarePlan.careTeam` to `0..0` (i.e., add `max = 0`
constraints in the `StructureDefinition` differential). Update the
profile's narrative / element guidance to state that these elements are
deliberately prohibited because PCDE uses `CarePlan` solely as a
container for ActiveTreatments and does not model clinical goals or care
team composition. Regenerate the IG so the profile snapshot, the
must-support / cardinality table, and any examples reflect the new
constraint, and add a release note describing the breaking change for
implementers already populating either element.

#### Justification

The reporter's framing is consistent with how PCDE scopes `CarePlan`:
treatments authorized under prior coverage, not clinical planning.
Profile-level `0..0` constraints are an established, well-supported
mechanism in FHIR for narrowing a base resource to the use case at hand,
and they remove any ambiguity for implementers about whether goals or
care team membership are expected. If the WG agrees the resource is
strictly a retrofit container, prohibiting the two elements is the most
direct way to encode that intent and prevent confusion — which is
precisely the harm the reporter cites.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `CarePlan.goal` and `CarePlan.careTeam` at their base
cardinalities (`0..*`) on the PCDE `CarePlan` profile, but address the
"clinical care plan" confusion through documentation rather than
schema. Specifically: (1) add explicit profile-level guidance that PCDE
uses `CarePlan` as a wrapper for ActiveTreatments and that `goal` /
`careTeam` are not part of the PCDE exchange contract and should
generally be omitted; (2) optionally tag both elements as Must Support
= false with element-level comments echoing the same guidance; and (3)
add a "Why CarePlan?" section to the IG's resource-usage page so the
modeling rationale is discoverable.

#### Justification

This addresses the reporter's underlying concern (implementer
confusion) while preserving the option Lloyd McKenzie raises in the
comment — that a payer could legitimately know goals or providers
attached to an ongoing therapy and that there is no FHIR-level reason
the data would be *wrong*. Documentation-only changes are
non-breaking, are reversible if PCDE later wants to embrace the
elements, and avoid prohibiting valid data exchange that some
trading partners may already perform. The trade-off versus
Disposition A is that nothing in the schema stops the data from
appearing, so confusion can recur for implementers who don't read the
narrative.

---

### Disposition C: Decline

#### Proposal

Decline to make any changes to the PCDE `CarePlan` profile in response
to this ticket, on the grounds that prohibiting valid base-resource
elements does not meet FHIR's usual bar for `0..0` constraints — i.e.,
that sending the data would be *wrong*. Close as "Persuasive with
modifier — no change" or equivalent, with a note pointing implementers
at the existing profile narrative for scope guidance.

#### Justification

This mirrors the position Lloyd McKenzie takes in the only comment on
the ticket: the elements are not semantically wrong in a payer
context, FHIR generally avoids prohibiting things that aren't wrong,
and a "we never expect this to be used" expectation does not by itself
justify a breaking schema constraint on an already-published IG.
Declining preserves maximum flexibility for trading partners and
keeps PCDE aligned with general FHIR profile-design conventions.

---

### Recommendation

**Recommended disposition:** B

The reporter's pain point is real — implementers do conflate `CarePlan`
with clinical care planning, and PCDE's use of the resource is
deliberately narrow — so doing nothing (Disposition C) leaves that
confusion in place. However, Lloyd McKenzie's comment is the only
recorded substantive discussion and it explicitly warns against
constraining the elements out in the initial release because doing so
fails FHIR's "would it be wrong to send this?" test and risks rejecting
data that is, in fact, legitimate for a payer to know. Disposition B
threads the needle: it directly addresses the confusion the reporter
identifies through profile guidance and Must Support shaping, without
locking out conformant data or introducing a breaking change. If a
future ballot cycle gathers concrete evidence that implementers are
populating these elements harmfully, the work group can revisit
Disposition A then with stronger justification.
