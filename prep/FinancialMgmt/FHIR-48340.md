## Summary

This ticket asks the CARIN Blue Button (C4BB) project to submit a defined
list of value sets (and any related code systems where applicable) from
the C4BB IG to HL7 Terminology (THO) so that they become reusable
terminology assets rather than IG-local artifacts. The reporter
enumerates nine specific C4BB value sets covering institutional claim
sub-type, institutional care-team roles, NUBC point of origin, NUBC
patient discharge status, ICD-9/10-CM diagnosis codes, inpatient and
outpatient institutional diagnosis-type, institutional procedure-type,
and present-on-admission. The reporter notes one of them (AHA NUBC
Present On Admission) likely overlaps existing THO content
(`ValueSet-v2-0895` and `ValueSet-POAIndicators`, where the 6.0.2
version of POAIndicators is more complete than 3.1.0), and flags that
moving `C4BBClaimInstitutionalCareTeamRole` to THO will need discussion
because the underlying CareTeamRole concept should stay open to
non-physicians even though only physicians submit claims. A follow-up
comment from the reporter additionally suggests that, while migrating
the inpatient `DiagnosisType` value set, the value set should be
expanded to also include concepts already present in the base
`Claim.diagnosis.type` example value set
(`http://hl7.org/fhir/R4/valueset-ex-diagnosistype.html`). Cross-
reference data confirms the ticket touches the FHIR artifacts
`Procedure.type` and `Claim.diagnosis.type`. No other Jira tickets,
Zulip discussions, or GitHub items are linked to FHIR-48340.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Submit each of the nine listed C4BB value sets (and any locally-defined
supporting code systems) to HL7 Terminology (THO), then refactor the
C4BB IG to bind to the THO-hosted canonical URLs instead of the
IG-local definitions. Concretely:

1. For each value set in the list except `AHANUBCPresentOnAdmission`,
   open a corresponding THO Jira ticket (project `UTG`) with the
   ValueSet (and its referenced CodeSystems where C4BB owns them, e.g.
   the `C4BB*` code systems) attached as the proposed THO content,
   preserving the canonical concept enumerations.
2. For `AHANUBCPresentOnAdmission`, do **not** create a new THO value
   set; instead retire the C4BB-local copy and rebind to the existing
   THO `ValueSet-POAIndicators` (using the 6.0.2-aligned version that
   includes all NUBC POA codes including `1`), and document the
   `v2-0895` relationship in the binding description.
3. For the inpatient `DiagnosisType` value set
   (`C4BBClaimInpatientInstitutionalDiagnosisType`), apply the
   reporter's comment: expand the proposed THO value set to also
   include the concepts from the base FHIR
   `http://hl7.org/fhir/ValueSet/ex-diagnosistype` example value set
   before submission, so the THO version is a superset usable by both
   institutional EOB profiles and base `Claim.diagnosis.type`.
4. After THO publishes, update the C4BB IG bindings on the relevant
   profiles (institutional inpatient EOB, institutional outpatient EOB,
   institutional CareTeam, etc.) to point to the new THO canonicals,
   and remove the now-redundant IG-local ValueSet/CodeSystem resources
   in a dedicated PR.

For `C4BBClaimInstitutionalCareTeamRole`, accept the migration but flag
in the THO submission that the value set is intentionally narrower than
the base `CareTeam.role` (physician-submitted institutional claims only)
so THO stewards understand the scope.

#### Justification

This is the canonical "promote IG terminology to THO" pattern that the
Vocabulary WG and project leadership have been encouraging for several
years; CARIN Blue Button is widely implemented (CMS interoperability
mandates) and its NUBC/AHA/ICD-anchored value sets are reused by other
financial and quality IGs (Da Vinci PDex, HRex, etc.). Hosting them in
THO removes accidental duplication, makes versioning explicit, and lets
non-C4BB IGs reuse the same canonical without copying definitions.
Accepting the reporter's `DiagnosisType` superset suggestion in the same
move avoids an immediate follow-up migration and keeps the THO copy
useful to the base `Claim` resource as well as the C4BB EOB profiles.
Reusing the existing THO `POAIndicators` (6.0.2) for present-on-
admission rather than re-publishing a near-duplicate honors the explicit
note in the ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Stage the migration in tranches based on ownership and risk, rather
than submitting all nine in one batch:

- **Tranche 1 (low-risk, externally-owned codes):** migrate the value
  sets that are pure enumerations over externally-stewarded code systems
  — `NUBCPointOfOriginForAdmission`, `DischargeStatus`,
  `CDCICD910CMDiagnosisCodes` — to THO first, since their content is
  not C4BB-authored and they are the most reusable by other IGs.
- **Tranche 2 (C4BB-authored type/category code systems):** migrate
  `InstitutionalClaimSubType`, `C4BBClaimInpatientInstitutionalDiagnosisType`
  (expanded per the comment), `C4BBClaimOutpatientInstitutionalDiagnosisType`,
  and `C4BBClaimProcedureType` after Vocabulary WG review of the
  underlying `C4BBClaim*` code systems — these are stewarded by C4BB
  and the move requires a stewardship transfer agreement, not just a
  submission.
- **Tranche 3 (deferred / discussion needed):** hold
  `C4BBClaimInstitutionalCareTeamRole` for joint Patient
  Administration / Financial Mgmt / Vocab discussion before submission,
  because narrowing `CareTeam.role` to physician-only at the THO level
  could leak into non-claims uses. In the interim, keep this value set
  IG-local.
- **Tranche 4 (rebind only, no new submission):** for
  `AHANUBCPresentOnAdmission`, skip THO submission entirely and just
  rebind the C4BB profile to the existing THO `POAIndicators` (6.0.2).

Each tranche becomes its own UTG ticket and its own C4BB rebind PR.

#### Justification

The nine value sets are not homogeneous — some wrap externally-owned
code systems (CDC/ICD, NUBC, AHA), some are C4BB-authored type
enumerations, one is potentially over-narrow for THO (CareTeam role),
and one is already represented at THO. Bundling them into a single
"submit to THO" effort risks the whole batch stalling on the
CareTeam.role discussion. Tranching also keeps the rebind PRs small
and reviewable, and lets implementers continue to rely on stable
canonical URLs for the harder cases while the easy ones move first. It
directly honors the "may need to discuss" caveat the reporter wrote
into the ticket for `ClaimInstitutionalCareTeamRole`.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-48340 as filed and close it as **Not Persuasive**, on the
basis that (a) `AHANUBCPresentOnAdmission` is already represented in
THO via `v2-0895` / `POAIndicators`, so no new submission is warranted
there; (b) the remaining value sets are tightly coupled to C4BB
profiles (e.g., the `C4BB*` type enumerations, the institutional
CareTeam role narrowing) and have not been requested by another IG, so
moving them out of the IG would create stewardship overhead at THO
without a demonstrated reuse benefit; and (c) no concrete external IG
or implementer has been cited as needing the THO canonicals. Direct
the reporter to instead document the binding rationale in the existing
C4BB ValueSet pages and revisit migration only when a concrete external
consumer files a request.

#### Justification

Promoting an IG-local value set to THO is a stewardship transfer, not
just a copy — it commits the Vocabulary WG (and, transitively, HL7) to
maintaining the value set across future versions, including answering
errata, processing new NUBC/AHA/ICD code system updates, and resolving
cross-IG conflicts. That cost is justified when there is demonstrable
cross-IG reuse, but FHIR-48340 does not cite any external consumer; the
ticket reads as a tidy-up rather than a response to a specific
implementation pain point. Combined with the explicit
`CareTeamRole`/physician-only concern flagged by the reporter, this
disposition argues that the safer course is to leave the value sets in
the IG where C4BB owns them outright.

---

### Recommendation

**Recommended disposition:** B

Disposition B (tranched migration) is the strongest fit. The reporter
has done the right thing by enumerating the candidate value sets, and
the long-term direction for FHIR terminology is unambiguously to
consolidate reusable value sets in THO — so a flat "decline"
(Disposition C) ignores that direction and the reporter's own NUBC/AHA
observations. However, Disposition A (accept as filed, batch) under-
weights two real risks the reporter themselves raised: the
`ClaimInstitutionalCareTeamRole` scoping problem requires cross-WG
discussion that should not block migration of the externally-anchored
NUBC/AHA/ICD value sets, and the `AHANUBCPresentOnAdmission` case is
explicitly a rebind-not-submit. Disposition B keeps the easy wins
moving (Tranche 1), correctly treats the C4BB-authored type
enumerations as a stewardship-transfer (Tranche 2), parks the
contentious CareTeamRole case (Tranche 3), and resolves the POA case
without creating a duplicate THO artifact (Tranche 4). It also makes
room to absorb the reporter's `DiagnosisType` superset comment inside
Tranche 2, where the change actually applies.
