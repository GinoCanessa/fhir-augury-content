## Summary

FHIR-46098 asks the Da Vinci **ATR** IG to fix misleading guidance in the
`$davinci-data-export` OperationDefinition. The current `exportType`
parameter description tells implementers that the PDEX value is simply
`"hl7.fhir.us.davinci-pdex"`, but PDEX in practice differentiates several
sub-uses of the operation via fragment-style identifiers — for example,
`hl7.fhir.us.davinci-pdex#payertopayer` and
`hl7.fhir.us.davinci-pdex#provider-delta`. The reporter (Carie Hammond)
proposes that the ATR description either enumerate every PDEX-defined
value or, preferably, defer to the PDEX IG as the source of truth so new
PDEX export-types can be added without round-tripping a change back into
ATR. The ticket has already been triaged with resolution **Persuasive**:
the agreed direction (per the resolution text) is to drop the embedded
`#payertopayer` example and rewrite the parameter documentation to point
readers to the PDex IG for the canonical list of export-type values. The
change was reviewed on the PDex Community Call on 2026-03-27 (per
ekivemark's comment) and is currently labelled for the **April 2026
block vote**.

No related Jira tickets surfaced through Augury cross-references. One
tangentially-related Zulip thread on Da Vinci PDex was found via search
(see below); it discusses CMS-vs-PDex divergence around
`$davinci-data-export` and is useful context for why pinning the
authoritative export-type list to PDEX matters.

## Details

**Description:**

> The description for exportType parameter in the $davinci-data-export
> operation profile indicates that for PDEX, the appropriate value is,
> "hl7.fhir.us.davinci-pdex". However, PDEX has multiple uses of the
> operation and differentiates them, for example,
> "hl7.fhir.us.davinci-pdex#payertopayer" or
> hl7.fhir.us.davinci-pdex#provider-delta to name a few.
>
> The description for this exportType parameters element should either
> list them all or direct readers to the PDEX IG for the appropriate
> value when using $davinci-data-export for PDEX. My preference is
> pointing to the PDEX IG, which allows for new values to be added
> without having to update the operation profile in ATR again.

**Resolution Description (Persuasive):**

> Will update the the $davinci-data-export exportType parameter's
> documentation, remove "and hl7.fhir.us.davinci-pdex#payertopayer for
> a PDex Payer to Payer Data Exchange Data Export.", instead will
> update the language to point to the PDex IG for export types defined
> in PDex.

## Proposed Dispositions

### Disposition A: Accept As Requested (point readers to the PDEX IG)

#### Proposal

In the **US Da Vinci ATR** IG, edit the `OperationDefinition` for
`$davinci-data-export` so that the `exportType` parameter's
`documentation` element no longer enumerates a single PDEX value (and
no longer hard-codes `hl7.fhir.us.davinci-pdex#payertopayer` as an
example). Replace that text with language that:

1. States that for PDEX-driven exports, the value of `exportType`
   MUST be one of the values defined by the **Da Vinci PDex IG** for
   `$davinci-data-export`; and
2. Links to the PDex IG section that lists those values (e.g., the
   PDex page that defines `payertopayer`, `provider-delta`, etc.).

Concretely, this means amending the `OperationDefinition.parameter`
entry whose `name` is `exportType` (the `documentation` markdown), and
keeping the parameter's binding/cardinality unchanged. The change is
documentation-only — no schema/cardinality/binding change to the
operation itself. This matches the **Persuasive** resolution already
recorded on the ticket.

#### Justification

This is exactly the resolution the ATR / PDex community has already
agreed to on the 2026-03-27 PDex Community Call, and it is the
reporter's stated preference. Pointing to PDEX as the authoritative
source for PDEX-specific `exportType` values:

- removes the "wrong/incomplete" guidance currently in ATR (only
  `hl7.fhir.us.davinci-pdex` is mentioned, omitting `#payertopayer`,
  `#provider-delta`, and any future entries);
- decouples the ATR publication cadence from PDEX evolution — new
  PDEX use-cases can be defined without re-balloting ATR, which is
  the operational pain the ticket calls out;
- mirrors a common FHIR pattern of letting the consuming IG own its
  own profile/operation usage details rather than the defining IG
  trying to keep an exhaustive cross-IG catalog in sync; and
- the supporting Zulip thread shows that implementers are already
  reading `exportType` as the discriminator for PDEX use-cases, so
  authoritative documentation reduces real interop risk.

---

### Disposition B: Alternative Approach — define the values via a CodeSystem/ValueSet referenced by the parameter

#### Proposal

Rather than letting the `exportType` documentation prose carry the
list of allowed values, define a **CodeSystem** (or extensible
**ValueSet**) for `$davinci-data-export` `exportType` codes — owned
by ATR or by PDex — and `binding` the parameter to it. The existing
PDEX values (`hl7.fhir.us.davinci-pdex`,
`hl7.fhir.us.davinci-pdex#payertopayer`,
`hl7.fhir.us.davinci-pdex#provider-delta`, …) become enumerated
concepts in that artifact. ATR's `OperationDefinition` would then say
"see binding" and link to the ValueSet; PDex (or other consuming IGs)
can grow the ValueSet over time. The parameter `documentation` still
refers readers to the PDex IG for narrative usage of each code.

#### Justification

This trades a documentation fix for a small terminology asset, but
gives three things prose alone cannot:

- **Machine-readable conformance:** validators can flag bad
  `exportType` values; today they can't, because the parameter is
  effectively free text.
- **A single, canonical, versioned list** of PDEX (and any other
  consumer's) `exportType` codes, decoupled from prose drift in the
  IG.
- **Future extensibility:** the ValueSet can be `extensible`, so
  non-PDEX use-cases can also slot in cleanly without further
  OperationDefinition edits.

Trade-offs vs. Disposition A: heavier change (new terminology
artifact, a binding on an existing parameter, new ATR↔PDex
ownership/maintenance question, longer review cycle), and arguably
out of scope for a ticket that the workgroup has already triaged as a
documentation reconciliation. It would not land cleanly in the
April 2026 block vote.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** on the grounds that the existing
`exportType` parameter documentation in ATR is sufficient (PDEX
implementers can read the PDex IG themselves) and that ATR should not
take on a new normative dependency on PDex's evolving
`exportType` values.

#### Justification

The defensible angle here is "scope": ATR defines a generic export
operation, and any consuming IG (PDEX or otherwise) is responsible
for telling its own implementers which `exportType` token to send.
Adding an outbound link to PDex from ATR creates a soft dependency
between two IGs that ballot independently and could easily drift.

However, the workgroup has **already** marked the ticket Persuasive
and reviewed it on the 2026-03-27 PDex Community Call, so this
disposition is documented for completeness rather than as an
expected outcome.

---

### Recommendation

**Recommended disposition:** **A — Accept As Requested.**

This recommendation aligns with the resolution already recorded on
the ticket (Persuasive), the outcome of the 2026-03-27 PDex
Community Call, and the reporter's stated preference. The fix is
documentation-only on a single OperationDefinition parameter in the
ATR IG, has no normative impact on the operation's signature, and
materially reduces implementer confusion that the Zulip discussion
shows is real (implementers are using `exportType` as the
discriminator for PDEX use-cases and need an authoritative,
forward-stable list). Disposition B (formal binding to a
CodeSystem/ValueSet) is the cleaner long-term solution but is
substantially larger work, would not make the April 2026 block vote,
and can be pursued as a follow-up Jira if the workgroup later wants
machine-checkable conformance for `exportType`. Disposition C runs
counter to the existing triage decision and would leave incorrect
guidance (`hl7.fhir.us.davinci-pdex` only) in the published IG.
