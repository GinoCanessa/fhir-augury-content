## Summary

FHIR-35407 asserts that the canonical defining URL for the CARIN IG for Blue
Button® (C4BB) ValueSet `AHANUBCTypeOfBill` should be anchored in
`terminology.hl7.org` (THO) rather than under `hl7.org/fhir/...`. The reporter
recommends opening a corresponding UTG ticket so the ValueSet is registered in
THO and the C4BB IG can bind to the THO-hosted artifact instead of carrying
its own copy. The ticket is part of CARIN BB STU2 Block-Vote-6 (FM 2022-07-19
WG call, voted 2022-07-26 / 13-0-0) where it was originally resolved as
**Considered for Future Use**: the ValueSet enumerates all codes from the
external AHA NUBC Type-of-Bill CodeSystem, and the FM WG decided to move it
to THO once a generic mechanism for "all codes" ValueSets exists, tracked by
[UP-316](https://jira.hl7.org/browse/UP-316) ("Create 'All Codes' ValueSets
for all External CodeSystems"). Lynn Laakso then **reverted** the prior
resolution and **re-opened** the ticket on 2022-11-28 "post STU2
publication", so it is back on the FM agenda awaiting a new disposition for
the next CARIN BB cycle.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the defining URL of `ValueSet-AHANUBCTypeOfBill` out of the C4BB IG
namespace and anchor it under THO. Concretely:

1. In `HL7/UTG`, add a new ValueSet resource that enumerates all codes of the
   AHA NUBC Type-of-Bill CodeSystem, with canonical
   `http://terminology.hl7.org/ValueSet/nubc-typeofbill` (final slug to be
   confirmed by Vocabulary / UTG governance) and an appropriate UTG
   stewardship statement. File a UTG ticket to track the addition (the
   description explicitly calls for this).
2. In `HL7/carin-bb`, after the THO ValueSet is published: remove the
   IG-local `ValueSet-AHANUBCTypeOfBill` resource and repoint every binding
   that currently references the C4BB canonical to the new THO canonical.
   Add a release-note entry flagging the canonical change for downstream
   implementers.
3. Where C4BB profiles bind to this ValueSet at strength `required` on a
   `code` data type, leave the binding strength unchanged; the change is to
   the URL only, not to the binding semantics.

#### Justification

This is exactly what the reporter and the workgroup's standing terminology
guidance ask for: external code systems and the "all codes" ValueSets that
wrap them belong in THO so that *all* IGs can bind to a single canonical
artifact rather than each IG carrying its own copy. The 2022-07-26 vote
(13-0-0) shows the FM WG already agrees with the destination; it deferred
only because the generic mechanism (UP-316) was not yet in place. With the
ticket now re-opened post-STU2, accepting it commits FM to making the move
in the next CARIN BB cycle and creating the UTG companion ticket the
description calls for.

---

### Disposition B: Alternative Approach

#### Proposal

Keep FHIR-35407 open but **explicitly bind it to UP-316** as a precondition,
and adopt the THO-hosted artifact via the generic "all codes ValueSet"
mechanism rather than minting a one-off THO ValueSet just for NUBC
Type-of-Bill:

1. Mark FHIR-35407 as *Persuasive with Modification — depends on UP-316*.
   Add a formal "is blocked by" link from FHIR-35407 to UP-316 in Jira so
   the dependency is visible.
2. Defer the C4BB-side change (removing `ValueSet-AHANUBCTypeOfBill` and
   repointing bindings) until UP-316 ships the all-codes ValueSet pattern
   in THO; then apply it uniformly to all sibling Block-Vote-6 tickets
   (FHIR-35390 / 35391 / 35392 / …) in a single coordinated cycle so the
   C4BB IG only takes one canonical-URL churn event.
3. In the interim, keep the IG-local ValueSet but add an editor's note in
   the C4BB terminology section flagging that the canonical will move to
   THO once the all-codes pattern lands.

#### Justification

This honours the original WG vote ("Considered for Future Use, waiting on
UP-316") rather than treating that decision as something to be reverted. It
also avoids a one-off THO ValueSet whose shape might disagree with the
eventual UP-316 pattern, which would force a *second* canonical change and a
second wave of implementer disruption. Coordinating with the sibling THO-
anchoring tickets (FHIR-35390/91/92/…) keeps implementer churn to a single
release boundary instead of trickling across cycles. The cost is calendar
time: this disposition gates resolution on UP-316 progress in the UTG
project, which is currently in *Proposal Draft*.

---

### Disposition C: Decline

#### Proposal

Close FHIR-35407 as *Not Persuasive — Considered for Future Use, no STU
change*: leave `ValueSet-AHANUBCTypeOfBill` defined in the C4BB IG under its
existing `hl7.org/fhir/...` canonical, document the long-term intent in the
IG terminology section, and revisit only when a broader THO migration is
funded across all C4BB external-code-system ValueSets.

#### Justification

A defensible-on-paper option: the bindings already work for implementers,
the canonical-URL change is breaking for any downstream consumer that has
hard-coded the C4BB URL, and the WG already voted 13-0-0 to defer in 2022.
Re-opening solely to change the host of a canonical URL — without the
upstream THO pattern in place — risks churn for marginal benefit.

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach)**.

The reporter's request (Disposition A) is correct in direction but premature
in timing: the FM WG's own 2022-07-26 vote tied this ticket's resolution to
UP-316, and that linkage is still the right sequencing. Acting unilaterally
in CARIN BB before UP-316 produces a stable "all codes ValueSet" pattern in
THO risks minting a bespoke THO artifact whose canonical URL or
representation later has to be changed again — exactly the kind of churn
this ticket is trying to prevent. Disposition C ignores the explicit re-open
("re-open post STU2 publication") and silently leaves the spec in a state
the WG has already said it doesn't want long-term. Disposition B keeps the
ticket alive, makes the UP-316 dependency explicit and tractable in Jira,
and lets FM batch the canonical move across the FHIR-35390/91/92/… cluster
into a single C4BB release — which is the implementer-friendly outcome.
