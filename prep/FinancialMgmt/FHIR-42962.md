## Summary

Lloyd McKenzie argues that the value set bound to `ExplanationOfBenefit.status` is conceptually mismatched with how an EOB is actually used. An EOB is a **point-in-time, document-like snapshot** of an adjudication: once issued it is not mutated — corrections are conveyed by issuing a new EOB instance. The current value set reuses the generic `fm-status` codes (`active | cancelled | draft | entered-in-error`), and in particular the code `active` is defined as *"The resource instance is currently in-force."* The reporter contends that a frozen snapshot cannot meaningfully be "not in-force": an EOB that exists is, by construction, valid as of its created timestamp; the only meaningful negative state is `entered-in-error`. He therefore asks Financial Mgmt to revise the status codes (or the binding) so that the codes align with EOB's document-like semantics, mirroring how `Composition.status` / `DocumentReference.status` use codes such as `final`, `current`, `amended`, `superseded`, etc.

Although no Jira tickets cross-reference FHIR-42962 directly, two adjacent Financial Mgmt issues frame the same surface:

- [FHIR-48304](https://jira.hl7.org/browse/FHIR-48304) (Applied / Persuasive with Modification) explicitly recognises the "one EOB replaces another" pattern by extending `EOB.related.claim` to allow referencing a prior `ExplanationOfBenefit`. That confirms the workgroup already accepts that EOBs are versioned/superseded snapshots — which is exactly the use case the current `active` definition struggles to describe.
- [FHIR-40502](https://jira.hl7.org/browse/FHIR-40502) (Published / Persuasive) sharpens the definition of `EOB.created` to be a timestamp on the resource itself, reinforcing the snapshot model.
- [FHIR-38927](https://jira.hl7.org/browse/FHIR-38927) (FHIR-I, Persuasive) revisited the definition of "frozen" for documents — relevant because Lloyd is asking EOB to inherit document-style status semantics (`final`, etc.).

## Details

**Description:**
> An EOB is a point-in time 'summary' that behaves in a document-like fashion. An instance gets created and it is then frozen. It doesn't get changed. Instead new instances are created.
>
> In other document-like resources (Composition, DocumentReference, etc.), the status for a document that is complete is 'final' or 'current', not 'active'. 'active', particularly with the definition of "The resource instance is currently in-force." doesn't really make sense for the resource. It's not possible for a frozen snapshot like EOB to **not** be in force. It could be flagged as erroneous, but other than that, an EOB that is created always exists and is always valid (for the timestamp on which it was created). Please adjust the statuses for this element to make more sense to the use of EOB.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-bind `ExplanationOfBenefit.status` (in `source/explanationofbenefit/`) to a **new EOB-specific value set** that uses document-lifecycle semantics rather than the shared `fm-status` codes. Concretely:

- Create a new value set `valueset-explanationofbenefit-status` whose codes are aligned with `Composition.status` / `DocumentReference.status`: `current | superseded | entered-in-error` (and optionally `preliminary` to cover "draft adjudication not yet final"). The new codes go into a new EOB-specific CodeSystem (so the shared `fm-status` is not disturbed).
- In the `ExplanationOfBenefit` StructureDefinition `<differential>`, change the binding of `ExplanationOfBenefit.status` to the new value set, keeping binding strength `required`.
- Rewrite the element definition and the new code definitions so that `current` explicitly means "the most recently issued snapshot for this adjudication", `superseded` means "a later EOB has been issued that replaces this one" (cross-linking with FHIR-48304's `EOB.related.claim` mechanism), and `entered-in-error` keeps its existing meaning.
- Update `source/explanationofbenefit/` narrative pages and the per-element guidance to explain the new lifecycle.

#### Justification

This is the change Lloyd actually asks for and it directly aligns the codes with how the workgroup has already chosen to model EOBs (point-in-time snapshots; FHIR-40502; FHIR-48304). It also resolves the implementer confusion documented in both CARIN BB Zulip threads — both Fred Harmon's "why would I ever use anything but `active`?" and Josh Lamb's "can I modify an `Active` EOB?" go away because the new codes have crisp semantics. Moving to an EOB-specific value set rather than redefining `fm-status` keeps the blast radius local: per the HL7/fhir briefing, `fm-status` is shared across other Financial Management resources, so leaving it untouched avoids breaking Claim/ClaimResponse/Coverage. The change is a `required`-binding swap, so it is a breaking change for R6 (or whatever the next major), but it is the right shape of fix.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the binding to `fm-status` but **constrain the value set** used at `ExplanationOfBenefit.status` and **rewrite the EOB-specific definitions** of those codes — without inventing a new code system. Concretely:

- In `ExplanationOfBenefit`'s `<differential>`, leave the binding pointed at `fm-status` (or a narrowed sibling value set) but limit the allowed codes to `active | cancelled | entered-in-error`, removing `draft` from the EOB-permitted set on the grounds that an EOB only exists once it is issued.
- Add EOB-specific text to `ExplanationOfBenefit.status` (and to the `source/explanationofbenefit/` narrative) explaining that for EOB:
  - `active` = "this EOB is currently the in-force snapshot for the adjudication" (i.e., it has not been superseded or marked in error). Document that "in force" is per-snapshot, not per-EOB-instance.
  - `cancelled` = "a later EOB has been issued that replaces or withdraws this one" (link to `EOB.related.claim` per FHIR-48304).
  - `entered-in-error` = unchanged.
- Optionally add an extension (or reuse `Provenance`) for finer-grained replace/supersede metadata rather than introducing new status codes.

#### Justification

This addresses the substantive complaint — that the codes are under-defined for the EOB use case — without creating a new value set, a new CodeSystem, or a UTG ticket. It is non-breaking for any implementer who is currently sending `active`, which preserves CARIN BB compatibility. It also leans on a mechanism (`EOB.related.claim` referencing another EOB) that the workgroup just accepted in FHIR-48304, so the lifecycle story is coherent across tickets. The downside is that it keeps `active` — a code Lloyd specifically calls out as a poor fit — and only papers over the conceptual mismatch with EOB-specific definition text.

---

### Disposition C: Decline

#### Proposal

Mark **Not Persuasive** on the grounds that (a) the existing `fm-status` codes are stable, widely implemented (CARIN BB and other downstream IGs), and changing them or the binding would be a breaking change with no concrete defect; (b) `entered-in-error` and `cancelled` already cover the only states Lloyd identifies as semantically meaningful for a frozen snapshot; and (c) the lifecycle of "this EOB has been replaced" is being addressed structurally via FHIR-48304 (`EOB.related.claim` → `ExplanationOfBenefit`) rather than via the status code, which is the FHIR-conventional mechanism.

#### Justification

The shared `fm-status` value set is bound across multiple Financial Management resources, and downstream IGs (CARIN BB in particular) have hard expectations on the existing codes; the briefing flags `fm-status` as a cross-cutting terminology asset. The Zulip threads show implementer confusion, but Pat Taylor's clarification ("yes, you can modify/replace an Active EOB") and Paul Church's "lastUpdated just means content changed" framing both describe behaviour that already works under the current codes — implementers want better documentation, not new codes. Combined with FHIR-48304 supplying explicit replace/supersede semantics outside the status field, the case for a breaking re-binding is weak.

---

### Recommendation

**Recommended disposition:** A.

The substantive critique is correct: `active` defined as *"the resource instance is currently in-force"* is incoherent for a snapshot resource that, by the workgroup's own acceptance of FHIR-48304 and FHIR-40502, is explicitly modelled as a point-in-time artifact that may be superseded by another instance. The CARIN BB Zulip threads are not noise — they are two independent rounds of competent implementers being unable to derive consistent meaning from the current codes, which is the textbook signal that the value set, not the documentation, is wrong. Disposition B is tempting because it avoids breakage, but it preserves the very code (`active`) the ticket calls out as the source of the problem and would leave us re-litigating this every ballot cycle. Disposition C ignores that even within Financial Management the lifecycle has *already* shifted (FHIR-48304) and that the status codes haven't followed. The right move is a localised, EOB-specific value set per Disposition A, scoped to `source/explanationofbenefit/` plus a coordinated `HL7/UTG` change, leaving the shared `fm-status` codes alone so other FM resources are not disturbed. The workgroup should weigh whether to land this in the next major (where binding swaps are tolerable) or to stage it as `extensible` first to give CARIN BB and other IGs a migration window.
