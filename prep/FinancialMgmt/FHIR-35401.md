## Summary

The submitter (Joan Harper, comment imported by Ron G. Parker) asks the
CARIN IG for Blue Button (CARIN BB) team to move the canonical URL of
the `ADAUniversalNumberingSystem` value set out of the
`hl7.org/fhir/...` namespace and anchor it in
`terminology.hl7.org` (THO), with a corresponding UTG ticket to
register the value set in THO. The general principle the reporter
articulates is that only value sets that are bound `required` to a
`code` data type should remain in the `hl7.fhir.org` namespace; all
others (including this one) belong in THO.

This is one item in a much larger CARIN BB STU2 sweep — the same
reporter filed parallel tickets (e.g., FHIR-35402 … FHIR-35447) making
the identical request for each of the IG's other value sets. They were
batched for review as Block-Vote-6 (see the related Zulip
announcement). No Jira-side cross-references to those siblings are
recorded against FHIR-35401, but the ticket should be understood as
part of that larger family. The block vote on 2022-07-26 dispositioned
this specific ticket as **Considered for Future Use**, deferring the
change until the upstream "all codes" ValueSet pattern (UP-316) is
available; that resolution was reverted on 2022-11-28 with the note
"re-open post STU2 publication", so the ticket is again Triaged and
awaiting WG action.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move `ValueSet-ADAUniversalNumberingSystem` out of the CARIN BB IG's
`http://hl7.org/fhir/us/carin-bb/ValueSet/...` namespace by retiring
the IG-defined value set and re-pointing every binding that currently
references it to a value set published under
`http://terminology.hl7.org/ValueSet/...`. Concretely: (1) file a UTG
(`HL7/UTG`) ticket to author the THO `ValueSet` resource (and, if
needed, register or re-anchor any associated CodeSystem) using the
ADA Universal Numbering System as a complete external code system;
(2) once that THO artifact is published, update each profile in CARIN
BB STU3 (e.g., the C4BB tooth/oral-related profile binding(s)) to
bind to the new THO canonical; (3) deprecate / mark the IG-local
value set as withdrawn in the next CARIN BB release notes.

#### Justification

This is exactly what the reporter asked for, and it is consistent with
the broader HL7 terminology-anchoring policy: value sets that simply
enumerate "all codes" of an external code system are governed
artifacts that belong in THO so they can be reused across IGs without
each IG re-publishing its own copy. The Nov-2022 comment from
lynn_laakso explicitly endorses this path ("when [UP-316] is complete,
the ValueSet in this IG can be removed and the bindings point to the
THO ValueSet"), and the reporter's framing — that only value sets
bound `required` to a `code` primitive should stay in the IG namespace
— aligns with FHIR's own conventions for code-bound primitives vs.
Coding/CodeableConcept bindings.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the namespace move until the upstream "all codes" value-set
pattern from [UP-316](https://jira.hl7.org/browse/UP-316) is published
in THO, but in the next CARIN BB release add (a) a published
narrative note on the value-set page stating the intent to re-anchor
in THO and pointing implementers at the future THO canonical, and (b)
keep the IG-local ValueSet canonical stable so that profiles continue
to validate. When UP-316 lands, do the actual re-binding in a single
coordinated CARIN BB release that retires all of the
similarly-affected IG-local value sets in one pass (the FHIR-35402…
FHIR-35447 family).

#### Justification

This was, in effect, the original block-vote resolution
("Considered for Future Use … when [UP-316] is complete"). It avoids
publishing a THO artifact whose canonical URL would have to change
again once the broader pattern is finalised, and it batches the
disruptive binding changes across the whole IG instead of one value
set at a time. The trade-off vs. Disposition A is that implementers
keep the temporary IG-local canonical for at least one more cycle,
but they get a clear migration signal in the published narrative.

---

### Disposition C: Decline

#### Proposal

Mark the request as **Not Persuasive** on the grounds that
`ValueSet-ADAUniversalNumberingSystem` is bound only inside this IG,
the ADA CDT licensing/terms make external publication awkward, and
the value set's content is fully specified by the external ADA code
system — implementers can resolve the codes without the value-set
canonical needing to live in THO. Close the ticket with no namespace
change.

#### Justification

Several siblings in the same Block-Vote-6 slate were dispositioned
"Not Persuasive" on the same WG call with very similar wording —
"ValueSet is not complex, fairly specific to this IG, and does not
warrant the maintenance of moving to THO" (e.g., FHIR-35432,
FHIR-35434, FHIR-35435, FHIR-35446, FHIR-35447). If the WG concludes
that the ADA value set falls into the same bucket — narrowly used,
licensed third-party content, and adequately served by the existing
IG canonical — declining is internally consistent with the rest of
the slate.

---

### Recommendation

**Recommended disposition:** B

The Nov-2022 reopen comment from the WG itself frames the issue as
"do this once UP-316 lands", not "do this immediately" or "do not do
this". That makes Disposition B the path that respects both the
reporter's intent (the value set should ultimately live in THO) and
the WG's already-stated dependency on UP-316, while avoiding the
churn of re-publishing a THO canonical now that may need to change
again when the "all codes" pattern is finalised. Disposition A is
defensible if UP-316 has progressed enough to give a stable target
canonical — in that case the WG should fold A back in. Disposition C
is hard to defend specifically for the ADA value set: unlike the
truly IG-specific ValueSets that were Not Persuasive in the same
block vote, the ADA Universal Numbering System is a complete external
code system reused across dental IGs, which is precisely the
"belongs in THO" pattern. The WG should adopt B now and revisit at
the next CARIN BB cycle once UP-316 status is known.
