## Summary

The reporter notes that the diagram and table in the Provider Bulk Access API
section (6.2.1) of the Da Vinci PDex IG implies the payer "sends" the bulk
response back to the requestor when processing completes. That is not how
FHIR Bulk Data works: the payer's initial response only returns a polling
location; the requestor polls that location until the job is complete and
then retrieves the NDJSON files itself. The ticket asks that the diagram /
table be updated to make this asynchronous request pattern explicit on the
Provider Access page.

This is the same clarification that was previously applied to the
payer-to-payer and bulk member-match pages under
[FHIR-49943](https://jira.hl7.org/browse/FHIR-49943) (and follow-up
FHIR-53256) — that work covered Payer-to-Payer Bulk Exchange and the
`$multi-member-match` operation, but the Provider Access page was left
with only the `Prefer: respond-async` header requirement and no narrative
description of the kick-off / poll / retrieve flow. FHIR-53616 closes that
remaining gap on the Provider Access API page. The ticket is already
labelled `Pre-Applied`: the change has been drafted in commit
`ab42fba4` ("Resolve FHIR-53616: document async request pattern for Provider
Bulk Access API") and is reflected in the IG's `changehistory.md` /
`changehistory.html`.

## Details

**Description:**
> In the diagram at this section showing the Bulk request, it appears to
> indicate that the Payer's service "sends" a response to the requestor
> when the Bulk query is complete. The actual pattern is that in Bulk the
> initial response gives the requestor a location that they then ping until
> the process is complete. Then they retrieve the NDJSON. It would be
> useful to update that interaction 6.2.1 Provider Bulk Access API table
> section.

**Resolution Description (already drafted, "Pre-Applied"):**
> The prior work (FHIR-49943 / FHIR-53256) had already covered the
> payer-to-payer and bulk-member-match pages. This ticket required the
> same treatment for the provider access page, which only had the
> `Prefer: respond-async` header requirement with no description of the
> actual async flow. `input/pagecontent/provider-access-api.md` now has
> §pdex-304a/304b with the explicit three-step pattern — kick-off → poll →
> retrieve NDJSON — and the explicit statement that the payer does not
> push data to the provider.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update `input/pagecontent/provider-access-api.md` in `HL7/davinci-epdx` so
that the §6.2.1 "Provider Bulk Access API" interaction table (and the
narrative immediately following it) explicitly documents the FHIR Bulk Data
asynchronous request pattern. Concretely:

1. After the existing `Prefer: respond-async` requirement, insert two
   numbered conformance statements (mirroring the resolution: §pdex-304a
   and §pdex-304b, or equivalently §pdex-340/341 already present in the
   file) that spell out the three-step flow:
   - **Kick-off:** Provider POSTs to
     `Group/[id]/$davinci-data-export` with `Prefer: respond-async`; the
     payer **SHALL** return HTTP 202 Accepted with a `Content-Location`
     header pointing at a status endpoint, and **SHALL NOT** return data
     in the kick-off response body.
   - **Poll:** Provider polls the `Content-Location` URL per the
     [Bulk Data Status Request](https://hl7.org/fhir/uv/bulkdata/STU2/export.html#bulk-data-status-request)
     guidance until the payer returns the completion manifest.
   - **Retrieve:** Provider downloads the NDJSON output files listed in
     the manifest from the payer's file server.
2. Add an explicit sentence stating that the payer does not push data to
   the provider — the provider polls and retrieves — and link to the
   normative [FHIR R4 Asynchronous Request Pattern](https://hl7.org/fhir/R4/async.html)
   and [FHIR Bulk Data Access STU2](http://hl7.org/fhir/uv/bulkdata/STU2/).
3. Update the existing §6.2.1 interaction diagram (or its caption) so that
   it shows the polling loop and the separate NDJSON retrieval step,
   rather than a single "response" arrow back from the payer. If the
   diagram is regenerated, refresh both `input/images-source/` and
   `input/images/` per the README's manual `image-sources/pdex` workaround.
4. Add a row to `input/pagecontent/changehistory.md` referencing FHIR-53616.
5. Re-run `_updatePublisher` + `_genonce` so that `fsh-generated/`,
   `output/`, and the template change-history copies regenerate; do not
   edit them by hand.

#### Justification

The reporter is factually correct: FHIR Bulk Data is a three-step
asynchronous pattern (kick-off → poll → retrieve), and a diagram that
implies the payer "sends" the result back when complete misrepresents the
specification. The fix is narrative-only on a single page, with no
profile, ValueSet, or operation impact, so the cost is minimal. It is also
fully consistent with the language already adopted by FHIR-49943 / FHIR-53256
on the payer-to-payer and bulk-member-match pages, so accepting the change
restores internal consistency across PDex's bulk narratives. The change is
already drafted in commit `ab42fba4` and the ticket is labelled
`Pre-Applied`, so accepting it just confirms the existing draft.

### Disposition B: Alternative Approach

#### Proposal

Instead of restating the async pattern inline in §6.2.1, replace the
ambiguous diagram and one-paragraph description with a single short
paragraph that **normatively defers** to two upstream specifications:

- The [FHIR R4 Asynchronous Request Pattern](https://hl7.org/fhir/R4/async.html), and
- The [FHIR Bulk Data Access IG STU2 Export flow](http://hl7.org/fhir/uv/bulkdata/STU2/export.html)
  (kick-off, status request, file request).

Keep the existing §pdex statements that bind PDex's Provider Bulk Access
API to those upstream behaviours (e.g. the existing `Prefer: respond-async`
requirement, plus a single new SHALL: "the Provider Access bulk export
**SHALL** follow the FHIR Bulk Data export interaction pattern, including
the kick-off response with `Content-Location`, polling of the status
endpoint, and retrieval of NDJSON output files"). Replace the misleading
diagram with the standard Bulk Data export sequence diagram (or a link to
it) rather than maintaining a PDex-specific copy. Do the same `changehistory`
+ regeneration steps as in Disposition A.

#### Justification

The async pattern is fully normative in FHIR Core and the Bulk Data IG;
PDex doesn't add any provider-specific async behaviour. Re-stating the
flow inline (as in Disposition A) creates a maintenance liability — if the
Bulk Data IG clarifies polling or 202/200 semantics in a future version,
PDex's local restatement risks drifting out of sync. A "delegate by
reference" approach is a common HL7 IG pattern and matches how PDex already
references US Core and HRex rather than restating their constraints.

The trade-off is reduced self-containment of the page: implementers must
follow the link to the Bulk Data IG to read the full flow, where Disposition
A keeps the description local and easier to scan. Given that PDex readers
are predominantly payer/provider implementers who likely already know the
Bulk Data spec, this is a defensible trade.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the existing requirement —
"`Prefer: respond-async` **SHALL** be present" — already binds the
operation to the FHIR Asynchronous Request Pattern by reference, and the
diagram is illustrative rather than normative. Leave §6.2.1 unchanged; if
needed, add only a one-line clarification under the diagram noting that the
"response" arrow represents the eventual NDJSON retrieval, not a push from
the payer.

#### Justification

A defensible argument for decline is that the Provider Access page already
mandates the async header, and the FHIR R4 async pattern unambiguously
defines what that means; a reader who follows the normative reference will
not misinterpret the flow. Spending IG real-estate on a re-explanation
duplicates content normatively owned by FHIR Core / Bulk Data IG, and a
"diagram is illustrative" disclaimer is a minimal, low-risk edit.

This rationale is weak in practice: the reporter is an experienced PDex
reviewer who *was* misled by the current diagram, which is a strong signal
that the illustrative content is materially misleading rather than merely
informal. The "Pre-Applied" label and the existing commit also indicate
the workgroup has already implicitly accepted that the page needs more than
a one-line tweak.

---

### Recommendation

**Recommended disposition:** A

The change has already been authored, committed (`ab42fba4`), and recorded
in the change-history (`changehistory.md` / generated `changehistory.html`),
and the ticket carries the `Pre-Applied` label and a `Persuasive` resolution.
That tells the workgroup that consensus has effectively been reached during
drafting; the formal disposition should ratify what is already in the
source tree rather than rework it. Disposition A also produces the most
readable Provider Access API page for implementers: the kick-off → poll →
retrieve pattern is spelled out where they will look for it, mirroring the
treatment FHIR-49943 / FHIR-53256 gave to the sibling Payer-to-Payer and
Bulk Member-Match pages, which is exactly the consistency the reporter is
asking for. Disposition B's "delegate by reference" idea is a reasonable
maintenance argument, but it is best raised as a future cleanup spanning
all three bulk pages at once, not as a reason to undo a clarification the
ticket reporter explicitly requested. Disposition C is not defensible
given the reporter (and presumably other readers) was misled by the
current text.
