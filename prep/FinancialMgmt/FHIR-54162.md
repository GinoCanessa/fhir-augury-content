## Summary

The ticket reports a typing inconsistency in a newly added `statusReason`
element owned by the Financial Management work group. The element has been
introduced as a `string [0..1]`, whereas the predominant pattern across
existing FHIR resources is to type `statusReason` as `CodeableConcept` so it
can carry both a coded value and free text. The reporter (Eric Haas) cites
`MedicationRequest.statusReason` and `ServiceRequest.statusReason` as
exemplars of the established pattern. The ticket has been triaged as
Persuasive, with the recorded resolution being to change the element to
`CodeableConcept`, migrate the existing string content into
`CodeableConcept.text`, and create an example `CodeSystem` and `ValueSet` to
seed coded usage. The change is flagged as an FM-R6 breaking change and is
queued under `FM-R5+_ToBeApplied`. No related Jira tickets, Zulip threads, or
GitHub items were surfaced via cross-references; the only cross-references are
the two FHIR artifact mentions used as exemplars.

## Details

**Description:**

> The newly added `statusReason` element is typed as `string`:
>
> > "statusReason : string [0..1]"
>
> However, most FHIR resources with status reason use `CodeableConcept` to
> allow coded values (e.g., MedicationRequest.statusReason,
> ServiceRequest.statusReason). Using `string` means:
>
> - Inconsistent with the pattern used elsewhere in FHIR

**Resolution Description:**

> Change statusReason to codeableConcept.
> This enables existing string input to be added to the
> codeableConcept.text element.
>
> Create an example codeSystem and valueSet.
>
> For next ballot this will be included in a list of Breaking Changes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Retype the newly added Financial Management `statusReason` element from
`string [0..1]` to `CodeableConcept [0..1]`, matching the prevailing pattern
exemplified by `MedicationRequest.statusReason` and
`ServiceRequest.statusReason`. As part of the same change:

- Migrate any existing string content into `CodeableConcept.text` so that
  pre-change instances can be losslessly upgraded.
- Create a new example `CodeSystem` and a bound `ValueSet` (with an
  `example` or `preferred` binding strength, consistent with how other FM
  `*statusReason`-style elements are bound) to seed coded usage.
- Update the resource's narrative/intro and any examples/test cases to use
  the new type, and add the change to the FM section of the R6 Breaking
  Changes list.
- Apply via the `FM-R5+_ToBeApplied` queue per the existing label.

#### Justification

This is the action specified in the ticket's recorded resolution
("Persuasive") and aligns the element with the well-established FHIR
pattern of `CodeableConcept` for status-reason semantics, which both the
ticket and its FHIR artifact cross-references
(`MedicationRequest.statusReason`, `ServiceRequest.statusReason`) call out
explicitly. `CodeableConcept.text` provides a clean migration path for any
implementer that is already populating the element as free text, which
keeps the breaking surface to the type signature itself rather than to
content semantics. Because the element was newly added (not yet in a
normative release), the breaking change is acceptable and the ticket is
already labelled `FM-R6-BreakingChange`, signalling that the work group
expects to absorb this in R6.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the element typed as `string` in the base resource but add a standard
extension (e.g., `http://hl7.org/fhir/StructureDefinition/statusReason-coded`
or an FM-scoped equivalent) of type `CodeableConcept` for implementations
that need to carry a coded value. Optionally retype to `CodeableConcept` in
a future release once a minimum viable `ValueSet` exists, rather than
shipping a half-populated example terminology in R6.

#### Justification

This avoids a breaking type change in R6 and lets the FM work group ship the
coded-value capability incrementally without committing to a normative type
choice before the value set is well understood. It also limits churn for
downstream IGs that have already started profiling the new element as a
string. The trade-off is that the inconsistency the ticket complains about
persists in the base resource, implementers must adopt a separate extension
URL to carry codes, and FM ends up with a non-standard shape compared with
`MedicationRequest`/`ServiceRequest`. It also creates more long-term
maintenance burden than Disposition A because a future "real" retype would
itself be a breaking change layered on top of an extension that then has to
be deprecated.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave the element typed as `string [0..1]`. Document
in the resource's notes that the element is intentionally a free-text reason
because the FM work group has no use cases for coded statusReason values on
this resource, and direct implementers who need codes to use a profile-level
extension.

#### Justification

A defensible rationale for declining would be that the FM work group has
deliberately scoped this element to free-text capture (e.g., user-entered
denial/cancellation explanations) and has no concrete use cases or candidate
terminologies justifying a `CodeableConcept`. However, the ticket has
already been triaged as **Persuasive** with a resolution that explicitly
accepts the change, and the reporter cites two well-known precedents
(`MedicationRequest.statusReason`, `ServiceRequest.statusReason`) where the
opposite choice was made, so this disposition is presented mainly for
completeness and would require the work group to actively reverse its
existing triage.

---

### Recommendation

**Recommended disposition:** A

Disposition A matches the ticket's existing Persuasive resolution and the
pattern set by sibling FM and clinical resources such as
`MedicationRequest.statusReason` and `ServiceRequest.statusReason`, which
the reporter explicitly cites. Because the element is newly added and not
yet normative, retyping it now is the cheapest moment to fix the
inconsistency — every release that ships the `string` form increases
migration cost for implementers. Migrating the existing string into
`CodeableConcept.text` keeps backwards-compatibility for content while
enabling coded usage going forward, and seeding an example `CodeSystem` and
`ValueSet` gives the FM community a concrete starting point to refine the
terminology in subsequent ballots. Disposition B's extension-based path
preserves short-term stability but locks in the very inconsistency the
ticket is reporting and creates a worse breaking change later. Disposition C
contradicts the existing Persuasive triage and the cited precedents, and
should only be revisited if the work group can produce specific use cases
that argue against coded values on this element.
