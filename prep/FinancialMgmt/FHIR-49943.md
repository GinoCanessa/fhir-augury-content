## Summary

Daniel Venton flagged that the `payertopayerbulkexchange` page in US Da
Vinci PDex (STU 2.1) makes a single combined SHALL statement —

> "The PDex `$multi-member-match` and the subsequent `$davinci-data-export`
> operations SHALL be submitted using a HTTP POST. The HTTP Header SHALL
> include: `Prefer: respond-async`"

— which implies that `$multi-member-match` itself is an asynchronous
operation, while the `OperationDefinition-BulkMemberMatch` resource is
written as a synchronous operation that returns inline `Matched`,
`NotMatched` and `ConsentConstrained` group `out` parameters. The ticket
asks the work group to either (a) clarify that only `$davinci-data-export`
is async and `$multi-member-match` is sync, or (b) modify the
`$multi-member-match` operation definition to be async.

The closely-related ticket [FHIR-53256](https://jira.hl7.org/browse/FHIR-53256)
("Multi-member-match response spec does not fit with async pattern",
filed by Casey Trauer) takes the second branch further: the IG already
*says* `$multi-member-match` SHALL be async (per
`payertopayerbulkexchange.html`), but the OperationDefinition still
declares the response as inline `out` Group parameters, which is
inconsistent with the FHIR R4 Async Request Pattern (which would deliver
the response via an ndjson manifest URL, like `$davinci-data-export`
does). FHIR-53256 asks for the diagram, narrative, and OperationDefinition
to be aligned around the async-with-manifest pattern.

The Da Vinci PDex Zulip "Bulk Member Match" topic (running from May 2024
through March 2025) shows the community had been arguing both
interpretations: Mark Scrimshire said in late 2024 that he considered
member match synchronous and wanted Connectathon testing to size-bound
it; Alexander Zautke, Casey Trauer, Jan Vissers and Michele Mottini
argued the opposite based on the IG wording, large-group timeout risk,
and the fact that real implementations may need to call MPIs / claim
processing / manual review. The ticket was filed during that thread.

The ticket has been resolved as **Persuasive with Modification** with a
three-step fix that lands on the async-with-manifest interpretation:
(1) split the combined SHALL sentence into two separate, unambiguous
async statements; (2) update the `$multi-member-match` (a.k.a.
`$bulk-member-match`) OperationDefinition to remove the inline Matched /
NotMatched / ConsentConstrained `out` parameters and document delivery
via the async completion manifest, mirroring `$davinci-data-export`;
(3) add an explanatory paragraph at the top of section 6.4.1 stating
that the Payer-to-Payer bulk flow is two sequential async operations,
each with its own kick-off / poll / retrieval steps. The fix has been
applied in commit `7712233c` ("Resolve FHIR-49943: clarify async usage
of `$multi-member-match`") on `HL7/davinci-epdx` and is recorded in the
IG changehistory.

## Details

**Description:**

> "The PDex $multi-member-match and the subsequent $davinci-data-export operations **SHALL** be submitted using a HTTP POST. The HTTP Header **SHALL** include:"
>
> Seems to state that the $multi-member-match is an async method. But the description of the operation is sync.
>
> Clarify this sentence to state that the $davinci-data-export is async and mmm is sync or modify the mmm operation definition to be an async method.

**Comments:**

- **JIRAUSER17326 (Daniel Venton, 2025-03-05):** Suggests that all
  references to "Bulk Member Match" (operation name `$multi-member-match`)
  be renamed to "Multi-Member Match" to remove any implied "works like the
  bulk operations do" reading.
- **michelemottini (2025-07-14):** Points out the IG narrative is
  stronger than an implication — `payertopayerbulkexchange.html` already
  states the `$multi-member-match` operation SHALL be performed as an
  asynchronous operation. Notes that "OperationDefinition is the same
  for async or sync" and argues async makes sense because member match
  may involve calls to MPIs, claim processing, or manual review.

**Resolution (Persuasive with Modification):**

Three steps applied:

1. **Split the combined sentence.** Replace the combined SHALL with two
   separate, unambiguous statements — one declaring `$multi-member-match`
   is a HTTP POST that SHALL be performed asynchronously following the
   FHIR R4 Async Request Pattern + Bulk Data Status Request protocol;
   one declaring `$davinci-data-export` is a HTTP POST performed as an
   asynchronous Bulk Data export per the FHIR Bulk Data Access IG.
2. **Update the `$bulk-member-match` OperationDefinition** to remove the
   inline `out` parameters (`Matched`, `NotMatched`,
   `ConsentConstrained` groups) and document instead that the response
   is delivered asynchronously via a manifest URL — consistent with how
   `$davinci-data-export` handles its output. This is the deeper fix
   that resolves both this ticket and FHIR-53256.
3. **Add an explanatory paragraph** at the top of section 6.4.1 making
   explicit that the entire Payer-to-Payer bulk flow consists of *two
   sequential async operations*, each with its own kick-off, polling,
   and retrieval steps — rather than one synchronous operation followed
   by one async one.

## Related Zulip Discussions

### [Da Vinci PDex > Bulk Member Match](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PDex/topic/Bulk%20Member%20Match/near/503286643)

Long-running thread (May 2024 – Sept 2025) in which the
community surfaces and debates the exact ambiguity this ticket
captures. Alexander Zautke opens by noting the IG lacks language saying
the operation should be executed asynchronously and asks for
confirmation; Mark Scrimshire confirms it is intended to be async.
Michael Taylor reports asking the same question on a PDex call months
earlier and being told it was *currently* synchronous but might need to
change. Daniel Venton argues bulk member match is not async ("you aren't
asking for GB of information", and the diagram shows a direct
execute/get-id flow with no polling step). Alexander counters that the
issue is *time-to-match* on large groups, not payload size, and that
implementations will time out without async. Mark Scrimshire (Dec 2024)
states he currently considers member match *synchronous* and proposes
Connectathon testing to set size limits. Casey Trauer (Mar 2025) then
cites the IG text — "`Prefer: respond-async`" — and concludes the IG
explicitly says it's async; Daniel Venton asks for a citation, Casey
provides it, Daniel files this Jira ticket the same day. Jan Vissers
(May 2025) confirms he also reads the IG as async and is waiting for the
ticket. Michele Mottini (Jul 2025) reiterates that async makes sense
because of MPI / claim processing / manual review. Devarajan Vairaperumal
(Sep 2025) asks what the polling endpoint is; Daniel Venton clarifies
there is no fixed status endpoint — the kick-off response carries the
status URL. Net consensus by the resolution date: async-with-manifest
is the right interpretation, which is what the resolution adopts.

Participants: Alexander Zautke, Mark Scrimshire, Michael Taylor, Daniel
Venton, Kyle Tilman, Casey Trauer, Jan Vissers, Michele Mottini,
Devarajan Vairaperumal.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the applied resolution exactly: in `HL7/davinci-epdx`, split the
combined SHALL sentence in
`input/pagecontent/payertopayerbulkexchange.md` (section 6.4.1) into
two unambiguous statements — one declaring `$multi-member-match` SHALL
be HTTP POST + `Prefer: respond-async` and SHALL follow the FHIR R4
Async Request Pattern + Bulk Data Status Request protocol; one
declaring `$davinci-data-export` SHALL be HTTP POST and SHALL be
performed as an asynchronous Bulk Data export per the FHIR Bulk Data
Access IG. In `input/fsh/Operation-bulk-member-match.fsh`, remove the
inline `Matched`, `NotMatched`, and `ConsentConstrained` Group `out`
parameters and document that the response is delivered via the async
completion manifest URL (mirroring `$davinci-data-export`'s
OperationDefinition shape). Add a short opening paragraph at the top
of section 6.4.1 making explicit that the Payer-to-Payer bulk flow is
two sequential async operations, each with its own kick-off / polling
/ retrieval. Update `input/pagecontent/changehistory.md` with the
FHIR-49943 (and FHIR-53256) entries under STU 2.1.0-ballot
reconciliation.

#### Justification

This is the resolution that has already been applied (commit
`7712233c` on `HL7/davinci-epdx`, marked "Pre-Applied" on the
ticket and the linked FHIR-53256). It picks the option the IG
narrative had already half-committed to (the existing
`Prefer: respond-async` / "SHALL be performed as an Asynchronous
operation" wording on `payertopayerbulkexchange.html`), aligns the
OperationDefinition with the same async-manifest pattern that
`$davinci-data-export` already uses (so implementers can reuse a
single async-polling code path for `$bulk-member-match`, `$export`,
and `$davinci-data-export` — the explicit goal Casey Trauer cites in
FHIR-53256), and addresses the timeout-on-large-groups concern
Alexander Zautke and others raised on Zulip. It also keeps both
`$multi-member-match` and `$davinci-data-export` describable in a
single architectural sentence ("two sequential async operations"),
which materially reduces the implementer ambiguity that Daniel
Venton, Casey Trauer, Michael Taylor, Jan Vissers and Devarajan
Vairaperumal hit independently on Zulip. The repo briefing flags the
flat `input/fsh/` layout as the only structural gotcha for the FSH
edit; the change does not cross repos.

---

### Disposition B: Alternative Approach

#### Proposal

Take only the *narrative* half of the fix and leave the
OperationDefinition shape alone. Specifically: split the combined SHALL
sentence in `payertopayerbulkexchange.md` so it is unambiguous that
*both* `$multi-member-match` and `$davinci-data-export` are async
HTTP POSTs that use `Prefer: respond-async`; keep the existing
`OperationDefinition-BulkMemberMatch` with its `Matched`, `NotMatched`,
and `ConsentConstrained` `out` Group parameters intact, and instead add
narrative + a sequence-diagram update under
`input/pagecontent/payertopayerbulkexchange.md` and
`input/images-source/` clarifying that the OperationDefinition's `out`
parameters describe the *logical* response shape, while on the wire
the response is delivered as an ndjson file via the async manifest URL
(documented as a non-normative implementation note rather than a
structural OperationDefinition change). Optionally also rename
"Bulk Member Match" → "Multi-Member Match" in headings and section
labels per Daniel Venton's first comment, to remove the implied
"works like Bulk Data Access" reading without altering the operation
canonical or `id`.

#### Justification

This addresses the literal text Daniel Venton flagged (the combined
SHALL sentence) and the renaming idea from his comment, while
minimising the structural impact on existing PDex implementers. The
Zulip thread shows real implementations have already shipped against
the existing `out`-parameter shape (Mark Scrimshire stated as recently
as Dec 2024 that he considered it sync; Devarajan Vairaperumal asked
in Sep 2025 about a status endpoint, implying live integration work).
Removing the `out` parameters from the published OperationDefinition
(Disposition A, step 2) is a visible STU change to a normative
artifact mid-flight; pushing that into a documentation note instead
keeps the resolution scoped to narrative, defers the deeper structural
realignment to a future major version, and avoids breaking any
implementer harness that currently keys on the inline Group response.
The trade-off versus Disposition A is the cost it imposes on FHIR-53256:
that ticket is *specifically* asking for the OperationDefinition
realignment, so this disposition would partially decline FHIR-53256
and lean on the narrative note to bridge the gap.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that the existing
`payertopayerbulkexchange.html` text — taken together with the
`Prefer: respond-async` HTTP header requirement and the explicit "SHALL
be performed as an Asynchronous operation" sentence elsewhere on the
same page — is already sufficient to convey that *both* operations
are async, and that the OperationDefinition's lack of async-specific
out-parameter shape mirrors how *every* OperationDefinition is
written (the OperationDefinition resource simply does not have an
"async vs sync" axis in FHIR R4). Direct readers to the existing FHIR
R4 Async Request Pattern (`https://hl7.org/fhir/R4/async.html`) and
to the Bulk Data Status Request section, and resolve the ticket as
"Not Persuasive — already addressed by existing text + the FHIR R4
async pattern reference."

#### Justification

A Decline is defensible if the work group concludes the existing
SHALL + `Prefer: respond-async` wording is normatively sufficient and
the OperationDefinition shape question is a general FHIR-spec issue
rather than a PDex one. However, this is a weak position in light of
the evidence: multiple independent implementers (Daniel Venton, Casey
Trauer, Michael Taylor, Jan Vissers, Devarajan Vairaperumal, Mark
Scrimshire himself) reached *different* conclusions reading the same
text, and Mark Scrimshire — the WG lead in the Zulip thread —
explicitly stated in Dec 2024 that he considered member match
synchronous, which is incompatible with what the IG actually says.
Declining would leave a known, demonstrated source of implementer
divergence in place. Listed for completeness only.

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as "Persuasive with Modification"
and the fix has been applied (commit `7712233c`, "Pre-Applied" label),
so endorsing Disposition A simply ratifies the in-flight resolution.
On its merits, A is also the right answer: it picks the
async-with-manifest interpretation the IG narrative had already
committed to (`Prefer: respond-async` + the "SHALL be performed as an
Asynchronous operation" sentence), and it closes the structural gap
that FHIR-53256 specifically calls out by aligning the
`OperationDefinition-BulkMemberMatch` response shape with the
already-deployed `$davinci-data-export` pattern. The Zulip thread
shows the alternative reading (sync member-match) was real and held
by the work-group lead as recently as late 2024, so leaving the text
as-is (Disposition C) is not viable — the demonstrated implementer
divergence is exactly what the ticket was filed to resolve.
Disposition B (narrative-only fix, keep the inline `out` parameters)
is a defensible fallback if the work group is uncomfortable changing
the published OperationDefinition shape mid-STU; if so, the work
group should explicitly carry the OperationDefinition realignment
forward as a tracked item against FHIR-53256 rather than closing it
silently. Either way, the work group should also weigh Daniel
Venton's renaming suggestion ("Bulk Member Match" → "Multi-Member
Match") as a separate, optional follow-up — it does not block this
resolution and was not part of the applied fix.
