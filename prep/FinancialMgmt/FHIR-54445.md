## Summary

The reporter notes that several `CodeableConcept` elements on the `InsurancePlan`
resource lack any value set binding, which means QA flags them as
informational issues and implementers have no normative or even suggestive
terminology to draw from when populating the resource. The ticket itself does
not enumerate the offending elements, but it sits alongside an essentially
identical, more detailed ticket — [FHIR-50876](https://jira.hl7.org/browse/FHIR-50876)
("InsurancePlan: Missing ValueSet Bindings") — which lists the specific
unbound elements: `InsurancePlan.type`, `InsurancePlan.generalCost.type`,
`InsurancePlan.specificCost.category`, `InsurancePlan.specificCost.benefit.type`,
`InsurancePlan.specificCost.benefit.cost.type`, and
`InsurancePlan.specificCost.benefit.cost.qualifier`. The September 2025 joint
PA/FM WGM minutes recorded on FHIR-50876 split the work: `InsurancePlan.type`
is being handled under [FHIR-24915](https://jira.hl7.org/browse/FHIR-24915)
("Harmonize Coverage and InsurancePlan type" — resolved Persuasive with
Modification, with a draft binding table to `insurance-coverage-type` /
`insuranceplan-plan-type`), leaving only the cost-related properties for
this ticket. Cooper Thompson clarified during that discussion that
`generalCost` was intended to model deductibles and `specificCost` to model
co-pays, which constrains what the missing value sets actually need to
cover. FHIR-54445 therefore reads as a duplicate of FHIR-50876 raised in the
next ballot cycle and should most likely be merged into that ticket's
resolution.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Resolve FHIR-54445 as a duplicate of [FHIR-50876](https://jira.hl7.org/browse/FHIR-50876)
and roll its substance into the FHIR-50876 resolution: add bindings (Example
strength, with reviewers free to upgrade to Preferred where confidence
allows) to the cost-related `InsurancePlan` CodeableConcept elements that
were called out by the QA report —

- `InsurancePlan.generalCost.type` — bind to a new THO value set
  `insurance-general-cost-type` seeded from Da Vinci PDex/Plan-Net
  patterns for deductibles, out-of-pocket maximums, and similar
  plan-level financial obligations (per Cooper Thompson's note that
  generalCost ≈ deductible).
- `InsurancePlan.specificCost.category` — bind to a small THO value set
  enumerating service-line groupings (e.g. medical / dental / vision /
  pharmacy / behavioral) consistent with the categories already used in
  US payer IGs.
- `InsurancePlan.specificCost.benefit.type` — bind (Example) to a value
  set enumerating benefit categories (primary care, specialty care,
  inpatient, outpatient, …), aligned with FHIR-24916's note about
  `coverage.benefit.type`.
- `InsurancePlan.specificCost.benefit.cost.type` — bind (Example) to a
  value set covering co-pay / co-insurance / deductible / out-of-pocket
  flavours of cost.
- `InsurancePlan.specificCost.benefit.cost.qualifier` — bind (Example)
  to a value set covering in-network / out-of-network / preferred /
  tier-1 etc.
- Defer `InsurancePlan.type` to [FHIR-24915](https://jira.hl7.org/browse/FHIR-24915),
  per the joint PA/FM WGM decision recorded on FHIR-50876, and link
  FHIR-54445 → FHIR-50876 → FHIR-24915 so the audit trail is intact.

#### Justification

The change literally fulfils what the reporter asked: every flagged
`CodeableConcept` on `InsurancePlan` ends up with a binding, the QA
INFORMATION lines disappear, and implementers gain at least an Example
"starter set" of codes. Using Example strength on the cost-side bindings
matches the FHIR design principle that a binding *should* be present even
when the WG cannot yet commit to a normative code list, and it mirrors the
draft strengths chosen on FHIR-24915 (`Preferred` / `Example`). Robert
Dieterle has already supplied recommendations in the FHIR-50876 attachment,
so there is concrete material to work from rather than speculative content.
Doing the work under FHIR-50876 (with FHIR-54445 closed as duplicate) keeps
a single conversation, comment thread and resolution attached to the
substantive change.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-54445 as a *terminology placement* ticket rather than a
core-spec change ticket: add only the minimally required bindings on the
core `InsurancePlan` resource (Example strength pointing at
`http://hl7.org/fhir/ValueSet/...` stubs that simply reference
`http://terminology.hl7.org/CodeSystem/...`), and push the substantive
value-set authoring out into THO and the relevant Da Vinci / NDH IGs.
Concretely:

- In the FHIR core spec, change each unbound element to carry an Example
  binding pointing at a THO value set whose initial content can be
  empty-ish or a thin set of starter codes, rather than minting full
  value sets inside the FHIR core build.
- Open companion tickets against THO (and ping Da Vinci PDex / NDH for
  candidate codes) to flesh out the value sets independently of the
  FHIR R6 ballot timeline.
- Continue to defer `InsurancePlan.type` to FHIR-24915 and the broader
  `InsuranceProduct` reshaping work.

#### Justification

The harder problem here is *agreeing on the codes*, not *adding the
binding*. The cost-related properties in `InsurancePlan` are notoriously
under-specified (generalCost vs specificCost, deductible vs co-pay,
in-network vs out-of-network) and FM has been circling them for several
ballot cycles via FHIR-24915, FHIR-24916, FHIR-50876 and now FHIR-54445.
Decoupling "give every CodeableConcept a binding" (a small core-spec edit)
from "produce real value sets" (a terminology effort spanning THO + IGs)
lets FM clear the QA INFORMATION items in the next ballot without locking
in code lists they may later regret. It also avoids the FHIR-24915 risk of
`Preferred` bindings to value sets whose membership is still actively
debated.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54445 as **Not Persuasive — duplicate of FHIR-50876** without a
spec change on this ticket. The reporter is asked to follow FHIR-50876
(and, for the `InsurancePlan.type` portion, FHIR-24915) for status. No
edits to the FHIR core spec are made under FHIR-54445.

#### Justification

FHIR-54445 contains a one-sentence description that adds no information
beyond what is already captured in FHIR-50876 (which itself enumerates the
exact unbound elements and has both a WG decision and Robert Dieterle's
recommendations attached) and FHIR-24915 (which already addresses
`InsurancePlan.type`). The Jira hygiene principle — one substantive
discussion per topic — argues for closing the later, thinner duplicate
rather than maintaining parallel conversations on the same elements with
the risk of divergent resolutions. This is a workflow disposition only;
the underlying request still gets satisfied through FHIR-50876.

---

### Recommendation

**Recommended disposition:** A

The QA report has been calling these elements out for multiple cycles, the
WG has already done the hard scoping work on FHIR-50876 (separating the
`type` discussion from the cost discussion), and Robert Dieterle has
attached concrete value-set recommendations. Disposition A captures all of
that progress, satisfies the reporter literally, and removes the
INFORMATION findings — while explicitly deferring `InsurancePlan.type` to
FHIR-24915 so the WG does not relitigate that decision here. Disposition B
is defensible if the WG does not yet have consensus on the cost code
lists, but in practice the binding strengths chosen on FHIR-24915
(Preferred / Example) already provide the same flexibility without the
overhead of a separate THO ticket per element. Disposition C is the
cleanest outcome from a Jira-hygiene standpoint, but it leaves the
substantive change to be made on a different ticket key than the one the
reporter raised; treating FHIR-54445 as a duplicate-but-rolled-up via
Disposition A gives the reporter visible closure and keeps the audit trail
intact.
