## Summary

The ticket reports that the US Da Vinci PAS profile of `ClaimResponse.outcome`
is mapped to the X12 278 Response BHT06 segment, but because
`ClaimResponse.outcome` is a `code` element with a required binding to a small
FHIR value set (`queued | complete | error | partial`), the full set of X12
BHT06 codes cannot be represented faithfully. The reporter notes that BHT06
codes 18 and 19 map cleanly to `complete` and `partial`, but `AT` (information
held / additional information requested) only partially maps to `partial` and
loses the "needs more information" semantic, while `RU` has no FHIR equivalent
at all. The request is to add an extension on `ClaimResponse.outcome` that can
carry the original X12 BHT06 code so that the full PAS / 278 semantics survive
the round-trip. The ticket has already been triaged with a `Persuasive`
resolution that proposes creating an optional extension on
`ClaimResponse.outcome` to carry the BHT06 code.

There are no direct Jira cross-references on this ticket; the only formal
cross-reference is to the FHIR artifact `ClaimResponse.outcome` itself. Closely
related prior work in PAS includes FHIR-25415 (which previously shifted
`queued/partial` semantics onto `ClaimResponse.reviewAction.code` driven by
HCR01) and FHIR-48616 (which extended the `reviewAction` extension to event
level so HCR-segment information could be communicated). FHIR-55208, against
core FHIR, asked for an "unknown/other" escape valve in the
`ClaimResponseStatus` value set and was resolved Not Persuasive — the present
PAS ticket is, in effect, the IG-level workaround for that same gap, scoped to
PAS rather than core.

## Details

**Description:**

> The ClaimResponse.outcome is currently mapped to BHT06. Since outcome is a
> code datatype and has a required binding, the complete semantics of the X12
> code is missing.
>
> 18 maps cleanly to complete and 19 maps cleanly to partial. AT could map to
> partial, but the extra information that the response has a request for
> additional information is missing. and RU just doesn't have a code.
>
> We need to create an extension to ClaimResponse.outcome that can carry the
> actual X12 BHT06 code.

**Resolution (already recorded on the ticket):**

> We will create the extension to carry the BHT06 code and have it as an
> optional extension on ClaimResponse.outcome.

**Comments:** None on the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Honour the existing `Persuasive` resolution on the ticket: define a new
optional extension in the US Da Vinci PAS IG, applied to
`ClaimResponse.outcome` on the PAS ClaimResponse profile (and, for symmetry,
the PAS Inquiry ClaimResponse profile), that carries the raw X12 BHT06 code.

Concretely:

- New extension `pas-claimresponse-bht06-code` (or similar) with context
  `ClaimResponse.outcome`, cardinality `0..1`, value `code` (or `Coding`)
  bound `required` to a new PAS-defined CodeSystem / ValueSet enumerating the
  X12 278 BHT06 codes actually allowed (at minimum `18`, `19`, `AT`, `RU`).
- Update the PAS ClaimResponse profile to slot this extension in, mark it
  Must Support, and document the BHT06 → outcome mapping rules in the
  X12-to-FHIR mapping table so that `outcome` is the FHIR-value-set bucket
  and the extension carries the precise X12 code.
- Map `AT` → `outcome=partial` + extension=`AT`, `RU` → an explicit
  `outcome` value (`partial` or `queued`, per FM consensus) + extension=`RU`,
  with the extension being the authoritative round-trip value.

#### Justification

This is the pattern the IG already uses for HCR-driven semantics
(FHIR-25415 / FHIR-48616 used `ClaimResponse.reviewAction` to carry the X12
code when the FHIR value set was too coarse). FHIR-55208 confirms that core
FHIR will not loosen the binding on `ClaimResponse.outcome`, so an IG-level
extension is the only durable mechanism. Making the extension Must Support
preserves the BHT06 round-trip required for X12 278 ↔ PAS conformance without
breaking implementers who only consume the base FHIR `outcome` value.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of inventing a new BHT06-specific extension, **reuse and extend the
existing `reviewAction` extension** (already established by FHIR-48616 at the
top-level adjudication / event level) so it carries BHT06 alongside HCR01.

Concretely, define a sibling sub-extension on the existing
`extension-reviewAction` (or a new top-level `extension-reviewDecision`) with
a `code` element bound to a value set that unions the BHT06 codes with the
existing HCR codes, plus a `system` discriminator distinguishing BHT06 vs
HCR. Keep `ClaimResponse.outcome` mapped per the current rules (FHIR value
set only) and document that the BHT06 round-trip lives on the same
event-level extension already used for HCR01 round-trip.

#### Justification

PAS already has one extension (`reviewAction`) carrying X12-segment-code
semantics for ClaimResponse. Adding a second, parallel extension specifically
for BHT06 fragments the X12 round-trip story and forces implementers to look
in two places for "the original X12 code that drove this outcome." Co-locating
BHT06 with HCR01 keeps the X12-side semantics in one extension family and
preserves the principle that `ClaimResponse.outcome` stays a coarse FHIR
bucket. Trade-off versus A: slightly more complex extension shape (must
discriminate BHT06 vs HCR) but a cleaner long-term mapping surface and easier
to extend if other BHT/HCR-style segments need the same treatment later.

---

### Disposition C: Decline

#### Proposal

Decline the request and close as Not Persuasive. Document in the X12-to-FHIR
mapping table that BHT06 codes `AT` and `RU` MAY be approximated by
`ClaimResponse.outcome=partial` (or `queued`) with explanatory text in
`ClaimResponse.disposition`, and rely on the existing
`ClaimResponse.reviewAction` extension (HCR-driven) for the precise
"additional information needed" semantics that BHT06=`AT` carries.

#### Justification

Defensible on the grounds that (a) FHIR-55208 already settled at the core
level that `ClaimResponse.outcome`'s value set will not be relaxed, (b) the
"additional information requested" semantic of BHT06=`AT` is arguably already
expressible via the existing `reviewAction` HCR pathway plus
`disposition`, and (c) BHT06 sits at the X12 transaction-envelope layer where
strict round-trip fidelity has historically been considered the X12 gateway's
problem, not the FHIR profile's. Adding more X12-mirroring extensions risks
"modelling FHIR after X12" (the concern raised in FHIR-36167) and increases
profile surface area for a code set most consumers will never read.

---

### Recommendation

**Recommended disposition:** B

The underlying need — preserving the BHT06 X12 code so a PAS round-trip is
loss-free — is real and is consistent with what the workgroup has already
agreed for HCR01 on the same resource (FHIR-25415, FHIR-48616). The current
ticket resolution (A) addresses that need, but does so by introducing a second,
parallel "carry the X12 code" extension on the same resource, which fragments
the X12-mirroring story across two extension families. Disposition B keeps the
already-Persuasive intent — yes, we add an extension; yes, it's optional / Must
Support; yes, BHT06 round-trips — but co-locates it with the existing
`reviewAction` extension so implementers have a single, consistent place to
look for "the original X12 code behind this outcome." Disposition C is hard to
defend now that FHIR-48616 has set the precedent that PAS *will* extend
`ClaimResponse` to carry X12-segment-level codes; declining only this one
would be inconsistent. The workgroup should treat the existing ticket
resolution (A) as the floor and discuss whether to consolidate onto the
`reviewAction` extension surface (B) before publishing.
