## Summary

The reporter is asking the PDex specification to make it possible (and
reliable) for a requester of `$bulk-member-match` to correlate each entry
in the response back to the Patient resource that was sent in the request.
Today the specification provides an extension on the matched-member entry
that points to a contained Patient representing the requester's input,
but the extension is `0..1`, so a responder could legally omit it; the
example response also labels the contained Patient's identifiers as
"Old Payer" when functionally those identifiers should belong to the
"New Payer"; and the prose does not say whether the contained Patient
must be the exact resource the requester submitted or whether the
original `id` and identifiers must be preserved. The ticket asks the WG
to (1) tighten the cardinality so the cross-reference is guaranteed,
(2) fix the example, and (3) add narrative pinning down what the
contained Patient must contain. The ticket has no cross-referenced Jira
tickets, no Zulip discussion, and no PR/issue links — only the resolution
commit and changehistory entries in `HL7/davinci-epdx`.

## Details

**Description:**

When we get back a $bulk-member-match response, the requester needs to
be able to cross reference matched responses with the Patients that are
part of the request. This is to facilitate the integration of data after
requesters retrieve the exported resources.

Here are the issues as the specification reads today:

1. We have an extension for matched member that supports pointing to a
   contained Patient resource representing what the requester sent.
   However, this extension has a cardinality of `0..1`. So we cannot be
   sure all implementations will include this.
2. The example for `$multi-member-match` response contains identifiers
   in the contained Patient resource labeled "Old Payer". Functionally,
   these should be identifiers from the "New Payer". Given we have so
   little information about what is expected to be in these contained
   resources, we should fix the example to be more clear.
3. Can we have more narrative around this contained resource on whether
   the expectation is that the referenced Patient resource is exactly
   the resource supplied by the requester (rather than, say, an
   abridged version)? We don't want responders determining what is
   important to return.
4. In the same vein, would the contained Patient resource retain the
   original Patient `id` submitted by the requester?

**Resolution (already pre-applied per the `Pre-Applied` label,
disposition `Persuasive`):**

1. **Cardinality enforced** — `ProfilePDexMemberMatchGroup.fsh` lines
   25 and 44: `matchedMember` / `nonMatchedMember` changed from `0..1`
   to `1..1 MS` on both profiles.
2. **Example labels fixed** — `ExampleMultiMemberMatchParameters.fsh`
   `input-1` through `input-4` now use
   `http://example.org/new-payer/identifiers/member` / "New Payer"
   (all 4 occurrences).
3. **Narrative added** — `payertopayerbulkexchange.md` gains a new
   "Cross-Referencing Match Results" subsection with §pdex-139b: the
   contained Patient SHALL be the exact submitted resource;
   responders SHALL NOT abridge or modify it.
4. **Original id requirement** — §pdex-139c: the contained Patient
   SHALL retain the original `id` and all identifiers supplied by the
   requester.

**Comments:** None recorded on the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply exactly the four changes already pre-applied in commit
`848d7eaa` on `HL7/davinci-epdx`:

1. In `input/fsh/ProfilePDexMemberMatchGroup.fsh`, change
   `matchedMember` (line ~25) and `nonMatchedMember` (line ~44) from
   `0..1` to `1..1 MS` on both the matched-member and non-matched-member
   PDex profiles, so every response entry is guaranteed to carry the
   contained-Patient cross-reference and implementers must support
   reading it.
2. In `input/fsh/ExampleMultiMemberMatchParameters.fsh`, change
   `input-1` through `input-4` so the contained Patient identifiers use
   `system = http://example.org/new-payer/identifiers/member` with
   display "New Payer" (replacing the current "Old Payer" labeling).
3. In `input/pagecontent/payertopayerbulkexchange.md`, add a new
   "Cross-Referencing Match Results" subsection containing
   conformance statement §pdex-139b: the contained Patient in each
   matched / non-matched member entry SHALL be the exact Patient
   resource submitted by the requester; responders SHALL NOT abridge,
   summarise, or otherwise modify it.
4. In the same subsection, add §pdex-139c: the contained Patient
   SHALL retain the original `id` and all `identifier` values supplied
   by the requester, so requesters can join responses back to their
   request payload by `id` and identifier.

After the FSH / narrative changes, refresh
`Requirements-fromNarrative.json` and `conformance_statements.csv` via
`process_conformance.py` so the new SHALL statements are tracked, and
add a corresponding row to `input/pagecontent/changehistory.md`.

#### Justification

The four changes line up exactly with the four numbered issues in the
ticket and are already implemented and `Pre-Applied` (resolution =
Persuasive), so accepting matches both the reporter's intent and the
work the IG team has already done. Tightening to `1..1 MS` is the only
way to make cross-referencing reliable across implementations — leaving
it `0..1` makes the cross-reference unusable in practice for any
requester that has to integrate exported resources from multiple
responders. Fixing the example removes a contradiction between the
narrative and the example (the response side belongs to the New Payer,
not the Old Payer). Adding §pdex-139b/§pdex-139c removes the open
question about responder discretion and pins down `id` / identifier
preservation, which is what makes the cross-reference deterministic.
The change is contained to one IG (`HL7/davinci-epdx`), the flat
`input/fsh/` layout makes the edits straightforward, and there is no
HRex or US Core dependency change implied.

---

### Disposition B: Alternative Approach

#### Proposal

Accept items 2, 3 and 4 as written, but **do not** raise cardinality
on the matched/non-matched member element to `1..1 MS`. Instead:

- Keep `matchedMember` / `nonMatchedMember` as `0..1` on the base PDex
  profile but add a stricter derived profile (e.g.
  `ProfilePDexMemberMatchGroup-Linked`) that constrains both to
  `1..1 MS` and reference it from the
  `pdex-payer-access-server-capability-statement.fsh` /
  payer-to-payer capability statement as the conformance target for
  bulk member-match responses.
- Mark the existing extension-pointer pattern as `MS` on the base
  profile (so implementers must support reading it where present) and
  add §pdex-139d clarifying that responders that omit it cannot
  participate in bulk payer-to-payer exchange.

Items 2–4 (example fix, §pdex-139b, §pdex-139c) are unchanged from
Disposition A and live in the same files
(`ExampleMultiMemberMatchParameters.fsh`,
`payertopayerbulkexchange.md`).

#### Justification

A profile-level `0..1` → `1..1` change is a breaking constraint for
any responder that has already declared conformance to the current
PDex 2.x profile and is producing responses without the contained
Patient. Splitting the strict cardinality into a derived profile keyed
off the capability statement preserves backward compatibility for
existing PDex conformers while still giving requesters a deterministic
profile to demand of responders that participate in payer-to-payer
bulk exchange. The trade-off versus Disposition A is more profile
surface area (one extra profile + one extra capability-statement
reference) and slightly weaker baseline guarantee — but it avoids
silently invalidating instances that conformed to PDex 2.x. The
narrative + example fixes still close the substantive ambiguity in
items 2–4, which is where the integration risk actually lives.

---

### Disposition C: Decline

#### Proposal

Decline the cardinality change and the new conformance statements.
Leave `matchedMember` / `nonMatchedMember` at `0..1`, leave the
existing extension-based contained-Patient pointer as the (optional)
mechanism, and rely on the existing HRex `$member-match` semantics
plus implementer guidance in connectathons to communicate that the
contained Patient should mirror the request. Optionally accept item 2
(the example identifier-label fix) as a clarifying errata-only change
since it has no normative effect.

#### Justification

Declining is defensible if the WG concludes that (a) the underlying
need is already addressed by the HRex `$member-match` definition that
PDex composes on top of, and (b) tightening cardinality and adding
SHALL statements about the contained Patient's exact byte-for-byte
content risks over-constraining responders that legitimately need to
strip PHI from contained resources for downstream redaction or audit
reasons. Declining keeps the PDex 2.x profile surface and conformance
language stable, avoids any breaking impact on existing implementers,
and defers the cross-referencing problem to operational guidance that
can evolve outside the published IG.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right answer. The reporter is identifying a
genuine integration defect: a `0..1` cross-reference is unusable for
the requester because the requester cannot tell, per response, whether
the absence of the contained Patient means "no match" semantics, "this
responder does not implement the cross-reference", or "this responder
implements it but chose to omit it for this entry". The PDex resolution
disposition is already `Persuasive`, the changes are `Pre-Applied` in
commit `848d7eaa`, and the edits are confined to one IG repo
(`HL7/davinci-epdx`) on well-understood files
(`ProfilePDexMemberMatchGroup.fsh`,
`ExampleMultiMemberMatchParameters.fsh`,
`payertopayerbulkexchange.md`) with no FhirCore, US Core, or HRex
breakage implied. Disposition B's backward-compatibility argument is
weak in practice — payer-to-payer bulk exchange is still maturing in
PDex 2.x and the cost of an extra derived profile + capability
statement reference outweighs the benefit relative to a clean `1..1 MS`
upgrade now. Disposition C ignores the substantive ambiguity called
out in items 2–4 (example mislabeling and silent responder discretion
over the contained Patient) and would leave requesters writing
implementation-specific reconciliation code, which is exactly what the
PDex bulk-member-match flow is supposed to prevent. The WG should
ratify the pre-applied change set and move on.
