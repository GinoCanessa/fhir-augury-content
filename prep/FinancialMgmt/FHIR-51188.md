## Summary

The submitter operates a FHIR store that is loaded incrementally on a daily
basis and exposes Bulk Data Export through the FHIR Bulk Data Access API. They
need a way to convey **inactive / terminated provider records** to Bulk Data
consumers so that downstream systems can retire records that have been
deactivated since a previous export. The reporter explicitly cites the PDex
Implementation Guide (referencing a "PDEX PG 1.1.0" version) and asks what the
recommended pattern is for surfacing inactive provider records through Bulk
FHIR APIs.

The ticket is filed against US Da Vinci PDex but the data in question is
**provider directory data**, which lives in the Da Vinci Plan-Net IG, not
PDex. The single comment from `ekivemark` (Mark Scrimshire) flags that PDex
has never published a 1.1.0 release and asks which IG is actually being
referenced. The ticket has been triaged with resolution
*Considered - Question answered* and a resolution note directing the
submitter to Plan-Net for any provider-data questions.

No related Jira tickets, Zulip discussions, or GitHub items were surfaced by
the cross-reference index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a substantive question against PDex and add a normative
"Inactive / Terminated Resources in Bulk Export" section to the PDex IG
narrative explaining how a Bulk Data server should communicate
deletions/inactivations of provider records to consumers. Specifically:

- Adopt the FHIR Bulk Data Access conventions for **deleted resources** —
  emit a `deleted` output set in the `$export` manifest containing
  transaction bundles whose `entry.request.method = DELETE` referencing the
  deleted resource (the pattern already specified by Bulk Data Access IG
  v2.0.0).
- Clarify that PDex servers MUST set `Practitioner.active`,
  `PractitionerRole.active`, `Organization.active`,
  `Location.status`, and `HealthcareService.active` (and the corresponding
  `period.end`) on inactive records, and that consumers running an
  incremental `_since`-based export MUST treat both *deleted* outputs and
  *active = false* records as evidence the record is no longer in use.
- Add a worked example to the PDex narrative showing the manifest shape
  with `output`, `deleted`, and `error` sections.

#### Justification

The submitter's underlying need — communicating provider-record
deactivation to Bulk Data consumers — is real and recurring for any payer
running incremental ingestion. The Bulk Data Access IG already defines the
`deleted` output mechanism, so this disposition would not invent new
machinery; it would simply make PDex's expectations explicit so payers and
consumers converge on the same pattern. PDex routinely re-states Bulk Data
constraints relevant to its scenarios (e.g., `_since`, `Group` export), so
calling out deletion semantics fits the IG's editorial pattern.

---

### Disposition B: Alternative Approach

#### Proposal

Decline to amend PDex and instead **route the ticket to Da Vinci Plan-Net**,
where provider-directory semantics live, and let Plan-Net author the
guidance. Concretely:

- Re-file (or clone) the ticket against the Da Vinci Plan-Net specification
  with the same description.
- In Plan-Net, add a "Bulk Data Export of Inactive Providers" subsection
  that combines (a) the `active` / `period.end` flagging conventions for
  Practitioner, PractitionerRole, Organization, Location, and
  HealthcareService; (b) the Bulk Data Access `deleted` output mechanism;
  and (c) guidance on whether servers should hard-delete vs. mark inactive
  (Plan-Net's preference is overwhelmingly soft-deletion via `active =
  false`).
- Leave PDex with at most a one-line cross-reference pointing readers to
  the new Plan-Net section.

#### Justification

The resolver's note already states this question belongs to Plan-Net, and
the ekivemark comment shows that even the version cited by the reporter
(PDex 1.1.0) does not exist — the reporter is conflating two IGs. Provider
directory data is squarely Plan-Net's remit; PDex's payer-to-payer member
data flow is a different problem domain. Putting the answer in Plan-Net
keeps the guidance with the resources it governs and avoids creating
duplicate, drift-prone normative text in two IGs. This also respects the
work-group boundary: Plan-Net is FM-owned but its provider-directory
content is more naturally maintained alongside the rest of Plan-Net than
inside PDex.

---

### Disposition C: Decline

#### Proposal

Close the ticket with the existing resolution (*Considered - Question
answered*) on the grounds that it is (a) a usage question rather than a
spec defect, and (b) already addressed by external normative content,
specifically:

- The **FHIR Bulk Data Access IG** already defines the `deleted` output set
  in the `$export` manifest for communicating deletions to consumers.
- The base FHIR resources already provide `active` / `status` /
  `period.end` for marking inactive providers.
- The submitter cited a non-existent PDex version (1.1.0); without a real
  defect against published normative text, no spec change is warranted.

Direct the submitter to the Bulk Data Access IG and the Da Vinci Plan-Net
IG, and close.

#### Justification

The ticket type is *Question*, not *Change Request*, and the resolution
field is already *Considered - Question answered*. The mechanism the
submitter is asking about already exists in Bulk Data Access; nothing in
PDex prevents its use. There is no community discussion (no Zulip threads,
no cross-referenced tickets, no GitHub PRs) suggesting this is a recurring
gap in the IG narrative — it appears to be a single implementer asking
"how do I do X?" rather than evidence of normative ambiguity. Spending IG
real estate on a re-statement of Bulk Data behaviour duplicates content
and risks drift.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need is legitimate but is unambiguously
provider-directory territory, which is Plan-Net, not PDex — both the
resolver's note and the ekivemark comment make that clear, and the
reporter even cited a PDex version that does not exist. Disposition C
(close as answered) is technically defensible but leaves the next
implementer to ask the same question; the lack of public discussion is
just as easily explained by "no one knew where to file it" as by "no one
needs it." Disposition A puts the answer in the wrong IG and risks drift
with Plan-Net. Disposition B captures the answer once, in the correct IG,
and is consistent with how FM has historically partitioned provider vs.
member data between Plan-Net and PDex. The actual normative content
Plan-Net would adopt is already settled in Bulk Data Access (use the
`deleted` output set) plus the existing `active` / `period.end` flags, so
the editorial lift is small.
