## Summary

FHIR-35395 requests that the canonical (defining) URL for the
`C4BBInstitutionalClaimSubType` CodeSystem in the US CARIN Blue Button
(C4BB) IG be re-anchored to `terminology.hl7.org` (THO) instead of
`hl7.org/fhir/...`, and that a corresponding UTG (Unified Terminology
Governance) ticket be filed to register the CodeSystem in THO. The
request originated as "Comment 7" imported by Ron G. Parker against C4BB
STU2 ballot.

The ticket was first dispositioned by the Financial Management WG on
2022-07-26 as **Considered for Future Use** (block vote, Corey Spears /
Kathleen Conner, 13-0-0) with the rationale that the CodeSystem may be
moved to THO in a future version based on harmonization work with other
IGs. On 2022-11-28 lynn_laakso reverted that disposition and re-opened
the ticket "post STU2 publication", returning it to **Triaged** status
so the WG can re-decide for the next C4BB cycle.

The cross-reference set surfaces only the original 2022-07-19 Block
Vote 6 announcement on Zulip; no explicit Jira issue links and no
GitHub PRs/issues/commits are recorded against this key. However, the
Zulip thread shows that FHIR-35395 is one of a large family of
sibling tickets (FHIR-35390, -35391, -35392, -35393, -35396, -35398,
-35399, -35401–-35407, -35409, -35413–-35417, -35424–-35428, -35431–
-35447, plus FHIR-36486/36487) all requesting the same
THO-anchoring change for other C4BB CodeSystems and ValueSets. Any
disposition for FHIR-35395 should be considered together with these
siblings to keep the C4BB terminology story consistent.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the `C4BBInstitutionalClaimSubType` CodeSystem to THO and
update C4BB to consume it from there. Concretely:

1. File a UTG ticket against `HL7/UTG` proposing the new THO canonical
   `http://terminology.hl7.org/CodeSystem/C4BBInstitutionalClaimSubType`
   (or the harmonized name agreed during UTG review), carrying over
   the existing concept set, definitions, and properties unchanged.
2. Once the UTG submission is accepted and published in a THO release,
   in `HL7/carin-bb` (authoring root for the IG):
   - Replace the in-IG CodeSystem resource with a reference to the
     external THO CodeSystem (or retire the in-IG copy and add a
     `notation` redirect / deprecation note as the WG prefers).
   - Update every ValueSet that currently includes
     `http://hl7.org/fhir/us/carin-bb/CodeSystem/C4BBInstitutionalClaimSubType`
     so the `compose.include.system` points to the new THO canonical.
   - Update any profile bindings (notably the C4BB Institutional EoB
     profile's `subType` binding) to reference the THO-anchored
     ValueSet.
   - Add migration / change-log narrative explaining the canonical
     change and any backward-compatibility notes for STU2 consumers.
3. Co-ordinate this change with the sibling tickets listed in
   *Related Jira Tickets* so all C4BB CodeSystems and ValueSets are
   migrated in a single coherent C4BB release rather than piecemeal.

#### Justification

The request matches HL7's standing terminology governance policy:
shared, reusable code systems belong in THO under the
`terminology.hl7.org` canonical, not under an IG-specific
`hl7.org/fhir/us/...` URL. C4BBInstitutionalClaimSubType is a
candidate for cross-IG reuse (NUBC-style institutional claim
sub-typing is relevant beyond CARIN BB), and the original WG vote on
2022-07-26 explicitly acknowledged this ("May move this CodeSystem to
THO for a future version based on harmonization work with other
IGs"). Reverting the *Considered for Future Use* disposition on
2022-11-28 signalled that the WG is now ready to revisit; accepting
moves the CodeSystem to its long-term home and unblocks the sibling
THO-anchoring tickets that the WG has been holding to a single
release wave.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the principle but bundle the work into a coordinated C4BB
"terminology re-anchoring" milestone rather than executing it as an
isolated change. Specifically:

1. Open a single UTG umbrella request that submits all eligible C4BB
   CodeSystems (the FHIR-3539x family — 35390, 35391, 35392, 35393,
   35395, 35396, 35398, 35399 — and the FHIR-36486/-36487
   CodeSystems) to THO together, ensuring consistent naming,
   property handling, and deprecation messaging.
2. In `HL7/carin-bb`, gate all in-IG CodeSystem retirement on UTG
   publication: until the THO version is published, keep the existing
   `hl7.org/fhir/us/carin-bb/CodeSystem/C4BBInstitutionalClaimSubType`
   canonical in place but mark the resource (and the corresponding
   ValueSet bindings) as `deprecated` / "moving to THO in next
   release", with a pointer comment to the UTG ticket.
3. Cut a single C4BB minor release that swaps every affected
   canonical at once, with one consolidated migration section in the
   IG narrative, instead of 8+ point releases.

#### Justification

FHIR-35395 is one of a large sibling set (see *Related Jira
Tickets*); migrating C4BBInstitutionalClaimSubType in isolation would
churn the IG, fragment implementer migration work, and risk
divergent UTG submissions for the related CodeSystems. A bundled
approach is also what the ticket's reporter implicitly asked for ("A
UTG ticket should be created…") at the policy level rather than the
mechanical one. It honours the original 2022-07-26 vote's intent
("based on harmonization work with other IGs") while making the
migration dependable and reviewable as a single change. The trade-off
vs Disposition A is a longer wall-clock time to closure for this
specific ticket — but lower implementer burden and lower risk of
inconsistent canonicals across C4BB.

---

### Disposition C: Decline (re-affirm "Considered for Future Use")

#### Proposal

Re-instate the original 2022-07-26 disposition: *Considered for
Future Use*, with the same rationale ("May move this CodeSystem to
THO for a future version based on harmonization work with other
IGs"). Leave `C4BBInstitutionalClaimSubType` published at its
current `hl7.org/fhir/us/carin-bb/CodeSystem/...` canonical for the
remainder of the STU2 / pre-STU3 life cycle. Do not file a UTG
ticket at this time.

#### Justification

The WG previously voted 13-0-0 to defer this change, and STU2 has
been published with the existing canonical in active implementer use.
A canonical URL change is a breaking change for consumers (FHIR
servers, ETL pipelines, validators, terminology services) and should
not be made without a clear THO landing target and a coordinated
release. Until the harmonization work referenced in the original
disposition has produced an agreed THO canonical (and until the
sibling tickets are ready to move together), the lowest-risk and
most defensible answer is to retain the deferral. Declining here does
not preclude accepting a future ticket once the THO submission and
the sibling slate are ready.

---

### Recommendation

**Recommended disposition: B — Alternative Approach (coordinated
re-anchoring milestone).**

The request is correct in principle (CodeSystem reuse belongs in
THO) and the WG's 2022-11-28 re-open signals an appetite to act
post-STU2, which weakens the pure "decline / defer again" answer in
Disposition C. However, executing FHIR-35395 in isolation
(Disposition A) would generate an asymmetric C4BB release where one
CodeSystem moves to THO while a dozen siblings (FHIR-35390/-91/-92/
-93/-96/-98/-99, plus FHIR-36486/-36487) remain at
`hl7.org/fhir/us/carin-bb`, doubling implementer migration work and
increasing the chance of inconsistent UTG submissions. Disposition B
preserves the accept-in-principle outcome the reverted vote points
to, ties it to a single UTG umbrella request and a single C4BB
release, and explicitly addresses the breaking-canonical concern that
motivated the original deferral. It is also the disposition most
consistent with the ticket-text's own framing ("A UTG ticket should
be created to register the Code System in THO") which is naturally
satisfied by a coordinated submission.
