## Summary

The submitter asks Financial Management to redefine how the Diagnosis Related
Group (DRG) is represented in the billing resources (`Claim`,
`ClaimResponse`, `ExplanationOfBenefit`). In R4 the DRG is carried by
`ExplanationOfBenefit.diagnosis.packageCode` (and is also reachable via
`Claim.diagnosis.packageCode`), bound to the example value set
`valueset-ex-diagnosisrelatedgroup`. The reporter argues that placement is
wrong because a DRG is the result of a grouper algorithm fed by procedure
codes, diagnosis codes, age, gender, length of stay, etc. ‚Äî it is not a
property of any single diagnosis. The proposed wording is to "Define the
Diagnosis Related Group (DRG) as a backbone element" so that it can carry
the DRG code together with its supporting metadata (e.g., grouper system,
version) rather than being misclassified under `diagnosis`.

The ticket was originally filed against CARIN BB but has been re-targeted
at FHIR Core. Its disposition history is non-trivial: it was initially
resolved "Considered for Future Use" (vote 18-0-5, Mark Roberts / Pat
Taylor) on 2020-07-21, then re-opened post-STU2 publication on 2022-11-28
(Lynn Laakso). A 2023 comment from Corey Spears reinforces the placement
concern, and a 2025-10-30 comment confirms it is now being treated as a
FHIR Core change request. R5 has already partially addressed the issue by
adding a flat `Claim.diagnosisRelatedGroup` (CodeableConcept 0..1) and
`Account.diagnosis.packageCode`, but the change has not been carried over
to `ClaimResponse` or `ExplanationOfBenefit`, and it was added as a flat
element rather than the backbone the submitter requested.

No incoming or outgoing cross-references were returned for FHIR-26712,
and no GitHub PRs/issues are linked to it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new BackboneElement, `diagnosisRelatedGroup` (0..1), to
`ExplanationOfBenefit`, `ClaimResponse`, and `Claim` (replacing or
co-existing with the R5 flat `Claim.diagnosisRelatedGroup` CodeableConcept
during a deprecation window). The backbone would carry at minimum:

- `diagnosisRelatedGroup.code : CodeableConcept` (0..1) ‚Äî bound to
  `valueset-ex-diagnosisrelatedgroup` (example), preserving the existing
  binding.
- `diagnosisRelatedGroup.grouper : CodeableConcept` (0..1) ‚Äî identifies
  the grouper system / variant (e.g., MS-DRG, AP-DRG, APR-DRG, IR-DRG).
- `diagnosisRelatedGroup.version : string` (0..1) ‚Äî grouper version /
  effective fiscal year.
- `diagnosisRelatedGroup.assigner : Reference(Organization)` (0..1) ‚Äî
  who assigned the DRG (provider vs. adjudicator), useful when both
  `Claim` and `ClaimResponse` carry DRGs that may differ.

Concurrently, deprecate `ExplanationOfBenefit.diagnosis.packageCode`
(and the corresponding `Claim.diagnosis.packageCode`) in favor of the
new top-level backbone, keeping the legacy element through one normative
cycle to preserve R4/R5 wire compatibility.

#### Justification

This matches the literal request and resolves Corey Spears's correctness
concern: a DRG is not a property of any single diagnosis row, so removing
it from `diagnosis.*` is the right structural fix. A backbone (rather than
a flat code) gives implementers a place to record the grouper family and
version, which is essential because the same numeric DRG code means
different things across MS-DRG vs. APR-DRG and across grouper years ‚Äî
information that today is either lost or stuffed into `supportingInfo`.
Adding the same backbone to `ClaimResponse` directly supports Paul
Knapp's Zulip observation that the adjudicator is authoritative for the
DRG and that the provider-supplied and adjudicated DRGs may differ.

---

### Disposition B: Alternative Approach

#### Proposal

Do not introduce a new backbone. Instead, complete the R5 pattern by
adding a flat `diagnosisRelatedGroup : CodeableConcept` (0..1) to
`ClaimResponse` and `ExplanationOfBenefit`, mirroring the existing
`Claim.diagnosisRelatedGroup` element added in R5. Mark
`ExplanationOfBenefit.diagnosis.packageCode` (and `Claim.diagnosis.packageCode`)
as deprecated in favor of the new top-level element, but leave them in
place for one cycle. Provide a standard core extension
`diagnosis-related-group-grouper` (CodeableConcept) and
`diagnosis-related-group-version` (string) for implementations that need
to qualify the DRG, rather than baking those into the resource.

#### Justification

The structural problem (DRG misplaced under `diagnosis`) is real, but the
ticket's "make it a backbone" framing is a means, not an end. R5 already
chose the simpler shape (a flat CodeableConcept on `Claim`); promoting
that same shape to `ClaimResponse` and `EoB` solves the placement problem
with less churn and stays consistent with what Cooper Thompson and Paul
Knapp described as the de-facto FM model. Implementations that need the
grouper system / version can use core extensions, which is how FHIR
typically handles attributes that matter to a minority of jurisdictions.
This approach also avoids inventing sub-elements (`grouper`, `version`,
`assigner`) without an articulated set of submitter requirements, which
historically leads to under-used backbone fields.

---

### Disposition C: Decline

#### Proposal

Decline as already addressed. Note in the resolution that R5 added
`Claim.diagnosisRelatedGroup` (CodeableConcept, 0..1) and
`Account.diagnosis.packageCode`, that the existing
`ExplanationOfBenefit.diagnosis.packageCode` continues to support DRG
exchange on the EoB side, and that grouper / version metadata can be
carried via `supportingInfo` or implementation-specific extensions.
Recommend that implementers needing additional structure profile the
existing elements rather than expand the core resource.

#### Justification

DRGs are a primarily US-jurisdictional concept and the R4 / R5 base
resources already let an implementer transmit the code. The submitter's
correctness concern (DRG should not live under `diagnosis`) is partially
mitigated in R5 by the addition of `Claim.diagnosisRelatedGroup` outside
the `diagnosis` backbone. Adding a backbone to three large, heavily-used
resources for a single coded value is a large surface-area change for
modest benefit, especially when extensions can carry grouper system /
version and `supportingInfo` can carry related context.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible starting point for the workgroup. It
fully addresses the submitter's structural concern ‚Äî DRG no longer lives
under `diagnosis` ‚Äî without manufacturing a backbone whose sub-elements
the ticket itself does not specify. It is consistent with what R5 has
already done on `Claim` and with the Zulip discussions, where Cooper
Thompson, Paul Knapp, and Lloyd McKenzie all described the DRG as a
single coded assertion that the provider supplies and the adjudicator
confirms. Carrying that same flat element forward to `ClaimResponse` and
`ExplanationOfBenefit` closes the obvious R5 gap (the adjudicated DRG
has nowhere to live on `ClaimResponse`) and makes round-tripping
provider-vs-adjudicator DRGs straightforward. Disposition A's grouper /
version / assigner fields can still be added later as a non-breaking
backbone refactor if implementer feedback shows that the standard
extensions are insufficient. Disposition C is hard to defend because it
leaves `ExplanationOfBenefit.diagnosis.packageCode` in the very place
that the submitter and Corey Spears have flagged as wrong.
