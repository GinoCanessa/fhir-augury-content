## Summary

The ticket asserts that the `Invoice` resource in FHIR Core lacks
terminology bindings on coded elements where bindings would normally be
expected, and that — depending on actual implementation uptake — the
resource's FMM (FHIR Maturity Model) level may need to be revisited. The
reporter (Paul Knapp, FM co-chair) has explicitly asked Simone Heckmann
whether Germany is currently using `Invoice`, and if so, whether the
German implementer community can supply concrete value-set or example
recommendations for the missing bindings. No related Jira tickets,
Zulip discussions, or GitHub items were surfaced via cross-references
or search, so this ticket is currently standing on its own as an
exploratory request that depends on implementer feedback before a
concrete change can be drafted.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the ticket and treat it as a two-part action item against the
`Invoice` resource in FHIR Core:

1. **Terminology audit.** The Financial Management WG inventories every
   coded element on `Invoice` (notably `status`, `cancelledReason`,
   `type`, `lineItem.priceComponent.type`,
   `lineItem.priceComponent.code`, `totalPriceComponent.type`,
   `totalPriceComponent.code`, and `paymentTerms`-adjacent elements),
   classifies each as *required / extensible / preferred / example*,
   and either reuses an existing HL7 / external value set (e.g.,
   IHE / X12 / UN-CEFACT / domain price-component code systems) or
   defines a new HL7 example/preferred value set under
   `terminology.hl7.org`. Where no canonical value set is yet
   available, ship example bindings populated with the German
   implementer-supplied codes Paul Knapp asked Simone Heckmann to
   provide.
2. **FMM reassessment.** Once real-world usage is confirmed (Germany
   was named explicitly as a likely user), reapply the FMM criteria
   (production use in ≥ N implementations, etc.). If Germany and at
   least one other realm are actively using `Invoice` in production,
   raise FMM. If usage is essentially nil, lower or hold FMM and label
   the resource accordingly.

#### Justification

This is exactly what the reporter asked for, and both halves are
defensible on their own. The terminology gap is real: `Invoice` is
notably sparser on bindings than peer financial resources (e.g.,
`Claim`, `ChargeItem`, `Account`), which makes interoperability
harder and creates implementation drift. The FMM portion is a
governance hygiene step the WG owes the community regardless — FMM is
supposed to track real-world maturity, and if Germany (and possibly
others) have been using `Invoice` in production for several releases,
the current FMM level is misleading either upward or downward.

---

### Disposition B: Alternative Approach

#### Proposal

Split the ticket into two narrower work items and resolve this one as
a tracking parent:

- **B1 — Terminology only.** Open a new ticket scoped solely to adding
  example/preferred bindings for `Invoice.lineItem.priceComponent.type`
  / `.code` and `Invoice.totalPriceComponent.type` / `.code`, seeded
  from the German implementer codes once Simone Heckmann responds.
  Defer any *required* or *extensible* bindings until at least two
  realms agree on a value set, to avoid locking in a single-realm
  vocabulary at the core-spec level.
- **B2 — FMM reassessment.** Open a separate, evidence-driven ticket
  that asks for concrete production-use citations (implementations,
  IGs profiling `Invoice`, conformance statements) and only then moves
  FMM. Until evidence is gathered, hold the current level.

The current ticket (FHIR-53100) is then closed as "split / superseded"
with links to B1 and B2.

#### Justification

The ticket as written conflates two governance decisions (binding
strength and maturity) that have very different evidentiary thresholds
and very different rollback costs. Adding example bindings is cheap
and safe; bumping FMM is hard to undo and signals to implementers that
the resource is stable. Splitting also lets the WG land the
terminology improvements quickly (low controversy) without holding
them hostage to the slower FMM evidence-gathering process, and avoids
the risk of accidentally promoting `Invoice` to a higher FMM solely on
the strength of one realm's usage.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" / decline as currently filed, with
the rationale that it is a request for action without an actionable
proposal: no specific elements are named, no candidate value sets are
proposed, no FMM target is suggested, and no evidence of production
use is supplied. Invite the reporter to refile once Simone Heckmann
(or any other implementer) has responded with concrete codes and
usage data, at which point the WG can act on a substantive proposal.

#### Justification

Tickets that say "the resource needs terminology" and "FMM should be
adjusted if it is in use" are essentially placeholders for work that
hasn't happened yet. The WG cannot ballot a binding it has not
authored, and cannot move FMM without the implementation evidence the
ticket itself acknowledges is missing. Closing with a clear "refile
when you have the codes and the evidence" reply keeps the backlog
honest and avoids leaving a vague item open for years.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures everything the reporter actually wants while
protecting the spec from the two failure modes the ticket flirts with:
(1) shipping required/extensible bindings sourced from a single
realm's vocabulary, and (2) moving FMM on anecdote rather than
evidence. It also gives the WG a clean way to make visible progress
quickly — example bindings can land as soon as the German codes
arrive — without entangling that progress with the slower FMM
governance question. Disposition A is acceptable but risks doing both
at once and is harder to revisit if the German codes turn out to be
realm-specific. Disposition C is technically defensible (the ticket
*is* underspecified) but throws away momentum the reporter has already
built by pinging a known implementer; splitting and tracking is more
collaborative than declining outright.
