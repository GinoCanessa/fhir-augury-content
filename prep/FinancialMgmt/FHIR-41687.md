## Summary

The reporter (Chris Cioffi) is requesting that the **CARIN Consumer-Directed Payer Data Exchange (CARIN BB)** Coverage profile add a new extension on the existing `Coverage.class:plan` slice named `productId` of type `string`. The stated purpose is to let consumers/clients distinguish a payer's **product** from its **plan**, because the reporter's organization treats the *provider network* as a cross-section of `product` and `plan id`. With only the existing `plan` value, downstream systems cannot disambiguate which provider network a member can actually access.

A workgroup comment from corey_spears (2023-11-16) confirms parallel work on slicing `Coverage.class` to include a `Product ID` slice and flags Amol Vyas as the author of a forthcoming proposal; a follow-up comment (2024-07-11) reminds that any change must also update the profile narrative and diagrams. No cross-referenced Jira tickets, Zulip threads, Confluence pages, or GitHub items were returned by the augury index for this issue (the only outgoing cross-reference is a textual mention of the `Coverage.class` FHIR artifact). Keyword-similar tickets that provide useful background — but are not formally linked — include [FHIR-34922](https://jira.hl7.org/browse/FHIR-34922) (Da Vinci PCT, Persuasive — added a `plan` slice on `Coverage.class` patterned after C4BB) and [FHIR-39855](https://jira.hl7.org/browse/FHIR-39855) (Resolved – Not Persuasive — affirmed that group/plan slicing is the correct pattern, citing CARIN BB and Da Vinci HRex).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **CARIN BB Coverage profile**, define a new FHIR extension and add it to the existing `Coverage.class:plan` slice:

- **Extension URL:** `http://hl7.org/fhir/us/carin-bb/StructureDefinition/CoverageClassPlanProductId` (final URL subject to IG-team naming conventions).
- **Context:** `Coverage.class` (intended for use on the `:plan` slice).
- **Cardinality on the slice:** `0..1 MS`.
- **Value:** `value[x]` constrained to `string` (`Extension.value[x] only string`).
- **Short / definition:** "A payer-assigned product identifier. Differentiates the *product* from the *plan id* on the same Coverage.class:plan slice; together they identify the provider network the member can access."
- Apply the extension as a named slice on `Coverage.class:plan.extension` with `mustSupport = true` so consumers must read it when present.
- Update the C4BB Coverage profile **narrative** and **class-diagram artifacts** so the new extension and its purpose appear in both the human-readable text and the rendered diagrams (per corey_spears' 2024-07-11 comment).
- Add at least one example Coverage instance demonstrating `productId` populated alongside the existing plan value.

#### Justification

The reporter's use case — that a member's accessible provider network is a cross-section of `product` and `plan id` — is a real US-payer modeling gap that cannot be expressed today on `Coverage.class:plan` (which carries a single coded value). Layering a typed extension on the existing `plan` slice is the lowest-disruption path: it preserves the slice shape established by FHIR-34922 and re-affirmed by the Financial Management resolution of FHIR-39855, requires no breaking change to existing implementers, and matches the CARIN BB convention of using extensions on Coverage to convey US-specific payer concepts. The corey_spears 2023-11-16 comment indicates the workgroup was already exploring this direction and has a proposal owner (Amol Vyas) lined up.

---

### Disposition B: Alternative Approach — Add `productId` as a New Slice on `Coverage.class`

#### Proposal

Instead of an extension on `Coverage.class:plan`, define a **new sibling slice** on `Coverage.class` named `product`:

- Slicing discriminator: `pattern` on `Coverage.class.type` (the existing discriminator pattern used for `group`, `plan`, etc.).
- New slice `Coverage.class:product`, `0..1 MS`, with:
  - `Coverage.class:product.type.coding 1..1 MS`, fixed to a `product` concept on `http://terminology.hl7.org/CodeSystem/coverage-class` (request the code via THO if `product` is not yet defined).
  - `Coverage.class:product.value 1..1 MS` (typed `string`, like the existing `plan` slice).
  - `Coverage.class:product.name 0..1 MS`.
- Keep `Coverage.class:plan` unchanged.
- Update narrative, diagrams, examples, and any client conformance text. Coordinate with HL7 Terminology (UTG) to register the `product` code if missing.

#### Justification

Treating *product* as a peer of *plan* (rather than as a child extension) mirrors the modeling used by Da Vinci PCT (FHIR-34922) and the Financial Management precedent set by FHIR-39855 that group/plan/etc. are *distinct first-class concepts* on `Coverage.class`. A first-class slice is more discoverable in tooling, easier to validate, and avoids the long-term anti-pattern of "core concept hidden inside an extension on another slice." The cost is higher: it requires a UTG terminology change (or use of an interim local code) and slightly broader profile updates, and it is a more visible change to existing implementers. It is, however, more architecturally honest about what `productId` actually is.

---

### Disposition C: Decline

#### Proposal

Reject the request as written, on the grounds that **CARIN BB Coverage already has a sanctioned mechanism — the `Coverage.class:plan` slice with its `value` element and free-text `name`** — to convey payer-assigned identifiers, and that consumers needing to distinguish *product* from *plan* should use the existing `Coverage.class.value` together with `Coverage.network`, `Coverage.payor`, and `Coverage.contract` to assemble the provider-network view, rather than introducing a CARIN-BB-specific element. Suggest the reporter raise the underlying use case (provider-network identification) as a Da Vinci PDex / Plan-Net work item if cross-payer interoperability is needed.

#### Justification

The artifact change is small in isolation but creates a CARIN-BB-only concept (`productId`) that other Coverage consumers (US Core, Da Vinci HRex/PCT, international Coverage users) will not understand. The 2023 FHIR-39855 thread shows the workgroup is actively defending the *current* slice shape; adding payer-specific extensions on top of `:plan` may invite more one-off requests from other payers and slowly fragment the profile. Declining preserves consistency with the broader Coverage modeling discussion and pushes the use case into the Plan-Net / network-discovery space where it more naturally belongs.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested), with one caveat.

The reporter's use case is concrete, the modeling gap is real, and the workgroup has already signaled willingness (corey_spears 2023-11-16). An extension on `Coverage.class:plan` is the **least disruptive** way to land the change inside the next CARIN BB ballot cycle: it is fully backward-compatible, requires no THO terminology changes, and mirrors how CARIN BB already extends Coverage for US-specific payer concepts. Disposition B is architecturally cleaner but pulls in a UTG dependency and a wider conformance impact that will likely slow the change past the reporter's needs; if the workgroup ultimately wants the first-class slice, that can be promoted *from* the extension in a later cycle without harming early adopters. Disposition C ignores a stated implementer pain point that today has no clean expression in the profile.

**Caveat for the workgroup:** before adopting Disposition A, please confirm that "productId" is indeed a *string* and not a coded identifier (`Identifier`/`code`); if it is in fact an identifier with a defined system per payer, use `Identifier` for `value[x]` instead of `string` so the assigning authority is preserved. This should be verified directly with the reporter and Amol Vyas (per the 2023-11-16 comment) prior to ballot submission. Also confirm that the C4BB Coverage narrative and class diagrams are updated as part of the same change (per the 2024-07-11 comment).
