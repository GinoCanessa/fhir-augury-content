## Summary

The reporter (Charles Willcutt, Epic) asserts that the status value set used
on the FHIR `Claim` resource (`active`, `cancelled`, `draft`,
`entered-in-error` — the standard `fm-status` value set) is incomplete
relative to how claims are actually managed in production billing
systems. He notes that real-world claims are categorised along a
lifecycle of **Initial, Subsequent, Final, Late, Replacement,** and
**Cancelled**, and argues that the current FHIR list cannot represent
common workflows such as sequential interim claims and addendums. The
ticket implicitly asks Financial Management to extend or replace the
status mechanism on `Claim` so these billing concepts can be expressed
without resorting to local extensions. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced for this issue, so the
request currently stands on its own without prior community discussion
captured in Augury.

## Details

**Description:**

> When claims are processed today, they have more status values than what are listed in the FHIR specification.
>
> Claims can be categorized as Initial, Subsequent, Final, Late, Replacement, or Cancelled.
>
> In the FHIR specification there is only active, cancelled, draft, entered-in-error
>
> This list of statuses is incomplete and not compatible with billing practices today such as sequential interim claims and addendums.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace or supplement `Claim.status` so it can carry the full billing
lifecycle the reporter cites. Concretely, in **HL7/fhir** under the
`Claim` resource definition (`source/claim/`):

1. Retain the existing required `Claim.status` element bound to
   `fm-status` (`active | cancelled | draft | entered-in-error`) for
   resource-level workflow state, since other FM resources share this
   binding.
2. Add a new element **`Claim.lifecycleStatus`** (0..1, `code`,
   required binding) bound to a new value set
   `claim-lifecycle-status` containing: `initial`, `subsequent`,
   `final`, `late`, `replacement`, `cancelled`. Define the code system
   alongside the value set under `source/valueset/` and
   `source/codesystem/`, with definitions aligned to the X12 837 claim
   frequency code semantics.
3. Add a corresponding `ClaimResponse.requestLifecycleStatus` (0..1)
   for response correlation, and a narrative section in the `Claim`
   page explaining how `lifecycleStatus` interacts with `status` and
   how to model addendums and sequential interim claims via
   `Claim.related` + `lifecycleStatus = subsequent | replacement`.

#### Justification

This directly addresses the reporter's gap: real-world billing
workflows (interim claims, addendums, replacements, late submissions)
cannot be conveyed today without out-of-band extensions, which harms
interoperability between payers and providers using FHIR. Separating
**resource workflow state** (`status`) from **billing lifecycle state**
(`lifecycleStatus`) preserves the existing `fm-status` semantics shared
across Financial Management resources and avoids a breaking change to
the required binding. Aligning the new code system with X12 837 claim
frequency codes leverages an existing, well-understood mapping that US
implementers already use, reducing the burden of code translation.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** modify `Claim.status`. Instead, formalise the billing
lifecycle through two existing mechanisms:

1. Define a **standard extension** `claim-lifecycle-status` (URL under
   `http://hl7.org/fhir/StructureDefinition/claim-lifecycle-status`)
   on `Claim`, bound to a new `claim-lifecycle-status` value set with
   the codes `initial | subsequent | final | late | replacement |
   cancelled`. Author it as a normal extension under the FHIR Core
   extensions area in **HL7/fhir**.
2. Tighten the documentation on **`Claim.related`** + **`Claim.related.relationship`** to
   show the canonical pattern for replacements and sequential interim
   claims (the `related-claim-relationship` value set already contains
   `prior`, and is open to additions such as `replaces`, `addendum-to`).
3. Optionally, add the new lifecycle codes as a US-specific profile in
   a downstream IG (e.g., a payer/provider IG) so that jurisdictional
   billing semantics live where they are actually constrained.

#### Justification

`Claim.status` in FHIR has always been a *workflow* state shared with
sibling resources (`ClaimResponse`, `Coverage`, etc.); overloading it
with payer-domain billing semantics risks confusing implementers and
forces a wider Financial Management review. An extension plus stronger
use of `Claim.related` is non-breaking, can ship in the next release,
and is consistent with how other domain-specific lifecycle concepts
have been added to the spec. It also keeps jurisdictional billing
specifics (which differ across US payers, NHS, AU MBS, etc.) out of
the base resource, which is preferable given that the codes
`initial / subsequent / final / late / replacement` map most cleanly
to US X12 837 conventions and may not generalise.

---

### Disposition C: Decline

#### Proposal

Decline the request as written and close as **Not Persuasive — already
addressed by existing mechanisms**. The conceptual states the reporter
lists are billing-workflow concepts, not resource-workflow concepts,
and they are already representable today via:

- `Claim.status` for resource workflow (`active`, `cancelled`,
  `draft`, `entered-in-error`).
- `Claim.use` (`claim`, `preauthorization`, `predetermination`) for
  the kind of interaction.
- `Claim.related` + `Claim.related.relationship` for replacements,
  addendums, and sequential interim claims (point to the prior
  `Claim` and qualify the relationship).
- `Claim.subType` and existing payer-specific extensions for
  jurisdictional billing nuances.

#### Justification

Adding `initial / subsequent / final / late / replacement` to the
required `fm-status` binding would be a breaking change to a normative
or near-normative resource and would conflate workflow state with
billing lifecycle. The cited workflows (sequential interim claims,
addendums) are already modelled in production FHIR-based payer/provider
exchanges using `Claim.related` and US-specific payer IGs, so the gap
is documentation and IG guidance rather than a missing core element.
Without concrete implementer evidence that `Claim.related` is
insufficient — and none is supplied in the ticket or in Augury's Zulip
index — a base-spec change is not warranted.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between the reporter's legitimate
concern and FHIR's existing patterns. The lifecycle states he lists
genuinely *are* used by real billing systems and are not cleanly
expressible today, so a flat decline (C) ignores a real implementer
pain point — especially given the reporter is from a major EHR vendor
and the comment trail already invites them to an FM call. At the same
time, jamming those codes into the required `fm-status` binding (A)
would be a breaking change to a value set shared across Financial
Management resources, would conflate two different notions of "status",
and would push US-flavoured X12 semantics into the base spec where
they may not generalise. A standard extension plus tightened guidance
on `Claim.related.relationship` lets FM ship a non-breaking solution
in the next release, validates the pattern with implementers, and
leaves the door open to promoting the extension to a core element in a
later cycle if uptake justifies it. The FM call requested by pknapp is
the right venue to confirm that the proposed codes match what Epic and
peer implementers actually need.
