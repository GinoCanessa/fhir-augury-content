## Summary

The reporter identifies an internal inconsistency in the US Da Vinci ATR
(Authorization, Treatment, and Referrals) IG, section 16.3.1.2
*Parameters* of the `$davinci-data-export` operation. The `_type`
parameter is declared with cardinality `0..1`, yet its description
explicitly states it carries "a string of comma-delimited FHIR resource
types" — i.e., it is intended to convey a list. The reporter argues the
cardinality should be `0..*`, and notes the same concern for
`_typeFilter`.

The reporter cross-references the FHIR Bulk Data Access IG STU2
([export.html](https://hl7.org/fhir/uv/bulkdata/STU2/export.html)),
where both `_type` and `_typeFilter` are defined with `0..*`
cardinality. Because the ATR IG's operation is layered on top of the
base Bulk Data export operation and its own documentation says the
parameter is "exactly the same parameter defined in the Bulk Data
Export operation," the divergent cardinality is at minimum confusing
and arguably an error. The Da Vinci PDex/ATR Implementer Support call
on 2025-11-21 has begun discussing the disposition (see linked
Confluence meeting page); commenter Mark S. proposed that simply
syncing cardinality with the base operation would be acceptable, while
yanheras flagged that the `0..1` constraint may have been chosen
deliberately because some base-spec parameters are *server-optional*
and the ATR profile may have wanted to lock down implementer behavior.

No related Jira tickets, GitHub PRs, issues, or Zulip threads were
surfaced by the cross-reference index for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Issue a technical correction to the US Da Vinci ATR IG
`OperationDefinition/davinci-data-export`. In §16.3.1.2 *Parameters*:

- Change `_type.max` from `1` to `*` (resulting cardinality `0..*`).
- Change `_typeFilter.max` from `1` to `*` (resulting cardinality
  `0..*`).

Leave the parameter `type`, `documentation`, `use` (`in`), and binding
unchanged. Update the rendered parameter table in the narrative section
so it matches the OperationDefinition. No other resources, profiles, or
operations are affected. Because this is purely a relaxation of an
upper bound, it is non-breaking for existing conformant clients (single
value remains valid) and aligns the operation signature 1:1 with the
underlying Bulk Data Access STU2 `$export`, which the IG's own prose
already claims it does.

#### Justification

The IG's own description text says each parameter "is exactly the same
parameter defined in the Bulk Data Export operation" and explicitly
describes a comma-delimited list — both statements are inconsistent
with `max=1`. FHIR convention for OperationDefinition parameters that
accept a comma-delimited list of values is to model the list using
`max="*"` (the Bulk Data IG itself does this). Aligning cardinality
removes a documented contradiction, eliminates an implementer trap,
and is non-breaking. This is the sort of in-place text/cardinality fix
that the *Technical Correction* type is intended for.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the parameters at `0..1` (a single string token) but **rewrite the
description** to make explicit that the single string is itself a
comma-delimited list, *and* tighten the conformance language so
implementers know the ATR profile is intentionally narrower than base
Bulk Data. Concretely:

- Retain `_type.max = 1` and `_typeFilter.max = 1`.
- In the parameter narrative, state: "The parameter is supplied as a
  single string whose value is a comma-delimited list of FHIR resource
  type names (e.g., `Patient,Coverage,ExplanationOfBenefit`). Servers
  conformant to this IG **SHALL** support this parameter (unlike the
  base Bulk Data operation where it is server-optional)."
- Add an implementer note that this is a deliberate ATR-profile
  constraint of the base operation, with a pointer back to
  Bulk Data §`_type` for the underlying semantics.

#### Justification

This addresses yanheras's concern that the original `0..1` choice may
have been deliberate: it gives the ATR profile a way to *strengthen*
implementer obligations relative to the base Bulk Data spec (where
`_type` and `_typeFilter` are server-optional) without changing the
OperationDefinition signature. It also avoids the small risk that
some implementations have been validating against `max=1` and would
need code changes. The downside is that it perpetuates the cosmetic
mismatch with the base operation, and "single string that is itself
a comma-delimited list" is a known anti-pattern in FHIR
OperationDefinitions — most invocations are GET URL parameters where
repeats and commas are interchangeable, so the distinction is
implementer-confusing.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as *Persuasive – Not Persuasive* (no spec change).
Document in the resolution that the cardinality of `_type` and
`_typeFilter` was set to `0..1` deliberately, that base Bulk Data
semantics for comma-delimited values still apply, and that the
description text already conveys list semantics; therefore no
correction is required.

#### Justification

This is the weakest of the three options. The reporter has identified a
real, in-spec contradiction (description vs. cardinality) and has
pointed at the authoritative base IG that uses the opposite
cardinality. Declining without a concrete mitigation would leave the
contradiction in place and ignore the comments already on the ticket
that lean toward agreement with the reporter. It would only be
defensible if the work group can document a hard interoperability
reason — e.g., a deployed conformance test suite or registered
implementations that depend on `max=1` — which has not surfaced.

---

### Recommendation

**Recommended disposition: A — Accept As Requested.**

The change is small, narrowly scoped to one OperationDefinition and its
narrative, non-breaking (relaxation of an upper bound), and resolves a
contradiction the IG itself acknowledges in its own parameter
descriptions ("exactly the same parameter defined in the Bulk Data
Export operation"). The base Bulk Data Access IG STU2 already uses
`0..*` for both parameters, and the ATR IG gains nothing operationally
from being narrower — at the wire level a comma-delimited string and
repeating parameters are equivalent for these search-style tokens, so
keeping `max=1` only signals a constraint that doesn't actually
constrain anything.

Disposition B's intent — making ATR's implementer obligations stronger
than base Bulk Data's — is worth preserving and can be folded into
Disposition A by adding a *Servers SHALL support `_type` and
`_typeFilter`* conformance note alongside the cardinality fix. That
captures yanheras's "server-optional in the base, mandatory here"
observation without leaving the cardinality contradiction in place.
The work group should confirm at the next Da Vinci PDex/ATR call,
where this is already on the agenda (per the 2025-11-21 meeting page).
