## Summary

The reporter requests that FHIR R6 enable coded representation of
`Coverage.class.value`. Today the element is typed as `Identifier`, which
is intended for system-unique identifiers, but real-world payer
implementations typically convey insurance class designations (e.g.,
group, plan, sub-plan, network tier) as enumerated codes drawn from
national or payer-defined ValueSets. Modeling these as `Identifier`
prevents terminology binding and validation against a ValueSet,
forcing implementers either to abuse the `Identifier` data type
(stuffing a code into `system`/`value`) or to layer custom extensions
that carry a `CodeableConcept`. The ticket proposes two candidate
remedies for FHIR R6: (1) allow `CodeableConcept` as an alternative or
replacement type for `Coverage.class.value`, or (2) define a standard
extension that carries a coded representation (with optional ValueSet
binding) alongside the existing `Identifier`. Cross-reference data
links the ticket to the FHIR artifacts `Coverage.class` and
`Coverage.class.value`; no related Jira tickets, Zulip discussions, or
GitHub items were surfaced by the cross-reference index.

## Details

**Description:**

The `Coverage.class` element is designed to describe classifications of
insurance coverage such as group, plan, or class, where `type` is a
codified classification category and `value` is an `Identifier`.
However, in real-world implementations, the actual value representing
the class of coverage is a code from a ValueSet, not a system-unique
identifier.

The current definition of `Coverage.class.value` as an `Identifier`
limits validation capabilities. In cases where the classification is
best expressed using codes (e.g., categories defined by national
insurance schemes), implementers are forced to either:

- Encode the value as an `Identifier`
- Or introduce custom extensions with a `CodeableConcept`

We suggest that FHIR R6 consider one of the following:

1. Allow `CodeableConcept` as an alternative or replacement for
   `Coverage.class.value`, where appropriate, or
2. Introduce a standard extension for a coded representation of the
   class value, including support for binding to a ValueSet.

**Comments:**

- **ekivemark (2025-08-19):** Cautions that `Coverage.class` is widely

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In R6, change the type of `Coverage.class.value` from `Identifier` to
a choice `value[x]` that admits both `Identifier` (for backward
compatibility with current usage) and `CodeableConcept` (for coded
classifications), bound to an example-strength ValueSet. Update the
element's short/definition text, comments, and invariants on the
`Coverage` resource so that exactly one of `valueIdentifier` or
`valueCodeableConcept` is present per class entry, and document that
the choice is paired with `Coverage.class.type` (which selects the
classification axis: group, plan, sub-plan, network, etc.). Provide a
mapping note for migrators describing how to translate existing
`Identifier`-based payloads to the new shape.

#### Justification

This directly satisfies the reporter's first option and brings the
element in line with how implementers are already using it in
practice, eliminating the need for the workaround extensions that the
ticket calls out. Treating the class value as a `CodeableConcept`
unlocks ValueSet binding and validation, which is the core capability
the ticket is asking for, and the choice type preserves the
`Identifier` representation for jurisdictions and payers that use
opaque, system-scoped class identifiers (e.g., a payer-internal group
number). The ekivemark comment about CARIN Blue Button is a real
concern, but a `value[x]` choice is additive at the type level —
existing `Identifier`-typed instances remain valid — so the breaking
surface is limited to tooling that hard-codes the element name
`Coverage.class.value` rather than the polymorphic family.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `Coverage.class.value` typed as `Identifier` and instead define
a standard extension on `Coverage.class` (e.g.,
`http://hl7.org/fhir/StructureDefinition/coverage-class-valueCoded`)
of type `CodeableConcept`, with an example-strength binding to a new
ValueSet aggregating commonly used class code systems. The extension
would be paired with `Coverage.class.type` the same way `value` is
today, and the spec would explicitly state that when a coded
classification is used, implementers SHALL populate the extension and
MAY omit `Coverage.class.value`. Document the extension in the
Financial Management module pages and add a guidance note to
`Coverage.class.value`'s definition pointing to it.

#### Justification

This is exactly what the ekivemark comment recommends and is the
least disruptive path for the large installed base — most prominently
CARIN Blue Button — that has serialized `Coverage.class.value` as an
`Identifier` for years. A standard extension delivers the validation
capability the ticket asks for without changing the resource's
element types, so existing instances, profiles, search parameters,
and conformance tooling continue to work unchanged. The trade-off
versus Disposition A is that conformant servers must learn an
additional access path (the extension URL) to read the coded class,
and the dual representation creates a "which one wins?" question if
both are present, which the spec text would need to resolve
explicitly.

---

### Disposition C: Decline

#### Proposal

Decline the request and clarify in `Coverage.class.value`'s element
definition that `Identifier.system` plus `Identifier.value` is the
intended carrier for coded classifications: the code system URI goes
in `system` and the code in `value`, with `Coverage.class.type`
selecting the classification axis. Add narrative guidance and a
worked example showing how to express, e.g., a national insurance
class code using this pattern, and note that ValueSet-style
validation of class codes can be performed by profiles that constrain
`Coverage.class.value.system` to a fixed code system URI and
`Coverage.class.value.value` to a member of an enumerated set.

#### Justification

This is the position implicit in the pknapp comment: the committee is
not yet convinced the existing `Identifier` shape is inadequate, and
`Identifier` does carry both a namespace (`system`) and a token
(`value`), which can in fact represent a code drawn from a code
system. Declining avoids any risk of churn against widely deployed
implementations and avoids inflating the Coverage resource with a
second representation. The cost is that it does not give implementers
a first-class FHIR terminology binding on the class value, so true
ValueSet-binding validation in generic tooling (without
profile-specific knowledge) remains unavailable.

---

### Recommendation

**Recommended disposition:** B

The Financial Management module should adopt the standard-extension
path. The reporter's underlying need — validating the class value
against an enumerated set of codes — is real and is not served well
by `Identifier`, so Disposition C understates the problem and leaves
implementers in the same situation that produced the ticket.
Disposition A does solve the problem cleanly but, as ekivemark flags,
`Coverage.class` is one of the most widely implemented parts of
Coverage (CARIN Blue Button and downstream payer APIs), and changing
the element to a `value[x]` choice forces every consumer, profile,
and validator to update their bindings even if the on-the-wire
`Identifier` representation continues to be accepted. A standard
coded-value extension delivers the validation capability the ticket
asks for, gives the WG a place to attach a ValueSet binding, and
keeps the existing `Identifier` payloads byte-for-byte compatible —
so it should be the starting point for committee discussion, with
Disposition A held in reserve if the WG decides R6 is the right
moment to take the breaking change.
