## Summary

The ticket asks the CARIN IG for Blue Button® (C4BB) editors to move the
defining canonical URL of `ValueSet-CMSPlaceofServiceCodes` out of the
IG's `hl7.org/fhir/...` namespace and into `terminology.hl7.org` (THO),
on the principle that only ValueSets used in `required` bindings to the
`code` data type should remain anchored in the IG's own URL space. A
companion UTG ticket would need to be filed to register the ValueSet in
THO. The ValueSet enumerates the externally-maintained CMS Place of
Service Code Set, so the request is part of a larger STU2 cleanup
sweep that touched dozens of CARIN BB ValueSets / CodeSystems (see the
Block-Vote-6 thread). FHIR-35438 was originally dispositioned in the
2022-07-26 Financial Management vote as **Considered for Future Use**
with the rationale "Plan on making change when external CodeSystem
'all codes' ValueSet is published (UP-316)" — meaning the C4BB editors
do not want to publish a hand-maintained THO ValueSet when UP-316 will
let them simply bind directly to a generated "all codes" ValueSet for
the external CodeSystem. That resolution was reverted on 2022-11-28
("re-open post STU2 publication"), which is why the ticket is back in
Triaged status awaiting fresh disposition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor `ValueSet-CMSPlaceofServiceCodes` so its canonical URL is
`http://terminology.hl7.org/ValueSet/CMSPlaceofServiceCodes` (or the
equivalent THO-managed slug) instead of an `hl7.org/fhir/us/carin-bb/...`
URL. Concretely: file a UTG ticket to register the ValueSet (and, if
not yet present in THO, the underlying CMS Place of Service code
system) in `terminology.hl7.org`; once the THO artifact is published,
remove the locally-defined `ValueSet-CMSPlaceofServiceCodes` from the
CARIN BB IG and update every binding that currently points at the
local URL to point at the THO canonical instead. Add a deprecation /
retirement note in the IG release notes.

#### Justification

This matches the long-standing HL7 terminology governance position
that ValueSets enumerating an externally-maintained code system
belong in THO rather than in an IG's own namespace, and it is exactly
what the ticket asks for. The CARIN BB editors already accepted the
direction in spirit when they originally voted *Considered for Future
Use* on the same content — the 2022-11-28 reopen makes clear the
deferral is no longer satisfactory. CMS Place of Service Codes are a
stable, externally-governed code set with broad cross-IG reuse
potential (claims, encounters, billing), so a THO home is justified
even without UP-316.

---

### Disposition B: Alternative Approach — Keep deferred to UP-316 ("all codes" ValueSet)

#### Proposal

Re-affirm the original 2022-07-26 disposition: keep
`ValueSet-CMSPlaceofServiceCodes` in the CARIN BB IG for now, but
commit to removing it and rebinding to the auto-generated THO "all
codes" ValueSet for the CMS Place of Service CodeSystem the moment
UP-316 lands. Track this commitment in the IG's terminology backlog
(or via a new follow-up ticket linked to UP-316) so it does not get
lost again, and add an editorial note on the local ValueSet's
narrative pointing implementers at the future THO replacement.

#### Justification

The original FM vote (13-0-0) was based on a sound technical argument:
this ValueSet is just "all codes from an external CodeSystem", which
is precisely the case UP-316 is designed to handle generically. Hand-
crafting a THO ValueSet now creates a maintenance artifact that
UP-316 will obsolete — two migrations instead of one. The reopen
note from lynn_laakso doesn't dispute that reasoning; it simply
observes that STU2 has now published, so the deferral can be
revisited. Choosing B keeps the IG aligned with the strategic UTG
direction while avoiding throwaway work.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Mark FHIR-35438 *Not Persuasive* on the same rationale already used
across this Block-Vote-6 family for IG-specific ValueSets:
"ValueSet is not complex, fairly specific to this IG, and does not
warrant the maintenance of moving to THO." Leave
`ValueSet-CMSPlaceofServiceCodes` defined under the CARIN BB
canonical URL.

#### Justification

This is the disposition the FM workgroup has already applied to
several siblings in the same block (FHIR-35432, -35434, -35435,
-35436, -35439, -35446, -35447). If the workgroup judges that the CMS
Place of Service ValueSet, as scoped here, is effectively only used
by CARIN BB profiles and that no other IG has asked to consume it,
the maintenance overhead of moving it to THO outweighs the
governance benefit, and a *Not Persuasive* close is defensible and
internally consistent with prior votes in the same block.

---

### Recommendation

**Recommended disposition:** B

The CMS Place of Service code set is externally governed and the
local CARIN BB ValueSet does not add IG-specific filtering on top —
it is the textbook case UP-316 is intended to solve. The original
13-0-0 FM vote was correct on the technical merits, and the only
new fact since then is "STU2 has shipped", which removes the
schedule pressure but does not change the underlying choice between
"build a one-off THO ValueSet now" and "wait one cycle for the
generic mechanism". Disposition B preserves the editors' technical
direction, avoids creating a THO artifact that UP-316 will
immediately deprecate, and gives the workgroup a concrete
follow-through (link to UP-316, editorial note on the ValueSet) so
the commitment survives the next ballot. Disposition A is the
right answer if UP-316 has stalled or been withdrawn — the
workgroup should confirm UP-316's current status before voting; if
UP-316 is alive, choose B, and if UP-316 is dead, fall back to A.
Disposition C is hard to justify here because, unlike the IG-
specific ValueSets that received *Not Persuasive*, this ValueSet
wraps a national external code system whose THO presence has
clear cross-IG value.
