## Summary

Lloyd McKenzie reports that the value set bound to `ExplanationOfBenefit.use`
is too narrow. Today the element only admits values that frame an EOB as a
response to a "process" request against a particular `Claim` instance —
i.e., `claim`, `preauthorization`, or `predetermination`. In practice an
EOB is also produced as a stand-alone, point-in-time assertion about the
state of a coverage: which claims, prior auths, and (potentially)
predeterminations have been processed to date, what has been paid, what
deductible remains, and so on. The ticket asks that
`ExplanationOfBenefit.use` (or its binding) be extended so this
"status / summary" usage of EOB can be expressed without misusing one of
the existing process-response codes.

The single cross-reference on the ticket is a FHIR artifact mention of
`ExplanationOfBenefit.use` itself; no related Jira tickets, Zulip
threads, or GitHub items were surfaced by Augury for this ticket key.

## Details

**Description:**

> EOBs aren't **only** used when submitting a claim, prior auth or
> predetermination.  EOB is also used to just provide a point-in time
> assertion of "Where are things at?" with respect to a coverage and
> will reflect information about claims, prior auths and potentially
> pre-determinations to date, as well as reflecting remaining
> deductibles, etc.  There is a need to communicate instances that
> were generated at a particular point-in-time rather than in response
> to a 'process' request for a particular Claim instance.

**Comments:**

- **Lloyd McKenzie (mckenzie) — 2024-02-01:** In Ontario, system users

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Broaden `ExplanationOfBenefit.use` to admit a stand-alone, point-in-time
"status / summary" usage. Concretely, extend the value set bound to
`ExplanationOfBenefit.use` (currently the `Use` value set covering
`claim | preauthorization | predetermination`) by adding a new code —
e.g., `coverage-status` (or `summary` / `point-in-time`) — to the
`http://hl7.org/fhir/claim-use` code system, and re-binding
`ExplanationOfBenefit.use` to the expanded value set. Update the
`ExplanationOfBenefit` resource narrative and the `use` element
definition to describe this case (an EOB instance that summarises
coverage state at a moment in time and is not the response to a single
`Claim`/`ClaimResponse` exchange), and clarify that for this usage
`ExplanationOfBenefit.claim` and `ExplanationOfBenefit.related` are
optional / may carry zero-or-many references to the underlying claims,
prior auths, and predeterminations being summarised.

#### Justification

The element name `use` is already meta-level ("what is this EOB
for?"), so adding a value that names a non-process-response purpose is
consistent with the element's intent and avoids inventing a parallel
element. Lloyd's Ontario example, plus the everyday patient-facing
quarterly/annual EOB summary, are concrete jurisdictional and
consumer-facing use cases that today have no clean encoding — implementers
are forced to either misuse `claim` for a non-claim-response artifact
or invent extensions, both of which fragment interoperability. Adding
a code is a low-disruption change: existing instances and their `use`
values remain valid, and senders that don't produce summary EOBs are
unaffected.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `ExplanationOfBenefit.use` and its bound value set unchanged, and
instead model the point-in-time / summary use case via a standard
extension on `ExplanationOfBenefit` — e.g., a boolean or CodeableConcept
extension `eob-summary-kind` with values such as `coverage-status`,
`periodic-summary`, `annual-summary`. The extension would document that
when present, the EOB is a summary artifact rather than a response to
a particular `Claim` and that `ExplanationOfBenefit.claim` may be
absent. Optionally, define a profile (e.g., `eob-coverage-summary`)
that constrains the extension as required, marks `claim` as optional,
and shows how `related` is used to enumerate the underlying claims and
prior auths covered by the summary.

#### Justification

This keeps the core `Use` value set tightly scoped to its historical
meaning ("what process produced this response?") and avoids any risk
that downstream consumers — including X12-aligned implementations and
Da Vinci IGs that already key behaviour off `use` — break or have to
re-interpret a new code. It also localises the change: jurisdictions
or projects that need the summary semantics can adopt the extension
or a profile without imposing a value-set change on every implementer
of `ExplanationOfBenefit`. The trade-off is that a reasonably common
real-world use case is signalled by an extension rather than a core
element, which is harder to discover and slightly weaker for
cross-jurisdictional interoperability.

---

### Disposition C: Decline

#### Proposal

Decline the change and direct implementers to keep using the existing
`use` codes (typically `claim`) together with the existing
`ExplanationOfBenefit.related` element to express that an instance
aggregates or references multiple prior claims/prior-auths. Document in
the `ExplanationOfBenefit` notes that point-in-time coverage summaries
are out of scope for the base resource and should be handled via
implementation guides if needed.

#### Justification

`ExplanationOfBenefit` is already one of the largest resources in the
Financial domain, and broadening `use` invites further requests to
loosen other process-oriented elements (`outcome`, `claim`, `created`
semantics, etc.). The X12 / claims-response heritage of the resource
gives `use` a precise, well-understood meaning today; admitting a
"summary" code blurs that. If the only known driver is a single
jurisdictional workflow (Ontario) and no Zulip / FM-WG discussion has
materialised on the ticket since 2024, the case for a normative change
to a widely-implemented resource is weak.

---

### Recommendation

**Recommended disposition:** A

The request describes a concrete, recurring real-world usage of EOB —
both jurisdictional (Ontario coverage-status queries) and consumer-facing
(periodic/annual summary EOBs) — that the resource is in fact already
expected to carry, since EOB is the only Financial-domain resource that
aggregates adjudicated benefit information. The mismatch is purely in
the `use` value set, not in the resource's data shape: `claim`,
`related`, `benefitBalance`, and `payment` already support the summary
view. Adding one well-named code (Disposition A) closes the gap with
minimal disruption and avoids forcing every implementer of the summary
pattern to adopt a project-specific extension (Disposition B), while
Disposition C leaves a real interoperability gap that implementers are
already working around. The Financial Management work group should
confirm the exact code (`coverage-status` vs `summary` vs
`point-in-time`) and update the `use` element definition and
`ExplanationOfBenefit.claim` cardinality guidance accordingly.
