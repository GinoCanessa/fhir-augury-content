## Summary

Nick Radov reports a documentation/content defect in the `Contract` resource:
the requirements text on `Contract.term.offer.topic` (originally referencing
`Contract.topic`) appears to have had an entire structure-definition fragment
inadvertently pasted into it. The visible string includes raw cardinality
markers (`[0..*]`, `[0..1]`, `[x]`), the literal definition of
`Contract.valuedItem.entity[x]`, and the type signature
`CodeableConcept|Reference(Any)`, all of which read as garbled prose rather
than guidance. The reporter believes the intent was a simple cross-reference
link from `Contract.term.offer.topic` to `Contract.topic` (and, by extension,
to `Contract.valuedItem` / `Contract.valuedItem.entity[x]`), and that the copy
of the structure definition was accidental. No related Jira tickets, Zulip
threads, or GitHub items were surfaced by cross-reference lookups; the
ticket's only cross-references are to the FHIR elements it mentions.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a definition-text bug on `Contract.term.offer.topic` and rewrite
the affected text in the element's `requirements` (or `comment`/`definition`,
whichever currently holds the garbled text — the ticket text reads "Answer:
The Contract.topic..." which is characteristic of a Q&A-style `requirements`
entry). Replace the embedded structure-definition fragment with a clean
sentence that simply cross-references the related elements, e.g.:

> The topic of the offer. This typically aligns with the overall
> `Contract.topic`, and may be described in more granular form via
> `Contract.term.asset` or, where pricing is in scope, via
> `Contract.term.asset.valuedItem` (whose subject is identified by
> `Contract.term.asset.valuedItem.entity[x]`).

Remove the literal cardinality tokens (`[0..*]`, `[0..1]`, `[x]`), the
"Definition:" label, and the `CodeableConcept|Reference(Any)` type signature
from the prose. Verify the same Q&A block is not duplicated on
`Contract.topic` itself or on `Contract.term.topic` (the element was renamed
between R4 and later versions), and clean up any sibling occurrences.

#### Justification

The reporter has correctly identified a clear authoring defect: structure-
definition metadata has been pasted into the human-readable element text,
producing prose that is unreadable and misleading to implementers. There is
no semantic change being requested — only a presentation fix — so the change
is low-risk, non-breaking, and consistent with FHIR's general expectation
that `requirements`/`comment` text be plain English. pknapp's 2021 note
("Being reviewed in light of recent changes") suggests the fix has been
deferred pending the Contract resource refactor; with that refactor now
largely settled in R5/R6, the cleanup can proceed against the current
element naming.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only patching the prose on `Contract.term.offer.topic`, perform a
broader sweep of the `Contract` resource's element text to find and fix all
similar copy/paste artifacts. Concretely: grep the Contract definitions
(across `Contract`, `Contract.term`, `Contract.term.offer`, `Contract.topic`,
`Contract.term.asset`, `Contract.term.asset.valuedItem`, and the
corresponding `.entity[x]` elements) for tell-tale fragments such as bare
`[0..*]`/`[0..1]` cardinality tokens, the literal substring
`CodeableConcept|Reference(Any)`, and stray `Definition:` labels inside
`requirements`/`comment` text. Replace each occurrence with a clean
cross-reference, and add a CI/lint check (or a one-off review checklist
item) so future paste-ins of structure-definition fragments are caught.

#### Justification

Copy/paste errors of this shape rarely occur in isolation; the same authoring
session that produced the garbled text on `Contract.term.offer.topic` likely
touched neighbouring elements. Fixing only the one element the ticket calls
out leaves the resource in a partially-cleaned state and risks a future
ticket being filed for each remaining instance. A broader sweep is still
non-breaking and only marginally more effort than the minimal fix, and the
lint check adds durable value across future Contract editing cycles. The
trade-off vs. Disposition A is scope: this expands the work beyond what was
literally requested, which the workgroup may prefer to handle as a separate
hygiene task.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive – Already Addressed** if a reviewer can
confirm that the offending text has already been removed or restructured by
later Contract edits (for example, if `Contract.term.offer.topic` no longer
exists in the current build, or if its `requirements`/`comment` text has
already been rewritten as part of the R5/R6 refactor). The disposition would
state that the original defect is no longer present in the current source,
cite the current rendered definition, and close without further change.

#### Justification

The ticket was filed in January 2019 against a much earlier Contract
structure, and the only comment (March 2021) explicitly notes that the
element is "being reviewed in light of recent changes." The Contract
resource has undergone substantial reshaping since then (notably the
introduction of `Contract.term.asset` and the move of `valuedItem` underneath
it). If the offending Q&A-style `requirements` block has already been
rewritten or the element renamed, there is nothing left to fix and keeping
the ticket open adds noise. Declining is only defensible after positively
verifying current state — otherwise Disposition A applies.

---

### Recommendation

**Recommended disposition:** A

This is a straightforward, low-risk text-cleanup defect with a clear,
specific reporter description and no controversy in the (very thin) review
trail — the only follow-up comment simply parks the ticket pending the
Contract refactor. With the refactor now largely settled, there is no reason
to leave the garbled `requirements` text in place: it actively misleads
implementers reading `Contract.term.offer.topic`. Disposition B is a
reasonable upgrade if Financial Mgmt has bandwidth to do a broader Contract
text sweep, and it can be folded in during the same edit, but the minimum
viable fix described in A should ship regardless. Disposition C is only
appropriate if a reviewer first confirms the text has already been corrected
in the current build; absent that confirmation, declining would silently
leave a known authoring bug in the resource.
