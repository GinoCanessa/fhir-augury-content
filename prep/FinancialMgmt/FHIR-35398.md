## Summary

The ticket asks the CARIN IG for Blue Button® Financial Management workgroup
to relocate the canonical URL of the `C4BBRelatedClaimRelationshipCodes`
CodeSystem so that it is anchored under `terminology.hl7.org` (THO) instead
of `hl7.org/fhir`. The reporter notes that a companion UTG ticket should be
created to register the CodeSystem in THO. The associated UTG ticket
[UP-313](https://jira.hl7.org/browse/UP-313) ("Create Related Claim
Relationship CS/VS that is not an example", reporter/assignee Corey Spears)
is in **Proposal Draft** status and is the upstream prerequisite: the
CARIN BB change can only land cleanly once the CodeSystem (and value set)
exists in THO. The ticket was originally part of CARIN BB STU2 Block-Vote-6
and was resolved as **Considered for Future Use** (vote 13-0-0 on
2022-07-26 by Corey Spears / Kathleen Conner) with the disposition "Plan
on moving when UP-313 is completed and published." Lynn Laakso later
re-opened the ticket post-STU2 publication so it can be re-dispositioned
once UP-313 lands.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive — relocate the `C4BBRelatedClaimRelationshipCodes`
CodeSystem (and its associated ValueSet, if applicable) so that its
canonical URL is anchored under `http://terminology.hl7.org/CodeSystem/...`
instead of `http://hl7.org/fhir/us/carin-bb/CodeSystem/...`. Concretely:

1. Wait for / drive UP-313 to publication so the CodeSystem (and any
   companion ValueSet) is registered in THO with a stable canonical.
2. In the CARIN BB IG, remove the locally-defined
   `C4BBRelatedClaimRelationshipCodes` CodeSystem resource and any
   inline ValueSet that compose-includes the local URL.
3. Update every `binding.valueSet` and `binding.system` reference in
   CARIN BB profiles (notably the ExplanationOfBenefit profiles that
   use `related.relationship`) to point to the new THO canonical(s).
4. Add a CARIN-BB-side dependency on the appropriate THO release in the
   IG `dependsOn` list / `package-list.json`, and update narrative /
   change-log to document the canonical migration.
5. Coordinate with downstream implementers via the CARIN BB Zulip
   stream so that test bundles and validators are refreshed against
   the new canonical URL before publication.

#### Justification

This is the resolution the workgroup already endorsed 13-0-0 on
2022-07-26 (recorded by Corey Spears / Kathleen Conner) and that Lynn
Laakso re-opened post-STU2 specifically to execute. It is also the
HL7-wide pattern for any CodeSystem that is not strictly an example:
non-example terminology belongs in THO so it is governed by UTG, can
be reused outside the originating IG, and gets a stable, canonical
URL independent of any one IG's release cadence. UP-313 is already
filed and assigned to Corey Spears, so the upstream prerequisite is
in motion. The `Terminology` label on this ticket explicitly flags it
as terminology-governance work, reinforcing that THO is the correct
home.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification — keep the CodeSystem authored inside
the CARIN BB IG sources for the next minor release, but change its
canonical URL to a stable `http://terminology.hl7.org/CodeSystem/...`
identifier and arrange for THO to publish the same artifact under that
canonical (an "extension content" / cross-publication arrangement
rather than a hard repo move). The CARIN BB IG would continue to ship
the resource bytes as part of its package; THO would re-publish the
same canonical so other IGs can resolve it.

This avoids blocking on UP-313 reaching a published THO release
before the CARIN BB editors can act, while still meeting the reporter's
core request that the canonical URL no longer be anchored under
`hl7.org/fhir/us/carin-bb`.

#### Justification

The literal blocker today is not "is the CodeSystem in THO?" but "does
its canonical URL claim a `hl7.org/fhir/...` namespace it should not
own?" Decoupling those two questions lets the workgroup ship the
canonical correction in the next CARIN BB release without waiting on
UTG's publication cycle. The trade-off vs. Disposition A: governance
of the CodeSystem stays partially with CARIN BB editors until UP-313
lands, which weakens the THO single-source-of-truth principle and
risks divergence if THO and CARIN BB ever publish different content
under the same canonical. It is, however, a well-trodden pattern
during the migration window for several other IGs in similar
situations.

---

### Disposition C: Decline

#### Proposal

Not Persuasive — close FHIR-35398 without action and leave the
CodeSystem canonical anchored under `hl7.org/fhir/us/carin-bb/...`.
Document in the resolution that the existing canonical is now in
production use across CARIN BB STU2 implementations and that any
move would constitute a breaking change for installed bases without a
clear migration benefit.

#### Justification

Defensible only on backward-compatibility grounds: CARIN BB STU2 is
published, and every payer / app developer that has loaded the IG
already resolves the CodeSystem at its current canonical. Moving it
breaks any persisted artifacts (Bundles, Provenance, validation
profiles) that referenced the old URL, and the workgroup has not (in
the indexed material) collected concrete implementer harm from leaving
it where it is. The cost: the workgroup walks back its own 13-0-0
vote and the explicit re-open by Lynn Laakso, and CARIN BB continues
to claim a `hl7.org/fhir` namespace for terminology it does not own —
which is exactly the architectural smell THO governance was created
to eliminate.

---

### Recommendation

**Recommended disposition: A (Accept As Requested) — Persuasive,
sequenced behind UP-313.**

The workgroup has already voted to move this CodeSystem to THO
(13-0-0), the reporter's request matches the broader HL7 governance
direction (non-example CodeSystems live in THO, not in IG namespaces),
and the only reason the ticket sat is that UP-313 had not yet
published. Re-opening was explicit and intentional. Disposition B is
attractive only if UP-313 stalls for an extended period; if that
becomes the situation, the workgroup can downgrade to B at execution
time without re-litigating the principle. Disposition C effectively
overturns a unanimous prior vote to preserve a known governance
defect, which is hard to defend on the record. Concretely, the
workgroup should: (1) confirm the resolution as Persuasive,
(2) make the disposition contingent on UP-313 publication, and
(3) ask the CARIN BB editors to track UP-313 so the canonical
migration ships in the next CARIN BB release after the THO release
that carries the new artifact.
