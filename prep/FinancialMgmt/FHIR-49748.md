## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the edits spelled out in the ticket's Resolution Description to the
CARIN-BB IG sources, in a post-STU 2.1 publication:

1. **CodeSystem `C4BBSupportingInfoType`** â€” rename the concept code
   `roudtrippurpose` â†’ `roundtrippurpose` (keep the display and definition
   unchanged). Because CARIN BB CodeSystems are published via
   terminology.hl7.org (THO), coordinate the code-rename with the THO
   publishing cycle as flagged in Corey Spears's comment.
2. **Profile
   `C4BB-ExplanationOfBenefit-Professional-NonClinician-Basis`** â€” rename the
   slice `supportingInfo:roudtrippurpose` â†’ `supportingInfo:roundtrippurpose`
   and update the fixed `category` binding from
   `C4BBSupportingInfoType#roudtrippurpose` to
   `C4BBSupportingInfoType#roundtrippurpose`.
3. **ValueSet `C4BBTransportationSupportingInfoType`** â€” update the included
   code reference from `C4BBSupportingInfoType#roudtrippurpose` to
   `C4BBSupportingInfoType#roundtrippurpose`.
4. **Examples `EOBProfessionalTransportation1` and
   `EOBProfessionalTransportation1-nonfinancial`** â€” update slice references
   `supportingInfo[roudtrippurpose]` â†’ `supportingInfo[roundtrippurpose]`
   and the `category` coding from `C4BBSupportingInfoType#roudtrippurpose`
   to `C4BBSupportingInfoType#roundtrippurpose`.

Target the next balloted CARIN-BB version (per the triage comment) so the
CodeSystem rename can ship alongside the THO refresh, not STU 2.1.

#### Justification

Renaming the code is consistent with FHIR practice for fixing obvious
misspellings in recently-published codes when the corrected spelling is the
plainly-intended one and there is no real-world usage that would be broken.
The slice name, value-set inclusion, fixed binding, and examples must all be
updated in lockstep because each of them either constrains to, or asserts,
the exact code string â€” leaving any of them pointing at the old spelling
would either invalidate the profile or silently misbehave in validators that
honor fixed-category bindings. The reporter and the workgroup lead have
already agreed in Zulip that this is a defect to be corrected.

---

### Disposition B: Alternative Approach

#### Proposal

Add the correctly-spelled code `roundtrippurpose` to
`C4BBSupportingInfoType` (with the intended display/definition) and
**deprecate** `roudtrippurpose` rather than deleting it: mark it
`status=retired` (or `deprecated`, per the CARIN-BB CodeSystem conventions)
with a `replacedBy` pointer to the new code. Then update the profile slice,
the value set, the fixed category binding, and the two examples to use the
new code. Retain a release-note entry telling implementers that any data
emitted against STU 2.1 using `roudtrippurpose` should be reissued against
`roundtrippurpose` going forward.

#### Justification

This is the canonical way to handle a code-identifier change in a
terminology artifact without breaking any implementer who may have started
producing data against the 2.1 CodeSystem. It preserves a resolvable URI for
the old code (important for historical EOB records), while giving new
implementations a clearly-indicated correct code and a machine-readable
pointer to it. The trade-off versus Disposition A is a slightly more
cluttered CodeSystem (one retired concept) in exchange for backward
resolvability of any data already created under STU 2.1 â€” a reasonable price
for a published IG. If the workgroup has evidence that no production payer
is emitting `roudtrippurpose` yet (plausible given 2.1 is very new), the
extra complexity is unnecessary and Disposition A is cleaner.

---

### Disposition C: Decline

#### Proposal

Close as Won't Fix, on the grounds that the misspelled code
`roudtrippurpose` is already locked into the published STU 2.1 CodeSystem
and corresponding profile/value-set, and renaming a published code is a
non-compatible change to a normative terminology artifact. Instead, document
the misspelling in the IG's release-notes / errata and leave the code,
slice, value-set, and examples unchanged.

#### Justification

Corey Spears explicitly flagged in the comment that this is "a substantive
non-compatible change." Once a code has shipped in a CodeSystem version
published on THO, any downstream system that has already persisted
`C4BBSupportingInfoType#roudtrippurpose` in claim data would see validation
breaks, broken lookups, and drift from archival EOB records if the code is
renamed. Since the misspelling does not block use of the feature (the
concept resolves; only its spelling is aesthetically wrong), declining is
defensible â€” CARIN BB could note the typo in documentation and let the
existing code live out its published lifecycle.

### Recommendation

**Recommended disposition:** A

The workgroup has already treated this as a confirmed defect in the Zulip
thread that spawned the ticket, the reporter and Corey Spears concur on the
fix, and the ticket's Resolution Description already enumerates the exact
artifact-level edits â€” the only open issue is *when*, not *what*. Corey's
triage comment ("wait until the CS is in THO with the next balloted
version") is a scheduling constraint, not an objection: it aligns the
CodeSystem rename with the normal THO publishing cycle, which is precisely
what Disposition A calls for. Disposition B's deprecate-and-replace pattern
is more conservative and worth adopting if there is any evidence of
real-world payer data already referencing `#roudtrippurpose`, but given the
code was introduced in STU 2.1 and the typo was caught essentially on first
review, the implementer footprint is almost certainly zero and the cleaner
rename in Disposition A is warranted. Disposition C understates the harm of
leaving a visibly-misspelled concept code in a published core CodeSystem
that implementers will see on every claim.
