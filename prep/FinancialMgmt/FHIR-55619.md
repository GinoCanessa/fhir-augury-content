## Summary

Daniel Venton is asking whether the `$provider-member-match` operation in the
US Da Vinci PDex IG should be marked/considered as an asynchronous operation,
applying the same justification that was used for the bulk member match
(`$bulk-member-match` / "multi-member-match"): that the data volume involved
is too large to be processed within the scope of a single synchronous HTTP
transaction. The reporter is essentially arguing for parity between
`$provider-member-match` and `$bulk-member-match` with respect to async
handling. The ticket was triaged with a proposed resolution of **Not
Persuasive**, on the grounds that async behavior in FHIR R4 is governed by
the `Prefer: respond-async` HTTP request header (the FHIR R4 Async Request
Pattern) rather than by any flag on the `OperationDefinition` itself, and
that `$provider-member-match` already follows the same setup as
`$bulk-member-match`, with the `Prefer: respond-async` requirement already
present in §pdex-162 of `provider-access-api-v2.md`. There are no
explicitly-linked Jira tickets and no Augury cross-references to other
Jira/GitHub items, but a closely-related discussion on Zulip (see below)
references the parallel and similarly-rejected ticket FHIR-55981, which
covered the same async-vs-OperationDefinition question for the output format.

## Details

**Description:**

> Shouldn't this operation be considered an ASYNC? Same justification that
> was used for the multi-member-match (bulk member match).
>
> Too much data to process within the scope of a single sync transaction.

**Resolution Description (proposed):**

> The `$provider-member-match` OperationDefinition itself doesn't need
> `async` flags — async behavior in FHIR R4 is governed by the
> `Prefer: respond-async` HTTP header pattern, not by fields on the
> OperationDefinition. The BulkMemberMatch has the same setup.
>
> `Prefer: respond-async` requirement already in §pdex-162 of
> `provider-access-api-v2.md`.
>
> The operation follows the same FHIR R4 Async Request Pattern already
> documented for `$bulk-member-match`.

## Related Zulip Discussions

### [Da Vinci PDex > Provider member match output format](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PDex/topic/Provider%20member%20match%20output%20format/near/581960322)

Rostislav Antonov asked why the Provider Access v2 page says
`$provider-member-match` "follows the FHIR Bulk Data API specification" and
returns a manifest, when the OperationDefinition declares direct OUT
parameters (`MatchedMembers`, `NonMatchedMembers`,
`ConsentConstrainedMembers`) that point to small `Group` references rather
than large per-member payloads. Michele Mottini answered that this is what
the **FHIR R4 async spec** prescribes — the completed status response must
point to NDJSON files even when the actual content is a single
`Parameters` resource — and explicitly cited the **rejected** ticket
[FHIR-55981](https://jira.hl7.org/browse/FHIR-55981) as the precedent.
Rostislav confirmed the implication ("So I should generate ndjson file with
only one line in it (Parameters resource)") and Michele agreed. This thread
directly reinforces the disposition rationale for FHIR-55619: the async
machinery (manifest + NDJSON) comes from the R4 Async Request Pattern
applied via the `Prefer: respond-async` header, not from any
OperationDefinition flag.

Participants: Michele Mottini, Rostislav Antonov.

### [Da Vinci PDex > Payer to Payer Single Member - Scoping to one Patient](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PDex/topic/Payer%20to%20Payer%20Single%20Member%20-%20Scoping%20to%20one%20Patient/near/581398433)

A long, multi-week thread on PDex member-match variants and Payer-to-Payer
scoping. Kelsey Hales summarized the four operations in play
(`$member-match` single sync, `$multi-member-match` bulk sync,
`$bulk-member-match` bulk async, `$provider-member-match` bulk async) and
asked how implementers are picking among them. Strong consensus from
Michele Mottini (CareEvolution) and Cooper Thompson (Epic) is to use
`$bulk-member-match` for payer-to-payer and `$provider-member-match` for

as a sync call because it does the same work. Disagreement remains
unresolved on whether single-member sync should stay normative; broad
agreement that `$provider-member-match` is bulk **and** async, and that
async output follows the Bulk Data manifest pattern (matching the
disposition logic of FHIR-55619). The thread also surfaces FHIR-52063
(`davinci-data-export` vs CMS-0057 `group-export` conflict), FHIR-55928,
FHIR-55939, FHIR-52121 as adjacent issues and the experimental flag bug on
`OperationDefinition-BulkMemberMatch`.

Participants: Kelsey Hales, Michele Mottini, Cooper Thompson, Daniel

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Mark `$provider-member-match` as an async operation in the IG by adding
explicit "asynchronous" semantics to the
`OperationDefinition-provider-member-match` artefact in the US Da Vinci
PDex IG (`HL7/davinci-epdx`). Concretely, this would mean (a) adding
narrative text to the OperationDefinition's `description`/`comment`
fields stating that the operation MUST be invoked using the FHIR R4 Async
Request Pattern (i.e. clients SHALL send `Prefer: respond-async` and SHALL
follow the Bulk Data status / manifest flow), and (b) cross-linking that
narrative back to the existing §pdex-162 requirement in
`provider-access-api-v2.md` so the async expectation is discoverable from
the OperationDefinition page itself, not only from the API page. This
mirrors how FHIR-55619 envisions parity with `$bulk-member-match`.

#### Justification

The reporter's underlying concern — that implementers may build a
synchronous handler and choke on the data volume — is real, even if the
mechanism (async via header, not via an OperationDefinition flag) is
already correct. Making the async expectation explicit on the
OperationDefinition resource itself is consistent with how FHIR
historically documents async-only / async-first operations in narrative,
removes a hop a reader must make from the operation page to the API
narrative, and aligns the two PDex member-match operations
(`$bulk-member-match` and `$provider-member-match`) in their on-resource
documentation. It is also a non-breaking, narrative-only change.

---

### Disposition B: Alternative Approach (Recommended)

#### Proposal

Keep the resolution as **Not Persuasive** for the literal request (no
`async` flag is added to the OperationDefinition, because no such flag
exists in FHIR R4), but **strengthen the IG narrative** so the async
expectation is impossible to miss. Specifically: (1) confirm and keep
§pdex-162 of `provider-access-api-v2.md` as the normative source of the
`Prefer: respond-async` SHALL; (2) add a short "Async Invocation" callout
block to the `OperationDefinition-provider-member-match` page (and the
matching `OperationDefinition-BulkMemberMatch` page, for consistency)
that says "This operation follows the FHIR R4 Async Request Pattern; see
§pdex-162"; (3) close FHIR-55619 with a resolution that points the
reporter at §pdex-162 and at the parallel rejection of FHIR-55981 (same
question, output-format angle). No `OperationDefinition` schema change
and no `async` boolean is invented.

#### Justification

This addresses the reporter's real concern (implementers missing the async
requirement) without inventing FHIR R4 machinery that doesn't exist.
Async behavior in R4 is unambiguously governed by `Prefer: respond-async`
and the Bulk Data status flow; trying to encode it as an
`OperationDefinition` flag would be either non-conformant or duplicative.
The Zulip "Provider member match output format" thread shows this is
exactly where implementers get confused — they read the OperationDefinition
in isolation and miss the Bulk Data tie-in — so the fix belongs in the
narrative, not the schema. It is also the option most consistent with the
already-rejected FHIR-55981, which is the Augury-linked precedent for this
exact disagreement.

---

### Disposition C: Decline (status quo)

#### Proposal

Keep the proposed resolution **Not Persuasive** as-is and close
FHIR-55619 with the existing resolution text: the OperationDefinition does
not need `async` flags because async is governed by `Prefer:
respond-async` per FHIR R4, and §pdex-162 of `provider-access-api-v2.md`
already imposes that requirement. Make no changes to the IG.

#### Justification

Defensible on three grounds: (1) FHIR R4 has no `async` field on
`OperationDefinition`, so the literal request cannot be implemented; (2)
the requirement the reporter wants is **already** normative in §pdex-162,
so there is no conformance gap; (3) `$bulk-member-match` is structurally
identical (no `async` flag, same `Prefer: respond-async` requirement) so
there is no parity gap to close. Risk: implementers reading only the
OperationDefinition page can still miss the async requirement, which is
the failure mode the reporter is implicitly raising and which the
"Provider member match output format" Zulip thread shows in the wild.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is the right balance. Disposition C is technically correct
and defensible — there is no `async` flag in R4 to add, and §pdex-162
already says SHALL — but it leaves the documented confusion (visible in
the "Provider member match output format" Zulip thread, which directly
cites the also-rejected FHIR-55981) in place, which means future tickets
on the same point are likely. Disposition A overshoots: it risks
implying a non-existent OperationDefinition-level async semantic. B keeps
the technically correct "Not Persuasive" stance on the literal ask, but
addresses the reporter's real concern by pulling the `Prefer:
respond-async` SHALL up onto the OperationDefinition page itself (and the
matching BulkMemberMatch page) as a callout. That should reduce the rate
of duplicate "shouldn't this be async?" / "what's the output format?"
tickets without inventing any new normative machinery, and it costs only a
small narrative edit in `HL7/davinci-epdx`.
