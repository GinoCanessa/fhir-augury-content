## Summary

The submitter (Mark Scrimshire) asks the Da Vinci PDex IG to publish a
single, machine-collected listing of every conformance statement
(`SHALL`/`SHALL NOT`, `SHOULD`/`SHOULD NOT`, `MAY`) that appears in the
IG's narrative pages (`input/pagecontent/*.md`), leveraging an upcoming
HL7 IG-Publisher feature designed for that purpose. The resolution
recorded on the ticket is **Persuasive**, with the agreed action being
to add a `narrative-conformance.md`/`.html` page to the IG that lists
all conformance statements found in the narrative. The ticket carries
the `STU2.1-upate` label, scoping the work to the in-progress STU 2.1
update; the only comment, from `ekivemark`, confirms the change will be
implemented "in accordance with FHIR Publisher capabilities". No
related Jira tickets surfaced via cross-references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the recorded resolution: add a new narrative page
`input/pagecontent/narrative-conformance.md` to `HL7/davinci-epdx`
that uses the upcoming HL7 IG-Publisher feature to enumerate every
conformance statement (`SHALL`, `SHALL NOT`, `SHOULD`, `SHOULD NOT`,
`MAY`) found across `input/pagecontent/*.md`. Wire the page into
`sushi-config.yaml` under `pages` (e.g.,
`pages.pagecontent.narrative-conformance.md.title: "Conformance
Statements in Narrative"`), and add a Table-of-Contents entry so the
generated `output/narrative-conformance.html` is reachable from the
IG's main menu. Land the change on the active STU 2.1 publication
branch (`pre-publish_stu2` per the README), and add a
`changehistory.md` entry citing FHIR-53793 (already present in the
cross-references).

#### Justification

This implementation is the lowest-risk way to deliver exactly what the
submitter asked for. It uses an IG-Publisher capability (per
ekivemark's comment) rather than custom tooling, so the listing stays
in lockstep with narrative edits automatically. It also leverages the
existing PDex precedent of treating conformance language as a
first-class concern (`Requirements-fromNarrative.json`,
`conformance_statements.csv`, `process_conformance.py` already exist in
the repo per the briefing), making the new page a natural extension
rather than a new direction. Because the change is narrative-only and
does not touch FSH, profiles, ValueSets, or capability statements,
there is no risk to the triple US Core pin or to dependent Da Vinci
IGs.

---

### Disposition B: Alternative Approach

#### Proposal

Implement the same listing but bind it to a richer, structured
artifact rather than a single rendered page. Specifically:

1. Continue to generate `narrative-conformance.md`/`.html` via the
   IG-Publisher feature (as in Disposition A), **and**
2. Replace the bespoke `process_conformance.py` /
   `conformance_statements.csv` / `Requirements-fromNarrative.json`
   pipeline with the publisher's output as the single source of truth,
   exporting a structured download (CSV + JSON) alongside the rendered
   page. Mark `process_conformance.py` as legacy in the README and
   remove the now-redundant CSV/JSON from the source tree once the
   publisher equivalents are in place.
3. Add per-statement anchors (`<a id="...">`) so individual
   conformance statements can be deep-linked from elsewhere in the IG
   and from external test suites / DTR / Inferno artifacts.

#### Justification

The repo already carries two parallel conformance-tracking artifacts
(`process_conformance.py` and the CSV/JSON pair). Adopting the
IG-Publisher feature *in addition to* leaving those in place creates
three competing sources for the same information, increasing the
chance they diverge silently. Disposition B preserves the user-visible
deliverable from Disposition A while collapsing the sources of truth
to one (publisher-produced) and adds anchorability so downstream
artifacts (Inferno conformance test cases, DTR rule references, payer
implementation guides) can cite specific statements stably. The cost
is modest: removal of a small Python helper and a regenerated CSV/JSON
download. The trade-off vs. A is a slightly larger scope and a small
risk that the publisher feature's schema differs from the current
`Requirements-fromNarrative.json` shape, requiring downstream
consumers (if any exist outside the repo) to update.

---

### Disposition C: Decline

#### Proposal

Decline to add a generated narrative-conformance listing on the basis
that PDex already maintains repo-local conformance tracking
(`Requirements-fromNarrative.json`, `conformance_statements.csv`, and
`process_conformance.py`). Direct readers who want a consolidated view
to those existing artifacts and resolve the ticket as **Not
Persuasive — already addressed by existing mechanism**.

#### Justification

If the workgroup judges that the existing repo-local tracking is
sufficient and that adding a publisher-generated page duplicates
effort without serving a new audience, Disposition C is defensible.
The bespoke pipeline is already wired and produces a structured
artifact that automation can consume, arguably with more flexibility
than a flat HTML page.

---

### Recommendation

**Recommended disposition:** B

The ticket's recorded resolution is **Persuasive**, the submitter and
implementer have already agreed in-thread to use the upcoming
IG-Publisher feature, and the change-history pages have been updated
to announce FHIR-53793 — so Disposition C contradicts the established
direction and should not be the workgroup's starting point. Between A
and B, the determining factor is that PDex already has two
hand-maintained conformance-tracking artifacts (CSV + JSON) plus a
helper script. Layering an IG-Publisher-generated page on top
(Disposition A) leaves three sources of truth that must be kept in
sync; Disposition B delivers the same user-visible artifact as A while
collapsing the sources to one and adding deep-link anchors that
downstream Da Vinci consumers (Inferno, DTR, payer IGs) can cite. The
incremental work over A is small — retiring `process_conformance.py`
and the redundant CSV/JSON — and it leaves PDex with a sustainable,
publisher-driven conformance listing instead of a tooling stack the
WG must continue to maintain.
