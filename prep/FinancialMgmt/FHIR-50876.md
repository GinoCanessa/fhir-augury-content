## Summary

A FHIR Core QA report flagged six coded elements on the `InsurancePlan`
resource that lack any ValueSet binding (reported at INFORMATION
severity): `InsurancePlan.type`, `InsurancePlan.generalCost.type`,
`InsurancePlan.specificCost.category`,
`InsurancePlan.specificCost.benefit.type`,
`InsurancePlan.specificCost.benefit.cost.type`, and
`InsurancePlan.specificCost.benefit.cost.qualifier`. The FHIR rule that
elements typed `CodeableConcept` or `Coding` must carry a binding is
violated for all six. During the joint PA/FM session at the September
2025 WGM, the work groups agreed to scope this ticket to **only the
`generalCost` and `specificCost` properties**, deferring the
`InsurancePlan.type` binding to the linked, already-resolved harmonization
ticket [FHIR-24915](https://jira.hl7.org/browse/FHIR-24915) ("Harmonize
Coverage and InsurancePlan type", resolution *Persuasive with
Modification*). FHIR-24915 establishes the pattern of binding
`InsurancePlan.type` to `http://terminology.hl7.org/ValueSet/insuranceplan-plan-type`
(Preferred), and is the source of truth for the `.type` element. Cooper
Thompson's read of the cost structure is that
`generalCost` models things like deductibles (cost independent of any
specific service) while `specificCost` models things like co-pays (cost
that varies with service); FM has asked Robert Dieterle (rcdieterle)
whether existing IGs (NDH, Da Vinci, etc.) already carry value sets
for these concepts that could be promoted to THO and bound here. As of
January 2026 Robert has provided recommendations in an attached document
that the reviewing group still needs to evaluate.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add ValueSet bindings to all six flagged elements on `InsurancePlan` in
the FHIR Core specification:

1. **`InsurancePlan.type`** — bind to
   `http://terminology.hl7.org/ValueSet/insuranceplan-plan-type`,
   strength **Preferred** (per the draft resolution in
   [FHIR-24915](https://jira.hl7.org/browse/FHIR-24915)).
2. **`InsurancePlan.generalCost.type`** — bind to a new (or existing
   THO) value set covering deductible-style cost categories drawn from
   Robert Dieterle's January 2026 recommendation attachment, strength
   **Example**.
3. **`InsurancePlan.specificCost.category`** — bind to an existing THO
   value set for benefit categories (e.g., the value set used for X12
   Service Type Codes / EOB.category-style concepts), strength
   **Example**.
4. **`InsurancePlan.specificCost.benefit.type`** — bind to a value set
   of benefit-type concepts (e.g., primary care, specialty care,
   inpatient, outpatient — the same gap noted on
   `InsurancePlan.coverage.benefit.type` in FHIR-24915), strength
   **Example**.
5. **`InsurancePlan.specificCost.benefit.cost.type`** — bind to a value
   set distinguishing cost components (e.g., copay, coinsurance,
   deductible-after, out-of-pocket-max), strength **Example**.
6. **`InsurancePlan.specificCost.benefit.cost.qualifier`** — bind to a
   value set for cost qualifiers (e.g., in-network, out-of-network,
   preferred, non-preferred), strength **Example**.

The actual value set URIs and strengths must be drawn from Robert
Dieterle's attached recommendations and reconciled against THO and the
NDH / Da Vinci PDex Plan-Net IGs to avoid creating duplicates.

#### Justification

This is exactly what the QA rule requires: every `CodeableConcept` or
`Coding` element must carry a binding, even if only at strength
*Example*. Binding strength *Example* explicitly signals "use this if
you don't have something better" without forcing implementer behaviour,
which is appropriate for cost concepts that vary by jurisdiction,
payer, and benefit design. Reusing existing THO value sets (and
patterns established by FHIR-24915 for `InsurancePlan.type`) keeps the
core spec consistent with the directory and payer IGs that are already
publishing implementations of `InsurancePlan`.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the QA rule literally only where there is real-world community
agreement on a value set, and use a placeholder approach for the
remainder:

- Bind `InsurancePlan.type` per [FHIR-24915](https://jira.hl7.org/browse/FHIR-24915)
  (Preferred) — no change from Disposition A.
- For the five `generalCost` / `specificCost` elements, bind each to
  `http://hl7.org/fhir/ValueSet/icd-10` … no — bind each to a single
  newly minted, intentionally minimal *Example* value set under THO
  (e.g., `valueset-insuranceplan-cost-type`,
  `valueset-insuranceplan-cost-qualifier`,
  `valueset-insuranceplan-benefit-category`,
  `valueset-insuranceplan-benefit-type`) populated from the IGs
  Robert Dieterle has identified, and explicitly call out in the
  element definitions that implementers in regulated markets (US ACA
  plans, EU SHI, etc.) are expected to slice these down via profiles.
- Where a comparable IG-defined value set already exists (e.g., NDH or
  Da Vinci PDex Plan-Net), bind to *that* value set directly with
  strength **Example** rather than re-publishing it under THO.

#### Justification

Cost-modelling vocabulary in `InsurancePlan` is one of the more
jurisdiction-specific corners of the spec (US copay/coinsurance/HSA
concepts do not map cleanly onto e.g. UK NHS or single-payer EU
schemes). Adopting too strong a binding too early — or inventing a
THO value set that competes with NDH/PDex/Plan-Net — risks creating
ecosystem fragmentation and forcing breaking changes later. Pinning
*Example* bindings to either tiny THO seed value sets or directly to
existing IG value sets satisfies the QA rule while leaving the real
terminology curation work to the IGs that have implementer commitments.
This also addresses Cooper Thompson's question to Robert Dieterle
("Are there value sets from an existing IG we could promote up to the
core spec?") with a "bind, don't necessarily promote" answer.

---

### Disposition C: Decline

#### Proposal

Do not add value set bindings to the five `generalCost` /
`specificCost` elements at this time; instead, mark the QA finding as
*Not Persuasive* (or *Persuasive — Considered for Future Use*) and
defer the work pending either (a) the broader review of
`InsurancePlan` planned for the next normative cycle, or (b)
adoption of cost-modelling value sets in a major payer/directory IG
(NDH, Da Vinci PDex Plan-Net, or US Core Plan-Net) that core can then
trace. `InsurancePlan.type` is still addressed via
[FHIR-24915](https://jira.hl7.org/browse/FHIR-24915).

#### Justification

The QA flag is at INFORMATION severity, not ERROR or WARNING — it does
not block publication. `InsurancePlan` is a Trial Use resource whose
cost-modelling sub-tree has known limitations and is the subject of
ongoing work in multiple IGs. Inventing core-level *Example* bindings
without strong IG consensus risks (a) being immediately overridden by
those IGs anyway, and (b) creating churn when the resource is
restructured. Declining keeps core stable and pushes the terminology
work to the venues actively curating it.

---

### Recommendation

**Recommended disposition:** A (with Disposition B as the working
implementation tactic).

The QA rule exists for a reason — every coded element should carry at
least an *Example* binding so implementers and tooling have a
discoverable starting point, and Cooper Thompson has already framed
the joint PA/FM review around finding usable value sets rather than
declining the request. Robert Dieterle has now (January 2026) provided
concrete recommendations, so the bottleneck that previously justified
deferral has cleared. Disposition C effectively kicks the can on a
problem the work group has been carrying since 2019 (cf.
[FHIR-24915](https://jira.hl7.org/browse/FHIR-24915)), and an
INFORMATION-level QA flag that has been open for years still represents
real implementer pain — implementations of `InsurancePlan` in NDH and
Da Vinci PDex Plan-Net have already been forced to invent bindings, so
core silence costs more than it saves. Disposition A (accept) is the
right answer at the *intent* level; in execution it should follow
Disposition B's pragmatic guidance — strength *Example* throughout the
cost sub-tree, reuse IG-defined value sets where they exist instead of
forking them into THO, and lean on the `InsurancePlan.type` decision
already locked in by FHIR-24915. The work group should walk through
Robert Dieterle's attachment in detail before finalising the URIs.
