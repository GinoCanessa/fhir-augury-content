## Summary

Lloyd McKenzie (FHIR Infrastructure) is asking the Da Vinci ATR (Member
Attribution List) authors — owned by Financial Management — to come to
FHIR-I to discuss a set of Group extensions defined in the
2.0.0-ballot of the Da Vinci ATR IG that appear to be either
under-defined or methodologically inappropriate. Two extensions are
called out by name:

1. **`ext-changeType`** — currently allows recording that a Group member
   has been *dropped*. Lloyd's concern is that "dropped" is a
   modifier-level fact about the member (it changes the meaning of the
   Group entry); per FHIR core rules this MUST be conveyed via the
   existing `Group.member.inactive` (a.k.a. "active") flag, and if it
   were to remain as an extension at all it would have to be a
   **modifier extension**, not a plain extension.
2. **`atr-any-resource-extension`** — the definition
   ("Specifies a resource that are associated with the member of the
   Group.") is too vague: it does not say *what kind* of association is
   meant, *which* resource types are valid, or *why* the link is needed,
   so it could be interpreted as "any resource with any kind of
   association," which is almost certainly broader than the IG intends.

The ticket does not propose a specific fix; it is a meeting-request
ticket asking FM to engage FHIR-I on the methodology questions. It is
still in **Triaged / Unresolved** status with no assignee.

There are no recorded cross-references to other Jira tickets, Zulip
threads, or GitHub items for this ticket. The closest related ticket

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a methodology defect against the Da Vinci ATR
2.0.0-ballot Group extensions and fix both flagged extensions in the
next ballot of the IG:

1. **`ext-changeType`** — Remove "member dropped" (and any other
   modifier-style values) from the value set used by this extension.
   Re-express member drop / inactivation by populating
   `Group.member.inactive = true` (and, where the use case requires it,
   also `Group.member.period.end`). Retain `ext-changeType` only for
   values that are *informational* about why the change occurred (e.g.,
   "added", "updated demographic", "updated coverage"); these can
   legitimately remain as a plain extension on `Group.member`. If the
   IG has a genuine need to flag *any* value as modifier-level (e.g.,
   "tentative" / "errored"), redefine that specific extension as a
   `modifierExtension` with a clear `isModifierReason`.
2. **`atr-any-resource-extension`** — Either (a) tighten the definition
   to enumerate the allowed target resource types (e.g.,
   `Coverage | Patient | Practitioner | Organization | …` per the
   actual ATR use cases), specify the nature of the association
   ("financially responsible party", "primary care provider for the
   attribution period", etc.), and rename it to something descriptive
   of that association, **or** (b) split it into multiple narrowly-scoped
   extensions (one per relationship type). In either case, the new
   definition must state *why* the resource is being attached and what
   conformant systems are expected to do with it.

The FM/Da Vinci ATR team should bring these proposed redefinitions to a
joint FM ↔ FHIR-I session (which is what the ticket literally asks
for) and capture the conclusions back on FHIR-43336 before applying.

#### Justification

This matches the ticket text directly. The "active flag" rule is a
core FHIR invariant: any element whose presence/absence changes the
clinical or business interpretation of a member entry is a modifier,
and FHIR core already supplies `Group.member.inactive` for exactly the
"dropped" case. Carrying a parallel "dropped" indicator in a plain
extension creates a real risk that conformant consumers honour
`inactive` but ignore the extension (or vice versa), producing
divergent member rosters. Tightening
`atr-any-resource-extension` is standard IG hygiene — extensions whose
definition does not constrain target type or semantics are unusable for
interoperability and routinely fail FHIR-I review. Both changes are
within FM's authority over the ATR IG and do not require any change to
core FHIR.

### Disposition B: Alternative Approach

#### Proposal

Rather than immediately rewriting the extensions, the WG accepts the
methodology critique in principle but reframes the work as an **ATR
modelling review** with FHIR-I:

1. Schedule a joint FM ↔ FHIR-I working session (the ticket's literal
   ask) and walk the *entire* set of `Group`-targeted extensions in the
   IG, not just the two named, against the FHIR extension methodology
   (plain vs. modifier, target typing, naming conventions, "must
   support" implications).
2. Produce a short methodology note as an output of that session
   describing, for each ATR Group extension: keep / convert to modifier
   / replace with core element / retire. Attach it to FHIR-43336.
3. Open follow-up tickets per affected `StructureDefinition` and
   resolve FHIR-43336 as "Persuasive with Mod – tracked by
   FHIR-XXXXX..XXXXX". The actual SD edits then land through the
   per-extension tickets in the next ballot cycle.
4. As a stop-gap in the interim publication, add narrative warnings on
   `ext-changeType` and `atr-any-resource-extension` ("Use of this
   extension to convey member drop is deprecated; populate
   `Group.member.inactive` instead.") so implementers are not led down
   a non-conformant path while the redesign is in flight.

#### Justification

The ticket itself is a *meeting request*, not a content proposal — it
explicitly says "come to FHIR-I to discuss." Treating it as a
methodology review (rather than two pinpoint SD edits) honours that
intent and is also more likely to surface other Group extensions in the
IG with the same plain-vs-modifier or under-typing problems, which the
two examples in the description strongly suggest exist
("**Several** of the extensions seem under-defined and **a few** seem
inappropriate"). The trade-off vs. Disposition A is timeline:
Disposition B defers the actual SD changes by one ballot cycle but
reduces the risk of fixing only the two cited extensions and leaving
the rest of the same class of problem in place. The stop-gap narrative
warning preserves implementer safety in the interim.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive**: keep `ext-changeType` and
`atr-any-resource-extension` as currently defined in the
ATR 2.0.0-ballot, on the grounds that the ATR IG is a US Realm Da
Vinci IG with a defined implementer community that has not reported
interoperability problems with these extensions, and that the
"dropped" semantic in `ext-changeType` is being used in practice as
audit/history rather than as the authoritative member-status flag.

#### Justification

This is the defensible path *only* if FM can demonstrate (a) that no
implementer is using `ext-changeType = dropped` to mean
"this member is no longer in the group" (i.e., everyone correctly
populates `Group.member.inactive` and uses `ext-changeType` purely as a
why/audit annotation), and (b) that
`atr-any-resource-extension` is in fact constrained somewhere downstream
(e.g., by a profile on `Group` that slices the extension by target
type) such that its loose base definition does not actually leak
ambiguity into conformance. Absent those two demonstrations, declining
is hard to defend against the FHIR-I methodology rules that the ticket
invokes — particularly the modifier-extension rule, which is a core
FHIR invariant rather than a stylistic preference.

---

### Recommendation

**Recommended disposition:** B

The ticket is, on its face, a request from FHIR-I for a *conversation*
about Group-extension methodology in the ATR IG, and the description
itself signals that the two named extensions are examples of a broader
pattern ("Several … and a few"). Going straight to Disposition A
risks fixing only the two examples and leaving sibling extensions with
the same defects, which would invite a follow-up FHIR-I ticket in the
next cycle. Disposition C is hard to defend because the
`ext-changeType = dropped` issue is a modifier-extension rule, not a
style preference, and FHIR-I will almost certainly push back. Disposition
B accepts the critique, schedules the meeting the ticket actually
asked for, scopes the fix to the right surface area (all ATR `Group`
extensions), and uses per-extension follow-up tickets to land the
actual SD changes — while protecting implementers in the interim with
narrative guidance on the two specifically-cited extensions. FM should
confirm at triage whether the FHIR-I joint session can be scheduled
before the next ballot freeze; if not, fall back to Disposition A on
just `ext-changeType` and `atr-any-resource-extension` so the most
clearly-broken pieces are not shipped again unchanged.
