## Summary

The reporter notes that Section 4.4 of the Da Vinci Payer Data Exchange
(PDex) US Drug Formulary / ATR (Member Attribution) IG currently
describes only one path for building a Member Attribution List: creating
an empty `Group` and then adding a single member at a time. They observe
that the `Group` profile actually permits multiple members to be carried
on a single POST, and ask the IG to clarify how each of the following
scenarios should be handled:

1. POST of a `Group` with multiple members marked with the
   `changeType` extension value `new`.
2. POST of a `Group` with multiple members marked with `changeType`
   value `dropped`.
3. POST of a `Group` mixing `new` and `dropped` members on the same
   submission.
4. POST of a `Group` to update an already-existing attribution list,
   with member additions and removals signalled via `changeType`.
5. Behaviour when `changeType` is omitted (since it is not mandatory)
   in any of the above scenarios.

There are no Jira cross-references and no related Zulip threads or
GitHub items in the cache for this ticket. The most relevant context is
the resolution note: the workgroup has agreed (2024-02-21 PDex / FM
discussion, re-affirmed in the 2026-03-27 PDex Community Call) that the
existing single-member narrative in §4.4 is only one example, and the
intent is to expand the narrative to describe these additional creation
and maintenance patterns. The ticket is flagged for the
`Block-Vote-2026-April` ballot disposition cycle as **Persuasive**.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update Section 4.4 of the Da Vinci ATR IG to expand the existing
single-member narrative into an explicit enumeration of the supported
member-attribution submission patterns. Concretely, add narrative (and
where helpful, small inline `Group` examples) covering:

1. POST `Group` containing N members all carrying
   `Group.member.extension:changeType = new` — interpreted as a bulk
   add of all listed members to the attribution list.
2. POST `Group` containing N members all carrying
   `changeType = dropped` — interpreted as a bulk removal of those
   members from the existing attribution list (members not listed are
   unchanged).
3. POST `Group` containing a mix of `new` and `dropped` members in the
   same submission — applied as a single transactional add/remove
   delta.
4. POST `Group` against an already-populated list to apply incremental
   add/remove deltas via `changeType`, distinguishing this delta-style
   submission from a full snapshot replacement.
5. Default behaviour when `changeType` is absent: state the IG's
   intended interpretation (e.g., treat the submission as a full
   snapshot of current membership, or treat each member as `new`,
   whichever the WG endorses) and recommend that producers populate
   `changeType` whenever the submission is a delta to avoid ambiguity.

Keep the existing one-member-at-a-time bullet as one of the listed
patterns. No conformance changes (cardinalities, MustSupport, value
sets, profile constraints) — narrative-only update consistent with the
"information/clarification only" note from the 2/21/2024 FM/PDex
discussion and the **Persuasive** resolution. Cross-link the
`changeType` extension page from the new narrative.

#### Justification

This is exactly what the workgroup agreed to in both the 2024-02-21 FM
discussion and the 2026-03-27 PDex Community Call: the §4.4 text
currently understates what the `Group` profile already supports, and
the fix is to broaden the narrative rather than alter the profile. The
ticket already calls out each scenario in actionable detail, the
`changeType` extension already exists to carry the semantics, and the
update poses no conformance or backward-compatibility risk. Adopting
the request as written also closes scenario 5 (absent `changeType`) by
documenting an explicit default, removing a real interpretation gap
that implementers have flagged.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same narrative expansion as in Disposition A, but factor it
into a dedicated "Submission Patterns for Member Attribution Lists"
sub-section (e.g., 4.4.x) rather than inlining a long enumeration into
the existing 4.4 bullet list. Structure the new sub-section around a
small table that maps each pattern to:

- HTTP interaction (`POST [base]/Group` vs. `PUT [base]/Group/{id}`
  vs. conditional update),
- value(s) of `Group.member.extension:changeType` on each member,
- expected server-side semantics (full replace vs. delta add/remove),
- and the recommended behaviour when `changeType` is omitted (with a
  SHOULD-level guidance note that producers populate `changeType` for
  any delta submission).

Optionally add one or two `Bundle`-wrapped example instances under the
IG's `examples/` content so that the patterns are testable and
referenceable from implementer questions.

#### Justification

The five scenarios in the ticket interact in subtle ways
(snapshot vs. delta, server-side reconciliation when `changeType` is
absent), and embedding them as additional bullet points in §4.4 risks
the same ambiguity the ticket is trying to resolve. A small table plus
one or two worked examples gives implementers an unambiguous reference
they can point CI tests at, while still respecting the workgroup's
decision that this is a clarification rather than a normative change.
This also makes future tickets easier to scope because the
attribution-submission semantics live in one named location instead of
being threaded through prose.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the FHIR
`Group` resource and the `changeType` extension already, by
themselves, describe how multiple members can be added or removed in a
single POST, and that further IG narrative is unnecessary. Direct
implementers to the base `Group` resource semantics and to the
`changeType` extension's own `StructureDefinition` page.

#### Justification

This would only be defensible if the WG concluded that the §4.4
narrative is not actually misleading and that the existing extension
documentation is sufficient. Even then it leaves scenario 5 (behaviour
when `changeType` is absent) genuinely undocumented, which is the one
concrete interoperability gap the ticket identifies. Decline is listed
here for completeness; the standing FM/PDex discussion notes already
contradict it.

---

### Recommendation

**Recommended disposition:** A

The workgroup has twice — in the 2024-02-21 FM/PDex discussion and
again in the 2026-03-27 PDex Community Call — landed on the same
position: §4.4 currently describes only one of several legitimate
patterns for building a Member Attribution List, and the right fix is
to expand the narrative to enumerate the rest. The ticket's resolution
is already recorded as **Persuasive** with a clear scope ("expand on
the explanation to include what is described in this ticket as one of
the ways of creating member attribution list"), and the change carries
no conformance risk. Disposition B is a refinement of A worth keeping
on the table during the apply step (especially the explicit handling
of absent `changeType`), but it should not delay acceptance. Disposition
C contradicts the standing WG decision and leaves the absent-
`changeType` ambiguity in place. Accept as requested, with the
understanding that the editor may choose the structural presentation
(inline bullets vs. dedicated sub-section + table) when applying.
