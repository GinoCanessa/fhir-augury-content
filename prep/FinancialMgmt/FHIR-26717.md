## Summary

The submitter requests that several NCPDP retail-pharmacy claim data
elements — specifically *Days supply*, *DAW (Dispense As Written) product
selection code*, *Refill number*, *Prescription origin code*, and *Plan
reported brand-generic code* — be promoted from being carried as
`supportingInfo` entries in implementation guides (notably CARIN BB) to
first-class backbone data elements on the FHIR Core financial resources
`Claim`, `ClaimResponse`, and `ExplanationOfBenefit`. The rationale is
that these data elements are required on every retail pharmacy claim,
so encoding them as opaque `supportingInfo` slices loses semantic
fidelity and forces every implementer to re-discover the slicing
convention. A FM committee comment on 2025-10-30 from Corey Spears
clarifies that while the original submission was filed against the
CARIN BB IG, it is in fact a FHIR Core request and should be evaluated
against the base specification. The ticket was originally dispositioned
"Considered for Future Use" by an 18-0-5 vote in 2020 (Mark Roberts /
Pat Taylor), but Lynn Laakso reverted that resolution and re-opened it
post-STU2 publication in November 2022, so the change is once again on
the FM committee's agenda. No related Jira tickets, Zulip threads, or
GitHub items were surfaced via cross-references.

### Disposition A: Accept As Requested

#### Proposal

Add the five named pharmacy data elements as first-class backbone
elements on `Claim`, `ClaimResponse`, and `ExplanationOfBenefit` in
FHIR Core. Concretely:

- On `Claim`, `ClaimResponse`, and `ExplanationOfBenefit`, extend the
  per-line backbone (e.g., `Claim.item`, `ExplanationOfBenefit.item`,
  `ClaimResponse.item`/`addItem`) with explicitly-typed elements:
  - `daysSupply` : `Quantity` (with UCUM `d`) — number of days the
    dispensed quantity is expected to last.
  - `dawCode` : `CodeableConcept` (binding example/preferred to NCPDP
    DAW Product Selection Code value set, 0–9).
  - `refillNumber` : `positiveInt` or `unsignedInt` — refill sequence
    (0 = original fill).
  - `prescriptionOriginCode` : `CodeableConcept` (binding to NCPDP
    Prescription Origin Code, 0–5).
  - `brandGenericIndicator` (Plan reported brand-generic code) :
    `CodeableConcept` bound to the corresponding NCPDP code list.
- Update the resource narratives, element definitions, examples, and
  the financial-summary pages on `claim.html`,
  `claimresponse.html`, and `explanationofbenefit.html` to describe
  the new elements and their NCPDP provenance.
- Mark the existing `supportingInfo` slice patterns used in CARIN BB
  STU2 for these elements as historical / superseded once the core
  elements ship; CARIN BB and other pharmacy-flavored IGs would then
  re-bind to the new backbone elements rather than to slice categories.

#### Justification

These five elements are not optional context — every retail pharmacy
claim in the U.S. carries them, and they are non-pharmacy-specific in
the same way that `Claim.item.quantity` and `Claim.item.unitPrice` are
not "medical" or "dental" specific. Hoisting them to backbone elements
matches the existing FHIR pattern of promoting universally-required
adjudication context out of `supportingInfo` (which is intentionally
the IG escape hatch) into typed elements. It also reduces interop cost:
CARIN BB consumers today must understand both the resource model and
the NCPDP-keyed `supportingInfo` slicing convention to read a basic
prescription. The 18-0-5 deferral in 2020 was explicitly "consider in
the next FHIR version", and STU2 of CARIN BB has now shipped, so the
gating concern that motivated the deferral has been resolved.

### Disposition B: Alternative Approach

#### Proposal

Leave the base resources untouched and instead define a small,
**FHIR-Core-published Pharmacy extension pack** plus a normative
profile (`pharmacy-claim`, `pharmacy-eob`, `pharmacy-claimresponse`)
that:

- Defines five named, typed extensions
  (`http://hl7.org/fhir/StructureDefinition/claim-daysSupply`,
  `claim-dawCode`, `claim-refillNumber`,
  `claim-prescriptionOriginCode`, `claim-brandGenericIndicator`) with
  the same data types and bindings as Disposition A.
- Publishes profiles on `Claim`/`ClaimResponse`/`ExplanationOfBenefit`
  that make those extensions `1..1` for the pharmacy context.
- Documents on `claim.html`/`claimresponse.html`/`explanationofbenefit.html`
  that pharmacy implementers SHOULD use the published extensions
  instead of `supportingInfo` slicing.
- Aligns CARIN BB to consume these new core extensions in its next
  release, retiring its `supportingInfo` slices over a deprecation
  window.

#### Justification

This avoids a backbone-element addition to three high-touch financial
resources, which is a cross-version compatibility consideration:
existing implementations parse `Claim`/`EOB` shapes and any new
required-by-pharmacy element added to the base would be invisible to,
or silently dropped by, older toolchains. Standard FHIR extensions
preserve schema stability while still giving the data elements typed,
discoverable, code-bound homes — which is the actual implementer pain
point Disposition A addresses. The trade-off is that pharmacy data
remains slightly second-class in the base resource (it lives under
`extension`), but with named published extensions plus a core profile,
the discoverability gap versus a backbone element is small. This is
consistent with how FHIR has historically handled domain-specific
adjudication context (e.g., dental and vision adjudication adjuncts).

---

### Disposition C: Decline

#### Proposal

Decline the request and direct pharmacy implementers to continue using
`supportingInfo` with the NCPDP-aligned categories already defined by
CARIN BB. Rationale category: **already addressed by existing
mechanism** (`Claim.supportingInfo` / `ExplanationOfBenefit.supportingInfo`
with category-coded slicing) and **out of scope for FHIR Core**
(pharmacy claim conventions are domain-specific and properly belong in
the CARIN BB / NCPDP-aligned IG layer).

#### Justification

`Claim.supportingInfo` and `ExplanationOfBenefit.supportingInfo` were
designed precisely so that domain-specific claim adjuncts — pharmacy,
dental, vision, workers' comp — could be carried in a uniform shape
without bloating the base resources. Promoting NCPDP-specific elements
to backbone status sets a precedent that every claim domain (dental
ADA, vision, transportation, DME) should expect equivalent first-class
elements, which would inflate `Claim`/`EOB` substantially. Adding new
backbone elements is also a non-trivial change for downstream
profilers and validators, and the CARIN BB `supportingInfo` slices
already provide a deployed, ballot-tested solution. The reverted 2020
disposition explicitly chose deferral rather than acceptance, and the
underlying engineering trade-off has not changed.

---

### Recommendation

**Recommended disposition:** B

The original 2020 vote (18-0-5 to defer rather than reject) and Lynn
Laakso's 2022 re-open both signal that the FM workgroup believes
pharmacy data elements deserve a more first-class home than
`supportingInfo`, but isn't yet committed to mutating the base
backbones. Disposition B threads that needle: it gives the five named
data elements typed, code-bound, discoverable definitions published in
FHIR Core (so they are no longer a CARIN-BB-only convention), without
changing the wire shape of `Claim`/`ClaimResponse`/`ExplanationOfBenefit`
for non-pharmacy implementers. It also avoids the precedent problem
called out in Disposition C — extensions are the right mechanism for
domain-specific adjuncts, and a *core-published* extension is much
stronger than the current IG-only slicing convention. Disposition A
remains a reasonable longer-term direction (and the right answer if FM
ultimately concludes pharmacy is part of base claim semantics), but it
should be revisited only after the extension-based path has bedded in
across CARIN BB and at least one other consuming IG.
