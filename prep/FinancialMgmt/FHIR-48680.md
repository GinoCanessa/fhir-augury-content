## Summary

Luke Duncan is asking that the standard `request-statusReason` extension
(part of the FHIR Extensions Pack, owned originally by FHIR-I but
applicable to workflow Request-pattern resources) be allowed on
additional financial Request resources — specifically `Claim`,
`EnrollmentRequest`, and `CoverageEligibilityRequest`. The extension
already supports several Request-pattern resources (e.g.,
`ServiceRequest`, `MedicationRequest`, `CommunicationRequest`,
`Task`-style request resources) so that implementers can record the
*why* behind a `status` value (e.g., why a request is `cancelled`,
`on-hold`, or `entered-in-error`). The reporter notes there may be other
Request resources that also belong in the context list.

A November 2024 comment from Lloyd McKenzie indicates FHIR-I is fine
with adding these contexts, and the ticket has been routed to Financial
Management because FM owns the three resources proposed for inclusion
and must concur before the extension's `context` element is widened.
There are no cross-referenced Jira tickets, no GitHub PRs/issues, and no
direct Zulip threads against this ticket key; older Zulip discussion of
the extension itself confirms that `request-statusReason` is intended to
carry the reason for a request's *state*, not the clinical/business
reason for the request itself (which lives in `.reason`).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add `Claim`, `EnrollmentRequest`, and `CoverageEligibilityRequest` to
the `context` element of the `request-statusReason` StructureDefinition
in the FHIR Extensions Pack (`HL7/fhir-extensions`). Concretely, append
three additional `context` entries with `type = element` and
`expression` values `Claim`, `EnrollmentRequest`, and
`CoverageEligibilityRequest` so the extension can be applied at the
root of each of those resources, mirroring the existing pattern used
for `ServiceRequest`, `MedicationRequest`, etc. No change to the
extension's value set, value type, or cardinality is implied; only the
applicability list grows. FHIR-I has already concurred (per Lloyd
McKenzie's 2024-11-25 comment); FM only needs to confirm the three
resources are appropriate hosts.

#### Justification

The extension's purpose — recording the reason a request is in its
current `status` (e.g., `cancelled`, `entered-in-error`, `on-hold`) —
applies cleanly to financial Request resources. `Claim`,
`EnrollmentRequest`, and `CoverageEligibilityRequest` all have a
`status` element whose lifecycle includes states (`cancelled`,
`entered-in-error`) for which an implementer may legitimately want to
attach a coded reason, and none of them have a native `statusReason`
element. Widening the context list is a non-breaking, additive change
and is consistent with how the extension is already exposed on other
Request-pattern resources. FHIR-I has signed off, so FM concurrence is
the last gate.

---

### Disposition B: Alternative Approach — Survey All Request-Pattern Resources First

#### Proposal

Before editing the StructureDefinition, perform a one-pass review of
every Request-pattern resource (FM-owned and otherwise) to identify the
complete set that should be in `request-statusReason.context`, then
update the extension once with the full list. The ticket reporter
explicitly hints at this ("There may be more"). Candidates to evaluate
alongside the three named resources include `Claim`,
`CoverageEligibilityRequest`, `EnrollmentRequest`,
`CommunicationRequest`, `DeviceRequest`, `NutritionOrder`,
`SupplyRequest`, `VisionPrescription`, `RequestOrchestration`, and any
other resource asserting the Request pattern. FM would coordinate with
FHIR-I, OO, and Pharmacy work groups as needed and update the extension
in `HL7/fhir-extensions` in a single commit.

#### Justification

Adding contexts piecemeal causes the extension's applicability list to
churn as each work group raises its own ticket. A single, coordinated
update reduces ballot/review overhead, gives implementers a stable
target, and avoids the situation where two near-identical tickets are
filed against the same extension six months apart. The trade-off is a
modest delay versus Disposition A, and the need to engage other work
groups for any non-FM resources surfaced by the survey.

---

### Disposition C: Decline

#### Proposal

Decline to add `Claim`, `EnrollmentRequest`, and
`CoverageEligibilityRequest` to the `request-statusReason` context.
Direct implementers to use the resource-specific mechanisms already
available — for `Claim`, the existing `Claim.status` plus narrative or
a profile-level extension; for `EnrollmentRequest` and
`CoverageEligibilityRequest`, file a focused change request to add a
native `statusReason` element if the use case is strong enough.

#### Justification

This is defensible only if FM judges that the financial Request
resources are sufficiently distinct from clinical Request-pattern
resources that overloading a workflow extension is undesirable, or if
FM prefers to add a first-class `statusReason` element to each
financial resource in a future version rather than carry an extension.
Note that the historical Zulip discussion (Jose Costa Teixeira, 2022)
flagged short-name collisions between `event-statusReason` and
`request-statusReason`; declining avoids further entrenching that
extension's footprint. The cost is forcing implementers to either go
without a coded status reason on these resources or to define a
workgroup-local extension that duplicates `request-statusReason`.

---

### Recommendation

**Recommended disposition:** A (with a light Disposition B follow-up
encouraged)

FHIR-I has already concurred, the change is purely additive to the
StructureDefinition's `context` element, and the three financial
Request resources have status lifecycles that genuinely benefit from a
coded reason for `cancelled` / `entered-in-error` / `on-hold`. The
existing pattern of attaching `request-statusReason` to other
Request-pattern resources is exactly the pattern this ticket asks FM
to extend, so the precedent is clear and the implementation cost is
trivial. FM should accept the three named resources now (Disposition
A) and, while the StructureDefinition is open, take a quick pass
through the remaining Request-pattern resources to fold in any other
obvious gaps in one edit (the Disposition B sweep), rather than
deferring the whole change while a wider survey runs.
