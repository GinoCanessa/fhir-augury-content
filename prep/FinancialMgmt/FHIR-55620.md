## Summary

The reporter (Daniel Venton) flags an apparent modeling inconsistency in the
US Da Vinci PDex IG's example for the Provider-Member-Match operation: the
`ProviderMemberMatchRequestExample` in
`input/fsh/ExampleProviderMemberMatchRequest.fsh` includes a `CoverageToLink`
parameter pointing to a brand-new `CoverageLinkExample1` Coverage instance.
The reporter observes that providers are not payers and therefore cannot
issue a *new* Coverage to be linked — making `CoverageToLink` an
inappropriate construct in the *provider*-member-match context (as opposed
to a payer-to-payer member-match scenario such as in HRex / payer-data
exchange where a new payer issues new coverage). The ticket has already
been resolved as **Persuasive** and is labeled **Pre-Applied**: commit
`bd9d8b9d` ("Resolve FHIR-55620: remove CoverageToLink from
provider-member-match example") removed the `CoverageToLink` parameter from
the Member 1 request block (lines 35–38) and deleted the entire
`CoverageLinkExample1` Coverage instance (~22 lines), and updated the
example description to call out that `CoverageToLink` is intentionally
omitted because providers do not issue new coverage. The `CoverageToLink`
parameter remains 0..1 in the operation definition itself — only the
example was changed. There are no Jira-linked tickets, no related Zulip
discussion was found, and the only cross-references are GitHub artifacts
inside `HL7/davinci-epdx` (the commit, the change-history page, and its
generated outputs).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change already applied in commit `bd9d8b9d` against
`HL7/davinci-epdx`:

1. In `input/fsh/ExampleProviderMemberMatchRequest.fsh`, remove the
   four-line `CoverageToLink` part from the Member 1 request block
   (the `parameter[=].part[+].name = "CoverageToLink"` block plus the
   following `resource = CoverageLinkExample1` and id assignment).
2. Delete the entire `Instance: CoverageLinkExample1 InstanceOf:
   http://hl7.org/fhir/us/davinci-hrex/StructureDefinition/hrex-coverage`
   block from the same file.
3. Update the `ProviderMemberMatchRequestExample` `Description` to state
   that `CoverageToLink` is intentionally omitted because providers do
   not issue new coverage.
4. Add a `changehistory.md` entry recording FHIR-55620.
5. Leave `Operation-provider-member-match.fsh` untouched — `CoverageToLink`
   stays 0..1 on the `OperationDefinition` so payers/intermediaries that
   *do* have a legitimate new-coverage scenario can still populate it.

#### Justification

The reporter's observation is correct on the modeling: in
provider-member-match, a *provider* is matching a patient against a
*payer*'s member records; the provider has no authority to issue a new
Coverage to be linked. Demonstrating `CoverageToLink` with a freshly-minted
HRex Coverage in the *provider*-side example was misleading and risked
implementers building provider-side flows that fabricate Coverage
resources. Keeping the parameter 0..1 on the OperationDefinition preserves
optionality for any unforeseen edge case (e.g., proxy/intermediary
implementations) without normalizing the misleading modeling in the
example. This matches the IG playbook's "modify an example" recipe
(Disposition is example-only; no profile, ValueSet, or operation contract
changes needed) and avoids any cascade through the triple US Core /
HRex pin set. The fix has already landed (label `Pre-Applied`,
resolution `Persuasive`); accepting locks the change in.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the example deletion from Disposition A **and** additionally
constrain or remove the `CoverageToLink` parameter on the
*provider*-member-match `OperationDefinition` itself
(`input/fsh/Operation-provider-member-match.fsh`). Two sub-options:

- **B1 (constrain):** Keep the parameter at 0..1 but add an inline
  comment / `documentation` string on the parameter clarifying that
  `CoverageToLink` is intended for payer-to-payer member-match flows
  (HRex `$member-match`) and is not expected to be populated by providers
  — making the operation definition self-explanatory rather than relying
  on the example narrative alone.
- **B2 (remove):** Drop the `CoverageToLink` part from
  `Operation-provider-member-match` entirely (`0..0` or removed), since
  providers have no scenario in which to populate it. Implementers needing
  the new-coverage flow would use HRex `$member-match` instead.

Either sub-option still applies the example edits in Disposition A and
records the change in `changehistory.md`.

#### Justification

The reporter's underlying objection ("providers don't issue new payer
coverage") is structural, not just cosmetic. Disposition A only fixes the
example, leaving a 0..1 parameter in the operation that has no realistic
provider-side use case — implementers reading the OperationDefinition
without the example may still mistakenly populate it. B1 is low-risk
documentation-only; B2 is cleaner but is a normative tightening of the
operation contract and would require a more careful look at any existing
implementations that may already populate `CoverageToLink` in
provider-member-match. Per the briefing this would touch
`Operation-provider-member-match.fsh` (a sibling under `input/fsh/`) and
should be cross-checked against HRex `$member-match` to keep the two
operations' parameter contracts complementary, not contradictory. Given
that `CoverageToLink` ships in the published 2.0.0 / current PDex (per
the IG version registration), B2 in particular needs a backward-
compatibility review.

---

### Disposition C: Decline

#### Proposal

Leave `ExampleProviderMemberMatchRequest.fsh` as originally authored,
including the `CoverageToLink` parameter and the `CoverageLinkExample1`
Coverage instance, on the rationale that the example is a permissive
illustration of *all* parameters available on the operation rather than
a recommended provider workflow.

#### Justification

A defensible decline argument: examples in IGs sometimes intentionally
exercise every optional parameter to give implementers a copy-and-paste
starting point, even when the modeled scenario is hypothetical. The
Description could be edited to call this out without removing content.
However, this is the weakest position because (a) the OperationDefinition
already documents `CoverageToLink` as 0..1, so an "exercise everything"
example is not necessary to communicate optionality, and (b) showing a
*provider* fabricating an HRex `hrex-coverage` Coverage instance models a
behavior PDex does not actually authorize. The decline is recorded here
for completeness but is not recommended.

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as **Persuasive** and labeled
**Pre-Applied**, with the change committed in `bd9d8b9d`. The applied
fix is example-only, surgically scoped to one authoring file
(`input/fsh/ExampleProviderMemberMatchRequest.fsh`) plus a one-line
`changehistory.md` entry, and preserves the `OperationDefinition`'s
existing 0..1 contract for `CoverageToLink` so no backward-compatibility
question is raised. Disposition A directly addresses the reporter's
concern (the misleading provider-as-payer modeling in the example)
without expanding scope into the operation contract or rippling through
the triple US Core / HRex pin set. Disposition B is a reasonable next
step that the workgroup may want to track as a follow-up ticket against
`Operation-provider-member-match.fsh` (especially the B1 documentation
tweak, which is essentially free), but conflating it with this ticket
risks turning a clean example fix into a normative-change discussion.
Disposition C is not defensible because it leaves the IG modeling a
provider behavior that PDex does not authorize. Adopt A as resolved;
optionally open a follow-up for B1.
