## Summary

Casey Trauer (reporting against the US Da Vinci PDex IG, Provider Access
API) flags ambiguity in how implementers should support the `#delta`
(`provider-delta`) `exportType` of the `$davinci-data-export` operation.
The IG previously required the payer to record per-Patient/per-Group
extensions (`lastTransmitted`, `lastResources`, `lastFilters`) so the
server could compute "what is new since the last export." In practice,
members of the same Group can have different extension values — for
example because a prior `#download` request only covered a subset of
patients, because newly-added members have no extensions yet, or because
a client previously used `_since`/`_until` not aligned with the request
date. Since `_since` only accepts a single value, the server cannot
honestly fulfil a per-member delta from a single Group request. The
reporter contrasts this with §6.2.4, which suggests the *client* is
responsible for grouping patients by `lastTransmitted` value, asking why
a server-side `#delta` exists at all if the burden actually sits on the
provider.

The ticket is tightly coupled to
[FHIR-55987](https://jira.hl7.org/browse/FHIR-55987) ("Provider access:
remove requirement to store bulk export details as extensions"), which
removed the three offending extensions (`base-ext-last-transmission`,
`base-ext-last-types`, `base-ext-last-typefilter`) and the corresponding
SHALL conformance statements (§pdex-269/276/277/280/281 in v2 and the
parallel §pdex-313/329/336/337/340/341 in v1). With the extensions gone,
this ticket's resolution closes the remaining gap by stating
unambiguously, in §pdex-266a (v2) and §pdex-326a (v1), that the **client
SHALL supply `_since`** on `provider-delta` requests, that the **client
owns the audit trail**, and that the **payer has no obligation to infer
a cutoff**. The change has been pre-applied on the `HL7/davinci-epdx`
working branch; the commit is recorded in the IG `changehistory` with
the FHIR-55987 dependency called out for traceability.

## Details

**Description:**

> We are struggling with the guidance the Provider Access API
> specification is providing us regarding supporting #delta export type.
>
> If we have a situation where members in the same Group have different
> extension values, how would we handle a #delta request? As I
> understand it, we would translate those extension values into
> `$davinci-data-export` parameter values, but we can only choose one
> `lastTransmitted` value to apply to `_since`.
>
> There is some dissonance in the guidance here. In section 6.2.4, it
> suggests that Provider clients can retrieve and evaluate those Groups
> and make requests for patients that have the same `lastTransmitted`
> value. This suggests that the burden is on the provider to ensure it
> is asking for new data. However, why do we then have a #delta export,
> which suggests the server side has the responsibility to determine
> what is new data (based on those extensions)?
>
> The edge cases that are raising this issue involve, for example:
> - `lastTransmitted` dates that vary within the group because they were
>   updated at different times due to a `#download` query that asks for
>   only a subset of patients.
> - If a payer adds a new member to the group, that member will have no
>   populated extension values.
> - If a provider client requests a set of data using `_since` and
>   `_until` that is not concurrent with the request date, then do we
>   update the `lastTransmitted`?

**Comments:**

No comments are recorded on this ticket in the data store.

**Resolution Description (already pre-applied):**

> With the removal of extensions the `#delta` becomes largely redundant.
> The IG has been updated as follows:
>
> 1. **FHIR-55987** (already committed) eliminated the root cause: the
>    `lastTransmitted` / `lastResources` / `lastFilters` extensions that
>    made it ambiguous whether the server or client was responsible for
>    tracking the delta cutoff, and introduced all the edge cases
>    (varying dates, new members, `_since`/`_until` interactions).
> 2. **This commit** closes the remaining gap with **§pdex-266a** (v2)
>    and **§pdex-326a** (v1), which clearly state:
>    - Client **SHALL** supply `_since` for `provider-delta` requests
>    - Client owns the audit trail — not the server
>    - Payer has no obligation to infer a cutoff
>
> The `changehistory` entry notes the FHIR-55987 dependency explicitly
> so the traceability is clear.

### [FHIR-55987: Provider access: remove requirement to store bulk export details as extensions](https://jira.hl7.org/browse/FHIR-55987)

Michele Mottini's request to drop the SHALL requirement that payers
write `lastTransmitted` / `lastResources` / `lastFilters` extensions
onto Patient/Group during bulk export. The objections — extensions get
overwritten by other clients sharing the same Patient/Group, read-only
clients would have to mutate resources, and the same Group is used
across multiple exports — are exactly the edge cases FHIR-51571
catalogues. Resolution (Persuasive, Pre-Applied) removed the three
extensions and their conformance statements (§pdex-269/276/277/280/281
in v2 and §pdex-313/329/336/337/340/341 in v1) and replaced them with a
SHOULD audit-trail recommendation, which is the precondition that makes
FHIR-51571's "client owns the cutoff" answer feasible.

Reporter: Michele Mottini. Assignee: Unassigned. Work group: Financial
Management. Status: Triaged / Resolution Persuasive / Pre-Applied.

## Proposed Dispositions

### Disposition A: Accept As Requested (ratify the pre-applied resolution)

#### Proposal

Accept as **Persuasive with Modification** and ratify the changes
already committed against `HL7/davinci-epdx`:

1. Keep the FHIR-55987 deletions in place: the three bulk-export
   extensions (`base-ext-last-transmission`, `base-ext-last-types`,
   `base-ext-last-typefilter`) and the related slice in
   `ProfilePDexProviderGroup.fsh` / `ExamplePDexProviderGroup.fsh`
   stay removed.
2. Retain the new conformance language in
   `input/pagecontent/provider-access-api-v2.md` **§pdex-266a** and
   `input/pagecontent/provider-access-api.md` **§pdex-326a**:
   - Client **SHALL** supply `_since` whenever the `exportType` is
     `provider-delta`.
   - Client owns the audit trail of prior export windows; the payer is
     not required to infer a cutoff.
   - Payer **MAY** maintain its own internal audit trail (SHOULD,
     mirroring the FHIR-55987 replacement language) but it is not
     normative output.
3. Retain the corresponding row in
   `input/pagecontent/changehistory.md` ("Clarify delta query
   responsibility after removal of export tracking extensions") with
   the explicit FHIR-55987 cross-link.
4. Keep §6.2.4 aligned: edit it (if not already) to point at the new
   client-owned semantics so it no longer reads as a competing
   server-side mechanism.

No FSH or terminology changes are required; no version bump beyond
what FHIR-55987 already triggered.

#### Justification

This is the change the IG has actually shipped on the working branch
and that the resolution wording explicitly endorses (Pre-Applied,
PDex-Vote-1). Putting cutoff ownership on the client is consistent with
how FHIR Bulk Data otherwise treats `_since` (Bulk Data §3.1.2 — the
client supplies `_since`, the server is stateless with respect to "what
was last seen"). It removes the three concrete edge cases the reporter
called out (mismatched per-member dates, new members with no
extensions, off-cycle `_since`/`_until` requests) without inventing new
machinery, and it satisfies FHIR-55987's complaint that read-only
clients were being forced to mutate Patient/Group resources just to
keep the delta watermark current. The repo briefing confirms this is
purely a `input/pagecontent/` narrative change — no profile, extension,
SearchParameter, CapabilityStatement, or terminology touch points — so
the cost/risk of accepting is minimal and the change is contained
within the active `pre-publish_stu2` branch already used for
FHIR-55987.

---

### Disposition B: Alternative Approach — keep server-driven `#delta` but make the contract explicit

#### Proposal

Instead of pushing all responsibility to the client, restore a
narrowly-scoped server-driven `provider-delta` mode by:

1. Re-introducing a single, **operation-level** delta watermark instead
   of per-member extensions. Concretely, define an
   `OperationOutcome`-style or `Parameters`-style extension on the
   `$davinci-data-export` *response* (not on Patient/Group) that
   reports the `lastTransmitted` value the server actually used. This
   would be authored as a new `Extension` in `input/fsh/Extensions.fsh`
   and referenced from the response example for
   `Operation-davinci-data-export.fsh`.
2. Update `input/pagecontent/provider-access-api-v2.md` §pdex-266a (and
   v1 §pdex-326a) to make `_since` **OPTIONAL** on `provider-delta`:
   if the client supplies `_since` it wins; if it does not, the server
   MAY use its own internal audit trail and MUST report the effective
   cutoff in the response extension.
3. Leave the FHIR-55987 deletions of `lastResources` / `lastFilters` in
   place — those were genuinely problematic — but allow the operation
   response to echo `_type` / `_typeFilter` filters used.
4. Update `changehistory.md` accordingly and add a connectathon track
   to validate the new response shape.

#### Justification

This preserves the original intent of having a `#delta` `exportType`
distinct from `#download`: namely, a one-shot "give me what is new"
call that does not force the client to maintain a per-member
high-water-mark cache, which is operationally hard for clients with
intermittent connectivity or Group churn. It also keeps section 6.2.4
meaningful instead of reducing it to a procedural note. The trade-off
versus Disposition A is significant: it requires a new extension and
operation-response shape (an additional `Extension*.fsh` in the flat
`input/fsh/` layout, plus an example update), it expands the v1/v2
parity surface, and it may collide with the publication timeline since
FHIR-55987 has already been merged on the assumption that all
extension-based tracking is gone. The repo briefing's "flat
`input/fsh/`" gotcha and the warning about co-versioned v1/v2 pages
both apply here, doubling the authoring footprint.

---

### Disposition C: Decline (revert and remove `#delta` from the IG)

#### Proposal

Decline the clarification and instead remove the `provider-delta`
`exportType` from the Provider Access API entirely. Specifically:

1. Drop `provider-delta` from the
   `$davinci-data-export` Operation definition
   (`Operation-…fsh`) and any binding ValueSet under
   `input/fsh/ValueSets*.fsh`.
2. Remove §pdex-266a / §pdex-326a (both the new and old wording) from
   `provider-access-api-v2.md` / `provider-access-api.md`.
3. Update §6.2.4 to instruct clients to use the existing FHIR Bulk Data
   `_since` parameter on a normal `provider-download` export instead.
4. Add a `changehistory.md` row marking the removal as the
   FHIR-51571 disposition.

#### Justification

Defensible only on the grounds that, with the FHIR-55987 deletions in
place, `provider-delta` is "largely redundant" (the resolution
description's own words) — it now means exactly the same thing as
`provider-download` plus `_since`. Removing it would simplify the
Operation surface and eliminate a source of implementer confusion
permanently. However, this is a **breaking change** for any STU 2.1.x
implementer that has wired up `provider-delta`, it would touch the
generated `Operation`, `CapabilityStatement`, and CodeSystem/ValueSet
artifacts (the briefing notes mixing FSH-defined and JSON-authored
SearchParameters causes duplicate-canonical errors — the same general
hazard applies when removing a code from a ValueSet that is referenced
elsewhere), and the immutability of `package-list.yml` /
`FHIR-us-davinci-pdex.xml` history means the deprecation has to be
done by version, not by deletion. None of this is justified given that
the reporter asked for clarification, not removal.

---

### Recommendation

**Recommended disposition:** A

The change has already been pre-applied, it is consistent with FHIR
Bulk Data's existing `_since` semantics, and it is the smallest change
that resolves both this ticket's edge cases and FHIR-55987's
underlying objection (read-only clients should not be writing
extensions). Disposition B is technically defensible but is a much
larger reopening of a question the workgroup just closed under
FHIR-55987, and it would require new FSH artifacts and v1/v2 parity
work right at the end of the STU 2.2 cycle. Disposition C is too
aggressive — `provider-delta` may be thin after the extension removal,
but removing it outright is a breaking change that the reporter did
not ask for and that no Zulip thread has been found to support. The
workgroup should ratify the pre-applied edits to §pdex-266a and
§pdex-326a (and the matching `changehistory.md` row) and close the
ticket as **Persuasive with Modification — Applied**.
