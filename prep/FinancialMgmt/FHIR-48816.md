## Summary

The submitter (Celine Lefebvre) is concerned that the Da Vinci PAS IG —
which is heavily structured around the X12 278 Prior Authorization
transaction — is hard to follow for implementers who want a pure
FHIR-to-FHIR Prior Authorization workflow that bypasses the X12
"black box" entirely. The request is to make the FHIR-only path easier
to find and understand, suggesting a separate page (or, at minimum, a
clear separation) so the FHIR-only workflow does not overlap with the
X12-driven content.

This ticket is part of **Block Vote 22** for Da Vinci PAS, where it was
originally proposed for "Considered for Future Use" (Jean Duteau / Jeff
Brown, 10-0-0 on 2025-10-21). That resolution was reverted by Lynn
Laakso on 2026-03-26 to be re-opened post STU2.2 publication. No
explicit Jira links or other Jira cross-references were found; the only
external thread is the PAS Block Vote 22 Zulip topic, where Jean Duteau
and Hans Buitendijk explicitly debated the disposition. Industry
commenters (notably JIRAUSER16610) have pushed back, arguing that a
FHIR-only PAS guide — including appeals — is "sorely needed" because
278 adoption is low and X12 codes are insufficient.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Within the existing Da Vinci PAS IG (`HL7/davinci-pas`), add a
dedicated narrative page — e.g. `input/pages/fhir-only-workflow.md`
(or equivalent) — titled something like *"FHIR-to-FHIR Prior
Authorization (no X12 intermediary)"*. The page would:

- State explicitly that the canonical PAS workflow is FHIR ↔ X12 278
  via an intermediary, and call out which conformance requirements,
  value sets, and slicing constraints exist *because of* the 278
  mapping.
- Identify the subset of the existing PAS profiles
  (`profile-claim`, `profile-claimresponse`, the PAS Bundle profiles,
  the `$submit` / `$inquire` operations) that an implementer can use
  in a FHIR-only deployment, and call out which X12-derived
  constraints (e.g. specific code system bindings, mandatory
  identifiers, X12-specific extensions) are NOT required when no X12
  hop exists.
- Add cross-links from the IG home page / TOC, and from the existing
  workflow narrative pages, so the FHIR-only audience does not have
  to read the 278-centric pages first.

No profile, Bundle, or operation changes; this is a documentation /
navigation change scoped to the PAS IG narrative.

#### Justification

This is the smallest change that directly addresses the reporter's
concern — implementers who want FHIR-to-FHIR cannot easily extract the
relevant guidance today because it is interleaved with 278 mapping
content. The Zulip thread shows Hans Buitendijk asking specifically
whether the community has agreed that scope expansion belongs in a
separate guide; an in-IG separation page sidesteps that scope debate
because it does not change conformance — it only re-presents existing
content for a different audience. JIRAUSER16610's pushback ("a
FHIR-only guide is sorely needed") is partially answered without
committing the workgroup to a new IG project.

---

### Disposition B: Alternative Approach — Considered for Future Use, scoped to a Separate IG

#### Proposal

Mark FHIR-48816 (and any sibling tickets calling for the same thing —
the Zulip thread implies these exist) as **Considered for Future
Use**, with a resolution note that:

1. The current Da Vinci PAS IG is intentionally scoped to "FHIR →
   intermediary → X12 278" and will not be expanded to cover a
   FHIR-only Prior Authorization workflow.
2. A FHIR-only Prior Authorization workflow — including features that
   the X12 278 cannot express today (e.g. automated appeals, decision
   letters, richer attachments) — should be addressed in a new,
   separate IG, to be proposed and chartered through the normal Da
   Vinci / Financial Management process post-STU2.2.
3. Add a single, short pointer in the current PAS IG's introduction
   that explicitly states "this guide does not cover pure
   FHIR-to-FHIR Prior Authorization; see the future *(name TBD)* guide
   for that workflow," so today's reader is not misled.

#### Justification

This matches the *intent* expressed by Jean Duteau and Bob in Zulip
("if Da Vinci was going to make a FHIR-only Prior Auth guide, it would
be a complete separate guide"), and answers Hans Buitendijk's request
that the community-level scope question be visible rather than buried
in a Not Persuasive disposition. "Considered for Future Use" is also
the disposition Hans recommended in Zulip, and is consistent with
Lynn's 2026-03-26 action of re-opening the ticket post-STU2.2 — i.e.
the workgroup is not closing the door, it is deferring. It also
preserves a narrow, factual update inside the current IG (the pointer)
so the reporter's navigation concern is at least partially addressed
without growing the STU2.2 scope.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Resolve the ticket as **Not Persuasive**, with a rationale that:

- The Da Vinci PAS IG's scope is explicitly "create an X12 PAS
  transaction from a FHIR Bundle," not a FHIR-only Prior
  Authorization workflow.
- A FHIR-only workflow is a different problem (different actors,
  different content — appeals, decision letters — and different
  conformance) and would require its own IG, project scope, and
  ballot, which is outside this ticket's authority to compel.
- The current IG already documents its scope, so adding FHIR-only
  guidance inside it would confuse implementers of the X12-bound
  workflow that the IG is built around.

#### Justification

This is essentially the disposition Jean Duteau argued for in Zulip
("Not Persuasive… we were never going to add a FHIR-only part to this
guide") and is internally consistent with the project team's stated
scope. It is also the cheapest disposition for the STU2.2 timeline.
However, it does not address Hans Buitendijk's concern that the
community has not actually agreed the scope question is settled, and
it leaves implementers and external commenters (e.g. JIRAUSER16610)
without a constructive path forward — which is a known risk given the
ticket has already been re-opened once.

---

### Recommendation

**Recommended disposition:** B (Considered for Future Use, scoped to a
separate IG, with a one-line scope pointer added to the current PAS IG
intro).

Disposition B is the only option that simultaneously (a) honors the
project team's stated scope ("the current IG is FHIR→X12, full stop",
per Jean Duteau), (b) acknowledges the real implementer pain the
reporter and JIRAUSER16610 raised, and (c) matches the procedural path
Hans Buitendijk asked for in Zulip — surface the scope question rather
than burying it. It also aligns with Lynn Laakso's 2026-03-26 action
of re-opening the ticket post-STU2.2: a "future use" disposition is
exactly what "we'll deal with this after STU2.2 publication" looks
like in resolution form. Disposition A overcommits the STU2.2 IG to
new authored content and risks the very scope-creep the project team
is trying to avoid; Disposition C is defensible on scope grounds but
ignores both the in-IG navigation problem the reporter raised and the
fact that the same disposition was already overturned once. The
workgroup should adopt B and use the (eventual) separate-IG charter as
the home for the appeals / decision-letter use cases that
JIRAUSER16610 and others want.
