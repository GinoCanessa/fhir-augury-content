## Summary

FHIR-35390 asserts that the CARIN Blue Button (C4BB) code system
`C4BBAdjudicationDiscriminator` should not be defined under the IG's own
canonical namespace (`http://hl7.org/fhir/us/carin-bb/CodeSystem/...`).
Instead, the reporter (Joan Harper, with imported comment context from
Ron G. Parker) wants the canonical URL anchored in
`terminology.hl7.org` (THO), and a corresponding UTG ticket filed to
register the code system there.

Historical context matters here: the ticket was originally dispositioned
on 2022-07-26 by Financial Management as **Considered for Future Use**
(vote Corey Spears / Kathleen Conner, 13-0-0), with the rationale that
the code system was "too immature and needs use in this IG and across
the larger industry to promote to THO." That resolution was then
**reverted by Lynn Laakso on 2022-11-28** with the comment
"re-open post STU2 publication," returning the ticket to a `Triaged`
state with `Resolution: Unresolved`. So the ticket is currently open
and pending re-disposition for STU3 / a subsequent CARIN BB cycle.

Cross-references against FhirAugury surface a single Zulip mention —
the CARIN BB STU2 Block-Vote-6 announcement that batched this ticket
with 25 others. No outgoing references and no GitHub cross-references
are recorded, so no repo briefings are required. A keyword search for
`C4BBAdjudicationDiscriminator` does, however, surface several
sibling tickets that materially affect the code system's content
(FHIR-38062, FHIR-38059) and its origin (FHIR-28532) — these are
included below as discovered context, not as cross-references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Promote `C4BBAdjudicationDiscriminator` to a THO-anchored canonical:

- **New canonical:** `http://terminology.hl7.org/CodeSystem/C4BBAdjudicationDiscriminator`
  (replacing `http://hl7.org/fhir/us/carin-bb/CodeSystem/C4BBAdjudicationDiscriminator`).
- File a paired UTG ticket against `HL7/UTG` to register the code
  system content (the post-FHIR-38062 / FHIR-38059 form, including
  `adjustmentreason`, `billingnetworkstatus`, `renderingnetworkstatus`,
  etc.) under THO governance.
- In `HL7/carin-bb`, remove the local CodeSystem definition and
  retain a redirect/note pointing to the THO canonical.
- Update every C4BB ValueSet that includes this CodeSystem (notably
  the `C4BBAdjudicationCategoryDiscriminator` value set) and every
  profile that uses it as a slicing discriminator
  (`ExplanationOfBenefit.item.adjudication` and
  `ExplanationOfBenefit.adjudication` slices in the
  C4BB-ExplanationOfBenefit-* profiles) to reference the new THO URL.
- Treat as a breaking change for STU3: instance-level `system` values
  on `Coding`s drawn from this CodeSystem will need to migrate from
  the IG URL to the THO URL.

#### Justification

The reporter's underlying principle is sound and matches HL7's stated
direction: shared terminology should live in THO, not in IG-local
namespaces, so it can be referenced by other IGs (PCT, Da Vinci, etc.)
without an IG-to-IG dependency. Recent ticket activity
(FHIR-38062 "align with PCT", FHIR-38059 "align with PCT") is direct
evidence that the code set is *already* being shaped for use beyond
CARIN BB, which weakens the original 2022-07-26 "too immature"
rationale that Lynn Laakso explicitly reverted on 2022-11-28.

### Disposition B: Alternative Approach

#### Proposal

Adopt a two-step migration that is committed but deferred:

- **In the next CARIN BB version (post-STU2):** keep the existing
  IG-anchored canonical
  (`http://hl7.org/fhir/us/carin-bb/CodeSystem/C4BBAdjudicationDiscriminator`)
  in the published IG, but mark it as "candidate for THO migration"
  in the CodeSystem narrative, and freeze further code additions
  (any new codes go straight to THO).
- **File the UTG ticket now**, but for a *parallel* THO publication
  of the same code system at
  `http://terminology.hl7.org/CodeSystem/C4BBAdjudicationDiscriminator`,
  initially flagged as a draft / informative mirror.
- **In the subsequent CARIN BB major version**, retire the IG-anchored
  CodeSystem and switch all profiles, value sets, and examples to the
  THO canonical, with a documented migration note for implementers
  and a `ConceptMap` between the two `system` URLs.

#### Justification

This addresses the original WG concern (Corey Spears / Kathleen
Conner, 13-0-0: "too immature, needs use in this IG and across the
industry") *while* moving in the requested direction. Because
FHIR-38062 and FHIR-38059 reshaped the code set during STU2, an
immediate THO move under Disposition A would publish the THO version
on top of an artifact that has just changed twice — exactly the
maturity concern the original vote raised. Splitting the change over
two cycles avoids a STU2-disruptive `system`-URL flip on already-issued
EOB instances (validators are sensitive to this; cf. FHIR-32337) and
gives implementers an explicit deprecation window.

### Disposition C: Decline

#### Proposal

Re-affirm the 2022-07-26 disposition: **Considered for Future Use /
Not Persuasive at this time.** Document that the code system is a
CARIN-BB-specific slicing discriminator, that it has been reshaped
twice in STU2 (FHIR-38062, FHIR-38059), and that it is not yet
referenced by any other published HL7 IG. Close FHIR-35390 with a
note that THO promotion will be reconsidered after the code set has
been stable for one publication cycle and at least one external IG
has expressed an intent to consume it.

#### Justification

The ticket carries no new evidence beyond what was in front of the WG
on 2022-07-26; the reopen comment is procedural ("re-open post STU2
publication") rather than substantive. Breaking the canonical URL of
a code system that is actively used as a slicing discriminator in
published profiles forces every conformant payer's already-issued
`ExplanationOfBenefit` instances to be revalidated against a new
`system` URL — a cost that is hard to justify without a concrete
cross-IG consumer.

### Recommendation

**Recommended disposition:** **B**

Disposition B is the right balance for an `Highest`-priority
`Block-Vote-6` ticket whose only stated rationale ("anchor in THO")
is principled but procedurally unfunded by the current evidence base.
The 2022-07-26 vote (13-0-0) said the code system was too immature
for THO; the 2022-11-28 reopen says STU2 is now published — but it
does **not** present a new external consumer or stability argument.
At the same time, FHIR-38062 and FHIR-38059 (both Published, both
"align with PCT") show that the code system *is* being pulled toward
cross-IG relevance, which makes outright decline (C) harder to defend
in a year. A flat accept (A) flips a `system` URL on an artifact that
has been edited twice in the same ballot cycle, and would land on
implementers with no deprecation window or `ConceptMap`. B commits to
the migration, files the UTG ticket now, and times the breaking
canonical change to a major-version boundary where implementers
expect breakage. The WG can collapse B into A if a co-sponsoring IG
(e.g., PCT) commits to consuming the THO version inside the next
cycle.
