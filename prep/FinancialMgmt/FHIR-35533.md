## Summary

The reporter requests that the canonical (defining) URL for the
`PCTGFEItemNDCVS` ValueSet (US Da Vinci PCT IG) be anchored in
terminology.hl7.org (THO) rather than under `hl7.org/fhir`. The
reporter notes that, as a matter of HL7 terminology stewardship policy,
only ValueSets bound at "required" strength to a `code`-typed element
should be anchored under `hl7.fhir.org`; everything else — including
this NDC-oriented GFE item value set — belongs in THO. A companion UTG
ticket would be needed to register the value set in THO. The ticket is
one of a family of sibling tickets filed against PCT
([FHIR-35529](https://jira.hl7.org/browse/FHIR-35529),
[FHIR-35532](https://jira.hl7.org/browse/FHIR-35532),

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the canonical URL of the `PCTGFEItemNDCVS` ValueSet from
`http://hl7.org/fhir/us/davinci-pct/ValueSet/PCTGFEItemNDCVS` to a
THO-hosted URL of the form
`http://terminology.hl7.org/ValueSet/v3-PCTGFEItemNDCVS` (or the
THO-naming-convention equivalent agreed with the Vocab WG). Concretely:

1. File a UTG (`HL7/UTG`) Change Request to register the value set in
   THO, including its NDC-based composition and any required
   `CodeSystem.valueSet` linkage.
2. In `HL7/davinci-pct`, replace the value set's canonical URL with the
   THO URL, retire the local definition (or convert it into a stub
   that simply imports the THO value set), and update every
   ElementDefinition binding in the PCT profiles that references the
   old canonical so it points to the new THO canonical.
3. Capture the URL change in the PCT IG release notes and add a
   Conformance section note that the value set now lives in THO.

#### Justification

This is exactly what HL7's terminology stewardship guidance — restated
by THO at the September 2022 WGM and reflected in
[UP-316](https://jira.hl7.org/browse/UP-316) — calls for: value sets
that are not strictly bound to a `code`-typed element, and which could
be reused by other IGs (NDC GFE items are a natural cross-IG concept),
should be defined in THO rather than buried inside an IG's namespace.
Lynn Laakso's 2025-10 reopen comment explicitly states that the FM/THO
migration is now underway "in earnest," which is the precondition the
2022/2025 Considered-for-Future-Use dispositions were waiting on.
Accepting now keeps FHIR-35533 inside that coordinated migration
rather than letting it drift again.

---

### Disposition B: Alternative Approach

#### Proposal

Roll FHIR-35533 into a single coordinated **PCT THO Migration**
tracking ticket (or epic) that owns the entire family — at minimum
FHIR-35529, -35532, -35533, -35534, -35535, -35536, -35537, -35540,
-35545, -35548, -35549, -35550, -45812, -45816, -45817, -45818 —
together with the corresponding UTG registrations and a single
deprecation/redirect strategy for the old `hl7.org/fhir/us/davinci-pct`
canonicals (e.g., publishing a retired stub that points to the THO
canonical). Disposition FHIR-35533 itself as "Persuasive with
Modification — handled under <epic key>" so it is not lost, but do the
actual move in one batched PR cycle against `HL7/davinci-pct` and one
batched PR against `HL7/UTG`, sequenced after the dependencies in
UP-316 / FHIR-36942 / FHIR-36837 are settled.

#### Justification

The block-vote rationale explicitly calls out that this VS "will
require a greater level of review across multiple IGs," and the same
block vote has already shown that FM is treating these tickets as a
class, not individually. Moving each VS in its own ticket cycle risks
inconsistent THO naming, mismatched UTG registration timing, and
conformance churn for implementers who have to chase one canonical
change per PCT release. A single coordinated migration is also
materially cheaper to communicate to PCT implementers (one breaking
change, one upgrade note) and avoids the trap where some sibling
tickets get "Persuasive — Move to THO" while others get "Considered
for Future Use," which is the inconsistent state the 2025-07 block
vote left behind.

---

### Disposition C: Decline

#### Proposal

Close FHIR-35533 as **Not Persuasive** on the grounds that
`PCTGFEItemNDCVS` is in active use by published PCT STU artifacts and
the IG-anchored canonical URL is already part of multiple deployed
implementations; defer the THO question to a PCT R2 / next major-version
boundary and rely on the existing IG canonical until then.

#### Justification

Two prior dispositions have already chosen "Considered for Future Use"
on the same reasoning that the broader THO move is not yet ready, and
the value set is referenced from required/preferred bindings in
deployed PCT profiles. A canonical-URL change is a breaking change for
validators and implementers, even when the value set's logical
contents are unchanged. If the FM/THO migration referenced by
lynn_laakso is not actually executable in the current cycle, declining
(rather than re-opening to defer again) at least gives implementers a
stable answer and removes a recurring item from the PCT backlog.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying request (Disposition A) is technically
correct and aligned with current HL7 terminology stewardship guidance,
and Disposition C ignores the fact that FM has now explicitly stated
the migration is underway. But handling FHIR-35533 in isolation is
exactly what got this ticket re-opened twice already — once after STU1
and again after STU2 — because the right unit of work is the full
PCT-to-THO migration, not one value set at a time. Disposition B
preserves Joan Harper's intent, acknowledges Lynn Laakso's 2025-10
reopen and the in-flight FM/THO program, and forces a coordinated
sequencing with [UP-316](https://jira.hl7.org/browse/UP-316) and the
sibling Block-2025-Jul tickets so that PCT implementers absorb one
canonical-URL break instead of a dozen. The workgroup should pick the
epic owner and a target ballot/publication window, then dispose this
ticket as Persuasive-with-Modification under that umbrella.
