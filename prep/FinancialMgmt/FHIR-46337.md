## Summary

This ticket is a follow-on spawned from [FHIR-45872](https://jira.hl7.org/browse/FHIR-45872)
(FHIR Infrastructure, resolved "Persuasive with Modification" on
2024-06-17, now Applied). FHIR-45872 extended the Request pattern's
*Fulfillment/Execution* section with an explanation of why FHIR tracks
authorization (Request) and fulfillment (Task) in separate resources,
and agreed that a short standard sentence would be added to the
*Boundaries and Relationships* section of each concrete Request
resource pointing readers back to that discussion. FHIR-46337 is the
EnrollmentRequest-specific instance of that roll-out: it asks that
EnrollmentRequest's *Boundaries and Relationships* section be amended
to include (with wording tweaks as needed):

> "The [EnrollmentRequest] resource represents an authorization for a
> service to be provided. Details about the fulfillment of the
> authorization are handled by the [Task] resource. For further
> information about this separation of responsibilities, refer to the
> [Fulfillment/Execution] section of the Request pattern."

The ticket is assigned to Financial Mgmt because EnrollmentRequest is
owned by that work group, even though the parent design decision was
made in FHIR Infrastructure.

## Details

per https://jira.hl7.org/browse/FHIR-45872, add this to the Boundaries

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update EnrollmentRequest's *Boundaries and Relationships* section to
append the standard pointer paragraph agreed in FHIR-45872, adapted
minimally for EnrollmentRequest's actual semantics:

> "The EnrollmentRequest resource represents a request to enroll a
> subject under a coverage. Details about the fulfillment of that
> request (e.g., the enrollment workflow, acceptance, and any
> follow-up activities) are handled by the Task resource. For further
> information about this separation of responsibilities, refer to the
> [Fulfillment/Execution](workflow.html#fulfillment) section of the
> Request pattern."

Keep the link target consistent with whatever anchor FHIR-45872 landed
on in the Request pattern page, and use EnrollmentRequest's existing
`[[EnrollmentRequest]]` / `[[Task]]` markdown-style link syntax so the
publisher resolves the resource references correctly.

#### Justification

FHIR-45872 explicitly committed to filing this change against each
concrete Request resource; accepting it here fulfils that commitment
for EnrollmentRequest and keeps the Request family consistent. The
original boilerplate is already approved text ("Persuasive with
Modification", Applied), so adopting it with only a light wording
tweak minimises editorial risk. Doing so closes a known documentation
gap where implementers conflate authorization status with fulfillment
status — particularly relevant for EnrollmentRequest, which in many
payer systems is treated as carrying its own workflow state. No
normative or schema change is required; this is a descriptive edit
only.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent of the ticket but adapt the wording more aggressively
to reflect EnrollmentRequest's actual role. Rather than repeat the
generic MyRequest/Task sentence verbatim, add a paragraph that:

1. Explicitly pairs EnrollmentRequest with EnrollmentResponse (its
   direct counterpart) and clarifies that EnrollmentResponse conveys
   the payer's acknowledgement/decision, *not* the fulfillment
   workflow.
2. Then adds the FHIR-45872 boilerplate pointing to Task and the
   Request pattern's Fulfillment/Execution section for any longer-lived
   workflow tracking (e.g., waiting on supporting documentation,
   provider outreach, manual review).

This keeps the FHIR-45872 roll-out intact but avoids confusing readers
who already know EnrollmentResponse exists and may otherwise wonder
how it and Task relate.

#### Justification

EnrollmentRequest differs from most Request resources in that it has a
dedicated paired response resource (EnrollmentResponse), which the
generic boilerplate does not mention. Implementers reading only the
verbatim FHIR-45872 sentence may (reasonably) assume Task supplants
EnrollmentResponse, which is not the intent. This disposition keeps
the cross-family consistency FHIR-45872 was trying to achieve while
addressing an EnrollmentRequest-specific boundary that Disposition A
leaves implicit. Editorial cost is modest and the change remains
non-normative.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive* on the basis that EnrollmentRequest is
a Financial resource whose real-world workflow is already documented
via EnrollmentResponse and associated claims-workflow guidance, and
that pushing implementers towards Task for enrollment fulfillment is
out of step with how payers actually process enrollment today.

#### Justification

The general FHIR-45872 argument (keep authorization status separate
from fulfillment status) is weaker for EnrollmentRequest than for
MedicationRequest/ServiceRequest/etc., because most payer
implementations treat enrollment as a short-lived request/response
interaction rather than a long-running task with business status.
Adding a Task-centric pointer could mislead Financial Mgmt implementers
who have no Task infrastructure and would otherwise never consult the
workflow module.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right call. FHIR-45872 is already Applied and
explicitly earmarked follow-up change requests against each concrete
Request resource — FHIR-46337 exists to execute that plan for
EnrollmentRequest, and declining (Disposition C) would undermine the
cross-family consistency that FHIR-45872 was designed to establish. The
change is documentation-only, non-breaking, and small. Disposition B's
concern (EnrollmentResponse vs. Task) is real but can be handled
editorially by the Financial Mgmt editor when inserting the sentence
(e.g., in a surrounding paragraph) without needing a separate
disposition path; it should not block adoption of the agreed
boilerplate. Financial Mgmt should coordinate with FHIR Infrastructure
to make sure the link target and anchor in the Request pattern page
match whatever FHIR-45872 published.
