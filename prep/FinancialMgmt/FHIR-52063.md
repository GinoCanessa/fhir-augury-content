## Summary

The ticket asks Da Vinci PDex to align the Provider Access API and the
Payer-to-Payer (bulk) API with the CMS-0057-F final rule by replacing — or
at least permitting — the use of the standard Bulk Data Access STU1
`Group/$export` operation in place of the PDex-defined
`$davinci-data-export` operation (which originated in the ATR IG and is
based on Bulk Data STU2). The reporter argues that the CMS rule
(45 CFR 170.215(d)(1)) explicitly mandates Bulk Data STU1 with
"group-export" for both APIs, so requiring implementers to additionally
support the PDex-specific operation forces them to build two parallel
bulk-export surfaces for the same regulatory purpose. No directly linked
or cross-referenced Jira tickets were surfaced via FhirAugury, but the
companion PDex Zulip discussions show the same friction being raised
from multiple angles (single-member vs. bulk member match, Provider
Access attribution patterns, and the practical consequence that vendors
such as Epic intend to implement plain `Group/$export` to meet CMS while
PDex still calls for `$davinci-data-export`).

## Details

**Description:**

> The Provider Access API and Payer to Payer API (bulk) pages describe
> using the `$davinci-data-export` operation which was created in the ATR IG.
>
> The CMS final rule points towards 45 CFR 170.215(d)(1) for both APIs
> which requires "FHIR® Bulk Data Access (Flat FHIR®) (v1.0.0: STU 1),
> including mandatory support for the 'group-export' 'OperationDefinition'".
>
> Requiring implementers to support two different sets of functionalities
> for the same purpose to be compliant with the regulations and PDex
> doesn't make sense.
>
> See also: [#Da Vinci PDex > Payer-to-Provider in PDex vs. CMS requirements?](https://chat.fhir.org/#narrow/channel/235286-Da-Vinci-PDex/topic/Payer-to-Provider.20in.20PDex.20vs.2E.20CMS.20requirements.3F)

**Resolution Description (proposed by WG, not yet voted):**

> The Bulk export API has some of the input parameters as optional for the
> server. So while Davinci data export and `$export` share the same
> parameters, `$davinci-data-export` requires the parameters to be
> supported to ensure consistent implementation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the PDex Provider Access API and Payer-to-Payer (Bulk) pages so
that the normative bulk export operation is the standard
`Group/[id]/$export` defined by Bulk Data Access IG STU1, replacing the
current `Group/[id]/$davinci-data-export` requirement. Specifically:

- On `payertopayerbulkexchange.html` ("`$davinci-data-export` operation"
  section) and on the Provider Access API page, change the SHALL on
  `$davinci-data-export` to SHALL `Group/[id]/$export` per Bulk Data
  STU1, and mark `$davinci-data-export` as MAY for backward
  compatibility (or deprecate it).
- Move the input parameters that PDex actually needs beyond STU1
  (e.g., POST kickoff support, `_typeFilter`, any PDex-specific
  filters such as `exportType`) into either (a) a thin extension/profile
  on the standard `$export` Parameters resource, or (b) IG-level
  conformance language that names which STU1-optional parameters PDex
  servers SHALL support — without forking the operation name.
- Re-point any conformance statements, CapabilityStatements, and
  examples to `Group/[id]/$export`.

#### Justification

This is the straightforward way to remove the dual-implementation
burden that the reporter (and downstream implementers in the related
Zulip threads, including Epic) are calling out. CMS-0057-F explicitly
names `group-export` from Bulk Data STU1 at 45 CFR 170.215(d)(1); under
the present text a PDex-conformant server is not automatically
CMS-conformant, and a CMS-conformant server is not automatically
PDex-conformant. Aligning on the standard operation name keeps PDex
inside the regulatory floor while still allowing PDex to add
conformance constraints (mandatory parameters, POST kickoff, consent
handling) on top — which directly addresses the WG's stated concern
in the proposed "Not Persuasive" resolution that some STU1 parameters
are server-optional and PDex needs them mandatory.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `$davinci-data-export` defined in the IG, but make it a *profile of*
the standard `Group/[id]/$export` rather than a sibling operation:

- Re-spec `$davinci-data-export` so that its `code` is `export` (i.e.,
  the operation is invoked as `Group/[id]/$export`) and the
  `OperationDefinition` becomes a constrained derivation of the Bulk
  Data STU1 export operation — adding the SHALL-level parameter
  support, POST kickoff, and any PDex-specific input filter
  (`exportType`, etc.) as additional constraints/extensions.
- Document explicitly that a PDex-conformant `Group/[id]/$export`
  endpoint also satisfies the CMS-0057-F `group-export` mandate, and
  that a generic STU1 client can call the same endpoint and ignore
  the PDex-specific parameters.
- Retain `$davinci-data-export` only as a documentation alias /
  historical name; remove it from CapabilityStatements going forward.

#### Justification

This preserves the WG's stated rationale for the current proposed
"Not Persuasive" resolution — that PDex needs to *guarantee* support
for parameters that STU1 leaves optional — while still letting a
single endpoint satisfy both PDex and CMS. It avoids a hard breaking
change for servers that have already shipped `$davinci-data-export`
and reframes the PDex requirement as a profile/constraint rather than
a parallel operation, which is the more idiomatic FHIR pattern.
Trade-off vs. Disposition A: it is a smaller editorial change and
keeps the historical name visible, but it requires care so that the
single endpoint can be discovered as both "standard `$export`" and
"PDex export" (e.g., via two `OperationDefinition` references in the
CapabilityStatement, both pointing at `Group/$export`).

---

### Disposition C: Decline (current proposed resolution)

#### Proposal

Resolve the change request as **Not Persuasive** and leave the IG as-is:
the Provider Access API and Payer-to-Payer Bulk page continue to require
`Group/[id]/$davinci-data-export`. The IG would add an explanatory note
that `$davinci-data-export` shares its parameter set with Bulk Data
STU1's `Group/$export` but tightens server obligations (the parameters
PDex needs are SHALL-supported instead of MAY-supported), which is why a
PDex-specific operation name is retained.

#### Justification

This is the disposition presently drafted on the ticket. Its defensible
core is real: STU1 leaves several useful parameters server-optional, so
a "plain" `Group/$export` server can technically be conformant without
supporting the inputs PDex relies on. Defining a distinct operation
gives the WG an unambiguous handle on which servers are *PDex-grade*
versus generically Bulk-Data-conformant. The downside — and the reason
the workgroup will likely hear pushback — is that it leaves the
regulatory misalignment in place: a PDex-only server still does not, on
the letter of CMS-0057-F (which names `group-export`), satisfy the
mandate, forcing implementers to expose two operations (or to ignore
PDex and ship plain `$export`, which is what at least one major vendor
in the related Zulip threads says it is doing).

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern is well-founded and is being
independently confirmed by implementers in the linked Zulip threads:
the cleanest way for a single endpoint to satisfy both CMS-0057-F and
PDex is to invoke `Group/[id]/$export`. At the same time, the
workgroup's drafted "Not Persuasive" rationale has a real point —
PDex genuinely needs guaranteed support for parameters that STU1
makes optional, and that is best expressed as a constraint, not as an
informal expectation. Disposition B captures both: it reframes
`$davinci-data-export` as a *profile of* the standard operation
(invoked as `$export`) with PDex-mandatory parameters layered on top,
which preserves the WG's conformance lever while letting one endpoint
serve both regulatory and PDex clients. Disposition A is a reasonable
fallback if the WG decides the historical operation name is not worth
preserving; Disposition C should only stand if the WG is prepared to
accept that "PDex-conformant" and "CMS-0057-conformant" remain
non-overlapping sets — a position the related Zulip discussions
suggest implementers will not silently absorb.
