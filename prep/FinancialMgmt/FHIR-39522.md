## Summary

The ticket asks the Da Vinci ATR (Authorization, Treatment, Referral) IG to
reconcile its `atr-coverage` profile with the `hrex-coverage` profile defined
by Da Vinci HRex. Today `atr-coverage` is an independent constraint on
`Coverage` that does not derive from `hrex-coverage`, which means implementers
who already conform to HRex must satisfy a parallel set of constraints when
exchanging coverage as part of an ATR transaction. The reporter requests that
the two profiles be brought into a derivation relationship (most naturally,
`atr-coverage` deriving from `hrex-coverage`) so the IGs share a single
coverage contract. The ticket was originally filed pre-STU2, deferred as
"Considered for Future Use" by a 10-0-0 vote on 2023-10-03 pending HREx
updates and analysis, and was re-opened post STU2 publication on 2024-01-09 by
Lynn Laakso so the work group can revisit it now that newer HRex versions are
available. No related Jira tickets, Zulip threads, or GitHub items were
surfaced by Augury for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the Da Vinci ATR IG, change the `atr-coverage` `StructureDefinition` so its
`baseDefinition` is the canonical of the current HRex `hrex-coverage` profile
(rather than `http://hl7.org/fhir/StructureDefinition/Coverage`). Re-express
the existing ATR-specific constraints as differential elements layered on top
of `hrex-coverage`, dropping any constraint that is already enforced by HRex
(or tightening it via cardinality / binding only where ATR genuinely needs
more). Pin the dependency on `hl7.fhir.us.davinci-hrex` to a specific
published version in the ATR `package.json` / IG dependencies, regenerate
snapshots, and update the ATR coverage page narrative and any references
(profile invariants, examples that conform to `atr-coverage`, search
parameters) accordingly.

#### Justification

This is what the ticket literally asks for and what implementers expect: ATR
participants overwhelmingly already implement HRex (member-match, consent,
coverage), so making `atr-coverage` derive from `hrex-coverage` removes a
parallel conformance surface and prevents drift between the two IGs. It is
consistent with how other Da Vinci IGs (PDex, CRD, DTR) layer on top of HRex
profiles rather than re-profiling base R4 resources. The ballot vote that
deferred this ticket explicitly conditioned acceptance on "HRex updates and
analysis"; with STU2 published and HRex revisions available, the prerequisite
is now satisfied.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of changing the `baseDefinition` of `atr-coverage`, keep it deriving
from base `Coverage` but bring it into structural alignment with
`hrex-coverage` element-by-element (matching cardinalities, must-supports,
slicing, and bindings) and add an explicit `compliesWith` /
`StructureDefinition.derivation` cross-reference plus IG narrative stating
that any instance valid against `atr-coverage` is also valid against
`hrex-coverage`. Add a CI validation step (or test instances) that asserts
ATR coverage examples validate against `hrex-coverage` to guarantee the two
stay aligned over time.

#### Justification

This achieves the practical goal — implementers can satisfy one and get the
other for free — without forcing a hard cross-IG dependency from ATR onto a
specific HRex version. That avoids the version-pinning churn that derivation
introduces (every HRex publication requires an ATR republish to absorb the
new snapshot), and it sidesteps any cases where HRex tightens a constraint in
a way ATR does not yet want to follow. Trade-off vs. A: alignment is
guaranteed only by tooling/process rather than by the type system, so it is
easier for the two to drift if either IG changes without the other noticing.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — keep `atr-coverage` independent of
`hrex-coverage`," documenting that ATR's coverage constraints are scoped to
the ATR transaction (provider attribution / treatment / referral) and that
HRex's coverage profile is scoped to member-match and payer-to-payer
exchange, so the two profiles are intentionally separate even where their
element constraints overlap.

#### Justification

This is defensible only if the work group concludes that the use cases are
genuinely distinct and that derivation would over-constrain ATR coverage
exchanges (for example, by inheriting HRex must-supports that ATR
implementers cannot reasonably populate). The reporter and the 2023 vote
explicitly anticipated revisiting this once HRex updates landed, so a
decline now would need to point to a specific incompatibility uncovered by
that analysis; absent such a finding, decline is the weakest of the three
options.

---

### Recommendation

**Recommended disposition:** A

The ticket has been pending exactly this analysis since 2023, the original
vote was a deferral rather than a rejection, and the conditioning event
("HREX updates and analysis") has occurred with STU2 published. Deriving
`atr-coverage` from `hrex-coverage` is the cleanest way to give ATR
implementers a single coverage contract and matches the layering pattern
the rest of the Da Vinci stack already uses. Disposition B is a reasonable
fallback if the work group finds, during the actual reconciliation work,
that a specific HRex constraint is incompatible with an ATR use case — in
that case the IG can switch to "aligned but not derived" without losing
much. Disposition C should only be selected if such an incompatibility is
identified and documented; on the current record there is no evidence for
it.
