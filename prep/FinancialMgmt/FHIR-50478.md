## Summary

The reporter asks Financial Management to revise the Financial Module
overview page (and the resource intro sections it points at) so that the
relationships between Coverage, InsurancePlan, and the newly-added
InsuranceProduct resource are made explicit. Two concrete gaps are
called out: (1) the page's index/landing section omits InsurancePlan,
InsuranceProduct, and "all other Financial focus resources," so a reader
landing on the module page cannot discover them; and (2) InsuranceProduct
is a new resource that is not introduced or defined anywhere on the
module page, and InsurancePlan is likewise unexplained, leaving readers
without a clear narrative for what each resource is for, how they
differ, and how they reference one another. The reporter explicitly asks
for an updated diagram alongside the prose. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were surfaced for this ticket.

## Details

**Description:**

- InsurancePlan, InsuranceProduct, and all other Financial focus resources should be added to the index section
- The InsuranceProduct resource is new and is never explained on the page (nor is insurancePlan). There should be clear definitions of these resources and their relationships (along with an updated diagram). It is not clear on this page what the relationship between Coverage, InsurancePlan and InsuranceProduct is, or how to reference between them.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revise the Financial Module overview page to fully address both gaps:

1. **Index / landing section** — Add InsurancePlan, InsuranceProduct,
   and every other resource currently scoped to the Financial
   Management workgroup (e.g., Coverage, CoverageEligibilityRequest,
   CoverageEligibilityResponse, EnrollmentRequest, EnrollmentResponse,
   Claim, ClaimResponse, Invoice, ChargeItem, ChargeItemDefinition,
   Account, PaymentNotice, PaymentReconciliation, ExplanationOfBenefit)
   to the module's resource index so that all FM resources are
   discoverable from the module landing page.
2. **Definitions** — Add a short, normative-tone definition block for
   InsurancePlan and InsuranceProduct on the module page itself
   (mirroring the resource's scope statement), making it clear that
   InsuranceProduct describes an offered insurance product/SKU and
   InsurancePlan describes a specific plan that can be subscribed to,
   while Coverage describes an individual's enrollment/benefit
   instance.
3. **Relationships and references** — Add an explicit "How these
   resources relate" subsection that walks the
   InsuranceProduct → InsurancePlan → Coverage chain, naming the
   reference elements actually used to link them
   (e.g., `Coverage.insurancePlan`, the InsurancePlan / InsuranceProduct
   linkages), and clarifying which side owns the reference.
4. **Diagram** — Replace the existing module diagram with one that
   includes InsurancePlan and InsuranceProduct as first-class boxes,
   draws the reference arrows between Coverage, InsurancePlan, and
   InsuranceProduct, and is consistent with the resource list in the
   refreshed index.
5. **Resource intro sections** — In the intro/scope sections of
   Coverage, InsurancePlan, and InsuranceProduct, add a short
   "Boundaries and Relationships" paragraph that points at the other
   two resources (and back to the module page) so the relationship is
   discoverable from any of the three landing pages.

#### Justification

The request is purely documentation and module-navigation work; it
introduces no normative changes and no breaking changes. The reporter
has named two specific, verifiable defects (missing index entries and
missing definitions for a newly-published resource), and FHIR module
overview pages are exactly the place readers expect this material to
live. Keeping the module page, the diagram, and the per-resource intros
mutually consistent is also routine spec-maintenance hygiene that the
workgroup is already responsible for. The cost is bounded (one HTML
page plus three resource intro snippets and one diagram), and the
benefit — implementers being able to find and disambiguate
InsurancePlan vs. InsuranceProduct vs. Coverage — directly supports
correct adoption of the newly-added InsuranceProduct resource.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same documentation goals as Disposition A, but split the work
and reduce the surface area of the module-page rewrite:

1. **Index fix only on the module page.** Update the Financial Module
   page to add the missing resources to the index and refresh the
   diagram so InsurancePlan and InsuranceProduct appear, but keep the
   module page's prose changes minimal (one paragraph naming the three
   resources and pointing to their resource pages).
2. **Push the detailed relationship narrative into the InsurancePlan
   and InsuranceProduct resource pages**, not the module page, by
   expanding their "Scope and Usage" / "Boundaries and Relationships"
   sections to carry the authoritative description of how Coverage,
   InsurancePlan, and InsuranceProduct relate, with a reciprocal link
   from Coverage's intro.
3. **Track the diagram update as a follow-up** via a short companion
   ticket so the module-page edit is not blocked on a re-render of the
   diagram asset.

#### Justification

Most readers reach the relationship discussion from a resource page,
not from the module overview, so investing the authoritative prose in
the resource intros gives it more reach and keeps the module page
serving its primary role as a directory. Splitting the diagram into a
follow-up de-risks the textual fix from getting stalled on a
visual-asset change. The trade-off vs. Disposition A is that the module
page itself stays thinner — readers landing there get pointers rather
than a self-contained explanation — which is acceptable if the resource
pages are kept in sync.

---

### Disposition C: Decline

#### Proposal

Decline as a standalone change request on the grounds that the gap is
already addressed by the InsurancePlan and InsuranceProduct resource
pages themselves (which carry their own scope statements and
boundaries sections), and that module overview pages are intentionally
kept lightweight. Close the ticket as "Not Persuasive" and direct
readers to the resource pages.

#### Justification

A defensible decline rests on the position that the FHIR Core
specification deliberately puts authoritative resource definitions on
the resource pages, not on module overview pages, and that the cost of
keeping a hand-maintained index/diagram in sync with every new resource
is high relative to the navigation already provided by the resource
list and search. This is a weak rationale here, however, because the
reporter is pointing at a real, demonstrable defect on a page the
workgroup owns; declining would leave InsuranceProduct effectively
invisible from the module landing page despite being a newly-published
resource.

---

### Recommendation

**Recommended disposition:** A

The ticket describes concrete, verifiable documentation defects on a
page the Financial Management workgroup owns, and the fix is bounded:
it is one overview page, three resource intro sections, and one
diagram, with no normative impact and no migration cost for
implementers. Disposition B is reasonable but trades discoverability
for editorial economy at exactly the moment InsuranceProduct is being
introduced — which is the worst time to leave the module landing page
silent about it. Disposition C is hard to defend because the reporter
has named specific, current defects that the workgroup is in the best
position to fix. Accepting as requested, and treating the diagram
refresh as part of the same change rather than a follow-up, gives
implementers a single, coherent landing experience for the financial
resources.
