## Summary

The ticket asks the Financial Management work group to reconsider the
required value-set binding on `Location.address.type` in the **US Da
Vinci CRD** profile, which currently restricts the element to `physical`
and `both` (excluding `postal`) — a tighter constraint than the
underlying FHIR US Core Location profile. The reporter accepts that
disallowing `postal` is functionally correct *within the ePA scenario*,
but argues that real-world FHIR servers will hold many `Location`
resources that do not qualify for ePA (and therefore may legitimately
carry an address type of `postal`). Because the same server is used for
configuration, documentation, and operational traffic, it is impractical
to guarantee that `postal` will never appear on a CRD-side request, and
test data may differ from operational data.

The reporter requests that the workgroup acknowledge this operational
reality — for `Location.address.type` and, by extension, for **all**
places where CRD further constrains a US Core profile — and provide
guidance allowing payers to respond with a "maybe" CDS Hooks card with
clarification, so end-users can either fix the data and re-issue the CRD
request or proceed to DTR. The only cross-reference surfaced is to the
FHIR element `Location.address.type` itself; no related Jira tickets,
Zulip threads, GitHub items, or Confluence pages were found in the
indexed sources.

## Details

**Description:**

> The CRD `Location.address.type` has a required binding to a restricted
> value set compared to FHIR US Core: `physical` and `both`, with
> `postal` not allowed. Functionally the makes sense in the context of
> ePA.
>
> However, fully ensuring that no locations would have an address type
> of "postal" in this flow is quite a challenge as the overall FHIR
> server would include Locations that do not qualify for ePA yet are
> valid elsewhere, and during configuration/documentation may still be
> used.
>
> During testing test data would/should not yield use of Locations, but
> operational transactions still may. Payers should be able to respond
> in those situations with a "maybe" with clarification on the card so
> the user can opt to progress to DTR or fix the problem and re-issue
> the CRD transaction. We suggest that such behavioral considerations
> are given to all places where the FHIR US Core profiles are further
> constrained for ePA, thus operational flows may result in sharing
> incorrect data such as a location type of "postal".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Keep the existing required binding on
`Location.address.type` in the CRD Location profile (values `physical`
and `both`) but add **conformance narrative** to the CRD IG (in
`HL7/davinci-crd`, IG narrative for the CRD Location profile / the
"CRD Workflow" or "Conformance" page) that:

1. Acknowledges that operational FHIR servers will carry Locations with
   `address.type = postal` that are valid for non-CRD use.
2. Defines a normative behavioural pattern for CRD service responses
   when an inbound CRD request references a Location that fails the
   profile (including `address.type = postal`): the payer **SHOULD**
   respond with a CDS Hooks "card" of indicator `warning` (the existing
   "maybe" pattern), carrying a human-readable explanation and a
   `suggestion` to either (a) correct the Location and re-submit CRD or
   (b) proceed to DTR with the understanding that downstream
   prior-auth processing may also fail.
3. Generalises the same pattern to **every** element where CRD tightens
   a US Core profile (cross-referencing each constrained element so the
   guidance is discoverable, not buried under Location).

This is purely a narrative + conformance-language change; no
StructureDefinition or ValueSet would be modified.

#### Justification

This disposition treats the constraint as correct *for the data CRD
needs to operate on* while addressing the reporter's real concern —
that today's IG offers no guidance for what a payer should *do* when
incoming data is non-conformant. The CDS Hooks "warning" card is
already the established CRD mechanism for "we can't decide; here's
what's wrong" responses, so this leverages an existing pattern rather
than inventing new conformance machinery. It also responds to the
reporter's explicit ask that the same consideration be given to *all*
US-Core-tightening constraints, not just `Location.address.type`,
which a single-element fix would miss.

---

### Disposition B: Alternative Approach — Relax the Binding to `extensible` or `preferred`

#### Proposal

In the CRD Location profile (in `HL7/davinci-crd`), change the binding
strength on `Location.address.type` from **required** to
**extensible** (or **preferred**), keeping the same value set
(`physical`, `both`) as the recommended set. Pair this with a Must
Support flag and a profile invariant (warning, not error) that fires
when `address.type = postal`, e.g.:

- `crd-loc-1: Location.address.type SHOULD be 'physical' or 'both';
  'postal' addresses are not actionable for ePA workflows`

Optionally also add the same narrative described in Disposition A so
payer responses can still leverage the CDS Hooks "warning" pattern.

#### Justification

Relaxing the binding to `extensible`/`preferred` codifies the
operational reality (servers *will* carry `postal` Locations) directly
in the conformance artifact, so a CRD instance with `address.type =
postal` becomes *technically conformant* while still flagged as
non-ideal via a warning invariant. This avoids the awkward situation
where well-formed real-world data fails strict profile validation,
which is what the ticket is reacting to. The trade-off is that some
payers may treat a relaxed binding as licence to ignore the constraint
entirely; a SHOULD-level invariant mitigates that. It is a heavier
change than Disposition A — it touches the StructureDefinition and the
binding strength, which has downstream effects on validators,
conformance test kits, and any RI that pins to a `required` binding —
and would warrant a substantive ballot comment and prior signal to
implementers.

---

### Disposition C: Decline

#### Proposal

Decline the change. Keep the required binding on
`Location.address.type` in the CRD Location profile as-is and do not add
new conformance narrative. The position is that the existing US Core /
FHIR base specification's CDS Hooks guidance, plus the general
"non-conformant input → reject or warn" expectation that already
applies to every profile, is sufficient. Implementers who need to
return a "maybe" today can already do so using a CDS Hooks card with
indicator `warning`; nothing in CRD prevents that.

#### Justification

Declining is defensible if the workgroup's view is that the CRD profile
should describe *the data CRD operates on*, and that the question of
what a payer does with non-conformant input is an
implementation/runtime concern outside the scope of the IG. The
existing CDS Hooks specification already permits warning cards with
explanatory text, and CRD already references CDS Hooks normatively, so
no new mechanism is strictly needed. The risk of declining is that the
IG remains silent on a recurring real-world question (what to do when
operational data fails a CRD-tightened constraint) for *every*
US-Core-tightening element in CRD, not just this one, which is exactly
what the reporter flagged.

---

### Recommendation

**Recommended disposition:** **A**

Disposition A is the lowest-risk way to address the reporter's actual
concern. The reporter is not arguing that `postal` should be valid for
ePA — they explicitly say the constraint "functionally makes sense" —
they are arguing that the IG offers no guidance for the (predictable)
case where operational data violates the constraint, and they want
that guidance applied uniformly to every CRD-tightening of US Core.
Disposition A delivers exactly that: it adds normative behavioural
narrative without disturbing the StructureDefinition, the binding
strength, validators, or the CRD reference implementation, and it
explicitly generalises the pattern beyond `Location.address.type`.
Disposition B (relax the binding) addresses the same concern but
introduces conformance-surface churn that risks ballot pushback and
implementer fragmentation — it is the right answer only if the
workgroup believes operational `postal` Locations are common enough
that they should be technically conformant. Disposition C leaves a
known gap in the IG; while the gap is arguably an "implementation
concern", the reporter's cross-cutting framing (apply this to *all*
tightened constraints) suggests the gap is wide enough that the IG
should speak to it.
