## Summary

This ticket asks Financial Management (in coordination with Patient
Administration) to harmonize the multiple `type` elements that span the
`Coverage` and `InsurancePlan` resources (and, in R6, `InsuranceProduct`).
Today there are several overlapping but inconsistently-defined slots:
`Coverage.type` (0..1, bound to `valueset-coverage-type`),
`InsurancePlan.type` (0..*, bound to `valueset-insuranceplan-type`),
`InsurancePlan.coverage.type` (no binding), `InsurancePlan.coverage.benefit.type`
(no binding) and `InsurancePlan.plan.type` (no binding). The reporter notes
inconsistent cardinality, inconsistent value-set bindings, and unclear semantic
boundaries between elements that all answer "what kind of insurance is this?".

The reporter recommends (a) widening `Coverage.type` to 0..* (tracked in a
separate FM ticket) and (b) aligning `InsurancePlan.type` and
`InsurancePlan.coverage.type` against the same value set, while clarifying why
both elements are needed. The remaining sub-type elements
(`InsurancePlan.coverage.benefit.type`, `InsurancePlan.plan.type`) need value
sets defined, even if only at example strength.

The closely related (and now duplicate) ticket
[FHIR-50357](https://jira.hl7.org/browse/FHIR-50357), filed by Ricardo Quintano,
re-raised the missing bindings on `InsuranceProduct.coverage.type` and
`InsuranceProduct.coverage.benefit.type` after the InsurancePlan→InsuranceProduct
split (FHIR-24827). That ticket was closed as a duplicate of this one and its
proposed resolution — adopt the THO `insurance-coverage-type` value set already
used by the US National Directory IG (NDH), at binding strength **Example** —
has been folded into this ticket's draft resolution.

The current draft resolution (still flagged TODO by FM) proposes:

| Property | ValueSet | Strength |
|---|---|---|
| `Coverage.type` | `http://terminology.hl7.org/ValueSet/insurance-coverage-type` | Preferred |
| `InsurancePlan.type` | `http://terminology.hl7.org/ValueSet/insuranceplan-plan-type` | Preferred |
| `InsuranceProduct.type` | `http://terminology.hl7.org/ValueSet/insurance-coverage-type` (TODO confirm) | Example |
| `InsuranceProduct.coverage.type` | `http://terminology.hl7.org/ValueSet/insurance-coverage-type` (TODO confirm) | Example |
| `InsuranceProduct.benefit.type` | TODO — value set to be selected | Example |

Open questions remain about whether `InsuranceProduct.type` and
`InsuranceProduct.coverage.type` should truly use the same value set (and if so,
how the elements are differentiated semantically), and what value set should
back `InsuranceProduct.benefit.type`.

## Details

**Description:**

> Variances to be harmonized: value set bindings, cardinality, etc.
>
> For example:
> * `Coverage.type` is 0..1 and binds to `valueset-coverage-type` (e.g. HMO, PPO, dental, LTC, vision)
> * `InsurancePlan.type` (product type) is 0..* and binds to `valueset-insuranceplan-type` (e.g. medical, dental)
> * `InsurancePlan.coverage.type` (one type per coverage) and has no binding, but definition implies medical, dental, LTC. Why is this different than `InsurancePlan.type`?
>
> Recommendation is to:
> * Change `Coverage.type` to 0..* (will log another FM request)
> * Change `InsurancePlan.type` and/or `InsurancePlan.coverage.type` to bind to the same value set (`valueset-coverage-type`) as well as differentiate between why both of these types are needed?
>
> --------------------------------------------------------
>
> These are additional types, but don't seem to have as much overlap as the aforementioned types. They do need value set bindings though.
> * `InsurancePlan.coverage.benefit.type` have no bindings, but definition implies primary care; specialty care; inpatient; outpatient.
> * `InsurancePlan.plan.type` is 0..1 and has no binding, but definition implies "Platinum" or "High Deductible".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the draft resolution table substantially as drafted in the ticket,
resolving the open TODOs as follows so the change can be implemented in the
R6 `Coverage`/`InsuranceProduct` source:

| Property | ValueSet | Strength |
|---|---|---|
| `Coverage.type` | `http://terminology.hl7.org/ValueSet/insurance-coverage-type` | Preferred |
| `InsurancePlan.type` (R5 / pre-split) | `http://terminology.hl7.org/ValueSet/insuranceplan-plan-type` | Preferred |
| `InsuranceProduct.type` | `http://terminology.hl7.org/ValueSet/insuranceplan-plan-type` | Example |
| `InsuranceProduct.coverage.type` | `http://terminology.hl7.org/ValueSet/insurance-coverage-type` | Example |
| `InsuranceProduct.benefit.type` | (new) `http://terminology.hl7.org/ValueSet/insurance-plan-benefit-type` derived from existing definition examples (primary care, specialty care, inpatient, outpatient) | Example |

In addition, change `Coverage.type` cardinality from 0..1 to 0..* (tracking
the companion FM tracker the reporter mentioned), add the same
`insurance-coverage-type` binding to `InsuranceProduct.coverage.type`, and
add definitional text on `InsuranceProduct.type` vs.
`InsuranceProduct.coverage.type` clarifying that **type** classifies the
overall product family (medical/dental/vision) while **coverage.type**
classifies an individual coverage line item under the product.

#### Justification

This is the harmonization the ticket asks for and is consistent with the
duplicate-closed FHIR-50357 disposition. The THO `insurance-coverage-type`
value set is already published at THO 6.2.0 and is the value set the US
National Directory IG (NDH) settled on for the same purpose, so adopting it
here aligns FM core with an existing implementer community rather than
inventing a new code list. Using **Preferred** for `Coverage.type` (the
primary instance-level slot) and **Example** for the `InsuranceProduct.*`
type slots reflects FM's repeated input on Zulip / WGM that the value sets
are still maturing and that NDH-style refinement will likely continue.
Widening `Coverage.type` to 0..* matches real-world plans where a single
coverage instance bundles, e.g., medical + dental + vision, which is the
underlying need that motivated the original ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-binding `InsurancePlan.type` and `InsuranceProduct.type` to
the same value set as the per-coverage element, formally split the semantics
in the spec text and bind each element to a *different*, narrower value
set:

- Keep `Coverage.type` at the existing `valueset-coverage-type`
  (HMO/PPO/dental/etc.) and only widen its cardinality to 0..*.
- Bind `InsuranceProduct.type` to a new, deliberately small "product
  family" value set (medical, dental, vision, LTC, pharmacy, behavioral
  health) at **Preferred** strength.
- Bind `InsuranceProduct.coverage.type` to the THO
  `insurance-coverage-type` value set at **Example** strength.
- Add a new value set for `InsuranceProduct.benefit.type` at **Example**
  strength (primary care, specialty care, inpatient, outpatient,
  pharmacy, mental health, etc.).
- Bind `InsuranceProduct.plan.type` (formerly `InsurancePlan.plan.type`)
  to a marketplace-tier value set (Platinum, Gold, Silver, Bronze, High
  Deductible, Catastrophic) at **Example** strength.

Spell out in the resource narrative that **product type** = "what kind of
insurance product the payer is selling", **coverage type** = "what kind of
benefit-line a member actually has under that product", **benefit type** =
"what category of service is being described for a single benefit", and
**plan type** = "marketplace tier / actuarial level".

#### Justification

The reporter's complaint isn't only about missing bindings — it's that the
elements appear redundant ("Why is this different than InsurancePlan.type?").
Binding two elements to the *same* value set, as Disposition A does for
`InsuranceProduct.type` and `InsuranceProduct.coverage.type` per the draft's
TODO, will perpetuate that confusion, not resolve it: implementers will
populate the same code in both places. Giving each element a distinct,
purpose-built value set forces the spec to articulate the semantic
distinction in a way the binding itself reinforces. It is more work — three
new (or refined) value sets — but FM has been actively curating coverage
terminology for two years (per Cooper Thompson's 2023/2024 WGM updates), so
the curation cost is largely sunk. It also leaves room for NDH and other IGs
to constrain to their own preferred value sets without having to fight the
core spec's overlap.

---

### Disposition C: Decline

#### Proposal

Decline the harmonization and instead resolve the ticket as **Not
Persuasive** on the grounds that (a) the InsurancePlan→InsuranceProduct
split (FHIR-24827) has already restructured this part of the model, (b)
implementer-specific value sets such as the NDH `insurance-coverage-type`
are better handled by IG-level constraints than by core re-bindings, and
(c) widening `Coverage.type` to 0..* is a non-trivial breaking change for
existing R4/R5 implementers who have built around the 0..1 cardinality.
Document in the spec narrative the recommended IG pattern (constrain
`Coverage.type` to a chosen code system, repeat `Coverage` instances when a
member has multiple coverage families) and close the ticket.

#### Justification

The ticket has been open for six years across multiple WGMs without FM
locking down a definitive value set, which suggests there isn't a single
right harmonized answer at the core level. The R5→R6 resource split already
addresses part of the original concern by separating the
plan/product/coverage concepts more clearly. Forcing a Preferred binding on
`Coverage.type` could break implementers who today populate codes outside
the THO `insurance-coverage-type` list, and a cardinality widening
(`0..1` → `0..*`) is a backward-incompatible change to a stable resource
element that FHIR maturity rules generally avoid post-normative-track.

---

### Recommendation

**Recommended disposition:** B

Disposition A is essentially what the ticket's own draft resolution
proposes, but it leaves two of the three TODOs unresolved precisely because
it tries to use the same value set for `InsuranceProduct.type` and
`InsuranceProduct.coverage.type`. That equivalence is what made the original
reporter file the ticket in the first place ("Why is this different than
InsurancePlan.type?"), so adopting it would close the ticket without
answering its core question and would invite a follow-up ticket the moment
implementers start populating the two elements identically.

Disposition C is too pessimistic given that FM has actually done the
terminology curation work over the last two years (Cooper Thompson's WGM
updates) and that an externally-validated value set already exists in THO
via the NDH IG. The 0..* cardinality widening is a real concern, but the
companion FM ticket the reporter mentioned in the original description is
the right venue for that debate, not this one.

Disposition B keeps the spirit of the ticket's draft resolution — adopt the
THO `insurance-coverage-type` value set rather than invent something new —
while honoring the harmonization request by giving each element a distinct
binding that reinforces a distinct meaning. It also positions FM cleanly
for the joint PA/FM call Brian Postlethwaite asked for in the January 2025
Zulip thread, because the four binding decisions (product type, coverage
type, benefit type, plan/tier type) map directly onto questions PA cares
about. The workgroup should treat the per-element value-set choices in the
table as the discussion items, but the recommendation is to adopt the
"distinct value set per element" pattern as the framing.
