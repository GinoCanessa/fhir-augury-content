## Summary

The ticket requests a correction to the human-readable display text bound
to NCI Thesaurus code **C133936** in the PQ/CMC (FHIR) specification.
The current display string contains an erroneous lower bound ("20 degrees
C (+/") that conflicts with the PQ/CMC Preferred Term (PT). The reporter
asks that the display value be changed to use the PQ/CMC PT, which (with
the erroneous portion struck through in the original ticket) reads
**"Storage at 5 degrees C."** This is a presentation-only correction to a
value-set/code-system display; it does not change the code itself. No
related Jira tickets, Zulip threads, or GitHub items were surfaced by
the cross-reference index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `display` for NCI Thesaurus concept code **C133936** in the
PQ/CMC FHIR IG so that it reads **"Storage at 5 degrees C."** wherever
it appears as an authored display in the IG's CodeSystem / ValueSet
content (e.g., the storage-condition / stability-related code system
fragment and any expansion that PQ/CMC ships with the IG). No change to
the code itself, no change to the binding strength, no change to the
underlying NCIt definition — purely a display string correction to
match the PQ/CMC Preferred Term and remove the erroneous "20 degrees C
(+/" prefix from the existing display.

#### Justification

This is a documented mismatch between the IG's published display and
the PQ/CMC PT for the same concept. Using the PT verbatim is the
established convention for PQ/CMC terminology bindings and prevents
implementer confusion (the current string implies a temperature range
that the PT does not). The change is non-breaking: codes are unchanged,
expansions remain valid, and any system that already binds against the
code (rather than the display) is unaffected. This is exactly the kind
of low-risk erratum that block votes are designed to clear.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring a corrected display string in the PQ/CMC IG,
remove the IG-local override of the display for C133936 entirely and
let the published NCI Thesaurus PT flow through the standard
CodeSystem-supplement / expansion mechanism. If PQ/CMC continues to
ship a snapshot/expansion of the storage-conditions value set, refresh
that snapshot from the current NCIt PT rather than maintaining a
hand-edited display. Document in the IG's terminology section that
display strings for C-codes are taken from the current NCIt PT.

#### Justification

The repeated re-opens of this ticket suggest that hand-maintained
display strings drift from the authoritative NCIt PT over time. Sourcing
the display from NCIt eliminates this entire class of erratum and
removes the per-code authoring burden going forward. Trade-off vs.
Disposition A: it is more invasive than a one-line display fix and may
require a tooling/build change in the IG, but it pays off across the
storage-conditions and stability value sets that PQ/CMC will continue
to expand into Stage 3.

---

### Disposition C: Decline

#### Proposal

Re-resolve as **Considered for Future Use** (the prior disposition for
this ticket in both the STU1 and STU2 ballots), with the rationale that
storage-conditions / stability content is not in active scope until
PQ/CMC Stage 3 Profiles for Stability are balloted, and the affected
value set will be re-authored as part of that work.

#### Justification

This is the workgroup's twice-recorded position (vote 12-0-0 in 2024,
vote 10-0-1 in 2025). The rationale — that storage conditions belong to
Stage 3 Stability, where the entire value set will be revisited — has
not changed. Fixing one display in isolation risks shipping a partial
correction in STU2 that is then re-touched in Stage 3, and increases
the chance of inconsistency between the published storage-conditions
content and what Stage 3 ultimately produces.

---

### Recommendation

**Recommended disposition:** A

The ticket is a documented, narrowly-scoped erratum: a single display
string for a single NCIt code that demonstrably does not match the
PQ/CMC PT it is supposed to convey. The previous "Considered for Future
Use" resolutions were driven by scope (the broader value set was
out-of-scope for that ballot cycle), not by any disagreement that the
display is wrong. The reporter has now re-opened the ticket twice and
it is carrying the `Block-Vote-3` label, which signals WG appetite to
clear it in this cycle. Accepting as requested is non-breaking
(code/binding unchanged), trivially auditable, and removes a real
implementer-facing defect in the published display. Disposition B is
the better long-term hygiene answer and should be considered as a
follow-up ticket against Stage 3, but it is disproportionate to gate
this single correction on. Disposition C is no longer defensible given
the re-opens; declining a third time without new substantive reasoning
will simply produce a fourth re-open after the next publication.
