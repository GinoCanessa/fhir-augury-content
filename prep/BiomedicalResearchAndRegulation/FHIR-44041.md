## Summary

This ticket asks the Biomedical Research & Regulation work group, on
behalf of the Pharmaceutical Quality (Industry) FHIR IG, to revise the
description and definition text on the data elements of the **Component
Building Block** profiles so that they carry PQ-specific wording rather
than the generic text inherited from the underlying base FHIR resources.
The intent is purely an editorial/profile-authoring change: the profile
structure, cardinalities, bindings, and slicing are not being modified,
only the human-readable `definition` / `comment` / `short` strings on
profiled elements where a PQ-specific clarification is warranted. The
ticket was originally voted by BR&R on 2024-02-21 (Diana Wright /
Rob Hausam, 15-0-0) as *Considered for Future Use*, then reopened on
2024-05-08 by Lynn Laakso "post STU1 publication" so the work can be
addressed in the next PQ release cycle. No related Jira tickets, Zulip
threads, or GitHub items were surfaced by the cross-reference search.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the change request and schedule an editorial pass over each
Component Building Block profile in the Pharmaceutical Quality (Industry)
IG. For every profiled `ElementDefinition` whose meaning has been
narrowed, constrained, or contextualised for PQ use, override the
inherited `short`, `definition`, and (where useful) `comment` strings
with PQ-specific text. Elements that retain the base-resource semantics
unchanged should be left alone so the IG does not gratuitously diverge
from the base spec. Track the affected profiles via a checklist in the
ticket and verify the change set with the IG Publisher's QA so that
overridden text shows correctly in the rendered profile snapshot and
differential views. No structural changes (cardinality, slicing,
bindings, must-support) are in scope.

#### Justification

This is the disposition the ticket literally asks for, and it is well
aligned with FHIR profiling best practice: profiles that re-use base
elements verbatim should inherit their text, but profiles that narrow
the *meaning* of an element for a specific domain should communicate
that narrowed meaning to implementers via the element's own
`definition`/`short`. Component Building Blocks are the foundational PQ
authoring units, so clarity here directly improves the usability of
every downstream PQ profile that composes them. The original BR&R vote
(15-0-0 to defer) was about timing rather than substance, and the
2024-05-08 re-open explicitly clears that timing constraint by anchoring
the work post-STU1.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than open-ended "update description and definition text where
appropriate", scope the work to a defined deliverable: (1) produce a
spreadsheet or markdown table enumerating every Component Building
Block profile element whose inherited base text is judged insufficient
for PQ, with the proposed replacement text and a one-line rationale per
row; (2) circulate that table for BR&R review and sign-off; (3) only
then apply the resulting overrides to the StructureDefinitions in a
single PR. Optionally, factor recurring PQ wording into reusable
snippets (e.g., a shared profile narrative include) so the same
clarification is not re-typed across multiple Building Blocks.

#### Justification

The ticket's "where appropriate" clause is the risk: without an explicit
list, two reviewers can disagree about which elements need PQ text,
the editorial pass can drift across multiple ballot cycles, and
re-published descriptions can creep away from the base resource in ways
the work group never explicitly approved. A review-then-apply workflow
gives BR&R a single artifact to vote on, makes the change auditable in
the ballot record, and avoids per-element re-litigation during ballot
reconciliation. It still delivers exactly what Disposition A delivers,
just with a controlled scoping step in front.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Modification → No Change", on the
grounds that FHIR profile inheritance already causes base-resource
`definition` text to render in profile views, and implementers are
expected to read the base resource definitions in conjunction with the
profile. Document the policy that PQ Component Building Block profiles
will *only* override base text when the profile changes cardinality,
binding strength, or slicing, and otherwise will inherit silently. Any
narrower clarification can be carried in the IG's prose pages rather
than per-element overrides.

#### Justification

Overriding base-resource `definition` text in a profile creates an
ongoing maintenance burden: when the base resource's definition changes
in a future FHIR release, the PQ override is silently left stale and
can directly contradict the normative base. There is also a real risk
of drift between profiles if multiple authors write parallel "PQ
flavour" prose for the same base element across different Building
Blocks. Declining (or, equivalently, restricting overrides to
structurally-changed elements) keeps the PQ IG aligned with the base
spec and pushes domain narrative into the IG's narrative pages, where
it belongs.

---

### Recommendation

**Recommended disposition:** B

Disposition B delivers the substance of what the reporter is asking for
— PQ-specific text on Component Building Block profile elements where it
genuinely helps implementers — while addressing the scoping ambiguity
that almost certainly drove the original 15-0-0 *Considered for Future
Use* vote. A pre-vetted table of "element → proposed PQ text" gives the
work group a concrete, reviewable artifact, prevents per-element
re-litigation during ballot reconciliation, and produces a single,
auditable PR rather than a drifting multi-cycle editorial pass. It also
naturally inherits the safeguard from Disposition C: any element whose
PQ meaning is unchanged from the base resource simply does not appear
in the table, which keeps the override surface minimal and limits
future drift against the base spec. Disposition A is acceptable but
unbounded; Disposition C under-serves the legitimate implementer-clarity
need that motivated the ticket's reopening post-STU1.
