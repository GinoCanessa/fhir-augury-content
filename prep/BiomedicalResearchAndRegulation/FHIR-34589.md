## Summary

The ticket asks the Biomedical Research & Regulation work group, in the
context of the *US ICSR Transfusion and Vaccination Adverse Event
Detection and Reporting (FHIR)* IG, to determine how to *refer to*
pre-defined NCI (National Cancer Institute) and NCI Meta value sets
rather than re-defining those value sets inside the IG itself. In other
words: where the IG currently inlines NCI / NCI Meta concept lists, the
authors should look for a stable, externally-maintained representation
of those value sets and bind to it instead. No related Jira tickets,
Zulip discussions, or GitHub items were surfaced through the augury
cross-reference and search indexes for this ticket key, so the request
stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace any in-IG-defined value sets in the *US ICSR Transfusion and
Vaccination Adverse Event Detection and Reporting* IG that mirror NCI
or NCI Metathesaurus content with bindings to externally-published
value sets sourced from NCI. Concretely:

1. Inventory each IG-internal `ValueSet` whose concepts come from NCI
   Thesaurus (`http://ncithesaurus.nci.nih.gov`, code system
   `http://ncithesaurus-stage.nci.nih.gov` / canonical
   `http://ncithesaurus.nci.nih.gov`) or NCI Metathesaurus
   (`http://ncimeta.nci.nih.gov`).
2. Identify, for each, an authoritative pre-published `ValueSet`
   canonical URL — preferably one published by NCI EVS (e.g., via
   the NCI EVS FHIR endpoint) or a recognised THO/UTG-hosted alias.
3. Update the affected profiles' element bindings (`element.binding.valueSet`)
   to point at those external canonical URLs at the appropriate
   `strength` (typically `extensible` or `preferred`, matching current
   IG intent).
4. Retire the now-redundant in-IG `ValueSet` resources (or convert
   them into thin wrappers that `compose.include` the external value
   set by reference, if local versioning is desired).

#### Justification

This is exactly what the ticket asks for and is consistent with FHIR's
general guidance to *reuse* canonical terminology artifacts rather
than fork them. Reusing NCI-published value sets keeps the IG aligned
with NCI's curation cadence, avoids drift between the IG copy and the
authoritative source, and reduces the IG's maintenance surface. NCI
EVS publishes both NCIt and NCI Metathesaurus content through FHIR
terminology services, which makes external referencing technically
viable.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than binding directly to NCI-hosted value sets, publish a
small set of *intensional* `ValueSet` resources in the IG (or in the
sponsoring realm's terminology IG) that `compose.include` concepts
**by code system + filter** from the NCI code systems, e.g.:

```
ValueSet.compose.include
  system = http://ncithesaurus.nci.nih.gov
  filter = concept is-a <NCIt root concept for the domain>
```

The IG's profile bindings would then point at these IG-owned, but
*externally-sourced*, `ValueSet` canonicals. Where an NCI Meta value
set is genuinely the right intent, include it the same way against
`http://ncimeta.nci.nih.gov`.

#### Justification

This preserves the spirit of the request (no hand-curated enumerations
of NCI codes living in the IG) while keeping the IG in control of the
canonical URL, version pinning, and the precise expansion semantics
that ICSR reporting needs. It avoids two real risks of pure
external-binding (Disposition A): (a) NCI's published ValueSets may
not match the exact slice of NCIt that ICSR needs, and (b) external
canonicals may evolve in ways that quietly change conformance for
implementers. Intensional value sets also degrade gracefully when
implementers expand against different terminology servers — they get
the same definition even if the expansion membership shifts with
NCIt versions.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as *Persuasive with Mod / Not Persuasive* on the basis
that no concrete change is required: re-defining NCI-derived value
sets locally in the IG is permitted under FHIR terminology rules, the
existing bindings already declare the NCI code systems by canonical
URL, and changing the bindings now would be a breaking change for
existing implementers of the ICSR transfusion/vaccination IG without
a demonstrated implementer pain point in the ticket.

#### Justification

The ticket is exploratory ("determine how to refer to…") rather than
identifying a defect. There is no attached use case where the
in-IG-defined value sets have caused interoperability problems,
no implementer feedback, no Zulip discussion, and no linked tickets
in five years. Switching established bindings to external canonicals
is itself non-trivial: it introduces a runtime dependency on NCI EVS
availability for validators, can change expansion membership over
time, and may not be reproducible across terminology servers — all of
which can destabilise a regulatory-adjacent IG.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between the ticket's stated intent
("don't redefine NCI content in the IG") and the operational realities
of a regulatory-adjacent specification. A pure external-binding
approach (A) is the most faithful reading of the request, but it
hands version and expansion control to NCI's publication cadence, and
the ticket does not establish that a suitable NCI-published `ValueSet`
canonical exists for every binding the IG currently defines. A flat
decline (C) ignores a legitimate maintenance concern: hand-curated
NCI enumerations *do* drift from NCIt over time. Publishing
intensional, IG-owned value sets that pull from the NCI code systems
by `compose.include` keeps the IG self-contained and version-pinnable,
removes the duplicated concept lists the ticket is worried about, and
leaves the door open to swap to a fully NCI-published canonical later
if and when one becomes the obvious authoritative choice. The work
group should confirm with the IG's terminology lead which NCI value
sets, if any, are already published in a form stable enough to bind
to directly — wherever that is true, those specific bindings can be
upgraded from "B" to "A" on a per-binding basis.
