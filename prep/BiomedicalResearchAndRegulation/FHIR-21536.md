## Summary

The reporter (Keith W. Boone) requests that the CDMH IG express its
mappings between Common Data Models (PCORnet, OMOP, i2b2, Sentinel) and
the BRIDG-aligned FHIR representation as FHIR `ConceptMap` resources
rather than as static prose tables. In a `ConceptMap`-based formulation,
the source-CDM column-name (e.g., `unit_concept_id`,
`value_as_number`) becomes a code in a source code system representing
that CDM, and the right-hand side becomes a code identifying the target
FHIR element / profile (for example, mapping `unit_concept_id` and
`value_as_number` in the OMOP `OBSERVATION` table to
`Observation.valueQuantity` rather than the polymorphic
`Observation.value[x]`). The ticket calls out one concrete correction in
the existing tables — `Observation.value[x]` should be replaced with the
specific choice type `Observation.valueQuantity` for that row — but the
broader ask is structural: replace (or augment) the manually-curated
mapping tables with machine-processable `ConceptMap` resources. The
ticket was resolved **Persuasive with Modification**; the work group's
resolution states it will use `ConceptMap` per the suggested comment
"where applicable", and the closing comments by `nageshbashyam` confirm
that **sample** `ConceptMap` resources have been added to the IG to
demonstrate the approach (rather than a full conversion of every
mapping table).

There are no cross-referenced Jira tickets to incorporate into this
summary; the only cross-references discovered are FHIR-artifact mentions
(`Observation.value[x]` and `Observation.valueQuantity`), which are the
targets of the example mapping called out in the description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Convert the CDMH-to-BRIDG/FHIR mapping tables in the CDMH IG into FHIR
`ConceptMap` resources — one `ConceptMap` per source CDM (PCORnet,
OMOP, i2b2, Sentinel) per target table grouping (e.g., OBSERVATION,
PERSON, MEASUREMENT, …). Each `ConceptMap.group` would carry:

- `source` = a CDM-specific `CodeSystem` whose codes are the column
  names of that CDM table (e.g., for OMOP OBSERVATION:
  `observation_concept_id`, `unit_concept_id`, `value_as_number`,
  `value_as_concept_id`, …).
- `target` = a CodeSystem of FHIR element paths / profile element
  references (e.g., `Observation.code`, `Observation.valueQuantity`,
  `Observation.valueQuantity.value`, `Observation.valueCodeableConcept`).
- `element.target.equivalence` populated (`equivalent`,
  `relatedto`, `wider`, `narrower`) so that the precise relationship is
  machine-checkable.

As part of this work, fix the specific defect called out in the ticket:
in the OBSERVATION table mapping for `unit_concept_id` and
`value_as_number`, change the right-hand side from the polymorphic
`Observation.value[x]` to the choice-specific
`Observation.valueQuantity` (with `unit_concept_id` →
`Observation.valueQuantity.code`/`system` and `value_as_number` →
`Observation.valueQuantity.value`).

The narrative tables in the IG would be retained but rendered from the
`ConceptMap` resources so the prose and the machine-readable artifact
cannot drift out of sync.

#### Justification

`ConceptMap` is the FHIR-native, machine-processable mechanism for
expressing exactly this kind of cross-model element mapping, and it is
already used elsewhere in HL7 IGs to publish curated mappings. Promoting
the tables to first-class artifacts lets implementers consume the
mappings programmatically (e.g., to drive ETL from OMOP/PCORnet/i2b2/
Sentinel staging tables into FHIR), supports validation and tooling, and
forces the more precise choice-type targeting (`valueQuantity` vs.
`value[x]`) that the ticket flagged. It is also consistent with HL7's
general direction of preferring conformance resources over prose for
anything implementers need to act on.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing prose mapping tables as the primary, human-curated
artifact, but publish a **representative set of sample `ConceptMap`
resources** in the IG (one per CDM, covering one or two
high-value tables such as OBSERVATION and PERSON) to demonstrate the
pattern, plus a short authoring guide. Independently, fix the
`Observation.value[x]` → `Observation.valueQuantity` correction in the
OBSERVATION table directly, and adopt a convention (e.g., `MAP-001:`
identifiers in the table) so that future complete `ConceptMap`
conversions can be cross-walked back to the prose tables without
rewriting them.

This is essentially what the resolution comments describe was actually
delivered ("Concept Maps have been added to the IG as samples to show
how it can be used").

#### Justification

A full conversion of every CDMH mapping table to `ConceptMap` is a
substantial editorial effort and demands stable source/target
`CodeSystem` definitions for each CDM's column vocabulary, several of
which (e.g., i2b2's i2b2-CRC star schema) do not have an HL7-published
code system today. A sample-plus-pattern approach unblocks implementers
who want to produce or consume `ConceptMap` representations, while not
holding the IG to a deliverable that exceeds the work group's editorial
capacity. It also matches what the work group actually did at
resolution, so adopting Disposition B retroactively documents the
ballot decision rather than reopening it.

---

### Disposition C: Decline

#### Proposal

Decline as **Already Addressed**. Close FHIR-21536 with no further
change beyond confirming that (a) sample `ConceptMap` resources are
present in the IG (per `nageshbashyam`'s 2021-12-01 comments) and (b)
the specific OBSERVATION-row defect (`Observation.value[x]` →
`Observation.valueQuantity`) has been corrected in the published
tables. No new normative or structural change to the CDMH IG is made.

#### Justification

The ticket has resolution **Persuasive with Modification** and the
closing comments assert that the modification (sample `ConceptMap`s)
has already been applied. If the spelling-of-the-resolution work is
complete, the residual ticket is administrative and can be closed. The
ticket carries the `Block-Vote-1` label, indicating it was already
batched with similar items for a single ballot decision; reopening it
for a broader conversion would re-litigate that vote.

---

### Recommendation

**Recommended disposition:** B

The ticket's underlying intent — making CDMH mappings machine-readable
— has merit and should not be lost, but a full, comprehensive
conversion of every CDM table into normative `ConceptMap` resources is
a much larger editorial commitment than the original ticket implied,
and it depends on stable column-level `CodeSystem`s for each CDM that
do not yet exist in HL7-published form. Disposition B captures the
correct precedent (the work group already shipped *sample* `ConceptMap`s
under "Persuasive with Modification") and pairs it with two concrete
follow-ups the work group can land cheaply: fix the specific
`Observation.value[x]` → `Observation.valueQuantity` defect that the
ticket flagged, and add a short authoring-pattern note so future
contributors can grow the `ConceptMap` coverage incrementally.
Disposition A is the right *long-term* direction but should be tracked
as a separate, scoped editorial work item rather than as the resolution
of this ballot ticket; Disposition C ignores the legitimate residual
defect (the polymorphic-vs-choice element error) the reporter
identified.
