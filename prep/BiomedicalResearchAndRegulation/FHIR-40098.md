## Summary

The ticket asks the Biomedical Research & Regulation work group to incorporate
the [Occupational Data for Health (ODH)](http://hl7.org/fhir/us/odh/) IG into
section 3 of the *Retrieval of Real World Data for Clinical Research* (RWD) IG.
The reporter, Genevieve Luensman, notes that ODH — a framework describing a
person's work (industry, occupation, employment status, work classification,
combat zone period, etc.) — is not formally part of IPS (the International
Patient Summary), but is materially important when assembling real-world data
for clinical research. The request was originally resolved as "Considered for
Future Use" in March 2023 on the grounds that the first iteration of the RWD IG
had no concrete ODH use cases, and was then re-opened post-STU publication on
2023-05-29 by Lynn Laakso so that ODH coverage can be reconsidered for the next
revision of the guide. No cross-referenced Jira tickets, Zulip discussions, or
GitHub items were surfaced for this ticket via `fhir-augury-cli`.

## Proposed Dispositions

Note: because no GitHub cross-references were available, the dispositions
below cannot name a specific authoring root or file path from a
`repo-analysis` briefing. They are scoped against the published RWD IG
(`hl7.org/fhir/us/rwd`) and the ODH IG (`hl7.org/fhir/us/odh`).

### Disposition A: Accept As Requested

#### Proposal

In the next revision of the *Retrieval of Real World Data for Clinical
Research* IG, add ODH coverage to section 3 ("Retrieving Patient Data" /
data-domain narrative). Specifically:

1. Add a dedicated subsection to section 3 titled "Occupational Data for
   Health (ODH)" that:
   - Cites the ODH US Realm IG (`http://hl7.org/fhir/us/odh/`) as a normative
     source for work-related observations.
   - Enumerates the ODH profiles the RWD IG expects servers to expose —
     minimally `odh-PastOrPresentJob`, `odh-EmploymentStatus`,
     `odh-UsualWork`, `odh-RetirementStatus`, `odh-CombatZonePeriod`, and the
     work-classification value sets they reference.
   - Documents `Observation` search expectations for these profiles
     (`patient`, `code`, `category=social-history`, `date`,
     `_profile=http://hl7.org/fhir/us/odh/StructureDefinition/...`) so a RWD
     consumer can pull a patient's occupational history with a deterministic
     bundle of queries.
2. Add a capability-statement entry (or supported-profile annotation on
   `Observation`) declaring the ODH profiles as `supportedProfile` for RWD
   servers/clients.
3. Add a worked example bundle to the RWD IG that includes at least one
   `odh-PastOrPresentJob` and one `odh-EmploymentStatus` resource so
   implementers see the expected shape end-to-end.

#### Justification

The reporter and the BR&R co-authors (Amanda Deering, Lori Fourquet, Lori
Fourquet's prior ODH work) treat occupational exposure and employment status
as primary covariates for many real-world clinical questions — environmental
and occupational disease, return-to-work outcomes, veteran/combat-exposure
research, and pandemic surveillance. ODH is the only US Realm normative IG
that carries this data in a profiled, codeable form, so reusing it (rather
than reinventing free-text occupation in the RWD IG) is consistent with FHIR's
"profile, don't redefine" guidance and with how US Core composes other US
Realm IGs (US Core → US Public Health Profiles, etc.). Re-opening the ticket
post-STU publication explicitly anticipates this kind of additive change for
the next ballot cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than embedding ODH-specific narrative and supported-profile claims into
section 3, take a lighter-weight, conformance-driven approach:

1. In section 3, add a single short paragraph stating that work-related
   observations relevant to real-world research SHOULD be exchanged using the
   ODH US Realm IG, with a normative reference to
   `http://hl7.org/fhir/us/odh/` and a non-exhaustive list of the ODH profile
   URLs.
2. Do **not** copy ODH search parameter or example detail into the RWD IG.
   Instead, in the RWD IG's CapabilityStatement(s), add the relevant ODH
   profiles to `Observation.supportedProfile`, and rely on the ODH IG itself
   for search semantics, value sets, and examples.
3. File a follow-up tracker (a separate Jira ticket) to revisit *required*
   ODH profile support once concrete RWD use cases — e.g., the cohorts the
   group is building examples around — actually exercise occupational data.

#### Justification

The original 2023-03-01 resolution ("no use cases yet") still has some force:
the RWD IG's value comes from the queries and bundles its use cases drive, and
if ODH is added without concrete RWD examples it risks becoming aspirational
text. A SHOULD-level reference plus a CapabilityStatement
`supportedProfile` claim gives implementers an unambiguous signal that ODH is
in scope without forcing the RWD IG to maintain a parallel narrative that can
drift from the upstream ODH IG. This also reduces the cross-IG maintenance
burden: when ODH revs, the RWD IG only needs to re-pin its profile URLs, not
re-author search/example content.

---

### Disposition C: Decline

#### Proposal

Decline the request and re-close as "Considered for Future Use," restating
that ODH inclusion will be reconsidered when (a) a specific real-world
research use case in the RWD IG's scope explicitly requires occupational data,
and (b) the BR&R work group has authoring capacity coordinated with the CDC
NIOSH ODH maintainers.

#### Justification

This essentially restores the 7-0-0 March 2023 vote. The argument is that the
RWD IG's section 3 is organized around use-case-driven data retrieval, and
adding an entire data domain without a concrete RWD use case to anchor it
sets a precedent that any clinically interesting US Realm IG can be appended
to RWD on first principles. Until BR&R has a documented RWD use case that
needs occupational data — e.g., a cohort retrieval scenario where occupation
or employment status is a query parameter or a required result element —
referring readers to the ODH IG as a sibling US Realm guide is sufficient.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance for a re-opened, post-STU change request
with no cross-referenced design discussion in Zulip or GitHub yet. It
satisfies the reporter's core concern — that ODH be acknowledged as in-scope
for real-world clinical research data retrieval — by creating an explicit,
implementer-actionable hook (a normative reference plus a
`supportedProfile` entry on `Observation`) without committing the RWD IG to
maintaining a parallel ODH narrative before any RWD use case has actually
exercised the data. Disposition A is the right end state once concrete RWD
use cases are written that touch occupational data, and Disposition B leaves
a clean upgrade path to A via the follow-up tracker. Disposition C ignores
the explicit re-open by the work group facilitator (Lynn Laakso) and the
reporter's named co-authors (Amanda Deering, Lori Fourquet), and is hard to
defend now that the ticket has been deliberately moved off "Considered for
Future Use."
