## Summary

Reporter Keith Salzman is asking whether the algorithms enumerated in the
*US ICSR Transfusion and Vaccination Adverse Event Detection and Reporting
(FHIR)* IG have been evaluated for accuracy — specifically, whether
sensitivity/specificity figures or any comparison of reported instances
versus IG-detected instances are available so implementers understand the
margin of error of automated case detection. The ticket carries no
"Existing Wording" or "Proposed Wording" content; it is essentially a
clarifying question about the evidentiary basis (and IG-published
performance characteristics) of the detection algorithms. A prior
disposition of *Considered for Future Use* was approved 16-0-0 (Duteau /
Gregory, 2021-11-23) noting that sensitivity and specificity were not
evaluated because false negatives could not be assessed, but positive
predictive value could potentially be reported after larger-scale
validation. That resolution was subsequently reverted and the ticket was
re-opened for re-consideration after STU1 publication. No Jira, Zulip,
or GitHub cross-references were surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the IG narrative (the *Algorithms* / detection-methodology pages
of the **US ICSR Transfusion and Vaccination Adverse Event Detection and
Reporting (FHIR)** IG) to add an "Algorithm Performance" subsection for
each algorithm enumerated in the guide. The subsection would:

- State explicitly which performance metrics have and have not been
  evaluated (sensitivity, specificity, PPV, NPV, F1).
- Where validation work exists, publish the metric values along with the
  reference cohort, denominator definition, and version of the algorithm
  evaluated.
- Where evaluation has not been performed, say so plainly (e.g.,
  "sensitivity and specificity have not been characterized because
  false-negative ascertainment was infeasible") and identify what would
  be needed to obtain them.
- Add (per the 2021-11-23 vote rationale) a roadmap statement that PPV
  may be reported in future iterations after larger-scale validation.

No structural FHIR artifact changes (no profile, extension, value set,
or operation changes) are implied — this is narrative content in the
IG's `input/pagecontent/` (or equivalent) authoring area.

#### Justification

This directly answers the reporter's question and satisfies the
re-open-after-STU1 instruction by giving implementers the transparency
they need to judge the margin of error of automated detection.
Publishing what *has* and *has not* been measured (rather than silently
omitting performance data) aligns with FHIR IG conventions for
algorithmic guidance and matches the substance of the original
*Considered for Future Use* rationale, which explicitly anticipated
PPV-style reporting once validation work matured.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than embedding algorithm-performance numbers in the IG itself,
add a normative-style "Algorithm Evaluation Expectations" page that:

1. Defines a minimum disclosure profile each implementer / algorithm
   author SHOULD publish (metric name, study design, denominator,
   algorithm version, dataset description, date of evaluation).
2. Points to an external companion artifact (HL7 Confluence page,
   companion repository, or publisher-maintained registry) where actual
   measurements are tracked over time, with a link from the IG.
3. Flags within the algorithm sections themselves which entries
   currently have published evaluations (linking out) versus which do
   not.

This keeps the IG free of metrics that age quickly while still
addressing the reporter's underlying need for transparency about
detection accuracy.

#### Justification

Algorithm performance characteristics drift as data sources, code
systems, and clinical practice evolve, so baking specific
sensitivity/specificity numbers into a balloted IG creates a maintenance
burden and risks publishing stale figures. Treating the IG as the
*framework* for disclosure and an external companion artifact as the
*living evidence base* is consistent with how other detection-oriented
IGs (e.g., public health and case-reporting work) handle algorithm
validation. It also respects the original 2021-11-23 finding that
sensitivity/specificity could not be meaningfully computed at the time,
without freezing that limitation into the IG narrative forever.

---

### Disposition C: Decline

#### Proposal

Re-affirm the original *Considered for Future Use* disposition (vote
16-0-0, 2021-11-23) without further change to the STU. Document that
the IG's role is to specify the data structures and exchange semantics
for adverse-event detection and reporting, and that algorithm
sensitivity/specificity measurement is the responsibility of
implementers and the studies that characterize each algorithm in
context, not of the IG itself.

#### Justification

Detection algorithms are typically deployed against widely varying data
sources (EHR phenotypes, claims, registries) where ground-truth
ascertainment of false negatives is infeasible at IG-author scope — the
exact reason the 2021-11-23 vote landed on *Considered for Future Use*.
Publishing single performance numbers in the IG could imply a
generalizability the underlying studies do not support, and could be
read as endorsing one algorithm over another. The original disposition
already acknowledged that PPV reporting may follow future validation
work; nothing in the re-open comment indicates that such validation has
since been completed.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need — implementers being able to reason
about the margin of error of automated detection — is legitimate and
should not be dismissed (which rules out C as a sole answer). At the
same time, the 2021-11-23 finding that sensitivity/specificity cannot
be soundly computed in this domain is still on the record and was not
contradicted by anything in the re-open. Disposition B threads that
needle: it commits the IG to *transparency about evaluation* (a
disclosure profile and an explicit "what has / has not been measured"
posture) without forcing the IG to publish numbers that would either
mislead or quickly age. It also leaves a clean path to fold in PPV or
other metrics as validation studies mature, which is exactly the future
the original vote envisioned. Disposition A is a reasonable fallback
if BR&R prefers to keep all evaluation content inside the IG; C should
only be the outcome if the workgroup concludes no new validation
material exists to disclose at all post-STU1.
