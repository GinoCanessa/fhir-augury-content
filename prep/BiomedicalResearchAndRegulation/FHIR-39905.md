## Summary

The reporter requests that the *Retrieval of Real World Data for Clinical
Research (FHIR)* (Vulcan RWD) implementation guide include support for
exchanging Device identification information by adding the appropriate
Device profile(s) to the IG. The ticket was originally resolved as
*Considered for Future Use* on 2023-03-01 (vote 7-0-0) on the basis that
the first STU's use cases did not include searching for devices, but it
was reopened post-STU publication on 2023-05-29 to be reconsidered for a
subsequent ballot cycle now that additional use cases (e.g., the ACS
example) and the IPS Device profile precedent are in scope. No
cross-referenced Jira tickets, Zulip threads, or GitHub artifacts were
surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new Vulcan RWD Device profile to the IG that constrains the base
`Device` resource for the real-world-data exchange use case. Concretely:

- Author a new `StructureDefinition-Device-rwd` profile (or
  similarly-named) under the IG's profile authoring area.
- Derive from the IPS Device profile
  (`StructureDefinition-Device-uv-ips`) as a starting point so the RWD
  IG aligns with the IPS precedent already cited in the ticket
  comments.
- Constrain `Device.identifier`, `Device.udiCarrier` (with `deviceIdentifier`,
  `issuer`, `jurisdiction`), `Device.type` (bound to an appropriate
  device-type value set), `Device.manufacturer`, `Device.distinctIdentifier`,
  `Device.lotNumber`, `Device.serialNumber`, `Device.expirationDate`, and
  `Device.patient` to the cardinalities/must-supports the RWD use cases
  (e.g., the ACS implantable-device example) require.
- Add the profile to the IG's resource list / capability statement, add
  examples (at minimum, an implantable-device example aligned with the
  ACS use case page), and reference the profile from the relevant
  use-case narrative pages.
- Document the search expectations on `Device` (`patient`, `type`,
  `identifier`, `udi-di`) in the IG's search section so the original
  workgroup concern about "we don't search for devices" is explicitly
  addressed for the post-STU iteration.

#### Justification

The reporter and the post-STU reopen comment together establish that
device identification is in scope for subsequent RWD ballots, and the
ACS use case already published in the IG demonstrates a real device
exchange need. Profiling rather than referencing IPS in name only gives
RWD implementers the same constraints they get for other RWD profiles
and avoids inconsistency with the rest of the guide. Deriving from IPS
Device aligns with the precedent the workgroup itself cited and reduces
the authoring burden.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring a brand-new RWD-specific Device profile, **adopt
the IPS Device profile by reference** for this iteration:

- Add a Device section to the IG that normatively requires conformance
  to `StructureDefinition-Device-uv-ips` for any Device resource
  exchanged in an RWD context.
- Add a thin RWD overlay only if a concrete RWD requirement cannot be
  satisfied by the IPS profile (for example, a tighter must-support on
  `Device.patient` or a specific binding for `Device.type` driven by an
  RWD use case); otherwise leave the IPS profile unchanged.
- Add at least one Device example tied to the ACS use case and update
  the use-case page to point at the IPS profile.

#### Justification

This is lower-cost, ships faster, and avoids the proliferation of
near-duplicate Device profiles across IGs that implementers already
complain about. It also matches the workgroup's stated incremental,
use-case-exploratory authoring style: introduce only the constraints
real RWD use cases force, and inherit everything else from IPS. If
later ballots surface RWD-specific gaps, an RWD-derived profile can be
added then without breaking implementations that already conform to
IPS.

---

### Disposition C: Decline

#### Proposal

Decline the request and restore the prior resolution of *Considered for
Future Use*, on the basis that the current STU's use cases still do not
include device search and that the ACS narrative can be supported with
the unprofiled base `Device` resource for now.

#### Justification

The original 7-0-0 vote captured a legitimate scope decision: without a
concrete search/exchange use case in this iteration, a Device profile
would be normative content the IG cannot validate against real
implementations. Publishing a profile that no implementer is required
to support risks freezing constraints that turn out to be wrong when
the use cases are actually written.

---

### Recommendation

**Recommended disposition:** B

The reopen comment makes it clear the workgroup no longer wants a flat
"deferred" answer, but the original 7-0-0 concern — that RWD doesn't
yet have its own validated device-exchange use cases — is still
substantially true. Adopting the IPS Device profile by reference (with
at most a thin overlay where an RWD use case demands it) gives
implementers a concrete, conformant target for the ACS-style scenario
already in the guide, aligns with the IPS precedent the workgroup
itself cited, and avoids committing the IG to RWD-specific Device
constraints before the use cases that would justify them have been
written. Disposition A remains the right path once future RWD ballots
add device-search use cases that IPS can't express; Disposition C is
hard to defend now that the ticket has been explicitly reopened.
