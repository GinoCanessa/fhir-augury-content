## Proposed Dispositions

> Caveat: the ticket body does not enumerate which sIRB pages, profiles, or
> elements are affected, and there are no cross-referenced artifacts or repo
> briefings to ground a change. The dispositions below therefore frame the
> *process* the work group should follow as much as the substantive outcome.

### Disposition A: Accept As Requested

#### Proposal

Accept the ticket as a tracked editorial/technical-correction change set
against the Single Institutional Review Board (sIRB) FHIR IG. As a
prerequisite, ask the reporter (Diane Rodden) to attach the concrete list of
"hidden fields for external data" edits and the accompanying technical
corrections — ideally as an itemized comment on the ticket or as a linked
working document — so that each change is independently reviewable. Once the
list is attached, apply the changes against the sIRB IG authoring sources,
treating each "hidden field" as a profile/element-level edit (e.g., adjusting
`mustSupport`, `min`/`max`, `isModifier`, slicing, or extension definitions
that govern whether external-data fields are surfaced) and treating the
"technical corrections" as editorial fixes to narrative pages, examples, and
ValueSet/CodeSystem references.

#### Justification

The ticket is filed by the responsible work group against its own IG, the
status is `Triaged` (i.e., the WG has already accepted that something needs
to be done), and "hidden fields" / "technical corrections" are normal IG
maintenance activities that do not require a change to the base FHIR spec.
The main risk in accepting it is procedural rather than technical: the ticket
as written is not actionable without the underlying change list, so any
acceptance must be conditioned on that list being captured in the ticket
itself before publication-time changes are made.

---

### Disposition B: Alternative Approach

#### Proposal

Convert FHIR-40058 from a single omnibus Change Request into a tracking
parent and split it into one child ticket per concrete change. Specifically:

- one ticket per "hidden field for external data" with the affected sIRB
  profile and element path named explicitly (e.g., the StructureDefinition
  and the `Element.id`/path being toggled, plus the intended cardinality,
  `mustSupport`, or `isModifier` change);
- one ticket per "technical correction" (typo, broken cross-link, bad
  example, mis-bound ValueSet, etc.) with the page or artifact identified.

The original ticket is then resolved as `Persuasive with Modification` once
all children are filed, and the substantive work happens in the children.

#### Justification

A grouped "hidden fields + technical corrections" ticket is hard to
disposition, hard to apply, and hard to audit later — reviewers cannot tell
which corrections were applied, and ballot/QA processes generally expect one
discrete change per ticket. Splitting also lets the WG land the
non-controversial editorial fixes immediately while the "hidden field"
changes (which can affect downstream implementers consuming external data)
get the additional scrutiny they warrant. This is the lowest-risk way to
honor the reporter's intent without committing the WG to a vague change set.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-40058 in its current form as **Not Persuasive — insufficient
detail to action**, with an explicit invitation for the reporter to refile
one or more replacement tickets that name the affected sIRB profiles, pages,
and elements and describe the desired change for each.

#### Justification

Defensible because the ticket as filed does not meet the basic actionability
bar: no specific artifact, page, element, or correction is named; there is no
attached working document; there are no cross-referenced Jira tickets, Zulip
discussions, or GitHub PRs in the Augury index that would let a reviewer
reconstruct the intended change set; and the ticket has been static since
mid-2023. Acting on it as written would either require the WG to invent the
change list (which substitutes editor judgement for the reporter's intent)
or to rubber-stamp an unspecified set of edits at publication time. Declining
without prejudice keeps the door open for a properly scoped refile.

---

### Recommendation

**Recommended disposition:** B

Disposition B (split into per-change child tickets, then resolve the parent
as `Persuasive with Modification`) is the best balance of honoring the
reporter's clear intent — the WG has already triaged the ticket, so there is
appetite to make these changes — while restoring the level of specificity the
sIRB IG maintainers and downstream implementers need. Disposition A is
acceptable in principle but only after the same itemization work is done, and
doing that work inside a single omnibus ticket makes the change history
harder to audit at ballot/publication time. Disposition C is too harsh given
that the work group itself surfaced the request and the underlying need
(hidden-field handling for external data and routine corrections) is
plausible IG-maintenance work; an outright decline risks losing the
reporter's institutional knowledge. Approaching this as a tracking parent
also creates the hook needed to commission a `repo-analysis` run against the
sIRB IG repository before any of the child tickets are implemented, which
this prep was unable to do because no GitHub repository is currently
associated with the ticket in the Augury index.
