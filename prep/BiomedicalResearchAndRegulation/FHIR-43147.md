## Summary

Lloyd McKenzie reports that a number of the extensions defined in the
**US Structured Product Labeling Mapping (FHIR)** IG declare an
extension `context` of `Element` (the root abstract type), which is far
broader than the obvious intended use of those extensions. He asks that
the IG authors tighten each extension's `Extension.context` so that it
names the specific resource(s) and/or element path(s) where the
extension is actually meant to be used.

This is one of a coordinated tranche of nearly identical "Please
tighten your extension contexts" Change Requests that Lloyd filed in
November 2023 against multiple IGs (e.g.,
[FHIR-43141](https://jira.hl7.org/browse/FHIR-43141),
[FHIR-43143](https://jira.hl7.org/browse/FHIR-43143),

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Walk every extension `StructureDefinition` published by the US SPL
Mapping IG. For each one whose `context` array currently contains an
entry with `type = element` and `expression = Element` (i.e., "this
extension may be used on any element anywhere"), replace that single
entry with a list of explicit `context` entries that name:

- the specific FHIR resource(s) the extension is meant to decorate
  (e.g., `MedicinalProductDefinition`, `PackagedProductDefinition`,
  `Ingredient`, `RegulatedAuthorization`, `AdministrableProductDefinition`,
  `ManufacturedItemDefinition`, etc., as applicable to each extension); and/or
- the precise element path(s) within those resources where the SPL
  mapping actually uses the extension (e.g.,
  `MedicinalProductDefinition.name`,
  `PackagedProductDefinition.packaging`, etc.).

Where an extension is genuinely reusable across multiple SPL artefacts,
list each applicable `(type, expression)` pair explicitly rather than
falling back to `Element`. Update the IG's narrative and any
`extensions.html`-style index pages so the published context matches
the new, narrower definition. Re-publish and re-validate the IG to
confirm no example instances are broken by the tighter contexts.

#### Justification

This is the literal, ticket-as-written remedy and aligns with FHIR's
core authoring guidance: `Extension.context` exists precisely so
implementers and validators can reject incorrect placement, and the
FHIR specification explicitly discourages publishing extensions with a
context broader than their intended use. SPL is a tightly scoped
mapping IG — every extension in it exists to carry a specific SPL
construct onto a specific FHIR resource — so there is nearly always a
demonstrably narrower context available. Tightening the context costs
nothing at runtime, improves validator output for SPL implementers,
prevents accidental reuse on unrelated resources, and brings the IG
into line with the same expectation Lloyd is enforcing across the
other tranche-43xxx IGs (a positive consistency signal across HL7).

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a hybrid approach:

1. For extensions whose intended scope is **unambiguous and singular**
   (the majority), apply the Disposition-A narrowing — replace
   `Element` with the specific resource/element path(s).
2. For extensions whose intended scope is **genuinely cross-cutting
   within SPL but bounded** (e.g., a "source SPL section reference"
   extension that may legitimately attach to many SPL-related elements),
   keep `context.type = element` but change `context.expression` from
   the bare `Element` to a constrained discriminator — e.g., enumerate
   each SPL-relevant resource (`MedicinalProductDefinition`,
   `PackagedProductDefinition`, `Ingredient`, …) as separate context
   entries, even if the list is long.
3. For any extension where narrowing would be premature because the
   SPL→FHIR mapping is still evolving, **document the intended context
   in the extension's `description` and `purpose`** and additionally
   apply the
   [`structuredefinition-extension-context-recommended`](https://jira.hl7.org/browse/FHIR-41453)
   pattern (or equivalent narrative-level marker) so future authors
   understand the constraint even though the formal `context` remains
   broader.

Track the per-extension decisions in a small spreadsheet attached to
the ticket so reviewers can see why each extension landed where it did.

#### Justification

A pure global narrowing (Disposition A) risks breaking downstream SPL
trial-use implementations that may already be attaching SPL extensions
to elements that the IG authors did not foresee. The hybrid approach
captures the same conformance benefit Lloyd is asking for on the easy
cases (which are most of them) while giving BR&R an explicit, auditable
escape valve for the few extensions where genuine cross-cutting use is
desirable. It also opens a constructive path with the Lloyd-tranche
FHIR-41453 work, which is exploring meta-context expression — useful
because SPL is one of the regulatory IGs most likely to have legitimate
"applies anywhere a regulated product appears" extensions.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Persuasive but Not Now / No Action**, with the
following rationale: SPL Mapping is published as a low-maturity /
trial-use FHIR IG and the extensions in it deliberately use
`context = Element` to remain forward-compatible while the underlying
SPL→FHIR mapping is still being refined with FDA stakeholders.
Tightening `Extension.context` now would force a re-publication cycle
without changing how the extensions are actually consumed by SPL
implementers (who follow the SPL prose, not the FHIR validator), and it
risks invalidating in-flight SPL test instances. Defer the cleanup
until the next substantive SPL Mapping release, and capture the work
as a tracking item on that release plan rather than acting on it now.

#### Justification

This disposition is defensible only if BR&R can show (a) that the SPL
IG is in active churn such that any context narrowing would have to be
re-done, and (b) that no current SPL implementer is asking the
validator to reject misplaced extensions. Even then it is the weakest
of the three options because the request itself is mechanical and
low-risk, and FHIR's published authoring guidance already says
`context = Element` should be avoided. Use this disposition only as a
deliberate "we will do it later, on our own schedule" signal, not as a
substantive disagreement with Lloyd's analysis.

---

### Recommendation

**Recommended disposition:** **B (Hybrid Narrowing)**.

Disposition B captures essentially all of the conformance benefit
Lloyd is asking for in Disposition A, while explicitly acknowledging
that an SPL→FHIR mapping IG is exactly the kind of regulatory artefact
where a small number of extensions legitimately want to span several
related resources. A pure A-style sweep is tempting because it is
mechanically simple and matches what the other tranche-43xxx work
groups will likely do, but it carries real risk of either (i) freezing
the IG to the current resource set prematurely or (ii) provoking a
follow-up "please loosen these contexts again" ticket the next time
the SPL mapping expands. Disposition C is hard to defend on the merits:
the underlying critique is correct, the FHIR spec is on Lloyd's side,
and "we'll do it later" is exactly the rationale that produced the
tranche in the first place. BR&R should adopt B, attach the
per-extension narrowing decisions to the ticket as evidence, and close
it as **Persuasive with Modification** once the next SPL Mapping
publication ships with the tightened contexts.
