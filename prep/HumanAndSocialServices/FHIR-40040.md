## Summary

The submitter (Greg Bloom, of the Open Referral / HSDS community) reports a
data-model alignment gap between the Human Services Directory (FHIR) IG and
the underlying HSDS source specification. In HSDS, multiple `contact` records
can be associated with either a `service` or an `organization`, and each
contact carries its own location/address and phone details. In the current
HSD IG, however, `Service` (i.e., `HealthcareService`) supports multiple
locations and telecom values but offers no way to identify which contact
person those details belong to, while the IG's `Organization` profile carries
contact information but lacks the richer location/address details that HSDS
attaches to a contact. The reporter proposes adding a "contact section" to
the Service profile to close this alignment gap and characterizes it as one
of the most significant — though not critical — limitations between the two
data models. No cross-referenced Jira tickets, Zulip threads, or GitHub items
were found via FhirAugury for this ticket.

## Details

**Description:**

In HSDS, multiple Contacts can be associated with a service or organization,
each with their own Location and Phone details. Within this IG, a Service can
have multiple locations and phone details but you can not specify the contact
to which they relate too. Meanwhile, Organization here *does* have contact
information, however *does not* have the full Location information, like
Contact does in HSDS.

Adding a contact section to the Service would improve alignment. We wouldn't
say this is a critical issue that absolutely needs to be addressed in this
release, but it is one of the most significant limitations in alignment
between the two data models.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persisted Considered for Future Use is reverted; treat the ticket as in scope
for the next version of the Human Services Directory (FHIR) IG. Add a new
`contact` slice / extension on the IG's `HealthcareService` profile that
mirrors the structure already available on `Organization.contact` —
specifically allowing 0..* contacts each carrying `purpose`
(CodeableConcept), `name` (HumanName), `telecom` (0..* ContactPoint), and
`address` (Address). The element should be defined as a complex extension
(e.g., `hsds-service-contact`) so it can be added without altering the
underlying R4 `HealthcareService` resource. Update the HSDS → FHIR mappings
page so HSDS `contact` records linked to a `service` map to this extension,
HSDS `contact` records linked to an `organization` continue to map to
`Organization.contact`, and HSDS `contact` records linked to
`service_at_location` map to the same extension on `HealthcareService` (with
guidance that the location context is conveyed via `HealthcareService.location`).
Mark the extension `MS` but optional (`0..*`) consistent with HSDS not
requiring contact per service.

#### Justification

This directly addresses the alignment gap the reporter identifies and follows
the architecture already endorsed in Chirag Bhatt's 2023-03-02 analysis on
the ticket — that contact information at the service level requires an
extension on `HealthcareService`. Modeling it after `Organization.contact`
keeps the IG internally consistent and minimizes implementer learning cost.
Keeping it optional respects HSDS's own optionality and matches the
empirical observation in Bhatt's analysis that, in OpenReferral test data,
only one service record had an associated contact. The lynn_laakso
2023-10-04 comment ("re-open post STU1 publication") explicitly clears this
ticket for action in a post-STU1 cycle, so accepting it now lines up with
the WG's own staging plan.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a new `HealthcareService.contact` extension, reuse
the existing R5 `ExtendedContactDetail` data type via the standard
`http://hl7.org/fhir/StructureDefinition/contactdetail` (or, in R4, the
`extended-contact-detail` extension already used by other IGs such as US
Core / PlanNet) and apply it as `HealthcareService.extension:contact`. The
mapping would say: each HSDS `contact` linked to a `service` becomes one
`ExtendedContactDetail` instance with `purpose`, `name`, `telecom`,
`address`, and (optionally) `organization`. Apply the same data type
consistently to `Organization` (replacing the IG's current use of
`Organization.contact` BackboneElement) and to `Location` so that
HSDS-style contacts can be attached uniformly anywhere HSDS allows them
(`organization`, `service`, `service_at_location`). Document a single
"HSDS Contact" profile of `ExtendedContactDetail` that other profiles slice
or reference.

#### Justification

`ExtendedContactDetail` exists precisely to carry the bundle of
purpose/name/telecom/address/organization that HSDS's `contact` entity
expresses, and adopting it everywhere HSDS allows contacts produces a more
uniform and forward-looking model than scattering bespoke complex extensions
across `HealthcareService`, `Location`, and `Organization`. It also
positions the IG for an eventual move to R5 with little rework. The
trade-off vs. Disposition A is greater short-term churn — the IG's current
`Organization.contact` usage would be redefined — but it avoids the
long-term cost of maintaining a parallel custom contact extension. It still
honors Bhatt's recommendation that the new capability be expressed as an
optional extension on `HealthcareService`, just using a standard data type.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive with Mod** (or restore the prior **Considered
for Future Use** disposition). Decline to add a service-level contact
mechanism in this version of the IG on the basis that (1) the requirement
is explicitly described by the reporter as non-critical, (2) HSDS itself
does not require contact per service and the OpenReferral test corpus
showed contact records present on only a single service, and (3) the
existing IG already permits an implementer to model a service's contact via
the `Organization` referenced from `HealthcareService.providedBy` or via
the back-pointer from `PractitionerRole`. Recommend that, if implementers
need this in production today, they use a generic `extended-contact-detail`
extension off-IG, and revisit the requirement once HSDS-FHIR alignment is
re-scoped for an R5-based revision.

#### Justification

This disposition is defensible because the reporter's own framing
explicitly de-prioritizes the change ("not a critical issue"), Chirag
Bhatt's analysis confirms the requirement is optional in HSDS, and the
existing model offers acceptable workarounds for organizations whose
contact relationship is service-wide. Declining avoids introducing a
purpose-built extension that may be obsoleted by an R5 migration to
`ExtendedContactDetail`. It is the lowest-risk option for the IG's
publication timeline.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest fit for the actual state of the ticket: the
WG already discussed the change, accepted Chirag Bhatt's analysis, and
explicitly re-opened the ticket "post STU1 publication" — i.e., for action
in the next IG cycle. Bhatt's recommendation in the comment thread is
specifically to add an extension on `HealthcareService`, mirroring
`Organization.contact`, and that is what Disposition A operationalizes.
Disposition B is conceptually attractive — `ExtendedContactDetail` is the
right long-term shape — but it expands the scope of the change well beyond
what the ticket asks for and beyond what the WG signaled it had agreed to
do, and risks destabilizing existing `Organization`-based contact usage in
the IG. Disposition C ignores the explicit re-open and the prior WG
analysis, and leaves the documented HSDS alignment gap unaddressed when a
low-cost, reviewer-endorsed remedy already exists. Adopt A; consider
flagging the `ExtendedContactDetail` direction as a follow-up tracker for
the IG's R5 work.
