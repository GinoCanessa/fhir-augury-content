## Summary

The ticket flags that the upcoming Open Referral **HSDS 3.0** release introduces several
backwards-incompatible changes to the source data model that the HL7 *Human Services
Directory (FHIR)* IG maps from, and asks the work group to review the IG's HSDS-to-FHIR
mappings for alignment. The reporter (Greg Bloom, the maintainer of HSDS at Open
Referral) walks through the specific HSDS 3.0 changes — merging `postal_address` /
`physical_address` into a single `address`, renaming `service_attributes` and
`other_attributes` to `attributes`, dropping the `eligibility` table in favour of
attributes, and deprecating `Organization.tax_id` in favour of a richer
`organization_identifier` table — and notes for each one whether the impact is on
mappings only or on FHIR-side definitions. The ticket also previews two additional
HSDS 3.0 deliverables that affect the IG's authoring strategy: a standardised JSON
representation / API specification (which could simplify how mappings are written), and
a planned HSDS profile that constrains HSDS to the FHIR-mapped fields and aligns its
taxonomies to the IG's value sets, with extensions filling any gaps.

The single cross-reference surfaced by Augury is the FHIR artifact
**`Organization.identifier`**, which is the natural FHIR-side landing place for the new
HSDS `organization_identifier` table that replaces `Organization.tax_id`. There are no
linked Jira tickets, no Zulip discussions, and no GitHub items associated with this
ticket in the indexed data, so the disposition discussion has to be driven from the
ticket text itself plus the publicly-known HSDS 3.0 change list. Lynn Laakso's most
recent comment (2023-10-04) re-opened the ticket post-STU1 publication after a previous
"Considered for Future Use" resolution, signalling that the work group now expects this
work to be picked up against HSDS 3.0 once it is published.

## Details

**Description:**

There are backwards incompatible changes in the upcoming HSDS 3.0.

Here is a list of the changes that will affect this IG, with notes on their
consequences:

- Merging of **postal_address** and **physical_address** to **address**
  - This should have very little effect on the IG itself as the FHIR representation
    already merges these. This will only affect the mappings, not the fields or
    definitions in the IG.
- **service_attributes and other_attributes** renamed to **attributes**
  - This should also have little effect and only affect the mappings.
- Removal of the **eligibility** table in preference to using attributes.
  - This is currently not mapped in the IG so should have very little effect.
- **Organization.tax_id** is being deprecated but being replaced with new
  **organization_identifier** table

- New table could supply more fields to the FHIR Organization.identifier section.
- Should only affect mapping

- A few small field name changes
- This should only affect the mappings also.

HSDS 3.0 will have a standardised JSON representation and API specification. This will
be a recommended way to publish HSDS in future. Here is an example of a service object
with all its fields
<https://raw.githubusercontent.com/openreferral/specification/3.0-dev/examples/service_full.json>.
Having the JSON representation could make the mappings easier and clearer as you would
not have to specify join conditions. We are, however, keeping the datapackage / database
schemas as recognized representation formats, so specifying the mappings like they
currently are should not be an issue.

Open Referral also aims to produce a profile of HSDS 3.0 that would help alignment with
FHIR and make the mappings simpler to implement. This will be done by containing the
fields in HSDS to the ones mapped, and constraining the taxonomies to map directly to
the FHIR Value Sets. Also for some cases where there are gaps in HSDS, extensions could
be made.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt HSDS 3.0 as the source-of-record version that the *Human Services Directory
(FHIR)* IG maps to in its next published version. Concretely, in the IG sources:

1. **Mappings** — Update every HSDS-source mapping in the IG's profiles to refer to
   HSDS 3.0 element names: collapse `postal_address` / `physical_address` mappings to
   the merged HSDS `address` table; rename `service_attributes` / `other_attributes`
   references to `attributes`; remove any residual references to the dropped
   `eligibility` table; and replace `Organization.tax_id` mappings with mappings from
   the new HSDS `organization_identifier` table to FHIR `Organization.identifier`,
   exposing any additional fields the new table provides (e.g., issuing authority /
   identifier scheme) via `Organization.identifier.system`, `.type`, and `.assigner`
   slices as appropriate.
2. **Profiles / value sets** — Where HSDS 3.0 introduces fields the current IG does not
   yet surface (notably the richer organization-identifier metadata), open the relevant
   `Organization.identifier` slice / extension on the IG's `Organization` profile so
   implementers can carry the new HSDS data without loss.
3. **Narrative** — Update the IG's HSDS-version statement, mapping pages, and any
   examples that quote HSDS field names (e.g., physical/postal address splits) to match
   3.0, and link to the HSDS 3.0 JSON example called out in the ticket as a canonical
   payload reference.
4. **Versioning** — Publish the change as a new STU update of the IG, bumping the
   declared HSDS-version dependency.

#### Justification

The reporter is the HSDS maintainer; their own analysis is that all but the
Organization-identifier change are mapping-only, and Lynn Laakso's 2023-10-04 comment
explicitly framed this as the work the group expected to do post-STU1 once HSDS 3.0
shipped. The IG's value proposition is "FHIR ↔ HSDS interoperability"; continuing to
map to a deprecated HSDS version directly undermines that. Aligning to 3.0 in a single
IG release also lets the work group pick up the richer `organization_identifier`
information in `Organization.identifier` in the same pass, keeping the mapping coherent
rather than spread across two releases.

---

### Disposition B: Alternative Approach

#### Proposal

Treat HSDS 2.x and HSDS 3.0 as **side-by-side mapping targets** in the IG instead of
performing a hard cutover:

1. Keep the existing HSDS-2.x mapping column in each profile's mappings table, and add
   a parallel **HSDS 3.0** mapping (using the
   `http://docs.openreferral.org/en/3.0/hsds/` namespace, or whatever URI HSDS 3.0
   ultimately publishes) so implementers on either version of HSDS can use the IG.
2. Address only the *FHIR-side* changes that HSDS 3.0 actually requires — i.e., extend
   `Organization.identifier` slicing / add an extension to carry the additional fields
   from the new HSDS `organization_identifier` table — without disturbing the rest of
   the profiles.
3. Defer publishing an HSDS-3.0 profile alignment (the second deliverable Open Referral
   describes in the ticket) to a later cycle, but explicitly point at the upstream HSDS
   3.0 profile from the IG's narrative once it exists.
4. Add a clear *"HSDS version selection"* page in the IG explaining which HSDS source
   version each implementer guide page applies to, and call out the changed/removed
   tables (`postal_address`/`physical_address` merge, `eligibility` removal,
   `service_attributes`/`other_attributes` rename, `tax_id` deprecation).

#### Justification

There is a real population of existing implementers running HSDS 2.x today; a clean
cutover (Disposition A) silently invalidates their mappings. The HSDS 3.0 changes are
also overwhelmingly source-side renames that don't change the FHIR shape — exactly the
case where carrying both `mapping` rows is cheap and doubles the IG's reach. This
approach also de-risks the schedule: the only FHIR-side work that *must* land in the
next release is the `Organization.identifier` enrichment for the new
`organization_identifier` table, which is small and well-scoped, while the larger
narrative/profile alignment with the upstream HSDS 3.0 profile can follow once Open
Referral has finalised it.

---

### Disposition C: Decline

#### Proposal

Decline to make any IG-level changes in response to HSDS 3.0 at this time. Document in
the IG that the current version maps to HSDS 2.x, note that HSDS 3.0 is in development,
and revisit only after HSDS 3.0 is published as a stable release with the planned
"FHIR-aligned profile of HSDS 3.0" that Open Referral has committed to producing. In the
meantime, close this ticket as *Considered for Future Use* (mirroring the original
2023-02-06 disposition that was later reverted).

#### Justification

The substantive work this ticket asks for — re-mapping every HSDS reference in the IG
and possibly adding new `Organization.identifier` material — only pays off once HSDS 3.0
itself is stable and Open Referral's promised FHIR-aligned HSDS 3.0 profile exists,
because the latter explicitly constrains HSDS to the IG's mapped fields and value sets
and so dictates the shape of the mappings. Doing the alignment work now, against the
`3.0-dev` JSON example linked in the ticket, risks rework when the upstream profile
lands. That said, this disposition was already tried (and reverted by the work group on
2023-10-04 with "re-open post STU1 publication"), so it is preserved here only for
completeness.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures everything the ticket actually demands — picking up the FHIR-side
benefit of the new `organization_identifier` table on `Organization.identifier`, and
acknowledging the renamed/merged HSDS tables in the IG's mappings — while protecting
existing HSDS 2.x implementers, who would be silently broken by a hard cutover
(Disposition A). It is also the disposition that survives best if the upstream HSDS 3.0
profile shifts during finalisation: a parallel HSDS 3.0 mapping column can be revised
without touching any FHIR-side cardinality or slicing decisions, whereas Disposition A
bakes 3.0-dev assumptions into the published profiles. Disposition C is not viable
because the work group has already explicitly reverted that path post-STU1; it is listed
only so the trade-off is visible. The work group should size the FHIR-side work in
Disposition B (the `Organization.identifier` enrichment) first, since that is the only
piece that can't be deferred without losing data fidelity, and let the rest of the HSDS
3.0 alignment land incrementally as the upstream HSDS 3.0 profile stabilises.
