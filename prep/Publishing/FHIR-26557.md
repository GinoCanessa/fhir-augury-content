## Summary

Jose Costa-Teixeira reports that the published page
`http://hl7.org/fhir/ehrsrle/auditevent-ehrsrle.html` (the AuditEvent profile
in the EHR-System Record Lifecycle Events module) renders its section heading
numbers as literal `?` characters instead of the expected dotted numbering
(e.g., `1.2.3`). The reporter spot-checked other profile pages in the same
specification and they appeared correct, so the defect may be confined to this
single page (or to a small set of pages with the same authoring pattern). No
cross-referenced Jira tickets were surfaced, so the ticket stands on its own
as a Publishing-tooling defect against the FHIR Core build of the EHR-SRLE
content.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a Publishing-tooling bug against the FHIR Core / EHR-SRLE
build pipeline. Reproduce the broken rendering on the current build of
`ehrsrle/auditevent-ehrsrle.html`, identify the source of the `?` characters
in the heading-number substitution (most likely a missing or unresolved
section/anchor key in the publisher's heading-number table for that page),
and patch the publisher (or the page's authoring template) so that the
generated headings carry their dotted numeric labels. Re-publish the IG and
confirm visually.

#### Justification

The reporter described an unambiguous rendering defect on a published HL7
page, and Grahame Grieve already accepted on Zulip that this is a Publishing
work-group concern. The fix is mechanical (a tooling/template patch, not a
normative spec change), so the cost is low and the value to readers of the
EHR-SRLE module is direct. There is no design controversy to weigh.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than patching only this one page, audit the publisher's heading-number
generation across **all** profile/IG pages produced by the same template path
(narrative-driven profile pages such as `*-ehrsrle.html` and analogous pages
in other modules). Add a publisher self-check that fails the build (or emits
a QA warning) when a generated heading contains an unresolved numbering
placeholder (e.g., a `?` where a digit is expected). Apply a single fix that
both repairs `auditevent-ehrsrle.html` and prevents recurrence on sibling
pages.

#### Justification

The reporter explicitly noted he wasn't sure whether the bug was confined to
this page. A targeted fix risks leaving silently-broken headings on other
profile pages that the reporter didn't check (or that exist in IGs published
later). A QA gate in the publisher is consistent with the way other rendering
defects (broken xrefs, missing examples) are caught today, and it is a
durable solution rather than a one-page hotfix.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Reproducible / No Longer Relevant**. The ticket is
five years old, the EHR-SRLE module has been re-published many times since
2020, and the publisher's heading-numbering machinery has been substantially
overhauled in the intervening releases. If the page renders correctly in the
current CI build, mark the ticket Resolved with resolution
"Persuasive with Mod" → "no change needed; verified fixed in current build"
and link to the current rendering for evidence.

#### Justification

The Jira ticket has had zero activity since the day it was filed, no comments,
no linked PRs, and the publisher pipeline has been rewritten in ways that
make it likely the underlying cause no longer exists. Spending workgroup
cycles on a five-year-old low-priority cosmetic bug is hard to justify
unless reproduction confirms it is still live. Declining (after verification)
clears the backlog without losing information — the Zulip thread and ticket
text remain searchable if the symptom returns.

---

### Recommendation

**Recommended disposition:** C, contingent on a quick reproduction check.

The ticket is a low-priority cosmetic Publishing defect from March 2020 with
no comments, no linked PRs, no follow-up Zulip activity, and an
already-Triaged status that has not moved in five years. The publisher and
the EHR-SRLE IG have both been re-published many times since then, and the
heading-numbering pipeline in `fhir-ig-publisher` has been reworked. The
correct first action is to load the current
`http://hl7.org/fhir/ehrsrle/auditevent-ehrsrle.html` (and the latest CI
build) and check whether the `?` characters are still present. If they are
gone, close as Disposition C with a verification note. If they are still
present, escalate to Disposition A (single-page fix) or Disposition B
(publisher-wide QA gate) — Disposition B is the better choice if any sibling
page exhibits the same defect, otherwise Disposition A is sufficient.
