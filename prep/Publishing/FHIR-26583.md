## Summary

The reporter, Harold Solbrig, observes that the hyperlinks the FHIR Core
publication exposes for the *raw* Shape Expressions (ShEx) definitions of
resources are broken. The problem is described as affecting **all
artifacts**, not only `Patient`, which implies the issue is in the
publication template/link-generation logic rather than in the ShEx file
for any single resource. No cross-referenced Jira tickets, Zulip threads,
or GitHub items were located by the augury cross-reference index, so the
request stands on its own as a defect report against the published
specification's downloads/links section for ShEx artifacts.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a publication-tooling defect against FHIR Core. Audit the
per-resource "Downloads" / "Detailed Descriptions" template that emits
the link to the raw ShEx file for each resource (the link adjacent to
the existing XML / JSON / Turtle raw-schema links). Confirm the actual
file path/extension produced by the publisher for ShEx (`.shex`) and
update the template's URL pattern so that the rendered `<a href="…">`
resolves to the file that the build actually writes to the output
directory. Re-publish a representative sample of resources (`Patient`,
`Observation`, `Bundle`, plus a primitive-heavy resource such as
`Account`) and verify that the previously-404 raw ShEx links now return
the expected ShEx text.

#### Justification

The ticket reports a verifiable, deterministic defect (a broken
hyperlink) that affects every resource page in the published spec. ShEx
is one of the formal definition formats FHIR advertises alongside XML
Schema, JSON Schema and Turtle, so a 404 on the "raw" link is a
straightforward correctness issue for the publication and is in scope
for the Publishing work group. Fixing the link template restores the
documented capability without changing the spec semantics or the ShEx
content itself, so the change is low-risk and reversible.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only repairing the existing per-resource link, replace the
current pattern with a single, build-validated downloads block: (1)
generate the ShEx file under a stable canonical path (e.g.
`<resource>.shex`) collocated with the other raw schema files, (2) emit
the link from the same template that emits the XML / JSON / Turtle raw
links so all four are produced and validated together, and (3) extend
the publisher's existing link-checker (or QA step) to include the raw
ShEx URL for every resource so that any future regression breaks the
build instead of shipping silently. If the project has effectively
de-emphasised the standalone ShEx artefacts, the alternative form of
this disposition is to **remove** the broken raw-ShEx links from the
per-resource pages (and any aggregate downloads page) and rely solely on
the consolidated `fhir.shex` bundle, with a short note pointing readers
there.

#### Justification

The reporter notes the breakage affects *all* artifacts, which is a
strong signal that the failure is structural rather than per-resource —
i.e. the link template and the build output disagree. Fixing only the
URL string (Disposition A) leaves the underlying coupling between the
publisher and the template intact, so the same class of defect can
recur the next time the ShEx output path or filename convention is
adjusted. Co-locating ShEx with the other schema artefacts and adding
it to the link-checker prevents recurrence; alternatively, consolidating
to the bundle reduces the surface area the publisher has to keep in
sync. Both forms address the underlying need (readers being able to get
to the ShEx for a resource) more durably than a one-shot URL fix.

---

### Disposition C: Decline

#### Proposal

Mark "Not Persuasive – Already Addressed" on the basis that the ShEx
artefacts continue to be published as a single consolidated bundle
(`fhir.shex`) reachable from the Downloads page, and that the FHIR ShEx
tooling (e.g. the `fhir-shex` ecosystem) consumes the bundle rather
than per-resource files. Recommend the reporter use the bundle and
treat the per-resource raw-ShEx links as obsolete; remove the dead
links in a future publication pass as a documentation tidy-up rather
than as a normative change.

#### Justification

If the per-resource raw ShEx files are no longer a supported output of
the build, the broken links are a symptom of a deliberate scoping
decision rather than a regression to fix. In that case, repairing the
links would imply re-introducing per-resource ShEx generation and
maintaining it indefinitely, which is a non-trivial publishing burden
for an artefact format with limited downstream consumers. Declining and
documenting the bundle as the canonical access path would be defensible
on cost-vs-benefit grounds.

---

### Recommendation

**Recommended disposition:** B

The defect as reported is real and worth fixing (so a flat "Decline" is
hard to justify against a Publishing-owned link that simply doesn't
resolve), but a pure URL patch (Disposition A) leaves the publisher and
the link template silently coupled — which is precisely the situation
that produced the original break and that the reporter implicitly
flagged by noting the breakage is universal across resources.
Disposition B fixes the immediate symptom *and* puts the raw ShEx link
on the same generated-and-link-checked path as the XML / JSON / Turtle
raw links, so the next time the ShEx output convention shifts the build
will fail loudly instead of shipping 404s. If the work group concludes
that per-resource raw ShEx is no longer worth maintaining, the
"remove + point at the bundle" variant of B is preferable to C because
it leaves the spec internally consistent rather than carrying known-bad
links until a future cleanup pass.
