## Summary

The ticket reports a publishing-tooling defect on the `ExampleServicePlaceCodes`
value set page. In the section that lists the resource elements bound to the
value set, the bullet for `Claim.item.location[x]` is hyperlinked to an anchor
that contains the literal `[x` characters
(`http://build.fhir.org/claim-definitions.html#Claim.item.location[x`),
producing a broken in-page link. The expected anchor for a choice-type element
is the `_x_`-suffixed form
(`http://build.fhir.org/claim-definitions.html#Claim.item.location_x_`),
matching the anchors actually emitted by the definitions page generator.

The reporter (Nick Radov) explicitly identifies this as a publishing tooling
issue rather than a content defect: the data model entry for
`Claim.item.location[x]` is correct, but the link generator that builds the
"used by" list on value-set pages is failing to translate `[x]` into the
canonical `_x_` anchor suffix used elsewhere in the IG publisher output.
Cross-references confirm only two FHIR mentions (`Claim.item.location` and
`Claim.item`); no related Jira tickets, Zulip threads, GitHub items, or
Confluence pages were found.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Fix the value-set page generator so that, when it emits the "used by" list
of elements bound to a value set (the section in which
`Claim.item.location[x]` appears on the `ExampleServicePlaceCodes` page),
the in-page anchor link for any element whose path ends in `[x]` is rewritten
to the canonical `_x_` suffix used by the definitions page (e.g.
`#Claim.item.location_x_` rather than `#Claim.item.location[x]`). The fix
should live in the publishing tooling that renders value-set pages — the
same code path that generates these bullet items — and should be applied
uniformly to every choice-type element link, not only `Claim.item.location[x]`.
Verification: re-render `valueset-example-service-place.html` (or the
current equivalent) and confirm that every `[x]` bullet in the "used by"
list resolves to a live anchor on the corresponding `*-definitions.html`
page.

#### Justification

The reporter is correct that this is a tooling bug, not a content bug:
the definitions pages already publish anchors in the `_x_` form, so the
mismatch is purely on the link-generation side of the value-set page
renderer. Fixing it at the source eliminates an entire class of broken
links across every value set whose binding list includes a choice-type
element, with no spec-text changes and no impact on downstream
implementers. It is also consistent with how the rest of the publisher
escapes `[x]` in element anchors, so the change is a small, local
normalization rather than a new convention.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than special-casing `[x]` → `_x_` rewriting in the value-set page
generator, introduce (or reuse) a single shared helper in the publishing
tooling that converts an element path to its canonical anchor id, and route
every page generator that emits element anchor links through it — value-set
"used by" lists, code-system pages, profile differential/snapshot tables,
search-parameter pages, etc. Then fix the originally reported symptom by
making the value-set page generator call that helper. As part of the same
change, add a publishing-time QA check (broken-anchor scan over generated
HTML) that fails the build when an internal `*-definitions.html#…` link
does not resolve to an emitted anchor.

#### Justification

The bug is almost certainly an instance of a class: anywhere the publisher
constructs `…-definitions.html#<elementPath>` by string concatenation it is
exposed to the same `[x]` mishandling, plus other edge cases (slicing
discriminators, names with dots, etc.). Centralizing the conversion and
adding a build-time link check eliminates this and future siblings in one
move, prevents regression, and gives the Publishing work group an
ongoing signal instead of relying on ad-hoc bug reports. The cost is a
slightly larger change with broader review surface, but the change is
mechanical and the QA check pays for itself the first time it catches a
regression. Trade-off vs. Disposition A: more work now, materially less
ticket churn later.

---

### Disposition C: Decline

#### Proposal

Close as "won't fix" on the grounds that (a) the URL in the report
points at `build.fhir.org` (a continuous build), (b) the ticket has sat
in `Triaged` since 2021 with no comments, no Zulip discussion, no PRs, and
no other linked tickets, and (c) the affected anchors may already render
correctly in current builds, making this a stale report against an
ephemeral artifact.

#### Justification

A defensible decline rests on verifying current behavior: if the present
build of the `ExampleServicePlaceCodes` page already produces `_x_`
anchors, the bug has been fixed incidentally and the ticket can be
closed as "Not Reproducible / Already Addressed". This avoids spending
Publishing-WG cycles on a four-year-old report that may have been
superseded by other tooling work. The risk is non-trivial: if the bug
still reproduces, declining leaves a known-broken link in the
specification, which is a poor outcome for a Technical Correction in
Core. Decline is therefore only safe after a quick reproduction check
on the current build.

---

### Recommendation

**Recommended disposition:** B

The reported symptom is small, but it is structurally a class of bug —
any publisher code path that builds a `*-definitions.html#…` URL by
concatenating an element path is susceptible to the same `[x]` mishandling
(and likely other edge cases). Centralizing element-path → anchor-id
conversion and adding a build-time broken-anchor check addresses the
specific report, eliminates siblings the work group has not yet noticed,
and gives Publishing an ongoing regression signal. Disposition A would
fix this exact bullet but leaves the same defect latent everywhere else;
Disposition C is only viable if a quick repro on the current build
shows the link already resolves, and even then it ignores the broader
class. If implementation bandwidth is the binding constraint, B can be
staged: ship A first as the minimal fix, then follow with the helper
consolidation and link checker as a tracked Publishing-tooling task.
