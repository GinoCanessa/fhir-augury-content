## Summary

Bryn Rhodes reports that the FHIR specification's tooling for documenting
`OperationDefinition` examples appears to support more than one example per
operation (per the build-time documentation at
`hl7-fhir.github.io/codesystem-operations.html`), but in practice the
publishing pipeline does not render multiple examples correctly. He cites the
`$evaluate` operation on `DecisionSupportServiceModule` as a case where he
authored multiple examples and the rendered output failed to separate them
visually. He further observes that even when multiple examples *do* render,
the current layout breaks the natural request/response pairing — all requests
are listed first, then all responses, instead of grouping each request with
its corresponding response. The ticket asks the Publishing work group to
enable (and document) authoring of multiple, correctly‑paired examples on
`OperationDefinition` pages. No cross‑referenced Jira tickets were found.

## Details

**Description:**

> The current mechanism for providing examples for OperationDefinitions only
> seems to allow one example to be provided. Looking at this page:
> https://hl7-fhir.github.io/codesystem-operations.html it seems to be
> possible, but when I try, I can't get the separation to come through (see
> the $evaluate operation definition in DecisionSupportServiceModule).
>
> In addition, this approach to multiple examples doesn't provide continuity
> between the request and response for each example (i.e. all the requests
> are listed, then all the responses).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR core publishing tooling that renders `OperationDefinition`
pages so that:

1. Authors can attach multiple, distinct examples to a single
   `OperationDefinition` (multiple example‑pair entries on the operation's
   intro page / spreadsheet definition), and each example renders as its own
   labeled block.
2. Each example block keeps its request and response paired together
   (Example 1: request → response; Example 2: request → response; …) instead
   of listing all requests followed by all responses.
3. The operations build documentation
   (`codesystem-operations.html` / equivalent guidance in the FHIR build) is
   updated to describe the supported authoring pattern for multiple examples
   so authors can reproduce it without reverse‑engineering Bryn's `$evaluate`
   case.

Concretely, this would touch the operation page generator in the FHIR Core
build (the publisher that renders `operation-<resource>-<name>.html`) plus
the operation intro/example fixtures that drive `DecisionSupportServiceModule
$evaluate` and similar pages.

#### Justification

The request is purely an authoring / publishing usability fix and aligns with
patterns already used elsewhere in the spec (e.g., resource example tabs that
preserve request/response or input/output continuity). Honoring request/
response continuity per example matches how implementers actually read
operation documentation — they need to see, for one scenario, what went in
and what came out. Bryn's report shows the *intent* to support multiple
examples already exists in the build's documentation page, so this is closing
a gap between documented behavior and rendered behavior rather than
introducing a new feature surface.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) extending the operation page renderer, treat
each operation example as a first‑class artifact:

- Standardize on storing each operation example as a `Bundle` (or paired
  `Parameters` request / `Parameters` response resources) under the spec's
  examples directory, with a naming convention such as
  `operation-<resource>-<name>-example-<n>.{json,xml}`.
- Have the `OperationDefinition` page generator discover all matching
  examples by name pattern and render one tabbed/collapsible block per
  example, each block showing request and response together with download
  links.
- Document this pattern once (in the operations build docs) so it applies
  uniformly to every operation in core and in IGs that reuse the build.

#### Justification

This avoids inventing operation‑specific markup in operation intro files and
instead leans on the same example‑resource discovery pattern the build
already uses for resource pages. It gives implementers downloadable,
runnable example payloads (a recurring community ask) as a side effect of
fixing the rendering, and it makes IG authors' lives easier because they
inherit the behavior automatically. The trade‑off vs. Disposition A is more
upfront work in the publisher and a one‑time migration of existing operation
examples to the new naming convention, but the result is more consistent and
reusable than patching the current single‑example path.

---

### Disposition C: Decline

#### Proposal

Close as **Won't Fix / Persuasive with Mod** on the grounds that the
underlying need — showing multiple worked examples for an operation — is
already achievable today by:

- Authoring the additional examples on the operation's intro/narrative page
  as inline code blocks paired by hand, or
- Linking out from the operation page to example `Bundle`s in the spec's
  examples directory.

Decline to invest in publisher changes given the ticket's age (2016), absence
of community discussion, and lack of a concrete implementer pain report
beyond the original author.

#### Justification

The ticket has sat in `Triaged` since 2016 with no comments, no Zulip
discussion, no linked tickets, and no GitHub activity. Publishing‑tool
changes carry real risk to every operation page in the spec and in
downstream IGs, and the workaround (inline narrative + linked example
resources) is functional even if visually inferior. Declining preserves
publisher stability and keeps Publishing capacity focused on tickets with
demonstrated, current implementer demand.

---

### Recommendation

**Recommended disposition:** B

The publishing tooling clearly fails to deliver what its own documentation
advertises (Disposition A's premise is sound), but the right fix is not to
patch the current single‑example path — it is to align operation examples
with how the rest of the spec already publishes example artifacts. Disposition
B fixes Bryn's request/response pairing complaint structurally, gives
implementers downloadable example payloads as a bonus, and produces a
pattern IG authors inherit for free. Disposition A leaves operation
examples as a one‑off rendering quirk; Disposition C ignores a real,
verifiable rendering bug. Recommend the work group accept the ticket with
Disposition B's approach and route the publisher change to the FHIR Core
build maintainers.
