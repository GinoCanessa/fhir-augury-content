## Summary

The ticket reports a broken hyperlink in the auto-generated "this Code
system is referenced in the content logical definition of the following
value sets" usage section on a CodeSystem page (the reporter copied the
list, which includes `DataAbsentReason`, the `Yes/No/Don't Know` example
value set, and `genderIdentity`). The `genderIdentity` link points to
`https://build.fhir.org/ig/HL7/fhir-extensions/valueset-gender-identity.html`,
which no longer resolves. The reporter notes the breakage exists in both
R5 and the CI build.

A subsequent comment from cooper.thompson (PA, May 2024 WGM, Tuesday Q4)
diagnoses the cause: Patient Administration moved the `gender-identity`
value set out of the FHIR Extensions Pack and into THO, where it now
lives at
`https://terminology.hl7.org/5.5.0/ValueSet-gender-identity.html`.
However, the back-references on the affected CodeSystem page (e.g.
`data-absent-reason`) still point at the old extensions-pack location,
producing the broken link. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this ticket by the augury
cross-reference index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a publication / cross-IG link-resolution defect against
the FHIR Extensions Pack and fix the back-reference so the
`genderIdentity` entry on the data-absent-reason (and any other
affected) CodeSystem usage section resolves correctly.

Concretely:

- In the FHIR Extensions Pack source (`HL7/fhir-extensions`), remove the
  retired `gender-identity` ValueSet artifact (or replace it with a
  redirect / deprecation stub) so the IG no longer claims to publish
  `valueset-gender-identity.html` at
  `build.fhir.org/ig/HL7/fhir-extensions/`.
- Ensure the now-canonical THO ValueSet
  (`http://terminology.hl7.org/ValueSet/gender-identity`, currently
  rendered at `terminology.hl7.org/5.5.0/ValueSet-gender-identity.html`)
  is the only artifact carrying that canonical URL, so back-reference
  generation in dependent IGs (R4 core, Extensions Pack) emits the THO
  link instead of the extensions-pack link.
- Re-publish the affected pages so the auto-generated "This Code system
  is referenced in the content logical definition of the following value
  sets" block on `data-absent-reason` (and `yes-no-dont-know` if it
  shares the same compose include) renders the THO URL.

#### Justification

The ticket is filed as a Technical Correction and the diagnosis from
cooper.thompson (PA, May 2024 WGM) is unambiguous: the value set has
already moved to THO, but the publication still has a stale duplicate
in the Extensions Pack that wins the back-reference. Fixing the source
of the duplicate is the minimal change that makes the rendered link
correct without touching the data-absent-reason CodeSystem itself, and
it aligns the published artifact set with the workgroup decision PA
already made. This sits squarely in THO's remit because the canonical
artifact now lives in the THO publication chain.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only deleting / redirecting the stale extensions-pack copy,
make the fix systematic at the publisher / cross-IG link level:

- Have the IG publisher honour the canonical-URL → publication-URL
  mapping from the THO package so any IG that includes a code system
  whose `useContext` / "used by" back-reference points at a value set
  whose canonical resolves into a different package emits the *resolved*
  URL (here: `terminology.hl7.org/.../ValueSet-gender-identity.html`),
  not the local guess.
- In parallel, in `HL7/fhir-extensions`, mark `gender-identity` as
  retired-from-this-IG with an explicit redirect note pointing to THO
  (so anyone landing on the old URL via cached search results gets a
  pointer instead of a 404).
- Audit other ValueSets that PA migrated out of the Extensions Pack into
  THO and apply the same treatment to avoid additional one-off broken
  back-references.

#### Justification

The literal fix in Disposition A clears this one symptom but does not
prevent the same class of bug recurring whenever an artifact is
relocated between an IG and THO — a common pattern for THO. Pushing the
fix into the publisher's link-resolution and into a small audit
addresses the underlying defect ("usage back-references hard-code the
local IG location") rather than chasing individual broken links. The
trade-off vs. A is more engineering effort and coordination with the IG
publisher team, but it is a better long-term posture for a workgroup
whose job is to be the canonical source of terminology that other
specs cite.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive – out of scope for THO / wrong queue":
the broken anchor appears on R4 core CodeSystem pages
(`hl7.org/fhir/R4/valueset-data-absent-reason.html`,
`valueset-example-yesnodontknow.html`) which are frozen normative
publications, and the offending source artifact lives in the
fhir-extensions IG (FHIR Infrastructure / Extensions). Recommend the
reporter re-file against the FHIR Extensions Pack maintainers (or
against the IG publisher) and decline as a THO action.

#### Justification

THO publishes the now-canonical `gender-identity` ValueSet correctly at
`terminology.hl7.org`; the broken link is on pages THO does not own and
is generated by tooling THO does not maintain. Declining is defensible
on jurisdiction grounds and avoids THO taking on a publication-pipeline
fix that belongs elsewhere. The downside is that the user-visible
broken link persists until another workgroup picks it up, which is
poor stewardship for a "Technical Correction" that has already been
diagnosed.

---

### Recommendation

**Recommended disposition:** A

The ticket is a clear, narrowly-scoped Technical Correction, the cause
is already diagnosed in the comment thread (PA migrated the value set
to THO; the Extensions Pack copy is stale and is winning the
back-reference), and the fix is mechanical: remove or redirect the
retired `gender-identity` artifact in `HL7/fhir-extensions` so
republication causes the dependent CodeSystem pages to point at the
THO URL. Disposition B is the right *direction* but is a publisher /
tooling change that should be tracked as a separate follow-up rather
than gating this correction; if the workgroup wants, it can accept A
now and spawn a follow-up ticket against the IG publisher to implement
the systemic fix from B. Disposition C is technically defensible on
turf grounds but leaves a known broken link in published material when
THO is the workgroup that actually owns the new canonical location and
is best placed to coordinate the cleanup with PA and the Extensions
Pack maintainers.
