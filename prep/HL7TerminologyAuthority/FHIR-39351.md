## Summary

HL7 Germany has adjusted the canonical URL it uses to identify the German
modification of ICD-10 ("ICD-10 GM"). The reporter asks the FHIR core
specification (specifically the "Using ICD with FHIR" page, `icd.html`) to
update the canonical for ICD-10 GM from the legacy DIMDI-based URI to
`http://fhir.de/CodeSystem/bfarm/icd-10-gm`, replace the broken/legacy
reference link `http://wiki.hl7.de/index.php?title=Kodesysteme` with
`https://ig.fhir.de/basisprofile-de/stable/Ressourcen-Condition-ICD-10-GM.html`,
and adjust the worked examples in sections 4.3.11.3.2 and 4.3.11.3.3 to
match the new canonical.

The change is motivated by a breaking change HL7 Germany already shipped
in v1.0.0 of the German base profiles (which restructures how
post-coordinated codes — asterisk/dagger and "!" codes — are represented
across linked Condition resources). HL7 Germany used that opportunity to
move the canonical out from under the renamed source organisation
(DIMDI → BfArM) so that the canonical is stable going forward.

There are no explicitly linked Jira tickets and the cross-reference index
surfaces no related Jira tickets. The ticket has been bouncing between
Vocabulary/Terminology Infrastructure (Vocab WG, now THO/Terminology
Infrastructure) and the requester since 2023: the WG's position has been
that because the "Using ICD with FHIR" page has moved to THO
(`terminology.hl7.org/ICD.html`), the canonical change must be processed
through the UTG / UP Jira project rather than as a FHIR core change. The
most recent comment (jsnell, 2025-10-20) signals the WG is preparing to
close the ticket and asks the requester to file a UP ticket if they want
the alternate identifier registered in THO.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR core "Using ICD with FHIR" page (`fhir/icd.html`,
section 4.3.11) so that the row identifying ICD-10 GM in the canonical
table cites `http://fhir.de/CodeSystem/bfarm/icd-10-gm` as the
canonical, marks the legacy DIMDI-based URL as deprecated, replaces the
reference link `http://wiki.hl7.de/index.php?title=Kodesysteme` with
`https://ig.fhir.de/basisprofile-de/stable/Ressourcen-Condition-ICD-10-GM.html`,
and updates the worked examples in sections 4.3.11.3.2 and 4.3.11.3.3 to
use the new canonical. Coordinate the change with Terminology
Infrastructure so the equivalent THO page (`terminology.hl7.org/ICD.html`)
and any THO `NamingSystem` / external-terminology metadata record for
ICD-10-GM are updated in lockstep.

#### Justification

This is the literal request and matches what HL7 Germany has already
shipped in the stable basisprofile-de v1.0.0. The "Using ICD with FHIR"
page is documentary — it tells implementers which canonical is in real
use for a given national variant — so it should follow national
authorities (HL7 Germany) on which canonical they actually publish.
Marking the old canonical as deprecated rather than silently replacing
it preserves backward compatibility for resources already in production
that carry the DIMDI-style URI.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the canonical recorded by the FHIR core spec; instead
register the new HL7-Germany canonical
(`http://fhir.de/CodeSystem/bfarm/icd-10-gm`) as an **alternate
identifier** for the ICD-10-GM code system in THO, via a UP-project
ticket (HL7 Terminology Authority's UTG process). Add a `NamingSystem`
entry in THO that lists both URIs, marks one as preferred, and links to
the German basisprofile-de documentation. Update `terminology.hl7.org/ICD.html`
(not `fhir/icd.html`) to surface the new canonical and the
deprecation note for the legacy URI. The FHIR core "Using ICD with
FHIR" page can then either redirect to the THO page (consistent with
how other external terminology pages have been migrated) or carry only
a brief pointer.

This is essentially the path most recently proposed in the WG comments
(jsnell, 2025-10-20; carmela_couderc, 2023-03-20): close FHIR-39351 in
favour of a UP ticket that drives the change through THO.

#### Justification

Multiple WG members have made the procedural point that ownership of
the "Using ICD with FHIR" content has moved from FHIR core to THO, and
that canonicals for external code systems are governed via UTG. Routing
the change through UP gets the right governance (HL7 Terminology
Authority) and the right artefact (a `NamingSystem` / THO metadata
record) so that *both* historical and new canonicals are
machine-discoverable as equivalent. It also addresses Michael Lawley's
concern about churn: if the source organisation is renamed again (DIMDI
→ BfArM → ?), the alternate-identifier mechanism makes it straightforward
to add another preferred URI without breaking existing data, which a
single canonical edit on `icd.html` does not. The trade-off vs.
Disposition A is that German implementers do not get the change until a
UP ticket lands and the THO record is published, which is slower than a
direct edit to `icd.html`.

---

### Disposition C: Decline

#### Proposal

Decline the change to the FHIR core spec. Close FHIR-39351 as "Not
Persuasive — Out of Scope (handled via UTG/THO)" with a pointer to the
UP project, and rely on the existing `terminology.hl7.org/ICD.html`
page (which already documents ICD-10 variants) to carry any future
changes. Do not edit `fhir/icd.html`.

#### Justification

The WG comments establish two grounds for declining at the FHIR core
level: (1) jurisdiction — the authoritative content has moved to THO,
so a FHIR core edit would either be redundant with or diverge from
THO; (2) governance — canonical changes for external code systems are
the UTG process's responsibility, and bypassing it sets a precedent
that any national affiliate can have a core-spec canonical changed via
a FHIR-project tracker. Michael Lawley's churn concern is also
unresolved (`bfarm` is itself an organisation-name fragment that could
change again). Declining in favour of the UP path is not refusing the
substantive request — it is refusing to satisfy it through this
specific change vehicle.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the cleanest expression of what the WG has consistently
asked for since January 2023, and it is the only option that addresses
both the substantive German need (have the new canonical formally
recognised) and the governance concerns raised on the ticket and on
Zulip (UTG ownership, deprecation handling, churn risk from
organisation-renamed canonicals). It avoids the precedent problem of
Disposition C (declining outright would leave the new canonical
undocumented anywhere in HL7-curated material and frustrate the
requester's clearly reasonable goal) and the silent-divergence risk of
Disposition A (changing `fhir/icd.html` while THO is the system of
record for "Using ICD with FHIR" would create two competing pages).
Concretely: close FHIR-39351 with a pointer to a new UP ticket; on the
THO side, register `http://fhir.de/CodeSystem/bfarm/icd-10-gm` as an
alternate identifier alongside the legacy DIMDI URI, mark the legacy
one as deprecated, and update `terminology.hl7.org/ICD.html` (and any
table link) to point to the German basisprofile-de Condition page. If
the WG wants the FHIR core "Using ICD with FHIR" page to stop being a
divergent source of truth, it can additionally fold a redirect/short
stub into the next FHIR core release, but that is a separate concern
from this ticket.
