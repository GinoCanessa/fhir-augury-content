## Summary

The reporter (Anthony "Tony" Julian) observes that the FHIR core ValueSet
`message-reason-encounter` (canonical
`http://hl7.org/fhir/ValueSet/message-reason-encounter`) currently carries a
maturity / standards-status of **Draft**, and proposes that it be reclassified
to **Informative**. The change would lower expectations of forward
compatibility and active maintenance for this value set: it would be published
as an informative artifact rather than a draft normative-track one, which is
appropriate when the artifact is documented for reference but is not under
active standardization.

The request sits at the intersection of terminology stewardship (HTA owns the
status of the published ValueSet) and the Infrastructure & Messaging work
group's stewardship of `MessageHeader`, which is the consuming resource for
this value set. No formal cross-references to other Jira tickets were returned
by the cross-reference index, but a related and resolved I&M ticket,
[FHIR-43807](https://jira.hl7.org/browse/FHIR-43807) ("Resource Message
Header"), is materially relevant: it deals with the unbound `MessageHeader.event`
slot, was resolved Persuasive-with-Modification to bind it to
`http://terminology.hl7.org/ValueSet/v2-0003` and to document
`MessageHeader.reason` (which is what `message-reason-encounter` binds to) as
an intentionally exemplar / unbound element. That resolution directly touches
the role and future of the value set this ticket is about.

## Related Zulip Discussions

### [fhir-messages > MessageHeader.event](https://chat.fhir.org/#narrow/stream/fhir-messages/topic/MessageHeader.event/near/540070070)

A September 2025 thread on what to bind `MessageHeader.event` (and, by
extension, `MessageHeader.reason` / `message-reason-encounter`) to. Tony
Julian opened by noting `MessageHeader.event` is currently only an Example
binding and asked whether `MessageHeader.reason` should still point at
`http://hl7.org/fhir/ValueSet/message-reason-encounter`. He explicitly called
out that `message-reason-encounter` "(although optional) is inadequate for
many use cases." Cooper Thompson argued that `MessageHeader.event` and
`SubscriptionTopic.trigger.event` should share a value set and that anything
v2-flavored is widely used in practice. Lloyd McKenzie agreed anything
tighter than Preferred would be a breaking change, was uneasy about codifying
v2 events in a FHIR core code system (because some are v2-paradigm-specific),
and confirmed the `reason` binding can be removed without breaking existing
instances. Tony then asked whether `reason` could be unbound; Lloyd answered
yes. Craig Newman noted that I&M had already approved FHIR-43807 (apply the
v2-0003 binding for `event` and document `reason` as exemplar / unbound) but
that it had not yet been applied. Kevin Mayfield and René Spronk added
implementer experience favoring a v2-aligned event vocabulary in practice. No
firm decision was reached about `message-reason-encounter` itself; the
direction is to deprecate or weaken its role rather than mature it.

Participants: Anthony (Tony) Julian, Cooper Thompson, Lloyd McKenzie, René
Spronk, Craig Newman, Kevin Mayfield, Brian Frankl.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the maturity / standards status of
`http://hl7.org/fhir/ValueSet/message-reason-encounter` from **Draft** to
**Informative** in FHIR Core. Concretely, update the `Standards-Status`
extension on the ValueSet definition (and the corresponding metadata in the
HL7 publishing tooling) from `draft` to `informative`, and update the
rendered "Standards Status" badge on `valueset-message-reason-encounter.html`
accordingly. Leave the ValueSet `status` element (publication status) as
`active` and leave the canonical URL, OID, and contents unchanged. Coordinate
with Infrastructure & Messaging (the steward of `MessageHeader.reason`, which
is the only core consumer) to confirm the reclassification before publication.

#### Justification

The reporter (and steward of the surrounding messaging space) is signaling
that this value set is not on a path toward normative maturation — the very
recent Zulip threads and the FHIR-43807 resolution point in the opposite
direction (treating `MessageHeader.reason` as exemplar / intentionally
unbound, and treating the value set as documentation rather than a target
for refinement). FHIR's standards-status model uses **Informative** precisely
for artifacts that are published as reference material but are not under
active standards development, which matches the actual posture HTA / I&M are
taking toward this value set. Reclassifying as Informative communicates that
posture honestly to implementers without removing the artifact (preserving
the canonical URL and any existing implementer references).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mark it Informative, **deprecate** `message-reason-encounter` and
weaken / replace its consumer binding. Specifically:

1. Set the ValueSet `status` to `retired` (or add the `Standards-Status`
   value `deprecated`) and add an explanatory note pointing implementers at
   the chosen replacement.
2. In coordination with I&M, change the binding on `MessageHeader.reason`
   from `message-reason-encounter` to either (a) no binding (intentionally
   unbound, mirroring the FHIR-43807 treatment of `MessageHeader.event`
   prior to v2-0003), or (b) an Example binding to the same value set used
   for `MessageHeader.event` (per Cooper Thompson's request that the two
   elements share a value set).
3. Keep the canonical URL resolvable for backward compatibility.

#### Justification

The Zulip discussion makes clear that the existing `message-reason-encounter`
value set is considered "inadequate for many use cases" by its own steward,
and that the I&M direction is to either leave the slot exemplar/unbound or
to align it with whatever value set ends up bound to `MessageHeader.event`.
"Informative" is honest about the artifact's status but does not address the
underlying problem that it is still the published Example binding for
`MessageHeader.reason`. Deprecating the value set and aligning the binding
removes the implicit endorsement that the Informative label leaves intact
(implementers tend to keep using whatever core publishes), and is more
consistent with FHIR-43807's resolution. The trade-off vs. A is a larger
coordinated change that requires I&M sign-off and a cross-resource review of
who uses the value set.

---

### Disposition C: Decline

#### Proposal

Decline the reclassification and leave the ValueSet at **Draft**, on the
grounds that the right resolution is being negotiated as part of the
`MessageHeader.event` / `MessageHeader.reason` binding work tracked under
[FHIR-43807](https://jira.hl7.org/browse/FHIR-43807) and the related Zulip
threads, and that the standards-status of the value set should be re-evaluated
once that work is applied (which the 2026-01-28 comment on FHIR-43807 notes
is still pending). Close FHIR-43438 with a pointer to FHIR-43807 and to the
`fhir-messages > MessageHeader.event` Zulip thread.

#### Justification

The community has not converged on whether the value set should be promoted,
deprecated, replaced, or simply unbound. Promoting it to Informative now
locks in a maturity statement that may have to be revised again once
FHIR-43807 is actually applied and once I&M and HTA decide whether
`MessageHeader.reason` should share a binding with `MessageHeader.event`. A
decline-with-pointer keeps the ticket's concern visible without committing to
a label that could shortly be at odds with a deprecation or rebinding
decision.

---

### Recommendation

**Recommended disposition:** B

The Informative reclassification proposed in A is technically defensible and
honest about HTA's posture, but it does not address the substantive concern
that surfaces repeatedly in the Zulip record and in FHIR-43807: the value
set's continued role as the (Example) binding for `MessageHeader.reason` is
itself the problem, and its own steward has called it "inadequate for many
use cases." Lloyd McKenzie has already confirmed that removing the `reason`
binding is non-breaking, and FHIR-43807 has set the precedent that the
companion `event` slot is being treated as intentionally exemplar / unbound.
Disposition B aligns the value set's lifecycle with that direction in a
single coordinated change, instead of using the Informative label to soften
an artifact that the workgroups are otherwise trying to disengage from.
HTA should bring this to I&M for joint disposition, and if I&M is not ready
to retire the value set, fall back to A as a clear interim signal rather
than holding at Draft.
