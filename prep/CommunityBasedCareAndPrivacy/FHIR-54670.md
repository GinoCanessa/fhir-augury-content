## Summary

The ticket reports that the Scalable Consent Management IG's Use Cases page
uses the term **"Actor"** ambiguously. The first diagram on the page depicts
*humans* as Actors (e.g., Consenter, Administrator), while subsequent diagrams
on the same page depict *technical / interoperability* Actors (system roles
that participate in FHIR exchanges). The reporter — John Moehrke — observes
that mixing a human-centric interaction model with a technical-actor model
without naming the distinction makes it hard to read the page or trace a
human role to the system that acts on its behalf. He suggests the IG follow
the common pattern of starting with a human-centric model and then explicitly
defining the technical Actors the humans use.

This ticket is one of a tightly clustered family of UseCases-labelled change
requests filed against the same IG in January 2026, all keyword-similar
(see *Related Jira Tickets* below). The workgroup gave them a common
**Persuasive** disposition: list the actors and systems with definitions,
scour the spec for consistent naming, link each use-case diagram to the
relevant technical operation, and add a table mapping use cases to required
operations.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the four-part remediation already recorded in the workgroup's
Persuasive resolution, anchored at the IG's **Use Cases** page:

1. **Add an "Actors and Systems" section at the top of the Use Cases
   page** that names every role used anywhere in the IG and labels each
   one as either a *Human Actor* (e.g., Consenter, Administrator,
   Delegate) or a *Technical Actor / System* (e.g., Client, Server,
   ConsentSystem). Each entry gets a one-paragraph definition.
2. **Pick one canonical name per role** and remove or redirect every
   alias (e.g., choose between "Server", "Consent Administration
   Service", "ConsentSystem", "consent management services" and use
   only the canonical form thereafter). The same canonical names must
   then appear in §6.1 formal definitions, on the Conformance page's
   Systems list, and in every requirement statement (including the MS-x
   conformance rules that today reference undeclared roles such as
   "Data Sources").
3. **In every use-case diagram, annotate each technical interaction
   arrow with a link to the FHIR operation that realises it** (e.g.,
   `Consent` create, `$consent-decision`, subscription notifications),
   so the reader can move from the human story to the wire-level
   mechanism in one click.
4. **Add a "Use Cases → Technical Operations" table** on the Use Cases
   page (or immediately after it) listing each use case row-by-row
   with the Actors involved, the Systems involved, and the FHIR
   operations the IG requires to implement it. This is the deliverable
   that also closes FHIR-55370.

#### Justification

This is the disposition the workgroup has already agreed to — the
ticket carries a `Persuasive` resolution and the resolution_description
spells out items 1–4 above. The same resolution text appears on
FHIR-54670, FHIR-54672, FHIR-54684, FHIR-54687 and FHIR-55370, so a
single coordinated edit pass closes the entire UseCases-labelled
cluster at once. Distinguishing **Human Actors** from **Technical
Actors / Systems** directly answers the ambiguity Moehrke raised, and
linking each diagram arrow to a FHIR operation makes the documentation
testable: a reviewer can verify that every step of every use case maps
to something in the Conformance page. None of the changes touch
normative profiles or value sets — they are page-level documentation
edits in the IG's `input/pagecontent/` (path not citable here because
the IG repo is not in the local cache).

---

### Disposition B: Alternative Approach

#### Proposal

Adopt items 1, 2 and 4 from Disposition A, but for item 3 use a
**single dedicated "Actors" page** (separate from `usecases.html`)
rather than embedding the Actor/System glossary at the top of the
Use Cases page, and reference that page from every use case, every
Conformance requirement, and §6.1. Concretely:

- Create `actors.html` (or rename §6.1 into a top-level Actors page)
  whose sole job is to enumerate Human Actors and Technical Systems
  with definitions, examples, and explicit non-examples (e.g.,
  "EHR is *not* a Client by itself; the patient-portal module of an
  EHR may act as a Client").
- Replace ad-hoc role names in the Use Cases narrative, §6.1
  glossary, Conformance "Systems" section, and every `MS-x` /
  requirement bullet with hyperlinks back to the canonical
  definitions on `actors.html`.
- Keep the per-diagram operation links and the Use Cases → Operations
  table from Disposition A (item 3 / item 4) so FHIR-55370 still gets
  closed.

#### Justification

The keyword-similar tickets — particularly FHIR-54684 ("§MS-1 mentions
*Data Sources*, which appears on neither page") and FHIR-54922
("definitions in 6.1 are very confusing … define them right at the
beginning") — show the inconsistency is *spec-wide*, not just a Use
Cases page problem. A single canonical Actors page is easier to keep in
sync than an Actors block embedded inside one chapter, and it gives
every other page (Conformance, Profiles, Requirements, §6.1) one URL to
link to instead of duplicating definitions. Trade-off vs. Disposition
A: slightly more page-restructuring effort, plus the risk that
reviewers who landed on the Use Cases page first miss the new Actors
page if cross-links are sparse — mitigated by ensuring every first
mention of an Actor or System on every page is itself a hyperlink.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive** on the basis that the ticket asks for a
documentation clarification on an IG page that the WG plans to
restructure substantially before publication anyway, and that a reader
can already infer "human vs. technical" from diagram context. Resolve
with a one-line note that subsequent ballot-driven edits to the Use
Cases page will incidentally clear the confusion.

#### Justification

This option is documented for completeness but is hard to defend. The
issue has been raised independently by at least three reporters
(Moehrke, Lorenzi, Lamy) across at least eight tickets in the same
month, all carrying the same `UseCases` label, and the inconsistency
demonstrably leaks into the *normative* `MS-x` conformance statements
(FHIR-54684), which means it is no longer a pure documentation concern.
Declining would also leave FHIR-55370's request for a use-case →
operations mapping unaddressed, since that mapping cannot be written
without first agreeing on canonical Actor/System names.

---

### Recommendation

**Recommended disposition:** **A**

The workgroup has already chosen this disposition (resolution =
Persuasive) and the resolution text already lists the four deliverables
in items 1–4 of Disposition A. The same wording closes a cluster of
keyword-similar tickets (FHIR-54672, FHIR-54684, FHIR-54687, FHIR-55370,
and overlaps significantly with FHIR-54922 and FHIR-53887), so committing
to A in one motion gives the most ballot-comment closure per unit of
editorial effort. Disposition B is a reasonable refinement worth
considering as an *implementation detail* of A (i.e., "where does the
Actors glossary physically live?") but it is not a different policy
decision and should not block accepting the request as filed.
Disposition C is included for symmetry only; the volume and consistency
of independent reviewer feedback, plus the leakage into normative
requirement language, makes Decline indefensible.
