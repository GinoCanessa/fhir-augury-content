## Summary

The reporter (via an imported comment from Ron G. Parker) asserts that the
Value Set canonical URL used in the US PACIO Re-Assessment Timepoints IG
should be anchored at `terminology.hl7.org` (THO) rather than under
`hl7.org/fhir`. The ticket proposes that a companion UTG ticket be opened
to register the Value Set in THO. The ticket title further nuances this:
only value sets that are the target of a `required` binding on a `code`
data type element should remain anchored at `hl7.org/fhir`; the rest
should move to THO. The ticket was originally resolved "Considered for
Future Use" in February 2022 (Dave Hill / Dave Pyke, 9-0-0) on the basis
that the IG and its terminology needed to mature before deciding whether
the value set could be absorbed elsewhere, kept IG-specific, or hosted on
THO. Lynn Laakso reopened it post-STU1 publication on 2022-09-28 and
reverted that resolution, so the work group must now reconsider whether
to move the value set's canonical URL to THO. No cross-references to
other Jira tickets, Zulip threads, or GitHub items were found via
fhir-augury for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

For the US PACIO Re-Assessment Timepoints IG, change the canonical URL
of the value set(s) at issue from a `http://hl7.org/fhir/...` namespace
to `http://terminology.hl7.org/ValueSet/<id>`, and open a corresponding
UTG (Unified Terminology Governance) ticket to register the value
set(s) — and any underlying code system(s) the IG defines — in THO.
Update all bindings in the IG's profiles, extensions, and narrative to
point at the new THO canonical URL. Per the title nuance, retain the
`hl7.org/fhir` anchoring only for value sets that are the target of a
`required` binding on a `code` data-type element (where the value set
must travel with the core artifact); move all others (e.g., value sets
bound to `Coding`/`CodeableConcept`, or non-`required` bindings) to
THO. Add migration notes to the IG's history page so existing
implementers can update references.

#### Justification

This aligns the IG with HL7's general direction to host
implementation-guide value sets in THO so they can be governed, versioned,
and discovered alongside other HL7 terminology rather than being scattered
across IG canonical roots. Anchoring in THO also makes the value sets
reusable by other IGs without a cross-IG dependency. Reserving the
`hl7.org/fhir` namespace for `required`-bound `code`-typed value sets
preserves the existing FHIR-core convention (where such value sets are
treated as part of the structural definition) while moving genuinely
terminological content to its proper home. Lynn Laakso's reopening
post-STU1 indicates the work group considers this ripe for action now
that real-world experience exists.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the current `hl7.org/fhir/...` IG-anchored canonical URLs for the
STU1-published value sets to avoid breaking existing implementers, but
publish parallel THO-anchored copies (same content, THO canonical) and
deprecate the IG-anchored versions in the next substantive IG release.
Open the UTG ticket now to register the THO copies. In the IG, add a
migration notice and update bindings in the **next** version to point at
the THO canonicals, with the old URLs maintained as historical aliases.

#### Justification

A canonical-URL change is, by FHIR conformance rules, effectively a new
artifact — every consumer that resolves the old canonical will need to
update. For an IG that has already shipped STU1, a hard switch risks
breaking deployed validators, terminology servers, and downstream
profiles. A "publish-in-THO-now, switch-bindings-later" path captures
the long-term governance benefit of THO hosting while giving implementers
a deprecation window. It also respects the original 2022-02-22
disposition's caution that the IG and its terminology are still
maturing.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep the value set canonical URLs anchored under
the IG's `hl7.org/fhir/...` namespace. Close the ticket as "Not
Persuasive" with the rationale that, for an IG-defined value set whose
sole consumer is this IG, IG-local hosting is appropriate and changing
the canonical post-STU1 would invalidate existing references for no
proportionate benefit.

#### Justification

If the value set is genuinely IG-specific (only used by US PACIO
Re-Assessment Timepoints) and there is no demonstrated reuse demand from
other IGs, the URL anchor is a stylistic preference rather than a
correctness issue. The original 2022-02-22 vote (9-0-0) explicitly
contemplated that the value set might appropriately remain
IG-specific. A canonical-URL change after publication imposes real cost
on implementers (validators, terminology caches, derived profiles) that
is not justified unless cross-IG reuse is concrete.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern — that IG value sets which are not
intrinsic to a `code`-typed `required` binding belong in THO — is
sound and matches HL7's broader terminology-governance direction, so an
outright decline (C) is hard to defend now that the work group has
reopened the ticket post-STU1. A clean accept (A), however, performs a
canonical-URL change on already-published artifacts, which is a known
hazard for downstream validators, derived profiles, and cached
terminology. Disposition B captures the governance win — opening the
UTG ticket and getting the value set(s) registered in THO immediately —
while sequencing the binding switch into the next substantive IG
release with a documented deprecation window. That is the disposition
most consistent with both the original 2022-02-22 caution and Lynn
Laakso's post-STU1 reopening, and it gives implementers a predictable
migration path.
