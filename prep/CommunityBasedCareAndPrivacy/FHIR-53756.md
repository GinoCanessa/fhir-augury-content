## Summary

Craig Newman raises two concrete defects in the **Must Have** element list
of an AuditEvent profile (the keyword analysis identifies it as
`FastConsentAuditEvent`) inside the Scalable Consent Management (FHIR)
specification:

1. The Must Have list mentions `period`, but it is unclear what that
   refers to. The required element on the underlying profile is
   `.recorded`, which is typed as `instant` rather than `Period`. The
   reporter asks for clarification of the term.
2. The element `.source` is constrained to `1..1` on the profile but is
   not surfaced in the Must Have list, even though required elements are
   normally expected to appear there.

The ticket is already triaged with resolution **Persuasive**, with the
work group's stated intent to "update the Must Have list to ensure it
corresponds to the profile." The work needed is therefore documentation
alignment rather than a normative profile change. There are no
cross-referenced Jira tickets, Zulip discussions, or GitHub items
associated with this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **Must Have** narrative list on the FastConsentAuditEvent
profile (Scalable Consent Management IG) to make it match the profile's
machine-readable conformance:

- Replace the entry "period" with **`AuditEvent.recorded` (instant,
  1..1)**, and explicitly note that this is an `instant` (not a
  `Period`) so readers are not misled by the previous wording.
- Add **`AuditEvent.source` (1..1)** — including its required
  sub-elements as constrained by the profile (typically
  `source.observer`) — so every element constrained to `1..1` on the
  profile is reflected in the Must Have list.
- While editing, sweep the rest of the Must Have list against the
  StructureDefinition's `min=1` elements and `mustSupport=true` flags so
  no other required elements are silently omitted.

#### Justification

This is the literal request from the ticket and matches the work group's
already-recorded resolution ("update the Must Have list to ensure it
corresponds to the profile"). The change is narrative-only: it does not
alter cardinalities, bindings, or invariants in the StructureDefinition,
so it has no implementer-visible conformance impact and cannot break
existing instances. Aligning narrative to the profile is also the
behaviour FHIR readers expect — Must Have lists are widely treated as a
quick index of required/mustSupport elements, and the current text both
under-reports (`source`) and uses an inaccurate type label (`period` for
an `instant` element), which is exactly the kind of drift that erodes
trust in the IG documentation.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hand-curating the Must Have list, **derive it from the
StructureDefinition** at IG build time (or replace it with the
auto-generated "Must Support" / "Mandatory" subsections that the IG
Publisher already emits from a profile's snapshot). Concretely:

- Remove the hand-maintained bullet list from the
  FastConsentAuditEvent narrative page.
- Insert the publisher-rendered constraint table (the standard
  `{{table-must-support}}` / mandatory-elements include) so the list is
  always in sync with the StructureDefinition.
- If the work group still wants a curated "what implementers must pay
  attention to" call-out, keep it as a short prose paragraph that points
  at the generated table rather than restating cardinalities.

#### Justification

The defect reported here is a *symptom* of a structural problem: any
hand-maintained Must Have list will drift from the profile every time
the SD changes. A generated list eliminates that drift class entirely,
which is more durable than fixing this single instance. The trade-off
is a small loss of editorial control over wording and ordering, and a
one-time refactor of the IG narrative page; both are modest given this
IG's relative youth. This approach also pre-empts any future ticket of
exactly the same shape against this or sibling profiles in the IG.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the StructureDefinition is the
normative source and already reflects the correct constraints
(`AuditEvent.recorded 1..1 instant`, `AuditEvent.source 1..1`). Treat
the narrative Must Have list as illustrative only and add a single
sentence to the page directing readers to the formal differential /
snapshot for the authoritative conformance statement.

#### Justification

In FHIR, narrative does not override the StructureDefinition, so an
implementer following the SD will already do the right thing regardless
of the Must Have list's wording. Declining avoids spending workgroup
cycles on documentation-only changes and discourages reliance on
hand-maintained lists that will inevitably drift again.

---

### Recommendation

**Recommended disposition:** A

The ticket is already resolved as Persuasive and the requested fix is
small, surgical, and entirely narrative. Disposition A directly
discharges the resolution the work group has already committed to and
removes a concrete misstatement (`period` for an `instant` element) plus
an omission (`.source`) that an implementer skimming the Must Have list
would actually trip over. Disposition B is the better long-term answer
and worth filing as a follow-up against the IG's narrative tooling, but
it is out of proportion to this ticket and should not gate the fix.
Disposition C is hard to defend given the work group has already agreed
the list should match the profile; leaving an inaccurate Must Have list
in place undermines the IG's credibility for a benefit (saved editing
time) that is negligible.
