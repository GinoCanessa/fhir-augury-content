## Summary

The ticket asks that a Code System defined in the US PACIO Re-Assessment
Timepoints IG be re-homed: its canonical URL should be anchored under
`terminology.hl7.org` (THO) rather than under `hl7.org/fhir`, and a
companion UTG ticket should be opened to formally register the Code
System in THO. This is framed as a hygiene/governance change rather
than a content change — the codes themselves are not in dispute, only
where their canonical identity lives. The request was originally
imported from a balloter's comment (Ron G. Parker, "Comment 5") and
was reopened by the work group after STU1 publication so that a
disposition could be reconsidered as the IG matures. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FhirAugury for this key.

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the affected CodeSystem in the US PACIO Re-Assessment
Timepoints IG under `http://terminology.hl7.org/CodeSystem/<id>`,
deprecating the existing `http://hl7.org/fhir/...`-style canonical.
Concretely:

1. File a companion **UTG** ticket against
   `HL7/UTG` requesting registration of the CodeSystem in THO,
   including the `id`, the proposed THO canonical URL, the steward
   (Community-Based Care and Privacy successor work group), and the
   current concept set.
2. In the IG's CodeSystem resource, change `CodeSystem.url` to the THO
   canonical, bump `CodeSystem.version`, and add a `replaces` /
   release-note entry pointing at the prior canonical so existing
   instances and ValueSets can be migrated.
3. Update every ValueSet `compose.include.system` reference inside the
   IG (and any profiles/binding sites) to the new THO URL.
4. Add a migration note to the IG's "Changes since previous version"
   page describing the canonical change and the THO registration.

#### Justification

This is the long-standing HL7 vocabulary governance pattern: shared
terminology lives in THO, not in the FHIR core publication root, so
that it can be reused across IGs and managed by the Vocabulary work
group's lifecycle rather than the FHIR publication cycle. The
balloter's comment and the work group's reopening note explicitly call
this out, and the 2022 deferral was conditional on "further experience
will inform whether this code system … should be … hosted on THO or
another terminology server" — post-STU1 is exactly when that
experience is supposed to feed back into the URL.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the CodeSystem **IG-local** (i.e., under the IG's own canonical
namespace, not core FHIR and not THO), but make the IG-local hosting
explicit and stable:

1. Move the CodeSystem `url` off `http://hl7.org/fhir/...` and onto
   the IG's own canonical base
   (`http://hl7.org/fhir/us/pacio-rt/CodeSystem/<id>`), so it is
   clearly scoped to this IG rather than appearing to be a FHIR-core
   artifact.
2. Document in the IG that the CodeSystem is IG-scoped and is not (yet)
   intended for cross-IG reuse, with a forward note that promotion to
   THO will be reconsidered if/when another IG needs the same codes.
3. Skip the UTG registration step for now, but record this decision in
   the IG's terminology section so the next maintainer knows why the
   codes are not in THO.

#### Justification

This addresses the legitimate concern in the original comment — that
`hl7.org/fhir/...` is the wrong namespace for IG-scoped vocabulary —
without taking on the heavier UTG registration workflow. It is
appropriate when the codes are genuinely PACIO-specific, when they are
expected to churn during the IG's STU lifecycle, or when the work
group does not yet have signals from other consumers that the codes
should be shared. It also matches the qualifier in the 2022 vote
("could be absorbed elsewhere, should be IG-specific, or hosted on
THO"), which left IG-local as an acceptable landing spot.

---

### Disposition C: Decline

#### Proposal

Decline the change for this ticket and leave the CodeSystem's
canonical URL where it is. Rationale category: **already addressed by
prior workgroup decision / out of scope for current maintenance** —
the work group voted 9-0-0 on 2022-02-22 to defer ("Considered for
Future Use") and the responsible work group is now marked
*[deprecated]*, so further canonical churn risks breaking downstream
implementers without an active steward to support migration.

#### Justification

Defensible if (a) no implementer has reported actual confusion or
collision from the current `hl7.org/fhir/...` URL, (b) the IG is in
maintenance mode under a deprecated work group with no active
successor lined up to own the THO entry, and (c) breaking the
canonical URL would invalidate already-published instances and
ValueSets that bind to the existing system. In that situation the
correct posture is to record the limitation, leave the URL alone, and
defer any re-homing to a future major version of the IG when there is
an owner committed to performing the migration end-to-end (UTG
registration, ValueSet updates, conformance regression, release
notes).

---

### Recommendation

**Recommended disposition:** A

The substance of the ticket is correct: a CodeSystem published by an
IG should not live under the `hl7.org/fhir` core canonical root, and
the canonical balloter comment that originated this ticket is calling
out a real governance bug rather than a stylistic preference. The
work group's own 2022 note explicitly anticipated revisiting the
question post-STU1, which is now. Disposition B (IG-local hosting) is
a reasonable compromise but only delays the same conversation, and
Disposition C is hard to defend on the merits — the only real argument
for declining is owner-availability, not correctness. The work group
should accept the change, file the companion UTG ticket, and treat
the canonical migration as a one-time cost paid in the next IG
release. If owner availability is genuinely the blocker (the work
group is marked *[deprecated]*), the right answer is to identify the
successor work group and route the UTG follow-up through them, **not**
to leave an incorrectly-anchored CodeSystem in place.
