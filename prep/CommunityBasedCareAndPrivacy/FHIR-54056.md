## Summary

The submitter (LukeBreyer) asks the Scalable Consent Management ("FASTConsent")
IG to either (a) document the rationale for fixing `Consent.scope` to the code
`patient-privacy`, (b) relax the constraint to allow other FHIR-defined consent
scopes (e.g., research, treatment, public health, adr), or (c) explicitly note
that `patient-privacy` is a technical placeholder rather than a substantive
legal/conceptual statement about the nature of the consent. The concern is that
fixing `Consent.scope` to a single code can be read as encoding a particular
interpretation of consent, which may create ambiguity in jurisdictions or use
cases where consent is operationally framed as research participation, program
enrollment, or statutory disclosure rather than a discretionary patient
privacy authorization.

The only cross-reference surfaced for the ticket is the FHIR element
`Consent.scope` itself (mentioned in the ticket body); there are no linked
Jira tickets, Zulip discussions, GitHub items, or related work-group
companion tickets in the cache. The ticket has already been triaged with a
**Persuasive** resolution: the FAST team has stated that the profile inherits
its constraints from the IHE Basic Consent profile and will add a note on the
profile explaining why the scope is fixed.

## Details

**Description:**

> The FASTConsent profile fixes Consent.scope to the code patient-privacy.
> The IG does not explicitly describe the rationale for this constraint or
> clarify why other FHIR-defined consent scope values are excluded.
>
> FHIR defines Consent.scope as a selector for the type of consent (e.g.,
> privacy, research, public health). Fixing it to a single value can be read
> as encoding a specific conceptual interpretation of consent.
>
> This may create ambiguity in contexts where consent is legally or
> operationally framed as research participation, program enrollment, or
> statutory disclosure rather than as a discretionary patient privacy
> authorization.
>
> Clarifying the intent of this constraint either by documenting the
> rationale for fixing the scope, relaxing it to allow additional appropriate
> values with guidance, or explicitly noting that patient-privacy is a
> technical placeholder rather than a legal definition would help ensure
> consistent interpretation across jurisdictions and use cases.

**Resolution Description (already recorded on the ticket):**

> We are using the IHE Basic Consent profile and have used the constraints
> that they impose. We will provide a rationale for why the scope is fixed
> in notes for this profile.

**Comments:** None recorded on the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the path the FAST team has already indicated in the resolution
description: keep `Consent.scope` fixed to `patient-privacy` on the
FASTConsent profile, but add an explicit **"Rationale for fixed scope"**
note in the profile's narrative / "Notes" section of the Scalable Consent
Management IG. The note should:

1. State that the profile is derived from the IHE Basic Consent profile and
   inherits its scope constraint from that source profile.
2. Clarify that `patient-privacy` here is the technical selector for the
   consent type covered by this IG (privacy / disclosure authorizations),
   not a substantive legal characterization of the underlying authorization.
3. Direct implementers whose use case is research participation, program
   enrollment, statutory disclosure, treatment consent, or public-health
   consent to use a *different* Consent profile (or the base `Consent`
   resource) with the appropriate `Consent.scope` value, rather than
   re-using FASTConsent.
4. Cross-link to the FHIR `Consent.scope` value set so readers can see the
   full set of allowed codes and understand what is intentionally out of
   scope for this IG.

No StructureDefinition / ValueSet / fixed-code change is required under
this option — the change is purely documentation.

#### Justification

This matches the resolution that the work group has already recorded
("Persuasive") and the rationale they have already articulated on the
ticket: the constraint comes from IHE Basic Consent, and the gap is one of
explanation, not of substance. It directly addresses the submitter's stated
ask ("document the rationale … or explicitly note that patient-privacy is a
technical placeholder") without changing conformance for any existing
FASTConsent implementer, and without re-opening alignment with IHE Basic
Consent. It is also the lowest-risk option for an IG that is past triage
and likely to ballot soon.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to the narrative note from Disposition A, **re-shape the
constraint on `Consent.scope`** so that the profile is more explicit about
what the IG covers without forcing every conformant instance to advertise a
single code. Specifically:

- Change `Consent.scope` from a *fixed* code (`fixed[x] = patient-privacy`)
  to a *required* binding to a small profile-specific value set whose
  initial content is `{patient-privacy}`, with documented expansion rules.
- Define a Scalable-Consent-specific value set (e.g.,
  `ScalableConsentScopeVS`) that today contains only `patient-privacy` but
  has a written extension policy: additional `Consent.scope` codes may be
  added in future versions of the IG when the work group adopts a use case
  for them (e.g., research, adr, treatment).
- Either (i) split FASTConsent into sibling profiles per scope (e.g.,
  FASTConsentPrivacy, FASTConsentResearch) gated behind that same value
  set, or (ii) keep one profile but make the slice on `Consent.scope` an
  explicit slicing point for future siblings.

#### Justification

This preserves backward compatibility for current FASTConsent instances
(every existing instance has `scope = patient-privacy`, which is still the
only legal value today) while removing the appearance that the IG is
encoding a single substantive interpretation of consent. It gives the work
group a clean, auditable extension path the next time a non-privacy use
case lands. The trade-off is non-trivial: it diverges from the IHE Basic
Consent constraint pattern, adds a new value set artifact to maintain, and
arguably scope-creeps an IG whose explicit charter today is privacy
consent. The work group should weigh that against the risk that future use
cases (research, ADR, public-health) will otherwise require a new sibling
IG.

---

### Disposition C: Decline

#### Proposal

Decline to relax `Consent.scope` and decline to add new normative narrative
beyond what the IG inherits from IHE Basic Consent. Close the ticket as
"Not Persuasive" with the rationale that (a) the IG's charter is privacy
consent, (b) `patient-privacy` is the correct `Consent.scope` selector for
that charter under the FHIR-defined value set, (c) the constraint is
inherited from IHE Basic Consent and changing it here would fork from that
profile, and (d) implementers needing a different scope should use a
different Consent profile or the base `Consent` resource directly.

#### Justification

A pure-decline is defensible on scope grounds — the IG is explicitly the
*Scalable Consent Management* IG for privacy consent, and FHIR already
provides `Consent.scope` precisely so that other consent types can be
modeled by other profiles. However, the submitter's underlying point that
the rationale is *not currently documented* is factually correct, so a
straight decline leaves the latent ambiguity in place. This option is
listed for completeness but is the weakest of the three.

---

### Recommendation

**Recommended disposition:** A

Disposition A matches the work group's already-recorded "Persuasive"
resolution and the submitter's primary ask (document the rationale or
clarify that `patient-privacy` is a technical selector rather than a
substantive legal claim). It is a documentation-only change, does not
re-open alignment with IHE Basic Consent, does not impose any new
conformance burden on existing FASTConsent implementers, and can be
delivered as a single edit to the profile's narrative before ballot.
Disposition B addresses a real future risk but should be considered as a
follow-up enhancement for a later version of the IG once the work group
has a concrete non-privacy use case in hand; pursuing it now would expand
scope past what this ticket actually requested. Disposition C is not
recommended because it leaves the documentation gap that triggered the
ticket unaddressed.
