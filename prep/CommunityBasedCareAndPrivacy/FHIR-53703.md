## Summary

The reporter (Richard Ettema) flags that the FAST Scalable Consent Management
`SubscriptionTopic` declares a `resourceTrigger.supportedInteraction` of
`delete`, but the IG provides no use case or implementation guidance describing
when or how a consent management server would actually delete a `Consent`
resource (rather than, for example, marking it `inactive` or `entered-in-error`).
Because the semantics of "deleting" a Consent are not specified — and because
the broader FHIR Consent guidance generally discourages hard deletion in favor
of state changes — the reporter argues the `delete` trigger is either
unimplementable as currently written, or needs to be backed by an explicit use
case and accompanying narrative in the *Implementation Notes — Exchanging
Consent* (Subscriptions) section.

The reporter offers two paths: (a) remove `delete` from the topic's
`supportedInteraction` list, or (b) keep `delete` and add a defined use case
plus implementation notes covering subscriber expectations when a Consent is
deleted. The ticket has been triaged with resolution **Persuasive** and a
resolution note indicating the work group intends to **remove `delete` as a
supported interaction from the subscription topic**, aligning with option (a).

There were no Jira, Zulip, or GitHub cross-references found for this ticket in
the FHIR Augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove `delete` from the `resourceTrigger.supportedInteraction` slice of the
FAST Scalable Consent Management `SubscriptionTopic` (the consent topic defined
in `HL7/fhir-consent-management`). After removal, the topic should declare
`supportedInteraction` of `create` and `update` only, with the narrative under
*Implementation Notes — Exchanging Consent → Subscriptions* updated to (1)
state explicitly that subscribers will not receive notifications when a
`Consent` is deleted, and (2) instruct consent servers that intend to signal
revocation/withdrawal to do so by an `update` that sets
`Consent.status = inactive` (or `entered-in-error`), which will fire the
existing `update` trigger.

This is the path the work group has already noted in the resolution
(*"We will remove 'delete' as a supported interaction from the subscription
topic."*).

#### Justification

`delete` semantics for `Consent` are not standardized: the FHIR base spec
treats `DELETE` as physical removal, but the Consent lifecycle in privacy/
security contexts is normally modeled by status transitions
(`inactive`, `entered-in-error`) so that the historical record is preserved
for audit. Allowing `delete` in the SubscriptionTopic without guidance creates
two practical problems:

1. Subscribers cannot rely on receiving delete notifications because most
   conformant consent servers will never issue a `DELETE` in production.
2. Servers that *do* honor `DELETE` and notify on it would be acting against
   common privacy-audit practice.

Removing the trigger and rerouting revocation through `update`/status change
gives subscribers a single, well-defined notification channel and matches the
work group's stated direction.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `delete` in `resourceTrigger.supportedInteraction` but constrain and
document it: add a sub-section to *Implementation Notes — Exchanging Consent →
Subscriptions* titled e.g. "Handling Consent deletion notifications" that:

- Names the specific use case under which a server is permitted to emit a
  delete notification (e.g., purge-on-request / right-to-erasure flows where
  the server is contractually required to physically remove the Consent).
- Specifies subscriber expectations: payload contents (likely an
  `id-only`/`empty` notification keyed by the deleted Consent's logical id),
  required local actions (e.g., invalidate cached authorizations), and the
  audit obligations of both publisher and subscriber.
- Clarifies that `update` to `inactive`/`entered-in-error` remains the
  preferred mechanism for ordinary revocation, and that `delete` is reserved
  for the documented erasure use case.

#### Justification

There are jurisdictions and contracts (GDPR Art. 17, certain US state
right-to-delete regimes) where a consent management server is genuinely
obligated to physically remove a `Consent` and must inform downstream
subscribers so they can purge dependent caches. Stripping `delete` outright
removes a notification path that those deployments need, forcing them to
either ignore the topic or invent a parallel out-of-band signal. Documenting
the trigger — instead of removing it — preserves the capability for the
narrow case while fixing the underlying complaint, which is the *absence of
guidance*, not the presence of the trigger.

The trade-off vs. Disposition A is that this approach is more work
(narrative + payload guidance + at least one worked example) and leaves a
seldom-used path in the topic that some implementers may misuse.

---

### Disposition C: Decline

#### Proposal

Decline the change. Leave `resourceTrigger.supportedInteraction` as-is
(including `delete`) and do not add narrative. Rationale: `delete` is a
standard `SubscriptionTopic` interaction value defined by the base R5
Subscriptions framework, and its semantics there are sufficient — implementers
are expected to know what a FHIR `DELETE` notification means without
IG-specific elaboration.

#### Justification

A pure-decline posture would lean on the base Subscriptions specification to
carry the semantics. In practice this is a weak position because the ticket's
core complaint — that there is no documented Consent-specific use case for
`delete` — is factually correct, and FAST Consent is a privacy IG where
implementers reasonably expect explicit guidance on lifecycle handling. Decline
is documented here for completeness only.

---

### Recommendation

**Recommended disposition:** A

The work group has already triaged this ticket as **Persuasive** with the
explicit resolution note that `delete` will be removed from the SubscriptionTopic,
so Disposition A directly executes the agreed direction and is the lowest-risk
path. It also matches established privacy practice, where Consent revocation is
modeled as a status change rather than a physical delete, and it gives
subscribers a single deterministic notification path
(`create` for new consents, `update` for any lifecycle transition including
revocation). Disposition B remains worth holding in reserve if, during PR
review, a concrete right-to-erasure use case is raised by an implementer — at
that point the work group can re-add `delete` *with* the missing guidance
rather than leaving the under-specified trigger in place today.
