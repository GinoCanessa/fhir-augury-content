## Summary

John Moehrke argues that the Scalable Consent Management (FHIR) IG should
not require any actor to **Create** AuditEvent resources in order to satisfy
its audit-related conformance. Drawing the analogy to US-Core — which
constrains how Observations are *searched* and *read* but says nothing about
how they must be *created* or stored internally — he proposes that the IG
constrain only the AuditEvent **search/read** surface (per IHE BALP, using
the ATNA Audit Record Repository's ITI-81 transaction). Under this model,
implementers may keep their existing non-FHIR audit log storage; conformance
attaches to what they expose to a queryable Audit Consumer, not to how they
record events internally. Consequently, a dedicated **Record Disclosure
operation** (and any other "actor X must create AuditEvent Y" requirement)
is unnecessary: the IG only needs to define which auditable events occur
and that some record of each event must be retrievable as a conformant
AuditEvent profile via ITI-81 search.

The ticket has no Jira cross-references, no related Zulip threads, and no
linked GitHub items in the cached index. The work group has already triaged
it with resolution **Persuasive**, and the resolution text states the
intent to remove the Record Disclosure operation and replace it with
guidance on FAST Audit Events that are searchable using the existing
profile.

## Details

**Description:**

> This page seems to be indicating a need for AuditEvents. Often there is
> concern when defining an AuditEvent that must be CREATED, as most systems
> have audit logs that are not FHIR AuditEvent compliant storage. The
> storage compliance is not important in many Implementation Guides,
> especially such as US-Core. Where US-Core does not define what a
> Blood-Pressure must be CREATED as, or format it must be stored in. It
> only defines how one would Search for Blood-Pressure, and what a
> Blood-Pressure would look like in the search results and Read operation.
>
> Same can be done with AuditEvents. Using IHE-BALP but using only the
> Audit Consumer actor with ITI-81 Transaction to the ATNA Audit Record
> Repository; means that the compliance with AuditEvent is ONLY across
> this ITI-81 transaction (the search).
>
> Once you define that the scope of AuditEvents is that they are
> searchable using ITI-81; and that the results must comply with A, B,
> and C profiles; then you have provided the necessary function of
> accessability to a given set of AuditEvent types. (Much like US-Core
> provides access to a given set of Observation types).
>
> Thus you do NOT need to define a Record Disclosure Operation, or in any
> other way indicate that some actor is responsible to record AuditEvent
> specifically. One needs to define the auditable event, and that some
> record is made of the event such that a given Profile of AuditEvent can
> be discovered on an ITI-81 search.
>
> References
>
> https://profiles.ihe.net/ITI/BALP/volume-1.html#1521-basicaudit-actors-and-content

**Resolution Description (already recorded by the work group):**

> We will remove the Record Disclosure operation.
>
> We will update the section on Record Disclosure to discuss FAST Audit
> Events and that they need to be searchable and that the existing
> profile defines the data that needs to be communicated. We will also
> indicate how these Audit Events are to be recorded and retrieved
> between systems.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the **Record Disclosure operation** from the Scalable Consent
Management (FHIR) IG and rewrite the corresponding section to specify
disclosure auditing in terms of **searchable AuditEvents** consistent with
IHE BALP. Concretely:

- Delete the OperationDefinition (and any references to it from
  CapabilityStatements, narrative pages, examples, and the IG's TOC) for
  Record Disclosure.
- Replace the section content with normative text that:
  - States that systems performing disclosure SHALL ensure each disclosure
    is represented by a discoverable AuditEvent that conforms to the
    relevant FAST / BALP profile already referenced by the IG (i.e.,
    reuse, do not redefine, the existing AuditEvent profiles for
    disclosure).
  - Constrains conformance to the **search/read** interaction on
    AuditEvent (IHE ITI-81 semantics) rather than the create interaction.
  - Clarifies that internal storage format is out of scope: a system may
    keep its audit log in any form so long as the required AuditEvent
    instances are returnable to an Audit Consumer via the specified
    search.
  - Adds an informative note describing how these AuditEvents are
    expected to be **recorded** at the source and **retrieved** between
    systems (Audit Source → Audit Record Repository → Audit Consumer
    flow), without elevating recording mechanics to normative
    conformance.
- Update the IG's CapabilityStatement(s) so that the Audit Record
  Repository / Audit Consumer roles list `search-type` (and `read`) on
  AuditEvent rather than the removed operation, and so that no actor is
  required to support `create` on AuditEvent solely to satisfy this IG.

#### Justification

This is the resolution the work group has already recorded as
**Persuasive**, so Disposition A is the literal codification of that
decision. It is also consistent with established FHIR IG design
patterns: US-Core scopes Observation conformance to search/read shape
without dictating how systems originate or store the underlying data,
and IHE BALP explicitly separates the Audit Source, Audit Record
Repository, and Audit Consumer actors so that a downstream profile only
has to constrain the consumer-facing transaction (ITI-81). Reusing the
existing AuditEvent profiles avoids duplicate normative material and
removes a real adoption barrier for systems whose audit infrastructure
predates FHIR. Defining a bespoke Record Disclosure operation, by
contrast, would force implementers either to wrap their existing audit
pipelines in a FHIR `$record-disclosure` endpoint or to refactor
internal storage — neither of which improves interoperability beyond
what searchable AuditEvents already provide.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a single, lightweight **disclosure-recording capability** in the IG
but redefine it so it is satisfied by *either* of two conformance paths,
rather than by a mandatory custom operation:

1. **Search-based conformance (preferred, per the reporter):** an actor
   exposes the existing AuditEvent profile(s) for disclosure via
   ITI-81-style `search-type` and `read`, and the IG's CapabilityStatement
   marks the `create` interaction as **MAY** rather than **SHALL**.
2. **Operation-based conformance (optional):** retain the Record
   Disclosure operation as an **optional** ("MAY support")
   OperationDefinition for deployments that genuinely need a uniform
   FHIR ingest endpoint into a centrally-managed audit repository (for
   example, when the consent-management actor and the audit-repository
   actor are owned by different organizations and need a contracted
   write API).

In both paths the underlying AuditEvent profile (FAST / BALP) is the
same; only the *mechanism by which a disclosure becomes
discoverable* differs. The IG narrative would explicitly recommend Path
1 and explain that Path 2 exists only for ecosystems that need it.

#### Justification

This preserves the practical benefit of Disposition A — implementers
with existing non-FHIR audit pipelines are no longer blocked — while
not throwing away the small set of integrations where a contracted
write endpoint is genuinely useful. It also lowers the cost of the
change for any early implementers who may already have built against the
Record Disclosure operation: they can keep their implementation, but
with an "optional" status. The trade-off versus A is added IG surface
area (two conformance paths to test, more CapabilityStatement
permutations) and a slightly weaker interoperability story, since
Audit Consumers still cannot assume the operation is present and must
fall back to search regardless. That weaker story is why this option is
listed as the alternative rather than the recommendation.

---

### Disposition C: Decline

#### Proposal

Reject the change and retain the **Record Disclosure operation** as a
required mechanism for recording disclosure AuditEvents in the
Scalable Consent Management (FHIR) IG, on the grounds that an explicit
FHIR-native write API gives the IG a single, testable conformance point
for the act of *recording* a disclosure, independent of how a
particular deployment later exposes the resulting AuditEvent.

#### Justification

A purely search/read-based conformance model (Disposition A) makes the
IG silent about *how* a disclosure becomes a discoverable AuditEvent in
the first place; two systems that both pass search-side conformance
could still disagree about which disclosures must be recorded and at
what fidelity, leaving a gap that the operation was originally meant to
close. Declining keeps the responsibility for recording explicit and
unambiguous, and avoids any perception that the IG is loosening
disclosure-audit requirements. However, this rationale is weak in
context: most real-world audit infrastructures predate FHIR and do not
expose a FHIR write endpoint, IHE BALP already addresses the
record/transport question through the Audit Source actor, and the work
group has already judged the reporter's argument **Persuasive** — so
Disposition C is included for completeness rather than as a credible
recommendation.

---

### Recommendation

**Recommended disposition:** A

The work group has already triaged this ticket as **Persuasive**, the
proposed resolution text matches Disposition A almost verbatim, and the
underlying argument is sound: scoping the IG to the search/read surface
of AuditEvent (per IHE BALP / ITI-81) is the same pattern US-Core uses
for Observation and is well understood by implementers. It also
removes a real adoption barrier — non-FHIR audit log storage — without
weakening what an Audit Consumer can actually retrieve, because the
existing AuditEvent profile is preserved. Disposition B is a reasonable
fallback if the work group surfaces concrete deployments that already
depend on the Record Disclosure operation; in that case the operation
can be demoted to optional rather than deleted. Disposition C should
only be reopened if someone produces a concrete interoperability gap
that search/read conformance cannot close, which the discussion to date
has not done.
