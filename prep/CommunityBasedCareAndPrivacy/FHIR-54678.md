## Summary

John Moehrke flags that the "Consent Network Relationship" page in the
Scalable Consent Management IG is marked **Informative**, yet it contains
language that reads as a normative requirement. The specific concern is
the example describing how "System A would add a system identifier to the
Consent to establish itself as the manager of the Consent" via the
`manager` extension once a Consent is shared. He notes this implies an
in-transit mutation pattern that is inconsistent with FHIR REST and
Resource semantics — under FHIR, a Resource is either updated at its
source (yielding a new version) or copied to a new Resource with a new
`id`; one cannot legitimately "add details in transit." He asks the WG
to either (a) align the wording with REST or (b) otherwise clarify the
intent. No related Jira tickets, Zulip threads, or GitHub items were
surfaced through cross-references for this issue.

## Details

**Description:**

> This page is identified as Informative, but it has implied requirements.
>
> "For example, if System A created a Consent, System A would add a
> system identifier to the Consent to establish itself as the manager of
> the Consent. This originator identifier is encapsulated using the
> `manager` extension. This attribute denotes to all recipients of the
> shared Consent what system originated it. Thus, when the Consent
> resource is shared with another system or systems, a relationship
> between sender and recipient system(s) is formed."
>
> Note that this is describing a methodology of object manipulation that
> is not consistent with FHIR RESTful or even Resource based objects.
> Either the Resource is updated at the source, or a new Resource and
> thus Resource.id is created as a copy of the original. One can't just
> add details in transit.
>
> Some change needs to be made to the text. Either it is aligned with
> REST or it is clarified somehow.

**Resolution Description (already captured in Jira):**

> We will ensure that any pages that have conformance statements are not
> marked as 'informative'.
>
> We will change the sentence:
>
> "For example, if System A created a Consent, System A would add a
> system identifier to the Consent to establish itself as the manager of
> the Consent."
>
> to read:
>
> "When the Consent record is created on System A, System A's system
> identifier SHALL be recorded in the manager extension of the Consent
> instance."

**Comments:**

- **david.pyke (2026-02-18):** "Will accept suggested changes from John."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already drafted on the ticket, applied to the
**Scalable Consent Management (FHIR)** IG's "Consent Network
Relationship" page:

1. Reclassify the "Consent Network Relationship" page from **Informative**
   to a normative classification (e.g., **Standards-track / Trial Use**),
   and audit sibling pages in the IG so that any page carrying conformance
   verbs is similarly reclassified. Explicitly state: "Any page that
   contains conformance statements SHALL NOT be marked as Informative."
2. Replace the offending sentence:
   > "For example, if System A created a Consent, System A would add a
   > system identifier to the Consent to establish itself as the manager
   > of the Consent."
   with:
   > "When the Consent record is created on System A, System A's system
   > identifier SHALL be recorded in the `manager` extension of the
   > Consent instance."
3. Re-read the surrounding paragraph to ensure the rewritten sentence is
   the only place that asserts who populates `manager`, and that the
   subsequent text about sharing describes the *consequence* of the
   already-set extension, not a new in-transit mutation.

#### Justification

This is exactly the modification proposed by the reporter and explicitly
endorsed by the WG triager (david.pyke: "Will accept suggested changes
from John."). It directly fixes both defects:

- The **Informative-vs-conformance** mismatch is resolved by reclassifying
  pages that carry SHALL/SHOULD language, which aligns the IG with HL7
  publishing rules.
- The **REST-semantics defect** is resolved by anchoring the `manager`
  extension to **creation time on the originating system**, removing any
  implication that downstream recipients (or the wire itself) mutate the
  resource. The new sentence preserves the design intent — that `manager`
  identifies the originator — without endorsing in-transit modification.

### Disposition B: Alternative Approach

#### Proposal

Keep the page Informative, but rewrite the example as a *narrative
walkthrough* rather than a conformance assertion, and move the
normative requirement about `manager` into the structural definition
of the `manager` extension itself (i.e., into the extension's
`StructureDefinition` description / comment, not the prose page).
Concretely:

- On the "Consent Network Relationship" page, soften the example to:
  > "In a typical originating workflow, the system that creates the
  > Consent populates the `manager` extension with its own system
  > identifier so that downstream recipients can identify the
  > originator."
  (No SHALL, no implication of in-transit edits.)
- In the `manager` extension definition, add a normative comment:
  > "`Consent.extension:manager` SHALL be populated by the system that
  > originally creates the Consent instance and SHALL NOT be modified by
  > recipients."
- Optionally add an invariant on the Consent profile asserting that
  `manager` is present once the Consent has been shared cross-system.

#### Justification

This separates *concept explanation* (which belongs on a narrative IG
page and is legitimately Informative) from *conformance* (which belongs
on the resource/extension definition where implementers and validators
will actually look). It addresses the underlying REST concern more
durably than Disposition A: even if future authors edit the prose, the
extension definition itself carries the rule. The trade-off is a slightly
larger edit footprint (StructureDefinition + page) and the need to
ensure the page no longer contains *any* SHALL/SHOULD statements, which
the WG would need to confirm during the audit.

### Disposition C: Decline

#### Proposal

Decline both the page reclassification and the rewording on the grounds
that the existing text is illustrative rather than normative — the
sentence begins with "For example" and is therefore explanatory prose.
Add a one-line clarifier instead, e.g., "(This narrative does not impose
a conformance requirement; see the `manager` extension definition for
normative rules.)"

#### Justification

A reader could argue that "For example…" framing already disclaims
conformance intent and that the rest of the paragraph is descriptive of
*what implementations will observe*, not what they must do. Under that
reading, no real interoperability defect exists and the ticket can be
closed with a minor clarifier. The cost of declining is that the
reporter's REST objection — that the prose still *describes* an
in-transit mutation — remains unaddressed and is likely to resurface in
ballot.

---

### Recommendation

**Recommended disposition:** A

The ticket is already at **Triaged** with resolution **Persuasive with
Modification**, the reporter authored the proposed wording, and the WG
triager has explicitly agreed to accept it. Disposition A captures
exactly that outcome and is the lowest-risk path to closure: it fixes
the page-classification policy violation (conformance language on an
Informative page) and replaces the REST-incompatible "would add … in
transit" example with a creation-time SHALL anchored to the originating
system. Disposition B is a stronger long-term design (moving the rule
into the extension definition), and the WG should consider folding its
extension-level SHALL into the resolution as a follow-up, but B should
not block this ticket. Disposition C is not defensible given the WG has
already agreed the change is warranted.
