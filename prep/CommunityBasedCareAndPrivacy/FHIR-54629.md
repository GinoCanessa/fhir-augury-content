## Summary

The reporter (John Moehrke) observes that the narrative for the Consent
resource in FHIR Core still uses the legacy term "exception", a
holdover from the pre-R4 element name `Consent.except` that was renamed
to `Consent.provision`. The ticket asks the work group to clarify the
narrative: where the prose still says "exception", determine whether it
genuinely means an exception (in the everyday English sense) or whether
it should be updated to refer to a `provision`. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced for this
issue, so the request stands on its own as an editorial / clarity fix
to the Consent resource narrative.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Audit the Consent resource narrative (the human-readable prose that
accompanies the resource definition) for every remaining occurrence of
the word "exception". For each occurrence, decide whether the prose is
talking about the structural concept that is now called
`Consent.provision` (in which case rewrite the sentence to use
"provision" / "nested provision" and link to the element) or whether it
genuinely means "exception" in plain English (in which case leave the
word but reword so the structural meaning cannot be misread).
Cross-check related narrative artifacts that ride along with the
resource page (e.g., notes / scope sections, profile pages, security
labels guidance) for the same legacy wording so the cleanup is
consistent.

#### Justification

`Consent.except` was renamed to `Consent.provision` in R4, and the
element-level rename has long since landed; leaving the legacy term in
the surrounding prose is a documented source of reader confusion,
especially for new implementers comparing the narrative to the
structure definition. This is an editorial fix with no normative
impact: no element names, cardinalities, bindings, or invariants
change, so it is low-risk and improves consistency with the current
element names. The reporter is the long-standing privacy SME for the
work group, and the ask is explicitly framed as a clarification, not a
behavior change.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than doing a one-shot "find and replace" pass on the Consent
narrative, take a slightly broader pass: (1) update the Consent
narrative as in Disposition A, (2) add a short "Terminology" note near
the top of the Consent page explaining that earlier drafts of the
resource used the term "exception" for what is now `Consent.provision`,
and (3) extend the audit to other Privacy / CBCC artifacts that
historically used "exception" language (e.g., examples, narrative
profiles, the Security & Privacy module page) so that returning readers
who learned the resource pre-R4 are explicitly oriented. The structural
spec is unchanged; only narrative and a small explanatory note are
added.

#### Justification

The literal request only addresses the immediate occurrence the
reporter noticed. Adding a brief historical note avoids future
"exception vs. provision" tickets from implementers who learned Consent
in R3 and are comparing old code to current docs. Touching adjacent
Privacy artifacts in the same pass is cheap when the narrative is
already being edited, and it is consistent with how editorial cleanups
are usually batched. The added "Terminology" sentence is the only
genuinely new content and is purely informative, so the trade-off
versus Disposition A is a slightly larger diff for a meaningfully
better reader experience.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the term "exception" in the
Consent narrative is being used in the ordinary English sense and not
as a stand-in for `Consent.provision`, and that the structural element
names already make the formal model unambiguous. Close as "Persuasive
With Mod" / "Not Persuasive" with a note pointing readers to the
`Consent.provision` element definition and indicating that the
narrative wording is intentional.

#### Justification

This disposition is only defensible if a careful read of the current
narrative confirms that every remaining "exception" is genuinely an
English-language usage rather than a leftover reference to the old
element name. If that is the case, the structural definitions (which
are normative) are already correct, and changing prose to remove a
common English word risks making the narrative more awkward for no
real implementer benefit. It is the weakest of the three options but
should be on the table so the work group is forced to actually inspect
the narrative before voting.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The reporter is John Moehrke, who has
been the primary maintainer of the Consent resource through the
`except` → `provision` transition, so the assertion that the narrative
still carries legacy wording should be taken at face value rather than
relitigated (which weighs against C). The literal fix in A is correct
but narrow; once the editor is already in the Consent narrative, a
short "Terminology" note plus a sweep of adjacent Privacy artifacts
heads off the next iteration of this exact ticket at marginal cost.
The change remains editorial and non-normative, so it is low-risk and
appropriate for the work group to accept with that small expansion of
scope.
