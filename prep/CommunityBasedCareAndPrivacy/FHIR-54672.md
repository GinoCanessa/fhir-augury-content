## Summary

The reporter observes that the "Scalable Consent Management (FHIR)"
specification's first use case, *Request Consent*, is internally
inconsistent: the narrative says the use case involves the
Administrator, Consenter, and ConsentSystem actors, but the
accompanying diagram does not show the ConsentSystem participating, and
the Actor list in that use case likewise omits ConsentSystem. The
reporter additionally notes that the stated post-condition — "the
Consenter is notified" — only strictly requires the Administrator to
notify the Consenter, suggesting the ConsentSystem's role in this use
case may be narrower than originally drawn or may not belong in the
first use case at all.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced by the tooling for this ticket, so the summary is based solely
on the ticket description and its snapshot. The ticket has already been
triaged with a resolution of **Persuasive** and a resolution description
indicating a broader cleanup is planned: enumerate and define the full
set of actors/systems, sweep the specification for consistent naming,
add links from each use case diagram to the relevant technical
operation, and provide a table that maps operations to use cases.

## Proposed Dispositions

(No GitHub cross-references were surfaced, so the dispositions below
cannot name specific authoring roots or files; they are framed in
spec-editorial terms against the "Scalable Consent Management (FHIR)"
IG. Whichever disposition is adopted, the concrete edits will land in
that IG's source (use case narrative pages and their embedded diagram

### Disposition A: Accept As Requested

#### Proposal

Apply the narrower, literal fix the reporter asks for in the *Request
Consent* use case only:

1. In the *Request Consent* use case narrative, remove ConsentSystem
   from the prose list of involved actors so it matches the diagram
   and the Actor list (which both already exclude it), leaving
   Administrator and Consenter as the only actors for this use case.
2. Leave the diagram unchanged.
3. Reword the post-condition/flow to make explicit that the
   Administrator notifies the Consenter, since no ConsentSystem
   participation is required to satisfy the stated post-condition.

No other use cases or global actor definitions are touched under this
option.

#### Justification

This is the smallest change that resolves the inconsistency the
reporter flagged and aligns the prose with the existing diagram and
Actor list. It respects the reporter's reading of the post-condition —
that only Administrator→Consenter notification is required — and
avoids re-opening design questions about ConsentSystem's role in
request flows elsewhere in the IG. It is also the lowest-risk option
for implementers already reading the current spec, because it does not
rename or redefine any actor.

---

### Disposition B: Alternative Approach (Recommended)

#### Proposal

Adopt the broader cleanup already outlined in the ticket's
**Persuasive** resolution, using FHIR-54672 as the tracking ticket:

1. Add an "Actors and Systems" section to the Scalable Consent
   Management IG that enumerates every actor/system referenced
   anywhere in the IG (at minimum: Administrator, Consenter,
   ConsentSystem, plus any others discovered during the sweep) and
   gives each one a normative definition.
2. Do a full-spec sweep to make sure those names are used consistently
   across narrative, actor lists, and diagrams — including fixing the
   *Request Consent* inconsistency that triggered this ticket (either
   by adding ConsentSystem to the diagram/actor list, or by removing
   it from the prose, whichever matches the intended design once the
   definitions are in place).
3. For each use case diagram, add a link to the specific technical
   operation(s) the use case is realised by.
4. Add a single "Use Case → Operations" table to the IG that maps
   every use case to the operations it requires, so readers can scan
   the operational footprint at a glance.

Revisit the *Request Consent* post-condition during step 2: if the
agreed definition of ConsentSystem is "the system of record for
consents," then it should appear in the use case and diagram (because
the Administrator's notification is presumably recorded there);
otherwise Disposition A's narrower wording is applied as part of the
sweep.

#### Justification

This is the approach the work group has already agreed to via the
Persuasive resolution, so it is both authoritative and in scope. It
addresses the reporter's concern not as a one-off prose edit but as a
symptom of a broader problem — the IG's actor/system vocabulary and
its diagrams have drifted out of sync — which is much more likely to
recur use case by use case if not fixed systemically. Defining actors
once, then enforcing consistent naming and linking each diagram to
concrete operations, also substantially improves implementer
onboarding and reduces ambiguity in later ballot comments. The
trade-off versus Disposition A is larger editorial effort and a wider
diff, but the change is entirely documentation/IG-narrative and
carries no conformance or backward-compatibility risk.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54672 as "Not Persuasive" on the grounds that the first
diagram and the Actor list are already internally consistent (neither
shows ConsentSystem in *Request Consent*) and the prose mention of
ConsentSystem can be treated as an informative, non-normative aside
about which system ultimately stores the resulting consent, not a
claim that ConsentSystem is an actor in that specific flow. No spec
change is made.

#### Justification

A defensible case for declining rests on the observation that the
diagram and actor list — the normative artifacts — are already
correct, and only the surrounding narrative prose is loose. If the
work group treats the narrative as informative framing rather than as
a list of participating actors, there is no actual conformance
problem to fix, and the editorial overhead of the broader cleanup
could be deferred to a later version.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because the work group has already
accepted the ticket with a **Persuasive** resolution whose text
matches Disposition B almost verbatim: enumerate and define actors,
sweep for consistent naming, link diagrams to operations, and publish
a use-case-to-operations table. Disposition A would leave the
underlying vocabulary drift in place and almost certainly produce
similar tickets against other use cases in the next ballot cycle;
Disposition C contradicts the recorded resolution and ignores the
reporter's legitimate observation that the narrative, diagram, and
actor list disagree. B is also the option that most improves the IG
for implementers, since it turns "which actors and operations does
this use case involve?" from a prose-reading exercise into a table
lookup. The editorial cost is real but bounded, and the risk is low
because no normative conformance surface changes.
