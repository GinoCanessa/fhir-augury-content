## Summary

The ticket asserts that the canonical URL for a Code System defined by the
US PACIO Re-Assessment Timepoints IG is anchored under `hl7.org/fhir`, and
argues that the canonical should instead be anchored under
`terminology.hl7.org` (THO), the HL7 home for shared terminology
artifacts. The submitter (carrying forward Comment 6 from Ron G. Parker)
also asks that a companion UTG ticket be opened so the Code System is
formally registered in THO. The ticket was originally resolved
"Considered for Future Use" in February 2022 with a 9-0-0 vote (Hill /
Pyke), on the rationale that more IG/terminology experience was needed to
decide whether the Code System belonged in the IG, on THO, or elsewhere;
in September 2022, lynn_laakso reverted that resolution and re-opened the
ticket post-STU1 publication, leaving it Triaged with the
"Block-Vote-1" label. No cross-referenced Jira tickets, Zulip threads,
GitHub items, or Confluence pages were returned by the data sources for
this ticket.

### Disposition A: Accept As Requested

#### Proposal

In the next normative cycle of the **US PACIO Re-Assessment Timepoints
(FHIR)** IG, change the canonical URL of the IG-defined Code System from
the current `http://hl7.org/fhir/...` form to a
`http://terminology.hl7.org/CodeSystem/...` URL, and remove the Code
System resource from the IG package so the IG only references it. In
parallel, file a UTG (Unified Terminology Governance) Jira ticket against
the UTG repository to register the Code System in THO under the new
canonical, including: code system metadata (name, title, status,
content), the full concept set carried in the IG today, and any value
sets in the IG that compose this Code System so they can be re-pointed
to the THO canonical.

#### Justification

The proposal aligns with HL7's standing terminology guidance: code systems
that are not version-locked to a specific FHIR core release belong in THO
under `terminology.hl7.org`, not under the FHIR specification namespace.
Moving the canonical now (post-STU1) and registering through UTG gives
the Code System a stable, terminology-server-friendly home, lets other
IGs (PACIO and beyond) reuse it without coupling to PACIO Re-Assessment
Timepoints, and is exactly the path the original 2022-02-22 resolution
deferred until the IG had more experience. Lynn's September 2022
re-open after STU1 publication signals that experience is now in hand.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the Code System defined inside the **US PACIO Re-Assessment
Timepoints** IG (retain the current IG-anchored canonical), but
explicitly mark it as IG-scoped: tighten its narrative to say it is
specific to PACIO Re-Assessment Timepoints, and add a documented
deprecation/migration path so that, if/when a broader PACIO or
cross-IG need emerges, the Code System can be lifted into THO with a
canonical change recorded in `ConceptMap` form. Defer the UTG
registration ask until at least one additional consumer outside the IG
materializes.

#### Justification

The original 9-0-0 resolution to "Consider for Future Use" was based on
the concern that it was not yet clear whether the Code System should be
"absorbed elsewhere, IG-specific, or hosted on THO." If concrete
external reuse has not actually appeared since STU1, moving the
canonical incurs migration cost (every implementer pinned to the
current canonical breaks) without a demonstrable reuse benefit. This
disposition honors the original concern, avoids a breaking canonical
change for early STU1 implementers, and keeps the THO option open for a
later cycle once reuse is real.

---

### Disposition C: Decline

#### Proposal

Decline the request and close as **Not Persuasive**, on the grounds
that (a) the IG is already published as STU1 with the existing
canonical, (b) no concrete cross-IG reuse of the Code System has been
cited in the ticket or its (empty) cross-references, and (c) the prior
workgroup resolution already evaluated this exact tradeoff and chose
"Considered for Future Use." Recommend that any future re-open be
accompanied by a specific external consumer or a UTG submission already
in flight.

#### Justification

A canonical URL change is a breaking change for every implementer who
has bound to the IG's Code System. Doing it without an attached UTG
ticket, an identified second consumer, or a coordinating PACIO-family
change set risks breaking STU1 conformance for purely speculative
benefit. The 2022 vote was unanimous (9-0-0) on essentially the same
information; absent new evidence in the ticket, declining is the most
defensible action and leaves the door open for a properly-scoped re-file
later.

---

### Recommendation

**Recommended disposition:** A

The submitter's ask is consistent with HL7's standard terminology
architecture (shared code systems live in THO), and the September 2022
re-open from lynn_laakso explicitly anchors the reconsideration in
"post-STU1 publication" — i.e., the IG is now mature enough to absorb a
canonical change in its next cycle, which is exactly the trigger the
2022-02-22 resolution was waiting for. The breaking-change concern from
Disposition C is real but is best handled by sequencing the canonical
move with the next normative/STU update of the IG and by publishing a
small `ConceptMap`/migration note, not by leaving the Code System
permanently parked under `hl7.org/fhir`. Disposition B is the safer
holding pattern, but it effectively re-applies the same "wait and see"
that the workgroup has already used once; choosing it again would push
the decision out indefinitely. The workgroup should adopt Disposition
A, file the companion UTG ticket as part of the disposition, and
schedule the canonical change for the next IG version.
