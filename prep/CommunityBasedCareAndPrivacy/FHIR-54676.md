## Summary

The Scalable Consent Management IG currently constrains the `Consent.status`
element so that only `active` or `inactive` are conformant. The reporter (John
Moehrke) observes that this excludes `entered-in-error`, which is a normal
real-world status — it covers the routine "oops" case where a Consent was
captured incorrectly and must be retracted. Without `entered-in-error` (and
other lifecycle statuses) being addressable by the IG, any Consent in such a
state is effectively non-compliant with the IG, even though it is a perfectly
valid FHIR Consent. The ticket asks the IG to either widen the allowed
statuses or, at minimum, explain how implementations should handle statuses
that fall outside `active` / `inactive`.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
linked to this ticket in the Augury index. The ticket has already been
triaged with resolution `Persuasive`; the recorded resolution direction is to
remove the IG's value-set restriction and instead provide guidance on the
FAST Consent profile distinguishing IG-covered statuses (`active`,
`inactive`) from statuses that may be present in a consent system but are
not the focus of the IG (e.g., `entered-in-error`, `draft`, `proposed`,
`rejected`).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an explanatory section to the Scalable Consent Management IG (most
naturally on the FAST Consent profile page) that explicitly calls out why
the IG narrows `Consent.status` to `active` and `inactive`, and that
explains how implementations should treat the other FHIR
`ConsentState` codes — in particular `entered-in-error`. Keep the current
required binding to the IG-defined value set ({active, inactive}) on the
FAST Consent profile, but add normative narrative such as:

- A Consent in `entered-in-error` is a valid FHIR Consent but is **out of
  scope** for exchange under this IG. Senders SHOULD NOT exchange Consent
  resources in that state; receivers MAY ignore them.
- Equivalent treatment for `draft`, `proposed`, and `rejected`, with a
  short rationale for each.
- Pointer back to the base FHIR `Consent.status` value set so readers
  understand what those codes mean in non-IG contexts.

#### Justification

This is the minimum-impact change: it preserves the IG's existing
constraint (which is intentional — the IG is about *exchanged, in-force*
consent) and removes the documentation gap the reporter raised. Keeping a
required binding to {active, inactive} maintains a clear conformance
target for implementers and validators, which is consistent with how other
HL7 IGs (e.g., US Core, IPS) narrow base-resource value sets when the use
case warrants. It also directly addresses the "oops" scenario by telling
implementers what the IG expects them to do with `entered-in-error`
records (handle them locally, don't exchange them under this IG).

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the resolution already recorded on the ticket: remove the IG's
required-binding value set on `Consent.status` and instead bind to the
full base FHIR `ConsentState` value set (preferred or required, as the
work group prefers), then add narrative guidance on the FAST Consent
profile that distinguishes:

- **IG-covered statuses** (`active`, `inactive`) — fully described,
  exchanged, and processed under this IG.
- **Statuses present in real systems but outside IG scope** — `draft`,
  `proposed`, `rejected`, `entered-in-error` — permitted on the wire so
  that implementations can faithfully round-trip them, with explicit
  caveats that processing semantics for these are deferred to other
  specifications or local agreement.

Per david.pyke's comment, include explicit guidance that when Consents
*are* exchanged, recipients should expect any of the base codes but
should treat anything outside `active`/`inactive` as informational only.

#### Justification

This matches what the work group has already concluded ("Persuasive"
resolution). It is more permissive than Disposition A and better reflects
how Consent resources actually flow in mixed environments — e.g., a
sending system may legitimately need to retract a previously-shared
Consent by re-sending it as `entered-in-error`. Loosening the binding
also avoids forcing implementers into IG non-conformance for a normal
FHIR lifecycle state, which was the reporter's core concern. The trade-off
is a slightly weaker conformance target (validators will no longer fail
on `entered-in-error`), but the narrative guidance preserves the IG's
intent for the in-scope statuses and matches base-resource semantics.

---

### Disposition C: Decline

#### Proposal

Leave the current `Consent.status` value-set restriction
({active, inactive}) in place and add no further narrative. State that
the IG is deliberately scoped to *currently-effective* consent for
exchange, and that lifecycle and error-correction states are out of
scope by design.

#### Justification

Defensible only on the grounds that the IG's name ("Scalable Consent
*Management*") is intentionally narrow and the constrained binding is the
explicit way the work group expresses that scope. However, this position
is hard to sustain given (a) the reporter's point that
`entered-in-error` is a legitimate base-resource state that real systems
will produce, (b) david.pyke's supportive comment, and (c) the work
group's own recorded resolution direction. A pure decline would also
leave the documentation gap the reporter raised, which is the easiest
part of the ticket to fix.

---

### Recommendation

**Recommended disposition:** B

The work group has already recorded a "Persuasive" resolution that aligns
with Disposition B (remove the value-set restriction, add narrative
guidance on the FAST Consent profile). That direction respects the
reporter's underlying point — `entered-in-error` is a normal FHIR Consent
state that should not, by itself, render an instance non-conformant with
this IG — and it incorporates david.pyke's suggestion to allow the
broader set with accompanying guidance about what is in and out of IG
scope. Disposition A is a smaller, narrative-only change and would be a
reasonable fallback if loosening the binding proves controversial on
review, but it leaves implementers in the awkward position of producing
valid FHIR Consents that are still IG non-conformant for what is
effectively a documentation issue. Disposition C should not be selected:
it contradicts both the comment thread and the recorded resolution, and
it leaves the original gap unresolved.
