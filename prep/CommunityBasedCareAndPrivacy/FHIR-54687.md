## Summary

The reporter (John Moehrke) flags an internal inconsistency in the
Scalable Consent Management (FHIR) implementation guide: the IG defines
the set of "systems" (and, by implication, actors) differently across
multiple pages. The page cited in the ticket distinguishes only two
system types — **Client** (gathers and files consents) and **Server**
(consent administration service that stores and serves consents) — but
other pages in the IG use a different set of definitions. The reporter
is unable to tell which definition is normative and asks the work group
to reconcile the terminology so readers and implementers have a single
canonical set of actors/systems to follow.

The ticket is already marked **Resolved: Persuasive**, with a resolution
plan to enumerate the actors and systems on the Use Cases page and then
sweep the rest of the specification for consistency. There are no
cross-referenced Jira tickets, Zulip threads, or GitHub items in the
Augury index, so the work group is being asked to confirm/refine the
"Persuasive" plan rather than choose between competing solutions raised
elsewhere.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already recorded on the ticket exactly as written:

1. On the **Use Cases** page of the Scalable Consent Management (FHIR)
   IG, add a single normative section that enumerates every actor and
   system the IG references (at minimum: **Client** — gathers and files
   consents, and **Server** — consent administration service that
   stores consents and handles consent requests; plus any additional
   actors used elsewhere in the IG, such as patient/consenter,
   organization, and any third-party requester role).
2. Provide a one-paragraph definition for each actor/system, written so
   that it is the canonical definition for the IG.
3. Sweep the remainder of the specification (overview, conformance,
   capability statements, profiles, narrative pages) and rewrite any
   inconsistent terminology (e.g., other names for the same role, or
   role names used with different meanings) to match the Use Cases
   page.
4. Add a glossary cross-link from each page that introduces an actor or
   system back to the Use Cases definitions.

#### Justification

This is the resolution already proposed by the work group ("Persuasive"
on the ticket) and is the lowest-risk way to remove the ambiguity the
reporter flagged. It is purely editorial — no profiles, value sets, or
conformance rules change — and consolidating actor/system definitions
on the Use Cases page is consistent with how most HL7 IGs (and FHIR
core) handle terminology: define once, link many. Naming a single
canonical page also makes future audits easy: any new mention of an
actor or system can be checked against one list rather than
reconciled across pages.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) prose definitions on the Use Cases
page, formalize the actor/system model with FHIR conformance artifacts:

1. Define an `ActorDefinition` (or, where the IG already targets an
   older publisher, an obvious-name extension on `CapabilityStatement`)
   for each actor — at minimum **Client** and **Server**, plus any
   ancillary actors discovered during the editorial sweep.
2. Reference those `ActorDefinition`s from the relevant
   `CapabilityStatement`(s) and from the `Requirements` /
   `ImplementationGuide` resources so the actor names appear once in
   machine-readable form.
3. On the Use Cases page, render the actor list directly from those
   `ActorDefinition`s (so the prose and the conformance artifacts
   cannot drift apart), and update every other page to link to the
   rendered list rather than redefining actors locally.
4. Keep the editorial sweep step from Disposition A for any prose that
   mentions an actor or system by name.

#### Justification

The underlying problem the reporter is describing — multiple, drifting
prose definitions of the same concept — is exactly the failure mode
`ActorDefinition` was added to FHIR R5 to prevent. Capturing actors as
first-class conformance artifacts means future divergence is mechanically
detectable rather than only catchable by careful proofreading, and lets
implementers (and downstream IGs that build on Scalable Consent
Management) reference the same actor set without re-typing the
definitions. Trade-off vs. Disposition A: more up-front editorial and
build-tooling work, and potential publisher-version constraints, in
exchange for a structurally durable fix.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the IG's
current two-tier "Client / Server" model is the canonical definition
and any other pages that appear to define systems differently are
either using "system" in a colloquial sense or describing sub-roles of
Client / Server. Add a short note to the ticket explaining the
intended reading rather than changing the IG.

#### Justification

This disposition is defensible only if the work group concludes (after
inspecting the IG pages the reporter is alluding to) that there is no
real inconsistency — only differing levels of detail or rhetorical
framing — in which case editorial churn would invite new ambiguity
without removing any. It is a poor fit for the present ticket because
the reporter explicitly says he cannot tell which definition to follow,
and the work group has already agreed the request is persuasive; this
disposition is included for completeness only.

---

### Recommendation

**Recommended disposition:** A

The ticket is already marked Persuasive with a concrete plan, the fix
is purely editorial, and the reporter's complaint is exactly the kind
of low-cost, high-clarity issue that should be closed by doing the
work rather than reopening the design discussion. Disposition B is
attractive in principle and worth keeping on the work group's roadmap
(especially as `ActorDefinition` adoption matures), but binding it to
this ticket risks delaying the fix the reporter actually asked for and
expanding scope into publisher / tooling territory the
Community-Based Care and Privacy work group may not be ready to take
on in this cycle. Recommend executing Disposition A now, and tracking
the `ActorDefinition` upgrade as a follow-up ticket if the work group
wants to pursue it.
