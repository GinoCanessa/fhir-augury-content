## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a **Plain Language Summary** section to the *Scalable Consent
Management (FHIR)* implementation guide as part of the publication
process. Concretely, add a short, jargon-light narrative (typically
under one page) to the IG's home/landing page (or a dedicated
`plain-language-summary.md`/`.xml` page linked from the TOC) that
describes, in accessible terms:

- what the guide is for (patient consent management at scale),
- who the intended audiences are (patients, clinicians, implementers,
  policy/privacy stakeholders),
- the core capabilities it enables (capturing, exchanging, and
  enforcing scalable consent decisions), and
- what the guide does *not* do (scope boundaries).

Wire the new page into the IG's menu/TOC and ensure it is linked from
the IG home page so it is discoverable at publication time. This
matches the existing resolution ("will be added when the guide is
published") and closes the editorial gap the reporter identified.

#### Justification

HL7 FHIR IGs increasingly ship a plain language summary as a baseline
editorial expectation, and the reporter's request is consistent with
that practice. The ticket is already resolved as **Persuasive**, so the
workgroup has implicitly agreed that the summary belongs in the guide;
Disposition A simply codifies what that deliverable looks like so the
publication editor can act on it. Because this is a purely editorial,
additive change, there is no impact on conformance, StructureDefinitions,
or existing implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring a new standalone Plain Language Summary page,
extend the IG's existing home/landing page with a clearly-labelled
"Plain Language Summary" section at the top, plus a small glossary of
key consent terms (e.g., *consent directive*, *policy*, *permission*,
*scope*) on a linked page. This keeps the summary in the first screen
the reader sees and pairs it with a lightweight glossary so the
plain-language commitment is actually usable. Content requirements are
the same as Disposition A (purpose, audiences, capabilities, scope
boundaries).

#### Justification

A top-of-home-page summary is harder to miss than a sibling page and
avoids creating a page that readers must navigate to. Pairing the
summary with a short glossary directly addresses the root need behind
"plain language" — it reduces the number of FHIR/privacy terms a
non-specialist reader must resolve on their own. The editorial burden
is only marginally higher than Disposition A and stays well within what
is expected at publication time.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for a spec change, on the grounds
that "missing plain language summary" is a publication/editorial task
that does not require a tracked change request, and close the ticket
without further action.

#### Justification

This is a defensible position only if the workgroup considers the
plain-language-summary commitment already covered by the existing
publication checklist. In that case, tracking it here adds no value
over the IG editor's normal pre-publication pass. However, the ticket
is currently marked **Persuasive**, which is inconsistent with a
decline, so adopting this disposition would require the workgroup to
re-resolve the ticket and explicitly rely on the publication checklist
to deliver the summary.

---

### Recommendation

**Recommended disposition:** A

The ticket is narrowly scoped, already resolved as **Persuasive**, and
has an explicit resolution note committing to add the plain language
summary at publication. Disposition A simply operationalises that
commitment by defining the deliverable (a linked Plain Language Summary
page covering purpose, audiences, capabilities, and scope) so the IG
editor can confidently close the loop at publish time. Disposition B is
a reasonable stylistic variant and the workgroup may prefer it; it can
be adopted as a refinement of A without re-opening the disposition.
Disposition C conflicts with the existing Persuasive resolution and is
not recommended.
