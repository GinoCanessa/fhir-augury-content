## Summary

The reporter (John Moehrke) flags an editorial inconsistency in the
Scalable Consent Management IG's pre-reading section: the prose links to
a page labelled **"Technical Background"**, but the link target
(`background.html`) is actually titled **"Reading this IG"** and its
content is scoped to FHIR Core rather than to this IG. The mismatch
between the link text and the destination page title — and the
narrowness of the destination's content relative to what a reader of
the Consent Management IG would expect — makes readers feel they have
landed on the wrong page.

The ticket has already been triaged with a resolution of *Persuasive
with Modification*: the working group plans to (a) rename the link in
the pre-reading so it matches the actual destination page, and (b)
broaden the destination page so it covers not only FHIR Core but also
Subscriptions Backport and the two relevant IHE guides that this IG
builds on.

No other Jira tickets, Zulip threads, or GitHub items cross-reference
this ticket in the FhirAugury index.

## Details

**Description:**

The following is found in the pre-reading; and is confusing as there is
no navigable page called "Technical Background", the link goes to a
page called "Reading this IG". As a reader online, I can click on the
link, but getting to something called something totally different makes
me feel I went to the wrong place. Further that destination page is
about FHIR core; not this IG.

- The [Technical Background](https://hl7.org/fhir/us/consent-management/2026Jan/background.html)
  page provides information about the underlying specifications and
  indicates what portions of them should be read and understood to have
  the necessary foundation to understand the constraints and usage
  guidance described here.

**Resolution Description (already on the ticket):**

We will change the name of the link since there is not a Technical
Background. We will also update the description of that page since it
is more about understanding the core specifications that this IG is
built on. It currently only discusses FHIR Core, so we will add
sections about Subscription Backport, and the two IHE guides.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply exactly the change captured in the ticket's resolution:

1. In the IG's pre-reading narrative, rename the link text from
   "Technical Background" to match the destination page's actual
   title — i.e., either change the link text to **"Reading this IG"**,
   or rename the destination page so it can legitimately be called
   "Technical Background". The simplest, lowest-risk option is to
   change the link text to **"Reading this IG"**.
2. Expand `background.html` so it covers more than FHIR Core: add
   sections describing how the IG depends on **Subscriptions Backport**
   and on the **two IHE guides** that the Consent Management IG builds
   on, alongside the existing FHIR Core material.
3. Update the surrounding pre-reading prose so it accurately describes
   what the destination page now contains.

#### Justification

This matches the resolution already agreed by the work group on the
ticket itself. It is purely editorial / narrative — no
StructureDefinition, ValueSet, profile, or example is affected — so
there is no implementer breakage risk. It eliminates the immediate
confusion the reporter raised and improves the IG's "where do I start
reading?" experience by making the prose, the link text, and the
destination page consistent. Restoring parity between link text and
page title is also consistent with HL7 publication practice for IG
narrative pages.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of expanding the existing `background.html` "Reading this IG"
page in place, restructure the pre-reading into two distinct pages:

- Keep `background.html` (or rename to `reading-this-ig.html`) as a
  short "How to read this IG" orientation page describing the IG's
  audience, conventions, and reading order.
- Add a new `technical-background.html` page that consolidates the
  technical dependencies — FHIR Core sections that must be understood,
  Subscriptions Backport, and the two IHE guides — and link to it from
  the pre-reading using the existing "Technical Background" wording.

Update the ToC / menu to surface both pages so the link target name in
the prose matches a real, navigable page.

#### Justification

This preserves the reporter's original mental model (a link called
"Technical Background" should land on a page called "Technical
Background") and keeps "how to read this IG" separate from "what other
specifications you need to understand". That separation tends to age
better as the IG grows: orientation content rarely changes, while the
list of underlying specifications (Subscriptions Backport version, IHE
profile versions, etc.) is more volatile and benefits from living on
its own page that can be revised without touching the orientation
prose. The cost is one additional narrative page and a menu entry,
which is small relative to the readability gain.

---

### Disposition C: Decline

#### Proposal

Close the ticket as *Not Persuasive*, on the grounds that the link in
the pre-reading is descriptive prose ("the Technical Background page
provides information about the underlying specifications…") rather
than a navigation label, and readers can self-correct once they see
the destination page title.

#### Justification

A defensible decline rests on the argument that the issue is purely
cosmetic — readers reach the right page, just under a different label
— and that publication time is better spent on normative content for
the next ballot cycle. This is the weakest position because the
reporter is an experienced FHIR participant flagging a real reader
experience problem, the resolution is cheap to implement, and the
decline would have to be revisited the next time anyone reviews the
IG's onboarding flow.

---

### Recommendation

**Recommended disposition:** A

The work group has already converged on *Persuasive with Modification*
and the proposed change is a low-risk, narrative-only edit that
directly addresses the reporter's concern. Disposition A captures that
agreement: rename the link to match the destination page, then expand
`background.html` to include Subscriptions Backport and the two IHE
guides so the page lives up to its role as the IG's "what you must
already understand" anchor. Disposition B is a reasonable upgrade if
the work group decides the orientation/dependencies split is worth a
second narrative page, but it is a larger structural change than the
ticket needs and can be deferred to a future editorial pass.
Disposition C should be rejected — the reporter's concern is concrete,
the fix is small, and declining would leave a known confusing link in
the published IG.
