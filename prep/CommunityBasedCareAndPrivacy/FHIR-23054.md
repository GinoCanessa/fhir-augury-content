## Summary

The ticket questions whether the eLTSS (Electronic Long-Term Services & Supports) IG should map "Service Start Date" and "Service End Date" to `ServiceRequest.occurrenceTiming.boundsPeriod` (the current STU2 design) or instead use `ServiceRequest.occurrencePeriod`. The reporter notes that `occurrenceTiming` only really earns its keep when richer schedule and quantity information (e.g., day-of-week, time-of-day, frequency, duration) is also present; for plain start/end dates `occurrencePeriod` would be the more natural choice. The conversation in comments evolves into a structural question for the FHIR base resource: because `ServiceRequest.occurrence[x]` is `0..1`, an eLTSS profile cannot simultaneously bind `occurrencePeriod` (for the service window) and `occurrenceTiming` (for the schedule/quantity inside that window). Two options are proposed for discussion with O&O: (1) raise `ServiceRequest.occurrence` cardinality from `0..1` to `0..*`, or (2) add a new top-level `ServiceRequest.period` element so that the timing slot can be reserved for the recurrence pattern. There are no cross-referenced Jira tickets, GitHub items, or Zulip threads in the indexed FHIR Augury data; the only cross-references found are FHIR element mentions (`ServiceRequest.occurrence`, `ServiceRequest.occurrencePeriod`, `ServiceRequest.period`).

## Details

**Description:**

> In reference to FHIR R4 Resource Elements(s):
> ServiceRequest > occurrenceTiming > boundsPeriod -> start
>
> ServiceRequest > occurrenceTiming > boundsPeriod -> end
>
> Question: Should this be using occurencePeriod instead of occurrenceTiming? occurrenceTiming makes sense only if it is meant to be used for other service schedule and quantity information (separate comment).

**Comments:**

### bheale (2024-04-02)

For STU2 the eLTSS IG adopted and balloted the use of `occurrenceTiming.boundsPeriod` to ascribe a start and end date — see the published STU2 ServiceRequest-eltss profile. The element documentation reads:

> "Please consider using this for timing information that fluctuates or is sufficiently complex. You will need to calculate end-date, or use occurrenceTiming.boundsPeriod to ascribe a start and end date. E.g. Give ServicePlan.quantity 5 with unit=[USD]/day with the occurrenceTiming.boundsPeriod of May 31, 2023 to June 23, 2023 on occurrenceTiming.dayOfWeek = Mon and Wed at occurrenceTiming.timeOfDay = 3 pm."

This is also the structure that carries `quantity` and `quantityRatio` for the service.

### lynn_laakso (2024-05-27)

Reverted the previous resolution ("Considered for Future Use", Amber Patel/Kathleen Connor: 12-0-1, 2020-07-21) which had reasoned: "ServiceRequest.occurrence is 0..1. In order to accommodate the service start/end dates as well as the interval and quantity elements, occurrenceTiming is used."

Two options to discuss with the Orders and Observations work group:
1. Change the cardinality on `ServiceRequest.occurrence` from `0..1` to `0..*` so a profile can bind `occurrencePeriod` for service start/end and `occurrenceTiming` for service interval/quantity.
2. Add a new element `ServiceRequest.period` to record start and end date.

If option 1 is executed, update the eLTSS mappings and profiles for "Service Start Date" / "Service End Date" to map to `ServiceRequest.occurrencePeriod`. If option 2 is executed, map them to `ServiceRequest.period` (or whatever the new element is called).

### lynn_laakso (2024-05-27)

Re-open post STU2 publication.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the eLTSS IG to map "Service Start Date" and "Service End Date" to `ServiceRequest.occurrencePeriod` instead of `ServiceRequest.occurrenceTiming.boundsPeriod`. Because `ServiceRequest.occurrence[x]` is a `0..1` choice, this requires upstream FHIR core to also accept the change so an eLTSS ServiceRequest can carry both a service window *and* a recurrence pattern. Submit a paired core change request to the Orders and Observations work group implementing **Option 1** from the May 2024 comment: relax the cardinality on `ServiceRequest.occurrence[x]` from `0..1` to `0..*` so a profile may bind both `occurrencePeriod` (for the service window) and `occurrenceTiming` (for schedule/quantity within the window). Once accepted, modify `StructureDefinition-ServiceRequest-eltss` so the Service Start/End Date elements slice on `occurrencePeriod`, and keep the existing `occurrenceTiming` binding for the schedule/quantity payload (`dayOfWeek`, `timeOfDay`, `boundsDuration`, `ServicePlan.quantity`, etc.).

#### Justification

The reporter's premise is correct: `occurrencePeriod` is the natural FHIR datatype for a bare start/end window, and `Timing.boundsPeriod` is intended as scope for a recurrence pattern, not as the primary place to record an unadorned service interval. The current STU2 mapping conflates the two only because `occurrence[x]` is single-valued. Loosening that cardinality preserves backward compatibility (existing `0..1` instances remain valid under `0..*`), aligns eLTSS with the way most workflow resources separate "when does this happen overall" from "how does it repeat", and lets implementers reason about service windows with simple `Period` queries rather than navigating into `Timing.boundsPeriod`.

---

### Disposition B: Alternative Approach

#### Proposal

Implement **Option 2** from the May 2024 comment: add a new dedicated `ServiceRequest.period` (`0..1 Period`) element to FHIR core, scoped explicitly as the validity / service-window for the request, and keep `ServiceRequest.occurrence[x]` at `0..1` for the recurrence/timing pattern. Update eLTSS to map "Service Start Date" / "Service End Date" to `ServiceRequest.period.start` and `ServiceRequest.period.end`, and leave `occurrenceTiming` reserved for `dayOfWeek` / `timeOfDay` / `boundsDuration` and the quantity-bearing schedule. As an interim, until O&O accepts a core change, eLTSS can ship a FHIR extension on ServiceRequest carrying a `Period` and document the mapping accordingly so implementers are not blocked.

#### Justification

This approach preserves the conceptual separation that the reporter is highlighting — a service *window* is a fundamentally different idea from a service *recurrence pattern* — and encodes that separation in the resource shape itself rather than relying on profile-level constraints to police a `0..*` choice. It avoids the validation ambiguity introduced by `occurrence[x] 0..*` (e.g., what does it mean if both an `occurrencePeriod` and an `occurrenceDateTime` are present?) and keeps queries simple: a search parameter on `ServiceRequest.period` is unambiguous, whereas a search on `occurrence` over a `0..*` choice with multiple datatypes is awkward. Trade-off vs. Disposition A: this is a larger ask of O&O (new element, new search parameter, R5/R6 timing) and may require an extension as a stopgap; Disposition A is cheaper but layers more meaning onto an existing element.

---

### Disposition C: Decline

#### Proposal

Decline the change and retain the STU2 design that maps "Service Start Date" and "Service End Date" to `ServiceRequest.occurrenceTiming.boundsPeriod`. Rationale category: *already addressed by existing mechanism*. Document the rationale on `StructureDefinition-ServiceRequest-eltss` so future implementers understand why `boundsPeriod` (rather than `occurrencePeriod`) is the binding.

#### Justification

The 2020 disposition ("Considered for Future Use", 12-0-1) and bheale's April 2024 comment both recognize that `Timing.boundsPeriod` is explicitly described in the FHIR specification as a way "to ascribe a start and end date" — the example in the element documentation literally shows a service-window use of `boundsPeriod` alongside `dayOfWeek` and `timeOfDay`. Because eLTSS needs to carry both the service window *and* the recurrence/quantity pattern in the same `ServiceRequest`, and `occurrence[x]` is `0..1`, `occurrenceTiming.boundsPeriod` is the only place those two facts can coexist today without a core change. Declining avoids a breaking change for STU2 implementers, avoids an external dependency on O&O accepting a core resource change, and keeps the IG self-consistent.

---

### Recommendation

**Recommended disposition:** A

The reporter and Lynn Laakso have correctly identified that the current STU2 mapping is a workaround forced by `ServiceRequest.occurrence[x]` being `0..1`. Disposition C is honest about the constraint but cements a design that most implementers will continue to find unintuitive — `Period` is the obvious FHIR datatype for a service window and `Timing` is the obvious datatype for a recurrence pattern. Disposition B is conceptually cleaner but is a heavier core change (new element, new search parameter, ballot exposure) and is therefore a longer path to relief for eLTSS. Disposition A — relax the choice to `0..*` and re-bind eLTSS to `occurrencePeriod` for start/end while keeping `occurrenceTiming` for the schedule/quantity payload — is the smallest change that actually solves the reporter's problem, is backward compatible (`0..1` instances remain valid under `0..*`), and lets eLTSS update its mapping with a localized profile change. The work group should bring Disposition A to Orders and Observations as the primary ask, with Disposition B held as a fallback if O&O resists overloading `occurrence[x]`.
