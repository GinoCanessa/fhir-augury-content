## Summary

John Moehrke requests that `Consent.provision` be able to express
location-scoped permits and denials, including the **negation** of a
location ("NOT this Location"). The motivating use case is U.S. state
privacy regimes that distinguish accesses originating from inside the
state from those originating outside, where a consent must be able to
say e.g. "permit access only when the requester is in State X" or
"deny access whenever the requester is outside State X." The reporter
notes that the existing `consent-location` extension supports the
positive case ("at this Location") but provides no way to express the
negative case. He suggests folding location handling into
`provision.actor`, while explicitly cautioning that "NOT this Location"
is **not** semantically the same as "NOT this actor" — i.e. a
location-negation cannot simply be expressed by re-using the existing
actor-negation pattern. There are no linked Jira tickets, no Zulip
threads, and no comments on the ticket; cross-referencing surfaces only
the `Consent.provision` artifact itself.

## Details

**Description:**

> Some states have rules that distinguish accesses from within the
> state from those outside the state. Thus a Consent.provision needs to
> be able to indicate a provision by Location, and also by NOT Location.
>
> There seems to be an extension consent-location; but there is no way
> to indicate NOT Location.
>
> This Location logic seems is needed more than an extension would
> indicate. So should be added to provision.actor
>
> Although the idea of NOT this Location is not the same as NOT actor.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add first-class location semantics to `Consent.provision` in
`HL7/fhir`, **including a negation flag**, by introducing a new
backbone slice `Consent.provision.location` with at minimum:

- `Consent.provision.location.reference : Reference(Location)`
  (cardinality `1..1`).
- `Consent.provision.location.exclude : boolean` (cardinality `0..1`,
  default `false`) — when `true`, the provision applies when the
  requester is **not** at the referenced location ("NOT Location"
  semantics).
- Repeating the backbone (`0..*`) so a single provision can list one or
  more in-scope or excluded locations.

Author the change as a `<differential>` edit on the Consent
StructureDefinition under `source/consent/`, leaving the publisher's
`<snapshot>` regeneration untouched, and update
`source/consent/consent-introduction.xml` and
`source/consent/consent-notes.xml` to document the new element,
including an explicit note that "NOT location" is distinct from "NOT
actor" (the reporter's own caveat) and how evaluators must combine it
with `provision.actor` and `provision.purpose`. Coordinate retirement
or deprecation of the `consent-location` extension in
`HL7/fhir-extensions` (since the new core element supersedes it) and
add a `Consent.provision.location` example in `source/consent/`
exercising both the positive and the `exclude=true` form.

#### Justification

The reporter — the long-time CBCC steward of Consent — explicitly says
the location dimension "is needed more than an extension would
indicate." The existing `consent-location` extension was already a
recognition that location is an evaluation dimension; promoting it to
core lets the resource carry the negation semantics the extension was
unable to express in a clean, modifierless way. Modelling
location-negation as a sibling `exclude` flag (rather than overloading
the existing `provision.actor` with a "NOT this actor" pattern)
respects the reporter's own caveat that location-negation is not the
same as actor-negation, and it sidesteps the publisher gotcha that
only `<differential>` content is hand-edited (the change is purely
additive at the differential level). Co-locating the change with the
narrative updates in `consent-introduction.xml` and `consent-notes.xml`
follows the FhirCore briefing's standing guidance that authoring sits
in `source/<resource>/` only.

---

### Disposition B: Alternative Approach

#### Proposal

Keep location handling in the **FHIR Extensions Pack**
(`HL7/fhir-extensions`) rather than promoting it to core, and resolve
the gap by **revising the existing `consent-location` extension to
carry both a `Location` reference and a sibling `exclude : boolean`
modifier-style child** (or by publishing a companion extension
`consent-location-exclude` that sits next to the existing one). Update
the FhirCore narrative in `source/consent/consent-introduction.xml` to
cross-link the revised extension(s) and to document the
provision-by-location and provision-not-by-location patterns by
example, but make no `<differential>` change to the Consent resource
itself. Specifically:

- In `HL7/fhir-extensions`, edit the `consent-location` StructureDefinition to add an `exclude` modifierExtension child (or define a new `consent-location-exclude` extension whose context is `Consent.provision`); add a tracker for backwards compatibility for any implementer already using the bare `consent-location` extension.
- In `HL7/fhir`, limit the change to a narrative paragraph in `source/consent/consent-notes.xml` describing how to combine the extension(s) with `Consent.provision.actor` and `Consent.provision.purpose`, and a note in `source/consent/consent-introduction.xml` that location-based provisions are expressed via extension.

#### Justification

`Consent.provision` is already an unusually wide constraint surface
(actor, action, purpose, class, code, dataPeriod, data,
securityLabel). Adding a brand-new core dimension for location raises
the design and review bar substantially: it forces every implementer
of Consent — including the IGs flagged in cross-refs (davinci-epdx and
the wider Da Vinci suite) — to consider it, and it locks the negation
semantics ("exclude" vs. an alternative such as a coded modifier) in
core where it cannot easily be revised. Keeping the dimension in
extensions preserves the option to iterate on the negation pattern
based on real implementer feedback before promoting it. The trade-off
versus Disposition A is reach: a core element is discoverable and
testable in conformance suites, whereas an extension can be ignored.
Implementers in the U.S. state-rules use case the reporter cites are
typically already extension-aware, which mitigates that reach gap.

---

### Disposition C: Decline

#### Proposal

Decline the request to model location at the level of
`Consent.provision`. State explicitly in the disposition that:

1. The base FHIR Consent resource is intentionally narrow; jurisdictional
   reach (in-state vs. out-of-state) is a **policy** concern best
   expressed by the `Consent.policy` / `Consent.policyRule` bindings
   (or by an Implementation Guide / consent directive), not by a
   structural element on every provision.
2. The existing `consent-location` extension remains available for
   implementers who need to attach a Location to a provision; the
   absence of "NOT Location" can be addressed at the **policy
   interpretation** layer ("the policy applies in State X" already
   implies "does not apply outside State X"), not by adding a negation
   modifier.
3. Reopen with concrete implementer evidence (multiple states /
   jurisdictions / vendors who actually need this in interchange, not
   just in policy text) before considering a core change.

No change is made under `source/consent/`, the `consent-location`
extension is left as-is, and the ticket is closed as **Persuasive with
Mod / Not Persuasive** depending on workgroup preference.

#### Justification

Adding a structural axis to `Consent.provision` for every consent
artifact (when the use case is concentrated in U.S. state-privacy
scenarios) trades breadth of impact for narrow utility. There is also
a real risk of design churn: the reporter's own write-up wavers
between two designs ("add to `provision.actor`" vs. "NOT location is
not the same as NOT actor"), with no comments, no Zulip discussion,
and no linked tickets establishing community consensus on the shape.
Declining keeps the resource lean, preserves the existing
`consent-location` extension as the documented mechanism, and keeps
the door open to a better-scoped proposal once a wider implementer
community speaks up.

---

### Recommendation

**Recommended disposition:** A — Accept As Requested

The reporter is the WG's standing subject-matter expert on Consent and
explicitly argues that location is structural rather than incidental,
and the existing `consent-location` extension is itself evidence that
the dimension already keeps re-emerging. Disposition B (extension-only)
is tempting because it carries less blast radius, but it institution-
alises an asymmetric pattern — positive location lives in an extension,
the negation sits in a *second* extension or a modifierExtension —
which is exactly the kind of "death by extension" that
`Consent.provision` was designed to avoid. Disposition C (decline)
under-weights the U.S. state-law driver the reporter calls out. The
authoring impact in `HL7/fhir` is contained (a `<differential>` edit
plus two narrative pages, all within the `source/consent/` authoring
root identified in the briefing), the publisher gotchas around
generated `<snapshot>` content do not apply, and downstream IGs
(including the davinci-epdx renderings surfaced in cross-refs) re-render
automatically. The workgroup should treat the explicit semantic caveat
in Disposition A's narrative — that "NOT location" is not the same as
"NOT actor" — as a non-negotiable, since that is the precise point the
reporter spent the second half of the ticket making.
