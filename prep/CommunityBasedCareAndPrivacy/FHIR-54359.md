alternative flow. There are no cross-referenced Jira tickets, Zulip
threads, or GitHub items associated with this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the "Review Consent" diagram on the Use Cases page of the
Scalable Consent Management (FHIR) IG to include the Alternate Flow
that is currently described only in the narrative. Specifically, add
the branch/decision and the steps that constitute the Alternate Flow,
and depict the termination condition for the review loop so that the
diagram visually matches the textual description. Keep the existing
primary flow intact and label the new branch clearly (e.g., "Alternate
Flow — [trigger condition]") so readers can map the diagram one-to-one
to the narrative.

#### Justification

The reporter is asking for a documentation/diagram fix, and the ticket
has already been resolved as "Persuasive" with a note committing to
exactly this update ("we will add the alternative flow to the
diagram"). Diagram/text alignment is a basic IG-quality expectation:
when a Use Cases page describes an Alternate Flow but the diagram
omits it, implementers are forced to reconcile the two themselves,
which leads to inconsistent reads of the spec. Accepting as requested
is low risk (no normative or schema change) and discharges the
explicit triage commitment.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding the Alternate Flow to the existing diagram,
restructure the Review Consent section so that the primary flow and
each alternate/exception flow are presented as separate, smaller
diagrams (one per flow) with shared swim-lanes/actors, plus a short
introductory diagram that shows the high-level decision points and
links to each detail diagram. Update the narrative to cross-reference
the matching diagram for every flow, and explicitly call out the loop
termination condition both in text and in the per-flow diagrams.

#### Justification

A single combined diagram that tries to show the primary flow, every
alternate flow, and loop termination tends to become dense and hard
to read, especially as more alternates are added in future versions.
Splitting by flow keeps each diagram focused, makes the
narrative-to-diagram mapping unambiguous, and is easier to maintain
when individual flows change. The trade-off versus Disposition A is
more authoring effort up front and slightly more page real estate,
but it scales better and is friendlier to readers who only care about
one path through Review Consent.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the Alternate Flow is already
fully described in the narrative of the Review Consent section, and
that the diagram is intended to convey only the primary ("happy path")
flow. Note in the resolution that the narrative is the normative
description of behavior and that the diagram is informative only.

#### Justification

If the IG's editorial convention is "diagrams illustrate the primary
flow; alternates and exceptions live in the narrative," then this
ticket is asking the diagram to take on a role it was not designed
for, and accepting it could create an expectation that every alternate
flow in every section must also be diagrammed — an open-ended
documentation burden. Declining is defensible *only* if such a
convention is documented and consistently applied elsewhere in the
guide; absent that, this rationale is weak.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been triaged "Persuasive" with the work group
explicitly committing to add the Alternate Flow to the diagram and to
show when the review loop ends, so Disposition A simply executes that
commitment. The change is purely editorial, has no normative or
conformance impact, and removes a real diagram/text mismatch that
would otherwise confuse implementers of Scalable Consent Management.
Disposition B is a reasonable longer-term improvement to consider
during a broader Use Cases refactor, but it is out of proportion to
what this single ticket requests and would expand scope unnecessarily.
Disposition C is hard to defend because the IG already commits the
narrative and diagram to describe the same flow in this section; the
mismatch is the bug, not a feature.
