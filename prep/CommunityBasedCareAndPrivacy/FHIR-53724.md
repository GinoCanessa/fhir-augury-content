## Summary

FHIR-53724 reports that the `Consent.policyRule` element — present in
R4 — has been removed from R5/R6, but stale references to `policyRule`
remain in the Consent resource's narrative text (notably in the
"Optimal Searching" section, where the remaining `policyRule` mention
should refer to `.policyBasis`). The reporter (John Moehrke) wants the
narrative cleaned up to reflect the current model and to clearly
explain how the replacement elements relate:

- `regulatoryBasis` — a code identifying the regulations on which the
  consent (or its backing policy) is based.
- `policyBasis` — points to the *organisation/jurisdiction-level*
  computable policy this consent is derived from (e.g. XACML, ODRL,
  or a FHIR `Permission`), via `.reference` or `.uri`.
- `policyText` — a reference to the *human-readable* policy.

Beyond fixing the narrative, the ticket asks that the resource provide
a way to point at *patient-specific* rules expressed in an external
standards language (XACML / ODRL) for implementers who do not want to
populate `.provision`. This is the same gap the related ticket
[FHIR-48988](https://jira.hl7.org/browse/FHIR-48988) — "Consent using
XACML example" — surfaced: John's own follow-up comment on FHIR-48988
notes that overloading `.policyBasis.url` with a per-patient rule URL
breaks the "find every consent based on last year's backing policy"
search use case, because `.policyBasis` is by definition the
*patient-independent* backing policy. So FHIR-53724 is effectively
asking the workgroup to (a) scrub the obsolete `policyRule` references
from Consent narrative and (b) decide on a clean home for
patient-specific external rule documents that does not collide with
`.policyBasis`.

## Details

**Description:**

While working on https://jira.hl7.org/browse/FHIR-48988, to provide an
example of a Consent that uses XACML rather than `.provisions` (see
comment in that ticket too):

Consent used to have a `.policyRule` element. This element is now
gone, but it is still referenced in narrative text. The `policyRule`
seems to have been renamed but not clear where the functionality went.

The new `regualtoryBasis` is more clear that it is a code indicating
the regulations upon which this consent (or backing policy) [is
based].

The new `policyBasis` is more clear that it is the indication of the
specific policy this consent is based upon (thus the policy language
that is derived from local business rules, medical records rules,
standards of practice, and the regulations indicated in
`.regulatoryBasis`). The new `policyBasis.reference` or `.uri` could
point at the computable policy that the organization has written.
Likely in XACML, ODRL, or FHIR `Permission`.

The new `policyText` is more clear that it is a reference to the human
readable policy.

`policyRule` is wrongly mentioned in the Optimal Searching section,
where I think it should be referring to `.policyBasis`.

There should be a way to point at the patient specific rules in
another standards language like XACML or ODRL; for those that don't
want to use the `.provision` element.

## Proposed Dispositions

(No GitHub cross-references / repo briefings were available, so the
authoring-root and file-path callouts below are inferred from the
artifact name — Consent — and standard FHIR Core layout. They should
be confirmed against `HL7/fhir` once a briefing exists.)

### Disposition A: Accept As Requested

#### Proposal

Make two coordinated edits to the `Consent` resource in FHIR Core
(R6 ballot):

1. **Narrative cleanup** — sweep the Consent intro / notes / examples
   pages for every remaining mention of `policyRule` (in particular
   the "Optimal Searching" section called out by the reporter) and
   replace each with the correct successor element. In the Optimal
   Searching section specifically, change `policyRule` → `policyBasis`
   when discussing the URL/reference to the backing computable
   policy, and change to `regulatoryBasis` where the prose is talking
   about the regulation code. Add a short "Migration from R4" note in
   the Consent intro mapping `R4 policyRule` → `R5/R6 regulatoryBasis
   + policyBasis + policyText`, with a one-liner per element. Also
   confirm the search-parameter table reflects today's element names.
2. **Patient-specific external rules** — add an explicit element
   (e.g. `Consent.patientPolicy` of type `Reference(DocumentReference
   | Permission) | url`, 0..1 or 0..*) intended to carry the
   patient-specific computable rules in XACML / ODRL / FHIR
   `Permission`. Document clearly that this is the *per-patient*
   companion to the *patient-independent* `.policyBasis`, and that
   either `.provision` **or** this new element (or both) MAY be
   populated.

#### Justification

The narrative half of this is unambiguous documentation debt: the R4
element was removed but the prose was not updated, which directly
misleads implementers (the reporter found this while building the
example requested by FHIR-48988). The new element half is the minimal
change that enables the FHIR-48988 use case without overloading
`.policyBasis`, preserving John's "find all consents based on last
year's backing policy" search semantic that he flagged in his
FHIR-48988 comment and again in the related Zulip thread. Splitting
patient-independent vs. patient-specific into two distinct elements
matches the conceptual model already documented for `policyBasis` and
keeps `policyText` (human-readable) cleanly parallel.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the narrative cleanup from Disposition A in full, but **do not
add a new core element** for patient-specific external rules.
Instead:

- Define a **standard core extension** (e.g.
  `http://hl7.org/fhir/StructureDefinition/consent-patientPolicy`) on
  Consent of type `Reference(DocumentReference | Permission) | url`,
  documented in the Consent extensions registry with the same
  semantics described in Disposition A.
- Update the Consent intro to point implementers who need this
  capability at the extension, with a worked snippet, and use the
  extension in the FHIR-48988 XACML example.
- Tighten the descriptions of `regulatoryBasis`, `policyBasis`, and
  `policyText` so the boundaries among the three are unambiguous, and
  state explicitly that `.policyBasis` is *patient-independent* (this
  is the assertion the reporter and the Zulip thread both rely on).

#### Justification

R6 is already in ballot churn for Consent, and adding a brand-new core
element invites another round of FMM/maturity discussion plus
implementer migration. A core extension delivers the same expressive
power for the relatively narrow population of implementers who choose
XACML/ODRL over `.provision`, keeps the base resource lean, and
avoids the risk that a new core element collides with the work the
group is already doing on `.provision`. It also lets the workgroup
ship the narrative fix — which is the higher-value, lower-risk half
of the request — immediately, without being blocked on element
design.

---

### Disposition C: Decline (split: accept narrative, decline new element)

#### Proposal

Treat the request as two separable issues. **Accept** the documentation
fix (replace `policyRule` with `policyBasis` / `regulatoryBasis` /
`policyText` in narrative, especially the Optimal Searching section)
as an editorial/Persuasive change. **Decline** the request for a new
mechanism to carry patient-specific XACML/ODRL rules, on the grounds
that `Consent.provision` already exists for patient-specific rules and
that implementers wanting an external standards language can either
(a) reference the external policy from a `DocumentReference` linked
via `Consent.sourceReference`, or (b) use a `Permission` resource
linked from `policyBasis.reference`, treating each patient's
permission instance as the "computable backing policy" for that
consent.

#### Justification

Adding a new element risks fragmenting how implementers describe
consent rules and creates two parallel-but-not-quite-equivalent ways
to say the same thing (one in `.provision`, another in the new
element). The Zulip thread shows the workgroup is already wrestling
with overlapping semantics among the existing rule-bearing elements;
piling on more is risky. The narrative fix is genuinely needed and
should not be held hostage to the element-design discussion.

---

### Recommendation

**Recommended disposition:** B

The narrative half of FHIR-53724 is plainly correct and should land
regardless — leftover R4 element names in normative-track prose are
actively misleading and the reporter has already pinpointed the
offending section. The disagreement is really about the
patient-specific external-rule slot. Disposition C is too tight: the
reporter's own analysis on FHIR-48988 shows that `.policyBasis` is the
*backing* (patient-independent) policy and that reusing it
per-patient breaks a documented search use case, and `.provision` is
exactly the structured form the implementer is trying to avoid — so
"just use what exists" doesn't actually solve FHIR-48988. Disposition
A solves it but at the cost of a new core element late in the cycle,
with all the FMM, mapping, and search-parameter ripple that implies.
Disposition B threads the needle: ship the narrative fix now,
deliver the missing capability via a standard core extension that the
FHIR-48988 example can demonstrate immediately, and revisit promotion
to a core element in a future cycle once there is real implementer
uptake. It also lines up with John Moehrke's own Zulip-stated
preference for moving lower-value rule metadata out of the base
resource (he floated converting `policyBasis` itself into a core
extension on the same grounds).
