## Summary

The reporter asks the Scalable Consent Management (FHIR) IG editors to
review and harmonize the names of the IG's CapabilityStatement resources
so they are consistent with each other and with the actor names used
elsewhere in the specification. She points out that the current
capability statements use vague terms such as "client" and "server"
without an "actor system" qualifier, and that the spec mixes variants
like "consent admin service", "consent administrative service", and
"client admin service" (the last of which she notes is incorrectly
attached to the server side). The request is that the IG pick a single,
clearly-defined set of actor / actor-system names and apply them
consistently across prose, use cases, and CapabilityStatement titles
and IDs to improve readability.

The reporter explicitly cross-references "my other issue on
standardizing and defining actor and actor systems and use
consistently" — that companion ticket is not surfaced by the
cross-reference index, but it sets the broader context: this ticket is
the CapabilityStatement-side rollout of a wider naming-cleanup effort
the reporter is driving on the same IG. The closely-related ticket
[FHIR-54738](https://jira.hl7.org/browse/FHIR-54738) (also from
Virginia Lorenzi against the same spec) covers analogous
naming/definition cleanup for "FAST Consent" vs "Scalable Consent
Management" and undefined acronyms ("PAS"), confirming a consistent
theme of terminology hygiene in this IG.

## Proposed Dispositions

The ticket has already been marked **Resolution: Persuasive** with a
recorded resolution that introduces "Consent Client" and "Consent
Server" actors and propagates those names into the
CapabilityStatements and the use cases. The dispositions below are
framed against that resolution so the workgroup can confirm or adjust
it. Because the ticket has no related GitHub repository in the
cross-reference index, no `repo-analysis` briefing is available; the
proposals below name spec artifacts (CapabilityStatements, actor
definitions, use-case prose) rather than concrete file paths in a
known authoring root.

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution that is already recorded on the ticket:

1. Define two canonical actors in the Scalable Consent Management IG:
   **Consent Client** and **Consent Server** (with optional
   "*-system" qualifiers if other tickets in this naming pass settle
   on that convention). Capture them in a single actor-definition
   page and reference that page from every other page that mentions
   an actor.
2. Rename every CapabilityStatement in the IG so its `name`,
   `title`, and `id` consistently reflect those two actors — for
   example `CapabilityStatement-ConsentClient` and
   `CapabilityStatement-ConsentServer`, replacing the current vague
   "client" / "server" names. Add narrative on each
   CapabilityStatement page identifying which actor it constrains.
3. Sweep the IG narrative and use cases to collapse the variants
   "consent admin service", "consent administrative service", and
   "client admin service" into a single chosen term tied to the
   correct actor. Where a former "client admin service" was
   describing server-side behaviour, retag it explicitly to the
   Consent Server actor.
4. Add a glossary / actors section that lists the canonical names and
   explicitly deprecates the old variants so future authors do not
   reintroduce them.

#### Justification

The existing capability-statement names are objectively inconsistent
both internally (the misnamed "client admin service" actually applies
to a server-side capability) and with the rest of the IG (use cases
and prose use richer names). FHIR IG conventions strongly favour one
CapabilityStatement per actor with a name that includes the actor —
this is the same pattern used by US Core, IPS, SMART App Launch, and
most HL7 IGs — so the proposed rename brings the IG in line with
community practice. The reporter's adjacent ticket
([FHIR-54738](https://jira.hl7.org/browse/FHIR-54738)) shows the IG
already has copy-and-paste terminology drift, so a single canonical
actor list is the right anchor for that broader cleanup.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the request but defer the concrete name choice
to the broader actor-naming ticket the reporter alludes to ("my other
issue on standardizing and defining actor and actor systems"). In
this disposition:

1. Treat FHIR-54925 as a child of that umbrella ticket and bundle
   both into a single naming-cleanup change set so the actor names,
   actor-system names, and the CapabilityStatement names are decided
   together rather than in two passes.
2. As an interim, make only the unambiguous corrections in this
   ticket — specifically, fix the "client admin service" misattribution
   (rename it to whatever the chosen Consent Server-side admin
   capability is called) and remove the "consent admin service" /
   "consent administrative service" variant collision by picking one
   spelling.
3. Park the broader rename of the per-actor CapabilityStatements
   (e.g., adoption of "Consent Client" / "Consent Server") until the
   actor-system naming ticket lands, so the IG only churns the
   CapabilityStatement IDs once.

#### Justification

If the reporter's referenced "actor and actor systems" ticket is
still open, locking in CapabilityStatement names here and then
revisiting them when that ticket resolves means two waves of
ID/name changes for implementers and two rounds of broken inbound
links. Bundling lets the workgroup choose between "Client / Server"
and "Client System / Server System" once, with the
CapabilityStatements following the decision, which directly answers
the reporter's complaint that "client and server is too vague"
without committing to specific names before the umbrella decision is
made.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54925 as **Not Persuasive**, on the basis that
"client" and "server" are the conventional and widely-understood
actor labels in HL7 capability statements (see the FHIR core
CapabilityStatements `capabilitystatement-base` and
`capabilitystatement-base2`, and equivalents in US Core, SMART, and
IPS), so the current naming is consistent with the broader
ecosystem even if it is terser than the reporter prefers. The
mis-attributed "client admin service" name (which actually
describes server behaviour) would be addressed as a narrow
technical correction in a separate ticket without renaming the
other CapabilityStatements.

#### Justification

This option is defensible only on the narrowest reading of the
ticket. It would, however, leave standing the documented
"consent admin service" vs "consent administrative service"
collision and the "client admin service" misattribution, both of
which the reporter calls out and neither of which is defensible on
its own merits. It is presented for completeness because the
workgroup may prefer to keep generic FHIR-style "client / server"
actor names and address only the bugs.

---

### Recommendation

**Recommended disposition:** A

The ticket is already triaged as Persuasive with a clear, recorded
resolution ("create Consent Client and Consent Server actors and use
them in the use case and update the capability statements to use
those names as well") that directly addresses the reporter's three
concrete complaints: the vagueness of "client / server", the
"consent admin service" vs "consent administrative service"
spelling drift, and the "client admin service" misattribution.
Disposition B is sensible in principle but only pays off if the
reporter's umbrella actor-naming ticket is still open and likely to
re-bikeshed the names — and that umbrella ticket does not show up in
the cross-reference index, so deferring on its behalf is
speculative. Disposition C is hard to justify because it leaves the
acknowledged inconsistencies in place. Accepting as requested
implements the recorded resolution, aligns the IG with standard FHIR
"one CapabilityStatement per actor" practice, and gives the wider
naming pass (FHIR-54738 and the alluded-to actor ticket) a stable
anchor to build on.
