## Summary

The reporter notes that Section 9.1.3 of the Scalable Consent Management
(FHIR) specification — the section that introduces the FAST Consent
Subscription Topic — is missing its descriptive prose. The
`SubscriptionTopic` instance is presented without the explanatory
narrative that other Subscription Topics in the IG carry, leaving
implementers without context on the topic's purpose, the events it
covers, or how it relates to the consent workflow. The ticket is
editorial in nature and is tracked under the
`Infrastructure/Editorial` label. No cross-referenced Jira tickets,
Zulip discussions, or GitHub items were surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add the missing narrative description to Section 9.1.3 of the Scalable
Consent Management (FHIR) IG so it accompanies the FAST Consent
`SubscriptionTopic` instance. Concretely, author a short paragraph (one
to three sentences) on the `SubscriptionTopic` instance's intro / page
content that:

1. Names the topic and identifies it as the FAST Consent Subscription
   Topic.
2. States the trigger / resource(s) it observes (e.g., changes to
   `Consent`).
3. Describes its intended use within the Scalable Consent Management
   workflow (notifying interested parties when a consent is created,
   updated, or revoked).

This matches the resolution already recorded on the ticket
("We will add a description for the Subscription Topic instance.")
and the `Infrastructure/Editorial` labelling.

#### Justification

The omission is purely editorial: the `SubscriptionTopic` artifact
exists, but its explanatory text does not. Adding the description has
no normative impact, no schema impact, and no implementation burden;
it improves IG readability and brings Section 9.1.3 in line with
sibling sections that already document their topics. The ticket's
existing resolution proposal is exactly this, the reporter has not
contested the wording, and there is no Zulip / GitHub discussion
suggesting any disagreement.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only writing prose into Section 9.1.3, expand the fix to
populate the `SubscriptionTopic` resource's own narrative-bearing
elements (`description`, and where useful `purpose` / `useContext`)
and let the IG page render that content. The Section 9.1.3 page would
then either render the resource's `description` directly or include a
short pointer plus the rendered resource view.

#### Justification

Putting the description on the `SubscriptionTopic` instance itself
makes the explanation travel with the artifact when it is consumed
outside the IG (e.g., loaded into a server, browsed in a registry, or
referenced by another IG). This is consistent with general FHIR
guidance that conformance resources should carry their own human-
readable description. The trade-off vs. Disposition A is a slightly
larger change (touching the resource JSON / FSH in addition to the
page text) and a small risk of duplicating prose between the resource
and the section. For an editorial ticket, that overhead is modest and
the durability win is real.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that Section 9.1.3 already renders
the `SubscriptionTopic` resource and a separate descriptive paragraph
is not strictly required for conformance.

#### Justification

This is the weakest option and is included only for completeness. The
ticket is editorial and low-cost; declining would leave a documented
gap in an IG section that the reporter found confusing, with no
offsetting benefit. There is no breaking-change risk, no scope
concern, and no existing mechanism that obviates the fix.

---

### Recommendation

**Recommended disposition:** A

The ticket is a small, uncontested editorial fix and the work group
has already recorded "Persuasive" intent on the Jira side, with the
resolution wording matching Disposition A almost verbatim. Disposition
B is a reasonable enrichment and could be folded in as an
implementation detail (i.e., when adding the section text, also
populate the resource's `description` so the prose lives in one
place), but it should not block accepting and closing the ticket.
Disposition C has no upside given the trivial cost of writing the
missing paragraph.
