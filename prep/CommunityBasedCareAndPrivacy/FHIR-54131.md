## Summary

The ticket reports that within the Scalable Consent Management (FHIR)
profile, the `Consent.provision` element's documentation is internally
inconsistent: in one definition/description it is called a "constraint",
while in another it is called an "exception". Hans Buitendijk notes that
these are conceptually two different things, and asks which meaning is
actually intended for `provision` in this profile. Although the reporter
acknowledges this is largely a FHIR Core concern (and a separate ticket
will be filed against FHIR R6), this ticket is scoped to the Scalable
Consent Management profile so the workgroup can clarify behavior for
implementers using the IG today. The cross-reference index returned no
related Jira tickets, Zulip threads, or GitHub items, so the request is
being evaluated on its own.

## Details

**Description:**

> While this mostly is a FHIR Core topic (a separate Jira will be
> submitted to FHIR R6), the provision attribute uses to different terms

> definition/description the term "constraint" is used and the other
> "exception". Those are two different concepts. Which definition is

**Resolution Description (already recorded on the ticket):**

> We can't change the base definitions but we will add a note on the
> profile that indicates that the terms "constraint" and "exception" are

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the request literally: pick one of the two terms
("constraint" or "exception") as the canonical term for
`Consent.provision` semantics in the Scalable Consent Management
(FHIR) profile, and rewrite the profile-level definition and any
element-level descriptions of `Consent.provision` so that only the
chosen term is used. Where the underlying base resource still uses the
other term, override the description on the profile's
StructureDefinition `differential` (and regenerate the snapshot) so that
implementers reading the profile see consistent language. Add a short
note in the profile narrative explaining which term the IG uses and why.

#### Justification

This is the most direct response to what the reporter actually asked:
"Which definition is actually used in this profile?" Choosing one term
and applying it everywhere removes the ambiguity rather than just
acknowledging it. FHIR profiles are explicitly allowed to tighten and
clarify element definitions, so overriding the description text on
`Consent.provision` is a well-supported mechanism. It also gives
implementers a single, unambiguous mental model for how provisions
behave in Scalable Consent Management without requiring them to consult
R4 base resource history.

---

### Disposition B: Alternative Approach (matches the recorded resolution)

#### Proposal

Leave the underlying base `Consent.provision` definitions untouched, and
instead add an explanatory note in the Scalable Consent Management
profile that explicitly states: in the context of this profile (and as
inherited from the R4 Consent resource), the terms "constraint" and
"exception" are used interchangeably and refer to the same concept.
Place this note both in the profile's narrative page and as a
`comment` / definition-extension on the `Consent.provision` element so
that it surfaces in generated implementation guides and tooling. Do not
attempt to change the base FHIR definitions in this ticket; the
reporter has already indicated a separate R6 Core ticket will track
that.

#### Justification

This is the disposition the workgroup has effectively already adopted
(the ticket is marked **Persuasive with Modification** with this exact
text). It honors the reporter's underlying need — implementers should
not be confused by the dual terminology — without taking on the
larger, cross-workgroup work of redefining `Consent.provision`
semantics in the base resource. It also avoids creating a situation
where the IG's terminology drifts from FHIR Core, which would be
confusing in its own right and would create ongoing maintenance
overhead each time the base Consent resource is updated. Keeping the
base wording and adding a clarifying note is a low-risk,
non-breaking change that is appropriate for a published profile.

---

### Disposition C: Decline

#### Proposal

Close the ticket as out of scope for the Scalable Consent Management
profile and defer the entire concern to the forthcoming FHIR R6 Core
ticket the reporter has indicated will be filed. Do not modify the
profile narrative or element definitions; mark the ticket as Not
Persuasive (or similar) with a pointer to the R6 Core ticket once it
exists.

#### Justification

The reporter himself characterizes this as "mostly … a FHIR Core
topic". One could argue that having two profiles' worth of clarifying
text — one in Scalable Consent Management, another (eventually) in
FHIR R6 — risks divergent wording over time and that a single
authoritative fix in Core is cleaner. Declining here would force the
fix to live in exactly one place. The trade-off is that implementers
using the current profile would be left without any clarification in
the meantime, which is the very problem the ticket raises.

---

### Recommendation

**Recommended disposition:** B

Disposition B matches the resolution the workgroup has already recorded
on the ticket ("Persuasive with Modification") and is the right balance
of responsiveness and restraint. The reporter's actual pain point is
implementer confusion when reading `Consent.provision` documentation;
that is fully addressed by an explicit note clarifying that "constraint"
and "exception" are used interchangeably in this profile (consistent
with R4 base usage). Disposition A, while more thorough, would have the
profile diverge from base FHIR terminology and create an ongoing
maintenance burden every time Core text changes — and it does not
actually resolve the underlying inconsistency, which lives in the base
resource. Disposition C leaves implementers without guidance until the
R6 Core ticket lands, which is a regression from where the workgroup
already is. Disposition B can be implemented quickly (a narrative
sentence plus an element-level comment), is non-breaking, and leaves
the deeper Core question to the R6 ticket the reporter has flagged.
