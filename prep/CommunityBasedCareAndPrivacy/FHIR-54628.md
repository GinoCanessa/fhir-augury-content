## Summary

John Moehrke flags multiple problems with the `consent-location`
extension in the FHIR Extensions Pack. He lists five concerns: (a) the
extension's narrative still uses the legacy term "exception" instead of
the current `provision`; (b) the binding point is ambiguous — he argues
it should sit on `Consent.provision.actor.reference` so it clearly
constrains "requests originating from this location" rather than
floating at the `provision` level; (c) when used at the `provision`
level it is not clear whether it scopes the *data* held at that location
or *requests originating from* that location; (d) it should arguably be
a modifier extension because changing the location materially changes
the meaning of a permit/deny rule; and (e) the location concept may be
important enough to be promoted to a first-class element on
`Consent.provision` rather than living as an extension.

This ticket is best read alongside the same-day, higher-priority
[FHIR-54627](https://jira.hl7.org/browse/FHIR-54627) ("Consent
provisions that are location specific"), in which Moehrke argues that
location semantics — including a "NOT this location" capability — are
strong enough that they should be added to the core `Consent.provision`
model rather than continuing to live as an extension. FHIR-54628
essentially supplies the editorial / placement / cardinality critique
that motivates the structural change requested in FHIR-54627. The
extension was originally introduced via
[FHIR-10712](https://jira.hl7.org/browse/FHIR-10712) (Persuasive with
Modification), which created `consent-location` as a "standard
extension" on what was then called `Consent.exception` — directly
explaining issue (a) in this ticket.

## Details

**Description:**

The consent-location extension

a. is still described using old language of "exception" rather than
"provision"

b. is unclear where this extension goes. It would seem best to be on
`provision.agent.reference`; not on `provision`. This would make it
clear that it applies to requests from that location.

c. If this extension is used on provision level, then what does it
mean? Is it meaning data within that location? Is it meaning requests
from that location?

d. this extension would seem should be a modifier extension

e. location concept might be strong enough to be an element rather than
extension

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply five surgical changes to `consent-location` in
`HL7/fhir-extensions`:

1. **(a)** Rewrite the `StructureDefinition.description`, `purpose`,
   `comment`, and any context narrative to replace every occurrence of
   "exception" with "provision".
2. **(b)** Change `Context.expression` from
   `Consent.provision` to `Consent.provision.actor.reference` so the
   extension only attaches to an actor reference. Remove
   `Consent.provision` from the context list.
3. **(c)** State explicitly in the `purpose` and `comment` text that
   the extension constrains the *physical/network location of the
   actor making the request*, not the location at which the data is
   stored or the patient is treated.
4. **(d)** Set `StructureDefinition.snapshot.element[0].isModifier` =
   `true` and add an `isModifierReason` explaining that the location
   constraint changes whether a permit/deny provision applies.
5. **(e)** Open a companion change in FHIR Core (already filed as
   FHIR-54627) to add a first-class `Consent.provision.actor.location`
   element, and once core has it, deprecate `consent-location` in the
   Extensions Pack with a redirect note.

#### Justification

Each item maps 1:1 to the reporter's bullets and is locally bounded.
(a), (b), and (c) are pure narrative / context fixes with no
breaking-on-the-wire impact. (d) is a normative change — flipping an
extension to modifier — which is defensible: a location restriction
*does* change whether a provision applies, which is the
canonical FHIR test for `isModifier`. (e) acknowledges that the right
long-term home for "location of access" is the core resource (per the
companion FHIR-54627), without blocking the immediate hygiene fixes
that this ticket calls out.

---

### Disposition B: Alternative Approach — Editorial Fix Now, Defer Structural Change

#### Proposal

Take only the safe, non-breaking parts of the request now, and split
the rest off:

1. Apply (a) and (c) — pure narrative cleanup in
   `HL7/fhir-extensions`: stop using "exception", and add explicit
   semantic guidance that the extension constrains the *requesting
   actor's location*.
2. Tighten (b) by **adding** `Consent.provision.actor.reference` to
   the context list (so consumers get clear guidance), but **leave**
   `Consent.provision` in place to preserve backward compatibility
   with existing instances.
3. Defer (d) — do *not* flip to `isModifier` in this cycle, because
   doing so retroactively invalidates every Consent currently using
   the extension that doesn't carry a `modifierExtension` slot.
4. Defer (e) entirely to FHIR-54627 and treat that ticket as the
   single decision point for "extension vs. core element"; mark this
   sub-issue Duplicate of FHIR-54627.

#### Justification

`isModifier` is the highest-cost flag in FHIR — once true, every
processor that doesn't understand the extension MUST reject the
resource. For a Trial-Use extension that has been published for years,
flipping that bit is a hard breaking change that should be justified
by demonstrated harm, not by intuition. Likewise, dropping
`Consent.provision` from the context expression invalidates any
existing instance that puts the extension at provision scope. This
disposition keeps the editorial wins, preserves existing instances,
and routes the structural debate (issue e) to the higher-priority
ticket where it already lives.

---

### Disposition C: Decline (Roll Into FHIR-54627)

#### Proposal

Close FHIR-54628 as **Not Persuasive – duplicate of FHIR-54627**, on
the grounds that the substantive question — *should "location of
access" be a core element on `Consent.provision.actor` rather than an
extension?* — is already being asked at higher priority by FHIR-54627,
and that the editorial nits in (a) and (c) will naturally be resolved
by whatever happens to the extension as a result of FHIR-54627
(promotion to core, retirement, or rewrite).

#### Justification

Five separate sub-points filed as one ticket make tracking and
balloting difficult, and three of the five (b, d, e) are really one
question: *what is the right shape of "location" in Consent?* That
question is owned by FHIR-54627. The remaining items (a, c) are
mechanical edits the editor can apply during the FHIR-54627 work
without needing a separate ballot disposition. Declining keeps the
workgroup's queue clean and avoids producing two parallel resolutions
that risk contradicting each other.

---

### Recommendation

**Recommended disposition:** B

The reporter's editorial complaints (legacy "exception" wording,
ambiguous semantics) are real and cheap to fix, so something *should*
ship this cycle — that argues against C. But Disposition A's
proposals to flip `isModifier` and to drop `Consent.provision` from
the context expression are silently breaking changes against published
content, and would require a separate substantive review on their own
merits. Disposition B captures the safe wins now (rename, clarify,
tighten guidance toward `actor.reference`), explicitly defers the
modifier-extension question until a use case demonstrates the
need, and routes the "promote to core element" decision to its proper
home in FHIR-54627. The workgroup can always reopen the deferred
sub-items if FHIR-54627 lands a different resolution.
