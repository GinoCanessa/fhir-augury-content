## Summary

This ticket asks the PACIO Re-Assessment Timepoints IG to move the canonical
URL of a Value Set out of the `hl7.org/fhir` namespace and anchor it instead
under `terminology.hl7.org` (THO), with a corresponding UTG ticket created to
register the value set in THO. The request reflects the long-standing HL7
convention that only value sets bound (typically with `required` strength) to
a `code`-typed element should live under `hl7.org/fhir`; all other value sets,
particularly those reused or potentially reusable across IGs, should be
hosted in THO. The ticket's title also notes the qualifier "Only those with
required binding to a 'code' data type should be anchored in hl7.fhir.org",
which is the policy basis for the move. No cross-referenced Jira, Zulip, or
GitHub items were surfaced by the FhirAugury index, so the request stands on
its own; the existing comment trail shows the resolution was reverted post
STU1 publication and the issue was re-opened for re-disposition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the affected PACIO Re-Assessment Timepoints value set's canonical URL
from `http://hl7.org/fhir/...` into the `http://terminology.hl7.org/ValueSet/...`
namespace, and file a companion UTG ticket to register the value set (and any
new code system it depends on) in THO. Concretely:

1. Identify the specific ValueSet resource(s) in the PACIO Re-Assessment
   Timepoints IG whose canonical begins with `http://hl7.org/fhir/` and
   whose binding does not meet the "required binding to a `code` element"
   criterion.
2. Update each ValueSet's `url` to the new THO canonical
   (`http://terminology.hl7.org/ValueSet/<name>`), keep the existing OID as
   an `identifier`, and bump `version`.
3. Update every binding in the IG (StructureDefinitions, profiles,
   extensions, examples) to reference the new canonical.
4. File a UTG ticket (e.g., against the UTG project) requesting that the
   value set be created in THO at the new canonical, citing FHIR-35578.
5. Add a brief migration note in the IG narrative documenting the URL
   change.

#### Justification

This matches established HL7 terminology governance: value sets that are not
narrowly tied to a `required`-bound `code` element belong in THO so they can
be reused, version-managed, and referenced by other IGs without coupling
them to a specific FHIR release. The reporter and the re-opening comment by
lynn_laakso explicitly call for this move, and the original 9-0-0 vote to
*Consider for Future Use* was conditional on the IG and terminology maturing
— which has now happened post STU1. Accepting as requested aligns the IG
with the current HL7 publishing rules and unblocks downstream reuse.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than a blanket move to THO, perform a per-value-set triage and split
the resolution:

- **Move to THO** any value set in the IG that (a) is bound at strengths
  weaker than `required`, or (b) is bound `required` but to a non-`code`
  data type (e.g., `Coding`, `CodeableConcept`, `Quantity.code`), or (c)
  has any plausible reuse outside PACIO. File a UTG ticket per value set,
  or a single grouped UTG ticket listing them.
- **Keep under the IG canonical** (`http://hl7.org/fhir/us/pacio-rt/...`,
  i.e., the IG's own namespace, not the core `hl7.org/fhir/ValueSet/...`
  space) any value set that is genuinely IG-specific and unlikely to be
  reused. The "anchor in `hl7.org/fhir`" wording in the ticket really means
  the core-spec namespace; IG-local value sets can legitimately live under
  the IG's own canonical root.
- For each kept-local value set, document the rationale in the IG so the
  decision is auditable.

#### Justification

The ticket title states the rule precisely: "Only those with required
binding to a 'code' data type should be anchored in hl7.fhir.org". Applied
literally that rule sweeps every other value set into THO, but in practice
many IGs retain narrowly-scoped, non-reusable value sets under their own IG
canonical root rather than THO. A per-value-set triage honors the spirit of
the rule (don't squat on the core namespace, don't lock reusable
terminology to a single IG) while avoiding unnecessary THO churn for value
sets nobody else will ever consume. It also produces a cleaner UTG
submission because each candidate value set carries an explicit reuse
rationale.

---

### Disposition C: Decline

#### Proposal

Decline as *Not Persuasive — already addressed* on the basis that the
PACIO Re-Assessment Timepoints IG is published as STU1 and the URL
conventions in question are HL7 publishing-process policy enforced by the
IG Publisher / FMG, not a normative content issue with the IG itself.
Recommend that any future re-publication of the IG follow current FMG/UTG
guidance at that time, but make no targeted change in response to this
ticket.

#### Justification

Defensible if the WG concludes that (a) renaming canonicals after STU1
publication causes more downstream breakage (broken external references,
re-validation of conformant content) than the governance benefit warrants,
and (b) the underlying rule is a publishing-process policy that will be
re-applied as a matter of course on the next ballot cycle. The original
2022 vote of *Considered for Future Use* effectively took this position;
the re-opening was procedural rather than driven by new evidence. Note
that this disposition leaves the value set in the wrong namespace until
the next IG revision and is harder to defend now that the ticket has been
flagged Block-Vote-1.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible because it honors the literal policy
the ticket cites without forcing every PACIO value set into THO regardless
of reuse potential. The reporter is correct that value sets sitting under
`hl7.org/fhir/...` that are not bound `required` to a `code` element should
not be there — but the right destination is not always THO; an IG-local
canonical (under the IG's own root) is appropriate for genuinely
IG-specific value sets. A per-value-set triage produces a clean UTG
submission only for the value sets that warrant promotion, documents the
rationale for the rest, and brings the IG into compliance with current HL7
URL governance with the minimum disruption to existing implementers.
Disposition A is a reasonable fallback if the WG prefers a simpler "move
everything" rule, and Disposition C should be rejected because the ticket
has already been re-opened and labeled Block-Vote-1, signalling that
"defer to future republication" is no longer acceptable.
