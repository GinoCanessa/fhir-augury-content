## Summary

dependency on a newer version of US Core — specifically US Core 6.1 or
US Core 7 — to "future-proof" the guide as new regulatory requirements

## Proposed Dispositions

> Note: the ticket is already marked Resolution = Not Persuasive with a
> resolution comment indicating the IG was in fact moved to US Core
> 6.1.0. The dispositions below are framed for a fresh workgroup
> review of the underlying request (which version of US Core SCM
> should depend on), so the discussion can confirm or revisit that
> outcome.

### Disposition A: Accept As Requested

#### Proposal

Update the Scalable Consent Management IG's `ImplementationGuide`
resource so that its `dependsOn` element references **US Core 6.1.0**
(package id `hl7.fhir.us.core`, version `6.1.0`) in place of any
earlier US Core dependency. Concretely, in the SCM IG sources this
means:

- Bumping the US Core entry under `dependencies` in the IG publisher
  configuration (e.g., `sushi-config.yaml` or equivalent
  `ImplementationGuide-*.json`) to `hl7.fhir.us.core: 6.1.0`.
- Re-running the IG publisher and resolving any profile/extension
  reference breakage introduced by the move from the prior US Core
  release to 6.1.0 (notably any Patient/Practitioner/Organization
  profile renames or must-support changes that affect SCM consent
  scoping).
- Updating any narrative pages in the SCM IG that name a specific US
  Core version.

This matches what the ticket's resolution description says was
ultimately done.

#### Justification

US Core 6.1.0 is the current US Realm baseline aligned with regulatory
expectations (USCDI v3-era), so anchoring SCM consent scopes,
identifiers, and demographic profiles to that baseline keeps SCM
implementable against the same artifacts US implementers are already
using. The community member raising the request is signaling concrete
implementer pain with an older US Core version, and the workgroup's
own resolution note confirms the change was feasible. Choosing 6.1.0
rather than US Core 7 is the conservative end of the requested range
and minimizes churn for early SCM adopters who have not yet moved to a
US Core 7 stack.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt **US Core 7** (the latest published US Core) as the SCM IG's US
Core dependency instead of 6.1.0, and explicitly document SCM's US
Core support policy in a new "Dependencies" section of the IG:

- Set `hl7.fhir.us.core: 7.0.0` in the SCM IG dependency
  configuration.
- Add a narrative page (or extend the existing dependency narrative)
  stating that SCM tracks the **most recent published US Core major
  release**, and that consent scoping profiles will be reviewed and
  re-pinned at each US Core major.
- Where SCM constrains or references US Core profiles
  (Patient, Practitioner, Organization, Encounter, Provenance, etc.),
  update the canonical URLs and any must-support overlays to the US
  Core 7 versions, and call out any newly-required elements that
  affect consent semantics.

#### Justification

The ticket specifically lists US Core 7 as an acceptable target, and
its stated motivation is "future-proofing" against forthcoming
regulation. Pinning to 6.1.0 satisfies the letter of the request but
re-creates the same lag the reporter is complaining about as soon as
US Core 7 becomes the de facto US baseline. Coupling a US Core 7
upgrade with an explicit, written tracking policy is a one-time cost
that pays down the recurring maintenance debt of "which US Core does
SCM target?" tickets. The trade-off vs. Disposition A is a larger
immediate diff and a brief window where SCM implementers running
against pre-7 stacks have to adapt; that cost is bounded because US
Core 7 is the trajectory regulated implementers are on anyway.

---

### Disposition C: Decline

#### Proposal

Decline to bump the US Core dependency at this time. Keep the SCM IG
on its current US Core dependency, and respond to the ticket noting
that the SCM workgroup will re-evaluate the US Core baseline at the
next substantive SCM release rather than as an out-of-cycle editorial
change. Categorize the rationale as "deferred to next release cycle /
no use cases blocked".

#### Justification

The ticket does not identify a specific consent-scoping element,
profile, or interoperability scenario that is broken by the current US
Core dependency; it is forward-looking. SCM is a relatively young IG,
and changing the US Core dependency mid-cycle has knock-on effects on
every SCM profile that derives from or references US Core (must-support
sets, terminology bindings, narrative pages). Without a concrete
implementer-blocking scenario, deferring the bump to the next planned
SCM release keeps publisher churn predictable and lets the workgroup
batch the US Core upgrade with related profile work.

---

### Recommendation

**Recommended disposition:** A

The ticket's resolution description already records that SCM was moved
to US Core 6.1.0, so Disposition A is consistent with what the
workgroup actually did and with the reporter's lower-bound ask. US
Core 6.1.0 is the broadly-deployed US Realm baseline, which keeps SCM
implementable today without forcing SCM adopters onto a US Core 7
stack that not all of them have yet. Disposition B (jump straight to
US Core 7) is defensible and arguably more durable, but it is a larger
content change than the ticket warrants on its own and is better
handled as part of a planned SCM release that can absorb the profile
churn. Disposition C is hard to justify given that the workgroup has
in fact already accepted the change. The workgroup should confirm the
disposition recorded on the ticket ("Not Persuasive") matches the
intended administrative outcome — if the IG was actually updated to US
Core 6.1.0, "Persuasive" with a pointer to the SCM IG dependency
declaration would more accurately reflect the result.
