## Summary

Hans Buitendijk reports that in the Scalable Consent Management (FHIR) IG, the
"Each Consent Must Support" list is internally inconsistent: items 2–5 read as
clear, parallel list items, but item 1 reads more like an introductory or
heading statement, leaving it ambiguous whether items 2–5 are intended to be
sub-bullets indented under item 1, or peer bullets at the same level. The
ticket also flags a terminology problem with "must have" appearing alongside
"must support": the FHIR community has a well-established meaning for
*Must Support* (the element may be absent, but a system must be able to
populate and consume it when present), and "must have" is not a defined
conformance term. The reporter asks whether "must have" is meant to imply a
minimum cardinality of 1 (i.e., a hard required element) and asks the editors
to disambiguate.

The ticket is recorded as **Triaged / Resolution: Persuasive (Resolved
2026-01-26)**. The resolution note already on the ticket states: *"We are
distinguishing between those elements that do have minimum cardinality of 1
versus those that do not but still must be supported. We will make this
clearer in the introductory statements."* The dispositions below are framed
to support the workgroup in confirming or refining that direction.

There are no Jira links, no Zulip cross-references, and no GitHub
cross-references attached to this ticket in the index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rewrite the "Each Consent Must Support" section in the Scalable Consent
Management (FHIR) IG so that:

1. Item 1 is reformatted as an explicit introductory paragraph (not a numbered
   list item), making clear that items 2–5 are peer bullets describing the
   elements every Consent instance must support.
2. The terminology "must have" is removed in favor of FHIR's defined
   conformance terms. Each enumerated element is annotated with **either**:
   - **Required (min cardinality ≥ 1)** — the element MUST be present and
     populated; the profile sets `min = 1`; or
   - **Must Support (min cardinality 0)** — the element MAY be absent, but
     producers MUST be able to populate it and consumers MUST be able to
     read/handle it when present, per the FHIR Must Support definition.
3. Add a one-sentence inline note linking the reader to the FHIR definition of
   Must Support so the distinction is unambiguous.

This matches the resolution already recorded on the ticket
("distinguishing between those elements that do have minimum cardinality of 1
versus those that do not but still must be supported").

#### Justification

This addresses both defects the reporter raised — list-structure ambiguity and
undefined "must have" terminology — without changing any conformance semantics
of the underlying Consent profile. It uses only terms already defined in the
FHIR specification (cardinality and Must Support), which is the standards-
consistent way to express the distinction. The change is purely narrative /
editorial and is the lowest-risk path to clarity. It is also exactly what the
ticket's existing resolution describes, so accepting as requested keeps the
resolution and the published narrative aligned.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to the narrative cleanup in Disposition A, restructure the section
as a **table** rather than a list. Columns:

| Element | Cardinality (in profile) | Conformance | Notes |
|---|---|---|---|
| ... | `1..1` / `0..1` / `1..*` | Required / Must Support | ... |

Each row covers one of the elements currently in items 2–5 (and any element
currently implied by item 1). The introductory paragraph above the table
states the intent ("Every conformant Consent instance must support the
following elements; some are also required to be present"). Optionally, link
each Element name to its anchor in the StructureDefinition / profile snapshot
so readers can jump straight to the element-level constraints.

#### Justification

Tables eliminate the list-vs-sub-list ambiguity entirely (there is no
"indent of 1" question), and they make the Required vs. Must Support
distinction visually obvious instead of relying on prose. This is consistent
with how many published HL7 IGs (US Core, IPS, Da Vinci) present per-element
conformance summaries, so it will feel familiar to implementers and reviewers
alike. The trade-off is slightly more authoring effort and a small risk of the
table drifting from the StructureDefinition over time; that risk can be
mitigated by generating the table from the profile if the IG already uses a
templating mechanism. For an IG in the Scalable Consent Management space,
where implementer clarity is the whole point, the upside outweighs the cost.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave the "Each Consent Must Support" list as it
stands, on the grounds that the existing list, read in context with the
underlying Consent profile's StructureDefinition, conveys the intended
constraints, and that the StructureDefinition (not the narrative) is the
normative source of cardinality and Must Support flags.

#### Justification

This is defensible only if the workgroup believes the narrative is purely
illustrative and that conformance is governed entirely by the
StructureDefinition. In that view, edits to the prose are unnecessary because
implementers should be reading the profile, not the IG narrative, for binding
conformance. However, this position is hard to sustain given that (a) the
ticket has already been resolved as **Persuasive**, meaning the workgroup has
effectively agreed an edit is warranted, and (b) "must have" is not a defined
FHIR conformance term, so leaving it in the published narrative is a real
defect, not a stylistic preference.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the smallest, most direct change that fully addresses both
problems the reporter raised — the list-structure ambiguity and the use of
the undefined term "must have" — and it is exactly what the ticket's existing
resolution ("Persuasive") commits to. It introduces no new conformance
semantics and no risk of changing implementer behavior; it just replaces a
non-standard term with the standard FHIR pair (cardinality + Must Support)
and clarifies the list structure. Disposition B (table form) is a strong
secondary option that the workgroup should consider as a follow-up editorial
improvement, especially if other sections of the IG would benefit from the
same treatment, but it is more work than this single ticket requires and can
be tracked separately. Disposition C is not recommended: the workgroup has
already judged the request persuasive, and shipping the published narrative
with an undefined conformance term would invite repeat tickets from
implementers.
