## Summary

The ticket asserts that a Code System defined within the US PACIO
Re-Assessment Timepoints (FHIR) IG has its canonical URL anchored under
`hl7.org/fhir` when it should instead be anchored under
`terminology.hl7.org` (THO). The submitter (per imported comment 4 by
Ron G. Parker) further requests that a companion UTG ticket be opened to
register the Code System in THO so that the canonical URL change is
backed by a real THO publication. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced for this ticket, so the
context is limited to the ticket text and its own comment history.

## Proposed Dispositions

> All three dispositions concern the canonical URL of the
> single PACIO Re-Assessment Timepoints CodeSystem flagged by the
> submitter. Because no GitHub cross-references were surfaced, no repo
> briefing was loaded; the IG repo (the PACIO Re-Assessment Timepoints
> publication source) would still be the implementing site for any
> change accepted, and the workgroup should confirm the exact CodeSystem
> id when applying the chosen disposition.

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the affected CodeSystem's canonical URL from
`http://hl7.org/fhir/...` to `http://terminology.hl7.org/CodeSystem/...`
within the US PACIO Re-Assessment Timepoints IG. Concretely:

1. Open a companion UTG ticket against `HL7/UTG` requesting registration
   of the CodeSystem under THO (proposing a stable identifier and
   maintenance owner — likely the PACIO project under CBCC stewardship).
2. Once the THO id is assigned, update the CodeSystem resource in the
   IG: set `CodeSystem.url` to the new
   `http://terminology.hl7.org/CodeSystem/<id>` value, leave
   `CodeSystem.identifier` and `CodeSystem.version` aligned with the THO
   record, and add a `replaces` / `extension[replaced-by]` pointer (or
   IG-level note) so existing implementers can follow the move from the
   old URL to the new.
3. Update every `system` reference inside the IG (ValueSets, profile
   bindings, examples, narratives) from the old `hl7.org/fhir` URL to
   the new THO URL.
4. Publish a migration note in the IG's change-history / errata page
   describing the URL move and recommending implementers update any
   instances or terminology server content that hard-codes the old URL.

#### Justification

This is the published HL7 governance pattern: code systems that are not
part of the FHIR core specification are expected to live in THO so they
have a single, canonical owner and a versioned publication independent
of any one IG. The reporter's request matches that pattern, and the
deferred 2022-02-22 resolution explicitly listed "hosted on THO" as an
acceptable future outcome — the September 2022 re-open signals that the
workgroup believes the IG has now matured enough to commit to that path.
Accepting as-requested also clears the original block-vote concern
without reopening the design question.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the CodeSystem hosted within the IG (retain an
`hl7.org/fhir/<ig>/CodeSystem/...` style canonical that is
IG-namespaced), but bring it into governance compliance by:

- Aligning the canonical URL with the IG's own canonical base (so it is
  unambiguously owned by the PACIO Re-Assessment Timepoints IG rather
  than the FHIR core namespace), and
- Adding a `CodeSystem.identifier` that is reserved in THO as a
  "registered, IG-published" entry — i.e., recorded in THO's catalogue
  of known external code systems without moving authorship out of the
  IG.

This preserves the IG's editorial control, avoids cross-repo
coordination overhead, and still satisfies the underlying "don't squat
on `hl7.org/fhir`" concern.

#### Justification

The deferred 2022-02-22 resolution explicitly listed *three* viable
futures: absorbed elsewhere, IG-specific, or hosted on THO. "IG-specific
with THO registration" is the lightest-weight option that still removes
the URL from the FHIR core namespace. It avoids the risk that THO
registration stalls (UTG queue depth and CodeSystem ownership debates
have historically delayed similar moves) while still giving implementers
a stable, correctly-namespaced canonical. It is also the better choice
if the codes are tightly coupled to PACIO Re-Assessment Timepoints
semantics and unlikely to be reused by other IGs.

---

### Disposition C: Decline

#### Proposal

Decline the URL change and close the ticket as **Not Persuasive** on
the grounds that the cost of moving a published canonical URL after
STU1 (broken references in deployed instances, terminology-server
content keyed off the old URL, downstream IGs that may already bind to
the existing system) outweighs the benefit, and that the IG can address
the underlying concern through documentation (clarifying ownership and
maintenance in the CodeSystem narrative) without re-anchoring the URL.

#### Justification

The earlier 9-0-0 vote had real substance: at the time the workgroup was
unwilling to commit to a single hosting outcome. STU1 has now been
published with the existing URL, and any implementer who has stored the
CodeSystem URL in instances or in a terminology server will see breakage
if the canonical changes. If no concrete implementer has reported harm
from the current URL, the cleaner action is to leave the URL stable and
record the policy rationale in the IG's terminology section, deferring a
THO move to the next major IG version where breaking changes are
expected anyway.

---

### Recommendation

**Recommended disposition:** A

The original ticket has stood for over half a year, the workgroup
explicitly re-opened it after STU1 to act on it, and the earlier
deferral identified THO hosting as an acceptable destination once the
IG matured. Accepting as-requested is the cleanest path: it puts the
CodeSystem in the namespace HL7 governance expects (THO), assigns a
single owner of record via UTG, and removes the standing block-vote
concern. Disposition B is defensible if the workgroup wants to minimise
cross-repo coordination, but it leaves the IG with a non-standard
hosting story and will likely re-surface the same complaint later.
Disposition C undervalues the re-open: had the workgroup wanted to
preserve the post-STU1 URL it would not have reverted the prior
resolution. The submitter's request to also file a UTG companion ticket
should be honoured as part of accepting; that is the lever that makes
the move safe rather than performative.
