## Summary

John Moehrke is asking the Scalable Consent Management (FHIR) IG to improve
the readability of the section that mandates a specific LOINC code on
`DocumentReference`. The author agrees that pinning a LOINC code is the
right design, but observes that the spec presents the bare code without
any human-readable context — implementers have to look the code up in
LOINC to understand what it represents. The request is to add narrative
that names and explains the chosen LOINC code (at minimum, surfacing the
LOINC display name) so that a reader of the IG can understand what the
mandated code means without leaving the page.

The ticket has no cross-referenced Jira tickets, Zulip threads, or
GitHub items in the FhirAugury index. It is already marked **Resolved /
Persuasive** with a resolution note committing to "add a description of
the loinc code and why it was chosen." The disposition discussion below
exists to confirm that resolution and to articulate what the actual
editorial change should look like in the IG source.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the Scalable Consent Management (FHIR) IG, locate the narrative that
mandates the specific LOINC code on `DocumentReference` (typically
either the `DocumentReference.type` binding/example narrative or the
profile-level "must use this code" callout). Add an inline editorial
block immediately adjacent to the code that contains, at minimum:

1. The LOINC code value itself (already present).
2. The LOINC **display name** for that code.
3. A one- or two-sentence explanation of *why* this LOINC code was
   chosen for Scalable Consent Management (i.e., what kind of document
   it represents and why it is the appropriate type for consent
   artifacts in this IG).

No structural change to the profile, value set, or binding is required —
this is a narrative-only addition that satisfies the ticket exactly as
filed and matches the resolution note already on the ticket.

#### Justification

The ticket is explicitly tagged `Infrastructure/Editorial` and the
reporter agrees the mandated code itself is correct ("the mandated
loinc code is good"). The requested change is purely about
discoverability: a reader should not have to leave the IG to understand
what the mandated code means. Adding the display name plus a short
rationale is consistent with how other HL7 IGs document mandated /
preferred codes (e.g., the IPS and US Core narrative blocks for fixed
LOINC codes), is fully backward-compatible, and matches the existing
**Persuasive** resolution. There is no implementer burden because no
exchanged instance changes.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) hand-authored narrative, expose the
LOINC code through a small, reusable mechanism in the IG so that the
display name and rationale stay in sync if LOINC ever updates the code's
display:

- Define (or reuse) a `ValueSet` / `CodeSystem`-supplement-style
  fragment in the IG that pins the LOINC code with its display, and
  render that fragment in the page via the IG publisher's standard
  code-rendering macro (so the display is sourced from the supplement,
  not duplicated as free text).
- Keep the human-facing rationale ("why this code") as a separate short
  paragraph next to the rendered code, but draw the *display name* from
  the rendered fragment.

This still resolves the ticket — readers see the display name and the
rationale on the page — but it removes the long-term editorial risk of
the narrative drifting from the actual LOINC display.

#### Justification

The base risk with Disposition A is that hand-typed LOINC display names
become stale: LOINC display strings have historically been adjusted,
and a hard-coded display in narrative will silently fall out of sync.
Routing the display through a publisher-rendered fragment keeps the
spec self-consistent and matches how IGs that depend on a single
mandated code (e.g., several Da Vinci IGs) typically present such
codes. The trade-off is more authoring overhead up front for a
single-code change, which the workgroup may consider disproportionate
for one LOINC code.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope / unnecessary. Treat the LOINC code as
self-documenting (LOINC is a normative external code system; readers
who need the display can resolve the code against `loinc.org`), and
make no narrative change.

#### Justification

This is the weakest disposition for this ticket. The only defensible
rationale for decline would be that the IG already links each LOINC
code to its LOINC.org page and the workgroup judges that sufficient.
However, the reporter — a long-standing contributor in this space — has
explicitly stated that the current presentation is inadequate, the
ticket is labeled `Infrastructure/Editorial` (i.e., low-risk, low-cost
to fix), and the workgroup has already accepted the change as
**Persuasive**. Declining now would contradict the existing resolution
and leave a known readability gap unaddressed.

---

### Recommendation

**Recommended disposition:** A

The ticket is small, editorial, already accepted as **Persuasive**, and
the requested change is exactly what readers of the IG need: a
human-readable explanation of a mandated LOINC code. Disposition A
delivers that with the least overhead and aligns with the resolution
note already on the ticket. Disposition B's drift-resistance argument
is real but disproportionate for a single mandated code — the workgroup
can adopt the publisher-rendered pattern later, IG-wide, if they decide
LOINC-display drift is a recurring issue. Disposition C should be
rejected because it conflicts with the existing **Persuasive**
resolution and ignores a reasonable editorial request from a senior
contributor.
