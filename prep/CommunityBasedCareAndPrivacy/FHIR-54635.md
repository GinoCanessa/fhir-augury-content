## Summary

The ticket questions whether `Consent.verification` is common enough across
implementations to justify its position as a core element of the `Consent`
resource, and asks the workgroup to consider whether it would be better
modeled as an extension. The reporter (John Moehrke) raises the concern at
the structural-design level only — no specific failure case or implementer
complaint is cited in the ticket itself, but the ticket follows directly on
the heels of a community discussion the reporter participated in (see the
[#implementers > Use Consent.verification for grantee accepting](https://chat.fhir.org/#narrow/stream/implementers/topic/Use%20Consent.verification%20for%20grantee%20accepting)
thread, started 2026-02-12 by Julianne Cohen) where the meaning and scope of
`Consent.verification` were debated. A directly-related sibling ticket,
[FHIR-55752](https://jira.hl7.org/browse/FHIR-55752) "Definition of
Consent.verification is too specific" (filed 2026-02-13 by Julianne Cohen
out of the same Zulip thread), proposes the *opposite* direction —
broadening the element's definition so it is clearly applicable beyond
treatment-instruction verification rather than removing/demoting it. Older
historical tickets touching the same element are
[FHIR-14181](https://jira.hl7.org/browse/FHIR-14181) (RIM mappings) and
[FHIR-14453](https://jira.hl7.org/browse/FHIR-14453) (Workflow report on
Consent), neither of which materially constrain the disposition here.

## Related Zulip Discussions

### [implementers > Use Consent.verification for grantee accepting](https://chat.fhir.org/#narrow/stream/implementers/topic/Use%20Consent.verification%20for%20grantee%20accepting)

The most directly relevant discussion (2026-02-12, 8 messages). Julianne
Cohen asks whether `Consent.verification` can be used to record that a
grantee has accepted a consent allowing another party to perform certain
actions, noting that the description seems to constrain the usage to
treatment instructions. John Moehrke replies that
`Consent.verification` is for recording **witnesses to the consent
ceremony** and that this meaning is not actually limited to treatment
instructions — and asks her to file a ticket to make that clearer (which
became FHIR-55752). The conversation then pivots to whether grantee/actor
information needs to be in `provisions` or can be left implicit when the
referenced policy already names the grantee. Lloyd McKenzie pushes back,
arguing it is "always best to be explicit" so consumers without access to
the underlying policy can still understand what is being authorized and so
that participants are searchable; John Moehrke counters that consumers
who don't understand the policy shouldn't act on the consent at all.
There is no consensus that `Consent.verification` should be removed or
demoted; the operative outcome of the thread is the request to clarify
the definition, not to extension-ify the element.

Participants: Julianne Cohen, John Moehrke, Lloyd McKenzie.

### [implementers > Consent.verification in R5](https://chat.fhir.org/#narrow/stream/implementers/topic/Consent.verification%20in%20R5)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove `Consent.verification` (BackboneElement, including
`Consent.verification.verified`, `Consent.verification.verificationType`,
`Consent.verification.verifiedBy`, `Consent.verification.verifiedWith`,
and `Consent.verification.verificationDate`) from the core `Consent`
resource definition for the next normative/ballot cycle, and replace it
with a standard `Extension` (e.g., `consent-verification` published from
HL7 fhir-extensions) that carries the same complex structure. Mark the
removed element as deprecated/retired with a migration note pointing
implementers to the new extension.

#### Justification

The reporter's stated rationale is that the element is not "common
enough" to deserve a core slot — implying that very few implementations
actually populate it. Moving rarely-used data into extensions is an
established FHIR design pattern, lowers the cognitive load of the base
resource, and aligns with the long-running effort to keep core resources
focused on the 80 % use case. If the workgroup has empirical evidence
that `verification` is unused (e.g., from registry data or implementer
surveys), this disposition formalizes that.

---

### Disposition B: Alternative Approach — Keep the element, fix the definition

#### Proposal

Leave `Consent.verification` in the core `Consent` resource and instead
broaden / clarify its definition so it is unambiguously applicable to any
Consent use case (not just treatment instructions). Concretely:

- Rewrite the `Consent.verification` short and definition to describe the
  element as a general "record of who, when and how the consent was
  verified or witnessed", and explicitly state that this is not limited
  to treatment-instruction consents.
- Tighten the `Consent.verification.verifiedBy` definition to consistently
  refer to the *grantor's* decision (resolving the wording issue noted in
  the older R5 thread / FHIR-34302).
- Add an example or two in the resource's narrative covering a
  non-treatment scenario (e.g., grantee accepting a permission to act).

This effectively merges the work of FHIR-54635 with the directly-related
[FHIR-55752](https://jira.hl7.org/browse/FHIR-55752) by treating both as
symptoms of the same underlying problem (a too-narrow definition), and
closes FHIR-54635 with a "addressed via FHIR-55752" disposition.

#### Justification

The recent Zulip thread is unambiguous: implementers (Julianne Cohen) are
*trying to use* `Consent.verification`, and the workgroup chair
(John Moehrke) confirms it is *intended* to apply beyond treatment
instructions. That is direct evidence that the element is wanted in
core — the friction is descriptive, not structural. Removing the element
to an extension (Disposition A) would impose migration cost on every
existing Consent implementer, break R4/R5 backwards compatibility for an
element that has already shipped through multiple releases, and conflict
with the stronger pro-explicitness argument Lloyd McKenzie raised in the
same thread. Definition changes are technical corrections; element
removal is a breaking change. Symmetry with how other resources expose
witness/verification structure (e.g., `Composition.attester`,
`Contract.signer`) also argues for keeping the element where it is.

---

### Disposition C: Decline (no change)

#### Proposal

Reject the request and close FHIR-54635 with the rationale that
`Consent.verification` is an established normative/structural element
that has shipped through R4 and R5, that there is no implementer evidence
that it is actively unused, and that any concrete clarity problem is
already being addressed under
[FHIR-55752](https://jira.hl7.org/browse/FHIR-55752). The ticket's
underlying observation ("would be better as an extension") is a design
opinion without a concrete use-case driver attached.

#### Justification

The ticket as filed contains a single sentence and a question mark, with
no impact analysis, no implementer survey, and no concrete failure mode.
The companion Zulip thread shows live implementer interest in *using*
the element, not in retiring it, and produced a different ticket
(FHIR-55752) that the workgroup can act on instead. Removing a core
element is a substantive breaking change and the bar for that should be
correspondingly higher than "unclear that it is common enough".

---

### Recommendation

**Recommended disposition:** **B** (Keep the element, fix the definition).

Disposition B is the disposition the available evidence actually
supports. The reporter's own follow-up Zulip activity, *together with*
the sibling ticket FHIR-55752 filed two days after FHIR-54635, points
firmly at a definitional/clarity problem rather than a "this element
shouldn't be here" problem. Disposition A (extension-ify the element) is
defensible only if the workgroup has evidence of non-use that is not
present in the ticket or in the discussion threads, and it carries real
breaking-change cost for existing Consent implementers; the workgroup
should not take that step on the strength of a one-line speculative
ticket. Disposition C (decline outright) leaves the legitimate clarity
issue on the table; even if FHIR-55752 is the formal vehicle for the
fix, FHIR-54635 should be closed with an explicit "addressed via
FHIR-55752 — definition broadened, element retained" note rather than a
silent decline, so the next reviewer sees the full reasoning trail.
