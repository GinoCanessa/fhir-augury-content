## Summary

The reporter (Alex Kontur) flags that the FAST Scalable Consent Management
implementation guide describes its two system actors inconsistently across
two pages. The **Conformance Expectations** page defines *Client systems*
as "typically EHRs … responsible for gathering consent records" and
*Server systems* as "consent administration systems," and links each to
its respective CapabilityStatement
([ConsentClientCapabilities](https://hl7.org/fhir/us/consent-management/2026Jan/CapabilityStatement-ConsentClientCapabilities.html)
and
[ConsentAdministrativeServerCapabilities](https://hl7.org/fhir/us/consent-management/2026Jan/CapabilityStatement-ConsentAdministrativeServerCapabilities.html)).
The **Technical Specification** page uses overlapping but non-identical
wording — "Client systems are systems that are responsible for gathering
and filing consents" and "Server systems are consent administration
services that store consents and handle requests for consents" — and does
not link the CapabilityStatements. The request is to harmonise the two
definitions so readers see one canonical actor model. The ticket has
already been resolved as **Persuasive**, with the workgroup proposing to
enumerate the canonical actor/system list once and then sweep the IG to
use the same names consistently. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were returned by the augury index for this
ticket.

>

>   consents and handle requests for consents.

## Proposed Dispositions

> Note: The augury index surfaced no GitHub repos for this ticket, so
> the dispositions below name the affected IG (`HL7/fhir-us-consent-management`,
> the canonical home of the FAST Scalable Consent Management IG) by

### Disposition A: Accept As Requested

#### Proposal

Adopt a single canonical "Systems / Actors" block in the IG narrative
and reuse it verbatim on both the Conformance Expectations and Technical
Specification pages. Concretely:

1. Add a new authoring page (e.g., `input/pagecontent/actors.md`) in
   the FAST Scalable Consent Management IG that defines exactly two
   actors:
   - **Client system** — "A system responsible for gathering and filing
     consent records. Client systems are typically EHRs but may be any
     system that captures consent." Linked to
     `CapabilityStatement-ConsentClientCapabilities`.
   - **Server system** — "A consent administration system that stores
     consent records and handles requests for them." Linked to
     `CapabilityStatement-ConsentAdministrativeServerCapabilities`.
2. Replace the diverging blocks on the Conformance Expectations page and
   the Technical Specification page with a `{% include actors.md %}`
   (or equivalent IG-publisher include) so the two pages stay in sync
   automatically.
3. Add a CapabilityStatement link from both pages (the Technical
   Specification page currently omits the links).
4. Sweep the rest of the IG narrative for stray uses of "client",
   "server", "consent administrator", "EHR", etc., and rewrite them to
   match the canonical names.

#### Justification

This is the action the workgroup has already accepted on the ticket
("list the set of actors and systems … then scour the specification to
make sure that we consistently use these names"). It follows standard
FHIR IG authoring practice — define each actor once, reference it
everywhere — and removes the implementer ambiguity that the reporter
identified. Centralising the definition via a shared include also
prevents the same drift from recurring in future ballots, which matches
how other HL7 IGs (e.g., several Da Vinci and FAST IGs) keep narrative
actor blocks aligned with their CapabilityStatements.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) a narrative-only fix, formalise the two
actors as **`ActorDefinition`** resources in the IG (FHIR R5 actor
mechanism, supported by IG Publisher), then derive both narrative pages
mechanically from those resources:

- `ActorDefinition/consent-client` (type = `system`) referencing
  `CapabilityStatement-ConsentClientCapabilities`.
- `ActorDefinition/consent-server` (type = `system`) referencing
  `CapabilityStatement-ConsentAdministrativeServerCapabilities`.

The Conformance Expectations and Technical Specification pages would
then either embed the rendered ActorDefinition narrative (via the
publisher's `{{ActorDefinition/…}}` template) or link out to the two
ActorDefinition pages, eliminating any free-text duplication entirely.

#### Justification

The underlying problem is duplication of definitional content in
narrative markdown — the same root cause that `ActorDefinition` was
introduced to solve. Going this route makes the actor model machine-
readable, lets `obligation`/`requirement` resources point at the actors
directly, and aligns the IG with the direction other FAST IGs are
moving. Trade-off vs. Disposition A: it is more invasive and requires
that the IG already (or imminently) targets a FHIR version that
publishes ActorDefinition cleanly; if the IG is still on a version where
ActorDefinition rendering is rough, the include-based approach in A is
faster and less risky for the current ballot cycle.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by existing CapabilityStatements**:
treat the two `CapabilityStatement` resources
(`ConsentClientCapabilities`, `ConsentAdministrativeServerCapabilities`)
as the normative actor definitions, and tighten only the Conformance
Expectations / Technical Specification narratives by adding a sentence
on each page that points at those CapabilityStatements as the source of
truth. No change to the wording itself.

#### Justification

Defensible only if the workgroup decides the narrative discrepancy is
cosmetic and that the CapabilityStatements already give implementers an
unambiguous, normative description of each system's responsibilities.
The two narrative blocks are not contradictory — both describe a Client
that gathers consents and a Server that administers them — so an
implementer following either page would build to substantively the same
shape. However: the ticket's resolution is already recorded as
**Persuasive**, and the reporter explicitly called this a mismatch
worth fixing, so this disposition is the weakest of the three and is
included mainly to make the trade-off explicit.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already resolved the ticket as **Persuasive** with a
plan that maps directly onto Disposition A — enumerate the actor/system
set once and then sweep the IG for consistent usage. Disposition A is
the lowest-risk way to deliver that resolution within the current
ballot, especially because the fix is almost entirely in
`input/pagecontent/`-level narrative and a shared include, with no
profile or terminology churn. Disposition B (formal `ActorDefinition`
resources) is a strictly better long-term answer and should be tracked
as a follow-up improvement, but folding it into this ticket would expand
the scope beyond what was reported. Disposition C does not honour the
already-recorded Persuasive resolution and is included only for
contrast.
