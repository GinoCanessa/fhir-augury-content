## Summary

The reporter raises a clarity concern about the redesigned `Consent`
resource. In R4, the `permit` / `deny` decision was attached to each
`provision`, where rich contextual qualifiers (time period, data, purpose,
actor, action) made the meaning of "permit what, by whom, for what
purpose" explicit. In the current FHIR build, `Consent.decision` sits at
the root of the resource and `provision` carries exceptions. The reporter
finds the root-level decision ambiguous: when a `Consent` says `permit`,
what exactly is being permitted, and against which context — the
`policyBasis`, `policyText`, `regulatoryBasis`, or some implicit "all
data / all purposes / all actors" baseline? The ticket asks the work
group either to explain the rationale for the model change since R4 and
to clarify in the `Consent` resource description what the root decision
means and how it composes with `provision`-level exceptions, or to
reconsider the design if the semantics cannot be made clear. No related
Jira tickets, GitHub items, or substantive Zulip threads were surfaced
by the cross-reference index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation / clarification change, not a structural
change, against the `Consent` resource in **HL7/fhir** (FHIR Core).
Specifically:

1. Expand the `Consent.decision` element definition to state explicitly
   that the root decision is the **default disposition** that applies
   to every request that is *not* matched by a more specific
   `provision`, and that the specificity of the decision context
   (purpose, actor, data, period, action) is carried by `provision`
   entries that override the root.
2. Add a paragraph to the `Consent` resource narrative (the resource
   intro / "Scope and Usage" / "Boundaries and Relationships" page)
   that:
   - States the rationale for the R5/R6 model change (a single
     enforceable default + exceptions, instead of an unordered set of
     per-provision permits/denies whose composition was implementation-
     defined).
   - Explains how `decision = permit` combined with `provision` entries
     of `type = deny` (and vice versa) is intended to be evaluated.
   - Cross-links `policyBasis`, `policyText`, and `regulatoryBasis` as
     the *governing policy*, distinct from `decision` which is the
     *default outcome under that policy*.
3. Add at least one worked example to the resource page showing
   `decision = permit` with one or more `deny` provisions (and the
   inverse), so the composition rule is unambiguous.

No element cardinalities, types, or value sets need change.

#### Justification

This directly answers the ticket: the reporter explicitly says "would
like have better understanding of this new model as well as the
rationale behind the change since R4. And suggest some explanation to
be included in the description of the resource." The follow-up comment
from `JIRAUSER26199` confirms that even after the work-group's plain-
English explanation, the semantics remain unclear — which is exactly
the symptom of a documentation gap, not a model defect. Documentation
clarification is a low-risk, non-breaking change that respects existing
implementations and aligns with HL7's normative-text-only correction
guidance.

---

### Disposition B: Alternative Approach

#### Proposal

Accept that the question reveals an ambiguity in the model itself, and
in addition to the documentation work in Disposition A, add an
**invariant** and (optionally) a **modifier-extension** that pin down
the evaluation semantics:

1. Add a FHIRPath invariant on `Consent` requiring that when
   `decision` is present, every `provision.type` value must be the
   *opposite* of `decision` (i.e., if `decision = permit`, all root-
   level `provision` entries must be `deny`, and vice versa). Mixed-
   polarity provisions at the same level are the main source of
   ambiguity in "what does the root permit cover?".
2. Add a short normative "Decision Composition" subsection on the
   `Consent` page describing the evaluation algorithm (most-specific
   matching `provision` wins; otherwise `decision`).
3. Optionally introduce an extension `consent-decision-scope` on
   `Consent.decision` that lets implementers state the implicit scope
   of the root decision (e.g., "all purposes / all actors / all data
   under the cited `policyBasis`") for cases where the policy text
   alone does not make the default scope obvious.

#### Justification

The follow-up Zulip-style exchange in the ticket comments shows two
experienced participants disagreeing about what `permit` at the root
means — that is signal that the model, not just the prose, leaves
something on the table. An invariant restricting mixed-polarity
provisions removes the worst ambiguity (overlapping permits and
denies at the same level) without breaking any existing valid
instances that already follow the "default + exceptions" pattern. The
extension is optional and additive, so it does not affect base-spec
conformance. Trade-off vs. Disposition A: more design effort and a new
conformance constraint to publish, but a stronger guarantee that
implementations interpret root-level `decision` consistently.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as **Persuasive with Mod / Not Persuasive — already
addressed**, on the grounds that the redesign of `Consent` (root
`decision` + `provision` exceptions) has been the subject of multiple
rounds of work-group review, that `policyBasis` / `policyText` /
`regulatoryBasis` are the canonical mechanism for stating the *scope*
the root decision applies to, and that the existing element
definitions already convey this. Close with a pointer to the relevant
governing-policy elements and to the prior R4→R5/R6 change rationale.

#### Justification

Defensible only if the work group judges the existing text to be
sufficient and the reporter's confusion to be a one-off
misunderstanding. Even then, the second commenter's restated question
("Am I consent everything if I'm using permit in the root?") makes
this a hard sell: when two different readers in the ticket itself
reach the same confusion, "the docs are fine" is not a strong
position.

---

### Recommendation

**Recommended disposition:** A

The reporter explicitly asks for documentation, not a structural
change, and the second commenter's restated question on 2025-06-13 is
strong evidence that the current resource description is not
self-explanatory for the root `decision` element. Disposition A
delivers exactly the clarification asked for, is non-breaking,
low-effort, and can be balloted against the FHIR Core resource
description and `Consent.decision` definition without touching
cardinalities, value sets, or invariants. Disposition B is an
attractive follow-on if the work group wants to harden the semantics —
the polarity invariant in particular is worth considering as a
separate, scoped ticket — but should not block landing the
documentation fix this ballot cycle. Disposition C is hard to defend
given that two reviewers in the ticket thread itself failed to converge
on the meaning.
