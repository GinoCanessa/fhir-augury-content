## Summary

The reporter (John Moehrke) requests that the *Scalable Consent Management
(FHIR)* IG add hyperlinks every place it names another specification, so that
readers can navigate directly to the version of the referenced specification
that this IG actually depends on. Specifically, references to **FHIR Core**
(Consent, AuditEvent, Subscription), **IHE-PCF** and **IHE-BALP**, and
**US Core** should each become hyperlinks to the corresponding revision of
those specifications. The reporter further recommends the IG use the IG
Publisher's `site.data.*` template variables (e.g., `site.data.fhir.version`
and `site.data.fhir.path`) so that the link target is automatically resolved
to the dependency-pinned version rather than being hard-coded. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the data sources for this ticket.

### Disposition A: Accept As Requested

#### Proposal

Apply an editorial pass across the *Scalable Consent Management (FHIR)* IG
narrative pages so that every prose reference to another specification is
rendered as a hyperlink to a version-specific page of that specification,
using the IG Publisher's `site.data.*` template variables (or the
equivalent `{{cross-version-anchor}}` / `[[link]]` markdown supported by
the publisher) so that the resolved URL tracks the IG's declared
dependency. Concretely:

- Replace bare textual references to **FHIR Core** resources (`Consent`,
  `AuditEvent`, `Subscription`) with links of the form
  `[Consent]({{site.data.fhir.path}}consent.html)` (and similarly for
  `auditevent.html`, `subscription.html`) so the link follows
  `site.data.fhir.version`.
- Replace references to **IHE-PCF** and **IHE-BALP** with links to the
  declared dependency revision of each profile (e.g., resolved through
  the IG's `dependsOn` entries / corresponding `site.data.<dep>.path`
  values, falling back to the canonical IHE Profile URL for that
  revision).
- Replace references to **US Core** with links to the version of US Core
  this IG depends on, again driven by the publisher template (e.g.,
  `site.data.uscore.path` if defined, otherwise the IG's `dependsOn`
  canonical pinned to that version).
- Add an introductory sentence in the IG's *Rationale* section (the one
  the reporter quoted, e.g., section 1.2) of the form
  *"This IG is founded on HL7 FHIR Revision {{site.data.fhir.version}}
  found at [{{site.data.fhir.path}}]({{site.data.fhir.path}})."*

#### Justification

This is exactly what the reporter requested and matches the resolution
already recorded on the ticket ("Everywhere that we reference a
specification, we will ensure that there is a link to it and that it is
a version-specific publisher generated link."). The change is
**editorial** (matches the `Infrastructure/Editorial` label), creates no
normative content, and aligns with standard HL7 IG authoring practice:
specification cross-references *should* be navigable and *should* track
the declared dependency rather than rotting against `current` or against
a hard-coded version. Using the IG Publisher's `site.data.*` mechanism
keeps the links correct automatically when the IG re-pins to a new
dependency revision, which is the long-term maintenance argument for
doing it template-driven instead of by raw URL.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same intent — version-pinned, navigable cross-spec links —
but implement it as a **mechanical, repo-wide convention** rather than a
one-time editorial pass:

1. Define (in the IG's authoring documentation, e.g. a
   `CONTRIBUTING.md` / `authoring-conventions.md` entry) a small set of
   canonical link macros that authors are required to use, e.g.
   `{{fhir:Consent}}`, `{{ihe-pcf:...}}`, `{{us-core:...}}`,
   resolved by `pagecontent` includes / `site.data.*` lookups against
   the IG's `dependsOn` block.
2. Land the editorial sweep from Disposition A on top of those macros so
   every existing reference uses the macro, not a hand-rolled URL.
3. Add a publisher / CI check (e.g., a regex lint over `pagecontent/`
   for bare strings like "FHIR Core Consent", "IHE-PCF", "US Core",
   etc., that are not inside a link) so that *future* edits cannot
   re-introduce un-linked references.

#### Justification

This addresses the underlying need — that readers can always reach the
right version of any referenced specification — without depending on
authors remembering the convention each time. It is more work up front
(macro definitions plus a lint) but eliminates the recurrence of this
same ticket on the next ballot cycle and makes dependency-version bumps
trivially correct. The trade-off vs. Disposition A is implementation
burden: A is "fix what's there now"; B is "fix what's there now *and*
prevent regression." For an IG that is already deprecating its work
group label, B may be over-investment relative to the IG's expected
future churn.

---

### Disposition C: Decline

#### Proposal

Decline the request as written, on the basis that bare textual
references to upstream specifications are an accepted IG-narrative
style, version pinning is already documented in the IG's `dependsOn`
block, and adding inline hyperlinks is a stylistic preference rather
than a defect. Close as *Not Persuasive — editorial preference*.

#### Justification

The only defensible reason to decline is scope/cost: the work group is
marked `[deprecated]`, an editorial sweep across every spec reference
plus `site.data.*` plumbing is non-trivial, and the information is
already discoverable via the IG's stated dependencies. However, this
rationale is weak because (a) the request *is* a real readability /
navigability defect for implementers reading the IG out-of-band, (b)
the work has already been accepted on the ticket as "Persuasive", and
(c) editorial fixes of exactly this shape are routine across HL7 IGs.
Disposition C is included for completeness; the case for it is thin.

---

### Recommendation

**Recommended disposition:** A

The ticket is small, editorial, and the resolution language already
recorded on the Jira issue ("Everywhere that we reference a
specification, we will ensure that there is a link to it and that it is
a version-specific publisher generated link.") matches Disposition A
almost word-for-word — the work group has effectively already accepted
this as Persuasive. Disposition B is the right *long-term* answer if
this IG had a long forward roadmap, but with the work group label
flagged `[deprecated]`, investing in lint infrastructure to prevent
regression is unlikely to pay back. Disposition C is hard to defend
against the existing resolution. Recommend the work group confirm
Disposition A and assign the editorial sweep to whoever is currently
shepherding the IG's narrative pages, with explicit instruction to use
`site.data.*` (and the equivalent `dependsOn`-driven variables for
IHE-PCF, IHE-BALP, and US Core) rather than hard-coded URLs.
