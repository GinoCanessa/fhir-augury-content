## Summary

The reporter (John Moehrke) observes that the **"Integration with other
Implementation Guides"** page of the **Scalable Consent Management**
FHIR IG only references **US-Core** and **Subscriptions**, and asks that
the IG also call out **IHE-PCF** (Privacy Consent on FHIR) and
**IHE-BALP** (Basic Audit Log Patterns) as guides that an implementer
should consult. The ticket has been triaged with a resolution of
**Persuasive**, with the resolution description stating "We will add
all guides that need to be consulted when implementing this IG."

No cross-referenced Jira tickets, Zulip conversations, or GitHub items
were surfaced for this ticket, so the request stands on its own as an
editorial / completeness fix to the IG's integration narrative.

## Proposed Dispositions

The ticket is editorial in nature: it asks the IG to acknowledge two
adjacent IHE-on-FHIR guides on a single narrative page. The
dispositions below reflect that scope. Because no GitHub repo
briefing is available, file paths are described conceptually (the
"Integration with other Implementation Guides" page in the Scalable
Consent Management IG sources) rather than as concrete authoring
paths.

### Disposition A: Accept As Requested

#### Proposal

Update the "Integration with other Implementation Guides" page of the
Scalable Consent Management IG to add explicit subsections (or, at
minimum, named bullets) for:

- **IHE-PCF — Privacy Consent on FHIR**: describe how PCF profiles
  Consent and the consent-decision/enforcement pattern relate to or
  complement Scalable Consent Management's consent model, and call out
  any Consent profile alignment / extension expectations.
- **IHE-BALP — Basic Audit Log Patterns**: describe how BALP
  AuditEvent profiles should be used to record consent-relevant events
  (consent capture, consent retrieval, consent-driven access decisions,
  disclosure events) produced by an implementation of this IG.

Each subsection should follow the same shape already used for the
US-Core and Subscriptions entries on that page (short narrative + link
to the canonical IG + statement of how/when an implementer should
consult it). Update the page's lead-in sentence so it no longer implies
the list is exhaustive at US-Core + Subscriptions.

#### Justification

This is precisely what the resolution description ("We will add all
guides that need to be consulted when implementing this IG") commits
to, and it matches the persuasive disposition already recorded on the
ticket. IHE-PCF and IHE-BALP are the recognized IHE-on-FHIR companion
guides for consent and consent-related audit, respectively, and
omitting them on a page titled "Integration with other Implementation
Guides" misleads implementers into thinking US-Core and Subscriptions
are the only relevant cross-IG dependencies. The change is editorial,
non-normative in the sense that it adds no new conformance constraints,
and carries effectively no implementation burden.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent of the ticket but generalize the fix instead of
hard-coding only IHE-PCF and IHE-BALP. Restructure the page so that:

1. A short rationale paragraph explains why an IG of this kind has
   integration touch points (consent capture, terminology, audit,
   notification, query).
2. A table lists *all* commonly-consulted companion guides grouped by
   concern — e.g., **Identity/Demographics** (US-Core, IPS),
   **Consent** (IHE-PCF), **Audit/Accounting of Disclosures**
   (IHE-BALP), **Notification** (Subscriptions /
   Subscriptions-Backport), **Provenance**, etc. — with one-line
   guidance and a link for each.
3. The IG governance text is updated to say which of these are
   *required* to be consulted vs. *recommended*, so future tickets
   that add a guide land in the right tier rather than just appending
   another bullet.

This still satisfies FHIR-54689 (IHE-PCF and IHE-BALP appear in the
list) but produces a more durable structure.

#### Justification

The reporter's observation — that the page reads as if US-Core and
Subscriptions are the entire universe of related IGs — points at a
structural weakness, not just a missing pair of bullets. Other tickets
of the same shape (asking for one more guide to be added) are likely.
Pre-grouping by concern with a *required vs. recommended* signal makes
the page resilient and gives implementers a clearer mental model.
Trade-off vs. Disposition A: it is a larger editorial change, requires
WG agreement on the required/recommended split, and risks the page
expanding into a list the WG cannot keep current.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave the "Integration with other
Implementation Guides" page unchanged, on the basis that IHE
publications (IHE-PCF, IHE-BALP) are governed outside HL7 and that
implementers should be expected to discover relevant IHE profiles via
the IHE site rather than via cross-pointers from an HL7 IG.

#### Justification

This is the defensible "out of scope" framing: the Scalable Consent
Management IG cannot warrant the stability or version-currency of
external (IHE) guides, and adding cross-pointers creates a maintenance
obligation (broken links, version drift) for content the WG does not
own.

---

### Recommendation

**Recommended disposition:** A

The ticket is small, editorial, and the WG has already recorded a
**Persuasive** resolution committing to "add all guides that need to be
consulted when implementing this IG." Disposition A delivers exactly
that with the minimum change footprint and matches the existing
treatment of US-Core and Subscriptions on the same page. Disposition B
is attractive structurally but is a larger scope than the ticket
asks for and would justify a separate ticket once the immediate gap
(IHE-PCF, IHE-BALP) has been closed. Disposition C contradicts the
already-recorded persuasive resolution and leaves implementers without
a pointer to the two most directly relevant companion IGs in the
consent + audit space, so it should not be the path chosen.
