## Summary

The reporter (Joseph M. Lamy) observes that the Scalable Consent Management
IG describes a set of use cases and a set of technical mechanisms (APIs,
operations, etc.), but the relationship between the two is not made
explicit in the published material. The ticket asks the workgroup to
provide an *informative* mapping between each use-case step and the
corresponding technical mechanism(s) that implement it, so that
implementers can readily see which API surface(s) satisfy which step. The
reporter calls out three concrete examples that illustrate the gap:
"Sign and File Consent" / "Revoke Consent" appear to be implementable
either via the defined API or via a system's internal mechanisms;
"Delegate Consent" has no obvious API mapping and may or may not be a
specialization of Sign-and-File (and could warrant normative
requirements); and "Propagate Consent Events" raises an open question
about how some triggers (e.g., "the consenter starting to review the
consent form") would even be detected. The ticket has already been
triaged with a resolution of **Persuasive**, with the workgroup's
documented intent being to add a link to the relevant technical operation
in each use-case diagram and to publish a table mapping use cases to the
operations they require. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were found for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already drafted on the ticket: in the Scalable
Consent Management IG, augment **each use-case page/diagram** with an
inline link from each use-case step to the specific technical mechanism
(FHIR operation, interaction, resource exchange, or notification topic)
that realizes it. In addition, publish a single **"Use Cases ↔ Technical
Mechanisms" mapping table** on a dedicated informative page (or as a
section appended to the existing use-cases page) that lists every use
case as a row, every required/optional operation as columns (or vice
versa), and marks the cells that apply. Treat the table and the
diagram-level links as **informative only**. For the three examples
called out by the reporter, address them as follows in the table notes:

- *Sign and File Consent* / *Revoke Consent* — map to the relevant
  Consent resource create/update interaction(s) defined by the IG, with
  a note that internal (non-FHIR) realizations are also acceptable.
- *Delegate Consent* — explicitly state in the mapping that it is a
  specialization of Sign-and-File Consent in which the signer is acting
  on behalf of the consenter, identify which Consent elements (e.g.,
  performer / grantor) carry the delegation, and add an informative
  note that no additional normative API is being introduced in this
  release.
- *Propagate Consent Events* — map to the IG's notification mechanism
  (e.g., FHIR Subscription / SubscriptionTopic, if defined) for events
  that have an observable state change; for the "consenter starting to
  review the consent form" example, add an informative note that this
  is a UI/workflow event that is **out of scope** for FHIR-level
  notification and would be handled by application-specific
  instrumentation.

#### Justification

The reporter is asking for purely informative documentation, not for
new normative content. The cost is low (no constraint changes, no
profile changes, no breaking changes for implementers) and the benefit
is high: implementers currently have to reverse-engineer which API
surface fulfills each use-case step. This is a well-established
documentation pattern in HL7 IGs, and it directly addresses the three
ambiguities the reporter raised — particularly Delegate Consent (clarified
as a specialization rather than a new API) and Propagate Consent Events
(explicitly bounding what is in scope for FHIR notification). It is also
consistent with the workgroup's already-recorded "Persuasive"
resolution, which describes essentially this approach.

---

### Disposition B: Alternative Approach

#### Proposal

Provide the mapping, but do so by adding **machine-processable metadata**
on each technical artifact (operation, SubscriptionTopic, profile)
rather than (or in addition to) a single static table. Concretely:

- Define an informative `useContext` / `purpose` convention (or a small
  IG-local extension) on each `OperationDefinition`, `SubscriptionTopic`,
  and `StructureDefinition` in the IG that names the use-case step(s)
  it realizes, using a code from a new informative `ConsentUseCase`
  CodeSystem/ValueSet authored in the IG.
- Generate the use-case ↔ mechanism table from those tags at IG-build
  time (rather than maintaining it by hand) and embed the generated
  table on the use-cases page.
- Keep the diagram-level links from Disposition A.

#### Justification

This buys the same reader-facing artifact as Disposition A, but the
mapping cannot drift from the artifacts because it is generated from
metadata that lives on the artifacts themselves. It also creates a
small, reusable vocabulary (`ConsentUseCase`) that downstream profiles
and companion IGs could reuse to declare which consent use cases they
support. The trade-off is non-trivial: it adds authoring burden,
introduces a new informative CodeSystem that must be maintained, and
requires IG-publisher tooling to render the generated table — all for a
benefit that is mostly maintainability rather than reader value. For an
IG at this maturity level, that overhead is likely premature.

---

### Disposition C: Decline

#### Proposal

Mark as **Not Persuasive**. State that the use cases in the Scalable
Consent Management IG are intentionally written at a business/workflow
level and are deliberately decoupled from any single technical
realization, so a one-to-one mapping to FHIR operations would either be
misleading (suggesting only the FHIR-defined APIs are conformant) or
incomplete (since several steps can be realized internally without any
FHIR exchange). Direct implementers to the existing operation,
profile, and subscription-topic pages, which already document each
mechanism in isolation.

#### Justification

This is defensible only if the workgroup wants to preserve maximum
flexibility for implementers and is concerned that publishing a mapping
will be read as normative even when labeled informative. However, the
reporter's questions (especially around Delegate Consent and Propagate
Consent Events) demonstrate that the current decoupling is *already*
causing implementer confusion, so declining largely leaves the gap in
place.

---

### Recommendation

**Recommended disposition:** A

This matches the workgroup's already-recorded "Persuasive" resolution
on the ticket and is the lowest-risk, highest-value response. The
reporter is explicitly asking for informative guidance, and the three
worked examples they raise (Sign-and-File, Delegate Consent, Propagate
Consent Events) are exactly the kind of cases that a use-case ↔
technical-mechanism table plus per-diagram links resolves cleanly,
without adding any normative obligations or breaking changes for
implementers. Disposition B reaches the same outcome via tag-driven
generation, but the tooling and CodeSystem overhead is not justified at
this stage of the IG's maturity. Disposition C leaves the documented
gap unresolved and is inconsistent with the disposition the workgroup
has already recorded; it should be reserved for cases where the
workgroup actively wants to discourage a tight coupling between
use-case steps and FHIR operations, which the reporter's feedback
suggests is not the situation here.
