## Summary

The reporter (Mohammad Jafari) is asking the Scalable Consent Management
(FHIR) IG to redo and elaborate on the architecture/overview diagram that
appears in the IG. Two distinct concerns are raised: (1) the components in
the diagram should be defined and aligned with terminology already
established in adjacent work — specifically the LEAP consent project and
the IHE "Consent on FHIR" profile — so readers can place this IG in the
broader consent ecosystem; and (2) the diagram itself should be made more
readable by relocating textual annotations out of the figure and by
re-rendering it in a scalable format (e.g., SVG) so it remains crisp
across devices.

The ticket has already been triaged by the work group with a resolution of
**Persuasive with Modification** ("We will create a new image that is more
readable.") and the reporter has noted that it is addressed by
[HL7/fhir-consent-management#14](https://github.com/HL7/fhir-consent-management/pull/14).
No other Jira tickets cross-reference FHIR-54709.

### Disposition A: Accept As Requested

#### Proposal

Accept the request and replace the current overview diagram in the
Scalable Consent Management (FHIR) IG with a redrawn version that:

1. Is rendered as an SVG (scalable, crisp on all devices) and stored as an
   editable source (e.g., draw.io / `.drawio.svg` or PlantUML) alongside
   the rendered asset under the IG's image authoring root.
2. Moves long textual annotations out of the figure into the surrounding
   narrative page so the figure itself is uncluttered.
3. Labels each component using terminology aligned with the LEAP consent
   project and IHE "Consent on FHIR", with a short legend / glossary in
   the narrative page that maps each labelled component to its definition
   and source. Where this IG's terminology must diverge from LEAP / IHE,
   call out the difference explicitly.

This is consistent with the work group's existing resolution
("Persuasive with Modification — We will create a new image that is more
readable") and the work apparently delivered in
[HL7/fhir-consent-management#14](https://github.com/HL7/fhir-consent-management/pull/14).

#### Justification

The request is purely editorial/infrastructural: it does not change any
FHIR resource, profile, value set, or conformance rule. It improves
comprehension of the IG and lowers the cost for implementers who are
already familiar with LEAP or IHE Consent on FHIR to map their mental
model onto this IG. SVG is the de facto standard for IG diagrams across
HL7 publications, so this also brings the IG in line with house style.
The reporter has already indicated PR #14 addresses the change, so
acceptance carries low risk and high readability payoff.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the readability/SVG portion of the request, but split the
LEAP/IHE alignment into a dedicated "Relationship to other consent
work" page (or a clearly-marked section on the existing overview page)
rather than embedding cross-IG terminology directly into labels on the
diagram. The diagram itself would use the IG's own component names; the
new page would provide a side-by-side mapping table from this IG's
components to (a) LEAP consent project terms and (b) IHE Consent on
FHIR actors/transactions, with citations.

#### Justification

Embedding external project terminology directly inside the diagram risks
two problems: (1) it visually couples this IG's normative content to the
naming conventions of projects whose terminology may evolve
independently, and (2) it can imply a stronger normative alignment than
actually exists. A separate mapping page keeps the overview diagram
focused on this IG's own architecture while still giving the reporter
what they ultimately want — a clear bridge to LEAP and IHE for readers
coming from those communities. It is also easier to maintain over time:
when LEAP or IHE update their terms, only the mapping page needs to be
updated, not the canonical diagram.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that it is a purely editorial /
documentation refinement request with no impact on conformance, and the
work group has limited editorial cycles. Direct the reporter to open a
PR with the proposed redrawn diagram if they would like the change made.

#### Justification

Strictly speaking, the existing diagram is functional and the IG can be
read without the proposed changes; nothing in the spec is incorrect or
ambiguous in a normative sense. Declining would conserve work group
effort for substantive change requests. However, this rationale is weak
given that (a) the reporter has already done the work in PR #14, and
(b) the work group has already accepted the change with resolution
"Persuasive with Modification". This disposition is included for
completeness and is not the recommended path.

---

### Recommendation

**Recommended disposition:** A (with the option of folding in
Disposition B's mapping-page idea as a follow-up enhancement).

The work group has already triaged this ticket as "Persuasive with
Modification" and the reporter has linked PR
[HL7/fhir-consent-management#14](https://github.com/HL7/fhir-consent-management/pull/14)
as the implementing change. Acceptance is therefore the cheapest path
that closes the ticket consistently with what has already been done. The
benefits — SVG rendering, decluttered labels, alignment with LEAP / IHE
vocabulary that implementers already use — are real and the cost is
purely editorial; there is no normative impact to weigh against. If the
work group has appetite for it, the LEAP/IHE alignment portion of
Disposition B (a dedicated "Relationship to other consent work" mapping
page) is a cleaner long-term home for cross-project terminology than
diagram labels and could be added as a follow-up rather than a blocker.
Disposition C is not recommended because the change is already in flight
and declining would create churn without saving work.
