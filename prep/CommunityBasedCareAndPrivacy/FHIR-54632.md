## Summary

The reporter (John Moehrke) is questioning the binding choice for
`Consent.provision.documentType` (and the parallel element on
`Contract`). He observes that Consent and Contract appear to use a
distinct, special-purpose ValueSet to enumerate "document type" codes,
while the more widely-deployed `Composition.type` and
`DocumentReference.type` elements use a different ValueSet. The ticket
asks for the rationale: why is a separate ValueSet needed for the
Consent/Contract use case rather than reusing the same code system /
ValueSet that already constrains document type elsewhere in FHIR?

The cross-reference index returns only one hit — the FHIR element path
`Consent.provision.documentType` itself — so this ticket stands on its
own; there are no linked or cross-referenced Jira tickets, Zulip
threads, or GitHub items to fold into the picture.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a request to harmonize the binding. Change the
binding on `Consent.provision.documentType` (and the equivalent element
on `Contract`) to point at the same ValueSet that is bound to
`Composition.type` and `DocumentReference.type` (the existing FHIR
"document type codes" ValueSet, which is LOINC-based). Retire the
Consent/Contract-specific ValueSet, or redirect it to the harmonized
one. Add a release note explaining the consolidation, and add a
migration note for implementers who were already using the
Consent-specific codes.

#### Justification

A single, consistent vocabulary for "what kind of document is this" is
easier for implementers, registries, and search infrastructure to
support across resource types. Using the same ValueSet that
Composition / DocumentReference already use lines Consent up with the
broader FHIR document model and aligns with the principle of reusing
existing terminology rather than creating bespoke value sets per
resource. The reporter's question implies he sees no domain-specific
reason for the divergence.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the bindings distinct but make the divergence intentional and
documented. Specifically:

1. Loosen the Consent/Contract binding strength (e.g., from `required`
   to `extensible` or `preferred`) so that implementers who want to
   reuse the document-type codes from Composition/DocumentReference
   are not blocked.
2. Update the element definition / comments on
   `Consent.provision.documentType` (and the Contract analogue) to
   explicitly state why the Consent/Contract list is curated
   differently — for example, that it scopes only the document classes
   meaningful to a privacy authorization (e.g., consent directives,
   advance directives, authorizations) rather than the full LOINC
   document-type catalog.
3. Optionally publish the Consent/Contract-specific list as a curated
   subset that derives from the broader document-type ValueSet, so the
   relationship between the two lists is machine-readable.

#### Justification

If there is a real privacy/legal-document reason for the curated list
(narrower, vetted set of document classes that make sense to authorize
or contract over), eliminating it could over-broaden the binding and
let implementers attach an authorization to document classes the
workgroup never intended. Recasting it as an explicitly-narrower
subset of the canonical document-type ValueSet preserves that intent,
documents the rationale for future readers, and still gives
implementers an escape hatch via a softer binding strength.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Modification** at most a
documentation change, or **Not Persuasive** if the workgroup believes
the existing comments are sufficient. Add a clarifying note on
`Consent.provision.documentType` (and Contract's equivalent)
explaining that the dedicated ValueSet is intentional — it scopes the
authorization to a vetted list of consent/contract-relevant document
classes — and is not intended to be reconciled with the broader
Composition/DocumentReference document-type ValueSet.

#### Justification

The Consent and Contract resources have legal/privacy semantics that
differ from generic clinical documentation. The codes that are
meaningful to authorize or contract over are a narrower set than the
full document-type catalog. If that was a deliberate workgroup
decision when the element was added, the right response to the
question is to record the rationale in the element definition rather
than restructure the binding.

---

### Recommendation

**Recommended disposition:** B

The ticket is fundamentally a "why does this differ?" question, and
both A and C take an absolute position (fully harmonize, or fully keep
the divergence) without confirming the original intent. Disposition B
splits the difference in a way that is defensible regardless of the
historical reason: it documents the rationale for the curated list,
explicitly relates it to the canonical Composition/DocumentReference
document-type ValueSet, and softens the binding strength so
implementers who legitimately need a code outside the curated subset
are not blocked. CBCP can then decide, with the rationale captured in
the element definition, whether a future change to fully merge the
ValueSets (Disposition A) is warranted. If the workgroup confirms in
discussion that there was never a deliberate scoping intent — i.e.,
the Consent/Contract ValueSet is just historical accident — escalate
to Disposition A and harmonize outright.
