## Summary

The submitter (Abderrazek Boufahja) observes that the Scalable Consent
Management IG defines a custom `SearchParameter` named `consent` on
`AuditEvent` (`fast-auditevent-consent`) for the "searching for
disclosures" use case, and points out that searching by the existing
core FHIR `AuditEvent.entity` (i.e. `entity` reference search parameter)
would yield the same result set. The ticket asks whether the IG can
roll back the new IG-specific `consent` SearchParameter and instead
mandate use of the native `entity` search parameter to find AuditEvents
that reference a `Consent` resource. There are no cross-referenced Jira
tickets, Zulip threads, or GitHub items in the FHIR Augury index for
this issue, so the request stands on its own. The ticket has already
been resolved as **Persuasive**, with the resolution stating that the
`consent` SearchParameter will be removed in favor of the core `entity`
parameter; this prep packages dispositions for workgroup confirmation.

## Proposed Dispositions

> Note: the ticket is already resolved as **Persuasive** with a
> resolution that aligns with Disposition A. Dispositions B and C are
> included to give the workgroup a structured set of options to
> reconfirm or reverse that decision.

### Disposition A: Accept As Requested

#### Proposal

Remove the IG-defined `SearchParameter` `fast-auditevent-consent`
(canonical id `consent`, resource = `AuditEvent`, expression
`AuditEvent.entity.what`, target = `Consent`) from the Scalable Consent
Management IG. Replace all guidance, examples, and capability-statement
expectations in the `searching-for-disclosures` section of
`technical.html` so that clients are instructed to search
`AuditEvent?entity=Consent/{id}` (or the equivalent
`entity=Consent/{id}` / `entity:Consent={id}` chained form) using the
core FHIR `entity` reference search parameter. Update the IG's
`CapabilityStatement` to drop the custom `consent` SearchParameter from
the `AuditEvent` resource entry and to require server support for the
core `entity` search parameter on `AuditEvent`. This matches the
resolution already recorded on the ticket.

#### Justification

Both the IG-defined `consent` parameter and the core `entity` search
parameter resolve against `AuditEvent.entity.what` (a `Reference` that
can target `Consent`), so the two queries produce the same result set
when filtering for AuditEvents whose `entity.what` is a `Consent`
resource. Defining a redundant IG-local SearchParameter increases
implementation burden (servers must register and index an extra
parameter), creates divergence risk between the IG-local and core
parameter behaviors over time, and offers no expressive power that the
core `entity` parameter does not already provide. FHIR's design
principles favor reusing core search parameters and only adding new
ones when they capture semantics that core cannot express. The
submitter — an experienced consent/privacy implementer — has already
flagged this as redundant, which strongly suggests the IG-local
parameter would be a friction point in real deployments.

---

### Disposition B: Alternative Approach

#### Proposal

Retain a way for clients to express "find AuditEvents whose entity is a
Consent", but instead of defining a new SearchParameter, address it
purely through normative narrative and example queries in the IG. Keep
the `fast-auditevent-consent` *page* (or repurpose it) as a guidance
section under "Searching for disclosures" that documents the canonical
query as `AuditEvent?entity:Consent={id}` (typed reference search) and
provides worked examples, but delete the `SearchParameter` resource and
remove it from the `CapabilityStatement`. Optionally add a profile-
level invariant or `CapabilityStatement` `searchParam` entry on
`AuditEvent` that lists the core `entity` parameter as **SHALL** be
supported when implementing the disclosure-search use case.

#### Justification

This delivers the same end-state as Disposition A — no IG-local
SearchParameter — while preserving the discoverability of the
disclosure-search recipe for implementers who came to the IG looking
for the `consent` keyword. It avoids dead links and lost context for
anyone who has already cited the existing
`SearchParameter-fast-auditevent-consent.html` page, by repurposing
that URL into guidance rather than removing it outright. The trade-off
versus A is slightly more editorial work (rewriting a page rather than
just deleting an artifact), but the ergonomics for first-time readers
are better and the technical position is identical.

---

### Disposition C: Decline

#### Proposal

Keep the `fast-auditevent-consent` SearchParameter as defined and do
not roll back to the core `entity` parameter. The IG continues to
require servers to advertise and support the IG-local `consent`
parameter on `AuditEvent`.

#### Justification

A defensible (though weaker) reason to keep it: a named, typed
`consent` parameter is more self-documenting for implementers scanning
a `CapabilityStatement` than a generic `entity` reference, and it
gives the IG a single, stable canonical URL to point to from
conformance language. However, this comes at the cost of duplicating a
core capability and forcing every consent-management server to
implement and index a parameter that is functionally equivalent to one
they already support. Because the submitter — and presumably others —
will simply use `entity` anyway, the IG-local parameter risks becoming
unused conformance overhead. This option should be selected only if
the workgroup believes the discoverability benefit outweighs the
redundancy.

---

### Recommendation

**Recommended disposition:** A

The resolution already on the ticket ("We will remove the consent
search parameter and just mandate the use of the entity search
parameter") matches Disposition A, and the technical argument supports
it: the IG-local `consent` parameter and core `entity` parameter
resolve against the same path on `AuditEvent` and target the same
`Consent` reference, so the two queries are equivalent. Removing the
redundant SearchParameter reduces implementer burden, eliminates
divergence risk, and aligns with FHIR's preference for reusing core
search parameters whenever they suffice. Disposition B is a reasonable
fallback if the workgroup wants to preserve the existing IG page URL
as guidance, and can in fact be folded into A as an editorial detail
during the IG cleanup. Disposition C is not recommended — keeping a
duplicate IG-local SearchParameter has no functional upside that
outweighs its conformance cost. The workgroup should confirm A and
proceed to remove the artifact, update `technical.html#searching-for-
disclosures`, and adjust the `CapabilityStatement` accordingly.
