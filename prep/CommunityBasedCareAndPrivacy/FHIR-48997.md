## Summary

The ticket requests that the existing FHIR Extensions Pack extension
`event-statusReason` be made usable on `Consent`. Today the extension's
context list does not include `Consent`, so implementers who want a coded
reason explaining a status change on a `Consent` (typically a revocation,
i.e. moving from `permit` to `deny`, or from `active` to `inactive` /
`rejected`) cannot attach this canonical extension. The reporter (John
Moehrke, CBCP) also asks whether other Event-pattern resources are
similarly missing from the extension's context list and should be added at
the same time.

The request originated in a chat.fhir.org `#implementers` discussion
("Revoking a Consent") in which Kevin Mayfield asked how to record a
business reason for revoking a previously-granted consent. The thread
established that `Consent` lacks a native `statusReason` element today,
and Lloyd McKenzie noted that the canonical pattern for recording the
reason behind a status change is one of: `statusReason` on the resource
itself, a `Task` that requested the change, or `Provenance`. Because
`event-statusReason` is the FHIR-Infrastructure–owned extension that
expresses option (a) generically, John filed this ticket against the
Extensions Pack to broaden its context to include `Consent`. There are no
linked Jira tickets and no related GitHub items; the only cross-reference
is the Zulip thread that motivated the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasively for the next FHIR Extensions Pack release, modify the
`event-statusReason` extension definition (`StructureDefinition/event-statusReason`)
so that its `context` list includes `Consent`. Concretely, add a new
`context` entry of type `element` with `expression = "Consent"` (mirroring
the way the existing Event-pattern resources are listed). While the
extension is being touched, audit the other Event-pattern resources
called out by the FHIR Event pattern (e.g. `ChargeItem`,
`ClinicalImpression`, `Communication`, `DeviceUsage`,
`GuidanceResponse`, `MedicationDispense`, `Procedure`, `SupplyDelivery`,
etc.) and add any that are demonstrably Event-pattern but absent from
the current context list, so a single change closes the broader gap the
reporter raised. CBCP signs off on the `Consent` addition; the
Extensions Pack steward (FHIR-I) makes the actual edit.

#### Justification

This is exactly what the reporter (the CBCP work-group lead) and the
Zulip discussion converged on: `Consent` is an Event-pattern resource
whose primary lifecycle event is revocation, and revocation is precisely
the use case `event-statusReason` was designed for. Lloyd's comment on
the ticket explicitly states FHIR-I is "fine with this in principle"
provided CBCP agrees, so the inter-WG approval path is straightforward.
Broadening an extension's `context` is a non-breaking change — existing
users of the extension are unaffected, and downstream profiles can
immediately bind a coded reason to a revoked `Consent` without waiting
for an R6 model change. Sweeping in any other missing Event-pattern
resources at the same time honours the reporter's parenthetical question
and avoids a stream of follow-up tickets.

---

### Disposition B: Alternative Approach

#### Proposal

Decline to broaden the `event-statusReason` extension and instead address
the underlying need natively in `Consent` for R6: add a `statusReason :
CodeableConcept` element directly on `Consent` (modelled after the
analogous element on other Event-pattern resources such as
`MedicationDispense.statusReason` and `Task.statusReason`), bound to an
example-strength value set seeded by CBCP (with placeholder codes for
common revocation rationales such as patient-withdrawal, expiry,
supersession, mistake-correction). For pre-R6 users, publish an
inter-version extension that mirrors the new R6 element so R4/R5
implementers get the same shape now. Leave `event-statusReason` as it is
today and direct implementers to the inter-version extension instead.

#### Justification

This is the second alternative Lloyd raised on the ticket. A native
`Consent.statusReason` is the most discoverable and self-documenting
solution — implementers reading the `Consent` resource page will find it
immediately, whereas an Extensions-Pack context tweak is invisible from
the `Consent` page itself. It also lets CBCP own the value set, which
matters because the codes appropriate for "why was this consent
withdrawn" (patient choice, regulatory, fraud, error correction, …) are
domain-specific to consent and are unlikely to live comfortably in a
generic Event extension's binding. The trade-off vs. Disposition A is
that this is an R6-cycle change with longer lead time and a larger
review surface, and implementers wanting to move now would have to adopt
a fresh inter-version extension rather than the existing
`event-statusReason`.

---

### Disposition C: Decline

#### Proposal

Decline the request and document, on the `Consent` page (or in CBCP
guidance), that the canonical way to record the reason behind a
`Consent` status change is one of the existing mechanisms already
available in R4/R5: (i) `Provenance` pointing at the updated `Consent`
with `Provenance.reason`, (ii) a `Task` whose `Task.reasonCode`/
`Task.statusReason` carries the rationale and whose execution updated
the `Consent`, or (iii) for the ceremony evidence,
`Consent.sourceAttachment` / `Consent.sourceReference`. Direct anyone
needing a code on the `Consent` resource itself to use a local extension
in their IG until CBCP has gathered concrete value-set candidates.

#### Justification

This is defensible on three grounds. First, John Moehrke's own Zulip
responses note that no codeSystem for consent-state rationales has yet
been presented to CBCP, so binding any element — native or via the
generic extension — risks shipping a slot with no community-agreed
codes. Second, the Event-pattern's status-change-reason story is
deliberately distributed across Provenance/Task/statusReason precisely
so that implementers with strong records-lifecycle requirements use
Provenance and lighter-weight implementers use the resource-local field;
broadening `event-statusReason` to `Consent` without first establishing
the codeSystem may push implementers toward (a) before the more robust
(b)/(c) paths are evaluated. Third, this avoids touching the Extensions
Pack at all, keeping its surface stable.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right starting point for the workgroup. The change
is small, surgical, non-breaking, and directly addresses the gap the
reporter identified; FHIR-I has already signalled it is acceptable in
principle pending CBCP concurrence, which is exactly what this ticket is
asking for. Disposition B (native `Consent.statusReason` in R6) is a
legitimate longer-term improvement and the two are not mutually
exclusive — accepting A now unblocks R4/R5 implementers immediately
without prejudicing a future R6 native element, which CBCP can pursue
once a value set has been collected. Disposition C is the weakest
option: implementers in the Zulip thread are actively looking for a
recorded, structured reason field today, and pushing them all to
Provenance overstates the ceremony required for the common case
(patient-initiated revocation captured at point of care). CBCP should
agree to A, optionally request that the Extensions Pack steward audit
other Event-pattern resources missing from the context list at the same
time, and separately log a CBCP-owned R6 work item to evaluate a native
`Consent.statusReason` once candidate codes are in hand.
