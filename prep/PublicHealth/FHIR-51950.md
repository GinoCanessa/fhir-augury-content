## Summary

The reporter asks the US Public Health work group to reconsider how the
"Characteristics of Home Environment" concept is modeled in the US Public
Health Profiles Library, specifically whether it should remain (or become) a
`Condition` profile or be expressed as an `Observation`, and how that choice
should align with the USCDI+ "Congregate Living" and "Housing Instability"
data elements. The ticket explicitly notes that the Gravity Project already
has well-defined, coded representations for housing insecurity, and asks the
work group to evaluate alternate modeling options based on that prior art
rather than inventing a parallel approach. No related Jira tickets, GitHub
items, or substantive Zulip discussions were surfaced via cross-references —
the request stands on its own as a modeling-direction question for the IG.

### Disposition A: Accept As Requested

#### Proposal

Refactor the `Characteristics of Home Environment` artifact in the US Public
Health Profiles Library to be modeled as one or more `Observation` profiles
(rather than as a `Condition`), aligned with Gravity Project SDOH patterns.
Concretely:

- Introduce an `Observation` profile (e.g.,
  `us-ph-housing-characteristics-observation`) whose `code` is bound to a
  value set that includes the Gravity Project codes for "congregate living"
  and "housing instability / insecurity", mapped to the corresponding
  USCDI+ data elements.
- Reuse Gravity's existing LOINC/SNOMED CT codings for the assessment
  questions and answer concepts rather than minting new ones.
- Deprecate the prior `Condition`-based representation (or retain it as a
  separate, narrower profile only where a clinical diagnosis of unstable
  housing is being asserted), with a clear migration note in the IG's
  history page.

#### Justification

Gravity has already done the hard terminology and modeling work for
housing-insecurity SDOH screening, and Gravity's pattern is to model SDOH
**screening responses** as `Observation` and only the resulting condition
(when clinically asserted) as `Condition`. Aligning US PH with that pattern
keeps US public-health reporting consistent with the broader SDOH
ecosystem (Gravity IG, USCDI+ SDOH, HL7 SDOH-CC) and avoids semantic drift
where the same real-world concept is a `Condition` in one IG and an
`Observation` in another. The reporter explicitly flagged this alignment as
the goal, and the prior workgroup vote (32-0-6 to *Consider for Future
Use*) shows the workgroup recognised the modeling question as legitimate
but deferred it; the post-STU2 reopen is the appropriate window to do the
work.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing `Characteristics of Home Environment` artifact in place
for backward compatibility, but **add** a parallel `Observation`-based
profile that carries the Gravity-aligned screening data, and publish
explicit **mapping guidance** between the two:

- Retain the current artifact (whatever its current base — `Condition` or
  otherwise) so existing implementers and downstream case-reporting
  consumers are not broken.
- Add a new `Observation` profile that binds to Gravity / USCDI+
  congregate-living and housing-instability value sets.
- Author a mapping table in the IG describing when each profile should be
  used (e.g., "use the `Observation` profile to convey USCDI+ Congregate
  Living / Housing Instability screening responses; use the existing
  artifact only for legacy case-reporting payloads").
- Mark the legacy artifact as *withdrawn at next major version* if/when
  uptake on the new profile is sufficient.

#### Justification

US PH profiles are referenced by other public-health IGs (e.g.,
case-reporting), so an outright replacement risks breaking existing
implementations and validation pipelines. A parallel-profile approach
respects the prior cautious "Considered for Future Use" disposition while
still delivering Gravity / USCDI+ alignment for new implementers.
Trade-off vs. Disposition A: more long-term maintenance burden for the
work group (two profiles to keep in sync), but materially lower disruption
for the installed base.

---

### Disposition C: Decline

#### Proposal

Do not change the modeling of `Characteristics of Home Environment` in the
US Public Health Profiles Library. Instead, document that implementers
needing Gravity-aligned housing-insecurity / congregate-living data should
use the **Gravity SDOH Clinical Care IG** (or USCDI+ SDOH artifacts)
directly, and add a cross-reference from the US PH IG pointing
implementers there. Close FHIR-51950 as *Considered for Future Use* /
*Not Persuasive* with that pointer.

#### Justification

The need described in this ticket — coded, well-defined housing-insecurity
screening — is already met by Gravity. Re-modeling US PH artifacts to do
the same job risks duplication, divergence, and ongoing harmonization
overhead, especially given that the original workgroup vote (32-0-6) was
to defer rather than act. If implementers can satisfy USCDI+ Congregate
Living / Housing Instability via Gravity profiles today, then the
defensible answer is to point at Gravity and avoid creating a US-PH-only
parallel artifact.

---

### Recommendation

**Recommended disposition:** B

The ticket reopens a legitimate modeling concern that the workgroup itself
acknowledged when it deferred the original vote, and outright declining
(C) ignores the reporter's well-founded observation that Gravity has
already solved this in a coded, USCDI+-aligned way. However, jumping
straight to a full refactor (A) is risky for an IG that is already
post-STU2 and is consumed by other public-health IGs (notably
case-reporting, where the existing `characteristics-of-home-environment`
StructureDefinition appears in tooling threads). A parallel
`Observation`-based, Gravity-aligned profile with explicit mapping
guidance gives new implementers the right pattern immediately while
giving existing implementers a graceful migration path. The workgroup
should also commission a `repo-analysis` pass on the US Public Health
Profiles Library repo before scoping the implementation, so the
disposition can be anchored to specific authoring paths and any
cross-repo touch points (e.g., case-reporting dependencies) are surfaced
before changes are made.
