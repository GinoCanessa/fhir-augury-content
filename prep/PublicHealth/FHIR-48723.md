## Summary

This ticket is a documentation-only clarification request against the **Cause
of Death Part 2** profile in the US Vital Records Death Reporting (VRDR) FHIR
IG. The reporter (Saul Kravitz) observes that the profile's current title and
descriptive prose drift from the established VRDR documentation language. He
asks that the profile be retitled "Other Significant Conditions Contributing
to Death", that the Forms mapping table on the profile page be updated to
match the new title, and that the description of the observation value
element (`value.text`, and ideally also `value[x]`) be rewritten so it
reflects the new title rather than the older "Cause of Death Part 2" phrasing.
The ticket explicitly notes that the proposed edits are documentary only and
do not change conformance, the StructureDefinition's machine-readable
constraints, or any wire-format / exchange behavior. No related Jira tickets
were surfaced by the cross-reference index, and no GitHub PR/issue activity
is linked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the documentation edits to the **Cause of Death Part 2** profile in the
VRDR IG exactly as the ticket lists them:

1. Update the profile's `title` (and the human-facing display name used by
   the IG publisher / artifact list) from "Cause of Death Part 2" to
   **"Other Significant Conditions Contributing to Death"**. Leave the
   profile `id`, canonical URL, and `name` (computable identifier) unchanged
   so that conformance resources, references from other VRDR profiles, and
   prior validators continue to resolve.
2. Update the **Forms mapping table** on the profile's narrative/intro page
   so that the row(s) currently labeled "Cause of Death Part 2" use the new
   title text, matching the existing convention for the corresponding death
   certificate field (Item 33 / "Other significant conditions contributing
   to death but not resulting in the underlying cause given in Part I").
3. Rewrite the human-readable `short` / `definition` text on
   `Observation.value[x]` and `Observation.value.text` (i.e., the element
   short labels and definitions in the StructureDefinition's
   differential/snapshot, plus any narrative copy on the profile page) so
   that they describe the captured value as "the other significant
   condition(s) contributing to death" rather than "the cause of death,
   part 2".

All three edits are documentary; the `StructureDefinition` cardinalities,
bindings, slicing, invariants, and the canonical URL are not touched.

#### Justification

The reporter is the long-standing technical lead for VRDR and is asking only
for prose alignment with the death-certificate-form vocabulary that the rest
of the IG (and US standard certificate of death, Item 33) already uses.
"Other Significant Conditions Contributing to Death" is the phrasing
implementers see on the source artifact, so the profile title and
element descriptions matching it reduces confusion for jurisdictions and
EDRS vendors. Because the change is restricted to `title`, narrative, and
element `short`/`definition` text — not `name`, `id`, canonical URL,
cardinalities, or bindings — there is no conformance or wire-format impact,
which the ticket itself confirms. This is the lowest-risk way to resolve
the inconsistency the reporter identified.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent but make the change in two coordinated steps rather than a
single in-place rename:

1. In the next VRDR publication, **add the new wording as the primary
   `title`** ("Other Significant Conditions Contributing to Death") and
   keep "Cause of Death Part 2" as a parenthetical alias in the narrative
   intro, the Forms mapping table caption, and the `Observation.value[x]`
   `short`. Example: *"Other Significant Conditions Contributing to Death
   (Cause of Death Part 2)"*.
2. Schedule the parenthetical alias for removal in the following ballot
   cycle once downstream consumers (VRDR C# / Java reference libraries,
   EDRS vendors, NCHS tooling) have had one publication to update any
   string-based UI labels, documentation, or test fixtures that key off
   the current title.

No `id`, `name`, canonical URL, or constraint changes occur in either step.

#### Justification

The "Cause of Death Part 2" phrasing has been in published VRDR releases for
several cycles and appears verbatim in the VRDR C# / Java reference libraries
and in implementer-facing documentation (e.g., the 2023 VRDR release notes
referenced in the Death on FHIR Zulip stream). Some implementers have
likely embedded the current title in human-readable forms, jurisdictional
crosswalks, or test reports. Carrying the old title as a parenthetical for
one cycle gives those consumers a clean migration window without
sacrificing the alignment Saul is asking for. The trade-off versus
Disposition A is one extra publication cycle of slightly noisier titles;
the upside is fewer "where did the Part 2 profile go?" support questions.

---

### Disposition C: Decline

#### Proposal

Decline the title and prose changes. Keep the profile titled "Cause of Death
Part 2" and leave the Forms mapping table and `value[x]` / `value.text`
descriptions as currently published. Optionally add a one-sentence
clarifying note in the profile narrative stating that "Cause of Death
Part 2" corresponds to "Other Significant Conditions Contributing to
Death" on the standard certificate of death.

#### Justification

The strongest argument for declining is implementer continuity: the current
title has been in published VRDR releases for years and is referenced by
human-readable documentation, vendor UIs, and the VRDR reference libraries.
A pure clarifying note would resolve the cross-reference ambiguity without
touching any published string. However, this rationale is weak because the
ticket touches only narrative text (no conformance impact), the reporter is
the IG's technical lead, and the proposed wording is the literal
death-certificate field name. There is no compelling FHIR design principle
or scope concern that justifies refusal.

---

### Recommendation

**Recommended disposition:** A

The change is purely documentary, the reporter has explicitly bounded it as
non-conformance-affecting, and the proposed title is the exact phrasing used
on the US standard certificate of death (Item 33), which is the authoritative
source the VRDR IG aims to model. Accepting as requested produces the
clearest IG and the smallest long-term maintenance burden. Disposition B's
parenthetical alias would be defensible if the WG hears, during review, that
specific implementers (NCHS tooling, jurisdictional EDRS vendors, the VRDR
reference libraries) have hard-coded the current title in user-facing
strings; absent that signal, the dual-title period is unnecessary noise.
Disposition C should be rejected because the inconsistency the reporter
identified is real and the fix is cheap. Care should still be taken during
implementation to leave the profile's `id`, `name`, and canonical URL
untouched so that existing conformance references and validator caches
continue to resolve.
