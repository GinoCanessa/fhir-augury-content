## Summary

The reporter (Craig Newman) observes that every value set defined in the
US Healthcare Associated Infections – Long Term Care Facility (HAI-LTCF)
FHIR IG is annotated with the statement **"This value set is not used"**.
He asks the Public Health work group to clarify the situation: if the
statement is accurate, why are these value sets published at all; and if
it is inaccurate, where and how are the value sets actually consumed in
the IG. The ticket is essentially a documentation / publication-quality
issue against the HAI-LTCF IG and seeks either a corrective edit to the
value-set descriptions, removal of unused value sets, or visible binding
of the value sets to the relevant profiles/elements so the "not used"
banner no longer appears.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
located for this ticket via the `fhir-augury-cli` search and
cross-reference endpoints.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation defect against the HAI-LTCF IG and resolve
it in the IG sources. Concretely: (1) audit every `ValueSet` resource
defined in the HAI-LTCF IG; (2) for each value set, determine whether
any profile, extension, questionnaire, or measure in the IG references
it (as a `binding`, `compose.include.valueSet`, `Questionnaire.item.answerValueSet`,
`Measure` population criteria, etc.); (3) for value sets that **are**
referenced, fix the IG so the publisher no longer emits the "This value
set is not used" banner — typically by ensuring the binding actually
resolves to the canonical URL and is detected by the IG publisher, and
by adding human-readable narrative explaining where the value set is
used; (4) for value sets that are genuinely unused, either bind them to
the appropriate profile element (if the omission was an oversight) or
remove them from the IG (and from the package) so the published IG
contains only artifacts that play a role.

#### Justification

The "This value set is not used" string is generated by the IG publisher
when nothing in the IG references the value set's canonical URL. If the
banner appears on every value set, either the bindings were never wired
up or the IG is shipping unused terminology. Both outcomes mislead
implementers — they cannot tell which value sets are normative
expectations of the IG. Fixing the wiring (or removing dead artifacts)
is the standard, low-controversy response and aligns with HL7's general
expectation that a published IG only contains artifacts that contribute
to its conformance story.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than removing or rewiring the value sets one-by-one, retain them
but add an explicit "Terminology Reference" page (or section in the
existing terminology page) to the HAI-LTCF IG that states the value sets
are published as **reusable terminology assets** for downstream HAI
reporting workflows (e.g., NHSN-aligned profiles, future HAI IGs, local
implementations) even though the current IG does not bind them to a
specific profile element. Add a `ValueSet.useContext` or narrative
disclaimer to each value set explaining the intended consumer, and
suppress / replace the publisher's auto-generated "not used" banner via
the IG's narrative override mechanism so readers see the curated
explanation instead of the misleading default text.

#### Justification

Some Public Health IGs intentionally publish value sets as shared
terminology that other IGs or local implementations will bind to, even
when the host IG does not itself bind them. If that is the design
intent for HAI-LTCF, removing the value sets (Disposition A's
fallback) would break downstream consumers, while leaving the
publisher's default banner in place — as today — leaves implementers
confused. A documentation override plus an explicit "reusable
terminology" framing addresses the reporter's underlying concern
("please clarify") without disturbing the published canonical URLs.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive / Out of Scope** on the grounds
that the HAI-LTCF IG has not been updated since the period when this
ticket was filed (2019) and there is no active maintenance project to
absorb the change; redirect the reporter to raise the issue against the
successor HAI reporting IG (or the relevant CDC/NHSN-driven
specification) once one is balloted.

#### Justification

The ticket has been sitting in Triaged status since 2019 with no
comments, no linked artifacts, and no cross-references in any of the
data sources surfaced through `fhir-augury-cli`. If the Public Health
work group is no longer maintaining HAI-LTCF as an active publication,
spending ballot/editorial cycles on cosmetic fixes to its value-set
narrative is unlikely to benefit implementers. Declining with a clear
forward pointer is more honest than accepting work that will not be
delivered.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a real, verifiable defect — every value set
in the HAI-LTCF IG carries a publisher-generated "not used" banner,
which is either factually wrong (bindings were lost) or correctly
indicates that unused artifacts are being published. Either way the
current state misleads implementers and undermines confidence in the
IG. Disposition A (audit and fix the bindings, remove genuine dead
terminology) is the cleanest resolution and is the response the IG
publisher's banner is explicitly asking the editors to perform.
Disposition B is reasonable only if the work group can confirm that the
value sets are deliberately published as reusable terminology assets;
absent that confirmation, accepting B risks codifying the current
confusing state with a thin narrative wrapper. Disposition C should be
held in reserve for the case where Public Health confirms the IG is
genuinely abandoned — but that decision should be made explicitly,
not by silently leaving the misleading banners in place.
