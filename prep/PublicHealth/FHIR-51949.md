## Summary

The ticket asks the US Public Health Profiles Library (USPHPL) work group to
decide how Chief Complaint should be modeled. The current STU2 draft maps
Chief Complaint in two places at once: a new `Chief Complaint Observation`
profile **and** `Encounter.diagnosis` with `use = "CC"`. The reporter
(Sarah Gaunt) recommends retaining only the `Chief Complaint Observation`
profile and dropping reliance on `Encounter.diagnosis.use = "CC"` because
(a) `Encounter.diagnosis` references `Condition`, which implies a
clinician-framed diagnosis rather than the patient's own words;
(b) Observation allows clean retrieval by `code` + `effective[x]`;
(c) mixing chief complaint into `Encounter.diagnosis` risks confusion with
provider diagnoses; and (d) a single representation improves data
consistency and accuracy. Cross-reference analysis surfaced only FHIR
element mentions of `Encounter.diagnosis` and `Encounter.diagnosis.use`
(no linked Jira, Zulip, or GitHub items). The ticket was originally
dispositioned *Considered for Future Use* on 2025-12-04 (vote 32-0-6,
Gaunt/Rutz) with the note "We will re-visit this next release"; Lynn
Laakso reverted that resolution on 2026-02-17 to re-open the item
post-STU2 publication, so it is now back on the agenda for the next
ballot cycle of the USPHPL.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the `Encounter.diagnosis` / `use = "CC"` path from USPHPL's
Chief Complaint guidance and retain `Chief Complaint Observation` as
the single, normative representation. Concretely, in the USPHPL IG:

1. Keep the `Chief Complaint Observation` profile (an `Observation`
   profile with `code` bound to the chief-complaint value set,
   `valueString` / `valueCodeableConcept` open enough to capture the
   patient's own words, and `effective[x]` populated).
2. In the Encounter-related profile(s), remove any slice / guidance
   that assigns `Encounter.diagnosis.use = "CC"` for Chief Complaint,
   or explicitly mark it as "do not use for chief complaint — use
   `Chief Complaint Observation` instead".
3. Update the IG narrative, examples, and any capability-statement
   expectations (search params, must-support flags) so that consumers
   retrieve chief complaint via
   `Observation?patient=…&code=…&date=…` rather than by walking
   `Encounter.diagnosis`.
4. Add a migration note for STU2 implementers who began wiring up the
   `Encounter.diagnosis.use = "CC"` pattern.

#### Justification

This mirrors exactly what the reporter asked for and aligns with the
bullets listed in the ticket: `Encounter.diagnosis` references
`Condition`, which is a clinician-authored, coded diagnosis and cannot
carry the patient's verbatim complaint; `Observation` is the standard
FHIR pattern for patient-reported findings at a point in time and
supports trivial retrieval by `code` + `date`. Collapsing to one
representation also avoids the well-known problem of systems populating
one path but not the other, which would produce inconsistent query
results across implementations. The November 2025 WG vote (32-0-6) to
defer this issue indicates a broad sense that the current dual mapping
is at best temporary; re-opening post-STU2 specifically to revisit this
supports a clean consolidation rather than another deferral.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both representations but make them non-overlapping and clearly
scoped:

1. Normative guidance: "Chief Complaint **in the patient's own words**
   SHALL be represented with `Chief Complaint Observation`."
2. Permissive guidance: `Encounter.diagnosis` with
   `use = http://terminology.hl7.org/CodeSystem/diagnosis-role#CC`
   MAY be used **only** when the reason-for-visit is already a
   clinician-coded `Condition` (not a patient statement), and systems
   SHOULD also surface the corresponding `Chief Complaint Observation`
   when patient-verbatim data exist.
3. Add an invariant or explicit text note reminding implementers that
   `Encounter.diagnosis` cannot carry patient-verbatim text because
   its target is `Condition`.
4. Keep a single recommended retrieval pattern (Observation) in the
   IG's "how to query" section so consumers are not forced to walk two
   paths.

#### Justification

Some EHRs already populate `Encounter.diagnosis` with a "CC"-roled
`Condition` from a structured pick-list and cannot easily retire that
path before STU3. Giving that pattern a narrow, documented home
preserves backward compatibility for those vendors while still
steering new implementations toward `Chief Complaint Observation` for
the patient-reported case. The trade-off vs. Disposition A is added
complexity and a continued risk of divergent implementations; the
trade-off vs. Disposition C is that the reporter's core concern
(confusion with provider diagnosis, inability to carry the patient's
own words) is explicitly addressed in normative text rather than
ignored.

---

### Disposition C: Decline

#### Proposal

Leave the STU2 dual mapping (`Chief Complaint Observation` **and**
`Encounter.diagnosis` with `use = "CC"`) in place for this release
cycle and close the ticket as *Not Persuasive* or *Considered for
Future Use* (again), on the grounds that: (a) STU2 has just been
published and churning the Chief Complaint mapping creates a breaking
change for early adopters; (b) implementers retain the option to use
whichever representation fits their source data; and (c) the issue can
be revisited when there is concrete implementer feedback showing harm.

#### Justification

This ticket was already dispositioned *Considered for Future Use* in
December 2025 by a strong 32-0-6 vote. The only new information since
then is Lynn Laakso's February 2026 procedural re-open post-STU2
publication — there is no new implementer evidence, no new Zulip
discussion, and no cross-referenced tickets indicating field pain.
Declining (or re-deferring) until STU3 preserves STU2 stability and
avoids re-litigating a decision the WG already took with a near-
unanimous vote.

---

### Recommendation

**Recommended disposition:** A

The ticket identifies a genuine modeling conflict that is not merely
stylistic: `Encounter.diagnosis` targets `Condition`, which structurally
**cannot** carry the patient's verbatim complaint, so the `use = "CC"`
path is fundamentally mis-scoped for the chief-complaint use case.
Keeping two representations (Disposition B) preserves a pattern that is
semantically wrong and guarantees inconsistent data across
implementations — exactly the "consistency in capturing data" problem
the reporter called out. Declining (Disposition C) is procedurally
defensible given the December vote, but that vote was expressly
conditioned on revisiting the issue next release, and Lynn Laakso's
re-open is that revisit; re-deferring a second time without new
evidence would be hard to justify to the reporter and to the
`2025_USPHPL_SEEK_FEEDBACK` audience. Disposition A is the cleanest
answer, matches the reporter's rationale, and simplifies the query
story for consumers; the main cost — a migration note for any
implementers who wired up `Encounter.diagnosis.use = "CC"` during STU2
— is modest and can be handled with a short "changes from STU2" entry
in the IG.
