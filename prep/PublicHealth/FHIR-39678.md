## Summary

This ticket is a Question raised against the **US MedMorph Research Data
Exchange Content IG**, asking how the IG's Knowledge Artifact "action
sequence" accommodates use cases in which a report must be compiled over
a **long clinical episode**, or in which **outcomes are tracked over a
period of time after an index event**. The reporter further asks
whether such a long-running / post-index-event scenario should be
documented as its own generic sequence in the IG. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced for this
ticket, so the request stands alone — it is a clarification /
gap-analysis question about the IG's modeling of the Knowledge Artifact
processing flow rather than a concrete change request.

## Details

**Description:**

How does the sequence accommodate use cases that require a report to be
compiled over the duration of a long episode, or look to track outcomes
over a period of time post-index event?  Should this scenario be laid
out in a generic sequence?

## Proposed Dispositions

The dispositions below are framed against the US MedMorph Research Data
Exchange (RDE) Content IG, which models its processing flow as a
Knowledge Artifact (`PlanDefinition`) whose `action` elements describe
the sequence the Backend Service Application executes (check trigger,
evaluate criteria, build report, submit, etc.). Because no GitHub
cross-references were returned, no specific repository / authoring root
is implicated; if the workgroup chooses an option that requires spec
changes, the affected repo is the MedMorph RDE IG repository under HL7
and the work would land in its `input/` authoring tree (PlanDefinition
examples, the IG narrative pages describing the sequence, and any
accompanying CapabilityStatement / extension definitions).

### Disposition A: Accept As Requested

#### Proposal

Add a new "long-running episode / longitudinal outcomes" generic
sequence to the RDE Content IG alongside the existing reporting
sequence. Concretely:

- Author a new narrative page in the IG that describes a generic
  long-episode sequence: trigger on index event, periodic
  re-evaluation over a configurable observation window, incremental
  report assembly, and a final / outcome-reporting submission.
- Provide a companion example `PlanDefinition` whose `action`
  elements model that sequence — e.g., an outer action with a
  timing/period that drives recurring child actions for
  data-collection, criteria re-evaluation, and partial / final report
  submission.
- Define (or reuse) the extensions needed to express
  observation-window duration, "index event" anchoring, and
  partial-vs-final report semantics on `MessageHeader` /
  `Bundle.meta` so receivers can correlate longitudinal submissions.

#### Justification

The existing RDE sequence is implicitly oriented around a discrete
encounter-style trigger → evaluate → report cycle. Long-episode and
post-index-event use cases are real public-health and research
scenarios (e.g., pregnancy episodes, oncology follow-up, post-discharge
outcomes), and asking implementers to invent their own pattern
fragments the ecosystem. A generic sequence in the IG gives
implementers a single, conformant pattern to extend, and keeps the
Knowledge Artifact authoring story aligned with the way clinicians and
researchers already think about longitudinal reporting.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** introduce a new top-level generic sequence. Instead, clarify
in the existing sequence narrative that long-episode / longitudinal
use cases are supported by the **same** sequence used recursively or
periodically:

- Add a non-normative subsection to the current sequence page
  explaining how `PlanDefinition.action.timing[x]` (e.g., `Timing` with
  a repeat) and `PlanDefinition.action.trigger` can be combined to
  re-fire the existing reporting sequence at intervals over an
  episode.
- Provide one or two worked example `PlanDefinition`s (a
  pregnancy-episode style example and a post-index follow-up style
  example) that show this pattern in the existing sequence.
- Cross-link from the example narrative to the generic sequence page
  so implementers find the pattern when they search for "longitudinal"
  / "follow-up" / "episode".

#### Justification

This addresses the underlying need (implementers don't know how to do
long-episode reporting with the IG today) without introducing a
parallel sequence that implementers must learn and reconcile against
the existing one. Keeping a single sequence and demonstrating its
flexibility through worked examples is generally lower implementation
burden, avoids divergent conformance requirements, and is easier to
maintain across IG versions. It also avoids prejudging whether
outcomes-reporting truly needs distinct semantics or just distinct
timing/trigger configuration — the workgroup can promote it to a
first-class sequence later if real implementations show that examples
are insufficient.

---

### Disposition C: Decline

#### Proposal

Close this ticket as **Not Persuasive — out of scope for the current
publication**. Record that the IG's current sequence is intentionally
generic and that long-episode / outcomes-tracking flows are considered
implementation choices on top of the existing PlanDefinition action
model. Suggest the question be re-raised against a future ballot of
the RDE IG (or against the MedMorph Reference Architecture IG) once
concrete long-episode use cases with implementer commitment are
brought forward.

#### Justification

The ticket is a Question with no concrete proposed change, no
cross-references, no Zulip discussion, no linked tickets, and has been
sitting in Triaged for an extended period. Without a sponsoring
implementer or a worked use case, designing a new generic sequence
risks adding speculative normative content. The existing sequence does
not actively prevent long-episode reporting, so declining is defensible
on a "no demonstrated gap, no proposed change" basis.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying question — "is this scenario supported, and
if so where is it shown?" — is most efficiently answered by clarifying
the existing sequence and adding worked long-episode examples, rather
than by either standing up a parallel generic sequence (Disposition A,
which adds normative surface area for a use case with no sponsoring
implementer recorded against this ticket) or by closing it outright
(Disposition C, which leaves the documentation gap in place).
Disposition B preserves the single-sequence story that the IG has
today, gives implementers concrete patterns they can copy for
long-episode and post-index-event reporting, and keeps the door open
for the workgroup to promote a dedicated sequence in a future ballot
cycle if real-world implementations show that examples alone are
insufficient.
