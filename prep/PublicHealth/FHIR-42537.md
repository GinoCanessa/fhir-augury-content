## Summary

Sarah Gaunt is asking the BSeR project team to revisit the *bundling
paradigm* used to convey BSeR payloads. Today the BSeR IG packages
referral payloads inside `Composition`-rooted document Bundles. The
ticket questions whether that document paradigm is the right choice for
all BSeR exchange patterns, particularly the **push** (initiator-driven)
pattern, and whether a `Bundle` of type `searchset` (a "query response
bundle") — or some other mechanism — would be more appropriate when the
recipient is pulling the payload from an EHR. The reporter explicitly
ties this to the broader question of how BSeR defines its payloads and
asks whether other bundling options should be considered. The ticket is
a Question (not a concrete change request) and is intended to seed a
workgroup conversation rather than mandate a specific change.

The cross-reference index returned **no direct cross-references** for
this ticket, so the related-by-keyword neighbours below were used to
provide additional context. The closest analogue is
[FHIR-30725](https://jira.hl7.org/browse/FHIR-30725) ("Consider document
paradigm for response.") — the same architectural question, but raised
against MedMorph, where the workgroup ultimately moved the response
bundle toward a Document/Composition structure (Persuasive with
Modification). [FHIR-41194](https://jira.hl7.org/browse/FHIR-41194)
covers the same push-vs-query trade-space for eRSD/eCR. The PSS items
[PSS-2489](https://jira.hl7.org/browse/PSS-2489) and
[PSS-2561](https://jira.hl7.org/browse/PSS-2561) record an explicit
Public Health initiative to use a FHIR Query & Response paradigm for
public-health data exchange. Together these establish a prior
Public-Health-WG pattern: prefer the document paradigm (Composition)
when the payload is a curated, signable, EHR-displayable artifact, and
prefer query/subscription mechanics for negotiation and transport.

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as the *intended* trigger for a workgroup conversation
and produce a normative answer in the BSeR IG that explicitly enumerates
the bundling paradigm for each BSeR exchange pattern. Concretely:

- Keep `Bundle.type = document` (Composition-rooted) as the canonical
  payload wrapper for the **push** path, where BSeR conveys a
  signed/curated referral package between organisations.
- Allow `Bundle.type = searchset` (a query response Bundle) as the
  canonical wrapper when the recipient is **querying** the holder of the
  referral data (e.g., an EHR being queried for referral content), and
  document the resource-selection rules.
- Add an "Exchange Paradigms" section to the BSeR IG that lays out the
  push vs query paths, the bundle type each path uses, the rationale,
  and any profiling constraints (e.g., MessageHeader/Communication
  expectations on the push path; `_include` / search-parameter
  expectations on the query path).
- File the structural changes against the BSeR IG repo (`HL7/bser`).

#### Justification

This is what the reporter is asking for in spirit — a clear answer to
"is Composition the best way?" — and it aligns with the precedent set by
[FHIR-30725](https://jira.hl7.org/browse/FHIR-30725), where the
Public-Health WG affirmed the document paradigm for MedMorph response
bundles while leaving room for other shapes when the use case differs.
Documenting both paradigms preserves BSeR's existing Composition-based
conformance (no breaking change for current implementers) while giving
query-driven implementers an officially-sanctioned shape, consistent
with the WG-level direction in
[PSS-2489](https://jira.hl7.org/browse/PSS-2489) /
[PSS-2561](https://jira.hl7.org/browse/PSS-2561).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than dual-mode the payload Bundle, **decouple bundling from
exchange paradigm**: keep the BSeR payload itself a
Composition-rooted document Bundle (so the curated, attestable referral
artifact is consistent across paths), and use *transport-layer* FHIR
mechanisms for the paradigm choice:

- **Push path**: convey the document Bundle via a `MessageHeader`
  message Bundle (or `Communication.payload` / `Task.input`), with the
  document Bundle carried as the focus / payload.
- **Pull path**: expose the same document Bundle through a FHIR
  search/operation (e.g., `$everything`-style operation or a focused
  search on `Composition` / `Task` filtered by referral identifier);
  the search response is naturally a `Bundle.type = searchset`
  containing the document Bundle (or its top-level resources) but the
  *payload definition* doesn't fork.
- Optionally adopt a topic-based **R5 subscription** notification to
  signal availability, mirroring the direction explored in
  [FHIR-41194](https://jira.hl7.org/browse/FHIR-41194) and
  [FHIR-38982](https://jira.hl7.org/browse/FHIR-38982).

#### Justification

Forking the payload shape (document vs searchset) by paradigm — as in
Disposition A — creates two parallel conformance profiles that
implementers must support, and risks subtle semantic drift between
"push BSeR" and "pull BSeR" payloads. Holding the *payload* invariant
(always a Composition-rooted document Bundle) and varying only the
*envelope* keeps the referral artifact canonical, signable, and
auditable, and avoids breaking existing BSeR implementations that
already treat the payload as a document. It also keeps BSeR aligned
with the MedMorph resolution in
[FHIR-30725](https://jira.hl7.org/browse/FHIR-30725) and lets BSeR
pick up the subscription-based notification machinery being explored
elsewhere in Public Health without a payload-shape change.

---

### Disposition C: Decline (Resolve "Not Persuasive" / "Question
Answered")

#### Proposal

Close the ticket with a written response on the ticket itself rather
than a spec change: confirm that BSeR's existing
`Composition`/document-Bundle wrapping is intentional and remains the
correct payload shape for both push and pull, on the grounds that (a)
the document paradigm is what gives BSeR its provenance/attestation
story, (b) a query response Bundle (`searchset`) is a *transport
artifact*, not a payload artifact, and is automatically the shape any
conformant FHIR server returns when BSeR resources are queried, so
nothing needs to be specified, and (c) the workgroup has already
trended this way in MedMorph (FHIR-30725 → document paradigm).

#### Justification

The ticket is a `Question`, not a `Change Request`, with no concrete
proposed wording. Declining (with a written rationale linking to
FHIR-30725) is defensible if the WG believes the current Composition
wrapping is correct and the apparent ambiguity is a documentation
issue at most. This avoids spec churn and avoids creating a
dual-conformance burden for implementers. The risk is that the
question genuinely surfaces an interoperability gap (push payloads vs
query results not being clearly distinguished) and that closing
without action leaves implementers to invent local conventions —
which is exactly the failure mode that triggered FHIR-41194 in the
eRSD/eCR space.

---

### Recommendation

**Recommended disposition: B (Alternative Approach).**

Disposition B best matches the WG's prior direction (FHIR-30725 affirmed
the document paradigm for a closely-related Public-Health IG) while
also taking the reporter's concern seriously: the underlying problem
isn't that Composition is wrong, it's that BSeR doesn't say enough
about what happens on the *pull* path. Holding the payload invariant
as a Composition-rooted document Bundle and addressing push vs pull at
the transport/envelope layer keeps BSeR's referral artifact canonical
and signable, avoids forking implementers' conformance work, and
positions BSeR to adopt R5 topic-based subscription notifications in
line with the broader Public-Health direction reflected in PSS-2489 /
PSS-2561 and explored in FHIR-41194 and FHIR-38982. Disposition A is a
reasonable second choice if the WG decides documenting both bundle
shapes is more pragmatic than introducing transport-layer machinery.
Disposition C is the weakest option — the ticket comes from an active
BSeR contributor (Sarah Gaunt) and silently closing it risks the same
ambiguity-driven divergence that has bitten other Public-Health IGs.
