## Summary

The reporter requests that the US Vital Records Death Reporting (VRDR) FHIR
Implementation Guide include additional example instances that illustrate the
exchange of death-related data between an Electronic Health Record (EHR) and
an Electronic Death Registration System (EDRS) — i.e., the EHR→EDRS workflow.
Today the VRDR IG centers its examples on the downstream VRO/Jurisdiction
exchange, and the EHR-to-EDRS portion of the pipeline is comparatively
under-illustrated. The request is purely additive (more examples), not a
change to profiles, value sets, or operations. The ticket title uses the
abbreviation "EHR-ERDS"; based on the comments and standard usage in the
VRDR space this should be read as "EHR-EDRS" (Electronic Death Registration
System).

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket via the augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Author and publish a new set of EHR→EDRS example instances in the VRDR IG,
covering at least the following scenarios:

1. A pronouncing/certifying clinician in an EHR transmits decedent
   demographics, encounter context, and observed cause-of-death information
   to an EDRS to seed a new death case.
2. An EDRS reconciles/updates an in-progress case after the EHR sends
   corrected or amended cause-of-death data.
3. A minimal "EHR-only known facts" handoff for an in-hospital death where
   the EHR is the source of pronouncement details but not the final
   certifier.

Each scenario should be expressed as a Bundle of the existing VRDR profiles
already used in the IG (Decedent Patient, Death Certification-related
Observations, Cause of Death conditions, Practitioner/Organization for the
EHR-side actors), referenced from a new "EHR-to-EDRS Examples" subsection
of the IG's examples/use-cases page. No profile, value set, or operation
changes are proposed; this is example content only.

#### Justification

The ticket asks for examples and the work-group commenter (cynthia_bush)
has already affirmed that "example instances will be created for EHR-EDRS
content." Adding examples is a low-risk, additive change that improves
implementer comprehension of a workflow the IG already supports
conceptually but under-illustrates. It also aligns with HL7 IG authoring
norms, which encourage worked examples for each major exchange in scope.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) adding raw example Bundles, add a narrative
**EHR-to-EDRS Workflow** page to the VRDR IG that describes the actors,
the trigger events, the expected payload shape, and the sequencing of
exchanges, and then anchor 1–2 canonical example Bundles to that page.
Defer authoring a broader matrix of examples until input has been gathered
from VROs and their EDRS partners (per cynthia_bush's 2025-09-05 comment),
and track that input-gathering as a follow-up sub-task before publishing
the additional Bundles.

#### Justification

The second comment explicitly calls out that real VRO/EDRS implementer
input is needed before locking down what the exchange looks like. Shipping
a single narrative page plus a small, vetted set of examples avoids
publishing many examples that may later need to be reworked when VRO/EDRS
feedback lands. It also gives implementers more pedagogical value than
example Bundles alone, which can be ambiguous about *why* each element is
populated. Trade-off vs. A: slower to deliver complete example coverage,
but lower risk of churn and better alignment with the work group's stated
need for partner input.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for the current VRDR ballot cycle on the grounds
that example content can be maintained in companion materials (e.g., a
VRDR examples repository, Connectathon track artifacts, or
education/training decks) rather than as normative content in the IG, and
that no specification defect has been identified.

#### Justification

The ticket does not allege an error or a gap in the conformance content of
the IG; it requests additional illustrative material. If the work group is
capacity-constrained, deferring example expansion to companion artifacts
is defensible and avoids enlarging IG scope mid-cycle. This is the
weakest option because the work-group commenter has already committed to
producing the examples, but it is offered for completeness.

---

### Recommendation

**Recommended disposition:** B

Disposition B best honors both the ticket's intent and the work group's
own stated caveat. Cynthia Bush has agreed examples will be authored
(supporting A), but she has also explicitly conditioned that work on
gathering input from VROs and their EDRS partners. Shipping a narrative
EHR-to-EDRS workflow page anchored by 1–2 carefully chosen example Bundles
satisfies the requester's underlying need (clarifying the exchange) while
leaving room to iterate on the broader example set once partner input is
in. Pure Disposition A risks publishing examples that get reworked after
partner review; Disposition C contradicts the work group's expressed
intent. The work group should confirm whether partner outreach can
realistically complete in time for the next publication, and adjust the
scope of the initial example set accordingly.
