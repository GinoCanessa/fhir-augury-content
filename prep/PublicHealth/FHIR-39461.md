## Summary

The ticket reports that the Decedent profile's `BirthRecordIdentifier`
extension — specifically the `component[birthYear]` slice that maps to the
IJE `IDOB_YR` (decedent's year of birth) field — cannot represent every
value `IDOB_YR` is allowed to carry. The IJE field can be a four-digit
year, the special sentinel `9999` meaning "unknown", or blank (not
specified). Because the FHIR component currently constrains the value to
a complete year, the `9999` "unknown" case has no clean representation.
Saul Kravitz proposes adding a `partialDate` extension on the value field
so implementers can flag the year as unknown rather than asserting a
real year. The ticket carries the `pre-STU3-ballot` label, was previously
resolved as "Considered for Future Use" (Passas/Kravitz, 25-0-0) on
2024-03-28, and was then re-opened post-STU3 publication by Lynn Laakso
on 2024-10-16 for consideration in the next ballot version.

## Details

**Description:**

> Current BirthRecordIdentifier.component[birthYear] supports the IDOB_YR
> field in IJE.
>
> IDOB_YR can be:
>  - year of death (YYYY)
>  - year of death - 1 (YYYY)
>  - unknown (9999)
>  - blank (not specified)
>
> The current profile can't support the 9999 case.
>
> Solution: Add a partialDate extension to the value field that would
> support the 9999 case.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an optional `partialDate` extension to the value of
`BirthRecordIdentifier.component:birthYear.value[x]` (typed as
`positiveInt` / `string` per the existing slice) on the Decedent profile
in the VRDR FHIR IG. The extension should follow the standard FHIR
`partial-date-time` style (e.g., a `data-absent-reason` companion or a
dedicated `unknownYear`/`partialDate` extension URL specific to VRDR)
and be defined so that when the year is unknown the `value[x]` may be
omitted (or carry a stand-in) and the extension carries
`absent-reason = "unknown"`. Update the IJE → FHIR mapping table for
`IDOB_YR` to specify:

- `YYYY` → `value[x]` populated with the four-digit year, no extension.
- `9999` → `value[x]` omitted (or carries `data-absent-reason`), with
  the new extension flagging the year as unknown.
- blank → component omitted entirely.

Document round-trip rules in the IJE-to-FHIR mapping section so
producers and consumers reconstruct the same `IDOB_YR` value.

#### Justification

This is what the reporter explicitly requested and the cleanest way to
preserve loss-less round-trip with IJE, which the VRDR IG is committed
to supporting. The 25-0-0 "Considered for Future Use" vote from
2024-03-28 was reverted specifically so the change could be considered
for the next ballot, signalling that the work group is open to
addressing it now. The IJE specification distinguishes "unknown"
(`9999`) from "not specified" (blank), and producers that drop one into
the other lose information that downstream public-health analytics rely
on. A targeted extension keeps the change additive and backward
compatible with existing STU3 instances that simply omit the year.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a new `partialDate` extension, re-use the
existing FHIR `data-absent-reason` extension on the
`component:birthYear.value[x]` element. Update the Decedent profile to:

1. Make `component:birthYear.value[x]` optional (cardinality `0..1`)
   where it is currently required.
2. Add a slice / guidance that when the value is absent and `IDOB_YR`
   was `9999`, the `data-absent-reason` extension MUST be present with
   code `unknown`.
3. Treat blank `IDOB_YR` as the component being entirely omitted (no
   change from today).

Document the IJE mapping the same way as Disposition A, but using the
core extension URL `http://hl7.org/fhir/StructureDefinition/data-absent-reason`
instead of a VRDR-specific `partialDate` extension.

#### Justification

`data-absent-reason` is a normative, widely-implemented core FHIR
extension whose `unknown` code maps directly to the IJE `9999`
sentinel. Re-using it avoids minting a VRDR-specific extension that
duplicates an existing concept, simplifies validator and tooling
support, and aligns VRDR with how other Public Health IGs (e.g., US
Core, BFDR) represent unknown values. The trade-off versus
Disposition A is expressiveness: `partialDate` can also carry a
partially-known year (e.g., decade, century), but the ticket only
identifies the binary "known year vs. unknown" case, so the simpler
mechanism is sufficient.

---

### Disposition C: Decline

#### Proposal

Decline the change and continue to instruct VRDR implementers to omit
the `birthYear` component when `IDOB_YR` is either `9999` or blank,
relying on the absence of the component to signify "year not known or
not specified". Add only a documentation note in the IJE mapping
section recording that the FHIR profile does not preserve the
distinction between `9999` and blank, and that systems requiring the
distinction should keep it in their IJE pipeline.

#### Justification

The previous Public Health work-group vote on 2024-03-28 was 25-0-0
for "Considered for Future Use" with the explicit comment "No
requirement has been identified for this ticket." Although that
resolution was reverted procedurally to make room for re-balloting, no
new use cases have been recorded against the ticket since 2022, and
no Zulip or GitHub discussion appears in Augury. Declining keeps the
profile stable, avoids introducing an extension whose only consumer is
a single IJE corner case, and preserves the existing community
consensus that the loss of the `9999` vs. blank distinction has not
caused a documented downstream problem.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the underlying interoperability need — being
able to assert that the decedent's birth year is unknown — without
inventing a VRDR-specific `partialDate` extension that overlaps with
existing core FHIR machinery. `data-absent-reason` with code `unknown`
is unambiguous, already supported by standard validators and FHIR
clients, and maps cleanly to IJE `9999`; blank `IDOB_YR` continues to
map to the component being absent. This satisfies the reporter's goal,
honours Lynn Laakso's call to reconsider the issue for the next ballot,
and respects the prior work-group concern that no concrete requirement
existed for a bespoke extension. Disposition A remains a defensible
fallback if the work group anticipates needing to express partially
known years (e.g., decade-only) in future revisions; Disposition C is
hard to defend now that the previous resolution has been explicitly
reverted for re-consideration.
