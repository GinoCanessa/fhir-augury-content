## Summary

The ticket reports a gap in the current VRDR STU3 (US Vital Records Death
Reporting) FHIR specification: the maternal-death linkage mechanism cannot
represent the case where the decedent's pregnancy ended in a multiple
birth (e.g., twins, triplets) or in one or more fetal deaths. Today the
linkage assumes a 1:1 relationship between the maternal death record and
a single birth or fetal-death record. Saul Kravitz proposes that VRDR
needs to be able to link to multiple birth/fetal-death events, or — at a
minimum — be able to indicate that the linked event was part of a
multiple birth, so that surveillance and follow-up activities can
correctly enumerate the affected children.

### Disposition A: Accept As Requested

#### Proposal

Modify the VRDR linkage mechanism so that the maternal death record can
reference **multiple** related birth and/or fetal-death records. Concretely:

- In the VRDR profile that today carries the maternal-death → birth/
  fetal-death linkage (e.g., the linkage Observation / extension /
  RelatedPerson reference used by the `Decedent`/`DecedentPregnancy`
  family of profiles), change the cardinality of the link element from
  `0..1` to `0..*`.
- Update the IG narrative pages that describe maternal-death linkage to
  document multiple-birth handling, including guidance that each linked
  child (live birth or fetal death) is represented by its own reference
  and that all linked references share the same delivery event.
- Add an example bundle demonstrating a maternal death linked to a twin
  birth (one live birth + one fetal death, plus a both-live-births
  variant).
- Add an `invariant` or narrative note clarifying that when more than
  one record is linked, all linked records must describe the same
  delivery (same date, same mother).

#### Justification

This is the most direct way to address the reporter's concern and aligns
with how multiple births are modeled elsewhere in FHIR (e.g.,
`Patient.multipleBirth[x]`). Vital records jurisdictions already collect
per-child certificates for multiple births, so the underlying data exists;
only the FHIR linkage in VRDR is too narrow. Widening cardinality is
backward-compatible for producers who only ever populate a single link,
which limits downstream disruption. It also gives surveillance use cases
(maternal mortality review committees, CDC-PMSS) the fidelity they need
to count affected children correctly.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of widening the cardinality of the existing linkage element,
keep the link as `0..1` but add a structured **"multiple birth"
indicator** alongside it:

- Add (or reuse) an Observation/extension carrying `multipleBirth[x]`
  (`boolean` + `integer` for birth order / set size) on the maternal
  death linkage, mirroring `Patient.multipleBirth[x]`.
- Document that the single linked record is the "primary" sibling, with
  the indicator signalling that other siblings exist outside the bundle.
- Optionally allow a `RelatedPerson` chain (or a list of `Reference`s in
  a separate "additional siblings" extension) for jurisdictions that do
  carry the other certificates, so adoption can be incremental.

#### Justification

This reduces the surface area of the change: the existing linkage
element keeps its current cardinality and meaning, and the new indicator
is purely additive. Some jurisdictions only have one of the sibling
certificates available at the time of maternal-death reporting, so an
indicator-plus-optional-list pattern matches real-world data
availability better than a hard `0..*` requirement to enumerate every
sibling. It also keeps consistency with the FHIR core convention of
representing multiple-birth status as a property of the patient rather
than as a list of sibling references.

---

### Disposition C: Decline

#### Proposal

Do not change the VRDR STU3 linkage in this ballot cycle. Document in a
known-issues / future-considerations note that representing multiple
births and fetal deaths in maternal-death linkage is recognized as a gap
to be addressed in a future VRDR version, and ask the reporter to bring
concrete jurisdictional examples and a full data-flow analysis to a
future Public Health WG call.

#### Justification

The ticket is exploratory ("May be necessary to link to multiple
events…") and there are no attached use cases, jurisdictional data
samples, or downstream consumer requirements in the ticket, comments,
Zulip, or GitHub. Changing cardinality on a published STU element risks
breaking existing implementers and producing inconsistent multi-link
semantics across jurisdictions if not designed carefully. Deferring
allows the WG to gather requirements first and avoid a hasty model
change in STU3.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a concrete fidelity gap: VRDR cannot today
represent a maternal death whose pregnancy ended in a multiple birth or
fetal death, even though that is a clinically and statistically
important class of cases for maternal mortality review. Disposition A
addresses that gap directly with a minimal, backward-compatible model
change (cardinality `0..1` → `0..*` on the maternal-death linkage),
and it brings VRDR into line with how multiple births are handled
elsewhere in FHIR. Disposition B is attractive on additivity grounds but
loses information when more than one sibling certificate *is* available,
which undercuts the original use case. Disposition C is hard to defend
given that the gap is real and the fix is small; deferring would push a
known correctness problem into another ballot cycle. The work group
should, however, take the opportunity (as part of A) to nail down the
delivery-consistency invariant and add at least one twin/fetal-death
example bundle so implementers have unambiguous guidance.
