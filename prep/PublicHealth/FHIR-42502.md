## Summary

The ticket asks the BSeR (Bidirectional Services eReferrals) authors to
revisit the implicit assumption baked into the BSeR Task state machine that
every workflow "event" — for example, the referral recipient sending
scheduled-visit information or an interim consult note — necessarily
advances the `Task.status` (and, by implication, `Task.businessStatus`).
Lloyd McKenzie's comment (imported as Comment 53) argues that some of
these events are purely informational outputs attached to the Task and
should not be treated as state transitions. The ticket therefore proposes
simplifying the BSeR state model so that a defined subset of actions
(scheduling a visit, sending an outcome, posting interim notes, etc.)
adds an `output` to the in-flight Task without changing its status.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were

Task lifecycle.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR Task lifecycle narrative and the BSeR Task profile so
that the state machine explicitly distinguishes **state-changing
actions** from **informational events**. Concretely:

1. In the BSeR IG narrative pages that describe the Task state machine
   (the BSeR Task lifecycle / state-diagram pages), reclassify
   "schedule visit information" and "interim consult note" (and any
   other "send an output" interactions) as events that
   **append a `Task.output`** rather than transitions that set a new
   `Task.status` or `Task.businessStatus`.
2. In the BSeR Task profile, leave `Task.status` and
   `Task.businessStatus` cardinalities and value sets unchanged, but
   add invariants/guidance that these elements MUST NOT be advanced
   solely because an interim output was posted.
3. Update the state diagram artifact (PlantUML/SVG) so that the
   informational actions appear as self-loops on the existing state
   instead of arrows between states.
4. Add a short conformance section enumerating which BSeR interactions
   are state-changing and which are output-only, so implementers can
   build deterministic state engines.

#### Justification

This is the literal change Lloyd requested and aligns BSeR with the
broader FHIR Workflow pattern, where `Task.output` is explicitly the
mechanism for accumulating intermediate results without forcing a
lifecycle transition. The Workflow module (Task resource narrative)
already recognises that fulfillers can produce multiple outputs over
the life of a Task; treating "schedule visit" and "interim consult
note" as outputs preserves a clean, auditable history while keeping
`Task.status` semantically meaningful (only meaningful lifecycle
transitions move it). Because the cardinality and value sets of
`status` / `businessStatus` are not narrowed, the change is
backward-compatible with existing BSeR exchanges that may already be
sending these outputs without status changes.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a single state machine in the BSeR Task profile, but introduce a
**dedicated BSeR `businessStatus` value set** that distinguishes
"in-progress / awaiting visit", "in-progress / interim notes posted",
and "in-progress / scheduled" as **business sub-states** of the
existing `Task.status = in-progress`. Continue to allow those events
to update `Task.businessStatus` (so consumers get a meaningful "what
just happened" signal), but lock `Task.status` so it only advances on
the canonical workflow transitions (`requested` → `accepted` →
`in-progress` → `completed` / `failed` / `cancelled`). Document the
event-to-businessStatus mapping in the BSeR Task lifecycle page and
add an invariant on the profile that `Task.status` cannot regress and
is unaffected by output-only events.

#### Justification

This honours the underlying concern — that informational events
shouldn't be conflated with lifecycle transitions — while preserving a
machine-readable signal that "something happened" (which some
referring providers' UIs rely on to refresh their views).
`Task.businessStatus` exists precisely for implementation- or
domain-specific sub-states, and using it here is more in line with the
Workflow module's intent than overloading `Task.status`. The trade-off
versus Disposition A is added profile complexity (a new BSeR
`businessStatus` value set and an invariant) in exchange for clearer
implementer guidance about what to do when an output arrives.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that the existing BSeR Task profile
and underlying Task resource already permit `Task.output` entries
without requiring a status change, and the perceived problem is one of
narrative/diagram clarity rather than a normative defect. Mark this
ticket as "Persuasive with Modification — clarify documentation only"
or close as "Not Persuasive" if the workgroup is satisfied that the
current text does not in fact mandate status changes for these
events.

#### Justification

Nothing in the base Task resource forces a status transition when an
output is added; the BSeR profile, as currently written, inherits that
freedom. If the only issue is that the BSeR state diagram or narrative
*implies* status changes when none are required, a documentation-only
clarification (potentially handled under a different `BSeR_STATE`
ticket that is already open) may be sufficient and avoids re-opening
the BSeR state machine for this single ticket.

---

### Recommendation

**Recommended disposition:** A

The ticket is reported by Lloyd McKenzie at `Highest` priority and
carries the `BSeR_STATE` label, signalling that the BSeR state-machine
review is the right venue and time for this change. Disposition A
makes the smallest set of substantive edits that directly address the
reporter's concern, aligns BSeR with the Workflow module's intended
use of `Task.output`, and is backward-compatible because it neither
changes element cardinality nor narrows existing value sets. Disposition
B is attractive when implementers genuinely need a sub-state signal,
but introducing a BSeR-specific `businessStatus` value set in the same
ballot cycle as the broader state-machine cleanup risks scope creep and
should be raised as a separate ticket if the workgroup decides it is
needed. Disposition C understates the problem: the existing diagrams
and narrative *do* lead implementers to wire status changes to every
event, so a normative-text + diagram update (Disposition A) is the
defensible minimum.
