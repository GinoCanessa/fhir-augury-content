## Summary

The reporter notes that on the **Use Cases** page of the *US Central Cancer
Registry Reporting (CCRR) Content IG*, section **2.3.5 Patient Journey**
contains the narrative step *"3/31/24: The completion of radiation therapy
triggers a non-encounter-based incremental report to be generated and sent to
the central cancer registry. **(NEB.2)**"*, but the reference token
**(NEB.2)** either has no link or a broken link to its corresponding example
bundle. The reporter expects (NEB.2) to behave like (NEB.1), which links out
to a bundle example page (e.g.,
`Bundle-ccrr-non-encounter-based-content-bundle-example.html`).

A secondary observation is that even when the (NEB.1) link does resolve, the
target Example Bundle page does not visibly identify itself as "NEB.1", so a
reader following the link cannot easily confirm they have landed on the
correct example. The reporter recommends that the Example Bundle page
heading (currently approximately *"7.39.1 Example Bundle: Non Encounter
based CCRR Content Bundle - Example"*) be amended to include the NEB token
(e.g., *"Example Bundle (NEB.1): Non Encounter based CCRR Content Bundle -
Example"*) so the reader can correlate the use-case label with the bundle
they are viewing.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
linked to this issue in Augury's index, so the request is being reviewed in
isolation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Make two coordinated narrative/IG-publisher fixes in the CCRR Content IG
sources:

1. **Fix the missing/broken (NEB.2) link in `pages/use-cases.md` (or the
   equivalent Use Cases page source)**, section 2.3.5 *Patient Journey*. The
   "3/31/24" radiation-therapy bullet that ends in **(NEB.2)** must link to
   the NEB.2 example bundle artifact in the same way the (NEB.1) bullet
   currently links to
   `Bundle-ccrr-non-encounter-based-content-bundle-example.html`. If the
   NEB.2 example bundle exists under a different ID, point at that page; if
   it does not yet exist, author the missing bundle example (e.g.,
   `Bundle-ccrr-non-encounter-based-incremental-content-bundle-example`)
   and link to its generated `Bundle-…-example.html`.
2. **Update the Example Bundle page titles** (FSH `^title` /
   `^short` on the example Bundle resources, which drive the
   `7.39.x Example Bundle …` headings in the published IG) so each title
   carries its NEB token, e.g. *"Example Bundle (NEB.1): Non Encounter based
   CCRR Content Bundle - Example"* and the analogous *"(NEB.2)"* title for
   the incremental bundle. Re-run the IG publisher and verify the rendered
   `use-cases.html` anchors resolve and that the destination page heading
   now contains the NEB token.

#### Justification

This is exactly what the reporter asked for, and it is a pure technical
correction with no normative impact. Use-case identifiers (NEB.1, NEB.2,
…) are only useful if a reader can click through from the narrative to the
concrete example artifact and visually confirm correlation on landing. A
broken link in the Patient Journey directly undermines that traceability,
and surfacing the NEB token in the bundle's title is a low-cost
publisher-side change that improves readability without affecting any FHIR
conformance or terminology. The change is scoped to IG narrative and
example metadata, so the risk of regression elsewhere in the spec is
minimal.

---

### Disposition B: Alternative Approach

#### Proposal

Address the underlying traceability problem more systematically rather
than only patching the two reported instances:

1. Introduce (or formalize) a small **NEB ↔ artifact mapping table** at the
   top of the *Use Cases* page (or in a new "Use Case Index" appendix) that
   lists each NEB.x identifier, a one-line description, and a link to its
   canonical example Bundle. Every narrative reference to (NEB.x) in the IG
   then links into that table anchor.
2. Encode the NEB token on each example Bundle resource as a structured
   identifier (e.g., a `Bundle.identifier` with a CCRR-specific system, or a
   tag/extension carrying the use-case label) **and** echo it in the
   resource's `^title`, so that the relationship is machine-readable as
   well as visible in the rendered page heading.
3. Audit the Use Cases page for any other NEB.x / EB.x style tokens that
   are unlinked or mis-linked, and fix them in the same pass.

#### Justification

The reported defect is almost certainly not isolated: a page that has one
broken use-case-to-example link is likely to have more, and any future
example added to the IG will face the same authoring pitfall. Adding a
single source-of-truth mapping (and a structured identifier on each
example) means later narrative references just point at the table anchor,
which is cheaper to maintain and harder to break than ad-hoc inline links.
The trade-off vs. Disposition A is a slightly larger edit and a small
amount of new structure on the example bundles, in exchange for durable
traceability. This approach also benefits implementers who script against
the IG, because the use-case label becomes discoverable from the resource
itself rather than only from the rendered HTML.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive – Already Addressable Editorially Without an
Errata**: acknowledge the reporter's observation, but rather than treating
it as a tracked specification change, fold the link fix and title tweak
into the next routine IG publication cycle as an editorial pass and close
the ticket with the corresponding disposition.

#### Justification

The ticket is filed as *Technical Correction* with *Low* priority against an
IG (not the FHIR core spec), and its substance is two narrative authoring
nits: a missing hyperlink and a missing parenthetical token in a page
heading. Neither alters conformance, terminology, structure, or behavior
for any implementer. Workgroup ballot disposition machinery is heavyweight
relative to "fix the link, add a token to the title". If the IG team is
already planning a publication refresh, declining the ballot disposition
in favor of an editorial fix avoids unnecessary process while still
delivering the change. This is the weakest of the three options because it
risks the fix being forgotten if not formally tracked, but it is defensible
when authoring bandwidth is constrained.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right starting point: the reporter's request is
narrowly scoped, technically correct, and completely consistent with how
the (NEB.1) reference already works on the same page. Fixing the (NEB.2)
link restores the use-case → example traceability the page is clearly
designed to provide, and adding the NEB token to the example Bundle titles
is a one-line authoring change per resource that materially improves the
reader's confidence on landing. Disposition B is attractive but should be
treated as a follow-up improvement (and probably a separate ticket) rather
than a precondition for fixing this defect — the workgroup should not let
"build a use-case index" block "fix the broken link the reporter pointed
at". Disposition C should be avoided: even though the change is small,
tracking it through normal ballot disposition ensures the fix actually
ships with the next publication and gives the reporter a clear answer.
