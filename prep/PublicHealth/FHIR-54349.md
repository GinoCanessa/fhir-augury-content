## Summary

Craig Newman has raised a Change Request against the **Organism
Susceptibility** profile in the Electronic Case Reporting (eCR) FHIR IG,
asking the Public Health work group to decide whether two element
bindings/types on the underlying `Observation` profile should be
constrained further:

1. Should the value set bound to `Observation.interpretation` be
   constrained to codes that specifically represent **susceptibility
   interpretations** (e.g., Susceptible / Intermediate / Resistant / SDD
   / Non-susceptible), or are other interpretation value types still
   relevant in this context?
2. Should the data type of `Observation.value[x]` be constrained from
   the full `value[x]` choice down to a single **`CodeableConcept`**
   (i.e., `valueCodeableConcept` only)?

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced by the Augury store for this ticket, so the request stands on
its own merits and is essentially a scope/tightening decision for the
profile author.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten both elements on the Organism Susceptibility profile:

- Change the binding on `Observation.interpretation` from whatever
  broader `ObservationInterpretation`-style value set is currently
  bound to a **required (or at minimum extensible)** binding against a
  susceptibility-specific value set containing only
  Susceptible / Susceptible-Dose Dependent / Intermediate / Resistant /
  Non-susceptible (e.g., the LOINC/SNOMED susceptibility answer list or
  the equivalent HL7 susceptibility code set used elsewhere in
  microbiology IGs).
- Constrain `Observation.value[x]` to **`valueCodeableConcept`** only
  (slice/type constraint removing Quantity, Ratio, string, etc.) and
  bind the resulting CodeableConcept to the same susceptibility value
  set used on `.interpretation` (or a closely-aligned one).

#### Justification

Organism susceptibility results in eCR are, by definition, a categorical
judgement about whether an organism is susceptible to a given
antimicrobial. Constraining both `.interpretation` and `.value[x]` to a
susceptibility-specific coded answer removes ambiguity for public health
receivers, aligns the profile with how labs actually report
susceptibility in HL7 v2 and in other FHIR microbiology profiles, and
eliminates downstream mapping logic for jurisdictions receiving eCR
data. A tight binding is consistent with the "one profile, one
semantic" design principle FHIR IGs typically apply to laboratory
observations.

---

### Disposition B: Alternative Approach

#### Proposal

Tighten `.interpretation` but keep `.value[x]` flexible:

- Bind `Observation.interpretation` to a susceptibility-specific value
  set with **extensible** strength (so the canonical S/I/R/SDD codes
  are required when applicable but additional interpretation codes
  remain usable).
- Leave `Observation.value[x]` open to a **choice of
  `CodeableConcept` or `Quantity`** (with `Quantity` used for MIC /
  zone-diameter numeric results and `CodeableConcept` for the
  categorical call). Bind the `CodeableConcept` slice to the
  susceptibility value set.
- Add guidance (and optionally an invariant) stating that when both a
  numeric MIC/zone result and a categorical S/I/R call are available,
  the categorical call belongs in `.interpretation` and the numeric
  result in `.value[x]`.

#### Justification

Real-world AST results are not purely categorical: labs routinely
report a numeric MIC or zone-diameter alongside (or instead of) the
S/I/R interpretation, and losing that channel would force senders to
hide quantitative data in extensions or drop it entirely. This option
preserves the clinical meaning the ticket is trying to protect
(categorical susceptibility lives in `.interpretation` under a tight
binding) while remaining compatible with US labs that report
quantitative susceptibility values. It also stays aligned with how base
FHIR and most US-oriented lab IGs model AST.

---

### Disposition C: Decline

#### Proposal

Keep the Organism Susceptibility profile as-is:
`Observation.interpretation` retains its current (broader) binding and
`Observation.value[x]` remains an open choice. Reject both proposed
constraints.

#### Justification

The profile is part of an eCR IG whose primary job is to carry case-
report payloads from EHRs to public health agencies; those payloads
already reflect what the sending lab produced. Over-constraining
`.interpretation` risks rejecting valid production messages that use
slightly different but semantically compatible codes (e.g., local lab
codes, SNOMED vs. HL7 answer lists), and constraining `.value[x]` to
`CodeableConcept` would silently drop MIC/zone-diameter numeric data
that many jurisdictions actually want. Declining defers the question to
the upstream microbiology modelling work (base FHIR / US Core / Lab IG)
rather than hard-coding a narrower shape in an eCR-specific profile.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best starting point for the work group discussion.
It gives Craig what he is actually protecting — an unambiguous,
susceptibility-only semantic on `.interpretation` — without breaking
the common real-world pattern of reporting a numeric MIC or zone
diameter in `.value[x]`. Disposition A is cleaner on paper but is
likely to reject otherwise-valid AST results that carry a numeric MIC,
which is a frequent pattern in US lab data and would generate pushback
during ballot. Disposition C preserves flexibility but leaves the
profile's semantics no tighter than plain `Observation`, which
undercuts the reason to have an Organism Susceptibility profile in the
first place. Public Health should confirm with the Orders & Observations
and lab-IG communities which susceptibility value set to reuse, so the
binding on `.interpretation` (and the `CodeableConcept` slice of
`.value[x]`) stays aligned across IGs.
