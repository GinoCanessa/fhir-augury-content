## Summary

The ticket asks the BSeR work group to decide whether a Code System
defined within the US Bidirectional Services eReferrals (BSeR) IG should
be re-homed (canonicalised) under `terminology.hl7.org` (THO) instead of
remaining in the BSeR IG's own canonical namespace. The ticket is part of
a wider `BSeR_THO`-labelled tranche that triages BSeR-defined code
systems for migration to HL7's central terminology repository, in line
with HL7's general policy that reusable code systems should live in THO
rather than in individual IGs. The ticket as filed does not identify a
specific Code System URL or OID — only the policy question is captured —
so disposition will need to confirm which Code System(s) are in scope
before any spec change is enacted.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the BSeR-defined Code System(s) called out by this ticket from
the BSeR IG's canonical namespace to `http://terminology.hl7.org/...`
under HL7 Terminology (THO). Concretely:

1. Identify the specific BSeR Code System(s) the reporter intended (the
   `BSeR_THO`-labelled tranche enumerates them; if this ticket is the
   umbrella question, treat each sibling ticket as the per-system unit
   of work).
2. Author a THO submission for each system: assign the THO canonical
   URL, copy the concept content, retain the existing OID, and mark the
   THO version as the authoritative copy.
3. In the BSeR IG, replace the in-IG `CodeSystem` resource with a stub /
   pointer (or remove it entirely) and update every `ValueSet` that
   includes from the old canonical to include from the new THO
   canonical. Update narrative pages and any `system` literals in
   examples accordingly.
4. Add a deprecation / redirect note in the BSeR IG so existing
   implementations following the old canonical can find the THO copy.

#### Justification

This is the path HL7 has been steering IGs toward for several cycles:
reusable code systems belong in THO so that other specifications
(US Core, public health IGs, downstream realm IGs) can reference them
without taking a dependency on BSeR. Anchoring in THO also gives the
content a stable governance home (Vocabulary WG / THO release cycle)
that is independent of BSeR's own publication cadence, which matches
the spirit of the `BSeR_THO` triage tranche this ticket sits inside.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a per-Code-System decision rule instead of a blanket "move to
THO" answer. For each BSeR-defined Code System, evaluate against
explicit criteria — (a) is the code set conceivably reusable outside
BSeR's bidirectional-eReferral context, (b) is the content stable
enough for THO's release cadence, (c) is there an existing THO or
external system that already covers the concepts — and only migrate
those that pass (a)+(b) and fail (c). For systems that are genuinely
BSeR-specific workflow codes (e.g., referral status enumerations that
mean nothing outside BSeR), keep them anchored in the BSeR IG canonical
namespace and document the rationale in the IG's terminology page.

This disposition would also instruct the work group to capture the
decision per system on the corresponding `BSeR_THO`-labelled child
ticket so the triage record is preserved.

#### Justification

A blanket migration risks pushing BSeR-internal workflow vocabularies
into THO where they will look like reusable terminology to unrelated
implementers, polluting THO's namespace and creating governance
overhead for codes only BSeR consumes. The per-system rule preserves
the policy intent (reusable terminology lives in THO) while keeping
truly local enumerations local, which is consistent with how other IGs
(e.g., US Core, IPS) draw the line between THO content and IG-scoped
content.

---

### Disposition C: Decline

#### Proposal

Close FHIR-42184 as "Not Persuasive" / "Already Addressed" on the
grounds that (i) the ticket as filed does not identify a specific Code
System or describe a concrete defect in the published BSeR IG, and
(ii) HL7's general policy on THO-anchoring of reusable code systems is
already documented and is being applied through the per-system tickets
in the `BSeR_THO` tranche. Direct any reviewer who lands on this
ticket to those per-system tickets for the actionable decisions.

#### Justification

A change request that asks an open-ended policy question without
naming the artifact to be changed cannot be acted on as a spec
modification: there is nothing to publish, no errata to issue, and no
test to write. If the BSeR_THO tranche is the operational vehicle for
the migration, this umbrella ticket adds no incremental signal and
keeps a "Highest"-priority item in Triaged indefinitely, which
distorts the work group's backlog metrics.

---

### Recommendation

**Recommended disposition:** B

The underlying intent — get BSeR's reusable terminology into THO — is
sound and aligns with HL7-wide direction, so a flat decline (C)
under-delivers on the reporter's concern and will likely be re-raised.
A blanket accept (A) over-delivers: BSeR almost certainly contains
both genuinely reusable code systems and BSeR-internal workflow
enumerations, and treating them identically risks adding maintenance
weight to THO for content nobody outside BSeR will ever consume.
Disposition B keeps the policy goal but binds it to an explicit,
auditable per-system decision, which is also the disposition that
maps most cleanly onto the `BSeR_THO` triage tranche this ticket sits
in. As a procedural note for the work group: before adopting B,
confirm which specific Code System (or "all BSeR-defined systems")
the reporter had in mind so the per-system evaluations can be scoped.
