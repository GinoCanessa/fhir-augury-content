## Summary

The ticket questions whether a "Recipient Referral Identifier" should be
exchanged at all in the BSeR (US Bidirectional Services eReferrals) IG
when communication occurs over a RESTful FHIR interface. The reporter
(Lloyd McKenzie, comment 65 imported from a prior review cycle) argues
that in a RESTful environment, the recipient's referral identifier will
never be populated on the originating ServiceRequest — if such an
identifier exists at all, it would live on a separate "filler-order"
instance held by the recipient. The reporter therefore challenges both
the necessity and purpose of carrying the recipient's identifier on the
referring side. The recorded resolution is **Persuasive** with a plan to
update the Referral ServiceRequest profile to remove the
`identifier:referralRecipientServiceRequestIdentifier` slice and re-evaluate
whether a recipient identifier needs to be recorded on this profile at all.
No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR **Referral ServiceRequest** profile to remove the
`identifier:referralRecipientServiceRequestIdentifier` slice entirely
(consistent with the recorded resolution). State explicitly in the
profile narrative that, in a RESTful exchange, only the placer-side
referral identifier (the originator's `ServiceRequest.identifier`) is
meaningful on this instance; any identifier assigned by the recipient
("filler-order") belongs on a separate ServiceRequest instance held by
the recipient and is not exchanged back on the original instance. Re-spin
any examples and IGs that populate the removed slice.

#### Justification

In RESTful FHIR, each party owns its own resource instance. A recipient's
locally-assigned identifier for a referral is a property of the
recipient's own ServiceRequest, not of the requester's instance, so
populating it on the referrer's profile cannot happen at the time of
sending and is semantically out of place. Removing the slice avoids
encoding a placer/filler messaging-style pattern into a REST interaction
model and removes a slot that implementers cannot meaningfully fill. This
matches the resolution already recorded on the ticket (Persuasive) and
the reporter's stated concern.

---

### Disposition B: Alternative Approach

#### Proposal

Keep an identifier slot for the recipient's identifier on the Referral
ServiceRequest profile but reshape it: rename the slice (e.g.,
`fillerReferralIdentifier`) and constrain its cardinality to `0..1`
**MustSupport=false**, with a `definition` that explicitly says it is
populated only when a recipient system has acknowledged the referral and
chosen to echo back its locally-assigned identifier on a subsequent
update of the referrer's record (out-of-scope for the basic REST
exchange). Optionally model this as an extension on the original
ServiceRequest rather than as an identifier slice, so that base
ServiceRequest semantics around `identifier` (business identifiers
assigned by the placer) are not muddied.

#### Justification

Some real-world BSeR deployments use a hybrid pattern in which the
recipient's identifier is fed back to the referrer as part of a status
update (e.g., when a Task carries the assigned identifier). Preserving a
narrow, optional, clearly-scoped place to record that identifier — or
moving it into an extension — keeps that workflow expressible without
implying that it is part of the initial REST POST. The trade-off versus
Disposition A is more profile surface area (and potential confusion
about when to populate), but it avoids forcing implementers who already
exchange a recipient identifier today to migrate to an out-of-profile
extension.

---

### Disposition C: Decline

#### Proposal

Leave the Referral ServiceRequest profile unchanged: keep the
`identifier:referralRecipientServiceRequestIdentifier` slice as-is, with
its existing cardinality and documentation, and clarify in narrative
that the slice is optional and intended for environments that exchange a
recipient-assigned identifier back to the referrer.

#### Justification

The slice is already optional, so it imposes no burden on REST-only
implementers who simply omit it. Removing a published slice is a
breaking change for any implementer that has built against it, and the
ticket does not present concrete interoperability harm — only a question
about purpose. Declining preserves backward compatibility.

---

### Recommendation

**Recommended disposition:** A

The ticket's resolution is already recorded as **Persuasive** with an
explicit plan to remove the
`identifier:referralRecipientServiceRequestIdentifier` slice and
re-evaluate whether a recipient identifier belongs on this profile at
all. The reporter's argument is structurally correct for a RESTful
model: a recipient's identifier is a property of the recipient's own
ServiceRequest, not of the referrer's instance, so a slot on the
referrer's profile cannot be populated meaningfully at send time and
invites incorrect implementations. Disposition B is a reasonable
fallback if the workgroup confirms an existing implementer base that
echoes back the recipient identifier on the referrer's resource — in
that case the slot should be reshaped (renamed and/or moved to an
extension) rather than left in place, and the migration path called out
in the IG's change log. Disposition C preserves backward compatibility
but does not address the underlying semantic problem the reporter
raised, and is inconsistent with the existing resolution on the ticket.
