## Summary

Lloyd McKenzie is asking the BSeR (Bidirectional Services eReferrals)
implementation guide to clarify the "participants" page. The page
currently relies on a number of diagrams to convey the participants and
their interactions, but the conformance implications of those diagrams
are not made explicit. Reviewers cannot tell whether the depicted
information is mandatory content that must appear in BSeR messages,
whether the same expectations apply to RESTful queries, or whether
implementations are expected to support specific `_include` parameters
and search parameters to surface the related data. The ticket asks the
authors to rewrite the page so that the conformance expectations — for
both the messaging and query exchange paradigms supported by BSeR — are
unambiguous, and so that any required `_include` / search-parameter
support is enumerated. No cross-referenced Jira tickets, Zulip
discussions, or GitHub items were located for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rewrite the BSeR IG "participants" page (the narrative page that
currently leads with the participant diagrams) so that each diagram is
preceded by prose that names the actors, the interaction it depicts,
and — crucially — the conformance paradigm it belongs to (FHIR
Messaging vs. RESTful query). Concretely:

1. Add an "Exchange paradigms" subsection that states which
   participant interactions are realised through messaging (referencing
   the relevant `MessageDefinition` / `MessageHeader.event` codes) and
   which are realised through RESTful query against a server (with
   pointers to the BSeR `CapabilityStatement`).
2. For every interaction that appears in a diagram, add a one-line
   conformance statement of the form "the *Recipient* SHALL/​SHOULD
   return …" so the reader can see, per arrow, whether the depicted
   payload is required, recommended, or illustrative only.
3. For the query-based interactions, add an explicit table listing the
   search parameters and `_include` (and where relevant `_revinclude`)
   values that conformant servers are expected to support, cross-linked
   to the corresponding profile and `SearchParameter` definitions
   already in the IG.
4. Cross-link the page from the IG's main TOC entry for the
   participants/actor section so readers reach the clarified narrative
   first.

#### Justification

The request is editorial / narrative in nature and does not change the
underlying conformance model — it makes implicit expectations explicit.
That is consistent with HL7 IG style guidance, which expects diagrams
to be backed by prose conformance statements rather than standing on
their own. Listing required `_include` and search parameters next to
the diagrams also matches the pattern used by other HL7 US-realm IGs
(e.g., US Core, Da Vinci) and reduces the risk of implementers guessing
at server expectations from pictures alone.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only rewriting the existing "participants" page, restructure
the IG so that the conformance content lives where implementers already
look for it, and the participants page becomes a pure orientation
page:

1. Keep the participants page diagrams, but reduce their accompanying
   text to a short orientation paragraph and add a banner that says
   "This page is informative; normative conformance for each interaction
   is defined on the linked pages below."
2. Move the per-interaction conformance statements (mandatory message
   contents, search parameters, `_include` values) onto the
   `CapabilityStatement` page and the per-message `MessageDefinition`
   pages, where they will appear next to the machine-readable artifact
   they constrain.
3. From each diagram, link the depicted interaction directly to the
   relevant `MessageDefinition` page (for messaging interactions) or
   the relevant section of the `CapabilityStatement` (for RESTful
   interactions).
4. Add a single "Search & _include expectations" table to the
   `CapabilityStatement` page enumerating the supported search
   parameters and includes per resource.

#### Justification

This addresses the same underlying need — implementers should not have
to infer conformance from diagrams — but locates the normative content
on the artifacts that actually define it. Mixing normative conformance
prose into a narrative page can drift out of sync with the machine
artifacts; co-locating it with the `CapabilityStatement` and
`MessageDefinition` pages keeps a single source of truth. The trade-off
is that readers who land on the participants page first must follow
links to find conformance details, which is slightly less convenient
than Disposition A but more maintainable over the IG's lifetime.

---

### Disposition C: Decline

#### Proposal

Decline the request on the basis that the conformance for BSeR is
already fully specified by the IG's `CapabilityStatement`,
`MessageDefinition`, profile, and `SearchParameter` artifacts, and the
participants page is explicitly an informative diagram-led overview.
Mark the ticket "Not Persuasive — informative page", with a note
inviting a follow-up ticket if there are specific conformance questions
the existing normative artifacts fail to answer.

#### Justification

In FHIR IGs, conformance is normatively expressed in the structured
artifacts (CapabilityStatements, profiles, MessageDefinitions, search
parameters), not in narrative pages. If the participants page is
labelled informative, asking it to carry conformance statements
duplicates content that lives elsewhere and risks the two going out of
sync. Declining preserves the separation between informative narrative
and normative conformance.

---

### Recommendation

**Recommended disposition:** A

The ticket is filed at the highest priority by a senior FHIR reviewer
and points at a real usability problem: the participants page is the
first place implementers look to understand who does what in BSeR, and
they cannot currently tell from that page whether a given arrow is a
hard requirement or an illustration. Disposition C ignores that
documentation problem and would likely attract the same comment again
in the next ballot cycle. Disposition B is architecturally cleaner but
requires more substantial restructuring across multiple pages and
artifact narratives, which is hard to land within a single ballot
cycle. Disposition A directly addresses the reviewer's concern with a
bounded, editorial change to one page (plus a small enumeration of
`_include` / search-parameter expectations), is consistent with how
other HL7 US-realm IGs document participant interactions, and leaves
the door open to migrate the conformance text onto the
`CapabilityStatement` / `MessageDefinition` pages later if the
workgroup decides to pursue Disposition B as a follow-up.
