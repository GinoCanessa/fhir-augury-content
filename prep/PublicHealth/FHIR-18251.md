## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the BSeR Patient profile so that its `baseDefinition` is the
current US Core Patient profile
(`http://hl7.org/fhir/us/core/StructureDefinition/us-core-patient`)
instead of the base FHIR `Patient` resource. Add a hard IG dependency
on the US Core version that BSeR is being aligned with (per the
Public Health work group's current US Core target â€” e.g., US Core
6.x or whichever version BSeR is presently rebased onto). Remove
from the BSeR Patient profile any constraints that are already
mandated by US Core Patient (e.g., `identifier`, `name`, `gender`,
`birthDate`, race / ethnicity / birth sex extensions) so the BSeR
profile only adds genuinely BSeR-specific differentials. Update the
IG narrative, the profile's intro page, and any examples / Task.for
references that previously bound to the BSeR Patient profile.
References inside other BSeR profiles (Task.for, ServiceRequest.subject,
etc.) should be reviewed and, where they are not already pointing at
US Core Patient directly, updated so that the BSeR Patient profile
remains a meaningful, narrower target than US Core Patient.

#### Justification

BSeR is explicitly US-realm; deriving from US Core Patient is the
established FHIR pattern for US-realm IGs and gives implementers a
single, consistent Patient profile across US Core, USCDI-driven IGs,
and BSeR. The 2024 BSeR Zulip discussion already confirms that
`Task.for` is constrained to US Core Patient, so the underlying
direction is already accepted; aligning the Patient profile itself
removes an inconsistency where the referral subject is US Core but
the named Patient profile is not. It also reduces duplication of
must-support and slicing requirements that US Core already enforces.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** rebase the BSeR Patient profile on US Core Patient.
Instead, leave the BSeR Patient profile derived from base
`Patient`, but require by IG-level conformance text and by reference
constraints elsewhere in the IG (Task.for, ServiceRequest.subject,
Encounter.subject, etc.) that any Patient instance exchanged in a
BSeR transaction `conformsTo` the US Core Patient profile in
addition to the BSeR Patient profile. Document the dual-conformance
expectation prominently in the IG's "Conformance" page and in each
profile's intro, and add `targetProfile` slices on the relevant
references so validators flag instances that are not US Core
Patient.

#### Justification

This avoids the maintenance and version-coupling burden of pinning
BSeR's Patient profile to a specific US Core release (US Core
versions evolve faster than BSeR can reasonably re-ballot) while
still guaranteeing implementers must produce US Core-conformant
Patients in BSeR exchanges. It mirrors the pattern several other
US-realm IGs have used when they wanted to remain technically
loose-coupled from US Core (e.g., for testing, or to support pre-
US-Core-adoption deployments) but operationally aligned with it.
It also avoids breaking existing BSeR implementers whose Patient
instances were validated only against the base-Patient-derived BSeR
profile.

---

### Disposition C: Decline

#### Proposal

Decline the request as overtaken by events. Note in the resolution
that BSeR has, in subsequent revisions, addressed the spirit of the
request by constraining the relevant references (most importantly
`Task.for`) to US Core Patient (per the BSeR Zulip thread on Core
Referral Architecture Profiles, Feb 2024). Recommend the work group
either close as Persuasive With Modification (pointing at the
existing constraints) or as Not Persuasive on the narrow question
of changing the Patient profile's `baseDefinition`.

#### Justification

The ticket is from 2018 and has had no activity since 2019; the
specification it targets has materially evolved, and the
practical concern raised â€” that BSeR exchanges should use US Core
Patient â€” has been addressed at the reference / Task.for layer
without restructuring the Patient profile. Reopening this as a
profile-rebase exercise now would impose churn on current BSeR
implementers without a corresponding new clinical or
interoperability requirement that the existing US Core constraint
on `Task.for` does not already meet.

---

### Recommendation

**Recommended disposition:** A

The request is small, well-aligned with standard US-realm FHIR
practice, and only partially addressed by the current BSeR design.
Today, BSeR constrains `Task.for` to US Core Patient but its own
Patient profile is still based on the base resource, which leaves a
visible inconsistency: a referral's subject must conform to US Core
Patient, yet the IG's named Patient profile does not. Rebasing the
BSeR Patient profile on US Core Patient (Disposition A) closes that
gap, removes duplicated must-support / slicing rules, and matches
how every other current US-realm IG handles Patient. The main risk
â€” version coupling to a specific US Core release â€” is real but
manageable through the IG's normal `package.json` dependency, and
is the same trade-off every other Public-Health-WG IG (e.g., the
ones in the Public Health space already depending on US Core) has
already accepted. Disposition B is reasonable as a fallback if the
work group wants to avoid bumping the IG's US Core dependency
right now, and Disposition C is appropriate only if the work group
explicitly decides the existing Task.for constraint is sufficient
and does not want to touch the Patient profile.
