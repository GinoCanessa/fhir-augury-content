## Summary

The ticket asks the Central Cancer Registry Reporting (CCRR) IG to refresh
the reporting value sets defined under Use Cases §2.3.7 ("Central Cancer
Registry Reporting ValueSets") so that they align with the most recent
authoritative reportability lists they are derived from — specifically the
CDC's 2026 ICD-10-CM Casefinding List for Cancer (FY26 spreadsheet
published by the National Program of Cancer Registries). In addition to
the version bump itself, the reporter (filed on behalf of a NAACCR
community member) flags that the current value sets contain discrepancies
between their human-readable descriptions and the inclusion/exclusion
criteria they declare, and asks that those be reviewed with subject matter
experts (SMEs) and corrected as part of the same update. There are no
cross-referenced Jira tickets, GitHub items, or substantive Zulip
discussions associated with this ticket.

## Details

**Description:**

Use Cases page, section 2.3.7, Central Cancer Registry Reporting ValueSets.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Refresh the value sets enumerated in the CCRR IG Use Cases page
§2.3.7 ("Central Cancer Registry Reporting ValueSets") so their
ICD-10-CM content matches the **CDC FY26 ICD-10-CM Casefinding List for
Cancer** (linked in the ticket). Concretely:

1. Re-derive the codes in each affected reporting ValueSet from the FY26
   casefinding spreadsheet (replace prior FY-year codings; add new FY26
   inclusions; remove codes that FY26 no longer designates as
   reportable).
2. Bump the ValueSet `version` element on each affected resource and
   record the FY26 source list (URL + retrieval date) in the ValueSet
   `description` and/or `copyright` so downstream implementers can trace
   provenance.
3. Convene an SME review (NAACCR / registry SMEs) of each affected
   ValueSet's `description`, `purpose`, and the narrative
   inclusion/exclusion criteria, and reconcile any discrepancies between
   that prose and the codes the ValueSet actually selects (intensional
   filters or extensional concept lists).
4. Regenerate the IG so the Use Cases §2.3.7 narrative, the ValueSet
   pages, and any expansions ship together in the next publication of
   the CCRR IG.

#### Justification

This is the literal request, and it is well-grounded: the casefinding list
is updated annually by CDC NPCR, and a public-health reporting IG whose
value sets lag the current reportability list will produce
under-reporting (missed FY26 additions) or over-reporting (codes CDC has
removed). Aligning ValueSet descriptions with their inclusion/exclusion
criteria is also basic terminology hygiene — discrepancies between the
prose and the actual code selection are a known source of implementer
confusion and audit findings, and SME reconciliation is the conventional
remedy. The request comes via a NAACCR community member, which is the
authoritative stakeholder community for cancer registry casefinding.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the FY26 update **and** institutionalize the maintenance pattern
rather than treating this as a one-off ticket. Specifically:

1. Do the FY26 refresh as in Disposition A (codes, descriptions,
   inclusion/exclusion criteria, regeneration).
2. In each affected ValueSet, replace inline ICD-10-CM enumerations with
   an **intensional** definition that filters CDC's published ICD-10-CM
   Casefinding code system / list by a stable property (e.g., a
   "reportable / casefinding inclusion" flag) wherever the source list's
   structure permits.
3. Add a small "Reportability list versioning" subsection under Use Cases
   §2.3.7 that documents (a) the policy that the IG tracks the most
   recent published FY casefinding list, (b) how the version is recorded
   on each ValueSet, and (c) the expected cadence of refresh tickets.
4. File a follow-up Jira ticket / standing work item for FY27 so the
   refresh is calendared rather than relying on community reports.

#### Justification

A literal one-time refresh (Disposition A) leaves the IG in exactly the
same drift-prone state next October when CDC publishes FY27. An
intensional binding (where feasible) and a documented refresh policy
shift the maintenance burden from "edit each ValueSet by hand" to "point
the binding at the new list and re-expand", which is the pattern used by
other terminology-driven Public Health IGs. The trade-off vs.
Disposition A is up-front effort: intensional filters require that
CDC's published artifact be modeled as a code system the IG can filter
on, which may not be true of every casefinding sub-list — so some
ValueSets may still need extensional content. That trade-off is worth
calling out in the disposition discussion.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as written, on the basis that the underlying request
(adopt the next FY casefinding list) is already implicit in the IG's
maintenance lifecycle and does not require a substantive design change —
the value sets will be refreshed as part of the IG's normal publication
cycle, and the description/inclusion-criteria reconciliation can be
handled editorially without a normative change request.

#### Justification

Defensible only if the workgroup judges that (a) the IG already commits
in a publication-process document to tracking the latest CDC casefinding
list, and (b) the noted prose-vs-criteria discrepancies are minor enough
to be handled by editorial cleanup. Even then, declining loses the
explicit SME-review step the reporter is asking for, and loses the
audit trail (a closed Jira ticket) that NAACCR can point to as evidence
the FY26 update was considered. This option is included for
completeness; it is not the recommended path.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The FY26 update is clearly required —
declining (C) is hard to justify against an external stakeholder
community that has explicitly flagged the gap, and accepting as written
(A) solves only this year's problem. Disposition B does everything A
does (so the immediate FY26 ask is satisfied) and additionally
documents a refresh policy and, where the source list shape allows,
moves to an intensional binding so the next refresh is mechanical
rather than another community-reported ticket. The main caveat the
workgroup should weigh is feasibility of the intensional binding for
each affected ValueSet; that should be confirmed during the SME review
step, and any ValueSets that cannot be expressed intensionally should
fall back to a documented extensional refresh under the same policy.
