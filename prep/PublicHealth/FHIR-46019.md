## Summary

The ticket challenges a constraint in the SANER (Situation Awareness for
Novel Epidemic Response) IG that fixes `MeasureReport.subject` to a
reference whose only populated component is `identifier`, rather than
allowing a direct `Reference` to a `Location` resource. The reporter
(Rick Geimer) argues that an actual `Location` resource — either
`contained` in the `MeasureReport` or transmitted alongside it in a
submission `Bundle` — can carry an identifier when needed *and* convey
additional information the recipient may require (address, position,
etc.). They consider the identifier-only restriction "heavy handed".

No other Jira tickets are cross-referenced, but the single cross-reference
to the FHIR artifact `MeasureReport.subject` confirms the subject of the
change request. Two long-running discussions on the `Covid-19 Response`
Zulip stream from 2020 already raised the same concern: that fixing
`MeasureReport.subject` to `identifier` is awkward, especially when an
implementation also needs to satisfy the SANER `ResourceLocation` profile
or to filter `MeasureReport` by `subject` reference.

## Details

**Description:**

What is the reason for requiring an identifier reference here instead of
a direct reference to a Location resource? The Location resource could
be contained in the MeasureReport itself, or in a submission Bundle. An
actual Location resource can contain the identifier if required (maybe
profile Location to require it) and also provide additional information
needed by some recipients (address, position, etc). The identifier
restriction seems heavy handed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the SANER `MeasureReport` profile so that `MeasureReport.subject`
is not constrained to identifier-only. Specifically: remove (or weaken)
the constraint that fixes `subject.identifier` while leaving
`subject.reference` unconstrained, so implementers can choose either:

- a literal `Reference` to a `Location` (or `Organization`) — typically
  to a profile derived from US Core Location / SANER `ResourceLocation`,
  optionally supplied as a `contained` resource or in the same
  submission `Bundle` — or
- a logical reference using `subject.identifier`, retained for
  back-compatibility with existing senders.

If `Location` profiling is the underlying concern, profile `Location` to
*require* `Location.identifier` so that consumers who only care about
the identifier can still extract it from the referenced resource.

#### Justification

This matches the literal request and the implementer feedback in both
Zulip threads. FHIR's general guidance is that `Reference.identifier`
is meant for cases where a literal reference cannot be resolved;
mandating it where a real `Location` is available inverts the usual
pattern and prevents clients (e.g., Nandina) from offering basic UX such
as "filter MeasureReports by Location". Allowing both forms preserves
existing senders while removing a friction point that has been raised
publicly more than once.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `MeasureReport.subject` open to a literal `Reference`, but instead
of fully unconstraining the element, define two SANER-specific patterns
and document them on the profile:

1. **Identifier-only** (current behaviour) — for senders that have only
   a logical identifier for the reporting site and no `Location`
   resource available. Slice `subject` so that this slice requires
   `subject.identifier` and forbids `subject.reference`.
2. **Resolvable reference** — for senders that can supply a `Location`.
   Slice `subject` so that this slice requires `subject.reference` to
   resolve to a SANER `ResourceLocation`-conformant `Location` (which
   itself requires `Location.identifier`). Allow the `Location` to be
   supplied either as a `contained` resource on the `MeasureReport` or
   as a `Bundle` entry in the same submission.

Document both patterns in the IG with examples and a CapabilityStatement
expectation that recipients SHALL accept either form.

#### Justification

This addresses Reece Adamson's point that the SANER `ResourceLocation`
profile is built on US Core Location and that simply unconstraining
`subject` shifts work elsewhere. It also addresses Josh Mandel's
concern that CDC/FEMA reporting (which sometimes only has a logical
identifier such as a CCN) must keep working. By formalising both
patterns rather than silently allowing them, recipients know exactly
what to implement, and the trade-off between "lightweight identifier"
and "rich `Location`" becomes a sender choice rather than an IG
restriction.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep `MeasureReport.subject` constrained to
`identifier`-only. Document the rationale in the IG: SANER's primary
exchange pattern is high-volume, low-payload aggregate reporting (e.g.,
to CDC/FEMA), where the recipient already has authoritative `Location`
data keyed by a known identifier (CCN, NPI, etc.) and does not want
senders embedding or referencing potentially divergent `Location`
resources.

#### Justification

A defensible argument exists that SANER deliberately chose an
identifier-only binding to keep the wire format minimal and to prevent
inconsistent `Location` payloads from competing with the recipient's
canonical site directory. If implementers want richer `Location` data,
they can fetch it out of band using the identifier. However, this
position is in tension with both the original requester and multiple
voices in the Zulip discussion, and would leave the connectathon-tested
client behaviour (subject-as-Reference) unsupported.

---

### Recommendation

**Recommended disposition:** B

Disposition B preserves the legitimate use case the current restriction
was likely designed for — terse identifier-only reporting against
recipient-curated site directories — while granting the
`Reference`-based pattern that implementers have repeatedly demonstrated
they need at connectathon. Pure Disposition A is also reasonable, but
slicing the element gives recipients a clearer contract than "anything
goes", and it explicitly accommodates the SANER `ResourceLocation`
profile (and its US Core Location heritage) without forcing all senders
through it. Disposition C contradicts the implementer evidence on
record and should only be selected if the work group can produce a
specific recipient requirement that an identifier-only `subject` is
mandatory.
