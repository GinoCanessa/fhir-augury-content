## Summary

Saul Kravitz (Public Health WG) reports that a ValueSet in the US Birth
Defect Reporting (BDR) FHIR IG is misnamed. Although its current name
(per the ticket text) refers to "Histological Placental Exams", the
ValueSet's actual content only enumerates whether a procedure was
*planned* or *performed* — i.e., it is a procedure-status concept and
has nothing to do with histology or placental examination. The
submitter requests that the ValueSet be renamed to
`ProcedurePlannedOrPerformed` (or an equivalently-scoped name) so that
the artifact's name reflects its semantics. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were found in FhirAugury for
this ticket, so the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rename the affected ValueSet in the US Birth Defect Reporting (BDR)
FHIR IG so that its `name`, `title`, `id`, and canonical URL all
reflect its actual content (procedure planned vs. performed) rather
than histological placental examination. Concretely:

- Change the ValueSet `name` to `ProcedurePlannedOrPerformed` (and
  set `title` to a human-readable equivalent such as "Procedure
  Planned Or Performed").
- Update the ValueSet `id` and the trailing canonical-URL segment to
  match (e.g., `…/ValueSet/procedure-planned-or-performed`), keeping
  the same composition / included codes.
- Update every reference to the old ValueSet — bindings on Procedure-
  derived profiles, narrative, IG pages, examples, and any
  ImplementationGuide resource entries — to point at the new
  canonical URL.
- Add a deprecation/redirect note (or a stub ValueSet under the old
  URL marked `status = retired` with `replacedBy` pointing to the
  new canonical) so that existing implementers can find the
  replacement.

Because the BDR IG is not yet normative and the ticket is `Triaged`
(not yet resolved), this is a low-risk renaming change handled
through a regenerate-and-publish cycle of the IG.

#### Justification

The submitter's observation is straightforward and verifiable from
the ValueSet's content: a binary planned/performed concept does not
belong under a name that implies histology or placental examination.
HL7 IG naming conventions ask that a ValueSet's `name` describe its
contents, and reusing the suggested name `ProcedurePlannedOrPerformed`
is consistent with how the broader FHIR ecosystem names
procedure-status enumerations. Fixing the name now — before the IG
matures further or downstream consumers depend on the canonical URL —
minimises future churn.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than renaming the BDR-local ValueSet, evaluate whether an
existing standard ValueSet already covers "procedure planned or
performed" and bind to that instead. Specifically:

- Check whether the relevant Procedure profile in BDR can bind to
  `Procedure.status` (or a small, IG-local subset of it) to express
  planned vs. performed, eliminating the bespoke ValueSet entirely.
- If a bespoke ValueSet must remain, define it once in a more
  general HL7 location (e.g., a Public-Health-wide terminology
  module) under the name `ProcedurePlannedOrPerformed` and have BDR
  reference that ValueSet by canonical URL.
- Retire the misnamed BDR ValueSet with `status = retired` and a
  `replacedBy` pointer to the chosen replacement.

#### Justification

The underlying need — distinguishing planned from performed
procedures — is not BDR-specific. Reusing `Procedure.status` (or a
shared Public-Health ValueSet) avoids duplicating concepts across
IGs, prevents the same naming problem from recurring elsewhere, and
gives implementers a single canonical answer. The trade-off versus
Disposition A is a wider scope (touching shared terminology and
possibly other Public Health IGs) in exchange for a more durable
fix; if the WG considers the rename a one-off cleanup, Disposition A
is lighter.

---

### Disposition C: Decline

#### Proposal

Decline the rename and instead address the mismatch by clarifying
the ValueSet's `description` and `title` — keeping the existing
`name`, `id`, and canonical URL — on the grounds that the canonical
URL is part of the IG's published interface and should not be
rewritten purely for naming aesthetics.

#### Justification

Renaming changes the canonical URL, which can break existing
implementations, validators, and downstream IGs that already
reference the artifact. If no implementers are known to be confused
by the existing name in practice, a documentation-only fix (clearer
`title`/`description` plus a `useContext` note) preserves backward
compatibility at the cost of leaving a misleading identifier in
place. This option is most defensible if the BDR IG already has
published versions in the wild.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly addresses what the submitter and the artifact
itself require: a ValueSet whose name actually describes its
content. The BDR IG is still maturing (the ticket has sat in
`Triaged` since 2024 with no resolution and no active
counter-proposals on Zulip), so the cost of renaming the canonical
URL is low — there is no evidence in the surveyed FhirAugury data of
external implementers depending on the misnamed URL. Disposition B
is attractive in the abstract but expands the scope from a BDR fix
to a cross-IG terminology decision and would likely stall the
ticket; it can be pursued as a follow-up once the immediate naming
defect is corrected. Disposition C preserves a known-misleading name
purely for hypothetical backward compatibility and would leave
future readers of the IG with the same confusion the submitter has
already flagged.
