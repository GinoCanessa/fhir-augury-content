## Summary

The ticket (originally logged as ODH issue #19 against the US Occupational
Data for Health FHIR IG) asks whether the `RetirementDateModel` logical
model should accommodate a **planned / future retirement date** in
addition to a past or current retirement date. The reporter notes that
retirement frequently triggers changes in income and healthcare coverage,
so capturing a patient's anticipated retirement date could let care teams
and public-health consumers of ODH data identify patients who are
preparing to adapt. The item was previously resolved by the Public
Health Work Group on 2023-01-16 as "Considered for Future Use"
(20‑0‑6, Newman / Luensman) but was reopened on 2023-03-22 by Lynn
Laakso to be revisited post ODH 1.3 publication. No related Jira
tickets, Zulip threads, or GitHub items surfaced in the cross-reference
index — the only link is back to the PHWG Confluence minutes page that
documents the 2023‑01 vote.

## Proposed Dispositions

Note: The ODH FHIR IG source was not available as a cached GitHub repo
at prep time, so the specific authoring path for `RetirementDateModel`
could not be cited from a briefing. Dispositions below refer to the
logical model conceptually; when implementing, confirm the exact file
path in the ODH FHIR IG source repo.

### Disposition A: Accept As Requested

#### Proposal

Extend the `RetirementDateModel` logical model in the US Occupational
Data for Health (FHIR) IG to allow the retirement date to be a **future
(planned / anticipated) date** in addition to a past or current
retirement date. Concretely:

- Keep the existing `retirementDate` (or equivalently named) element as
  a `dateTime`/`date` with cardinality `0..1`.
- Add a companion flag element such as `retirementDateStatus` bound to
  a small value set (e.g., `actual` | `planned` | `anticipated`) so
  consumers can distinguish a recorded historical retirement from a
  self-reported planned retirement.
- Update the element definition/comments to explicitly state that the
  date may be in the future when `retirementDateStatus = planned`.
- Add an example instance illustrating a future-dated planned
  retirement, and a note in the narrative that implementers SHOULD
  refresh planned dates as the event approaches or is realized.

#### Justification

The reporter's clinical rationale is sound: retirement typically
triggers insurance (commercial → Medicare), income, and access-to-care
changes, and identifying patients preparing for that transition has
clear public-health and care-coordination value. FHIR logical models
routinely accept future dates (e.g., `Appointment.start`,
`CarePlan.period`), so allowing a future value is not novel. Adding a
small status/qualifier element — rather than repurposing the date
alone — preserves the ability for downstream analytics to separate
historical employment data from prospective planning data, which is
important for public-health reporting where "retired" is usually an
observed status. The 2023‑01 PHWG vote (20‑0‑6) showed strong
interest in revisiting the concept, and Lynn Laakso's 2023‑03
reopening explicitly set the expectation that it be considered
post‑1.3.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** modify the `RetirementDateModel` data shape. Instead,
address the underlying need by:

1. Clarifying in the ODH narrative that `RetirementDateModel` captures
   a single retirement event date and that **planned / anticipated**
   retirement should be represented using more general FHIR resources
   outside the ODH logical models — for example a `Goal` with
   `target.dueDate` set to the planned retirement date, or an
   `Observation` with a LOINC-coded "anticipated retirement date" and
   `effectiveDateTime` in the future.
2. Optionally add a non-normative example to the IG showing how an
   implementer can represent planned retirement using these existing
   mechanisms, and cross-linking from the `RetirementDateModel` page.

#### Justification

`RetirementDateModel` is a **logical model** in an IG that is
primarily focused on *reporting employment and occupational history*
for public-health use. Mixing an anticipated, patient-reported future
date into the same element raises semantic and data-quality risks —
analytic consumers would need to filter out planned values every time
they summarize retirement prevalence, and systems currently populating
the field from HR or employment records would have no source for a
"planned" value. Using `Goal`/`Observation` for prospective planning
keeps the ODH model crisp (observed occupational history) while still
giving implementers a standards-based path for the reporter's use
case, and avoids a breaking change to any system already consuming the
existing element. This aligns with the FHIR convention of putting
forward-looking patient intent in resources designed for that purpose
(`Goal`, `CarePlan`, prospective `Observation`) rather than in
historical fact models.

---

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope for the ODH FHIR IG**, and close
as "Not Persuasive / Considered for Future Use". Rationale: ODH is
scoped to exchanging observed occupational history for public-health
and clinical use, not patient-reported prospective life-event planning;
future retirement intent is better modeled by other HL7 artifacts
(care planning, social determinants screenings), and no use cases,
implementer requests, or connectathon tracks have accumulated against
this item in the ~7 years it has been open.

#### Justification

The original 2023‑01 PHWG vote (20‑0‑6) explicitly noted "This is
not part of the considerations for the current model", which is a
strong signal from the work group that planned retirement sits outside
ODH's scope. The reopen in 2023‑03 was procedural (tied to post‑1.3
publication) rather than driven by new implementer demand, and the
cross-reference index shows **no** linked Jira tickets, Zulip threads,
or GitHub issues/PRs accumulating around this concept — i.e. no
evidence of active implementer pull. Adding a planned-retirement
concept would also expand ODH's scope into patient-reported
prospective data, which has significant modeling, provenance, and
conformance implications (who sources it, how often it is refreshed,
how it interacts with later observed retirement). Declining keeps ODH
focused and leaves the prospective-planning use case to
better-suited mechanisms.

---

### Recommendation

**Recommended disposition:** B

Disposition B best matches both the clinical motivation in the ticket
and the scope discipline the PHWG already expressed in the 2023‑01
vote. The reporter's concern — that clinicians and public-health
consumers benefit from knowing a patient is preparing to retire — is
real and worth addressing, so a flat decline (C) undersells the
request. But pushing a prospective, patient-reported concept into
`RetirementDateModel` (A) risks polluting a model that is otherwise a
clean record of observed occupational history and creates a
data-quality burden for every downstream consumer. Using existing
FHIR mechanisms (`Goal.target.dueDate`, LOINC-coded prospective
`Observation`) lets implementers satisfy the use case today, keeps
ODH's logical models tight, and avoids a breaking change; the IG can
help by adding a short narrative note and example pointing to those
patterns. If the work group subsequently sees sustained implementer
demand for a first-class "planned retirement" element inside ODH,
Disposition A remains available as a follow-up in a later ballot
cycle.
