## Summary

The reporter (Craig Newman) observed that, in the BSeR (US Bidirectional Services eReferrals) IG — specifically the 2018Sep ballot publication — every link to the "Common Transaction Profiles" page returns a 404 File Not Found error. A follow-up comment enumerates three specific pages that are also broken within the same publication:
`ReferralRequestTransactionProfiles.html`, `ReferralFeedbackTransactionProfiles.html`, and `ImplementationProfileBundles.html`. The ticket is an IG publication/build defect rather than a modeling change: the authored content references page names that were either not generated by the publisher or were generated under different names/paths, leaving a set of dead internal links in the ballot output. No related Jira tickets, Zulip threads, or GitHub items were surfaced via cross-references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Fix the broken links in the BSeR IG so that all references to the Common Transaction Profiles resolve successfully. Concretely, in the BSeR source (the narrative/markdown pages that link to `ReferralRequestTransactionProfiles.html`, `ReferralFeedbackTransactionProfiles.html`, and `ImplementationProfileBundles.html`): (1) identify the actual generated page names / anchors produced by the IG publisher for the corresponding transaction-profile content, (2) update the authoring links to point at those pages, and (3) rebuild and republish the BSeR IG so the fixes flow through. Where a "Common Transaction Profiles" landing page was intended but never authored, author it and add it to the IG's table of contents / toc.xml so the links have a valid target.

#### Justification

The reporter has demonstrated concrete 404s at three canonical pages under the `2018Sep` ballot output, which indicates an authoring or publication defect rather than a modeling issue. Broken internal navigation directly harms implementer usability of the IG and is a routine, low-risk editorial fix. Fixing the links (or authoring the missing page and wiring it into the toc) is consistent with standard IG publisher hygiene and does not require any change to profiles, value sets, or transactions. Because no GitHub cross-references or cached repo briefings for BSeR are available in this workspace, the exact source path cannot be named here; the change would occur in the BSeR IG's authored input pages / toc.xml under its IG publisher source (HL7 `bser` IG repository).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than patching a retired ballot snapshot (`2018Sep`), redirect the fix forward: leave the 2018Sep ballot output as-is (it is a historical snapshot), but (1) ensure the *current* BSeR publication has working Common Transaction Profiles pages (`ReferralRequestTransactionProfiles`, `ReferralFeedbackTransactionProfiles`, `ImplementationProfileBundles`), and (2) add redirects or a clearly labelled "superseded — see current build" notice at the 2018Sep URLs that currently 404, pointing users to the equivalent page in the latest BSeR publication. If the same authoring defect still exists in the current build, fix it there using the steps in Disposition A.

#### Justification

The 2018Sep ballot is a point-in-time artifact; regenerating it cleanly years later can be fragile (toolchain drift, IG publisher version differences, dependencies on historical core/US-Core versions). Forward-fixing the current build and providing navigation redirects from the broken historical URLs gives implementers what they actually need — access to working Common Transaction Profiles — without destabilising the ballot archive. This approach also scales: any other latent 404s in `2018Sep` get the same "redirect to current" treatment rather than being patched individually.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive / Out of Scope — historical ballot snapshot." The `2018Sep` URLs are a frozen ballot reconciliation artifact; the work group's responsibility is the current normative/published build of BSeR, not long-dormant ballot outputs. Implementers should be directed to the current BSeR publication (`http://hl7.org/fhir/us/bser/`), and the ticket closed without modifying the 2018Sep output.

#### Justification

The ticket was filed in 2018-09 against a ballot snapshot and has been idle since 2019-02 with no follow-up from reporter or work group, suggesting the reporter's practical need was satisfied by another route. HL7 generally does not rebuild superseded ballot publications to fix broken internal links, and doing so risks touching a build chain that no longer matches current tooling. Declining is defensible provided the current BSeR build's Common Transaction Profile pages are intact; if they are *also* broken, Disposition A or B should be preferred instead.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most honest reflection of how HL7 IG maintenance typically works in 2025: the 2018Sep ballot is a snapshot, but the underlying defect — users landing on 404s when following links to Common Transaction Profiles — can and should be addressed in the *current* BSeR publication, with lightweight navigation help from the stale ballot URLs. Disposition A is correct in spirit but spends effort rebuilding a retired ballot publication, which is rarely worth it. Disposition C is tempting given the ticket's age, but closing without confirming the current build's Common Transaction Profiles pages actually resolve would leave the reporter's original concern unverified. Public Health should confirm the current BSeR build exposes working `ReferralRequestTransactionProfiles`, `ReferralFeedbackTransactionProfiles`, and `ImplementationProfileBundles` pages, fix any that are still broken there, and add a "superseded — see current build" notice at the 2018Sep URLs before resolving the ticket.
