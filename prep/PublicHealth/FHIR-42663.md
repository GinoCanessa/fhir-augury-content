## Summary

The reporter raises a SNOMED CT modeling concern in the US Bidirectional
Services eReferrals (BSeR) IG: several `Observation.code` value set
entries draw from the SNOMED CT *finding* hierarchy (e.g., `364826005
| Finding related to ability to perform breast-feeding (finding)`),
which is semantically inappropriate for `Observation.code` unless the
observation is reported with `valueBoolean` (i.e., asserting the
finding's presence). The recommended pattern is to use codes from the
SNOMED CT *observable entity* hierarchy (or a LOINC code, requesting one
if needed) for `Observation.code`, leaving finding-hierarchy concepts
for `Observation.value[x]`. A concrete substitution is offered —
`289079006 | Ability to perform breast-feeding (observable entity)` in
place of `364826005` — and the reporter requests that the work group
review the appropriateness of eight additional codes currently used in
BSeR observations (covering pain management, isolation, healthful diet,
fatigue, medication compliance, breastfeeding support, maternal
concern, and nipple shield). The cross-reference index links the ticket
only to the FHIR element `Observation.code`; no related Jira, Zulip, or
GitHub items were surfaced. The ticket has been triaged as Persuasive,
indicating the work group accepts the concern in principle.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the BSeR IG, replace the `Observation.code` binding entry
`364826005 | Finding related to ability to perform breast-feeding
(finding) | SCT` with `289079006 | Ability to perform breast-feeding
(observable entity) | SCT` everywhere it appears (value sets, code
systems supplements, profile-level fixed/pattern codes, examples, and
narrative). Then perform a focused terminology review of the other
eight codes the reporter flagged and, where they are bound to
`Observation.code`, swap each finding-/regime-therapy-/procedure-/
physical-object-hierarchy concept for an equivalent
`observable entity` concept (or, where no suitable SCT observable
exists, request a new LOINC code and use that instead). Codes from the
finding hierarchy may remain only where they are used as
`Observation.value[x]` paired with `Observation.code` describing the
observable being asserted, or where the observation is genuinely a
boolean assertion (`valueBoolean`) of the finding's presence. Update
the IG narrative on observations to state this modeling rule
explicitly so future contributors do not reintroduce finding-hierarchy
codes as `Observation.code`.

#### Justification

The reporter's rule reflects long-standing FHIR + SNOMED CT modeling
guidance: `Observation.code` answers "what was observed," which maps
to the SCT *observable entity* hierarchy, while *finding* concepts
answer "what was concluded," which maps to `Observation.value[x]` (or
to a `Condition`). Using a finding code as `Observation.code` collapses
the question and the answer into the same field, which both confuses
implementers and breaks the standard pattern that other Public Health
and US-realm IGs follow. The work group has already triaged the
ticket as Persuasive and the resolution notes confirm
`289079006` as the intended substitute, so accepting the request is
consistent with the disposition the work group has already signaled.
The fact that the reporter explicitly invites a review of the other
eight codes makes it efficient to handle them in the same change set
rather than waiting for follow-up tickets per code.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the modeling principle but address it at the value set /
binding layer rather than by piecemeal code substitution. Specifically:

1. Define (or adopt) a BSeR-scoped value set for `Observation.code`
   that is intensionally constrained to descendants of SCT
   `363787002 | Observable entity` plus selected LOINC codes, and bind
   it (extensible) to every BSeR Observation profile's `code` element.
2. Move the existing finding-hierarchy concepts that were being used
   as `Observation.code` into a parallel value set bound to the
   matching `Observation.value[x] (CodeableConcept)`, so the clinical
   intent (e.g., "feeling isolated", "fatigue", "maternal concern") is
   preserved as the *answer* rather than the *question*.
3. For concepts where neither an SCT observable nor an existing LOINC
   captures the question (e.g., "Nipple shield" as a physical object
   is clearly not an observable), redesign the observation: either
   model it as `DeviceUseStatement` / `Procedure` / `CarePlan`
   activity, or submit a LOINC request for a properly worded
   observation code.
4. Add a FHIRPath / invariant on the BSeR Observation profiles (or at
   least a `mustSupport` + narrative guidance) that rejects
   `Observation.code` concepts from the finding hierarchy except when
   `value[x]` is `valueBoolean`.

#### Justification

A binding-level fix is more durable than a one-by-one code swap: it
prevents the same modeling error from creeping back in as the IG
evolves and makes the intent machine-checkable for implementers
running validation. It also forces an honest answer for the codes that
are not really observables at all (e.g., `335389008 | Nipple shield
(physical object)`, `408883002 | Breastfeeding support (regime/
therapy)`), which Disposition A would only paper over. The trade-off
is scope: this is a larger editorial effort that touches every BSeR
Observation profile and may delay publication. It also relies on
SNOMED International / Regenstrief turnaround time for any
LOINC requests, which can stretch the timeline.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — out of scope for the current
BSeR publication," on the grounds that (a) the BSeR Observation
profiles are bound to value sets whose authority sits with the
clinical use-case sponsors and SNOMED International rather than the
IG editors, and (b) the current bindings have been validated against
real implementer feedback in the field. Add narrative guidance noting
the modeling concern and defer the actual code changes to a future
ballot cycle.

#### Justification

This disposition is the weakest of the three and is included only for
completeness. The only defensible reason to decline is timing — if the
work group is mid-publication and cannot afford a terminology churn,
deferring is preferable to a half-done substitution. However, the
ticket has already been triaged Persuasive and a target replacement
code has already been identified in the resolution notes, so an
outright decline would contradict the work group's own signal. It
would also leave a known modeling defect in a published IG, which
implementers will continue to flag.

---

### Recommendation

**Recommended disposition:** A (with a deliberate path toward B in the
next BSeR cycle).

The work group has already characterized the ticket as Persuasive and
the resolution notes commit to `289079006 | Ability to perform
breast-feeding (observable entity)` as the substitute, so Disposition A
matches the direction the group has already chosen and is the smallest
change that resolves the reporter's specific request. The other eight
codes the reporter flagged should be reviewed in the same change set,
because the cost of opening the value set / profile editing flow is
amortized across all of them and a per-code follow-up ticket trail
would be needlessly noisy. Disposition B is the more architecturally
correct answer and should be scheduled as a follow-up: once the
immediate substitution is in, a structural binding tightening
(observable-only value set for `Observation.code`, finding value set
for `Observation.value[x]`, plus an invariant) will keep the IG from
regressing. Disposition C is not recommended — declining now would
contradict the existing Persuasive triage and leave a known terminology
defect in a published Public Health IG.
