## Summary

The reporter (Ron G. Parker, originally captured as imported Comment 32)
contends that the BSeR IG's minimum cardinality of `1..` on `Task.for` is
overly constrained. His argument: details about the subject are already
contained inside the referenced `ServiceRequest`, so requiring the subject
also to be conveyed via `Task.for` duplicates the same information in two
places and creates an opportunity for the two pointers to disagree. The
request is therefore to relax the BSeR profile on `Task` so that
`Task.for` is no longer required (i.e., make it optional).

The only cross-reference Augury surfaces is to the FHIR artifact
`Task.for` itself (the element being constrained). There are no
cross-referenced Jira tickets, no related Zulip threads, and no related
GitHub items.

The ticket has already been triaged by the Public Health work group with a
resolution of **Not Persuasive**, on the rationale that (a) the constraint
was set through deliberate stakeholder engagement aligned with Gravity
SDOH and others, and (b) `Task.for` is a reference to the beneficiary, so
the data is pointed to rather than duplicated — the same pattern used by
`subject` on most other resources.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify the BSeR `Task` profile so that `Task.for` is no longer required:
relax the minimum cardinality from `1..1` to `0..1`. Concretely, in the
BSeR IG's `Task` profile differential, change the constraint on the
`for` element from `min: 1` to `min: 0` (keep `max: 1`), and update the
profile's narrative to explain that when `Task.basedOn` references a
`ServiceRequest` that itself carries `subject`, the implementer may
omit `Task.for`.

#### Justification

This directly addresses the reporter's concern that the subject is
already conveyed through `ServiceRequest.subject` and that requiring it
again on `Task.for` invites inconsistency between the two pointers. FHIR
core defines `Task.for` as `0..1` and explicitly notes it is "Typically,
but not always, this is a Patient." Relaxing back to the core
cardinality avoids a profile-level requirement that goes further than
the underlying spec without a documented use case unique to BSeR.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Task.for` as required (`1..1`) but add a profile invariant
ensuring it cannot disagree with the subject of the referenced
`ServiceRequest`. The invariant would be expressed in FHIRPath along the
lines of:

> `basedOn.resolve().ofType(ServiceRequest).subject.reference.exists()
> implies for.reference = basedOn.resolve().ofType(ServiceRequest).subject.reference`

In addition, add narrative guidance to the BSeR `Task` profile and to
the `Task.for` element definition explaining that the duplication is
intentional (to allow `Task` to be processed without resolving
`basedOn`) and that the two pointers MUST identify the same subject.

#### Justification

This preserves the work group's stated reason for keeping the field
required — `Task` consumers (e.g., a referral inbox) often need to
filter, route, or display tasks by subject without dereferencing the
embedded `ServiceRequest`, which may not be locally available. Adding
the equality invariant directly answers the reporter's concern about
"potential for error if they do not [agree]" by making the agreement a
machine-checkable conformance rule rather than relying on implementer
discipline.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Close the ticket as **Not Persuasive** and retain `Task.for` at its
current required cardinality in the BSeR `Task` profile. Document in the
resolution that:

1. The constraint reflects a deliberate decision reached through
   stakeholder engagement, including alignment with Gravity SDOH and
   other downstream IGs that consume BSeR `Task` resources.
2. `Task.for` is a reference, not a copy of the subject's data, so the
   "duplication" concern conflates a pointer with the referenced
   content. The same pattern is used by `subject` on virtually all
   request and event resources in FHIR.
3. Keeping `for` populated lets `Task` be indexed, filtered, and
   processed (e.g., subject-scoped task lists) without resolving
   `Task.basedOn`.

#### Justification

This is the disposition already adopted by the work group on
2025-04-01. The reporter's concern — that `for` and the embedded
`ServiceRequest.subject` could disagree — applies equally to every other
FHIR resource that uses both `subject` and a `basedOn` link, and the
canonical FHIR answer has consistently been "the references must
identify the same logical subject," not "drop the redundant pointer."
Loosening the cardinality after stakeholder consensus would also create
churn for existing BSeR implementers who have already built to the
`1..1` profile, with no concrete failing use case in evidence.

---

### Recommendation

**Recommended disposition:** C

The work group has already resolved this ticket as Not Persuasive on
substantive grounds: `Task.for` is a reference (a pointer), not a
duplication of subject data, and keeping it required was an explicit
outcome of cross-IG alignment with Gravity SDOH and other stakeholders.
The reporter has not provided a concrete use case in which the `1..1`
constraint blocks a real exchange — only a theoretical risk of
disagreement between `Task.for` and `ServiceRequest.subject`, which is
inherent to any FHIR resource that carries both a subject and a
`basedOn` link. Disposition A would force an unnecessary breaking change
on existing BSeR implementers, and Disposition B's invariant — while
defensible in principle — adds conformance complexity that the work
group has already judged unwarranted. Confirming the existing Not
Persuasive resolution gives reviewers a clean record of why the
constraint stays in place; if the reporter follows up with a concrete
implementation problem, a future ticket can revisit it with that
evidence in hand.
