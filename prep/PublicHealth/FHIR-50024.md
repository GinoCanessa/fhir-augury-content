## Summary

The ticket asks the VRDR (US Vital Records Death Reporting) IG to fix
its reference to the companion Vital Records FHIR Messaging (VRFM) IG.
The current link is incorrect or imprecise; it should instead point to
the published-versions list for VRFM and explicitly state that any
published version of VRDR is compatible with the current published
version of VRFM. The reporter notes that the fix has already been
applied to the build (CI) version of VRDR, so the request is
effectively to carry that build-version correction through to a
published erratum / next release of VRDR. No cross-referenced Jira
tickets were found, so the scope is confined to this single
documentation/linking issue in VRDR.

### Disposition A: Accept As Requested

#### Proposal

Update the VRDR IG narrative so that the place where it currently
references VRFM:

1. Links to the **published-versions list** for VRFM
   (`https://hl7.org/fhir/us/vrfm/history.html`) rather than to a
   specific version or to the build URL.
2. Adds a short compatibility statement of the form: "Any published
   version of VRDR is intended to work with the current published
   version of VRFM."
3. Carries the existing build-version correction forward into the next
   published release / errata of VRDR (the description confirms the
   fix is already in the CI build).

The change is confined to IG narrative pages — no profile,
ValueSet, CodeSystem, extension, or example resource needs to change.

#### Justification

This is a documentation defect, not a modeling change. Pointing at a
history/published-versions list is the standard HL7 convention for
loosely-coupled cross-IG references and avoids future link rot every
time VRFM publishes a new STU. Stating compatibility explicitly
matches what implementers already need to know to choose a VRFM
version when deploying VRDR. Because the fix is already in the build,
acceptance is essentially confirmation plus a publication action.

---

### Disposition B: Alternative Approach

#### Proposal

Same narrative correction as Disposition A, but go one step further
and replace the inline VRFM mention with a small, reusable
**"Companion Specifications"** section in the VRDR IG that lists each
companion IG (VRFM and any others — e.g., BFDR, etc.) with:

- IG name and short description,
- canonical URL,
- link to the IG's published-versions / history page,
- an explicit version-compatibility statement.

Where the VRFM reference appears in body text, link into this section
rather than directly to VRFM. Apply the same pattern in the VRFM IG
pointing back to VRDR, so the relationship is documented symmetrically.

#### Justification

A dedicated companion-IG section centralises cross-IG version
relationships, making future link/version corrections one-line edits
instead of grep-and-replace exercises. It also gives readers a single
place to discover the broader Vital Records family of IGs. Trade-off:
slightly more authoring work and coordination with the VRFM editors
to mirror the section, in exchange for a maintenance pattern that
removes whole classes of "IG link is wrong / pinned to wrong version"
tickets in the future.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as already-resolved-in-build, on the basis that the
description itself states the correction has been made in the build
version of VRDR, and that no normative or interoperable artifact is
affected. The change would then ride along with the next ordinary
VRDR republication without a tracked ticket.

#### Justification

The defensible rationale would be administrative: the work is already
done in the build, no profile/resource is changing, and there is no
implementer harm waiting on a fast-track. However, this rationale is
weak because (a) the published IG still carries the broken/imprecise
reference until republication, and (b) closing without a tracked
disposition makes it harder to confirm the fix actually shipped.

---

### Recommendation

**Recommended disposition:** A

The ticket is a precise, low-risk narrative correction the reporter
has already implemented in the build; accepting it as requested simply
confirms that fix and gets it into the next published cut of VRDR with
a paper trail. Disposition B is attractive as a longer-term
maintenance improvement and the workgroup may wish to open a separate
follow-up for it, but bundling it into this ticket would expand scope
beyond what was requested and would require coordination with the
VRFM editors before VRDR can republish. Disposition C should be
avoided because it leaves the published IG carrying the incorrect
reference with no tracked closure.
