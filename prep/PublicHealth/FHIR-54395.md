## Summary

The ticket, submitted by AshleyKnealing on behalf of a NAACCR community
member, proposes that the CCRR (US Central Cancer Registry Reporting
Content) IG's "Patient Industry Code" use-case data element capture how
long the patient was active in the occupation/industry being reported.
The reporter recommends that, when populated, the element convey either
a number of years (e.g., 15) and/or a date range (e.g., 1980–1995),
framed as a `period` with `start` and `end`, or at least a range. The
intent is to give cancer-registry consumers temporal context for
occupational exposures associated with the reported industry/occupation.

No related Jira tickets, Zulip threads, or (meaningfully) related
GitHub items were surfaced. The single GitHub cross-reference returned
is a US-Core commit whose message text happens to contain the string
`FHIR-54395` alongside unrelated US-Core SearchParameter/ignoreWarnings
changes; see "Related GitHub Items" below.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the CCRR "Patient Industry Code" use-case data element so that,
when populated, it carries the time the patient was active in the
reported occupation/industry. Concretely, ensure the element backing
Patient Industry Code exposes a `period` (with optional `start` and
`end` of type `date`) and/or a human-readable duration representation
(e.g., years active). Update the IG's usage guidance for the element
to:

- State that `period.start` SHOULD be populated when the start date is
  known, and `period.end` SHOULD be populated unless the occupation is
  current.
- Provide at least one example instance in the IG showing a populated
  `period` (e.g., 1980-01-01 to 1995-12-31).
- Add a narrative note explaining how consumers should interpret
  missing `period.start`/`period.end` (unknown vs. current).

#### Justification

The requester's underlying need — temporal context for occupational
exposure — is clinically and epidemiologically meaningful for cancer
registry reporting. Making `period` explicitly visible (and
illustrating it with an example) addresses the reporter's concern
without inventing new machinery: the comment thread confirms an
occupation `period` element already exists. The gap the ticket really
flags is **discoverability**, not absence, of the capability.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the structural model as-is (since `period.start`/`period.end`
already exist on the occupation element) but address the reporter's
concern with **IG-level guidance changes only**:

- Add a usage-notes block to the Patient Industry Code page explicitly
  citing `period.start` / `period.end` semantics as quoted in the
  ticket comment (start = began; missing start = unknown; end =
  ended; missing end on current occupation = ongoing).
- Add a non-normative "Duration" note showing how implementers can
  derive "years active" from `period` for downstream consumers that
  expect a duration rather than a date range.
- Explicitly document that NAACCR does not currently supply a
  start/end-date mapping, so the period is populated only when a FHIR
  source system has it, not when deriving from a NAACCR record.

No structural changes to the profile; no new extensions.

#### Justification

This preserves the existing profile and avoids churn for implementers
and validators, while still giving the reporter and reviewers a clear
answer: the capability exists, and the IG now documents it. It also
preserves alignment with NAACCR, which the commenter notes does not
supply mappings for the period endpoints — so making `period`
structurally more prominent (Disposition A beyond an example) could
create an expectation the source data cannot satisfy.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Close the ticket as **Not Persuasive**, matching the disposition
recommendation already recorded in the ticket comment by
JIRAUSER30293. Rationale to be recorded verbatim on the ticket:

> The occupation element already exposes a `period` with `start` and
> `end` carrying the semantics requested. There is no NAACCR data
> element that maps to period start/end, so making the range mandatory
> or adding a parallel "years active" representation would create an
> authoring expectation the source data cannot meet.

#### Justification

This is defensible because (a) the structural capability the reporter
asks for already exists on the occupation element, and (b) the CCRR IG
is derived from NAACCR data that has no corresponding period mapping,
so tightening the spec would not improve interoperability and could
degrade conformance for senders who genuinely lack the dates. Declining
avoids breaking-change risk and keeps the IG aligned with its source
data model.

---

### Recommendation

**Recommended disposition:** B

Disposition C is technically correct — the element and its `period`
already exist — but closing the ticket as Not Persuasive without any
IG change leaves the original reporter (and the NAACCR community
member they represent) no better off: the next reader of the IG will
have the same question. Disposition A over-delivers, given the
commenter's note that NAACCR has no source mapping for the dates, and
risks setting an authoring expectation senders can't satisfy.
Disposition B is the middle path: keep the structure unchanged, but
update the Patient Industry Code page to quote the existing
`period.start`/`period.end` semantics, show a worked example, and
explicitly note the NAACCR mapping gap. That addresses the reporter's
concern with guidance rather than structure, and it gives the Public
Health WG a low-risk edit to ship in the same CCRR_Block1 pass.
