## Summary

The reporter (Caroline Macumber) observes that the NHSN Digital Quality
Measure (dQM) Reporting IG defines two value sets that do not appear to
be referenced anywhere in the IG (no profile bindings, no element
slicing, no inclusion in any other terminology artifact). The ticket
asks the Public Health work group to confirm whether these unused value
sets are intentional, and if not, to remove or otherwise account for
them. The resolution recorded on the ticket is *Persuasive with
Modification*: the value sets carry codes that are used in the IG's
*examples*, but because no profile actually binds to them and the
examples already cite the underlying code system directly, the value
sets are effectively unreferenced and can be removed — provided
removal does not break the IG build. If removal does break the build,
the alternative is to keep the value sets and document explicitly that
they exist solely to support the example resources.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with FHIR-48102 in the FhirAugury index.

## Proposed Dispositions

The two value sets are not named in the ticket text. Each disposition
below therefore phrases the action against "the two unreferenced
IG-defined value sets" — the resolver should identify them by
inspecting the IG sources and confirming that no `ValueSet`,
`StructureDefinition` (binding), `CodeSystem`, or other artifact in
the NHSN dQM IG references their canonical URLs. With no GitHub
cross-references in the FhirAugury index, the affected repository for
this IG must be confirmed by the Public Health work group before any
edit lands; the dispositions below assume that confirmation.

### Disposition A: Accept As Requested

#### Proposal

Remove the two IG-defined `ValueSet` resources that are not referenced
by any profile binding or other artifact in the NHSN dQM Reporting IG.
Concretely:

1. Identify the two value sets by their canonical URLs in the IG's
   value-set folder (typically `input/resources/valueset/` or
   `input/fsh/valuesets/` depending on whether the IG is FSH-authored).
2. Confirm via an IG build (`_genonce` / Sushi + IG Publisher) that no
   profile, extension, search parameter, or other value set references
   either canonical URL.
3. Verify the example instances still validate after removal — they
   should, because they already reference the underlying `CodeSystem`
   directly (per the resolver's note) rather than going through the
   value set.
4. Delete the two value-set definition files and any IG-resource
   manifest entries (e.g., `sushi-config.yaml` resource lists or
   `ig.ini`-managed inclusions) that point at them.
5. Re-run the IG build and confirm zero new errors and that the
   removed canonical URLs no longer appear in the published artifact
   list.

#### Justification

This is precisely what the recorded resolution endorses, and it is
the cleanest outcome for downstream implementers. Publishing
ValueSets that no profile actually binds to invites confusion: tooling
indexers and downstream IGs may pick them up, implementers may assume
they are normative, and registry curators may treat them as
authoritative terminology. Removing dead terminology artifacts is
consistent with general FHIR IG hygiene (cf. similar cleanup in
FHIR-46140 for CQF-M, where unused program / value-set artifacts were
demoted to examples). Because the example resources already use the
`CodeSystem` directly, removal has no functional impact on the
examples and no impact on conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Retain both ValueSets but explicitly mark them as **example-only**
support artifacts:

1. Add a clear narrative statement to each ValueSet (`ValueSet.text`
   or the IG's auto-generated description) such as: "This value set is
   provided to support the example instances in this IG and is not
   bound by any profile. Implementers should not infer normative
   binding from its presence."
2. Set `ValueSet.experimental = true` and lower the maturity / FMM
   indicator if one is published, so tooling consumers can distinguish
   it from binding-grade terminology.
3. Optionally move the two ValueSet files into an `examples/` or
   `support/` grouping in the IG's resource index so the published TOC
   surfaces them in a non-normative section.

This is the fallback the resolver themselves named for the case where
removal turns out to break the IG build (e.g., a `Questionnaire`,
`Library`, `Measure`, or other resource silently depends on the
canonical URL).

#### Justification

Disposition A assumes the IG build stays clean after removal. If a
`Measure`, `Library`, or CQL artifact in the dQM IG actually
references either canonical URL — which is plausible in a measure
reporting IG, where Measure / Library resources frequently cite
ValueSet canonicals even without a binding on a profile element —
removal will produce broken references. Keeping the artifacts but
clearly labelling them as example-supporting is a low-risk middle
ground that resolves the reporter's confusion (the artifacts are
documented, not orphaned) without risking publication errors. It also
preserves stable canonical URLs in case any external implementer has
already pinned them.

---

### Disposition C: Decline

#### Proposal

Take no action. Leave both ValueSets in the IG as-is on the grounds
that (a) they are valid FHIR resources, (b) they cause no
conformance issue, and (c) IGs routinely publish supporting
terminology that is not directly bound, e.g., for downstream profiling
or discovery.

#### Justification

Defensible only if the work group concludes that the artifacts have
external value (e.g., other IGs are expected to bind to them, or
NHSN's own non-FHIR processes consume them). In practice the
reporter's observation — that nothing in *this* IG references them —
is correct, and the resolver's note explicitly judged the artifacts
removable, so a Decline disposition would contradict the documented
analysis. Listed for completeness.

---

### Recommendation

**Recommended disposition:** A

The reporter is correct, the resolver concurs, and the recorded
resolution is already *Persuasive with Modification* with a clear
plan: remove the two ValueSets and only fall back to documenting them
if removal breaks the IG build. Disposition A operationalises that
plan directly. Disposition B should be the automatic fallback if step
3 of A's proposal surfaces a broken reference (most likely in a
`Measure` / `Library` artifact of a dQM IG); the work group should
treat B as the contingency rather than the first move. Disposition C
is included for completeness but is not consistent with the
resolver's own analysis. The change is low-risk, mechanical, and
improves IG hygiene with no implementer impact.
