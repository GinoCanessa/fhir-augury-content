## Summary

Craig Newman is asking the eCR (Electronic Case Reporting) authors to clarify the
expected population and data type of `Observation.value` on the eICR Organism
Detected Observation profile. Specifically, he is asking (a) whether there is any
realistic scenario in which `.value` would *not* carry the detected organism's
name, and, if not, (b) whether the cardinality should therefore be tightened to
`1..1` and the type constrained to a specific data type — `string` and/or
`CodeableConcept`. The ticket has no Augury-tracked cross-references, but the
keyword-similarity search surfaces a tightly-related cluster of sibling tickets
filed against the same eCR ballot by the same reporter on the same day:
[FHIR-54348](https://jira.hl7.org/browse/FHIR-54348) (align micro-result modelling
with R5/R6 base-spec guidance), [FHIR-54349](https://jira.hl7.org/browse/FHIR-54349)
(parallel `.value` data-type question on the Susceptibility profile),
[FHIR-54350](https://jira.hl7.org/browse/FHIR-54350) (specimen guidance),
[FHIR-54420](https://jira.hl7.org/browse/FHIR-54420) (susceptibility code value-set
too broad), and [FHIR-54439](https://jira.hl7.org/browse/FHIR-54439) (Sarah Gaunt's
proposal to collapse Susceptibility into components on Organism Detected). Together
these tickets form a coordinated review of the eICR Organism Detected /
Susceptibility profiles, and FHIR-54347 should be dispositioned in the context of
that wider redesign rather than in isolation. A precedent disposition for an
analogous "constrain Observation.value" question exists at
[FHIR-29421](https://jira.hl7.org/browse/FHIR-29421) (Patient Care, US PACIO
Cognitive Status, resolved Persuasive in 2021), where the workgroup *removed* a
`CodeableConcept`-only constraint and instead provided narrative guidance.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the eICR Organism Detected Observation profile so that `Observation.value[x]`
is `1..1` and sliced/typed to `CodeableConcept` only (i.e., remove the other
`value[x]` choices on this profile, keeping `valueCodeableConcept`). Bind the
`.value` element to a SNOMED-CT-based "Organisms" value set (e.g., refining or
re-using the existing eICR organism value set bound on `.code`-context profiles)
with an Extensible binding, so that the detected organism is always present and
always machine-processable as a code. Update the profile narrative on Organism
Detected Observation to state explicitly that the value carries the identified
organism and that absence is not a valid state for this profile.

#### Justification

The clinical semantics of an "Organism Detected" Observation make a 0..1 untyped
`value[x]` close to nonsensical: by definition, an instance of this profile
asserts that an organism was detected, so a populated, code-typed value is the
only way for the resource to carry its core payload to a downstream Public
Health consumer. `CodeableConcept` (rather than `string`) is appropriate because
identified organisms have well-curated terminology (SNOMED CT, NCBI taxonomy);
free-text would defeat the eCR use case of automated public-health reporting.
This is also consistent with how related US Public Health profiles already
constrain identifying observations to coded values.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Observation.value[x]` as `0..1` on the profile, but (a) constrain the
type *choice* to `CodeableConcept | string` (drop Quantity / numeric / Range /
etc.), (b) add a `valueCodeableConcept` slice with the SNOMED organism value
set as Extensible, and (c) add a profile-level invariant + narrative stating
that when an organism has been definitively identified, `valueCodeableConcept`
SHALL be present, while `valueString` is permitted as a fall-back for
laboratory-reported free-text where no code can yet be assigned. Resolve this
ticket *together with* [FHIR-54348](https://jira.hl7.org/browse/FHIR-54348) and
[FHIR-54439](https://jira.hl7.org/browse/FHIR-54439): if FHIR-54439 is accepted
and Susceptibility folds into components on Organism Detected, document the
component slices in the same change.

#### Justification

The reporter's own precedent ticket
[FHIR-29421](https://jira.hl7.org/browse/FHIR-29421) was resolved Persuasive by
*removing* a hard `CodeableConcept`-only constraint and providing guidance
instead, on the grounds that hard `value[x]` slicing is brittle and
implementer-hostile when an unknown future use case emerges. Allowing
`valueString` as a documented fall-back also accommodates intermediate lab
states ("acid-fast bacilli observed, identification pending") that real-world
micro reports do produce, while still strongly steering implementers to coded
values via the invariant and narrative. Coordinating with FHIR-54348 and
FHIR-54439 avoids re-balloting the same profile twice when the surrounding
discrete-micro-results model is already on the table.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54347 as written, on the grounds that it should be subsumed by
[FHIR-54348](https://jira.hl7.org/browse/FHIR-54348) and
[FHIR-54439](https://jira.hl7.org/browse/FHIR-54439). Close as "Not Persuasive
— Considered for Future Use," with a comment directing the reporter to track
the broader Organism Detected / Susceptibility re-modelling effort and to
re-raise the cardinality and data-type question once that re-modelling lands.
Make no profile changes for this ticket alone.

#### Justification

FHIR-54348 may move eCR to the R5/R6 discrete-micro-results pattern, and
FHIR-54439 may collapse Susceptibility into components on Organism Detected.
Either change would significantly restructure where the "organism name" lives
on the resource (potentially in a component rather than `.value`), making any
isolated decision on `.value` cardinality on the current profile shape
premature and likely to require rework. The "decline now, re-raise later"
pattern is appropriate when an in-flight architectural ticket would invalidate
the resolution.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between the two real risks here. Disposition A
is clinically defensible but commits eCR to a hard `value[x]`-only-CodeableConcept
shape just as FHIR-54348 and FHIR-54439 are proposing structural changes that
could move "organism name" off `.value` entirely or change its surrounding
cardinality, and the closely-analogous precedent
[FHIR-29421](https://jira.hl7.org/browse/FHIR-29421) shows that the wider HL7
community has preferred *guidance* over *hard slicing* in exactly this
situation. Disposition C is the safest from a re-work perspective but leaves
the reporter's substantive question unanswered and risks the ticket dying in
the queue while the bigger re-modelling tickets churn. Disposition B answers
the reporter — yes, `.value` is intended to carry the organism, yes it should
be coded, and yes there should be an invariant requiring a coded value when an
organism has been identified — while keeping the door open for FHIR-54348 /
FHIR-54439 to add component slices around it. It also explicitly batches the
disposition with its sibling tickets, which is how the work group is most
likely to actually land a coherent change.
