## Summary

The reporter (Forrest White) has identified a presentation/quality issue in
section **7.12.1.2 Logical Definition (CLD)** of the US Central Cancer Registry
Reporting Content IG. The first bullet point in that section's "Logical
Definition" lists codes drawn from SNOMED CT (`http://snomed.info/sct`,
version "Not Stated — use latest from terminology server") **without any
human-readable display alongside the codes**. The reporter asks whether a
display (i.e., a description of each included code) should be expected
in this rendering, implying that the absence of displays makes the CLD hard
to read and review.

The ticket carries no formal cross-references in Augury — no linked Jira
tickets, no Zulip threads, and no GitHub issues/PRs are recorded against
`FHIR-54699`. Keyword-based similarity surfaces a recurring pattern across

inconsistent, or where SNOMED/LOINC/ICD codes need additional context
(e.g., FHIR-32349, FHIR-34488, FHIR-26056, FHIR-54129), which suggests
this is a known class of authoring/rendering issue rather than a one-off.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the value set referenced from section **7.12.1.2 Logical Definition
(CLD)** of the US Central Cancer Registry Reporting Content IG so that
each SNOMED CT code listed in the first bullet of the CLD is rendered
with a human-readable display in the IG. Concretely:

- In the value set's `ValueSet.compose.include[0].concept[*]` (the
  enumerated SNOMED CT inclusions), populate `display` for every concept
  using the current SNOMED CT preferred term (PT) for the code.
- Re-run the IG publisher so the rendered "Logical Definition" table /
  bullet on page `7.12.1.2` shows `code — display` rather than bare
  codes.
- Leave the version pin as "Not Stated (use latest from terminology
  server)" — the request is purely about display, not versioning.

#### Justification

This is the minimal change that answers the reporter's question
("Is a display of a description of the code expected?") affirmatively
and consistently with how the rest of the IG (and FHIR generally)
renders enumerated value-set inclusions. Authoring-time displays make
the CLD reviewable without a terminology-server round-trip and aid both
balloters and downstream implementers. It is consistent with the
recurring pattern in tickets such as FHIR-32349 and FHIR-34488 where
displays were added or aligned for the same readability reason. As a
"Technical Correction" at Low priority against an already-Triaged
ticket, the change is well within scope and carries no semantic
implications.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hand-populating displays in the value-set source, address
the underlying rendering by:

1. Confirming the value set is authored as an **intensional** include
   (e.g., a `filter` or a `valueSet` reference) rather than an
   enumerated `concept` list — and if it is enumerated, leave the
   source as bare codes.
2. Configuring the IG publisher / terminology server expansion so that
   the rendered CLD on page 7.12.1.2 *expands* the include against the
   bound terminology server and shows the server-supplied displays
   (PTs) at publish time.
3. Adding a small editorial note under the CLD bullet explaining that
   displays are sourced from the terminology server at publication and
   may evolve with SNOMED releases.

This avoids baking SNOMED display strings into the IG source (which
would otherwise need re-syncing every SNOMED release) while still
giving readers the displays they expect.

#### Justification

The reporter's underlying need is "I cannot tell what these codes mean
when I read the CLD." That need is satisfied by either authoring-time
or publish-time display population. Option B is preferable when the IG
intentionally tracks "latest from terminology server" — hard-coding
displays in the source would create a maintenance debt and a risk of
drift between the IG text and the live SNOMED release. The trade-off
versus Disposition A is that B depends on publisher/expansion behaviour
being correctly configured, which the workgroup will need to verify
against their build pipeline.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — already addressed by terminology
server expansion**. Document in the resolution that:

- The CLD intentionally renders bare codes because the binding declares
  "version Not Stated (use latest from terminology server)" and displays
  are expected to be resolved by consumers against that server.
- Implementers and reviewers needing displays should expand the value
  set against a SNOMED CT terminology server.

#### Justification

Defensible only on the grounds that the IG's authoring convention is
"do not pin display text in source when the version is unpinned" and
that the reporter's expectation (displays in the rendered CLD) is a
preference rather than a conformance gap. However, this position is
weak: bare-code CLDs are widely treated as a readability defect across
HL7 IGs, and the same pattern has been corrected in adjacent tickets
(FHIR-34488, FHIR-32349). Declining would also leave the IG less
balloter-friendly for the next publication round.

---

### Recommendation

**Recommended disposition:** B

The reporter is right that bare codes in the rendered CLD are a real
readability problem, so Disposition C (decline) is hard to defend on
its merits. The choice between A and B comes down to *where* the
displays should live. Because this IG explicitly pins SNOMED CT to
"version Not Stated — use latest from terminology server", baking
displays into the value-set source (Disposition A) introduces a
maintenance liability: every SNOMED release risks a drift between the
hard-coded display in the IG and the current PT on the server, which
is exactly the kind of low-grade correction this very ticket
exemplifies. Disposition B keeps the source aligned with the IG's
stated versioning policy while still satisfying the reporter, and it
generalises to every other CLD in the IG that has the same shape. The
workgroup should pick B unless the IG's publisher pipeline cannot be
relied on to expand SNOMED includes at build time, in which case
falling back to A is acceptable as a pragmatic measure for this
ballot cycle.
