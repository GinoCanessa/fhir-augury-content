## Summary

This ticket, raised against the BSeR (Bidirectional Services eReferrals) IG by
Lloyd McKenzie as part of his September 2023 ballot review (imported as
"Comment 59"), reports that one of BSeR's tables of observation concepts is too
sparse to drive interoperable implementations. Using "able to latch" as an
example, the comment notes that the table does not state how the listed code
is meant to be used in an `Observation` instance: should the SNOMED concept go
in `Observation.code` with an `answerBoolean` value (i.e., a yes/no question),
or is the concept itself the answer? The submitter asks that the table be made
explicit about the intended `Observation.code` / `Observation.value[x]` binding
pattern, and notes that the same ambiguity also applies to the corresponding
"responses" table.

No formal cross-references were returned from the augury cross-reference index
for this ticket, but it sits inside a clear cluster of Lloyd McKenzie's BSeR
ballot comments about the observation-code tables, including
[FHIR-42479](https://jira.hl7.org/browse/FHIR-42479) (calling for full
LOINC/answer-data-type/value-set constraints on every BSeR observation code),
[FHIR-42480](https://jira.hl7.org/browse/FHIR-42480) (questioning the
consistency / completeness of the data elements selected for each program),
and [FHIR-42499](https://jira.hl7.org/browse/FHIR-42499) (asking for richer
descriptions in BSeR tables; resolved Persuasive). FHIR-42508 is essentially
the "make the usage pattern explicit" sibling of those constraint and
description complaints and should be dispositioned consistently with them.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR Referral Observations table (the one containing the "able to
latch" row) and the corresponding Response Observations table in the BSeR IG
narrative pages so that each row explicitly documents the intended
`Observation` instance shape. Concretely, add columns (or per-row prose) that
state, for every listed concept:

- Where the concept is carried — typically `Observation.code` for the
  question concept;
- The expected `Observation.value[x]` data type — `valueBoolean`,
  `valueCodeableConcept`, `valueQuantity`, etc.;
- For coded answers, the value set / answer list to bind against;
- For boolean rows like "able to latch", an explicit note that the concept is
  the question and the answer is conveyed in `valueBoolean` (i.e.,
  `Observation.code = <SNOMED>` + `Observation.valueBoolean = true|false`).

Mirror the same change on the responses-side table. Where the same concept
already exists as (or aligns with) a LOINC question, prefer LOINC in
`Observation.code` and reserve SNOMED for the answer side, consistent with
established FHIR Observation guidance.

#### Justification

This is exactly what the ticket asks for and is consistent with how
Persuasive sibling tickets in the same batch have already been resolved
(see [FHIR-42499](https://jira.hl7.org/browse/FHIR-42499), which committed to
adding per-row descriptions to a different BSeR table). The cost is low
(narrative IG edits, not structural profile changes), the benefit is direct
(implementers stop guessing whether the SNOMED code goes in `code` or
`value[x]`), and it aligns with FHIR's general principle that
`Observation.code` carries the question while `Observation.value[x]` carries
the answer.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than fixing the ambiguity only in narrative table prose, replace the
table entries with **per-row Observation profile slices (or per-row
ObservationDefinitions)** that formally constrain the structure for every
listed concept. For each row of the Referral / Response observation tables,
publish a profile that fixes:

- `Observation.code` to the specific code (SNOMED or LOINC, as appropriate);
- `Observation.value[x]` to a single chosen type (`boolean`, `Quantity`,
  `CodeableConcept`, …);
- For coded answers, an `answerValueSet`-style binding;
- Cardinality and `mustSupport` flags.

Cross-link each table row to its profile so implementers see both the
machine-readable contract and the human-readable description in one place.
This subsumes FHIR-42508 (the usage pattern becomes self-evident from the
profile) and also addresses the closely-related
[FHIR-42479](https://jira.hl7.org/browse/FHIR-42479) call for full constraints
in the same change.

#### Justification

The narrative-only fix in Disposition A solves the "what does this row mean?"
problem but leaves the deeper interoperability gap from FHIR-42479
(unconstrained answer types, missing value sets, missing units) unaddressed.
Solving both with one structural pass is more efficient than two narrative
passes and produces machine-readable artifacts implementers can validate
against. The trade-off is a substantially larger editorial workload, the need
for the BSeR team to commit to specific code systems / value sets per row
(which may not yet be agreed), and a potentially more disruptive ballot
re-circulation.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-42508 as **Not Persuasive — duplicate / will be addressed by**
[FHIR-42479](https://jira.hl7.org/browse/FHIR-42479). Rationale: the
substantive interoperability ask (constrain the observation rows so the
question/answer pattern is unambiguous) is already captured by FHIR-42479's
broader call for constraints; resolving 42479 will inherently make the
code-usage pattern explicit, so 42508 does not need a separate disposition.

#### Justification

FHIR-42508 reads as a narrower restatement of FHIR-42479 from the same
submitter in the same batch. Closing it as a duplicate avoids two parallel
editorial workstreams against the same tables and avoids the risk of the
narrative fix (Disposition A) and the structural fix (Disposition B) drifting
apart. The cost is purely process: the workgroup must commit to actually
resolving FHIR-42479 in a way that explicitly addresses the
`code`-vs-`value[x]` ambiguity, otherwise this concern will resurface.

---

### Recommendation

**Recommended disposition:** A

The submitter's specific complaint — that a reader of the table cannot tell
whether the SNOMED concept belongs in `Observation.code` or in
`Observation.value[x]` — is a narrative-clarity problem and is solvable with
the same kind of per-row description enrichment that the Public Health WG has
already accepted as Persuasive on the sibling
[FHIR-42499](https://jira.hl7.org/browse/FHIR-42499). Doing it as a narrative
pass keeps FHIR-42508 small, low-risk, and ballot-cycle-friendly, while
leaving the larger structural conversation (constraints, value-set bindings,
units) to play out under [FHIR-42479](https://jira.hl7.org/browse/FHIR-42479)
on its own merits. Disposition B is the technically stronger answer but
materially expands scope and depends on per-row decisions BSeR may not yet be
ready to make; Disposition C risks losing the explicit "make the usage
pattern visible in the table" requirement if FHIR-42479's eventual resolution
focuses only on bindings and not on table presentation. Accepting FHIR-42508
as requested, and noting the dependency on FHIR-42479 for the deeper
constraints, gives the workgroup the cleanest two-step path.
