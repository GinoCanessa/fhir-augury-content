## Summary

This Public Health ticket against the **Vital Records Common Profiles Library (FHIR)** asserts that the IG's code systems and value set URLs are not properly anchored at `terminology.hl7.org` (THO), contrary to the TSC-approved [Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers) policy. The reporter (Ron G. Parker, comment 45 in an imported batch of 29-style anchoring tickets) requests that the Vital Records project initiate the UTG process so the affected value set — specifically a Yes/No/NA value set called out in the prior disposition — is migrated into THO. The ticket was originally resolved as "Considered for Future Use" on 2024-05-30 (Passas/Wright, 23-0-0; the workgroup explicitly agreed that "Creating a THO Valueset for Yes/No/NA is appropriate"), but lynn_laakso reverted that resolution and re-opened the ticket post-STU2 publication so the actual UTG submission can be tracked and completed. No Jira, Zulip, or GitHub cross-references were returned by FHIR Augury for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive — With Modification. Initiate the UTG process to publish the Vital Records Yes/No/NA value set (and its underlying code system, if a new one is needed) under the `http://terminology.hl7.org` namespace, then update the Vital Records Common Profiles Library to reference the THO canonical(s) and retire the IG-local definition(s). Concretely:

1. File a UTG proposal (Jira "UP" project) requesting a new value set `http://terminology.hl7.org/ValueSet/v3-YesNoNA` (or the agreed THO-conformant slug), bound to an existing THO code system where possible (e.g., `http://terminology.hl7.org/CodeSystem/v2-0136` extended with `NA`, or `v3-NullFlavor` for `NA` plus `v2-0136` for `Y/N`), or to a newly-minted THO code system if no existing one fits.
2. Once UTG publishes the artifact, change every binding in the Vital Records Common Profiles Library that currently points at the IG-local `Yes/No/NA` value set to the new THO canonical, and remove the local copy from the IG.
3. Apply the same pattern to any sibling artifacts flagged by the broader "imported comment 45" batch so the IG ends up free of locally-anchored terminology.

#### Justification

This is exactly what the TSC-approved [Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers) policy requires, and it matches the workgroup's own May-2024 vote (Passas/Wright, 23-0-0) that "Creating a THO Valueset for Yes/No/NA is appropriate." The previous "Considered for Future Use" resolution was explicitly reverted on 2024-10-11 to make this happen post-STU2, so accepting now both honours that vote and clears a known compliance gap before the next ballot cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive — With Modification. Rather than minting a new VRCL-specific Yes/No/NA value set in THO, **bind to an existing THO terminology** that already covers the concept. Two viable existing anchors:

- `http://terminology.hl7.org/ValueSet/v2-0136` (Yes/No indicator) for the Yes/No portion, combined with `http://terminology.hl7.org/CodeSystem/v3-NullFlavor#NA` for "Not Applicable", expressed as a small composed value set in THO; or
- A profile-level `extensible` binding to `v2-0136` plus a documented usage rule that `NA` is conveyed via `dataAbsentReason`/`nullFlavor` rather than as an in-band code.

In either case, the Vital Records IG would still drop its locally-anchored value set, but the UTG submission would be a binding/composition change rather than the creation of a brand-new VRCL-flavoured artifact.

#### Justification

THO already contains well-vetted Yes/No vocabulary; introducing a new VRCL-branded `YesNoNA` value set risks fragmenting how "not applicable" is represented across HL7 IGs and creates a maintenance burden for TSMG. Reusing `v2-0136` plus the standard null-flavor mechanism keeps the IG aligned with the rest of the v2/v3/FHIR ecosystem, satisfies the "anchor in THO" policy, and avoids a new code system entirely. The trade-off vs. Disposition A is some implementer ergonomics (callers must populate two places to express NA) in exchange for tighter ecosystem consistency.

---

### Disposition C: Decline

#### Proposal

Not Persuasive. Close the ticket without action on the grounds that (a) the IG has already shipped STU2 with the locally-anchored value set, (b) implementers have built against those canonicals, and (c) re-anchoring to THO is a breaking change that would force every VRCL implementer and downstream IG to retest bindings, with no clinical or safety benefit.

#### Justification

This position is defensible on stability grounds alone — `Trial Use` artifacts are still expected to maintain canonical-URL stability between ballots where reasonable, and a wholesale re-anchor of value sets touches every conformance resource in the IG. However, this rationale is weak in light of the TSC policy explicitly requiring THO anchoring and the workgroup's own 23-0-0 vote that the THO migration is appropriate; a "decline" disposition would essentially override both the policy and the prior vote.

---

### Recommendation

**Recommended disposition:** B

The workgroup has already agreed (23-0-0) that the Yes/No/NA concept belongs in THO, and the TSC policy gives no room to leave it locally-anchored, so Disposition C is off the table. Between A and B, **B is preferable** because THO already publishes mature Yes/No vocabulary (`v2-0136`) and a standard mechanism for "not applicable" (`v3-NullFlavor#NA` / `dataAbsentReason`); creating yet another `YesNoNA` value set — even under the THO namespace — adds a parallel artifact to maintain and risks divergence from the broader HL7 ecosystem. Reusing existing THO content honours both the TSC policy and TSMG's de-duplication goals while still producing the same end-state for the VRCL IG (no IG-local terminology, all canonicals under `terminology.hl7.org`). If TSMG reviews the UTG proposal and concludes that an explicit composed `YesNoNA` value set is operationally clearer for implementers, the workgroup can fall back to Disposition A without losing any work, since both paths require the same UTG submission and the same VRCL binding swap.
