## Summary

The ticket asks the Vital Records Common Profiles Library (VRCL) to add a
dedicated "test jurisdiction" code — `TT` (display: *Test Jurisdiction*) —
to the jurisdiction value set used by vital records resources, so that
test records carrying a jurisdictional identifier can validate without
colliding with a real US state, territory, or federal jurisdiction code.
In a follow-up comment the reporter broadens the request to also add
`TS` (a test jurisdiction reserved for STEVE — the State and Territorial
Exchange of Vital Events — interoperability testing) and `ZZ` (an
"unknown jurisdiction" code). There are no cross-referenced Jira
tickets, Zulip threads, or GitHub items in the Augury index, so the
request stands on its own and must be evaluated on the merits of the
VRCL jurisdiction value set design.

## Proposed Dispositions

Because no GitHub cross-references are attached, the dispositions below
reference the VRCL specification by name. The concrete authoring
location (CodeSystem / ValueSet source files) would be confirmed at
implementation time from the HL7 VRCL IG repository.

### Disposition A: Accept As Requested

#### Proposal

Extend the VRCL jurisdiction CodeSystem / ValueSet with three additional
concepts so that jurisdictional test records are conformant:

- `TT` — *Test Jurisdiction* — "Test Jurisdiction. Reserved for use
  with test records that must carry a jurisdictional identifier but do
  not refer to any real US state or territory."
- `TS` — *Test Jurisdiction (STEVE)* — "Test Jurisdiction reserved for
  STEVE (State and Territorial Exchange of Vital Events) interoperability
  testing."
- `ZZ` — *Unknown Jurisdiction* — "Jurisdiction unknown or not
  otherwise specified."

Add the three codes to the VRCL jurisdiction CodeSystem, include them in
the bound ValueSet, update the narrative of any profile that binds to
that ValueSet to mention the test/unknown semantics, and ship the change
in the next VRCL publication. The codes should be documented as *not*
for use in production records.

#### Justification

The request is a narrow, backward-compatible additive change to a VRCL
code system. Having explicit sentinel codes for test and unknown
jurisdictions is a well-established pattern in vital-records exchange
(STEVE already reserves test jurisdictions in practice), and it
eliminates the current conformance gap where implementers are forced to
reuse a real jurisdiction code (e.g., a small state) for test traffic,
which pollutes downstream analytics and reporting. Adding `ZZ` for
"unknown" aligns with the long-standing ISO/USPS convention of using
two-letter sentinels outside the assigned state/territory range and
gives implementers a conformant way to represent missing jurisdiction
data without resorting to data-absent-reason extensions on a required
element.

---

### Disposition B: Alternative Approach

#### Proposal

Add only the test-jurisdiction codes (`TT` and `TS`) to the VRCL
jurisdiction CodeSystem, and address "unknown jurisdiction" through the
standard FHIR data-absent-reason mechanism rather than a code-system
entry. Concretely:

1. Add `TT` and `TS` to the jurisdiction CodeSystem and ValueSet with
   the definitions above, clearly marked as test-only.
2. For unknown jurisdiction, update the binding profile(s) so the
   jurisdiction element is either optional or, where it must be
   populated, instruct implementers to use
   `http://terminology.hl7.org/CodeSystem/data-absent-reason#unknown`
   (via `valueCodeableConcept`'s data-absent-reason extension, or by
   making the element a `CodeableConcept` that can carry
   `dataAbsentReason`).
3. Document the guidance in the jurisdiction ValueSet / profile
   narrative.

#### Justification

Mixing genuine test sentinels with a semantic "unknown" marker in the
same CodeSystem blurs two distinct concerns: the first is a conformance
convenience for synthetic records, the second is a real-world data
quality signal. FHIR already has a standard pattern —
`data-absent-reason` — for "value could not be determined," and using it
keeps the jurisdiction CodeSystem focused on jurisdictions (test or
real). This also avoids the risk that downstream analytics silently
treat `ZZ` as just another jurisdiction. The trade-off versus
Disposition A is a slight increase in implementation burden (profiles
must allow the extension) in exchange for cleaner semantics.

---

### Disposition C: Decline

#### Proposal

Do not add `TT`, `TS`, or `ZZ` to the VRCL jurisdiction CodeSystem.
Instead, document in the VRCL implementation guidance that test traffic
should either (a) use a reserved example jurisdiction already employed
in VRCL example bundles, or (b) carry a dedicated test-record marker
(e.g., a `meta.tag`, an `Extension`, or a `Composition.status` value)
rather than overloading the jurisdiction code. For unknown jurisdiction,
rely on `data-absent-reason` as in Disposition B's point 3.

#### Justification

VRCL's jurisdiction binding is intended to identify the real
state/territory/federal jurisdiction responsible for a vital event.
Introducing non-jurisdictional sentinel codes into that value set
permanently mixes test and production semantics in a terminology that
will be consumed by analytics, registries, and downstream exchange
partners who may not honor the "test-only" narrative caveat. The
underlying need — making test records conformant — can be met without
changing the code system, by marking test records at the resource level
(e.g., `meta.tag` with an `ActReason` test code) and by using
`data-absent-reason` for the unknown case. Declining avoids the risk of
real production data being coded `TT`/`TS`/`ZZ` by accident and then
flowing through jurisdictional reporting pipelines.

---

### Recommendation

**Recommended disposition:** A

The request is small, well-scoped, and responds to a concrete
conformance problem that VRCL implementers hit today: there is no
conformant way to send a jurisdictional test record. The STEVE ecosystem
already uses `TS` in practice, so adding it to the published
CodeSystem aligns the specification with existing field behavior rather
than inventing anything new. The concern that motivates Disposition C —
test codes leaking into production data — is real but is better handled
with clear "not for production use" definitions and examples than by
declining the change, because implementers who need a test code will
otherwise pick a real state code, which is a worse outcome. The only
genuinely debatable piece is `ZZ` (test sentinel vs. missing-data
semantics); if the Public Health work group is uncomfortable mixing
those, the pragmatic compromise is to accept `TT` and `TS` now and
treat `ZZ` as a separate decision, routed through
`data-absent-reason` per Disposition B.
