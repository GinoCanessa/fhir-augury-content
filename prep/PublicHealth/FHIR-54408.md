## Summary

The submitter, writing on behalf of a NAACCR community member, asserts that
a Central Cancer Registry (CCR) report under the US Central Cancer Registry
Reporting Content IG should not be triggered (and should not be considered
of any value to CCRs) unless a minimum set of patient-identifying data
elements is present as discrete content in the EHR. The required minimum
they propose is: first name, last name, address state, sex, date of birth,
**and** a Primary Cancer Condition. The request is therefore both a data
quality gate ("require these elements to be discrete in the source EHR")
and a reporting-trigger gate ("do not fire a CCR report unless those
elements have content").

No related Jira tickets, Zulip threads, or GitHub items were surfaced by
the cross-reference index for this ticket, so the request currently stands
on its own without prior community discussion or implementation work
attached to it.

## Details

**Description:**

> I expect at minimum patient identifiers of first and last name, address

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the US Central Cancer Registry Reporting Content IG so that the
five named patient demographics plus a Primary Cancer Condition are
treated as a hard prerequisite for both reporting and triggering:

- On the CCRR Patient profile (the IG's profile of US Core Patient),
  raise `Patient.name.family`, `Patient.name.given`, `Patient.gender`,
  `Patient.birthDate`, and `Patient.address.state` to **min = 1** and
  **mustSupport = true** for the reportable bundle / Composition. (Where
  US Core already requires the element, mirror the constraint and add
  the IG-level invariant that the value is non-null/non-empty in the
  reported instance.)
- Add an invariant on the CCRR Composition / reporting bundle profile
  asserting that all five demographics are populated and that at least
  one Primary Cancer Condition (the IG's existing `Condition` profile)
  is included as a section entry.
- Update the IG's reporting-trigger definition (the PlanDefinition /
  eligibility logic that drives "should this report fire?") so that the
  trigger does not fire — or fires into a hold/queued state — until all
  six elements have content. Document this as a pre-condition in the
  trigger's `condition` element rather than only as post-hoc validation.
- Reflect the same minimums in the IG's narrative "Reporting
  Requirements" / "Triggering" pages so EHR implementers see them
  alongside the technical constraints.

#### Justification

The five demographics are already understood to be foundational for
patient matching, deduplication, and jurisdictional routing at central
cancer registries; without them a CCR cannot reliably link an incoming
report to the right person or even to the right state registry, which
is exactly the submitter's concern. Requiring a Primary Cancer Condition
is consistent with the IG's purpose — a cancer-reporting bundle with no
cancer condition has no reporting value. Encoding the requirement in
both the profile (min = 1 / invariants) **and** the trigger
pre-condition gives CCRs a clean signal and avoids the worst failure
mode the submitter is calling out: low-quality reports being generated
and transmitted at all.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the spirit of the request but split it into two layered mechanisms
rather than a single hard `min = 1` everywhere:

- **Profile layer (must-support, not mandatory):** Keep cardinality on
  `Patient.name`, `Patient.gender`, `Patient.birthDate`, and
  `Patient.address.state` aligned with US Core's existing constraints
  (must-support, not raised to mandatory beyond what US Core already
  imposes), and add explicit IG guidance that producers SHALL populate
  these when known.
- **Quality / trigger layer (hard gate):** Define a CCRR-specific
  `eligibility` invariant or a PlanDefinition `condition` expression
  that evaluates the candidate report bundle and only allows the report
  to be triggered/transmitted when (a) all five demographics resolve
  to a non-empty value and (b) at least one Primary Cancer Condition
  is present. Reports failing the gate are surfaced to the EHR
  user/administrator as "not reportable yet — missing required data"
  rather than being silently sent.
- Provide a small, named ValueSet/CodeableConcept-free invariant
  (FHIRPath) that implementers can reuse, plus a worked example in the
  IG showing the gate behavior.

#### Justification

Raising base-profile cardinalities to `min = 1` on a US Core-derived
Patient profile is a heavier change than it looks: it can break round-
tripping with US Core, complicate validation against historical or
imported records (e.g., older cancer cases with sparse demographics),
and force EHRs to fabricate placeholder values to pass validation —
which actively undermines the submitter's goal of *real* discrete data.
Putting the hard requirement on the **trigger / eligibility** layer
keeps the profiles compatible with US Core, lets the EHR retain
incomplete records non-destructively, and still guarantees the
submitter's outcome: no CCR report leaves the EHR until the named
elements are populated. It also gives the work group a single place to
evolve the rule (e.g., add or remove elements) without re-balloting
profile cardinalities each time.

---

### Disposition C: Decline

#### Proposal

Decline the request as written, on the grounds that (1) the IG already
relies on US Core Patient, which establishes must-support semantics for
the named demographics, and (2) the broader "what makes a CCR report
reportable" gate is appropriately the responsibility of the registry
business rules and the implementing EHR's reporting workflow, not of
the FHIR IG's profile cardinalities or trigger definition. Close the
ticket as "Not Persuasive" with a short note pointing the submitter at
the existing US Core must-support requirements and at the IG's
narrative on reporting completeness.

#### Justification

If every jurisdictional CCR data-quality preference were lifted into
profile cardinality or trigger pre-conditions, the IG would quickly
diverge from US Core and accumulate registry-specific rules that don't
generalize. A standards-track IG is not the right vehicle to enforce
operational data quality on every implementer; that belongs in the
registry's submission/acceptance rules, where reports lacking the
named elements can be rejected with feedback. Declining is defensible
when the request is essentially a policy statement about local data
quality rather than a missing technical capability.

---

### Recommendation

**Recommended disposition:** B

The submitter's core concern is real and the work group should respond
to it: a CCR report with no name, no state, no DOB, no sex, and no
cancer condition is genuinely unusable, and the IG should not pretend
otherwise. However, Disposition A's blanket `min = 1` on Patient
elements that derive from US Core risks breaking US Core conformance
and incentivizes implementers to insert junk values just to satisfy
validation — which makes the underlying data quality *worse*, not
better. Disposition C does nothing and leaves the submitter's concern
unaddressed. Disposition B captures the intent — no report leaves the
EHR until the named elements are populated — by placing the gate on
the trigger/eligibility layer, where it belongs, while keeping the
profiles compatible with US Core. Because the ticket has no Zulip
discussion, no linked tickets, and no community signal beyond the
submitter, the work group should also use this disposition as an
opportunity to confirm with NAACCR/CCR stakeholders that the proposed
six-element gate (5 demographics + Primary Cancer Condition) is the
right minimum before locking it into the IG.
