## Summary

The Public Health work group, acting on a request from the FHIR-I Workflow
group's 27 Oct 2025 call, is asking that a `.status` element be added at the
root of `ImmunizationRecommendation`. The element would be cardinality `1..1`,
type `Coding`, marked `IsModifier = true`, and `Required`-bound to a new value
set drawn from the existing `request-status` code system
(`http://hl7.org/fhir/request-status`). The motivating use case is the ability
to retroactively mark a recommendation as `entered-in-error`; a second value,
`generated`, is also wanted from `request-status` but the addition of that
code is being tracked separately because Patient Care owns that code system.

The request lands in a cluster of three sibling tickets filed the same day
out of the same FHIR-I Workflow call, all aimed at aligning
`ImmunizationRecommendation` more closely with the Request pattern:
[FHIR-53253](https://jira.hl7.org/browse/FHIR-53253) re-maps several existing
elements (`recommendation.forecastStatus` → `Request.doNotPerform`,
`recommendation.vaccineCode` → `Request.product`, `recommendation.reason` →
`Request.reason`), and
[FHIR-53254](https://jira.hl7.org/browse/FHIR-53254) proposes a new
`.requestor` (0..1 Reference(Device)) so the CDS engine that produced the
forecast can be identified. There is also relevant prior art in
[FHIR-40759](https://jira.hl7.org/browse/FHIR-40759) (Apr 2023, *Resolved -
No Change*), where a similar request to add `.intent` to
`ImmunizationRecommendation` was rejected on the grounds that an intent of
"proposed" is implied by the resource and other intent values are out of
scope. The current ticket is narrower — it targets only `.status`, with
`entered-in-error` as the primary driver — so the prior decision does not
automatically settle it, but it should be weighed.

## Details

**Description:**

> On 20251027, the FHIR-I workflow group requested that
> ImmunizationRecommendation contain a .status data element in order to
> accommodate instances where a recommendation might be retroactively marked
> as "entered-in-error". The data element would have the following
> characteristics:
>
> - Cardinality - 1..1
> - Data Type - coding
> - IsModifier - True
> - ValueSet Binding Strength - Required
> - ValueSet - new value set containing entered-in-error and a new value of
>   "generated" from the existing request-status code system
>   (http://hl7.org/fhir/request-status) — Note that a separate Jira ticket
>   will be created to add "generated" as the code system is owned by Patient
>   Care (although FHIR-I will likely be interested as well).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new `status` element to the root of the `ImmunizationRecommendation`
`StructureDefinition` in `HL7/fhir`, in its authoring location under
`source/` (likely `source/immunizationrecommendation/structuredefinition-ImmunizationRecommendation.xml`,
to be confirmed against the current clone layout — see Repo Context gap).
The element's `<differential>` definition is:

- Cardinality `1..1`.
- Type `Coding`.
- `isModifier = true`, with a `modifierReason` describing that
  `entered-in-error` invalidates the recommendation.
- Binding strength `required` to a new value set
  (e.g., `ImmunizationRecommendationStatus`) defined under the same
  resource folder as `valueset-immunizationrecommendation-status.xml`. Today
  the value set contains only `entered-in-error` from
  `http://hl7.org/fhir/request-status`; once the companion ticket against
  Patient Care lands, expand to include `generated`.
- Update `ImmunizationRecommendation` introduction/notes prose to describe
  the new modifier semantics and the (interim, single-code) value set.
- Open / link a companion Public-Health-side ticket against Patient Care for
  adding `generated` to `request-status`, and gate the value-set expansion
  on that resolution.

#### Justification

Aligns the resource with the Request pattern that FHIR-53253 and FHIR-53254
are also moving toward, so the three sibling tickets land coherently.
Directly addresses the implementer pain point captured in the Mar 2022
*Cancel current ImmunizationRecommendation* Zulip thread, where users had
to wrap recommendations in CarePlan purely to express invalidation. Using
the existing `request-status` code system (rather than minting a new one)
is consistent with FHIR's terminology-reuse principle and keeps governance
in Patient Care. `isModifier = true` is correct because `entered-in-error`
fundamentally invalidates the recommendation. Caveat acknowledged in the
Repo Context: introducing a `1..1` modifier element is a breaking change
for downstream profiles and existing instances, which the work group must
accept as the cost of correcting a known modelling gap.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same semantic outcome but with two modelling changes that reduce
breakage risk:

1. Make the new element `0..1` instead of `1..1`, still `Coding`, still
   `isModifier = true`, still `Required`-bound. The `Required` binding
   continues to police the legal codes, and `0..1` lets existing instances
   and profiles remain valid until they are explicitly upgraded. Document
   in the introduction that the absence of `.status` is to be interpreted
   as `active`/`generated`-equivalent.
2. Defer landing the value-set expansion until Patient Care has added
   `generated` to `request-status`. In the interim, ship the
   `entered-in-error`-only value set so the primary use case
   (retroactive correction) is supported on day one without depending on
   another work group's release schedule.

All other authoring locations and recipes are as in Disposition A.

#### Justification

Keeps the door open to the use case the requester actually cares about
(retroactive `entered-in-error`) while avoiding an avoidable breaking
change for every existing implementation. Consistent with how several
status-like fields elsewhere in FHIR are modelled (e.g., several Request-
pattern resources allow optional status interpretations in early drafts).
Also avoids coupling the FHIR Core release to a Patient-Care-owned
terminology change — the briefing flags this as a real cross-WG sequencing
risk. Trade-off vs. A: weaker conformance — implementers are not *forced*
to declare `active`/`generated`, which means tooling cannot rely on the
element being populated. The work group should weigh that against the
breakage cost of `1..1`.

---

### Disposition C: Decline

#### Proposal

Reject the request and instead recommend that implementers express
`entered-in-error` via one of the existing FHIR mechanisms:

- The `Resource.meta.tag` mechanism, using
  `http://terminology.hl7.org/CodeSystem/v3-ObservationValue#invalidated`
  (or similar) to mark the resource as no-longer-valid, **or**
- A wrapping `CarePlan` whose status carries the workflow state, which is
  the pattern Artem Pavlov was already using in the Mar 2022 Zulip thread,
  **or**
- Resource deletion / version history, where the prior version remains
  retrievable and the current version is gone.

Document whichever mechanism is preferred in the
`ImmunizationRecommendation` notes page, and close the ticket with that
guidance.

#### Justification

Carries forward the precedent set by
[FHIR-40759](https://jira.hl7.org/browse/FHIR-40759), where the work group
declined to add `.intent` on the basis that
`ImmunizationRecommendation` is a forecast and most workflow-pattern
elements don't apply. Adding a `1..1 isModifier` element is a breaking
change for every implementer and every profile, and FHIR already provides
mechanisms for marking arbitrary resources as invalid. Defensible if the
work group treats `entered-in-error` as a generic concern rather than a
resource-specific one. Weakness: implementers have repeatedly asked for a
direct mechanism, and `meta.tag`-based invalidation is not as
discoverable or as reliably honoured by tooling as a true modifier element.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the real-world need — implementers want a documented
way to mark an `ImmunizationRecommendation` as `entered-in-error` — without
imposing the breaking change of a mandatory `1..1` modifier on every
existing instance and profile in the FHIR ecosystem. The Zulip evidence
(particularly the Mar 2022 *Cancel current ImmunizationRecommendation*
thread) establishes that the gap is real, so Disposition C is too
conservative and re-runs the FHIR-40759 conversation without engaging with
the substantively different driver here (`entered-in-error` as a true
modifier, not an aspirational `.intent`). Disposition A is the cleanest
modelling outcome and best aligns with the sibling tickets FHIR-53253 and
FHIR-53254, but `1..1` plus a Patient-Care-owned terminology dependency
creates a coordinated landing problem the work group should not take on
unless the breakage cost has been explicitly weighed. Disposition B
delivers the user-visible behaviour now, lets FHIR-53253 / FHIR-53254
proceed in parallel, and leaves a clear upgrade path to `1..1` in a future
ballot once `generated` exists in `request-status` and downstream profiles
have had a cycle to set the new element.
