## Summary

The ticket reports an interoperability/usability problem in the US Birth
and Fetal Death Reporting (BFDR) IG. The BFDR Composition resources
reference the Mother Patient and Child Patient profiles defined in the
Vital Records Common Library (VRCL): the child is the subject of the
composition, and the mother is the focus of several sections. However,
when a consumer receives a BFDR Bundle, there is no required, easily
queryable element that distinguishes a Mother Patient instance from a
Child Patient instance. Because both conform to base `Patient`, a simple
FHIRPath expression cannot reliably pick out "the mother" or "the child"
from the bundle without traversing every Composition reference.

The reporter, Saul Kravitz, proposes adding a differentiating "role"
extension to the Mother and Child Patient profiles in VRCL — analogous
to the existing role pattern used by the Practitioner / PractitionerRole
profiles in the IG — so that a single, simple FHIRPath expression can
fish out the appropriate Patient instance from a BFDR Bundle.

were surfaced for this ticket by the cross-reference index, so the
discussion below is grounded entirely in the ticket text and the BFDR /

### Disposition A: Accept As Requested

#### Proposal

Add a new "role" extension to VRCL and bind a required (or at least
must-support) instance of it on the **Mother Patient** and **Child
Patient** profiles. Concretely:

- Define a new extension in VRCL, e.g.
  `http://hl7.org/fhir/us/vrcl/StructureDefinition/patient-role`, with
  context `Patient`, value type `CodeableConcept` (or `code`), bound
  required to a small VRCL-defined value set, e.g.
  `vrcl-patient-role-vs` containing at minimum `mother` and `child`
  (and reserving room for other vital-records subjects such as
  `father`, `decedent` if useful for VRDR alignment).
- On the **Mother Patient** profile: slice
  `Patient.extension` to require exactly one occurrence of this
  extension, fixed (or pattern-bound) to `mother`. Make it `1..1` and
  must-support.
- On the **Child Patient** profile: do the same with the value fixed
  to `child`.
- Document the resulting FHIRPath idiom in the BFDR/VRCL guidance, e.g.
  `Bundle.entry.resource.ofType(Patient).where(extension('http://hl7.org/fhir/us/vrcl/StructureDefinition/patient-role').value = 'child')`.
- Update BFDR Composition narrative / examples to demonstrate the new
  extension on contained Patient instances.

#### Justification

This directly solves the reported problem: a single, deterministic
FHIRPath expression can identify the mother and child Patient instances
without traversing Composition.subject / Composition.section.focus.
It also mirrors a pattern the reporter explicitly cites as already
present in the IG (the Practitioner / PractitionerRole role pattern),
which means consumers learn one idiom and reuse it. Because the
extension is introduced at the VRCL layer, downstream IGs that reuse
VRCL Mother/Child Patient (BFDR, VRDR, fetal death, etc.) inherit the
discriminator for free, improving cross-IG consistency for vital
records reporting.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introduce a new VRCL extension, use existing FHIR machinery
to make the Mother / Child Patient instances self-describing:

- Require, on the Mother Patient profile, a
  `Patient.meta.profile` slice that *must* include the canonical URL of
  the VRCL Mother Patient profile (`1..*`, must-support, fixed to the
  profile canonical). Do the same for Child Patient with its canonical.
- Equivalently or additionally, require a fixed `Patient.identifier`
  slice with a vital-records role system (e.g.
  `system = "http://hl7.org/fhir/us/vrcl/CodeSystem/patient-role"`,
  `value = "mother" | "child"`), if `meta.profile` is felt to be too
  fragile across servers that strip or normalize meta.
- Document the canonical FHIRPath:
  `Bundle.entry.resource.ofType(Patient).where(meta.profile.contains('http://hl7.org/fhir/us/vrcl/StructureDefinition/Patient-Mother'))`.

#### Justification

This avoids minting a new extension when the platform already provides
two well-understood discriminators: profile assertion via `meta.profile`
and typed identifiers. It is zero-cost at the schema level (no new
StructureDefinition), and `meta.profile`-based discrimination is
already the convention many validators and FHIRPath-based tools use to
slice bundles. The trade-off is fragility: not every server preserves
`meta.profile` faithfully, and identifier-based discrimination requires
agreeing on a system URL that consumers have to know about — which is
arguably no less work than learning a single role extension. It is also
slightly less ergonomic in FHIRPath than a direct `extension('…').value`
lookup.

---

### Disposition C: Decline

#### Proposal

Decline the request as already addressed by the existing BFDR /
Composition structure and by `Patient.meta.profile`. Clarify in BFDR
implementer guidance (a small documentation page, not a profile change)
that the canonical way to identify the mother and child within a BFDR
Bundle is:

1. The `Composition.subject` reference resolves to the Child Patient.
2. The mother is identifiable via `Composition.section.focus` on the
   sections that pertain to her, and/or via
   `Patient.meta.profile` containing the VRCL Mother Patient canonical.

No normative changes to VRCL or BFDR profiles.

#### Justification

The information needed to tell mother from child is already present in
the bundle — it is reachable, just not via a one-line FHIRPath. A
documentation-only resolution avoids both a new VRCL extension and a
profile change that ripples through every IG that reuses VRCL
Mother/Child Patient. It also avoids creating a redundant axis of truth
(role extension vs. profile assertion vs. Composition role) that
implementers then have to reconcile when they disagree. The downside is
that consumer code remains more verbose, and authors who don't read the
guidance will continue to hit the same usability cliff the reporter
describes.

---

### Recommendation

**Recommended disposition:** A

The reporter is describing a real, recurring usability problem: a BFDR
Bundle contains two Patient instances that are semantically very
different (mother vs. child), and today there is no required, locally
inspectable element on the Patient resource that says which is which.
Disposition C (documentation only) leaves implementers writing
non-trivial FHIRPath that traverses Composition references, and
Disposition B (lean on `meta.profile`) is technically clean but
operationally fragile because not all intermediaries preserve
`meta.profile` and it is not a discriminator FHIR was designed to
*require*. Disposition A is the smallest normative change that gives
implementers exactly what the reporter asked for — a single, reliable
FHIRPath idiom — and it reuses a pattern (a role-style code carried as
an extension) that the IG already uses for Practitioner /
PractitionerRole, so the cognitive cost to implementers is low. Placing
the extension at the VRCL layer also lets sibling vital-records IGs
benefit, which is consistent with the purpose of VRCL as a common
library. The workgroup should confirm with the VRCL maintainers that
adding a required extension to Mother/Child Patient is acceptable from
a backwards-compatibility standpoint before adopting.
