## Summary

FHIR-54591 ("ODH Recommendation") is a ballot-style comment against the
Electronic Case Reporting (eCR) FHIR IG that recommends the IG retain its
existing Occupational Data for Health (ODH) capture mechanisms while
*also* adding a US Core–based option. The reporter (Amanda Deering)
argues that keeping the existing ODH profiles preserves continuity for
early adopters and supports richer occupation-related data capture
beyond USCDI, while the US Core option — being constrained to USCDI
data elements — would be a better on-ramp for new implementers. The
comment notes that both options reference the same value sets used in
USCDI v3 / v3.1, so terminology alignment is preserved either way. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by Augury for this ticket, so the request is being evaluated
on its own merits.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the eCR FHIR IG to formally support **two** parallel ODH
representations and document them as such:

1. **Retain** the existing ODH-based occupation profiles currently
   referenced by eCR (the ODH Occupation/Employment Status profiles, or
   their eCR-aliased equivalents) as the default / continuing pathway
   for implementers already in production.
2. **Add** a US Core–constrained option (US Core Observation
   Occupation / Employment Status, scoped to the USCDI v3 / v3.1
   element set) as a second supported pathway, intended for new
   implementers whose scope is bounded by USCDI.

In the eCR IG, this means: (a) keep the existing ODH must-support
references intact; (b) add the US Core occupation-related profiles to
the IG dependency list and to the relevant Composition / bundle slicing
so either profile satisfies the slot; (c) add narrative guidance in the
"Occupational Data" section of the IG explaining when each option is
appropriate and confirming both bind to the same USCDI value sets.

#### Justification

This directly answers the comment as written. ODH is the long-standing
HL7 mechanism for occupation data and is already implemented by early
eCR adopters; removing it would force a breaking migration with no
functional gain, since US Core's occupation observations are a strict
subset. Offering both keeps backward compatibility for production
senders while giving new implementers a lower-burden, USCDI-aligned
entry point. Because the comment explicitly notes that both options
reference the same value sets used in USCDI v3 / v3.1, the terminology
surface is unchanged and downstream receivers can normalize on the
shared codes.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than carrying two parallel profile slots, keep **one slice per
data element** (e.g., one "occupation" slice, one "employment status"
slice) but allow the slice to be satisfied by **either** the ODH profile
**or** the US Core profile via an `Observation` slicing discriminator on
`meta.profile` (or on `code` where the codes already differ). Document
ODH as the recommended profile for senders that already capture the
richer ODH element set, and US Core as the minimum-conformant profile
for USCDI-only senders. Receivers MUST process either.

This keeps the IG's bundle structure stable, avoids the appearance of
two competing "occupation" sections, and lets the choice of profile
remain a sender-side decision driven by data availability rather than
an IG-level either/or.

#### Justification

A single slot with two acceptable profiles is closer to how US Core
itself handles overlapping concepts and avoids confusing implementers
with two visually distinct "occupation" entry points in the
Composition. It also reduces the risk that a receiver implements one
path but not the other. Because both profiles bind to the same USCDI
value sets, a discriminator on `meta.profile` is unambiguous and tooling
(e.g., the IG publisher, validator) handles it cleanly. The trade-off
vs. Disposition A is slightly less narrative prominence for the ODH
pathway, which may matter to early adopters who want explicit IG-level
acknowledgement that ODH remains first-class.

---

### Disposition C: Decline

#### Proposal

Decline the comment on the grounds that the eCR IG's current direction
on occupation data is already settled (single ODH-based pathway, or
single US Core–based pathway, whichever the WG has currently adopted),
and that introducing a second parallel option expands the conformance
surface without a demonstrated implementer need beyond the reporter's
recommendation.

#### Justification

Defensible only if the WG has already concluded — through prior tickets
or connectathon evidence not surfaced here — that supporting both
pathways doubles the receiver-side conformance burden without
proportionate benefit, or if the WG's intent was an explicit migration
*from* ODH *to* US Core for harmonization with other USCDI-aligned IGs.
Declining is weakest absent that prior context: no Zulip thread, linked
ticket, or GitHub discussion was found that argues against retaining
ODH, and the reporter's argument (continuity for early adopters, richer
data capture, same value sets) is on its face reasonable.

---

### Recommendation

**Recommended disposition:** A

Accepting as requested is the right starting position for the WG
discussion. The reporter's argument is concrete and grounded in real
implementer concerns (early adopters already using ODH, richer
occupational data available via ODH, USCDI-aligned simplicity for new
implementers), and the cost of accepting is low because both pathways
bind to the same USCDI value sets — terminology drift is not a risk.
Disposition B is a strong fallback if the WG is uncomfortable with two
parallel "occupation" entries in the Composition; the choice between A
and B is largely a packaging decision and can be settled in
implementation. Disposition C should only be selected if the WG has
prior context (not surfaced by Augury for this ticket) showing that
dual-pathway support has already been considered and rejected.
