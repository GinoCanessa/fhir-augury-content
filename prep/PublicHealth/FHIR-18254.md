## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the entire class-diagram section from the BSeR IG narrative. In
practice this means deleting (or commenting out) the page/section in the
IG's narrative content (typically an `input/pagecontent/*.md` page in the
IG Publisher source, or the equivalent legacy DSTU2-era HTML page that was
in use when this ticket was filed in 2018) that hosts the diagram, along
with the diagram source/image asset (e.g., `.plantuml`/`.png`/`.svg`) and
any cross-references to it from the IG's table of contents (`menu.xml` or
the equivalent `pages.xml`/`ig.json` entry).

#### Justification

The reporter — an experienced FHIR IG author — explicitly states the
diagram does not tie into the profiles "in any meaningful way." A class
diagram in a profile-driven IG that doesn't map to the profiled resources
is actively misleading: implementers may try to reconcile it against the
StructureDefinitions and lose time. Removing rather than rewriting also
matches the comment cost (a single-sentence ballot comment with no
counter-suggestion); the reporter is signalling the diagram has no
salvageable value, not asking for a rework.

---

### Disposition B: Alternative Approach

#### Proposal

Retain the section but rework it so the diagram is grounded in the BSeR
profiles. Concretely: replace the existing class diagram with one generated
from (or visibly aligned to) the IG's actual profiles — e.g., a
`ServiceRequest` / `Task` / `MessageHeader` interaction diagram for the
referral exchange, with each box labelled with the BSeR profile canonical
URL and each association labelled with the profiled element (e.g.,
`ServiceRequest.basedOn`, `Task.focus`). Add a short paragraph explaining
what scenario the diagram illustrates and link each shape to the relevant
profile page in the IG.

#### Justification

A scoped, profile-aligned diagram can be genuinely useful for implementers
who want a one-page mental model of the referral workflow before diving
into individual StructureDefinitions — this is a common pattern in mature
IGs (e.g., US Core's "must support" overview, IPS bundle diagram). The
trade-off vs. Disposition A is editorial effort: someone must own the
rewrite, and if no one steps up, this disposition silently degrades into
"do nothing," which is worse than removal. It also assumes the underlying
intent of the original diagram can be recovered, which the ticket itself
casts doubt on.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope or insufficiently justified at this
time. Mark the diagram section as "informative" / "non-normative" if it
isn't already, and defer any structural change to a future BSeR ballot
cycle when broader narrative cleanup is on the agenda.

#### Justification

This is a narrative-only complaint with no normative impact on
conformance, no implementer harm beyond confusion, and no cross-references
from Zulip or GitHub indicating wider community concern. Spending ballot
disposition cycles on removing a diagram — when no replacement is offered
and no other commenter has chimed in — is hard to defend if the WG has
heavier substantive tickets queued. Declining keeps the IG stable until a
coordinated narrative refresh is undertaken.

---

### Recommendation

**Recommended disposition:** A

The ticket is six years old (filed 2018, last touched 2019), still sits in
`Triaged` with `Unassigned`, and has attracted zero follow-up discussion
in Jira, Zulip, or GitHub. That pattern strongly suggests no one has
volunteered to either defend the diagram (Disposition C) or rewrite it
(Disposition B). Disposition A is the lowest-cost action that actually
resolves the reporter's concern: the diagram is, by the reporter's
account, a net negative for implementer comprehension, and removing it
costs only a narrative edit with no normative consequences. If, during
the disposition discussion, an author steps forward willing to own the
rewrite proposed in Disposition B, the WG can pivot — but the default
should be removal rather than indefinite deferral.
