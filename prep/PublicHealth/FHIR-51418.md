## Summary

The ticket proposes extending the US Vital Records Death Reporting (VRDR) IG
to support a pre-certification exchange flow in which an EHR transmits
decedent information to an Electronic Death Registration System (EDRS)
before the death certificate is certified. Today, VRDR is scoped primarily
to the jurisdiction-to-NCHS reporting flow, so several of its profiles
impose constraints that are not feasible at the EHR-to-EDRS stage. The
reporter suggests three concrete directions: (1) defining a new Bundle
(type `document` or `message`) to carry the pre-certification payload, (2)
marking VRDR elements that the EDRS (not the EHR) is expected to populate
as optional — possibly via the IG's Conformance section rather than profile
edits — and (3) adding slices to relevant profiles. The Death Record
Identifier is called out as a specific example, because its format (year +
jurisdiction + jurisdiction-assigned death certificate number) cannot be
satisfied by an EHR that has not yet received a certificate number from
the jurisdiction — the same concern tracked in
[FHIR-50344](https://jira.hl7.org/browse/FHIR-50344). A comment from
cynthia_bush (2025-09-23) confirms the direction: a new bundle will be
created for this use case and the IG's Conformance section will be
updated.

## Linked Jira Tickets

### [FHIR-50344: Clarify requirement of Death Record Identifier](https://jira.hl7.org/browse/FHIR-50344)

FHIR-50344 is referenced directly in the body of FHIR-51418 and is the
concrete trigger for the broader request. It points out that the VRDR
Death Record Identifier format — year of death + jurisdiction of death +
jurisdiction-assigned death certificate number — is not feasible for the
initial EHR-to-jurisdiction interaction because the death certificate
number is assigned downstream by the jurisdiction. FHIR-51418 generalises
that observation into a structural proposal (new Bundle, optional
elements, profile slicing) so that pre-certification EHR→EDRS payloads
can be valid VRDR instances. FHIR-50344 is currently "Waiting for Input"
and shares the same reporter and work group.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the VRDR IG to add an explicit "EHR → EDRS pre-certification"
exchange. Concretely:

1. Define a new bundle profile — e.g. `Bundle-death-pre-certification`
   (type `document`, paralleling the existing
   `Bundle-document-death-certificate`) — whose `Composition` entry points
   at a new `Composition-death-pre-certification` profile. The bundle
   would collect the decedent, cause-of-death observations (where known),
   injury/external cause observations (where known), certifier/attestor,
   and any identifiers the EHR can assert.
2. Add sliced `identifier` elements on the existing Death Record
   Identifier / Certificate Number profiles so that the jurisdiction-
   assigned certificate number slice is `0..1` (optional) at the
   pre-certification stage, while remaining `1..1` on the post-
   certification document bundle — aligning with FHIR-50344.
3. Relax cardinalities on EDRS-assigned elements (certifier licence
   identifier, manner-of-death determination when it is the certifier's
   responsibility, etc.) on the pre-certification profiles, keeping the
   existing VRDR document bundle unchanged.
4. Update the IG's Conformance section to describe the two flows (EHR →
   EDRS pre-certification, jurisdiction → NCHS post-certification) and
   which profiles apply to each.

#### Justification

This matches exactly what the reporter described and what cynthia_bush's
comment confirms is already the working direction ("A new bundle created
will be created to support this use case, and will add updates to
conformance section of the IG"). Creating a parallel bundle — rather
than relaxing cardinalities on the existing certified death record
profiles — preserves the validation guarantees that downstream
jurisdiction and NCHS consumers rely on, while giving EHR vendors a
conformance target they can actually meet before a certificate number
exists. Slicing the Death Record Identifier also cleanly resolves
FHIR-50344 without reinterpreting the current format.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a single set of VRDR profiles but introduce the pre-certification
exchange as a **capability-statement / use-case layer** rather than new
Bundle and Composition profiles. Specifically:

1. Add a new section to the IG's Conformance / Use Cases page describing
   the EHR → EDRS pre-certification flow.
2. Define one or more `CapabilityStatement` resources (e.g., "EHR
   pre-certification source", "EDRS pre-certification receiver") that
   declare which elements are `MustSupport` in each role, explicitly
   marking elements assigned by the jurisdiction (certificate number,
   certifier licence, etc.) as **not** `MustSupport` for the EHR role.
3. Address FHIR-50344 narrowly by relaxing the Death Record Identifier
   slice's `value` expectations (or sliding the jurisdiction-assigned
   slice to `0..1`) rather than authoring a second bundle.
4. Leave the existing document Bundle as the single structural target.

#### Justification

This keeps the VRDR artefact surface smaller and avoids a duplicate set
of nearly-identical profiles that must be maintained in lockstep. It
leans on FHIR's existing mechanism for expressing "different actors care
about different elements" (CapabilityStatement + `MustSupport`) instead
of proliferating profiles. The trade-off versus Disposition A is that
validation tools will not, by themselves, catch an EHR that submits a
post-certification-shaped bundle — the conformance distinction lives in
the CapabilityStatement and narrative, not in the StructureDefinition.
For implementers that already consume VRDR, this is less disruptive;
for new EHR implementers, it is arguably less clear.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for the current VRDR release and track the
EHR→EDRS pre-certification use case as a future ballot cycle item.
Address FHIR-50344 on its own terms (clarifying the Death Record
Identifier expectation for initial EHR submissions) but defer the
broader Bundle / Conformance rework.

#### Justification

The request spans new profiles, conformance narrative, and potentially a
new workflow actor — a significant scope increase late in a ballot
cycle. If community use cases and EDRS vendor alignment are still being
collected, locking in a `Bundle-death-pre-certification` profile now
risks having to revise it in the next cycle. Declining here is
defensible if the workgroup judges that (a) the single-profile + Death
Record Identifier clarification from FHIR-50344 is enough for the next
release, and (b) there is not yet enough implementer input to design a
stable pre-certification bundle.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already indicated (via cynthia_bush's 2025-09-23
comment) that a new bundle plus Conformance updates is the direction
being pursued, which is the Disposition A proposal in all but name.
Disposition A also gives EHR vendors a concrete, validatable target that
cleanly handles the Death Record Identifier problem raised in
FHIR-50344 without weakening the post-certification document bundle that
jurisdictions and NCHS rely on. Disposition B is lighter-weight but
leaves the real-world "EHRs cannot satisfy the certificate number
constraint" problem to narrative rather than schema, which is the exact
failure mode FHIR-50344 complained about. Disposition C punts on a
problem the reporter and a work-group member have already agreed needs
to be solved. Recommend proceeding with Disposition A, explicitly
mirroring FHIR-50344's resolution in the new pre-certification profile's
Death Record Identifier slicing.
