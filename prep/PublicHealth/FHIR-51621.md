## Summary

The reporter (Rebecca Angeles) notes that on the History page of the US
Central Cancer Registry Reporting Content IG (FHIR), the link to the
"current" build is not working. This is a publication / website
infrastructure defect rather than a content modeling change: visitors
who navigate from the History page to the current CI build land on a
broken link. No related Jira tickets, Zulip discussions, or GitHub

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a publication-side defect against the US Central Cancer
Registry Reporting Content IG and fix the History page so the
"current" entry resolves to the active CI build URL. Concretely:

- Inspect the IG's `package-list.json` (or equivalent `history`
  configuration consumed by the IG history template) and confirm the
  `current` entry points to the canonical CI build URL
  (`https://build.fhir.org/ig/HL7/<repo>/`).
- Re-publish the History page (regenerate `history.html` /
  `history.cfml` from the standard HL7 history template) so the
  rendered "current" link matches the corrected entry.
- Verify in CI that the rebuilt History page produces a working link
  to the latest CI build.

No FHIR conformance artifacts (resources, profiles, value sets) need
to change.

#### Justification

The ticket describes a broken link, not a modeling concern. Accepting
as requested keeps the fix narrowly scoped to publication metadata,
which is the lowest-risk change and directly addresses the reporter's
observation. It also preserves the History page's intended
navigation, which IG consumers rely on to find the most recent build
when comparing to published versions.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the immediate broken link as in Disposition A, **and** add a
lightweight CI check that validates the "current" link on the History
page returns HTTP 200 against the published CI URL. This could be
implemented either in the IG's existing CI workflow (a small
`curl --fail` step against the rendered History page) or by
parameterizing the History template so the "current" URL is derived
from the IG's `canonical` plus a known CI path, eliminating the class
of hand-edit-induced broken links.

#### Justification

The History page is generated from a small set of inputs, but those
inputs are easy to get out of sync when an IG is renamed, moved, or
re-homed under a different `HL7/<repo>` path. A one-line link check
prevents this exact ticket from recurring and is consistent with
similar guards used in other HL7 IGs. The trade-off is a small
amount of additional CI maintenance, which is justified by the
recurring nature of "current build link" tickets across the IG
portfolio.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the basis that the History page and "current"
build link are produced by the shared HL7 IG publisher / history
template tooling rather than by content authored in this IG, and that
the underlying publication infrastructure is owned by FHIR
Infrastructure (or the IG Publisher project) — not Public Health.
Recommend the reporter re-file against the IG Publisher / history
template repository if the issue persists after the next IG
re-publish.

#### Justification

Public Health, as the sponsoring work group, owns the IG's content
but typically does not own the History page template or the link
resolution behavior used by `history.html`. If the broken link is
caused by a defect in the shared template (rather than a wrong entry
in this IG's `package-list.json`), the fix belongs upstream and a
content-side disposition would not actually resolve the user's
complaint.

---

### Recommendation

**Recommended disposition:** A

This is, on the available evidence, a self-contained publication
defect specific to the CCRR IG's History page. The most likely root
cause is a stale or incorrect `current` entry in the IG's
`package-list.json` (or an out-of-date rendered `history.html`),
both of which are within Public Health's authoring scope and can be
fixed with a single re-publish. Disposition B is a sensible
follow-on if the work group wants to prevent recurrence, but it
should not block closing this ticket. Disposition C is only
appropriate if investigation shows the bug is in the shared history
template; the work group should fall back to it only after
confirming the IG's own `package-list.json` is correct and the link
is still broken.
