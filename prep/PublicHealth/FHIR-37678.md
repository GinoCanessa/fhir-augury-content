## Summary

The ticket asks the SMART Health Cards: Vaccination and Testing (FHIR) IG to add a quick-navigation feature — a keyboard-driven "quick jump" / command-palette style affordance (think VS Code Ctrl-K) that lets readers navigate the published IG without using the mouse. The motivation, captured almost entirely outside the ticket body itself (the Jira description was filed empty), is accessibility and speed of navigation: the reporter (Max Masnick) and other implementers (Josh Mandel, Grahame Grieve) saw real value in keyboard-only navigation for users with RSI or other accessibility needs, and a working implementation based on the [`ninja-keys`](https://github.com/ssleptsov/ninja-keys) component was developed and demonstrated in `HL7/fhir-shc-vaccination-ig` (see commits `fe69212`, `5fe7c38`, `9b375aa`). The most recent comment (2023-08-01) notes that the custom-template implementation has been **removed from this IG** in favor of pursuing the feature as a general enhancement to the default HL7 IG template, with a pointer to the Zulip thread `IG-creation > .js file in custom template`. As of the snapshot, the ticket is **Triaged / Unresolved / Unassigned** and there is no linked successor ticket against `fhir-ig-publisher` or `ig-template-base` recorded in the cross-reference graph.

## Related Zulip Discussions

### [IG creation > .js file in custom template](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/.js%20file%20in%20custom%20template)

The thread starts as a build-error question: Max Masnick added a custom JavaScript file (`custom-template/content/assets/js/something.js`) to the SHC vaccination IG and the IG publisher refused to execute it because the custom template is "not trusted" (`has file extensions: [.js]`). Lloyd McKenzie and Jose Costa Teixeira clarify that this is intentional — the CI build only runs scripts from HL7-managed templates, and the path forward for organization-specific scripting is either (a) submit changes upstream to the `base` template via PR or (b) propose a new HL7-managed template on the Tuesday IG calls. Grahame Grieve later refines the trust model: the gate is not strictly "controlled by HL7" but "controlled by someone Grahame chooses to trust," because templates can run code (Ant, JS, XSLT) on HL7's build server.

The conversation then pivots to the actual feature when Josh Mandel praises the "Quick Jump (Ctrl-K)" navigator that Max had shipped on the SHC IG and asks for it in the base template. Reactions split: Max, Josh, Grahame, and (more cautiously) Lloyd and Jose all support keyboard navigation as an accessibility win, with Lloyd noting it is essentially a richer ToC replacement and that a left-side collapsing menu would arguably be better still. John Moehrke and Eric Haas push back, calling it a "distraction" and a "niche" feature given the existing ToC page. Lloyd closes the loop with a concrete next step — file an issue against `fhir-ig-publisher` — and Max commits to doing so, noting two prerequisites: (1) a computable ToC that JS can parse, and (2) upstream changes to `HL7/ig-template-base`. Max also flags that he wants to investigate `fuse.js` for true fuzzy search before opening the upstream issue, and Jose suggests bundling related template improvements (section numbering, side menu) into the same rework. No final consensus on scope was recorded in this thread, and no `fhir-ig-publisher` issue link was captured in the cross-reference data.

Participants: Max Masnick, Jose Costa Teixeira, Lloyd McKenzie, Josh Mandel, Jean Duteau, John Moehrke, Grahame Grieve, Eric Haas.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restore the quick-navigation feature in `HL7/fhir-shc-vaccination-ig` by reinstating the vendored `ninja-keys`-based command palette that was added in `fe69212` and removed in `d5c971b`. Concretely: revert `d5c971b` (or cherry-pick the asset re-introduction), keep the inlined-JS approach from `5fe7c38` so the IG publisher's trust check passes, and bind a Ctrl-K / Cmd-K shortcut wired against the IG's existing TOC data so that readers of the published SHC Vaccination & Testing IG get a keyboard-driven page picker. No upstream changes to `ig-template-base` or `fhir-ig-publisher` are made; the feature stays scoped to this IG.

#### Justification

This is exactly what the reporter originally implemented and what Josh Mandel and Grahame Grieve explicitly endorsed on Zulip. The accessibility argument (RSI, keyboard-only navigation) is real and was not seriously contested — the dissent (John Moehrke, Eric Haas) was about whether the feature deserved upstream investment, not whether it was harmful in the SHC IG itself. The CI-build trust gotcha is already solved by the `5fe7c38` inlining pattern, so the only operational risk is maintenance of the vendored asset within this single IG. Cost is low and the change is fully contained in one repository.

---

### Disposition B: Alternative Approach

#### Proposal

Close FHIR-37678 against the SMART Health Cards: Vaccination and Testing (FHIR) IG with a "Persuasive — addressed elsewhere" resolution, and instead pursue the feature upstream:

1. File a tracker issue against [`HL7/fhir-ig-publisher`](https://github.com/HL7/fhir-ig-publisher) requesting a publisher-emitted, machine-readable table of contents (JSON) that any template can consume — this is the prerequisite Max Masnick identified on Zulip.
2. File a companion PR against [`HL7/ig-template-base`](https://github.com/HL7/ig-template-base) adding a Ctrl-K command-palette built on the vendored `ninja-keys` (or equivalent) that consumes the publisher-emitted TOC, behind an opt-in `ig.ini` / `sushi-config.yaml` flag so individual IGs can enable it without forcing it on everyone.
3. Once the upstream feature ships, the SHC Vaccination & Testing IG opts in via the flag and ships no custom JS of its own.

This matches the resolution direction Max recorded in the 2023-08-01 comment ("removing this for now in favor of future enhancements to the default IG template").

#### Justification

The Zulip thread converges on this as the right architectural answer: Lloyd asked for an `fhir-ig-publisher` issue, Max agreed, and Grahame voiced support for keyboard navigation in general. Solving it upstream avoids per-IG vendoring of a JS dependency, side-steps the per-template trust review Grahame must otherwise perform for every IG that wants the feature, and gives every published HL7 IG (not just the SHC vaccination IG) the accessibility benefit. Trade-off vs. Disposition A: it is materially more work and depends on bandwidth in the IG publisher / base template repos, both of which sit outside Public Health's scope; in exchange the change is durable and reusable. The "side menu / section numbering" extension Jose floated can be tackled in the same upstream rework without blocking the keyboard-shortcut piece.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-37678 "Not Persuasive" and close it. The Triaged status, the empty description, the absence of any linked successor ticket against `fhir-ig-publisher` or `ig-template-base`, and the reporter's own 2023-08-01 comment ("Removing this for now…") together indicate that the request as originally framed — *add this to the SHC Vaccination & Testing IG specifically* — has already been withdrawn by the reporter. The existing TOC page provides functionally equivalent navigation, and the publisher's trust model intentionally prevents per-IG custom JS from being executed on the CI build server.

#### Justification

A defensible Decline rests on three points: (1) the reporter has already removed the implementation and signalled intent to pursue the feature elsewhere, so leaving the ticket open against this IG no longer corresponds to any active work; (2) John Moehrke and Eric Haas raised legitimate concerns that an in-IG keyboard shortcut is a niche scope creep over a publisher / base-template responsibility; (3) the IG publisher's trust gate (Lloyd, Grahame) makes per-IG custom JS the wrong layer to fix this at, which means accepting the request as-is would create an exception to a deliberate security boundary. Decline is harm-free here because the underlying need is not lost — anyone who wants the feature can still file the upstream issue Max committed to in the Zulip thread.

---

### Recommendation

**Recommended disposition:** B

Disposition B aligns with where the discussion actually landed on Zulip (Lloyd: "file an issue against `fhir-ig-publisher`"; Max agreeing to do so) and with the reporter's own 2023-08-01 comment that explicitly defers the feature to "future enhancements to the default IG template." It captures the accessibility benefit Josh Mandel and Grahame Grieve championed without forcing every implementer to vendor JS into every custom template, and it cleanly respects the CI-build trust boundary Grahame highlighted. Disposition A is workable but parochial — it leaves every other IG without the feature and accepts perpetual maintenance of vendored JS inside a single Public Health IG. Disposition C is honest about the current state of the ticket but throws away the documented community appetite for the upstream feature; if Public Health closes the ticket without first ensuring the upstream `fhir-ig-publisher` / `ig-template-base` tracker exists, the work simply gets lost. Workgroup action under B should therefore be: (i) confirm with the reporter that no upstream tracker has been filed in the intervening time, (ii) if not, file the two upstream items described in the Proposal, and (iii) close FHIR-37678 with a "Persuasive — addressed elsewhere" resolution that links to those upstream items.
