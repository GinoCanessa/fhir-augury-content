## Summary

The reporter asks the Public Health work group to enrich the narratives
of the profiles published in the US Public Health Profiles Library
(USPHPL) so that each profile explicitly identifies — or links to — the
specific public health use cases that actively rely on it. The intent
is to give implementers a quick way to see, from the profile page
itself, which downstream PH implementation guides or reporting
workflows actually consume the profile, rather than having to infer
that mapping from the wider ecosystem. The ticket was voted in the
Public Health work group on 2022-09-21 (Sarah Gaunt / John Loonsk,
15-0-0) and was re-opened post-STU publication for action in the next
ballot cycle. No cross-referenced Jira tickets, Zulip threads, or
GitHub items were found by FhirAugury.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update each profile in the US Public Health Profiles Library so its
narrative (the profile description / `StructureDefinition.text` and the
human-readable intro on the profile page) explicitly enumerates the
public health use cases that actively use the profile. Concretely:

1. For every PH profile in the USPHPL (e.g., the PH-flavoured Patient,
   Condition, Observation, Encounter, Immunization, etc. profiles),
   add a "Public Health Use Cases" section to the profile page
   listing each consuming PH implementation guide / use case (e.g.,
   Electronic Case Reporting, Reportability Response, Immunization
   Reporting, Birth & Fetal Death Reporting, Cancer Reporting, etc.)
   along with a hyperlink to the relevant published IG.
2. Where a profile is *not* used by any current PH use case, state
   that explicitly so reviewers can decide whether to retire or
   continue maintaining it.
3. Reflect the same use-case list in the profile's
   `StructureDefinition.description` so it surfaces in tooling that
   consumes the raw artifact, not just the rendered IG site.
4. Establish a maintenance rule (in the IG's authoring guidance) that
   any new PH IG which adopts a USPHPL profile must add itself to the
   profile's use-case list as part of its own publication checklist.

#### Justification

This is exactly what the ticket asks for, and it was already endorsed
by Public Health 15-0-0 on 2022-09-21. Surfacing the consuming use
cases directly in the profile narrative gives implementers a strong
"why does this profile look the way it does" signal, helps reviewers
gauge the blast radius of any proposed change to a USPHPL profile, and
makes the profiles' relevance auditable from the artifact itself rather
than from tribal knowledge. It is consistent with how other HL7
products (e.g., US Core) document downstream consumers in their profile
intros.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of hand-curating a use-case list inside each profile's
narrative, capture the relationship in structured metadata and let the
publishing tooling render it:

1. Tag each USPHPL profile with one or more `useContext` entries (or a
   small extension on `StructureDefinition`) whose `code` identifies
   the consuming PH use case / IG (e.g., `eCR`, `RR`, `IZ-Reporting`).
2. Maintain a single registry artifact in the USPHPL repo (e.g., a
   `usecases.json` or a Confluence-backed table) listing each PH use
   case, its canonical IG URL, and which USPHPL profiles it consumes.
3. Have the IG publication step generate the "Public Health Use Cases"
   section on each profile page automatically from that registry, so
   the human-readable narrative and the structured metadata can never
   drift apart.
4. Optionally surface the same data as a cross-reference page ("Which
   profiles does *my* use case depend on?") in the rendered IG.

#### Justification

The literal request (Disposition A) addresses the *display* problem but
puts the burden of keeping the list correct on every author of every PH
IG, in every profile narrative they touch. A structured approach
trades a one-time tooling investment for ongoing maintenance: the
canonical truth lives in one place, gets surfaced consistently on every
profile page, and is queryable by external tooling (e.g., for impact
analysis when a profile changes). It also avoids divergence between
the rendered narrative and the underlying StructureDefinition, which
is a recurring source of bugs in narrative-only updates.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds of scope: the USPHPL is intentionally a
*library* of reusable profiles, and per-IG use-case lists belong in
each consuming IG rather than in the library that supplies the
building blocks. Recommend instead that each PH IG continue to state
which USPHPL profiles it consumes (which they already do via their
profile lists / dependency declarations), and close FHIR-38524 as
"Persuasive with Modification — addressed elsewhere".

#### Justification

Embedding consumer lists in a shared library inverts the normal
dependency direction (libraries shouldn't know about their consumers)
and creates an editorial coupling: every time a new PH IG ships, the
USPHPL must be re-balloted or errata'd to mention it. The information
the ticket wants is technically already discoverable by walking the
`dependsOn` declarations of the consuming IGs, and a reader who needs
"which use cases use this profile" can be served by a generated
cross-reference page maintained outside the profile narrative itself.

---

### Recommendation

**Recommended disposition:** B

Disposition A is what the ticket literally asks for and what the work
group voted to accept, but in practice profile narratives drift quickly
when their maintenance is manual — especially across a library of
profiles consumed by multiple, independently-balloted PH IGs. Capturing
the profile ↔ use-case relationship as structured metadata and
rendering the narrative from it preserves the user-visible outcome the
work group endorsed (every profile page shows its consuming PH use
cases), satisfies Lynn Laakso's note to land the change in the next STU
publication, and gives Public Health a single source of truth that can
be reused for impact analysis when profiles evolve. Disposition C is
defensible architecturally but ignores a unanimous work-group vote and
the implementer pain point the ticket is trying to solve, so it should
only be revisited if the tooling investment in B proves infeasible
within the STU timeline — in which case fall back to A.
