## Summary

The reporter argues that the **Vaccine Credential Patient Assertion** profile
in the US Public Health Profiles Library (`us-ph-vaccine-credential-patient-assertion`,
2025Sep build) does not actually model what its name implies. The profile
binds `Observation.code` to LOINC **11370-4 (Immunization status — Reported)**,
whose answer set is *Yes / No / Unknown / Not Asked*. That code only captures
whether the patient or provider has asserted some immunization status; it does
not assert that the patient possesses a physical or electronic **vaccine
credential** (e.g., a paper certificate or a SMART Health Card). The reporter
proposes three avenues for fixing this: (1) submit a new LOINC request for a
true *Vaccine Credential* observation code; (2) restructure the profile along
the lines of the **US Core Observation ADI Documentation** profile, which
pairs an observation with a referenced supporting document; and/or (3) replace
or supplement 11370-4 with LOINC **105580-5 (Immunization certificate)**,
pending guidance from the Terminology Infrastructure (TI) WG on whether a
LOINC DOC code can be used as an `Observation.code`. No formal cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced for this issue, so
the discussion record is limited to the ticket's own comment thread.

## Details

**Description:**

The modeling of the **Vaccine Credential Patient Assertion** profile
([StructureDefinition-us-ph-vaccine-credential-patient-assertion](https://hl7.org/fhir/us/ph-library/2025Sep/StructureDefinition-us-ph-vaccine-credential-patient-assertion.html))
appears not to say what is intended.

The current LOINC code in use ([11370-4](https://loinc.org/11370-4/)) is part
of a panel that also includes specific immunization information (e.g.,
[55168-9](https://loinc.org/55168-9)). As designed, 11370-4 (*Immunization
status - Reported*) with answer options (*Yes, No, Unknown, Not Asked*) does
not actually represent whether a patient has a physical or electronic vaccine
credential (e.g., certificate document or SMART Health Card). Rather, it only
reflects that the patient (or provider) has **asserted a vaccination status**
about one or more immunizations.

At best, this implies the patient has some sort of immunization status —
which may mean they have had no immunizations, or have not had a particular
immunization — but it does not equate to possession of a vaccine credential.

**Recommendation**

- Consider submitting a **new LOINC request** to represent a true *Vaccine
  Credential* assertion.
- Model the profile similarly to
  [US Core Observation ADI Documentation](http://hl7.org/fhir/us/core/StructureDefinition/us-core-observation-adi-documentation).
- Allow the profile to optionally reference a supporting document, such as
  the LOINC code for *Immunization record* ([105580-5](https://loinc.org/105580-5/)).
- OR evaluate whether 105580-5 could be used in place of 11370-4, pending
  guidance from TI/LOINC on whether it is acceptable to use a DOC code as
  an observation code. If acceptable, the profile would be significantly
  more meaningful than its current design — conceptually it would then
  indicate that a US Public Health patient has a certificate, does not have
  a certificate, that the status is unknown, or that the patient was not asked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure the `us-ph-vaccine-credential-patient-assertion` profile in the
US Public Health Profiles Library to model **possession of a vaccine
credential** rather than a generic immunization-status assertion. Concretely:

- Change `Observation.code` from a fixed/required binding to LOINC
  **11370-4** to a binding that uses a code intended to represent a vaccine
  credential. As a near-term step, bind to LOINC **105580-5 (Immunization
  certificate)**, pending TI confirmation that a LOINC DOC code may be
  used in `Observation.code`. As a longer-term step, submit a new LOINC
  request for a purpose-built *Vaccine Credential — asserted* code and
  switch the binding once it exists.
- Keep `Observation.value[x]` as `CodeableConcept` with the existing
  Yes / No / Unknown / Not Asked answer set so the profile can record
  whether the patient *has* a credential, *does not have* one, the status
  is *unknown*, or the patient was *not asked*.
- Add an optional `Observation.derivedFrom` (or `.focus` /
  `.hasMember`) reference to a `DocumentReference` (or similar) that
  carries the credential itself (paper certificate, SMART Health Card,
  etc.), modeled after
  [US Core Observation ADI Documentation](http://hl7.org/fhir/us/core/StructureDefinition/us-core-observation-adi-documentation).
- Update the profile narrative, examples, and any conformance statements
  in the ph-library IG to reflect the new semantics, and call out the
  break from the prior STU2 modeling in the Change Log.

#### Justification

The reporter's analysis is technically correct: LOINC 11370-4's intent
and answer set describe an *immunization status assertion*, not the
existence of a credential document. The proposed approach mirrors a
well-established US Core pattern (ADI Documentation) where an
`Observation` records that a document exists and an optional reference
points at the document itself, which is the standard FHIR way to model
"this patient has a *thing*" without conflating the assertion with the
artifact. The `lynn_laakso` comment confirms the WG explicitly chose to
defer changes "for the next release" and re-opened the ticket post-STU2
publication, so a substantive remodel is on-scope for the next ballot
cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `Observation.code` bound to LOINC **11370-4** but **rename and
re-scope the profile** to match what it actually models, and address
the credential use case with a separate, additive artifact:

- Rename the existing profile to something like
  `us-ph-immunization-status-assertion` (or update only the `title`,
  `description`, and narrative if a URL change is undesirable for
  STU2 implementers) so it accurately describes "patient/provider
  asserted an immunization status" rather than "vaccine credential".
- Introduce a **new** companion profile,
  `us-ph-vaccine-credential-documentation`, that:
  - Constrains `Observation.code` to LOINC **105580-5** (or a future
    purpose-built code) — gated on TI guidance regarding LOINC DOC
    codes in `Observation.code`.
  - Requires (or strongly recommends) `Observation.derivedFrom` →
    `DocumentReference`, again following the US Core ADI Documentation
    pattern.
- Cross-link the two profiles in narrative and add usage guidance so
  implementers pick the one that matches their intent.

#### Justification

This option preserves backwards compatibility for any STU2 implementers
who are already populating the existing profile (and the underlying
11370-4 code) and avoids the risk of TI/LOINC pushing back on the
DOC-code-as-Observation.code question for the *single* artifact in
US PH. Splitting the two concerns into two profiles makes the modeling
crisper than Disposition A, which overloads one profile with an
"either-or" binding strategy. The trade-off is added IG surface area
and a potential renaming churn for the existing profile, plus two
artifacts to maintain instead of one.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep the
`us-ph-vaccine-credential-patient-assertion` profile as published in
STU2, on the basis that the profile is intentionally an *assertion*
profile and the semantic mismatch between its name and its binding is
better resolved by a documentation/erratum clarifying that the profile
captures a self-/provider-asserted vaccination status, not possession
of a credential document. Direct implementers who need true credential
documentation to use a `DocumentReference` directly (out of scope for
this profile) and revisit only if TI/LOINC publish a purpose-built
*Vaccine Credential* code.

#### Justification

This is the lowest-cost path and avoids any risk of breaking STU2
consumers. However, it is hard to defend on the merits: the reporter's
critique about LOINC 11370-4 not representing credential possession is
correct, and the Public Health WG already explicitly reverted the
"Considered for Future Use" resolution and re-opened this ticket
"post-STU2 publication" with a stated intent to revisit in the next
release. Picking C now would directly contradict that WG decision and
would only be defensible if the next release is so close that there is
no realistic engineering window, or if TI/LOINC guidance comes back
strongly negative on every alternative code option.

---

### Recommendation

**Recommended disposition:** B

Disposition B (rename/clarify the existing profile and add a separate,
additive `us-ph-vaccine-credential-documentation` profile) is the most
defensible path. It directly addresses the reporter's core concern —
that the current artifact's *name* and its *binding* mean different
things — without forcing a hard rebinding of an already-published STU2
profile, which Disposition A would do. It also keeps the dependency on
TI/LOINC guidance scoped to a single, new, optional artifact (the
documentation profile), so the rest of the US PH library does not block
on that conversation. Disposition C is at odds with the WG's
documented decision (lynn_laakso, 2026-02-17) to reopen this for the
next release and would leave a known semantic defect in place. The WG
should decide explicitly whether the existing profile's URL/id can be
renamed or whether only its title/description/narrative will change;
either way, the substantive remediation lives in the new companion
profile.
