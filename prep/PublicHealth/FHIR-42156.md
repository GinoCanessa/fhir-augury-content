## Summary

This ticket records feedback from Nutrition subject-matter experts on the
US Bidirectional Services eReferrals (BSeR) implementation guide: the
SMEs support sending the `ServiceRequest` resource **in addition to** the
`Task` resource as part of the BSeR "Notify of Referral" exchange (see
the `BSeR_NOTIFY_SEND_CONTENTS` label, which scopes the request to the
contents of that step). Today BSeR's notify step is centred on `Task`,
and the ticket asks the work group to broaden that payload so the
referral's clinical intent — captured in `ServiceRequest` — travels with
the `Task` rather than being implied by it. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were surfaced for this ticket;
the request is self-contained feedback rather than part of a broader
discussion thread tracked in Augury. The Public Health work group has
already acknowledged the feedback in the resolution and committed to
considering sending both `Task` and `ServiceRequest` for the Notify of
Referral step.

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR IG so that the **Notify of Referral** message payload
includes a `ServiceRequest` instance alongside the existing `Task`
instance. Concretely:

- Extend the BSeR Notify-of-Referral message Bundle profile (and the
  associated MessageDefinition / "send contents" guidance the
  `BSeR_NOTIFY_SEND_CONTENTS` label points at) so that a
  BSeR-`ServiceRequest` profile entry is required (cardinality `1..1`)
  in addition to the BSeR-`Task`.
- Require the `Task.focus` reference on the BSeR-`Task` to resolve to
  that bundled `ServiceRequest`, so the Task's clinical intent is
  explicit and machine-resolvable rather than implied.
- Reuse the existing nutrition-referral `ServiceRequest` profile (or
  add one if BSeR currently lacks a discrete profile for the Notify
  step) with the same `code`, `subject`, `requester`, `performer`, and
  `reasonCode` slices already defined for the Send-Referral step, so
  that the Notify payload is a faithful restatement of what was sent.
- Update the corresponding examples, narrative, and "Send Contents"
  table in the IG to show the two-resource payload.

#### Justification

This matches the feedback verbatim and the work group's own resolution
("we will consider sending both the Task and Service Request for the
Notify of Referral step"). It also aligns with the prevailing FHIR
workflow pattern: `Task` carries the *state of work* and `ServiceRequest`
carries the *thing being requested*; bundling both is how the rest of
the FHIR workflow module models a referral notification, and it lets the
receiver render the clinical request without dereferencing back to the
sender. For Nutrition SMEs specifically, the `ServiceRequest` is where
diet/nutrition-counselling intent, indications, and `reasonCode` live,
so omitting it forces receivers to reconstruct clinical context from
`Task` extensions that were never designed to carry it.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of *requiring* a bundled `ServiceRequest`, **make it
conditionally required and resolvable by reference**:

- Keep `Task` as the mandatory anchor of the Notify-of-Referral message.
- Define `Task.focus` on the BSeR-`Task` profile as `1..1` and typed to
  the BSeR-`ServiceRequest` profile.
- Specify that `Task.focus` MAY be either (a) a contained/bundled
  `ServiceRequest` (the Disposition A behaviour) **or** (b) an absolute
  URL to a `ServiceRequest` already exchanged in the prior Send-Referral
  step, and require senders to use the bundled form when the receiver
  has not previously received the `ServiceRequest`.
- Add a "MUST support" expectation on receivers to resolve `Task.focus`
  in either form.

#### Justification

Many BSeR exchanges happen between partners that have already exchanged
the `ServiceRequest` during the Send-Referral step; in those cases
re-sending the full resource in every Notify message duplicates payload
and risks divergence if the sender mutates the copy. The conditional /
referenced form preserves the SMEs' underlying need (the receiver can
always *get to* the `ServiceRequest`) while keeping payloads small and
authoritative. It is also more forgiving for early implementers who may
not yet have a BSeR-`ServiceRequest` profile wired into their Notify
pipeline. The trade-off vs. Disposition A is added implementation
optionality, which the IG can constrain with capability-statement
expectations.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep the Notify-of-Referral payload `Task`-only,
documenting that the original `ServiceRequest` is already available to
the receiver from the Send-Referral step and that `Task.basedOn` /
`Task.focus` references are sufficient to dereference it.

#### Justification

The BSeR workflow already exchanges the `ServiceRequest` in the
Send-Referral step; the Notify step is, by design, a lightweight status
update keyed on `Task`. Re-sending the `ServiceRequest` in every Notify
message risks the two copies drifting (different `meta.versionId`,
different `status`, different `note`s), and forces all receivers — not
just nutrition use cases — to ingest a resource they already have.
Declining preserves the message's narrow contract and avoids a breaking
change for existing BSeR implementers; the SMEs' need can be met by
clarifying how to dereference the existing `ServiceRequest` rather than
by adding it to the bundle.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible middle path. The Nutrition SMEs'
underlying need is for the receiver to have direct, unambiguous access
to the clinical intent at Notify time, not specifically to receive a
duplicate `ServiceRequest` on every notification. Pinning
`Task.focus` to a BSeR-`ServiceRequest` profile and allowing either a
bundled instance or a resolvable reference satisfies the SMEs (the
information is always reachable), avoids the duplication and drift
concerns that motivate Disposition C, and lets the IG keep a single
profile for the resource without over-constraining payloads. Disposition
A is a reasonable fallback if the work group concludes that many BSeR
exchanges are stateless or cross trust boundaries where the Notify
receiver is not the same actor that received the Send-Referral; in that
case the bundled-only form is operationally simpler. Disposition C
should only be chosen if the work group concludes the SME feedback is
already addressed by existing dereferencing guidance — but the ticket's
resolution text ("we will consider sending both") indicates the group
itself does not view the status quo as sufficient.
