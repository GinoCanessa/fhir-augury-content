## Summary

The reporter (Vassil Peytchev) raises a design concern about the BSeR
implementation guide's use of batch `Bundle`s to query for "supporting
information" needed to complete an eReferral. He observes that this
batch-bundle-as-query mechanism is a novel/custom pattern within FHIR
and suggests it would be better served by FHIR's standard
publish/subscribe machinery — specifically by defining BSeR-specific
`SubscriptionTopic`s and notification shapes — which is a less bespoke,
more idiomatic FHIR mechanism for the same data-pull workflow. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
found that elaborate on this concern, so the request stands on its
own.

### Disposition A: Accept As Requested

#### Proposal

Replace the BSeR "supporting information" batch-Bundle query mechanism
with a FHIR Subscriptions–based pattern. Concretely:

1. Define one or more BSeR `SubscriptionTopic` resources covering the
   supporting-information classes the receiver currently asks for via
   batch Bundle (e.g., relevant `Observation`, `Condition`,
   `MedicationStatement`, `DocumentReference`, `Encounter` instances
   tied to the referral subject).
2. Define the notification shape (the `Bundle` produced by the
   `$status`/notification delivery) so receivers know exactly what
   resources and references to expect.
3. Deprecate the batch-Bundle query construct in the next BSeR
   publication, retaining it only as a transitional/legacy option with
   a migration note.
4. Update the BSeR conformance/CapabilityStatements to require
   Subscriptions (R5 Subscriptions, or R4/R4B Backport Subscriptions
   IG, depending on the BSeR target version) on the data-holder side.

#### Justification

This aligns BSeR with the FHIR-standard mechanism for "tell me when
data of shape X exists/changes," which is precisely the semantic the
batch-Bundle query is emulating. Using `SubscriptionTopic` plus a
defined notification `Bundle` reduces the implementation surface
specific to BSeR, lets implementers reuse generic Subscriptions
tooling, and benefits from ongoing FHIR core work on Subscriptions
rather than maintaining a bespoke query convention. It also addresses
the reporter's concern directly: the batch-Bundle pattern is novel and
not used elsewhere in FHIR, so divergence from the rest of the
ecosystem creates an avoidable interoperability and tooling tax.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the batch-Bundle query mechanism, but formally constrain and
document it as the "BSeR Supporting Information Query" pattern, and
publish a parallel, normative-equivalent Subscriptions binding for
implementers who prefer push semantics. Specifically:

1. Profile the request `Bundle` (`Bundle.type = batch`) and the
   response `Bundle` (`Bundle.type = batch-response`) with explicit
   slicing on `Bundle.entry.request` so the allowed queries are
   enumerated, not free-form.
2. Add a non-normative section (or companion page) showing the
   equivalent `SubscriptionTopic` definitions and notification
   `Bundle` shapes, marked as an implementer option.
3. Add CapabilityStatement guidance describing which interaction
   (batch query vs. subscription notification) a server claims to
   support, so a referring system can negotiate.

#### Justification

Replacing the batch-Bundle mechanism outright (Disposition A) is a
breaking change for any BSeR pilot or production deployment that has
already implemented it, and Subscriptions adoption is uneven across
EHRs and HIEs — particularly the R5-style `SubscriptionTopic`
machinery. A two-track approach preserves existing investment, gives
implementers a documented migration path, and lets the workgroup
collect real deployment data on which mechanism wins before forcing a
single answer. It also addresses the spirit of the reporter's concern
— that the batch-Bundle pattern is under-specified — by tightening the
profile rather than removing the construct.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the current BSeR cycle.
Document the rationale: the batch-Bundle query mechanism is an
intentional design choice motivated by the synchronous,
request-response workflow BSeR targets between the referring and
receiving systems, and switching to a Subscriptions model would be a
major redesign requiring re-balloting and would shift the interaction
pattern from pull to push. Optionally re-file as a forward-looking
issue against the next major version of BSeR.

#### Justification

The reporter raises a design preference rather than identifying a
defect or interoperability blocker, and Subscriptions (especially
`SubscriptionTopic`-based R5 Subscriptions) impose substantial server
infrastructure requirements that not all BSeR participants are
positioned to meet. If the existing batch-Bundle mechanism is working
in pilots, churning it now risks destabilising BSeR adoption without
clear evidence the alternative is better in this specific workflow.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying observation is correct: a custom
batch-Bundle query convention is harder to justify than reusing FHIR's
standard pub/sub plumbing, and the current ticket has been "Triaged"
at "Highest" priority for a long time precisely because that concern
is real. However, jumping straight to Disposition A (rip-and-replace
with `SubscriptionTopic`) is likely too aggressive for a deployed
implementation guide whose implementers have already built against
the batch-Bundle pattern, and Subscriptions support across the BSeR
implementer community is still maturing. Disposition B threads that
needle: it tightens the existing mechanism (addressing the "novel and
under-specified" critique), publishes the Subscriptions-based
equivalent so the migration path is visible, and lets the workgroup
gather adoption signal before committing to deprecation. If/when
Subscriptions adoption proves out, a future ticket can promote the
notification path to required and deprecate the batch-Bundle path,
landing where Disposition A would have started.
