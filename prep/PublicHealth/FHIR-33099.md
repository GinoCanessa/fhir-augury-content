## Summary

David Pyke filed FHIR-33099 against the SANER (Situation Awareness for Novel
Epidemic Response) implementation guide, reporting that an example was missing
for the "Measure Expectation" content in the IG. The absence of an example was
producing a warning in the IG QA build, which the work group needed to clear

and a follow-up comment from kwboone (Keith Boone) on 2021-08-23 stating the
work was "Completed". There are no cross-referenced Jira tickets, Zulip

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a conformant example resource for the "Measure Expectation" content in the
SANER IG so the IG publisher QA warning is cleared. Concretely, author a JSON
(and matching XML) example illustrating the use of the `Measure` extension /
guidance referenced on the "Measure Expectation" page — most likely a
fully-populated `Measure` instance demonstrating the expectation pattern as
described — register it in the IG `ImplementationGuide` resource (or the
sushi-config `instances` list, depending on the IG's tooling), and reference
it from the page so the example narrative renders. Re-run the IG build and
confirm the QA warning is gone.

#### Justification

The ticket is a pure publication-quality fix: the IG is already shipping the
guidance, it just lacks the illustrative example needed to satisfy the IG
publisher's "every page should have an example" QA rule. Adding the example
is consistent with HL7 IG authoring practice, requires no normative or
conformance changes, has no implementer impact beyond providing clearer
guidance, and is exactly what the resolution comment on the ticket already
indicates was done ("Completed" — kwboone, 2021-08-23).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a single example, suppress the QA warning at its source by
either (a) marking the section as not requiring an example via the IG
publisher's standard suppression mechanism (e.g., adding the page/artifact to
the appropriate `qa-warnings.txt` / `ignoreWarnings.txt` allowlist with a
documented rationale), or (b) restructuring the "Measure Expectation"
content so it is presented as conformance text on an existing example-bearing
artifact (e.g., embedded into the guidance for an existing example `Measure`)
rather than its own page. Either variant clears the QA warning without
requiring the authoring team to invent an additional standalone example.

#### Justification

This avoids creating example artifacts that exist only to satisfy tooling
rather than to aid implementers. It also reduces long-term maintenance
burden — every example must be kept in sync with the model and re-validated
on each build. Suppression is appropriate when the guidance is genuinely
abstract and a manufactured example would not add real value. Trade-off vs.
Disposition A: implementers lose a concrete worked example, which the SANER
audience (rapidly mobilized public-health implementers) tends to rely on
heavily, and suppressing QA warnings sets a precedent that erodes the
publisher's quality signal.

---

### Disposition C: Decline

#### Proposal

Decline to act on the ticket on the grounds that the QA warning is cosmetic
and the SANER IG can publish with the warning visible, with no normative or
implementer-facing impact. Close the ticket as "Not Persuasive" with a note
that the QA finding is acknowledged but not blocking publication.

#### Justification

This would only be defensible if the SANER work group considered the QA
warning to be informational noise rather than a publication blocker, and if
no implementer had complained about missing examples for Measure Expectation.
In practice, HL7 publication rules treat unaddressed QA warnings as a quality
signal against the IG, and the reporter is an experienced HL7 contributor
explicitly asking for the example. Declining would also be inconsistent with
the work group's existing behavior on this ticket (already resolved
Persuasive and marked Completed).

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right outcome and is in fact what the work group already
did — the ticket is marked **Persuasive**, the resolution is "Create example",
and Keith Boone's 2021-08-23 comment confirms the work was completed. The
request is small, editorial, and improves implementer guidance in a way that
is fully aligned with HL7 IG authoring norms. Disposition B (suppression /
restructuring) is a viable fallback only when authoring an example is
genuinely impractical, which is not the case here. Disposition C (decline) is
not defensible for a public-health IG where worked examples materially help
implementers stand up surveillance measures quickly. The work group should
confirm the example landed in the published SANER build, then close the
ticket as Resolved.
