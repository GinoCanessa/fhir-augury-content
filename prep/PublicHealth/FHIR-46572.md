## Summary

[FHIR-46572](https://jira.hl7.org/browse/FHIR-46572) — *Managing Data
Discrepancies* — is a Public Health / VRDR change request flagging that
several fields in a death record can be sourced from more than one
party (funeral home, attending/certifying provider, and/or
Medical Examiner / Coroner), and that VRDR may need an explicit
mechanism for representing or reconciling discrepancies between those
sources. The ticket was deliberately left open at the end of the STU3
cycle as a "breadcrumb" so the issue gets revisited during STU4. It is
currently `Triaged`, `Unresolved`, carries the `Block-Vote-2` label,
and was previously dispositioned `Considered for Future Use`
(SKravitz/Forest, 36-0-0) before being reverted by Lynn Laakso on
2024-10-16 with the note that it must be reconsidered during STU4.

The ticket has no formal cross-references in Augury, but its
keyword-nearest neighbour is
[FHIR-46571](https://jira.hl7.org/browse/FHIR-46571) — *Additional Use
Cases for Funeral Homes* — another Saul Kravitz STU3-end breadcrumb
that explicitly expands funeral-home interaction use cases in VRDR.
That ticket is the natural companion: 46571 enlarges *who* contributes
to a death record (more funeral-home flows), while 46572 asks how VRDR
should behave *when* those multiple contributors disagree on the same
field. Other neighbouring tickets ([FHIR-44782](https://jira.hl7.org/browse/FHIR-44782)
*Support Funeral Director within VRDR* and
[FHIR-33715](https://jira.hl7.org/browse/FHIR-33715) *Source Record
type (MFIELD) is missing*) reinforce that source-of-record provenance
is an active VRDR concern.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an explicit "data discrepancy / multi-source attribution" mechanism
to VRDR for STU4. Concretely, for the small, named set of
death-record fields that can legitimately be reported by more than one
party (e.g., decedent demographics, date/time/place of death, cause /
manner of death contributions, disposition method/place), introduce a
standard pattern that records *which* party contributed *which* value
and how to flag a disagreement. The most VRDR-idiomatic shape is:

1. Profile a `Provenance` resource per contribution, scoped to the
   target Observation/Composition section, with
   `Provenance.agent.who` distinguishing funeral home, attending /
   certifying provider, and ME/C; and `Provenance.entity.what`
   pointing at the specific Observation or Composition section the
   value populates.
2. Add a "discrepancy" extension (e.g.,
   `vrdr-data-discrepancy`) on the affected Observation /
   Composition.section that carries a coded reason
   (`agreed | disagreed-superseded | disagreed-unresolved`), a
   reference to the alternate Provenance(s), and an optional free-text
   reconciliation note.
3. Mark the affected fields in the IG narrative ("Managing Data
   Discrepancies" page) listing the canonical source-precedence rules
   (NCHS / state-vital-records guidance) so implementers have a
   default reconciliation order even when the discrepancy extension is
   absent.

#### Justification

This stays inside the existing VRDR information model (Observations
inside `DeathCertificateDocument`) and reuses the FHIR base-spec
mechanisms that already exist for exactly this problem
(`Provenance` for attribution, an extension for the "two values
disagree" flag). It directly answers Saul's STU3 breadcrumb, addresses
the multi-contributor reality FHIR-46571 will magnify, and gives
implementers a deterministic reconciliation rule rather than letting
each EDRS invent its own. It is also the path most consistent with the
adjacent provenance/source-of-record work in FHIR-44782 and
FHIR-33715.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a discrepancy primitive to VRDR. Instead, declare the
death record one-writer-per-field by policy and push reconciliation
out of FHIR:

- In the IG narrative, designate a *single* authoritative source per
  field (the existing NCHS field-ownership tables can drive this),
  e.g., funeral-home owns demographics, certifier owns cause of death,
  ME/C owns manner-of-death where applicable.
- Where a non-authoritative party still wants to record what *they*
  observed, capture it as a separate Observation profile (e.g.,
  `Observation-FuneralHome-DecedentName-AsReceived`) rather than as a
  competing value on the canonical Decedent / certificate field.
- Use `Bundle.entry.request` semantics on inbound submissions so the
  EDRS can reject (or down-grade to a "received-but-not-authoritative"
  Observation) any submission that overwrites an authoritative field.

#### Justification

Trades modelling complexity for governance simplicity. It avoids
introducing a new VRDR-specific extension and avoids the
"two-source-of-truth" anti-pattern in EDRS storage. The drawback vs.
Disposition A is that it leaves no in-instance signal that two parties
*disagreed* — that information either gets lost or has to be inferred
from a parallel `Observation-AsReceived` resource — so analytics and
audit tooling become harder. It also pre-supposes that NCHS field
ownership is settled enough to encode in the IG, which the workgroup
should confirm before committing.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Persuasive with Mod / Not Persuasive — already
addressed" on the basis that FHIR's existing `Provenance` resource
plus the Observation/Composition section model already lets an
implementer record who contributed which value, and that
reconciliation between competing values is an EDRS / jurisdictional
policy concern, not a VRDR conformance concern. Close
the ticket with a normative-narrative-only update pointing implementers
at `Provenance` and at jurisdictional source-precedence rules.

#### Justification

Defensible if the workgroup believes (a) no implementer is currently
blocked by the absence of a VRDR-specific discrepancy primitive, and
(b) standardising a reconciliation rule risks conflicting with state
vital-records statutes that already specify which source wins. Risk:
this is the path that was previously chosen ("Considered for Future
Use", 36-0-0) and was *explicitly reverted* by Lynn Laakso on
2024-10-16 so it would be reconsidered in STU4 — re-declining without
new substance would undo that signal.

---

### Recommendation

**Recommended disposition:** A

The ticket exists precisely because the prior "park it" disposition was
revoked so the workgroup would actually engage with the problem in
STU4. The companion ticket FHIR-46571 will, if accepted, *increase*
the rate at which multiple parties write to the same death-record
fields, which makes a deterministic, in-instance discrepancy
representation more — not less — necessary. Disposition A keeps the
work inside well-trodden FHIR machinery (`Provenance` plus a small
VRDR extension and a narrative reconciliation table) and avoids the
governance gamble in Disposition B (assuming NCHS field-ownership is
settled enough to hard-code) while still answering the original
question. Disposition C should only be revisited if the STU4 scoping
discussion shows that no implementer has hit the discrepancy case in
practice.
