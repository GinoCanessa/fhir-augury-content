## Summary

This ticket asserts that the canonical URL for the
`hypertensive-disorders-pregnancy` ValueSet defined in the Maternal and
Infant Health Research (MIHR) FHIR IG is anchored at
`http://www.hl7.org/fhir/us/mihr/ValueSet/...` but, per TSMG guidance,
should instead be anchored under `terminology.hl7.org` (THO). The
reporter (relaying comment 5 from Ron G. Parker) further notes that
only ValueSets supporting a *required* binding to a `code` data type
should remain anchored under `hl7.org/fhir`; all other ValueSets
should be migrated to THO, with a UTG ticket created to register the
ValueSet there. No cross-referenced Jira tickets, Zulip discussions,
or GitHub items were found for this ticket — its context is entirely
internal to the MIHR IG and the broader TSMG/THO canonical-URL policy.

## Details

**Description:**

The Value Set canonical url should be anchored in terminology.hl7.org
(THO), not hl7.org/fhir. A UTG ticket should be created to register
the Value Set in THO.

Only those with required binding to a 'code' data type should be
anchored in hl7.org/FHIR.

Contact TSMG if you have further questions.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persistent For Next Version. Re-anchor the
`hypertensive-disorders-pregnancy` ValueSet so that its canonical URL
is `http://terminology.hl7.org/ValueSet/hypertensive-disorders-pregnancy`
(or an equivalent THO-anchored canonical chosen with TSMG), and update
all bindings within the MIHR IG that reference the old MIHR-anchored
URL to point at the new THO canonical. File a companion UTG ticket to
register the ValueSet (and its associated CodeSystem definitions, if
any are MIHR-defined) in THO. Audit the rest of the MIHR IG's
ValueSets and apply the same treatment to every ValueSet that is *not*
the target of a required binding to a `code` data type, leaving only
those required-binding-to-`code` ValueSets anchored under
`hl7.org/fhir/us/mihr`.

#### Justification

This is the disposition that literally implements TSMG's
canonical-URL policy as relayed in the ticket: terminology artifacts
belong in THO, and only the narrow class of ValueSets bound `required`
to a `code` data type stays under the IG's own canonical root. The
`lynn_laakso` 2023-03-29 comments explicitly re-open the ticket "post
STU publication" and revert the earlier "Considered for Future Use"
resolution, which had been parked pending IG maturity and the
finalization of the draft THO process — both of those gating
conditions have since been resolved, so the original action can now
be executed. Doing the work as a single, IG-wide sweep (rather than
artifact-by-artifact) is also more consistent with how MIHR's sibling
Public Health IGs have handled the same TSMG guidance.

---

### Disposition B: Alternative Approach

#### Proposal

Persistent For Next Version, but scope the change narrowly. For the
next MIHR publication, re-anchor *only* the
`hypertensive-disorders-pregnancy` ValueSet to THO and file the
corresponding UTG registration ticket. Do not perform an IG-wide
sweep in this cycle; instead, open a separate tracker ticket against
the MIHR IG (with `Terminology` and `Block-Vote-1` labels) listing
every other ValueSet that needs the same treatment, to be addressed
in a subsequent ballot cycle. Add a NamingSystem / `OtherCanonical`
mapping or an explicit retirement notice on the old MIHR-anchored
URL so existing implementations that resolved the old canonical are
not silently broken.

#### Justification

The ticket as written calls out one specific ValueSet and one
specific URL; treating that ValueSet as the only in-scope artifact
keeps the change request and its block-vote disposition tightly
aligned with what was actually filed and reviewed by the workgroup.
The sweep contemplated in Disposition A is real work, but it touches
canonical URLs across the IG and therefore carries non-trivial
implementer-impact risk; deferring it to a dedicated ticket lets the
workgroup weigh that impact separately rather than rolling it into a
"Low" priority block-vote item. The explicit
deprecation/`OtherCanonical` step is added because moving canonicals
without a forwarding signal is the well-known
breaking-change failure mode of THO migrations.

---

### Disposition C: Decline (Persuasive)

#### Proposal

Persuasive — Not Persuasive. Decline the change on the grounds that
the MIHR-anchored canonical
`http://www.hl7.org/fhir/us/mihr/ValueSet/hypertensive-disorders-pregnancy`
has already been published in an STU release of the MIHR IG and is
in active use by implementers; changing the canonical URL post-STU
would silently break every conformance resource and instance that
references the existing URL. Recommend instead that the ValueSet
retain its current MIHR canonical and that any future MIHR-defined
terminology be authored in THO from inception, governed by a
forward-looking IG-level policy rather than a retroactive rename.

#### Justification

Canonical URLs in FHIR are deliberately stable identifiers; a
ValueSet's canonical is what bindings, profiles, conformance
resources, and instance `Coding.system` / membership-check logic
resolve against. Renaming a published canonical is a breaking change
to every downstream consumer, and TSMG's "anchor in THO" guidance
is generally applied prospectively rather than as a retroactive
rewrite of already-balloted IG content. The original 2022 disposition
(*Considered for Future Use*) reflected exactly this concern, and the
2023-03-29 re-open does not, on its face, introduce new evidence that
the breaking-change risk has been mitigated.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible balance of three competing
pressures: (1) TSMG's clear policy direction, which the workgroup has
already accepted in principle by re-opening the ticket post-STU; (2)
the very real implementer-impact cost of moving a published canonical,
which Disposition C correctly identifies but which is more
appropriately *managed* (via deprecation / `OtherCanonical`) than used
as a reason to decline outright; and (3) the scope-creep risk of
Disposition A, which converts a single-ValueSet ticket into an IG-wide
terminology migration without the workgroup having explicitly weighed
the broader impact. Filing the UTG registration and re-anchoring the
named ValueSet honours the original request and the TSMG guidance,
while spinning the IG-wide audit into a dedicated tracker keeps that
larger conversation visible without forcing it through a "Low" priority
block-vote slot.
