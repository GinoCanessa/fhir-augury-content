## Summary

The reporter (Hans Buitendijk, imported as Comment 7) flags that a section title
in the US Helios Query and Response (FHIR) IG, combined with copied wording
about TEFCA and QHINs, is ambiguous and could be read to imply that a QHIN
will always *broker* a FHIR query on behalf of a Public Health Agency (PHA).
The reporter argues this overstates what QHINs do today: QHINs facilitate
endpoint discovery (including via a Record Locator Service) and establish
trust so that the PHA can directly initiate the FHIR query itself. Brokering
is permitted but not guaranteed — not every QHIN offers it, and a brokered
FHIR query is not always the most appropriate model for PHA queries. The
ticket asks the work group to revise the section title and the existing
narrative to remove the implicit assumption of QHIN brokering and to describe
the QHIN role more accurately.

There are no linked Jira tickets, no cross-referenced Jira/Zulip/GitHub items,
and no related repository artifacts surfaced by Augury for this ticket — it
is a self-contained editorial / wording change request against the IG
narrative.

## Details

**Description:**

> The section title is confusing and in combination with the existing
> wording copied it may imply that a QHIN always will broker a FHIR query.
> Rather, the QHINs facilitate the discovery of the endpoint (with a record
> locator service as needed) and establish trust so the PHA can directly
> initiate a query. There is no requirement these be brokered by the QHIN.
> It is reasonable that another party could do so, and a QHIN may offer it.
> However, that can currently not be expected from every QHIN, nor may a
> brokered FHIR query turn out to be the most appropriate for PHA queries.
>
> **Existing Wording:**
>
> Under TEFCA, QHINs will provide a single point of connection for PHAs to
> reach out to the nationwide network of responders. QHINs help PHAs use
> FHIR to connect to a network of networks, Record Locator Services and a
> Directory of Requestors/Responders, following a federally-supported set
> of standards.
>
> *(Comment 7 - imported by: Hans Buitendijk)*

**Resolution (already recorded):**

> Changes made as requested. Added clarifying text:
> *"Under TEFCA, QHINs may provide a single point of connection for PHAs to
> reach out to the nationwide network of responders to facilitate the
> discovery of the endpoint with a record locator service, and a Directory
> of Requestors/Responders, and establish trust so the PHA can directly
> initiate a query, while following a federally supported set of standards."*

## Proposed Dispositions

> Note: This ticket has already been marked **Persuasive with Modification**
> by the work group, with the resolution text recorded above. The
> dispositions below are framed as if reviewing the request fresh, so the
> work group can confirm (or revisit) that decision.

### Disposition A: Accept As Requested

#### Proposal

Revise the section header and the TEFCA/QHIN narrative paragraph in the
US Helios Query and Response (FHIR) IG to remove the implication that a
QHIN always brokers the FHIR query. Specifically:

- Reword the section header so it does not imply a QHIN-brokered query
  pattern (e.g., from "QHIN-brokered FHIR Query" / similar wording to
  "Using TEFCA / QHINs to support PHA FHIR Queries" or equivalent neutral
  phrasing that covers both directly-initiated and (optionally) brokered
  flows).
- Replace the existing paragraph with the resolution text already
  recorded on the ticket:
  *"Under TEFCA, QHINs may provide a single point of connection for
  PHAs to reach out to the nationwide network of responders to facilitate
  the discovery of the endpoint with a record locator service, and a
  Directory of Requestors/Responders, and establish trust so the PHA can
  directly initiate a query, while following a federally supported set of
  standards."*

This is a narrative-only change against the IG; no StructureDefinitions,
profiles, value sets, or operations are affected.

#### Justification

The reporter's concern is factual: TEFCA/QHIN responsibilities, as
currently published, center on trust, directory, and RLS services rather
than on brokering FHIR transactions. The original wording ("will provide
a single point of connection", "QHINs help PHAs use FHIR to connect to a
network of networks") reads as if the QHIN is the FHIR endpoint, which
will mislead implementers about whose system terminates the FHIR
exchange. Softening "will" to "may" and explicitly separating
*facilitation* (discovery + trust) from *initiation* (PHA-driven query)
matches how PHAs and QHINs actually operate today and is consistent with
the IG's stated goal of describing realistic PHA query patterns. Because
this is editorial, the change is low-risk and does not break any
conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the wording correction in Disposition A, **and** add a short
"QHIN role spectrum" paragraph or table immediately after the revised
sentence that enumerates the realistic QHIN involvement levels for a PHA
query:

1. QHIN provides trust + Directory only; PHA queries the responder's
   FHIR endpoint directly.
2. QHIN additionally provides RLS / endpoint discovery; PHA still
   initiates the FHIR query directly.
3. QHIN optionally brokers the FHIR exchange (only where the QHIN
   offers it and both sides agree).

Cross-link this paragraph from the section header so future readers see
all three patterns are in scope.

#### Justification

The original wording's ambiguity is symptomatic of a deeper IG-level
problem: the spec does not currently disambiguate the QHIN involvement
levels. A simple wording fix (Disposition A) removes the misleading
implication but leaves implementers to infer the spectrum from prose.
Calling out the patterns explicitly costs little additional text, makes
the IG more useful as implementation guidance, and pre-empts the next
round of "does the QHIN proxy this request?" tickets. The trade-off is a
slightly larger editorial change and a small risk of having to revise
the enumeration as TEFCA evolves.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the existing wording, on the basis that
the current paragraph paraphrases TEFCA/QHIN materials and is intended
as a high-level orientation rather than a normative description of the
QHIN's role in any specific FHIR exchange.

#### Justification

This would only be defensible if the work group considered the section
to be unambiguously aspirational or if the section already contained
nearby qualifying text that made the QHIN role clear. Neither appears to
be the case here: the cited paragraph reads as a factual claim about
QHIN behavior ("will provide a single point of connection"), not as
aspiration, and the reporter — an active TEFCA/PHA participant — found
it misleading. Declining would leave a known wording defect in a
balloted IG and risk implementer confusion about who terminates the
FHIR query.

---

### Recommendation

**Recommended disposition:** A

The work group has already adopted Disposition A in substance ("Persuasive
with Modification"), and the recorded resolution text cleanly addresses
the reporter's concern: it changes "will" to "may", separates discovery /
trust from query initiation, and explicitly states that the PHA can
*directly* initiate the query. That is exactly the disambiguation the
reporter asked for, with minimal editorial blast radius and no
conformance impact. Disposition B is attractive as a follow-up — adding
an explicit QHIN-role spectrum would further harden the IG against
future ambiguity tickets — but it should be tracked as a separate change
request rather than expanding the scope of this block-vote item.
Disposition C is not defensible given that the existing wording is
demonstrably misleading about QHIN responsibilities.
