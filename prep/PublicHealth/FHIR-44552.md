## Summary

The reporter requests that the US Birth and Fetal Death Reporting (BFDR)
FHIR Implementation Guide update its dependency so that its profiles derive
from **US Core 6.1.0** instead of the currently-referenced **US Core 5.0.1**.
The stated rationale is regulatory alignment: US Core 6.1.0 is the version
named in ONC's HTI-1 final rule, so deriving BFDR profiles from 5.0.1 leaves
BFDR out of step with the version implementers are being required to adopt
for certified health IT. The ticket was originally dispositioned "Considered
for Future Use" in March 2024 (vote 20-0-0, Passas/Kravitz) on the grounds
that "there seems to be little advantage for vital records to be at the
bleeding edge of US Core implementation." That resolution was subsequently
reverted and the ticket re-opened post-STU2 publication (2024-10-15,
lynn_laakso) so Public Health can reconsider it in the context of a future
BFDR release.

## Proposed Dispositions

Because the cross-reference service returned no linked GitHub repos for
this ticket, the dispositions below name the BFDR IG at the
specification level (its published dependency on US Core) rather than a
specific briefing-backed authoring root. If Public Health elects to act
on this ticket, the assignee should confirm the exact authoring file in
the BFDR IG source repo (typically the IG resource / `sushi-config.yaml`
`dependencies` entry for `hl7.fhir.us.core`) before implementation.

### Disposition A: Accept As Requested

#### Proposal

In the next BFDR release, update the IG's US Core dependency from
`hl7.fhir.us.core` **5.0.1** to **6.1.0**. This means:

- Bump the `hl7.fhir.us.core` dependency declaration in the BFDR IG
  configuration (e.g., `sushi-config.yaml` / `ImplementationGuide`
  resource `dependsOn`) from `5.0.1` to `6.1.0`.
- Re-derive all BFDR profiles that declare a `parent` on a US Core
  profile (e.g., `us-core-patient`, `us-core-practitioner`,
  `us-core-observation-*`, `us-core-encounter`, `us-core-condition`) so
  they inherit from the 6.1.0 versions.
- Reconcile any constraints/slicing/must-support flags that differ
  between US Core 5.0.1 and 6.1.0 (notably around Patient race/ethnicity
  extensions, SDOH-related Observation profiles, and any elements whose
  bindings or cardinalities tightened in 6.1.0).
- Update examples and narrative to reference 6.1.0 where versions are
  stated.
- Update the IG's published dependency table and cross-version notes.

#### Justification

US Core 6.1.0 is the version named in ONC's HTI-1 final rule, which is
the regulatory baseline BFDR implementers (state vital records agencies
and their EHR partners) must meet. Keeping BFDR on US Core 5.0.1 forces
implementers to maintain two US Core versions concurrently — the one
their certified EHR is obligated to expose under HTI-1 (6.1.0), and the
one BFDR's profiles derive from (5.0.1) — which complicates validation,
data mapping, and terminology handling. Aligning the dependency also
avoids BFDR-specific slicing on elements whose US Core definitions have
since been adjusted in 6.1.0.

---

### Disposition B: Alternative Approach

#### Proposal

Keep US Core 5.0.1 as the declared derivation parent for the current
STU, and instead:

1. Add an explicit BFDR **companion guide / cross-version statement**
   documenting which BFDR profiles are compatible with US Core 6.1.0
   and listing any 5.0.1 → 6.1.0 deltas implementers should expect.
2. In the next BFDR ballot cycle, publish a **parallel variant** (or a
   separate minor release) of BFDR whose profiles derive from US Core
   6.1.0, so consumers can pick the variant that matches their
   certified-EHR baseline.
3. Pin the 6.1.0 derivation as the forward default once the next major
   BFDR version is under development.

#### Justification

The original workgroup vote (20-0-0 in March 2024) concluded that
vital-records systems should not be on the "bleeding edge" of US Core.
That concern — stability for state agencies running on older
infrastructure — has not gone away simply because HTI-1 now names
6.1.0. A parallel/companion approach preserves the 5.0.1 deployment
path for agencies not yet ready to move while giving HTI-1-aligned
implementers a supported 6.1.0 derivation, at the cost of the IG
maintenance burden of shipping two derivation lines. It also avoids
disrupting STU2, which was only just published when this ticket was
re-opened.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep BFDR derived from US Core 5.0.1 for the
current STU. Rationale category: **insufficient justification for
breaking-change churn to a just-published STU**; HTI-1 alignment is a
derivation-parent concern rather than a conformance blocker because
data produced against BFDR 5.0.1-derived profiles is still structurally
valid US Core data.

#### Justification

This matches the workgroup's earlier position (March 2024 vote, 20-0-0)
that vital-records implementations gain little from chasing the latest
US Core. STU2 has only just published; forcing a parent-version bump
now would invalidate freshly-deployed implementations, re-open every
profile for reconciliation review, and shift maintenance effort away
from outstanding BFDR-specific change requests. Implementers who need
HTI-1-aligned US Core data can continue to produce it on the wire; the
derivation parent version does not prevent that.

---

### Recommendation

**Recommended disposition:** A

HTI-1 has moved US Core 6.1.0 from "latest release" to "the regulatory
baseline," which materially changes the calculus the workgroup used in
March 2024. Vital-records systems are not being asked to ride a moving
edge — they are being asked to align with the single US Core version
that their certified EHR trading partners are now obligated to expose.
Leaving BFDR derived from 5.0.1 permanently desynchronises the IG from
its own ecosystem and pushes reconciliation work onto every
implementer. Disposition B preserves optionality but at a real
maintenance cost (two parallel derivation lines) that is hard to
justify for a small IG footprint, and Disposition C perpetuates the
mismatch. The workgroup should plan the 6.1.0 bump for the next BFDR
release (not a patch to STU2), scope the derivation delta explicitly,
and use the release notes to signal the HTI-1 alignment to
implementers.
