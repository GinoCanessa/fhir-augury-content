## Summary

This ticket, originally filed by Ron G. Parker against the Vital Records
Common Profiles Library (VRCL) FHIR IG, asserts that the IG defines (or
relies on) one or more code systems and/or value sets whose canonical URLs
are not anchored in `terminology.hl7.org` (THO). Per the TSC-approved
"Terminology Guidance for IG Authors" policy, such terminology artifacts
are required to live in THO rather than in IG-local namespaces. The
specific artifact at issue, surfaced in the most recent comment, is a
"Yes/No/Unknown"–style value set for which the workgroup previously voted
(23-0-0) that "creating a THO valueset named Yes/No/Unknown is
appropriate." The ticket asks the workgroup to start the UTG process so
this value set (and any peers in the same situation in VRCL) is migrated
into THO and referenced from the IG by its THO canonical.

The ticket is one of a numbered series of "Code Systems and Valuesets
should be anchored in THO (terminology.hl7.org). 31"-style requests
filed against various Public Health / Vital Records artifacts, so the
disposition here should be consistent with how its sibling tickets have
been (or will be) handled. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were returned by the augury index for this
key, so this prep is grounded only in the ticket itself and the cited
TSMG policy.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Open the corresponding UTG (Unified Terminology Governance) proposal to
move the affected value set — most directly the "Yes/No/Unknown" value
set called out in the workgroup's prior 23-0-0 vote — from its current
VRCL-local canonical into `terminology.hl7.org`. Concretely:

1. File a UTG ticket (or set of tickets) against the THO repo proposing
   a new THO-hosted value set (and any underlying code system, if one
   does not already exist in THO) for Yes/No/Unknown semantics, using
   the canonical pattern `http://terminology.hl7.org/ValueSet/<name>`.
2. Once accepted and published in a THO release, update the Vital
   Records Common Profiles Library to reference the new THO canonical
   in place of the IG-local URL on every binding currently pointing at
   the IG-local artifact.
3. Retire (or mark deprecated, redirecting to the THO canonical) the
   IG-local value set definition in VRCL.
4. Repeat the same migration for any other VRCL-local code
   systems/value sets that fail the THO-anchoring policy, so this
   ticket and its siblings in the "...31" series are resolved as a
   coherent batch.

#### Justification

This is the literal request of the ticket and is directly aligned with
the TSC-approved "Terminology Expectations for IG Developers" policy
cited in the description. The workgroup has already taken a 23-0-0
position that a THO-hosted Yes/No/Unknown value set is appropriate, so
the substantive question is settled — the only remaining work is
process (UTG proposal + VRCL re-binding). Reopening "post STU2
publication" implies the workgroup explicitly preserved this work for
the next publication cycle rather than declining it; Disposition A
honours that intent.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than migrating each VRCL-local artifact individually under its
own ticket, treat FHIR-44457 (and its visibly-numbered siblings) as a
single VRCL terminology-cleanup work item:

1. Inventory every code system and value set currently defined in VRCL
   whose canonical is not under `terminology.hl7.org`.
2. For each, classify as: (a) general-purpose / cross-IG (e.g.,
   Yes/No/Unknown) → migrate to THO via UTG; (b) genuinely
   VRCL-specific → keep in the IG namespace but document the rationale
   in the IG's terminology section so future audits do not re-flag it;
   (c) duplicates of existing THO content → retire and re-bind to the
   existing THO canonical with no new UTG submission needed.
3. Track the resulting work as a single epic with one UTG proposal per
   (a)/(c) artifact and a documentation PR for the (b) artifacts.
4. Resolve FHIR-44457 by pointing at that epic and the specific
   Yes/No/Unknown UTG proposal it spawns.

#### Justification

The ticket title and the existence of a numbered series ("...31")
strongly suggest a per-artifact ticket fan-out that is expensive to
review and easy to handle inconsistently. A single inventory-driven
plan lets the workgroup apply the TSMG policy uniformly, avoid creating
THO entries for genuinely VRCL-specific concepts, and produce a single
publication-ready terminology section for VRCL. It also preserves the
already-decided Yes/No/Unknown migration from Disposition A without
losing it in the reorganisation.

---

### Disposition C: Decline

#### Proposal

Decline the ticket with the rationale "already addressed by existing
mechanism / out of scope for VRCL changes": the TSMG policy is owned by
TSC/TSMG and is enforced through UTG, not through per-IG change
requests. The workgroup notes that the appropriate venue is a UTG
proposal against the THO repo, and closes FHIR-44457 as "Not Persuasive
- With Mod" pointing the requester at the UTG submission process.

#### Justification

This disposition would only be defensible if the workgroup intended to
take no action in VRCL at all and to push the entire matter to UTG /
THO. Given that VRCL must ultimately re-bind to whatever THO canonical
results, and given the prior 23-0-0 vote in favour of creating the THO
value set, this disposition is unlikely to reflect the workgroup's
actual position — but it is included for completeness as the
"do-nothing in VRCL" option.

---

### Recommendation

**Recommended disposition:** B

The substantive answer (move Yes/No/Unknown into THO) is already
settled by the prior 23-0-0 vote, so Disposition A's outcome is going
to happen regardless. However, the title — "Code Systems and Valuesets
should be anchored in THO ... 31" — and the explicit "re-open post
STU2 publication" comment together signal that this is one instance of
a structural cleanup, not a one-off binding fix. Handling it as a
VRCL-wide terminology audit (Disposition B) avoids 30+ near-duplicate
dispositions, lets the workgroup distinguish genuinely VRCL-specific
terminology from migration candidates, and produces a single coherent
plan for the next publication cycle. Disposition A remains the right
*mechanical* answer for the Yes/No/Unknown artifact and should be
preserved as a sub-task inside the Disposition B work plan.
