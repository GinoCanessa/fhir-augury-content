## Summary

The ticket asks the Public Health work group to consider disallowing
"aborted" medications from being reported through the Electronic Case
Reporting (eCR FHIR) IG. In practice this means tightening the
medication-related profiles in the eCR FHIR IG so that medication
resources whose lifecycle indicates the medication was never actually
administered or was abandoned (e.g., a `MedicationRequest` that was
`cancelled`/`entered-in-error`, a `MedicationAdministration` with
`status = not-done`, or a `MedicationStatement` that was
`entered-in-error`) are excluded from an eICR (electronic Initial Case
Report) bundle. There are no cross-referenced Jira tickets, Zulip
threads, or GitHub items linked to this ticket in Augury, so the
request is taken at face value from the title and description alone.

## Proposed Dispositions

Note: because no GitHub cross-references exist for this ticket, the
dispositions below name the FHIR resources, elements, and IG-level
mechanisms that would change but do not yet name a specific authoring
file path. Before implementation, the chosen disposition should be
re-grounded against the eCR FHIR IG repo briefing.

### Disposition A: Accept As Requested

#### Proposal

Add IG-level constraints to the eCR FHIR profiles for medications so
that "aborted" medication entries cannot be included in an eICR
bundle. Concretely:

- On the eCR `MedicationRequest` profile, add an invariant disallowing
  `MedicationRequest.status in {cancelled, entered-in-error}` (and
  optionally `stopped` if the working group considers it equivalent to
  "aborted" in this context).
- On the eCR `MedicationAdministration` profile, add an invariant
  disallowing `MedicationAdministration.status in {not-done,
  entered-in-error}`.
- On the eCR `MedicationStatement` profile (if used in eICR), add an
  invariant disallowing `MedicationStatement.status in
  {entered-in-error, not-taken}`.
- Update narrative guidance in the relevant eCR IG pages
  (Medication-related profile pages and the eICR composition guidance)
  to state that medications whose status indicates they were never
  given / were retracted MUST NOT be included in an eICR submission.

#### Justification

Including medications that were never actually administered (or that
were retracted) in a public-health case report risks misleading
downstream public-health agencies — both as evidence of treatment and
as potential triggers for reporting decision logic. FHIR already
provides the relevant status codes
(`MedicationRequest.status = cancelled | entered-in-error`,
`MedicationAdministration.status = not-done | entered-in-error`,
`MedicationStatement.status = entered-in-error | not-taken`), so the
change is a conformance tightening rather than a new modelling
problem. Encoding the rule as an invariant on the eCR profiles makes
the constraint machine-checkable and consistent with how the IG
already constrains other Must-Support medication elements.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of an outright prohibition via invariants, address the
underlying need with **narrative guidance plus a Bundle-level slicing
rule**:

- Keep the medication profiles permissive of all FHIR-defined status
  codes (so that a record system MAY hold an "aborted" medication
  conformantly), but
- Add guidance and a slicing/filter rule on the eICR Composition
  (and/or the eICR Bundle) that the `section.entry` references for
  medications MUST resolve to medication resources whose `status` is
  not in the "aborted" value set.
- Define a small intensional value set
  (`eICR-aborted-medication-status`) listing the codes that count as
  "aborted" per resource type, and reference that value set from the
  Composition guidance / a Bundle-level invariant.

#### Justification

This keeps the eCR medication profiles reusable by record systems that
legitimately need to represent every medication state on the source
record (including aborted ones) while still preventing aborted entries
from reaching the public-health receiver. It also localizes the rule
to a single, named value set, which is easier to evolve (e.g., adding
or removing a status code) than per-profile invariants spread across
three resource types. The trade-off vs. Disposition A is that the
constraint moves from the per-resource profile to the eICR
Composition / Bundle, which is slightly less obvious to implementers
reading a single profile in isolation.

---

### Disposition C: Decline

#### Proposal

Decline the request and document, in the eCR FHIR IG narrative, that
filtering out aborted medications is the responsibility of the
eICR-producing system rather than a conformance constraint on the
medication profiles. Rationale category: *already addressed by
existing FHIR mechanisms*.

#### Justification

Every relevant FHIR medication resource already carries a `status`
element with codes that explicitly cover the "aborted" cases
(`cancelled`, `stopped`, `not-done`, `entered-in-error`,
`not-taken`). Receivers can — and arguably should — filter on these
codes themselves, and senders can choose to omit aborted entries
without an IG-level prohibition. Encoding the prohibition as an
invariant risks rejecting otherwise-valid eICR bundles when a record
system legitimately includes an aborted medication for completeness
(for example, a stopped antibiotic that is clinically relevant to the
reportable condition). The ticket as filed ("Consider disallowing
aborted medications") does not include a use case showing concrete
public-health harm from the current behaviour, so a hard prohibition
may be premature.

---

### Recommendation

**Recommended disposition:** B

Disposition B best balances the public-health intent behind the
ticket — keep retracted / never-given medications out of submitted
eICRs — with the realities of source EHR data, where "aborted"
medications often exist for valid clinical reasons and where multiple
profiles (`MedicationRequest`, `MedicationAdministration`,
`MedicationStatement`) would otherwise need parallel invariants.
Putting the rule on the eICR Composition / Bundle and backing it with
a single named value set keeps the medication profiles reusable
outside eICR, makes the "what counts as aborted" decision explicit and
reviewable in one place, and avoids the over-rejection risk of
Disposition A. Disposition A remains a reasonable fallback if the
work group prefers per-profile machine-checkable invariants;
Disposition C should be reserved for the case where the work group
concludes the filtering decision genuinely belongs to senders /
receivers rather than the IG. Because no GitHub cross-references or
Zulip discussions are linked to this ticket in Augury, the work group
should also confirm during review whether there is prior off-ticket
discussion (e.g., on the eCR call) that should sharpen the
recommendation before implementation.
