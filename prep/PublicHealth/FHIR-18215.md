## Summary

The ticket asks whether the `Observation.valueQuantity` element on the
Body Height profile in the US Bidirectional Services eReferrals (BSeR)
implementation guide should be constrained to centimeters, as is
suggested by the UML expression for that profile. As filed, BSeR's Body
Height profile leaves the unit of measure unconstrained on
`valueQuantity`, even though the UML diagram shows centimeters as the
expected unit. The reporter (Craig Newman) is flagging this
inconsistency between the UML model and the computable profile and
asking whether the profile should be tightened so that conformant
instances are required to express height in centimeters (e.g., UCUM
`cm`).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR Body Height profile so that `Observation.valueQuantity`
is constrained to UCUM centimeters. Concretely, on the Body Height
profile (an `Observation` profile derived from / aligned with the FHIR
core `vitalsigns` / `bodyheight` profile), add the following bindings
on `Observation.valueQuantity`:

- `valueQuantity.system` fixed to `http://unitsofmeasure.org`
- `valueQuantity.code` fixed to `cm`
- `valueQuantity.unit` documented as `cm` (display only)

This brings the computable profile into alignment with the UML
expression that already shows centimeters as the intended unit.
Re-publish the BSeR IG with the tightened constraint and note the
change in the IG's change log so implementers know to send height in
centimeters specifically (rather than inches, meters, etc.).

#### Justification

The UML model is the human-readable contract for BSeR; if the UML
shows centimeters but the StructureDefinition allows any unit, then
two conformant systems can legitimately exchange Body Height values
that aren't directly comparable, which defeats the purpose of having
a profile at all. The FHIR core `bodyheight` profile already uses a
UCUM-bound `valueQuantity` and lists `cm` and `[in_i]` as the
allowed codes; BSeR is well within the spirit of FHIR profiling to
narrow that further to a single unit when its referral use cases only
need one. Locking to `cm` is also consistent with how most other
public-health-oriented profiles handle anthropometrics.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than fixing `valueQuantity.code` to `cm`, keep the FHIR core
`bodyheight` UCUM binding (which already restricts the unit to either
`cm` or `[in_i]`) and instead update the BSeR UML expression and
narrative to clarify that **either** centimeters **or** inches is
acceptable, with centimeters being the preferred / canonical
representation. Add an IG-level note (and, optionally, an invariant
or guidance paragraph) that receivers MUST be able to consume both
units and SHOULD store / forward in centimeters.

#### Justification

US clinical systems frequently capture height in inches, and forcing
senders to convert to centimeters at the edge introduces rounding
error and an extra implementation burden for what is fundamentally
display data. The FHIR core profile already constrains the unit space
to `cm` or `[in_i]`, so the comparability concern raised by the
ticket is largely addressed without BSeR having to add a stricter
constraint. Aligning the UML to what's actually achievable in US
EHRs avoids breaking existing implementers while still giving the
unit of measure a defined, machine-comparable answer.

---

### Disposition C: Decline

#### Proposal

Decline the change as out of scope for the BSeR profile and rely on
the FHIR core `http://hl7.org/fhir/StructureDefinition/bodyheight`
profile, which already binds `Observation.valueQuantity` to UCUM with
required codes `cm` or `[in_i]`. Update the BSeR Body Height
narrative to point at the core profile's existing constraint and
explain that BSeR does not further restrict it; close the ticket as
"already addressed by existing mechanism".

#### Justification

The underlying ask — that height be expressed in a known, comparable
unit — is already satisfied by the FHIR core vital-signs profile that
BSeR builds on. Adding a BSeR-specific tightening risks fragmenting
US public-health profiles (BSeR vs. US Core vs. other PH IGs) when
the right place to constrain vital-sign units is the shared
vital-signs layer, not each downstream IG. Declining keeps BSeR
aligned with US Core / vital-signs conventions and avoids a
backwards-incompatible change for any current BSeR implementer who
is sending inches today.

---

### Recommendation

**Recommended disposition:** A

The ticket is a straightforward internal-consistency defect: the BSeR
UML for Body Height says centimeters, but the StructureDefinition
allows any unit. That mismatch is exactly the kind of thing the
authoring workgroup is best positioned to resolve, and the cleanest
fix is to make the StructureDefinition match the UML by fixing
`valueQuantity.code` to `cm`. BSeR is a referral-flow IG with a
relatively small implementer base, so the migration cost of
tightening the unit is low; the long-term comparability win for
public-health analytics is high. Disposition B is a reasonable
fallback if the workgroup hears from implementers that inch-only
capture is common in their referral sources, and Disposition C is
defensible only if the workgroup decides Body Height should not be a
BSeR-specific profile at all. Absent that signal, Accept As Requested
is the right call.
