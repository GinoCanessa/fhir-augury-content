## Summary

The reporter notes that the US Public Health Profiles Library (v2.0.0
snapshot) provides **two distinct locations** at which a Specimen
Collection Date/Time can be recorded for a laboratory result:

1. On the **US Public Health Laboratory Result Observation Profile** via
   `Observation.effective[x]`, where the IG text describes the value as
   "usually either the time of the procedure or of specimen collection".
2. On the **US Public Health Specimen Profile** via
   `Specimen.collection.collectedDateTime`, which is explicitly tagged
   as the (USCDI+) Specimen Collection Date/Time element.

The ticket asks the Public Health work group to clarify the intended
behavior for implementers: is a single canonical location preferred (and
if so, which one), or are systems expected to populate both locations
with the same value? The reporter requests that, if feasible, the IG
choose a single location to remove ambiguity. There are no
cross-referenced Jira tickets, no Zulip discussions, and no GitHub items
linked to this ticket; the only cross-references are direct mentions of
the two FHIR element paths above.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Designate **`Specimen.collection.collectedDateTime`** (in the US Public
Health Specimen profile) as the **single canonical location** for the
Specimen Collection Date/Time, and remove specimen-collection semantics
from `Observation.effective[x]` in the US Public Health Laboratory
Result Observation profile.

Concretely, in the US Public Health Profiles Library:

- In `StructureDefinition-us-ph-lab-result-observation`, change the
  guidance/description on `Observation.effective[x]` to explicitly
  describe it as the **clinically relevant time of the observation /
  result** (e.g., the time the result applies to clinically) and add a
  short, normative-style note: *"For the date/time the specimen was
  collected, use `Specimen.collection.collected[x]` on the referenced US
  Public Health Specimen instance. Do not duplicate the specimen
  collection date/time on `Observation.effective[x]`."*
- In `StructureDefinition-us-ph-specimen`, retain
  `Specimen.collection.collected[x]` (with `collectedDateTime` as the
  expected choice) as the USCDI+ Specimen Collection Date/Time and
  reinforce that this is the authoritative location.
- Optionally tighten `us-ph-lab-result-observation` to require
  `Observation.specimen` when a specimen exists, so the
  `Specimen.collection.collected[x]` value is always reachable from the
  Observation.

#### Justification

Having two equally-valid locations for the same datum is the principal
source of ambiguity the ticket calls out and is exactly the kind of
clarification implementers need before going to production. Anchoring
on `Specimen.collection.collected[x]` aligns the IG with USCDI+, which
already names that element as the Specimen Collection Date/Time, and
keeps the public-health profiles consistent with how
`Observation.effective[x]` is used elsewhere in FHIR (the *clinically
effective* time of the observation, not the specimen-handling time).
Centralizing on the Specimen also avoids encouraging systems to compute
or guess `Observation.effective[x]` when only the specimen draw time is
known.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both elements available, but make the relationship between them
**explicit and rule-based** rather than choosing one canonical location:

- Update both profile narratives with a shared "Specimen Collection
  Date/Time" guidance block that defines exactly what each element means
  in the public-health laboratory context:
  - `Specimen.collection.collected[x]` — the date/time the **specimen
    was physically collected** from the subject (the USCDI+ data
    element).
  - `Observation.effective[x]` — the **clinically effective time** of
    the result; for laboratory results derived from a specimen this
    *may* be the collection time, the analysis time, or another
    clinically relevant instant, depending on the test.
- Add an invariant / "must-support" expectation on
  `us-ph-lab-result-observation` that when a `Specimen` is referenced,
  `Specimen.collection.collected[x]` SHALL be present, and when both
  `Observation.effective[x]` and `Specimen.collection.collected[x]` are
  present, the producing system SHALL ensure they are semantically
  consistent (with implementer guidance describing what "consistent"
  means: equal when the result is the specimen draw time; otherwise the
  effective time may be later).
- Provide a worked example in the IG showing the recommended population
  pattern (specimen draw on Specimen, clinically effective time on
  Observation).

#### Justification

Many laboratory workflows already populate both elements because
upstream LIS/LIMS systems emit both, and forcing a single location may
break downstream consumers (e.g., consumers who only read
`Observation.effective[x]` because the base FHIR Observation pattern
trains them to). Explicit, named semantics for each element preserves
the existing behavior, satisfies USCDI+ on the Specimen side, and gives
implementers a deterministic rule for when they may diverge — which is
the underlying ask of the ticket — without a breaking profile change in
the 2.0.0 cycle.

---

### Disposition C: Decline

#### Proposal

Decline the request to pick a single location and instead close as
"Not Persuasive — already addressed by the base FHIR semantics":
`Observation.effective[x]` and `Specimen.collection.collected[x]` are
documented in core FHIR with distinct meanings (effective time of the
observation vs. time the specimen was collected), and US Public Health
Profiles Library inherits those definitions. The IG should not
re-define base-resource semantics, and implementers should follow the
base FHIR guidance plus USCDI+ for the Specimen.

#### Justification

Carving the public-health IG away from base FHIR semantics introduces
divergence from every other IG that uses the same elements (US Core,
mCODE, the Diagnostics IGs) and risks confusing implementers who work
across multiple jurisdictions and domains. The "two locations" the
ticket points at are not actually duplicative in the base spec — they
record different concepts that often happen to coincide. A short FAQ
entry or implementer guide note can address the confusion without
re-shaping the profiles.

---

### Recommendation

**Recommended disposition:** B

Disposition A produces the cleanest IG, but unilaterally removing the
specimen-collection wording from `Observation.effective[x]` is a
breaking change for systems already populating that element from
LIS/LIMS feeds and is likely to draw negative ballot comments from
laboratory implementers. Disposition C under-delivers: the ticket is a
real, reasonable implementer question, and "read the base spec" is not
adequate guidance from a public-health IG that explicitly calls out
both locations in its own profile text. Disposition B threads the
needle — it preserves backward compatibility, makes the USCDI+
Specimen element the authoritative collection date/time as the ticket
requests, and removes the ambiguity by giving each element a precise,
non-overlapping meaning plus an explicit consistency rule. The work
group should confirm the wording with the laboratory implementer
community (e.g., via Zulip in `#public health`) before publishing, and
consider promoting the consistency rule to a formal invariant in a
later release once tooling is in place.
