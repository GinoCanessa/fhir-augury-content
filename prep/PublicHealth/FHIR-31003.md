## Summary

This change request, raised against the Vital Records Common Profiles
Library (VRCL), asks whether any of the profiles defined in VRCL are in
fact reusable outside the Vital Statistics / Vital Records domain, and
if so, whether they should "live at a different level in the stack"
rather than being defined inside a use-case-specific IG. The reporter
(Lorraine Constable) explicitly notes that this is really a
methodology / cross-cutting concern more appropriate for MnM than for
the Public Health work group, motivated by an observed pattern of
multiple IGs each redefining profiles for the same underlying concept.
The ticket has been resolved twice as "Considered for Future Use"
(2021-03-25 and 2024-03-07) and re-opened twice (post-STU1.1 and
post-STU2), indicating the work group has accepted the principle but
deferred the actual move pending a willing higher-level home. No
related Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-reference or search.

## Proposed Dispositions

The following dispositions all turn on a question of *governance and
packaging* rather than profile content. None require a code/spec
change inside a single specific repo (no GitHub cross-references were
surfaced), so they instead describe project-level commitments by VRCL
and any candidate "higher-level" library.

### Disposition A: Accept As Requested

#### Proposal

Treat this as a persistent VRCL governance commitment. Specifically:

1. In the next VRCL publication, add a "Profile Reuse and Layering"
   section to the IG's methodology / introduction page that:
   - Names each VRCL profile and classifies it as either
     **vital-records-specific** (intended to remain in VRCL) or
     **potentially reusable** (candidate to move "up the stack" into
     US Core, a Public Health common library, or a future US-Realm
     profile library).
   - For each *potentially reusable* profile, identifies the proposed
     target home (e.g., US Core, a Public Health library, a new
     US-Realm library).
2. File a tracking ticket per *potentially reusable* profile against
   the receiving project (US Core, Public Health library, etc.)
   asking it to absorb the profile, with VRCL committing to switch to
   `derives-from` / direct reference once the receiving project
   adopts it.
3. Establish, in coordination with MnM and US Realm Steering, a
   standing review at each VRCL ballot cycle to re-evaluate the
   classification.

#### Justification

This honors the ticket's intent literally — it forces a per-profile
answer to "which of these belong elsewhere?" — while preserving the
work group's established position (twice voted, 19-0-0 and 21-0-0)
that the actual move can only happen when a willing higher-level home
exists. By codifying a classification table in the IG itself, the
question stops being implicit and becomes auditable across
publications, which addresses Lorraine's underlying concern about
multiple IGs independently redefining the same concept. It also
aligns with the pattern US Core and other US-Realm IGs already use
when nominating profiles for promotion.

---

### Disposition B: Alternative Approach

#### Proposal

Re-scope the ticket from a per-profile classification exercise to a
**methodology referral to MnM** (which is what the reporter actually
suggests in the description):

1. Close FHIR-31003 against VRCL with a pointer comment.
2. Open a new MnM-targeted ticket (or Confluence methodology page)
   asking MnM, in coordination with the FHIR-I and US Realm Steering
   groups, to define general guidance for:
   - When a profile defined in a use-case IG should be promoted to a
     shared library (criteria for "reusable").
   - The mechanical handoff (deprecation in the source IG, redirect,
     versioning).
   - Where US-Realm shared profiles should live in the absence of a
     dedicated US-Realm common library (US Core vs. a new artifact).
3. VRCL commits to apply that guidance once published, but does not
   itself produce the per-profile classification until then.

#### Justification

The ticket explicitly states this is "not a Public Health Workgroup
question … should be considered by MnM," and the 2024-03-07 vote
language already reached the same conclusion ("This process should be
driven by the developer of the higher level library"). Solving the
general problem at MnM avoids VRCL inventing a one-off classification
process that other IGs would then ignore, and prevents the same
ticket from being re-opened ballot-after-ballot against VRCL when the
real blocker lives elsewhere. It also honors the reporter's framing
without forcing VRCL to take ownership of cross-IG governance.

---

### Disposition C: Decline

#### Proposal

Persistent disposition: **Not Persuasive — already addressed by
existing process.** Close with a comment that:

- VRCL has on two prior occasions (2021-03-25, 2024-03-07) voted with
  unanimous consent that profile relocation will be driven by the
  receiving library, not by VRCL.
- VRCL profiles already use canonical URLs and are referenceable from
  any other IG today; nothing in the current packaging prevents
  reuse. There is therefore no defect in VRCL itself.
- The ticket should not be re-opened against VRCL on subsequent
  publications absent a concrete proposal naming a specific receiving
  library and specific profiles to move.

#### Justification

The request as written has no actionable change to VRCL's normative
content — it is a question, not a defect. The work group has
deliberated it twice, both times reaching the same answer with
overwhelming support, and the only thing that has changed between
re-openings is that VRCL has published another STU. Continuing to
re-open the ticket without a concrete receiving-library proposal
absorbs ballot bandwidth without producing a different outcome.
Existing FHIR mechanisms (canonical references, `derives-from`,
package dependencies) already let any future US-Realm or US Core
library absorb VRCL profiles when one becomes available.

---

### Recommendation

**Recommended disposition:** B

The ticket has been re-opened twice with the same outcome, which is a
strong signal that simply voting it down a third time (Disposition C)
will not make it stay closed — the underlying concern is real and
recurring. At the same time, Disposition A asks VRCL to do work
(per-profile classification, opening receiving-side tickets) that the
work group has twice rejected as out of its lane, and that arguably
*should* be driven by the receiving library per the 2024-03-07 vote.
Disposition B matches what the reporter actually asked for ("should be
considered by MnM"), preserves the prior VRCL position, and converts
a recurring sticky ticket into a one-time methodology referral. It
gives the next VRCL ballot a clean answer ("we are awaiting MnM
guidance / a willing higher-level home, tracked under <new ticket>")
instead of inviting yet another re-open.
