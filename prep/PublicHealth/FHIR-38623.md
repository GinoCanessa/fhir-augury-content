## Summary

The reporter (Saul Kravitz) observes that in the VRDR IG, the
`CauseOfDeathPart1` profile carries the cause-of-death text only in the
`valueCodeableConcept.text` of an Observation, while ICD‚Äë10 coded values are
carried separately on the `EntityAxisCauseOfDeath` (EAC) profile, indexed by
line and position. The ticket argues this split is awkward and not
"FHIR‚Äëlike": if there is already a coded representation per line/position,
that code should be expressible on the same Observation that carries the
text. Two alternatives are proposed: (1) bind
`CauseOfDeathPart1.valueCodeableConcept` to the same ICD‚Äë10 ValueSet that EAC
uses today and document that profile as the place where the coded form
lives; or (2) do (1) and then eliminate the separate `EntityAxisCauseOfDeath`
profile, formally mapping EAC into `valueCodeableConcept.coding`.

The ticket was originally voted "Considered for Future Use" (Passas/Kravitz,
25‚Äë0‚Äë0) on 2024‚Äë03‚Äë28 with the intent to consider it for the next ballot
version, then reverted post‚ÄëSTU3 publication and re‚Äëopened (lynn_laakso,
2024‚Äë10‚Äë16) so it can be addressed in the next VRDR ballot cycle. There are
no Jira, GitHub, or recorded Zulip cross-references attached to the ticket

implementers actually use the line/position model in production
interjurisdictional exchange (see *Related Zulip Discussions* below).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt option (1) from the ticket: in the next VRDR ballot, bind
`CauseOfDeathPart1.valueCodeableConcept` (and the parallel
`CauseOfDeathPart2.valueCodeableConcept`) to the same ICD‚Äë10 ValueSet that
`EntityAxisCauseOfDeath` uses today (the ICD‚Äë10 cause-of-death codes used
by NCHS), as a `preferred` (or `extensible`) binding. Update the profile
narrative to document that, when an ICD‚Äë10 coded value is available for
the *line as a whole* (Part 1 line 1 / 2 / 3 / 4 and Part 2), it MAY be
expressed on `valueCodeableConcept.coding` alongside the literal
`valueCodeableConcept.text` written by the certifier. Keep the
`EntityAxisCauseOfDeath` profile unchanged; it remains the place to carry
the per-token (line + position) entity-axis breakdown produced by NCHS
coding, including the e-code indicator. Update the IG's
"Recommended Best Practices" / cause-of-death section to describe this dual
representation and explain that EAC is *additional* detail, not the only
home for coded values.

#### Justification

This change is conservative and additive: existing implementations that
only populate `valueCodeableConcept.text` continue to validate, and
existing EAC consumers continue to function. It also addresses the
reporter's "FHIR-like" complaint without losing information ‚Äî the EAC's
line/position/e‚Äëcode structure (clearly required in the field per the two
Death-on-FHIR Zulip threads above) is preserved. Allowing the
line-level ICD‚Äë10 code on `CauseOfDeathPart1` itself is consistent with
how other VRDR observations already carry coded values in
`valueCodeableConcept`, and it gives downstream consumers a simpler path
when they only need the line-level code.

---

### Disposition B: Alternative Approach

#### Proposal

Decline option (2) outright (do **not** eliminate `EntityAxisCauseOfDeath`),
and instead of adding a binding to `CauseOfDeathPart1`, address the
reporter's underlying concern with documentation only: keep the current
structure (text on `CauseOfDeathPart1.valueCodeableConcept.text`, codes on
`EntityAxisCauseOfDeath` keyed by `lineNumber`/`position`), and add a
prominent "Coded vs Literal Cause of Death" section to the VRDR IG that
diagrams how the two profiles relate, why Part 1 lines need a `lineNumber +
position` keyed structure (multiple ICD‚Äë10 entity-axis codes per certifier
line plus an e-code indicator), and how implementers should reconstruct a
"Part 1 line N coded value" by aggregating EAC entries with `lineNumber =
N`. Optionally add an extension on `CauseOfDeathPart1` that *links* to the
matching `EntityAxisCauseOfDeath` instances rather than duplicating the
codes inline.

#### Justification

The Zulip thread "Question on NCHS TRX response ‚Äì EntityAxis & RecordAxis"
makes it explicit that NCHS's coded response is intrinsically multi-token
per Part 1 line (1:1, 1:2, ‚Ä¶, 2:1, ‚Ä¶) and includes an e-code indicator
that doesn't have a natural home on `valueCodeableConcept`. Trying to
collapse that into a single `valueCodeableConcept.coding` on
`CauseOfDeathPart1` either loses information (no positions, no e-code
flag) or forces implementers to choose which entity-axis code is "the"
code for the line ‚Äî a value judgement the IG currently (deliberately)
avoids. Treating this as a documentation/clarification problem keeps the
existing wire format stable for jurisdictions that have already implemented
STU3, addresses Saul's point that the relationship is currently obscure,
and explicitly rejects the "eliminate EAC" half of the request that would
be a hard breaking change for production exchange (KY, CT, NAPHSIS STEVE,
NCHS TRX).

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive": leave `CauseOfDeathPart1` /
`CauseOfDeathPart2` carrying only `valueCodeableConcept.text`, and keep
`EntityAxisCauseOfDeath` and `RecordAxisCauseOfDeath` as the sole homes
for coded NCHS output. Rationale to record on the disposition: the
existing factoring matches both the IJE reference model and the way NCHS
TRX returns coded data (entity axis = per-token, indexed by line +
position with an e‚Äëcode flag; record axis = up to 20 codes for the record
as a whole). Adding a second, line-level binding on `CauseOfDeathPart1`
would create two valid coded representations for the same datum and force
trading partners to define precedence rules.

#### Justification

The ticket's premise ‚Äî that text on Part 1 and codes on EAC is "not
FHIR‚Äëlike" ‚Äî is at odds with how mortality coding actually works
operationally. Certifier‚Äëwritten Part 1 lines are free text; NCHS coding
produces multiple tokenized ICD‚Äë10 codes per line plus a record-axis
summary. The current IG structure mirrors that. The Death-on-FHIR
threads above show that implementers can and do locate the coded
information on EAC; the friction reported (e.g. KY/CT) is about MOU /
data‚Äësharing policy and missing MRE elements, not about the location of
EAC. A change that adds a redundant binding risks divergence between the
two coded representations without solving an in-the-field problem.

---

### Recommendation

**Recommended disposition:** B

The Death-on-FHIR Zulip evidence makes a strong case that EAC's
`lineNumber` + `position` + e‚Äëcode indicator structure is doing real work
that does not collapse cleanly into a single `valueCodeableConcept.coding`
on `CauseOfDeathPart1`, so option (2) of the ticket (eliminate EAC) is a
non‚Äëstarter. At the same time, Disposition C ignores a legitimate
clarity problem: the IG currently makes implementers walk between two
profiles to assemble a coded Part 1, and the ticket reporter is not the
first person to be confused. Disposition B preserves the wire-stable
current factoring (which the Public Health WG just published in STU3),
addresses the documentation gap directly, and leaves the door open to a
narrowly-scoped extension that links a `CauseOfDeathPart1` line to its
matching EAC instances if the WG decides that helps consumers. If the
WG instead has appetite to carry a *line-level* ICD‚Äë10 code on
`CauseOfDeathPart1.valueCodeableConcept.coding`, Disposition A is the
right way to add it ‚Äî additively and without removing EAC ‚Äî and the WG
should explicitly call that out in the disposition so the binding choice
is on the record.
