## Summary

The reporter (Craig Newman) flags that Section 4.2 of the US Vital Records
Death Reporting (VRDR) IG contains a broken link to the *NCHS Vital Records
FHIR Messaging Implementation Guide* (the "Nightingale" messaging IG that
specifies the messaging envelopes used to exchange VRDR content between
Jurisdictional Vital Records Offices and NCHS). The link points to a
continuous-build URL
(`https://build.fhir.org/ig/nightingaleproject/vital_records_fhir_messaging_ig/branches/main/index.html`)
that no longer resolves. The ticket asks the workgroup to either update the
link to a current, working location or remove it.

No related Jira tickets, Zulip discussions, or GitHub cross-references were
returned by the cross-reference query, so the request stands on its own as
an editorial / specification-hygiene fix.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the link in Section 4.2 of the VRDR IG so it points to a stable,
working location for the NCHS Vital Records FHIR Messaging IG, rather
than the dead `build.fhir.org/ig/nightingaleproject/...` continuous build
URL. Concretely, replace the URL with the current canonical / published
location for the Messaging IG (the most recent published or balloted
version under the NCHS / nightingaleproject publication path; if no
published version exists, link to the project's current `main` build
under its present hosting). The link text ("NCHS Vital Records FHIR
Messaging Implementation Guide") and the surrounding sentence describing
the IG's role (messaging envelopes for JVRO ↔ NCHS exchange) should be
preserved unchanged.

#### Justification

The Messaging IG is genuinely a companion specification to VRDR — the
narrative in Section 4.2 already explains its role, and removing the
pointer would leave implementers without a discoverable path to the
envelope spec they need. Repointing to a stable URL fixes the immediate
defect with the lowest possible disruption and keeps the editorial
intent of the section intact. This is consistent with how the FHIR
ecosystem generally treats links to companion / dependency IGs: prefer
the latest published version, falling back to a stable CI URL only if no
publication exists.

---

### Disposition B: Alternative Approach

#### Proposal

Replace the bare link with a versioned, dependency-style reference: cite
the NCHS Vital Records FHIR Messaging IG by its canonical URL and a
specific version (the latest published version known at the time of
VRDR's next publication), and add a short note that readers should
consult the IG's own publication history page for newer versions. If a
published version does not yet exist, soft-deprecate the inline link in
Section 4.2 and instead surface the dependency in the IG's
"Dependencies" / cross-IG references section, where it can be maintained
alongside other companion specs.

#### Justification

The root cause of this ticket is a class of problem, not a one-off typo:
narrative-embedded `build.fhir.org/.../branches/main/...` URLs rot
whenever the upstream IG renames a branch, moves repositories, or
changes its CI pipeline. Pinning to a canonical/versioned URL — and/or
moving cross-IG pointers into a single dependencies section — makes
future breakage less likely and easier to maintain. The trade-off vs.
Disposition A is a slightly larger edit and a small loss of inline
discoverability, in exchange for durability.

---

### Disposition C: Decline

#### Proposal

Decline to update the link and instead remove it entirely from Section
4.2, leaving only the descriptive sentence about the existence of the
NCHS Vital Records FHIR Messaging IG without a URL. Rationale category:
*defer to the companion IG's own publication channels* — VRDR readers
who need the messaging spec can locate it via the NCHS / nightingaleproject
project pages, and VRDR avoids ongoing maintenance burden for an
external link.

#### Justification

Declining a *fix* in the strict sense is hard to justify for a clearly
broken link, but removing the link altogether is a defensible minimal
action: it eliminates the defect, removes the maintenance liability of
tracking another IG's URL changes, and is consistent with how some HL7
specifications avoid embedding direct links to externally-hosted
companion IGs. The cost is reduced discoverability for implementers who
would otherwise follow the link directly from VRDR.

---

### Recommendation

**Recommended disposition:** A

A is the right starting point because the ticket is a small, well-scoped
editorial defect: the section already commits to pointing at the NCHS
Messaging IG, the descriptive sentence is correct and useful, and the
only thing wrong is the URL. Repointing to a working location restores
the intended behaviour with minimal change and no risk to the rest of
the spec. Disposition B's durability concerns are real and worth
adopting *if* the workgroup wants to make a broader pass over external
links in the IG, but that is out of scope for this single ticket and
should not block the fix. Disposition C trades a real loss of
implementer convenience for a small reduction in maintenance and is
hard to defend when a working URL almost certainly exists. The
workgroup's main decision under A is which target URL to use — the
latest published version of the Messaging IG if one exists, otherwise
its current canonical CI build location.
