## Summary

The reporter (Craig Newman) observes that the BSeR Obesity Referral Request
Bundle appears to permit only a single occurrence of the `PatientAllergy`
profile, and that `PatientAllergy` itself only documents a single allergy.
Together these constraints would limit a referral to a single patient
allergy, which is clinically unrealistic — patients commonly have multiple
allergies that may be relevant to a referral. The reporter flags that the
Arthritis Referral Request Bundle has the same issue.

The change is therefore a cardinality / structural fix in the BSeR IG: the
allergy entry in the affected referral bundles should be repeatable (or the
modeling reworked so that all of a patient's relevant allergies can be
conveyed). No Jira links, comments, or cross-references to other tickets,
Zulip threads, or GitHub items are recorded against this ticket in the
indexed sources, so the disposition has to lean on the BSeR IG's normal
profiling pattern (one `AllergyIntolerance` per allergy) and on the
adjacent BSeR completeness tickets surfaced by keyword similarity.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a cardinality / structure defect in the BSeR IG and fix it
in two places:

1. **Bundle entries.** In the Obesity Referral Request Bundle and the
   Arthritis Referral Request Bundle profiles, change the slice that
   currently constrains the `PatientAllergy` entry to `0..1` so it
   becomes `0..*` (or, equivalently, define the slice with `min=0`,
   `max=*`). Apply the same change to any other condition-specific BSeR
   referral request bundles (Hypertension, Diabetes, etc.) that mirror
   the Obesity/Arthritis pattern, to keep the IG internally consistent.
2. **PatientAllergy profile.** Confirm that `PatientAllergy` is
   structured as a profile of a single `AllergyIntolerance` (one profile
   instance == one allergy). If, instead, `PatientAllergy` itself
   internally tries to model multiple allergies, refactor it so it is
   exactly a single-allergy profile and let repetition happen at the
   bundle-entry level.

The semantic effect is: a sender can include zero, one, or many
`PatientAllergy`-conformant `AllergyIntolerance` resources in the bundle.

#### Justification

This matches the way every comparable FHIR IG conveys multiple allergies
(one `AllergyIntolerance` per allergy, repeated as needed) and matches
how BSeR already conveys other repeating clinical concepts. The
underlying clinical assumption — that a referral might need to convey
multiple drug/food/environmental allergies — is uncontroversial. Adjacent
BSeR tickets (FHIR-22567, FHIR-22615) show the work group has previously
restructured these same bundles in response to completeness/clinical
gaps, so the editorial pathway is well-trodden. The change is purely a
relaxation of cardinality (`0..1` → `0..*`), so it is **non-breaking for
existing senders** — any bundle that conforms today will still conform
after the change.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the bundle slicing as it is, but redefine `PatientAllergy` as a
**list-style profile** that wraps the patient's full allergy list — for
example, a profile on `List` whose `entry.item` references one or more
`AllergyIntolerance` resources (each constrained by an
`AllergyIntoleranceProfile`), with a constraint requiring at least one
entry per known allergy and an explicit `emptyReason` slice for "no known
allergies." The bundle would then continue to carry exactly one
`PatientAllergy` resource (the list), and that list repeats the allergies
internally.

#### Justification

This preserves the current "one profile per concern" shape of the
referral request bundle (one PatientAllergy slot, one PatientMedication
slot, etc.), which can make consumer parsing simpler — a consumer always
looks in the same slot to find the allergy story, including the
explicit "no known allergies" assertion. It also gives a natural place to
attach list-level metadata (date asserted, asserter, mode = `snapshot` vs
`changes`). The trade-off: it is a heavier change than Disposition A
(introduces a new `List` profile, requires senders to switch from
emitting `AllergyIntolerance` directly to wrapping them, and is
**breaking** for any current implementer), and it does not match the
pattern used by other comparable referral IGs that simply repeat
`AllergyIntolerance`. Worth raising as an option specifically because
BSeR does separately care about "we asked, and the patient has none"
which is awkward to express with a bare repeating entry.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive": no change to the BSeR profiles or
bundles. Document that implementers needing to convey multiple allergies
should continue to use the existing slicing (e.g., emit additional
`AllergyIntolerance` resources outside the `PatientAllergy` slice or in
an unsliced/open `Bundle.entry`), and update the IG narrative to make
that pattern explicit.

#### Justification

Defensible only if the work group decides the current `0..1` slicing was
intentional — for example, if BSeR is deliberately scoping
`PatientAllergy` to "the single most clinically-relevant allergy for
this referral" and expects the referral receiver to look up the full
allergy list out-of-band (USCDI/USCore patient pull, etc.). In that case
the request is based on a misreading of the IG's intent and the right
fix is documentation, not a cardinality change. This is the weakest of
the three: nothing in the ticket, the IG's clinical intent, or the
keyword-similar BSeR tickets supports a "one allergy is enough" reading,
and forcing implementers to invent their own out-of-bundle conveyance
defeats the purpose of using a self-contained referral request bundle.

---

### Recommendation

**Recommended disposition:** A — Accept As Requested.

Disposition A is a small, non-breaking cardinality relaxation against a
clinical assumption (patients can have more than one allergy) that no
reasonable reviewer will contest. It aligns BSeR with the universal
FHIR pattern of "one `AllergyIntolerance` per allergy, repeated", keeps
the existing `PatientAllergy` profile shape, and matches the editorial
pattern the Public Health work group has already used on these same
referral request bundles in FHIR-22567 and FHIR-22615. Disposition B
solves a real but secondary problem (explicit "no known allergies"
assertion and list-level metadata) at the cost of a breaking change and
a divergence from comparable IGs; it should be raised as a follow-up
ticket if the work group wants to express list-level semantics, not
folded into this one. Disposition C requires the work group to defend a
clinical position ("one allergy per referral") that is essentially
indefensible, so it should only be picked if the work group is prepared
to make that argument explicitly.
