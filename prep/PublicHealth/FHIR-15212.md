## Summary

Catherine Staes raised a 2018-Jan ballot comment (eCR #28) against the
(now deprecated) Electronic Case Reporting (eCR) FHIR IG, questioning
the value set scoping for occupation in the eCR profiles. Two concerns
are bundled in the comment:

1. The IG only binds an "occupational" value set, and the commenter
   asks why that is the only value set included for the
   occupation-related element.
2. The bound terminology uses **SOC** (Standard Occupational
   Classification) codes, but in practice occupation in U.S. public
   health reporting is conveyed using the **CDC Census occupation
   codes**, not SOC. The implication is that the wrong code system has
   been bound.

The ticket was triaged and resolved in 2018 as **Persuasive with
Modification**, meaning the workgroup agreed the binding/value-set
scope needed to change but applied a modified fix rather than the
literal request. No comments, attachments, or explicit Jira links are
recorded on the issue itself, and no GitHub artefacts cross-reference
it. The only cross-reference is the automated `fhir_bot` Zulip tracker
post announcing the ticket. This prep is therefore primarily a
historical disposition review for an already-resolved item against a
deprecated specification.

## Proposed Dispositions

> Note: The ticket is already resolved as **Persuasive with
> Modification** against a deprecated IG. The dispositions below are
> framed as if the workgroup were re-deciding today, so the historical
> resolution can be confirmed, revisited, or formally closed.

### Disposition A: Accept As Requested

#### Proposal

Replace (or extend) the SOC-based occupation value set bound on the
eCR occupation element with one drawn from the **CDC Census
Occupation classification** (the system the commenter identifies as
the actual code system used for occupation in U.S. public health
reporting). Concretely: define/rebind the eCR occupation value set so
that its primary `system` is the CDC Census Occupation code system,
mark the binding strength explicitly (preferred or extensible, in line
with sibling demographics bindings), and add additional value sets
(e.g., industry, usual occupation vs. current occupation) only if
those elements actually exist on the profile.

#### Justification

The commenter's two concerns map cleanly onto a single corrective
action: bind the right code system to the right element. SOC and CDC
Census Occupation classifications differ in granularity and intent,
and using SOC where CDC Census is the de facto standard for case
reporting would produce data that public health agencies cannot
ingest without re-coding. Aligning the binding with the system
actually used downstream is consistent with FHIR's general guidance
to bind to terminologies that match the operational use case.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than swapping SOC out wholesale, restructure the occupation
element as an **extensible binding to a composite value set** that
includes the CDC Census Occupation codes as the primary/preferred
system while still permitting SOC (and other jurisdictionally
required systems) as alternates. Document in the profile narrative
which system implementers SHOULD send when the consumer is a U.S.
public health agency, and add an example showing CDC Census usage.
Optionally, expose the choice via an extension (e.g.,
`occupation-classification-system`) so consumers can negotiate
expectations.

#### Justification

eCR is consumed by jurisdictions whose terminology preferences are
not uniform; some downstream systems do consume SOC for workforce or
surveillance analytics even if CDC Census is the dominant case-report
system. A composite value set with a clear preferred system avoids
breaking implementations that already send SOC, gives CDC Census
first-class status, and keeps the binding strength loose enough to
accommodate jurisdictional variance — at the cost of more
implementer guidance and slightly weaker interoperability guarantees
than Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline to act on the ticket on the grounds that the underlying
specification — *Electronic Case Reporting [deprecated] [deprecated]
(FHIR)* — is deprecated, and current eCR work has moved to the
successor IG. Close the ticket as **Not Persuasive — Out of Scope
(Deprecated Specification)**, with a pointer telling the commenter to
re-file against the active eCR IG if the same terminology issue
persists there.

#### Justification

Modifying a doubly-deprecated specification yields no implementer
benefit: nobody is expected to build against it, and any errata would
not propagate to the spec people actually use. The 2018 resolution
("Persuasive with Modification") presumably already drove a fix into
whatever the contemporary successor was; re-opening or carrying the
change forward without verifying the active IG is wasted workgroup
cycles. Declining and redirecting concentrates effort on the live
specification.

---

### Recommendation

**Recommended disposition:** B

The substantive concern — that SOC alone is the wrong binding for
occupation in U.S. public health case reporting — is real and has
not gone away with the IG's deprecation; it is exactly the kind of
issue that tends to recur in successor IGs. Disposition A is the
cleanest fix in isolation, but it risks invalidating any existing
implementations that send SOC and does not acknowledge jurisdictional
variability. Disposition C is tempting given the deprecation, but it
discards a useful signal that should be confirmed against the active
eCR IG. Disposition B preserves the corrective intent (CDC Census
becomes the preferred system), keeps the door open for SOC where it
is genuinely used, and gives the workgroup a concrete artefact (a
composite value set + narrative guidance) to carry forward into the
current eCR IG. The workgroup should also verify whether the 2018
"Persuasive with Modification" resolution was applied to the
successor IG; if not, this ticket's recommendation should be ported
there.
