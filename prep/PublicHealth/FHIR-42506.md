## Summary

This ticket, filed against the US Bidirectional Services eReferrals (BSeR)
implementation guide, questions a constraint in the BSeR message/workflow
design that restricts the "outcome" package (the bundle/section of artifacts
conveying outcomes such as Observations and reports) to only the *end* of a
referral exchange. The reporter, Lloyd McKenzie, points out that real-world
referral workflows often need to convey *interim* or *work-in-progress*
outcomes — partial Observations, status reports, intermediate findings —
before the referral is fully closed out. The request is therefore to relax
the "outcome only at end" rule so that outcome content can also be exchanged
mid-workflow. The ticket has been triaged as Persuasive, with the recorded
resolution committing to "add the ability to send interim in-progress
outcomes." There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items linked in the Augury index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the BSeR specification so that the **outcome package** is permitted at
points other than the terminal step of a referral exchange. Concretely:

- In the BSeR referral state machine / workflow narrative, remove (or
  reword) the constraint that limits the outcome package to the end of the
  exchange. Replace it with language allowing the outcome package to be
  exchanged whenever interim outcomes are available (e.g., from any state
  in which the receiving party has clinically meaningful in-progress
  results to share).
- Add an explicit "interim outcome" mode: either a distinct message event
  / Task business status (e.g., `in-progress-outcome`) or a flag /
  qualifier on the existing outcome message that distinguishes interim
  from final outcomes.
- Update the corresponding BSeR message Bundle profile(s) and
  ServiceRequest / Task workflow guidance to permit (and ideally require
  appropriate distinguishing metadata for) repeated outcome submissions
  prior to closure.
- Clarify in narrative that a final outcome package is still expected at
  closure of the referral, so that interim outcomes do not eliminate the
  closing exchange.

This matches the ticket's stated request and the recorded "Persuasive"
resolution ("We will add the ability to send interim in-progress
outcomes.").

#### Justification

The reporter's use case — sending in-progress Observations, partial
reports, or status updates before the referral is fully closed — is a
common clinical pattern (e.g., long-running consults, multi-visit
follow-ups, staged diagnostic workups). Forcing all outcome content to the
end of the exchange either delays clinically useful information or forces
implementers to misuse other message types. FHIR's general workflow
pattern (Task / ServiceRequest with `intent`, `status`, `businessStatus`)
already supports interim updates, so aligning BSeR with that pattern is
consistent with broader FHIR design. The Public Health work group has
already marked this Persuasive, signalling agreement with the principle.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than redefining the existing outcome package to be sendable
anywhere in the workflow, introduce a **separate, lightweight "interim
update" message** alongside the existing terminal outcome package:

- Keep the current "outcome package" semantics tied to referral closure
  (terminal, comprehensive, the authoritative outcome).
- Add a new event/profile — e.g., `bser-interim-update` — whose Bundle is
  permitted to carry Observations, DiagnosticReports, and Communication
  resources mid-workflow, but is explicitly *not* a closure event.
- Specify that interim updates do not change the referral's overall Task
  status (it remains `in-progress`), and that the eventual outcome
  package at closure is still required and may consolidate / supersede
  the interim content.

#### Justification

This preserves backwards compatibility for existing BSeR implementers who
have built logic around "outcome package == referral is closing." Mixing
interim and final outcomes in the same message event risks ambiguity:
receivers may auto-close referrals on receipt of an outcome package, or
file results without realizing more are coming. A distinct interim-update
event keeps the closure semantics crisp while still satisfying the
clinical need. The trade-off vs. Disposition A is a slightly larger
surface area (an additional event/profile) in exchange for clearer
state-machine semantics and lower migration risk for existing
implementations.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the current rule that the outcome package
is exchanged only at the end of the referral. Direct implementers who
need to convey interim findings to use existing FHIR mechanisms outside
the BSeR outcome package — e.g., free-standing `Communication` /
`Observation` exchanges, or Task `note` / `output` updates — without
changing the BSeR message structure.

#### Justification

Declining is defensible only if the work group concludes that the
existing outcome package semantics are load-bearing for downstream
processing (state transition, billing, registry submission, etc.) and
that any interim-results channel should live outside the BSeR message
framework to avoid confusing closure semantics. However, this leaves the
reporter's underlying need unaddressed within BSeR itself and pushes the
problem to implementers, which is hard to reconcile with the work group
having already marked the ticket Persuasive.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been triaged Persuasive with a written resolution
committing to "add the ability to send interim in-progress outcomes,"
which directly maps to Disposition A. Accepting the request is the
lowest-friction path to honouring that resolution and addresses the
reporter's clinical use case head-on. Disposition B's concern about
preserving closure semantics is real but can be mitigated within
Disposition A by requiring a clear interim-vs-final qualifier on the
outcome message (e.g., a Task `businessStatus` or a message-event
distinction), so a separate event type is not strictly necessary.
Disposition C contradicts the work group's own Persuasive disposition
and should only be revisited if new information surfaces during review.
The work group should focus discussion on *how* to mark interim vs.
final outcomes (status code, event code, or extension) rather than
*whether* to allow them.
