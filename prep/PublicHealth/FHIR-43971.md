## Summary

The submitter requests that the US Respiratory Virus Hospitalization
Surveillance Network (RESP-NET) FHIR IG be extended to capture **employment
status and job information** for hospitalized patients, modeled on the
profiles already defined in the
[US Occupational Data for Health (ODH) IG](https://hl7.org/fhir/us/odh/).
The rationale is epidemiological: many respiratory viruses are transmitted in
the workplace, so surveillance data without occupational context misses an
important transmission vector. The ticket previously received a
"Considered for Future Use" disposition (vote 31-0-0, 2024-04-18) on the
grounds that the RESP-NET IG mirrors the existing CDC Case Reporting Forms
(CRFs), which do not collect employment status or job information. That
resolution was reverted on 2025-06-20 by Lynn Laakso to be re-opened post
STU1 publication, putting the request back on the work group's agenda. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found in
the Augury cache for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add occupational data to the RESP-NET IG by **referencing the ODH IG
profiles** rather than redefining them locally. Concretely, in the next
RESP-NET version:

1. Add a normative dependency on `hl7.fhir.us.odh` (current published
   version) in the IG's `package.json` / `sushi-config.yaml`.
2. Update the patient-centric profile(s) used by RESP-NET (e.g. the
   RESP-NET Patient profile, derived from US Core Patient) so that the
   bundle/composition / message structure references the relevant ODH
   Observation profiles:
   - `odh-EmploymentStatus` — current employment status (employed,
     unemployed, retired, etc.).
   - `odh-UsualWork` — usual occupation and industry.
   - `odh-PastOrPresentJob` — for relevant exposures during the
     incubation window (optional, 0..*).
3. Mark the new occupational Observations as **MustSupport = false,
   cardinality 0..*** so reporters that do not yet have the data can
   continue to conform.
4. Add a narrative section to the IG's data dictionary / mapping page
   explaining how each ODH element maps onto the current CRF fields
   (and explicitly flagging that the CRFs do not yet collect them, so
   transmission of these elements is conditional on local availability).
5. File a companion request with the CDC RESP-NET CRF maintainers to
   add employment status / occupation fields to the source forms, so
   that the IG additions become populated upstream over time.

#### Justification

The ODH IG already models exactly the data this ticket asks for, and is
the HL7 canonical source for US occupational data; re-using its profiles
avoids duplicate definitions and keeps RESP-NET consistent with other
public-health IGs (e.g., eCR, eCC) that also reference ODH. Making the
elements optional respects the original 2024 vote concern (the CRFs do
not collect this data today) by avoiding a conformance burden on
existing reporters, while still providing a normative landing place for
sites that *can* contribute occupational context. The reverted
resolution and re-open comment from 2025-06-20 indicate the work group
already wants to revisit this post-STU1, so adding it now in an
optional, ODH-aligned form is the lowest-friction way to honour the
re-open.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add ODH profiles into the core RESP-NET IG. Instead, publish a
**RESP-NET "occupational supplement" companion profile / extension
page** in the next minor release that:

1. Defines an extension `respnet-occupational-context` on the RESP-NET
   Patient or Encounter profile, whose value is a `Reference(Observation)`
   slice constrained to the three ODH Observation profiles listed above.
2. Documents the extension as **Trial-Use within RESP-NET**, off the
   conformance critical path, intended for jurisdictions that already
   collect ODH-conformant data.
3. Adds an "Occupational Context" page to the IG narrative that explains
   when and why a sender would populate the extension and how it relates
   to ODH.
4. Keeps the base RESP-NET resources unchanged so that current CRF-aligned
   senders are not affected.

#### Justification

Disposition B preserves the "IG mirrors the CRFs" design principle that
drove the 2024 vote, while still giving early adopters and research
partners a standardised way to attach ODH data when they have it. Using
an extension rather than profile slicing avoids a breaking change to the
RESP-NET Patient profile and means tooling that already handles ODH
Observations can plug in directly. The trade-off vs. Disposition A is
discoverability: an extension is easier to overlook than a first-class
slice, and it pushes the burden of advocating for CRF changes back onto
the requester rather than embedding the ask in the IG itself.

---

### Disposition C: Decline

#### Proposal

Re-affirm the original 2024-04-18 resolution and mark the ticket
**Not Persuasive** (or **Considered for Future Use**) on the basis that
the RESP-NET IG is intentionally a faithful FHIR representation of the
current CDC RESP-NET Case Reporting Forms, which do not collect
employment status, usual occupation, or job history. Direct the
requester to (a) the ODH IG, which they can use directly today, and (b)
the CDC RESP-NET CRF governance process, which is the upstream place to
get occupational fields added; once the CRFs change, the IG will follow.

#### Justification

The IG's stated scope is to support the existing RESP-NET surveillance
workflow, not to extend that workflow. Adding optional ODH elements that
no current CRF populates risks producing "always empty" elements in
production traffic, which is misleading to receivers and adds modelling
maintenance for no current data flow. There is no implementer evidence
in the Augury cache (no Zulip threads, no related tickets, no
GitHub PRs) suggesting a downstream system is ready to send or consume
these data, so the change would be speculative. Declining keeps the
STU1-published profiles stable and preserves the design discipline that
the 31-0-0 ballot reflected.

---

### Recommendation

**Recommended disposition:** B

The 2025-06-20 revert and "re-open post STU1 publication" comment make a
plain Disposition C politically difficult — the work group has already
signalled it wants to take another look. At the same time, the original
ballot concern (the CRFs do not collect this data) has not changed, so
forcing ODH Observations into the conformance critical path of a public
health surveillance IG (Disposition A) risks producing empty slots and
imposing modelling overhead on jurisdictions that have nothing to
report. Disposition B threads the needle: it provides a standardised,
ODH-aligned, RESP-NET-blessed way to attach occupational context for
the (currently small) population of senders that have it, without
breaking existing conformance or pretending the CRFs already cover this
ground. It also leaves a clean upgrade path — if/when the CDC CRFs add
employment and occupation fields, the extension can be promoted to a
first-class slice (effectively Disposition A) in a future major
version.
