## Summary

This ticket — one of a large family of comments filed by Ron G. Parker
against the Vital Records Common Profiles Library (VRCL) FHIR IG —
requests that the IG's locally-defined code systems and value sets be
re-anchored in `terminology.hl7.org` (THO) rather than `hl7.org/fhir`
or any IG-local namespace, in accordance with the TSC-approved
[Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy. The reporter explicitly asks that the Vital Records work group
initiate the UTG (Unified Terminology Governance) process to migrate
the affected value set into THO. The ticket is "Comment 30" in the
imported batch (comment number 46 in the source); the prior
disposition history captured on the ticket notes that the previous
"Considered for Future Use" outcome (Robert Passas / Diana Wright,
23-0-0, 2024-05-30) has been reverted, with the recorded vote rationale
that "Creating a THO Valueset for Yes/No/Unknown/NA is appropriate."
The ticket was reopened post STU2 publication so the work can proceed.
There are no recorded cross-references in Augury for this specific
ticket key, but the BM25 keyword neighbours (FHIR-44303, FHIR-44299,
FHIR-44439, FHIR-44441, FHIR-44453, FHIR-44455, FHIR-44457, FHIR-44426,
FHIR-44230, FHIR-44404, FHIR-44384, FHIR-44383, FHIR-44386, FHIR-44194,
FHIR-44399) are the parallel "comment N" tickets in the same THO-
anchoring sweep and should be considered as a single body of work.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Initiate the UTG process to migrate the VRCL-local "Yes/No/Unknown/NA"
value set (and its underlying code system, if VRCL-local) into THO.
Specifically:

1. Submit a UTG proposal to create the value set under
   `http://terminology.hl7.org/ValueSet/<chosen-id>` (and, if a new
   local code system is being promoted, the corresponding
   `http://terminology.hl7.org/CodeSystem/<chosen-id>`), using the
   contents that comment 30 of FHIR-44230/FHIR-44456 corresponds to.
2. Once UTG publishes the THO artifact(s), update the VRCL IG so the
   profile/element bindings that previously referenced the IG-local
   canonical now reference the THO canonical(s), and remove the
   IG-local definitions to avoid duplication.
3. Coordinate the change with the parallel "comment N" tickets
   (FHIR-44303, FHIR-44299, FHIR-44439, FHIR-44441, FHIR-44453,
   FHIR-44455, FHIR-44457, FHIR-44426, FHIR-44404, FHIR-44384,
   FHIR-44383, FHIR-44386, FHIR-44194, FHIR-44399, and umbrella
   FHIR-44230) so all VRCL-local terminology is migrated in one UTG
   submission window rather than dribbled out one comment at a time.

#### Justification

This is exactly what the TSC-approved
[Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy requires: HL7-defined code systems and value sets used across
HL7 products belong in THO, not in an IG's local namespace. The work
group has already voted (23-0-0) that "Creating a THO Valueset for
Yes/No/Unknown/NA is appropriate," and the earlier "Considered for
Future Use" deferral was reverted on 2024-10-11 specifically so this
work could be done post-STU2. Accepting as requested aligns the IG with
TSC policy, removes duplicate/competing canonicals, and makes the
Yes/No/Unknown/NA concept reusable by other Vital Records (and
non-VRCL) artifacts.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than addressing FHIR-44456 in isolation, treat it as one row in
a single coordinated UTG submission covering **all** comments in the
THO-anchoring sweep against VRCL (FHIR-44230 umbrella plus the
per-comment children listed under "Related Jira Tickets"). The work
would be:

1. Produce one inventory spreadsheet of every VRCL-local code system
   and value set, mapping each row to (a) the corresponding "comment
   N" ticket, (b) whether the concepts already exist in THO (and can
   be re-bound rather than re-defined), and (c) the proposed THO
   canonical URL if a new artifact is needed.
2. File a single UTG proposal package referencing that inventory,
   rather than per-ticket UTG submissions.
3. After THO publication, do one VRCL release that swaps all bindings
   over and deprecates the IG-local artifacts in one cycle, with a
   clear migration note for implementers.
4. Resolve FHIR-44456 (and siblings) "Persuasive with Mod" pointing at
   that single delivery.

#### Justification

The body of FHIR-44456 is generic and identical to its siblings, and
the work group has already expressed policy-level agreement. Doing
this comment-by-comment generates per-ticket UTG submissions, multiple
VRCL releases, and asymmetric migration windows where some bindings
point at THO while others still point at the IG-local namespace —
which is worse for implementers than the current state. Bundling
trades a slightly longer single delivery for a much cleaner
implementer experience and a single round of UTG review. It also lets
TSMG/Vocab review the migration as a coherent set, which is what they
typically prefer for IG-to-THO migrations.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-44456 "Not Persuasive" / "Already Addressed" on the grounds
that the policy compliance request is already captured by the
umbrella ticket (FHIR-44230) and the work group's standing 2024-05-30
position that "Creating a THO Valueset for Yes/No/Unknown/NA is
appropriate." Track the actual UTG submission against the umbrella
only, and close the per-comment children (including FHIR-44456) as
duplicates.

#### Justification

This option avoids 15+ near-identical disposition discussions in the
same call. It is defensible because the per-comment tickets do not add
substantive new information beyond what the umbrella already says, and
the work group has already voted on the principle. The risk is that
"close as duplicate" without naming the specific value set being
migrated loses the per-row traceability that the comment-N
enumeration was designed to give, so this disposition is only
defensible if the inventory recommended in Disposition B is captured
elsewhere (e.g., on the umbrella ticket or in a UTG tracker) before
closing.

---

### Recommendation

**Recommended disposition:** B

The ticket as written is plainly correct on policy: VRCL-local
Yes/No/Unknown/NA-style terminology should live in THO, the work
group has already voted that it is appropriate, and the
"Considered for Future Use" deferral has been explicitly reverted so
that the work can be done post-STU2. So the substantive answer is
"yes." However, FHIR-44456 is one of ~15 keyword-identical comment
tickets in the same sweep, all filed by the same reporter, all asking
for the same migration mechanism (UTG → THO → re-bind in VRCL). Doing
this one comment at a time produces per-ticket UTG churn and an
inconsistent intermediate state for implementers, which is exactly
the harm the TSC terminology policy is trying to prevent. Disposition
B preserves the policy-correct outcome of Disposition A while
delivering it as a single coordinated migration coordinated against
the umbrella ticket (FHIR-44230). Disposition C is tempting for
agenda-management reasons but should only be used if the per-row
inventory is captured before the children are closed; otherwise the
work group loses the traceability that the comment-N enumeration was
created to provide.
