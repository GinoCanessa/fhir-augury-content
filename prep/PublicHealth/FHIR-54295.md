## Summary

The ticket asks the eCR (Electronic Case Reporting) work group to consider
adopting the **US Core Occupation** and **US Core Occupation Industry**
observation profiles in place of — or alongside — the **Occupational Data for
Health (ODH)** profiles that eCR currently references for capturing patient
work-history data. The submitter (Sarah Gaunt) provides no implementation
detail in the description; it is a one-line "consider" request, but it
mirrors a parallel ticket and a more elaborated companion ticket from the
same ballot cycle.

[FHIR-54299](https://jira.hl7.org/browse/FHIR-54299) is a verbatim duplicate
of this request, also filed by Sarah Gaunt against the eCR specification two
minutes later, and should be tracked together with FHIR-54295 (likely a
double-submission). [FHIR-54422](https://jira.hl7.org/browse/FHIR-54422)
("Answer to Notes to Balloters related to ODH profile", reporter Ulrike
Merrick) supplies the substantive analysis behind the ask: the US Core
Occupation profile implements the USCDI Occupation and Occupation Industry
data elements, but it only overlaps with **one** of the observation slices
required by ODH's `Composition.section.entry` (specifically
`odh-PastOrPresentJob`). Merrick's recommendation is therefore that eCR use
the US Core Occupation profile in place of `odh-PastOrPresentJob`, while
**retaining** the other ODH entry slices (Employer, UsualWork,
RetirementStatus, CombatZoneHazardousDuty), since US Core has no equivalent
for them. Together these tickets frame the decision as an alignment /
USCDI-compliance question rather than an outright replacement of ODH.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the eCR IG (in `HL7/case-reporting`) so that the
`odh-OccupationalDataForHealth` Composition profile's
`Composition.section.entry:odh-PastOrPresentJob` slice **also accepts** (or
preferentially binds to) the
`http://hl7.org/fhir/us/core/StructureDefinition/us-core-observation-occupation`
profile, alongside (or in place of) the ODH `odh-PastOrPresentJob` profile.
Where the eCR IG documents how senders represent current/past employment,
add normative narrative directing implementers to use the US Core Occupation
profile (and US Core Occupation Industry where applicable) for the
PastOrPresentJob slot to satisfy USCDI Occupation/Occupation Industry, and
keep the other ODH entry slices
(`odh-Employer`, `odh-UsualWork`, `odh-RetirementStatus`,
`odh-CombatZoneHazardousDuty`) bound to their existing ODH profiles. Treat
FHIR-54299 as a duplicate and resolve it with the same disposition; cite
FHIR-54422 as the analysis basis.

#### Justification

This reflects the position spelled out in FHIR-54422: US Core Occupation
implements USCDI 1:1 for Occupation/Occupation Industry but only overlaps
with one ODH entry slice. Aligning with US Core for that slice gives eCR
USCDI conformance without forcing implementers to maintain two parallel
representations of the same datum, and preserves ODH for the work-history
elements US Core does not cover. The Zulip "ODH codes" thread shows the
ODH terminology stack is being actively refreshed by NIOSH/Dogwood, so the
remaining ODH-only slices remain a sound foundation. A "both profiles
allowed, US Core preferred" binding on the PastOrPresentJob entry is the
standard FHIR pattern (`supportedProfile` / multi-target slicing) for this
kind of alignment.

---

### Disposition B: Alternative Approach

#### Proposal

Do not modify the eCR `Composition.section.entry` slicing in this ballot
cycle. Instead, add an informative section to the eCR IG narrative
explaining the relationship between ODH and US Core Occupation/Industry
profiles, and publish a **mapping** artifact (e.g., a `ConceptMap` or a
narrative cross-walk table) showing which ODH entry slices have US Core
equivalents and which do not. Defer any binding change until the eCR and
US Core work groups jointly agree on whether US Core should grow profiles
for the remaining ODH slices (Employer, UsualWork, RetirementStatus,
CombatZoneHazardousDuty), at which point a single coordinated alignment can
be done across all entry slices.

#### Justification

FHIR-54422 explicitly notes that US Core only covers one of the ODH entry
slices. Accepting the request literally now (Disposition A) creates a
hybrid profile set where one slice points at US Core and the others stay on
ODH, which implementers will reasonably read as an inconsistency. It also
introduces a new dependency on the US Core Occupation profile's required
PHIN VADS value-set binding (`PHVS_Occupation_CDC_ONET-SOC2010_ODH`), which
the "Unable to access Occupation ONETSOC Detail (ODH) value set" Zulip
thread shows has had real-world availability problems. Holding the binding
change until the cross-IG picture is coherent avoids ratifying a
half-aligned profile set during ballot reconciliation, while still
delivering the implementer-facing clarity the request is really asking for.

---

### Disposition C: Decline

#### Proposal

Persuasive — Not Persuasive (or Considered for Future Use). Keep eCR's
existing ODH-only profile set on the
`Composition.section.entry:odh-PastOrPresentJob` slice and decline to add
the US Core Occupation/Industry profiles as alternative or required entry
profiles for this publication. Resolve FHIR-54299 the same way as a
duplicate, and treat FHIR-54422 as out-of-scope advisory content.

#### Justification

Defensible if the work group judges that (a) the request as filed is
non-actionable (one sentence, no proposed normative change), (b) eCR's
public-health use case is fundamentally broader than USCDI Occupation —
ODH was designed to capture combat-zone exposure, retirement status, usual
work, and employer details that drive case-reporting decisions, and US
Core Occupation does not — and (c) introducing a second profile family
mid-ballot would create cardinality and slicing ambiguity for senders and
receivers already conformant to the ODH-based composition. The Zulip
thread on PHIN VADS availability also flags an operational risk in
mandating the US Core profile's required binding. If declined, the work
group should explicitly note that USCDI Occupation/Industry conformance is
already achievable in eCR by populating the existing ODH PastOrPresentJob
observation with the same codes.

---

### Recommendation

**Recommended disposition:** A

The ticket, read together with its companion FHIR-54422, is a concrete and
narrowly-scoped alignment ask: USCDI conformance for Occupation and
Occupation Industry is increasingly a baseline expectation for any US
realm IG, and the US Core Occupation profile is the canonical way to claim
that conformance. Disposition A grants the substance of the request while
respecting the analysis in FHIR-54422 — only the PastOrPresentJob slice is
realigned, the remaining ODH slices are preserved, and ODH continues to
own everything US Core does not cover. Disposition B's hesitation is
reasonable but punts an alignment that implementers and reviewers are
already asking for in ballot comments; Disposition C is hard to defend
when the duplicated submission and the Notes-to-Balloters response together
demonstrate clear demand. The work group should pair the disposition with
(i) a duplicate-resolution on FHIR-54299, (ii) a coordinated note to the
US Core team about extending coverage to the remaining ODH entry slices,
and (iii) confirmation with the terminology team that the
`PHVS_Occupation_CDC_ONET-SOC2010_ODH` value set is reliably hosted before
the binding goes normative.
