## Summary

The reporter asks the BFDR (US Birth and Fetal Death Reporting) IG to
extend the **jurisdiction-level** birth and fetal death reports so they
carry the mother's and father's **current job**, in addition to the
"usual work" / longest-held occupation that BFDR (and VRCL) already
exchange. This ticket is the jurisdiction-report counterpart to its
sibling [FHIR-43974](https://jira.hl7.org/browse/FHIR-43974), which
makes the same "add mother's and father's current job information per
ODH" request at the parent / source-data level. The reporter's own
clarifying comment (2024-01-20) spells out that the change "pertains to
4 sections, two named in FHIR-43974 and two named in FHIR-43975," with
43975 covering the jurisdiction-report sections (and possibly examples).
Both tickets were originally resolved 2024-02-15 as "Considered for
Future Use" based on guidance from Amanda Deering / Cindy Bush at NCHS
(occupation/industry are not currently collected on the facility or
patient worksheets that feed birth/fetal-death certificates, and "usual
work" was retained because it is part of the inter-jurisdictional
element set). Lynn Laakso reopened the ticket on 2024-10-15 "post STU2
publication," so it is now active again for STU3 consideration by the
Public Health WG and the NVSS CoP.

## Proposed Dispositions

(Because no repo briefings are available for the BFDR IG, the
dispositions below name the BFDR IG and its conventional structure
without pinning specific authoring file paths. Implementation-side
file-level decisions should be confirmed against the BFDR source repo
once a `repo-analysis` briefing exists for it.)

### Disposition A: Accept As Requested

#### Proposal

Extend the BFDR (STU3) jurisdiction-level birth report and
jurisdiction-level fetal death report bundles to carry **two new
optional ODH-aligned Observations per parent**:

- `Observation-MotherCurrentOccupation` (current job title)
- `Observation-MotherCurrentIndustry` (current industry)
- `Observation-FatherCurrentOccupation`
- `Observation-FatherCurrentIndustry`

Each profile would mirror the existing BFDR "Usual Work" pattern but
use distinct ODH category codes (PastOrPresentJob with appropriate
effectiveTime semantics indicating *current*) so they can coexist
unambiguously with the existing usual-work observations. The
jurisdiction report Composition / Bundle definitions for both birth and
fetal death would gain optional sections (or extend the existing
parent-demographics section) that reference these observations, and
sample instances would be updated to demonstrate populated and absent
cases. Cardinality should be `0..1` per parent per element to match
"usual work," preserving backward compatibility for jurisdictions that
do not collect this data.

#### Justification

This is exactly what the reporter and the sibling ticket FHIR-43974 ask
for, and it is consistent with the existing BFDR pattern of representing
parent occupational data via ODH-conformant Observations referenced
from the jurisdiction Composition. Making the new observations optional
means jurisdictions that do not yet capture current-job data on their
worksheets are not forced to populate them, which directly addresses
Cindy Bush's NCHS concern that occupation/industry "is not collected
within the facility and patient worksheets" today. The reopen comment
explicitly anticipates STU3 inclusion once the NVSS CoP weighs in, so
adding the elements now (gated on optionality) lets the IG ship with
the structure ready while jurisdictional adoption catches up.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting four new BFDR-specific profiles, **adopt the
existing US Public Health Profiles Library / Occupational Data for
Health (ODH) "PastOrPresentJob" Observation profile by reference** and
add narrative + IG guidance to the BFDR jurisdiction report
Composition stating that, when present, current-job data SHALL be
exchanged as ODH PastOrPresentJob Observations distinguished from the
existing "usual work" observations by `effectiveTime` (open-ended,
ongoing) and/or by an explicit ODH "current job" qualifier. Add a
single parameterized section in the Mother and Father parts of the
jurisdiction Composition that allows 0..* such Observations rather
than fixed slices per data element.

#### Justification

ODH already defines occupation/industry exchange semantics; spinning
up BFDR-local profiles risks profile drift and double-maintenance
against ODH. Referencing ODH directly keeps BFDR aligned with the
broader US public health stack (the same approach BFDR already takes
for "usual work") and avoids hard-coding the cardinality at four
slices, which the reporter's own comment hints may be insufficient
("may also pertain to additional sections, e.g., examples"). Trade-off
vs. Disposition A: jurisdictions get less prescriptive structure (must
discriminate current vs. usual via ODH semantics rather than a
distinct profile URL), which makes validation and tooling slightly
harder but keeps the IG lighter and more future-proof.

---

### Disposition C: Decline (Persistent Considered for Future Use)

#### Proposal

Restore the prior resolution of **Considered for Future Use** without
making STU3 normative changes, on the grounds that NCHS has stated the
underlying data are not currently collected on the facility/patient
worksheets that feed jurisdictional reporting, and that adding them now
would create IG-level optional elements with no real-world data flow
behind them. Document the deferral with a pointer to the planned NVSS
Community of Practice presentation and reopen for STU4 once
jurisdictional collection practices have been polled.

#### Justification

This mirrors the original 2024-02-15 disposition (Saul Kravitz / Erin
Holt: 25-0-2) and the explicit NCHS guidance from Cindy Bush. Adding
optional elements that no jurisdiction populates risks "dead" profile
surface area, harms IG readability, and creates an implementation
expectation that cannot yet be met. Declining at the IG level does not
preclude the underlying ODH PastOrPresentJob observations from being
exchanged ad hoc — implementers can already do so via core FHIR — so
the cost of waiting is low.

---

### Recommendation

**Recommended disposition:** B

Disposition B best honors the post-STU2 reopen (so the request is not
simply re-deferred a second time) while respecting the NCHS feedback
that drove the original deferral. It avoids creating four new
BFDR-local profiles that would duplicate ODH semantics and that
jurisdictions currently have no source data to populate, but it does
add the IG-level *guidance and Composition slot* needed so that any
jurisdiction that *does* collect current-job data has an
inter-jurisdictionally-portable way to exchange it. That is the
smallest change that lets the NVSS Community of Practice and individual
jurisdictions move forward without forcing a hard schema commitment in
STU3, and it is consistent with how BFDR already leans on ODH for
"usual work." Disposition A is a reasonable fallback if the WG
concludes that explicit per-element profiles (rather than guidance over
ODH) are needed for jurisdictional tooling; Disposition C should be
avoided because it repeats the 2024 outcome without progress, and the
ticket has now been explicitly reopened.
