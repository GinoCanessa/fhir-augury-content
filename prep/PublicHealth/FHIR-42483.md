## Summary

The reporter (Lloyd McKenzie) flags that the BSeR IG diagrams depict a
"Retrieve Payload" interaction in which a referral recipient pulls the
nominal batch Bundle of searches that defines what data should be shared
for a given referral type — but the IG never specifies *how* that
retrieval actually happens. The reporter expects this to be a standard
FHIR operation: the requester would pass in a service code (or possibly a
full ServiceRequest, though that may raise PHI concerns) and receive back
the Bundle of search definitions describing the data set appropriate to
that referral type. No such operation is currently defined in the IG.

This ticket sits inside a tightly-coupled cluster of Lloyd McKenzie's
comments against the same BSeR ballot pass:

- [FHIR-42482](https://jira.hl7.org/browse/FHIR-42482) explains that the
  batch-based query / x-fhir-query architecture is only revealed at the
  end of the spec and asks for it to be surfaced earlier — i.e., the
  *what* this ticket asks to formalize an operation around.
- [FHIR-42473](https://jira.hl7.org/browse/FHIR-42473) and
  [FHIR-42467](https://jira.hl7.org/browse/FHIR-42467) argue that the
  Notify-and-Pull pattern (recipient queries after notification) needs
  much more detail, which directly motivates a defined retrieval
  operation.
- [FHIR-42490](https://jira.hl7.org/browse/FHIR-42490) discusses where
  the *filter* of allowed content should live, suggesting the Bundle
  definition retrieved here should be applied server-side at the
  referral source rather than trusted to the recipient — that constrains
  what an operation needs to return.
- [FHIR-41901](https://jira.hl7.org/browse/FHIR-41901) (Persuasive with
  Modification) is the broader request to clarify the link between
  ServiceRequest and the referral response payload, which provides the
  ServiceRequest hook this ticket would key off of.

Together these indicate the WG already accepts that the
batch-query-Bundle approach needs to be made first-class; FHIR-42483 is
the specific ask to nail down the retrieval mechanism as an operation
rather than an unspecified "interaction".

### Disposition A: Accept As Requested

#### Proposal

Define a new FHIR operation (working name `$retrieve-payload-definition`
or `$bser-data-requirements`) on the BSeR IG that takes a referral
service code (e.g., the LOINC/SNOMED code identifying the type of
referral) as an input parameter and returns a Bundle of `Parameters` /
search-definition entries (the same x-fhir-query batch Bundle the IG
already uses internally) describing the data to be shared for that
referral type. The operation would be defined as an
`OperationDefinition` in the IG, invoked at the system level
(`[base]/$retrieve-payload-definition?serviceCode=...`), and explicitly
called out in the interaction diagrams that currently show an
unlabeled "Retrieve Payload" arrow. ServiceRequest itself would *not*
be the input parameter (per the reporter's own PHI caveat); only the
service code would be exchanged at this step. The returned Bundle
matches the existing per-referral-type Bundles described in
FHIR-42482's "batch-based query definition".

#### Justification

This is the most direct interpretation of the reporter's request and
aligns the IG with standard FHIR practice: anything shown as a
client/server interaction in a diagram should be backed by either a
RESTful interaction or a named `OperationDefinition`. Using the service
code (rather than ServiceRequest) as input addresses the reporter's own
PHI concern, and it lets the same operation serve discovery use cases
(e.g., a referral source publishing what it expects, a recipient
checking what it is allowed to send) without disclosing patient
context. It also dovetails with FHIR-42482 (introduce the architecture
earlier) and FHIR-42473 / FHIR-42467 (Notify-and-Pull is the dominant
pattern), and gives FHIR-42490 a concrete place to anchor the
filter-server-side discussion: the same operation can be defined on the
*referral source* so the source both publishes and applies the filter.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of minting a new operation, formalize the retrieval as a
RESTful read against curated, canonical artifacts published by the IG
itself: define one `Bundle` (or `Library` / `Questionnaire`-style)
artifact per referral type, give each a canonical URL of the form
`https://hl7.org/fhir/us/bser/Bundle/data-requirements-{serviceCode}`,
and document that retrieval is a plain `GET` of the appropriate
canonical (or a `Bundle` search by `code`/`identifier`). The
"interaction" in the diagrams would be relabeled as
"GET Bundle/data-requirements-{serviceCode}". Optionally, the IG could
expose a thin `$data-requirements` operation on `ServiceRequest` (per
the existing `Resource/$data-requirements` pattern in core FHIR R5) as
a convenience, but the source of truth would be the published
canonical Bundles.

#### Justification

This avoids inventing a new IG-specific operation when a vanilla
canonical-resource read covers the use case, and it keeps the
"definitions" in version-controlled, balloted artifacts rather than
behind server logic that varies by implementer. It also leverages an
existing FHIR mechanism (`$data-requirements`) the WG can point at
instead of designing a fresh one, reducing implementation burden for
anyone who already supports R5 conformance/data-requirements machinery.
Trade-off vs. A: less flexibility for servers to compute definitions
dynamically (e.g., per-jurisdiction variants), and FHIR-42490's
"filter applied server-side at the referral source" still needs a
separate hook because plain canonical reads are not policy points.
However, for a straightforward "tell me what to send for a referral of
this type" question, this is the lower-ceremony answer.

---

### Disposition C: Decline

#### Proposal

Decline the request to define a dedicated operation. Document
explicitly in the IG that the "Retrieve Payload" arrow in the
diagrams represents an out-of-band exchange (e.g., a configuration step
between trading partners, or use of an existing mechanism such as the
recipient's local copy of the IG-published Bundles) and is not a
runtime FHIR interaction. Update the diagrams and narrative to remove
the implication that a wire-level retrieval is required, and rely on
implementers loading the IG-published x-fhir-query Bundles as static
configuration.

#### Justification

Defensible only on the grounds that the IG's published artifacts are
already sufficient: implementers can ship the Bundles as configuration,
and a runtime operation is unnecessary overhead for a relatively static
catalog that changes at IG-version cadence rather than at request
cadence. This avoids adding new conformance surface area to BSeR and
sidesteps the questions FHIR-42490 raises about who is trusted to apply
the filter. The cost is real, however: the diagram-vs-narrative
mismatch the reporter identified would only be partially resolved
(diagrams updated, but the underlying "how do partners actually
discover what to send" question is pushed to out-of-band agreement),
and FHIR-42473 / FHIR-42467 still need a defined query mechanism for
the data itself, even if discovery of the *definition* is offline.

---

### Recommendation

**Recommended disposition:** A

The reporter's underlying ask — "this is shown as an interaction in the
diagrams; please specify the operation behind it" — is a basic
conformance-quality fix the IG should make regardless of which
mechanism it picks. Disposition A is recommended because it is the
option that simultaneously closes this ticket *and* gives FHIR-42490 a
natural home (the same operation can live on the referral source and
apply the allowed-content filter), aligns with the
Notify-and-Pull narrative requested by FHIR-42473 / FHIR-42467, and
follows the WG's existing direction to use the batch x-fhir-query
Bundle as the architectural anchor (FHIR-42482). Disposition B is
attractive in its simplicity but is weaker on the filter-application
question and gives the WG less control over future per-jurisdiction or
per-implementer variation; it would still leave a gap that FHIR-42490
must fill separately. Disposition C should only be taken if the WG
concludes that the IG's batch Bundles are intended purely as static
configuration, which contradicts how the diagrams are drawn today and
how the related comment cluster reads the spec. Before action, run
`repo-analysis HL7/bser if-stale` so the OperationDefinition can be
authored against the current `input/` layout.
