## Summary

The reporter (Keith W. Boone) asks the Situation Awareness for Novel
Epidemic Response (SANER) IG to incorporate a `ConceptMap` translating
between the HAVBED (HHS / ASPR Hospital Available Beds) bed-location
vocabulary and the CDC bed-location vocabulary, and "perhaps more related
content" mapping between the HAVBED data set and SANER (in both
directions). The ticket is scoped to the SANER FHIR specification. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found
in FHIR Augury for this ticket, so the request stands on its own. The
ticket has already been worked through one cycle in the work group: it
was originally dispositioned "Considered for Future Use" (vote 18-0-0,
Kreisler/Gaunt, 2021-03-25), then re-opened post-STU1 publication for
re-consideration, and was ultimately resolved on 2024-01-11 as
**Not Persuasive** with the rationale that "no interest has been shown
for the HAVBED requirements or codesystems."

## Proposed Dispositions

> Note: this ticket already carries a recorded resolution of
> **Not Persuasive** (2024-01-11). The dispositions below are framed as
> options the work group could *re-affirm or revisit* if FHIR-29446 is
> brought back up for review.

### Disposition A: Accept As Requested

#### Proposal

Author and ship a `ConceptMap` resource in the SANER IG that maps
between the HAVBED bed-location code system and the CDC bed-location
code system. Concretely:

- Add a `CodeSystem` (or `NamingSystem` reference) for the HAVBED bed
  locations if one is not already in the IG, and confirm the canonical
  URL of the CDC bed-location code system that SANER should target.
- Author `ConceptMap/HAVBED-to-CDC-BedLocation` (and optionally the
  inverse `ConceptMap/CDC-to-HAVBED-BedLocation`) covering the HAVBED
  bed-location concepts in scope, using `equivalent` /
  `wider` / `narrower` / `inexact` relationship codes as appropriate.
- Optionally add narrative pages and a couple of `MeasureReport` /
  `Measure` examples in SANER that demonstrate translating HAVBED
  submissions into the CDC-vocabulary measures the IG already supports.

#### Justification

This directly fulfills what Keith requested and would let HAVBED-aligned
implementers participate in SANER measure reporting without re-coding
their bed-location data. ConceptMaps are the standard FHIR mechanism
for exactly this kind of vocabulary bridging, and SANER, as a
situational-awareness IG built around bed availability, is the natural
home for them. It also concretely answers the "more related content"
half of the request by adding worked examples.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than carry HAVBED↔CDC mappings inside the SANER IG itself,
publish them in a smaller, dedicated terminology/mapping artifact (for
example a Public Health work-group "bed-location vocabulary mappings"
companion IG, or a UTG submission for the HAVBED code system plus a
standalone `ConceptMap`), and have SANER reference those canonicals
rather than own them. The SANER IG would link to the external
`ConceptMap` from its terminology page and use it in narrative
guidance, but would not bundle it.

#### Justification

The HAVBED↔CDC mapping is fundamentally a terminology concern, not a
SANER-specific measurement concern; keeping it in a separate artifact
lets it be re-used by any other Public Health IG that ingests HAVBED
data and avoids coupling SANER's release cycle to changes in either
source vocabulary. It also makes governance cleaner — UTG / a dedicated
terminology artifact is the more conventional FHIR home for cross-code
system maps than an event-response IG.

---

### Disposition C: Decline (re-affirm "Not Persuasive")

#### Proposal

Re-affirm the existing 2024-01-11 disposition of **Not Persuasive**:
do not add HAVBED↔CDC ConceptMaps or related HAVBED content to the
SANER IG. Close the ticket with the existing resolution text, "No
interest has been shown for the HAVBED requirements or codesystems."

#### Justification

The ticket has been on the books since 2020 and went through at least
two work-group passes (2021 deferral, 2024 Not Persuasive). The work
group's recorded rationale is that no implementer community has shown
sustained interest in the HAVBED requirements or its code systems, and
the ticket itself provides no concrete use case, scope of concepts to
be mapped, or volunteer to author the mappings. Adding terminology
artifacts that nobody is asking to consume creates ongoing
maintenance burden (re-publication on every SANER release, keeping the
mappings synchronised with HAVBED / CDC vocabulary changes) for
unclear benefit, which is a defensible reason to decline.

---

### Recommendation

**Recommended disposition:** C

The recorded history on this ticket — a 2021 deferral followed by a
2024 "Not Persuasive" with the explicit observation that no one is
asking for HAVBED tooling — is the strongest signal here. There are no
cross-referenced Jira tickets, no Zulip discussion, and no GitHub
activity on FHIR Augury for this request, which is consistent with the
work group's read that implementer demand never materialised. Without
a sponsor willing to author and maintain the ConceptMap(s) and at least
one consumer willing to use them, Disposition A would create
maintenance debt for the SANER editors, and Disposition B only makes
sense if the underlying demand exists somewhere else in Public Health
— which the available evidence does not support. The work group should
re-affirm Not Persuasive, and leave the door open for a future ticket
if a HAVBED-aligned implementer steps forward with a concrete mapping
and use case.
