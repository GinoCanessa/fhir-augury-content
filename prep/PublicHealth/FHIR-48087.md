## Summary

The reporter requests that the temporary "Healthcare Capacity Reporting Code System"
defined in the NHSN dQM Reporting IG be replaced with equivalent LOINC codes, so that
the IG relies on the published LOINC terminology rather than carrying its own ad-hoc
local code system. The ticket itself contains only a one-line change request and no
implementer-supplied LOINC mappings. No Jira, Zulip, GitHub, or Confluence
cross-references were surfaced for this ticket, so there is no community discussion
record tied to it. The Public Health work group has already triaged the request and
recorded a resolution of "Not Persuasive" with the note that the code system in
question "has moved to the US SAFR IG" — i.e., the artifact the ticket targets is no
longer maintained inside the NHSN dQM IG. A keyword search for "US SAFR IG" surfaces a
cluster of related dQM/US SAFR tickets (e.g., [FHIR-50304](https://jira.hl7.org/browse/FHIR-50304),

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Within the NHSN dQM Reporting IG, retire the local
"Healthcare Capacity Reporting Code System" CodeSystem and re-bind every
ValueSet that draws from it to the equivalent LOINC concepts
(numerator/denominator hospital bed and ICU capacity observations, occupancy
counts, staffed-bed counts, ventilator availability, etc.). Concretely:
identify each concept in the local CodeSystem; map it 1-for-1 to a LOINC
code (using the LOINC panels already curated for hospital capacity / NHSN
COVID-19 reporting where they exist, and submitting LOINC requests for
genuine gaps); update each affected CodeSystem/ValueSet/ConceptMap and any
example Observations / Measure expressions; deprecate the local CodeSystem
with an explanatory note pointing to the new LOINC bindings; and ship a
ConceptMap from the legacy local codes to the new LOINC codes for
implementer migration.

#### Justification

LOINC is the FHIR-preferred terminology for laboratory and clinical
observations, including the NHSN hospital-capacity observation set, and is
already used widely across U.S. public-health reporting (including prior
NHSN COVID-19 capacity reporting). Replacing a temporary local code system
with LOINC eliminates a known terminology-debt item, removes the
maintenance burden of an IG-local CodeSystem, and improves
interoperability with downstream systems that already speak LOINC.

---

### Disposition B: Alternative Approach

#### Proposal

Do not modify the NHSN dQM Reporting IG. Instead, treat this ticket as
already addressed by the relocation of the Healthcare Capacity Reporting
content into the US SAFR IG (per the work group's resolution note), and
file/forward the LOINC-replacement work against the US SAFR IG so the
LOINC bindings are introduced where the content now lives. In the NHSN dQM
IG, leave the local CodeSystem deprecated/marked as moved, and add a
narrative pointer from the dQM IG to the US SAFR IG so implementers find
the LOINC-bound successor artifacts.

#### Justification

The work group's recorded rationale ("This code system has moved to the
US SAFR IG") makes any in-place LOINC-binding change in the NHSN dQM IG
either redundant or actively misleading — implementers would have two
parallel terminologies for the same hospital-capacity concepts in two IGs.
Performing the LOINC migration in the IG that actually owns the content
going forward (US SAFR) avoids divergence, keeps a single authoritative
binding, and lets the NHSN dQM IG retire the temporary CodeSystem cleanly
without paying the cost of mapping/binding/example updates that will not
be carried forward.

---

### Disposition C: Decline

#### Proposal

Reject the ticket as Not Persuasive against the NHSN dQM Reporting IG, on
the grounds that the targeted artifact (Healthcare Capacity Reporting
Code System) is no longer in scope for this IG. Close with a brief
disposition note explaining the relocation to the US SAFR IG and pointing
the requester to that IG if they wish to pursue LOINC bindings for the
same content. (This matches the resolution already entered.)

#### Justification

The ticket asks for a change to a CodeSystem that is no longer maintained
in the NHSN dQM IG. Acting on the request inside this IG would either
revive an artifact that has been intentionally moved out, or duplicate
work that more properly belongs to the US SAFR maintainers. Declining is
defensible because the requester's underlying need (LOINC-bound capacity
reporting) is not made worse by closing the ticket against this IG; it
simply needs to be raised against the IG that now owns the content.

---

### Recommendation

**Recommended disposition:** B

The work group has already concluded that the Healthcare Capacity Reporting
Code System has moved out of the NHSN dQM Reporting IG and into the US SAFR
IG, and that conclusion is consistent with the broader dQM / US SAFR
boundary work visible in surrounding tickets (FHIR-50304, FHIR-50655,
FHIR-50476, FHIR-50330). A pure "Decline / Not Persuasive" close
(Disposition C, the current resolution) is technically defensible but
leaves the requester's legitimate ask — get this content onto LOINC — with
no clear next step and no audit trail across IGs. Disposition A is wrong
for this IG: doing the LOINC remap here would re-introduce content that
was intentionally relocated. Disposition B preserves the work group's
existing close on the dQM IG side, but pairs it with an explicit forward
of the LOINC-replacement work to the US SAFR IG (and a narrative pointer
in the dQM IG so implementers aren't left guessing). That gives the
workgroup a clean disposition while ensuring the underlying terminology
cleanup actually lands somewhere.
