## Summary

The ticket asks the Public Health work group to remove the `MustSupport`
flags currently asserted on the slices of `ServiceRequest.identifier`
(specifically the slices that pin the identifier `type` to `PLAC`
(placer) and `FILL` (filler)) in the BSeR Referral Service Request
profile. The reporter argues that requiring implementers to support
these specific identifier slices is overly constrained for the BSeR
use case, because other elements on `ServiceRequest` and the related
`Task` already model the actor / object relationships in the eReferral
state machine that PLAC/FILL identifiers would otherwise carry. No
related Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-references for FHIR-42223; the only cross-reference found was a
mention of the FHIR element `ServiceRequest.identifier` itself.

The ticket has been resolved as **Persuasive**, with a resolution note
indicating that the BSeR Referral Service Request profile will be
updated to remove the `MustSupport` constraints on the `identifier`
slices, and that the work group will additionally reconsider whether
the recipient and referral identifiers need to be recorded at all or
whether existing `ServiceRequest` elements already cover that need.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the BSeR (US Bidirectional Services eReferrals) IG, edit the
`ReferralServiceRequest` (Referral Service Request) profile so that the
slices defined on `ServiceRequest.identifier` that pin
`identifier.type` to `PLAC` and `FILL` are no longer flagged
`mustSupport = true`. Concretely, in the StructureDefinition (the
profile's authoring source), change `mustSupport` from `true` to
`false` (or remove the element where the only purpose of the
differential entry was to set MustSupport) on:

- `ServiceRequest.identifier:placer` (and any `placer.type`,
  `placer.system`, `placer.value` sub-elements pulled in just to carry
  MustSupport),
- `ServiceRequest.identifier:filler` (and equivalent sub-elements).

Leave the slice *definitions* (slicing discriminator + the slices
themselves) intact so that implementers who do choose to send placer
or filler identifiers continue to do so in a structurally consistent
way. Update the profile's narrative / "Notes" section to explain that
PLAC/FILL identifiers are optional in BSeR and that linkage between
actors and the referral is carried by existing `ServiceRequest`
elements (e.g. `requester`, `performer`, `subject`, `basedOn`,
`replaces`) and by the BSeR `Task` resources in the workflow.

#### Justification

This matches the resolution already recorded on the ticket
("Persuasive — will update the Referral Service Request profile and
remove the MustSupport constraints on Slices for *identifier*"), and
is the smallest change that addresses the reporter's concern. FHIR's
guidance on `mustSupport` is that profile authors should only flag an
element as MustSupport when downstream behaviour depends on the
implementer being able to handle it; the reporter's point is that the
BSeR workflow already conveys placer/filler semantics via other
ServiceRequest and Task elements, so demanding MustSupport on these
identifier slices imposes implementation cost (consumers must be able
to read and interpret PLAC/FILL slices) without a corresponding
behavioural requirement in the IG. Removing MustSupport (rather than
removing the slices) is also non-breaking for senders that already
populate them.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only relaxing MustSupport on the existing PLAC/FILL slices,
remove the slicing on `ServiceRequest.identifier` entirely from the
`ReferralServiceRequest` profile (or demote it to an informative
example) and instead add explicit profile guidance — and where useful,
invariants — that direct implementers to use the dedicated
`ServiceRequest` elements (`requester`, `performer`, `subject`,
`basedOn`, `replaces`) and the BSeR `Task` resources for
actor/workflow linkage. If a placer/filler identifier is genuinely
needed in some BSeR exchanges, expose it via an optional, unsliced
`identifier` with documentation on how to use HL7 v2 PLAC/FILL codes
in `identifier.type`, rather than as named, normatively-MustSupported
slices.

#### Justification

The ticket's underlying concern is not just MustSupport, but whether
the placer/filler identifiers serve any functional purpose in BSeR at
all — the reporter explicitly says "other elements exist to link the
task to actors and objects in the state machine", and the resolution
note repeats this by asking whether the identifiers "need to be
recorded at all". If the answer is "no", carrying named slices in the
profile (even without MustSupport) is dead weight that future
implementers will still discover, infer significance from, and
populate inconsistently. Removing the slices removes that ambiguity
and keeps the profile aligned with the actual BSeR information model.
The trade-off vs. Disposition A is implementation churn for any
existing BSeR sender that already populates the slices, and a slightly
larger conformance change.

---

### Disposition C: Decline

#### Proposal

Keep the `ServiceRequest.identifier` PLAC and FILL slices, and keep
their `mustSupport = true` flag, in the BSeR `ReferralServiceRequest`
profile. Close the ticket as "Not Persuasive — existing constraints
serve a documented interoperability purpose" and instead clarify in
the profile narrative why the placer and filler identifiers are
required to be supported (e.g. for reconciliation with HL7 v2-based
order entry / order filler systems on the boundary of the eReferral
exchange).

#### Justification

PLAC/FILL identifier types are widely used to bridge FHIR
ServiceRequest with HL7 v2 ORM/ORR-style placer/filler order workflows
that many participating jurisdictions and EHRs still operate. If BSeR
implementers are expected to interoperate with those legacy systems at
the edges of the public-health referral, MustSupport on the slices
ensures that consumers can at least read and round-trip placer/filler
identifiers when they appear, which is exactly the behavioural
guarantee MustSupport is intended to provide. Declining preserves
backward compatibility with current BSeR implementations and avoids
re-opening a settled design choice.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been adjudicated as Persuasive, the resolution
text on the Jira issue commits to exactly the change in Disposition A
("remove the MustSupport constraints on Slices for *identifier*"), and
this is the minimum-impact change that addresses the reporter's
concern without breaking any sender that currently populates PLAC/FILL.
Disposition B (removing the slices entirely) is more architecturally
clean and lines up with the resolution's secondary question about
whether the identifiers are needed at all, but it is a larger
conformance break and should be handled as a follow-up ticket once the
work group has decided whether `requester` / `performer` / `Task`
linkage is sufficient on its own; tying it into this ticket risks
delaying a change the work group has already agreed to. Disposition C
is inconsistent with the recorded Persuasive resolution and with the
reporter's (uncontested) observation that the BSeR workflow already
expresses placer/filler semantics through other elements, so it is not
recommended.
