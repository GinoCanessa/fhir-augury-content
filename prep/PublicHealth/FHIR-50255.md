## Summary

Saul Kravitz reports an inconsistency in how the BFDR Compositions reference
the mother and father between the **submission** Composition profiles
(Provider Live Birth Report, Provider Fetal Death Report) and the **coded**
Composition profiles (e.g., Coded Race & Ethnicity, Coded Industry &
Occupation). After his self-correcting comment, the corrected statement of
the issue is:

- The **submission** Composition references the mother/father as
  `Composition.section.focus` in the appropriate sections.
- The **coded** Composition references the mother/father as
  `Composition.section.entry` in the appropriate sections.

He notes that the published **examples** for the coded Composition were
already aligned with the submission pattern (i.e., examples used
`section.focus`) and that he has fixed the examples in the build version.
The remaining question is whether the *profiles* themselves should be
brought into alignment so that submission and coded Compositions use the
same mechanism (focus vs. entry) to point at the parent(s).

This is the third in a small thread of related Public Health tickets that
have iteratively tightened how mother/father references are conveyed in
BFDR section structures: see [FHIR-30189](https://jira.hl7.org/browse/FHIR-30189)
(made `section.focus` required, 1..1, on the coded race profile) and
[FHIR-30191](https://jira.hl7.org/browse/FHIR-30191) (asked whether
mother/father section focus should be required across all compositions
and whether RelatedPerson should be allowed). Both were resolved
"Persuasive" and Published. FHIR-50255 surfaces residual drift between the
submission and coded composition families that those earlier changes did
not fully reconcile.

## Proposed Dispositions

Note: `Composition.section.focus` (Reference, 0..1 in base FHIR) is the
"primary subject" of a section when it differs from the Composition's
overall `subject`. `Composition.section.entry` is the list of
references to the resources whose narrative is summarized in the
section. Aligning the two BFDR Composition families means picking which
of these slots carries the mother/father pointer — and applying the
same choice consistently across both submission and coded Compositions.

### Disposition A: Accept As Requested

#### Proposal

Reconcile the BFDR Composition profiles so that **all** mother/father
linkages in the appropriate sections are conveyed via the same element
in both the submission and the coded families. Concretely (assuming we
align *to the coded side*, where `section.focus` was already tightened
to 1..1 by FHIR-30189):

1. In each `Composition - Provider Live Birth Report` and `Composition -
   Provider Fetal Death Report` section that today carries the
   mother/father reference as `section.entry` (mother / motherInformation,
   father / fatherInformation, and any analogous sections), add a
   `section.focus` slice/constraint that is 1..1 and constrained to the
   appropriate target type (`Patient-Mother` for mother sections;
   `RelatedPerson-Father-Natural` for father sections) — mirroring the
   pattern established by FHIR-30189 for the coded race profile.
2. Where the same reference is currently carried only as `section.entry`,
   either remove it from `entry` (so `focus` is the single source of
   truth) or downgrade `entry` to optional and document that `focus` is
   the authoritative pointer.
3. Audit every other Composition profile in the IG (Coded Cause of
   Fetal Death, Coded Industry & Occupation, etc.) and apply the same
   `focus`-only pattern. Re-publish the affected
   `StructureDefinition-Composition-*.json` (or `.fsh`) sources in
   `HL7/fhir-bfdr` and regenerate snapshots.
4. Update narrative pages and intro `*.md` files to state explicitly:
   "Mother/father references are conveyed via `section.focus`; they are
   not duplicated in `section.entry`."
5. Confirm Saul's already-applied example fix is consistent with this
   choice, and update remaining example bundles accordingly.

#### Justification

This is what the ticket is literally asking for — internal consistency
across the IG. It also extends the trajectory established by
[FHIR-30189](https://jira.hl7.org/browse/FHIR-30189) ("require
`section.focus` on coded race profile") and
[FHIR-30191](https://jira.hl7.org/browse/FHIR-30191) ("review all
subject/focus mappings"), both Persuasive/Published, so it lands
inside an established design direction the workgroup has already
endorsed. Picking `section.focus` (rather than `section.entry`) as the
canonical slot is consistent with the FHIR base definition of `focus`
as "what the section is about" when the section subject differs from
the composition subject — which is exactly the mother/father case.

---

### Disposition B: Alternative Approach

#### Proposal

Use **both** `section.focus` *and* `section.entry` consistently, but in
distinct semantic roles, and document that contract explicitly:

- `section.focus` (1..1, profile-constrained to `Patient-Mother` /
  `RelatedPerson-Father-Natural`) identifies *who the section is about*.
- `section.entry` (1..*) lists the actual data resources (Observations,
  Conditions, Procedures, etc.) that constitute the section's content;
  the parent reference is **not** duplicated in `entry`.

Apply this to **every** BFDR Composition (submission and coded) so that
the inconsistency disappears not by removing one of the two elements
but by giving them disjoint, documented responsibilities. Add a
constraint (FHIRPath invariant) on the affected sections asserting that
`entry` does not contain a Reference to the same resource as `focus`.

Optionally allow `section.focus` to be either `Patient` or
`RelatedPerson` per FHIR-30191's earlier discussion, with profile-level
slicing constraining the choice per section.

#### Justification

This preserves the expressive distinction between "who" (focus) and
"what data" (entry) that FHIR's Composition model was designed for,
rather than collapsing everything into one slot. It also avoids a
breaking change for any implementer who today populates `entry` with
the parent reference on the coded side — they keep doing what they're
doing for the *data* entries, and just add `focus` for the parent. The
new invariant prevents the duplication that caused this ticket to be
filed in the first place. Trade-off vs. A: more text in the
StructureDefinitions and a new invariant, but fewer behavioural changes
for existing implementers and a cleaner long-term semantic model.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** on the basis that (a) the example fixtures
have already been corrected in the build version per Saul's first
comment, and (b) the submission and coded Composition families serve
different downstream consumers (jurisdictional EBRS submission vs.
NCHS-bound coded payloads) and there is no real interoperability harm
from the two families using `focus` vs. `entry` differently for
mother/father. Document the existing pattern in the IG narrative
(without changing profiles) so that implementers know what to expect
in each Composition family, and move on.

#### Justification

The mechanical inconsistency does not, on its own, prevent
implementations from validating or exchanging instances; the receiving
profiles in each pipeline already know which slot to read. Changing
the `Composition - Provider Live Birth Report` /
`Composition - Provider Fetal Death Report` profiles to require
`section.focus` is potentially a breaking change for jurisdictional
submitters who have built against the existing submission Composition
shape (FHIR-30191 explicitly noted that the coded path can't accept
`RelatedPerson - Mother` for downstream NCHS reasons — symmetric
considerations may apply on the submission side). Declining keeps
implementer cost low and confines the fix to documentation and
examples, which is in fact what has already happened.

---

### Recommendation

**Recommended disposition:** A

The BFDR workgroup has already, in two prior tickets
([FHIR-30189](https://jira.hl7.org/browse/FHIR-30189),
[FHIR-30191](https://jira.hl7.org/browse/FHIR-30191)), committed to a
design where `Composition.section.focus` is the canonical slot for the
mother/father pointer (going so far as to make it 1..1 on the coded race
profile and to ask for a review of all subject/focus mappings). Leaving
the submission Compositions out of that alignment perpetuates the very
"review all subject/focus mappings" debt that FHIR-30191 left open, and
it is what generated this ticket. Disposition A finishes the work, lines
up the IG with its own established direction, and matches the example
fixtures Saul has already applied in the build version — so for most
implementers tracking the build, A actually *reduces* surprise.

Disposition B is the strongest alternative and should be the fallback if
the workgroup discovers, during the change, that `section.entry` in some
existing submission Composition genuinely encodes content that isn't
just the parent reference (in which case keeping both elements with
distinct contracts is safer than removing `entry`-as-parent outright).
Disposition C is hard to defend now that two prior Persuasive tickets
have already moved the IG in the opposite direction; "documentation
only" would re-open the same inconsistency for the next reviewer.
