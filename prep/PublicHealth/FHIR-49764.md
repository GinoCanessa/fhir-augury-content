## Summary

The submitter (Saul Kravitz) reports that current Inter-Jurisdictional
Exchange (IJE) submissions carry two distinct "unknown" sentinel values
in the state/territory/province field that today cannot be represented
in the FHIR Vital Records value set bound to that field:

- **`XX`** — "Country is Canada but the Canadian province is unknown,
  **or** any country that is known but is neither U.S. nor Canada."
- **`ZZ`** — "Unknown or blank U.S. state or territory, **or**
  unknown/unclassifiable country."

Because these sentinels are absent from the
StatesTerritoriesAndProvinces value set used by the Vital Records
Common Profiles Library (VRCL) — and inherited by both VRDR (death
reporting) and BFDR (birth/fetal-death reporting) — IJE→FHIR→IJE
round-tripping is lossy: jurisdictions cannot faithfully signal
"unknown province" or "unknown state" in FHIR. The submitter notes
that the parallel **Country** value set already carries `ZZ` for the
analogous purpose, so the precedent and pattern are established. The
ticket asks the workgroup to mirror that pattern in the
state/territory/province value set, and flags that **technical
corrections to all three IGs (VRCL, VRDR, BFDR)** are likely required
because the value set lives in VRCL but is consumed by the two
downstream IGs.

The cross-reference index returned no direct links from this ticket to
other Jira items, Zulip threads, or GitHub PRs/issues. However,
keyword-similarity surfaces a tight cluster of historically resolved
tickets against the same value set (FHIR-33152 added the `YC` New
York City code; FHIR-36404 normalized State/Jurisdiction encoding to
strings bound to that value set; FHIR-33193 dealt with the related
ISO 3166-2 problem on `birthState`; FHIR-44462 covered the broader
renaming of non-VR-specific terminology artifacts). The current
ticket is a direct continuation of that line of work — extending the
same value set with two more administrative sentinel codes rather
than re-architecting the binding.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add two new concept codes — `XX` and `ZZ` — to the
StatesTerritoriesAndProvinces code system / value set maintained in
the **Vital Records Common Profiles Library** (the artifact named in
the ticket's Specification field, and the artifact FHIR-44462
established as the home for shared VR terminology). Definitions
should mirror the IJE wording precisely:

- `XX` — "Country is Canada but Canadian province is unknown, or any
  country that is known but is not U.S. or Canada."
- `ZZ` — "Unknown or blank U.S. state or territory, or
  unknown/unclassifiable country."

Apply the change as a **technical correction** in VRCL, then issue
matching technical-correction releases of VRDR and BFDR so the
extended value set propagates to all three IGs as called out in the
ticket. Update the value-set narrative to describe the sentinels and
to point implementers at the corresponding `ZZ` already present in
the Country value set so the symmetry is documented.

#### Justification

This is the lowest-friction, highest-fidelity fix. It restores
lossless IJE↔FHIR round-tripping for the two real, in-the-wild
sentinel values the submitter cites, without changing the binding
shape that FHIR-36404 already standardized (string bound to a
2-letter postal-code value set, plus YC). It mirrors a precedent the
workgroup already accepted for the parallel Country value set (which
already contains `ZZ`), so the design rationale is consistent.
Because VRCL owns the artifact (per FHIR-44462), a single edit
flows to both VRDR and BFDR, matching the submitter's expectation
that "technical corrections to all 3 IGs are probably necessary."
The change is additive and backwards-compatible — existing instances
remain valid; new instances gain the ability to represent
"unknown/non-US/non-Canada" jurisdictions explicitly.

---

### Disposition B: Alternative Approach — Use a Coded Null Flavor Pattern

#### Proposal

Instead of overloading the postal-code value set with two
non-postal sentinels, model "unknown jurisdiction" using the FHIR
DataAbsentReason / NullFlavor pattern. Specifically: keep the
StatesTerritoriesAndProvinces value set restricted to genuine
jurisdiction codes (postal + YC), and require that when the IJE
sender meant `XX` or `ZZ`, the FHIR instance carry the field with
`extension[dataAbsentReason]` set to:

- `unknown` (for IJE `ZZ`), and
- a new `dataAbsentReason` concept (or `unknown` plus a country-code
  qualifier) for IJE `XX`.

Document an explicit IJE↔FHIR mapping table in the VRCL
implementation guidance so converters round-trip the two sentinels
deterministically without putting non-postal codes inside the
postal-code value set.

#### Justification

This keeps the postal-code value set semantically clean — `XX` and
`ZZ` are not really jurisdictions, they're "I don't know which
jurisdiction," and FHIR already has a first-class mechanism
(`DataAbsentReason`) for exactly that. It avoids the long-tail
problem of downstream consumers (mappers, terminology services,
display logic) treating `XX`/`ZZ` as if they were postal codes and
trying to render them as territories. However, it requires (a)
implementers to understand and emit the extension, (b) a defined
mapping for the `XX` "known non-US/non-Canada country" case which
`DataAbsentReason.unknown` does not cleanly capture (it conflates
"unknown province" with "known foreign country"), and (c) a
conversion utility on every IJE↔FHIR boundary. Compared to
Disposition A this is more architecturally pure but materially more
work for VRDR/BFDR implementers and arguably loses information
relative to A because the two distinct IJE meanings collapse onto a
single `DataAbsentReason.unknown` unless a new concept is also
minted.

---

### Disposition C: Decline

#### Proposal

Do not change the value set. Direct submitters to either (a) leave
the field absent when the jurisdiction is unknown and rely on the
country field plus `DataAbsentReason` already available in FHIR, or
(b) carry the IJE sentinel in a separate IJE-specific extension that
is stripped on conversion to canonical FHIR.

#### Justification

A principled "decline" rationale would be that postal-code value
sets should contain postal codes only, and that round-tripping a
non-FHIR legacy format is not, by itself, justification to mint
non-postal "codes" inside a postal value set. However, this
position is **weak** in this specific case because (i) the workgroup
has already accepted the same accommodation in the parallel Country
value set (`ZZ`), so consistency would have to be rolled back there
too, (ii) FHIR-36404 explicitly accepted the precedent of mixing a
non-postal entry (`YC`) into the same value set, and (iii) the
ticket reports a real, current data-loss problem on production IJE
exchanges, not a hypothetical one. Decline is recorded for
completeness but is not the recommended path.

---

### Recommendation

**Recommended disposition:** **A** (Accept As Requested).

Disposition A is the right call. The workgroup has already
established — twice — that this exact value set is allowed to carry
non-postal entries when there is a real interoperability need
(`YC` for NYC via FHIR-33152, blessed by FHIR-36404), and the
parallel Country value set already carries `ZZ` for the analogous
"unknown/unclassifiable" case. Refusing the same accommodation here
would be inconsistent both internally (Country vs.
State/Territory/Province) and historically (precedent set by
FHIR-33152/36404). VRCL is the right home for the change per
FHIR-44462, so a single technical-correction edit there propagates
to both VRDR and BFDR exactly as the submitter anticipates. The
DataAbsentReason alternative (Disposition B) is conceptually
cleaner but materially harder for implementers, loses the
distinction between IJE `XX` and `ZZ` unless a custom concept is
also minted, and creates a special case that diverges from how the
sibling Country value set already handles the same problem. The
workgroup should adopt Disposition A and process it as a coordinated
technical correction across VRCL, VRDR, and BFDR.
