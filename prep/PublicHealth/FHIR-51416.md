## Summary

The reporter (NatanShpringman) requests that the US Vital Records Death
Reporting (VRDR) FHIR IG add additional, explicitly-listed use cases
covering an EHR → EDRS workflow in the IG's Background section. Today,
the Background appears to under-document how data flows from an
Electronic Health Record (EHR) into an Electronic Death Registration
System (EDRS) — for example, when a clinician or hospital initiates the
death record from the encounter system rather than from within the EDRS
itself. A subsequent comment from cynthia_bush on 2025-09-23 confirms
the work group's intent: a use case and an accompanying workflow diagram
will be added to support the EHR-EDRS scenario. No external Jira,
Zulip, or GitHub cross-references were surfaced for this ticket, so the
scope is whatever VRDR editorial work is needed to satisfy the
clarified Background content.

### Disposition A: Accept As Requested

#### Proposal

Expand the VRDR IG Background section to add a dedicated "EHR → EDRS
Workflow" subsection that enumerates, at minimum, the following
narrative use cases:

1. **Clinician-initiated death notification** — a hospital/EHR detects
   patient death in an inpatient encounter and pushes initial
   demographic, encounter, and clinical data (Patient, Encounter,
   Practitioner, Location, Condition/Observation pertinent to cause of
   death) to an EDRS for the certifier to complete the certificate.
2. **Pronouncement-only flow** — a pronouncing clinician in the EHR
   supplies pronouncement data (date/time, location) that an EDRS uses
   to seed a record completed later by a separate certifier.
3. **Cause-of-death authoring in the EHR** — the certifying clinician
   drafts cause-of-death information in the EHR and transmits it to
   the EDRS for jurisdictional review and registration.
4. **Supplemental clinical data pull** — the EDRS, after a record is
   opened, requests additional clinical context (medications, problem
   list, recent encounters) from the EHR to support certifier
   review.

Add a workflow diagram (the figure cynthia_bush already committed to
producing) that depicts the EHR, the certifier, the EDRS, and the
state vital records jurisdiction, and shows which VRDR Bundles /
profiles are exchanged at each step. Cross-link the new subsection
from the existing "Actors" / "Workflow" content in the IG so a reader
encounters it in context.

#### Justification

This is exactly what the reporter asked for and what the work group
already signaled it would do in the 2025-09-23 comment. The Background
section is narrative-only — no profiles, value sets, or capability
statements change — so the editorial cost is low and the risk to
implementers is essentially nil. Listing concrete EHR→EDRS use cases
also improves discoverability for EHR vendors who typically read the
Background first to decide whether VRDR is in scope for them, which
directly supports the IG's adoption story. Aligning the new prose with
an explicit workflow diagram follows the pattern used elsewhere in
HL7 Public Health IGs (e.g., eCR, CRD), where a visual workflow
anchors the narrative.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than embedding a list of EHR→EDRS use cases inline in the
Background, create a new dedicated **"Use Cases"** page in the IG (a
sibling of Background and Actors) and place all use cases — including
the existing funeral-director-initiated and jurisdiction-initiated
flows as well as the new EHR→EDRS scenarios — there. Reduce the
Background to a brief pointer ("see Use Cases for detailed
scenarios"). Each use case page entry would have a short prose
description, the actors involved, the FHIR Bundles/profiles
exchanged, and (where helpful) a sequence diagram. The EHR→EDRS
workflow diagram cynthia_bush proposed becomes one of several diagrams
hosted on this consolidated page.

#### Justification

Use-case content tends to grow over successive ballots; if every new
workflow is appended to Background, that section becomes a catch-all
and loses its orienting function. Splitting use cases out is a
well-established pattern in larger HL7 IGs (US Core's "Must Support",
USCDI alignment pages; eCR's Use Cases page) and gives the work group
a stable home for future scenarios (e.g., medical-examiner workflows,
fetal-death variants) without re-litigating Background structure each
time. Trade-off vs. Disposition A: this is a slightly larger editorial
change (one new page, plus minor refactor of Background) and could
delay the next publication if it is not in scope for the current
cycle. The reporter's literal request — "in the Background section" —
is also not honored, which the work group should weigh against the
long-term IG ergonomics.

---

### Disposition C: Decline

#### Proposal

Decline to add new EHR→EDRS use cases to the Background, on the basis
that VRDR's normative payload (Death Certificate Document Bundle, Cause
of Death Coding Content Bundle, etc.) is transport- and
originator-agnostic and the Background already states that any
authoring system (EHR, EDRS, funeral home system) may produce the
data. Resolve the ticket as "Not Persuasive — already addressed by
existing narrative" and point the reporter at the current Actors /
Workflow content.

#### Justification

This would be defensible only if the reviewer believed the existing
Background already conveys the EHR→EDRS workflow with enough
specificity for implementers, and that adding more prose would create
redundancy or imply EHR-specific conformance requirements that VRDR
does not actually impose. The risks of declining are: (a) the work
group has already publicly committed (in the 2025-09-23 comment) to
adding the use case and diagram, so declining now would reverse that
commitment and likely be re-filed by the same or another reporter;
and (b) EHR vendors are an explicit growth audience for VRDR, so
withholding workflow narrative aimed at them works against adoption.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly satisfies the reporter, matches what
cynthia_bush has already committed the work group to delivering, and
costs essentially nothing in conformance terms because Background is
narrative-only. Disposition B is structurally appealing for the long
term but is a larger refactor that should be raised as its own ticket
rather than bundled with this one — doing it here risks scope creep
and a missed publication window. Disposition C contradicts the work
group's own stated direction and gives EHR-vendor implementers no new
on-ramp into VRDR. If the work group later decides Background has
become too long, the use-case content added under Disposition A is
straightforward to lift into a dedicated Use Cases page (Disposition
B) in a future ballot cycle.
