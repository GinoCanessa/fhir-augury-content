## Summary

The ticket reports that on the US Public Health Library lab-result Observation
profile (`us-ph-lab-result-observation`), the `Observation.category:Laboratory`
slice is a required element but only carries the base FHIR extensible binding to
the `observation-category` value set. Because this slice is, by name and
intent, the "Laboratory" category slice, the reporter argues the broad
extensible binding is too permissive: only the `laboratory` code from the value
set is appropriate in this slot. The reporter proposes constraining the slice
to a fixed value of `laboratory` (from the `observation-category` code system)
rather than relying on the extensible binding. The only cross-reference Augury
surfaced is the FHIR element `Observation.category` itself; no other Jira
tickets, Zulip threads, or GitHub items are linked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

On the `us-ph-lab-result-observation` profile in the US Public Health Profiles
Library, change the `Observation.category:Laboratory` slice so that the slice's
`Observation.category.coding` carries a fixed `Coding` of
`{ system: "http://terminology.hl7.org/CodeSystem/observation-category",
code: "laboratory" }` (or, equivalently, a `patternCodeableConcept` with that
single coding). The slice's binding can then be tightened/removed because the
fixed value already satisfies any conformance check; the slice discriminator
(pattern on `coding`) becomes self-consistent with the fixed value. The base
`Observation.category` element binding remains unchanged, so other categories
on a non-Laboratory observation are not affected.

#### Justification

The slice is named "Laboratory" and the profile scopes the resource to
laboratory results, so any value other than `laboratory` in this specific slice
would be semantically wrong regardless of what the extensible binding allows.
Fixing the value matches the pattern used by other US-realm observation
profiles (e.g., US Core lab result) where the laboratory category slice pins
the coding rather than relying on the value set binding. It also makes the
slice's pattern discriminator unambiguous, which is the reason the slice exists
in the first place.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a single fixed `Coding`, tighten the slice using a
`patternCodeableConcept` containing the `laboratory` coding **and** strengthen
the slice's binding from `extensible` to `required` against a profile-specific
value set that contains only `http://terminology.hl7.org/CodeSystem/observation-category|laboratory`.
Leave `Observation.category` cardinality and the slicing rules otherwise
untouched. This keeps the door open for additional codings to be added
alongside `laboratory` on the same `CodeableConcept` (e.g., a jurisdictional
category code) without invalidating the slice.

#### Justification

A hard fixed value forbids implementers from sending any additional translation
or jurisdictional coding alongside `laboratory` in the same `CodeableConcept`,
which can be useful in public-health data flows that carry local category
codes. Using a pattern + a single-code required binding still guarantees that
`laboratory` is present (so the slice discriminator works) while preserving
extensibility on the `CodeableConcept` itself. This is the more conservative,
implementer-friendly form of the same constraint.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave the `Observation.category:Laboratory` slice with
its current extensible binding to `observation-category`, on the grounds that
the slice discriminator (pattern on `coding` = `laboratory`) already enforces
the intent: any instance that lands in this slice must carry the `laboratory`
coding, and the extensible binding on the broader element is the standard FHIR
pattern that the US PH Library inherits intentionally.

#### Justification

This is defensible only if the workgroup believes the discriminator alone is
sufficient and that adding a fixed value or required binding would create
unnecessary churn for implementers who have already validated against the
current profile. It is the weakest of the three options because the reporter's
underlying observation is correct — the binding as written permits codes that
are obviously wrong for a laboratory-result profile — but it may be chosen if
the WG wants to defer until a broader review of all category slices in the
library.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the reporter's intent (the Laboratory slice must carry
the `laboratory` code) while remaining tolerant of additional translations or
jurisdictional codings in the same `CodeableConcept`, which is a realistic
public-health data shape. Disposition A is essentially correct and aligns with
US Core's treatment of the same slice, but a strict fixed `Coding` is more
restrictive than necessary and can break instances that legitimately carry
multiple codings. Disposition C should be rejected: the reporter's point —
that an extensible binding to the full `observation-category` value set is
inappropriate for a slice whose entire purpose is "laboratory" — is correct,
and leaving it as-is keeps a real conformance gap in the profile. The Public
Health WG should adopt B (or fall back to A if WG consensus prefers strict
fixed values consistent with US Core).
