## Summary

Brett Marquard reports that the eCR example value set `valueset-mrtc-example`
(canonical `http://hl7.org/fhir/us/ecr/ValueSet/valueset-mrtc-example`) only
contains medication codes, despite being intended to support reportable-
condition trigger evaluation that also covers vaccinations. The reporter asks
the Public Health work group to publish a separate example value set scoped to
immunizations so that example queries such as
`Immunization?vaccineCode:in=…/valueset-mrtc-example` are meaningful, and to
consider sourcing those example codes from the eRSD definitions published by
AIMS (https://ersd.aimsplatform.org/). The ticket has already been triaged with
a resolution of "Considered for Future Use" and a maintainer note that the
upstream RCTC medications and immunizations value sets are currently identical
(the medications value set in fact carries vaccine codes), so the fix is
out of the eCR IG's direct control and needs to be raised with the value-set
stewards. No related Jira tickets, Zulip discussions, or GitHub items were
surfaced via cross-reference or keyword search.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Author and publish a new immunization-scoped example value set in the eCR IG —
e.g. `ValueSet/valueset-mrtc-example-immunizations` (canonical
`http://hl7.org/fhir/us/ecr/ValueSet/valueset-mrtc-example-immunizations`) —
populated with a representative handful of CVX vaccine codes (and, where
applicable, RxNorm vaccine product codes) drawn from the eRSD/RCTC
immunization grouping at AIMS. Update the Subscription / topic example query
in the eCR narrative to use the new immunization value set
(`Immunization?vaccineCode:in=…/valueset-mrtc-example-immunizations`), keep
the existing `valueset-mrtc-example` for medications, and add a
documentation note clarifying that these are *example* value sets — the
authoritative grouper remains the AIMS-published RCTC.

#### Justification

This is the most direct response to the reporter's complaint. The current
example query is misleading because the referenced value set, despite its
generic "mrtc-example" name, contains only medication codes; an
implementer copy-pasting the example will get zero matches against
`Immunization.vaccineCode`. Splitting the example into medication- and
immunization-scoped value sets aligns the IG's example assets with how
RCTC content is actually consumed in practice (separate trigger groupings
per resource type) and avoids confusing readers about the role of
`vaccineCode` vs. medication coding systems.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting a new example value set, fix the documentation
defect in place: rename / re-narrate `valueset-mrtc-example` so its
title and description make clear it is a medications-only illustrative
sample, replace the misleading `Immunization?vaccineCode:in=…` example
with a `MedicationRequest?code:in=…` example that actually matches the
value set's contents, and add a normative pointer to the
AIMS/eRSD-published RCTC immunization grouping
(https://ersd.aimsplatform.org/) as the canonical source for any
implementer who needs an immunization trigger value set. Optionally,
include a small inline `composition.include.concept` snippet of two or
three CVX codes purely for illustration in the narrative, without
publishing it as a standalone IG-owned ValueSet resource.

#### Justification

The maintainer's resolution note already concedes that the *real* fix
lives upstream at AIMS, not in the eCR IG. Publishing a brand-new
IG-owned example value set risks implementers treating it as
authoritative and skewing toward an incomplete vaccine list that will
drift from RCTC. A documentation-only correction removes the bug
(the broken example query) without taking on long-term curation
responsibility for vaccine code content the IG does not own, and
explicitly steers implementers to the authoritative AIMS source.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the eCR IG. Mark the example
value set issue as a known limitation referencing AIMS/RCTC governance,
close the ticket with the existing "Considered for Future Use"
resolution, and direct the reporter to raise the immunization /
medication separation directly with the RCTC stewards at AIMS.

#### Justification

Per the maintainer comment already on the ticket, the upstream RCTC
medications and immunizations value sets are *defined* to be identical
today — the medications grouping carries the vaccine codes — and
neither the IG editors nor Public Health WG control that content. An
IG-published example that diverges from RCTC could create the
appearance of a parallel, conflicting grouper. Since the substantive
change must happen at AIMS, the eCR IG's only obligation is to not
mislead, which can be handled as an editorial follow-up rather than a
new ballot-tracked artifact.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern is real — the example query in the
eCR IG is broken because `valueset-mrtc-example` does not contain any
codes that `Immunization.vaccineCode` would bind to — but the
maintainer's own resolution note correctly identifies that the
authoritative split between medication and vaccine codes belongs to the
AIMS/RCTC stewards, not the IG. Disposition A would saddle the eCR IG
with curating an immunization vaccine-code list it doesn't own and that
will inevitably drift from RCTC. Disposition C leaves the broken
example query in the published IG, which is the actual implementer
pain. Disposition B fixes the implementer-visible defect (the
misleading example) with a tightly-scoped editorial change, points
readers at the authoritative AIMS source, and respects the upstream
governance boundary the maintainers have already flagged. It can be
delivered as a small narrative + example-snippet change to the eCR IG
without adding a new ValueSet artifact or new ballot surface area.
