## Summary

This BSeR (US Bidirectional Services eReferrals) change request — tracked
upstream as BSeR issue #138 — proposes that **every profile in the BSeR
Implementation Guide carry a brief introductory narrative section**,
consistent with the intro that already exists on the *BSeR Arthritis
Feedback Observation* profile. Today only a handful of BSeR profiles
have such an intro, leaving most profiles without prose context that
explains intent, scope, and how the profile fits into a referral
workflow. The reporter asks that the intro pattern be generalized
across the entire profile set so implementers get a consistent,
human-readable explanation alongside the structural definition. The
ticket has been Triaged and resolved as **Persuasive** with the
resolution note "Will update as proposed." It is now labelled
`Ready-For-Vote` for confirmation by the Public Health work group.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items
were found via `fhir-augury-cli` for this ticket.

### Disposition A: Accept As Requested

#### Proposal

Add a brief introductory narrative section to **every profile** in the
US BSeR FHIR IG, modelled on the existing intro on
*BSeR Arthritis Feedback Observation*. Concretely:

- For each `StructureDefinition` (profile) in the BSeR IG — Patient,
  Practitioner, PractitionerRole, Organization, Location, Encounter,
  ServiceRequest, Task, Coverage, Consent, the various Observation
  profiles, the Bundle/Composition profiles used for referral and
  feedback packages, and any extensions that carry their own page —
  author a short prose intro (typically 2–6 sentences) covering:
  - the profile's role in the referral / feedback workflow,
  - the actor(s) expected to create or consume it,
  - any must-support or cardinality highlights worth flagging up front,
  - how it relates to neighbouring profiles in the same exchange.
- Place the intro using the IG-publisher convention used today for the
  Arthritis Feedback Observation profile (e.g.
  `input/pagecontent/StructureDefinition-<id>-intro.md`, or whatever
  fragment include the BSeR IG already wires into its profile pages).
- Keep wording aligned with the profile's `description` element so the
  spec page narrative and the StructureDefinition metadata don't drift.

#### Justification

The request is purely additive editorial guidance and does not change
any conformance bits. It improves the IG's usability for implementers
who today get only the auto-generated structure tables on most profile
pages, and it brings the profile set in line with a pattern the WG has
already endorsed on at least one profile. The ticket is already
resolved **Persuasive** with the note *"Will update as proposed,"* so
Disposition A simply executes the existing decision and reflects the
`Ready-For-Vote` state.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the intent of the request but **scope it deliberately and add a
template** rather than committing to "every profile, all at once":

1. Author and publish (in the BSeR IG's contributor docs) a short
   *Profile Intro Template* with required headings (Purpose, Actors,
   Workflow Context, Notable Constraints, Related Profiles) and
   examples drawn from the Arthritis Feedback Observation intro.
2. Apply the template first to a prioritized tier — the
   referral-bundle, feedback-bundle, ServiceRequest, Task, and the
   Observation profiles — in the next BSeR publication.
3. Track remaining profiles as follow-up tickets so that intros are
   added in tranches with explicit WG review of the prose, rather than
   as a single bulk edit.
4. Where a profile's `StructureDefinition.description` already carries
   the same content, lift it into the intro page and keep the two in
   sync via the page include rather than duplicating text.

#### Justification

Disposition B preserves the user-visible benefit (every profile
eventually gets an intro) but reduces the risk of low-quality, copy-
pasted prose appearing across many profiles in a single PR. It also
gives the Public Health WG a reusable template for future BSeR
profiles, which is more durable than a one-off bulk update. Trade-off
vs. A: slower rollout, and the IG will temporarily continue to have
intro coverage gaps on lower-priority profiles.

---

### Disposition C: Decline

#### Proposal

Decline to add intro sections across all profiles, on the grounds that
the `StructureDefinition.description` element and the IG-publisher's
auto-generated profile pages already provide a place for narrative
context, and that prose maintained separately from the
StructureDefinition risks drifting out of sync with the formal
definition.

#### Justification

Declining is defensible in principle — duplicated narrative is a known
maintenance hazard, and BSeR could instead invest in richer
`description` text on each profile, which the IG publisher renders
automatically. However, this rationale is weak in this specific case:
the Arthritis Feedback Observation precedent shows the WG has already
chosen the intro-page pattern over relying on `description` alone, and
the ticket is documentation-only with no conformance impact.

---

### Recommendation

**Recommended disposition:** A

The ticket is already resolved **Persuasive** with the explicit
resolution *"Will update as proposed"* and is now `Ready-For-Vote`, so
the workgroup has effectively pre-committed to Disposition A. The
change is editorial, carries no conformance risk, and harmonises the
BSeR profile set with a pattern the WG itself established on the
Arthritis Feedback Observation profile. Disposition B's template idea
is genuinely useful and should be adopted *as part of* executing
Disposition A — i.e., write the intro template first, then apply it to
every profile in the next BSeR publication — rather than used as a
reason to slow the rollout. Disposition C should be rejected because
the WG's prior precedent already settled the "intro page vs.
description-only" question in favour of intro pages.
