## Summary

Reporter Ron G. Parker (via imported "Comment 29") asks the Vital Records
Common Profiles Library (VRCL) project to anchor its code system and value
set URLs in `terminology.hl7.org` (THO), in line with the TSC-approved
"Terminology Guidance for IG Developers" policy. Concretely, the request
asks the work group to initiate the UTG (Unified Terminology Governance)
process so that the VRCL-defined value set(s) can be migrated to THO and
referenced from THO URIs in a future publication of the IG. The most
recent comment (lynn_laakso, 2024-10-11) reopens the ticket post-STU2
publication, having reverted an earlier "Considered for Future Use"
disposition (vote 23-0-0, 2024-05-30) on the grounds that the value set
"has nothing to do with vital records and is of general utility," and
states the work group will begin the process of moving it to THO so a
later version of the IG can reference the THO copy.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FhirAugury for this key, so the request is being treated on
its own terms; the underlying terminology asset is only referenced
generically ("this value set") in the imported comment and is not
identified by URL inside the visible ticket content.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Initiate the UTG process to migrate the IG-defined CodeSystem(s) and
ValueSet(s) currently authored under the VRCL canonical base into THO.
For each terminology artifact:

1. Submit a UTG proposal that moves the artifact's canonical URL from
   the VRCL IG namespace to a `http://terminology.hl7.org/...` URL,
   preserving the resource's identifier and codes.
2. Once the THO entry is published, in a subsequent VRCL release replace
   the in-IG `CodeSystem`/`ValueSet` resource with a reference to the
   THO canonical URL (or retain a thin re-publication that defers to THO
   if needed for backward compatibility with STU2 consumers).
3. Add a migration note to the VRCL release notes calling out the
   canonical-URL change and the equivalence between the STU2 URL and the
   new THO URL.

#### Justification

This is exactly what the TSC-approved "Terminology Expectations for IG
Developers" policy calls for and what the reopening comment explicitly
commits the work group to. The 2024-10-11 comment also confirms the
asset is of general utility ("nothing to do with vital records"), which
is the canonical trigger for a UTG move under the policy. Doing the
migration cleanly post-STU2 keeps the STU2 URLs valid for existing
implementers while letting the next VRCL publication reference the
authoritative THO copy.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the URL change in VRCL itself and instead handle the move as a
THO-side republication: file the UTG proposal so the artifact appears in
THO with a THO-canonical URL, but in VRCL keep the existing IG-canonical
CodeSystem/ValueSet for the duration of the current normative cycle and
add explicit cross-references (via `ValueSet.identifier` /
`CodeSystem.identifier` and a narrative note) pointing at the THO
equivalent. Schedule the canonical-URL swap for the next major VRCL
release rather than this maintenance cycle.

#### Justification

Re-anchoring canonical URLs is a breaking change for any implementer who
has indexed VRCL-defined value sets by URL. Decoupling "publish in THO"
(low risk, addresses the policy and the reporter's literal ask of
"start the UTG process") from "rewrite VRCL to point at THO" (higher
risk, affects published conformance artifacts) lets the work group
satisfy the policy now while giving implementers a deprecation window.
This also matches the spirit of the lynn_laakso comment, which already
contemplates "a future version of this IG should reference the
valueset."

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** on the basis that the ticket as imported is
under-specified — it does not name the specific CodeSystem or ValueSet
to be moved, nor does it provide a concrete THO target — and ask the
reporter (or originating WG) to refile a per-artifact UTG request
through the standard UTG submission channel rather than as a VRCL
change request.

#### Justification

The ticket is a generic restatement of existing TSC/TSMG policy and
contains no actionable specification-level change for VRCL: no resource
ID, no element to alter, no constraint to add or remove. UTG migrations
are normally tracked as UTG proposals, not as IG change requests, so
processing this in VRCL risks duplicating governance and producing no
artifact-level diff. Routing it back to UTG keeps the work where the
governance lives.

---

### Recommendation

**Recommended disposition:** A

The reopening comment from lynn_laakso explicitly commits Public Health
to "begin the process of moving to THO," and the TSC-approved
Terminology Expectations policy clearly states IG-defined terminology of
general utility belongs in THO. Disposition A simply executes that
commitment end-to-end (UTG submission *and* eventual VRCL repointing),
which is what implementers actually need to stop pinning STU2 URLs into
their systems long-term. Disposition B is a reasonable phasing tactic
and could be folded into A as an implementation detail (publish in THO
first, swap VRCL canonicals in the next major release) if the work
group wants to manage breakage carefully — but it should not be used as
a reason to indefinitely postpone the canonical change. Disposition C
is hard to defend given the work group has already voted, in effect, to
proceed; declining now would re-litigate a decision that has been made.
The next concrete step the work group should take from this review is
to identify the specific VRCL CodeSystem/ValueSet artifact(s) the
imported "Comment 29" refers to and attach that list to this ticket so
the UTG submission can proceed.
