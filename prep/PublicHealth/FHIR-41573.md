## Summary

The ticket asks the BFDR (US Birth and Fetal Death Reporting) IG to define an
Observation profile that records whether the mother used specific drugs during
pregnancy. The reporter, Saul Kravitz, frames this as an intra-jurisdictional
exchange need — for example, a BFDR sender forwarding data to a Public Health
Surveillance receiver — and explicitly notes that NCHS is not interested in
receiving this content, so the new artifact must be optional (or at least
non-mandatory) from NCHS's perspective. The ticket points to LOINC code
[89495-6](https://loinc.org/89495-6/) ("Drugs of abuse used by mother during
this pregnancy") and the associated answer list
[LL4903-2](https://loinc.org/LL4903-2) as a near-perfect existing terminology
fit, and supplies a six-row table from the underlying source system covering
amphetamines, heroin, methadone, marijuana, cocaine, and "any other illegal
drugs", each captured as a Yes / No / Unknown flag. The ticket has been
re-opened twice (post-STU1.1 in 2023 and post-STU2 in 2024) after prior
"Considered for Future Use" votes, indicating sustained interest in landing
this in a future ballot rather than abandoning it. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were returned by the data source.

## Details

**Description:**

To support intra-jurisdictional exchange (e.g., BFDR to Public Health
Surveillance) an observation is required to record mother's use of drugs
during pregnancy. NCHS is not interested in receiving this information.

There is an existing LOINC <https://loinc.org/89495-6/> and elements of its
answer list <https://loinc.org/LL4903-2> that seems tailor made to capture
this info.

The initial use case includes the following drugs:

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new `Observation`-based profile to the BFDR IG (e.g.,
`ObservationMotherDrugsUsedDuringPregnancy`) with:

- `Observation.code` fixed to LOINC `89495-6` ("Drugs of abuse used by
  mother during this pregnancy [Reported]").
- `Observation.subject` constrained to the BFDR `Patient-Mother` profile.
- `Observation.focus` (or `extension`) referencing the related encounter /
  pregnancy / child Patient as appropriate to align with existing
  BFDR `Observation` profiles for prenatal data.
- A repeating `Observation.component` for each substance, each with:
  - `component.code` bound to LOINC answer list `LL4903-2` (or a BFDR
    value set derived from it) covering at minimum amphetamines, heroin,
    methadone, marijuana, cocaine, and "other illegal drugs";
  - `component.value[x]` as `CodeableConcept` bound to a small required
    value set (`yes` / `no` / `unknown`), with the source-system blank
    mapped to `unknown` (`http://terminology.hl7.org/CodeSystem/v3-NullFlavor#UNK`
    or the BFDR-standard absent/unknown coding).
- Mark the profile as **MS but not required** in the BFDR-to-Public-Health
  Surveillance exchange profile, and explicitly excluded from the
  BFDR-to-NCHS submission bundle (consistent with the reporter's note that
  NCHS does not want this content). Update the relevant `Composition`
  section definitions and `MotherPrenatalObservations` slicing in the IG
  to reference the new profile.

#### Justification

The request is well-scoped, has a clear public-health use case
(jurisdiction-to-jurisdiction exchange), and a tailor-made LOINC code plus
answer list already exists, which keeps terminology authoring cost low and
maximises downstream analytics interoperability. Modelling the substances
as components of a single `Observation` (rather than six separate
profiles) mirrors the structure of the LOINC concept and the source
system's grouped table, and keeps the resource count manageable. The
ticket has now been re-opened twice and survived two ballot cycles, so
the work group has implicitly signalled that "defer indefinitely" is no
longer the right answer.

---

### Disposition B: Alternative Approach

#### Proposal

Define the artifact at the **profile-of-profile** layer rather than as a
new top-level Observation, by:

1. Adding a single new BFDR `Observation` profile
   `ObservationMotherSubstanceUseDuringPregnancy` that uses
   `Observation.code` = LOINC `89495-6` and `Observation.value[x]` as a
   repeating `CodeableConcept` bound to LL4903-2 — i.e., one Observation
   per substance reported, rather than a single Observation with six
   components. Substances the mother did *not* use are omitted entirely;
   "Unknown" is conveyed via `dataAbsentReason` on a single header
   Observation.
2. Scoping it explicitly to the **Public Health Surveillance** exchange
   profile only (a separate exchange-flavoured artifact, e.g.
   `BundleDocumentBirthReportToPublicHealth`), and **not** including it
   in the NCHS-bound bundle.
3. Leaving the door open for additional substances (opioids, fentanyl,
   alcohol — already partly covered by other BFDR observations) by binding
   to the LOINC answer list with `extensible` strength rather than
   `required`.

#### Justification

A "one Observation per reported substance" model aligns more closely with
how `Observation` is generally used elsewhere in FHIR (one observation =
one finding) and avoids the awkwardness of recording six explicit
"No" answers per birth — most births will report no substance use at all,
and the component-per-substance model produces large, sparse Observations.
It also degrades more gracefully as the substance list grows
(fentanyl, e-cigarettes, etc.), because new answers can be added to the
value set without re-versioning the profile structure. The `extensible`
binding leaves room for jurisdictions that already track substances not
in the initial table, which is realistic for a public-health surveillance
audience. The exchange-profile separation directly addresses the "NCHS is
not interested" constraint at the right architectural layer (bundle /
composition section), rather than relying on individual profile MS flags.

---

### Disposition C: Decline

#### Proposal

Mark as **Persuasive with Mod / Considered for Future Use** again,
deferring to a later STU on the basis that:

- The use case is intra-jurisdictional (BFDR → state Public Health
  Surveillance) and is not part of the core NCHS submission scope that
  drives the BFDR IG's primary conformance targets.
- Jurisdictions that need this today can use a generic `Observation`
  with LOINC `89495-6` without a BFDR-specific profile, and any
  conformance enforcement can live in the receiving jurisdiction's
  own profile.
- The 2024-02-15 vote (25-0-2) explicitly noted "currently of limited
  interest to potential users", and no new evidence of broad demand has
  been recorded on the ticket since.

#### Justification

The ticket has twice been voted "Considered for Future Use" by sizeable
margins (29-0-0 in 2023, 25-0-2 in 2024) on the grounds of limited
implementer interest. Re-opening after publication does not by itself
change that signal. Adding a BFDR-specific profile that NCHS will not
consume risks creating an artifact that only one or two jurisdictions
actually use, which is hard to maintain across IG versions and dilutes
the IG's focus on the federally-aligned dataset.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the reporter's underlying need — recording
mother's substance use during pregnancy for intra-jurisdictional public
health exchange — while addressing the structural concerns that the
literal ask (Disposition A) raises. A component-per-substance Observation
forces every birth record to carry six explicit "No" components even when
nothing was used, and bakes the *current* substance list into the profile
structure; a profile that uses one Observation per reported substance
with an extensible LOINC LL4903-2 binding is more idiomatic FHIR, scales
better as new substances of concern emerge, and keeps the artifact easy
to evolve. Scoping the new profile to the Public-Health-Surveillance
exchange composition (and explicitly excluding it from the NCHS bundle)
honours Saul Kravitz's "NCHS is not interested" constraint at the
correct layer of the IG. Disposition C is no longer defensible: the
ticket has been re-opened by Lynn Laakso twice, and continuing to defer
without engaging with the design risks the work group looking unwilling
to support a documented public-health surveillance use case.
