## Summary

The reporter, Saul Kravitz, observes that the documentation for the Mother's
**Residence** and **Mailing** addresses in the US Birth and Fetal Death
Reporting (BFDR) IG is unclear: both labels resolve to the same backing
element, `PatientMotherVitalRecords.address`, with no in-IG explanation that
the two are distinguished by the FHIR `Address.use` slicer (`home` for
residence, `billing` for mailing). The ask is to make this distinction
explicit in the IG documentation so implementers do not have to infer the
mapping. The follow-up comment broadens the scope: rather than a one-off
documentation fix for the mother's two addresses, the work should be a
sweep of **all** address fields in BFDR (and VRDR, as a sibling Vital
Records IG) to add `use` constraints/labels where they are needed, and to
call out `MotherResidence` and `MotherBilling` as two known instances that
must be updated.

No cross-referenced Jira tickets, Zulip discussions, GitHub PRs/issues, or
Confluence pages were returned for this key by the FHIR Augury indices, so
the request currently stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BFDR IG narrative for the Mother's address fields so that
the two logical fields ("Mother's Residence Address" and "Mother's
Mailing Address") explicitly document that they both bind to
`PatientMotherVitalRecords.address` and are disambiguated by
`Address.use`:

- `MotherResidence` → `Address.use = home`
- `MotherBilling`  → `Address.use = billing`

Concretely, in the `PatientMotherVitalRecords` profile page, add a
short prose section under the `address` element that lists the
defined `use` values and the IG field name they correspond to. If the
profile already slices `address` on `use`, surface the slice
definitions in the rendered profile snapshot/differential. If it does
not, add a `slicing` declaration on `Patient.address` with
discriminator `value` on `use` and named slices `residence` (use=home)
and `mailing` (use=billing), each with cardinality `0..1`.

#### Justification

This is the minimal, conservative change that resolves the reporter's
specific complaint. Address slicing on `use` is the standard FHIR
pattern for distinguishing residence from mailing addresses (`Patient`
already permits multiple `address` entries, with `Address.use` as the
discriminator). Making the binding explicit in the IG eliminates the
need for implementers to guess the mapping from the IG's labelled
field names to the underlying FHIR element, and aligns the BFDR
profile with how every other US Realm IG that distinguishes home vs.
mailing addresses (e.g., US Core Patient) treats the same problem.

---

### Disposition B: Alternative Approach (Spec-Wide Sweep per Comment)

#### Proposal

Adopt the broader scope proposed in Saul Kravitz's comment: treat
FHIR-48883 as the entry point for a documentation/profile sweep of
**all** address-bearing elements in BFDR (and, in coordination with
the VRDR maintainers, in VRDR). Concretely:

1. Inventory every IG-labelled "address" field in BFDR and VRDR
   (mother, father, child/decedent, place of birth, place of death,
   place of injury, etc.).
2. For each, decide whether the distinction is by `Address.use`, by
   `Address.type`, by a separate profile/extension, or by a different
   resource entirely (e.g., `Location.address`).
3. Where the distinction is by `Address.use` (or `Address.type`), add
   an explicit slice to the relevant profile and document the IG
   field-name → slice mapping in the profile narrative.
4. Where there is no current distinction but the IG label implies
   one, file the gap and propose a slicing/extension model.
5. Use `MotherResidence` and `MotherBilling` as the two worked
   examples in the methodology section so future IG fields follow the
   same convention.

#### Justification

This addresses the reporter's underlying need (implementers cannot
map IG field names to FHIR elements unambiguously) at the level the
reporter himself identified in his follow-up comment, rather than
fixing only the two fields that triggered the ticket. It also avoids
a near-certain cycle of follow-up tickets each repeating the same
"this address field is ambiguous" pattern. The trade-off versus
Disposition A is scope: the sweep is more work, requires coordinating
with the VRDR specification owners, and may surface profile changes
(new slices, new extensions) that go beyond a documentation update.
That said, the sweep is normative-light — most of the change is
narrative and structural slicing — and it makes the BFDR/VRDR pair
internally consistent, which is a recurring Public Health Work Group
goal.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the information needed to
distinguish residence from mailing addresses is already present in
the FHIR core specification (`Address.use` value set, with `home` and
`billing` codes, plus `Patient.address` cardinality `0..*`), and the
BFDR IG inherits this directly via its profile of `Patient`.
Implementers wishing to populate or consume the two address types can
use the standard FHIR mechanism without an IG-level documentation
change.

#### Justification

This position is defensible only on the narrow reading that the IG
need not re-document FHIR core semantics. It would, however, leave
unanswered the reporter's actual concern: the BFDR IG's labelled
fields ("Mother's Residence Address", "Mother's Mailing Address") do
not state, in IG-visible documentation, which `Address.use` value
realises which IG label. That mapping is an IG-specific binding
decision (the IG could just as defensibly have used
`Address.type = postal`/`physical`), so leaving it implicit
contradicts the IG's own purpose of constraining FHIR for the BFDR
use case. Decline is included for completeness; it is unlikely to
satisfy implementers.

---

### Recommendation

**Recommended disposition: B (Alternative Approach — spec-wide sweep).**

Disposition A solves the literal complaint, but the reporter's own
follow-up comment explicitly broadens the ask to all BFDR/VRDR
address fields and asks that `MotherResidence`/`MotherBilling` be
listed as known examples of the pattern. Treating the ticket as the
entry point for that sweep — rather than as a two-field documentation
patch — captures the reporter's intent, makes BFDR and VRDR
internally consistent (a stated Vital Records goal), and avoids a
trickle of follow-up tickets reporting the same ambiguity on other
address fields. Disposition C is included for completeness but is
hard to defend: the IG's own labelled fields create an IG-specific
mapping question that the FHIR core spec cannot answer on its own.

Before drafting the change, the workgroup should run the
`repo-analysis` skill against `HL7/fetal_records` (and the VRDR
repo, if VRDR is included) so that the implementation plan can name
the specific profile and narrative files that need to be edited. The
current FHIR Augury indices do not link any GitHub PRs or commits to
this ticket, so the touched paths cannot be derived from prior work.
