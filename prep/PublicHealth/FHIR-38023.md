## Summary

Reporter Craig Newman asks the Public Health work group whether the US
Public Health Profiles Library (USPHPL) "disability profile" should be
rebased on the US Core Observation Survey Profile
(`us-core-observation-survey`, introduced in US Core STU5.0.1) rather
than being authored from scratch on the FHIR base `Observation`. The
ticket is a design / alignment question rather than a defect: USPHPL
already carries a Disability Status profile, and US Core has since
shipped a Survey profile that captures the structural patterns
(question/answer linkage to a survey instrument, `derivedFrom →
QuestionnaireResponse`, value coding, etc.) that disability-status
observations typically exhibit.

A previous disposition closed this as "Considered for Future Use"
(2022-12-01, vote 23-0-4: "We will consider using the US Core
Observation Survey Profile when we move to a later version of US Core
(this IG is currently dependent on US Core 4.0.0 which does not contain
that profile).") The ticket was reopened post-STU publication (2023-08-17,
lynn_laakso) so the WG can revisit it now that USPHPL is in a position
to upgrade its US Core dependency.

No direct or cross-referenced links were returned by the FhirAugury
index, but several keyword-related Jira tickets give important context:

- [FHIR-38424](https://jira.hl7.org/browse/FHIR-38424) — Align Disability
  Status with USCDI v3 (USPHPL; resolved Persuasive with Modification).
- [FHIR-38589](https://jira.hl7.org/browse/FHIR-38589) — Clarify what
  happens when US Core adds the same content (USPHPL; Persuasive).
- [FHIR-44544](https://jira.hl7.org/browse/FHIR-44544) — Disability
  Status should have its own Observation (US Core; Not Persuasive).
- [FHIR-37463](https://jira.hl7.org/browse/FHIR-37463) — QI-Core created
  its own Observation Survey Profile consistent with US Core 5.0.0
  (Persuasive with Modification) — direct precedent for an IG
  re-anchoring a survey-style profile on the US Core Survey profile.

Together these establish that (a) USPHPL has already committed to
USCDI v3 / US Core alignment for disability status, (b) US Core
declined to carve out a dedicated Disability Status observation, and
(c) at least one peer IG (QI-Core) has adopted the US Core Observation
Survey Profile as the basis for survey-shaped observations.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-derive the USPHPL Disability Status Observation profile from
`http://hl7.org/fhir/us/core/StructureDefinition/us-core-observation-survey`
(US Core ≥ 5.0.1) instead of from base `Observation`. Concretely:

- Bump the USPHPL IG `dependsOn` for `hl7.fhir.us.core` from 4.0.0 to
  the lowest US Core version that publishes the Survey profile
  (5.0.1 / STU5.0.1, or the current published version at re-ballot
  time).
- Change the Disability Status profile's `baseDefinition` to
  `us-core-observation-survey`.
- Re-author the profile's differential to *only* add USPHPL-specific
  constraints on top of what the Survey profile already requires:
  Disability-Status–specific `code` binding, value-set bindings on
  `value[x]`, `derivedFrom → QuestionnaireResponse` slicing tied to
  the disability instrument(s), and any USCDI v3-driven Must Support
  flags that are not already inherited.
- Update the IG narrative and examples to show the survey-style usage
  (instrument identifier in `code`, individual answers in `component`
  or in linked Observations per the Survey profile's pattern).
- Add a migration note to the IG version history per the answer USPHPL
  gave on FHIR-38589, explaining the change of `baseDefinition` and
  what implementers must adjust.

#### Justification

This is the literal request and the option most aligned with the
Public Health WG's existing direction:

- **USCDI v3 alignment (FHIR-38424).** USPHPL has already accepted
  Persuasive with Modification on aligning Disability Status with
  USCDI v3 / US Core; deriving from `us-core-observation-survey` is
  the structural counterpart to that semantic alignment.
- **Migration story (FHIR-38589).** USPHPL has already accepted
  Persuasive on documenting "what happens when US Core adds the same
  content"; rebasing on the US Core Survey profile is exactly the
  kind of forward-compatible move that answer envisions.
- **Precedent (FHIR-37463).** QI-Core has already done this
  re-derivation against US Core 5.0.0's Survey profile — there is a
  validated authoring pattern to follow.
- **The original blocker is gone.** The 2022-12-01 vote explicitly
  conditioned acceptance on USPHPL being able to upgrade US Core; that
  upgrade is now feasible.

The FHIR-42831 multi-value gotcha must be addressed in the Justification
of the IG change PR: the Survey profile does not by itself solve the
"vision *and* hearing impairment at the same point in time" pattern, so
the USPHPL profile must keep guidance on whether to use `component`,
multiple Observations, or `valueCodeableConcept` with multiple `coding`
entries — but that guidance is independent of which `baseDefinition` is
chosen, and is easier to attach to a Survey-derived profile than to a
bespoke one.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** change `baseDefinition`. Instead:

- Keep the USPHPL Disability Status profile derived from base
  `Observation`.
- Add a normative compatibility section to the IG that documents
  element-by-element alignment with `us-core-observation-survey`
  (Must Support, cardinality, slicing on `derivedFrom`, value-set
  bindings) so an instance that conforms to the USPHPL profile is
  *also* a valid `us-core-observation-survey`.
- Add a conformance test / IG QA rule that asserts this alignment,
  and add the alignment table to the IG so implementers can see it
  without having to diff the StructureDefinitions.
- Optionally publish a derived profile-on-profile in a future
  USPHPL minor release, once US Core 6.x adoption is widespread in
  Public Health implementer space.

#### Justification

Some Public Health implementers are still on US Core 3.x / 4.x
(Disability Status reporting frequently flows through state public
health agencies whose IT modernization lags US Core releases). A hard
re-derivation forces every USPHPL implementer onto the newer US Core
the moment they pick up the next USPHPL release, which can stall
adoption of *other* USPHPL improvements.

This approach delivers most of the interoperability win — instances
remain conformant to `us-core-observation-survey` — without forcing the
US Core dependency bump, and it preserves a clean upgrade path
(switching `baseDefinition` later becomes a no-op if alignment is
already verified).

The trade-offs vs. Disposition A: more long-term maintenance burden
inside USPHPL (the alignment can drift each US Core release), no
inherited Must Support / slicing changes when US Core evolves the
Survey profile, and the answer to FHIR-38589 is weaker — USPHPL still
has to manually re-converge whenever US Core moves.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive**: keep the USPHPL Disability Status
profile derived from base `Observation` and explicitly document in the
IG that the USPHPL Disability Status profile is intentionally *not*
based on `us-core-observation-survey`, citing (a) the multi-value
pattern that disability status requires (vision + hearing at the same
encounter, per FHIR-42831) and (b) the public-health–specific
QuestionnaireResponse linkage and value-set requirements that do not
sit cleanly on top of the Survey profile's Must Support set.

#### Justification

Defensible if and only if the WG concludes that the Survey profile's
constraints actively get in the way:

- **FHIR-44544** shows that US Core itself declined to create a
  dedicated Disability Status profile, signalling that US Core does not
  view disability status as a survey-shaped artifact in the strict
  sense.
- **FHIR-42831** confirms US Core will not loosen `Observation.value`
  cardinality, so the multi-value disability-status pattern has to be
  solved at the IG layer regardless. If the WG judges that solving it
  on top of `us-core-observation-survey` is harder than solving it on
  top of base `Observation`, declining is reasonable.
- A decline still leaves room for the FHIR-38589 commitment by
  documenting *why* USPHPL diverges from US Core's Survey profile —
  which is itself a form of "what happens when US Core adds the same
  content".

This option does, however, walk back from the implicit direction set by
the 23-0-4 "Considered for Future Use" vote and from the QI-Core
precedent, so it should only be selected if the WG can articulate a
concrete technical conflict with the Survey profile's constraints.

---

### Recommendation

**Recommended disposition:** A

The 2022-12-01 vote (23-0-4) already showed strong WG appetite for this
change, and the only stated blocker — being pinned to US Core 4.0.0 —
is no longer in force. USPHPL's prior commitments on
[FHIR-38424](https://jira.hl7.org/browse/FHIR-38424) (USCDI v3
alignment) and [FHIR-38589](https://jira.hl7.org/browse/FHIR-38589)
(documenting US Core convergence) point in the same direction, and the
[FHIR-37463](https://jira.hl7.org/browse/FHIR-37463) QI-Core precedent
shows the re-derivation is a well-trodden path. Disposition B is a
reasonable fallback if the WG hears from implementers that a US Core
dependency bump is genuinely disruptive, and Disposition C should be
reserved for the case where the WG can name a specific Survey-profile
constraint that conflicts with disability-status semantics. The
multi-value gotcha raised in
[FHIR-42831](https://jira.hl7.org/browse/FHIR-42831) needs to be
explicitly addressed in the IG narrative regardless of which option is
chosen, and is not by itself a reason to prefer B or C over A.
