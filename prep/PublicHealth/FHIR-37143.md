## Summary

This ticket asks the Maternal and Infant Health Research (MIHR) FHIR IG to
change the canonical defining URL of the `PregnancyDeliveryDiagnosis` value
set from the IG's own `http://www.hl7.org/fhir/us/mihr/ValueSet/...` namespace
to a `terminology.hl7.org` (THO) namespace, and to file a corresponding UTG
ticket so the value set is registered and maintained in THO. The submitter
(reporting on behalf of TSMG / Ron G. Parker, "Comment 8") states the general
rule that only value sets that are required-bound to a `code` data type
should be anchored under `hl7.org/fhir`; everything else belongs in THO.
FHIR-37143 is one of nine sibling tickets opened against the MIHR IG on the
same day for the same TSMG-driven cleanup — FHIR-37136 through FHIR-37144 —
each targeting a different MIHR value set. The ticket was originally resolved
"Considered for Future Use" (vote 21-0-2, 2022-06-02, Ruby Nash / Sarah
Gaunt) pending IG maturity and finalization of the draft THO process, and
was then re-opened post-STU publication by Lynn Laakso on 2023-03-29 so the
disposition could be revisited now that those preconditions are closer to
being met.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

For the MIHR IG's `PregnancyDeliveryDiagnosis` value set:

1. File a UTG ticket to register the value set in THO under a
   `http://terminology.hl7.org/ValueSet/<name>` canonical (preserving the
   current logical name unless TSMG asks otherwise) and to assign / publish
   its OID and identifiers through the THO maintenance process.
2. Once the THO ticket is approved and the value set is published in THO,
   change the value set's `ValueSet.url` in the MIHR IG from
   `http://www.hl7.org/fhir/us/mihr/ValueSet/PregnancyDeliveryDiagnosis` to
   the new THO canonical, and remove the in-IG copy in favour of an
   `imports`/external reference.
3. Update every `Binding.valueSet` reference inside the MIHR IG that points
   at the old canonical to point at the new THO canonical.
4. Add a redirect / alias note at the old canonical so existing implementer
   references continue to resolve, and call out the canonical change in the
   IG's change-history page.
5. Apply the same treatment to the eight sibling tickets (FHIR-37136 through
   FHIR-37142 and FHIR-37144) so the MIHR IG ends up with a clean THO
   anchoring across the whole `Block-Vote-1` set.

#### Justification

This is exactly the policy TSMG has been driving across HL7 IGs: only value
sets that are required-bound to a FHIR `code` primitive belong under
`hl7.org/fhir`; everything else lives in THO so it can be governed,
versioned, and reused independent of any single IG. The original 2022-06-02
"Considered for Future Use" disposition explicitly conditioned acceptance on
two things — IG maturity and a finalised draft THO process — both of which
are materially closer now that the IG has reached STU publication and the
re-open comment from lynn_laakso (2023-03-29) acknowledges the timing has
shifted. Treating all nine siblings as a single block vote (the ticket
already carries `Block-Vote-1`) prevents partial migrations that would leave
the IG with a confusing mix of canonical namespaces.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than migrate the canonical URL piecemeal per ticket, defer the
mechanical canonical change and instead:

1. Open a single coordinating UTG / TSMG ticket that owns the migration of
   *all* MIHR non-`code`-bound value sets to THO as one batched operation,
   with the nine FHIR-3713x tickets linked as children.
2. In the next MIHR publication, mark the existing
   `http://www.hl7.org/fhir/us/mihr/ValueSet/PregnancyDeliveryDiagnosis`
   canonical as deprecated (via a publication-level note plus, where the
   tooling supports it, a `ValueSet.status = retired` or analogous
   indicator) but keep it resolvable for one IG cycle.
3. Land the actual canonical migration and binding rewrites in the cycle
   *after* the THO entries are published, so implementers get one IG release
   of overlap before the old URL is removed.

#### Justification

The blocker called out in the original 2022 disposition was process
maturity, not the substance of the change — TSMG was right then and is
right now, but a hard cut-over of nine canonicals in a single IG release
risks breaking implementers who already have artifacts pointing at
`hl7.org/fhir/us/mihr/ValueSet/...`. A coordinated, two-cycle migration with
a single tracking ticket gives implementers an explicit deprecation window,
keeps the THO maintenance work on one PR rather than nine, and reduces the
chance that some siblings land in one publication and others slip to the
next (which is exactly what creates the messy mixed namespace state TSMG is
trying to avoid).

---

### Disposition C: Decline

#### Proposal

Decline the requested canonical change for `PregnancyDeliveryDiagnosis` and
close FHIR-37143 (along with its eight siblings) as "Not Persuasive" with
the rationale that this value set's scope is intrinsic to the MIHR IG and
not expected to be reused outside it, so there is insufficient cross-IG
benefit to justify the cost of moving it (and migrating every binding) into
THO.

#### Justification

The TSMG policy is a *should*, not an absolute *shall*: value sets that are
genuinely IG-local can defensibly remain under the IG's own canonical,
particularly when the value set's content is expected to evolve with the
IG's clinical model rather than independently. Declining preserves the
existing implementer contract (no broken canonical references) and avoids
the THO maintenance overhead for a value set whose only consumer is MIHR
itself. This stance only stands up if the work group can demonstrate that
the value set is not (and is unlikely to become) reusable outside MIHR;
otherwise Disposition A or B is the better answer.

---

### Recommendation

**Recommended disposition:** B

The substance of TSMG's request is correct and the original 2022
"Considered for Future Use" deferral has effectively expired — IG STU
publication has happened and the THO process is materially further along —
so straight Decline (C) is hard to defend on policy grounds. Pure Accept-as-
Requested (A) is right in spirit but operationally fragile across nine
sibling tickets in one IG: any partial landing produces exactly the mixed
`hl7.org/fhir` + `terminology.hl7.org` state TSMG is trying to eliminate,
and offers implementers no deprecation runway. Disposition B keeps the
endpoint state TSMG wants (all non-`code`-bound MIHR value sets anchored in
THO), batches the work behind a single coordinating UTG ticket so the nine
FHIR-3713x tickets move as a unit, and gives implementers one publication
cycle of overlap before the old canonicals are removed — which is a
materially smaller blast radius than a single-cycle hard cut-over while
still resolving the underlying request.
