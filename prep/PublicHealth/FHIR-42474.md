## Summary

Lloyd McKenzie raises that the BSeR (US Bidirectional Services eReferrals)
workflow currently treats the **Evaluation** step as a binary decision
(accept / reject) on a referral. Real-world referral handling is rarely
that clean: the receiving party may need more information, may decline
the requested service but offer an alternative, or may initially accept
and later change their mind. The ticket asks the BSeR specification to
discuss — and likely model — these intermediate and reversible
evaluation outcomes rather than collapsing the result of an evaluation
into a single yes/no value.

There were no cross-referenced Jira tickets, Zulip threads, or GitHub
items returned for this ticket, so the request stands on its own as a
modeling/guidance gap inside the BSeR IG's referral workflow.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the Evaluation step in the BSeR referral workflow narrative and
its supporting Task/ServiceRequest profiles so that the outcome of an
evaluation is explicitly modelled as a multi-valued, reversible state
rather than a binary accept/reject. Concretely:

1. In the BSeR workflow narrative ("Referral Workflow" / "Evaluation"
   pages), add a section describing the legitimate non-binary outcomes:
   *information requested*, *alternative offered / counter-proposed*,
   *accepted then withdrawn*, *deferred*, *partially accepted*.
2. Profile `Task` (the BSeR referral Task) to constrain
   `Task.status` to the workflow-relevant values from the FHIR Task
   status code system (e.g., `received`, `accepted`, `rejected`,
   `on-hold`, `in-progress`, `cancelled`, `completed`) and require
   `Task.statusReason` whenever the status changes to `on-hold`,
   `rejected`, or `cancelled`, so the *why* of a non-acceptance is
   captured.
3. Allow / require `Task.businessStatus` to carry the BSeR-specific
   sub-state (e.g., `awaiting-additional-info`, `alternative-offered`),
   bound to a new BSeR ValueSet defined alongside the existing workflow
   ValueSets.
4. For the "alternative offered" case, document the pattern of
   responding with a new (or revised) `ServiceRequest` linked to the
   original via `ServiceRequest.basedOn` (or via a child `Task`
   referencing the original).
5. Document the reversal pattern: a previously-`accepted` Task may
   transition to `cancelled` or `rejected` with `statusReason`
   populated, and the receiver SHALL not delete prior state — the
   history is preserved via Task version history.

#### Justification

This is the literal reading of the request: stop treating Evaluation as
binary and give implementers a defined vocabulary for the messy real
world. FHIR's workflow pattern (Task / ServiceRequest) was designed
precisely for this — `Task.status`, `Task.statusReason`, and
`Task.businessStatus` are the canonical hooks for richer workflow
outcomes, and the FHIR Workflow module already endorses non-binary task
lifecycles. Aligning BSeR with the broader Task state machine improves
interoperability with other workflow-driven IGs (e.g., Da Vinci PDex,
SDOH Clinical Care) and reduces the risk that BSeR implementers invent
private extensions to express the same concepts. The ticket's `Highest`
priority and the lone `BSeR_WORKFLOW` label both suggest this is a
known, blocking gap in the workflow guidance.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than profiling `Task` more tightly and minting new BSeR
ValueSets, address the underlying need by adding **non-normative
guidance** to the BSeR IG that maps the four scenarios Lloyd describes
(deferred decision, request for more info, counter-offer, change of
mind) onto the **existing** FHIR Task / ServiceRequest state machine,
and provide one worked example per scenario:

- *Deferred*: `Task.status = on-hold`, `statusReason = awaiting-info`,
  with a `Communication` or `CommunicationRequest` carrying the question.
- *Counter-offer*: receiver responds with a new `ServiceRequest`
  (`status = active`, `intent = proposal`, `basedOn` = original
  ServiceRequest) and sets the original Task to `rejected` with
  `statusReason = alternative-proposed`.
- *Change of mind after acceptance*: a new `Task` revision moves status
  from `accepted` → `cancelled` with `statusReason = withdrawn`; a
  `Provenance` resource records the actor and reason.

No new profiles, no new ValueSets — just narrative, examples, and a
state-transition diagram in the IG.

#### Justification

BSeR currently ships with a relatively thin workflow narrative; the
practical problem implementers hit is "what do I *do* when the answer
isn't yes-or-no?", not "the spec doesn't allow it." The base FHIR Task
state machine already supports every scenario in the ticket. A
guidance-first response (a) ships much faster, (b) avoids constraining
implementers who are already using the base Task codes, and (c)
preserves room for Disposition A as a follow-up if real-world
implementations diverge. It also avoids inventing BSeR-only codes for
concepts that are likely to show up in other referral IGs, where a
shared HL7 Workflow contribution would be more appropriate than a
BSeR-local ValueSet.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed by existing FHIR mechanisms**. The base
FHIR Task resource's `status`, `statusReason`, and `businessStatus`
elements, together with `Task` history (versioning) and the Workflow
module's defined state transitions, already cover deferred decisions,
requests for more information, counter-proposals (via a new
ServiceRequest with `basedOn`), and change-of-mind reversals. Close as
"Persuasive with Mod" pointing the requester at the relevant Workflow
module documentation rather than changing the BSeR IG.

#### Justification

If the workgroup judges that the ticket is really a documentation-of-
existing-capability concern rather than a gap, declining with a pointer
to the FHIR Workflow module is defensible. The risk is real, however:
the ticket comes from a senior FHIR contributor (Lloyd McKenzie) and is
flagged `Highest` priority with a `BSeR_WORKFLOW` label, both of which
suggest the workgroup itself believes the BSeR-specific narrative is
insufficient. A pure decline is therefore the weakest of the three
options.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the substance of Lloyd's complaint — that BSeR
currently reads as if Evaluation is binary — without committing the
workgroup to a constraining set of new profiles and ValueSets before
real implementations have shaken out which states actually matter.
Because the ticket has no cross-referenced implementations, GitHub PRs,
or Zulip discussions to anchor a specific code system, jumping straight
to Disposition A's profiled vocabulary risks codifying the wrong terms.
Disposition B delivers the *guidance and examples* implementers
actually need now, leaves the door open for a tighter Disposition-A
follow-up once usage patterns are clearer, and keeps BSeR aligned with
the base FHIR Workflow module rather than diverging from it. Disposition
C is not recommended: the `Highest` priority, the `BSeR_WORKFLOW`
label, and the explicit "There should be a discussion of this" wording
all indicate the workgroup itself sees a real documentation gap that
should not simply be closed.
