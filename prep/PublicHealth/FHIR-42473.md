## Summary

Lloyd McKenzie reports that the BSeR (US Bidirectional Services eReferrals)
implementation guide diagrams should be updated to reflect a query-based
referral notification pattern. Specifically, in most real-world referral
notification flows the notification message itself does **not** carry the
clinical payload; instead, the receiving client uses the notification as a
trigger to subsequently query the source system for the actual referral data.
The ticket requests that BSeR's interaction diagrams (and any associated
narrative) make this notify-then-query pattern explicit, since transmitting
payloads inline is undesirable for security (avoid pushing PHI to endpoints
that might not be authorized for it), reliability (avoid duplicating large
payloads in subscription/notification channels), and control (let the source
remain the system of record and authoritative gatekeeper for access). No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found in
the FHIR Augury index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR IG to make the **notify-then-query** pattern the documented
default for referral notifications. Concretely:

- In the BSeR interaction / sequence diagrams (in the "Referral Initiation"
  and "Referral Status Notification" workflow pages), redraw the
  notification arrow so it carries only a *pointer* to the referral
  (e.g., a `Subscription`-style notification or `MessageHeader` with a
  `focus` reference), followed by a return arrow showing the recipient
  performing a `GET ServiceRequest/{id}` (and/or related `$everything` /
  bundle query) against the source server.
- Add narrative text on the relevant workflow page that explicitly states:
  notifications do not convey the clinical payload; recipients MUST query
  the source system for the referenced resources after receiving a
  notification, and explain the security, reliability, and access-control
  rationale supplied in the ticket.
- Where the IG currently includes example bundles or message payloads that
  embed the full referral content in the notification, replace them with
  examples showing an empty-/reference-only notification plus a follow-up
  query/response pair.
- Cross-check and update any conformance language (SHALL/SHOULD) so that
  servers MUST support the query interaction and notifications SHOULD NOT
  include the full payload.

#### Justification

This matches the stated request verbatim and aligns BSeR with broader FHIR
Subscriptions R5 / Subscription Backport guidance, which similarly favors
notification-as-trigger over notification-as-payload when PHI is involved.
The three rationales Lloyd cites — security (do not push PHI to endpoints
that may not be authorized to receive it), reliability (avoid duplicating
potentially large payloads through messaging infrastructure), and control
(keep the source as the authoritative system of record and access-control
point) — are well-established design principles in HL7 work and have been
applied consistently in other US Realm IGs that touch referrals and care
coordination. Because this is a diagram and narrative change rather than a
structural model change, it carries low implementation risk.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mandating notify-then-query as the sole pattern, document
**both** a "notify-only" (pointer) pattern and a "notify-with-payload"
pattern in the BSeR IG, and let trading partners select between them based
on the trust relationship and transport. The diagrams would show two
parallel flows side-by-side, with explicit guidance on when each is
appropriate:

- *Notify-only / query-back*: default; required when the recipient is not
  pre-authorized for the payload, when payloads are large, or when the
  source must retain access-control over each retrieval.
- *Notify-with-payload*: optional; permitted when the channel is already
  trusted (e.g., a Direct/secure-messaging endpoint with a pre-existing
  BAA) and reliability concerns are mitigated.

Conformance would require all servers to support the notify-only pattern
and make notify-with-payload an optional capability declared in the
`CapabilityStatement` and/or BSeR `MessageDefinition` profile.

#### Justification

Some BSeR deployments (particularly those using existing Direct
secure-messaging infrastructure between a referring clinician and a
specialty/public-health program) already operate with payload-bearing
notifications, and forcing all of them onto a query-back model would impose
a non-trivial migration cost and require the recipient to stand up a
pull-capable server they may not have today. A two-pattern approach
preserves the security/control benefits Lloyd identifies as the **default**,
while not orphaning current implementers. The trade-off versus Disposition A
is added documentation surface and the risk of implementer confusion about
which pattern to choose; this is mitigated by clear conformance language
making notify-only mandatory-to-support.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that BSeR's current diagrams already
permit (and arguably already imply) a query-back interaction through the
underlying FHIR RESTful and Subscription mechanisms, and that no specific
diagram or narrative defect has been identified in the ticket. Close as
"Persuasive with Mod – No Change" and note that implementers needing a
query-back pattern can already realize it using base FHIR Subscriptions
without IG modification.

#### Justification

The ticket does not point to a specific diagram, page, or sentence that is
incorrect; it expresses a general preference for the notify-then-query
pattern. If the underlying message and Subscription infrastructure used by
BSeR already supports both styles, then the IG is not technically wrong —
only incomplete with respect to guidance. A workgroup could reasonably
decide that adding the guidance is desirable but not a *change* to the
specification, or that the change should wait until a coordinated revision
of the BSeR notification model (the broader `BSeR_NOTIFY_SEND_CONTENTS`
label suggests this is one of a cluster of related tickets that may be
better dispositioned together).

---

### Recommendation

**Recommended disposition:** A

Disposition A directly addresses the request, has low implementation cost
(diagram + narrative changes, plus a conformance tightening), and aligns
BSeR with the prevailing HL7 design pattern of notification-as-trigger when
PHI is in play. The reporter's three rationales — security, reliability,
and control — are exactly the rationales cited elsewhere in HL7 work for
preferring query-back over payload-bearing notifications, and applying that
pattern to BSeR keeps the IG consistent with peer specifications. Disposition
B is attractive if the work group has evidence of significant existing
payload-bearing deployments, but absent that evidence (and none is captured
in cross-references or Zulip discussion), the simpler single-pattern guidance
is preferable. Disposition C is hard to defend given that the reporter is the
HL7 methodologist who originally helped author the relevant patterns; his
observation that the diagrams *should* show the notify-then-query interaction
is itself the defect being reported. Reviewers should also consider
batch-dispositioning this ticket alongside the other `BSeR_NOTIFY_SEND_CONTENTS`-labeled
tickets so that the diagram updates and the conformance language move
together.
