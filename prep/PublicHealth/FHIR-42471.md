## Summary

The BSeR (US Bidirectional Services eReferrals) IG currently does not
define — nor does it explicitly reference — the `SubscriptionTopic`
resource(s) that BSeR conformant systems should subscribe to in order to
support the bidirectional notification flow that BSeR implies. Lloyd
McKenzie (as part of imported review comment 22) is asking the Public
Health work group to standardise this: the IG itself should either
publish the canonical `SubscriptionTopic` instance(s) that BSeR uses or
reference an externally published topic, so that the trigger criteria
(resource type, filter, supported interactions, notification shape) are
fixed by the IG rather than left for each implementer to invent. The
ticket is highest priority and triaged but unresolved, with no comments
recorded in Augury and no other Jira tickets, Zulip threads, or GitHub
items currently cross-referenced to it in the knowledge base.

## Proposed Dispositions

> Note: there are no cross-referenced repos, Zulip threads, or linked

> the Public Health portfolio. They will need to be re-grounded against

### Disposition A: Accept As Requested

#### Proposal

Author SubscriptionTopic instance(s) inside the BSeR IG that codify the
bidirectional referral notification flows BSeR depends on. Concretely,
add at least one canonical `SubscriptionTopic` per direction the IG
already describes (e.g., "BSeR ServiceRequest status change" for the
referral side and "BSeR Task / referral-feedback update" for the
response side). Each topic should fix:

- a stable canonical URL under the IG's canonical base
  (e.g., `http://hl7.org/fhir/us/bser/SubscriptionTopic/<name>`),
- the `resourceTrigger` (resource type plus `supportedInteraction` and
  `queryCriteria`/`fhirPathCriteria` aligned with the BSeR-profiled
  `ServiceRequest`/`Task`/feedback resources),
- `notificationShape` listing the `include`/`revInclude` set
  implementers must surface in the bundle (referral, feedback Task,
  associated Patient, MeasureReport, etc.),
- conformance language in a new "Subscriptions" page in the IG that
  mandates support for these topics in the bidirectional channel.

#### Justification

This is exactly what the reporter asked for. SubscriptionTopic is the
R5/backport mechanism the wider FHIR community has standardised on for
exactly this kind of "well-known event" the IG already implies.
Defining the topics in-IG removes the per-implementer variation Lloyd
flags, and it gives certifiers and test tooling a single canonical URL
to assert against. It is also consistent with how other Public Health
IGs (e.g., eCR Now, MedMorph) publish their topics: the IG owns the
trigger semantics, not the deployment.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting new BSeR-specific `SubscriptionTopic` instances,
have the IG **reference** an externally published topic set — e.g., a
common Public Health "referral lifecycle" topic published by MedMorph
or by the FHIR Subscriptions / Public Health joint topic registry —
and only profile/extend it with BSeR-specific `notificationShape`
includes via the IG. The IG's "Subscriptions" page would then point at
the external canonical URL and add a small conformance overlay
(required filters, required includes for BSeR profiles).

#### Justification

The ticket explicitly allows "define **or reference**" the topics, so
referencing satisfies the underlying need while avoiding fragmentation
of the Public Health topic ecosystem. If a suitable referral-lifecycle
topic already exists (or is on the near-term roadmap for a sibling PH
IG), reusing it keeps trigger semantics consistent across PH use-cases
that share the same underlying `ServiceRequest`/`Task` lifecycle, and
it lets server vendors implement one topic that serves multiple IGs.
The trade-off vs. Disposition A is governance: BSeR loses unilateral
control over the trigger definition and inherits the maintenance
cadence of the upstream topic.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave SubscriptionTopic definition out of the
BSeR IG, on the grounds that BSeR's normative content is the
referral/feedback content model itself, and the choice of notification
mechanism (R5 SubscriptionTopic vs. R4 Subscription criteria vs. a
non-FHIR transport) should remain a deployment-level decision.

#### Justification

This is defensible only if the work group concludes that BSeR is not in
fact prescribing a notification model and that interoperability is
already adequately constrained by the profiled `ServiceRequest` /
`Task` / feedback resources. It would, however, leave Lloyd's stated
problem unsolved — implementations would continue to vary in how (or
whether) they expose subscription topics for BSeR events, undermining
the "bidirectional" promise of the IG. Decline is therefore only
appropriate if the WG actively decides BSeR should *not* mandate any
topic-based subscription support.

---

### Recommendation

**Recommended disposition:** A

Accepting as requested directly resolves the variability Lloyd flagged
and aligns BSeR with the prevailing Public Health pattern of shipping
canonical `SubscriptionTopic` instances inside the IG. Disposition B is
attractive in principle, but is only viable once a concrete external
topic exists that the WG is willing to depend on; today there is no
cross-referenced PH topic registry the IG could point at without
inventing one. Disposition C leaves the original interoperability gap
in place and contradicts BSeR's own bidirectional framing. The WG
should plan to publish the topic(s) in the IG, and revisit Disposition
B opportunistically if/when a shared PH referral-lifecycle topic
materialises and BSeR can re-anchor to it without behavioural change.
