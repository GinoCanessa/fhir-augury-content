## Summary

This ticket requests that ImmunizationRecommendation's "Boundaries and
Relationships" section adopt the standard cross-link to Task / the Request
pattern's Fulfillment/Execution section that was agreed for all "Request"
resources in the parent ticket
[FHIR-45872](https://jira.hl7.org/browse/FHIR-45872) (FHIR Infrastructure,
Resolved "Persuasive with Modification"). FHIR-45872's resolution committed to
filing change requests against each Request resource asking them to add the
sentence: *"The [MyRequest] resource represents an authorization for a service
to be provided. Details about the fulfillment of the authorization are handled
by the [Task] resource. For further information about this separation of
responsibilities, refer to the [Fulfillment/Execution] section of the Request
pattern."* — with allowance for re-wording on resources where "authorization"
is not the right framing (CarePlan, Appointment, Claim, etc.).

The ImmunizationRecommendation Focus Group has pushed back: the resource is
explicitly *only* a proposal (never an authorization) and is not directly
fulfilled — clinical action flows through a separately generated
MedicationRequest (linked via `MedicationRequest.basedOn`) and is documented
via Immunization. Craig Newman (2024-07-19) raised that the proposed wording
contradicts existing ImmunizationRecommendation text and signaled a leaning
toward "Not Persuasive." Jose Costa-Teixeira (2024-10-21) offered a softened
re-wording ("proposal" instead of "authorization"; "can be handled by Task
*and other resources*") to satisfy the workflow-pattern consistency goal
without misrepresenting the resource. Nathan Bunker (2025-01-31) reported
that the Immunization Focus Group concluded the group does *not* see a need
for any direct fulfillment reference on ImmunizationRecommendation, noting
the resource fits the Request pattern only tenuously.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the ImmunizationRecommendation resource's "Boundaries and
Relationships" section to add the standard FHIR-45872 cross-link sentence,
with the targeted re-wording invited by FHIR-45872's resolution to fit a
proposal-only resource: *"The ImmunizationRecommendation resource represents
a proposal for one or more immunization services to be provided. Where
fulfillment activity needs to be tracked, it is handled via separately
generated MedicationRequest and Immunization resources (linked back via
`MedicationRequest.basedOn`) and may additionally be coordinated using the
[Task] resource. For further information about the separation between
authorization and fulfillment responsibilities, refer to the
[Fulfillment/Execution] section of the Request pattern."* Place the new
paragraph adjacent to the existing MedicationRequest/Immunization paragraph
in Boundaries and Relationships to preserve the existing narrative.

#### Justification

This satisfies the cross-FHIR consistency goal of the parent FHIR-45872
(every Request-pattern resource cross-links to the new
Fulfillment/Execution rationale) while honoring Craig Newman's correct
observation that ImmunizationRecommendation is never an authorization.
FHIR-45872's resolution explicitly anticipated that some Request resources
would need re-worded variants — this disposition uses that latitude rather
than ignoring it. It also reinforces, rather than contradicts, the existing
ImmunizationRecommendation Scope and Usage statement that the resource's
intent is always "proposal," and it surfaces the MedicationRequest /
Immunization hand-off that implementers most need to find.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add the FHIR-45872 sentence to ImmunizationRecommendation's
Boundaries and Relationships. Instead, add a short forward-pointer in
ImmunizationRecommendation's Scope and Usage (immediately after the
existing "intent is always 'proposal'" sentence) reading approximately:
*"Because ImmunizationRecommendation is purely a proposal, it is not itself
fulfilled; clinical fulfillment is tracked on the MedicationRequest(s)
generated from a recommendation (see
[MedicationRequest](medicationrequest.html)) and the resulting
[Immunization](immunization.html) records. See the Request pattern's
[Fulfillment/Execution](workflow.html#fulfillment) section for the general
separation between authorization and fulfillment."* Leave the Boundaries
and Relationships paragraph that already describes the
MedicationRequest/Immunization relationship intact, and file a companion
ticket against the workflow/Request pattern page noting that
ImmunizationRecommendation is a Request-pattern resource that is never
itself fulfilled, so implementers should not look for a Task hand-off on it.

#### Justification

This addresses Nathan Bunker's substantive concern (the resource is not
itself fulfilled, so directing readers to Task is misleading) while still
giving implementers the cross-link to the Request-pattern fulfillment
discussion that FHIR-45872 was designed to surface. Co-locating the link
with the existing "intent is always 'proposal'" sentence — rather than in
Boundaries and Relationships, where the requested wording would land
awkwardly — keeps the two pieces of guidance reinforcing each other.
Filing the workflow-pattern companion change documents
ImmunizationRecommendation as a known exception so future readers of the
Request pattern do not assume the standard sentence is missing by oversight.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive**. Take no change to ImmunizationRecommendation
based on FHIR-45872. Capture in the resolution that
ImmunizationRecommendation is, by definition (Scope and Usage), only a
proposal and is never directly fulfilled — fulfillment is tracked on the
MedicationRequest(s) generated from the recommendation and on the resulting
Immunization records, both of which are already documented in the existing
Boundaries and Relationships text and linked via `MedicationRequest.basedOn`.
Recommend that FHIR-45872's pattern-level guidance call out
ImmunizationRecommendation as an explicit exception.

#### Justification

This is the position the Immunization Focus Group reached on 2025-01-31
("Group does not see a need for ImmunizationRecommendation to have any
reference to fulfillment directly") and it is consistent with Craig Newman's
2024-07-19 analysis that the proposed sentence contradicts the resource's
own Scope and Usage. The hand-off implementers actually need
(ImmunizationRecommendation → MedicationRequest → Immunization) is already
present and is more accurate than a Task pointer. Because FHIR-45872 itself
acknowledged that some Request resources may not fit the standard wording,
declining here is fully consistent with that ticket's resolution.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible compromise. Disposition C accurately
reflects what the Immunization Focus Group resolved, but it leaves
implementers cross-walking from the workflow/Request pattern page to
ImmunizationRecommendation with no breadcrumb back, which is precisely the
gap FHIR-45872 was filed to close. Disposition A inserts the link in the
section the original ticket asked for, but the "Boundaries and
Relationships" paragraph is the wrong home for it — that section already
describes the MedicationRequest/Immunization hand-off correctly, and
inserting a Task-fulfillment sentence there would create the very
contradiction Craig Newman flagged. Disposition B keeps the cross-link the
parent ticket wanted (consistent with FHIR-45872's anticipation that some
resources need re-worded variants), places it where it actually fits (next
to the "intent is always 'proposal'" statement that defines why this
resource is different), and pairs it with a companion workflow-page change
so the Request pattern itself records ImmunizationRecommendation as a
known exception. The Public Health workgroup may, of course, prefer
Disposition C if they want to make a stronger point that the resource's
Request-pattern classification is itself questionable — Nathan Bunker's
comment opens that door.
