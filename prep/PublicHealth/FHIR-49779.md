## Summary

Bryn Rhodes is asking the Public Health work group to consider moving the
`receiver-address` extension out of its current home and into the FHIR
Extensions Pack so that it is available for reuse outside of any single
public-health implementation guide. The extension was originally defined
in eCR and has since migrated to the US Public Health Profiles Library
(USPHPL) — per the most recent comment from `minigrrl` (2025-11-17) —
which is why the ticket was re-routed onto the USPHPL specification.
The underlying question (does this extension belong in a Public Health
IG, or in a cross-IG extensions pack?) is unchanged by that move. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FhirAugury for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the `receiver-address` extension definition out of the US Public
Health Profiles Library (USPHPL) and into the HL7 FHIR Extensions Pack
(`HL7/fhir-extensions`). Concretely:

- Author a new `StructureDefinition-receiver-address` in the
  Extensions Pack, preserving the current canonical semantics
  (context(s), `value[x]` typing, cardinality, and short/definition
  text) so existing instances remain valid.
- Assign the extension a new canonical URL under the Extensions Pack
  namespace (e.g., `http://hl7.org/fhir/StructureDefinition/receiver-address`).
- In USPHPL (and any remaining eCR profile that still references the
  extension), replace the local extension reference with the new
  Extensions Pack canonical and add a deprecation/redirect note in the
  USPHPL artifact pointing readers at the Extensions Pack copy.
- Coordinate with the eCR maintainers so the previous canonical (if
  one was published from eCR) is marked retired and pointed at the new
  canonical.

#### Justification

The Extensions Pack exists specifically to host extensions that are
useful across multiple IGs and that are not bound to a single domain
profile. A "receiver address" concept is general — it describes the
postal/electronic address of a message receiver — and is plausibly
useful outside Public Health (e.g., other messaging-based IGs that
need to convey a receiver address that doesn't fit naturally on
`MessageHeader.destination`). Centralising it in the Extensions Pack
avoids three different IGs each minting their own near-duplicate
extension, and matches the recent direction of work on common
extensions. The reporter (Bryn Rhodes) is well-placed on the broader
FHIR-I / cross-IG side to identify candidates for that pack.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the `receiver-address` extension in the US Public Health Profiles
Library for now, but formally elevate it to a "shared" USPHPL
extension (clearly documented as reusable and stable), and open a
follow-up ticket against the Extensions Pack to evaluate adoption
there in the next Extensions Pack ballot cycle. Specifically:

- Leave the canonical URL where it is (USPHPL) so no consumers break.
- Update the USPHPL narrative for the extension to clarify it is
  intended for reuse beyond eCR, and document its context(s) and
  intended use cases.
- File a new ticket on the Extensions Pack project ("Adopt
  USPHPL receiver-address extension") referencing FHIR-49779, so the
  Extensions Pack maintainers can evaluate whether the semantics are
  generic enough and whether they want to take ownership.

#### Justification

Moving an already-published extension between IGs is not free: it
changes the canonical URL, requires coordination with current
implementers (eCR senders/receivers), and risks instances that
reference the old canonical becoming non-conformant. If the extension
is currently only used by USPHPL/eCR consumers, moving it now buys
generality at the cost of churn for the people using it today. A
two-step path — stabilise it where it is, then have the Extensions
Pack adopt it on its own ballot timeline with the appropriate
versioning/redirect machinery — gets the same long-term outcome with
less near-term disruption to Public Health implementers.

---

### Disposition C: Decline

#### Proposal

Decline the move. Keep the `receiver-address` extension in the US
Public Health Profiles Library, on the grounds that its semantics and
context are tied to public-health messaging patterns (in particular
eCR-style routing of case reports to a named receiver) and that it
does not have demonstrated cross-IG demand. Close the ticket as "not
persuaded" with a note that the request can be re-opened if and when
another IG wants to reference the same extension.

#### Justification

The Extensions Pack is intentionally curated; not every reusable-looking
extension belongs there. The ticket itself is a one-line "consider"
with no enumerated consumers outside Public Health, and the most recent
movement on it has been re-homing within Public Health (eCR → USPHPL),
not outward expansion. Without at least one concrete non-PH consumer
asking for the extension, hosting it in the Extensions Pack adds
maintenance load on that pack's stewards without a clear payoff, and
risks setting a precedent of speculative migration of every
domain-specific address/contact extension.

---

### Recommendation

**Recommended disposition:** B

The reporter's instinct is sound — `receiver-address` reads like a
generic concept that other IGs could reuse, and the Extensions Pack is
the right long-term home for that kind of artifact. But the extension
has just been moved (eCR → USPHPL) and the ticket carries no concrete
cross-IG consumer yet, so an immediate second move risks churning
implementers without a demonstrated payoff. Disposition B captures
both realities: stabilise the extension's documentation and reuse
intent in USPHPL now, and hand the migration question to the
Extensions Pack maintainers as a tracked follow-up so they can adopt
it on their own ballot cycle (with the canonical-URL/redirect work
done properly). If the Extensions Pack is willing to pick it up
quickly and the Public Health WG is comfortable with the churn,
Disposition A is a reasonable upgrade path; Disposition C should only
win if the WG concludes the extension's semantics are genuinely
PH-specific.
