## Summary

The ticket asks the VRDR work group to decide how to encode "Place of Injury"
(INJPL) so that jurisdictions can submit records using the **SuperMICAR**
letter codes in addition to the NCHS **TRX** numeric codes that VRDR already
supports. The IJE specification permits either coding system for INJPL, and
the SuperMICAR value set is a strict superset of TRX — it adds 7 concepts
(Military Residence, Hospital, Garage/Warehouse, Mine/Quarry, Public
Recreation Area, Institutional Recreation Area, Other Specified Place) that
do **not** have entries in the LOINC answer list (`LL1051-3`) currently used
by VRDR. The reporter asks two concrete questions: (1) should VRDR support
the full 17-concept value set, and (2) if so, should new local codes be
minted for the 7 missing concepts (some can be covered by SNOMED CT —
roughly 4/7 precisely, 1/7 imprecisely, leaving 2 needing local codes)?

There are **no cross-referenced Jira tickets, Zulip threads, or GitHub
items** in the FHIR Augury index for this ticket. The history visible on the
ticket itself shows the work group disposed it as **"Considered for Future
Use" (25–0–0) on 2024-03-28**, with the rationale "no requirement has been
identified". Lynn Laakso then **reverted that resolution and reopened the
ticket post-STU3 publication on 2024-10-16**, which is its current state
(Triaged, Unresolved). Saul Kravitz had previously noted in 2023 that it
would not be addressed in STU2.2.

## Details

**Description:**

The IJE specification says that INJPL (coded place of injury) can be
specified with EITHER SuperMICAR codes (letters) or TRX codes (numbers).
The SuperMICAR value set (first column) is larger than the TRX value set
(2nd column) by 7 codes. NCHS uses the TRX value set. Some of the
jurisdictions wanted to create records using SuperMICAR codes.

The answer list that is the source of our LOINC codes for this does not
include the highlighted 7 concepts — see [LL1051-3](https://loinc.org/LL1051-3/).

**Questions:**

1. Should we support the full 17-code value set for INJPL in the VRDR IG?
2. Should we define local codes for the 7 missing concepts? There are no
   existing LOINC codes. An initial look finds that there are SNOMED
   environment codes available that would cover 4 of the 7 precisely, 1/7
   imprecisely. Leaving 2 that would require local codes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the VRDR IG's "Place of Injury" value set so it covers the full
17-concept SuperMICAR/TRX union used by IJE INJPL. Concretely, in the VRDR
ValueSet that backs `Observation-InjuryIncident.component:PlaceOfInjury`
(currently bound to the LOINC answer-list concepts derived from
`LL1051-3`):

1. Keep the existing 10 LOINC `LA…` codes that are equivalent across
   SuperMICAR and TRX.
2. Add the 4 SNOMED CT codes that map cleanly to SuperMICAR concepts:
   - D Military Residence → `SCT 224683003`
   - E Hospital → `SCT 22232009`
   - J Mine/Quarry → `SCT 257640001`
   - L Public Recreation Area → `SCT 702703009` (Outdoor Public Site)
3. For the imprecise mapping (H Garage/Warehouse → `SCT 272506004` or
   `257727001`), pick one SCT code as the IG's normative choice and
   document the imprecision.
4. Mint VRDR-local codes (in a new VRDR-namespaced `CodeSystem`) for the
   2 remaining SuperMICAR concepts that have no good SCT equivalent
   (M Institutional Recreation Area, P Other specified place).
5. Publish a `ConceptMap` from the VRDR value set to the TRX numeric
   codes so jurisdictional payloads using SuperMICAR letters can still
   round-trip to NCHS TRX submissions.

#### Justification

This directly answers both questions the reporter posed and removes the
asymmetry between what jurisdictions can encode in IJE and what the VRDR
FHIR IG accepts. Using SCT where it exists keeps the IG aligned with FHIR's
terminology preferences, and a VRDR-local CodeSystem for the 2 residual
concepts is a well-established VRDR pattern (the IG already defines several
local code systems where LOINC/SCT are insufficient). The accompanying
`ConceptMap` to TRX preserves NCHS interoperability, which was the
original concern in the 2024-03-28 vote ("no requirement identified") —
explicit jurisdictional demand is now the requirement.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the bound value set, keep the VRDR-required binding
to the existing 10-concept LOINC answer-list value set (extensible
binding) and define an **extension** —
`StructureDefinition-PlaceOfInjury-SuperMICAR-Source` — that lets a
producer also carry the original SuperMICAR letter code as a
`Coding` alongside the standard answer. Populate the IG's IJE-to-FHIR and
FHIR-to-IJE mapping pages so that any of the 7 SuperMICAR-only letters
maps to an `Observation.component` whose primary `Coding` is the closest
LOINC `LA…` (or `OTH (Other)` for the residual cases) and whose extension
carries the literal SuperMICAR letter for fidelity.

#### Justification

This preserves the existing terminology surface — no new local CodeSystem,
no new SCT codes, no `ConceptMap` to maintain — and keeps STU3 consumers
working unchanged. It addresses the underlying need (jurisdictions must be
able to submit SuperMICAR-coded data without information loss) without
broadening the normative value set that downstream tools have already
implemented against. The trade-off is that the SuperMICAR letter is no
longer first-class terminology in VRDR; consumers that care about the
extra distinctions (Military Residence, Hospital, Mine/Quarry, etc.) must
read the extension rather than just the bound code.

---

### Disposition C: Decline

#### Proposal

Decline the request and document VRDR as TRX-aligned for INJPL. Close
FHIR-39603 as **Persuasive with Mod** with a note that the VRDR IG
intentionally tracks the NCHS TRX 10-concept value set (which is what
NCHS actually consumes), and that jurisdictions that need to retain a
SuperMICAR-letter-faithful record should do so in their internal systems,
mapping to the closest TRX code on submission. Add a normative note to the
PlaceOfInjury element documenting this position so it does not get raised
again.

#### Justification

This is the position the work group reached on 2024-03-28 (25-0-0): "No
requirement has been identified". The IG is built around NCHS submission,
NCHS uses TRX, and the 10-concept LOINC answer list already covers the
TRX value set with equivalent semantics. Adding 7 more concepts plus a
new local CodeSystem plus a ConceptMap is a non-trivial maintenance burden
for an edge case (only "some jurisdictions" want SuperMICAR fidelity), and
several of the missing concepts (Hospital, Military Residence) are
already inferable from other VRDR data (Disposition.method, decedent
context) — meaning the loss is smaller than it appears.

---

### Recommendation

**Recommended disposition:** A

The work group's earlier "Considered for Future Use" (Disposition C in
spirit) was explicitly **reverted post-STU3 publication** by Lynn Laakso
on 2024-10-16 — the very rationale used to dispose it ("no requirement
has been identified") no longer holds, because jurisdictional demand is
now the trigger for reopening. Disposition C therefore re-litigates a
decision the work group already chose to undo.

Disposition B (extension) is the safer-looking middle path, but it
solves the round-trip problem without solving the **semantic** problem:
downstream consumers of VRDR FHIR data still cannot distinguish a
hospital injury from "other" without parsing an extension that most tools
will ignore. For mortality analytics that is a real loss.

Disposition A is the right answer because (a) the IJE source of truth
already permits the full 17-concept set, (b) the technical lift is
modest — most missing concepts have clean SNOMED CT mappings, only 2
genuinely need VRDR-local codes — and (c) a `ConceptMap` from the
expanded VRDR value set back to TRX preserves NCHS submission paths
unchanged. The work group should treat this as a STU3-cycle change and
publish the expanded value set, the local CodeSystem for the 2 residual
concepts, and the SuperMICAR↔TRX `ConceptMap` together so jurisdictions
and NCHS see a single coherent update.
