## Summary

Lloyd McKenzie (originally Comment 30 imported into a parent ticket) raises
that the BSeR IG names many "Referral Observation" concepts (e.g., "Ability
to latch onto breast for feeding", "Maternal concern", "Baby's name",
"Baby's sex") without specifying enough to allow interoperability. For each
such observation the IG should pin down: the LOINC (or other) code on
`Observation.code`, the answer data type for `Observation.value[x]`, and —
when coded — the value set, or — when quantitative — the unit. He also
flags semantic questions for non-Observation data, e.g., whether "Baby's
sex" expects `Patient.gender` or one of the US Core sex/gender extensions
(this is the only element captured as a cross-reference on the ticket:
`Patient.gender`), and how multiple "Baby's name" instances would be
represented.

Beyond per-element constraints, the ticket asks the IG to specify
*how much* of each observation to share — for example, whether
"hypertension blood pressure" means the most recent BP, the last year's
worth (including continuous in-hospital readings), or a lifetime — and
*which* `Condition`s/diagnoses are valid responses for things like
hypertension. This is a request for substantially tighter conformance
across BSeR Observation profiles and their associated request/response
bundles.

The cross-reference index attaches no other Jira tickets, Zulip threads,
Confluence pages, or GitHub items to FHIR-42479 directly — the only edge
is the FHIR element mention `Patient.gender`. Several other BSeR tickets
(FHIR-41901, FHIR-41859, FHIR-20831, FHIR-20856, FHIR-22613) raise
adjacent under-specification concerns; they are summarised under
"Related Jira Tickets" below for context.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten every BSeR Referral Observation profile so that, for each named
concept (e.g., "Ability to latch onto breast for feeding", "Maternal
concern", "Baby's name", "Baby's sex"), the profile fixes:

1. `Observation.code` to a specific LOINC code (or, when LOINC truly does
   not cover the concept, an explicitly justified BSeR-minted code in the
   IG's CodeSystem) with cardinality 1..1.
2. `Observation.value[x]` to a single permitted data type (`CodeableConcept`,
   `Quantity`, `string`, `boolean`, `integer`, …) with the irrelevant
   `value[x]` slices removed.
3. For coded answers: a `required` (or, where extensibility matters,
   `extensible`) binding to a named, versioned BSeR ValueSet.
4. For quantitative answers: a fixed UCUM unit (or a UCUM-bound ValueSet
   when more than one unit is sensible — see FHIR-22613's kg/lbs
   precedent) and required `Observation.valueQuantity.system` /
   `.code`.
5. For each "scope" question raised in the description, add either a
   profile-level invariant or normative IG narrative stating the
   expected window of observations to share (e.g., "most recent BP
   within 12 months for hypertension referrals") and the bound value set
   of acceptable response Conditions for each clinical concern (e.g.,
   hypertension → ICD-10/SNOMED CT subset).
6. For "Baby's sex", explicitly state in the IG (and, if needed, in the
   profile differential) whether the recipient should populate
   `Patient.gender` or a US Core sex/gender extension; the existing
   cross-reference to `Patient.gender` already pins the discussion to
   that element.

Track this work as a coordinated set of edits across all BSeR Observation
profiles plus the BSeR ValueSets / CodeSystems pages of the IG.

#### Justification

This is what the ticket literally asks for, and it is consistent with how
the WG has handled directly comparable BSeR tickets: `Observation.category`
cardinality and `subject` reference target tightening (FHIR-20831, *Applied
/ Persuasive*), the A1c LOINC code question (FHIR-20856, *Applied /
Persuasive*), and the BSeR Body Weight UCUM unit question (FHIR-22613,
*Applied*). The WG has already accepted that the BSeR response shape needs
to be tightened (FHIR-41901, *Persuasive with Modification*), so applying
the same standard at the per-Observation conformance level is in-line with
established direction. Without these constraints, two BSeR implementations
can claim conformance while exchanging mutually-uninterpretable instances,
which defeats the IG's stated interoperability goal.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than treating FHIR-42479 as a single ticket to "fix everything", split
it into a tracked work programme and address it in two layers:

1. **IG-level conformance policy.** Add a new "Observation conformance
   requirements" page to the BSeR IG that states, normatively, that every
   Referral Observation profile MUST fix `Observation.code` (1..1 with a
   single code or a `required`-bound ValueSet), MUST constrain
   `Observation.value[x]` to a single data type, MUST bind coded answers
   to a named ValueSet, and MUST fix UCUM units (or bind a UCUM ValueSet)
   for quantitative answers. Also state the scope-of-data expectations
   ("recipients SHALL accept the most recent N observations within window
   W, unless requested otherwise"), and require each profile page to
   explicitly state the response Condition value set.
2. **Per-profile child tickets.** Spawn one child ticket per
   under-specified concept (e.g., "Ability to latch onto breast for
   feeding", "Maternal concern", "Baby's name", "Baby's sex",
   "Hypertension blood pressure scope", "Hypertension response
   diagnoses", …) under FHIR-42479, each carrying the LOINC / value set
   / unit / cardinality decision so it can be reviewed by the
   appropriate clinical SME without blocking the others.

For "Baby's sex" specifically, prefer using the US Core sex / gender
extensions over `Patient.gender` (since `Patient.gender` is the
administrative concept and US Core already provides the more nuanced
clinical extensions that BSeR's neonatal use cases need); document that
choice in the IG's "Patient" profile narrative.

#### Justification

This addresses the same underlying need as Disposition A but acknowledges
two practical issues. First, "tighten every Observation profile" is a
large body of work that will require LOINC and value-set decisions from
clinical SMEs that the ticket itself does not provide; bundling those
decisions into a single ticket invites stalling. Second, the ticket
already conflates two distinct concerns (per-element conformance vs.
scope/recency of data exchanged) — a layered policy plus per-concept
children separates an IG-level decision (one vote) from many
clinical-content decisions (many votes), which is the same pattern the
WG used implicitly when it accepted FHIR-20831, FHIR-20856, and
FHIR-22613 as separate, narrowly-scoped tickets that all "tighten
Observation profiles". It also delivers immediate value (the policy page)
even if some per-concept children remain open between ballot cycles.

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-42479 as *Not Persuasive* (or *Considered — No Action
Required*), on the grounds that BSeR is a referral *transport* IG and
that the appropriate place to fix LOINC codes, value sets, and units for
clinical observations is the receiving / authoring IGs (e.g., US Core,
US Public Health Profiles, condition-specific IGs), not BSeR itself.
Direct the reporter to file the per-concept tightening requests against
the profile owners.

#### Justification

There is a defensible scoping argument: BSeR's job is to package a
referral and a response into bundles and to route them; pinning down
which LOINC code "Ability to latch onto breast for feeding" maps to is a
clinical-content decision that arguably belongs to a maternal-child or
public-health observation IG, not to a referral IG. Declining on those
grounds keeps BSeR's footprint narrow and avoids BSeR forking
clinical-observation conformance from the rest of US Realm. This option
is included for completeness; the existing precedents below make it the
weakest of the three.

---

### Recommendation

**Recommended disposition:** **B**

Disposition A captures exactly what Lloyd asked for, and the WG has a
strong precedent of accepting that kind of tightening on a
per-Observation basis (FHIR-20831, FHIR-20856, FHIR-22613 are all
*Applied / Persuasive*). However, FHIR-42479 as written is genuinely two
asks — "fix per-element conformance on every Observation profile" and
"fix scope/recency/response-Condition expectations" — and resolving it
as one monolithic change request risks either (i) the ticket being
re-triaged repeatedly while clinical SMEs work each LOINC binding, or
(ii) the WG accepting it as a bare commitment without the substantive
follow-through. Splitting it into a normative IG policy page plus
per-concept child tickets gives the WG a single decision they can act
on this cycle, and a tracked queue of clinical bindings that can be
attacked incrementally without blocking the IG.

Disposition C is not recommended: BSeR has historically accepted
*Persuasive* dispositions on the same class of concerns (kg/lbs units,
A1c LOINC selection, `Observation.category` cardinality), so deferring
the entire question to other IGs would contradict that precedent and
leave the IG materially under-specified for interoperability — which is
precisely what the ticket says it already is.
