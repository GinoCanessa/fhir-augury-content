## Summary

The ticket asks why the BSeR (US Bidirectional Services eReferrals) IG's
use of `Task` does not specify a standardized code system for
`Task.input.type` and `Task.output.type`. Lloyd McKenzie's position is
that interoperability requires senders and receivers to agree on
identifier codes for the inputs and outputs Task carries, and that
`Task.input` is the slot that should reference the various "requester
package" content (with `Task.output` carrying the corresponding
"response package" content). Without a fixed code set, two implementers
exchanging BSeR referrals cannot reliably identify which package payload
each `Task.input`/`Task.output` element represents.

Closely related upstream work has since landed at the FHIR Core level:
[FHIR-50258](https://jira.hl7.org/browse/FHIR-50258) — *"Add new Task
code and input and output codes to support data retrieval tasks"* — was
resolved **Persuasive** by Orders & Observations and is now **Applied**
in the core spec, defining a new `task-code` value (`data-request`) plus
a `task-inputoutput-parameter-type` code system. That work generalises a
pattern Da Vinci had been using and migrates the codes into core/UTG.
Other BSeR Task-related tickets reported by the same author —
[FHIR-42518](https://jira.hl7.org/browse/FHIR-42518) (Task.owner should
be US Core), [FHIR-42501](https://jira.hl7.org/browse/FHIR-42501)
(Cancellation Request), [FHIR-42520](https://jira.hl7.org/browse/FHIR-42520)
(don't use a fixed v2 event code), [FHIR-42507](https://jira.hl7.org/browse/FHIR-42507)
(Encounter scheduling not part of Task), and
[FHIR-42456](https://jira.hl7.org/browse/FHIR-42456) (package
determination) — establish that BSeR's overall Task usage needs a more
disciplined, code-driven design, of which this ticket is one piece.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR Task profile (and any companion BSeR Task-derived
profiles) to bind `Task.input.type` and `Task.output.type` to a
**required** (or at minimum **extensible**) BSeR-defined CodeSystem /
ValueSet whose codes enumerate the supported BSeR "requester package"
inputs and "response package" outputs (e.g., one code per supported
referral package type). Document, on the BSeR Task profile page, that
`Task.input` SHALL point at the requester-package content and
`Task.output` SHALL point at the corresponding response-package content,
with the bound code identifying which package kind the referenced
resource represents.

#### Justification

This is the most direct response to what the ticket asks for and
addresses the interoperability concern head-on: senders and receivers
can only agree on payload semantics if they share an identifier
vocabulary for what each `Task.input`/`Task.output` slice carries.
Defining the codes at the BSeR layer keeps governance with the IG that
actually knows the package taxonomy, and aligns BSeR with the broader
FHIR pattern of profiling the open `Task.input.type` / `Task.output.type`
bindings into something deterministic per use case.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the new core FHIR Task terminology landed via
[FHIR-50258](https://jira.hl7.org/browse/FHIR-50258) — namely the new
`task-inputoutput-parameter-type` code system (and, where applicable,
the `data-request` value in `task-code`) — and bind
`Task.input.type` / `Task.output.type` in the BSeR Task profile to a
ValueSet drawn from that core system, **extending it via UTG** with any
BSeR-specific package codes that aren't already covered. Where an
existing core/Da Vinci code already names a payload BSeR exchanges,
reuse it instead of minting a BSeR-local synonym.

#### Justification

FHIR-50258 has been Applied as Persuasive in core FHIR with the
explicit goal of giving Task a sharable, non-Da-Vinci-specific
terminology for input/output parameters. Rebasing BSeR onto that core
vocabulary avoids a second, parallel, BSeR-only code system and is more
consistent with HL7's preference (called out in FHIR-50258 itself) that
Task parameter codes be managed in UTG rather than per-IG. Trade-off
vs. Disposition A: BSeR loses some short-term autonomy over the code
list and inherits a dependency on core/UTG release cadence, but the
long-term interoperability win — one Task parameter vocabulary across
BSeR, Da Vinci, and other IGs — is significant, and the Justification
on FHIR-50258 explicitly anticipates this kind of downstream adoption.

---

### Disposition C: Decline

#### Proposal

Decline as out-of-scope-here / superseded: close FHIR-42519 as
*Persuasive with Modification* pointing implementers at FHIR-50258 and
the new core `task-inputoutput-parameter-type` system, and take **no
BSeR-specific change** beyond a documentation note in the BSeR Task
guidance that implementers should use the new core codes for
`Task.input.type` / `Task.output.type`.

#### Justification

The substantive standards work the ticket is asking for has effectively
already happened at the core level via FHIR-50258. If BSeR adds nothing
beyond what core now provides, the cheapest defensible action is to
acknowledge that and avoid duplicating terminology governance inside
BSeR. The risk this carries is that BSeR's actual referral package
taxonomy may not map cleanly onto the core code set, in which case
declining without a profile-level binding leaves implementers with the
same interoperability gap the ticket originally raised.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right balance. The ticket's core complaint —
"there are no standardized codes, so senders and receivers can't
interoperate" — is genuine, and Disposition C alone does not solve it
for BSeR implementers because a documentation note is not a binding.
Disposition A solves it but at the cost of inventing a parallel BSeR-only
vocabulary at exactly the moment FHIR Core has defined a sharable one
([FHIR-50258](https://jira.hl7.org/browse/FHIR-50258), Applied as
Persuasive). Disposition B keeps the BSeR Task profile concrete and
implementable (a real binding on `Task.input.type` /
`Task.output.type`), but anchors it on the new core
`task-inputoutput-parameter-type` system and only extends via UTG for
genuinely BSeR-specific package types. This minimises long-term
maintenance for Public Health, maximises cross-IG interoperability with
Da Vinci and other Task consumers, and discharges Lloyd McKenzie's
original concern in the way the broader FHIR community has now
endorsed.
