## Summary

Floyd Eisenberg observes that the BSeR (US Bidirectional Services
eReferrals) IG lists `ServiceRequest.doNotPerform` (boolean, 0..1) in
the Key Elements Table of its ServiceRequest profile. He argues that,
in a referral context, a request to **do** something
(`doNotPerform = false`) is semantically very different from a request
**not** to do something that has been planned
(`doNotPerform = true`), and that the IG should therefore split
ServiceRequest into two disjoint profiles — one for the positive
referral and one for an explicit "service not requested" referral —
mirroring the pattern used by the QI-Core IG (which defines distinct
profiles for the affirmative and negated cases). The only structured
cross-reference for this ticket is the FHIR element
`ServiceRequest.doNotPerform` itself; no related Jira tickets, Zulip
threads, or GitHub items were surfaced by the data sources.

## Details

**Description:**

> In your ServiceRequest, you include `.doNotPerform` in the Key
> Elements Table as a boolean (cardinality 0..1). However, for the
> purpose of the IG, the referral may be either a request to do
> something (which would make `doNotPerform = false`) or a specific
> request not to do something that has been planned (i.e.,
> `doNotPerform = true`). That would suggest that the IG needs a
> disjoint set of profiles for ServiceRequest (the positive) and for
> ServiceNotRequested — see QI-Core IG currently in ballot for a
> reference: https://build.fhir.org/ig/HL7/fhir-qi-core/profiles.html.

**Resolution Description (Not Persuasive, 2025-04-03):**

> The `doNotPerform` data element is not a data element we have
> highlighted for our use case. It is showing up in the Key Elements
> tab because it is a modifier element. Note that it also shows up in
> the Key Elements tab of the base US Core ServiceRequest profile,
> that our profile depends on. Implementers of **any** profile have to
> pay attention to that element.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Split the BSeR `ServiceRequest`-derived profile into two disjoint
profiles in the BSeR IG:

1. **`BSeR-ServiceRequest`** (the affirmative referral): constrain
   `ServiceRequest.doNotPerform` to `0..1` with the value fixed (or
   pattern) to `false`, or constrain it to `0..0` so the element
   cannot appear true in this profile.
2. **`BSeR-ServiceNotRequested`** (the negated referral): a sibling
   profile that fixes `ServiceRequest.doNotPerform = true` and is
   `1..1`, mirroring the QI-Core "negation" pattern at
   <https://build.fhir.org/ig/HL7/fhir-qi-core/profiles.html>.

Update each BSeR use-case-specific profile (e.g., the diabetes
prevention, hypertension, obesity, and tobacco cessation referral
profiles) to derive from one of the two new base profiles, and update
the IG narrative / Key Elements documentation to make the affirmative
vs. negated distinction explicit. Also update example resources and
CapabilityStatements to reflect the two profile URLs.

#### Justification

Disjoint profiles communicate the semantic difference between
"please do X" and "explicitly do not do X" much more clearly than a
single profile that exposes a modifier flag, and the QI-Core IG cited
in the ticket sets a precedent for this pattern in another HL7
US-realm IG. For implementers building referral workflows, having a
distinct profile URL is easier to validate against and easier to route
than a runtime boolean check on a modifier element.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a single BSeR ServiceRequest profile but add explicit IG
guidance and (optionally) a constraint that addresses the negation
case without forking profiles:

- Add narrative in the BSeR ServiceRequest profile page (and the
  IG home page) explaining how `ServiceRequest.doNotPerform` is to
  be used in BSeR referrals, with worked positive and negative
  examples.
- Tighten cardinality of `doNotPerform` in the BSeR profile to
  `1..1` to force senders to make the affirmative/negated stance
  explicit (rather than leaving it absent and ambiguous).
- Optionally add an `invariant` requiring `intent` and other
  status-bearing elements to be consistent with `doNotPerform =
  true` (e.g., reasonCode required when `doNotPerform = true`).
- Document that the element is inherited as a modifier from the base
  resource and US Core ServiceRequest, so its presence in the Key
  Elements Table is intentional and not unique to BSeR.

#### Justification

This addresses Floyd's underlying concern — that consumers of a BSeR
ServiceRequest must not silently miss a "do not perform" signal —
without doubling the number of profiles or requiring a breaking
change to existing BSeR consumers, conformance statements, and
implementations already in production. It also avoids divergence
from US Core ServiceRequest, which BSeR derives from and which itself
exposes `doNotPerform` as a modifier element rather than splitting
into two profiles. Explicit cardinality + invariants give validators
enough to catch unsafe omissions, which is where the modifier-element
risk really lives.

---

### Disposition C: Decline

#### Proposal

Mark the request **Not Persuasive** (as the work group has already
done on 2025-04-03) and make no profile or narrative change to the
BSeR IG. The rationale, per the existing resolution, is that
`doNotPerform` appears in the Key Elements Table only because it is
a FHIR R4 modifier element, the same way it appears in the
**US Core ServiceRequest** profile that BSeR derives from.
Implementers of *any* ServiceRequest profile are already obligated
by the FHIR specification to inspect modifier elements, so calling
it out in BSeR is correct, not a defect.

#### Justification

`isModifier = true` elements are surfaced in the Key Elements view
by design across the FHIR tooling; their presence is informational,
not a statement that the IG is endorsing one value over another.
Splitting the profile in two would (a) diverge from the US Core
parent profile, (b) require every downstream BSeR-specific use-case
profile and example to be re-pointed, and (c) introduce a churn cost
on BSeR consumers without a documented use case in the ticket where
the current single profile actually fails. The QI-Core analogy is
weak: QI-Core's negation profiles exist to support measure-logic
patterns that look up "evidence of negation," which is not BSeR's
referral scope. The work group has already accepted this rationale.

---

### Recommendation

**Recommended disposition:** C

The work group's existing **Not Persuasive** resolution is the
right call and should stand. `ServiceRequest.doNotPerform` appears
in BSeR's Key Elements Table for the same structural reason it
appears in US Core ServiceRequest — it is a FHIR modifier element,
and the IG's tooling correctly highlights it. The ticket does not
identify a concrete BSeR exchange that fails under the current
single-profile design; it argues by analogy to QI-Core, whose
negation profiles serve a different (measure-logic) purpose.
Disposition A would force a breaking restructure of every BSeR
use-case profile and diverge from the US Core base, while
Disposition B's tightened cardinality and narrative could be a
useful follow-up if real implementer confusion is reported, but
should be raised as a separate, scoped change request rather than
folded into this ticket. Closing as Not Persuasive — with the
existing resolution text retained — is defensible and consistent
with how modifier elements are handled elsewhere in the US-realm
ecosystem.
