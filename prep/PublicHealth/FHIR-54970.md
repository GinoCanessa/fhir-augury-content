## Summary

The ticket asks the US Central Cancer Registry Reporting Content IG (CCRR)
authors to consider basing some of the IG's profiles on profiles from the
**US Public Health Profiles Library (US PH Library, `hl7.fhir.us.ph-library`)**
rather than profiling base FHIR resources (or US Core) directly. Sarah
Gaunt — a long-standing CDC / public-health IG author and the editor of
the US PH Library — filed the request at "Highest" priority during
triage. No specific CCRR profile is named in the description; the request
is a general call to align the IG with the cross-domain PH library so
that public-health concepts (patient demographics, conditions,
observations, encounters, etc.) reuse common building blocks rather than
re-deriving them. There are no cross-referenced Jira tickets, GitHub
items, or Zulip threads specifically tied to FHIR-54970, so the
workgroup needs to decide which CCRR profiles are good candidates for
re-basing on PH-Library equivalents.

## Proposed Dispositions

> Note: Because no `repo-analysis` briefings are available, the
> dispositions below name the candidate IG repositories
> (`HL7/fhir-us-central-cancer-registry-reporting-ig` for the CCRR IG
> and `HL7/fhir-us-ph-library` for the US PH Library) and the
> conventional authoring root for HL7 IGs (`input/fsh/profiles/` for the
> StructureDefinitions, `sushi-config.yaml` for the IG manifest, and
> `input/pagecontent/` for narrative). They should be confirmed against
> the actual repos before implementation.

### Disposition A: Accept As Requested

#### Proposal

Perform a profile-by-profile audit of the CCRR IG and re-base every CCRR
profile whose intent overlaps a US PH Library profile so that
`StructureDefinition.baseDefinition` points at the PH-Library profile
instead of the base FHIR resource (or US Core profile). Concretely:

1. In `HL7/fhir-us-central-cancer-registry-reporting-ig`, enumerate all
   profiles under `input/fsh/profiles/` (and any JSON
   `StructureDefinition`s under `input/resources/`) and map each one to
   a candidate US PH Library profile from
   `http://hl7.org/fhir/us/ph-library/STU2`
   (e.g., `Patient` → `us-ph-patient`, `Condition` → `us-ph-condition`,
   `Observation` → `us-ph-observation` family, `Encounter` →
   `us-ph-encounter`, `Practitioner` / `Organization` / `Location`
   variants, etc.).
2. Update each candidate FSH profile's `Parent:` to the PH-Library
   profile and reconcile differential constraints (cardinality, MS
   flags, slicing on `category` / `code`, bound value sets) so the CCRR
   profile no longer re-states what the PH-Library parent already
   guarantees.
3. Add `hl7.fhir.us.ph-library` (matching the latest STU2 edition) to
   `dependencies:` in `sushi-config.yaml`, drop redundant direct US Core
   dependencies that the PH Library already pulls in, and update IG
   narrative in `input/pagecontent/` (Dependencies / Profiles pages) to
   document the new derivation chain.
4. Re-run IG Publisher; resolve any QA issues caused by tightened
   parents (e.g., terminology bindings or slicing patterns inherited
   from PH Library that conflict with current CCRR content) and submit
   for re-ballot.

#### Justification

The US PH Library exists precisely to provide reusable, public-health-
oriented base profiles so individual PH IGs do not each re-derive the
same constraints from US Core / base FHIR. Aligning CCRR with the PH
Library reduces drift between PH IGs (eCR, Birth & Fetal Death,
Hypertension, Contact Tracing, etc.), simplifies implementer guidance
("if you already conform to US PH X, you are most of the way to CCRR
X"), and matches the direction the reporter (Sarah Gaunt, the PH
Library editor) and the broader Public Health WG have been moving in
since the PH Library Release 1 (2023) and STU2 (2026-02) publications
referenced in Zulip's Announcements stream.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than committing to re-base profiles in the current ballot cycle,
add a **dependency on `hl7.fhir.us.ph-library` STU2** to the CCRR IG
and produce a written **alignment plan** that classifies each CCRR
profile as one of:

- **Re-base now** — PH-Library parent is a clean superset; switch
  `Parent:` in the next minor release.
- **Re-base after reconciliation** — PH-Library parent is close, but
  CCRR currently constrains an element more loosely or differently;
  capture the delta as a Jira sub-task and defer until reconciled.
- **Keep current parent** — no suitable PH-Library equivalent (e.g.,
  cancer-registry-specific resources with no PH-wide analogue).

Land the PH-Library dependency and the alignment-plan page in
`input/pagecontent/dependencies.md` (or a new `ph-library-alignment.md`)
in this ballot, and stage the actual `Parent:` switches across one or
two subsequent ballots, one resource family at a time.

#### Justification

The ticket says "consider using … as the base for **some** profiles,"
not all. An across-the-board re-parent in a single ballot risks
breaking implementers who already conform to the current CCRR STU2
ballot (`hl7.fhir.us.central-cancer-registry-reporting#2.0.0-ballot`
published 2025-12-15) and risks QA churn from PH-Library constraints
that conflict with cancer-registry-specific tightenings. A staged
approach delivers the architectural benefit the reporter is asking for
(consistent PH base profiles) while letting the WG sequence the actual
re-bases by risk and by available reviewer bandwidth, and gives
implementers a published roadmap rather than a surprise breaking
change.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-54970 as **Not Persuasive** for this ballot cycle on the
grounds that (a) the request is a general direction rather than a
concrete change against a specific profile, (b) re-parenting any CCRR
profile is a breaking change for current implementers and requires a
specific, justified delta per profile, and (c) the current CCRR ballot
already targets US Core (and base FHIR) parents that are well
understood by NAACCR / cancer-registry implementers. Invite the
reporter to refile per-profile tickets that name the specific CCRR
profile, the proposed PH-Library parent, and the expected differential
changes.

#### Justification

A "consider using X" request without a named profile or concrete
differential is not actionable as a single change at the ballot level;
implementing it as written would either be a no-op (just "we
considered") or a sweeping breaking change with no per-profile
justification on file. Declining with a clear request for
per-profile follow-up tickets keeps the IG's change history auditable
and gives each re-parent its own reconciliation discussion.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best fit. The reporter (Sarah Gaunt) is the editor
of the US PH Library and clearly intends a strategic alignment, and the
WG has consistently moved PH IGs onto the PH Library since its
Release 1; outright declining (C) would ignore that direction and waste
the reporter's signal. But the ticket as filed names no specific
profile and the CCRR IG just published a 2.0.0 ballot in
December 2025, so an immediate full re-base (A) would create avoidable
breakage and reviewer load. Adding the PH-Library dependency now,
publishing an alignment plan that classifies each CCRR profile, and
staging the actual `Parent:` switches across the next one to two
ballots gives the WG the architectural win the ticket is asking for
while protecting current implementers and giving each re-parent its
own reviewable change.
