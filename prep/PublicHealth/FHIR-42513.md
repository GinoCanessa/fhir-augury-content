## Summary

This ticket, originally raised against the US Bidirectional Services
eReferrals (BSeR) FHIR IG by Lloyd McKenzie, asserts that the BSeR Consent
profile is too thin to be useful for interoperability. The reporter argues
that simply marking a `Consent.provision.type = permit` (with `Consent.category`
as the only other discriminator) does not actually communicate **what** is
being permitted â€” what data may be disclosed, to whom, for what purpose, or
under what conditions. The ticket asks the work group to clarify the
interoperability target for the profile: what a recipient of a BSeR Consent
instance is supposed to be able to *do* with it, and how that affects their
disclosure decisions. It is a design-level critique of the profile rather
than a request for a single, specific element change. The Augury index found
no cross-referenced Jira tickets, Zulip threads, Confluence pages, or GitHub
items linked to this ticket, so the request stands on its own at this point.

### Disposition A: Accept As Requested

#### Proposal

Substantively expand the BSeR Consent profile so that a recipient can
mechanically determine what a given Consent instance permits, without
needing out-of-band context. Concretely:

- Define and document the **interoperability intent** of the profile in
  its narrative: what decision a BSeR actor is expected to make on
  receipt of the Consent (e.g., "may share referral packet containing
  X with receiving provider Y for purpose Z").
- Constrain `Consent.scope` to a fixed BSeR-appropriate value (e.g.,
  `patient-privacy`) and bind `Consent.category` to a required
  ValueSet whose codes carry actual semantic weight for BSeR (not
  just "consent").
- Require `Consent.provision` and tighten its sub-elements:
  - `provision.type` (permit/deny) with a required usage rule.
  - `provision.purpose` bound to a required ValueSet (e.g., `TREAT`,
    `HPAYMT`, `HOPERAT`, plus referral-specific purposes).
  - `provision.actor` (or `actor.reference` / `actor.role`) made
    must-support so the recipient knows *who* is authorized.
  - `provision.action` bound to a required ValueSet covering at least
    `disclose` / `access` so the permitted action is explicit.
  - `provision.class` and/or `provision.code` bound (required or
    extensible) to identify the resource types / data classes covered
    (e.g., the BSeR referral bundle, ServiceRequest, observations).
  - `provision.period` made must-support so disclosures have a known
    validity window.
- Add invariants that reject instances which assert `permit` without
  populating at least one of `actor`, `action`, and `class`/`code`.

#### Justification

This is the most direct response to the reporter's complaint: the
current profile under-specifies the contract between sender and
receiver, and FHIR's Consent resource already exposes the elements
needed to express *who*, *what*, *for what purpose*, and *for how long*
a disclosure is permitted. Tightening these elements aligns BSeR with
how Consent is profiled in adjacent ecosystems (e.g., consent profiles
in Da Vinci and SDOH-CC work), keeps the profile inside the base
Consent model rather than inventing extensions, and lets conforming
implementations reason about disclosure decisions deterministically â€”
which is exactly the interoperability target the reporter says is
missing.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than try to encode all permission semantics inside the existing
Consent profile, redefine BSeR's use of Consent as a **thin, signed
attestation** and move the substantive disclosure rules into a small,
named set of pre-coordinated **policy artifacts** referenced from the
Consent:

- Keep the BSeR Consent profile minimal but require
  `Consent.policy.uri` (or `Consent.policyRule`) to point at one of a
  short, registry-style list of named BSeR consent policies
  (e.g., "BSeR-Referral-Standard-Disclosure-v1",
  "BSeR-BehavioralHealth-Restricted-v1").
- Publish those policies as first-class IG artifacts (HTML pages plus
  machine-readable identifiers) that spell out, in one place, the
  actors, purposes, data classes, and duration each policy authorizes.
- Require `Consent.patient`, `Consent.dateTime`, and (where used)
  `Consent.sourceAttachment` / `sourceReference` so the attestation
  itself is verifiable, but leave `provision` optional / informative â€”
  the policy URI is the source of truth.
- Provide implementer guidance on how a recipient resolves the policy
  URI to a known rule set and what to do if the URI is unrecognized
  (reject, fall back, escalate).

#### Justification

The reporter's underlying problem is that the *meaning* of a BSeR
consent is not knowable from the instance. Disposition A fixes that by
loading the meaning into the instance; Disposition B fixes it by
naming the meaning out-of-line in a small, governed catalog. This
mirrors the way real-world consent regimes actually operate (a signed
form references a named policy rather than re-stating it), reduces
profile churn when policy text needs to evolve, and avoids the trap of
trying to model every disclosure rule in `Consent.provision`
discriminators â€” which historically has produced consent profiles that
are both verbose and still ambiguous. Trade-off versus A: recipients
must maintain a policy-URI lookup, and pure-data validators cannot tell
a "good" instance from a "bad" one without consulting the policy
catalog.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as **Not Persuasive** for this BSeR ballot cycle on
the grounds that (a) the Consent profile in BSeR is intended only to
record that *a* consent was captured for the referral exchange, and
(b) the substantive disclosure semantics live in the underlying
jurisdictional / organizational policy rather than in the FHIR
instance â€” which is consistent with how many real-world referral
workflows operate. Document this scoping decision in the BSeR Consent
profile's narrative so future readers do not re-raise the same
critique, and point them at the base FHIR Consent guidance for richer
profiling.

#### Justification

Defensible only if the work group can affirmatively state that BSeR
does not need machine-actionable disclosure rules in the Consent
instance because every BSeR participant is already operating under a
shared, known policy framework outside the exchange. In that case the
profile is a *receipt*, not a *rule*, and adding `provision`
constraints would impose burden without changing real implementer
behaviour. Risk: the reporter explicitly asked "what is the
interoperability you're trying to achieve here?", and "none beyond
acknowledgement" is a coherent answer only if it is written down â€” so
declining without updating the narrative would simply re-surface the
same ticket later.

---

### Recommendation

**Recommended disposition:** B

The reporter's critique is correct as stated â€” the current BSeR
Consent profile does not let a recipient act on the consent â€” and
straight acceptance (A) is tempting because it directly answers that
critique. However, encoding the full disclosure rule set inside
`Consent.provision` for a referral IG tends to produce profiles that
are both heavy to author and still under-specified at the edges
(actors expressed as roles vs. references, purpose codes that don't
align with payer/jurisdictional vocabularies, etc.), and Public Health
implementations in particular often need to evolve disclosure policy
faster than an IG ballot cycle allows. Disposition B keeps the FHIR
instance small and verifiable while moving the substantive semantics
into a short, named catalog of BSeR consent policies that the work
group controls â€” which gives recipients a deterministic
"what-am-I-allowed-to-do" answer (the original ask) without locking
the policy text into the profile. Disposition C should only be
revisited if the work group confirms BSeR was never intended to carry
machine-actionable consent semantics, in which case the profile's
narrative must be updated to say so explicitly.
