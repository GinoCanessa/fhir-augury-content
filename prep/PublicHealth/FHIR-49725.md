## Summary

The ticket requests removal of three extraneous codes from the VRDR
`vrdr-observations-cs` CodeSystem in the US Vital Records Death Reporting
(VRDR) FHIR IG. The reporter (Saul Kravitz) states that these codes —
`emergingissues`, `codedraceandethnicity`, and `inputraceandethnicity` —
along with their associated Observations have already been transitioned
to the Vital Records Common Library (VRCL) CodeSystem under similar
names, leaving the VRDR copies redundant and a source of duplication
between the two specifications.

No cross-referenced Jira tickets, Zulip threads, or GitHub items are
recorded against `FHIR-49725` in the augury index. A keyword search for
`VRDR codedraceandethnicity` surfaced only a 2022 Death-on-FHIR Zulip
message that pre-dates the VRCL migration and merely uses the code in an
example payload, so it does not provide additional context for the
disposition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the three concepts `emergingissues`, `codedraceandethnicity`, and
`inputraceandethnicity` from the VRDR `vrdr-observations-cs`
CodeSystem (canonical
`http://hl7.org/fhir/us/vrdr/CodeSystem/vrdr-observations-cs`) in the
next VRDR publication. As part of the same change:

- Remove the corresponding Observation profiles / examples that bind to
  these codes from the VRDR IG, since the ticket states the observations
  themselves were also moved to VRCL.
- Remove or update any ValueSets in VRDR that enumerate or expand to
  include these three codes (so the removal is reflected in any binding
  that previously surfaced them).
- Update VRDR narrative pages and IG guidance that reference these
  codes to point to the corresponding VRCL concepts.
- Bump the VRDR CodeSystem version and add a clear note in the IG
  changelog calling out the removal and the VRCL replacements.

#### Justification

The reporter — who is a long-standing VRDR/VRCL contributor — states
that the codes and their associated Observations have already been
moved to VRCL with similar names. Leaving duplicates in
`vrdr-observations-cs` causes:

- Two canonical homes for the same concept (VRDR vs VRCL), which
  encourages divergent bindings across implementers.
- Risk of NCHS pipelines and jurisdictional senders picking the wrong
  code system in produced/consumed Observations.
- Continued maintenance burden on a CodeSystem that the IG owners have
  explicitly decided should no longer host these concepts.

Removing them aligns the published artifact with the design decision
that has already been executed in VRCL and is consistent with FHIR's
preference for a single authoritative source for a given coded concept.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hard-removing the three codes from
`vrdr-observations-cs`, retain them in the CodeSystem but mark each one
as **deprecated** for at least one publication cycle, using the standard
mechanisms:

- Set `CodeSystem.concept.property` `status = deprecated` (or
  `retired`), and add a `deprecationDate` property.
- Add a `replacedBy` property on each concept pointing at the
  corresponding VRCL code (canonical URI + code), so tooling and
  validators can surface the migration automatically.
- Update the concept `definition` text to say the code has been moved
  to VRCL and should not be used in new instances.
- Update VRDR Observation profiles to bind to the VRCL replacements
  while keeping the deprecated VRDR codes resolvable for older
  instances.
- Schedule full removal for a subsequent VRDR release once jurisdictions
  have had time to migrate.

#### Justification

VRDR is implemented by NCHS and by U.S. state/territorial vital
records jurisdictions, many of whom run multi-year release cycles. A
silent removal (Disposition A) breaks `$validate` against historical
death-record bundles and any jurisdictional pipeline that still emits
the VRDR-namespaced codes. Deprecating with `replacedBy` preserves
historical record validity, gives implementers a machine-readable
migration target, and matches the deprecation pattern used elsewhere in
HL7 terminologies for code retirement. The trade-off versus
Disposition A is one extra release cycle of clutter in the CodeSystem
in exchange for a smoother migration for downstream implementers.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep `emergingissues`, `codedraceandethnicity`,
and `inputraceandethnicity` in `vrdr-observations-cs` as currently
published, on the grounds that VRDR is a published US Realm IG and
removing concepts from an existing CodeSystem is a breaking change to
any instance, profile, or jurisdictional pipeline currently using
those codes. Document the VRCL replacements only in narrative.

#### Justification

This option is defensible only if the workgroup concludes that the
implementer impact of removal outweighs the duplication harm — for
example, if survey of jurisdictions shows substantial production usage
of the VRDR-namespaced codes. It would not be the recommended path
because (a) the reporter, who is closely involved with both VRDR and
VRCL, has explicitly asked for removal; (b) maintaining knowingly
duplicated codes contradicts the stated VRDR/VRCL division of
responsibility; and (c) it leaves the IG in an inconsistent state where
the Observations have moved but their codes have not.

---

### Recommendation

**Recommended disposition:** B

Disposition B (deprecate with `replacedBy`, then remove in a later
release) is the right balance for a published US Realm IG with
production implementers at NCHS and state vital-records jurisdictions.
It honours the reporter's underlying intent — these codes should no
longer be authored in VRDR namespace — while protecting validation of
existing death-record instances and giving jurisdictions a
machine-readable migration target to VRCL. Disposition A achieves the
end state more quickly but at the cost of breaking `$validate` for any
historical or in-flight bundle that still uses the VRDR codes;
Disposition C preserves stability but locks in known duplication
between VRDR and VRCL, which is exactly the situation the ticket is
trying to resolve. Workgroup discussion should focus on (1) confirming
the VRCL replacement codes and canonical, and (2) agreeing the release
in which the deprecated codes will be fully removed.
