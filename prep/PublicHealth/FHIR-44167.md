## Summary

The reporter requests that the "Yes", "No", and "&lt;x&gt;" nullFlavor-style
value sets be added to Terminology.HL7.Org (THO) so they can be reused
across many FHIR specifications and use cases. The ticket originates from
the Vital Records Common Profiles Library (FHIR), where these "yes / no /
unknown-style" answer sets are a recurring need, but the ask is explicitly
to host them centrally in THO rather than in any single IG. There are no
cross-referenced Jira tickets, Zulip threads, or GitHub artifacts in the
Augury index, so the ticket stands on its own; the only history is on the
ticket itself, where an earlier disposition of "Considered for Future Use"
(2024-03-07, vote 21-0-0, with the note "Will start process to add to
THO") was reverted and the ticket re-opened post STU2 publication on
2024-10-11.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Author and publish three new ValueSet resources in
Terminology.HL7.Org (THO) that combine an affirmative/negative answer
with a nullFlavor "exception" code:

1. **Yes / No** — composed of `v2-0136` (or equivalent SNOMED CT codes)
   "Y" and "N".
2. **Yes / No / Unknown** — adds `v3-NullFlavor#UNK` to the above.
3. **Yes / No / &lt;x&gt;** — a parameterized pattern (one ValueSet per
   commonly-needed nullFlavor: `UNK`, `NI`, `OTH`, `ASKU`, `NASK`),
   each composing the Yes/No answers with a single nullFlavor code.

Publish them under the THO `ValueSet/` authoring root using stable
canonical URLs in the `http://terminology.hl7.org/ValueSet/...`
namespace, with `experimental = false`, an explicit steward, and a
description that documents the intended pattern (binary answer plus a
controlled exception code). Update the Vital Records Common Profiles
Library to bind to the new THO canonicals where it currently inlines
equivalent value sets, and announce the new resources to other public
health and administrative IGs that have re-implemented the same
pattern.

#### Justification

The need for "yes / no plus unknown / not asked / other" answer sets
recurs across vital records, public health case reporting, social
determinants questionnaires, and patient-reported outcomes. Hosting
canonical versions in THO avoids the proliferation of near-duplicate
ValueSets in individual IGs and aligns with THO's stated purpose as the
shared terminology home for HL7. The 2024-03-07 Vital Records vote was
21-0-0 in favor of starting the THO process, so there is an existing
work group mandate; reverting the disposition was procedural (post-STU2
publication) rather than substantive.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting net-new "Yes/No/&lt;x&gt;" ValueSets, publish in THO
a single small **CodeSystem** (or reuse `v2-0136`) plus a documented
**authoring pattern / template** showing IG authors how to compose a
Yes/No answer set with any nullFlavor code they need. Provide one
canonical reference ValueSet — `Yes/No/Unknown` — as the worked example,
and leave the long tail of `Yes/No/<other-nullFlavor>` combinations to
the IGs that actually need them, each binding by composition rather than
by importing a pre-baked ValueSet.

#### Justification

The set of useful "Yes/No plus exception" combinations is open-ended
(`UNK`, `NI`, `OTH`, `ASKU`, `NASK`, `MSK`, …), and pre-publishing every
combination in THO risks creating a sprawl of nearly-identical artifacts
that implementers must choose between. A documented composition pattern
plus one canonical example keeps the THO surface area small while still
removing the duplication burden the reporter is concerned about, and it
sidesteps governance debates about which exact nullFlavor codes deserve
their own pre-built ValueSet.

---

### Disposition C: Decline

#### Proposal

Decline the request as written and instead instruct authors to bind
locally in their IGs by composing existing THO content
(`v2-0136` for Yes/No and `v3-NullFlavor` for the exception code). Note
that the Vital Records Common Profiles Library may continue to define
the value sets it needs in its own ValueSet authoring area, as it does
today.

#### Justification

Both inputs already exist in THO as code systems, and FHIR ValueSet
composition is the intended mechanism for assembling answer lists from
existing code systems. Adding pre-composed ValueSets to THO does not
expose any new codes; it only saves authors a few lines of ValueSet
JSON, at the cost of permanent THO maintenance, version pinning, and
the governance overhead of choosing which combinations to bless. With
no cross-referenced tickets or Zulip threads showing broader demand
beyond Vital Records, the request may not clear the bar for THO
inclusion.

---

### Recommendation

**Recommended disposition:** A

The 2024-03-07 Vital Records vote (21-0-0, "Will start process to add
to THO") already represents a clear work-group consensus to publish
these value sets in THO; the ticket was only re-opened because the
process was deferred past STU2 publication, not because the substance
was rejected. The artifacts are small, well-bounded, and reuse code
systems that are already in THO, so the maintenance burden is low.
Disposition B is the strongest fallback if the Terminology Authority
pushes back on hosting multiple `Yes/No/<x>` permutations — in that
case, publish `Yes/No/Unknown` as the canonical example and document
the composition pattern. Disposition C should be reserved for the case
where THO governance explicitly declines to host pre-composed answer
sets at all.
