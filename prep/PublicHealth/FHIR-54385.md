## Summary

The ticket asks the US Central Cancer Registry Reporting (CCRR) IG authors
to add a "trigger code" indicator as a discrete data element on the
Central Cancer Registry Reporting Composition profile (section 7.4.1),
mirroring the pattern already established by the eCR / Case Reporting IG.
In eCR, the composition's entry slices each carry an extension
(`eicr-trigger-code-flag-extension`) on the `Composition.section.entry`
reference, which flags the referenced clinical resource as the one whose
matching trigger code caused the report to be generated. The submitter
(entered on behalf of a NAACCR community member) wants the same explicit,
machine-readable flag in CCRR so that consumers can identify which
referenced resource(s) satisfied a reporting trigger rather than having to
re-derive that match downstream. There are no cross-referenced Jira
tickets, Zulip threads, or GitHub items currently linked to this ticket
in the index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the eCR pattern in the CCRR IG's Central Cancer Registry Reporting
Composition profile (section 7.4.1):

1. Author a new StructureDefinition for an extension named (suggested)
   `ccrr-trigger-code-flag-extension`, modeled directly on
   `eicr-trigger-code-flag-extension`. Context: `Composition.section.entry`.
   Type: `boolean` (with the convention that *true* = "this referenced
   resource matched a reporting trigger"). Cardinality 0..1 per entry.
2. In the Central Cancer Registry Reporting Composition profile, on each
   defined `section.entry` slice, add an `extension` slice (sliced on
   `url`) referencing the new extension StructureDefinition with
   cardinality 0..1.
3. Update the IG narrative for section 7.4.1 to document when senders
   SHOULD set the flag (i.e., on the entry corresponding to the
   resource(s) whose data satisfied the cancer reporting trigger), with a
   worked example mirroring the eCR example page.

#### Justification

- Consistency with a sibling Public Health IG (eCR / Case Reporting) is
  itself a strong design principle: implementers and toolchains that
  already handle the eCR flag will need only minimal adaptation.
- The flag is additive (0..1) and therefore non-breaking for existing
  CCRR senders and receivers.
- It addresses a concrete downstream need: cancer registries currently
  must re-derive which referenced resource triggered the report by
  matching codes against trigger value sets, which is brittle and
  duplicates work the sender has already done.
- The request comes from a NAACCR community member, i.e., a primary
  consumer of CCRR submissions, which is the right voice to ask for
  receiver-facing affordances.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting a CCRR-specific extension, **reuse the existing
`eicr-trigger-code-flag-extension`** from the Case Reporting IG by:

1. Adding a normative dependency from the CCRR IG on the Case Reporting
   IG (or, if that dependency footprint is undesirable, lifting the
   extension into a shared "Public Health common" location — e.g., an
   extensions IG owned by the Public Health WG — and depending on that
   instead).
2. Slicing `Composition.section.entry.extension` in the CCRR Composition
   profile against the existing canonical URL.
3. Optionally narrowing the binding/usage rules in CCRR (e.g., requiring
   the flag whenever the entry references a Condition that was the
   matched trigger) where CCRR's reporting model is more deterministic
   than eCR's.

#### Justification

- Avoids two near-identical extensions in the Public Health space, which
  is exactly the kind of fragmentation that bites implementers building
  cross-program (eCR + cancer + …) reporting pipelines.
- Aligns with the FHIR principle of preferring an existing, in-use
  artifact over a parallel new one when the semantics match.
- Trade-off vs. Disposition A: introduces an IG dependency (Case
  Reporting) or requires WG coordination to relocate the extension to a
  neutral home, which is more process work up front but pays off in
  long-term maintainability. If the WG is unwilling to take a hard
  dependency on the eCR IG version cycle, the "shared extensions IG"
  variant is the right compromise.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped, with the rationale that the
trigger-code provenance can already be conveyed without a new flag:
either (a) by including the trigger `Observation`/`Condition` itself in
the bundle and letting receivers match `Condition.code` (or
`Observation.code`) against the published reportable-condition trigger
value sets, or (b) by using `Provenance` resources to record which
resource(s) were the basis for generating the report. If declined, close
as "Persuasive with mod" suggesting better narrative guidance instead of
a structural change.

#### Justification

- A defensible argument for declining is that the matching can be
  recomputed deterministically from the trigger value sets the IG
  already publishes, so the flag is convenience rather than capability.
- However, this rationale is weak in practice: trigger value sets evolve,
  reporters and receivers may be on different versions, and the current
  ticket is a direct receiver-side ask. Decline should only be chosen if
  the WG decides the cost of a new extension (and possible eCR
  divergence) outweighs the implementer convenience.

---

### Recommendation

**Recommended disposition:** B

The eCR trigger-code-flag extension already exists, is in active use in a
sibling Public Health IG, and solves exactly the stated problem. Minting
a near-duplicate CCRR-specific extension (Disposition A) would deliver
the requested capability but at the cost of long-term Public Health
fragmentation — receivers building unified reporting infrastructure
would have to special-case each program. Declining (Disposition C)
ignores a legitimate receiver-side need from NAACCR. Disposition B —
reuse the existing extension, either by depending on the Case Reporting
IG or by promoting the extension to a shared Public Health extensions
home — gives the submitter what they asked for, preserves cross-IG
consistency, and is non-breaking. The WG should pick between "depend on
eCR" and "promote to a shared home" based on its appetite for the eCR
release cadence; either path is preferable to a parallel CCRR-only
extension.
