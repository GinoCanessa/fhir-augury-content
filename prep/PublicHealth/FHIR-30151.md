## Summary

The reporter (Floyd Eisenberg) flags the eCR (Electronic Case Reporting)
FHIR IG's apparent inclusion of the Oregon REAL D (Race, Ethnicity, Language,
and Disability) data capture approach and asks the Public Health work group
to clarify why a state-specific (Oregon) data set is being introduced into a
US Realm specification. The request, raised against the eCR IG, is for a
mechanism — most likely an extension comparable to the US Core Race and
Ethnicity extensions — that would allow the additional REALD-style race and
ethnicity granularity to be captured if it is in fact intended for US-wide
use, or otherwise to remove/explain its presence in the US Realm IG.

The cross-reference search returned no related Jira tickets, Zulip threads,
or GitHub items for FHIR-30151, so the analysis below relies on the ticket's
own description, comments, and resolution narrative.

### Disposition A: Accept As Requested

#### Proposal

Add a REALD-aligned race/ethnicity capture mechanism to the US Public
Health Profiles Library (USPHPL) — and, by extension, expose it to eCR
via the US Public Health Patient profile that was migrated from eCR into
USPHPL. Concretely, define two USPHPL extensions modelled on the US Core
patterns (`us-core-race`, `us-core-ethnicity`) but bound to Oregon REALD
value sets (or to a published HL7-curated superset that contains the
REALD codes), with the same nested `ombCategory` / `detailed` / `text`
sub-extension shape. Slice `Patient.extension` on the USPHPL patient
profile to allow these REALD extensions alongside the existing US Core
race/ethnicity extensions, with cardinality `0..*` for `detailed` and
`0..1` for the top-level extension. Document the use case as
"jurisdictional public-health reporting where REALD data are mandated."

#### Justification

This is the literal ask in the ticket and aligns with the migration path
described in the resolution narrative: the US Public Health Patient now
lives in USPHPL, so REALD-style granularity belongs there rather than in
eCR. Modelling on the US Core race/ethnicity extension shape preserves
implementer familiarity, keeps the OMB top-level categories interoperable
with existing US Core consumers, and contains the REALD-specific
granularity in a sub-extension/value-set binding so consumers that don't
understand REALD can still process the OMB rollups. It also addresses the
real-world driver named in the comments: REALD elements are mandatory in
Oregon for COVID-19 case reporting, which is precisely what eCR carries.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add REALD constructs directly to USPHPL or eCR. Instead, route
the REALD requirement through USCDI/USCDI+ first, and let any resulting
elements flow into US Core (and from there into USPHPL via reuse), as
the ticket's resolution narrative already anticipates. In the interim,
publish a small Oregon-specific (or Public Health–scoped) companion IG
that defines the REALD extensions and a thin Patient profile derived
from the US Public Health Patient, and reference it from eCR/USPHPL
non-normatively as "jurisdictions with REALD-style mandates may use
these extensions." Keep the core US Realm IGs unchanged with respect to
REALD until USCDI uptake is confirmed.

#### Justification

This honours the original objection (REALD is state-specific and does
not belong in a US-wide IG) without losing the implementation path for
Oregon. It also matches the resolution's own logic: USPHPL is being
aligned with USCDI+, and US Core takes from USCDI, so the durable way to
get REALD into US-wide artefacts is to get it into USCDI(+) first. A
separate jurisdictional IG keeps the governance burden on the
jurisdiction that needs it and avoids forcing every eCR/USPHPL
implementer to reason about an Oregon-specific value set. Trade-off vs.
A: slower for Oregon implementers in the short term, but cleaner
layering and lower risk of US-wide profiles carrying a state-specific
binding.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** (which matches the current
recorded resolution). State explicitly that (a) the eCR IG does not, and
should not, normatively require Oregon REALD elements; (b) the US Public
Health Patient has been moved into USPHPL, where alignment with USCDI+
is the governing mechanism for adding new race/ethnicity granularity;
and (c) jurisdictions that need REALD today can already do so via
locally-defined extensions on `Patient` without a change to the US Realm
IGs. No spec change is made.

#### Justification

The ticket itself argues that a state-specific data set has no clear
justification for inclusion in a US-wide specification, and the
resolution narrative concedes that point — the migration to USPHPL plus
USCDI+ alignment is the agreed forward path, and that path does not
require a REALD-specific change in eCR. FHIR's extension model already
permits jurisdictions to add granular race/ethnicity capture without
core profile changes, so declining does not block Oregon implementers;
it simply refuses to bake state-specific bindings into a US Realm IG.
This is also the lowest-risk option for downstream implementers and
preserves the existing US Core race/ethnicity extension contract.

---

### Recommendation

**Recommended disposition:** C

The ticket is effectively self-resolving: the reporter's core
objection — that a state-specific dataset doesn't belong in a US Realm
IG — is correct, and the work group's own resolution narrative
acknowledges that the durable path for any REALD content is through
USCDI(+) and then into US Core / USPHPL, not via a bespoke change to
eCR. The "Not Persuasive" resolution already on the ticket is the
honest outcome and should stand. If Oregon (or another jurisdiction)
wants REALD now, FHIR extensions on `Patient` are sufficient without a
US Realm spec change, and Disposition B remains available as a
follow-on if and when USCDI movement materialises. Adopting Disposition
A today would lock a state-specific binding into a US-wide profile
ahead of the USCDI alignment that USPHPL is explicitly being shaped
around, which the work group should avoid.
