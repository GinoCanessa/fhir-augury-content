## Summary

[FHIR-38266](https://jira.hl7.org/browse/FHIR-38266) is a placeholder
change request asking the US Public Health Profiles Library to migrate to
the new R5 "gender identity" extension once that extension is published in
the base FHIR R5 specification. The reporter (Rob McClure) anticipates
that the R5 extension — most likely
`http://hl7.org/fhir/StructureDefinition/individual-genderIdentity` (the
extension carved out of R5 `Patient.genderIdentity` for cross-version
use) — will become the canonical mechanism for representing gender
identity on `Patient`-like resources, and that the US Public Health
Profiles Library should adopt it when feasible. The ticket was
originally voted "Considered for Future Use" (Sarah Gaunt / John Loonsk,
18-0-0) on 2022-09-21, then re-opened by Lynn Laakso on 2023-08-17
"post STU publication" so it can be carried into the next publication
cycle. No cross-referenced Jira tickets, Zulip threads, or GitHub items
were surfaced by the augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the US Public Health Profiles Library so that any profile that
currently constrains or references gender identity on a `Patient`-like
resource (e.g., USPH Patient profile, and any derived
case-report/lab-report Patient profiles) adopts the R5 individual
gender identity extension —
`http://hl7.org/fhir/StructureDefinition/individual-genderIdentity` —
as a `mustSupport` extension on `Patient.extension`. Remove any
references to the prior R4 `patient-genderIdentity` extension or
narrative-only handling, and add an `Invariants` / guidance note that
implementers SHALL use the R5 extension for new instances. Bind the
extension's `value[x]` to whatever value set the R5 extension specifies
(or a USPH-preferred value set if Public Health WG has documented one).

#### Justification

The reporter explicitly asked for this swap, and the precondition (R5
publication of the extension) has been met. Adopting the official R5
cross-version extension keeps the USPHPL aligned with the broader FHIR
ecosystem, avoids divergence from US Core and other US realm IGs that
are migrating to the same extension, and gives implementers a single
canonical URL to bind to. Lynn Laakso's 2023-08-17 re-open note —
"re-open post STU publication" — signals that Public Health WG already
considers the next publication cycle the right window for this change.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than referencing the R5 extension URL directly, define a thin
USPH wrapper / forwarding extension (or simply add `mustSupport`
guidance) that says: "Use the gender identity representation of the
underlying base profile (US Core Patient) as published at the time of
the USPHPL release." In practice this means USPHPL stops carrying its
own gender identity constraint and instead defers to US Core
Patient (which itself carries the R5 individual gender identity
extension as of US Core 7+). USPHPL would add a profile-level note and,
where needed, a `mustSupport` flag on `Patient.extension:genderIdentity`
inherited from US Core.

#### Justification

USPHPL's Patient profiles are layered on top of US Core Patient, and US
Core has already taken on the responsibility of selecting and binding
the gender identity extension for the US realm. Hard-coding the R5
extension URL in USPHPL risks divergence from US Core if the US realm
ever picks a different binding (e.g., a USCDI-aligned value set).
Deferring to US Core keeps USPHPL thin, reduces maintenance, and avoids
double-publishing the same constraint. The trade-off versus
Disposition A is a small loss of explicitness in the USPHPL artifacts
themselves, mitigated by clear narrative pointing at US Core.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Mod" → No change to USPHPL itself,
on the grounds that gender identity representation in US realm Patient
profiles is owned by US Core (and downstream by USCDI), not by USPHPL.
Document in the resolution that any implementer wanting the R5 gender
identity extension should consume it via the US Core Patient base, and
that USPHPL will pick it up automatically the next time it re-bases on
a newer US Core version.

#### Justification

Defensible only on scope grounds: USPHPL profiles inherit Patient from
US Core, so adding an explicit gender identity constraint at the USPHPL
layer arguably duplicates work already done elsewhere and creates a
second place that has to be kept in sync with US Core / USCDI. The
weakness of this disposition is that the reporter (Rob McClure) and
the WG vote (18-0-0 for "future use", later re-opened) clearly
expected USPHPL itself to be touched, so declining without coordinating
with the reporter would likely be seen as ignoring the request rather
than re-routing it.

---

### Recommendation

**Recommended disposition:** B

Disposition B best balances the reporter's intent with the realities of
the US realm profile stack. The R5 individual gender identity extension
is genuinely the right target, but in the US realm that decision is
already being made one layer down, in US Core Patient. Having USPHPL
re-bind the same extension itself (Disposition A) risks the two IGs
drifting on value sets, `mustSupport` flags, or modifier handling, and
saddles the Public Health WG with maintenance every time US Core
revises its gender identity guidance. Declining outright (Disposition
C) ignores the reporter's expectation that USPHPL will visibly adopt
the R5 extension. The middle path — re-base USPHPL Patient profiles on
the current US Core Patient, add explicit `mustSupport` and narrative
pointing implementers at the R5 `individual-genderIdentity` extension
inherited from US Core, and close FHIR-38266 with a clear note
describing the inheritance — gives implementers everything the ticket
asks for while keeping the IG thin and aligned. The WG should confirm
the target US Core version as part of accepting this disposition.
