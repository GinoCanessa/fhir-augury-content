## Summary

The ticket asks the US Bidirectional Services eReferrals (BSeR) FHIR
Implementation Guide to be explicit, on a per-artifact basis, about
which artifacts are normative/STU versus informative, and to publish
relative FHIR Maturity Model (FMM) levels for the artifacts it defines.
Today the IG does not surface this information consistently, so
implementers cannot tell which profiles, extensions, value sets, and
operations are stable enough to build against and which are still early
work. The reporter (Lloyd McKenzie) raised this during ballot review
(imported as comment 37). No cross-referenced Jira tickets, Zulip
threads, or GitHub items were found in the index for this key, so the
request stands on its own and should be triaged against current BSeR IG
publishing practice.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR IG so that every conformance artifact (profiles,
extensions, value sets, code systems, operations, capability
statements, examples that double as profile illustrations) carries an
explicit publication-status pair:

1. A standards-status flag (e.g., `informative`, `trial-use`,
   `normative`, `deprecated`) using the `structuredefinition-standards-status`
   extension on the artifact's `StructureDefinition` /
   `ValueSet` / `CodeSystem` / `OperationDefinition`, mirroring the
   pattern used by core FHIR.
2. A numeric FMM level (0–5) using the
   `structuredefinition-fmm` extension, again following the core FHIR
   convention.

In addition, add an "Artifact Maturity" page (or section in the IG
home page) that summarises the matrix of every BSeR artifact, its
standards status, and its FMM level so reviewers and implementers can
see the picture at a glance. Default any artifact whose level the
authors cannot defend to FMM 1 (or 0 for clearly draft material) and
mark it as `trial-use` (since the IG itself is STU), reserving
`informative` for artifacts that are illustrative only.

#### Justification

This is exactly what the reporter asks for, and it aligns the BSeR IG
with the well-established convention used by the FHIR core spec and by
sibling HL7 IGs (e.g., US Core, IPS) that already carry per-artifact
FMM and standards-status badges. The FMM and standards-status
extensions are the sanctioned mechanism, are recognised by the IG
publisher's templates (badges render automatically in the artifact
header), and require no spec invention. Surfacing this metadata is a
non-breaking, additive change that materially improves implementer
guidance — particularly important for a public-health IG whose
artifacts are at varying levels of field experience.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than annotating every individual artifact, declare a single,
IG-wide standards-status and FMM level on the
`ImplementationGuide` resource and document any per-artifact deviations
in a short narrative section on the IG home page. Concretely:

- Set `ImplementationGuide.status = active` and use the
  `structuredefinition-standards-status` extension on the
  `ImplementationGuide` itself to assert `trial-use` for the IG as a
  whole.
- Add an "Artifact Maturity" home-page section that lists only the
  exceptions — artifacts that are informative-only, deprecated, or at
  a notably lower/higher FMM than the IG default.
- Optionally tag those exception artifacts with the per-artifact
  extensions, but do not require it for artifacts that match the
  IG-wide default.

#### Justification

This costs much less editorial effort than tagging every artifact and
still satisfies the spirit of the ticket: an implementer can answer
"what is STU here and what is informative?" by reading one section.
It is also more honest in IGs where the authoring group does not yet
have field-experience data to defend distinct FMM levels per artifact;
inventing fine-grained levels just to fill in metadata can mislead
implementers more than a single conservative IG-wide level. The
trade-off vs. Disposition A is loss of tooling support — the
publisher's per-artifact maturity badges will not appear, and
downstream consumers that scrape the extensions cannot reason about
individual artifacts. For an IG with relatively few profiles this is a
reasonable trade.

---

### Disposition C: Decline

#### Proposal

Close as Persuasive – Not Persuasive (or "Not Persuasive with Mod") on
the grounds that BSeR is published as an STU IG, that the IG-level STU
designation already conveys the appropriate maturity expectation to
implementers, and that the FHIR core FMM scale was designed for
core-spec artifacts rather than IG-defined profiles. Document the
IG-wide STU status more prominently on the home page in lieu of
per-artifact FMM tagging.

#### Justification

This option is defensible if the workgroup believes the BSeR artifacts
are uniformly at the same maturity (typical for a first-published STU)
and that introducing FMM scoring would imply a level of operational
field experience the IG does not yet have. It also avoids the
maintenance burden of keeping per-artifact FMM levels in sync with
real-world implementer feedback over successive ballots. The downside
is that it does not address the reporter's underlying concern about
distinguishing informative content (examples, narrative-only profiles)
from conformance content, which is a real implementer pain point even
within an STU IG.

---

### Recommendation

**Recommended disposition:** A

The reporter's concern is concrete and implementer-facing: without
per-artifact standards-status and FMM annotations, consumers of the
BSeR IG cannot tell which profiles and extensions are safe to build
against and which are still in flux. The
`structuredefinition-standards-status` and
`structuredefinition-fmm` extensions are the HL7-standard mechanism
for exactly this, the IG publisher renders them automatically as
badges, and adopting them is additive and non-breaking. Disposition B
is a reasonable fallback if the workgroup feels it cannot defend
per-artifact FMM levels yet, but at minimum it should adopt the
informative-vs-STU distinction at artifact granularity, which is the
half of the request that has the highest implementer value. Disposition
C leaves the original problem unsolved and is hard to defend given that
peer IGs already publish this metadata.
