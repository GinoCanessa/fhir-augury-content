## Summary

The reporter is asking a documentation/scope question about how the BSeR
implementation guide depicts intermediary participants (a CBO Hub or a
Healthcare Information Exchange (HIE)) in its interaction diagrams.
Today, the BSeR IG describes two different operating modes for an
intermediary: (1) the intermediary acts *on behalf of* the CBO and
exposes the data to the CBO via web access at the intermediary, and
(2) the intermediary acts as a *pass-through* and the CBO retains full
FHIR functionality at its own endpoint. The current "with intermediary"
interaction diagrams only show one of these two modes per transaction —
one mode is illustrated against the Referral-Push (with intermediary)
use case and the other against the Referral-Notify-and-Pull (with
intermediary) use case. The question for the work group is whether both
intermediary modes should be diagrammed for **both** transaction
patterns, so that implementers can see all four combinations rather
than only the two illustrative ones currently published.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the BSeR IG narrative pages for the two intermediary-mediated
transactions — *Referral - Push (with intermediary)* and *Referral -
Notify and Pull (with intermediary)* — so that **each** transaction
shows both intermediary operating modes side-by-side. Concretely, for
each of the two transaction pages:

1. Retain the existing interaction diagram and label it explicitly as
   the "intermediary acts on behalf of the CBO (web access at the
   intermediary)" variant **or** the "intermediary as pass-through (full
   FHIR functionality at the CBO)" variant, whichever it currently
   depicts.
2. Add a second sequence/interaction diagram showing the other variant
   for the same transaction, using the same actors and message labels
   so the two diagrams are visually comparable.
3. Add a short narrative paragraph above the diagrams introducing the
   two intermediary modes and pointing the reader to both diagrams.
4. Cross-link the two pages so that an implementer reading one
   transaction can quickly find the equivalent diagram on the other.

#### Justification

The reporter's underlying need is documentation completeness: both
intermediary modes are real-world deployments in BSeR, and the IG
already acknowledges both modes in prose. Showing only one mode per
transaction creates the false impression that mode and transaction are
coupled, which can lead implementers to conclude (incorrectly) that
"Push" requires one intermediary architecture and "Notify and Pull"
requires the other. Diagramming all four combinations explicitly closes
that gap and aligns the diagrams with the narrative the IG already
publishes. This is a documentation-only change with no impact on
conformance, profiles, value sets, or capability statements.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than producing four separate transaction diagrams, introduce a
single *intermediary architecture* page in the BSeR IG that:

1. Defines the two intermediary modes ("on behalf of CBO with web
   access" vs. "pass-through with full FHIR at CBO") once, with one
   generic diagram per mode that is independent of the specific
   transaction.
2. Notes explicitly that **either** mode can be combined with **either**
   transaction (Referral - Push or Referral - Notify and Pull), so all
   four combinations are supported.
3. Updates the two existing transaction pages to keep their current
   single diagram as an illustrative example, but adds a one-line
   pointer back to the new intermediary architecture page for the full
   matrix of combinations.

#### Justification

Duplicating diagrams across both transactions doubles the maintenance
surface — any future change to actor naming, message labels, or
sequencing has to be made in four places instead of two, and the
diagrams will inevitably drift. Factoring the *intermediary mode* as an
orthogonal concern from the *transaction pattern* matches how the IG
actually describes them in prose and reduces the risk that implementers
read one diagram as normative for a particular combination. The
trade-off versus Disposition A is that readers no longer see the
combination they care about as a single rendered diagram; they have to
mentally combine the transaction diagram with the intermediary-mode
diagram. For an IG aimed at experienced implementers this is usually
acceptable.

---

### Disposition C: Decline

#### Proposal

Close FHIR-42533 as *answered, no change*: the existing diagrams plus
the surrounding narrative are sufficient, because the prose already
states that either intermediary mode may be used with either
transaction. Add (at most) a one-sentence clarifying note on each of
the two "with intermediary" transaction pages stating that the diagram
shown is illustrative and that the other intermediary mode is equally
valid.

#### Justification

The interaction diagrams in BSeR are illustrative, not normative —
conformance is driven by the profiles, capability statements, and
operations, none of which constrain the intermediary architecture.
Adding diagrams for all four combinations risks implying that the
diagrams *are* normative and that any deployment that doesn't match a
diagram is non-conformant. A brief clarifying sentence achieves the
reporter's goal (avoiding the false coupling between intermediary mode
and transaction) at minimal cost and without expanding the diagram
inventory the work group has to maintain.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the reporter's concern — that only one of two
intermediary modes is shown per transaction — without doubling the
diagram inventory. By factoring intermediary architecture out as a
separate, orthogonal concern, the IG can state once that any
intermediary mode can be paired with any transaction, and the existing
transaction diagrams can remain focused on the message exchange itself.
Disposition A solves the same problem but at a higher long-term
maintenance cost and with a non-trivial risk of diagram drift between
the two transaction pages. Disposition C is the cheapest option but
leaves the diagrams in a state where readers have to infer the missing
combinations from prose, which is precisely the situation the reporter
is flagging as confusing. The work group should confirm with the BSeR
authors that the two intermediary modes really are orthogonal to the
transaction choice (the ticket strongly implies they are); if that is
confirmed, B is the right structural fix. If the work group instead
wants to preserve the current page-per-transaction layout for editorial
reasons, fall back to Disposition A.
