## Summary

This ticket, raised by Lloyd McKenzie against the US Bidirectional Services
eReferrals (BSeR) IG, asks for clearer documentation of the BSeR workflow
state-transition table. Lloyd notes that event labels such as "Send
Cancellation Request" are not self-explanatory — for example, it is not
obvious why that event would move a referral status from `received` to
`in progress`. He asks for more detailed descriptions for each event in the
table and further suggests that each event likely warrants its own
`MessageDefinition` resource as well as a corresponding event definition
code. The ticket has been triaged as Persuasive; the resolution accepts
adding detailed event descriptions and flags the per-event
`MessageDefinition` plus event-code work as a follow-up "ToDo." No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the request as stated. In the BSeR IG narrative that contains the
referral state-transition table, expand each event row (e.g., *Send
Referral Request*, *Send Cancellation Request*, *Send Update Request*,
*Send Feedback Response*, etc.) with a short prose description that
explains (1) what the event represents in the real-world referral
workflow, (2) which actor initiates it, and (3) why it causes the listed
status transition (e.g., why "Send Cancellation Request" moves a referral
from `received` to `in progress` rather than directly to a terminal
state). In addition, define a discrete `MessageDefinition` resource per
event and a corresponding event code in the IG's event/message-event code
system / value set so that each row in the table is grounded in a
machine-readable artifact that implementers can bind to.

#### Justification

This matches the ticket's literal ask and the already-recorded
Persuasive resolution. It directly addresses the reviewer's confusion:
the table today is effectively a glossary of unexplained labels, and
ambiguous status transitions are a known source of interoperability
defects in workflow IGs. Per-event `MessageDefinition` resources are the
FHIR-canonical way to document a messaging-style interaction (one
trigger event → one expected message → one or more responses), and
having an explicit event code per event lets implementers reason about
the workflow without reverse-engineering it from prose. It is also
consistent with how other HL7 messaging-oriented IGs document their
events.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the documentation half of the request — add detailed prose
descriptions and a justification column for each status transition in the
BSeR table — but defer the per-event `MessageDefinition` and
event-definition-code expansion. Instead, define a single BSeR event
`CodeSystem` / `ValueSet` whose codes enumerate the events (one code per
row), and bind the table to those codes. Keep the existing
`MessageDefinition`(s) at their current granularity until implementer
feedback shows the consolidated form is actually causing problems.

#### Justification

The descriptive ambiguity is the user-visible bug; the structural
proliferation of `MessageDefinition` resources is a heavier change with
real cost (more artifacts to maintain, more conformance surface,
potentially duplicate Bundle profiles per message type). FHIR allows a
single `MessageDefinition` to cover multiple related events via its
`event[x]` and `allowedResponse` slots, and BSeR's events are closely
related variations on the same underlying referral message. Splitting
them out is reversible later if the workgroup gets feedback that
implementers actually need per-event definitions; consolidating them
after they are published is much harder. This disposition addresses the
reviewer's core complaint (table is opaque) at lower IG churn.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive* on the structural change (no per-event
`MessageDefinition`, no new event-definition codes) and decline the
documentation expansion as well, on the basis that the existing
narrative around the BSeR workflow already explains the state machine
and the table is intended as a quick-reference summary, not a
self-contained specification.

#### Justification

This is the weakest path and is included for completeness. It would only
be defensible if the workgroup believed (a) the surrounding BSeR
workflow narrative truly does explain each event in enough depth that
the table is unambiguous in context, and (b) the cost of adding event
codes and `MessageDefinition`s outweighs the implementer benefit. Given
that the reporter is Lloyd McKenzie — a long-time HL7 reviewer who
explicitly could not infer the meaning from the current text — that
position is hard to defend, and the workgroup has in fact already
recorded a Persuasive resolution.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already recorded the resolution as Persuasive with a
"ToDo" that mirrors Disposition A almost exactly (detailed descriptions
plus per-event `MessageDefinition` plus event codes). Disposition A
formalizes that path and keeps BSeR aligned with how FHIR messaging IGs
are normally documented: each trigger event is a first-class artifact,
not a row in an unexplained table. Disposition B is a reasonable
fallback if the workgroup, when actually drafting the changes, finds
that the events collapse cleanly onto a single `MessageDefinition` with
multiple `event[x]` entries — in that case the recommendation should be
to keep the prose-and-codes parts of A and downgrade only the
`MessageDefinition` proliferation to B's consolidated form. Disposition
C contradicts the existing Persuasive resolution and should not be
pursued.
