## Proposed Dispositions

Note: because no GitHub cross-references were surfaced, the
dispositions below cannot name a specific authoring root or file path
from a `repo-analysis` briefing. They reference the affected
specification (Vital Records Common Profiles Library, owned by the
Public Health WG) at the conceptual level only. Once the NIOSH/ONC
update is published and the VRCL repo is in scope, a follow-up
implementation plan should re-load the relevant repo briefing.

### Disposition A: Accept As Requested

#### Proposal

Keep the ticket open and tracked against the VRCL IG. Once NIOSH and
ONC publish the updated industry and occupation vocabulary, update the
VRCL IG's I&O terminology bindings (the value set(s) and/or code
system reference(s) used by the decedent / patient industry and
occupation elements) to point at the new artifacts. Re-publish any
affected profiles and examples, and add a release-notes entry
describing the vocabulary change and any migration impact. Coordinate
the change with Amanda Deering and Lori Fourquet as requested by the
reporter.

#### Justification

The ticket is explicitly a forward-looking placeholder for an upstream
terminology change driven by NIOSH/ONC. Lynn Laakso has already
re-opened it post-STU2 specifically so the VRCL IG can absorb that
change in a future cycle. Updating the binding (rather than restating
the codes inline) is consistent with FHIR's terminology layering: the
IG owns the binding, the value set / code system owns the membership.
Accepting honors the reporter's intent and the prior 22-0-2 WG vote to
keep this in scope for a future ballot.

---

### Disposition B: Alternative Approach

#### Proposal

Split the ticket into two work items: (1) a tracking ticket that stays
open until the NIOSH/ONC vocabulary is formally published, and (2) a
concrete change request opened only once the new vocabulary exists,
specifying the exact value set OIDs / canonical URLs to bind, the
target VRCL profiles and elements, and any back-compat shim (for
example, retaining the old binding strength as `extensible` for one
release while the new binding is `preferred`, or vice versa, depending
on what NIOSH/ONC publish). The split ticket would also call out
example updates and IG publication-tooling impacts (NIOSH/ONC code
system caches, IG-publisher terminology server expectations).

#### Justification

The current ticket cannot be acted on as written — the upstream
artifact does not yet exist, so any binding change would be
speculative and potentially churn the IG twice. Splitting separates
the *governance* signal (we intend to adopt the update) from the
*implementation* change (the actual binding edit), which is cleaner
for ballot tracking and lets the WG vote substantively on the binding
change once the codes and value sets are stable. It also provides a
natural place to evaluate breaking-change implications and migration
guidance, which a one-line tracking ticket cannot capture.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Not Persuasive — Insufficient Information**
(or re-resolve as **Considered for Future Use** without re-opening),
on the grounds that no concrete change can be evaluated until the
NIOSH/ONC vocabulary update is actually published. Ask the reporter
to file a new, specific ticket once the updated value set(s) and code
system(s) are available, with the exact canonical URLs and target
elements identified.

#### Justification

The ticket as written is a notice of an upstream activity, not a
change request the WG can evaluate or implement. There is no
proposed value set, no proposed binding strength, and no enumerated
list of changing codes. The original 2024-02-22 vote (22-0-2)
already chose "Considered for Future Use" precisely because the
change was contingent on an upstream artifact that did not exist —
re-opening without that artifact does not change that calculus.
Closing keeps the tracker free of indefinitely-open placeholders;
the WG will see the substantive change when the concrete ticket is
filed.

---

### Recommendation

**Recommended disposition:** B

The reporter's intent (track the upstream NIOSH/ONC update and adopt
it in VRCL) is legitimate and should not be lost — Disposition C
risks the change falling off the radar between tracker hygiene
sweeps. But Disposition A as written commits the WG to "update the
binding" before the binding target exists, which is not actionable
and would have to be re-opened or amended once NIOSH/ONC actually
publish. Splitting into a tracking ticket plus a concrete
implementation ticket (Disposition B) preserves the governance
signal *and* lets the WG vote substantively on the real change
later, with the actual canonical URLs, binding strengths, and
backward-compatibility analysis on the table. It also gives Amanda
Deering and Lori Fourquet — who the reporter asked to be looped in —
a clear, scoped artifact to comment on when the update lands.
