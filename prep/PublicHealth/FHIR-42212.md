## Summary

Ron G. Parker (the reporter, importing his own Comment 24 from the BSeR
ballot) raises a guidance/clarity concern about the US Bidirectional
Services eReferrals (BSeR) IG. As a national IG intended to drive
vendor harmonization for referrals, BSeR currently presents **four
technical interaction patterns with eight total options**, which the
reporter argues is too broad to be scalable across implementation
contexts. He acknowledges that BSeR must accommodate a wide range of
public-health programs and locale-specific privacy regimes, but asks
the workgroup to (a) designate a **recommended default pattern** that
implementers should adopt absent strong reasons otherwise, and (b)
provide a **note explaining when derivative/downstream guides** —
especially those targeting referral types with greater sensitivity to
the receiving system seeing data — **may legitimately enact an
alternative pattern such as "Notify and Pull"**, with the
justification (the "x, y, z") spelled out in the IG.

There are no formally cross-referenced Jira tickets, Zulip threads, or
GitHub items recorded for this ticket in Augury. Keyword-similarity
search surfaces a cluster of sibling BSeR ballot comments from the
same review batch (e.g., FHIR-42467 "Need much more detail on pull
option", FHIR-42456 "How is package determination done?", FHIR-42455
"Unclear sentence", FHIR-42204 on duplicative Task requirements) that
together indicate the workgroup received broad feedback that the
interaction-pattern material in the IG is under-explained and
under-prioritized. FHIR-42212 sits at the top of that cluster as the
strategic ask: pick a default, and explain when departing from it is
appropriate.

## Proposed Dispositions

Because no GitHub cross-references were surfaced, the dispositions
below cannot name a specific authoring root or file path from a
repo-analysis briefing. They are framed in terms of the BSeR IG
narrative content and the IG's existing pattern catalog, which the
BSeR workgroup will need to map onto the actual BSeR IG repository at
disposition time.

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR IG's interaction-patterns narrative to:

1. **Designate a single recommended default interaction pattern.** Based
   on the existing four patterns / eight options, select the one that
   best supports the broadest set of BSeR vendors and referral types
   (most plausibly the simplest "Push" / direct-send option, but the
   workgroup must confirm). Mark this option in the IG narrative and
   in any pattern-comparison table with a clear "Default —
   conformant servers SHOULD support this pattern unless a derivative
   guide says otherwise" callout.
2. **Add a "When to deviate" subsection** immediately after the
   default callout that lists the legitimate reasons a downstream /
   derivative IG (e.g., behavioral health, HIV/STI, substance-use,
   minor-consent referrals) may instead require **"Notify and
   Pull"** or another non-default pattern. Spell out the named
   reasons (the "x, y, z") — minimally: heightened receiver-side
   sensitivity / consent gating, jurisdictional privacy law (e.g.,
   42 CFR Part 2), payer or program rules requiring the receiver to
   re-pull only what it is authorized to see, and audit / minimum
   necessary obligations.
3. Reflect the same default-vs-alternative framing in the
   CapabilityStatements and any conformance language so vendors have
   a single target for "baseline BSeR" while leaving the other
   patterns explicitly available to derivative guides.

#### Justification

This is exactly the guidance the reporter (and, by the keyword
clustering, several other ballot commenters on the same review)
asked for. It addresses the core risk the reporter raises — that a
national harmonization IG which leaves all four patterns equally
"on the table" effectively harmonizes nothing, because each vendor
can pick a different one and still claim conformance. Naming a
default while explicitly blessing "Notify and Pull" for sensitive
downstream use cases preserves BSeR's required breadth (broad
programs, varied locales, varied privacy regimes) without forcing
every implementer to support all eight options. It is also a
narrative-only / conformance-clarification change — no resource,
profile, or extension model is altered — so the implementation
burden on existing BSeR adopters is small relative to the
interoperability payoff.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than designating a single default at the top level of BSeR,
restructure the patterns section into a **decision-tree / capability
matrix**:

- Define a small number of named **deployment scenarios** (e.g.,
  "general clinical referral", "sensitive-services referral",
  "closed-network payer-mediated referral", "cross-jurisdiction
  public-health referral").
- For each scenario, recommend **one** interaction pattern (and, where
  appropriate, one fallback) drawn from the existing four / eight, with
  a one-paragraph rationale.
- Mark all other pattern/option combinations as "permitted but not
  recommended for this scenario" rather than "equally valid".
- Add explicit guidance that **derivative IGs SHALL identify which
  scenario(s) they target** and SHOULD only narrow, never broaden, the
  scenario's recommended pattern set.

#### Justification

This addresses the same underlying scalability concern but avoids the
risk that a single global default is wrong for a meaningful slice of
BSeR's audience. Public-health referral programs vary enough that
"the default is X" can be technically correct but operationally
useless for, say, a behavioral-health referral network where X is
never actually used. A scenario-keyed matrix gives implementers a
sharper signal ("for my use case, do this") than a single default
plus a deviation note, and it gives derivative IG authors a
prescribed slot to plug into rather than re-litigating the pattern
choice. The trade-off vs. Disposition A is more authoring effort
inside BSeR and a slightly higher cognitive load for first-time
readers, but it avoids encoding a default that the workgroup may
later need to walk back.

---

### Disposition C: Decline

#### Proposal

Decline as **out of scope for this ballot cycle / already addressed by
existing narrative**, on the basis that:

- BSeR intentionally exposes the full pattern set so that derivative
  IGs (which are the appropriate venue for narrowing) can choose
  without re-defining the patterns themselves.
- The current narrative already explains when each pattern applies,
  and adding a "default" risks being interpreted as a normative
  conformance requirement that BSeR does not intend to make.

Reject the request and respond "Not Persuasive" with a pointer to the
existing pattern-selection narrative, asking the reporter to raise
specific gaps rather than a global "pick a default" change.

#### Justification

A defensible-but-weak case. It rests on the position that BSeR is a
*framework* IG whose job is to expose mechanisms, leaving policy to
derivative guides — which is a legitimate stance for some HL7 IGs.
The reason this disposition is unlikely to satisfy the workgroup is
that the reporter's concern is precisely that without a default,
"vendor harmonization" — BSeR's stated purpose — is not actually
achieved. Declining also leaves the cluster of related ballot
comments (FHIR-42467, FHIR-42456, FHIR-42455, FHIR-42204) without
the unifying guidance that would resolve several of them at once.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers what the reporter asked for, is the
lowest-cost change to the IG (narrative + a CapabilityStatement
clarification, no resource/profile breakage), and aligns BSeR's
stated purpose ("drive harmonization around vendor capabilities") with
its actual conformance guidance. Disposition B is intellectually
stronger and would be the better long-term answer, but it requires
the workgroup to enumerate and agree on a stable set of deployment
scenarios — work that is unlikely to land cleanly in a ballot
reconciliation cycle and that risks blocking other related comments
(FHIR-42467, FHIR-42456, FHIR-42455). Disposition C should be avoided:
declining a "Highest"-priority ballot comment that explicitly asks
for harmonization guidance on a national harmonization IG is the kind
of disposition that tends to attract a Negative vote on
reconciliation. The pragmatic path is to accept (Disposition A) and
optionally signal that a future BSeR version may move toward the
scenario matrix described in Disposition B.
