## Summary

This ticket challenges an apparent assumption in the US Bidirectional
Services eReferrals (BSeR) IG that a referral response is delivered as a
single, monolithic payload. Lloyd McKenzie — submitting the comment as
part of the BSeR ballot import (Comment 27) — argues that real-world
referral workflows generate interim results, that the data a recipient
needs varies by the practitioner/role consuming the response, and that
the level of detail returned (just the report, the supporting
observations, the device/observer provenance, etc.) should be driven by
the requester rather than baked into a one-shot response. The ticket
asks BSeR to either justify the single-payload assumption or extend the
specification to allow incremental, role-aware, and request-shaped
responses.

The cross-reference index returned no incoming or outgoing links for
this ticket, and no Zulip or GitHub items reference it directly. It is,
however, part of a tightly clustered set of Lloyd's BSeR ballot
comments. The most directly paired sibling is
[FHIR-42472](https://jira.hl7.org/browse/FHIR-42472) ("Why is there a
presumption of a single request payload?", Comment 23), which raises
the symmetric concern on the request side — that follow-up questions,
drill-downs from one report to another, and image/source retrieval all
imply a multi-message exchange rather than a single send. Other
keyword-similar siblings reinforce the same theme:
[FHIR-42467](https://jira.hl7.org/browse/FHIR-42467) ("Need much more
detail on pull option"),
[FHIR-42483](https://jira.hl7.org/browse/FHIR-42483) ("Retrieve Payload
Data Requirements not defined"),
[FHIR-42492](https://jira.hl7.org/browse/FHIR-42492) ("How are
responses associated to referral"), and
[FHIR-42506](https://jira.hl7.org/browse/FHIR-42506) ("Outcome package
shouldn't only be at end"). Together they describe a BSeR exchange
model that the commenter believes is under-specified for iterative,
multi-payload referral conversations.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR IG to explicitly drop the single-response-payload
assumption and to specify a multi-payload, incrementally-deliverable
response model. Concretely:

- In the BSeR exchange-pattern narrative, replace "the response is
  delivered as a single payload" with language that allows zero-or-many
  interim response payloads followed by a terminal "final" payload, all
  correlated back to the originating `ServiceRequest`/`Task`.
- Add a `Task.businessStatus` (or equivalent) progression covering at
  least `in-progress / interim-available / completed`, with a
  conformance requirement that interim results are exposed via the
  same Task chain that delivers the final result.
- Define a "response detail level" parameter on the pull/retrieve flow
  (e.g., a coded set of `report-only`, `report+observations`,
  `report+observations+provenance`, `full`) so the requester chooses
  how much supporting data to pull. This parameter lives on the
  retrieval operation (e.g., as an `_include`/`_revinclude` profile or
  a dedicated query parameter on the BSeR retrieve endpoint).
- Add MUST-level conformance that each response payload — interim or
  final — references the originating `ServiceRequest.id` so that
  recipients can correlate, addressing the concern surfaced in
  FHIR-42492.

#### Justification

The submitter is the FHIR methodology lead and is describing the
real-world referral workflow accurately: referral responses are almost
never single, complete deliverables, and the right level of supporting
detail is recipient- and role-dependent. Aligning the IG with that
reality removes a baked-in mismatch between BSeR and how implementers
will actually build it. It is also internally consistent with the
sibling tickets (FHIR-42467 detail on pull, FHIR-42483 retrieve data
requirements, FHIR-42492 correlation, FHIR-42506 interim outcomes), so
adopting this disposition resolves the cluster coherently rather than
piecemeal.

---

### Disposition B: Alternative Approach

#### Proposal

Keep BSeR's primary "response = one payload" pattern as the simple,
conformance-required path, but **add a profiled subscription / query
companion pattern** that explicitly handles the incremental,
role-aware, and detail-tunable cases:

- Define a BSeR `Subscription` (R4 Backport or R5 topic-based)
  subscribed to `Task`/`Communication` updates on the originating
  referral. Recipients opt in if they want interim notifications.
- Define a BSeR retrieve operation (or profile FHIR's `$everything` /
  `_include` semantics) keyed off `ServiceRequest.id` that lets the
  requester choose what to pull (report only, with observations, with
  provenance/devices). Document the supported parameter set and
  conformance.
- Leave the existing single-payload "push" flow in place for the
  simple, fully-resolved case so existing implementations are not
  invalidated.

This treats the underlying need — "interim, role-aware, requester-shaped
data" — as an opt-in capability rather than a redefinition of the core
exchange.

#### Justification

This addresses every concrete need Lloyd raises (interim availability,
role-driven content, requester-driven detail) without forcing every
BSeR implementer to rebuild the response side. It also aligns with
existing FHIR mechanisms (Subscriptions, `_include`, `$everything`)
rather than inventing IG-specific machinery, lowering implementation
burden. The trade-off vs. Disposition A is that conformance becomes
two-tier: simple senders can stay on the push path, while richer
exchanges move to the subscription/query path. That two-tier shape is
arguably honest about where deployments actually sit today.

---

### Disposition C: Decline

#### Proposal

Decline as **out of scope for the current BSeR publication cycle**.
Capture the concern in the IG's "Future Considerations" section and
defer multi-payload / interim-results / requester-shaped response
mechanics to a subsequent BSeR version. Close FHIR-42476 with a
"Persuasive with Mod – Future" or "Not Persuasive – Future
Consideration" disposition, depending on the workgroup's read.

#### Justification

The ballot comment is framed as a question, not a defect, and it lands
on a high-impact architectural seam. Reworking the response model
late in the cycle risks destabilizing the already-implemented BSeR
exchange and entangles this ticket with at least four siblings
(FHIR-42467, FHIR-42483, FHIR-42492, FHIR-42506) that would all need
coordinated resolution. If the workgroup's bandwidth is constrained,
deferring is more defensible than shipping an under-specified
multi-payload model that implementers cannot reliably interoperate
against.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the strongest balance of the three. The submitter's
underlying concerns are real and well-evidenced by the sibling cluster,
which makes outright decline (C) hard to justify on substance —
especially since the same author keeps surfacing the same issue from
multiple angles. At the same time, Disposition A's wholesale
redefinition of the response model is a large surface-area change that
would need coordinated resolution of FHIR-42467, FHIR-42483,
FHIR-42492, and FHIR-42506 in lockstep, and risks late-cycle
destabilization. Disposition B preserves the simple push path for
implementations that already work, while adding a conformance-defined
subscription + retrieve profile that directly answers the "interim,
role-aware, requester-shaped" need using existing FHIR primitives. It
also gives the workgroup a clean way to roll the sibling tickets into
a single coherent capability rather than patching each one separately.
