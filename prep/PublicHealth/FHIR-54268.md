## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

On the `central-cancer-registry-primary-cancer-condition`
StructureDefinition (canonical
`http://hl7.org/fhir/us/central-cancer-registry-reporting/StructureDefinition/central-cancer-registry-primary-cancer-condition`),
set `mustSupport = true` on the `Condition.stage` element differential
(and any sliced children that previously carried `mustSupport = true`
in the pre-US-Core-6.1 version of the profile, e.g., `stage.summary`
and `stage.assessment` if they were originally flagged). Regenerate the
profile snapshot, refresh the IG output, and update the change log to
note the restoration of the `mustSupport` flag that was inadvertently
dropped during the US Core 6.1 alignment. No companion changes are
required to bindings, cardinality, or constraints.

#### Justification

The reporter and the triage commenter both characterise the missing
flag as an oversight from the prior US Core 6.1 re-alignment work, not
an intentional design choice, and the ticket already carries a
"Persuasive" resolution with the note "We will apply change as
requested." US Core 6.1 itself flags `Condition.stage` as `mustSupport`
on its profiled cancer/condition guidance, so restoring the flag on the
CCRR primary cancer condition profile re-aligns CCRR with the parent
profile's expectations rather than diverging from it. The change is
narrowly scoped to a single element on a single StructureDefinition
and carries no ripple effect on cardinality or terminology bindings,
making it low-risk for the `CCRR_Block1` cleanup ballot.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the `mustSupport` flag as in Disposition A, but at the same time
audit every element on `central-cancer-registry-primary-cancer-condition`
(and the other CCRR profiles touched in the US Core 6.1 alignment pass)
against the corresponding US Core 6.1 profile and the prior CCRR STU1
profile, and produce a single block ticket / PR that restores any other
`mustSupport` (and `must-support-modifier`) flags that were dropped by
the same alignment edit. Document the audit results in the IG's change
log so reviewers can see the full delta in one place.

#### Justification

The fact that this flag was lost during a profile re-work strongly
suggests that the same alignment pass may have dropped other
`mustSupport` annotations elsewhere in the profile (or in sibling
profiles touched in the same commit). Bundling the audit with this fix
costs only marginal effort, prevents a series of one-element follow-up
tickets in later ballot cycles, and gives the work group a defensible
"as of CCRR_Block1, the profile's mustSupport surface is fully
reconciled with US Core 6.1" statement for ballot reconciliation.
The downside is a wider blast radius and a slightly slower turnaround
for this specific ticket, which is why it is positioned as an
alternative rather than the primary recommendation.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive**: do not add `mustSupport` on
`central-cancer-registry-primary-cancer-condition.stage`. Implementers
that need to populate stage information can do so via the underlying
`Condition.stage` element from the base resource without an explicit
`mustSupport` flag, and the IG can document an expectation in narrative
prose rather than as a profile constraint.

#### Justification

`mustSupport` semantically obliges every conformant CCRR producer to
populate `Condition.stage` when known and every CCRR consumer to be
able to handle it. If there is any CCRR data-source population for
which staging information is genuinely out of scope (e.g., minimal
reportability submissions where stage is captured elsewhere in the
bundle), flagging the element `mustSupport` would impose an
implementation burden that the original re-alignment may have
deliberately removed. Declining preserves that flexibility.

---

### Recommendation

**Recommended disposition:** A

The ticket is unambiguous, the missing flag is explicitly described as
an oversight rather than a design choice, the existing "Persuasive"
resolution and the triage commenter's note already point in this
direction, and the change touches a single element on a single
StructureDefinition. Disposition C is hard to defend given the
reporter's framing and the fact that US Core 6.1 itself treats
`Condition.stage` as `mustSupport`; declining would re-introduce
divergence rather than preserve a deliberate design. Disposition B has
real merit as a follow-up — the work group should seriously consider
opening a companion audit ticket against the rest of the
US-Core-6.1-alignment commit — but bundling the audit into this
specific ticket risks delaying a clearly-resolved fix that is already
slated for the `CCRR_Block1` block. Apply the flag now (Disposition A)
and track the broader audit separately.
