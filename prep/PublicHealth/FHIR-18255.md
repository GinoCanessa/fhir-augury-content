## Summary

The reporter (Eric Haas) asserts that the **US Bidirectional Services
eReferrals (BSeR) FHIR IG** lacks the critical "how to USE the artifacts"
guidance. The IG ships an interaction diagram, but the diagram is not tied
back to the conformance artifacts (profiles, operations, search parameters,
example bundles) it references. The ticket proposes adding step-by-step
narrative content that walks an implementer through the eReferral workflow
end-to-end, naming each artifact and FHIR API call as it appears, with
**inline request/response examples** at each step.

No cross-references were returned by FhirAugury for this ticket (no linked
Jira issues, Zulip threads, or GitHub items were indexed against
`FHIR-18255`). However, keyword-similarity surfaces a clear pattern: the
same reporter has filed essentially identical "narrative + inline-examples"
asks against several other IGs — most directly
[FHIR-30173](https://jira.hl7.org/browse/FHIR-30173) (Subscriptions
Backport), [FHIR-25710](https://jira.hl7.org/browse/FHIR-25710)
(Da Vinci ATR), and [FHIR-18261](https://jira.hl7.org/browse/FHIR-18261)
(eCR, also Public Health WG, filed in the same week as this ticket). All
three were resolved **Persuasive** or **Persuasive with Modification**,
which establishes a strong precedent for accepting this kind of editorial
ask in BSeR as well.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level **"Workflow Walkthrough"** narrative page (or expand
the existing interaction-diagram page) in the BSeR IG that walks the
end-to-end eReferral workflow step by step. For each step, the
narrative must:

1. Name the actor(s) and the FHIR interaction (e.g., `POST
   ServiceRequest`, `GET Task?...`, the relevant `$operation`, etc.).
2. Link by name to the conformance artifact(s) used at that step — the
   profile(s), Task code, OperationDefinition, SearchParameter, and any
   ValueSet bindings.
3. Embed an **inline, abbreviated request and response example** drawn
   from the IG's existing example bundles (or add new examples where
   gaps exist), with anchor links into the full example bundle.
4. Reference the matching node in the interaction diagram so the diagram
   and the narrative are explicitly tied together.

Concretely, in an IG-Publisher-driven IG this means new/expanded
narrative pages under the IG's `input/pages/` (or `input/pagecontent/`)
authoring root, plus any additional `input/examples/` instances
referenced inline. The exact paths must be confirmed against a BSeR
repo briefing once one exists (see Repo Context gap).

#### Justification

This is exactly the editorial approach the same reporter has asked for
across multiple IGs and that has been resolved **Persuasive** /
**Persuasive with Modification** at least three times
([FHIR-30173](https://jira.hl7.org/browse/FHIR-30173),
[FHIR-25710](https://jira.hl7.org/browse/FHIR-25710),
[FHIR-18261](https://jira.hl7.org/browse/FHIR-18261)). Tying narrative,
artifacts, and inline examples together is consistent with FHIR IG best
practice (artifact-centric guidance is the established pattern in
mature IGs like US Core and Da Vinci PDex) and directly improves
implementer onboarding — the primary audience for an IG. No structural
or conformance change is required; this is editorial content.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than producing one large monolithic walkthrough, attach the
narrative + inline example to **each conformance artifact's intro
file** (the per-profile, per-operation, per-Task-code intro pages that
the IG Publisher already concatenates above the generated artifact
detail). Specifically:

- For each profile used in the workflow, add an "Used in BSeR
  workflow" section to its intro that names the step(s) it participates
  in and embeds a minimal request/response excerpt.
- For each `$operation` and each Task business-status code, add a
  similar contextual block.
- Keep the existing interaction diagram page as the single index, but
  let it deep-link into each artifact's intro rather than restating
  every step inline.

This co-locates "how to use it" with "what it is", and keeps the
narrative discoverable from any entry point a reader lands on.

#### Justification

A single long walkthrough page tends to drift out of sync with the
artifact set as the IG evolves (steps reference profiles by name only;
when a profile is renamed or split, the walkthrough silently rots).
Attaching the guidance to the intro file of each artifact ties the
narrative to the source of truth, so that any change to the artifact
set forces the editor to revisit the relevant guidance fragment.
Trade-off vs. Disposition A: a brand-new reader still benefits from a
single linear narrative, so this approach should arguably be done **in
addition to** A rather than instead of it. As a pure alternative, B is
weaker for first-time readers but stronger for long-term maintenance.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that BSeR's
existing interaction diagram and per-artifact pages collectively
describe the workflow, and that adding a separate step-by-step
narrative duplicates content the IG Publisher already generates from
the conformance resources.

#### Justification

The most defensible decline rationale would be "duplicative content"
— but in practice this is hard to defend: the reporter explicitly
states the diagram is *not* tied to the artifacts, and the same
editorial gap has been judged Persuasive across multiple sibling IGs by
multiple workgroups. Declining would be inconsistent with the
workgroup's own precedent on
[FHIR-18261](https://jira.hl7.org/browse/FHIR-18261) and would leave
implementers without the orientation material that has become standard
practice in HL7 IGs. Recorded here for completeness; not recommended.

---

### Recommendation

**Recommended disposition:** A (with the per-artifact augmentation from
B as an editorial follow-on).

The request is editorial, well-scoped, and aligns directly with how
sibling IGs have resolved the identical ask from the same reporter. The
workgroup's own track record on
[FHIR-18261](https://jira.hl7.org/browse/FHIR-18261) (Persuasive,
"Resolved - change required") makes Decline indefensible. Disposition A
delivers the linear, reader-facing walkthrough the ticket actually
asks for; Disposition B is a sound maintenance discipline that should
be folded into the same editorial pass so the new walkthrough doesn't
degrade as BSeR continues to evolve. Before assigning the work, the
workgroup should ask the FhirAugury operator to add the BSeR IG repo
to the tracked repo set and run the `repo-analysis` skill, so that the
implementer can be pointed at the exact `input/pages/` authoring paths
and existing example bundles to extend.
