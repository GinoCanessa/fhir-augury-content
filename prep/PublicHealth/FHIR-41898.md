## Summary

Floyd Eisenberg, responding to a BSeR ballot question about the "Notify of
Referral" step in the *Referral - Notify and Pull* transaction, recommends
that **both** the `Task` and the `ServiceRequest` be sent (rather than the
`Task` alone). He uses the response to flag a broader concern about how `Task`
is being applied in the IG: as currently scoped, the receiving organization or
practitioner is expected to perform the referred clinical activity (procedure,
evaluation, observation), which makes `Task` feel like an internal worklist
item that adds clinician burden. He suggests that `Task` would be more
appropriate if it were directed at a scheduling application — i.e., a request
to schedule an appointment that fulfills the `ServiceRequest` — and notes that
the IG does not currently reference `Schedule`, `Appointment`, or
`AppointmentResponse`, all of which seem more relevant to that pattern. The
ticket is essentially two things in one: (1) a direct ballot answer ("send
both"), and (2) a design critique recommending the Public Health WG reconsider
how `Task` and the scheduling resources are layered into the BSeR referral
workflow. There were no Jira, Zulip, or GitHub cross-references discovered
through the FHIR Augury index.

## Details

**Description:**

In response to your request to ballot commenters "In the Notify of Referral
step in [Referral - Notify and Pull](http://hl7.org/fhir/us/bser/2023Sep/bi-directional_service_ereferral_transactions.html#referral-notify-and-pull)
should both the Task and SerivceRequest be sent, or should just the Task be
sent?":

High level response - send both

More complete response:

Task seems to be a challenge in that an organization or practitioner expected
to perform a referral must do some action (e.g., procedure, evaluation leading
to some sort of observation). Task seems to be an internal health organization
activity that implies a "task" is added to a clinician's activity
list/workflow (definitely adding burden) and that task is to perform a
procedure and report back the findings to the original requester. I would
consider a "task" more reasonable if it was to a scheduling application to
indicate a need to schedule an appointment to perform the requested action(s).
And, that means to schedule a visit, confirm an appointment to fulfill the
ServiceRequest. However, the Schedule resource
(http://hl7.org/fhir/R4B/schedule.html) is not referenced at all, nor is
Appointment (http://hl7.org/fhir/R4B/appointment.html) or AppointmentResponse
(http://hl7.org/fhir/R4B/appointmentresponse.html), both of which seem more
relevant to use of the Task resource in your IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR IG so that the *Referral - Notify and Pull* transaction
(`bi-directional_service_ereferral_transactions.html#referral-notify-and-pull`)
explicitly requires the Notify-of-Referral payload to contain **both** the
referral `Task` and the underlying `ServiceRequest` (referenced from
`Task.focus`/`Task.basedOn`), rather than the `Task` alone with the
`ServiceRequest` only being retrievable via the subsequent Pull. Tighten the
narrative and the Bundle profile / example for the notification message so
that the `ServiceRequest` is a mandatory entry, and add a conformance
statement to that effect. Leave the existing `Task`-centric workflow
otherwise unchanged for this ticket; treat Floyd's broader Schedule/
Appointment/AppointmentResponse critique as a separate follow-up item the WG
opens for the next ballot cycle.

#### Justification

This is the literal ballot response from the commenter ("send both") and is
seconded by `rbaker@acc.org`. Sending the `ServiceRequest` inline removes a
required round-trip on the Notify side and lets the receiver act on the
clinical request immediately — which matches typical implementer expectations
when a referral is "pushed". It is consistent with how other US Realm
referral / order-notification IGs bundle the order with the workflow wrapper,
and it does not require any breaking changes to the underlying Task or
ServiceRequest profiles. Splitting off the broader Task-vs-Scheduling concern
keeps this change small enough to land in the current cycle while still
resolving the balloted question.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the "send both" answer as in Disposition A, **and** at the same time
open the door to Floyd's broader concern by introducing an optional
scheduling-oriented variant of the Notify transaction. Concretely:

1. Make the Notify Bundle carry both `Task` and `ServiceRequest` (as in A).
2. Add IG guidance that distinguishes two receiver patterns: (a) a
   *performing* receiver, where `Task` represents the clinical work to be
   done and reported back (today's behaviour), and (b) a *scheduling*
   receiver, where `Task.code` indicates "schedule an appointment to fulfill
   this ServiceRequest" and the receiver responds via `Appointment` /
   `AppointmentResponse` rather than directly performing the service.
3. Reference `Schedule`, `Appointment`, and `AppointmentResponse` in the IG,
   and add (or reuse from US Core / base FHIR) lightweight profiles plus an
   example Bundle for the scheduling pattern.
4. Mark the scheduling variant as optional / informative for this cycle so it
   can mature without blocking the ballot fix.

#### Justification

This addresses both halves of the ticket: it answers the balloted question
directly, and it gives the WG a concrete place to take the design feedback
that `Task` as currently used adds clinician worklist burden. The scheduling
pattern Floyd describes is a real-world referral flow (referral → schedule
visit → perform service → report back), and explicitly modelling it keeps
BSeR aligned with how `Task`, `Appointment`, and `ServiceRequest` are
intended to compose in base FHIR R4B. The trade-off vs. Disposition A is
scope: the WG has to commit to authoring/maintaining the scheduling variant,
and implementers will see two notify patterns instead of one — but flagging
the scheduling pattern as optional/informative keeps backward compatibility
and lets adoption be driven by demand.

---

### Disposition C: Decline

#### Proposal

Decline the request to mandate sending both `Task` and `ServiceRequest` in
the Notify step, and decline the broader suggestion to introduce
`Schedule` / `Appointment` / `AppointmentResponse` references. Keep the
Notify payload `Task`-only and rely on the existing Pull step for the
receiver to retrieve the `ServiceRequest` (and any other referenced
resources) on demand. Document this position explicitly in the
*Referral - Notify and Pull* page so future ballot commenters understand
the design intent.

#### Justification

The defensible rationale is "already addressed by the existing Pull
mechanism": a receiver that needs the `ServiceRequest` can dereference it
from `Task.focus` immediately after Notify, so making it part of the Notify
payload is convenience, not capability. Mandating it would also enlarge the
Notify message, duplicate state that already lives behind a stable
reference, and could cause divergence between the Notify-time snapshot of
the `ServiceRequest` and its current authoritative version on the
requester's server. The scheduling-resources critique, while reasonable, is
out of scope for a ballot reconciliation item of this size and would be
better served by a dedicated WG discussion or a separate change request.

---

### Recommendation

**Recommended disposition:** A

The balloted question is narrow ("send both, or just Task?") and has a clear
answer from the commenter, seconded by another reviewer, with no contrary
position visible in Jira, Zulip, or GitHub. Disposition A resolves that
question definitively and ships a small, low-risk change in the current
cycle. Disposition B is attractive on the merits — the
`Task`-vs-`Schedule`/`Appointment` concern is a real design issue — but
bundling it into this ticket risks scope creep and a longer path to
reconciliation; the WG can spin that off as a follow-on change request and
still get full credit for the feedback. Disposition C is hard to defend
given that a balloter explicitly asked the question and answered it, and a
second reviewer agreed; declining without new counter-evidence would look
like the WG ignoring direct ballot input.
