## Summary

Lloyd McKenzie requests that the BSeR (Bidirectional Services eReferrals) IG
remove its `Composition` profile and drop `Composition` from the IG examples.
His rationale is that `Composition` is intended for human-curated narrative
packaging of information into a story, whereas BSeR exchanges are essentially
machine-driven query/response interactions whose payloads are better
represented as a `Bundle` of type `batch-response` (or similar) rather than
wrapped in a `Composition`. The ticket therefore proposes simplifying the IG
by deleting the BSeR Composition profile and replacing the example payloads
that currently use it with plain Bundles. No other Jira tickets, GitHub items,
or Zulip threads were surfaced as cross-references for this ticket.

## Details

**Description:**

> Composition is relevant if a human is packaging the information into a
> collection of information in a narrative order intended to tell a story.
> However, if you're just executing a Batch of queries, there's no reason to
> not just attach a Bundle that is the batch-response for those queries.
> Remove this profile and drop it from the examples.
>

## Proposed Dispositions

> Note: With no GitHub cross-references and no loaded repo briefings, the
> dispositions below describe spec-level changes to the BSeR IG. Any actual
> implementation would need to target the BSeR IG source repository (the IG
> for "US Bidirectional Services eReferrals (BSeR) (FHIR)"), which the
> Public Health work group should confirm and, if needed, request a
> `repo-analysis` briefing for before the change is executed.

### Disposition A: Accept As Requested

#### Proposal

Remove the `BSeR-Composition` (or equivalently named) `Composition` profile
from the BSeR IG entirely:

- Delete the StructureDefinition for the BSeR Composition profile from the
  IG's profile set, along with any companion search parameters,
  CapabilityStatement entries, and value-set bindings that exist solely to
  support that profile.
- Update the IG's `Bundle` profile(s) so that BSeR exchange payloads no
  longer require a `Composition` as `Bundle.entry[0].resource`. Where the
  IG previously profiled a "document" Bundle, replace the constraint with
  a `Bundle` of type `batch-response` (or another non-document Bundle
  type) appropriate for the query/response interaction.
- Remove every example instance whose root resource is the deleted
  `Composition` profile and replace each with an equivalent
  `Bundle` example carrying the same constituent resources directly as
  entries.
- Remove all narrative pages, diagrams, and "Conformance Resources"
  table rows that reference the BSeR Composition profile, and add a brief
  release-notes entry explaining the removal.

#### Justification

The reporter's argument is consistent with the FHIR specification's intent
for `Composition`: a Composition is a curated, narrative, attested
"document header" intended to assemble clinical content into a coherent
story for human consumption. BSeR exchanges are system-to-system referral
query/response interactions; there is no human author attesting to a
narrative, no notion of a signed clinical document, and no real consumer
of the Composition's `section` structure. Replacing the document Bundle
with a `batch-response` Bundle better represents what is actually
happening on the wire and removes profile machinery (author, attester,
sections) that implementers must populate without semantic value.

---

### Disposition B: Alternative Approach

#### Proposal

Retain the `Composition` profile in the IG but mark it **deprecated** and
introduce a parallel, non-document `Bundle` profile (e.g.,
`BSeR-ExchangeBundle` of `Bundle.type = batch-response`) as the
RECOMMENDED payload going forward:

- Add a deprecation banner on the BSeR Composition profile page indicating
  it is no longer recommended and will be removed in a future version.
- Publish the new `BSeR-ExchangeBundle` profile and update the
  CapabilityStatement so both the document-style payload and the
  batch-response-style payload are listed, with the batch-response form
  marked "preferred" / SHALL for new implementations.
- Re-author the example set so the primary examples use the new Bundle
  profile, retaining one or two Composition-based examples for backwards
  compatibility reference until the next major release.
- Open a follow-up ticket to actually delete the Composition profile in
  the next major BSeR release once implementers have migrated.

#### Justification

BSeR has been published in multiple ballot/STU releases that already use
the Composition profile, and existing implementers may have built sender
and receiver code against it. A two-step deprecation path preserves
backwards compatibility for one release cycle, gives implementers a clear
migration target (the new Bundle profile), and avoids a hard breaking
change in a Public Health IG where deployed systems can be slow to
re-validate. The end state matches Disposition A; only the timing
differs.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep the BSeR `Composition` profile and its
examples as-is. Close the ticket as "Persuasive with Mod" → "Not
Persuasive" with the rationale that the Composition profile reflects an
intentional design choice by the BSeR authors to package eReferral
content as a FHIR Document so that the referral payload is
self-describing, narrative-bearing, and persistable as a single
attested artifact (e.g., for record-keeping and audit at the
referral-receiver site).

#### Justification

If the BSeR work group genuinely uses the Composition's `author`,
`attester`, `date`, narrative, and `section` structure to convey
referral provenance and human-readable summaries to receiving
clinicians, then the Composition is doing real work and a bare
`batch-response` Bundle would lose that semantics. Removing the
profile is also a breaking change that invalidates existing example
instances and any conformant implementations. Declining is defensible
if the work group can point to deployed BSeR users that depend on the
document structure.

---

### Recommendation

**Recommended disposition:** B

The reporter's analysis is technically sound — `Composition` is
genuinely a poor fit for system-to-system query/response payloads, and
the IG would be cleaner without it (favoring Disposition A on technical
merits). However, BSeR has been published and exchanged in real
implementations under the Public Health umbrella, and an outright
deletion of a top-level profile (and the corresponding examples) is a
breaking change that warrants a deprecation window. Disposition B
captures the same end-state as A — the Composition profile goes away
and `Bundle` (batch-response or similar) becomes the canonical payload
— while giving implementers one release cycle to migrate and giving
the work group an opportunity to confirm whether any deployed users
actually rely on Composition-level semantics. If the work group
confirms no implementer dependency on the Composition profile, fall
back to Disposition A and remove it immediately. Disposition C should
only be selected if the work group can articulate a concrete
human-narrative or attestation use case that the Composition is
serving today.
