## Summary

The reporter (on behalf of TSMG, via imported Comment 9 from Ron G. Parker) asks
the Maternal and Infant Health Research (MIHR) FHIR IG to move the canonical URL
of the `PregnancyProcedureDelivery-CPT` value set out of the
`http://hl7.org/fhir/us/mihr/...` namespace and into the
terminology.hl7.org (THO) namespace. The general guidance cited is that only
value sets that are required bindings to a `code` data type should remain
anchored under `hl7.org/fhir`; all other value sets should live in THO. The
ticket also calls for a corresponding UTG ticket to register the value set in
THO. No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the lookup, so context is limited to the ticket itself and the
two reopen comments from Lynn Laakso.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the value set canonical from
`http://hl7.org/fhir/us/mihr/ValueSet/PregnancyProcedureDelivery-CPT` to
`http://terminology.hl7.org/ValueSet/PregnancyProcedureDelivery-CPT` (or the
THO-assigned slug), file a companion UTG ticket against `HL7/UTG` to register
the value set under THO, and update every binding in MIHR that references the
old canonical (profiles, extensions, IG narrative, examples, comparison
tables) to point at the new THO canonical. Add a `predecessor` / retirement
note on the old canonical in the IG so existing implementers can follow the
move. Coordinate publication so the THO release containing the value set is
available before (or simultaneously with) the next MIHR release.

#### Justification

This matches the standing TSMG guidance the ticket cites: only value sets that
are *required* bindings to the `code` data type stay under `hl7.org/fhir`;
everything else (especially terminology-driven value sets like CPT-based
procedure subsets) belongs in THO so it can be reused across IGs and managed
under UTG governance. The reopen comment from Lynn Laakso explicitly notes the
prior "Considered for Future Use" disposition was contingent on the THO
process maturing — that process is now the established path, removing the
original blocker.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the `PregnancyProcedureDelivery-CPT` value set defined in MIHR for this
publication, but (a) deprecate the `hl7.org/fhir/us/mihr` canonical with a
clear "moving to THO" note, and (b) commit, via a tracked UTG ticket
referenced from the IG, to migrating the value set to THO in the next
substantive MIHR release. In the meantime, mirror the value set definition in
THO with the same content but a THO canonical, and have MIHR's bindings
continue to reference the IG canonical for STU-stability. Implementers get a
single source of truth in THO for downstream reuse without breaking existing
MIHR conformance resources.

#### Justification

A canonical-URL move is a conformance-affecting change: any system validating
against the current MIHR canonical will need to be updated, and any derived
profile or external IG that bound to the old URL has to follow. Doing the
relocation in two steps (publish-in-THO → swap-bindings) lets TSMG and the
Public Health WG honour the TSMG guidance without forcing implementers
through a hard URL break in a maintenance/STU-update cycle, and gives time to
align with whatever comparable "anchor in THO" tickets are filed against the
sibling MIHR value sets.

---

### Disposition C: Decline

#### Proposal

Reject the change for this version of MIHR on the grounds that the value set
is purely an IG-local subset of CPT used by a single IG, that no cross-IG
reuse has been requested, and that moving the canonical after STU publication
introduces unnecessary churn for implementers. Document the TSMG guidance in
the IG's terminology section as an aspirational future direction but leave
the existing canonical in place.

#### Justification

The ticket has sat at "Considered for Future Use" once already because the
THO process was not mature, and the only ongoing activity since is the 2023
reopen note. If MIHR has no near-term cross-IG reuse driver and no other
value set in the IG is being relocated in this cycle, a one-off move imposes
implementer cost (binding updates, validator-cache invalidation, derived-IG
churn) for limited near-term benefit. Decline is defensible as long as the IG
narrative records the TSMG direction and the value set is flagged for THO
relocation in a future major version.

---

### Recommendation

**Recommended disposition:** B

The TSMG direction the ticket cites is correct policy and should be honoured
— Disposition C ignores standing terminology governance — but a clean
single-step canonical move (Disposition A) on a published STU is unfriendly
to implementers and difficult to coordinate without UTG release alignment
already locked in. Disposition B preserves the policy intent (the value set
*does* move to THO and *is* registered via UTG), satisfies Lynn Laakso's
reopen rationale (the THO process is now mature enough to act), and gives
MIHR a controlled deprecation window so that implementers and any derived
profiles can re-bind in a subsequent release without an unannounced break.
The Public Health WG should also confirm with TSMG whether the sibling
"… 1" through "… N" tickets on this IG should be batched into the same UTG
submission, since the cost of one coordinated migration is much lower than
the cost of N separate ones.
