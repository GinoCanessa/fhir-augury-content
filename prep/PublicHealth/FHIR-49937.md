## Summary

Saul Kravitz reports that the Vital Records Common Profiles Library (VR
Common Library) "Partial Dates and Times" guidance at
`usage.html#partial-dates-and-times` documents how to convert a FHIR
`dateTime` (which carries a time zone whenever a time component is
present) **into** an IJE (Inter-Jurisdictional Exchange) record, but
does not address the inverse case. IJE has no representation for time
zone, and FHIR `dateTime` requires a time zone whenever a time of day is
included, so a fully-specified IJE date+time cannot be losslessly
expressed in a FHIR `dateTime`. The ticket asks the editors to extend
the narrative — specifically the example table on that page — to show
that when an IJE source provides a date plus a time but no time zone,
the value should be carried in the existing **Partial Date Time
Extension** (e.g., `2023-12-23-1328`) rather than synthesised into a
`dateTime` with an invented zone. A proposed redline of the table (with
the new "(none)" FHIR `dateTime` row plus the populated
year/month/day/time extension components) is included in the
description. No cross-referenced Jira tickets, Zulip threads, or GitHub
items were returned by the data tools.

## Details

**Description:**

The current documentation of the use of partial dates and times on
<https://hl7.org/fhir/us/vr-common-library/usage.html#partial-dates-and-times>
touches on the conversion of FHIR date times to IJE, but does not
provide guidance about converting IJE times to FHIR. IJE does not have
a way to express time zone. FHIR Date times require a timezone if a
time is included. So, there is no way to express an IJE time in FHIR.

The new text is highlighted in red.

| FHIR dateTime | Partial Date Time Extension Components YYYY-MM-DD TTTT | IJE YYYY | IJE MM | IJE DD | IJE TT | Description |
|---|---|---|---|---|---|---|
| 2023-12-23 T13:28:17-05:00 | (none) | 2023 | 12 | 23 | 1328. (uses local time) | Correct usage **if data originates in FHIR and Time Zone is known** |
| **(none)** | **2023-12-23-1328** | **2023** | **12** | **23** | **1328. (uses local time)** | **Correct usage for converting IJE content where time zone is not known to FHIR.** |

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the "Partial Dates and Times" section of the VR Common Library
usage page (`input/pagecontent/usage.md` — i.e. the source for
`usage.html#partial-dates-and-times`) so that the example table matches
the redline in the ticket. Concretely:

1. Add a leading sentence to the section explaining the asymmetry: FHIR
   `dateTime` requires a time zone whenever a time component is present,
   while IJE has no time-zone field, so any IJE record that carries
   YYYY/MM/DD/TT but no zone cannot be round-tripped through a FHIR
   `dateTime`.
2. Add the second row from the ticket's table (FHIR `dateTime` =
   `(none)`, Partial Date Time Extension = `2023-12-23-1328`, IJE
   components `2023 / 12 / 23 / 1328`) and label it "Correct usage for
   converting IJE content where time zone is not known to FHIR."
3. Tighten the existing first row's Description column to read "Correct
   usage if data originates in FHIR and Time Zone is known", matching
   the ticket's redline.
4. Add one or two sentences immediately under the table reminding
   implementers that the Partial Date Time Extension already defined in
   this IG is the canonical home for the date+time-without-zone case,
   and that a synthetic time zone (e.g., assuming UTC or jurisdiction
   local) MUST NOT be invented when converting from IJE.

No StructureDefinition, ValueSet, or example resource changes are
required — this is a narrative-only change to one usage page.

#### Justification

The reporter has accurately identified a gap: the existing table only
illustrates the FHIR → IJE direction, which leaves implementers
building IJE → FHIR converters with no documented landing spot for the
IJE TT field when the originating system has no zone information. The
extension the disposition points to (`PartialDateTime`) is already a
first-class artifact of this IG, so no normative or schema change is
needed — only a clearer narrative and an additional example row.
Accepting the change as requested is also the lowest-risk option: it
documents existing behaviour, does not affect conformance, and matches
the redline the reporter has already drafted, minimising editorial
back-and-forth.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying need but broaden the fix beyond the single table:

1. Add a short, named subsection — e.g., "IJE → FHIR conversion of
   dates and times" — under "Partial Dates and Times" that states the
   asymmetry rule once, lists the four IJE → FHIR cases (date-only,
   date+time with known zone, date+time with unknown zone, time-only),
   and shows the canonical FHIR representation for each (plain `date`,
   plain `dateTime`, `PartialDateTime` extension on a `date` element,
   `PartialDateTime` time component only).
2. Replace the current single example row with a small matrix that
   covers those four cases, and link each row to the relevant element
   of the `PartialDateTime` extension definition.
3. Add a reciprocal "FHIR → IJE conversion" subsection that captures
   the truncation rules already implied by the existing row, so the two
   directions are documented symmetrically.
4. Add a "MUST NOT invent a time zone" implementer note as in
   Disposition A.

#### Justification

The reporter's redline solves the immediate ambiguity, but the
underlying problem — that the page only documents one direction and
only one example — will resurface for date-only and time-only IJE
fields as soon as another implementer hits them. Spending the editorial
cycles once to produce a directional, case-based table avoids a stream
of follow-up tickets and turns the section into a true conversion
reference. The cost is modest (still narrative-only, no schema work)
and the result is more useful to the IJE bridging tools that the VR
community is actively building. The trade-off versus Disposition A is
slightly more editorial scope and review time.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive — already addressed by existing mechanism**.
The `PartialDateTime` extension and its descriptive text already cover
the case where a FHIR `dateTime` cannot be populated because a time
zone is unknown; converters between IJE and FHIR are expected to
consult the extension definition rather than rely on the usage-page
example table. Close without spec change and direct the reporter to
the `PartialDateTime` StructureDefinition narrative.

#### Justification

A defensible (if minimalist) reading of the current IG is that the
`PartialDateTime` extension definition is the normative home for the
"date+time without zone" case, and that the usage page is illustrative
only. Under that reading, the omission the ticket flags is a
documentation gap rather than a correctness gap, and the workgroup
could choose to absorb the request by clarifying the extension's own
text in a future cycle. This option is least disruptive to the current
ballot/publication queue.

---

### Recommendation

**Recommended disposition:** A

The ticket reports a real, narrowly-scoped documentation gap on a
single usage page, and the reporter has supplied a complete redline
including the additional table row and the qualifier on the existing
row. Disposition A applies that redline verbatim, which is the cheapest
way to remove the ambiguity for IJE → FHIR converter authors and
carries effectively zero conformance risk. Disposition B is attractive
in principle but is a larger editorial effort with no evidence in the
ticket or in cross-referenced discussions that the workgroup currently
has appetite for a full directional rewrite; it would be a reasonable
follow-on ticket once the immediate fix lands. Disposition C is hard to
defend: the reporter has demonstrated that a careful reader of the
usage page is genuinely misled about how to represent an IJE time
without a zone, and pointing them at the extension definition does not
remove the misleading example.
