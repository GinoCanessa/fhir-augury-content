## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Open a BSeR design item to **replace the current Batch Query mechanism
for payload definition with a profile-driven approach**: define one
`StructureDefinition` per supported referral payload (one per BSeR use
case â€” e.g., diabetes prevention, hypertension, early childhood nutrition,
arthritis, obesity, tobacco cessation), and have the referral request
identify the desired payload by `Bundle` profile URL on the
`ServiceRequest`/`Task`. The Batch Query operation would be retired in
favour of either a standard FHIR fetch (`Bundle` returned via the existing
referral fulfilment `Task`) or a `$everything`-style operation scoped by
the target patient/encounter and constrained to the requested payload
profile.

#### Justification

This aligns BSeR with how most FHIR IGs define data payloads (profile +
constraint, not bespoke query language), makes payloads validatable with
standard FHIR validators, and removes the need for implementers to learn
a BSeR-specific query syntax. It also lets implementers reuse existing
US Core profiles inside the payload Bundles, which reduces the
implementation burden for systems already conformant to US Core.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the Batch Query *concept* but **re-express it as a `Questionnaire` /
`PlanDefinition` pair** that the referral recipient renders to know
which data elements to return:

- A `PlanDefinition` per BSeR use case enumerates the required and
  optional resources (e.g., `Observation:LOINC|...`,
  `Condition:SNOMED|...`, `MedicationStatement:RxNorm|...`).
- The referral request points to the `PlanDefinition` via
  `ServiceRequest.instantiatesCanonical`.
- The recipient assembles the response `Bundle` by walking the
  `PlanDefinition.action[]` entries and including matching resources
  for the patient.

This keeps the dynamic, query-like nature of Batch Query (the request can
ask for a *subset* of a use case's full payload) while replacing the
custom syntax with standard FHIR knowledge artifacts.

#### Justification

`PlanDefinition`/`Questionnaire` are the FHIR-standard way to describe
"what data should be collected/returned for this clinical workflow."
Several CDS-Hooks, CQF, and Da Vinci IGs already use this pattern, so
BSeR implementers gain transferable skills and tooling. It preserves the
flexibility that Batch Query offers (per-request payload tailoring) that
a pure profile-only approach (Disposition A) would lose.

---

### Disposition C: Decline

#### Proposal

Decline the request as **insufficiently specified**. The ticket asks the
workgroup to "consider other options" without naming a concrete defect
in the current Batch Query design or proposing a specific alternative.
Close as `Not Persuasive` (or `Persuasive with Mod` deferred) with a
note inviting the reporter to file a follow-up ticket that (a) describes
a concrete problem with the current Batch Query mechanism and
(b) proposes at least one specific alternative for evaluation.

#### Justification

Triaged Comment-type tickets that lack a concrete proposal typically
generate open-ended discussion without producing actionable spec
changes. Replacing the payload-definition mechanism in BSeR is a
breaking change for existing implementers, so the bar for accepting
such a change should be a documented problem and a documented
alternative â€” neither of which this ticket supplies. Declining now,
with an explicit invitation to refile, focuses workgroup time on
tickets that can be dispositioned.

---

### Recommendation

**Recommended disposition:** B

The ticket is real (BSeR's Batch Query mechanism is a recurring
discussion point), so an outright Decline (C) is likely to push the
problem rather than resolve it. A full profile-only rewrite (A) is
cleaner architecturally but loses the per-request tailoring that BSeR
implementers rely on and would be a hard breaking change. Disposition B
preserves the dynamic-payload behaviour the BSeR community values, moves
the mechanism onto well-understood FHIR knowledge artifacts
(`PlanDefinition` / `Questionnaire`), and gives the workgroup a concrete
path to prototype before committing to the larger surgery in A. If the
prototype shows that per-request tailoring is rarely used in practice,
the workgroup can then collapse to Disposition A in a future cycle.
