## Summary

The reporter is asking a clarifying question about the BSeR (US Bidirectional
Services eReferrals) IG: whether a referral can be represented by more than one
`Task` resource (for example, intermediate progress / status steps in addition
to the initiating "start" Task and a terminal "end" Task), and if so, how those
additional Tasks are linked to the start and end Tasks. The ticket reads as a
request for guidance and/or a documentation/profile clarification rather than a
specific change to a structure definition. No cross-referenced Jira tickets,
Zulip discussions, or GitHub items were surfaced through the FHIR Augury index
for this ticket, so the request stands on its own as a documentation/guidance
clarification request against BSeR.

## Details

**Description:**

> Can a referral task have multiple tasks (e.g. progress steps and status)? If
> so, how are these placed in relationship with the start and ending task?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add explicit guidance to the BSeR IG narrative (and, where appropriate, to the
BSeR Task profile(s)) describing how multiple Tasks belonging to a single
referral are chained together. Concretely:

- Document a recommended pattern in the BSeR "Workflow" / "Use Cases" page that
  uses `Task.partOf` to link intermediate progress / status Tasks to the
  originating referral Task (the "start" Task), and `Task.basedOn` from the
  terminal "end" Task back to the start Task.
- Use `Task.businessStatus` (in addition to `Task.status`) on each Task to
  convey BSeR-specific progress states (e.g., received, scheduled,
  in-progress, completed, cancelled), and define or reference a value set for
  those business statuses.
- Add narrative + an example bundle showing a referral with: 1 start Task, N
  intermediate Tasks (each `partOf` the start Task), and 1 end Task that
  references the start Task via `basedOn` (or `partOf`, depending on the
  pattern chosen) and carries the final outcome.

#### Justification

The reporter is asking how to model something that the base FHIR Task resource
already supports — `Task.partOf`, `Task.basedOn`, `Task.focus`, and
`Task.businessStatus` are the standard chaining elements. The fact that the
question was raised at all indicates the BSeR IG does not currently make this
pattern obvious. Documenting the recommended pattern (rather than inventing a
new mechanism) keeps BSeR aligned with how Workflow / FHIR-I uses Task in
other IGs, gives implementers a deterministic answer, and resolves the ticket
without changing any cardinalities or invariants.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding narrative, define a small BSeR-specific extension or
profile-level constraint that captures the start/intermediate/end role
explicitly:

- Define a code system + value set (e.g., `bser-task-role` with codes
  `start`, `progress`, `end`) and bind it to either `Task.businessStatus` or
  to a new `Task.meta.tag` slice on the BSeR Task profile.
- Profile `Task.partOf` on intermediate / end Tasks to be `1..1` and constrain
  the target to the BSeR start Task profile, so that conformance tooling can
  validate the chain mechanically.
- Provide example bundles, as in Disposition A, plus a conformance invariant
  that an "end" Task must reference exactly one Task tagged as "start".

#### Justification

This approach goes beyond clarification and gives implementers (and
validators) a machine-checkable way to express the intent of each Task in a
referral. It is more invasive than Disposition A — it adds new BSeR
artifacts (value set + profile constraints) and may require ballot-level
review — but it removes the ambiguity that prompted the question in the first
place and produces a stable contract that downstream systems can rely on
without re-reading narrative. The trade-off is increased authoring/maintenance
burden in the BSeR IG and the usual risk that overly tight cardinality
constraints (`partOf` 1..1) break legitimate-but-uncommon flows.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Answered / Not Persuasive** on the grounds that base
FHIR Task already provides `Task.partOf`, `Task.basedOn`, `Task.focus`, and
`Task.businessStatus` for exactly this purpose, and BSeR does not need to
restate that. Direct the reporter to the FHIR R4 Task resource definition and
the FHIR Workflow module for the canonical chaining pattern.

#### Justification

The question is purely a "how do I use FHIR Task?" question. It does not
identify a defect in any BSeR profile, value set, example, or narrative
section, and no community discussion (Zulip, GitHub) has been linked that
demonstrates broader confusion. Declining keeps the BSeR scope tight and
avoids spending ballot cycles on what is effectively user education that the
base spec already covers.

---

### Recommendation

**Recommended disposition:** A

The ticket is a real-world implementer question raised against a Public Health
IG, and the fact that it was filed at all is evidence that the BSeR IG does
not currently make the multi-Task chaining pattern obvious. Disposition A
(documentation + example) gives implementers a clear, IG-specific answer
without taking on the maintenance and ballot cost of new conformance
artifacts that Disposition B would require, and it is materially more helpful
than Disposition C, which leaves the next implementer to ask the same
question. Disposition B remains a reasonable follow-up if, after the
narrative+example land, the workgroup sees continued divergence in how
implementers chain referral Tasks; at that point the role codes / `partOf`
constraints can be added against a known baseline.
