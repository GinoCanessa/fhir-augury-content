## Summary

The reporter is requesting that the **Birth Sex** element in the US
Central Cancer Registry Reporting Content IG (US CCRR) be tightened
from its current "Not required, Must Support" handling to a
cardinality of **1..1** with **dataAbsentReason allowed (`mustSupport
= true`)**, on the grounds that sex-at-birth is a critical data point
for cancer-incidence reporting and that any ambulatory EHR submitting
a cancer report can reasonably be expected to know it (or to be able
to assert "unknown" via dataAbsentReason). A follow-up comment from
the submitter (entered on behalf of a NAACCR community member) makes
the same argument more plainly: the current "Not required, Must
Support" status is confusing, and the element should be **required
and not absent**, with an explicit "unknown" valid value when it is
truly not known.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the CCRR IG **Patient** profile (the profile that scopes the
patient who is the subject of the cancer report), change the
constraint on the **US Core Birth Sex extension**
(`http://hl7.org/fhir/us/core/StructureDefinition/us-core-birthsex`)
from its current "Must Support, optional" treatment to:

- `extension:birthsex` cardinality = **1..1**
- `mustSupport = true`
- Allow `dataAbsentReason` on the extension's `value[x]` (or, if the
  US Core extension does not natively expose `value.dataAbsentReason`,
  bind a slice that admits an `unknown` code from
  `http://terminology.hl7.org/CodeSystem/v3-NullFlavor` /
  `data-absent-reason`) so that "truly unknown" is representable
  without violating the new 1..1.

Update the IG narrative for Birth Sex to state explicitly that
senders MUST populate the element on every cancer-report bundle, and
that `unknown` is the only acceptable way to assert absence.

#### Justification

- Matches the submitter's stated need exactly: cancer-incidence
  analysis is stratified on sex-at-birth, and a missing value
  silently degrades the registry's downstream statistics.
- The IG is profile-only over US Core; tightening cardinality on a
  US-Core-defined extension in a downstream IG is an established
  pattern (the same pattern is used elsewhere for race/ethnicity
  in public-health IGs).
- Allowing `dataAbsentReason` preserves the "truly unknown" case the
  reporter explicitly called out in the second comment, so 1..1 does
  not force senders to invent a value when none is known.
- The change is profile-tightening only; it does not change US Core
  itself, so blast radius is contained to CCRR consumers and
  conformant senders.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the underlying extension cardinality at **0..1 Must Support**
on the profile, but add an **invariant** on the CCRR Patient profile
that says, in effect, "either `extension:birthsex.value[x]` is
populated, or `extension:birthsex.value[x].extension:dataAbsentReason`
is populated with a code from `data-absent-reason` (preferred:
`unknown`)." Phrase the invariant so it is enforceable by validators
(`expression`: `extension('…us-core-birthsex').value.exists() or
extension('…us-core-birthsex').value.extension('…dataAbsentReason').exists()`).
Strengthen the IG narrative and conformance text to require senders
to make a conscious decision (assert a value or assert
data-absent-reason) rather than silently omit the element.

#### Justification

- Achieves the registry's analytic goal — every report must say
  *something* about birth sex — without setting `min = 1` on an
  extension that some upstream EHR data models still treat as
  optional, which can cause spurious validation failures during the
  early conformance period.
- An invariant provides a clearer error message at validation time
  than a raw cardinality violation ("birth sex must be present or
  data-absent-reason must be given"), which is friendlier to the
  ambulatory EHRs the submitter specifically called out.
- Lower migration burden for senders that today produce CCRR bundles
  without any birth-sex element: they can add `dataAbsentReason =
  unknown` once and stay conformant while they fix their data
  pipelines.
- Keeps the downstream IG's relationship to US Core asymmetric only
  where it has to be (constraint vs. cardinality change on an
  upstream artifact).

---

### Disposition C: Decline

#### Proposal

Leave the Birth Sex element at its current "Must Support, optional"
treatment in the CCRR IG. Update the IG narrative only, to clarify
that "Must Support" already obliges receivers to be able to handle a
populated value and obliges senders to populate it whenever the
information is available in the source system, and that registries
should treat a missing element as "unknown at the source." Close
the ticket as "Persuasive with Modification" only on the narrative
clarification.

#### Justification

- US Core 6+ has been deliberate about *not* making sex-at-birth
  required at the base level because real-world source systems
  (especially in ambulatory care) sometimes genuinely lack the data,
  and forcing 1..1 pushes those systems either to fabricate values
  or to fail validation outright.
- "Must Support" is the conformance lever specifically designed for
  this case: senders SHALL be able to provide the element when they
  have it; receivers SHALL be able to consume it. Tightening
  cardinality on top of Must Support arguably duplicates intent
  while adding rigidity.
- Defensible as "out of scope for this IG, addressed by existing
  US Core mechanism" if the workgroup judges the registry's analytic
  need is better served by a data-quality scoring approach than by
  spec-level cardinality.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — invariant
requiring value or dataAbsentReason).

Disposition B captures the submitter's actual goal — "every cancer
report must say something about birth sex, even if that something is
'unknown'" — without taking the harder step of pinning a
US-Core-defined extension to `min = 1` in a downstream IG. The
submitter's own follow-up comment is the strongest argument for B:
they explicitly want a valid "unknown" representation, which is
exactly what `dataAbsentReason` provides. An invariant gives the
registry the analytic guarantee it needs, gives ambulatory EHRs an
honest escape hatch when the data really is missing, and produces a
clearer validator error than a bare cardinality violation. If the
workgroup later concludes that almost all senders are using the
escape hatch routinely, escalating from B to A is straightforward in
a future ballot cycle; going the other direction (relaxing 1..1 once
implementers have tooled to it) is much harder.

Disposition A is the right answer if Public Health WG is comfortable
constraining the US Core extension's cardinality in the CCRR
profile and is prepared to defend that choice during ballot
reconciliation; it is a defensible second choice. Disposition C
(decline) is only appropriate if the WG concludes that the
registry's analytic need is already met by Must Support plus
data-quality reporting outside the spec — a position that is hard to
hold given the submitter's explicit, NAACCR-sourced statement that
the current treatment is causing real confusion.
