## Summary

Eric Haas reports that the R6 type change for `Immunization.protocolApplied.doseNumber` and `Immunization.protocolApplied.seriesDoses` — moving them from `string` (R5) / `positiveInt|string` (R4) to `CodeableConcept` — is over-engineered and semantically inappropriate. Dose numbers are inherently ordinal positions in a sequence (1, 2, 3, …) or short descriptive labels ("booster", "catch-up"), and `seriesDoses` is a count of recommended doses; neither is naturally a coded concept. The `CodeableConcept` choice further implies the existence of a standardised code system, but the bindings are only `Example` and the referenced value sets (`DoseNumber`, `SeriesDoses`) appear to be undefined. The reporter predicts that virtually all implementers will simply populate `CodeableConcept.text` with "1", "2", "3", which adds parsing burden without any interoperability gain. The proposal is to revert the type to `string`; implementers who do want to use codes can still bind a `string` to a value set. The single comment from `marti_velezis` (2026-03-23) confirms the ticket was filed against the wrong work group (Devices) and re-routes it to Public Health, which owns the Immunization family of resources. There are no cross-referenced Jira tickets, Zulip threads, or GitHub items recorded in Augury for this ticket.

## Details

**Description:**

### `protocolApplied.doseNumber` Type Change - Breaking and Semantically Odd

Changed from R5's `string` (previously `positiveInt|string` in R4) to `CodeableConcept` in R6.

This is semantically problematic:

- Dose numbers are typically ordinal (1, 2, 3...) or descriptive ("booster", "catch-up")
- `CodeableConcept` implies there's a standardized code system for dose numbers
- The binding is only "Example" with no actual value set specified (`Binding: DoseNumber (Example)` — but the value set appears to be undefined/missing)

**Issue:** Using `CodeableConcept` for what is fundamentally a position in a sequence is over-engineering. Most implementations will just put "1", "2", "3" in `text` with no coding.

### `protocolApplied.seriesDoses` - Same Issue as doseNumber

Changed from `string` to `CodeableConcept`. The definition says: *"The recommended number of doses to achieve immunity."*

A count (e.g., "3 doses") is not naturally a coded concept. The binding shows `SeriesDoses (Example)` but again appears to lack a defined value set.

Propose using reverting type to string. If an implementer chooses to use a set of codes, string can be bound to them.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revert the type of `Immunization.protocolApplied.doseNumber`, `Immunization.protocolApplied.seriesDoses`, and the parallel elements on `ImmunizationEvaluation` (`doseNumber`, `seriesDoses`) and `ImmunizationRecommendation.recommendation` (`doseNumber`, `seriesDoses`) from `CodeableConcept` back to `string` for R6. Remove the `DoseNumber` and `SeriesDoses` example bindings (or keep them as informative, attached to the `string` element), and update the element definitions and short descriptions to reflect that the value is a free-text positional/ordinal label or count. Note in the resource changelog and the Immunization "Changes since R5" section that this is a breaking change relative to the current R6 ballot draft and a partial revert toward the R5 representation.

#### Justification

Aligns with the reporter's analysis: the element semantics are ordinal position / count, not concept membership. Reverts an R6-only breakage rather than perpetuating it through a normative cycle, and removes two phantom value sets (`DoseNumber`, `SeriesDoses`) that have no defined content. `string` preserves the ability to express "1", "2", "booster", "catch-up" exactly as R5 already supported, and matches the de-facto pattern implementers will adopt anyway (`CodeableConcept.text` with no coding). Lower implementer burden than `CodeableConcept`, and consistent with the historical R4 choice of `positiveInt | string` (this option intentionally drops `positiveInt` to keep the cardinality of types at one, matching R5).

---

### Disposition B: Alternative Approach

#### Proposal

Keep the underlying intent of allowing structured dose identification, but replace `CodeableConcept` with a narrower choice type `positiveInt | string` (the R4 model) on all six elements. Define `positiveInt` as the canonical representation for sequential doses ("1", "2", "3") and `string` as the fallback for non-numeric labels ("booster", "catch-up", "series complete"). Drop the empty `DoseNumber` / `SeriesDoses` value sets; if Public Health later identifies a real coded vocabulary for dose labels, expose it via an extension on the `string` (or the `positiveInt` element via an `iso21090-int-codedValue`-style extension) rather than promoting the base type to `CodeableConcept`.

#### Justification

Restores the most expressive prior model (R4) and gives implementers an unambiguous typed slot for the common ordinal case while still permitting descriptive labels. Avoids the trap, identified in the ticket, of advertising a coded concept where no code system exists. The trade-off vs. Disposition A is added implementer complexity (a choice type with `[x]` rename of the elements) and the loss of strict R5-compatibility on the wire; the gain is type-checked numeric doses, which several immunization registries already validate today. Use of an extension for the rare "coded dose" use case is consistent with FHIR's general guidance to prefer extensions over upgrading core types when the use case is narrow.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain `CodeableConcept` for `doseNumber` and `seriesDoses` on `Immunization`, `ImmunizationEvaluation`, and `ImmunizationRecommendation.recommendation`. Address the reporter's concern about the empty bindings by either populating the `DoseNumber` and `SeriesDoses` value sets with at least an indicative set of codes (numeric ordinals plus common labels such as "booster", "catch-up", "series complete") or by re-stating the example binding as informative, with explicit guidance that `CodeableConcept.text` is acceptable when no code applies.

#### Justification

`CodeableConcept` is a strict superset of both `string` (via `text`) and a coded value, so no implementer is forced to use a code. The R5 → R6 change was made deliberately to allow registries that already have local code systems for dose identification (catch-up schedules, series identifiers, booster nomenclature) to express them in a structured way without resorting to extensions. Reverting at this stage compounds churn for implementers who have already begun migrating toward R6, and the underlying "missing value set" complaint is fixable without a type change. This option also preserves wire-format consistency with the rest of FHIR R6 where coded clinical concepts use `CodeableConcept`.

---

### Recommendation

**Recommended disposition:** A

The reporter's argument is sound and aligns with how Immunization has historically modelled these elements (R5 used `string`; R4 used `positiveInt | string`). The R6 promotion to `CodeableConcept` is paired with example bindings whose value sets are not defined, which is a strong signal that no coded vocabulary actually exists and that the type is over-specified. `CodeableConcept` for an ordinal position invites every implementer to either invent a local code system or — far more likely — populate `text` with the digits, both of which are worse than just sending a `string`. Disposition B (restore the R4 `positiveInt | string` choice) is attractive on technical grounds but adds the cost of a choice type and breaks R5 producers and consumers without a strong corresponding gain; Disposition C papers over the empty-binding problem without addressing the semantic mismatch the reporter raised. Public Health should accept the change request, revert the six elements to `string`, and document the change clearly in the R6 Immunization changelog so that any work-in-progress implementations of the current ballot draft can be adjusted before publication.
