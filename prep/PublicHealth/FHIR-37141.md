## Summary

The ticket asserts that the canonical URL for the Maternal and Infant Health
Research (MIHR) IG's `PregnancyNonLive` value set is currently anchored under
`hl7.org/fhir` (specifically
`http://www.hl7.org/fhir/us/mihr/ValueSet/PregnancyNonLive`) and should
instead be anchored under `terminology.hl7.org` (THO). The reporter (on
behalf of TSMG / Ron G. Parker, per the imported "Comment 6" provenance)
states that only value sets bound as a *required* binding to a `code` data
type should remain anchored under `hl7.org/fhir`; everything else should
live in THO. A companion UTG ticket would be required to register the value
set in THO. No cross-referenced Jira tickets, Zulip threads, or GitHub
artifacts were surfaced for this ticket, so the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the canonical URL of the `PregnancyNonLive` value set out of the
MIHR IG namespace and into THO. Concretely:

1. File a UTG ticket to register the value set in THO with a new
   canonical such as `http://terminology.hl7.org/ValueSet/mihr-pregnancy-non-live`
   (exact slug subject to UTG / TSMG conventions).
2. In the MIHR IG, remove the locally-defined `ValueSet/PregnancyNonLive`
   resource and replace every binding reference (currently
   `http://www.hl7.org/fhir/us/mihr/ValueSet/PregnancyNonLive`) with the
   new THO canonical.
3. Treat the old MIHR canonical as retired; if backwards compatibility
   is required for STU adopters, redirect / alias rather than republish
   under the IG namespace.

This applies the TSMG rule literally: because `PregnancyNonLive` is not
the value set behind a *required binding to a `code`* element, it
belongs in THO rather than under `hl7.org/fhir/us/mihr`.

#### Justification

This is the disposition that matches the explicit TSMG / Vocab guidance
quoted by the reporter and is consistent with how other HL7 IGs have
been migrating reusable value sets into THO. The previous "Considered
for Future Use" rationale (THO process maturity) was cited as the
blocking concern — Lynn Laakso's 2023 comment re-opening the ticket
post-STU acknowledges that maturity gate has now been cleared, so the
work can proceed. Anchoring shared/reusable value sets in THO also
makes them re-usable across IGs without dragging in MIHR-specific
versioning.

---

### Disposition B: Alternative Approach

#### Proposal

Address the underlying intent (THO-anchored vocabulary) but do so as a
single, IG-wide migration tracked under one umbrella ticket rather than
per-value-set tickets. Specifically:

1. Treat FHIR-37141 (and its sibling "…anchored in THO. 1..N" tickets, if
   any are filed against MIHR) as a single batch.
2. Inventory every value set in MIHR, classify each as
   *required-binding-to-code* (stays under `hl7.org/fhir/us/mihr`) vs
   *anything else* (moves to THO), and produce one consolidated UTG
   submission.
3. Apply the URL changes in MIHR as a coordinated commit so the IG's
   bindings, narrative, and IG resource list all flip together — avoiding
   a partially-migrated state.

#### Justification

The TSMG rule applies to the whole IG, not just `PregnancyNonLive`.
Migrating value sets one ticket at a time risks long-lived intermediate
states where some bindings point at THO and others still point at
`hl7.org/fhir/us/mihr`, which is harder for implementers to reason about
than a single coordinated cut-over. It also avoids re-litigating the
same disposition N times in workgroup. The trade-off vs. Disposition A
is timing: the umbrella migration is more disruptive in one release but
cleaner over the lifetime of the IG.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the canonical
`http://www.hl7.org/fhir/us/mihr/ValueSet/PregnancyNonLive` as-is.
Rationale category: *breaking change to a published canonical URL*.

#### Justification

The MIHR IG has reached STU publication, and changing a value set's
canonical URL is a breaking change for any conformance resource (profile,
StructureDefinition binding, instance) that references it. If no STU
implementers are known to be relying on the existing canonical, the harm
is small — but if any are, declining preserves stability. This
disposition is defensible if the workgroup judges the THO-anchoring
benefit to be insufficient to justify a normative URL change at this
stage of the IG's lifecycle.

---

### Recommendation

**Recommended disposition:** B

The TSMG guidance the ticket invokes is a generic rule about *all*
non-`code`-bound value sets, not a `PregnancyNonLive`-specific concern.
Doing this work one ticket at a time (Disposition A) will repeatedly tax
Public Health WG meetings with near-identical votes and risks an MIHR
release in which the binding namespaces are inconsistent. Disposition C
treats canonical-URL stability as paramount, but the ticket has been
explicitly re-opened post-STU on the understanding that the THO process
is now mature enough to move on this — declining would re-create the
exact "wait for THO maturity" state that already justified one revert.
The clean path is to bundle FHIR-37141 with its siblings, do one
inventory + UTG submission + MIHR migration, and ship the new bindings
together. If the workgroup is uncomfortable with the batch-migration
scope, fall back to Disposition A and accept the per-ticket overhead.
