## Summary

The reporter (Lloyd McKenzie) questions an implicit architectural
assumption in the US Bidirectional Services eReferrals (BSeR) IG:
namely, that a referral response can be satisfied by a single request
payload. The ticket points out that real-world referral workflows
frequently involve follow-up questions and drill-down requests — for
example, a surgery report that references another report the recipient
now wants to see, or an imaging report that was "based on" an image
that was not originally transmitted but is now needed. The ticket asks
the BSeR workgroup to describe (and/or specify) how such follow-up /
drill-down interactions are handled in the BSeR exchange pattern. It
is framed as a Question rather than a defect, but the "Highest"
priority and the `BSeR_FOLLOWUP` label indicate the workgroup intends
to address it. No cross-referenced Jira tickets, Zulip threads, or
GitHub items were surfaced for this ticket.

## Proposed Dispositions

The BSeR IG is not surfaced by any GitHub cross-references in the
Augury cache, and no `repo-analysis` briefing is therefore loaded. The
dispositions below are phrased in IG-authoring terms (narrative pages,
Task / ServiceRequest / Communication profiles, exchange-pattern
diagrams) rather than naming specific file paths. If the workgroup
decides to act, the BSeR IG repository should be added to the Augury
cache and a `repo-analysis` briefing produced so that a follow-up
implementation plan can point at concrete authoring roots.

### Disposition A: Accept As Requested

#### Proposal

Extend the BSeR IG to explicitly describe and profile a *follow-up /
drill-down* exchange pattern, layered on top of the existing
request-response flow. Concretely:

1. Add a narrative section ("Follow-up and drill-down requests") to the
   BSeR IG exchange-pattern page describing when and how a recipient of
   a referral payload may request additional information after the
   initial response.
2. Profile `Task` (or, if the workgroup prefers, `CommunicationRequest`)
   as a *BSeR Follow-up Request* that carries a reference to the
   originating referral `Task` / `ServiceRequest` and identifies the
   additional artifact being requested — either by
   `DocumentReference`/`Reference(Resource)` pointer copied from the
   prior payload, or by a free-text `description` plus
   `code`/`reasonCode` describing the need.
3. Profile the follow-up response (either a `Bundle` keyed by the
   originating follow-up `Task`, or a new `Communication` payload) that
   carries the requested resources (e.g., the referenced
   `DiagnosticReport`, `ImagingStudy`, `DocumentReference`, or `Media`).
4. Add at least two examples to the IG: (a) "request referenced surgery
   report B after receiving report A", and (b) "request ImagingStudy
   after receiving an ImagingReport that cites it".
5. Update the state diagram for the BSeR workflow so that multiple
   follow-up cycles are shown as a supported loop, not an exceptional
   path.

#### Justification

The ticket describes workflows that are not edge cases — in real
clinical referrals, responses very often reference artifacts that are
not included in the initial payload, and the recipient may need to
drill into them to complete their review. Treating this as a
first-class pattern in the IG (rather than leaving it implicit)
directly answers Lloyd McKenzie's question, improves implementer
certainty, and is consistent with how other FHIR exchange IGs (e.g.,
Da Vinci PDex, CRD, DTR) explicitly model multi-turn workflows using
`Task` chains. The `BSeR_FOLLOWUP` label on the ticket indicates the
workgroup has already identified this area as needing a follow-up
deliverable.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing new BSeR-specific follow-up profiles, add a
normative "Out-of-band retrieval" section to the BSeR IG that:

1. States explicitly that BSeR does **not** presume a single request
   payload is sufficient and that recipients MAY issue subsequent
   requests.
2. Directs implementers to use the existing FHIR RESTful API (`GET
   [base]/DiagnosticReport/B`, `GET [base]/ImagingStudy/X`) against the
   sender's FHIR endpoint to retrieve referenced-but-not-included
   resources, gated by the usual authorization (e.g., SMART
   app-launch / backend-services profile already referenced by BSeR).
3. Recommends that senders populate `Reference.reference` (absolute URL
   where possible) and/or `DocumentReference.content.attachment.url`
   so the recipient can resolve references without a new Task
   exchange.
4. Adds a short "Scenarios" subsection showing the surgery-report and
   imaging-report drill-down examples as pure REST reads, not as new
   BSeR messages.
5. Leaves `Task`-driven follow-up as future work, only to be introduced
   if REST-based retrieval proves insufficient after implementer
   feedback.

#### Justification

This approach answers the ticket without inflating the BSeR IG's
profile surface. FHIR already provides a well-defined mechanism for
"I have a reference, I want the target" — it's the REST API — and
duplicating that behind a `Task`-shaped envelope adds complexity and
yet another state machine for implementers to manage. It is also
lower-risk for the current BSeR publication cycle: it is purely
additive narrative and does not require new profiles, examples, or
conformance statements. The trade-off versus Disposition A is that
asynchronous, workflow-driven scenarios (e.g., the original referral
handler has gone off-shift and someone else must fulfil the follow-up)
are less well supported than they would be with a first-class `Task`
chain.

---

### Disposition C: Decline

#### Proposal

Close the ticket as *Not Persuasive — Out of Scope for BSeR*, with
direction that follow-up / drill-down behaviour is a general FHIR
exchange concern and not a gap that BSeR itself needs to fill.
Rationale to be recorded on the ticket:

1. BSeR specifies the referral request/response envelope; subsequent
   access to referenced artifacts is governed by the base FHIR REST
   API and by the implementer's access-control framework, which are
   already referenced by BSeR.
2. The examples given (drill into referenced report, fetch an
   `ImagingStudy` for a report that cited it) are addressable today
   using existing mechanisms (RESTful reads, `$everything`, or the
   relevant Da Vinci / USCDI data-access IGs) without requiring a
   BSeR-specific follow-up profile.
3. Adding a BSeR-specific follow-up pattern risks diverging from, or
   competing with, those more general exchange IGs.

#### Justification

Declining is defensible on scope grounds: BSeR's charter is the
eReferral *request* and *response*, not ongoing data access between
the referral parties. Other IGs in the HL7 ecosystem already address
"I have a pointer, get me the target" more comprehensively, and
building a parallel mechanism in BSeR could fragment implementer
effort. The weakness of this disposition is that it does not actually
answer the ticket's substantive question ("how is drill-down
handled?"); at minimum, even if the workgroup declines to *add
profiles*, a narrative pointer to the REST / other-IG mechanism (as in
Disposition B) is probably still warranted.

---

### Recommendation

**Recommended disposition:** B

Disposition B most directly answers Lloyd McKenzie's actual question —
"How is the drill-down process handled?" — without committing the BSeR
workgroup to a new set of profiles, examples, and state transitions on
short notice. The reporter's concern is that the IG *reads* as though
a single request payload is presumed sufficient; the remedy for that
reading is primarily narrative. Explicitly stating that BSeR does not
presume single-payload sufficiency, and pointing implementers at the
REST API (and the existing authorization machinery BSeR already
references) for retrieving referenced artifacts, closes the gap the
ticket identifies and keeps the IG's scope coherent. If, after
publishing that guidance, implementer feedback shows REST-based
retrieval is insufficient for asynchronous follow-up workflows, the
workgroup can revisit Disposition A in a subsequent cycle — at that
point ideally with a `repo-analysis` briefing of the BSeR IG repository
so specific authoring paths and profile names can be named in the
proposal. Disposition C is not recommended because it leaves the
ticket's question unanswered in the IG itself, which is the minimum
outcome the `BSeR_FOLLOWUP` label implies the workgroup wants.
