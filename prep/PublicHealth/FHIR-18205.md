## Summary

The reporter (Craig Newman) flags an internal inconsistency in the BSeR
Arthritis Observation profile: the differential/snapshot lists
`Observation.code.code` with a cardinality of `1..1`, but the hover
tooltip — driven by the underlying conformance/invariant statement
identified as `4383-426` — states that there can be `0..1` codes. The
two views of the same element disagree, which leaves implementers
uncertain whether a code value is required or optional. The reporter
suggests the cardinality should be aligned (most likely to `1..1`,
matching the profile constraint). No other Jira tickets, Zulip
discussions, or GitHub issues are cross-referenced to this ticket in
Augury, so the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR Arthritis Observation profile so that the
`Observation.code.code` element is consistently constrained to `1..1`
in **both** the profile cardinality and the underlying conformance
statement / invariant referenced in the tooltip (the rule labelled
`4383-426`). Concretely, in the StructureDefinition for the Arthritis
Observation profile:

- Set `ElementDefinition.min = 1` and `ElementDefinition.max = "1"` on
  the `Observation.code.code` slice/element (already the stated
  intent).
- Rewrite the invariant currently producing the "0 or 1 codes" hover
  text so it asserts exactly one `code` (e.g.,
  `code.coding.where(...).code.exists()` rather than `.count() <= 1`).
- Regenerate the snapshot and IG so the hover/tooltip text matches the
  cardinality.

This fulfils the reporter's literal suggestion ("These should be the
same (probably 1..1)") and removes the implementer-facing ambiguity.

#### Justification

The hover tooltip and the cardinality column are two presentations of
the same conformance contract; when they disagree, implementers cannot
tell whether a missing code is a profile violation. The reporter
explicitly recommends `1..1`, and that is the more restrictive of the
two readings, which matches the typical FHIR profiling pattern of using
the cardinality column as the source of truth and writing invariants
that reinforce — not relax — it. Aligning the invariant to `1..1`
restores internal consistency without changing what conformant systems
already have to do (since they were already obligated by the `1..1`
cardinality).

---

### Disposition B: Alternative Approach

#### Proposal

Resolve the inconsistency in the **other** direction: relax
`Observation.code.code` to `0..1` to match the existing invariant
`4383-426`, and update the snapshot/tooltip accordingly. Optionally,
move the "code SHOULD be present" guidance into a `MustSupport` flag
plus narrative guidance rather than a hard cardinality, so that BSeR
Arthritis Observations can carry a `Coding` whose `code` element is
absent (e.g., text-only, or `display` + `system` only) when the source
system genuinely lacks a coded value.

#### Justification

If the invariant `4383-426` was authored deliberately at `0..1`, that
may reflect real-world referral data where the sending system has a
display string and a code system but not a discrete code. Relaxing the
cardinality preserves that flexibility while still letting the IG
require the element via `MustSupport` and conformance prose. This
avoids a silent breaking change for any early implementer who has been
relying on the looser invariant interpretation, and it keeps BSeR
aligned with the base `Observation.code` (a `CodeableConcept`, in which
`Coding.code` is `0..1`).

---

### Disposition C: Decline

#### Proposal

Decline the change as not actionable in its current form: close as
"Not Persuasive" or "Persuasive with Modification — withdrawn pending
republish", on the grounds that the BSeR ballot/IG version that
exhibited the `4383-426` inconsistency is no longer the current
publication, and that the apparent mismatch is a stale-snapshot
artefact rather than a substantive profile defect. Ask the reporter to
re-verify against the current BSeR build and reopen if the
inconsistency persists.

#### Justification

The ticket dates from September 2018 and was last touched in May 2019;
there are no comments, no linked PRs, no Zulip discussion, and no
linked GitHub repository in Augury. BSeR has been republished since,
and the invariant numbering (`4383-426`) is the kind of identifier
that does not survive snapshot regeneration. Spending workgroup
disposition cycles on a possibly-already-fixed tooltip mismatch — with
no implementer follow-up in six years — is hard to justify when the
underlying profile may already be self-consistent in the current
build.

---

### Recommendation

**Recommended disposition:** A

The reporter's diagnosis is concrete (cardinality column vs. invariant
`4383-426` disagree on `Observation.code.code`) and the suggested fix
(`1..1` everywhere) is the least disruptive way to remove the
ambiguity: it matches what the cardinality column already advertises,
so any implementer who has been honouring the snapshot is already
compliant. Disposition B is defensible only if the workgroup has
positive evidence that BSeR Arthritis Observations legitimately need
to carry an uncoded `Coding`, and Disposition C only makes sense if
the workgroup can confirm the current published BSeR build no longer
shows the mismatch — in which case the ticket can be closed as
"Already Fixed" with a pointer to the current version. Absent that
confirmation, Accept As Requested is the cleanest path: align the
invariant to `1..1`, regenerate, and close.
