## Summary

Rick Geimer requests that all references to "gender identity" be removed
from the HL7 NHSN dQM Reporting Implementation Guide (narrative,
profiles, and examples) so that the IG aligns with what the reporter
characterizes as initial guidance issued in response to U.S. Executive
Order "Defending Women." The reporter specifically calls out the
example `Measure/NHSNdQMAcuteCareHospitalInitialPopulation` as
containing such a reference that needs to be removed, and asks
maintainers to sweep the rest of the IG to confirm that "gender
identity" does not appear elsewhere as a data element or in narrative
text. The ticket has been triaged with a resolution of **Persuasive**,
and the resolution comment indicates the work group has agreed to
remove the gender-identity references. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove every occurrence of the phrase "gender identity" from the HL7
NHSN dQM Reporting IG. Concretely:

1. Strip the `gender-identity` slice / element from the example
   `Measure/NHSNdQMAcuteCareHospitalInitialPopulation` referenced in
   the description (and any sibling example resources that include the
   same construct), regenerating the rendered JSON/XML/Turtle pages.
2. Sweep the IG sources for the literal text "gender identity"
   (case-insensitive) across narrative pages, profile element
   definitions, slicing discriminators, value-set bindings,
   `Observation`/`Patient` extension references, and example
   instances. Remove or rewrite each hit.
3. If any structure definition currently profiles a Patient or
   Observation extension carrying gender-identity (e.g., the US Core
   `us-core-birthsex` / `us-core-genderIdentity` extension), drop the
   profile reference and any `must-support` flagging in the dQM
   profiles.
4. Update the IG narrative, change log, and ballot notes to record
   that gender-identity content has been removed and to cite this
   ticket.

#### Justification

This matches what the reporter asked for and what the work group
already endorsed via the **Persuasive** resolution — the resolution
comment explicitly says "Will remove any gender identity references."
The example called out in the description is a verifiable artifact in
the published build, so the change is concrete and bounded to the dQM
IG. Because the request is scoped to a single IG (not core FHIR or a
shared terminology), the blast radius is small and the change can be
validated by re-publishing the IG and running the IG publisher.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than deleting the data element wholesale, retain the
gender-identity-bearing constructs in the underlying profiles /
examples but mark them **optional / not in scope for NHSN dQM
reporting** in this IG, and remove all *narrative* prose that
discusses or instructs on gender identity. Specifically:

1. In the dQM profiles, leave any inherited extension slots intact
   (don't actively forbid them with `0..0`), but remove any
   `mustSupport=true`, slicing, or binding that calls out gender
   identity.
2. Replace gender-identity-valued example fields with neutral
   placeholders or omit them from the example bundle, while keeping
   the example structurally valid against the underlying base
   resources.
3. Delete every narrative mention of "gender identity" from the IG's
   markdown pages, change log, and ballot notes.

This decouples the IG's *guidance* (which is what the reporter targets)
from its *technical conformance surface*, so downstream implementers
who already populate gender-identity data in their FHIR servers do not
have their resources rejected by dQM-profile validators.

#### Justification

Outright `0..0`-style removal of inherited extensions can cause
validation failures for production systems that already exchange the
underlying base resources with gender-identity content. Removing the
guidance and examples (Disposition B) accomplishes the reporter's
stated compliance goal — "gender identity" no longer appears in the IG
as something this guide tells implementers to do — without forbidding
data that base FHIR / US Core still defines. It also future-proofs the
IG: if guidance changes, only narrative needs to be re-added, not the
profile machinery.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the requested change is driven
by a U.S. executive-order interpretation rather than by a clinical,
technical, or interoperability requirement, and that HL7 IGs should
not be edited to track shifting national executive guidance,
especially when (a) "gender identity" is a recognized clinical concept
in US Core and base FHIR, and (b) NHSN itself has not (per the ticket
record) issued a formal program-level requirement to strip the field
from dQM submissions. Close as **Not Persuasive**, leaving the IG
unchanged, and direct the reporter to NHSN program guidance as the
appropriate channel.

#### Justification

This is defensible because (1) HL7 specifications normally change in
response to standing program / regulatory requirements, not interim
executive-branch guidance documents that may be revised or rescinded;
(2) other HL7 IGs (US Core, USCDI-aligned IGs) actively define and
support gender-identity elements, so a unilateral removal here creates
an inconsistency across the HL7 publication; and (3) the ticket cites
no specific NHSN reporting rule that prohibits the data — only a
political directive. Declining preserves consistency with the broader
HL7 ecosystem and avoids the IG having to be re-edited if guidance
shifts again.

---

### Recommendation

**Recommended disposition:** A

The work group has already triaged this ticket with a **Persuasive**
resolution and a stated intent to "remove any gender identity
references," so Disposition A is the path most consistent with the
decision the work group has already taken on the record. The change
is narrowly scoped to a single IG (NHSN dQM), the affected artifact
is specifically named (the `NHSNdQMAcuteCareHospitalInitialPopulation`
Measure example), and the work is straightforward to verify by
re-publishing the IG. Disposition B is the right fallback if, during
implementation, the NHSN dQM team finds that gender-identity content
is structurally entangled with inherited US Core profiles in a way
that makes hard removal disruptive to existing implementers — in that
case, narrative-only removal preserves the compliance intent without
breaking conformance for downstream systems. Disposition C is not
recommended given that the work group has already accepted the
request; raising it again at disposition time would be re-litigating
a decision rather than executing it.
