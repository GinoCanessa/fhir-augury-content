## Summary

[FHIR-21649](https://jira.hl7.org/browse/FHIR-21649) is a tracker ticket
opened against the US Bidirectional Services eReferrals (BSeR) FHIR IG
asking the Public Health work group to consider profiling
`ServiceRequest` with **slices on `supportingInfo`** so that the kinds
of supporting information attached to a referral (e.g., relevant
observations, conditions, documents, prior assessments) are constrained
and discoverable per BSeR use case rather than being a free-form
heterogeneous reference list. The ticket itself is intentionally
minimal — the description is literally "Adding a tracker item for
following this topic" — and was filed by Vassil Peytchev in May 2019,
triaged at Medium priority, and has had no activity (no comments, no
linked tickets, no Zulip cross-references) since. It has remained in
**Triaged** status for ~6 years, indicating the work group has not yet
disposed of the underlying design question.

There are no Jira, Zulip, GitHub, or Confluence cross-references for
this ticket in the FHIR Augury index, so the only signal available is
the title plus the BSeR specification context.

### Disposition A: Accept As Requested

#### Proposal

In the BSeR IG, update each use-case-specific `ServiceRequest` profile
(e.g., the per-condition referral request profiles such as the
Arthritis Referral Request, Hypertension Referral Request, etc.) to
**slice `ServiceRequest.supportingInfo`** by `resolve().meta.profile`
(or by a structurally appropriate discriminator such as
`resolve().code` for `Observation`-shaped slices). For each use case,
define named slices for the supporting clinical artifacts the
referral type requires — for example:

- `vitalSigns` — references to the BSeR vital-signs `Observation`
  profile, cardinality per use-case requirements,
- `relevantConditions` — references to the BSeR `Condition` profile(s)
  carrying the indication for the referral,
- `priorAssessments` — references to the BSeR functional/screening
  `Observation` or `QuestionnaireResponse` profiles,
- `attachments` — references to `DocumentReference` for free-form
  supporting documents,

with a final open slice (`rules: open`) so that implementers can still
attach use-case-specific extras. Each slice should set a clear
cardinality (e.g., `1..*` for required clinical context, `0..*` for
optional attachments) and reference the appropriate BSeR-profiled
target resources.

#### Justification

Slicing `supportingInfo` is the FHIR-canonical way to turn a
free-form `Reference(Any)` bag into a contract that implementers can
validate against. The BSeR IG already constrains the request itself
(category, code, performer, etc.) per use case; constraining the
supporting payload is the natural completion of that pattern and
aligns with how US Core, QI-Core, and other IGs slice
`supportingInfo` / `hasMember`. It also addresses the underlying need
visible in the keyword-related ticket [FHIR-22615](https://jira.hl7.org/browse/FHIR-22615),
which asks effectively the same question for the Arthritis referral
bundle — slicing per-use-case answers "what is in the bundle that is
unique enough to indicate it is for Arthritis."

---

### Disposition B: Alternative Approach

#### Proposal

Instead of slicing `ServiceRequest.supportingInfo`, model the
referral payload as a **constrained Bundle** (or use a profile on the
existing BSeR Referral Request Bundle if one exists) whose entries are
sliced by profile, and keep `ServiceRequest.supportingInfo` as a
free-form `Reference(Any)` that simply points into entries of that
Bundle. The per-use-case constraints (which Conditions, Observations,
QuestionnaireResponses must be present) live on the Bundle profile
rather than on the `ServiceRequest`.

#### Justification

BSeR is a messaging/referral pattern where the unit of exchange is a
Bundle, not a standalone `ServiceRequest`. Putting the slicing on the
Bundle keeps the contract co-located with the thing actually
transmitted, lets the Bundle be validated end-to-end without resolving
references mid-validation (slicing by `resolve().meta.profile` on
`supportingInfo` requires the validator to dereference, which is
notoriously fragile across implementations), and avoids duplicating
the contract in every per-use-case `ServiceRequest` profile. The
trade-off is that consumers who receive only the `ServiceRequest`
out-of-context (e.g., via a `_include` rather than the original
referral Bundle) lose the structural guarantees that Disposition A
would give them.

### Disposition C: Decline (Close — Not Persuasive / Stale Tracker)

#### Proposal

Close FHIR-21649 as **Not Persuasive** (or **Persuasive with
Modification** pointing at FHIR-22615 if the work group prefers to
consolidate). Rationale: the ticket is a 2019 tracker placeholder
with no description, no comments, no linked tickets, no Zulip
discussion, and no GitHub activity in the six years since it was
opened. The substantive design question it was meant to track is
already represented by concrete, scoped tickets such as
[FHIR-22615](https://jira.hl7.org/browse/FHIR-22615). Keeping an
empty tracker open dilutes the BSeR backlog and risks duplicate
disposition.

#### Justification

HL7 ticket hygiene: tickets should describe a specific change or
question. A ticket whose description is "Adding a tracker item for
following this topic" with no follow-up is exactly the shape the work
group typically retires when triaging a backlog. The technical
question (slice `supportingInfo`?) is not lost — it can be addressed
on the use-case-specific tickets that actually carry concrete
scenarios.

---

### Recommendation

**Recommended disposition:** **C** (close as Not Persuasive, with a
pointer to the use-case tickets), with **A** as the fallback if Public
Health wants to keep the tracker open and act on it now.

The ticket has been static for ~6 years, has no description beyond
"tracker," no participants beyond the original reporter, no
cross-references, and no community discussion. The substantive
modeling question — *should BSeR slice `ServiceRequest.supportingInfo`
per use case?* — is real and worth answering, but it is better
answered on the per-use-case tickets where the work group already has
concrete clinical scenarios (e.g., FHIR-22615 for Arthritis) than on
an empty placeholder. If Public Health instead wants to use this
ticket as the umbrella to drive the change, escalate to Disposition A
and open a follow-up to onboard the BSeR IG source repository into
`cache/github/repos/` so a proper `repo-analysis` briefing can ground
the implementation plan; without that briefing, any implementation
proposal here is necessarily abstract.
