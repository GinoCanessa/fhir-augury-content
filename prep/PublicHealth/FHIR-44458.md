## Summary

The ticket asks the Vital Records Common Profiles Library (VRCPL) to stop
defining a local Yes/No/Unknown/NotAsked value set and instead use an
already-existing Terminology Authority (THO) value set
[`yes-no-unknown-not-asked`](https://terminology.hl7.org/ValueSet-yes-no-unknown-not-asked.html).
If that THO value set is unsuitable, the reporter proposes that a new THO
value set (named "Yes No Unknown") be created and reused, because the
underlying concepts are general-purpose and not specific to Vital Records.

A 2024-05-30 vote (Passas / Wright, 23-0-0) resolved the item as
"Considered for Future Use" on the grounds that substituting the existing
THO `yes-no-unknown-not-asked` was not appropriate — "Unknown" and
"Not Asked" are semantically distinct — but agreed that creating a THO
value set named Yes/No/Unknown would be appropriate. That resolution was
reverted on 2024-10-11 and the ticket was re-opened post STU2 publication
so the work can actually be carried out against THO and the VRCPL IG in
the next publication cycle.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced by `cross-referenced` or Zulip `search` queries for this key or
its obvious keyword set.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the path the reporter's fallback already identifies and the
2024-05-30 vote already endorsed: create a new THO value set named
`Yes/No/Unknown` (tentative id `yes-no-unknown`) in
`terminology.hl7.org`, containing only the three concepts
`yes` / `no` / `unknown` drawn from the appropriate HL7 code systems
(e.g., `http://terminology.hl7.org/CodeSystem/v2-0136` for Yes/No and
`http://terminology.hl7.org/CodeSystem/v3-NullFlavor#UNK` for Unknown,
matching the composition pattern used by
`yes-no-unknown-not-asked`). Then update the Vital Records Common
Profiles Library to retire the locally-defined Yes/No/Unknown/NotAsked
value set and bind the relevant elements to the new THO value set —
handling the "Not Asked" concept separately (e.g., via a `dataAbsentReason`
extension or `nullFlavor` on the element) rather than as a member of the
reused value set.

#### Justification

This is exactly what the 23-0-0 vote endorsed before the resolution was
reverted, and it directly addresses the reporter's stated concern that
the concepts are general-purpose and do not belong in a Vital-Records-only
artifact. Hosting general Yes/No/Unknown concepts in THO is consistent
with HL7's long-standing pattern of promoting reusable terminology out of
individual IGs, and it lets other IGs (not only Vital Records) adopt the
same binding without redefining it. The comment thread explicitly rules
out substituting the existing `yes-no-unknown-not-asked` value set
because "Unknown" and "Not Asked" are semantically distinct, so the new
THO value set avoids that conflation while still achieving the reuse
goal.

---

### Disposition B: Alternative Approach

#### Proposal

Leave the VRCPL value set in place as a local artifact, but refactor its
definition so it *composes* / includes the new THO `Yes/No/Unknown`
value set (once created) plus a single additional "Not Asked" concept
sourced from an existing code system (e.g.,
`http://terminology.hl7.org/CodeSystem/v3-NullFlavor#ASKU` or equivalent).
This makes the Yes/No/Unknown core reusable across IGs while keeping the
Vital-Records-specific "Not Asked" semantic intact for existing VRCPL
profiles and any downstream implementers relying on that element
without changing the binding.

#### Justification

This is a backward-compatible middle ground: existing VRCPL profiles
retain the same value set binding URL and the same four-concept
enumeration, so implementers deployed against STU2 are not forced to
migrate data capture flows that already distinguish Unknown from Not
Asked at the element-value level. It still answers the reporter's real
complaint — that the Yes/No/Unknown trio is general-purpose — by
promoting *that* trio to THO. The cost is that VRCPL continues to own a
local value set, which is the thing the ticket is trying to eliminate;
Disposition A removes that ownership entirely.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" / decline. Retain the existing VRCPL
Yes/No/Unknown/NotAsked value set as defined today, on the basis that
the four-concept enumeration is already stable, is bound into published
STU2 profiles, and that the distinction between "Unknown" and "Not
Asked" is material to Vital Records data collection workflows where
non-response must be recorded explicitly rather than inferred.

#### Justification

Declining is defensible on scope/backward-compatibility grounds: the STU2
publication has shipped with this value set, and any refactor — even a
compose-based one — risks subtle differences in concept URIs, display
strings, or binding semantics that downstream Vital Records
implementations would have to re-certify. The 2024-05-30 vote already
confirmed that substituting the existing THO
`yes-no-unknown-not-asked` value set is not appropriate, so if the
workgroup concludes that creating yet another THO value set is
disproportionate to the benefit, the status quo is a coherent outcome.

---

### Recommendation

**Recommended disposition:** A

Disposition A is what the workgroup already voted 23-0-0 to do before the
resolution was reverted, and the reason for reopening the ticket
(2024-10-11, "re-open post STU2 publication") was operational — to
let the work actually be executed in the next cycle — not a change of
heart about the direction. Externalizing the Yes/No/Unknown trio into
THO solves the reporter's stated concern (general-purpose concepts do
not belong in a Vital-Records-only value set), preserves the
Unknown-vs-Not-Asked distinction that the prior vote correctly
identified as semantically significant (by handling "Not Asked" via
`nullFlavor` / `dataAbsentReason` rather than as a member of the reused
set), and aligns with HL7's broader pattern of promoting reusable
terminology to THO. Disposition B keeps a local VRCPL artifact the
reporter explicitly wanted retired, and Disposition C ignores a clear
existing consensus; neither is as strong as simply re-executing the
original vote.
