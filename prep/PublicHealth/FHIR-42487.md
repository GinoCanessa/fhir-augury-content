## Summary

This ticket asserts that the BSeR (US Bidirectional Services eReferrals)
implementation guide is non-compliant with HL7 policy on locally defined
code systems. Per the policy referenced by the reporter, an IG may only
declare its own `CodeSystem` resources when one of the following holds:
(a) the IG's FHIR Maturity Model (FMM) level is below 2 **and** the code
system is explicitly labeled as temporary (or uses a Terminology.HL7.org
URL that has been reserved for eventual migration), or (b) the HL7
Terminology Services Management Group (TSMG) has granted the IG explicit
permission for those codes to remain permanently IG-specific. The
reporter (Lloyd McKenzie) is asking the BSeR work group to bring the
guide's code systems into conformance with that policy by marking them
temporary (or migrating to reserved THO URLs / obtaining TSMG sign-off
for any that should be permanent). No cross-referenced Jira tickets,
Zulip discussions, or GitHub items were surfaced for this ticket, so the
review needs to be grounded in the BSeR-specific code-system inventory
itself.

## Details

**Description:**
IGs aren't allowed to declare code systems unless they're either &lt;FMM2

### Disposition A: Accept As Requested

#### Proposal

Bring every `CodeSystem` resource declared in the BSeR IG into
conformance with the HL7 IG terminology policy, as a single
coordinated change:

1. Inventory all `CodeSystem` resources currently published by BSeR
   (those with a canonical URL under the BSeR IG, not external
   re-publications).
2. For each one, choose one of three landing states and apply it:
   - **Temporary**: set `CodeSystem.status = "draft"` (or `active` with
     `experimental = true` per current HL7 guidance) **and** add a
     prominent "temporary — pending migration to THO" notice in the
     `CodeSystem.description` / IG narrative; keep the canonical URL on
     the BSeR domain only if BSeR is below FMM2.
   - **Migrated to a reserved THO URL**: coordinate with Vocabulary /
     TSMG to reserve a `http://terminology.hl7.org/CodeSystem/...` URL,
     then change the BSeR `CodeSystem.url` to that reserved THO URL and
     update every binding / `Coding.system` reference in BSeR profiles,
     value sets, examples, and narrative.
   - **Permanently IG-specific**: file a TSMG request, and only keep
     the code system on a BSeR-local URL after TSMG has approved it in
     writing; record the TSMG decision in the IG.
3. Update the IG's terminology narrative page to enumerate each code
   system's chosen state and link to the TSMG decisions where
   applicable.
4. Re-run the IG publisher and the FHIR terminology QA so the
   "IG-defined code system" warnings are cleared (or are accompanied
   by an explicit TSMG-approved exception note).

#### Justification

This is the action the policy literally requires, and the reporter
(Lloyd McKenzie, who maintains a lot of the IG-conformance tooling)
filed the ticket against the BSeR IG specifically because BSeR is
currently out of compliance. Marking code systems temporary, migrating
qualifying ones to THO, or getting TSMG sign-off for the remainder is
exactly the standard remediation HL7 has applied to other IGs in the
same situation; doing so removes the IG-publisher warnings and aligns
BSeR with the broader HL7 terminology stewardship model. Treating it
as a single coordinated pass (rather than ticket-by-ticket per
CodeSystem) avoids dragging the work across multiple ballot cycles.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-classifying every BSeR `CodeSystem` in place, treat
this as a **terminology migration project** with two phases:

- **Phase 1 (immediate, this cycle):** mark every BSeR-declared code
  system as `experimental = true` and append a standard "BSeR-local,
  pending terminology review" stanza to each `CodeSystem.description`.
  Add an IG-level terminology page that lists them and explicitly
  flags BSeR as below FMM2 (so the temporary status is policy-valid
  without needing TSMG action up-front).
- **Phase 2 (next cycle):** for each code system, decide individually
  whether it should (i) be retired in favor of an existing
  THO/SNOMED/LOINC code system, (ii) be migrated to a reserved THO
  URL with Vocabulary/TSMG, or (iii) be brought to TSMG for
  permanent-IG-specific approval. Each decision is tracked as its
  own follow-up ticket.

This separates "stop being non-conformant" from "decide the long-term
home for each code system," which are two different kinds of work
(editorial vs. governance/terminology).

#### Justification

The policy compliance gap can be closed quickly and cheaply (Phase 1
is purely metadata + narrative), which gets BSeR back inside the
fence by the next publication without blocking on TSMG turnaround.
Phase 2 is then a normal terminology backlog that the work group can
prioritize against other BSeR work, instead of being forced to make
permanent decisions about every code system under deadline pressure.
This staged approach is friendlier to a Public Health IG whose
authors may not have the bandwidth to negotiate every code system
through TSMG in a single cycle, while still satisfying the reporter's
underlying concern that BSeR not silently mint permanent IG-local
code systems.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that it is a **policy reminder
rather than a defect against a specific BSeR artifact**: it neither
identifies a specific `CodeSystem` resource nor proposes concrete IG
text. Close as "Not Persuasive — already governed by HL7 IG
terminology policy and TSMG process," and instead handle conformance
case-by-case as TSMG / Vocabulary review surfaces specific BSeR code
systems that need attention.

#### Justification

Defensible only on procedural grounds: the ticket has no identified
artifact, no proposed wording, and no comments to clarify scope, so it
cannot be acted on as written. A reviewer could argue that BSeR's
adherence to the IG terminology policy is enforced by the IG
publisher's terminology QA and by TSMG review at ballot, and that the
remediation work should be tracked by terminology-focused tickets that
name the specific code systems. This disposition is included for
completeness; it is the weakest of the three because the reporter is
flagging a real conformance gap, not asking a hypothetical question.

---

### Recommendation

**Recommended disposition:** B

Disposition B closes the policy gap quickly (Phase 1 is purely a
metadata / narrative pass that any BSeR editor can execute in one
cycle without external dependencies) while still committing the work
group to the deeper terminology decisions in Phase 2. Disposition A
is technically the "right answer" but bundles editorial cleanup with
TSMG negotiation and possible URL migrations into a single change,
which historically has been slow for Public Health IGs and risks the
compliance fix slipping past another publication. Disposition C is
not defensible: the reporter is a knowledgeable maintainer flagging
an actual conformance issue, and ignoring it leaves BSeR
out-of-policy. The work group should adopt B, log the per-code-system
follow-ups as new tickets at the time of the Phase 1 pass, and revisit
this ticket only to confirm Phase 1 has shipped.
