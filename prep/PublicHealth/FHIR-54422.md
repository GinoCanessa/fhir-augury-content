## Summary

The reporter is responding to a Notes-to-Balloters question in the Electronic
Case Reporting (eCR) FHIR IG concerning how Occupation & Industry data should
be represented. The note proposes that eCR adopt the US Core 6.1.0
[US Core Occupation Profile](https://hl7.org/fhir/us/core/STU6.1/StructureDefinition-us-core-observation-occupation.html)
(which implements the USCDI Occupation and Occupation Industry requirements)
in place of the [HL7 US Occupational Data for Health (ODH)](http://build.fhir.org/ig/HL7/us-odh/)
profiles currently used by eCR.

The reporter observes that the US Core Occupation Profile appears to map to
only **one** of the observations sliced into
`Composition.section.entry` of the ODH Composition profile — specifically the
[`odh-PastOrPresentJob`](https://build.fhir.org/ig/HL7/us-odh/StructureDefinition-odh-OccupationalDataForHealth-definitions.html#diff_Composition.section.entry:odh-PastOrPresentJob)
slice. The other ODH entry slices (e.g., usual-work, employment-status,
retirement-date, combat-zone-period, etc.) have no equivalent in the US Core
Occupation Profile. The reporter concludes that, unless additional US Core
mappings are intended that they have missed, **both** profiles should be used
in eCR — though they support replacing the `odh-PastOrPresentJob` slice with
the US Core Occupation Profile.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
recorded in the FHIR Augury index for this ticket as of the time of report.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the eCR IG so that the `odh-PastOrPresentJob` slice in the eCR
Composition's occupation/industry section is replaced by (or aliased to) the
US Core 6.1.0 `us-core-observation-occupation` profile, while **retaining**
the remaining ODH entry slices (e.g., `odh-UsualWork`,
`odh-EmploymentStatus`, `odh-RetirementDate`, `odh-CombatZonePeriod`, and any
other ODH-defined `Composition.section.entry` slices currently required by
eCR). Concretely:

- In the eCR Composition profile that today references ODH, change the
  `Composition.section.entry` slice currently bound to `odh-PastOrPresentJob`
  so its `type.profile` constraint targets
  `http://hl7.org/fhir/us/core/StructureDefinition/us-core-observation-occupation`.
- Leave the other ODH entry slices unchanged so the USCDI Occupation/Industry
  requirement is satisfied via US Core for the past-or-present-job
  observation, while the broader occupational history that ODH supports
  continues to be conveyed via the existing ODH profiles.
- Update narrative, conformance statements, and Notes-to-Balloters in the eCR
  IG to explain the dual use (US Core for USCDI Occupation/Industry; ODH for
  the additional occupational data elements).

#### Justification

This directly addresses the ballot note and the reporter's analysis: the US
Core Occupation Profile and the ODH `PastOrPresentJob` profile cover the same
conceptual observation, so aligning eCR with US Core there improves
interoperability with the broader US Core ecosystem and satisfies USCDI
without forcing implementers to support two near-duplicate observations. At
the same time, the other ODH slices have no US Core equivalent, so retaining
them preserves eCR's existing public-health surveillance content and avoids
data loss for case reporting workflows that rely on usual-work, employment
status, etc.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of swapping in the US Core profile slice-for-slice, define the eCR
slot for the past-or-present-job observation so it accepts **either**
profile, and prefer US Core where available:

- Change the relevant `Composition.section.entry` slice to allow
  `type.profile` of either
  `http://hl7.org/fhir/us/core/StructureDefinition/us-core-observation-occupation`
  **or** the ODH `odh-PastOrPresentJob` profile (e.g., open the slice
  discriminator on `profile` with `rules = open` and list both as permitted
  profiles, or use a `must support` chooser pattern).
- Add a conformance rule (and narrative guidance) that senders SHOULD use the
  US Core Occupation Profile when populating USCDI Occupation/Industry data,
  and MAY also include an ODH `odh-PastOrPresentJob` instance when richer ODH
  semantics are needed.
- Keep the remaining ODH entry slices required as today.

#### Justification

This avoids breaking existing eCR senders that already produce
ODH-conformant occupation observations, gives implementers a smooth
migration path to US Core, and preserves the option for jurisdictions or
sender systems that need ODH-specific elements on the past-or-present-job
observation (industry coding system bindings, occupation/industry
classifications, etc.) to continue using ODH. It is more conservative than
Disposition A on backwards compatibility but at the cost of some
implementer ambiguity about which profile to prefer.

---

### Disposition C: Decline

#### Proposal

Decline to change the eCR Composition's occupation/industry slicing in this
ballot cycle. Update the Notes-to-Balloters response to clarify that:

- US Core 6.1.0 only covers the `PastOrPresentJob` use case, not the other
  ODH `Composition.section.entry` slices, so a wholesale replacement of ODH
  by US Core in eCR is not feasible.
- Coordination with the Public Health and Patient Care work groups and with
  the ODH project is needed before partial replacement is undertaken.
- Defer any actual profile change to a follow-up ballot once an aligned
  cross-IG strategy is documented.

#### Justification

The ticket is fundamentally a clarification request: the reporter is
flagging an incomplete mapping in the original Notes-to-Balloters proposal
rather than asking for a specific normative change. Making a normative
change now risks fragmenting the ODH ecosystem and creating inconsistency
between eCR, US Core, and any other public-health IG that profiles the same
observations. A "decline + clarify + defer" disposition is defensible if the
work group does not have time in this cycle to coordinate with US Core and
ODH stewards.

---

### Recommendation

**Recommended disposition:** A

The reporter explicitly states they "support using the US Core Occupation
profile over the `odh-pastOrPresentJob` profile artifact," and the
substantive concern is only that the *other* ODH entry slices have no US
Core analog and therefore should be retained. Disposition A makes that exact
trade — adopt US Core where it covers the use case (USCDI Occupation /
Industry, i.e., the `PastOrPresentJob` slice), keep ODH where it does not —
and is the cleanest answer to the Notes-to-Balloters question. Disposition B
hedges in a way that increases implementer ambiguity, and Disposition C
leaves the inconsistency the ballot note was created to resolve unaddressed.
The work group should still confirm with the ODH project that the
PastOrPresentJob slice can be safely sourced from US Core (terminology
bindings, must-support elements, and cardinality compatibility) before
publishing.
