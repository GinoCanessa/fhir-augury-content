## Summary

This ticket, titled "Expand on Setup Coordination," asks the BSeR IG to
tighten its currently open-ended guidance on connection setup. Lloyd McKenzie
(reporting on behalf of imported ballot Comment 21) raises two coupled
concerns. First, BSeR does not commit to a concrete security posture for
exchanges between Sender and Receiver: the IG should narrow the field to one
or two named options (e.g., SMART on FHIR, SMART Backend Services, mutual
TLS) rather than enumerating possibilities and leaving the choice to
implementers. Second, BSeR's subscription story is similarly ambiguous —
the IG must say whether implementations are required to support REST-based
creation/update of Subscription resources, or whether subscription wiring
may be performed entirely out-of-band (e.g., by manual configuration), and
whether requiring out-of-band setup is even compliant with FHIR's
expectations for conformant exchange. The fix is editorial-plus-normative
clarification rather than new resource design: pick a small set of
conformant security profiles, declare which are SHALL/SHOULD/MAY, and do
the same for subscription provisioning.

No cross-references, comments, GitHub items, or Zulip discussions were
returned by the augury index for this ticket. The strongest contextual
signals are similarity matches with other Lloyd-McKenzie BSeR ballot
tickets (FHIR-42455, FHIR-42471) and a number of related "pin down the
security mechanism" tickets in adjacent specifications (FHIR-26216,
FHIR-34039, FHIR-35384), summarised under "Related Jira Tickets" below.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new normative "Security and Subscription Setup" section to the BSeR
IG that constrains both halves of the ticket:

1. **Security profiles.** Declare that conformant BSeR exchanges between
   Sender and Receiver SHALL use one of exactly two named profiles:
   (a) **SMART Backend Services** (system-to-system OAuth 2.0 client
   credentials with asymmetric keys) for production server-to-server
   referral exchange, and (b) **SMART App Launch** for any
   user-mediated/launched flow. Mutual TLS MAY be layered as a transport
   constraint but is not by itself sufficient. All other mechanisms
   (basic auth, plain TLS without authentication, bespoke tokens) are
   explicitly out of scope.
2. **Subscription setup.** Declare that BSeR Receivers SHALL support
   RESTful creation and update of `Subscription` resources by Senders
   (R4 Subscription, or R4B/R5 topic-based per the IG's FHIR version),
   constrained to the SubscriptionTopic(s) defined by the companion
   ticket FHIR-42471. Out-of-band provisioning MAY be additionally
   supported but SHALL NOT be the only mechanism — i.e., a
   conformant Receiver always offers a programmatic path.

#### Justification

This is what the ticket literally asks for: pick one or two options and
stop leaving the field open. Both choices are well-trodden — SMART
Backend Services is the de-facto US baseline for system-to-system FHIR
and is already cited as the target by FHIR-26216's resolution path. The
"REST is mandatory, out-of-band is optional" rule for subscriptions is
consistent with FHIR's general principle that conformant exchange must
be discoverable and exercisable at runtime, which directly answers the
ticket's last question ("Is it compliant to **require** out-of-band
configuration?" — no). Pairing this with the FHIR-42471 disposition
gives implementers a single coherent setup story.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than embedding a security profile in BSeR itself, **adopt by
reference** an external security IG and constrain only the
subscription-setup question in BSeR text:

1. Add a normative statement that BSeR exchanges SHALL conform to the
   security and identity requirements of the relevant FAST / UDAP-based
   security IG (or HRex's security section, depending on what BSeR's
   current dependency graph already pulls in). Provide a short
   conformance table mapping BSeR roles (Sender, Receiver, Intermediary)
   to the referenced IG's actor definitions.
2. For subscriptions, take a softer line than Disposition A: REST-based
   subscription management is **SHOULD**, out-of-band configuration is
   **MAY**, and the IG defines a small CapabilityStatement assertion
   (`rest.resource.interaction` for `Subscription` = `create` + `update`)
   that a Receiver advertises iff it supports the REST path, so Senders
   can detect the mode at runtime instead of being told by side channel.

#### Justification

Picking a security profile inside BSeR risks (a) duplicating policy that
already lives in cross-cutting security IGs and (b) diverging from
sibling Public Health and Da Vinci IGs that adopt FAST/UDAP centrally.
Reference-by-IG keeps BSeR's security story aligned automatically when
the upstream IG advances. The softer SHOULD/MAY split for subscriptions
addresses FHIR-35384's counter-argument: a deployment that runs over a
private network or pre-provisioned channel should not be forced to stand
up REST endpoints just to be conformant, provided it advertises that
fact in its CapabilityStatement so Senders can negotiate. Cost: more
indirection, and BSeR readers must follow the reference to learn the
actual rules.

---

### Disposition C: Decline

#### Proposal

Mark Not Persuasive (or Considered for Future Use). Rationale: BSeR is
a profiling IG over US Core / FHIR R4 and intentionally inherits its
security posture from the deployment environment (typically a TEFCA /
QHIN / state HIE arrangement). Subscription provisioning is similarly
expected to be governed by the trust-framework agreement under which two
BSeR participants exchange data, not by the IG itself. Defer the
"required" question to that trust framework and do not add normative
security or subscription-provisioning text to BSeR.

#### Justification

The pattern is established: many US Realm IGs (e.g., US Hybrid/Intermediary
Exchange, per FHIR-35384) explicitly avoid dictating transport security
because deployments already have channel-level controls (VPN, TEFCA
participant agreements). BSeR's value is in the data-content profiles,
not in re-stating SMART or mTLS rules. The cost of declining is that
implementers continue to face the exact ambiguity Lloyd is calling out,
which makes early implementations non-interoperable on day one.

---

### Recommendation

**Recommended disposition:** A

The ticket is reporter Lloyd McKenzie's distillation of imported ballot
Comment 21, filed at Highest priority and explicitly framed as "should
nail this down to one or two options." That is a request for a concrete
normative decision, not an editorial one, and the prior art on adjacent
IGs (FHIR-26216, FHIR-34039) shows the work group has consistently
landed on naming a profile rather than enumerating choices. Disposition
A delivers exactly that on both halves of the ticket, and dovetails
cleanly with the companion ticket FHIR-42471 (which defines the
SubscriptionTopics that A requires Receivers to expose via REST).
Disposition B is a credible second choice and worth holding in reserve
if the work group prefers to keep BSeR thin and lean on a FAST/UDAP
reference; if so, the reference target should be named in committee
before publication so we don't ship another open-ended pointer.
Disposition C should be declined: leaving security and subscription
setup unspecified is what produced this ballot comment in the first
place, and answering "Is it compliant to require out-of-band
configuration?" with silence is exactly what the reporter is objecting
to.
