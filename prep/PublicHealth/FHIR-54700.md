## Summary

The ticket is a low-priority technical correction against the **US Central
Cancer Registry Reporting Content IG (FHIR)**. In section **6.2.2 Detailed
Description of Changes**, the "Change Description" column for the row
"Article = All (where applicable)" currently reads "Added references to **US
PH Library profiles**". The reporter, Forrest White, observes that the
referenced library's correct name is the **US Public Health Profiles
Library**, and that the text should both use the full name and link to the
library's IG page at
[https://build.fhir.org/ig/HL7/fhir-us-ph-common-library-ig/](https://build.fhir.org/ig/HL7/fhir-us-ph-common-library-ig/).

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
in the FHIR Augury index for this ticket, so the request stands on its own:
a wording fix plus a hyperlink in the IG's change-history narrative.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit section **6.2.2 Detailed Description of Changes** of the US Central
Cancer Registry Reporting Content IG so that, in the "Change Description"
column of the "Article = All (where applicable)" row, the phrase "Added
references to **US PH Library profiles**" is rewritten as "Added references
to profiles in the **[US Public Health Profiles
Library](https://build.fhir.org/ig/HL7/fhir-us-ph-common-library-ig/)**",
i.e. spell out the official IG name and turn it into a hyperlink to the
library IG. No code, profile, value-set, or terminology change is required;
this is a narrative-only edit in the change-history section of the IG.

#### Justification

The reporter has correctly identified that "US PH Library" is informal
shorthand for the published HL7 IG titled *US Public Health Profiles
Library* (`HL7/fhir-us-ph-common-library-ig`). Using the canonical name
plus a deep link is consistent with how other Public Health IGs reference
shared libraries, improves discoverability for implementers reading the
change log, and removes ambiguity about which "PH Library" is meant. The
ticket is classified as a Technical Correction at Low priority, which is
exactly the bucket this kind of narrative fix is intended for, and there is
no community disagreement on the index that argues against it.

---

### Disposition B: Alternative Approach

#### Proposal

Make the same wording fix as Disposition A, but at the same time introduce
a short defined term — e.g., "US Public Health Profiles Library (the *US PH
Profiles Library*)" — at the first reference in the IG's introduction (or
a glossary/abbreviations section), and then use that defined short form in
the change-history table and other inline references. The hyperlink to
`https://build.fhir.org/ig/HL7/fhir-us-ph-common-library-ig/` is attached
to the first occurrence (and re-attached in the change log entry), but
later mentions can use the defined short form without re-linking.

#### Justification

The IG plausibly references the library in multiple places (change log,
introduction, profile prose). Defining a canonical short form once, and
using it consistently, reduces churn the next time someone notices another
spot using "US PH Library" or "PH Library profiles". It also keeps the
change-history row readable rather than swelling every mention to the full
title. Trade-off vs. Disposition A: it is slightly more work and touches
more pages than the literal request, so it is only worth doing if a quick
scan of the IG turns up additional inconsistent references; if the
change-log row is genuinely the only spot, the extra structure is
unjustified.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the change-history table in section
6.2.2 documents what was done in a prior publication cycle and that
historical entries should be preserved verbatim, with corrections handled
through a follow-on change-history row in the next published version
rather than by editing the existing row.

#### Justification

Some IGs treat past entries in the "Detailed Description of Changes" table
as an immutable record of what shipped in a given version, on the basis
that retroactively editing the log makes it harder to reconcile against
prior published builds. Under that policy, the right response would be to
leave the existing row alone and, in the *next* version, add a row noting
that references to the *US Public Health Profiles Library* have been
clarified.

---

### Recommendation

**Recommended disposition:** A

This is a low-risk, narrative-only correction in the IG's own change-log
section. The reporter has supplied both the canonical name and the correct
URL, the ticket is filed as a Technical Correction at Low priority, and
there is no Zulip, GitHub, or related-ticket context suggesting a broader
scope or any disagreement. Disposition B is a reasonable enhancement *if*
a quick editorial pass confirms additional inconsistent references
elsewhere in the IG, but it should not block the simple fix; the editor
applying Disposition A can promote it to B at their discretion. Disposition
C asks the workgroup to defend an "immutable change log" policy that this
IG does not appear to have adopted, and would leave a known-incorrect
shorthand in the published narrative for at least one more cycle, which
is the opposite of what a Technical Correction ticket is for.
