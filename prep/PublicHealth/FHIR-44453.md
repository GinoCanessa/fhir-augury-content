## Summary

FHIR-44453 is one entry in a series of "Code Systems and Valuesets should be
anchored in THO (terminology.hl7.org)" change requests filed by Ron G. Parker
against the **Vital Records Common Profiles Library (VRCPL, FHIR)** IG. Each
entry in the series corresponds to a specific code system / value set that the
IG defines locally rather than under `terminology.hl7.org`. The "28" suffix in
the title and the body's "Comment 43" reference indicate this row was imported

`http://terminology.hl7.org/...` and start the UTG (Unified Terminology
Governance) process to migrate the artifact into THO**, in compliance with the
TSC-approved [Terminology Expectations for IG
Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-44453 as covering the **Units of Age** value set referenced by the
2024-10-11 vote note. Concretely:

1. Identify the locally-defined Units-of-Age value set in the VRCPL IG (e.g.,
   `http://hl7.org/fhir/us/vrdr-common/.../ValueSet/...UnitsOfAge` or the
   equivalent VRCPL canonical) and confirm it is not already a thin wrapper
   over a UCUM/v3 binding.
2. Open a UTG proposal to publish the value set under
   `http://terminology.hl7.org/ValueSet/...` (and any new code system, if
   needed, under `http://terminology.hl7.org/CodeSystem/...`), following the
   normal HL7 UTG submission workflow in `HL7/UTG`.
3. After the UTG artifact is published, update the VRCPL IG (in
   `HL7/vr-common-library`) to **remove the locally-defined ValueSet
   resource** and replace any `ValueSet` references / bindings with the new
   THO canonical URL. Apply the same pattern to all sibling tickets
   (FHIR-44197, FHIR-44300, FHIR-44296, FHIR-44439, FHIR-44441, FHIR-44455,
   FHIR-44456, FHIR-44457, FHIR-44426, FHIR-44404, FHIR-44384, FHIR-44383,
   FHIR-44386, FHIR-44194, FHIR-44399) so the batch is dispositioned as a
   unit.
4. Track the cross-IG migration as a coordinated change between the Public
   Health and Vocabulary work groups, since UTG publishing is a Vocabulary /
   TSMG-governed pipeline.

#### Justification

This is exactly what the TSC-approved [Terminology Expectations for IG
Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy prescribes, and it is what the 2024-10-11 vote-revert note already
endorses ("Creating a THO Valueset for Units of Age is appropriate."). The
prior "Considered for Future Use" disposition was explicitly reverted, and
the ticket was re-opened post-STU2 specifically so that this policy
alignment can land in the next VRCPL cycle. Anchoring under THO also
unlocks reuse of the value set by other IGs (FHIR Core, US Core, other PH
specs) without forcing dependents to take a hard dependency on VRCPL.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the THO-anchoring intent but **do it as a single coordinated VRCPL
disposition pass** rather than per-ticket:

1. Triage FHIR-44453 together with all sibling tickets surfaced by keyword
   similarity (FHIR-44197, FHIR-44300, FHIR-44296, FHIR-44439, FHIR-44441,
   FHIR-44455, FHIR-44456, FHIR-44457, FHIR-44426, FHIR-44404, FHIR-44384,
   FHIR-44383, FHIR-44386, FHIR-44194, FHIR-44399) as one workgroup ballot
   item.
2. Produce a single tracking spreadsheet that maps each ticket → affected
   `CodeSystem`/`ValueSet` URL → target THO canonical → owning UTG proposal.
3. Submit one UTG batch proposal covering all migrations rather than 15
   independent proposals.
4. Apply a single coordinated PR in `HL7/vr-common-library` to swap local
   canonicals for THO canonicals once UTG publishes them, and close the whole
   ticket cluster against that single change set.

#### Justification

The Augury data shows these ~15 tickets are template-level duplicates of one
policy assertion against different value sets; dispositioning each
independently risks inconsistent decisions, partial migrations, and
duplicated UTG paperwork. A batched approach matches how UTG normally
prefers to receive related submissions, gives Vocabulary / TSMG a single
review surface, and lets VRCPL ship one IG release that completes the
THO-anchoring policy rather than dribbling it across multiple cycles. The
trade-off vs. Disposition A is process overhead (one combined ballot item
vs. fast per-ticket closure), but the per-artifact action taken on each
value set is the same.

---

### Disposition C: Decline

#### Proposal

Decline as **out of scope for VRCPL at this time**: leave the value set
defined locally in the VRCPL IG, do not initiate a UTG submission, and close
the ticket as "Not Persuasive" or "Considered for Future Use" (the original
2024-05-30 disposition before it was reverted).

#### Justification

A defensible "decline" rationale would rest on (a) the affected value set
being VRCPL-specific with no realistic reuse outside Vital Records, (b) the
UTG submission cost outweighing the policy benefit for a small enumeration,
or (c) sequencing — VRCPL has just published STU2 and the workgroup may
prefer to defer terminology-pipeline migrations to a dedicated future
release. However, this rationale **conflicts directly** with the TSC-approved
THO-anchoring policy and with the workgroup's own 2024-10-11 reversal of the
earlier "Considered for Future Use" vote, so it would need explicit
re-endorsement by Public Health and acknowledgement that VRCPL is choosing
not to comply with TSMG guidance for this artifact.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B (accept the request, but execute it as a single coordinated
VRCPL THO-anchoring pass across the full sibling cluster) is the strongest
choice. The substantive answer is the same as Disposition A — the policy
applies, the workgroup has already reversed the prior "future use" vote, and
the 2024-10-11 note explicitly endorses creating a THO value set for Units of
Age — but FHIR-44453 is one of ~15 functionally identical tickets, and
disposing of them one-by-one would create needless UTG and editorial churn,
plus the real risk of inconsistent outcomes across siblings. Approving them
as a batch lets Public Health hand a single, well-scoped deliverable to
Vocabulary / TSMG and to the VRCPL editors. Disposition C should only be
considered if the workgroup explicitly decides to walk back the 2024-10-11
reversal, which the Augury record gives no evidence of.
