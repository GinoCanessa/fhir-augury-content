## Summary

Craig Newman is asking how the FHIR `Observation` profiles defined in the BSeR
Implementation Guide's StructureDefinitions relate to the BSeR Domain Analysis
Model (DAM). The DAM (specifically its last two pages) enumerates observations
that fit into various referral program types, but he cannot find an "Arthritis
Feedback" observation analogue in the IG. The IG defines an arthritis grouper,
but the program-specific feedback data types in the DAM appear more granular
than what the StructureDefinitions provide. He further questions whether the
BSeR Observation profiles should be bound to specific LOINC codes, and whether
the terminology binding work is incomplete. A follow-up comment from the
reporter asks whether SNOMED CT code `3723001` ("Arthritis") is intended as the
`Observation.code` value for the relevant profile.

The cross-reference query returned **no** linked Jira tickets, Zulip threads,
Confluence pages, or GitHub items, and a content search for the ticket key
across configured sources returned no hits. There is therefore no community
discussion or implementation activity surfaced in FHIR Augury that bears on
this question; the ticket appears to have been triaged in 2018 and left to sit.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reconcile the BSeR IG's `Observation` profiles with the DAM's program-specific
feedback observation list. Concretely:

1. Add a profile (or set of profiles) covering the **Arthritis Feedback**
   observation that the DAM enumerates but the IG currently lacks, alongside
   any other DAM feedback observations that are missing from the IG. These
   would sit beside the existing arthritis grouper profile rather than
   replacing it.
2. For every BSeR Observation profile, fix `Observation.code` to a
   `required`-strength binding against a BSeR-curated value set. Where the DAM
   names a concept (e.g., the reporter's candidate SNOMED CT `3723001`
   "Arthritis"), pin it as the canonical code; otherwise add LOINC codes for
   the feedback observations and document which code system / answer list each
   feedback question uses.
3. Add an explicit narrative section in the BSeR IG that maps each DAM
   observation row to the IG profile (or extension) that realises it, and
   flags any DAM concepts that were intentionally **not** profiled.

#### Justification

The reporter is highlighting two well-formed gaps: (a) DAM-to-IG traceability
is incomplete, and (b) terminology bindings on `Observation.code` are
under-specified. Both are normal expectations for a US Realm IG that derives
from a DAM — implementers need to know which DAM concept a given profile
realises, and `Observation.code` without a strong binding leaves
interoperability to chance. Adding the missing profile(s) and tightening the
bindings is the most direct way to honour the request.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding new profiles for every missing DAM feedback observation,
publish a **DAM-to-IG conformance map** as supporting material in the BSeR IG
and rely on a single, extensible "BSeR Feedback Observation" profile bound to
an extensible value set:

1. Author a new IG page ("DAM Coverage") that tabulates every DAM observation
   against either an existing BSeR profile, an existing US Core profile, or an
   explicit "out-of-scope for this version" marker. The Arthritis Feedback row
   would point at the existing arthritis grouper plus a generic feedback
   profile.
2. Define one `BSeRFeedbackObservation` profile whose `Observation.code` is
   bound (`extensible`) to a new BSeR Feedback Codes value set, populated with
   LOINC where available and SNOMED CT (e.g., `3723001`) where LOINC has no
   match. Program-specific feedback variants are then realised by binding the
   value set per use, not by minting one profile per program.
3. Treat any remaining program-specific gaps as Jira items for the next BSeR
   release rather than blocking on them now.

#### Justification

The DAM's feedback observation list will grow with each new referral program;
minting one profile per row (Disposition A) creates ongoing maintenance and a
combinatorial explosion of nearly-identical StructureDefinitions. An extensible
value set on a single feedback profile lets implementers cover new programs
without IG changes, and a published DAM-coverage table preserves the
traceability the reporter is asking for without committing the workgroup to a
profile-per-concept strategy. Trade-off vs. A: less prescriptive validation
(extensible vs. required) and slightly weaker out-of-the-box conformance, in
exchange for substantially lower long-term IG churn.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Modification — No Change**, on the grounds that the
ticket is a question, not a defect, and the BSeR IG has moved on substantially
since 2018 (the ticket has sat in `Triaged` with `Unassigned` for several
years). Respond to the reporter that:

- The DAM and the IG are intentionally not 1:1 — the DAM scopes the analysis
  while the IG profiles only the artifacts needed for the current published
  use cases.
- Terminology binding work for BSeR Observation profiles is being handled in
  the active tickets for the current BSeR ballot cycle, not under this
  question ticket; new binding gaps should be filed against the current
  version.
- If the reporter still believes a specific Arthritis Feedback profile is
  missing from the **current** BSeR release, the workgroup invites a fresh
  ticket scoped to that one concept with a concrete proposed code.

#### Justification

The ticket as written is exploratory ("How do … relate to the DAM?", "Is that
part of the terminology that hasn't been fully fleshed out?") rather than a
concrete change proposal, has no community discussion attached (no Zulip
threads, no linked tickets, no PRs), and is years old. Acting on it now risks
designing against a frozen snapshot of the DAM and a long-superseded version
of the IG. The defensible path is to close it without action and ask the
reporter (or a successor) to refile against the current spec with a specific,
actionable change.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern is real — DAM-to-IG traceability and
`Observation.code` binding strength are legitimate IG-quality issues — so
declining outright (C) leaves a known gap on the table and disrespects a
well-formed question, even if it has aged. Disposition A fixes the symptom
but commits the workgroup to authoring and maintaining one profile per DAM
feedback row, which scales poorly as new referral programs are added to BSeR.
Disposition B captures the traceability the reporter wants (via a published
DAM-coverage table) and gives implementers a single extensible feedback
profile with a curated value set, which is the pattern most modern US Realm
IGs use for survey- and feedback-style observations. It also gives the
workgroup a clean place to land the reporter's specific Arthritis question
(SNOMED CT `3723001` becomes a value-set entry, not a profile decision). The
workgroup should sanity-check current BSeR content against this proposal
before adopting it, since the cached FHIR Augury data does not include the
BSeR IG repo and the recommendation has not been ground-truthed against the
current sources.
