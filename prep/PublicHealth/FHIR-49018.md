## Summary

Saul Kravitz reports an internal cardinality inconsistency in the
`UsualWork` profile of the US Occupational Data for Health (ODH) FHIR
implementation guide. As currently authored, the profile makes the
`Observation.value[x]` element (carrying the *Usual Occupation*) optional
with cardinality `0..1`, while the *Usual Industry* component is
constrained to `1..1` (required). The reporter argues that a profile
named "UsualWork" should treat its two defining concepts symmetrically:
both Usual Occupation and Usual Industry should be either both required
(`1..1`) or both optional (`0..1`), because making one required while
the other is optional is confusing for implementers and suggests an
unintended modeling choice.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
artifacts associated with this ticket in the indexed sources.

## Proposed Dispositions

### Disposition A: Accept As Requested — Tighten Usual Occupation to 1..1

#### Proposal

In the `UsualWork` profile of the US ODH IG, change the cardinality of
`Observation.value[x]` (Usual Occupation) from `0..1` to `1..1` so it
matches the existing `1..1` constraint on the `Usual Industry`
component. Concretely:

- StructureDefinition: `UsualWork` (the ODH `Observation`-based profile
  for usual work).
- Differential change: set `Observation.value[x].min = 1`.
- Update the profile narrative / "Mandatory" table and any examples
  that currently omit `value[x]` so they validate against the tightened
  profile.
- Add a brief change note to the IG history page.

#### Justification

This honours the reporter's intuition that the two defining concepts of
`UsualWork` — occupation and industry — should be treated symmetrically.
A `UsualWork` instance without a Usual Occupation is arguably not a
useful representation of "usual work"; making both required is
consistent with the way most occupational-health use cases (BLS, NIOSH,
public-health surveillance, EHR data exchange) treat occupation and
industry as a paired classification. It also resolves the asymmetry the
reporter flags as confusing without expanding the profile's scope.

The main risk is breaking conformance for any production data that
currently sends a `UsualWork` Observation with only the industry
component populated; that risk should be discussed with implementers
before adopting.

---

### Disposition B: Alternative Approach — Relax Usual Industry to 0..1

#### Proposal

Instead of tightening Usual Occupation, relax Usual Industry by
changing the cardinality of the relevant component slice (the
`UsualIndustry` component in the `UsualWork` profile) from `1..1` to
`0..1`, so both elements are optional and symmetric. Concretely:

- StructureDefinition: `UsualWork`.
- Differential change: on the `Observation.component` slice
  representing Usual Industry, set `min = 0`.
- Add an `obs-X` invariant (or strengthen the existing slicing rules)
  requiring that *at least one* of `value[x]` (Usual Occupation) or the
  `UsualIndustry` component be present, so the profile still
  communicates meaningful work data.
- Update narrative, "Mandatory" table, and examples accordingly.

#### Justification

This achieves the symmetry the reporter is asking for without breaking
existing senders that omit Usual Occupation but populate Usual Industry
(a real pattern for some surveillance data sources where industry is
captured but a job-title/occupation code is not). The added "at least
one of" invariant preserves the IG's semantic intent — a `UsualWork`
instance is required to convey *some* work information — while letting
data sources contribute partial information that they actually have.

Trade-off vs. Disposition A: this is more permissive and slightly more
complex (introduces an invariant), but it is backward-compatible with
existing conforming instances that only have one of the two elements,
which Disposition A is not.

---

### Disposition C: Decline — Persuasive but Not Persuasive

#### Proposal

Mark the ticket "Persuasive but Not Persuasive" (or similar Decline
resolution). Document that the asymmetry is intentional: Usual Industry
(the employer's industry) is the more reliably-coded data element in
the source systems and surveillance use cases that drove the original
ODH design, while Usual Occupation (the worker's job role) is often
free-text or missing, so the profile deliberately requires the more
reliable element and treats the other as optional. Point implementers
to the IG narrative for the rationale and decline to change the
cardinalities.

#### Justification

Declining is defensible if the ODH workgroup confirms — from the
original design history — that the asymmetric cardinality is an
intentional reflection of the underlying data sources rather than an
oversight. In that case, changing it would either break existing
conformance (Disposition A) or weaken a constraint that was put there
on purpose (Disposition B). The corrective action is then editorial: a
clarifying note in the profile description explaining *why* Usual
Industry is required while Usual Occupation is optional.

The weakness of this disposition is that, absent that documented
rationale, the asymmetry really does read as a modeling oversight, and
"because we said so" will not satisfy a careful reviewer.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — relax Usual
Industry to 0..1, with a "one-of" invariant).

The reporter's complaint is fundamentally about *symmetry and
consistency*, not about which specific element should be mandatory.
Disposition B delivers that symmetry while remaining backward-compatible
with any production data that currently relies on the existing
`UsualWork` profile (data that has only Usual Industry populated will
continue to validate). Disposition A produces a cleaner profile but is
a breaking change for any sender that today omits Usual Occupation —
and given that occupation coding is widely known to be the *less*
reliable of the two elements in real-world employment data, tightening
that side first is exactly the wrong direction. Disposition C is only
viable if the ODH authors can document a deliberate design rationale
for the asymmetry; in the absence of such documentation in this
ticket's snapshot, sustaining the asymmetry will keep generating the
same implementer confusion the reporter raised.

The workgroup should confirm, before adopting B, whether real
implementations exist that send `UsualWork` with only Usual Industry —
that would lock in the choice between A and B. If no such
implementations exist, A becomes equally attractive on cleanliness
grounds.
