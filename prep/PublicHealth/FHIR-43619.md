## Summary

The reporter requests that the US Birth and Fetal Death Reporting (BFDR) FHIR
IG add an EHR-to-EBRS (Electronic Birth Registration System) interface,
modeled on the existing EHR-to-MDI (Medical Examiner / Death Investigation)
API in the mortality reporting space. The proposed interface would let an EHR
(a) query the EBRS to determine whether a birth record already exists, and
(b) create or update a record in the EBRS, all via standard FHIR
POST/GET interactions secured with an OAuth2 flow. The ticket was originally
resolved as "Considered for Future Use" on 2024-05-30 (vote 29-0-0), then
re-opened post-STU2 publication on 2024-10-15 by lynn_laakso, who noted that
STU2 was scoped to polishing the Provider-VRO and VRO-NCHS interfaces (using
FHIR Messaging in line with mortality reporting), and that the EHR-EBRS
interface "requires further discussion with EHR and EDRS implementers." No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were found
in the FHIR Augury index for this ticket.

## Details

**Description:**

Add an API similar to the MDI API that an EHR could use to query for the
existence of a birth record in the EBRS, and create/update a record in the
EBRS. All of these operations are completed through basic POST/GET operations
on a FHIR server. An OAuth2 security flow is recommended for restricting
access to the API.

## Proposed Dispositions

> Note: No GitHub cross-references were indexed for this ticket, so no
> repo-analysis briefing was loaded. The dispositions below name the
> spec/IG (US Birth and Fetal Death Reporting, BFDR) and the analogous
> mortality-reporting work (MDI / VRDR) that the ticket itself cites,
> but cannot pin a specific authoring path without a briefing.

### Disposition A: Accept As Requested

#### Proposal

Add a normative EHR-to-EBRS REST interface to the BFDR IG, modeled directly
on the EHR-MDI API used in the mortality space. Concretely:

1. Define a CapabilityStatement for the EBRS server that documents:
   - `GET /Composition?...` (or a dedicated `$ebrs-find` OperationDefinition)
     for an EHR to query whether a birth/fetal-death record already exists
     for a given mother + date-of-event + facility key set.
   - `POST /Bundle` (transaction) — or `POST /Composition/$ebrs-submit` —
     for an EHR to create a new birth/fetal-death record in the EBRS.
   - `PUT`/`PATCH` semantics (or a `$ebrs-update` operation) for amending
     an existing record while preserving EBRS-side identifiers.
2. Reuse the existing BFDR Bundle/Composition profiles (Birth and Fetal
   Death) as the payloads, adding any EHR-context profile constraints
   (e.g., requiring a stable EHR-side identifier on the Composition) that
   the EBRS needs to dedupe against the query in step 1.
3. Document a SMART-on-FHIR / OAuth2 client-credentials (system-level)
   security profile, with scopes scoped to the EBRS endpoints listed above.
4. Add an "EHR-EBRS Interface" page to the IG narrative paralleling the
   existing Provider-VRO and VRO-NCHS pages, with sequence diagrams for
   the query / create / update flows.

#### Justification

This is exactly the symmetry the mortality-reporting stack already provides
(EHR ↔ MDI, MDI ↔ VRO, VRO ↔ NCHS), and the BFDR IG already commits to
"the same mechanisms used for mortality reporting" per lynn_laakso's STU2
note. Adding the EHR-EBRS leg closes the obvious gap and gives EHR vendors
a single normative target instead of N bespoke jurisdictional integrations.
Using REST (not Messaging) for the EHR leg matches the MDI precedent and
matches what most EHRs already implement for SMART-app and bulk-data work.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than standing up a new EHR-EBRS REST interface inside BFDR, extend
the existing **Provider-VRO interface** to be the single EHR/Provider entry
point and document EHR-specific guidance (auth, identifier reuse, dedupe
query) as a profile/usage layer on top of it. Specifically:

1. Keep the wire mechanism aligned with the existing Provider→VRO interface
   (whether REST or Messaging as currently specified for STU2) so EBRS and
   VRO implementations don't diverge.
2. Add an `$exists` (or equivalent search) operation to the
   Provider-VRO surface that an EHR can call to dedupe before submission;
   make this the *only* new operation introduced by this ticket.
3. Publish an "EHR Implementer Guide" page in the BFDR IG that maps EHR
   workflows (admit/discharge for L&D, fetal-death documentation) to the
   existing Provider-VRO request/response pairs and an OAuth2 client-creds
   security profile.
4. Defer any net-new resource/profile work; reuse existing BFDR Birth and
   Fetal Death Composition/Bundle profiles unchanged.

#### Justification

The reporter's underlying need is "let an EHR check for, create, and update
a birth record in the EBRS." That need can be met by clearly documenting the
EHR as a valid Provider in the existing Provider-VRO interface, plus one
small dedupe operation, instead of introducing a parallel EHR-EBRS surface
that the workgroup would then need to maintain in lock-step with
Provider-VRO. It also avoids the risk of two divergent submission paths
(EHR-EBRS vs. Provider-VRO) producing inconsistent EBRS records, and is far
less invasive for an STU.next ballot.

---

### Disposition C: Decline

#### Proposal

Re-resolve as **Considered for Future Use** (restoring the 2024-05-30
disposition) and explicitly defer EHR-EBRS interface work to a post-STU2
project, contingent on documented commitment from EHR and EDRS implementers
to participate in design and connectathon testing.

#### Justification

The 2024-10-15 re-open comment is itself the rationale: "This requires
further discussion with EHR and EDRS implementers." Standardizing an
EHR-facing API without active EHR vendor input risks producing a spec that
nobody implements, and the STU2 release was deliberately scoped to the
Provider-VRO and VRO-NCHS legs. Declining now (with a clear "future use"
marker) preserves the option without forcing the workgroup to ballot an
under-specified interface, and matches the prior 29-0-0 vote.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle that the 2024-10-15 re-open comment
identified: it gives EHRs a documented, normative path into the EBRS
without introducing a second submission surface that would have to be kept
in sync with Provider-VRO. Disposition A is the most faithful to the
reporter's literal request, but it risks bifurcating the IG's submission
story right after STU2 deliberately consolidated on Provider-VRO /
VRO-NCHS, and would need substantial EHR-vendor input to land safely.
Disposition C is the safe option but effectively re-litigates the prior
29-0-0 vote that has already been reverted; the workgroup has signaled it
does want to move forward, just carefully. Disposition B captures that
"move forward, but reuse the interface we already have" intent and gives
the workgroup something concrete (one new dedupe operation + an EHR
Implementer Guide page) to ballot in the next cycle.
