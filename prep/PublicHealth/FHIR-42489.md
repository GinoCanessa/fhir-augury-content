## Summary

Lloyd McKenzie (in an imported comment-40 ballot item against the BSeR FHIR
IG) argues that an implementation guide that *defines messaging solutions*
is incomplete unless it ships normative `MessageDefinition` resources for
every message it expects implementers to send. The ticket asks the BSeR
work group to add `MessageDefinition` instances that (a) bind to the
specific event code for each business event, (b) describe the focus and
graph of the message content (i.e., the bundle entries and their
profiles/cardinalities), and (c) enumerate the allowed response message
types. No cross-referenced tickets, Zulip threads, GitHub PRs, or
Confluence pages were surfaced for this key by the Augury indexes;
however, keyword neighbours (FHIR-42488 "standard event codes for all of
the messages", FHIR-42520 "Don't use a fixed v2 event code", FHIR-42521
"MessageHeader profile needs more", FHIR-42496 "Inappropriate
CapabilityStatement") are clearly part of the same ballot tranche on BSeR
messaging and should be tracked together.

## Details

### Disposition A: Accept As Requested

#### Proposal

Author one normative `MessageDefinition` resource per business event
defined in the BSeR IG. Each `MessageDefinition` shall:

- Set `eventCoding` to the BSeR event code defined per FHIR-42488 /
  FHIR-42520 (i.e., FHIR-native codes from a BSeR code system, not a
  hard-coded v2 trigger).
- Set `category` to `notification` or `consequence` as appropriate per
  message.
- Populate `focus` with the resource type that is the subject of the
  message (typically the BSeR `ServiceRequest` profile, with the
  appropriate `min`/`max` and `profile` reference).
- Use `MessageDefinition.parent` (or narrative + a profile cross-link)
  to point at the BSeR `MessageHeader` profile so the bundle graph is
  fully described.
- Populate `allowedResponse.message` with references to the
  `MessageDefinition`(s) that are valid responses (acknowledgement,
  follow-up request for more info per FHIR-42509, completion, error,
  etc.), each with an explicit `situation`.
- Be listed in `CapabilityStatement.messaging.supportedMessage` (paired
  fix for FHIR-42496) with `mode = sender` or `receiver` per actor.

Add a new IG page that tabulates the message catalogue (event code →
MessageDefinition → focus profile → allowed responses) and link it from
the IG's messaging chapter.

#### Justification

`MessageDefinition` is the FHIR-defined conformance artifact for exactly
this purpose: it is what `CapabilityStatement.messaging.supportedMessage`
points at, and it is the only computable way to declare the event code,
focus, message bundle graph, and allowed responses in one place. Without
it, implementers must reverse-engineer the message catalogue from prose,
which contradicts the stated goal of "defining messaging solutions".
This is consistent with how other HL7 messaging IGs (e.g., Da Vinci
PDex, V2-to-FHIR notifications) ship their message catalogues. It also
enables tooling (validators, the IG publisher, IG examples) to
automatically check that conformant senders actually populate the bundle
the way the IG intends.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring a separate `MessageDefinition` instance per
business event, define **one base `MessageDefinition`** for BSeR that
declares the common bundle graph (MessageHeader → ServiceRequest focus →
supporting resources) and **derive per-event `MessageDefinition`s via
`MessageDefinition.parent`** that only override `eventCoding` and
`allowedResponse`. Pair this with a single tabular IG page that lists
every event and links to the derived `MessageDefinition`. Update the
`CapabilityStatement` to enumerate the derived definitions, but keep the
authoring burden in one place.

#### Justification

BSeR's messages are highly homogeneous — all are referral-flow
variations of the same `ServiceRequest`-centric bundle. Maintaining N
near-identical `MessageDefinition` files is duplicative and a known
source of drift (the IG already exhibits this in
profile-vs-profile drift flagged elsewhere in this ballot, e.g.
FHIR-42521). A parent/child pattern keeps the graph definition
DRY, addresses Lloyd's underlying need (every event has a
machine-readable definition with focus + allowed responses), and makes
the answer to FHIR-42509 (request-more-info response) a single
`allowedResponse` slot inherited by every child. Trade-off vs.
Disposition A: tooling support for `MessageDefinition.parent` is less
exercised than for flat MessageDefinitions, so the IG publisher should
be smoke-tested before committing to this shape.

---

### Disposition C: Decline

#### Proposal

Decline the request as stated and instead document, in IG narrative
only, the list of business events, the event code for each, and the
expected message bundle structure. Rely on the existing `MessageHeader`
profile + a prose table for the message catalogue, without authoring
`MessageDefinition` instances.

#### Justification

Defensible only on resourcing grounds. Stated rationale would be that
BSeR is a small, tightly-scoped IG, the message catalogue is short, and
adding `MessageDefinition` instances now would block the ballot
publication on additional authoring, validation tooling, and
example-bundle work that the work group cannot absorb in the current
cycle. Risk: the IG remains conformance-incomplete (a messaging IG
without `MessageDefinition`s cannot be referenced from a conformant
`CapabilityStatement.messaging.supportedMessage`), and FHIR-42496 and
FHIR-42509 would have to be declined with parallel rationale. This
position would be hard to defend at ballot reconciliation given the
priority is **Highest**.

---

### Recommendation

**Recommended disposition:** A

`MessageDefinition` is the FHIR-native answer to literally every part of
this ticket — event binding, message graph, allowed responses — and the
fix unblocks the closely-related FHIR-42488, FHIR-42496, FHIR-42509, and
FHIR-42521 in one coordinated edit pass. The ticket is **Priority
Highest** under "Triaged" and was raised by Lloyd McKenzie, so a Decline
(Disposition C) is unlikely to survive reconciliation. Disposition B
(parent/child MessageDefinitions) is the right *implementation tactic*
once the work group commits to A, and the work group should adopt A as
the disposition with a note that the authoring may follow B's
parent/child shape if IG-publisher support is verified. Pair the
resolution with the related tickets above so that
`CapabilityStatement.messaging`, `MessageHeader`, and the event-code
value set all land in the same revision.
