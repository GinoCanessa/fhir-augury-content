## Proposed Dispositions

> All three dispositions below are constrained by the fact that the
> ticket's substance lives in an inaccessible PDF attachment. Each
> disposition therefore proposes both a substantive direction *and* the
> minimum information the workgroup should extract from the PDF before
> committing to it.

### Disposition A: Accept As Requested

#### Proposal

Open the attached PDF, transcribe its enumerated defects into the
ticket as a structured list (one bullet per defect), and apply each
correction directly to the bilirubin Example instance in the Maternal
and Infant Health Research (FHIR) IG. Typical defects in lab-result
examples of this kind include: incorrect or missing LOINC code on
`Observation.code`, wrong or missing UCUM unit on
`Observation.valueQuantity` (mg/dL vs. µmol/L), missing
`Observation.subject` reference resolution, missing
`Observation.effective[x]`, missing `Observation.status`, missing
`Observation.category` (`laboratory`), missing or incorrect
`Observation.referenceRange` for a neonate, and missing `performer`
or `specimen` linkage. Each item identified in the PDF would be fixed in
the example XML/JSON/Turtle source under the IG's `input/examples/`
(or equivalent) directory and the example regenerated.

#### Justification

Examples in a published IG are normative-by-influence: implementers copy
them, and downstream profiles and test suites reference them. If a
domain expert (the reporter is a recognized lab-data SME) has compiled a
written defect list, the lowest-risk action is to apply those
corrections so the IG's example accurately reflects FHIR R4/R5
Observation conformance and Public Health expectations for newborn
bilirubin reporting. There is no spec-level change required — only
example-instance corrections — so the change cannot break conformant
implementations and only improves implementer guidance.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than patching the single instance in place, replace the existing
bilirubin example with a small **example set** that demonstrates the
clinically meaningful variants the PDF likely highlights:

1. A **total serum bilirubin** Observation for a neonate (LOINC
   58941-6 *Bilirubin.total [Mass/volume] in Serum or Plasma --Neonatal*,
   value with UCUM `mg/dL`, age-appropriate reference range, `category =
   laboratory`, full `subject`/`effective`/`performer`/`specimen`
   linkages).
2. A matching **direct bilirubin** Observation (LOINC 1968-7) on the same
   specimen, demonstrating `Observation.specimen` reuse.
3. A **transcutaneous bilirubin** Observation (LOINC 76981-7), to
   illustrate the non-invasive measurement modality common in neonatal
   workflows.

Cross-link the three with a parent panel Observation (`hasMember`) so the
example also documents the panel pattern. Update the IG narrative page
that references the bilirubin example to point at the new set and call
out the differences.

#### Justification

The original ticket is "multiple issues" — that phrasing suggests the
defects aren't a single typo but a pattern of underspecification.
Rebuilding the example as a small didactic set is more durable than
patching one instance: it covers the modalities clinicians actually
encounter, gives implementers test data for several LOINC/UCUM
combinations, and demonstrates the panel/hasMember relationship that
laboratory implementers frequently get wrong. It carries marginally more
authoring effort than Disposition A but produces lasting value beyond
this ticket and reduces the chance of follow-up "the example is still
wrong for X" tickets in the next cycle.

---

### Disposition C: Decline

#### Proposal

Decline the change *as filed* and request that the reporter re-submit
with the defect list inline in the ticket (or as itemized child tickets,
one per defect), rather than as a PDF attachment. Mark FHIR-46853 as
"Not Persuasive — Need More Information" until the workgroup has a
machine-readable list of the asserted issues.

#### Justification

Defensible only on process grounds, not substantive ones. Accepting a
"see attached PDF" change request as-is creates a poor audit trail:
disposition tracking, ballot reconciliation, and downstream
implementer-facing change logs all rely on the issue text being legible
in Jira itself. Without the defect list in the ticket body, the
workgroup cannot triage piecemeal, cannot assign individual items to
different sub-resolvers, and cannot demonstrate to balloters what
specifically was changed. Declining-pending-restructure protects
ballot-reconciliation hygiene and is consistent with how other HL7
workgroups handle attachment-only change requests. The substantive
defects, if real, can be re-filed with full visibility.

---

### Recommendation

**Recommended disposition:** A

The reporter is a credible lab-data SME and the change is scoped to an
*example instance* in an IG, not to normative spec content. That makes
the cost of accepting the corrections low, the risk to implementers
essentially zero, and the upside (a correct, copy-paste-safe newborn
bilirubin example) clearly worth taking. Disposition B is attractive but
expands the ticket's scope; it should be considered only if, on opening
the PDF, the workgroup finds that the defects span enough modalities
(total/direct/transcutaneous, panel structure, reference ranges) that a
single corrected instance can't represent them coherently — in that
case, fold B in as the implementation of A. Disposition C should be
avoided: penalizing a substantive contributor on an attachment-format
technicality discourages the kind of detailed lab-domain review the
MIHR IG benefits from. Before final disposition, the lead reviewer
should download the PDF from Jira and paste its enumerated defect list
into a comment on FHIR-46853 so the disposition record is
self-contained.
