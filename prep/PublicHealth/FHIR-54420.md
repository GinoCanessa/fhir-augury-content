## Summary

The reporter observes that the eICR (Electronic Case Reporting / FHIR) Organism Susceptibility Observation profile extensibly binds `Observation.code` to the US Core Laboratory Test Codes value set (`http://hl7.org/fhir/us/core/STU6.1/ValueSet-us-core-laboratory-test-codes.html`), which encompasses the entire universe of laboratory test LOINCs. For an antimicrobial susceptibility observation, this binding is far too broad: only LOINCs that actually represent susceptibility testing (phenotypic, and possibly genotypic) are appropriate. The reporter recommends asking Regenstrief (LOINC's steward) for a curated value-set definition scoped to susceptibility tests so the eICR profile can bind to a clinically meaningful concept domain rather than the full lab-test catalog.

The single cross-reference recorded for this ticket is the FHIR artifact `Observation.code`, which confirms the change is targeted at the `code` element on the Organism Susceptibility Observation profile rather than at any structural change to Observation itself. No related Jira tickets, Zulip discussions, or GitHub items were discoverable through the augury index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the binding on the eICR Organism Susceptibility Observation profile so that `Observation.code` is bound (extensibly) to a new, narrowly-scoped value set covering only susceptibility test LOINCs — minimally the phenotypic AST panel codes, optionally extended to genotypic resistance testing codes. Engage Regenstrief / the LOINC committee to author or endorse the value-set definition (e.g., a LOINC-defined intensional expression scoped to AST-related properties) and host it either in the eICR IG itself or as a referenced LOINC-managed value set. Replace the current reference to `us-core-laboratory-test-codes` on this profile only; leave other eICR observation profiles unchanged.

#### Justification

The current binding inherits a value set sized for "any US Core lab test", which provides essentially no constraint for an Organism Susceptibility Observation and gives implementers no guidance about what `code` should carry. Narrowing the binding to susceptibility-test concepts aligns the profile's `code` with its `value[x]` (typically a susceptibility interpretation such as S/I/R) and matches the Observation profile's stated clinical purpose. Asking Regenstrief is the correct path because LOINC already curates the AST families and is the authoritative steward; an IG-defined ad hoc enumeration would risk drift from LOINC. Keeping the binding extensible (rather than required) preserves room for newer susceptibility codes that emerge between LOINC releases and avoids breaking implementers using less common but valid susceptibility LOINCs.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring a brand-new value set, change the binding to the existing LOINC-curated grouping for susceptibility tests — i.e., bind `Observation.code` to a value set defined by intensional inclusion of LOINCs whose `COMPONENT` matches "Susceptibility" (or equivalent LOINC class/parent such as the `MICRO` class with method `Susc*`). Define this value set inside the eICR IG with an intensional `compose.include.filter` against LOINC, and bind the profile's `code` to it (extensible). Additionally, add an invariant or guidance note instructing implementers that, when reporting genotypic resistance, an alternative `Observation.code` from the molecular-resistance LOINC family is acceptable.

#### Justification

This avoids the latency and external dependency of a Regenstrief request and lets the eICR project ship the constraint in its next ballot cycle. An intensional LOINC filter also self-maintains: as LOINC adds new susceptibility codes they automatically fall into the value set without an IG republish. The trade-off versus Disposition A is governance — an IG-defined intensional set is less authoritative than a LOINC-stewarded one — but for an extensible binding the practical constraint on implementers is equivalent. Splitting phenotypic vs. genotypic via guidance rather than two value sets keeps the profile shape simple while still acknowledging the reporter's "phenotypic at minimum, genotypic possibly" framing.

---

### Disposition C: Decline

#### Proposal

Keep the existing extensible binding to `us-core-laboratory-test-codes`. Document, either in the profile's `code` element definition or in narrative guidance on the Organism Susceptibility Observation page, the expected subset of LOINCs implementers should use for susceptibility reporting (with examples), but do not change the formal binding.

#### Justification

An *extensible* binding does not force implementers to use the listed value set when no suitable code exists; it requires them to use it when one does. Because US Core's lab-test value set already contains susceptibility LOINCs, conformant systems will land on appropriate codes today, and the perceived "breadth" of the value set is largely a documentation concern. Replacing the binding adds maintenance burden (a new value set to steward, plus possible Regenstrief coordination) and risks blocking valid edge-case LOINCs. This option is defensible if the workgroup judges that the reporter's concern is best addressed through guidance rather than a normative binding change.

---

### Recommendation

**Recommended disposition:** B

Disposition B delivers the substantive narrowing the reporter is asking for — a value set actually scoped to susceptibility testing — without taking on the schedule risk and external dependency of waiting for Regenstrief to publish a curated set (Disposition A). An intensional LOINC filter authored inside the eICR IG can be reviewed and balloted on the workgroup's own cadence and will pick up new susceptibility LOINCs automatically, which is a meaningful operational win for a Public Health IG that ingests data from a long tail of laboratory systems. Disposition A remains a good follow-on if the workgroup wants to migrate the binding to a Regenstrief-stewarded set in a future release, and Disposition C is unattractive because the reporter's point — that an extensible binding to "all US lab tests" provides effectively no guidance for an Organism Susceptibility Observation — is well taken and is exactly the kind of clarity an IG should provide.
