## Summary

FHIR-42185 is a one-line BSeR ballot comment — imported into Jira as
"Comment 2 - imported by: Ron G. Parker" — asking whether *the* Code
System referenced by the comment should be anchored under
`terminology.hl7.org` (THO) rather than under the BSeR IG's own
canonical (`hl7.org/fhir/us/bser/CodeSystem/...`). The ticket itself
does not name the specific Code System; the question is the same
generic policy question raised against BSeR multiple times in the same
ballot batch and tagged `BSeR_THO`.

Sibling [FHIR-42184](https://jira.hl7.org/browse/FHIR-42184) ("Comment
1 - imported by: Ron G. Parker") is the immediately preceding ballot
comment in the same series and carries identical metadata (BSeR,
Public Health, `BSeR_THO`, Highest, Triaged), so the two should
almost certainly be dispositioned together. The same comment template
was also imported against the CRMI specification as
[FHIR-42229](https://jira.hl7.org/browse/FHIR-42229) and
[FHIR-42230](https://jira.hl7.org/browse/FHIR-42230); both were
resolved **Not Persuasive** with the rationale that, because the
relevant terminology is bound `required` from a normative/ballot
artifact, it is "ballot-bound" and therefore correctly anchored in
`hl7.org/fhir` rather than THO. The older
[FHIR-35619](https://jira.hl7.org/browse/FHIR-35619) (Gravity / US
SDOH Clinical Care) is the canonical precedent for the question and
remains open pending review of which code systems are mature/reusable
enough to be migrated to THO with each new IG release.

## Proposed Dispositions

> The ticket does not name the specific Code System under review.
> Each disposition below assumes that, when worked, Public Health
> will identify the BSeR Code System(s) flagged by Comment 2 (the
> `BSeR_THO`-labelled batch makes the working list explicit) and
> apply the chosen disposition consistently to them, in coordination
> with sibling FHIR-42184.

### Disposition A: Accept As Requested

#### Proposal

Migrate the BSeR Code System(s) referenced by this comment from the
IG-local canonical (`http://hl7.org/fhir/us/bser/CodeSystem/<id>`) to
the Unified Terminology Governance (UTG) namespace
(`http://terminology.hl7.org/CodeSystem/<id>`). Concretely:

1. Open a UTG ticket per affected Code System to register it in THO,
   transferring stewardship metadata (publisher, copyright,
   maintainer) from the BSeR IG.
2. Replace the `CodeSystem.url` and any in-IG `ValueSet.compose.include.system`
   references in the BSeR IG sources with the new
   `terminology.hl7.org/CodeSystem/...` canonical.
3. Add a deprecation/redirect note on the old IG-local canonical so
   existing implementers can find the new home.
4. Coordinate with the BSeR sibling tickets tagged `BSeR_THO`
   (starting with FHIR-42184) so the migration is performed as a
   single batch rather than per-comment.

#### Justification

This is the long-standing HL7 stewardship pattern: code systems that
are reusable beyond a single IG belong in THO, where they can be
versioned and consumed independently of BSeR's publication cadence.
The Gravity precedent in FHIR-35619 is explicit that the migration
is *expected* once code systems mature ("we will revisit moving
completed code systems / value sets to THO with the next release of
the IG"). For BSeR, where the eReferral workflow is intended to be
reused across multiple referral domains, anchoring shared code
systems in THO reduces churn for downstream IGs.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the Code System anchored in the BSeR IG canonical for now, but
formally classify each affected Code System against the
THO-vs-IG-local criteria and document the decision per Code System
in the BSeR IG. Specifically:

1. Build a small inventory of BSeR-defined Code Systems (the
   `BSeR_THO`-labelled tickets are effectively that worklist).
2. For each, decide based on (a) whether the Code System is bound
   `required` from a ballot artifact, (b) whether it is reused
   outside BSeR, and (c) whether its content is stable.
3. Migrate to THO only the Code Systems that are reusable and
   stable; for the rest, retain the `hl7.org/fhir/us/bser` canonical
   and add a note in the IG explaining the choice.
4. Resolve FHIR-42185 (and FHIR-42184) with that per-system
   disposition recorded in the resolution.

#### Justification

The CRMI siblings FHIR-42229 / FHIR-42230 were resolved Not
Persuasive precisely because *required-bound* terminology used by
ballot artifacts is treated as ballot-bound and is allowed (indeed
preferred) to stay under `hl7.org/fhir`. Some BSeR Code Systems will
fall in that category and some will not; a one-size-fits-all
migration risks either dragging immature/IG-scoped vocabulary into
THO or leaving genuinely shareable vocabulary stranded under BSeR.
A documented per-system decision avoids both failure modes and
gives implementers a clear rationale.

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-42185 as **Not Persuasive / No Change** on the basis
that the BSeR Code Systems in question are bound `required` from
BSeR profiles/extensions and are therefore ballot-bound terminology
that is correctly anchored in `hl7.org/fhir/us/bser`. Reference
FHIR-42229 and FHIR-42230 in the resolution text as the controlling
precedent.

#### Justification

If the affected Code Systems are (a) only consumed by the BSeR IG,
(b) bound `required` from ballot artifacts, and (c) versioned in
lockstep with the BSeR IG release, then the CDS WG's 2023-09-11
disposition of the analogous CRMI tickets applies directly: such
terminology belongs under the IG's canonical, not THO. Declining is
also defensible on resourcing grounds — the import added a large
batch of single-line `BSeR_THO` tickets without naming the affected
Code Systems, and absent a curated list, blanket migration is
disproportionate to the documented harm.

---

### Recommendation

**Recommended disposition:** B

The BSeR_THO ballot batch is a policy question masquerading as many
identical tickets, and neither blanket Accept nor blanket Decline is
defensible across all of the BSeR-defined Code Systems. Disposition A
(blanket migrate) ignores the CRMI precedent that
required-and-ballot-bound terminology is allowed to live under
`hl7.org/fhir`; Disposition C (blanket decline) ignores the Gravity
precedent that mature, reusable Code Systems are expected to be
migrated to THO with each IG release. Disposition B forces Public
Health to do the small amount of curation needed — list the BSeR
Code Systems, classify each one, and decide per-system — which is
the only way the disposition can be defended to ballot reviewers and
the only way the resolution will scale across FHIR-42184/42185 and
the rest of the `BSeR_THO`-tagged batch. The classification can
piggyback on the next BSeR publication cycle, mirroring how Patient
Care framed FHIR-35619.
