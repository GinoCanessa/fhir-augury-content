## Summary

In the proposed BSeR (US Bidirectional Services eReferrals) exchange model,
the referral *source* prepares a Bundle of allowed content and pushes it to
the referral *recipient*. Lloyd McKenzie observes that this push-only
pattern does not fit a RESTful exchange, where the natural pattern is for
the recipient to query the source. The complication is that the recipient
is not trusted to determine, on its own, which content it is permitted to
receive. The ticket asks BSeR to surface an alternative pattern in which
the recipient queries broadly (e.g., for the resources it wants, or via
`$everything`), and the source uses a previously-agreed query Bundle (the
"filter" describing what the recipient is allowed to receive) to constrain
what it actually returns. The core question the ticket raises is *where
that filter definition should live* — i.e., which artifact carries the
allowed-content definition, who curates it, and how the source resolves it
at query time.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the augury data sources for this ticket, so the request stands
on its own without surrounding workgroup context.

## Details

> and preps a bundle that they push to the referral recipient. However, in
> a RESTful world, that doesn't fly. Instead, the referral recipient must
> query for the data. However, the referral recipient (presumably) isn't

> invoke $everything), and the referral source then retrieves the query

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative section to the BSeR IG describing a RESTful "pull"
companion to the existing push pattern, anchored on a server-side
**allowed-content filter** expressed as a query Bundle:

- Define the filter as a `Bundle` of type `batch` (or `collection` of
  `Parameters`/`SearchParameter` entries) whose `entry.request` URLs
  enumerate the searches the recipient is permitted to receive results
  for. Persist this artifact on the referral source, keyed by the
  combination of (referral `ServiceRequest`, recipient organization, and
  service category).
- Specify that the recipient may either (a) execute any of the queries in
  that Bundle directly, or (b) invoke `Patient/[id]/$everything` (or a new
  BSeR-specific operation) on the source.
- Require the source, on receiving a query from the recipient, to resolve
  the applicable allowed-content Bundle and apply it as a server-side
  filter — returning only the intersection of what the recipient asked
  for and what the Bundle authorizes.
- Reference the Bundle from the referral itself via a
  `ServiceRequest.supportingInfo` (or new BSeR extension) pointing at the
  filter resource so the recipient can discover what is in scope.

#### Justification

This is exactly what the ticket describes and addresses the legitimate
RESTful-architecture concern Lloyd raises: the recipient drives the
exchange (which is what FHIR REST assumes), while the source remains the
authority on what is permissible to disclose. Locating the filter on the
source (and merely *referenced* from the referral) keeps the trust
boundary in the right place — the recipient never has to be trusted to
self-limit. Modeling the filter as a Bundle of search requests reuses an
existing FHIR construct rather than inventing new machinery, which is
consistent with how FHIR has historically expressed "a set of queries"
(e.g., `GraphDefinition`, the batch Bundle pattern used by `$everything`
implementations).

---

### Disposition B: Alternative Approach

#### Proposal

Instead of standing up a new BSeR-specific filter Bundle, leverage existing
FHIR access-control / consent machinery and define BSeR's contribution as
*profiling* and *workflow guidance*:

- Express the allowed-content scope as a **`Consent`** resource (with
  `provision` entries enumerating permitted resource types, codes, and
  date ranges) rather than as a query Bundle. Profile a BSeR
  `Consent` for "referral disclosure scope".
- The recipient queries normally (`$everything`, type-level search, etc.).
  The source enforces the `Consent` on response — which is a pattern
  already understood by FHIR servers implementing the Security &
  Privacy / Consent IGs.
- Optionally, pair this with a BSeR `OperationDefinition` (e.g.,
  `$bser-export`) that wraps `$everything`-style behavior and documents
  that the response is filtered by the referenced `Consent`.
- Keep the existing push flow as an option for partners that cannot expose
  a queryable endpoint.

#### Justification

The underlying need — "the source filters what the recipient can see" — is
the canonical use case for `Consent`-driven access control, and reusing
that machinery means BSeR inherits tooling, validators, and security
review work that has already been done elsewhere (Da Vinci, IPA, Bulk
Data, SMART). Modeling the filter as a query Bundle (Disposition A) risks
re-implementing access control in an artifact that wasn't designed for
it, and creates ambiguity about precedence when the recipient's query
overlaps but does not match the Bundle's literal request URLs (e.g., the
recipient searches `Observation?category=vital-signs&date=ge2024` but the
Bundle has `Observation?patient=X&category=vital-signs`). A `Consent`
provision evaluates per-resource, which side-steps that
query-shape-matching problem entirely. The trade-off is that
implementations must support `Consent`-based filtering, which is a higher
bar for the smaller referral-management vendors BSeR targets.

---

### Disposition C: Decline

#### Proposal

Decline as **out of scope for the current BSeR ballot cycle**. Close with
guidance that BSeR's current push-Bundle pattern remains the only
normative exchange pattern for this version, and defer the RESTful-pull
companion to a future version once a partner ecosystem with REST-capable
referral sources exists to validate it.

#### Justification

The ticket is, in effect, a request to add a second, fundamentally
different exchange topology (pull instead of push) and to design a new
authorization artifact to accompany it. That is a substantial body of
work that touches security review, conformance testing, and partner
implementations — none of which can be completed by tweaking the existing
push artifacts. Without at least one identified implementer ready to
build the pull side, the workgroup risks balloting a speculative pattern
that will need to be revised once real implementations surface concerns.
Declining now and tracking the requirement as a future-version item is a
defensible way to keep BSeR shippable.

---

### Recommendation

**Recommended disposition:** B

The architectural concern Lloyd raises is real and should not be
declined: a referral exchange that only works push-only will be
out-of-step with every other US Realm exchange IG by the next ballot
cycle, so Disposition C only delays the inevitable. Between A and B,
Disposition B is the better starting point because the problem the
ticket frames — "the source must filter what the recipient is allowed to
receive" — is structurally an *access-control* problem, not a
*query-rewriting* problem, and FHIR already has a resource (`Consent`)
purpose-built for that. Building the filter as a literal query Bundle
(A) couples the authorization decision to the exact shape of the query
the recipient happens to send, which is fragile in practice. The
workgroup should discuss whether `Consent` (B) is implementable by the
typical BSeR source today; if it is not, a hybrid that profiles a
simpler "allowed resource types + codes" structure (still server-side,
still on the source, but lighter than full `Consent`) would be a
reasonable refinement of B and is preferable to A.
