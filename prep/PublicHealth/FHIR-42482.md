## Summary

Lloyd McKenzie reports that the BSeR (US Bidirectional Services eReferrals)
implementation guide does not adequately introduce or explain its
batch-based query definition approach until the very last page of the
specification. He is asking that the architectural pattern — using a set
of bundled `x-fhir-query` queries to define what data is to be returned
for a referral — be described in narrative text and surfaced much earlier
in the IG. He further requests that:

- The data-element tables that already enumerate the information to be
  retrieved should also expose the technical filter expressions carried
  in the example Bundles, so readers can see the actual query semantics
  alongside the human-readable list.
- The IG should explicitly call out that R5 `x-fhir-query` syntax is
  being used to define the filters, and should mention `now()`-based
  date filtering as an available capability.
- The undefined query parameters `patient-id-baby` and
  `patient-id-mother` are flagged as a gap: they are referenced but
  never defined, and implementers will instead have to use FHIRPath /
  reference navigation to traverse from the baby's record to the
  mother's (or vice versa). The reporter also raises an unresolved
  question about how the query pattern behaves when there are multiple
  children associated with a single mother.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure the BSeR IG narrative so the batch-query architecture is
introduced up front and reflected consistently throughout:

1. Add a new top-level "Querying for Referral Data" (or equivalent)
   page near the start of the IG (e.g., just after the IG home /
   conformance overview) that explains:
   - The conceptual model: each BSeR referral profile defines its
     payload as a *batch of `x-fhir-query` requests* carried in an
     example Bundle, which servers execute to assemble the data set to
     return.
   - That the query expressions use the **R5 `x-fhir-query` syntax**,
     with a forward-link to the R5 definition.
   - That date-bounded filters can use `now()` (and other supported
     date functions), with at least one worked example
     (e.g., `date=ge{{%now-1y}}` or the appropriate BSeR pattern).
2. For every BSeR-specific data-element table that currently lists
   "what is retrieved," add a column (or an adjacent rendering) that
   shows the **actual `x-fhir-query` filter** drawn from the example
   Bundle, so readers see the technical filter beside the
   human-readable description rather than only on the final page.
3. Define the currently-undeclared query parameters
   `patient-id-baby` and `patient-id-mother` explicitly:
   - Document each as a named placeholder used by the batch queries.
   - Show how a server resolves them via reference / FHIRPath
     navigation (baby→`Patient.link` or `RelatedPerson` → mother, and
     vice versa).
   - Add explicit guidance for the multi-child case: how the batch is
     expected to behave when one mother has multiple children
     (e.g., one Bundle per child, server iterates, or unsupported and
     out of scope — the WG must pick).
4. Cross-link the new architecture page from the existing "last-page"
   batch-definition section so it is reachable from both directions.

#### Justification

The reporter is explicit that the architectural model is essential to
understanding the IG and is currently buried. Surfacing it early is a
documentation-only change for items 1–3 and matches standard IG
authoring practice (architecture/conformance pages precede artifact
pages). Item 4 (defining `patient-id-baby` / `patient-id-mother` and
the multi-child behavior) is *not* purely editorial — the IG cannot
claim conformance for queries whose parameters are undefined, so this
is a substantive interoperability gap that the WG should close
regardless. Calling out R5 `x-fhir-query` and `now()` is also a
correctness fix: implementers reading only the data tables today have
no way to know which query dialect they must support.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of expanding narrative and table content in place, restructure
BSeR's payload-definition mechanism to use a more declarative,
machine-readable artifact, and let the narrative wrap around it:

1. Introduce (or adopt) a small companion artifact per referral type —
   e.g., a `Library` or a `GraphDefinition` — that formally captures
   the set of `x-fhir-query` expressions that today live only inside
   the example Bundles. This becomes the normative source of truth for
   "what is retrieved."
2. Generate the data-element tables (and the technical-filter column
   requested in Disposition A) **from** that artifact, so the
   human-readable view and the executable view cannot drift.
3. Define `patient-id-baby` and `patient-id-mother` as named
   parameters on that artifact (e.g., `Library.parameter` or a
   `GraphDefinition` start parameter), with documented resolution
   rules and a defined behavior for multi-child scenarios (e.g., the
   artifact is invoked once per child Patient).
4. Keep the early architecture page from Disposition A, but point it
   at the new artifact rather than embedded prose tables.

#### Justification

The reporter's underlying complaint is that the *connection* between
the narrative tables and the actual query semantics is invisible until
the final page. A declarative artifact addresses that root cause:
implementers get a single, parseable description of the batch query,
and the IG narrative can be regenerated rather than hand-maintained,
which avoids the same drift problem recurring in future versions. It
also gives `patient-id-baby` / `patient-id-mother` and the multi-child
case a formal home (parameter typing, cardinality, documentation)
rather than relying on prose. Trade-off vs. A: more authoring effort
up front and a learning curve for IG readers familiar only with
example Bundles, but better long-term maintainability and a stronger
conformance story.

---

### Disposition C: Decline

#### Proposal

Decline the request as filed and close as "works as intended" /
"editorial only, deferred." Rationale category: the batch-query
mechanism is already documented (on the final page the reporter
reached), the example Bundles are the canonical source for the filters,
and the undefined `patient-id-baby` / `patient-id-mother` symbols are
intentional placeholders that implementers are expected to bind from
context.

#### Justification

Defensible only on narrow grounds: if the WG considers the BSeR IG
frozen for the current ballot cycle and has no cycles to restructure
narrative ordering, declining preserves stability and avoids reopening
the payload-definition design. However, this position does not address
the substantive gap that `patient-id-baby` and `patient-id-mother` are
referenced but undefined and that multi-child behavior is unspecified
— those are interoperability issues, not editorial preferences, so a
pure decline leaves real implementer ambiguity in place.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly addresses what the reporter — an experienced FHIR
methodologist — actually asked for: move the batch-query architecture
explanation to the front of the IG, expose the technical filters
alongside the data-element tables, name the query dialect (R5
`x-fhir-query`) and its date-filter capabilities, and close the
`patient-id-baby` / `patient-id-mother` / multi-child gaps. Most of the
work is documentation, which the Public Health WG can land without a
substantive design change, and the parts that are *not* documentation
(defining the undeclared parameters and the multi-child semantics) are
real conformance gaps that need to be resolved regardless of which
disposition wins. Disposition B is attractive long-term but is a
significant architectural change for what the reporter framed as a
narrative/clarity problem, and it would expand the scope well beyond
the original request; it could be pursued as a follow-on ticket if the
WG sees value. Disposition C should be avoided because the undefined
query parameters are not just an editorial nit — they leave
implementers without a normative way to traverse the mother↔baby
relationship, especially in the multi-child case the reporter
specifically flags.
