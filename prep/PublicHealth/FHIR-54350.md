## Summary

The reporter requests that the Electronic Case Reporting (eCR FHIR) IG
add explicit guidance on how to populate `Observation.specimen` for
laboratory-result observations carried in an eICR/RR — specifically for
microbiology results where two distinct specimens commonly exist:

1. The **original sample** collected from the patient (e.g., throat
   swab, blood culture bottle, sputum), and
2. The **derived "isolate" / organism specimen** that represents the
   organism actually identified and tested (e.g., the cultured isolate
   on which susceptibility testing is performed).

The ticket asks the work group to decide which specimen `.specimen`
should reference, and — if the derived isolate is the correct target —
whether the IG should also constrain or describe the relationship
between the original and derived specimens (e.g., via
`Specimen.parent`) within the eCR FHIR payload. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were located that already
discuss this question for eCR.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative-style guidance to the eCR FHIR IG profile(s) that
constrain laboratory-result `Observation` resources used in the eICR
(notably the microbiology / organism-identification observations) that:

- For microbiology Observations whose `Observation.code` represents an
  organism identification or antimicrobial-susceptibility result,
  `Observation.specimen` **SHALL** reference the **derived isolate
  Specimen** (the cultured organism actually tested), not the original
  patient-collected sample.
- For all other laboratory-result Observations, `Observation.specimen`
  references the **original collected Specimen**.
- When an isolate Specimen is referenced, the isolate `Specimen`
  instance **SHALL** populate `Specimen.parent` to point at the
  original collected `Specimen`, and **SHOULD** populate
  `Specimen.type` with a code distinguishing the isolate (e.g., an
  appropriate SNOMED CT "isolate" concept) from the parent's
  collection-site code.
- Add a narrative section ("Representing organism isolates in eICR
  microbiology results") to the IG with at least one worked example
  bundle that includes the patient, the original Specimen (e.g.,
  throat swab), the derived isolate Specimen with `parent` populated,
  and the organism-ID / susceptibility Observations referencing the
  isolate.

#### Justification

This matches established FHIR practice: `Specimen.parent` exists
specifically to model derivation chains (aliquots, isolates, pooled
samples), and base-spec guidance already says
`Observation.specimen` should reference "the specimen the observation
was made on," which for organism-ID and susceptibility testing is the
isolate, not the swab. Making the rule explicit eliminates ambiguity
across senders, lets receivers (public-health jurisdictions)
deterministically reason about both the patient-side collection
context (via `Specimen.parent`) and the tested material (via the
isolate), and is consistent with how US-Core / lab IGs typically
handle aliquots. It also avoids the failure mode where two senders
populate `.specimen` differently for the same logical result, which
today makes downstream organism analytics unreliable.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mandating that `.specimen` always point at the isolate,
adopt a **"point at the most specific Specimen the lab actually
tested, and always populate `Specimen.parent` when a derivation
exists"** rule, expressed as guidance (SHOULD-level) plus profile
invariants:

- `Observation.specimen` SHOULD reference the most specific Specimen
  the laboratory tested for that Observation (which for
  organism-identification and susceptibility is the isolate, but for
  e.g. a Gram stain on the original swab is the original specimen).
- Whenever a referenced Specimen is derived from another (isolate,
  aliquot, pooled), `Specimen.parent` SHALL be populated.
- Provide a decision-tree figure in the IG mapping common
  microbiology workflows (culture → isolate → ID → AST; direct
  molecular detection on primary specimen; serology) to which
  Specimen `.specimen` should reference.
- Do **not** constrain `Specimen.type` to a fixed isolate value set —
  defer to the lab's coding while encouraging SNOMED CT.

#### Justification

The original-vs-isolate question is not actually binary: a Gram stain
or direct molecular test legitimately points at the original
specimen, while culture-based organism ID points at the isolate. A
hard "always isolate" rule (Disposition A) would be wrong for those
cases and would push senders to either fabricate isolate Specimens
for non-culture workflows or violate the rule. This approach gets the
same downstream benefit (receivers can always walk
`Observation.specimen` → `Specimen.parent*` to find the patient
collection) without overconstraining workflows that genuinely test
the original specimen. It also keeps the IG aligned with the way
LIS systems already structure microbiology results internally.

---

### Disposition C: Decline

#### Proposal

Decline to add eCR-specific guidance and instead rely on the FHIR
core `Observation.specimen` / `Specimen.parent` definitions. Resolve
the ticket as "Persuasive with Modification — no IG change," with a
short FAQ entry (or wiki note) pointing senders at the existing core
definitions and at any equivalent guidance already published by US
Core / Laboratory Report / Public Health Lab Reporting IGs.

#### Justification

`Specimen.parent` and the base `Observation.specimen` definition
already describe derivation semantics, and microbiology
isolate-vs-original is a general lab-reporting concern, not unique to
eCR. Encoding eCR-specific rules risks divergence from US Core Lab,
the eventual HL7 Laboratory Report IG, and CDC's lab reporting
profiles, which would create reconciliation work later and could push
senders to maintain two different specimen-population rules for the
same underlying data. Declining keeps the eCR IG lean and defers the
canonical guidance to whichever IG owns lab-result modeling.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the real-world workflow accurately: eCR
microbiology results genuinely span both "tested on the original
specimen" (Gram stain, direct molecular) and "tested on the derived
isolate" (organism ID, AST), and a single hard rule (A) will
mis-model one or the other. At the same time, the question Craig
raised is a real interoperability gap — declining outright (C) leaves
two senders free to point `.specimen` at different things for the
same logical result, which is exactly the situation public-health
receivers cannot reconcile today. B threads that needle: keep
`.specimen` semantically faithful to "what the lab actually tested,"
make the derivation chain explicit and required via `Specimen.parent`
whenever it exists, and give implementers a worked decision tree so
the SHOULD is operationally unambiguous. The work group should
confirm the SHOULD-vs-SHALL strength on `Specimen.parent` and decide
whether the decision tree lives in the eCR IG or is contributed
upstream to a shared lab-reporting page.
