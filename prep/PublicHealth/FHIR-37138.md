## Summary

The reporter (Joan Harper, originally raised by Ron G. Parker as
"Comment 3") asks that the canonical `url` of the value set
`http://www.hl7.org/fhir/us/mihr/ValueSet/hypertensive-disorders-pregnancy-ICD`
be moved off the `hl7.org/fhir` namespace and anchored under
`terminology.hl7.org` (THO) instead. Per TSMG guidance cited in the
ticket, only value sets used as a **required** binding on a `code` data
type should remain in the `hl7.org/fhir` namespace; everything else —
which includes this ICD-coded value set — belongs in THO and must be
registered there via a UTG ticket.

This ticket is one of an identified series of identical change requests
filed against the same MIHR IG, each targeting a different value set
(see Related Jira Tickets — FHIR-37136 through FHIR-37144 plus
FHIR-37151..37156, FHIR-37164, FHIR-37193, all surfaced as
keyword-similar siblings with score >117). The series was originally
dispositioned **Considered for Future Use** on 2022-06-02 (Ruby Nash /
Sarah Gaunt: 21-0-2) "based on the maturity of the IG and when the
draft THO process has been finalized," then re-opened by Lynn Laakso
on 2023-03-29 with the note "re-open post STU publication." It is now
labeled **Block-Vote-1**, indicating Public Health intends to dispose
of the whole batch together.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the canonical URL of the value set currently published as

```
http://www.hl7.org/fhir/us/mihr/ValueSet/hypertensive-disorders-pregnancy-ICD
```

to a THO-namespaced canonical of the form

```
http://terminology.hl7.org/ValueSet/hypertensive-disorders-pregnancy-ICD
```

(or whatever code/short-name TSMG approves at registration time), and:

1. File a companion UTG ticket against **HL7/UTG** to register the
   value set in THO with the new canonical, the existing
   ICD-coded compose, and the existing metadata (`name`, `title`,
   `description`, `experimental`, jurisdiction, etc.).
2. In the **MIHR IG** repository, replace the in-IG value set
   definition with a `ValueSet`-pointer/usage that references the THO
   canonical (or remove it entirely once THO publishes), and update
   every binding that currently references the old `us/mihr/...`
   canonical to reference the THO canonical instead.
3. Apply the same treatment in lockstep to the rest of the
   Block-Vote-1 batch (FHIR-37136..37144 and the apparent extended
   set FHIR-37151..37156 / 37164 / 37193) so the IG remains
   internally consistent.
4. Coordinate the publication so that the THO entry is live before
   the next MIHR republish that drops the old canonical, to avoid a
   period where existing instances point at a 404 canonical.

#### Justification

This is the literal ask in the ticket and it tracks the explicit TSMG
policy cited in the description: value sets used outside a `code`-type
required binding belong in THO, not in `hl7.org/fhir`. The
ICD-coded "hypertensive disorders of pregnancy" value set is
domain-level terminology that other IGs (and MIHR's own profiles)
should be able to reuse, which is exactly what THO is for. The 2023
re-open note ("re-open post STU publication") suggests the original
"Considered for Future Use" deferral was driven by IG maturity and
the draft-state of the THO process — both of which have since
advanced, removing the original blocker. Treating the whole
Block-Vote-1 batch as a single coordinated migration also matches the
intent of the label.

---

### Disposition B: Alternative Approach — Accept the Move, but Stage It Through a Persistent Redirect

#### Proposal

Same end-state as Disposition A (canonical lives in THO), but stage
the cut-over so the existing `hl7.org/fhir/us/mihr/ValueSet/...`
canonicals continue to resolve for at least one full STU cycle:

1. Register the value set in THO (UTG ticket) under a new
   THO canonical, exactly as in A.
2. In the MIHR IG, **keep** the existing `us/mihr/ValueSet/
   hypertensive-disorders-pregnancy-ICD` entry but mark it
   `status = retired`, set its `compose` to a single `import` of the
   THO canonical, and add a `replacedBy` extension /
   `ValueSet.relatedArtifact` entry pointing to the THO version.
3. Update profile bindings inside MIHR to reference the THO canonical
   directly — so new instances are validated against THO — but allow
   existing instances pointing at the old canonical to keep
   resolving via the retired (import-only) value set.
4. Apply the same staged pattern to all Block-Vote-1 siblings.

#### Justification

The MIHR IG has already been STU-published (per Lynn Laakso's
re-open comment), so deployed instances may legitimately carry
`ValueSet` references to the old `us/mihr/...` canonicals in
`Coding.system` / extension URLs / DTR questionnaires / etc. A hard
re-anchor (Disposition A) silently breaks those references.
Disposition B accepts TSMG's direction and gets the value sets into
their proper namespace, but uses the retire-and-redirect pattern that
FHIR core itself uses for relocated artifacts, which is materially
less disruptive for implementers. The trade-off vs. A is more files
to maintain in the MIHR IG (a stub for every retired VS) and a
slightly more complex publication, in exchange for not breaking
existing MIHR consumers at the canonical-resolution layer.

---

### Disposition C: Decline (as worded) — Persuasive With Mod

#### Proposal

Decline to re-anchor on the grounds that **at least some** of the
value sets in the Block-Vote-1 batch may legitimately remain in the
`hl7.org/fhir/us/mihr` namespace if they are IG-specific
compositions (e.g., a narrow ICD-restricted subset that is only
meaningful inside MIHR) rather than reusable terminology assets.
Specifically, ask the work group to triage each VS in the batch
individually against TSMG's "reusable beyond this IG" test, and:

- For value sets that are reusable (the typical ICD/SNOMED/LOINC
  reference sets), apply Disposition A or B.
- For value sets that are genuinely MIHR-specific compositions, mark
  the ticket(s) **Persuasive With Mod** and document in the IG that
  the VS is intentionally IG-scoped.

#### Justification

TSMG's guidance is "only required bindings to `code` should stay in
`hl7.org/fhir`" — but that policy is about *where unique terminology
content is owned*, not about whether every IG-specific VS that
filters/composes external code systems must move. There is a real
risk in mechanically moving every VS in MIHR to THO that the work
group ends up registering hyper-specific clinical bundles in the
shared THO namespace where they crowd out more general assets and
acquire a maintenance contract they can't honour. This disposition
preserves the spirit of the TSMG ask while leaving the IG room to
keep narrow compositions in-IG. The cost is that the Block-Vote-1
batch can no longer be voted as a single unit, which is the
labelling intent — so this should only be picked if the work group
is willing to break the batch.

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach — Accept and
stage through a retired-with-import stub).**

The substantive ask is correct: value sets composed of external code
systems (here, ICD) and intended for cross-IG reuse belong in THO,
not in `hl7.org/fhir/us/mihr`. The 2023 re-open note explicitly
records that the original "Considered for Future Use" deferral was
about THO-process maturity, not about disagreement with the
direction, so an outright decline (C) would be a regression on the
work group's own prior intent and would also undermine the
Block-Vote-1 grouping. A literal Accept (A) is also defensible, but
because MIHR has already shipped at STU, a hard re-anchor risks
breaking deployed instances and DTR/CRD-style downstream consumers
that have already pinned the old canonical. Disposition B gets the
same end-state as A while keeping deployed clients resolving, which
is exactly the discipline FHIR core uses for relocated canonicals
and which the Public Health work group can defend on Zulip /
`#terminology` without re-litigating the core TSMG question. The
extra cost — one retire-and-import stub per VS in the batch — is
small and one-time, and it can be removed at the next MIHR major
version. Pair the disposition with a UTG companion ticket against
**HL7/UTG** for each VS and coordinate publication order so THO is
live before the MIHR republish.
