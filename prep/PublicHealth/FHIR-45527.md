## Summary

The ticket asks the VRDR work group to extend the US Vital Records Death
Reporting (VRDR) FHIR Implementation Guide so that the death data it
carries can be leveraged by the U.S. Food and Drug Administration (FDA)
as real-world data (RWD). Specifically, the reporter wants VRDR to
include use cases that support FDA active-surveillance workflows and
registry data needs. The ticket was originally dispositioned in May 2024
as "Considered for Future Use" (vote Saul Kravitz / Diana Wright, 23-0-0)
on the basis that, while the work group was open to engaging with FDA,
no concrete requirements had been provided. Lynn Laakso reverted that
resolution and re-opened the ticket post STU3 publication so it can be
revisited; no cross-referenced Jira tickets, Zulip threads, or GitHub
items were found in the augury index to add further context.

### Disposition A: Accept As Requested

#### Proposal

Add a dedicated "FDA Real-World Data Use Cases" section to the VRDR
Implementation Guide that enumerates concrete FDA RWD scenarios for
death data, covering at minimum: (1) active post-marketing safety
surveillance against mortality outcomes, and (2) registry data exchange
where death status / cause-of-death is a key endpoint. For each use
case, document the actor model (data submitter, FDA receiver),
triggering events, and the VRDR resources / profiles already in scope
that satisfy the exchange (e.g., `Bundle - Death Certificate Document`,
`Patient - Decedent`, `Observation - Cause of Death Part 1/2`,
`Observation - Manner of Death`, `Observation - Decedent Pregnancy
Status`). Where current VRDR profiles are insufficient, list them as
"gaps" so they can be split into follow-on change requests. Update the
IG's CapabilityStatement narrative to reference the new use cases and,
if FDA confirms specific query patterns, add corresponding
SearchParameter expectations.

#### Justification

Lynn Laakso's re-open comment makes clear the work group is open to
engaging with FDA; the original "Considered for Future Use" vote turned
on the absence of concrete requirements rather than a substantive
objection. Adding documented use cases is the lowest-risk first step:
it is normative-narrative-only (no profile breaking changes), it
provides a stable hook for FDA to confirm/refine requirements against,
and it discharges the ticket's literal ask ("Include Use Cases for FDA
RWD - Death Data"). The `Block-Vote-1` label suggests the work group
expects a tractable disposition that can be batch-voted, and a
narrative-only addition fits that profile.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding FDA-specific use cases directly into the core VRDR
IG, sponsor a **companion VRDR-RWD supplement** (a separate small IG
or VRDR appendix) that profiles VRDR resources for FDA active
surveillance and registry use. The companion would: (a) define a
`VRDR-RWD-Bundle` profile constraining the existing Death Certificate
Document for de-identified RWD submission, (b) add an `Observation`
profile or extension capturing cohort / exposure linkage identifiers
required by FDA Sentinel-style workflows, and (c) document conformance
expectations for registries reusing VRDR death data. Engage FDA
formally (e.g., via a connectathon track or joint Public Health / BR&R
call) to ratify the requirements before publishing.

#### Justification

FDA RWD requirements typically include de-identification, linkage
identifiers, and audit metadata that are out of scope for vital-records
jurisdictional reporting; folding them into core VRDR risks
overloading a specification whose primary audience is state vital
records offices. A companion supplement keeps the core VRDR profiles
stable for jurisdictions while still meeting the ticket's underlying
need, and it gives FDA a clean artifact to reference. The
work-group-of-record is already Public Health, so co-sponsorship with
BR&R or the FDA-aligned project is feasible without re-balloting core
VRDR.

---

### Disposition C: Decline

#### Proposal

Re-apply the original "Considered for Future Use" resolution. Decline
to add FDA RWD use cases to VRDR until FDA submits concrete, prioritized
requirements (e.g., specific data elements, query patterns, identifier
strategies, de-identification expectations) through a follow-on change
request or a project-scoped IG.

#### Justification

The reverted comment from lynn_laakso explicitly notes "At present
there are no specific requirements to drive VRDR development for these
use cases." The original 23-0-0 vote endorsed exactly this position.
Adding speculative use cases without FDA-confirmed requirements risks
publishing narrative that does not match how FDA actually intends to
consume VRDR data, which would then need to be revised — arguably more
disruptive than waiting. A decline is defensible while remaining open:
the work group can re-open (as it just did) the moment FDA brings
concrete requirements.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been through one decline cycle and was
deliberately re-opened post-STU3, which signals that the work group
wants to make tangible forward motion rather than re-parking the
request. Disposition A is the cheapest path that materially advances
the ticket: a use-cases section is normative narrative, does not break
existing implementations, and gives FDA a concrete artifact to react
to — which is the most reliable way to elicit the very requirements
whose absence drove the original decline. Disposition B is the right
end-state if and when FDA brings substantive RWD constraints, but
launching a companion IG before requirements are in hand is premature
and would over-invest WG capacity. Disposition C is defensible but
effectively repeats history; given the explicit re-open it is unlikely
to satisfy the reporter or FDA stakeholders. The work group should
adopt A now, treat it as the seed for FDA engagement, and use the
follow-on dialogue to decide whether B is warranted in a future
release.
