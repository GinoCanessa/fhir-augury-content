## Summary

Craig Newman is asking why FHIR Observation profiles handle two core
elements inconsistently: (1) some profiles (e.g., Body Height) fix
`Observation.status` to a single allowed value while others (e.g.,
Blood Pressure) leave the full status value-set in play; and (2) some
profiles constrain `Observation.code` differently — Blood Pressure
treats it as an ordinary `CodeableConcept` (allowing the full pattern
of bindings/coding entries) while Body Height appears to lock it down.
The reporter asserts that comparable observation profiles within the
same family ought to be aligned. The ticket was filed against the BSeR
specification by Public Health, but the cited examples (Body Height,
Blood Pressure) are FHIR core Vital Signs profiles, so the underlying
question is really about consistency in the Vital Signs profile family
rather than anything BSeR-specific. There are no cross-referenced Jira
tickets that elaborate the request.

## Details

**Description:**

> Why do some observation profiles restrict Observation.status to a
> value of Complete while others do not? For example Body Height fixes
> the status to Complete but Blood Pressue does not. It seems like
> they should be consistent.
>
> Similarly why is Observation.code sometimes a codeableConcept in
> some profiles (Blood Pressure) but not others (Body Height)?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Align `Observation.status` and `Observation.code` constraints across
the Vital Signs profile family in FHIR core. Specifically:

- For `Observation.status`, apply the same constraint pattern across
  every Vital Signs profile (Vital Signs base, Body Height, Body
  Weight, Body Temperature, Heart Rate, Respiratory Rate, Oxygen
  Saturation, Blood Pressure, BMI, Head Circumference, Body Length,
  etc.). Either (a) fix `Observation.status` in the Vital Signs base
  profile so all derived profiles inherit the same constraint, or (b)
  remove the fixed value from any profile that currently has one and
  let the resource-level `ObservationStatus` value-set apply
  uniformly.
- For `Observation.code`, re-state the binding the same way in every
  Vital Signs profile (base type `CodeableConcept` with a `pattern[x]`
  for the LOINC code in single-concept profiles such as Body Height,
  and a comparable `pattern[x]` on Blood Pressure's panel code), so
  the typing surface presented to implementers is uniform.

The change would be made in the FHIR core Vital Signs profile sources
and would be a coordinated edit across that profile family rather than
a one-off tweak.

#### Justification

Profile-family inconsistency for two of the most-implemented
observation profiles is a real interoperability and tooling problem:
validators, FHIRPath authors, and code generators all behave
differently when one profile fixes a status and a sibling does not, or
when the declared type of `Observation.code` differs across profiles
that are otherwise structurally similar. Aligning these two elements
across the Vital Signs family is a low-risk consistency fix that
matches the FHIR design principle that profiles in a coherent family
should differ only where the clinical content requires it.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the request as a documentation / profiling-guidance issue rather
than a constraint change. Specifically:

- Add a short "Conformance conventions for Vital Signs profiles"
  section to the Vital Signs profile narrative in FHIR core that
  explicitly states: (i) `Observation.status` is intentionally left
  unrestricted in the Vital Signs base so that in-progress and
  amended observations remain conformant, and any tighter constraint
  in a derived profile is local to that profile's intended use; (ii)
  `Observation.code` is always typed `CodeableConcept` and the
  individual profiles use `pattern[x]` to lock the LOINC concept,
  which can read as "fixed code" in tooling but is not a type change.
- Confirm in the BSeR IG (the specification this ticket is filed
  against) that BSeR does not need to redefine these constraints
  itself.

No element-level changes to the Vital Signs profiles are made.

#### Justification

If the differences the reporter sees are mostly presentational (Forge
/ snapshot rendering of a `pattern[x]` looking like a fixed value, or
a derived profile narrowing status for that profile's use case), then
the right fix is to explain the convention rather than to change the
profiles. Editing fixed values on widely-implemented Vital Signs
profiles risks breaking existing validation tooling and downstream
IGs (US Core, IPS, BSeR itself), so a documentation-only response is
considerably less disruptive while still answering the question the
reporter raised.

---

### Disposition C: Decline

#### Proposal

Decline the change as filed against this work group / specification.
Rationale: the ticket is filed against the BSeR (Public Health) IG,
but the cited inconsistencies are in FHIR core Vital Signs profiles
that BSeR does not own. Close as `Not Persuasive — wrong
specification` and direct the reporter (or a follow-up tracker) to
re-file the consistency question against FHIR core (Orders &
Observations work group) if it is still considered a problem in a
current FHIR release.

#### Justification

Public Health / BSeR cannot resolve a constraint-consistency question
about FHIR core Vital Signs profiles, and acting on it from BSeR would
either be no-op (if BSeR doesn't redefine these elements) or would
create a BSeR-local divergence from the rest of the ecosystem. The
ticket has also been open and untouched since 2019, suggesting the
underlying concern may already be addressed in current FHIR
publications or has lost its original implementer context. Declining
with a clear redirect lets the right work group decide whether a
change is still warranted.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern is legitimate but the most likely
root cause is presentational: `Observation.code` is always a
`CodeableConcept` in every Vital Signs profile, but `pattern[x]`
bindings make some of them appear "fixed" in tooling, and
`Observation.status` is deliberately left open in some profiles to
preserve clinical workflow flexibility. A documentation pass on the
Vital Signs profile family — owned by Orders & Observations, not
Public Health — is far less disruptive than editing fixed values on
profiles that are heavily reused by US Core, IPS, and downstream IGs
including BSeR itself, and it directly answers the reporter's
question. If the workgroup agrees the issue is really a core /
O&O concern, B can be paired with a redirect (per C) so the
documentation actually lands in the right place. Disposition A should
only be picked if the workgroup has evidence that real implementations
are diverging in interoperable behaviour today, which this ticket does
not provide.
