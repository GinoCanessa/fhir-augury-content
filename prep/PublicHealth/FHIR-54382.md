## Summary

The reporter, on behalf of a NAACCR community member, is asking that the
**Central Cancer Registry Reporting Primary Cancer Condition** profile
(§7.6.1 of the US Central Cancer Registry Reporting Content IG) stop using a
custom extension to carry `assertedDate` and instead reuse the `assertedDate`
data element that is already exposed by the **US Core Condition Encounter
Diagnosis** profile from which it is derived. The request additionally asks
that the inherited `assertedDate` element be tightened to **1..1**
cardinality on the CCRR profile, since the registry workflow always requires
that date. No cross-referenced Jira tickets, Zulip threads, or GitHub items
were found for this change, so this review is grounded entirely in the
ticket itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **US Central Cancer Registry Reporting Content IG**, profile
**Central Cancer Registry Reporting Primary Cancer Condition**
(§7.6.1):

1. Remove the existing custom `assertedDate` extension slice/declaration on
   the CCRR Primary Cancer Condition profile.
2. Constrain the inherited `Condition.assertedDate` element (already
   defined on the US Core Condition Encounter Diagnosis Profile via the
   US Core / FHIR `assertedDate` extension surfaced as a top-level
   element in that profile) to **1..1**.
3. Update the profile narrative, examples, and any IG guidance pages that
   currently reference the local extension so they instead reference the
   inherited `assertedDate` element.
4. Add a brief migration note in the IG change log indicating the
   extension is no longer used and that producers/consumers must populate
   and read `Condition.assertedDate` directly on instances of this
   profile.

#### Justification

This is exactly what the reporter (and the recommending reviewer) asked
for, and the IG workgroup has already marked the resolution *Persuasive*
("We will apply change as requested."). It also follows core FHIR
profiling discipline: when a parent profile already exposes a needed
element, derived profiles should reuse it rather than re-introduce the
same semantics through a parallel extension. Tightening the cardinality
to 1..1 reflects the registry use case — assertedDate is the date the
clinician asserted the cancer condition, which a central cancer registry
report must always carry. Reusing the inherited element also avoids the
ambiguity of having two conflicting representations (extension vs.
element) for the same concept on the same instance.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the same change as Disposition A, but **stage it across two
publications**:

- *This release:* Mark the current `assertedDate` extension on CCRR
  Primary Cancer Condition as **deprecated** (warning in the profile
  narrative, `must-support = false`, and an `obligation` / guidance note
  pointing implementers at `Condition.assertedDate`). At the same time,
  expose `Condition.assertedDate` as **must-support 0..1** on the
  profile.
- *Next release:* Remove the extension and tighten
  `Condition.assertedDate` to **1..1**, as in Disposition A.

#### Justification

Functionally identical end-state, but it gives existing CCRR senders and
receivers a release in which both representations are valid, which
limits the breakage risk for early implementers. Trade-off: it requires
two ballot cycles, leaves a temporary "two ways to say the same thing"
inconsistency on the wire, and delays the cleaner profile that the
reporter is asking for. Worth considering only if there is evidence of
existing production use of the extension; the ticket and surrounding
discussion give no indication that there is.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the existing `assertedDate` extension on
CCRR Primary Cancer Condition with its current cardinality, on the
grounds that the change would alter the wire format of an already-
derived profile.

#### Justification

This is the weakest option. The only defensible reason to decline would
be active production use of the extension that cannot tolerate even a
deprecate-then-remove path; nothing in the ticket, the comments, or the
(absent) Zulip / GitHub trail suggests that is the case. Without such
evidence, declining would leave the profile carrying a redundant
extension for a concept the parent profile already models — exactly the
kind of duplication FHIR profiling guidance discourages — and would
ignore the workgroup's own *Persuasive* recommendation.

---

### Recommendation

**Recommended disposition:** A

The ticket is a small, well-scoped profiling cleanup whose semantics are
unchanged: `assertedDate` continues to mean the same thing, just sourced
from the inherited element instead of a parallel extension. The
workgroup has already converged on *Persuasive*, the requesting
community (NAACCR) is the primary implementer of this IG, and tightening
to 1..1 matches the registry reporting use case. Disposition B is a
reasonable fallback only if implementer feedback surfaces existing
production use of the extension during ballot reconciliation; absent
that signal, the staged approach buys little and costs a release cycle.
Disposition C is not defensible on the current evidence.
