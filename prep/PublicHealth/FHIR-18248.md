## Summary

The ticket reports that the BSeR Tobacco Request Support Bundle constrains
each referral to a single `SmokingStatus` and a single
`TobaccoCessationReferralObs`, and that neither of those observations
exposes the operational data elements called out in the underlying BSeR
Domain Analysis Model (DAM) — for example `bestDayToCall`,
`bestTimeToCall`, and `leaveMessageIndicator`. The reporter asks how
implementers are expected to convey those DAM-defined data elements when
sending a tobacco cessation referral request. The reporter also flags in
a follow-up comment that other BSeR subject matter areas (e.g.,
Childhood Nutrition) likely have the same gap.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items linked to this ticket in the cached data, so the broader scope of
the request must be inferred from the ticket itself and from the BSeR
DAM versus the FHIR IG content.

## Details

**Description:**

The Tobacco Request Support Bundle only allows one each of SmokingStatus
and TobaccoCessationReferralObs neither of which seems to include a way
to send data elements set out in the DAM such as bestDayToCall,
bestTimeToCall, leaveMessageIndicator, etc. How are these data elements
supposed to be send in a referral request?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BSeR Tobacco Cessation Referral profiles and the Tobacco
Request Support Bundle so that the DAM-defined contact-preference data
elements (`bestDayToCall`, `bestTimeToCall`, `leaveMessageIndicator`,
and any other operational/contact attributes called out in the DAM) can
actually be transmitted in a referral request. Concretely:

1. Relax the cardinality on `SmokingStatus` and
   `TobaccoCessationReferralObs` slices in the Tobacco Request Support
   Bundle from `1..1` to `0..*` (or `1..*` where at least one is
   required) so that multiple observations of the same kind can be
   carried — needed because contact preferences are conceptually
   separate observations, not facets of smoking status.
2. Add a new BSeR profile (e.g., `TobaccoCessationContactPreferenceObs`,
   or a more general `BSeRContactPreferenceObs` reusable across subject
   areas) on `Observation` whose `component` slices cover
   `bestDayToCall`, `bestTimeToCall`, and `leaveMessageIndicator`,
   bound to appropriate value sets (day-of-week, time-of-day, boolean).
3. Add that new profile as an additional supportingInfo /
   `Bundle.entry` slice in the Tobacco Request Support Bundle (and
   parallel updates to the other subject-matter referral bundles, per
   Craig Newman's comment about Childhood Nutrition, etc.).

#### Justification

This is the most direct way to close the gap the reporter raised: the
DAM defines these data elements as part of the referral payload, so the
FHIR IG should provide a conformant means to send them. Adding a
purpose-specific `Observation` profile rather than overloading the
existing tobacco-specific observations preserves the semantic meaning
of `SmokingStatus` and `TobaccoCessationReferralObs` and avoids
inventing tobacco-only extensions for cross-cutting concepts (call
preferences are not specific to tobacco cessation). Loosening the
1..1 cardinality is necessary because the underlying issue — that the
bundle currently can carry only one instance of each — is real and
prevents conformant transmission of additional referral-context
observations. Generalising the contact-preference profile addresses
the comment that Childhood Nutrition and other BSeR subject areas have
the same issue, instead of fixing it once per workflow.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a new `Observation` profile, model the
operational referral attributes (`bestDayToCall`, `bestTimeToCall`,
`leaveMessageIndicator`) as **extensions on the BSeR
`ServiceRequest`** (i.e., the referral request itself), since these are
properties of *how to fulfil the referral*, not clinical observations
about the patient. Define a single complex extension —
`BSeRReferralContactPreferences` — with sub-extensions for each
attribute, bound to suitable value sets, and slot it into the BSeR
`ServiceRequest` profile used by every subject-matter referral bundle
(Tobacco, Childhood Nutrition, etc.). Leave the existing
`SmokingStatus` and `TobaccoCessationReferralObs` cardinalities and
semantics untouched.

#### Justification

`bestDayToCall`, `bestTimeToCall`, and `leaveMessageIndicator` are
operational/administrative metadata about contacting the patient to
schedule the referred service — they are not clinical findings that
naturally fit `Observation`. Modelling them as a `ServiceRequest`
extension keeps the referral-time logistics co-located with the
referral itself, simplifies receiver processing (no scanning bundle
entries for a side-channel observation), and is reusable across all
BSeR subject-area bundles without touching any of the existing
condition-specific observation profiles. It also avoids loosening
cardinalities on the tobacco-specific observations, which is a more
contained change with less risk to existing implementers. The trade-off
versus Disposition A is that extensions are slightly less discoverable
than first-class profiles and require receivers to know the extension
URL.

---

### Disposition C: Decline

#### Proposal

Decline the request on the basis that the listed DAM data elements
(`bestDayToCall`, `bestTimeToCall`, `leaveMessageIndicator`) are
patient-administrative contact preferences that are out of scope for
the BSeR FHIR IG bundle payload and should instead be conveyed via the
referenced `Patient` resource (e.g., `Patient.contact`,
`Patient.telecom.period`, or jurisdictional patient-communication
profiles) or out-of-band scheduling workflows, rather than embedded in
the referral observation set.

#### Justification

A defensible argument for declining is that BSeR's FHIR payload is
intentionally a clinical-context bundle for the referral, and recurring
contact-preference attributes belong on the patient demographics record
where they can be reused for any encounter or referral, rather than
being duplicated inside each referral bundle. If the existing
`Patient`-level mechanisms are judged sufficient, no IG change is
required. This option should only be chosen if the workgroup
collectively concludes that the DAM elements were aspirational and
that conforming implementations are expected to satisfy them via the
`Patient` resource already present in the bundle.

---

### Recommendation

**Recommended disposition:** B

Disposition B — modelling the contact-preference attributes as a
reusable extension on the BSeR `ServiceRequest` — best matches the
nature of the data (operational referral logistics, not clinical
observations) and addresses Craig Newman's follow-up point that the
same gap exists across subject areas (Childhood Nutrition, etc.) with a
single artifact reused everywhere. It also avoids relaxing
`1..1` cardinalities on `SmokingStatus` and
`TobaccoCessationReferralObs`, which keeps the change contained and
backward-compatible for senders that don't need to convey contact
preferences. Disposition A is workable but introduces more moving
parts (new profile + cardinality changes + bundle slice changes) and
arguably misclassifies administrative contact metadata as clinical
observations. Disposition C should be rejected unless the workgroup is
confident that `Patient`-level demographics already cover the DAM's
intent — the reporter's question implies they do not, since BSeR
explicitly carries these elements in its DAM.
