## Summary

FHIR-42632 questions the design of the BSeR "batch query" Bundles that
implementations send to gather supporting clinical information at the time
of a referral. The reporter (Bas van den Heuvel) observes that the
canonical batch queries — both the worked examples and the formal
definitions in the BSeR IG — are written to retrieve **every** historical
value ever recorded for the patient for a given concept, rather than the
most recent (or otherwise clinically relevant) values. He asks whether the
"return everything ever" behavior is intentional and, if it is not
clinically required, whether the queries should be constrained to a
bounded time window (e.g., last *N* months/years) so that responders are
not forced to return unbounded history just to satisfy a referral.

The ticket is filed against the same BSeR batch-query mechanism that is
the subject of two other open Public Health tickets that share the
`BSeR_BATCH_QUERY` label and were filed in the same review window:
[FHIR-42151](https://jira.hl7.org/browse/FHIR-42151) (Rick Geimer) asks
whether the IG considered using `Measure` + CQL instead of a batch
`Bundle` of queries, and
[FHIR-42356](https://jira.hl7.org/browse/FHIR-42356) (Vassil Peytchev)
suggests that the same use case might be better served by Subscriptions
and notification topics. FHIR-42632 is therefore part of a broader
cluster of feedback challenging both the **shape** (batch Bundle vs.
Measure/CQL/Subscription) and the **scope** (full history vs. bounded
window) of the BSeR supporting-information retrieval pattern; this
ticket is specifically the "scope" critique.

## Details

**Description:**

> The batch queries (and the formal definitions in the spec) contain
> queries that provide all values that are ever recorded for a patient
> and not the most recent values. Is that intended?
>
> If the information is not clinical relevant/required, it should not be
> included. Can we limit these searches to a time-frame?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive. Update each of the per-referral batch-query Bundle
definitions in the BSeR IG so that every search entry that retrieves
clinical observations, conditions, medications, etc. carries an explicit
time-frame constraint and (where appropriate) a "most recent" selector.
Concretely:

1. Add `date=geYYYY-MM-DD` (or equivalent `period`/`onset` parameter)
   bounds to every batch query that today is unbounded — using a per-
   query lookback window that the workgroup defines per referral type
   (e.g., 12 months for vitals/labs feeding a hypertension referral,
   24 months for diagnoses, lifetime for immunizations, etc.).
2. Where the receiver only needs the latest value, add `_count=1` and
   `_sort=-date` to the relevant entries so responders return one
   most-recent resource rather than full history.
3. Document the lookback windows and "most recent vs. history" choices
   in the IG narrative for each referral type, plus a single normative
   statement that batch queries SHALL constrain results to the
   documented window unless the receiver explicitly requests history.
4. Regenerate the example Bundles and the formal definitions
   (`Bundle`/`Composition` artifacts plus the corresponding test
   examples) so the published artifacts match the new narrative.

#### Justification

The reporter's clinical premise is sound: a referral receiver typically
needs current clinical state, not lifetime history, and unbounded
queries impose disproportionate load on the responding system and
disproportionate processing burden on the receiver. FHIR search
parameters (`date`, `_count`, `_sort`, `_lastUpdated`) already provide
the exact mechanisms needed, so this is a low-mechanism, high-value
tightening of the existing pattern. It is also the smallest change that
addresses the stated concern without disturbing the broader batch-Bundle
mechanism that FHIR-42151 / FHIR-42356 are challenging — i.e., it is
forward-compatible with either keeping or replacing the batch-Bundle
approach.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive – consider for further review. Rather than parameter-tuning
each batch query in place, replace the per-referral batch-Bundle
definitions with a Measure / CQL artifact per referral type (aligning
with the suggestion in [FHIR-42151](https://jira.hl7.org/browse/FHIR-42151)).
Each referral's "supporting information set" would be expressed as a CQL
library that explicitly returns "most recent value within window N" for
each clinical concept; the IG would then publish one `Measure` per
referral type and define a `$evaluate-measure`-style retrieval profile
instead of a batch `Bundle`. The clinical-relevance and time-window
concerns that motivate FHIR-42632 are then expressed once in CQL and
inherited by every responder.

#### Justification

CQL is purpose-built for "give me the most recent X within the last N
months" semantics and is already used widely in quality-measure and
decision-support contexts in the same Public Health and CDS ecosystem
that BSeR sits in. This addresses both FHIR-42632 (scope/recency) and
FHIR-42151 (mechanism choice) in one stroke, and it removes the
hand-maintenance burden of per-referral batch Bundles. The trade-off is
larger: it is a mechanism change, requires implementers to support CQL
evaluation (or the receiver to interpret the library), and risks
overlap with FHIR-42356's separate suggestion to switch to
Subscriptions. The workgroup should not adopt this without a coordinated
disposition across all three `BSeR_BATCH_QUERY` tickets.

---

### Disposition C: Decline

#### Proposal

Not Persuasive. Close FHIR-42632 with a Not Persuasive resolution on
the grounds that the unbounded queries are intentional: BSeR is a
referral-initiation specification whose receiver typically has *no
prior relationship* with the patient, so providing the full available
history (rather than a sender-defined slice) is the safer default for
clinical decision-making. Implementers who need a narrower scope can
already constrain their own server responses, and bilateral trading
partners can negotiate tighter windows out-of-band without the IG
mandating one.

#### Justification

This is the defensible position only if the workgroup affirmatively
believes that "send everything you have" is the correct contract for a
referral hand-off. It avoids a normative change that could break
existing implementations that rely on receiving full history, and it
keeps the IG out of the business of picking arbitrary lookback windows
per clinical concept (which would invite endless re-litigation). The
weakness is that it does not actually answer the reporter's concrete
question — "is that intended?" — and it leaves responders bearing the
full cost of unbounded queries.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested), explicitly
coordinated with FHIR-42151 and FHIR-42356.

The reporter has identified a real, narrowly-scoped defect in the
current batch-query definitions: they are unbounded, the IG does not
say whether that is deliberate, and FHIR's own search parameters
provide a clean fix. Disposition A delivers that fix using only
existing FHIR mechanisms, regenerates the published artifacts, and
makes the IG's intent explicit — which is the minimum the workgroup
owes the reporter. Disposition B (Measure/CQL) is more elegant but is
really an answer to FHIR-42151, not FHIR-42632, and adopting it here
would entangle three tickets and a mechanism change in one decision;
it should be considered separately and, if pursued, would supersede A
naturally. Disposition C (Decline) is hard to defend without comments
or Zulip discussion supporting the "send everything" contract. The
workgroup should triage all three `BSeR_BATCH_QUERY` tickets together,
adopt A for FHIR-42632 now, and use that work to inform the
mechanism-level decision on FHIR-42151 and FHIR-42356.
