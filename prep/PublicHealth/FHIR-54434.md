## Summary

The reporter raises a documentation/scoping concern about the US Central Cancer
Registry Reporting (CCRR) IG and its relationship with the separate Cancer
Pathology Data Sharing (US Cancer Reporting / `us/cancer-reporting`) IG. The
CCRR composition definition references the `us-pathology-diagnostic-report`
profile from the Cancer Pathology IG, which itself drags in specialized
profiles (`us-pathology-service-request`, `us-pathology-specimen`,
`us-pathology-observation`) that the reporter cannot find listed anywhere in
the CCRR IG. The reporter further observes that the pathology diagnostic
report profile appears to be allowed only in the Notes section
(`Composition.section:sliceNotesSection`) of the CCRR composition and is
*not* permitted in the Results section
(`Composition.section:sliceResultsSection`), which seems counter-intuitive for
pathology results. The ticket asks the workgroup to update the IG's
"Background to Cancer Reporting in the USA" use cases page with explicit
narrative describing the relationship between the two IGs and clear guidance
on when each is expected to be used. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this ticket, so the request is
self-contained.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CCRR IG's `usecases.html` page (anchor
`#background-to-cancer-reporting-in-the-usa`) to add a new sub-section titled
"Relationship to the Cancer Pathology Data Sharing IG" that:

1. Names the companion IG (`hl7.org/fhir/us/cancer-reporting`, STU2) and its
   anchor profile (`us-pathology-diagnostic-report`) plus the chained
   profiles it pulls in (`us-pathology-service-request`,
   `us-pathology-specimen`, `us-pathology-observation`).
2. Explains scope: CCRR is the cancer-registry-facing reporting workflow
   (case report bundle anchored on `ccrr-composition-definitions`), while
   the Cancer Pathology Data Sharing IG defines the structured pathology
   report payload exchanged between lab and other consumers.
3. States explicitly which profiles from the Pathology IG are reused by
   CCRR vs. which are merely transitive dependencies (and therefore not
   re-listed in CCRR's Profiles index but still in scope when
   `us-pathology-diagnostic-report` instances are sent).
4. Documents the deliberate slicing decision in
   `ccrr-composition-definitions`: that
   `us-pathology-diagnostic-report` is currently allowed in
   `Composition.section:sliceNotesSection` and *not* in
   `Composition.section:sliceResultsSection`, with the rationale.
5. Adds a small decision table or flow ("when do I send CCRR vs. when do I
   send a standalone Cancer Pathology bundle?") so registry submitters and
   pathology systems can self-route.

#### Justification

The reporter's confusion is a concrete documentation defect that costs
implementers time on every onboarding. The fix is purely narrative — no
profile, value-set, or cardinality changes — so it is low-risk for a
content IG and aligns with FHIR's general principle that an IG's
home/use-cases page should make external dependencies and their scope
explicit, especially when one IG transitively imports profiles from
another. Reusing rather than redefining the pathology profiles is already
the right design choice; the IG just needs to say so.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the documentation update from Disposition A *and* re-examine the
Composition slicing in `ccrr-composition-definitions` as a substantive
change:

1. Add `us-pathology-diagnostic-report` to the allowed profile list on
   `Composition.section:sliceResultsSection` (in addition to keeping it on
   `sliceNotesSection`), if pathology results are in fact a legitimate
   "results" payload for a central cancer registry submission.
2. Optionally surface the chained `us-pathology-specimen` and
   `us-pathology-observation` profiles in the CCRR Profiles index as
   "referenced profiles" (Markdown table or generated section), so
   implementers do not have to discover them by chasing the
   `DiagnosticReport.specimen` / `DiagnosticReport.result` references at
   runtime.
3. Add an `extension` or markdown panel on the
   `StructureDefinition-ccrr-composition-definitions.html` page itself
   explaining the slicing rationale, so the answer is reachable from where
   the reader first hits the question (not only from `usecases.html`).

#### Justification

The reporter's second observation — that pathology DRs are allowed only in
Notes, not Results — looks like a real modeling question rather than a pure
documentation issue. If cancer registries do receive pathology *results*
(not just narrative notes) as part of a case report, the slicing
constraint is wrong and should be relaxed; if they don't, the rationale
needs to be stated. Either way, the workgroup should not patch only the
narrative without confirming that the slicing was intentional. This
disposition addresses both halves of the ticket.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" / "Already documented", on the grounds that the
CCRR IG already declares its dependency on the Cancer Pathology Data
Sharing IG via the `dependsOn` entry in the published `ImplementationGuide`
resource, and that profile authors are expected to consult the dependency
graph. No narrative change is required.

#### Justification

This stance is hard to defend. FHIR IGs are expected to be navigable by
implementers, not just by tooling: a dependency declared only in the
machine-readable `ImplementationGuide` resource is invisible to a human
reading `usecases.html`. The reporter is a known FHIR contributor and
explicitly states they could not find the relationship documented, which
is itself evidence that the current docs are insufficient. Decline is only
appropriate if the workgroup deliberately wants the two IGs to remain
loosely coupled in narrative — an unusual position for a public-health
content IG that explicitly composes the other.

---

### Recommendation

**Recommended disposition:** B

The ticket bundles two issues that look superficially like one. The pure
documentation fix (Disposition A) is necessary but not sufficient: the
reporter's second observation — that `us-pathology-diagnostic-report` is
permitted in the Notes slice of the CCRR Composition but blocked from the
Results slice — is a substantive modeling question that the workgroup
should answer before publishing more narrative. Public-health stakeholders
on the call are well-placed to confirm whether central cancer registries
ingest pathology results as structured `DiagnosticReport` instances (in
which case Results-slice support should be added) or strictly as
attached/notes content (in which case the rationale needs to be written
down). Resolving both together avoids a near-term repeat ticket and gives
the IG editors a single coherent edit pass on
`usecases.html` and `ccrr-composition-definitions`.
