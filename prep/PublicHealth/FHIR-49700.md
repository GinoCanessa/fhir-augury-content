## Summary

The ticket asks the Public Health work group to stop maintaining a US
Public Health–specific "expansion parameters" extension and instead reuse
the cross-cutting `cqf-expansionParameters` extension already published
in the HL7 FHIR Extensions Pack
(`http://hl7.org/fhir/StructureDefinition/cqf-expansionParameters`).
The reporter (Bryn Rhodes) argues the existing CQF extension supports
the use case generally, so the US PH–specific clone is redundant and
should be removed.

The request was effectively rolled into [FHIR-52764](https://jira.hl7.org/browse/FHIR-52764)
on the **US Public Health Profiles Library (USPHPL)** IG, which has the
same title and intent. FHIR-52764 was resolved **Persuasive** with the
disposition "We will use the `cqf-expansionParameters` extension (we
will use this extension where the US Public Health equivalent is used)
and remove the US Public Health Expansion Parameters Extension from the
IG." That ticket is now **Published**, and FHIR-49700 was therefore
closed with **Persuasive with Modification**, pointing to FHIR-52764 as
the actual implementation vehicle.

## Proposed Dispositions

Note: FHIR-49700's substantive change has already been committed via
FHIR-52764 on the USPHPL IG (resolved Persuasive, now Published). The
dispositions below are framed as the formal action this ticket should
take in response to that fact.

### Disposition A: Accept As Requested

#### Proposal

Resolve FHIR-49700 as **Persuasive** (rather than the current
*Persuasive with Modification*) with the disposition: "Replace all uses
of the US Public Health–specific Expansion Parameters Extension with
the FHIR Extensions Pack's `cqf-expansionParameters`
(`http://hl7.org/fhir/StructureDefinition/cqf-expansionParameters`),
and remove the US PH–specific extension definition from the source
specification (originally targeted by this ticket). Update any profile
differentials and IG narrative that referenced the removed extension to
point at `cqf-expansionParameters`. This work has already been carried
out on the USPHPL IG via FHIR-52764." Close FHIR-49700 with a pointer to
the FHIR-52764 changes.

#### Justification

This is exactly what the reporter asked for and exactly what FHIR-52764
implemented on the IG that now owns the affected profile. Reusing
`cqf-expansionParameters` is consistent with FHIR's preference for
cross-IG extension reuse over per-domain clones, removes a divergent
definition that implementers would otherwise have to special-case, and
aligns Public Health terminology-binding behaviour with how CQF/CPG
content already configures terminology server expansion. Because
FHIR-52764 is Published, declaring FHIR-49700 fully Persuasive makes the
two tickets internally consistent.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve FHIR-49700 as **Persuasive with Modification** (its current
state) and explicitly scope the change as: (1) prefer
`cqf-expansionParameters` going forward and (2) retire the US Public
Health–specific extension by **deprecating** rather than deleting it
for at least one ballot cycle. Add a deprecation notice to the US
PH–specific extension's StructureDefinition pointing implementers to
`cqf-expansionParameters`, keep the canonical URL resolvable, and
schedule removal for the cycle after USPHPL's next published release.
Document the migration path (element-by-element mapping from the US PH
extension's parameters to a `Parameters` payload understood by
`cqf-expansionParameters`) in the IG's Conformance / Migration page.

#### Justification

If any downstream Public Health implementations have already deployed
the US PH–specific extension (it predates the USPHPL consolidation),
hard-removing it — as FHIR-52764 elected to do — risks silently breaking
those implementations. A deprecation window preserves the FHIR-49700
intent (single canonical extension) while giving early adopters a
documented migration path. This option also matches the existing
"Persuasive with Modification" resolution code that was already applied
to FHIR-49700, and it is compatible with what FHIR-52764 has done on
USPHPL (the deletion there can be re-characterised as the end-state of
the deprecation window).

---

### Disposition C: Decline

#### Proposal

Decline FHIR-49700 as **already addressed** — i.e., close it without
further action on the original specification, with a resolution note
that the substantive change is fully tracked and Published under
FHIR-52764 on the US Public Health Profiles Library IG, and that no
additional spec changes are required against the original
specification.

#### Justification

This is the cleanest book-keeping outcome when a ticket has been
superseded by another that has already shipped. It avoids re-opening a
debate whose substantive answer ("yes, use `cqf-expansionParameters`")
has already been adopted and balloted in USPHPL, and it prevents
parallel disposition text from drifting between FHIR-49700 and
FHIR-52764. The downside is that "Decline" understates what actually
happened — the request was accepted, just on a different artifact —
which is why this is the weakest of the three options on its own
merits.

---

### Recommendation

**Recommended disposition:** A

The reporter's request was substantively accepted and is already
Published via FHIR-52764: `cqf-expansionParameters` is now the canonical
mechanism, and the US Public Health–specific extension is being removed
from USPHPL. Recording FHIR-49700 as straight **Persuasive** (with a
resolution note pointing at FHIR-52764) keeps the two tickets aligned
and gives implementers a single, unambiguous answer when they search
Jira for either key. Disposition B's deprecation window is worth
mentioning during the workgroup discussion only if anyone surfaces a
known external implementation of the US PH–specific extension; absent
that evidence, a clean removal — as USPHPL has already done — is the
right call. Disposition C is not recommended because "Decline"
mischaracterises the outcome.
