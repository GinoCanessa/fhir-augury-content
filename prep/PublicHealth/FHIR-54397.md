## Summary

This ticket asks the **US Central Cancer Registry Reporting Content IG**
to expand its handling of the use-case data element **"Patient Smoking
Status"** so that, when the element is populated, it carries enough
exposure detail to be useful for cancer-registry risk assessment —
specifically, the **number of years of active smoking** (e.g., `25 years`)
and/or the **active-smoking period** (e.g., `1990–2015`), rather than
only a current status code. The submission was made by NAACCR on behalf
of a community member.

Comments on the ticket reflect non-trivial discussion among reviewers:
they note significant **semantic differences between how EHRs collect
smoking data and what a central cancer registry actually needs**, point
out that the relevant element is in **draft USCDI V7**, cite literature
([Bian et al., JCO CCI 2020](https://ascopubs.org/doi/10.1200/CCI.20.00187))
on these gaps, and explicitly recommend **discussing with the registry
submitter before proposing a resolution**. They also flag that *most
registries today are unable to process and store the deeper exposure
information* this proposal would convey, and that early-incidence
reporting may be better served by the simpler status values, while full
exposure history belongs in case ascertainment / full case reporting.

There are **no cross-referenced Jira tickets** in Augury. The only
cross-reference is to a Confluence page hosting the NAACCR
"Tobacco Use Proposal v5" spreadsheet (page id `35720239`), which is
not ingested into Augury and so cannot be summarised inline.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the IG's "Patient Smoking Status" use-case data element so that,
when populated, it carries quantitative tobacco-exposure detail in
addition to the current status code:

- Add an **`activeSmokingYears`** quantity (integer years; e.g., `25`).
- Add an **active-smoking `period`** with `start` and `end`
  `dateTime`s (e.g., `1990-01-01` → `2015-12-31`).
- Bind these as **optional (0..1)** alongside the existing smoking
  status, so registries that cannot capture the additional exposure
  detail are not forced to.

Realise the change as profile-level guidance against `Observation`
(consistent with US Core's existing **Smoking Status** profile and with
draft USCDI V7), with companion `component` slices for
`activeSmokingYears` (Quantity, UCUM `a`) and `activePeriod`
(`valuePeriod` on a dedicated component code), plus must-support
flags set to `MS = false` so the additions are non-breaking for
implementers who do not populate them.

#### Justification

The submission is well-grounded: the element is in draft USCDI V7, and
the literature cited
([JCO CCI 2020](https://ascopubs.org/doi/10.1200/CCI.20.00187)) confirms
that bare status codes lose information central-cancer-registries need
for risk assessment. Modeling the additions as **optional `Observation`
components** keeps backward compatibility for incidence-only flows
while enabling registries that do collect exposure detail to convey it.
This aligns with the FHIR design principle of *"add detail through
optional structure, not by replacing the simpler shape"*, and mirrors
how US Core layers smoking-status semantics onto `Observation` today.

---

### Disposition B: Alternative Approach

#### Proposal

**Do not** add new fields to the IG's smoking-status element itself.
Instead:

1. Keep the existing smoking-status element as-is for **early incident
   reporting**.
2. Reference (rather than duplicate) a separate **tobacco exposure
   history** profile in the **case-ascertainment / full case-reporting**
   workflow of the IG. The natural carrier is one or two extra
   `Observation` instances — for example a `pack-years`-style
   observation (LOINC `74010-0` "Pack years of cigarette smoking")
   and/or an exposure-period observation — surfaced via the IG's
   case-reporting bundle, not the smoking-status element itself.
3. Add IG narrative explicitly stating the **separation of concerns**
   the reviewers identified: status for incidence, full exposure
   history for ascertainment.

If the IG already aligns with US Core / USCDI for the status element,
prefer **profiling/extending existing US Core Smoking Status** rather
than creating IG-specific smoking-status structure.

#### Justification

This directly addresses the strongest review-comment concern — that
*"trying to do both may fail to meet either need"*. It avoids burdening
EHR sources and registries that don't capture exposure history, while
still giving the small population of registries that do a defined,
LOINC-coded place to put the data. It also keeps the IG aligned with
US Core / draft USCDI V7 instead of forking smoking-status semantics,
reducing implementer surprise and downstream IG drift.

The trade-off vs. Disposition A is one extra round of model design
(deciding the exposure-history `Observation` shape and code bindings)
in exchange for cleaner semantics and a smaller blast radius on
implementers who cannot supply the new data.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently framed and resolve as **Persuasive
With Mod / Not Persuasive** pending submitter discussion. Rationale
category: *insufficiently scoped use case for IG-level change; existing
mechanisms suffice.*

Specifically:

- Smoking status as collected by EHRs is already representable via the
  existing US Core / USCDI smoking-status pattern.
- Detailed tobacco-exposure history (years, periods, pack-years) is
  **already expressible** via additional `Observation` instances using
  established LOINC codes (e.g., `74010-0` "Pack years"); the IG does
  not need to mint new structure to permit this.
- The reviewers themselves note that **most registries cannot process
  or store the additional detail today**, weakening the case for
  IG-level normative change.

Recommend the IG instead add **non-normative implementation guidance**
pointing implementers at the existing LOINC-coded observations, and
close the ticket without changing the data element.

#### Justification

The change request is best understood as a *recommendation*, not a
defect — the submitter says "if/when populated, include …". FHIR's
existing `Observation` machinery and standard LOINC codes already
support the requested information without IG-specific schema change.
Adding required or near-required structure that few registries can
consume risks creating a non-conformance trap and pushing complexity
onto EHR producers for marginal real-world benefit. Reviewer comments
explicitly recommend **talking to the submitter first** and flag the
EHR↔registry semantic mismatch — both are reasonable grounds to defer
spec change.

---

### Recommendation

**Recommended disposition: B — Alternative Approach.**

Disposition B is the best balance of the three. The submitter's
underlying need — quantitative tobacco-exposure data for cancer-registry
use — is real and well-supported by literature and by draft USCDI V7,
so a flat decline (C) would lose useful information. But Disposition A
risks exactly what reviewers warned against: conflating an
*early-incidence* status element with a *case-ascertainment* exposure
history, in a way that burdens EHR producers and most registries who
cannot consume the extra detail.

By keeping the existing smoking-status element unchanged and conveying
years / period / pack-years via **separate, optional `Observation`
instances** in the case-reporting workflow (ideally reusing existing
LOINC codes such as `74010-0`), the IG can satisfy registries that do
collect exposure history while staying aligned with US Core and draft
USCDI V7 for everyone else. The reviewers' explicit ask — *discuss
with the registry submitter before resolving* — should be a
prerequisite to closing the ticket either way; that discussion is most
productive against a concrete alternative proposal, which is what this
disposition gives the workgroup to react to.
