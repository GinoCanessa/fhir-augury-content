## Summary

The reporter — submitting on behalf of a NAACCR (cancer registry) community
member — asks a clarifying **question** about the trigger model in the US
Central Cancer Registry Reporting (CCRR) Content IG: *"Will an incremental
report be triggered every time chemotherapy is administered?"* In other
words, does each individual MedicationAdministration event (e.g., each
chemotherapy infusion or dose) produce a separate non-encounter-based (NEB)
or incremental Bundle to the CCR, or are administrations batched into a
single per-encounter report?

The question sits inside an active cluster of CCRR ballot items about how
and when reports fire. By keyword similarity it relates closely to:
[FHIR-54409](https://jira.hl7.org/browse/FHIR-54409) (concern that
laboratory- and chemotherapy-driven triggers will produce excessive report
volume; suggests no more than every 30 days after T0),
[FHIR-54384](https://jira.hl7.org/browse/FHIR-54384) (request to re-add
time-based triggers because medication updates between encounters can be
missed), and [FHIR-54430](https://jira.hl7.org/browse/FHIR-54430)
(clarification that the report bundle is anchored on **encounter close**,
not encounter start, with `Bundle.timestamp` MS and 1..1). Together these
indicate the WG is still tuning trigger granularity; FHIR-54378 is a
specific instance of "what does the trigger model do for medication
administrations within (or across) encounters?"

## Proposed Dispositions

> The ticket is filed as a **Question**, so each disposition below frames
> the response both as an answer to the reporter and as the consequent
> spec/IG change (or non-change) that should accompany it. Because no
> GitHub repository surfaced via cross-references, the dispositions name
> the IG and its trigger / NEB-bundle pages descriptively rather than
> citing specific authoring file paths.

### Disposition A: Accept As Requested

#### Proposal

Treat the question as a documentation-clarification request and answer
**"No — not per administration; per encounter close."** Concretely:

- In the CCRR IG's *Use Case / Triggers* page, add a paragraph (and a
  worked chemotherapy example) stating that medication administrations
  (including chemotherapy) do **not** independently fire incremental
  reports. Instead, MedicationAdministration events are gathered into the
  reporting Bundle assembled when the **encounter closes** (consistent
  with the resolution of [FHIR-54430](https://jira.hl7.org/browse/FHIR-54430)).
- Cross-link this clarification from the NEB bundle profile/example
  page (cf. [FHIR-54698](https://jira.hl7.org/browse/FHIR-54698)) so
  implementers reading the NEB example see the same rule.
- Reply on FHIR-54378 with the answer + the link to the new page
  section, and resolve as **Not Persuasive / Already Specified** once
  the wording is in.

#### Justification

This is the answer most consistent with the WG's existing design
direction: FHIR-54430 has already been resolved Persuasive with
Modification to anchor `Bundle.timestamp` on encounter close, which
implicitly determines per-administration vs. per-encounter behavior.
Codifying that answer in the use-case prose closes a real ambiguity
the reporter found, requires no profile/cardinality change, and is
backwards-compatible with implementations following the current draft.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the question as evidence that the trigger model needs an explicit,
**bounded throttling rule** for high-frequency clinical events
(chemotherapy administrations, frequent labs), and add that rule to the
spec rather than only clarifying current behavior:

- Add a normative statement in the *Triggers* page that, even when an
  EB or NEB trigger condition is met, a CCR-bound report SHALL NOT be
  generated for the same patient/cancer more frequently than a defined
  minimum interval (e.g., 30 days) post-T0, except for explicitly
  designated reportable events.
- Define a small `extension` (or reuse an existing reporting-cadence
  element) on the reporting Bundle / sender configuration to record the
  throttle interval actually applied, so receivers can tell whether a
  report was suppressed by cadence vs. genuinely absent.
- Update the NEB bundle example
  ([FHIR-54698](https://jira.hl7.org/browse/FHIR-54698)) to demonstrate
  the throttled case (multiple chemotherapy administrations in a 30-day
  window producing one consolidated NEB bundle).

#### Justification

FHIR-54378 is one of three sibling tickets from the same NAACCR reporter
([FHIR-54378](https://jira.hl7.org/browse/FHIR-54378),
[FHIR-54409](https://jira.hl7.org/browse/FHIR-54409),
[FHIR-54384](https://jira.hl7.org/browse/FHIR-54384)) that collectively
describe a single design problem: trigger granularity needs a stated
ceiling so registries are not flooded, while still leaving room for
catch-up via time-based triggers. Answering FHIR-54378 in isolation
(Disposition A) leaves the volume problem from FHIR-54409 unaddressed
and can be re-asked the next ballot. A throttling rule directly resolves
the underlying operational concern, is aligned with the stated CCR
intake reality, and fits within the existing trigger framework without
requiring new resources. The trade-off versus A is more spec text and a
small new conformance commitment; the trade-off versus C is the
opposite — it commits the WG to a number that future implementers will
debate.

---

### Disposition C: Decline

#### Proposal

Answer the question on the ticket — *"the encounter-based trigger
fires on encounter close, so multiple chemotherapy administrations
inside a single encounter result in one report; administrations across
separate encounters result in one report per encounter"* — and
**decline to make any spec change**, on the basis that the behavior is
already determined by the encounter-close anchor adopted under
[FHIR-54430](https://jira.hl7.org/browse/FHIR-54430) and the existing
EB/NEB trigger definitions. Resolve as **Not Persuasive** (question
answered, no normative change required).

#### Justification

The IG's trigger model is intentionally specified in terms of
encounter and reportable-event semantics, not per-resource events.
Adding either prescriptive prose (A) or a throttling rule (B) risks
locking the spec to particular operational policies (30-day cadence,
encounter-close anchor wording) that vary across CCRs and that the WG
may wish to leave to local implementation guidance. A polite ticket
reply — combined with the FHIR-54430 resolution already in flight — is
sufficient. This option is defensible if the WG judges that the
cross-cutting volume question belongs in FHIR-54409 / FHIR-54384 and
not here.

---

### Recommendation

**Recommended disposition:** **A**

The ticket is a clarifying question, not a change request, and the WG
has already chosen the substantive answer when it resolved
[FHIR-54430](https://jira.hl7.org/browse/FHIR-54430) Persuasive with
Modification (encounter-close-anchored bundle, `Bundle.timestamp` MS
1..1). The most useful action is to make the implication of that
resolution explicit on the *Triggers* / use-case page so that the next
implementer asking the same question finds it written down rather than
inferred. Disposition B is the right follow-on — but it answers a
different ticket (the volume concern is properly
[FHIR-54409](https://jira.hl7.org/browse/FHIR-54409) and
[FHIR-54384](https://jira.hl7.org/browse/FHIR-54384), and the
throttling design should be debated there with full registry input).
Disposition C is technically defensible but leaves the documentation
gap that produced this ticket in the first place. Recommend A here and
flag in the WG meeting that the related volume/cadence design question
is being handled under the FHIR-54409/FHIR-54384 thread.
