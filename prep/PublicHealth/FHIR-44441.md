## Summary

Ron G. Parker requests that the code system and value set URLs used by the
Vital Records Common Profiles Library (VRCL) be anchored under
`terminology.hl7.org` (THO), in line with the TSC-approved
[Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy. Specifically, the ticket asks the work group to initiate the
UTG (Unified Terminology Governance) process to migrate the affected
value set into THO, after which a future version of the IG should
reference the THO-hosted artifact rather than carrying it locally. The
ticket originated as comment 21/31 imported during the VRCL ballot
reconciliation cycle. No cross-referenced Jira tickets, Zulip threads,
or GitHub items were surfaced for FHIR-44441 in the FHIR Augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive – With Modification. Initiate the UTG process to relocate
the in-scope code system(s) and value set(s) currently defined inside
the Vital Records Common Profiles Library (VRCL) to
`terminology.hl7.org` (THO), preserving canonical URLs as
`http://terminology.hl7.org/CodeSystem/{id}` and
`http://terminology.hl7.org/ValueSet/{id}`. Concretely:

1. Inventory the VRCL `input/resources/CodeSystem-*.json` and
   `input/resources/ValueSet-*.json` artifacts that are not specific
   to vital records and are of "general utility" (per Lynn Laakso's
   2024-10-11 comment).
2. Open a UTG harmonization proposal (UP) per the
   [Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
   policy, with Vocabulary as steward unless a more specific work
   group claims it.
3. Once UTG accepts the artifact and it ships in a THO release, retire
   the local copy in the VRCL and update all `binding`,
   `compose.include.system`, and `compose.include.valueSet`
   references in VRCL profiles to the THO canonicals.
4. Track the migration with a follow-up Jira ticket against the
   next VRCL STU version so the binding swap is not lost.

#### Justification

This is exactly what the TSC-approved policy requires: terminology that
is reusable across IGs must live in THO rather than in an IG-private
namespace. Lynn's reverted resolution explicitly recognises the value
set is "of general utility" and commits Public Health to begin the move,
so accepting the request is consistent both with HL7 policy and with
the work group's already-stated intent. The two-phase approach
(initiate UTG now, retarget bindings in a future VRCL release)
acknowledges that the THO publication cadence is independent of the
VRCL ballot cycle and avoids blocking VRCL on UTG turnaround.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive – With Modification (split scope). Rather than treating
FHIR-44441 as a single migration, fan it out into a tracking ticket
plus one child UP per code system / value set that needs to move:

- Keep FHIR-44441 open as the umbrella tracker linking to each child
  UP in UTG and any per-artifact Jira tickets.
- For artifacts that are demonstrably *vital-records-specific*
  (i.e., where Lynn's "general utility" caveat does not apply), keep
  them in the VRCL namespace under
  `http://hl7.org/fhir/us/vrcl/CodeSystem|ValueSet/...` rather than
  forcing them through THO.
- For each generally-useful artifact, open a UTG proposal with a
  proposed steward (Vocabulary by default; Patient Care, OO, etc.
  where the content is clearly theirs) and only retarget VRCL bindings
  once the THO artifact is published.

#### Justification

The original ticket text ("this value set") and the imported comment
number suggest it was filed against a specific item in the ballot
spreadsheet but is being read here as a blanket directive over all
VRCL terminology. Splitting the work surfaces *which* artifacts are
actually in scope, lets the work group discuss steward assignments
per artifact, and avoids accidentally moving content that is genuinely
vital-records-specific (e.g., jurisdictional reporting codes) into
THO where it would be awkwardly hosted. It is also more transparent
to implementers, because the umbrella ticket gives them a single place
to watch progress without conflating unrelated migrations.

---

### Disposition C: Decline

#### Proposal

Not Persuasive. Close FHIR-44441 without action, on the grounds that
(a) STU2 has already published with the value sets defined locally in
the VRCL namespace, (b) a future-version-only retarget is already
implicit in HL7's general guidance, and (c) the request as written
does not name a specific code system or value set.

#### Justification

Declining is defensible only on procedural grounds: the ticket was
imported as ballot comment 31 with no concrete artifact named, and
the substantive commitment (move to THO) was already recorded in
Lynn's October 2024 comment. Closing as "Not Persuasive – No Action"
would, however, contradict TSC's
[Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy, which makes THO anchoring a normative expectation, not an
optional improvement. It would also leave the VRCL out of compliance
with TSMG guidance going into the next ballot cycle, so this option
is included for completeness rather than as a serious recommendation.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because it operationalises Disposition A
without losing nuance. The ticket as written treats VRCL terminology
monolithically, but Lynn's own comment distinguishes content that is
"of general utility" from content that is genuinely vital-records-
specific — and only the former belongs in THO. Turning FHIR-44441 into
an umbrella tracker with per-artifact UPs forces that triage to happen
explicitly, gives Vocabulary (or another steward) a clean per-artifact
surface to react to, and lets VRCL retarget its bindings incrementally
as each THO publication lands. Disposition A is the right destination
but, executed as a single batch, risks stalling on the slowest UTG
proposal and entangles unrelated artifacts. Disposition C is included
only as a procedural counterpoint and conflicts with TSC policy, so
the work group should not adopt it.
