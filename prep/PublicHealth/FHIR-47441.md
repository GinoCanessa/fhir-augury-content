## Summary

The reporter (Corey Spears) observes that the NHSN dQM Reporting IG currently
defines only a single submission profile — the **NHSN MeasureReport Bundle** —
intended to carry an entire submission as one Bundle. For high-volume
submitters, that single Bundle can plausibly reach gigabyte scale, which is
incompatible with chunked / resumable transports such as the FHIR Bulk Data
export pattern (NDJSON, one resource per line). The ticket asks the Public
Health work group to define **additional submission profiles** that decompose
a submission into smaller units that can be transported via Bulk Data or
similar mechanisms — for example, a "Header Measure Bundle" that carries
submission-level metadata plus per-subject "Individual Report Bundles" that
each carry one subject's MeasureReport(s) and `MeasureReport.evaluatedResources`.
The ticket explicitly points at the Da Vinci DEQM pattern of organizing Bundles
by subject so that resources shared across measures are not duplicated.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items in
the Augury index — the only cross-reference is a FHIR element mention of
`MeasureReport.evaluatedResources`. Comment history shows the work group
originally voted (35-0-0) to defer to a future version (Considered for Future
Use, 2024-12-12), citing in-flight Argonaut Bulk Data work and the HRSA UDS+
import-manifest pattern as efforts to align with. That resolution was
**reverted on 2025-07-09** by Lynn Laakso to re-open the ticket post-STU1
publication, so the work group now needs a fresh disposition.

## Details

**Description:**

This IG notes that submissions may be large. The only submission profile in
this IG is the NHSN MeasureReport Bundle which is a single bundle. It is
conceivable that a single submission could measure in the GB or larger. A
single resource does not lend itself to assured and uninterrupted transmission
in smaller chunks to support methods like Bulk Data Transport IG (as that
required a resource to be on a single line with NDJSON). While it may be
possible to support large single resource transport in smaller chunks, it
would require non-FHIR ways of doing so.

Create submission profiles that enable the use of bulk data and more granular
level transport. (Suggest considering created a Header Measure Bundle profile
and an Individual Report Bundle profile (e.g. that support an individual level
measure report).

To align with DEQM, the Bundles used in the DEQM operations enable the
evaluation and exchange of data for multiple measures, while also constraining
duplicate data. Each Bundle SHOULD contain the resources, including
MeasureReports and data of interest (`MeasureReport.evaluatedResources`), for
all of the measures that apply to a single subject. Organizing the Bundles by
subject means that resources are less likely to be duplicated when used by
multiple measures. (See
<https://hl7.org/fhir/us/davinci-deqm/2024Sep/guidance.html#deqm-operation-bundles-organized-by-subject>.)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add two new submission Bundle profiles to the NHSN dQM Reporting IG alongside
the existing **NHSN MeasureReport Bundle**:

1. **NHSN Header Measure Bundle** — a thin submission-level Bundle carrying
   submission metadata (reporting period, reporting Organization, Measure
   reference(s), submission identifier, counts/manifest of related Individual
   Report Bundles) but no per-subject `MeasureReport` payload. This becomes
   the entry point for a multi-file submission and gives receivers a manifest
   they can reconcile against.
2. **NHSN Individual Report Bundle** — a subject-scoped Bundle constrained to
   carry the `MeasureReport`(s) and supporting `MeasureReport.evaluatedResources`
   for **one subject** across all measures in the submission, modeled on the
   DEQM "Bundles organized by subject" pattern
   (<https://hl7.org/fhir/us/davinci-deqm/2024Sep/guidance.html#deqm-operation-bundles-organized-by-subject>)
   so shared resources are not duplicated across measures.

Update the IG's transport / submission narrative to describe how a submission
is decomposed into one Header Bundle plus N Individual Report Bundles, and
permit the Individual Report Bundles to be exchanged via FHIR Bulk Data
($export / NDJSON one-resource-per-line) in addition to the existing single-
Bundle path. Retain the current NHSN MeasureReport Bundle profile for small
submissions and backward compatibility, and add conformance language stating
when each profile is appropriate (e.g., size threshold or submitter election).

#### Justification

This is what the reporter literally asked for, and it lands the IG on a
pattern that is already well-established in adjacent specifications: the
DEQM subject-organized Bundle pattern is explicitly cited in the ticket and
solves the de-duplication concern for `MeasureReport.evaluatedResources`
shared across measures. Splitting submissions by subject also makes each
"unit of work" small enough to fit comfortably into the FHIR Bulk Data
NDJSON model, which the ticket correctly identifies as incompatible with a
single GB-scale Bundle. Adding a Header Bundle gives receivers a manifest
to reconcile partial / interrupted submissions, which is the central
reliability problem the ticket raises.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting two new core submission profiles in this STU cycle,
publish a **non-normative "Bulk / Chunked Submission" guidance page** in the
NHSN dQM Reporting IG that:

- describes the size problem and the single-Bundle limitation;
- defines a lightweight **manifest pattern** (modeled on HRSA UDS+
  `uds-plus-import-manifest`,
  <https://fhir.org/guides/hrsa/uds-plus/StructureDefinition-uds-plus-import-manifest.html>,
  and the in-flight Argonaut Bulk Data file-organization work,
  <https://build.fhir.org/ig/HL7/bulk-data/branches/argo24/export.html#bulk-data-output-file-organization>);
- recommends — but does not yet normatively profile — splitting large
  submissions into per-subject NDJSON files keyed off `MeasureReport.subject`
  with `MeasureReport.evaluatedResources` colocated in the same file; and
- explicitly defers a normative Header / Individual Report Bundle profile
  pair to the next ballot cycle, once Argonaut and HRSA work have stabilised.

Optionally, ship a single **slice / variant** on the existing NHSN
MeasureReport Bundle profile that constrains a "single-subject" form, so
implementers can experiment without a profile proliferation.

#### Justification

The work group's original 2024-12-12 disposition (Considered for Future Use,
35-0-0) was specifically motivated by the desire to align with the Argonaut
Bulk Data and HRSA UDS+ manifest work that was still in flight. That
rationale has not gone away — committing now to a Header / Individual Report
Bundle pair risks publishing a profile shape that is incompatible with the
manifest conventions those efforts ultimately settle on, forcing a breaking
change in a later version. A guidance-plus-manifest-pattern release lets
NHSN signal the direction of travel and enable early implementer experiments
without locking in normative artifacts that may need to be re-cut.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — Out of Scope for this STU revision**.
State that the FHIR Bulk Data Access IG and the underlying $export operation
are the appropriate venue for chunked / NDJSON transport of large FHIR
payloads, and that NHSN dQM submitters with very large submissions should
either (a) use Bulk Data $export against their own data store and submit the
resulting NDJSON files out-of-band, or (b) split their submissions along
reporting-period or facility boundaries using the existing NHSN MeasureReport
Bundle, rather than have this IG mint additional submission Bundle profiles.

#### Justification

The IG's current single-Bundle submission profile is internally consistent
and was balloted as such; introducing a Header / Individual Report Bundle
pair mid-cycle expands the IG's surface area, increases conformance burden
on receivers (who must now accept three Bundle shapes), and partly duplicates
work being done in Bulk Data, Argonaut, and HRSA UDS+. If the underlying
problem is transport-level reliability, that is properly the concern of the
Bulk Data spec, not of a measure-reporting IG.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is the strongest fit for the current state of the world. The
ticket reporter is correct that a single GB-scale NHSN MeasureReport Bundle
is operationally untenable, so Disposition C (decline) understates a real
implementer pain point and would likely return as a ballot comment. But
Disposition A's Header / Individual Report Bundle pair is exactly the kind
of profile that the work group's earlier 35-0-0 vote chose to defer
specifically to keep alignment with the Argonaut Bulk Data file-organization
work and the HRSA UDS+ import-manifest pattern that the reporter himself
flagged in the comments. Re-opening the ticket post-STU1 is a good moment
to publish *guidance* — naming the size problem, pointing at DEQM's
subject-organized Bundles as the design north star, and aligning with the
emerging manifest convention — without committing the IG to a normative
profile shape that may need to be re-cut once those upstream efforts land.
A normative Header / Individual Report Bundle pair (Disposition A) should
then be the headline feature of the next ballot cycle, built on whatever
manifest convention the broader community converges on.
