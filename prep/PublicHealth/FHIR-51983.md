## Summary

Craig Newman observes an internal inconsistency in the **US Helios Query and
Response (FHIR)** implementation guide between two pieces of guidance about
how broadly a public-health querier should ask for clinical data. The
*Minimization of Queries* section recommends using `Observation.category`
codes (e.g., `vital-signs`, `laboratory`) to retrieve a comparatively wide
slice of data and then filter it down on the requester side. The *FHIR Query
Basics* section, by contrast, urges queriers to be as narrow as possible —
its example explicitly says that a flu follow-up query should not pull
back STI-related lab results. The reporter is asking the IG authors to
add a focused discussion of the trade-offs (round-trips, server load,
specificity of search parameters, and confidentiality / minimum-necessary
obligations) so implementers understand when each strategy applies and
why the IG appears to recommend both. The ticket has been triaged
**Persuasive with Modification**, with the resolution noting that
language updates suggested by the reporter will be incorporated.

No cross-referenced Jira tickets, Zulip threads, GitHub items, or
Confluence pages were surfaced for this issue by the FHIR Augury
indexes.

## Details

**Description:**

> In the Minimization of Queries section, it talks about using the
> category (vital signs or lab) to retrieve a larger set of data, but
> this seems at odds with the concept of Minimum Necessary and
> elsewhere in the document where the advice is to be as specific as
> possible (such as in FHIR Query Basics where it says that flu follow
> ups shouldn't ask for data related to STI lab results). I think there
> should be a focused discussion of the pros and cons of each approach.

**Resolution Description:**

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new, focused subsection — e.g., **"Choosing query breadth: category
filtering vs. narrow search parameters"** — within the *Minimization of
Queries* page of the US Helios Query and Response IG. The subsection
would explicitly enumerate the trade-offs between the two approaches:

- **Broad category-based query + post-filtering** (e.g.,
  `Observation?category=laboratory&patient=…`): fewer round-trips,
  simpler responder implementation, easier reuse of server-side search
  parameters, but the responder transmits data the requester does not
  strictly need, increasing privacy / Minimum-Necessary risk and
  payload size.
- **Narrow, condition-specific query**
  (e.g., `Observation?code=…&patient=…` scoped to flu-relevant LOINC
  codes): better aligned with HIPAA Minimum Necessary and with the
  *FHIR Query Basics* example that excludes STI labs from a flu
  follow-up; but requires the requester to maintain a curated
  value-set, may need multiple queries, and depends on responder
  support for the relevant search parameters.

Add cross-links from both the *Minimization of Queries* section and the
*FHIR Query Basics* section to this new discussion so a reader who
arrives at either page sees the same reconciled guidance.
Source-file edits are expected under the Helios Q&R IG's
`input/pagecontent/` authoring area (file names to be confirmed in the
IG's repository).

#### Justification

This is exactly what the reporter asked for and matches the recorded
**Persuasive with Modification** resolution ("Updated language as
recommended by Craig Newman"). The change is narrative-only, so it does
not affect conformance, profiles, or value sets, and carries no
implementation burden for systems that already conform to the IG. It
removes a real internal contradiction that implementers will hit on
their first careful read, and it acknowledges the HIPAA Minimum
Necessary obligation that public-health queriers operate under.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a single new subsection, **restructure the guidance
into a decision framework** keyed off the *use case* the querier is
serving:

1. **Surveillance / case-finding** (broad cohort discovery) — broad
   category queries are appropriate, with explicit normative-style
   `SHOULD` language about responder-side or requester-side
   confidentiality filtering.
2. **Case investigation / follow-up on a known condition** — narrow,
   code-scoped queries are required; broad category queries `SHOULD
   NOT` be used.
3. **Mixed cases** — guidance on issuing multiple narrow queries vs. a
   single broad query plus post-filter, including a worked example
   that mirrors the existing flu / STI illustration.

Move the *FHIR Query Basics* STI / flu example into this framework so
the example is the canonical illustration of (2), and rewrite the
*Minimization of Queries* paragraph as the canonical illustration of
(1). Add a short table summarizing trade-offs (round-trips, payload
size, Minimum Necessary alignment, responder load, search-parameter
support).

#### Justification

This goes beyond reconciling two paragraphs and gives implementers an
actionable rule for *when* to apply each strategy, which is the
underlying need behind the ticket. It is more durable than a single
"pros and cons" subsection because future readers won't have to
re-derive the rule from the prose. Trade-off vs. A: it is a larger
editorial change, touches two existing sections rather than just adding
to one, and risks introducing new ambiguity if the use-case taxonomy is
not vetted by the workgroup. Because this IG is currently in
ballot-comment cycles (see the `BlockVote6HeliosQR` label), reviewers
may prefer the smaller, surgical edit in A unless there is appetite to
re-open the broader narrative.

---

### Disposition C: Decline

#### Proposal

Decline to make a substantive change. Rationale: the two sections are
addressing different operational contexts (cohort-style surveillance
vs. targeted follow-up), and a careful reader can reconcile them. Close
as **Not Persuasive** with a brief comment pointing the reporter at
both sections and noting that Minimum Necessary is already discussed
elsewhere in the IG.

#### Justification

Defensible only if the workgroup believes the existing text is already
clear and that adding more discussion risks making the IG longer
without adding normative clarity. Cite the existing *FHIR Query
Basics* STI example as evidence that the IG already privileges
specificity when the use case calls for it.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been triaged **Persuasive with Modification**
with the resolution noting that the reporter's own wording will be
adopted, so the workgroup has effectively pre-selected option A. The
change is small, narrative-only, and removes a contradiction that will
otherwise generate repeat ballot comments in future cycles. Disposition
B is appealing as a longer-term restructuring, but it expands scope
beyond what the reporter requested and is risky to undertake mid-ballot
under the `BlockVote6HeliosQR` block. Disposition C is hard to
justify on the record because the reporter is correct that the two
sections, as written, give conflicting guidance — declining would
likely be re-raised on the next ballot. The Public Health workgroup
should adopt A as drafted, and if there is appetite later, fold A's
new subsection into a B-style decision framework in a future
publication.
