## Summary

Reinhard Egelkraut requests an improvement to the ConceptMaps published in
the DICOM SR to FHIR Resource Mapping IG, specifically the tables in
§6.24.1 *ConceptMap — DICOM Imaging Measurement Group to FHIR Mapping*
and §6.26.1 *ConceptMap — DICOM Imaging Measurements Container to FHIR
Mapping*. Because some source codes in these tables are drawn from more
than one coding scheme (e.g., DCM, SCT/SNOMED CT, LOINC, UCUM), the
displayed entries are ambiguous when only the code value is shown. The
ticket asks that every source code in those tables be **fully qualified**
with its coding scheme so a reader can unambiguously identify the
intended concept.

The request is part of the September 2024 SR ConceptMap review tranche
(label `II_SEP_2024_SR_ConceptMap`) and sits alongside several sibling
tickets against the same IG: [FHIR-47941](https://jira.hl7.org/browse/FHIR-47941)
(use of `Observation.bodySite` when only a code is present —
*Persuasive/Applied*), [FHIR-47510](https://jira.hl7.org/browse/FHIR-47510)
(rename §6.24.1 section title to align with FHIR terminology — still
*Triaged*), and [FHIR-48643](https://jira.hl7.org/browse/FHIR-48643)
(specification unclear on how to handle concept modifiers — *Triaged*).
[FHIR-51155](https://jira.hl7.org/browse/FHIR-51155) (filed against the
EU Imaging Study Report IG) addresses a closely related but distinct
quality issue — incorrect ConceptMap target URLs — and was closed
*Not Persuasive*. Together these tickets indicate a focused, low-risk
clean-up pass on the ConceptMap presentation layer of the DICOM SR IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the ConceptMap tables in §6.24.1 (*ConceptMap — DICOM Imaging
Measurement Group to FHIR Mapping*) and §6.26.1 (*ConceptMap — DICOM
Imaging Measurements Container to FHIR Mapping*) of the DICOM SR to
FHIR Resource Mapping IG so that every entry in the *Source Code*
column is fully qualified with its coding scheme. Concretely:

- Add (or restore) a *Source Coding System* column on both tables, or
  render each cell as `system | code | display` so it is clear whether
  a code originates in DCM, SNOMED CT (SCT), LOINC, UCUM, or another
  vocabulary.
- Apply the same treatment to the *Target Code* column where it can
  draw from more than one system, for symmetry.
- Where the underlying ConceptMap resources already carry
  `group.source` / `group.target` system URLs, regenerate the page so
  the rendered table reflects them per source row rather than only at
  the group header.

This is a presentation / authoring fix in the IG; the underlying
ConceptMap resource model (`group.element.code` plus `group.source`)
already supports per-row systems when groups are split correctly.

#### Justification

The request is squarely within the same intent as the rest of the
`II_SEP_2024_SR_ConceptMap` tranche: make the published ConceptMap
tables unambiguous for implementers. Mixed-vocabulary source codes
shown without their system are a real source of ambiguity in
DICOM↔FHIR mapping (DICOM uses DCM-coded items alongside SCT/LOINC/UCUM
in the same TIDs), and FHIR Coding semantics are explicit that code
values are only meaningful in the context of a system. The change is
backwards-compatible (no change to ConceptMap resource semantics, just
to how rows are displayed) and aligns with sibling ticket
[FHIR-47510](https://jira.hl7.org/browse/FHIR-47510), which is also a
presentation fix on §6.24.1.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (only) editing the rendered tables, restructure the
underlying ConceptMap resources for §6.24.1 and §6.26.1 so that each
source coding system is its own `ConceptMap.group` (with `source` /
`sourceCanonical` set to the appropriate system URL — DCM, SCT, LOINC,
UCUM, etc.). Then re-run the IG publisher so the auto-generated tables
inherit a per-group system header, removing the need to introduce a
new column. As part of the same change, document in §6.24 / §6.26
narrative that "Source Code" rows must be read in the context of their
group's source system.

#### Justification

This treats the ambiguity as a *data* defect rather than a *display*
defect, which is more durable: any downstream tooling that consumes
the ConceptMap resources (validators, mapping engines, IG publishers
in other contexts) gets the corrected systems too, not just human
readers of the HL7-published HTML. It is consistent with how other
multi-system FHIR ConceptMaps in the broader ecosystem are authored
(e.g., the per-group pattern recommended in
[FHIR-33061](https://jira.hl7.org/browse/FHIR-33061)). The trade-off
is that it is a larger edit (multiple ConceptMap resources rather than
two narrative pages) and requires regeneration plus republication of
the IG.

---

### Disposition C: Decline

#### Proposal

Close FHIR-47693 as *Not Persuasive*, on the grounds that the source
coding scheme for each row in §6.24.1 / §6.26.1 is already implied by
the section text and by the DICOM TID context, and that adding scheme
qualification to every row would clutter the tables without changing
the normative ConceptMap content.

#### Justification

This would be defensible only if the workgroup believes the implicit
context (TID number, IG-wide convention that "DCM" is the default
unless otherwise stated) is sufficient for the IG's intended
audience — DICOM and FHIR implementers who already know the DCM coding
system. It is, however, a weak position: the ticket is reported by an
implementer who explicitly states the tables are ambiguous in
practice, and the broader tranche of `II_SEP_2024_SR_ConceptMap`
tickets shows the community treats these tables as the canonical
mapping reference, not as an internal aide-mémoire. Cite this
disposition only if the workgroup wants to push the request into a
broader "redesign the ConceptMap presentation" work item rather than
patch §6.24 / §6.26 in isolation.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested), with the
authoring change executed in a way that, where practical, also
realises Disposition B's structural improvement.

The request is small, well-targeted, raised by an implementer, and
aligned with the rest of the September-2024 ConceptMap tranche
(notably FHIR-47941, which has already been *Applied*, and FHIR-47510,
which sits on the very same section). Fully qualifying the source
codes is a strict clarity improvement with no normative downside and
no breaking impact on resources or profiles. Where the underlying
ConceptMap groups can cleanly be split by source system, the
implementer should prefer that route (Disposition B's data-side fix)
because it propagates the disambiguation to anyone consuming the
ConceptMap resources directly; where a single mixed group is
unavoidable, an explicit per-row *Source Coding System* column
satisfies the ticket. Disposition C is not recommended — declining
would leave a real, implementer-reported ambiguity unaddressed in the
IG's principal mapping tables.
