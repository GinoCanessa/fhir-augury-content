## Summary

The ticket reports a small but concrete error in the FHIR Core specification's
DICOM Tag Mapping table for `ImagingStudy` (section 10.4.9.5). The row for
`numberOfInstances` at the **series** level is currently labelled as "DICOM
instance metadata" when, by definition, it describes the series and should
read "DICOM series metadata". This is a documentation/mapping-table
correction only — no schema, cardinality, or terminology change is
implied — and the request is anchored to the canonical DICOM standard
(<https://dicomstandard.org/current>) for verification. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were found, so the request stands
on its own.

>

> while it should be "DICOM series metadata".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the DICOM Tag Mapping table in the `ImagingStudy` specification
(section **10.4.9.5 DICOM Tag Mapping**) so that the row whose FHIR path is
`ImagingStudy.series.numberOfInstances` shows the metadata-source column as
**"DICOM series metadata"** instead of "DICOM instance metadata". In the
`HL7/fhir` repository this mapping table is authored under
`source/imagingstudy/` (the per-resource folder under the `source/` authoring
root used by the bespoke FHIR publisher); the change is a single
text/cell edit in the mapping page/table for `ImagingStudy` and requires no
StructureDefinition, cardinality, ValueSet, or example changes. No
publisher behaviour, schema, or downstream profile is affected.

#### Justification

The DICOM standard models `(0020,1209) Number of Series Related Instances`
as a **Series-level** attribute (it is part of the General Series module at
the Series IE), so a row that lives under the Series block of the
ImagingStudy DICOM mapping must, by definition, be sourced from "DICOM series
metadata", not from instance metadata. The change therefore restores
internal consistency between the section's row grouping (Series level) and
its "metadata source" annotation, and aligns the FHIR mapping with the
authoritative DICOM specification cited in the ticket. The fix is
purely documentary, has no implementer impact, and is well within the kind
of editorial correction the FHIR publisher's prose-edit workflow handles in
a single PR.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the correction, but take the opportunity to do a **targeted
audit** of the entire DICOM Tag Mapping table in `ImagingStudy` (section
10.4.9.5) to confirm that every row's "metadata source" annotation matches
its section grouping (Study / Series / Instance). Apply any further
corrections in the same PR. No structural changes to the table layout —
just verifying each metadata-source value against the DICOM IE/module the
attribute lives in (e.g., General Study / Patient Study modules at the
Study IE; General/Specialized series modules at the Series IE; SOP common
and IOD-specific modules at the Instance IE).

#### Justification

A miscategorisation in one row in a hand-maintained mapping table is a
soft signal that other rows may have drifted as well, especially for
attributes whose IE assignment in DICOM has changed across CP cycles.
Bundling a quick audit with the requested fix avoids opening (and
re-reviewing) a follow-up ticket for the same table, gives reviewers one
authoritative diff to compare against the DICOM standard, and produces a
mapping table the Imaging Integration WG can confidently cite. The cost is
small (one table, three IE columns) compared with the cost of repeated
single-row fixes in successive ballot cycles.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — already addressable as an editorial
fix outside the change-request stream**, on the grounds that the requested
edit is a one-cell label correction with no normative content change and
no implementer impact, and could be handled as routine editorial cleanup
without consuming a workgroup disposition slot.

#### Justification

The ticket carries no semantic risk, no implementer-visible behaviour
change, and no downstream profile or terminology impact; it is the kind of
typo-class correction that an editor could pick up during pre-publication
QA. Declining as a change request (while still making the edit
editorially) keeps the change-tracking ledger focused on substantive
disposition discussions. This is the weakest of the three options because
the ticket has been formally lodged and the corrected wording is
unambiguous, but it is recorded here as the principled "do not adopt"
alternative.

---

### Recommendation

**Recommended disposition:** A

The fix is unambiguous, the source of truth (the DICOM standard) is cited
directly in the ticket, and the row in question is structurally located
under the Series block of the `ImagingStudy` DICOM Tag Mapping — so
labelling it "DICOM series metadata" is the only internally consistent
value. There is no implementer cost, no downstream profile to renegotiate,
and no Zulip or PR discussion suggesting a contrary view. Disposition B
is attractive but expands scope beyond what the reporter raised and
should be tracked separately if the WG wants a full table audit;
Disposition C understates the value of having a citable, dispositioned
record of the correction in Jira. Apply A in `HL7/fhir` under
`source/imagingstudy/` and reference FHIR-54784 in the commit message so the
R6 DICOM Tag Mapping ledger (label `R6-DICOMTagMapping`) stays complete.
