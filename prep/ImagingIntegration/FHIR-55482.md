## Summary

The ticket asks the Imaging Integration work group to consider changing
`ImagingStudy.series.description` from its current `string` datatype to
`CodeableConcept`, so that a series description can carry a coded value
(e.g., from the RSNA RADLex Playbook Series effort) in addition to – or
instead of – free text. The reporter notes that, as an alternative to
retyping the existing element, a *new* sibling element on
`ImagingStudy.series` could be introduced to hold the coded equivalent.
The motivating use case is alignment with DICOM, where the Series
Description Code Sequence `(0008,103F)` provides a coded counterpart to
the free-text Series Description `(0008,103E)`, and where RSNA's RADLex
Playbook Series project is producing a candidate code system for these
descriptions. There are no cross-referenced Jira tickets, Zulip threads,
or GitHub items linked to this issue in Augury at the time of review.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the datatype of `ImagingStudy.series.description` from `string`
to `CodeableConcept` in FHIR Core (HL7/fhir, R6 / next normative cycle),
keeping cardinality `0..1`. Add an example binding (Strength: `example`,
or `preferred` if the work group is comfortable) to a new value set
seeded from the RSNA RADLex Playbook Series codes, and document the
alignment with DICOM `(0008,103F)` Series Description Code Sequence in
the element definition. Free-text equivalents would move into
`CodeableConcept.text`, preserving the human-readable Series Description
content currently held by the `string`.

#### Justification

This is the most direct fulfillment of the ticket: a single element
captures both the free-text label (via `.text`) and the coded
representation (via `.coding`), neatly mirroring how DICOM exposes
`(0008,103E)` and `(0008,103F)` as paired concepts. `CodeableConcept`
is FHIR's idiomatic answer to "string-or-code" and avoids proliferating
sibling elements. However, this is a **breaking change** to a published
resource element: existing R4/R5 instances populated as `string` will
not validate against the new schema, and every implementer (PACS, VNAs,
viewers, FHIRcast clients) that reads or writes
`ImagingStudy.series.description` will need to migrate. That cost is
material and should be weighed before recommending Disposition A.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `ImagingStudy.series.description` as `string` and add a new
sibling element `ImagingStudy.series.descriptionCode : CodeableConcept
[0..1]` (or a similarly named element such as `series.code`, if the
work group prefers naming consistent with `ImagingStudy.modality` /
`ImagingStudy.series.modality`). Bind the new element with `example` or
`preferred` strength to a value set seeded from the RSNA RADLex
Playbook Series, and add invariants/notes that the two elements should
be semantically aligned when both are populated. Map the new element
explicitly to DICOM `(0008,103F)` Series Description Code Sequence and
keep the existing `description` mapped to `(0008,103E)`.

#### Justification

This approach is non-breaking: existing instances and implementations
continue to work unchanged, and the coded representation is opt-in for
sites and vendors ready to adopt RADLex. It also more faithfully
mirrors the DICOM information model, which keeps `(0008,103E)` and
`(0008,103F)` as distinct attributes rather than collapsing them.
Trade-offs vs. Disposition A: two elements are slightly more verbose
and risk drift between the free-text and coded forms, and reviewers
may dislike adding another `series.*` element for what the reporter
frames as essentially one concept. These are real but, in the Imaging
Integration domain where DICOM parity matters, generally outweighed by
the avoided breakage.

---

### Disposition C: Decline

#### Proposal

Decline the request and direct implementers to use the existing
extension mechanism (e.g., a published extension such as
`http://hl7.org/fhir/StructureDefinition/imagingstudy-series-descriptionCode`,
defined in the `HL7/fhir-extensions` IG) to attach a `CodeableConcept`
on `ImagingStudy.series` when a coded series description is needed.
Rationale category: *already addressable by an existing mechanism;
breaking change to a normative-track element is not justified by the
current evidence (a single ticket, no surfaced Zulip/GitHub demand,
and a value set that is still under development at RSNA).*

#### Justification

`ImagingStudy` is widely deployed and the RADLex Playbook Series codes
are still being curated by RSNA, so the value set that would back any
binding is not yet stable. A core-spec change — whether retyping
`description` (Disposition A) or adding a sibling element (Disposition
B) — locks in a design before the terminology has matured. An
extension keeps the door open for early adopters without imposing
schema or behavioral changes on the broader implementer base, and can
be promoted into core in a future cycle once the RADLex value set, and
real-world demand, are firmer.

---

### Recommendation

**Recommended disposition:** B

Adding a new `ImagingStudy.series.descriptionCode : CodeableConcept`
element (Disposition B) gets the work group most of the way to what
the reporter asked for, while preserving the existing `string`
contract that R4/R5 implementers depend on. It also matches the DICOM
information model — `(0008,103E)` text and `(0008,103F)` code as two
distinct attributes — which is the model PACS and VNAs already speak
fluently. Disposition A is conceptually cleaner but the breaking-change
cost on a heavily implemented resource is hard to justify on the
strength of a single ticket with no surfaced community discussion or
GitHub work. Disposition C is defensible if the work group judges the
RADLex value set too immature to commit to in core, but an extension
will not give the imaging community the discoverability or DICOM
mapping that a first-class element provides — and the work group can
always defer the binding strength (`example`) and value set
finalization while still landing the structural change. If the
Imaging Integration WG wants a middle path, it can adopt B now with an
`example` binding and tighten the binding once RSNA's RADLex Playbook
Series codes stabilize.
