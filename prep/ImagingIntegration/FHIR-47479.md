## Summary

The reporter notes that DICOM SR Template TID 300 defines four
post-coordination concept-modifier rows — `$ModType`, `$Method`,
`$Derivation`, and `$TargetSite` — that "allow the explicit definition
of terms for post-coordination of the measurement concept name." In the
current DICOM SR to FHIR Resource Mapping IG, the
`DICOM SR Imaging Measurement Mapping to Observation` profile maps
`$Method` (to `Observation.method`) and `$TargetSite` (to
`Observation.bodySite`/`bodyStructure` per the resolution of
[FHIR-37380](https://jira.hl7.org/browse/FHIR-37380)), but there is **no
mapping target for `$ModType` or `$Derivation`**. As a result, those
post-coordination terms are silently dropped, which loses information
that is required to fully reconstruct the measurement's concept name.

The reporter recommends that the IG provide standard extensions on the
mapped `Observation` profiles so that `$ModType` and `$Derivation` can
be communicated in a uniform, parseable way rather than each
implementation inventing its own pattern.

This question is closely related to a broader, still-open issue raised
on the same IG by Harald Zachmann via Jonathan Whitby
([FHIR-48643](https://jira.hl7.org/browse/FHIR-48643)), which asks the
same general question — "how should TID 300 (and TID 1501) concept
modifiers be mapped?" — and proposes using `Observation.component`
code/value pairs for *all* such modifiers. FHIR-48643 is the natural
companion ticket and any disposition here should align with it. Two
already-applied tickets on the same IG provide local precedent for
mapping individual TID 300 rows:
[FHIR-47648](https://jira.hl7.org/browse/FHIR-47648) (device mapping
correction) and
[FHIR-47518](https://jira.hl7.org/browse/FHIR-47518) (description
cleanup). On the wider FHIR side, Orders & Observations resolved
[FHIR-54904](https://jira.hl7.org/browse/FHIR-54904) ("Support recording
multiple details of Observation Context and/or Method") in R6 by
clarifying `Observation.interpretationContext` so that contextual
information about the observation value (including imaging timing,
view, and mode) is in scope — a relevant base-spec lever this IG can
lean on.

## Details

**Description:**

> According to TID300, $ModType, $Method, $Derivation and $TargetSite
> "allow the explicit definition of terms for post-coordination of the
> measurement concept name". Currently it is not possible to capture
> the values for $ModType nor $Derivation. This would lead into loss of
> relevant elements needed for the post-coordination.
>
> It would be good if the IG provides the corresponding extensions to
> the Observation, in order to have a standard way of communicating
> such information.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add two new IG-defined extensions to the
`DICOM SR Imaging Measurement Mapping to Observation` profile (and, in
parallel, to `DICOM SR Imaging Qualitative Evaluation Mapping to
Observation` so Qualitative Evaluation mappings have the same coverage):

- `imaging-measurement-mod-type` — `Extension(CodeableConcept)`,
  cardinality 0..*, bound (extensible) to a value set listing TID 300
  Row 2 ("Modifier Type") concept codes from DCM, mapped from TID 300
  `$ModType`.
- `imaging-measurement-derivation` — `Extension(CodeableConcept)`,
  cardinality 0..*, bound (extensible) to a value set listing TID 300
  Row 6 ("Derivation") concept codes from DCM, mapped from TID 300
  `$Derivation`.

Update the `ImagingMeasurement` (and `ImagingQualitativeEvaluation`)
StructureDefinitions in `HL7/dicom-sr` (the same family of files
amended in commit
[`7d18dd0`](https://github.com/HL7/dicom-sr/commit/7d18dd044dc1fa8fac4af33b13c58f61b5b89efd))
to declare the extensions on `Observation`, and add the corresponding
TID 300 → extension rows to the mapping narrative pages
(`StructureDefinition-imaging-measurement.*`). Mark FHIR-48643 as
"resolved by the same change" if the WG agrees the same extensions cover
the broader concept-modifier question.

#### Justification

This is what the reporter asked for, and it preserves the IG's
existing modeling pattern: each TID 300 post-coordination row is mapped
to a *named* place on the `Observation` profile rather than into a
generic bag. That mirrors the precedent already set by
[FHIR-37380](https://jira.hl7.org/browse/FHIR-37380), where
`$TargetSite` was mapped onto the typed `Observation.bodyStructure`
element with named sub-modifiers (Finding Site, Laterality,
Topographical modifier), and by `$Method` → `Observation.method`. Named
extensions are easy for downstream profiles (mCODE, breast-radiology
IG, etc.) to point at and constrain. The cardinality of 0..* on each
extension also resolves the "combinatorial explosion" concern raised by
Kevin O'Donnell in
[FHIR-54904](https://jira.hl7.org/browse/FHIR-54904) without requiring
any base-spec change to `Observation`.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of two new IG-defined extensions, extend the existing IG
guidance to specify that **all** TID 300 / TID 1501 concept modifier
rows that don't have a typed Observation slot — currently `$ModType`
and `$Derivation` for measurements, plus the parallel modifier rows on
qualitative evaluations — are mapped to `Observation.component`
entries, where:

- `Observation.component.code` carries the DICOM concept name code
  identifying *which* modifier (e.g., `121401 "Derivation"`,
  `370129005 "Measurement Method"`-equivalents for `$ModType`), and
- `Observation.component.value[x]` (typically `valueCodeableConcept`)
  carries the post-coordinated term.

Add a new section to the IG narrative explicitly listing the supported
modifier-name codes and the rule for round-tripping them back to TID
300 rows. Bundle the resolution of
[FHIR-48643](https://jira.hl7.org/browse/FHIR-48643) into the same
change since it asks for exactly this generalized pattern. The change
again lands in `HL7/dicom-sr`.

#### Justification

This addresses both FHIR-47479 and FHIR-48643 with a single, uniform
mechanism, and it avoids minting per-row extensions for a long tail of
modifier rows that may emerge later (TID 1419, TID 1501 row 11/11b,
etc.). It uses only base FHIR machinery, so consumers don't need to
know about IG-specific extension URLs to read the data, and round-trip
to DICOM SR is straightforward because the modifier-name code is
preserved in `component.code` rather than encoded in an extension URL.
The trade-off is that information that *is* a "first-class" concept of
the measurement (such as derivation) becomes harder to query than if it
had a dedicated extension — but the IG already treats `Observation.
component` as the catch-all for non-primary measurement data, and
[FHIR-48172](https://jira.hl7.org/browse/FHIR-48172) (Devices) shows
that other WGs have landed in the same place after weighing component
vs. extension. If the WG worries about searchability, the IG can
publish search-parameter recipes alongside the mapping table.

---

### Disposition C: Decline

#### Proposal

Resolve as "Considered – No action required": do not add new IG
extensions or generalized component-mapping guidance. Instead, document
in the IG that implementations needing to convey `$ModType` or
`$Derivation` should either:

(a) wait for / use the R6 `Observation.interpretationContext`
    clarification from
    [FHIR-54904](https://jira.hl7.org/browse/FHIR-54904), which now
    explicitly covers "timing, view, or imaging mode" as contextual
    information relevant to an imaging measurement, or

(b) carry the post-coordinated terms in the existing
    `Observation.code` (the measurement concept name) as additional
    `coding[]` entries, since DICOM SR post-coordination is, by
    definition, refining the concept name.

Mark the issue resolved with no change to the IG StructureDefinitions.

#### Justification

Defensible *only* on the grounds that the underlying need is partially
addressed by the R6 base-spec clarification of
`Observation.interpretationContext` and that the IG should avoid
publishing extensions whose long-term home may shift to base FHIR. The
weakness is that `$ModType` and `$Derivation` are post-coordination of
the **measurement concept**, not contextual information about
*interpretation* of the value, so option (a) is a stretch on
semantics; option (b) (stuffing into `Observation.code.coding[]`)
forces consumers to know how to recombine post-coordinated codings
without help from the IG, which is exactly what the reporter is asking
the IG to avoid.

---

### Recommendation

**Recommended disposition: B.**

Disposition B (generalized `Observation.component`-based mapping) is
the right answer because FHIR-47479 and the still-open
[FHIR-48643](https://jira.hl7.org/browse/FHIR-48643) are the same
question asked twice — narrowly for two specific TID 300 rows, and
broadly for "all TID 300 / TID 1501 concept modifiers." The WG should
resolve them together, and a per-row extension strategy (Disposition
A) doesn't scale: TID 300 has more modifier rows than `$ModType` and
`$Derivation`, and TID 1501 (qualitative evaluation) has its own. A
component-based pattern with a documented modifier-name code list
covers both tickets, requires no base-spec changes, and is consistent
with how the Devices WG resolved the analogous question in
[FHIR-48172](https://jira.hl7.org/browse/FHIR-48172). Disposition A is
a reasonable fallback if the WG strongly prefers typed extensions for
discoverability; Disposition C should be rejected because the R6
`interpretationContext` clarification, while helpful, is about
contextualizing the *value* and does not solve post-coordination of
the *concept name* that this ticket is about.

`cache/github/repos/HL7_dicom-sr/repo-analysis/briefing.md` (currently
