## Summary

The reporter notes that the example narrative on the `ImagingSelection`
resource page illustrates a "volume rendered view" use case where
`ImagingSelection.code` is populated with RadLex `RID35862` (Volume
Rendering) and `ImagingSelection.category` is populated with DICOM
Image Type values such as `DERIVED` and `SECONDARY`. Neither of those
codes is drawn from the value sets that are bound (preferred) to
`code` and `category` on the resource. The reporter further observes
that the preferred value sets bound to `code` and `category` are
currently identical, which makes the distinction between the two
elements unclear to readers. They request that the example be revised
to use values that are actually drawn from the bound value sets, and
that the spec text explain why the two bindings share a common value
set (or differentiate them if they should not).

The ticket has already been triaged with a resolution of **Persuasive
with Modification**. The agreed change is to (a) update the example
`code` to DCM `121347` "Volume corresponding to spatially-related
acquisition frames", (b) update the example `category` to DCM `128196`
"For Segmentation", and (c) update the surrounding narrative to match
— noting that `code` and `category` will then be drawn from different
DICOM concepts even though both bindings reference DCM-derived value
sets.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the cross-reference query, and a Zulip search for the
ticket key returned no hits. The disposition below is therefore based
solely on the ticket itself.

## Details

> "*For example, if an ImagingSelection is used to identify a set of

> I suggest using example values from the actual preffered value sets
> with an explanation why the values sets are the same.

**Resolution Description (already recorded on the ticket):**

- Update `code` to use DCM#121347 "Volume corresponding to
  spatially-related acquisition frames".
- Update `category` to use DCM#128196 "For Segmentation".
- Update accompanying text to match.
- Note that the example code systems for `code` and `category` are
  no longer the same.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already recorded on the ticket verbatim, in the
`ImagingSelection` resource page of FHIR Core:

- Replace the example `code` value (currently illustrated as RadLex
  `RID35862` "Volume Rendering") with DCM `121347` "Volume
  corresponding to spatially-related acquisition frames" — a value
  drawn from a DCM-based value set appropriate to
  `ImagingSelection.code`.
- Replace the example `category` value (currently illustrated as
  DICOM Image Type `DERIVED` / `SECONDARY`) with DCM `128196` "For
  Segmentation" — a value drawn from a DCM-based value set
  appropriate to `ImagingSelection.category`.
- Update the surrounding narrative on the `ImagingSelection` page to
  reflect the new example codes and to explicitly note that, although
  both bindings are preferred and currently share overlap, the
  example now uses distinct DCM concepts that are individually
  appropriate to each element's intent (one identifying *what* the
  selection represents, the other classifying the *kind* of
  selection).
- Leave the bindings themselves and their strength unchanged for this
  ticket.

#### Justification

The reporter's underlying complaint is real: an example that uses
codes outside the bound value sets is misleading and undermines the
educational purpose of the narrative. The triaged resolution already
identifies concrete DCM codes that fall inside the bound value sets
and that meaningfully distinguish `code` from `category`, which
directly answers both halves of the reporter's request (in-binding
example values **and** clarification of the relationship between the
two bindings). This is a low-risk, narrative-only change that does
not affect conformance, instance validation, or implementer behavior.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to fixing the example as in Disposition A, also revisit
the value set bindings on `ImagingSelection.code` and
`ImagingSelection.category`:

- If the two bindings are intentionally identical because the same
  DCM concept set is reusable in both roles, add a non-normative
  paragraph on the resource page (and a comment on each element
  definition) explaining the rationale and giving guidance on which
  subset of concepts is appropriate for each element.
- If, on review, the two bindings should differ, narrow each to a
  context-appropriate subset (e.g., a "selection purpose" value set
  for `code` and a "selection classification" value set for
  `category`), keeping both at preferred strength to avoid breaking
  existing implementations.

#### Justification

The reporter raised two concerns: misleading example values *and* the
fact that `code` and `category` currently share a value set. Disposition
A resolves the first; Disposition B additionally resolves the second
by either documenting the intent or refining the bindings. This is a
more thorough fix and prevents the same confusion from recurring on
the next reader. The trade-off is scope: changing bindings (even at
preferred strength) is a more substantive edit than a narrative tweak
and warrants Imaging Integration discussion before being committed,
which is why it is offered as an alternative rather than the primary
path.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** on the basis that example narrative is
illustrative only and is not required to be drawn from the bound
value sets. Leave the existing example (RadLex `RID35862` for `code`
and DICOM Image Type values for `category`) and the existing bindings
unchanged.

#### Justification

This would only be defensible if the workgroup judged that examples
intentionally range outside the bound value sets to demonstrate the
breadth of acceptable codes under a `preferred` binding strength.
Given that the ticket has already been triaged with resolution
**Persuasive with Modification** and that the workgroup has agreed on
specific replacement codes, declining at this point would contradict
the workgroup's own disposition and is included here only for
completeness.

---

### Recommendation

**Recommended disposition:** A

The ticket is narrowly scoped, the workgroup has already agreed on
specific in-binding DCM codes (`DCM#121347` for `code` and
`DCM#128196` for `category`), and the change is confined to the
`ImagingSelection` example narrative. Disposition A executes that
agreement with no surface area beyond the example and its
surrounding text, which makes it the right starting point for ballot
disposition. Disposition B is worth holding in reserve if, during the
edit, the workgroup decides the shared-binding question itself
deserves a documentation or binding-narrowing follow-up — but that
should be a separate ticket rather than expanding the scope of
FHIR-54827. Disposition C is not recommended because it would
contradict the resolution already recorded on the ticket.
