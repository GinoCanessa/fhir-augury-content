## Summary

Jeroen Medema reports an error in the FHIR Core specification's **DICOM Tag
Mapping** table for the ImagingStudy resource (section 10.4.9.5). The row for
`numberOfInstances` currently labels the source of the value as **"DICOM
instance metadata"**, which is incorrect: the mapping should reference
**"DICOM study metadata"**. This is a localized, factual correction to a
mapping table entry and does not change the structure or semantics of the
ImagingStudy resource itself. No related Jira tickets, Zulip threads, or
GitHub issues/PRs were found via cross-reference or text search for the
ticket key, so this appears to be an isolated, narrow fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir`, edit the ImagingStudy DICOM Tag Mapping table (rendered
as spec section 10.4.9.5, authored under `source/imagingstudy/` — most
likely `source/imagingstudy/imagingstudy-notes.html`) so that the row
for `numberOfInstances` references **"DICOM study metadata"** instead
of **"DICOM instance metadata"**. Make no other content changes; in
particular do not modify the row's DICOM tag column, the resource
element name, or the structure of the table. Verify that the same
"instance metadata" wording is not duplicated in any sibling rows that
should remain unchanged.

#### Justification

`ImagingStudy.numberOfInstances` is defined in FHIR as the **total
number of SOP Instances in the study** — a study-level aggregate
property, not a per-instance attribute. In the underlying DICOM model
this count is exposed at the study level (e.g., as part of study
metadata such as the Study IOD's instance summary), not as part of
per-instance metadata. The ticket's correction therefore aligns the
mapping documentation with both the FHIR element's semantics and DICOM's
own data-organisation level. The change is purely textual, has no
schema or tooling impact, is local to a single authored HTML page, and
matches the briefing's standard "Update narrative / prose" recipe for
HL7/fhir.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the correction in Disposition A, **and** in the same edit
audit the rest of the DICOM Tag Mapping table for the ImagingStudy
resource for the same class of error — i.e., review every row whose
mapping currently says "DICOM instance metadata" or "DICOM series
metadata" and confirm the level (study / series / instance) actually
matches the underlying DICOM source for that ImagingStudy element.
Where additional mismatches are found, fix them in the same change set
and note them in the commit message; where the existing wording is
correct, leave it untouched. Keep the scope strictly to mapping-level
labels (study vs. series vs. instance) — do not retitle the table,
restructure rows, or change the linked DICOM tag values.

#### Justification

The reporter has identified one row but the underlying class of mistake
(citing the wrong metadata level) can plausibly affect other rows in
the same table, since the table groups together elements that come
from different DICOM levels (study, series, instance). Fixing the
single reported row leaves any sibling errors in place to be filed as
separate tickets later, with the same reviewer overhead repeated each
time. Bundling a tightly-scoped audit of metadata-level labels in the
same change keeps the table internally consistent without expanding
the change beyond textual mapping corrections, and remains compatible
with the briefing's "Update narrative / prose" recipe (no
StructureDefinition edits, no cross-repo impact). The risk is small
and bounded: the only files touched are still in
`source/imagingstudy/`.

---

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope / no change**, on the basis that
the existing wording can be defended as referring to "metadata about
the instances belonging to the study" rather than per-instance
metadata, and that readers can already disambiguate via the linked
DICOM standard.

#### Justification

This is the weakest of the three options and is included only for
completeness. The wording "DICOM instance metadata" is not naturally
read as "study-level summary information about the study's instances";
in DICOM parlance "instance metadata" is well-established as
per-instance attributes. Declining would leave a documented inaccuracy
in a normative-track core resource's mapping table, which the Imaging
Integration WG has historically been unwilling to do for verifiable
DICOM-conformance issues, especially under an `R6-DICOMTagMapping`
label that signals an intentional pass over these mappings for R6.

---

### Recommendation

**Recommended disposition:** A

This is an unambiguous factual correction to a single cell in a single
mapping table. The ticket reporter has cited the relevant DICOM
authority (`dicomstandard.org/current`); `ImagingStudy.numberOfInstances`
is by definition a study-level count; and the change has no schema,
tooling, or downstream-IG impact (HL7/fhir authoring root only, no
cross-repo touch points). Disposition B is attractive in principle but
risks expanding a narrowly-scoped errata fix into a broader audit that
the reporter did not ask for and for which no other rows have been
flagged; it can be revisited later as a follow-up if the WG sees value
in a full sweep. Disposition C is not defensible on the merits. Accept
as requested, land the one-line text change in `HL7/fhir` under
`source/imagingstudy/`, and close the ticket.
