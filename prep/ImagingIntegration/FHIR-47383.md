## Summary

The reporter notes that the *DICOM SR to FHIR Resource Mapping* Implementation
Guide does not contain a mapping section for **Device Algorithm Identification**
(the DICOM SR concept used to identify the algorithm — typically embodied in a
Device or its software — that produced or processed an SR observation). The
ticket asks the Imaging Integration Work Group to clarify whether that section
is simply missing from the IG and needs to be authored, or whether the IG
intentionally omits it because the concept does not require a dedicated
mapping. No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FhirAugury for this ticket, so the request stands on its own and
needs a substantive WG decision rather than a roll-up of prior discussion.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new mapping section to the *DICOM SR to FHIR Resource Mapping IG* titled
**"Device Algorithm Identification"** (placed alongside the other "Device-"
sections, so it is discoverable from the IG's mapping-sections index). The
section maps the DICOM Algorithm Identification Macro / Device Algorithm
Identification context group concepts to FHIR `Device`, including:

- DICOM Algorithm Name → `Device.deviceName` (with `type = manufacturer-name`
  or `model-name` as appropriate) and/or `Device.type` (CodeableConcept) when a
  coded algorithm type is present.
- DICOM Algorithm Version → `Device.version.value`.
- DICOM Algorithm Source / Manufacturer → `Device.manufacturer`.
- DICOM Algorithm Parameters → `Device.property` (or, if structured parameters
  are out of scope for this IG cycle, an explicit "out of scope" note pointing
  to a future ticket).
- Linkage from the SR-derived `Observation` to this `Device` via
  `Observation.device` (and, where the algorithm acted on prior data,
  `Observation.derivedFrom` to the source `Observation`/`ImagingStudy`).

Include a worked example (an SR with an algorithm-produced measurement) and
add the new section to the IG's table of contents and mapping-coverage matrix.

#### Justification

The reporter explicitly asked whether the section is missing, and the WG
already escalated the ticket to a Change Request — both signals that the IG is
expected to grow this content rather than disclaim it. DICOM SR routinely
carries Algorithm Identification (e.g., AI-derived measurements, CAD findings),
and downstream FHIR consumers need a deterministic place to look for the
algorithm provenance so that two implementers do not invent different bindings.
`Device` is the FHIR resource the spec already uses for software/algorithm
identity, so the mapping is consistent with existing IG patterns and with the
broader FHIR convention of attaching the originating algorithm via
`Observation.device`.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introduce a standalone "Device Algorithm Identification" section,
fold algorithm identification into the **existing Device / equipment mapping
section** as a clearly-labelled subsection ("Device — Algorithm Identification
variant") and capture algorithm-specific provenance via **`Provenance`** rather
than only via `Device`. Concretely:

- Document that the algorithm is represented as a `Device` instance whose
  `Device.type` distinguishes it from physical equipment (e.g., a code such as
  "software algorithm").
- Map the SR Algorithm Name/Version/Source onto the same `Device` element set
  as Disposition A (`deviceName`, `version`, `manufacturer`).
- Recommend that derived `Observation`s point to that `Device` *and* that a
  `Provenance` resource record `Provenance.agent.who = Device` plus
  `Provenance.entity` referencing the source images/SR, so that the
  "algorithm-produced" relationship is explicit and queryable rather than
  inferred from `Observation.device` alone.
- Add a short "When to use a separate Device vs. Provenance" guidance box.

#### Justification

This keeps the IG's section count flat (the WG has expressed a preference in
other Imaging tickets for consolidating Device-related guidance rather than
fragmenting it), and it leverages `Provenance` — a resource specifically
designed for "who/what produced this and from what" — instead of overloading
`Observation.device`. It is more faithful to FHIR's separation of concerns:
`Device` answers "what software was it", `Provenance` answers "what did it
do, to which inputs, when". The trade-off vs. Disposition A is discoverability
(a reader hunting for "Algorithm Identification" in the TOC won't find a
top-level entry); that is mitigated by adding an index/anchor entry that
points into the Device section.

---

### Disposition C: Decline

#### Proposal

Decline to add a dedicated Device Algorithm Identification mapping section.
Resolve the ticket as **"Persuasive with Modifier — No Change"**: respond to
the reporter that the IG intentionally does not enumerate every DICOM
Algorithm Identification element because it is already covered transitively by
the existing Device mapping (any algorithm is a `Device` whose
name/version/manufacturer are mapped exactly the same way as physical
equipment), and add a one-line clarifying note to the Device mapping section
saying so.

#### Justification

If the WG considers the existing Device mapping sufficient, then a separate
section duplicates content and creates a maintenance burden whenever the
Device mapping changes. Declining is defensible *only if* the WG can point at
the existing Device section and confirm that an implementer reading just that
section would correctly map all of: Algorithm Name, Algorithm Version,
Algorithm Source/Manufacturer, and the Observation→Device linkage. The
downside is that the reporter (and likely other implementers) already could
not find the mapping; declining without at least a clarifying note effectively
re-creates the original confusion that prompted the ticket.

---

### Recommendation

**Recommended disposition:** A

The ticket was deliberately re-classified from a clarification into a
substantive Change Request by the WG itself (jwhitby, 2024-09-21), which
strongly suggests the WG already concluded that *some* IG edit is warranted —
ruling out Disposition C. Between A and B, Disposition A best serves
implementers: Device Algorithm Identification is a first-class, frequently-used
DICOM SR concept (especially as AI/CAD-derived SRs proliferate), and a named,
top-level section makes the mapping discoverable from the IG's TOC and
coverage matrix without forcing readers to know that "algorithm" lives inside
the generic Device section. Disposition B's `Provenance` guidance is valuable
and should be **incorporated as a subsection within Disposition A** rather
than chosen instead of it — that captures both the discoverability win and
the cleaner separation of "what the algorithm is" (Device) from "what it did"
(Provenance). Recommend the WG accept A with the Provenance guidance from B
folded in.
