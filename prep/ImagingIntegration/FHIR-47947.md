## Summary

This ticket asks the Imaging Integration WG to flesh out the
**DerivedMeasurements** mapping in the *DICOM SR to FHIR Resource Mapping IG*.
Today the IG defines only a single Observation profile for DerivedMeasurements,
even though the underlying DICOM SR structure is parallel to that of
**ImagingMeasurements**. The reporter (Bas van den Heuvel) argues that
DerivedMeasurements should generate the same family of FHIR artifacts that
ImagingMeasurements does — i.e., a *group* Observation that anchors the set,
one *derived* Observation per measurement, and matching **ImagingSelection**
resources to point back to the source pixels/coordinates. As filed, the ticket
has no explicitly linked Jira tickets and no GitHub or Confluence
cross-references in Augury; the only adjacent community signal is a tangential
Zulip thread in `#implementers > DICOM SR Mappings` about the broader scope of
the IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the *DICOM SR to FHIR Resource Mapping IG* so that the
**DerivedMeasurements** mapping produces the same artifact set as
**ImagingMeasurements**, namely:

1. A **group Observation** profile for the DerivedMeasurements container,
   carrying `Observation.code`, `subject`, `effective[x]`, `derivedFrom` (to
   the source ImagingMeasurements group), and `hasMember` linking the
   per-measurement Observations.
2. One **derived measurement Observation** profile per numeric/coded
   measurement value, with `Observation.value[x]`, `code`, `derivedFrom` to
   both the parent group and any source measurements, and `bodySite` /
   `method` populated from the SR concept-name/value pairs.
3. An **ImagingSelection** profile per derived measurement that references
   the same DICOM image / frame / coordinate region used by the corresponding
   ImagingMeasurement, so derived values remain traceable to the source
   pixels.
4. Add narrative + mapping tables to the IG's "Derived Measurements" page
   explaining the parallel structure, and add example bundles that round-trip
   a DerivedMeasurements SR fragment through the new profiles.

#### Justification

The IG already defines this exact pattern for ImagingMeasurements, so
extending the same shape to DerivedMeasurements keeps the mapping internally
consistent and lets implementers reuse their existing transformation logic.
The reporter's framing (that the two SR structures "overlap") matches how
DICOM itself models derived measurements — they are produced from the same
measurement-group machinery — so the FHIR projection should mirror that.
Preserving an ImagingSelection per derived value is also the only way to
keep the chain from *image → measurement → derived measurement* navigable on
the FHIR side, which is the IG's stated goal of supporting downstream
consumers such as the IHE AI Results profile (per Jonathan Whitby's comment
in the related Zulip thread).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a full parallel set of profiles, **refactor the
ImagingMeasurements profiles so they are reusable for derived measurements**
and reference them from the DerivedMeasurements mapping. Concretely:

- Promote the existing group / measurement / ImagingSelection profiles from
  `ImagingMeasurements*` to a more neutral name (e.g., `MeasurementGroup`,
  `Measurement`, `MeasurementImagingSelection`) or keep the names but mark
  them as the canonical profiles for *any* TID-1500-derived measurement set.
- Define DerivedMeasurements as a **specialization** that adds only the
  delta: a constraint on `Observation.derivedFrom` requiring a link to the
  source measurement group, and any additional `code` bindings specific to
  derived measurements.
- Document the relationship in the IG so implementers see DerivedMeasurements
  as "ImagingMeasurements + a derivation link," not as a separate parallel
  artifact family.

#### Justification

This addresses the same underlying need — full structural coverage of
DerivedMeasurements — while avoiding profile duplication. Two near-identical
profile families would force the IG (and downstream implementers) to maintain
parallel mapping tables, examples, and conformance tests every time the SR
template evolves, which is exactly the kind of drift the reporter is already
flagging. A specialization approach also makes the *derivation* relationship
first-class in the model rather than implicit in the artifact name, which is
useful for AI/post-processing pipelines that need to distinguish primary from
derived findings. The trade-off vs. Disposition A is a small amount of
profile-renaming churn against substantially less ongoing maintenance.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the single-Observation mapping for
DerivedMeasurements, on the grounds that the existing mapping is sufficient
for the current scope of the IG (the specific SR templates the WG has
committed to) and that any consumer needing the full
group/measurement/ImagingSelection structure can already reuse the
ImagingMeasurements profiles directly without a spec change.

#### Justification

This is the weakest of the three options because it leaves a known
inconsistency in the IG documented by the reporter. The only defensible
ground for declining is scope: if the WG concludes that DerivedMeasurements
in the templates the IG actually covers never carry more than a single
scalar value, then a richer projection would be over-engineering. Given the
reporter's framing and Jonathan Whitby's Zulip statement that the WG is open
to extending the IG, that scope argument is hard to sustain.

---

### Recommendation

**Recommended disposition:** B

Disposition B delivers exactly what the reporter is asking for — a
DerivedMeasurements mapping that produces the full group + per-measurement +
ImagingSelection structure — while avoiding the profile-duplication cost that
Disposition A would lock in. Because the IG is still relatively young and is
explicitly tied to specific SR templates (per the assignee's Zulip remark),
this is the right moment to factor the shared measurement structure out
once, rather than ship a parallel `DerivedMeasurements*` family that the WG
would then have to keep in lock-step with the `ImagingMeasurements*` family
forever. If the WG judges that renaming the existing profiles is too
disruptive at this stage of the IG's lifecycle, falling back to Disposition
A is acceptable; Disposition C should only be selected if the WG decides
that the SR templates currently in scope simply do not produce derived
measurement *groups* in practice.
