## Summary

The reporter (Bas van den Heuvel) notes that FHIRcast (and the underlying
ImagingStudy context it references) binds several elements to DICOM-defined
ValueSets via URLs that point to DICOM's HTML documentation pages at
`dicom.nema.org`, rather than to machine-readable, canonical ValueSet
resources. The actual ValueSet JSON can only be reached by following a
link on the HTML page to an FTP endpoint
(`ftp://medical.nema.org/medical/dicom/resources/valuesets/fhir/json/...`).
As a result, the FHIR publisher and validator cannot retrieve these
ValueSets to validate resources, and ImagingStudy-derived instances used by
FHIRcast emit warnings. The ticket asks that these bindings be updated to
use ValueSet canonical URLs that the FHIR tooling can resolve (and
preferably versioned references, e.g., `...|6.0.0-cibuild`). No
cross-referenced Jira tickets were returned; the only cross-references are
Zulip GitHub-notification messages announcing that `bvdh` created and
pushed branch `FHIR-42817` on `HL7/fhircast-docs`, indicating work was
started on a fix in that repo.

## Details

**Description:**

The fields in the resource that refer to the DICOM-defined valuesets point
to the HTML end-point, e.g.:

> `"valueSet" : "http://dicom.nema.org/medical/dicom/current/output/chtml/part16/sect_CID_244.html"`

The ValueSet can only be retrieved by following a link on that HTML page
to an FTP endpoint such as
`ftp://medical.nema.org/medical/dicom/resources/valuesets/fhir/json/ValueSet-dicom-cid-244-Laterality.json`.
This makes it very hard/impossible for the FHIR publisher/validator to
retrieve the ValueSet and check its contents in order to validate
resources. It also causes warnings in ImagingStudy instances used by
FHIRcast.

The expected behaviour of the endpoint provided in a binding is that it
allows retrieval of the ValueSet directly; the requested `Content-Type`
defines the type of results. It is also common practice to use versioned
references (e.g. `http://hl7.org/fhir/ValueSet/all-languages|6.0.0-cibuild`)
to clearly indicate the version of the ValueSet referred to. The reporter
asks that these references be updated to point to versioned ValueSet
definitions so the validator works properly.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update every DICOM-backed ValueSet binding in the FHIRcast IG source so
that `valueSet` points to a canonical, resolvable, versioned ValueSet URL
rather than a DICOM HTML page. Concretely, in `HL7/fhircast-docs`:

1. Inventory every binding whose `valueSet` currently points at
   `http://dicom.nema.org/medical/dicom/current/output/chtml/part16/...`.
2. Replace each such URL with the corresponding DICOM-published canonical
   ValueSet URL (e.g.,
   `http://dicom.nema.org/resources/ontology/DCM` /
   `http://dicom.nema.org/.../ValueSet/dicom-cid-244-Laterality`) pinned
   to a specific version string (e.g. `|2024.1.20240120`), matching the
   ValueSets shipped in the `fhir.dicom` package.
3. Add the `fhir.dicom` package as an IG dependency in the FHIRcast
   publication config (per `john_moehrke`'s comment):
   ```yaml
   dependencies:
     fhir.dicom:
       uri: http://fhir.org/packages/fhir.dicom
       version: 2024.1.20240120
   ```
4. Rebuild the IG and confirm the publisher/validator resolves the
   ValueSets and the ImagingStudy-derived warnings clear.

#### Justification

This is exactly what the reporter and `bvdh` have asked for, and it
matches the mechanism John Moehrke describes for consuming DICOM codes in
HL7 IGs. DICOM WG-06 has confirmed (per `jwhitby`, 2024-01-30) that they
will not fix the upstream URLs, so the HL7 side must adapt. Using the
`fhir.dicom` package also gives implementers and tooling a versioned,
resolvable, machine-readable source — consistent with FHIR's general
expectation that `ValueSet.url` is dereferenceable to the ValueSet
resource, not an HTML page. A working branch (`FHIR-42817` on
`HL7/fhircast-docs`) already exists as a starting point.

---

### Disposition B: Alternative Approach

#### Proposal

Two-phase fix. **Phase 1 (near-term, low-risk):** replace the
`ftp://medical.nema.org/...` and HTML-page URLs with their HTTP
equivalents on `dicom.nema.org` without introducing the package
dependency — this matches the WGM 30 Jan 2024 consensus (`jwhitby`) that
"changing FTP resources to HTTP alone would be an improvement". **Phase 2
(next FHIRcast release):** migrate to the `fhir.dicom` package with
versioned canonical references as in Disposition A, after validating that
the package covers every code system / ValueSet FHIRcast binds to and
coordinating version bumps with the broader Imaging Integration IG set.

#### Justification

Phase 1 is a quick win that removes the FTP-only / HTML-only blockers and
can ship without waiting for coordination on package versioning. Phase 2
delivers the full benefit the reporter is after (validator resolution,
versioned bindings) but gives the workgroup time to (a) verify that
`fhir.dicom` contains every ValueSet FHIRcast references at a suitable
version, (b) align on a version string shared with other imaging IGs so
downstream implementers aren't forced to juggle conflicting DICOM
package versions, and (c) regression-test the publisher build. It also
preserves reviewer attention on the substantive change (package
dependency) rather than bundling it with a trivial URL-scheme swap.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the ValueSet URL form is dictated by DICOM
upstream (which has declined to change it — see `jwhitby`, 2024-01-30),
and that any real fix requires coordination across the full set of HL7
imaging IGs that bind DICOM ValueSets (ImagingStudy, FHIRcast,
ImagingSelection, etc.), not just a patch to FHIRcast. Close this ticket
with a pointer to a new umbrella tracker covering all imaging IGs.

#### Justification

Fixing only FHIRcast while leaving the same broken HTML/FTP bindings in
ImagingStudy and related IGs would create an inconsistent developer
experience and leave the underlying validator warnings elsewhere. If the
workgroup is not yet ready to commit to the `fhir.dicom` package as the
canonical dependency for all its IGs, a point fix here will have to be
redone.

---

### Recommendation

**Recommended disposition:** A

The reporter's request is straightforward, well-scoped to FHIRcast, and
the mechanism to satisfy it already exists: Grahame Grieve's
`fhir.dicom` package is the standard way HL7 IGs consume DICOM codes and
ValueSets, and John Moehrke explicitly endorsed it on this ticket. DICOM
WG-06 has declined to fix the upstream URLs, so the change must happen on
the HL7 side, and `bvdh` has already started a branch. Disposition B's
"HTTP-first, package later" two-phase plan is attractive but in practice
doubles the editing work on the same bindings and delays the actual
benefit (validator-resolvable, versioned ValueSets) that the ticket is
asking for; the Phase 1 HTTP-only change should only be the fallback if
`fhir.dicom` turns out to be missing coverage for a specific ValueSet.
Disposition C is not warranted: the ticket is legitimate, there is a
known fix, and declining would simply defer the same work to a future
umbrella ticket without reducing its scope. The workgroup should accept
the change, adopt the `fhir.dicom` package dependency in the FHIRcast IG
publication config, and use the existing `FHIR-42817` branch on
`HL7/fhircast-docs` as the starting point — while verifying in review
that the same approach is then applied consistently to the other imaging
IGs that share these bindings.
