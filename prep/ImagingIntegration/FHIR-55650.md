## Summary

Eric Haas (US Core / profiling perspective) raises a concern that the
extensible bindings on `ImagingStudy.modality` and
`ImagingStudy.series.modality` point at code systems (the DICOM CID 33
"Modality" terminology) that may not be in use by every jurisdiction or
implementation community. He recommends weakening the binding strength
from **extensible** to **preferred** so that implementers in
communities that do not use the DICOM modality code system are not
trapped by the extensible-binding rules (which require a code from the
specified value set unless no suitable code exists).

No other Jira tickets cross-reference FHIR-55650, and no GitHub items
or repo cross-references were found. Two Zulip threads on the
`imaging` and `terminology` streams discuss the same underlying
binding/code-system question and provide useful background but were not
formally linked to this ticket. The Imaging Integration WG has already
triaged the ticket and applied a "Not Persuasive" resolution on the
basis that no viable alternative code system to the DICOM modality
terminology has been identified.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the binding strength on `ImagingStudy.modality` and
`ImagingStudy.series.modality` from **extensible** to **preferred**,
keeping the existing DICOM CID 33 ("Modality") value set as the
preferred value set. No change to the value set itself, the
recommended code system URI
(`http://dicom.nema.org/resources/ontology/DCM`), or any examples.
This would relax the conformance requirement so that implementers
whose jurisdiction or community does not use DICOM CID 33 are free to
use a locally appropriate code system without violating the binding.

#### Justification

This is the literal change Eric Haas requested. A preferred binding
still signals that DICOM CID 33 is the recommended terminology and
preserves interoperability for the majority of implementers, while
removing the "must use a code from this value set unless none fits"
obligation that extensible bindings impose. It mirrors a pattern US
Core and other realm profiles have used on similar bindings where
local terminologies are common.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the **extensible** binding on
`ImagingStudy.modality` / `ImagingStudy.series.modality` as-is, but
(1) clarify in the element's `Comments` and the
`http://hl7.org/fhir/ValueSet/...` value-set narrative that the bound
DICOM modality terminology is the expected choice and the canonical
`Coding.system` is `http://dicom.nema.org/resources/ontology/DCM`, and
(2) explicitly note that jurisdictions or implementation communities
that need to use a different terminology may add additional `Coding`
entries to the modality `CodeableConcept` (since the type is
`CodeableConcept`, multiple `Coding`s are already permitted) — the
extensible binding only constrains that *one* of the codings come from
the bound value set when a suitable code exists.

#### Justification

This addresses the underlying concern (jurisdictional flexibility)
without weakening the binding or losing the strong signal that DICOM
CID 33 is the standard modality vocabulary in imaging interoperability.
The Imaging Integration WG comment on the ticket (jwhitby, 2026-03-18)
confirms the WG's view that DICOM is the only code system that can
fully represent the source data; weakening the binding strength risks
fragmenting modality coding across implementations. The
`CodeableConcept`-based escape valve is also consistent with the
guidance the Zulip thread "Coding.system for ImagingStudy.modality"
implicitly assumes.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** (already the resolution recorded on
the ticket) and leave the existing extensible binding to the DICOM
CID 33 modality value set unchanged on both
`ImagingStudy.modality` and `ImagingStudy.series.modality`.

#### Justification

The DICOM modality terminology is the de facto vocabulary used by
imaging modality vendors and PACS/VNA systems worldwide; the WG is not
aware of an alternative code system that can fully represent imaging
modality, and the reporter himself acknowledged he is not a domain
expert and could be convinced no competing terminology exists.
Weakening the binding to preferred would create a real risk of
implementations using ad-hoc local codes for modality, undermining
cross-system interoperability of imaging studies — precisely the
property `ImagingStudy` exists to support. Where a community truly
needs a local code, the `CodeableConcept` type already permits
additional `Coding` entries alongside the DICOM-bound code.

---

### Recommendation

**Recommended disposition:** C

The Imaging Integration WG has already triaged this ticket as "Not
Persuasive" and the substantive comment on the ticket from jwhitby
captures the right reason: imaging modality is one of the few areas
where a single international code system (DICOM CID 33) genuinely
dominates and is what downstream consumers of `ImagingStudy` expect to
see. Loosening the binding to preferred (Disposition A) would solve a
hypothetical community-fit problem at the cost of a real
interoperability problem — implementers could legitimately ship
modality codings from arbitrary code systems and remain conformant.
Disposition B is a reasonable compromise the WG could fall back to if
pushed, but it is mostly a documentation clarification of behaviour
that the existing `CodeableConcept` cardinality already allows;
nothing in the ticket or the related Zulip threads suggests
implementers are actually being blocked, only that the rule is felt to
be strict. Decline, with a note pointing the reporter at the
`CodeableConcept`-multiple-Coding escape valve, is the cleanest
outcome and aligns with the resolution already recorded.
