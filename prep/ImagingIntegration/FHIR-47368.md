## Summary

The reporter flags a confusing sentence in the DICOM SR to FHIR Resource
Mapping IG: *"All algorithm devices will populate use the Equipment Device
to populate the parent element."* The sentence is grammatically broken
("will populate use") and ambiguous about which `Device` plays which role
in the parent/child relationship. The reporter asks the work group to
clarify the intended meaning so implementers can correctly populate the
parent `Device` reference for algorithm devices derived from a DICOM
Equipment module. There are no cross-referenced Jira tickets, Zulip
threads, or GitHub items associated with this ticket in the FhirAugury
index.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the offending sentence in the DICOM SR to FHIR Resource Mapping IG
narrative (Device mapping section) to read, in substance:

> "Every algorithm `Device` instance SHALL set `Device.parent` to the
> `Device` instance representing the DICOM Equipment that produced the
> algorithm result."

Concretely, replace the broken sentence "All algorithm devices will
populate use the Equipment Device to populate the parent element." with a
single grammatical sentence that (a) names the resource (`Device`),
(b) names the element (`Device.parent`), (c) identifies the referenced
target (the `Device` derived from the DICOM Equipment module), and
(d) uses a single, clear conformance verb (SHALL or "is populated with").

#### Justification

The ticket is purely editorial — the underlying intent (algorithm devices
point to the equipment device via `Device.parent`) is already the
documented model in the IG; only the prose is broken. Fixing the wording
in place is the lowest-risk action, requires no normative change, and
removes a real source of implementer confusion. Using FHIR's standard
conformance vocabulary and explicit element paths brings the sentence in
line with the rest of the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Replace the single sentence with a small, structured block in the Device
mapping section:

- One sentence stating the rule in plain English ("Algorithm devices
  reference the equipment that produced them via `Device.parent`.").
- A short bullet list mapping the DICOM source to the FHIR target:
  - DICOM Equipment module → `Device` (kind = equipment)
  - Algorithm identification → `Device` (kind = algorithm), with
    `Device.parent` → the equipment `Device` above
- An inline example (or link to an existing example in the IG) showing
  the two `Device` instances and the populated `parent` reference.

#### Justification

The ambiguity in the original sentence suggests the prose alone is not
carrying its weight; readers have to reverse-engineer which `Device` is
parent and which is child. A short structured block plus an example makes
the relationship unambiguous on first read and is consistent with how
other DICOM-to-FHIR mappings in the IG are documented. The cost is
slightly more authoring effort than a one-line fix, but it eliminates the
class of confusion this ticket reports rather than just this instance.

---

### Disposition C: Decline

#### Proposal

Decline as "not persuasive" / "no change" on the grounds that the
surrounding section already conveys the parent/child Device relationship
and the sentence, while awkward, can be inferred in context.

#### Justification

A defensible-but-weak position: the IG is a Standard for Trial Use and
editorial-only changes consume work-group cycles. If reviewers conclude
the broader section makes the relationship clear, declining preserves the
published text. This rationale is weak because the reporter is an
implementer who explicitly could not parse the sentence, which is direct
evidence the prose fails its job.

---

### Recommendation

**Recommended disposition:** A

This is a straightforward editorial clarification of a sentence that is
demonstrably unparseable as written ("will populate use the Equipment
Device to populate the parent element"). Disposition A is the smallest
change that fixes the bug-in-prose, names the resource and element
explicitly, and uses standard FHIR conformance language. Disposition B is
a reasonable upgrade if the work group wants to invest in a structured
mapping block, and the editor applying A could opt into B opportunistically
if the surrounding section is being touched. Disposition C should be
avoided: the reporter's inability to parse the sentence is itself the
evidence that the current text does not stand on its own.
