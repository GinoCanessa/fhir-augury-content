## Summary

The reporter (Jeroen Medema) observes that DICOM UIDs are encoded
inconsistently across the `ImagingStudy` resource: the **DICOM Study
Instance UID** is carried in `ImagingStudy.identifier` (an `Identifier`
datatype, requiring the `urn:dicom:uid` system and a `urn:oid:`-prefixed
value), whereas the **Series Instance UID** (`ImagingStudy.series.uid`)
and **SOP Instance UID** (`ImagingStudy.series.instance.uid`) are
carried as the `id` datatype directly on dedicated `uid` elements. The
ticket asks the workgroup to either (a) document a justification for
the divergent modeling on the `ImagingStudy.identifier` page, or (b)
make the encoding consistent across the three DICOM UID slots.

The ticket has already been triaged with a resolution of "Persuasive
with Modification" and an in-progress resolution proposal that adds a
new sentence ("DICOM UIDs are typically encoded in FHIR using the `id`
datatype.") plus updated text on the study explaining that the Study
Instance UID is encoded in `ImagingStudy.identifier` because it is a
primary identifier for the resource. In a follow-up comment Jeroen
notes that the proposed resolution text still does not actually
explain *why* `ImagingStudy.identifier` is not an `id` like the series
and instance UIDs, and asks for that justification. There are no
explicit Jira links and no GitHub cross-references; the only
cross-reference recorded is to the FHIR artifact
`ImagingStudy.identifier` itself.

## Related Zulip Discussions

### [implementers > Referencing DICOM instances contained in FHIR ImagingStudy](https://chat.fhir.org/#narrow/stream/implementers/topic/Referencing%20DICOM%20instances%20contained%20in%20FHIR%20ImagingStudy)

David Flade opened the topic on 2025-01-14 asking how, in FHIR R4, to
reference a specific DICOM SOP Instance from an `Observation` when the
`ImagingStudy` resource contains many instances. The discussion drifts
across (i) whether to model one `ImagingStudy` per SOP instance vs one
`ImagingStudy` per DICOM Study, (ii) how to fall back to R4 when
`ImagingSelection` (introduced in R5) is not yet available, and (iii)
how to use `ImagingStudy.identifier` together with
`Observation.derivedFrom.identifier` to carry Study and SOP Instance
UIDs side-by-side. John Moehrke recommends treating R4 `ImagingStudy`
as a stand-in for `ImagingSelection` and putting the SOP Instance UID
into `derivedFrom.identifier`; Kari Heinonen and others raise concerns
about mutability and the limited target types of
`Observation.derivedFrom`; Bas van den Heuvel and Gino Canessa point
to the cross-version-extension / "detail the use" guidance on
`hl7.org/fhir/versions.html#extensions`. There is no firm consensus on
*the* correct R4 pattern, but the thread implicitly relies on exactly
the modeling the present ticket questions: Study Instance UID lives in
`ImagingStudy.identifier`, while series/instance UIDs live in their
own typed fields — so it is useful supporting context for whether
"consistency" or "documented justification" is the better resolution.

Participants: David Flade, John Moehrke, Kari Heinonen, Bas van den
Heuvel, Elliot Silver, Forest, Gino Canessa (Jonathan Whitby and

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the in-progress "Persuasive with Modification" resolution and
**also fully address the reporter's follow-up comment** by adding the
missing rationale text. Concretely, on the `ImagingStudy` resource
page (and the `ImagingStudy.identifier` element notes):

1. Add the general guidance sentence: *"DICOM UIDs are typically
   encoded in FHIR using the `id` datatype."*
2. Replace the current `ImagingStudy.identifier` notes with text such
   as: *"Because the DICOM Study Instance UID is the primary business
   identifier for the imaging study resource — it is the value used to
   look up, share, and correlate the study across systems — it is
   carried in `ImagingStudy.identifier` (an `Identifier`) rather than
   in a dedicated `id`-typed `uid` element. This allows the standard
   FHIR identifier search/match machinery (system + value, optional
   `type`, `period`, `assigner`) to be applied to the Study Instance
   UID, and lets the same study be referenced from
   `Reference.identifier` on other resources. The `series.uid` and
   `series.instance.uid` elements, by contrast, are not primary
   identifiers of separately-instantiated FHIR resources and are
   therefore modeled as plain `id`-typed values inside the parent
   resource."*
3. Cross-link from `ImagingStudy.series.uid` and
   `ImagingStudy.series.instance.uid` to that note so the divergence
   is discoverable from either side.

#### Justification

This is the minimum change that closes the ticket on the reporter's
own terms — the reporter explicitly offered "give a good reason …
*and document that here too*, or make them consistent" as the choice,
and the workgroup has already chosen the "document the reason" path
via *Persuasive with Modification*. The reporter's open comment makes
clear that the current draft only restates *what* the model is, not
*why*; adding the rationale above (primary-identifier semantics +
search/`Reference.identifier` interoperability) addresses that gap
without any wire-format change. It is also consistent with how other
resources expose primary business identifiers via `Identifier` (e.g.
`Patient.identifier`, `Specimen.identifier`).

---

### Disposition B: Alternative Approach

#### Proposal

Make the encoding consistent across the three DICOM UID slots by
introducing a typed `ImagingStudy.uid` element (datatype `id`) that
carries the Study Instance UID directly, mirroring
`series.uid` / `series.instance.uid`. Keep `ImagingStudy.identifier`
as an *additional, optional* slot that may carry the same UID encoded
as `Identifier` (system `urn:dicom:uid`, value `urn:oid:<uid>`) so
that existing `Reference.identifier`-based lookups continue to work.
Mark the new `ImagingStudy.uid` as the canonical location of the
Study Instance UID in narrative and invariants; add an invariant
asserting that, when both are present, `ImagingStudy.uid` and the
`urn:dicom:uid` entry in `ImagingStudy.identifier` carry the same
underlying UID. Mark the change as a non-breaking additive change in
the next normative-eligible release; provide migration guidance for
implementers currently relying on `ImagingStudy.identifier`.

#### Justification

This directly satisfies the "or make them consistent" branch of the
reporter's request, removes a long-standing developer-experience
papercut (every imaging implementer has to learn that the Study UID
lives in a different shape than the Series/Instance UIDs), and aligns
`ImagingStudy` internally with itself. The trade-off vs Disposition A
is real: it is a structural change to a widely-implemented resource,
it adds redundancy between `uid` and `identifier`, and it forces every
downstream IG, profile, and validator to learn the new element. In
practice this also affects FHIR-shaped tooling that has special-cased
`urn:dicom:uid`/`urn:oid:` parsing on `ImagingStudy.identifier`, and
it would need coordinated guidance for ImagingSelection and the
DICOM-SR IG.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by existing mechanisms**:
keep `ImagingStudy.identifier` as the sole carrier of the Study
Instance UID with the documented `urn:dicom:uid` / `urn:oid:` pattern,
keep `series.uid` and `series.instance.uid` as `id`-typed fields, and
make no spec or narrative change beyond what is already published.
Close the ticket as Not Persuasive on the grounds that (a) the
existing text on the `ImagingStudy` page already states the modeling,
(b) the divergence is intentional because the Study UID is the
resource-level primary identifier that needs to participate in
identifier-based search and `Reference.identifier`, and (c) any
breaking-or-additive change would impose migration cost without a new
clinical or interoperability use case.

#### Justification

Defensible only on stability grounds: the current model ships in R4,
R4B and R5, has wide deployment, and the request is a documentation /
ergonomics complaint rather than a bug or interoperability defect. The
weakness — and the reason this is the least attractive option — is
that the reporter's open comment shows the current narrative does not
in fact justify the divergence, so "already addressed" is only true
for the *what*, not the *why*; outright declining leaves that gap in
place.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already landed on *Persuasive with Modification*,
and the reporter's only remaining objection is that the in-progress
resolution text restates the modeling without explaining the
rationale. Disposition A finishes that work cheaply, in narrative
only, with no wire-format impact and no risk to deployed
implementations or to downstream IGs (DICOM-SR, ImagingSelection
guidance, etc.). Disposition B genuinely solves the consistency
complaint but pays a structural-change cost that the ticket itself
does not justify — the reporter offered documentation as an
acceptable alternative, and the Zulip *implementers* discussion shows
the community is already navigating the existing model rather than
asking for a redesign. Disposition C should be avoided because it
ignores the explicit follow-up comment asking for the rationale to be
written down. Recommend proceeding with A and explicitly closing the
loop with Jeroen on the new rationale text before marking the ticket
Resolved.
