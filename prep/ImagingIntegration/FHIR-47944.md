## Summary

The ticket questions whether the `ImagingSelection` profiles defined in
the DICOM SR to FHIR Resource Mapping IG implicitly require any
referenced DICOM instances to belong to the **same study**. The reporter
(Bas van den Heuvel) observes that several `ImagingSelection`
definitions in the IG are used to point at other DICOM instances, and
notes that the current mappings appear to suggest the referenced
instances must share the parent study (i.e., the `studyUid` and
`seriesUid` are pinned). He asks the WG either to confirm that the
single-study constraint is correct or, if it is not, to remove
`studyUid` / `seriesUid` from the profiles and only populate them when
actually known.

The ticket was originally resolved as **Not Persuasive** on 2024-09-22
on the grounds that the WG could not locate where the constraining
mapping is specified (Jeremy Whitby asked Bas to point at the specific
mapping he was concerned about). Bas re-opened the discussion on
2025-08-14 with new information drawn from the underlying DICOM
template: per his reading of TID 1500, measurements can reference any
study present in TID 1600 (Image Library) via the `IMAGE` fields, and
therefore — in theory — an `ImagingSelection` produced from such a
reference can legitimately point at instances in a *different* study
than the SR's parent study. That re-opening is what brings the ticket
back to the WG, with Status now `Triaged`.

There are **no cross-referenced Jira tickets, Zulip threads, or GitHub
items** on file in Augury for this issue, so the analysis below relies
solely on the ticket itself and on documented DICOM SR / FHIR
ImagingSelection semantics.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat `studyUid` and `seriesUid` on the IG's `ImagingSelection`
profiles as **truly optional / "populate only when known"**, and
loosen any mapping prose that implies they must match the parent
SR's study. Concretely, in the DICOM SR → FHIR Resource Mapping IG:

- For every `ImagingSelection` profile derived from TID 1500 / TID
  1600 IMAGE references, change the mapping narrative so it does
  **not** assert that `ImagingSelection.studyUid` /
  `ImagingSelection.seriesUid` are inherited from the SR's parent
  study; state instead that they are taken from the referenced
  IMAGE content item's UIDs and may differ from the SR's study /
  series.
- Keep the elements optional (per base `ImagingSelection`
  cardinality), and add an explicit note: "If the referencing
  IMAGE content item does not carry the study/series UIDs, omit
  these elements rather than back-filling from the SR's parent
  study."
- Update any examples that currently re-use the SR's parent
  study/series UIDs without justification, so they show at least
  one cross-study reference (illustrating the TID 1500 ↔ TID 1600
  pattern Bas cited).

#### Justification

This aligns the IG with the underlying DICOM semantics Bas raised in
his 2025-08-14 comment: TID 1500 measurements may legitimately
reference IMAGE entries in TID 1600 that come from a different
study, so a hard "same study" assumption in the mapping would
silently produce incorrect FHIR. It also keeps the IG consistent
with base R5 `ImagingSelection`, where `studyUid` and `seriesUid`
are 0..1 — the IG should not narrow that contract by side-channel
prose.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `studyUid` / `seriesUid` populated by default (because most
real-world SRs do reference instances in the parent study), but
**document the cross-study case explicitly** in the IG mapping
guidance instead of changing cardinality or removing the elements:

- Add a "Scope of reference" note to each affected
  `ImagingSelection` mapping: by default the referenced instances
  share the SR's study; populate `studyUid` / `seriesUid` from the
  IMAGE content item itself, and acknowledge that the resulting
  values may differ from the SR's parent study when TID 1500
  measurements reference TID 1600 entries from another study.
- Add at least one worked example in the IG showing a cross-study
  IMAGE reference being mapped to `ImagingSelection` with a
  different `studyUid` than the SR's study.
- Leave the profiles themselves unchanged (cardinality stays as in
  base R5).

#### Justification

This addresses the reporter's underlying concern (the IG appearing
to mandate same-study) without removing useful structural
information from typical SRs, where same-study is the common case.
It is also the lowest-risk change for downstream implementers who
have already built against the current profiles — no cardinality or
binding shifts, only normative-prose clarification plus an
illustrative example.

---

### Disposition C: Decline

#### Proposal

Re-affirm the original 2024-09-22 *Not Persuasive* resolution: ask
the reporter to point at the **specific mapping table or profile
constraint** in the IG that asserts a same-study requirement. If no
such constraint is found, close as Not Persuasive on the grounds
that the IG never made the claim in the first place — `studyUid`
and `seriesUid` already carry their base R5 cardinality (0..1) and
the IG only populates them in examples, which are illustrative, not
normative.

#### Justification

Jeremy Whitby's 2024-09-22 comment ("I cannot find any cases where
study / series is included in the `ImagingSelection` mapping except
in the examples") is still on the record and was never rebutted with
a citation. If the IG does not in fact constrain `studyUid` /
`seriesUid` beyond what base `ImagingSelection` says, then there is
nothing to change and the request is moot. Declining preserves the
WG's existing reading until the reporter or another reviewer
identifies the offending text.

---

### Recommendation

**Recommended disposition:** B

The reporter's 2025-08-14 comment supplies a real DICOM-side
justification (TID 1500 measurements can cite IMAGE entries from any
study in TID 1600), which makes a pure "Not Persuasive" close (C)
hard to defend a second time even if no normative text in the IG
explicitly forces same-study — readers will reasonably *infer* the
constraint from examples and silent prose. At the same time,
Disposition A's blanket "remove studyUid / seriesUid unless known"
change overcorrects: in the dominant SR case the parent study *is*
the right answer, and stripping those elements would lose
round-trippable context that downstream renderers rely on. B
threads the needle: it keeps the profiles and cardinalities as-is,
adds explicit "scope of reference" guidance that aligns with the TID
1500 / TID 1600 semantics Bas cited, and adds a cross-study worked
example so the IG is no longer silent on the case the reporter is
worried about. The WG should still ask Bas to confirm a concrete
mapping passage to update so the editorial change is anchored, but
that fact-finding belongs in the disposition write-up, not in
another full re-triage cycle.
