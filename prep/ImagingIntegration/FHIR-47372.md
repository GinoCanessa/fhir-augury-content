## Summary

The reporter notes that the DICOM SR to FHIR Resource Mapping IG describes a
parent/child relationship between an Algorithm Identification Device and the
Equipment Device — quoting the IG narrative: *"If both have a defined
Algorithm Identification Device, both devices will be created and each will
have the Equipment Device as its parent."* — but this constraint is not
expressed in the `AlgorithmIdentificationMapping` profile itself. They request
that the profile's `Device.parent` element be typed (or constrained) to point
at the IG's General Equipment Device profile so the textual rule is enforced
by the artifact.

The ticket is already marked **Resolved: Persuasive** (resolved 2025-10-17),
with the resolution note pointing at HL7/dicom-sr commit
[b78d87f8](https://github.com/HL7/dicom-sr/commit/b78d87f8176ba3a841cfacea3af752436b845228).
The reporter's only follow-up comment ("Looks good") confirms acceptance of
that change. No related Jira tickets, Zulip threads, or indexed GitHub items
surfaced via cross-reference or search.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the DICOM SR to FHIR Resource Mapping IG, update the
`AlgorithmIdentificationMapping` profile so that its `Device.parent` element
is typed/constrained as `Reference(<canonical of the IG's General Equipment
Device profile>)`. Concretely, set a `targetProfile` on the
`Device.parent` element differential pointing at the General Equipment
Device profile defined elsewhere in the IG, so the textual rule
("each will have the Equipment Device as its parent") is structurally
enforceable. Regenerate the IG and confirm the change appears in the
profile snapshot, the IG narrative is consistent, and any example instances
that exercise this profile still validate.

#### Justification

This is exactly what the reporter requested and what the published
resolution says was done (commit b78d87f on HL7/dicom-sr; reporter
confirmed "Looks good"). Constraining `Device.parent` via `targetProfile`
is the standard FHIR mechanism for asserting that a referenced resource
must conform to a specific profile, and it eliminates the gap between IG
narrative and machine-checkable profile constraints. Because the IG already
defines a General Equipment Device profile, no new artifacts are required;
the change is local to one profile differential.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) a `targetProfile` constraint, encode the
parent-device requirement with an invariant on the
`AlgorithmIdentificationMapping` profile — e.g., a FHIRPath constraint of
roughly the form
`parent.exists() implies parent.resolve().conformsTo('<canonical of General
Equipment Device profile>')` — and keep `Device.parent` typed as a generic
`Reference(Device)`. Optionally pair this with non-normative narrative
cross-references between the AlgorithmIdentificationMapping and General
Equipment profiles so readers see the relationship from either side.

#### Justification

A `targetProfile` is enforced at static profile resolution time and may be
treated as a hard requirement by some validators, which can be too strict
when an Algorithm Identification Device is supplied without a fully
profiled parent (e.g., a generic `Device` reference from a system that
hasn't adopted the IG's General Equipment profile). An invariant lets the
IG state the rule clearly while leaving validators flexibility about
severity, and it avoids tightly coupling two profile canonicals at the
type level. It's a softer, more interoperable constraint, at the cost of
being slightly less discoverable than a typed reference in the snapshot.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the IG's narrative already specifies the
parent/child relationship and that `Device.parent` in core FHIR is already
typed as `Reference(Device)`, which the General Equipment Device profile
satisfies by construction. Add a sentence in the AlgorithmIdentificationMapping
profile description pointing readers at the General Equipment Device profile,
but do not add a structural constraint.

#### Justification

This is the minimum-change option and avoids any risk of over-constraining
implementers who legitimately want to point `Device.parent` at a Device
resource that has not been re-profiled to the IG's General Equipment Device.
It also avoids potential validator churn if the General Equipment Device
profile evolves. The cost is that the IG continues to carry a rule that is
only enforced by prose, which is what the reporter is asking us to fix.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been dispositioned **Persuasive** by the work group
and the requested change has been merged on HL7/dicom-sr (commit b78d87f),
with the reporter explicitly approving the result. Disposition A simply
ratifies that outcome: typing `Device.parent` against the IG's General
Equipment Device profile is the most direct way to make the IG's narrative
machine-checkable, the change is local to one profile differential, and it
adds no new artifacts. Disposition B is worth keeping in mind only if
implementer feedback later shows the `targetProfile` is too strict in
practice, in which case the constraint can be relaxed to an invariant
without removing the AlgorithmIdentificationMapping profile itself.
Disposition C is not recommended because it leaves the original gap — a
narrative-only rule with no structural enforcement — in place, which is
exactly what the reporter (correctly) flagged.
