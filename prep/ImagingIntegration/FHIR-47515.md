## Summary

Ana Kostadinovska reports that an example in the DICOM SR to FHIR Resource
Mapping IG appears to be incorrect. The flagged example pairs an observation
*code* of "radiographic measurement of the lung" with an observation *value*
of "Solitary nodule of lung". The reporter argues this pairing is
semantically wrong: if the observation is a volume (or other quantitative)
measurement of the lung, then the value should be a measurement result —
either a `Quantity` (e.g., a numeric volume with units) or an ordered
qualitative assessment (low / normal / high) — and not a coded finding
describing a lesion. The label `II_SEP_2024_SR_Example` indicates the
ticket was raised against the September 2024 batch of SR example reviews.
No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FhirAugury for this key, so the ticket appears to stand on its
own pending Imaging Integration triage.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the flagged example as defective and replace it. In the example
artifact in the DICOM SR to FHIR Resource Mapping IG that pairs the code
"radiographic measurement of the lung" with the value "Solitary nodule of
lung":

- Keep the code "radiographic measurement of the lung" (or its DCM/LOINC
  equivalent) as `Observation.code`.
- Replace `Observation.valueCodeableConcept` with an `Observation.valueQuantity`
  representing a realistic measurement result (e.g., a numeric nodule
  volume in mm³, or a long-axis diameter in mm with appropriate UCUM
  units).
- If the original SR document genuinely encoded a coded finding rather
  than a measurement, split the example into two related Observations:
  one measurement Observation (code = the measurement concept, value =
  Quantity), and one finding Observation (code = "Pulmonary nodule
  finding" or similar, value = "Solitary nodule of lung"), linked via
  `Observation.derivedFrom` or `Observation.hasMember` as appropriate to
  the SR template.

#### Justification

The reporter is correct on FHIR semantics: `Observation.code` names *what
was observed* and `Observation.value[x]` carries *the result of that
observation*. Pairing a measurement code with a finding-style coded value
violates the modeling intent of Observation and is inconsistent with
guidance in the FHIR Observation resource and Observation profiling work
(e.g., vital signs, quantitative imaging). Examples in a normative-track
mapping IG are particularly load-bearing: implementers copy them. Fixing
the example aligns the IG with FHIR Observation semantics and removes a
foot-gun for DICOM-to-FHIR converter authors.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than rewriting only the single flagged example, treat
`II_SEP_2024_SR_Example` as a class of issues and:

1. Audit every example in the DICOM SR to FHIR Resource Mapping IG that
   carries the `II_SEP_2024_SR_Example` label (or that was sourced from
   the same September 2024 SR sample set), looking for the same
   code-vs-value category mismatch.
2. Add a short normative-style guidance section to the IG explaining how
   DICOM SR "concept name + measured value" rows map onto
   `Observation.code` + `Observation.value[x]`, with explicit rules for
   when a coded SR value should become `valueQuantity`,
   `valueCodeableConcept`, or a separate companion Observation.
3. Apply the fix from Disposition A as part of the audit, plus any
   analogous fixes the audit surfaces.

#### Justification

The reported defect is almost certainly not unique — the September 2024
example batch was generated/curated as a group, and the same conceptual
confusion (treating a coded finding as a measurement value) tends to
propagate across sibling examples. A targeted audit plus a small
normative paragraph costs more editorial effort than a one-line fix but
prevents a steady drip of future tickets against the same example set,
and gives implementers a citable rule rather than only a corrected
example. Trade-off vs. Disposition A: more work for the WG and authoring
team, slightly later fix-publish time, but materially better long-term
quality of the IG's example set.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the example is intentionally
illustrating an SR row in which the DICOM source content already
contained that coded value, and the IG's job is to faithfully represent
the SR mapping rather than to silently "correct" the SR. Mark the ticket
as Not Persuasive / Not Related and close.

#### Justification

If — and only if — the example is sourced verbatim from a real DICOM SR
document where the SR template legitimately uses a coded finding as the
"value" of a "measurement" concept, then the IG is documenting reality,
not endorsing it. In that case the right answer would be to add a short
note to the example explaining the SR-side oddity rather than to change
the example. This rationale is fragile, however: even if the SR source
is faithful, the *FHIR mapping* still has to land on a sensible
Observation shape, so this disposition would only be defensible if such
a note already exists or is added.

---

### Recommendation

**Recommended disposition:** B

The reporter's specific complaint is well-founded and should be acted on
(Disposition A's fix is the right local change), but the ticket label
`II_SEP_2024_SR_Example` strongly suggests this example came from a
batch and that the same code-vs-value confusion is likely present in
sibling examples. Spending the marginal extra effort to audit the
September 2024 example set and to add a short paragraph on how SR
concept-name/value rows map onto `Observation.code` /
`Observation.value[x]` — and only then applying the Disposition A fix —
will give Imaging Integration a durable improvement to the IG rather
than a one-off patch. Disposition C is hard to defend without explicit
evidence that the SR source genuinely contained a coded finding in the
value slot, and even then would still warrant Disposition B's guidance
addition.
