## Summary

The reporter flags a notation error on the *Architecture / 3.1.1.2
ImagingSelection* page of the DICOM SR to FHIR Resource Mapping IG. The
text uses the form *"region content item CID 111030 'Image Region'"*,
which mis-uses the DICOM acronym **CID** (Context Group Identifier) as
if it were a code identifier. `111030` is **not** a CID — it is the DCM
code that represents the coded concept *Image Region*
(`(111030, "DCM", "ImageRegion")`, an Enumerated Value). The reporter
notes that other places in the IG already use the canonical notation
`Concept Name <coding scheme>#<code> "meaning"` (e.g.,
`DCM#111030 "Image Region"`) and asks that the same notation be applied
consistently in this section, and to the other concept references in
the same paragraph. The ticket has been Triaged with resolution
**Persuasive** and a concrete resolution proposal that rewrites the
ImagingSelection (and adjacent BodyStructure) bullet list to use the
`DCM#<code> "meaning"` form, with hyperlinks to the corresponding
DICOM PS3.16 chapter D entries and to each profile in the IG. No
related Jira tickets, Zulip threads, or GitHub items were surfaced by
cross-reference or content search, so this is treated as a
self-contained editorial fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution exactly as already drafted on the ticket: rewrite
the *Architecture / 3.1.1.2 ImagingSelection* sub-section so each
profile bullet uses the canonical `Concept Name <coding scheme>#<code> "meaning"`
notation — i.e., replace every occurrence of *"CID 111030 'Image Region'"*
(and the analogous `121191`, `121214`, `130488`, `121231`, `126100`,
`121200`, `130401` references) with `DCM#<code> "meaning"`, hyperlinking
the code to the corresponding DICOM PS3.16 chapter D anchor
(`https://dicom.nema.org/medical/dicom/current/output/chtml/part16/chapter_D.html#DCM_<code>`)
and the profile name to its StructureDefinition page in the IG. Make
the matching pass on the BodyStructure bullet (`SCT#363698007 "Finding
Site"`) so that the page is internally consistent. The change is
editorial-only: no StructureDefinitions, ValueSets, CodeSystems, bindings
or examples are touched. Implementation lands in the IG narrative source
(the `input/pagecontent/` Architecture page in the DICOM SR IG repo,
once that repo is brought into the local cache via `repo-analysis`).

#### Justification

The reporter's analysis is technically correct: in the DICOM standard
**CID** denotes a Context Group ID (e.g., `CID 7000`), while `111030`
is a *coded concept* identifier within the DCM coding scheme — the two
are categorically different. Using "CID 111030" mis-states the DICOM
information model and is also inconsistent with the rest of the IG,
which already uses the `DCM#<code> "meaning"` shorthand. The drafted
resolution is minimal, surgical, and self-consistent with the IG's own
established convention; the workgroup has already marked the ticket
**Persuasive**, signalling agreement with the substance.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the *intent* of Disposition A but, while making the edit, also
add a one-paragraph **notation key** at the top of the Architecture
chapter (or in the IG's introductory conventions section) that
explicitly defines the IG's preferred shorthand:

> Throughout this guide, coded concepts are written as
> `<scheme>#<code> "meaning"` (e.g., `DCM#111030 "Image Region"`).
> *CID nnnn* refers to a DICOM Context Group, **not** to an individual
> code.

In the same pass, sweep the Architecture chapter (and ideally the rest
of the IG narrative) for any other occurrences of *"CID <numeric>"*
that are actually code references rather than Context Groups, and
correct them. File the sweep as a sibling editorial change so the
review surface stays small.

#### Justification

The reporter explicitly notes *"Same applies to other referenced
concepts in this section"*, which signals that the mis-notation may
recur elsewhere. A documented convention plus a sweep prevents the
issue from re-emerging in future ticket cycles and reduces ambiguity
for implementers who are simultaneously reading the DICOM PS3.16 (where
`CID` has a strict meaning). The cost is incremental — the resolution
already requires editing the same page — and it converts a one-shot fix
into a durable improvement in the IG's editorial conventions.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive* on the grounds that the existing text,
while informally worded, is unambiguous in context (the surrounding
sentence makes clear that `111030` identifies the Image Region concept)
and that re-editing every occurrence is not worth the editorial churn.
Close without change.

#### Justification

A pure editorial fix has no implementer impact on conformance: no
profile, ValueSet, or binding changes; no examples break; no migration
is required. If the workgroup is constrained on editorial bandwidth or
wants to bundle small fixes into a larger pass, deferring or declining
is defensible.

---

### Recommendation

**Recommended disposition:** **B**

The substance of the request is correct — *CID* genuinely is the wrong
DICOM term for a coded concept identifier — and the resolution already
on the ticket (Disposition A) is uncontroversial; the workgroup has
already evaluated it as **Persuasive**. Disposition B preserves
everything Disposition A does and additionally pays down the
underlying root cause: the IG lacks an explicit, discoverable notation
convention, which is why the same mistake was reported and may exist
elsewhere (the reporter says as much). Disposition C is hard to
defend: the cost of the edit is low, the correctness benefit is real
for an IG that sits at the boundary between two standards (DICOM and
FHIR) where terminology precision matters to implementers. Disposition
B is therefore the strongest starting point for the workgroup
discussion; it can be downgraded to A trivially if scope concerns are
raised.
