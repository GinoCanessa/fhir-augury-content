## Summary

The reporter (Ana Kostadinovska) observed that an example in the *DICOM SR
to FHIR Resource Mapping IG* shows only a DICOM representation and asks
whether the corresponding mapped FHIR representation has been omitted by
mistake. The ticket suggests that the example be extended so that **both**
the DICOM source and its FHIR-mapped counterpart are presented side by
side, so readers can see the result of applying the IG's mappings.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned for this ticket, so the request stands on its own without
broader linked discussion.

## Details

**Description:**

> The example provided only shows DICOM information. Is the mapped FHIR
> representation missed?
> Suggest including both the DICOM and FHIR representations.

**Comments:**

- **jwhitby (2024-09-24):** Clarifies that the example was not intended to
  create a full FHIR `DiagnosticReport` representing the contents of the
  DICOM SR — it was used as a mechanism to surface a DICOM JSON SR as a
  FHIR Example. He notes that the contents are mostly mapped already in
  the relevant section, and asks whether producing a `DiagnosticReport`
  alongside the DICOM SR would be confusing given that producing such a
  resource is not an intended output of this IG.

**Resolution:** Not Persuasive (2025-10-30).

> As noted in comment, the goal of the IG is **not** to produce a
> representation of the entire SR as a DiagnosticReport or Composition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the IG example currently showing only a DICOM SR JSON payload to
additionally include the mapped FHIR representation (a
`DiagnosticReport` — and/or `Composition` — populated from the DICOM SR
contents per the IG's mapping rules, with referenced `Observation`
instances for the SR content items as appropriate). Render both
representations together on the affected example page so a reader can
visually compare the DICOM source with the FHIR output produced by the
IG's mappings.

#### Justification

For an Implementation Guide whose explicit purpose is to map DICOM SR
content to FHIR resources, presenting only the DICOM side of an example
arguably under-delivers on the IG's value proposition. Showing both
sides makes the mapping concrete, removes ambiguity about exactly which
SR content items become which FHIR elements, and helps implementers
validate their own mappings against an authoritative example.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than building a full `DiagnosticReport`/`Composition` for the
example (which is **not** an intended output of this IG per the
assignee's comment), keep the DICOM SR JSON as the example payload but
add an adjacent section to the example page that presents the
**element-level mapped FHIR fragments** the IG actually produces — for
example, the individual `Observation` resources or extension values
derived from the SR content tree. Add a short narrative cross-link
explaining what is mapped and what is intentionally not mapped, and
reference the IG's mapping section so readers see exactly which
mappings the example exercises.

#### Justification

This addresses the reporter's underlying concern — that the FHIR side of
the mapping is invisible in the example — without introducing artifacts
(`DiagnosticReport`, `Composition`) that the IG explicitly does not
intend to produce. It avoids confusing readers about the IG's scope
while still demonstrating the FHIR-side outputs the IG does cover.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Decline the request to add a full FHIR representation alongside the
DICOM SR in the example. Document in the resolution that the IG's
purpose is to map DICOM SR **content items** to FHIR elements, not to
produce a `DiagnosticReport` or `Composition` representation of an
entire SR, and that the example was intentionally a DICOM JSON payload
used as a vehicle to surface SR content. Optionally clarify the
example's narrative to make the IG's scope explicit so future readers
do not draw the same inference.

#### Justification

The assignee has stated that producing a `DiagnosticReport`/`Composition`
for the SR is out of scope for this IG, and the existing mapping section
already documents the element-level mappings. Adding a synthetic
`DiagnosticReport` for example purposes risks misleading implementers
into believing such a resource is an expected output of conformant
implementations.

---

### Recommendation

**Recommended disposition:** C

This matches the disposition Imaging Integration ultimately applied
(Resolution: *Not Persuasive*, 2025-10-30) and is the right call: the
IG explicitly does **not** scope the production of a full
`DiagnosticReport`/`Composition` for a DICOM SR, and adding one to an
example would set an implementer expectation that contradicts the IG's
stated intent. That said, the reporter's confusion is legitimate
signal — if the workgroup wants to soften the decline, Disposition B
(adding the *element-level* mapped FHIR fragments and a clarifying
narrative on the example page) addresses the underlying concern without
expanding the IG's scope, and could be tracked as a documentation
follow-up rather than reopening this ticket.
