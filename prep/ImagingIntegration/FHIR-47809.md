## Summary

Rob McClure reports that the Terminology section of the *DICOM SR to FHIR
Resource Mapping IG* is completely empty, and asks the Imaging Integration
work group to populate it with substantive guidance. Specifically, the
section needs to (a) explain how implementers are expected to access and use
DICOM-published terminology (e.g., DCM / DICOM Code Definitions and
referenced external coding systems such as SNOMED CT, LOINC, RadLex) when
implementing this guide, (b) clarify what the IG expects of profiles with
respect to DICOM terminology beyond the cases where a fixed code is bound,
and (c) describe the general approach to terminology mapping used throughout
the guide. No related Jira tickets, Zulip threads, or GitHub items were
surfaced by the available cross-reference and search APIs, so the request
stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Author the Terminology section of the *DICOM SR to FHIR Resource Mapping
IG* as a dedicated narrative page covering, at minimum:

1. **Source terminologies in scope** — DICOM Controlled Terminology
   (`http://dicom.nema.org/resources/ontology/DCM`), SNOMED CT, LOINC,
   RadLex, UCUM, and any IG-defined CodeSystems/ValueSets used by the
   profiles.
2. **Access guidance** — where to obtain each terminology (DICOM PS3.16
   Annex D for DCM, the THO publication of DCM, terminology server
   expectations, license/redistribution notes for SNOMED/LOINC/RadLex).
3. **Binding philosophy** — explain the IG's use of `required`,
   `extensible`, `preferred`, and `example` strengths, and clarify that
   profiles fix individual codes only where DICOM SR templates require
   a specific concept; otherwise the value set remains open to the
   underlying DICOM context group.
4. **Mapping approach** — describe how DICOM SR Code Sequences (Coding
   Scheme Designator + Code Value + Code Meaning) translate to FHIR
   `Coding` (system + code + display), including the canonical
   `system` URIs the IG uses for each DICOM Coding Scheme Designator,
   and how ConceptMaps (if any) are organised.
5. **Implementer expectations** — when validators should accept any
   member of a DICOM context group, how to handle private/local codes,
   and how unknown DICOM coding schemes should be represented.

Land this as a new IG page (e.g., `input/pagecontent/terminology.md`)
referenced from the IG ToC, with cross-links from each affected profile.

#### Justification

This is exactly what the reporter asked for and is the minimum content
expected of any FHIR IG that profiles resources whose semantics are
carried in coded values. Without it, implementers have no consistent
way to interpret the IG's bindings, particularly the ones that are
deliberately open. Populating the section also discharges the wider
ballot expectation that IGs explain their terminology strategy, and
gives reviewers a single place to evaluate whether the mappings from
DICOM SR Code Sequences to FHIR `Coding` are coherent.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring one large narrative page, split the Terminology
guidance into two smaller, more targeted artifacts:

1. A short Terminology overview page that states the IG's binding
   philosophy and lists the source terminologies and their canonical
   `system` URIs (one screen of content).
2. A machine-readable **per-Coding-Scheme-Designator → FHIR `system`**
   mapping table generated from a NamingSystem (or a small CSV checked
   into the IG) and rendered into the IG, plus an inline "Terminology
   notes" callout on each profile page that uses non-trivial DICOM
   bindings.

Defer deep mapping prose (ConceptMaps, code-by-code rationale) to the
profile pages where the bindings actually live.

#### Justification

The reporter's underlying need is that implementers can answer "what
terminology do I use here, and where do I get it?" without hunting.
Co-locating that information with the profile that uses it is often
more useful than a single monolithic chapter, and the NamingSystem /
table approach makes the Coding-Scheme-Designator → `system` mapping
authoritative and testable rather than buried in prose. This also
keeps the central Terminology page small enough to be maintained as
the IG evolves, reducing the risk of it drifting back to "empty or
stale".

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for this ballot cycle on the grounds that DICOM
terminology guidance is already published normatively by DICOM (PS3.16,
PS3.17) and that the IG can simply reference those documents without
restating them, leaving the Terminology section as a brief pointer.

#### Justification

A defensible argument exists that an IG should not duplicate normative
DICOM content. However, this disposition is weak because the reporter is
not asking for DICOM normative text to be restated — they are asking for
the IG-specific guidance (binding strengths, system URIs, mapping
approach, expectations on profiles) that *only* this IG can provide.
Declining therefore leaves a real gap, and should only be selected if
the work group genuinely intends to ship the IG with no Terminology
section.

---

### Recommendation

**Recommended disposition:** A

The Terminology section is a structural expectation of any FHIR IG that
profiles coded data, and the reporter — Rob McClure, who reviews this
material across many work groups — has flagged it explicitly with
priority "Highest". The section's current emptiness is not a stylistic
issue but a substantive gap that prevents implementers from reasoning
about the IG's bindings or about how DICOM SR Code Sequences round-trip
into FHIR `Coding`. Disposition B is an attractive refinement and could
reasonably be folded into the implementation of A (the central page can
be kept tight and lean on per-profile callouts), but the work group
should commit to producing the central guidance rather than dispersing
it. Disposition C is not defensible given the reporter's specific
request for *IG-level* guidance that DICOM's own publications do not
provide.
