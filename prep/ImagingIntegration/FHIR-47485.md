## Summary

The reporter notes that the DICOM specification for Qualitative Evaluations
permits encoding a flat list of name–value pairs whose answers may be either
coded values **or** free text. The current DICOM SR to FHIR Resource Mapping
IG profile (`imaging-qualitative-evaluation`) only documents an encoded
(coded) value, with no clear path for a text-only answer. Diana asks the
Imaging Integration work group to extend the value definition so that
text-only qualitative answers can also be represented when round-tripping
DICOM SR content to FHIR.

No related Jira tickets were surfaced via the augury cross-reference index,
and no Zulip threads or GitHub PRs/issues directly reference this ticket.
The ticket has already been triaged with a resolution of *Not Persuasive
with Modification*, meaning the work group concluded that the underlying
capability already exists (via `CodeableConcept.text`) but the mapping
guidance can be made clearer.

## Details

**Description:**

According to DICOM the Qualitative Evaluations allow encoding a flat list of
name-value pairs that are coded questions with coded or **text** answers.

The current value definition only allows encoded data but no text. Please
add this possibility.

**Resolution Description (already recorded on the ticket):**

> `CodeableConcept` data types include the ability to encode text-only
> values: https://hl7.org/fhir/R5/datatypes.html#CodeableConcept
>
> The mapping table includes text and coded values as the source for the
> `valueCodeableConcept`:
> https://hl7.org/fhir/uv/dicom-sr/2024SEP/StructureDefinition-imaging-qualitative-evaluation-mappings.html
>
> It would be possible to split the mapping line to make this clearer –
> e.g. `text -> valueCodeableConcept.text` and
> `code -> valueCodeableConcept.coding`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `imaging-qualitative-evaluation` profile in the DICOM SR to FHIR
Resource Mapping IG so that the value-binding guidance explicitly permits a
text-only answer. Concretely, leave the element as `valueCodeableConcept`
(no type change), but in the profile's element documentation and in the
DICOM-to-FHIR mapping table, split the current single mapping row into two
rows:

- DICOM coded answer → `Observation.valueCodeableConcept.coding`
- DICOM text answer → `Observation.valueCodeableConcept.text`

Add a short note in the profile's narrative that text-only qualitative
answers are represented by populating only `valueCodeableConcept.text`
(with `coding` absent), and that mixed coded+text answers populate both.

#### Justification

This directly addresses the reporter's concern by making the supported
behaviour explicit at the place an implementer reads it (the profile and
its mapping table), without introducing any breaking change. It is the
modification path the resolver already sketched in the resolution
description, which keeps the disposition consistent with prior work-group
thinking.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the underlying element as `valueCodeableConcept` (text capability is
already inherent to `CodeableConcept`), but instead of expanding the
mapping table, add a normative-style "Handling text-only evaluations"
section to the IG narrative for the Qualitative Evaluation profile. This
section would:

1. State that DICOM TID-style text-only answers map to
   `Observation.valueCodeableConcept.text`.
2. Provide a worked JSON example of a qualitative evaluation with only
   `text` populated and `coding` omitted.
3. Cross-link to the FHIR R5 `CodeableConcept` definition for the formal
   semantics.

No structural change to the profile. The mapping table remains a single
row but is annotated with a footnote pointing at the new narrative
section.

#### Justification

Implementers most often miss this capability because the *mapping table* is
read as a constraint rather than as a source-side description. Anchoring
the answer in narrative + a worked example reduces the chance that future
readers re-raise the same ticket, and avoids subdividing a mapping row in
a way that could be misread as two distinct target elements. Trade-off vs
Disposition A: less mechanical/compact in the mapping table, but stronger
on implementer comprehension.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive*: take no change to the profile, the
mapping table, or the narrative. Respond that
`valueCodeableConcept.text` is already a documented FHIR capability of
`CodeableConcept`, that the current mapping does not forbid populating
it, and that the IG therefore already supports the requested behaviour.

#### Justification

Defensible on the grounds that no spec or profile change is strictly
necessary to enable text-only qualitative answers — `CodeableConcept.text`
is base-FHIR machinery that any conformant implementation will accept.
However, this position ignores the discoverability problem that the
ticket actually raises: the mapping table reads as if only coded values
are in scope, which is why the reporter (a competent implementer) filed
the change request in the first place.

---

### Recommendation

**Recommended disposition:** A (closely aligned with the work group's
already-recorded *Not Persuasive with Modification* resolution).

The reporter's substantive concern is real but mechanical: the IG's
mapping table for `imaging-qualitative-evaluation` reads as if coded
values are the only sanctioned target, even though the chosen FHIR data
type (`CodeableConcept`) inherently supports text. Splitting the mapping
row into `text -> valueCodeableConcept.text` and
`code -> valueCodeableConcept.coding` (Disposition A) is the smallest,
most surgical change that resolves the ambiguity at the exact place
implementers consult, requires no structural profile change, and matches
the modification path already articulated in the ticket's resolution
description. Disposition B is a reasonable enhancement to layer on top of
A if work-group reviewers want a worked example, but on its own it leaves
the mapping table — the document implementers actually copy from — still
ambiguous. Disposition C is hard to defend given that a domain expert
already misread the current text; declining outright would invite the
same ticket to be re-filed.
