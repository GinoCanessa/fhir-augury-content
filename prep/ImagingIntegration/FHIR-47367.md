## Summary

The ticket asks why the **Derived Imaging Measurement** profile is omitted
from the IG rule that maps SR content carrying a child with Concept Name
DCM#111001 ("Algorithm Name") to an **Algorithm Identification Device**.
The existing rule covers **Imaging Measurement Group**, **Imaging
Measurement**, and **Imaging Qualitative Evaluation**, but not Derived
Imaging Measurement. The reporter (filing on behalf of Ignacio Jauregui)
notes that derived observations — calculations, scores, and similar
algorithmically-produced values — are exactly the kind of content most
likely to have been generated by an identifiable algorithm, and so the
mapping to Algorithm Identification Device is at least as relevant for
Derived Imaging Measurement as for the three profiles already listed.

The ticket is a narrow, surgical correction to the IG's mapping prose.
The resolution recorded on the ticket ("Persuasive") already proposes
adding Derived Imaging Measurement to both the trigger list and the
target list in the affected section. No related Jira tickets, Zulip
threads, or GitHub items were surfaced by Augury for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the "Algorithm Identification Device" mapping section of the DICOM SR
to FHIR Resource Mapping IG, add **Derived Imaging Measurement**
(`StructureDefinition-derived-imaging-measurement.html`) to **both** the
trigger list and the target list, so the rule reads:

> If an Imaging Measurement Group, Imaging Measurement, Imaging
> Qualitative Evaluation **or Derived Imaging Measurement** has a child
> with Concept Name DCM#111001 "Algorithm Name", then it is mapped to an
> Algorithm Identification Device.

This is a prose/narrative-only update to the IG. It does not alter any
StructureDefinition, ValueSet, ConceptMap, or example. Apply the change
in both occurrences of the list (the trigger sentence and the target
sentence) so the two stay in sync, exactly as called out in the ticket's
recorded resolution.

#### Justification

The four imaging-measurement profiles in this family form a single
content tree in DICOM SR (TID 1500 / TID 300x). Algorithmic provenance —
expressed via the DCM#111001 "Algorithm Name" concept — can appear under
any of them, and is in fact most frequently associated with derived
results (calculations, scores, ratios). Excluding Derived Imaging
Measurement from the mapping rule is a documentation gap, not a design
decision: the underlying SR pattern is identical, and the target Device
profile (Algorithm Identification Device) is profile-agnostic with
respect to which of the four observation profiles points at it. The
reporter has confirmed the proposed wording ("Looks good", 2025-10-31),
and the ticket is already marked **Persuasive**.

---

### Disposition B: Alternative Approach

#### Proposal

Replace the enumerated profile list with a single generic statement that
references the common parent pattern, e.g.:

> If **any imaging measurement profile** (Imaging Measurement Group,
> Imaging Measurement, Imaging Qualitative Evaluation, Derived Imaging
> Measurement, and any future profile derived from this family) has a
> child with Concept Name DCM#111001 "Algorithm Name", then it is mapped
> to an Algorithm Identification Device.

Optionally, define a named group (e.g., "Imaging Measurement family") at
the top of the IG and reference it from this and any similar mapping
rules.

#### Justification

The root cause of the omission is enumerating profiles by name in prose.
Every time a new profile is added to the family, the same kind of bug
will recur for every rule that lists them. A generic phrasing — or a
single named group used in multiple places — eliminates this class of
error and makes the IG more maintainable. The trade-off is that "any
imaging measurement profile" is slightly less precise than an explicit
list and may require a glossary entry; reviewers may prefer the
explicit, ticket-as-written change for clarity in this normative-style
mapping document.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that Derived Imaging Measurement was
intentionally excluded — i.e., that derived results are required to
inherit algorithm attribution from the source observation(s) they are
derived from (`Observation.derivedFrom`) rather than carrying a direct
mapping to Algorithm Identification Device.

#### Justification

This disposition is only defensible if the IG editors can document a
deliberate design intent that derived measurements should never carry
their own DCM#111001 child, and instead always inherit algorithm
identity from upstream observations. There is no evidence of such intent
in the ticket history, the reporter has confirmed the change, and the
ticket already carries a **Persuasive** resolution. Listed here only to
make the option explicit for the work group.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the change the reporter asked for, the change the
work group has already marked **Persuasive**, and the change the
reporter has explicitly signed off on ("Looks good"). It is a tightly
scoped narrative fix with no schema, terminology, or example impact,
and no risk of breaking downstream implementers — the rule is currently
silent on Derived Imaging Measurement, so adding it can only widen
correct mapping coverage. Disposition B (genericising the profile list)
is a sound maintainability improvement and could be accepted *in
addition* to A as a follow-up ticket, but it should not block landing
the specific fix this ticket requests. Disposition C is included for
completeness but is not supported by the ticket history.
