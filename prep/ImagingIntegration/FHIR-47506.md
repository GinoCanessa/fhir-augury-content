## Proposed Dispositions

> FhirAugury. The proposals below name the IG and the conceptual

### Disposition A: Accept As Requested

#### Proposal

In the example tagged `II_SEP_2024_SR_Example` within the **DICOM SR to
FHIR Resource Mapping IG**, replace the entry **`Device`** with
**`Practitioner`** in the "created FHIR resources" list / mapping table
for that SR example. Update any prose, narrative, and (if present)
companion example bundle so the produced resources include a
`Practitioner` instance in place of the `Device` instance, and adjust
references from the SR-derived resource (e.g., DiagnosticReport,
Observation, ImagingStudy) so that the participant slot that previously
pointed to a `Device` now points to the `Practitioner`. Verify that the
SR DCM concept being mapped (typically a Person Observer / "Observer
Context — Person" code in DICOM SR) is the one driving this row; if so,
Practitioner is the correct FHIR target.

#### Justification

In the standard DICOM SR → FHIR mapping pattern, **Person Observer** /
"Observer Type = Person" content items are mapped to **Practitioner**
(or PractitionerRole), while **Device Observer** / "Observer Type =
Device" items map to **Device**. If the SR example in question is a
person-observed report and the IG currently lists `Device` in the
"created FHIR resources" column, that is a straightforward editorial
error and Practitioner is the consistent target. Accepting as requested
keeps the IG aligned with how implementers already convert SR Observer
Context, and matches the reporter's reading of the example.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than a blanket swap of `Device` → `Practitioner`, **list both
resources conditionally** in the SR example: clarify that the "created
FHIR resources" depend on the **Observer Type** carried by the SR
instance (TID 1002 / Observer Context). Show:

- when Observer Type = Person → create `Practitioner` (and optionally
  `PractitionerRole`),
- when Observer Type = Device → create `Device`,

and include a short narrative note that the example was authored
assuming a person observer, so `Practitioner` is the resource produced
for the example as currently written. If the example bundle is wrong
about the observer, fix the bundle to use a `Practitioner`; if the
narrative is wrong, fix only the narrative. This disposition splits the
correction into "fix this example" plus "make the rule explicit so the
next author does not repeat the mistake".

#### Justification

The reporter's complaint is consistent with a single-example error, but
the underlying cause — that the IG enumerates "created FHIR resources"
without naming the SR Observer Type that drives the choice — is likely
to bite again in the next tranche of examples. Adding the conditional
guidance pre-empts a class of tickets and keeps the IG honest about the
DICOM-to-FHIR mapping rule, while still applying the editorial fix the
reporter asked for. Trade-off vs. Disposition A: slightly larger
editorial scope and a small risk that adding conditional language
re-opens unrelated mapping discussions during balloting.

---

### Disposition C: Decline

#### Proposal

Decline on the basis of **insufficient information to confirm the error
without re-examining the source SR**. Ask the reporter to attach (or
link) the specific SR instance / example identifier and the page in the
IG she is reading, then re-triage. If, on review, the example is
deliberately a Device-observed SR (e.g., an automated CAD report), the
existing `Device` mapping is correct and no change is needed.

#### Justification

A blanket text swap based on a one-line description carries a small but
real risk of breaking an example that was intentionally Device-observed
(automated detection, AI-derived findings, modality-generated
measurements all legitimately produce Device-observed SRs). Declining
pending confirmation is defensible if the WG cannot match the
description to a specific example without help. This is the weakest of
the three options because the reporter is a known Imaging Integration
contributor and the label `II_SEP_2024_SR_Example` already pins down
the tranche; in practice the WG should be able to locate the example
quickly.

---

### Recommendation

**Recommended disposition:** B

The reporter is almost certainly right about the immediate example, so
some flavour of "accept" is warranted — Disposition C only makes sense
if the WG genuinely cannot locate the example, which is unlikely given
the focused `II_SEP_2024_SR_Example` label. Between A and B, Disposition
B does the same editorial fix while taking the small additional step of
naming the underlying rule (Person Observer → Practitioner; Device
Observer → Device) in the IG. That extra clarification is cheap, it
costs nothing in terms of conformance, and it materially reduces the
chance of another ticket of the same shape landing against the next
tranche of SR examples. The Imaging Integration WG should adopt B,
falling back to A if the editorial scope of "add conditional guidance"
proves contentious in the meeting.
