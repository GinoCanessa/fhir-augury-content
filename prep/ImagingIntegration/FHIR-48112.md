## Summary

The reporter, working with the DICOM SR to FHIR Resource Mapping IG, asks the
Imaging Integration WG to relax the requirement that the `ImagingSelection`
profile defined in the IG references an `ImagingStudy` resource via
`ImagingSelection.derivedFrom`. In the DICOM SR → FHIR mapping flow, a
corresponding `ImagingStudy` FHIR resource frequently does not exist — it is
typically only created when there is an independent reason to do so, and the
images themselves are not necessarily mapped to FHIR when an
`ImagingSelection` is generated from a DICOM SR. The reporter therefore
proposes that `derivedFrom(ImagingStudy)` should not be required by the
profile.

The ticket has already been triaged and resolved as **Persuasive with
Modification**, with the resolution describing a concrete adjusted approach:
remove the `derivedFrom(ImagingStudy)` requirement, instead require that the
DICOM Study Instance UID (`studyUid`) is captured (which is always known when
an SR is being mapped), and add a mapping that specifies where `studyUid`
is sourced from in the DICOM SR.

No related Jira tickets, Zulip threads, or GitHub items were surfaced by the
cross-reference index or by content search across Zulip / GitHub for this
ticket key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `ImagingSelection` profile in the **DICOM SR to FHIR Resource
Mapping IG** so that `ImagingSelection.derivedFrom` referencing
`ImagingStudy` is **not required** (drop any `min = 1` constraint and any
must-support obligation that forces a populated `ImagingStudy` reference on
this profile). Make no other structural changes — leave the existing
`derivedFrom` element available as an optional reference for implementers
that do have an `ImagingStudy` resource on hand.

#### Justification

This is what the reporter literally asks for, and it is consistent with the
real-world DICOM SR → FHIR mapping flow they describe: the SR can be mapped
into an `ImagingSelection` even when no FHIR `ImagingStudy` instance exists
for the underlying study. Forcing `derivedFrom(ImagingStudy)` on this
profile creates an artificial dependency on producing a second FHIR
resource that the source workflow may have no reason to materialize, which
is at odds with the IG's stated purpose of mapping SR content into FHIR.
Relaxing the cardinality is non-breaking for systems that already populate
`derivedFrom`.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the **Persuasive with Modification** plan already drafted on the
ticket:

1. In the IG's `ImagingSelection` profile, remove the requirement that
   `ImagingSelection.derivedFrom` reference an `ImagingStudy` (drop
   `min = 1` / must-support coupling).
2. Require that the DICOM Study Instance UID is conveyed on the
   `ImagingSelection` itself by constraining `ImagingSelection.studyUid`
   to `min = 1` (and marking it must-support), with profile guidance
   stating that this value is always available from the source DICOM SR.
3. Add a concept map / mapping entry in the IG that explicitly identifies
   the DICOM SR source element (DICOM tag for Study Instance UID,
   `0020,000D`) as the source for `ImagingSelection.studyUid`.

#### Justification

This addresses the reporter's underlying concern — that the IG should not
force creation of an `ImagingStudy` FHIR resource — while preserving the
ability for downstream consumers to still identify the study the
selection came from. Carrying `studyUid` directly on `ImagingSelection`
keeps the study identity in the resource where it is needed, without
requiring a second FHIR resource to exist. Adding the DICOM SR → FHIR
mapping for `studyUid` is consistent with the IG's mapping-IG nature and
gives implementers an unambiguous source for the value. This is also the
approach already recorded as the resolution direction on the ticket, so
adopting it formalizes work that has effectively already been agreed.

---

### Disposition C: Decline

#### Proposal

Keep `ImagingSelection.derivedFrom(ImagingStudy)` as currently
constrained in the IG profile (i.e., do not relax the requirement). Close
the ticket as "Not Persuasive" with guidance that implementations
mapping DICOM SR to FHIR should construct a minimal `ImagingStudy`
resource alongside `ImagingSelection` when they need to expose
selections.

#### Justification

`ImagingSelection` was modeled with `derivedFrom(ImagingStudy)` to keep
selections explicitly tied to a FHIR-native representation of the study
they reference, which simplifies traversal and reuse in downstream FHIR
workflows. Requiring implementers to also produce an `ImagingStudy`
keeps the FHIR view internally consistent and avoids divergent patterns
where some `ImagingSelection` instances point at FHIR studies and others
carry only a raw DICOM identifier. Declining preserves the current
profile shape and avoids any spec churn.

---

### Recommendation

**Recommended disposition:** B

The ticket has already been triaged as **Persuasive with Modification** and
the resolution text on the ticket spells out exactly the modified plan
described in Disposition B (drop `derivedFrom(ImagingStudy)` requirement,
require `studyUid`, add a mapping for its source). That plan accepts the
reporter's underlying concern — DICOM SR mappers should not be forced to
produce an `ImagingStudy` FHIR resource — while keeping the study
identity discoverable on the `ImagingSelection` itself via `studyUid`,
which the reporter agrees is always known. Disposition A is acceptable
but weaker than B because it loses the study linkage entirely for
implementers that do not produce an `ImagingStudy`. Disposition C
ignores the reporter's stated workflow constraint and the WG's existing
triage outcome. The workgroup should ratify Disposition B, assign the
profile and mapping edits in the DICOM SR to FHIR Resource Mapping IG,
and move the ticket from Triaged → Resolved/Applied.
