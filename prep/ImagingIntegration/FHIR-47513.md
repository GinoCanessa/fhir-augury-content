## Summary

Reporter Ana Kostadinovska (Philips), reviewing the September-2024 ballot of
the **DICOM SR to FHIR Resource Mapping IG** (Imaging Integration WG), flags
that in the Example section the IG enumerates **Device** as one of the FHIR
resources created for the worked DICOM SR example, where she believes the
created resource should actually be **Observation**. The fix she requests is
purely editorial: correct the listed resource type so that the Example
section's "created resources" wording matches the resources the example
actually instantiates.

The ticket is one of a coherent batch of `II_SEP_2024_SR_Example` issues she
filed the same day reviewing the Example section. The closest sibling,
[FHIR-47506](https://jira.hl7.org/browse/FHIR-47506), is the same shape —
"Practitioner resource instead of Device resource is created" — confirming the
problem is a systematic mis-labelling of the resource list in that section
rather than a one-off typo. Other siblings such as
[FHIR-47466](https://jira.hl7.org/browse/FHIR-47466) (focus vs. derivedFrom),
[FHIR-47477](https://jira.hl7.org/browse/FHIR-47477) (unnecessary
BodyStructure creation), [FHIR-47673](https://jira.hl7.org/browse/FHIR-47673)
(misalignment between Mapping and Example sections), and
[FHIR-47223](https://jira.hl7.org/browse/FHIR-47223) (Example section mixes
normative content with examples) provide the broader context: the Example
section is widely viewed as inconsistent with the Mapping section and with
the example instances themselves, and FHIR-47513 is a small, concrete data
point in that pattern.

The ticket carries no description-level URL, no comments, no Zulip discussion,
and no GitHub cross-references, so the precise sub-section / resource being
flagged must be confirmed against the IG narrative at review time, but the
substance — "this list of created resources names the wrong resource type" —
is unambiguous.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a pure editorial correction in the **DICOM SR to FHIR
Resource Mapping IG** Example section. In the sub-section where the example
currently states that a **Device** resource is created, replace the word
`Device` with `Observation` in the "created resources" enumeration so that
the listed resource type matches the actual example instance (an
`Observation`). Make no other changes. Coordinate with FHIR-47506 (same
shape, asks for `Device → Practitioner`) and apply both fixes in a single
editorial commit so the Example section's resource inventory is internally
consistent in one pass.

#### Justification

The reporter's claim is a factual one ("the listed resource is wrong") rather
than a design proposal, and the sister ticket FHIR-47466 establishes that
small, narrowly-scoped editorial corrections from the same reviewer have
been accepted as **Persuasive** by Imaging Integration on this IG. Doing
nothing leaves a published example that contradicts itself, which is exactly
the kind of inconsistency FHIR-47673 and FHIR-47223 also flag in this
section. The fix is low-risk (no schema, no profile, no terminology change),
fully backwards-compatible, and improves implementer trust in the IG's
worked example.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying need but address it as part of a broader cleanup of the
Example section rather than a one-line edit. Specifically: (1) rebuild the
"created resources" list in each Example sub-section programmatically from
the actual example instances bundled with the IG (so the narrative cannot
drift from the examples); (2) fold FHIR-47513, FHIR-47506, FHIR-47477, and
the structural concerns from FHIR-47673 / FHIR-47223 into a single
`II_SEP_2024_SR_Example` epic; and (3) close FHIR-47513 as **Persuasive
with Modification**, pointing at the epic. The end state for FHIR-47513
specifically is the same — `Observation` instead of `Device` — but the fix
is delivered as part of a structural change that prevents the same class of
error reappearing.

#### Justification

The Example section's resource lists are clearly hand-maintained and have
drifted from the example instances (FHIR-47513, FHIR-47506, FHIR-47477 are
all the same defect class). Spot-fixing one ticket at a time leaves the
underlying authoring process unchanged and invites the same set of tickets
again at the next ballot. Generating the resource list from the example
bundles is consistent with how other HL7 IGs handle this and aligns with
FHIR-47673's complaint that the Mapping and Example sections are not
co-maintained. The trade-off is delivery time: this is a larger piece of
work than a single word swap, and the WG must be willing to defer the
individual ticket close.

---

### Disposition C: Decline

#### Proposal

Close FHIR-47513 as **Not Persuasive** on the grounds that the ticket does
not pin down the section it is referring to (the description says "here" but
contains no link, anchor, or quoted heading), and ask the reporter to
re-file with a precise section reference if the issue still applies in the
current build of the IG.

#### Justification

The description's lack of a concrete pointer is a real defect in the ticket
itself, and on a strict reading the WG cannot verify the claim without doing
the reporter's locating work for them. However, this position is weak: the
reporter is a known and active contributor on this IG, the
`II_SEP_2024_SR_Example` label scope is narrow (a single section in a single
IG), and sister tickets (notably FHIR-47506) confirm the pattern. Declining
on a technicality would also be inconsistent with how FHIR-47466 was handled
by the same WG. This disposition is included for completeness but is hard
to defend on the merits.

---

### Recommendation

**Recommended disposition:** A

The ticket is a small, low-risk editorial correction in a published IG
Example section, the reporter is a credible reviewer with an accepted track
record on this IG (FHIR-47466 → Persuasive / Applied), and the fix is
trivially verifiable against the example instance. The WG should take the
one-line fix now (paired with FHIR-47506 in the same commit so the Example
section's "created resources" list is internally consistent) and treat
Disposition B's structural cleanup as a separate, larger work item tracked
under the Example-section umbrella tickets (FHIR-47673, FHIR-47223). That
sequencing gets the immediate inconsistency out of the published IG without
blocking on a larger refactor, and it keeps the ticket's editorial intent
intact.
