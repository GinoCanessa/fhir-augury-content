## Summary

The reporter (Jeroen Medema) flags a documentation-accuracy defect in the
ImagingStudy "DICOM Tag Mapping" table (section 10.4.9.5). The row for
`ImagingStudy.numberOfSeries` currently labels its DICOM source as "DICOM
instance metadata", but the value actually originates from DICOM
study-level query/retrieve metadata. The row should be corrected to refer
to "DICOM study metadata" (or be otherwise rephrased to make clear that
the value comes from a query/retrieve attribute rather than from any tag
embedded in an instance dataset). No cross-referenced Jira tickets,
Zulip threads, or GitHub items were located for this ticket.

## Details

**Description:**

10.4.9.5 DICOM Tag Mapping ([https://dicomstandard.org/current](https://www.dicomstandard.org/current))

The row `numberOfSeries` states "DICOM instance metadata" but should refer

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the "DICOM Tag Mapping" table in the ImagingStudy resource page
(section 10.4.9.5) and change the source-column entry on the
`numberOfSeries` row from "DICOM instance metadata" to "DICOM study
metadata". Make no other edits to the row (DICOM tag reference,
cardinality, transformation notes) and no edits to surrounding rows.

#### Justification

This is the literal, minimal correction the reporter asked for and
matches `jwhitby`'s explanation that the value comes from study-level
metadata rather than from any tag inside an instance. It is a pure
documentation fix: no schema, profile, or terminology change, no impact
on validators, and no implementer-visible breakage. It is the lowest-risk
way to remove an objectively misleading statement in the published
specification.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only swapping "instance" for "study" on the `numberOfSeries`
row, sweep the entire 10.4.9.5 DICOM Tag Mapping table and rephrase the
"source" column so it consistently distinguishes between (a) values
sourced from DICOM tags inside an instance dataset, (b) values sourced
from study- or series-level query/retrieve attributes, and (c) values
derived/aggregated by the server. Apply the corrected wording to
`numberOfSeries`, `numberOfInstances`, and any other rows where the
current label misrepresents a query/retrieve attribute as an "instance
metadata" tag (Jeroen's comment explicitly notes "the same holds for a
few other elements too"). Add a short lead-in paragraph defining the
three source categories so future rows have a clear template.

#### Justification

The reporter and `jwhitby` both indicate that the underlying problem is
a category label that does not actually distinguish query/retrieve
attributes from in-dataset DICOM tags, and that more than one row is
affected. Fixing only the one row Jeroen happened to call out leaves the
same defect in place elsewhere and invites a follow-up ticket. A
one-pass cleanup of the table — same change window, same reviewers — is
proportionate to the issue and produces a more useful artifact for
implementers, at the cost of a slightly larger diff and the need to
audit the other rows.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" / "No Change" on the grounds that the column
heading already implies that the listed sources are descriptive (the
table is a mapping aid, not a normative statement about where any given
DICOM system stores the value), and that "DICOM instance metadata" can
be read broadly enough to cover study-level attributes returned via
query/retrieve.

#### Justification

This disposition is defensible only if the WG concludes that the
existing wording is, on a strict reading, not actually wrong and that
the cost of any spec edit (re-publication, re-review of adjacent rows)
outweighs the clarity benefit. Both the reporter and the original
author of the row (jwhitby) have already conceded on-ticket that
rephrasing is warranted, which materially weakens this position.

---

### Recommendation

**Recommended disposition:** B

The on-ticket exchange between the reporter and `jwhitby` makes it clear
that (1) the current wording on `numberOfSeries` is misleading, (2) the
intent of the row is to flag a query/retrieve attribute rather than an
in-dataset DICOM tag, and (3) the same mislabelling affects more than
one row. Disposition A fixes only the single row the reporter happened
to file against and predictably triggers another ballot/tracker cycle
when the next reader notices the same defect on a sibling row.
Disposition C is hard to defend given that both parties on the ticket
already agree some rephrasing is warranted. Disposition B costs a
slightly larger diff and a one-time audit of the table, but it resolves
the defect class — not just the one instance — and gives future
contributors a clear template for the "source" column. The work group
should, as part of accepting B, decide whether to (a) keep
`numberOfSeries` in the table at all if it is purely a query/retrieve
convenience, or (b) keep it but explicitly tag its provenance.
