## Summary

The reporter is reviewing the DICOM SR to FHIR Resource Mapping IG and asks
for clarification on the `ImagingMeasurement` Observation mapping table.
Specifically, the diagram indicates that the value of `bodyStructure` can
be a "parent BodyStructureFindingSite", and the reporter wants confirmation
on whether that "parent" refers to the `BodyStructureFindingSite` instance
defined in the Group Mapping section, or to a different artifact. The
ticket asks for the IG text/diagram to make the cross-reference explicit
so implementers can resolve the reference unambiguously.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the data sources, so the request is being treated on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the DICOM SR to FHIR Resource Mapping IG so the `ImagingMeasurement`
Observation mapping table (and the accompanying diagram) explicitly
identifies which `BodyStructureFindingSite` is meant when the
`bodyStructure` value is described as "parent BodyStructureFindingSite".
Concretely:

- In the `ImagingMeasurement` mapping table row for `bodyStructure`,
  replace the bare phrase "parent BodyStructureFindingSite" with a
  hyperlinked, fully qualified reference to the
  `BodyStructureFindingSite` profile/instance defined in the Group
  Mapping section (e.g., "the `BodyStructureFindingSite` resource
  instantiated by the parent Group Mapping — see [Group Mapping →
  BodyStructureFindingSite]").
- Add the same explicit cross-reference to the diagram caption / legend
  so the relationship is visible without reading the surrounding prose.
- Add a one-sentence note in the IG explaining the parent/child
  containment model: an `ImagingMeasurement` Observation inherits the
  `bodyStructure` from the `BodyStructureFindingSite` produced by its
  enclosing Group Mapping unless overridden locally.

#### Justification

The reporter's resolution was already recorded as **Persuasive**, which
indicates the workgroup agreed the IG was ambiguous on this point. The
fix is editorial — no normative change to mappings, cardinalities, or
profiles is required — and it directly removes a real source of
implementer confusion identified by an external reviewer. Explicit
cross-references between mapping tables and the artifacts they consume
are a standard FHIR IG authoring convention and improve traceability
between the SR concept (parent finding site) and its FHIR
representation.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only patching the wording in the `ImagingMeasurement` row,
restructure the IG so the `BodyStructureFindingSite` is defined once as
a named, anchored artifact in a dedicated "Shared Mappings" or
"Reusable Targets" section, and have both the Group Mapping and the
`ImagingMeasurement` Observation mapping link to that single
definition. The `ImagingMeasurement.bodyStructure` row would then read
"reference to the [BodyStructureFindingSite](#shared-mappings-bsfs)
established by the enclosing Group Mapping". The diagram would be
re-rendered to show the shared node with inbound arrows from both the
Group Mapping and the `ImagingMeasurement` Observation.

#### Justification

The same ambiguity is likely to recur wherever the IG re-uses
`BodyStructureFindingSite` (or any other inherited finding-site
concept). Centralising the definition removes the "which one?" question
class-wide instead of one row at a time, and makes future additions
(e.g., other measurement-style Observations that inherit body site)
consistent by construction. The trade-off is more editorial churn now
and a slightly larger IG re-render than Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the IG already states (in the
Group Mapping section) how `BodyStructureFindingSite` is constructed
and that the word "parent" in the `ImagingMeasurement` table is meant
in the SR-tree sense (the enclosing container), so no IG change is
required. Respond to the reporter with a clarifying note pointing at
the Group Mapping section.

#### Justification

If the workgroup believes the existing prose around the diagram is
sufficient for an SR-literate reader, then the request can be answered
with guidance rather than a spec edit. This avoids re-rendering the
diagram and re-balloting editorial text.

---

### Recommendation

**Recommended disposition:** A

The reporter is an external implementer who could not resolve the
reference from the IG as written, and the workgroup has already marked
the resolution **Persuasive** — that is effectively a pre-commitment to
a clarifying edit, so Disposition C is inconsistent with the existing
disposition. Disposition B is the cleaner long-term structure but is a
larger editorial undertaking and may not be in scope for the cycle that
produced this ticket; it can be tracked as a follow-up if the WG sees
the same ambiguity recurring elsewhere. Disposition A delivers exactly
what the reporter asked for, matches the recorded resolution, and is
low-risk because it changes only narrative/diagram text, not mappings
or profiles. If the WG wants to reduce future occurrences without
committing to a full restructure, Disposition A can be applied now and
Disposition B opened as a follow-on tracker.
