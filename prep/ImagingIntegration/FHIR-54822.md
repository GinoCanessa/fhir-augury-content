## Summary

The reporter (Ana Kostadinovska) asks the Imaging Integration WG to update
the ImagingSelection "Imaging Selection Subsets" notes section so that
`imageRegion2D` is listed alongside the other options the section
enumerates. The section was authored under [FHIR-47654](https://jira.hl7.org/browse/FHIR-47654)
("ImagingSelection - subset", Applied) and currently enumerates four
defined types of `instance.subset`: Frames, Referenced Content Item
Identifiers, Segments, and ROIs — keyed off `instance.sopClass`. The
ticket reports that `imageRegion2D` is omitted from this enumeration
even though it is "a valid option for imaging selection subset."

The request lands in the middle of an active R6 clean-up of this same
Notes block (label `R6-Notes-ImagingSelectionSubsets` is shared with
[FHIR-54824](https://jira.hl7.org/browse/FHIR-54824),
[FHIR-54826](https://jira.hl7.org/browse/FHIR-54826) and
[FHIR-54830](https://jira.hl7.org/browse/FHIR-54830)). FHIR-54826 (Persuasive,
applied via HL7/fhir commit `c429239`) already removed an incorrect
sentence from the same notes section, and FHIR-54830 (Persuasive, applied
in the same commit) added a note to §10.5.5.3.3 / §10.5.5.3.4 stating that
`imageRegion2D` and `imageRegion3D` are mutually exclusive. The
broader effort [FHIR-52900](https://jira.hl7.org/browse/FHIR-52900) is
splitting `instance.subset` into multiple, mutually-exclusive elements,
which may overtake the precise wording targeted here. Importantly,
`imageRegion2D` is a sibling element of `instance.subset` (not one of
the SOP-class-determined subset types), so the request can be read in
two ways: (a) literally add `imageRegion2D` to the enumerated subset
list, or (b) clarify the notes so a reader does not perceive
`imageRegion2D` as missing from the family of "ways to identify a
sub-portion of an instance".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the "Imaging Selection Subsets" notes block on
`imagingselection.html` (rendered §10.5.5.1; authored under
`source/imagingselection/` in `HL7/fhir`) to add `imageRegion2D` as a
fifth bullet in the enumerated list of defined subset types — alongside
Frames, Referenced Content Item Identifiers, Segments, and ROIs — and
add a corresponding `e.g.` example tying a 2D-region selection to a
representative SOP Class UID (e.g., a 2D image SOP Class). Also extend
the closing note about "each imaging selection instance can only contain
one type of subset" to include `imageRegion2D` so it does not become
inconsistent.

#### Justification

This is the literal reading of the reporter's request and matches the
WG's recent pattern of taking R6 ballot notes work directly into the
same paragraph (FHIR-54826 / FHIR-54830, both Persuasive, both applied
via commit `c429239`). It keeps the disposition trail with the reporter
(Ana Kostadinovska) consistent across her four sibling tickets and
removes the perceived omission in one editing pass.

### Disposition B: Alternative Approach

#### Proposal

Do not add `imageRegion2D` to the enumerated subset list. Instead, edit
the same "Imaging Selection Subsets" notes block to (1) state up front
that the section describes the values that may appear in the
`instance.subset` element specifically, and (2) add a closing
cross-reference paragraph that names `imageRegion2D` and `imageRegion3D`
as the *separate* sibling elements used to identify a 2D / 3D region
within an image instance, with a forward link to §10.5.5.3.3 /
§10.5.5.3.4 (the subsections that FHIR-54830 just clarified). Coordinate
the wording with the in-flight FHIR-52900 split so the change survives
that restructuring. Edit the authored notes fragment under
`source/imagingselection/` in `HL7/fhir`.

#### Justification

`imageRegion2D` is structurally a sibling of `instance.subset`, not a
value of it: per FHIR-47654's applied resolution the four subset types
are determined by `instance.sopClass`, and per FHIR-48612 the
`imageRegion2D` element is independently described under its own
Definition / Requirements. Inserting it into the SOP-class-keyed subset
enumeration would fold two distinct modelling concepts into one list
and undo editorial decisions made in FHIR-47654. The reporter's
phrasing ("a valid option for imaging selection subset") is best read
as a reader-facing gap — the Notes section reads as if `imageRegion2D`
has been forgotten — and the right fix is to disambiguate, not to
re-classify. This also resolves the "is the Notes block about the
`subset` element or about subsetting in general?" ambiguity that
underlies FHIR-54822, FHIR-54824, and FHIR-54826 collectively. The
HL7/fhir briefing flags FHIR-52900 (PR #3729) as restructuring the
subset element family on branch `imaging-202510`; coordinating this
edit with that branch addresses the merge-conflict gotcha called out
in the briefing.

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive** on the grounds that `imageRegion2D` is
not a value of `ImagingSelection.instance.subset` and therefore does
not belong in the "Imaging Selection Subsets" enumeration. Note in the
disposition that the underlying need — making readers aware that
`imageRegion2D` exists alongside `subset` — is already satisfied by
FHIR-54830's added note in §10.5.5.3.3 / §10.5.5.3.4 (commit
`HL7/fhir@c429239`) and by FHIR-48612's Definition / Requirements
restructure (PR `HL7/fhir#3319`).

#### Justification

The four-bullet enumeration in the Notes section was deliberately keyed
off `instance.sopClass` (FHIR-47654 resolution); `imageRegion2D` is not
keyed off SOP class and is modelled as a separate, mutually-exclusive
sibling element. Adding it to the enumeration would introduce a
modelling inconsistency that downstream implementers would have to
unwind. Existing R6 changes (FHIR-54830, FHIR-48612) already give
`imageRegion2D` first-class treatment in the prose, so the reader-facing
gap is arguably already closed.

### Recommendation

**Recommended disposition:** B

Disposition B is the cleanest fit for the facts on the ground. The
reporter's other tickets in the `R6-Notes-ImagingSelectionSubsets`
batch (FHIR-54826, FHIR-54830) were Persuasive and applied, so flat
declining FHIR-54822 (Disposition C) ignores a real reader-facing
problem the same WG has been actively addressing. But Disposition A
quietly reverses FHIR-47654's deliberate scoping of the Notes block to
SOP-class-keyed subset types and would be undone almost immediately by
FHIR-52900's restructure of `instance.subset`. Disposition B captures
the reporter's underlying intent — make `imageRegion2D` visible from
the subset notes — without conflating two distinct modelling concepts,
and it slots cleanly into the in-flight `imaging-202510` branch the
briefing flags. The WG should also consider rolling FHIR-54824 into
the same edit pass, since it touches the same paragraph.
