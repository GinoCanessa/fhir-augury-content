## Summary

The ticket reports that the *DICOM SR to FHIR Resource Mapping IG* leaves the
handling of **concept modifiers** unspecified for two key profiles:
`Observation - DICOM SR Imaging Measurement Mapping to Observation`
(`imaging-measurement`) and `Observation - DICOM SR Imaging Qualitative
Evaluation Mapping to Observation` (`imaging-qualitative-evaluation`). DICOM SR
templates such as **TID 300** (measurements) and **TID 1501** (qualitative
evaluations, rows 11 & 11b) permit one or more concept modifiers attached to a
coded measurement or evaluation, but the current IG mappings give no guidance
on where those modifiers should land in the resulting `Observation`.

The submitter (Harald Zachmann, via reporter Jonathan Whitby) proposes that
each concept modifier be mapped to an entry in `Observation.component`, using
the modifier's concept as `component.code` and its value as the
`component.value[x]`. This naturally accommodates an unbounded number of
modifiers per observation. The submitter asks whether there is any reason not
to adopt this approach and document it in the IG.

No related Jira tickets, Zulip discussions, or GitHub items were surfaced for
this ticket via cross-reference, search, or related-item queries.

## Proposed Dispositions

> Note: with no cross-referenced GitHub repository, file-path-level proposals
> below cite the canonical IG source location (`HL7/dicom-sr` build URLs) and
> the affected profile artifacts (`imaging-measurement`,
> `imaging-qualitative-evaluation`). A repo briefing for `HL7/dicom-sr` would
> let a future revision pin authoring paths exactly; if the workgroup wants
> that, run `repo-analysis HL7/dicom-sr` and re-prep the ticket.

### Disposition A: Accept As Requested

#### Proposal

Adopt the submitter's proposal: map each DICOM SR **concept modifier**
attached to a coded measurement (TID 300) or a qualitative evaluation (TID
1501 rows 11 & 11b) to a separate entry in `Observation.component`, where:

- `Observation.component.code` ← the concept modifier's *concept name* coding
  (typically a DCM/SCT code from the modifier's "concept name" attribute).
- `Observation.component.value[x]` ← the concept modifier's value, choosing
  the FHIR datatype that matches the SR value type (e.g. `valueCodeableConcept`
  for CODE-typed modifiers, `valueQuantity` for NUM, `valueString` for TEXT,
  etc., consistent with how the IG already maps the parent measurement value).

Update both profiles in the *DICOM SR to FHIR Resource Mapping IG*:

- `StructureDefinition-imaging-measurement` (Observation profile for TID 300
  numeric measurements).
- `StructureDefinition-imaging-qualitative-evaluation` (Observation profile
  for TID 1501 row 11 / 11b coded evaluations).

For each profile, add:

1. Narrative guidance in the profile intro describing the concept-modifier →
   `component` mapping rule, including how to populate
   `component.code.coding.system` for DCM vs. SCT modifier concepts and how to
   pick the matching `value[x]` choice.
2. A worked example (one for measurement, one for qualitative evaluation)
   showing one or more concept modifiers serialised into the `component`
   array.
3. Either an updated `mapping` element on `Observation.component` in the
   StructureDefinition pointing at the relevant DICOM SR template rows, or a
   row added to the existing concept map / mapping table that the IG renders.

Leave `component`'s cardinality (`0..*`) untouched — the existing slot already
accepts an unbounded number of modifiers.

#### Justification

`Observation.component` is exactly the FHIR-defined extension point for
"additional information related to the observed value … that share the same
attributes as the primary value" (Observation invariant `obs-7` / component
guidance). Concept modifiers in DICOM SR semantically qualify the parent
measurement or evaluation (e.g. *finding site*, *laterality*, *measurement
method*) — i.e. the same data shape FHIR's `component` is designed for. Reusing
`component` keeps the mapping consistent with how other FHIR IGs (Vital Signs,
US Core BMI, US Core Pulse Oximetry) attach related coded qualifiers to a
primary value, and avoids inventing a parallel mechanism for what is, in FHIR
terms, an already-solved problem. It also preserves the unbounded cardinality
that TID 300 / TID 1501 require.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (only) overloading `Observation.component`, document a tiered
mapping that distinguishes *intrinsic context* modifiers from *qualifying
value* modifiers:

1. **Intrinsic-context modifiers** with first-class FHIR slots map to those
   slots, not to `component`. Examples:
   - *Finding Site* → `Observation.bodyStructure` (R5) or
     `Observation.bodySite` plus the existing
     `bodyStructure`/`Observation-bodyPosition` extension pattern already used
     elsewhere in the IG;
   - *Laterality* → `Observation.bodySite` qualifier or the
     `http://hl7.org/fhir/StructureDefinition/bodySite` extension on the body
     site code;
   - *Measurement Method* → `Observation.method`;
   - *Subject Relationship Context* → `Observation.focus`.
2. **Residual / qualifying modifiers** — those without a dedicated
   `Observation.*` element — map to `Observation.component` exactly as
   Disposition A describes.
3. Add an explicit *concept-modifier routing table* to the IG that lists, by
   DICOM concept name code, the target FHIR element. Modifiers not in the
   table fall through to `component`.

Update both `imaging-measurement` and `imaging-qualitative-evaluation`
profiles to reflect this routing, and add narrative + at least one worked
example per category.

#### Justification

Mapping every concept modifier indiscriminately to `component` will create
`Observation` instances where queryable, validated information (body site,
method, laterality) is hidden inside a generic code/value pair, defeating the
clinical query patterns FHIR consumers rely on (e.g. "find all measurements
on the left kidney"). Concept modifiers in DICOM SR are not all equal — some
correspond directly to first-class `Observation` elements, and the IG already
maps a few of them that way for the primary measurement. Extending that
pattern is more work than Disposition A but yields a mapping that downstream
implementers can actually search against, and avoids divergence between
"primary measurement context" and "modifier context" within the same
`Observation`. The fallback to `component` ensures completeness for the
long tail of TID-300/1501 modifiers that have no natural FHIR home.

---

### Disposition C: Decline

#### Proposal

Do not add a normative mapping for concept modifiers in this revision.
Instead, mark the behaviour as **out-of-scope** for the current ballot of the
*DICOM SR to FHIR Resource Mapping IG* and add a single sentence to each of
the two profile intros stating that mapping of TID 300 / TID 1501 concept
modifiers is implementation-defined pending future work, and pointing
implementers at `Observation.component` as a permitted but not required
landing spot.

#### Justification

This is the weakest of the three options and is included for completeness.
The DICOM SR templates in question are widely implemented and the mapping
gap is real, so leaving it implementation-defined effectively forces every
implementer to invent the same wheel — and to invent it differently, which
is precisely what the IG exists to prevent. The only defensible reason to
decline is if the workgroup believes the modifier set is too large or
heterogeneous to map normatively in the current ballot window; even then,
the right move is usually B (a routing table that can grow) rather than C.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the submitter's underlying request — close the
documentation gap so implementers know where to put concept modifiers — while
preserving FHIR's first-class semantics for body site, laterality, method,
and other context attributes that already have dedicated `Observation`
elements. It is a strict superset of Disposition A: the long-tail modifiers
still land in `Observation.component`, exactly as the submitter asked, but
clinically queryable context isn't buried there. The cost is a one-time
authoring effort to produce the routing table, which the Imaging Integration
WG is well-positioned to do because the same modifier-to-FHIR-element
decisions have already been made (informally) for the primary measurement
mapping in these profiles. Disposition A is the acceptable fallback if WG
bandwidth is tight; Disposition C should be rejected because it preserves
the very ambiguity the ticket is asking to resolve.
