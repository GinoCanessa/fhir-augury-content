## Summary

The reporter asks the Imaging Integration work group to tighten the
wording in the ImagingSelection narrative that describes how
`instance.frameNumber` is used when an Enhanced CT image is selected.
The current text says that, when `instance.sopClass` equals the Enhanced
CT SOP Class UID (`urn:oid:1.2.840.10008.5.1.4.1.1.2.1`), the instance
is multiframe and therefore `instance.frameNumber` *can* be included to
specify a list of frames. The reporter contends that "can be included"
understates the requirement — for a multiframe Enhanced CT instance,
identifying the selected frames is the whole point of populating the
subset, so the prose should read "should be included" (or similar
mandatory-on-condition language). The change is editorial / narrative
only; no resource definition, cardinality, binding, or invariant is
proposed to change. No cross-referenced Jira tickets, Zulip discussions,
or GitHub items were surfaced by FHIR Augury for this ticket.

## Details

**Description:**

> "*If instance.sopClass has a value of urn:oid:1.2.840.10008.5.1.4.1.1.2.1, this indicates that the instance is Enhanced CT, which is a multiframe image instance and therefore instance.frameNumber can be included to specify a list of frames*". Here, "*can be included to specify a list of names*" I think should be "*should be included…*". PLease update.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the editorial fix exactly as the reporter requests in the
ImagingSelection notes narrative. In the bullet describing
`instance.sopClass = urn:oid:1.2.840.10008.5.1.4.1.1.2.1` (Enhanced
CT), change:

> "…and therefore `instance.frameNumber` **can** be included to specify
> a list of frames"

to:

> "…and therefore `instance.frameNumber` **should** be included to
> specify a list of frames"

This is a single-word change in the ImagingSelection narrative prose
("Use of ImagingSelection.subset" / Enhanced CT bullet). It does not
alter the StructureDefinition, cardinalities, bindings, search
parameters, examples, or invariants for ImagingSelection. The label
`R6-Notes-ImagingSelectionSubsets` indicates this is being tracked as
an R6 ballot note resolution against the subset narrative, which lines
up with an editorial-only change.

#### Justification

The reporter's reading is correct in spirit: when an ImagingSelection
points at an Enhanced CT (multiframe) instance, the entire purpose of
`ImagingSelection.instance.subset` is to identify *which* frames are
being selected. Saying `frameNumber` *can* be included makes the
multiframe-frame-selection case sound optional in a context where
omitting it would defeat the use case. "Should" matches FHIR's normal
usage for strong but non-machine-enforceable narrative guidance and is
consistent with how the sibling Enhanced SR bullet immediately below
describes `instance.referencedContentItemIdentifier`. The change is
backwards compatible (no schema impact) and carries essentially zero
implementer cost.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the narrative but rewrite the bullet to be conditional rather
than swap a single modal verb. Replace the sentence with something
like:

> "If `instance.sopClass` has a value of
> `urn:oid:1.2.840.10008.5.1.4.1.1.2.1` (Enhanced CT — a multiframe
> image SOP Class), `instance.frameNumber` **SHALL** be populated when
> the selection refers to a specific subset of frames; senders that
> intend to reference the entire multiframe instance MAY omit
> `instance.frameNumber`."

Apply the parallel rewrite to the Enhanced SR bullet for
`instance.referencedContentItemIdentifier`. Optionally back the
narrative with an invariant on `ImagingSelection.instance` such as
"when `sopClass` identifies a multiframe SOP Class and the selection
is intended to denote specific frames, `frameNumber` is present."

#### Justification

This addresses the reporter's underlying concern — that "can" is too
permissive — but distinguishes the two real cases: selecting *some*
frames (where `frameNumber` is genuinely required to be useful) versus
selecting the *entire* multiframe instance (where omitting
`frameNumber` is legitimate). It also keeps the Enhanced CT and
Enhanced SR bullets symmetrical in their conformance language. The
trade-off is that this is a larger editorial change than the reporter
asked for, may need a quick sanity check from DICOM-side reviewers,
and an invariant (if added) widens the scope from a pure narrative fix
to a normative artifact change against R6 — heavier process for what
is essentially the same intent as Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave "can be included" in place. Rationale
category: existing wording already conveys the intent given the
surrounding context, and the workgroup considers "should" too strong
because there are legitimate Enhanced-CT selection cases (e.g., a
selection that designates the entire multiframe instance with no
frame-level subset) where `frameNumber` is intentionally absent.

#### Justification

`ImagingSelection.instance.frameNumber` has cardinality `0..*`, and
the spec deliberately allows an instance-level selection without
naming individual frames. A bare modal swap from "can" to "should"
risks implying a near-mandatory expectation in cases where omitting
`frameNumber` is correct. Where the surrounding narrative already
explains that `frameNumber` exists "to specify a list of frames", a
reader who actually wants to select a list of frames will populate it;
no normative gap exists. The R6 ballot-notes label suggests this can
be resolved as "considered — no change" with a brief note explaining
the optional-by-design behaviour.

---

### Recommendation

**Recommended disposition:** A

This is a one-word editorial fix in a narrative bullet, on a ticket
explicitly tagged `R6-Notes-ImagingSelectionSubsets`, with no
cross-cutting impact, no breaking change, no normative-artifact churn,
and no implementer burden. The reporter's reading is consistent with
how the Enhanced SR bullet that immediately follows is phrased and
with FHIR's normal use of "should" for strong narrative guidance.
Disposition B captures a real nuance (whole-instance selection vs.
frame-subset selection) and is worth keeping in mind, but it expands
the scope beyond what the ticket requests and — if it grows an
invariant — pushes a trivial editorial item into normative territory
late in R6. Disposition C is hard to defend: the existing wording is
weak enough that a community reporter has already filed a ticket
against it, and the cost of the fix is negligible. Accept as
requested; if the work group wants the Disposition B nuance, capture
it as a follow-up ticket rather than blocking this one.
