## Summary

Lloyd McKenzie, on behalf of the January 29, 2021 CDISC call, asks the
**CDISC Lab (FHIR)** Implementation Guide to add narrative guidance about
how implementers should handle CDISC *supplemental variables* (a.k.a.
SUPPQUAL / supplemental qualifiers). Specifically the IG should explain
two related points:

1. When the IG provides **no explicit FHIR-element mapping** for a piece
   of CDISC information, that information can still be carried into a
   study as a supplemental variable when it is study-relevant.
2. **CDISC publishes a registry of "standard" supplemental variables**,
   and the IG currently does **not** include mappings to those registered
   variables — readers should be aware of this gap.

The cross-reference graph is empty for this key (no incoming or outgoing
links of any kind were recorded), but several BR&R / CDISC tickets
surface as semantically related and corroborate the request:

- [FHIR-30536](https://jira.hl7.org/browse/FHIR-30536) (Revisit sex
  mapping) explicitly suggests that ambiguous gender information *"can
  sometimes be conveyed as supplemental qualifiers"* — exactly the
  pattern this ticket asks the IG to document generally.
- [FHIR-30528](https://jira.hl7.org/browse/FHIR-30528) (Many CDISC AE
  variables not included in the mapping) and
  [FHIR-30538](https://jira.hl7.org/browse/FHIR-30538) (Should be more
  DM mappings) reflect the same underlying gap: implementers keep
  hitting CDISC variables the IG does not yet map.
- [FHIR-50469](https://jira.hl7.org/browse/FHIR-50469) (Need guidance on
  supplemental guides) is a different IG (US Da Vinci PAS) but reinforces
  the broader pattern that "supplemental" handling needs explicit
  guidance text.

The reporter has already labelled the ticket **Ready-For-Vote** and the
working resolution is **Persuasive**, so the workgroup discussion here
is about the *form* of the guidance text, not whether to act.

## Proposed Dispositions

Note: the working resolution is already **Persuasive** and the ticket
is labelled **Ready-For-Vote**, so all three dispositions below are
framed for the BR&R workgroup vote rather than as open-ended options.

### Disposition A: Accept As Requested

#### Proposal

Add a new short narrative subsection to the CDISC Lab (FHIR) IG —
recommended location: a "Handling unmapped data" or "Supplemental
variables" section under the IG's Mapping Guidance / Conformance page —
that states, in roughly this form:

> **Handling unmapped data with supplemental variables.** When this
> implementation guide does not define an explicit FHIR mapping for a
> piece of CDISC information, implementers may still convey that
> information into a study as a CDISC *supplemental variable*
> (SUPPQUAL) when the information is relevant to the study. CDISC
> additionally publishes a registry of *standard* supplemental
> variables that implementers may wish to leverage for cross-study
> consistency. **This implementation guide does not currently define
> mappings to variables from that registry**; doing so is a candidate
> for a future version. Implementers using standard supplemental
> variables should follow the registry definitions and document the
> mapping locally until a normative mapping is added here.

No structural changes (no new resources, profiles, extensions, value
sets, or constraints) are needed. This is a narrative-only change.

#### Justification

The request is exactly what was discussed and minuted on the Jan 29,
2021 CDISC call (per Lloyd McKenzie's comment), the resolution is
already marked Persuasive, and the change is purely additive guidance.
The two bullets in the description map cleanly onto two paragraphs of
narrative; there is no design controversy to resolve. The pattern of
"information that can't be cleanly mapped is appropriate to convey via
suppqual" is already used in disposition text for sibling ticket
[FHIR-30536](https://jira.hl7.org/browse/FHIR-30536), so the
terminology and framing are consistent with prior BR&R decisions.

---

### Disposition B: Alternative Approach — Accept with Modification (cross-IG note + future-mapping marker)

#### Proposal

Adopt the narrative from Disposition A, but also:

1. Place the same (or a near-identical) supplemental-variables
   narrative into a shared location — either the CDISC Mappings (FHIR)
   IG's Mapping Guidance page, or a small reusable narrative fragment
   that both IGs include — so the guidance does not have to be
   re-litigated each time a new CDISC-aligned IG is published, and so
   tickets like [FHIR-30528](https://jira.hl7.org/browse/FHIR-30528)
   and [FHIR-30538](https://jira.hl7.org/browse/FHIR-30538) get a
   single canonical answer.
2. Add an explicit "Known mapping gaps" subsection (or task list /
   future-work note) to the Lab IG that names the CDISC standard
   supplemental-variable registry as a deferred-but-tracked item, with
   a tracker reference back to this ticket and to the existing
   coverage-gap tickets in the related set. This makes the
   "doesn't currently cover" admission visible and actionable rather
   than a single-paragraph aside.

#### Justification

The literal request (Disposition A) only adds narrative inside one IG.
Because the same gap reappears in CDISC Mappings (FHIR) tickets
(FHIR-30528, FHIR-30538, FHIR-30536), placing the suppqual narrative in
a shared/canonical location avoids divergent wording across the CDISC
IG family and reduces future maintenance. Naming the registry-mapping
gap explicitly (rather than just mentioning it) gives the workgroup
something to point at in subsequent ballots and makes prioritisation of
follow-on mapping work easier. Trade-off: slightly more editorial work
now and a small cross-IG coordination cost vs. Disposition A.

---

### Disposition C: Decline

#### Proposal

Reject the change on the grounds that supplemental-variable handling is
a CDISC concept already documented by CDISC, and the FHIR IG should not
restate normative CDISC guidance. Close as *Not Persuasive* and direct
implementers to the CDISC SDTM Implementation Guide and the CDISC
supplemental-variable registry directly.

#### Justification

Defensible only on a strict "scope discipline" reading: the IG defines
FHIR↔CDISC mappings, and how a CDISC consumer fills in unmapped data is
arguably a CDISC-side concern rather than a FHIR-side concern.
Restating CDISC guidance risks drift if the registry or SUPPQUAL
mechanics evolve. **However**, this disposition runs against the
already-recorded Persuasive resolution and the explicit Jan 29 CDISC
call decision, and it leaves implementers without a clear in-context
answer for the recurring "what do I do when there's no mapping?"
question that drives the surrounding ticket cluster. Included here for
completeness.

---

### Recommendation

**Recommended disposition: B (Accept with Modification).**

Disposition A is the safe, minimal vote and is fully consistent with
the existing Persuasive resolution. However, the related-ticket cluster
(FHIR-30528, FHIR-30536, FHIR-30538, and the conceptually adjacent
FHIR-50469) shows that the same "unmapped CDISC variable → use
suppqual" question has already had to be answered piecemeal across
multiple BR&R tickets and across at least two IGs (CDISC Lab and CDISC
Mappings). Spending a small amount of additional editorial effort now
to (a) state the suppqual narrative in a shared/canonical location and
(b) explicitly mark the standard-supplemental-variable registry as a
known, tracked mapping gap will let the workgroup answer future tickets
in this family with a pointer rather than a fresh discussion, and will
make the IG's coverage limitations more honest and more actionable for
implementers. Disposition C is not recommended; it would contradict the
already-recorded Persuasive resolution and the Jan 29 CDISC call
decision.
