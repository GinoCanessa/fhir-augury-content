## Summary

The reporter (Ricardo Quintano) observes that the *Scope and Usage* section
of the `MedicationStatement` resource page in FHIR R6 ballot 4
(`medicationstatement.html#scope`) jumps straight into a clarifying paragraph
about `MedicationStatement.adherence` — explaining that adherence is specific
to a single `MedicationStatement` instance and that adherence over time is
expressed via multiple instances bounded by `effectivePeriod` — without first
introducing what `MedicationStatement.adherence` is or what it is for. The
ticket requests that an introductory sentence (or short paragraph) describing
the purpose of `MedicationStatement.adherence` be prepended so that the
existing clarifying paragraph reads in context. This is a documentation /
narrative change to the resource page; no schema, cardinality, or binding
change is requested. No other Jira tickets are cross-referenced from this
issue.

## Details

**Description:**

> In the scope and usage section at
> https://hl7.org/fhir/6.0.0-ballot4/medicationstatement.html#scope, there is
> the following paragraph:
>
> > "MedicationStatement includes an adherence element. Note that this
> > adherence is specific to that instance of MedicationStatement. If
> > MedicationStatement.adherence is being tracked over time, then instances
> > of MedicationStatement would report adherence for the interval noted in
> > effectivePeriod."
>
> I suggest adding an introductory explanation describing the purpose of
> MedicationStatement.adherence. This would help readers better understand
> the paragraph and the intended use of the element.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the *Scope and Usage* section of the `MedicationStatement` resource
page (the source for `medicationstatement.html#scope` in HL7/fhir) to
prepend an introductory sentence (or short paragraph) before the existing
"MedicationStatement includes an adherence element…" text. The new
introduction should state, in plain language, what `MedicationStatement.adherence`
is (a structured assessment of whether the patient is taking the
medication as prescribed), why the resource carries it (so that
adherence can be recorded alongside the statement of use rather than in
a separate observation), and that the adherence assessment applies to
the medication use described by *this* `MedicationStatement` instance —
which then makes the existing paragraph about per-instance scope and
`effectivePeriod` flow naturally. The change is narrative-only: no
element, cardinality, binding, or invariant changes; only the
introductory text is added.

#### Justification

The reporter's observation is correct on a plain reading of the page:
the existing paragraph presupposes the reader already knows what
`MedicationStatement.adherence` is, and only clarifies a corner case
(per-instance scope, tracking over time via `effectivePeriod`). Adding
an introduction is a low-risk, high-value editorial fix that improves
implementer comprehension and is consistent with how other key elements
(e.g. `status`, `effectivePeriod`) are introduced in adjacent narrative
on the same page. The Pharmacy WG already maintains the resource
narrative, and an "improve scope text" change is precisely the kind of
edit that "Technical Correction"-typed tickets exist for. The Zulip
evidence (recurrent confusion about adherence vs. status across
multiple streams) reinforces that even seasoned implementers benefit
from an explicit definition in the Scope and Usage section.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a new sentence in *Scope and Usage*, expand and
clarify the **element-level definition** of `MedicationStatement.adherence`
itself (the `definition` and `comments` text on the element in the
StructureDefinition / `medicationstatement-definitions.html` page), and
add a short cross-reference from *Scope and Usage* to that element
definition (e.g. "see the definition of `MedicationStatement.adherence`
below"). Keep the existing per-instance / `effectivePeriod` clarifying
paragraph in *Scope and Usage* as it is. Ensure the element definition
explains the purpose of the element (recording assessed adherence for a
given statement of medication use), the relationship to
`effectivePeriod`, and — where helpful — the distinction from R4
`MedicationStatement.status`.

#### Justification

The element definition is the authoritative location implementers see
when they hover the element name, generate a logical model, or render
the resource tree. Strengthening the definition there benefits every
downstream rendering (tree, snapshot, differential, IG profile pages),
not just the prose of one section. *Scope and Usage* would still get a
small pointer so the reader on that page is led to the explanation. The
trade-off vs. Disposition A is that the change touches an element
definition (which is more visible and slightly more invasive in
diff/regen terms) instead of being a pure narrative tweak; on the other
hand, the benefit propagates beyond the single page that the reporter
cited.

---

### Disposition C: Decline

#### Proposal

Decline the request as already addressed by existing material: the
`MedicationStatement.adherence` backbone element carries its own
`definition` and `comments` in the resource definition table on the
same page, and the *Scope and Usage* paragraph is intentionally a
clarifying note about a known per-instance / temporal subtlety — not a
definition. Recommend the reporter consult the element definition
table for the purpose of the element. No spec change.

#### Justification

The Scope and Usage section's role is to highlight non-obvious
modelling choices, not to redefine elements that already have formal
definitions elsewhere on the same page. Declining avoids editorial
churn during ballot reconciliation for a change the reporter could
satisfy by reading the adjacent element-definitions table. If the
existing element definition is genuinely insufficient, the right ticket
is one against that definition, not against the Scope and Usage prose.

---

### Recommendation

**Recommended disposition:** A

The reporter is asking for a small, well-scoped editorial improvement
to a paragraph that, on its own, presupposes knowledge it does not
introduce — exactly the kind of issue that "Technical Correction"
tickets are meant to absorb. The fix is narrative-only, has no
normative impact, no breaking-change risk, and no cross-resource
implications, so it is cheap to merge and easy for the WG to review.
Disposition B is a defensible variant and the WG may wish to do *both*
(strengthen the element definition **and** add an introductory
sentence in Scope and Usage) — but the additional element-definition
work should not be a precondition for accepting the simple narrative
fix the reporter actually requested. Disposition C undersells a real
clarity gap that adjacent Zulip discussion confirms is a recurring
source of confusion among implementers.
