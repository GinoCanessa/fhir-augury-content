## Summary

Eric Haas (Pharmacy) reports that `MedicationDispense.notPerformedReason`
is defined as the reason a dispense was *not performed*, but the
`MedicationDispense.status` value set offers several codes whose
"not-performed-ness" is unclear: `cancelled`, the new R6 `unfulfilled`,
`declined`, and `on-hold`. There is no spec text or invariant that ties
`notPerformedReason` to a specific subset of those statuses, so
implementers cannot tell when populating it is required, allowed, or
forbidden. The ticket proposes adding either narrative guidance or an
invariant clarifying which `status` values require/allow
`notPerformedReason`.

The ambiguity sits at the intersection of several recent Pharmacy
threads: [FHIR-49940](https://jira.hl7.org/browse/FHIR-49940) (Applied,
Persuasive with Modification) introduced the European "dispense decline"
use case that drove much of the modeling around refusal/decline;
[FHIR-40692](https://jira.hl7.org/browse/FHIR-40692) (Applied,
Persuasive) re-evaluated the MedicationDispense status codes in light of
BiologicallyDerivedProductDispense (which is where `unfulfilled`
originates); and [FHIR-54372](https://jira.hl7.org/browse/FHIR-54372)
(Triaged, opened the same day by the same reporter) asks for parallel
status-↔-timestamp invariants for `whenPrepared`/`whenHandedOver`. Older
context comes from [FHIR-46696](https://jira.hl7.org/browse/FHIR-46696)
(QI-Core, Not Persuasive) and
[FHIR-34000](https://jira.hl7.org/browse/FHIR-34000) (Published, Not
Persuasive with Modification), both of which examined the
`statusReason`/`notPerformedReason` definition without resolving the
"which statuses?" question. The current ticket squarely targets that
unresolved gap.

## Details

**Description:**

> The element `notPerformedReason` is defined as: *"Indicates the reason
> why a dispense was **not performed**."*
>
> However, the status values that indicate "not performed" are
> ambiguous:
>
> - `cancelled` — Was it not performed?
> - `unfulfilled` — Explicitly not performed (new in R6)
> - `declined` — Not performed (patient declined?)
> - `on-hold` — Temporarily not performed but might be later
>
> **Issues:**
>
> - When exactly should `notPerformedReason` be populated? Only for
>   `unfulfilled`? For `cancelled`? For `declined`?
>
> **Propose** Adding guidance or an invariant on which status values
> require/allow `notPerformedReason`.

**Comments:** None on the ticket as of the snapshot.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a constraint (FHIRPath invariant) to `MedicationDispense` that ties
`notPerformedReason` to the not-performed status codes. Concretely, in
`HL7/fhir` under `source/medicationdispense/` (the authoring root for
the resource):

- **`source/medicationdispense/structuredefinition-medicationdispense.xml`**
  — add an invariant, e.g. `mdd-2`:
  - **Severity:** error
  - **Human:** *"`notPerformedReason` SHALL only be present when
    `status` is `cancelled`, `declined`, or `unfulfilled`."*
  - **Expression:**
    `notPerformedReason.empty() or status in ('cancelled' | 'declined' | 'unfulfilled')`
  - **XPath:** equivalent guard on `f:notPerformedReason` /
    `f:status/@value`.
- **`source/medicationdispense/medicationdispense.html`** (and
  `medicationdispense-notes.html` if used by the resource) — add a
  short "When to use `notPerformedReason`" subsection in the resource
  narrative that mirrors the invariant and explains the rationale per
  status code (notably calling out `on-hold` as *temporarily* halted
  and therefore **not** a not-performed state).
- **`source/medicationdispense/medicationdispense-spreadsheet.xml`** —
  if still used as authored input for this resource, mirror the
  invariant there so the publisher does not regenerate it away.

Pair with FHIR-54372 so that the `whenPrepared` / `whenHandedOver`
invariants and this `notPerformedReason` invariant share the same
"performed vs. not performed" partition of the status code list.

#### Justification

The reporter's question — "when exactly should this be populated?" — is
exactly the question an invariant answers, and FHIR has a strong
existing pattern of using `xxx-N` invariants on resources to pin
context-sensitive cardinality (e.g., the existing `mdd-1` on
MedicationDispense already constrains
`whenHandedOver`/`whenPrepared`). FHIR-40692 (Applied) deliberately
introduced `unfulfilled` and aligned `declined` with the BDP "abandoned"
semantics, and FHIR-49940 (Applied) brought the dispense-decline use
case into core; together those resolutions imply a clear "performed /
not-performed" partition that this invariant simply formalises.
Excluding `on-hold` is consistent with its definition as a *temporary*
halt where a dispense may yet occur — the dispense has not been
*not-performed*, only paused. Co-ordinating with FHIR-54372 (same
reporter, same partition) avoids two overlapping invariants landing
inconsistently.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a hard invariant; instead, add **normative narrative
guidance** in the MedicationDispense resource page plus an
element-definition comment on `notPerformedReason`, and add a
**warning-level** invariant rather than an error.

Specifically in `HL7/fhir` under `source/medicationdispense/`:

- **`structuredefinition-medicationdispense.xml`** — strengthen the
  `ElementDefinition.comment` (and `requirements`) on
  `notPerformedReason` to read approximately: *"This element is
  expected when `status` is `unfulfilled` or `declined`. It MAY be
  used with `status = cancelled` to record why a dispense that had
  been started was cancelled. It SHALL NOT be used with `status =
  on-hold`, which represents a temporary pause rather than a
  not-performed dispense."*
- **`medicationdispense.html`** — add an "Interpreting the status
  codes" table that, for each `status` value (including the new R6
  `unfulfilled`), spells out whether `notPerformedReason` is
  Required / Permitted / Forbidden.
- Add `mdd-2` as a `severity = warning` invariant with the same
  expression as Disposition A, so trading-partner profiles (e.g.,
  US-Core, IHE MPD) can promote it to `error` without breaking
  generic R5 instances that may already exist.

#### Justification

FHIR-46696 was closed *Not Persuasive* in part because changes to the
status/`notPerformedReason` relationship reach across realms, and
FHIR-49940 explicitly noted that European decline-of-dispense usage
diverges from the assumptions in the resource scope. A hard error
invariant added late in R6 risks invalidating R5 instances that already
populate `notPerformedReason` alongside `cancelled` (a perfectly
defensible interpretation today, since the spec doesn't forbid it). A
warning-level constraint plus prescriptive narrative achieves the
clarification the reporter is asking for, gives implementers actionable
guidance, and lets profiles like QI-Core (the FHIR-46696 originator) and
IHE MPD tighten the rule for their own scope without core having to make
the call for everyone. It is also the lower-risk option this close to
ballot, where invariant changes attract scrutiny.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive — already addressed**: the existing
definition of `MedicationDispense.notPerformedReason` ("the reason a
dispense was not performed") together with the published definitions of
each `medication-dispense-status` code is sufficient to determine when
the element applies, and no spec change is required. The
recommendation to the reporter would be to use the established status
definitions (e.g., `on-hold` = temporary pause, `unfulfilled` =
explicitly not performed, `declined` = patient declined, `cancelled` =
order cancelled) and to populate `notPerformedReason` whenever a
human-readable not-performed reason is meaningful.

#### Justification

This is the path FHIR-46696 (CQI) and FHIR-34000 (the original
statusReason definition review) effectively took: *Not Persuasive* /
*Not Persuasive with Modification*, on the basis that the existing
element + status-code definitions are descriptive enough. Declining is
defensible if the WG concludes that an invariant would over-constrain
real-world workflows — for example, recording why an `on-hold` dispense
was put on hold can be clinically useful even if "on-hold" is not a
not-performed state — and if the WG prefers the principle that
cardinality/context guidance for codeable elements lives in the value
set + element definition rather than in invariants.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach)

The ticket identifies a real, reproducible interpretation gap — both
FHIR-46696 and FHIR-34000 stalled because the same gap is awkward to
close cleanly — so a *Decline* (Disposition C) leaves the ambiguity in
place for another cycle and effectively repeats the FHIR-46696 outcome
that the reporter is implicitly objecting to. *Accept As Requested*
(Disposition A) is the right *technical* answer, but landing a new
error-level invariant on a long-published resource late in the R6 cycle
risks invalidating instances that today legitimately populate
`notPerformedReason` alongside `cancelled`, and the FHIR-49940 record
shows that the "decline / refusal" semantics are still being negotiated
across realms (EU MPD vs. US Core vs. QI-Core). Disposition B threads
that needle: it gives implementers the unambiguous "Required /
Permitted / Forbidden" table the reporter is asking for, anchors it in
the element definition where most implementers actually look, and uses
a warning-level invariant that downstream profiles (US Core, IHE MPD,
QI-Core) can tighten to an error in their own scope. The disposition
should be coordinated with FHIR-54372 so a single, consistent partition
of the status codes governs both `notPerformedReason` and the
`whenPrepared`/`whenHandedOver` timing constraints.
