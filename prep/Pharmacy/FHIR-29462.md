## Summary

The reporter (Daniel Zhang) flags a precision/wording issue in the PDMP
FHIR Implementation Guide's "Request Data Element" mapping table for
**Requestor Role**. As written, the mapping cites the NCPDP "`<Qualifier>`
element" as the source mapped to `PractitionerRole.code`. In the NCPDP
SCRIPT/PDMP XML schema, however, `Qualifier` is not a free-standing
element — it is an **attribute** carried on the `<Header>/<From>` element
that types the identifier conveyed there (i.e., it qualifies what the
"From" identifier represents, such as DEA, NPI, NCPDP Provider ID, etc.,
which is what the IG is trying to express as a requestor role). The
ticket asks the WG to correct the mapping wording so that it accurately
references the **`Qualifier` attribute of `<Header>/<From>`** rather
than a non-existent `<Qualifier>` element.

The only cross-reference returned by Augury is the FHIR artifact
`PractitionerRole.code`, which is the target side of the mapping. No
linked Jira tickets, no related Jira tickets, no Zulip discussions, and
no GitHub items were found. Comment history shows the ticket was
originally imported from a spreadsheet (2020), dispositioned
"Considered for Future Use" in Jan 2023 because the IG had been
withdrawn, and then **reopened in Nov 2023 by smrobertson** under the
PSS‑2225 *PDMP on FHIR* project, which has resumed work on the IG —
making the mapping wording fix relevant again.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the wording of the *Requestor Role* row in the PDMP IG's
**"PDMP Request Data Element"** mapping table so that the source side
no longer says "`<Qualifier>` element" and instead reads, e.g.:

> *Qualifier* attribute of `<Header>/<From>` (NCPDP SCRIPT) →
> `PractitionerRole.code`

Concretely:

- Edit the relevant data-element mapping page in the US PDMP FHIR IG
  (the "Request Data Element" mapping section) so the *Source* column
  reads **"`Qualifier` attribute of `<Header>/<From>`"** (or the
  IG's preferred convention for naming an XML attribute on a parent
  element).
- Leave the *Target* unchanged: `PractitionerRole.code`.
- Add a brief parenthetical that the `Qualifier` attribute typifies
  what identifier the `<From>` carries (e.g., DEA, NPI, NCPDP Provider
  ID), so readers understand why this conveys the requestor's role.
- Apply the same correction anywhere else in the IG narrative that
  references the same misnamed `<Qualifier>` element (search the IG
  source for `<Qualifier>` / "Qualifier element" to catch siblings).

No structural change to the FHIR mapping itself — `PractitionerRole.code`
remains the FHIR target. This is an editorial / mapping-precision fix.

#### Justification

This is a documentation accuracy issue, not a design issue. The NCPDP
SCRIPT/PDMP XML schema does carry `Qualifier` as an **attribute on
`<Header>/<From>`**, not as a stand-alone element, so the existing IG
text mis-describes the source artifact and is misleading to
implementers building NCPDP↔FHIR bridges. Accepting the change costs
nothing in spec design, eliminates a real reader confusion in a
normative-feeling mapping table, and aligns the IG with the actual
NCPDP wire format. The ticket has the highest priority, the original
"Considered for Future Use" disposition was driven by the IG being
withdrawn (a workflow reason, not a substantive disagreement), and the
PSS‑2225 reopening explicitly puts editorial cleanup of the old PDMP
issues back on the table.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying correction, but use it as the trigger to do a
**broader pass on the PDMP Request/Response data-element mapping
tables**, rather than a one-line wording edit:

- Audit every row in the PDMP Request Data Element and Response Data
  Element mapping pages where the source column references an NCPDP
  XML construct, and normalize the convention used to refer to
  elements vs. attributes (e.g., introduce an explicit notation such
  as `<Header>/<From>@Qualifier` or "`<Header>/<From>` element,
  `Qualifier` attribute").
- For *Requestor Role* specifically, also reconsider whether the FHIR
  target should remain `PractitionerRole.code` alone, or whether the
  qualifier-coded identifier (DEA / NPI / NCPDP Provider ID) is more
  faithfully expressed as `PractitionerRole.practitioner.identifier`
  with a qualifying `system`, while keeping `PractitionerRole.code`
  for the *role*. Document the chosen split.
- Capture the convention in the IG's editorial guidance section so
  future mappings stay consistent.

#### Justification

The ticket is one symptom of a class of issues — the PDMP IG's
data-element mappings predate disciplined element/attribute notation,
and the "old PDMP issues" label suggests there are siblings. Doing the
fix in isolation risks leaving inconsistent neighbours. It also gives
the WG a cheap opportunity to check whether the *Requestor Role*
mapping is conflating a participant's **role** with the **identifier
qualifier** that types their ID — which is a real semantic question,
not just wording. Trade-off vs. Disposition A is scope: this requires
a small editorial sweep and a brief WG decision on identifier
modelling, in exchange for a more durable mapping. Backwards
compatibility is preserved because the IG is being re-published under
PSS‑2225 anyway.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as *Not Persuasive* (or *Persuasive with Mod — No
Change Required*) on the basis that the *Requestor Role → 
`PractitionerRole.code`* mapping is conceptually correct as published
and that the `<Qualifier>` reference in the source column should be
read as informal shorthand for the NCPDP SCRIPT qualifier construct.
No spec or IG change is made.

#### Justification

A defensible decline rests on two observations: (1) the ticket
identifies a wording issue, not a behavioural or interoperability
defect — implementers who already understand NCPDP SCRIPT will read
"Qualifier" correctly without the IG re-stating it; and (2) the
original 4‑0‑0 "Considered for Future Use" vote effectively concluded
that PDMP IG editorial cleanup was not worth the WG's time when the
IG was withdrawn. If the WG considers the PDMP IG re-publication out
of scope for the current cycle, this disposition keeps the ticket
backlog clean without committing editorial effort.

---

### Recommendation

**Recommended disposition:** **A** (Accept As Requested), optionally
absorbing a lightweight slice of B.

The ticket is precise, narrow, and substantively correct: in NCPDP
SCRIPT XML, `Qualifier` is an attribute on `<Header>/<From>`, not an
element, and the PDMP IG's "Request Data Element" mapping table
mis-states this. The fix is editorial, has zero impact on the FHIR
mapping target (`PractitionerRole.code` is unchanged), and removes a
real source of implementer confusion in a mapping-heavy IG. The 2023
"Considered for Future Use" vote was driven by the IG being withdrawn;
PSS‑2225 has reopened the IG, so that procedural reason no longer
applies — declining (Disposition C) would now be hard to justify on
its merits. Disposition B is attractive but should not block A: I
recommend the WG accept the wording correction now, and separately
add an action item to sweep the rest of the PDMP mapping tables for
the same element/attribute confusion under the PSS‑2225 work plan.
