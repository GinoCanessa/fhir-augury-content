## Summary

The ticket, raised by Scott M. Robertson out of a Connectathon discussion,
asks the US Prescription Drug Monitoring Program (PDMP) FHIR IG to allow
medications to be coded with **both NDC and RxNorm**, rather than the
RxNorm-only binding that the PDMP profiles inherit today via the
[Medication Clinical Drug value set](https://vsac.nlm.nih.gov/valueset/2.16.840.1.113762.1.4.1010.4/expansion).
The stated real-world expectation is that **NDC is the most common
medication identifier in PDMP traffic (SHALL)**, while **RxNorm is
supported (SHOULD / MAY)**. The ticket has been triaged as Persuasive,
with the resolution noting that PDMP stakeholders work almost
exclusively with NDCs and that the IG will support both codings, with
specific guidance needed for the case where an RxNorm code is not sent
(because RxNorm is "required" by US Core profiles that PDMP derives
from). No related Jira tickets, Zulip threads, GitHub PRs/issues, or
other cross-references were surfaced by Augury for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the PDMP IG so that the medication-bearing profiles
(MedicationRequest and MedicationDispense, plus any embedded
`contained` Medication profile) re-bind `Medication.code` /
`medicationCodeableConcept` to a value set that **unions NDC and
RxNorm** instead of the RxNorm-only Medication Clinical Drug value set
(VSAC OID 2.16.840.1.113762.1.4.1010.4). Concretely:

- Define (or reuse) a PDMP-specific value set, e.g.
  `pdmp-medication-codes`, that includes the NDC code system
  (`http://hl7.org/fhir/sid/ndc`) and the RxNorm semantic clinical drug
  subset currently in use.
- Bind that value set to `Medication.code` /
  `medicationCodeableConcept` with binding strength **extensible** (or
  **preferred**), so NDC is first-class.
- Make `CodeableConcept.coding` slicing on the profiles require an
  **NDC slice with min=1** ("SHALL include NDC") and a **RxNorm slice
  with min=0 / mustSupport=true** ("SHOULD include RxNorm"), matching
  the SHALL/SHOULD/MAY language in the ticket.
- Add narrative guidance describing the divergence from US Core (which
  treats RxNorm as required) and the rationale for accepting NDC-only
  representations on the PDMP wire.

#### Justification

This is exactly what the ticket requests and what the resolution text
("Will support both in IG") commits to. Reflecting real PDMP traffic —
which is overwhelmingly NDC — in the binding avoids forcing senders to
fabricate or look up RxNorm codes they don't have, which would
otherwise turn an interoperability profile into a barrier. Using slice
cardinality (NDC min=1, RxNorm min=0/MS) keeps the SHALL/SHOULD/MAY
intent enforceable at validation time without losing RxNorm where it is
available. The Pharmacy WG has previously used multi-coding patterns
for medication identifiers, so this is consistent with existing
practice.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-binding the existing element, **derive PDMP profiles
directly from the FHIR base (or from a Pharmacy-WG-curated parent)
instead of from US Core**, and bind `Medication.code` to a value set
that includes both NDC and RxNorm with binding strength **required**
(NDC SHALL be present; RxNorm MAY be present in an additional
`coding`). Document the non-derivation from US Core explicitly, and
provide a conformance variant / capability statement assertion for
systems that want to *also* claim US Core conformance (in which case
RxNorm `mustSupport` continues to apply). Optionally publish a small
extension `pdmp-medication-source-system` to flag whether the canonical
identifier on the wire was NDC or RxNorm, for downstream PDMP
analytics.

#### Justification

The hardest part of the request — per the resolution note — is the
clash with US Core's effective requirement that RxNorm be present on
medication resources. Quietly weakening that requirement inside a
US-Core-derived profile creates a confusing situation where a resource
is PDMP-conformant but not US-Core-conformant, even though it claims
US-Core lineage. Severing the derivation makes the divergence honest
and inspectable, and the optional dual-conformance capability statement
preserves a path for systems that need both. This trades implementation
clarity for a slightly larger one-time spec change, and matches how
other "wire-format" IGs (e.g., NCPDP-aligned work) have handled
US-Core-vs-domain-reality tension.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep the existing RxNorm-only binding to the
Medication Clinical Drug value set. Document, in narrative, that
implementers MAY include an NDC `coding` alongside the required RxNorm
coding, but make no normative change to the binding or to the slice
cardinalities.

#### Justification

Defensible only on the grounds of US Core alignment: PDMP profiles
inherit from US Core, and US Core has already made the RxNorm-required
call across the US realm. Declining preserves that alignment and avoids
fragmenting medication semantics across US-realm IGs. The cost is that
it explicitly ignores the Connectathon evidence that real PDMP traffic
is NDC-first, and it leaves implementers to either fabricate RxNorm
mappings or fail validation — which is precisely why the ticket was
filed and why it has been triaged Persuasive.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the smallest change consistent with the ticket's
ask, with the resolution text already committed by the Pharmacy WG
("Will support both in IG"), and with the actual PDMP wire reality.
Modeling NDC as a `min=1` slice and RxNorm as a `min=0`/mustSupport
slice on `Medication.code` directly encodes the SHALL/SHOULD/MAY
intent and is enforceable by the standard validator, without needing
to break the US Core derivation chain that Disposition B proposes.
The US-Core-conflict the resolution flags is real but narrower than
it looks: PDMP is a domain-specific exchange, and tightening the NDC
side while leaving RxNorm as mustSupport-but-optional is a defensible
profile-level constraint over a US Core base. Disposition B should be
held in reserve in case the Pharmacy WG and the US Core stewards
decide the derivation is causing more confusion than it's worth;
Disposition C should be rejected because it directly contradicts the
already-recorded "Persuasive" resolution.
