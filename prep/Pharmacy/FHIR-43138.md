## Summary

Scott M. Robertson opened this ticket on behalf of an external implementer
(Ansuman Nayak / Jean Hall at LogiCoy) who is mapping pharmacy data
(NCPDP / PMIX) into the US PDMP FHIR specification. The ticket is filed
as a **Question** (not a defect or change request) and asks the Pharmacy
work group for guidance on four specific modeling questions about the
PDMP FHIR profiles:

1. How to link a Practitioner (the prescriber) to a MedicationDispense.
2. How to link a pharmacy Organization to a MedicationDispense.
3. Whether MedicationAdministration is needed for PDMP reporting given
   that PMIX / NCPDP source data does not appear to provide an explicit
   administration mapping.
4. If MedicationAdministration *is* needed, how it should be linked back
   to the corresponding MedicationDispense.

No cross-referenced Jira tickets, Zulip discussions, or GitHub artifacts
were returned by the FHIR Augury index for this ticket, so the request
stands on its own without a broader thread of community discussion.

## Proposed Dispositions

Because this ticket is a **Question** about how to use the existing
PDMP FHIR specification — not a defect report or change request — the
dispositions are framed as reviewer-response options rather than
spec-edit options. Each disposition still names concrete elements in
the affected resources so the work group can act on it directly.

### Disposition A: Accept As Requested

#### Proposal

Resolve the ticket by providing an authoritative answer that points the
implementer at the existing FHIR R4 elements on `MedicationDispense` and
`MedicationAdministration`, then close the ticket as **Answered /
Persuasive** with no normative change to the PDMP IG. The answer
posted on the ticket should explicitly cover:

1. **Prescriber → MedicationDispense:** The prescriber is not carried
   directly on `MedicationDispense`. It is reached via
   `MedicationDispense.authorizingPrescription` →
   `MedicationRequest.requester` (a `Reference(Practitioner |
   PractitionerRole | Organization | Patient | RelatedPerson | Device)`).
   For PDMP reporting, populate `MedicationRequest.requester` with the
   prescribing `Practitioner` (or `PractitionerRole` if the practice
   role/organization context is needed) and reference that
   MedicationRequest from each MedicationDispense.
2. **Pharmacy Organization → MedicationDispense:** Use
   `MedicationDispense.performer.actor` with a `Reference(Organization)`
   for the dispensing pharmacy (and/or
   `MedicationDispense.location` for the physical dispensing site). If
   both the pharmacist and pharmacy are reported, add a second
   `performer` entry referencing the `Practitioner` /
   `PractitionerRole` for the pharmacist.
3. **MedicationAdministration for PDMP:** Confirm that PDMP reporting is
   built around *dispense* events, not *administration* events. NCPDP /
   PMIX source data does not carry an administration record because the
   patient self-administers outpatient prescriptions; therefore
   MedicationAdministration is **not required** for PDMP.
   MedicationAdministration is appropriate only when the workflow being
   reported is a clinician-administered dose (e.g., an in-clinic
   injection) that the implementer also wants to surface.
4. **MedicationAdministration → Dispense linkage (if used):** When an
   administration *is* recorded against a dispensed supply, link it via
   `MedicationAdministration.partOf` →
   `Reference(MedicationAdministration | Procedure)` for procedural
   context, and via `MedicationAdministration.request` →
   `Reference(MedicationRequest)` to tie back to the same prescription
   the dispense was authorized by. The dispense and administration are
   then siblings under a common `MedicationRequest`, which is the
   FHIR-idiomatic linkage.

#### Justification

These mappings already exist in the FHIR R4 Medication module and are
the canonical patterns used by every other pharmacy-oriented IG (e.g.,
US Core, IPS, Pharmacy IG). Answering the question "as requested"
respects the implementer's framing and gives them a directly actionable
response without modifying the PDMP IG. Because the questions are
purely interpretive — they do not assert a spec gap — closing as
*Persuasive / Answered* is consistent with the work group's standard
handling of question-typed tickets.

### Disposition B: Alternative Approach

#### Proposal

Treat the ticket as evidence that the **PDMP IG narrative is missing an
implementer-facing mapping section** for these high-frequency
NCPDP/PMIX-to-FHIR linkages, and resolve it by:

1. Posting the same answer described in Disposition A on the ticket.
2. Opening a follow-up tracker against the PDMP IG to add a short
   "Mapping NCPDP / PMIX to PDMP FHIR Resources" guidance page (or
   expanding an existing one) that documents, with examples:
   - Prescriber linkage via
     `MedicationDispense.authorizingPrescription` →
     `MedicationRequest.requester`.
   - Pharmacy organization and pharmacist via
     `MedicationDispense.performer.actor` and
     `MedicationDispense.location`.
   - Explicit statement that MedicationAdministration is **out of scope
     for PDMP** dispense reporting, with a one-paragraph rationale.
   - A labelled diagram showing
     `Patient ← MedicationRequest ← MedicationDispense (←
     MedicationAdministration, optional)`.
3. Resolving FHIR-43138 with a "Persuasive with Modification"
   disposition that points at the new guidance page.

#### Justification

The fact that an implementer with NCPDP/PMIX experience had to file a
Jira ticket to ask these four questions strongly suggests the PDMP IG's
implementer narrative leaves the dispense-side resource topology
under-documented. Adding the guidance once is cheaper than answering
the same question repeatedly on Jira/Zulip, and it strengthens the
spec without requiring any structural / conformance change. The trade-off
versus Disposition A is the additional work of an IG narrative edit,
but the cost is small and the benefit accrues to every future PDMP
implementer.

---

### Disposition C: Decline

#### Proposal

Close FHIR-43138 as **Not Persuasive / Not a Specification Issue** on
the grounds that the ticket asks general "how do I model X in FHIR"
questions that are already answered by the base FHIR R4 specification
for `MedicationRequest`, `MedicationDispense`, and
`MedicationAdministration` and are not specific to the US PDMP IG.
Direct the reporter to:

- The R4 resource pages for `MedicationDispense` (`performer`,
  `location`, `authorizingPrescription`) and `MedicationAdministration`
  (`partOf`, `request`).
- The HL7 `pharmacy` Zulip stream for implementer Q&A.

#### Justification

Jira is a normative-change tracker; questions of the form "how do I
link X to Y in my implementation" are typically routed to chat.zulip
or the implementer mailing list rather than disposed by the work group
as ticket work. Declining keeps the Pharmacy work group's queue focused
on actionable IG/spec changes. The downside is that the implementer
gets no answer on the ticket itself, which is a poor experience for a
question that takes only a few minutes to answer correctly.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because it does both of the things the
work group should care about: (1) it gives the LogiCoy implementer a
direct, concrete answer (the same one Disposition A provides), and
(2) it captures the durable lesson — the PDMP IG's narrative does not
adequately describe the NCPDP/PMIX-to-FHIR resource topology — by
opening a small follow-up to harden the IG. Disposition A leaves the
next implementer to file the same ticket again. Disposition C is
defensible procedurally but is poor stewardship for a question that
the spec authors can answer in a paragraph and that points at a real
documentation gap in a US-Realm IG. Disposition B costs slightly more
work than A but materially improves the IG and avoids repeat traffic.
