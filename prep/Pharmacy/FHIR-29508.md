## Summary

The reporter argues that the introduction to the (then-named) US PDMP /
US Medication FHIR Implementation Guide does not make a convincing case
for adoption. The submitter notes that (a) the FHIR resources cited in
the guide are not widely used in the US, (b) other standards — notably
NCPDP and PMIX — already cover most PDMP use cases in production, and
(c) the guide as written does not appear to support querying multiple
state PDMPs simultaneously, a capability that already exists in
deployed tooling. The ticket therefore asks the workgroup to revise the
Introduction so that it clearly articulates the value proposition of
this FHIR-based guide *relative to* existing PDMP infrastructure,
rather than describing the guide in a vacuum.

The ticket has had a complicated lifecycle: it was originally imported
from a tracking spreadsheet (comment #58), was dispositioned in 2023-01
as "Considered for Future Use" (4-0-0; John Hatem / Tim McNeil) on the
basis that the IG had been withdrawn, and was then re-opened in
2023-11 by smrobertson when the IG work was picked back up under
[PSS-2225 PDMP on FHIR](https://confluence.hl7.org/display/PHAR/US+Prescription+Drug+Monitoring+Program+%28PDMP%29+FHIR+Implementation+Guide).
There are no Augury-cross-referenced Jira tickets, Zulip threads, or
GitHub items linked to FHIR-29508 at this time.

## Proposed Dispositions

> Note: There are no Augury-surfaced GitHub cross-references for this
> ticket, so the dispositions below are framed against the IG as
> identified in the ticket metadata — *US Medication FHIR Implementation
> Guide* — and the re-opening reference to PSS-2225 (US PDMP on FHIR).
> If the WG decides to act, the concrete file/path targets should be
> confirmed against the current IG repo before drafting changes.

### Disposition A: Accept As Requested

#### Proposal

Rewrite the IG's Introduction (typically `input/pages/index.md` or the
equivalent narrative page in the Pharmacy IG repo) to add a
clearly-labeled "Why this guide?" / "Relationship to existing
standards" subsection. The new content must, at minimum:

1. Acknowledge the existing US PDMP landscape, naming **NCPDP SCRIPT
   / PMP** transactions and **PMIX/PMP InterConnect** as the dominant
   in-production solutions for prescriber/dispenser PDMP queries.
2. State the specific gap(s) this FHIR guide is designed to close —
   e.g., bringing PDMP data into the EHR-native FHIR workflow alongside
   `MedicationRequest` / `MedicationDispense` / `MedicationStatement`,
   reducing duplicate context-switching for clinicians, and enabling
   reuse of SMART-on-FHIR security and FHIR-based audit/consent
   patterns rather than bespoke PDMP portals.
3. Address the multi-state-query concern explicitly: either describe
   how the IG composes with PMIX/InterConnect at the back end (i.e.,
   the FHIR layer is an EHR-facing facade over existing federation),
   or, if the IG genuinely does not solve federation, say so plainly
   and scope the guide to the in-EHR consumption use case.
4. Add a short "Out of scope" bullet list so readers stop expecting
   capabilities the IG does not deliver.

#### Justification

The reporter's complaint is fundamentally an editorial one — the IG's
narrative does not justify its own existence to a US audience that is
already served by NCPDP/PMIX. That is straightforwardly fixable in the
Introduction without any conformance impact. The 2023 vote to defer
("Considered for Future Use") was driven by IG withdrawal, not by
disagreement with the substance of the comment; now that PSS-2225 has
revived the work, addressing the comment is a prerequisite for the
revived IG to ballot credibly. Naming NCPDP and PMIX directly is also
consistent with HL7's general practice of citing related standards in
IG introductions so that implementers can self-select.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only revising the Introduction prose, treat this ticket as
the trigger to publish a separate, normative-tone **Scope and
Relationship to Other Standards** page in the IG (e.g.,
`input/pages/scope.md`) and to expand the IG's **Use Cases** page with
concrete personas:

- An EHR-embedded clinician workflow that retrieves PDMP data via
  FHIR at the point of prescribing.
- A health-system aggregation workflow that queries PDMP data via FHIR
  for population-level analytics / opioid stewardship.
- An explicit non-use-case statement that bulk multi-state federation
  is delegated to PMIX/InterConnect, with a pointer to those specs.

The Introduction would then carry a one-paragraph summary and a
prominent link into the new Scope page, instead of trying to cram the
full justification into the Introduction itself.

#### Justification

The ticket's surface request (fix the Introduction) is symptomatic of a
deeper documentation gap: the IG does not have a page that
implementers can cite when pitching a FHIR-based PDMP project against
an existing NCPDP/PMIX stack. Splitting "marketing" prose from
"scope/relationship" content keeps the Introduction readable while
giving procurement-style readers a stable URL to anchor on. This
approach also de-risks future similar tickets — additional
"introduction" comments can be redirected to amending the Scope page
without churning the front door of the IG.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the
Introduction's role is to set context, not to defend the IG against
every alternative standard. The Pharmacy WG can instead point readers
to the standard HL7 mechanism for cross-standards positioning (the
project's PSS, the WG mission statement, and the published IG-to-IG
relationship pages) and leave the Introduction substantially as-is.

#### Justification

This is defensible if the WG takes the position that comparative-value
arguments belong in the **PSS-2225** project documentation and the
HL7-wide IG-comparison materials, not in the IG's own front matter —
and that adopting the change risks turning the Introduction into a
sales document. It is also defensible if the WG concludes that the
multi-state federation concern is genuinely out of scope and that
restating that in the Introduction adds no normative value. Decline
should only be chosen if the WG is confident readers already have
enough context to evaluate the IG; the reporter's lived experience
suggests they do not.

---

### Recommendation

**Recommended disposition:** B

The ticket is a legitimate documentation problem, not a conformance
problem, so Disposition C (Decline) materially under-serves the
reporter and does not address the credibility issue that PSS-2225's
revived IG will face at ballot. Disposition A is correct in spirit but
risks ballooning the Introduction into a multi-page essay that is hard
to maintain as PDMP standards evolve. Disposition B keeps the
Introduction lean while creating a durable home — a Scope /
Relationship-to-Other-Standards page plus expanded Use Cases — for the
NCPDP/PMIX comparison and the explicit treatment of multi-state
federation. That structure also gives the WG a clean place to absorb
any follow-on "introduction" tickets without re-litigating the front
page each time. The WG should pair this with an editorial decision to
name NCPDP SCRIPT / PMP and PMIX/InterConnect by name, since refusing
to name them is itself part of what triggered this ticket.
