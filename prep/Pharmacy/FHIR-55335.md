## Summary

The Pharmacy work group is asked to evaluate whether four code systems (and the
value sets that wrap them) currently shipped inside the FHIR Core
MedicationRequest resource should be migrated out to the HL7 Terminology
ecosystem (THO / `terminology.hl7.org`). The four candidates and their bindings
on MedicationRequest are:

| CodeSystem | ValueSet | Bound element |
|------------|----------|---------------|
| `medicationrequest-category` (`MedicationRequestCategoryCodes`) | `medicationrequest-category` | `MedicationRequest.category` |
| `medication-intended-performer-role` (`MedicationIntendedPerformerRole`) | `medication-intended-performer-role` | `MedicationRequest.performerType` |
| `medicationrequest-dispenser-instructions` (`MedicationRequestDispenserInstructions`) | `medicationrequest-dispenser-instructions` | `MedicationRequest.dispenseRequest.dispenserInstruction` |
| `medication-dose-aid` (`MedicationDoseAid`) | `medication-dose-aid` | `MedicationRequest.dispenseRequest.doseAdministrationAid` |

The originator (Lloyd McKenzie, imported from comment 299 on a separate
review) explicitly notes that migration is **not required**. The criteria
proposed for considering migration are (a) the value-set definition is
realistically expected to need to change over the coming years, and/or (b)
the codes might be useful outside FHIR — in CDA, V2, or other HL7
specifications. No specific migration target tier within THO is stipulated;
the request is for the WG to assess each of the four sets individually and
decide which (if any) should move.

The ticket carries **Highest** priority and is currently in **Triaged**
status with no assignee. There are no formally linked Jira issues; the
cross-reference index only surfaces the four FHIR elements named above as
mentions inside the description itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate all four CodeSystem/ValueSet pairs from FHIR Core into THO, retire
the in-spec definitions in `HL7/fhir` (under the MedicationRequest source
tree), and re-point each MedicationRequest binding at the new THO canonical
URLs:

- `medicationrequest-category` (CS + VS) → THO, bound to
  `MedicationRequest.category`.
- `medication-intended-performer-role` (CS + VS) → THO, bound to
  `MedicationRequest.performerType`.
- `medicationrequest-dispenser-instructions` (CS + VS) → THO, bound to
  `MedicationRequest.dispenseRequest.dispenserInstruction`.
- `medication-dose-aid` (CS + VS) → THO, bound to
  `MedicationRequest.dispenseRequest.doseAdministrationAid`.

Authoring would happen in `HL7/UTG`, with the existing FHIR Core artifacts
deleted (or replaced by stub redirects, per the standard THO migration
playbook) and the MedicationRequest StructureDefinition / element bindings
in `HL7/fhir` updated to reference the new THO canonicals. Existing
`http://hl7.org/fhir/...` canonical URLs would be preserved as historical
aliases per current THO migration practice so that prior IGs continue to
resolve.

#### Justification

Both criteria the ticket calls out apply to the strongest of the four:

- `medicationrequest-category` codes are semantically generic ("inpatient",
  "outpatient", "community", "discharge", …) and are demonstrably used
  outside FHIR Core today — the Zulip thread cited above notes US Core,
  CMS, and HEDIS measure sets all reuse these codes, satisfying the
  cross-spec reuse criterion.
- `medication-intended-performer-role` codes describe practitioner roles
  that overlap with V2 and CDA participation models and are therefore
  reasonable cross-spec candidates.
- `medicationrequest-dispenser-instructions` and `medication-dose-aid` are
  smaller, narrower vocabularies whose definitions are likely to evolve as
  pharmacy practice matures (especially dose-aid devices), satisfying the
  "definition likely to change" criterion.

Migrating now, while these sets are still small, avoids a larger and more
disruptive migration after R6 publication and aligns MedicationRequest with
the trajectory other clinical resources have already followed.

---

### Disposition B: Alternative Approach

#### Proposal

Migrate only the value sets that meet a clear external-reuse bar today —
namely `medicationrequest-category` and `medication-intended-performer-role`
— and leave the two dispense-time vocabularies
(`medicationrequest-dispenser-instructions` and `medication-dose-aid`)
inside FHIR Core for this cycle. Concretely:

- Move `medicationrequest-category` (CS + VS) and
  `medication-intended-performer-role` (CS + VS) to THO via `HL7/UTG`,
  re-pointing the corresponding MedicationRequest bindings.
- For `medicationrequest-dispenser-instructions` and `medication-dose-aid`,
  open follow-up Pharmacy WG tickets to track them for re-evaluation in the
  next cycle, but make no change to FHIR Core in this ballot.

For the two migrated sets, follow the same canonical-URL preservation
pattern described in Disposition A.

#### Justification

The ticket explicitly says migration is **not required** and asks the WG to
weigh each set against two criteria. A split decision honours that framing:
the two "category / role" vocabularies have strong, demonstrated cross-spec
reuse (US Core, HEDIS, CMS, V2/CDA participation), while the two
"dispense aid / dispenser instructions" vocabularies are still maturing,
have narrower extant reuse, and risk dragging an unsettled definition into
THO before the Pharmacy WG has converged on its semantics. Migrating only
the well-understood sets reduces churn for implementers, keeps the
cross-spec wins, and defers the riskier moves until evidence accrues.

---

### Disposition C: Decline

#### Proposal

Decline the migration request for this cycle and keep all four
CodeSystem/ValueSet pairs in FHIR Core under the MedicationRequest source
tree, with current bindings unchanged. Document the WG's rationale on the
ticket and resolve as "Persuasive with Mod" → "no change" or
"Not Persuasive", with a note that re-evaluation may occur in a future
cycle if external-reuse evidence grows.

#### Justification

Two defensible reasons support declining:

1. **Cost vs. benefit at this stage of the cycle.** A THO migration is not
   a free editorial change — it touches `HL7/UTG` authoring, FHIR Core
   binding URLs, downstream IGs that hard-code the
   `http://hl7.org/fhir/...` canonicals, and validators / tooling caches.
   Doing four migrations late in an R6-style cycle for value sets that are
   already functional risks introducing regressions for marginal benefit.
2. **Migration is explicitly optional.** The ticket states migration "is
   not required" and only suggests it where definitions are likely to
   evolve or the codes are useful outside FHIR. The WG can legitimately
   conclude that none of the four cross the threshold in the current cycle
   — particularly for the two dispense vocabularies that have no documented
   non-FHIR consumers.

---

### Recommendation

**Recommended disposition: B (Alternative Approach).**

The Zulip discussion in `committers > PR Reviews` makes a concrete case that
`medicationrequest-category` is already being treated as a shared
vocabulary by US Core, CMS, and HEDIS — exactly the cross-spec reuse the
ticket's criterion targets — and the same logic extends naturally to
`medication-intended-performer-role` given its overlap with V2/CDA
participation roles. Migrating those two now captures the consolidation
benefit while it is still cheap. By contrast,
`medicationrequest-dispenser-instructions` and `medication-dose-aid` have
no documented external consumers in the indexed evidence, and dose-aid
semantics in particular are still maturing in clinical practice. Locking
those into THO now risks publishing a vocabulary that the Pharmacy WG will
want to redefine within one or two cycles, which is precisely the kind of
churn THO migration is meant to avoid. Disposition B captures the high-value
moves, defers the speculative ones with an explicit follow-up trail, and
remains faithful to the ticket's "consider, don't mandate" framing better
than either A (over-commits) or C (forgoes a clear win).
