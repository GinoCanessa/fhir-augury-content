## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the *US Medication FHIR IG* successor of the ASAP 2.1A â†’ FHIR
"Mappings for PDMP Response" table, add a second row for
**Medicare Number** mapped to `Patient.identifier`, modelled exactly
the same way SSN is modelled today. Concretely:

- Keep the existing row: `Patient Identifier | SSN* | SSN | Patient.identifier`.
- Add a new row: `Patient Identifier | Medicare Number* | Medicare Number | Patient.identifier`.
- Carry forward the `*` "multiple data elements" footnote so the
  semantics match the ASAP source.
- In the IG narrative immediately under the table, document how to
  distinguish the two identifiers on the wire â€” i.e., that both ASAP
  values are sliced into `Patient.identifier` and the receiver tells
  them apart by `Patient.identifier.type` (SSN: `SS`, Medicare Number:
  `MA` or, for current MBI, the HL7 v2-0203 `MB` / a CMS-defined code)
  and/or by `Patient.identifier.system` (e.g.,
  `http://hl7.org/fhir/sid/us-ssn` for SSN and the CMS MBI system URL
  for the Medicare Beneficiary Identifier).
- Fix the spelling ("Identifer" â†’ "Identifier",
  "Patient.identifer" â†’ "Patient.identifier") that appears in the
  reporter's proposed wording â€” do not propagate the typos into the IG.
- If the IG already publishes a `PDMPPatient` (or equivalent) profile,
  add a second `Patient.identifier` slice for Medicare Number alongside
  the SSN slice, with the corresponding `type` and `system` fixed
  values.

#### Justification

This is what the reporter asked for, it matches what ASAP 2.1A actually
permits (Medicare Number is a defined patient identifier in the ASAP
response), and it is consistent with how every other US-realm IG
(US Core, Da Vinci PDex, CARIN BB, etc.) models multiple patient
identifier kinds â€” one `Patient.identifier` repetition per kind, with
`type` and `system` distinguishing them. The change is documentation +
(optionally) one new identifier slice on a profile; it does not break
existing implementations and does not require any base-FHIR change. The
2023-11-28 reopen via PSS-2225 explicitly intended for these old PDMP
items to be re-evaluated now that the IG is being taken up again, which
removes the only reason the workgroup previously deferred this
("Considered for Future Use" pending IG revival).

### Disposition B: Alternative Approach

#### Proposal

Instead of enumerating Medicare Number as a second hard-coded mapping
row, generalise the table: replace the SSN-only row with a single row
that maps "Patient Identifier (any ASAP-permitted kind)" to
`Patient.identifier`, and move the per-identifier detail into a
separate "Patient identifier kinds" sub-section that lists each
permitted ASAP value (SSN, Medicare Number, Driver's License,
Military ID, â€¦) together with its required FHIR
`Patient.identifier.type` code (from `http://terminology.hl7.org/CodeSystem/v2-0203`
plus any CMS extensions for MBI) and `Patient.identifier.system` URL.
Bind `Patient.identifier.type` in the IG profile to a value set that
enumerates the ASAP-permitted identifier kinds, and require `system` to
be present.

#### Justification

ASAP 2.1A allows several patient identifier kinds in addition to SSN
and Medicare Number (driver's licence, state ID, passport, military
ID, etc.). Adding them one ticket at a time produces a long, brittle
mapping table and another round of "what about X?" tickets. Driving
identifier kinds through a value-setâ€“bound `Patient.identifier.type`
is the same pattern used by US Core and Da Vinci IGs and gives
implementers a single, machine-checkable answer rather than narrative
prose. The trade-off versus Disposition A is that this is more work
than the reporter strictly asked for and may need a Pharmacy-side value
set or reuse of an existing one; it also delays the Medicare Number
fix until the broader sub-section is drafted.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** (or "Already Specified") on the
basis that `Patient.identifier` is already an open, repeating element
whose semantics are governed by `type` and `system`, so any
implementation is already free to send a Medicare Number as a
`Patient.identifier` repetition with the appropriate `type`/`system` â€”
no spec change is required, and the ASAP-to-FHIR table can be left as
an SSN exemplar rather than an enumeration.

#### Justification

`Patient.identifier` is `0..*` and identifier kinds are intentionally
distinguished by `type` and `system` rather than by adding new FHIR
elements. From a pure-FHIR conformance standpoint the IG does not need
to list every ASAP-permitted identifier kind for implementations to be
correct. Declining keeps the IG narrative short and avoids the table
becoming a maintenance burden every time ASAP adds another permitted
identifier value.

### Recommendation

**Recommended disposition:** A (with the spelling fixes and `type` /
`system` guidance from the proposal â€” and a follow-up ticket carved
off to do Disposition B's value-set generalisation later).

The ticket is small, the request is correct on the substance (Medicare
Number genuinely is an ASAP-permitted patient identifier and an
implementer reading the current table would reasonably assume it is
not), and the only historical reason for deferral â€” the original PDMP
IG having been withdrawn â€” has been explicitly removed by the
2023-11-28 reopen under PSS-2225. Disposition C is technically true
but unhelpful to implementers, who routinely use these mapping tables
as the authoritative answer to "which ASAP fields does this IG
support?"; saying nothing about Medicare Number leaves a real
ambiguity in the wire format. Disposition B is the right *eventual*
shape of the section but should not block this fix â€” the workgroup can
land Disposition A now, file a companion ticket to convert the table
to a value-setâ€“driven sub-section, and resolve any further "what about
identifier X?" tickets against that single follow-up rather than one
at a time. The Justification in Disposition A also commits the editor
to specifying `Patient.identifier.type` and `Patient.identifier.system`
for both SSN and Medicare Number, which is what an implementer
actually needs to interoperate.
