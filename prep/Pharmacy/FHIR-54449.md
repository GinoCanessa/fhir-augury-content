## Summary

Eric Haas reports that the semantics of `MedicationStatement.adherence.reason`
are confusing and likely the result of copy/paste from `MedicationRequest`. The
current `adherence.reason` definition ("Captures the reason for the current use
or adherence of a medication"), short ("Details of the reason for the current
use of the medication"), and comment (which references "exception statuses such
as entered-in-error" and points readers to `reasonCode`) all describe *why a
medication is being taken*, which overlaps with the resource-level `reason`
element. The reporter argues this conflates *adherence* (whether the medication
is being taken as instructed) with *indication* (why it is being taken) and
proposes rewording the definition, short, and comment so that
`adherence.reason` is understood as the reason a medication has not been
consumed according to instructions.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were returned
by the data sources for this ticket, so this review relies on the ticket
content and its single comment alone.

## Details

**Description:**

`adherence` Structure - Confusing Semantics

`adherence.code`: *"Type of adherence for the medication"*
Binding: MedicationStatementAdherenceCodes (Example)

`adherence.reason`: *"Captures the reason for the current use or adherence of a
medication"*

&nbsp;&nbsp;&nbsp;&nbsp;short: *"Details of the reason for the current use of
the medication"*

&nbsp;&nbsp;&nbsp;&nbsp;comment: *"This is generally only used for 'exception'
statuses such as 'entered-in-error'. The reason for performing the event at
all is captured in reasonCode, not here."*

**Issues:**

- `adherence.reason` talks about "reason for current use" - but adherence is
  about WHETHER a medication is being taken according to instructions, not
  WHY.
- The `reason` element (at the resource level) already captures "why the
  medication is being/was taken".
- The comment seems to be a copy/paste error from the
  `MedicationRequest.reasonCode` element.

**Propose:**

- Change `adherence.reason` definition to: *"reason a medication **has not**
  been consumed according to instructions."*
- Change `adherence.reason` short to: *"Reason for lack of adherance"*
- Update comment to: *"Only used if the patient is not taking the medication
  as instructed."*

**Comments:**

- **costateixeira (2026-04-06):** "I don't think this is only for reason why a
  medication has not been consumed according to instructions. It can be a
  reason for on-hold, or reason to resuming."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the reporter's wording verbatim to `MedicationStatement.adherence.reason`
in FHIR Core:

- **definition:** "Reason a medication **has not** been consumed according to
  instructions."
- **short:** "Reason for lack of adherence" (correcting the spelling of
  "adherance" in the proposal).
- **comment:** "Only used if the patient is not taking the medication as
  instructed."

Leave `MedicationStatement.adherence.code` and the
`MedicationStatementAdherenceCodes` value set unchanged. Leave the
resource-level `MedicationStatement.reason` element unchanged so it continues
to carry the indication / "why the medication is being taken".

#### Justification

This is the most direct fix for the reporter's concern: it removes the textual
overlap between `adherence.reason` and the resource-level `reason` and removes
the comment block that was copy/pasted from `MedicationRequest.reasonCode`
(which is not even an element on `MedicationStatement`, so the current text is
actively misleading). It is a definition/short/comment edit only, with no
schema, cardinality, or binding impact, so it is non-breaking for existing
implementations and instances.

### Disposition B: Alternative Approach

#### Proposal

Adopt the *intent* of the reporter's change but broaden the wording to cover
the full set of adherence states surfaced by costateixeira's comment. Concretely:

- **definition:** "Reason for the current adherence state recorded in
  `adherence.code` — for example, why the medication is being taken
  intermittently, why it is on hold, why it has been stopped, or why the
  patient is not taking it as instructed. Distinct from
  `MedicationStatement.reason`, which captures *why* the medication was taken
  in the first place."
- **short:** "Reason for the current adherence state."
- **comment:** "Use this to explain the value chosen in `adherence.code` (e.g.
  on-hold, not-taking, taking-as-directed). The clinical indication for the
  medication itself belongs in `MedicationStatement.reason`, not here."

Optionally add an invariant or guidance noting that `adherence.reason` SHOULD
only be populated when `adherence.code` is present, to reinforce the link
between the two elements.

#### Justification

costateixeira's comment is the only piece of community feedback on the ticket
and it explicitly pushes back on narrowing `adherence.reason` to "not taken as
instructed", noting that on-hold and resumption reasons are also legitimate.
The `MedicationStatementAdherenceCodes` value set already enumerates states
beyond simple non-adherence (taking, not-taking, on-hold, stopped, unknown),
so the definition should describe the reason as scoped to whichever
`adherence.code` value applies, not just the negative case. This still removes
the conflation with the resource-level `reason` element (the core defect the
reporter identified) and still scrubs the misleading copy/paste comment, but
without locking the element to a single status. It is also non-breaking.

### Disposition C: Decline

#### Proposal

Do not change the definition, short, or comment of
`MedicationStatement.adherence.reason`. If the comment about
`MedicationRequest.reasonCode` is genuinely confusing, the only edit is to
strike the second sentence of the existing comment so that it reads
approximately: "This is generally only used for 'exception' statuses such as
'entered-in-error'." Leave the definition and short as-is.

#### Justification

The definition wording, while imperfect, has been in the spec long enough that
implementations and downstream IGs may be relying on it. The reporter's
proposed wording narrows the element to a single use case ("not taken as
instructed") that the work group's own commenter (costateixeira) explicitly
disagrees with, and there is no Zulip thread, linked ticket, or implementer
feedback in the cross-references suggesting the current wording is causing
real-world data quality problems. A "no change" disposition keeps the element
stable while the work group decides whether a deeper redesign of the
`adherence` BackboneElement is needed.

---

### Recommendation

**Recommended disposition:** B

Disposition A is faithful to the reporter but is contradicted by the only
other voice on the ticket: costateixeira's comment makes a substantive point
that `adherence.reason` legitimately covers on-hold and resumption reasons,
not just non-adherence, and that point is consistent with the broader value
set bound to `adherence.code`. Disposition C preserves the status quo but
leaves the genuine defect — the copy/pasted `MedicationRequest.reasonCode`
comment and the textual overlap with the resource-level `reason` — in place,
which is what triggered the ticket. Disposition B keeps the corrective
edits the reporter is really asking for (decoupling `adherence.reason` from
the indication-style `reason`, and removing the misleading comment) while
honoring the in-thread feedback that the reason can apply to *any* adherence
state. It is also a documentation-only change with no schema or binding
impact, which keeps it well within scope for a Triaged Medium-priority
clarification ticket.
