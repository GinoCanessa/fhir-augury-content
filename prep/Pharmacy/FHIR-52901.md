## Summary

FHIR-52901 is a Pharmacy work-group **Question** asking how cyclical
medication regimens — the canonical example being oral contraceptives
("21 tablets daily, then 7 days off") — should be represented in R6 so
that systems can safely interpret and exchange them. The reporter
(TimSchuller) notes these patterns are clinically common and
safety-critical, and there is no clear, documented R6 pattern.

The ticket is closely entangled with two other Pharmacy items surfaced
by keyword similarity and explicit comment links:

- **[FHIR-51540](https://jira.hl7.org/browse/FHIR-51540)** —
  *"MaxDosePerPeriod to apply across multiple concurrent dosage
  instructions"* (Resolved – Persuasive with Modification). Christof
  Teixeira flagged FHIR-52901 as **dependent** on the FHIR-51540
  resolution, because both questions push on how multiple Dosage
  instances combine semantically.
- **[FHIR-54252](https://jira.hl7.org/browse/FHIR-54252)** —
  *"Adding a Repeat cycle element to DosageDetails"* directly proposes a
  concrete structural answer to FHIR-52901 (a new
  `repeatPeriodCyclicalSchedule` element on DosageDetails, modelled on
  the Nictiz extension), and uses the same 21-on / 7-off example.

The most recent comment (Christof Teixeira, 2026-01-25) says the
question "should be reassessed with the new DosageDetails type",
acknowledging that the answer surface has shifted since the ticket was
filed. The September WGM discussion (frank_mckinney) committed to (a)
investigating RequestOrchestration as a cycle-repeat mechanism with the
CDS WG, (b) considering other approaches, and (c) creating an example
for the R6 ballot. To date, that example has not been added.

## Details

**Description:**

> Some therapies follow a cyclic regimen (e.g., oral contraceptives:
> 21 tablets daily, then 7 days break. These regimens are clinically
> common and safety-critical. What is the recommended way within R6 to
> model these cyclic patterns so that systems can safely interpret and
> exchange them?

**Comments:**

- **JIRAUSER30126 (2025-09-17):** Points to the Netherlands' Nictiz
  extension currently used for these instructions:
  `https://simplifier.net/packages/nictiz.fhir.nl.r4.medicationprocess9/2.0.0-rc.2/files/2961568`.
- **frank_mckinney (2025-09-17):** Discussed at the Sep WGM. Confirmed
  there is no example in the current FHIR spec. Identified
  RequestOrchestration as a possible mechanism. To-do list: investigate
  RequestOrchestration (CDS WG owner), explore other approaches, and
  produce an R6 ballot example jointly with CDS.
- **costateixeira (2025-09-24):** "We should indeed add an example of
  RequestOrchestration which would be more thorough and complicated.
  For the case at hand, where it is the same medication, we could also
  see how to do this in the possible/expected changes in the dosage
  data type, which, with nesting, should support this with ease."
  Proposed disposition: add **two** complex cyclic-dosage examples —
  (1) RequestOrchestration aligned with COW and CDS, (2) a nested
  Dosage structure as discussed in FHIR-51540.
- **costateixeira (2025-09-24, follow-up):** Marks FHIR-52901 as
  dependent on FHIR-51540.
- **jduteau (2025-10-06):** "Create an example that uses the current
  structure to see if we can just add that to the spec." (i.e., try
  current Timing/Dosage first before changing the model.)
- **costateixeira (2026-01-25):** "This should be reassessed with the
  new DosageDetails type." — i.e., the answer surface has materially
  changed since the ticket was filed; the reassessment should consider
  the DosageDetails redesign tracked in adjacent tickets such as
  FHIR-54252.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-52901 as a **documentation gap** and close it by adding
worked R6 examples that demonstrate cyclical regimens with the
**existing** R6 building blocks, exactly as jduteau requested
(2025-10-06) and as frank_mckinney committed to in the September WGM:

1. Update the Dosage examples page (currently §14.7, including the
   ambiguous Example 5 cited in FHIR-54252) to add a
   **"Cyclic regimen — 21-on / 7-off"** example expressed using
   `Timing.repeat` (`boundsDuration` of 28 days, with `frequency`/
   `period` and a `dayOfWeek` / explicit-event constraint) plus a
   plain-language `Dosage.text`. Note explicitly that the current
   "frequency=21 in period=28d" reading is unsafe and must not be used.
2. Add a **second example** on the RequestOrchestration page (in
   coordination with the Clinical Decision Support WG, owners of
   RequestOrchestration) showing the same 21-on / 7-off regimen modelled
   as a RequestOrchestration of two MedicationRequests with explicit
   timing windows, per Christof Teixeira's 2025-09-24 disposition.
3. Add a short narrative section ("Representing cyclic regimens") under
   MedicationRequest cross-linking the two examples.

No structural changes to Dosage, DosageDetails, or Timing. No new
elements. Resolution would be **Persuasive** (or **Not Persuasive with
Modification** if the WG considers the question already answered by
the new examples).

#### Justification

This is the lowest-risk, fastest path to ballot and matches both the
September WGM minutes and jduteau's "try the current structure first"
position. It addresses the literal request ("what is the recommended
way in R6?") by giving implementers two concrete, copy-pasteable
patterns. It also lets Pharmacy ship something for R6 ballot without
waiting on the larger DosageDetails redesign or on the FHIR-51540
nesting semantics to fully settle. The downside is acknowledged: it
leaves cyclic schedules expressed in a verbose form that is easy to
get subtly wrong, which is precisely the safety concern Tim Schuller
raised.

---

### Disposition B: Alternative Approach

#### Proposal

Couple FHIR-52901 to **FHIR-54252** and resolve it by adopting a
**native cyclic-schedule element on DosageDetails**:

1. On `DosageDetails`, add a new element
   `repeatPeriodCyclicalSchedule` (0..1, BackboneElement) modelled on
   the Nictiz `ext-InstructionsForUse.RepeatPeriodCyclicalSchedule`
   modifier extension cited by JIRAUSER30126 and FHIR-54252. The
   backbone should at minimum carry: `cycleLength` (Duration),
   `activePhase` (Duration or count of timing units),
   `inactivePhase` (Duration), and an optional `cycleAnchor`
   (CodeableConcept | Period) to handle event-relative starts (the
   FHIR-13317 requirement).
2. Replace the broken Example 5 in the Dosage examples page with the
   "21-on / 7-off" case expressed using the new element.
3. Document interaction with `maxDosePerPeriod` so the FHIR-51540
   semantics compose correctly across the active vs inactive phase
   (i.e., the `inactivePhase` contributes `0` to `maxDosePerPeriod`).
4. Resolve FHIR-52901 as **Persuasive with Modification**, marking
   FHIR-54252 as the implementing change, and add the
   RequestOrchestration narrative example from Disposition A as a
   secondary, non-normative pattern for the multi-medication cycle case.

#### Justification

This addresses Tim Schuller's underlying safety concern rather than
just papering over it: a single, explicit element is dramatically less
error-prone than overloading `Timing.repeat.frequency`/`period` and
hoping implementers read the example. It directly aligns with Christof
Teixeira's 2026-01-25 reassessment ("should be reassessed with the
new DosageDetails type") and with FHIR-54252's existing proposal, so
the WG does not split the discussion across two tickets.

The principal trade-off versus Disposition A is scope and timing:
adding a normative element to DosageDetails for R6 requires Pharmacy
to land FHIR-54252, and it should be designed alongside the
FHIR-51540 nesting semantics so cycles compose with concurrent
instructions. The renderedInstruction discussion in FHIR-54014 should
be kept in scope so that even systems that don't yet support the new
element render the cycle unambiguously.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** on the grounds that the existing R6
toolkit is sufficient: cyclic regimens can already be represented by
(a) `Timing.repeat` with explicit `boundsDuration` and `dayOfWeek` /
event constraints, (b) two MedicationRequests joined by
RequestOrchestration, or (c) `Dosage.text` plus a structured "best
effort" Timing for the active phase. Recommend the reporter use one
of these patterns and close FHIR-52901 without spec changes, while
leaving FHIR-54252 to handle any structural follow-up on its own
merits.

#### Justification

Defensible only if the WG concludes that cyclical regimens are
sufficiently rare or sufficiently well-handled by `Dosage.text` that
investing in either documentation or new structure is not warranted
for R6. This is a hard sell given (i) the reporter explicitly flagged
safety-critical exchange, (ii) frank_mckinney's WGM minutes record a
commitment to add a ballot example, and (iii) Christof Teixeira's most
recent comment treats the question as still open and expects DosageDetails
to evolve to cover it. Including this disposition for completeness so
the WG can explicitly reject it on the record.

---

### Recommendation

**Recommended disposition: B (Alternative Approach).**

The discussion thread on FHIR-52901 has visibly converged on
DosageDetails as the right home for the answer: jduteau asked for a
worked example, frank_mckinney scoped a joint piece of work with CDS,
Christof Teixeira twice (2025-09-24, 2026-01-25) pushed the question
toward a nested / DosageDetails-based encoding, and FHIR-54252 already
proposes the concrete element with the **same** 21-on / 7-off example
that this ticket asks about. Resolving FHIR-52901 by simply adding
examples (Disposition A) leaves the underlying safety problem in
place — implementers will still encode cycles with overloaded
`Timing.repeat.frequency` and the Example-5 ambiguity will recur.
Declining (Disposition C) ignores both the WGM commitment and the
reporter's safety framing.

Pragmatically, the WG should **resolve FHIR-52901 as Persuasive with
Modification, pointing to FHIR-54252 as the implementing change**, and
**also include the RequestOrchestration narrative example from
Disposition A** as a non-normative companion pattern for cycles that
span multiple medications (where a single Dosage cannot carry the
cycle). FHIR-51540's nesting semantics and FHIR-54014's
`renderedInstruction` requirement should be kept in scope when the
DosageDetails element is finalised so the three changes compose
safely. This gives Pharmacy a single coherent answer to ship in the
R6 ballot rather than three loosely related half-answers.
