## Summary

The reporter asks how to distinguish, in FHIR, the **main (active) ingredient
medication** from accompanying **diluent / vehicle medications** when an IV
order is recorded in source EHR systems as multiple, separate
`MedicationAdministration` instances that share the same patient and the same
start/end times. The example given is a Potassium Chloride 2 mEq/mL IV order
that the EHR materialises as three separate administrations — Potassium
Chloride (the active drug), Sodium Chloride (the diluent), and Lidocaine — all
linked together via `MedicationAdministration.partOf`. The reporter has the
"main ingredient vs diluent" flag in the source EHR and wants to know whether
that flag should be carried in `MedicationAdministration.category`, in a new
extension, or somewhere else. The question is scoped to the **US Medication
FHIR Implementation Guide** but the underlying mechanism is in base FHIR
`MedicationAdministration`. Cross-reference scan surfaced only a textual
mention of `MedicationAdministration.partOf`; no related Jira tickets, no
Zulip threads, and no GitHub items were found via the augury index.

## Details

**Description:**

> For IV administrations, I have seen that some EHRs represent their data as
> separate administrations of the main ingredients and diluents.
>
> For example, suppose a physician orders Potassium chloride 2 mEq/mL
> intravenous solution for their patient — which is represented as one
> administrations for Potassium Chloride, another for a Sodium Chloride
> intravenous solution, and another for Lidocaine. This means that in the
> MedicationAdministration resource, we have separate administrations for the
> Potassium Chloride, Sodium Chloride, and Lidocaine with the same start/end
> date for the same patient. We can link these administrations together using
> the MedicationAdministration.partOf field, however for downstream
> applications we want to be able to flag which of these medications is the
> main ingredient vs the diluent. We have this data in the EHR but I am not
> sure how to represent it in FHIR. Should this information go in the
> medicationAdministration.category field or in a separate extension?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Provide a normative answer in the **US Medication FHIR Implementation Guide**
(US Medication IG) clarifying how to distinguish the main ingredient from
diluents in a multi-resource IV admixture pattern, and back it with a small
modeling artifact:

1. Define a new **role / kind extension** on `MedicationAdministration`
   (e.g., `us-medication-administration-component-role`) bound to a small
   value set with codes such as `active-ingredient`, `diluent`, `vehicle`,
   and `additive`. The extension is asserted on each child
   `MedicationAdministration` whose `partOf` points at the parent
   administration that represents the overall IV order.
2. Update the IG narrative to explicitly describe the
   "one-parent-per-admixture, one-child-per-component" pattern, give the
   Potassium Chloride / Sodium Chloride / Lidocaine example verbatim from
   this ticket, and call out that `MedicationAdministration.category` is
   **not** the right place for this flag (see Justification).
3. Add an example instance set in the IG showing the parent + three children
   with the new extension populated.

#### Justification

`MedicationAdministration.category` is bound (in base FHIR) to a value set
describing **where/how the medication is administered** (inpatient,
outpatient, community, etc.) — not the **role of a component within a
compound administration**. Re-purposing `category` for ingredient role would
overload the element's semantics and break interpretation by any consumer
that expects the standard codes. A purpose-built extension keeps
`MedicationAdministration.partOf` as the structural link (which the EHR
data already supports) and adds a clean, queryable role flag on each
child resource. Codifying this in the US Medication IG (rather than asking
implementers to invent local extensions) is consistent with how the IG
already constrains medication patterns for the US realm and gives
downstream PDMP / analytics consumers a single, predictable element to
read.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new extension on `MedicationAdministration`. Instead, push
the "what role does this substance play" information **down into the
`Medication` resource** that each child administration references, using
the existing `Medication.ingredient.isActive` boolean (and, where richer
semantics are needed, `Medication.ingredient.strengthRatio` /
`strengthQuantity`). The IG would:

1. Recommend that each component administration reference a `Medication`
   instance in which the component substance appears as an
   `ingredient`, with `isActive = true` for the main ingredient and
   `isActive = false` for diluents/vehicles.
2. Where the EHR represents components as standalone medications (the
   case in the ticket), recommend a thin profile on `Medication` that
   requires `ingredient.item` and `ingredient.isActive` to be present
   so the role can be derived without extension lookup.
3. Continue to use `MedicationAdministration.partOf` for the structural
   grouping, unchanged.

#### Justification

FHIR already has a first-class concept of "active vs non-active component"
on `Medication.ingredient.isActive`. Routing the role through `Medication`
keeps the modeling aligned with the rest of the medication graph
(MedicationRequest → Medication → ingredient) and avoids creating an
IG-specific extension that duplicates an existing core concept. The
trade-off vs Disposition A is that consumers must dereference each
child administration's `medicationReference` to discover the role,
which is a heavier read pattern than a flag directly on
`MedicationAdministration`; it also forces the source system to author
proper `Medication` resources rather than just CodeableConcepts. For
US Medication IG implementers who are already expected to materialise
`Medication` resources, that cost is small, and the resulting graph is
more consistent with base FHIR semantics.

---

### Disposition C: Decline

#### Proposal

Decline to add a new element, extension, or guidance. Close the ticket as
**"Persuasive with mod — answered"**: respond to the reporter that the
existing combination of `MedicationAdministration.partOf` (to group the
administrations) plus the referenced `Medication.ingredient.isActive`
flag is sufficient to convey ingredient vs diluent, and that
`MedicationAdministration.category` is the wrong element because it
encodes setting/kind of administration, not component role.

#### Justification

Pharmacy WG has limited bandwidth for new US Medication IG content, and
the underlying capability already exists in base FHIR via
`Medication.ingredient.isActive`. Adding another extension or value set
would create a US-specific divergence for a problem solvable with
existing R4/R5 elements, and would risk inconsistent adoption across
implementations that have already mapped EHR data using the
`Medication.ingredient` path. Declining keeps the IG lean and pushes
the answer to the implementer in documentation form rather than as a
new normative artifact.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right balance for a Pharmacy WG **Question** ticket:
it answers the reporter's actual need (a queryable "main ingredient vs
diluent" flag) by leaning on `Medication.ingredient.isActive`, which is
already the core-FHIR home for that concept, and it avoids minting a new
US-Medication-IG-only extension that would duplicate base semantics.
Disposition A is technically clean but introduces a parallel mechanism
on `MedicationAdministration` that consumers would have to learn in
addition to `Medication.ingredient`; over time that tends to fragment
implementations. Disposition C is too thin — the reporter explicitly
asked between two specific modeling choices (`category` vs extension)
and deserves a written, normative answer in the IG, not just a comment
on the ticket. Either way, the WG should also explicitly state in the
IG narrative that `MedicationAdministration.category` is **not** the
correct landing spot, since that misuse is the most likely default if
no guidance is published.
