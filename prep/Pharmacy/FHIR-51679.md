## Summary

The ticket asks the CARIN RTPBC IG (Pharmacy WG) to retire the four
locally-defined `NamingSystem` resources that STU1 used to reference
external NCPDP and Canada Post terminologies, and instead bind directly
to the corresponding code-system canonical URLs that have since been
added to HL7 Terminology (THO). The change touches three profiles
(RTPBC Pharmacy Organization, RTPBC Patient, RTPBC Response
ClaimResponse) and three ValueSets (RTPBC Pharmacy Type, RTPBC Formulary
Status, RTBPC State and Province). Because production RTPBC
request/response instances carry these `Coding.system` values on the
wire (NCPDP pharmacy type, NCPDP provider ID, NCPDP reject code, and
Canadian province codes), the swap is explicitly flagged by the
reporter as a **substantive (breaking) change** — implementers will
need to emit and accept the THO/Canada Post canonical URIs in place of
the IG-local NamingSystem identifiers.

No cross-referenced Jira tickets, GitHub PRs/issues, or recent Zulip
threads were surfaced for this key. The ticket was filed 2025-08-07,
triaged, and resolved **Persuasive** on 2025-12-12, indicating the
Pharmacy WG has already accepted the request in principle; this report
prepares the formal disposition write-up for ratification.

## Details

**Description:**

Replace NamingSystems used in the IG to reference external NCPDP and
Canada Post terminologies with the THO code systems that were added
after STU1 was published.

Note that this will change the `coding.system` values used in RTPBC
request and responses for NCPDP codes (pharmacy type, provider ID and
reject code) and Canada province codes, so this is a substantive
change.

**Resolution (already recorded as Persuasive on 2025-12-12):**

Replace the following NamingSystems with references to corresponding
codesystem definitions in THO and adjust associated bindings in
profiles:

- **NCPDP Provider Identification Number** → `http://terminology.hl7.org/CodeSystem/NCPDPProviderIdentificationNumber`
  - Affects **RTPBC Pharmacy Organization** profile.
- **NCPDP Pharmacy Type** → `http://terminology.hl7.org/CodeSystem/NCPDPPharmacyType`
  - Affects **RTPBC Pharmacy Organization** profile and **RTPBC Pharmacy Type** ValueSet.
- **NCPDP Reject Code** → `http://terminology.hl7.org/CodeSystem/NCPDPRejectCode`
  - Affects **RTPBC Formulary Status** ValueSet and **RTPBC Response ClaimResponse** profile.
- **Canada Post Province Code** → `http://canadapost.ca/CodeSystem/ProvinceCodes`
  - Affects **RTBPC State and Province** ValueSet, **RTPBC Patient** and **RTPBC Pharmacy Organization**.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution exactly as drafted by the reporter:

1. Remove (or deprecate and stop referencing) the four IG-local
   `NamingSystem` resources for NCPDP Provider Identification Number,
   NCPDP Pharmacy Type, NCPDP Reject Code, and Canada Post Province
   Code from the RTPBC IG sources.
2. Update the affected profiles to bind/use the THO and Canada Post
   canonical code-system URIs:
   - **RTPBC Pharmacy Organization** — change the `Identifier.system`
     slice for the NCPDP provider ID to
     `http://terminology.hl7.org/CodeSystem/NCPDPProviderIdentificationNumber`;
     change the pharmacy-type `CodeableConcept.coding.system` to
     `http://terminology.hl7.org/CodeSystem/NCPDPPharmacyType`;
     change the address-state binding to use the Canada Post system
     where applicable.
   - **RTPBC Patient** — change the address-state binding to the new
     RTBPC State and Province ValueSet (which itself now points at the
     Canada Post canonical URL).
   - **RTPBC Response ClaimResponse** — change the reject-code
     `Coding.system` to
     `http://terminology.hl7.org/CodeSystem/NCPDPRejectCode`.
3. Update the affected ValueSets to compose from the new canonical
   systems:
   - **RTPBC Pharmacy Type** ValueSet — `compose.include.system` =
     `http://terminology.hl7.org/CodeSystem/NCPDPPharmacyType`.
   - **RTPBC Formulary Status** ValueSet — replace the NCPDP reject
     code system reference with
     `http://terminology.hl7.org/CodeSystem/NCPDPRejectCode`.
   - **RTBPC State and Province** ValueSet — replace the Canada Post
     NamingSystem reference with `http://canadapost.ca/CodeSystem/ProvinceCodes`.
4. Refresh examples (request and response Bundles, and any
   contained-resource examples on the IG's "Submit Operation" / "other"
   pages) so all `Coding.system` and `Identifier.system` values use the
   new canonical URIs.
5. Add a prominent "Substantive change since STU1" entry to the IG's
   change log / migration guidance describing the wire-level system
   URI change for each of the four code systems, and call it out in
   ballot intro material.

This work lands in the `HL7/carin-rtpbc` IG (FSH/StructureDefinition
sources, ValueSets, and example bundles); the THO entries themselves
already exist and do not need to be authored under this ticket.

#### Justification

The four THO code systems (`NCPDPProviderIdentificationNumber`,
`NCPDPPharmacyType`, `NCPDPRejectCode`) and the Canada Post canonical
(`http://canadapost.ca/CodeSystem/ProvinceCodes`) are the
community-sanctioned canonical identifiers for exactly the concepts the
RTPBC IG is naming. The original 2019 Zulip thread shows the
NamingSystems were a deliberate stop-gap precisely because canonical
URIs were "in process"; with THO now publishing them, continuing to use
IG-local NamingSystems creates a parallel identifier graph that
implementers must reconcile, blocks reuse with neighboring pharmacy
specifications (Da Vinci, US Core, NCPDP-aligned IGs), and prevents
terminology services from validating RTPBC payloads against the
authoritative code systems. The reporter's framing as substantive is
correct and aligns with FHIR versioning policy, which is the right
trade for moving onto the standard URIs once at the next substantive
publication rather than carrying the workaround forward indefinitely.
The Pharmacy WG has already marked the resolution **Persuasive**, which
matches Disposition A.

---

### Disposition B: Alternative Approach — Phased Migration with Dual-Binding

#### Proposal

Take the same end-state as Disposition A, but stage the wire-level
break across two publications:

1. In **this** publication, leave the existing NamingSystems in place
   and **add** the THO/Canada Post canonical URIs as additional allowed
   `Coding.system` values, using:
   - Slicing on `Coding.system` (open slicing) on the affected
     `CodeableConcept` elements so receivers MUST accept either the
     legacy NamingSystem URI or the THO canonical URI.
   - For `Identifier.system` (NCPDP Provider Identification Number on
     RTPBC Pharmacy Organization), document both URIs as valid and
     specify a preferred form for senders (the THO canonical).
   - Update the affected ValueSets to `compose.include` **both** the
     legacy NamingSystem-named system and the THO canonical, with a
     deprecation note on the legacy entry.
2. Mark the four IG-local NamingSystems `status = retired` and add
   guidance that new senders SHALL use the THO/Canada Post canonical
   URIs and existing senders SHOULD migrate.
3. Schedule the actual removal of the legacy systems for the next
   substantive RTPBC ballot/publication after this one.

#### Justification

RTPBC is an active, deployed real-time interaction; an immediate flip
of `Coding.system` on production traffic is observable to every
intermediary, payer adjudicator, and pharmacy benefit manager that
parses the response. A dual-binding window lets payers and submitters
upgrade independently and removes the "big bang" risk that a single
trading partner trips on the URI change at runtime. The trade-off is
real: receivers carry temporary parsing complexity, terminology
services must be configured for both URIs, and the IG carries
deprecated NamingSystems for one more cycle. This is a defensible
compromise when the underlying concern is operational safety rather
than theoretical correctness, and it is consistent with how other
HL7 IGs (e.g., US Core for `birthsex` extension migrations) have
handled wire-level identifier changes.

---

### Disposition C: Decline (Status Quo)

#### Proposal

Keep the existing IG-local `NamingSystem` resources in place for this
publication and do not bind to the THO / Canada Post canonical
systems. Document in the IG that the THO entries exist and are
informationally equivalent, but that RTPBC's normative
`Coding.system` / `Identifier.system` values continue to be the
NamingSystem-defined URIs from STU1.

#### Justification

The status-quo case rests on (a) avoiding any substantive,
implementer-visible change in this cycle, and (b) minimizing
coordination risk with operational RTPBC trading partners. Because the
NamingSystems already work end-to-end and the THO additions are
strictly additive, the IG can technically continue without change.
However, this position does not address the underlying motivation for
the request — alignment with the broader FHIR terminology ecosystem —
and it carries a growing maintenance debt: every neighboring pharmacy
spec that adopts the THO canonical URIs will diverge from RTPBC, and
new implementers will face an avoidable mapping step. Decline is
defensible only as a "not now" position tied to a concrete commitment
to migrate at the next ballot, not as an indefinite deferral.

---

### Recommendation

**Recommended disposition: A (Accept As Requested).**

The Pharmacy WG has already recorded the resolution as **Persuasive**,
the requested THO code systems exist and are stable, and the change
is exactly the kind of post-STU "clean up the stop-gap once the
canonical URIs ship" work that ballot cycles are designed for. The
reporter (who authored the original NamingSystems back in 2019) is
explicit that the change is substantive — meaning the WG and ballot
process will surface the wire impact to implementers without needing a
dual-binding mechanism to soften it. Disposition B's phased approach is
attractive for production safety, but it doubles the IG's terminology
surface area for at least one cycle and historically tends to leave
deprecated systems in place far longer than planned. Disposition C
postpones a problem that only grows with each adjacent IG that adopts
the THO canonicals. Accept as requested, and pair the publication with
a clearly-labeled migration note in the IG change log and ballot
introduction so implementers know to update their `Coding.system` /
`Identifier.system` emitters and parsers in lockstep.
