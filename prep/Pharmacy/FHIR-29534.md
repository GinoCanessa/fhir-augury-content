## Summary

This ticket reports a typographical error in the US Prescription Drug
Monitoring Program (PDMP) FHIR Implementation Guide (filed against the
"US Medication FHIR Implementation Guide" Jira specification, which is
the Pharmacy-tracked spec for the PDMP IG). The PDMP Responder
conformance text reads "request **coorelation**" where it should read
"request **correlation**" in the SHALL statement requiring support for
the `X-Request-ID` HTTP header used to correlate requests between a
PDMP Requester and PDMP Responder. The reporter (Scott M. Robertson)
notes in a 2023 comment that the typo has been fixed, but also raises
a follow-on question about whether the Capability Statement should be
referencing `X-Request-ID` at all — i.e., whether the conformance
hook is in the right artifact. The ticket has been reopened under the
revived PSS-2225 PDMP on FHIR project after the IG had previously been
withdrawn (which had caused the original "Considered for Future Use"
disposition to be reverted). No cross-referenced Jira tickets, Zulip
threads, or GitHub items were found via the augury index.

## Details

### Existing Wording:
The PDMP Responder SHALL support the HTTP Header parameter X-Request-ID for request **coorelation** between the PDMP Requester and PDMP Responder

### Proposed Wording:
The PDMP Responder SHALL support the HTTP Header parameter X-Request-ID for request **correlation** between the PDMP Requester and PDMP Responder

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the spelling correction exactly as proposed: in the PDMP
Responder conformance statement of the US PDMP FHIR Implementation
Guide, change "request **coorelation**" to "request **correlation**".
The corrected sentence reads: *"The PDMP Responder SHALL support the
HTTP Header parameter X-Request-ID for request correlation between
the PDMP Requester and PDMP Responder."* No semantic, conformance, or
Capability Statement change is made; this is a pure technical
correction. The fix should be applied wherever the sentence appears
(narrative page describing PDMP Responder requirements and any
copy/paste duplicates), with the existing SHALL strength on
`X-Request-ID` left intact for the next ballot/publication of the
revived IG (PSS-2225).

#### Justification

The change is a non-substantive editorial fix that improves
readability and removes an obvious misspelling, with zero impact on
implementers, conformance, or interoperability. It matches the
ticket's `Technical Correction` type and `Highest` priority for an
otherwise trivial change, and the reporter has already confirmed the
typo is corrected in their working copy. Treating it as a pure typo
fix avoids conflating it with the open design question raised in the
2023-11-29 comment about whether the Capability Statement is the
right home for `X-Request-ID` — that concern is a separable issue and
should not block landing the spelling correction.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spelling correction (as in Disposition A) **and**, in the
same change set, address the reporter's follow-up concern by splitting
this ticket: spin off a companion Pharmacy ticket to evaluate whether
the `X-Request-ID` SHALL belongs on the **Capability Statement**
itself or in the narrative conformance section for PDMP
Requester/Responder actors. The companion ticket would review whether
expressing an HTTP header requirement via a `CapabilityStatement.rest`
constraint is supported / idiomatic in FHIR (HTTP-header-level
expectations are typically narrative, not Capability Statement
machinery), and either (a) move the SHALL to the actor narrative and
remove the Capability Statement reference, or (b) keep it in the
Capability Statement with a clarifying note. The current ticket would
still close on the typo fix, but with a cross-link to the new ticket
so the open question is not lost.

#### Justification

The 2023-11-29 comment makes clear the reporter sees a second issue
beyond the typo, and a Triaged ticket reopened under a revived IG is
exactly the right moment to surface that. Splitting keeps each ticket
single-purpose: the typo fix is unambiguous and can land immediately,
while the Capability Statement question requires Pharmacy WG
discussion (and possibly FHIR-I input on `CapabilityStatement`
expressivity for HTTP headers). This avoids two failure modes: (1)
holding up a trivial editorial correction behind a design discussion,
and (2) closing the ticket with the design concern silently lost.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that the entire PDMP IG conformance
text — including the misspelled sentence — should be re-evaluated
holistically as part of PSS-2225 before any per-sentence corrections
are applied, and therefore mark this ticket as "Considered for Future
Use" until the revived PDMP project produces a refreshed draft. No
edit is made to the existing (withdrawn) IG content in the interim.

#### Justification

The IG was previously withdrawn, which is exactly why this ticket was
originally dispositioned "Considered for Future Use" in January 2023.
A defensible (if conservative) view is that piecemeal editorial fixes
to an unpublished/withdrawn artifact create churn without value:
implementers are not consuming the current text, and PSS-2225 is
likely to rewrite or restructure the conformance section anyway —
including potentially answering the reporter's own question about the
Capability Statement's role. Declining here keeps the ticket queue
focused on substantive design issues that the revived project must
resolve.

---

### Recommendation

**Recommended disposition:** B

A single-character spelling fix in a SHALL statement is exactly the
kind of change that should land immediately and without ceremony — it
costs nothing and removes a small but real source of reader
distraction in a normative sentence — so blocking it (Disposition C)
is hard to justify even with the IG's history of being withdrawn,
especially now that PSS-2225 has revived the project and the reporter
has explicitly pulled this ticket back into scope. At the same time,
Disposition A leaves the reporter's own 2023-11-29 question about
whether `X-Request-ID` belongs in the Capability Statement
unaddressed, and that question is a real design issue (HTTP-header
SHALLs are typically expressed in narrative actor conformance, not
inside `CapabilityStatement.rest`). Disposition B captures both: land
the typo fix now under this ticket, and spin off a focused companion
ticket so Pharmacy can decide where `X-Request-ID` is properly
asserted as part of the PSS-2225 work, without coupling that
discussion to a one-character edit.
