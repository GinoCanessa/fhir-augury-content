## Summary

The ticket asserts that the US Prescription Drug Monitoring Program (PDMP)
FHIR Implementation Guide is using the wrong date element when describing
the date range of dispenses returned in a PDMP query. The current guide
text references
`MedicationDispense.authorizingPrescription.dispenseRequest.validityPeriod`,
which is the date range during which the *original prescription* is valid
for dispensing — i.e., a property of the authorizing `MedicationRequest`.
PDMP queries, however, ask "which dispenses actually occurred within a
given date range," so the relevant timestamp is the date the medication
was actually dispensed to the patient. The reporter proposes the IG use
`MedicationDispense.whenHandedOver` (the date the medication left the
pharmacy to the patient) and notes `MedicationDispense.whenPrepared` as a
fallback for systems that do not capture handover. No related Jira
tickets, Zulip discussions, or GitHub items were surfaced via the
cross-reference index; the only outbound references from this ticket are
the two `MedicationDispense` element paths just discussed. The ticket was
originally resolved "Considered for Future Use" (2023-01-18) when the
PDMP IG was withdrawn, then reopened in November 2023 under the renewed
PSS-2225 PDMP-on-FHIR project, and currently sits at status *Triaged*
with priority *Highest*.

## Details

**Description:**

MedicationDispense.authorizingPrescription.dispenseRequest.validityPeriod
is the date range on the original prescription when the pharmacy can
dispense the medication. But the PDMP request is for dispenses which have
occurred during a date range. This would be based upon
MedicationDispense.whenHandedOver (or .whenPrepared if the pharmacy
system does not record the handover).

**Existing Wording:**

`MedicationDispense.authorizingPrescription.dispenseRequest.validityPeriod`

**Proposed Wording:**

`MedicationDispense.whenHandedOver`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the US PDMP FHIR Implementation Guide narrative and any
associated `SearchParameter` / query-parameter definitions so that the
"date range of dispenses" filter is bound to
`MedicationDispense.whenHandedOver` rather than
`MedicationDispense.authorizingPrescription.dispenseRequest.validityPeriod`.
Concretely:

1. In the IG's PDMP query / response narrative, replace every reference
   to `MedicationDispense.authorizingPrescription.dispenseRequest.validityPeriod`
   (used as the dispense date filter) with
   `MedicationDispense.whenHandedOver`.
2. If the IG defines a custom `SearchParameter` for "dispense date
   range" (or reuses the base `MedicationDispense-whenhandedover`
   parameter), make that explicit and document it as the canonical
   filter for PDMP date-range queries.
3. Add a SHALL-level conformance statement on the PDMP
   `MedicationDispense` profile requiring `whenHandedOver` to be
   populated when the dispense has been handed to the patient, and a
   SHOULD-level statement allowing senders to supply `whenPrepared` when
   handover is not recorded by the pharmacy system, with implementer
   guidance that responders may fall back to `whenPrepared` for the
   filter in that case.

#### Justification

The reporter's analysis is correct on the FHIR semantics.
`MedicationRequest.dispenseRequest.validityPeriod` (reached via
`MedicationDispense.authorizingPrescription.…`) is, per the base
`MedicationRequest` definition, the period during which the prescription
is *valid for dispensing* — a property of the order, not of the
dispensing event. `MedicationDispense.whenHandedOver` is defined as the
time the dispensed product was handed over to the patient, which is
exactly the event whose temporal distribution PDMP regulators care
about. Using the prescription validity period would incorrectly pull in
dispenses whose underlying prescription was authorized to run during the
window even if no actual dispensing event occurred there, and would
exclude dispenses that occurred during the window against
prescriptions whose validity period started earlier — both of which
defeat the regulatory purpose of PDMP. The fallback to `whenPrepared`
acknowledges real-world pharmacy systems that may not capture an
explicit handover timestamp, mirroring the way the base
`MedicationDispense-whenhandedover` and `-whenprepared` search
parameters are used in practice.

---

### Disposition B: Alternative Approach

#### Proposal

Specify the PDMP "dispense date range" filter as a logical query that
**prefers `MedicationDispense.whenHandedOver` and falls back to
`MedicationDispense.whenPrepared` when handover is absent**, expressed as
a single composite/`OR` search rather than choosing one element. In the
IG:

1. Define (or reference) a custom PDMP `SearchParameter`,
   e.g. `pdmp-dispense-date`, with expression
   `MedicationDispense.whenHandedOver | MedicationDispense.whenPrepared`
   and document that responders SHALL evaluate the filter against
   `whenHandedOver` when present and against `whenPrepared` otherwise.
2. Add a profile invariant on the PDMP `MedicationDispense` profile
   requiring that at least one of `whenHandedOver` or `whenPrepared` be
   populated (`whenHandedOver.exists() or whenPrepared.exists()`).
3. Retain the narrative clarification from Disposition A that
   `authorizingPrescription.…validityPeriod` is *not* a dispense date.

#### Justification

The reporter already flags the `whenPrepared` fallback parenthetically;
formalising it as the IG's filter contract removes ambiguity for
implementers and avoids two interoperability failure modes that would
otherwise arise from a strict `whenHandedOver`-only definition: (a)
pharmacy systems that record only `whenPrepared` would silently drop out
of PDMP responses, and (b) different PDMP states could disagree on
whether to substitute `whenPrepared`, producing inconsistent
multi-jurisdiction queries. The composite search-parameter pattern is
already idiomatic for Pharmacy resources (e.g., the way base
`MedicationDispense` exposes both date elements) and gives state PDMP
operators a single filter with a deterministic evaluation rule. The
trade-off vs. Disposition A is a slightly more complex
`SearchParameter` definition in exchange for a stable contract that
covers the long tail of pharmacy systems.

---

### Disposition C: Decline

#### Proposal

Do not change the IG. Mark the ticket *Persuasive — Not Persuasive* on
the grounds that the corrected element is already obvious from the base
`MedicationDispense` definitions and that the PDMP IG can rely on the
base specification rather than restating it.

#### Justification

This is the weakest of the three options and is included for
completeness. It would only be defensible if the current IG text were
already aligned with `MedicationDispense.whenHandedOver` and the ticket
were stale — which is not the case here. The reporter is the person who
originally reopened the ticket explicitly because the renewed PSS-2225
project intends to revise the IG, so declining would re-create the
exact ambiguity that prompted the change request and would be
inconsistent with the workgroup's decision to reopen the previously
deferred PDMP backlog.

---

### Recommendation

**Recommended disposition:** B

The reporter's core diagnosis is unambiguously correct, so some form of
*Persuasive* is warranted; Disposition C is included only for
completeness. Disposition A is the literal answer to the ticket and is
defensible, but the reporter himself flags `whenPrepared` as a
necessary fallback for pharmacies that do not record handover, and
PDMP queries cross many pharmacy systems whose recording practices
vary by vendor and jurisdiction. Codifying the
`whenHandedOver`-or-`whenPrepared` rule as a single composite
PDMP search parameter (Disposition B) gives state PDMP operators a
deterministic filter, prevents silent data loss for prepared-but-not-yet-
handed-over dispenses, and matches the existing idiom of the base
`MedicationDispense` resource. It costs only a small amount of extra
specification (one composite `SearchParameter` plus a one-line
invariant) over Disposition A and avoids a foreseeable round of follow-
up tickets once implementers hit the `whenPrepared`-only case in the
field.
