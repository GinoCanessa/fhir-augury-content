## Summary

The ticket asserts that the Patient-related data mapping in the (now-revived)
US Prescription Drug Monitoring Program (PDMP) FHIR Implementation Guide
covers only patient first name, last name, and date of birth, but that the
NCPDP "Data Elements for Patient" set contains additional elements that
should also be mapped to FHIR. The reporter is asking the Pharmacy work
group to extend the PDMP IG's NCPDP-to-FHIR Patient mapping to cover the
full NCPDP patient data element set rather than just the three identity
elements. No cross-referenced Jira, Zulip, or GitHub items were found for
this ticket; context is limited to the ticket body and its administrative
comments.

## Proposed Dispositions

Because no GitHub cross-references were surfaced, these dispositions do
not name a concrete repo/authoring-root path. They reference the
PDMP FHIR IG authoring artifacts generically (the NCPDP Patient data
element mapping table and the PDMP Patient profile/differential). A
follow-up repo-analysis pass on the PDMP IG source repo will be needed
to refine file-level paths before implementation.

### Disposition A: Accept As Requested

#### Proposal

Extend the NCPDP "Data Elements for Patient" mapping table in the US
PDMP FHIR Implementation Guide so that every patient-scoped NCPDP data
element is mapped to its FHIR Patient (or associated resource)
counterpart, not just the three identity elements currently documented
(first name, last name, date of birth). Specifically:

- Add rows to the PDMP Patient mapping section for the remaining NCPDP
  patient data elements, including but not limited to: middle name,
  name prefix/suffix, gender, address (street/city/state/postal code/
  country), telephone number(s), email, patient identifier(s) (e.g.,
  medical record number, driver's license number where used by PDMP),
  species (where applicable in PDMP reporting), and any other elements
  enumerated in the NCPDP SCRIPT / PDMP patient segment the IG targets.
- For each added row, map to the appropriate `Patient` element
  (`Patient.name.given[1]`, `Patient.name.prefix`, `Patient.name.suffix`,
  `Patient.gender`, `Patient.address.*`, `Patient.telecom`,
  `Patient.identifier[...]`, etc.), and add profile-level `mustSupport`
  or slice guidance in the PDMP Patient profile where the data element
  is required by PDMP reporting.
- Where an NCPDP element has no direct FHIR Patient counterpart, define
  (or reuse) an extension on Patient and document the mapping with a
  comment explaining the rationale.

#### Justification

The ticket's complaint is literally that the current mapping table is
incomplete relative to the NCPDP patient data element list the IG
claims to cover. Completing the table is the narrowest change that
resolves the reporter's concern and aligns IG documentation with the
NCPDP source. The Pharmacy work group's 2023-11-28 revert of the
"Considered for Future Use" resolution indicates the IG is active
again, so the original "defer because the IG is withdrawn" rationale
no longer applies. FHIR's `Patient` resource already accommodates
nearly all NCPDP patient fields, so most of the work is documentation
and mapping rather than new modelling.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hand-authoring a one-off expanded mapping table, adopt a
structured NCPDP-to-FHIR Patient mapping artifact in the IG:

- Publish the full NCPDP Patient data element set as a
  `ConceptMap` (NCPDP Patient segment codes → FHIR Patient element
  paths / extensions), rendered in the IG as the canonical mapping
  table.
- Tighten the PDMP Patient profile to slice `Patient.identifier`,
  `Patient.telecom`, and `Patient.address` according to NCPDP
  reporting needs, and mark elements `mustSupport` based on NCPDP
  conditionality.
- Keep only a short narrative section in the IG page; let the
  `ConceptMap` and profile drive the generated mapping documentation.

#### Justification

A `ConceptMap` / profile-driven approach is more maintainable than an
ad-hoc markdown table: when NCPDP revises its patient segment or when
the PDMP IG rebases on a newer US Core Patient, the mapping stays
machine-checkable. It also lets implementers consume the mapping
programmatically, which is the pattern other US Realm IGs (US Core,
USDI-derived IGs) have converged on. The trade-off vs. Disposition A
is higher up-front authoring effort (building a `ConceptMap` and
slicing the profile) in exchange for lower long-term maintenance and
a more usable artifact.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive – Insufficient Detail" on the
basis that the ticket does not enumerate which NCPDP patient data
elements are missing, which PDMP IG version or page the complaint
targets, or which FHIR elements the reporter believes should receive
the mapping. Ask the reporter (or the revived PDMP-on-FHIR project
team under PSS-2225) to file a new, element-specific ticket per
missing mapping so each can be triaged on its merits.

#### Justification

The ticket as written is effectively a work item ("there is more to
map") rather than a specific change request. Pharmacy traditionally
asks for ticket-per-element so votes can be recorded per concrete
change. Declining here is defensible because the revived PDMP IG
project (PSS-2225) is expected to re-scope the mapping holistically;
absorbing this open-ended ticket into project-level scope rather than
resolving it in place avoids a partial fix that later has to be
revisited. This is the least-risk option but also the least useful
to the IG authors.

---

### Recommendation

**Recommended disposition:** A

The ticket is narrowly and reasonably scoped: the IG's NCPDP Patient
mapping table is demonstrably incomplete, and the remedy is to
complete it. Disposition A directly delivers that remedy and
unblocks downstream ticket resolution (the reopen via PSS-2225
signals that Pharmacy intends to work these legacy PDMP issues, not
keep deferring them). Disposition B is attractive in the long run,
but a `ConceptMap`-first refactor is a larger piece of work than the
ticket asks for and is better tracked as its own PSS-2225-aligned
task once the element-level mappings in Disposition A are in place.
Disposition C under-serves the work group: the reporter's concern is
legitimate and specific enough (the NCPDP Patient segment is a
finite, externally-defined list), so asking for one ticket per
element would generate busywork without adding triage signal.
Recommend accepting and tasking the PDMP-on-FHIR project (PSS-2225)
to enumerate the remaining NCPDP patient elements and extend the
IG's Patient mapping and profile accordingly.
