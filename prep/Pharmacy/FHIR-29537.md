## Summary

[FHIR-29537](https://jira.hl7.org/browse/FHIR-29537) is a Pharmacy
work-group change request, originally filed in 2020 against the
US Prescription Drug Monitoring Program (PDMP) FHIR IG (now tracked
under the broader **US Medication FHIR Implementation Guide** spec
slot). The reporter raises a set of clarifying questions about the
IG's reliance on the **PMIX/NIEM** message format used by intermediaries
to bridge NCPDP requests to State PDMP systems. Specifically, the
ticket asks:

1. Whether PMIX/NIEM is already in production use by a majority of
   States and whether it is a mandatory requirement.
2. Whether NIEM is an ANSI-recognized standard and, if not, whether it
   is appropriate for an HL7 standard to depend on (or normatively
   reference) a non-ANSI-recognized format.
3. How implementations have validated the translation between
   HL7/NCPDP content and the PMIX/NIEM payloads exchanged with State
   PDMPs.

The ticket targets the IG's narrative sentence: *"The intermediary
typically converts this incoming NCPDP request to PMIX/NIEM format and
initiates a request to the State PDMP systems."*

It is essentially a request for justification and clarification of an
external-format dependency rather than a request for a specific
artifact change. The history is non-trivial: it was originally
disposed of as **Considered for Future Use** on 2023-01-18 (vote
4-0-0, John Hatem / Tim McNeil) on the grounds that the PDMP IG had
been withdrawn, then **reopened** by smrobertson on 2023-11-28 in
connection with the PSS-2225 project to revive the PDMP-on-FHIR work.
Cross-reference indices and unified search returned **no** related
Jira tickets, Zulip threads, or GitHub items for this key, and the
Confluence source is not currently configured in this Augury instance,
so the linked PSS-2225 page could not be fetched programmatically.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a **narrative-clarification** change request against the
revived PDMP-on-FHIR IG (under the *US Medication FHIR IG* spec slot
or its successor) and update the offending sentence so that the IG
explicitly:

1. Frames PMIX/NIEM as a **descriptive observation** about how some
   intermediaries operate today, not as a requirement of this IG. Make
   the sentence non-normative ("MAY", "in current deployments") rather
   than declarative.
2. Adds a footnote/sidebar that (a) names PMIX as the **Prescription
   Monitoring Information eXchange** and NIEM as the **National
   Information Exchange Model**, (b) acknowledges NIEM is a US federal
   information-sharing framework rather than an ANSI-accredited SDO
   standard, and (c) makes clear that conformance to this IG is
   defined entirely against the FHIR/NCPDP boundary it specifies — not
   against PMIX/NIEM.
3. Adds (or links to) an **Implementation Notes** section that points
   intermediary implementers at the relevant PMIX/NIEM specifications
   for the State-facing leg, and at any conformance/validation
   guidance the project sponsor has published, so the question
   "how have implementers validated the translation?" has a documented
   answer rather than being left implicit.

#### Justification

The ticket as written is not asking for a structural/resource change;
it is asking for justification and clarification of an external-format
dependency baked into IG narrative. The cleanest way to discharge that
request without reopening the IG's data model is to (a) make clear
the IG does **not** mandate PMIX/NIEM, and (b) give implementers
enough context (definitions, scope statement, validation pointers) to
answer the reporter's three questions from the spec itself. This is
consistent with HL7's normal handling of references to non-ANSI
information frameworks in adjacent ecosystems (we describe them, we
don't conform-test against them) and aligns with the reopen comment,
which explicitly ties the reopen to a project (PSS-2225) whose remit
is to revive the IG, i.e., a context in which narrative edits are in
scope.

---

### Disposition B: Alternative Approach

#### Proposal

**Batch-resolve as part of a PSS-2225 PDMP-IG narrative-cleanup pass.**
Rather than dispositioning FHIR-29537 in isolation, mark it as part of
a coordinated cleanup of the `PDMP-old-issues`-labelled backlog:

1. Pull every Jira ticket carrying the `PDMP-old-issues` label
   (Pharmacy WG, US Medication FHIR IG / former PDMP IG) into a single
   tracking sheet.
2. Triage them against the current PSS-2225 scope: classify as
   *narrative-only*, *structural*, or *out-of-scope-for-revived-IG*.
3. For FHIR-29537 specifically, apply the wording change from
   Disposition A as one entry in that batch, and use the same batch to
   answer (or formally defer, with rationale) the reporter's questions
   about ANSI recognition and translation validation in a single
   IG-level "PMIX/NIEM relationship" appendix rather than scattering
   the same explanation across multiple narrative edits.
4. Run the `repo-analysis` skill against the IG's source repo once
   PSS-2225 has identified it, so the actual edits land in the
   correct authoring root.

#### Justification

The ticket was reopened *because* PSS-2225 reopened the underlying
IG, and the comment trail explicitly says these items were deferred
"until the IG is taken up again by the project sponsor and/or the
workgroup." Several `PDMP-old-issues`-tagged tickets are likely to
raise overlapping concerns about external-format dependencies and
conformance scope; resolving them one at a time risks inconsistent
narrative ("PMIX/NIEM is descriptive" in one section, "the IG
requires PMIX/NIEM" in another). Batching also lets the workgroup
decide once on the IG's posture toward non-ANSI frameworks rather
than re-litigating it per ticket. The trade-off vs. Disposition A is
slower close-out for this individual ticket in exchange for higher
overall consistency and lower review cost.

---

### Disposition C: Decline

#### Proposal

**Persuasive — Not Persuasive with Mod.** Decline to make any spec
change in response to this ticket on the grounds that it consists of
*questions* rather than a concrete change request, and the underlying
narrative was already reviewed and dispositioned (as "Considered for
Future Use") when the IG was withdrawn. Close the ticket and instead
file a new, scoped ticket under PSS-2225 if and when the revived IG
re-introduces the PMIX/NIEM sentence and the workgroup wants
narrative tightening at that point.

#### Justification

This disposition is defensible on three grounds. First, the ticket
literally asks questions rather than requesting a specific change to a
resource, profile, value set, or constraint — the workgroup is not
obligated to make spec changes in response to information requests.
Second, the ticket is currently filed against a spec slot
(*US Medication FHIR IG*) that may or may not inherit the PDMP IG's
PMIX/NIEM language verbatim; reviewing the wording before the revived
IG actually reproduces it risks fixing something that is no longer
there. Third, the original 2023-01-18 vote (4-0-0) already reflected
a workgroup judgement that these items should be parked until the IG
is alive again — that judgement is still arguably correct, and the
2023-11-28 reopen could be read as procedural (because PSS-2225
reopened the IG) rather than as a substantive reversal of it. The
trade-off is leaving the reporter's questions formally unanswered in
the issue tracker until the IG re-emerges.

---

### Recommendation

**Recommended disposition:** B (with A as the fallback if PSS-2225
batching is not practical in this cycle).

The ticket is genuinely substantive — the reporter's questions about
the IG mandating a non-ANSI framework, and about translation
validation, are exactly the sort of questions that *should* be
answerable from the IG narrative — so a flat decline (C) leaves real
value on the table and re-creates the same review burden the next
time someone reads the IG's PMIX/NIEM sentence. At the same time,
disposition A in isolation risks producing a one-off narrative patch
that is inconsistent with how the rest of the `PDMP-old-issues`
backlog is treated under PSS-2225. Disposition B threads that needle:
it commits to making the wording change (so the reporter's concerns
are addressed in the IG itself), but it does so as part of a
coordinated, project-sponsored cleanup whose scope is set by
PSS-2225. It also gives the workgroup a natural moment to decide
once, and document, the IG's posture toward non-ANSI exchange
frameworks like NIEM — which is the underlying question this ticket
is really asking. If the workgroup judges that PSS-2225 is not yet
mature enough to absorb such a batch, fall back to A and apply the
wording change individually rather than declining.
