## Summary

The reporter notes that the `<FillStatus>` element used in the US PDMP / US
Medication FHIR IG does not capture whether a particular dispense was a
**partial fill**. NCPDP transactions carry a dedicated Partial Fill
Indicator data element, and the IG's `FillStatus` value set / element does
not currently provide an equivalent signal, so PDMP submissions and queries
cannot distinguish a partial fill from a complete fill.

The ticket was originally dispositioned **Considered for Future Use**
(2023‑01‑18, John Hatem / Tim McNeil, 4‑0‑0) on the basis that the IG had
been withdrawn. It was then **reopened on 2023‑11‑28** by smrobertson under
the renewed PSS‑2225 PDMP on FHIR project, indicating that Pharmacy intends
to revisit it when the IG is taken back up. No related Jira tickets, Zulip
discussions, or GitHub items were surfaced via cross-reference or search,
so this Summary is grounded solely in the ticket itself and the comment
trail.

## Proposed Dispositions

The ticket targets an IG (US PDMP / US Medication FHIR IG) that was
withdrawn and is being revived under PSS-2225. Without a current source
repo surfaced through cross-references, the dispositions below describe
the change in IG-authoring terms rather than naming a specific
authoring path; the workgroup should map the chosen disposition to the
revived project's repository when it is re-established.

### Disposition A: Accept As Requested

#### Proposal

Add an explicit Partial Fill Indicator to the `<FillStatus>` construct in
the US Medication / PDMP FHIR IG so that a dispense event can be reported
as a partial fill versus a complete fill, aligned to the NCPDP Partial
Fill Indicator data element. Concretely:

- Extend the `FillStatus` ValueSet (or its successor profile element) with
  codes that distinguish at minimum: `complete-fill`, `partial-fill`, and
  (where NCPDP defines them) `completion-of-partial-fill`.
- Bind the extended ValueSet to the `MedicationDispense.type` slot used
  by the IG's PDMP dispense profile, or — if `type` is already used for
  another purpose — introduce a dedicated extension
  `partial-fill-indicator` on the dispense profile referencing the new
  ValueSet.
- Update the IG narrative for the dispense / fill section to describe the
  new code semantics and provide a mapping table to the NCPDP Partial
  Fill Indicator field.

#### Justification

This matches the request literally and preserves direct traceability to
the NCPDP source data element, which is the canonical interoperability
contract for PDMP submissions in the US. PDMP recipients (state PMPs,
prescribers, pharmacists) genuinely need to distinguish partial fills to
compute remaining quantity, days-supply exhaustion, and refill-eligibility
windows; conflating them with complete fills creates clinical and
regulatory risk. Aligning to NCPDP terminology also minimizes mapping
burden for pharmacy systems already producing those transactions.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than extending `FillStatus` itself, model the partial-fill signal
on `MedicationDispense` using the resource's existing structural elements
plus a small, focused extension:

- Use `MedicationDispense.quantity` together with the prescribed
  quantity (resolved via `MedicationDispense.authorizingPrescription`) to
  let consumers compute "partial vs. complete" deterministically.
- Add a US-PDMP-scoped extension `partial-fill-indicator` (boolean or
  small code: `partial | complete | completion-of-partial`) on
  `MedicationDispense` to carry the explicit NCPDP indicator without
  changing the meaning of the `FillStatus` ValueSet.
- Optionally constrain `MedicationDispense.type` (TypeOfMedicationDispense)
  with US Core / US-Pharmacy guidance instead of inventing a new fill
  status code system.

#### Justification

This keeps `FillStatus` semantically stable (avoids breaking any existing
implementer that mapped to the legacy ValueSet) and reuses base-resource
mechanisms already in FHIR. It also separates the *fact* of partiality
(extension/derivable from quantities) from the *workflow status* of the
dispense, which is what `status`/`FillStatus`-style elements are usually
intended to represent. For a revived IG, a thin extension is cheaper to
publish, easier to deprecate later if the base resource gains a first-class
element, and less likely to fork pharmacy terminology.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the ticket targets a withdrawn IG
and that a US-PDMP-specific `FillStatus` extension is out of scope until
the IG is formally re-balloted. Mark as **Persuasive with Mod / Considered
for Future Use** pending the PSS-2225 PDMP-on-FHIR project producing a
revised draft, at which point the partial-fill requirement should be
addressed as part of that draft rather than as an isolated change to a
withdrawn artifact.

#### Justification

The IG was explicitly withdrawn (per the 2023-01-18 vote) and is only now
being revived under PSS-2225; making targeted edits to a withdrawn
specification without a coherent ballot package risks publishing
inconsistent guidance. The underlying clinical need is real, but it is
better captured as a requirement against the revived IG than as a point
fix here. This disposition has historical precedent — it is the
disposition that was originally applied before smrobertson reverted it.

---

### Recommendation

**Recommended disposition:** A

The clinical and regulatory case for distinguishing partial from complete
fills in PDMP data is strong (it directly affects remaining-quantity and
refill-eligibility calculations), the source data element exists and is
well-defined in NCPDP, and the ticket has been explicitly reopened under
PSS-2225 precisely so issues like this can be resolved as the IG is
revived. Disposition A delivers the clearest, most discoverable solution
for downstream PDMP implementers and avoids leaving the indicator
implicit in computed quantities (Disposition B), which historically has
been a source of inconsistent reporting between dispensers. Disposition C
restores the previous "defer indefinitely" stance, which the workgroup
already overturned when reopening the ticket. If the workgroup is
concerned about ValueSet churn, Disposition B is a reasonable fallback,
but the recommendation is to absorb the small terminology change now
while the IG is being re-shaped rather than after publication.
