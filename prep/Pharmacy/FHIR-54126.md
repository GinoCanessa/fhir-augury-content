## Summary

Brett Marquard is challenging the wording of the recently-applied change to
`MedicationStatement.medication` (made under
[FHIR-53425](https://jira.hl7.org/browse/FHIR-53425), resolved
"Persuasive with Modification") that says the element "now explicitly supports
representing statements such as 'no medications are known / taken.'" The
reporter observes that "no medications known" and "no medications taken"
are semantically distinct assertions, yet the example value set being
shipped contains only the SNOMED CT concept `787481004 |No known medications
(situation)|`. He proposes the workgroup either (a) tighten the prose and
the value-set scope so that only "No known medications" is conveyed via this
mechanism, or (b) add a separate concept that explicitly covers "no
medications taken." The change is therefore a clarification/follow-up on the
FHIR-53425 resolution rather than a new feature request.

## Related Jira Tickets

### [FHIR-53425: Change MedicationStatement.medication to .code to support "No Medication"](https://jira.hl7.org/browse/FHIR-53425)

Originating Pharmacy ticket (Applied, "Persuasive with Modification") that
introduced the very change FHIR-54126 is now critiquing. FHIR-53425 asked to
align `MedicationStatement` with `AllergyIntolerance` by renaming
`.medication` to `.code` and broadening the value set to allow negation
codes; the workgroup declined the rename but agreed to (1) widen the
definition of `MedicationStatement.medication` to allow "no known
medication," (2) add some terms to the example value set, and (3) add a note
that this is **not** the recommended approach. FHIR-54126 surfaces the gap
that the resolution text mentioned both "no medications known" and "no
medications taken," but only the former actually made it into the value
set.

Reporter: Jose Costa-Teixeira. Assignee: Unassigned.

## Related Zulip Discussions

### [Medication > No current medications](https://chat.fhir.org/#narrow/stream/Medication/topic/No%20current%20medications)

Long-running thread (Oct–Nov 2025) that drove FHIR-53425 and frames
FHIR-54126. Heath Frankel (AU) opened it asking how to record "no current
medications" in alignment with US Core / AU Core and the IPS IG, which uses
SNOMED CT `787481004 |No known medications|` on `MedicationStatement`. Jose
Costa-Teixeira argued that `MedicationStatement` is the right home (since
`MedicationRequest` carries the wrong semantics — "patient should not take
meds" rather than "patient is not taking meds") and noted `787481004` is a
*finding*, so it is a poor fit for `MedicationStatement.medication` and
might better suit an `Observation`. Jean Duteau reported Canadian practice
of using `Observation` for "no meds" and "no allergies." Peter Jordan
pushed for IPS alignment and was uncomfortable framing nil-knowns as
observables. Stephen Chu and others raised that `MedicationStatement` is a
point-in-time assertion and that combining `effectiveDateTime` +
`status=active` makes "no known medications" semantically fragile for CDS
consumers months later; he preferred a dedicated structure but conceded
"that ship has sailed." Grahame Grieve drove the procedural decision to
ship the change in R6 ballot ("change the definition, add example codes,
add a note that this is not recommended — easier to revise in ballot than
add later") with the caveat that any such statement must carry an
`effective[x]` or `dateAsserted` and that codes must clearly distinguish
"no medications of any kind" from narrower assertions like "no NSAIDs." A
follow-up `/poll` from Jose enumerated four candidate approaches
(allergy-style negation on the resource, DAR on `MedicationStatement`, a
dedicated `Observation`, or a new modifier element). Heath Frankel reported
that AU PS will use AU-only SCT `1234391000168107 |No known current
medicines|` because it carries the temporal "current" context that the
international concept lacks — a code Grahame is "troubled" to use but
considers another sailed ship.

Participants: Heath Frankel, Jose Costa-Teixeira, Peter Jordan, Jean
Duteau, Lloyd McKenzie, Eric Haas, Stephen Chu, Grahame Grieve, Rutt
Lindström.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the FHIR-53425 change so that the `MedicationStatement.medication`
mechanism *only* asserts "no known medications," matching the single
SNOMED CT concept (`787481004 |No known medications (situation)|`)
currently in the example value set:

- Update `MedicationStatement.medication` definition prose (and any
  comment/usage notes added by FHIR-53425) to drop the phrase "no
  medications are known / taken" and replace it with the narrower "no
  known medications" formulation.
- Leave the example value set as-is (single concept `787481004`).
- Update the cautionary note added by FHIR-53425 ("this is not the
  recommended approach") to call out that the mechanism does **not**
  cover the distinct assertion "patient is not taking any medication"
  and to refer implementers to `Observation` or to the discussion
  recorded under FHIR-53425 for that case.

#### Justification

Matches the principle established by Grahame and Jose in the Zulip
thread that "no known medications of any kind" must not be conflated with
narrower or differently-scoped assertions. Aligns wording with the only
concept actually shipped in the value set, removing a documentation/value-set
mismatch that implementers would otherwise have to reconcile. Preserves the
FHIR-53425 resolution intent (allow nil expression on
`MedicationStatement.medication`) without expanding scope into "not taking,"
which the thread shows is contested and arguably belongs in `Observation` or
in a future modifier element.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the broader "known / taken" framing in the definition prose, but back
it up by extending the example value set so that both flavors are
representable:

- Add a second example concept covering "no medications taken" (e.g., a
  SNOMED CT concept for "patient not currently taking medication," or a
  HL7 `data-absent-reason`-style code) alongside `787481004`.
- Document each concept's intended meaning inline (a small table or
  bulleted note) so implementers can choose between "we asked and the
  patient confirmed no current intake" and "we have no record of any
  medications."
- Reinforce Grahame's guidance from the Zulip thread that any such
  `MedicationStatement` SHALL populate `effective[x]` or `dateAsserted`
  and SHOULD distinguish "no medications of any kind" from narrower
  negations (e.g., drug classes).

#### Justification

Follows Disposition B in the spirit of FHIR-53425's resolution ("change the
example value set to include some terms") while honouring the reporter's
objection that the current single-concept value set under-delivers on the
prose. It also accommodates AU PS's real-world distinction between "no
known" and "no current" medicines (Heath Frankel reported AU using the
SCT-AU concept `1234391000168107 |No known current medicines|` precisely
because temporal context matters). The trade-off versus Disposition A is a
larger surface area to maintain and a higher risk of misuse, which is why
the cautionary note FHIR-53425 already added must remain prominent.

---

### Disposition C: Decline (Not Persuasive — Already Addressed)

#### Proposal

Decline FHIR-54126 on the grounds that the FHIR-53425 resolution explicitly
flagged this approach as **not recommended** and that the issue raised here
is best addressed by other mechanisms (`Observation` with a finding code,
or a future modifier element à la Jose's Option 4 in the Zulip poll), not
by further enlarging the `MedicationStatement.medication` value set or
re-litigating the wording. Keep the current text and the single example
concept; rely on the FHIR-53425 cautionary note to steer implementers
elsewhere.

#### Justification

The Zulip thread shows real disagreement on whether
`MedicationStatement.medication` should be carrying *any* nil assertion.
Stephen Chu was explicit that he is "particularly not comfortable" with
the approach, and Grahame himself argued the mechanism should ship only
because it is "procedurally easier to do it now and revise it in ballot."
Declining FHIR-54126 keeps the door open for a cleaner R6 ballot
resolution (dedicated `Observation` guidance, or the proposed "not
taking/taken" modifier) without adding more codes — and more semantic
ambiguity — to the example value set in the meantime.

---

### Recommendation

**Recommended disposition:** A

The ticket is fundamentally a documentation/scope mismatch: the prose
added by FHIR-53425 advertises two distinct capabilities ("known" and
"taken") but the value set delivers only one. Disposition A resolves the
mismatch with the smallest possible footprint, keeps the FHIR-53425
resolution intact, and avoids re-opening the contested question of whether
`MedicationStatement.medication` should carry "patient is not taking"
semantics — a question the Zulip thread shows is unresolved and which Jose
explicitly proposed handling via a future modifier element or
`Observation`. Disposition B is defensible but expands a mechanism the
workgroup has already labelled "not recommended," and would commit Pharmacy
to picking a second example code (likely SNOMED) before the broader
"current vs. known" temporal-scope debate (per the AU PS experience) is
settled. Disposition C reads as procedurally tidy but leaves a known
contradiction in the published spec — exactly the kind of documentation
defect ballot reviewers will (rightly) re-file.
