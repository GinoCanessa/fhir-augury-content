## Summary

The ticket asks the Pharmacy work group to clarify two related conformance
gaps in the US Prescription Drug Monitoring Program (PDMP) on FHIR
Implementation Guide. First, when the PDMP responder encounters an error,
should the IG mandate a mapping between FHIR-level error semantics
(e.g., `OperationOutcome` issues, HTTP status codes) and the equivalent
error handling defined by other PDMP-relevant standards (e.g., NCPDP
SCRIPT, ASAP Web Services)? Second, when the responder cannot match the
requested patient and therefore has no dispense history to return, how
should that "no match" condition be conveyed — distinct from the
"patient matched but no dispenses on file" outcome that already has a
natural FHIR representation (an empty search Bundle).

The ticket has no Jira, Zulip, GitHub, or Confluence cross-references on
record, so it must be evaluated on its own. Comment history shows the
ticket was previously closed as **Considered for Future Use** in
January 2023 because the IG had been withdrawn, then **reopened** in
November 2023 under the reactivated PSS-2225 project. It is currently
**Triaged / Highest** with no assignee.

## Proposed Dispositions

Because there are no GitHub cross-references and no published PDMP-on-FHIR
authoring repo surfaced through FhirAugury, the dispositions below are
framed at the IG-content level rather than naming specific source files.
If the workgroup adopts Disposition A or B, the editor will need to apply
the changes to the active PDMP IG authoring repo once it is re-stood-up
under PSS-2225.

### Disposition A: Accept As Requested

#### Proposal

Add an explicit **Error Handling and Negative Responses** section to the
PDMP on FHIR IG conformance pages with two normative parts:

1. **Error mapping table.** Define a one-way mapping from FHIR error
   responses (HTTP status + `OperationOutcome.issue.code` /
   `details.coding`) to the equivalent error/exception codes in each
   PDMP-relevant non-FHIR standard the IG already references (at minimum
   ASAP Web Services and the NCPDP SCRIPT request/response patterns
   used by PMP gateways). Require responders to populate
   `OperationOutcome.issue` with the listed codes, and require the
   mapping table itself to be authoritative for cross-standard
   interoperability.
2. **"No patient match" response shape.** Specify that a request that
   completes successfully but cannot identify a patient SHALL NOT be
   conflated with "patient matched, no dispenses." The responder SHALL
   return an HTTP 200 search-set `Bundle` whose `total` is `0` **and**
   include an `OperationOutcome` entry with
   `issue.severity = information`, `issue.code = not-found`, and a
   defined `details.coding` (e.g., a new IG-defined CodeSystem with at
   least `no-patient-match` and `ambiguous-patient-match`) that calls
   out the no-match condition explicitly. The "patient matched but zero
   dispenses" case continues to be an empty Bundle with no
   `OperationOutcome` warning.

#### Justification

This is the change the ticket literally asks for: a normative answer to
both questions (error mapping + no-match representation). The chosen
mechanisms (`OperationOutcome` for error/informational signaling,
empty search Bundle for "no data", an IG-defined CodeSystem for the
PDMP-specific reasons) are the conventional FHIR patterns and align
with how other US Realm IGs handle similar concerns. Distinguishing
"no match" from "no dispenses" is also clinically and
operationally important — implementers in PMP/dispense workflows must
not treat an unidentified patient as evidence that the patient has
no controlled-substance history.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than maintaining a normative cross-standard error mapping inside
the IG, scope the IG's conformance to FHIR semantics only and address
the underlying need with two narrower changes:

1. **FHIR-only error conformance.** Require responders to use standard
   FHIR error responses (`OperationOutcome` with HL7-defined
   `issue-type` codes and standard HTTP status codes) and explicitly
   state that mapping to NCPDP SCRIPT / ASAP Web Services error codes
   is **out of scope** for this IG and is the responsibility of the
   gateway/integration layer. Provide an informative (non-normative)
   appendix with example mappings rather than a normative table.
2. **No-match as a dedicated SearchParameter outcome.** Define a single
   informational `OperationOutcome` reason code
   (`no-patient-match`) returned alongside an empty search-set Bundle,
   and document explicitly that the absence of that
   `OperationOutcome` means "patient was identified, no dispenses
   found." Skip the broader "ambiguous match" / multi-state taxonomy
   unless concrete use cases are brought forward.

#### Justification

Maintaining a normative cross-standard error mapping is a recurring
maintenance burden — every revision of NCPDP SCRIPT or an ASAP profile
risks invalidating IG-published normative content, and PMP gateways in
practice already do this translation. Keeping the IG FHIR-pure protects
the IG from churn driven by other SDOs while still answering the
"no patient match" half of the ticket with a concrete, testable
mechanism. This disposition trades some completeness against
Disposition A for a smaller, more durable normative surface.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as **Not Persuasive** (or **Considered for Future Use**
again) on the grounds that (a) FHIR core already provides sufficient
mechanisms — `OperationOutcome`, HTTP status codes, and empty search
Bundles — to express both error scenarios and "no results" outcomes,
and (b) the PDMP IG should rely on those generic mechanisms plus its
existing security/`SMART on FHIR` profile rather than introducing
PDMP-specific error or no-match semantics.

#### Justification

This is defensible only if the workgroup believes that implementers can
reasonably infer the right behavior from FHIR core alone. The risk —
which is what the ticket calls out — is that without IG-level guidance,
two PDMP responders may legitimately use different status codes and
different shapes for "no patient match", undermining cross-state PMP
interoperability. Decline is the weakest option because the ticket is
flagged **Highest** priority, was explicitly **reopened** when the IG
project restarted under PSS-2225, and was originally raised by an
implementer review of the IG's conformance section (comment #24 from
the IG comment spreadsheet). Declining a Highest-priority,
implementer-sourced conformance gap should require strong evidence
that the gap is theoretical, which is not present here.

---

### Recommendation

**Recommended disposition:** A

The ticket asks two well-formed, implementer-facing conformance
questions on a Highest-priority IG that has been deliberately reopened
under PSS-2225. Disposition A directly answers both questions with
standard FHIR mechanisms (`OperationOutcome`, search Bundle, an
IG-defined reason CodeSystem) and also provides the cross-standard
error mapping that PMP gateways need in order to bridge FHIR
responders into the existing NCPDP / ASAP ecosystem — which is the
distinguishing characteristic of PDMP relative to a pure FHIR IG.
Disposition B is a reasonable fallback if the workgroup judges that
maintaining the cross-standard mapping is too costly, and the
no-match half of B can be lifted into A unchanged. Disposition C
should be rejected: leaving "no patient match" indistinguishable from
"no dispenses on file" is a clinical-safety-relevant ambiguity in a
controlled-substances context, and the workgroup has already once
deferred this ticket and then chosen to reopen it.
