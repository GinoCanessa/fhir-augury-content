## Summary

Frank McKinney (Pharmacy WG) is requesting that the CARIN Real-time
Pharmacy Benefit Check (RTPBC) IG add a **non-PHI request option** for
use against data sources that can return benefit or pricing information
without requiring patient demographics. Today the STU1 RTPBC Request
mandates a `Patient` resource, which forces submitters to release
protected health information even in scenarios (e.g., generic price
lookups, formulary/benefit checks keyed only on plan + drug + general
location) where the data source neither needs nor uses it. The ticket
proposes a parallel non-PHI request flow alongside the existing
PHI-bearing request, leaving the choice of which to send to the
submitter based on what the downstream data source supports.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items linked to this ticket in the available indexes, so the request
stands on its own. The resolution text (already drafted, marked
**Persuasive**) outlines a concrete implementation: new "Non-PHI"
profiles for `Patient`, `MedicationRequest`, `Claim`, and the request
`Bundle`; modifications to the request `MessageHeader` /
`MessageDefinition` to admit either flavor; and updates to the
Information Content, Use Cases, and Business Rules narrative pages.

## Details

**Description:**

> it is not necessary to provide Patient details when retrieving certain
> types of benefit-related and price-related information. In those
> circumstances, using the STU1 RTPBC Request – which includes a
> mandatory Patient resource – unnecessarily releases protected health
> information to the data source.
>
> This is a request to include a non-PHI request option in the IG to be
> used when the data source is able to return the desired benefit or
> pricing information without receiving patient information.

**Resolution Description (already drafted, "Persuasive"):**

> Create resources and guidance for submission of an RTPBC request that
> excludes patient information – to be used in situations where the data
> source can supply a response without patient details.
>
> New resource profiles:
> - **Non-PHI Patient** – using base FHIR `Patient` as the parent profile
>   because PHI elements such as `name` and `birthDate` would inherit
>   `Must Support` from US Core, which would be contradictory and
>   confusing to implementers.
> - **Non-PHI MedicationRequest** to convey minimal non-PHI information
>   incl. medication, quantity, preferred pharmacy, general location /
>   ZIP. Use base `MedicationRequest` as parent b/c US Core requires
>   reference to US Core Patient.
> - **Non-PHI RTPBC Claim** that references the non-PHI Patient and
>   MedicationRequest and doesn't require `careTeam.provider` or
>   `insurance.coverage`.
> - **Non-PHI Request Bundle** containing the non-PHI resources above
>   - omit practitioner and coverage
>   - make pharmacy organization optional
> - Modify the **RTPBC Request `MessageHeader`** and **`MessageDefinition`**
>   to allow PHI or non-PHI content.
>
> Update the **Information Content** page in the IG with the non-PHI
> request option. Add guidance to the **Use Cases** page and modify
> graphics as needed. Update the **Business Rules** page with
> discussion of when to use the non-PHI vs PHI requests.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution as drafted. Concretely, in the CARIN RTPBC IG
(STU2 candidate):

- Add four new profiles:
  - `RtpbcNonPhiPatient` — parent: base FHIR `Patient` (deliberately
    *not* US Core Patient, to avoid inheriting `Must Support` on
    `name`/`birthDate`/`identifier`/etc.). Constrain or zero out PHI
    elements; require only the minimum needed to anchor the bundle
    (e.g., a synthetic / opaque `id`).
  - `RtpbcNonPhiMedicationRequest` — parent: base FHIR
    `MedicationRequest`. `Must Support`: `medication[x]`,
    `dispenseRequest.quantity`, `dispenseRequest.performer` (preferred
    pharmacy), and an extension or existing element carrying a general
    location / ZIP. `subject` references the Non-PHI Patient profile.
  - `RtpbcNonPhiClaim` — references the Non-PHI Patient and Non-PHI
    MedicationRequest. Relax/remove the existing PHI-flavor
    requirements on `careTeam.provider` and `insurance.coverage`.
  - `RtpbcNonPhiRequestBundle` — `Bundle` profile that contains the
    Non-PHI Claim + MedicationRequest + Patient (and optionally a
    pharmacy `Organization`); explicitly omits `Practitioner` and
    `Coverage` slices.
- Update the **RTPBC Request `MessageDefinition`** focus list (and the
  request `MessageHeader` profile) to admit either the existing PHI
  request bundle or the new Non-PHI request bundle.
- Narrative updates to the **Information Content**, **Use Cases**
  (including refreshed sequence/flow graphics), and **Business Rules**
  pages, with explicit decision guidance on when to send a Non-PHI vs
  PHI request.

#### Justification

The reporter's privacy argument is sound and aligns with HIPAA's minimum
necessary standard and broader FHIR guidance to avoid transmitting PHI
where it is not required for the operation. Splitting into a parallel
profile family (rather than relaxing the existing PHI request) preserves
backward compatibility for STU1 implementers while providing a cleanly
labeled non-PHI path. Choosing **base FHIR `Patient`** as the parent
(per the resolution) is the right call: parenting on US Core Patient
would forcibly inherit Must Support on name/DOB/identifiers, which
directly contradicts the privacy intent and would mislead implementers
about what they are obligated to send. The same reasoning applies to
basing the Non-PHI MedicationRequest on core (not US Core), since US
Core MedicationRequest mandates a reference to US Core Patient. The
`MessageDefinition` change is the minimum-touch way to let a single
endpoint accept both flavors.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a parallel family of "Non-PHI" profiles, take a
**slice-and-extension** approach on the existing RTPBC profiles plus a
**capability/operation flag**:

1. Tag the request `Bundle` with a "non-PHI" usage extension (e.g.,
   `rtpbc-request-mode = phi | non-phi`) on `MessageHeader` or the
   request Bundle itself.
2. In the existing PHI-flavor profiles, weaken the constraints on
   `Patient`, `careTeam.provider`, `insurance.coverage`, `Practitioner`,
   and `Coverage` from `Must Support 1..1` to `0..1` and add an
   invariant: *"when `request-mode = non-phi`, these elements SHALL be
   absent or contain no PHI."*
3. Express server support (and per-data-source support) for the non-PHI
   mode via the IG's `CapabilityStatement` rather than via two distinct
   request shapes.

This keeps the profile count flat, leaves implementers with a single
RTPBC request profile to learn, and uses an invariant + extension to
police the privacy posture.

#### Justification

The trade-off vs. Disposition A is **fewer artifacts and a single
canonical request shape**, at the cost of a less self-evident contract:
implementers have to read the invariant to know what is allowed in
each mode, and validators have to evaluate the extension before
applying constraints. It also avoids the awkwardness of having two
near-identical Bundle profiles that diverge only by what they require.
However, this approach is harder to explain on the IG narrative pages,
weaker as a static contract (a tooling-only validator that ignores the
extension would not catch PHI leakage), and goes against the resolution
author's stated reason for *not* parenting on US Core — namely,
removing Must Support inheritance — because the existing PHI profiles
likely already inherit US Core. Disposition A's clean separation is
therefore more defensible if PHI minimization is the primary driver.

---

### Disposition C: Decline

#### Proposal

Decline the request. State that the existing STU1 RTPBC Request is the
single supported shape, and that implementers needing to suppress PHI
should populate the required `Patient`, `Practitioner`, and `Coverage`
slots with **opaque or pseudonymous identifiers** (a "minimum-PHI"
convention), documented as guidance only — without introducing new
profiles, a new Bundle, or `MessageDefinition` changes.

#### Justification

A defensible decline rationale would be (a) scope: STU1 is in the
field, and adding a parallel profile family plus `MessageDefinition`
changes is a non-trivial STU2 work item; (b) interoperability:
implementers and intermediaries currently keyed on a single request
shape would need to detect and route two; (c) sufficiency of an
existing mechanism: pseudonymization can in principle satisfy the
minimum-necessary rule without changing the wire contract.

This rationale is weak on the merits, however. Pseudonymization still
requires the requester to *transmit* a `Patient` resource and to invent
fake names/DOBs to satisfy US Core inheritance, which both leaks
metadata and creates operational risk (mistakenly sending real values).
The use case (price/benefit lookups that do not need the patient) is
real and recurring, and the resolution author has already done the
design work. Decline should only be selected if the WG concludes the
non-PHI use case is rare enough to leave unaddressed in this IG.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been worked through to a **Persuasive**
resolution with a concrete, internally consistent design, and the
underlying privacy argument is unambiguous: it is not appropriate to
ship PHI to a data source that does not need PHI to answer. Disposition
A's parallel-profile approach is the cleanest way to make the privacy
posture **statically visible in the profile tree** — implementers can
see at a glance whether they are populating an instance that is allowed
to carry PHI or not, and validators can enforce it without needing
extension-aware logic. The deliberate choice to parent on base FHIR
`Patient` (rather than US Core) is the linchpin and is correctly
motivated by US Core's Must Support inheritance on PHI elements;
collapsing this into Disposition B's single-profile-with-invariant
loses that benefit. The cost is a handful of new profiles and a small
`MessageDefinition` change — proportional to the value delivered. The
WG should focus its discussion on (1) the exact element set retained on
the Non-PHI MedicationRequest (especially how "general location" is
expressed — ZIP via extension vs. an `Address` constrained to postal
code only), and (2) whether the pharmacy `Organization` should be
optional or required for the non-PHI flow given matching/routing
implications downstream.
