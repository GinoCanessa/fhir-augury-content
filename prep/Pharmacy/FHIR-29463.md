## Summary

The ticket asks how the FHIR representation of a NCPDP SCRIPT 2017071
**Requesting Facility identification** (specifically the facility's
`<DEANumber>` and `<NPI>`) should be mapped when a prescriber performs a
PDMP query. The reporter notes that the existing `Prescriber/Identification`
mapping is not appropriate (those identifiers describe the prescriber, not
the facility), and the `Pharmacy/Identification` mapping is also not
appropriate because the query is not originating from a pharmacy. NCPDP
SCRIPT 2017071 already provides a `<Facility>/<Identification>` element
intended for exactly this case, and the ticket asks the workgroup to define
the corresponding FHIR mapping path.

The ticket was originally raised against the US Prescription Drug
Monitoring Program (PDMP) FHIR Implementation Guide; the workgroup voted
2023-01-18 to mark it "Considered for Future Use" (4-0-0, Hatem/McNeil)
because the PDMP IG was withdrawn, then reopened it on 2023-11-28 in
connection with the project reboot referenced as
[PSS-2225 PDMP on FHIR](https://confluence.hl7.org/display/PHAR/US+Prescription+Drug+Monitoring+Program+%28PDMP%29+FHIR+Implementation+Guide).
It currently sits under the "US Medication FHIR Implementation Guide"
specification in Jira but is functionally a PDMP IG ticket awaiting that
IG's revival.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define an explicit FHIR mapping for NCPDP SCRIPT 2017071
`<Facility>/<Identification>` in the (revived) US PDMP FHIR
Implementation Guide. Concretely:

- Add a row to the IG's NCPDP-to-FHIR mapping table that maps
  `Facility/Identification/DEANumber` →
  `Organization.identifier:DEA.value` (with system
  `http://hl7.org/fhir/sid/us-dea`) and
  `Facility/Identification/NPI` → `Organization.identifier:NPI.value`
  (with system `http://hl7.org/fhir/sid/us-npi`).
- In the PDMP query message bundle/profile, carry the requesting
  facility as a `Reference(Organization)` distinct from the
  prescriber's `Practitioner` and from any `Pharmacy` actor —
  e.g. `MessageHeader.sender` (when modelled as messaging) or a new
  `Task.requester` / `Task.location` / context-encounter
  `Encounter.serviceProvider` slot, depending on the IG's chosen
  interaction pattern.
- Slice `Organization.identifier` to require at least one of `NPI` or
  `DEA` (matching NCPDP's "Requesting Facility ID" element) and
  document binding/cardinality consistent with the rest of the IG.

#### Justification

This is exactly what the ticket asks for: a published, normative path in
the IG's mapping table so implementers do not have to invent their own
representation of the requesting facility's identifiers. The DEA and
NPI identifier systems already exist as well-known FHIR identifier
namespaces, and `Organization.identifier` is the conventional FHIR
landing spot for facility-scoped identifiers, so the change introduces
no new mechanism — only an explicit mapping. Doing this in the PDMP IG
(rather than the US Medication IG it is currently filed under) keeps
the change localized to the specification that originally raised the
gap.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve the mapping using existing FHIR mechanisms without inventing a
new IG slot or extension, and document that resolution as guidance:

- Map `Facility/Identification` to **`Organization.identifier`** with
  two pre-defined identifier slices:
  - `DEA` — system `http://hl7.org/fhir/sid/us-dea` (matches
    `<DEANumber>`).
  - `NPI` — system `http://hl7.org/fhir/sid/us-npi` (matches `<NPI>`).
- Carry the requesting facility in the query as the **`Encounter.serviceProvider`**
  reference on the contextual `Encounter`, or — if there is no
  Encounter resource in the query interaction — as
  **`Task.owner`/`Task.location.managingOrganization`** chained from the
  PDMP query Task. This reuses the same `Reference(Organization)` /
  `Organization.identifier` pattern US Core already profiles, with no
  new extension and no IG-specific element.
- Add a single normative note in the mapping table that explicitly
  contrasts `Prescriber/Identification` (→ `Practitioner.identifier`),
  `Pharmacy/Identification` (→ `Organization.identifier` of the
  pharmacy actor), and `Facility/Identification` (→
  `Organization.identifier` of the requesting facility), so the three
  NCPDP identification slots map to three distinct FHIR resource
  instances rather than three locations on one resource.

#### Justification

Disposition A and B reach the same end state for the wire: an
Organization with NPI/DEA identifiers. B differs by (a) deliberately
**not** introducing any new IG-defined element or extension, and (b)
explicitly disambiguating the three NCPDP `Identification` slots with
guidance text. That disambiguation is the actual root cause of the
reporter's confusion; without it, future implementers will hit the same
question even if a Facility row is added. B also keeps the IG aligned
with US Core's existing identifier-slicing pattern, which lowers the
implementation burden and reduces the risk of conformance drift if the
US PDMP IG is later harmonized with US Core or the US Medication IG.
The trade-off vs. A is that B leans on guidance text rather than a
distinct mapping row, which some workgroup members may consider less
discoverable.

---

### Disposition C: Decline (Defer Pending IG Revival)

#### Proposal

Close the ticket as **Considered – No Action Required, Pending IG
Resumption**: re-affirm the 2023-01-18 disposition (which was reverted
on 2023-11-28 only because PSS-2225 was filed). Until the US PDMP FHIR
IG is formally re-chartered with an active project sponsor, an active
authoring repository, and a current ballot target, the workgroup
should not cut a normative mapping for `Facility/Identification`.
Re-open and re-triage this ticket, together with the rest of the
`PDMP-old-issues` set, at the start of the next PDMP IG cycle.

#### Justification

The ticket has been open since 2020 against an IG that was withdrawn,
was deferred by formal vote in 2023, and was reopened only as a
bookkeeping consequence of PSS-2225 being filed — not because a
substantive design decision was reached. There is no active authoring
repo for the US PDMP IG visible in FhirAugury, no Zulip discussion, and
no recent activity on the ticket. Cutting a mapping in isolation now
risks landing it in the wrong specification (the ticket is currently
filed under the US Medication FHIR IG, which is *not* the right home
for an NCPDP query mapping) and being inconsistent with whatever
interaction model the revived IG ultimately chooses (messaging vs.
Task-based vs. operation-based query). Deferring is honest about the
ticket's status and is the lowest-risk path.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is the right starting point for workgroup discussion. The
underlying question — "which FHIR element carries the requesting
facility's NPI and DEA when a prescriber queries PDMP?" — has a clean
answer in existing FHIR mechanics (`Organization.identifier` sliced by
the standard DEA/NPI systems, referenced from
`Encounter.serviceProvider` or `Task.owner`), and that answer can be
documented as guidance without waiting on the IG's full re-charter.
Disposition A is acceptable but introduces an explicit IG-specific
mapping row whose phrasing will likely need to be re-litigated once
PSS-2225 settles on an interaction pattern; B's guidance-first framing
is more durable across that uncertainty. Disposition C is defensible
on process grounds — and the workgroup may legitimately prefer it given
the IG's withdrawn status — but it leaves the reporter's concrete
question unanswered for another IG cycle, and the disambiguation
between Prescriber/Pharmacy/Facility identification is a low-cost win
that should not have to wait. If the workgroup elects C, the
recommendation is to at least record the B-style guidance in the
ticket's resolution comment so the next cycle inherits the analysis.
