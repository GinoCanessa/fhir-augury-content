## Summary

The submitter (Frank McKinney) asks the Pharmacy work group to consider
adding deductible-accumulator information to the Consumer RTPBC response
so that a Real-Time Prescription Benefit response can convey the
patient's deductible status alongside the cost estimate. The request is
grounded in NCPDP RTPB Standard Version 11 (January 2021), which added
two situational fields — Accumulated Deductible Amount (NCPDP field
512-FC) and Remaining Deductible Amount (513-FD) — that are returned for
both the submitted product and any alternative products on an RTPB
response. The business need is to let the prescriber or pharmacist
explain to the patient how filling the identified medication would
affect their deductible position. The ticket includes worked examples
from the NCPDP RTPB IG showing how the accumulator values are reported
relative to the Patient Pay Amount and existing on-file accumulators,
and notes the underlying NCPDP requirement: the fields are required when
the plan provides details of how the product could impact the patient's
deductible status.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items
were surfaced for FHIR-36294, so the request stands on its own. The
ticket has been resolved as **Persuasive**, with the documented
resolution being to add `accumulated-deductible` and
`remaining-deductible` codes to the RTPBC Patient Pay Type code system
used in the RTPBC response (the FHIR ClaimResponse profile).

## Details

**Description:**

To support reporting deductible status information at time of inquiry,
NCPDP added to the RTPB Standard Version 11 (published in January 2021)
situational fields for communicating Accumulated Deductible Amount
(512-FC) and Remaining Deductible Amount (513-FD) for submitted and
alternative products.

---

**Business Need/Purpose:**

Used by the Provider or Pharmacist to explain the patient's deductible
status if they were to obtain the identified medication.

---

**Information about data elements from NCPDP Data Dictionary:**

*Accumulated Deductible Amount:*

Amount in dollars met by the patient/family in a deductible plan.
Field length: 11; field format: s9(9)v99; Format=s$$$$$$$$$cc.

*RTPB Example:* A Real-Time Prescription Benefit Request is submitted
where the Patient Pay Amount is determined to be $50 and the
beneficiary's current accumulated deductible amount is $100. The
Accumulated Deductible Amount returned on the RTPB response would be
$150. This field would reflect: `1500{`. If a subsequent RTPB request is
submitted, the same current accumulated deductible amount of $100 would
be leveraged.

*Remaining Deductible Amount:*

Amount not met by the patient/family in the deductible plan.
Field length: 11; field format: s9(9)v99; Format=s$$$$$$$$$cc.

*RTPB Example:* A Real-Time Prescription Benefit Request is submitted
where the Patient Pay Amount is determined to be $50 and the
beneficiary's current remaining deductible amount is $400. The
Remaining Deductible Amount returned on the RTPB response would be $350
and this field would reflect: `3500{`. If a subsequent RTPB request is
submitted, the same current remaining deductible amount of $350 would
be leveraged.

Use of deductible accumulator data elements in the RTPB Standard
Implementation Guide: required when the plan provides details of how
the product could impact the patient's deductible status.

---

**Pricing guidance about deductible accumulator fields from RTPB
Standard Implementation Guide:**

The Amount Applied to Periodic Deductible component of the Estimated
Patient Financial Responsibility amount is based on plan benefit
accumulators, such as the accumulated deductible and remaining
deductible. The accumulated deductible and remaining deductible amounts
can be communicated in the RTPBResponse RTPB Product
Segment/ResponseProduct and Response RTPB Alternative Product
Segment/ResponseAlternativeProduct segments. These are estimated
amounts calculated by the pharmacy benefit manager based on the
patient's plan benefit structure and current accumulated deductible
amount. A RTPB request does not change the deductible amounts on file
in the payer system. If multiple RTPB requests are submitted
concurrently for a patient to a single prescription benefit plan, each
is processed individually using the benefit accumulators and other
financial information on file with the processor at the time of the
request — that is, the accumulator values returned on the RTPB response
will only reflect the financial changes for the submitted drug and each
of the alternatives versus the cumulative financial changes that occur
with multiple claim billing transactions.

**Resolution Description (recorded on ticket):**

Add `accumulated-deductible` and `remaining-deductible` codes to the
RTPBC Patient Pay Type code system used in the RTPBC response (FHIR
ClaimResponse profile).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already recorded on the ticket: extend the RTPBC
**Patient Pay Type** code system used by the Consumer RTPBC response
(the RTPBC ClaimResponse profile) with two new concepts:

- `accumulated-deductible` — "Amount in dollars met by the
  patient/family in a deductible plan" (mirrors NCPDP 512-FC,
  Accumulated Deductible Amount).
- `remaining-deductible` — "Amount not met by the patient/family in a
  deductible plan" (mirrors NCPDP 513-FD, Remaining Deductible Amount).

Use these codes in `ClaimResponse.item.adjudication.category` (or the
equivalent slice on the RTPBC ClaimResponse profile) so that one
adjudication entry per accumulator can be returned for both the
submitted product and each alternative product, with `amount` carrying
the dollar value. Update the RTPBC IG response page to document the new
categories, link them to NCPDP 512-FC / 513-FD, and reproduce the
NCPDP-supplied usage rule ("required when the plan provides details of
how the product could impact the patient's deductible status") as the
RTPBC conformance expectation. Mirror the same wording used in the
NCPDP RTPB Standard v11 examples so cross-walk to the X12-/NCPDP-style
field semantics is unambiguous.

#### Justification

This is the disposition the resolution text already commits to and is
the smallest-surface change that closes the gap with NCPDP RTPB
Standard v11. Adding two coded categories on
`ClaimResponse.item.adjudication` preserves the established RTPBC
pattern of expressing per-product financial breakdowns through the
adjudication slice, avoids any new resource shape, and remains backward
compatible — existing RTPBC responses without accumulator data continue
to validate. The NCPDP source material in the ticket gives unambiguous
definitions, format guidance, and a worked example, so the IG text and
the code system descriptions can be lifted directly without
interpretation risk. Because the underlying NCPDP rule is *situational*
("required when the plan provides details…"), modeling the values as
optional adjudication entries (rather than required ClaimResponse
elements) is the natural fit.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting two new local Patient Pay Type codes, model the
deductible accumulators using the existing FHIR
`adjudication.category` value set extended via the standard CARIN /
ClaimResponse adjudication category code system (or the equivalent
HL7-level value set used elsewhere in CARIN BB / Da Vinci PCT), and
attach an extension on each adjudication entry that explicitly carries
the NCPDP field reference (`512-FC`, `513-FD`) to make the cross-walk
machine-readable. Where a code already exists in
`http://terminology.hl7.org/CodeSystem/adjudication` (or a CARIN-aligned
code system) that means "deductible met to date" / "deductible
remaining", reuse that code; otherwise propose the new concepts upstream
in CARIN/THO so the RTPBC IG inherits them rather than carrying its own
RTPBC-specific concepts.

#### Justification

The Pharmacy work group has historically tried to keep RTPBC's coded
vocabulary aligned with the broader CARIN adjudication categories so
implementers building both BB and RTPBC don't have to maintain two
parallel terminologies for the same concept (deductible status).
Choosing reused/upstream codes reduces long-term maintenance and
improves interoperability with payer-side systems that already emit
deductible totals through CARIN BB. The NCPDP-field extension makes the
NCPDP origin explicit for implementers who must bridge to the NCPDP RTPB
v11 message. The trade-off is more upfront coordination (with THO and
potentially CARIN BB) and a longer path to publication than Disposition
A's local-codes change.

---

### Disposition C: Decline

#### Proposal

Decline to add deductible-accumulator concepts to the RTPBC response on
the grounds that the existing RTPBC Patient Pay Type values already
allow implementers to convey the patient's expected out-of-pocket
amount, and that conveying *plan-side accumulator state* (current
accumulated deductible, remaining deductible) is a payer-eligibility
function better served by CARIN Blue Button or a coverage/eligibility
operation rather than by RTPBC, which is scoped to per-product price
estimates.

#### Justification

This disposition is defensible only if the work group judges that
deductible accumulator state is out of scope for a per-prescription
price-check transaction and that adding it would expand RTPBC into
territory already covered by CARIN BB / coverage eligibility. It also
avoids the (small) risk of RTPBC and CARIN BB drifting on how
"accumulated deductible" is named.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved Persuasive with a concrete,
narrowly-scoped change: add two codes to the RTPBC Patient Pay Type
code system. The submitter has supplied the upstream NCPDP definitions,
the NCPDP field numbers (512-FC / 513-FD), the situational-use rule,
and worked examples — everything required to write the code-system
entries and the IG narrative without further interpretation. The
NCPDP RTPB v11 standard has been published since January 2021, so
RTPBC responses that don't carry these accumulators are demonstrably
behind the underlying NCPDP transaction they are intended to mirror;
that is exactly the gap this ticket asks to close. Disposition B's
upstream-alignment concerns are real, but they are better handled as a
follow-up harmonization with CARIN BB / THO than as a precondition for
publishing the NCPDP-aligned codes RTPBC needs now; nothing in
Disposition A blocks Disposition B's harmonization later. Disposition C
under-serves the original NCPDP use case — a provider or pharmacist
explaining at the point of inquiry how filling this drug will move the
patient's deductible — which is an RTPBC scenario, not a CARIN BB
scenario.
