## Summary

The ticket reports that a sample API call shown in the US Prescription Drug
Monitoring Program (PDMP) FHIR Implementation Guide is described as the
"exact" API to use, when in fact it is only an example — most obviously
because it still contains the unresolved `[base]` placeholder. The reporter
asks that the IG be re-worded to clearly present the snippet as an example,
improve the surrounding readability, and ideally provide a side-by-side
breakdown (such as a two-column table of "call parts" and "explanation")
that maps each segment of the URL/operation to its meaning. The ticket also
proposes specific replacement wording that re-frames "the exact API" as "an
example API call" and tightens the sentence describing the bundle returned
(MedicationDispense plus the associated MedicationRequest, Practitioner,
Organization, Patient, and Medication resources). No related Jira tickets,
Zulip discussions, or GitHub items were surfaced via cross-references for
this issue, so the request is being evaluated on its own merits within the
PDMP IG narrative.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the reporter's proposed editorial changes verbatim to the PDMP IG
narrative page that hosts the multi-search-parameter example. Specifically:

1. Replace "The following is the exact API that uses multiple search
   parameters …" with "The following is **an example** API **call** that
   uses multiple search parameters …".
2. Replace "The above API will fetch all MedicationDispense …. The bundle
   returned will include MedicationDispense, MedicationRequest,
   Practitioner, Organization, Patient and Medication." with "The above
   API **call** will fetch all MedicationDispense …. The bundle returned
   will include MedicationDispense **resources, and associated**
   MedicationRequest, Practitioner, Organization, Patient and Medication
   **resources**."
3. Add a two-column "call parts / explanation" table immediately after the
   sample URL that breaks the example into its constituent segments
   (`[base]`, resource type, search parameter names, parameter values,
   `_include`/`_revinclude` if present) with a one-line explanation for
   each. Where `[base]` appears, footnote it as a placeholder for the
   PDMP Responder's service base URL.

#### Justification

The change is purely editorial / clarifying: it does not alter conformance,
search behaviour, or bundle composition. Calling the snippet "an example"
correctly characterises it (the literal string contains `[base]`, which is
not a resolvable URL and is by HL7 convention a placeholder), and the
reporter's wording also tightens the sentence about the returned bundle by
clarifying that the additional resource types are returned because they
are *associated* with MedicationDispense — matching how `_include`-style
search semantics actually work. The proposed call-parts table is a common
pattern in other FHIR IGs (e.g., narrative pages that walk through search
URLs) and meaningfully improves implementer comprehension at very low
authoring cost. The original "Considered for Future Use" disposition was
explicitly rolled back because the IG is being picked up again
(PSS-2225), so the historical reason for deferral no longer applies.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substantive intent of the ticket but generalise the fix beyond
the single sentence:

1. Sweep the PDMP IG narrative for every occurrence of "exact", "the API",
   or similar wording that could be read as normative URL syntax, and
   replace each with "example" framing — not just the one snippet called
   out in this ticket.
2. Standardise on the FHIR-style `[base]` convention with a single
   IG-wide note (e.g., in an "Examples" or "Conventions" section)
   defining the placeholder once, instead of footnoting it on every
   page.
3. Introduce a reusable "API call walkthrough" pattern: each example URL
   in the IG is followed by the same call-parts/explanation table format
   so readers learn the convention once. Apply it to this example first,
   then to other example URLs as they are touched.
4. Where the example demonstrates `_include`/`_revinclude` behaviour,
   also add a short note that the *included* resources (MedicationRequest,
   Practitioner, Organization, Patient, Medication) come back because of
   the include semantics, not because MedicationDispense itself contains
   them.

#### Justification

The wording problem the reporter found is almost certainly not unique to
this one paragraph — IGs that have been through several authors tend to
have the same "exact API" / missing-`[base]`-explanation pattern in
multiple places. Doing a sweep at the same time is cheap incremental
work given the page must already be opened, and it avoids a stream of
near-duplicate tickets later. Defining the `[base]` convention and the
walkthrough-table pattern centrally also makes future PDMP IG additions
easier to author consistently. The trade-off vs. Disposition A is scope
creep: it touches more pages and may slow down resolution, and the
work-group must be willing to own the broader editorial pass.

---

### Disposition C: Decline

#### Proposal

Close the ticket as *Persuasive with Modification – Not Persuasive* on
the grounds that (a) the `[base]` placeholder is a long-standing,
IG-wide FHIR convention defined elsewhere (typically in the FHIR core
spec's "Service Base URL" section) and is therefore not misleading to
the IG's expected audience of FHIR implementers, and (b) the existing
prose is unambiguous in context: an implementer cannot literally
execute a URL containing `[base]`, so they will treat it as an example
regardless of whether the page calls it "exact" or "example".

#### Justification

This is the weakest of the three options and is included only for
completeness. It would be defensible if the work group wished to defer
narrative polish on a specification that is still being re-stood up
under PSS-2225, on the grounds that there are likely larger structural
issues to settle first and minor wording tweaks can be batched into a
later editorial pass. It is *not* defensible on the grounds that the
request is wrong — the reporter is correct that "exact" is the wrong
word and that `[base]` is a placeholder.

---

### Recommendation

**Recommended disposition:** A, with the door open to expanding to B if the
editor finds the same pattern elsewhere on the page.

The change is small, low-risk, purely editorial, and the reporter has
already supplied finished replacement wording — accepting it as requested
is essentially free relative to the cost of deferring or arguing the
point. The only judgement call is whether to add the call-parts table
now (recommended) or as a follow-up; given the table is the part of the
request most likely to actually help implementers read the example, it
should land with the wording fix rather than be split off. Disposition B
is a sensible upgrade if the editor, while in the file, sees the same
"exact API" framing reused elsewhere — in that case the sweep should be
folded in rather than tracked as a new ticket. Disposition C is not
recommended: the prior "deferred because IG withdrawn" rationale was
explicitly reverted by the reporter when PSS-2225 brought the IG back
into scope, and there is no remaining substantive objection to the
change.
