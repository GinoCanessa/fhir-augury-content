## Summary

While Pharmacy was applying a tranche of related tickets that move several
"category" CodeSystems back from THO into FHIR Core
([FHIR-31632](https://jira.hl7.org/browse/FHIR-31632) — example value sets/code
systems back to the FHIR build;
[FHIR-51726](https://jira.hl7.org/browse/FHIR-51726) — move
`MedicationRequestAdministrationLocationCodes` back to core, renaming it to
`medicationrequest-category`; and
[FHIR-50142](https://jira.hl7.org/browse/FHIR-50142) — recreate
`http://terminology.hl7.org/CodeSystem/medicationrequest-admin-location` in
core), reviewers (notably Lloyd McKenzie, on the
[committers > PR Reviews Zulip thread](https://chat.fhir.org/#narrow/stream/committers/topic/PR%20Reviews/near/553195702))
flagged a methodology rule: a CodeSystem that lives in FHIR Core must either
(a) be marked `experimental = true` (and therefore only used as `example`
binding), or (b) be a `code`-typed binding; otherwise it belongs in THO.

The reporter raises three concrete observations on the medication category
CodeSystems being added by the PR(s):

1. The codes should either be marked `experimental` or moved into THO.
2. The four medication-resource category code systems are effectively the
   same set of values applied to elements with identical purpose, and could
   plausibly be combined into a single category CodeSystem.
3. The `MedicationAdministration.category` definition still says
   *"The type of medication administration (for example, drug
   classification like ATC, where meds would be administered, legal category
   of the medication)."* The "drug classification" wording no longer belongs
   in this element.

The proposed fix in the ticket is to mark the affected category CodeSystems
`experimental` and add a comment that the committee will revisit them — most
likely consolidating into a single CodeSystem and moving it to
[terminology.hl7.org](http://terminology.hl7.org). The Pharmacy work group
already met on this and recorded the resolution as **Persuasive with
Modification**: add a comment to the spec, but do **not** flip the codes to
experimental, and open a follow-up Jira to (1) remove the per-resource
CodeSystems from core, (2) merge them into a single CodeSystem in THO with
adjusted designations, and (3) bind to the THO CodeSystem.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FHIR Core spec, mark the medication category CodeSystems introduced /
restored by FHIR-51726 / FHIR-50142 — i.e.
`http://hl7.org/fhir/CodeSystem/medicationrequest-category`,
`medicationdispense-category`, `medicationstatement-category`,
and `medication-admin-category` — with `experimental = true`, and add an
explanatory note on each saying the Pharmacy WG will revisit these codes
and likely consolidate them into a single CodeSystem published from THO.
Also tighten the `MedicationAdministration.category` element definition to
remove the obsolete "drug classification (e.g. ATC)" example wording. The
underlying ValueSet bindings on the four `*.category` elements stay
`example`.

#### Justification

This is the literal request, and it satisfies Lloyd McKenzie's "code
systems with experimental=true (or `code`-typed bindings) may live in core;
everything else needs to be in THO" rule without removing or renaming
anything in the build window. It cleanly unblocks
[HL7/fhir#3896](https://github.com/HL7/fhir/pull/3896) /
[HL7/fhir#3911](https://github.com/HL7/fhir/pull/3911) and aligns with
[FHIR-31632](https://jira.hl7.org/browse/FHIR-31632)'s methodology of
returning *example* terminology to the build.

---

### Disposition B: Alternative Approach (the WG's recorded resolution)

#### Proposal

Do **not** change the new medication category CodeSystems to
`experimental`. Instead:

1. Leave the in-core CodeSystems and `example` bindings produced by
   FHIR-51726 / FHIR-50142 in place for this cycle.
2. Add a non-normative editorial note on each of the four `*.category`
   elements (and on each CodeSystem) stating that Pharmacy intends to
   consolidate these into a single CodeSystem and republish it from
   [terminology.hl7.org](http://terminology.hl7.org) in a future release.
3. Tighten the `MedicationAdministration.category` element definition to
   drop the "drug classification (e.g. ATC, legal category)" example
   wording, since drug classification no longer belongs in this element.
4. Open a follow-up Jira (per the recorded resolution) covering: (a)
   removing the per-resource CodeSystems from core; (b) merging them into
   a single THO CodeSystem with reconciled designations (e.g.
   "Includes orders in…" vs. "Includes administrations in…"); (c)
   re-binding all four medication `*.category` elements to that THO
   ValueSet.

#### Justification

This matches what is already recorded as the ticket's resolution
(*Persuasive with Modification*) and what Lloyd McKenzie and Bryn Rhodes
both argued for on Zulip: Lloyd noted these are real, non-example values
("it would be odd to define those codes and say 'but you're not allowed to
use them'"), and Bryn pointed out US Core / CMS / HEDIS already bind to
the existing THO `medicationrequest-category` value set in a non-example
way. Flipping to `experimental` would mis-signal usability and break
alignment with downstream IGs already in production. Splitting the THO
consolidation into a separate Jira keeps this PR mergeable now while
preserving the methodology fix for the next cycle.

---

### Disposition C: Decline

#### Proposal

Decline the methodology change in the form proposed (mark experimental /
plan to consolidate). Take only the editorial sub-fix: tighten the
`MedicationAdministration.category` element definition to drop the
"drug classification" example wording, and otherwise leave the four
medication category CodeSystems and their `example` bindings exactly as
introduced by FHIR-51726 / FHIR-50142.

#### Justification

These CodeSystems were just deliberately moved into core under FHIR-31632
and FHIR-51726, and equivalent codes are already in active production use
via THO bindings (US Core, CMS, HEDIS — per Bryn Rhodes on Zulip). Marking
them `experimental` immediately after restoring them sends a contradictory
signal to implementers, and consolidating into a single THO CodeSystem is a
non-trivial cross-resource change that has not yet been costed by the WG.
Defensible if the WG concludes the policy concern Lloyd raised is real but
should be addressed only as part of a future, scoped THO migration ticket
rather than amended onto the in-flight PR.

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach — the WG's recorded
resolution).**

Disposition B is what Pharmacy already agreed on the ticket and what the
two strongest voices on the Zulip review thread (Lloyd McKenzie, Bryn
Rhodes) explicitly endorsed. It threads the methodology rule (CodeSystems
in core must be experimental or `code`-typed) by acknowledging it in a spec
comment and committing to migrate, without breaking real-world bindings in
US Core / CMS / HEDIS that already use these codes non-experimentally.
Disposition A is technically defensible but mis-signals usability and was
specifically rejected on Zulip ("those look like 'real' values to me");
Disposition C drops the consolidation commitment entirely and leaves the
methodology issue unowned. The follow-up Jira called out in the recorded
resolution is the right place to land the THO consolidation work in a
future cycle.
