## Summary

Lloyd McKenzie (as part of a broader sweep to identify FHIR-defined
terminologies that are candidates for migration to Terminology.HL7.org
(THO)) flags two CodeSystem/ValueSet pairs in the MedicationAdministration
resource as potentially re-usable beyond FHIR:

- `CodeSystem/medication-admin-category` + `ValueSet/medication-admin-category`
  (bound to `MedicationAdministration.category`), and
- `CodeSystem/administration-subpotent-reason` +
  `ValueSet/administration-subpotent-reason` (bound to
  `MedicationAdministration.subPotentReason`).

Migration to THO is explicitly **not required**, but the ticket asks
Pharmacy to consider it where (a) the definitions are likely to evolve
independent of FHIR release cadence, or (b) the concepts are reasonably
re-usable in CDA, V2, or other HL7 product families. The ticket is a
governance / terminology-hygiene prompt rather than a content change
request; the expected decision is "migrate", "keep in FHIR", or "migrate
selectively" with rationale.

No cross-referenced Jira tickets were returned for this key, and no
Zulip discussions or GitHub PRs/issues were found. The only
cross-references the orchestrator surfaced are the two FHIR element
mentions that inspired the ticket (`MedicationAdministration.category`
and `MedicationAdministration.subPotentReason`).

## Proposed Dispositions

Because no GitHub cross-references were surfaced, the dispositions
below are framed at the specification/terminology-governance level.
Any that are accepted will ultimately touch the FHIR core build
(authoring MedicationAdministration bindings) and THO (new
CodeSystem/ValueSet homes), but concrete repo/path targeting will need
a repo-analysis-backed follow-up.

### Disposition A: Accept As Requested

#### Proposal

Migrate both terminology artifacts out of FHIR core into THO:

1. Move `CodeSystem/medication-admin-category`
   (MedicationAdministrationCategoryCodes) to THO under the HL7
   Terminology namespace (e.g.,
   `http://terminology.hl7.org/CodeSystem/medication-admin-category`),
   retiring the `http://hl7.org/fhir/CodeSystem/medication-admin-category`
   definition and replacing it with a "moved to THO" stub per HL7
   terminology migration policy.
2. Do the same for
   `CodeSystem/administration-subpotent-reason`
   (AdministrationSubPotentReason), moving it to
   `http://terminology.hl7.org/CodeSystem/administration-subpotent-reason`.
3. Re-point the two FHIR ValueSets (`medication-admin-category`,
   `administration-subpotent-reason`) to compose from the new THO
   CodeSystem URIs. Preserve the ValueSet canonical URLs so existing
   bindings on `MedicationAdministration.category` and
   `MedicationAdministration.subPotentReason` keep working.
4. Coordinate with Vocabulary WG on URI redirection / obsolete-codesystem
   handling so R4/R4B/R5 implementers see a clean deprecation story.

#### Justification

Both code systems are small, HL7-authored, and conceptually
cross-paradigm: "category of administration" (inpatient / outpatient /
community) and "reason a dose was sub-potent" are not FHIR-shaped
concepts — they describe real-world medication-administration
workflow and are plausibly useful to CDA (CCD medication-administered
templates), V2 (RXA segments), and emerging specs. Hosting them in
THO decouples their maintenance from the FHIR release cycle, which
is the stated test in the ticket. Preserving ValueSet canonical URLs
keeps the change invisible to conformance resources and IGs that
bind to them.

---

### Disposition B: Alternative Approach

#### Proposal

Migrate **only** `administration-subpotent-reason`; keep
`medication-admin-category` in FHIR core.

- `administration-subpotent-reason` moves to THO (as in Disposition
  A, step 2); its ValueSet is repointed.
- `medication-admin-category` stays under
  `http://hl7.org/fhir/CodeSystem/medication-admin-category`, but is
  flagged in its narrative as "under review for future THO migration"
  so Pharmacy can revisit once usage outside FHIR is demonstrated.

#### Justification

The two code systems have very different cross-product reuse profiles.
`administration-subpotent-reason` is a workflow/clinical-reason
vocabulary (partial dose, expired product, parent refusal, etc.) that
is clearly reusable in CDA/V2 medication administration flows and is
more likely to need vocabulary changes (new sub-potency reasons) on
a cadence independent of FHIR. `medication-admin-category`, by
contrast, is a tight three-ish-value operational classification
(inpatient/outpatient/community) that has essentially not changed and
has no demonstrated non-FHIR consumer. Migrating everything
indiscriminately adds churn (stub CodeSystems, redirection entries,
publisher tooling changes) without clear benefit for the second
artifact. This disposition accepts the ticket's intent while applying
its own stated test ("realistic to change" / "useful in other specs")
asymmetrically.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — existing location is appropriate."
Rationale: both CodeSystems are FHIR-native, have stable definitions,
have no demonstrated external consumers, and the ticket itself
acknowledges migration is not required. Pharmacy elects to keep them
in FHIR core and revisit if/when a concrete cross-product need
materializes or a definition change is requested that FHIR's release
cadence cannot accommodate.

#### Justification

The ticket is a terminology-hygiene prompt, not a defect or a
use-case-driven request. Migration has non-trivial costs (stub
CodeSystem management, ValueSet compose repointing, publisher config
changes in both the FHIR core build and THO, downstream IG
republication, implementer confusion about canonical URIs) and those
costs are only justified when there is an actual cross-paradigm
consumer or an expected definition change the FHIR cadence cannot
support. Neither has been demonstrated for either code system.
Declining defers the work without foreclosing it — if a future
request requires an update to either vocabulary between FHIR
releases, the migration can be reopened then.

---

### Recommendation

**Recommended disposition:** B

Disposition B best honors the ticket's own stated test. The
`administration-subpotent-reason` vocabulary describes
cross-paradigm clinical reasons (relevant to CDA medication
administration and V2 RXA flows) and is the kind of list that
genuinely grows over time as new sub-potency reasons are recognized —
so THO's independent release cadence is a real benefit.
`medication-admin-category`, by contrast, is a tiny stable
operational classification with no demonstrated non-FHIR consumer,
and migrating it would pay migration costs for no observed benefit.
Disposition A is defensible but over-applies the policy; Disposition
C under-applies it for the sub-potent-reason vocabulary, which really
is a THO-shaped artifact. Pharmacy should coordinate the
sub-potent-reason migration with Vocabulary WG to sequence it with
any other Pharmacy THO-candidate migrations (sibling tickets are
likely pending for other medication resources) so the THO and FHIR
publisher changes land together rather than piecemeal.
