## Summary

Arianne van de Wetering (Pharmacy WG) is asking that the new
`DosageDetails` datatype introduced in the FHIR R6 ballot be renamed to
`InstructionsForUse`. Her rationale is that the scope and contents of
`DosageDetails` are wider than dosage proper — the structure now bundles
ordering steps, safety, conditions, additional instructions, route,
timing, and other patient-facing instructions — so the current name
under-represents what the type actually is. The change is filed as a
"Highest" priority Change Request against FHIR Core and remains
Triaged.

The request sits inside a much larger cleanup effort by the Pharmacy WG
around the new dosing model. Closely related tickets in the same family
include [FHIR-54336](https://jira.hl7.org/browse/FHIR-54336)
("move Dosage.route to DosageDetails.route"),
[FHIR-54338](https://jira.hl7.org/browse/FHIR-54338)
("move Dosage.additionalInstruction to DosageDetails.additionalInstruction"),
[FHIR-54330](https://jira.hl7.org/browse/FHIR-54330)
(redundancy between `DosageDetails.step.count` and
`step.component.timing.repeat.count`),
[FHIR-54557](https://jira.hl7.org/browse/FHIR-54557)
(clarification of `DosageSafety` placement throughout `DosageDetails`),
and [FHIR-53721](https://jira.hl7.org/browse/FHIR-53721)
(`ActivityDefinition.dosage` cardinality after the redesign). Together
these confirm that `DosageDetails` is a brand-new R6 type still being
shaped, so a rename is technically cheap but substantively affects
every other open ticket against the type.

Costa Teixeira (Pharmacy co-chair) pushed back on the comment thread,
agreeing the name *can* change while the structure is new but warning
that "InstructionsForUse doesn't seem a good name and we should not
iterate much on this." Mattias Colliander voiced the opposite view in
the Medication Zulip stream — "the DosageDetails name has bugged me as
well, happy to have a Jira for it" — so there is community appetite for
a rename, just not consensus on the target name.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rename the `DosageDetails` datatype to `InstructionsForUse` across the
FHIR Core specification before R6 publishes. Concretely this requires:

- Renaming the `StructureDefinition` for the datatype (id, name, url,
  type) from `DosageDetails` to `InstructionsForUse`.
- Updating every resource that currently references `DosageDetails` to
  reference `InstructionsForUse` (at minimum: `MedicationRequest`,
  `MedicationDispense`, `MedicationStatement`,
  `MedicationAdministration`, `ActivityDefinition` per FHIR-53721, plus
  any profile/extension that targets it).
- Renaming the datatype's spec page, examples
  (`dosage-examples.html` content), search parameters, and the FHIRPath
  references that use the type name.
- Updating sibling tickets in flight (FHIR-54336, FHIR-54338,
  FHIR-54330, FHIR-54557) to land against the new name in the same
  ballot cycle so the implementer-visible name only changes once.

#### Justification

`DosageDetails` was introduced fresh in R6 ballot 4, so the rename
window is open without breaking any normative content. The reporter's
core argument — that the type is an envelope for patient-facing
instructions (steps, components, safety, conditions, additional
instructions, route) and not just "dosage details" — is supported by
the related tickets (FHIR-54336/54338 moving route and
additionalInstruction up; FHIR-54557 documenting safety living at
multiple levels). Mattias Colliander's Zulip comment shows there is
implementer demand for a clearer name. Doing the rename now, in
coordination with the other open `DosageDetails` tickets, is cheaper
than doing it post-publication.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying concern (the name `DosageDetails` does not
reflect the type's scope) but choose a different target name than
`InstructionsForUse`. Specifically, propose a Pharmacy WG short-list of
candidate names — for example `DosageInstructions`,
`MedicationInstructions`, `DosingPlan`, or `DoseInstructions` — and
pick one through a single WG vote to avoid iterative renames. Apply
the chosen name with the same coordinated rollout described in
Disposition A (StructureDefinition, references, examples, in-flight
tickets).

#### Justification

This addresses Costa Teixeira's two stated concerns: (1) he agrees the
name *can* change while the type is new, and (2) `InstructionsForUse`
specifically "doesn't seem a good name and we should not iterate much
on this." A single WG-driven vote on a small candidate set guards
against churning implementers with successive renames. It also
acknowledges that "InstructionsForUse" carries regulatory baggage in
some jurisdictions (it is the title of the printed product
leaflet/IFU), which could be confused with the FHIR datatype that
describes administration to a specific patient. A name like
`DosageInstructions` keeps continuity with the legacy `Dosage` family
while accurately describing the broader scope.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** and keep the name `DosageDetails`. Note
in the resolution that the type's scope is documented in its narrative
description and that further restructuring tickets (FHIR-54336,
FHIR-54338, FHIR-54557) will refine what the type actually contains
before a rename is justified.

#### Justification

`DosageDetails` is brand new in R6 ballot 4 and a rename — no matter
how cheap technically — forces every implementer who has begun
prototyping against the ballot to update tooling, validators,
documentation, and example libraries. Costa Teixeira (co-chair) has
already signalled that he does not consider the proposed target name
an improvement, and the Zulip thread did not produce consensus on an
alternative. If the WG accepts the rename premise but cannot agree on
a better name, the safest course is to keep the existing name and
revisit later if a clearly better candidate emerges.

---

### Recommendation

**Recommended disposition:** B

The reporter's complaint is legitimate and is independently echoed in
Zulip ("the DosageDetails name has bugged me as well"), and the related
tickets confirm the type is steadily becoming an envelope for far more
than dosing numbers. That argues against Disposition C. However, Costa
Teixeira's pushback against `InstructionsForUse` is well-founded — the
term collides with regulatory IFU language and is awkward as a FHIR
datatype name — so accepting the rename literally (Disposition A) risks
locking in a name the WG itself does not love and then re-renaming
later. Disposition B threads the needle: take the rename concern
seriously, but make the WG choose a better target through a single,
bounded decision and land the change in the same ballot cycle as the
in-flight `DosageDetails` restructuring tickets so implementers absorb
exactly one name change. Co-chairs should drive that short-list at the
upcoming Medication call already scheduled in the Zulip backlog.
