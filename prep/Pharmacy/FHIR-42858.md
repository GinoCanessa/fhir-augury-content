## Summary

This ticket, raised out of HL7 Connectathon 34 against the US Prescription
Drug Monitoring Program (PDMP) FHIR IG, asks the Pharmacy work group to
define how a PDMP query response should communicate exception cases —
specifically failures, "no data found," and similar non-success
conditions — that today are not consistently represented in the IG's
response patterns. The reporter (Scott M. Robertson) flagged that the
existing PDMP query/response flow lacks a uniform, machine-readable way
for a responder to signal these conditions, and explicitly suggested
`OperationOutcome` as a likely vehicle. The ticket was triaged with a
"Persuasive" resolution noting that an `OperationOutcome` for exception
cases was incorporated into the May 2024 PDMP ballot. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced for this
ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt `OperationOutcome` as the canonical mechanism for communicating
exception cases on PDMP query responses, mirroring the change already
landed in the May 2024 PDMP ballot referenced in the resolution.
Concretely, in the PDMP IG:

- Define a profile (e.g., `PDMPOperationOutcome`) that constrains
  `OperationOutcome.issue.severity`, `issue.code`, and
  `issue.details.coding` for the PDMP-relevant cases: `not-found`
  (no records for the patient/timeframe), `processing` /
  `transient` (responder error), `security` (auth/consent failures),
  and `business-rule` (e.g., jurisdictional restriction).
- Update the PDMP query Bundle / response narrative so that a
  successful "no data" response returns a Bundle whose `entry` includes
  an `OperationOutcome` with `severity = information` and
  `code = not-found`, while error responses return the
  `OperationOutcome` either as a top-level resource (per FHIR REST
  rules) or as a Bundle entry, depending on the binding.
- Add a value set / code system entry for PDMP-specific issue detail
  codes covering the agreed exception cases, and bind
  `OperationOutcome.issue.details` to it (extensible).
- Add IG-level guidance pages enumerating each exception case, the
  expected `OperationOutcome` shape, and example instances.

#### Justification

This is exactly what the reporter proposed ("possibly OperationOutcome")
and what the work group has already adopted in the May 2024 ballot
("included OperationOutcome for exception cases is 2024May ballot").
`OperationOutcome` is the FHIR-canonical mechanism for conveying
operation status, errors, warnings, and informational messages, and
using it keeps PDMP consistent with the wider FHIR ecosystem (US Core,
Da Vinci, etc., all use `OperationOutcome` for the same purpose).
Profiling `OperationOutcome` and binding a PDMP-specific code system
gives implementers a deterministic, machine-readable contract for
exception handling without inventing a parallel mechanism.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) profiling `OperationOutcome`, model
PDMP exception cases primarily through HTTP semantics and Bundle
metadata:

- Specify the exact HTTP status codes responders MUST use for each
  exception class (e.g., `200` with empty searchset for "no records",
  `403` for consent/jurisdictional denial, `422` for validation
  failure, `503` for transient backend issues).
- For "no data" specifically, require an empty `searchset` Bundle with
  `total = 0`, optionally including an `OperationOutcome` entry only
  when the responder needs to convey *why* (e.g., "patient not known
  to this PDMP" vs. "patient known, no dispensations in window").
- Reserve `OperationOutcome` strictly for error/warning conditions,
  not for normal "no data" responses, to avoid implementers conflating
  informational outcomes with errors.

#### Justification

This approach keeps the happy-path response shape stable (a Bundle is
always a Bundle) and pushes the bulk of exception signalling to the
transport layer that every FHIR client already inspects. It reduces
the risk that implementers treat an `OperationOutcome` with
`severity = information` as an error, which is a known interoperability
pitfall. The trade-off vs. Disposition A is less expressive failure
detail in some scenarios — but PDMP responders and consumers are a
relatively constrained, regulated community where coarse-grained HTTP
semantics plus a small set of well-defined error codes may be enough.
Note that this does not contradict the May 2024 ballot content; it
narrows where `OperationOutcome` is required.

---

### Disposition C: Decline

#### Proposal

Close the ticket as already addressed by the May 2024 ballot content
(the resolution notes `OperationOutcome` for exception cases is
included) and, beyond that, decline to add further normative exception-
handling machinery to the PDMP IG. Defer any additional structure
(profiles, value sets, status-code tables) to implementer feedback
gathered during ballot reconciliation and subsequent connectathons.

#### Justification

The ticket itself is explicitly framed as a "FUTURE" discussion item
from Connectathon 34, and the work group has already taken the minimum
persuasive action by adding `OperationOutcome` to the May 2024 ballot.
Without concrete implementer pain points (no Zulip discussion, no
linked GitHub issues, no follow-up Jira tickets surfaced), expanding
the scope risks over-specifying behaviour that real implementers may
not need or may want to shape differently after the ballot. Declining
further work here is defensible until ballot comments or connectathon
results justify a richer design.

---

### Recommendation

**Recommended disposition:** A

The reporter's suggestion, the work group's resolution, and the May
2024 ballot are already aligned on using `OperationOutcome` for PDMP
exception cases. The remaining work is to make that adoption usable
and testable: a profile, a small value set, and clear narrative
guidance for each exception class. Disposition A captures that work
explicitly and turns the "Persuasive" resolution into something
implementers can build against. Disposition B raises a legitimate
concern about overloading `OperationOutcome` for "no data," and that
nuance should be folded into Disposition A's narrative (i.e., be
explicit about when `OperationOutcome` is required vs. optional in
the response Bundle), but it is not a reason to walk back the
fundamental choice. Disposition C is too thin given that the ballot
has already moved in this direction; the work group should finish the
job rather than leave the IG with `OperationOutcome` mentioned but
under-specified.
