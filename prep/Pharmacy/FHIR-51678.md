## Summary

The ticket is a technical correction against the US CARIN Real-time Pharmacy
Benefit Check (RTPBC) FHIR IG. The reporter, Frank McKinney, identifies that
the four published example Bundles that demonstrate RTPBC request/response
exchanges contain incorrect internal resource references (i.e., `Bundle.entry`
instances whose `fullUrl` / `reference` linkage between the operation
parameters, MedicationRequest, Patient, Coverage, etc. do not resolve cleanly
within each Bundle). The four affected examples are the bundled RTPBC
request to an insurer, the bundled RTPBC request to a medication pricing
source, the bundled RTPBC response from an insurer, and the bundled RTPBC
response from a medication pricing source. The ticket asks that the
internal references in all four Bundle examples be corrected so the examples
are self-consistent and validate. The Pharmacy WG has already triaged the
ticket as **Persuasive**, indicating intent to apply the corrections in the
next publication of the IG. There are no cross-referenced Jira tickets and
no related Zulip discussions surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply targeted corrections to the four RTPBC example Bundles:

1. Bundled RTPBC request to an insurer
2. Bundled RTPBC request to a medication pricing source
3. Bundled RTPBC response from an insurer
4. Bundled RTPBC response from a medication pricing source

For each Bundle:

- Ensure every `Bundle.entry.fullUrl` is a syntactically valid URI (UUID
  `urn:uuid:...` or absolute URL) and is unique within the Bundle.
- Ensure every internal `Reference.reference` value (on the operation
  Parameters, MedicationRequest, Patient, Coverage, Organization,
  Practitioner, etc.) resolves to a sibling `Bundle.entry.fullUrl` in the
  same Bundle, using the form documented in the FHIR Bundle spec
  (`urn:uuid:...` matches `urn:uuid:...`, relative `ResourceType/id`
  matches an entry whose `fullUrl` is the matching absolute URL).
- Re-run the IG publisher / FHIR validator across the four Bundles and
  confirm zero unresolved-reference / dangling-reference warnings.
- Leave business content (codes, prices, member identifiers, scenarios)
  unchanged — this is a referencing-only fix.

#### Justification

The request is exactly what the WG already endorsed (Resolution =
Persuasive, label = `typos-and-fixes`). Internal-reference errors in
example Bundles directly mislead implementers, who frequently copy the
published examples as the starting point for their conformant payloads.
Fixing the four named Bundles in place is the smallest, lowest-risk
change consistent with the IG's current structure: it touches example
fixtures only, does not alter conformance resources (StructureDefinitions,
ValueSets, OperationDefinitions), and therefore does not require a
substantive re-ballot. It also aligns with FHIR's published guidance that
intra-Bundle references SHALL resolve.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the corrections as in Disposition A, **and additionally** add an
automated example-validation gate to the IG build so this class of defect
cannot recur silently:

- Add (or enable, if already present) an IG-publisher / Sushi
  configuration that fails the build when a Bundle example contains an
  unresolved internal reference.
- Where practical, refactor the four examples to share a small set of
  named "actor" resources (Patient, Coverage, Organization,
  PrescribingPractitioner, PharmacyOrganization) so the same reference
  pattern is reused across the request and response Bundles, reducing
  the surface area for future drift.
- Document, in the IG's "Examples" narrative page, the convention used
  for `fullUrl` (e.g., `urn:uuid:` style) and how implementers should
  read the cross-resource references.

#### Justification

The WG has just spent reviewer cycles catching what is fundamentally a
mechanical defect that should have been caught by the build. Pairing the
content fix with a process fix keeps the same problem from re-surfacing
in subsequent publications and gives implementers a clearer mental model
for how the example Bundles are wired together. The added cost is small
(configuration + a short narrative section) and is contained within the
RTPBC IG; it does not affect downstream profiles or other Pharmacy IGs.

---

### Disposition C: Decline

#### Proposal

Decline to make the corrections in this publication cycle and instead
attach an errata note to the affected example pages indicating that the
internal references in the four named Bundles are illustrative only and
should not be treated as canonically resolvable.

#### Justification

This would only be defensible if the examples were considered
non-normative narrative and the WG had no near-term publication
opportunity. Given the ticket has already been triaged Persuasive and
the change is entirely within example fixtures (no conformance impact),
this option is included for completeness but is the weakest of the three.

---

### Recommendation

**Recommended disposition:** B

The literal request (Disposition A) is correct and must be done — the
examples are wrong and the WG has already resolved the ticket as
Persuasive. The reason to prefer Disposition B is that the underlying
failure mode is mechanical: dangling intra-Bundle references survived
publication because nothing in the build asserted that they resolve.
Pairing the content fix with a small build-time validation gate (and an
optional refactor to shared actor resources) costs little, scopes
cleanly to the RTPBC IG, and prevents the WG from reopening the same
class of ticket against the next publication. Disposition C is not
recommended because the examples are the primary on-ramp implementers
use when wiring up RTPBC exchanges, and leaving them broken with only an
errata note would actively mislead implementers.
