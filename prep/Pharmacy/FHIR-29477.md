## Summary

The ticket reports a wording defect in the US Prescription Drug Monitoring
Program (PDMP) FHIR Implementation Guide (part of the broader US Medication
FHIR IG family, owned by Pharmacy). In the section listing the information a
PDMP Responder must collect about each PDMP Requestor, the first bullet
currently reads "Requestor Identity namely name". The reporter argues the
phrase "namely name" is a copy/editorial artifact (likely an unintended echo
of "Requestor Identity") and proposes the bullet be reduced to
"Requestor Identity name".

A 2023 reviewer comment from `smrobertson` notes that the literal "namely
name" wording has since been corrected, but raises a follow-on concern that
the same bullet list (specifically the 4th bullet — "Requestor Facility
information") may inappropriately describe internal actions/functions of the
PDMP Responder rather than interoperability requirements. The ticket itself
was previously dispositioned **Considered for Future Use** in 2023-01 (vote
4-0-0, John Hatem / Tim McNeil) on the basis that the IG had been withdrawn,
then **reopened** on 2023-11-28 by `smrobertson` because the IG was being
picked up again under PSS-2225 ("PDMP on FHIR"). It is currently `Triaged`
with `Highest` priority and `Unresolved`.

There are no linked Jira tickets, no GitHub artifacts, and no substantive
related Zulip discussions (the only cross-reference is an incidental "Done"
link from an unrelated IG-creation thread).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the requested narrative correction to the PDMP Responder
Requirements section of the US PDMP FHIR Implementation Guide. Change
the first bullet from:

> Requestor Identity namely name

to:

> Requestor Identity name

Leave the remaining three bullets (Requestor Role; Requestor DEA, NPI
Number; Requestor Facility information) unchanged. Per the smrobertson
comments this edit appears to already exist in the working copy of the
IG; this disposition formalises that as the resolution of FHIR-29477 and
closes the ticket as **Persuasive — Already Applied** once the published
build reflects the change. The follow-on concern about whether the 4th
bullet (Requestor Facility information) describes internal Responder
behaviour rather than interoperable content is **out of scope** for this
ticket and should be split into a new tracker.

#### Justification

The ticket is a `Technical Correction` filed against narrative text only
— no resource, profile, value set, extension, or operation is affected.
"Requestor Identity namely name" is a clear editorial artifact ("namely"
is a redundant connector that produces the awkward phrase "namely
name") and the proposed wording "Requestor Identity name" is consistent
with the parallel bullets ("Role", "DEA, NPI Number", "Facility
information"), each of which is a noun phrase. A reopened-on-revival
"old issue" labeled `PDMP-old-issues` with a trivial textual fix is the
canonical case for an Accept disposition: it costs nothing, removes a
documented confusion, and clears backlog ahead of any substantive PDMP
ballot work under PSS-2225. Splitting the Facility-information concern
into its own ticket keeps each disposition narrow and auditable.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the textual fix from Disposition A, **and** in the same edit
restructure the bullet list so it makes the Responder requirement
unambiguously about *information that must be captured/audited about
each Requestor* rather than about Responder-internal processes. Concrete
changes:

1. First bullet: change "Requestor Identity namely name" → "Requestor
   name (identity)".
2. Third bullet: change "Requestor DEA, NPI Number" → "Requestor
   identifiers (e.g., DEA, NPI)" so the bullet is generalised and
   future-proof.
3. Fourth bullet: change "Requestor Facility information" → "Requestor
   facility (organization name and identifier(s))" so it names the
   captured data points instead of suggesting an internal Responder
   function — directly addressing the smrobertson follow-on concern.
4. Add a one-sentence lead-in clarifying that these are the *audit/
   identification data points the Responder retains*, not a description
   of Responder business logic.

This bundles FHIR-29477 with the closely-related editorial concern
raised in the comments rather than spawning a second tracker.

#### Justification

The reopening reviewer (smrobertson) explicitly questioned bullet 4 in
the same review pass, so the workgroup will end up touching this
paragraph anyway as part of the PSS-2225 revival. Doing one coherent
narrative pass is cheaper than two sequential `Technical Correction`
disposition cycles, and it eliminates the risk that bullet 4 is left in
its current ambiguous form once FHIR-29477 closes. The trade-off
against Disposition A is scope creep: this disposition expands the
ticket beyond what the reporter literally asked for, which weakens its
audit trail and may trigger a second round of review on the broader
edits.

---

### Disposition C: Decline

#### Proposal

Close FHIR-29477 as **Not Persuasive — Already Resolved / No Further
Action**. Per smrobertson's 2023-11-28 comment the literal "namely
name" wording has already been corrected in the IG source. If the
correction is in fact present in the current published build, no
further editorial action is required and the ticket is moot. The
broader bullet-4 concern is logged in the comments and can be reopened
or filed separately if Pharmacy chooses to act on it.

#### Justification

If the fix is genuinely already in the IG, accepting the ticket creates
a paper "change" with no actual delta. Declining-as-already-resolved
keeps the disposition record honest and avoids implying that the IG was
modified in response to FHIR-29477 when the edit predates this
disposition. This is defensible only if Pharmacy can verify the
published narrative and the build source already read "Requestor
Identity name" (or equivalent) before this ticket is dispositioned;
otherwise Disposition C risks closing a still-broken page.

---

### Recommendation

**Recommended disposition:** A

This is a one-word narrative typo in a withdrawn-then-revived IG, with
the reporter's proposed wording already substantively in place per the
last reviewer comment. The cleanest workgroup outcome is to accept the
ticket as requested, confirm the corrected wording is live in the
PSS-2225 working build, and close FHIR-29477 as Persuasive (or
Persuasive — Already Applied). Disposition B is tempting because the
same paragraph has a known follow-on concern, but bundling unrequested
edits into a `Technical Correction` ticket muddies the audit trail and
risks reopening a settled point; the Facility-information / internal-
behaviour question deserves its own tracker so it can be debated on its
own merits. Disposition C is only safe if the workgroup can verify the
fix is already published; without that verification, declining risks
leaving the original defect in the rendered IG. Accept as requested,
verify, and split out the bullet-4 concern as a new ticket against the
revived PDMP IG.
