## Summary

The ticket reports a gap in the US Prescription Drug Monitoring Program
(PDMP) FHIR Implementation Guide's MedicationDispense search behavior.
The Responder is required to support only a fixed set of chained patient
search parameters — `subject:Patient.name.given`,
`subject:Patient.name.family`, and `subject:Patient.birthdate` — plus
`authorizingPrescription.dispenseRequest.validityPeriod`. The reporter
(Erin Holt) asks what a Responder is expected to do when those three
demographic parameters do not resolve to a unique patient: should it
return "no match"? More importantly, because the conformance statement
restricts the Requester to only first name, last name, and date of birth,
there is no defined way for the Requester to refine the query and
disambiguate. The reporter requests that the IG address this scenario at
least in narrative/context for implementers, and (implicitly) consider
allowing additional disambiguating parameters.

The cross-reference index surfaces only FHIR element mentions
(`Patient.name.given`, `Patient.name.family`, `Patient.birthdate`) drawn
from the ticket text itself — there are no related Jira tickets, Zulip
threads, GitHub items, or Confluence pages indexed against this ticket.
Comment history shows the ticket was previously dispositioned
"Considered for Future Use" (vote Hatem/McNeil 4-0-0, 2023-01-18) when
the PDMP IG was withdrawn, then reopened on 2023-11-28 by smrobertson
when the project (PSS-2225) was picked back up. The ticket has been
quiet since.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the US PDMP FHIR IG narrative for the MedicationDispense Search
interaction to explicitly describe Responder behavior when the chained
patient demographics (`subject:Patient.name.given` +
`subject:Patient.name.family` + `subject:Patient.birthdate`) do not
resolve to a unique patient. Specifically:

1. Add an "Ambiguous / Non-Unique Patient Match" subsection under the
   MedicationDispense Search conformance text that states:
   - The Responder **SHALL** treat ambiguous matches as a search miss
     and **SHALL NOT** return MedicationDispense records for any candidate
     patient.
   - The Responder **SHALL** return an empty `Bundle` (HTTP 200, total = 0)
     accompanied by an `OperationOutcome` of severity `warning` with
     `issue.code = multiple-matches` (or equivalent) so the Requester
     can distinguish "no PDMP record" from "could not uniquely identify
     the patient."
   - The Responder **SHALL NOT** disclose how many candidates matched or
     any candidate identifiers (privacy / minimum necessary under
     state PDMP statutes).
2. Add an Implementer Note pointing out that, because state PDMP laws
   typically constrain the demographics that may be queried, Requesters
   that receive a `multiple-matches` outcome should escalate to a
   manual workflow rather than re-query with additional PHI; the IG
   does not define a programmatic refinement loop.

This is a narrative/conformance-text change only; it does not alter the
declared SearchParameter set or the CapabilityStatement.

#### Justification

This addresses exactly what the reporter asks: define the expected
Responder behavior when the demographics do not uniquely identify a
patient, and document the practical consequence for Requesters. An
`OperationOutcome` with a distinct issue code is the standard FHIR
mechanism for communicating "could not process" alongside an empty
result set, and it is consistent with patterns already used in IHE PDQm
and other patient-matching contexts. Keeping the existing parameter set
unchanged is consistent with the Pharmacy WG's prior decision to
constrain PHI exposure on the PDMP query channel and avoids re-opening
the broader scope of patient identification, which is a separate
policy/statutory question.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to the narrative change in Disposition A, expand the
Responder's required search-parameter combination to allow optional
disambiguating parameters that the Requester MAY include and the
Responder SHALL honor when present, without making them mandatory.
Concretely, on the MedicationDispense Search conformance:

- Add `subject:Patient.address-postalcode` (SHOULD support) and
  `subject:Patient.gender` (SHOULD support) as optional refinement
  parameters that the Requester MAY combine with the existing
  name+DOB triple.
- Document that when a Responder returns the `multiple-matches`
  `OperationOutcome` from Disposition A, the Requester MAY retry the
  search with one or more of these optional parameters.
- Leave the existing first-name / family-name / birthdate combination
  as the SHALL-supported minimum so existing Requester implementations
  remain conformant.

#### Justification

This actually closes the loop the reporter describes ("the PDMP
requester is not able to refine the query parameters and research") by
giving the Requester a defined refinement path instead of forcing every
ambiguous case into a manual workflow. Using `SHOULD support` for the
new parameters preserves backward compatibility for existing PDMP
Responders, and constraining the additions to gender and postal code
keeps the disclosed PHI surface small. The trade-off vs Disposition A
is real: many state PDMP statutes enumerate exactly which demographics
may be transmitted, so any added parameter risks putting the IG ahead
of state law and forcing implementers into per-jurisdiction profiling.

---

### Disposition C: Decline

#### Proposal

Mark the issue *Persuasive with Modification → Not Persuasive* (or
*Considered for Future Use*) on the basis that (a) the ticket conflates
two distinct concerns — Responder behavior on ambiguous match (a
narrative gap) and Requester-side refinement (a policy/scope question)
— and (b) the underlying patient-matching question is out of scope for
the US Medication / PDMP IG, which deliberately constrains the query
surface to comply with state PDMP statutes. Direct the reporter to the
Patient $match operation and to state-specific PDMP policy as the
appropriate venues.

#### Justification

The PDMP Responder operates against a state-controlled data store whose
allowable query parameters are typically fixed by statute, not by FHIR
profiling. Attempting to define a programmatic disambiguation loop in
the IG risks creating a normative expectation that conflicts with state
law, and the existing FHIR machinery (empty bundle + standard
`OperationOutcome`) is already sufficient for a Responder that wishes
to communicate "ambiguous." Declining keeps the IG narrowly scoped.
The weakness of this disposition is that it leaves the documentation
gap the reporter actually called out unaddressed, which is what got
the ticket reopened in 2023 in the first place.

---

### Recommendation

**Recommended disposition:** A

The ticket is, at its core, a request for the IG to *say* something it
currently does not — what a Responder must do when name+DOB are not
unique. Disposition A directly fills that gap with a small, defensible
conformance addition (empty bundle + `multiple-matches` OperationOutcome)
that uses standard FHIR mechanisms and does not push the IG beyond what
state PDMP statutes typically allow. Disposition B is attractive because
it actually solves the Requester's refinement problem, but it materially
changes the PHI exposure surface of the PDMP query and almost certainly
needs separate legal/regulatory review per state — that work is better
tracked as its own ticket against the next PDMP IG ballot cycle rather
than bundled into a clarification fix. Disposition C is not recommended:
the ticket has been on the backlog since 2020, was reopened explicitly
because the IG project resumed (PSS-2225), and declining without
addressing the narrative gap leaves implementers in exactly the
position the reporter described. Recommend the WG adopt A now and, if
there is appetite for B, spin out a separate change request scoped to
"optional disambiguating search parameters" so the policy review can
proceed independently.
