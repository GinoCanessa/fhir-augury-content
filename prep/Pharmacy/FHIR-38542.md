## Summary

The ticket asks the Pharmacy work group to move the canonical URLs of
value sets defined in the US Specialty Rx (FHIR) IG from the
`hl7.org/fhir` namespace to `terminology.hl7.org` (THO), on the
principle that only value sets with required bindings to the `code`
data type should remain anchored under `hl7.org/FHIR`. The request
further instructs that a companion UTG ticket be raised to register the
value set(s) in THO, and it points to TSMG as the authority for further
questions. The "10" in the title indicates this is one entry in a
batch of similar THO-anchoring comments imported by Ron G. Parker. No
other Jira, Zulip, or GitHub items were cross-referenced or surfaced
via search, so the ticket stands on its own content.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-home the canonical URL of each value set defined by the US Specialty
Rx (FHIR) IG that does **not** participate in a required binding to a
`code`-typed element. Concretely, for every such `ValueSet`:

1. Change `ValueSet.url` from
   `http://hl7.org/fhir/us/specialty-rx/ValueSet/<id>` to
   `http://terminology.hl7.org/ValueSet/<id>` (using the THO id
   convention the value set is registered under).
2. File a companion UTG ticket requesting that the value set be
   registered in THO, with the same id, title, description, compose
   rules, and copyright as in the IG.
3. Retire the IG-local definition once the THO definition is published,
   leaving the IG to reference the THO canonical by URL only.
4. Update every element binding inside the IG's profiles/extensions
   that references the old canonical to point at the THO canonical
   (including any `valueSetVersion` annotations).
5. Keep anchoring in `hl7.org/FHIR` only for value sets whose bindings
   are `required` and whose underlying data type is `code`.

#### Justification

This is exactly the policy direction the ticket (and TSMG via the
"comment 10" batch) is asking for, and it is consistent with HL7's
broader move to centralise reusable terminology in THO. The 2023-05-04
comment from `lynn_laakso` explicitly makes the previous "Considered
for Future Use" disposition contingent on the anchoring policy being
finalised; if the policy is now clear enough to act on, accepting the
request keeps US Specialty Rx aligned with every other IG following
the same guidance and avoids duplicate governance of the same value
sets in both the IG and THO.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent of the ticket but stage the work behind a
TSMG/Vocab-confirmed policy and an explicit inventory step:

1. Produce an inventory of every `ValueSet` currently defined in US
   Specialty Rx, tagged with (a) binding strength, (b) binding data
   type, and (c) whether the codes involved are owned by HL7 or by an
   external SDO (NCPDP, SNOMED, etc.).
2. Apply the THO anchoring only to the subset that both (i) fails the
   "required binding to `code`" carve-out and (ii) has no external-SDO
   ownership issues that would make THO hosting awkward.
3. For the remainder, leave the canonical under `hl7.org/fhir/us/specialty-rx`
   and document the rationale (external ownership, bespoke to the IG,
   etc.) in the IG's Terminology page.
4. Batch the UTG registrations so a single UTG ticket covers all
   affected value sets, rather than one per value set, to reduce churn.

#### Justification

The request as written is categorical ("anchor in THO unless
required-binding-to-code"), but real IGs tend to have value sets whose
codes are curated externally or whose scope is too IG-specific to
belong in THO. Doing an inventory first lets the work group apply the
policy where it clearly belongs and defend the exceptions, rather than
risk registering IG-specific or externally-owned value sets in THO and
having to un-do them later. A single batch UTG ticket also reduces the
coordination load on the Vocabulary WG, which is the main practical
cost of Disposition A.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — Policy Not Yet Final**, on the
basis that (a) the 2023-05-04 comment explicitly predicates action on
finalisation of HL7's policy on anchoring IG-defined terminology, and
(b) the US Specialty Rx guide has already been published at STU2 with
the current canonicals in use by implementers. Leave the existing
`hl7.org/fhir/us/specialty-rx` canonicals in place and revisit the
question when TSMG/Vocabulary publishes a ratified anchoring policy.

#### Justification

Changing a published `ValueSet.url` is a breaking change for any
implementer who has bound to the current canonical: existing
`CodeSystem`/`ValueSet` references, validator configurations, and
terminology server caches will all need to be updated. Until the
anchoring policy is final, making that change risks having to do it
again in the opposite direction, which would be worse than the current
inconsistency. The original 6-0-0 "Considered for Future Use" vote on
2023-01-30 already reflected this caution, and nothing in the ticket's
history indicates the policy has since been ratified.

---

### Recommendation

**Recommended disposition:** B

Disposition B best matches the state of the evidence. The ticket is a
batch-imported TSMG instruction, not a concrete set of value-set
mappings, and the most recent substantive comment (`lynn_laakso`,
2023-05-04) makes action contingent on a policy that must be
finalised. Going straight to Disposition A without an inventory risks
re-anchoring value sets that either belong elsewhere (externally-owned
terminology) or are too IG-specific to warrant THO hosting, while
Disposition C ignores the clear directional intent of TSMG and leaves
US Specialty Rx out of step with peer IGs. Disposition B commits the
Pharmacy WG to the right policy direction, produces a defensible
inventory, and batches the UTG work — giving TSMG and Vocabulary a
clean artifact to react to while protecting implementers from
unnecessary churn on value sets that were never going to move to THO
anyway.
