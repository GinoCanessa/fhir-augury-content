## Summary

This ticket — titled "Only json?" — questions whether JSON is intended to
be the *only* permitted wire format for PDMP (Prescription Drug Monitoring
Program) FHIR interactions, as the existing IG text states "Support json
resource formats for all PDMP interactions." The reporter is asking the
Pharmacy work group to clarify whether XML (and potentially RDF /
Turtle) are also permitted, or whether the IG genuinely intends to
mandate JSON-only conformance — which would diverge from the FHIR core
specification's expectation that conformant servers support at least
JSON or XML and ideally both. The ticket was originally resolved
"Considered for Future Use" in 2023-01-18 (Hatem / McNeil, 4-0-0)
because the PDMP IG had been withdrawn, and was reopened 2023-11-28 by
the reporter once the IG project (PSS-2225) was being taken up again.
There are no Augury-cached cross-references to other Jira tickets,
Zulip threads, GitHub items, or Confluence pages for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a request for clarification and update the PDMP IG
"Existing Wording" so that it explicitly states JSON is the **required**
format for PDMP interactions and that XML / RDF support is **out of
scope** for this IG. Concretely, in the PDMP IG narrative page that
contains "Support json resource formats for all PDMP interactions",
replace the sentence with: "All PDMP actors **SHALL** support the JSON
representation of FHIR resources (`application/fhir+json`) for every
interaction defined by this guide. Support for the XML representation
is **OPTIONAL**; servers that elect to support XML MUST do so per the
core FHIR specification." Reflect the same constraint in the IG's
CapabilityStatement(s) by setting `CapabilityStatement.format` to
include `application/fhir+json` (and listing `application/fhir+xml`
only if the IG decides to allow it).

#### Justification

The existing wording is ambiguous — "support json" reads as a minimum
rather than an exclusive requirement, which is exactly the confusion
the reporter is flagging. Pinning the IG to JSON-mandatory aligns with
how nearly every modern US realm IG (US Core, Da Vinci, etc.) handles
wire format in practice (JSON-required, XML-optional), keeps PDMP
implementers — many of whom integrate via lightweight web stacks —
from having to ship two parsers, and is consistent with the original
PDMP project sponsor's intent as captured in the ticket text. It also
gives the work group a single, testable conformance statement to put
into the IG's CapabilityStatement, rather than leaving format support
to implementer interpretation.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mandating JSON-only, soften the IG to align with the FHIR
core conformance rules: require that PDMP servers support **at least
one** of `application/fhir+json` or `application/fhir+xml`, with JSON
**RECOMMENDED**, and require clients to honour the format(s) the
server's CapabilityStatement advertises. Update the PDMP IG narrative
to read: "PDMP servers **SHALL** support at least one of the JSON or
XML FHIR representations and **SHOULD** support JSON. Servers
**SHALL** advertise supported formats via
`CapabilityStatement.format`. Clients **SHALL** honour content
negotiation per the core FHIR REST specification." Leave
`CapabilityStatement.format` unconstrained beyond the core rules.

#### Justification

This avoids creating a PDMP-specific divergence from FHIR core
conformance and preserves interoperability with existing HL7
infrastructure (validators, reference servers, terminology services)
that emit both JSON and XML. It also future-proofs the IG against
regulatory environments where state-level PDMP regulators have
historically required XML payloads (e.g., NCPDP-adjacent stacks), and
keeps the IG aligned with sibling Pharmacy IGs. The trade-off vs.
Disposition A is mild ambiguity for implementers about which format to
choose; that is mitigated by the SHOULD-JSON guidance and by the
CapabilityStatement advertisement requirement.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive / Already Addressed**. The base FHIR
specification (Resource Formats / RESTful API sections) already governs
how servers advertise supported MIME types via
`CapabilityStatement.format` and how clients perform content
negotiation; the PDMP IG's "support json" sentence is best read as
profile-level guidance layered on top of those core rules and does not
need to be re-litigated in this IG. Resolve the ticket with a note
pointing the reporter at the core FHIR conformance text and at
`CapabilityStatement.format`.

#### Justification

The question is genuine but the answer is fully covered by the core
FHIR specification, so adding new normative text in the PDMP IG risks
either restating core (low value) or contradicting it (harmful).
Declining is also defensible because the IG is currently being
re-scoped under PSS-2225, and any wire-format wording change is more
appropriately handled as part of the broader IG re-write rather than
as a one-off ticket disposition on a 2020-vintage spreadsheet import.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of three considerations: (1) it
directly resolves the reporter's confusion by stating, in the IG, what
formats are required vs. optional; (2) it does so without inventing a
PDMP-specific divergence from FHIR core conformance, which would create
work for implementers who already support both representations and
problems for tooling that assumes the standard rules; and (3) it
preserves the work group's ability to tighten to JSON-only later
(Disposition A) once PSS-2225 has produced concrete implementer
feedback, whereas going JSON-only first and loosening later would be a
breaking change. Disposition C is tempting given the IG's withdrawn /
re-emerging status, but the ticket has been explicitly reopened by the
reporter precisely because the IG is being taken up again, so closing
without an answer leaves the same ambiguity in the rebooted IG.
