## Summary

Eric Haas raises a semantic-clarity concern on `MedicationRequest`: the
relationship between `priorPrescription` ("Reference to an order/prescription
that is being **replaced** by this MedicationRequest") and `basedOn` ("A plan
or request that is **fulfilled** in whole or in part by this medication
request") is not clearly differentiated for two very common pharmacy scenarios
— refills and dose changes. For each scenario it is unclear which element the
implementer should populate. The reporter offers two paths forward:
(1) collapse `priorPrescription` into `basedOn` (i.e., remove the dedicated
element and rely on the workflow-pattern element), or (2) keep both elements
but add clarifying documentation describing the use cases for each.

The question lands directly on top of recent community discussion. A
[Zulip exchange in `#implementers`](https://chat.fhir.org/#narrow/stream/implementers/topic/Definition%20of%20MedicationRequest.priorPrescription)
in October 2025 asked the same question and was answered by Jean Duteau —
"It was always meant to be used for renewals or replacements. We just made
that clearer in R5". A second thread in
[`#workflow > Prolongation / refills`](https://chat.fhir.org/#narrow/stream/workflow/topic/Prolongation%20%2F%20refills)
on the same day this ticket was filed shows Lloyd McKenzie agreeing with
`priorPrescription` for refill/renewal, while Jose Costa Teixeira pushes back
that "succeeds" and "replaces" mean different things and that the
specification needs more explanation — Lloyd explicitly suggests "It might be
worth a change request to clarify", and this ticket appears to be the
result.

The same reporter has another open Pharmacy ticket,
[FHIR-54367](https://jira.hl7.org/browse/FHIR-54367), raising a parallel
"redundant concept" complaint about `basedOn` vs `authorizingPrescription`
on `MedicationDispense`, which suggests the WG is being asked to think
holistically about how `MedicationRequest`/`MedicationDispense` use the
generic workflow `basedOn` element vs. medication-specific reference
elements. A previous Pharmacy ticket,
[FHIR-39142](https://jira.hl7.org/browse/FHIR-39142) (Published,
Persuasive), already rewrote the `priorPrescription` short description and
is the source of the wording the reporter is now criticising; and
[FHIR-54876](https://jira.hl7.org/browse/FHIR-54876) (FHIR Infrastructure,
Persuasive with Modification) is doing exactly the requested
documentation-clarification job for the parallel `Request.replaces`
element on `ServiceRequest`.

## Details

**Description:**

> `priorPrescription` vs `basedOn` — Confusing Relationship Semantics
>
> - `priorPrescription`: *"Reference to an order/prescription that is being **replaced** by this MedicationRequest"*
> - `basedOn`: *"A plan or request that is **fulfilled** in whole or in part by this medication request"*
>
> **Confusion:**
> - Refill scenario: Is the refill `basedOn` the original? Or does it `replace` via `priorPrescription`?
> - Dose change: New order with different dose. Is the old one `priorPrescription` or was the new one `basedOn` the old?
>
> When would a MedicationRequest be "based on" another MedicationRequest vs "replace" it?
>
> Propose:
> 1) Rolling `priorPrescription` into `basedOn`, or
> 2) Provide clarifying documentation to support the use cases for both

No comments have been posted on the ticket as of the report date.

## Related Zulip Discussions

### [implementers > Definition of MedicationRequest.priorPrescription](https://chat.fhir.org/#narrow/stream/implementers/topic/Definition%20of%20MedicationRequest.priorPrescription)

Patrick Werner (October 2025) raised exactly the question this ticket asks:
the R4 definition was "a link to a resource representing an earlier order
related order or prescription", which covered renewals; in R5+ it became
"Reference to an order/prescription that is being replaced", which sounds
narrower. Patrick asked whether this was a deliberate scope change and
whether `priorPrescription` should still be used for renewals. Jean Duteau
(Pharmacy) replied authoritatively: "It was always meant to be used for
renewals or replacements. We just made that clearer in R5." Consensus on
the thread: `priorPrescription` *is* the right element for renewal,
replacement, and dose-change scenarios — but the wording does not convey
that scope to readers, which is exactly the ambiguity FHIR-54447 wants
fixed. Participants: Patrick Werner, Jean Duteau.

### [workflow > Prolongation / refills](https://chat.fhir.org/#narrow/stream/workflow/topic/Prolongation%20%2F%20refills)

Started by Jose Costa Teixeira on 2026-01-15 (the day before this ticket
was filed) asking how to model "after 5 physiotherapy sessions, 5 more are
needed". Lloyd McKenzie answered that for `MedicationRequest` the right
element is `priorPrescription` and for `ServiceRequest` it is `replaces`,
characterising the relationship as "succeeds" rather than invalidates.
Jose pushed back hard that "succeed" and "replace" are semantically
different and that conflating them affects downstream behaviour
(reimbursement reset, billing counts), and asked whether this is a
Workflow-call or Clinical-on-FHIR-call topic. Lloyd concluded "I'm not
sure they're being used differently in the model. It might be worth a
change request to clarify." That suggestion is plausibly the immediate
trigger for FHIR-54447 (and FHIR-54876 in the parallel `ServiceRequest`
case). The thread shows there is real implementer disagreement about
whether replace = succeed = renew, which means a documentation-only
disposition must address all three terms, not just one.
Participants: Jose Costa Teixeira, Lloyd McKenzie.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's option (2): retain both `MedicationRequest.basedOn`
and `MedicationRequest.priorPrescription`, and tighten the documentation
on both. Concretely, in the FHIR Core specification (Pharmacy WG):

1. Change the short description of `MedicationRequest.priorPrescription`
   from "Reference to an order/prescription that is being replaced by
   this MedicationRequest" to wording that matches the WG's actual intent
   as stated by Jean Duteau — for example: *"Reference to a prior
   medication order that this MedicationRequest renews, replaces,
   continues, or otherwise succeeds (e.g., refills, dose changes,
   substitutions)."*
2. Add a definition-level comment on `priorPrescription` enumerating the
   intended scenarios (refill, renewal, dose change, route change,
   substitution) and explicitly distinguishing it from `basedOn`.
3. Add a complementary comment on `MedicationRequest.basedOn` clarifying
   that it points to the protocol, plan, CarePlan, ServiceRequest,
   ImmunizationRecommendation, RequestOrchestration, or higher-level
   MedicationRequest that this request *fulfils* — and that for "replaces
   a previous prescription" the implementer should use
   `priorPrescription`, not `basedOn`.
4. Add a short narrative paragraph (or a small table) to the Workflow
   section of `medicationrequest.html` walking the two example scenarios
   in the ticket (refill, dose change) and showing which element is
   populated in each.

Decline option (1) — collapsing `priorPrescription` into `basedOn` — as
out of scope here; see Disposition C for why.

#### Justification

The Zulip evidence is unambiguous that the WG already has the answer
("`priorPrescription` is for renewal **and** replacement, made clearer in
R5") but that the spec wording does not convey it. FHIR-39142 is the
ticket that introduced the current wording, and the WG resolved it as
Persuasive on essentially the same grounds — a documentation tightening.
Doing the same kind of fix here keeps the two elements distinct (which
matters because `basedOn` is a Workflow-pattern element shared across all
Request resources and was added through FHIR-10456 as part of that
alignment), while removing the gap that is causing implementers to ask
the same question on Zulip year after year. It is also consistent with
how FHIR Infrastructure resolved the parallel FHIR-54876 against
`ServiceRequest.replaces` (also Persuasive with Modification, also a
short-description + comment expansion).

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-54447 as one half of a coordinated cleanup of the medication
ordering relationship elements, and resolve it together with the
reporter's companion ticket FHIR-54367 plus the FHIR-I work on
FHIR-54876. Concretely:

1. Apply the documentation tightening from Disposition A to
   `priorPrescription` and `basedOn` on `MedicationRequest`.
2. Adopt the same disposition style on `MedicationDispense.basedOn` vs
   `MedicationDispense.authorizingPrescription` (FHIR-54367) so the two
   resources tell the same story about "generic Workflow `basedOn` vs.
   medication-specific authorizing/prior reference".
3. Coordinate with FHIR Infrastructure (FHIR-54876) to align the
   `Request.replaces` short description and comment language so that
   `priorPrescription` reads as the medication specialisation of the same
   concept.
4. Add a single Pharmacy module page section ("Relating
   MedicationRequests / Dispenses to prior orders") that holds the
   normative explanation and is linked from each affected element
   definition.

#### Justification

The Zulip threads make clear this is one underlying concept that surfaces
in multiple places (`Request.replaces`, `MedicationRequest.priorPrescription`,
`MedicationDispense.authorizingPrescription`, `MedicationRequest.basedOn`,
`MedicationDispense.basedOn`). Fixing them in lock-step is much more
useful to implementers than fixing each in isolation, and avoids a future
in which the medication and workflow definitions drift again. The trade-off
versus Disposition A is that this requires cross-WG coordination
(Pharmacy, FHIR Infrastructure, possibly Workflow) and a small amount of
extra editorial scope, but no additional structural change.

---

### Disposition C: Decline

#### Proposal

Decline the request. Specifically:

1. Reject option (1) — do **not** roll `priorPrescription` into `basedOn`.
   `basedOn` is the Workflow Request-pattern element shared across every
   Request resource (`ServiceRequest`, `MedicationRequest`,
   `DeviceRequest`, …) and means "fulfils a higher-level plan/protocol/
   request". `priorPrescription` is the medication-specific peer of
   `Request.replaces` and means "supersedes a prior order of the same
   type". Conflating them would lose information, would be a breaking
   change for every implementer that already uses either element, and
   would have to be coordinated with Workflow / FHIR-I rather than handled
   as a Pharmacy edit.
2. On option (2), point the reporter at FHIR-39142 (already Persuasive
   and Published, which produced the current wording) and at the
   `#implementers > Definition of MedicationRequest.priorPrescription`
   thread, where the question is already answered authoritatively by a
   Pharmacy co-chair. Mark FHIR-54447 as duplicate / not persuasive on
   the grounds that the desired clarification has been delivered.

#### Justification

Defensible for option (1): conflating `priorPrescription` and `basedOn`
would be a breaking change to a normative resource and would violate the
Workflow Request pattern. Less defensible for option (2): although
FHIR-39142 has run, the Zulip evidence (October 2025 and January 2026)
shows that experienced implementers still cannot tell the elements apart
from the spec text alone, so labelling FHIR-54447 a duplicate is hard to
justify. This disposition is included for completeness but is the weakest
of the three.

---

### Recommendation

**Recommended disposition: B.**

Disposition B is recommended over A because the surrounding evidence
already points beyond just `MedicationRequest`. The same reporter (Eric
Haas) has a parallel open ticket (FHIR-54367) about the analogous
`basedOn` vs. `authorizingPrescription` confusion on `MedicationDispense`,
and FHIR Infrastructure is concurrently fixing exactly the same wording
problem on `ServiceRequest.replaces` in FHIR-54876 — the very thread that
appears to have triggered FHIR-54447 ends with Lloyd McKenzie suggesting
a change request to clarify. Solving these in isolation will produce
three near-identical edits with three slightly different wordings, which
is the situation that produced the ambiguity in the first place. A
coordinated documentation pass (still no structural change, still no
breaking change) directly answers the question Eric is asking, aligns
Pharmacy with the Workflow Request pattern, and gets the Pharmacy
medication elements off the recurring Zulip FAQ list. Disposition A is a
perfectly acceptable fallback if cross-WG coordination is not feasible
in this cycle; Disposition C should be avoided because the recurring
Zulip discussions are direct evidence that FHIR-39142's earlier fix did
not land for implementers.
