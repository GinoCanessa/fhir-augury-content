## Summary

The reporter, May Terry, requests that the date elements on the US
Standardized Medication Profile (SMP) `MedicationStatement` profile —
specifically `effective[x]` and `assertedDate` — be promoted from optional
to **Must Support**. The rationale is that recording when a medication is
(or was) being taken, and when that information was asserted, is essential
for establishing a usable medication timeframe in interoperable medication
records. The ticket has been triaged with a resolution of **Persuasive**,
indicating the Pharmacy work group agrees with the proposed change. No
related Jira tickets, Zulip discussions, or GitHub items were surfaced via
cross-reference or search.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Standardized Medication Profile (SMP) `MedicationStatement`
profile, change the **Must Support** flag from `false` to `true` on:

- `MedicationStatement.effective[x]` (covering both `effectiveDateTime`
  and `effectivePeriod` choices), and
- `MedicationStatement.assertedDate` (the extension carrying the date
  the information was asserted; in R5-aligned profiles this is
  `MedicationStatement.dateAsserted`).

Cardinality remains `0..1`. Add Must Support guidance text to the SMP
profile narrative explaining that producers SHALL populate at least one
of these elements when known and consumers SHALL be able to process
them, in line with the US Realm Must Support definition. Update the
SMP profile differential, the StructureDefinition snapshot, the
profile's element-level descriptions, and any conformance test fixtures
/ examples that previously omitted these elements. Regenerate the IG.

#### Justification

Must Support on the timing elements directly addresses the ticket's
stated need to "establish a medication timeframe", and the work group
has already marked the resolution as **Persuasive**, signalling
agreement. The change is a constraint tightening on a US-Realm profile
(not the base resource), so it does not affect base FHIR conformance and
follows the established US-Realm pattern (e.g., US Core
MedicationRequest / MedicationStatement profiles already flag timing /
asserted-date elements as Must Support). The change is low-risk, scoped
to a single profile, and consistent with how clinical timing data is
treated elsewhere in US medication profiles.

---

### Disposition B: Alternative Approach

#### Proposal

Mark `MedicationStatement.effective[x]` as **Must Support**, but treat
`assertedDate` differently:

- Either (i) leave `assertedDate` as optional but add a profile
  invariant / "Must Support if known" guidance stating that systems that
  capture an assertion date SHALL expose it, or
- (ii) replace the Must Support flag on `assertedDate` with an
  obligation (`obligation-actor` + `will-populate` /
  `can-populate`) targeted at specific actors (e.g., source EHRs vs.
  aggregators), so that only systems for which the date is meaningful
  are obligated to support it.

Pair this with examples in the SMP IG showing both "effective only" and
"effective + assertedDate" patterns.

#### Justification

`effective[x]` is unambiguously needed to establish a medication
timeframe and is the strongest part of the request. `assertedDate`,
however, is provenance-style metadata: not every source system records
or surfaces it (paper-derived data, reconciled lists, claims-derived
medication statements), and forcing Must Support on all conformant
producers can create populate-with-best-guess anti-patterns. An
obligation-based or conditional-MS approach preserves the reporter's
intent (the date is honoured when present) without burdening producers
that genuinely cannot supply it, which keeps the SMP profile achievable
across the full range of US medication data sources.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave both `effective[x]` and `assertedDate` as
optional in the SMP `MedicationStatement` profile, on the basis that
(a) the elements are already available and conformant systems may
populate them today, and (b) Must Support obligations should be driven
by published SMP use cases rather than by general desirability of
timing data.

#### Justification

This option is included for completeness. It would be defensible only
if the work group concluded that a meaningful subset of SMP source
systems cannot reliably supply either element and that flipping Must
Support would harm conformability more than it helps interoperability.
Given that the resolution is already recorded as **Persuasive**, this
disposition is the weakest of the three.

---

### Recommendation

**Recommended disposition:** A

The ticket is narrow, the work group has already marked the resolution
**Persuasive**, and the requested change aligns with established
US-Realm practice of flagging clinically essential timing elements as
Must Support. Disposition A delivers exactly what the reporter asked
for with minimal scope (a profile-level constraint tightening on a
single SMP profile) and no impact on base FHIR resources. Disposition B
is a reasonable fallback if Pharmacy hears, during balloting, from
implementers who cannot reliably populate `assertedDate`; in that case
the obligation-based variant for `assertedDate` is the cleanest
mitigation. Disposition C should only be revisited if new evidence
emerges that Must Support on these elements is unachievable for a
material share of SMP producers.
