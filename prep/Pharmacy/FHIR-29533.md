## Summary

FHIR-29533 questions whether the **US Prescription Drug Monitoring Program
(PDMP) FHIR Implementation Guide** is correct in mandating SMART Backend
Services for *every* PDMP Responder, or whether that requirement should only
apply when the implementation is itself a SMART on FHIR app. The ticket cites
the PDMP Responder authorization paragraph that reads "The PDMP Responder
SHALL support the SMART Backend Services Authorization Guide for verifying
authentication and providing authorization to PDMP Requestors." and asks
whether that SHALL is appropriate when the deployment is a pure
system-to-system integration with no end-user SMART app in the picture.

The ticket has had a notable lifecycle: it was originally dispositioned
"Considered for Future Use" on 2023-01-18 because the PDMP IG had been
withdrawn, then re-opened on 2023-11-28 by Scott Robertson when the IG was
picked back up under [PSS-2225 PDMP on FHIR](https://confluence.hl7.org/display/PHAR/US+Prescription+Drug+Monitoring+Program+%28PDMP%29+FHIR+Implementation+Guide).
It currently sits in **Triaged** state at **Highest** priority and is
unassigned, with no resolution recorded since the revert.

There are **no cross-referenced Jira tickets, Zulip threads, Confluence
pages, or GitHub items** in FhirAugury for FHIR-29533. The PDMP IG is
published at `http://hl7.org/fhir/us/pdmp/STU1`, but the corresponding
authoring repo (historically `HL7/FHIR-ONC-Meds`) is not present in the
GitHub source — see "Repo Context" below.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the conformance verb on the PDMP Responder authorization
requirement so it is conditional on deployment context. Concretely, in
the PDMP IG's authorization narrative (the "Existing Wording" the
reporter quotes) and in the PDMP Responder CapabilityStatement's
`security` element, change

> "The PDMP Responder **SHALL** support the SMART Backend Services
> Authorization Guide for verifying authentication and providing
> authorization to PDMP Requestors."

to

> "When the PDMP Responder is consumed by a SMART on FHIR app or by an
> external organisation over the public internet, it **SHALL** support
> the SMART Backend Services Authorization Guide. For other deployment
> contexts (e.g., closed B2B integrations using a different mutually
> agreed authorization mechanism), SMART Backend Services **SHOULD** be
> supported."

The same softening must be mirrored in the corresponding requirement on
the PDMP Requestor side and in any conformance test plan tied to that
clause.

#### Justification

This is the most literal reading of the ticket: the reporter is
explicitly questioning whether the SHALL applies when "the implementation
is not a SMART on FHIR app". Conditional SHALL/SHOULD wording is a
well-established FHIR pattern (e.g., US Core's conditional support
guidance) and gives non-SMART deployments a defensible path while still
making SMART Backend Services the default expectation. It also matches
the stated intent of SMART Backend Services itself, which is the
authorization profile for *backend* (no end-user) clients — i.e., the
profile is already aimed at the case the ticket is worried about, but
the IG text doesn't make that connection.

### Disposition B: Alternative Approach — Keep SHALL, Clarify the Wording

#### Proposal

Retain the SHALL but rewrite the surrounding narrative to remove the
implied coupling between "SMART Backend Services" and "SMART on FHIR
app". Specifically, in the PDMP IG's PDMP Responder authorization
section, replace the current paragraph with text that:

1. States plainly that **SMART Backend Services is a system-to-system
   OAuth 2.0 client-credentials profile** and is **independent of**
   whether the PDMP Requestor is a SMART on FHIR end-user app, an EHR
   backend, a state HIE service, or another B2B integration.
2. Keeps the conformance statement: "The PDMP Responder **SHALL** support
   the SMART Backend Services Authorization Guide for authenticating
   PDMP Requestors and authorizing PDMP queries, regardless of whether
   any human-facing SMART app is involved."
3. Adds a short non-normative note pointing to the SMART Backend
   Services spec section that explains the no-end-user use case, and
   to the PDMP Requestor CapabilityStatement so implementers can see
   the matching requirement on the requestor side.

No CapabilityStatement value changes are required under this option; the
change is purely narrative and (optionally) a `description` refinement on
the `security` element.

#### Justification

The ticket reads as a question driven by terminology confusion — the
word "SMART" suggests an end-user app, even though SMART Backend Services
explicitly targets backend clients. Clarifying the wording rather than
weakening the conformance verb preserves a single, predictable
authorization profile across all PDMP Responders, which is the stronger
position for an information-sensitive domain like prescription monitoring
where divergent per-deployment auth mechanisms are a security and
interoperability liability. It also avoids the conformance-test churn
that Disposition A would cause and matches how comparable IGs
(Da Vinci PDex, US Core server profiles, Bulk Data) handle the same
distinction.

### Disposition C: Decline (Not Persuaded)

#### Proposal

Decline the change. Mark FHIR-29533 as "Not Persuaded" with the
explanation that the existing SHALL is intentional: a single mandatory
authorization profile is a deliberate design choice for the PDMP IG, and
SMART Backend Services is the correct profile for system-to-system
clients regardless of whether a SMART app is in front of them. The IG
text remains as published.

#### Justification

PDMP data is among the most sensitive clinical data exchanged over FHIR
and is subject to state-level regulation; a uniform, well-defined
authorization profile materially reduces the attack surface and
implementation variance compared with "use SMART Backend Services *or*
something else". SMART Backend Services is, by design, the no-end-user
B2B profile, so the premise of the ticket — that the SHALL is awkward
when there is no SMART app — is already addressed by the cited
specification itself. Declining is defensible on those grounds, though it
leaves the underlying readability problem in the IG text unaddressed.

### Recommendation

**Recommended disposition: B (Alternative Approach — Keep SHALL,
Clarify the Wording).**

Disposition B addresses the real problem the ticket surfaces — the
wording invites readers to assume "SMART Backend Services" implies a
SMART on FHIR end-user app, which it does not — without giving up the
single mandatory authorization profile that makes a national-scope PDMP
IG tractable for implementers and regulators. Disposition A would
fragment PDMP authorization across deployments and create a conformance
testing burden disproportionate to the underlying issue, which is
fundamentally one of explanation rather than substance. Disposition C
leaves the same ambiguous paragraph in the published IG, all but
guaranteeing the same question is asked again. Given the ticket is at
**Highest** priority, was reopened specifically because the IG is being
revived under PSS-2225, and is currently unassigned, the work group's
best move is to take Disposition B, assign an editor to the
corresponding narrative section, and confirm the active authoring repo
so the change can be made and a repo-analysis briefing produced for any
follow-up tickets in this tranche.
