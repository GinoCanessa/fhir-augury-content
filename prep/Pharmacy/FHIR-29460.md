## Summary

The reporter asks the Pharmacy work group to make the NCPDP SCRIPT version
explicit in the US Prescription Drug Monitoring Program (PDMP) FHIR
Implementation Guide's mapping section. The current heading "NCPDP Mappings
for PDMP Request" is ambiguous as to whether mappings were authored against
NCPDP SCRIPT 10.6 or NCPDP SCRIPT 2017071. Because 2017071 is the current
named version of the standard, the request is to (a) state the source
version explicitly and (b) standardize all mappings going forward on
NCPDP SCRIPT 2017071 RxHistoryRequest, renaming the section to
"NCPDP SCRIPT 2017071 RxHistoryRequest Mappings for PDMP Request". No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found
through the FHIR Augury index, so the request stands on its own and must
be evaluated against the current state of the PDMP IG (which was
withdrawn and is being reopened under PSS-2225).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US PDMP FHIR Implementation Guide's mapping page (currently
titled "NCPDP Mappings for PDMP Request"):

1. Rename the page/section heading to **"NCPDP SCRIPT 2017071
   RxHistoryRequest Mappings for PDMP Request"**.
2. Add an explicit version statement at the top of the page identifying
   the source standard as **NCPDP SCRIPT Standard, Implementation Guide,
   Version 2017071** and the specific transaction as
   **RxHistoryRequest**.
3. Audit each mapping row in the table for any element that was authored
   against NCPDP SCRIPT 10.6 and either re-map to the 2017071 equivalent
   or note the divergence. Where a 10.6 segment/field has no 2017071
   counterpart (or vice versa), call this out explicitly.
4. Update any cross-page references (TOC, narrative pages that link to
   the mapping page, conformance statements) to use the new title.
5. Adopt 2017071 as the baseline NCPDP SCRIPT version for the IG going
   forward, so future mapping work in the PDMP IG starts from 2017071
   rather than 10.6.

#### Justification

The reporter's request is editorial-plus-correctness: an IG that maps
between FHIR and an external standard must name the version of that
external standard, otherwise implementers cannot validate the mapping.
NCPDP SCRIPT 2017071 has been the current named version since 2017 and
is what real-world PDMP implementations exchange today, so anchoring the
IG on 2017071 aligns the spec with the deployed reality. The change is
narrative + mapping-table accuracy; it does not alter FHIR resource
shapes or generated artifacts. Because the PDMP IG was withdrawn and is
now being reopened under PSS-2225, this is exactly the right moment to
correct version labeling before any new ballot cycle locks the wording
in again.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substance of the request, but model NCPDP SCRIPT version as
explicit, structured metadata rather than only a renamed heading:

1. Keep a generic mapping page title (e.g., "NCPDP SCRIPT Mappings for
   PDMP Request") but introduce a **Source Standard** metadata block
   ("Standard: NCPDP SCRIPT, Version: 2017071, Transaction:
   RxHistoryRequest") rendered consistently at the top of every mapping
   page in the IG.
2. For each mapping row, add a **Source Version** column (default
   2017071) so future revisions can extend mappings to a newer NCPDP
   SCRIPT version without renaming the page or fragmenting it into
   per-version pages.
3. Optionally publish a small CodeSystem or NamingSystem entry capturing
   the NCPDP SCRIPT versions the IG knows about, so other artifacts
   (CapabilityStatement, ConceptMap) can reference the version
   canonically.

#### Justification

A bare title rename solves today's ambiguity but bakes the version into
prose. NCPDP SCRIPT will revise again, and when it does, the IG will be
back in the same situation — re-titling pages and updating prose. A
structured "Source Standard" header plus a per-row version column gives
implementers the same clarity as Disposition A while making future
version bumps a one-line metadata change rather than a documentation
sweep. The trade-off is more initial editorial work and a slightly more
complex page template; the upside is that the same pattern can be
reused if/when other NCPDP transactions (e.g., Telecom standards) are
added to the PDMP IG family.

---

### Disposition C: Decline

#### Proposal

Decline as **No change required** on the basis that the request was
already dispositioned (Considered for Future Use, 2023-01-18) at a time
when the IG was withdrawn, and the reopening under PSS-2225 should
re-evaluate version labeling holistically rather than via a four-year-old
spreadsheet-imported comment.

#### Justification

The original disposition was procedurally sound for a withdrawn IG, and
PSS-2225 is the appropriate vehicle to set the IG's normative NCPDP
SCRIPT version. Resolving this single comment in isolation risks
inconsistency with whatever version policy the reopened project formally
adopts — e.g., if PSS-2225 ultimately targets a successor to 2017071,
applying the rename now creates a second editorial pass. Declining here
and asking the project team to address NCPDP SCRIPT versioning as a
PSS-2225 deliverable keeps version policy in one place.

---

### Recommendation

**Recommended disposition:** A

The request is small, concrete, demonstrably correct (NCPDP SCRIPT
2017071 is the current named version), and has been blocked only by the
IG's prior withdrawn status — which the 2023-11-28 reopen note explicitly
removes. Disposition A directly answers the reporter, lands minimal
narrative changes, and is the cheapest way to remove a real ambiguity for
implementers reading the mapping page. Disposition B is attractive on
paper but is properly a separate editorial pattern decision for the
reopened PDMP project; it should not block this ticket. Disposition C
re-defers a question that has already been deferred once and that the
work group is now equipped to answer. Accepting now also gives PSS-2225
an authoritative version baseline (2017071) to build subsequent mapping
work on top of.
