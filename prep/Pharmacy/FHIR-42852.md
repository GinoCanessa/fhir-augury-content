## Summary

Raised out of Connectathon 43 against the US PDMP (Prescription Drug
Monitoring Program) FHIR IG, this Change Request asks the Pharmacy work
group to add explicit data-rendering requirements to PDMP query
responses. Some U.S. states mandate that PDMP responses be presented to
the requesting clinician in a specific human-readable form (commonly a
state-prescribed text or HTML report) in addition to (or instead of)
discrete data. The reporter therefore wants the IG to support rendering
of every response bundle as text, HTML, and potentially other formats —
either by carrying the rendered representation inline or by carrying a
link to it — so that responders can satisfy state presentation rules
while still returning discrete FHIR data. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were surfaced for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative section to the US PDMP IG titled "Rendered
Representation of Response Bundles" that requires every PDMP query
response Bundle to carry a human-readable rendering of all discrete data
in the Bundle, in either or both of two forms:

1. **Inline rendering** — a `Composition` resource added as the first
   entry of the response Bundle, with `Composition.section.text` (and/or
   a `Composition.section.entry`-referenced `Binary`) carrying the
   text/HTML rendering. Profile this as `PDMPRenderedResponse`
   (Composition profile) and bind `Composition.type` to a small value
   set covering at least `text/plain` and `text/html`.
2. **By-reference rendering** — a `DocumentReference` resource included
   in the Bundle, with `DocumentReference.content.attachment.url`
   pointing to the externally-hosted rendered document and
   `attachment.contentType` indicating the media type.

Add an IG-level invariant on the response Bundle (`pdmp-rendering-1`)
requiring at least one of these two representations to be present, and
extend the existing PDMP query operation page to describe both
options and the state-driven selection between them.

#### Justification

This matches the request literally: it puts the rendering obligation in
the IG itself rather than asking implementers to invent it. Composition
plus DocumentReference are the FHIR-canonical mechanisms for
"human-readable document derived from data" and "pointer to a rendered
document," respectively, which keeps PDMP aligned with how other
US-Realm IGs (e.g., US Core ProvenanceTargetProfile patterns,
IPS Composition) handle attested narrative content. Stating the
invariant in the IG — rather than relying on each state's profile to
re-state it — gives implementers a single conformance target that
satisfies the strictest state requirements.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mandating a new Composition/DocumentReference structure in
the base PDMP IG, **strengthen and document the existing narrative
mechanisms** and add a single optional carrier for state-specific
rendered output:

1. Tighten the response Bundle profile to require
   `Resource.text.status = "generated"` (or `"extensions"`) and
   non-empty `Resource.text.div` on every clinical resource in the
   Bundle (Patient, MedicationRequest, MedicationDispense, Practitioner,
   etc.), so a conforming response is already self-rendering at the
   resource level.
2. Define **one** optional extension —
   `http://hl7.org/fhir/us/pdmp/StructureDefinition/rendered-response`
   — typed as `Attachment`, slot-able on the response Bundle's
   `Composition` (if present) or on `MessageHeader`/`Parameters`
   wrapper. The Attachment carries either inline base64 (`data`) or a
   `url` to the state-mandated rendered document, with `contentType`
   identifying the format (`text/plain`, `text/html`,
   `application/pdf`, …).
3. Document, but do not mandate, that state-specific PDMP profiles MAY
   make this extension `mustSupport` or `1..1` and MAY constrain
   `contentType` to the format their statute requires.

#### Justification

Every clinical FHIR resource already has a narrative slot specifically
designed for human rendering, and that slot is already widely supported
by tooling. Reusing it — and only adding a single small extension for
the state-mandated "exact rendering" case — keeps the base IG cheap to
implement (no new resources to populate just to satisfy conformance) and
pushes the truly state-specific obligation into state-specific profiles
where it belongs. It also avoids forcing every responder, including
those serving states with no rendering requirement, to assemble a
Composition just to clear conformance. From a backward-compatibility
standpoint, an optional extension is non-breaking for existing PDMP
implementations that have already shipped against the current IG.

---

### Disposition C: Decline

#### Proposal

Decline the change request. Resolve the ticket as **Persuasive with
Modification → No Change** (or **Not Persuasive**), with the rationale
that core FHIR already provides sufficient mechanisms for human-readable
rendering — `Resource.text`, `Composition`, `DocumentReference`,
`Binary`, and `Attachment` — and that *which* of those a responder uses
to satisfy a state mandate is properly the concern of state-specific
PDMP profiles or implementer agreements, not of the base US PDMP IG.
Add a short non-normative note to the IG's "Use of Narrative" page
pointing implementers at those existing mechanisms.

#### Justification

The ticket describes a real implementer need but not a gap in FHIR.
Building a new IG-level rendering obligation risks (a) duplicating
narrative semantics that already exist on every resource, (b) freezing
one of several reasonable carriers (Composition vs. DocumentReference
vs. extension) into the spec when state requirements actually differ,
and (c) breaking conformance for already-deployed PDMP responders that
rely on narrative-only rendering. A "no change, document existing
mechanisms" disposition preserves implementer flexibility while still
acknowledging the connectathon feedback.

---

### Recommendation

**Recommended disposition:** B

The underlying need (states require a specific rendered presentation,
sometimes inline and sometimes by reference) is real and was raised at
connectathon, so a flat decline (C) under-serves the reporter and the
states being modeled. However, fully accepting (A) — making
Composition+DocumentReference a normative part of *every* PDMP response
— pushes IG-wide cost onto responders in jurisdictions that have no
such mandate, and prematurely picks a carrier when state statutes
disagree on whether the rendering must be inline or merely retrievable.
Disposition B threads the needle: it leans on `Resource.text`, which
PDMP responders should already be populating, and adds a single, small,
optional `rendered-response` Attachment extension that state profiles
can promote to `mustSupport` or `1..1` as needed. It is the lowest-cost
change that still gives state profilers a well-defined slot to bind to,
and it keeps the base US PDMP IG implementable by responders whose
states impose no extra rendering rule. Pharmacy should confirm with the
ticket reporter (and any state representatives present at the next
connectathon) whether an Attachment-typed extension is expressive
enough, or whether a small choice between `Attachment` and
`Reference(DocumentReference)` is warranted.
