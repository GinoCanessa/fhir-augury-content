## Summary

The ticket reports that the element name `<RequestingState>`, used in the
US Prescription Drug Monitoring Program (PDMP) FHIR Implementation Guide
(originally filed against the predecessor "US Medication FHIR IG") to
represent the state code of a requesting facility, does not actually
exist as a defined field in any version of the NCPDP SCRIPT standard.
The reporter (Daniel Zhang) points out that the equivalent value is
already conveyed in NCPDP SCRIPT 2017071 via the
`<Facility>/<Address>/<State>` element path, and implies that the IG
should align its element name / mapping with the real NCPDP SCRIPT
structure rather than invent a non-standard `<RequestingState>` field.
The ticket was originally deferred in January 2023 with the resolution
"Considered for Future Use" because the IG had been withdrawn, then
reopened on 2023-11-28 by smrobertson once PSS-2225 (PDMP on FHIR)
revived the IG project. It is currently Triaged at Highest priority and
unassigned.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with this ticket in the index, so the request must be
evaluated on its own facts and on the workgroup's understanding of the
revived PDMP IG scope.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next published version of the US PDMP FHIR Implementation Guide
(under PSS-2225), remove the invented `<RequestingState>` element from
the request profile and replace it with a mapping/element that mirrors
the NCPDP SCRIPT 2017071 structure: the State code of the requesting
facility should be sourced from
`Facility.Address.State` (i.e., the FHIR equivalent of NCPDP's
`<Facility>/<Address>/<State>`). Concretely, on whichever profile
currently carries `RequestingState` (typically the request/query
resource in the PDMP IG, e.g., a Task or MedicationRequest-derived
profile or a custom Bundle entry representing the requesting facility):

- Drop the standalone `RequestingState` element / extension.
- Add or strengthen a slice/binding on the requesting facility's
  `Organization.address.state` (or the `Location.address.state` /
  `Address.state` of the facility component) to carry the state code,
  with the NCPDP-aligned mapping documented in the element's "Mapping"
  table pointing at `Facility/Address/State` in NCPDP SCRIPT 2017071.
- Update narrative and any example bundles in the IG so the state code
  appears on the facility's address rather than as a top-level
  `RequestingState` field.

#### Justification

The reporter's claim is verifiable against the NCPDP SCRIPT 2017071
schema: there is no `<RequestingState>` element, and the state of the
requesting facility is in fact carried under
`<Facility>/<Address>/<State>`. Inventing an IG-only field that does
not map to the underlying standard breaks the IG's stated purpose of
being interoperable with NCPDP SCRIPT and forces implementers to do a
non-obvious translation. Moving the state to the facility's
`Address.state` is also consistent with how every other US Realm IG
(US Core, US Public Health Profiles, etc.) carries jurisdictional
state codes, so it lowers the integration cost for implementers
already handling US Core Organization/Location.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than removing the element, **rename and re-bind** it so that the
IG keeps a single, named slot for the requesting facility's state code
but documents it as a derived/convenience element with an explicit
NCPDP mapping. Concretely:

- Rename `RequestingState` to something that signals provenance
  (e.g., `requestingFacilityState`) and define it as a profile-level
  invariant or extension that **must equal** the
  `Organization.address.state` (or `Location.address.state`) of the
  resource representing the requesting facility in the same Bundle.
- In the element's Mapping table, point the source explicitly at NCPDP
  SCRIPT 2017071 `Facility/Address/State` and note that no native
  NCPDP element of that name exists.
- Mark it `0..1` and `MS` so legacy implementations that previously
  populated `RequestingState` continue to work, while new
  implementations are guided to populate the canonical
  `Facility.Address.State` path.

#### Justification

This preserves backward compatibility with any early adopter (or
prototype) systems that already wrote to `RequestingState` while still
fixing the underlying defect — the element is now correctly named,
correctly mapped to the real NCPDP path, and explicitly described as
derived rather than as a "real" NCPDP field. It also keeps a
single, easily queryable element on the request profile for consumers
that don't want to traverse the facility resource to find the state,
which is a frequent usability complaint with deeply nested PDMP data.
The cost is that the IG now carries a redundant element with an
invariant to maintain.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Persuasive with Modification – No Change** (or
re-resolve as *Considered for Future Use*) on the grounds that the
underlying IG was withdrawn and any element-level corrections should be
absorbed wholesale into the new PSS-2225 PDMP on FHIR work rather than
patched in isolation. Document in the resolution that the new IG must
not reintroduce a `<RequestingState>` element and must instead carry
the state code on `Facility.Address.State`, but do not produce a
discrete change against the withdrawn artifact.

#### Justification

The Pharmacy WG previously voted 4-0-0 to defer this exact ticket as
*Considered for Future Use* on 2023-01-18 specifically because the IG
was withdrawn. Although the reopening under PSS-2225 reactivated the
ticket, the substantive rebuild of the PDMP IG will likely re-author
the request profile from scratch; adopting a targeted fix to a
withdrawn artifact spends WG cycles for little durable benefit.
Closing this with explicit guidance for the new IG captures the
intent without committing the WG to maintaining the legacy artifact.

---

### Recommendation

**Recommended disposition:** A

The defect described is unambiguous and verifiable: `<RequestingState>`
genuinely is not an NCPDP SCRIPT element, and a correctly-named
`Facility.Address.State` mapping already exists in NCPDP SCRIPT
2017071. Because PSS-2225 has revived the PDMP IG and the WG will be
authoring/republishing the request profile anyway, this is the
cheapest possible moment to do the right thing — drop the invented
element and mirror NCPDP. Disposition B's renamed/derived element
preserves legacy populates but introduces an invariant the WG would
have to maintain forever for an implementer base that is realistically
very small (the IG was withdrawn for years), so the
backward-compatibility argument is weak. Disposition C is technically
defensible given the prior vote, but it punts a clean, factual
correction into a larger redesign and risks the same mistake being
copied forward. Accepting as requested, scoped to the PSS-2225
revision, both fixes the standard-conformance issue and eliminates a
known source of confusion for NCPDP-aligned implementers before the
new IG ships.
