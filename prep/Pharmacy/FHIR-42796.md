## Summary

This Pharmacy ticket — also known as the "1-step vs. 2-step question" — asks
whether the US Prescription Drug Monitoring Program (PDMP) FHIR IG should
support submitting a request that contains *only* patient demographics (with
no dispense/medication parameters) so that a requester can first resolve which
patient(s) match before issuing a second, patient-specific query for dispense
records. The reporter raises this as an interoperability concern: real-world
PDMP implementations vary in whether they support a discrete patient-lookup
step versus always returning patient + dispense data in a single response.
The ticket has been triaged with a resolution of **Not Persuasive**, on the
basis that existing PDMPs do not implement a patient-only request, and that
the existing Bundle-based response design already accommodates either
behaviour (returning a patient pick-list when there are multiple matches, or
returning all matching patients with their dispense records). No
cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced
by the FHIR Augury index for this ticket.

## Details

**Description:**

Also known as the 1-step vs 2-step question.

Do PDMPs support querying for *only* patient information — for example, to
determine which patient dispense records to return in a second request?

**Resolution:**

Not implemented as such in existing PDMPs. PDMPs may return a patient list
where there are multiple matches, expecting the requester to re-request with
the specific patient identifier(s). Or, PDMPs may return multiple patients
and all associated dispense records for the requester to parse.

Bundle design permits either option to occur. But the PDMP Stakeholders did
not see any utility in the patient-only request.

**Comments:**

- **smrobertson (2023-09-27, PDMP call):** Discussed the 2-step query
  approach (query for patient, get multiple, then query for the specific
  patient). Pick-lists are used in TN and CA but not in other states. When
  a query matches multiple patients, options observed include responding
  with all records for all matching patients, or responding with "multiple
  matches, look at portal." Multiple-patient matches are fairly rare (OH
  reports <1%); integrations require "full names" plus additional
  parameters. The IG needs to support both a pick-list option and a "fail
  state" for multiple patient matches. When multiple matches are reported,
  consolidators try to deconsolidate them if they are profiles within the
  same state.
- **smrobertson (2023-09-25, Tech call):** Multiple patient matches will be
  common. Some states do probabilistic matching and return multiple
  patients. Some states allow patient selection but do not allow multiple
  patients in responses; if multiple patients are found, the requester is
  redirected to the portal.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Explicitly add a "patient-only" query mode to the US PDMP FHIR IG. Concretely:
define (or reuse) a Patient-search interaction on the PDMP server endpoint
that accepts only patient demographic search parameters (e.g., `family`,
`given`, `birthdate`, `gender`, `address-postalcode`, identifier) and
returns a Bundle of zero, one, or many `Patient` resources without any
`MedicationDispense` or other dispense-related entries. Add a normative
section to the IG describing the 2-step workflow (patient-only request →
client picks identifier → patient-scoped dispense request), define a
distinct OperationOutcome / response pattern for the "multiple matches"
fail state, and constrain the search parameters so a server can recognise
and honour a patient-only request.

#### Justification

A formally specified patient-only mode would give implementers a portable
way to perform interactive disambiguation across state PDMPs. The pick-list
behaviour already in use in TN and CA, and the "multiple matches → portal"
fallback noted on the 9/25/2023 tech call, both presuppose a discrete
patient-resolution step; standardising it would reduce per-state
integration variance and make probabilistic-match outcomes easier for
clients to handle uniformly.

---

### Disposition B: Alternative Approach

#### Proposal

Do not introduce a separate patient-only request, but document the
**two equivalent response shapes** that PDMPs may legitimately return for a
single dispense query:

1. A Bundle containing a single `Patient` plus that patient's
   `MedicationDispense` resources (the unambiguous-match case).
2. A Bundle containing multiple `Patient` resources and **no**
   `MedicationDispense` entries (the ambiguous-match case), accompanied by
   an `OperationOutcome` (or response code) signalling "multiple matches —
   re-query with a specific patient identifier."

Add IG guidance and conformance language requiring clients to handle both
shapes, and requiring servers to use option (2) when probabilistic matching
yields more than one candidate patient. Optionally constrain the multi-
patient response to omit `MedicationDispense` so the boundary between
"resolve patient" and "fetch dispenses" is unambiguous.

#### Justification

This preserves the single-request workflow that existing PDMPs already
implement while still giving clients a deterministic way to drive a 2-step
UX when needed. It directly leverages the existing Bundle design that the
resolution notes as already permitting either option, and avoids
introducing a new endpoint or interaction that no current PDMP supports.
It also aligns with the 9/27/2023 PDMP-call observation that multi-patient
responses are rare (<1% in OH) — making a dedicated patient-only endpoint
hard to justify on volume grounds while still ensuring the rare case is
handled consistently.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Close the ticket as **Not Persuasive** with the resolution text already
recorded on the ticket: existing PDMPs do not implement a patient-only
request; the existing Bundle design permits either a single patient +
dispenses response or a multi-patient (no-dispense) response, and the PDMP
stakeholders did not see utility in adding a dedicated patient-only
request. No spec changes.

#### Justification

This matches the position reached by the PDMP stakeholders during triage
and is consistent with the implementation reality reported on both the
9/25/2023 tech call and the 9/27/2023 PDMP call: no surveyed PDMP exposes
a discrete patient-only mode, multi-match cases are rare, and the
behaviours that *do* matter in practice (pick-list, "go to the portal")
are already representable in a Bundle response. Adding a new query mode
that no implementer would realistically deploy increases IG surface area
and conformance burden without addressing a demonstrated implementer or
clinical need.

---

### Recommendation

**Recommended disposition:** C

The recommendation is to confirm the existing **Not Persuasive** resolution.
The two committee discussions captured in the comments converge on the same
empirical point: real PDMPs do not implement (and stakeholders do not want)
a patient-only request, and the multi-patient case that motivates the
question is both rare (<1% in OH) and already handleable inside the
Bundle-based response shape. Disposition A would standardise a workflow no
PDMP currently offers, raising conformance cost for zero current
implementers. Disposition B is the strongest fallback — it would be worth
adopting *if* the workgroup decides the IG today is silent on the
multi-match response shape and that ambiguity is harming interoperability;
in that case B can be tracked as a separate clarification ticket rather
than a reopening of FHIR-42796. Absent that signal, declining (C) is the
defensible outcome and matches the disposition the workgroup already
recorded.
