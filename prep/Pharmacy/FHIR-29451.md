## Summary

The reporter argues that the PDMP (US Prescription Drug Monitoring Program)
FHIR Implementation Guide should not mandate that the patient's Social
Security Number (SSN) be returned in the PDMP response. The current
specification marks SSN with an asterisk (`*`) on the response table,
indicating it is a required element. The ticket proposes removing that
asterisk so that providing SSN becomes optional, leaving the choice to
each individual state PDMP. The ticket has already been resolved
**Persuasive**, with the work group agreeing that the response should not
require the patient's SSN; instead, the PDMP Patient profile will require
at least one identifier from a set (SSN, Driver's License, MRN, or other),
and the response table is to be updated to reflect that flexibility. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by FhirAugury for this ticket.

## Details

**Description:**

The standards should probably not mandate sending SSN. It should be up to

**Existing Wording:** SSN has an `*` on the response, meaning it is required.

**Proposed Wording:** Remove `*` next to SSN.

**Resolution Description:**

Agree response should not require patient's SSN. The PDMP Patient profile
requires at least one identifier (SSN, DL, MRN, other). This table will be

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the request literally: in the **US Prescription Drug Monitoring
Program (PDMP) FHIR Implementation Guide** (Pharmacy WG, "US Medication
FHIR IG" specification entry), edit the PDMP response page so that the
asterisk (`*`) next to the **SSN** entry in the response table is
removed, and adjust the surrounding legend/text so SSN is documented as
**optional** rather than required in the response. In the underlying
**PDMP Patient profile**, ensure that `Patient.identifier` slice for SSN
has `min = 0` and `mustSupport = true` (rather than `min = 1`), while a
profile-level constraint (or narrative) requires that **at least one**
identifier from the set {SSN, Driver's License, MRN, other} be present.
This is the same shape as the resolution description already recorded on
the ticket.

#### Justification

This matches the resolution recorded by the work group ("Persuasive") and
honours the privacy concern raised by the reporter: states differ in
whether SSN may legally be transmitted in a PDMP response, and FHIR
profiles should not force a jurisdiction to violate state policy in order
to be conformant. Requiring "at least one of SSN/DL/MRN/other" preserves
the patient-matching intent of the response without singling out SSN as
mandatory, which is consistent with how identifier slices are usually
constrained in US Realm IGs.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of only relaxing the SSN cardinality on the response table, model
the matching requirement explicitly with a profile invariant on the PDMP
Patient profile (e.g., `pdmp-pat-1: identifier.where(...).exists()`) that
requires **at least one** of the named identifier types (SSN system,
DL system per state, MRN, or "other" with a documented system) to be
present, and mark each individual identifier slice as `min = 0`,
`mustSupport = true`. Additionally, add explicit narrative on the
response page describing the privacy considerations for SSN and pointing
implementers at the state-by-state guidance, and tag SSN with a security
label / usage note rather than a required asterisk.

#### Justification

This addresses the same underlying need (do not force SSN) but encodes
the "at least one identifier" rule in a machine-checkable invariant
rather than relying on prose under a response table — which is more
robust for validators, IG publishers, and downstream profiling. It also
gives implementers a clearer signal about *why* SSN is being relaxed
(privacy / state law) instead of merely dropping an asterisk. The
trade-off versus Disposition A is a slightly larger edit (invariant +
narrative + table) and a minor risk that existing implementations
relying on the SSN-required reading need to update their validation
expectations.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep SSN marked as required (`*`) in the PDMP
response table, on the grounds that consistent, deterministic patient
matching across PDMP queries depends on a strong identifier and that the
US PDMP ecosystem has historically relied on SSN for that purpose.

#### Justification

This is the weakest of the three options for this ticket because the
work group has already accepted the change as Persuasive and because
state-level prohibitions on SSN transmission make a hard requirement
non-conformant in practice for some jurisdictions. It would be defensible
only if implementer feedback showed that no other identifier combination
yields acceptable match rates — but no such evidence is on the ticket,
in cross-references, or in the (empty) Zulip search. Including this
option for completeness rather than as a serious recommendation.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved **Persuasive** by the Pharmacy work
group, and the recorded resolution description is essentially Disposition
A: drop the SSN required marker on the response and require "at least one
of {SSN, DL, MRN, other}" on the PDMP Patient profile. Disposition B is
a stronger engineering answer (encoding the rule as an invariant rather
than as table semantics), and the work group may choose to fold its
invariant + narrative improvements into the same change — but the
minimum action that closes this ticket is the literal edit in
Disposition A. Disposition C contradicts the existing resolution and
the privacy / state-variability rationale and should not be adopted.
