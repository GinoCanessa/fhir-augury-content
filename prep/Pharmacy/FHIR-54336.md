## Summary

The reporter questions whether a single `DosageDetails` ever needs to
carry multiple different `Dosage.route` values across its per-step
`Dosage` elements. The argument is that `route` is typically a property
of the pharmaceutical product / order (e.g., oral, IV, subcutaneous) and
therefore constant across the dosing steps of one `MedicationRequest`.
On that basis, the ticket proposes moving `Dosage.route` **up** one
level, from the repeating `Dosage` (step) element to the containing
`DosageDetails` element (i.e., from `DosageDetails.dosage.route` to
`DosageDetails.route`), so it is expressed once for the whole
instruction set rather than repeated on every step. No other Jira
tickets are cross-referenced, although the same Zulip thread that
surfaces this ticket also queues the sibling ticket
[FHIR-54340](https://jira.hl7.org/browse/FHIR-54340) (rename
`DosageDetails` to `InstructionsForUse`) for the same Pharmacy backlog
call, indicating this change is being considered as part of a broader
tidy-up of the new `DosageDetails` structure in 6.0.0-ballot4.

## Details

**Description:**

> Are there usecases where multiple [Dosage.route](https://hl7.org/fhir/6.0.0-ballot4/dosage.html#Dosage) elements can differ from each other within the same DosageDetails element?
> Isn't the Dosage.route for each MedicationRequest dependent on the pharmaceutical product and thus the same?
>
> Suggestion to move Dosage.route to [DosageDetails](https://hl7.org/fhir/6.0.0-ballot4/dosage.html#DosageDetails).route.

**Comments:**

### costateixeira (Jose Costa Teixeira) — 2026-04-13

> We considered the case take sublingual vs swallow. I think there were
> other cases where it was clear that route was not always the same.

This comment directly contradicts the ticket's premise: at least one
well-known use case (buccal/sublingual administration followed by
swallowing, or staged administration across steps) exists where `route`
legitimately varies between sequential `Dosage` steps within a single
`DosageDetails`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the `route` element from the repeating `Dosage` (step) backbone to
the containing `DosageDetails` backbone in FHIR Core 6.0.0:

- Remove `DosageDetails.dosage.route` (currently `Dosage.route`,
  0..1, CodeableConcept, bound to `RouteOfAdministration`).
- Add `DosageDetails.route`, same type, same value-set binding,
  cardinality 0..1.
- Update downstream types that embed `DosageDetails` (most notably
  `MedicationRequest.dosageInstruction`,
  `MedicationDispense.dosageInstruction`,
  `MedicationAdministration` cross-refs, `MedicationStatement`, and any
  IG derivatives such as US Core / IPS profiles) to read `route` from
  the new location; publish a mapping note for 5.x → 6.0 implementers.
- Update invariants / constraints that reference `Dosage.route`; adjust
  ballot examples under `dosage-examples.*`, `medicationrequest-example-*`,
  etc., to move `route` one level up.
- Document the migration in the resource change note on `dosage.html`
  and the 6.0 change log.

#### Justification

For the common clinical case (a single oral tablet, a single IV
infusion, a single subcutaneous injection) `route` is in fact a property
of the product-plus-order and is uniform across dosage steps; under
6.0.0-ballot4 implementers have to repeat it on every `Dosage`, which
adds redundancy and invites divergent values that have no clinical
meaning. Moving it up aligns `route` with how `method`, `site`, and
`asNeeded` are conceptually scoped (per order, not per step in the
dominant case) and reduces the authoring burden for the majority of
prescriptions. The change is a simple element relocation inside a
resource already being actively reshaped for 6.0 ballot (FHIR-54340,
FHIR-54337, FHIR-54524, FHIR-54651 are all in flight in the same
backlog), so the disruption window is the one already being paid.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Dosage.route` where it is and instead add `DosageDetails.route`
as a new, optional default that applies to all contained `Dosage`
elements that do not themselves carry a `route`. Specifically:

- Add `DosageDetails.route` (0..1, CodeableConcept, same
  `RouteOfAdministration` binding).
- Retain `DosageDetails.dosage.route` (0..1).
- Add an invariant (e.g., `dos-route-1`) stating: if
  `DosageDetails.route` is absent, at most one distinct
  `Dosage.route` value should appear across the contained dosages
  unless clinically justified; if `DosageDetails.route` is present, a
  `Dosage.route` that differs from it represents a deliberate
  override for that step.
- Non-normatively, guide implementers to populate
  `DosageDetails.route` for the common single-route case and only
  populate `Dosage.route` when a step legitimately deviates (e.g.,
  sublingual-then-swallow).

#### Justification

Jose Costa Teixeira's comment explicitly records that the WG has
already considered the sublingual-vs-swallow case and that `route` is
not always constant across steps; other multi-route patterns (e.g.,
IV loading dose followed by oral maintenance expressed as one
instruction set) are equally plausible. A pure move (Disposition A)
would lose that expressiveness and force senders to either split a
single logical instruction set into multiple `MedicationRequest`
resources or to misuse other elements. A header + per-step override
pattern keeps backward-compatibility of intent with 6.0.0-ballot4,
satisfies the reporter's common-case ergonomics goal, and mirrors the
"summary field with per-step refinement" pattern already used
elsewhere in FHIR. It is more conservative during a ballot cycle where
`DosageDetails` is still in flux.

---

### Disposition C: Decline

#### Proposal

Decline the move and keep `Dosage.route` at the step level unchanged
in 6.0.0. Close as "Persuasive with Mod" only to the extent of
adding a non-normative note to `dosage.html` clarifying that for the
common case where `route` is uniform across steps, senders should
populate the same value on each `Dosage.route` (or leave absent where
implicit from `medication`/`medicationCodeableConcept`).

#### Justification

The WG has a documented counter-example (sublingual-vs-swallow; the
comment suggests further cases) showing `route` genuinely varies across
sequential dosage steps. Moving it up would either (a) lose that
expressiveness, or (b) require compensating extensions/overrides that
would leave the spec strictly more complex than today. `Dosage` is the
long-standing locus of route in FHIR R4/R5; moving it in 6.0 creates a
breaking structural change (element path, search-parameter, profile and
IG impact across US Core, IPS, MedicationRequest-derived IGs) for
marginal ergonomic benefit that can also be achieved through publisher
guidance. The ticket's framing is a question ("Are there usecases…?")
and the answer the WG has already arrived at is "yes", which is the
classical basis for declining a structural change.

---

### Recommendation

**Recommended disposition:** B

A straight move (A) is attractive for the common single-route order,
but the sublingual-vs-swallow case already noted on the ticket, plus
the realistic pattern of IV-to-oral transitions expressed in one
`DosageDetails`, are concrete expressiveness regressions the WG should
not pay for at this stage of the 6.0 ballot. A flat decline (C) ignores
the reporter's legitimate ergonomic complaint — redundant `route`
values on every step in the overwhelmingly common single-route case.
Disposition B gives the reporter the header they want (`DosageDetails.route`
as the place to state route once), preserves the minority but real
multi-step differing-route cases via the existing `Dosage.route`, and
fits the "tidy-up of `DosageDetails`" theme of the Pharmacy backlog
(FHIR-54337, FHIR-54340, FHIR-54524, FHIR-54651) without compounding the
breaking-change surface. It is the disposition most likely to generate
consensus on the call the Zulip thread was convened to prepare for.
