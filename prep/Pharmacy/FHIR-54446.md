## Summary

Eric Haas requests that `MedicationRequest.isRecordOfRequest` be marked
as a modifier element. His argument is that the element fundamentally
changes the interpretation of the resource — when set to `true`, the
instance is *not* a real order and "should not be used for medication
dispense | supply and administration" — which is the textbook definition
of modifier behavior, yet the current element is not flagged with
`isModifier`.

The ticket sits at the intersection of two earlier, opposing Pharmacy WG
decisions. [FHIR-49863](https://jira.hl7.org/browse/FHIR-49863) (Stephen
Chu, resolved *Persuasive with Modification* on 2025-02-24) renamed the
prior `MedicationRequest.reported` to `isRecordOfRequest` **and**
explicitly resolved to make it a modifier element.
[FHIR-53305](https://jira.hl7.org/browse/FHIR-53305) (Jose
Costa-Teixeira, opened 2025-11-01) takes the opposite position, citing
PR review comments on
[HL7/fhir#3896](https://github.com/HL7/fhir/pull/3896) and the Request
pattern (which treats Requests as not-actionable-by-default) to argue
that the `isModifier` flag should be *removed*. FHIR-54446 effectively
re-asserts FHIR-49863's intent against FHIR-53305's pushback.

The most recent comment on FHIR-54446 (Lloyd McKenzie, 2026-01-25) goes
further and questions whether the element should exist at all, arguing
the `actionable` `meta.tag` (or `Task` / message / operation) is the
correct mechanism for differentiating "for your information" copies of a
Request from "please fulfill" copies, and that using a `meta.tag`
preserves digital signatures when responsibility moves between systems.
Costa-Teixeira's comment simply links back to FHIR-53305.

## Details

**Description:**

> isRecordOfRequest fundamentally changes the interpretation of the
> element or resource (it's not a real order), but it's NOT marked as a
> modifier element.
>
> - From isRecordOfRequest definition: " *...is not request or order
>   itself and should not be considered as such, i.e. the information
>   should not be used for medication dispense | supply and
>   administration.*"
>
> Propose making it a modifier element.

**Comments:**

- **mckenzie (2026-01-25):** Request resources are *always* records of
  requests; for them to be authoritative, they need a tag. Questions the
  existence of the element entirely — why have an element doing the job
  the tag is intended for? Tags (rather than data elements) avoid
  breaking digital signatures when copying an order from one system to
  another and needing to flip "isActionable" while retaining signature
  validity.
- **costateixeira (2026-01-25):** Pointer to
  [FHIR-53305](https://jira.hl7.org/browse/FHIR-53305) (the
  counter-ticket arguing the element should *not* be a modifier).

### [FHIR-49863: MedicationRequest :: add MedicationRequest.recordOfRequest as new modifier element in base resource or modifier extension elementor as](https://jira.hl7.org/browse/FHIR-49863)

The originating ticket (Stephen Chu, 2025-02-23). Argued that in
continuity-of-care / transition-of-care use cases (IPS, the Australian
Patient Summary, the Australian Aged Care Clinical Document IG),
MedicationRequest is included to communicate that there is a *record* of
a request, not a live order — and that `MedicationRequest.reported` is
not semantically explicit enough, especially when paired with
`status = active` and `intent = order`. Asked Pharmacy WG to add a
modifier element (or modifier extension) to make the
record-vs-order distinction unambiguous. Resolved on 2025-02-24 as
*Persuasive with Modification*: rename `reported` → `isRecordOfRequest`,
mark it as a modifier element, and rewrite the short and description.
This is the resolution that FHIR-54446 wants enforced and FHIR-53305
wants reversed.

Reporter: Stephen Chu; Assignee: Jose Costa-Teixeira; Commenter: Jose
Costa-Teixeira (2025-11-03 — "We will wait for the issue creator to
confirm if we need to add something else (comment, isModifier..)").

## Related Zulip Discussions

### [IPS > Day 2 Madrid Connectathon](https://chat.fhir.org/#narrow/stream/IPS/topic/Day%202%20Madrid%20Connectathon/near/517338257)

This is the substantive community thread on the same question. Stephen
Chu summarised the 2025-02-24 Pharmacy WG resolution and asked IPS
whether they would pre-adopt the rename + `isModifier` flag (or use a
modifier extension) before IPS 2.0 publication. **Lloyd McKenzie pushed
back hard**: he argued that all FHIR instances are records, so
`isRecordOfRequest` is a poor name; that `reported` is about provenance
(source-of-truth vs. patient/caregiver hearsay), not about whether the
record is a directive to act; and that the proper way to differentiate
"please fulfill" from "FYI" copies is the `actionable` `meta.tag` (or
`Task`/message/operation), because `meta.tag` changes don't break
digital signatures when an order moves between pharmacies. Grahame
Grieve clarified that IPS is not trying to convey workflow — only
clinical information — and quoted the disposition wording. Stephen Chu
maintained that the `isRecordOfRequest` name carries the semantic
clarity that "reported" lacks. Peter Jordan and Dave deBronkart raised
the practical IPS angle (a large fraction of prescriptions are never
filled, so `MedicationStatement` is a better signal of actual use). **No
consensus was reached**; the disagreement is essentially the same one
that now divides FHIR-54446 and FHIR-53305.

Participants: Stephen Chu, Lloyd McKenzie, Grahame Grieve, Peter Jordan,
Dave deBronkart, Peter Williams, John D'Amore, Justin McReynolds,

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir`, edit
`source/medicationrequest/structuredefinition-MedicationRequest.xml`
(differential only) to set `isModifier` = `true` on the
`MedicationRequest.isRecordOfRequest` ElementDefinition, and add an
`isModifierReason` such as: *"When `isRecordOfRequest` is `true`, the
instance is a record of a previously authored MedicationRequest and must
not be acted upon as a request for dispense, supply, or administration;
ignoring this flag changes the safety semantics of the resource."*
Update the resource intro / narrative on
`source/medicationrequest/medicationrequest.html` to call out the
modifier status. Re-publish examples (e.g.,
`source/medicationrequest/medicationrequest0340.xml`) to confirm no
publisher errors. Close FHIR-53305 as superseded with a pointer to this
disposition. No changes to `<snapshot>` (regenerated).

#### Justification

This finishes the work agreed by the Pharmacy WG on 2025-02-24 in the
FHIR-49863 resolution, which explicitly committed to *"We (Pharmacy WG)
will make it a modifier element."* The element's normative description
("the information should not be used for medication dispense | supply
and administration") is the textbook condition for `isModifier = true`
under the FHIR conformance rules: a downstream system that ignores the
flag will misinterpret the instance in a clinically unsafe way. Stephen
Chu's argument in the IPS Zulip thread — that `reported` carries
provenance semantics (source-of-truth vs. hearsay) and is not
self-explanatory about actionability — also stands. The breaking-change
gotcha is real but bounded: `isRecordOfRequest` is a recently renamed
element, and downstream profiles (IPS, AU PS, AU Aged Care) are still in
flight, which is exactly the cheapest moment to flip the flag.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** flip `isModifier` on the core element. Instead, in
`HL7/fhir`:

1. Leave `MedicationRequest.isRecordOfRequest` as a non-modifier boolean
   that captures the authoring intent ("the populator considers this a
   record of a request, not the request itself").
2. Document in the resource narrative
   (`source/medicationrequest/medicationrequest.html`) that the
   *workflow* signal — whether a recipient is authorised to act — is
   carried by the existing `actionable` `meta.tag` (or by `Task` /
   message / operation), per Lloyd McKenzie's Zulip argument and the
   Request workflow pattern.
3. Publish a **modifier extension**
   (e.g. `http://hl7.org/fhir/StructureDefinition/medicationrequest-isRecordOfRequest`,
   under `HL7/fhir-extensions`) that downstream IGs needing
   modifier-strength semantics (IPS, AU PS, AU Aged Care) can apply on
   their profiles, satisfying FHIR-49863's safety intent without forcing
   modifier semantics on every implementer of core MedicationRequest.
4. Resolve FHIR-53305 in alignment by **not** marking the element
   `isModifier`.

#### Justification

This threads the disagreement between FHIR-54446 / FHIR-49863 (pro
modifier) and FHIR-53305 plus McKenzie's Zulip pushback (anti
modifier). It addresses the safety concern (record-of-request instances
must not be filled) where it is actually exchanged today — in patient
summaries — via a modifier extension that those IGs can mandate, while
preserving signature stability and avoiding a breaking modifier flip on
every existing core implementer. It also concedes McKenzie's structural
point that "actionability" is fundamentally a workflow/tag concern
rather than a content-element concern, without going as far as deleting
the element (which would re-open the rename battle). Trade-off vs.
Disposition A: profile authors must opt into the modifier extension,
which is more work and slightly less safe by default; but it avoids
imposing modifier semantics on workflow-only MedicationRequest exchanges
that have no record-vs-order ambiguity.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54446. Keep `MedicationRequest.isRecordOfRequest` as a
non-modifier boolean. In the resolution, point requesters to the
`actionable` `meta.tag` and the Request workflow pattern as the
already-existing mechanisms for differentiating "this record is a
directive to act" from "this record is informational," exactly as Lloyd
McKenzie laid out in the IPS Zulip thread. Resolution category:
*"already addressed by existing mechanism"* (the tag-based workflow
machinery), with the further note that flipping `isModifier` on a
recently renamed element would impose a breaking conformance change on
all consumers of core `MedicationRequest` for a problem that downstream
IGs have not yet shown the tag-based mechanism cannot solve. Pair with
either resolving FHIR-53305 in the same spirit, or — per McKenzie's
broader comment on FHIR-54446 — opening a follow-up to consider removing
`isRecordOfRequest` entirely in favour of the tag.

#### Justification

This is defensible because (a) FHIR already has a deliberately designed
mechanism (`actionable` `meta.tag`, plus `Task`/message/operation) to
separate authoritative-and-actionable Requests from informational ones,
and the Request pattern explicitly treats "not actionable by default" as
the baseline — this is exactly the
existing-mechanism-already-addresses-it case; (b) marking an existing
element `isModifier` is a non-trivial breaking change that downgrades
implementer trust in the safety promises of "minor" updates; (c) the
tag-based mechanism preserves digital signatures when responsibility
moves between systems, which the element-based mechanism does not (a
concrete clinical-safety advantage McKenzie called out and that was not
rebutted in the Zulip thread). The cost is that Stephen Chu's
self-explanatory-naming argument is left unanswered and the IPS / AU
Aged Care use cases must rely on the tag, which they currently do not.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because the workgroup is genuinely split
(FHIR-49863 vs. FHIR-53305, mirrored by Chu vs. McKenzie on Zulip), and
both sides have substantive arguments that should not be dismissed.
Disposition A satisfies the FHIR-49863 commitment but ignores both
McKenzie's signature-stability point and FHIR-53305's countervailing
position, and imposes a breaking modifier flip on every core
implementer for a use case (IPS-style record-of-request) that only
specific downstream IGs actually need. Disposition C is intellectually
clean but leaves Chu's IPS / AU PS / AU Aged Care safety concern
unaddressed and effectively overturns the 2025-02-24 Pharmacy WG
resolution without offering those IGs a concrete migration path.
Disposition B keeps core MedicationRequest stable for existing
implementers, gives the IGs that need modifier-strength semantics a
sanctioned modifier extension to apply, and aligns FHIR-54446 and
FHIR-53305 on a single, reviewable resolution. The workgroup should be
prepared to coordinate with FHIR-I (per McKenzie's note that workflow /
pattern changes belong with the workflow project) and with the IPS
editors before publication of IPS 2.0.
