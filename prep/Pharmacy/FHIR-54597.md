## Summary

LawrenceMcKnight requests that an explicit sequence number (sequence
identifier) be added back to `DosageDetail.step`. Today, multi-step
tapered/scheduled dosages are conveyed as an ordered list of `step`
entries, with no element on each step that records its position in the
sequence. The reporter argues this creates a strong, fragile dependency
on consumers preserving list order during processing — which is not
guaranteed for asynchronous pipelines, certain XSLT transforms, or any
serialisation/handling path that does not preserve element order.
Because the sequence carries clinical meaning (e.g., a tapering regimen),
silent re-ordering could change the regimen in clinically significant
ways and would be very hard to detect after the fact. The reporter
therefore requests an explicit sequence identifier on each step so that
out-of-order processing can be detected and recovered.

## Details

**Description:**

> Each step has relative time start/end, and there is instruction
> "When a DosageDetails contains multiple steps, the steps are
> sequential — the dosage course follows through the steps one at a
> time. Each step has to specify an end condition — either by
> explicitly linking the step to some external event, or by specifying
> a limit on the timing of the event using either Timing.repeat.bounds[x]
> or Timing.repeat.count on the Timing data type."
>
> The example 14.7.1.4 *Example 3: Tapered dose, with some interlaced
> dosages* also shows steps with `repeat.count` and
> `repeat.boundDuration`.
>
> However, this creates a strong dependency on the list itself to ensure
> step ordering. There is no other reference on each step item to
> indicate that the step item itself has an important conditionality on
> the prior steps in the list being completed. This seems risky and a
> potential safety issue because as processing occurs on the list, the
> list ordering is not always guaranteed to keep sequence (e.g., lists
> processed asynchronously or with XSLT that does not make sequence
> guarantees). Further, if/when this sort of error occurs it would be
> hard to trace or identify that the sequence was lost. So, while it
> may be possible for some implementations to take precautions and
> ensure sequencing in the list processing, the risk is high for an
> error to occur that results in dramatically different clinical
> meanings. I thus strongly would favor adding back an explicit
> sequence identifier on the step so that if the list was processed
> where sequence was not kept it would at minimum become apparent what
> happened.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new mandatory `sequence` element to the `DosageDetail.step`
backbone element (in the Dosage data type) typed as `positiveInt`,
with cardinality `1..1`. Add an invariant on `DosageDetail` requiring
that, when more than one `step` is present, the set of `step.sequence`
values forms a contiguous run starting at 1 with no duplicates and no
gaps (e.g., `step.all(sequence.exists()) and step.sequence.distinct().count() = step.count() and step.sequence.toInteger().sum() = (step.count() * (step.count() + 1)) / 2`,
or an equivalent FHIRPath formulation). Update the Dosage page
narrative — including the existing "steps are sequential" paragraph and
the tapered-dose example in 14.7.1.4 — to (a) state that processors
MUST honour `step.sequence` (not document order) when interpreting the
regimen, and (b) re-render the Example 3 (tapered dose) instance to
include explicit `sequence` values.

#### Justification

The reporter's safety argument is concrete: the sequence of steps
carries clinical meaning (e.g., a taper), and FHIR's wire formats and
many downstream pipelines do not guarantee that XML/JSON element order
survives intact. Making the order explicit on each step is the
standard FHIR pattern for cases where ordering is semantically
important (cf. `Bundle.entry` historically, `List.entry` ordering
debates, and the use of `sequence` on `Coverage.class`,
`Claim.diagnosis`, `Claim.procedure`, `ExplanationOfBenefit.diagnosis`,
etc.). A required `positiveInt` plus an invariant detects accidental
re-ordering at validation time rather than as a silent clinical error,
which directly addresses the reporter's "hard to trace" concern.

---

### Disposition B: Alternative Approach

#### Proposal

Add an **optional** `sequence : positiveInt` to `DosageDetail.step`
(cardinality `0..1`) instead of making it required, and add a
*should*-level (warning) invariant that recommends populating it when
more than one `step` is present. Strengthen the narrative on the
Dosage data type page to state that conformant processors MUST preserve
step ordering, and that when `sequence` is present it is authoritative
over document order. Optionally, also add a companion note that
`sequence` SHOULD start at 1 and be contiguous when supplied. Do not
re-spec existing examples to require the new element, but update
Example 3 (tapered dose) to demonstrate `sequence` as best practice.

#### Justification

This addresses the same safety concern but at a lower compatibility
cost. Making the element required is a breaking change for every
existing instance and every profile/IG that uses the Dosage data type
(notably US Core MedicationRequest, IPS, AU Base, IPA, and the various
medication-administration IGs). An optional element with strong
narrative guidance lets implementers who care about robust pipelines
opt in immediately, while not invalidating instances that rely on
document order — which is, today, the only conformant way to convey
multi-step dosages. It also keeps the door open for a future,
mature-level promotion to `1..1` once adoption is observed.

---

### Disposition C: Decline

#### Proposal

Decline the request. Retain the current model in which
`DosageDetail.step` is an ordered list whose semantics are conveyed by
position, and clarify the narrative on the Dosage page to state
unambiguously that processors MUST preserve element order through any
transformation, serialisation, or pipeline. Optionally add a
data-handling warning that processors which cannot preserve list
ordering MUST NOT process Dosage instances containing multiple
`step` entries.

#### Justification

FHIR already relies on document order for many ordered constructs
(e.g., `Questionnaire.item`, `List.entry` when `mode = working`,
`Bundle.entry` for transactions, the components of `Timing.event`).
Both XML and JSON preserve element/array order by spec, and conformant
FHIR tooling is required to round-trip that order. Adding a sequence
element to one specific backbone element creates an inconsistent
pattern across the spec and shifts compliance burden onto every
producer of Dosage instances to satisfy a new invariant, in order to
mitigate a class of bug (order-losing pipelines such as non-conformant
XSLT) that is arguably a defect in those pipelines rather than in the
data model.

---

### Recommendation

**Recommended disposition:** B

The safety concern Lawrence raises is real — multi-step regimens are
exactly the kind of clinical content where a silent re-ordering would
matter, and "the wire format preserves order" is cold comfort once an
instance has passed through an integration pipeline that does not.
However, making `sequence` required (Disposition A) is a breaking
change that would invalidate every existing Dosage instance and force
coordinated updates across US Core, IPS, IPA, AU Base, and the
medication-administration IGs in the same cycle. Disposition B
captures the safety upside — implementers who care can adopt
`sequence` immediately and validators can flag mismatches between
`sequence` and document order — without breaking the installed base,
and leaves a clean path to tighten cardinality in a later release once
adoption and field experience justify it. Pharmacy should weigh this
against any signal from MnM about consistency with other "ordered
list" patterns before finalising the conformance level of the new
element.
