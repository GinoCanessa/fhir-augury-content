## Summary

The reporter (Chris Moesel) asks the Pharmacy work group to clarify, in the
`MedicationStatement` resource, how a "no known medications" code in
`MedicationStatement.medication` interacts with other elements on the same
resource — notably `category`, `dosage`, `adherence`, and `reason`. The ticket
poses three concrete scenarios: (a) does specifying a `category` scope the
"no known medications" assertion to that category only; (b) does a populated
`dosage` qualify the negation (e.g., "no known oral medications"); and (c)
how should an `adherence` code of `not-taken` be interpreted alongside a
"no known medications" assertion? The reporter suggests two possible
directions — either (1) document interaction semantics for these elements,
or (2) treat the "no known medications" code as overriding/forbidding such
qualifiers — and notes the spec need not enumerate every combination but
should say *something* about how qualifiers interact with statement-style
medication codes. There are no other linked or cross-referenced Jira tickets,
Zulip threads, or GitHub items; the only cross-references found are to the
FHIR artifacts named above.

## Details

**Description:**

Clarify how the use of a "no known medications" code interacts with other
elements in the resource. Do those other elements *qualify* the "no known
medications" statement?

For example,

- Does "no known medications" with a category specified mean no known
  medications in the context of that specific category (but there may be
  medications in the context of other categories)?
- Does "no known medications" with specific "dosage" specified mean no known
  medications taken as specified by the dosage instructions (e.g., "no known
  oral medications", "no known daily medications", etc?
- How would one interpret "no known medications" with adherence code
  specified as "not-taken"? There are no known medications that are not
  being taken?

Or does the semantic or implied meaning of the "no known medications" code
potentially override any other qualifiers (in which case such qualifiers
should be disallowed)?

Perhaps the specification cannot (or should not) explore every possible
combination, but it would be beneficial to say *something* about how the
other elements interact (or not) with medication codes that specify a
situation rather than a medication.

**Comments:**

### chuscmi (2026-04-07)

`MedicationStatement.medication = 787481004 | No known medications` is
intended to be a global, positive assertion that, to the best knowledge of
the asserter / statement author, no information about a patient's
medication is known to the asserter / author.

As this is intended to be a global, positive assertion:

- **Q1** — Does "no known medications" with a category specified mean no
  known medications in the context of that specific category (but there
  may be medications in the context of other categories)?
  - *Comment* — it may represent a global statement about all categories
    (i.e. care settings: inpatient | outpatient | community | discharge).
    `MedicationStatement.category` has cardinality `0..*`, which does not
    preclude all category values being included in a statement. If any
    category values are included, it should be use-case or clinical-
    scenario specific. For example, in a hospital discharge summary,
    `MedicationStatement.category = "inpatient"` would limit the "no known
    medications" assertion to "no known inpatient medications at the time
    of discharge", and other interpretations should be inferred from
    context.
- **Q2** — Does "no known medications" with specific "dosage" specified
  mean no known medications taken as specified by the dosage instructions
  (e.g., "no known oral medications", "no known daily medications", etc.)?
  - *Comment* — In the context of a global, positive assertion of no
    known medication, clinical common practice is expected **not** to
    populate `MedicationStatement.dosage`, `.adherence`, or `.reason`.
    Even when the medication code expresses "no known class of
    medications" (e.g., "no known ACE Inhibitor", "no known
    anticoagulant") or "no known fluoxetine", clinical logic leads to
    the same outcome — those qualifier elements should not be populated.
- **Q3** — How would one interpret "no known medications" with adherence
  code specified as "not-taken"?
  - *Comment* — If no information is known about the patient's
    medication, it is unlikely that adherence information exists.

These questions raise a key requirement: clear implementation guidance on
the use of "no known medication" is required.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative-flavored implementation guidance to the `MedicationStatement`
resource page (and the element definitions of `medication`, `category`,
`dosage`, `adherence`, and `reason`) describing how a "statement-style"
medication code — such as SNOMED CT `787481004 | No known medications` and
the related "no known X class" / "no known X drug" codes — interacts with
the other elements on the resource. Concretely:

1. In the `MedicationStatement` resource's narrative (e.g., "Notes" or
   "Scope and Usage"), add a "No known medications and other negation
   codes" subsection that explicitly states: when `medication` carries a
   negation/assertion code, the statement is a global positive assertion
   about the *absence* of medication knowledge for the asserter, and
   `dosage`, `adherence`, and `reason` SHOULD NOT be populated.
2. In the same subsection, document that `category`, when present
   alongside a negation code, scopes the assertion to that care setting
   (e.g., "no known inpatient medications") and is the one qualifier
   whose use is permitted and meaningful.
3. Add element-level comments on `dosage`, `adherence`, and `reason`
   cross-referencing the new subsection so implementers see the guidance
   in the element definitions.
4. Optionally add an `invariant` (warning-level, not error-level) on
   `MedicationStatement` that fires when `medication` is a member of a
   "no known medication" value set and `dosage`, `adherence`, or `reason`
   are populated — leaving it as a warning preserves backward
   compatibility while signalling the intent.

#### Justification

This directly addresses Chris Moesel's request and aligns with the
disposition already taken by the work-group commenter (chuscmi), who
spelled out exactly the semantics described above. Documenting the
interaction in the resource narrative and on the affected elements gives
implementers a single source of truth and avoids the trap of expecting the
spec to enumerate every combination. Keeping the `dosage`/`adherence`/
`reason` rule as SHOULD NOT (and, optionally, a warning invariant) preserves
the resource's flexibility and avoids breaking any existing data while
clearly signalling the intended pattern.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than embedding interaction rules in `MedicationStatement` itself,
introduce a dedicated value set (e.g., `MedicationKnowledgeAbsence`) that
enumerates the codes that are "statement-style" assertions about absence
of medication knowledge ("no known medications", "no known ACE inhibitor",
"no known fluoxetine", etc.) and add a single, focused note on
`MedicationStatement.medication` that:

- Identifies this value set as the canonical list of negation-style codes,
- Makes `dosage`, `adherence`, and `reason` `0..0` (or constrained to
  `empty()`) via a profile in the Pharmacy module (e.g.,
  `MedicationStatement-no-known-medication`) that implementers can derive
  from when expressing a negation, and
- States that `category` MAY be used to scope the assertion to a care
  setting.

The base `MedicationStatement` resource itself stays untouched apart from
a short pointer to the profile and value set.

#### Justification

This keeps the base resource clean and pushes the negation semantics into
a profile + value-set pair, which is the FHIR-idiomatic way to express
"this kind of statement has different element rules". It also gives
implementers a machine-checkable artifact (the profile) rather than only
prose. The trade-off vs. Disposition A is that the rule only "bites" when
implementers explicitly use the profile — base-resource consumers still
need the prose guidance — and creating a new value set adds maintenance
burden on the Pharmacy work group.

---

### Disposition C: Decline

#### Proposal

Decline to add new spec text. Rationale: the existing element definitions
are intentionally permissive (`category 0..*`, `dosage 0..*`, `adherence
0..1`, `reason 0..*`), and the interaction semantics for the small set of
SNOMED-CT "no known X" codes are already a matter of clinical common
sense, not FHIR semantics. The work group can publish a Pharmacy-module
FAQ or a Zulip-pinned message instead of changing the specification.

#### Justification

The reporter himself acknowledges the spec "cannot (or should not)
explore every possible combination". One could argue that codifying
negation interaction rules in the base resource invites endless edge
cases (what about `effective[x]`? `informationSource`?
`statusReason`?) and that the cleaner answer is to leave the base
resource open and rely on implementer judgment. This disposition is
defensible only if the work group is satisfied that the question will
not recur and that downstream IGs can carry any guidance themselves.

---

### Recommendation

**Recommended disposition:** A

The work-group commenter (chuscmi) has already articulated the intended
semantics in detail and concluded that "clear implementation guidance
information on the use of 'no known medication' is required". That is
precisely Disposition A, and capturing it in the base
`MedicationStatement` page makes the guidance discoverable by every
implementer rather than only those who happen to find a profile or FAQ.
Disposition B is attractive on FHIR-modeling grounds but adds a new
profile + value-set the work group must maintain and ballot, and it does
not help implementers who consume the base resource directly. Disposition
C contradicts the work group's own commenter and leaves the original
ambiguity in place. Adopting A — with the optional warning-level
invariant — gives implementers actionable, normatively-located guidance
without introducing breaking changes.
