## Summary

The reporter proposes moving the `additionalInstruction` element from the
nested `Dosage` datatype up to the parent `DosageDetails` datatype in the
FHIR R6 (ballot 4) Dosage model. The underlying argument is that
`additionalInstruction` (free-text or coded patient-facing instructions
such as "take with food") is semantically a property of the prescription
or medication-administration order as a whole, and therefore the same
across every individual `Dosage` repetition that belongs to a single
`MedicationRequest` (or other DosageDetails-using resource). The reporter
asks whether there are any real use cases where two sibling `Dosage`
entries inside the same `DosageDetails` would carry *different*
`additionalInstruction` values; if not, the element belongs one level up.

No related Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-references or full-text search for this ticket key, so the request
must be evaluated on the basis of the reporter's description and the
published R6 ballot 4 Dosage / DosageDetails structure.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the `additionalInstruction` element from `Dosage` to `DosageDetails`
in the FHIR R6 Dosage model:

- Remove `Dosage.additionalInstruction` (currently `0..*
  CodeableReference(ObservationDefinition)` / patient-facing instruction
  text in the ballot 4 Dosage type).
- Add `DosageDetails.additionalInstruction` with the same cardinality
  (`0..*`), type, binding, short, definition, and comments as the current
  `Dosage.additionalInstruction`.
- Update DosageDetails comments/usage notes to clarify that
  `additionalInstruction` applies to the entire dosing regimen described
  by the contained `Dosage` repetitions.
- Update consumers of DosageDetails (notably MedicationRequest,
  MedicationDispense, MedicationStatement, MedicationAdministration,
  NutritionOrder, ActivityDefinition.dosage, etc. — wherever DosageDetails
  is used) so any examples / search parameters / extension definitions
  pointing at `Dosage.additionalInstruction` are repointed to
  `DosageDetails.additionalInstruction`.
- Add a backwards-compatibility note to the R6 changes / migration page
  describing the move from `Dosage.additionalInstruction` to
  `DosageDetails.additionalInstruction`.

#### Justification

The reporter's premise is consistent with how `additionalInstruction` is
typically used in real prescriptions: auxiliary labelling such as "take
with food", "do not drive", "shake well before use" is associated with
the order/regimen as a whole rather than with a single dose phase (e.g.,
the morning-vs-evening split, or the tapering-down repetitions of a
steroid course). Promoting the element to `DosageDetails` removes the
need to repeat identical text on every sibling `Dosage`, eliminates the
risk of contradictory values inside the same regimen, and aligns the
model with how dispensing/administration systems already render Sig +
auxiliary labels. Because R6 is still in ballot, this is the cheapest
window to make the structural correction.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Dosage.additionalInstruction` where it is **and** add
`DosageDetails.additionalInstruction` as a peer, with an invariant that
clarifies precedence:

- Add `DosageDetails.additionalInstruction` (`0..*`,
  `CodeableReference(ObservationDefinition)`) for instructions that apply
  to the regimen as a whole.
- Retain `Dosage.additionalInstruction` for instructions that genuinely
  vary between dose phases (e.g., a tapering course where the morning
  dose is "with food" but the evening dose is "at bedtime, with water").
- Add an invariant / usage note: an instruction that applies to all
  contained `Dosage` repetitions SHOULD be expressed once at
  `DosageDetails.additionalInstruction` rather than duplicated on each
  child `Dosage`.

#### Justification

This preserves expressive power for the (admittedly uncommon) case where
auxiliary instructions differ between dose phases — for example a
multi-phase chemotherapy regimen, a steroid taper with phase-specific
warnings, or paediatric formulations whose handling instructions change
between titration steps — while still giving implementers the cleaner,
non-duplicated representation for the common case. It also mitigates the
backwards-compatibility cost: existing R5/R6-early implementations
populating `Dosage.additionalInstruction` continue to validate, and
implementers can adopt the new top-level slot opportunistically. The
trade-off is a more complex model with two valid locations for the same
information, requiring profilers to write a "don't duplicate" guidance
note.

---

### Disposition C: Decline

#### Proposal

Decline the move. Keep `additionalInstruction` solely on `Dosage`, on
the basis that the element's per-Dosage placement is intentional and
that DosageDetails-level patient-facing instructions can already be
expressed via `DosageDetails.patientInstruction` (or the equivalent
top-level free-text instruction slot in DosageDetails) plus repetition
on each `Dosage` where coded auxiliary instructions are needed.

#### Justification

There are real, if minority, use cases where auxiliary instructions
differ between sibling `Dosage` entries — multi-phase regimens, taper
schedules, and PRN-vs-scheduled splits within a single order. Moving the
element forces those use cases either to lose information or to fall
back on extensions, which is a worse outcome than the current modest
duplication. If the concern is purely about avoiding repeated text in
the common case, that is better solved by tooling/UI and by guidance in
the Dosage page, not by relocating the element and breaking every
implementation that already populates `Dosage.additionalInstruction`.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the recommended starting point for Pharmacy WG
discussion. The reporter's observation is correct for the overwhelming
majority of prescriptions — auxiliary instructions like "take with food"
or "do not drive" really do belong to the order, not to an individual
dose phase — but Disposition A's clean relocation removes a genuinely
useful (if rarely exercised) capability for multi-phase regimens such as
steroid tapers, chemotherapy cycles, and split scheduled/PRN orders.
Adding `additionalInstruction` at the DosageDetails level while keeping
the per-Dosage slot, plus usage guidance that the regimen-level slot is
the preferred location when the instruction is uniform across all
contained Dosages, captures the reporter's intent without sacrificing
expressivity or breaking existing implementations as R6 settles. If the
WG is comfortable that the per-Dosage variation use cases are not real
in practice (i.e., can be handled by splitting into multiple
MedicationRequests), Disposition A becomes preferable on simplicity
grounds — that is the key question to put to the room.
