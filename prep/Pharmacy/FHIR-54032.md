## Summary

The reporter proposes changing the datatype of `DosageDetails.renderedInstruction`
from FHIR `markdown` to plain `string`. The argument is that the rendered
instruction is the human-safe, human-readable form of the dosing directions
intended to be shown directly to end users, and FHIR `markdown` is defined such
that rendering is *optional*: a conformant consumer may display the raw markdown
source verbatim or may render it as GitHub-Flavored CommonMark. That optionality
creates a clinical-safety concern, because identical bytes can convey different
meanings depending on whether the consumer chooses to render the markdown â€” most
notably HTML-style comments (`<!-- ... -->`), emphasis, and link syntax can
suppress, alter, or visually obscure clinically relevant tokens such as
"NEVER". A `string`-typed renderedInstruction would force the source to encode
the dosing message in a single unambiguous form that all consumers must display
identically.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were returned
by the FhirAugury index for this key, so the request stands on its own.

## Details

**Description:**

> **Suggestion:** DosageDetails.renderedInstruction should simply be a string
> datatype, rather than the [FHIR markdown](https://build.fhir.org/datatypes.html#markdown)
> datatype.
>
> Per [Dosage - FHIR v6.0.0-ballot3](https://build.fhir.org/dosage.html#text),
> it is important to display the renderedInstructions to the end user. Recall
> that FHIR markdown is defined to be a string that may *optionally* be rendered
> using the Github-Flavored Markdown extension of the CommonMark format.
>
> As such, the information contained in these instructions should be simple for
> all humans to read, regardless of interpretation or optional rendering of this
> markdown.
>
> As an *extreme* example, consider how differences in implementation may
> display the renderedInstructions differently to the end user:
>
> **As freetext:**
>
> YOU SHOULD &lt;!-- NEVER --&gt; TAKE MORE THAN ONE TABLET PER DAY
>
> **Render using Github-Flavored Markdown:**
>
> YOU SHOULD TAKE MORE THAN ONE TABLET PER DAY

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the datatype of `DosageDetails.renderedInstruction` from `markdown`
to `string` in the FHIR Core specification (R6 ballot 3 build, `dosage.html`
/ the DosageDetails structure definition). Update the element's short
description and definition to make explicit that the value is intended for
direct display to humans without any markdown rendering pass, and add a usage
note warning implementers that consumers MUST NOT interpret embedded markup
(including HTML comments, emphasis, or link syntax) as anything other than
literal characters. As a `string`, the element retains the existing 1 MiB size
constraint and gains predictable, render-agnostic display semantics. Mark this
as a breaking type change against R5 / earlier R6 ballots and call it out in
the substantive-changes appendix.

#### Justification

The reporter's safety argument is concrete: FHIR `markdown` is defined as
"may *optionally* be rendered" (datatypes.html#markdown), so two conformant
consumers can legitimately display the same `renderedInstruction` value
differently. For an element whose entire purpose is to be the canonical
human-readable rendering of the dose, that optionality is at odds with the
element's own contract on `dosage.html#text`. A `string` datatype removes the
ambiguity and aligns the element's type with its stated intent.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `DosageDetails.renderedInstruction` as `markdown`, but add a normative
invariant and definition tightening that (a) requires the value to be safely
displayable as plain text â€” i.e., the source SHALL author the content such
that the unrendered character sequence is itself the intended human-readable
message â€” and (b) recommends that consumers display the value as plain text
unless they have an explicit, verified rendering pipeline (HTML-comment
stripping, link/script sanitisation, etc.). Optionally introduce a sibling
`renderedInstructionFormat` code or a `renderedInstructionPlain` companion
element of type `string` for consumers that cannot safely render markdown.

#### Justification

This preserves backwards compatibility with R5 and earlier R6 ballots â€” no
type change in the datatype, no migration burden for existing
MedicationRequest / MedicationDispense / MedicationStatement instances and
profiles. It also keeps the door open for richer presentation (line breaks,
emphasis, lists) when the consumer is known to render markdown, which is a
genuine implementer ask in printed labels and patient-facing apps. The
trade-off vs. Disposition A is that the safety concern is mitigated by
specification language and invariants rather than by the type system; this is
weaker enforcement but avoids a breaking change.

---

### Disposition C: Decline

#### Proposal

Reject the change. Keep `DosageDetails.renderedInstruction` as `markdown` with
no spec-level changes. The risk described is treated as an implementation /
authoring concern: sources that produce dosage instructions for safety-critical
display are responsible for not embedding constructs (HTML comments, link
syntax) that can change meaning under rendering, and consumers are responsible
for using a sanitising renderer.

#### Justification

`markdown` is used widely across FHIR for human-readable text fields
(`Resource.text.div` aside, see `*.note.text`, `*.description`, etc.) and the
"may optionally be rendered" contract is well understood in the ecosystem. The
specific failure mode shown (an HTML comment hiding the word "NEVER") relies
on a consumer that renders GitHub-Flavored Markdown *and* strips HTML
comments â€” a behaviour that itself is a sanitiser bug, not a datatype bug.
Changing the type is a breaking change and risks driving authors toward
workarounds (jamming structured content into a single line of plain text) that
degrade usability without materially improving safety.

---

### Recommendation

**Recommended disposition:** A

The whole point of `renderedInstruction` is to be the display-ready string the
end user sees. As long as the type is `markdown`, the spec is explicitly
telling implementers "you may render this, or you may not", which is exactly
the wrong contract for a clinically-significant, display-oriented element.
Disposition B's invariant-based mitigation is real but weaker â€” it relies on
authors and consumers reading and obeying prose, where Disposition A makes the
safe behaviour the *only* behaviour permitted by the schema. Disposition C
under-weights the safety argument: even if the demonstrated exploit assumes a
flawed renderer, the broader truth is that the same byte sequence can produce
two different human-visible messages, which is unacceptable for a medication
dose. The cost is a breaking type change between R5/earlier ballots and R6,
which is significant but tractable in a ballot cycle and should be flagged in
the substantive changes list. Pharmacy should adopt A and ask FHIR-I for
review of the breaking-change implications.
