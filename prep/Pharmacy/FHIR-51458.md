## Summary

FHIR-51458 reports that the current SMP (US Standardized Medication Profile)
Bundle profile only allows a single medication list per Bundle. Because
implementers often need to exchange more than one medication list (for
example, a current-meds list alongside a discharge or reconciliation list),
forcing each list into its own Bundle is structurally limiting and adds
transactional overhead. The reporter asks the Pharmacy work group to
loosen the cardinality so multiple medication lists may appear in the same
SMP Bundle. No other Jira tickets were cross-referenced; the only
cross-referenced artifact is the GitHub pull request that implemented the
fix (`HL7/smp-ig#2`). The ticket has been marked **Persuasive** and the
change has already been applied to the CI build per the latest comments.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the cardinality of the medication-list slice inside the SMP Bundle
profile so that more than one List (or MedicationStatement-bearing
collection) entry may be present per Bundle. Concretely:

- In the SMP Bundle StructureDefinition, change the `Bundle.entry` slice
  that constrains the medication-list entry from `1..1` to `1..*` (or
  `0..*` if no list is genuinely required).
- Update any invariants, slicing discriminators, and narrative/IG prose
  that currently assume exactly one medication list per Bundle.
- Refresh example Bundle instances to demonstrate multi-list usage and
  add at least one example exhibiting two lists (e.g., active meds +
  reconciliation).
- Confirm downstream profiles/operations that reference the SMP Bundle
  tolerate the new cardinality.

This mirrors the change already landed via
[HL7/smp-ig#2](https://github.com/HL7/smp-ig/pull/2).

#### Justification

The reporter's concern — that one-list-per-Bundle forces multiple Bundles
for common pharmacy use cases (admission + reconciliation, current + prior,
etc.) — is a real implementation friction. Multi-cardinality on Bundle
entries is idiomatic FHIR (Bundles are inherently collections), and the
Pharmacy work group has already indicated agreement in the ticket comments
and resolved the item as **Persuasive**. The fix is already in the CI
build, so Disposition A simply formalizes what has occurred.

---

### Disposition B: Alternative Approach

#### Proposal

Retain the `1..1` medication-list slice on the SMP Bundle but introduce a
sibling slice (e.g., `medicationListAdditional`, `0..*`) typed to the same
List profile, distinguished by a `List.code` discriminator (e.g., current
vs. discharge vs. reconciliation). Define a small value set of
medication-list purposes to drive the slicing discriminator, and require
that when multiple lists are present, each carries a distinct
`List.code`. Provide guidance that a single "primary" list remains
mandatory for backwards-compatible consumers.

#### Justification

This preserves the existing producer/consumer contract (there is always
exactly one "primary" medication list at a well-known slice) while still
unblocking the multi-list use cases. It is less disruptive for
implementers who have already built to the STU1 shape, and it forces
producers to semantically tag each extra list, which reduces ambiguity on
the consumer side. The trade-off is extra profile complexity and a new
value set; Disposition A is simpler but gives consumers no hint about
*what* each list is for.

---

### Disposition C: Decline

#### Proposal

Decline the cardinality change and instruct implementers to continue
packaging each medication list in its own SMP Bundle, optionally grouped
via a containing MessageHeader/Composition or a batch/transaction Bundle
envelope at the exchange layer. Document this pattern explicitly in the
SMP IG.

#### Justification

Defensible only on a "one Bundle = one clinically-meaningful payload"
narrative and to avoid any risk of breaking STU1 consumers that assumed a
singleton list. However, given the work group has already agreed the
request is persuasive and the fix is merged, declining now would reverse
an already-accepted change and contradict the Resolution field. This
option is included for completeness.

---

### Recommendation

**Recommended disposition:** A

The ticket is already **Persuasive**, the change has been implemented in
[HL7/smp-ig#2](https://github.com/HL7/smp-ig/pull/2), merged to a
`preapply` branch, and subsequently landed in the CI build. The reporter's
use case (multiple medication lists per exchange) is well-motivated and
Bundle multi-cardinality is idiomatic FHIR. Disposition B's typed-sibling
slice is an attractive refinement, but it adds profile complexity and a
new value set for a problem that multi-cardinality already solves; the
Pharmacy WG can still layer an optional `List.code`-based slicing
convention on top of the accepted `0..*` / `1..*` change in a follow-up
ticket if implementer feedback shows ambiguity in practice. Disposition C
contradicts the already-recorded resolution and should not be pursued.
