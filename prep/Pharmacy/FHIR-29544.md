## Summary

Rick Geimer reports that the MedicationDispense profile in the US Prescription
Drug Monitoring Program (PDMP) FHIR Implementation Guide (filed under the
"US Medication FHIR Implementation Guide" specification in Jira) flags
`dosageInstruction` as must-support, but does not propagate any must-support
flags to its child elements, which are all optional. The reporter argues that
at minimum `dosageInstruction.text` — the SIG (the human-readable directions
for use of the dispensed medication) — should also be marked must-support, so
that a conformant PDMP responder is required to surface the SIG to consumers
of the dispense record. The ticket was originally dispositioned as
"Considered for Future Use" in January 2023 (the IG had been withdrawn at
that point), then reopened in November 2023 by smrobertson under the
PSS-2225 effort to revive PDMP on FHIR. There are no Jira, Zulip, or GitHub
cross-references on file for this ticket.

## Proposed Dispositions

> Note: No GitHub cross-references were surfaced for this ticket, so no
> repo-analysis briefings are loaded here. The dispositions therefore
> reference the PDMP IG repository at the conceptual level (the
> MedicationDispense profile and its `dosageInstruction.text` element). If
> the workgroup wishes to act on this ticket, the implementer should run
> `repo-analysis` on the active PDMP IG repo (e.g., the HL7 PDMP IG repo
> revived under PSS-2225) before writing the actual profile change, so the
> Justifications can be re-grounded in the briefing's authoring roots and
> warnings.

### Disposition A: Accept As Requested

#### Proposal

In the PDMP IG's MedicationDispense profile, change the must-support flag on
the child element `MedicationDispense.dosageInstruction.text` from `false` to
`true`, leaving the parent `dosageInstruction` (`0..1`, MS) unchanged. Mark
all other `Dosage` children as not-must-support, so the constraint is
narrowly scoped to the SIG string. Update the IG narrative accompanying the
profile to state explicitly that responders must populate
`dosageInstruction.text` whenever `dosageInstruction` is present, and that
consumers must be able to display it.

#### Justification

The reporter's premise — that flagging a complex element MS without flagging
any of its children leaves the actual data optional in practice — is the
core argument the FHIR profiling community uses for "MS on at least one
meaningful child." `dosageInstruction.text` is the canonical
human-readable SIG, is universally available in PDMP source data, and is
the field consumers of a dispense record most need. Constraining only
`.text` (rather than the full Dosage structure) keeps the burden minimal
for responders that don't have structured dosing data, which matches the
pragmatic profile of US PDMP submissions.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) marking `dosageInstruction.text` as
must-support, add a profile-level invariant on MedicationDispense requiring
that whenever `dosageInstruction` is present, **either** `dosageInstruction.text`
**or** a structured `dosageInstruction.timing` + `dosageInstruction.doseAndRate`
combination is populated. Mark `dosageInstruction.text`,
`dosageInstruction.timing`, and `dosageInstruction.doseAndRate` all as
must-support, with the invariant ensuring at least one branch is filled.

#### Justification

PDMP source systems vary widely: some pharmacies submit only a free-text
SIG, others submit structured dose/frequency. Forcing `.text` MS alone risks
dropping structured data on the floor when it is available, while forcing
both unconditionally penalises text-only sources. A "one-of" invariant
preserves the intent of the request (the SIG must be conveyable) without
losing structured directions when they exist, and aligns with how other US
Realm IGs (e.g., US Core) handle Dosage. Trade-off vs. A: more authoring
work and a new invariant to test, but better long-term fidelity.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the current revival cycle of the
PDMP IG, on the basis that (a) the ticket is part of the bulk
`PDMP-old-issues` backlog imported from a 2020 spreadsheet, (b) the IG was
formally withdrawn and the prior 2023-01-18 disposition explicitly deferred
this and similar items, and (c) the base FHIR `MedicationDispense` resource
already permits `dosageInstruction.text`, so any responder that needs to
convey the SIG can do so today without an MS flag.

#### Justification

The original 4-0-0 vote ("Considered for Future Use") chose deferral
precisely because the IG was inactive. Reopening the ticket under PSS-2225
re-queues it but does not, on its own, justify the change — especially when
the broader PDMP profile has not yet been re-baselined. Declining keeps the
backlog focused on changes that can be evaluated against the new
project's actual scope rather than a 2020-era profile.

---

### Recommendation

**Recommended disposition:** A

The change is small, narrowly scoped to a single element flag, addresses a
real and well-known profiling anti-pattern (MS parent with no MS children),
and directly supports the consumer-side use case the PDMP IG exists to
serve: conveying dispense information, including the SIG, between dispensers
and PDMP queriers. Disposition B is technically more elegant but introduces
a new invariant whose value depends on PDMP submission conformance data the
workgroup likely does not yet have for the revived IG; B can be revisited
once the new IG draft is further along. Disposition C is defensible only if
the workgroup intends to triage every `PDMP-old-issues` ticket the same way,
which would defeat the purpose of reopening them under PSS-2225. Pharmacy
should accept A now and flag B as a candidate refinement during the next
PDMP IG ballot cycle.
