## Summary

The ticket asserts that Value Set canonical URLs defined by the US Specialty
Rx FHIR IG should be anchored under `terminology.hl7.org` (THO) rather than
under `hl7.org/fhir`, and that a corresponding UTG ticket should be opened
to register each affected Value Set in THO. The reporter (Joan Harper,
relaying TSMG comment 7 imported by Ron G. Parker) notes the only Value
Sets that should remain anchored under `hl7.org/fhir` are those required
bindings to a `code` data type. The ticket originated from a TSMG review
of US Specialty Rx and was previously resolved as "Considered for Future
Use" pending finalization of the IG-terminology-anchoring policy, then
re-opened by Lynn Laakso for re-consideration after STU2 publication. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
discovered for this ticket in the indexed sources.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor every Value Set defined by the US Specialty Rx FHIR IG that is
**not** bound as `required` to a `code` data type so that its canonical
URL lives under `http://terminology.hl7.org/ValueSet/...` instead of
`http://hl7.org/fhir/us/specialty-rx/ValueSet/...`. Concretely:

1. Inventory every `ValueSet` resource in the US Specialty Rx IG and
   classify each as either (a) required-bound to a `code` element
   (stays in `hl7.org/fhir/...`) or (b) any other binding strength /
   data type (moves to THO).
2. For each (b) Value Set, file a UTG ticket requesting registration of
   the Value Set (and any IG-local CodeSystems it depends on) under
   `terminology.hl7.org`, then update the IG's `ValueSet.url` to the
   new THO canonical.
3. Update every `ElementDefinition.binding.valueSet` reference in
   profiles, extensions, and operation definitions in the IG to point
   at the new THO canonicals.
4. Add release-notes / migration guidance to the IG narrative noting
   the URL change for downstream implementers.

#### Justification

This is exactly what TSMG recommended (per the imported Comment 7) and
aligns with the long-standing terminology-stewardship pattern that
shared, reusable terminology lives in THO so it can be governed
independently of any one IG release cycle. Required bindings to `code`
elements are intentionally excluded because those code values are
structural to the FHIR core / IG and cannot be revised through THO
governance. Re-anchoring also reduces the risk of duplicate canonicals
appearing once another IG wants to reuse the same Value Set.

---

### Disposition B: Alternative Approach

#### Proposal

Do not relocate every non-required-binding Value Set unconditionally.
Instead, apply a triage rule: only Value Sets that are (i) reusable
outside US Specialty Rx, or (ii) drawn entirely from CodeSystems already
hosted in THO, are migrated to `terminology.hl7.org`. Value Sets that
are intrinsically scoped to US Specialty Rx workflows (e.g., enumerations
that only make sense inside a Specialty Rx prescription exchange) remain
under `hl7.org/fhir/us/specialty-rx/ValueSet/...`. Pair this with an
explicit "anchoring decision" table in the IG narrative documenting why
each Value Set lives where it does, and still file UTG tickets for the
ones that move.

#### Justification

The original TSMG guidance pre-dates the formal policy on IG-defined
terminology anchoring, which Lynn Laakso's 2023 comment explicitly
acknowledges is still pending. A blanket move can create churn for
implementers and adds governance overhead at THO for Value Sets that no
other IG will ever consume. A reusability-driven triage preserves the
spirit of the TSMG guidance (shared terminology in THO) without paying
the migration cost on truly IG-local enumerations, and it gives the
Pharmacy WG and TSMG something concrete to evaluate when the anchoring
policy is finalized.

---

### Disposition C: Decline

#### Proposal

Decline the change for the current STU cycle on the grounds that the
underlying HL7 policy on IG-defined terminology anchoring has not been
finalized. Document the rationale on the ticket, leave the Value Set
canonicals as currently published in US Specialty Rx STU2, and defer
any re-anchoring work until TSMG / FHIR-I publish the binding policy.

#### Justification

This is consistent with the previous workgroup vote (6-0-0 on
2023-01-30) to consider the request for future use, and with the
explicit reasoning recorded on that vote: "This change will be
considered when the policy related to anchoring of IG-defined
terminology is finalized." Re-anchoring Value Sets is a breaking change
for any implementer who has already bound to the current canonicals;
making that change before the governing policy is settled risks doing
it twice if the policy ultimately disagrees with the TSMG comment as
worded.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between TSMG's terminology-stewardship
intent and the practical reality that the IG-anchoring policy is still
in flight. Disposition C (decline) was already tried and was reverted by
Lynn Laakso post-STU2 publication, which signals the WG no longer wants
to sit on the request. Disposition A (accept as requested) commits the
WG to a blanket migration plus a UTG ticket per Value Set before the
governing policy is settled, which risks rework and unnecessary
implementer churn for Value Sets that are intrinsically Specialty-Rx-
specific. A reusability-driven triage (B) lets the Pharmacy WG make
visible progress now — moving the Value Sets that clearly belong in THO
and filing UTG tickets for them — while preserving optionality for the
IG-local Value Sets until TSMG finalizes the policy. The "anchoring
decision" table also gives reviewers and TSMG a concrete artifact to
react to, which should accelerate closure of this ticket family.
