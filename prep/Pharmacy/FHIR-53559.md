## Summary

The ticket reports a typo / narrative inconsistency on the
[SMP Client CapabilityStatement](https://build.fhir.org/ig/HL7/smp-ig/CapabilityStatement-smp-client.html)
page of the US Standardized Medication Profile (SMP) IG. The quoted
narrative describes "the expected capabilities of the SMP Server actor"
on a page that defines the **SMP Client** CapabilityStatement, and
references "SMP clients" in the same sentence — mixing the Server and
Client actors in the prose. It also references the US Core
CapabilityStatement as an expected pairing. The requested fix is to
correct the narrative so the client-actor page describes SMP Client
capabilities consistently (and conversely for the server-actor
CapabilityStatement narrative if the same error appears there).

No cross-referenced Jira tickets were surfaced for this issue; the only
external link is to the GitHub PR that implements the fix
(`HL7/smp-ig#10`).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the reported typo as valid and adopt the narrative fix already
merged in [`HL7/smp-ig#10`](https://github.com/HL7/smp-ig/pull/10) for
the SMP Client CapabilityStatement (and mirror-correct the SMP Server
CapabilityStatement narrative if the same Server/Client confusion
appears there). Specifically:

- On the `CapabilityStatement-smp-client.html` page, change "This
  resource defines the expected capabilities of the SMP Server actor"
  to "This resource defines the expected capabilities of the SMP Client
  actor" and ensure the remaining sentences consistently refer to
  **SMP Clients** (not Servers).
- Verify the analogous `CapabilityStatement-smp-server.html` narrative
  refers to the **SMP Server** actor throughout.
- Leave the reference to the US Core CapabilityStatement pairing in
  place — that cross-reference is correct and unrelated to the typo.

Once the PR is confirmed merged to `main`, close this ticket with
resolution **Persuasive** (already recorded).

#### Justification

The reported text is demonstrably inconsistent with the page it appears
on (the Client CapabilityStatement page describes "the SMP Server
actor"). Narrative typos on actor-scoped CapabilityStatement pages
directly confuse implementers reading the conformance narrative, so
correcting the actor wording is the minimum change needed to make the
IG read correctly. The author has already merged the fix in
`HL7/smp-ig#10`, which makes Accept-As-Requested the pragmatic closing
disposition for a pre-STU2-ballot correction.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the narrative correction (as in Disposition A) **and**
additionally generalize the narrative so it is less prone to
Server/Client copy-paste errors in future edits. Options:

- Replace the hand-written opening sentence in both CapabilityStatement
  narratives with an IG-publisher-driven actor phrase (e.g., using a
  templated fragment or macro that substitutes the actor name) so that
  Client/Server narratives cannot diverge from the actor they describe.
- Add a one-line regression note in the IG's internal authoring
  guidance (e.g., a `README`/`authoring.md` in the source repo, pending
  confirmation from `repo-analysis`) instructing authors to match the
  narrative actor to the CapabilityStatement `mode` when editing these
  pages.

#### Justification

This prevents recurrence: the same typo is the kind of error that
reappears when CapabilityStatement narratives are copied between actor
pages. The added authoring guidance is cheap, and a templated actor
phrase eliminates the class of defect at modest authoring cost. The
trade-off vs. Disposition A is scope — this is still a narrative-only
change but adds authoring-process burden that the WG may view as
unnecessary for a pre-ballot IG.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the narrative error is (a)
already addressed by a merged CI-build PR and (b) too minor to require
a tracked workgroup disposition.

#### Justification

Not defensible here. The ticket documents a real narrative defect on a
normative-conformance artifact (CapabilityStatement); closing without a
recorded disposition loses the ballot-relevant record of the fix. The
ticket is correctly left **open** until formally resolved against the
merged PR. Decline is included only for completeness.

---

### Recommendation

**Recommended disposition:** A

Disposition A matches the ticket's existing resolution
(`Persuasive`) and the merged PR `HL7/smp-ig#10`, which already
implements the narrative correction. The reported text is unambiguously
wrong for the page it appears on, so there is no substantive debate
about whether the change is warranted. Disposition B's extra authoring
hardening is worth considering as a follow-up item but should not hold
up closing a pre-STU2-ballot typo fix. Disposition C is not defensible
for a narrative defect on a conformance artifact. Before closing, the
workgroup should confirm (via `repo-analysis HL7/smp-ig` and a spot
check of the merged PR) that the Server CapabilityStatement narrative
is correct as well, so both actor pages are consistent.
