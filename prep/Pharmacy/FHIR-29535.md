## Summary

The ticket questions whether the chained search parameter
`authorizingPrescription.dispenseRequest.validityPeriod` (defined on
`MedicationDispense` to reach into the authorizing `MedicationRequest`'s
`dispenseRequest.validityPeriod`) is actually needed in the US Prescription
Drug Monitoring Program (PDMP) FHIR Implementation Guide. The reporter,
Scott M. Robertson, argues that if his related "PDMP date range for
dispenses" negative ballot comment is correct, then this search parameter
becomes unnecessary. The description also notes that Fig 5b Option 3 and
Fig 5c Option 5 of the IG describe non-SMART, "FHIR API only" query
patterns — i.e., the patterns in which a PDMP query would typically need
to filter dispenses by date. The ticket was originally dispositioned
"Considered for Future Use" on 2023-01-18 (4-0-0, John Hatem / Tim McNeil)
because the IG had been withdrawn, but was reopened on 2023-11-28 when
the PDMP project was revived under PSS-2225. The cross-reference index
returned no related Jira tickets, Zulip threads, or GitHub items for
this key, so the ticket should be evaluated alongside the companion
"PDMP date range for dispenses" ticket it depends on (referenced only in
prose, not via a Jira link).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the `authorizingPrescription.dispenseRequest.validityPeriod`
chained `SearchParameter` from the US PDMP FHIR Implementation Guide
(US Medication IG), conditional on the companion "PDMP date range for
dispenses" ballot comment being accepted. Specifically: drop the
`SearchParameter` resource that defines this chain on
`MedicationDispense`, remove its references from the IG narrative
(including the chained-search-parameters listing and any worked
examples in the Fig 5b Option 3 / Fig 5c Option 5 query patterns), and
update the conformance section so PDMP responders/requesters are not
expected to support it.

#### Justification

The reporter himself flags this parameter as derivative — it only earns
its place in the IG if PDMP queries are expected to filter dispenses by
the *prescription's* validity period rather than by an attribute of the
dispense itself (e.g., `MedicationDispense.whenHandedOver` or
`whenPrepared`). Chained search parameters are expensive for servers to
implement and tend to surprise implementers, and the FHIR design
guidance is to avoid chained parameters when an equivalent direct
parameter on the target resource would do. If the companion ticket
concludes that PDMP date filtering belongs on the dispense itself, this
chain becomes dead weight in the IG and should be removed cleanly
rather than left in place "just in case".

---

### Disposition B: Alternative Approach

#### Proposal

Keep the chained search parameter in the IG but **demote it from
required/SHALL to optional/MAY**, and add explicit narrative explaining
when a client would use it (i.e., to find dispenses tied to
prescriptions that were valid during a particular window) versus the
preferred dispense-date filter (e.g., `whenHandedOver`). Also add a
worked example in Fig 5b Option 3 / Fig 5c Option 5 showing the
"FHIR API only" (non-SMART) query that uses the dispense-date parameter
as the primary filter and the chained `validityPeriod` parameter as a
secondary filter. The `SearchParameter` resource itself is retained
unchanged.

#### Justification

PDMP queries have legitimate variants: jurisdictions that audit
*prescriptions* (and their dispenses) over a window will want to filter
by `dispenseRequest.validityPeriod`, while jurisdictions that audit
*dispense events* will filter by dispense-date. Removing the chained
parameter outright forecloses the first pattern. Demoting rather than
deleting preserves the option for implementers who need it, keeps
backwards compatibility with any pilot implementations from the
withdrawn 2020 IG that may already support it, and lets the workgroup
test demand during the next ballot cycle without paying the cost of
re-introducing a search parameter later.

---

### Disposition C: Decline

#### Proposal

Decline the change. Keep the `authorizingPrescription.dispenseRequest.validityPeriod`
chained search parameter in the IG as currently defined. Resolve as
"Not Persuasive" with a note that the parameter remains useful for the
PDMP "FHIR API only" query options, and that the companion "PDMP date
range for dispenses" ticket should be evaluated independently rather
than treated as a precondition for removing this parameter.

#### Justification

The ticket's own argument is conditional — it only proposes removal *if*
a separate negative comment is accepted. If the workgroup decides that
PDMP date-range filtering is best expressed via the prescription's
validity period (because a prescription can authorize multiple dispense
events and PDMP reporting is often anchored to the prescription
lifecycle), then this chained parameter is the right mechanism and
should stay. Declining also avoids churn on a parameter that has been
in the IG since 2020 and that any pilot implementations are likely
already coded against.

---

### Recommendation

**Recommended disposition:** B

The ticket is explicitly conditional on a sibling ballot comment whose
disposition is not yet settled, so a hard-delete (Disposition A) is
premature and a flat decline (Disposition C) ignores the reporter's
real concern that the IG may be over-specifying its query surface. The
demote-and-clarify path in Disposition B addresses the underlying
worry — that implementers will be forced to support an awkward chain
they may not need — without irrevocably removing a parameter that some
PDMP query patterns (especially the non-SMART "FHIR API only" options
in Fig 5b/5c) genuinely use. It also gives the workgroup a clean way to
re-evaluate the parameter against real implementer feedback during the
next PSS-2225 ballot cycle, at which point Disposition A becomes a
trivial follow-on if demand never materializes. Pair this disposition
with the resolution of the companion "PDMP date range for dispenses"
ticket so the two stay consistent.
