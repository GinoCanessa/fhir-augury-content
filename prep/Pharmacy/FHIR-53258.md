## Summary

This ticket asks the Pharmacy work group to add a new status code, `generated`, to the MedicationRequest status code system (which Pharmacy maintains separately from the shared Request status value set). The proposal originates from a Workflow channel discussion (linked below) and the parent ticket [FHIR-53252](https://jira.hl7.org/browse/FHIR-53252), where it was observed that decision-support systems and computable artifacts often emit Request-pattern instances (ImmunizationRecommendation, ServiceRequest, MedicationRequest, etc.) that have no real lifecycle — they are never actioned, never updated, and the existing values (`draft`, `active`, `completed`, `on-hold`, `cancelled`, `entered-in-error`) all misrepresent that situation. The reporter (also a participant in the Zulip thread) confirmed in the ticket comments that "when systems propose requests, no other status makes sense", and the original proposal pairs `status=generated` with `intent=proposal`, enforced by an invariant.

Important context: while Pharmacy's recorded resolution on this ticket is **Persuasive with Modification** to add `generated` plus the proposal-only invariant, the upstream FHIR-I ticket [FHIR-53252](https://jira.hl7.org/browse/FHIR-53252) was resolved differently — FHIR-I decided to introduce `activated` (with `active` and `completed` becoming specializations of it) rather than `generated`. The two resolutions are not aligned, and Pharmacy needs to decide whether to (a) follow Pharmacy's own original `generated` resolution, (b) re-align with FHIR-I's `activated` outcome, or (c) close as already-addressed by the upstream change to the shared Request Status code system.

After Patient Care punted ownership to either FHIR-I or OO, FHIR-I assumed ownership of the code system and ultimately resolved the request as **Persuasive with Modification** — but with a different code name and semantics than `generated`: a new status `activated` is being added, and `active` / `completed` are being redefined as specializations of `activated`. This directly affects FHIR-53258 because Pharmacy's MedicationRequest status code system was called out in the originating Zulip thread as a separate code system that should "consider incorporating the code there too" — and the relevant upstream code is now `activated`, not `generated`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new code `generated` to the **MedicationRequest status** code system (the Pharmacy-owned code system, not the shared Request status code system), with the definition: "The request was generated by an automated system (e.g., a decision support engine or order-set expansion) as a proposal intended to be taken up by a requester who would prepare a proper MedicationRequest. The status will not be updated as the request progresses." Add the code to the bound `MedicationRequest.status` value set. Add an invariant on MedicationRequest (e.g., `mrq-gen-1`) requiring that when `status = 'generated'`, `intent` SHALL be `'proposal'`. Update the MedicationRequest narrative/state diagram to show `generated` as a terminal status reachable only when `intent = proposal`.

#### Justification

This is the resolution Pharmacy already recorded on the ticket ("Persuasive with Modification") and is exactly what the reporter and the Workflow Zulip thread converged on for the MedicationRequest case. It honours Pharmacy's autonomy over its own status code system, gives implementers an unambiguous code for decision-support output, and the proposal-only invariant prevents misuse on actionable requests. The Zulip thread shows no opposition to the underlying need on the Pharmacy side, and the reporter explicitly worked through the intent/immutability question with Lloyd before filing the ticket.

---

### Disposition B: Alternative Approach — Adopt `activated` to Stay Aligned with FHIR-I

#### Proposal

Instead of introducing `generated` to the MedicationRequest status code system, mirror the upstream FHIR-I resolution of [FHIR-53252](https://jira.hl7.org/browse/FHIR-53252): add `activated` to MedicationRequest.status and redefine `active` and `completed` as specializations of `activated`. Adjust the MedicationRequest narrative/state diagram accordingly. If Pharmacy still wants to flag decision-support output specifically, do so via `intent = proposal` alone (already supported), or via a `usage` extension, rather than a new status code.

#### Justification

The MedicationRequest status code system is conceptually a Pharmacy-flavoured restatement of the Request pattern's status codes, and divergence between it and the shared Request status code system is a long-standing source of implementer confusion. FHIR-I — the new owner of the Request status code system — explicitly decided that `generated` is not the right name/semantics and chose `activated` instead. Adopting a code (`generated`) that the upstream pattern owner has already rejected creates exactly the kind of cross-resource inconsistency that the Workflow group has spent years trying to reduce, and would force Pharmacy to either re-do this work or carry a permanent divergence. The "decision support proposal" use case is already adequately conveyed by `intent = proposal`; the new shared `activated` code covers the lifecycle-unknown case that motivated FHIR-53252.

---

### Disposition C: Decline — Already Addressed Upstream

#### Proposal

Resolve as **Not Persuasive** (or **Already Addressed**) with the rationale that the underlying need — a status for Request-pattern instances that will never see a lifecycle update — has been addressed at the Request-pattern level by FHIR-53252's introduction of `activated`. Implementers needing to mark a MedicationRequest as machine-generated can already use `intent = proposal` (and, once FHIR-53252 lands, the upstream `activated` code if the work moves to align MedicationRequest.status with the shared Request status code system on a future cycle). No change to MedicationRequest in this cycle.

#### Justification

Pharmacy's status code system intentionally tracks the Request pattern's status semantics. Adding `generated` here, after FHIR-I has chosen `activated` for the same conceptual gap, would lock in a divergence that Pharmacy will eventually want to undo. Declining now preserves optionality: Pharmacy can adopt the upstream `activated` code in a coordinated cross-pattern update rather than introducing a one-off MedicationRequest-only code that would need to be deprecated later. The Zulip thread itself notes that the original `generated` proposal predated the FHIR-I ownership and naming decisions, so the resolution recorded on this ticket reflects a moment that has since been overtaken.

---

### Recommendation

**Recommended disposition:** B

The committee should adopt the alternative approach. Pharmacy already recorded "Persuasive with Modification" toward `generated`, but that resolution was written before FHIR-I — the new owner of the Request status code system — settled on `activated` (with `active`/`completed` as specializations) in [FHIR-53252](https://jira.hl7.org/browse/FHIR-53252). The originating Zulip thread explicitly framed the change to MedicationRequest as "consider incorporating the code there too", i.e., as a follow-on to the shared code system, not as a standalone Pharmacy-only invention. Shipping `generated` in MedicationRequest.status while the rest of the Request pattern uses `activated` would re-create the cross-resource status drift that the Workflow group is actively trying to eliminate, and would almost certainly need to be deprecated in a later cycle. Disposition B keeps the implementer story consistent across MedicationRequest, ServiceRequest, and the rest of the Request pattern, and still gives Pharmacy a clean way to express the decision-support case (`intent = proposal`, plus the new `activated` status when the lifecycle is unknown). If the workgroup believes there is a Pharmacy-specific semantic that `activated` plus `intent = proposal` does not capture, that case should be made explicitly back to FHIR-I before introducing a divergent code.
