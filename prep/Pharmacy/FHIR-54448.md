## Summary

Eric Haas proposes a small editorial change to the definition text of
`effectiveTiming[x]` on the medication resources. The current wording
references "multiple `dosageInstruction` lines" (e.g., for tapering doses)
as the scenario in which `effectiveTiming[x]` represents the earliest start
date and the latest end date. Because R6 work has narrowed
`dosageInstruction` to a single occurrence (0..1) on the affected
medication resources, the "multiple lines" framing is no longer accurate.
The ticket asks the definition to instead say "multiple steps in
`dosageInstruction`" — i.e., the multi-step semantics now live inside the
single `Dosage` (via its repeating sub-structure / sequencing) rather than
across multiple sibling `dosageInstruction` elements. There are no
explicit Jira links, no GitHub cross-references, and no comments on the
ticket itself, but the change is consistent with the broader Dosage
restructuring being discussed in the Pharmacy WG (see Zulip "Medication >
Dosage").

## Details

**Description:**

> Propose changing the `effectiveTiming[x]:`
>
> from: *"...Where there are **multiple dosageInstruction lines** (for
> example, tapering doses), this is the earliest date and the latest end
> date...."*
>
> to: *"...Where there are **multiple steps in dosageInstruction** (for
> example, tapering doses), this is the earliest date and the latest end
> date...."*
>
> Reason: `dosageInstruction` is now 0..1 (single), so the "multiple
> lines" scenario is handled differently in R6.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the editorial change verbatim in the `effectiveTiming[x]`
element definition on every medication resource that currently carries
the "multiple dosageInstruction lines" phrasing — at minimum
`MedicationRequest.effectiveTiming[x]`,
`MedicationDispense.effectiveTiming[x]`,
`MedicationAdministration.effectiveTiming[x]`, and
`MedicationStatement.effectiveTiming[x]`. Replace the substring
"multiple dosageInstruction lines" with "multiple steps in
`dosageInstruction`". Leave the rest of the sentence (semantics of
"earliest date and latest end date") and the parenthetical example
("for example, tapering doses") unchanged. No structural / cardinality
changes; definition text only.

#### Justification

The change is purely editorial and corrects a now-incorrect description
that survived from when `dosageInstruction` was 0..*. With
`dosageInstruction` constrained to 0..1 in R6 on these resources, the
"multiple lines" wording is misleading: the only way to express
multi-step regimens (e.g., tapering) is via the repeating internal
structure of a single `Dosage` (the `Dosage` data type itself supports
sequencing/multiple instructions). The replacement phrase aligns the
definition with the actual structural reality and matches how the
Zulip "Medication > Dosage" thread is talking about the design. The
risk surface is essentially zero — no schema, profile, code system,
example, or instance becomes invalid as a result.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the wording fix in spirit, but rewrite the `effectiveTiming[x]`
definition more comprehensively in the same edit so that the WG does
not have to revisit it during the larger Dosage restructuring discussed
in [Medication > Dosage](https://chat.fhir.org/#narrow/stream/Medication/topic/Dosage/near/553957808).
Concretely: (a) make the definition explicitly say
"`effectiveTiming[x]` represents the overall envelope (earliest start,
latest end) of the regimen described by the single `dosageInstruction`,
including any internal sequencing/tapering steps"; (b) add a forward
reference to the new Dosage page section that Grahame proposed
covering the relationship between `renderedDosageInstruction`,
`effectiveTiming[x]`, and `dosageInstruction`; (c) apply this uniformly
across MedicationRequest, MedicationDispense, MedicationAdministration,
and MedicationStatement.

#### Justification

The proposed two-word edit fixes the immediate inaccuracy but leaves
the definition just as terse and just as easy to misread as it has
been since 2017 (see Lloyd's old implementers thread). The Pharmacy WG
is already actively reorganizing Dosage documentation; bundling a
clearer definition into this ticket avoids a second editorial pass
and gives implementers a single, self-contained explanation of what
`effectiveTiming[x]` means in the new single-`dosageInstruction`
world. Trade-off: slightly larger diff and a small risk of pre-empting
decisions still in flight on the larger restructuring (e.g., if
`dosageInstruction` is later renamed to `dosageDetails` per Grahame's
Nov 10 proposal, the new wording would need another touch).

---

### Disposition C: Decline

#### Proposal

Decline as premature. Hold the change until the broader Dosage
restructuring tracked under
[FHIR-51540](https://jira.hl7.org/browse/FHIR-51540) and the
"Medication > Dosage" Zulip thread is resolved, since that work may
rename or restructure `dosageInstruction` (e.g., into
`dosageDetails`), at which point `effectiveTiming[x]`'s definition
would need to be rewritten anyway. Mark FHIR-54448 as a duplicate /
dependent of that larger ticket.

#### Justification

If the WG adopts Grahame's Nov 10 proposal (collapsing
`dosageInstruction`/`renderedDoseInstructions` into a `dosageDetails`
type), the wording proposed by this ticket becomes obsolete almost
immediately. Making the change now risks two churn events on the same
sentence within a single ballot cycle. Declining is defensible
strictly on "wait for the bigger decision" grounds — though it does
leave R6 with demonstrably incorrect text in the interim, which is the
weakness of this position.

---

### Recommendation

**Recommended disposition:** A

The ticket is a low-risk, surgical correction of factually wrong text:
the current definition references a cardinality (`dosageInstruction`
0..*) that no longer exists in R6, and any reader will be misled.
Disposition A makes the definition correct today with negligible
risk and no dependency on the unresolved Dosage restructuring.
Disposition B is attractive but conflates an editorial cleanup with
design choices that the Pharmacy WG has not yet settled, and would
likely stall this ticket behind FHIR-51540. Disposition C leaves
known-wrong text in place for an indeterminate time, which is harder
to defend to implementers than re-editing the same sentence later if
the larger restructuring lands. Accept as requested, and revisit the
wording (along with all `effectiveTiming[x]` / Dosage prose) if and
when the larger restructuring is approved.
