## Summary

Eric Haas is requesting an editorial change to `MedicationDispense` to clarify the relationship between two semantically overlapping elements: `basedOn` (the workflow-pattern element shared by all FHIR Event resources) and `authorizingPrescription` (the Pharmacy-specific element that references `MedicationRequest`). Both elements model "what this dispense fulfills," but the existence of two parallel mechanisms for the same concept breaks consistency with the rest of the FHIR Event Resource Pattern, where `basedOn` is the universal way to point at the authorizing request. The proposed remedy is narrow: add a comment to `MedicationDispense.basedOn` directing implementers to use `authorizingPrescription` when the authorizing request is a `MedicationRequest`. No cross-referenced Jira tickets were surfaced, but the request directly echoes a long-running implementer discussion (see Related Zulip Discussions) about why `MedicationDispense` deviates from the workflow pattern.

## Details

**Description:**

> `authorizingPrescription` and `basedOn` are Redundant Concept. Both represent "what this dispense fulfills" but use different element names and reference different resource types. Since Other FHIR event resources are typically use `basedOn` for ALL authorizing requests. MedicationDispense has a special-case element just for MedicationRequest, breaking the pattern.
>
> Propose making a comment in basedOn: "For Order/Prescription that authorizes the dispense use `authorizingPrescription`"

**Comments:**

- **mckenzie (2026-03-09):** Suggests the clarifying comment should be bidirectional — present on **both** `basedOn` and `authorizingPrescription` and explain what belongs in each. Also raises two structural points: (1) if the elements remain separate, `authorizingPrescription` should be tightened to `0..1` because a single dispense cannot be authorized by multiple prescriptions, and (2) the two elements should be positioned adjacent to each other in the resource definition so they are visible together when read in element order.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an editorial-only comment to `MedicationDispense.basedOn` in FHIR Core stating, in substance:

> "For an Order/Prescription that authorizes this dispense, use `MedicationDispense.authorizingPrescription` instead of `basedOn`. Use `basedOn` only for non-prescription requests (e.g., a `CarePlan`, `ServiceRequest`, or other authorizing Request) that the dispense fulfills."

Per Lloyd McKenzie's comment, also add a reciprocal comment on `MedicationDispense.authorizingPrescription` pointing back to `basedOn` for non-prescription authorizing requests, so the two elements document each other. No changes to types, cardinality, bindings, or invariants — purely a documentation/`comment` change to the element definition(s).

#### Justification

This is the minimum-impact change that closes the documented implementer confusion seen in the `implementers > Medication workflow and reference to MedicationRequest` Zulip thread, where multiple newcomers had to be told verbally that `authorizingPrescription` is the Pharmacy-specific name for the workflow-pattern `basedOn` slot. It is editorial-only, so it does not break any existing instances, profiles, IGs, or implementations; it is non-substantive for normative-comment purposes. It also aligns with FHIR's general convention of using `comment` on overlapping elements to disambiguate — the same pattern used elsewhere in the spec when a resource carries both a generic Event-pattern element and a domain-specific specialization.

---

### Disposition B: Alternative Approach

#### Proposal

Accept Disposition A's bidirectional comments **and** make the two structural adjustments raised by Lloyd McKenzie:

1. Tighten `MedicationDispense.authorizingPrescription` cardinality from `0..*` to `0..1`, on the grounds that a single dispense cannot logically be authorized by more than one prescription.
2. Reorder the resource definition so that `basedOn` and `authorizingPrescription` appear adjacent to each other (and adjacent to `partOf`), so the two "what this dispense relates back to" elements are visible together when reading the resource in element order.

The `basedOn` cardinality stays as it is (it remains `0..*` to allow other request types).

#### Justification

This addresses the underlying confusion *and* a latent modeling defect. A `0..*` `authorizingPrescription` is, in practice, almost never used with more than one entry — Pharmacy systems do not split a single dispense across multiple prescriptions; that would be modeled as multiple dispenses. Tightening the cardinality matches real-world workflow, prevents implementations from inventing meaning for the multi-valued case, and harmonizes with `MedicationDispense` already being a one-event-per-resource pattern. Co-locating the two elements in the resource definition means the `comment` text actually has a chance of being read alongside the element it refers to. The trade-off is that tightening cardinality is a **breaking change** for any implementation that has authored multi-valued `authorizingPrescription` — Pharmacy WG should confirm there are no known implementations doing so before adopting; if any exist, fall back to Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the duplication is intentional and well-established Pharmacy WG modeling, and that adding cross-referencing comments will encode the *exception* to the Event Pattern as if it were a problem to be managed, rather than a deliberate domain convention. The current spec is already correct; implementers needing to link to non-`MedicationRequest` authorizing resources can do so via `basedOn` (or via extension, as Lloyd McKenzie advised in the Zulip thread).

#### Justification

The Pharmacy WG made an explicit, documented choice (visible in the Zulip thread) to use the domain-intuitive name `authorizingPrescription` rather than the generic `basedOn`, on the principle that "common names are helpful for those systems that deal with multiple resources, but those are, by definition, more sophisticated than those systems that are pharmacy-specific." Adding a comment that effectively reads "don't use the obvious element, use the other one instead" risks signaling that the design is broken when the WG considers it correct. Implementers who hit the question already get the answer from the existing element documentation and from community channels; a spec comment is unlikely to materially change behavior of implementers who are not already reading the element definitions carefully.

---

### Recommendation

**Recommended disposition:** B

Disposition A clears up the documented confusion at zero risk and should be adopted at minimum. However, Lloyd McKenzie's comment on the ticket — from a senior, active reviewer — flags two adjacent issues (the unjustified `0..*` cardinality on `authorizingPrescription` and the awkward element ordering) that are easy to fix in the same touch and that the Pharmacy WG would otherwise have to revisit in a separate ticket. Rolling them in now is more efficient and produces a cleaner end state. The cardinality tightening is the only part of Disposition B that carries any real risk; the WG should poll members for known implementations relying on `authorizingPrescription` cardinality > 1 before committing. If any are found, fall back to Disposition A. Disposition C is not recommended — the Zulip thread is direct evidence that the current text is **not** self-explanatory to implementers, and editorial clarification is exactly the lightweight tool the spec has for this situation.
