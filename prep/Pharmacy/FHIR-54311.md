## Summary

Stephen Chu (Pharmacy WG) requests that the `MedicationAdministration`
resource be extended to better support multi-cycle chemotherapy protocols,
where a full course of treatment consists of several repeated cycles (typically
4–6) and where clinicians must progressively track the cumulative dose a
patient has received against a clinically safe lifetime/total-dose limit.
The ticket specifically proposes adding two new data elements on
`MedicationAdministration`: `cycle` (to identify which cycle in the protocol
this administration belongs to) and `cumulatedDose` (to record the running
total of drug delivered to date). The use case is illustrated with the
Hodgkin lymphoma BACOPDac escalated-dose regimen, where bleomycin carries a
400,000 IU total-dose toxicity ceiling that must not be exceeded across the
entire 6-cycle course. No related Jira tickets were surfaced by
cross-reference search; the only cross-references are to the Pharmacy WG
Confluence agenda page where this item was discussed and to the two proposed
element paths themselves.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add two new elements directly to the base `MedicationAdministration`
resource in FHIR Core:

- `MedicationAdministration.cycle` — a structured element identifying which
  cycle of a multi-cycle protocol this administration belongs to. Likely
  shape: a BackboneElement with `number` (positiveInt, the cycle ordinal),
  `total` (positiveInt, total planned cycles, 0..1), and an optional
  `protocol` reference (CodeableReference(PlanDefinition) or
  Reference(CarePlan)) so the cycle can be tied back to the regimen
  definition.
- `MedicationAdministration.cumulatedDose` — a 0..* element (BackboneElement
  or repeating Quantity/Ratio) capturing the running cumulative dose of the
  medication delivered to the patient at the point of this administration,
  across the cycles-to-date. Shape would mirror existing dose modelling
  (Quantity with UCUM, optional denominator for per-BSA/per-kg expressions)
  and should allow multiple entries to represent different dose views (e.g.
  per-cycle subtotal vs. lifetime total) via a `type` coded element.

Both elements would be Trial-Use, with guidance in the resource narrative
and invariants clarifying that `cumulatedDose` is informational (not a
substitute for server-side computation) and must be consistent with the
administration's own `dosage.dose`.

#### Justification

The request is grounded in a concrete, high-severity patient-safety scenario
(bleomycin pulmonary toxicity above 400,000 IU lifetime dose) that is common
across oncology regimens. Representing cycle number and cumulative dose
directly on `MedicationAdministration` keeps the safety-critical information
co-located with the event it qualifies, so downstream consumers (EHR
summaries, CDS, pharmacy verification) don't have to reconstruct it by
walking `basedOn`/`partOf` chains or summing prior administrations
themselves. It also aligns with existing Pharmacy precedent where
administration events carry contextual dose information (e.g. `dosage.rate`,
`dosage.dose`). The Confluence discussion (PHAR 2025-08-25) explicitly
framed this as a data-model gap rather than a guidance problem, supporting
a core-resource change.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add `cycle` or `cumulatedDose` as first-class elements on the
base `MedicationAdministration` resource. Instead, address the underlying
need through a combination of existing mechanisms plus a single, more
generic hook:

1. Model the chemotherapy regimen as a `PlanDefinition` (cycle structure,
   per-day drug list, total cycles) and the patient's course as a
   `CarePlan` or a `Task` hierarchy, using the existing `partOf`/`basedOn`
   chain to associate each `MedicationAdministration` with its cycle
   iteration.
2. For the cumulative-dose safety signal, use `Observation` (code: e.g.
   SNOMED "cumulative drug dose") with `derivedFrom` pointing at the
   contributing administrations, rather than embedding the running total in
   the administration itself.
3. If a shorter-path carrier is still needed, introduce a **single**
   extension at the Pharmacy module level —
   `http://hl7.org/fhir/StructureDefinition/medicationadministration-cycle`
   (BackboneElement with `number`, `total`, optional `protocol`
   reference) — and leave cumulative dose to `Observation`. This extension
   could be promoted to a core element later if adoption warrants it.

#### Justification

As costateixeira noted on the ticket, cross-event progress tracking is
conventionally a `Task`/`CarePlan` concern, not a per-event concern. Adding
`cumulatedDose` to `MedicationAdministration` risks data-integrity problems:
the value can become stale or inconsistent with the sum of prior
administrations, and it invites different systems to compute the total
differently (ideal vs. actual BSA, inclusion of partial doses, etc.). Using
`Observation` for the cumulative view lets the running total be an
*asserted* clinical fact with its own provenance, recorded-at time, and
method, while keeping the administration resource focused on "what was
given right now." A single cycle extension avoids a breaking/scope-widening
core change while still giving implementers the 80% win (knowing which
cycle an event belongs to), and it preserves the option of a future
core-element promotion if real-world usage proves out.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — existing mechanisms sufficient." The
clinical need is real, but FHIR already provides every primitive required
to express it:

- Cycle membership is representable today via `MedicationAdministration.partOf`
  (→ `MedicationAdministration` for the cycle's grouping event) and
  `basedOn` (→ `MedicationRequest` linked to a `PlanDefinition` that
  defines the cycle structure).
- Cumulative-dose safety ceilings are representable today via
  `Observation` with an appropriate SNOMED/LOINC code, `derivedFrom` the
  contributing administrations, and optionally a `DetectedIssue` for
  threshold breaches.
- The BSA-based per-administration dose is already carried on
  `MedicationAdministration.dosage`.

Recommend the Pharmacy WG instead publish implementation guidance (in the
MedicationAdministration notes page, or a Pharmacy IG) that shows the
canonical encoding of a multi-cycle chemo protocol using these existing
elements.

#### Justification

Adding elements to a widely-implemented, Normative-track resource to solve
a pattern that is already expressible carries a real cost: implementers
must decide whether to populate the new elements, the old linkage, or both,
and validators/terminology must be updated. The patient-safety concern is
addressed more robustly at the Observation/CDS layer than by a denormalised
`cumulatedDose` on each administration event, which can silently drift from
the true sum. "Not Persuasive — existing mechanisms sufficient" is the
correct disposition for change requests that are fully served by current
modelling primitives plus guidance.

---

### Recommendation

**Recommended disposition:** B

The clinical need is real and well-evidenced (BACOPDac / bleomycin
400,000 IU lifetime-dose ceiling is a genuine safety signal, not a
theoretical edge case), so an outright Decline (C) under-serves the
submitter and leaves the Pharmacy community without an interoperable way
to flag which cycle an administration belongs to. At the same time, the
single reviewer comment on the ticket explicitly flags that cross-event
progress tracking belongs at the `Task`/protocol level, and denormalising
a running total onto every `MedicationAdministration` invites drift and
double-sourcing — so Accept-As-Requested (A) is heavier than the use case
justifies, especially for a resource heading toward Normative. The
Alternative Approach (B) splits the two requested elements along their
natural seam: `cycle` becomes a lightweight Pharmacy-module extension (low
cost, directly addresses the gap the submitter identified), while
cumulative dose is expressed as an `Observation` derived from the actual
administrations (preserves provenance, avoids drift, and plays well with
CDS and DetectedIssue). This path gives the Pharmacy WG a concrete
deliverable for the next ballot cycle, leaves room to promote `cycle` to a
core element if adoption is strong, and keeps `MedicationAdministration`'s
surface area stable.
