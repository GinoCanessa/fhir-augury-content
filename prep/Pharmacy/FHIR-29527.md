## Summary

This ticket — filed against the US Prescription Drug Monitoring Program
(PDMP) FHIR Implementation Guide (now tracked under the broader US
Medication FHIR Implementation Guide work) — raises two related
questions about the IG's Security Considerations section. First, the
reporter (Scott M. Robertson) questions whether the existing wording
that says implementers "should pay attention to FHIR Security
considerations" is too weak, and whether such attention should instead
be a normative requirement ("shall"). Second, the ticket asks whether
implementations that are plain FHIR (i.e., not SMART on FHIR) are still
required to use the SMART Backend Services Authorization Guide for
securing PDMP transactions between the PDMP Requestor and the PDMP
Responder, or whether alternative authorization mechanisms are
permitted.

The ticket was originally dispositioned "Considered for Future Use"
(Hatem / McNeil, 4-0-0, 2023-01-18) on the basis that the IG had been
withdrawn, but it was reopened in November 2023 when the PDMP on FHIR
project (PSS-2225) was picked back up. No cross-referenced Jira
tickets, Zulip discussions, or GitHub issues/PRs were found by the
Augury indexes for this ticket — it is effectively a stand-alone
question awaiting workgroup direction.

### Disposition A: Accept As Requested

#### Proposal

Tighten the IG's Security Considerations section to make the security
posture normative and unambiguous:

1. Replace "All implementers of FHIR servers and clients **should** pay
   attention to FHIR Security considerations" with "All implementers of
   FHIR servers and clients **SHALL** comply with the FHIR Security
   considerations referenced from this IG, in addition to any
   jurisdictional requirements for controlled-substance data."
2. Restate the SMART Backend Services requirement as an unconditional
   normative rule: "PDMP transactions between the PDMP Requestor and
   the PDMP Responder **SHALL** be secured using the SMART Backend
   Services Authorization Guide, regardless of whether the broader
   client implementation is also a SMART on FHIR app." Add a sentence
   clarifying that SMART Backend Services applies because PDMP
   exchanges are system-to-system / no-user-context, which is
   independent of whether the client also supports SMART App Launch.
3. Add a short conformance table (or bullet list) enumerating the
   security controls the IG mandates (TLS version floor, audit
   logging, token introspection where applicable) so reviewers don't
   have to chase the prose for SHALLs.

#### Justification

Pharmacy data, and PDMP data in particular, is high-sensitivity
controlled-substance information that is subject to state and federal
regulation. A "should" verb in the IG leaves implementers room to
ship insecure conformant systems, which both undermines the IG's
value and creates regulatory risk. SMART Backend Services is already
the FHIR-ecosystem-standard answer for system-to-system OAuth, and
calling it out as SHALL — independent of any SMART App Launch usage —
matches the security model the IG was clearly written around.
Strengthening the wording is also aligned with how other US Realm IGs
(e.g., US Core, Da Vinci) phrase security obligations, so this brings
the IG in line with peer guides.

---

### Disposition B: Alternative Approach

#### Proposal

Keep "should" for the broad "pay attention to FHIR Security
considerations" sentence, but split the SMART Backend Services
requirement into two normative cases driven by the deployment profile:

1. Define two PDMP capability statements / actor profiles in the IG:
   - **PDMP Backend Client** (system-to-system, no end-user context):
     **SHALL** use SMART Backend Services.
   - **PDMP Interactive Client** (clinician-facing, user-context
     present): **SHALL** use SMART App Launch with the appropriate
     scopes; backend-services tokens are not permitted for this actor.
2. For "plain FHIR" deployments that are neither, require a
   conformant equivalent (mutually authenticated TLS plus an OAuth
   2.0 client-credentials flow that meets the same audience, JWT, and
   key-rotation rules as SMART Backend Services), and explicitly state
   that informal API-key or basic-auth schemes are non-conformant.
3. Keep the broader "FHIR Security considerations" reference as
   guidance ("should") because the upstream FHIR security page is
   itself a mix of normative and informative content, and elevating
   the umbrella reference to SHALL would create unverifiable
   conformance claims.

#### Justification

This addresses the reporter's underlying concern — that "plain FHIR"
deployments might escape the security obligation — without
overloading a single SHALL on top of an external page that the IG
does not control. Splitting by actor profile is consistent with how
SMART itself and Da Vinci IGs scope their security requirements, and
it gives implementers a clear conformance target to test against. It
also resolves the second question explicitly: a non-SMART
implementation does still owe an equivalent security posture, and the
IG names what "equivalent" means rather than leaving it to
interpretation.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive / Already Addressed**, on the
grounds that:

- The IG's intent that PDMP transactions be secured via SMART Backend
  Services is already stated, and the surrounding HL7 framework
  (SMART Backend Services, the FHIR Security page, and US Realm
  jurisdictional rules) supplies the normative force without the IG
  needing to restate it as SHALL.
- The "should pay attention to FHIR Security considerations" sentence
  is informative scaffolding pointing at an external normative
  resource; rewriting it as SHALL would make the IG conformance test
  unbounded (it would inherit every future change to the FHIR
  Security page).
- Plain-FHIR (non-SMART) deployments are out of scope for the PDMP IG
  by construction, since the IG's transaction definitions assume the
  SMART Backend Services token model.

#### Justification

This is defensible if the workgroup wants to minimize churn in a
recently-revived IG and rely on the broader SMART/FHIR security
ecosystem. The ticket is an old (2020) imported question with no
community follow-up on Zulip or GitHub, so there is no evidence of
real-world implementer confusion driving the request. Declining
preserves the existing wording the implementer community has already
read against.

---

### Recommendation

**Recommended disposition:** B

Disposition B best honors the reporter's two underlying concerns —
weak verb on the security obligation, and ambiguity for non-SMART
deployments — without producing the unbounded-conformance problem
that a blanket SHALL on the FHIR Security page would create
(Disposition A) and without dismissing legitimate confusion that the
ticket has been raised about (Disposition C). Tying the SMART Backend
Services SHALL to a concrete actor profile, and explicitly defining a
conformant non-SMART equivalent, gives implementers a testable bar
and gives the workgroup a clean place to evolve the security story
when the IG is reballoted under the revived PSS-2225 project. It is
also the disposition most likely to be acceptable on a re-vote, since
it materially improves conformance clarity while leaving the
upstream "FHIR Security considerations" reference as the
informative pointer it was always intended to be. Pharmacy WG should
confirm the correct authoring repository for the revived PDMP IG and
generate a `repo-analysis` briefing for it before turning this
disposition into an actual edit.
