## Summary

This is a clarifying question raised by Ron G. Parker (imported from "Comment 33" of an earlier review pass) asking whether `DeviceDefinition` is the intended target type for `MedicationAdministration.performer.actor` (or analogous `performer` element) so that the resource can express *the kind of device* permitted to carry out the administration, rather than a specific physical instance. The ticket is paired with its companion [FHIR-55008](https://jira.hl7.org/browse/FHIR-55008) ("Should performer still be allowed to reference a specific Device?") — together the two tickets ask the Pharmacy work group to clarify the intended semantics of "performer" when the performer is a device: does the spec mean a device *kind* (`DeviceDefinition`), a device *instance* (`Device`), or both? The single comment on FHIR-55007 explicitly defers to FHIR-55008 for the substantive change discussion, so this ticket is best disposed in tandem with that one.

## Proposed Dispositions

The dispositions below address the substantive question asked by FHIR-55007. Because the Cross-Reference index surfaced no related GitHub PRs/commits and no `repo-analysis` briefings are required, the proposals are framed at the FHIR Core spec level (the `MedicationAdministration` resource definition and its `performer.actor` allowed reference targets). Any of these dispositions should be coordinated with FHIR-55008.

### Disposition A: Accept As Requested

#### Proposal

Confirm and clarify in `MedicationAdministration` that `performer.actor` (typed `CodeableReference`) is intended to permit `DeviceDefinition` as a target so that an administration can identify the *kind* of device authorized/used to perform the administration (e.g., "any infusion pump of model X"), not only the specific physical instance. Concretely:

1. Ensure the allowed reference targets on `MedicationAdministration.performer.actor` include `DeviceDefinition` alongside the existing actor-style targets (Patient, Practitioner, PractitionerRole, RelatedPerson, Device).
2. Add a sentence to the `MedicationAdministration` notes/element definition explaining that `DeviceDefinition` is used when the performer is described by *device kind/model*, and `Device` when a specific physical instance is recorded.
3. Mirror the same wording on the related `MedicationAdministration.device` element so the two elements use a consistent device-kind-vs-instance framing.
4. Add an example illustrating each case (instance vs. kind) in the resource examples directory.

This treats the question as "yes — and the spec should state it explicitly," and pairs naturally with retaining `Device` per FHIR-55008 Disposition A.

#### Justification

`DeviceDefinition` was introduced specifically to represent device *kinds* (manufacturer/model/configuration) distinct from the physical instance modeled by `Device`. Allowing `DeviceDefinition` as a `performer.actor` target is consistent with the way other R5/R6 resources differentiate kind from instance (e.g., `Medication` vs `MedicationKnowledge`, `NutritionProduct` vs `NutritionIntake.nutritionItem`). Reporters in real-world infusion-pump and smart-pump scenarios frequently capture the model/kind that performed the administration when a specific serial-numbered instance is unknown or not recorded; explicit support avoids forcing implementers into extensions. The FMG enumeration thread above confirms `performer.actor` is already a `CodeableReference`, which is the data type designed for exactly this kind/instance flexibility.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add `DeviceDefinition` directly to `MedicationAdministration.performer.actor`. Instead:

1. Keep `performer.actor` constrained to the *acting party* targets (Patient, Practitioner, PractitionerRole, RelatedPerson, Device — i.e., something that actually *performed* the act). The performer is, by definition, an instance that did something at a point in time.
2. Use the existing `MedicationAdministration.device` element (already typed `CodeableReference`) to record the *kind* of device involved, since `device` already accepts `DeviceDefinition` for that purpose. When the device is also the performer, point `performer.actor` at the `Device` instance (or omit if unknown) and put the device-kind in `MedicationAdministration.device` (and/or use the `CodeableConcept` half of the `CodeableReference` to carry a coded device type).
3. Add explicit guidance in the `performer` and `device` element definitions explaining this division of labor, and add an example showing a smart-pump administration where the kind is captured in `device` and the instance (if any) is captured in `performer.actor`.

#### Justification

This preserves the semantic invariant that a "performer" is something that actually *acts* (and therefore is necessarily an instance, not a definitional/catalog entry), while still letting implementers record device-kind information without resorting to extensions. It avoids stretching `performer.actor` to mean "the kind of thing that could have performed this," which is a different concept and risks confusing query semantics (e.g., "find administrations performed by Dr. Smith" vs. "find administrations performed by *some pump of model X*"). It also leverages an element (`MedicationAdministration.device`) that already exists and is already a `CodeableReference`, so no breaking change to `performer.actor` target lists is required. Trade-off vs. A: implementers wanting a single field for "who/what performed it" must consult two elements, but the semantic clarity is worth it.

---

### Disposition C: Decline

#### Proposal

Decline the implied request to add or document `DeviceDefinition` as a `performer.actor` target. Resolve FHIR-55007 as **Answered** with the following clarification placed on the ticket (not in the spec):

> `MedicationAdministration.performer.actor` is intended to record the party that actually performed the administration. To capture *the kind of device* permitted or used, use `MedicationAdministration.device` (which is a `CodeableReference` and already supports `DeviceDefinition`), or carry the device kind in the `CodeableConcept` half of `performer.actor`'s `CodeableReference`. No spec change is required.

Pair with declining the substantive change in FHIR-55008 (i.e., keep `Device` instance on `performer.actor` and make no additions).

#### Justification

The ticket is filed as a **Question**, not a Change Request, and the underlying need (recording device kind on an administration) is already met by existing R6 mechanisms: (a) `MedicationAdministration.device` accepting `DeviceDefinition` via `CodeableReference`, and (b) the `CodeableConcept` half of any `CodeableReference` element. Adding `DeviceDefinition` to `performer.actor` introduces the conceptual oddity of a non-instance "performer" and would force implementers and validators to reason about whether a definition can act, with knock-on effects for search parameters and chaining. Declining preserves consistency with how `performer`-style elements are modeled across R6 admin resources (per the 2019 implementers thread, `performer.actor` historically targets *instances*).

---

### Recommendation

**Recommended disposition:** B

This question landed in Pharmacy because the `DeviceDefinition` resource was introduced after `MedicationAdministration.performer` was originally designed, and Ron G. Parker is rightly asking whether the work group consciously routed device-kind capture through `performer.actor` or whether it slipped through unreviewed. Disposition B is the cleanest answer: keep "performer" meaning *the party that acted* (an instance), and direct device-kind capture to `MedicationAdministration.device`, which is already a `CodeableReference` and already has `DeviceDefinition` in scope. This preserves the semantic distinction between an act's performer and the catalog/model of equipment involved, avoids broadening `performer.actor`'s target list (which costateixeira explicitly flagged on FHIR-55008 as something to avoid doing accidentally), and requires only a small clarifying note in the resource definition rather than a structural change. Disposition A has implementer appeal but conflates two concepts FHIR has otherwise been careful to keep separate; Disposition C is technically defensible but leaves the spec silent on a question that has already been asked twice (FHIR-55007 + FHIR-55008) and will keep recurring. The work group should resolve FHIR-55007 with the B-style clarification and use the same framing to drive FHIR-55008's disposition.
