## Summary

The ticket asks the CARIN Real-time Pharmacy Benefit Check (RTPBC) IG to add
structured representation of two NCPDP Real-Time Prescription Benefit Standard
data elements that, as of NCPDP's February 2022 ballot, support codifying
information that today is typically conveyed in free-text response messages:

1. **Formulary Status** (mandatory in the NCPDP message) — a coded status
   describing where a drug sits in the payer's formulary, with values such as
   *Brand Preferred, Carve-Out, Non-Formulary, On Formulary, Preferred,
   Unknown, Non-Reimbursable without authorization,* and *Non-Reimbursable*.
2. **Preference Level** (situational; required when formulary status is
   non-formulary, on-formulary, or preferred and preference levels exist) —
   a numeric ordinal in the range 1–99 where a lower number indicates a more
   preferred alternative.

The request is scoped to the RTPBC IG so that both the submitted product and
any returned alternative products on the response can carry formulary
positioning and preference order in a structured form, supporting the
patient/prescriber decision about pricing and substitution. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced
by the data layer for this key.

The ticket has been resolved **Persuasive**, with the resolution committing
to add a Formulary Status code system, value set, and extension covering the
requested codes (with one additional value, `Z` "Zero Dollar Copay"), plus
an extension carrying a Preference Level integer 1–99.

## Details

**Description:**

To support codifying information communicated via free text messages,
NCPDP's February 2022 ballot includes proposed new data elements to the
Real-Time Prescription Benefit Standard to communicate formulary status and
preference level. This allows for the communication of the formulary status
of both submitted product and alternative products to help understand
pricing on the response.

Information about data elements from NCPDP Data Dictionary and External
Code List:

- **Formulary Status field (mandatory)**
  - Definition: Status of the drug within the formulary.
  - Format: x(2)
  - Value Descriptions: Brand Preferred; Carve-Out; Non-Formulary;
    On Formulary; Preferred; Unknown; Non-Reimbursable without
    authorization; Non-Reimbursable.
- **Preference Level field (situational)**
  - Definition: If there are multiple alternatives for a given Source drug,
    this is the payer's order of preference (a lower number equals greater
    preference).
  - Format: 9(2)
  - Situation of Use: Required when values for formulary status are
    non-formulary, on-formulary, or preferred and preference levels exist.
    1 through 99, lower is more preferred.
  - Value descriptions: Least preferred; Ordered preference between least to
    most; Most preferred.

**Resolution Description (Persuasive):**

Add a Formulary Status code system, value set, and extension with the
requested values:

- `B` — *Brand Preferred* — "Indicates the payer prefers the brand over the
  generic"
- `C` — *Carve-Out* — "This medication is processed by a third party, not
  the payer providing the formulary"
- `N` — *Non-Formulary* — "Response code indicating that the prescribed drug
  is not included in the plan formulary"
- `O` — *On Formulary* — "The medication submitted is included in the list
  of payable products in that patient's plan formulary but that there may be
  a more preferred product in the therapeutic category"
- `P` — *On Formulary/Preferred* — "The medication is the preferred payable
  products in that patient's plan formulary"
- `U` — *Unknown* — "Unknown"
- `X` — *Non-Reimbursable without authorization* — "This medication is not
  covered although an exception or appeal process may exist"
- `Y` — *Non-Reimbursable* — "The medication will never be reimbursable by
  the payer with no appeal process"
- `Z` — *Zero Dollar Copay* — "This medication does not have a copay. The
  cost is $0 to the patient"

Add an extension for Preference Level with values 1–99, where 1 = most
preferred and 99 = least preferred.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the CARIN RTPBC IG, add structured carriers for both NCPDP elements:

1. **Formulary Status**
   - Define a new `CodeSystem` (e.g., `CodeSystem/FormularyStatus`) and a
     bound `ValueSet` enumerating the eight NCPDP-aligned values
     (`Brand Preferred`, `Carve-Out`, `Non-Formulary`, `On Formulary`,
     `Preferred`, `Unknown`, `Non-Reimbursable without authorization`,
     `Non-Reimbursable`), using single-letter codes consistent with NCPDP's
     External Code List (`B`, `C`, `N`, `O`, `P`, `U`, `X`, `Y`).
   - Define an `Extension` (e.g., `Extension/formulary-status`) of type
     `CodeableConcept` bound `required` to the value set, applicable on the
     RTPBC response profiles for both the submitted product and each
     alternative product (the `ClaimResponse.item` / `addItem` slices used
     today to convey the source drug and alternatives).
2. **Preference Level**
   - Define an `Extension` (e.g., `Extension/preference-level`) of type
     `positiveInt` with an invariant constraining the value to `1..99`,
     applicable on the same alternative-product elements.
   - Document the situational rule in the extension's description: required
     when formulary status is `Non-Formulary`, `On Formulary`, or
     `Preferred` and preference levels exist; lower number = more preferred.

This mirrors NCPDP's February 2022 ballot semantics and lets implementers
exchange formulary positioning structurally rather than as free text.

#### Justification

This is exactly what the NCPDP standard adds and what the requester (an
NCPDP/Pharmacy SME) asked for; it is also consistent with the resolution
already recorded on the ticket (Persuasive). RTPBC's stated purpose is to
mirror the NCPDP RTPB transaction in FHIR, so divergence from the NCPDP
data dictionary on a *mandatory* field would actively harm round-tripping
between NCPDP and FHIR implementations. Using a code system + value set +
extension pairing keeps the change additive (no cardinality changes on
existing RTPBC profiles) and is the conventional FHIR pattern for
introducing payer-domain data elements that aren't yet in the base
ClaimResponse model.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt Formulary Status and Preference Level, but **lean on existing FHIR /
external code systems** rather than minting an RTPBC-only `CodeSystem`:

- For Formulary Status, bind the new extension to a value set whose
  concepts come from an **external NCPDP code system URI** (declared as a
  `not-present` / external `CodeSystem` resource pointing at NCPDP's
  External Code List), or check whether
  `http://terminology.hl7.org/CodeSystem/v3-ActCode` /
  `formulary` already covers the semantics for a subset, and only mint
  HL7-owned codes for any gaps.
- For Preference Level, instead of a new extension, evaluate reuse of
  `ClaimResponse.item.adjudication` with a category code such as
  `priority` carrying a `Quantity`/`integer` value, which already exists
  for ranking adjudicated line items.

#### Justification

External-CodeSystem reuse avoids HL7 owning a parallel code list that
must be maintained in lockstep with NCPDP every time NCPDP adds a value
(e.g., "Zero Dollar Copay" was added in the resolution but is *not* in the
original ticket's enumeration, illustrating the drift risk). Reusing
adjudication slots for preference order avoids extension proliferation on
profiles that already have a rich adjudication structure. The trade-off
versus Disposition A is implementer ergonomics: adjudication-based
preference is less obvious to RTPBC implementers than a named extension,
and external-CodeSystem references depend on stable NCPDP URIs that HL7
does not control.

---

### Disposition C: Decline

#### Proposal

Decline to add Formulary Status and Preference Level to RTPBC at this
time, on the grounds that the underlying NCPDP elements were still in
ballot at submission and that implementers can continue to convey the
information either in `ClaimResponse.item.noteNumber` / `processNote.text`
free-text or in implementer-defined extensions until NCPDP normatively
publishes the elements.

#### Justification

The ticket itself notes the elements are part of an NCPDP *ballot*; if the
ballot had failed or the codes had shifted, an HL7-defined code system
would have to follow. Declining defers HL7 maintenance burden until NCPDP
finalizes. The downside, and the reason this is the weakest option, is
that it institutionalizes the very free-text problem the ticket is trying
to fix and forces every implementer to invent local extensions.

---

### Recommendation

**Recommended disposition:** A

The ticket's existing **Persuasive** resolution already commits to
substantively the same change as Disposition A (CodeSystem + ValueSet +
Formulary Status extension; Preference Level extension over 1–99), so the
workgroup has effectively pre-endorsed this path. RTPBC's value to
implementers depends on faithfully representing NCPDP's RTPB transaction;
a *mandatory* NCPDP element with no FHIR carrier is a structural gap, not
a stylistic one. Disposition B's instincts about reducing HL7's
terminology-maintenance footprint are worth carrying forward as a
follow-up — specifically, ensuring the new CodeSystem documents its
relationship to the NCPDP External Code List and includes the additional
`Z` "Zero Dollar Copay" code surfaced in the resolution — but they are
not strong enough to override the implementer-ergonomics win of named
RTPBC extensions. Disposition C should not be pursued; deferring would
leave the IG misaligned with the very NCPDP standard it mirrors.
