## Summary

The ticket captures feedback from the Kansas PDMP (Gayle Donaldson) on the
draft US PDMP-on-FHIR IG, relayed via project facilitator Scott Robertson.
The submitter reports that approximately 99% of their PDMP integrations are
no longer NCPDP-based and instead follow a **two-call interaction pattern**:
a first call that "queues up" a search on the PDMP server, followed by a
second call that records an executed search and returns the result data.
The PDMP must be able to **distinguish these two request types in its audit
trail** in order to report usage data for federal grants and state
performance measures. Scott Robertson's inline response acknowledges the
pattern and notes that some IGs describe a 2-step query in which the initial
request returns an identifier (e.g., a GUID) and a follow-up HTTP request
with that identifier retrieves the data; he flags that Frank McKinney's
input may be needed and that the topic warrants discussion. The ticket is
filed as a Question (not a defect or enhancement request) and is asking the
Pharmacy WG to confirm how — or whether — the IG will support and
distinguish the two phases of this two-call workflow. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were found for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the US PDMP-on-FHIR IG to **explicitly describe and support a
two-step query pattern** and to **require that the request "type" (queue
vs. execute) be carried in the request payload** so it can be recorded in
the responder's audit trail. Concretely:

- Define two distinct interactions in the IG's "Querying a PDMP" page:
  1. A "queue / prepare" query whose response returns a server-issued
     identifier (e.g., a `Bundle.identifier` value or an `OperationOutcome`
     carrying a GUID/token) but no PDMP result data.
  2. A "retrieve / execute" query that resolves that identifier and returns
     the actual result Bundle.
- Specify the wire-level mechanism for step 2 — most naturally a FHIR
  `$pdmp-search` (or similarly named) operation with an input parameter
  carrying the identifier from step 1, or a parameterized search such as
  `GET [base]/MedicationDispense?_query=pdmp-execute&token={guid}`.
- Mandate that requesters tag every request with the phase (e.g., a
  `requestType` parameter or a coded value bound to a small IG-defined
  ValueSet `pdmp-request-type` with codes such as `prepare` and `execute`),
  and require the PDMP server to record that code on the corresponding
  `AuditEvent` (e.g., on `AuditEvent.action`, a `subtype` coding, or an
  `AuditEvent.entity.detail`).
- Profile `AuditEvent` for PDMP usage so that grant- and
  performance-reporting tooling has a stable element path for the request
  phase.

#### Justification

The submitter is the operator of a real production PDMP and reports that
~99% of their integrations already use this two-call pattern. The
distinction matters operationally — federal grant and state performance
reporting depend on counting *executed* searches, not *prepared* ones, and
the IG cannot be deployed in Kansas (and likely other states with similar
reporting obligations) unless that distinction is unambiguous on the wire
and in the audit log. The 2-step pattern is also the pattern Scott
Robertson explicitly identified in his reply as "may match what you are
asking for," and it is consistent with the asynchronous-search style used
in other FHIR IGs (e.g., Bulk Data $export, async patterns in R5). Adopting
it deliberately and documenting the request-type element gives the
community one canonical answer instead of letting each PDMP invent its own.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a new IG-specific 2-step interaction, **adopt FHIR's
existing async-request pattern** (the `Prefer: respond-async` header /
`Bundle` async polling defined in the FHIR Async Request Pattern, and used
by Bulk Data) for any PDMP query that the responder chooses to defer, and
**handle the audit-trail "request type" requirement out-of-band** in an
`AuditEvent` profile only:

- Do not invent a new "prepare vs. execute" pair of FHIR interactions.
  Instead, document that PDMP servers MAY accept `Prefer: respond-async`
  on the standard PDMP query and, when they do, return a polling URL per
  the standard pattern; the requester polls that URL to retrieve results.
- Define a PDMP `AuditEvent` profile that requires
  `AuditEvent.action` / a subtype coding to distinguish three concrete
  audited events: `query-submitted`, `result-prepared`, and
  `result-retrieved`. This lets a PDMP that *internally* uses a two-call
  workflow record both phases without forcing every PDMP and every client
  to implement two distinct FHIR endpoints.
- Provide implementation guidance that maps the Kansas PDMP's existing
  two-call workflow onto these events, and clarify that the wire-level
  shape of the queries themselves is the existing FHIR query/operation,
  not a new IG invention.

#### Justification

The submitter's hard requirement is the **audit-trail distinction**, not
the wire shape per se. Solving the audit need with an `AuditEvent` profile
is lower-impact than adding a new IG-specific interaction model, avoids
divergence from the rest of the FHIR ecosystem, and lets PDMPs that don't
need the two-call shape keep a simple synchronous endpoint. Reusing the
established async pattern for the deferred case keeps the IG consistent
with Bulk Data and other FHIR IGs that have already solved
"queue-then-fetch" — implementers benefit from libraries and patterns they
already know. This also preserves the option for any PDMP, including
Kansas, to keep their internal two-call pipeline while exposing whatever
interaction shape best fits their clients.

---

### Disposition C: Decline

#### Proposal

Decline to change the IG. Respond to the submitter that the IG already
supports their need:

- A PDMP can implement a synchronous FHIR query today; the "first call"
  the submitter describes is an implementation-internal concept and does
  not need to be modeled as a separate FHIR interaction.
- Audit-trail granularity (recording prepare vs. execute) is a server-side
  audit concern that any PDMP can already capture with `AuditEvent`
  without IG changes — the IG does not need to define a `requestType`
  element to enable that.
- Close this Question ticket as "answered, no IG change required," and
  document the response on the PDMP IG FAQ page so future implementers
  see the same answer.

#### Justification

The ticket is filed as a **Question**, not a change request. Its primary
purpose is to clarify whether the FHIR design accommodates the Kansas
two-call workflow; the answer is "yes, it accommodates it, and how you
audit your internal phases is a local server concern." Adding an
IG-mandated `requestType` element risks over-fitting the IG to one
state's operational pipeline and would force every other PDMP and every
PDMP client to implement and validate a distinction they may not need.
`AuditEvent` is already extensible enough to record any phase the server
wants to log, without IG normative text.

---

### Recommendation

**Recommended disposition:** B

Disposition A correctly identifies that "request type in the request
elements" and the two-call workflow are real, deployed concerns, but
inventing PDMP-specific `prepare` / `execute` interactions duplicates
work the wider FHIR community has already done in the Async Request
Pattern and Bulk Data, and risks isolating PDMP-on-FHIR from those
ecosystems. Disposition C under-serves the submitter: the audit-trail
requirement is concrete and tied to federal grant reporting, and dismissing
it as a purely local concern leaves Kansas (and other similarly-situated
PDMPs) without portable, comparable audit data across PDMPs — which is
exactly what an IG should standardize. Disposition B threads that needle:
it solves the audit-trail distinction with a PDMP-specific `AuditEvent`
profile (the place where this distinction actually belongs), it points
implementations that need deferred delivery at the existing FHIR async
pattern instead of inventing a new one, and it leaves PDMP servers free to
keep their internal two-call pipelines. The Pharmacy WG should also use
this ticket as a prompt to invite the submitter to the next PDMP IG call so
that the `AuditEvent` profile's coded request-type ValueSet is grounded in
the events PDMPs actually need to count for federal and state reporting.
