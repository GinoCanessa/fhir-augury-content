## Summary

FHIR-38538 is one in a numbered series of TSMG-imported change requests
("Defining URL should be anchored in THO. 6") raised by Joan Harper against
the **US Specialty Rx (FHIR)** implementation guide. The boilerplate body —
imported on behalf of TSMG by Ron G. Parker — asks the Pharmacy work group
to move a Value Set's canonical URL out of the `hl7.org/fhir` namespace and
into `terminology.hl7.org` (THO), and to file a corresponding UTG ticket so
the Value Set is registered in THO. The TSMG guidance allows only Value
Sets bound as `required` to a `code`-typed element to remain anchored under
`hl7.org/fhir`; everything else should live in THO.

The ticket was originally dispositioned **"Considered for Future Use"** on
2023-01-30 by a 6-0-0 vote (Frank McKinney / John Hatem), with the comment
"This change will be considered when the policy related to anchoring of
IG-defined terminology is finalized." Lynn Laakso reverted that resolution
and re-opened the ticket on 2023-05-04 ("re-open post STU2 publication"),
so it now sits in Triaged/Unresolved state for re-disposition. The ticket
body does not name the specific Value Set (the "6" in the title indexes the
TSMG batch, not a ValueSet identifier), so the work group will need to
re-establish the target artifact during disposition.

There are no cross-referenced Jira tickets, Zulip threads, GitHub items, or

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the TSMG guidance as written for the Value Set targeted by this
ticket: in the US Specialty Rx (FHIR) IG, change the Value Set's
`ValueSet.url` from `http://hl7.org/fhir/us/specialty-rx/ValueSet/<id>`
to `http://terminology.hl7.org/ValueSet/<id>` (or the equivalent THO
canonical), update every binding reference (in StructureDefinitions,
narrative, IG resources, and examples) to the new canonical, and add a
deprecated/retired alias for the old URL so existing implementers are not
broken silently. File a paired UTG ticket against `HL7/UTG` to register
the Value Set in THO with the same identifier and codes as the IG copy.
Apply the same treatment consistently across the sibling tickets
(FHIR-38381, -38450, -38452, -38530, -38539 – -38542, -38553, -38565, …).
Keep anchored under `hl7.org/fhir` only those Value Sets that are bound
`required` to a `code`-typed element, per the TSMG rule.

#### Justification

This is the disposition TSMG explicitly asked for and is consistent with
the broader HL7 policy of hosting cross-IG terminology in THO so it can
be reused without dragging an entire IG canonical along. It also aligns
with how the FHIR Core specification has been moving Value Sets to THO,
making the US Specialty Rx IG more consistent with sibling Pharmacy and
US-Realm IGs. Lynn Laakso's 2023-05-04 reopen explicitly notes "post STU2
publication," meaning the IG has already shipped under the wrong
canonicals; the longer the relocation is deferred, the more implementers
will harden against the `hl7.org/fhir` URLs and the more painful the
eventual move becomes.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than disposition each numbered TSMG ticket individually, re-scope
this ticket (and its siblings) into a single batch effort: produce one
inventory ticket against US Specialty Rx that lists every IG-defined
Value Set, classifies each as "required-binding-to-code → keep on
hl7.org/fhir" or "everything else → relocate to THO," and tracks one
omnibus UTG submission. Close FHIR-38538 (and the other numbered
duplicates) as superseded by the inventory ticket, and execute the
relocation in a single coordinated PR + UTG submission for the next
publication of the IG. Treat the affected Value Sets as a versioned set
so the URL change ships with a clear migration note in the IG's change
log and an extension on the old canonical pointing to the new one.

#### Justification

The 20+ sibling tickets carry identical boilerplate and no per-ticket
context beyond a serial number. Disposition-by-disposition voting will
generate the same outcome 20+ times while obscuring the actual scope of
the change for implementers. A single inventory + omnibus UTG submission
gives the Pharmacy WG one decision to make, gives implementers one
migration to plan for, and gives TSMG one place to verify the binding
classification was applied correctly. It also avoids the failure mode
where some siblings are accepted, others declined, and the IG ends up
with Value Sets split across two canonicals with no overarching rule.

---

### Disposition C: Decline

#### Proposal

Decline the TSMG request for US Specialty Rx, on the grounds that (a)
the IG has already published as STU2 with the existing
`hl7.org/fhir/us/specialty-rx/ValueSet/...` canonicals and changing the
canonical URL is, by FHIR rules, a breaking change even if the codes are
unchanged; (b) the original 2023-01-30 "Considered for Future Use"
disposition was correct in waiting for the IG-defined-terminology
anchoring policy to be finalized, and that policy has not yet been
ratified; and (c) TSMG has not provided a per-Value-Set list, only a
generic template, so the work group cannot verify which Value Sets
actually fail the "required-binding-to-code" test. Mark as Persuasive
with Mod / Considered for Future Use until TSMG provides a concrete list
and the anchoring policy is published.

#### Justification

The ticket body is generic and does not name the affected Value Set;
acting on it without that specificity risks moving Value Sets that
legitimately should remain on `hl7.org/fhir`. Changing canonical URLs
post-STU2 has real implementer cost: every CapabilityStatement,
profile reference, and validation rule that points at the old URL must
be updated, and silent-fail terminology lookups (`expand`, `validate-code`)
against the old canonical will degrade. This was the rationale for the
original "Considered for Future Use" outcome; nothing in the reopen
comment indicates the IG-defined terminology anchoring policy has since
been finalized.

---

### Recommendation

**Recommended disposition:** B

The TSMG guidance itself is sound — Disposition A captures the right
*direction* — but executing it ticket-by-ticket against 20+ identical
boilerplate tickets is the wrong *vehicle*. The risk of inconsistent
outcomes across the batch is high: each numbered ticket will be voted on
in isolation, with no shared per-Value-Set context, and the IG can end
up half-migrated. Disposition B preserves the substance of TSMG's
request, gives implementers a single coordinated migration aligned with
the next IG publication, and lets the Pharmacy WG vote once on the
classification rule rather than 20+ times on its consequences. It also
gives the WG a clean handle for the policy-readiness concern that drove
the original 2023-01-30 deferral: the inventory can sit ready, with the
omnibus UTG submission gated on the IG-defined-terminology anchoring
policy being finalized. If the WG prefers to honor the literal TSMG ask,
fall back to Disposition A and apply it uniformly across every sibling
ticket in the same vote; Disposition C should only be selected if TSMG
declines to provide the per-Value-Set list when asked.
