## Summary

Craig Newman is asking the Pharmacy WG to clarify, in the R6 Dosage rework,
how the timing-bearing elements at different nesting levels of `DosageDetails`
interact and whether one takes precedence over another. Specifically, the
`DosageDetails.step` sub-element (the ticket text refers to this loosely as
"DosageDetails contains .start and .end") carries `step.start` and `step.end`
of the new `RelativeTime` data type, while the required `step.component`
sub-element uses the `Dosage` data type, which itself contains a `Dosage.timing`
element of type `Timing`. The reporter wants the spec to explain how these two
timing layers are meant to be used together, and what implementers should do
when they appear to disagree.

The ticket has no Jira links and no automated cross-references in Augury, but
it sits squarely within the wider R6 `DosageDetails`/`step`/`component` rework
discussed at length in the `Medication` Zulip stream (topic *Dosage*, leading
to PR [HL7/fhir#3974](https://github.com/HL7/fhir/pull/3974) and the rework
preview at `https://build.fhir.org/branches/2025-11-gg-dosage-rework/dosage.html`)
and the follow-up topic *R6 DosageCondition vs. dosage-conditions extension*,
which raises the same `step.*` vs `component.timing` interaction concern (and
extends it to `step.count`).

## Details

**Description:**

> Given that DosageDetails contains .start and .end elements and the required
> .component element uses the Dosage data type that also contains a .timing
> element, please clarify how these elements are used together and which (if
> either) has precedence if there is a discrepancy.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add explicit normative prose and an example to `dosage.html` (the `DosageDetails`
section of the R6 Dosage rework) that defines the relationship between the
`step`-level temporal elements and the inner `component[].timing` element. The
recommended semantics, consistent with how Grahame has described `step` in the
*Dosage* Zulip thread, are:

- `DosageDetails.step.start` and `DosageDetails.step.end` (both
  `RelativeTime`) define the **outer envelope** of the step: when the step
  begins relative to some anchor event and when it ends. They describe the
  step as a whole, not any single component dose.
- `DosageDetails.step.component[].timing` (`Timing`) describes the schedule
  of doses **within** that envelope for that component. It is interpreted
  **relative to the step's start/end window**, not as an absolute schedule
  that can override the step boundaries.
- If `component.timing.repeat.bounds[x]` (or
  `component.timing.event`) appears to extend outside `step.start`/`step.end`,
  the step boundaries win: doses are not administered before `step.start`
  or after `step.end`. This makes `step.start`/`step.end` authoritative for
  the temporal envelope, and `component.timing` authoritative for the
  intra-step pattern.
- Add at least one worked example to `dosage.html` showing a multi-step
  course where the same `component.timing` is reused under different
  `step.start`/`step.end` windows, plus a constraint warning implementers
  about the precedence rule.

#### Justification

This is exactly what the reporter is asking for — clarification, not a
structural change — and it codifies the design intent that came out of the
*Dosage* Zulip discussions (Grahame Grieve and Jose Costa Teixeira). The
precedence rule (step envelope wins) is consistent with how the rework
positions `step` as the outer scheduling layer and `component` as the dose
content. It also closes the open question raised by Kari Heinonen on the
*R6 DosageCondition vs. dosage-conditions extension* thread on 14 April 2026,
where the meaning of `step.start`/`step.end` vs the inner schedule was
explicitly flagged. No element changes are required, so it is non-breaking
within the rework.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only writing prose, **constrain** the model so the discrepancy
the reporter is worried about can't arise:

- Add an invariant on `DosageDetails.step` requiring that, if both
  `step.start`/`step.end` and any `step.component[].timing.repeat.bounds[x]`
  (or `component.timing.event`) are populated, the `component.timing`
  bounds/events SHALL fall within the `step.start`–`step.end` window
  (interpreting `RelativeTime` against the same anchor).
- Add a companion invariant that `component.timing.repeat.count` /
  `count`-style repetition SHALL NOT cause administration outside the step
  envelope.
- Document the precedence rule (step envelope wins) and back it with the
  invariants so validators flag inconsistent dosages instead of leaving
  interpretation to implementers.
- Optionally, if the WG accepts the symmetry concern Kari Heinonen has
  raised repeatedly, consider exposing a `step.timing` (or equivalent)
  element so simple repetition at the step level doesn't need to be encoded
  in `component.timing` for single-component steps. This is a larger change
  and would be a separate proposal.

#### Justification

Clarifying prose alone (Disposition A) leaves implementers free to write
inconsistent instances; an invariant turns the ambiguity into a hard validation
error and is therefore safer for a clinical data type. The R6 Dosage rework
is already a breaking change, so adding invariants now — before R6 normatives
the structure — has minimal cost and addresses the same kind of safety
concern Grahame raised when defending the `DosageOrchestration`/`doseLimits`
ideas in the original *Dosage* thread. The downside is that invariants on
`RelativeTime` comparison are non-trivial when anchors differ, which the
Justification for the invariant must call out.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as Not Persuasive (or Persuasive With Modification, deferring
to a follow-up R7 ticket) on the grounds that the intended interaction is
already implicit in the published model: `step` describes the step, and
`component.timing` describes the dose pattern within the step. No spec change
is made; the reporter is pointed at the existing `dosage.html` examples and
the R6 Dosage rework PR ([HL7/fhir#3974](https://github.com/HL7/fhir/pull/3974)).

#### Justification

The R6 Dosage rework is a large, recently-balloted change set that the
Pharmacy WG is still stabilizing. Adding precedence prose or invariants now
re-opens design discussion at a moment when Grahame has signalled that R6
needs to close. If implementers genuinely run into the discrepancy in the
field, a targeted change can be made in R7 with concrete examples in hand.

---

### Recommendation

**Recommended disposition:** A

The reporter is asking a clarification question, not a structural one, and
the answer (step boundaries are the outer envelope; `component.timing` is the
intra-step pattern; step boundaries win on conflict) is already the working
mental model in the *Dosage* Zulip thread — it just isn't written down in
`dosage.html`. Codifying it in prose plus an example is cheap, non-breaking,
directly answers Craig Newman's request, and discharges the same open
question Kari Heinonen has raised on the *R6 DosageCondition* thread.
Disposition B is the right longer-term move, but the WG should land the
prose first (A) and treat the invariants and any `step.timing` symmetry
work as a follow-up — landing both in one ticket risks re-opening the
broader rework debate that Grahame has explicitly asked the group not to
re-open before R6 closes. Declining (C) leaves a known ambiguity in a
safety-critical data type and should be avoided.
