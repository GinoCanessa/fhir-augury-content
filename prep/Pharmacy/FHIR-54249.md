## Summary

The ticket asks the Pharmacy work group to relax the cardinality of
`MedicationRequest.effectiveTiming[x]` (the choice element that captures
when the medication is to be taken) from its current `0..1` to `0..*`.
The reporter motivates this by example: a prescription such as "take x
starting on Jan. 1st 2026 for three months" naturally combines two
representations of the same choice — a `Duration` (three months) and a
`Period` whose `start` is anchored (Jan. 1st 2026) — and they argue the
modeller should be allowed to express both pieces simultaneously rather
than being forced to pick a single variant.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items linked to this ticket; the only outgoing references are mentions
of the FHIR artifacts named above. A single comment from `costateixeira`
pushes back on the proposal as a likely overlap with `dosage` and a
source of conflict between competing representations of the same
information.

## Details

**Description:**

There are examples where the period over which the medication has to be
taken is defined as a startdate with a duration such as "take x starting
on Jan. 1st 2026 for three months". This could be expressed by combining
`MedicationRequest.effectiveTimingDuration` and
`MedicationRequest.effectiveTimingPeriod.start`.

Suggest changing cardinality of `MedicationRequest.effectiveTiming[x]`
to `0..*`.

**Comments:**

- **costateixeira (2026-04-06):** "I think that would overlap with
  dosage, and it would also cause confusion around the intended purpose
  of the element — this is the time period in which the medication is
  taken. Allowing several ways to define it might risk conflicting
  entries and I think this is supposed to be a simple element. Also I
  don't understand why effective can be a range."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the cardinality of `MedicationRequest.effective[x]`
(internally typed as `dateTime | Period | Duration | Timing`) from
`0..1` to `0..*` in the FHIR Core MedicationRequest resource. Add
explicit invariant text and a comment on the element clarifying that
when more than one value is present, the repetitions must describe the
same effective period (e.g., a `Period` whose `start` is anchored plus a
`Duration` that gives its length) and must not contradict each other.
Add a `MedicationRequest`-level invariant such as
`mrq-effective-consistent` that requires any combination of `Period` /
`Duration` / `Timing` repetitions to be temporally consistent.

#### Justification

This is the literal request and addresses the modelling pain point the
reporter raises: today an authoring system that knows both the start
date and the planned length is forced to drop one of those facts (or
shoehorn them into `dosage[*].timing`, which describes the *cadence*
of administration rather than the overall effective window). FHIR has
precedent for choice elements with `0..*` cardinality (e.g.,
`Observation.value[x]` is `0..1` but `Observation.component.value[x]`
patterns and several `*.identifier` choice-bearing structures have shown
that multi-cardinality choices are workable when paired with an
invariant that constrains the combinations).

---

### Disposition B: Alternative Approach

#### Proposal

Leave `MedicationRequest.effective[x]` at `0..1` and instead direct the
"start + duration" use case at the existing `Timing` data type.
`Timing.repeat.boundsPeriod` already carries `start`/`end` and
`Timing.repeat.boundsDuration` already carries the length; a single
`effectiveTiming` value populated with `repeat.bounds[x]` set to either
`Period` (with `start` only) or `Duration`, plus
`Timing.repeat.count` / `frequency` if relevant, captures the same
information without changing core cardinality. Document this pattern
explicitly in the MedicationRequest narrative and in the `effective[x]`
element comments, and add an example bundle that uses
`effectiveTiming.repeat.boundsPeriod.start` + `boundsDuration` together.

If the work group agrees the spec text is the gap, file a paired
documentation-only change to MedicationRequest's `effective[x]`
element comments.

#### Justification

This avoids the design risk that `costateixeira` flagged: introducing
multiple competing representations of the same fact, with no clearly
defined precedence rule, invites conflicting instances and complicates
every consuming system (CDS, dispensing, claims). The `Timing` data
type was specifically engineered to compose `boundsPeriod` and
`boundsDuration` together, so the requested capability is already
available — the issue is discoverability, not modelling. Keeping
`effective[x]` at `0..1` also preserves backward compatibility for every
existing implementation guide that profiles MedicationRequest (US Core,
IPS, AU Base, etc.) and avoids forcing those IGs to add slicing or
`max=1` constraints just to retain their current contract.

---

### Disposition C: Decline

#### Proposal

Decline the request with rationale "already addressed by existing
mechanism". Close the ticket as `Persuasive with Modifier` pointing to
the `Timing` data type's `repeat.bounds[x]` (`boundsPeriod`,
`boundsDuration`, `boundsRange`), and optionally add a one-line clarifying
note to `MedicationRequest.effective[x]`'s element definition.

#### Justification

The combination "start date + length" is already first-class in FHIR via
`Timing.repeat.boundsPeriod` plus `Timing.repeat.boundsDuration` (or
`boundsRange`). Promoting `effective[x]` to `0..*` would (1) overlap
with `dosage[*].timing.repeat.bounds[x]`, (2) introduce a new class of
invariant ("repetitions must be consistent") that has historically been
hard to enforce and validate, and (3) be a breaking change for every
downstream profile that asserts `max=1` implicitly. The single comment
on the ticket already raises the overlap concern, and there is no
recorded community demand (no Zulip threads, no related tickets, no
linked PRs) suggesting the existing mechanism is insufficient.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need — expressing "starting on date X for
duration Y" on a MedicationRequest — is real and worth addressing, but
relaxing `effective[x]` cardinality (Disposition A) creates an invariant
burden and a breaking-change ripple through profiles that the single
comment thread already foreshadows as problematic. Pure decline
(Disposition C) leaves the reporter without an answer and does nothing
to prevent the next person hitting the same modelling wall.
Disposition B threads the needle: it acknowledges the gap as a
*documentation* problem (the `Timing.repeat.bounds[x]` pattern is not
discoverable from `effective[x]`), produces a concrete spec-text /
example change in the FHIR Core MedicationRequest narrative, and avoids
both the invariant complexity and the backward-compatibility cost that
Disposition A would impose. Because the ticket has no GitHub or Zulip
context to override this analysis, the work group should open the
discussion from B and only fall back to A if implementers can produce
concrete cases that `Timing.repeat.bounds[x]` cannot already handle.
