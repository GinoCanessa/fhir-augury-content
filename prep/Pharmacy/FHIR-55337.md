## Summary

The ticket flags two value sets bound on `MedicationStatement` as potential candidates for migration from the FHIR Core specification to Terminology.HL7.org (THO). Specifically: the `medicationrequest-category` value set (backed by the `MedicationRequestCategoryCodes` code system), bound to `MedicationStatement.category`, and the `medication-statement-adherence` value set (backed by `MedicationStatementAdherenceCodes`), bound to `MedicationStatement.adherence.code`. Lloyd McKenzie (who imported the original review comment as Comment 301) notes that migration is *not* required, but should be considered if either (a) the value-set definitions are likely to evolve over the coming years, or (b) the codes might be useful in CDA, V2, or other HL7 specifications outside FHIR Core. This ticket is one of a coordinated series of "THO-candidate value sets" reviews Lloyd has filed across many resources — closely related Pharmacy entries include [FHIR-55335 (MedicationRequest)](https://jira.hl7.org/browse/FHIR-55335), [FHIR-55333 (MedicationAdministration)](https://jira.hl7.org/browse/FHIR-55333), [FHIR-55334 (MedicationDispense)](https://jira.hl7.org/browse/FHIR-55334), and [FHIR-55332 (Medication)](https://jira.hl7.org/browse/FHIR-55332) — so the disposition here will likely set the tone for the others. Note that the `medicationrequest-category` value set is shared with `MedicationRequest.category`, so any migration decision must be coordinated with FHIR-55335 to avoid splitting one value set across two homes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate both value sets and their backing code systems from FHIR Core to THO:

1. **`medicationrequest-category` / `MedicationRequestCategoryCodes`** — relocate the CodeSystem and ValueSet definitions to THO (under the Pharmacy stewardship namespace). Retire the FHIR Core definitions; replace the binding on `MedicationStatement.category` (and on `MedicationRequest.category`, in coordination with FHIR-55335) with the new THO canonical URL. Preserve existing canonical URLs via redirects/equivalence so existing implementers continue to resolve the binding correctly.
2. **`medication-statement-adherence` / `MedicationStatementAdherenceCodes`** — relocate to THO under Pharmacy stewardship. Update the binding on `MedicationStatement.adherence.code` to the THO canonical URL. Preserve old URLs.

Coordinate the migration with the THO maintainers and ensure the changes ship together with FHIR-55335 so the shared `medicationrequest-category` value set is not split across two homes.

#### Justification

Both value sets meet the criteria Lloyd lays out for migration: (a) they are pharmacy-domain code lists whose contents are realistically going to evolve as practice and policy change, and (b) at least the category concept (inpatient/outpatient/community/discharge) is exactly the kind of cross-paradigm concept that V2 and CDA implementers can also reuse. Centralizing in THO gives a single, version-managed, cross-product source of truth, reduces FHIR-Core release coupling for terminology updates, and matches the broader stewardship pattern this ticket is part of (FHIR-55063, 55067, 55216, 55220, 55266, 55292, 55332, 55333, 55334, 55335).

---

### Disposition B: Alternative Approach

#### Proposal

Migrate **only** `medicationrequest-category` / `MedicationRequestCategoryCodes` to THO, and **retain** `medication-statement-adherence` / `MedicationStatementAdherenceCodes` in FHIR Core for now.

Rationale for the split:

- `medicationrequest-category` is already shared between `MedicationRequest.category` and `MedicationStatement.category`, encodes generic care-setting concepts (inpatient, outpatient, community, discharge), and is genuinely reusable in V2/CDA. It is the strongest THO candidate.
- `medication-statement-adherence` is tightly coupled to `MedicationStatement.adherence.code` semantics, has no obvious consumer outside FHIR Core today, and its values are unlikely to change frequently. Move it to THO only when a concrete cross-spec reuse case appears.

Update the binding on `MedicationStatement.category` to the new THO URL (in lock-step with FHIR-55335). Leave `MedicationStatement.adherence.code` bound to the existing FHIR Core ValueSet, and revisit at the next major release.

#### Justification

THO migration carries real costs — coordination with the THO maintainers, dual-canonical-URL handling for existing implementers, and ongoing release-train coordination between FHIR Core and THO. Doing it for value sets that genuinely benefit from cross-spec reuse is worthwhile; doing it for value sets that don't is overhead without payoff. This split keeps the high-value migration (the shared category code) and defers the lower-value one until evidence justifies the cost. It also limits blast radius: only one binding change per medication resource, and the adherence code remains a stable, FHIR-Core-managed concept.

---

### Disposition C: Decline

#### Proposal

Do not migrate either value set. Retain both `medicationrequest-category` and `medication-statement-adherence` (with their backing code systems) inside FHIR Core, owned by the Pharmacy work group. Close the ticket as "Persuasive with Mod" → no change, with a note that migration may be reconsidered if a concrete CDA/V2 reuse request emerges or if the definitions are proposed for substantive revision.

#### Justification

Lloyd explicitly states migration is *not required*, only "should be considered" if change is realistic or cross-spec reuse is realistic. Neither value set has demonstrated demand from CDA or V2 implementers in the indexed Zulip / GitHub corpus, and neither has a track record of churn that would justify decoupling the release cadence. Leaving them in FHIR Core keeps the bindings, governance, and release process simple, and avoids the dual-URL/redirect complexity that THO migration imposes on existing implementers. If circumstances change, a future ticket can revisit.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right balance for this ticket. The `medicationrequest-category` value set is already a shared, reusable, care-setting code list — it meets *both* of Lloyd's migration criteria and is the obvious THO candidate, especially because the same value set is also in scope for [FHIR-55335](https://jira.hl7.org/browse/FHIR-55335) on `MedicationRequest.category`. Migrating it once, in lock-step across both tickets, gives Pharmacy a clean, defensible THO migration with real cross-spec payoff. Conversely, `medication-statement-adherence` is purpose-built for `MedicationStatement.adherence.code`, has no surfaced cross-spec consumer, and its concepts are stable; the cost of migrating it (coordination, dual canonicals, release coupling) outweighs the benefit today. Disposition A over-migrates and pays cost without reuse evidence; Disposition C under-migrates and ignores the genuinely-shared category code, leaving the same value set bound from two resources in a place that is harder to reuse from V2/CDA. B picks the migration that earns its keep and defers the one that doesn't — which is exactly the discipline Lloyd's "should be considered" framing asks for.
