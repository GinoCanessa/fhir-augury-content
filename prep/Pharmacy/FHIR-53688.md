## Summary

Craig Newman is asking the Pharmacy work group to clarify, in the
DosageDefinition data type's narrative, the mechanism by which an
individual component of a multi-component dosage is bound to a specific
clinical condition. The example currently lists "different dosages for
different conditions" as a justification for splitting a dose into
multiple `component` entries, but the spec text does not actually tell
the reader how a given `component` is associated with the condition that
it applies to. The reporter speculates that this might be done via an
element on the `component` itself — possibly through the
`Dosage.condition.details` element (or the analogous element on
DosageDefinition) — and is asking that the example be expanded to make
the wiring explicit.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the FHIR Augury index for this key, so the request stands on
its own at this point.

## Details

**Description:**

> In the example reason for multiple components for "different dosages

## Proposed Dispositions

Because no GitHub cross-references were surfaced, the dispositions below

### Disposition A: Accept As Requested

#### Proposal

Update the DosageDefinition data type page so that the example which
cites "different dosages for different conditions" as a reason to use
multiple `component` entries explicitly states the mechanism by which a
component is tied to its condition. Concretely:

- In the explanatory text accompanying the example, add a sentence that
  reads (or equivalent): "When the rationale for multiple components is
  that each component applies to a different clinical condition, the
  binding between the component and the condition is carried on the
  component itself via `Dosage.condition` (e.g.,
  `Dosage.condition.details` for a CodeableConcept reference to the
  condition, or `Dosage.condition.reference` when a Condition resource
  is available)."
- Augment the worked example so at least two `component` entries are
  shown side-by-side, each populated with a different
  `condition.details` value (e.g., one component for "renal impairment"
  and one for "normal renal function"), making the wiring visible at a
  glance.
- No structural change to DosageDefinition is implied — this is a
  narrative + example clarification only.

#### Justification

The reporter is not asking for a new element; he is asking for the
existing element that already does this job to be named in the
example. FHIR's general editorial pattern is that whenever the spec
gives a "use this when…" example, it should also show the modelling
mechanism. The change is low-risk (narrative + example only), aligns
with the data type's existing capability, and matches the priority
("Very Low") and type ("Change Request") on the ticket. There is no
community opposition recorded — Zulip returned no related discussion —
so a clean documentation fix is the path of least friction.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (only) editing the DosageDefinition example, generalize the
fix:

1. Add a short, dedicated subsection on the DosageDefinition page —
   e.g., "Pairing components with conditions, schedules, or routes" —
   that enumerates each justification for splitting a dose into multiple
   components and points to the specific element used to express each
   pairing (`condition` for condition-specific components, `timing` for
   schedule-specific components, `route` / `site` for
   administration-route-specific components, etc.).
2. Cross-link this subsection from each bullet in the existing "reasons
   to use multiple components" list, so the next reader who hits any of
   those bullets immediately learns the corresponding modelling
   mechanism.
3. Optionally, add an invariant or guidance note recommending that when
   `component` entries are present and represent condition-specific
   variants, each component SHOULD populate `Dosage.condition`.

#### Justification

The reporter's confusion is a symptom of a broader documentation gap:
the page lists several reasons for multi-component dosages but does not
explain the modelling mechanism for any of them. Fixing only the
"different conditions" example leaves the same confusion latent for
"different schedules" and "different routes." Disposition B costs only
modestly more editorial effort, prevents follow-up tickets of the same
shape, and gives implementers a single index of "if you split for
reason X, the linkage element is Y." The trade-off is more editorial
work and, if the optional SHOULD invariant is included, a slightly
larger change with potential for further review.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Persuasive with Modifier" or "Not Persuasive" on the
grounds that the existing data type definition already exposes
`Dosage.condition` (and `Dosage.condition.details`) on each `component`,
that the data type page's element table already documents this element,
and that no normative change is being requested. The reporter may be
directed to the element table on the same page rather than expanding
the narrative example.

#### Justification

DosageDefinition's element table already lists `condition` (with
`details` and `reference`) on the per-component path, and a careful
reader can derive the answer by inspecting it. Declining keeps the
narrative compact and avoids over-fitting examples to every plausible
question. The risk is that the next implementer who asks the same
question will file the same ticket again — so this disposition is only
defensible if the work group considers the element table self-documenting.

---

### Recommendation

**Recommended disposition:** A

The reporter is making a small, surgical editorial request against an
example that is demonstrably under-specified, the priority is "Very
Low," and there is no Zulip or GitHub discussion suggesting any
controversy or larger design question hiding behind it. Disposition A
delivers exactly what was asked, costs almost nothing, and removes the
ambiguity for the next reader. Disposition B is appealing as a
"prevent the next ticket" move and would be the right call if the
work group is already planning broader cleanup of the DosageDefinition
page; if so, A's text can be folded into B's subsection without
rework. Disposition C is hard to defend at this stage — the reporter
explicitly read the example, formed a hypothesis about the linkage
element, and asked for confirmation in the narrative, which is exactly
the signal that the example needs the clarification.
