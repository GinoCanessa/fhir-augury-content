## Summary

The ticket reports that the US Prescription Drug Monitoring Program (PDMP)
FHIR Implementation Guide references the SMART Backend Services
Authorization Guide in a way that does not point at the published, released
version on hl7.org. The reporter (Josh Mandel) asks that the citation be
updated to the released SMART Backend Services Authorization Guide URL on
hl7.org, and additionally raises a normative-strength question: should the
PDMP IG mandate SMART Backend Services as the sole authorization mechanism
for PDMP Requestor ↔ PDMP Responder transactions, or should it be
recommended (with room for alternatives) until real-world deployment
experience justifies tightening the constraint? The ticket was originally
imported from a spreadsheet review of the PDMP IG; it was previously
resolved as "Considered for Future Use" because the IG had been withdrawn,
then reopened in November 2023 in connection with PSS-2225 (the relaunched
PDMP on FHIR project). No cross-referenced Jira tickets, Zulip threads, or
GitHub items were located by FhirAugury for this key.

## Details

**Description:**

> In addition to the above, the PDMP transactions between the PDMP
> Requestor and the PDMP Responder need to be secured using the SMART
> Backend Services Authorization Guide as shown in Figure 6 below.

This should be pointing to the released version of the smart back-end
Services authorization guide published on hl7.org. It's worth considering
whether implementers should be constrained to only using this guy, or
whether this should be simply a recommended option. Is there real world
application experience pushing this into production? If not, it would be
good to leave options open and tighten over time.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the PDMP IG security narrative (the section containing Figure 6
covering PDMP Requestor ↔ PDMP Responder transactions) so that the
citation of the SMART Backend Services Authorization Guide resolves to
the **released** version published at
`http://hl7.org/fhir/uv/bulkdata/authorization/index.html` (the SMART
Backend Services Authorization profile of the SMART App Launch /
Bulk Data IGs, currently maintained at
`http://hl7.org/fhir/smart-app-launch/backend-services.html`). Replace
any draft/CI-build links with the canonical published URL and pin to a
specific released version in the references section. Retain the **SHALL**
conformance strength on the security mechanism so PDMP transactions
SHALL be secured using SMART Backend Services as currently written.

#### Justification

The reporter's primary concrete ask is fixing the citation, which is a
straightforward editorial correction the workgroup can apply during the
PSS-2225 relaunch of the PDMP IG. SMART Backend Services has been
balloted, published, and adopted broadly (notably as the foundation of
US Core / Bulk Data deployments and as the security layer mandated by
ONC's Standards Version Advancement Process), so requiring it does not
impose an unproven dependency. Holding the requirement at SHALL gives
PDMP a single, interoperable security baseline and avoids the
fragmentation that would arise if every PDMP responder picked a
bespoke OAuth profile.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the citation as in Disposition A (point at the released
hl7.org-published SMART Backend Services Authorization Guide and pin a
specific version), but **soften the conformance strength** in the PDMP
IG's security section: change the "SHALL be secured using the SMART
Backend Services Authorization Guide" text to a recommendation
("**SHOULD** be secured using SMART Backend Services; alternative
authorization mechanisms MAY be used where agreed bilaterally between
PDMP Requestor and PDMP Responder, provided they meet the security
properties enumerated in §X"). Add an explicit list of the security
properties that any alternative must satisfy (mutual authentication,
client identity attestation, scoped access tokens, audit, etc.) and a
note that the workgroup intends to tighten this to SHALL once
real-world deployment evidence supports it.

#### Justification

This is the disposition the reporter actually proposed in the second
half of the ticket: leave options open until production deployment
experience justifies a hard constraint. PDMP transactions involve a
heterogeneous set of state-run responders, several of which have
pre-existing federated authorization arrangements that pre-date SMART
Backend Services; mandating SMART Backend Services on day one risks
non-conformance for otherwise-secure deployments. Capturing the
required security *properties* (rather than a single mechanism) is
consistent with how other HL7 IGs (e.g., IHE-aligned profiles) have
handled multi-jurisdictional security baselines, and the explicit
"intend to tighten" note signals the migration path to implementers.

---

### Disposition C: Decline

#### Proposal

Decline the change. Keep the existing reference and the existing SHALL
constraint in the PDMP IG as authored, on the grounds that (a) the
citation will be normalized to the released hl7.org version
automatically when the IG is re-published under PSS-2225 (so a
ticket-driven edit is redundant), and (b) the conformance-strength
question was already adjudicated when the original PDMP IG was
balloted; reopening it would re-litigate a settled design decision
without new substantive input.

#### Justification

The ticket has been dormant since 2020, was previously closed as
"Considered for Future Use" in January 2023, and was reopened only as
an administrative consequence of PSS-2225 reactivating the IG. No new
implementer feedback, no Zulip discussion, and no GitHub activity
linked to FHIR-29489 were surfaced by FhirAugury, suggesting the
community has not raised new concerns that would justify revisiting
the SHALL/SHOULD design. Editorial citation hygiene is normal IG
republication work and does not require a tracked change request.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most faithful reading of the reporter's actual
request: he asked for both an editorial fix *and* a substantive
reconsideration of whether SMART Backend Services should be the sole
permitted authorization mechanism. Because PSS-2225 represents a
relaunch of the PDMP IG, the workgroup has a one-time opportunity to
set the conformance strength deliberately rather than inheriting it by
default. Real-world PDMP deployments today span state agencies with
heterogeneous existing authorization stacks; locking the IG to SHALL on
SMART Backend Services before the relaunched IG has any production
implementations risks shipping a guide that is technically correct but
operationally non-conformant in practice. Disposition B preserves a
strong default (recommend SMART Backend Services, cite the released
version), enumerates the security properties any alternative must meet
so interoperability is not lost, and explicitly stages the intended
tightening to SHALL in a future version once the workgroup has
deployment evidence — exactly the "tighten over time" path the reporter
suggested. Disposition A is acceptable but forecloses the substantive
question; Disposition C ignores it.
