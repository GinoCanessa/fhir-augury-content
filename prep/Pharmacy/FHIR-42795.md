## Summary

This ticket, raised out of Connectathon 34 discussion of the US Prescription
Drug Monitoring Program (PDMP) FHIR IG, asks whether a PDMP query should
allow the requester to enumerate "other states" to be searched as part of
the query payload itself. The reporter's framing is that most states already
share data with other states subject to data-sharing agreements, and that
including a per-request list of states to search does not appear necessary
or appropriate. The ticket was filed to validate that observation with the
Pharmacy work group before the PDMP IG locked in its request profile.
Subsequent Pharmacy tech-call discussion (9/25, 9/27, 10/2/2023) and a 2024
follow-up confirmed the position: inter-state access is controlled at the
hub/network level via pre-registered organizational entitlements and
state-to-state agreements, not on a per-transaction basis, so the IG should
not carry a "states to query" parameter. The ticket was resolved
**Not Persuasive** (i.e., the request to add or retain such a parameter is
declined) on 2023-09-27, and the resolution text was finalized accordingly.
No cross-referenced Jira, Zulip, or GitHub items were surfaced by the
augury index for this key.

## Details

>
> Multiple State query
> - Most states will include other states as allowed by data sharing agreements
> - Does NOT appear necessary (or appropriate) to include "states to search" in query
>

**Resolution Description:**

> Organizations (e.g., pharmacies, hospitals) register for inter-state
> access. Subsequent queries from the organization includes the
> pre-registered states. A state-to-state agreement must also exist.
>
> There is no need for "which states to query" in the request.

**Comments:**

  healthcare organizations are permitted to make inter-state requests; each
  org has to be vetted for the inter-state requests they want to perform.
  This is built at the hub level, not the transactional level. So there is
  no apparent need to include "states to query" in the query. Mail-order
  dispenses are reported to the state where the patient will receive the
  dispense.
- **smrobertson (2023-09-27)** — 9/27/2023 PDMP call: hubs are central to
  inter-state access. Some current queries do include the states to query,

  be queried".
- **smrobertson (2023-10-02)** — 10/2/2023 Tech Call: discussed in regard
  to PDMP request and response content; confirmed that "state selection" is
  not advisable in individual requests.
- **smrobertson (2024-09-16)** — In further discussion with the
  stakeholders, the states to query are set up when the requester is
  registered with the PDMP.

## Proposed Dispositions

The ticket is already in `Triaged` status with resolution `Not Persuasive`,
reflecting the Pharmacy tech-call consensus that the PDMP IG should *not*
expose a per-request "states to query" parameter. The dispositions below are
phrased so the work group can confirm, soften, or reverse that resolution
on review.

### Disposition A: Accept As Requested

#### Proposal

Confirm and ratify the existing **Not Persuasive** resolution. Specifically:
the US PDMP IG's query/request profile (the PDMP $query-style operation
and any associated request bundle / parameters resource defined under the
US PDMP IG) MUST NOT define an element, search parameter, or `Parameters`
entry that lets a caller enumerate the set of "states to search" on a
per-transaction basis. Where prior drafts or examples included such a
field (e.g., a `state[]` / `requestedState` parameter on the request), it
should be removed from the operation definition and from all conformance
examples. Add a short narrative section to the IG's "Query Design" /
"Inter-state Access" page stating that inter-state scope is determined by
(a) the requester organization's pre-registered entitlements at the
hub/network and (b) state-to-state data-sharing agreements, and that
per-request state selection is explicitly out of scope.

#### Justification

This matches the resolution text already attached to the ticket and the
three Pharmacy tech-call discussions from late September / early October
2023 plus the September 2024 stakeholder follow-up, all of which
independently confirmed that hubs (not requesters) control inter-state
access and that newer integrations are discouraged from carrying state
selectors in the request. Removing the parameter prevents implementers
from constructing requests that *appear* to scope inter-state access
client-side while in fact being silently re-scoped by the hub, which is a
real-world source of audit-log confusion for PDMPs. It also aligns the IG
with the operational reality that mail-order dispenses are routed to the
state of receipt regardless of the requester's stated state list, so a
client-supplied list would be misleading rather than authoritative.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the resolution as **Not Persuasive** for the *required* request
shape, but add an **optional, advisory** mechanism — most cleanly a
standard FHIR extension on the request `Parameters` resource (e.g.,
`pdmp-requested-state-scope`, valued as a list of US 2-letter state
codes) — that a requester MAY include to declare *which* of their
already-entitled states they want included in this particular query. The
hub MUST treat the extension as advisory only: it may *narrow* the query
to a subset of states the organization is already entitled to, but it
MUST NOT be used to *expand* scope, and the hub MUST ignore any state in
the extension for which the organization is not pre-registered. Document
the extension under the IG's extension pack with a "MAY ignore"
conformance statement and an example showing a hub silently restricting
the response to entitled states.

#### Justification

Some live integrations today *do* carry a state list in the request
(noted on the 9/27/2023 call: "currently, some queries include the
states to query"). A clean break that simply forbids the field forces
those integrations to either drop the data or migrate to a non-standard
extension of their own. An IG-defined optional, advisory extension
preserves the resolution's central principle (entitlement and inter-state
scope are configured at the hub, not at the transaction) while giving
existing implementers a standards-conformant migration path and giving
clinicians a way to express intent (e.g., "only my home state today")
without violating the hub's entitlement model. The trade-off versus
Disposition A is added surface area in the IG and the risk that
implementers misread "advisory" as "authoritative"; the conformance
language and examples are critical to mitigate that.

---

### Disposition C: Decline

#### Proposal

Reverse the current `Not Persuasive` resolution and treat the ticket as
**Persuasive**: define `requestedState` (0..* `code`, bound to a value
set of US state and territory codes) as a first-class parameter on the
PDMP query operation, with conformance text requiring hubs to honor it
when the requester organization is entitled to those states.

#### Justification

This disposition is presented for completeness so the work group can
explicitly weigh it, but it is hard to defend on the record before the
group. The Pharmacy tech-call discussions and the 2024 stakeholder
follow-up uniformly land on the position that inter-state scope is a
configuration concern, not a transactional concern, and that allowing
clients to declare states per request invites misalignment between the
request's apparent scope and the response the hub is actually able to
return (with downstream audit and consent-tracking implications). The
only argument in favor is preserving compatibility with legacy
integrations that already populate a state list — and that need is
better served by the optional extension proposed in Disposition B than
by promoting the field to a normative request parameter.

---

### Recommendation

**Recommended disposition:** A

The Pharmacy work group has already converged on this answer across
three 2023 tech calls and confirmed it again in 2024 with stakeholders,
and the operational rationale (hub-controlled entitlement, state-to-state
agreements, mail-order routing to state-of-receipt) is independently
sound. Disposition A simply ratifies that record and tightens the IG so
implementers cannot accidentally construct misleading requests.
Disposition B is worth keeping in reserve **only** if review surfaces
concrete live integrations whose migration would be blocked by removing
the field outright; if no such integrations are named on the call, the
extra extension is unnecessary surface area. Disposition C should be
declined for the reasons above.
