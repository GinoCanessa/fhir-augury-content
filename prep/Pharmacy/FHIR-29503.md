## Summary

This ticket is a Pharmacy work-group change request originally raised against
the US Prescription Drug Monitoring Program (PDMP) FHIR Implementation Guide
(carried under the umbrella "US Medication FHIR Implementation Guide"
specification entry in Jira). The reporter notes that **NCPDP RxFill** is the
established mechanism used to convey whether a prescription has been picked up
or otherwise dispensed/not-dispensed (i.e., the "purpose" of detecting "Not
Taken" medications). The existing IG wording instead points readers to the
core FHIR `MedicationStatement` Resource Notes section for guidance on
determining whether the patient has taken a medication. The ticket asks the
work group to consider replacing or supplementing that pointer with guidance
that references the NCPDP RxFill transaction, which is the real-world data
source PDMP implementers rely on to know that a dispense did or did not
occur.

The Augury cross-reference index returns **no** related Jira tickets, Zulip
threads, or GitHub items for FHIR-29503, so this Summary is built solely
from the ticket itself and its comment history. The 2023-11-28 reopen by
`smrobertson` ties the issue to **PSS-2225 PDMP on FHIR**, indicating it
should be re-considered when the PDMP IG project is restarted.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the US PDMP FHIR Implementation Guide narrative that currently reads:

> "Not Taken" Medications — For specific guidance on how to determine if the
> patient has taken the medication, see the MedicationStatement Resource
> Notes section in the FHIR Specification.

so that it instead (or additionally) directs implementers to the **NCPDP
SCRIPT RxFill** transaction as the authoritative signal that a prescription
was, or was not, dispensed/picked up. The new wording should:

1. Explain that, in the U.S. pharmacy ecosystem, "not taken" at the PDMP
   layer is operationalized as "not dispensed/not picked up," and that this
   is conveyed via NCPDP RxFill notifications between pharmacy and
   prescriber systems.
2. Map the relevant RxFill outcomes onto the FHIR resources the PDMP IG
   already uses — primarily `MedicationDispense.status` (e.g.,
   `not-taken`/`cancelled`/`declined`) rather than `MedicationStatement` —
   and provide a short example or table.
3. Retain a secondary pointer to `MedicationStatement` only for cases where
   patient- or provider-asserted non-adherence (outside the dispense
   workflow) is in scope.

#### Justification

The reporter's underlying point is correct: in U.S. PDMP workflows, whether
a medication was "taken" is inferred almost entirely from dispense activity,
which is communicated via NCPDP RxFill, not from a separately-asserted
`MedicationStatement`. Pointing implementers at `MedicationStatement` for
this question is misleading because PDMP data feeds rarely contain
patient-asserted statements; they contain dispense events. Re-anchoring the
guidance on RxFill / `MedicationDispense` aligns the IG with how the data
actually flows and is consistent with HL7 Pharmacy's general guidance that
`MedicationDispense` is the right resource for pharmacy-system-sourced
dispense facts.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than rewriting the "Not Taken" Medications paragraph in place, add a
new, dedicated **"Determining Non-Dispense / Non-Pickup"** sub-section to
the PDMP IG that:

- Explicitly names NCPDP RxFill as the source-of-truth transaction for
  pharmacy-side non-pickup events.
- Defines a small profile or extension on `MedicationDispense` (or reuses
  an existing US Core / US Medication IG profile) that constrains
  `status` and `statusReason` value sets to the RxFill-derived codes the
  PDMP community actually exchanges.
- Leaves the existing `MedicationStatement` cross-reference intact for the
  narrower "patient-reported non-adherence" case, with a clear
  signpost telling the reader which sub-section applies to which data
  source.

#### Justification

FHIR-29503's text is a one-line spreadsheet comment, and we know from the
2023-11-28 history that the PDMP IG was withdrawn and is only now being
restarted under PSS-2225. A dedicated sub-section is more durable than an
in-line edit: it lets the work group expose the RxFill ↔
`MedicationDispense.status`/`statusReason` mapping explicitly (including
value-set guidance), which is what implementers will actually need, while
preserving the existing `MedicationStatement` pointer for the legitimate
patient-asserted-non-adherence case. This approach also avoids overloading
`MedicationStatement` semantics in a US pharmacy context where almost all
data comes from dispense systems.

---

### Disposition C: Decline

#### Proposal

Close FHIR-29503 as **Considered — No Change** (or **Persuasive with Mod /
Not Persuasive** depending on the work group's preferred disposition
vocabulary), on the grounds that:

- The US PDMP FHIR IG was withdrawn in 2023 and the spreadsheet-imported
  comment is too terse to drive a specific normative change on its own.
- General guidance on dispense-vs-taken semantics is already covered by
  the core FHIR `MedicationDispense` and `MedicationStatement` resource
  pages, and a U.S.-specific RxFill mapping is better handled by the
  refreshed PDMP-on-FHIR project (PSS-2225) as part of its own design
  rather than as a back-fill change to a withdrawn IG.

#### Justification

This is one of a batch of "PDMP-old-issues"-labeled tickets imported from a
spreadsheet against an IG that has since been withdrawn. The ticket as
written does not propose specific replacement wording, value sets, or
profile changes; acting on it without the PDMP-on-FHIR project re-engaging
risks producing guidance that the new project then has to undo. Declining
(while noting the substantive concern is valid and should be carried into
PSS-2225's design backlog) is defensible on scope grounds.

---

### Recommendation

**Recommended disposition:** B

The reporter's substantive point is correct and should not be lost — RxFill
*is* the U.S. pharmacy mechanism that signals non-pickup, and the current
pointer to `MedicationStatement` Resource Notes does mislead PDMP
implementers. Disposition A captures that, but a one-paragraph in-line
rewrite is fragile and risks blurring `MedicationStatement` (patient
assertion) and `MedicationDispense` (pharmacy event) semantics. Disposition
C is the safest procedural answer but loses real implementer value.
Disposition B is the strongest middle path: it preserves the existing
`MedicationStatement` reference for patient-asserted non-adherence, adds a
dedicated RxFill → `MedicationDispense.status`/`statusReason` sub-section
that future PDMP-on-FHIR balloters can extend with concrete value-set
bindings, and gives PSS-2225 a clean place to land its work. The work
group should adopt B and explicitly route the detailed value-set work into
the PSS-2225 project plan.
