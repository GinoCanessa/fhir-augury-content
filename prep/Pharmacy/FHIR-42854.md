## Summary

The ticket, filed from Connectathon 34, requests that the US PDMP FHIR
Implementation Guide support conveying "Alerts" in the PDMP response to a
prescription-history query. Alerts may originate from either a state PDMP or
an Integrator acting on behalf of a state, and are associated with the
subject Patient (who is consolidated by the PDMP via probabilistic matching
across pharmacy-sourced data). The reporter notes that PMIX/NIEM already
defines an enumerated alert list (PMIX §4.1.2.3) that includes an "other"
entry, so the FHIR representation must support both the enumerated codes and
free-text, and should accommodate content such as "look in the portal"
pointers. The reporter flagged the `Flag` resource as a promising candidate
mechanism. No cross-referenced Jira tickets, Zulip threads, or GitHub items
were surfaced by FhirAugury for this ticket, so the request stands on its
own within the Pharmacy workstream.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add first-class support for PDMP "Alerts" to the PDMP history response by
defining an `Alert` slice on the `history-response-bundle` whose entries are
`DetectedIssue` instances. Bind `DetectedIssue.code` to a value set built
from the PMIX §4.1.2.3 enumerated alert codes (including the "other" code),
with binding strength `extensible` so that states can convey free-text /
narrative detail via `DetectedIssue.detail` (and/or `DetectedIssue.code.text`
when "other" is used). Reference the subject Patient via
`DetectedIssue.patient`, and allow an `Alert`-to-portal pointer to be
carried as a `DetectedIssue.reference` (or an extension) so "look in the
portal" alerts have a canonical place to live. Add a narrative note in the
IG describing how Integrators-acting-on-behalf-of-a-state should populate
these alerts, and how the PMIX Alert codes map into `DetectedIssue`.

#### Justification

This is what the reporter ultimately asked for: a structured but
free-text-tolerant alert channel on the PDMP response, aligned with PMIX's
existing enumerated list. `DetectedIssue` is already the FHIR resource for
"something the responder wants the requester to be aware of about this
patient's medication picture," and it naturally carries a code plus
narrative detail plus a patient subject — matching the ticket's three
requirements (enumerated code, free-text/"other", patient association).
Using an extensible binding preserves PMIX fidelity while letting states
that are "still figuring it out" convey alerts they cannot yet code. The
ticket's own recorded resolution ("Persuasive with Modification",
incorporated prior to the MAY2024 ballot) confirms this is the path the
work group actually took.

---

### Disposition B: Alternative Approach

#### Proposal

Use the `Flag` resource (as the reporter initially suggested) rather than
`DetectedIssue` for the Alert slice: define a PDMP `Alert` profile on `Flag`
in the IG, bind `Flag.code` to the PMIX alert value set with an extensible
binding, require `Flag.subject` = the consolidated PDMP Patient, and use
`Flag.category` to distinguish PDMP-sourced alerts from Integrator-sourced
alerts. Carry portal-pointer alerts via `Flag.code.text` plus a
`Flag`-level extension (or a contained `DocumentReference`) for the URL.
Slice the `history-response-bundle` on `Flag` instead of `DetectedIssue`.

#### Justification

`Flag` is purpose-built in FHIR for "prospective warnings about a patient
that should be visible to clinicians," which is arguably a closer semantic
match to a PDMP alert than `DetectedIssue`, whose canonical use is
identification of a specific clinical/medication issue detected during an
analysis. `Flag` also has stronger built-in support for lifecycle
(`status`, `period`) and for categorization, both of which matter when a
state-run PDMP wants to distinguish persistent patient-level warnings from
transient response-scoped annotations. The trade-off versus Disposition A
is that `DetectedIssue` more naturally carries "evidence" / "detail"
structures that the PMIX alert set sometimes implies (e.g., multi-provider
/ multi-pharmacy signals), and the workgroup has already invested in
`DetectedIssue` tooling inside the IG.

---

### Disposition C: Decline

#### Proposal

Decline to add Alerts to the PDMP response at this revision of the IG, on
the grounds that PMIX alerts are "still being figured out by the states",
that taxonomy-coded alerts are not currently carried in the PMIX message
flow, and that "look in the portal" behavior is already achievable via
existing Bundle narrative / out-of-band mechanisms. Defer a formal alert
structure to a future ballot cycle once the state-side semantics stabilize.

#### Justification

Standardizing a FHIR structure on top of a source system whose own
semantics are in flux risks locking the IG into a shape that will need
breaking changes as states converge. Declining preserves optionality and
avoids forcing implementers to re-code alert handling twice.

---

### Recommendation

**Recommended disposition:** A

Disposition A aligns with what the workgroup actually did: the ticket is
recorded as "Persuasive with Modification" with a resolution that adds an
`Alert` slice on `history-response-bundle` backed by `DetectedIssue`, with
PMIX Alert codes bound in the IG and a clarifying usage note. That path
directly satisfies the three explicit requirements in the ticket (PMIX
enumerated codes, "other"/free-text, patient association), lets Integrators
populate alerts on behalf of states, and gives "look in the portal" alerts a
stable home via `DetectedIssue.detail` / `reference` without inventing a
new resource. Disposition B (Flag) is a defensible alternative on semantic
grounds, but it would displace the work already in the IG and split tooling
between two resource types; Disposition C understates how concretely PMIX
§4.1.2.3 is already defined and would leave PDMP responses without any
structured alert channel at all. Accept as requested, with the modifications
captured in the resolution.
