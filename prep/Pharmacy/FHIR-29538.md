## Summary

[FHIR-29538](https://jira.hl7.org/browse/FHIR-29538) is a 2020 ballot
question against the **US Prescription Drug Monitoring Program (PDMP)
FHIR Implementation Guide** (a sub-IG / draft companion of the US
Medication FHIR IG). The reporter, Suzanne Gonzales-Webb, observes that
the IG's "last bullet" of some (unidentified) section refers to a
**"Private Key"**, but the term *Private Key* does not appear anywhere
else in the document — there is no glossary entry, no preceding
discussion of key material or PKI, and no description of how the key
is generated, exchanged, used, rotated, or protected. The ticket asks
the workgroup to clarify what "Private Key" means in that context.

The ticket was originally dispositioned **"Considered for Future Use"**
on 2023-01-18 (vote 4-0-0, Hatem/McNeil) on the basis that the PDMP IG
had been withdrawn and would only be revisited if the project sponsor
or workgroup picked it back up. The disposition was then **reverted on
2023-11-28** by smrobertson and the ticket re-opened, tied to
project-services entry **PSS-2225 PDMP on FHIR**, which signaled that
work on PDMP on FHIR was being restarted. As of 2024-03-19 the ticket
sits in **Triaged / Unresolved** with the historical `PDMP-old-issues`
label.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next published version of the **US PDMP FHIR Implementation
Guide** (the PDMP-on-FHIR work being revived under PSS-2225, hosted
under the Pharmacy WG), locate the bullet that introduces the term
**"Private Key"** and either:

1. Replace the bare phrase "Private Key" with a fully specified
   reference — for example, *"the SMART Backend Services client's
   asymmetric **private signing key**, paired with the public key
   published in the client's JWKS and used to sign the
   `client_assertion` JWT per
   [SMART Backend Services Authorization](https://hl7.org/fhir/uv/bulkdata/authorization/index.html)
   §"Registering a SMART Backend Service"*; **and**
2. Add a short **Glossary / Terminology** entry (or a sentence in the
   IG's Security section) defining "Private Key" in the same terms,
   so the term is no longer first-and-only-mentioned in a bullet.

If the underlying intent of that bullet was something other than SMART
Backend Services key material (e.g., PDMP-state-issued credentials),
the bullet should be rewritten to name the specific key system and
cite the governing specification.

#### Justification

The ticket is, on its face, a documentation defect: a security-
relevant term appears once, in a bullet, with no definition, no
referent, and no cross-reference to any standard. That is the textbook
case for "clarify in place + add to glossary," and it is consistent
with how FHIR core, US Core, and SMART IGs handle similar terms (key,
JWK, JWKS, `client_assertion`). Tying the wording to **SMART Backend
Services** is also consistent with sibling
[FHIR-29516](https://jira.hl7.org/browse/FHIR-29516), which already
asks the PDMP IG to harden its backend-services conformance language;
the two should be reconciled together so the spec doesn't end up
naming "Private Key" in one section and "JWK" / "client assertion
signing key" in another.

---

### Disposition B: Alternative Approach — Defer-and-Bundle Pending PSS-2225 Restart

#### Proposal

Rather than edit the withdrawn 2020 PDMP IG text in isolation, leave
FHIR-29538 in **Triaged**, attach it to the active **PSS-2225 PDMP on
FHIR** project, and treat it as an input requirement for the
**security/authentication** chapter of the next-revision PDMP IG. As
part of that chapter rewrite, the project must (a) standardize on
**SMART Backend Services** (or whatever auth scheme the revived
project ultimately adopts) and define the term "Private Key" exactly
once, in the security overview, and (b) batch-resolve all
`PDMP-old-issues`-tagged tickets that touch the same chapter
(including FHIR-29516, FHIR-29478, FHIR-29479, FHIR-29480) in a single
coordinated disposition pass.

Mechanically: do not "Persuasive / change applied" this ticket alone.
Mark it "Considered — for next revision" linked to PSS-2225, and
record an explicit acceptance criterion that the next revision's
Security section cannot ship with an undefined "Private Key" mention.

#### Justification

The original 2023-01-18 disposition ("Considered for Future Use, IG
withdrawn") was reverted in November 2023 specifically because
PSS-2225 picked the work back up. That suggests the workgroup's real
state is *"we will fix this, but we will fix it in the new IG, not in
the withdrawn one."* Patching only the bullet (Disposition A) without
also reconciling the broader security narrative risks producing a
glossary entry that the next revision then has to immediately rewrite.
Bundling all `PDMP-old-issues` security/auth tickets together gives
the revived project a clean, internally consistent security chapter
on first publish, and avoids whack-a-mole disposition work for the
WG.

---

### Disposition C: Decline (Not Persuasive — Withdrawn IG)

#### Proposal

Close FHIR-29538 as **Not Persuasive — withdrawn IG, no longer
maintained** with an explicit note that the US Prescription Drug
Monitoring Program (PDMP) FHIR IG was withdrawn and that ballot
comments raised against the withdrawn version will not be
individually applied. If and when a successor IG is balloted under
PSS-2225, the successor will be reviewed afresh against current
SMART/security guidance, at which point any residual "undefined
Private Key" defect will be caught by the new ballot cycle rather
than carried forward as an inherited resolution.

#### Justification

This is essentially the position the workgroup took on 2023-01-18
(4-0-0). The basis is defensible: there is no current published
artifact for the change to land in; the ticket is more than four
years old; the underlying complaint is a one-sentence documentation
clarification with no standing implementer impact (no implementer is
shipping against the withdrawn IG); and any successor IG will have to
re-justify its security text from scratch, so a 2020 disposition has
limited durable value. Declining also keeps the WG's `PDMP-old-issues`
backlog from inflating the apparent scope of PSS-2225 with paperwork
items that the new project will redo anyway.

The risk of declining is symbolic — it signals to the original
reporter that the comment was not actionable — and the cost is small
because PSS-2225 review will catch the same defect again if the
successor IG repeats the wording.

---

### Recommendation

**Recommended disposition: B (Defer-and-Bundle Pending PSS-2225
Restart).**

The November 2023 revert is the most informative signal in this
ticket's history: the workgroup explicitly decided this comment is no
longer "withdrawn IG, ignore" but is instead an input to the revived
PDMP-on-FHIR project. Disposition A would technically resolve the
text defect, but it does so against a withdrawn document and risks
being thrown away by the next revision; Disposition C would
re-impose the very stance that smrobertson reverted in 2023, which
the WG should not silently re-adopt without a fresh discussion of
PSS-2225's status. Disposition B respects the revert, keeps the
ticket actionable, and — by bundling FHIR-29538 with FHIR-29516 and
the rest of the `PDMP-old-issues` security/auth cluster — gives the
next-revision authors a single, coherent security chapter to draft
rather than a sequence of one-bullet patches.

If the WG concludes during discussion that PSS-2225 is itself
dormant, the right fallback is C (close as not-persuasive on the
withdrawn IG); A should only be selected if the WG intends to publish
an erratum / clarification against the existing withdrawn document,
which is unusual.
