## Summary

The reporter asserts that in "example 8" of the FHIR Core specification
(Pharmacy work group, almost certainly a `MedicationRequest` /
`MedicationDispense` / `MedicationAdministration` `dosageInstruction`
example whose `Dosage.timing.repeat` carries a `startOffset` element),
the value of `startOffset` is incorrect: it is currently shown as `8`
but, per the surrounding narrative of the example, it should be `9 days`.
The request is a small editorial correction — change the literal value
from `8` to `9` in example 8. There are no cross-referenced Jira
tickets, Zulip threads, or GitHub items linked to FHIR-54341, so the
scope is limited to whatever instance(s) of "example 8" the Pharmacy WG
identifies during triage. A single comment on the ticket flags it as a
technical correction (TC) on an example that was voted on as negative,
which means the workgroup needs to formally dispose of it as a ballot
comment rather than treat it as a free-form editorial fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the editorial/technical correction exactly as the reporter asks:
in the Pharmacy module's "example 8" (the `Dosage`-bearing example —
most likely a `MedicationRequest` example whose
`dosageInstruction.timing.repeat.startOffset` is presently encoded with
the value `8`), change the `startOffset` value from `8` to `9` (interpreted
as 9 days, consistent with the narrative of the example). Update both
the XML and JSON serializations of the example, plus any rendered
narrative or summary table that quotes the value, so the example is
internally consistent. Mark the ticket as a Technical Correction on the
ballot and resolve the negative vote as Persuasive.

#### Justification

The reporter and the commenter both characterise this as a Technical
Correction tied to a negative ballot vote on an example. The fix is
local to a single example file and does not change normative content,
schema, or invariants. Accepting as requested is the lowest-risk
resolution that satisfies the negative voter, removes the inconsistency
between the example value and its surrounding narrative, and avoids
leaving an erroneous example in the published spec — a class of defect
that disproportionately confuses implementers because they tend to
copy examples verbatim. There is no Zulip or cross-ticket discussion
suggesting any other interpretation of the intended value.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this as a prompt to audit the *narrative* of example 8 rather
than the data, and reconcile the two in whichever direction the
example's clinical scenario actually requires. Concretely: before
changing `startOffset` from `8` to `9`, have the Pharmacy WG verify the
intent of the example (the dosing schedule it is meant to illustrate)
and then update either (a) the `startOffset` value to `9`, **or**
(b) the surrounding narrative/comment to say "8 days", whichever
preserves the originally-intended teaching point. If audit shows the
example is meant to demonstrate a different `Timing.repeat` pattern
entirely (e.g., `boundsDuration` plus `frequency`/`period` rather than
`startOffset`), refactor the example to use the more idiomatic pattern
and add a one-line comment explaining what the example illustrates.

#### Justification

`startOffset` on `Timing.repeat` is a relatively uncommon element and
is easy to misuse in examples — it offsets the *event* from the start
of the timing period, which is not the same as "the schedule starts on
day N". A bare `8 → 9` swap fixes the symptom (numeric mismatch with
narrative) but does not confirm that the example is actually
illustrating a clinically and semantically valid use of `startOffset`.
This disposition costs slightly more WG time than Disposition A but
hardens an example that has already drawn a negative ballot vote, and
it lets the WG decide whether the underlying narrative or the data was
the one in error. Backward compatibility is unaffected either way
because only example content changes.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Not Persuasive** on the grounds that no
defect has been substantiated: the reporter has not cited where in
example 8 the discrepancy arises, has not shown that "9 days" is the
intended value (versus the existing `8` being correct and the
narrative being what should change), and there is no Zulip
discussion, linked artifact, or cross-referenced ticket establishing
the intended schedule. Close without changing the example.

#### Justification

The ticket body is two sentences and provides no quotation of the
narrative, no link to the specific example file, and no clinical
rationale for `9` over `8`. In principle, ballot comments require
enough specificity for the workgroup to verify the defect and choose a
fix; this one does not. Declining is defensible as a procedural matter,
particularly if the WG cannot independently confirm the intent of
example 8 from its source.

---

### Recommendation

**Recommended disposition:** A

The commenter has already classified this as a Technical Correction on
an example tied to a negative ballot vote, the requested change is
purely editorial (a single literal value in one example), and the cost
of leaving an internally-inconsistent example in the published spec
substantially exceeds the cost of the fix. Disposition B is sound
hygiene and the WG should still take a brief look at the example's
intent during application of the change, but there is no need to
escalate the resolution itself beyond "accept" — a one-character data
change is the right shape for a TC. Disposition C is hard to defend
when a balloter has gone to the trouble of pointing to a specific
example value and another WG member has corroborated that it is a
TC; declining would invite the negative vote to persist into the next
ballot cycle.
