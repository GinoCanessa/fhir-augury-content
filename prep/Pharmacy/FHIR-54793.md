## Summary

The reporter notes an asymmetry between the `Dosage` and `DosageDetail`
datatypes around patient-facing instructions. In `Dosage`, two
audience-specific elements exist — `Dosage.additionalInstruction`
(`CodeableConcept`, codified additional guidance such as "with food")
and `Dosage.patientInstruction` (free-text instructions targeted at the
patient). The newer `DosageDetail` datatype currently exposes only
`renderedInstruction`, which is audience-neutral and intended to carry a
human-readable rendering of the structured dose for any consumer
(clinician, pharmacist, patient). The ticket proposes adding a
`patientInstruction` element on `DosageDetail` so that patient-targeted
text can again be carried alongside the structured dose without
overloading `renderedInstruction`. The single workgroup comment so far
(Costa Teixeira) asks the submitter to clarify *which renderings are
needed where* — i.e., to articulate the audience model for
`DosageDetail` before adding new fields.

There are no formal Jira links and no cross-references recorded in the
Augury index, but several adjacent Pharmacy / CDS tickets opened in the
same R6 ballot window touch the same `DosageDetail` redesign and are
relevant background:

- [FHIR-53721](https://jira.hl7.org/browse/FHIR-53721) — proposal that
  `ActivityDefinition.dosage` change from `0..* Dosage` to
  `0..1 DosageDetail` to match the new datatype split.
- [FHIR-54597](https://jira.hl7.org/browse/FHIR-54597) — proposal to add
  an explicit sequence number to `DosageDetail.step` to avoid
  list-order safety risks.
- [FHIR-54990](https://jira.hl7.org/browse/FHIR-54990) /
  [FHIR-54991](https://jira.hl7.org/browse/FHIR-54991) — questions about
  whether `MedicationRequest` / `MedicationStatement` still need their
  own `route`/`method` elements now that `DosageDetail` carries them.

Together these tickets indicate that the `Dosage`→`DosageDetail` /
`DosageSafety` / `DosageCondition` split is still being shaken out in
ballot, and that several elements that lived on legacy `Dosage` need
explicit decisions about whether they are reproduced, replaced, or
dropped on `DosageDetail`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new element `DosageDetail.patientInstruction` of type `string`,
cardinality `0..1`, summary `false`, with a definition mirroring
`Dosage.patientInstruction` ("Free-text instructions for the patient,
explaining the dose in language they can act on"). Implement by adding
the element row to `source/datatypes/dosagedetails.xml`
(Excel-spreadsheet authoring source) and updating
`source/dosage-definitions.html` and `source/dosage.html` so the
narrative explicitly contrasts the three instruction-bearing elements:
`renderedInstruction` (audience-neutral, full structured-dose
rendering), `patientInstruction` (free-text patient-facing), and the
existing `additionalInstruction` on `Dosage` (codified additional
guidance such as "with food"). At least one example under
`source/medicationrequest/` should be extended to populate the new
`patientInstruction` so reviewers can see the intended usage.

#### Justification

This is the simplest fix and restores parity with the legacy `Dosage`
datatype, where `patientInstruction` has shipped for several FHIR
versions and is in active use by implementers (notably Swedish
ePrescription per the reporter, and broadly elsewhere). Carrying
patient-facing free text inside `renderedInstruction` is undesirable
because `renderedInstruction` is the canonical full rendering of the
*structured* dose for any audience — overloading it with patient-only
prose loses round-trip fidelity and confuses non-patient consumers
(pharmacist verification systems, EHR clinician views). Adding the
element as `0..1 string` is fully backward-compatible (additive) and
costs implementers nothing if they don't populate it. The change also
lets profiles (e.g., US-Core, IPS, national medication IGs) constrain
or require the element where patient-facing text is mandatory without
needing an extension for what is clearly a core concept.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding a new sibling element, model patient-targeted
rendering as an audience-tagged variant of `renderedInstruction`. Two
concrete shapes are possible:

- **B1 (preferred sub-option):** Change `DosageDetail.renderedInstruction`
  cardinality from `0..1 string` to `0..* (BackboneElement)` with two
  children — `audience` (`code`, bound to a small value set such as
  `clinician | patient | pharmacist | other`) and `text` (`string`).
  This lets implementers carry one rendering per audience (and per
  language, if combined with the existing translation extension) without
  inventing additional named elements every time a new audience is
  identified.
- **B2 (lighter-weight alternative):** Keep `renderedInstruction` as
  `string` but bind a standard `audience` extension (already used in
  several IGs) and document its use on the `DosageDetail` page. This
  would be authored under `source/dosage-definitions.html` /
  `source/dosage.html` and the extension would live in
  `HL7/fhir-extensions` per the FhirCore playbook's "extensions removed
  from core" pattern.

In addition, document on the `Dosage`/`DosageDetail` narrative page
(addressing Costa Teixeira's comment) which audience each existing
element is intended for, so that implementers do not have to infer.

#### Justification

The reporter is solving one instance of a more general problem — the
same need will recur for clinician summaries, pharmacist verification
text, and translations into multiple languages. Adding named
audience-specific siblings (`patientInstruction`, then later
`pharmacistInstruction`, `clinicianInstruction`, …) would proliferate
elements where a single audience-tagged structure scales. B1 is the
more durable design and is consistent with how FHIR has handled
audience- and language-tagged narrative elsewhere; B2 is the
lower-friction, ballot-cycle-feasible variant if a model change is too
disruptive this late in R6. Either variant is also more honest about
what `renderedInstruction` actually is (one of several possible
human-readable renderings, not the only one), which is the point Costa
Teixeira is asking for in the comment thread.

---

### Disposition C: Decline

#### Proposal

Do not add `DosageDetail.patientInstruction`. Direct implementers who
need patient-facing free text to (a) place it in
`DosageDetail.renderedInstruction` and tag it as patient-targeted via
profile or extension, or (b) continue carrying patient text on the
parent `Dosage` element where `patientInstruction` already exists, or
(c) carry patient instructions at the resource level (e.g.,
`MedicationRequest.note` / `Communication`). Document this guidance on
`source/dosage-definitions.html` and close the ticket as *Persuasive
with Mod* with a pointer to the existing mechanisms.

#### Justification

`DosageDetail` was deliberately scoped to *structured* dose
specification — the recently-split `Dosage` / `DosageDetail` /
`DosageSafety` / `DosageCondition` decomposition was driven by a desire
to keep each datatype focused. Free-text instructions to a specific
audience are arguably a presentation/communication concern rather than
a dosing concern, and FHIR already has multiple places to carry them.
Declining keeps the new datatype lean, avoids re-introducing the
"every datatype carries its own narrative" pattern that the split was
meant to escape, and forces implementers to be explicit about *where*
patient-facing text lives in their workflow.

---

### Recommendation

**Recommended disposition:** **A** (Accept As Requested), with the
narrative clarification from B folded in.

The reporter's request is concrete, additive, and restores parity with
a long-standing element on legacy `Dosage` that real implementers
(including national ePrescription programs) already depend on. The
patient-facing instruction is not just a presentation concern — it is
clinically distinct content (e.g., "Take with a full glass of water and
do not lie down for 30 minutes") that is authored and reviewed
separately from the structured dose, and burying it in
`renderedInstruction` loses that distinction. Disposition C therefore
under-serves a well-established use case. Disposition B's
audience-tagged model is the more elegant design and is worth
considering as a separate, broader-scope change — but pursuing it
inside this single ballot ticket risks delaying a low-cost,
backward-compatible win and entangles FHIR-54793 with an architectural
debate that should happen across all rendering elements. Recommend
Pharmacy adopt Disposition A for FHIR-54793 and, in the same workgroup
session, agree to update `source/dosage-definitions.html` /
`source/dosage.html` with a short audience-model paragraph (responding
directly to Costa Teixeira's comment), and open a separate tracker for
the audience-tagged-rendering proposal so it can be evaluated against
*all* `Dosage*` instruction elements together. Coordinate the change
with the in-flight FHIR-53721 / FHIR-54597 / FHIR-54990 / FHIR-54991
tickets so the `DosageDetail` revisions land as a coherent set.
