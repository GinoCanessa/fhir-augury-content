## Summary

Lloyd McKenzie (importing comment 298) is asking the Pharmacy work group
to evaluate whether three value sets currently defined inside the FHIR
core specification for `MedicationDispense` should be migrated to
Terminology.HL7.org (THO). The three value sets — and their underlying
HL7-authored code systems — are
`medicationdispense-status-reason` (bound to
`MedicationDispense.notPerformedReason`),
`medicationdispense-category` (bound to `MedicationDispense.category`),
and `medication-dose-aid` (bound to `MedicationDispense.doseAdministrationAid`).
The reporter explicitly notes that THO migration is **not required**,
but should be considered when (a) the value set definition is likely to
change in the coming years independently of FHIR's release cadence, or
(b) the value set is plausibly useful to CDA, V2, or other HL7
specifications. No related Jira tickets, Zulip threads, or GitHub items
were surfaced for this issue, so the work group is the first venue
weighing in.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate all three value sets and their underlying HL7-authored code
systems from FHIR core to THO:

1. Move CodeSystem `medicationdispense-status-reason`
   (`MedicationDispenseStatusReasonCodes`) and ValueSet
   `medicationdispense-status-reason` to THO under the
   `terminology.hl7.org` URL space, retaining their existing canonical
   identifiers via a redirect/alias so existing bindings keep
   resolving.
2. Move CodeSystem `medicationdispense-category`
   (`MedicationDispenseCategoryCodes`) and ValueSet
   `medicationdispense-category` to THO on the same terms.
3. Move CodeSystem `medication-dose-aid` (`MedicationDoseAid`) and
   ValueSet `medication-dose-aid` to THO on the same terms.
4. Update `MedicationDispense.notPerformedReason`,
   `MedicationDispense.category`, and
   `MedicationDispense.doseAdministrationAid` bindings in the FHIR
   core resource to point at the THO-hosted ValueSet canonicals (URLs
   may stay stable depending on UTG migration policy; binding strength
   and version semantics are preserved).
5. Coordinate with UTG to land the migration in the next UTG
   publication cycle and announce the move so downstream IGs (e.g.,
   US Core, IPA, IPS Pharmacy profiles) can pick up the new
   provenance.

#### Justification

All three value sets are HL7-authored, narrowly scoped to medication
dispensing concepts that are equally relevant to V2 (RXE/RXD segments,
e.g., dose-aid concepts) and CDA medication entries, which is exactly
the reuse case the reporter flags. THO is the standing home for
HL7-maintained terminology that is shared across product families and
that needs an out-of-band release cadence — both criteria fit here.
The reporter's framing ("if it's realistic for the value set
definition to need to change in the coming years") applies in
particular to `medicationdispense-status-reason` and
`medication-dose-aid`, where new operational reasons and assistive
devices are routinely identified by implementers between FHIR
releases.

---

### Disposition B: Alternative Approach

#### Proposal

Migrate only the value sets where cross-product reuse or off-cycle
maintenance is clearly justified, and leave the others in FHIR core:

- **Migrate to THO:** `medication-dose-aid` (CodeSystem +
  ValueSet). Dose-administration aids (blister packs, dosettes, etc.)
  are a real-world inventory that grows independently of FHIR releases
  and is directly relevant to CDA medication administration entries
  and V2 RXE/RXG dispensing flows.
- **Keep in FHIR core for now:**
  `medicationdispense-status-reason` and
  `medicationdispense-category`. Both are tightly coupled to the
  semantics of the `MedicationDispense` resource itself
  (`notPerformedReason` is a workflow concept whose meaningful set
  mirrors the resource's status machine; `category` mirrors a
  resource-level partitioning that does not obviously generalize to
  CDA/V2). Re-evaluate at the next normative ballot if cross-product
  demand emerges.

In both cases, where migration occurs, follow the same canonical
preservation + UTG coordination steps from Disposition A.

#### Justification

THO migration carries non-trivial governance overhead (UTG ballot
cycles, version pinning across IGs, redirect maintenance). The
reporter explicitly says migration is **not required** and should be
weighed case-by-case. A selective migration honors that guidance:
`medication-dose-aid` clearly meets the "useful to CDA / V2 / other
specs" bar, while the two status/category value sets are essentially
FHIR-resource-shaped and have low evidence of external reuse today.
This avoids paying the migration cost for value sets that are unlikely
to evolve independently of `MedicationDispense` itself.

---

### Disposition C: Decline

#### Proposal

Decline migration for all three value sets. Document the decision (and
the criteria considered) on the resource's terminology page so the
question is not re-litigated each ballot cycle. Re-open if and when a
concrete cross-product (CDA, V2, or another FHIR resource) consumer
appears or a definitional change is needed off-cycle.

#### Justification

All three value sets are currently scoped to one FHIR resource
(`MedicationDispense`), are HL7-authored, and have not — based on the
absence of any related Jira tickets, Zulip discussions, or GitHub
issues for this ticket — generated demonstrable demand from CDA, V2,
or other FHIR resources. Migration to THO is voluntary per the
reporter; without an identified consumer or impending definitional
change, the work is overhead that does not improve interoperability
today. Keeping them in FHIR core preserves a single point of
maintenance and avoids splitting governance across two release trains.

---

### Recommendation

**Recommended disposition:** B

A blanket "migrate everything" answer (A) is the safest stewardship
posture, but it spends UTG/Pharmacy bandwidth on value sets
(`medicationdispense-status-reason`, `medicationdispense-category`)
whose semantics are intrinsically tied to the `MedicationDispense`
resource's own state machine and partitioning — there is no obvious
CDA/V2 reuse story for them today, and the reporter is explicit that
the bar is "realistic" reuse or off-cycle change, not theoretical.
Conversely, declining outright (C) ignores the genuine cross-product
relevance of dose-administration aids, which already show up in CDA
medication entries and V2 dispensing flows and which evolve with
real-world device inventories. Disposition B (migrate
`medication-dose-aid`, keep the other two) targets the concrete reuse
case, defers the governance cost where it is not yet justified, and
leaves a clear re-evaluation hook (`if a cross-product consumer
appears or a definitional change is needed, revisit`). The work group
can override this if it has evidence of CDA/V2 alignment work for the
status-reason or category value sets that this prep didn't surface.
