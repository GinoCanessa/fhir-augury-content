## Summary

The reporter observes that in the **PMIX Mappings for PDMP Response** mapping
table inside the US Prescription Drug Monitoring Program (PDMP) FHIR
Implementation Guide, the **Patient Identifier** row lists only **SSN** as the
identifier value (mapped from PMIX `<nc:IdentificationID>` to FHIR
`Patient.identifier`). A footnote ("`*`") below the table indicates that the
Patient Identifier element actually covers multiple identifier kinds, including
**Medicare Number**, but the table itself does not enumerate Medicare Number as
a row. The reporter asks that the table be made internally consistent with its
own footnote by listing Medicare Number alongside SSN as an explicit row, both
mapped to `Patient.identifier`.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items for
this issue. The only cross-references the augury index surfaces are FHIR
artifact mentions of `Patient.identifier` (and the misspelled `Patient.identifer`)
inside the ticket body itself. The ticket has had a non-trivial life cycle:
imported from a spreadsheet in 2020 (comment #48), voted "Considered for Future
Use" 2023-01-18 (4-0-0, John Hatem / Tim McNeil) on the basis that the IG had
been withdrawn, then reopened on 2023-11-28 by **smrobertson** under project
**PSS-2225 PDMP on FHIR** when work on the IG resumed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the PDMP FHIR IG, in the **PMIX Mappings for PDMP Response** table, replace
the single `Patient Identifier — SSN*` row with two explicit rows that both
map to `Patient.identifier`:

| Element | Value | PMIX (NIEM) | FHIR |
|---|---|---|---|
| Patient Identifier | SSN | `<nc:IdentificationID>` (with identifier type / system designating SSN, e.g. `http://hl7.org/fhir/sid/us-ssn`) | `Patient.identifier` |
| Patient Identifier | Medicare Number | `<nc:IdentificationID>` (with identifier type / system designating MBI, e.g. `http://hl7.org/fhir/sid/us-medicare`) | `Patient.identifier` |

Retain the `*` footnote, but reword it from "this row also covers other
identifiers including Medicare Number" to "Patient.identifier MAY repeat;
multiple identifier instances may be present in a single response, distinguished
by `Patient.identifier.type` and/or `Patient.identifier.system`." Cross-link
each row to a profile or slice of `Patient.identifier` (e.g., a `ssn` slice and
an `mbi` slice on the PDMP Patient profile) so implementers know which
identifier system / type code to use.

Apply the same fix in any sibling mapping tables (PDMP Request, PDMP Refill,
etc.) where the same footnote/short-row pattern appears.

#### Justification

This is a low-risk documentation correction that brings the mapping table into
agreement with its own footnote. The current presentation is misleading: a
casual reader sees only SSN and may build implementations that omit Medicare
Number, even though the IG's intent (per the footnote) is to support both.
Listing each identifier kind as its own row is consistent with how PMIX→FHIR
mapping tables typically expand multi-valued NIEM elements onto repeated FHIR
elements, and aligns with FHIR's standard pattern of slicing
`Patient.identifier` by `system` / `type`. Pharmacy already owns the IG via
PSS-2225, so authoring access exists once the IG repo is reactivated.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding one row per identifier type to the mapping table, **redesign
the row as a one-to-many mapping** and pair it with a normative slicing of
`Patient.identifier` in the PDMP Patient profile:

1. Keep a single mapping-table row of the form:

   | Element | PMIX (NIEM) | FHIR | Notes |
   |---|---|---|---|
   | Patient Identifier (repeats) | `<nc:IdentificationID>` (with `nc:IdentificationCategoryText` / typed sub-element) | `Patient.identifier` (repeats; sliced by `type`) | See PDMP-Patient profile for the value set of allowed identifier types. |

2. Define a PDMP-Patient profile that slices `Patient.identifier` by
   `Patient.identifier.type.coding.code`, with named slices for at least
   `SS` (Social Security Number) and `MC` (Patient's Medicare Number) bound
   to a value set (e.g., a PDMP-Patient-Identifier-Type VS) drawn from
   `http://terminology.hl7.org/CodeSystem/v2-0203`. Bind each slice's
   `system` to the corresponding canonical identifier system
   (`http://hl7.org/fhir/sid/us-ssn`, `http://hl7.org/fhir/sid/us-medicare`).

3. Replace the table footnote with a pointer to the value set and to the
   profile's slice definitions, so the enumerable list of accepted identifier
   types lives in one normative place (the profile / value set), not in two
   places that can drift.

#### Justification

The reporter's underlying complaint is *table/footnote inconsistency* and the
risk that Medicare Number is invisible to implementers. Disposition A solves
that by enumeration, but enumeration is fragile: every additional identifier
type the PDMP community recognises (e.g., Driver's License, Tribal ID, Patient
MRN issued by a state PDMP) means another mapping-table row, and any
table/profile drift recreates exactly the bug being reported. Pushing the
enumeration into a sliced profile + bound value set keeps the mapping table
short, makes the conformance machinery (validators, IG publisher) the source
of truth, and lets future identifier additions be a value-set change rather
than an IG-text edit. This is also the dominant pattern in adjacent US Realm
IGs (US Core Patient slices `identifier` by `type` and binds to v2-0203).

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Persuasive with Modification — No Change** (or
re-apply the original 2023-01-18 *Considered for Future Use* disposition):
acknowledge that the existing footnote is intended to convey that Medicare
Number is in scope, declare the existing wording sufficient, and close the
ticket without editing the mapping table or the profile.

#### Justification

This is the disposition's weakest path, included for completeness. It is
defensible only if the workgroup judges that (a) the IG remains effectively
dormant despite the PSS-2225 reopen and there is no near-term publication
window in which to land an editorial fix, or (b) the table footnote is
unambiguous to the IG's actual reader audience and the cost of a
mapping-table edit (revalidation, re-balloting if substantive) is not
justified by an editorial gain. Both predicates are weak: PSS-2225 was
explicitly cited as the trigger for reopening, and the reporter — an
implementer-facing reader — *did* find the footnote unclear, which is the
empirical evidence against (b).

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the reporter's actual concern (the table is
inconsistent with its footnote and Medicare Number is invisible to a casual
reader) while also fixing the *category* of bug rather than just this
instance. Disposition A is a perfectly reasonable editorial fix and the
workgroup should fall back to it if profile/value-set work is judged out of
scope for the current PDMP-on-FHIR cycle, but A leaves the IG one identifier
type away from re-introducing the same inconsistency. Disposition C is hard
to defend now that PSS-2225 has explicitly reactivated the IG and an
implementer has flagged the wording as unclear. The recommendation is B with
A as the acceptable fallback if the Pharmacy WG decides the profile/slicing
work belongs in a later cycle; in that case the editorial change from A
should ship now and a follow-up ticket should be opened to track the
profile/value-set work.
