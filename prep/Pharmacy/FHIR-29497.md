## Summary

The reporter (Karin Frank) raises a question against the US Prescription Drug
Monitoring Program (PDMP) on FHIR Implementation Guide (originally tracked
under the US Medication FHIR IG umbrella). In the NCPDP-to-FHIR mapping table
for the PDMP Response, the only patient identifier called out is SSN
(`<SocialSecurity>` mapped to `Patient.identifier`). The reporter asks whether
this row should also accommodate other identifier types — specifically
**Medicare Number** (`<MedicareNumber>`) — and points out that the existing row
is flagged with `*` ("repeats"), so multiple identifier rows should be
explicitly enumerated rather than implied by a single SSN row. The proposed
wording adds a second `Patient.identifier` row for Medicare Number alongside
the existing SSN row.

The ticket has had a turbulent disposition history: it was originally voted
"Considered for Future Use" on 2023-01-18 (John Hatem / Tim McNeil, 4-0-0)
because the PDMP IG had been withdrawn, then re-opened on 2023-11-28 by
smrobertson under PSS-2225 ("PDMP on FHIR") when the workgroup picked the
project back up. It currently sits in **Triaged** status with no assignee.
There are no cross-referenced Jira tickets, Zulip threads, GitHub items, or
linked specification pages for this issue — the only system-detected reference
is a textual mention of the FHIR artifact `Patient.identifier`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the NCPDP-to-FHIR mapping table for the PDMP Response in the US PDMP on
FHIR IG to enumerate **Medicare Number** as a second `Patient.identifier`
mapping row alongside the existing SSN row, exactly as proposed by the
reporter:

| Field | NCPDP element | FHIR path |
|-|-|-|
| Patient Identifier | `<SocialSecurity>` (SSN) | `Patient.identifier` (with `type` = `SS`, `system` = `http://hl7.org/fhir/sid/us-ssn`) |
| Patient Identifier | `<MedicareNumber>` | `Patient.identifier` (with `type` = `MC` or `MA` from v2-0203, `system` = `http://hl7.org/fhir/sid/us-medicare`) |

In the IG narrative, also clarify that the `*` repetition marker on the
NCPDP-side row means **any number of identifier instances** may be present,
each carrying a different `Patient.identifier.type` and `system`, and that the
table now lists the two NCPDP-defined identifier subtypes the PDMP Response
explicitly carries.

#### Justification

The reporter is correct on both points. First, NCPDP SCRIPT / PDMP messages
already model Social Security Number and Medicare Number as distinct subtypes
of the patient identifier element, so a one-row "SSN-only" mapping
under-specifies the source content. Second, FHIR's `Patient.identifier` is
explicitly `0..*` and is the standard pattern for carrying multiple typed
identifiers differentiated by `type` and `system` — the same pattern is used
by US Core Patient and by virtually every US realm IG that handles SSN +
Medicare. Adding the row keeps the mapping faithful to the NCPDP source and
aligns the IG with established US-realm identifier conventions. There is no
breaking change risk: both rows produce additional optional `Patient.identifier`
slices, not constraints on existing ones.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than enumerating Medicare Number as a second hard-coded row, **rewrite
the PDMP Response mapping table once** to present `Patient.identifier` as a
single repeating mapping driven by an **identifier-type code system / value
set**, and reference (or define) a small PDMP value set that enumerates the
identifier types the PDMP Response is allowed to carry — at minimum SSN,
Medicare Number, and a "driver's license / state ID" placeholder commonly
required in PDMP workflows. The NCPDP `<SocialSecurity>` and
`<MedicareNumber>` elements would each be shown as concrete examples mapping
into a `Patient.identifier` slice differentiated by `Patient.identifier.type`
drawn from that value set, with `Patient.identifier.system` constrained per
slice.

#### Justification

The narrow change in Disposition A fixes the immediate complaint but leaves
the same defect for the next identifier type that comes up (driver's license,
MRN, state-issued PMP ID, etc.) — all of which are common in real PDMP
exchanges. Modeling the row as a typed, sliced repeating identifier is more
durable, gives implementers a clear binding to validate against, and matches
how US Core and other US-realm IGs already handle multi-typed identifiers. The
trade-off is that it is a larger editorial change than the reporter asked for
and requires the workgroup to agree on the initial PDMP identifier value set,
which may slow down republication of the IG.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** (or **Considered for Future Use**) on the
grounds that the PDMP on FHIR IG is currently in re-scoping under PSS-2225
and the underlying mapping table is itself subject to wholesale revision. Any
narrow row-level edit risks being thrown out when the IG is re-balloted, and
does not on its own change conformance — `Patient.identifier` is already
`0..*` in FHIR Patient and implementers can already populate Medicare Number
today.

#### Justification

This is the path the workgroup originally took on 2023-01-18 when the IG was
withdrawn. The argument still has some force: the mapping-table cell does not
constrain or enable behavior — it documents NCPDP↔FHIR provenance — and a
PDMP Response producer is free to emit a Medicare Number `Patient.identifier`
slice today regardless of whether the table lists it. Declining (or deferring)
keeps the ticket out of the way of the larger PSS-2225 rewrite. The downside
is that the original reporter raised a real, low-effort documentation defect
in 2020, and re-deferring it five years later sends an unhelpful signal.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a concrete, low-risk documentation defect:
the NCPDP PDMP Response carries multiple typed patient identifiers and the
mapping table currently shows only one. Disposition A directly answers the
question, costs almost nothing editorially, and is consistent with how
`Patient.identifier` is already used across US-realm IGs. Disposition B is
the technically cleaner outcome and the workgroup should keep it in mind when
PSS-2225 reaches the table-rewrite phase, but it is disproportionate as a
response to a single missing row and would block this ticket on broader value
set decisions. Disposition C was already tried in 2023 and was explicitly
reverted when the project was re-picked-up under PSS-2225, so deferring again
is hard to justify. Accept A now; if the workgroup, while implementing it,
sees enough additional identifier types to warrant the typed-slice rewrite,
they can promote A into B as part of the same edit pass.
