## Summary

The ticket asks the Pharmacy WG to revisit the modifier-flag decision made
when implementing [FHIR-49863](https://jira.hl7.org/browse/FHIR-49863).
FHIR-49863 was resolved "Persuasive with Modification" by renaming
`MedicationRequest.reported[x]` to `MedicationRequest.isRecordOfRequest`,
keeping it as a boolean, and **flagging it as a modifier element** so that
consumers cannot ignore the distinction between a primary order and a
record/copy of one. The reporter (Jose Costa-Teixeira, who also assigned
FHIR-49863) now argues that modifying the resource semantics is unnecessary:
because `MedicationRequest` follows the Request pattern — and Request
instances "are not actionable orders per se" — a system that ignores
`isRecordOfRequest` is not put at risk in the way a modifier element is
intended to prevent. The reviewer comment on
[HL7/fhir PR #3896](https://github.com/HL7/fhir/pull/3896#pullrequestreview-3407698941)
referenced in the description makes the same point. The proposed change is
narrow: keep the new `isRecordOfRequest` element exactly as introduced by
FHIR-49863, but **remove the `isModifier` flag** from it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reverse the FHIR-49863 modifier decision on the new
`MedicationRequest.isRecordOfRequest` element while keeping every other
aspect of FHIR-49863 intact (the rename from `reported[x]`, the boolean
type, the new short, the new definition, and the mapping to the Request
workflow pattern's `reported[x]`).

Concretely, in `HL7/fhir` under `source/medicationrequest/`:

1. In the MedicationRequest StructureDefinition `<differential>`, locate
   the element `MedicationRequest.isRecordOfRequest` and set
   `<isModifier value="false"/>`.
2. Remove (or clear) the corresponding `<isModifierReason>` element so the
   resource validates and so the publisher does not surface a stale
   modifier-reason.
3. Confirm that the element's short and definition (set by FHIR-49863) do
   not need re-wording now that it is no longer a modifier — the existing
   text already explains that the instance "is a record of … and should
   not be considered" an actionable order, which is enough advisory
   language for a non-modifier flag.
4. Do not edit the regenerated `<snapshot>` block, the
   `medicationrequest-spreadsheet.xml`, or any sushi/IG-publisher artefacts
   (none exist in this repo).
5. Coordinate with the Zulip pre-apply already done by Jose Costa-Teixeira
   so the build branch and the Pharmacy WG decision land in sync.

#### Justification

The Request workflow pattern is explicit that a Request resource describes
something that has been requested, not something that *will be acted on*
without further orchestration; a system that processes a `MedicationRequest`
without consulting `isRecordOfRequest` is not at risk of dispensing or
administering medication on the basis of a record-of, because acting on a
`MedicationRequest` already requires honouring `status` and `intent` and a
downstream MedicationDispense / MedicationAdministration workflow. That is
the standard FHIR test for whether a flag must be a modifier — "ignoring
this changes the meaning of the resource in a clinically unsafe way" — and
it is not met here. The PR #3896 review comment cited by the reporter makes
the same argument. Lloyd McKenzie's "yes" on the Zulip pre-apply check, and
Jose's note that the change has already been pre-applied to the build,
indicate that the committers most familiar with the modifier-element bar
have already concluded this is the right call. The change is also
operationally cheap: a single `<isModifier>` flip in
`source/medicationrequest/`, no terminology, no extension, no
narrative-page rewrites, and no cross-repo coordination per the FhirCore
briefing.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `MedicationRequest.isRecordOfRequest` as a modifier element, but
sharpen the supporting metadata so the modifier flag is actually carrying
its weight. Specifically, in `source/medicationrequest/`:

1. Leave `<isModifier value="true"/>` in place on
   `MedicationRequest.isRecordOfRequest`.
2. Tighten `<isModifierReason>` to spell out exactly which downstream
   interpretations change when the value is `true` (e.g., "Recipients must
   not treat this MedicationRequest as authority to dispense, supply, or
   administer; it represents a previously known request being reported for
   continuity-of-care purposes only.").
3. Optionally add an invariant or a comment noting that `isRecordOfRequest
   = true` combined with `intent = order` and `status = active` is a
   record-of-an-order, not an active order, and is the original motivating
   case from FHIR-49863.
4. Document the decision in the MedicationRequest narrative
   (`source/medicationrequest/medicationrequest.html` style page, per the
   FhirCore briefing's authoring root) so implementers reading the page
   understand the safety-by-modifier reasoning.

#### Justification

FHIR-49863 was filed precisely because real-world IPS / APS / Aged Care
deployments confuse an actionable medication order with a *report* of one,
even when `status` and `intent` should already make that distinction
clear. The Pharmacy WG's original reasoning when resolving FHIR-49863
"Persuasive with Modification" — that the safety bar is high enough to
warrant a modifier — is defensible: implementations that silently drop or
ignore unfamiliar booleans are common, and a modifier flag forces parsers
and validators to take notice. The Request-pattern argument used in
FHIR-53305 is correct in theory but assumes a level of pattern-aware
implementation that is not universal in IPS-consuming systems. Keeping the
modifier flag and instead investing in a clearer
`<isModifierReason>` preserves the FHIR-49863 safety guarantee at low
cost, and avoids re-litigating the same question once IPS implementers
start exchanging real records.

---

### Disposition C: Decline

#### Proposal

Reject FHIR-53305 and leave FHIR-49863 as resolved. Take no edit to
`source/medicationrequest/`; revert the pre-apply on the build branch if
necessary so the published spec continues to mark
`MedicationRequest.isRecordOfRequest` as a modifier element.

#### Justification

FHIR-49863 was discussed and resolved by the same Work Group only weeks
earlier, with an explicit decision to make the new element a modifier.
Reopening that decision on the basis of one PR-review comment, with no new
implementer evidence and no Zulip discussion beyond the pre-apply
mechanics, sets a precedent that any committer-level reviewer can flip a
WG resolution. The Request-pattern argument is also rebuttable: many
Request-pattern resources (e.g., `ServiceRequest`, `CommunicationRequest`)
do route into actionable workflows in deployed systems, and the
record-vs-order distinction is exactly the class of semantic that the
modifier mechanism exists to protect. If the WG later finds that the
modifier flag is causing real harm (e.g., breaking validators that treat
unknown modifiers as errors), a follow-up ticket can revisit it then.

---

### Recommendation

**Recommended disposition:** A

The Request pattern explicitly warns that Request-resource instances are
not, by themselves, authority to act, and the safety chain for actually
administering medication runs through MedicationDispense /
MedicationAdministration — not through silent consumption of a
`MedicationRequest`. That is the canonical FHIR test for whether a flag
needs to be a modifier, and `isRecordOfRequest` does not pass it. The
reporter, the cited PR-review comment, and Lloyd McKenzie's "yes" on the
committers stream are aligned, the change has already been pre-applied to
the build, and the FhirCore briefing confirms the work is a single-file
`<isModifier>` flip in `source/medicationrequest/` with no terminology,
extension, or downstream-IG ripple. Disposition B is the right fallback if
Pharmacy WG members raise concrete IPS-implementation evidence on Monday
that ignoring the flag would cause clinical harm; Disposition C should
only be entertained if the WG concludes that overturning a decision made
weeks earlier without new field evidence is itself the larger risk. Absent
that pushback, accept as requested.
