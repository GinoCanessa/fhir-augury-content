## Summary

The reporter flags Example 6 in the FHIR Core "Dosage Regime Examples"
page (§14.7.1.7) as ambiguous. The narrative reads *"Iron sucrose:
'Mon/Wed/Fri 09:00 × 5 doses total'"*, but the accompanying structure
sets `Timing.repeat.count = 5`, `period = 1`, `periodUnit = d`, and
`dayOfWeek = Mon|Wed|Fri`. A reader can plausibly interpret this as
either (a) five total administrations spread across the Mon/Wed/Fri
pattern (matching the prose), (b) five administrations *per dosing day*
(yielding 15 doses), or (c) repeating the Mon/Wed/Fri block five times
(also 15 doses). The reporter asks the Pharmacy work group to adjust
Example 6 so prose and structure unambiguously convey the same meaning.
No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced via the augury index for this ticket.

## Details

**Description:**

> In example 6 (14.7.1.7) the dosage instruction says "Iron sucrose:
> 'Mon/Wed/Fri 09:00 × 5 doses total'".
>
> What does '5 doses' mean? Is it five occurrences in total? Meaning on
> Monday, Wednesday, Friday then Monday and Wednesday in the next week?
> Or five doses every day?
>
> When looking to the structure of the example it could be interpreted
> as repeat "mon/wed/fri at 09:00" five times, which makes it a total
> of 15 doses and not 5 doses. 5 doses is not the same as "repeat it
> for 5 times".
>
> Suggestion is to adjust example 6.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit Example 6 on the Dosage Regime Examples page (§14.7.1.7) to remove
the ambiguity in both the narrative and the structure:

1. **Narrative (textual line above the example).** Replace
   *"Iron sucrose: 'Mon/Wed/Fri 09:00 × 5 doses total'"* with an
   explicit, unambiguous phrasing such as:
   *"Iron sucrose: one dose at 09:00 on Mon/Wed/Fri, stop after 5
   total doses (i.e. completes within ~2 weeks)."*

2. **Structure / JSON / XML tabs.** Keep `count = 5` (this already
   means *5 total occurrences* per `Timing.repeat.count`), but add a
   short bullet under the existing "Structure" list reaffirming the
   semantics, e.g.:
   *"`count` is the total number of administrations across the
   `dayOfWeek` pattern — not five per day and not five repeats of the
   Mon/Wed/Fri block. Five doses therefore complete on the second
   Wednesday."*

3. **Optional belt-and-braces.** Add `boundsDuration` (e.g.
   `value = 11, unit = d`) or `boundsPeriod` so the example is
   self-bounding and a reader cannot mistake `period = 1 / periodUnit
   = d` for "five doses per day".

#### Justification

This is what the reporter asked for and the cheapest fix that resolves
the ambiguity at the point of confusion. The reading the reporter is
worried about ("5 doses per day" or "repeat the block 5 times = 15
doses") contradicts the normative definition of `Timing.repeat.count`
("A total count of the desired number of repetitions across the
duration of the entire timing specification"), so the structure is
already correct — only the prose and a clarifying bullet need to land.
Authoring this change is a narrative-only edit on the existing
examples page (no schema, no StructureDefinition, no resource example
JSON re-publication), so the regression surface is minimal and there
are no downstream IGs that would have to be re-floored.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than touching Example 6 in isolation, add a short
**"Interpreting `count`, `period`, and `dayOfWeek` together"** callout
box at the top of the Dosage Regime Examples page (§14.7) and then
update Example 6 (and any other examples that combine `count` with
`dayOfWeek` / `timeOfDay`) to point back at it. The callout would
explicitly state:

- `count` is the total number of administrations, *not* per-day and
  *not* per-cycle;
- when `dayOfWeek` is present, `period` / `periodUnit` describe the
  cadence *within* the constrained set of days, not an independent
  daily schedule;
- worked arithmetic for the Mon/Wed/Fri × `count = 5` case showing the
  five concrete dose times.

Example 6's narrative is then simplified to *"Iron sucrose: one dose
at 09:00 on Mon/Wed/Fri, total of 5 doses (see callout above)."*

#### Justification

The reporter's confusion is almost certainly not unique to Example 6 —
any example that combines `count` with `dayOfWeek` is open to the same
mis-reading, and the FHIR Core spec has historically attracted Pharmacy
clarifications about `Timing.repeat` semantics. Fixing the underlying
documentation gap once, and pointing each example at the shared
explanation, is more durable than re-wording a single example in
isolation. The cost is a slightly larger edit on the same single
authored page (`dosage-examples.html`) plus a cross-reference from
Example 6 — still narrative-only, no normative change.

---

### Disposition C: Decline

#### Proposal

Decline the change as **"already addressed by existing mechanism"**.
The behaviour the reporter is worried about is fully specified by the
`Timing.repeat.count` definition ("A total count of the desired number
of repetitions across the duration of the entire timing
specification"). Example 6's structure (`count = 5`) and narrative
("5 doses total") are mutually consistent under that definition; the
ambiguity is in the reader's interpretation, not in the example. Close
as "Not Persuasive" with a comment pointing at the `count` element
definition and the existing section §14.7 introduction.

#### Justification

This option is defensible only on strict-reading grounds: the spec is
already correct and the example matches the spec. However, it ignores
that the reporter is a domain expert (Pharmacy) who *did* read the
spec and *still* found Example 6 ambiguous, which is the canonical
signal that the documentation — not the underlying model — needs work.
Use this disposition only if the work group concludes that no plausible
re-wording would help and that the reporter's reading is an outlier.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the actual root cause — that `count` combined
with `dayOfWeek` is consistently misread by implementers — instead of
patching only the example the reporter happened to land on. It costs
essentially the same authoring effort as Disposition A (a single
narrative edit to `dosage-examples.html` on the FHIR Core spec, no
StructureDefinition or resource changes), it makes Example 6
self-evidently unambiguous, and it inoculates the other count-plus-
constrained-days examples on the same page against the next ticket of
this shape. Disposition A is an acceptable fallback if the work group
prefers to keep the change strictly scoped to Example 6; Disposition C
should be rejected because the reporter is a Pharmacy domain reader
and a Highest-priority "Triaged" ticket from that audience is itself
evidence that the prose is not doing its job.
