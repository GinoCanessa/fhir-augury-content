## Summary

FHIR-29488 reports that the PDMP API example in the (now-withdrawn) **US
Medication FHIR Implementation Guide** is unclear: when the example is
expanded, it appears to invoke a search parameter that does not exist —
referred to in the example as `.validityPeriod` — and the reporter (Josh
Mandel) suggests the intent may have been the standard `authoredon`
search parameter on `MedicationRequest`/`MedicationDispense`. The
underlying issue is a documentation/clarity problem on the example, but
it has implementation consequences because a reader cannot tell whether
the IG actually requires a non-standard search parameter or whether the
example is simply mis-rendered.

This ticket sits inside a tightly-related cluster of "PDMP-old-issues"
filed against the same example block:

- [FHIR-29487](https://jira.hl7.org/browse/FHIR-29487) — formatting of
  the same API example (backticks, indentation, line-breaks).
- [FHIR-29526](https://jira.hl7.org/browse/FHIR-29526) — wording fix
  ("not 'exact', it is an example") plus a request to break the example
  into a parts/explanation table.
- [FHIR-29535](https://jira.hl7.org/browse/FHIR-29535) — confirms the
  intended chained search parameter is
  `authorizingPrescription.dispenseRequest.validityPeriod` (i.e., the
  IG was reaching the `MedicationRequest.dispenseRequest.validityPeriod`
  element via a chained query through
  `MedicationDispense.authorizingPrescription`), and questions whether
  it is needed at all.
- [FHIR-29475](https://jira.hl7.org/browse/FHIR-29475),
  [FHIR-29481](https://jira.hl7.org/browse/FHIR-29481),
  [FHIR-43139](https://jira.hl7.org/browse/FHIR-43139) — sibling
  formatting / API-design tickets in the same PDMP section.

Per the comments on this ticket, the original `us-meds` IG was
**withdrawn** (resolution previously set to "Considered for Future Use"
on 2023-01-18, then reverted on 2023-11-28 when the project was picked
back up under PSS-2225). Subsequent Pharmacy WG activity moved PDMP into
its own IG (`HL7/fhir-pdmp`, *US Prescription Drug Monitoring Program
(PDMP) FHIR IG*, STU1 published 2024-10-15). The disposition therefore
needs to consider whether to retire the ticket against the withdrawn
host IG, port the fix to the new PDMP IG, or both.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the report as a **wording / search-parameter correction** on the
PDMP API example. Rewrite the example so that the chained search
parameter currently shown as `…validityPeriod` is replaced by the
intended FHIR search semantics:

- If the intent really is "dispense events by date" — the most common
  PDMP query — change the example to use
  `MedicationDispense?…&whenhandedover=geXXXX&whenhandedover=leYYYY`
  (or `&authoredon=…` if the intent is the prescription's authored
  date, as Josh Mandel speculates).
- If the original intent really was the prescription's validity window,
  keep the chained form but spell it out exactly:
  `MedicationDispense?…&authorizingPrescription.dispenseRequest.validityPeriod=…`
  and add a sentence noting that this is a chained search through
  `MedicationRequest`, not a search parameter on `MedicationDispense`.

Make the change in the *US Prescription Drug Monitoring Program (PDMP)
FHIR IG* (`HL7/fhir-pdmp`, the live successor) and mark the ticket
"Persuasive with Mod" against the withdrawn `us-meds` IG. Cross-link
FHIR-29487 and FHIR-29526 in the resolution comment so the example is
re-rendered (newlines, indentation, no stray backticks, parts/
explanation table) at the same time.

#### Justification

FHIR-29535 confirms the IG was using
`authorizingPrescription.dispenseRequest.validityPeriod` as a chained
search expression, so FHIR-29488's "non-existent search parameter" read
is genuinely a documentation defect — the example is right "in spirit"
but unreadable as rendered. Fixing the example aligns with Pharmacy WG's
broader cleanup of the PDMP section (FHIR-29475, FHIR-29487, FHIR-29526)
and avoids leaving implementers to guess between `validityPeriod`,
`authoredon`, and `whenhandedover`. Doing the fix in the live
`fhir-pdmp` IG (rather than the withdrawn host IG) is what the
2023-11-28 reopening explicitly anticipates.

---

### Disposition B: Alternative Approach — Re-scope as a Persuasive duplicate of FHIR-29535 + FHIR-29487/29526, fix once

#### Proposal

Rather than treat FHIR-29488 as a standalone ticket, **resolve it as
"Persuasive — Duplicate"** of the cluster {FHIR-29487, FHIR-29526,
FHIR-29535} and roll all four into a single editorial change to the API
example block in `HL7/fhir-pdmp`:

1. Reformat the `GET [base]/MedicationDispense?…` example with one
   parameter per line and proper indentation (FHIR-29487).
2. Replace the word "exact" with "example" and convert the parameters
   into a parts/explanation table (FHIR-29526).
3. Either keep the chained
   `authorizingPrescription.dispenseRequest.validityPeriod` parameter
   (FHIR-29535) with an explanatory sentence, or drop it in favour of a
   `whenhandedover` / `authoredon` filter — Pharmacy WG to confirm.
4. The clarification on the search parameter name (this ticket,
   FHIR-29488) is automatically delivered by steps 1–3.

Track the unified change via FHIR-29535 (which carries the most
specific, actionable wording) and close FHIR-29488/29487/29526 as
duplicates pointing at it.

#### Justification

These four tickets all touch the same paragraph in the same withdrawn
IG. Making four independent changes invites drift and requires four
separate ballot dispositions; making one consolidated change in the
successor `HL7/fhir-pdmp` IG is materially less work for the editor and
gives reviewers a single coherent block to vote on. It also avoids
having FHIR-29488 inadvertently introduce a *new* search-parameter
semantic (e.g., switching to `authoredon`) without considering FHIR-29535's
suggestion that the parameter may not be needed at all — that decision
is best made once, in context, rather than per-ticket.

---

### Disposition C: Decline — Not Persuasive (host IG withdrawn; defer to successor IG's own ballot)

#### Proposal

Close FHIR-29488 as **Not Persuasive** with the rationale that the host
specification (*US Medication FHIR Implementation Guide* / `us-meds`)
has been withdrawn and is no longer maintained. Add a comment directing
implementers to the published successor *US Prescription Drug Monitoring
Program (PDMP) FHIR IG* (`hl7.fhir.us.pdmp` STU1, published 2024-10-15)
and noting that any remaining defect should be re-filed against that
IG's tracker. Apply the same disposition consistently to other
"PDMP-old-issues" labelled tickets that target only the withdrawn IG.

#### Justification

The 2023-01-18 vote (John Hatem / Tim McNeil 4-0-0) already deferred
this class of issue precisely because the IG was withdrawn. Although
that resolution was reverted on 2023-11-28 to keep the ticket alive
during PSS-2225, the practical outcome — that PDMP work has moved to
`HL7/fhir-pdmp` — has now occurred. Spending workgroup cycles editing
a withdrawn artefact is low-value, and any reader who hits the broken
example today is on a frozen historical page rather than current
normative guidance. Declining here keeps the Pharmacy backlog focused
on the live IG, where the same readers will land.

---

### Recommendation

**Recommended disposition:** **B**

The defect FHIR-29488 reports is real, but it is one of four
overlapping tickets describing the same broken paragraph. Disposition C
under-delivers — the example is now republished essentially verbatim in
the live PDMP IG, so closing without fixing leaves the same confusion in
front of implementers. Disposition A fixes the right thing but in
isolation, which risks landing a `validityPeriod` → `authoredon`
substitution that contradicts FHIR-29535's read of the original intent
(a chained search through `MedicationRequest.dispenseRequest`).
Disposition B captures the substance of A while letting Pharmacy make
one coherent decision — keep the chained `validityPeriod` form with
proper formatting, or drop it in favour of `whenhandedover` /
`authoredon` — and pay the editorial cost once. Verification of the
exact path in `HL7/fhir-pdmp` should be done by whoever picks up the
consolidated ticket, since no `repo-analysis` briefing was loaded for
that repo as part of this prep.
