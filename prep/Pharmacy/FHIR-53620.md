## Summary

The ticket requests that the CARIN Real-time Pharmacy Benefit Check
(RTPBC) IG update its base-profile dependencies from **US Core 3.1.1**
to **US Core 8.0.1**. Specifically, the RTPBC profiles that currently
derive from US Core 3.1.1 versions of Patient, MedicationRequest, and
Coverage should be re-parented to their US Core 8.0.1 equivalents, with
any resulting conflicts resolved in the RTPBC profile content. The
reporter's stated resolution is to bump the base IG dependency and the
parent profile references on **RTPBC MedicationRequest**, **RTPBC
Patient**, and **RTPBC Coverage**, then reconcile any constraints or
slicing that diverge between the current RTPBC content and the
tightened US Core 8 requirements. No related Jira tickets, Zulip
discussions, or GitHub items were surfaced through cross-references or
search, so this appears to be a standalone technical-update request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's proposed resolution verbatim: in the RTPBC IG,
update the `dependsOn` on US Core from 3.1.1 to 8.0.1 in the
ImplementationGuide resource, and change the `baseDefinition` on the
three RTPBC profiles to the US Core 8.0.1 canonicals:

- RTPBC Patient → `http://hl7.org/fhir/us/core/StructureDefinition/us-core-patient|8.0.1`
- RTPBC MedicationRequest → `http://hl7.org/fhir/us/core/StructureDefinition/us-core-medicationrequest|8.0.1`
- RTPBC Coverage → `http://hl7.org/fhir/us/core/StructureDefinition/us-core-coverage|8.0.1`

Then resolve any snapshot/differential conflicts: reconcile
must-support elements, terminology bindings (especially the
MedicationRequest status and intent bindings and Patient identifier /
race / ethnicity extensions that tightened in US Core 4→8), and any
slicing or cardinality changes. Update IG narrative and examples to
conform. Publish as a technical-update release of the RTPBC IG.

#### Justification

This is the straightforward, direct response to the ticket's stated
need and matches the resolution text the reporter already supplied.
US Core 3.1.1 is multiple major versions behind current US Core and is
no longer the reference baseline for US realm implementers; leaving
RTPBC anchored to 3.1.1 forces implementers to reconcile two divergent
US Core views (their EHR's current US Core vs. RTPBC's 3.1.1 anchor).
Re-basing to US Core 8.0.1 aligns RTPBC with the version most US
implementers are converging on and is consistent with how other
CARIN/Da Vinci IGs have handled US Core version bumps. The
`technical-updates` label on the ticket signals the workgroup already
views this as a maintenance change rather than a new substantive
requirement.

---

### Disposition B: Alternative Approach

#### Proposal

Re-parent the three RTPBC profiles onto US Core 8.0.1 as in
Disposition A, but **package the change as a new major version of
RTPBC** (rather than a technical-update patch) and explicitly scope
the conflict resolution:

- For each of RTPBC Patient, MedicationRequest, and Coverage, produce
  a documented delta table showing every must-support, binding, and
  cardinality change inherited from US Core 8.0.1, and each
  RTPBC-side adjustment made to accommodate it.
- Where a US Core 8 tightening would break existing RTPBC exchanges
  (e.g., a newly-required element that RTPBC payers cannot populate),
  narrow the inheritance with a profile-level must-support override
  and record the rationale in the IG.
- Retain the US Core 3.1.1-based profiles as deprecated, versioned
  artifacts for one publication cycle so current implementers have a
  defined migration window.

#### Justification

Bumping from US Core 3.1.1 to 8.0.1 crosses several major US Core
releases; the cumulative delta on MedicationRequest (intent/status
value sets, reported[x] handling, dosageInstruction must-supports),
Patient (sex/gender/race/ethnicity extension churn across US Core 5–7),
and Coverage (new must-supports and identifier slicing) is
non-trivial. Treating this as a plain technical-update risks silently
breaking conformance for payers that are live today on RTPBC against
US Core 3.1.1. Publishing as a major RTPBC version with an explicit
delta and a brief deprecation window honours the intent of the request
while giving implementers a defensible migration path, and it matches
how adjacent Da Vinci/CARIN IGs have handled similar base-version
jumps.

---

### Disposition C: Decline

#### Proposal

Close without adopting. Leave the RTPBC base-profile anchor at US Core
3.1.1 for the current publication cycle. Instead, add IG narrative
that clarifies the minimum US Core baseline is 3.1.1 and that RTPBC
profiles are compatible-by-derivation with later US Core versions to
the extent that those later versions only add must-supports or
narrower bindings.

#### Justification

This would only be defensible if the Pharmacy workgroup concludes that
the current RTPBC implementer base is still predominantly on US Core
3.1.1 and that re-parenting would impose more migration cost than it
relieves. Given the `Persuasive` resolution already recorded on the
ticket and the `technical-updates` label, this does not appear to be
the workgroup's intent; Disposition C is included for completeness as
the counter-position a reviewer might raise.

---

### Recommendation

**Recommended disposition:** B

Disposition A is already effectively endorsed by the ticket's
`Persuasive` resolution, so the interesting choice for the workgroup
is how to package the change, not whether to make it. Jumping from US
Core 3.1.1 to 8.0.1 spans several years of US Core evolution and
touches the three most exchange-critical RTPBC resources (Patient,
MedicationRequest, Coverage). Handling that as an opaque
"technical-update" risks silently re-defining conformance for live
RTPBC exchanges, whereas Disposition B delivers the same technical
outcome but adds a documented delta, explicit must-support overrides
where US Core 8 would otherwise break current implementers, and a
one-cycle deprecation window — all low-cost, high-signal additions for
a US-realm payer IG. Pharmacy should pursue B and use the ticket's
resolution text as the core of the migration guide.
