## Summary

The ticket reports a typo on the home page of the US Prescription Drug
Monitoring Program (PDMP) FHIR IG: the word "futher" appears in a sentence
referring readers to the ISO 3166 country-codes web page and should read
"further". The reporter requests that the spelling be corrected. The PDMP
work group's investigation (recorded in the comments) confirmed the typo
reproduces with the most recent SUSHI (v3.10.0) and IG Publisher
(publisher.jar 1.6.8), and concluded that the offending text is not
authored in the PDMP IG itself — it is part of a global "intellectual
property" / boilerplate statement injected by the FSH / SUSHI / IG
Publisher tool chain. As a result, the assignee indicated the comment
would be forwarded to FHIR Infrastructure (FHIR-I) for resolution. There
are no cross-referenced Jira tickets, Zulip discussions, or GitHub items
linked to this ticket in the FhirAugury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive — Pharmacy accepts the change in principle but recognises the
fix cannot be made in the PDMP IG. Formally re-route the ticket to FHIR
Infrastructure (FHIR-I), or to the SUSHI / IG Publisher project that
emits the global "intellectual property" / country-codes boilerplate, so
that the source template containing "futher" is corrected to "further".
Once the upstream tool chain is patched, a routine PDMP IG re-publish
will pick up the corrected wording on the home page with no PDMP-side
content edits. Add a comment on FHIR-45695 linking to the upstream
ticket / PR and resolve PDMP-45695 as "Persuasive — referred upstream"
once the link is recorded.

#### Justification

The reporter's intent — fixing the spelling on the PDMP home page — is
clearly correct; this is an unambiguous typo in user-facing text. The
PDMP work group has already confirmed (smrobertson, 2024-05-20) that the
text is generated by the FSH/SUSHI/IG Publisher pipeline and is not
editable from inside the PDMP IG, so the only path that actually delivers
the requested change is an upstream fix. Treating this as
"Persuasive / referred" rather than a no-op preserves the audit trail
that the change was accepted on its merits, while putting it in the hands
of the group that can actually ship the correction. The
`PDMP-ballot-referred` label already signals the work group's
expectation that the issue is being handed off.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification — keep the substantive resolution
(upstream fix) but, in addition, ship a localised PDMP-side workaround
so the typo disappears from the published PDMP home page in the next
PDMP re-publish, without waiting for the upstream tool chain. Concretely:
override the affected boilerplate on the PDMP IG home page by either
(a) supplying a custom IG-Publisher template / page-content fragment for
the home page that re-states the ISO 3166 reference with the correct
spelling, or (b) adding a small `pagecontent` snippet in the PDMP IG
that explicitly re-introduces the country-codes guidance worded
correctly, so the misspelled global statement is visually superseded.
Continue to forward the underlying defect to FHIR-I / SUSHI / IG
Publisher so the global template is fixed for everyone.

#### Justification

This addresses the trade-off in Disposition A: an upstream fix is the
right long-term answer, but it can take a full SUSHI / IG-Publisher
release cycle to land and propagate, during which the PDMP IG continues
to ship a visible typo immediately above an ISO link. A targeted
PDMP-side override removes the user-visible defect at the next PDMP
re-publish while preserving the upstream referral. The cost is a small
amount of PDMP-specific authoring drift that has to be cleaned up once
the upstream fix lands; for a single-word typo in boilerplate, that is a
modest burden.

---

### Disposition C: Decline

#### Proposal

Not Persuasive (or "Not Related") — close FHIR-45695 against the PDMP
IG on the basis that the misspelled string is not authored in the PDMP
IG. The text in question is part of a global IP / country-codes
statement emitted by the FSH/SUSHI/IG Publisher tool chain, so PDMP has
no surface on which to apply the change. Direct the reporter to file (or
allow Pharmacy to file on the reporter's behalf) the equivalent ticket
against FHIR Infrastructure or the appropriate IG-Publisher /
HL7 / fhir-ig-publisher repository, and close this ticket with a
pointer to the upstream issue.

#### Justification

The PDMP work group's own analysis (smrobertson, 2024-05-20) explicitly
states "this is not correctable in the IG", which is a defensible basis
for declining the ticket *as filed against PDMP*. Doing so keeps the
PDMP ballot-disposition log honest — the work group should not record a
"Persuasive" change it cannot actually implement — and ensures the
issue moves to a tracker where it can be acted on. The downside is
purely procedural: ballot reporters often expect "Persuasive — referred"
rather than "Not Persuasive" when the substantive intent is endorsed,
which is why this disposition is the weakest of the three.

---

### Recommendation

**Recommended disposition:** A

The substantive change requested ("futher" → "further") is unambiguously
correct and should be acknowledged on its merits, so a flat decline
(Disposition C) understates Pharmacy's agreement with the reporter and
risks looking dismissive of a valid ballot comment. A PDMP-local
workaround (Disposition B) is technically feasible but introduces
authoring drift in the PDMP IG to paper over a single-word typo in
upstream boilerplate, which is poor leverage for a Technical Correction
of this size. Disposition A — accept the comment as Persuasive, formally
refer the fix to FHIR-I / the SUSHI / IG-Publisher tool chain, and link
the upstream ticket from FHIR-45695 — matches what the assignee already
indicated would happen, preserves the audit trail that Pharmacy agreed
with the reporter, and keeps the IG free of one-off overrides.
