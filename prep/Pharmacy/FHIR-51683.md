## Summary

The ticket asks the Pharmacy work group to convert all conformance
artifacts in the **US CARIN Real-time Pharmacy Benefit Check (RTPBC)** IG
from raw XML StructureDefinition / ValueSet / CodeSystem / example
files into **FHIR Shorthand (FSH)** sources, processed through SUSHI as
part of the IG build. The motivation is purely maintainability — the
underlying conformance content (profiles for resource, messaging, and
terminology, plus their value sets, code systems, and examples) is
expected to be unchanged, except where some other change request
specifically modifies it. The reporter (Frank McKinney) is the long-time
RTPBC IG editor, and the Jira resolution has been pre-set to
**Persuasive**, indicating the proposer expects the work group to
accept the change in concept and that the ticket primarily exists to
authorise the rework. There are no cross-referenced Jira tickets,
Zulip threads, GitHub PRs/issues, or Confluence pages tying in
additional context — this is a self-contained tooling/format change
ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Authorise the editor to perform a one-shot conversion of every
conformance artifact currently authored as raw XML in the RTPBC IG into
FSH, processed by SUSHI as part of the IG build. Concretely, in the
RTPBC IG source repository (the CARIN RTPBC IG repo maintained by the
Pharmacy WG), this means:

- Move existing `input/resources/StructureDefinition-*.xml` (profiles
  and extensions for resource, messaging, and terminology profiles) to
  `input/fsh/profiles/*.fsh` and `input/fsh/extensions/*.fsh`.
- Move existing `input/resources/ValueSet-*.xml` and
  `input/resources/CodeSystem-*.xml` to `input/fsh/terminology/*.fsh`.
- Move existing `input/examples/*.xml` to `input/fsh/instances/*.fsh`
  (or keep curated XML examples under `input/examples/` only where FSH
  cannot faithfully express them, e.g., narrative-bound or hand-curated
  Bundles).
- Add / update `sushi-config.yaml` so SUSHI emits the same canonical
  URLs, ids, names, and versions that the current XML uses.
- Adopt the "round-trip diff" acceptance gate spelled out in the
  proposed resolution: regenerate the package and confirm that every
  produced StructureDefinition / ValueSet / CodeSystem / example is
  byte-equivalent (or, more realistically, semantically equivalent
  after canonicalisation) to the previously-published artifact, except
  where a separate, already-accepted change ticket has modified it.
- No substantive change is made to the IG's normative content as part
  of this ticket.

#### Justification

This matches the reporter's request verbatim and is consistent with
HL7-wide practice: the great majority of currently-balloted IGs author
conformance artifacts in FSH and treat raw-XML authoring as legacy.
Maintainability gains are real — FSH diffs are reviewable, profile
changes can be made without hand-editing element-by-element XML, and
new contributors can use the same toolchain (`sushi`, IG Publisher,
fsh-validator, SUSHI snapshots) they use on every other HL7 US Realm
IG. The acceptance gate built into the resolution ("Confirm resulting
artifacts are unchanged unless otherwise modified in response to a
separate change request") preserves substrate stability and means
implementers will see no behavioural change. The reporter has already
flagged the proposed resolution as **Persuasive**, indicating editor
buy-in.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the conversion in principle, but stage it across two ballot
cycles instead of doing it as a single bulk rework, and pair each
staged conversion with explicit publication-equivalence evidence:

1. **Cycle 1 — terminology and examples first.** Convert all
   `CodeSystem`, `ValueSet`, and example resources to FSH. These are
   the lowest-risk artifacts (no snapshotting concerns, no slicing,
   minimal differential surface). Publish a CI build and attach a diff
   report against the previously published package so reviewers can
   verify equivalence. Hold the converted profiles back.
2. **Cycle 2 — profiles, extensions, and messaging definitions.**
   Convert the StructureDefinitions for resource, messaging, and
   terminology profiles, including any extensions, in the next ballot
   cycle. These are higher-risk (slicing, discriminators, snapshot
   regeneration, must-support flags, narrative regeneration) and
   benefit from being reviewed alongside an explicit canonical diff
   between old-XML and new-FSH outputs.

In both cycles, retain the original XML files in repo history (and,
optionally, under `input/resources-legacy/` excluded from the IG build)
for one cycle so that the WG can refer back during ballot review.

#### Justification

A bulk XML-to-FSH conversion in one PR is a large, opaque diff:
reviewers cannot easily distinguish "purely cosmetic FSH rewrite" from
"accidentally changed slicing discriminator" or "lost a must-support
flag" without artifact-level diffing. Splitting the work isolates the
risky surface (profiles / slicing / snapshots) into its own cycle, and
publishing a canonical-diff report against the previously published
package makes the "no substantive change" promise auditable rather than
trust-based. The trade-off vs. Disposition A is a longer rollout and
some duplicate editor effort, but it lowers the chance of an accidental
implementer-visible regression slipping in under the "format change
only" banner.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for substantive RTPBC change
tickets and leave the existing raw-XML authoring in place. Rationale
category: *technical/tooling change without implementer impact, not
warranting WG-managed change control*.

#### Justification

The change does not alter any conformance behaviour an implementer can
observe — the published canonical artifacts are intended to be
identical. If the WG considers tooling/authoring choices to be the
editor's prerogative rather than a balloted change, then a Jira ticket
is not the right vehicle: the editor can make the conversion under a
maintenance PR without WG disposition, on the condition that the
published package round-trips. Declining the ticket on that basis is
defensible if the WG wants to avoid setting a precedent of using Jira
to authorise pure refactors. (This disposition would not block the
work; it would simply remove WG-level authorisation as a prerequisite.)

---

### Recommendation

**Recommended disposition:** A

The reporter is the IG editor, the proposed resolution already
includes a built-in equivalence-of-published-artifacts acceptance
gate, and the change aligns RTPBC with the dominant authoring pattern
across HL7 US Realm IGs (FSH + SUSHI, raw XML only where unavoidable).
The maintainability case is strong: future Pharmacy WG tickets that do
need to change RTPBC profiles will be far cheaper to review and apply
once the source is FSH. Disposition B's two-cycle staging is more
cautious and would be the right fallback if the WG is uncomfortable
with a single bulk rework, but in the absence of slicing-heavy
profiles or community-reported concerns it is probably more ceremony
than the change warrants. Disposition C is not recommended: even if
one accepts that pure refactors don't strictly need WG authorisation,
running this through the normal disposition path gives the editor
explicit cover and creates an auditable record of the equivalence
check.
