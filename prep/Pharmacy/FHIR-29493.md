## Summary

Josh Mandel raises two related concerns about the search workflow defined in
the US Prescription Drug Monitoring Program (PDMP) FHIR Implementation Guide
(see [pdmp.html#search](http://hl7.org/fhir/us/meds/2020Sep/pdmp.html#search)).
First, the IG mandates a chained-search pattern that combines patient
demographics (name, family name, birthdate) with prescription queries in a
single request; many servers cannot execute chained queries performantly, and
the requirement bakes in a single, brittle patient-matching strategy. Mandel
proposes a two-step workflow instead — search for the patient first, then
search for that patient's prescriptions — which would let the patient-matching
mechanism evolve over time (e.g., to the `$match` operation or richer
identifiers) without changing the prescription-search contract. Second, the
name-based chained search is semantically wrong for patients with multiple
`HumanName` entries: independent matching of `family` and `given` across
different `name` instances would return false positives (the example given is
a person with names "Sherlock Smith" and "John Holmes" matching a query for
"Sherlock Holmes"). The ticket has been Triaged at Highest priority, was
previously closed as "Considered for Future Use" when the IG was withdrawn,
and was reopened in November 2023 under PSS-2225 PDMP on FHIR. There are no
cross-referenced Jira tickets, Zulip threads, or GitHub items recorded for
this issue in the local index.

## Details

**Description:**

> http://hl7.org/fhir/us/meds/2020Sep/pdmp.html#search
>
> Why require chained search parameters here? Many servers cannot support
> these in a performant way and it seems unnecessary. Why not simply have a
> first step to search for a patient and a second step to search for relevant
> prescriptions for that patient? This approach has the additional advantage
> that a patient matching technique could be adopted over time rather than
> forcing name, family name, and birthdate to be the only options for
> identifying a patient. The approach to searching based on names is also
> wrong in a subtle way for a patient that has more than one name in the
> record. The example given would find a person named "Sherlock Smith" who
> later changed their name to "John Holmes" — in other words, someone who
> was never named "Sherlock Holmes" would still match the query.

## Proposed Dispositions

> Note: With no GitHub repository surfaced for the US Medication / PDMP IG
> in the cross-references, the dispositions below describe the spec-level
> changes that would be required if/when the workgroup re-engages the IG
> per PSS-2225. They do not name a concrete authoring root or file path
> because no repo-analysis briefing is available for this IG in the local
> cache.

### Disposition A: Accept As Requested

#### Proposal

Restructure the PDMP IG's search guidance (currently at
`pdmp.html#search` in the 2020Sep snapshot) to define a two-step workflow:

1. **Step 1 — Patient resolution.** Replace the chained-search example
   with a normative requirement to first locate the Patient resource via
   `GET [base]/Patient?...`. Allow servers to advertise one or more
   patient-resolution mechanisms (demographic search, identifier search,
   and/or the `Patient/$match` operation) via the IG's
   CapabilityStatement, and require the client to use whichever mechanism
   the server advertises. Drop the implied requirement that
   `name`/`family`/`birthdate` are the only patient-identification inputs.
2. **Step 2 — Prescription retrieval.** Define the prescription query as
   a non-chained search on the resolved patient reference (e.g.,
   `GET [base]/MedicationRequest?patient=Patient/{id}&...` plus any
   PDMP-specific filters), and remove the chained-search SearchParameter
   examples from the IG.
3. **Name-matching correction.** Add explicit IG guidance that any
   demographics-based patient search MUST evaluate `family` and `given`
   within the same `HumanName` instance (matching the FHIR core search
   semantics for multi-valued nested elements), and call out the
   "Sherlock Smith / John Holmes" anti-pattern in a worked example so
   implementers don't reproduce it.

#### Justification

The change addresses both concerns Mandel raised. The two-step workflow
is the dominant pattern in other US Realm IGs that touch prescriber
queries (e.g., US Core's read-then-search guidance), so it is the
lower-friction path for implementers and avoids depending on chained
search — which several common server stacks do not support efficiently.
Decoupling patient resolution from prescription retrieval also lets the
IG layer in `Patient/$match` later without re-versioning the
prescription-search contract, which is exactly the extensibility benefit
called out in the ticket. The name-matching clarification aligns the IG
with the FHIR R4 search semantics on repeated complex elements and
removes a real source of false positives when a patient has multiple
`HumanName` entries.

---

### Disposition B: Alternative Approach

#### Proposal

Keep chained search as an *optional* convenience pattern, but make the
two-step workflow the *normative* path:

- Mark the existing chained-search example as informative ("servers MAY
  support this for low-volume convenience clients") and add a clear
  capability flag a server uses to declare support.
- Define the two-step workflow (Patient resolution → MedicationRequest
  query keyed on the resolved patient reference) as the SHALL-supported
  flow for all PDMP servers.
- Add a profile or guidance section for `Patient/$match` as the
  recommended forward-looking patient-resolution mechanism, but do not
  yet require it (acknowledging current implementer maturity).
- Apply the same name-matching correction described in Disposition A
  (intra-`HumanName` matching, worked counterexample).

#### Justification

Some existing PDMP server implementations may have already implemented
the chained-search pattern as documented; outright removing it could
strand them. A "two-step is normative, chained is optional" framing
preserves the requested architectural improvement while limiting
disruption for early adopters. It also keeps a migration runway for the
`$match`-based future state without making it a Day-1 requirement that
could block re-publication of the IG. The trade-off versus Disposition A
is mild ambiguity (servers must declare which path they support), but
that ambiguity is contained to the CapabilityStatement.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the affected guidance lives in
the 2020-September snapshot of the US Medication IG, which has been
formally withdrawn (per smrobertson's 2023-11-28 comment) and is no
longer on a publication track until the project sponsor and workgroup
re-engage under PSS-2225. Recommend that the ticket be parked as
"Considered for Future Use — re-evaluate when PSS-2225 produces a new
draft" rather than reverted-and-actioned now.

#### Justification

There is no current IG version against which the change can be applied,
no GitHub repository surfaced in the index for the US Meds / PDMP IG,
and no active community discussion (no Zulip threads, no cross-refs).
Any normative change made today would attach to a withdrawn artifact
and would need to be re-reviewed when the IG is rebooted. Declining now
avoids spending workgroup cycles on rework. The original 2023-01-18
"Considered for Future Use" disposition reached this same conclusion
4-0-0; reverting that disposition without a concrete IG re-engagement
plan risks reopening the issue without the ability to act on it.

---

### Recommendation

**Recommended disposition:** B

The substance of Mandel's feedback is correct on both points: chained
search is a known performance and capability liability in production
PDMP-style servers, and the documented name-matching example does
produce false positives across multi-`HumanName` patients. The workgroup
should bank that design decision now, while the ticket is reopened and
the issue is fresh, so that the next PDMP IG draft (under PSS-2225)
starts from the right architecture. Disposition B captures the same
architectural improvement as Disposition A but is materially safer for
any implementer who built against the 2020Sep guidance, and it gives
the workgroup an explicit place to layer in `Patient/$match` once
implementer maturity catches up. Disposition C is defensible given the
withdrawn-IG status, but the ticket has already been re-opened
specifically because PSS-2225 is reviving the work — pre-resolving the
design question now avoids re-litigating it later.
