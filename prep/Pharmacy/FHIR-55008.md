## Summary

The ticket asks whether `MedicationAdministration.performer.actor` should
continue to permit a reference to a specific `Device`, now that
`MedicationAdministration` has a dedicated `device` element (added when
the resource was reworked to use `CodeableReference`). The submitter
(Ron G. Parker) appears to be raising this as imported "Comment 34" on
behalf of a ballot/review process. Pharmacy WG triage feedback (from
costateixeira) frames two possible reads of the request: (1) make sure
both `Device` and `DeviceDefinition` are usable wherever a device is
referenced, or (2) confirm/clarify the semantic distinction between
`performer.actor = Device` (the device acts as the performer) and the
new `MedicationAdministration.device` element (the device used during
administration). No cross-referenced Jira tickets, GitHub issues/PRs,
or directly-linked Zulip threads were surfaced for this ticket; related
prior FMG/committers/Implementers discussions confirm both
`performer.actor` and `device` were changed to `CodeableReference` as
part of the R5 rework, but did not address whether `Device` should be
removed from `performer.actor`'s target list.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove `Device` from the target-profile list of
`MedicationAdministration.performer.actor` (the `CodeableReference`
target set). Tighten the element definition to state that
`performer.actor` is restricted to human/organizational actors —
`Practitioner`, `PractitionerRole`, `Patient`, `RelatedPerson`,
`CareTeam`, `Group` — and explicitly direct implementers who need to
record the physical device that delivered the medication to use the
new `MedicationAdministration.device` (`CodeableReference(Device |
DeviceDefinition)`) element instead. Add a short note to the
`MedicationAdministration` notes/scope page explaining the split
("`performer` = who is responsible; `device` = what equipment was
used") and update the `MedicationAdministration-performer` SearchParameter
target list accordingly.

#### Justification

The introduction of `MedicationAdministration.device` (confirmed in the
fmg "CodeableReference usage in Core" inventory and the prior
committers thread) created a dedicated home for "the device that
administered the medication." Continuing to allow `Device` on
`performer.actor` in addition is redundant and ambiguous: the same
device could be expressed in two places, with no rule to disambiguate.
Pharmacy WG triage commentary already raised this as an oversight to
"check after introduction of devicedefinition." Restricting
`performer.actor` to non-device actors aligns with how `performer` is
modelled on most other clinical resources and removes the ambiguity at
the cost of a (small) breaking change for any implementer who put the
device in `performer.actor` rather than `device`.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Device` on `MedicationAdministration.performer.actor` but
clarify the semantic split rather than narrow the type set. Update the
element definition for `performer.actor` to say that `Device` is only
appropriate when the device is the *responsible actor* (e.g., an
autonomous infusion pump or closed-loop insulin system that decided
and carried out the administration without a human in the loop), and
update `MedicationAdministration.device` to say it captures the
physical equipment used during administration (including when a human
is the responsible performer). Add an invariant or guidance note that
discourages duplicating the same `Device` in both `performer.actor`
and `device`, and add an example showing the two patterns
(human-performer + device used; autonomous-device performer).

#### Justification

There are real use cases for a device acting as a responsible
performer (closed-loop / autonomous administration), as raised in the
implementers "Immunization FHIR block vote" thread when the same
question was discussed for `Immunization.performer.actor`. Removing
`Device` from `performer.actor` (Disposition A) would lose the ability
to record those scenarios with the same modelling pattern used
everywhere else and would be a normative breaking change for
implementers already using it that way. A guidance-only fix preserves
both expressivity and backward compatibility while addressing the
ambiguity that motivated the ticket. The cost is that the constraint
becomes "soft" (documentation + invariant) rather than enforced by
the structural type set.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-55008 as "Persuasive with Mod – Not Persuasive" /
"Considered – No Change": close without changing the target list of
`MedicationAdministration.performer.actor`. State that the existing
behaviour is intentional and that the new
`MedicationAdministration.device` element complements rather than
replaces the ability to record a `Device` as a responsible performer.

#### Justification

`MedicationAdministration.performer.actor` has allowed `Device` since
before R5 (see the 2019 implementers thread "Specialization in Admin
resources" and the 2022 implementers / Immunization discussion).
Removing it now would be a normative breaking change against an
element used by existing implementers and would diverge from how
`performer.actor` is modelled on sibling resources
(`MedicationDispense`, `Procedure`, `ServiceRequest`) where `Device`
is also a valid actor. The submitter ("Comment 34") asks the question
but does not provide a concrete interoperability problem, ambiguous
instance, or implementer pain point that would justify a normative
change at this stage of the cycle.

---

### Recommendation

**Recommended disposition:** B

The ticket is fundamentally a clarification request, not a bug report
— Ron G. Parker is asking *whether* `Device` should still be allowed,
and Pharmacy triage explicitly framed it as either a duplication issue
or a semantic-clarity issue. Disposition A (remove `Device`) is the
cleanest model but is a normative breaking change that would invalidate
existing implementations of autonomous-device administration scenarios
that the WG has previously acknowledged as valid (closed-loop pumps,
autonomous administration). Disposition C (decline) leaves the
ambiguity in place, which is precisely what the submitter is flagging.
Disposition B threads the needle: it preserves backward compatibility
and the autonomous-performer use case while resolving the ambiguity
between `performer.actor = Device` and the new `device` element
through tightened definitions, an example pair, and a discouraging
invariant on duplication. It is also the lowest-risk path to convert a
"Comment 34"-style observation into actionable spec text without
opening a wider redesign of `performer.actor` across the medication
resources.
