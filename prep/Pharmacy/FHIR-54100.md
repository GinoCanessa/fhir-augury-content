## Summary

The ticket asks the Pharmacy work group to add explicit guidance to the
`MedicationStatement` resource — specifically in its *Scope and Usage* narrative
— on how implementers should represent the case where a medication was **not**
administered (or, more broadly, not taken). Today the resource carries a
`status` element whose codes include `not-taken`, but the narrative does not
walk the reader through the modelling pattern (status code, reason, dosage
context) that should be used when recording a non-event. The single comment
from `costateixeira` proposes that the guidance simply spell out the semantics
of `status = not-done` (the equivalent in event-resource terminology) — i.e.
"this event, described with all of these details, did not happen" — so that
implementers understand the negation applies to the fully-described event, not
to an unspecified omission.

There are no direct Jira cross-references and no linked GitHub PRs/issues for
this ticket, so the request stands on its own. The closest community context
comes from older Zulip threads about `MedicationStatement.status` semantics and
the boundary between `MedicationStatement`, `MedicationRequest` and
`MedicationAdministration`, which inform but do not pre-decide the disposition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a short subsection to the `MedicationStatement` *Scope and Usage*
narrative that walks the implementer through representing non-administration /
non-taking. Concretely:

- State that to record that a medication was **not** taken or **not**
  administered, set `MedicationStatement.status` to `not-taken` (and use the
  analogous "this event, fully described, did not happen" semantics that
  `costateixeira` proposed for `not-done` in the comment).
- Clarify that the rest of the resource (medication, dosage, effective[x],
  subject, etc.) describes the *event that did not occur*, not a separate
  hypothetical event.
- Recommend `statusReason` (and/or `reasonCode` where appropriate) to capture
  *why* the medication was not taken (e.g. patient refusal, contraindication,
  not available).
- Link explicitly to `MedicationAdministration` for the case where the
  not-administered event is being recorded by a clinician/agent at the point
  of care, so readers can choose the right resource.

#### Justification

This is a `Technical Correction` against narrative that already implies the
behaviour but does not state it. The `not-taken` code already exists on the
resource, and the work group has consistently positioned `MedicationStatement`
as the patient-assertion view of medication use (per the
`MedicationStatement.status` and "MedicationRequest prescribed vs
dispensed/administered" Zulip threads). Codifying the pattern in *Scope and
Usage* is low-risk, non-breaking, aligns with `costateixeira`'s comment, and
removes a recurring source of implementer confusion without changing any
element definitions, value sets, or invariants.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding the guidance only to `MedicationStatement`'s *Scope and
Usage*, address non-administration as a **cross-resource pattern** in the
medication module's overview page:

- Add a new "Representing non-administration / non-taking of a medication"
  section to the medications module overview that contrasts:
  - `MedicationStatement.status = not-taken` for patient-asserted non-use,
  - `MedicationAdministration.status = not-done` (with `statusReason`) for
    clinician-recorded non-administration at the point of care,
  - `MedicationRequest.doNotPerform = true` for explicitly *prohibiting*
    administration.
- From `MedicationStatement`'s *Scope and Usage*, add a short paragraph and
  link into that shared section, rather than duplicating the full guidance.

#### Justification

The underlying confusion is not unique to `MedicationStatement` — implementers
routinely conflate "patient didn't take it", "clinician didn't give it", and
"clinician forbade it". A single, authoritative cross-resource explanation is
more durable than per-resource prose that may drift, and it uses the existing
`doNotPerform` and `MedicationAdministration.status = not-done` machinery
instead of treating `MedicationStatement` as the only home for the concept.
The trade-off versus Disposition A is broader scope (touches the module page
and possibly cross-links from `MedicationAdministration` and
`MedicationRequest`) and a longer ballot review, but the resulting guidance
is more useful to implementers.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed by existing mechanism**: close as "Not
Persuasive" with a note that `MedicationStatement.status` already includes a
`not-taken` code, that the `code system / value set` definitions describe its
intent, and that `statusReason` / `reasonCode` are available for the
explanation. Recommend the reporter raise a more specific defect if a
particular sentence in *Scope and Usage* is misleading.

#### Justification

The mechanism to record non-administration on `MedicationStatement` already
exists and is documented in the element definitions and the status value set.
A purely narrative addition risks duplicating, and eventually contradicting,
the normative element definitions. If the workgroup is conservative about
narrative changes during the current cycle (e.g. wanting to avoid a new round
of translation/style review for a non-defect), declining and pointing the
reporter at the existing artifacts is defensible.

---

### Recommendation

**Recommended disposition:** A

The request is small, well-scoped, has an explicit constructive comment from
`costateixeira` already describing the wording approach, and is filed as a
`Technical Correction` rather than a substantive change. The status code
(`not-taken`) and supporting elements (`statusReason`, `reasonCode`) are
already in place, so all the change does is make the existing capability
discoverable from the resource's *Scope and Usage* — exactly the kind of
narrative tightening Pharmacy can land without ripple effects on profiles,
value sets, or implementer code. Disposition B is attractive in principle but
expands scope beyond what the reporter asked for and would be better filed as
a follow-up ticket against the medications module overview if the WG agrees
the pattern is worth documenting cross-resource. Disposition C understates
the real-world confusion the Zulip history shows and would leave a
known-fixable narrative gap in place.
