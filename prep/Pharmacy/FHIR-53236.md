## Summary

Rik Smithies is asking the Pharmacy work group to decide whether the
`Reference(Substance)` choice on `Medication.ingredient.item` is still
needed. The question is a follow-on to
[FHIR-43166](https://jira.hl7.org/browse/FHIR-43166) (BR&R, resolved
Persuasive — Duplicate), under which `Medication.ingredient.item` is
gaining an additional `Reference(SubstanceDefinition)` target so that
definitional information can still be carried after the Substance
resource was tightened to "instance only" semantics. With
`SubstanceDefinition` covering the definitional case, the only remaining
reason to keep `Reference(Substance)` on `Medication.ingredient.item` is
when the medication is an actual instance of a compounded drug whose
ingredients are themselves known instances of substance lots. Rik's
proposal is to keep `Reference(Substance)` on the assumption that this
instance-level use case does occur, but he is explicitly soliciting
confirmation from Pharmacy; if the work group believes the ingredient is
never an actual `Substance` instance, the `Reference(Substance)` choice
should be dropped from `Medication.ingredient.item`.

## Details

**Description:**

For [FHIR-43166](https://jira.hl7.org/browse/FHIR-43166) this element is
getting a extra reference to SubstanceDefinition, so that definitions
can still be carried, even though Substance resource no longer allows
that.

I don't plan to remove the reference to Substance, in case an instance
is needed.

Is the ingredient ever an actual known instance of a substance?
Presumably yes, when Medication is an actual instance of a compounded
drug (uncommon, but possible).

But if not, then Substance needs to be removed.

## Linked Jira Tickets

### [FHIR-43166: Substance vs. SubstanceDefinition](https://jira.hl7.org/browse/FHIR-43166)

Hans Buitendijk's BR&R ticket asked the project to stop letting the
`Substance` resource straddle both "definitional" and "instance" use,
and to make `SubstanceDefinition` the unambiguous home for definitions
(including a canonical URL so other resources can reference it). It was
resolved Persuasive and closed as Duplicate in 2025-09. The fallout from
that resolution — adding `Reference(SubstanceDefinition)` to
`Medication.ingredient.item` so a definitional ingredient can still be
expressed — is exactly what triggered FHIR-53236, which now asks whether
the legacy `Reference(Substance)` target is still earning its place.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Leave `Medication.ingredient.item` as a `CodeableReference` whose
allowed Reference targets are `Substance | SubstanceDefinition |
Medication` (i.e., keep `Reference(Substance)` alongside the new
`Reference(SubstanceDefinition)` added under FHIR-43166). No spec text
or schema change is required beyond confirming, in `Medication.ingredient`
guidance, that `Substance` is reserved for the case where the ingredient
is an actual instance — typically for compounded medications where the
specific lot of an ingredient is being recorded — and `SubstanceDefinition`
is the definitional target for everything else.

#### Justification

This matches Rik's stated default and preserves the `Substance`
modelling path that implementers in the related Zulip threads
(`Link Medication with Dosage`, `multiple profile references on choice
item`) have demonstrated using. Compounding pharmacies and certain
high-control workflows do track ingredients down to the specific
`Substance` instance/lot, so dropping the target would force those
implementers into an extension. Because the FHIR-43166 work is adding
`SubstanceDefinition` as a new target rather than replacing one, keeping
`Substance` is a backwards-compatible no-op that does not introduce
ambiguity (Substance is now narrowly scoped to instance use, so the two
targets do not overlap semantically).

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Reference(Substance)` on `Medication.ingredient.item` for R6, but
deprecate it via a `standards-status` / `obligation`-style note in the
element comments stating that `Reference(Substance)` is intended only
for compounded-medication / lot-level ingredient instances and that
implementers SHOULD prefer `Reference(SubstanceDefinition)` (or the
inline `CodeableConcept`) for all other cases. Also add an example to
`Medication.ingredient` that shows the `SubstanceDefinition` pattern as
the default, and one compounding example that shows the `Substance`
pattern. Revisit removal in R7 once usage data is in.

#### Justification

This addresses the underlying design concern — the new
`SubstanceDefinition` target should be the obvious default — without
silently breaking the implementers documented in the Zulip threads who
already point at `Substance`. The dual-target choice on a
`CodeableReference` is exactly the kind of element where implementers
benefit from explicit guidance about which target to pick in which
situation; without that guidance, FMG's experience (per the
`CodeableReference usage in Core` thread) is that picker confusion
becomes a recurring question. Deferring the actual removal to a future
release also gives the work group time to confirm whether the
compounding use case is real in practice or only theoretical, which is
the open question Rik flagged.

---

### Disposition C: Decline (Remove `Reference(Substance)`)

#### Proposal

Treat the technical correction as confirmed: remove `Reference(Substance)`
from `Medication.ingredient.item`, leaving the allowed targets as
`SubstanceDefinition | Medication` (plus the existing `CodeableConcept`
side of `CodeableReference`). Update `Medication.ingredient` element
comments to state that ingredient identity is always definitional, and
add a migration note that any prior use of `Reference(Substance)` should
move to `Reference(SubstanceDefinition)`.

#### Justification

This is defensible if Pharmacy's position is that an ingredient is
inherently a *kind of* substance, not a particular lot — i.e., the
identity that matters in `Medication.ingredient` is always definitional,
even for compounded medications, where lot-level traceability would
typically be carried elsewhere (e.g., on `MedicationDispense` or via
`AdministrableProductDefinition` / `ManufacturedItemDefinition`). It
honours the FHIR-43166 cleanup intent — making `SubstanceDefinition` the
unambiguous home for substance definitions — and removes a target that
is rarely justified in practice. The cost is breaking instances and
profiles that currently bind to `Reference(Substance)` (the
`multiple profile references on choice item` thread shows at least one
QI-Core profile in that shape), so the change has to be called out as a
breaking technical correction with a clear migration note.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible starting point because the ticket
itself frames the question as open ("Presumably yes … but if not, then
Substance needs to be removed"). Disposition A locks in the status quo
without resolving the ambiguity that FMG and implementers already
encounter on `CodeableReference` choices, and Disposition C removes a
target that real-world threads (`Link Medication with Dosage`,
`multiple profile references on choice item`) show implementers using —
which would convert this from a technical correction into a breaking
change without a confirmed mandate from Pharmacy. Documenting
`SubstanceDefinition` as the default while leaving `Substance` available
for compounded-medication / lot-level ingredient use lets the work
group land the FHIR-43166 follow-through cleanly now and revisit
outright removal in a future release once compounding implementers have
been polled.
