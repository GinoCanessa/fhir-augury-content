## Summary

The ticket asks the Pharmacy Work Group to re-anchor the canonical URL(s)
for one or more Code Systems defined by the US Specialty Rx (FHIR) IG so
they live under `terminology.hl7.org` (THO) rather than under
`hl7.org/fhir` (or the IG's own canonical base). A companion UTG ticket
would then be opened to register the Code System(s) in THO. The reporter
points to TSMG's January 2022 Code System Identifier Task Force material
and to the R5 terminologies guidance as best-practice references, and
notes that TSMG should be consulted for further questions. The ticket
was originally resolved "Considered for Future Use" in January 2023
("This change will be considered when the policy related to anchoring of
IG-defined terminology is finalized.") and was then reverted and
re-opened by Lynn Laakso on 2023-05-04 "post STU2 publication," meaning
the change is now in scope for the next publication cycle of US
Specialty Rx. The "2" in the title indicates this is a follow-up /
duplicate of an earlier identically-themed THO-anchoring ticket from the
same import batch.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the request and re-anchor every Code System currently defined
inside the US Specialty Rx (FHIR) IG so its canonical `url` is under
`http://terminology.hl7.org/CodeSystem/<id>` instead of any
`hl7.org/fhir/...` or IG-local base. Concretely, for the next
publication of US Specialty Rx, the Pharmacy WG would (1) inventory
every `CodeSystem` resource the IG publishes, (2) change each
`CodeSystem.url` to a THO canonical, (3) update every `ValueSet`,
profile, example, and narrative reference inside the IG to the new
canonical, (4) keep the old canonical published as a deprecated alias
note in the IG history page where feasible, and (5) raise a companion
UTG ticket per Code System to register the Code System (and its `url`
move) in THO so the binding will resolve from the central terminology
server. Coordination with TSMG is part of the disposition, per the
description.

#### Justification

This is the disposition the description literally asks for and is
explicitly aligned with the TSMG Code System Identifier Task Force
guidance (Jan 2022) and the R5 terminologies-section best practice
linked from the ticket. Anchoring IG-defined Code Systems in THO is the
established HL7 pattern for any code system the WG expects to be
referenced from outside its source IG; it lets implementers resolve the
Code System from a single terminology home and avoids the
"IG-canonical-equals-publication-URL" coupling that has caused issues
for other IGs. The original "Considered for Future Use" resolution was
reverted specifically because STU2 had been published and the WG no
longer had the "policy not yet finalized" excuse to defer.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the THO move surgically rather than wholesale: keep IG-internal,
draft-quality Code Systems on the IG's existing canonical base, but move
to THO **only** those Code Systems that (a) are normative or
trial-use-stable, (b) are or are likely to be referenced from outside US
Specialty Rx, or (c) overlap conceptually with codes already curated in
THO. For each Code System that does move, file the corresponding UTG
ticket and add a `CodeSystem.identifier` carrying the prior canonical
so existing implementer code that resolves the old URL has a documented
migration path. Code Systems that remain on the IG canonical get a
narrative note explaining why they are out of scope for THO anchoring
(e.g., experimental, IG-internal enumerations).

#### Justification

A blanket move can be heavier than the underlying need warrants and
incurs UTG-registration overhead even for Code Systems that are
genuinely IG-private. A risk-tiered approach preserves the spirit of
the TSMG guidance (any externally-referenced Code System lives in THO)
while limiting churn and UTG workload for the Pharmacy WG. It also
gives implementers a clean migration story — `CodeSystem.identifier`
preserving the old canonical is the FHIR-native way to bridge the
rename without breaking existing references.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Mod / Not Persuasive" on the basis
that the ticket is a generic duplicate of an earlier Pharmacy THO ticket
("...THO. 2" indicates it was the second comment of an imported batch),
that it does not name a specific Code System, and that the underlying
TSMG anchoring policy has since been re-stated normatively elsewhere.
Direct any concrete THO-anchoring work into per-CodeSystem tickets that
identify the specific resource to move and the target THO canonical,
rather than carrying a broad-brush ticket forward.

#### Justification

The ticket as written is policy advice, not an actionable spec change:
no specific `CodeSystem.url` is named, no migration plan is attached,
and the "Considered for Future Use" history shows the WG has previously
treated it as policy-bound rather than IG-bound. If the Pharmacy WG
wants the move to actually happen, ticket-per-CodeSystem is more
auditable for ballot reconciliation than one umbrella ticket that is
hard to mark "done."

---

### Recommendation

**Recommended disposition:** B

The TSMG guidance the ticket cites is real and the post-STU2 re-open
makes deferral hard to defend, so a flat decline (C) ignores the
substantive reason the ticket exists. But Disposition A is broader than
the evidence in the ticket supports — the description does not actually
enumerate which Code Systems are in scope, and a wholesale move drags
every IG-internal enumeration through UTG with it. Disposition B keeps
the Pharmacy WG aligned with TSMG best practice for any Code System
that is or could be referenced from outside US Specialty Rx, files the
UTG tickets that the description explicitly asks for, and protects
existing implementers via `CodeSystem.identifier` aliases on the moved
canonicals. It is also the disposition most likely to survive TSMG
review without a second round-trip, because it forces the WG to
articulate, per Code System, why each is or is not THO-bound.
