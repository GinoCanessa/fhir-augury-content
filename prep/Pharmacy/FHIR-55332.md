## Summary

This ticket asks the Pharmacy work group to evaluate whether one of the
value sets used by the `Medication` resource — specifically the
`medication-ingredientstrength` value set (backed by the
`MedicationIngredientStrengthCodes` code system) bound to
`Medication.ingredient.strength[x]` — is a candidate for migration from
the FHIR Core specification to HL7 Terminology (THO). The reporter notes
that migration is not required, but should be considered when the value
set definition is likely to evolve over the coming years and/or when the
value set could be useful in other product families such as CDA or
V2. The ticket was imported from a comment (Comment 296) on a broader
THO-candidate review by Lloyd McKenzie. No companion Jira tickets were
surfaced for this specific item — it stands alone as the Pharmacy
work group's instance of that broader THO grooming exercise.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the `medication-ingredientstrength` value set and its backing
`MedicationIngredientStrengthCodes` code system from the FHIR Core
specification (`Medication` resource definition area) to HL7
Terminology (THO). After migration, update the binding on
`Medication.ingredient.strength[x]` so that it points to the THO-hosted
canonical URLs for the value set (and, where referenced, the code
system). Preserve the existing binding strength and concept content,
add THO maintenance/governance metadata, and ensure the FHIR Core
build retains a redirect/reference so that existing consumers
resolving the legacy canonical continue to work.

#### Justification

This is precisely the action the ticket invites: the value set is
narrow in scope, descriptive of an attribute (ingredient strength
qualifier) that is plausibly reusable across CDA and V2 medication
representations, and small enough that a migration carries low
disruption risk. THO hosting also gives the codes a maintenance
lifecycle independent of FHIR's release cadence, which is helpful if
new strength qualifiers (e.g. additional reference-based strength
modifiers) need to be added between FHIR ballots. Aligns with the
broader THO-migration grooming exercise from which this ticket was
imported.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the value set in FHIR Core for the upcoming release, but formally
**flag it for THO migration in the next major cycle** by (a) adding a
note to the value set's narrative declaring the migration intent and
target THO canonical, and (b) coordinating with the Vocabulary work
group and the Pharmacy CDA/V2 stewards to confirm cross-product demand
before changing the canonical URL. No binding change in this cycle.

#### Justification

Migration changes the canonical URL of the value set, which is a
breaking change for any implementer or profile that pinned the FHIR
Core canonical. Doing the analysis-and-announce step first lets the
work group confirm there is genuine cross-product appetite (CDA / V2)
before paying that migration cost, and gives implementers a release of
warning before the canonical moves. This is consistent with the
ticket's own framing — migration is "not required" and should be
considered against the realistic likelihood of definitional change and
cross-product reuse, neither of which has been demonstrated yet for
this particular value set.

---

### Disposition C: Decline

#### Proposal

Decline the migration. Leave `medication-ingredientstrength` and
`MedicationIngredientStrengthCodes` in FHIR Core, bound as today on
`Medication.ingredient.strength[x]`. Close the ticket as "not
persuasive" (or equivalent) with a note that the value set is FHIR-
Medication-specific, has been stable, and there is no evidence of CDA
or V2 demand to justify the canonical-URL churn that migration would
cause.

#### Justification

The ticket itself states migration is optional. The
`medication-ingredientstrength` value set is small, has been stable
across recent releases, and is tightly coupled to a single FHIR
element. Without a concrete cross-product use case or a pending change
to the value set's contents, migrating purely on the speculative
grounds of "might be reusable" imposes a real cost (canonical change,
cross-IG ripple effect, profile updates) for no demonstrated benefit.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible starting point for the work
group. Disposition A commits to a breaking canonical change before
either the cross-product demand (CDA, V2) or the likelihood of future
definitional churn has been substantiated, and the ticket itself
acknowledges those are the two preconditions that should drive
migration. Disposition C is too conservative because the ticket is the
output of an explicit THO-grooming exercise that the Pharmacy WG
agreed to participate in — closing it without any analysis ignores
that commitment. Disposition B preserves the migration option,
captures the intent in the value set narrative so implementers are not
surprised later, and forces the work group to do the lightweight
cross-product check that should precede any canonical move.
