## Summary

The reporter asks the consumer-facing US CARIN Real-time Pharmacy Benefit
Check (RTPBC) to expose, as part of the response, the amount that is paid
by the PBM (i.e., the plan-paid portion) so that a patient can understand
how a given fill consumes their pharmacy benefit and can estimate how much
benefit remains before they hit a plan-defined maximum (e.g., an annual
benefit cap). The request is targeted at the value set drawn from
`http://hl7.org/fhir/us/carin/CodeSystem/rtpbc-patient-pay-type`, which
currently enumerates ten patient-pay categories but does not include any
plan-paid / accumulator / remaining-benefit concept.

The ticket was originally dispositioned "Considered for Future Use"
(12-0-0, 2020-04-13, Frank McKinney / Daniel Zhang) and re-opened by
Lynn Laakso after STU1 publication so that accumulator / balance
information that could impact the patient's cost in this or future fills
could be revisited.

A closely related, more recent ticket — [FHIR-36294](https://jira.hl7.org/browse/FHIR-36294)
— covers exactly the accumulator side of this concern: it asks for
deductible accumulator fields (NCPDP RTPB v11 Accumulated Deductible
Amount 512-FC and Remaining Deductible Amount 513-FD) to be added to the
RTPBC response. FHIR-36294 has already reached resolution **Persuasive**
with the resolution text "Add accumulated-deductible and
remaining-deductible codes to the RTPBC Patient Pay Type code system
used in the RTPBC response (FHIR ClaimResponse profile)". That work is
the natural vehicle for satisfying the spirit of FHIR-26507 — but it
addresses *deductible* accumulators, not the broader "amount paid by the
PBM" / remaining-benefit view that this ticket describes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend `CodeSystem/rtpbc-patient-pay-type` (and the bound ValueSet) with
new concept(s) representing the plan/PBM-paid portion and the remaining
benefit, so the RTPBC `ClaimResponse` can return them via existing
`adjudication` slices (the same pattern used by C4BB). Concretely:

- Add `plan-paid` (display: "Amount paid by the plan / PBM") — the dollar
  amount the PBM/plan is paying for this fill at this pharmacy under this
  benefit.
- Add `accumulated-benefit` and `remaining-benefit` (display: amount
  applied to / remaining under a defined plan benefit maximum, e.g.,
  annual pharmacy benefit cap), distinct from the already-agreed
  `accumulated-deductible` / `remaining-deductible` codes from
  FHIR-36294.
- In the RTPBC ClaimResponse profile, allow these codes in
  `ClaimResponse.item.adjudication.category` (and require `amount` /
  `value` accordingly), and add narrative guidance that these values are
  estimates calculated by the PBM at request time and do not modify the
  patient's stored accumulators (mirroring the NCPDP RTPB IG language
  cited in FHIR-36294).

#### Justification

This is the smallest change consistent with how RTPBC already models
financial breakdowns and is a natural extension of the work already
agreed in FHIR-36294 (which extends the same code system for
accumulated/remaining deductible). Reusing
`ClaimResponse.item.adjudication` keeps RTPBC aligned with C4BB's
adjudication pattern and with the FM-WG-recommended use of `Claim` /
`ClaimResponse` in predetermination mode (per the implementers Zulip
thread). It directly satisfies the Block-Vote-4 expectation captured in
the re-open comment that the IG should inform patients of "accumulator /
balance information that could impact their cost in this or future
fills." The NCPDP RTPB v11 standard (referenced by FHIR-36294) provides
precedent for both the field semantics and the "informational / estimate"
caveats that need to ride alongside.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add new patient-pay codes for plan/PBM-paid amounts. Instead,
satisfy the underlying need by:

1. Folding FHIR-26507 into the already-Persuasive disposition of
   FHIR-36294 — the deductible accumulator addition addresses the
   majority of the "what's left in my coverage" concern in the U.S.
   pharmacy benefit context (where the deductible drives most patient
   cost behaviour).
2. For the literal "PBM-paid amount", reuse FHIR `ClaimResponse.payment`
   (or `ClaimResponse.total` with category `benefit`) rather than
   introducing a new `rtpbc-patient-pay-type` code. The ClaimResponse
   resource already has standard slots for plan-paid totals; `Coverage`
   and `ExplanationOfBenefit` provide separate mechanisms for benefit
   maxima.
3. Add only narrative guidance to the RTPBC IG describing how
   implementers should populate these existing elements when the PBM
   wishes to disclose the plan-paid portion.

#### Justification

The `rtpbc-patient-pay-type` code system is, by name and by the existing
ten concepts, scoped to *patient pay* categories. Stretching it to carry
plan-paid and benefit-cap concepts overloads the vocabulary and blurs the
boundary the C4BB / financial resources have been carefully drawing
between adjudication categories. The deductible-accumulator codes already
in flight under FHIR-36294 satisfy the dominant use case the reporter
described. For the residual "what the PBM paid" need, FHIR already
provides typed elements (`ClaimResponse.payment`, `ClaimResponse.total`)
that don't require new vocabulary. This avoids a potentially breaking
expansion of a published STU value set and reduces interop ambiguity for
existing implementers. The trade-off vs. Disposition A is reduced
expressivity for "remaining annual pharmacy benefit" specifically — but
that information is arguably a `Coverage` / EOB concern, not an RTPBC
predetermination concern.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-26507 as **already addressed** by FHIR-36294. State on the
ticket that the deductible accumulator codes added under FHIR-36294
(`accumulated-deductible`, `remaining-deductible`) provide the
patient-cost / remaining-coverage visibility that motivated this request,
and that out-of-pocket maxima and annual benefit caps are out of scope
for a per-fill predetermination check — they belong in `Coverage` /
`ExplanationOfBenefit`. Close as "Not Persuasive — Already Addressed" and
link to FHIR-36294.

#### Justification

The ticket is from 2020 and was re-opened in 2020 specifically to
revisit "accumulator / balance information." FHIR-36294 (2022, resolution
Persuasive in 2025) is exactly that follow-up and has already passed the
WG. RTPBC is intentionally a per-fill predetermination check and not a
benefit-summary report; the literal "what's left before I hit my PBM
max" question is a `Coverage` / `ExplanationOfBenefit` question, not a
`ClaimResponse` predetermination question (consistent with the
implementers Zulip thread on resource-choice boundaries). Adding
plan-paid / benefit-maximum concepts to the patient-pay code system
risks mis-scoping the IG and creating duplication with the broader
financial resources. This would also avoid expanding a published STU
value set with concepts that have not been validated against PBM
implementer feedback.

---

### Recommendation

**Recommended disposition:** B

The ticket's underlying intent — give the patient enough information to
understand how a fill affects their pharmacy benefit balance — is
substantially met by the FHIR-36294 deductible-accumulator additions
that already cleared the WG with Persuasive resolution; that work
extends the very code system this ticket cites. Disposition C captures
that fact but throws away the small but real residual ask (visibility of
the actual plan/PBM-paid amount), which is genuinely useful and is also
consistent with how `ClaimResponse` is used elsewhere. Disposition A
solves the residual ask but does so by overloading
`rtpbc-patient-pay-type` with non-patient-pay concepts and inventing a
new "annual benefit cap" model on RTPBC that arguably belongs in
`Coverage` / `ExplanationOfBenefit`. Disposition B threads both: pick up
FHIR-36294's accumulator codes for the dominant use case, and use
`ClaimResponse.payment` / existing typed elements (with IG narrative)
for the literal plan-paid amount, without expanding a published STU
value set in a way that may collide with future C4BB / financial-resource
harmonisation. The implementers Zulip discussion (Frank McKinney /
Lloyd McKenzie / niraj631) underscores that the financial-resource
boundaries are still evolving — keeping this change inside existing
typed slots minimises rework if those boundaries shift again.
