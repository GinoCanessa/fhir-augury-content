## Summary

The ticket requests that a new boolean element named `timingExact` be added
to the "Dosage Safety" grouping on the `Dosage` data type. The reporter
points out that some dosing instructions — antibiotics are given as the
canonical example — must be administered as close as possible to the
prescribed time, and that there is currently no first-class way to
communicate this safety-relevant fact in `Dosage` itself. The reporter
explicitly references the existing R4 modifier extension
`http://hl7.org/fhir/StructureDefinition/timing-exact` (and its use in the
Dutch nictiz `medicationprocess9` package) as the prior art that should be
promoted into the core `Dosage` type. No cross-referenced Jira tickets
were surfaced by the cross-reference index, but a closely related
historical thread (`Timing datatype puzzle` on chat.fhir.org) and the
related ticket [FHIR-39336](https://jira.hl7.org/browse/FHIR-39336)
("Timing datatype needs Time to time bounds", resolved Persuasive in
FHIR Infrastructure) already establish that imprecise/strict dosing time
semantics are a recurring real-world need.

### [FHIR-39336: Timing datatype needs Time to time bounds](https://jira.hl7.org/browse/FHIR-39336)

Filed by Bart Decuypere (FHIR Infrastructure) in 2022 and resolved
"Persuasive" in late 2022, this ticket asked for a way to express a
between-time-and-time bound on `Timing` (e.g., "administer between 13:00
and 15:00"). It was dealt with as part of the broader imprecise-period
extensions created under FHIR-38726. It is conceptually adjacent to
FHIR-54339: both arise from the observation that the existing
`Timing`/`Dosage` machinery is not expressive enough about how strictly
an administration time should be honoured. FHIR-39336 dealt with the
"window" side of the problem; FHIR-54339 deals with the "must be exact"
modifier side.

Reporter: Bart Decuypere; Assignee: Unassigned.

extension as the established Dutch (Nictiz) pattern: pick a `timeOfDay`

agreement that "how strict is this time?" is a real, recurring,
safety-relevant need; the disagreement is mostly about *which* mechanism
to add to core (boolean exactness flag, time-to-time bounds,
uncertain-date-style extension, or all of the above).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new optional boolean element `Dosage.timingExact` (cardinality
`0..1`) under the existing "Dosage Safety" grouping in the core `Dosage`
data type, with semantics aligned to the existing modifier extension
`http://hl7.org/fhir/StructureDefinition/timing-exact`:

- `true` — the administration must occur as close to the time(s)
  expressed in `Dosage.timing` as possible (e.g., antibiotics whose
  half-life requires near-exact dosing).
- `false` / absent — normal clinical flexibility around the stated time
  is acceptable.

Because the existing extension is a *modifier* extension, the new
element should also be marked `isModifier = true` with
`isModifierReason = "Changes how the timing in Dosage.timing must be
interpreted; non-exact timing is acceptable in normal clinical
practice, but exact timing changes the safety profile of the order."`.
Retire / deprecate the `timing-exact` extension in favour of the new
core element, with a forward-mapping note in the extension definition.

#### Justification

The pattern is already standardised as a published HL7 modifier
extension and has demonstrated real-world implementation (Nictiz
`medicationprocess9` 2.0.0-rc.1). The "must be taken on time" semantics
are explicitly safety-relevant for antibiotics, anticoagulants
(Marcoumar was given as an example in the Zulip thread), antiretrovirals
and many other classes — exactly the kind of concept Pharmacy's "Dosage
Safety" grouping is meant to surface to consuming systems. Promoting
established modifier extensions to core when they have broad uptake is
a recognised lifecycle pattern, and doing so is more discoverable and
interoperable than leaving every implementer to find the extension by
URL. The Zulip "Timing datatype puzzle" thread shows multi-jurisdiction
agreement (NL, BE, US, INT) that the current `Timing` model
under-expresses strictness.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add `timingExact` as a dedicated boolean. Instead, address
the underlying need in two coordinated ways:

1. Republish / refresh the existing
   `http://hl7.org/fhir/StructureDefinition/timing-exact` modifier
   extension into the R6 FHIR Extensions Pack as the *recommended* way
   to express timing strictness, and reference it prominently from the
   `Dosage` and `Timing` pages and from the Pharmacy
   MedicationRequest/MedicationAdministration guidance. Pair it with
   the "time-to-time bounds" work that resolved FHIR-39336 so that
   implementers have a coherent toolkit (window vs. strictness).
2. If the workgroup judges that a boolean is too coarse, model
   strictness instead as a coded element (e.g.,
   `Dosage.timingTolerance` with a small value set such as
   `exact | tight | normal | flexible`) so that future safety-driven
   refinements (for example "±15 minutes" tolerances for IV
   antimicrobials) are not boxed in by a yes/no flag.

#### Justification

Promoting an extension into core is a one-way door for `Dosage`, which
is already a heavily-used, deeply-profiled type. The Zulip threads make
clear that "how strict?" is genuinely a spectrum, not a binary — Bart
Decuypere's case 1 (Marcoumar, must be on time), case 2 (tube feeding
across a 4-hour window) and case 3 (antibiotic must run in 30 minutes)
each pull toward different shapes. A coded tolerance element is more
future-proof than a boolean. Keeping the mechanism as a (refreshed)
extension also avoids forcing every Pharmacy profile, terminology
binding and validator across the ecosystem to migrate when the actual
implementer demand can be met today by the existing extension.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by an existing mechanism**.
Close as "Not Persuasive – existing extension". Direct the reporter to
`http://hl7.org/fhir/StructureDefinition/timing-exact`, which is a
published, modifier extension that already conveys exactly the
semantics requested and is in production use (Nictiz
`medicationprocess9`). Recommend that any IG that needs first-class
visibility of the concept profile `Dosage` to make
`extension:timing-exact` a `must support` slice rather than asking for a
core element.

#### Justification

`Dosage` is a foundational, widely-implemented type, and every new
element — especially a `isModifier` element — has a measurable cost on
implementers, profiles, IGs, examples, search parameters and
downstream tooling. The Zulip "real-world usage" thread (jy95, 2025)
shows that even baseline `Dosage` elements have uneven uptake; adding a
boolean modifier when a published modifier extension already exists
risks creating two parallel ways to express the same concept (the new
element and the lingering extension in installed R4 deployments) for no
new expressive power. The extension mechanism exists precisely so the
core spec does not have to absorb every safety annotation that some
jurisdiction needs.

---

### Recommendation

**Recommended disposition:** B

The need is real and safety-relevant, and Pharmacy should not simply
decline (Disposition C undersells the demonstrated demand visible in
both Zulip threads and in the Nictiz package). However, jumping
directly to a core boolean (Disposition A) would lock the Dosage Safety
grouping into a binary representation of what the
`Timing datatype puzzle` thread shows is genuinely a spectrum, and it
would do so on the strength of one ticket with no comments and no
in-person sponsorship recorded yet. Disposition B keeps the existing
`timing-exact` extension in play as the immediate answer for
implementers (no migration burden), while opening the door to a
better-shaped core element — likely a small coded value set rather than
a boolean — once Pharmacy has discussed it alongside the related
work that resolved FHIR-39336. If the workgroup decides during
discussion that a boolean is sufficient and the extension already has
broad-enough uptake to justify promotion, B can be upgraded to A
without any of the data-gathering being wasted.
