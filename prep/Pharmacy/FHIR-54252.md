## Summary

The submitter (Nictiz) reports that the current R6 `DosageDetails` model
cannot cleanly express a **cyclical repeat schedule** — i.e., a dose that is
taken for a window of days and then paused for a window of days, with the
whole window repeating (the canonical example being oral contraceptives:
"1 tablet daily for 21 days, then 7 days off, repeated every 28 days").
The ticket points at the published Example 5 (`dosage-examples.html#2`,
spec section 14.7.1.6) and demonstrates that the existing encoding
(`Timing.repeat.frequency=21` with `period=28 d`) is genuinely ambiguous:
it can be read as "take 1 tablet 21 times every 28 days," which would
also permit multiple tablets per day during the 21-day window. There is
no element today that says "this whole `step`/`dosageInstruction` repeats
on a 28-day cycle."

The proposal is to introduce a new element (working name
`repeatPeriodCyclicalSchedule`, typed `Duration`) on `DosageDetails` (or
`DosageDetails.step`) modeled after the Dutch national extension
[`ext-InstructionsForUse.RepeatPeriodCyclicalSchedule`](http://nictiz.nl/fhir/StructureDefinition/ext-InstructionsForUse.RepeatPeriodCyclicalSchedule).
Subsequent comments expand the scope into a small cluster of related
DosageDetails refinements: (1) reduce redundancy between
`step.component.Timing.repeat.count` and a `step.count`; (2) decide
whether to add `period`/`periodUnit` (or a single `Duration`-typed cycle
element) at `step` and/or `DosageDetails` level; (3) clarify whether
multiple `component`s within a `step` are *concurrent* (overlapping in
time) or *parallel* (occurring in the same time slot but as
distinguishable activities) and write that guidance into the spec; and
(4) clarify which `startOffset` (`step.start.offsetDuration` vs
`component.timing.repeat.startOffset`) to use for sequential dosing.

The cross-reference index returns no formally linked Jira tickets,
Zulip threads, GitHub PRs, or commits for this ticket key. However, a
keyword search shows the request lives squarely inside the active
DosageDetails redesign that was voted on in the 6.0.0 ballot cycle (see
`#Medication > Dosage`, `#Medication > backlog for next call`, and
`#Medication > How bolus dose in MedicationRequest`), and Jose
Costa-Teixeira (DosageDetails redesign author) has already engaged in
the comments asking the submitter to bring a concrete case forward.

## Details

**Description:**

In example 5 (14.7.1.6) a dosageInstruction "1 tablet daily for 21 days,
then 7 days off" is expressed with a `Timing.repeat.frequency` set to 21
and the period set to 28 days. This can be interpreted as "take 1 tablet
21 times every 28 days except for the last seven days"; it would also be
valid to take 1 tablet multiple times a day. With the current structures
and elements it is not possible to express a repeat cycle.

The submitter suggests adding a new element to `DosageDetails` (or
`DosageDetails.step`) to express the repeat cycle, modeled on the Dutch
national modifier extension
`http://nictiz.nl/fhir/StructureDefinition/ext-InstructionsForUse.RepeatPeriodCyclicalSchedule`
(see Nictiz `medicationprocess9` package). With the new element the
contraceptive example becomes:

```xml
<MedicationRequest xmlns="http://hl7.org/fhir">
  <medication>
    <concept><text value="Microgynon tab"/></concept>
  </medication>
  <dosageInstruction>
    <repeatPeriodCyclicalSchedule>
      <valueDuration>
        <value value="28"/><unit value="days"/>
        <system value="http://unitsofmeasure.org"/><code value="d"/>
      </valueDuration>
    </repeatPeriodCyclicalSchedule>
    <step>
      <component>
        <timing><repeat>
          <boundsDuration>
            <value value="21"/><unit value="days"/>
            <system value="http://unitsofmeasure.org"/><code value="d"/>
          </boundsDuration>
          <frequency value="1"/><period value="1"/><periodUnit value="d"/>
        </repeat></timing>
        <doseAndRate><doseQuantity>
          <value value="1"/><unit value="tablet"/>
          <system value="http://unitsofmeasure.org"/><code value="{tablet}"/>
        </doseQuantity></doseAndRate>
      </component>
    </step>
  </dosageInstruction>
</MedicationRequest>
```

**Comments:**

- **LauriRondolo (Nictiz, comment 1)** — "I've added more examples in
  the comments and questions."

- **LauriRondolo (Nictiz, comment 2)** — Posts a long follow-up linked
  from a Nictiz wiki, expanding the request into two themed proposals:
  - **(1) Reduce redundancy in `step.component`.** Use the existing
    `step.count` instead of `step.component.Timing.repeat.count`. Then
    debate whether to (a) add `period`/`periodUnit` directly on `step`,
    or (b) add a single `Duration`-typed `repeatPeriodCyclicalSchedule`
    on `step` (matching the Nictiz extension). Three worked Epirubicin
    examples (`ex4-proposed-structure-with-step-count`,
    `…-with-step-count-and-period`,
    `…-with-step-count-and-repeatPeriodExt`) illustrate each variant.
  - **(2) Define "concurrent" vs "parallel" components.**
    `DosageDetails` currently says "When there are more than one
    component dosages in a step, all components within the same step
    are considered concurrent." The submitter argues *concurrent*
    (simultaneous execution) and *parallel* (simultaneous occurrence)
    are not synonymous and should not be used interchangeably. Three
    sub-points: (a) if components are reserved for parallel dosing,
    sequential dosing must use multiple `step`s — which then re-creates
    the same redundancy problem at `step` level; (b) clarify whether to
    use `step.start.offsetDuration` or
    `component.timing.repeat.startOffset` for sequential cases; (c)
    introducing `period` / `repeatPeriodCyclicalSchedule` / `count` at
    `DosageDetails` level would let the redundant per-step values be
    factored out (`ex4-proposed-structure-with-dosageDetails-repeatPeriodExt-count`).

- **LauriRondolo (Nictiz, comment 3)** — Direct reply to Jose
  Costa-Teixeira after the Jan WGM, with two concrete questions:

    without `PlanDefinition`/`RequestOrchestration`: setting
    `boundsDuration=2 d` and `frequency=1` with `period=3 w` does not
    convey "take it again on day 4."

    incorrect/ambiguous and asks for the canonical R6 encoding.

- **costateixeira (2026-01-28)** — Confirms the modelling rules of the
  current redesign: "steps are sequential; within each step, components
  are in parallel; components have timing that can be used for offsets
  and timings." States hard constraints the workgroup should respect:
  the `Timing` data type cannot be changed; very complex dosages remain
  the province of `PlanDefinition` / `RequestOrchestration`; the
  decisive evidence for any change is "common cases that are *broken*
  and impossible," with "unusual implementations due to ambiguity"
  strengthening the case. Asks LauriRondolo to present a concrete case
  to demonstrate where the current approach is insufficient.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new element on `DosageDetails` (i.e., at the
`MedicationRequest.dosageInstruction` level, *not* on `step`) named
`repeatPeriodCyclicalSchedule` with cardinality `0..1` and datatype
`Duration`. The element captures the length of the recurring cycle
within which the contained `step`(s) execute and then pause; the
"on" portion of the cycle is conveyed by
`step.component.timing.repeat.boundsDuration`. The Microgynon example
becomes the encoding shown in the ticket description (28 d cycle,
21 d boundsDuration, frequency 1/d, 1 tablet). Semantics: an
implementer must repeat the entire `step` sequence every
`repeatPeriodCyclicalSchedule` until the prescription
ends/`MedicationRequest.dispenseRequest.validityPeriod`/`numberOfRepeatsAllowed`
is exhausted. Rewrite Example 5 (`dosage-examples.html#2`,
spec §14.7.1.6) to use the new element so the published example is
unambiguous, and add explanatory prose in the `DosageDetails` section
of `dosage.html`.

#### Justification

This is the minimal change that *does* close the gap the ticket
identifies: the contraceptive case ("21 days on, 7 days off,
repeating") provably cannot be expressed unambiguously today, which
matches Jose Costa-Teixeira's own bar for action ("common cases that
are broken and impossible"). The proposed element has a
production-tested precedent — the Dutch national extension
`ext-InstructionsForUse.RepeatPeriodCyclicalSchedule` — so the
semantics, name, and datatype are not invented for this ticket. Putting
it on `DosageDetails` (rather than on `step`) avoids the redundancy the
submitter calls out in proposal (1): a single cycle applies to the
whole sequence of `step`s without having to be repeated on each `step`.
It also respects the hard constraint Jose Costa-Teixeira reaffirmed in
the comments — the `Timing` data type itself does not change.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of one new element, fold the cyclical-schedule problem into a
small, coherent set of additions on `DosageDetails.step`, addressing
all four of Lauri Rondolo's sub-issues at once:

1. Add `DosageDetails.step.cyclePeriod : Duration` `0..1` to express
   the recurring cycle window (e.g., 28 d), replacing the proposed
   `repeatPeriodCyclicalSchedule`. The "on" portion of the cycle
   continues to be expressed by
   `step.component.timing.repeat.boundsDuration`.
2. Move `step.component.timing.repeat.count` up to `step.count`
   wherever it applies to the whole step (already mooted by Jose
   Costa-Teixeira and consistent with the R6 redesign), eliminating
   one source of redundancy.
3. Tighten the spec text so that *concurrent* is replaced with the
   precise term — define *parallel* `component`s (same step, same
   timing slot, distinguishable activities, e.g., diluent + active)
   and contrast with *sequential* `step`s. Specifically rewrite the
   sentence "all components within the same step are considered
   concurrent" in `dosage.html#DosageDetails`.
4. Add normative guidance that `step.start.offsetDuration` is the
   canonical place to express "this step starts N units after the
   previous step," with `component.timing.repeat.startOffset`
   reserved for offsets *within* a step's parallel components.

Then rewrite Examples 4 (Epirubicin) and 5 (Microgynon) to use the
revised structure, and write up the medrol pulse-therapy case as a new
example so the resolution of the comment-3 question is part of the
spec.

#### Justification

This disposition treats the ticket honestly — Lauri Rondolo's second
comment is not really one request, it is four interrelated ones, and
solving only the cyclical-schedule piece (Disposition A) leaves the
redundancy and concurrent-vs-parallel ambiguity unresolved. It still
respects the workgroup's hard rule against changing `Timing`, since all
new structure lives on `DosageDetails.step`. It aligns with the
direction Jose Costa-Teixeira and Grahame Grieve have already taken in
the R6 rework branch (`2025-11-gg-dosage-rework`, PR HL7/fhir#3974) and
with the backlog Jose posted to `#Medication > backlog for next call`,
so the ticket's resolution becomes a coherent member of that batch
rather than a one-off element addition. The trade-off vs A is scope:
this is more work, more breaking, and harder to slip into a late
ballot — but the alternative is to ship A now and immediately re-open
the same area in a follow-up cycle to handle the rest.

---

### Disposition C: Decline

#### Proposal

Decline the new element on the grounds that the cyclical-repeat case
is already addressable using existing R6 mechanisms — combine
`step.component.timing.repeat.boundsDuration` (the "on" window) with
either `step.start.offsetDuration` (to position the next iteration) or
`PlanDefinition` / `RequestOrchestration` (for arbitrarily complex
cycles). Where the published Example 5 is genuinely ambiguous, treat
that as a documentation defect and fix the example wording without
adding a new element — i.e., resolve as "Not Persuasive, fix
example/guidance only."

#### Justification

The comments thread already names the relevant escape valve:
Costa-Teixeira explicitly notes that "for very complex dosages we have
`PlanDefinition` / `RequestOrchestration`," and the workgroup's stated
bar for changing the data type is "common cases that are broken and
impossible." A defender of this disposition could argue that the
oral-contraceptive case, while real, is at the edge of what
`Dosage`/`DosageDetails` is meant to carry — anything more complex
than "21 on / 7 off" already needs orchestration — and that adding
yet another element to `DosageDetails` compounds the very modelling
debt the submitter is also complaining about (see the
concurrent-vs-parallel confusion). This option is the least disruptive
to the R6 ballot timeline.

---

### Recommendation

**Recommended disposition:** B

The submitter has, across the description and three comments,
documented four interlocking defects in `DosageDetails`: the missing
cycle element, the `step`/`component` count redundancy, the
*concurrent*-vs-*parallel* terminology problem, and the duplicate
`startOffset` paths. Disposition A fixes only the first; Disposition C
defers all four to `PlanDefinition` and tooling guidance — which
contradicts Jose Costa-Teixeira's own "common cases that are broken
and impossible" test (the Microgynon case, on the workgroup's own
published Example 5, is exactly that). Disposition B is the option
that matches both the breadth of the ticket and the direction of
travel already visible in `#Medication > Dosage` and the R6 rework PR
HL7/fhir#3974: take the ticket as the trigger to land a coherent set
of `DosageDetails.step` refinements, with the cyclical element at the
core. The risk is scope and ballot timing — the workgroup should be
prepared either to accept the breaking-change implications now (Jose
already named breaking change as on-the-table for this rework) or to
sequence the four sub-changes across two ballots, with the cyclical
element first and the redundancy/terminology cleanups immediately
after. Either way, the discussion should start from B, not A or C.
