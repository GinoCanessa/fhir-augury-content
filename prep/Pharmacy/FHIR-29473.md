## Summary

The ticket asks the Pharmacy work group to provide more granular mapping
guidance between the compound `<Quantity>` field (and its multiple
required subfields) carried in the `MedicationDispensed` segment of
NCPDP SCRIPT versions 2017071 and 10.6, and the corresponding
`MedicationDispense.quantity` representation in FHIR. The current IG
text simply labels the source element ("NCPDP Data Element for
Quantity") without telling implementers how to map each NCPDP subfield
(value, unit of measure qualifier, potency, etc.) into
`Quantity.value`, `Quantity.unit`, `Quantity.system`, and
`Quantity.code`. The ticket was originally closed as "Considered for
Future Use" in January 2023 because the hosting IG (PDMP on FHIR) had
been withdrawn; it was reopened in November 2023 after the PDMP on
FHIR IG was re-chartered under PSS-2225, and has since been triaged at
Highest priority but remains unassigned. There are no cross-referenced
Jira tickets, Zulip threads, or GitHub items on file, so this
disposition review is based on the ticket content alone.

## Proposed Dispositions

The dispositions below target the US Medication FHIR IG (the
ticket's stated specification) and its sister PDMP-on-FHIR IG, both
owned by the Pharmacy work group. No GitHub briefings are on file, so
path-level guidance is limited to the IG's conventional mapping/
narrative assets (mapping tables and `MedicationDispense` profile
pages); the workgroup should confirm exact file paths when executing.

### Disposition A: Accept As Requested

#### Proposal

Add an explicit NCPDP SCRIPT → FHIR mapping table for the
`MedicationDispensed` `<Quantity>` compound element covering both
NCPDP SCRIPT 2017071 and 10.6. The table should enumerate each
required NCPDP subfield (e.g., Quantity/Value, Quantity/CodeListQualifier,
Quantity/UnitSourceCode, and — where applicable — Potency) and bind
each to a specific FHIR target on `MedicationDispense.quantity` (a
`SimpleQuantity`):

- NCPDP Quantity/Value → `MedicationDispense.quantity.value`
- NCPDP Quantity/CodeListQualifier → fixed discriminator used to
  select the value set for `MedicationDispense.quantity.system`
  (e.g., `http://unitsofmeasure.org` for UCUM vs. the NCPDP
  billing-unit system) rather than appearing as a FHIR element
  itself
- NCPDP Quantity/UnitSourceCode (or equivalent unit-of-measure
  qualifier) → `MedicationDispense.quantity.code`, with the matched
  system populated in `MedicationDispense.quantity.system`
- Human-readable unit label → `MedicationDispense.quantity.unit`
- Potency/PotencyUnit subfields → mapped to
  `MedicationDispense.dosageInstruction.doseAndRate.doseQuantity`
  (or an appropriate extension on `MedicationDispense`) when not
  expressible inside a single `SimpleQuantity`

Publish this as a dedicated "NCPDP SCRIPT Quantity Mapping" section
on the `MedicationDispense` profile page of the US Medication FHIR
IG, with a worked example showing one row of NCPDP SCRIPT XML and the
corresponding FHIR `MedicationDispense.quantity` JSON/XML fragment.
If the same mapping is reused by the PDMP on FHIR IG (PSS-2225),
either duplicate the table there or reference the US Medication IG
section canonically.

#### Justification

The ticket explicitly asks for "more granular guidance" and the
current IG text ("NCPDP Data Element for Quantity") does not tell an
implementer which NCPDP subfield lands in which FHIR element. A
tabular, subfield-by-subfield mapping is the standard treatment in
HL7 IGs that span NCPDP and FHIR (e.g., the NCPDP Formulary/Benefits
and Pharmacist eCare Plan mappings), so this is consistent with
existing Pharmacy WG practice. It also directly unblocks PDMP on
FHIR (PSS-2225), which is the reason the ticket was reopened in
November 2023, and it removes a known round-trip hazard — two
implementers seeing the same NCPDP quantity and emitting different
FHIR shapes.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) adding narrative mapping text,
publish a machine-processable `ConceptMap` plus a small `StructureMap`
in the US Medication FHIR IG that transforms
`MedicationDispensed/Quantity` (both NCPDP SCRIPT 2017071 and 10.6)
into `MedicationDispense.quantity`. Concretely:

- A `ConceptMap` resource mapping NCPDP Quantity CodeListQualifier
  values to the FHIR `Quantity.system` URIs the IG expects (UCUM,
  NCPDP billing-unit code system, etc.).
- A `StructureMap` resource that consumes a logical model of the
  NCPDP Quantity compound field and populates
  `MedicationDispense.quantity` (value/unit/system/code), with a
  branch for potency that routes to
  `dosageInstruction.doseAndRate.doseQuantity` or an extension.
- A companion narrative page on the `MedicationDispense` profile
  that embeds the mapping table generated from those artifacts and
  points to at least one conformance example.

#### Justification

A narrative-only mapping (Disposition A) risks drifting out of sync
with the actual conformance resources and is hard to test. Shipping
the mapping as FHIR `ConceptMap` / `StructureMap` artifacts makes it
executable: implementers can validate their transforms against the
IG's own mapping, and the NCPDP code-list-qualifier edge cases (UCUM
vs. non-UCUM units, billing-unit vs. dispense-unit) are encoded once
rather than prose-described in multiple places. It does cost more
authoring effort and pulls in `ConceptMap`/`StructureMap` tooling
that the Pharmacy WG may not have set up for this IG, so it is worth
weighing against the simpler table-plus-example approach.

---

### Disposition C: Decline

#### Proposal

Close FHIR-29473 as "Not Persuasive" on the grounds that
`MedicationDispense.quantity` is a standard FHIR `SimpleQuantity` and
that NCPDP-to-FHIR quantity mapping is a general FHIR/UCUM concern
already covered by the base `Quantity` data-type documentation and
the NCPDP/HL7 joint mapping materials. No IG-specific guidance is
added; implementers are referred to the base FHIR `Quantity`
datatype page and any existing NCPDP/HL7 cross-standard mapping
guides.

#### Justification

This disposition is defensible only if the workgroup concludes that
the compound NCPDP Quantity is generic enough that IG-level guidance
would merely restate base-spec material. In practice, though, the
NCPDP Quantity element carries required subfields (value + code-list
qualifier + unit source code, and sometimes potency) whose routing
into `SimpleQuantity.value/unit/system/code` is genuinely
ambiguous — so this rationale is weak. It is included here because
the ticket previously took a similar "defer" path (closed 2023-01-18
as Considered for Future Use when the IG was withdrawn), and the
workgroup may prefer to formalise that posture if the PDMP/US
Medication IG lifecycle again stalls.

---

### Recommendation

**Recommended disposition:** A

Accepting as requested is the right starting point. The ticket is a
legitimate, concrete guidance gap — a compound NCPDP element with
required subfields mapping into a single FHIR `SimpleQuantity` is
exactly the class of problem an IG is supposed to pin down, and the
ticket has now been reopened specifically because PSS-2225 (PDMP on
FHIR) needs it. The Pharmacy WG already has precedent for publishing
NCPDP-to-FHIR mapping tables on profile pages, so the authoring cost
is low and the reviewer can validate the output without new tooling.
Disposition B (ConceptMap/StructureMap) is the better long-term
shape and should be kept open as a follow-on once the narrative
table is in place; treating it as the immediate answer risks
stalling on tooling while implementers still have no guidance.
Disposition C is not recommended — the base `Quantity` datatype page
does not resolve NCPDP's code-list-qualifier vs. unit-source-code
split, and declining would leave the reopened PDMP-on-FHIR IG
shipping with the same ambiguity that caused the ticket in the
first place.
