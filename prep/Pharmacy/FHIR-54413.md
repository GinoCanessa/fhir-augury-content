## Summary

The submitter (Eric Haas) is asking the Pharmacy work group to clarify what the
`status` value `completed` means when `MedicationRequest.doNotPerform = true`.
The current definition of `doNotPerform` â€” *"If true, indicates the provider is
ordering a patient should not take the specified medication"* â€” is silent on
the lifecycle of such an order, so it is unclear whether `status = completed`
is a legal/meaningful combination, and if so, what real-world event marks the
"completion" of an instruction *not* to do something. The proposal is to add a
constraint or narrative guidance â€” and ideally a state diagram for the
`doNotPerform` lifecycle (e.g., `draft â†’ active`, `draft â†’ active â†’ cancelled`,
or another set of permitted transitions) â€” so authors and validators have an
unambiguous answer.

No cross-referenced Jira tickets were surfaced by the cross-reference index for
FHIR-54413, but a content search turned up directly relevant prior art:
[FHIR-28104](https://jira.hl7.org/browse/FHIR-28104) (QI-Core, Eric Haas, 2020)
hit the same conceptual gap from the validator side â€” a CQL measure expected
`MedicationRequest.doNotPerform = true and status in {'completed', 'cancelled'}`
but the QI-Core profile constrained `doNotPerform` to `false`; that ticket was
ultimately retracted in favour of the dedicated `MedicationNotRequested`
profile pattern. Several QI-Core / QDM mapping tickets
([FHIR-53323](https://jira.hl7.org/browse/FHIR-53323),
[FHIR-53324](https://jira.hl7.org/browse/FHIR-53324),
[FHIR-47869](https://jira.hl7.org/browse/FHIR-47869),
[FHIR-48117](https://jira.hl7.org/browse/FHIR-48117)) also wrestle with which
`status` values are appropriate on negation profiles. A parallel workflow-level
clarification of `Task.doNotPerform` is in flight as
[FHIR-47394](https://jira.hl7.org/browse/FHIR-47394).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `MedicationRequest` definitional page in FHIR Core (Pharmacy work
group) to add explicit guidance and a constraint covering `status` when
`doNotPerform = true`. Concretely:

1. Add normative narrative under the Notes / Boundaries section of
   `MedicationRequest` (and the `doNotPerform` element definition itself)
   stating: *"For a `MedicationRequest` with `doNotPerform = true`, the
   `status` reflects the lifecycle of the prohibition itself: `draft` and
   `active` indicate the prohibition has been authored and is in force;
   `on-hold` indicates the prohibition is temporarily suspended; `cancelled`,
   `stopped`, `ended`, or `entered-in-error` indicate the prohibition has been
   withdrawn or was incorrect; `completed` SHALL be used only when the
   prohibition has a defined end (e.g., 'do not give NSAIDs for the next 7
   days') and that bounded period has elapsed. `unknown` retains its general
   meaning."*
2. Add a FHIRPath invariant on `MedicationRequest` of the form
   `doNotPerform.exists() and doNotPerform = true and status = 'completed'
   implies dispenseRequest.validityPeriod.end.exists() or dosageInstruction.timing.repeat.bounds.exists()`
   (warning-level), so that "completed" is only used with a bounded prohibition
   and tooling can flag the otherwise-ambiguous case.
3. Add a state diagram to the MedicationRequest page showing the two flows
   side-by-side: the existing positive-order lifecycle and the
   `doNotPerform = true` lifecycle (`draft â†’ active â†’ {on-hold â†” active} â†’
   {cancelled | stopped | ended | completed (bounded only)}`).

#### Justification

This is what the submitter literally asked for and it answers the question for
authors, validators, and downstream consumers (CQL measures, IPS/US-PCS
summaries) in one place. It is consistent with the direction the Workflow group
is taking on `Task.doNotPerform` per FHIR-47394 (completion is only meaningful
for time-bounded prohibitions), and it resolves the latent conflict that
FHIR-28104 surfaced between QI-Core's `doNotPerform = false` constraint and CQL
measure expressions that test `status in {'completed', 'cancelled'}`. The
warning-level invariant keeps existing instances valid (no breaking change)
while signalling the previously-undocumented expectation.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than add a Core constraint that conditions one status value on the
presence of a bounded period, ratify the existing implicit pattern by
**(a)** adding only narrative guidance on `MedicationRequest.doNotPerform`
that the `status` continues to describe the lifecycle of the *order*
(authored, in force, withdrawn, or â€” for time-bounded prohibitions â€” elapsed),
and **(b)** pointing implementers who need a strongly-typed "negative
medication statement" at the existing `MedicationNotRequested` / equivalent
profiles in QI-Core and similar IGs, or at `MedicationStatement` with
`status = not-taken`. No invariant is added; no state diagram is added to Core;
the workgroup instead publishes a short cookbook page (or extends the existing
`MedicationRequest` Notes section) to summarise the choices.

#### Justification

A Core invariant that depends on the presence of a `validityPeriod.end` or
`timing.bounds` will trip a long tail of existing instances that legitimately
use `doNotPerform = true` with `status = completed` against an implicit
encounter boundary (the QI-Core / VTE1 measure case in FHIR-28104 is exactly
this). The lighter-touch narrative answer matches how CQI repeatedly resolved
the analogous "Not Requested" profile tickets (FHIR-47869, FHIR-48117 â€”
*Resolved â€“ No Change*, no fixed status) and avoids forcing every implementer
to adopt the new constraint mid-cycle. It also keeps Pharmacy out of the harder
state-machine argument that Workflow is still having on FHIR-47394, deferring
to that group for the cross-resource pattern.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-54413 as *Resolved â€“ Not Persuasive* (or *Persuasive with
Modification â†’ No Change*) on the grounds that the existing `status` value set
on `MedicationRequest` already supplies the needed expressivity, the
`doNotPerform` element is clearly defined, and the question of "what does
`completed` mean" is a use-case-specific question that should be answered by
the IG or implementer applying the resource (IPS, US-PCS, QI-Core, CQL
measures), not by FHIR Core. Direct the submitter to existing negation
patterns (`MedicationNotRequested` profile in QI-Core; `MedicationStatement`
`status = not-taken`).

#### Justification

The gap the ticket identifies is real but is a definitional / educational gap,
not a wire-format gap. Adding a Core constraint risks invalidating existing
instances and existing IG profiles (QI-Core in particular), and adding a state
diagram for one boolean modifier risks setting a precedent that every
`doNotPerform`-bearing resource (Task, ServiceRequest, DeviceRequest,
NutritionOrder, â€¦) needs its own. Workflow's still-open FHIR-47394 thread shows
no community consensus yet on what "completion of a prohibition" means; Core
should not enshrine a definition ahead of that consensus.

---

### Recommendation

**Recommended disposition:** A

The submitter is right that the current MedicationRequest definition is silent
on a question that downstream consumers (CQL measures per FHIR-28104, IPS /
US-PCS summaries per the IPS Zulip thread, QI-Core "Not Requested" profile
authors per FHIR-47869 / FHIR-48117) keep having to answer for themselves â€”
inconsistently. Disposition C punts that ambiguity back to IGs and effectively
reaffirms today's status quo, which has produced repeated tickets in adjacent
work groups. Disposition B (narrative only) is a defensible compromise but
leaves validators with no machine-checkable signal and risks Pharmacy and
Workflow publishing different verbal answers for `MedicationRequest` vs.
`Task`. Disposition A â€” narrative *plus* a warning-level invariant *plus* a
state diagram â€” costs Pharmacy a single intro-section edit and one invariant,
is non-breaking (warning, not error), aligns with the direction Bryn Rhodes
articulated for `Task.doNotPerform` in the FHIR-47394 Zulip thread, and gives
QI-Core / IPS authors a single Core-level reference to point at. The work group
should adopt Disposition A and coordinate the wording with Workflow before
publication so the same answer applies to other request resources later.
