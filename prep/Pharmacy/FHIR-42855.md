## Summary

This ticket, raised out of Connectathon 34 against the US Prescription Drug
Monitoring Program (PDMP) FHIR IG, requests support in the specification for
expressing opioid dose-equivalence metrics — most notably Morphine Milligram
Equivalents (MME) and Lorazepam Milligram Equivalents (LME), along with
related measures such as MED (Morphine Equivalent Dose, referring to an
individual dose rather than a daily total). The reporter notes that
MME/LME refer to a total daily dose whereas MED refers to an individual
dose, asks for a full list of such measures (suggesting they might be
carried as units of measure), and points out that NIEM v4 and PMIX already
support these concepts. No other Jira tickets were surfaced as cross-
references, so this request stands on its own within the PDMP IG. The
ticket has already been resolved as Persuasive, with MME and LME added as
extensions on MedicationDispense and MedicationAdministration for the 2024
May ballot; the review here is therefore a confirmation of the chosen
disposition rather than an open decision.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add first-class support in the PDMP (FHIR) IG for dose-equivalence
measures (MME, LME, MED, and any additional equivalents identified as in
scope), carried on MedicationDispense and MedicationAdministration. The
concrete shape is a pair of complex extensions — one for total-daily-dose
equivalents (MME, LME) and one for per-dose equivalents (MED) — each
carrying a `Quantity` with a code drawn from a bound value set of
equivalence measures, plus optional metadata (calculation method,
reference drug, period of aggregation). This is what has already been
delivered: MME and LME extensions on both MedicationDispense and
MedicationAdministration, published in the 2024 May ballot.

#### Justification

The request names a concrete, well-understood clinical need (opioid
stewardship and PDMP reporting) that is already implemented in adjacent
standards (NIEM v4, PMIX). Expressing these as extensions on the
existing Medication* resources keeps the change additive, avoids a
breaking change to the base FHIR resources, and scopes the work to the
PDMP IG where the requirement originates. Binding the measure to a
coded concept leaves room to grow the list of supported equivalents
without further structural churn.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) dedicated extensions, express dose
equivalence using a UCUM-style unit on the existing `dosage.doseAndRate`
/ `dispenseRequest.quantity` elements, with a `Quantity.code` such as
`mg{MME}/d` or `mg{MED}` carried in an annotation/unit value set defined
by the PDMP IG. Author guidance that instructs implementers to populate
the equivalence alongside the actual dispensed/administered quantity
rather than in a parallel extension. If the base FHIR UOM story proves
too restrictive, fall back to a single generic `doseEquivalence`
extension keyed by a coded measure so one element shape covers MME, LME,
MED, and future additions.

#### Justification

The reporter explicitly asks whether this "might [be] support[ed] as a
UOM." A UOM-first approach keeps the equivalence value co-located with
the quantity it derives from, avoids a proliferation of one-off
extensions (one per equivalence measure), and lines up with how UCUM
annotations are already used elsewhere in FHIR. The generic-extension
fallback preserves additivity while still giving a single place for
implementers to look. The trade-off versus Disposition A is that UOM
annotations are not first-class FHIR concepts in the same way
structured extensions are, and validation/tooling support for free-text
UCUM annotations is weaker.

---

### Disposition C: Decline

#### Proposal

Decline to add MME/LME/MED support to the PDMP (FHIR) IG and direct
implementers that need these values to convey them out-of-band (e.g., in
a narrative, in an external report, or via a jurisdiction-specific
profile outside HL7). Rationale category: out of scope — dose
equivalence is a derived/calculated clinical measure whose algorithm
varies by jurisdiction and is arguably a consumer-side concern rather
than a PDMP-IG concern.

#### Justification

MME calculation rules are not standardized (different state PDMPs use
different conversion tables and rounding), so any value the IG carries
is only as trustworthy as the sender's algorithm. Declining avoids
codifying a contested calculation into the normative spec and keeps the
PDMP IG focused on representing dispense/administration events rather
than derived analytics. This is the defensible "do no harm" path if the
workgroup is uncomfortable sanctioning a specific equivalence model.

---

### Recommendation

**Recommended disposition:** A

Disposition A matches what the Pharmacy work group has already resolved
and balloted (Persuasive, 2024-May ballot): MME and LME extensions on
both MedicationDispense and MedicationAdministration. It satisfies the
explicit PDMP/NIEM/PMIX interoperability need the reporter raised, keeps
the change additive (extensions rather than base changes), and scopes
the equivalence vocabulary to a coded value set that can be grown over
time. Disposition B's UOM angle is attractive on paper but collides with
real-world UCUM tooling limitations for annotation-style units;
Disposition C is hard to defend given that PDMP implementers
demonstrably need these values today and an adjacent standard already
carries them. The workgroup should confirm the as-implemented extension
shape, review the coded value set for completeness (at minimum MME, LME,
MED), and consider whether calculation-method metadata needs to be
required rather than optional to address the variability concern raised
in Disposition C.
