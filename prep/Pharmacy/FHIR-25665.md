## Summary

Eric Haas observes that the CARIN RTPBC `carin-rtpbc-request-bundle`
profile (a message `Bundle`) is an incomplete way to express the
structure of the RTPBC request: profiling only the `Bundle` — without
also constraining the graph of entries — leaves the relationships
between the `MessageHeader` focus resource, the `Patient`,
`Coverage`, `MedicationRequest`, and other entry resources
underspecified. The reporter asks the Pharmacy work group to either
profile the entries with the `aggregate` element to define the graph,
or to use a `GraphDefinition` alongside (or instead of) the bundle
profile to make the intended message shape explicit and verifiable.
The ticket was initially resolved as "Considered for Future Use" in
April 2020 (vote 11-0-1) but was re-opened after STU1 publication in
August 2020 at Eric Haas's request to discuss why the profiles cannot
be updated or a `GraphDefinition` used to do this properly. No related
Jira tickets, Zulip threads, or GitHub items were surfaced by the
cross-reference service.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a `GraphDefinition` resource to the CARIN RTPBC IG that formally
describes the message graph rooted at the `rtpbc-request-message`
`MessageHeader`: `MessageHeader.focus` → `rtpbc-request` (the RTPBC
`Claim`/request resource), with links out to the `Patient`,
`Coverage`, `MedicationRequest`/`MedicationKnowledge`, and any
`Practitioner`/`Organization` participants the request carries. In
parallel, tighten `carin-rtpbc-request-bundle` so that each allowed
entry slice references a named, profiled resource, and set the
appropriate `ElementDefinition.aggregate` values (`bundled` for
resources that must travel inside the bundle, `referenced` for anything
that may be resolved out of band). Reference the new `GraphDefinition`
from the bundle profile's narrative and from the RTPBC messaging page
so implementers have a single authoritative picture of the request
graph (and mirror the treatment for the response bundle).

#### Justification

This is the most direct way to address what the reporter actually
asked for and closes a real gap: a `Bundle` profile alone does not
constrain the shape of the graph its entries form, which is exactly
what a message-oriented exchange like RTPBC needs to validate. Using
`GraphDefinition` plus `aggregate`-constrained entry slices is the
mechanism FHIR provides for this — it is the same pattern other
message-based IGs use — and it gives implementers and validators a
machine-readable contract rather than prose. The work group's original
resolution explicitly acknowledged that this should be considered in a
future version of the IG; re-opening post-STU1 is the natural point to
act on that commitment.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a `GraphDefinition`. Instead, address the underlying
"incomplete specification" concern purely within the bundle profile:
introduce named, closed slices on `Bundle.entry` for each RTPBC
payload resource (request, patient, coverage, medication, etc.), bind
each slice to its RTPBC profile via `entry.resource` type/profile
constraints, and set `ElementDefinition.aggregate` on each slice to
document whether the referenced resource must be bundled or may be
referenced externally. Add explicit invariants (FHIRPath `constraint`
expressions) on the bundle profile to enforce cardinality between
`MessageHeader.focus`, the request resource, and the supporting
resources. Publish a narrative diagram of the resulting graph on the
RTPBC messaging page.

#### Justification

`GraphDefinition` tooling and validator support were (and still are)
uneven across implementations, and RTPBC is a fairly small, fixed
message shape. Capturing the graph structurally in the `Bundle`
profile via entry slicing + `aggregate` + invariants delivers most of
the enforcement benefit Disposition A targets while staying inside the
mechanism implementers already consume (`Bundle` validation). It also
avoids introducing a new conformance artifact into an IG that has
already balloted and published STU1, which lowers implementer churn
and keeps the change surface area smaller.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" / "Considered for Future Use (reaffirmed)":
leave `carin-rtpbc-request-bundle` (and its response counterpart) as
published in STU1, and document in the IG narrative that the intended
graph is described prose-only for now, with a pointer to this ticket
as a tracked future enhancement.

#### Justification

STU1 has shipped, implementers are coding against the current bundle
profile, and there is no evidence in this ticket (or in the
cross-reference graph — which is empty) of implementer interop
failures caused by the current level of specification. Adding a
`GraphDefinition` or re-slicing the bundle profile at this point is a
normative-shape change that would break validators and test harnesses
for marginal gain, and the original work group vote (11-0-1) already
concluded this was a "future version" concern. Declining the ticket
again, explicitly, is defensible on scope and stability grounds.

---

### Recommendation

**Recommended disposition:** B

Disposition B best balances the reporter's legitimate concern — that a
bare `Bundle` profile under-specifies the RTPBC message graph — with
the practical reality that RTPBC has already published STU1 and that
`GraphDefinition` remains less-well-supported by validators than
entry-slicing with `aggregate`. Tightening the existing
`carin-rtpbc-request-bundle` (and the response bundle) via named entry
slices, profile bindings, `ElementDefinition.aggregate`, and
invariants gives implementers a verifiable contract using mechanisms
they already consume, and it keeps the IG's conformance surface area
coherent. Disposition A is the "textbook correct" answer and should
be kept on the table if the work group is prepared to adopt
`GraphDefinition` IG-wide; Disposition C ignores the reporter's
re-opening rationale and a prior WG acknowledgement that this was a
real gap.
