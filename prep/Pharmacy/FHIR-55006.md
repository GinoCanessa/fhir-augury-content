## Summary

This ballot comment, imported from an HL7 ballot pool by Ron G. Parker (Comment 32), questions whether the new `MedicationRequest.isRecordOfRequest` element is the correct mechanism, asking whether the underlying intent — recording a known order without acting on it — should instead be conveyed by `MedicationStatement`. The element in question is a renaming of the existing `MedicationRequest.reported` boolean and is intended to flag that the instance faithfully reproduces a prior order but is not itself an authorisation to dispense or administer.

The Pharmacy WG has multiple closely related tickets in flight that should be triaged together: [FHIR-55005](https://jira.hl7.org/browse/FHIR-55005) (already resolved Persuasive — agrees to move `isRecordOfRequest` next to `supportingInformation` because it is a new element), [FHIR-54446](https://jira.hl7.org/browse/FHIR-54446) (Eric Haas; proposes the element should be a modifier — Lloyd McKenzie comments that the element may not be needed at all because tags are the right mechanism for "is authoritative" semantics so signatures aren't broken), and [FHIR-53305](https://jira.hl7.org/browse/FHIR-53305) (Jose Costa-Teixeira; argues the opposite — that `isModifier` should be removed). The IPS Madrid Connectathon Zulip thread (Day 2, 11 May 2025) records the original Pharmacy/IPS agreement to rename `reported` → `isRecordOfRequest` and exposes the live design tension: Stephen Chu (Pharmacy/IPS) wants an explicit, faithful "this is a record of a prior request, do not act on it" flag for IPS use cases, while Lloyd McKenzie and Grahame Grieve argue that all FHIR resources are records, that authorisation is governed by workflow/tags, and that `MedicationStatement` is the right resource when the patient is the source of truth. FHIR-55006 is essentially asking the workgroup to revisit that May 2025 IPS decision in light of this ballot.

## Details

**Description:**

> Is the element isRecordOfRequest appropriate?  Wouldn't that be a MedicationStatement?
>
> *(Comment 32 - imported by: Ron G. Parker)*

**Comments:**

- **costateixeira (2026-01-25):** "This should be high priority as it has a high impact - how to represent a statement of an order? A full scenario should be described."
- **chuscmi (2026-04-07):** Argues the element is appropriate. Key points:
  - `MedicationRequest.isRecordOfRequest` is a rename of `MedicationRequest.reported`; setting `MedicationStatement.isRecordOfRequest = true` does **not** turn a `MedicationRequest` into a `MedicationStatement`.
  - There are use cases — IPS and AU-PS in particular — in which `MedicationRequest` instances must be exchanged faithfully (preserving the original prescription) but must not be used to dispense.
  - IPS `Composition.section:sectionMedications.entry:medicationStatementOrRequest` is `0..* Reference(MedicationStatement (IPS) | MedicationRequest (IPS))`, so the IPS profile already needs to carry both.
  - These imported `MedicationRequest` instances are not `MedicationStatement`s because they are intended to be a faithful representation of the original prescription, including authoring practitioner, dose, and timing.

## Related Zulip Discussions

### [IPS > Day 2 Madrid Connectathon](https://chat.fhir.org/#narrow/stream/IPS/topic/Day%202%20Madrid%20Connectathon)

This is the thread that spawned the rename `MedicationRequest.reported` → `MedicationRequest.isRecordOfRequest` and is the substantive design discussion underlying FHIR-55006. Stephen Chu reports the Day 1 Pharmacy/IPS agreement that the IPS `MedicationRequest` profile is **not** intended to trigger dispensing and proposes the rename plus a definition stating the instance "is a record of the Medication Request and is not request or order itself." Lloyd McKenzie pushes back: all FHIR instances are records; IPS is a summary, not a workflow channel; "isActionable" semantics belong on a tag (not an element) so that signatures survive cross-system copies; if the patient is the source, `MedicationStatement` (with `informationSource = Practitioner`) is the appropriate resource. Grahame Grieve sides with Stephen on the clinical-information case (whether a fill was *requested* matters in a summary even outside workflow) but agrees with Lloyd that all resources are records and that some context-setting (tag or otherwise) is required. Peter Jordan supports the IPS need pragmatically (20–30 % of prescriptions are never filled, so `MedicationRequest` is a backstop). The thread did **not** close out the modifier-vs-tag-vs-element question — exactly the gap FHIR-55006, FHIR-54446, and FHIR-53305 are now trying to fill.

*Participants:* Stephen Chu, Lloyd McKenzie, Grahame Grieve, Peter Jordan, John D'Amore, Dave deBronkart, Justin McReynolds, Peter Williams, Danielle Tavares-Rixon, Nikola Tanjga, Rob Hausam (mentioned), Jose Costa-Teixeira (mentioned).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove `MedicationRequest.isRecordOfRequest` from the FHIR Core MedicationRequest resource (and the matching `MedicationStatement.isRecordOfRequest` if present). For the IPS / AU-PS use case the reviewer is gesturing at, document in the MedicationRequest narrative and in the IPS profile that summary-only / non-actionable medication information should be conveyed via `MedicationStatement` (with `informationSource` populated to indicate the original prescriber as the source). Reconcile/withdraw the related changes in [FHIR-55005](https://jira.hl7.org/browse/FHIR-55005) (which positions the element next to `supportingInformation`), [FHIR-54446](https://jira.hl7.org/browse/FHIR-54446) (modifier-flag debate), and [FHIR-53305](https://jira.hl7.org/browse/FHIR-53305), and restore (or formally retire) the predecessor `MedicationRequest.reported` boolean.

#### Justification

This is the literal reading of the ballot comment — the reviewer asserts the use case is `MedicationStatement`. Lloyd McKenzie's comments on both FHIR-54446 and the IPS Zulip thread reinforce this reading on FHIR design grounds: all FHIR Request resources are by definition records of requests; "is/is-not authoritative" is a workflow/provenance concern that belongs on a tag (so that copying an instance across systems does not invalidate signatures); and where the patient or a downstream summary is the information source, `MedicationStatement.informationSource` is precisely the modelled mechanism. Adopting this disposition removes the modifier-flag controversy entirely (FHIR-53305 / FHIR-54446 become moot) and avoids introducing a new element whose semantics overlap with existing tagging.

---

### Disposition B: Alternative Approach (Recommended)

#### Proposal

Keep `isRecordOfRequest` but reframe and constrain it so that the IPS / AU-PS use case is satisfied without conflating with `MedicationStatement`:

1. Retain `MedicationRequest.isRecordOfRequest : boolean` (consistent with the rename agreed in the IPS Madrid Connectathon thread and locked in by FHIR-55005's persuasive disposition).
2. Sharpen the element definition to make clear it indicates **provenance** ("this instance reproduces a previously-issued request") and is **not** a workflow signal — actionability is governed by `status`, `intent`, and the receiving system's workflow context, not by this flag.
3. Resolve the modifier debate by **leaving `isModifier = false`** (aligning with FHIR-53305, against FHIR-54446): the flag does not change the meaning of the *MedicationRequest* (it is still a request); it qualifies the *origin* of the instance. Document this rationale explicitly to close out FHIR-54446.
4. Add invariants/guidance to the IPS `MedicationRequest` profile requiring `isRecordOfRequest = true` in summaries, and amend the IPS profile narrative to direct implementers to `MedicationStatement` for patient-reported medication use, with `MedicationRequest` reserved for faithful reproductions of known prior orders.
5. Cross-link the disposition with FHIR-55005, FHIR-53305, and FHIR-54446 so reviewers see the package decision.

#### Justification

This addresses the reviewer's underlying concern (clear separation of "patient is on this medication" from "there exists a known prescription for this medication") without forcing IPS/AU-PS implementers to choose between losing prescription fidelity (Disposition A — moves everything to `MedicationStatement`, which loses the original prescriber, dosing, and timing structure) and accepting a workflow-bearing modifier (Disposition C path, or FHIR-54446's proposal). It honours the FHIR-55005 disposition that has already been adopted, respects Stephen Chu's IPS use cases captured in the Madrid Zulip thread, and uses Lloyd's "all instances are records" point as the definitional anchor — `isRecordOfRequest` becomes a *provenance hint*, not an *actionability switch*. Leaving `isModifier = false` is consistent with FHIR's Request pattern (a request is not an order until workflow says so), which is also Costa-Teixeira's position in FHIR-53305.

---

### Disposition C: Decline

#### Proposal

Decline the comment as **Considered – No Change**, on the grounds that (a) the rename `reported` → `isRecordOfRequest` and the element's semantics were debated and agreed at the IPS Madrid Connectathon (May 2025) with Pharmacy and IPS WG participation, (b) the sibling ticket FHIR-55005 is already Resolved – Persuasive on the same element, and (c) the reviewer's alternative ("use MedicationStatement") loses the structured prescription fidelity that IPS/AU-PS explicitly require. Direct the reviewer to the existing element definition and invite a follow-up ticket if a specific clinical scenario cannot be represented.

#### Justification

The element is not novel — it is a rename of the long-standing `MedicationRequest.reported` boolean, which already carried this exact "this instance was reported by someone other than the original prescriber" semantic. Removing it now would be a breaking change for every implementer using `reported`, and the IPS profile demonstrably needs to carry faithful `MedicationRequest` instances (Composition `section:sectionMedications` allows both `MedicationStatement (IPS)` and `MedicationRequest (IPS)`). The reviewer's question is best read as a request for clarification, not a request for removal.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the only path that simultaneously (i) honours the already-adopted persuasive disposition on FHIR-55005, (ii) preserves the IPS / AU-PS use case that drove the Madrid Connectathon rename, (iii) gives the workgroup a clean answer to the open modifier-flag question raised by FHIR-53305 and FHIR-54446, and (iv) addresses Lloyd McKenzie's substantive design objection by reframing the element as provenance rather than as a workflow/actionability switch. Disposition A is principled but breaks a freshly-adopted disposition (FHIR-55005) and discards real implementer need; Disposition C dismisses a substantive design concern that has now surfaced in three separate tickets and a multi-day Zulip thread, so it is unlikely to hold under further review. The workgroup should adopt B, package the resolution with FHIR-55005/FHIR-53305/FHIR-54446, and take the explicit `isModifier = false` decision on the record.
