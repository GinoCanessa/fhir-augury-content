## Summary

The ticket asks the Pharmacy work group to clarify the intended use of
`Timing.repeat` in the context of `Dosage`. The reporter notes that the
data type permits semantically equivalent but structurally different
representations of the same dosing schedule — for example,
`boundsDuration = 5 d` is equivalent to `count = 5` with `period = 1 d`.
The reporter argues that this redundancy creates implementation,
producer, and consumer challenges (different systems may emit one form
and expect the other) and asks whether FHIR should establish a
preferred pattern, or otherwise provide normative guidance about how
the alternatives are to be reconciled when used in `Dosage` (e.g.,
`MedicationRequest.dosageInstruction.timing`,
`MedicationStatement.dosage.timing`,
`MedicationDispense.dosageInstruction.timing`).

There are no cross-referenced Jira tickets in the Augury index for this
item, so the request currently stands on its own.

## Details

**Description:**

> There are alternative representations for determining how Timings are
> repeated, e.g. using boundsDuration or count. E.g. boundsDuration =
> 5 d is same as count = 5, period 1 d. This creats challenges for
> implementers, producers and consumers of this data. Should there be
> a preference for any of the patterns?

### [implementers > GF#12881 - Timing.repeat.count](https://chat.fhir.org/#narrow/stream/implementers/topic/GF%2312881%20-%20Timing.repeat.count)

MnM raised the question of how `Timing.repeat.count` interacts with
`frequency`/`period` — concretely, for "twice daily for 5 days", is
`count` 5 (the number of periods) or 10 (the number of administrations)?
Implementer responses converged: Richard Townley-O'Neill, Jenni Syed
and Michelle Miller all stated they use `count` as the total number of
events/doses (i.e., 10), with the rationale that this gives a
`frequency`-independent meaning. No formal resolution to enshrine this
in a normative invariant or example is recorded in the thread.

Participants: Grahame Grieve, Richard Townley-O'Neill, Jenni Syed,
Michelle (Moseman) Miller.

Discussion about invariant `tim-9` on `Timing.repeat`, which uses an

and proposed rewriting it with `when.all($this in (…))`. The
conversation concluded that a tracker should be raised against FHIR-I
both to fix the invariant and to clarify spec language about how
constraint exceptions are interpreted (FHIR-35990 was filed). Relevant
to FHIR-54842 because it shows that `Timing.repeat` invariants are an

Participants: Ose Umolu, Bryn Rhodes, Nick George.

Participants: Michel Rutten, Lloyd McKenzie.

### [Medication > start and end date of therapy](https://chat.fhir.org/#narrow/stream/Medication/topic/start%20and%20end%20date%20of%20therapy)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative guidance to the `Timing` data type page (and a
corresponding note on the `Dosage` page) that establishes a *preferred*
representation when alternatives are semantically equivalent, and
documents how implementers should reconcile equivalent forms. Concretely:

1. On `Timing.repeat`, add a "Choosing between equivalent
   representations" subsection that states a preference order:
   - Use `bounds[x]` (preferring `boundsPeriod` when calendar-anchored
     start/end is known, otherwise `boundsDuration`) to express the
     overall therapy window.
   - Use `count`/`countMax` to express a finite number of
     administrations when the schedule is *event-counted* rather than
     *time-windowed* (e.g., "10 doses").
   - Do **not** use `boundsDuration` together with `count` + `period`
     to express the same duration twice; if both are present they must
     be consistent.
2. Add a worked-example table showing the canonical encoding for
   common phrases ("twice daily for 5 days", "every 4 hours up to 6
   doses", "from 1 Mar to 7 Mar twice daily").
3. Add invariant `tim-N` on `Timing.repeat`: if `boundsDuration` is
   present and `count` and `period`/`periodUnit` are also present, then
   `boundsDuration` must equal `count * period` (in the same unit),
   tolerating UCUM normalization.
4. In `Dosage`, link to the new guidance from the `timing` element
   description.

#### Justification

This directly answers the reporter's question, gives implementers a
single canonical encoding to target, and codifies the implementer
consensus already visible in the GF#12881 thread (`count` = number of
administrations) and the WG guidance from the
"start and end date of therapy" Medication thread (`bounds[x]` is the
home for therapy window). Adding a consistency invariant rather than
forbidding one form preserves backwards-compatibility while making
non-conformant combinations machine-detectable. This is consistent with
the prior pattern of incrementally tightening `Timing.repeat`
invariants (e.g., GF#15839 for negative `count`).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than declaring one form preferred in the *base* `Timing` data
type, address the ambiguity at the **profile** level for medication
contexts:

1. Leave the base `Timing` data type unchanged (all forms remain
   equally valid), but expand the descriptive guidance on `Timing` to
   explicitly state that the alternatives are *semantically
   equivalent* and that systems must be prepared to consume any of
   them.
2. Add an informative "Dosing Pattern" guidance page under the Pharmacy
   module that defines a small set of canonical encodings for
   medication dosing and recommends them for use in Pharmacy-related
   resources (`MedicationRequest`, `MedicationStatement`,
   `MedicationDispense`, `MedicationAdministration`).
3. Encourage / require IGs (e.g., national medication IGs) to constrain
   `dosageInstruction.timing` to the canonical forms via profiles, and
   provide example slicing/invariants they can reuse.
4. Optionally publish the consistency rule from Disposition A as a
   `Dosage` profile invariant, not a base `Timing` invariant.

#### Justification

The reporter's pain point is specifically about *Dosage* —
non-medication users of `Timing` (Schedule, ActivityDefinition,
ServiceRequest) may have legitimate reasons to encode in different
forms (e.g., a one-shot `boundsPeriod` for a service window, vs. a
repeating `count` for a protocol step). Restricting at the base type
risks breaking those uses or invalidating existing instances in the
wild. Pushing the canonical-form decision into Pharmacy-owned guidance
(or downstream IGs) keeps the base type permissive while still giving
medication implementers a single target. It also avoids cross-WG
ownership friction with FHIR-I, which owns the base data type.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-54842 as "Persuasive with Modifier — No Change" (or
"Not Persuasive"), responding that:

- The alternative representations of `Timing.repeat` are intentional;
  `bounds[x]`, `count`/`countMax`, `period`/`periodUnit`, and
  `frequency`/`frequencyMax` describe genuinely different aspects of a
  schedule and are not strictly redundant in all cases (e.g.,
  `boundsDuration` does not by itself say *how many doses* or *at what
  cadence*).
- Establishing a single normative preferred form would invalidate
  equally-valid existing instances and force implementers to migrate
  data without a clear interoperability gain.
- Implementers should follow existing element descriptions and rely on
  IG-level profiles where a tighter local pattern is needed.

#### Justification

The example in the ticket (`boundsDuration = 5 d` vs. `count = 5,
period = 1 d`) is only equivalent under the assumption that exactly
one event happens per period and the schedule is contiguous — neither
of which `Timing.repeat` requires. The WG can reasonably conclude that
the apparent redundancy is the consequence of a flexible model serving
many use cases (orders, schedules, observations, protocols) and that
the cost of a normative preference (data migration, churn in mature
implementations such as US Core / national medication IGs) outweighs
the benefit.

---

### Recommendation

**Recommended disposition:** B

The reporter's concern is real and well-grounded in the medication
domain, and the Augury-surfaced threads
([GF#12881](https://chat.fhir.org/#narrow/stream/implementers/topic/GF%2312881%20-%20Timing.repeat.count)
and the Medication
[start/end of therapy](https://chat.fhir.org/#narrow/stream/Medication/topic/start%20and%20end%20date%20of%20therapy)
discussion) show that implementers already converge on a *de facto*
canonical encoding for dosing — but that consensus has never been
written down. Disposition A would write it down at the base-type
level, but `Timing` is consumed far beyond Pharmacy and changing
normative behaviour there risks invalidating instances in unrelated
domains (and crosses ownership lines into FHIR-I). Disposition C
preserves flexibility but leaves the documented gap that the reporter
is pointing at. Disposition B threads the needle: it ratifies the
existing implementer consensus where it actually matters (Pharmacy /
Dosage) via a guidance page and optional `Dosage`-level invariants,
without disturbing the base data type. The work group should weigh
whether to additionally lift the consistency invariant
(`boundsDuration = count * period`) into the base type as a non-breaking
clarification — this is the part of Disposition A most likely to be
non-controversial and could be combined with B if desired.
