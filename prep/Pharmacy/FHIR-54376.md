## Summary

Eric Haas (Pharmacy) flags an apparent overlap between two package-related
constructs on `MedicinalProductDefinition` in R6: the newly-added
`packageSize` backbone (with `numberOfItems` and `amountPerItem`) and the
existing `instance` backbone, which is described as "Information that only
applies to packages." Because both convey package-level information, the
ticket asks the workgroup to either (1) consolidate by moving `packageSize`
under `instance`, or (2) clarify in the specification when implementers
should use `packageSize` versus `instance` for package-related data. A
reviewer comment on the ticket additionally observes that the current
`instance` definition itself is incorrect — it should describe information
that applies to physical occurrences of the product, not to "packages,"
which is precisely the source of the overlap with `packageSize`.

No related Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-reference or text search.

## Details

**Description:**

> New `packageSize` backbone added in R6:
>
> - `numberOfItems`: *"The number of items in the package"*
> - `amountPerItem`: *"The amount per each item, e.g. 10 ml / vial"*
>
> Meanwhile, `instance` is defined as: *"Information that only applies to **packages**"*
>
> *Both relate to package information so propose either:*
>
> 1. *Consolidate the package information by moving `packageSize` under `instance`*
> 2. *Clarify the difference and when to use `packageSize` vs `instance` for package-related data.*

**Comments:**

  packages' seems wrong. should be 'Information that only applies to
  physical occurrences of the product'."

### Disposition A: Accept As Requested (Option 2 — clarify both elements)

#### Proposal

Retain both `MedicinalProductDefinition.packageSize` and
`MedicinalProductDefinition.instance` as currently structured in R6, and
fix the definitions so the distinction between "package size description"
(an attribute of the abstract product packaging) and "physical occurrence
of a package" (a concrete instance of a package, e.g., a specific bottle
on a shelf) is unambiguous. Concretely:

- Rewrite the `instance` element definition from "Information that only
  applies to packages" to "Information that only applies to physical
  occurrences of the product (a specific manufactured package), as opposed
  to the abstract product description" — directly addressing the reviewer
  comment from costateixeira.
- Add a sentence to the `packageSize` element comment clarifying that it
  describes the *abstract* package design (how many items, and how much per
  item) and is not used to identify or describe an individual physical
  pack — for that, use `instance` (e.g., `instance.identifier`,
  `instance.expiry`).
- Cross-link the two elements in their `comment` fields so the
  abstract-vs-physical distinction is discoverable from either side.

#### Justification

The reviewer comment from costateixeira pinpoints the actual defect: the
overlap is in the *prose*, not the model. `packageSize` (count and amount
per item) and `instance` (data about a specific manufactured pack —
typically identifier, expiry, lot) describe genuinely different concepts
that consistently coexist in IDMP-aligned medicinal-product modelling.
Tightening the definitions resolves the implementer-facing ambiguity
without restructuring an R6 backbone that is already implemented and
referenced by downstream IGs. This is a low-risk, definition-only change
appropriate for a Triaged change request at this point in the cycle.

---

### Disposition B: Alternative Approach (Option 1 — consolidate under `instance`)

#### Proposal

Move `packageSize` under `MedicinalProductDefinition.instance` (e.g., as
`instance.size` with the same `numberOfItems` and `amountPerItem`
sub-elements), deprecate the top-level `packageSize` backbone in R6, and
update derived profiles, search parameters, and any IG references
accordingly. Keep the deprecated path available with guidance pointing to
the new location for at least one release cycle.

#### Justification

If the workgroup concludes that `instance` truly is the canonical
"package" container — the option-1 reading the ticket itself proposes —
then collocating size with the rest of the per-instance data makes the
model self-consistent and removes the temptation to split package-level
information across two parallel backbones. The trade-off is significant:
`packageSize` is a new R6 backbone, and moving it after publication would
be a breaking restructuring for early implementers and any IG that has
already profiled it. Backward-compatibility burden, downstream IG churn,
and the fact that `instance` semantically describes physical occurrences
(per the ticket's own commenter) all weaken the case for this option, but
it remains the cleaner long-term shape if the workgroup intends `instance`
to be *the* package container.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that `packageSize` and `instance` model
distinct concepts that are already separated in R6, and that no
implementer-reported defect or interoperability failure has been cited —
only a reading ambiguity in two element definitions. Mark the request
"Not Persuasive — definitions adequate" and rely on existing authoring
guidance.

#### Justification

Defensible only if the workgroup believes the current definitions are in
fact clear when read in context. The weakness of this position is
explicit in the ticket: a second reviewer (costateixeira) independently
arrived at the same confusion the reporter raised, which is direct
evidence the prose is misleading. Declining without at least the
definition fix in Disposition A is hard to justify against the
"Persuasive with Modification" bar.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly resolves the only concrete defect anyone has
identified — the misleading definition of `instance` — while preserving
both R6 backbones and avoiding the breaking restructuring that
Disposition B would impose this late in the cycle. The reviewer comment
from costateixeira essentially writes the corrected `instance`
definition for the workgroup, and `packageSize` (a count plus an
amount-per-item) is a different shape of information from a per-physical-
pack record (identifier, expiry, lot). Disposition B is the right answer
only if the workgroup's intent for R6 was always for `instance` to be
*the* canonical package container; in that case it should be tracked as a
separate, larger restructuring ticket rather than folded in here.
Disposition C ignores the second reviewer's confirmation that the prose
is wrong and is hard to defend.
