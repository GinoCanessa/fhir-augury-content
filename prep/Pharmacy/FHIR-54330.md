## Summary

The reporter observes that in the FHIR R6 (6.0.0-ballot4) Dosage rework, every
example on the [`dosage-examples.html`](https://hl7.org/fhir/6.0.0-ballot4/dosage-examples.html)
page expresses repetition by means of `DosageDetails.step.component.timing.repeat.count`,
and none of the published examples actually exercise the newly introduced
`DosageDetails.step.count` element. They therefore cannot tell from the spec
when an authoring system should pick `step.count` over the equivalent
`step.component.timing.repeat.count`, and they question whether `step.count`
adds expressive power that cannot already be carried inside the timing of a
step's component(s).

The reporter proposes one of two corrective actions: either **remove
`DosageDetails.step.count`** from the data type because the existing
`Timing.repeat.count` carried by each component already covers the use case,
**or** add at least one worked example to the dosage-examples page that can
only be expressed via `step.count` (i.e., that demonstrates a semantic that
component-level timing cannot capture).

There are no explicit Jira links, no Zulip cross-references, and no GitHub
cross-references stored against this ticket; however, the broader Pharmacy
WG R6 Dosage redesign discussions on chat.fhir.org provide the design
context (see "Related Zulip Discussions" below). In those discussions the
WG has clarified that, in the new model, **steps are consecutive in time**
while **components within a step are concurrent**, which sets the conceptual
boundary the ticket is asking about.

## Proposed Dispositions

### Disposition A: Accept As Requested — Remove `DosageDetails.step.count`

#### Proposal

Delete the `DosageDetails.step.count` element from the R6 `DosageDetails`
data type for ballot 5 / final. Rely on
`DosageDetails.step.component.timing.repeat.count` (i.e. `Timing.repeat.count`
on each component's timing) to express the number of administrations a step
is to be carried out for. Update the dosage-examples page only to the extent
needed to remove any latent reference to `step.count`. Add a short
clarifying paragraph on the `DosageDetails.step` element definition stating
that step-level repetition is expressed through the `Timing` of each
component, and that distinct repetitions of the *step itself* should be
modelled as separate `step` entries (or, where applicable, by the
`Timing.repeat` of every component in lockstep).

#### Justification

The reporter's empirical observation is correct: every published example in
6.0.0-ballot4 uses component-level `Timing.repeat.count`, and the WG has
not yet been able to articulate a use case `step.count` uniquely covers.
The "How bolus dose in MedicationRequest" thread shows the WG already
treating *steps* as semantically "consecutive in time" — i.e., a step is a
**phase**, not something you naturally "count". Counting a phase is more
plausibly modelled either by repeating the phase as additional `step`
instances (which is what the bolus/continuous example does) or by the
component timing. Removing `step.count` is also consistent with the
WG's current trim direction on `DosageDetails` (removal of
`DosageDetails.simple` per the "recycling constraint ids" and "backlog for
next call" Zulip threads, plus FHIR-54337). Fewer elements means fewer
modelling choices that must be reconciled by implementers and fewer
inconsistencies between MedicationRequest/MedicationDispense/MedicationStatement
producers and consumers.

---

### Disposition B: Alternative Approach — Keep `step.count`, Define It as Step-Cycle Count and Add an Example

#### Proposal

Retain `DosageDetails.step.count` but tighten its definition so that it
unambiguously means **"the number of times this entire step (including all
of its concurrent components) is to be repeated as a unit before moving to
the next step"** — i.e., a *cycle count* at the step level. Add a worked
example to `dosage-examples.html` that genuinely cannot be expressed by
`step.component.timing.repeat.count`, for instance:

- A simple chemotherapy cycle: a step containing two concurrent drug
  components (e.g., drug A every 6 h and drug B every 12 h), with
  `step.count = 4` to indicate the whole cycle of components is repeated
  four times before the regimen moves on to the next step.
- An induction phase (`step.count = 3`) followed by a maintenance phase
  (`step.count = 12`), where each phase contains multiple components with
  their own intra-phase `Timing.repeat`.

Add an invariant / guidance line on `DosageDetails.step.count` clarifying
that it does **not** override or scale `Timing.repeat.count` on the
contained components — it counts repetitions of the whole step, not of
individual administrations.

#### Justification

The "How bolus dose in MedicationRequest" thread establishes that "step"
is meant to be a higher-order, consecutive-in-time unit. Once that is
accepted, *counting steps* is genuinely distinct from *counting
administrations within a step*: e.g., "do this step 4 times" is awkward to
encode by inflating each component's `Timing.repeat.count` if the
components have different intra-step periodicities, because their
repetition counts would need to be derived per-component instead of
declared once at the step level. Keeping the element preserves
expressivity for cyclical regimens (a major Pharmacy use case in oncology
and infectious disease) without forcing implementers to reverse-engineer
cycle structure from heterogeneous component timings. Adding the example
the reporter asked for also discharges the spec's own documentation debt:
the reporter's *primary* complaint is that they cannot tell from the
examples when to use the element.

---

### Disposition C: Decline — No Change

#### Proposal

Close the ticket as "Not Persuasive" / "Working as Designed". Take no
modelling change and no examples change, on the basis that
`DosageDetails.step.count` is intentionally a step-level convenience
element distinct from `Timing.repeat.count`, and that the existing element
definition is sufficient.

#### Justification

This is defensible only if the WG is confident that the existing element
text already disambiguates `step.count` from `step.component.timing.repeat.count`
to a competent implementer. Given (a) the reporter's "Highest" priority
filing, (b) the absence of any worked example in 6.0.0-ballot4, and (c)
the WG's own ongoing rework of `DosageDetails` (removal of `.simple`,
proposed rename to `InstructionsForUse`, restoration of step ordering),
the position that "no clarification is needed" is the weakest of the
three. It is included for completeness but should be chosen only if the
WG explicitly judges that `step.count` is well-understood inside the
group and the absence of an example is not material.

---

### Recommendation

**Recommended disposition:** B — Keep `step.count`, define it as a
step-cycle count, and add a worked example.

The reporter's underlying concern is documentation, not modelling: they
explicitly offer "remove it **or** add a differentiating example." The
Pharmacy WG's own R6 design conversations (Zulip "Dosage", "How bolus
dose in MedicationRequest", "R6 DosageCondition vs. dosage-conditions
extension") already treat a `step` as a higher-order, consecutive-in-time
unit with its own metadata (`step.start`, `step.end`), so a cycle-count
*at the step level* is conceptually consistent with the rest of the
emerging R6 model and meaningfully harder to express via per-component
`Timing.repeat.count` once a step has multiple, non-uniform components.
Removing the element (Disposition A) would simplify the data type but
would force implementers to encode chemotherapy- and other cyclical
regimens by inflating per-component repeat counts, which is the kind of
implementer-hostile workaround the R6 rework was meant to avoid. Disposition
C ignores the documented gap. Disposition B addresses the reporter's
real ask (an example, plus disambiguating prose) and aligns with the
WG's expressed direction without requiring a breaking removal mid-ballot.
