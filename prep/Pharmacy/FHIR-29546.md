## Summary

The ticket requests that the US Medication FHIR Implementation Guide (originally
filed against the now-paused US Prescription Drug Monitoring Program (PDMP) on
FHIR effort under the Pharmacy work group) add `US Core PractitionerRole
Profile` as an allowed target type on the `actor` reference that today only
permits `US Core Practitioner Profile` and `US Core Organization Profile`. The
reporter argues that `PractitionerRole` should be the preferred reference
because the bare `US Core Practitioner` profile carries no information about the
practitioner's organization or role context, which is important for prescribing
and dispensing scenarios. The proposed wording is `actor S 1..1 Reference(US
Core PractitionerRole Profile | US Core Practitioner Profile | US Core
Organization Profile)`. No related Jira tickets were surfaced via cross-reference
search.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Medication FHIR Implementation Guide, update the `actor` reference on
the affected profile/resource (the page from which the existing wording
`Reference(US Core Practitioner Profile | US Core Organization Profile)` is
drawn — most likely the prescriber/dispenser actor on the medication
prescription/dispense profile) to add `US Core PractitionerRole Profile` as an
allowed target. The new constraint becomes:

```
actor S 1..1 Reference(
  US Core PractitionerRole Profile |
  US Core Practitioner Profile |
  US Core Organization Profile)
```

In the accompanying narrative, add a note that `US Core PractitionerRole
Profile` is the **preferred** target when the sender can provide it, because
`PractitionerRole` carries the practitioner-to-organization linkage and the
role/specialty context that bare `Practitioner` does not.

#### Justification

`PractitionerRole` is the standard FHIR vehicle for binding a practitioner to
the organization, role, specialty, and location in which they were acting at
the time of an event — exactly the context a PDMP, prescription, or dispense
record needs in order to be unambiguous (e.g., "Dr. Smith acting as a
prescribing physician at Clinic X" vs. "Dr. Smith acting as a pharmacist at
Pharmacy Y"). US Core publishes `PractitionerRole` as a first-class profile and
already recommends it for that purpose, so adding it to the allowed targets
aligns the US Medication IG with broader US Core guidance and with how other
US Realm IGs (e.g., US Core itself, Da Vinci PDex, QI-Core) model
practitioner-in-context references. Retaining `US Core Practitioner Profile`
and `US Core Organization Profile` preserves backward compatibility for
existing senders.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the union of allowed target profiles on a single
reference, **slice the `actor` reference by profile** and require
`US Core PractitionerRole Profile` as the primary slice (1..1, MustSupport),
with `US Core Practitioner Profile` and `US Core Organization Profile` as
optional fallback slices (0..1, MustSupport). Equivalently, add a constraint /
guidance statement that says: *if a `PractitionerRole` is available for the
acting practitioner, it SHALL be referenced; bare `Practitioner` or
`Organization` references are permitted only when no `PractitionerRole` is
known.*

#### Justification

The literal proposal in Disposition A only adds `PractitionerRole` to a flat
union, which leaves implementers free to keep sending bare `Practitioner`
references — defeating the reporter's underlying intent that
`PractitionerRole` be **preferred**. Slicing the reference (or adding a
conformance constraint) makes the preference machine-checkable and gives
validators something to enforce, which is more effective than a narrative
"preferred" note. It also keeps a clean upgrade path: senders who only have
`Practitioner` data today can still conform via the fallback slice while being
nudged toward `PractitionerRole`. The trade-off vs. A is slightly higher
authoring complexity in the IG and a small additional implementation burden
on senders that only persist `Practitioner` references.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the US Medication FHIR Implementation
Guide it targets is a successor to the **withdrawn** US PDMP on FHIR IG, and
that any reference-target decisions for `actor` should be made fresh against
the current US Medication IG ballot scope rather than carried forward from a
five-year-old PDMP-era request. Close as **Not Persuasive — Out of Scope** (or
**Considered for Future Use**, restoring the 2023-01-18 disposition that was
reverted in November 2023) until the work group reaffirms that this exact
`actor` element still exists in the current IG and that the reporter's
rationale still applies.

#### Justification

The ticket was originally filed in October 2020 against the PDMP on FHIR IG,
which the work group itself acknowledged as withdrawn in the 2023-01-18 vote.
The 2023-11-28 reopening was procedural (tied to the PSS-2225 reboot) and did
not include any new technical analysis. There is no Zulip discussion, no
linked or cross-referenced Jira tickets, and no GitHub artifact that confirms
the targeted `actor` element exists in the same form in the current US
Medication IG. Acting on a five-year-old proposed wording without that
verification risks editing a profile element that has since been renamed,
restructured, or removed.

---

### Recommendation

**Recommended disposition:** B

Disposition B best reflects what the reporter is actually asking for: not just
"please also allow `PractitionerRole`", but "`PractitionerRole` is *preferred*
because it carries organizational context that bare `Practitioner` does not."
A flat union (Disposition A) records that preference only in narrative, where
it is easy for implementers to ignore and impossible for validators to enforce.
Slicing the reference (or adding a conformance constraint with the same
effect) makes the preference normative, which is consistent with how other US
Realm IGs handle the `Practitioner` vs. `PractitionerRole` choice in clinical
exchange contexts. Disposition C is defensible given the ticket's age and the
absence of recent Zulip / GitHub activity, but the work group has already
chosen to re-open it in service of the relaunched PDMP / US Medication effort,
so a substantive technical disposition is more useful than a procedural close.
Before locking B in, the work group should confirm against the current US
Medication IG draft that the targeted `actor` element still exists with this
profile-union signature, and adjust the slice/constraint wording to match the
draft's actual element path.
