## Summary

This change request asks the Pharmacy Work Group to clarify the scope of
the US Prescription Drug Monitoring Program (PDMP) FHIR Implementation
Guide with respect to who acts as the "PDMP Requestor". Today, NCPDP-based
integration patterns allow both prescribers and pharmacies/pharmacists to
query a PDMP, and the data elements that must be supplied differ depending
on which actor is querying. The reporter (Daniel Zhang) notes that the
existing "PDMP Data Elements and Mappings" section of the IG presents a
single set of NCPDP mappings without distinguishing the requestor type. If
the IG is intended to support both actor types (which the reporter believes
to be the case), the NCPDP mappings need to be updated so that the required
versus optional data elements are correctly described per requestor role
(prescriber vs. pharmacist). No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this ticket; the only related
context is the project's Confluence page (linked above) and the
project-management ticket PSS-2225 ("PDMP on FHIR") referenced in the
re-open comment.

## Details

**Description:**

Current integration methods allow both prescribers and pharmacies to
perform queries of PDMPs. Is the intention of this implementation guide
to cover both of those users as a PDMP Requestor? If yes, NCPDP mappings
at least need to be updated to reflect how queries use different data
elements depending on whether the requestor is a prescriber or pharmacist.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Confirm in the US PDMP FHIR Implementation Guide that **both** prescribers
and pharmacists/pharmacies are in-scope as the "PDMP Requestor" actor, and
update the **PDMP Data Elements and Mappings** section so that NCPDP
mappings (NCPDP SCRIPT / PMIX query messages and their data dictionary)
are presented per-requestor-role. Concretely:

- Add a "Requestor Role" axis to the existing PDMP Data Elements and
  Mappings table so each row indicates whether the element is
  Required / Required-if-known / Optional / Not-applicable for a
  *prescriber* requestor versus a *pharmacist* requestor.
- For elements whose presence/optionality differs by role (e.g.,
  prescriber identifiers such as DEA number, NPI; pharmacy identifiers
  such as NCPDP Provider ID; patient identifiers used by each workflow),
  split the mapping row or add per-role footnotes citing the relevant
  NCPDP query message field.
- In the IG actor / capability statement narrative, name the two
  requestor sub-roles explicitly (e.g., "Prescriber-Requestor",
  "Pharmacy-Requestor") and link each to the corresponding NCPDP query
  pattern.
- Where FHIR resources surface the requestor (e.g., the requesting
  `Practitioner` / `PractitionerRole` / `Organization` referenced from
  the PDMP query operation parameters or supporting `Provenance`),
  document which slice/profile applies for each role.

#### Justification

This is what the reporter asked for and it directly removes ambiguity that
implementers will hit on day one. NCPDP's PDMP query patterns (SCRIPT /
PMIX-derived) genuinely do require different identifiers depending on
whether a prescriber or a pharmacist is asking, so a single undifferentiated
mappings table is factually incomplete. Splitting the table by role is a
narrative / mapping change rather than a structural change to FHIR
resources, so it is low-risk for downstream profiles. It also aligns with
the IG's restart under PSS-2225 ("PDMP on FHIR"): the re-open comment makes
clear that deferred items are being revisited as the IG is taken up again,
and clarifying the requestor scope is foundational to every other PDMP
query mapping in the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the single PDMP Data Elements and Mappings table
in place, **introduce an explicit Actor / Use-Case layer** in the IG and
restructure the mappings around it:

- Define two distinct actors in the IG (e.g., "Prescriber PDMP
  Requestor" and "Pharmacy PDMP Requestor") with their own
  CapabilityStatements describing which PDMP query operation(s) they
  invoke and which search parameters / operation parameters they must
  populate.
- Replace the monolithic PDMP Data Elements and Mappings section with
  **two parallel mapping pages**, one per actor, each cross-walking that
  actor's required FHIR inputs to the corresponding NCPDP fields. Keep a
  shared "common elements" page for fields that are role-invariant
  (e.g., patient demographics).
- If the FHIR query surface is the same operation for both actors (e.g.,
  a single `$pdmp-query` Operation), add an `OperationDefinition`-level
  conformance section that documents per-actor parameter obligations
  rather than duplicating the operation.

#### Justification

Splitting only the mapping rows (Disposition A) addresses the immediate
defect but leaves the IG's actor model under-specified; implementers and
testers still have to infer which role they are conforming to. Promoting
"requestor role" to a first-class actor makes downstream artifacts —
CapabilityStatements, examples, test plans, and the eventual
implementer-facing search/operation definitions — explicit and
testable. It also future-proofs the IG against additional requestor
types that the work group has discussed historically (e.g., delegated
staff, integrated EHR services), which would be cumbersome to bolt onto
a single combined mappings table. The trade-off is more authoring work
up front and a larger diff for the next ballot, which is acceptable
given the IG is being reopened from a withdrawn state under PSS-2225 and
will be republished anyway.

---

### Disposition C: Decline

#### Proposal

Decline the change and either (a) re-apply the prior disposition of
**"Considered for Future Use"** until the IG has a confirmed scope
statement under PSS-2225, or (b) close as **"Not Persuasive"** on the
basis that the IG, as currently scoped, is intended only for a single
requestor archetype and the mapping table is therefore correct as
written.

#### Justification

The ticket's first sentence is itself a question about scope ("Is the
intention of this implementation guide to cover both of those users…?"),
and the IG was withdrawn for years before being reopened. If the work
group decides the restarted IG (PSS-2225) will scope to only one
requestor type — most plausibly the prescriber-side query, which is the
historically dominant US PDMP integration pattern — then the existing
NCPDP mappings are accurate for that scope and no change is required.
Declining (or re-deferring) is also defensible procedurally: the prior
"Considered for Future Use" decision was reverted only as part of
restarting the project, not because the substantive answer changed, and
re-deferring keeps the comment open against the restarted IG without
committing the work group to a non-trivial mapping rewrite before scope
is settled.

---

### Recommendation

**Recommended disposition:** A

The reporter's underlying observation is factually correct: NCPDP PDMP
queries do behave differently depending on whether a prescriber or a
pharmacist is the requestor, and the current IG section presents a
single mapping table that does not reflect that. With the IG being
reopened under PSS-2225, the work group has a natural opportunity to
make this correction, and Disposition A is the smallest viable change
that fixes the documented defect without committing to the broader
authoring effort implied by Disposition B. Disposition B is attractive
and would arguably be the right end state, but it depends on actor /
capability decisions that the restarted project should make
deliberately rather than be forced into by a single change request;
those decisions can be tracked as a follow-up. Disposition C only makes
sense if the work group has already decided to scope the restarted IG
to a single requestor type, which is not reflected in any of the
material attached to this ticket. Recommend accepting and updating the
PDMP Data Elements and Mappings section per Disposition A, with a
follow-up ticket opened to evaluate the per-actor restructuring in
Disposition B during the reopened-IG planning under PSS-2225.
