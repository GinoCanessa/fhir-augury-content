## Summary

This ticket captures external feedback (from Gayle Donaldson, Assistant
Director, Kansas Board of Pharmacy) on **Section 7 "Mappings"** of the US
Prescription Drug Monitoring Program (PDMP) FHIR IG, together with
inline responses from the reporter, Scott M. Robertson (annotated as
"SMR-…" within the description). The submitter observes that Section 7
documents existing data-field mapping situations from prior standards
(NCPDP, PMIX, etc.) at a minimum level, but does **not** state what FHIR
itself is proposing as required data or mapping, and lists three
specific concerns drawn from existing mappings: (1) the **requester
identifier** must accommodate state license numbers for pharmacists who
lack NPIs and for VA nurses accessing PDMPs through the VA EMR, while
remaining unique per individual (no shared pharmacy NPIs / placeholders);
(2) the **requester role** must be carried in the query so a PDMP can
enforce state-specific authorization rules tied to provider role; and
(3) a **response type / status indicator** must appear in the response so
implementers and auditors can distinguish success, error, and "no data
found" outcomes for compliance and integration monitoring. SMR's inline
replies clarify that mappings are non-normative and exist for traceability;
that FHIR's `Identifier` model already supports multiple, qualified
identifiers (including state-issued licenses) per actor; that
authentication/authorization (including role) is layered through the
Security section's reference to US Core (OAuth 2 / SMART) but should be
surfaced more explicitly in this IG; and that the FHIR query response
model already echoes the query alongside errors or returned data, so the
remaining gap is **examples and narrative** rather than missing
machinery. No cross-referenced Jira tickets, Zulip threads, or GitHub
items were found.

## Details

**Description:**

**Section 7 Mappings**

While this section spells out existing data field mapping situations (at
minimum values), it does not outline what FHIR is proposing for required
data or mapping.

> **[SMR — the mappings are not normative content. Mappings show how the
> proposed FHIR resource elements correspond to data elements in the other
> standards. This provides insight into the nature of the data (as in
> your comment on state license number for pharmacists) and traceability
> for troubleshooting.]**

Considerations that should be made from existing NCPDP, PMIX, etc.,
mappings:

- Requester identifier needs to accommodate a state license number for
  pharmacists without NPIs and the VA health system, which allows nurses
  to access PDMPs through the VA EMR. All requester identifiers must be
  unique, so pharmacy NPIs or other placeholders cannot be used in place
  of individual identifiers.

> **[SMR — FHIR can accommodate any number of different identifiers,
> including multiple types of identifiers from one organization (an
> employee number and a practitioner ID from an organization) and the
> same kind of identifier from different organizations (e.g., state
> licenses from different states). Regarding the query request, the
> requestor does need to be uniquely identified (authentication) and
> verified as an allowed user (authorization) — which are part of
> Security (which references US Core) and likely needs additional text
> to explicitly address your concern.]**

- Requester role must be included in the request elements. Authorization
  for access to the PDMP is granted by each state after careful
  consideration of each registrant. The requester role needs to be
  captured to ensure provider authorization requirements are being met
  for access to data by appropriate role types.

> **[SMR — Authentication can include Role. It's not apparent in the IG
> as the Security discussion refers to US Core. Generally, FHIR IGs
> don't repeat "established and reused" content to minimize the need to
> update multiple IGs when an underlying capability is updated. However,
> points of concern, such as this, can and should be included in our
> IG, while still referring to US Core for the technical details.
>
> _Related: US Core for Authentication and Authorization focus on OAuth 2
> and SMART. The group needs to discuss if this is viable, or another
> Authentication scheme is needed._]**

- Response type needs to be included in the response elements. We need
  to know whether the request was successful, had an error, found data,
  etc. This helps us identify if there are issues within the integration
  producing a high number of errors and helps us with compliance and
  investigative work to be able to access our audit trail to identify
  whether a search was successful and/or produced data.

> **[SMR — The FHIR query response model includes the query in the
> reported response along with either errors or the found data. I think
> this points to the need for examples, and some additional narrative.]**

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update **Section 7 "Mappings"** of the US PDMP IG to (a) add an
introductory paragraph stating that the mappings are informative and
exist to show how PDMP FHIR elements correspond to NCPDP / PMIX / other
legacy data elements, and (b) extend the section with three new
sub-sections — or cross-references to existing IG pages — that
explicitly call out, with examples:

1. **Requester identifier handling** — show a `Practitioner.identifier`
   slice/example that uses a state pharmacy-board license number
   (system per state, e.g., `urn:oid:…` or a state-board URI) for
   pharmacists without an NPI, plus a separate slice/example for VA
   nurses using a VA-issued identifier. Reinforce that the identifier
   must be individual (no shared NPIs / placeholders).
2. **Requester role in the query** — bind a request parameter or
   `requester.extension` to a coded role (e.g.,
   `practitioner-role` from US Core or an IG-defined ValueSet) and
   require it on the query CapabilityStatement so PDMPs can enforce
   state-specific role authorization.
3. **Response type / status** — require the response Bundle to carry an
   `OperationOutcome` (and/or a coded response-status extension)
   distinguishing `success-with-data`, `success-no-data`, `partial`,
   and `error`, with examples of each.

Accompany the changes with at least one round-trip example in Section 7
showing query → response for both a successful match and a "no data
found" result.

#### Justification

This is the disposition that most directly answers the submitter's
three concrete asks while staying consistent with the reporter's own
inline replies (mappings remain informative; identifier multiplicity is
already supported; the response model already returns query + errors +
data — what is missing is **explicit narrative and examples** in this
IG). It avoids new normative machinery, leans on existing FHIR
constructs (`Identifier`, `OperationOutcome`, US Core role
vocabularies), and converts a generic "see US Core for Security" pointer
into PDMP-specific guidance — which the reporter already conceded is
appropriate for points of concern.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding Section 7 in place, **restructure the concerns out
of "Mappings" and into the IG sections that actually own them**, leaving
Section 7 strictly informative:

- Move the **requester-identifier** guidance into the PDMP query
  Practitioner profile (or a dedicated "Actors and Identifiers" page),
  defining identifier slices for NPI, state pharmacy/medical license,
  and VA identifier, with a `must-support` rule that at least one
  individually-attributable identifier is present.
- Move the **requester-role** requirement into the query
  CapabilityStatement / SearchParameter definition (e.g., a required
  `requester-role` query parameter bound to a PDMP-specific ValueSet
  that extends US Core practitioner roles), and document the
  authorization expectation in the Security page (still referencing US
  Core / SMART).
- Define a small **PDMP response profile** on `Bundle` (or extend the
  existing response artifact) that requires an `OperationOutcome` entry
  whose `issue.code`/`issue.severity` distinguishes success, no-data,
  partial, and error outcomes — and document this on the response page
  rather than in Section 7.

Section 7 would then contain only mapping tables plus pointers to these
home pages.

#### Justification

This treats the submitter's concerns as **conformance / profile** issues
rather than mapping-narrative issues, which is where FHIR IGs normally
encode "must be present" or "must be coded this way" rules. It avoids
mixing normative requirements into a non-normative mappings section,
keeps the mapping tables stable for traceability, and makes the
constraints machine-checkable via IG validators. The trade-off vs.
Disposition A is greater editorial churn (touches profiles,
CapabilityStatement, Security and response pages) and a longer review
cycle, but the resulting artifacts are more enforceable.

---

### Disposition C: Decline

#### Proposal

Decline the request as filed against Section 7 on the grounds that
Section 7 is **informative mapping content** and is not the right venue
for prescribing required data, identifier rules, role-based
authorization, or response status semantics. Note in the resolution
that the underlying capabilities the submitter is asking for are
**already addressed elsewhere** in the FHIR stack — `Identifier`
multiplicity, `OperationOutcome`, US Core / SMART for
authentication+role — and close as "Not Persuasive / Already addressed",
optionally inviting the submitter to file targeted tickets against the
specific profile, CapabilityStatement, or Security page if any concrete
gap remains.

#### Proposal context

The reporter's own inline annotations (SMR-…) already explain that
mappings are non-normative, that FHIR identifiers can carry the cases
called out, that role can ride on authentication, and that the response
model already returns query + errors + data.

#### Justification

Defensible on scope grounds: Section 7 is intentionally informative and
should not grow into a quasi-normative requirements list. The submitter's
substantive needs are met by mechanisms that already exist in FHIR /
US Core / SMART, and forcing them into the mappings section would blur
the normative/informative boundary other IGs rely on. The risk is that
declining without any IG change leaves the underlying complaint
("nothing in the IG points me at these capabilities") unanswered, which
is the reason most reviewers will be uncomfortable closing without at
least a narrative pass.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the smallest change that takes the submitter's three
concerns seriously and matches what the reporter's own inline replies
say is missing — **examples and explicit narrative** within the PDMP
IG — without re-architecting profiles, CapabilityStatements, and the
Security page in a single ticket. It also avoids the "Decline" optics of
Disposition C, which the workgroup will struggle to defend given that
the submitter is a state Board of Pharmacy describing real operational
constraints (pharmacists without NPIs, VA nurses, state authorization
rules, audit/compliance reporting). Disposition B is the more
architecturally satisfying answer and should be held in reserve: if the
workgroup decides during discussion that any of the three asks needs to
be **enforceable** rather than merely **described**, the relevant piece
of B (most likely the requester-role parameter and the response-status
profile) can be split into a follow-up ticket against the appropriate
profile or CapabilityStatement, leaving Section 7's narrative changes
from A intact.
