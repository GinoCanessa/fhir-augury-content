## Summary

The ticket asks the Pharmacy work group to update the version of the
NCPDP SCRIPT standard cited in the US Medication FHIR Implementation
Guide (originally the US Prescription Drug Monitoring Program / PDMP
FHIR IG context) from "NCPDP SCRIPT 10.6" to "NCPDP SCRIPT 2017071".
The reporter notes that regulation now requires all electronic
prescribing (eRx) transactions to employ SCRIPT 2017071, so any
narrative or normative text in the IG that still references the older
10.6 version is out of date and contradicts current regulatory
requirements. The ticket was originally deferred in 2023-01 as
"Considered for Future Use" because the PDMP IG had been withdrawn,
then reopened in 2023-11 when the PDMP-on-FHIR project (PSS-2225) was
picked back up; it was subsequently resolved as Persuasive with a
resolution note that the IG has been updated to reference SCRIPT
2017071. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items recorded in FHIR Augury for this issue, so this prep is
based solely on the ticket itself.

## Proposed Dispositions

Because the cross-reference index, Zulip search, and GitHub index all
returned no related items, none of the dispositions below can name a
specific source-file path; they instead name the IG narrative location
where the SCRIPT version string lives. If the work group wants
file-level pointers, the PDMP / US Medication FHIR IG repository
should be ingested and a `repo-analysis` briefing produced before
acting on this ticket.

### Disposition A: Accept As Requested

#### Proposal

Update every occurrence of "NCPDP SCRIPT 10.6" in the US Medication
FHIR Implementation Guide (the successor to the withdrawn US PDMP on
FHIR IG, per PSS-2225) to "NCPDP SCRIPT 2017071". This includes the
IG's external-standards / dependencies narrative, any conformance
statements that pin a SCRIPT version, the introduction/scope page,
and any glossary entry. Also update any version-bearing labels,
publication-box entries, or reference URLs (e.g., the NCPDP standards
page citation) so they consistently advertise SCRIPT 2017071. Confirm
that no profile, ValueSet, CodeSystem, or example still embeds the
literal string "10.6" in `description`, `purpose`, or comment fields.
The resolution note already on the ticket ("IG has been updated to
reference SCRIPT 2017071") indicates this work is at least partially
done; this disposition is the formal "yes, do exactly that, and
verify it is complete" path.

#### Justification

The change is purely editorial / version-pinning: it does not alter
any FHIR resource definition, profile constraint, terminology, or
exchange semantics. The reporter's stated regulatory driver — that
all eRx transactions are required to use SCRIPT 2017071 — means
keeping the old 10.6 reference in the IG would actively misinform
implementers and could cause non-conformant deployments. The
work-group already voted Persuasive, and the prior 2023-01 deferral
("Considered for Future Use") was withdrawn solely because the IG was
on hiatus, a condition that no longer holds now that PSS-2225 has
revived the work. There is no community disagreement on record (no
Zulip threads, no linked tickets) that would argue against making the
straightforward update.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hard-coding "NCPDP SCRIPT 2017071" in the IG text, change
the IG to reference NCPDP SCRIPT *generically* (e.g., "the current
federally-mandated version of NCPDP SCRIPT") and centralize the
version pin in a single dedicated location — for example a "Standards
Versions" / "External Specifications" page or a publication-parameter
entry — that is cited from every other place the standard is
mentioned. Optionally, capture the version string in a dedicated
CodeSystem / NamingSystem entry (or a publish-box parameter) so that
the next regulatory bump (e.g., from 2017071 to a future SCRIPT
revision) is a one-line change rather than a sweep across the IG.

#### Justification

Pharmacy regulation around SCRIPT has historically advanced
(10.6 → 2017071 → and likely further), and each bump has
predictably re-opened the same kind of editorial ticket against
NCPDP-aware IGs. Centralizing the version reference reduces the
maintenance burden, lowers the risk that a future sweep misses a
stale citation in an obscure narrative block, and makes it obvious
to ballot reviewers exactly which version the IG claims conformance
to. The trade-off versus Disposition A is a slightly larger
refactor up-front and a marginally more verbose narrative ("see the
External Specifications page") in exchange for substantially easier
future updates. It does not change any normative semantics, so it
remains compatible with the reporter's regulatory motivation.

---

### Disposition C: Decline

#### Proposal

Close the ticket as Not Persuasive on the basis that the request is
already substantively satisfied — the ticket itself carries a
resolution description stating "IG has been updated to reference
SCRIPT 2017071" — and that no separate change is needed. If anything
remains, treat it as a documentation nit and address it as part of
normal IG maintenance rather than as a tracked Change Request.

#### Justification

Declining is defensible only if the work-group can confirm by
inspection that every reference to "SCRIPT 10.6" has in fact been
removed from the current source of the US Medication FHIR IG and
that the published artefact already cites SCRIPT 2017071 throughout.
In that case, leaving the ticket open conveys no actionable work
item and risks being re-deferred again. The downside is that, with
no cached repo briefing available to verify the IG's current text,
"decline as already done" is a leap of faith; if even one narrative
page still says 10.6, declining would leave a regulatory mismatch
in a published spec.

---

### Recommendation

**Recommended disposition:** A

The ticket asks for a narrowly-scoped, regulation-driven editorial
correction with no controversy on record (no Zulip debate, no
cross-referenced tickets pulling in a different direction) and
already carries a Persuasive disposition and a resolution note
claiming the change is done. Disposition A is the lowest-risk
path: do exactly what was asked, then verify by grep that no
"SCRIPT 10.6" string survives in the IG source. Disposition B is
attractive as follow-up housekeeping — the SCRIPT version *will*
move again — but it is best filed as its own improvement ticket
rather than scope-creeping a regulatory correction. Disposition C
should be avoided unless and until the IG source has actually been
inspected end-to-end and confirmed clean, which cannot be done from
the data available in FHIR Augury for this ticket today.
