## Summary

The reporter (Lloyd McKenzie) observes that several extensions defined in
the **FHIR for FAIR** Implementation Guide declare an extension `Context`
of `Element` — the broadest possible type-level scope that allows the
extension to be applied anywhere in any resource. Because the extensions
are clearly intended for a much narrower set of use sites, the reporter
asks the FHIR for FAIR authors to tighten the `Context` declarations on
each extension so they assert only the resource(s), datatype(s), or
specific element paths where the extension is actually expected to
appear. No cross-referenced Jira tickets, Zulip threads, or GitHub items
were found by the augury index for this ticket, so the request stands on
its own.

## Details

**Description:**

> A number of the extensions defined in this IG have a scope of 'Element',
> which is broader than the obviously intended scope of the extensions.
> Please assert contexts that are narrower and reflect where the
> extensions are actually expected to be used.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Audit every `StructureDefinition` of `kind = "extension"` defined in the
FHIR for FAIR IG and replace any `context` entry of type `element` with
`expression = "Element"` (i.e., the maximally broad scope) with one or
more narrower entries that name the actual intended use sites. For each
extension:

1. Determine, from the IG's narrative and any examples, which
   resources, datatypes, or element paths are the intended targets.
2. Replace the broad `context` with one or more typed contexts such as
   `{ "type": "element", "expression": "Patient" }`,
   `{ "type": "element", "expression": "Observation.code" }`, or
   `{ "type": "fhirpath", "expression": "..." }` as appropriate.
3. Re-run IG publication and FHIR validation to confirm no existing
   examples or profiles in the IG (or known dependents) are invalidated
   by the tightened contexts.
4. Document the tightened contexts in the IG's change log entry for the
   next publication.

#### Justification

This matches FHIR's documented extension-design guidance: an extension's
`Context` should be as narrow as is meaningful for the use case so that
tooling, validators, and implementers can detect misuse. `Element` is
effectively "anywhere", which is rarely what an IG actually means and
undercuts validator value. Tightening the contexts is a low-risk,
documentation-style change for an IG: it does not alter the wire format
of any instance that was already using the extension correctly, and it
strengthens conformance. The reporter (Lloyd McKenzie) is the FHIR core
methodology lead, so this is consistent with the broader pattern of QA
sweeps applied to IGs across HL7.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same goal as Disposition A, but proceed in two phases:

1. **Phase 1 (this ballot/publication):** Tighten only the extensions
   whose narrower context is uncontroversial — i.e., the extension is
   only ever used in one resource or one element path inside the IG's
   own profiles and examples. For these, set a single concrete
   `context`.
2. **Phase 2 (follow-up ticket):** For extensions where the narrower
   context is genuinely ambiguous (e.g., the extension was intended to
   be reusable across several FAIR-related resources), open a
   targeted discussion with the FHIR for FAIR authors and any known
   downstream users to agree on the intended scope, then tighten in the
   following publication. Track the deferred extensions in a follow-up
   Jira ticket linked from FHIR-43150.

In both phases, prefer `type = "element"` with the narrowest
resource/element expression possible; only fall back to `type =
"fhirpath"` when an FHIRPath constraint is genuinely required.

#### Justification

Tightening an extension's `Context` is technically backward-compatible
for *correct* prior use, but it can break (i.e., emit validation errors
on) any out-of-band downstream consumer that was leaning on the loose
`Element` scope. Splitting the work lets the FHIR for FAIR team capture
the easy wins immediately without blocking the publication on
potentially contentious scope decisions for the trickier extensions, and
gives downstream implementers a clear deferred-work signal.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Persuasive with Mod" → "Not Persuasive" (decline) on
the basis that the FHIR for FAIR IG authors deliberately scoped the
extensions to `Element` to maximise reusability across FAIR data
representations (which are, by design, cross-resource), and that the
generic FHIR validator will surface no real implementer harm from the
broader scope.

#### Justification

This is the weakest of the three options and is included for
completeness only. It is defensible only if the FHIR for FAIR authors
can demonstrate, with concrete examples, that each `Element`-scoped
extension is genuinely intended to apply across the full FHIR resource
graph (not just "we were not sure where it would be used"). In practice
that bar is rarely met for an IG-defined extension, and "we were not
sure" is exactly the situation the reporter is asking the IG to resolve.

---

### Recommendation

**Recommended disposition:** B

The substantive request (tighten the contexts) is correct and well-aligned
with FHIR core methodology, so straight decline (C) is not defensible.
However, doing all the tightening in a single sweep (A) risks either
churn — if the team has to revisit decisions made under time pressure —
or breaking external implementers who relied on the loose `Element`
scope. The phased approach (B) lets the FHIR for FAIR work group apply
the obvious tightenings now, signal the remaining work explicitly via a
follow-up Jira ticket, and engage the FAIR community on the contested
extensions before locking in their final contexts. Workgroup discussion
should focus on identifying which extensions can move in Phase 1 and
nominating an owner for the Phase 2 follow-up.
