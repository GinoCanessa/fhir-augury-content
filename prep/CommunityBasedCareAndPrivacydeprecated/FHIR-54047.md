## Summary

LukeBreyer is asking the Community-Based Care and Privacy work group to
clarify which portions of the Scalable Consent Management IG's "consent
ceremony" narratives — spread across the Overview, Use Cases, and
Implementation Notes — are intended as normative conformance requirements
versus illustrative, informative examples. The ticket notes that consent
workflows are highly jurisdiction- and policy-specific, so without clear
signaling readers may misread descriptive scenarios as prescribing
required patient-facing or operational behavior. The reporter wants
explicit guidance on whether conformance depends on any particular
consent workflow or user-interaction pattern, and on which sections are
informative only. The ticket has been triaged as Persuasive with a
resolution direction of adding a link to the relevant technical
operation in each use case diagram and providing a table showing which
operations are required for each use case. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were found in the index.

## Details

**Description:**

The IG includes detailed descriptions of end-to-end "consent ceremony"
interactions across the Overview, Use Cases, and Implementation Notes.
These narratives provide helpful context, but it is not always clear
which aspects are intended to be normative requirements versus
illustrative examples.

Because consent workflows are highly jurisdiction-specific and
policy-driven, clearer signaling would help ensure these descriptions
are not interpreted as prescribing required patient-facing or
operational behavior, rather than demonstrating possible interactions.

Clarification would be helpful on whether conformance depends on any
specific consent workflow or user interaction pattern, and which
sections are intended as informative guidance only.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already proposed in Jira ("Persuasive"): in the
Scalable Consent Management IG, explicitly mark the consent-ceremony
narratives in the Overview, Use Cases, and Implementation Notes as
**informative**. Concretely:

- Add a short "Conformance scope" callout at the top of the Overview,
  Use Cases, and Implementation Notes pages stating that the narrative
  walkthroughs and ceremony descriptions are illustrative and that
  conformance is defined exclusively by the Consent profile(s),
  operation definitions, CapabilityStatement(s), and any conformance
  rules called out with normative keywords.
- For each use case diagram, add a link from the diagram caption to the
  specific technical operation (e.g., the relevant `$consent-*`
  operation, search interaction, or profile) that backs that step.
- Add a **"Operations by Use Case"** table to the Use Cases page that
  lists, for each use case, which operations / interactions / profiles
  are *required* for conformance versus which are *optional* or
  *example-only*.
- Audit the existing prose for normative keywords (SHALL / SHOULD / MAY)
  inside the illustrative sections and either move those sentences into
  a clearly-marked normative section or downgrade the keyword.

#### Justification

This matches the resolution the work group has already accepted as
Persuasive and directly addresses the reporter's concern: the ambiguity
is in the narrative framing, not in the underlying artifacts, so the fix
belongs in the IG's prose and use-case presentation rather than in the
profiles or operations themselves. It also follows long-standing FHIR IG
guidance that conformance is carried by the structured artifacts plus
explicit normative language, with narrative serving as illustration.
Adding the operations table gives implementers a single, unambiguous
checklist to point at when scoping a conformant implementation, which is
particularly valuable in a jurisdictionally-variable domain like
consent.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only annotating existing pages, restructure the IG so that
narrative and conformance content are physically separated:

- Split the existing Overview / Use Cases / Implementation Notes into
  two parallel page families: an **"Informative Guide"** family
  (ceremonies, walkthroughs, jurisdictional considerations, sample
  flows) and a **"Conformance"** family (Consent profile reference,
  operation definitions, required interactions, CapabilityStatement
  expectations).
- The Conformance pages own all SHALL/SHOULD/MAY language and the
  required-operations table; the Informative pages carry an explicit
  "non-normative" banner on every page and link *into* the Conformance
  pages instead of restating requirements.
- Optionally tag every example workflow with a small "Example —
  non-normative" badge so the distinction survives copy-paste into
  downstream documentation.

#### Justification

The reporter's concern is that today's structure makes it easy to
mistake illustrative ceremony prose for required behavior. Disposition A
fixes that with callouts and a table, but the underlying interleaving of
normative and informative content remains. A structural split makes the
distinction load-bearing in the IG's information architecture, so future
edits cannot accidentally re-introduce the ambiguity. The trade-off is a
larger documentation refactor and renumbered TOC, which is more
disruptive to readers familiar with the current layout and to any
external references already pointing at the existing page anchors.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that FHIR IG conformance conventions
already define how to read normative versus informative content
(normative keywords, the conformance section, and the structured
artifacts), and the existing IG content does not introduce SHALL-level
requirements in the narrative sections. Close as "Not Persuasive — works
as designed," with at most an editorial note pointing readers to the
existing conformance guidance.

#### Justification

This disposition is defensible only if a careful re-read of the
Overview, Use Cases, and Implementation Notes confirms that no
normative keywords are used in the ceremony narratives and that the
profiles / operations are the sole conformance surface. In that case the
ambiguity is a reading issue rather than a content issue, and adding
extra callouts could imply (incorrectly) that other IGs without such
callouts *do* treat narrative as normative. The downside is that the
reporter has explicitly raised confusion, which is itself evidence the
current signaling is insufficient — declining leaves the underlying
usability problem unresolved.

---

### Recommendation

**Recommended disposition:** A

The work group has already triaged this as Persuasive with a concrete
resolution (per-use-case links to technical operations plus a required
operations table), and that resolution maps cleanly onto Disposition A.
It is the lowest-risk, highest-clarity option: it directly addresses the
reporter's concern, requires only IG narrative edits and a new table
rather than a structural refactor, and preserves existing page anchors
and reader expectations. Disposition B is attractive long-term but is a
larger lift than the ticket warrants and should be considered separately
if reader confusion persists after Disposition A lands. Disposition C is
hard to defend given that an implementer has explicitly reported the
ambiguity. Proceeding with A also lets the work group close the ticket
in line with the resolution already recorded in Jira.
