## Summary

The ticket challenges the eLTSS mapping guidance that the Plan Monitor
(care manager) for a person-centered plan be conveyed indirectly via
`CarePlan.encounter` → `EpisodeOfCare.careManager` → `Practitioner`. The
reporter argues that source systems frequently lack a formal Encounter
or EpisodeOfCare — particularly at the moment a plan is first authored —
so making Plan Monitor identification dependent on those resources is
brittle. Because `CarePlan` already exposes `author`, `contributor`, and
`careTeam` references that can point directly at a Practitioner (or a
CareTeam containing one), the ticket asks the workgroup to drop the
indirection through Encounter/EpisodeOfCare and let Plan Monitor be
expressed directly on the CarePlan. A 2020 disposition closed the
ticket as "Considered for Future Use" pending implementer input;
in May 2024 lynn_laakso reverted that disposition, re-opened the
ticket post-STU2, and recorded concrete suggestions that align with
the reporter's intent: remove the EpisodeOfCare eLTSS Profile, add a
CareTeam eLTSS Profile that carries the Plan Monitor name and phone
number, update the mapping spreadsheet (XML + HTML), add (or identify)
a SNOMED code for Plan Monitor, and coordinate with the Security WG.
There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items in the index.

> → encounter to reference the EpisodeOfCare that defines the plan

> references the Practitioner who is monitoring the plan.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the May-2024 Suggestions block verbatim against the eLTSS FHIR
IG:

1. Retire the eLTSS `EpisodeOfCare` profile and remove EpisodeOfCare
   from the eLTSS resource list.
2. Introduce a new eLTSS `CareTeam` profile that carries the Plan
   Monitor's name and phone number. Concretely:
   - Slice `CareTeam.participant` so that one slice is constrained to
     `participant.role` = "Plan Monitor" (using the new SNOMED code
     once selected; if no suitable code is available at publication,
     bind to a code from a small eLTSS CodeSystem with the SNOMED
     binding noted as the future direction).
   - On that slice, require `participant.member` to reference
     `Practitioner` (or `PractitionerRole`) and require the referenced
     Practitioner to expose `name` and a `telecom` of system
     `phone`.
3. Re-route the `Additional Mapping Details` for "Plan Monitor Name"
   and "Plan Monitor Phone Number" so they target the new CareTeam
   profile (`CareTeam.participant[planMonitor].member.resolve()
   .name` and `.telecom.where(system='phone').value`) rather than
   `CarePlan.encounter.resolve().careManager.resolve()…`. Update the
   eLTSS Mapping spreadsheet, both the XML source and the rendered
   HTML.
4. File a SNOMED CT request (or pick an existing concept) for
   "Plan Monitor" — explicitly distinct from "Case Manager" — and
   record the chosen code in the new CareTeam profile and the
   mapping spreadsheet.
5. Engage Security WG to confirm that conveying the Plan Monitor via
   CareTeam (rather than EpisodeOfCare.careManager) does not regress
   the consent / access-control assumptions baked into the current
   eLTSS guidance.

#### Justification

This is the path the workgroup has already articulated in the May
2024 reopening, and it directly addresses the reporter's core
objection: source systems often have no formal Encounter or
EpisodeOfCare when a plan is first authored, so making Plan Monitor
identification depend on those resources is fragile. CareTeam is the
FHIR-idiomatic surface for "people responsible for a CarePlan" —
`CarePlan.careTeam` is already the canonical link — and the role
slice pattern is consistent with how other US Realm IGs (e.g.,
US Core CareTeam) convey participant function. Retiring the eLTSS
EpisodeOfCare profile also removes an artifact the workgroup has
flagged as not earning its place in the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `EpisodeOfCare` in the IG (do not retire its profile), but
*decouple* Plan Monitor identification from it: add a new eLTSS
`CareTeam` profile (as in Disposition A) and update the
"Plan Monitor Name / Phone Number" mapping rows to reference
`CarePlan.careTeam → CareTeam.participant[planMonitor]`.
EpisodeOfCare remains an *optional* alternative pathway for sites
that already model their LTSS engagement as an episode, with
`EpisodeOfCare.careManager` documented as a permitted secondary
source for the same data when present. No SNOMED dependency is made
blocking: ship with a small eLTSS-local CodeSystem entry for
"Plan Monitor" and replace it with a SNOMED code in a follow-up
publication once one is identified or requested.

#### Justification

This preserves backward compatibility for any existing eLTSS
implementer that has already wired up EpisodeOfCare-based
provenance for the care manager, while still removing the *required*
indirection that motivated the ticket. It also unblocks publication
on a realistic timeline by not making the change contingent on a
SNOMED CT terminology submission, which can take multiple cycles.
The trade-off vs. Disposition A is two competing pathways for the
same data — implementers will need clear guidance on which to
populate when both are available — but it lowers migration cost for
current adopters.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive**. Rationale: the current
guidance is non-normative mapping advice in the "Additional Mapping
Details" section, and STU2 already does not *preclude* implementers
from conveying the Plan Monitor via `CarePlan.author`,
`CarePlan.contributor`, or `CarePlan.careTeam` (as bheale notes in
the 2024-04-02 comment). Implementers who lack an Encounter /
EpisodeOfCare can simply use those direct references today. No
breaking change to the IG is required.

#### Justification

The ask is essentially a documentation / guidance issue: the FHIR
data model already permits the direct reference the reporter wants.
A profile-level change (retiring EpisodeOfCare, adding a CareTeam
profile, updating the mapping spreadsheet, requesting a SNOMED
code, coordinating with Security WG) is a sizeable multi-artifact
edit for a question that can be resolved by a sentence in the
mapping page. Declining keeps STU2 stable and avoids a partial
refactor that may need to be reopened anyway when the SNOMED code
question is settled.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The reporter's underlying complaint —
that Plan Monitor identification must not be predicated on the
existence of an Encounter or EpisodeOfCare — is legitimate, was
endorsed in substance by the May-2024 reopening, and should be
fixed in the IG rather than papered over with a documentation note
(so Disposition C is too thin). Disposition A is the workgroup's
own draft plan and is the right long-term shape, but it bundles
three independently-risky moves: retiring an existing profile,
introducing a new one, *and* taking a SNOMED CT terminology
dependency, plus a Security WG handshake. Tying publication to all
three lengthens the ballot cycle and gives the workgroup very
little room to fall back if the SNOMED request stalls or Security
WG raises concerns about losing the EpisodeOfCare-mediated
care-manager signal. Disposition B delivers the substantive fix
(direct CareTeam-based Plan Monitor mapping) on the next ballot,
keeps EpisodeOfCare available for current adopters, and lets the
SNOMED code and the EpisodeOfCare retirement be revisited in a
follow-up ticket once implementer feedback on the CareTeam
profile is in hand.
