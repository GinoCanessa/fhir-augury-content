## Summary

The ticket reports a publisher build INFORMATION message on the `Consent`
resource: `Consent.verification.verificationType` — the child element
`verificationType` repeats the name of its parent backbone element
`verification`. The reporter (David Pyke) recommends renaming the child to
`type` so the fully-qualified path becomes `Consent.verification.type`,
eliminating the redundant prefix.

This ticket is part of a paired clean-up of the same `verification`
backbone: its sibling [FHIR-50871](https://jira.hl7.org/browse/FHIR-50871)
asks for the analogous rename of `Consent.verification.verificationDate`
to `Consent.verification.date`. Both were filed on the same day by the
same reporter against the same build message class. Both have already
been marked **Persuasive** and are implemented together in
[HL7/fhir#3957](https://github.com/HL7/fhir/pull/3957) by Mohammad
Jafari, which renames `verificationType` → `type` and `verificationDate`
→ `date` in `source/consent/structuredefinition-Consent.xml`, the
example, the search-params bundle, and the resource diagram. The PR is
merged.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rename `Consent.verification.verificationType` to
`Consent.verification.type` in the Consent resource. Concretely, in
`HL7/fhir` at clone `db79dcd`, edit the `<differential>` of
`source/consent/structuredefinition-Consent.xml` so that the element
currently identified as `Consent.verification.verificationType` (path,
id, short, definition, mapping references) becomes
`Consent.verification.type`; update the example
`source/consent/consent-example-notThem.xml` to use `<type>` inside
`<verification>`; update any SearchParameter expression in
`source/consent/bundle-Consent-search-params.xml` that references the
old path; and update the diagram `source/consent/consent.svg`. This is
exactly the change made in PR
[HL7/fhir#3957](https://github.com/HL7/fhir/pull/3957) (which also
applies the parallel `verificationDate` → `date` rename for
[FHIR-50871](https://jira.hl7.org/browse/FHIR-50871)).

#### Justification

The build INFORMATION message exists precisely because FHIR's element
naming convention discourages restating the parent's name in the child
(the child is already qualified by its parent in the path). Within the
`verification` backbone, `type` is unambiguous and consistent with
similar shortened child names elsewhere (e.g., `Patient.contact.name`,
`Encounter.participant.type`). The clean-up has already been applied
together with its sibling FHIR-50871 in a single, small, merged PR
touching only `source/consent/`, exactly the authoring root the
briefing identifies for FhirCore content. The disposition is recorded
as Persuasive in Jira, which aligns with this proposal.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the rename only as a **publisher-side aliasing / mapping** rather
than as a hard rename of the element name on the wire. In the Consent
StructureDefinition `<differential>`, keep
`Consent.verification.verificationType` as the canonical element name
but add an `<alias value="type"/>` and update the short/definition to
use the friendlier name in narrative. Optionally raise the build
INFORMATION message's severity threshold (or suppress it for this
specific path via `suppressed-messages.txt`) to silence the warning
without changing the element identifier.

#### Justification

The trade-off vs. Disposition A is purely about backward compatibility.
`Consent.verification.verificationType` is already a published element;
renaming it is, by FHIR's compatibility rules, a breaking change for
any instance, profile, search expression, validator, or generated code
that uses the old name. Aliasing leaves all of those consumers
untouched while still acknowledging the naming-convention issue in the
human-readable spec text. If the work group has reason to believe there
is meaningful downstream usage (e.g., from US Core, IHE, regional
IGs), B avoids forcing a coordinated breaking change for what is, at
heart, a stylistic clean-up flagged at INFORMATION (not WARNING/ERROR)
severity.

---

### Disposition C: Decline

#### Proposal

Decline the rename and keep `Consent.verification.verificationType`
(and, by extension under FHIR-50871, `verificationDate`) as published.
Add an entry to `suppressed-messages.txt` (or the equivalent
build-suppression mechanism) for the two `Name of child overlaps with
name of parent` INFORMATION messages on `Consent.verification.*` so
they no longer surface in the build, with a rationale comment pointing
back to FHIR-50872 / FHIR-50871.

#### Justification

The publisher emits this at INFORMATION severity precisely because it
is advisory, not a defect. Declining is defensible when the cost of
churn (every existing Consent instance, every profile that constrains
`verification.verificationType`, every search-parameter expression,
every implementer code-gen artefact, and every conformance test) is
weighed against a purely cosmetic improvement on a resource that has
already shipped through multiple FHIR releases. C is the
backward-compatibility-first option and, with the suppression entry,
removes the recurring noise from future builds.

---

### Recommendation

**Recommended disposition:** A

The work group has already moved this way: the resolution is recorded
as **Persuasive**, and PR [HL7/fhir#3957](https://github.com/HL7/fhir/pull/3957)
is **merged** with the rename applied to all four authored files under
`source/consent/`. The change is small, contained to a single
authoring root that the FhirCore briefing explicitly calls out
(`source/consent/`), follows the same naming convention used on
analogous backbone children elsewhere in FHIR, and is paired with the
sibling FHIR-50871 fix so the `verification` backbone is cleaned up in
one stroke. The principal risk is the breaking-change concern raised
under Disposition B, but the work group's acceptance of the rename
indicates it judges the long-term consistency benefit to outweigh the
migration cost — and the tooling cost (regenerating snapshots,
updating downstream profiles/search params) is modest and
mechanically driven by the rename. Disposition A should be confirmed
and the implementing PR signed off; Disposition B is the fall-back if
the work group revisits the breaking-change calculus.
