## Summary

Craig Newman is asking the Community-Based Care and Privacy work group to
clarify how the conformance language in Section 7.2.3 of the *Scalable Consent
Management (FHIR)* specification relates to the entries on the Use Case page.
Section 7.2.3 states that a "consent administration service" must support
filing and revoking consent, but the spec also uses other names — "consent
administration system" and (in the Subscription Topic profile) "Consent
Registry" — for what may be the same actor, and it is not clear whether these
are intended to be the same thing. The ticket also questions whether "filing
consent" in 7.2.3 is meant to subsume the request/review use cases or only
the formal filing step. Finally, because only one entry on the Use Case page
is marked optional, the implication is that the remaining use cases are
mandatory, which appears to over-constrain conformance compared with what
7.2.3 actually requires. The resolution captured on the ticket already
indicates a "Persuasive" disposition with a plan to (a) add an informative
mapping from business use cases to technical operations and (b) standardise
on "Consent Client Systems" and "Consent Server Systems" terminology
throughout the IG.

## Details

**Description:**

Section 7.2.3 requires that a consent administration service (presumably the
same as a consent administration system and perhaps "Consent Registry" as
used in the Subscription Topic profile) must support filing and revoking
consent. Do these relate to the similarly named use cases on the Use Case
Page? Does filing consent include requesting and reviewing consent or just
the formal filing? Only a single use case on the Use Case Page is listed as
optional. This suggests that all other use cases are requirements which may
not align with 7.2.3.

**Resolution Description (already recorded on the ticket):**

> We are going to provide an informative mapping from the business use cases
> to the technical operations to show when the operations are used. We will
> also ensure we are using consistent language of Consent Client Systems and
> Consent Server Systems.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already drafted on the ticket. Specifically, in the
Scalable Consent Management (FHIR) IG:

1. **Standardise actor terminology.** Replace every occurrence of "consent
   administration service", "consent administration system" and the
   Subscription Topic profile's "Consent Registry" with the two-actor pair
   **Consent Client System** and **Consent Server System**, and add a short
   actor-definition block (probably in the IG's "Actors" or introductory
   section) that defines them once. Update Section 7.2.3 and the
   Subscription Topic profile narrative to use the new terms.
2. **Add an informative use-case → operation mapping.** Insert a new
   informative subsection (e.g., "Use Case to Operation Mapping") on or
   near the Use Case page that lists each business use case and identifies,
   for each one, the technical operation(s) (file, revoke, request, review,
   etc.) that realise it. Mark the table explicitly as informative so it
   does not add new conformance requirements.
3. **Re-align Section 7.2.3 with the Use Case page.** Restate 7.2.3 in
   terms of the technical operations a Consent Server System SHALL support
   (file, revoke) and clarify that "filing consent" refers only to the
   formal filing operation, not to request/review. Re-mark optional vs.
   required use cases on the Use Case page so that only those use cases
   whose underlying operations are SHALL in 7.2.3 are presented as
   required; the rest become optional / informative.

#### Justification

This is exactly what the reporter asked for and it matches the resolution
text already on the ticket ("Persuasive"). It removes three sources of
implementer confusion in one pass: inconsistent actor names across the IG,
ambiguity between the business-level "use cases" and the SHALL-level
conformance operations, and an over-broad reading of Section 7.2.3 caused
by the "only one use case is optional" framing. Because the operation
inventory itself does not change — only the language around it and an
informative mapping — this can be done without breaking existing
implementations.

---

### Disposition B: Alternative Approach

#### Proposal

Address the underlying confusion via a structural rewrite rather than a
terminology pass plus an informative mapping. Specifically:

1. Move all conformance language out of Section 7.2.3 prose and into a
   formal **CapabilityStatement** (and/or per-actor conformance
   requirements section) for the Consent Server and Consent Client actors,
   listing the supported operations and their `expectation` (SHALL /
   SHOULD / MAY).
2. Demote the Use Case page to purely descriptive narrative — strip the
   "optional / required" markers entirely — and let the CapabilityStatement
   be the single source of truth for what a conformant system must do.
3. Keep the actor renaming from Disposition A (Consent Client System /
   Consent Server System) so the CapabilityStatement and Subscription Topic
   profile align.

#### Justification

This approach is more durable than informative prose: future readers do not
have to reconcile a use-case table against a conformance section, because
conformance lives in machine-readable artifacts. It also matches how most
mature FHIR IGs separate "what the IG is for" (use cases) from "what a
conformant system must do" (CapabilityStatement). The trade-offs vs.
Disposition A are (a) more authoring effort, (b) a likely larger ballot
delta if 7.2.3 is being reworked rather than just clarified, and (c) some
risk that implementers who have already keyed off the Use Case page will
need to re-map their understanding to the new CapabilityStatement.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that Section 7.2.3 is the binding
conformance statement and the Use Case page is — by section heading —
intended as illustrative material. Add at most a one-sentence editorial
note at the top of the Use Case page reminding readers that the page is
illustrative and that conformance is governed exclusively by 7.2.3.

#### Justification

A defensible "decline" rests on the position that no normative content is
actually wrong: 7.2.3 already states the SHALL requirements, and use-case
pages in FHIR IGs are conventionally non-normative. Under that reading the
ticket is asking for clarification of something that is already true,
which can be handled with a single editorial sentence rather than
terminology and structural changes.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified three concrete, implementer-visible problems
(inconsistent actor names, ambiguity over what "filing" covers, and an
over-broad reading of "required" on the Use Case page), and the work group
has already recorded a "Persuasive" resolution with the same fix. Disposition
A delivers exactly that fix with the smallest possible ballot delta and no
breaking change to the operations themselves. Disposition B is attractive
in the long run — a CapabilityStatement-driven IG is cleaner — but it is a
larger rework than this ticket scopes and is better raised as its own
follow-up. Disposition C is hard to defend given that the reporter has
shown the existing wording is in fact misleading; a single editorial
sentence will not fix the actor-name inconsistency or the
file/request/review ambiguity. Recommend the work group accept A and, if
desired, log a separate ticket to consider B as a future ballot cycle
clean-up.
