## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Author and insert a Pre-Condition subsection for Section 3.4 ("Use Case
3.4") of the Scalable Consent Management (FHIR) IG, mirroring the
Pre-Condition structure used in the surrounding use-case sections (e.g.,
3.1–3.3, 3.5+). The new Pre-Condition should enumerate, at minimum:

- The actors that must already exist (e.g., Patient, Consent grantor,
  custodian/organization, requesting party).
- The Consent and supporting resources that must already be on record
  (e.g., an active `Consent` resource with the relevant
  `category`/`scope`/`provision` for the scenario in 3.4).
- The system/state assumptions (e.g., the Consent decision service is
  reachable, identity for the requester has been established, applicable
  policies are loaded).
- Any data prerequisites referenced later in the Main Flow of 3.4 so the
  flow is internally consistent.

The change is editorial/narrative only — IG narrative page for Use Case
3.4 — with no StructureDefinition, ValueSet, or example artifact changes.

#### Justification

The ticket is explicitly an editorial gap report, not a model change, and
the workgroup has already coded the resolution as Persuasive with the
note "We will add pre-condition information." Adopting the request as
asked restores narrative consistency across Section 3 (every other use
case carries a Pre-Condition), removes a known reader-facing defect, and
is low-risk because no normative artifact, schema, or terminology is
affected. It is also the smallest change that fully discharges the
ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring Section 3.4's Pre-Condition in isolation, perform
a narrow audit of all use-case sections in Section 3 of the Scalable
Consent Management IG and:

1. Define a single canonical "Use Case Template" (Pre-Condition,
   Trigger, Main Flow, Post-Condition, Variations) in the IG's
   contributor guidance.
2. Backfill Section 3.4's Pre-Condition against that template.
3. Spot-check the other Section 3.x use cases for the same omission and
   open follow-on tickets (or fix in-line) for any others that are
   missing required template slots.

The textual content added to 3.4 is the same as Disposition A; the
difference is that the fix is delivered alongside a structural guard
that prevents the same gap from recurring elsewhere.

#### Justification

The reported defect — a missing template slot in a use-case section — is
a class of bug that often appears in more than one place when an IG's
use-case sections were authored over time by different contributors.
Standardising the template costs a small amount of additional editorial
work but yields a durable fix and makes future review (including ballot
review) cheaper. The trade-off vs. Disposition A is a slightly larger
edit footprint and a longer review cycle for what is, on its face, a
single-section ticket.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54358 as "Not Persuasive — already addressed editorially" if
the workgroup determines that the Pre-Condition for Section 3.4 is
already conveyed by the section's narrative prose or by a shared
"Common Pre-Conditions" block elsewhere in the IG, making a dedicated
Pre-Condition heading redundant.

#### Justification

Decline is only defensible if Section 3.4's existing prose already
satisfies the informational need that a Pre-Condition heading would
serve. If that is the case, adding a separate Pre-Condition would
duplicate content and introduce a maintenance hazard (two places to keep
in sync). This is the weakest of the three options because the
workgroup has already self-coded the ticket as Persuasive, indicating
they accept the gap exists; declining would require reversing that
finding.

---

### Recommendation

**Recommended disposition:** A

The ticket is a tightly scoped editorial fix on a single IG narrative
section, the workgroup has already marked it Persuasive with an explicit
intent to add the missing content, and there is no cross-cutting signal
(no related Jira tickets, no Zulip discussion, no GitHub work) that
suggests a broader problem worth bundling. Disposition A discharges the
ticket with the minimum disruption to the IG and matches the resolution
the workgroup has already recorded. Disposition B is a reasonable upsell
if the workgroup independently wants to harden Section 3's structure,
but it should be pursued as a separate housekeeping ticket rather than
being loaded onto FHIR-54358. Disposition C is not recommended because
the workgroup's own resolution note contradicts it.
