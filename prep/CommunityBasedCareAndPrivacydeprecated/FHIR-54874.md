## Summary

John Moehrke is asking the Scalable Consent Management IG to make explicit
guidance that an existing `Consent` instance should not be updated once it is
active. Instead, when the terms of consent change, the prior `Consent` should
be marked inactive and a new `Consent` resource created carrying the new
terms. The request is grounded in two concerns: (1) legally, changing the
terms of a consent is treated as forming a new contract, so mutating an
existing instance conflates two distinct legal artifacts; and (2)
operationally, ingesting consent terms into a downstream access-control
engine is expensive, so consumers need confidence that an active `Consent`
will not silently change underneath them. The reporter highlights this as
particularly important data/consent governance for shared consent registries
that multiple organizations rely on. The IG editor has already triaged the
request as **Persuasive** and noted the historical context (the dedicated
"Update Consent" operation was previously removed for exactly this reason)
and committed to adding an "Update Consent" subsection to the technical page
stating that consents SHOULD NOT be updated.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned for this ticket, so the scope is fully captured by the description
and resolution above.

## Details

> (mostly never) updated. That if there is a change in Consent, that the

> the new terms. This may be intended, but is not clear. The rational is
> that changing terms of a consent is legally considered a new contract; and
> ingesting the terms into an access control engine is an expensive process.
> Thus there should be confidence that a Consent, once marked active, will

> Update Consent on the technical page to indicate that consents SHOULD NOT
> be updated.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the editor's stated resolution as-is: add an **"Update Consent"**
subsection to the IG's technical page in the Scalable Consent Management
(FHIR) IG that explicitly states a `Consent` resource SHOULD NOT be updated
once it is in `active` status. The guidance should:

- Direct implementers, when terms change, to set the prior `Consent.status`
  to `inactive` (and optionally link forward via `Consent.relatedReference`
  / `Consent.policyBasis` or an equivalent IG-defined linkage) and to
  create a **new** `Consent` resource carrying the new terms.
- Define the narrow exceptions where update IS permitted (e.g., correcting
  pre-active drafts in `proposed`/`draft` status, fixing administrative
  metadata that does not affect the consent terms — clearly enumerated).
- Note the rationale: legally a change of terms forms a new contract, and
  downstream access-control engines and shared consent registries need a
  stable, ingestible artifact.
- Use SHOULD NOT (RFC 2119) consistent with other IG conformance language,
  and cross-reference the historical removal of the `$update-consent`
  operation so readers understand why no operation is provided.

#### Justification

This is exactly what the reporter asked for, what the editor agreed to, and
what is consistent with the IG's prior design choice to remove the
`$update-consent` operation. Codifying the rule on the technical page
closes a documentation gap rather than introducing a new constraint, so it
carries no breaking-change risk for conformant implementations. SHOULD NOT
(rather than SHALL NOT) preserves room for the legitimate edge cases
(pre-active drafts, administrative corrections) without weakening the
core governance guarantee that consumers and shared registries depend on.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the request but address it more comprehensively than a
single prose subsection. Specifically:

1. Add the "Update Consent" SHOULD NOT subsection from Disposition A.
2. Additionally define a normative **Consent lifecycle** section on the
   technical page that diagrams the supersession pattern: `active` →
   `inactive` with a forward link from the inactive instance to its
   replacement (using `Consent.relatedReference` with an IG-defined code,
   or a small IG extension if no suitable element exists).
3. Add a conformance requirement on consumers (e.g., a CapabilityStatement
   note or a SHALL on registry actors) that they MUST honour
   `Consent.status = inactive` and refresh from the superseding instance,
   so the immutability guarantee is matched by a consumer-side obligation.
4. Provide one or two worked examples (a superseded `Consent` and its
   replacement) in the IG's example set.

#### Justification

The reporter's underlying need is not just "say it in prose" — it is that
shared consent registries and access-control engines can rely on the
immutability guarantee. A bare SHOULD NOT in narrative achieves the
documentation goal but leaves the supersession/linkage mechanics implicit,
which is where interoperability tends to fragment. Defining the linkage
element, the consumer-side obligation, and worked examples makes the
guidance actionable and testable. The cost is more editorial work and a
slightly larger conformance footprint; the benefit is that downstream
implementers (especially registry operators) get a complete pattern rather
than a rule with no recipe.

---

### Disposition C: Decline

#### Proposal

Decline to add explicit guidance, on the rationale that FHIR's general
`Resource.meta.versionId` and the existing `Consent.status` value set
already give implementers the tools to express supersession, and that
local/jurisdictional policy is the appropriate place to set
"don't-update" rules rather than a global IG SHOULD NOT.

#### Justification

A "decline" position is defensible only if one believes IG-level guidance
would over-constrain jurisdictions that legitimately treat certain
administrative updates (e.g., correcting an identifier, attaching a
late-arriving signature) as in-place edits. Declining keeps the IG silent
and leaves the choice to deployments. However, this position has to
acknowledge the editor's contrary view on the ticket and the historical
removal of `$update-consent` for the same reason — both of which point
toward the IG actually wanting to take a position here.

---

### Recommendation

**Recommended disposition:** A

The reporter, the editor, and the IG's own design history (removing
`$update-consent`) all converge on the same answer: the IG should state
plainly that an active `Consent` SHOULD NOT be updated and that term
changes are expressed by inactivating the prior instance and creating a
new one. Disposition A delivers exactly that with minimal editorial
surface, no breaking changes, and no new conformance machinery, which
matches the ticket's "Persuasive" triage. Disposition B is attractive and
arguably the right long-term shape of the guidance, but it expands scope
beyond what the ticket asked for and beyond what the editor committed to
on the ticket; it is better filed as a follow-up enhancement once the
basic SHOULD NOT is in place. Disposition C is not recommended — it
contradicts both the reporter's well-reasoned governance argument and the
IG's own prior decision to remove the update operation. The workgroup
should accept A and, if appetite exists, open a follow-up to layer in the
linkage element, consumer-side obligation, and worked examples from B.
