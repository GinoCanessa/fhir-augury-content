## Summary

The Scalable Consent Management IG uses inconsistent terminology to refer
to the parties that participate in its scenarios. The Use Case page
introduces one set of names it labels "Actors"; the Conformance page
instead refers to the equivalent parties as "Systems"; and the
Requirements text uses neither vocabulary. The reporter (John Moehrke)
flags §MS-1 of the Conformance page as a concrete example: it places a
SHALL on "Data Sources," a label that does not appear on either the Use
Case page or the Systems list. The problem is described as pervasive
across the requirements, making it hard for an implementer to map a
requirement back to the actor/system that must satisfy it. The ticket
has been triaged as **Persuasive**, with an authored resolution
committing to (1) define a single canonical list of actors and systems,
(2) sweep the IG to use those names consistently, and (3) link each use
case diagram to the corresponding technical operation and add a table
mapping use cases to required operations.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already authored on the ticket verbatim:

1. Add (or upgrade) a single section in the Scalable Consent Management
   IG that enumerates every party that participates in the use cases or
   carries normative obligations, splitting them explicitly into:
   - **Actors** — human / organisational roles that drive scenarios on
     the Use Case page; and
   - **Systems** — software components that carry conformance
     obligations on the Conformance page.
   Provide a one-paragraph definition for each entry and a stable
   anchor (e.g. `#actor-data-source`, `#system-data-source`) so other
   pages can link to it.
2. Sweep the IG (Use Case page, Requirements, Conformance page,
   profile narratives, capability statements) and rewrite every
   informal label (e.g. "Data Sources" in §MS-1) to use the canonical
   actor/system name from step 1. Where a requirement applies to more
   than one system, name each one rather than collapsing them.
3. On every use case diagram, add a hyperlink from the depicted
   interaction to the corresponding technical operation (e.g.
   `Consent.$find`, `Consent` search, `$consent-decision`, etc.).
4. Add a "Use Case → Operations" table that, for each use case, lists
   the operations a conformant system SHALL or SHOULD support.

#### Justification

The reporter and the work group already converged on this approach —
the ticket is marked **Persuasive** with an authored resolution. The
change is purely editorial and traceability-improving: it neither
removes nor weakens any normative statement, but it does make
conformance testable by allowing an implementer to determine which
system bears each obligation. The §MS-1 example is a textbook
traceability defect (a SHALL pinned to a label that is not defined
anywhere else in the IG); fixing it once at the glossary level
dissolves the entire class of issue. Coupling the terminology cleanup
with the use case → operation mapping table also addresses the
"requirements don't use either of these" half of the complaint, which
a glossary alone would leave untouched.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent but tighten the resolution into a single layered
model rather than two parallel lists:

1. Define one canonical glossary of **Systems** (software roles that
   carry conformance obligations) — e.g. *Consent Authoring System*,
   *Consent Repository*, *Consent Consumer*, *Policy Decision Point*.
   This is the only vocabulary used in the Conformance page,
   Requirements, profiles, and capability statements.
2. Treat **Actors** on the Use Case page as a strictly narrative
   overlay: each Actor is annotated with the System(s) that realise it
   in a deployment, using a short table at the top of the Use Case
   page (e.g. *"Patient Portal" → Consent Authoring System,
   Consent Consumer*).
3. Drop human-only labels like "Data Sources" from normative text
   entirely; use a defined system name (e.g. *Consent Repository*) in
   §MS-1 and equivalents.
4. Add the use case → operation table proposed in Disposition A; this
   piece is uncontroversial.

#### Justification

The two-list approach in Disposition A risks recreating the original
defect at a smaller scale: implementers will still have to reason
across two vocabularies (Actors vs Systems) and decide which one a
given requirement attaches to. A single conformance vocabulary, with
Actors layered on top as a narrative convenience, removes the
ambiguity at the source: every normative statement attaches to a
System; every use case statement names an Actor and the System(s) that
realise it. The trade-off is that this is a slightly larger editorial
sweep than what the ticket's resolution text describes, and reviewers
familiar with the existing Use Case page will need to confirm the
Actor → System mapping is faithful. It is, however, a one-time cost
that prevents the same ticket from reappearing the next time someone
adds a requirement.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** on the grounds that the IG already
defines its actors and systems on their respective pages, and that the
appearance of "Data Sources" in §MS-1 is an isolated copy-edit issue
that can be fixed in place without an IG-wide terminology project.

#### Justification

Declining would let the work group avoid the editorial sweep across
the Use Case page, Conformance page, Requirements, profiles, and
capability statements, and instead handle the §MS-1 wording as a
local fix. This is the least disruptive option for an IG already on a
ballot cycle.

It is, however, a weak position: the reporter explicitly states the
problem "exists across all requirements," and the work group has
already accepted the request by marking it Persuasive with an
authored resolution. Reverting that disposition would require new
evidence that the inconsistency is localised, which the ticket and
the cited example do not support.

---

### Recommendation

**Recommended disposition:** A

The ticket is already marked **Persuasive** with a concrete,
implementable resolution authored by the work group, and there is no
Zulip or GitHub debate on file that would justify reopening that
decision. Disposition A simply executes the agreed plan — define the
actors and systems, sweep the IG, link diagrams to operations, and
publish the use case → operations table — and so is the path of least
risk and least churn. Disposition B is the more architecturally clean
answer and worth flagging in discussion, but it expands scope beyond
what the work group has signed off on; if the editor doing the sweep
finds that an Actors-and-Systems split keeps producing edge cases, B
can be adopted as a follow-up without losing any of the work done
under A. Disposition C contradicts the existing Persuasive
disposition and the reporter's evidence and should only be considered
if new information surfaces during the sweep.
