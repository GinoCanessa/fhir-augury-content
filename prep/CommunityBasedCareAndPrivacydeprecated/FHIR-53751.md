## Summary

The Scalable Consent Management (FHIR) IG uses two parallel sets of system
terminology that the reporter, Craig Newman, finds inconsistent and
under-specified. The Must Support expectations describe systems as "data
sources" and "data consumers", while Section 6.1 instead defines the
participating systems as **Client** and **Server**. The reporter asks
whether either the Client or the Server can play either role (data
source vs. data consumer), or whether each system type is locked to a
single role.

The ticket also flags that the Use Cases narrative introduces a
third term, **ConsentSystem**, without aligning it to the Client /
Server pair defined elsewhere (where the Client is typically the EHR
and the Server is the Consent Management System). For example, in
Section 3.3 ("Sign and File"), it is unclear whether the
ConsentSystem is the EHR (which records the Consenter's approval) or
the Consent Management System (with the Consenter operating in the
EHR). The reporter is asking the work group to make the system / role
mapping explicit and consistent across the Must Support, capability
statements, and use case sections.

The ticket has already been triaged with a resolution of **Persuasive
with Modification**: the work group plans to update the use cases to
introduce an explicit client system (typically the EHR) alongside the
existing Consent System — i.e. **Consenter → Consent Client System →
Consent Server System** — and to rename the capability statements to
**ConsentClient** and **ConsentServer**. There are no cross-referenced
Jira tickets, Zulip threads, or GitHub items in the Augury index for
this ticket.

## Details

**Description:**

> The Must Support expectations use the terms "data sources" and "data
> consumers" but Section 6.1 defines the systems as Client and Server.
> Is it expected that either client or server could act as either data
> source or consumer or is the system tied to a particular role.
>
> Furthermore, the Use Cases include the ConsentSystem. Other pages
> define the Client (usually the EHR) and Server (the Consent
> Management System) but it's not clear how these systems function in
> the use cases. For example, in Section 3.3 Sign and File, is the
> ConsentSystem the EHR which records the Consenter's approval or is
> that the Consent Management System (with the Consenter acting in the
> EHR)? Please clarify how these systems fit into the use cases.

**Resolution Description (already on the ticket):**

> We will update the use cases to have an explicit client system
> (generally representing an EHR) along with the current Consent
> System.
>
> e.g., Consenter → Consent Client System → Consent Server System.
>
> Then we will change the capability statements to be ConsentClient
> and ConsentServer.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already drafted on the ticket. Concretely:

1. **Use cases (incl. Section 3.3 "Sign and File" and the broader Use
   Cases narrative):** introduce an explicit **Consent Client System**
   actor — described as "generally representing an EHR" — alongside
   the existing **Consent Server System**. Restructure each use case
   sequence as **Consenter → Consent Client System → Consent Server
   System**, replacing every standalone reference to "ConsentSystem"
   with whichever of those two actors is performing the step.
2. **Capability statements:** rename the existing Client / Server
   capability statements to **ConsentClient** and **ConsentServer**
   (and update their `id`, `name`, `title`, canonical URL, and any
   internal cross-links in the IG to match the new names).
3. **Must Support expectations (Section 6.1 and the Must Support
   page):** keep "data source" and "data consumer" as *role* labels,
   but explicitly state which of ConsentClient / ConsentServer plays
   which role in each interaction (or, if either system can play
   either role for a given exchange, say so explicitly so readers
   stop trying to infer it). Add a single short subsection that maps
   actor → capability statement → role (data source / data consumer)
   so the three vocabularies line up in one place.

#### Justification

This is the resolution the work group has already recorded on the
ticket as "Persuasive with Modification", and it directly answers
both halves of the reporter's question: the use case actor is named,
and the Client / Server capability statements get unambiguous names
that match the use case actors. Naming the capability statements
ConsentClient / ConsentServer follows the same pattern many other
HL7 IGs use (e.g., `*Client` / `*Server` capability statement pairs)
and gives implementers a stable canonical to declare conformance
against. Keeping "data source" / "data consumer" as role labels (rather
than removing them) preserves the Must Support framing while making
the actor-to-role mapping explicit, which is the specific gap the
reporter called out.

---

### Disposition B: Alternative Approach

#### Proposal

Make the same actor split (Consenter, Consent Client System, Consent
Server System) and the same capability-statement rename (ConsentClient
/ ConsentServer), **but** also collapse the dual vocabulary in the
Must Support section: drop "data source" / "data consumer" from the
Must Support expectations and re-express each Must Support obligation
in terms of ConsentClient and ConsentServer directly (e.g., "The
ConsentServer SHALL support …", "The ConsentClient SHALL be able to
…"). The Use Cases would still describe information flow in
plain-English terms ("the EHR provides the patient demographics") but
Must Support, Section 6.1, and the capability statements would all
use a single set of system names.

#### Justification

The reporter's underlying complaint is that the IG uses three
overlapping vocabularies — Client/Server, data source/data consumer,
and ConsentSystem — and a reader must reconstruct the mapping. The
literal resolution (Disposition A) keeps two of those three
vocabularies and just adds a mapping table; this alternative keeps
*one*. That is editorially simpler and harder to get out of sync
during future ballot cycles, and it matches how most FHIR IGs phrase
Must Support (anchored on a named actor / capability statement rather
than on an abstract role). The trade-off is more text churn than
Disposition A and a small loss of expressiveness if the work group
later wants to talk about a system playing a non-default role
(e.g., a ConsentClient acting as a data source for some flows); that
case can be handled by saying so explicitly in the affected
interaction.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" on the grounds that the existing
text — Client/Server in Section 6.1, data source/data consumer in
Must Support, and ConsentSystem in the use cases — is internally
consistent if read together, and that adding a new actor and renaming
the capability statements would constitute editorial churn for no
substantive normative gain.

#### Justification

This is the defensible "do nothing" path: no normative behaviour
changes, no canonical URL change for the capability statements, and
no downstream effect on existing implementer guidance. It should
only be taken if the work group concludes the reporter's confusion is
idiosyncratic and not representative of how implementers are reading
the IG. Given that the reporter is an experienced FHIR reviewer
(Craig Newman) and that the same pattern is exactly what triggered
companion editorial tickets against this spec (e.g., FHIR-54738 on
references to other guides), that conclusion is hard to defend; this
disposition is included for completeness rather than as a real
candidate.

---

### Recommendation

**Recommended disposition:** A

Disposition A is already the work group's recorded resolution on the
ticket and directly answers both questions the reporter raised: it
gives the use cases an explicit Client actor (resolving the Section 3.3
ambiguity about who "ConsentSystem" really is), and it renames the
capability statements to ConsentClient / ConsentServer so the
Section 6.1 vocabulary aligns with what the use cases now say.
Disposition B is editorially cleaner but is a larger change than the
reporter asked for and would re-open Must Support text that is
otherwise stable; it is a reasonable amendment if the work group wants
to take the opportunity to deduplicate vocabularies, but it should not
block adopting A. Disposition C is documented only for completeness —
the reporter's confusion is exactly the kind of cross-section
inconsistency a ballot reviewer is meant to surface, and the cost of
the Disposition A change (a new actor, a capability-statement rename,
and a small mapping subsection) is low compared to the implementer
clarity it buys.
