## Summary

The reporter, Abderrazek Boufahja, asks the Scalable Consent Management
IG authors to clarify what is meant by the term "systems" in the
conformance statements identified as §OP10 and §OP11 on the IG's
[Conformance Statements in This Guide](https://hl7.org/fhir/us/consent-management/2026Jan/conformance.html#conformance-statements-in-this-guide)
page. Specifically, it is unclear whether "systems" refers only to
server-side systems, only to client-side systems, or to both. Without an
explicit definition, implementers cannot reliably tell which side of an
exchange a given conformance statement applies to, which undermines the
testability of the guide's normative behavior. The ticket has already
been resolved as **Persuasive**, with the work group noting that two
distinct system actors — a **Consent Client** system actor and a
**Consent Server** system actor — should be defined and consistently
used throughout conformance statements and capability statements. There
are no cross-referenced Jira tickets, Zulip threads, or GitHub items in
the Augury index for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already recorded on the ticket: introduce two
explicit system actors in the Scalable Consent Management IG —
**Consent Client** and **Consent Server** — and rewrite every
conformance statement on the *Conformance Statements in This Guide*
page (including §OP10 and §OP11) so each statement names the actor it
applies to. Add a short "Actors" subsection (or extend the existing
glossary) on the conformance page that defines each actor and ties it
to a corresponding `CapabilityStatement` resource (e.g.,
`CapabilityStatement/consent-client` and
`CapabilityStatement/consent-server`). Update the IG's existing
CapabilityStatements to reflect the same actor split, so the
conformance page and the CapabilityStatements are mutually consistent.

#### Justification

This is the option already endorsed by the work group when the ticket
was resolved Persuasive, so it directly answers the reporter's
question. Splitting client and server conformance is consistent with
how other FHIR IGs (e.g., US Core, SMART App Launch, Bulk Data) make
their normative statements testable: each statement has an unambiguous
target. The change is editorial-plus-modeling rather than breaking,
since the underlying operations are unchanged — only the wording and
the structuring of CapabilityStatements are tightened. It also closes a
real interoperability risk: today, a vendor could legitimately argue
that §OP10/§OP11 do not apply to them because the actor is unspecified.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing two new IG-specific actor names, define the
audience for each conformance statement by **reusing existing FHIR
conformance vocabulary** — i.e., qualify each statement with
`Client` / `Server` (the same terms used in
`CapabilityStatement.rest.mode` and in the FHIR core conformance
chapter). The narrative on the *Conformance Statements in This Guide*
page would adopt a uniform pattern such as "Servers SHALL …" /
"Clients SHALL …" for each numbered statement, and a single
introductory paragraph would point readers at the existing FHIR
definitions of those roles. Capability statements would still be split
into client/server modes but would not be promoted to named "actors."

#### Justification

This avoids inventing new IG-local actor names that implementers must
learn and that may collide with actor terminology used elsewhere in
HL7 (IHE profiles, other privacy/consent IGs). Using the
already-normative `Client`/`Server` vocabulary keeps the IG aligned
with the rest of FHIR, lowers the documentation burden, and still
fully resolves the reporter's ambiguity. The trade-off vs. Disposition
A is that the IG loses the ability to attach actor-level requirements
that don't naturally fit the REST client/server split (e.g.,
back-channel processing systems), but the current §OP10/§OP11 wording
suggests that is not needed here.

---

### Disposition C: Decline

#### Proposal

Leave §OP10 and §OP11 unchanged and decline to introduce explicit
actor definitions, on the basis that "systems" in this IG should be
read as "any conformant system" — i.e., the statements apply to both
clients and servers wherever the behavior is observable.

#### Justification

This is defensible only if the work group judges that every numbered
conformance statement on the page is genuinely intended to apply to
*all* systems regardless of role, in which case adding an actor split
would over-constrain implementers and force artificial choices. It
would also avoid editorial churn in the 2026Jan publication. However,
this position is hard to sustain given that the ticket has already
been resolved Persuasive and that the reporter — an experienced IG
implementer — could not determine the intent from the text. Declining
also leaves the IG's CapabilityStatements without a clear normative
home for the requirements.

---

### Recommendation

**Recommended disposition:** A

The ticket is already resolved Persuasive and the work group has
committed to defining Consent Client and Consent Server actors, so
Disposition A is the path of least disruption and matches the
documented intent. Naming the actors explicitly (rather than relying
only on `Client`/`Server` from core FHIR, as in Disposition B) gives
the IG a stable hook for any future requirements that don't map cleanly
onto REST roles — for example, audit, provenance, or notification
behavior that may be added in later versions of the Scalable Consent
Management guide. The work group should, however, borrow Disposition
B's discipline when wording the statements: each rewritten
§OPnn should still use SHALL/SHOULD/MAY language that names the
actor, and each actor should be backed by a published
`CapabilityStatement` so the conformance page and the machine-readable
artifacts remain in lockstep.
