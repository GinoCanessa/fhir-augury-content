## Summary

The reporter flags a defect in the "Review Consent" use-case diagram in
the Scalable Consent Management (FHIR) specification: as drawn, the loop
in the diagram has no new input on each iteration and no exit condition,
so it would never terminate if read literally. The ticket asks for two
corrections to the diagram — (1) include the "sent" step inside the loop
so each iteration consumes/produces something, and (2) add an explicit
exit condition that fires when the review is complete. The change is
purely a documentation/diagram fix to the use-case description; it does
not alter any FHIR resource, profile, operation, or value set. The
ticket has already been triaged as **Persuasive** with a resolution
note that mirrors the request.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
found in the Augury index for FHIR-54789.

## Proposed Dispositions

> Note: this ticket has already been triaged with resolution
> **Persuasive** and a resolution statement aligned with Disposition A.
> The dispositions below are framed for workgroup confirmation /
> reconsideration on that record.

### Disposition A: Accept As Requested

#### Proposal

Update the "Review Consent" use-case diagram in the Scalable Consent
Management (FHIR) specification to:

1. Move the "sent" step (the action that transmits / publishes the
   reviewed consent) **inside** the loop body so each iteration has a
   meaningful input/output and the loop is not an infinite no-op.
2. Add an explicit **exit guard** on the loop that terminates iteration
   when the review is complete (e.g., a UML guard such as
   `[review complete]` on the loop's outgoing edge, or an
   `[else]`-style alt fragment that breaks out when no further consent
   items remain to review).

The change is confined to the use-case narrative page in the Scalable
Consent Management IG source — no resources, profiles, value sets,
operations, or examples are affected. Editor should also re-render the
diagram (PlantUML / source asset) and confirm the rendered SVG/PNG in
the published IG matches the corrected source.

#### Justification

The reporter's defect description is correct on its face: a loop with
no body change and no guard is a documentation bug regardless of FHIR
opinion, and a use-case diagram is meant to communicate intent to
implementers — leaving it ambiguous undermines that purpose. The
proposed fix preserves the original intent of "iterate over consent
items being reviewed" while making the iteration semantics actually
expressible. It is also the minimum-impact change: narrative-only,
no normative artifacts touched, no implementer rework required. This
matches the ticket's already-recorded resolution of **Persuasive**.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than patching the existing loop fragment in place, **replace the
loop with a clearer flow construct**: either

- an explicit UML `loop [for each consent under review]` fragment with
  the body containing both the review step and the "sent" step and a
  `break [review complete]` inner fragment, **or**
- decompose the diagram into two sequence fragments — a
  per-consent-item interaction (review → send) and an outer
  orchestration that enumerates the items — so the termination
  condition becomes a property of the enumerator rather than a guard
  glued onto a loop.

Optionally, add a short narrative paragraph above the diagram
describing the intended iteration semantics in prose, so the diagram
and text reinforce each other and a future reader can spot the same
class of defect more easily.

#### Justification

Disposition A fixes the immediate defect but leaves the diagram
structurally close to the broken version, which makes regressing it
easy the next time the page is edited. A clearer construct (named
`loop` fragment with an explicit `break`, or split diagrams) makes the
termination semantics syntactically obvious and matches common UML
sequence-diagram practice. The trade-off is more editorial work and a
larger visual delta in the published IG, which reviewers may push back
on if they are conservative about narrative changes between ballot
cycles.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the diagram is illustrative
rather than normative, and the surrounding prose is sufficient for a
reader to infer the intended iteration. Mark the ticket
**Not Persuasive** and close without altering the IG source.

#### Justification

Use-case diagrams in FHIR specifications are informative, and
implementers are not expected to derive conformance from them. If the
narrative on the page already conveys the loop's intent, the cost of
re-rendering the diagram and re-balloting the page may exceed the
benefit. This rationale is weak in the present case — the reporter
explicitly identifies the diagram as misleading, and there is no
counter-evidence on the ticket — but it is the only defensible
"do nothing" path and is included for completeness.

---

### Recommendation

**Recommended disposition:** A

The defect is real, the fix is small, and the workgroup has already
recorded **Persuasive** as the resolution with a statement that matches
Disposition A almost verbatim. There is no Zulip discussion, no linked
ticket, and no GitHub PR contradicting that direction, so there is no
evidence of community disagreement to weigh against it. Disposition B
is a reasonable editorial upgrade and could be folded into the same
edit if the editor has appetite for it, but it should not block
acceptance. Disposition C is not supportable given the reporter's
specific, accurate critique and the absence of any argument that the
current diagram is adequate.
