## Summary

The ticket — filed against the US PACIO Re-Assessment Timepoints (FHIR) IG and
routed to the (now deprecated) Community-Based Care and Privacy work group —
asserts that an IG-defined Value Set's canonical URL is anchored under the
`hl7.org/fhir` namespace when it should instead be anchored under
`terminology.hl7.org` (THO). The reporter (Joan Harper, with an imported
comment originally authored by Ron G. Parker) requests that the affected
Value Set be re-anchored in THO and that a corresponding UTG ticket be opened
to register the Value Set in THO. The disposition history shows the ticket was
initially resolved as "Considered for Future Use" (vote 9-0-0 on 2022-02-22)
deferring the decision until the IG and its terminology matured, and was then
re-opened by lynn_laakso post-STU1 publication (2022-09-28) so it can be
re-evaluated now that the IG has shipped. No formal cross-references to other
Jira tickets, Zulip threads, or GitHub PRs/issues/commits have been recorded
in FhirAugury for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the affected Value Set's canonical URL from the
`http://hl7.org/fhir/...` namespace into the
`http://terminology.hl7.org/ValueSet/...` namespace in the next published
version of the US PACIO Re-Assessment Timepoints IG, and open a corresponding
UTG ticket to register the Value Set (and any CodeSystem(s) it depends on, if
similarly mis-anchored) in THO. The IG's ValueSet resource(s) and any
StructureDefinition / profile binding that references the old canonical must
be updated in lock-step, with the prior canonical retained only as a
deprecated alias if back-compat for STU1 implementers is required. Note: the
specific ValueSet identity that this ticket targets is not named in the
ticket body; the workgroup will need to confirm which Value Set Comment 1
originally referred to before the UTG ticket can be filed.

#### Justification

Aligns with HL7's standing publishing convention that terminology artifacts
(CodeSystems and ValueSets) intended for cross-IG reuse, and especially those
used as the target of `required` bindings to `code`-typed elements, live
under `terminology.hl7.org` rather than under an IG-specific
`hl7.org/fhir/...` canonical. It is also consistent with how the same
boilerplate request is being raised across the related sibling tickets
(FHIR-35486 / -35529 / -35551 / -35578 / -35579 / -35614 and the
FHIR-38528–38531 series), so accepting here keeps the PACIO family on a
common terminology baseline. lynn_laakso's 2022-09-28 re-open note
explicitly acknowledged that this should be revisited post-STU1 once IG
experience exists.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the THO re-anchoring **only** to those Value Sets in the IG that are
the target of a `required` binding on a `code`-typed element (i.e., the
narrower scope hinted at in the ticket title: *"Only those with required
binding to a 'code' data type should be anchored in hl7.fhir.org"*). Value
Sets used solely for `extensible` / `preferred` / `example` bindings, or
bound to `Coding`/`CodeableConcept`-typed elements, remain in their current
IG-local canonical namespace. For each Value Set that does qualify, file a
single bundled UTG ticket covering all of the in-scope ValueSets (and their
CodeSystems) at once, rather than one UTG ticket per FHIR-Jira ticket.

#### Justification

The title text itself recognises that the THO-anchoring rule is only strictly
expected for required-bound `code` Value Sets, where THO hosting unlocks
cross-IG reuse and validator-friendly resolution. Value Sets used for weaker
bindings, or bindings to `Coding`/`CodeableConcept`, do not have the same
reuse pressure and the cost of churning their canonical URL (breaking
existing instances, profiles, and validator caches) may exceed the benefit.
Bundling the UTG submissions also reduces overhead for the UTG WG and keeps
the PACIO terminology migration coherent — important because the
related-by-keyword family above shows this same change request appears
across many sibling tickets that would otherwise generate many small UTG
filings.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** (or restore the prior **Considered for Future
Use** disposition) on the basis that (a) the IG has already shipped STU1
with the current canonical, (b) re-anchoring the Value Set is a breaking
change for any implementer who has stored or referenced the old canonical,
and (c) the ticket as written does not identify which specific Value Set is
being challenged, making the request not actionable in its current form.
Suggest the reporter re-file with the specific ValueSet canonical URL(s) in
question if they wish the WG to reconsider.

#### Justification

The 2022-02-22 vote (9-0-0) already decided that the right time to migrate
PACIO IG terminology into THO is when the terminology has matured enough
that the WG knows whether each ValueSet should be IG-local, absorbed
elsewhere, or hosted on THO. Acting on a single under-specified ticket
ahead of that maturity assessment risks an inconsistent half-migration
across the IG's Value Sets, and changing a published canonical URL is a
recognised hazard for downstream implementers (validators, profiles, and
recorded instances all carry the canonical). Declining now does not foreclose
the migration — it simply waits for the WG to do it as a coherent batch
once the in-scope ValueSets are known.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible starting point for the workgroup
discussion. It honours the spirit of the original request — and the title's
own hedge that THO-anchoring is required only for `required`-bound `code`
Value Sets — while avoiding the "re-anchor everything" cost that pure
Disposition A would impose on a freshly-published STU1 IG. It also addresses
the practical problem flagged in Disposition C (the ticket does not name the
specific Value Set) by reframing the work as a scoped audit of the IG's
required `code` bindings, which the WG can complete deterministically
without depending on the original reporter to clarify. Disposition A is a
reasonable fallback if the audit shows that essentially all of the IG's
Value Sets fall into the required-`code` bucket; Disposition C is appropriate
only if the WG concludes that the PACIO Re-Assessment Timepoints terminology
should remain IG-local indefinitely, which the 2022-09-28 re-open note
indicates is not the WG's current intent. Note that the work group of record
("Community-Based Care and Privacy") is now marked deprecated, so before any
disposition is enacted the ticket will likely need to be reassigned to the
PACIO-owning work group.
