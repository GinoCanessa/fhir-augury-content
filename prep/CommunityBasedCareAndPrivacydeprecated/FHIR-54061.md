## Summary

LukeBreyer requests clarification on the FASTConsent profile (part of the
Scalable Consent Management IG) regarding its mandatory `1..*` cardinality on
the R5-backported `Consent.grantee` extension. Base FHIR Consent supports a
range of consent paradigms — recipient-based, prospective, program-based,
implied, and policy-driven — but the IG does not describe how `grantee` should
be populated when authorization is policy-driven and recipients are not
enumerated at consent capture time. The reporter notes that, without guidance,
implementers will model abstract scenarios (opt-out HIE participation,
program-level enrollment, implied consent) inconsistently — naming a program
authority, a broad participant set, or a placeholder — which undermines
interoperability. The reporter asks the IG to either document an accepted
pattern for abstract grantees or relax the `1..*` requirement so non-recipient-
enumerated models are valid. The only cross-reference detected for this ticket
is the FHIR artifact `Consent.grantee` itself; no related Jira tickets, Zulip
threads, or GitHub items were surfaced by FhirAugury.

## Details

**Description:**
The FASTConsent profile requires use of the R5-backported Consent.grantee
extension with cardinality 1..*. The IG would benefit from additional
clarification regarding the intended role of this requirement across different
consent paradigms.

FHIR Consent supports multiple consent models, including recipient-based,
prospective, program-based, implied, and policy-driven consent. The
Consent.grantee element can represent abstract or programmatic entities (e.g.,
an exchange authority), which enables modeling of prospective authorization.
However, the IG does not explicitly describe how grantee is intended to be
represented when authorization is primarily policy-driven and recipients are
not enumerated at the time consent is recorded.

Without guidance, implementers may model similar scenarios differently (e.g.,
naming a program authority vs. broad participant sets vs. placeholders),
leading to inconsistent semantics and interoperability challenges. This is
particularly relevant in opt-out HIE participation, program-level exchange
enrollment, or implied consent models.

Clarifying the intended semantics of Consent.grantee in these scenarios via
narrative guidance and/or examples would help ensure consistent interpretation.
If the IG intends to support cases where recipients are implicit or
policy-defined, it may also be helpful to either document an accepted pattern
for abstract grantees or relax the 1..* requirement for the sake of
non-recipient-enumerated models.

**Resolution (per Jira snapshot — Resolved 2026-02-20, Persuasive):**

- Relax the cardinality on the `grantee` extension to `0..*`.
- Provide guidance on what the IG expects `grantee` to represent, especially
  for implicit consents.

**Comments:**

- **david.pyke (2026-02-18):** Recommend following what the base Consent
  resource does. Suggests filing a separate ticket against core Consent to
  handle the underlying use case.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already recorded on the Jira ticket against the
**FASTConsent** profile in the **Scalable Consent Management (FHIR) IG**:

1. **Relax cardinality** of the R5-backported `Consent.grantee` extension on
   the FASTConsent profile from `1..*` to `0..*`.
2. **Add narrative guidance** to the FASTConsent profile page (and/or a
   dedicated "Grantee semantics" section in the IG) covering:
   - When `grantee` should reference an explicit, enumerated recipient
     (recipient-based / prospective consent).
   - When `grantee` may reference a programmatic / authority entity (an
     exchange operator, an HIE program) for program-based or prospective
     authorization.
   - When `grantee` may legitimately be omitted (implied / policy-driven
     consent where recipients are determined at access time by policy rather
     than at capture time).
3. **Add at least two examples** to the IG: one program-authority grantee for
   prospective HIE enrollment, and one with no `grantee` for an implied
   opt-out scenario.

#### Justification

This is exactly what the resolution already negotiated between the reporter
and the WG specifies, and it is responsive to the original concern: relaxing
to `0..*` removes the artificial barrier for implied / policy-driven models,
while the accompanying guidance prevents the inconsistency the reporter
flagged (program authority vs. broad set vs. placeholder). The change is
backward-compatible for existing implementations that already populate
`grantee` — they continue to validate. david.pyke's review comment ("follow
what the base Consent resource does") is honored, since base R5
`Consent.grantee` is `0..*`; aligning the FASTConsent extension's cardinality
to `0..*` reduces the gap between the IG and the underlying resource.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Consent.grantee` at `1..*` on the FASTConsent profile, but introduce a
small set of **named conventions** for abstract grantees rather than
permitting the element to be absent. Concretely:

1. Define an IG-owned CodeSystem / ValueSet of "abstract grantee" identifiers
   — e.g., `program-authority`, `policy-defined-recipients`,
   `implied-opt-out`, `unknown-at-capture` — to be used as
   `Consent.grantee.identifier` (or as a coded `Consent.grantee.display`
   pattern) when no concrete party can be named.
2. Add narrative guidance and at least one example for each abstract value.
3. Add an invariant or recommendation that, when an abstract grantee is used,
   `Consent.policy`/`Consent.policyRule` MUST identify the policy that resolves
   the recipient set at access time.
4. File a companion ticket against base **Consent** (per david.pyke's comment)
   to evaluate first-class support for abstract / policy-resolved grantees in
   a future R6 cycle.

#### Justification

This preserves the original FAST design intent — that every consent
explicitly names *someone*, even if that someone is an authority — and
forces implementers off the inconsistent free-text patterns the reporter
called out. It also keeps a stricter interoperability floor than Disposition
A: every `Consent` instance still carries a recipient slot, so consumers
never have to special-case missing data. Trade-off: it is more invasive (new
CodeSystem, new invariant, more examples) and arguably steps beyond what
the reporter asked for; it also keeps the IG out of step with base R5
`Consent.grantee` cardinality, which the reviewer comment explicitly
discouraged.

---

### Disposition C: Decline

#### Proposal

Decline the request. Retain the FASTConsent profile's `1..*` cardinality on
the R5-backported `Consent.grantee` extension and add no new guidance.
Rationale: FASTConsent's scope is recipient-based / prospective consent, and
implied / policy-driven consent capture is out of scope for the profile.
Implementers needing those models should use base `Consent` directly rather
than this profile, and any clarification belongs against core Consent, not
the FASTConsent profile.

#### Justification

Defensible only on a strict-scope reading of FASTConsent: if the profile
genuinely intends to constrain itself to scenarios where a grantee is always
known at capture time, the existing `1..*` is a feature, not a bug. The
WG-internal review comment from david.pyke partially supports this framing
("put a ticket against core Consent"). The cost of declining, however, is
real: the reporter's interoperability concern is left unaddressed for any
implementer who reasonably reads FASTConsent as the FAST-aligned starting
point for *all* HIE consent flows, and the profile remains stricter than the
underlying R5 `Consent.grantee` (`0..*`) without a stated reason.

---

### Recommendation

**Recommended disposition:** A

The Jira resolution is already recorded as **Persuasive** with an explicit
two-part plan that matches Disposition A (relax to `0..*`, add guidance),
and the reviewer comment from david.pyke is consistent with that plan
(aligning to base Consent's cardinality and addressing the abstract-grantee
use case in narrative). Disposition A is the lowest-friction, backward-
compatible path that directly resolves the reporter's concern without
inventing new IG-specific machinery, and it leaves the door open for the
companion core-Consent ticket david.pyke suggested. Disposition B is worth
keeping in reserve only if the WG decides that "no grantee at all" is
unacceptable for FAST-aligned exchange — but absent that strong signal, it
adds complexity (new value set, new invariant) for marginal interoperability
gain. Disposition C should be rejected: FASTConsent is positioned as a
profile of `Consent`, and tightening cardinality below the base resource
without published guidance is precisely what produced this ticket.
