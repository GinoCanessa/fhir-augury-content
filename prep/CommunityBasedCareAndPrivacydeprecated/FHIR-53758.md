## Summary

Craig Newman asks why the `Subscription.status` element is constrained
to the fixed value `active` in the Scalable Consent Management (FHIR) IG
(the "fast subscription" profile of `Subscription` used to subscribe to
`Consent` resources). He argues that a subscriber should be able to move
the subscription out of `active` — for example, to `off` — when it
chooses to stop receiving notifications, which the current `active`-only
constraint prevents. The submitter's resolution note records agreement
("There is no reason why we fixed this to `active`. We will remove this
constraint."), so the substantive question for the workgroup is how to
relax the constraint, not whether to do so. There are no cross-referenced
Jira tickets, Zulip threads, or GitHub items associated with FHIR-53758
in the indexed sources.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the fixed/required-binding constraint on `Subscription.status` in
the Scalable Consent Management (FHIR) "fast subscription" profile so
that the element falls back to the base `Subscription` definition, where
`status` is bound (required) to the `SubscriptionStatus` value set
(`requested | active | error | off | entered-in-error`). Concretely,
edit the StructureDefinition for the constrained `Subscription` profile
in the Scalable Consent Management IG to drop the `fixedCode`/pattern on
`Subscription.status` (and any narrative or `mustSupport`/example
material that asserts the value is always `active`), and update the
profile's narrative / examples to show at least one non-`active`
transition (e.g., `off`) so implementers see the intended lifecycle.

#### Justification

This matches what the ticket asks for and what the submitter's
resolution note already commits to ("we will remove this constraint").
It restores the standard `Subscription` lifecycle so that a subscriber
can deactivate its own subscription (`off`), the server can mark a
broken endpoint (`error`), and tooling can record `entered-in-error` —
all of which are needed for a real, long-lived subscription against
`Consent`. Fixing `status` to `active` actively prevents the normal
"unsubscribe" path, which is a behaviour the base `Subscription`
resource is explicitly designed to support. The change is a profile
relaxation, so it does not break instances that currently carry
`status = active`.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of dropping the constraint outright, narrow it to the subset of
codes that make sense for a client-managed "fast subscription" against
`Consent`: constrain `Subscription.status` to a profile-specific value
set containing `active | off | error | entered-in-error` (i.e., remove
only `requested`, on the grounds that this profile is created in the
already-active state). Document the expected state transitions in the
profile narrative (active → off for a clean unsubscribe; active → error
for endpoint failure; active → entered-in-error for mistaken creation).

#### Justification

This still solves the submitter's concrete pain point — a subscriber can
move to `off` to unsubscribe — while preserving some of the original
intent of the constraint, namely that this profile is not used to model
a "pending"/`requested` workflow. It gives implementers a tighter
contract than full base behaviour and keeps the IG opinionated about
lifecycle, which is consistent with how other "fast" / scalable
profiles in the consent space tend to lock down optional flexibility.
Trade-off vs. Disposition A: it is slightly more spec work (a new value
set + binding) and any future addition to `SubscriptionStatus` would
have to be reconsidered for inclusion.

---

### Disposition C: Decline

#### Proposal

Keep `Subscription.status` fixed to `active` in the Scalable Consent
Management profile. Direct subscribers who want to stop receiving
notifications to delete the `Subscription` instance (HTTP `DELETE`)
rather than transitioning its `status`.

#### Justification

A defensible rationale would be that the "fast subscription" profile
is intentionally a thin, single-state shape — if the subscription is
present, it is active; if it is not wanted, the client removes it.
That keeps server-side state machines trivial and avoids implementers
having to reason about `off` vs. `error` vs. `entered-in-error` for
this constrained use case.

---

### Recommendation

**Recommended disposition:** A

The submitter's own resolution note already records the workgroup's
intent to remove the constraint, and the substantive argument in the
ticket is sound: a `Subscription` whose `status` cannot leave `active`
has no in-band way to be cleanly unsubscribed, deprecated, or marked as
broken, which forces every consumer to fall back to `DELETE`-only
lifecycle management and loses the audit value of the status
transition. Disposition A is the smallest, lowest-risk change (a
profile relaxation, backward compatible with existing `active`
instances) and it re-uses the well-understood base `SubscriptionStatus`
value set rather than inventing IG-local terminology. Disposition B is
a reasonable fallback if the workgroup wants to keep the profile
opinionated about not allowing `requested`, but that nuance can be
added later as a follow-up tightening if real-world use shows it
matters; it is not a reason to delay the basic fix.
