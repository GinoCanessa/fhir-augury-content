## Summary

The reporter is reviewing the FAST Scalable Consent Management IG and
notes that the `FASTConsent` profile restricts `Consent.identifier` to a
maximum cardinality of 1, allowing only a single identifier per consent
instance. They are asking the work group to justify this restriction.
The implicit ask is to relax the cardinality so multiple identifiers
(e.g., business identifier plus an external system identifier) can be
carried on a single consent. The ticket has been triaged with a
"Persuasive" resolution and a proposed resolution to change
`Consent.identifier` cardinality to `1..*`. No related Jira tickets,
Zulip threads, or GitHub items were surfaced by cross-reference; the
only cross-reference is to the FHIR element `Consent.identifier` itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already proposed on the ticket: in the FAST
Scalable Consent Management IG (`Scalable Consent Management (FHIR)`),
relax the `Consent.identifier` constraint on the `FASTConsent` profile
(`StructureDefinition-FASTConsent`) from `0..1` to `1..*`. Concretely:

- In the `FASTConsent` `StructureDefinition`, change the differential
  for `Consent.identifier` so `min = 1` and `max = "*"`.
- Update the profile narrative / page text on
  `StructureDefinition-FASTConsent.html` to explain that an instance
  MUST carry at least one identifier and MAY carry additional
  identifiers from other systems (e.g., source EHR identifier, registry
  identifier, downstream exchange identifier).
- If existing examples or supporting profiles assume `identifier 0..1`
  or `1..1`, update them to remain consistent with the new cardinality.

#### Justification

This matches the ticket's "Persuasive" resolution and aligns with the
base FHIR R4/R5 `Consent.identifier` cardinality of `0..*`. Restricting
to a single identifier on a profile is unusual for a resource-level
business identifier and conflicts with common FHIR practice, where
implementers routinely need to track an artifact across multiple
systems (originating EHR, consent registry, exchange intermediary). The
change is backward-compatible for any conformant instance that already
carries 0 or 1 identifiers — those instances remain valid under
`1..*` only if they carry at least one identifier, so the only behavior
change is that a previously-valid `identifier`-less instance becomes
invalid. That is a minor tightening that is appropriate for a
profile whose purpose is to make consents reliably identifiable for
exchange.

---

### Disposition B: Alternative Approach

#### Proposal

Relax the upper bound but leave the lower bound at `0`: change
`Consent.identifier` on `FASTConsent` from `0..1` to `0..*`, matching
the base resource exactly, and add slicing guidance (or an `identifier`
slice on a designated business-identifier `system`) in the profile
narrative so implementers know which identifier the IG considers the
primary FAST consent identifier. Optionally, define a `must-support`
slice for the canonical FAST identifier rather than a hard `min=1`.

#### Justification

`0..*` matches the base `Consent` resource and avoids introducing a new
mandatory element on a profile, which can break existing producers that
were conforming to the `0..1` profile or to base `Consent`. Slicing
gives the IG a way to call out the "primary" FAST identifier (with
`mustSupport = true`) without forcing every instance to populate it,
which is friendlier to bridges from systems that don't always have a
stable consent business identifier (e.g., transient or session-scoped
consents). The trade-off versus Disposition A is weaker
identifiability: instances may legitimately omit any identifier, which
complicates dedup and lifecycle tracking across exchanges.

---

### Disposition C: Decline

#### Proposal

Keep `Consent.identifier` at `0..1` on the `FASTConsent` profile and
close the ticket with a "Not Persuasive" rationale, on the basis that a
single canonical identifier per consent simplifies reconciliation
across the FAST exchange ecosystem and that secondary identifiers can
be carried via an extension (or via `Consent.sourceReference` /
`Consent.sourceAttachment`) rather than by relaxing the cardinality.

#### Justification

A single profile-mandated identifier reduces ambiguity about which
identifier is "the" consent identifier in a federated exchange, which
is one of the goals of the FAST consent work. If the only motivating
use case is carrying alternate-system identifiers, that can be modeled
as an extension on `identifier` or as part of the source reference
chain, without changing the profile contract. Declining is defensible
when the restriction was a deliberate exchange simplification.

---

### Recommendation

**Recommended disposition:** A

The work group has already marked the ticket "Persuasive" and proposed
the exact change in Disposition A, and that change reflects standard
FHIR practice: business-identifier elements on resource-level profiles
are very rarely capped at 1, because real-world instances accumulate
identifiers from multiple systems (originating EHR, registry,
intermediary). Going to `1..*` (rather than the base `0..*` of
Disposition B) preserves the IG's clear intent that every FAST consent
instance be identifiable for exchange, while removing an artificial
cap that has no documented rationale on the ticket. Disposition B is a
reasonable fallback if the work group hears, on review, that some
producers genuinely cannot supply an identifier; Disposition C is hard
to defend without a written rationale for the original `0..1` cap,
which the ticket explicitly asks for and which has not surfaced in any
linked Jira, Zulip, or GitHub discussion.
