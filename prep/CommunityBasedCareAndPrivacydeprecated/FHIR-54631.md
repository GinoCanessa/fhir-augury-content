## Summary

The reporter, John Moehrke, observes that the `Permission` resource has
introduced a `rule.limit` concept that lets a permit carry obligation/refrain
codes (for example, "do not print" or "must discard after 48 hours") that
must be honored by any actor consuming data exposed by that rule. `Consent`
currently lacks an equivalent way to attach such residual handling
constraints to a `Consent.provision`. The ticket asks the work group to
choose between (a) adding a `limit`-style element directly to
`Consent.provision` to align it with `Permission.rule.limit`, or (b)
defining an extension that achieves the same effect without modifying the
core `Consent` structure. The reporter notes that approach (b) carries the
practical advantage of being usable in R4 as well as the current build, and
in his only comment on the ticket links to a demonstration extension he has
already authored in his `ConsentAboutAI` IG
(`StructureDefinition-permissionRuleLimit`). No related Jira tickets were
surfaced by the cross-reference index, so the request stands on its own,
but it is closely tied to the existing `Permission.rule.limit` design.

## Details

**Description:**

> Permission has introduced the concept of a rule.limit; these are limits to
> be placed on a Permit that is used. Thus one can have a rule that permit
> some activity, and the limit is applied to that activity. (e.g. obligation
> code to not print is applied to the use of that data exposed by that rule)
>
> a) Consent could add this to .provision
>
> b) Consent could define an extension that could be used on .provision
>
> The (b) solution has the added benefit that the extension can be used in
> R4.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt option (a) from the ticket: add a `limit` element directly to
`Consent.provision`, mirroring the structure and semantics of
`Permission.rule.limit`. Concretely, introduce
`Consent.provision.limit` as `0..*` of the same datatype used by
`Permission.rule.limit` (a `BackboneElement` carrying `limit.element`,
`limit.tag`, and any obligation/refrain coding), and bind its coded
fields to the same value sets used by `Permission.rule.limit` (e.g.,
`v3-ObligationPolicy`, `v3-RefrainPolicy`, security labels). Update the
`Consent` narrative and the cross-references between `Consent` and
`Permission` to describe the parallel structure. Because `Consent`
already nests `provision` recursively, place `limit` on the
`provision` backbone so that it can be attached to the top-level
`provision` and to nested provisions alike.

#### Justification

This is the most direct alignment with the `Permission` resource and
keeps the two privacy-policy resources structurally and semantically
parallel, which is consistent with how the work group has been
positioning `Consent` and `Permission` as complementary expressions of
the same underlying access-policy model. It avoids forcing every
implementer that needs limits on a consent provision to discover and
adopt an HL7-published extension, and the Zulip background on
`Permission.rule.limit` shows the work group has already accepted that
residual rules (e.g., do-not-print, time-bound retention) belong on the
permit/consent side rather than purely in security tags. As a core
change it is a R6-only addition (additive `0..*`, so non-breaking),
which the work group has historically tolerated for `Consent`.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt option (b) from the ticket: do **not** modify `Consent` core in
R6. Instead, publish a standard HL7 extension —
`http://hl7.org/fhir/StructureDefinition/consent-provision-limit` —
whose definition mirrors `Permission.rule.limit` exactly (same
backbone, same element/tag/coding sub-structure, same value set
bindings). Anchor the extension to `Consent.provision` with `0..*`
cardinality, and reference it from the `Consent` resource's
"Extensions" section as the recommended way to attach residual
obligation/refrain rules to a consent provision. Use the reporter's
`StructureDefinition-permissionRuleLimit` from the `ConsentAboutAI` IG
as the starting point for the extension definition.

#### Justification

This was the reporter's preferred option in the original write-up
("the (b) solution has the added benefit that the extension can be
used in R4"), and the reporter has already produced a working model in
his `ConsentAboutAI` build. An extension is non-breaking by
construction, ships independently of the FHIR ballot cycle, and
critically lets implementers on R4 and R5 use exactly the same mechanic
without waiting for R6 — which matters for jurisdictions whose
real-world consent infrastructure is locked to earlier FHIR versions.
If experience shows the extension is widely used, promotion into core
on a later release is straightforward and the extension URL can be
retired with a published mapping.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped, on the grounds that the same
expressive power is already available today by attaching the existing
obligation/refrain security-label codes to `Consent.provision.code`
(category) and to the `Consent.provision.securityLabel` element, and
by referencing a `Permission` instance from the consent flow when
richer residual rules are required. Document this pattern in the
`Consent` resource's narrative and in the security-labels guidance
rather than introducing a parallel `limit` mechanism on `Consent`.

#### Justification

`Consent.provision` already exposes `securityLabel`, `purpose`,
`actor`, `action`, `class`, and `code`, several of which are bound to
exactly the same vocabularies (`v3-ObligationPolicy`,
`v3-RefrainPolicy`, security labels) that `Permission.rule.limit`
draws on. The ticket and the Zulip threads acknowledge that residual
rules of the kind motivating this request ("do not print", "discard
after 48 hours") are unusual; introducing a second, parallel mechanism
on `Consent` risks two equally valid ways to express the same
constraint and a corresponding interoperability split. Where richer
limit semantics are genuinely needed, the work group's stated
direction is to use `Permission` for the detailed rules and have
`Consent` carry the patient-authorization layer — which is a documented
pattern, not a gap.

---

### Recommendation

**Recommended disposition:** B

Option (b) gets the work group everything it needs from option (a)
without taking on a core spec change at this point in the cycle. The
reporter has already published a working extension definition in his
`ConsentAboutAI` IG, so the cost of standardizing it as an HL7
extension is essentially editorial review plus value-set alignment with
`Permission.rule.limit`. The R4 forward-compatibility argument is real:
a large fraction of deployed `Consent` infrastructure is on R4 today
and would benefit immediately. Disposition C understates how awkward
it is to express the full `limit.element` + `limit.tag` shape using
only `securityLabel` and `code`, and Disposition A locks the change to
R6 only, which the reporter himself flagged as a drawback. Promotion
of the extension into core in a later release remains an open path if
adoption justifies it, so option B is the lower-risk, faster-to-deploy
choice and the recommended starting point for the work-group
discussion.
