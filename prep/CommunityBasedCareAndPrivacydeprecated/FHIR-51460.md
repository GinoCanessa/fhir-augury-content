## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **US PACIO Re-Assessment Timepoints (FHIR)** IG to declare dependencies on US Core 6.1.0, 7.0.0, and 8.0.0 in addition to (or in place of) US Core 3.1.1. Concretely:

- Bump the `dependsOn` entries in the IG's `ImplementationGuide` resource (and the corresponding `sushi-config.yaml` / `ig.ini` / NPM package manifest) to add `hl7.fhir.us.core` at versions `6.1.0`, `7.0.0`, and `8.0.0`.
- For every profile in the IG that derives from a US Core profile (Patient, Practitioner, and any clinical observations used for re-assessment timepoints), add parallel profile variants or rebase the existing profiles onto the newest supported US Core version, retaining a 3.1.1 variant only if backward compatibility is required.
- Re-run the IG publisher, fix any QA errors arising from US Core element-tightening or terminology changes (e.g., USCDI v3/v4-aligned must-support flags, race/ethnicity extensions, address line constraints), and refresh examples to validate against the new profiles.
- Update the IG's narrative pages ("Dependencies", "Profiles") to document which US Core version each profile targets and how implementers should choose.

#### Justification

The reporter's underlying concern is real: US Core 3.1.1 corresponds to USCDI v1 and is several years out of date. CMS, ONC, and most US realm IGs now reference US Core 6.1.0+ (USCDI v3) and 7.0.0/8.0.0 (USCDI v4/v5). A PACIO IG that only depends on 3.1.1 will increasingly be incompatible with implementer stacks built against newer USCDI baselines, which directly defeats the IG's interoperability purpose. Multi-version support is the established pattern for US Core consumers (e.g., Da Vinci, CARIN) precisely because realm guides cannot force their implementers onto a single US Core line.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than maintaining parallel profile variants for four US Core versions, **rebase the IG onto a single newer US Core line — US Core 6.1.0 as the floor — and retire the 3.1.1 dependency**. Specifically:

- Replace the `dependsOn hl7.fhir.us.core 3.1.1` with `dependsOn hl7.fhir.us.core 6.1.0`.
- Document compatibility with 7.0.0 and 8.0.0 via a "Supported US Core Versions" page that calls out which differential elements are stable across 6.1.0 → 8.0.0, rather than shipping three sets of profiles.
- For implementers still on US Core 3.1.1, publish a short migration note pointing at US Core's own version-bridge guidance instead of carrying the legacy profiles inside this IG.
- Cut a major version of the Re-Assessment Timepoints IG to signal the breaking dependency change.

#### Justification

Maintaining four parallel profile sets multiplies authoring, QA, and example-management cost for a small workgroup, and the resulting IG becomes harder for implementers to read because every page must qualify which US Core flavor applies. A single floor of US Core 6.1.0 captures the USCDI v3 baseline that most current US implementers have adopted, while compatibility with 7.0.0 and 8.0.0 can usually be asserted (and tested) without literally re-publishing the profiles, because US Core's policy is to keep changes between minor lines additive where possible. This also avoids the risk of subtle divergence between the 3.1.1 and 8.0.0 variants of "the same" PACIO profile.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the **Community-Based Care and Privacy** work group is marked **[deprecated]** and the US PACIO Re-Assessment Timepoints IG is no longer being actively maintained at this work group. Refer the request to whichever work group has inherited PACIO stewardship (Patient Care or the active PACIO project leadership), or close the ticket with a note that the IG is in maintenance-only status and will not take a US Core dependency bump.

#### Justification

The work group label literally reads "Community-Based Care and Privacy [deprecated]", which strongly suggests that neither this work group nor this ticket queue is the right venue to land a substantive IG change. Rather than letting the ticket sit in Triaged indefinitely, declining (with a redirect) is more honest than promising work the deprecated WG will not deliver. If the IG itself is also retired or superseded, a US Core dependency bump would be effort spent on a guide implementers should be migrating away from anyway.

---

### Recommendation

**Recommended disposition:** B

The change the reporter is asking for is legitimate and overdue — US Core 3.1.1 is no longer a credible floor for a US realm IG — so outright decline (C) only makes sense if the IG is being retired, which the ticket does not establish. Between A and B, B is the better starting point: a single US Core 6.1.0 floor with documented forward compatibility delivers the implementer value the reporter wants without committing the (deprecated) work group to maintaining four parallel profile sets it has no resources to keep in sync. If the workgroup discussion surfaces a concrete population still pinned to US Core 3.1.1, A can be revisited as a transitional measure, but the default direction should be "move forward to 6.1.0+ and document compatibility upward". The disposition discussion should also explicitly resolve the deprecated-work-group question raised in C — i.e., confirm who owns the IG before any version-bump work is scheduled.
