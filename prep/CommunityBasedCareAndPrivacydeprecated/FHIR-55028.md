## Summary

The reporter (David Hill) asks whether the Scalable Consent Management
specification should require a confirmation back to the consenter when a
consent is **delegated** or **revoked**, in the same way a confirmation is
already provided after consent **activation**. The ticket is effectively a
consistency / completeness request: today the activation flow tells the
consenter that their action succeeded, but the corresponding delegate and
revoke flows do not appear to provide an equivalent acknowledgement,
which leaves the consenter without assurance that their instruction was
acted upon. The ticket has already been resolved as "Persuasive" with the
note that "We will add the confirmation step in all use cases where
needed," so the workgroup's working position is that the gap is real and
should be closed across the affected use cases. There are no
cross-referenced Jira tickets, Zulip threads, or GitHub items recorded
against this ticket, so the only available context is the ticket itself.

## Proposed Dispositions

The Scalable Consent Management specification is not represented by a
GitHub repository in the available cross-references, so the dispositions
below describe the change at the use-case / specification-text level
rather than at a specific authoring path. If the workgroup chooses to
act, the change will land in the Scalable Consent Management IG
authoring source (likely the use-case narrative pages plus any
sequence diagrams) — the exact path should be confirmed against that
IG's repo before editing.

### Disposition A: Accept As Requested

#### Proposal

Update the Scalable Consent Management specification so that **every**
consenter-initiated use case — at minimum *delegate consent* and
*revoke consent*, alongside the existing *activate consent* flow —
includes an explicit "confirmation to consenter" step. Concretely:

- Add a normative (or conformance-level appropriate) statement to the
  delegate and revoke use-case sections that the system MUST/SHOULD
  return a confirmation to the consenter when the action completes
  successfully (and a corresponding failure notification on error).
- Update the sequence diagrams / interaction tables for those use cases
  to show the confirmation message as a final step, mirroring the
  activation diagram.
- Where the confirmation is delivered as a FHIR artifact, recommend
  using `Communication` (with a `basedOn`/`partOf` link to the
  triggering `Consent`) for the notification and `Provenance` /
  `AuditEvent` for the system-of-record entry, consistent with how the
  activation confirmation is handled today.

#### Justification

This is exactly what the ticket asks for and matches the resolution
already recorded ("We will add the confirmation step in all use cases
where needed"). It is a consistency fix: the activation flow already
sets the precedent, so extending the same pattern to delegate and
revoke removes a surprising asymmetry, gives the consenter assurance
their instruction was honoured, and aligns with general privacy
principles (transparency to the data subject about actions taken on
their consent). It is also a low-risk change because it adds an
expected step rather than changing an existing one.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a per-use-case confirmation step, define a **single
cross-cutting "consenter notification" requirement** in a shared
section of the IG and reference it from each use case. The shared
section would:

- State that any state-changing consent action initiated by, or on
  behalf of, the consenter (activate, modify, delegate, revoke,
  re-affirm, expire-with-notice, …) MUST result in a confirmation
  delivered to the consenter through their preferred channel.
- Specify the artifact pattern once (e.g., `Communication` referencing
  the affected `Consent` plus a `Provenance` entry) and the minimum
  data set the confirmation must carry (action taken, timestamp,
  affected scope, who/what triggered it, how to dispute).
- Have each use-case section simply cite the shared requirement and
  show it on the diagram, instead of restating it.

#### Justification

This addresses the same underlying need — the consenter should always
know what happened to their consent — but does so in a way that
generalises beyond the three flows currently called out and avoids
divergence between use cases as the IG grows. It is more
maintenance-friendly: future use cases (e.g., automated expiry,
policy-driven revocation, delegate-of-delegate) inherit the
requirement automatically, and implementers have one normative
statement to conform to instead of N near-identical ones. The
trade-off is that the change is slightly larger in scope than what
the ticket literally asked for, and it requires a new shared section
that does not exist today.

---

### Disposition C: Decline

#### Proposal

Decline to add explicit confirmation steps to the delegate and revoke
use cases on the basis that consenter notification is an
**implementation / deployment concern** rather than a
specification-level interoperability requirement, and that the
existing `Consent` resource state (plus standard `AuditEvent` /
`Provenance` records) already gives the consenter — or any system
acting on their behalf — the ability to verify the outcome by
reading back the resource.

#### Justification

The IG can reasonably take the position that *what* happens (the
consent state changes) is in scope, while *how the consenter is told
it happened* is a UX/notification concern that varies by deployment
(portal banner, email, SMS, push, in-clinic acknowledgement, etc.).
Mandating a confirmation step risks either being too vague to be
testable or too prescriptive about a delivery channel the IG cannot
realistically constrain. This option preserves the current scope and
avoids spec-level requirements that are hard to certify against.

---

### Recommendation

**Recommended disposition:** B

The ticket has already been resolved as Persuasive, so Disposition C
(Decline) is inconsistent with the workgroup's recorded position and
should not be the starting point. Between A and B, Disposition B
delivers exactly what the reporter asked for *and* prevents the same
gap from re-appearing in future consenter-initiated flows — which is
likely why the resolution text deliberately says "in all use cases
where needed" rather than naming only delegate and revoke. Pulling the
confirmation requirement into a single shared section, with a stated
artifact pattern (`Communication` + `Provenance`/`AuditEvent`), keeps
the IG internally consistent, gives implementers one conformance
target, and only marginally increases editorial effort over Option A.
If the workgroup wants the smallest possible change for this ballot
cycle, Option A is an acceptable fallback and can be refactored into
the shared pattern later without breaking conformance.
