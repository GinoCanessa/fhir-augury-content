## Summary

The ticket asks the US Military Service (FHIR) implementation guide to
re-root the canonical URL of the `military-service-military-occupation-vs`
ValueSet under `terminology.hl7.org` (THO) rather than under
`hl7.org/fhir`. The reporter (Ron G. Parker) notes that the ValueSet
should also be registered in THO via a companion UTG ticket. The request
reflects the broader HL7-wide convention that vocabulary artifacts (code
systems and value sets) live in THO, while resource-bearing
specifications live under `hl7.org/fhir`. The cross-reference search
surfaced **no related Jira tickets, Zulip threads, GitHub items, or
Confluence pages** in the FhirAugury index, so this prep is built from
the ticket and its single comment alone.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-canonicalize the `military-service-military-occupation-vs` ValueSet so
that its `ValueSet.url` becomes
`http://terminology.hl7.org/ValueSet/military-service-military-occupation-vs`
(or an equivalent THO-rooted slug agreed with the Vocabulary WG), and
file a paired UTG ticket to register the ValueSet (and any underlying
CodeSystem(s) it composes) in THO. Update every reference to the old
`http://hl7.org/fhir/...` canonical inside the US Military Service IG
(profiles, extensions, narrative, examples) to point at the new THO URL,
and add a `ConceptMap`/aliasing note (or retain a redirect) so existing
implementations are not silently broken.

#### Justification

This is the canonical pattern HL7 has converged on: vocabulary content
lives in THO, resource/profile content lives in `hl7.org/fhir/<ig>`. The
reporter explicitly calls out that this is the intended end-state. The
ticket's own resolution text concedes the direction is normative in
FHIR R5 — adopting it now in the US Military Service IG aligns the IG
with that direction proactively rather than carrying an inconsistency
forward.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing `hl7.org/fhir`-rooted canonical for the IG's *current*
publication to avoid a breaking change, but (a) author and publish a
THO-rooted *companion* ValueSet via the UTG process, (b) mark the
IG-rooted ValueSet as `status = retired` (or `deprecated`) once the THO
copy is live, and (c) point all profiles/bindings in the next IG release
at the THO canonical. Document the migration in the IG's change log and
in a small "URL migration" page.

#### Justification

A straight rename of an already-published canonical URL invalidates any
downstream binding, instance, or terminology-server cache that uses the
old URL. The two-step deprecate-then-redirect pattern delivers the same
end-state requested by the ticket while giving implementers a clean
migration window. It also respects the resolution-text observation that
the THO-rooting rule is *not* expressly stated in R4 — a soft migration
is defensible under R4 in a way that a silent breaking rename is not.

---

### Disposition C: Decline

#### Proposal

Mark the request **Considered for Future Use / Non-Persuasive** (the
disposition already recorded on the ticket): leave the existing
`hl7.org/fhir`-rooted canonical in place for the current US Military
Service IG publication, and defer any THO-rooting of this ValueSet to
the IG's next major revision when it adopts FHIR R5 conventions.

#### Justification

The IG was balloted and published with the existing canonical URL.
Per the ticket's resolution text, the THO-rooting requirement is not
expressly stated in FHIR R4, so the current URL is not non-conformant.
The ticket already carries a "Non-persuasive" comment from the WG
reviewer (seanmuir) and a resolution of "Considered for Future Use",
which is consistent with declining the immediate change while leaving
the door open to a future, coordinated migration.

---

### Recommendation

**Recommended disposition:** B

Disposition C matches what the WG has already recorded on the ticket and
is the lowest-effort path, but it leaves the IG knowingly out of step
with HL7's accepted vocabulary-governance pattern and pushes the same
problem onto a future ballot cycle. Disposition A is the right
*end-state* but a hard rename of a published canonical URL silently
breaks every downstream binding and cached terminology resolution that
already references the `hl7.org/fhir`-rooted URL — that is a
disproportionate cost for an IG with active implementers. Disposition B
delivers the THO-rooted artifact the reporter is asking for, opens the
companion UTG ticket the description explicitly calls for, and gives
implementers a documented deprecation window before the IG's
profiles/bindings are flipped to the new canonical. It is the option
that respects both the governance direction and the existing
implementer base.
