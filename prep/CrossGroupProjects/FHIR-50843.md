## Summary

The reporter (Kenneth Lord) raises a semantic round-trip concern with the
medication mapping guidance in the *US C-CDA on FHIR* IG. The IG documents
bi-directional mappings between CDA `substanceAdministration` and the FHIR
medication resources (`MedicationAdministration`, `MedicationRequest`, and —
implicitly — `MedicationStatement`/related), but the mappings as written do
not preserve resource identity across a round trip:

- **Use Case 1 (FHIR → CDA → FHIR):** Starting from a FHIR
  `MedicationAdministration`, the natural target in CDA is
  `substanceAdministration` with `moodCode = EVN`. However, the IG's
  reverse mapping says CDA `substanceAdministration[moodCode=EVN]` maps
  back to a FHIR `MedicationRequest` with `intent = planned`. The
  starting and ending FHIR resources differ, so the round trip is not
  semantically lossless.
- **Use Case 2 (CDA → FHIR for an event that actually occurred):** It is
  unclear from the current guidance how a CDA `substanceAdministration`
  representing an administration that was actually given should populate
  a FHIR resource that records administration (i.e.,
  `MedicationAdministration`) rather than a request/plan resource.

The reporter is asking the workgroup to clarify and correct the
mapping table so that `moodCode` (and any companion attributes such as
`statusCode`/`negationInd`) drive the choice of FHIR resource in a way
that is consistent with the FHIR resource semantics (`event` →
administration/statement, `request`/`intent` → request).

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the augury index for this ticket, so the request is taken
at face value from the description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revise the Medication Mappings table in the *US C-CDA on FHIR* IG so
that `substanceAdministration.moodCode` drives a FHIR resource choice
that preserves act semantics on a round trip:

- `substanceAdministration[moodCode = EVN]` → FHIR
  `MedicationAdministration` (administration that occurred) or, when
  the act is reported by the patient/another source rather than
  performed, `MedicationStatement`. The reverse mapping (FHIR →
  C-CDA) should map `MedicationAdministration` and `MedicationStatement`
  back to `substanceAdministration[moodCode = EVN]`.
- `substanceAdministration[moodCode = INT | RQO | ARQ | PRMS]` → FHIR
  `MedicationRequest` with `intent` set per the table
  (`proposal | plan | order | …`), and the inverse mapping from
  `MedicationRequest` back to the corresponding `moodCode`.
- Explicitly call out that the prior text mapping `moodCode = EVN` →
  `MedicationRequest.intent = planned` was incorrect: `EVN` denotes
  an event (something that happened), not a plan, and `intent =
  planned` belongs to a `RQO`/`INT`-flavoured act.

The change is editorial-plus-table within the IG: update the mapping
narrative, the bi-directional mapping table(s), and any companion
examples that exercise the round trip. Provide a worked example for
each of the two use cases the reporter raised.

#### Justification

This aligns the mapping with the established FHIR "event vs. request"
resource pattern (`MedicationAdministration`/`MedicationStatement` are
event resources; `MedicationRequest` is a request resource with an
`intent`) and with the V3 RIM semantics of `moodCode` (`EVN` is an
event mood, `INT`/`RQO` are intent/request moods). The reporter's
round-trip concern is a direct consequence of the current text
violating that alignment, so accepting and correcting the table
restores semantic equivalence across the round trip and removes the
ambiguity the reporter highlighted in Use Case 2.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only correcting the table cells, restructure the
Medication Mappings page so that **`moodCode` is the primary axis**
and each mood maps to a small, ranked set of FHIR resource choices
qualified by additional CDA attributes (e.g., `statusCode`,
`negationInd`, presence of `author`/`performer`,
`entryRelationship` typeCode). Concretely:

1. Add a decision-tree (or table keyed first on `moodCode`, then on
   secondary attributes) that yields exactly one FHIR target resource
   per CDA permutation, and vice versa.
2. Document the reverse direction symmetrically, so for each FHIR
   medication resource the IG names the canonical CDA `moodCode` plus
   the supporting attributes that must be set for round-trip fidelity.
3. Add a normative "Round-Trip Conformance" subsection that states
   which round trips MUST be lossless, which MAY lose annotation-only
   data, and which are out of scope (e.g., when a CDA act has no FHIR
   equivalent and falls back to `Basic`/extension).
4. Provide at least two end-to-end examples (one starting from FHIR,
   one starting from CDA) covering the use cases the reporter named.

#### Justification

The reporter's confusion is rooted not in a single wrong table cell
but in the table being organised in a way that makes it easy to land
on an incorrect target when only `moodCode` is considered. Restructuring
the guidance around `moodCode` plus secondary discriminators eliminates
the class of error rather than just the instance, and gives implementers
an unambiguous algorithm. It is more work than Disposition A but is
likely to prevent further round-trip tickets in the same area.

---

### Disposition C: Decline

#### Proposal

Decline as written, on the grounds that the IG's medication mapping
intentionally trades strict round-trip identity for accurate
representation in each model: a CDA `substanceAdministration` carries
information beyond what a single FHIR resource captures, and the
"correct" FHIR target depends on context that is not always present
in the CDA act itself. Direct the reporter to the existing C-CDA on
FHIR mapping narrative and to use site-level transformation profiles
where strict round-trip fidelity is required.

#### Justification

Defensible only if the workgroup's position is that the mapping is
informative guidance, not a normative bidirectional transform, and
that consumers are expected to apply local rules to choose between
`MedicationAdministration`, `MedicationStatement`, and
`MedicationRequest`. Even then, the *specific* mapping cell the
reporter cites (`moodCode = EVN` → `MedicationRequest.intent =
planned`) appears to be a semantic error rather than a defensible
trade-off, which weakens this disposition substantially.

---

### Recommendation

**Recommended disposition:** B

The reporter has identified a real semantic defect (Use Case 1) and a
real gap (Use Case 2), and Disposition A would fix both at the level
of the offending table cells. However, the underlying problem — that
`moodCode` alone does not deterministically select a FHIR medication
resource and that the table is not organised to make that explicit —
is what produced the reporter's confusion in the first place and is
likely to produce more tickets in this area until it is addressed.
Disposition B subsumes Disposition A (the mood-keyed structure forces
the `EVN → MedicationAdministration/MedicationStatement` correction)
and additionally gives implementers an unambiguous algorithm and a
stated round-trip conformance contract. Disposition C is not
recommended because the specific mapping the reporter calls out is
materially incorrect, not merely a contextual judgement call.
