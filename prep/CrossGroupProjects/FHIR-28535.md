## Summary

The reporter (Bryn Rhodes) observes that the introduction to the *US
Vital Signs with Qualifying Extensions (FHIR)* IG discusses the
background and motivation for the qualifying‑extensions work but never
states clearly that the IG's profiles **derive from and build upon both
the base FHIR Vital Signs profiles and the US Core profiles**. He asks
that the introduction be updated to make this dependency explicit —
naming US Core as a parent IG, identifying the specific version of US
Core that is being built upon, and providing a link to that IG. He also
suggests (optionally) a dependency diagram to make the relationship
between the IGs visually obvious. No cross‑referenced Jira tickets,
Zulip discussions, or GitHub items were surfaced for this ticket, so
this is the full extent of the request.

## Details

**Description:**

The introduction discusses the background for development of the
profiles, but it doesn't make clear that the profiles derive from and
build upon the base FHIR profiles, and the current US Core profiles. A
diagram might help, but at the very least a clear statement about
dependency on US Core, which version, and a link to that IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the IG's introduction page (the landing/index narrative of the
*US Vital Signs with Qualifying Extensions* IG) to add an explicit
"Relationship to other IGs" subsection that:

1. States that the qualifying‑extensions profiles are **derived from
   the base FHIR Vital Signs profiles** (cite the base FHIR Vitals
   profile family) and **further constrain / extend the US Core Vital
   Signs profiles** of US Core.
2. Pins the **specific US Core version** the IG builds on (i.e., the
   version listed in the IG's `dependsOn` package list in
   `ImplementationGuide`/`sushi-config.yaml`), and links to that
   version of US Core (`hl7.fhir.us.core` published page).
3. Optionally embeds a small dependency diagram (PlantUML or static
   image) showing: base FHIR Vital Signs → US Core Vital Signs → US
   Vital Signs with Qualifying Extensions.

The change is purely narrative; no profile, extension, value set, or
conformance artifact is altered. The dependency itself already exists
in the IG's package manifest — this just makes it visible to readers.

#### Justification

The request is uncontroversial and aligns with general HL7 IG
authoring guidance: every IG that depends on another IG should make
that dependency obvious in its introduction so implementers know which
"stack" of profiles applies. Pinning the specific US Core version is
particularly important for vital signs because US Core has revised the
Vital Signs profiles across versions, and conformance changes
materially with the version pinned. The cost (a paragraph plus
optional diagram) is low and produces a real reader benefit.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding prose to the introduction, take a slightly
broader documentation pass:

1. Add the explicit dependency statement and US Core version link in
   the introduction (as in Disposition A).
2. **Additionally**, on each profile page that derives from a US Core
   profile, surface a "Derived from US Core" callout linking to the
   parent US Core profile at the pinned version.
3. Add a single canonical "IG Dependencies" page (linked from the
   introduction) that lists every upstream IG (base FHIR, US Core, any
   terminology IGs) with version, canonical URL, and a one-line
   description of what is reused.

This treats the ticket's underlying need — implementers being able to
trace the IG's lineage — as a navigation problem solved at multiple
entry points, not just the introduction.

#### Justification

Readers do not always start at the introduction; many land directly on
a profile page from a search engine or a tooling link. Putting the
dependency story only in the introduction leaves those readers with no
context. A per-profile "derived from" callout plus a dedicated
dependencies page costs a bit more authoring effort but produces
durable benefits across the whole IG and is consistent with how other
mature US-realm IGs (e.g., US Core's own derivations of base FHIR)
present lineage. The trade-off versus Disposition A is scope: this
touches more pages but gives the introduction change much more
leverage.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the basis that the dependency is **already
machine‑readable** in the IG's `ImplementationGuide` resource /
`sushi-config.yaml` `dependsOn` list and is rendered in the published
IG's auto‑generated "Dependencies" tab/table. No editorial change is
required.

#### Justification

If the IG already publishes a generated dependency table that names US
Core and pins its version, a reader who needs that information can
find it via the standard IG navigation. Adding redundant prose risks
drift (the prose says one US Core version while the package manifest
pins another). Declining keeps the manifest as the single source of
truth.

---

### Recommendation

**Recommended disposition:** A

The cost of Disposition A is genuinely tiny (a paragraph and a link in
one Markdown/HTML page in the IG sources) and the benefit is
disproportionately large: implementers approaching the IG cold should
not have to infer from a `dependsOn` list that this IG is layered on
top of US Core Vital Signs. Disposition C's "the manifest already says
so" argument underweights the reality that introductory narrative is
where new readers form their mental model of an IG's scope; leaving
the relationship unstated is precisely the gap the reporter
identified. Disposition B is attractive and would be a strict
improvement, but it expands the ticket beyond what was asked and
should be tracked separately if the workgroup wants the broader
documentation pass — accepting A first lets the originally‑requested
fix land quickly, with the per‑profile callouts and dedicated
dependencies page handled as a follow‑up enhancement. To mitigate
Disposition C's drift concern, the introduction text should reference
the US Core version by linking to the IG's own dependency declaration
rather than hard-coding the version number in the prose.
