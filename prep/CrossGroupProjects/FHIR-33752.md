## Summary

The ticket asks the US Military Service (FHIR) IG to relocate the canonical
URL of the `military-service-separation-reason-vs` Value Set so that it is
rooted under `terminology.hl7.org` (THO) rather than under `hl7.org/fhir`.
The reporter additionally asks that a companion UTG ticket be created to
register the Value Set in THO, so that the terminology artifact lives in
the cross-WG terminology space rather than embedded inside the IG's own
canonical namespace. No cross-referenced Jira tickets, Zulip threads, or
GitHub items were surfaced for this ticket; the request stands on its
own as a canonical-URL / terminology-stewardship concern raised against
an early publication of the US Military Service IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reissue the `military-service-separation-reason-vs` Value Set under a
THO-rooted canonical URL (e.g.,
`http://terminology.hl7.org/ValueSet/military-service-separation-reason`)
and retire the `hl7.org/fhir`-rooted canonical from the US Military
Service IG. Concretely:

1. Open a companion UTG ticket to register both the Value Set and any
   IG-local CodeSystem it expands under `terminology.hl7.org`,
   following standard UTG submission rules (CodeSystem first, then
   ValueSet that references it).
2. In the next publication of the US Military Service IG, replace the
   in-IG ValueSet definition with a reference to the THO-published
   canonical, and add a deprecation note pointing the old canonical at
   the new one (or remove it and rely on UTG redirection if available).
3. Update any bindings inside the IG (StructureDefinitions, profiles,
   examples) that reference the old canonical to point at the THO
   canonical.

#### Justification

This is the pattern HL7 has trended toward for cross-WG and
cross-publication terminology: terminology artifacts live in THO and
are referenced by IGs, not embedded in IG canonical namespaces. Doing so
removes ambiguity about stewardship (UTG owns the artifact, the WG owns
the binding), allows reuse outside this single IG, and aligns with the
direction the project as a whole has taken since this ticket was filed.
Even though seanmuir's comment notes that R4 doesn't *require* this,
adopting the THO-rooted canonical keeps the IG forward-compatible and
avoids minting a new IG-rooted canonical that would need to be
deprecated later anyway.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing `hl7.org/fhir`-rooted canonical for backwards
compatibility on already-published versions of the US Military Service
IG, but for the *next* major republish:

- Define the new authoritative ValueSet (and any underlying CodeSystem)
  in THO via a UTG ticket as in Disposition A.
- In the IG, retain the old canonical as a *deprecated* alias that is
  bound nowhere new, and have all current/new bindings point at the THO
  canonical.
- Add an IG-level note documenting the move, and (if THO supports it)
  publish a "see also" / replaced-by relationship between the two
  canonicals.

#### Justification

This addresses the same underlying need (THO stewardship, cross-WG
discoverability) without invalidating already-deployed implementations
that resolve the old canonical. It treats the canonical-URL move as a
versioned transition rather than a breaking rename, which is consistent
with how FHIR generally handles canonical relocations and with the
"non-persuasive at this time" framing in seanmuir's comment — the
intent is honoured, but the cost to existing implementers is bounded.

---

### Disposition C: Decline

#### Proposal

Maintain the current resolution of **Considered for Future Use** and
leave the `military-service-separation-reason-vs` canonical rooted at
`hl7.org/fhir` for the FHIR R4-based US Military Service IG. Document
in the IG that any future major version (FHIR R5+ rebuild) will adopt
THO-rooted canonicals as a matter of course, and close this ticket
without further action against the R4 IG.

#### Justification

This matches the WG's existing position as recorded in seanmuir's
comment: FHIR R4 does not expressly require terminology canonicals to
be THO-rooted, so the original vote was deemed non-persuasive at the
R4 time horizon. Reissuing the canonical now would force a churn cycle
on every implementer of the current US Military Service IG for a
benefit that R4 does not mandate. Deferring the work to a future R5+
republish lets the WG do the THO migration once, cleanly, alongside
other modernizations rather than as a mid-life canonical rename.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of the principle the reporter is
defending (THO stewardship, cross-WG terminology hygiene) and the
practical concern the WG already noted (R4 doesn't require it; surprise
canonical changes are costly to implementers). Disposition A is
directionally correct but treats the move as a hard relocation, which
is harder to justify against a published R4 IG and is exactly the kind
of change the WG flagged as non-persuasive in 2022. Disposition C is
defensible against the literal text of R4 but leaves the IG with an
IG-rooted ValueSet canonical that everyone — WG, UTG, and implementers —
already expects to migrate eventually, just deferred. Doing the THO
registration now (via the UTG ticket the reporter asked for) and
retaining the old canonical as a deprecated alias gets the artifact
into the right home without breaking current consumers, and lines the
IG up to drop the alias cleanly at the next major republish.
