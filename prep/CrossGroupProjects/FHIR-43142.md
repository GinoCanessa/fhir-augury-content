## Summary

Lloyd McKenzie (in the Cross-Group Projects work group) reports that the
`military-service-veteran-status` extension defined in the US Military Service
FHIR IG declares its context as `Element`, which is too broad. As written, the
extension is structurally allowed to appear on virtually any element of any
data type or resource — for example on `Address.postalCode`,
`Encounter.dischargeDisposition`, `Condition.status`, etc. — even though it is
clearly only meaningful when describing a person/patient's veteran status.
The ticket requests that the extension's context be tightened to the specific
resources and/or elements where "veteran status" is semantically appropriate
(typically `Patient`, `Person`, and `RelatedPerson`).

No directly cross-referenced Jira tickets were returned by the FHIR Augury
index for this issue.

### [implementers > Extension:Context](https://chat.fhir.org/#narrow/stream/implementers/topic/Extension%3AContext/near/433346649)

An implementer (Syed Junaid) asked how to interpret the context ("Element ID
Element") of the `military-service-veteran-status` extension and whether they
could legitimately use it on a `Person` resource. Yunwei Wang explained that
the extension is technically permitted on any property whose data type is
`Element`. Lloyd McKenzie confirmed that the extension was improperly defined
with a universal context — so while there is no structural problem using it
on `Person`, it could just as wrongly be applied to `Condition.status`,
`Encounter.dischargeDisposition`, `Address.postalCode`, etc. This thread is
the practical manifestation of the issue raised in FHIR-43142 and reinforces
the need to tighten the context.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Military Service IG, change the `context` of
`StructureDefinition/military-service-veteran-status` from a single entry of
type `element` with `expression = "Element"` to a tightened set of typed
contexts that match where "veteran status" is meaningful. Concretely, replace
the existing `context` element with multiple entries:

- `{ "type": "element", "expression": "Patient" }`
- `{ "type": "element", "expression": "Person" }`
- `{ "type": "element", "expression": "RelatedPerson" }`

(Optionally also `Practitioner` if the work group considers that staff
veteran status is in scope; the ticket does not require it.) Republish the
StructureDefinition in the next IG release and note the change in the
IG's change log as a tightened constraint. Existing instances that already
place the extension on `Patient` / `Person` / `RelatedPerson` remain valid;
no FHIRPath or invariant changes are required.

#### Justification

This is exactly the change the ticket requests, and it aligns with the FHIR
guidance that an extension's `context` should reflect the elements where it
is semantically valid, not merely where it is syntactically expressible
(see the Extensibility page and `StructureDefinition.context` definition).
The Zulip thread above demonstrates real-world implementer confusion that
the current `Element` context produces. Restricting to `Patient`, `Person`,
and `RelatedPerson` matches the IG's narrative use of veteran status as a
person-level attribute and prevents nonsensical placements such as on
`Address.postalCode` or `Condition.status`.

---

### Disposition B: Alternative Approach

#### Proposal

Tighten the context, but anchor it on a single shared logical surface
rather than enumerating individual resources. Two flavours:

1. **Use `extension` as the context type** and list the `Patient.extension`,
   `Person.extension`, and `RelatedPerson.extension` paths explicitly, e.g.
   `{ "type": "element", "expression": "Patient" }` plus
   `{ "type": "extension", "expression":
   "http://hl7.org/fhir/StructureDefinition/patient-..." }` style references
   where applicable. This keeps the door open for the extension to ride
   along on profiles that already extend Patient/Person/RelatedPerson via
   composition.
2. **Introduce a "person-like resources" pattern** — keep the context on
   `Patient` and `RelatedPerson` only (the resources actually used in
   clinical exchange), and explicitly *exclude* `Person` on the grounds
   that `Person` is an administrative linkage resource and veteran status
   should travel with the clinically-relevant record. Document the choice
   in the IG.

#### Justification

Either variant addresses the same underlying problem (over-broad context)
but trades off coverage vs. precision. Variant 1 future-proofs for
profile-driven adoption but is harder for validators to reason about.
Variant 2 is the most defensible from a "where do consumers actually look
for veteran status" perspective and avoids ambiguity about whether the
data lives on the demographic-aggregation `Person` resource or the
clinical `Patient`. This may be preferable if the work group has a clear
position that `Person` is out of scope for the IG's use cases.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave the extension's context as `Element`. Document
in the IG that the extension is *intended* for use only on Patient, Person,
and RelatedPerson, and rely on narrative + validator profiles to police
misuse rather than the StructureDefinition context.

#### Justification

This is hard to defend. The only argument for declining is "we don't want
to break any consumer that may be relying on the broad context to attach
veteran status somewhere unusual," but no such use case has been raised on
the ticket or in the Zulip thread, and the change is non-breaking for the
documented intent. Declining would leave the IG in a state that publicly
contradicts the FHIR Extensibility guidance and that has already confused
implementers in production. This option is included for completeness but is
not advisable.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest, lowest-risk fix and is exactly what the
ticket asks for. The reporter is the FHIR methodology lead, and the bug is
straightforward: the extension was defined with the default/loose context
and never tightened. Restricting to `Patient`, `Person`, and
`RelatedPerson` is non-breaking for any sensible existing usage, removes
the confusion shown in the cited Zulip thread, and brings the extension
into line with FHIR's guidance on `StructureDefinition.context`. If the
work group has a substantive disagreement about whether `Person` belongs,
that can be resolved as a refinement of Disposition A (drop `Person` per
Disposition B variant 2) without changing the overall direction. Declining
(Disposition C) is not justified given the absence of any use case for
attaching veteran status to non-person elements.
