## Summary

The reporter requests that the US Military Service (FHIR) IG add slicing to the
"Industry" element so that, in addition to whatever industry vocabulary is
currently bound, implementers can also use the Occupational Data for Health
(ODH) industry value set — specifically *Industry NAICS Detail (ODH)* (PHIN VADS
OID `2.16.840.1.114222.4.11.7900`). The justification given is that ODH provides
a more complete and more readily collectable industry vocabulary than the
existing binding. No cross-referenced Jira tickets, Zulip threads, or GitHub
items were found for this issue, so the request stands on its own. The ticket
has already been triaged with a "Not Persuasive" resolution and a comment
indicating the underlying profile inheritance was modified, rendering the
specific change request OBE (overtaken by events).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a slice to the Industry element on the relevant US Military Service
profile (the patient/occupation-style profile that carries Industry) that binds
to *Industry NAICS Detail (ODH)*
(`http://phinvads.cdc.gov/vads/ViewValueSet.action?oid=2.16.840.1.114222.4.11.7900`).
The existing binding is preserved as the default slice; the new ODH NAICS
Detail slice is added alongside it (slicing discriminator on the coding
`system`, with `rules = open`). Document in the profile narrative that
implementers SHOULD use the ODH NAICS Detail coding when industry is captured
via ODH-aligned workflows.

#### Justification

ODH is the HL7-published standard for occupation and industry data and
explicitly uses the NAICS Detail value set the reporter cites. Adding an
ODH-aligned slice (rather than swapping the binding) preserves backward
compatibility for existing US Military Service implementers while enabling
ODH-conformant systems to exchange the more granular, more collectable
vocabulary the reporter describes. Slicing on `coding.system` is the standard
FHIR pattern for permitting multiple bindings on the same element.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding a profile-level slice, broaden the value-set binding on the
Industry element to an extensible (or "preferred + example bindings") binding
whose primary value set is the existing US Military Service industry list and
whose additional binding includes *Industry NAICS Detail (ODH)*. Cross-link the
US Military Service IG narrative to the ODH IG so implementers know ODH NAICS
Detail is an explicitly-blessed alternative without forcing every consumer to
parse a slice.

#### Justification

Slicing adds parsing/validation cost for every downstream consumer of the
profile, even those that never use ODH. Using `additional` value-set bindings
(R5-style) or an extensible binding with documented alternates achieves the
same interoperability outcome (ODH NAICS Detail is permitted) with less
structural complexity. This approach also keeps the US Military Service IG out
of the business of versioning ODH terminology — implementers follow ODH
directly. Trade-off vs. Disposition A: weaker machine-enforcement of which
codings are acceptable, but lower implementation burden.

---

### Disposition C: Decline

#### Proposal

Mark the request **Not Persuasive — Out of Date / OBE**, on the basis that the
US Military Service profile's inheritance has already been modified (per
seanmuir's 2022-10-10 comment) such that the Industry element now flows from
an upstream profile that resolves the vocabulary concern, and the reporter
acknowledged the change ("ok will update"). No additional slicing is required
on the US Military Service profile itself.

#### Justification

This is the disposition already recorded in Jira. The substantive concern —
that ODH NAICS Detail be reachable for industry data — was addressed by
restructuring the profile inheritance rather than by adding a slice in this
IG. Re-opening the ticket to add a slice now would either duplicate work
already done upstream or introduce a second, conflicting binding. There is no
new evidence in the ticket, comments, or community discussion (no Zulip
threads, no linked tickets, no GitHub PRs) suggesting the upstream fix is
insufficient.

---

### Recommendation

**Recommended disposition:** C

The ticket is already resolved "Not Persuasive" with an explicit OBE rationale
from the profile author and an acknowledgement from the reporter. Both parties
on the ticket agreed the underlying inheritance change addressed the concern,
and no community signal (Zulip, GitHub, related Jira) has surfaced in the four
years since to suggest otherwise. Re-opening to pursue Disposition A or B
would re-litigate a settled item without new use-case evidence and risks
introducing a second, conflicting binding on top of the inheritance fix that
already landed. The workgroup should ratify the existing "Not Persuasive — OBE"
disposition; if a fresh use case for an explicit ODH NAICS Detail slice
emerges, it should be filed as a new ticket against the current profile
structure.
