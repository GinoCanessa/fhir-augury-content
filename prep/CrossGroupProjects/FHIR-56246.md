## Summary

FHIR-56246 is a non-substantive clarification request against the US Core
(FHIR) IG. The reporter (Eric Haas) proposes a minor wording change to the
Must Support narrative on `must-support.html#must-support---complex-elements`,
rephrasing "is represented in the structure" to "is present in an instance"
when describing the precondition under which servers SHALL support sub-elements
that are flagged Must Support on a complex parent. The intent is to make the
test for triggering Must Support obligations refer to actual instance data
rather than the more ambiguous "structure," matching how implementers and
testers actually evaluate conformance.

The ticket is already triaged as **Persuasive** with the resolution "update
as proposed," and is bundled into US Core V9 Block Vote 3 (announced on the
`#argonaut > US Core V9 Ballot Reconciliation` Zulip topic) as a
non-substantive clarification. No related Jira tickets were surfaced via
cross-references; the only code touch point is a US Core IG narrative file
(`input/pagecontent/changes.md` and, by inference from the page slug, the
must-support narrative source).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Core (FHIR) IG, on the `must-support.html` page, in the
"Must Support — Complex Elements" section, replace the phrase
"is represented in the structure" with "is present in an instance" in the
sentence:

> If any sub-element is marked as Must Support [for a complex element] and
> the parent element is not… [and] the parent element **is represented in
> the structure**, Servers **SHALL** support the sub-element(s) marked as
> Must Support.

so it reads:

> If any sub-element is marked as Must Support [for a complex element] and
> the parent element is not… [and] the parent element **is present in an
> instance**, Servers **SHALL** support the sub-element(s) marked as Must
> Support.

The change is a one-page edit to the must-support narrative source in
`HL7/US-Core` (most plausibly `input/pagecontent/must-support.md` — to be
confirmed against the repo briefing) plus a one-line entry in
`input/pagecontent/changes.md` recording it as a
Non-substantive (Clarification) Requirements QA item. No StructureDefinition,
profile, value set, or example needs to change. Mark the change as
Non-substantive (Clarification) in the IG change log and include it in
US Core V9 Block Vote 3.

#### Justification

The Must Support obligation for sub-elements of a complex parent has always
been intended to fire when the parent actually appears in an instance, not
when the parent merely exists as a node in the differential / snapshot.
"Is represented in the structure" reads as a statement about the
StructureDefinition; "is present in an instance" matches the runtime
conformance test that implementers and testers actually apply. The reword
brings the prose into line with how Must Support is exercised against real
data and aligns with the broader US Core conformance language. The CGP/US
Core call already triaged the ticket as Persuasive with a "Non-substantive
(Clarification)" change impact, the Zulip Block Vote 3 announcement records
no objections to including it, and the resolution text on the Jira issue is
literally "update as proposed."

---

### Disposition B: Alternative Approach

#### Proposal

Apply the requested wording change, but, in the same edit, also add a
one- or two-sentence parenthetical immediately after the affected sentence
that explicitly anchors the rule to instance data — for example:

> (That is: the SHALL applies on a per-instance basis. If a complex parent
> element is absent from a given instance, no Must Support obligation is
> raised for its Must-Supported children in that instance.)

This is still one edit to the must-support narrative source in `HL7/US-Core`
(plausibly `input/pagecontent/must-support.md`, to be confirmed against the
repo briefing) plus the corresponding entry in `input/pagecontent/changes.md`,
but it removes any residual ambiguity about whether the rule is evaluated
per-StructureDefinition or per-instance.

#### Justification

The wording swap from "is represented in the structure" to "is present in an
instance" is a clear improvement, but it leaves implicit *what* "present in
an instance" means in a population sense (any one instance? every instance?).
A short parenthetical pinning the rule to per-instance evaluation pre-empts
the next round of "what does this mean" trackers without enlarging scope or
changing conformance, and it would still qualify as Non-substantive
(Clarification). The trade-off vs. Disposition A is slightly more text in
the narrative and a small additional review burden, against a measurable
reduction in re-litigation risk on a perennially debated paragraph.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-56246 as **Not Persuasive** and leave the must-support narrative
text in `HL7/US-Core` unchanged. Rationale: already addressed by the
existing wording when read in context with the rest of the Must Support
section.

#### Justification

This disposition is hard to defend on the merits. The base FHIR
must-support guidance already evaluates Must Support against instance data,
and replacing "is represented in the structure" with "is present in an
instance" makes the US Core sentence consistent with that established
reading without changing any conformance obligation. Declining would
preserve a phrase that has been demonstrably misread by implementers and
QA reviewers — the very reason the ticket was raised — and would reject a
zero-cost, already-Persuasive clarification that the work group has
already agreed to.

---

### Recommendation

**Recommended disposition:** A

The change is a true clarification: it does not alter any Must Support
obligation, does not touch any StructureDefinition or value set, and only
edits a single narrative sentence plus a `changes.md` entry. It is already
triaged as **Persuasive** on the Jira issue with the resolution "update
as proposed," it is already bundled into US Core V9 Block Vote 3 as a
Non-substantive (Clarification), and the Zulip thread shows no objection
to including it in that block. Disposition B's parenthetical is defensible
but inflates the scope of a clarification that the work group has already
decided to take "as proposed"; if reviewers feel the per-instance reading
needs further reinforcement, that can be raised as a separate follow-up
ticket against the same paragraph rather than gated on this one. Disposition
C is unsupported by the discussion record. Recommend approving the wording
change exactly as written in the ticket.
