## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the ticket's two narrative edits verbatim in the US Core IG
(`HL7/US-Core`):

1. In the Must Support Requirements narrative (the source for
   `must-support.html`, under `input/pagecontent/`), change the word
   **"can"** in the relevant sentence to the RFC 2119 keyword
   **MAY** (bold, to match the IG's typographic convention for
   normative keywords).
2. Delete the sentence: *"The U.S. Core Data for Interoperability
   (USCDI) may require additional elements such as `Patient.suffix`.
   §,§"* — including the trailing footnote markers. Re-flow any
   footnote references that now point at nothing.
3. Keep the existing change-log entry in `input/pagecontent/changes.md`
   crediting FHIR-56217 under Block Vote 3 with change impact
   **Non-substantive (Clarification)**.

No FSH, profile, or ValueSet changes are required; the edit is purely
narrative.

#### Justification

This is exactly what the ticket, the CGP/US Core reconciliation call,
and the Block Vote 3 Zulip announcement agreed to, and the change is
already captured as the intended resolution ("update as proposed",
**Persuasive**). Replacing an informal "can" with the RFC 2119 **MAY**
aligns the sentence with the IG's own conformance-keyword discipline
and with the sibling tickets in the same block (FHIR-56267, FHIR-56272,
FHIR-56271) that are harmonising "may/can/MAY" usage. Removing the
USCDI sentence avoids an out-of-date and slightly ambiguous aside about
USCDI possibly requiring extra elements such as `Patient.suffix` — the
IG's Patient profile already addresses `Patient.suffix` in its own
Must Support machinery, so the aside is redundant and prone to drift.
The commit `cec272b27a` shows the implementer has already batched
this change with the rest of the Block Vote 3 edits, which reduces
review cost.

---

### Disposition B: Alternative Approach

#### Proposal

Accept edit (1) — change "can" to **MAY** — but **replace** (rather
than delete) the USCDI sentence with a short, neutral pointer
paragraph, e.g.:

> "Implementations subject to USCDI may need to support additional
> elements beyond those marked Must Support here; consult the current
> USCDI release for authoritative requirements."

and move the `Patient.suffix` example (if it is still illustrative)
into the Patient profile page or the USCDI ↔ US Core mapping table
referenced by sibling ticket FHIR-54216.

#### Justification

The existing sentence does serve a real purpose: it warns implementers
that conformance to US Core's Must Support set is **not** equivalent to
conformance to USCDI, which can require more. Deleting it outright
(Disposition A) removes that warning entirely and relies on readers to
discover the USCDI↔US Core mapping elsewhere. Replacing it with a
version-agnostic pointer preserves the signal, drops the potentially
stale `Patient.suffix` example, and avoids cross-page duplication.
This approach also dovetails with FHIR-56221 ("Remove text") and
FHIR-54216 (USCDI↔US Core mapping table) — sibling tickets in the same
reconciliation pass — by ensuring the mapping story has exactly one
canonical home.

---

### Disposition C: Decline

#### Proposal

Decline edit (2) (do not remove the USCDI sentence) and accept only
edit (1), re-scoping the ticket to the "can → MAY" fix. Rationale
category: **Partial — preserve useful implementer warning**.

#### Justification

The "can → MAY" change is squarely an RFC 2119 clarification and
should proceed. The USCDI sentence, however, performs a small but
non-trivial role: it signals to implementers that US Core's Must
Support set is a floor, not a ceiling, relative to USCDI. Until an
equivalent, discoverable statement exists elsewhere in the IG
(for example in a USCDI mapping page), removing it without
replacement risks implementers silently assuming US Core MS ≡ USCDI.
Declining the removal is defensible as "no replacement identified
yet"; the ticket can be reopened or replaced once a better home for
the warning exists.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already aligned on "Persuasive / Non-substantive
(Clarification)" both in the ticket's resolution and in the Block
Vote 3 Zulip announcement, and the corresponding implementation
commit (`cec272b27a`) has been prepared alongside the sibling
requirements-QA tickets. The "can → MAY" change is pure RFC 2119
hygiene and is uncontroversial. The USCDI sentence being removed is
narrowly scoped: it names a single example element (`Patient.suffix`)
that is already covered by the Patient profile's own Must Support
machinery, and the broader "US Core MS is a floor, not a ceiling"
message is better delivered in a dedicated USCDI-mapping context
(the mapping table tracked by FHIR-54216) than as a drive-by
footnote on the Must Support page. Disposition B would be a
reasonable fallback if the workgroup wants to preserve the warning
inline, but given the existence of a dedicated mapping table in the
IG and the preference for non-substantive, low-risk edits in this
ballot cycle, the simpler deletion in Disposition A is appropriate
and matches the already-agreed resolution.
