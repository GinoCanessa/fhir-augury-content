## Summary

The reporter asks that the Section 4 title in the CardX Hypertension
Management (FHIR) Implementation Guide be changed from "Card X" (with a
space) to "CardX" (no space), to match the project's branded name. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found
in FhirAugury for this ticket; the available context is the ticket text
plus two comments from the work group. The ticket was previously
dispositioned "Considered for Future Use" (Bray / Tcheng vote 8-0-1 on
2024-06-27) on the basis that FHIR Shorthand (FSH) does not currently
permit two capital letters in the same word, which prevented updating
the header to "CardX"; that resolution was reverted and the ticket
re-opened post-STU1 publication on 2024-12-12 so it can be revisited.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the Section 4 heading in the CardX Hypertension Management IG
narrative from "Card X" to "CardX". Because the work-group comment
records that FSH cannot generate a heading containing two consecutive
capitals in the same word, accepting the request requires bypassing
that limitation: edit the IG's narrative authoring source directly
(e.g., the page-content markdown / XHTML used to render Section 4)
rather than relying on auto-generation from a FSH `Title:` keyword, or
post-process the published header. The IG `package-list.json` /
publication request remain unchanged; only the Section 4 page title
text changes. A short note can be added to FSH developers (per
lynn_laakso's 2024-12-12 comment) so the underlying FSH limitation is
addressed for future builds.

#### Justification

This is a low-risk cosmetic correction that aligns the published IG
with the project's branded name, "CardX". The reporter's request is
unambiguous and the change does not affect any normative content,
profiles, value sets, or examples — only a section heading. Treating
the FSH limitation as an authoring constraint to be worked around
(narrative override) rather than as a reason to refuse the fix
preserves the user-facing branding while leaving the underlying FSH
issue to be addressed on its own track.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the rendered Section 4 title as a single token but use a
typographic separator that FSH does accept, so the title is
auto-generatable today. Two viable variants:

1. Use a non-breaking-space-free single token by replacing the space
   with a hyphen or thin separator that FSH allows — e.g., "Card-X" or
   "Card_X" — only if the project's branding guide will accept it; OR
2. Leave the heading as "Card X" but add a one-line note in the IG
   introduction that the project name is properly written "CardX",
   keeping the FSH-generated heading unchanged.

Either variant avoids manual narrative overrides and keeps the IG
build fully driven by FSH.

#### Justification

This honours the underlying authoring tooling: the work group already
identified FSH's two-capital-letter restriction as the blocker, and
working around it by hand-editing rendered output creates a recurring
maintenance burden every time the IG is rebuilt. A
typographically-distinct but FSH-legal title (B.1) keeps the heading
and the brand visually associated, while the introduction-note variant
(B.2) is the most conservative path and preserves the current build
chain entirely.

---

### Disposition C: Decline

#### Proposal

Persuasive — Considered for Future Use. Decline to change the heading
in the published STU1 (the previous 8-0-1 resolution direction), but
record the request to be revisited once FSH supports headings with two
adjacent capitals. File the corresponding FSH enhancement ticket
referenced in the 2024-12-12 comment and link it from this ticket so
the dependency is explicit.

#### Justification

The original work-group vote (8-0-1) reached the same conclusion: the
fix is desirable but not currently achievable through the IG's
authoring toolchain without unusual workarounds, and the cosmetic
benefit does not justify diverging from FSH-driven generation for STU1.
Re-opening the ticket so it tracks the FSH enhancement (rather than
attempting to apply a narrative-override fix in the IG) keeps the
branding correction in scope without committing the IG team to
maintenance overhead in the meantime.

---

### Recommendation

**Recommended disposition:** A

The change requested is a one-character correction to a section
heading, the reporter is correct that the project is branded "CardX",
and the only obstacle on record is a FSH authoring restriction — not a
specification, profile, or terminology concern. A narrative override
for one heading in one IG is a small, well-scoped maintenance cost,
and it delivers the user-visible fix immediately rather than blocking
on an upstream FSH enhancement of indeterminate timing. Pair the fix
with the FSH developer ticket already promised in the 2024-12-12
comment so that future IG rebuilds can return to fully-generated
headings without losing the branding correction in the interim.
Disposition B is a reasonable compromise if the work group is
unwilling to maintain a narrative override; Disposition C should only
be selected if the work group decides to defer all branding
corrections until FSH catches up.
