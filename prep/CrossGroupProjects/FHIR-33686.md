## Summary

John Moehrke (HL7 Cross-Group Projects, US Military Service IG) is reporting
that the US Military Service implementation guide currently provides no way to
retrieve "all relevant Military Service resources" for a given subject as a
single, coherent set. He suggests two possible mechanisms: (a) a defined REST
search that returns a search-set `Bundle` containing every Military Service
profile instance for a patient, or (b) a defined `Composition` (and resulting
document `Bundle`) that brings the Military Service content together as a
clinical summary, analogous to how IPS/CCDA-style summaries assemble related
content for human consumption. The ticket is filed against the US Military
Service IG (a Cross-Group Projects deliverable) and was originally voted
"Considered for Future Use" in 2022-07-28, then re-opened by Lynn Laakso
post-STU1 publication in 2023-05-30 so that the work group can reconsider it,
with Sean Muir noting the group will discuss Composition support jointly with
the CDA-on-FHIR projects.

No other Jira tickets cross-reference FHIR-33686, no Zulip threads were
returned by the search, and no GitHub PRs/issues link to this ticket; the
US Military Service IG repository is not present in the local repo cache,
so there is no `repo-analysis` briefing available.

## Proposed Dispositions

The dispositions below are scoped to the US Military Service IG (the
specification recorded on the ticket). Because no `repo-analysis`
briefing is available for that IG, paths and authoring-root details are
described in IG-publishing-conventions terms (e.g., `input/fsh/`,
`input/pagecontent/`) rather than verified against a cached repo. If
the work group accepts a code-touching disposition, a `repo-analysis`
run on the US Military Service IG repo should be commissioned before
implementation begins.

### Disposition A: Accept As Requested

#### Proposal

Add a "Summarization" page to the US Military Service IG that defines
**both** retrieval mechanisms John proposed, with the search-set option
as the normative baseline and the Composition option as an additional
profile:

1. **Search-set Bundle:** Define a documented "fetch all Military
   Service content for a patient" interaction — a single
   `GET [base]/[Resource]?patient=[id]` query per Military Service
   profile, plus a recommended `Bundle` shape (type `searchset`)
   assembled by the client, OR (preferred) a server-side `$everything`
   style operation scoped to Military Service profiles, defined as a
   new `OperationDefinition` in the IG (e.g.,
   `OperationDefinition/military-service-everything`).
2. **Composition profile:** Add a new `Composition` profile
   (`StructureDefinition/military-service-summary`) whose `section`
   slices reference each Military Service profile (service period,
   deployments, exposures, separation, etc.). Add a corresponding
   document `Bundle` profile (`type = document`) that anchors the
   Composition.
3. Author the changes in the IG sources (typically `input/fsh/` for
   the Composition / OperationDefinition and `input/pagecontent/` for
   the new "Summarization" narrative page) and add the new artifacts
   to the IG's table of contents and `sushi-config.yaml` resource
   list.

#### Justification

This directly answers the reported gap: today an implementer has no
documented way to assemble Military Service content for a subject. Both
mechanisms are well-established FHIR patterns (search-set bundles and
clinical-document Compositions), and Sean Muir's 2022-08 comment shows
the work group has already accepted that Composition support is
in-scope and would be coordinated with CDA-on-FHIR. Defining both
gives REST-style implementers and document-style implementers a
sanctioned answer, and the `Composition` route aligns naturally with
how comparable "summary of X" use cases are handled (IPS, US-PCS,
PACIO summaries), so reviewers and tooling already understand the
pattern.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt only the **Composition / document-Bundle** half of Disposition A
and explicitly defer the operation/search-set half to implementer
guidance:

- Add a single `Composition` profile
  (`military-service-summary`) plus a `Bundle` profile of
  `type = document` to the US Military Service IG, with section slices
  for each Military Service profile already in the guide.
- Add an "Assembling a Military Service Summary" narrative page that
  shows two worked examples: one using the Composition/document
  pattern, and one using parallel REST queries (`GET
  [base]/Observation?patient=...&_profile=...` etc.) without
  introducing a new operation.
- Coordinate the Composition shape with the CDA-on-FHIR projects (per
  Sean Muir's 2022-08 commitment) so the same logical summary can
  round-trip to/from a CDA Continuity-of-Care-style document.
- No new `OperationDefinition` is published; servers remain free to
  expose `$everything`-like operations, but the IG does not require
  them.

#### Justification

This addresses the underlying need — "give me a coherent summary of a
patient's military service" — without committing the IG to maintain a
new operation, which historically introduces conformance burden on
servers and tooling support requirements. A Composition is also
inherently more useful for the clinical-document audience driving the
US Military Service / Cross-Group Projects work, and it preserves the
CDA-on-FHIR coordination already promised on the ticket. Implementers
who only need REST-style aggregation can still do so against the
existing profiles using the documented narrative example, so no
capability is lost.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive — already addressed by base-FHIR
mechanisms." Rationale: any FHIR client can already retrieve all
Military Service content for a patient using existing base-spec
mechanisms — `_profile`-scoped searches per resource type, or, where
the server supports it, `Patient/[id]/$everything` filtered
client-side. The IG does not need to introduce a new artifact to
enable behavior that is already possible with base FHIR.

#### Justification

The base FHIR API already provides search-set Bundles, the `_profile`
search parameter, and, on capable servers, `$everything`. Adding a
new operation or a Composition profile risks divergence from how
analogous "summary of X" content is being defined in IPS, US-PCS, and
the CDA-on-FHIR work — none of which the Military Service IG is yet
aligned with. Until a concrete consumer use case is documented (e.g.,
"VA needs to render a Military Service summary in this specific
context"), introducing new normative artifacts is premature and adds
maintenance load to a small IG.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of the three. The reporter's core
need is a *coherent* summary of a patient's military service, not a
new search semantic — and a `Composition`/document-`Bundle` pair is
the standard FHIR answer for that, already validated by IPS and the
broader CDA-on-FHIR work the Cross-Group Projects work group is
explicitly coordinating with (per Sean Muir's 2022-08 comment).
Disposition A bundles in a new `OperationDefinition` that has no
demonstrated demand on the ticket and would commit servers to extra
conformance surface; that can always be added in a later cycle if
implementer feedback warrants it. Disposition C is defensible but
ignores the fact that the work group has already, on the record,
agreed to "consider supporting a Composition," and the ticket's
re-opening post-STU1 signals that the group still wants to act on
that commitment rather than walk away from it. Approving B closes the
gap, honors the prior commitment, and keeps scope tight enough to
land in the next STU update.
