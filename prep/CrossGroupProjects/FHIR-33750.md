## Summary

The ticket asserts that the canonical URL for the `military-service-branch-vs`
ValueSet — defined by the US Military Service (FHIR) IG — should be rooted at
`terminology.hl7.org` (THO) rather than under `hl7.org/fhir`. The reporter
also asks that a corresponding UTG ticket be opened to register the ValueSet
in THO so the canonical move is backed by the proper governance path. The
ticket has a recorded resolution of "Considered for Future Use", with the
explanatory note that the requirement to root non-FHIR-core terminology
artefacts in THO is explicit in FHIR R5 but not in R4, so for the moment the
votes are being treated as "non-persuasive". No cross-referenced Jira
tickets, Zulip threads, or GitHub items were returned by the FHIR Augury
data sources for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-root the canonical URL of `military-service-branch-vs` so that it lives
under `http://terminology.hl7.org/ValueSet/military-service-branch-vs`
instead of `http://hl7.org/fhir/...`, and open a companion UTG ticket to
register (and own going forward) the ValueSet in THO. As part of the move,
publish a deprecation note for the old canonical, update any referencing
profiles or extensions in the US Military Service IG to point at the new
canonical, and ensure the THO-hosted artefact carries the same identifier,
status, and code system bindings as the existing IG-hosted definition.

#### Justification

This aligns the artefact with the FHIR R5 direction that non-core
terminology should be authored and versioned in THO, and it removes the
ambiguity of a domain-specific value set living in the core FHIR canonical
namespace. It also gives the value set a stable home that other IGs can
reuse without taking a dependency on the US Military Service IG's
publication cadence, and it follows the standard UTG governance flow that
the reporter explicitly asked for.

---

### Disposition B: Alternative Approach

#### Proposal

Leave the existing `hl7.org/fhir`-rooted canonical in place for the current
publication of the US Military Service IG to avoid a breaking canonical
change, but open the UTG ticket now to publish a parallel
`terminology.hl7.org`-rooted ValueSet with the same content. Mark the
IG-hosted ValueSet as deprecated and direct implementers at the THO copy
for future use, switching the IG's bindings to the THO canonical at the
next substantive republication of the IG.

#### Justification

This addresses the underlying governance concern (the value set belongs in
THO) without breaking any in-flight implementations that already resolve
the existing canonical. It honours the resolution note that R4 does not
mandate THO-rooting, while still moving the artefact toward the R5
direction. The trade-off is that there are temporarily two canonicals for
the same concept set, which has to be managed via a clear deprecation
notice — but that is a smaller blast radius than swapping the canonical
out from under existing consumers in a single step.

---

### Disposition C: Decline

#### Proposal

Mark the change request as "Not Persuasive — for R4". The
`military-service-branch-vs` ValueSet keeps its current
`hl7.org/fhir`-rooted canonical for the R4 publication of the US Military
Service IG. Re-rooting in THO becomes an item to revisit when the IG is
re-published against R5 (or a later version where the THO-rooting rule is
explicit), at which point a fresh ticket can drive the move and the UTG
registration in one coordinated step.

#### Justification

This is the position implied by the recorded resolution: the policy basis
for forcing a THO canonical is explicit in R5 but not in R4, so changing
the canonical now would impose a breaking change on implementers without a
normative rule to point at. Declining for R4 also avoids creating
two long-lived canonicals for the same ValueSet, and it keeps all
re-rooting work bundled with the next planned republication so that
binding updates, UTG registration, and deprecation notices land together
rather than piecemeal.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of the governance ask and the
practical constraints flagged in the resolution. The reporter's underlying
concern — that domain terminology should live in THO and be governed via
UTG — is legitimate and is the direction the standard is moving in R5, so
ignoring it (Disposition C) just defers the same work without solving the
ownership question. At the same time, hard-swapping the canonical now
(Disposition A) imposes a breaking change in R4 with no normative rule to
back it, which is exactly the basis on which the original vote was deemed
non-persuasive. Publishing the THO-rooted ValueSet in parallel and
deprecating the IG-hosted one lets Cross-Group Projects open the UTG
ticket immediately, gives implementers a clear migration target, and
leaves the canonical switch in IG bindings to land cleanly at the next
substantive republication.
