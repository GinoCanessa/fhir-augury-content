## Summary

This ticket asks the US Core editors to remove the per-profile narrative
"requirements" content that is currently reproduced inside the US Core
**CapabilityStatement(s)** and instead point the reader to where that
content now lives. Today the CapabilityStatement repeats every SHALL /
SHOULD bullet from each profile's *Profile-Specific Implementation
Guidance* section. The reporter (Eric Haas) notes that this hand-curated
duplication is now fully covered by the new US Core
**Requirements Tables** and **Requirements resources** introduced in v9,
making the CapabilityStatement copy redundant and a maintenance burden.
The proposal is therefore to (a) delete the duplicated bullets from the
US Core Server and Client CapabilityStatements and (b) add a short
pointer in the CapabilityStatement introduction explaining where the
authoritative requirements list now lives (the Requirements page /
Requirements resources). The ticket has already been triaged as
**Persuasive** and is queued for **Block Vote 3** at the 2026-04-09
CGP / US Core Ballot Reconciliation call alongside other clarifying
narrative-requirements clean-ups (FHIR-56272, FHIR-56271, FHIR-56267,
FHIR-56260, FHIR-56246, FHIR-56244, FHIR-56221, FHIR-56217). No
explicit Jira "is-related-to" links are recorded; the related-ticket
context comes from the Block Vote 3 announcement on Zulip.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change as written, with resolution **Persuasive**:

1. In the US Core **Server** CapabilityStatement and the US Core
   **Client** CapabilityStatement (sources under `HL7/US-Core` →
   `input/`), remove the per-profile bullet lists that reproduce each
   profile's SHALL / SHOULD *Profile-Specific Implementation Guidance*
   statements. Leave the structural conformance content
   (resource interactions, supported profiles, search params,
   operations, must-support flags) untouched.
2. In the CapabilityStatement introduction (the `capabilitystatements`
   narrative page and/or the per-CS intro), add a short pointer such as:
   *"The normative SHALL / SHOULD conformance requirements that were
   previously reproduced here are now maintained in the
   [Requirements page](requirements.html#requirements-tables) and the
   underlying Requirements resources."*
3. Confirm `input/pagecontent/changes.md` lists the change under
   v9.0.0 (already done by commit `93f97b7`) and link from the change
   entry to the per-resource Requirements Table anchors so reviewers
   can see the new authoritative location.
4. Vote as part of US Core v9 **Block Vote 3** with change impact
   **Non-substantive (Clarification)**.

#### Justification

The duplicated content is, by the reporter's own analysis and the
existence of the new Requirements Tables / Requirements resources, a
genuine duplicate of normative content that now has a single source of
truth. Removing the duplicate eliminates a known drift risk: any future
edit to a profile's implementation-guidance bullet would otherwise need
to be hand-mirrored into the CapabilityStatement. Keeping a short
pointer in the CapabilityStatement intro preserves discoverability for
implementers who still navigate from the CapabilityStatement first.
The resolution is already triaged Persuasive, the change is classified
Non-substantive (Clarification), and the surrounding Block Vote 3
companions on the Zulip thread show the editors are deliberately
tightening normative-keyword usage in this same review pass.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the request but **replace the human-curated
bullets with a generated stub** instead of deleting them outright. Have
the IG publish step (or a small pre-publish script) auto-emit, for each
profile referenced by the CapabilityStatement, a one-line entry of the
form:

> *Patient — see [Patient profile requirements](requirements.html#hl7.fhir.us.core-patient).*

…driven by the same Requirements resources that back the Requirements
Tables. Keep the CapabilityStatement introduction pointer from
Disposition A. Substantively this still removes the hand-curated SHALL
/ SHOULD bullets but preserves a per-profile breadcrumb directly inside
the CapabilityStatement page rendering.

#### Justification

Some implementers and downstream tooling (e.g., conformance dashboards,
test harnesses such as Inferno, Touchstone) read the
CapabilityStatement page as their primary entry point and may rely on
finding *some* per-profile requirement context inline. A generated stub
keeps that affordance while still eliminating the manual-curation
burden the ticket flags, because the stub is sourced from the
Requirements resources rather than typed by hand. The trade-off
relative to A is added publish-time complexity (a small generator) and
a slightly higher risk of stale links if a profile is renamed without
updating the generator. It also pushes the work into v9.x rather than
landing cleanly in v9.0.0 Block Vote 3.

---

### Disposition C: Decline

#### Proposal

Decline the request as **Not Persuasive**. Leave the per-profile
SHALL / SHOULD bullets in the US Core Server and Client
CapabilityStatements unchanged for v9.0.0. Optionally accept the
*pointer* sentence in the CapabilityStatement introduction as a
discoverability aid for the new Requirements Tables, but do not
remove the existing inline content.

#### Justification

A defensible Decline rests on three points: (1) the
CapabilityStatement is the artifact that conformance tooling and many
implementers historically consult first; removing inline normative
bullets shifts a load-bearing affordance onto a brand-new
Requirements page whose stability across publishes is not yet proven.
(2) The Requirements Tables and Requirements resources were only
introduced in v9 itself, so there is no v8.x → v9.0.0 deprecation
window for tooling vendors to react to the relocation. (3) FHIR-56261
is a small, low-risk improvement that could equally well land in
v9.1.0 once the new Requirements story has been exercised in the field.
This option exists for completeness; it is unlikely the workgroup
would prefer it.

---

### Recommendation

**Recommended disposition:** **A — Accept As Requested.**

Disposition A is recommended. The ticket has already been triaged
Persuasive, the change has been bundled into Block Vote 3 with no
objections raised on the Zulip thread, the editors have published a
companion entry in `input/pagecontent/changes.md`, and the underlying
problem the ticket identifies (manually mirrored normative bullets in
the CapabilityStatement vs. the new Requirements resources) is a real
single-source-of-truth violation that will only get worse as US Core
evolves. The change is classified Non-substantive (Clarification),
which is appropriate: the normative content is not being weakened —
only relocated to the artifact (Requirements resources / Requirements
Tables) that now owns it. Disposition B is intellectually attractive
but introduces v9.0.0-blocking publishing-tool work for a benefit
(per-profile breadcrumbs inline in the CS page) that the simple
introductory pointer in A largely already delivers; B is better filed
as a post-v9.0.0 enhancement if implementer feedback after publication
shows the pointer is insufficient. Disposition C would entrench the
duplication the new Requirements infrastructure was designed to
retire, and the Block Vote 3 sweep is precisely the right vehicle to
clean it up.
