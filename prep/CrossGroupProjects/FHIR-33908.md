## Summary

The ticket asks the US Military Service (USMS) FHIR implementation guide to add
slicing to the occupation element so that the Occupational Data for Health
(ODH) ONETSOC-based vocabulary
([PHIN VADS OID 2.16.840.1.114222.4.11.7901](https://phinvads.cdc.gov/vads/ViewValueSet.action?oid=2.16.840.1.114222.4.11.7901))
can be used alongside whatever occupation coding the IG already binds. The
reporter (Genevieve Luensman, NIOSH/ODH) argues the ODH vocabulary is more
collectable and more complete for representing occupation than the existing
binding, so military-service profiles should accept ODH-coded values via a
sliced binding rather than forcing a single code system. The ticket was
resolved **Persuasive** on 2022-04-21 and tagged **Block-Vote-2**, indicating
the work group accepted the change in principle as part of a block vote and
that the IG authors agreed to update accordingly (the lone comment, from the
reporter, simply says "ok will update"). No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced for this issue, so this prep
focuses on confirming/ratifying the change rather than re-litigating it.

## Proposed Dispositions

> Note: The ticket is already resolved **Persuasive** under Block-Vote-2.
> The dispositions below frame the choices the work group had at resolution
> time and that any follow-up confirmation ballot would re-evaluate.

### Disposition A: Accept As Requested

#### Proposal

In the US Military Service IG, on the profile element that carries
occupation (e.g., the `Observation.value[x]` / `Observation.code` pair, or
the equivalent `code`/`value` element of the relevant occupation profile),
add discriminated slicing on `coding` (discriminator
`value` on `coding.system`) and define an additional slice bound to the ODH
**Occupation ONETSOC Detail** value set
(`https://phinvads.cdc.gov/vads/ViewValueSet.action?oid=2.16.840.1.114222.4.11.7901`,
canonical `urn:oid:2.16.840.1.114222.4.11.7901`) at strength **extensible**.
Retain the existing binding as a separate slice so historical/military-
specific codes remain valid. Document the new slice in the profile's
narrative and add an example instance using an ONET-SOC code.

#### Justification

This is the literal request and matches FHIR's standard mechanism for letting
a single coded element accept values from more than one terminology
(`Coding`-based slicing is the pattern used across US Core, ODH, and other
NIOSH-aligned IGs). The reporter is the ODH lead and confirmed she would
update the IG, and the work group already marked the ticket Persuasive in
Block-Vote-2 — i.e., the substantive debate is closed. Aligning USMS with the
ODH vocabulary improves cross-IG consistency for occupation reporting and
removes a barrier for implementers who already capture ONET-SOC.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of in-profile slicing, define (or reuse) a single composed value set
that unions the existing USMS occupation codes with the ODH ONETSOC Detail
value set, and rebind the occupation element to that composed value set at
strength **extensible**. Optionally add a NIOSH-recommended preferred binding
via an `elementdefinition-preferredBinding` extension pointing at the ODH
value set so implementers know ONET-SOC is the preferred vocabulary while
other systems remain conformant.

#### Justification

A composed value set keeps the profile differential simple (one binding, no
slices), avoids the validator complexity that discriminator-on-system slicing
sometimes introduces, and keeps the door open for additional code systems
without future StructureDefinition churn. It also concentrates the
terminology decision in a reusable ValueSet artifact that other military /
veteran-health IGs can adopt. Trade-off vs. A: implementers lose the
explicit, machine-readable signal that "this slice is the ODH slice", which
some downstream profiling tools rely on.

---

### Disposition C: Decline

#### Proposal

Decline the request and instruct implementers who need to record ODH
occupation data to use the existing **ODH Occupation** profile from the ODH
IG directly (referenced or contained from the USMS bundle) rather than
extending the USMS occupation element's binding.

#### Justification

The ODH IG already defines a fully specified Occupation observation with the
ONETSOC binding; duplicating that binding inside USMS risks two profiles
drifting out of sync and forces USMS maintainers to track ODH value-set
versioning. Declining is defensible on a "single source of truth /
out-of-scope" basis. However, this disposition was effectively rejected by
the work group when the ticket was resolved Persuasive, and the reporter (an
ODH/NIOSH SME) considered it inadequate for military-service capture
workflows.

---

### Recommendation

**Recommended disposition:** A

The work group already settled this on substance — the ticket was resolved
Persuasive under Block-Vote-2 and the reporter committed to updating the IG
— so any further review should be a confirmation ballot rather than a
re-litigation. Disposition A is the literal commitment made at resolution
time, uses the standard FHIR slicing pattern that other NIOSH/ODH-aligned
IGs already follow, and keeps the existing USMS occupation codes intact
while letting implementers contribute ODH-coded data. Disposition B is a
reasonable engineering refinement and could be folded in as an
implementation detail (a composed value set behind one of the slices), but
it should not delay landing the change. Disposition C is not viable: it
contradicts the work group's earlier decision and ignores the SME input
that motivated the ticket.
