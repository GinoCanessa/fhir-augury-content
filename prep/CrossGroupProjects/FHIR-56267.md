## Summary

This US Core ballot tracker requests a sweeping editorial pass that replaces lower-case conformance verbs ("may", "can", "are not required", "only need to", etc.) with the formal, RFC 2119–style **MAY** in 13 specific narrative passages across the US Core IG. The reporter (Eric Haas) frames the rationale as: using **MAY** "creates a formal, testable optional capability" rather than a casual prose hint. The 13 affected passages span profile-specific implementation guidance for US Core DiagnosticReport-Note, MedicationDispense / MedicationRequest, Observation Clinical Result, Pulse Oximetry, Patient (including the Project US@ note), Practitioner, ServiceRequest, Specimen, AllergyIntolerance, plus screening-and-assessments category usage, clinical-notes support requirements, scopes/SMART scopes, and the DocumentReference / DiagnosticReport-Note "minimum supported categories" guidance. Several edits also reword the surrounding sentence so that **MAY** governs an actual server/client capability statement (e.g., "Systems **MAY** choose not to support `Observation.component.dataAbsentReason`"), rather than just substituting the verb in place. No related Jira tickets are surfaced by cross-references; the change is companion to the broader Block Vote 3 conformance-language cleanup announced in the `[argonaut] US Core V9 Ballot Reconciliation` Zulip thread (which also bundles FHIR-56272, FHIR-56271, FHIR-56261, FHIR-56260, FHIR-56246, FHIR-56244, FHIR-56221, FHIR-56217). Resolution on the ticket is already marked **Persuasive**, and a commit (`93f97b7…` "FHIR-56267 FHIR-56261 Update the what's new section") shows the change has begun being applied in `HL7/US-Core`.

- *Rationale:* use **MAY** to create a formal, testable optional capability.

- *Note:* URL where the narrative is listed in parentheses.

1. Other [diagnostic report] categories ~~may~~ ***MAY*** be supported [when using US Core DiagnosticReport Profile for Report and Note Exchange].
   (`StructureDefinition-us-core-diagnosticreport-note.html#profile-specific-implementation-guidance`)

2. When using a code [to represent a medication for a medication dispense], USCDI recommends NDC as an *optional* terminology. They ~~can~~ ***MAY*** be supplied as an additional coding element.
   (`StructureDefinition-us-core-medicationdispense.html#profile-specific-implementation-guidance`, `…-medicationrequest.html#…`)

3. Systems that never provide a component observation without a component value ~~are not required~~ ***MAY choose not*** to support `Observation.component.dataAbsentReason`.
   (and the parallel sentence about "without a value")
   (`StructureDefinition-us-core-observation-clinical-result.html#profile-specific-implementation-guidance`)

4. Inspired oxygen therapy ~~may~~ ***MAY*** be represented with [component] observations when measured at the same time as the pulse oximetry measurements [in the US Core Pulse Oximetry profile].
   (`StructureDefinition-us-core-pulse-oximetry.html#profile-specific-implementation-guidance`)

5. For new and updated records, Certifying Systems **SHALL** and non-Certifying Systems **SHOULD** enable users entering patient addresses to follow the Project US@ Technical Specification for Patient Addresses Final Version 1.0, which is required by the USCDI Patient Demographics Data Class (CONF-0469, CONF-0470).
   - Consult that specification for the format of Address datatype elements, especially `Patient.address.line` and `Patient.address.city`.
   - Note: ***Both certifying and non-certifying systems MAY choose not to use Project US@ for h***istorical records or documents that are not exposed through FHIR-based APIs (CONF-0471).
   (`StructureDefinition-us-core-patient.html#profile-specific-implementation-guidance`)

6. Although `Practitioner.address` is marked as Must Support, the Server system***s*** ~~is not required~~ ***MAY choose not*** to support it if they support the US Core PractitionerRole Profile.

7. The Must Support `ServiceRequest.category` is bound, at a minimum, to the [US Core ServiceRequest Category Codes], and other category codes ~~can~~ ***MAY*** be used.
   (`StructureDefinition-us-core-servicerequest.html#profile-specific-implementation-guidance`)

8. Since the binding for `Specimen.type` is extensible, when a code is unavailable ***servers MAY use*** ~~just~~ text ~~is allowed~~.

9. *No Known Allergies* ~~may~~ ***MAY*** be represented using the US Core-AllergyIntolerance profile (CONF-0305) with the appropriate negation code in `AllergyIntolerance.code` (CONF-0306) and the appropriate verification status in `AllergyIntolerance.verificationStatus` (CONF-0307).
   (`StructureDefinition-us-core-allergyintolerance.html#profile-specific-implementation-guidance`)

10. API consumers ~~can~~ ***MAY*** query by category when accessing patient information. Each profile's *Quick Start* section shows searches by category if the category element is defined in the profile. US Core defined these USCDI Health Assessments Data Elements category codes…
    (`screening-and-assessments.html#category-codes`)

11. …a Client ~~can~~ ***MAY*** determine the note and report types supported by a Server by invoking the standard FHIR Value Set Expansion `[$expand]` operation defined in the FHIR R4 specification.
    (`clinical-notes.html#support-requirements`)

12. Implementations meeting US EHR certification requirements must support all US Core's required scopes (CONF-0124). Other systems ~~only need to~~ **MAY** support scopes for the US Core APIs they support (CONF-0125).
    (`scopes.html#smart-scopes`)

13. This guide requires systems to implement the US Core DocumentReference Profile (CONF-0223) and to support a minimum of all ten Common Clinical Notes listed above. Systems ~~may~~ ***MAY*** extend their capabilities to the complete US Core DocumentReference Type Value Set (CONF-0222). Likewise the IG requires the DiagnosticReport-Note profile and a minimum of three report categories (Cardiology LP29708-2, Pathology LP7839-6, Radiology LP29684-5); Systems ~~may~~ ***MAY*** support other categories as well (CONF-0225, CONF-0226).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the 13 narrative substitutions exactly as enumerated in the ticket Description, in `HL7/US-Core` under `input/pagecontent/` (the `StructureDefinition-us-core-*.md` files for diagnosticreport-note, medicationdispense, medicationrequest, observation-clinical-result, pulse-oximetry, patient, practitioner, servicerequest, specimen, allergyintolerance, plus `screening-and-assessments.md`, `clinical-notes.md`, and `scopes.md`). Render the conformance verb in **bold MAY** (no italics) per the reporter's "only conformance verbs will be Bold in final text" note. Where the substitution requires rephrasing the surrounding clause to make **MAY** govern an actual server/client capability (items 3, 5, 6, 8, 12), apply the rephrasing as shown, and add a brief change-log entry to `input/pagecontent/changes.md` under the v9.0.0 reconciliation section (already partially applied in commit `93f97b7`). Mark the change impact as **Non-substantive (Clarification)** to match the Block Vote 3 categorization. Also confirm the linked CONF-IDs (CONF-0124/0125/0222/0223/0225/0226/0305/0306/0307/0469/0470/0471) still resolve to existing requirement anchors after the rewrite — those anchors are referenced by the new text and must continue to exist.

#### Justification

The change is purely editorial: it elevates loose verbs ("may", "can", "is not required", "only need to") to the RFC 2119 keyword **MAY**, which US Core already imports and uses elsewhere in the same prose. Doing so produces formal, testable optional capabilities (the reporter's stated rationale) and aligns these passages with the rest of US Core's conformance vocabulary and with the new Narrative Requirements Tables that drove Block Vote 3. The Resolution is already **Persuasive**; the Cross-Group Projects ballot thread bundles the change into Block Vote 3 alongside FHIR-56272/56271/56261/56260/56246/56244/56221/56217 with no withdraw requests recorded; and the change has already begun being applied to the change-log via commit `93f97b7`. Sibling tickets in the same block (e.g., FHIR-56217 "Change 'can' to 'MAY' …") attest to the same pattern being applied consistently across the IG.

---

### Disposition B: Accept With Modification — Split Verb Substitution from Capability Rephrasing

#### Proposal

Apply items that are pure verb swaps (1, 2, 4, 7, 9, 10, 11, 13) as written. For items that *also* rephrase the sentence so that **MAY** governs a new capability shape — specifically:

- Item 3: ~~are not required~~ → ***MAY choose not*** to support `Observation.component.dataAbsentReason`
- Item 5: ~~may not meet this requirement, and systems~~ → ***MAY choose not to use Project US@ for h***istorical records (CONF-0471)
- Item 6: ~~is not required~~ → ***MAY choose not*** to support `Practitioner.address`
- Item 8: ~~just text is allowed~~ → ***servers MAY use*** text
- Item 12: ~~only need to~~ → **MAY** support scopes for the US Core APIs they support (CONF-0125)

…apply the verb swap but keep the rephrased sentence as a follow-up review for the workgroup, so reviewers can confirm that "MAY choose not" carries the same meaning as the prior "is not required" and that CONF-0471 / CONF-0125 / Must-Support semantics for `Practitioner.address` are not weakened. Document the per-item disposition (split into two trackers if helpful, or capture the carve-out as a comment on FHIR-56267).

#### Justification

Five of the thirteen edits do more than swap a verb — they change "is not required" / "are not required" / "only need to" into a positive **MAY** statement. While the *intent* (this is optional) is preserved, "MAY choose not to support X" interacts with the FHIR Must-Support definition (especially for `Practitioner.address`, which is explicitly Must Support) and with the certification-grounded CONF-0471 / CONF-0125. The workgroup may want a deliberate read of those five edits even if the other eight ride through Block Vote 3 unchanged. This preserves the reporter's intent and Block Vote 3 momentum while adding a small safety net for the items that touch certification language.

---

### Disposition C: Decline

#### Proposal

Do not apply the 13 substitutions in v9.0.0. Defer the cleanup to a future US Core errata or to a coordinated cross-spec conformance-language pass, on the grounds that it is purely editorial and not required for a publication-blocking issue.

#### Justification

A defensible decline rationale would be (a) the change is non-substantive and could slip to a 9.0.x technical correction without affecting interoperability, (b) bundling many small narrative edits into a publication carries a non-zero risk of introducing typos or breaking CONF-anchor resolution in `requirements.html` (CONF-0469, CONF-0470, CONF-0471, CONF-0305–0307, CONF-0124, CONF-0125, CONF-0222–0226), and (c) replacing "is not required" with "MAY choose not to support" near Must-Support elements (`Observation.component.dataAbsentReason`, `Practitioner.address`) opens a re-litigation of the Must-Support contract that is out of scope for an editorial cleanup. Declining preserves v9.0.0's narrative as it shipped to ballot.

### Recommendation

**Recommended disposition:** **A — Accept As Requested.**

The ticket is explicitly part of US Core v9.0.0's Block Vote 3 conformance-language cleanup, the reporter's rationale (formal, testable optional capability via RFC 2119 **MAY**) is exactly the standards-grounded reason US Core already imports the keyword, and the resolution is already marked Persuasive in Jira. The Cross-Group Projects ballot-reconciliation thread shows no withdraw requests against FHIR-56267, and a commit (`93f97b7`) has already begun applying the change. Disposition B's caution about the five "rephrase" items is reasonable but better handled inline during application (a brief reviewer pass on items 3, 5, 6, 8, and 12 before merge) than by carving them out of the block vote and slowing publication. Disposition C would leave the IG with two parallel conformance vocabularies in adjacent paragraphs, which is exactly the inconsistency the Narrative Requirements Tables work is trying to remove. Apply A; raise any item-3/5/6/8/12 wording concerns as in-line comments on the application PR rather than as a new tracker.
