## Summary

The Observation ADI Documentation Profile in US Core lists a `patient + category + date` combination as a **SHALL**-supported search parameter combination on its Mandatory Search Parameters table. The `date` search parameter on `Observation` is bound to `Observation.effective[x]`, but in the ADI Documentation profile that element carries no Must Support, USCDI, or cardinality requirement — relevant date information for these documentation observations lives on the linked `DocumentReference` resources instead. The reporter (James Jahns) argues that mandating support for a search parameter against an element that the profile does not require servers to populate is unclear: a conformant server could "support" the parameter by returning zero results whenever it is supplied, which is technically valid but probably not the intent. He proposes either (a) removing the `date`-bearing combination from the SHALL-support list, or (b) adding narrative that explains the intent (for example, that `date` SHALL be supported at the Observation endpoint level even when this profile does not require the underlying element to be populated).

The cross-reference index surfaces no related Jira tickets, Zulip threads, or GitHub items for this ticket; the only structured cross-reference is to the FHIR artifact `Observation.effective`, which is the element at the center of the issue. The April 9, 2026 CGP call (recorded in comments) reached working-group consensus that requiring this search parameter on a profile where the underlying element is not Must Support does not make sense.

## Details

**Description:**

For the Observation ADI Documentation Profile, under Mandatory Search Parameters, it says:

> **SHALL** support searching using the combination of the **[`patient`](https://hl7.org/fhir/us/core/SearchParameter-us-core-observation-patient.html)** and **[`category`](https://hl7.org/fhir/us/core/SearchParameter-us-core-observation-category.html)** and **[`date`](https://hl7.org/fhir/us/core/SearchParameter-us-core-observation-date.html)** search parameters

However, the `date` parameter corresponds to `Observation.effective[x]`, which has no MS / USCDI / cardinality requirements on this profile. Because of this, the search parameter requirements here are unclear. This might be a technical correction or a change request depending on what is intended.

I assume that `Observation.effective[x]` intentionally has no requirements here; given the nature of this profile, relevant date information would be found within the linked DocumentReference(s) anyway. Given that, mandatory support for this parameter is a bit unusual. Servers that don't return `Obs.effective[x]` for this profile could certainly "support" the parameter by returning no results whenever the parameter is provided, which is technically correct, but I'm not sure if that's what US Core intends.

My suggestion is to remove this combination of parameters from the SHALL support list, or to add additional narrative guidance to explain the intent. (I understand that, in some sense, this parameter SHALL be supported for the Observation search endpoint as a whole, and thus all profiles, even if it filters out all results in this case; if that is the point, it would be helpful to be explicit about it.)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Core Observation ADI Documentation Profile (`StructureDefinition-us-core-observation-adi-documentation`), edit the **Mandatory Search Parameters** section so that the SHALL-support combination is reduced to `patient + category` and the `date` parameter is removed from the required combination for this profile specifically. The `date` search parameter remains supported at the Observation endpoint level (per other US Core Observation profiles), but is no longer a mandatory part of the combined search for the ADI Documentation profile, where `Observation.effective[x]` is not Must Support. Update the profile's CapabilityStatement-driven search-parameter table accordingly so the rendered IG matches.

#### Justification

This is the cleanest way to remove the contradiction the reporter identified: it is incoherent to require a search-parameter combination keyed on an element that the same profile does not require servers to populate. The April 9, 2026 CGP call reached this conclusion explicitly ("WG agrees that `Observation.effective[x]` is not MS … and it does not make sense to have a required search parameter on that element"). Removing the combination aligns the conformance contract with the profile's actual data shape and avoids the absurd-but-conformant pattern of "support" by always returning empty results. Per cooper.thompson's clarification, this can ride a normal US Core release while USCDIv3 remains the certification floor and does not require an immediate US Core Patch.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the `patient + category + date` combination in the SHALL-support list for consistency with other US Core Observation profiles, but add explicit narrative to the Observation ADI Documentation profile's Mandatory Search Parameters section clarifying that: (1) the `date` search parameter is required at the Observation endpoint level for this profile, (2) servers are not required to populate `Observation.effective[x]` on instances conforming to this profile, and (3) when `Observation.effective[x]` is absent, a server's correct behavior in response to a `date`-bounded query is to exclude that instance from the result set. Optionally cross-reference DocumentReference date elements as the place where authoritative date information for ADI documentation lives.

#### Justification

This preserves uniformity across US Core Observation profiles — every other Observation profile carries the `patient + category + date` combination, and divergence here forces implementers to special-case the ADI Documentation profile in their server configuration and conformance test suites. The reporter himself flags this as a viable alternative ("…or to add additional narrative guidance to explain the intent"). The cost is that the conformance contract remains weaker than it appears at first read, and reviewers must trust the narrative to understand how `date` actually behaves on this profile. It is also more verbose than Disposition A and depends on implementers reading prose rather than relying on the structured search-parameter table.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing text already binds servers to support `date` for the Observation endpoint as a whole, and the lack of Must Support on `Observation.effective[x]` does not, on its own, prevent that endpoint-level conformance. Treat the ticket as not actionable and close as "Persuasive — Not Persuasive" / "Working as Designed".

#### Justification

This disposition is hard to defend in light of the working group's own April 9 CGP-call consensus, which already agreed with the reporter. The only situation in which Decline would be appropriate is if CGP later concludes that the existing endpoint-level requirement is sufficient and that no profile-specific change or narrative is warranted; in that case the ticket can be closed with a comment pointing to the endpoint-level conformance language. As written today, however, the reporter's analysis is correct and a "no change" resolution would leave the documented contradiction in place.

---

### Recommendation

**Recommended disposition:** A

The April 9, 2026 CGP call has already reached working-group consensus that `Observation.effective[x]` is not Must Support on the Observation ADI Documentation Profile and that requiring a search parameter against it does not make sense — Disposition A is the literal codification of that consensus. It eliminates the contradiction at the structural level rather than papering over it with narrative, which is more defensible to implementers building conformance test suites and certification harnesses. Disposition B is a reasonable fallback if CGP later decides that cross-profile uniformity of the search-parameter table outweighs the inconsistency, but it leaves implementers with a profile whose machine-readable conformance is stricter than its real-world expectations. Per cooper.thompson, this can land in a normal US Core release without invoking the US Core Patch process, since USCDIv3 remains the certification floor and there is runway before USCDIv5 would force the issue.
