## Summary

The ticket asks the US Military Service IG maintainers to constrain the
`military-service-veteran-status` extension's `Context` so that it can
only be applied to `Patient` (and optionally `Practitioner` and
`RelatedPerson`). As published in STU1 the extension declares its
context as the abstract type `Element`, which means it is
syntactically valid on any element of any resource or data type
(e.g., `Condition.status`, `Address.postalCode`). This is almost
certainly an authoring error: "veteran status" only has semantic
meaning for an individual, so the extension should be scoped to the
resources that represent individuals.

There are no other Jira tickets cross-referenced to FHIR-43110 in the
Augury index, but several Zulip discussions confirm the issue is real
and is being encountered by implementers in the wild — see the
"Related Zulip Discussions" section below.

## Details

**Description:**

> The veteran-status extension has a context of 'Element', meaning it
> can appear on any resource or data type element, which makes no
> sense. This should be constrained to having a context of Patient.
> (Or if you wished, possibly Practitioner and RelatedPerson too.)

**Comments:**

No comments have been recorded on the Jira ticket.

## Related Zulip Discussions

### [implementers > Extension:Context](https://chat.fhir.org/#narrow/stream/implementers/topic/Extension%3AContext/near/433346649)

Syed Junaid asked (April 2024) whether the `military-service-veteran-status`
extension could be applied to a `Person` resource, given its declared
context of "Element". Yunwei Wang confirmed that, as written, the
extension is valid on any element of type `Element`. Lloyd McKenzie
explicitly stated that "the extension was improperly defined as being
valid everywhere" — i.e., it could be stuck on `Condition.status` or
`Address.postalCode` — and noted the silver lining is that this means
there is currently no conformance problem with using it on `Person`.
This thread is the clearest community confirmation of the bug
FHIR-43110 is reporting and signals that any tightening of the context
will be a breaking change for at least some implementers.

Participants: Syed Junaid, Yunwei Wang, Lloyd McKenzie

### [implementers > Extension](https://chat.fhir.org/#narrow/stream/implementers/topic/Extension/near/321827258)

A long-running thread in which implementers ask whether single-resource
extensions (such as `patient-religion` and `patient-importance`) can be
reused on `Person`/`Practitioner`. Eric Haas explicitly recommends
filing a FHIR tracker to broaden the context of `patient-religion` to
"other individuals like Person, Practitioner, and RelatedPerson",
exactly mirroring the secondary suggestion in FHIR-43110. Lloyd
McKenzie reinforces that using an extension outside its declared
context produces a non-conformant instance. The same reasoning applies
to `military-service-veteran-status`: the *correct* fix is to define
the context to match the populations the extension is meaningful for.

Participants: Syed Junaid, Eric Haas, Jean Duteau, Lloyd McKenzie,
Yunwei Wang, Mareike Przysucha, Ioana Singureanu, vivek nayak, Deepa
Shahi, David Pyke

### [implementers > Problematic extensions](https://chat.fhir.org/#narrow/stream/implementers/topic/Problematic%20extensions/near/283334992)

Piotr Sokoliński reported (May 2022) that the validator only emits an
*information* (not an error) for a `valueBoolean` use of
`http://hl7.org/fhir/us/military-service/StructureDefinition/military-service-veteran-status`
on `Patient`, which is consistent with the fact that the extension's
context is so loose it cannot tell whether the placement is
appropriate. Grahame Grieve clarifies that the validator only treats
unknown extensions outside the `hl7.org/fhir` namespace as
informational. While the thread is mostly about validator behaviour,
it illustrates how the loose context defeats validator tooling that
implementers rely on.

Participants: Piotr Sokoliński, Grahame Grieve

### [implementers > individual-pronoun](https://chat.fhir.org/#narrow/stream/implementers/topic/individual-pronoun/near/443994085)

A short tangential thread in which the `military-service-veteran-status`
extension appears as an example of an extension being placed on
`Patient` alongside `individual-pronouns`. Not directly about the
context bug, but confirms that real implementations are using the
extension on `Patient`, which is the intended target of the proposed

Participants: Syed Junaid

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify the `military-service-veteran-status` `StructureDefinition` in
the US Military Service IG so that `StructureDefinition.context` is
replaced with a single entry:

```
context: [
  { type: "element", expression: "Patient" }
]
```

Bump the extension's `version` and re-publish the IG (next STU
ballot or technical correction, as the WG prefers). Add a short note
in the IG narrative explaining that the extension is intended for
patient instances and that prior usage on non-`Patient` elements,
while syntactically tolerated by FHIR-43110-era publications, is
considered an authoring error.

#### Justification

This matches the literal request in the ticket and aligns with FHIR's
authoring guidance that extension `Context` should describe the
elements where the extension *makes sense*, not the most permissive
type. The Zulip "Extension:Context" thread confirms the current
context is widely viewed as a defect even by the original requester
(Lloyd McKenzie) and a recognised expert (Yunwei Wang). Restricting
to `Patient` alone is the narrowest fix and the one most consistent
with the IG's stated focus on patient veteran status. Implementers
that have placed the extension on other resources are, by Jean
Duteau's and Lloyd's repeated guidance in the related Zulip threads,
already non-conformant in spirit, so the change clarifies rather than
re-introduces a divergence.

---

### Disposition B: Alternative Approach

#### Proposal

Set the extension's context to the union of the FHIR resources that
represent an individual person, i.e.:

```
context: [
  { type: "element", expression: "Patient" },
  { type: "element", expression: "Practitioner" },
  { type: "element", expression: "RelatedPerson" },
  { type: "element", expression: "Person" }
]
```

This is the parenthetical alternative Lloyd offered in the ticket
("Or if you wished, possibly Practitioner and RelatedPerson too"),
extended with `Person` because the most-cited Zulip use case
(`Extension:Context`, April 2024) is exactly an implementer trying
to record veteran status on a `Person` instance.

#### Justification

The "veteran status" concept is meaningful for any individual
represented in FHIR, not only patients — for example, a clinician's or
contact's veteran status may be tracked for benefits, eligibility, or
program-membership reasons. The Zulip evidence shows real
implementers attempting exactly these placements. Broadening the
context preserves backward compatibility for those instances while
still removing the absurd cases (extension on `Address.postalCode`,
`Encounter.dischargeDisposition`, etc.). Eric Haas applied the same
reasoning to `patient-religion` in the "Extension" thread and
explicitly recommended filing a tracker to broaden that extension's
context — the same principle applies here. Cost is small: it is one
extra line in the `StructureDefinition` and a slight broadening of the
narrative, and the fix lives entirely in the
`HL7/fhir-military-service` IG (no Core or US Core change required).

---

### Disposition C: Decline

#### Proposal

Leave the extension's `Context` as `Element` and instead add a
narrative note to the extension's description in the IG explaining
that the extension is "intended for use on individuals (e.g.,
`Patient`)" but is technically tolerated elsewhere. Close FHIR-43110
as "Will not fix — narrative clarification only".

#### Justification

A defensible-but-weak rationale for declining is backward
compatibility: any instance that currently carries the extension on a
non-individual element (or on `Person`/`Practitioner`/`RelatedPerson`)
is already valid against STU1 and would become invalid if the context
is tightened to `Patient`. The April-2024 "Extension:Context" thread
shows at least one implementer is actively using the extension on a
non-`Patient` resource, and Lloyd's own comment there points out the
upside that "there's no problem using it on Person". Declining the
change in favour of a narrative warning preserves those implementations
and keeps the IG's published binary surface stable. This rationale is
weaker than Dispositions A or B because it leaves the underlying
authoring defect in place and continues to defeat validator tooling
(see the "Problematic extensions" thread).

---

### Recommendation

**Recommended disposition:** B

Disposition B (broaden the context to `Patient`, `Practitioner`,
`RelatedPerson`, and `Person`) is the strongest of the three
options. It cleanly fixes the authoring defect identified in the
ticket — closing off the absurd placements on non-individual elements
that drove FHIR-43110 — while preserving the only real-world non-
`Patient` use cases the Zulip record actually shows (most notably the
"Extension:Context" thread, where the implementer wanted to put
veteran status on a `Person`). It is also the option Lloyd flagged as
acceptable in the ticket itself and is consistent with how the wider
community has reasoned about analogous extensions (`patient-religion`,
`individual-pronouns`). Disposition A is defensible but
unnecessarily narrow given the evidence; Disposition C does not
actually address the bug and just papers over it with narrative.
Implementation lives entirely in the `HL7/fhir-military-service` IG,
so the change is low-risk and contained, and the WG can sequence it as
a technical correction or roll it into the next STU update at its
discretion.
