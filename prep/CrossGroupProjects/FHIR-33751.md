## Summary

The ticket asserts that the `military-service-paygrade-vs` Value Set in the
US Military Service FHIR IG has its canonical URL incorrectly rooted under
`hl7.org/fhir` and should instead be rooted under `terminology.hl7.org`
(THO). The reporter (Ron G. Parker) further requests that a companion UTG
ticket be opened to register the Value Set in THO so the canonical URL
resolves to a curated, governed terminology artifact.

The ticket is one of a cluster of near-identical "Defining URL (Canonical
URL) should be in THO not hl7.org/fhir.N" change requests filed by the same
reporter against the same IG, each targeting a different terminology
artifact. The closely-related siblings (see "Related Jira Tickets" below)
cover the IG's Code System (`military-service-solor-extension`) and the
other Value Sets (`military-service-branch-vs`, etc.). Together they
represent a single editorial position about where authoring of HL7
terminology artifacts should live.

The ticket has already been triaged and resolved as **Considered for
Future Use**, with the workgroup noting that the THO-rooting policy "is
apparently part of FHIR 5… not expressly stated in FHIR 4" and that votes
in this series would be treated as **non-persuasive** at the time. This
review re-examines the ticket so the workgroup can decide whether to
revisit that disposition (e.g., now that a future ballot of the IG is on
the table) or leave it as-is.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-open the ticket and adopt the requested change for the
`military-service-paygrade-vs` Value Set in the US Military Service FHIR
IG:

1. Change the Value Set's `ValueSet.url` from
   `http://hl7.org/fhir/us/military-service/ValueSet/military-service-paygrade-vs`
   (or whichever `hl7.org/fhir/...` form it currently uses) to
   `http://terminology.hl7.org/ValueSet/military-service-paygrade-vs`
   (final segment to be confirmed against existing THO conventions).
2. File a companion UTG ticket against **HL7/UTG** to register the Value
   Set under THO with the new canonical URL, including the existing
   compose definition and any required metadata (status, experimental,
   jurisdiction, copyright).
3. Update all in-IG references (profiles' `binding.valueSet`, narratives,
   markdown links) to point at the new THO canonical.
4. Apply the same treatment in lockstep to the six sibling tickets
   (FHIR-33749, -33750, -33752, -33753, -33754, -33755) so the IG's
   terminology artifacts move to THO together rather than piecemeal.

#### Justification

This matches the editorial direction that became explicit in FHIR R5 and
in HL7's overall terminology-governance posture: terminology artifacts
authored by HL7 belong in THO, not under per-IG `hl7.org/fhir/...` paths.
The reporter is correct on the policy direction; the original "Considered
for Future Use" disposition deferred on timing, not on principle. Doing
all seven IG artifacts at once preserves internal consistency of the IG
and avoids two cohorts of canonicals coexisting indefinitely.

---

### Disposition B: Alternative Approach

#### Proposal

Leave the canonical URL of `military-service-paygrade-vs` **unchanged** in
the currently published version of the IG, but bind the migration to the
IG's next substantive republication:

1. Add an editorial note to the IG (and to UTG planning) that, on next
   republish, all seven `military-service-*` Code System / Value Set
   artifacts will move their canonicals to THO and will be registered via
   UTG.
2. At republish time, perform a coordinated rename: publish the THO
   artifact first, then update the IG to bind to the new THO canonical,
   then retire (or alias / mark deprecated) the old `hl7.org/fhir/...`
   canonical.
3. Track this as a single umbrella tracker item that supersedes the seven
   per-artifact tickets, so the workgroup can disposition the cluster
   once instead of seven times.

#### Justification

Canonical URL changes are not silent: any downstream profile,
ConceptMap, NamingSystem, or implementation that has hard-coded the old
URL will break the moment the URL flips. Pinning the migration to a
republication boundary lets the IG document the change in its release
notes, lets THO host the artifact before the IG retargets it, and gives
implementers a defined window. This honors the original "non-persuasive
at this time" framing — the policy is right, but executing it
mid-cycle creates breakage for little benefit.

---

### Disposition C: Decline

#### Proposal

Maintain the existing disposition: **Considered for Future Use /
non-persuasive**. Do not change the canonical URL of
`military-service-paygrade-vs`, do not file a UTG registration ticket,
and apply the same outcome consistently to the six sibling tickets in
this series. Document in the workgroup's response that the THO-rooting
policy was not normative at the time the IG was published and that the
IG's authors retained authority to root the artifact under
`hl7.org/fhir/us/military-service/...`.

#### Justification

The original disposition is defensible on the FHIR R4 record: there is
no R4 normative requirement that HL7-authored terminology artifacts be
rooted in THO. Re-litigating each per-artifact ticket six years later,
with no implementer pressure surfacing in the cross-reference, Zulip, or
GitHub indexes, has limited workgroup ROI. If the project ever
republishes the US Military Service IG, the THO-rooting decision can be
revisited then as a forward-looking design choice rather than a
retroactive correction.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach)

Disposition A captures the right end-state — HL7-authored terminology
belongs in THO — but doing the URL flip *now*, on a long-quiet IG, is
the kind of change that silently breaks downstream bindings and
ConceptMaps without any compensating implementer benefit. Disposition C
preserves the status quo at the cost of leaving a known editorial
inconsistency on the books indefinitely. Disposition B threads the
needle: it commits the workgroup to the THO-rooting outcome the reporter
asked for, but binds execution to the IG's next republish so that THO
registration (via UTG) and the IG's binding updates land together. It
also collapses the seven sibling tickets into a single umbrella decision,
which matches how the cluster was actually filed and how it will
realistically be implemented. The workgroup should re-disposition all
seven tickets as "Persuasive with Modification — defer execution to next
IG republish, track via umbrella" rather than letting the original
"non-persuasive at this time" stand without a forward path.
