## Summary

The ticket reports a navigation/usability defect on the **US Vital Signs with Qualifying Extensions** IG (2020-Sep snapshot). On the IG's **Profiles & Extensions** index page, the hyperlink for each extension (e.g., `Associated Precondition`) targets the extension's *Detailed Descriptions* page (`StructureDefinition-AssociatedPreconditionExt-definitions.html`) instead of the extension's main overview page (`StructureDefinition-AssociatedPreconditionExt.html`). Landing on the detailed-descriptions page leaves the user without the usual tabs/links to the rest of the StructureDefinition (overview, narrative, snapshot/differential, mappings, etc.). The reporter additionally observes that the link is rendered as an "external" link even though the target is a profile/extension defined within the same IG, and asks that the rendering be corrected so the link is treated as internal. No comments and no cross-references (Jira, Zulip, Confluence, or GitHub) were found via FHIR Augury, so this report relies solely on the ticket text.

## Details

**Description:**

The links to extensions from the Profiles & Extensions page are to the Detailed Descriptions for the extension, rather than the overall page. For example, the link for Associated Precondition is:

http://hl7.org/fhir/us/vitals/2020Sep/StructureDefinition-AssociatedPreconditionExt-definitions.html

This puts the user on the detailed descriptions of the elements in the profile, but in a state where the rest of the structure definition cannot be accessed.

Consider linking to the overview page of the structure definition:

http://hl7.org/fhir/us/vitals/2020Sep/StructureDefinition-AssociatedPreconditionExt.html

Also, consider making sure that the link is not defined as external, as it is to a profile defined in this implementation guide.

## Proposed Dispositions

> Note: With no GitHub cross-references and no repo briefings to anchor file-path-level proposals, the dispositions below are scoped to the *kind* of change required (IG-publisher / template / source markdown for the Vital Signs IG) rather than to specific files. ### Disposition A: Accept As Requested

#### Proposal

In the next publication of the **US Vital Signs with Qualifying Extensions** IG, fix the **Profiles & Extensions** index page so each extension row links to the StructureDefinition overview page (e.g., `StructureDefinition-AssociatedPreconditionExt.html`) rather than the `-definitions.html` variant, and ensure those links are emitted as internal links (no `class="external-link"`, no `target="_blank"`/`rel="nofollow noopener"`). Concretely: (1) audit the source of the Profiles & Extensions page in the IG (typically a hand-authored page or a `{% include list-simple-extensions.xhtml %}` / `list-extensions.xhtml` fragment generated by the IG Publisher) and replace any `*-definitions.html` hrefs with the bare `StructureDefinition-<id>.html` form; (2) if the page is generated from an IG-Publisher template, update the template to emit the overview href; (3) republish the IG so the corrected links propagate to the canonical (non-`2020Sep`) build.

#### Justification

The reported behaviour is unambiguously a defect: the detailed-descriptions page is a *secondary* tab on a StructureDefinition rendering and does not expose the overview/narrative/differential/snapshot tabs in the same way the main page does. Pointing the index page at the overview is the convention used by other HL7 IGs (e.g., US Core, IPS) and by the FHIR core spec itself, so accepting the request restores consistency with established IG-Publisher norms. Removing the "external" treatment is also straightforward and correct — both pages live under the same IG canonical (`hl7.org/fhir/us/vitals/...`), so there is no reason to mark the link as external.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hand-fixing the IG-source Profiles & Extensions page, raise this as a downstream fix in the **IG Publisher / IG template** project so that the auto-generated extension list always links to the StructureDefinition overview page and never to the `-definitions.html` page, and emits internal-link markup. Apply the resulting Publisher/template fix to US Vital Signs by re-running the publisher against the latest sources (no IG-source change needed beyond a republish). File a tracking note on the US Vital Signs IG referencing the upstream fix, and close FHIR-28538 once the next Vital Signs publication picks up the corrected output.

#### Justification

The same defect is very likely present in *any* IG that renders a Profiles & Extensions index from the same Publisher/template fragment. Fixing it once at the Publisher/template layer prevents regressions in other IGs and avoids per-IG hand-edits to generated content. It also avoids the risk of the Vital Signs IG re-introducing the bug on the next clean republish if the IG source page is later regenerated from the template. The trade-off is timing — a Publisher fix has its own release cadence and may delay user-visible resolution compared to Disposition A.

---

### Disposition C: Decline (Persistent Versioned Snapshot)

#### Proposal

Decline to change the `2020Sep` snapshot specifically (it is a frozen, versioned publication) and treat FHIR-28538 as already-addressed if the current published Vital Signs IG already links to overview pages. The fix, if any is needed, should be applied only to the active development branch and the next published version; the `2020Sep` artefact would be left as-is per HL7's normal "do not edit published snapshots" practice.

#### Justification

HL7 IGs treat dated snapshot publications (`/2020Sep/`, `/2021Jan/`, etc.) as immutable historical artefacts. Editing the `2020Sep` HTML in place would diverge from HL7 publishing convention and mask the historical state. If the current (undated/canonical) Vital Signs publication already routes the Profiles & Extensions links to the overview page, the ticket can be closed as "Persistent" / "Already Addressed". If it does not, Disposition A or B applies — but only against the current sources, not the snapshot.

---

### Recommendation

**Recommended disposition:** B (with a Disposition-A-style follow-through if the Publisher/template fix is non-trivial or slow to land).

The defect description is clearly about IG-Publisher-generated index behaviour (linking to `-definitions.html` and emitting "external"-style markup for in-IG links), and the same pattern appears across many HL7 IGs that use the standard Profiles-and-Extensions index. Fixing it once in the IG-Publisher / template layer is the highest-leverage change, prevents the bug from recurring on republishes, and benefits other IGs without per-IG hand-edits. If the Publisher fix is already in place in current builds, the workgroup should verify against the *current* US Vital Signs publication and either close as Persistent (Disposition C) or, if the bug still appears, apply the targeted IG-source fix (Disposition A) as an interim measure until the next clean republish picks up the upstream change. Disposition C alone is not recommended unless verification of the current publication confirms the issue is already gone.
