## Summary

Craig Newman reports that the Military Service Episode profile in the US
Military Service IG binds every `.component` slice value set with a binding
strength of **Required**. Because several of those value sets are
US-specific (e.g., PHINVADS-sourced lists) or otherwise narrowly scoped,
the Required strength prevents implementers in non-US jurisdictions from
using the profile without violating the binding. The submitter suggests
relaxing those bindings to **Extensible** or **Preferred** so the IG
remains useful outside the US while still steering implementers toward
the canonical code system.

The ticket sits inside a coordinated cluster of submissions Craig Newman
filed on 2021-09-08 against the same IG ([FHIR-33438](https://jira.hl7.org/browse/FHIR-33438),
[FHIR-33440](https://jira.hl7.org/browse/FHIR-33440), [FHIR-33441](https://jira.hl7.org/browse/FHIR-33441),
[FHIR-33434](https://jira.hl7.org/browse/FHIR-33434)), plus a follow-on
from Cooper Thompson ([FHIR-34109](https://jira.hl7.org/browse/FHIR-34109)).
Each sister ticket targets a specific component slice / value set on the
Military Service Episode (or its peer Deployment Episode) profile and
makes the same essential argument: the binding strength is too strict for
international use. All of the sister tickets are already **Published**
with **Persuasive** (or Persuasive with Modification) resolutions that
loosened the corresponding bindings, so this ticket is the last
unpublished member of an otherwise resolved cluster. The recorded
resolution on FHIR-33442 itself is "Will update binding," consistent with
the disposition pattern applied to its siblings.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the binding strength on **every** `.component` slice value set on
the Military Service Episode profile in the US Military Service IG from
**Required** to **Extensible** by default, falling back to **Preferred**
for value sets where Extensible cannot be justified (i.e., where the US
codes are genuinely not the only sensible choice). Concretely, for each
slice on the Military Service Episode profile that today carries
`binding.strength = required`:

1. Change `binding.strength` to `extensible` (or `preferred` where the
   value set is openly extensible in spirit).
2. Leave the `valueSet` reference unchanged so US implementers continue
   to receive the same guidance.
3. Update the IG narrative on the profile page to call out the new
   binding strength and note that non-US implementers may use codes
   outside the value set when an appropriate code is not present.

This mirrors what was done on the sister tickets ([FHIR-33438](https://jira.hl7.org/browse/FHIR-33438),
[FHIR-33440](https://jira.hl7.org/browse/FHIR-33440)) for individual
slices.

#### Justification

The submitter's argument is sound: PHINVADS-sourced US-specific value
sets bound at Required strength make the profile unusable elsewhere
without a profile-level break. Extensible preserves the intent — US
codes are the canonical answer when one exists — while admitting the
reality that international implementers will encounter codes the value
set does not enumerate. The workgroup has already accepted this exact
argument for the closely analogous sibling tickets ([FHIR-33438](https://jira.hl7.org/browse/FHIR-33438)
"Persuasive — Will update the binding accordingly", [FHIR-33440](https://jira.hl7.org/browse/FHIR-33440)
"Persuasive — Will update binding"), so accepting this ticket as
requested keeps the disposition pattern consistent across the cluster
and resolves the last open member.

---

### Disposition B: Alternative Approach — Per-slice Triage (mirroring FHIR-34109)

#### Proposal

Rather than blanket-loosening every `.component` slice on the profile,
perform a per-slice triage and pick the right binding strength for each
slice individually:

- Slices whose value set is openly extensible in spirit (e.g.,
  occupation, branch / service component) → **Extensible**.
- Slices whose value set is intentionally short and was always meant to
  be illustrative (the same situation that drove [FHIR-34109](https://jira.hl7.org/browse/FHIR-34109)
  to "Example") → **Example**, with a comment in the binding's
  description explaining why.
- Slices whose value set is genuinely closed and authoritative in the
  US (if any remain after analysis) → **Preferred**, with documentation
  of why a stricter strength is not appropriate for this IG's
  international audience.

Document each per-slice decision in the IG's binding rationale section
so future maintainers understand why the strengths differ.

#### Justification

The workgroup demonstrably has not always landed on the same answer
across this cluster: most siblings went to Extensible, but
[FHIR-34109](https://jira.hl7.org/browse/FHIR-34109) was disposed
**Persuasive with Modification** to **Example** because the underlying
value set was too short to be authoritative. That precedent says "binding
strength on the Military Service Episode component slices is not a
one-size-fits-all knob." A per-slice triage produces a more defensible
end state than the blanket change in Disposition A and avoids
overshooting on slices where Example is actually the honest answer. The
trade-off is more analysis work up front and a slightly more complex
ballot reconciliation for the IG editor.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that it is too coarse to act on as
written ("all of the `.component` slices") and that the workgroup has
already addressed the underlying concern through the per-slice tickets
that the submitter filed on the same day ([FHIR-33438](https://jira.hl7.org/browse/FHIR-33438),
[FHIR-33440](https://jira.hl7.org/browse/FHIR-33440)) and the follow-on
[FHIR-34109](https://jira.hl7.org/browse/FHIR-34109). Mark FHIR-33442
as a duplicate / superseded by those tickets and close.

#### Justification

The decline reading is that FHIR-33442 is essentially the umbrella
version of the per-slice tickets the same reporter filed simultaneously,
and that the workgroup's per-slice dispositions on those siblings have
already produced the requested change for the slices that warranted it.
A blanket "loosen everything" disposition risks regressing slices where
a stricter binding is correct, and the workgroup has no per-slice
analysis on FHIR-33442 to guide such a sweep. The risk in declining is
that a remaining slice (one not individually ticketed) keeps an
inappropriate Required binding indefinitely, and the published resolution
note ("Will update binding") becomes misleading.

---

### Recommendation

**Recommended disposition:** **B**

The recorded resolution on FHIR-33442 is already **Persuasive** ("Will
update binding"), so a clean Decline (Disposition C) is hard to defend
in light of the workgroup's prior signal. Disposition A is the
conservative read of that signal and matches what was done for the two
sibling Required→Extensible tickets — but the workgroup's own behavior
on [FHIR-34109](https://jira.hl7.org/browse/FHIR-34109) shows it is
willing, and sometimes prefers, to drop binding strength all the way to
**Example** when the underlying value set cannot carry Extensible
weight. A blanket Required→Extensible sweep would re-litigate that
decision badly.

Disposition B (per-slice triage, with Extensible / Preferred / Example
chosen on the merits of each value set, mirroring the differentiated
pattern the workgroup already established on the sister tickets) is the
right way to honour the Persuasive resolution without flattening the
nuance the workgroup has already shown. It is the disposition most
likely to produce a published profile that is both internationally
implementable and internally consistent with the rest of the IG.
