## Summary

Ron G. Parker requests that the canonical URL of the
`military-service-discharge-status-vs` Value Set in the US Military Service
(FHIR) IG be rooted in `terminology.hl7.org` (THO) rather than under
`hl7.org/fhir`. To support this re-rooting, he asks that a companion UTG
ticket be opened to register the Value Set in THO so the canonical can move
without losing the artifact's identity. The ticket was filed during
reconciliation against the US Military Service IG and is one of a class of
"defining URL should be in THO" comments raised across HL7 specifications at
that time. No related Jira tickets, Zulip threads, or GitHub items were
surfaced by the cross-reference index for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the canonical URL of the `military-service-discharge-status-vs` Value
Set out of the `http://hl7.org/fhir/...` namespace and re-root it under
`http://terminology.hl7.org/ValueSet/military-service-discharge-status` (or
the THO-policy-conformant slug agreed with the Vocabulary WG). Concretely:

1. File a UTG change request to register the Value Set (and any underlying
   CodeSystem(s) it depends on, if they are also IG-local) in THO under the
   appropriate steward.
2. Once UTG accepts, update the US Military Service IG so the ValueSet
   resource carries the new THO-rooted `url`, and replace every binding
   reference (in profiles, extensions, and narrative) with the new
   canonical.
3. Add a `ConceptMap` / retirement note (or at minimum a release-note entry
   and a `ValueSet.identifier` carrying the old canonical) so existing
   implementers can trace the move.

#### Justification

THO is the agreed home for HL7-stewarded terminology, and re-rooting
defining URLs there is the documented direction for specifications going
forward (the practice was made explicit in the FHIR R5 cycle, as the
resolution comment notes). Doing the migration as part of the normal IG
publication — paired with a UTG registration — keeps the canonical stable
under the steward that actually owns the content and avoids future churn
when the same comment is raised again in a later ballot cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing `hl7.org/fhir/...` canonical as the durable identifier of
record for the published version of the IG, but **mirror** the Value Set in
THO going forward: file the UTG ticket to publish the Value Set at
`http://terminology.hl7.org/ValueSet/military-service-discharge-status`,
mark the THO copy as authoritative for future maintenance, and add an
explicit `ValueSet.identifier` (and a release note) on the IG-local
artifact pointing at the THO canonical. Future versions of the IG would
then bind to the THO canonical, while the original URL continues to
resolve for already-deployed systems.

#### Justification

A literal canonical change is a breaking identity change for any
implementer that has hard-coded the existing URL in profiles, validators,
or terminology services. Mirroring captures the intent of the request
(THO becomes the steward) without invalidating prior conformance
artifacts. It also matches how several other IGs migrated terminology
into THO in the R4 → R5 window, and lets the WG defer the binding-rewrite
work to the next substantive revision of the IG rather than tying it to
this single comment.

---

### Disposition C: Decline

#### Proposal

Mark the request as **Considered for Future Use / Non-Persuasive** — i.e.,
acknowledge that THO is the long-term home for stewarded terminology but
decline to change the canonical URL of `military-service-discharge-status-vs`
in this cycle. This is the disposition currently recorded on the ticket
(per @seanmuir's comment dated 2022-02-07).

#### Justification

The cited rule that defining URLs belong in THO was made explicit in
FHIR R5 and was not a normative requirement in R4, which is the basis the
US Military Service IG was reconciled against. The commenter agreed at
the time that votes of this class would be treated as non-persuasive
pending a clearer cross-WG policy. Re-canonicalising a published Value
Set is also a breaking change for downstream implementers, and absent a
specific R4-era requirement there is no strong forcing function to incur
that cost mid-cycle.

---

### Recommendation

**Recommended disposition:** B

The literal request (Disposition A) is directionally correct and aligns
with the post-R5 norm that stewarded terminology lives in THO, but
flipping the canonical URL of a published ValueSet is a breaking identity
change that the workgroup historically declined to take in this very
ticket (Disposition C). Disposition B threads that needle: it commits
the WG to register the Value Set in THO via a UTG ticket — which is the
*operational* substance of what the reporter asked for — while preserving
the existing `hl7.org/fhir/...` canonical for already-deployed
implementers and deferring the binding rewrite to the next substantive
revision of the IG. That gives the workgroup a defensible "yes, with a
migration path" posture rather than re-stamping the original
non-persuasive disposition, and it sets up a clean canonical move at the
next IG version without invalidating R4-era conformance artifacts in the
interim.
