## Summary

The reporter observed that several profiles published under the *US Vital
Signs with Qualifying Extensions (FHIR)* IG have `StructureDefinition.kind`
set to `complex-type` rather than `resource`, even though they constrain
the `Observation` resource (whose own `kind` is `resource`). Specifically
called out are the BMI, Body Weight, and Blood Pressure Panel profiles.
The reporter is asking whether this is intentional and correct, or whether
the published `StructureDefinition` resources are mis-typed. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the FhirAugury data store for this key, so this report is
based solely on the ticket itself.

## Details

**Description:**

> Profiles are designated as complex types instead of Resource:
> In the xml of your profiles there is: `<kind value="complex-type"/>` (in JSON it is `"kind": "complex-type"`,)
> In the profiles, it conforms to (observation) "kind" is "Resource": is: `<kind value="resource"/>` (in JSON it is `"kind": "resource"`,)
>
> I only checked BMI, Body Weight and Blood Pressure panel. Is this intentional and correct?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a publication-tooling defect in the *US Vital Signs with
Qualifying Extensions (FHIR)* IG and correct the affected
`StructureDefinition` resources so that `StructureDefinition.kind` is
`resource` (matching the base resource `Observation`) rather than
`complex-type`. Concretely:

- Audit every `StructureDefinition` in the IG whose `baseDefinition`
  resolves (transitively) to a resource (e.g., `Observation`,
  `Patient`, `Encounter`) and confirm `kind = "resource"`.
- Re-publish the IG so that both XML (`<kind value="resource"/>`) and
  JSON (`"kind": "resource"`) representations are consistent across the
  BMI, Body Weight, Blood Pressure Panel, and any other affected
  profiles.
- Investigate the IG's authoring/publication pipeline (likely the IG
  Publisher inputs or hand-authored `StructureDefinition` source files)
  to identify why `kind` was emitted as `complex-type`, and add a
  validation step (e.g., a sushi-config / IG Publisher check, or a
  QA assertion) so that a profile's `kind` cannot diverge from its
  base's `kind` on future builds.

#### Justification

Per the FHIR core spec, `StructureDefinition.kind` for a profile must
match the kind of the resource/type being constrained â€” a profile of
`Observation` (`kind = resource`) cannot itself be `kind =
complex-type`. Tooling that consumes profiles (validators, IG
Publisher, FHIR Path engines, $validate operations) keys off `kind` to
decide how to instantiate, validate, and serialize instances. Leaving
`kind = complex-type` on profiles of `Observation` will make those
profiles invalid against the base specification and may cause
downstream tools to silently mis-handle them. The reporter's
observation is therefore a real defect, not a stylistic question, and
the right disposition is to fix the IG and harden the build to prevent
recurrence.

---

### Disposition B: Alternative Approach

#### Proposal

Acknowledge the defect in this ticket but, instead of treating it as a
one-off IG fix, route the underlying problem to the IG Publisher /
publication-tooling backlog: open (or link to) a tooling ticket
requiring the IG Publisher to *automatically derive*
`StructureDefinition.kind` from the resolved `baseDefinition` (or fail
the build with a clear error if a hand-authored value disagrees).
Resolve `FHIR-28541` itself as "fixed in IG re-publish" once the US
Vital Signs IG is re-built with the corrected `kind`, and cross-link
the tooling ticket so future IGs benefit.

#### Justification

The literal request â€” "is this intentional and correct?" â€” only needs a
yes/no answer plus a corrected publication. But the more useful,
durable outcome is to make this class of error impossible to ship.
Hand-maintaining `kind` on every `StructureDefinition` is exactly the
kind of derivable metadata that publication tooling should compute
from `baseDefinition`. Doing the fix only in the US Vital Signs IG
leaves every other IG exposed to the same mistake; doing it in tooling
fixes it once for the whole ecosystem. The trade-off versus
Disposition A is scope and timing: a tooling change requires
coordination with the IG Publisher maintainers and won't be
instantaneous, so the IG-level fix should still ship in parallel.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Not Persuasive** / **Question Answered** on the
basis that, at the time it was filed (2020-09), the published IG output
faithfully reflected what the IG's authoring inputs declared, and the
ticket has since been overtaken by subsequent publications of the IG
that no longer exhibit the issue. No spec-level change is taken from
this ticket.

#### Justification

This disposition is only defensible if the WG can confirm by
inspection that the current published `StructureDefinition` resources
for the US Vital Signs profiles already carry `kind = resource`, in
which case there is nothing left to act on and the ticket is stale.
The reporter's observation would still have been correct at filing
time, so this disposition is essentially "no change required today";
it should not be selected if any currently-published artifact in the
IG still shows `kind = complex-type`.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a clear, objective defect: a profile of
`Observation` cannot validly declare `StructureDefinition.kind =
complex-type`, and the affected artifacts (BMI, Body Weight, Blood
Pressure Panel, and likely siblings) need to be republished with `kind
= resource`. Disposition A delivers that fix and additionally hardens
the IG's QA so the same mistake doesn't recur. Disposition B is
strictly better in the long run, but it depends on a separate
tooling-side change and shouldn't block the IG fix; the WG can layer
B on top of A by opening a follow-up tooling ticket against the IG
Publisher. Disposition C is only viable if a current re-publish has
already silently fixed this, which the WG should verify before
considering it.
