## Summary

FHIR-56244 is a US Core v9 ballot QA tracker raised by Eric Haas as part of the
Cross-Group Projects Requirements clean-up. It asks the editors to tighten four
pieces of US Core narrative that currently use soft language ("strongly
encouraged", "are encouraged to", "must", "can use") so they instead use the
RFC-2119 keyword **SHOULD**. The rationale is that "SHOULD" produces a formal,
testable best-practice capability that downstream conformance tooling and
implementer reviewers can act on, whereas the existing wording is subjective.

The four narrative locations called out in the description are:

1. **General Requirements** – server search behaviour without `status`
   (`general-requirements.html#search-for-Servers-requiring-status`).
2. **US Core Device** – UDI-specific implementation guidance pointing at the
   FDA GUDID (`StructureDefinition-us-core-device.html#udi--specific-implementation-guidance`).
3. **Screening and Assessments** – the rule that point-in-time facts that need
   to be searched/trended/etc. be surfaced as `Observation` rather than
   `QuestionnaireResponse`
   (`screening-and-assessments.html#choosing-between-questionnaireresponse-and-observation`).
4. **US Core Coverage** – guidance on which `Coverage.type` codes to use to
   differentiate Medicare Parts A/B/C/D
   (`StructureDefinition-us-core-coverage.html#profile-specific-implementation-guidance`).

The ticket has been triaged with a proposed disposition of **Persuasive** /
"update as proposed" and grouped into **Block Vote 3** for the 2026-04-09
CGP/US Core Ballot Reconciliation call. It has already been pre-applied to the
US Core CI build via commit `cec272b2` ("FHIR-56217 FHIR-56221 FHIR-56244
FHIR-56246 FHIR-56260 FHIR-56271 FHIR-56272") and recorded in
`input/pagecontent/changes.md` with a "Non-substantive (Clarification)" change
impact. There are no explicitly linked Jira tickets and no other directly
cross-referenced trackers, but the Block Vote 3 announcement on Zulip groups
this ticket with eight sibling SHOULD/MAY narrative-keyword fixes
(FHIR-56217/56221/56246/56260/56261/56267/56271/56272).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the four wording edits exactly as written in the ticket description, in
the US Core authoring sources under `HL7/US-Core/input/pagecontent/`
(and the corresponding `input/intro-notes/` files for the Device and Coverage
profile-specific guidance):

1. `general-requirements.md` (anchor `search-for-Servers-requiring-status`):
   replace "Servers are strongly encouraged to support a query for resources
   without requiring a status parameter." with "Servers **SHOULD** support a
   query for resources without requiring a status parameter."
2. `StructureDefinition-us-core-device-intro.md` (anchor
   `udi--specific-implementation-guidance`): replace "Implementers are
   encouraged to use the FDA Global UDI Database (GUDID) …" with
   "Implementers **SHOULD** use the FDA Global UDI Database (GUDID) …".
3. `screening-and-assessments.md` (anchor
   `choosing-between-questionnaireresponse-and-observation`): replace "…
   anything that meets one of the preceding criteria must be surfaced as an
   Observation." with "… anything that meets one of the preceding criteria
   **SHOULD** be surfaced as an Observation."
4. `StructureDefinition-us-core-coverage-intro.md` (anchor
   `profile-specific-implementation-guidance`): in each of the three Medicare
   Part bullets, replace "can use" with "**SHOULD** use".
5. Add a single change-log row in `input/pagecontent/changes.md` under v9.0.0
   crediting FHIR-56244 (already done on the CI build via commit
   `cec272b2`).

This is the resolution Eric Haas has put forward as "Persuasive — update as
proposed" and that has been pre-applied on the CI build for Block Vote 3.

#### Justification

The ticket's rationale is sound: in HL7 specification prose, "SHOULD" is an
RFC-2119 conformance keyword and is interpreted by implementers,
ballot-reconciliation reviewers, and conformance tooling as a formal,
recommendable best practice. "Strongly encouraged", "are encouraged to",
"must" (lowercase) and "can use" carry no such weight; they read as opinion or
an unscoped requirement that may either be ignored (encouraged/can) or be
treated as SHALL (must), depending on the reader. Aligning these four
sentences with SHOULD makes the requirements consistent with the rest of the
US Core narrative and with the new US Core Narrative Requirements Tables that
the CGP team specifically built this v9 ballot reconciliation around. The
change is text-only, lives entirely under `input/pagecontent/` /
`input/intro-notes/`, requires no FSH or resource regeneration, and matches
the bulk pre-applied commit that already covers seven sibling trackers in
Block Vote 3. No counter-position has been raised on Zulip and no commenter
has asked to pull FHIR-56244 from the block vote.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the SHOULD wording for items **1, 2, and 3** as written, but treat
item **4 (Medicare `Coverage.type` codes)** differently: keep "SHOULD use"
for the Medicare Part A/B (`121`) bullet, but downgrade Parts C and D to
**MAY** (Part C lists three alternative codes — `111`/`112`/`113` — with no
guidance on which to pick, so a SHOULD on a multi-valued enumeration is
ambiguous; Part D's `122` is a plausible SHOULD candidate, but
implementations of Medicare Advantage / supplemental coverage often legitimately
use payor-specific codes). At the same time, fix the bracketed editorial
fragments in that same Coverage paragraph (`[when sending Coverage.type]`,
`[For]`, `[use]`) that are currently rendered as `class="error"` in the
source. Apply edits in the same authoring files as Disposition A, and split
the Coverage edit into its own change-log row so that any future
reconsideration is traceable.

#### Justification

Items 1–3 each constrain a single, unambiguous behaviour and are clean SHOULD
candidates. Item 4 is different: SHOULD on a sentence that lists multiple
acceptable codes ("Part C SHOULD use 111, 112, 113") will be read by some
implementers as "use all three", by others as "pick one", and by validators as
"any of the three is conformant" — none of which were the original author's
intent. MAY captures the "if you choose to use these codes, here's the list"
spirit more honestly. This alternative also lets the workgroup correct the
already-flagged `class="error"` fragments in the same edit window so the
section comes out of v9 with no lingering editorial markers. The cost is one
extra paragraph of prose and a separate change-log entry, which is small
relative to the ballot-reviewer clarity gained.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-56244 as **Not Persuasive — out of scope for v9 narrative cleanup**
and leave the four sentences using their current wording ("strongly
encouraged", "are encouraged to", "must", "can use"). Re-open the underlying
question via a single, methodology-level tracker that defines a US-Core-wide
glossary for narrative conformance keywords (when to use SHOULD vs MAY vs
"can"/"might") and apply that glossary uniformly across the IG in the next
release.

#### Justification

This is the weakest of the three options. It is only defensible if the
workgroup believes that promoting four ad-hoc narrative sentences to formal
SHOULD-level conformance — without a corresponding methodology decision and
without checking that downstream test-suites / Inferno profiles are ready to
assert SHOULD on these sentences — risks creating four new "Why didn't this
implementation get flagged?" complaints from US-Core implementers who treated
the prior wording as informative. The case for declining is undermined by the
fact that (a) the US Core Narrative Requirements Tables work that the CGP did
this ballot cycle is exactly that methodology decision, and (b) the change is
already pre-applied on the CI build with a "Non-substantive (Clarification)"
classification, so reverting it now would itself be a churn cost.

### Recommendation

**Recommended disposition:** A

The ticket is a small, surgical narrative cleanup that aligns four
inconsistent sentences with the RFC-2119 keyword the surrounding US Core text
uses everywhere else. The Cross-Group Projects ballot-reconciliation team has
already designed Block Vote 3 around exactly this pattern (nine sibling
tickets, all "Persuasive / Non-substantive (Clarification)"), the change has
been pre-applied on the CI build via commit `cec272b2`, and no participant on
the Zulip thread has asked to pull FHIR-56244 for separate discussion. The
nuance Disposition B raises about the Medicare `Coverage.type` paragraph is
genuine and worth airing on the call, but it is small enough that it can be
handled either as a friendly amendment to Disposition A or as a follow-up
tracker against v9.0.1 — it should not derail Block Vote 3. Disposition C is
not warranted; the methodology rationale (formal, testable best-practice
capability) is sound and is exactly what the new US Core Narrative
Requirements Tables are meant to enforce.
