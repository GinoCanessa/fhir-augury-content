## Summary

The ticket asserts that the `military-service-combat-episode-mission-vss`
ValueSet, defined in the US Military Service (FHIR) IG, has a canonical URL
rooted at `hl7.org/fhir/...` and argues that it should instead be rooted at
`terminology.hl7.org` (THO), the canonical home for HL7-curated terminology
content. The reporter, Ron G. Parker, also asks that a companion UTG ticket be
opened to register the ValueSet in THO so that it lives alongside other shared
HL7 terminology rather than inside the IG's own canonical namespace. There are
no cross-referenced Jira tickets, Zulip threads, or GitHub items recorded for
this issue in the FhirAugury index. The ticket has already been resolved as
"Considered for Future Use" with a single committee comment of
"Non-persuasive", on the rationale that explicit direction to root such
ValueSets in THO is part of FHIR R5 guidance but is not stated in FHIR R4.

### Disposition A: Accept As Requested

#### Proposal

Re-root the canonical URL of the `military-service-combat-episode-mission-vss`
ValueSet from the US Military Service IG's `hl7.org/fhir/...` namespace to a
`http://terminology.hl7.org/ValueSet/...` URL, and open a companion UTG ticket
to register and publish the ValueSet (and its underlying CodeSystem if one is
locally defined) in THO. In the IG, replace the ValueSet definition with a
reference to the new THO canonical, update every binding that points at the
old URL, and add a deprecation/redirect note in the IG so that consumers
following the old URL are pointed at the THO version. The migration would be
done as part of the next non-trivial republication of the IG so the URL change
is tied to a new IG version.

#### Justification

THO is the designated home for HL7-curated terminology that is intended to be
shared across IGs and projects, and rooting reusable ValueSets there is the
direction explicitly stated in FHIR R5 publishing guidance. Even under R4,
where the rule is not normatively stated, several Cross-Group Projects and IG
authors have already migrated reusable ValueSets to THO precisely so they can
be referenced from multiple IGs without coupling consumers to a single IG's
canonical namespace. The reporter's request follows that established pattern
and the linked UTG step is the standard mechanism for publishing into THO.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the ValueSet's canonical URL where it is for the current published
version of the US Military Service IG (to avoid breaking existing references),
but in the next major version of the IG: (1) publish a *new* THO-rooted
ValueSet via UTG with the same content, (2) deprecate the IG-local ValueSet
and mark it as superseded by the THO ValueSet using `ValueSet.replaces` /
narrative guidance, and (3) update all in-IG bindings to point at the THO
canonical. This preserves backward compatibility for any implementer that
already consumed the original URL while still satisfying the underlying intent
of moving stewardship to THO.

#### Justification

A canonical URL is, by FHIR convention, a stable identifier; silently
re-rooting it (Disposition A) is a breaking change for any implementer that
has already pinned to the IG-local URL or built validation against it.
Disposition B accomplishes the same end-state — the ValueSet lives in THO and
is reusable across IGs — without invalidating prior conformance claims, and it
aligns with the way other HL7 work groups have handled similar migrations
(parallel publish, then deprecate). The cost is a longer transition window and
the maintenance overhead of two canonicals during the deprecation period.

---

### Disposition C: Decline

#### Proposal

Decline the request and close as **Considered for Future Use / Non-persuasive**
(which is, in fact, the resolution already recorded on the ticket). Document
that R4-era IGs are not required to root reusable ValueSets in THO, that the
existing IG-local canonical is conformant under R4 publishing rules, and that
any migration to THO will be considered when the IG is next reballoted —
ideally against R5 — at which point the THO-rooting expectation is explicit
and the migration can be done as a single coherent change rather than
piecemeal.

#### Justification

This is the disposition the workgroup has already taken, and its rationale is
defensible: under FHIR R4 there is no normative requirement to root reusable
ValueSets in THO, the existing canonical is technically valid, and changing it
mid-cycle would either break consumers (if done in place) or impose a
multi-version deprecation tail (if done via parallel publish) for a ValueSet
that is narrowly scoped to a single IG. Deferring the change until the IG is
republished against R5 lets the work happen in lockstep with the wider HL7
direction on terminology stewardship and avoids spending committee cycles on
a per-ValueSet basis.

---

### Recommendation

**Recommended disposition:** C

The workgroup has already resolved this ticket as "Considered for Future Use"
with a "Non-persuasive" comment, and that resolution is the right one for an
R4-era IG: the THO-rooting expectation is an R5-era publishing convention, the
current canonical is conformant under R4, and there is no recorded community
discussion (no Zulip threads, no related tickets, no GitHub activity) pushing
for an immediate migration. The right time to act on the underlying intent is
when the US Military Service IG is next reballoted against R5, at which point
both the canonical re-rooting and the UTG registration can be done as a
coherent package rather than a one-off URL change. Disposition A or B should
be revisited at that point; for the current cycle, declining (and leaving the
existing resolution in place) is the recommendation.
