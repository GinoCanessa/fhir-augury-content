## Summary

The ticket asks the Cross-Group Projects authors of the *US Vital Signs with
Qualifying Extensions (FHIR)* IG to surface the example instances that the
guide already publishes by linking them from each affected profile and
extension definition. Examples exist in the IG, but because they are not
referenced from the StructureDefinition narratives, readers landing on a
profile or extension page have no obvious way to navigate to a worked
example, which makes the IG harder to use for implementers. The reporter,
Bryn Rhodes, frames this as a usability/discoverability fix rather than a
content change to the profiles themselves: keep the existing examples,
just cross-link them from the artifact pages where implementers naturally
land. No related Jira tickets, Zulip threads, Confluence pages, or GitHub
issues/PRs were surfaced for this ticket via FhirAugury cross-reference
or text-search lookups.

## Details

## Proposed Dispositions

> Note: The dispositions below are intentionally generic about file
> paths because no `repo-analysis` briefing exists for the IG hosting
> the *US Vital Signs with Qualifying Extensions* spec. They name the
> conceptual targets (profile and extension StructureDefinitions, IG
> publisher example bindings, and intro/notes pages) that any IG built
> with the HL7 IG Publisher exposes.

### Disposition A: Accept As Requested

#### Proposal

For every profile and extension `StructureDefinition` shipped by the
*US Vital Signs with Qualifying Extensions (FHIR)* IG, declare each
applicable example instance as an `example` against that profile in the
IG's `ImplementationGuide` resource (i.e., set
`ImplementationGuide.definition.resource.exampleCanonical` /
`exampleBoolean = true` and point it at the profile's canonical URL).
This is the IG-publisher-supported mechanism that causes the rendered
profile and extension pages to grow an "Examples" section listing every
linked example, with hyperlinks to the example's rendered page. Where
an example exercises multiple profiles/extensions, repeat the
declaration so the example shows up on each artifact's page.

In parallel, update the profile and extension intro/notes Markdown so
that any narrative discussion of "see also" examples uses an explicit
hyperlink (relative IG link) to the example page, so the cross-reference
is robust to publisher changes.

#### Justification

This is exactly what the reporter asked for and matches the IG
Publisher's intended mechanism for example discoverability. It does not
modify the profile or extension semantics, so it is non-breaking,
ballot-neutral, and low-risk. It improves implementer onboarding, which
is the explicit motivation in the ticket. Many sibling HL7 IGs (e.g.,
US Core, mCODE) already use this pattern, so reviewers and implementers
will recognize the rendered output.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) wiring each example to its profile via
`ImplementationGuide.definition.resource`, add a dedicated "Examples"
page to the IG that groups examples by clinical concept (vital sign
group, qualifying extension category, etc.) and link to that page from
the profile and extension intro Markdown via a single, prominent
"Worked examples" call-out. Keep `meta.profile` set on each example so
that the IG Publisher's automatic profile-conformance pages still work,
but treat the curated index page as the canonical entry point.

#### Justification

When a profile is exercised by many similar examples, the auto-generated
"Examples" section on the profile page can become a long, undifferentiated
list. A curated index lets the workgroup explain *why* each example
matters and group qualifying-extension permutations meaningfully, which
is more useful for implementers than a flat list. The Markdown link
from the profile intro still satisfies the reporter's discoverability
concern. The trade-off versus Disposition A is the additional editorial
burden and the risk that the curated page drifts from the actual example
set over time — Disposition A's auto-generated section never drifts
because the IG Publisher regenerates it on every build.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Persuasive with Mod* / *Not Persuasive* with the
rationale that the IG already publishes the example instances at stable
URLs and provides a global "Artifacts" / "Examples" index, and that the
authoring workgroup considers that index sufficient for example
discoverability without per-profile cross-links.

#### Justification

This is the weakest of the three options because the reporter's
observation — that landing on a profile or extension page does not
surface relevant examples — is factually correct for the IG as
published. Declining would only be defensible if the workgroup
explicitly considers per-artifact example linking out of scope for this
IG (for example, because the IG is in maintenance mode and is not
accepting non-substantive editorial enhancements), and that position is
not visible in the ticket history.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right starting point: it is exactly what the
reporter asked for, it uses the IG Publisher's first-class mechanism for
attaching examples to profiles and extensions, it has no impact on
profile semantics or conformance, and it brings the IG into line with
peer HL7 IGs that already do this. Disposition B is worth considering
as a follow-on enhancement if the example set is large enough that a
curated narrative index would help implementers, but it should not
block the simple discoverability fix. Disposition C is hard to defend
given that the reporter's underlying complaint is accurate and the fix
is editorial rather than normative.
