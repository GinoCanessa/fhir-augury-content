## Summary

The reporter notes that the R6 Vital Signs Base Profile now places the
"interoperability" LOINC codes (e.g., `8867-4` heart rate, `9279-1`
respiratory rate, `85354-9` blood-pressure panel) on
`Observation.category` rather than on `Observation.code`. However, the
example searches still shown in the Observation Vital Signs Profiles
introduction continue to query by `code` using those very LOINCs
(e.g., `GET [base]/Observation?patient=1186747&code=8867-4`). The
profile text and the search examples are therefore inconsistent: a
server that conforms to the new R6 profile placement would not return
the expected resources for the documented `code=…` queries. The
reporter asks that the examples either be rewritten to search by
`category` (when interoperability LOINCs are involved) or that the
intended search behaviour be clarified given the new element
placement.

The single comment on the ticket (from a triager, 2026-02-03) raises
clarifying questions for the Vital Signs SMEs:

1. Is `category=vital-signs` (HL7 category code) equivalent to
   `category=85353-1` (LOINC interoperability category for the panel),
   or is the latter potentially a subset?
2. Are `category=vital-signs,9279-1` and `category=9279-1` both valid
   ways to find respiratory-rate observations?
3. Is `code=59408-5` alone (i.e., a more specific LOINC) a valid
   search even without the categorisation? If so, none of the example
   searches in §10.1.19 are technically *wrong*, since each
   "interoperability category" LOINC is also bound into the `.code`
   value-set bindings — they are just no longer the recommended /
   profile-aligned pattern.
4. New examples should likely be added that demonstrate finding vital
   signs purely by category code.

This ticket is the R6 follow-on to a thread of earlier vital-signs
search-example tickets:

- **[FHIR-50392](https://jira.hl7.org/browse/FHIR-50392)** — proposed
  the same swap (replace `code=8867-4` with `category=8867-4` in the
  example block) but was closed *Resolved – No Change / Not Persuasive*
  in October 2025 with the rationale "This is related to the decision
  to continue to identify Vital Signs using `.code`, not `.category`."
  That decision has since been **reversed** in R6 (see FHIR-53553),
  which is precisely why FHIR-53902 has been re-raised against the new
  ballot.
- **[FHIR-53553](https://jira.hl7.org/browse/FHIR-53553)** — the R6
  Ballot Note for Observation, which explicitly records that R6
  "removed references to the LOINC 'magic value' code, and instead
  refer[s] to it as the 'interoperability category'", created
  per-profile preferred-binding value sets on `Observation.code`, and
  "updated the vital signs examples where needed to be consistent with
  the updated profiles." FHIR-53902 documents an example block that
  was missed during that sweep.
- **[FHIR-50452](https://jira.hl7.org/browse/FHIR-50452)** — a related
  R5 complaint that the Base profile already mentioned the
  "interoperability category" requirement at the wrong level; closed
  *Not Persuasive* under R5 semantics.
- **[FHIR-43353](https://jira.hl7.org/browse/FHIR-43353)** — relaxed
  the required UCUM binding on Vital Signs components; *Applied,
  Persuasive with Modification*. Establishes the OO pattern of
  preferring narrower bindings on the base profile rather than
  prescriptive ones.

## Details

**Description:**

> The profile specifies that interoperability LOINC codes (e.g.,
> 8867-4, 9279-1, 85354-9) now reside in the `category` element rather
> than `code`. However, the search examples still demonstrate
> searching by `code` parameter with these LOINCs:
>
> `GET [base]/Observation?patient=[id]&code=[vital sign LOINC{,LOINC2,LOINC3,...}]`
>
> `GET [base]/Observation?patient=1186747&code=8867-4`
>
> ...
>
> Suggested change: Update all examples to search by `category` when
> using interoperability LOINC codes, or clarify the intended search
> behavior given the new element placement.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit `source/observation/observation-vitalsigns-profile-introduction.xml`
in `HL7/fhir` to rewrite the Vital Signs example searches so that
queries which target the *interoperability category* LOINCs are issued
against `Observation.category`, not `Observation.code`. Concretely:

- Replace the generic template
  `GET [base]/Observation?patient=[id]&code=[vital sign LOINC{,LOINC2,LOINC3,...}]`
  with
  `GET [base]/Observation?patient=[id]&category=[interoperability-category LOINC{,LOINC2,LOINC3,...}]`.
- Replace the worked single-code example
  `GET [base]/Observation?patient=1186747&code=8867-4` with
  `GET [base]/Observation?patient=1186747&category=8867-4`, and the
  multi-code variant
  `…&code=8867-4,9279-1,85354-9` with
  `…&category=8867-4,9279-1,85354-9`.
- Add (or retain) a separate example that shows querying by
  `Observation.code` for a *specific* per-profile LOINC drawn from the
  R6 preferred-binding value set (e.g.,
  `…&code=59408-5` for SpO₂ by pulse oximetry), to make the
  category-vs-code split explicit.
- Add a one-sentence note explaining the R6 placement: interoperability
  LOINCs identify the *kind* of vital sign and are searched via
  `.category`; per-profile codes identify the specific measurement and
  are searched via `.code`.

#### Justification

This literally restores consistency between the R6 profile constraints
and the example searches. The same fix was previously requested in
FHIR-50392 and declined under R5 because R5 identified vital signs by
`.code`; that premise has since been reversed by the R6 ballot
(documented in FHIR-53553), which is exactly what FHIR-53902 is
flagging. The change is confined to a single narrative XML file in
`HL7/fhir` (`source/observation/observation-vitalsigns-profile-introduction.xml`),
matches the FhirCore briefing's *Update narrative / prose* recipe,
touches no `<snapshot>` content, and has no cross-repo implications.
It also follows the pattern endorsed in FHIR-34640 — search examples
must illustrate codes / parameters the surrounding profile actually
constrains.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the request as a **clarification** rather than a wholesale swap,
and address the SME questions raised in the existing comment:

- Keep one example using `code=` (so readers see that interoperability
  LOINCs *are* still bound into `Observation.code` via the value-set
  bindings and a `code=…` query is not strictly invalid), but add a
  preceding paragraph that names this the *legacy* pattern and points
  readers at the canonical R6 pattern.
- Add a new section "Searching by interoperability category vs. by
  specific code" under the existing examples in
  `source/observation/observation-vitalsigns-profile-introduction.xml`,
  with worked examples for: (a) all vital signs via
  `category=vital-signs`; (b) all vital-signs by panel via
  `category=85353-1`; (c) specific vital-sign family via
  `category=8867-4`; (d) most-specific measurement via
  `code=59408-5`; and (e) the combination `category=2708-6&code=59408-5`.
- Cross-link the section to the per-profile preferred-binding value
  sets created for R6 (FHIR-53553) so implementers can discover the
  "specific code" candidates per profile.

#### Justification

The triager's comment surfaces real ambiguity — in particular, whether
`category=vital-signs` and `category=85353-1` are equivalent, and
whether `code=8867-4` is wrong or merely non-canonical. A pure swap
(Disposition A) silently makes a normative-feeling decision in
narrative without explaining it; servers built against earlier R6
drafts may still answer `code=8867-4` queries because the value-set
bindings on `Observation.code` permit those LOINCs. A clarifying
restructure preserves backward compatibility for early R6 implementers,
explicitly answers the SME questions in the spec itself, and is more
useful to readers than a one-line edit. The change still lives in the
same single file (`observation-vitalsigns-profile-introduction.xml`),
so the implementation cost is similar.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-53902 as **Resolved – No Change / Not Persuasive** on the
grounds that the existing examples are not strictly wrong: each
"interoperability category" LOINC remains in the `Observation.code`
value-set bindings of the relevant per-vital-sign profiles, so the
documented `code=8867-4` query continues to return conformant
resources. Recommend the reporter use the per-profile preferred
bindings on `.code` to discover the canonical per-measurement LOINCs
and rely on category for grouping. No spec change.

#### Justification

This mirrors the precedent set by FHIR-50392 (closed *Not Persuasive*
on a similar example-rewrite request) and FHIR-50452 (closed *Not
Persuasive* on a related Base-profile narrative complaint). Declining
keeps the narrative stable across the late R6 ballot cycle and avoids
the risk of introducing a new inconsistency between the introduction
prose and the per-profile conformance text if the swap is done
incompletely.

---

### Recommendation

**Recommended disposition:** B (with elements of A).

The R6 ballot (FHIR-53553) explicitly relocated the interoperability
LOINCs to `Observation.category` *and* added per-profile preferred
bindings on `Observation.code`. In that reshaped world, the existing
example block — which uses interoperability LOINCs as `code=` query
arguments — is misleading even if it is not technically illegal under
the value-set bindings, because it is the worst possible illustration
of the new shape: it teaches readers exactly the pattern the R6
remodelling was meant to move them away from. A pure swap (A) is the
right *technical* edit but leaves the SME questions in the existing
comment unanswered and risks reintroducing the same confusion the next
time someone reads the page in isolation. A clarifying restructure (B)
that includes A's swap, adds a worked category-vs-code example matrix,
and references the R6 preferred-binding value sets is the disposition
that closes both the literal inconsistency and the underlying
documentation gap. Disposition C is not defensible now that the R5
premise FHIR-50392 was declined under has been reversed; reusing that
rationale would conflict with FHIR-53553 and continue to ship a
ballot whose spec text and examples do not agree.
