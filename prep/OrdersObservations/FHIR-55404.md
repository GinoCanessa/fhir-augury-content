## Summary

The reporter (Jose Costa-Teixeira) asks the Clinical Order Workflow (COW)
IG to add a "Bidding" pattern: a functional introduction explaining what
bidding-style ordering is, the typical scenarios it covers, and links from
that introduction to the relevant technical patterns, FHIR resources, and
worked examples. The ticket observes that useful bidding guidance has
already been discovered/written in pieces (e.g., on the
`BMS-Ballot-Updates` branch's `workflow-state-overview.html`), but there
is no narrative entry-point that orients implementers to the topic before
they hit the technical machinery. The cross-reference index returned no
hits for FHIR-55404 in either direction, and no Zulip threads cite the
ticket key directly. However, Zulip's `workflow > Bidding on Tasks` topic
(37 messages, 2023-11 through 2024-05) is the obvious upstream community
discussion behind this request, and FHIR Infrastructure ticket
[FHIR-43312](https://jira.hl7.org/browse/FHIR-43312) — *"Add `offer` as
new `intent` for ServiceRequest"* (Persuasive with Modification, applied
to the build) — is the technical primitive the bidding pattern would
build on.

### [FHIR-43312: Add `offer` as new `intent` for ServiceRequest](https://jira.hl7.org/browse/FHIR-43312)

Andy Bond (FHIR Infrastructure) proposed a new `offer` value on
`ServiceRequest.intent` so that a prospective filler can propose
fulfillment of an existing ServiceRequest by issuing their own
ServiceRequest with `intent = offer` — supporting bid-on-token /
bid-on-paper-form scenarios and competitive offers (location, time,
cost). The ticket was resolved **Persuasive with Modification** and
applied to the FHIR Core build (PR HL7/fhir#3677, commit `954ae5d`).
This is the technical primitive on which a Bidding pattern in the COW
IG would be documented; FHIR-55404 is essentially asking COW to provide
the narrative + scenario layer over the FHIR Core change made by
FHIR-43312.

Reporter: Andy Bond. Assignee: John D.L. Nolen. Commenters: jdlnolen.

## Related Zulip Discussions

### [workflow > Bidding on Tasks](https://chat.fhir.org/#narrow/stream/workflow/topic/Bidding.20on.20Tasks)

(37 messages, 2023-11-03 through 2024-05-21.) Lloyd McKenzie opened the
thread describing a Portuguese MoH workflow where a placer creates an
"open" request and multiple potential fillers respond — an offer/bid
pattern that the workflow spec did not cover well. The discussion
explored several modeling options:

- **Sub-Task per respondent.** Lloyd's initial proposal: one Task per
  candidate filler, but the spec lacks a clean status to convey "offer"
  (distinct from "accept"). Andy Bond noted this is what their
  pathology/imaging implementations already do, including barcode-/
  token-based claim against either an individual Task or a group Task
  for a RequestGroup.
- **A `claim` operation on Task.** Andy described existing
  implementations using a `$claim`-style operation against Task; Lloyd
  pushed back on operation-based approaches and on focusing the claim
  on the ServiceRequest rather than the Task (since a Task may seek
  fulfillment of only part of a ServiceRequest, e.g., one MRI of a
  multi-MRI authorization).
- **A new `ServiceOffer` resource** (Andy) — Lloyd preferred reusing
  ServiceRequest with a new `intent`.
- **A new `offer` intent on ServiceRequest/Task.** Lloyd's
  recommendation; the prospective filler creates a *new* ServiceRequest
  with `intent = offer` describing exactly what they will do (code,
  method, timing, …), linked via `basedOn` to the original. Andy
  agreed and filed [FHIR-43312](https://jira.hl7.org/browse/FHIR-43312)
  on 2023-11-30.
- **Workflow Pattern F discussion** (Bram Wesselo, May 2024). Bram
  asked whether mutating `Task.intent` from `proposal` to `order` is
  acceptable; Lloyd answered firmly **no** — `Task.intent` is
  immutable, and changing intent values would make a solution
  non-conformant. Instead, a new Task instance with `intent = order`
  must be created (the order points to the proposal Task via
  `basedOn`), and whether the proposal Task is then closed/cancelled
  is an implementer choice.

**Range of opinions / open questions.** There is consensus that bidding
*needs* explicit support and that immutable-intent semantics must be
preserved, but the community did not converge on a single canonical
representation — sub-tasks per respondent vs. ServiceRequest-with-offer-
intent vs. operation-style claim were all live. There is also an open
question (Lloyd, last message) about whether/why anyone would actually
*cancel* a proposal once a bid is selected. These unresolved points are
exactly what the requested COW narrative needs to settle for
implementers.

Participants: Lloyd McKenzie, Andy Bond, Grahame Grieve, Marc de
Graauw, Bram Wesselo, Jose Costa Teixeira.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level "Bidding" pattern page to the Clinical Order
Workflow IG (`HL7/fhir-cow-ig`). Concretely:

- Create `input/pagecontent/bidding.md` (or equivalent IG authoring
  path) introducing the pattern in narrative terms: what bidding is in
  a clinical-order context, why it is distinct from a single-filler
  fulfillment flow, and the typical scenarios (open request to a panel
  of fillers, paper- or token-based claim against a group order, MoH
  open-tender flows, hospital-to-care-home transfer offers, etc.).
- For each scenario, link to the technical primitives that already
  exist: `ServiceRequest.intent = offer` (delivered by
  [FHIR-43312](https://jira.hl7.org/browse/FHIR-43312)), Task with
  `basedOn` chains for proposal → order transitions, and the relevant
  Workflow Patterns (notably Pattern F).
- Cross-link from `workflow-state-overview.html` and the IG TOC into
  the new page so readers find it from existing entry points.
- Add at least two worked examples (FSH/JSON) showing the canonical
  shape: (1) placer issues a ServiceRequest + solicitation Task; (2)
  one or more fillers respond with `ServiceRequest.intent = offer`
  linked by `basedOn`; (3) placer accepts one offer and a new Task
  with `intent = order` is created pointing to the chosen offer.
- Register the new page in the IG menu/TOC and rebuild.

#### Justification

This is exactly what the ticket asks for: a functional introduction to
Bidding that consolidates the guidance that already exists in
fragments and binds it to concrete FHIR mechanics. The technical
underpinnings already exist — `ServiceRequest.intent = offer` was
applied to the FHIR Core build via FHIR-43312 — so the COW IG can
document a complete pattern today without waiting on further core
changes. The `workflow > Bidding on Tasks` thread shows this is a
recurring real-world need (Portugal MoH, Dutch hospital→care-home
transfers, AU pathology) but also that the community has not reached
consensus on a canonical shape; an explicit, opinionated narrative in
the IG is the appropriate venue to settle that. The reporter's link to
`workflow-state-overview.html` on the `BMS-Ballot-Updates` branch
suggests material is already in flight that this page can absorb or
reference.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding one new "Bidding" page, treat bidding as a *worked
example* slotted into an expanded **Workflow Pattern F** ("solicitation
and competitive response") section of the COW IG, plus a short
glossary entry that points there. Specifically:

- Extend the Workflow Pattern F page in `HL7/fhir-cow-ig` with a
  named sub-section "Bidding / Open Solicitation" describing the
  placer-issues-solicitation → multiple-filler-offers →
  placer-selects-offer flow.
- Add a one-paragraph "Bidding" entry to the IG glossary that links
  into that sub-section.
- Use the same examples and `ServiceRequest.intent = offer` mechanics
  as Disposition A, but without a free-standing pattern page.
- Defer any topic-page dedicated to Bidding until additional patterns
  (auction with reserve, sealed-bid, time-boxed solicitation) emerge
  that justify a parent topic.

#### Justification

The Zulip thread is unambiguous that bidding is a *specialization* of
existing request/fulfillment workflows, not a new modality — every
proposed solution reused ServiceRequest, Task, `basedOn`, and the
existing intent vocabulary. Documenting bidding as a Pattern F
specialization keeps the IG's narrative spine intact and avoids
implying that bidding requires a new conceptual stack. It also reduces
the risk of orphaning a thinly-populated top-level page if the only
realized scenarios for some time are minor variants of one underlying
pattern. The trade-off vs. Disposition A is discoverability: a
top-level "Bidding" page is easier for an implementer Googling the
term to land on than a sub-section, but a glossary entry largely
mitigates that.

---

### Disposition C: Decline

#### Proposal

Decline as **out of scope for the current ballot cycle** of the COW IG,
on the grounds that (a) the underlying technical primitive
(`ServiceRequest.intent = offer`) was only recently applied to the
FHIR Core build via [FHIR-43312](https://jira.hl7.org/browse/FHIR-43312)
and has not yet seen broad implementation feedback, and (b) the Zulip
discussion did not converge on a single canonical bidding shape
(sub-task-per-respondent vs. offer-intent vs. claim-operation). Park
the ticket as "deferred — needs implementer feedback on offer-intent"
and revisit after a Connectathon track exercises the pattern.

#### Justification

Writing a definitive "Bidding pattern" page commits the workgroup to
an opinionated shape before that shape has been validated against
multiple independent implementations. The thread surfaced at least
three distinct designs in active use, and Lloyd's open question about
whether proposal Tasks should ever be cancelled remains unanswered.
Publishing prematurely risks codifying guidance the WG would need to
re-cut shortly after ballot. Declining for now and routing the topic
into a Connectathon scenario gives the WG implementer-grounded
material to write against in the next cycle.

---

### Recommendation

**Recommended disposition:** A

The community discussion has been running for over 18 months across
two distinct national contexts, the technical primitive (`offer`
intent) has already been adopted into the FHIR Core build via
FHIR-43312, and the reporter has identified concrete prose already
present on the `BMS-Ballot-Updates` branch that lacks a narrative
home. The cost of writing a focused pattern page is modest, while the
cost of *not* writing one is to leave implementers to re-derive the
pattern from a 37-message Zulip thread and a single intent value in
the FHIR spec. Disposition B is a reasonable fallback if the WG feels
a top-level page over-promises, but the discoverability win of a
named "Bidding" page in the COW IG is worth the slightly heavier
authoring commitment. Disposition C is not recommended: the
implementer pressure is real and a deferral would simply leave the
existing partial guidance scattered. The recommended disposition
should explicitly call out the open question from the thread (whether
proposal Tasks should be closed once an offer is accepted) and
document the WG's chosen answer rather than silently picking one.
