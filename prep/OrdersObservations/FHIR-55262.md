## Summary

`DeviceRequest.intent` is a 1..1 element with a `required` binding to the
`request-intent` value set / code system. The reporter (Lloyd McKenzie) notes
that the bound value set has no `unknown`, `other`, or equivalent "escape
valve" concept, so implementers cannot legally populate the element in cases
where the requester's intent is genuinely unknown or does not fit any of the
listed concepts. The ticket asks that such an escape-valve concept be added.

The same defect was logged against the other resources that bind their
`.intent` to the same value set: [FHIR-55275](https://jira.hl7.org/browse/FHIR-55275)
(ServiceRequest), [FHIR-55267](https://jira.hl7.org/browse/FHIR-55267)
(NutritionOrder) and [FHIR-55322](https://jira.hl7.org/browse/FHIR-55322)
(CommunicationRequest). On the 2026-02-12 OO main call, OO concluded that the
value set is owned by **FHIR-I** and the underlying code system by **CDS**, so
OO can record a recommendation but cannot itself make the change. A consolidated
ticket against FHIR-I — [FHIR-56142](https://jira.hl7.org/browse/FHIR-56142),
"Request.intent has no escape valve" — has since been raised by Michelle
Miller and resolved **Persuasive**, naming FHIR-55262 as one of the tickets
that would be settled by that single change. FHIR-55262 itself remains
**Triaged / Unresolved** in OO and needs a disposition that hands the work
off cleanly.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the request literally for `DeviceRequest.intent`: extend the bound
value set so that an "escape valve" concept (`unknown` and/or `other`) is
available without changing the binding strength or the cardinality. Because
`DeviceRequest.intent`'s required binding resolves to
`http://hl7.org/fhir/ValueSet/request-intent`, which draws codes from
`http://hl7.org/fhir/CodeSystem/request-intent`, the actual edits land in
**FHIR-I** (value-set owner) and **CDS** (code-system owner), not in OO's
own resource files. OO's role under this disposition is to record the
recommendation on FHIR-55262 and re-assign the ticket — to FHIR-I for the
value-set change, with a note that CDS must add the new code(s) to
`CodeSystem/request-intent` first. No change to `DeviceRequest.intent`
itself (still `1..1`, still `required`).

#### Justification

This is exactly what the reporter asked for and exactly what closes the
defect: today, an implementer who genuinely does not know the requester's
intent has no legal value to send for a `1..1 required` element. Adding an
`unknown`/`other` code is the standard FHIR-wide pattern for such cases and
preserves cross-resource consistency with the other `*.intent` elements
that share the same value set. It also matches the path taken at FHIR-I:
[FHIR-56142](https://jira.hl7.org/browse/FHIR-56142) ("Request.intent has
no escape valve") has already been resolved **Persuasive** against FHIR-I,
naming FHIR-55262 as one of the tickets a single fix would settle. OO
recording "accept as requested, please action upstream" simply lines up
behind work that's already in flight in the right place.

---

### Disposition B: Alternative Approach

#### Proposal

Do not modify the value set. Instead, for `DeviceRequest.intent` only,
relax the element to `0..1` and (where structurally permitted) loosen the
binding so that absence of intent is the spec's escape valve — i.e., when
the requester does not know the intent, the element is omitted. This was
the path FHIR-55275 briefly took on 2026-02-20 ("Persuasive with
Modification: relax cardinality for .intent to 0..1; change binding to
extensible; ask FHIR-I to consider a similar change to the request
pattern", 5-0-0).

#### Justification

This avoids touching a value set / code system OO does not own and lets OO
ship a fix without waiting on FHIR-I and CDS. It also acknowledges that
"unknown intent" is arguably a *missing* value, not a *coded* value, which
is what the FHIR data-absent / cardinality story is for. **However**, this
approach ran into a hard objection on FHIR-55275 within four days
(2026-02-23 / 2026-02-24): a `code` element cannot be re-bound as
`extensible`, and the reporter explicitly clarified that the ask was to
extend the value set, not to weaken the binding. The motion was reopened
6-0-0. The same objections apply verbatim to `DeviceRequest.intent`, which
is also `code`-typed and is part of the same shared `Request` pattern, so
relaxing cardinality on DeviceRequest in isolation also breaks the
pattern's promise that intent is always populated. Disposition B is listed
for completeness and as a fallback if FHIR-56142's fix is delayed past R6.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** for OO with an explicit note that the
defect is real but is being handled at the responsible work group:
`DeviceRequest.intent` is bound to a value set owned by **FHIR-I** and a
code system owned by **CDS**, both of which are out of OO's editorial
scope, and the upstream fix is already tracked under
[FHIR-56142](https://jira.hl7.org/browse/FHIR-56142). Close FHIR-55262
without action in OO; once FHIR-56142 lands, the new code(s) flow into
`DeviceRequest.intent`'s value set automatically with no OO-side change.

#### Justification

OO genuinely cannot resolve this ticket itself — the editable artifacts
live in other groups' authoring trees. The 2026-02-12 OO main-call decision
("complete our review … note our recommendation, and then re-assign the
ticket") and the 2026-03-19 Zulip thread (Lloyd McKenzie: "tracker item for
the 'request' pattern … tag it as grouping 'workflow'") both point at the
same outcome: the right place for the fix is FHIR-I/CDS, and that work has
already been opened and ruled persuasive. Declining cleanly avoids
duplicating tracking on a ticket OO can't action, and keeps the audit
trail in one place. The downside is that "Not Persuasive" reads, to a
reader scanning the ticket history, like the *defect itself* was rejected,
which is not what OO means; the comment text needs to be very explicit
about that.

---

### Recommendation

**Recommended disposition:** A.

The substantive ask is correct (the element really does lack an escape
valve) and the upstream fix is already in flight at FHIR-I via
[FHIR-56142](https://jira.hl7.org/browse/FHIR-56142), which is **Resolved
/ Persuasive** and which explicitly names FHIR-55262 as one of the
tickets it closes. The cleanest, most defensible OO action is to mark
FHIR-55262 **Persuasive** mirroring FHIR-56142, append a comment naming
FHIR-56142 as the implementing change, and re-assign FHIR-55262 to FHIR-I
so the disposition history follows the work. Disposition B is the option
OO already tried on FHIR-55275 and which was reverted within four days
because `code`-typed elements can't carry an extensible binding; applying
it here would re-litigate that same dead end and would diverge
`DeviceRequest.intent` from the rest of the `Request` pattern. Disposition
C reaches the right end state (no OO file changes) but is easy to
misread as rejecting the defect, so it should only be used if FHIR-56142
unexpectedly stalls.
